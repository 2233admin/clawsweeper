## Summary

Keep open. Current `main` still lacks the proxy-fetch normalization this PR proposes, while the Discord proxy path on `main` still builds global `FormData` multipart requests and routes them through injected proxy fetches. Nothing in the current code or tests shows the report is obsolete, already shipped, or safe to auto-close.

## Evidence

- **Current main still passes proxy fetch inputs straight through to undici:** `makeProxyFetch` and `resolveProxyFetchFromEnv` both cast `input` to `string | URL` and forward `init` plus `dispatcher` directly to `undiciFetch`. There is no Request flattening, no FormData conversion, and no header/body normalization in the current implementation.
  - file: [src/infra/net/proxy-fetch.ts:15](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/infra/net/proxy-fetch.ts#L15)

- **Main already has analogous FormData normalization elsewhere, but not in proxy-fetch:** `src/infra/net/runtime-fetch.ts` contains `normalizeRuntimeFormData` and strips `content-length`/`content-type` after rebuilding runtime FormData, which reinforces that cross-runtime FormData handling is a known problem in this repo. That logic is absent from `proxy-fetch.ts` on current `main`.
