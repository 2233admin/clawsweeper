## Summary

Keep #71555 open. Current main still pins `@mariozechner/pi-coding-agent` 0.70.2 and the embedded runner still opens the transcript with `SessionManager.open(params.sessionFile)` before creating the Pi agent session. The pinned upstream `SessionManager.open()` source still fully reads/parses the file once to inspect the header and then the constructor calls `setSessionFile()`, which reads/parses the same file again; `createAgentSession()` then calls `sessionManager.buildSessionContext()`. Existing fixes for stale lock cleanup and tool-result truncation are useful adjacent mitigations, but they do not remove the reported double-load/large-transcript crash path for existing large JSONL sessions.

## Best Possible Solution

Keep this open as a core embedded-runner/session-management bug. The best fix is either an upstream `pi-coding-agent` update or an OpenClaw-owned adapter that avoids double full-file reads for `SessionManager.open()`, reads only the header when resolving cwd, and bounds or streams large transcript parsing before `createAgentSession().buildSessionContext()`. Add focused regression coverage for large JSONL restore/rebuild behavior, and consider making post-compaction transcript truncation or doctor repair more proactive for existing oversized transcripts.
