## Summary

Keep open. Current `main` still has macOS LaunchAgent lifecycle paths that explicitly fall back to `launchctl bootout` and can leave the service unloaded, and the CLI/doctor code now contains dedicated recovery logic for exactly the installed-but-unloaded LaunchAgent state described in the issue. That is evidence of mitigation, not proof the underlying lifecycle bug is fixed.

## Evidence

- **Current stop path can unload the LaunchAgent on failure:** `stopLaunchAgent()` disables the service, then falls back to `bootout` when `disable`, `stop`, or stop-state confirmation fails, with user-facing warnings that it 'left service unloaded'. This matches the issue's core symptom of the job disappearing from the launchd domain.

That means current main still treats 'unloaded after fallback' as an expected outcome, not a fixed regression.
