## Summary

Keep this issue open. Current main still has correct runtime cron timezone evaluation, but the reported bug is model argument selection for CronCreate/cron.add. The model-facing cron tool schema and description still do not explicitly tell the model that a cron expression paired with `tz` must use that timezone's local wall-clock time and must not be UTC-converted.

## Best Possible Solution

Make a narrow model-facing fix in `src/agents/tools/cron-tool.ts`: explicitly state that for `schedule.kind="cron"`, `expr` is written in the supplied `tz` timezone's local wall-clock time and must not be converted to UTC. Include the reported example, such as Shanghai 6pm -> `expr: "0 18 * * *"`, `tz: "Asia/Shanghai"`, and add focused schema/description regression coverage in `src/agents/tools/cron-tool.schema.test.ts` or the existing cron tool tests.
