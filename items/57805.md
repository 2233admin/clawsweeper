## Summary

Keep open. Current `main` no longer uses the PR’s old `provider-capabilities.ts` path, but the replacement replay-policy path still appears not to apply Mistral `strict9` tool-call ID handling when the owning provider is `openrouter` or another proxy. The underlying request does not look implemented on `main`.

## Evidence

- **Replay policy is resolved from the owning provider plugin only:** `resolveTranscriptPolicy` asks `resolveProviderRuntimePlugin` for the current provider and uses only that plugin’s `buildReplayPolicy`; it does not consult foreign provider plugins for replay policy.
  - file: [src/agents/transcript-policy.ts:177](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/transcript-policy.ts#L177)
  - command: `nl -ba src/agents/transcript-policy.ts | sed -n '169,210p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Runtime plugin lookup is provider-id scoped:** `resolveProviderRuntimePlugin` filters by `providerRefs: [params.provider]` and returns the plugin matching that provider ID or alias. A Mistral plugin replay hook will not be selected for an `openrouter` request.
  - file: [src/plugins/provider-hook-runtime.ts:149](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/plugins/provider-hook-runtime.ts#L149)
  - command: `nl -ba src/plugins/provider-hook-runtime.ts | sed -n '149,176p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **OpenRouter does not define a strict9 replay policy:** The OpenRouter plugin spreads `PASSTHROUGH_GEMINI_REPLAY_HOOKS`. That helper only adds Gemini thought-signature handling and does not set `sanitizeToolCallIds` or `toolCallIdMode` for OpenAI-compatible requests.
  - file: [extensions/openrouter/index.ts:140](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/openrouter/index.ts#L140)
