## Summary

Keep this PR open. Current main still lacks configurable per-session lane concurrency, and the PR head now addresses the earlier Greptile hot-reload/test-reset feedback, but it is still not mergeable as-is: the new public config key is absent from the strict schema, and the diff removes unrelated gateway reload, SDK export, and public type surface behavior.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is a narrow rebase on current main that preserves existing gateway reload/security code, SDK exports, and public config type fields; adds `sessionLaneMaxConcurrent` to the strict schema, generated config metadata/help/docs, and focused resolver/queue/reload tests; and leaves #53023 open for the broader auto-yield and lane-wait-threshold work unless those are implemented separately.

## Risks / Open Questions

- PR head likely breaks gateway startup because `startManagedGatewayConfigReloader` is still imported but no longer exported.
- The new `agents.defaults.sessionLaneMaxConcurrent` option is part of the public config type in the PR but is not accepted by the strict zod schema or backed by generated config metadata/docs/tests.
- The diff removes managed reload behavior around secrets snapshots, shared-auth generation rollback, channel reload deferral, and MCP runtime disposal; that is security-sensitive and unrelated to session lane concurrency.
- The SDK barrel change can break bundled plugin runtime imports, including WhatsApp auto-reply monitoring paths that depend on `getRuntimeConfigSourceSnapshot`.
- The PR only addresses the session-concurrency part of #53023; the broader auto-yield and lane-wait-threshold requests remain separate open work.
