## Summary

Keep open. Current `main` still does not accept or propagate `threadId` on `POST /hooks/agent`, while the downstream delivery path already has explicit thread support. The report matches current code and docs, and there is already related active work referenced from the issue (`#69859`).

## Evidence

- **Hook payload type omits threadId:** `HookAgentPayload` includes `message`, `name`, `agentId`, `sessionKey`, `deliver`, `channel`, `to`, `model`, `thinking`, and `timeoutSeconds`, but no `threadId` field.
  - file: [src/gateway/hooks.ts:213](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/hooks.ts#L213)
  - command: `nl -ba src/gateway/hooks.ts | sed -n '208,245p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Hook normalizer drops threadId:** `normalizeAgentPayload(...)` reads and returns the same fields above and never parses `payload.threadId`, so a caller-supplied thread id is ignored before dispatch.
  - file: [src/gateway/hooks.ts:399](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/hooks.ts#L399)
  - command: `nl -ba src/gateway/hooks.ts | sed -n '399,470p'`

  - file: [src/gateway/server-http.ts:683](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/server-http.ts#L683)

This supports the issue's claim that the gap is at hook payload parsing, not deeper delivery plumbing.
