## Summary

Keep open: current `main` still routes active-memory aborts through an embedded-runner cleanup path that waits on `waitForIdle()` with a default 30s ceiling and no abort-aware fast path, so the reported 15-20s timeout overshoot remains plausible and no current-main fix or timing regression test was found.

## Evidence

- **Active-memory measures elapsed time around the full embedded run promise:** `maybeResolveActiveRecall()` starts its timer before creating an `AbortController`, aborts at `config.timeoutMs`, then still awaits `runRecallSubagent()`. The logged `elapsedMs` is computed only after that awaited call returns, so any delayed runner cleanup directly inflates the plugin's reported timeout duration.
  - file: [extensions/active-memory/index.ts:1733](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/active-memory/index.ts#L1733)
  - command: `nl -ba extensions/active-memory/index.ts | sed -n '1721,1815p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Abort signal is forwarded into the embedded runner:** `runRecallSubagent()` passes the active-memory `AbortSignal` through to `runEmbeddedPiAgent()`, so the issue is not a missing signal handoff from the plugin layer.
  - file: [extensions/active-memory/index.ts:1664](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/active-memory/index.ts#L1664)
  - command: `nl -ba extensions/active-memory/index.ts | sed -n '1587,1718p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Embedded attempt always waits for cleanup before returning:** The attempt runner returns its result object, but its `finally` block still awaits `cleanupEmbeddedAttemptResources()` before the promise settles. That means an abort does not let callers return until cleanup finishes.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:3022](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/pi-embedded-runner/run/attempt.ts#L3022)
  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.ts | sed -n '2968,3105p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Cleanup uses wait-for-idle with no abort override:** `cleanupEmbeddedAttemptResources()` calls `flushPendingToolResultsAfterIdle()` with `clearPendingOnTimeout: true` but no shorter timeout and no abort signal. The helper itself races `waitForIdle()` only against `DEFAULT_WAIT_FOR_IDLE_TIMEOUT_MS = 30000`, then flushes or clears pending tool results. There is no abort-aware early exit in this helper.
  - file: [src/agents/pi-embedded-runner/wait-for-idle-before-flush.ts:10](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/pi-embedded-runner/wait-for-idle-before-flush.ts#L10)
  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.subscription-cleanup.ts | sed -n '17,45p'; nl -ba src/agents/pi-embedded-runner/wait-for-idle-before-flush.ts | sed -n '1,58p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Current tests do not cover prompt return latency after timeout:** The active-memory regression test only proves that late payloads are ignored after the timeout signal fires. It does not assert that the embedded run returns promptly after abort, so it does not contradict the reported 15-20s overshoot.
  - file: [extensions/active-memory/index.test.ts:1186](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/active-memory/index.test.ts#L1186)
