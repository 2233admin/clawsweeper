## Summary

Keep open. Current `main` still performs strict timestamp ordering checks for task audit findings, so this PR is not already implemented and the proposed change still affects a documented user-visible warning surface.

## Evidence

- **Current main still flags any backward timestamp without tolerance:** `findTimestampInconsistency` on current `main` uses strict `<` comparisons and truthiness guards (`task.startedAt && ...`, `task.endedAt && task.startedAt && ...`). There is no jitter window, so a 1 ms inversion would still produce `inconsistent_timestamps`.
