## Summary

Keep #69936 open. Current main still does not consult `approvals.{exec,plugin}.targets[].accountId` when native approval runtimes choose which Telegram account should handle an approval request, so the PR is not obsolete or implemented on main.

## Best Possible Solution

Keep the PR open or replace it with a maintainer-owned implementation that adds a conservative shared native approval account-binding fallback for unambiguous configured approval targets, gated by the same enabled/filter predicates as the forwarder and covered by Telegram plus shared helper regression tests. Coordinate with #69916 as the user-facing bug tracker.
