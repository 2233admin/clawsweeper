## Summary

Keep open. Current main still treats session persistence as filesystem-backed metadata plus JSONL transcript files: config exposes `session.store` as a path, session store load/save uses local file APIs, gateway discovery scans disk-backed `sessions.json` files, and the embedded runner opens per-session transcript paths through `SessionManager`. The plugin runtime exposes helper access to the current store, but not a backend registration API. Related #57497 is a narrower open PostgreSQL request, not a shipped implementation or canonical duplicate for this broader external unified storage abstraction.

## Best Possible Solution

Keep this issue open for core session persistence design. The likely product path is an explicit session persistence contract that covers both session metadata and transcript/history semantics, preserves the filesystem backend as the default, adds config/schema/docs, and proves at least one multi-instance-safe backend. A plugin-facing backend registration API should follow only after the core contract is explicit enough for third-party backends.

## Risks / Open Questions

- A metadata-only external store would not satisfy the reported multi-instance loss because transcript history, repair, compaction, and runner paths still depend on local JSONL files.
- A correct implementation needs maintainer design for backend contracts, transcript semantics, locking/concurrency, TTL/maintenance, migration, config/schema/docs, and secure credential handling.
