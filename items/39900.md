## Summary

Keep #39900 open. Current main still accepts provider-owned TTS config broadly, but the bundled OpenAI speech path does not normalize or forward `extraBody` / `extra_body` into the `/audio/speech` request body. The closed #39909 item is a duplicate-style cross-reference, not a canonical replacement; #26952 is LLM completions-only, and #5423 only covered `baseUrl`.

## Best Possible Solution

Implement a narrowly scoped `extraBody` / `extra_body` option in the bundled OpenAI speech provider: normalize it as a plain JSON object, pass it to `openaiTTS`, merge it into the `/audio/speech` body with documented precedence, add focused tests under `extensions/openai`, and document it on the public TTS/OpenAI provider docs. If maintainers want arbitrary custom TTS transports to be plugin-first instead, record that explicit product decision while leaving this issue open until the OpenAI-compatible TTS gap is addressed or superseded by a canonical item.

## Risks / Open Questions

- Adding this needs a clear collision policy for generated keys such as `model`, `input`, `voice`, `response_format`, `speed`, and `instructions`.
- Tests should cover both official OpenAI behavior and custom `baseUrl` behavior so provider-specific passthrough does not break standard TTS requests.
