## Summary

Current `main` now documents the post-2026.3.13 WebSocket device-auth contract and the browser pairing/migration flow, and the code/tests explicitly scope the break-glass bypass to Control UI only. That satisfies the issue's documentation request on `main` even though the old token-only `webchat-ui` behavior was not restored.

## Close Comment

Closing from a Codex review because current `main` now documents and codifies the behavior that was unclear when this was filed. `docs/gateway/protocol.md:574` states that WebSocket clients must send device identity, with only narrow Control UI exceptions at `docs/gateway/protocol.md:575`-`578`. The browser pairing flow is now documented in `docs/web/control-ui.md:36`-`75`, and the migration details for `connect.challenge` / `device.nonce` are documented in `docs/gateway/troubleshooting.md:167`-`215`. On the implementation side, the break-glass bypass is intentionally scoped to Control UI operator sessions in `src/gateway/server/ws-connection/connect-policy.ts:121`, and `src/gateway/server.auth.browser-hardening.test.ts:348` explicitly verifies that non-Control-UI browser clients still require pairing. This means the documentation/migration path requested here is implemented on `main`, while `gateway.controlUi.dangerouslyDisableDeviceAuth` remains Control UI-only by design. Best proving commit on `main`: `3d8e6a3aa3a26870c4ae61fd27853914fc66cf96`.
