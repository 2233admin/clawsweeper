## Summary

Keep open. This is a fresh, concrete runtime bug report against v2026.4.24/current gateway behavior, and current main only shows partial Telegram/channel watchdogs. I found no shipped fix, duplicate, or superseding item that covers the reported gateway-wide CPU spin where Telegram ingress continues but replies and loopback status RPC stall.

## Best Possible Solution

Keep the issue open. The best maintainer path is to reproduce with the reporter's sanitized gateway/stability data, add useful event-loop/CPU/stuck-child diagnostics, and then harden the gateway with a watchdog or supervisor-level recovery that still works when the event loop is starved. Also isolate or block interactive channel login commands from active agent/gateway reply sessions, with regression coverage for status RPC responsiveness while a channel login or child process is stuck.
