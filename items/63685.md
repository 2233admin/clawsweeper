## Summary

Keep open. Current main supports the corrected Google Gemma 4 model id `gemma-4-26b-a4b-it`, but the reported `gemma-4-26b` id is still accepted as a forward-compatible Gemma model, normalized unchanged, and sent to Google as `models/gemma-4-26b`. The issue discussion identified the likely correct id, but current main still lacks an alias, validation, or setup/model-selection fix for the reporter's path.

## Best Possible Solution

Keep this issue open. The best fix is in the Google provider/setup surface: verify whether onboarding, model selection, or docs can lead users to `gemma-4-26b`; replace any such choice with `gemma-4-26b-a4b-it`, or add provider-owned validation/alias handling that fails before dispatch with a clear hint. Add regression coverage for the reported `gemma-4-26b` path separately from the existing corrected-id coverage. Review #68996 independently for native-route behavior, but do not treat it as a shipped resolution for this issue until merged and verified.
