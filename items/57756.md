## Summary

Keep open. Current main still resolves the reported session-key RPCs and subscriptions by method scope plus caller-supplied key, without a caller-device/session ownership check. The docs also describe sessions as gateway-owned rather than client-owned, so the right outcome needs explicit maintainer product/security judgment or an implementation that defines this new session access model.

## Best Possible Solution

Keep this issue open for maintainer review. The best path is to either implement a documented session-access authorization layer across `sessions.list/get/subscribe/messages.subscribe`, `chat.send`, session mutators, and session event delivery, or explicitly decide and document that `operator.read/write/admin` are gateway-wide session capabilities and close manually with that product/security rationale.

No caller-device/session ownership input is checked in these handlers.

## Risks / Open Questions

- A fix needs a clear ownership model for sessions that are intentionally shared across devices, channels, Control UI, TUI, subagents, ACP, and webchat flows.
- Changing this could break existing gateway-wide operator workflows unless admin bypasses, shared-session grants, and migration/default behavior are specified and tested.
