## Summary

Keep open. Current `main` still appears to authorize these gateway session surfaces only by method scope, then resolve/subscribe/broadcast by raw `sessionKey` without a caller-to-session ownership check.

## Evidence

- **Gateway RPC auth is method-scope-only:** `handleGatewayRequest` authorizes requests by role plus `authorizeOperatorScopesForMethod(method, scopes)` and does not add any per-session ownership check before dispatching session methods.
  - file: [src/gateway/server-methods.ts:44](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/server-methods.ts#L44)
  - command: `sed -n '1,120p' src/gateway/server-methods.ts`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Session listing exposes raw keys from the combined store:** `sessions.list` loads the combined gateway session store and returns `listSessionsFromStore(...)`; `GatewaySessionRow` includes `key`, and `buildGatewaySessionRow(...)` returns that key directly. No caller/device filtering is applied in the list path.
  - file: [src/gateway/server-methods/sessions.ts:579](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/server-methods/sessions.ts#L579)

- **Session subscription paths have no ownership guard:** `sessions.subscribe` unconditionally registers the caller connection in the global session-event subscriber set, and `sessions.messages.subscribe` canonicalizes the supplied key then subscribes that connection to the per-session registry without validating a caller/session relationship.
  - file: [src/gateway/server-methods/sessions.ts:594](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/server-methods/sessions.ts#L594)
  - command: `sed -n '260,760p' src/gateway/server-methods/sessions.ts`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Subscriber registries are global/keyed registries, not owner-scoped:** `createSessionEventSubscriberRegistry()` stores one global `Set<connId>`, and `createSessionMessageSubscriberRegistry()` stores `sessionKey -> Set<connId>` mappings. The broadcast handlers fan out `session.message`/`sessions.changed` to those registry members. No owner check is present there either.
  - file: [src/gateway/server-chat.ts:271](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/server-chat.ts#L271)

The same file shows reset/delete/patch/compact handlers operating on the supplied key without a caller/session authorization layer.
