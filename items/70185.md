## Summary

Keep open. Current main still does not register memory publicArtifacts in the CLI-metadata loader path, and memory-core's CLI metadata entry still registers only the CLI command. Main does include a guard that prevents bridge import from pruning everything when no memory capability is loaded, but that does not solve the reported 0-artifact CLI bridge import behavior.

## Best Possible Solution

Keep the PR open and ask for maintainer review or an updated patch that makes `openclaw wiki bridge import` load the selected memory plugin's `publicArtifacts` provider in the CLI process without bypassing memory slot selection. The best fix should include the CLI-metadata `registerMemoryCapability` handler with the same memory-slot guard and rollback behavior as full registration, update `memory-core/cli-metadata.ts` or the current command-loading path as needed, and add a regression test that exercises the actual CLI bridge import path instead of only directly registering memory state.
