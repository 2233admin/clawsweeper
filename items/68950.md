## Summary

Keep #68950 open. Current main still makes `memory-lancedb` an OpenAI API-key-only plugin: its config and manifest require `embedding.apiKey`, registration constructs the embeddings client from that key, and the OpenAI plugin only exposes a memory embedding provider for `openai`, not `openai-codex`. Docs now explicitly state that Codex OAuth covers chat/completions only and does not satisfy embedding requests. Active Memory is useful adjacent shipped work, but it does not close the specific LanceDB/OAuth auto-recall/capture gap reported here.

## Best Possible Solution

Keep the issue open for a maintainer-owned memory/provider fix. The clean implementation path is to either migrate `memory-lancedb` embedding resolution onto the shared memory embedding provider adapter surface or add a documented provider-routed embedding mode that can use an authenticated provider session when the provider actually supports embeddings. If OpenAI Codex OAuth cannot support embeddings, update memory-lancedb setup/validation docs to say so up front and document the supported OAuth/keyless alternative, such as Active Memory with GitHub Copilot, Bedrock, Ollama, or local embeddings.
