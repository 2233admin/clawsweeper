## Summary

Keep #64301 open. Current main has related malformed-tool-name handling, but it only covers narrower `functionsread3`/`functions.read`-style cases and does not implement the PR's central reverse-lookup behavior for stripped underscores, leading-dot variants, dropped-leading-character variants, or long hex suffixes such as `read5f3787893b0d4c49a`. The PR is a plausible focused bug fix, not clearly obsolete or duplicated by the provided related items.

## Best Possible Solution

Keep the PR open for maintainer review, but require it to be rebased onto current `main` and adapted to the existing `buildStructuredToolNameCandidates`/`inferToolNameFromToolCallId` flow. The best fix is a small, fail-closed reverse-lookup helper with focused tests for `read5f3787893b0d4c49a`, `functionsread5f3787893b0d4c49a`, stripped-underscore names, leading-dot names, dropped-leading-character names, and false positives like `blacklist1a2b3c` or `readfeed`.
