## Summary

Keep open. Current main still has timestamped dreaming narrative session keys and routes memory-core cleanup through admin-scoped `sessions.delete`; the PR addresses a real remaining surface, but should not be auto-closed or auto-merged as-is because its proposed synthetic-scope path needs to cover request-scoped clients and the branch includes unrelated runtime dependency changes.

## Best Possible Solution

Keep #70464 open for maintainer/author follow-up. The best fix belongs in memory-core plus the gateway plugin runtime: make narrative session keys stable per workspace/phase while keeping timestamp/run uniqueness in `idempotencyKey`; add bounded pre-run/final cleanup; and provide a narrowly authorized memory-core cleanup path or lifecycle-independent session cleanup API that works under both fallback and request-scoped clients, with regression tests for wrong plugin/session denial and request-scoped success. Drop unrelated provider runtime-deps changes from this PR.
