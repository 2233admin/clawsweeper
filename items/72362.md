## Summary

Keep open. Current `main` still routes non-Windows `writeFileWithinRoot` and `copyFileWithinRoot` through the Python pinned-write helper, and `normalizePinnedWriteError` still rewrites any non-`SafeOpenError` into the misleading `path is not a regular file under root` message while preserving the real spawn failure only as `cause`. The report is concrete, recent, non-maintainer-authored, has no protected labels, and describes an actionable core/runtime bug rather than a ClawHub/plugin boundary issue.

## Best Possible Solution

Keep the issue open and fix the core fs-safe write path. The best maintainer path is to make pinned writes robust without an undocumented runtime dependency: either remove the Python subprocess requirement, or classify `spawn python3` `ENOENT`/`EACCES`/`ENOEXEC` and fall back safely or throw an actionable `SafeOpenError` that names the missing interpreter. Add regression tests for `writeFileWithinRoot`/`copyFileWithinRoot` when the pinned write helper cannot spawn, and update Docker/runtime docs only if Python remains a supported requirement.
