## Summary

Keep open. Current main still has the reported write-path shape: non-Windows writes go through the pinned Python helper, the helper falls back to spawning `python3` when no known executable is found, and `writeFileWithinRoot` still normalizes non-`SafeOpenError` helper failures to `path is not a regular file under root`. The reported session-memory and `tools.fs.workspaceOnly: true` surfaces both still call this write path.

## Best Possible Solution

Fix the write path in core. Best maintainer path is to add spawn-error handling for `runPinnedWriteHelper` comparable to the pinned path helper, either falling back to a safe Node implementation when the Python helper cannot spawn or returning a clear `SafeOpenError` that names the missing Python dependency. Also preserve/log `cause` for session-memory failures and add regression tests for missing pinned write helper behavior; if Python remains required, document it in the Docker install docs and image build guidance.
