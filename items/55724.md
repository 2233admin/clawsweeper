## Summary

Keep open. Current `main` does not implement this PR's behavior; it explicitly allows owner-only `/tools/invoke` access for owner/admin contexts and documents that shared-secret bearer auth on this endpoint is a full trusted-operator boundary.

## Evidence

- **HTTP handler uses caller owner status, not forced non-owner:** `src/gateway/tools-invoke-http.ts:223-244` computes `senderIsOwner` via `resolveOpenAiCompatibleHttpSenderIsOwner(...)`, passes it into `resolveGatewayScopedTools(...)`, then filters with `applyOwnerOnlyToolPolicy(tools, senderIsOwner)`. That is materially different from this PR's proposed `applyOwnerOnlyToolPolicy(..., false)` behavior.
  - file: [src/gateway/tools-invoke-http.ts:223](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/gateway/tools-invoke-http.ts#L223)
  - command: `nl -ba src/gateway/tools-invoke-http.ts | sed -n '223,244p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Current tests assert the opposite policy:** `src/gateway/tools-invoke-http.test.ts:814-851` verifies owner-only tools are denied for `operator.write` callers but allowed for `operator.admin` callers and for shared-secret bearer auth on `/tools/invoke`. This conflicts with the PR's intended always-non-owner HTTP policy.
  - file: [src/gateway/tools-invoke-http.test.ts:814](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/gateway/tools-invoke-http.test.ts#L814)

  - file: [docs/gateway/tools-invoke-http-api.md:54](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/docs/gateway/tools-invoke-http-api.md#L54)
