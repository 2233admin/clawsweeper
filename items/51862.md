## Summary

Current main already supports per-group/per-channel context isolation through exact peer bindings to dedicated agents/workspaces, which covers the underlying request without adding a new `workspaceFiles` field on channel config.

## Close Comment

Codex review: closing as already implemented on current `main`.

OpenClaw now has first-class per-conversation agent routing, which already provides the requested context isolation by routing a specific Telegram group or Discord channel to a dedicated agent workspace. The current docs define an agent as its own workspace files + session store (`docs/concepts/multi-agent.md:8`), and the channel routing docs explicitly say groups/channels stay isolated and that exact peer bindings choose which agent workspace/session store is used (`docs/channels/channel-routing.md:34`, `docs/channels/channel-routing.md:62`). The config contract also documents `bindings[].match.peer` for pinning one direct/group/channel target to an agent (`src/config/schema.help.ts:499`). On the Telegram path, runtime routing builds the group peer id and applies configured binding routing before the run continues (`extensions/telegram/src/conversation-route.ts:37`, `extensions/telegram/src/conversation-route.ts:94`). Verified against `main` at `aa3a5f9ff7907c2a41d6259d24b529bda41f293a`.

So while the exact `workspaceFiles` config shape proposed here is not present, the underlying per-group/per-channel workspace isolation request is already supported on `main` via dedicated agents plus `bindings[].match.peer`.
