## Summary

Keep #58626 open. Current main has related bundled-plugin path cleanup, but it does not cover the reported failure mode: a versioned pnpm/npm global package path from a previous OpenClaw install remains in `plugins.load.paths`, discovery still treats that missing path as a config error, and gateway startup still fails on invalid config. The issue is recent, externally reported with a concrete reproduction, and a commenter confirmed the same root cause for npm global installs.

## Best Possible Solution

Keep the issue open and implement a doctor-owned repair for stale packaged bundled plugin paths from prior npm/pnpm global package roots. A conservative fix should recognize paths ending in `node_modules/openclaw/(dist|dist-runtime)/extensions/<plugin>` or `openclaw/extensions/<plugin>` for known bundled plugin ids, remove redundant `plugins.load.paths` aliases during `openclaw doctor --fix`, preserve unrelated custom plugin paths, migrate or refresh stale install records safely, and add regression tests using an old versioned pnpm-style package path like the ACPX report.

## Evidence

- **Issue report and confirmation:** The issue reports `plugins.load.paths` pointing at `/Users/link/Library/pnpm/global/5/.pnpm/openclaw@2026.3.28.../node_modules/openclaw/dist/extensions/acpx` after upgrading to 2026.3.31, causing `openclaw gateway start` to fail. The lone comment confirms the same absolute-path upgrade breakage for npm global installs and suggests auto-repairing the path.
- **Missing plugin load path still invalidates config:** Plugin discovery emits an error when a configured load path does not exist, with message `plugin path not found: ${resolved}`. Config validation maps that diagnostic to `plugins.load.paths`, matching the reported failure surface.
