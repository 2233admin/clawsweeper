## Summary

Keep this PR open. The reported Telegram portrait-video bug is still plausible on current main because both outbound Telegram video paths call `sendVideo` without width/height, but the submitted diff should not be merged as-is: it crosses the plugin boundary, adds a bare `ffprobe` execution path, changes an unrelated global media limit, and lacks regression coverage for the requested behavior.

## Best Possible Solution

Keep this PR open and request a focused revision: implement a public or Telegram-appropriate video-dimension helper through `openclaw/plugin-sdk/media-runtime` that uses the trusted ffprobe resolver, pass width and height only for regular Telegram `sendVideo` calls in `extensions/telegram/src/send.ts` and `extensions/telegram/src/bot/delivery.replies.ts`, keep GIF animations and video notes excluded, add regression tests for both outbound paths, and remove or split the unrelated global `MAX_VIDEO_BYTES` change.
