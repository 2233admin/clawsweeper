## Summary

Current `main` does not contain this PR's Control UI buffering changes, and the remaining UI code path still allows image attachments to be absent at send time while `FileReader` work is in flight. The gateway already has a newer attachment pipeline on `main`, so the PR is not implemented as-is, but the underlying webchat send bug is not conclusively resolved either.

## Evidence

- **UI still waits for all async reads before exposing attachments:** `readDroppedFilesIntoAttachments` only calls `onAttachmentsChange` after every `FileReader` finishes (`pending === 0`). Until then, the attachment list visible to send logic stays unchanged, so a fast send can race ahead of image staging.
