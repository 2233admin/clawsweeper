## Summary

Keep open. Current main still only logs and swallows task registry restore failures; the SQLite-backed task store has no corruption detection, bad-DB snapshot, `.recover` path, rebuild fallback, or task-registry audit finding for malformed `tasks/runs.sqlite`. The issue is recent, author is not protected, and related ClawSweeper matches cover different task/image/chmod symptoms rather than superseding this SQLite corruption report.

## Best Possible Solution

Keep the issue open and implement a focused core recovery path in the task registry SQLite store. The likely fix belongs around `src/tasks/task-registry.store.sqlite.ts`/`src/tasks/task-registry.ts`: classify SQLite corruption/open/load errors, close any cached handle, snapshot `runs.sqlite` and WAL sidecars, rebuild or recover into a fresh DB when feasible, expose an operator-visible restore-failure/recovery event or audit finding, and add targeted tests for malformed/unreadable registry images plus docs for the recovery behavior.
