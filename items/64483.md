## Summary

Current main still keeps `modelstudio` as a Qwen compatibility alias and still runs Qwen built-in suppression before the custom configured-provider fallback can resolve the reported `modelstudio/qwen3.6-plus` setup. This issue should remain open.

## Best Possible Solution

Keep this issue open and fix the provider-precedence contract. A conservative implementation would preserve exact custom provider ids such as `models.providers.modelstudio` when explicitly configured, restrict `modelstudio` aliasing to legacy Qwen compatibility when no explicit custom provider owns that id, run custom configured-provider resolution before Qwen built-in suppression for this case, and add a regression test for `modelstudio/qwen3.6-plus` with `api: "openai-completions"` and a Coding Plan base URL. Update the Qwen docs once the compatibility rule is settled.
