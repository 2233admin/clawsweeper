## Summary

Keep #57389 open. Current main still does not accept `api_key` as an auth-profile API-key value alias and still resolves only `key`/`keyRef`; a profile containing only `api_key` is parsed without a credential value, becomes ineligible/missing, and is skipped. The docs now describe the canonical `{ provider, key }` shape, but the reported silent alias failure remains a real, focused product bug. The linked fix PR #57418 is closed/unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open and implement a narrow compatibility fix in `src/agents/auth-profiles/persisted.ts`: map `api_key` to canonical `key` when `key` is absent, preserving canonical-field precedence over aliases. Add focused coverage next to the existing alias cases in `src/agents/auth-profiles.ensureauthprofilestore.test.ts`, and consider a load-time diagnostic for unknown credential fields so future auth-profile shape mistakes are not silent. The closed #57418 PR is a useful prior attempt, but the final patch should be small and current-main based.
