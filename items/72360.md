## Summary

Keep #72360 open. Current main still skips the `configWrites` authorization path for `/allowlist ... --store`: `parsed.target === "store"` makes `shouldUpdateConfig` false, the only `resolveConfigWriteDeniedText` call is inside the config branch, and the store branch still calls `updatePairingStoreAllowlist` directly. The supplied timeline links open PR #72361 as the proposed fix, but it is not merged into main, so closing the issue now would be premature.

## Best Possible Solution

Keep #72360 open until main includes a fix. The likely product fix is to apply `resolveConfigWriteDeniedText` to store-targeted allowlist mutations before `updatePairingStoreAllowlist`, using the same channel/account target semantics as config-backed edits, and add a regression test for `/allowlist add|remove --store` with `channels.<provider>.configWrites=false` and account-level `configWrites=false`. Review or port PR #72361 as the active candidate rather than closing the issue as implemented.
