## Summary

Keep #72360 open. Current main at 998e37fcb3d5b937fcf70d7509b677efba939f10 still has the reported `/allowlist --store` gap: store-targeted edits skip the config-update branch where `resolveConfigWriteDeniedText` runs, then call `updatePairingStoreAllowlist` directly. The linked PR #72361 is open and unmerged, so it is useful related work but not a shipped fix. One nuance: current code does reject non-owner add/remove commands before this path, so the maintainer framing should be the documented channel `configWrites` boundary bypass for owner-initiated channel writes, not an arbitrary authorized non-owner path unless another owner-authorizing path is shown.

## Best Possible Solution

Keep the issue open and review/merge PR #72361 or an equivalent narrow patch: apply `resolveConfigWriteDeniedText` before store-targeted `/allowlist add|remove --store` mutations using the explicit channel/account config-write target, and add regression tests for channel-level disabled, account-level disabled, and allowed store edits. Consider adjusting severity wording to reflect the current owner-only command gate while still treating `configWrites=false` as the documented channel write boundary.
