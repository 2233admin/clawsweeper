## Summary

Keep open. Current main has an intentional default daily reset at 4:00 AM local time, which plausibly explains the 11 PM to 5 AM WeChat context loss, but the issue is fresh and lacks the session/config/log evidence needed to confirm expected behavior versus a real regression in session routing, WeChat plugin session keys, transcript persistence, or provider context replay.

## Best Possible Solution

Keep the issue open for maintainer triage. Ask the reporter for session.reset, session.resetByChannel, session.dmScope, WeChat plugin version, /status before and after the overnight gap, the affected session key, redacted sessions.json timestamps (sessionStartedAt, lastInteractionAt, updatedAt), and gateway logs around 4:00-5:00 AM. If the default daily reset explains it, the best product fix is likely clearer first-turn reset notice or docs/config guidance for long-lived WeChat sessions; if not, add a focused regression test around WeChat/direct-session reuse or channel session-key stability.
