## Summary

Current `main` still contains the reported failure path. The Qwen plugin intentionally strips `qwen3.6-plus` from Coding Plan configs/catalogs, and gateway image-capability resolution still returns `false` on catalog miss instead of consulting explicit `models.providers.*.models[]` config. The issue remains actionable and should stay open.

## Evidence

- **Qwen Coding Plan still removes `qwen3.6-plus` from explicit provider config:** The bundled Qwen provider `normalizeConfig` drops `qwen3.6-plus` whenever the configured base URL is a Coding Plan endpoint, so an explicit `models.providers.qwen.models[]` entry for that model is not preserved through provider normalization.
  - file: [extensions/qwen/index.ts:168](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/qwen/index.ts#L168)
  - command: `nl -ba extensions/qwen/index.ts | sed -n '160,200p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Provider normalization is applied before models.json/model catalog generation:** `normalizeProviders` calls `normalizeProviderSpecificConfig` for each configured provider before the merged provider set is finalized, so the Qwen normalization above affects the generated gateway model catalog.
  - file: [src/agents/models-config.providers.normalize.ts:116](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/models-config.providers.normalize.ts#L116)
  - command: `nl -ba src/agents/models-config.providers.normalize.ts | sed -n '115,140p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Gateway image support still fails closed on catalog miss:** `resolveGatewayModelSupportsImages` only checks the loaded gateway model catalog plus narrow Foundry/Claude shims. If the model is absent from the catalog, it returns `false`; there is still no fallback to explicit provider config as proposed in the issue/comment.
  - file: [src/gateway/session-utils.ts:1053](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/session-utils.ts#L1053)
  - command: `nl -ba src/gateway/session-utils.ts | sed -n '1052,1112p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Bundled tests still assert Coding Plan does not advertise `qwen3.6-plus`:** The current Qwen provider tests expect `qwen3.6-plus` to be absent on Coding Plan endpoints and present only on Standard endpoints, matching the behavior the reporter identified as the trigger for the attachment drop.
  - file: [extensions/qwen/provider-catalog.test.ts:21](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/qwen/provider-catalog.test.ts#L21)
  - command: `nl -ba extensions/qwen/provider-catalog.test.ts | sed -n '18,27p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Issue discussion already identified the same missing fallback and proposed fixes:** The provided timeline shows a confirming root-cause comment plus two follow-up PRs (#67766 and #67768) linked from the issue. Current `main` does not contain the corresponding fallback in `resolveGatewayModelSupportsImages`, which supports keeping the issue open rather than closing as already fixed.
