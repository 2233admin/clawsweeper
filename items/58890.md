## Summary

Keep open. Current main contains useful launchd restart hardening, but it does not close this issueâ€™s specific auto-update failure path: the gateway still launches `openclaw update --yes --json` through `runCommandWithTimeout`, which keeps stdout/stderr piped to the gateway parent, and the update flow still refreshes the LaunchAgent with `gateway install --force` before running the prepared restart script. That refresh path still reaches `launchctl bootout` in the install activation path, so the reported parent-pipe/restart-handoff ordering concern remains actionable rather than implemented or fully superseded by the related closed restart issues.

## Best Possible Solution

Keep the issue open and fix the auto-update-specific handoff. The safest direction is to make gateway-owned auto-update independent of the gateway parent before any LaunchAgent bootout can occur: either run the auto-update subprocess detached with ignored stdio or schedule the launchd restart/recovery handoff before `refreshGatewayServiceEnv`, and add a regression test that covers `OPENCLAW_AUTO_UPDATE=1`, `gateway install --force`, and restart-script ordering. Also consider changing the install refresh path so it avoids booting out the currently running LaunchAgent from inside its own process tree.
