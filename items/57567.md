## Summary

Keep open. Current `main` has newer config recovery work, but the reported Feishu upgrade path is not clearly fixed: legacy `accounts.*.botName` is still invalid under the strict Feishu schema, legacy `accounts.default` is still treated as a real default account if present, and invalid-JSON recovery is wired to `.last-good`/prefix stripping rather than a Feishu-specific migration or `.bak` rollback.

## Evidence

- **Legacy `botName` still rejected by current Feishu schema:** `FeishuAccountConfigSchema` allows `name` but not `botName`, and the schema is strict, so the legacy `accounts.main.botName` shape from the issue would still fail validation on current `main`.
  - file: [extensions/feishu/src/config-schema.ts:191](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/feishu/src/config-schema.ts#L191)
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Legacy `accounts.default` sentinel is still selected as the active default account:** If `channels.feishu.defaultAccount` is unset and `accounts.default` exists, current account resolution returns `default` as the mapped default account. That means the old sentinel-style `accounts.default` entry is still treated like a real account, which matches the upgrade hazard described in the issue.
  - file: [extensions/feishu/src/accounts.ts:144](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/feishu/src/accounts.ts#L144)
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Doctor migrations depend on channel compatibility hooks, but Feishu exposes no doctor migration hook here:** `applyLegacyDoctorMigrations()` only applies the shared legacy migration list plus channel compatibility normalizers. The Feishu public setup/contract surfaces shown here export setup/runtime APIs but no `normalizeCompatibilityConfig` or other doctor migration hook for this schema change.
  - file: [src/commands/doctor/shared/legacy-config-compat.ts:4](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/commands/doctor/shared/legacy-config-compat.ts#L4)

  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Invalid JSON recovery is `.last-good`/prefix based, not `.bak`/doctor migration based:** On startup, invalid config recovery tries `recoverConfigFromLastKnownGood()` and then `recoverConfigFromJsonRootSuffix()`. The recovery implementation restores from `openclaw.json.last-good`, not from `openclaw.json.bak`, and `readConfigFileSnapshotInternal()` reports raw JSON5 parse failure without a `.bak` restore path.
