## Summary

Keep open. Current main mitigates the narrow Anthropic default trigger from #39903, but it does not implement this PR's general fail-fast startup auth guard. The PR is still a plausible focused fix, but it needs revision because its proposed auth check is narrower than runtime auth resolution and the review feedback about false-positive failures and circular suggestions remains valid.

## Best Possible Solution

Keep the PR open for revision. The best path is to implement the guard in the current post-attach startup flow by resolving the selected default model and preflighting it through the same auth contract runtime execution uses, such as hasAvailableAuthForProvider or a non-mutating resolveApiKeyForProvider wrapper. Preserve the skipChannels bypass, avoid circular model suggestions, and add focused coverage for missing auth plus valid config apiKey, Bedrock/AWS SDK, local synthetic/plugin auth, and the openai-codex example edge case.

## Risks / Open Questions

- Merging the PR as-is could block valid configurations that authenticate through models.providers.<provider>.apiKey, Bedrock/AWS SDK fallback, synthetic local auth, or plugin-owned auth flows.
- The proposed hardcoded example can suggest switching to the same failing openai-codex/gpt-5.4 model.
- Closing this PR as implemented would lose a still-unimplemented general startup guard for explicitly configured default models with missing auth.
