## Summary

Keep #69102 open. Current main does not expose or forward Gemini Flex `service_tier` for the bundled Google/Gemini provider; existing service tier support is limited to other provider paths.

## Best Possible Solution

Add bundled Google provider support for `agents.defaults.models["google/<model>"].params.serviceTier` and/or `service_tier`, mapping it to top-level Gemini REST `service_tier` for direct `google-generative-ai` streaming requests. Cover it with focused tests in `extensions/google/transport-stream.test.ts` or `src/agents/pi-embedded-runner/extra-params.google.test.ts`, and document it in `docs/providers/google.md` using the public provider docs surface.

## Evidence

- **Upstream Gemini Flex contract:** Google's Gemini Flex documentation says Flex is selected by adding `service_tier: "flex"` to the request body; the JavaScript SDK example uses `config: { serviceTier: "flex" }`, while REST sends top-level `service_tier`. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/flex-inference))
- **Gemini transport request shape omits service tier:** The direct Gemini request type only contains `cachedContent`, `contents`, `generationConfig`, `systemInstruction`, `tools`, and `toolConfig`; `buildGoogleGenerativeAiParams` only maps temperature, max tokens, thinking config, cached content, system prompt, tools, and tool choice before serializing the request.

## Risks / Open Questions

- The implementation should not reuse the OpenAI service-tier wrapper blindly; Gemini Flex needs the Gemini REST/SDK field shape and should be tested on the direct `google-generative-ai` transport.
- Flex may need timeout guidance or headers in docs, because the upstream docs describe higher latency and timeout considerations for Flex requests.
