## Summary

Current `main` does not preserve `/new` across later routing-key changes. Session reset still applies to the current `sessionKey`, and when DM routing switches back to `agent:main:main`, `initSessionState()` will load that older entry again. The only dmScope migration logic on `main` retires legacy delivery metadata; it does not migrate or rebind the fresh session created by `/new`.

## Evidence

- **Explicit inbound session key wins:** `resolveSessionKey()` returns the explicit `ctx.SessionKey` immediately. If a channel/router later starts emitting `agent:main:main` again, session resolution follows that key rather than the earlier per-channel `/new` lane.
