## Summary

Keep open. Current `main` still writes `/acp spawn --label` through the requester's `storePath` instead of resolving the spawned ACP session's store from its `sessionKey`, so the cross-agent label-persistence bug described in PR #57597 still appears present and the PR's core fix is not already implemented on `main`.

## Evidence

- **Current main still persists the label to the requester store path:** `persistSpawnedSessionLabel` updates `params.commandParams.sessionStore` and then calls `updateSessionStore(params.commandParams.storePath, ...)`; it never resolves the spawned session's target store from `sessionKey`.
  - file: [src/auto-reply/reply/commands-acp/lifecycle.ts:458](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/auto-reply/reply/commands-acp/lifecycle.ts#L458)
  - command: `nl -ba src/auto-reply/reply/commands-acp/lifecycle.ts | sed -n '448,478p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Spawned ACP sessions are keyed to the target agent, not necessarily the requester agent:** `handleAcpSpawnAction` creates the new session key as `agent:${spawn.agentId}:acp:...`, so cross-agent spawns naturally point at a different agent-owned store than the command's current session may use.
  - file: [src/auto-reply/reply/commands-acp/lifecycle.ts:518](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/auto-reply/reply/commands-acp/lifecycle.ts#L518)
