## Summary

Keep open. Current `main` still appears to have the reported gap: normal `openclaw memory status` rebuilds `dirty` from whether index metadata exists, not from incomplete source coverage or a persisted failed-sync marker, while concurrent per-file indexing can leave partial rows behind after a transient embeddings failure.

## Evidence

- **Status-only dirty is derived only from meta presence:** `resolveInitialMemoryDirty()` returns `!hasIndexedMeta` for status-only managers. Once any prior index metadata exists, a fresh status-only manager starts clean even if the stored index is incomplete.
  - file: [extensions/memory-core/src/memory/manager-status-state.ts:26](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/memory-core/src/memory/manager-status-state.ts#L26)
  - command: `nl -ba extensions/memory-core/src/memory/manager-status-state.ts | sed -n '1,40p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **`memory status` uses the status-only manager path:** `runMemoryStatus()` selects `purpose: "status"` unless `--index` is passed, so the normal CLI status command goes through that meta-based dirty initialization path.
  - file: [extensions/memory-core/src/cli.runtime.ts:683](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/memory-core/src/cli.runtime.ts#L683)
  - command: `nl -ba extensions/memory-core/src/cli.runtime.ts | sed -n '665,706p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Incremental memory sync writes files independently before the whole pass succeeds:** `syncMemoryFiles()` builds one task per file, calls `indexFile()` inside each task, then awaits the task pool. There is no transaction or rollback around the overall incremental pass, so already-written files can remain indexed if another file later fails.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:721](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/memory-core/src/memory/manager-sync-ops.ts#L721)
  - command: `nl -ba extensions/memory-core/src/memory/manager-sync-ops.ts | sed -n '664,742p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Concurrency helper permits partial completion before surfacing the first error:** `runTasksWithConcurrency(..., errorMode: "stop")` stops scheduling new work after the first failure, but in-flight workers are still awaited with `Promise.allSettled()`. That matches a partial-index outcome when one embedding request fails transiently mid-run.
  - file: [src/utils/run-with-concurrency.ts:21](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/utils/run-with-concurrency.ts#L21)

- **Tests currently encode the same status-only assumption:** The existing status-state test asserts that status-only managers are dirty only when no prior index metadata exists; there is no current test covering partial indexed counts with `Dirty: no`.
