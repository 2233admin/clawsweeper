## Summary

Keep open. Current `main` still only refreshes gateway service metadata during `openclaw update` when the service is actively loaded. An installed-but-not-loaded macOS LaunchAgent can therefore keep stale plist `ProgramArguments` across package updates, and the later start/restart recovery path only re-bootstraps the existing plist instead of reinstalling it.

## Evidence

- **Update has a dedicated LaunchAgent refresh step:** `refreshGatewayServiceEnv()` explicitly reruns `gateway install --force`, so this is the intended path for rewriting stale service metadata after update.
  - file: [src/cli/update-cli/update-command.ts:256](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/cli/update-cli/update-command.ts#L256)
  - command: `nl -ba src/cli/update-cli/update-command.ts | sed -n '256,310p'`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Refresh is still gated on `isLoaded()`:** `openclaw update` sets `refreshGatewayServiceEnv = true` only inside `if (loaded)`, so an installed plist that is present but not currently loaded is skipped entirely.
  - file: [src/cli/update-cli/update-command.ts:1395](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/cli/update-cli/update-command.ts#L1395)
  - command: `nl -ba src/cli/update-cli/update-command.ts | sed -n '1390,1410p'`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Start/restart recovery does not reinstall the plist:** When the service is not loaded on macOS, `runDaemonStart`/`runDaemonRestart` call `recoverInstalledLaunchAgent()`, and that helper only checks for plist existence and runs `repairLaunchAgentBootstrap()`; it does not rewrite `ProgramArguments`.
  - file: [src/cli/daemon-cli/lifecycle.ts:152](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/cli/daemon-cli/lifecycle.ts#L152)

- **Reinstall would prefer the new entrypoint order:** Gateway install entrypoint resolution still prefers `dist/index.js` before legacy `dist/entry.js`, so skipping reinstall can preserve the exact stale entrypoint mismatch reported in the issue.
  - file: [src/daemon/gateway-entrypoint.ts:4](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/daemon/gateway-entrypoint.ts#L4)
