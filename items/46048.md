## Close Comment

Thanks for the proposal. I reviewed this against current `main` as a Codex maintainer pass, and this looks better scoped as an external OpenClaw plugin or ClawHub skill rather than a core change.

The current project vision is explicit here: `VISION.md:54` says optional capability should usually ship as plugins, `VISION.md:58` says plugin authors should host and maintain those plugins in their own repositories, `VISION.md:69` says new skills should go to ClawHub first, and `VISION.md:101` / `VISION.md:106` / `VISION.md:107` exclude new core skills that can live on ClawHub plus agent-hierarchy / heavy orchestration layers in core. The plugin docs also already document the external path: `docs/plugins/building-plugins.md:11` and `docs/plugins/building-plugins.md:16` say plugins can add agent tools and other capabilities and do not need to be added to the OpenClaw repo.

Given that scope, I’m closing this as `clawhub` rather than a core OpenClaw issue. If you want to pursue `workflow-as-list`, the evidence on current `main` points to shipping it as a standalone plugin/skill package and distributing it through ClawHub or npm.
