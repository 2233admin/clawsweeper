## Summary

Keep open. Current main can send Telegram media when outbound delivery already has a `mediaUrl`, but the read-tool bridge still only forwards media when the tool result includes `details.media`, `MEDIA:` text, or `details.path`. The OpenClaw read-tool wrapper does not add that path metadata, and the current tests still document the no-path image case as producing no outbound media artifact.

## Evidence

- **Telegram outbound send supports media when given a media URL/path:** `telegramOutbound.sendMedia` forwards `mediaUrl` into `sendMessageTelegram`, and `sendMessageTelegram` loads the media and calls Telegram media APIs such as `sendPhoto`/`sendDocument`. This shows the channel can send images once the outbound payload has a media path.
  - file: [extensions/telegram/src/outbound-adapter.ts:176](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/extensions/telegram/src/outbound-adapter.ts#L176)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Telegram sender uses mediaUrl to send photos/documents:** When `mediaUrl` is present, `sendMessageTelegram` loads the media, classifies it, and sends via `api.sendPhoto` for images or `api.sendDocument` otherwise.
  - file: [extensions/telegram/src/send.ts:814](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/extensions/telegram/src/send.ts#L814)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Tool-result media extraction still requires explicit media metadata/path:** `extractToolResultMediaArtifact()` only returns outbound media from `details.media`, `MEDIA:` lines in text blocks, or a fallback `details.path` when image content exists. There is no fallback from the read tool call arguments alone.
  - file: [src/agents/pi-embedded-subscribe.tools.ts:282](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/agents/pi-embedded-subscribe.tools.ts#L282)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Read-tool wrapper does not add outbound media path metadata:** `createOpenClawReadTool()` runs adaptive paging, image MIME normalization, and image sanitization, then returns the result. In this wrapper there is no code adding `details.path`, `details.media.mediaUrl`, or `MEDIA:` lines for image reads.
  - file: [src/agents/pi-tools.read.ts:674](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/agents/pi-tools.read.ts#L674)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)

That matches the reported symptom: image metadata exists, but nothing is queued for channel delivery.
  - file: [src/agents/pi-embedded-subscribe.tools.media.test.ts:111](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/agents/pi-embedded-subscribe.tools.media.test.ts#L111)
