## Summary

Keep #68977 open. Current main still exposes `enqueueSystemEvent` as an ephemeral in-memory queue with a boolean return and only consecutive duplicate-text suppression. It does not accept a scoped `idempotencyKey`, return `EnqueueResult`, or durably persist dedupe keys atomically with enqueue. Related dedupe primitives exist, but they are separate APIs and do not solve this requested core/plugin-runtime contract.

## Best Possible Solution

Keep the issue open for maintainer/API design. The likely path is an additive change around `src/infra/system-events.ts`, plugin runtime types, and plugin docs: accept a scoped `idempotencyKey`, return a structured enqueue result, persist the key with the queued event at least until consumption, and add regression tests for same-session duplicates, same key across different sessions, payload mismatch behavior, restart durability, and cleanup/TTL semantics. Existing `persistent-dedupe` or session-delivery queue code may be useful implementation material, but the public `enqueueSystemEvent` contract still needs the actual core integration.
