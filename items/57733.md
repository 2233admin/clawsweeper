## Summary

Keep open. Current `main` does not implement configurable non-batch memory embedding concurrency, and the PR's documented config is still absent from the strict runtime schema, so there is no strong basis to close this as already implemented or obsolete.

## Evidence

- **Non-batch indexing still uses the hardcoded default:** `getIndexConcurrency()` on current `main` still returns batch concurrency when enabled, otherwise the fixed `EMBEDDING_INDEX_CONCURRENCY` value. There is no provider-specific or config-driven non-batch override.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:466](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/memory-core/src/memory/manager-embedding-ops.ts#L466)
  - command: `nl -ba extensions/memory-core/src/memory/manager-embedding-ops.ts | sed -n '462,472p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Resolved memory config has no `nonBatchConcurrency` field:** `ResolvedMemorySearchConfig.remote` only contains `baseUrl`, `apiKey`, `headers`, and `batch`; merge logic also only propagates those remote fields.
  - file: [src/agents/memory-search.ts:21](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/memory-search.ts#L21)

  - file: [docs/reference/memory-config.md:337](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/docs/reference/memory-config.md#L337)

## Risks / Open Questions

- Users on slower non-batch embedding providers such as Ollama still have no supported way on `main` to reduce non-batch indexing concurrency.
- The PR as submitted appears incomplete because the runtime schema omission would make the documented config key unusable even if the other code changes landed unchanged.
