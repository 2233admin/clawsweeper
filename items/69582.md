## Summary

Keep open. Current main has adjacent safeguards for malformed tool-call streams and optional repetitive-call detection, but it does not yet address this issue's specific value-level `{}` contamination or the `process(action={})` path that returns the same `sessionId is required for this action` failure.

## Best Possible Solution

Keep the issue open and fix it in the core agent/tool boundary. A conservative path is to add runtime validation for `process.action` against allowed actions, return a terminal/actionable invalid-parameter error for wrong types, and add a focused provider/tool-argument repair or rejection path for GLM/OpenAI-compatible `{}` suffix contamination in string fields. Add regression tests for `process({ action: {} })` and edit parameters such as `path: "file.md{}"` and `newText: "value{}"`, plus a loop-bounding assertion so the model cannot keep retrying the same malformed call indefinitely.

## Evidence

- **Issue report is specific and recent:** The provided issue context reports OpenClaw 2026.3.24 on Windows 11 with local/glm-5, including concrete malformed parameters such as `edit.newText` ending in `{}`, `edit.path` ending in `{}`, and `process(action={})`; the reporter also commented that multiple March/April versions reproduced it.
- **Process runtime still accepts the bad shape until it falls through:** `createProcessTool` casts `args` to the expected shape without runtime type validation.
