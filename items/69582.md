## Summary

Keep #69582 open. Current main has related guardrails, but they do not fully address the reported bug: `process(action={})` still reaches the process tool as a cast object and returns the same `sessionId is required for this action` failure path, trailing `{}` inside valid string arguments is not cleaned, and general tool-loop detection remains opt-in rather than default protection.

## Best Possible Solution

Keep the issue open and fix it in the agent/tool boundary. A conservative path is to add runtime argument validation for `process.action` before sessionId handling, return an explicit wrong-type/wrong-action error with retry guidance, and add regression coverage for `{ action: {} }` plus GLM/OpenAI-compatible argument contamination. If maintainers choose to sanitize trailing `{}` in string fields, scope it narrowly to provider/tool-call argument repair with tests proving it cannot silently change legitimate file contents or command text.

## Evidence

- **Reporter context describes a concrete GLM/tool-argument corruption pattern:** The issue body reports OpenClaw 2026.3.24 on Windows 11 with local/glm-5, including `edit` string values ending in `{}` and `process` receiving `action: {}`; the follow-up comment says multiple March/April versions were tried with the same problem.
