## Summary

Keep open. Current `main` does not implement the PR’s configless-Ollama auth behavior, and current Ollama plugin/docs still require either env/profile auth or a meaningful `models.providers.ollama` config before synthetic auth is minted.

## Evidence

- **Core auth path still rejects providers with no config-backed synthetic auth:** `resolveSyntheticLocalProviderAuth()` still returns `null` immediately when `resolveProviderConfig()` is absent, and `resolveApiKeyForProvider()` later throws `No API key found for provider ...` when no other auth source exists.
  - file: [src/agents/model-auth.ts:309](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/model-auth.ts#L309)
  - command: `nl -ba src/agents/model-auth.ts | sed -n '297,345p' && nl -ba src/agents/model-auth.ts | sed -n '559,601p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Ollama synthetic auth is only minted for meaningful explicit provider config:** The bundled Ollama plugin’s `resolveSyntheticAuth` hook returns auth only when `hasMeaningfulExplicitOllamaConfig(providerConfig)` is true; otherwise it returns `undefined`. That does not cover `cfg: {}` / direct `models.primary` only.
