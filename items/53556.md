## Summary

Current main still only partially covers the requested Feishu backward-compatibility cases. Legacy `groupPolicy: "allowall"` and `topicSessionMode` are supported, but the reported `botId` to `appId` path remains missing from schema, runtime credential resolution, setup/status detection, doctor migration hooks, docs, and tests.

## Best Possible Solution

Keep this open and scope the fix to Feishu-owned compatibility: add explicit `botId` to `appId` handling for top-level and account configs, preferably through Feishu `normalizeCompatibilityConfig`/`legacyConfigRules` or an equivalent schema-normalization seam; preserve `appId` precedence when both fields exist; update plugin metadata/setup status/credential resolution as needed; add focused schema, account, and doctor tests; and document the upgrade path on https://docs.openclaw.ai/channels/feishu.
