## Summary

Keep open. Current `main` still only applies temporal decay to top-level `memory/YYYY-MM-DD.md` files, so nested dated files under `memory/` are not covered; an unmerged PR already targets this exact bug.

## Evidence

- **Current matcher is flat-path only:** `DATED_MEMORY_PATH_RE` is still `/(?:^|¥/)memory¥/(¥d{4})-(¥d{2})-(¥d{2})¥.md$/`, and both `parseMemoryDateFromPath` and `isEvergreenMemoryPath` depend on it. That means nested files like `memory/daily-log/2026-04-10.md` do not parse as dated and fall through the evergreen branch.
  - file: [extensions/memory-core/src/memory/temporal-decay.ts:15](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/extensions/memory-core/src/memory/temporal-decay.ts#L15)
  - command: `nl -ba extensions/memory-core/src/memory/temporal-decay.ts | sed -n '1,170p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Direct reproduction against current regex:** A read-only `node -e` check on `main` returns `true` for `memory/2026-04-16.md` but `false` for `memory/daily-log/2026-04-10.md` and `memory/session-log/2026-04-15-1200.md`. The reported path shapes still do not match current code.
