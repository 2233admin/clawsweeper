## Summary

Keep open. Current main has partial cleanup for some session-switch paths, but the specific stale active compaction state described in #64220 is still plausible: a `compaction` start sets `compactionStatus.phase = "active"`, the UI renders active/retrying status indefinitely, and terminal chat/reconnect/clear paths still clear run/stream state without consistently clearing `compactionStatus` and `fallbackStatus`. Related #39908 is similar but does not supersede this more specific Control UI state-machine issue.

## Best Possible Solution

Keep #64220 open and fix it in the Control UI state layer. The best path is to add a run/session-scoped stale timeout for active compaction, centralize clearing of compaction/fallback status plus timers in chat-state cleanup, invoke that cleanup on terminal chat events, reconnect/disconnect, clear/reset/send-failure paths, and add regression tests for a missed compaction end after a fully rendered reply plus reconnect/session-switch cleanup.
