## Summary

Keep open. Current `main` still routes plugin CLI command registration through a non-activating load, and the wiki bridge/status paths still read memory public artifacts only from process-global memory capability state. On a fresh CLI process that leaves `memoryPluginState.capability` unset, which matches the reported `0 artifacts` behavior.

## Evidence

- **CLI command registry uses non-activating plugin loads:** Plugin CLI commands are registered from `loadPluginCliRegistrationEntriesWithDefaults(...)`, which calls `loadPluginCliCommandRegistryWithContext(...)`; that loader hard-codes `activate: false` and `cache: false` for CLI command registration.
  - file: [src/plugins/cli-registry-loader.ts:107](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/plugins/cli-registry-loader.ts#L107)
  - command: `nl -ba src/plugins/cli.ts | sed -n '40,70p'; nl -ba src/plugins/cli-registry-loader.ts | sed -n '107,126p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Non-activating loads restore the previous memory capability instead of keeping the plugin's new registration:** During `loadOpenClawPlugins(...)`, snapshot/non-activating loads capture `previousMemoryCapability` and immediately restore it after plugin registration. In a fresh CLI process that previous value is `undefined`, so memory capability registration from `memory-core` does not persist.
  - file: [src/plugins/loader.ts:2873](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/plugins/loader.ts#L2873)
  - command: `nl -ba src/plugins/loader.ts | sed -n '2868,2918p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Artifact listing falls back to empty when no memory capability is registered:** `listActiveMemoryPublicArtifacts(...)` returns `[]` whenever `memoryPluginState.capability?.capability.publicArtifacts` is missing.
  - file: [src/plugins/memory-state.ts:274](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/plugins/memory-state.ts#L274)
  - command: `nl -ba src/plugins/memory-state.ts | sed -n '274,279p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **memory-wiki bridge and status both depend on that global artifact accessor:** Bridge import calls `listActiveMemoryPublicArtifacts({ cfg: params.appConfig })`, and status computes `bridgePublicArtifactCount` from the same accessor. If capability state is unset in the CLI process, both surfaces report zero artifacts.
  - file: [extensions/memory-wiki/src/bridge.ts:224](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/memory-wiki/src/bridge.ts#L224)
  - command: `nl -ba extensions/memory-wiki/src/bridge.ts | sed -n '202,228p'; nl -ba extensions/memory-wiki/src/status.ts | sed -n '217,227p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **memory-core does export public artifacts on normal activation:** `memory-core` registers `publicArtifacts.listArtifacts = listMemoryCorePublicArtifacts`, so the reported failure is not explained by missing export code in the plugin itself.
  - file: [extensions/memory-core/index.ts:33](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/memory-core/index.ts#L33)
  - command: `nl -ba extensions/memory-core/index.ts | sed -n '24,40p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Existing regression tests only cover preserving an already-registered capability:** The loader tests added for this area first do an activating load, then verify that a later `activate:false` load preserves that existing capability. They do not cover a fresh CLI process where the first relevant load is already `activate:false`.
  - file: [src/plugins/loader.test.ts:3364](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/plugins/loader.test.ts#L3364)
  - command: `nl -ba src/plugins/loader.test.ts | sed -n '3364,3443p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Recent discussion still reports the bug on newer builds:** The issue thread includes multiple confirmations after the original report, including reproduction on 2026.4.15 and another confirmation on 2026.4.21 on 2026-04-24, all describing the same silent `0 artifacts` bridge result with `unsafe-local` as the workaround.

## Risks / Open Questions

- If some other startup path activates the memory plugin before wiki CLI registration in a specific environment, the impact could be narrower than the code path suggests, but the fresh-process CLI path still appears unresolved.
