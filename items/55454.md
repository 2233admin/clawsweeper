## Summary

Current `main` already handles Windows runtime-overlay symlink permission failures by falling back to copying into `dist-runtime`, and the same implementation is present in shipped release `v2026.4.22` with a focused regression test.

## Evidence

- **Windows symlink fallback in current main:** `scripts/stage-bundled-plugin-runtime.mjs:15` defines a Windows-only fallback for `EPERM`/`EINVAL`/`UNKNOWN`, and `scripts/stage-bundled-plugin-runtime.mjs:33` applies that fallback around `fs.symlinkSync(...)` so denied runtime-overlay links are copied instead of aborting the build.
  - file: [scripts/stage-bundled-plugin-runtime.mjs:15](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/scripts/stage-bundled-plugin-runtime.mjs#L15)

## Close Comment

Closing from Codex review: current `main` already implements the Windows symlink-permission fallback this report asks for. `scripts/stage-bundled-plugin-runtime.mjs:15` and `scripts/stage-bundled-plugin-runtime.mjs:33` now catch Windows `EPERM`/`EINVAL`/`UNKNOWN` from runtime-overlay symlink creation and fall back to copying into `dist-runtime` instead of failing the build, and `test/scripts/stage-bundled-plugin-runtime.test.ts:21` covers that exact regression by simulating a Windows `EPERM` and asserting the file is copied. I also checked the shipped `v2026.4.22` release commit (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and it already contains the same logic, so this looks fixed in current releases as well.
