## Summary

Keep open. Current main authenticates `trusted-proxy` users for gateway access, but the WebSocket/webchat path does not persist that authenticated user onto the connection or inject it into `chat.send` message context. The agent still receives gateway client identity (`openclaw-control-ui` / `webchat-ui`) rather than the proxy-authenticated human identity, so the reported gap appears real on `main` at `af56926e2fc410809fcb0399435787c7fa8daef8`.

## Evidence

- **Trusted-proxy auth does resolve a user header:** `authorizeTrustedProxy` reads the configured `trustedProxy.userHeader`, trims it, and uses it for allowlist checks, so the authenticated upstream user is available during gateway auth.
  - file: [src/gateway/auth.ts:263](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/gateway/auth.ts#L263)
  - command: `nl -ba src/gateway/auth.ts | sed -n '263,312p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Trusted user is not stored on the WebSocket client record:** `GatewayWsClient` has no field for a trusted-proxy user, and the connect handler populates `nextClient` with `connect`, auth/session-generation flags, presence, and canvas data only.
  - file: [src/gateway/server/ws-types.ts:4](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/gateway/server/ws-types.ts#L4)
  - command: `nl -ba src/gateway/server/ws-types.ts | sed -n '1,40p'; nl -ba src/gateway/server/ws-connection/message-handler.ts | sed -n '1290,1352p'`

- **chat.send uses gateway client identity, not proxy-authenticated user identity:** `chat.send` builds `MsgContext` with `SenderId`, `SenderName`, and `SenderUsername` from `client.connect.client.id/displayName`.
