## Summary

Keep #70729 open. Current main still authenticates trusted-proxy users and exposes the parsed user in the auth result, but the identity is not persisted on the WebSocket client state and `chat.send` builds agent message sender fields from static Control UI client metadata instead. The related #23585 fallback/default-user item is separate because #70729 has an identity header present and asks for propagation to webchat agent context.

## Best Possible Solution

Keep the issue open. Implement a core gateway fix that persists the trusted-proxy authenticated user/source on the WebSocket client after handshake, threads it into webchat-originated `chat.send` and session initial-send context, documents the sender/authenticated-user contract, and adds a regression test for trusted-proxy webchat message context. Treat #23585 as a separate missing-header fallback feature, not a duplicate.
