## Summary

Keep #70729 open. Current main still authenticates trusted-proxy users but does not retain that identity on the WebSocket client or inject it into webchat `chat.send` message context. The related #23585 fallback/default-user item is separate: this report has an identity header present and asks for propagation to agent context.

## Best Possible Solution

Keep the issue open. Implement a core gateway fix that persists the trusted-proxy authenticated user/source on the WebSocket client after handshake, threads it into webchat-originated `chat.send` and `sessions.create` initial-send context, documents whether it maps to `SenderId`/`SenderName` or a distinct authenticated-user field, and adds a regression test for trusted-proxy webchat message context.
