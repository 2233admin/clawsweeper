## Summary

Keep open. Current main still only partially addresses deleted-agent state: agent deletion prunes config, removes core session-store entries, and trashes agent/workspace/transcript directories, but it does not remove or disable cron jobs targeting the deleted agent and does not run per-session/plugin/context-engine cleanup for the purged sessions. There is also still no first-class `agents reset` or clean-state command matching the issue's requested repair path.

## Best Possible Solution

Keep this issue open. The best fix is a scoped cleanup/repair path: extend `agents delete` and `agents.delete`, or add an `agents reset --clean-state` command, so deleted-agent cleanup purges core session-store/transcript state through the existing session mutation lifecycle, removes/disables/reports cron jobs targeting the agent, and gives context-engine/plugins a versioned lifecycle hook to clean session-owned persistence. Add regression coverage for CLI deletion, Gateway deletion, cron jobs targeting a deleted agent, and plugin/context-engine cleanup behavior.

## Risks / Open Questions

- The exact reported `TypeError: Cannot read properties of undefined (reading 'length')` was not reproduced in this read-only review; a fix should add a logs-backed fixture or focused reproduction before closing.
- Cron cleanup needs product judgment: deleting an agent could remove matching jobs, disable them, or fail/report actionable cleanup.
