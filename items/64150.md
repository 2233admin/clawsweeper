## Summary

Keep this PR open. Current main still has the OAuth Claude Code tool-name gap and still does not register `find`/`grep`/`ls`, so the reported bug is real. The PR is not ready as-is because it appends upstream filesystem-discovery tools without OpenClaw workspace/sandbox guards, policy catalog integration, or regression tests; on current main it also uses `findTool`/`grepTool`/`lsTool` names that the pinned `@mariozechner/pi-coding-agent@0.70.2` package does not export.

## Best Possible Solution

Keep this PR open as a real but incomplete implementation candidate. The best path is a focused revision or successor that adds guarded OpenClaw wrappers around `createFindTool`, `createGrepTool`, and `createLsTool`, routes them through sandbox filesystem operations and `tools.fs.workspaceOnly`, adds them to the core tool catalog/profile/group docs, handles any `fd`/`rg` dependency policy explicitly, and adds regression tests for Anthropic OAuth `Glob` -> `find`, `Grep` -> `grep`, `LS` -> `ls`, plus `group:fs` allow/deny behavior.

## Risks / Open Questions

- Merging the PR as submitted could expose `find`/`grep`/`ls` outside the `tools.fs.workspaceOnly` and sandbox bridge paths used by existing OpenClaw filesystem tools.
- Merging the PR as submitted could make `group:fs`, profiles, allowlists, denylists, and docs inaccurate because the new filesystem-reading tools are not in the core tool catalog.
- The PR introduces a runtime path where missing `fd`/`rg` binaries may be downloaded from latest GitHub releases and executed, without a pinned asset/version or OpenClaw policy review.
- The patch shape appears stale against current main because the pinned package exports `createFindTool`/`createGrepTool`/`createLsTool`, not the `findTool`/`grepTool`/`lsTool` symbols used in the PR.
