## Summary

Keep this PR open. Current main still appears to have the Anthropic OAuth Claude Code tool-name gap and still does not register local `find`, `grep`, or `ls` tools. The PR addresses a real failure mode, and Greptile's comment correctly identifies the intended `Glob` -> `find` mapping, but the submitted patch is not merge-ready because it is stale against current main, uses dependency symbols that are not exported by the pinned package, and would add filesystem discovery tools without OpenClaw's workspace/sandbox policy integration.

## Best Possible Solution

Keep this PR open as a real but incomplete implementation candidate. The best path is a focused revision or successor that uses `createFindTool`, `createGrepTool`, and `createLsTool`, wraps filesystem access through the same workspace/sandbox policy surfaces as `read`/`write`/`edit`, updates `tool-catalog` and the public tools docs, explicitly handles the `fd`/`rg` executable acquisition/execution policy, and adds regression tests for Anthropic OAuth `Glob` -> `find`, `Grep` -> `grep`, `LS` -> `ls`, plus `group:fs` allow/deny behavior.

## Risks / Open Questions

- Current main likely still fails Anthropic OAuth sessions when Claude returns `Glob` and no registered local `find` tool exists.
- Merging the PR as submitted could expose filesystem discovery outside the guarded `tools.fs.workspaceOnly` and sandbox filesystem bridge paths used by existing OpenClaw filesystem tools.
- Merging the PR as submitted could make `group:fs`, profile allowlists/denylists, and public docs inaccurate because the new filesystem-reading tools are not added to the core catalog.
- The PR is stale against current main and the pinned dependency API, because it imports `findTool`, `grepTool`, and `lsTool` values that are not exported by `@mariozechner/pi-coding-agent@0.70.2`.
- The upstream discovery tools can download and execute `fd`/`rg` through the dependency's tool manager path; OpenClaw should make an explicit policy decision before exposing that path in core.
