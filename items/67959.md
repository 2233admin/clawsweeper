## Summary

Keep open. Current `main` already parallelizes gateway-side channel status work, but the CLI still injects a `10000` ms default for `openclaw channels status --probe`, which prevents the intended `30000` ms probe default from taking effect. The PR still carries a real user-facing fix plus missing regression coverage.

## Evidence

- **Gateway-side parallelization is already on main:** `channels.status` now schedules both per-account and per-plugin work through `runTasksWithConcurrency`, so the probe parallelization portion of the PR is largely present on current `main`.
  - file: [src/gateway/server-methods/channels.ts:270](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/server-methods/channels.ts#L270)

- **Command layer expects a 30s default for probe mode:** `channelsStatusCommand` uses `opts.timeout ?? (opts.probe ? 30_000 : 10_000)`, which would give probe mode a 30s default only if the CLI leaves `opts.timeout` unset.
  - file: [src/commands/channels/status.ts:162](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/commands/channels/status.ts#L162)

That leaves the current CLI/default mismatch unguarded.
