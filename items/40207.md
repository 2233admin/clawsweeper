## Summary

Current main still writes the CLI-managed macOS Gateway LaunchAgent stdout/stderr paths under the OpenClaw state directory, so a symlinked `~/.openclaw` can still be regenerated into `StandardOutPath` and `StandardErrorPath`. The issue has concrete macOS launchd exit-78 confirmations and remains an actionable core daemon/CLI bug, not a stale support request.

## Best Possible Solution

Keep this issue open and fix the CLI-managed macOS LaunchAgent path in `src/daemon`: write `StandardOutPath` and `StandardErrorPath` to a launchd-safe non-symlinked user log location such as `~/Library/Logs/openclaw`, or resolve the generated log directory with `realpath` before writing the plist. Add regression coverage for symlinked state directories and plist regeneration, update runtime/status/docs hints, and consider an install-time health check that reports launchd EX_CONFIG failures clearly.
