Keep open. Current main has useful generic diagnostics, context/memory docs, and several startup/performance fixes, but it still does not ship the requested systematic native-VPS reasoning-latency analysis or instrumentation plan with measured or estimated contribution by layer.

## Best Possible Solution

Keep this open and scope it into a bounded native-Linux gateway performance investigation. Capture one trace/runbook covering bootstrap/context assembly timing, prompt/token estimation, memory query embedding/search and Active Memory recall, compaction trigger/model-call duration, queue waits, subagent spawn and announce-back, Telegram preview/final delivery, hook execution, auth fallback decisions, and process RSS/heap/GC. Then publish quick-win config guidance in the Linux server/performance docs and split narrower optimization issues for measured hot spots.
