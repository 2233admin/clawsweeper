## Summary

Keep open. Current main still exposes gateway health as a service/channel/session snapshot, and there is no connection-level `connected`, `rttMs`, `lastHeartbeatAt`, or `gateway.ping` surface. The existing tick keepalive/watchdog proves the transport has a heartbeat mechanism, but that state remains private to clients and is not returned by the `health` RPC. The related provided reports are about channel health/deaf sockets and WhatsApp status mismatches, not a canonical tracker for this gateway connection-level API request.

## Best Possible Solution

Keep the issue open. The best implementation would be an additive gateway core change: either add an optional connection/transport section to `health` or introduce a lightweight `gateway.ping` method that returns `connected`, measured `rttMs`, and a timestamp derived from the active WS heartbeat/request path. Update the gateway protocol/types, CLI/UI health types, docs, and focused tests around the health handler/client keepalive behavior.
