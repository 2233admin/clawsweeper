## Summary

Keep open. Current `main` does not appear to have a general fix for Gemini LLM requests honoring plain `HTTP_PROXY`/`HTTPS_PROXY` env vars by default, and I did not find evidence that the CLI/WebUI silent-failure report is obsolete. The code path still requires an explicit model-provider `request.proxy` override to enter env-proxy mode.

## Evidence

- **Gemini chat transport still uses the shared guarded model fetch path:** Google's streaming LLM transport builds its request with `buildGuardedModelFetch(model)` and sends the Gemini request through that shared provider transport layer, so the default model-fetch policy here is the relevant path for the issue.
  - file: [extensions/google/transport-stream.ts:638](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/extensions/google/transport-stream.ts#L638)

  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Default model transport policy does not auto-enable env proxy:** `resolveModelRequestPolicy` only merges per-model/provider request overrides plus the debug proxy override. `buildGuardedModelFetch` then passes the resulting dispatcher policy into `fetchWithSsrFGuard`; there is no automatic `HTTP_PROXY`/`HTTPS_PROXY` opt-in for ordinary LLM requests.
  - file: [src/agents/provider-transport-fetch.ts:126](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/agents/provider-transport-fetch.ts#L126)
