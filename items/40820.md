## Summary

Keep open. Current main does not implement the Telegram `/settings` inline keyboard panel, the `cfg_*` callback path, or the provider-restricted command registry seam from the PR. The PR still addresses a real Telegram UX gap, and the review thread shows the author iterated on concrete auth issues rather than abandoning low-signal code.

## Best Possible Solution

Keep the PR open, but require a rebase into the current `extensions/telegram` plugin layout. The best path is to preserve the PR's shared settings auth design, implement the `/settings` command and `cfg_*` callbacks in the current Telegram plugin code, add colocated tests under `extensions/telegram/src`, and keep docs aligned with the shipped Telegram config-write behavior. If current plugin command/interactive APIs are not sufficient for this safely, add a narrow documented SDK seam rather than closing the user-facing request.
