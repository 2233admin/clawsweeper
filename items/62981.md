## Summary

Current main only partially mitigates the report by delaying lock acquisition until after cold bootstrap/plugin/tool setup. The CLI still falls back to embedded execution with the original session selectors after any Gateway timeout, while the accepted Gateway run can keep running and hold the same session write lock through the active attempt.

## Best Possible Solution

Keep this issue open as the focused tracker for Gateway-timeout-to-embedded-fallback session lock contention. Fix `src/commands/agent-via-gateway.ts` so a timed-out accepted Gateway run is handled via `agent.wait`, abort-and-wait, or an explicitly fresh user-visible session before embedded execution. Add regression coverage in `src/commands/agent-via-gateway.test.ts` for an accepted Gateway run timing out before fallback, and coordinate remaining broader dispatch-timeout work with #71605.
