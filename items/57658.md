## Summary

Keep open. Current main already supports durable scope-limited device/operator access, so the issue's reconnect-overwrites-scopes rationale is partly obsolete, but the titled feature is still not implemented: gateway roles are still hardcoded to `operator`/`node`, unknown roles are rejected at handshake, and `GatewayConfig` has no custom-role definition surface.

## Evidence

- **Gateway roles are hardcoded:** `GATEWAY_ROLES` only includes `operator` and `node`, and `parseGatewayRole` rejects anything else.
  - file: [src/gateway/role-policy.ts:3](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/role-policy.ts#L3)

- **Unknown roles are rejected on connect:** The WS handshake parses `connectParams.role`; if it is not a known gateway role, the server returns `invalid role` and closes the connection.
  - file: [src/gateway/server/ws-connection/message-handler.ts:453](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/server/ws-connection/message-handler.ts#L453)
