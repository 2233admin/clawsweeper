## Summary

Current `main` still appears to have the reported stale-lineage path. The requester depth for `sessions_spawn` is derived from stored `spawnDepth`/`spawnedBy`, and normal `agent` runs still write preserved lineage back into the same session entry without clearing it for top-level sessions. The provided timeline also shows the proposed fix in PR #67946 is still open, so this issue should remain open.

## Evidence

- **Normal agent runs persist existing lineage back into the session entry:** When `agent` handles an existing `requestedSessionKey`, it canonicalizes `entry.spawnedBy`, then writes `spawnedBy` and `spawnDepth` straight back into `nextEntryPatch` before merging the entry. There is no top-level-session branch here that clears stale lineage first.
  - file: [src/gateway/server-methods/agent.ts:634](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/server-methods/agent.ts#L634)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Merge semantics preserve omitted subagent metadata:** `mergeSessionEntryWithPolicy` builds `next` as `{ ...existing, ...patch, ... }`, so fields omitted from the patch, such as `subagentRole` and `subagentControlScope`, remain on the entry during ordinary sends.
  - file: [src/config/sessions/types.ts:376](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/config/sessions/types.ts#L376)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Stored lineage drives computed requester depth:** `getSubagentDepthFromSessionStore` accepts any session key, returns stored `spawnDepth` when present, and otherwise recursively follows `spawnedBy`. It does not first require the session key to be an actual `subagent:*` session.
  - file: [src/agents/subagent-depth.ts:117](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/subagent-depth.ts#L117)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **sessions_spawn rejects based on that stored depth:** `spawnSubagentDirect` computes `callerDepth` via `getSubagentDepthFromSessionStore(requesterInternalKey, { cfg })` and rejects when `callerDepth >= maxSpawnDepth`, producing the same depth-limit error described in the report.
  - file: [src/agents/subagent-spawn.ts:605](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/subagent-spawn.ts#L605)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Existing tests still allow subagent metadata on a main-session row:** Gateway agent tests populate `agent:main:main` with `spawnedBy`, `spawnDepth`, `subagentRole`, and `subagentControlScope`, then assert those fields are broadcast back on `sessions.changed`. That indicates current test coverage does not yet enforce clearing stale lineage from top-level runs.
  - file: [src/gateway/server-methods/agent.test.ts:616](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/server-methods/agent.test.ts#L616)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
