## Summary

Keep this issue open. Current main still lacks first-class JSON Schema input, validation, repair/retry, and machine-readable schema failure semantics on the requested `openclaw infer model run` and `openclaw agent` CLI surfaces. The bundled `llm-task` plugin is useful prior art, but it is an optional tool path and throws on invalid JSON/schema rather than defining the requested core CLI contract.

## Best Possible Solution

Keep this issue open. The best maintainer path is to implement a narrow provider-neutral JSON Schema contract on `openclaw infer model run` first, likely `--json-schema <path-or-inline>`, parse and validate final JSON with a shared validator, define bounded repair/retry behavior, return structured validation and repair-exhausted errors in `--json` mode, and add an additive gateway protocol field with matching tests and docs. After that lands, maintainers can decide whether `openclaw agent` should expose the same contract or whether workflow-only use cases should stay with plugin tools like `llm-task`.

## Risks / Open Questions

- A real fix needs an additive gateway protocol field so local and gateway transports can share one schema contract instead of silently diverging.
- Repair/retry semantics need to be explicitly bounded and reflected in `--json` failures; simply validating and throwing would not satisfy the requested machine-readable repair-exhausted behavior.
- Provider-native schema modes can be used opportunistically, but OpenClaw still needs a provider-neutral validation and error contract so behavior does not vary by backend.
