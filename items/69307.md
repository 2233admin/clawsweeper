## Summary

Keep #69307 open. Current main still lacks a narrow modifying hook between the natural assistant reply and TTS directive parsing/synthesis. Existing support covers static TTS configuration, per-agent/channel/account overrides, chat-scoped auto on/off, model-output observation, and post-TTS delivery rewriting, but not the requested post-LLM/pre-TTS text transform or per-message skip decision.

## Best Possible Solution

Keep the issue open as an additive Plugin SDK design request. The conservative implementation path is a narrow modifying hook such as `before_tts` or a generalized reply-payload transform, registered in `src/plugins/hook-types.ts` and `src/plugins/hooks.ts`, invoked in `src/auto-reply/reply/dispatch-from-config.ts` immediately before each `maybeApplyTtsToReplyPayload()` call that should be transformable. It should include typed session/channel/account/conversation context, support deterministic text modification and TTS skip behavior, and ship with docs plus tests for directive injection, skip semantics, hook ordering, and conversation-access policy.

## Risks / Open Questions

- Static per-agent/channel/account TTS overrides and peer bindings may satisfy some users' voice-routing needs, so maintainers should decide whether the best product surface is a hook, a declarative per-session voice profile, or both.
- A post-LLM text transform exposes raw assistant content to plugins and should align with the existing conversation-access policy and docs.
- The hook scope needs design: final replies only versus final/tool/block/synthetic payloads, and whether `skip` should bypass only synthesis or the entire TTS cleanup/directive path.
