## Summary

Keep #69307 open. Current `main` still lacks a narrow modifying hook between generated reply text and TTS directive parsing/synthesis. Existing hooks either observe only, run before the model reply exists, claim the entire reply dispatch path, or run after TTS/media preparation.

## Best Possible Solution

Keep the issue open as an additive Plugin SDK design request. The best path is a narrow modifying hook such as `before_tts` or a generalized reply-payload transform, registered in `src/plugins/hook-types.ts` and `src/plugins/hooks.ts`, invoked immediately before each transformable `maybeApplyTtsToReplyPayload()` call in reply dispatch, and documented with typed session/channel/account/conversation context plus tests for directive injection, skip semantics, ordering, and access policy.

## Risks / Open Questions

- A plugin can approximate this with `reply_dispatch` only by taking over the whole dispatch path, which is materially heavier than the requested deterministic text transform and risks bypassing default delivery behavior.
- Maintainers should decide whether the product shape should be a narrow `before_tts` hook, a more general reply-payload transform, declarative per-chat TTS profiles, or a combination.
