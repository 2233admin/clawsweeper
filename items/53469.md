## Summary

Keep this issue open. Current main has partial private-network support for SearXNG web_search and model-provider request transport, and remote image references now inherit the existing web_fetch SSRF policy. The central request is still not implemented for generic web_fetch because its public config/schema/runtime only support the RFC2544 benchmark-range exception, not localhost/LAN/private ranges. Gateway OpenResponses URL inputs also still hard-code private-network denial for input_image and input_file fetches.

## Best Possible Solution

Keep this issue open and scope the remaining work around a narrow explicit private-network opt-in for tools.web.fetch.ssrfPolicy, remote media/image references that inherit that policy, and gateway.http.endpoints.responses.{files,images} URL fetches. Preserve default blocking, add schema/help/docs, decide separately whether gateway.mode=local should auto-enable anything, and cover web_fetch, redirects, Gateway URL inputs, cache behavior, and image/media reference handling with SSRF tests. Treat the shipped SearXNG provider and models.providers.*.request.allowPrivateNetwork support as partial precedent, not full resolution.

## Risks / Open Questions

- Private-network access is SSRF-sensitive; any implementation should remain default-off except for an explicitly reviewed local-mode rule.
- The web_fetch cache key must include any private-network policy so denied and allowed fetches cannot share cached payloads.
- Redirects, DNS pinning, hostname allowlists, Gateway input_file/input_image URL fetching, and remote media references need regression coverage for both denied-by-default and explicit opt-in behavior.
