## Summary

Keep this issue open. Current main still sends Feishu image media by uploading the image and then calling Feishu message create/reply without any stable provider retry key, so the reported duplicate-send path remains plausible and unaddressed.

## Best Possible Solution

Fix this in the Feishu plugin media path. Verify the Feishu/Lark message idempotency contract, thread a stable outbound send retry key through the CLI/gateway/channel outbound context into sendMediaFeishu, apply it to message.create/reply where supported, and add regression coverage that simulates a duplicate retry without blocking intentional later sends.
