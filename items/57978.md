## Summary

Keep #57978 open. Current main still has the Feishu WebSocket park-on-abort implementation: it starts one WSClient and then waits only for abort, with no supervisor loop, no getReconnectInfo polling, and no backoff after the Lark SDK exhausts retries. The PR is a focused fix for open bug #52618 and its latest discussion shows the author addressed several concrete review findings rather than abandoning low-signal work.

## Best Possible Solution

Keep #57978 open for maintainer review. The best product path is to land a vetted Feishu-owned supervisor loop in extensions/feishu/src/monitor.transport.ts with regression tests around SDK retry exhaustion, successful reconnects, abort cleanup, bot identity preservation across restart cycles, and non-recoverable credential errors. If maintainers prefer a different implementation, keep #52618 as the canonical bug and ask the PR author to rebase or narrow the patch rather than closing as obsolete.
