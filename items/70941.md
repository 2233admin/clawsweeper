## Summary

Keep #70941 open. The issue is recent and low-detail, but the title identifies a concrete path: current `openclaw models set` still writes through the shared config writer, and current `main` still rejects sharp config size drops as `CONFIG_WRITE_REJECTED`. I found no comments, timeline events, duplicate, shipped fix, or regression coverage proving this specific `models set` size-drop false positive is fixed or superseded.

## Best Possible Solution

Keep the issue open. A maintainer should reproduce with the reporter's large config shape, then make `openclaw models set` preserve the authored source shape or route the narrow model-primary mutation through a config mutation path that cannot look like a destructive size drop. Add a regression test covering `models set` against the triggering large config while keeping the existing destructive-write guard green.
