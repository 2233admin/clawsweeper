## Summary

Current `main` already fixes the child-logger minLevel inheritance bug, and the fix was shipped in the `2026.4.7` changelog entry for #44646.

## Evidence

- **Child logger now inherits parent threshold:** `src/logging/logger.ts:248-259` sets `minLevel` to `opts?.level ? levelToMinLevel(opts.level) : base.settings.minLevel`, so a child without its own override keeps the parent logger threshold instead of passing `undefined`.
  - file: [src/logging/logger.ts:248](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/logging/logger.ts#L248)

  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Regression tests cover the exact failure mode:** `src/logging/level-filter.test.ts:100-132` asserts that child loggers inherit the parent `minLevel`, do not fall back to `0`, and that pino-style child loggers also propagate the parent threshold.
  - file: [src/logging/level-filter.test.ts:100](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/logging/level-filter.test.ts#L100)

## Close Comment

Closing as implemented on current `main`. A Codex review checked `src/logging/logger.ts:248-259`, where `getChildLogger()` now falls back to `base.settings.minLevel` when no child-specific level is provided, so child loggers no longer wipe out the parent threshold with `undefined`. That applies to the reported cron path as well: `src/gateway/server-cron.ts:152` creates the cron child logger, and the reported `"cron: timer armed"` line in `src/cron/service/timer.ts:670-672` goes through it. The regression is also covered by `src/logging/level-filter.test.ts:100-132`, and the changelog shows this shipped in release `2026.4.7` under `#44646`. If anyone still sees DEBUG lines leaking into file logs on a current build, please open a fresh issue with the exact version and reproduction details.
