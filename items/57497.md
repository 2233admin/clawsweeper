## Summary

Keep #57497 open. Current main still stores session metadata in filesystem-backed `sessions.json` files and conversation transcripts in per-session JSONL files. There is no PostgreSQL/MySQL session backend, no `sessionStore`/`storeType` config, and no plugin API that would let a third-party plugin replace core session persistence today. Related #58550 is an open SQLite PR and #63492 tracks broader external session storage, but neither is a shipped PostgreSQL implementation or a clear canonical duplicate for this narrower request.

## Best Possible Solution

Keep the issue open as a concrete core session-persistence enhancement. The best path is to design a versioned session storage abstraction, keep filesystem JSON/JSONL as the default backend, add config/schema/docs and migration tooling, then implement PostgreSQL or expose a reviewed backend/plugin API that can safely support PostgreSQL without bypassing gateway session semantics.
