## Summary

Keep open. Current `main` still renders code-mode markdown tables by `string.length`, so CJK and emoji grapheme widths are not accounted for. Telegram and Discord still route markdown tables through that path, and the current tests do not cover the reported alignment cases.

## Evidence

- **Code-mode table width is still based on code-unit length:** `src/markdown/ir.ts:509`-`577` computes `width = cell?.text.length ?? 0` and pads with `widths[i] - (cell?.text.length ?? 0)`. That logic is not display-width aware, so wide CJK cells and emoji graphemes will misalign in monospace table output.
  - file: [src/markdown/ir.ts:509](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/markdown/ir.ts#L509)
  - command: `nl -ba src/markdown/ir.ts | sed -n '500,580p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Telegram and Discord still use code-table conversion on main:** `src/config/markdown-tables.ts:88`-`104` defaults markdown tables to `"code"`. `extensions/telegram/src/format.ts:75`-`85` passes `tableMode` into `markdownToIR(...)`, and `extensions/discord/src/send.outbound.ts:148`-`158` calls `convertMarkdownTables(text ?? "", tableMode)` for outbound text.
  - file: [src/config/markdown-tables.ts:88](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/config/markdown-tables.ts#L88)
  - command: `nl -ba src/config/markdown-tables.ts | sed -n '88,105p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Current markdown table tests do not cover CJK or emoji width alignment:** `src/markdown/ir.table-code.test.ts:4`-`89` only checks style-overlap behavior inside code-mode tables. There are no current assertions for CJK display width, emoji grapheme width, or divider alignment in markdown code tables.
  - file: [src/markdown/ir.table-code.test.ts:4](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/markdown/ir.table-code.test.ts#L4)
  - command: `nl -ba src/markdown/ir.table-code.test.ts | sed -n '1,140p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Main already has width-aware logic elsewhere, but markdown tables are not using it:** `src/terminal/ansi.ts:89`-`118` implements grapheme-aware `visibleWidth(...)`, with tests in `src/terminal/ansi.test.ts:24`-`34` and `src/terminal/table.test.ts:144`-`167`. That makes the markdown code-table path stand out as still missing the same width handling.
  - file: [src/terminal/ansi.ts:89](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/terminal/ansi.ts#L89)
