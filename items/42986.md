## Summary

Keep this issue open. Current main still does not implement native Telegram thread-bound subagent session spawning. Telegram has conversation binding infrastructure and ACP topic binding, but the native `sessions_spawn({ mode: "session", thread: true })` path still requires a `subagent_spawning` hook result, and Telegram does not register subagent lifecycle hooks. Related #49073 was correctly closed as a duplicate of this remaining gap, and open PR #42707 appears to be a relevant but not-yet-merged implementation candidate.

## Best Possible Solution

Keep this issue open and implement Telegram native subagent lifecycle hooks analogous to Discord, Matrix, and Feishu. Rebase/review #42707 if it remains the active implementation candidate, or land an equivalent focused change that registers Telegram `subagent_spawning`, `subagent_delivery_target`, and `subagent_ended` hooks, uses the existing Telegram `SessionBindingAdapter` and thread-binding manager, honors `channels.telegram.threadBindings.enabled` and `spawnSubagentSessions`, cleans up bindings on end/failure, resolves completion delivery targets, and updates `docs/tools/subagents.md` plus Telegram tests to distinguish native subagent support from ACP topic binding.

## Risks / Open Questions

- Closing as implemented would conflate Telegram ACP topic binding with native OpenClaw subagent session binding; the requested `sessions_spawn` flow still lacks Telegram hook wiring.
- Telegram config and UI hints expose `channels.telegram.threadBindings.spawnSubagentSessions`, so users may reasonably expect behavior that current native subagent spawning does not provide yet.
- Any fix should be reviewed for routing/security correctness because it will mutate Telegram session bindings, delivery targets, account scoping, and lifecycle cleanup.
