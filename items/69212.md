## Summary

Keep #69212 open. Current main does not implement the PR’s central behavior for the HTTP transports: `openai-completions` still records replayable reasoning as a plain field-name signature and ignores streamed `encrypted_content`, while HTTP `openai-responses` / `openai-codex-responses` params still always send full `input` without a `previous_response_id` continuation plan. The existing WebSocket path has similar concepts, but that does not cover the PR’s HTTP transport scope. The PR also has substantive review feedback, so the best maintainer path is to refresh and fix those edge cases rather than close it as obsolete.

## Best Possible Solution

Keep the PR open, refresh it onto current main, and address the review comments before merge. The likely product direction is to adapt the existing WebSocket replay/continuation patterns into the HTTP transports with provider/route capability gates, preserving `encrypted_content` through streaming coalescing and buffer accounting, plus focused payload-shape regression tests for Volcengine/Doubao completions and native OpenAI/Codex Responses continuations.
