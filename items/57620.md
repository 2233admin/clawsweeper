## Summary

Current `main` already removes successful one-shot cron jobs when `deleteAfterRun` is true, and both code and regression coverage match the expected behavior described in the issue.

## Close Comment

Closing this as implemented on current OpenClaw.

- Codex review of current `main` found that successful one-shot jobs with `deleteAfterRun: true` are marked for deletion and then removed from the cron store, rather than only being disabled (`src/cron/service/timer.ts:463`, `src/cron/service/timer.ts:612`).
- The manual `openclaw cron run` path uses the same delete-and-persist flow (`src/cron/service/ops.ts:657`).
- The CLI still wires `--delete-after-run` to `deleteAfterRun: true`, while `--keep-after-run` maps to `false` (`src/cli/cron-cli/register.cron-add.ts:75`, `src/cli/cron-cli/register.cron-add.ts:223`).
- Regression coverage now expects removal after a successful one-shot run, and the CLI docs describe one-shot `--at` jobs as deleting after success by default (`src/cron/service.runs-one-shot-main-job-disables-it.test.ts:445`, `docs/cli/cron.md:30`).
