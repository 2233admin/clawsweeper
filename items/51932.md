## Summary

Current `main` already handles the reported plain-HTTP Control UI approval path by preserving operator scopes for localhost `allowInsecureAuth` sessions, and the repo includes regression coverage proving a device-less localhost Control UI client can still perform an `operator.admin` RPC. That means the reported `scopes=[] -> no-approval-route -> unknown or expired approval id` flow no longer matches current `main`.

## Close Comment

Closing from a Codex maintainer review: current `main` already covers this case. The plain-HTTP localhost Control UI path now preserves operator scopes when `gateway.controlUi.allowInsecureAuth=true` instead of clearing them during the missing-device handshake (`src/gateway/server/ws-connection/connect-policy.ts:129`, `src/gateway/server/ws-connection/message-handler.ts:616`). There is also regression coverage proving a device-less localhost Control UI session can connect and immediately succeed on an admin RPC (`src/gateway/server.auth.control-ui.suite.ts:100`, `src/gateway/server.auth.control-ui.suite.ts:324`), which would not be possible if the session still ended up with `scopes=[]` and no approval route. This behavior is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
