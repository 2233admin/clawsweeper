## Summary

Keep open. Current main still leaves the central bug actionable: the default plugin discovery and manifest cache windows are 1 second, activation still performs filesystem discovery/manifest registry construction before filtering runtime imports to requested plugin ids, and the checked-out repo still has a large bundled plugin surface. Main has useful mitigations for narrower scanner problems, including node_modules/dist skipping, persisted registry support, and reload debounce, but those do not close this issue's request for production-safe defaults or a constrained-host performance mode.

## Best Possible Solution

Keep the issue open and narrow it into a measurable plugin discovery performance task: profile gateway startup/reload and plugin inventory paths on low-core ARM-class hardware, then either lengthen or adapt the default discovery/manifest cache windows, use the persisted plugin registry more directly for activation planning, add plugin-state reload hysteresis where still missing, or introduce an explicit constrained-host performance preset with regression coverage.
