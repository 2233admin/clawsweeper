## Summary

Keep #71195 open. Current main already has the backend/browser Realtime Talk path and OpenAI browser-session provider, but the native macOS Talk runtime still uses SFSpeechRecognizer -> chat.send/chat.history -> local or gateway talk.speak playback. The issue discussion also has an explicit maintainer comment to keep this open and narrow it to native macOS integration, not close it as already implemented.

## Best Possible Solution

Keep the issue open and scope the implementation to native macOS Talk consuming the existing talk.realtime.session/OpenAI Realtime flow: add a GatewayConnection wrapper, implement a Swift realtime WebRTC/audio session, wire TalkModeRuntime overlay/cancellation/barge-in state to it, and retain the current SFSpeechRecognizer -> chat.send -> talk.speak path as fallback. Split any remaining ElevenLabs repeated-403 behavior into a smaller Mac Talk TTS issue if it still reproduces after configuring outputFormat.
