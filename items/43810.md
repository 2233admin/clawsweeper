## Summary

Keep this PR open. Current main still has the one-time pairing reply behavior that this PR is trying to change, so it is not obsolete or implemented. The submitted one-line patch is not safe to merge as-is because it would resend pending codes without preserving the existing first-created semantics for `onCreated`, and it does not update tests or docs that explicitly encode the current behavior.

## Best Possible Solution

Keep this PR open and request a complete implementation: resend the existing code when `upsertPairingRequest` returns a non-empty code, preserve `onCreated` so it only fires when `created === true` or introduce a separately named reply-sent callback, then update the shared pairing tests, affected channel tests, and public pairing/security docs if repeated reminders become the intended behavior.

## Risks / Open Questions

- The submitted patch would broaden `onCreated` from first request creation to every repeat pending request, producing duplicate logs and possible repeated one-time side effects in channel code.
- Changing resend behavior without docs and tests would leave public pairing/security guidance and channel expectations inconsistent with runtime behavior.
- Repeatedly resending codes may be an intentional UX improvement, but maintainers should decide whether it needs throttling or copy changes to avoid outbound-message spam from unapproved senders.
