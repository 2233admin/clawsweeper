## Summary

Keep open. Current main has useful pieces of the requested behavior, including optimistic chat submission, WebChat streaming/readiness UI, and non-blocking `chat.send` acknowledgements, but the broader request is not implemented. End-to-end UI/Gateway/provider/first-visible-token instrumentation is still missing, config editing still uses coarse page/controller busy flags rather than per-control save/test/progress isolation, and related open issues still track unresolved Control UI slowness and composer runtime-status work.

## Best Possible Solution

Keep the issue open, or split it into focused follow-ups: add end-to-end latency tracing for UI action -> Gateway RPC -> provider/model -> first visible update/final response; improve config screens with granular pending/test/save states, cancellation, and stale-response guards; and continue using the existing non-blocking/streaming chat path while filling the remaining status and measurement gaps.
