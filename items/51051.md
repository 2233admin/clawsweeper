## Summary

Current `main` still has the reported `nodes list`/`nodes status` split, and the PR as described has unresolved scope and JSON-contract risks. Conservative cleanup decision: keep open.

## Evidence

- **`nodes list` still uses pairing store as primary source on `main`:** The `list` subcommand calls `node.pair.list` first and only consults `node.list` for `--connected` / `--last-connected` filtering, so an empty pairing store still drives the unfiltered paired count/output.

  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **`nodes status` is built from a broader merged catalog:** `node.list` combines `listDevicePairing()`, `listNodePairing()`, and live connected nodes via `createKnownNodeCatalog()` / `listKnownNodes()`, so it can legitimately see nodes that `nodes list` misses.
