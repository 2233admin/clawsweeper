## Summary

Keep open. Current main implements the config-level `plugins.entries.<id>.hooks.allowConversationAccess` path, but not the requested manifest-owned `permissions.conversationAccess` contract. The manifest type/normalizer does not carry a permissions field, the loader passes only config entry hooks into hook policy, and enforcement still blocks non-bundled conversation hooks unless that config policy is explicitly true. The existing maintainer/Codex issue comment also kept this open for the same core API reason.

## Best Possible Solution

Keep #71428 open. The best path is an additive manifest-level permission contract, likely `permissions.conversationAccess`, normalized through `src/plugins/manifest.ts` and `src/plugins/manifest-registry.ts`, then merged with the existing `plugins.entries.<id>.hooks.allowConversationAccess` policy in the loader/registry path. Preserve the existing config flag as an operator override/backward-compatible path, and add focused loader/manifest tests plus updates to the public plugin manifest and hook docs.
