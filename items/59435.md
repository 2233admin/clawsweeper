## Summary

Keep open. Current main has launchd restart improvements, but the `openclaw update` service-refresh path still re-runs `gateway install --force` with the current invocation environment rather than first preserving the target plist's `OPENCLAW_CONFIG_PATH`. That leaves the reported multi-instance plist overwrite plausible and actionable in this repo.

## Best Possible Solution

Keep #59435 open and patch the update-time service refresh to target the installed service's existing command/plist environment. A conservative fix would read the selected service command before `gateway install --force`, preserve the target `OPENCLAW_CONFIG_PATH`, `OPENCLAW_STATE_DIR`, profile/label, and port unless explicitly overridden, and add a regression test for two macOS LaunchAgent instances with mismatched invocation env.

## Risks / Open Questions

- The failure mode is most likely in custom/env-driven multi-instance installs where the selected launchd label and the invoking `OPENCLAW_CONFIG_PATH` can diverge; profile-based installs may avoid it.
- A fix should preserve the intended package-update service environment refresh while making the target service/plist identity and config path authoritative.
