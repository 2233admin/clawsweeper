## Summary

Keep #72087 open. Current main still has a distinct `src/entry.ts` main-module bootstrap path before `runCli()`, while the lower OpenAI Codex image provider path is implemented and tested separately. I did not find source, test, docs, or history evidence that the reported Linux `dist/entry.js` vs direct `runCli()` fetch-failure boundary has been fixed on main.

## Best Possible Solution

Keep the issue open and investigate the `src/entry.ts` main-module prelude, respawn/env/profile normalization, and any process-global network/Undici side effects against the direct `runCli()` path. The best fix should make `dist/entry.js`, `openclaw.mjs`, and direct `runCli(argv)` behave identically for `openclaw infer image generate --model openai/gpt-image-2` on dual-stack Linux, then add a targeted regression or packaged-entry smoke that exercises the entry path rather than only the provider helper.
