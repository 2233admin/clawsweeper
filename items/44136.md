## Summary

Keep open. Current `main` still appears to overwrite a user-configured `models.providers.github-copilot.baseUrl` during GitHub Copilot runtime-auth setup, so this is not implemented on `main`. The open PR is also stale against the current architecture: Copilot token exchange now lives in the provider plugin/runtime-auth seam rather than the old `run.ts` path the PR edits, and the review thread already surfaced additional correctness risks.

## Evidence

- **Configured provider baseUrl still enters the resolved model on main:** `resolveModelAsync` still builds the runtime model from `providerConfig.baseUrl`, and the github-copilot model test asserts that a configured `https://proxy.example.com/v1` baseUrl is preserved on the resolved model before runtime auth runs.
  - file: [src/agents/pi-embedded-runner/model.ts:366](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/pi-embedded-runner/model.ts#L366)
  - command: `nl -ba src/agents/pi-embedded-runner/model.ts | sed -n '340,410p'; nl -ba src/agents/pi-embedded-runner/model.test.ts | sed -n '1468,1506p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **GitHub Copilot plugin unconditionally returns token-derived baseUrl:** The current GitHub Copilot provider plugin's `prepareRuntimeAuth` always returns `baseUrl: token.baseUrl` from `resolveCopilotApiToken(...)` and does not check `ctx.model.baseUrl`, so a user-configured custom baseUrl is not respected there.
  - file: [extensions/github-copilot/index.ts:168](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/github-copilot/index.ts#L168)
  - command: `nl -ba extensions/github-copilot/index.ts | sed -n '110,220p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Runner always applies prepared runtime-auth baseUrl to the effective model:** `applyPreparedRuntimeRequestOverrides` overwrites both `runtimeModel` and `effectiveModel` with `preparedAuth.baseUrl` whenever a plugin returns one, so the token-derived Copilot baseUrl wins over the earlier configured model baseUrl during auth setup/refresh.
  - file: [src/agents/pi-embedded-runner/run/auth-controller.ts:74](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/pi-embedded-runner/run/auth-controller.ts#L74)
  - command: `nl -ba src/agents/pi-embedded-runner/run/auth-controller.ts | sed -n '70,120p;368,440p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Current architecture moved this behavior out of the old run.ts patch site:** The plugin contract explicitly makes runtime auth, including an optional base URL override, provider-owned via `prepareRuntimeAuth`. That means the PR's edits against the old embedded-runner token flow are stale against current `main`, even though the underlying bug still looks real.
  - file: [src/plugins/types.ts:1391](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/plugins/types.ts#L1391)

That makes the proposed change not obviously ready to land as-is.
