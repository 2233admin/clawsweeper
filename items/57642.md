## Summary

Current `main` already strips malformed UTF-16 surrogate halves before provider request payloads are serialized, and regression tests cover preserving valid emoji while removing lone surrogate code units. That addresses the reported Discord-label poison path well enough to close as already implemented on `main`.

## Close Comment

This looks fixed on current `main`, so I'm closing it.

- Codex review found a shared transport guard in `src/agents/transport-stream-shared.ts:22` that strips lone UTF-16 surrogate halves before provider payloads are serialized.
- That sanitizer is applied when building outbound text payloads for the current OpenAI, Anthropic, and Google HTTP transports (`src/agents/openai-transport-stream.ts:244`, `src/agents/anthropic-transport-stream.ts:218`, `extensions/google/transport-stream.ts:297`), so persisted Discord/session labels cannot keep poisoning later JSON requests through those paths.
- Regression tests in `src/agents/transport-stream-shared.test.ts:10` and `src/agents/openai-transport-stream.test.ts:332` verify the intended behavior: malformed surrogate halves are removed, while valid emoji surrogate pairs are preserved.
- Codex also confirmed the same sanitizer/test are already present in the latest release snapshot reviewed here (`v2026.4.23`, release SHA `9b18ae877018a34933e27bb7b2986cb97e9602aa`).
