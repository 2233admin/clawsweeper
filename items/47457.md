## Summary

Current main still reserves `mode:"session"` for thread-bound conversations, but the reported headless orchestration blocker no longer matches behavior: spawned sessions default to `cleanup:"keep"`, remain available until auto-archive, and can be re-entered with `sessions_send` against the existing `sessionKey`. That covers the WhatsApp follow-up/session-retention use case without thread binding.

## Close Comment

Closing from a Codex review of current `main` (`0e7250f`). The reported blocker no longer matches current behavior. On current main, `sessions_spawn` defaults `cleanup` to `keep` (`src/agents/tools/sessions-spawn-tool.ts:192`), keep-mode child runs are retained instead of being immediately archived (`src/agents/subagent-registry-run-manager.ts:304`), and the docs now say sub-agent sessions remain until `agents.defaults.subagents.archiveAfterMinutes` (`docs/tools/subagents.md:150`). Follow-up work is handled through `sessions_send`, which is documented as delegating to an existing session (`src/agents/tool-description-presets.ts:27`) and starts a new `agent` run against the target `sessionKey` (`src/agents/tools/sessions-send-tool.ts:55`). `mode="session"` is still intentionally thread-bound (`src/agents/subagent-spawn.ts:578`), but the headless WhatsApp orchestration use case described here is already achievable on main with `mode:"run"` plus the default kept session and `sessions_send`, so I could not reproduce the reported “must respawn and recontextualize every time” limitation.
