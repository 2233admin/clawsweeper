## Summary

Keep #59078 open. Current main still cannot express or evaluate the reported “env var OR skill-owned config file” credential setup path: skill status only treats `requires.env` as satisfied by process env, `skills.entries.<key>.env`, or `skills.entries.<key>.apiKey` tied to `primaryEnv`, while `requires.config` is limited to truthy paths inside `openclaw.json`. The reported ima-skill is external and belongs on ClawHub, but the blocker is a missing generic core/skill metadata setup-status contract, so this should not be closed as ClawHub-only cleanup.

## Best Possible Solution

Keep the issue open and design a generic skill setup-status extension point. A conservative path would add a declarative requirement form for credential alternatives, such as env vars OR specific user-owned file-presence checks, or a trusted plugin setup-status hook that returns redacted readiness. Once that core contract exists, ima-skill can publish corrected metadata through ClawHub without adding ima-specific logic to OpenClaw core.

## Risks / Open Questions

- A naive executable setup-status hook could run untrusted skill code during UI/status refresh; a safer fix should prefer declarative checks or a trusted plugin-owned setup hook with clear security boundaries.
- File-based credential checks must avoid leaking secret values or overly specific private paths in status payloads, logs, and UI.
- The contract should define host vs sandbox/remote semantics, because current skill env injection is scoped to host agent runs.
