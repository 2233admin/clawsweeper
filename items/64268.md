## Summary

Current main still does not provide per-LLM-call `llm_output` events or a new `llm_call` hook for agentic loops. The existing embedded runner accumulates assistant usage during streamed messages and emits one aggregate `llm_output` after the model attempt. The related PR #45990 remains an implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open. The best path is to review and either land or replace PR #45990 with an additive, documented per-call observability contract: either a new `llm_call` hook or a backwards-compatible `llm_output` expansion that preserves aggregate semantics while exposing per-call usage, provider/model, run/session/agent ids, optional latency, and tool-call data. The implementation should update hook types, embedded subscription/attempt threading, plugin docs, and focused multi-round agentic-loop tests.
