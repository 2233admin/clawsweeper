## Summary

Keep open. Current main still has the reported stale fallback path: when a path-scoped Control UI key is missing, `loadSettings()` falls back to the shared legacy localStorage key and preserves a sibling path's gateway URL. The PR is a focused in-repo UI storage fix with relevant tests/discussion, not something already implemented or safely obsolete.

## Best Possible Solution

Keep the PR open for maintainer review or supersede it with an equivalent patch in `ui/src/ui/storage.ts`. The best path is a regression-tested storage migration that prevents legacy-key cross-contamination for path-prefixed same-host instances while preserving intentional custom gateway URLs, with focused tests in `ui/src/ui/storage.node.test.ts` for `/a` to `/b`, upgraded unflagged legacy data, and same-host custom WebSocket path overrides.

## Risks / Open Questions

- The fix must preserve explicit user-configured same-host but different-path WebSocket endpoints, because docs and UI support a configurable gateway URL independent of the Control UI origin.
- Legacy unflagged localStorage migration is subtle: a solution that only marks new writes may not fix the first upgraded visit after a user switches from one path-prefixed instance to another.
