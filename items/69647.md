## Summary

Current main still appears to have the stale gateway URL behavior described by PR #69647. `loadSettings()` can fall through to the shared legacy localStorage key for a different path-prefixed instance and preserves that sibling `gatewayUrl`, and main does not yet contain the PR's page-scoped or auto-derived metadata fix.

## Best Possible Solution

Keep PR #69647 open for maintainer review. The best path is a narrow Control UI storage fix that distinguishes page-derived defaults from explicit user gateway overrides, seeds/updates a page-derived settings key for path-prefixed instances, and adds regression tests for `/a` -> `/b`, same-host custom WebSocket paths, cross-host overrides, and single-instance behavior.

## Evidence

- **Legacy fallback still preserves sibling-path gateway URL:** `loadSettings()` derives the current page/default gateway URL, reads the current gateway-scoped key, then `default`, then the shared legacy key. It only substitutes the page default when the stored URL exactly equals the current page-derived URL, so a legacy `wss://host/a` value loaded while visiting `/b` remains selected.
