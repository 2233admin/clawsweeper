## Summary

Current main and the latest shipped release treat `sk-ant-oat*` as Anthropic token auth on purpose, not API-key auth, and now have dedicated setup-token onboarding plus test/QA coverage showing Opus 4.6 works on that path. The report's claim that Bearer routing itself breaks current Anthropic setup-token usage no longer matches the codebase or shipped docs/tests.

## Evidence

- **Current transport intentionally routes `sk-ant-oat*` through Bearer auth:** `isAnthropicOAuthToken()` matches `sk-ant-oat` tokens, and the Anthropic transport sends those via `authToken`/`Authorization: Bearer` instead of `x-api-key`.
  - file: [src/agents/anthropic-transport-stream.ts:190](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/anthropic-transport-stream.ts#L190)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Setup-token is now a first-class token flow, not API-key flow:** The bundled Anthropic provider exposes `setup-token` as `kind: "token"` with the hint `Manual bearer token path`, separate from `api-key`.
  - file: [extensions/anthropic/provider-contract-api.ts:29](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/anthropic/provider-contract-api.ts#L29)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)

## Close Comment

Closing this as `cannot_reproduce` after a Codex review against current `main` and the latest shipped release.

- `src/agents/anthropic-transport-stream.ts:190` and `src/agents/anthropic-transport-stream.ts:561` intentionally treat `sk-ant-oat*` as token auth and send it as `Authorization: Bearer`, not `x-api-key`.
- `extensions/anthropic/provider-contract-api.ts:29` and `extensions/anthropic/register.runtime.ts:121` now model Anthropic `setup-token` as a separate `token` flow, not `api_key`.
- `src/agents/anthropic-transport-stream.test.ts:282` asserts that Bearer path for `sk-ant-oat*`, and `src/agents/anthropic.setup-token.live.test.ts:181` plus `qa/scenarios/models/anthropic-opus-setup-token-smoke.md:12` cover successful Opus 4.6 setup-token usage on the regular Anthropic provider.
- `docs/providers/anthropic.md:280` documents token auth as expiring/revocable legacy token auth and recommends API keys for new setups.
- The same behavior and coverage are present in the latest release `v2026.4.23`, so this no longer matches current shipped behavior.

If someone can still reproduce a real failure on a current release, they should open a fresh issue with the exact auth method used (`setup-token`, Claude CLI, or API key), the selected model ref, and a current request/response error trace.
