## Summary

Keep open. Current `main` still routes `status` through the built-in `memorySearch` config gate, so a third-party memory-slot plugin with `agents.defaults.memorySearch.enabled: false` still resolves to `null` and renders as `enabled (plugin …) · unavailable`.

## Evidence

- **Shared status snapshot still hard-gates on built-in memory config:** `resolveSharedMemoryStatusSnapshot()` returns `null` when `resolveMemoryConfig(cfg, agentId)` is null, before asking the active memory plugin manager for status. That path is still present on current `main`.
  - file: [src/commands/status.scan.shared.ts:197](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/commands/status.scan.shared.ts#L197)
  - command: `nl -ba src/commands/status.scan.shared.ts | sed -n '178,220p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Built-in memory config resolver returns null when memorySearch is disabled:** `resolveMemorySearchConfig()` returns `null` whenever the merged `memorySearch.enabled` is false. That means third-party memory-slot setups that disable built-in memory search still fail the status snapshot gate.
  - file: [src/agents/memory-search.ts:387](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/memory-search.ts#L387)
  - command: `nl -ba src/agents/memory-search.ts | sed -n '380,392p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Null memory snapshot renders the exact unavailable string:** `buildStatusMemoryValue()` renders `enabled (plugin <slot>) · unavailable` whenever `memoryPlugin.enabled` is true but `memory` is missing.
  - file: [src/commands/status.command-sections.ts:153](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/commands/status.command-sections.ts#L153)
