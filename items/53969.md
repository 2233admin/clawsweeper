## Summary

Keep open. Current main has already shipped several important parts of the request, including atomic JSON writes, Codex CLI bootstrap-only behavior, same-provider/profile refresh serialization, and refresh-token-reused recovery. The issue is still actionable because OAuth-producing auth writes still use the unlocked upsert path, and duplicate enrollment protection is still keyed to provider/profileId rather than a stable enrollment or explicit copied-store/single-writer boundary.

## Best Possible Solution

Keep this issue open and narrow it to the remaining hardening: route OAuth-producing auth writes through updateAuthProfileStoreWithLock or upsertAuthProfileWithLock, or make the default OAuth upsert path lock-safe without breaking plugin SDK callers; add feasible duplicate-enrollment detection within one store; and document the single-writer boundary for copied auth-profiles.json files, multiple hosts/stores, and Codex CLI coexistence. Preserve the shipped atomic writes, same-profile refresh serialization, refresh-token-reused recovery, and Codex bootstrap-only behavior.

## Risks / Open Questions

- Any fix should preserve public plugin SDK compatibility while moving OAuth-producing writes onto a lock-safe path or clearly documenting an intentional boundary.
