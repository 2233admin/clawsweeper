## Summary

Keep open. Current `main` still misroutes TUI `/model list` and `/model status`, but this PR’s local alias fix would bypass the richer, agent-aware shared `/model` command behavior already implemented elsewhere on `main`.

## Evidence

- **Current TUI still treats any `/model <args>` as a model-id patch:** `handleCommand()` has a dedicated `case "model"` that opens the selector only for bare `/model`; otherwise it calls `client.patchSession({ key, model: args })`. On current `main`, `/model list` and `/model status` still fall into that patch path.
  - file: [src/tui/tui-command-handlers.ts:353](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/tui/tui-command-handlers.ts#L353)
  - command: `nl -ba src/tui/tui-command-handlers.ts | sed -n '348,372p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **The PR’s `/model list` alias would use the global model RPC, not the agent-aware shared command path:** The local selector path calls `client.listModels()`. In the gateway-backed TUI client, `listModels()` issues `models.list` with no params, and the server method validates an empty params object then builds the catalog from gateway config alone. That path has no `agentId`, `agentDir`, or `sessionEntry` inputs.

- **Shared `/model` handling on `main` already gives agent/session-aware `/model list` behavior:** The shared command implementation recognizes `list` and routes it through `resolveModelsCommandReply(...)`, explicitly passing `activeAgentId`, `agentDir`, and `sessionEntry` so model browsing is resolved in the active agent/session context.
