## Summary

Keep open. Current `main` still accepts malformed trailing-`@` ClawHub specs in the parser and routes them into normal package lookup, so the reported bug remains plausible and the PR's fix is not implemented on `main`.

## Evidence

- **Current parser still accepts trailing-@ specs:** `parseClawHubPluginSpec` returns `{ name: spec }` whenever `atIndex <= 0 || atIndex >= spec.length - 1`. That means `clawhub:demo@` and `clawhub:@scope/pkg@` are still treated as package names instead of invalid specs.
  - file: [src/infra/clawhub-spec.ts:16](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/clawhub-spec.ts#L16)
  - command: `nl -ba src/infra/clawhub-spec.ts | sed -n '1,40p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Install path still converts malformed specs into package lookup errors:** `installPluginFromClawHub` only returns `invalid_spec` when the parser returns null or no name. Otherwise it fetches package detail, and `mapClawHubRequestError` converts a 404 package lookup into `package_not_found`. With the current parser, malformed trailing-`@` specs still fall into that later error path.
  - file: [src/plugins/clawhub.ts:146](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/plugins/clawhub.ts#L146)
  - command: `nl -ba src/plugins/clawhub.ts | sed -n '136,170p'; nl -ba src/plugins/clawhub.ts | sed -n '762,782p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current tests do not cover the malformed trailing-@ regression:** The parser test on `main` covers `clawhub:demo`, `clawhub:demo@1.2.3`, and rejecting raw `@scope/pkg`, but it has no assertion for `clawhub:demo@`, `clawhub:@scope/pkg@`, or the scoped valid case discussed in the PR thread.
  - file: [src/infra/clawhub.test.ts:37](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/clawhub.test.ts#L37)

- **Docs show only package and package+version forms as valid syntax:** The user-facing docs document `clawhub:<package>` and `clawhub:<package>@<version>` as the supported locator forms, which supports the PR's premise that a trailing `@` with no version is malformed rather than a valid package name.
