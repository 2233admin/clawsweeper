## Summary

Keep open. Current `main` does not include the PR's ACP metadata propagation for thread/topic-bound sessions, and ACP resolution still requires `entry.acp` on the exact session key.

## Evidence

- **Current `recordSessionMetaFromInbound` lacks ACP copy logic:** On `main` at `01bc49c88c0cef9807b15df4d7486a0c5fe4afbd`, `recordSessionMetaFromInbound` only derives the inbound patch, merges it, and stores it. There is no branch that derives a base session key from `:thread:`/`:topic:` and copies `acp` from the parent entry, which is the core behavior proposed by PR #48967.
  - file: [src/config/sessions/store.ts:680](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/config/sessions/store.ts#L680)
  - command: `nl -ba src/config/sessions/store.ts | sed -n '680,725p'`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **ACP resolution still fails when the looked-up session entry has no `acp`:** `AcpSessionManager.resolveSession()` reads ACP metadata for the requested session key and returns `stale` with `ACP_SESSION_INIT_FAILED` when that session has no `acp`. There is no fallback here to a base thread/topic session.
  - file: [src/acp/control-plane/manager.core.ts:172](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/acp/control-plane/manager.core.ts#L172)
