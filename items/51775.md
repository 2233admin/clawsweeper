## Summary

Keep open. Current `main` still does not apply the proposed embedded-run / queue / sub-agent cleanup in `chat.abort`, so this is not implemented on main, but the PR's current approach appears too broad: it would turn run-scoped or partially authorized aborts into session-wide cleanup and can interrupt unrelated work on the same session.

## Evidence

- **Current main still lacks the PR change:** `chat.abort` on current `main` aborts controllers and persists partials, then responds. There are no `abortEmbeddedPiRun`, `clearSessionQueues`, or `stopSubagentsForRequester` calls in either the session-key or runId paths.
  - file: [src/gateway/server-methods/chat.ts:1956](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/gateway/server-methods/chat.ts#L1956)

  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Session aborts are intentionally owner-scoped:** The existing authorization test proves session-scoped `chat.abort` must only abort the requester's owned runs and leave another device's run active.
  - file: [src/gateway/server-methods/chat.abort-authorization.test.ts:85](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/gateway/server-methods/chat.abort-authorization.test.ts#L85)

## Risks / Open Questions

- The underlying bug reported by the PR likely still exists on current `main`, because `chat.abort` still only aborts gateway-side controllers while `/stop` performs broader cleanup in `src/auto-reply/reply/abort.ts:291-335`.
- Merging the PR as described would risk aborting another in-flight or queued prompt on the same session, and could break device-owner isolation for partially authorized session aborts.
