## Summary

Keep open. Current main still logs failed configured/bundle MCP server startup at the gateway/runtime layer and then continues with a catalog containing only successfully connected servers/tools. I did not find an implemented agent-visible notification, required-server block, or tool-diff awareness for the embedded Gateway/Telegram path described in the issue.

## Best Possible Solution

Add a generic MCP startup diagnostic contract to the session MCP runtime/catalog, propagate failed-server details through embedded tool materialization into the agent-visible turn context, and decide product semantics for `mcp.servers.<name>.required` so required failures can fail fast and notify the active channel. Cover it with a focused embedded runner/runtime test for a configured server that fails to connect.
