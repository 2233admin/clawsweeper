## Summary

Current main still silently degrades configured bundle MCP tools when a server fails during startup. The runtime logs the failure, disposes that server session, deletes it from the connected sessions map, and returns a catalog containing only successful servers/tools. I did not find agent-facing diagnostics, required-server blocking semantics, expected-vs-available tool diff support, or docs/tests indicating the requested behavior has shipped.

## Best Possible Solution

Keep this issue open and implement a generic bundle-MCP startup diagnostic path: preserve redacted failed-server records in the session MCP catalog, expose those diagnostics to embedded runner prompt/channel context, add typed `mcp.servers.<name>.required` semantics that can block agent start and notify the active channel, add expected-vs-available tool awareness where feasible, and cover stdio/SSE/streamable-HTTP startup failures with focused runtime, runner, config, and docs tests.

## Risks / Open Questions

- Startup diagnostics must redact sensitive command args, URLs, headers, and environment values before reaching the model or channel.
- Required-server semantics need a clear transport-agnostic contract so one optional MCP failure does not block unrelated tools, while a required server can fail fast with a user-visible message.
- Because unknown MCP server fields are currently accepted without behavior, users can configure `required` or `onStartFailure` and mistakenly believe they are enforced.
