## Summary

Keep this issue open. Current main still returns the explicit-token error before trying thread-bound ACP resolution, so the reported `/acp close <non-session-token>` bound-thread case remains broken. The related fix PR #66338 was closed unmerged, and current tests cover bound-thread close only when no bad token is present.

## Best Possible Solution

Keep this issue open and land a focused resolver fix: if explicit token resolution returns null, try `resolveBoundAcpThreadSessionKey` before returning the explicit-token error. Preserve the existing bad-token error when no bound thread target resolves, and add regression coverage for `/acp close <non-session-token>` inside a bound thread plus the no-binding bad-token case.
