## Summary

Keep open. Current main still intentionally retries the configured primary on the next direct turn after an auto fallback, so it does not provide the requested cache-aware sticky fallback window across future requests. The related OpenRouter provider-routing work is present and useful for #62992, but it only pins routing inside OpenRouter requests; it does not solve cross-provider prompt-cache bouncing after automatic failover.

## Best Possible Solution

Implement a core/session failover policy that records a successful automatic fallback with an expiry, honors it across future direct turns before clearing the auto override, probes the configured primary once after expiry, and extends the sticky window when that probe fails. Put config near auth cooldown/model failover settings, support fixed and cacheRetention-derived durations, preserve explicit user overrides, and add tests for primary timeout -> fallback success -> next turn starts on fallback -> expiry probes primary. Keep the closed OpenRouter provider-routing work as a separate mitigation for #62992, not as the replacement for this issue.

## Risks / Open Questions

- A sticky fallback policy will need careful interaction with explicit user model overrides, inherited parent-session overrides, live model switching, auth-profile cooldown probes, and cacheRetention-derived TTLs.
- The default duration needs product judgment so OpenClaw avoids rapid prompt-cache bouncing without trapping a session on a degraded fallback longer than intended.
