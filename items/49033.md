## Summary

Current main already covers the reported missing-dependency crash class. The packaged-runtime fix shipped in v2026.4.15 moved bundled plugin runtime deps back to their owning extensions, current bootstrap code treats bundled plugin load failures as non-fatal instead of crashing CLI startup, and the affected deps are now declared on the owning plugins with repair paths in place.

## Close Comment

Closing from a Codex review: current `main` already covers this bug well enough. The packaged-runtime work shipped in `v2026.4.15` moved bundled plugin runtime deps back to their owning extensions (`CHANGELOG.md:562`, merge `c727388f937ffc9128ba4e4d2ae37d4c72487bc1`), so the affected deps are no longer expected to live in the root `package.json`. Current bootstrap loading also no longer hard-crashes when a bundled plugin import throws; it marks that plugin missing and continues (`src/channels/plugins/bootstrap-registry.ts:117`, with regression coverage in `src/channels/plugins/bundled-root-caches.test.ts:148`). The deps named here are declared on their owning plugins today (`extensions/googlechat/package.json:7`, `extensions/nostr/package.json:6`, `extensions/zalouser/package.json:6`), and bundled runtime-deps repair is still wired through install/doctor (`package.json:1410`). If this still reproduces on a current release, please open a fresh issue with the exact image/tag and failing command output.
