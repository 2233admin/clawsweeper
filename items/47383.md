## Summary

Current main already fixes the reported Copilot runtime-baseUrl problem: the provider returns token-derived `baseUrl` during runtime auth preparation, and the embedded runner applies that `baseUrl` to the live model before requests. The same behavior is present in the latest release commit.

## Evidence

- **GitHub Copilot provider returns runtime `baseUrl`:** The bundled provider's `prepareRuntimeAuth` now returns `{ apiKey, baseUrl, expiresAt }` from `resolveCopilotApiToken`, so the token-derived endpoint is preserved instead of discarded.
  - file: [extensions/github-copilot/index.ts:168](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/extensions/github-copilot/index.ts#L168)

## Close Comment

Closing from a Codex review because the reported runtime `baseUrl` problem is already implemented in current OpenClaw and in the latest released build. The GitHub Copilot provider now returns runtime auth with the token-derived `baseUrl` in `extensions/github-copilot/index.ts:168`, and the pi-embedded runner applies that `preparedAuth.baseUrl` to both the runtime and effective model before requests are sent in `src/agents/pi-embedded-runner/run/auth-controller.ts:97`. The other lightweight completion path also carries the exchanged Copilot endpoint and has regression coverage in `src/agents/simple-completion-runtime.test.ts:246`. I also verified the same implementation is present in release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` (`v2026.4.22`).

The separate shared cache-file concern in `src/agents/github-copilot-token.ts:18` looks like a different issue from the title/body here. If cross-profile Copilot token cache contamination is still reproducible on a current build, please open a new issue focused on that behavior with the exact profile setup and repro steps.
