## Summary

Keep open. Current main still classifies Moonshot/Kimi billing-shaped HTTP 429 payloads as rate_limit before billing evidence or provider-specific hooks can win. Adjacent model-switch and failover paths have improved, but the central insufficient-balance 429 misclassification remains actionable.

## Best Possible Solution

Keep this open. Add a regression test for a Moonshot/Kimi HTTP 429 payload whose message/code clearly says insufficient balance or credits, then adjust failover classification so billing evidence or a Moonshot provider-owned classifyFailoverReason hook can override the broad 429 rate-limit rule. Verify the user-facing message becomes billing-specific and auth-profile state uses billing disable/backoff rather than the short rate-limit cooldown.
