## Summary

Keep open. Current `main` still has abort primitives, but it does not yet implement the issue's requested unified run-generation invalidation or stale-output fencing across the reply pipeline. The linked draft PR #70363 is active evidence that the work is still in progress rather than obsolete.

## Evidence

- **Reply run registry has abort state but no generation/currentness contract:** `ReplyOperation` exposes `abortSignal`, phase, and result state, but no run-generation field or `isCurrent()`-style API. The registry only tracks active runs/session IDs and aborts by signaling/canceling the backend.
