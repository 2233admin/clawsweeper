## Summary

Keep #70634 open. Current main still serializes agent run lanes and busy-session followups with FIFO arrays, and I found no shipped priority tier that moves human-originated inbound messages ahead of inter-session `sessions_send`/agent-originated work. The newer agent-to-agent ping-pong cap reduces runaway reply-back chains, but it does not implement the requested human-over-agent queue ordering. The KinthAI comment is useful supporting evidence because it reports the same starvation pattern and describes a working two-tier dispatcher design, but that design is not present in OpenClaw core.

## Best Possible Solution

Keep the issue open and implement a small core queue priority contract: classify inbound work at the channel/gateway boundary using existing provenance or an explicit queue priority, treat external human/user input as priority over `inter_session` agent work within the same session lane, preserve FIFO order within each tier, and add regression tests for a `sessions_send` loop with a later human Discord/webchat message. The existing `agentToAgent.maxPingPongTurns` cap should remain as loop back-pressure, but it is not a substitute for human-message priority.
