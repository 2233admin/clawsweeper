## Summary

Keep open. Current main still has the reported channel auto-reply tool-policy gap, and this PR is a plausible security-hardening candidate, but the latest diff still needs maintainer/security review for policy scope and runtime authorization correctness.

## Best Possible Solution

Keep this PR open for maintainer/security review. Revise the patch so dangerous filesystem/exec tools are denied for untrusted external channel auto-reply contexts, internal/trusted providers and memory-flush read/write behavior are explicitly preserved, and any exec owner-only hardening is enforced through `src/agents/tool-policy.ts` or exec tool metadata. Review alongside the related open same-author PR #64718, which covers the exec-default side of the same threat model.

## Risks / Open Questions

- Closing this PR now would leave current main's external-channel dangerous-tool exposure unchanged.
- Merging the latest diff as-is may strip exec/read/write/edit tools from trusted internal providers such as webchat, heartbeat, cron-event, and exec-event, which is outside the stated untrusted-channel threat model.
- The current exec owner-only hardening in the PR is not wired through the runtime owner-only policy or exec tool metadata, so it can create a false sense of defense-in-depth.
- Channel memory-flush runs rely on read/write tools; a broad default deny can prevent persistence unless the final policy explicitly preserves that path.
