## Summary

Keep open. Current `main` still embeds managed env values into the gateway install environment, which matches the bug described by #70612, and this PR also has an unresolved review concern about accidentally purging service-owned keys like `PATH`.

## Evidence

- **Current main still persists managed env values into the install environment:** `buildGatewayInstallEnvironment` merges `managedEnvironment` into `environment`, then only records `OPENCLAW_SERVICE_MANAGED_ENV_KEYS`; there is no follow-up removal of the managed keys themselves on current `main`.
  - file: [src/commands/daemon-install-helpers.ts:223](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/commands/daemon-install-helpers.ts#L223)
  - command: `nl -ba src/commands/daemon-install-helpers.ts | sed -n '215,255p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Runtime dotenv loader skips keys already present in process.env:** `loadRuntimeDotEnvFile` continues early when `process.env[key] !== undefined`, so values embedded by the service manager will prevent refreshed `.env` values from loading at startup.
  - file: [src/infra/dotenv.ts:166](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/infra/dotenv.ts#L166)
  - command: `nl -ba src/infra/dotenv.ts | sed -n '150,230p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Tests on main still expect literal managed secrets in the install plan:** The existing tests assert that managed values like `GOOGLE_API_KEY`, `CUSTOM_VAR`, `SAFE_KEY`, `OPENAI_API_KEY`, and `ANTHROPIC_TOKEN` remain present in `plan.environment`, which is the behavior the PR is trying to change.
  - file: [src/commands/daemon-install-helpers.test.ts:197](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/commands/daemon-install-helpers.test.ts#L197)
  - command: `nl -ba src/commands/daemon-install-helpers.test.ts | sed -n '180,335p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Unresolved review concern on the proposed fix:** The PR discussion already contains a substantive review comment against commit `6b4780ca4f1e202fd7b18523adb0f81046249a18`: deleting every key named in `OPENCLAW_SERVICE_MANAGED_ENV_KEYS` could also remove service-owned variables such as `PATH` if they enter the managed set, and the author replied that this would be a possible follow-up rather than addressing it in the PR.
  - sha: [6b4780ca4f1e](https://github.com/openclaw/openclaw/commit/6b4780ca4f1e202fd7b18523adb0f81046249a18)
- **Durable service env collection does not special-case PATH today:** `collectDurableServiceEnvVars` reads state-dir `.env` and config env vars, filtering only the dangerous host-env blocklists; it does not explicitly exclude `PATH`, while service install logic separately merges a curated service PATH.
