## Summary

Current `main` still contains the reported Telegram inbound placeholder mismatch. The Telegram body builder still falls back to `<media:image>` for every non-audio attachment without consulting `allMedia.contentType`, and the Telegram media resolver still does not provide a MIME hint for `msg.photo`. I did not find current tests covering the reported non-image or mixed-media placeholder cases, so there is no evidence this was fixed on `main`.

## Evidence

- **Telegram inbound body still hardcodes image placeholder for all non-audio media:** `resolveTelegramInboundBody()` sets `bodyText` to `<media:image>` whenever `allMedia.length > 0` and no audio is present. It does not inspect `allMedia[*].contentType`, so a non-image or unknown attachment still produces an image placeholder.
