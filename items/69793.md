## Summary

Keep open. Current main still appears to have the reported Telegram mismatch: photo messages are classified from the Telegram message shape as `<media:image>` before considering the resolved attachment `contentType`, while the session payload separately carries the sniffed media type. The linked PR context does not supersede this because #69893 was closed unmerged, and the existing Telegram body tests do not cover non-image or generic MIME media attached through `allMedia`.

## Best Possible Solution

Keep the issue open and fix it in the Telegram plugin body-building path, preferably by adding a shared media-placeholder helper that derives the marker from resolved `allMedia` MIME kinds. Add focused tests for Telegram photo-shaped messages whose resolved media is `application/octet-stream` or another non-image type, plus mixed/multiple media cases, and then consider applying the same helper to other channel plugins that build media placeholders manually.
