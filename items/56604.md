## Summary

Keep open. Current `main` still treats Voice Call settings as one global config. The dialed `to` number is captured on inbound events, but there is no typed per-number route table, no per-call effective config resolution, and downstream response/TTS paths still read the single global plugin config.

## Evidence

- **Voice Call config has only global response/TTS fields:** `VoiceCallConfigSchema` defines top-level global keys like `fromNumber`, `toNumber`, `inboundGreeting`, `tts`, `agentId`, `responseModel`, and `responseSystemPrompt`, but no `numbers` map or similar per-dialed-number routing structure.
  - file: [extensions/voice-call/src/config.ts:300](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/voice-call/src/config.ts#L300)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Call manager keeps one shared config for all calls:** `CallManagerContext` exposes a single `config: VoiceCallConfig`, and `CallManager.getContext()` passes `this.config` through unchanged. There is no stored per-call effective config alongside the call record.
  - file: [extensions/voice-call/src/manager.ts:283](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/voice-call/src/manager.ts#L283)
