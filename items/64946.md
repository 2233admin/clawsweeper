## Summary

Keep open. Current main still has the validation boundary needed for the reported false-positive plugin config noise: plugin-owned schemas are documented and applied under `plugins.entries.<id>.config`, but loader paths still pass `entry?.config` into strict manifest schemas and `validatePluginConfig` coerces missing values to `{}` before logging `invalid config`. The related #70811 is still an open implementation candidate, not a merged fix on current main.

## Best Possible Solution

Keep this issue open and fix the plugin loader validation boundary. The likely maintainer path is to land or extend #70811 so non-activating snapshot/CLI metadata paths defer validation when plugin config is genuinely unavailable, while real activation and explicit config validation remain strict. Add focused regression coverage for a strict-schema plugin configured under `plugins.entries.<id>.config` and for non-activating missing-config paths that must not emit false `invalid config` logs.
