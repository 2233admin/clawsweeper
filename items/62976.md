## Summary

Keep this issue open. Current main still treats invalid known plugin entry config as a hard config validation failure, intentionally skips last-known-good recovery when all issues are under plugins.entries.*, and then rejects gateway startup. Doctor repair only covers stale unknown plugin IDs and bundled plugin load-path aliases, not quarantine/disablement or deterministic recovery for a known third-party plugin with invalid config. Related #70371 and open PR #70394 are useful overlap, but the relevant behavior is not shipped on main and does not cover the full doctor/safe-mode recovery scope here.

## Best Possible Solution

Keep this issue open and fix it in core. Add a narrow startup degradation path that disables or quarantines only the bad third-party plugin entry while keeping the gateway bootable, surface the failed plugin and schema path in plugin/status/doctor diagnostics, and extend openclaw doctor --fix or a safe-mode/plugin-quarantine command to repair plugins.entries, plugins.allow, and related plugins.load.paths deterministically. Coordinate with #70371/#70394, but do not close this issue until startup degradation plus doctor/safe-mode recovery behavior is merged with regression tests.
