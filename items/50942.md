## Summary

Current main already supports canonical cron object schedules (`{ kind: "cron", expr: "..." }`) and treats string schedules as legacy store-compatibility only, so the reported docs/code mismatch is no longer present.

## Close Comment

Closing as implemented on main after a Codex review. Current OpenClaw accepts canonical cron object schedules: `src/gateway/protocol/schema/cron.ts:119` defines the cron schedule as `{ kind: "cron", expr, tz?, staggerMs? }`, and `src/cron/normalize.ts:67` preserves `schedule.expr` when it is present instead of stripping it. There is also regression coverage for canonicalizing legacy cron fields in `src/cron/normalize.test.ts:211`, and the docs now describe string schedules as legacy store-compatibility migration rather than the canonical persisted shape (`docs/gateway/doctor.md:292`, `src/commands/doctor-cron-store-migration.ts:265`). This implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still see `invalid cron schedule: expr is required` on a newer build, please open a fresh issue with the exact `~/.openclaw/cron/jobs.json` entry before and after the rewrite/save cycle plus the gateway log around that rewrite path.
