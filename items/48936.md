## Summary

Current `main` already fixes the reported false-negative classification for gateway probes: successful WebSocket connect is tracked separately from slower follow-up read RPCs, active loopback probes use the full caller budget, and the shipped CLI now reports post-connect problems as `Read probe: failed/limited` instead of `Connect: failed`. The same behavior is present in the released `v2026.4.22` tree.

## Evidence

- **Probe logic separates connect success from follow-up read/detail work:** After `hello-ok`, the probe records `connectLatencyMs` and only then runs detail RPCs under a separate timer, with an explicit comment that slow follow-up RPCs should not downgrade the target to unreachable.
  - file: [src/gateway/probe.ts:262](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/probe.ts#L262)
