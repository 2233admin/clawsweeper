## Summary

Keep open. Current `main` still appears to reject the exact probe shape described in #67703: unauthenticated loopback probes are created without a device identity, while the WebSocket handshake still requires device identity for non-Control-UI operator sessions unless shared-secret auth succeeded. The linked RFC #69066 also suggests this is still considered an unresolved core design gap rather than a stale report.

## Evidence

- **Probe drops device identity on unauthenticated loopback:** `probeGateway()` explicitly returns `null` for `deviceIdentity` when probing a loopback URL without a token or password, so `gateway.auth.mode=none` probes connect without device auth material.
  - file: [src/gateway/probe.ts:157](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/probe.ts#L157)
  - command: `nl -ba src/gateway/probe.ts | sed -n '148,167p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Probe client defaults to operator role:** `GatewayClient` uses `operator` when no explicit role is provided, which is the path `probeGateway()` takes.
  - file: [src/gateway/client.ts:447](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/client.ts#L447)
  - command: `nl -ba src/gateway/client.ts | sed -n '438,452p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **`mode=none` does not count as shared auth for device-skip logic:** `resolveConnectAuthState()` marks `sharedAuthOk` only for token/password or trusted-proxy success. `auth.mode=none` returns `authOk`, but not `sharedAuthOk`, so it does not unlock the operator device-identity bypass.
  - file: [src/gateway/server/ws-connection/auth-context.ts:114](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/server/ws-connection/auth-context.ts#L114)

  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Missing-device policy still rejects this path:** `evaluateMissingDeviceIdentity()` only allows device-less operator connects for Control UI bypass/trusted-proxy cases or when `roleCanSkipDeviceIdentity()` succeeds; otherwise it returns `reject-device-required`.
  - file: [src/gateway/server/ws-connection/connect-policy.ts:129](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/server/ws-connection/connect-policy.ts#L129)

  - file: [docs/gateway/protocol.md:513](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/docs/gateway/protocol.md#L513)

## Risks / Open Questions

- I did not run a live reproduction because the review is read-only and the local environment does not have `pnpm` available, so this conclusion is from current code/docs inspection rather than an executed runtime trace.
- The exact cascading symptoms in the report depend on higher-level callers, but the underlying `1008 device identity required` probe failure remains plausible on current `main`.
- Regular backend gateway clients are less affected because `GatewayClient` attaches a device identity by default; the evidence here is specific to probe-style clients that intentionally omit it on unauthenticated loopback.
