## Summary

Keep #44786 open. Current main still does not implement provider-level `contextWindow` as a default override: the provider config type and schema do not expose provider-level `contextWindow`, and model resolution still chooses per-model or discovered metadata values. The related exact-fix PR #44955 remains open/unmerged, so the reported bug is still actionable in-repo.

## Evidence

- **Provider config type lacks provider-level contextWindow:** `InlineProviderConfig` includes `baseUrl`, `api`, `models`, headers/auth/request fields, but no provider-level `contextWindow`, so the resolver cannot read the requested override from this typed path.
  - file: [src/agents/pi-embedded-runner/model.inline-provider.ts:19](https://github.com/openclaw/openclaw/blob/b1b29a8fc210a0772eb3728e03783890586e07f8/src/agents/pi-embedded-runner/model.inline-provider.ts#L19)

  - sha: [b1b29a8fc210](https://github.com/openclaw/openclaw/commit/b1b29a8fc210a0772eb3728e03783890586e07f8)
- **Config schema lacks provider-level contextWindow:** `ModelProviderSchema` accepts `baseUrl`, auth/API/header/request fields, and `models`, while `contextWindow` is only defined on `ModelDefinitionSchema`; a top-level provider `contextWindow` is not part of the current schema.
  - file: [src/config/zod-schema.core.ts:347](https://github.com/openclaw/openclaw/blob/b1b29a8fc210a0772eb3728e03783890586e07f8/src/config/zod-schema.core.ts#L347)

- **Discovered model merge still ignores provider-level contextWindow:** When a discovered registry model is merged with config, `contextWindow` is resolved as `metadataOverrideModel?.contextWindow ?? discoveredModel.contextWindow`; there is no `providerConfig.contextWindow` precedence between per-model config and discovered metadata.
  - file: [src/agents/pi-embedded-runner/model.ts:396](https://github.com/openclaw/openclaw/blob/b1b29a8fc210a0772eb3728e03783890586e07f8/src/agents/pi-embedded-runner/model.ts#L396)
