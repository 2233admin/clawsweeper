## Summary

Keep PR #69675 open. Current main has not incorporated the requested ACP follow-up orchestration note or standalone doc, and the related open PR #34542 is about broader thread-bound fallback guidance rather than this parent-rewrite/verbatim-relay distinction. However, the PR's proposed semantics are not just already-shipped docs cleanup: current main docs and runtime still describe and implement bound ACP follow-ups as direct messages to the ACP session, while the PR body explicitly says the deeper runtime behavior change is future work. That needs maintainer/product review rather than conservative auto-close.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to choose the product contract first: if parent-orchestrated rewriting should be the default for thread-bound ACP follow-ups, implement that behavior in the session-continuation/sessions_send path and add regression coverage, then update docs and the accepted-note copy. If the current direct-bound-session model should remain, narrow the PR to wording that clarifies explicit relay without contradicting docs/tools/acp-agents.md, and consider folding the guidance into the existing ACP Agents page instead of adding an orphan standalone page.
