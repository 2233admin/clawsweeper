## Summary

Keep open. Current `main` already addresses part of the motivation behind this RFC with canonical session-key handling, configurable `session.mainKey`/`dmScope`, session-binding infrastructure, a shipped `sessions.create` RPC, and plugin-based shared memory. But the RFC itself is not implemented end-to-end: the session store type still lacks archive/tag/channel-binding metadata, the gateway method surface has no `sessions.archive`/`sessions.pin`, and the Control UI sessions screen is still a list/patch/delete surface rather than create/archive/restore/pin management.

## Evidence

- **Per-channel DM isolation already exists on main:** `buildAgentPeerSessionKey` supports `dmScope` values including `per-peer`, `per-channel-peer`, and `per-account-channel-peer`, and the routing tests assert that `per-channel-peer` yields a channel-scoped direct-message session key such as `agent:main:whatsapp:direct:+15551234567`.
  - file: [src/routing/session-key.ts:130](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/routing/session-key.ts#L130)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Configured main-session canonicalization is already implemented:** `canonicalizeMainSessionAlias` explicitly maps legacy hardcoded `agent:main:<mainKey>` aliases back to the configured agent’s canonical main session, and the round-trip tests assert that the write path matches `resolveMainSessionKey(cfg)`. That makes the RFC body’s `session.mainKey` complaint at least partly obsolete on current `main`.
  - file: [src/config/sessions/main-session.ts:47](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/config/sessions/main-session.ts#L47)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Gateway already ships session creation:** `sessions.create` is a first-class gateway method, and the server test covers creating a dashboard session with label/model/parent linkage and a transcript file. So one piece of the RFC is already present, even though the Control UI does not expose the broader management flow requested here.
  - file: [src/gateway/server-methods/sessions.ts:797](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/gateway/server-methods/sessions.ts#L797)

- **Control UI sessions page does not implement the RFC management UX:** The Sessions view renders refresh/filter/search, per-session patch controls, compaction checkpoint actions, and bulk delete. It does not expose a New button, Active/Archived/All tabs, archive/restore actions, or pinning.
  - file: [ui/src/ui/views/sessions.ts:247](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/ui/src/ui/views/sessions.ts#L247)

That is not the RFC’s requested archived-session state with visible restore/pin support.
  - file: [ui/src/ui/controllers/sessions.ts:242](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/ui/src/ui/controllers/sessions.ts#L242)

A repo-wide search for `sessions.(archive|pin)` only found unrelated test fixture strings and no OpenClaw gateway/UI implementation.
  - file: [src/gateway/server-methods-list.ts:77](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/gateway/server-methods-list.ts#L77)

## Risks / Open Questions

- The RFC mixes several concerns: some routing/main-session complaints are already partly fixed on `main`, while archive/pin/UI/session-architecture work is still unshipped. That makes the item easy to misread as either 'already done' or 'still entirely unstarted'.
- Because `sessions.create` exists at the RPC layer but is not exposed as the requested Control UI workflow, maintainers may want follow-up work to be split into narrower UI/API/architecture issues instead of treating this RFC as resolved.
