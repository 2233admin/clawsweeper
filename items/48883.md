## Summary

Keep open. Current `main` does not already implement this change, and the proposed PR still appears to carry at least two concrete risks: it removes the cron `deliverToUser`/no-relay behavior that current heartbeat code and tests still depend on, and its new main-session text validation would likely reject benign edits on pre-existing main-session jobs because `applyJobPatch()` validates the full job after unrelated patches.

## Evidence

- **Current main still uses relay-style cron prompts plus an internal-only branch:** `buildCronEventPrompt()` on current `main` still embeds reminder prose and still switches on `deliverToUser`, including the internal-only `Handle this reminder internally` path.
  - file: [src/infra/heartbeat-events-filter.ts:7](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/infra/heartbeat-events-filter.ts#L7)
  - command: `nl -ba src/infra/heartbeat-events-filter.ts | sed -n '1,80p'`
- **Heartbeat runner still depends on the cron deliver/no-relay gate:** `buildHeartbeatPrompt()` still passes `params.canRelayToUser` into `buildCronEventPrompt()`, and the test suite still expects the cron prompt to stay internal-only when heartbeat target is `none`. That means the PR's `_opts` no-op change would be a behavioral regression, not dead-code cleanup.
  - file: [src/infra/heartbeat-runner.ts:717](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/infra/heartbeat-runner.ts#L717)

- **Main-session reminder prose is still part of the current public contract:** Current tool guidance and tests still describe `sessionTarget="main"` `systemEvent` payloads as freeform text, and `contextMessages` currently appends `Recent context:` onto `systemEvent` reminder text. The docs quick start also still recommends `--session main --system-event "Reminder: ..."`. That means this PR is not already implemented on `main` and would need wider contract/docs alignment before cleanup-closing.
