## Summary

Keep open. Current main still allows the reported Telegram transport shape to arm sticky IPv4 fallback: the fallback classifier can qualify `fetch failed` errors through `ETIMEDOUT` or `UND_ERR_SOCKET`, the fallback policy still forces IPv4, and the resolved fetch still sticks to the successful fallback dispatcher. I did not find a loopback-IPv4 `ECONNREFUSED` plus IPv6 `verbatim` regression test or exception on main.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin transport layer. Add a focused regression test for an aggregate `fetch failed` containing loopback IPv4 `ECONNREFUSED` and IPv6 timeout under `dnsResultOrder: "verbatim"`, then adjust `extensions/telegram/src/fetch.ts` so intentionally disabled loopback IPv4 does not arm or preserve sticky IPv4 fallback when IPv6 is configured or known healthy. The fix should explicitly decide, with tests, whether `UND_ERR_SOCKET` should still trigger sticky fallback.
