## Summary

Keep PR #69413 open. Current main still hydrates Signal mention placeholders into @uuid/@phone text, then applies only configured/display-name regex mention matching; it does not compare structured Signal mention metadata against the bot account phone or UUID. The PR is a focused fix for a real remaining Signal channel gap, and the provided review context shows the author addressed the span-validation concern with a follow-up commit.

## Best Possible Solution

Keep #69413 open for maintainer review. The best path is to merge or adapt the PR's Signal-owned fix so the Signal handler treats structured dataMessage.mentions entries matching deps.accountUuid or deps.account as native bot mentions, with the span validation from the follow-up commit and regression tests for UUID, phone, and non-bot mentions. Consider a follow-up for canDetectMention when native Signal identity detection is available without text mentionPatterns.
