## Summary

Keep open. Current main has useful visible-history defenses for heartbeat prompts, runtime context envelopes, tagged active-memory context, directive tags, tool-call XML, and silent control replies, but it still does not handle the reported plain-text `## Memory system - ACTION REQUIRED` / auto-recall `memory_search` instruction block at the `chat.history` projection boundary. The prior issue discussion remains valid: this is a customer-visible transcript filtering/provenance problem, not a WebChat-only renderer issue.

## Best Possible Solution

Keep this issue open. The best maintainer path is to ensure memory and heartbeat/system-injection content is structurally internal before it can enter visible history, or extend `projectChatDisplayMessages` and session-history SSE projection with narrowly scoped handling for known internal memory action blocks. Add regression coverage for string and text-block content covering the reported memory action block, heartbeat prompt with workspace-file suffix, legitimate user text that merely mentions those phrases, `chat.history`, and session-history SSE.

## Risks / Open Questions

- A literal string filter for phrases like `## Memory system` could hide legitimate user-authored text; structural internal provenance or display-hidden message metadata is safer.
- External memory plugins may use different wording, so exact phrase matching would remain incomplete unless paired with a documented plugin/API path for hidden prompt context.
