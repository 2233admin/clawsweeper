## Summary

Keep #66792 open. Current main has an Active Memory hard-timeout wrapper and timeout-budget docs, but it does not implement or prove the requested lightweight runEmbeddedPiAgent bootstrap optimization. The fresh v2026.4.24 comment reports the symptom still persists, and current Active Memory still goes through the normal embedded runner setup path before prompt/model start.

## Best Possible Solution

Keep the issue open and narrow it to core embedded-runner cold-path profiling and optimization for blocking plugins. Preserve the shipped Active Memory hard-timeout guard, add stage timing around lane queueing, auth/profile setup, context-engine resolution, session locking, bootstrap/context resolution, tool/runtime materialization, system prompt/session setup, and prompt start, then cache or bypass safe per-run invariants for bootstrapContextMode: "lightweight". Update the Active Memory docs with measured latency budgets once the core path is profiled.
