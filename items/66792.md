## Summary

Keep open. Current main still runs Active Memory as a blocking `before_prompt_build` embedded-agent call, and the code shows timeout/fail-open mitigation plus adjacent startup/cache/lock improvements rather than a proven fast `bootstrapContextMode: "lightweight"` bootstrap path. The issue also has fresh v2026.4.24 evidence that the symptom persisted after the original report.

## Best Possible Solution

Keep this issue open as a focused Active Memory embedded-runner cold-path profiling task. Add stage timings around lane wait, runtime plugin load, hook/model/harness/auth-profile resolution, context-engine initialization and assembly, tool registry materialization, session manager/resource loader/session creation, and prompt start. Then cache or bypass safe per-run invariants for lightweight blocking runs, preserve Active Memory's hard timeout/fail-open behavior, and update the public Active Memory docs with measured timeout and latency guidance once the root cause is fixed.
