## Summary

Keep open. Current main still has session-memory indexing and QMD session export pieces, but the reporter's exact QMD configuration remains insufficient: `agents.defaults.memorySearch.experimental.sessionMemory` plus `sources: ["memory", "sessions"]` enables the builtin memory-search session source, while QMD transcript export is still controlled separately by `memory.qmd.sessions.enabled`. Even when session hits exist, `memory_search` filters them through session-tool visibility, whose default `tree` policy will hide same-agent gateway/DM sessions outside the current session tree.

## Best Possible Solution

Keep this issue open. The maintainer path is to either bridge `agents.defaults.memorySearch.experimental.sessionMemory` plus `sources: ["sessions"]` into QMD transcript export when `memory.backend` is `"qmd"`, or explicitly document and diagnose that QMD transcript recall requires both `memory.qmd.sessions.enabled: true` and a wider session visibility setting such as `tools.sessions.visibility: "agent"` or `"all"` for same-agent gateway-to-DM recall. Add a regression test with a gateway-style `agent:<id>:paperclip:issue:<uuid>` session queried from a DM key such as `agent:<id>:whatsapp:direct:*`.

## Risks / Open Questions

- A fix must preserve conservative transcript visibility defaults; automatically surfacing all session transcripts through `memory_search` could expose unrelated channel, user, or cross-agent conversations.
- If maintainers intentionally want separate builtin and QMD session-memory switches, the safer path may be docs, doctor diagnostics, and runtime warnings rather than automatic config bridging.
- QMD also has `memory.qmd.scope`; any bridge or diagnostic should explain how that scope interacts with `tools.sessions.visibility` for gateway-dispatched sessions.
