## Summary

Keep open. Current main has a WhatsApp watchdog that can eventually reconnect when app-level silence continues, but the shipped/default path still waits far longer than the reported 10-17 minute message-loss window when transport frames keep arriving. The related open maintainer PR #72621 also explicitly says this related report should remain open unless its exact symptoms are proven fixed.

## Best Possible Solution

Keep this issue open as the WhatsApp-owned stale message-flow/message-loss bug. The best maintainer path is to add a configurable app-silence or message-flow stale threshold separate from transport activity, wire it through config types/schema/docs, and add a regression test where transport frames continue while inbound app messages stop within the reported 10-17 minute failure window. Coordinate with #63939 and the narrower open #72621 listener/auth-state work rather than closing this as already implemented.
