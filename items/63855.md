## Summary

Keep open. Current main has WhatsApp stale-listener recovery, but the reported gap remains: the WhatsApp watchdog is still a 30-minute inbound-silence timeout, it is not exposed as user config, and the generic gateway stale-socket monitor only acts on provider-proven transport activity that WhatsApp snapshots do not currently publish.

## Best Possible Solution

Keep #63855 open. The best fix is to make WhatsApp stale-socket recovery match the reported message-loss window: either expose a configurable WhatsApp message-flow timeout with a safer default around 10 minutes after maintainer product judgment, or publish provider-proven WhatsApp transport activity so the generic gateway health monitor can restart connected-but-dead sockets without relying only on app-level inbound messages. Add regression coverage for a connected WhatsApp listener that receives no inbound event for the 10-17 minute class of failure and verify status moves from healthy/listening to stale/reconnecting.
