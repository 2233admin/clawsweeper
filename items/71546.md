## Summary

Keep #71546 open. It is a fresh, current-version Discord ingest-lag report with concrete timing evidence, no protected-author/label blocker, and current main does not appear to ship the requested RESUME replay/catch-up behavior or a narrower canonical fix. The related PR #68159 is present in the changelog as a startup readiness fix, but the issue explicitly reports a post-login reconnect/RESUME delay after that change.

## Best Possible Solution

Keep this issue open and investigate in the bundled Discord plugin. The best path is to add targeted telemetry around Carbon RESUME, replayed dispatch delivery, listener registration lifetime, and inbound enqueue timestamps; then fix the responsible layer. If Carbon replay cannot be trusted, add a bounded opt-in Discord REST catch-up after successful RESUME using last seen message ids plus the existing inbound replay/dedupe guard, with focused lifecycle and inbound-worker tests.
