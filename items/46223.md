## Summary

Close as `clawhub`. Current `main` already has plugin-owned webhook ingress primitives and a bundled generic webhooks plugin, while the project vision explicitly keeps optional integrations out of core and prefers plugin/ClawHub delivery for this kind of platform-specific automation.

## Risks / Open Questions

- There is no bundled GitHub webhook plugin on `main` today, so closing this in core scope assumes maintainers want the feature to live as a plugin/ClawHub package instead of as a first-party bundled integration.
- If maintainers later decide GitHub issue-comment automation is a strategic default workflow, this scope call could change.

## Close Comment

Closing this as a Codex scope review. The idea is useful, but on current `main` it fits the plugin/ClawHub path rather than OpenClaw core. The project vision explicitly says optional capability should usually ship as plugins and that new skills should go to ClawHub first (`VISION.md:54-70`, `VISION.md:99-107`). The repo already provides the right extension seams for this: plugin-owned HTTP routes with `auth: "plugin"` for webhook/signature verification (`docs/plugins/architecture-internals.md:529-558`), the bundled generic Webhooks plugin for trusted external automation (`docs/plugins/webhooks.md:11-92`), and existing plugin-owned implementation with auth/rate-limit/body-guard handling (`extensions/webhooks/index.ts:19-38`, `extensions/webhooks/src/http.ts:665-759`). A GitHub-specific `@claw` issue/comment bridge should be built and published as a plugin or ClawHub package on top of those surfaces, not merged into core gateway behavior.
