## Summary

Keep open. Current `main` still has the Feishu startup-blocking behavior this PR is trying to address, but the PR as proposed appears unsafe because it detaches the single-account `startAccount` task from the gateway lifecycle and would likely trigger restart-loop regressions.

## Evidence

- **Current main still serializes startup probes and waits for all monitors:** `monitorFeishuProvider()` still probes accounts one-by-one, then awaits `Promise.all(monitorPromises)`, so multi-account startup remains blocking on current `main`. The code comment explicitly says probes are sequential.
