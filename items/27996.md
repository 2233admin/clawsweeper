## Summary

Current `main` fixes the CLI/API edit path by recomputing `state.nextRunAtMs` when `patch.schedule` changes, but it still preserves a persisted future `nextRunAtMs` on reload/startup. Because the store loader merges runtime state from `jobs-state.json` and `recomputeNextRuns()` only recalculates when the stored value is missing or already due, out-of-band schedule edits on disk can still leave a stale future next run. That is not strong enough evidence to close the issue as implemented.
