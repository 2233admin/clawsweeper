## Summary

Keep open. Current main still requires `models.providers.<id>.models` at config validation time, so the PR's central bug is not implemented. The generated config schema also still marks `models` required, while the memory embedding runtime would use `models.providers.<provider>.baseUrl` if the provider entry survived validation. The linked bug #39589 remains open, and the related alternative PRs in the provided context are closed unmerged rather than shipped replacements.

## Best Possible Solution

Keep this PR or the linked bug #39589 open until main accepts credential/baseUrl-only `models.providers.<id>` entries. The best implementation is a small config-contract change that defaults omitted provider `models` to `[]`, keeps `ModelProviderConfig.models` as a parsed array, regenerates the generated config schema, and adds focused regression coverage for schema parsing plus the memory embedding provider `baseUrl` fallback with SSRF/DNS stubs.
