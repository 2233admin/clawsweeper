## Summary

Keep open. Current `main` still defaults real Slack thread replies to separate `:thread:<threadTs>` session keys, and the public Slack config/docs expose `thread.historyScope` and `thread.inheritParent` but no `thread.sessionMode`-style option to route thread replies onto the parent channel session. The only non-thread path I found is an internal runtime conversation-binding hook covered by tests, not a user-facing Slack setting or documented feature.

## Evidence

- **Slack routing still forks thread replies by default:** `resolveSlackRoutingContext(...)` only skips thread suffixing when a runtime conversation binding already exists; otherwise it calls `resolveThreadSessionKeys(...)` with the Slack thread id.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:83](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L83)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Thread helper still appends `:thread:` suffixes:** `resolveThreadSessionKeys(...)` defaults `useSuffix` to true and returns `${baseSessionKey}:thread:${normalizedThread}` whenever `threadId` is present.
  - file: [src/routing/session-key.ts:237](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/routing/session-key.ts#L237)

  - file: [src/config/types.slack.ts:92](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/config/types.slack.ts#L92)
