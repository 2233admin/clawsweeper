## Summary

Current main still leaves the concrete webchat/openclaw-tui outbound delivery hook gap open. TUI sends through the same gateway `chat.send` path as webchat, that path creates its own reply dispatcher without the `message_sending` before-delivery hook wiring, and there is no `message_sent` emission in the gateway chat/TUI surface. Some related hooks now have coverage elsewhere (`agent_end`, guarded transcript writes), but the central `message_sending`/`message_sent` outbound lifecycle problem remains. The related #12584 is closed unmerged, and #50126 remains an open umbrella rather than a shipped fix for this specific surface.

## Best Possible Solution

Keep the issue open and fix the gateway chat outbound path. The likely implementation is to put webchat/openclaw-tui replies through the same lifecycle observer as shared outbound delivery, or minimally add a `beforeDeliver`/post-delivery hook layer around the `chat.send` dispatcher with session/run/channel context and regression tests covering `message_sending` modification/cancel and `message_sent` success/failure for both webchat and openclaw-tui clients. Document any intentionally unsupported hook/channel combinations until the matrix is complete.
