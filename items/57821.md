## Summary

Current `main` already routes `anthropic-messages` models through OpenClaw's own boundary-aware Anthropic transport, and that transport defensively ignores unknown content-block and usage fields instead of depending on a strict upstream parser. The same implementation is present in the released `v2026.4.23` tree.

## Close Comment

Current `main` already appears to handle this case, so this Codex review recommends closing it as implemented.

- OpenClaw now routes `anthropic-messages` models through its own boundary-aware Anthropic transport instead of relying on the generic parser path: `src/agents/pi-embedded-runner/stream-resolution.ts:44`, `src/agents/provider-transport-stream.ts:13`.
- That transport initializes `output.content` before parsing and only keys off `content_block.type`; for `type: "text"` it ignores unrelated fields like an extra `id`, so the reported non-standard block shape does not hit a `push` on `undefined`: `src/agents/anthropic-transport-stream.ts:748`, `src/agents/anthropic-transport-stream.ts:808`.
- The same parser also ignores unrelated `usage` keys and only reads the known token counters: `src/agents/anthropic-transport-stream.ts:786`, `src/agents/anthropic-transport-stream.ts:973`.
- The released `v2026.4.23` tag already contains the same Anthropic transport/routing implementation reviewed here, so this is not just on unreleased `main`.
