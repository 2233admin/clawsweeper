## Summary

Keep this issue open. Current main still has generic exact-parameter tool-loop detection, polling/no-progress handling, ping-pong detection, and a global circuit breaker, but not the requested exec-side read-only command classifier, cwd/path normalization, line-window collapsing, same-file or same-file-pair thresholds, or pre-execution synthetic warning result. The linked #64426 remains a companion read-tool proposal, not a replacement for this exec-runtime request.

## Best Possible Solution

Keep this issue open. The best implementation path is a narrow core exec/tool-loop enhancement that classifies obvious read-only shell readbacks, normalizes cwd plus target file paths while collapsing line-window variants, tracks same-file and same-file-pair counts per run/session, emits a warning-only synthetic tool result before the next redundant read executes, logs structured trigger telemetry, and adds focused tests for sed -n, nl -ba | sed -n, cat, bounded rg, alternating file pairs, thresholds, and false-positive cases. Coordinate with #64426 without treating it as a replacement for the exec-runtime work.

## Risks / Open Questions

- A low-threshold exec reread detector can false-positive during legitimate debugging unless it is narrowly scoped to obvious read-only commands and starts warning-only.
- The hook point needs to avoid executing redundant commands while preserving host routing, sandbox/node/gateway behavior, approval handling, environment checks, preflight diagnostics, and structured logging.
- #64426 should be coordinated so first-class read-tool and exec readback guards share semantics where practical, but it does not supersede this exec-specific command-normalization work.
