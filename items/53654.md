## Summary

Keep this issue open. Current main still registers Discord message-create, reaction, thread, and optional presence listeners, but not Carbon messageUpdate/messageDelete listeners, and the inbound worker/dedupe path has no source Discord message-id to active run abort mapping needed for delete-to-cancel. Related #38505 remains open and covers only the edit half, so it does not supersede this broader request.

## Best Possible Solution

Implement this in the bundled Discord plugin. Add Carbon MessageUpdateListener and MessageDeleteListener handling in extensions/discord/src/monitor/listeners.ts, register them from extensions/discord/src/monitor/provider.startup.ts, route edits through the existing Discord preflight/message handling path with mention/content-change gating and edit-aware replay semantics, and maintain a precise Discord source-message-id to pending job or active run abort mapping so delete events cancel only matching active work. Coordinate the edit half with #38505, but keep this broader tracker open until both edit-to-reprocess and delete-to-cancel ship.

## Risks / Open Questions

- Carbon messageUpdate payloads may be partial, so an implementation likely needs hydration and content-change gating before reprocessing.
- Delete-to-cancel should cancel only the run triggered by the deleted Discord message; cancelling by channel/session alone could abort newer unrelated work.
- Edit replay needs an explicit dedupe/replay design so edited content is not dropped as an already-processed Discord message id.
