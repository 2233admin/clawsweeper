## Summary

Current main still lacks the requested immediate, channel-visible session-busy reply for ordinary messages sent while the same session has an active long-running run. The existing behavior serializes or queues follow-up work, exposes adjacent `/stop` and queue modes, and can keep typing indicators alive, but it does not return a scripted busy/waiting message with cancel/new-session guidance before model execution.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the generic active-session queue/runner path around resolveActiveRunQueueAction, resolvePreparedReplyQueueState, and runReplyAgent: add a configurable, channel-safe busy/status reply for normal messages that arrive during an active run, with optional `/stop` and `/new` guidance, while preserving queued follow-up behavior. Add focused runner tests plus at least one channel-level test proving an ordinary message sent during an active turn receives the busy/status reply and still preserves intended follow-up handling.

## Risks / Open Questions

- Existing `/stop`, `/new`, `/queue interrupt`, typing indicators, and the internal '[Queued messages while agent was busy]' follow-up prompt are adjacent features but do not satisfy the ordinary-message busy-reply UX requested here.
