## Summary

Keep open. Current main serializes same-session inbound messages through the command queue and exposes manual stop/reset controls, but it does not send the requested immediate user-facing "session busy / waiting" reply with cancel/new actions. Busy followups are queued silently, and any residual session-lock failure still falls back to generic error text rather than a dedicated busy-status path.

## Evidence

- **Per-session queueing is the current collision strategy:** The command queue docs say inbound auto-reply runs are serialized per session, the default queue mode for all surfaces is `collect`, and typing indicators fire on enqueue rather than a scripted busy reply.
  - file: [docs/concepts/queue.md:10](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/docs/concepts/queue.md#L10)

  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Default runtime behavior is `collect`:** `defaultQueueModeForChannel()` returns `collect`, so same-session followups are queued by default unless config overrides it.
  - file: [src/auto-reply/reply/queue/settings.ts:7](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/auto-reply/reply/queue/settings.ts#L7)

  - file: [src/auto-reply/reply/agent-runner.ts:1005](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/auto-reply/reply/agent-runner.ts#L1005)

## Risks / Open Questions

- The original report describes session-lock timeouts; on current main the more accurate maintainer framing is a queued-followup UX gap rather than a raw concurrency bug for normal auto-reply traffic.
- Some channels may still show typing or preview activity from the original run, so the remaining user-facing confusion is likely channel-dependent even though the requested explicit busy reply is still missing.
