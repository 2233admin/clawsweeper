## Summary

Keep #71744 open. Current main has useful fallback attempt data in `FallbackSummaryError` and structured `model_fallback_decision` diagnostic logs, and trajectory capture records per-run lifecycle events, but there is still no persisted `fallback_step`/runtime-event record for each fallback transition or chain exhaustion. The mixed-cause terminal failure path also intentionally hides the detailed fallback summary from the default operator-visible reply, so the reported observability gap remains actionable.

## Best Possible Solution

Keep the issue open and implement core-owned, sanitized fallback-step observability from the existing model fallback path. Emit a structured event when moving from one candidate to the next and when the chain is exhausted, including from/to provider and model, classified reason, truncated detail, chain position, next/final outcome, and run/session correlation. Add focused tests for primary failure followed by fallback failure, and ensure the event is available through the documented runtime-event/trajectory or agent-event surface without requiring verbose user-facing replies.

## Risks / Open Questions

- Adding fallback-step events must preserve current redaction/truncation behavior so provider error details do not leak secrets into support bundles, logs, or channel-visible surfaces.
- Maintainers should choose the intended sink explicitly: OpenClaw currently has trajectory events, agent lifecycle events, diagnostic logs, and plugin runtime event hooks, but no first-class `runtime_events.jsonl` writer in this checkout.
