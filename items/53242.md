## Summary

Keep open. Current main still applies a default 8,000-character `chat.history` text cap before the byte-budget checks, and the normal Control UI/WebChat history load does not pass a larger `maxChars`. The related config work provides a workaround, but it does not satisfy the requested default behavior. The same-author PR #53243 is also still an open implementation candidate with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review/land PR #53243 or an equivalent gateway fix. The best implementation should raise or derive the default chat-history text cap consistently with the byte budget, keep UTF-8-heavy messages readable instead of replacing normal content with placeholders, add regression coverage for long assistant text that fits the history budget, and align config help/generated docs with the shipped default.
