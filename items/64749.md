## Summary

Keep open. Current main still does not implement the proposed durable `worker-state.json` snapshot, and this PR remains an open implementation candidate rather than obsolete cleanup. It also mixes the task snapshot feature with unrelated `doctor --json` work and has material existing review feedback around weakening doctor gateway health validation.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to split or drop the unrelated doctor JSON work, restore the live gateway health/auth check if doctor JSON stays, rebase the task-registry observer on current main, and decide whether `worker-state.json` is a supported operator contract with focused restore/upsert/delete/default-wiring tests and docs.

## Risks / Open Questions

- The PR bundles unrelated `doctor --json` behavior with the worker-state feature, making review harder and increasing regression surface outside the stated task snapshot scope.
- Existing review feedback is correctness-significant: replacing the live gateway RPC health check can make doctor report healthy without proving gateway reachability or token/auth correctness.
- Security review: the provided changed-file list shows no workflow, dependency, lockfile, package-publishing, downloaded-artifact, or third-party execution changes. The security-sensitive concern is functional and privacy-related: weakening doctor health validation and introducing a durable task snapshot containing task labels, progress summaries, errors, owner/session/run identifiers that should be treated as state-dir-trusted operator data.
- `worker-state.json` would likely become an operator-facing contract, so maintainers should confirm filename, schema stability, deletion semantics, sanitization/privacy expectations, and docs before shipping.
