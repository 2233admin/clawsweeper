## Summary

Keep open. Current `main` still does not expose `dreaming.execution` in the public `memory-core` config schema, and the internal dreaming execution type only supports a single optional `model` override rather than a fallback chain, so the feature request is not implemented.

## Evidence

- **Public memory-core schema blocks extra dreaming keys:** The shipped `memory-core` manifest sets `dreaming.additionalProperties` to `false` and only allows `enabled`, `frequency`, `timezone`, `verboseLogging`, `storage`, and `phases`; there is no top-level `execution` field, so the requested config shape would be rejected by the public schema.
  - file: [extensions/memory-core/openclaw.plugin.json:21](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/extensions/memory-core/openclaw.plugin.json#L21)
  - command: `nl -ba extensions/memory-core/openclaw.plugin.json | sed -n '1,220p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Internal resolver has execution knobs, but no fallback list:** `MemoryDreamingExecutionConfig` includes `speed`, `thinking`, `budget`, optional `model`, `maxOutputTokens`, `temperature`, and `timeoutMs`. It does not define any `fallback` field or fallback-model array.
  - file: [src/memory-host-sdk/dreaming.ts:58](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/memory-host-sdk/dreaming.ts#L58)

- **Resolver reads internal execution config only:** The host resolver reads `dreaming.execution.defaults` plus per-phase `light.execution`, `deep.execution`, and `rem.execution`, which shows partial internal support exists, but that support is not aligned with the public manifest schema the user config must pass through.
