## Summary

Keep open. Current main has moved multipart audio filename handling into the shared helper and avoids the missing-filename crash, but it still uses platform `path.basename`, so Windows-style path-shaped filenames can still pass through unchanged on POSIX hosts. The PR should be updated to fix the shared helper rather than adding an OpenAI-only resolver.

## Best Possible Solution

Keep the PR open and ask for a rebase/update that patches `resolveAudioTranscriptionUploadFileName` in `src/media-understanding/shared.ts` to strip both POSIX and Windows path segments before AAC/M4A normalization, with focused regression tests for POSIX paths, Windows paths, missing or blank filenames, and AAC remapping. That fixes OpenAI-compatible audio plus the shared plugin SDK helper used by other STT providers.
