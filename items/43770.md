## Summary

Keep open, but request changes rather than merge as-is. The PR addresses a real config-validation failure, yet the proposed passthrough-only fix would accept top-level plugin-specific keys without delivering them to plugin validation or runtime config, turning an actionable startup/schema error into silent misconfiguration.

## Best Possible Solution

Request changes and keep the PR open for maintainer review. The best fix is to keep the core plugin entry shape explicit, add any documented entry-level fields such as apiKey/env deliberately if they are still part of the contract, and solve plugin-owned settings under plugins.entries.<id>.config with regression tests that prove values pass through base validation, plugin schema validation, normalization, and api.pluginConfig at runtime.

## Risks / Open Questions

- Merging as-is would likely convert a hard validation failure into silent ignored configuration for top-level plugin-specific keys.
- Arbitrary passthrough at plugins.entries.<id> would broaden the core config surface outside plugin-owned configSchema validation, which can obscure unsupported keys or misplaced credentials.
- There is a separate documented/core-schema mismatch around plugins.entries.<id>.apiKey and env: docs and metadata mention them, but the current Zod/type entry shape does not accept them. That should be handled deliberately, not by allowing every unknown key.
