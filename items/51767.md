## Summary

Keep this issue open. Current main still makes `cron.sessionRetention` a cron-run-only retention sweep, while default keep-mode subagent and tracked ACP/session spawns are not covered by an equivalent restart-safe, short-retention cleanup path. The closed implementation PR #51861 remains unmerged, and #52327 was closed as a duplicate rather than becoming the canonical tracker.

## Best Possible Solution

Keep this issue open and implement a restart-safe retention path for completed subagent and ACP session-store entries. The likely direction is either a generic ephemeral-session reaper or a documented subagent/ACP-specific retention setting that preserves active and intentionally persistent sessions, archives/deletes transcripts consistently, runs on startup/store-write paths, and adds focused tests for expired cron runs, completed subagent/ACP runs, running subagents, keep-mode/session-mode behavior, and transcript cleanup. PR #51861 is useful prior art but needs review against current main and the restart-safety details from #52327.

## Risks / Open Questions

- A fix must distinguish active, resumable, keep-mode, and intentionally persistent session-mode rows from completed ephemeral rows before deleting store entries or transcripts.
- General `session.maintenance` can prune or cap old rows with broad defaults, but it is a blunt all-session policy and does not satisfy the reported `cron.sessionRetention: "3h"` behavior for completed subagent/ACP sessions.
- The existing registry sweep may remove registry metadata before any future session-retention pass can use that metadata, so the implementation should rely on durable session-store markers such as `endedAt`, `abortedLastRun`, cleanup markers, or another documented terminal-state contract.
