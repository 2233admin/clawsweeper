## Summary

Keep open. Current `main` still has the outbound normalization gap the PR is targeting, but this PR's fix is not safe to close out conservatively because it applies a UI/user-message sanitizer that also strips leading timestamp-like text from any string, which can alter legitimate assistant replies.

## Evidence

- **Current outbound normalization still does not strip inbound metadata:** `createOutboundPayloadPlanEntry()` normalizes reply text with `formatBtwTextForExternalDelivery(...)` only; there is no inbound-metadata stripping in the delivery path on `main`.
  - file: [src/infra/outbound/payloads.ts:152](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/infra/outbound/payloads.ts#L152)
  - command: `nl -ba src/infra/outbound/payloads.ts | sed -n '120,180p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **The proposed helper is documented for display/user-message cleanup, not outbound assistant delivery:** `stripInboundMetadata()` is described as stripping OpenClaw-injected inbound metadata from user-role text before UI display, and it removes a leading timestamp prefix before doing any sentinel check.
  - file: [src/auto-reply/reply/strip-inbound-meta.ts:1](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/auto-reply/reply/strip-inbound-meta.ts#L1)
