## Summary

Keep #69401 open. Current main has useful adjacent hardening for untrusted metadata, history/context blocks, external-content wrappers, and a Discord-specific guild-message body wrapper, but the general inbound channel body path is not wrapped. Slack and Telegram still pass the raw inbound body as `BodyForAgent`, prompt assembly appends that raw body after the untrusted metadata prefix, and memory-session ingestion later strips metadata and indexes the remaining user text without channel provenance filtering.

## Best Possible Solution

Keep the issue open and implement a generic core/SDK path for channel-originated message bodies: wrap `BodyForAgent` and relevant history/thread bodies with nonce or escaped untrusted-content boundaries, preserve raw `CommandBody` for command detection, add channel `InputProvenance` such as `external_user` plus `sourceChannel`, and carry that provenance into memory/dreaming so retrieval/promotion can filter untrusted channel-derived content. Reuse or extend `wrapExternalContent` rather than inventing per-channel wrappers, then add regression tests for Slack, Telegram, at least one plugin channel, and session-memory ingestion.

## Risks / Open Questions

- The fix crosses shared prompt assembly, channel SDK contracts, command parsing, transcript storage, and memory ingestion, so it needs maintainer design review rather than automated closure.
- Existing partial defenses could create false confidence: untrusted metadata/history and Discord guild bodies are protected more strongly than common Slack/Telegram current-message bodies.
