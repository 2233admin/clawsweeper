## Summary

Current `main` still archives reset transcripts, but `sessions_history` only fronts `chat.history` for the current session entry and does not follow `.jsonl.reset.*` archives. A reset-archive fallback exists in the bundled session-memory hook, which reinforces that the discoverability gap reported in #56131 is still real rather than already fixed on `main`.

## Evidence

- **`sessions_history` has no archive-follow option:** The tool schema only accepts `sessionKey`, `limit`, and `includeTools`, and its execution path forwards only `sessionKey` and `limit` to `chat.history`; there is no `includeArchived`/`followResets` parameter or archive-chain lookup in the tool itself.
  - file: [src/agents/tools/sessions-history-tool.ts:26](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/tools/sessions-history-tool.ts#L26)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **`chat.history` reads only the active session entry:** Gateway `chat.history` resolves the current session entry for the requested session key and reads messages from that entry's `sessionId`/`sessionFile`. It does not walk prior fragments after a reset.
  - file: [src/gateway/server-methods/chat.ts:1891](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-methods/chat.ts#L1891)
