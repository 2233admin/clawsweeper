## Summary

Current `main` already prevents heartbeat/cron/exec system events from resetting stale sessions, which covers the reported main-session loss after QQ Bot reconnects.

## Evidence

## Close Comment

Codex review: current `main` already implements the relevant fix for this report. `initSessionState()` now treats `heartbeat`, `cron-event`, and `exec-event` as system events that must not trigger session resets or conversation-binding retargeting, and it bypasses stale-session expiry for those events (`src/auto-reply/reply/session.ts:230`, `src/auto-reply/reply/session.ts:431`). It also preserves the existing delivery route instead of letting synthetic heartbeat routing leak into the main session (`src/auto-reply/reply/session.ts:538`). There is dedicated regression coverage for stale heartbeat sessions and daily-reset mode in `src/auto-reply/reply/session.heartbeat-no-reset.test.ts:75` and `src/auto-reply/reply/session.heartbeat-no-reset.test.ts:126`. QQ Bot still re-identifies on gateway close code `4009` (`extensions/qqbot/src/engine/gateway/reconnect.ts:133`), but on current `main` that path no longer resets the main conversation session. Closing as implemented on `main` based on commit `4630ce3d9e2f089062ccd386b0ae241325961d1a`.
