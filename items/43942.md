## Summary

Current main still does not resolve the reported OpenRouter/Hunter Alpha failure mode. The issue discussion includes a concrete assistant message where `[[reply_to_current]]` and the visible answer arrive only inside a `type: "thinking"` block. Main now has a safer OpenRouter `reasoning_details.response.output_text` path and generic reasoning-only retry coverage for GPT/Gemini/Ollama-style cases, but Hunter Alpha remains reasoning-enabled, still receives OpenRouter proxy reasoning injection, and bare `reasoning`/`reasoning_content`/`reasoning_text` deltas are still stored as thinking blocks that visible-text extraction and reply directive parsing ignore.

## Best Possible Solution

Keep this issue open. Add a targeted regression for an OpenRouter/Hunter Alpha completions stream where the answer starts with `[[reply_to_current]]` but arrives only in a plain reasoning field, then fix the owner path conservatively: disable proxy reasoning injection for Hunter Alpha if that route cannot separate hidden reasoning from visible content, or add a narrowly scoped OpenRouter compatibility path that surfaces only proven answer text. The resulting visible text must go through normal reply directive parsing so webchat and Telegram receive a normal text reply with reply metadata.

## Risks / Open Questions

- A broad fallback that promotes every thinking-only block into visible text could leak genuine hidden reasoning. The fix should be scoped to the affected OpenRouter route or to provider-marked visible output.
- OpenRouter route behavior may change upstream, so regression coverage should mock the observed bare `reasoning`, `reasoning_content`, or `reasoning_text` shape rather than depending on live availability.
