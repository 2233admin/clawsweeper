## Summary

Current main implements the requested per-call LLM latency telemetry through the sanitized `model_call_started` / `model_call_ended` plugin hooks. `model_call_ended` carries per-call `durationMs` plus provider/model/call metadata, and the embedded runner wraps each provider stream call so these events exclude tool execution time.

## Best Possible Solution

Keep the current `model_call_started` / `model_call_ended` telemetry as the canonical per-provider-call latency surface. Leave broader per-iteration `llm_output` usage/content changes to the related #64268 discussion if maintainers still want that payload shape.

## Risks / Open Questions

- The field is exposed as `durationMs` on `model_call_ended`, not as `latencyMs` on `llm_output`; consumers should use the model-call hook for latency telemetry.
- Per-call token/content payload work is separate from latency and remains covered by the related open issue #64268.
