## Summary

Keep this PR open. Current main still uses the fixed 2-hour maintenance threshold for stale cron run locks while execution timeouts are resolved per job, so the PR targets a real remaining cron reliability gap. The current PR head should not merge as-is because its 2-hour ceiling can clear locks before valid explicit long-running job timeouts finish.

## Best Possible Solution

Keep the PR open as a real cron reliability candidate, but request changes before merge: rebase on current main, derive the stuck-lock threshold from `resolveCronJobTimeoutMs(job) + buffer` for normal positive timeouts, keep a conservative fallback for `undefined` no-timeout jobs, handle only timer-overflow-sized values with an explicit compatible policy, and add regression coverage for explicit long timeouts as well as short timeout, systemEvent, and no-timeout cases.
