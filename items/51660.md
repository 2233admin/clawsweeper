## Summary

Current `main` does not implement a configurable umask or file/directory mode setting for agent-created user-facing files. The only filesystem-tool config surface is `tools.fs.workspaceOnly`; host writes still use plain `fs.writeFile` with inherited process umask, while workspace-contained writes default new files to `0o600` or preserve an existing file's mode. This issue is recent and actionable, so it should remain open.

## Evidence

- **No config knob for file permissions:** The tool config type defines `tools.fs.workspaceOnly` only; there is no `umask`, `filePermissions`, `fileMode`, or `dirMode` setting in the filesystem tool config.
  - file: [src/config/types.tools.ts:296](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/config/types.tools.ts#L296)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Non-workspace host writes inherit process defaults:** `writeHostFile` creates parent directories and then calls `fs.writeFile(resolved, content, "utf-8")` with no explicit mode or umask handling. `createHostWriteOperations(..., { workspaceOnly: false })` uses that path for write/edit tool output outside the workspace.
  - file: [src/agents/pi-tools.read.ts:751](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/pi-tools.read.ts#L751)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Workspace-contained writes default new files to 0600:** The safe write path uses `const fileMode = params.mode ?? 0o600` when opening/creating files, so new workspace-contained files are created with owner-only permissions unless an existing file mode is being preserved.
  - file: [src/infra/fs-safe.ts:536](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/infra/fs-safe.ts#L536)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Pinned workspace writer preserves existing mode, otherwise falls back to 0600:** `resolvePinnedWriteTargetWithinRoot` initializes `mode = 0o600`, replaces it only when an existing target file already exists, and returns `mode: mode || 0o600` for the atomic write helper.
  - file: [src/infra/fs-safe.ts:885](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/infra/fs-safe.ts#L885)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Default behavior is still unrestricted path scope, not configurable permissions:** The current tests assert that `tools.fs.workspaceOnly` defaults to `false`; this is the only tested filesystem-tool default exposed in config. No tests or schema entries cover configurable output permissions.
  - file: [src/agents/tool-fs-policy.test.ts:8](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/tool-fs-policy.test.ts#L8)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Repository search found no user-facing umask/file-permission config:** A repo search for `filePermissions|umask|fileMode|dirMode` only found temp-dir/test references and the internal safe-write defaults above, not a user-facing config or docs entry for agent-created file modes.
