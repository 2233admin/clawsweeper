## Summary

Keep #70533 open. Current main has startup scoping that prevents out-of-scope plugin modules from being imported, but it still discovers bundled plugin directories and builds the manifest registry before applying `onlyPluginIds`/allowlist filtering. The related #48644 closure does not supersede this narrower discovery-time memory report.

## Best Possible Solution

Keep the issue open and implement scoped cold discovery: derive the startup/plugin activation scope before manifest materialization, pass that scope into bundled/installed manifest registry loading, filter candidate directories before `openclaw.plugin.json`/zod work where possible, include scope in the relevant cache keys, and add regression tests that an out-of-scope bundled plugin manifest is not read or validated under a restrictive startup/allowlist config.

## Risks / Open Questions

- The issue text says `tools.allow`, while the current plugin activation path is driven by `plugins.allow`/startup `onlyPluginIds`; maintainers should clarify whether the requested filter should derive from plugin activation state, tool-policy plugin ids, or both.
- A correct fix must preserve channel/provider/setup discovery that intentionally needs manifest metadata for plugins that are not explicitly enabled by plugin id.
