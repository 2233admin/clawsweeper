## Summary

Keep #70533 open. Current main narrows gateway startup to a computed plugin id set, but the loader still performs full bundled/global discovery and manifest registry construction before applying `onlyPluginIds` or plugin allowlist enablement. The central memory/startup concern remains plausible on current main even though related #48644 and #69250 addressed narrower startup/runtime-load issues.

## Best Possible Solution

Keep the issue open and implement a scoped discovery/manifest-registry path for gateway startup. The likely fix is to thread the computed startup plugin ids and/or `plugins.allow` through discovery before manifest parsing, preferably reusing the installed plugin registry/index where possible, while preserving full manifest reads for config schema, doctor, setup, registry refresh, and other control-plane commands that explicitly need all plugins. Treat root `tools.allow` as tool-surface policy, not a plugin trust policy, unless maintainers intentionally define a separate compatibility mapping.
