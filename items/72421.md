## Summary

Keep this issue open. Current main still maps Discord/current-session work through per-session command lanes, and the queue only frees an active lane when the task promise settles. There is restart-time lane reset and a session write-lock watchdog, but I did not find an in-process stale active-lane watchdog, auto-release policy, or `status --lanes` diagnostic surface that would resolve the reported symptom.

## Best Possible Solution

Keep this issue open as a core auto-reply/session-lane reliability bug. Add lane active-task metadata tied to run/session/LLM liveness, regression coverage for cron or heartbeat system-event completion on a Discord session key, and a read-only lane diagnostic such as `openclaw status --lanes`; add defensive auto-release only when it can distinguish stale bookkeeping from an actually active run.

## Risks / Open Questions

- A naive stale-lane auto-release could allow concurrent writes to the same session if the task is still genuinely active.
- The report is intermittent and lacks a tight unit repro, so the first fix should add run/lane liveness instrumentation or a focused regression around cron/heartbeat system-event completion before changing recovery policy.
