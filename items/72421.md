## Summary

Keep #72421 open. The report is fresh, concrete, and not covered by the supplied related items. Current main has related safeguards, including heartbeat busy-lane skips and embedded-attempt session-lock cleanup, but it still lacks a lane watchdog/auto-release path and `openclaw status --lanes`, and the per-session queue still depends on the active task promise settling to clear the lane.

## Best Possible Solution

Keep the issue open. Maintainers should audit the cron/heartbeat/system-event paths through `runHeartbeatOnce`, `getReplyFromConfig`, `runEmbeddedPiAgent`, and reply operation completion; add a regression for a completed/stale per-session lane that blocks a Discord channel; and consider a conservative gateway lane watchdog plus a read-only lane-state status surface. Coordinate with #64810/#64963 if the active-reply-run gap is confirmed as the same root cause, but do not close #72421 until the Discord stale-lane symptom is fixed or clearly superseded.
