## Summary

Keep open. Current main still copies state-dir `.env` values and env-backed auth profile values into the gateway install plan, and the macOS LaunchAgent writer serializes every planned environment entry into `EnvironmentVariables`. Runtime dotenv loading still does not override existing process env values, so the reported stale LaunchAgent precedence problem remains possible after rotating a key in `~/.openclaw/.env` and running `openclaw gateway restart`. The related fix PRs #53452 and #53462 were closed unmerged, and current regression tests still assert that `.env` API-key values are persisted into the service environment.

## Best Possible Solution

Keep this issue open and fix the macOS gateway install path. The likely direction is to persist only internal service metadata and explicitly supported service-level fields in LaunchAgent `EnvironmentVariables`, stop copying state-dir `.env` secrets and env-backed auth/profile values into the plist, add install-plan and LaunchAgent regression coverage proving user-managed API keys are excluded, and add repair guidance for stale already-installed plists.

## Risks / Open Questions

- Existing macOS LaunchAgent plists may already contain stale copied API keys, so the eventual fix should include repair, doctor, or reinstall guidance to remove old persisted values.
- A macOS fix should preserve the documented ability for daemon-managed gateways to read durable keys from `~/.openclaw/.env` at runtime, while avoiding install-time snapshots that shadow later edits.
- Linux systemd currently has separate `EnvironmentFile` behavior for state-dir `.env`; the fix should stay platform-specific enough not to regress that path.
