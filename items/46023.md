## Summary

Keep open. Current `main` still computes Telegram `mediaMaxBytes` in the bot setup path but does not thread that value into `deliverReplies`/`deliverMediaReply`, so outbound final media replies still load media without the configured byte cap.

## Evidence

- **Telegram computes a per-account media limit:** `mediaMaxBytes` is derived from `opts.mediaMaxMb ?? telegramCfg.mediaMaxMb ?? 100` in the current bot setup path.
  - file: [extensions/telegram/src/bot-core.ts:483](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/telegram/src/bot-core.ts#L483)
  - command: `nl -ba extensions/telegram/src/bot-core.ts | sed -n '478,490p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Reply delivery path has no mediaMaxBytes parameter:** `deliverMediaReply` accepts `mediaLocalRoots` and a `mediaLoader`, but no `mediaMaxBytes` or `maxBytes` field.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:248](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/telegram/src/bot/delivery.replies.ts#L248)
  - command: `nl -ba extensions/telegram/src/bot/delivery.replies.ts | sed -n '248,278p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Outbound final media loads omit maxBytes:** Inside `deliverMediaReply`, `loadWebMedia` is called through `buildOutboundMediaLoadOptions({ mediaLocalRoots: params.mediaLocalRoots })`, which passes local roots only and omits `maxBytes`.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:273](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/telegram/src/bot/delivery.replies.ts#L273)
  - command: `nl -ba extensions/telegram/src/bot/delivery.replies.ts | sed -n '248,278p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Caller still does not propagate the limit:** `deliverReplies` calls `deliverMediaReply` with `mediaLocalRoots` and other reply metadata, but does not pass any computed media byte limit.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:732](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/telegram/src/bot/delivery.replies.ts#L732)
  - command: `nl -ba extensions/telegram/src/bot/delivery.replies.ts | sed -n '726,748p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Direct Telegram send path already does the correct thing:** `send.ts` computes `mediaMaxBytes` and passes it into `buildOutboundMediaLoadOptions({ maxBytes: mediaMaxBytes, ... })`, showing the intended pattern exists elsewhere but not in final reply delivery.
  - file: [extensions/telegram/src/send.ts:633](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/telegram/src/send.ts#L633)
  - command: `nl -ba extensions/telegram/src/send.ts | sed -n '631,640p;815,824p'`

- **Current tests cover local roots but not maxBytes on reply delivery:** The existing reply-delivery test asserts `localRoots` forwarding only; there is no corresponding assertion for `maxBytes` propagation in this path.
  - file: [extensions/telegram/src/bot/delivery.test.ts:487](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/telegram/src/bot/delivery.test.ts#L487)
