## Summary

Keep open. Current main still has a byte-size-drop config write guard that can reject a large formatting-only shrink, and `models auth login` still updates `openclaw.json` after writing the auth profile. The issue's exact silent-success wording may be partly improved by current `CONFIG_WRITE_REJECTED` logging/throwing, but the reported central problem remains: PowerShell/BOM/verbose formatting can block the config-side auth profile persistence path.

## Best Possible Solution

Keep the issue open and fix the config write safety logic for this path: compare normalized parsed JSON or changed paths before treating a large byte shrink as destructive, handle UTF-8 BOM/PowerShell formatting as a formatting normalization case, and add a regression test that exercises a verbose/BOM `openclaw.json` through the `models auth login` config mutation path. Also consider sequencing or rollback so `auth-profiles.json` is not left partially updated when `openclaw.json` is rejected.

## Evidence

## Risks / Open Questions

- OAuth login can persist `auth-profiles.json` before the config-side `auth.profiles` metadata is committed, leaving confusing partial state when the config write is rejected.
- The current guard is useful for real clobber prevention, so the fix should preserve fail-closed behavior for semantic data loss while allowing formatting-only size shrink plus the intended auth metadata change.
