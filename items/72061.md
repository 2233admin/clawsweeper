## Summary

Current main should not auto-close this bug. The reload planner is source-config based, but the WebUI skills toggle still goes through `skills.update` -> `loadConfig()` -> `writeConfigFile()`, and the write notification can still pass `nextCfg` as `sourceConfig` instead of the persisted/source-shaped config when no runtime source snapshot is active. That matches the reported runtime-default/plugin-entry paths leaking into the reload diff.

## Best Possible Solution

Keep #72061 open and fix the write/reload contract. The most direct path is to have `writeConfigFile` notify reload listeners with the persisted/source-shaped config returned by the lower-level writer, or otherwise guarantee a runtime source snapshot is active before WebUI config writes. Add a regression test that runs the `skills.update`/`writeConfigFile` notification path with runtime-materialized defaults and `plugins.entries.*.config`, then asserts the reload diff contains only `skills.entries.<name>.enabled` and does not schedule `onRestart`.
