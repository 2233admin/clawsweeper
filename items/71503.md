## Summary

Keep open. Current main at 631552c554095b0c520376db9131dac0f5427f94 still uses the older LAN-address selection behavior: core gateway only prefers exact en0/eth0 before falling back to the first external IPv4, pairing setup picks the first private IPv4, and the bundled device-pair plugin also returns the first matching private IPv4. I did not find the PR’s default-route preference, bridge/CNI deprioritization, shared snapshot picker, or regression tests on main, so PR #71503 remains actionable even though linked issue #71493 is closed.

## Best Possible Solution

Keep PR #71503 open for maintainer review. The best path is to land or revise the targeted gateway LAN picker, pairing setup, and bundled device-pair consistency changes, with regression coverage for physical/default-route preference, Docker/bridge/CNI deprioritization, and fallback when bridge-style interfaces are the only candidates.
