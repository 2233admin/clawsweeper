## Summary

Keep open. Current `main` still contains the reported uncancelled `stop()` timeout, so this is not implemented on `main`, but the proposed PR is not safe to auto-close as-is because its timer cleanup patch would remove the only fallback kill path on reused `IMessageRpcClient` instances: `closed` is created once in the constructor and never reset on `start()`, so a second `stop()` would race against an already-resolved promise and return immediately.

## Evidence

- **Current main still has the uncancelled stop timer:** `IMessageRpcClient.stop()` still uses `Promise.race([this.closed, new Promise(setTimeout(..., 500))])` without storing or clearing the timeout handle. The same code is present on current `main` and in release `v2026.4.23`, so the reported leak is not already fixed on shipped code.

  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **PR review comment identifies a real lifecycle regression:** The PR's Codex review comment is substantive: if the timeout is cleared but `closed` remains a one-shot promise, then after one successful shutdown the next `stop()` resolves immediately from stale `this.closed`, cancelling the new fallback timer before it can `SIGTERM` a hung child.

- **The reused-closed-promise behavior is visible in current code:** `closed` is initialized once in the constructor, stored as `private readonly closed`, and only resolved from child `error`/`close` handlers. `start()` does not recreate it. That means repeated `start()`/`stop()` on the same instance would observe an already-resolved `closed` promise after the first shutdown.
