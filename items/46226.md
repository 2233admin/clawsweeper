## Summary

Current main already gives active local loopback gateway probes the full caller timeout, so the reported healthy-loopback 800ms false timeout path is not present.

## Risks / Open Questions

- The text UI still prints a single top-level `Probe budget: <overall>ms` line even though some non-primary targets keep shorter caps (`1500ms` remote, `2000ms` SSH, `800ms` inactive local loopback), so a broader budget-label UX issue may still exist outside the reported healthy active local loopback case.

## Close Comment

Closing from a Codex review. Current `main` already avoids the reported healthy local-loopback false timeout: `src/commands/gateway-status/helpers.ts:132` only keeps the `800ms` cap for inactive `localLoopback` targets, while active local loopback and explicit loopback URLs use the full caller budget, and `src/commands/gateway-status/probe-run.ts:124` passes that resolved budget into `probeGateway`. The active-loopback behavior is covered in `src/commands/gateway-status/helpers.test.ts:297`. I also checked the `v2026.4.22` tag and it already contains the same logic, so this is implemented in shipped code. If you can still reproduce this on a current build, please open a new issue with the exact config mode, version, and `openclaw gateway probe --json` output.
