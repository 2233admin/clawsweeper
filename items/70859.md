## Summary

Keep open. Current main has already moved setup and the ordinary status scan onto `buildPluginCompatibilitySnapshotNotices`, but PR #70859 still contains an unlanded `status --all` path change and an unresolved review concern that the snapshot path can suppress hook-derived compatibility notices. I did not find a canonical follow-up that cleanly tracks that remaining `status --all` question.

## Best Possible Solution

Keep the PR open for maintainer review or author revision. The best path is to narrow it to the remaining `status --all` performance question, add focused coverage proving snapshot-based compatibility notices still preserve `hook-only` and `legacy-before-agent-start` warnings, and drop unrelated timeout/TTS test changes. If maintainers decide `status --all` should remain a full diagnostics surface, close this PR manually as superseded by the shipped setup/normal-status metadata-path work.
