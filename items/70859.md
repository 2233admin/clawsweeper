## Summary

Keep open. Current main has already moved ordinary `status` and setup compatibility notices onto `buildPluginCompatibilitySnapshotNotices`, but `openclaw status --all` still calls the heavier `buildPluginCompatibilityNotices` diagnostics path. The PR therefore is not obsolete. It does need maintainer review/rework before landing because the unresolved review concern is grounded in current code: snapshot reports are built with `loadModules:false`, and the notice derivation still depends on populated `hooks`/`typedHooks` arrays for hook-only and legacy `before_agent_start` notices.

## Best Possible Solution

Keep the PR open, but require a narrower revision: preserve the already-shipped setup and ordinary status snapshot behavior, move `status --all` to a cold path only after compatibility notice derivation is correct for snapshot reports, and split or drop unrelated test/runtime changes. A good fix likely updates the snapshot-compatible notice source to use manifest/plugin-record metadata for hook-only and legacy-hook detection, then adds focused `status --all` regression coverage proving the full diagnostics loader is avoided without losing expected notices.
