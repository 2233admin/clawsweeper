## Summary

Keep open. Current `main` does not include the PR’s early-exit lock cleanup, and the gateway startup path still acquires the lock well before it registers shutdown handlers, so the reported stale-lock window remains plausible.

## Evidence

- **`acquireGatewayLock()` still has no early-exit cleanup on `main`:** Current `main` opens the lock file, writes the payload, and returns a handle whose only cleanup is `release()`. There is no `process.on("exit")` or other synchronous cleanup registered between acquisition and return.
  - file: [src/infra/gateway-lock.ts:274](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/infra/gateway-lock.ts#L274)
  - command: `nl -ba src/infra/gateway-lock.ts | sed -n '260,420p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Gateway startup still registers signal cleanup after lock acquisition:** `runGatewayLoop()` acquires the gateway lock at line 43, but only installs `SIGTERM`/`SIGINT`/`SIGUSR1` handlers at lines 253-255. That leaves the exact early-startup gap described in the PR body.
  - file: [src/cli/gateway-cli/run-loop.ts:43](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/cli/gateway-cli/run-loop.ts#L43)
  - command: `nl -ba src/cli/gateway-cli/run-loop.ts | sed -n '1,320p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **OpenClaw already uses an exit-handler pattern for a similar lock elsewhere:** `src/agents/session-write-lock.ts` registers an `exit` handler that synchronously releases held locks, which matches the approach proposed in this PR and underscores that `gateway-lock.ts` has not yet adopted the same protection.
  - file: [src/agents/session-write-lock.ts:264](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/session-write-lock.ts#L264)
