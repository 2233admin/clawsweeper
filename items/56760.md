## Summary

Current main has the right building blocks for isolated DM session keys and forwards `sessionKey` into agent hook contexts, but I did not find strong proof that `before_agent_start` and `agent_end` now fire end-to-end for per-channel-peer DM turns. The remaining `sessionKey` vs `runtimePolicySessionKey` split means the report is not safely closable from read-only inspection.

## Evidence

- **Telegram DM routing now carries scoped session keys:** Telegram conversation routing builds session keys from channel/account/peer identity with `dmScope`, and inbound Telegram context writes that routed key into `SessionKey`.
  - file: [extensions/telegram/src/conversation-route.ts:48](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/telegram/src/conversation-route.ts#L48)
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Inbound Telegram context uses routed session key:** The finalized inbound context sets `SessionKey: route.sessionKey`, so current Telegram inbound turns are intended to arrive with the routed peer session key rather than always collapsing to main.
  - file: [extensions/telegram/src/bot-message-context.session.ts:322](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/telegram/src/bot-message-context.session.ts#L322)
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Agent hook contexts use the run session key:** Embedded runner passes `params.sessionKey` into prompt-build / legacy `before_agent_start` hook context and into `agent_end` hook context.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2017](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/agents/pi-embedded-runner/run/attempt.ts#L2017)
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Auto-reply still has a separate runtime-policy session key path:** `resolveRuntimePolicySessionKey()` derives a per-direct-peer key when the active session key is a main-session alias, so current main still distinguishes between the run `sessionKey` and a derived peer-scoped runtime-policy key.
  - file: [src/auto-reply/reply/runtime-policy-session-key.ts:86](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/auto-reply/reply/runtime-policy-session-key.ts#L86)
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Current tests still model main-session runs with separate peer policy keys:** `agent-runner-direct-runtime-config.test.ts` explicitly sets `followupRun.run.sessionKey = "agent:main:main"` while using a per-peer `runtimePolicySessionKey`, which preserves the exact split that could leave hooks attached to the main session key.
  - file: [src/auto-reply/reply/agent-runner-direct-runtime-config.test.ts:175](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/auto-reply/reply/agent-runner-direct-runtime-config.test.ts#L175)
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **No direct regression found for this report:** Searching `before_agent_start` / `agent_end` tests under auto-reply, Telegram, and embedded runner only found unit coverage for hook helpers, not an end-to-end regression proving per-channel-peer DM turns fire those hooks.
  - command: `rg -n "before_agent_start|agent_end" src/auto-reply/reply extensions/telegram src/agents/pi-embedded-runner -g '*.test.ts'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)

## Risks / Open Questions

- If DM turns still execute with `sessionKey=agent:main:main` and only carry the peer key in `runtimePolicySessionKey`, plugin hook consumers that key off `ctx.sessionKey` can still miss per-channel-peer isolation.
- The issue comment says the behavior was inconsistent even under `dmScope: "main"`, which suggests there may be a second hook-delivery bug beyond routing alone.
- Because this was a read-only review with no live reproduction, closing would overstate the evidence.
