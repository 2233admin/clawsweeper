## Summary

Keep #38640 open. Current main still does not implement the PR's prompt-level LLM retry wrapper with exponential backoff: the runner calls `activeSession.prompt(...)` directly, while the existing behavior is SDK retry-wait capping, auth-profile/model failover, and unrelated channel/generic retry infrastructure. Related configurable LLM/provider retry work in #49376/#61443 is still open, not a landed supersession.

## Best Possible Solution

Keep #38640 open for maintainer judgment. The best path is to either explicitly supersede it with a chosen configurable LLM/provider retry design from #49376/#61443, or rework this PR at the selected runner/provider boundary with abortable backoff, shared retry/failover status parsing, clear interaction with SDK retries and model fallback, and focused tests for classification, retry-after parsing, cancellation, and no duplicate side effects.

## Risks / Open Questions

- Prompt-level retries can interact with provider SDK retries, auth-profile cooldowns, model fallback, cancellation, and token/tool side effects.
- The PR still has unresolved design/review concerns, especially aborting retry backoff promptly and sharing status/error classification with existing failover code.
- There is meaningful overlap with #49376/#61443, so maintainers may want one explicit canonical retry design rather than multiple independent retry loops.
