## Summary

Current `main` sends Discord outbound messages through the regular channel message API, and Discord's current docs say regular messages already parse user/role/everyone mentions by default when `allowed_mentions` is omitted. The PR's premise does not reproduce for this code path, and the proposed `parse: ["users"]` change would narrow existing mention behavior rather than fix it. ([docs.discord.com](https://docs.discord.com/developers/resources/message))

## Close Comment

Closing after Codex review. On current `main`, OpenClaw's Discord outbound path sends regular channel messages via `extensions/discord/src/send.shared.ts:349` and `extensions/discord/src/send.shared.ts:405`, not webhook/interaction sends. Discord's current Message Resource docs say regular messages already parse user, role, and everyone mentions by default when `allowed_mentions` is omitted, so the reported "mentions render as plain text unless we add `allowed_mentions`" behavior does not reproduce for this code path. ([docs.discord.com](https://docs.discord.com/developers/resources/message))

The proposed `allowed_mentions: { parse: ["users"] }` change would also narrow the existing default behavior for role and `@everyone`/`@here` mentions, and Discord documents that `SUPPRESS_NOTIFICATIONS` only suppresses push/desktop notifications rather than all mention badges, so applying explicit mention parsing unconditionally is not a safe cleanup. ([docs.discord.com](https://docs.discord.com/developers/resources/message))
