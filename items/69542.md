## Summary

Keep open. Current main still selects a single ACP runtime backend from session metadata or global config and only has the existing same-backend fresh-handle retry; it does not expose or honor `acp.fallbacks`. The PR is not implemented on main, not clearly duplicated or superseded, and the capability likely belongs in the ACP control-plane/config layer rather than ClawHub because failover crosses core session metadata, runtime-cache, and backend registry behavior. It should not merge as-is: the provided review discussion identifies multiple correctness issues and the PR also contains unrelated workflow/runtime-deps changes.

## Best Possible Solution

Keep the item open and ask for a narrower ACP failover revision. The best implementation belongs in the ACP control plane/config surface: add a documented ordered fallback backend config, derive candidates from resolved session metadata while preserving binding/agent/global precedence, fail over only before any streamed output or side-effectful work, close/evict old handles cleanly, include init-stage and turn-stage unavailable/rate-limit classifications, preserve observability/session-state bookkeeping, and add targeted regression tests. Split unrelated install-smoke and runtime-deps fixes into separate PRs.
