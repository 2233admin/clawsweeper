## Summary

Keep this issue open. Current main still has the CLI fallback retry gap described in the issue: fallback attempts can reuse a stored CLI backend session while submitting the prompt again, and the explicit orphaned trailing user repair remains limited to the embedded runner path.

## Best Possible Solution

Keep this issue open. The best fix is a CLI-owned fallback retry repair in `src/agents/command/attempt-execution.ts`, `src/agents/cli-runner.ts`, `src/agents/cli-runner/execute.ts`, and the Claude live-session helper where needed: when `isFallbackRetry` would reuse a stored CLI session after a rate-limit or timeout, either repair/branch the backend transcript where supported or deliberately clear and reseed a fresh CLI session instead of resuming and submitting the same prompt again. Add focused coverage in `src/agents/command/attempt-execution.cli.test.ts` or `src/agents/cli-runner.reliability.test.ts` for a resumed CLI session whose first attempt fails after prompt submission and whose fallback does not append a duplicate user turn.

## Risks / Open Questions

- A naive prompt-skip may lose the task if the backend cannot continue from an already-appended orphaned user turn; the fix should model failure after prompt submission, not only before process spawn.
- Different CLI backends have different resume semantics, so the repair likely needs a CLI-owned policy with backend-specific reset, reseed, or branch behavior where transcript repair is unavailable.
- I kept this read-only and did not run tests; the decision is based on current source, docs, test inspection, and the provided issue discussion.
