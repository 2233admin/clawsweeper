## Summary

Keep open. Current main still lacks `--profile-id` support for `openclaw models auth login`; the existing code only supports custom profile IDs for `paste-token`, while login persists whatever profile IDs provider auth methods return. The Codex OAuth identity fixes reduce the original overwrite problem by deriving email or stable JWT-based profile names, but they do not provide the requested operator-chosen OAuth alias feature.

## Best Possible Solution

Keep the issue open until a safe implementation lands. The likely product path is to add `--profile-id <id>` to `models auth login`, validate that the profile ID belongs to the selected provider and does not collide unexpectedly, plumb it through login options/provider auth persistence, reject ambiguous multi-profile remaps, and update CLI docs plus parser/auth persistence tests. #49315 appears to be the current candidate PR to re-evaluate if it lands or is replaced.
