## Summary

Keep open. Current `main` still decodes child-process stdout/stderr with per-chunk `chunk.toString()`, which is exactly the UTF-8 boundary bug described in the PR. A simple Node reproduction shows that split multibyte characters decode as `�` under the current approach, while `StringDecoder` preserves the character. This is not implemented on `main`, and the PR is addressing a real bug rather than obsolete behavior.

## Evidence

- **Current adapter still uses boundary-unsafe decoding:** `createChildAdapter` registers `stdout`/`stderr` listeners that call `chunk.toString()` directly on each `data` event, with no stateful UTF-8 decoder and no flush on `end`. That matches the corruption mechanism described in the PR.
