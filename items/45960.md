## Summary

Keep open. Current `main` still uses a fire-and-forget `wsClient.start()` path in Feishu transport, so this is not implemented on `main` and there is no strong basis to close as obsolete. At the same time, the PR as proposed is not ready for cleanup-close either, because its patch imports a core `src/**` module from bundled plugin code, which violates the extension boundary and is risky for packaged installs.

## Evidence

- **Current main still drops async `start()` failures:** `monitorWebSocket` still calls `void wsClient.start({ eventDispatcher })` inside a synchronous `try/catch`. That only catches synchronous throws; it does not handle promise rejection from an async `start()` path, which is the exact bug the PR claims to fix.

## Risks / Open Questions

- The reported Feishu websocket zombie/unhandled-rejection path still appears plausible on current `main` because async `wsClient.start()` rejection is not handled.
- Closing this PR as implemented or unreproducible would discard a likely real bug report without evidence that current `main` fixed it.
- Merging or closing in favor of the current PR patch would be premature because the proposed fix crosses the bundled-plugin boundary and likely needs a different seam, such as a plugin-safe runtime helper.
