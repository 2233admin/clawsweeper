## Summary

Keep #40760 open. Current `main` still has the user-visible gap the PR targets: reply setup uses the typing controller's fixed default TTL and does not budget typing for timeout/fallback cycles, and timeout-before-reply failures still fall through to generic failure copy. The PR is not obsolete or superseded by the adjacent configurable-TTL threads, but it should be revised before merge because the provided review discussion identifies concrete correctness issues in the proposed timeout classification and fallback budget math.

## Best Possible Solution

Keep the PR open, or replace it with a narrow updated patch. The best fix is to compute a bounded typing TTL from the same effective fallback candidate budget used by `runWithModelFallback`, including configured-primary append and the one transient retry cycle, then add explicit timeout-final messaging using a narrow timeout check such as `isTimeoutError`. Add focused tests for fallback budget sizing, true timeout copy, and non-timeout network errors such as `ECONNREFUSED`/`EAI_AGAIN` staying out of the timeout message. Adjacent configurable-TTL PRs can proceed separately because they do not fully supersede this timeout/fallback-budget behavior.

## Risks / Open Questions

- Closing this PR as stale or obsolete would drop a real remaining UX bug: typing can stop before long fallback/timeout runs finish, and timeout failures remain generic on current `main`.
- Merging the PR as submitted could mislabel bare socket or DNS connectivity failures as model timeouts unless it switches to a narrower timeout predicate such as `isTimeoutError` and adds regression coverage.
- Merging the PR as submitted could still under-budget typing TTL in model/channel override cases where `runWithModelFallback` appends the configured primary candidate.
