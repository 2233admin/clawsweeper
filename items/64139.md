## Summary

Keep #64139 open. Current main has related Gateway auth and exec improvements, but I did not find a high-confidence implementation that resolves the macOS LaunchAgent `crontab` SIGKILL report, and the CLI pairing follow-up remains only partially addressed: local backend `callGateway` paths can avoid stale paired-device scope baselines, but `openclaw cron` still runs as a CLI client and current pairing code intentionally requires manual approval for scope upgrades.

## Best Possible Solution

Keep the issue open and split the maintainer investigation into two concrete checks: reproduce `crontab -l` and `echo ... | crontab -` from a macOS LaunchAgent-backed current-main Gateway exec path with logs/diagnostics, then decide the Gateway cron CLI policy separately. For the CLI path, the likely product/code options are a documented token-only local CLI mode, method-scoped cron CLI credentials, or an explicit maintainer-approved pairing workflow change; any change must preserve the current scope-upgrade protection tested in `src/gateway/server.silent-scope-upgrade-reconnect.poc.test.ts`.
