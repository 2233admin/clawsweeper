## Summary

Keep this issue open. Current main still does not show a source, docs, tests, or history-backed fix for macOS `/usr/bin/crontab` failing through exec, and the related `openclaw cron` CLI path still intentionally uses paired CLI Gateway auth for admin-scoped cron mutations. Recent devices-CLI fallback work gives a terminal approval workaround, but it does not satisfy the requested same-host automatic cron authorization path or explain/fix raw system `crontab` SIGKILL/timeouts.

## Best Possible Solution

Keep this issue open. Reproduce `crontab -l` and `echo ... | crontab -` through OpenClaw exec on a current macOS LaunchAgent install, then either fix the exec environment/TTY/timeout handling or document a supported alternative. Separately make an explicit product/security decision for same-host CLI cron administration: keep manual admin scope upgrades with documented `openclaw devices list/approve`, add a terminal approval flow directly to cron errors, or add a narrowly scoped same-host token/device reuse path that preserves the Gateway trust boundary.
