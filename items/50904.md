## Summary

The request mixes two things: core model fallback, which OpenClaw already has, and prompt/task-based model routing, which current project scope pushes into plugins/skills or external routing layers rather than new core orchestration.

## Risks / Open Questions

- There is no bundled ClawHub/plugin implementation in this checkout, so closing on scope may disappoint users who wanted a first-party declarative `openclaw.json` router.
- If maintainers later decide task-based routing is core product direction, this classification would need to be revisited.

## Close Comment

Thanks for the suggestion. I reviewed current `main` in a Codex maintainer pass, and this looks better scoped to a plugin/ClawHub solution than new OpenClaw core config.

Current `main` already covers the fallback half in core: `docs/concepts/model-failover.md:10` documents built-in auth rotation plus ordered model fallback through `agents.defaults.model.fallbacks`. For prompt/task-based routing, OpenClaw already exposes a plugin seam that can inspect the prompt and attachments and override the provider/model before the run starts (`src/plugins/hook-before-agent-start.types.ts:7`). That lines up with the project scope in `VISION.md:54` and `VISION.md:99`, which keeps optional capability in plugins and explicitly avoids heavy new orchestration layers in core.

There are also existing routing-oriented provider paths today, like LiteLLM (`docs/providers/litellm.md:9`) and OpenRouter automatic routing via `openrouter/auto` (`docs/providers/openrouter.md:25`). With that combination, this request reads as useful, but better as a plugin or external routing layer rather than a new core `agents.routers.tasks` feature, so I’m closing this one on that basis.
