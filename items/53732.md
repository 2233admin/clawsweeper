## Summary

Keep this issue open. Current main still runs BOOT.md by embedding its contents verbatim in a normal agent turn, uses model fallback with the original prompt preserved, and only suppresses the final agent reply with deliver:false. I did not find a boot-run marker, fallback prompt sanitization, or message-tool echo guard that would prevent a fallback model from sending echoed BOOT.md text through the message tool.

## Best Possible Solution

Keep this issue open. Implement a narrow internal boot-run guard that tags BOOT.md executions, rejects or suppresses substantial literal echoes of the generated BOOT.md prompt before both final reply projection and message-tool delivery, and add deterministic regression tests for primary-model failure followed by fallback-model echo while preserving legitimate BOOT.md-requested outbound messages.

## Risks / Open Questions

- The reported symptom is model-dependent, so regression coverage should simulate a primary failure and fallback model echo instead of relying on live Gemini behavior.
- Disabling the message tool for every BOOT.md run would break documented BOOT.md startup notifications; the fix needs a narrow literal-echo guard or internal boot context.
- Current tests cover the boot prompt shape and fallback prompt preservation, but not suppression of fallback BOOT.md prompt echoes before message-tool delivery.
