## Summary

Keep open. Current main still does not honor `approvals.exec.targets[].accountId` in the Telegram native approval handler path. Explicit forwarded delivery preserves target `accountId`, but the per-account native approval runtimes still decide with `shouldHandleTelegramExecApprovalRequest(...)`; when an approval request has no Telegram turn-source account and no persisted Telegram session binding, the matcher returns true for every enabled/eligible Telegram account. The open PR #70568 appears to track the remaining work, but the fix is not on current main.

## Best Possible Solution

Keep #69916 open until the native Telegram approval routing is patched. The best fix is to filter native approval delivery by the explicit Telegram `approvals.exec.targets[].accountId` when that target unambiguously identifies one account, while preserving existing turn-source/session binding precedence and leaving ambiguous/missing target account configs unchanged. Add a multi-account regression where several Telegram accounts share the approver chat ID but only the configured target account returns true from `shouldHandleTelegramExecApprovalRequest(...)` and delivers the native prompt.
