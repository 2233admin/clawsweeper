## Summary

Keep #72278 open. Current main has useful low-level SecretRef batching, but it does not implement the requested product-level external-vault provider pattern: the core contract is still only env/file/exec, the pass cache only memoizes exact refs and file payloads, configure only captures generic exec commands/args, and the public 1Password/Vault examples still show one single-value exec command per secret with `id: "value"`. The related #17311 is relevant but not a clear canonical duplicate for this specific batching/memoization contract.

## Best Possible Solution

Keep #72278 open as core SecretRef product/design work. The best path is to make the supported external-secret pattern batchable by default: one provider represents a backend/account/context, SecretRef ids represent backend secret paths, a single startup pass batches through that provider, and resolver/backend code can memoize repeated container/item/entity/payload lookups in memory for that pass. Update `openclaw secrets configure`, apply/audit preflight, docs, and regression tests accordingly. If maintainers want backend adapters outside core, expose or document the missing secret-provider/plugin API first, then keep core docs and configure flows pointed at the batchable contract.
