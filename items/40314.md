## Summary

Keep this PR open. Current main still emits internal message:received and message:sent hooks without draining hook event.messages into delivered replies, the linked bug #39688 is still open, and the maintainer successor #41375 is open/unmerged with a narrower replyable-surface contract. This PR is a plausible implementation candidate but needs consolidation with the maintainer path and the unresolved loop/session-context review concerns before merge or closure.

## Best Possible Solution

Keep this PR or an explicit successor open until #39688 is resolved. Maintainers should consolidate it with #41375, decide whether message:sent is a replyable hook surface, implement a shared reply-drain helper only where a safe target exists, preserve session/media context, persist any hook-loop suppression through queue recovery if outbound replay can happen, and update regression tests plus https://docs.openclaw.ai/automation/hooks to match the final contract.

## Risks / Open Questions

- The PR's message:received hook reply path omits sessionKey, so hook replies can diverge from normal routed replies for agent-specific response prefixes unless that is intentional and documented.
- The PR's recursive message:sent reply delivery omits the originating session context, which is harmless for current text-only hook messages but is weaker if the hook reply surface expands.
- The proposed skipMessageHooks suppression is not represented in the current delivery queue payload/recovery contract, so queued or recovered hook replies could re-emit message:sent hooks and produce unexpected follow-up sends or loops.
- The intended message:sent reply contract remains unsettled because maintainer successor #41375 intentionally limits auto-delivery to replyable surfaces.
