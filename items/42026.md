## Summary

Keep this issue open. Current main has useful adjacent pieces, including multi-agent config/state isolation, hot config reload, in-process agent harness selection, runtime-plan policy plumbing, and Gateway node/operator roles. It still does not implement the requested split into a lightweight Gateway control plane plus separately registered per-agent compute runtimes, nor local-split/distributed deployment modes or per-agent runtime lifecycle CLI commands.

## Best Possible Solution

Keep this issue open for maintainer architecture review. The best path is an additive design that preserves embedded Gateway mode, defines a stable Gateway-to-agent-runtime protocol and runtime role/capability handshake, adds a local process split with per-agent start/stop/restart and health, then layers container/Kubernetes backends and stricter per-agent secret/resource isolation. The design should reuse the existing runtime plan, harness registry, config reload, session routing, sandbox, node auth lessons, and Gateway protocol versioning rather than treating them as unrelated systems.

## Evidence

TinyTb's comment usefully narrows the first valuable milestone to per-agent start/stop/restart without restarting the Gateway, which is also not present on current main.
- **Docs still describe a single embedded agent runtime:** The agent runtime concept page says OpenClaw runs a single embedded agent runtime, one agent process per Gateway, with its own workspace/bootstrap/session contract. ## Risks / Open Questions

- The RFC is broad enough that maintainers should probably split it into milestones: protocol boundary, local process split, per-agent lifecycle CLI, secret/resource isolation, then orchestration backends.
- Related #12429 remains relevant for Kubernetes worker pairing, but it is narrower infrastructure work and does not supersede this lower-level agent compute runtime split.
- Current hot reload, node transport, ACP/subagent paths, in-process harnesses, sandboxing, and runtime-plan policy plumbing are adjacent; a future design should reuse them deliberately instead of creating parallel control paths.
