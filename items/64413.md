## Summary

Keep open. Current main has delivery diagnostics, but it does not expose normalized delivery failure reasons across cron and messaging runs. Cron still has only coarse delivery statuses (`delivered`, `not-delivered`, `unknown`, `not-requested`) plus raw `deliveryError`, while task/messaging delivery uses a separate status vocabulary (`pending`, `delivered`, `session_queued`, `failed`, `parent_missing`, `not_applicable`). Channel-not-configured and unavailable cases still surface as raw error strings or logs rather than stable reason codes. The related #49190 remains open and is adjacent cron warning/status work, not a complete duplicate of this broader normalization request.

## Best Possible Solution

Keep this open and implement a shared delivery outcome/reason contract that cron runs, task/messaging delivery, Control UI, CLI JSON output, and docs all consume. A good maintainer path is to add stable reason codes at the delivery boundary, map existing outbound/channel errors into those codes, preserve raw error text as detail, and then update cron run logs, task records, UI filters/status cards, and troubleshooting docs together. #49190 can remain the narrower cron warning/status follow-up unless maintainers choose to merge it into this broader delivery semantics work.
