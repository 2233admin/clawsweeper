## Summary

Keep this issue open. Current main still exposes cron and message delivery outcomes through coarse status enums, booleans, and raw error strings; it does not provide the shared normalized delivery reason codes requested for cron runs, status/UI surfaces, JSON output, and messaging actions.

## Best Possible Solution

Keep this issue open. The best path is an additive normalized delivery outcome model shared across cron run history, message action results, background task delivery, Gateway protocol schemas, Control UI filters/cards, CLI JSON/text output, and docs. Preserve existing compatibility fields, add stable reason codes such as not_configured, channel_unavailable, skipped, attempted_failed, and rate_limited, and cover routing/config failures, outbound adapter failures, provider/rate-limit failures, best-effort skips, fallback delivery, and UI/API filtering in focused tests.

## Risks / Open Questions

- A fix should be additive because existing clients already consume status, deliveryStatus, delivered, deliveryError, and task deliveryStatus fields.
- Reason classification needs to preserve raw provider/plugin diagnostics while adding stable operator-facing reason codes.
- The change spans protocol schemas, cron run logs, task delivery state, message CLI/tool results, Control UI filters/cards, and docs, so partial implementation would likely create another inconsistent status surface.
