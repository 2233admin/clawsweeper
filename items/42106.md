## Summary

Keep open. Current `main` still matches the report: paragraph separators are consumed by `EmbeddedBlockChunker`, the remaining buffer has leading newlines stripped, and downstream block-reply delivery only carries plain text with no boundary metadata once a flush occurs. The only post-`v2026.4.22` related change in this area was `block-reply-pipeline` whitespace-insensitive dedupe, which does not preserve paragraph boundaries.

## Evidence

- **Chunker consumes paragraph separators and strips them from the remainder:** In `flushOnParagraph` mode, the emitted chunk stops before the paragraph break, then `start` advances by the matched separator length and the leftover buffer is normalized with `stripLeadingNewlines()`. That removes the blank-line boundary from both the emitted chunk and the next buffered fragment.
  - file: [src/agents/pi-embedded-block-chunker.ts:166](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/pi-embedded-block-chunker.ts#L166)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Existing test still encodes separator-dropping behavior:** The current test expects `"First paragraph.\n\nSecond paragraph."` to emit as one chunk and expects the remaining buffer to be exactly `"Third paragraph."`, not `"\n\nThird paragraph."`. That is the behavior described in the issue.
  - file: [src/agents/pi-embedded-block-chunker.test.ts:49](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/pi-embedded-block-chunker.test.ts#L49)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Coalescer only preserves separators while chunks stay in the same buffer:** `createBlockReplyCoalescer()` flushes a plain `ReplyPayload { text: bufferText, ... }` and resets state. The only separator restoration is `const nextText = bufferText ? `${bufferText}${joiner}${text}` : text`, which works only before a flush. After flush, no boundary metadata survives.
  - file: [src/auto-reply/reply/block-reply-coalescer.ts:57](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/auto-reply/reply/block-reply-coalescer.ts#L57)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Downstream payload normalization and dedupe are whitespace-insensitive:** `resolveSendableOutboundReplyParts()` trims text with `text.trim()`, and `BlockReplyPipeline.hasSentPayload()` compares streamed fragments after `text.replace(/\s+/g, "")`. Current main therefore still treats boundary whitespace as non-semantic downstream.
  - file: [src/plugin-sdk/reply-payload.ts:145](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/plugin-sdk/reply-payload.ts#L145)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **No chunker/coalescer fix landed after the latest release:** `git diff --stat v2026.4.22..HEAD -- src/agents/pi-embedded-block-chunker.ts src/agents/pi-embedded-block-chunker.test.ts src/auto-reply/reply/block-reply-coalescer.ts src/auto-reply/reply/block-reply-pipeline.ts src/plugin-sdk/reply-payload.ts src/auto-reply/reply/reply-delivery.test.ts` shows only `block-reply-pipeline.ts` and `reply-delivery.test.ts` changed; the chunker and coalescer implicated by the report are unchanged from the latest release.
