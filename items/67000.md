## Summary

Current main still does not implement warm standby or persistent embedded-agent reuse for active-memory. The issue discussion is current and corroborated by two separate user reports. On cache misses, active-memory still creates a fresh sub-agent session id/key and transcript file, calls `runEmbeddedPiAgent`, and the embedded runner still performs queueing, workspace/plugin/model/auth/hook setup per call. Existing `cacheTtlMs` only reuses identical recall results, not initialized embedded sessions. The broader #50963 is open, but it is not a clear canonical superseding item for this active-memory-specific latency problem.

## Best Possible Solution

Keep this issue open as active-memory embedded-agent warm reuse work. A maintainer-quality fix should likely live in the active-memory plugin as a bounded per-parent-session reuse pool, or in a generic embedded-runner/session API if multiple plugin-owned embedded agents need the same semantics. It should define TTL and invalidation, preserve isolation/security boundaries, and add focused tests proving repeated active-memory turns reuse the intended initialized state or skip the expensive setup path.

## Risks / Open Questions

- A warm reuse implementation needs explicit invalidation for config, model, auth, prompt, tools, transcript, and plugin runtime changes to avoid stale or cross-session state leaks.
- The broader #50963 may eventually become the canonical session-reuse tracker, but the available context does not show a maintainer decision that it supersedes this active-memory-specific issue.
- The local checkout lacks runnable `pnpm`, so this review used source, docs, tests, and provided GitHub context rather than executing tests.
