## Summary

Keep open. Current main still has the cron parser weakness this PR targets, and the PR's cron-side change is plausible. It should not be closed as cleanup, but it also should not be merged as-is because it bundles an unrelated `openclaw update` early return that changes post-update behavior.

## Best Possible Solution

Keep this PR open for maintainer review, but ask the author to split or remove the update-command change. Land the cron fix as a focused parser/call-site hardening patch with a regression test for malformed `at` schedules in cron list/show or persisted-row formatting. Treat the `openclaw update` no-op behavior as separate work tied to #69412 and review it against plugin sync, channel persistence, restart, and doctor semantics.
