## Summary

Keep this issue open. Current main still only treats an exact silent-token payload as silent; mixed reasoning/preamble text followed by `NO_REPLY` remains deliverable after token stripping or, for glued `here.NO_REPLY`, can remain visible. The issue also has open unmerged implementation candidates with closing syntax, so it should stay open until maintainers choose and land the intended silent-token policy.

## Best Possible Solution

Keep this issue open as the canonical tracker and review the active implementation PRs #66755 and #69394. Maintainers should decide the intended silent-token policy, add regression coverage for bare `think`, `<think>...</think>`, whitespace-delimited trailing tokens, and glued `here.NO_REPLY` cases, then close this issue only after the chosen fix lands.

## Risks / Open Questions

- Maintainers still need to choose the product policy: narrowly suppress reasoning-marker-wrapped silent replies, or broadly suppress any payload with a trailing silent token.
- A final-payload normalizer fix may not cover streaming/block delivery if reasoning text is sent before the final `NO_REPLY` token arrives.
- Broad trailing-token suppression would intentionally change behavior currently covered by the #30916 regression tests that preserve substantive text and strip only the token.
