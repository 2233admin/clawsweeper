## Summary

Keep open. Current `main` has atomic temp-file replacement and last-known-good recovery, but the config write path still does an unlocked read/merge/write sequence with no advisory lock or in-lock reread. That means the issue's requested fix is not implemented on `main`, and the recovery machinery only mitigates corruption after the fact.

## Evidence

- **Config writes are still unlocked read/merge/write:** `writeConfigFile` reads a snapshot, derives `persistCandidate`, then writes a temp file and renames it into place. There is no advisory file lock, no cross-process serialization, and no reread inside a critical section before commit.
  - file: [src/config/io.ts:1624](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/config/io.ts#L1624)

  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Atomic replace exists, but only as temp-file rename:** The low-level JSON/text helpers provide atomic temp-file replacement and an in-process async lock helper, but the config write path is not using a shared lock here. Atomic rename prevents partial writes; it does not serialize concurrent writers.
  - file: [src/infra/json-files.ts:38](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/infra/json-files.ts#L38)
