## Summary

Current `main` already provides the requested capability in practice: built-in automatic compaction with preserved recent context, configurable compaction-only model/provider and memory flush, plus a second provider-side compaction layer for OpenAI Responses and plugin-owned context-engine compaction hooks.

## Risks / Open Questions

- The implementation is distributed across core compaction, context engines, and provider integrations rather than a single feature named "dual-layer context compression".
- Provider-side server compaction is currently explicit for supported backends such as OpenAI Responses, not a universal gateway-side layer for every provider.

## Close Comment

Closing this as implemented on current `main` after a Codex review.

- OpenClaw already does automatic context compaction when sessions near the model limit or hit overflow, and it keeps recent turns intact while persisting a compaction summary for continuity (`docs/concepts/compaction.md:9`, `src/agents/pi-embedded-runner/run.ts:1094`).
- The core compaction path already exposes the knobs requested here: a separate compaction-only model, pluggable compaction providers, optional user notices, and pre-compaction memory flush (`docs/gateway/config-agents.md:532`).
- The safeguard compaction flow preserves recent turns and split-turn/tool-state suffix context, and it generates structured summary sections rather than a bare freeform blob (`src/agents/pi-hooks/compaction-safeguard.ts:781`).
- There is also a second compaction layer for supported providers: OpenAI Responses requests get automatic server-side `context_management` compaction with a default threshold derived from the model context window (`src/agents/openai-responses-payload-policy.ts:45`, `docs/providers/openai.md:671`).
- For more specialized structured-session behavior, current `main` already supports plugin-owned context engines with `assemble`, `compact`, and `afterTurn` lifecycle hooks, including engine-owned compaction (`docs/concepts/context-engine.md:67`).
