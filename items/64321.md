## Summary

Keep open. Current main partially mitigates the CLI path by listing from `sessions.json` metadata, but the Gateway `sessions.list` implementation still can synchronously read and parse full JSONL transcript files for rows missing fresh usage/context/cost metadata. The issue is recent, has a concrete 112MB/124-file report, and the author followed up that they were still seeing ~20s runtimes.

## Best Possible Solution

Keep the issue open and implement a fast default `sessions.list`: use persisted session-store metadata or a small metadata index, avoid full JSONL reads during default row construction, apply limit/active filters before any optional hydration, and move transcript-derived usage/title/message previews behind explicit bounded/lazy options with cache invalidation by transcript mtime/size. Add targeted tests covering metadata-only listing plus a large-transcript guard.

## Risks / Open Questions

- Removing the transcript fallback outright could regress UI/session usage fields for legacy sessions unless replaced with cached metadata, lazy hydration, or an explicit opt-in detail path.
- A robust fix should include a performance regression test or benchmark with many large transcripts, because the current unit tests validate correctness but not the large-history latency target.
