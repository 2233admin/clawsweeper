## Summary

Current `main` already applies a per-attachment Discord download timeout with abort/fallback handling, so the reported 30-minute inbound-worker block is addressed and the same implementation is present in release `v2026.4.23`.

## Evidence

- **Inbound processing now applies attachment-specific timeouts:** `processDiscordMessage` passes both `readIdleTimeoutMs` and `totalTimeoutMs` into Discord media resolution before continuing message handling, so attachment fetches no longer rely only on the 30-minute worker timeout.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:177](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/discord/src/monitor/message-handler.process.ts#L177)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.process.ts | sed -n '172,194p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Attachment timeout constants are far below the worker timeout:** Discord attachment reads now use a 60s idle timeout and 120s total timeout, while the worker timeout remains 30 minutes.
  - file: [extensions/discord/src/monitor/timeouts.ts:3](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/discord/src/monitor/timeouts.ts#L3)
  - command: `nl -ba extensions/discord/src/monitor/timeouts.ts | sed -n '1,12p'`

- **Each attachment fetch is raced against its own aborting timeout:** `fetchDiscordMedia` starts an `AbortController`, merges it with the worker abort signal, aborts the fetch on timeout, and rejects with `discord media download timed out ...` instead of letting the fetch hang until the worker-level 1800s timeout.
  - file: [extensions/discord/src/monitor/message-utils.ts:313](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/discord/src/monitor/message-utils.ts#L313)

## Close Comment

This looks fixed on current `main`, and the same implementation is already shipped in `v2026.4.23`.

- Codex review of `extensions/discord/src/monitor/message-handler.process.ts:177` shows Discord inbound processing now passes `readIdleTimeoutMs` and `totalTimeoutMs` into attachment resolution before the worker continues.
- `extensions/discord/src/monitor/timeouts.ts:3` sets attachment-specific limits to `60000` ms idle and `120000` ms total, while the separate inbound worker timeout remains `1800000` ms.
- `extensions/discord/src/monitor/message-utils.ts:313` now races each attachment/sticker download against its own aborting timeout and falls back instead of letting the worker sit blocked for the full 30 minutes.
- `extensions/discord/src/monitor/message-utils.test.ts:680` covers the reported shape directly by mocking a never-resolving attachment fetch and asserting the handler returns fallback media data after the shorter timeout.
- I also compared the relevant Discord implementation files between release `v2026.4.23` and current `main`; there is no code delta there, only a docs change, so this was already present in the shipped release.

If you still hit a 30-minute block on `v2026.4.23` or newer, please open a fresh issue with current version details and a log excerpt from the attachment fetch path so maintainers can check for a different regression.
