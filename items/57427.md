## Summary

Keep open. Current main still implements sandbox exec as an isolated environment: `.env`/host `process.env` values are loaded for the gateway, but sandbox exec env is built only from explicit sandbox env and per-call env. The OpenShell backend attaches configured provider names during `openshell sandbox create`, but it does not resolve or forward gateway `.env` secrets into the remote command/provider path. The single issue comment is a separate OpenShell runtime-status question, not a resolution or duplicate.

## Best Possible Solution

Keep the issue open and resolve it with an explicit sandbox/OpenShell env contract rather than silent host-env inheritance. A good fix would add a narrowly scoped, documented way to pass selected resolved env values into sandbox exec or OpenShell provider credentials, preserve secret filtering and precedence, clarify whether `sandbox.docker.env` is intentionally shared with non-Docker backends or should become a generic sandbox env surface, and add regression coverage for `.env`/config env, OpenShell `providers`, placeholder handling, and non-inheritance of unrelated secrets.

## Risks / Open Questions

- Forwarding all gateway `.env` values into remote sandboxes would conflict with the existing sandbox isolation model and needs an explicit security/product decision.
- The unresolved `openshell:resolve:env:*` placeholder may also require an upstream OpenShell change, but OpenClaw still has an open question about whether and how it should pass selected resolved env values to OpenShell-backed sandbox exec/provider flows.
