## Summary

Keep this PR open. Current main has changed the explicit restart path to kickstart-first behavior, so the PR needs rebase and rescoping, but the core requested recovery is not fully implemented: install/force activation and doctor repair still perform single raw `launchctl bootstrap` calls without an EIO-specific retry, and current tests do not cover transient `Bootstrap failed: 5: Input/output error` recovery.

## Best Possible Solution

Keep this PR open for maintainer review, require a rebase onto current main, and adapt the implementation around a shared EIO-only `launchctl bootstrap` retry helper used by `bootstrapLaunchAgentOrThrow`, doctor repair, and any current restart fallback/handoff bootstrap path. Decide separately whether install force-reinstall still needs old-PID waiting or escalation under the current activation flow; if retained, add the strongest available PID identity guard. Cover transient EIO retry, non-retry of GUI-domain errno 125 and permission errors, repair recovery, and any retained SIGKILL timing behavior with focused tests.

## Risks / Open Questions

- The PR is stale against current main: `restartLaunchAgent` has moved to kickstart-first behavior, so the patch should be rebased and scoped to the current install, repair, restart-fallback, and handoff surfaces rather than merged as-is.
- The SIGKILL escalation is security-sensitive. The PR's final liveness check reduces PID-reuse risk, but maintainers should prefer a stronger process-identity check if direct PID killing remains part of the solution.
- No local tests or live macOS launchd reproduction were run because this was a read-only review; after rebase, focused launchd unit tests and a macOS launchd integration run should validate behavior.
