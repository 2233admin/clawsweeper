## Summary

Keep this issue open. Current main has partially improved one-shot `capability model run`/`infer model run` behavior by using no-tools probe mode, but the normal OpenClaw agent runtime still treats dynamic OpenRouter models as tool-capable when OpenRouter metadata omits `tools`. The reported Sonar Deep Research path still needs a provider/runtime fix rather than cleanup closure.

## Best Possible Solution

Fix this at the OpenRouter provider/runtime boundary: preserve whether OpenRouter `supported_parameters` includes `tools`, map that into `compat.supportsTools` for dynamic OpenRouter models, define a conservative migration/default for old cache entries, and add regression coverage for `openrouter/perplexity/sonar-deep-research` proving normal agent runs no longer send tool schemas or fail late via provider fallback.

## Risks / Open Questions

- A fix needs an explicit policy for existing on-disk OpenRouter capability cache entries that predate any new `supportsTools` field; treating missing cache data as tool-capable would preserve this bug for cached users.
- Maintainers still need to choose runtime behavior for non-tool-capable agent models: strip optional tools when the run can proceed, or fail early with a clear compatibility error when OpenClaw tools are required.
- The `capability model run`/one-shot path appears to have moved to no-tools mode, so regression coverage should include a normal agent run path as well as the probe path.
