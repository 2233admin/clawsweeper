## Summary

Keep open. The issue and follow-up comment identify a real OpenRouter model-runtime capability mismatch: `perplexity/sonar-deep-research` is reported as present but not tool-capable, while OpenClaw still treats dynamic OpenRouter models as tool-capable unless `compat.supportsTools` is explicitly false. Current main has OpenRouter catalog metadata fetching, but that metadata only feeds name/input/reasoning/context/tokens/cost into the dynamic model and does not convert `supported_parameters` lacking `tools` into `compat.supportsTools: false`. Related Perplexity/OpenRouter web-search items are separate from this agent/model runtime tool gating path.

## Best Possible Solution

Keep the issue open and fix it in the OpenRouter provider/runtime path: extend OpenRouter model capabilities with a `supportsTools` value derived from `supported_parameters.includes("tools")`, apply `compat.supportsTools: false` for dynamic/configured OpenRouter models that lack tools, and add a regression test using a Sonar-style catalog fixture plus an agent/model-run assertion that no tool schema is sent or that a clear compatibility error is raised before fallback.
