## Summary

Keep #69327 open. The report is recent, non-maintainer-authored, and not protected, but current main does not provide strong close evidence. The issue includes concrete reproduction data plus a follow-up root-cause analysis pointing at the subagent sandbox inheritance path. Source inspection shows `sessions_spawn` still passes only group/workspace context into `spawnSubagentDirect`, while Docker env/setup is resolved afresh from the target agent config and no parent resolved sandbox Docker env/config is threaded through. Current code does have related protections for workspace inheritance, config-hash recreation, and `--agent` recreation matching subagent keys, but that is not enough to close the env propagation/stale runtime report as fixed without a regression test or live reproduction proof.

## Best Possible Solution

Keep the issue open and treat it as an actionable sandbox/subagent bug. The best maintainer path is to add a focused regression test for sandboxed `sessions_spawn` covering `sandbox.docker.env`, `setupCommand` after `openclaw sandbox recreate --agent`, and workspace freshness, then either thread the parent resolved sandbox Docker config for same-runtime `sandbox="inherit"` semantics or explicitly document/implement target-agent-only env behavior with clear user-facing diagnostics.
