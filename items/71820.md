## Summary

Keep PR #71820 open. Current main still only hydrates BlueBubbles inbound reply context from webhook metadata or the process-local reply cache; there is no shipped opt-in HTTP API fallback for cache misses. The related #43764/#65447 work is outbound Private API status refresh, not inbound reply-context recovery. The PR is a focused draft with tests and review comments that need fixups, not conservative-cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to revise the fallback to extend/reuse the guarded BlueBubbles client request path for `GET /api/v1/message/{guid}`, keep the behavior opt-in unless maintainers choose otherwise, preserve channel-level config inheritance when per-account configs omit the flag, and retain focused tests for cache misses, failures, dedupe, cache population, config inheritance, and SSRF policy.
