## Summary

Keep PR #71820 open. Current main still only hydrates BlueBubbles inbound reply context from webhook-provided fields or the in-memory reply cache; it has no API fallback for cache misses. The related #43764 work is already shipped but covers outbound Private API status-cache refresh, not inbound original-message lookup. The PR is focused and potentially useful, but the draft needs review fixes for SSRF policy threading and config inheritance before it could land.

## Best Possible Solution

Keep the PR open for maintainer review. The right path is to revise the BlueBubbles plugin implementation so the cache-miss original-message lookup reuses the existing guarded client/policy path, preserves channel-level config inheritance for multi-account setups, includes focused tests for those review comments, and updates the BlueBubbles configuration docs if the option is accepted.

## Risks / Open Questions

- The proposed helper must not bypass BlueBubbles' current always-guarded SSRF policy; current main centralizes that invariant in `BlueBubblesClient` and its tests.
- The config schema change should preserve channel-level defaults for account entries instead of materializing per-account `false` values that override a global opt-in.
- The fallback should add user-facing docs/config reference coverage if accepted, because the current docs do not expose this setting.
