## Summary

Keep PR #71820 open. Current main still has only webhook-provided reply metadata plus the process-local BlueBubbles reply cache for inbound reply context; it does not ship the PR's opt-in HTTP API fallback for cache misses. The related #43764/#65447 work is a different outbound Private API status refresh path, and the PR remains a focused bundled BlueBubbles behavior change with review comments to address rather than an obsolete or duplicate item.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to revise the BlueBubbles fallback so inbound cache misses can optionally fetch the original message through the existing guarded client request path, preserve channel-level defaults across account merges, keep the behavior opt-in unless maintainers choose otherwise, and retain focused tests for cache miss success/failure, dedupe, SSRF policy, and config inheritance.
