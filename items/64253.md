## Summary

Keep open. The issue is recent, non-maintainer-authored, and has concrete Windows/subagent timeout logs. Current main has resilience around subagent announce delivery, but I did not find a source-backed fix for the central symptom: an alive gateway becoming broadly unresponsive under sustained subagent/provider load. The exact 120s completion announce timeout path still exists and is covered by tests as a retry/give-up behavior, not as a gateway hang fix.

## Best Possible Solution

Keep the issue open for a targeted maintainer investigation. The likely product path is to reproduce on Windows with sustained subagent completion delivery, inspect gateway event-loop/RPC responsiveness and provider timeout interaction, then either add a concrete fix in the subagent announce/gateway client/command lane path or link a narrower canonical issue once root cause is known. Diagnostics and configurable concurrency can be mentioned as mitigation, but they should not be treated as the core fix.

## Evidence

- **Issue context:** #64253 reports OpenClaw 2026.4.5 on Windows 10 with repeated `Subagent announce completion direct announce agent call ... gateway timeout after 120000ms`, provider failover timeouts, lane wait exceeded, and eventual Telegram/channel silence until manual gateway restart. There are no comments resolving it; the only timeline cross-reference is #71699, which is still open and explicitly describes a distinct Windows hard-crash/post-restart wedge.
