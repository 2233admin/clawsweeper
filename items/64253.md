## Summary

Current main still preserves the reported risky path: subagent completion delivery calls the direct Gateway `agent` RPC first, uses the same 120000 ms announce timeout, classifies `gateway timeout` as transient/retryable, and has regression coverage expecting four direct `agent` attempts after `gateway timeout after 120000ms`. Related reports #47975 and #71699 are open and adjacent, but neither is a merged canonical fix or duplicate for this Windows alive-but-unresponsive completion-timeout report.

## Best Possible Solution

Keep this issue open as the focused Windows alive-but-unresponsive Gateway/subagent completion-timeout report. The maintainer path is to reproduce or add a current-main stress regression around concurrent subagent completion announcements and saturated Gateway `agent` calls, then decide whether completion delivery should fail over faster, avoid direct parent-agent calls under saturation, add an unresponsive-Gateway health watchdog for Windows service installs, or address lane/provider starvation. Cross-link #47975 and #71699 as related but distinct reports rather than closing this issue now.
