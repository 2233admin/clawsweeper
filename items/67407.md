## Summary

Keep this issue open. Current main still has adjacent safeguards, including config validation and the separate lossless-claw context-engine fix, but it does not implement `openclaw gateway preflight [--json]` or a shared no-bind gateway startup validation path that loads plugins/context engines and checks bindability before serving traffic.

## Best Possible Solution

Keep this issue open and implement `openclaw gateway preflight [--json]` as a first-class Gateway CLI command. The implementation should extract a shared no-bind startup validation routine used by both `gateway run` and `gateway preflight`: read and validate config, resolve required SecretRefs/auth, plan/load startup plugins enough to catch duplicate IDs and missing dependencies, resolve the selected context engine factory, check configured port bindability without accepting connections, and emit stable human/JSON results with exit code 0 or 1. Add focused CLI/startup tests and Gateway docs; `openclaw update --safe` can later compose stop, install, preflight, restart, and rollback around that routine.

## Risks / Open Questions

- A shallow schema-only command would not catch the plugin/context-engine startup failures described in the report.
- Directly calling the current startup path as preflight would be unsafe because startup can perform side-effectful work such as config/auth seeding, plugin runtime dependency staging, startup maintenance, session migration, sidecar startup, and finally port binding.
