## Summary

Keep this issue open. Current main still lacks a pinned key-value facts editor and lacks the broader manifest/gateway/SPA seam for persistent Control UI plugin panels that the recent discussion refined. Existing behavior is raw workspace markdown editing plus chat-local pinned messages/tool sidebars, not a persistent Control UI panel with guaranteed context injection semantics.

## Best Possible Solution

Keep this open unless maintainers split a narrower canonical issue. The best path is an additive, manifest-first Control UI panel contract in the plugin manifest/schema, loader, gateway protocol, and SPA mounting layer, followed by a pinned facts editor backed by a documented structured store or `USER.md` section with tests/docs proving those facts are included in the agent context on every turn.

## Risks / Open Questions

- This overlaps with broader Control UI/sidebar work in #46630 and #49828; maintainers may want to split a canonical manifest-first Control UI panel API issue and leave this issue focused on pinned facts UX.
- A panel API needs product and security design around tool refresh, canvas or iframe mounting, permissions, and persistence before third-party plugins can safely render persistent Control UI surfaces.
- A pinned facts editor still needs explicit storage and context-injection semantics; raw `USER.md` editing alone would not satisfy the requested structured and discoverable workflow.
