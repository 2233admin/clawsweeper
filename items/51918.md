## Summary

Keep open. Current `main` has configurable `steer`/`steer-backlog` injection and hard `interrupt`, but it does not implement owner-priority delivery, a `priority` flag, or a soft-preemption mode that preserves pending work. The issue still describes a real gap on current `main`, not something already fixed or obsolete.

## Evidence

- **Queue docs: default remains collect; no owner-priority mode:** The documented queue modes are `steer`, `followup`, `collect`, `steer-backlog`, `interrupt`, and legacy `queue`, with `collect` as the default for all surfaces.
  - file: [docs/concepts/queue.md:25](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/docs/concepts/queue.md#L25)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Steering is not mid-tool soft preemption:** Docs say queued steering is delivered only after the current assistant turn finishes executing its tool calls, before the next LLM call. That is narrower than the issue's requested next-tool-boundary owner preemption.
  - file: [docs/concepts/agent.md:85](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/docs/concepts/agent.md#L85)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Runtime only steers when queue mode is explicitly steer/steer-backlog:** `shouldSteer` is true only for `steer` and `steer-backlog`; `collect` and `followup` are followup-only paths.
  - file: [src/auto-reply/reply/get-reply-run.ts:633](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/auto-reply/reply/get-reply-run.ts#L633)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Interrupt is hard abort, not soft delivery:** When queue mode is `interrupt`, the runtime aborts the active run and waits for it to end before continuing.
  - file: [src/auto-reply/reply/get-reply-run-queue.ts:29](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/auto-reply/reply/get-reply-run-queue.ts#L29)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Config surface has no owner-priority or soft-preemption fields:** The queue config exposes only `mode`, `byChannel`, `debounceMs`, `debounceMsByChannel`, `cap`, and `drop`. Queue mode enums likewise contain no `owner-priority`, `interrupt-soft`, or similar mode.
  - file: [src/config/types.messages.ts:13](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/config/types.messages.ts#L13)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Repo search found no priority feature implementation:** Search for `ownerPriority|priorityMode|interrupt-soft|steer-priority|priority: true` returned no matches in `src`, `docs`, or `test`.
  - command: `rg -n "ownerPriority|priorityMode|interrupt-soft|steer-priority|priority: true" src docs test`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Owner status is used for auth grouping, not queue priority:** Collect-mode drain keys on `senderIsOwner`, but only to split authorization-compatible batches. Tests show owner and non-owner queued items are delivered as separate collect batches, not that owner messages preempt an active run.
  - file: [src/auto-reply/reply/queue/drain.ts:62](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/auto-reply/reply/queue/drain.ts#L62)
