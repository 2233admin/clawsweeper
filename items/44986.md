## Summary

Keep open. Current `main` does not implement this install-skip optimization, and the PR still has an unresolved correctness issue: its skip check would trust any `openclaw` found on `PATH`, not specifically the npm-managed install.

## Evidence

- **Current main still always runs the npm install path:** `install_openclaw()` resolves the target version only for messaging, then unconditionally builds `install_spec` and calls `install_openclaw_npm`. There is no early-return skip path and no `OPENCLAW_FORCE_INSTALL`/`--force` handling on `main`.
  - file: [scripts/install.sh:1991](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/scripts/install.sh#L1991)
  - command: `nl -ba scripts/install.sh | sed -n '1988,2050p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **The version probe currently trusts PATH before npm's global package tree:** `resolve_openclaw_version()` first uses `OPENCLAW_BIN` or `command -v openclaw` and only falls back to `$(npm root -g)/openclaw/package.json` if no version was found. A skip optimization built on this helper would treat any matching binary on `PATH` as proof that the npm-managed install is current.
  - file: [scripts/install.sh:2148](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/scripts/install.sh#L2148)
  - command: `nl -ba scripts/install.sh | sed -n '2140,2218p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Binary resolution also prefers PATH over npm global bin:** `resolve_openclaw_bin()` returns `type -P openclaw` before checking the npm global bin directory, reinforcing the review concern that installer logic can pick up a different wrapper or binary ahead of the npm-managed one.
  - file: [scripts/install.sh:1844](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/scripts/install.sh#L1844)
  - command: `nl -ba scripts/install.sh | sed -n '1838,1878p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Review discussion already contains a concrete unresolved repro:** In the PR review context, `altaywtf` reported reproducing a bad skip by placing a fake `openclaw` earlier on `PATH` while `npm root -g` contained no `openclaw` package; the branch still skipped installation. The follow-up commit mentioned in the thread only addressed the PATH warning on early return, not this PATH-vs-npm ownership bug.

## Risks / Open Questions

- Closing this PR would discard useful work even though the requested behavior is still absent from `main`.
- Merging the PR as-is could skip installation when an unrelated `openclaw` binary on `PATH` reports the target semver, leaving the npm-managed package missing or stale.
