## Summary

Keep open. Current main still has the original race-shaped behavior: `maybeNotifyOnExit()` only checks `backgrounded`, `notifyOnExit`, and `exitNotified`, while `process poll` waits asynchronously without marking an active poll or setting `exitNotified` when it observes completion. The PR is a focused, recent bug-fix candidate with tests and live validation in the discussion, not something already implemented or superseded. It likely needs maintainer review/changes for the overlapping-poll race called out in review.

## Best Possible Solution

Keep the PR open as a plausible focused fix. Ask for a small revision that tracks in-flight polls with a counter/token instead of a boolean, keeps `exitNotified` success-only in the kill path, and adds regression coverage for overlapping polls plus the active-poll notify suppression path. Then validate with the changed agent/exec test lane before considering merge.

## Risks / Open Questions

- The PR's boolean `pollActive` approach can reopen the race under overlapping polls; current main tests already exercise concurrent `process poll` calls, so a counter/token plus regression test would be safer before merge.
- The author-noted gateway-restart ghost scenario is out of scope for this PR and should remain separate if maintainers want to track it.
- Related issues #25592 and #52305 are broader than this PR, so merging this would not close every channel-routing or async-completion leak problem.
