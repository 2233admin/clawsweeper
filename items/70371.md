## Summary

Keep #70371 open. Current main still treats plugin-local config/schema failures as invalid startup config and has tests asserting gateway startup rejects that state; the registry loaders also still use strict `throwOnLoadError: true` paths. The related fix PR #70394 is still open/unmerged in the provided context, and the current checkout does not contain its proposed behavior.

## Best Possible Solution

Keep this open and fix it in core. The likely product/code path is to quarantine plugin-local config validation failures under `plugins.entries.<id>.config` into per-plugin error/disabled state, skip that plugin's runtime registration and tools, keep unrelated gateway/channels/plugins booting, and make doctor surface the exact failing config path plus a repair/fix path. A complete fix should cover startup config validation, runtime/metadata registry strict-load behavior, CLI bootstrap behavior, and regression tests for gateway startup, plugin registry status, and doctor reporting.
