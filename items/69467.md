## Summary

Keep open. Current main still has the reported live-session hazard: `current` is persisted as `session:<actual-session-key>`, explicit `session:*` jobs are run on that requested session key, and `deleteAfterRun` direct-delivery cleanup still calls `sessions.delete` on `params.agentSessionKey` without distinguishing cron-owned isolated sessions from live/current session bindings. The recent #72292 fix isolates cron context-engine keys for isolated jobs, but its regression test explicitly preserves requested keys for session-bound jobs, so it does not close this issue.

## Best Possible Solution

Keep the issue open and fix cron session cleanup semantics. The safest implementation is to make `deleteAfterRun` cleanup delete only cron-owned isolated run/session keys, never a user/live key reached through `current` or explicit `session:<id>`. For one-shot reminders created from live chats, either reuse the live session without deleting the binding afterward or create a separate run-scoped working session and clean up only that working session. Add regression coverage for a non-default chat session where `current` normalizes to `session:<live-key>` and `deleteAfterRun: true` direct delivery or `delivery.mode: "none"` plus message-tool send does not call `sessions.delete` on the live key and does not rebind the live session.
