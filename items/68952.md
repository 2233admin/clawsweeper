## Summary

Keep #68952 open. Current main still suggests the untagged local default `gemma4`, appends discovered `/api/tags` names such as `gemma4:latest` as separate models, and only skips the pull when the selected model name exactly matches a discovered tag. That leaves the reported duplicate picker entries and misleading `Downloading`/`Downloaded` copy reachable when Ollama reports only `gemma4:latest`.

## Best Possible Solution

Keep the issue open and fix it in the bundled Ollama provider/onboarding path. The likely maintainer-quality fix is to canonicalize Ollama local tag aliases before model picker construction and before pull checks, treating an untagged requested model as satisfied by `<name>:latest`; prefer the discovered concrete tag in UI/defaults when present; and update the pull/probe copy so availability checks say something like `Querying Ollama` and already-present models end with `Using <model>` instead of `Downloaded <model>`. Add focused tests for `tags: ["gemma4:latest"]` covering picker/config dedupe and no `/api/pull` call for selecting `ollama/gemma4`.
