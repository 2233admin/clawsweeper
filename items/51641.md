## Summary

Keep open. Current `main` does not contain this PR's `subsystem.ts` or `help.ts` changes, and the existing stderr-routing on `main` is intentionally narrower: it is enabled for suppressed/machine-output flows (`--json`) and plugin loading under that mode, not for every non-TTY stdout case.

## Evidence

The PR's proposed `!process.stdout.isTTY` branch is not present on `main`.

## Risks / Open Questions

- The underlying non-JSON piped-output bug still looks plausible on `main` for commands like `message send`, because plugins load for that command family while stderr routing is disabled outside `--json` mode.
- The PR's implementation is also not obviously safe to close as already-handled, because current `main` appears to intentionally avoid routing all non-TTY info logs to stderr; the open design question is broader than a simple duplicate-of-main fix.
