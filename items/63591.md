## Summary

Current main still has the reported behavior: `sessions_spawn(runtime="acp")` is advertised and routed based on ACP/backend availability, but `spawnAcpDirect` immediately sends the first turn through the normal `agent` path, where `acp.dispatch.enabled=false` raises `ACP_DISPATCH_DISABLED` before `runTurn`. The prior ClawSweeper comment remains accurate against current main, and the related ACP items cover different transcript, placement, timeout, and action-support work rather than this manual-vs-automatic dispatch policy split.

## Best Possible Solution

Keep this issue open. Implement a focused ACP policy split: carry an explicit manual or parent-owned spawn context from `sessions_spawn` and `spawnAcpDirect` into the initial `agentCommand` ACP turn, bypass only `acp.dispatch.enabled` for that context if maintainers want the reported semantics, keep the global ACP/backend/sandbox/allowlist gates intact, and preserve `src/auto-reply/reply/dispatch-acp.ts` blocking automatic bound-thread follow-ups when dispatch is disabled. Add regression coverage for manual ACP spawn success with dispatch disabled plus automatic dispatch still returning `ACP_DISPATCH_DISABLED`, then align the ACP docs.

## Risks / Open Questions

- Changing this without a narrow manual-spawn context could weaken an intentional global ACP execution gate. The safer product path is to bypass only the dispatch-only gate for explicit manual or parent-owned spawn turns, while preserving `acp.enabled`, backend availability, sandbox, and `acp.allowedAgents` checks.
- Current docs say `acp.dispatch.enabled=false` blocks execution, so maintainers need to confirm the desired semantics and update docs/tests with the code change.
