## Summary

Keep open. Current `main` still documents cron timezone behavior inconsistently with runtime behavior, so this is not implemented on `main`; and the PR's CLI wording is likely still too ambiguous for remote-gateway use.

## Evidence

- **Runtime fallback is host local timezone when `tz` is omitted:** `resolveCronTimezone()` returns `Intl.DateTimeFormat().resolvedOptions().timeZone` when no `tz` is provided, so cron schedules do not default to UTC.
  - file: [src/cron/schedule.ts:9](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/cron/schedule.ts#L9)
  - command: `nl -ba src/cron/schedule.ts | sed -n '1,20p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **LLM-facing cron tool text on `main` is still misleading:** The tool schema shows cron with optional timezone, then immediately states `ISO timestamps without an explicit timezone are treated as UTC`, without scoping that statement to `schedule.kind=\"at\"`.
  - file: [src/agents/tools/cron-tool.ts:456](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/agents/tools/cron-tool.ts#L456)
  - command: `nl -ba src/agents/tools/cron-tool.ts | sed -n '456,465p'`

- **CLI help on `main` still lacks the default-timezone clarification:** `cron add` and `cron edit` both describe `--tz` only as `Timezone for cron expressions (IANA)`, with no note about the default timezone source.
  - file: [src/cli/cron-cli/register.cron-add.ts:87](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/cli/cron-cli/register.cron-add.ts#L87)

## Risks / Open Questions

- Closing as implemented would be incorrect because the misleading text is still present on current `main`.
- Closing as obsolete would discard a valid docs/help issue; the runtime and user-facing copy still disagree.
- Even if this PR merged as-is, the CLI phrase `system local timezone` could still mislead users who target a remote gateway.
