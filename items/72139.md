## Summary

Keep #72139 open. Current main still allows the stale-token retry path the issue reports, and `chat.history` still performs full synchronous transcript read/parse/projection work before response limits are applied. Existing docs/tests show partial auth guidance and history size caps, but not the requested terminal auth failure behavior or non-starving history handler.

## Best Possible Solution

Keep the issue open for a core Gateway/Control UI fix: make `AUTH_TOKEN_MISMATCH` terminal after the single trusted device-token retry, and terminal immediately when that retry cannot be attempted; surface a clear token re-entry action. Separately, make `chat.history` non-starving by using bounded tail reads, async/yielded transcript parsing, or another bounded projection path, and add regression coverage for token-mismatch no-loop behavior plus large-history responsiveness while Slack-class timers remain healthy.
