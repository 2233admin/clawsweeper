## Summary

Keep #72057 open. Current main still shows a real Google LLM proxy mismatch: Google Gemini LLM streams use `buildGuardedModelFetch`, while the provider HTTP helper path used by Google TTS/media requests auto-upgrades to env-proxy mode when `HTTP_PROXY`/`HTTPS_PROXY` applies. The existing explicit `models.providers.*.request.proxy.mode: "env-proxy"` path is documented, but that does not resolve the reporter's regression claim that ambient Docker proxy env vars alone used to work for Google LLM requests.

## Best Possible Solution

Keep the issue open and add parity for model-provider LLM transports, most likely by making `buildGuardedModelFetch` apply the same conservative env-proxy auto-upgrade used by provider HTTP helpers when the target is proxyable and no explicit dispatcher policy is present. Add a focused regression test for a Google Gemini LLM request with `HTTPS_PROXY` set, plus documentation clarifying ambient env proxy versus explicit `request.proxy` behavior.

## Evidence

## Risks / Open Questions

- No live Google API/proxy reproduction was run during this read-only review, so the exact network failure mode still needs a focused regression test or live maintainer repro.
- Maintainers need to decide whether Google/native LLM model transports should gain the same ambient env-proxy auto-upgrade as provider HTTP helpers, or whether docs/onboarding should require explicit `models.providers.google.request.proxy.mode: "env-proxy"` for LLM requests.
