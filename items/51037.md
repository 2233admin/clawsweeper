## Summary

Current main already supports per-account Telegram native command control. `channels.telegram.accounts.<id>.commands.native` and `.nativeSkills` are part of the Telegram account config, account settings are merged before runtime, and Telegram startup resolves those per-account settings when registering menus. The same implementation is present in release `v2026.4.22`.

## Evidence

- **Per-account Telegram config includes command overrides:** `TelegramAccountConfig` defines `commands?: ProviderCommandsConfig`, which is the config surface for `channels.telegram.accounts.<id>.commands.native` and `.nativeSkills`.
  - file: [src/config/types.telegram.ts:94](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/config/types.telegram.ts#L94)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Account config is merged per account before use:** Telegram runtime merges channel-level defaults with the selected account entry via `mergeTelegramAccountConfig`, so account-specific command settings override the provider defaults for that bot.
  - file: [extensions/telegram/src/account-config.ts:16](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/telegram/src/account-config.ts#L16)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Native command enablement is resolved from the selected account config:** Telegram startup reads `telegramCfg.commands?.native` and `telegramCfg.commands?.nativeSkills` for the current account, falling back to global `cfg.commands` only when the account override is unset.
  - file: [extensions/telegram/src/bot-core.ts:467](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/telegram/src/bot-core.ts#L467)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Menu sync is scoped by account and bot identity:** Telegram menu synchronization keys its cache by `accountId` and `botIdentity`, and registration passes the current `accountId` plus bot token when syncing commands. That is consistent with separate per-bot menu handling in multi-account setups.
  - file: [extensions/telegram/src/bot-native-command-menu.ts:217](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/telegram/src/bot-native-command-menu.ts#L217)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Same behavior is present in the latest release:** The release tag `v2026.4.22` contains the same account-level `commands?: ProviderCommandsConfig` surface, account-config merge, and per-account Telegram native-command resolution (`git show v2026.4.22:src/config/types.telegram.ts`, `git show v2026.4.22:extensions/telegram/src/account-config.ts`, `git show v2026.4.22:extensions/telegram/src/bot.ts`).
  - command: `git show v2026.4.22:src/config/types.telegram.ts | sed -n '94,106p'; git show v2026.4.22:extensions/telegram/src/account-config.ts | sed -n '16,36p'; git show v2026.4.22:extensions/telegram/src/bot.ts | sed -n '465,478p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Risks / Open Questions

- The report may reflect mis-mapped Telegram account ids or stale Telegram/BotFather menu state rather than a missing config capability.
- This review was read-only and could not live-reproduce against Telegram, so it cannot distinguish config/operator error from a separate runtime bug outside the reported request.

## Close Comment

Codex review: this is already implemented on current `main`, and the same support is present in released OpenClaw `v2026.4.22`. Telegram account config already exposes per-account command overrides at `channels.telegram.accounts.<id>.commands.native` / `.nativeSkills` (`src/config/types.telegram.ts:94`), those account settings are merged before runtime (`extensions/telegram/src/account-config.ts:16`), and Telegram startup resolves the selected account's `telegramCfg.commands?.native` / `.nativeSkills` when registering menus (`extensions/telegram/src/bot-core.ts:467`). In other words, the config shape requested in this issue is already supported for multi-account Telegram bots. Closing as implemented based on this Codex review. If you still see the wrong menu on a current release, please open a new bug with the exact `channels.telegram.accounts` config, which bot username maps to which account id, and the startup logs around `deleteMyCommands` / `setMyCommands` for each bot.
