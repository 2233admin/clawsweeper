## Summary

Current main still lacks the requested OS-level TCP keepalive for the WhatsApp WebSocket. The WhatsApp plugin creates a Baileys socket with auth/version/logger/proxy options and only adds a WebSocket error handler afterward; existing keepAlive code is watchdog/reconnect logic, not TCP socket keepalive. The issue is recent, has a same-problem comment, and the related WhatsApp timeout report remains open, so this should stay open for a WhatsApp-local fix and test.

## Best Possible Solution

Keep this issue open. Implement the mitigation in the WhatsApp plugin socket setup, likely near extensions/whatsapp/src/session.ts or a small WhatsApp-local helper, by enabling TCP keepalive on the underlying Node socket after it exists with a conservative initial delay such as 15000 ms. Add a focused unit test with a mocked Baileys/ws socket proving setKeepAlive is applied while preserving ambient proxy agent behavior.
