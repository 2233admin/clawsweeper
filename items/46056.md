## Summary

Current `main` still appears to gate shell builtins under allowlist mode. Builtins like `cd` do not resolve to a filesystem path, `matchAllowlist` rejects entries without a resolved path, and `requiresExecApproval` prompts on any allowlist miss. The docs now say allowlist patterns must resolve to binary paths and that every chain segment must satisfy allowlist rules, but they still do not explicitly document shell builtin behavior or the reported workaround.

## Evidence

- **Unresolved executables stay pathless:** `buildExecutableResolution` sets `resolvedPath` from executable-path lookup and otherwise leaves it unset while keeping the raw token as `executableName`. A shell builtin such as `cd` has no binary path to match.
  - file: [src/infra/exec-command-resolution.ts:62](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/infra/exec-command-resolution.ts#L62)
  - command: `nl -ba src/infra/exec-command-resolution.ts | sed -n '62,80p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Allowlist matcher rejects commands without resolved paths:** `matchAllowlist` returns `null` when `resolution?.resolvedPath` is missing unless the entry is a bare `*`, so pathless builtins cannot satisfy a normal allowlist entry.
  - file: [src/infra/exec-command-resolution.ts:318](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/infra/exec-command-resolution.ts#L318)

- **Shell allowlist evaluation depends on that path-based match:** `resolveSegmentAllowlistMatch` feeds the segment's policy target into `matchAllowlist`; there is no builtin-specific bypass or safe-builtin handling in this path.
  - file: [src/infra/exec-approvals-allowlist.ts:410](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/infra/exec-approvals-allowlist.ts#L410)
