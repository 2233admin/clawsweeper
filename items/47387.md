## Summary

Keep open. Current `main` still does not thread Telegram proxy/transport settings into outbound remote media prefetch for either direct sends or reply delivery, so the PR's requested fix is not already implemented on `main` and should not be conservatively closed.

## Evidence

- **Direct Telegram send path still loads remote media without transport options:** `sendMessageTelegram` calls `loadWebMedia(mediaUrl, buildOutboundMediaLoadOptions(...))` with only `maxBytes`, local roots, readFile, and image optimization flags. It does not pass `proxyUrl`, `fetchImpl`, or other Telegram transport fields, even though the same file builds Telegram API client fetch options separately via `resolveTelegramFetch`.
  - file: [extensions/telegram/src/send.ts:815](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/extensions/telegram/src/send.ts#L815)

  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)

  - file: [extensions/telegram/src/bot/delivery.replies.ts:273](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/extensions/telegram/src/bot/delivery.replies.ts#L273)
