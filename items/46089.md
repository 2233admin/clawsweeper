## Summary

Current `main` already addresses the reported embedded-run timeout behavior: the old hard 600s path is gone, agent/LLM timeouts are configurable, idle stalls are retried once then surfaced as explicit errors instead of leaving the session apparently hung, and the queueing behavior during an active run is now documented with Stop/Steer controls. This shipped by `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Close Comment

Closing from a Codex review because current `main` already covers the reported behavior. The embedded runner no longer relies on the old hard-coded 600s path: timeout resolution is configurable in `src/agents/timeout.ts:3`, LLM idle stalls are configurable via `agents.defaults.llm.idleTimeoutSeconds` (`src/config/types.agent-defaults.ts:500`, `src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:19`), and `runEmbeddedPiAgent()` now retries one silent idle-timeout and then returns an explicit timeout error instead of leaving the turn apparently finished but still blocking the queue (`src/agents/pi-embedded-runner/run.ts:1662`, `src/agents/pi-embedded-runner/run.ts:1779`). There is regression coverage for that behavior in `src/agents/pi-embedded-runner/run.timeout-triggered-compaction.test.ts:225` and `src/agents/pi-embedded-runner/run/assistant-failover.test.ts:192`. The queueing you described is also now documented as expected while a run is active, with Stop/Steer controls (`docs/web/control-ui.md:170`). This implementation is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this issue is being closed as implemented on main.
