## Summary

Keep open. Current main has count-based history limiting and separate session reset/retention controls, but no configurable time-based session context window that filters prompt history to the last N hours across turns.

## Evidence

- **History limiting is turn-count based, not time based:** `limitHistoryTurns` keeps the last N user turns; `getHistoryLimitFromSessionKey` resolves only `historyLimit` / `dmHistoryLimit` values from channel config. There is no timestamp or hour-window parameter in this path.
  - file: [src/agents/pi-embedded-runner/history.ts:13](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/pi-embedded-runner/history.ts#L13)
  - command: `nl -ba src/agents/pi-embedded-runner/history.ts | sed -n '10,120p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Main request assembly uses only the turn-count limiter:** Before a prompt is sent, the runner truncates with `limitHistoryTurns(..., getDmHistoryLimitFromSessionKey(...))`. This is the live per-turn context reduction path, and it does not filter by message age.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:1689](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/pi-embedded-runner/run/attempt.ts#L1689)

## Risks / Open Questions

- There is some overlap with existing `historyLimit`/`dmHistoryLimit`, compaction, idle reset, and `session.maintenance.pruneAfter`, so maintainers may want to decide whether this belongs as a new prompt-assembly policy or as an extension of existing history-limit config.
- I could not use `gh` in this environment because `GH_TOKEN` is unset, so this review is based on the checked-out `main` code and docs only.
