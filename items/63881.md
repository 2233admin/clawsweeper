## Summary

Keep open. Current main still has the MCP plugin-tools asymmetry the PR is trying to address: the plugin-tools MCP path passes resolved plugin tools into the MCP server and the handler only filters `ownerOnly` tools plus `before_tool_call` hooks, while the gateway HTTP path has a separate default deny list and `gateway.tools` override handling. The PR is actionable, but it needs maintainer review/rebase and scope cleanup rather than conservative closure.

## Best Possible Solution

Keep the PR open for maintainer security review. The best path is to rebase it, drop or separate the axios workspace change, implement a config-aware MCP plugin-tools deny policy in the MCP serving/handler path using an explicitly named shared default deny set, preserve the existing `ownerOnly` and `before_tool_call` approval boundaries, and add focused tests for `listTools`, `callTool`, `gateway.tools.allow`, and `gateway.tools.deny`. Also reconcile the HTTP deny-list docs/constants for `whatsapp_login` as part of the surface-specific policy decision.
