## Summary

Keep this PR open. Current main still lacks a default dangerous-tool deny policy on the plugin-tools MCP stdio path, while the HTTP `/tools/invoke` path has a default deny boundary. The PR remains a plausible security hardening candidate, but it needs a current-main rebase and should stay focused on MCP policy rather than dependency release-age cleanup.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto current main, add a universal dangerous-tool deny policy in the shared MCP plugin-tools handler/server path, preserve existing `ownerOnly` filtering and `before_tool_call` hooks, wire `gateway.tools.allow` and `gateway.tools.deny` with tested precedence for both listing and invocation, and split or drop the unrelated dependency release-age change.

## Risks / Open Questions

- The PR patch is stale relative to the current MCP handler split; the policy should be implemented in the active `createPluginToolsMcpHandlers` / `createToolsMcpServer` flow, not only the older server wrapper shape.
- Changing MCP default exposure can break operators or plugins relying on high-risk tool names through the opt-in bridge; the fix needs clear `gateway.tools.allow` and `gateway.tools.deny` precedence tests for both `listTools` and `callTool`.
- The provided PR context includes an unrelated supply-chain-sensitive package policy change. Keep dependency release-age or axios cleanup separate from the MCP deny-list fix unless a maintainer explicitly wants them coupled.
- Current docs/source around `whatsapp_login` appear to have drifted: docs list it in the HTTP deny list, while current source constant contains only the 12 high-risk entries. A rebase should align the universal/HTTP-only classification with current intended behavior.
