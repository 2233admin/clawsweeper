## Summary

Current `main` still short-circuits `resolvePluginCapabilityProviders(...)` to the active registry for non-memory capability families whenever any active providers exist, even when `cfg` is present. That matches the bug described in PR #66572 rather than an implementation already landed on `main`, so this should stay open for maintainer review.

## Evidence

- **Current runtime still bypasses cfg-compatible resolution when active speech providers exist:** `resolvePluginCapabilityProviders` returns `activeProviders` immediately for every key except `memoryEmbeddingProviders`, before using the cfg-compatible load path. That means a partial active `speechProviders` registry still wins over cfg-shaped resolution on current `main`.
