## Summary

Keep this issue open. Current main still has live tool-result guarding, provider replay hygiene, and restart recovery, but it does not implement the requested durable reopen/startup seam that appends parentId-aware synthetic tail repair rows for unresolved tool calls or an explicit assistant marker for orphan user tails.

## Best Possible Solution

Keep this issue open. The best fix is a narrow first-party reopen/startup transcript repair near session-file load/open that inspects only the active transcript tail, appends parentId-valid synthetic error toolResult rows for unresolved eligible tool calls, appends one explicit assistant recovery marker for a persisted orphan-user tail, never rewrites/deletes existing rows, is idempotent across repeated reopen, and adds focused tests for missing tail results, partial tail results, orphan-user tail, healthy no-op, and repeated reopen no-op.

## Risks / Open Questions

- A durable fix must coordinate with the existing in-memory replay repair and live pending-tool-result guard so repeated reopen does not duplicate synthetic toolResult rows.
- Eligibility for aborted or errored assistant tool-call turns needs explicit policy, since current guard and replay repair intentionally avoid synthesizing missing results for those cases.
- The implementation should write through SessionManager or an equivalent parentId-aware seam; raw JSONL appends could corrupt the active transcript branch.
