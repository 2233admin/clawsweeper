## Summary

Keep PR #72209 open. Current main still does not implement the PR's central behavior: passive recovery prompt discipline, local recovery bundle/ledger storage, or checkpoint writes from update_plan, sessions_yield, and sessions_spawn. Existing restart/orphan recovery and background task-ledger code are adjacent but solve different problems, and the Greptile P2 findings are normal review work rather than a conservative cleanup close reason.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to refine the proposed core implementation: address the redaction and synchronous-I/O review comments, keep recovery context passive and informational, ensure prior approvals/auth/permissions are never inherited, and preserve focused tests for update_plan, sessions_yield, sessions_spawn, recovery-state storage, and prompt composition. Maintainers should also decide whether the new recovery-state surface needs public docs or should integrate with existing task/session recovery concepts.
