## Summary

Keep open. Current main still intentionally documents and generates the Slack manifest without `features.assistant_view` or `assistant_thread_started`, the Slack channel implementation does not register assistant-thread events, and a maintainer already said on 2026-04-19 that the team is deliberately holding off because of Slack UI/default-experience considerations. That is product/maintainer judgment, not a conservative auto-close case.

## Evidence

- **Public Slack docs still omit the proposed manifest changes:** The current Slack manifest example in `docs/channels/slack.md` still includes `assistant:write` but does not include `features.assistant_view`, and its default `bot_events` list does not include `assistant_thread_started`.
  - file: [docs/channels/slack.md:131](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/docs/channels/slack.md#L131)
  - command: `nl -ba docs/channels/slack.md | sed -n '120,310p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Generated setup manifest matches the current docs:** `buildSlackManifest()` in the bundled Slack extension emits the same manifest shape: `assistant:write` is requested, but there is no `assistant_view` block and no `assistant_thread_started` subscription. This means the PR is not just a docs typo; it would diverge from the setup manifest OpenClaw currently generates.
  - file: [extensions/slack/src/setup-shared.ts:10](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/extensions/slack/src/setup-shared.ts#L10)

  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Slack runtime does not handle assistant-thread start events:** The Slack monitor registers handlers for `message` and `app_mention` events only. There is no `assistant_thread_started` or `assistant_thread_context_changed` event registration on current main, so the PR's added assistant-related manifest/event docs would imply support beyond the current implementation.
  - file: [extensions/slack/src/monitor/events/messages.ts:148](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/extensions/slack/src/monitor/events/messages.ts#L148)

- **Maintainer already framed this as deferred, not resolved:** In the PR discussion on 2026-04-19, maintainer `zimeg` said the team is aware of the warnings but is holding off on this feature for now because of the current default experience across the Slack UI, and that the warning should not cause problems for the recommended defaults.
  - command: `Provided GitHub context for PR #67650`

## Risks / Open Questions

- A docs-only merge would drift from the manifest OpenClaw currently generates during Slack setup.
- The PR mixes documentation of assistant-view setup with assistant-thread event subscriptions that current main does not implement, so landing it as-is could overstate supported Slack behavior.
- Closing now would discard contributor work even though the available evidence points to an unresolved maintainer/product decision rather than an obsolete or already-shipped change.
