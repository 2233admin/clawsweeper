## Summary

Keep PR #69261 open. Current main still resolves Ollama embedding auth as remote key, then provider key, then env key, and attaches the selected bearer token to whatever embedding base URL wins. The current tests also lock in two behaviors the PR is trying to change: sending the `ollama-local` placeholder as a bearer token and sending an env-backed `OLLAMA_API_KEY` to the local Ollama embedding URL. The related #69222 context is useful, but it is web-search-specific and does not supersede this embedding-provider fix. The review comments on #69261 identify P2 follow-up work, so the best maintainer action is rework/review, not conservative closure.

## Best Possible Solution

Keep #69261 open and ask for a focused update: implement embedding-specific auth scoping in `extensions/ollama/src/embedding-provider.ts`, filter non-secret/env markers consistently, prevent provider keys from crossing `memorySearch.remote.baseUrl` host boundaries, attach env `OLLAMA_API_KEY` only to the Ollama Cloud host, and add regression tests for local env leakage, provider/remote host mismatch, placeholder opt-out, and the P2 provider opt-out edge case. #69222 remains useful precedent for web search but is not the canonical fix for embeddings.
