## Summary

Keep open. Current main still requires the configured trusted-proxy identity header and fails with `trusted_proxy_user_missing` when it is absent; there is no `defaultUser`/`fallbackUser` config field or schema support. The related discussion explicitly says #25293 did not fix this and #17761 is a separate trusted-proxy fallback problem.

## Best Possible Solution

Keep tracking this as a core gateway auth/config product decision. A real fix would need to add a documented and schema-backed trusted-proxy fallback/default user option, update `authorizeTrustedProxy`/WebSocket auth behavior carefully, and add regression coverage for missing identity headers from an otherwise trusted proxy source. Maintainers should also decide whether this weakening of identity-bearing auth is acceptable for Cloudflare Zero Trust/IP-policy deployments.
