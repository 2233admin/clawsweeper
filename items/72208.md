## Summary

Keep #72208 open. It is a fresh non-maintainer bug report with concrete Windows/npm-global regression steps, and current main still has the same pricing-fetch startup surface: gateway runtime services start the model pricing refresh during gateway startup, the refresh fetches both OpenRouter and LiteLLM catalogs with 60s abort signals, and the existing tests only assert timeout logging/background scheduling, not that the Control UI remains reachable or Ctrl+C remains responsive under stalled pricing requests.

## Best Possible Solution

Keep the issue open and investigate the gateway startup pricing refresh path on Windows. A good fix would prove that pricing catalog fetch stalls cannot block Control UI readiness or shutdown: move/guard pricing refresh fully after readiness if needed, make in-flight pricing fetches cancellable on shutdown, and add a regression test or focused manual proof that stalled OpenRouter/LiteLLM fetches leave the dashboard reachable and Ctrl+C responsive.
