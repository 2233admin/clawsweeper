## Summary

Current main still does not implement the requested readiness-aware `until_ready` or equivalent public queue mode. The public queue type, config schema, `/queue` command surface, parser, docs, and generic followup drain remain mode/debounce based. The subagent announce queue has internal busy-parent deferral with a 15s cap, but that is limited to subagent completion announces and is not an operator-selectable `messages.queue` behavior for ordinary chat/status/completion messages.

## Best Possible Solution

Keep this open as core auto-reply queue work. Add an operator-selectable readiness-aware `messages.queue` mode or option, wire it through exported config types, zod schema, generated/help labels, `/queue` parsing/metadata, docs, and focused tests. Implement the generic drain condition against real session/run readiness with an explicit starvation cap, using the subagent announce deferral behavior only as prior art.
