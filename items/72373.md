## Summary

Keep open. Current main has configurable tool-loop detection and a narrow exec script-file preflight, but it does not reject literal placeholders or malformed shell syntax before OpenShell sends commands through `/bin/sh -c`. Generic identical tool repeats are also still intentionally warning-only until the separate no-progress global circuit breaker, so the issue's requested fail-fast and hard-abort behavior remains real work.

## Best Possible Solution

Keep the issue open and implement this as a narrow runtime hardening change: add an exec/OpenShell preflight before sandbox shell invocation for malformed generated-command hazards, add focused tests for OpenShell/SSH exec command rejection and successful non-hazard commands, and update loop detection so repeated failing or malformed identical tool calls hard-block after a clear threshold without overblocking legitimate progress. Also lint or rewrite bundled skill/help examples that show copy-executable placeholder commands.

## Risks / Open Questions

- A naive `<...>` rejection could break legitimate shell redirection, here-docs, XML-like args, or documentation text passed intentionally; the fix should validate token context rather than blanket-ban every angle-bracket string.
- Hard-aborting all generic repeats could block legitimate retry or polling workflows unless it is tied to failure/no-progress evidence or remains configurable.
- No live OpenShell sandbox run was performed in this read-only review; the keep-open decision is based on current source, docs, and tests, which are sufficient to show the requested behavior is not implemented.
