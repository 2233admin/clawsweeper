## Summary

Keep open. Current main still sends OpenShell exec text through the shared SSH `/bin/sh -c` wrapper without malformed-shell or literal-placeholder preflight, and generic same-tool/same-args loop detection remains warning-only until separate no-progress breakers fire. Related #58726 covers the loop-abort half, but this issue also tracks OpenShell command validation, so it should not be closed as a duplicate or implemented.

## Best Possible Solution

Keep this issue open. The maintainer-grade fix belongs in the exec/sandbox runtime path: add shared OpenShell/SSH preflight for obvious generated placeholders and malformed shell structure before remote `/bin/sh -c` construction; make repeated identical failed or no-progress generic tool calls escalate to a critical block at the configured threshold; add focused regression tests for OpenShell/shared SSH command construction and before-tool-call blocking; and separately clean up bundled skill/help examples that look copy-executable while containing placeholders.

## Risks / Open Questions

- A command preflight must be carefully scoped so legitimate shell redirection, comparisons, command substitution, and literal angle-bracket data are not rejected incorrectly.
- Generic repeated-call hard-abort should likely be failure/no-progress gated and configurable so legitimate repeated status or polling workflows are not blocked prematurely.
