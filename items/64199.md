## Summary

Keep open. Current main still maps a Discord thread that falls back to a configured parent-channel ACP binding onto the parent conversation id, and the configured ACP session key is still hashed from that materialized conversation id. That leaves the reported cross-thread persistent ACP session sharing path present on main.

## Best Possible Solution

Keep the issue open until the Discord configured ACP binding path materializes thread-scoped session identity for inherited parent-channel bindings, without breaking parent-channel routing fallback or Feishu/Telegram canonical remapping semantics. A good fix should add a regression test proving two Discord thread ids under the same parent channel resolve to distinct configured ACP session keys, likely building on or superseding PR #64322.
