## Summary

Keep open. Current main still does not validate the transcript session header in `readLatestSessionUsageFromTranscript`, and the candidate resolver can still return a sessionFile whose header belongs to a different session. The PR is a focused fix with a regression test; it likely needs a small test style cleanup before landing, not conservative closure.

## Best Possible Solution

Keep #63978 open and land the header-mismatch guard in `readLatestSessionUsageFromTranscript` with the regression test updated for current test style. The best product behavior is for usage fallback to return `null` when a discovered transcript declares a different session id, while preserving legacy/custom transcripts that have no parseable session header.
