## Summary

Keep #59273 open. Current main still has no visible config fingerprint or generation check on cached ACP runtime handles, and the Discord inbound queue still preserves a captured cfg in delayed jobs. The related stale-config cleanup in #57896 does not fully supersede this ACP/Discord-specific path.

## Best Possible Solution

Keep the issue open and fix the cache boundary directly: add an effective runtime/config fingerprint or generation to ACP runtime cache validity, or explicitly clear ACP runtime handles on gateway config restart. Also update Discord restart handling so preserved inbound jobs re-resolve current config or are rebuilt after restart. Add regression coverage for a queued Discord/ACP turn and for same-session ACP handle reuse after safeBins or tools.exec.security changes.

## Evidence

- **ACP cache reuse omits config:** ensureRuntimeHandle accepts cfg, but the cached-handle reuse path compares backend, agent, mode, cwd, metadata, and runtime liveness before returning the cached runtime handle; it does not compare the effective runtime config or an exec-config fingerprint.

  - sha: [4ad8b613c9b6](https://github.com/openclaw/openclaw/commit/4ad8b613c9b60466224d9b3737fff7d63ada652a)
- **Cached state has no config identity:** CachedRuntimeState stores runtime, handle, backend, agent, mode, cwd, and appliedControlSignature. There is no field for a config hash, config generation, safeBins value, or tools.exec.security value.
  - file: [src/acp/control-plane/runtime-cache.ts:3](https://github.com/openclaw/openclaw/blob/4ad8b613c9b60466224d9b3737fff7d63ada652a/src/acp/control-plane/runtime-cache.ts#L3)
