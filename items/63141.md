## Summary

Keep this issue open. Current main still uses a static 100000-token parent-fork cap, skips oversized parent inheritance with only a session-init log warning, and has tests/docs that explicitly preserve fresh-child behavior rather than auto-compacting, channel-warning, or model-aware tuning. The related compaction provider and checkpoint work is useful adjacent infrastructure, but it is not wired into the parent-fork overflow branch.

## Best Possible Solution

Keep open. Implement the fix in the session owner path: update src/auto-reply/reply/session.ts and src/auto-reply/reply/session-fork.ts so an oversized parent either gets compacted/checkpointed before fork or emits a one-time visible/operator warning when inheritance is skipped; optionally make resolveParentForkMaxTokens() model-aware through a narrow context-window resolver. Update the focused session tests and session/config docs to cover the chosen behavior.
