## Summary

Current `main` still matches the reported bug: `readLaunchAgentRuntime()` marks any non-zero `launchctl print` as `missingUnit: true`, and the CLI still converts `missingUnit` into `Service unit not found` plus reinstall guidance even when the LaunchAgent plist can still exist. I did not find evidence of a shipped fix on `main` or in the latest release commit.

## Evidence

- **Runtime still conflates any launchctl print failure with a missing unit:** `readLaunchAgentRuntime()` returns `{ status: "unknown", missingUnit: true }` for every non-zero `launchctl print` result, without first checking whether the LaunchAgent plist exists in that branch.
  - file: [src/daemon/launchd.ts:287](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/daemon/launchd.ts#L287)
  - command: `nl -ba src/daemon/launchd.ts | sed -n '270,310p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Gateway state can still report the service as installed from the plist:** `readGatewayServiceState()` sets `installed: command !== null`, and `readLaunchAgentProgramArguments()` reads directly from the LaunchAgent plist path. That means an existing plist can make the service look installed while runtime simultaneously reports `missingUnit`.
  - file: [src/daemon/service.ts:93](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/daemon/service.ts#L93)
