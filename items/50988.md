## Summary

Keep open. Current `main` fixed a narrower Slack regression for top-level channel messages, but real thread replies still resolve to thread-scoped session keys and there is no `channels.slack.thread.sessionMode` or equivalent config to route them to the parent session by default.

## Evidence

- **Slack routing still creates thread-scoped session keys for thread replies:** `resolveSlackRoutingContext()` computes `roomThreadId` for actual thread replies, passes it as `threadId`, and does not set `useSuffix: false`; `threadInheritParent` only affects `parentSessionKey`.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:80](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L80)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/prepare-routing.ts | sed -n '40,130p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Thread suffix remains the default behavior in shared routing helper:** `resolveThreadSessionKeys()` defaults `useSuffix` to `true`, so any non-empty `threadId` becomes `:thread:<id>` unless a caller explicitly disables it.
  - file: [src/routing/session-key.ts:250](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/routing/session-key.ts#L250)

  - file: [src/config/types.slack.ts:92](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/config/types.slack.ts#L92)

## Risks / Open Questions

- The issue body bundled two behaviors: current `main` appears to have fixed the older per-top-level-message fragmentation regression, but the requested parent-session routing for actual thread replies is still not implemented.
- Runtime conversation bindings can redirect a specific Slack thread to another session, but that is not a general `channels.slack.thread.sessionMode` replacement and does not justify closing this issue.
