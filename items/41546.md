## Summary

Keep open. Current `main` intentionally clears the gateway token when the WebSocket URL changes, scopes tokens to the selected gateway URL, and sends the current token to the currently selected gateway on connect. This PR removes that guard and flips the test expectation, so it would introduce a credential-scoping regression rather than clean up an obsolete item.

## Evidence

- **Overview UI intentionally drops the token when the gateway URL changes:** The current `@input` handler updates `gatewayUrl` and resets `token` unless the trimmed URL is unchanged, so token clearing is deliberate behavior on `main`, not an accidental bug.
  - file: [ui/src/ui/views/overview.ts:266](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/ui/src/ui/views/overview.ts#L266)

  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Browser test locks in the current behavior:** `navigation.browser.test.ts` explicitly asserts that after editing the gateway URL, `refreshed.settings.token` becomes an empty string. The PR rewrites this expectation, so it is changing an existing contract, not just adding missing coverage.
  - file: [ui/src/ui/navigation.browser.test.ts:311](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/ui/src/ui/navigation.browser.test.ts#L311)

If the UI preserved an old token while the URL changed to a different host, the old credential would be sent to the new endpoint.

  - file: [ui/src/ui/storage.ts:105](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/ui/src/ui/storage.ts#L105)
