## Summary

Keep open. Current `main` contains useful WhatsApp proxy support that may help some handshake-timeout cases, but I did not find source, tests, docs, changelog, or related-item evidence tying #64296's Windows 11 408 login failure to that fix. The current WhatsApp login path still treats non-515/non-401 connection close errors, including a 408 opening-handshake timeout, as a terminal failed login and prints the same `WhatsApp Web connection ended before fully opening` message.

## Best Possible Solution

Keep #64296 open for maintainer or reporter follow-up. Ask affected users to retry v2026.4.24 or newer with `--verbose`, include proxy-related environment details (`HTTPS_PROXY`, `HTTP_PROXY`, `NO_PROXY`, redacted), and confirm whether the current proxy-aware WhatsApp socket fixes their Windows login. If 408 still reproduces, the likely code path is `extensions/whatsapp/src/connection-controller.ts`: add a deliberate 408 handling policy such as bounded retry/backoff, clearer diagnostics, or a configurable connect timeout, with a unit test that exercises a 408 `waitForWaConnection` failure.
