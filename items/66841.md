## Summary

Keep open. Current main still tails configured gateway file logs via `logs.tail` and falls back to `readConfiguredLogTail` for local loopback pairing/scope failures; it does not prefer the active Linux systemd journal or print the requested service provenance and mismatch warnings. The reporter's validated local prototype is useful implementation evidence, but that behavior is not present in the checked-out main tree.

## Best Possible Solution

Implement the local Linux/systemd path in `src/cli/logs-cli.ts`: detect the active `openclaw-gateway.service` using existing daemon service helpers, prefer `journalctl --user _PID=<pid>` for local `--follow` when available, print CLI version, service ExecStart, service PID, service version, and exact log source before tailing, warn on mismatches, keep explicit `--url` and Control UI `logs.tail` file-log behavior unchanged, add focused CLI tests for pairing/scope fallback to journal, and update the public logs/logging docs.

## Risks / Open Questions

- Operators can still be misled during side-by-side cutover because an old CLI/install context may tail a configured rolling file instead of the active gateway service journal.
- A fix needs to preserve explicit remote `--url` RPC behavior and Control UI `logs.tail` file-log behavior while adding the Linux/systemd local CLI path only where appropriate.
- Any provenance added to `--json` output should be designed deliberately so existing scripts consuming `openclaw logs --json` are not broken unexpectedly.
