## Summary

Current `main` routes unresolved-model misses to structured `Unknown model` / `model_not_found` errors before `normalizeResolvedModel()` is reached, so the reported crash path does not match the present code.

## Risks / Open Questions

- `normalizeResolvedModel()` still assumes a non-undefined model object, so a future unguarded caller could reintroduce a similar crash even though the current mainline path does not reproduce this report.

## Close Comment

Closing this as `cannot_reproduce` after a Codex review against current `main`.

- `normalizeResolvedModel()` is only called from guarded paths in `src/agents/pi-embedded-runner/model.ts`; each caller already has a concrete model or returns early before normalization.
- Current model-resolution misses are handled by `resolveModel()` / `resolveModelAsync()` returning `buildUnknownModelError(...)`, not by passing `undefined` into `normalizeResolvedModel()`.
- The PI runner now checks `if (!model)` right after `resolveModelAsync()` and raises `FailoverError(..., { reason: "model_not_found" })` in `src/agents/pi-embedded-runner/run.ts`.
- Current regression coverage in `src/agents/pi-embedded-runner/model.startup-retry.test.ts` expects an `Unknown model` error on a miss, not a startup crash.
- The additional `agentId` note also does not match current hook wiring; current run and prompt-build hook contexts populate `agentId` before invoking plugins.

If this still reproduces on a newer checkout, please open a fresh issue with the exact plugin, provider/model ref, current `main` SHA, and the full stack trace.
