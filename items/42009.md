## Summary

Keep open. Current `main` still routes `openclaw agent --agent <id> --to <target>` through the agent's explicit main-session alias before considering `--to`, so the PR's requested behavior is not implemented on `main`.

## Evidence

- **CLI contract says `--to` derives the session key:** The user-facing CLI docs define `--to <dest>` as the recipient used to derive the session key, while `--agent <id>` selects the agent.
  - file: [docs/cli/agent.md:26](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/docs/cli/agent.md#L26)
  - command: `nl -ba docs/cli/agent.md | sed -n '1,60p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Gateway command passes both `agentId` and `to` into session resolution:** `agentViaGatewayCommand()` calls `resolveSessionKeyForRequest({ cfg, agentId, to: opts.to, sessionId: opts.sessionId })`, so this resolver is the deciding path for `--agent + --to` behavior.
  - file: [src/commands/agent-via-gateway.ts:113](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/commands/agent-via-gateway.ts#L113)
  - command: `nl -ba src/commands/agent-via-gateway.ts | sed -n '96,120p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Current resolver still lets `agentId` override `--to`:** `resolveSessionKeyForRequest()` computes `explicitSessionKey = opts.sessionKey?.trim() || resolveExplicitAgentSessionKey({ ... agentId ... })`, then uses `explicitSessionKey ?? resolveSessionKey(...)`. With `--agent` set and no explicit session key, the agent main-session alias wins before the `to`-derived key is considered.
  - file: [src/agents/command/session.ts:140](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/command/session.ts#L140)
  - command: `nl -ba src/agents/command/session.ts | sed -n '130,154p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Explicit agent session helper always returns the agent main session:** `resolveExplicitAgentSessionKey()` returns `resolveAgentMainSessionKey(...)` whenever `agentId` is present, which is the root cause of the `--to` value being ignored in the current resolver.
  - file: [src/config/sessions/main-session.ts:36](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/config/sessions/main-session.ts#L36)
  - command: `nl -ba src/config/sessions/main-session.ts | sed -n '36,45p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Current tests do not cover the reported `--agent + --to` regression:** The present dedicated resolver tests cover duplicate `sessionId` selection and scoped store lookup, but there is no current regression case for `agentId + to`, so there is no evidence that `main` already protects this behavior.
