## Summary

Current `main` already gives spawned ACP child runs a push-based completion path back into the parent/orchestrator session via the task-completion announce flow, which covers the reported no-polling post-completion workflow even though it is not exposed as a literal `acp:session:complete` internal hook name.

## Risks / Open Questions

- This does not add a separately named internal hook event `acp:session:complete`; it covers the workflow through the built-in task-completion announce channel instead.

## Close Comment

Codex review: this is already covered on current `main`.

- `sessions_spawn({ runtime: "acp" })` now registers accepted ACP child runs in the same completion registry used for auto-announce, with `expectsCompletionMessage` enabled by default when not using `streamTo: "parent"` (`src/agents/tools/sessions-spawn-tool.ts:245-326`).
- That registry listens for child lifecycle `end` / `error` events and completes the tracked run instead of requiring polling (`src/agents/subagent-registry.ts:657-700`).
- Completion is turned into an internal `task_completion` event and sent back to the requester session as an inter-session `agent` call; for orchestrator sessions this is `deliver: false`, which is a real session wake/new turn rather than a passive system note (`src/agents/subagent-announce.ts:502-545`, `src/agents/subagent-announce-delivery.ts:302-328`).
- The ACP docs now explicitly describe parent-owned one-shot ACP children as reporting completion through this built-in task-completion announce path, and skipping A2A follow-up because the completion path already wakes the parent (`docs/tools/acp-agents.md:346-368`).
- Tests cover both waking an active requester on completion and nested session-internal completion delivery (`src/agents/subagent-announce-delivery.test.ts:126-138`, `src/agents/subagent-announce.test.ts:385-423`).
