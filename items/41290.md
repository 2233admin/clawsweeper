## Summary

Keep open. Current `main` still accepts oversized heartbeat intervals and passes the resulting delay straight to `setTimeout`, so the 32-bit timer overflow described in this PR is still plausible and the PR’s clamp/validation/test coverage is not present in shipped code.

## Evidence

- **Current interval resolution has no upper bound:** `resolveHeartbeatIntervalMs` parses the configured duration and returns any positive millisecond value unchanged; there is no clamp to the Node timer maximum.
  - file: [src/infra/heartbeat-summary.ts:44](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/infra/heartbeat-summary.ts#L44)
  - command: `nl -ba src/infra/heartbeat-summary.ts | sed -n '1,220p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Current scheduler uses unclamped timeout delay:** `scheduleNext()` computes `const delay = Math.max(0, nextDue - now)` and passes that directly to `setTimeout`, so a heartbeat interval above `2_147_483_647` ms can still overflow in Node.
  - file: [src/infra/heartbeat-runner.ts:1366](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/infra/heartbeat-runner.ts#L1366)

  - file: [src/config/zod-schema.agent-runtime.ts:46](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/config/zod-schema.agent-runtime.ts#L46)
