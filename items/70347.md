## Summary

Current main still does not implement the requested cron outer-timeout lifecycle emission. The scheduler aborts and rejects from the wall timer, records the cron run as an error, and existing gateway persistence only finalizes sessions when a lifecycle start/end/error event is actually emitted. Related work such as #67785 and #72317 improved lane scoping and CLI abort propagation, but neither adds a synchronous lifecycle.error from the cron timeout path.

## Best Possible Solution

Keep this issue open. The best fix is to make the cron outer-timeout path publish a terminal lifecycle error with the active run session key/session id before returning the timeout result, then add focused regression coverage that sessions.json is finalized synchronously without relying on later watchdog or runner-specific cleanup.

- The regression test should cover scheduled, manual, and startup catch-up timeout paths if they share executeJobCoreWithTimeout.
