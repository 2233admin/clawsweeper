## Summary

Keep open. Current `main` still matches the PR's root-cause claim: task-registry maintenance treats cron jobs as live only when `isCronJobActive(jobId)` is true, but the startup catch-up path and the manual-run path do not call `markCronJobActive`/`clearCronJobActive`. The same gap is present in the latest shipped release `v2026.4.23`, so this is not implemented on main and not a stale cleanup close.
