## Summary

Keep PR #71040 open. Current main still has the active cron job signal wired for regular due-job and executeJob paths, but not for startup catch-up or manual-run execution. Task registry maintenance still treats authoritative cron tasks as backed only by isCronJobActive(jobId) after the 5-minute grace window, so the PR addresses a real remaining gap rather than stale or duplicated work.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land a small producer-side fix that marks and clears active cron job IDs around startup catch-up and manual-run execution, with the proposed regression tests or equivalent coverage, while preserving the existing durable recovery behavior as a complementary terminal-state safeguard.
