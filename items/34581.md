## Summary

Keep this PR open. Current main still uses strict `express.json()` parsing for authenticated Microsoft Teams Bot Framework webhook bodies and has no raw-body repair path for invalid JSON escape sequences, so the reported Azure backoff failure remains plausible. The PR is focused, but it needs maintainer review/rebase because it changes security-sensitive webhook parsing and HTTP 200 acknowledgement behavior after later auth hardening on main.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. The best path is a focused rebase in `extensions/msteams/src/monitor.ts` that preserves the current bearer and Bot Framework JWT gates before body parsing, keeps `MSTEAMS_WEBHOOK_MAX_BODY_BYTES`, repairs only truly bare invalid JSON escapes, returns HTTP 200 only for authenticated unrecoverable Bot Framework JSON to avoid Azure backoff, and adds integrated middleware tests for repair success, unrecoverable acknowledgement, adapter invocation/no-invocation, ordering, and payload limits.

## Risks / Open Questions

- Landing the stale PR without rebasing could disturb the current bearer/JWT-before-body-parse security ordering or the existing webhook body-size limit.
- Returning HTTP 200 for unrecoverable malformed JSON is intentional for Azure backoff avoidance, but it is security- and observability-sensitive and should apply only after Bot Framework authentication succeeds.
- The PR's helper-level repair tests are useful, but they do not by themselves prove full Express middleware ordering, adapter call/no-call behavior, payload-limit preservation, or exact HTTP status behavior on current main.
- Regex-based JSON repair needs careful maintainer review because the PR discussion already identified edge cases around valid doubled backslashes and invalid bare escapes.
