## Summary

Keep #40864 open. Current main supports `sessions_send` fire-and-forget acknowledgement and A2A announce follow-up, but it still does not provide the requested durable per-agent inbox / at-least-once delivery queue for `sessions_send` timeout or failure recovery. Existing persistent queues cover outbound channel delivery and restart-continuation recovery, not general agent-to-agent `sessions_send` delivery.

## Best Possible Solution

Keep the issue open and implement this as a core session-tool/gateway reliability feature, likely by reusing or extending the existing session-delivery queue primitives for `sessions_send`: persist before dispatch, return a delivery/ack identifier, drain on startup/heartbeat or wake, track idempotency, and add regression tests for timeout, gateway restart, duplicate replay, and final acknowledgement semantics. Docs should clarify the difference between immediate `accepted`, target-run completion, and durable delivery acknowledgement.

## Risks / Open Questions

- A durable at-least-once implementation will need explicit duplicate-suppression semantics so replayed messages do not create repeated agent turns.
- The existing outbound and restart-continuation queues are easy to confuse with this request, but they do not currently close the `sessions_send` reliability gap.
