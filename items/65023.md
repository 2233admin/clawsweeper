## Summary

Keep this PR open. Current main still does not implement the proposed `google-genai` provider, `google-genai-sdk` auth mode, or `models.providers.google-genai.vertexai` config contract, while the existing bundled Google plugin uses a different `google` / `google-vertex` surface. The PR is plausible related work for Google Vertex/ADC issues, but it needs maintainer design and security review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is either to revise it so unified Google GenAI/Vertex auth fits the bundled Google plugin/provider architecture and current transport behavior bar, or supersede it with a narrower PR that fixes the remaining `google-vertex` ADC/proxy behavior in the existing provider path with focused tests, docs, and request-policy preservation.

## Risks / Open Questions

- The PR changes shared model-provider config/auth contracts, including making `baseUrl` optional and adding a keyless mode, so it needs explicit maintainer API design review.
- The PR's keyless auth path must be propagated through every caller that currently requires an API key, including `/btw` and simple completion paths, not only embedded run auth checks.
- A new SDK transport must preserve current Google behavior for system prompts, tools, generation controls, cancellation, guarded fetch/proxy/TLS/request policy, usage, and finish metadata.
- Importing `@google/genai` from core without root dependency ownership would be a package-resolution and supply-chain hygiene risk.
