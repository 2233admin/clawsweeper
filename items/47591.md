## Summary

Keep open. Current main has adjacent pieces: per-channel preview/tool-progress updates on some channels, typing keepalive, and queued outbound recovery/retry. But I did not find the requested gateway-native silence watchdog, per-turn response delivery confirmation with correlation IDs, operator alerts on final delivery failure, or generic no-response alerting.

## Evidence

- **Current visibility is channel streaming, not a gateway watchdog:** The streaming docs describe current operator-visible progress as channel block streaming and preview streaming, with tool-progress preview edits on Discord/Slack/Telegram/Mattermost. They explicitly frame this as channel behavior rather than a gateway-enforced watchdog.
  - file: [docs/concepts/streaming.md:12](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/docs/concepts/streaming.md#L12)
- **Preview streaming is optional, not a default runtime guarantee:** Telegram channel config documents `streaming` as default off and opt-in to avoid preview-edit rate limits. That does not match the issueâ€™s proposed default-on safety guarantee.
  - file: [docs/gateway/config-channels.md:184](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/docs/gateway/config-channels.md#L184)
- **Mid-turn tool visibility is implemented inside channel adapters:** Telegram currently pushes `onToolStart`, item events, plan updates, approvals, command output, and patch summaries into draft preview progress lines. Similar channel-local hooks exist in Discord and Slack, which is partial mitigation but not a gateway-native silence timer.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:969](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/extensions/telegram/src/bot-message-dispatch.ts#L969)
- **Delivery handling is retry/recovery, not per-turn confirmed delivery with alerts:** The retry policy explicitly warns that generic retries on non-idempotent sends can duplicate messages, and pending-delivery recovery only replays queued entries on startup with backoff/move-to-failed behavior. I did not find per-turn `response_id` correlation, operator alerts, or final no-response detection.
  - file: [src/infra/retry-policy.ts:84](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/infra/retry-policy.ts#L84)
- **Silent zero-visible-reply turns remain valid behavior on main:** OpenClaw still treats exact `NO_REPLY` as a supported silent outcome, with groups/channels and internal orchestration allowing silence by default. That means the proposed generic `alertOnNoResponse` behavior is not implemented today.
  - file: [docs/concepts/messages.md:154](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/docs/concepts/messages.md#L154)
- **No matching watchdog/delivery-confirmation config or runtime symbols found:** Repo-wide search found no current `silenceWatchdog`, `responseDelivery`, `alertOnFailure`, `alertOnNoResponse`, `preserveUndelivered`, `resetOnStreaming`, or `resetOnTypingOnly` symbols on main.
  - command: `rg -n "silenceWatchdog|responseDelivery|alertOnFailure|alertOnNoResponse|preserveUndelivered|resetOnStreaming|resetOnTypingOnly|maxBeacons|thresholdSeconds" src docs extensions`

## Risks / Open Questions

- Existing preview/tool-progress support reduces silence only on some channels and only when streaming is enabled; it does not provide the runtime-enforced guarantee requested here.
- Outbound queue recovery helps after failures or restarts, but it is not the same as authoritative per-turn delivery confirmation plus operator-visible failure alerting.
- Any future no-response detector will need policy exceptions for intentional `NO_REPLY` flows to avoid false alarms on groups, internal orchestration, and subagent-related silence.
