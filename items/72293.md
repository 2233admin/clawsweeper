## Summary

Keep #72293 open. Current main has some adjacent hardening, including session-store mapping for lifecycle events that explicitly carry `stopReason: "aborted"`, but the subagent completion path still has gaps: `agent.wait` and the subagent registry primarily classify terminal state from `data.aborted`, so a resolved run whose final assistant message is `stopReason: "aborted"` can still be treated as an OK completion and announced as `(no output)` rather than a parent-visible failure.

## Best Possible Solution

Keep the issue open and fix the core subagent lifecycle/announce path. The likely fix is to normalize assistant-terminal `stopReason: "aborted"` and abort-like `errorMessage` into a failed or aborted run outcome even when `data.aborted` is false, propagate that through `agent.wait`, subagent run records, `abortedLastRun`, and announce delivery, then add a regression test for a subagent transcript ending with `content: []`, `stopReason: "aborted"`, and no visible output.
