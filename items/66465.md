## Summary

Keep open. Current main still delegates ACPX subprocess ownership to the pinned `acpx@0.6.1` runtime, and neither OpenClaw’s ACPX plugin config nor the published runtime contract exposes persistent stderr logging, a log directory, or retention/redaction controls. The prior Codex review comment is consistent with the current checkout; related PR #53997 has useful artifact ideas but is still open/unmerged and not present on main.

## Best Possible Solution

Keep this issue open. The best path is to add bounded, redacted per-process stderr teeing in upstream `openclaw/acpx` `AcpClient.start()`, expose typed runtime options for log directory and retention, wire those through `extensions/acpx` config/schema/manifest/docs, add startup and post-initialize stderr capture tests, then bump the bundled `acpx` dependency in OpenClaw. Review #53997 for artifact-design ideas, but treat it as an implementation candidate until it is merged or superseded.

## Risks / Open Questions

- Persisting raw child stderr can capture secrets, auth diagnostics, filesystem paths, prompt/tool content, or environment details; any implementation needs redaction, restrictive permissions, retention, and clear operator discoverability.
- The scoped ACPX guide says reusable ACP runtime behavior belongs in `openclaw/acpx`; implementing only in the OpenClaw wrapper risks dependency drift unless the upstream runtime exposes typed options and OpenClaw wires them through.
- Open PR #53997 may overlap with the artifact design, but it is open/unmerged and not a reason to close this issue before the work lands.
