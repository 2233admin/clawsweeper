## Summary

Keep open. Current `main` does not show an OpenClaw-side implementation of persistent ACPX child `stderr` logging, the bundled ACPX plugin config/docs expose no such control surface, and the report is recent and concrete. The evidence also suggests the real fix likely belongs in the shared `acpx` runtime rather than being auto-closed from OpenClaw core cleanup.

## Evidence

- **Runtime wrapper forwards only existing ACPX options:** The bundled ACPX service constructs `AcpxRuntime` with `cwd`, `sessionStore`, agent registry, probe agent, MCP servers, permission settings, and timeout only. There is no OpenClaw-side stderr tee/log-file wiring here.
  - file: [extensions/acpx/src/service.ts:43](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/acpx/src/service.ts#L43)
  - command: `nl -ba extensions/acpx/src/service.ts | sed -n '43,60p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **No stderr log config surface in bundled ACPX schema:** The ACPX plugin config schema exposes `cwd`, `stateDir`, `probeAgent`, permission knobs, timeout, MCP servers, and agent overrides, but no `stderrLogDir`, retention, or similar post-mortem stderr persistence setting.
  - file: [extensions/acpx/src/config-schema.ts:26](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/acpx/src/config-schema.ts#L26)
