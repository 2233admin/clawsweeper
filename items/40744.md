## Summary

Keep PR #40744 open. Current main still does not strip malformed non-array `required` values from shared tool schema normalization, so the Bailian/GLM-5 provider rejection described in #40568 can still survive the generic/OpenAI-compatible path. The PR needs rebasing onto the newer `normalizeToolParameterSchema` structure rather than closure.

## Best Possible Solution

Keep the PR open and ask for an update against current main. The best fix is to strip own non-array `required` fields in shared tool-parameter normalization before provider-specific cleaning, preserve valid arrays, cover the Bailian/OpenAI-compatible top-level path, and add direct nested Gemini coverage if the Gemini cleaner is changed.
