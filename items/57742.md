## Summary

## Risks / Open Questions

- Explicit `/acp spawn codex` on `webchat`/TUI still appears to be a fallback path rather than the primary supported same-conversation workflow; this closure is based on the requested workflow now being implemented via native Codex conversation binding on `main`, not on ACP gaining webchat thread/conversation binding parity.

## Close Comment

Closing because current `main` now has a supported same-session Codex workflow for this use case.

- Codex review on `main` `7e49cc87`: the docs now direct current-conversation Codex control to the native Codex app-server path, not ACP (`docs/tools/acp-agents.md:15`, `docs/tools/acp-agents.md:29`).
- `/codex bind` binds the invoking conversation itself through plugin conversation binding, so the active chat stays the host surface instead of falling back to agent `main` (`extensions/codex/src/command-handlers.ts:238`, `src/plugins/commands.ts:232`, `src/channels/conversation-resolution.ts:313`).
- Native Codex now sends `request_user_input` prompts back to the originating chat and consumes the next queued reply there, which covers the interactive blocker/question loop this report was asking for (`extensions/codex/src/app-server/user-input-bridge.ts:46`, `extensions/codex/src/app-server/run-attempt.ts:388`).
- The behavior is backed by tests and changelog entries on `main` (`extensions/codex/src/app-server/user-input-bridge.test.ts:13`, `extensions/codex/src/app-server/run-attempt.test.ts:648`, `CHANGELOG.md:54`).
- ACP remains documented as an explicit fallback/background path for Codex, so this report's workflow gap is considered implemented on `main` via the native Codex binding path rather than by adding `webchat` ACP thread binding.
