## Summary

Keep open. Current `main` has partial mitigation for orphaned trailing user turns during provider replay, but I did not find strong evidence that `webchat` `chat.send` is fixed at the transcript level or that a single send is now guaranteed to persist exactly one user row in session JSONL.

## Evidence

- **Webchat user-update path does not attach transcript idempotency:** `chat.send` builds the eager user transcript update with only `role`, `content`, `timestamp`, and optional media fields, then emits it. This path does not include an `idempotencyKey` or transcript `messageId` for the user turn.
  - file: [src/gateway/server-methods/chat.ts:741](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/gateway/server-methods/chat.ts#L741)
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Assistant transcript appends have idempotency dedupe, but that is separate infrastructure:** `appendExactAssistantMessageToSessionTranscript` checks `transcriptHasIdempotencyKey(...)` before appending and reuses the existing message id. That proves transcript-level dedupe exists for this helper, but I did not find equivalent proof on the `webchat` user-turn path.
  - file: [src/config/sessions/transcript.ts:194](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/config/sessions/transcript.ts#L194)
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Current main only repairs a trailing orphaned user leaf during replay:** The embedded runner now merges/removes one orphaned trailing user message before prompt replay. This is a runtime repair step for provider turn ordering, not a direct fix proving duplicate JSONL user rows are no longer written.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2074](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/agents/pi-embedded-runner/run/attempt.ts#L2074)
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Tests cover single-orphan repair, not webchat JSONL dedupe:** The e2e coverage I found only asserts that an orphaned user message can be repaired and the run continues. I did not find a regression test for `webchat` where one `chat.send` results in exactly one persisted user transcript entry.
  - file: [src/agents/pi-embedded-runner.e2e.test.ts:657](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/agents/pi-embedded-runner.e2e.test.ts#L657)
  - command: `rg -n "chat.send.*transcript|user transcript update|duplicate user messages|orphaned user messages" src/gateway/*.test.ts src/gateway/server-methods/*.test.ts src/agents/*.test.ts src/agents/pi-embedded-runner*.test.ts`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Gateway tests still validate emitted update shape only:** `chat.directive-tags` tests assert that `chat.send` emits a user transcript update with `role/content/timestamp`, including no-one-row-per-send JSONL assertion or retry/dedup regression for persisted user messages.
  - file: [src/gateway/server-methods/chat.directive-tags.test.ts:1647](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/gateway/server-methods/chat.directive-tags.test.ts#L1647)

- The checkout is shallow and `gh` was unavailable without `GH_TOKEN`, so historical PR attribution was limited; that lowers confidence for an `implemented_on_main` close.
