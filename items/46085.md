## Summary

Current main already avoids the reported false-error path for default/no-target heartbeats: unset heartbeat delivery resolves to `none`, cron/exec heartbeat prompts stay internal-only when delivery is disabled, and regression coverage asserts the run finishes successfully without outbound sends.

## Close Comment

Closing from a Codex review because current `main` already avoids the reported failure mode. With no explicit heartbeat target, `resolveHeartbeatDeliveryTarget` now resolves to `target-none` (`src/infra/outbound/targets.ts:103`), and `runHeartbeatOnce` treats that case as an internal-only cron/exec heartbeat and returns success without attempting outbound delivery (`src/infra/heartbeat-runner.ts:1191`). The regression test at `src/infra/heartbeat-runner.returns-default-unset.test.ts:1542` covers the same shape: `heartbeat.target: "none"`, cron event queued, status stays `ran`, and outbound sends stay at zero. `CHANGELOG.md:3978-3979` also records the heartbeat-routing/defaults change that made blocked or disabled cron heartbeats internal-only. The low-level empty-account normalization still exists, but the false `Slack bot token missing for account "default"` path described here is implemented away on current `main`.
