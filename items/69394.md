## Summary

Keep PR #69394 open. Current `main` still strips a trailing `NO_REPLY` from auto-reply text and delivers the remaining preamble, and the colocated tests still encode that behavior. The PR addresses a real unresolved auto-reply bug, but it needs maintainer review/changes because its `stripped.length < text.length` guard can falsely suppress non-trailing `NO_REPLY` text with trailing whitespace. Related PR #66755 is narrower and does not clearly supersede this broader trailing-token proposal.

## Best Possible Solution

Keep this PR or one canonical sibling open. The best fix is to move the cron path's safer trailing-token detection pattern into the shared auto-reply normalization path, add regression tests for reasoning preamble suppression and non-trailing `NO_REPLY` with trailing whitespace, and explicitly decide how media/channelData and previously substantive `#30916` cases should behave.
