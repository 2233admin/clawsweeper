## Summary

Keep open. Current main still implements the strip-and-deliver behavior for non-streaming auto-reply payloads that end with `NO_REPLY`, and tests explicitly preserve that behavior. Related PR #66755 is open but narrower, while #49854 was closed unmerged, so this PR is not safely implemented, duplicate, or superseded.

## Best Possible Solution

Keep this PR open for maintainer review. The likely path is to resolve #66701 by choosing the intended policy between #66755's narrow reasoning-marker approach and #69394's broader trailing-token approach, implement a precise trailing-token detector in `normalizeReplyPayload`, update the #30916/#19537 tests to match that policy, and track streaming-block suppression separately if needed.

## Risks / Open Questions

- Merging #69394 unchanged may regress legitimate messages that contain `NO_REPLY` non-trailingly with trailing whitespace, as noted in the existing review comment.
- The remaining product choice is real: broad trailing-token suppression fixes more reasoning leaks, but it reverses currently tested #30916 behavior for substantive text ending in `NO_REPLY`.
- The PR explicitly does not address already-streamed block delivery before a final `NO_REPLY` arrives, so a separate streaming-buffering follow-up may still be needed.
