## Summary

Keep open. Current main still lacks the reported cross-HOME update safeguard: update/service discovery and forced service refresh are rooted in the current process environment, while systemd unit lookup and config defaults still follow the effective HOME. Existing hardening covers same-invocation env normalization and stale service env filtering, but not detecting, preserving, migrating, warning, or blocking when an update runs as HOME=/root after the gateway/config lived under /home/openclaw.

## Best Possible Solution

Keep this issue open and fix the update service refresh/install path directly. Before regenerating an installed gateway service, discover the prior service/config root even when the updater runs under a different HOME, compare HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH, then either preserve the prior root when safe or emit an explicit migration/warning/block. Add a regression test for a service installed with HOME=/home/openclaw and refreshed by an update process running with HOME=/root.

## Evidence
