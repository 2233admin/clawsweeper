## Summary

Keep open. Current `main` makes the PR body’s plaintext `ws://` reproduction mostly obsolete by rejecting non-loopback insecure WebSocket targets at the gateway client layer, but `gateway-status` still does no hostname/IP SSRF validation of its own and will pass arbitrary `wss://` targets through to probing. The proposed patch is also not ready to close out the item because the review discussion identified a real IPv4-mapped-IPv6 bypass that the PR’s string-prefix check would miss, while OpenClaw already has stronger shared SSRF helpers that cover that case.

## Evidence

## Risks / Open Questions

- Closing this PR now would discard a still-valid hardening gap: `gateway-status` itself still lacks host-level SSRF validation for accepted `wss://` targets and config remote URLs.
- Treating the item as already fixed would be incorrect because the only existing protection relevant to the PR body is the later plaintext-`ws://` transport guard, not the requested hostname/IP validation in `gateway-status`.
- Merging the PR unchanged would leave the IPv4-mapped-IPv6 link-local bypass described in review, so the item likely needs revision rather than closure.
