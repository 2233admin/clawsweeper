## Summary

Keep open. Current `main` still suppresses duplicate assistant delivery only from committed successful messaging-tool sends, while this issue is specifically about a failed send followed by a retry in the same turn. The current code and tests do not show a fix for that path; they instead preserve non-suppression after tool error, so the reported duplicate remains plausible on `main`.

## Evidence

- **Pending messaging sends are tracked but not used for suppression:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:693` records message-tool targets/texts into pending maps at tool start, but `src/agents/pi-embedded-subscribe.handlers.tools.ts:862` only promotes them into `messagingToolSentTexts` / `messagingToolSentTargets` when `!isToolError`. That means a failed first send contributes no committed suppression state before a same-turn retry.
  - file: [src/agents/pi-embedded-subscribe.handlers.tools.ts:693](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-subscribe.handlers.tools.ts#L693)
