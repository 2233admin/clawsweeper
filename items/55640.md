## Summary

Current `main` still appears vulnerable to the reported retry-in-same-turn duplicate-delivery case. The code tracks pending messaging sends at tool start, but all suppression paths inspected still read only committed successful sends, so a retried `message` tool call can still leave suppression blind until the retry succeeds.

## Evidence

- **Pending targets are tracked but not committed on failure:** `handleToolExecutionStart` stores a pending messaging target at tool start, but `handleToolExecutionEnd` only moves it into `messagingToolSentTargets` when `!isToolError`.
  - file: [src/agents/pi-embedded-subscribe.handlers.tools.ts:672](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/pi-embedded-subscribe.handlers.tools.ts#L672)
  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.tools.ts | sed -n '672,686p' ; nl -ba src/agents/pi-embedded-subscribe.handlers.tools.ts | sed -n '842,858p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Streaming suppression explicitly ignores pending sends:** `emitBlockChunk` has an inline comment saying it checks only committed successful messaging-tool texts and does not use pending texts, because suppressing on pending could drop the only user-visible response if the tool later fails.
  - file: [src/agents/pi-embedded-subscribe.ts:630](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/pi-embedded-subscribe.ts#L630)
