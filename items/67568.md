## Summary

Keep open. Current `main` still writes session-memory filenames and headings from UTC ISO timestamps, so the reported local-day mismatch is real. There is partial mitigation when startup context reads daily memory, but no write-side fix on `main` or in the latest release evidence reviewed.

## Evidence

- **Session-memory hook still stamps filenames from UTC:** `saveSessionToMemory()` derives `dateStr` with `new Date(event.timestamp).toISOString().split("T")[0]`, then uses that for the output filename. It also derives the fallback slug and heading time from `toISOString()`, so the saved artifact date/time are UTC, not `agents.defaults.userTimezone`.
  - file: [src/hooks/bundled/session-memory/handler.ts:84](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/hooks/bundled/session-memory/handler.ts#L84)
  - command: `nl -ba src/hooks/bundled/session-memory/handler.ts | sed -n '83,185p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Write path explicitly labels saved session memory as UTC:** The bundled hook docs show the output heading as `# Session: 2026-01-16 14:30:00 UTC`, matching the current implementation rather than a user-local calendar day.
  - file: [src/hooks/bundled/session-memory/HOOK.md:32](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/hooks/bundled/session-memory/HOOK.md#L32)

This is a read-side workaround for exactly the class of mismatch reported, not a write-side fix.

That shows the codebase currently expects this mismatch to exist.

## Risks / Open Questions

- A fix will need a compatibility story for existing UTC-dated `memory/YYYY-MM-DD-slug.md` files, because startup-context currently includes differing UTC dates to avoid dropping them.
- Any change should audit other consumers that sort or group by date-prefixed memory filenames so behavior does not regress for historical artifacts.
