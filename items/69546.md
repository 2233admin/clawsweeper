## Summary

Keep #69546 open. The report is recent, has a concrete QQBot reproduction, and current main still has the same likely failure mode: QQBot starts its gateway with a config snapshot, passes that same snapshot into each inbound route resolution, while the gateway reload planner treats top-level `bindings` edits as dynamic/no-restart changes. Core routing itself does prefer peer bindings over account bindings when both are present, so the remaining issue is QQBot not reading the updated config at message time. The related ACP/plugin-binding reports in the provided context do not supersede this top-level route-binding hot-reload bug.

## Best Possible Solution

Keep the issue open and fix QQBot routing to resolve inbound route state from the current runtime config at message handling time, or otherwise provide a narrow live config getter to the QQBot engine before `resolveAgentRoute` and reply dispatch. Add a QQBot-focused regression test that starts with an account binding, updates top-level `bindings` with a peer-specific binding, and verifies the next C2C message matches `binding.peer` without restarting the gateway.

## Risks / Open Questions

- A fix needs focused QQBot regression coverage for changing top-level `bindings` after gateway start; otherwise this can regress silently because normal route precedence tests already pass.
- Restarting QQBot on every top-level binding edit would work around the stale snapshot but would conflict with the documented dynamic-read reload model; a fresh per-message config read is the more aligned direction.
