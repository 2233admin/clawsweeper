## Summary

Keep #69546 open. Current main still matches the reported failure mode: QQBot inbound routing resolves against the gateway-start config snapshot, while top-level `bindings` changes are classified as dynamic/no-restart, so updated peer bindings can be visible to CLI config readers without being used by an already-running QQBot gateway.

## Best Possible Solution

Keep the issue open. The best fix is to make QQBot inbound routing use the current runtime config at message handling time, for example by loading a fresh runtime config before `resolveAgentRoute` and reply dispatch or by passing a narrow live config getter into the QQBot engine. Add a regression test that starts QQBot routing with an account binding, updates top-level `bindings` with a peer-specific C2C binding, and verifies the next inbound message matches `binding.peer` without a gateway restart.
