## Summary

Keep open. Current main still has the reported default Chrome MCP spawn path and does not fully close the bounded-response or orphan-cleanup concern: existing-session profiles still default to `npx -y chrome-devtools-mcp@latest`, the browser plugin still constructs the upstream SDK `StdioClientTransport` directly, and `/tabs` reachability/listing paths still do not propagate the 300ms route budget into `listChromeMcpTabs`. Current mitigations reduce symptoms, but they are not a complete fix for this issue.

## Best Possible Solution

Keep this open and implement a focused browser-plugin fix: propagate timeout and AbortSignal through Chrome MCP availability/list calls and tab-management routes, close and force-clean failed pending sessions reliably, replace or adapt the Chrome MCP subprocess transport so stuck `npx` child trees are reaped, and add a regression that simulates a hung default `npx`/`client.connect`/`list_pages` path through `/tabs` proving bounded response time and cleanup. Keep `mcpCommand`/`mcpArgs` as a documented workaround, not the primary fix.

## Risks / Open Questions

- Current 30s handshake and tool-call timeout mitigations reduce some symptoms, so the remaining fix should be scoped to timeout/AbortSignal propagation and subprocess cleanup rather than a broad browser rewrite.
- The browser plugin should not deep-import core `src/agents` internals; if reusing OpenClaw's process-tree-aware stdio transport, expose a proper plugin-safe seam or implement an extension-local equivalent.
- No live macOS Chrome MCP reproduction was run in this read-only review, but current source, tests, docs, and upstream transport behavior are enough to show the reported default-path gaps remain.
