## Summary

Keep this PR open. Current main still lacks the `/v1/chat/completions` tool protocol behavior this PR targets: request `tools` / `tool_choice` parsing, forwarding `clientTools`, OpenAI-compatible `tool_calls` responses, streaming `delta.tool_calls`, and `tool_call_id` follow-up handling. The linked #54405 remains open and this PR is the focused implementation candidate, though the stream-finalization review concern still needs maintainer review or a fix before landing.

## Best Possible Solution

Keep this PR open. The best path is to review or rework it on current main, resolve the stream finalization concern, then land a focused gateway adapter fix for `/v1/chat/completions` that parses `tools` and `tool_choice`, forwards `clientTools`, emits OpenAI-compatible non-stream and streaming `tool_calls`, accepts `role: "tool"` plus `tool_call_id` follow-up messages, and keeps regression coverage in `src/gateway/openai-http.test.ts`.

- The open P1 review comment says the PR's stream finalization timeout may still close the SSE response before `agentCommandFromIngress` resolves under slow providers or event-loop stalls, dropping `delta.tool_calls` / `finish_reason: "tool_calls"`. That should be fixed or explicitly accepted before merge.
- The PR expands Chat Completions behavior to accept caller-supplied function-tool schemas on a full-operator endpoint. This matches the existing Responses contract, but review should confirm the Chat adapter only forwards definitions and returns pending tool calls, without adding server-side execution or broader auth/secrets behavior.
