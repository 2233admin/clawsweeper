## Summary

Keep this issue open. Current main still routes the reported OpenRouter/OpenAI-compatible model path through the guarded Undici-based provider fetch path, and the same user-facing error string is still produced for generic fetch/connection failures. The systemd proxy workaround in the discussion is useful evidence for some commenters, and related #58150 was closed as a duplicate of this issue, but I found no shipped RISC-V64-specific fix, diagnostic, or node:https transport fallback on current main.

## Best Possible Solution

Keep this issue open as the canonical RISC-V64/provider-transport tracker. Maintainers should ask reporters for exact board/distro, Node binary source, `process.arch`, `process.versions`, direct `fetch` and `node:https` results against the same provider, redacted gateway service proxy environment, and logs preserving the underlying provider failure cause. If stale proxy injection is confirmed, fix doctor/docs or service environment handling; if RISC-V64 Undici/fetch is confirmed, add a guarded transport fallback or explicit compatibility diagnostic in the shared provider transport path.
