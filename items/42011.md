## Summary

Keep open. Current `main` still appears to have the same gap: Control UI chat state is cleared by `chat` final/error/aborted events, but the `chat.send` started-run path still skips a fallback `broadcastChatFinal(...)` when dispatch settles without a terminal lifecycle event. The only explicit fallback I found is in `agent.wait`, which races lifecycle against dedupe and has a regression test for chat runs that finish without lifecycle events.

## Evidence

- **Control UI only leaves running state on chat terminal events:** `handleChatEvent(...)` clears `state.chatRunId` only for `payload.state === "final"`, `"aborted"`, or `"error"`. If none of those chat events arrive, the UI keeps the active run id and therefore stays in the Stop state.
  - file: [ui/src/ui/controllers/chat.ts:456](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/ui/src/ui/controllers/chat.ts#L456)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '437,500p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Started chat runs still rely on lifecycle-driven finalization:** In `chat.send`, once `onAgentRunStart` flips `agentRunStarted = true`, the settled `.then(...)` path no longer calls `broadcastChatFinal(...)`; it only emits a transcript update in the `else` branch. The same block still writes a success dedupe entry afterward, so non-UI waiters can finish even though Control UI did not get a terminal chat event.
  - file: [src/gateway/server-methods/chat.ts:2498](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/gateway/server-methods/chat.ts#L2498)

  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Fallback exists for agent.wait, not for Control UI chat events:** `agent.wait` explicitly races lifecycle against terminal gateway dedupe, and there is a regression test named `agent.wait resolves chat.send runs that finish without lifecycle events`. That shows current main still expects some chat runs to complete without lifecycle events, but the fallback is wired to `agent.wait`, not the Control UI chat event stream.
