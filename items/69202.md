## Summary

Keep #69202 open. Current main has a useful partial recovery path: context-overflow-shaped embedded errors and classified compaction failures rotate the channel session and send a user-facing reset notice. That does not yet cover the full report, which asks for recovery from compaction failure for any reason, including timeout, API error, quota/rate limit, and reasoning-constraint failures. The current classifier intentionally excludes several of those cases from compaction-failure recovery, and timeout compaction failures still fall through to normal timeout/error handling rather than force-rotating to a fresh session.

## Best Possible Solution

Keep the issue open as the canonical hard-failure watchdog item. The maintainer-quality fix should broaden compaction-failure classification/recovery so timeout, provider 4xx/5xx, quota/rate-limit, and reasoning-constraint compaction failures either try configured compaction fallbacks or rotate to a fresh channel session with a user-visible notice. It should also clear/reset lifecycle status for the new session and add regression coverage around the non-overflow failure classes named in the report. #52012 and #64962 can remain narrower follow-ups for fallback model resolution and failed timeout-compaction escalation details.
