## Summary

Closing the PR. Current `main` treats `workspace` and `agentDir` path overrides as intentional admin-configurable host paths, so the reported behavior does not reproduce as a bug against the present product contract. The proposed patch would also break current documented/tested behavior and is not mergeable as submitted.

## Risks / Open Questions

- If maintainers want to reduce trust in admin-configured `workspace`/`agentDir`, that is a product/security design change and would need coordinated docs, config, and migration work.
- Current behavior still allows trusted config to point at arbitrary host paths; the intended protection boundary is sandbox/tool policy, not implicit path clamping in these resolvers.

## Close Comment

Thanks for the PR. I reviewed this with Codex against current `main`, and I’m closing it because the reported behavior does not reproduce as a bug under the current OpenClaw contract.

Today `resolveAgentWorkspaceDir()` and `resolveAgentDir()` intentionally honor configured host paths in `src/agents/agent-scope-config.ts:149` and `src/agents/agent-scope-config.ts:174`. The docs also explicitly say an agent workspace is the default cwd, not a sandbox, and that absolute paths can reach other host locations unless sandboxing is enabled (`docs/concepts/multi-agent.md:48`). Per-agent `agentDir` overrides are likewise documented and tested as supported config (`docs/gateway/config-agents.md:892`, `src/commands/agents.test.ts:13`, `src/config/config.multi-agent-agentdir-validation.test.ts:8`).

Separately, the submitted patch is not in a mergeable state: the new helper is scoped incorrectly in the diff, and the review comments correctly note remaining gaps around absolute workspace paths and symlink handling. Since this PR would change documented/tested behavior rather than fix an accepted bug on `main`, I’m closing it.
