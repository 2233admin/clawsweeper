## Summary

Keep this PR open. Current main still falls back to queued followup when `steer` arrives during an active but non-streaming run, so the linked soft-steer request #10960 is not implemented on main. The PR remains a plausible implementation candidate, but the current diff aborts before active-run queue policy can drop heartbeat turns, creating a real availability/regression risk.

## Best Possible Solution

Keep this PR open as an implementation candidate. Revise the implementation so active non-streaming `steer` aborts/unblocks only after queue policy confirms the turn should become a real followup, preserve the streaming injection fast path, keep active heartbeat turns dropped without aborting user runs, clean up the misleading `steer-backlog` test name and unused `isStreaming` contract noted in review, then rebase and run the targeted changed tests.
