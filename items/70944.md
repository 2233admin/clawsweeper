## Summary

Keep open. Current `main` still limits `identityLinks` to routing/session-key canonicalization and does not expose or thread a Discord `sender_principal`/`TrustedSenderPrincipal` signal into inbound prompt metadata. The PR is active, coherent, and already incorporated review feedback on the PluralKit sender-ID edge case, so there is no conservative auto-close basis.

## Evidence

That means the requested Discord prompt-context behavior is not already shipped.
