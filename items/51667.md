## Summary

Keep open. Current `main` still handles inbound voice notes as transcription-only media understanding, then injects transcript text into the agent context. The model catalog/schema and chat transports are still image-only for non-text media, so native audio parts for omni models are not implemented on `main` as of `aa3a5f9ff7907c2a41d6259d24b529bda41f293a`.

## Evidence

- **Docs describe transcription-only audio flow:** The user-facing audio docs say inbound audio is processed by media understanding, then on success `Body` is replaced with an `[Audio]` block and `{{Transcript}}` is set; all config examples use transcription models such as `gpt-4o-mini-transcribe`.
  - file: [docs/nodes/audio.md:12](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/docs/nodes/audio.md#L12)

- **Audio runner requires transcription provider:** The media-understanding runner hard-codes audio to `provider.transcribeAudio`, errors if that hook is absent, and returns output kind `audio.transcription`. There is no branch here for forwarding audio as a native model input part.
  - file: [src/media-understanding/runner.entries.ts:575](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/media-understanding/runner.entries.ts#L575)

## Risks / Open Questions

- Implementing this is broader than a config toggle: it would require widening model input contracts, provider catalogs, and transport serialization beyond image-only binary content.
- The audit/snapshot concern raised in the issue comments is still unresolved in current `main`; native audio would need an explicit continuity/audit story rather than reusing the transcript-only path.
