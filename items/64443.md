## Summary

Keep #64443 open. The report is recent, concrete, and hardware-specific; current main still has visible macOS WebChat compositor-heavy paths, and I found no source, test, changelog, or related issue evidence proving the WindowServer CPU issue is fixed or obsolete. The two issue comments are generic external-guide comments and do not provide OpenClaw evidence or a workaround.

## Best Possible Solution

Keep the issue open and treat it as a macOS native WebChat rendering-performance bug. Best maintainer path: profile the visible chat window on similar Intel Retina hardware, identify whether NSVisualEffectView/material layers, transparent windows, streaming scroll animations, or repeat animations drive WindowServer load, then ship a reduced-compositing or low-motion fallback with focused coverage and a documented workaround if needed.
