## Summary

Keep #69513 open. Current main still validates announce cron delivery channels at create/update time, but it does not validate delivery.accountId or failureDestination.accountId against configured channel accounts. The reported invalid account can still pass schema/normalization as a non-empty string and is later preserved as the explicit delivery account. The related fix PR #69518 was closed unmerged, so it is not shipped on main.

## Best Possible Solution

Keep the issue open and land a focused gateway-side fix similar to the closed #69518 approach: during cron.add and cron.update, validate delivery.accountId and announce-mode delivery.failureDestination.accountId against cfg.channels.<channel>.accounts when an explicit accounts map exists, while preserving omitted accountId and flat single-account configs. Add regression tests in src/gateway/server-methods/cron.validation.test.ts or src/gateway/server.cron.test.ts for invalid and valid account IDs, including update/failure-destination coverage if implemented.
