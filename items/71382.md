## Summary

Keep open. Current `main` does not already implement a WhatsApp channel/account `readOnly` control, and existing `session.sendPolicy` is narrower than the PR's goal: it suppresses agent reply delivery, but it does not hard-gate WhatsApp transport-level outbound actions such as connect presence, pairing replies, or read receipts. The PR is therefore addressing a real gap, but the provided review discussion also surfaces unresolved behavior issues in the patch, so conservative maintainer cleanup should not auto-close it.

## Evidence

- **`session.sendPolicy` is documented as a session-level delivery policy, not a WhatsApp transport mute:** The config docs describe `session.sendPolicy` under session settings and say it matches by `channel`, `chatType`, `keyPrefix`, or `rawKeyPrefix`; there is no WhatsApp-specific channel/account `readOnly` control documented on `main`.
  - file: [docs/gateway/config-agents.md:1144](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/docs/gateway/config-agents.md#L1144)
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **`sendPolicy` is enforced inside reply dispatch, not as a channel transport guard:** `dispatch-from-config.ts` resolves `sendPolicy` and comments that under `sendPolicy: "deny"` the agent still processes inbound but no outbound reply/notice/indicator is allowed. That scope is the reply dispatcher, not WhatsApp socket behavior generally.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:503](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/auto-reply/reply/dispatch-from-config.ts#L503)

  - file: [extensions/whatsapp/src/inbound/monitor.ts:133](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/whatsapp/src/inbound/monitor.ts#L133)
