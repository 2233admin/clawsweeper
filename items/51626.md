## Summary

Keep open. Current `main` still does not route `sessions_send` announce delivery through the target session's stored Discord `lastAccountId`/`deliveryContext.accountId`. For Discord, the announce target is still derived from the session key without an account id, and the Discord send path resolves a missing account id back to the channel default account.

## Evidence

- **Discord announce target still skips session lookup:** `resolveAnnounceTarget()` returns the parsed fallback immediately unless the channel plugin sets `meta.preferSessionLookupForAnnounceTarget`. Only the later lookup path reads `deliveryContext.accountId` / `lastAccountId`.
  - file: [src/agents/tools/sessions-announce-target.ts:26](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/tools/sessions-announce-target.ts#L26)

  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Current test still expects Discord announce targets without accountId:** The `resolveAnnounceTarget` test for Discord expects `{ channel: "discord", to: "group:dev" }` and explicitly asserts `sessions.list` is not called. The neighboring WhatsApp tests are the ones that hydrate `accountId` from session state.
