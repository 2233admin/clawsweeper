## Summary

Current main only partially covers this PR. Shared outbound delivery now propagates `audioAsVoice` to generic `sendMedia` and iMessage text chunking is configured, but the iMessage plugin still lacks a `sendPayload` path, drops `audioAsVoice` at its runtime boundary, and the low-level iMessage sender still converts empty audio attachments into `<media:audio>`. The PR is stale against the moved `extensions/imessage` layout, but the underlying bug remains actionable.

## Best Possible Solution

Keep this PR open and ask for a narrowed rebase or rewrite against `extensions/imessage`. The best fix is an iMessage-owned `sendPayload` path on the current plugin adapter that forwards `audioAsVoice` through `channel.runtime.ts` into `send.ts`, suppresses the generic `<media:audio>` placeholder for voice-intent audio, sends visible caption text separately when needed, preserves configured text chunking for text-only payloads, and adds focused iMessage regression tests.
