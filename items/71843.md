## Summary

Keep PR #71843 open. Current main still lets Slack `already_reacted` bubble from the agent `react` action because `handleSlackAction()` awaits `reactSlackMessage()` without a catch, and the shared helper itself directly awaits `client.reactions.add()`. The PR is a focused live fix for linked issue #69005, but maintainer feedback now asks for the behavior to move into `reactSlackMessage()` so every Slack caller gets idempotent duplicate-reaction handling.

## Best Possible Solution

Keep #71843 open for maintainer review. The best path is to revise this PR or land an equivalent Slack plugin fix that makes `reactSlackMessage()` treat Slack `already_reacted` as idempotent success using the structured SDK error shape (`data.error`) with a safe fallback, preserves propagation for unrelated errors, removes the now-duplicate status-adapter catch where appropriate, and keeps focused helper-level plus agent-runtime regression coverage. Maintainers can then use this PR to supersede the older overlapping Slack reaction PRs if it also covers their remaining scope.
