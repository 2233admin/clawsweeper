## Summary

Keep open. Current `main` has a real docs/UX mismatch around Android setup-code pairing: the Android setup-code flow passes a `bootstrapToken`, and gateway tests assert QR/setup-code bootstrap auto-approves with no pending pairing request left behind, but current CLI/docs still tell users to run `openclaw devices approve <requestId>`. That approval path intentionally returns `unknown requestId` when nothing is pending, which matches the report too closely to close as obsolete or unreproducible.
