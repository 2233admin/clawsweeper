## Summary

Keep open. Current `main` still implements Feishu typing with `im.messageReaction.create/delete` and stores a `reaction_id`, while docs and tests still describe that behavior as "typing reactions." The issue is recent, concrete, and from a non-maintainer, and I did not find a main-branch fix that migrates Feishu to a separate native typing-status API.

## Evidence

- **Current main still uses reaction API for typing:** `addTypingIndicator()` calls `client.im.messageReaction.create(...)` with `reaction_type.emoji_type = "Typing"`, and `removeTypingIndicator()` calls `client.im.messageReaction.delete(...)` by `reaction_id`. That is the same reaction-based approach described in the issue, not a separate typing-status API.
  - file: [extensions/feishu/src/typing.ts:99](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/typing.ts#L99)
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Reply flow still wires channel typing through that helper:** Feishu reply dispatch still starts typing by calling `addTypingIndicator()` and stops typing by calling `removeTypingIndicator()` on the reply target message, so current runtime behavior still depends on the reaction implementation.
