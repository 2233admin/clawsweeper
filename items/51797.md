## Summary

Current main no longer matches the reported Windows exec-space-loss mechanism: host exec on Windows sends the full command as a single PowerShell `-Command` payload, regression tests cover space-preserving Windows/PowerShell parsing, and the Windows exec guidance now explicitly tells users not to wrap commands with `cmd /c` or `powershell -Command`.

## Close Comment

Closing from a Codex review of current `main` (`aa3a5f9ff7907c2a41d6259d24b529bda41f293a`). I could not match the reported Windows `exec` “parameter space loss” behavior against the current implementation. On Windows, host exec now routes through PowerShell (`src/agents/shell-utils.ts:55`) and passes the full command through as a single `-Command` payload (`src/agents/bash-tools.exec-runtime.ts:669`) rather than rebuilding `/PID`-style native arguments from split tokens at the OpenClaw boundary. Current regression tests also cover Windows quoted executables and PowerShell payloads with space-containing arguments (`src/infra/exec-approvals-analysis.test.ts:474`), and the Windows exec guidance now explicitly tells users not to wrap commands in `cmd /c` or `powershell -Command` (`src/agents/bash-tools.descriptions.ts:33`, `docs/tools/exec.md:75`). Based on current `main`, this report no longer reproduces as filed, so I’m closing it. If you can still reproduce it on `v2026.4.22` or newer, please open a new issue with the exact command sent, the exact command OpenClaw displayed, the actual output/error, and whether it was `gateway`, `sandbox`, or `node` exec.
