## Summary

Current main still does not implement Telegram inline-query UX. The Telegram plugin has inline-query names only in fallback/update/status metadata, while runtime handlers cover reactions, callback queries, messages, migration, and channel posts only. Config, generated metadata, docs, and tests expose inline buttons, not inline queries. The closed PR #51963 remains useful reference material but was not merged or shipped.

## Best Possible Solution

Keep this issue open as the canonical Telegram inline-query tracker. The likely fix belongs in the bundled Telegram plugin: add an explicit inline-query config/schema/docs surface, handle inline_query and chosen_inline_result updates, authorize senders conservatively, route bounded inline requests through the agent flow, answer with Telegram InlineQueryResultArticle results, and add focused Telegram plugin tests. Treat closed PR #51963 as reference material unless a new reviewed implementation lands.

## Risks / Open Questions

- Inline queries can originate from arbitrary Telegram chats, so a future implementation needs explicit sender authorization, privacy/session boundaries, rate limiting or debounce, and bounded execution before routing requests through the agent flow.
- The existing status output can mislead operators because inlineQueries reflects Telegram/BotFather account capability, not whether OpenClaw can answer inline queries.
