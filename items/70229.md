## Summary

Keep PR #70229 open. Current main still hard-codes Docker `--security-opt no-new-privileges` for sandbox containers and does not expose `sandbox.docker.noNewPrivileges` in config types, validation, docs, generated schema, or dangerous-config reporting. The linked bug #43996 remains open and has concrete reporter + maintainer discussion confirming the host-level EPERM trigger. The PR branch is a narrow, relevant implementation with secure default behavior and follow-up coverage for the bot review comment.

## Best Possible Solution

Keep #70229 open for maintainer review or landing. The right product/code path is to add `sandbox.docker.noNewPrivileges?: boolean`, resolve it with a secure default of true, guard the Docker arg emission, expose it in schema/types/docs, and report `noNewPrivileges=false` as a dangerous sandbox config flag. That directly addresses #43996 without removing the default hardening.
