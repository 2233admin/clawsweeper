## Summary

Keep this issue open. Current main still resolves the `llm-task` provider and model as separate fields, only strips a duplicate selected-provider prefix, and dispatches the resulting pair directly to the embedded runner. That does not handle a `model` override like `google/gemini-3-flash-preview` when the selected/default provider remains Anthropic, nor does it resolve configured aliases before plugin allowlist validation. The reporter's distinction from the stale Haiku alias issue is still valid: the shared resolver exists and is tested for provider/model refs and aliases, but `llm-task` does not use it.

## Best Possible Solution

Keep this issue open. Resolve `llm-task` model overrides through the same alias/provider-ref resolver used by normal model selection before allowlist validation and before `runEmbeddedPiAgent`, then pass the resolved provider plus bare model ID. Add regression coverage for `model: "gemini-flash"`, `model: "google/gemini-3-flash-preview"`, an Anthropic alias, and an allowlist case that validates the resolved canonical `provider/model` key. If the bundled plugin boundary should not import the resolver directly, add a narrow plugin/runtime seam for resolving model refs.
