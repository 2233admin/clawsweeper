## Summary

Keep open. The Windows `pnpm ui:build` failure is a concrete regression with an independent Windows reproduction, and current `main` still appears to pass a `.cmd` runner path directly into `spawn(..., { shell: true })` without wrapping/escaping the command path. The linked PR #48289 tracks a proposed fix, but it is still open and unmerged, so the issue is not implemented on main.

## Evidence

- **Issue discussion includes independent Windows reproduction:** stoppieboy reproduced `pnpm ui:build` on Windows with Node v24.14.0 and pnpm 10.32.1, showing Node's `shell: true` warning followed by `'C:\Program' is not recognized as an internal or external command`, matching a path-with-spaces split.
- **Current main still uses shell mode for Windows command shims:** `createSpawnOptions` enables `shell: true` for Windows `.cmd/.bat/.com` runners, but the command path itself is passed through separately rather than converted to a quoted `cmd.exe /c` command line.
