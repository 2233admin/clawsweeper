## Summary

Keep #39807 open. Current main still resolves inline `models.providers.<id>.apiKey` credentials without a durable auth profile id, while billing-disable/backoff recording and fallback skipping remain keyed to auth profiles. The related synthetic-profile fix in PR #39836 is closed unmerged and is not present on main.

## Best Possible Solution

Keep the issue open and implement a maintained synthetic-profile or provider-level cooldown key for inline provider credentials. The likely path is to have `resolveApiKeyForProvider()` return a stable `profileId` for inline `models.providers.<id>.apiKey`, ensure `markAuthProfileFailure()` can persist usage state for that key, and add regression coverage proving a billing failure disables the inline provider and lets configured fallbacks run on later attempts.
