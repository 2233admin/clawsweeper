## Summary

Keep #39807 open. Current main at 8ba82534e6c98410e19b226f3534b285fd75d278 still resolves inline `models.providers.<id>.apiKey` credentials without a durable auth-profile/cooldown identity, while billing disable and fallback skip logic remain profile-store based. The provided related PR #39836 proposed the synthetic `inline:<provider>` fix, but it is closed and unmerged.

## Best Possible Solution

Keep the issue open and implement a stable cooldown identity for inline `models.providers.<id>.apiKey` credentials. A good fix would either synthesize a persisted auth-profile identity for inline provider keys or add provider-level cooldown state, then prove with regression coverage that a billing failure disables that inline identity and later turns skip it and proceed to configured fallbacks.
