## Summary

Keep PR #58244 open. Current main still retains dynamic command-queue lane entries in the global `lanes` Map after work completes: `getLaneState()` creates and stores lane state, while current `drainLane`, `clearCommandLane`, and `resetAllLanes` do not prune idle non-persistent lanes. Dynamic session and auth-probe lane producers still exist. The PR is a focused core bug fix with regression tests and appears to have addressed the prior concurrency-preservation review feedback, but it is now dirty against main and needs normal rebase/validation rather than auto-close.

## Best Possible Solution

Keep the PR open. The best path is to rebase or reimplement the scoped queue fix on current main, preserving well-known lanes and custom `maxConcurrent > 1` lane settings, and keep the lane-count regression tests that prove dynamic lanes are pruned after task completion, `clearCommandLane`, and `resetAllLanes`.
