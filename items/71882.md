## Summary

Keep #71882 open. The PR targets a real remaining core status path: current main can return `null` before consulting the selected memory runtime, so a runtime-backed external memory plugin can still show as unavailable. The linked plugin-side `memory-lancedb-pro` runtime work is relevant, but this PR needs rework before merge because its core patch hard-codes one third-party plugin id and its console filter does not cover raw plugin `console.log` output in routed JSON status.

## Best Possible Solution

Keep the PR open and ask for a generic rework: make status detect runtime-provided memory status through a plugin capability/manifest/runtime contract rather than a hard-coded slot id, consult the selected runtime before builtin sqlite/qmd store gates when that capability is present, and make routed JSON status keep stdout clean for raw plugin console output during runtime loading/probing. Add regression tests using a generic external-style memory plugin/runtime, not `memory-lancedb-pro` as a baked-in core id.

## Risks / Open Questions

- Merging the PR as written would couple shared core status logic to `memory-lancedb-pro` and leave other runtime-backed memory plugins with the same false-negative behavior.
- The proposed console subsystem filter would not protect strict JSON consumers from raw plugin `console.log` / `console.info` output during routed `status --json --all`.
- Closing the PR as external-plugin-only would hide a remaining core status probe ordering problem once external plugins correctly register a memory runtime.
