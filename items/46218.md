## Summary

Current `main` already contains the handshake hardening that matches this report: the gateway pre-auth handshake timeout now defaults to 10 seconds instead of 3, and the CLI gateway call path explicitly yields one event-loop turn before starting the WebSocket client so `connect.challenge` can be processed under heavy startup load. The changelog records that fix in release `v2026.3.22`.

## Evidence

- **Pre-auth handshake timeout raised from 3s to 10s:** `DEFAULT_PREAUTH_HANDSHAKE_TIMEOUT_MS` is `10_000`, and the gateway WebSocket server uses `getPreauthHandshakeTimeoutMsFromEnv()` when arming the unauthenticated handshake timer. That directly addresses the issue report's server-side `handshakeMs: 3000` timeout behavior.
  - file: [src/gateway/handshake-timeouts.ts:1](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/gateway/handshake-timeouts.ts#L1)

## Close Comment

Closing as implemented on current `main` after a Codex review. The gateway handshake path has already been hardened in the areas this report describes: `src/gateway/handshake-timeouts.ts:1` now sets the default pre-auth handshake timeout to `10_000ms`, `src/gateway/server/ws-connection.ts:366` uses that value for the server-side handshake timer, and `src/gateway/call.ts:479` explicitly yields one event-loop turn before starting the CLI WebSocket client so `connect.challenge` is not starved by heavy startup work. There is also regression coverage for that yield in `src/gateway/call.test.ts:489`. The changelog records the handshake timeout fix in `v2026.3.22` at `CHANGELOG.md:2202` and `CHANGELOG.md:2442`. If this is still reproducible on a current build, please open a new issue with the exact version and fresh debug logs so maintainers can separate it from this older v2026.3.13 failure mode.
