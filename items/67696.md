## Summary

Keep open. Current `main` still has a shared completions-stream path that appends `choice.delta.content` without a runtime string guard, so non-string/object deltas can still leak through as `[object Object]`. Telegram and Mattermost both forward that shared `text` payload unchanged, and the nearby shipped streaming fix in the changelog is explicitly OpenRouter-specific rather than a Mistral/general `delta.content` normalization fix.

## Evidence

- **Shared stream path still trusts `delta.content` blindly:** `processOpenAICompletionsStream` checks `if (choice.delta.content)` and then passes `choice.delta.content` directly into `queuePostToolCallDelta(... text: choice.delta.content)` / `appendTextDelta(choice.delta.content)` with no `typeof === "string"` guard or content-array flattening on the response path.
  - file: [src/agents/openai-transport-stream.ts:1322](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/openai-transport-stream.ts#L1322)
