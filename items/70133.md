## Summary

Keep open. Current main still routes ordinary Feishu same-chat P2P messages through an unbounded per-chat promise chain, and each queued task awaits the full agent dispatch before the next same-chat message can enter the gateway reply pipeline. The related #54409 discussion is useful but explicitly describes a different collect-mode batching failure, not this starvation/release-guard bug.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin owner path, likely around `extensions/feishu/src/sequential-queue.ts` and `extensions/feishu/src/monitor.message-handler.ts`. Add a regression test that simulates a same-chat dispatch that never settles and verifies a later P2P message is released, timed out, or admitted to the core follow-up queue without breaking cross-chat concurrency or the existing control lanes.

## Evidence
