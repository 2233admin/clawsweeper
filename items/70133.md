## Summary

Keep #70133 open. Current main still has the Feishu same-chat Promise chain described in the report: normal P2P text resolves to the base per-chat sequential key, the handler awaits the queued task, and the queue only advances after the prior task settles. The generic reply dispatcher also waits for the full agent dispatch before releasing its dispatcher reservation, while the default agent timeout is 48 hours, so there is no narrow Feishu timeout/release guard that would prevent later same-chat messages from being stuck behind a hung or very slow dispatch. The linked #54409 remains related but not a duplicate because it tracks collect-mode batching, while this issue needs explicit starvation/timeout acceptance criteria.

## Best Possible Solution

Keep the issue open and implement a bounded Feishu per-chat queue release strategy or redesign the Feishu inbound path so later same-chat messages can reach the gateway active-run/follow-up queue even when the first dispatch is slow or wedged. The change likely belongs in `extensions/feishu/src/monitor.message-handler.ts` and `extensions/feishu/src/sequential-queue.ts`, with tests that simulate a stuck first task, verify later P2P work is not held indefinitely, and preserve the existing control and `/btw` lane behavior.
