## Summary

Keep this PR open. Current main still lacks the explicit message-tool agent routing path the PR is trying to add: the message tool has no `agentId` option, derives routing from `agentSessionKey`, and the embedded coding-tool path does not pass its resolved active agent into `createMessageTool`. The PR diff is focused and the discussion shows a real fallback hazard was identified and addressed in PR head, so this is still useful maintainer work rather than cleanup-close material.

## Best Possible Solution

Keep this PR open and rebase a guarded version onto current main: add an explicit `agentId` path through `createOpenClawCodingTools` -> `createOpenClawTools` -> `createMessageTool`, prefer it only when it is a trusted runtime-provided explicit value, keep fallback routing config-aware via `resolveSessionAgentId` after config is available, add focused tests for explicit override and opaque-session default-agent behavior, and fix the changelog attribution if the entry remains.
