## Summary

Keep open. Current `main` still does not implement the PR’s `enabled: true` force-enable behavior in `shouldIncludeSkill`, and status reporting still ignores that override, so the reported bug remains live rather than already fixed or obsolete.

## Evidence

- **Runtime filter still ignores `enabled: true`:** `shouldIncludeSkill` only short-circuits for `enabled === false`, then immediately applies bundled allowlist and unconditionally calls `evaluateRuntimeEligibility`. There is no `enabled === true` fast path on current `main`, so missing bins / OS / env / config checks can still exclude a skill even when the config sets `skills.entries.<key>.enabled: true`.
  - file: [src/agents/skills/config.ts:73](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/skills/config.ts#L73)
  - command: `git show HEAD:src/agents/skills/config.ts | nl -ba | sed -n '73,105p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Eligibility path still rejects the exact conditions named in the PR:** `evaluateRuntimeEligibility` returns `false` for OS mismatch and for unmet `requires.bins`, `requires.anyBins`, `requires.env`, or `requires.config`. Because `shouldIncludeSkill` still delegates there, the reported asymmetry is still present in current code.
  - file: [src/shared/config-eval.ts:60](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/shared/config-eval.ts#L60)

That matches the PR review feedback that CLI/UI status would disagree with the proposed runtime behavior unless this file also changed.
