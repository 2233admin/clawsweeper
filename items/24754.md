## Summary

Keep this issue open. Current main still provides Gateway WebSocket WebChat/Control UI chat, browser device identity and pairing, Tailscale/HTTPS access, PWA/web push, OpenAI Realtime browser Talk, and a separate VoiceClaw/Gemini realtime endpoint, but it does not implement the requested no-third-party browser E2EE chat transport with browser-held per-device/per-agent keys and channel isolation. The prior maintainer-associated Codex review comment also kept this open, and current main still supports that conclusion.

## Best Possible Solution

Keep this open as a product/security architecture item. The best path is a maintainer-led design across Gateway protocol/auth, Control UI/WebChat, session/channel routing, transcript storage, docs, and tests. If maintainers want a plugin-first implementation, first define the missing core/plugin API that lets an external browser-native E2EE transport integrate without bypassing pairing, auth scopes, session isolation, or transcript semantics.
