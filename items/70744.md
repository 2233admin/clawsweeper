## Summary

Current `main` still surfaces overflow/compaction exhaustion as a blocked error that tells the user to `/reset` or `/new`, but I did not find an automatic Telegram/session-local recovery path that would quarantine, fork, or reset a poisoned direct session. The provided issue discussion also broadens this beyond Telegram into a general session-recovery problem after effective context-window overflow, so this is not strong enough to close conservatively.

## Evidence

- **Overflow recovery still ends in blocked error, not automatic session recovery:** `runEmbeddedPiAgent` logs overflow recovery attempts, retries explicit auto-compaction up to the configured limit, and then returns a `context_overflow`/`compaction_failure` error payload with `livenessState: "blocked"` and advice to use `/reset` or `/new`. I did not find any automatic fork/reset/quarantine of the bad session in this path.
