## Summary

Keep open. Current `main` still filters `system.execApprovals.get/set` out of the advertised node command surface, while the separate exec-approvals RPC path still invokes those commands directly. That matches the reported mismatch between `nodes describe` / Control UI and `openclaw approvals get --node`.

## Evidence

- **Headless node host declares exec-approvals commands:** The node host connects with `commands: [...NODE_SYSTEM_RUN_COMMANDS, ...NODE_EXEC_APPROVALS_COMMANDS, ...pluginNodeHost.commands]`, so `system.execApprovals.get/set` are intended to be advertised by the host.
  - file: [src/node-host/runner.ts:192](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/node-host/runner.ts#L192)
  - command: `nl -ba src/node-host/runner.ts | sed -n '170,220p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Gateway default allowlist omits exec-approvals commands on macOS/Linux/Windows:** `SYSTEM_COMMANDS` includes `system.run.prepare`, `system.run`, `system.which`, `system.notify`, and `browser.proxy`, but not `system.execApprovals.get/set`; `PLATFORM_DEFAULTS` for `macos`, `linux`, and `windows` are built from that set.
  - file: [src/gateway/node-command-policy.ts:54](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/node-command-policy.ts#L54)

- **Gateway rewrites advertised node commands through the filtered allowlist before registration:** During node connect, `reconcileNodePairingOnConnect(...)` computes `effectiveCommands` from `normalizeDeclaredNodeCommands(... allowlist ...)`, and the websocket handler overwrites `connectParams.commands = reconciliation.effectiveCommands` before the session is registered.
  - file: [src/gateway/node-connect-reconcile.ts:62](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/node-connect-reconcile.ts#L62)

If those commands are filtered out on connect, both surfaces hide the node.
  - file: [src/cli/nodes-cli/register.status.ts:254](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/cli/nodes-cli/register.status.ts#L254)

- **`approvals get --node` bypasses advertised-command filtering:** The gateway `exec.approvals.node.get/set` handlers directly call `context.nodeRegistry.invoke(... command: "system.execApprovals.get/set" ...)`, while `node.invoke` explicitly rejects `system.execApprovals.*` and tells callers to use `exec.approvals.node.*`. That explains how the CLI approvals command can still work even when the node does not advertise those commands.
