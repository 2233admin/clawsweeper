## Summary

Keep PR #70391 open. Current main still reloads chat history after every same-run final chat event, even when `handleChatEvent` already appended a displayable assistant final payload locally. The PR is a focused fix for that remaining reload amplifier; related work does not provide a merged or clearly canonical superseding fix.

## Best Possible Solution

Keep the PR open and refine it: make `shouldReloadHistoryForFinalEvent` skip reloads only for well-formed displayable assistant final payloads, add gateway tests for same-run final assistant payloads and deferred `session.message` replay behavior, and keep #66875 as the broader tracking issue for remaining webchat reload races.

## Risks / Open Questions

- The PR should address the Greptile review concern before merge: a final message with missing or non-string `role` should not be treated as a confirmed assistant payload unless the gateway contract explicitly allows role-less assistant finals.
- The Codex review comment on the PR raises a real product tradeoff: suppressing a deferred `session.message` reload after a final assistant payload can hide unrelated transcript updates from another client. That needs maintainer judgment or a narrower replay condition, but it does not make the PR auto-closeable.
