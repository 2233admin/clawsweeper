## Summary

Keep PR #69495 open. Current main still has the reported heartbeat-model fallback gap: `agents.*.heartbeat.model` is string-only in schema/types/docs, the heartbeat runner only forwards a single `heartbeatModelOverride`, and the reply fallback path still inherits `agents.defaults.model.fallbacks` rather than any heartbeat-specific chain. The PR remains relevant, though it needs follow-up for the posted review concern about preserving explicit `fallbacks: []`.

## Best Possible Solution

Keep the PR open as useful unmerged work. The best product path is to implement heartbeat-specific model fallback support on current main by changing `HeartbeatSchema` and public config types to accept `AgentModelConfig`, resolving heartbeat primary plus ordered fallbacks in `runHeartbeatOnce`, threading an explicit `modelFallbacksOverride` through `GetReplyOptions` and `FollowupRun`, preserving `fallbacks: []` as an empty override, updating pricing/schema/help/docs, and adding regression tests for defaults, per-agent merge, retriable failover plumbing, and empty-array semantics.
