## Summary

Keep open. Current `main` does not implement outbound `[[react:...]]` / `[[react_to_current:...]]` handling for Telegram, and the existing Telegram delivery path still drops reaction-only replies. This PR appears to address a real gap rather than duplicate shipped behavior.

## Evidence

- **Reply directive parsing still ignores reaction tags:** `parseReplyDirectives` only returns text/media/reply/silent fields; there is no `reactionEmoji` or `reactionTag` handling on current `main`. A `[[react_to_current:🔥]]` directive would remain in text instead of being normalized into structured channel data.
  - file: [src/auto-reply/reply/reply-directives.ts:5](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/auto-reply/reply/reply-directives.ts#L5)
  - command: `nl -ba src/auto-reply/reply/reply-directives.ts | sed -n '1,220p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Outbound normalization has no Telegram reaction projection:** `createOutboundPayloadPlanEntry` normalizes text/media/reply/audio fields, then drops non-renderable non-silent payloads. There is no merge step that carries a parsed reaction into `channelData.telegram.reaction`.
  - file: [src/infra/outbound/payloads.ts:152](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/infra/outbound/payloads.ts#L152)
  - command: `nl -ba src/infra/outbound/payloads.ts | sed -n '140,240p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Telegram outbound adapter does not send reactions:** `sendTelegramPayloadMessages` only reads `buttons` and `quoteText` from `channelData.telegram` and never calls `reactMessageTelegram`, so outbound payload delivery on current `main` cannot honor Telegram reaction directives.
  - file: [extensions/telegram/src/outbound-adapter.ts:75](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/telegram/src/outbound-adapter.ts#L75)
  - command: `nl -ba extensions/telegram/src/outbound-adapter.ts | sed -n '1,220p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Reaction-only replies are still rejected in direct Telegram delivery:** `deliverReplies` treats replies with no text and no media as `reply missing text/media` and skips them. That blocks reaction-only payloads from being delivered on current `main`.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:669](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/telegram/src/bot/delivery.replies.ts#L669)
  - command: `nl -ba extensions/telegram/src/bot/delivery.replies.ts | sed -n '620,720p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Telegram already has a native reaction primitive:** `reactMessageTelegram` already exists and is covered by tests, which supports the PR’s claim that this is missing wiring for an existing Telegram capability rather than an incoherent or out-of-scope request.
  - file: [extensions/telegram/src/send.ts:1047](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/telegram/src/send.ts#L1047)
