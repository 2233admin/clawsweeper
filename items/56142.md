## Summary

Keep open. Current `main` still routes Slack `block_actions` into queued system events only; it does not dispatch them through `handleSlackMessage` or pair the enqueue with a heartbeat wake. That matches the reported symptom that button clicks can sit idle until some later wake path runs.

## Evidence

- **Interaction registration still omits message dispatch:** `registerSlackMonitorEvents()` passes `handleSlackMessage` only to `registerSlackMessageEvents()`. `registerSlackInteractionEvents()` is still called with just `{ ctx, trackEvent }`, so the issue's proposed wake-through-message path is not present on `main`.
  - file: [extensions/slack/src/monitor/events.ts:18](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/slack/src/monitor/events.ts#L18)
  - command: `nl -ba extensions/slack/src/monitor/events.ts | sed -n '1,40p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Block actions only enqueue a system event:** `enqueueSlackBlockActionEvent()` resolves a session key and calls `enqueueSystemEvent(...)`, but the handler does not call `handleSlackMessage()` or `requestHeartbeatNow()`. This is the exact code path behind Slack button clicks.
