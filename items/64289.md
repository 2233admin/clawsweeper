## Summary

Keep open. Current main does not provide strong evidence that the reported idle-return WebChat disconnect/flash is fixed. Some startup root-cause details appear weaker now because deferred gateway plugins are loaded before WebSocket handlers attach, but current main still has session-scoped bundled MCP runtime idle eviction at the same 10-minute window, creates those runtimes lazily during agent attempts, and Control UI still visibly transitions through disconnected/reconnect state on socket close. The exact named plugins in the report appear external/not in-tree, so closing would be premature.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The best path is to either preserve Control UI state across recoverable local WebSocket reconnects, add an explicit prewarm/eager-load path for session-scoped plugin or bundled MCP runtimes where the memory tradeoff is acceptable, or document/configure mcp.sessionIdleTtlMs=0 as a workaround while collecting a current repro with the affected external plugins and gateway logs.
