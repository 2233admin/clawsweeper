## Summary

Keep open. Current main still lacks the Mattermost message-tool thread-context inheritance requested in #45082, and the cron/heartbeat follow-up case from the issue comment is still visible in the Mattermost monitor path. The linked PRs #52120 and #52236 track plausible fixes, but they are still open/draft in the provided GitHub context, so the bug remains actionable in-repo rather than implemented or safely superseded.

## Evidence

- **Message tool send path still uses only explicit reply IDs:** Mattermost `handleAction` destructures only `action`, `params`, `cfg`, and `accountId`; for send actions it derives `replyToId` solely from `params.replyToId` or `params.replyTo` before calling `sendMessageMattermost`. There is no `toolContext.currentThreadTs` fallback in the current path.
  - file: [extensions/mattermost/src/channel.ts:113](https://github.com/openclaw/openclaw/blob/1531123d3547f9ff8ee8094590fc773305f018f2/extensions/mattermost/src/channel.ts#L113)

  - sha: [1531123d3547](https://github.com/openclaw/openclaw/commit/1531123d3547f9ff8ee8094590fc773305f018f2)
- **Mattermost plugin still has no threading buildToolContext:** The Mattermost threading adapter exposes account reply mode resolution and reply transport mapping, but no `buildToolContext`; `rg -n "buildToolContext" extensions/mattermost/src/channel.ts` returned no matches. Without that channel-specific builder, the message tool does not get Mattermost `MessageThreadId` converted into the tool context described by the linked PR #52120.
