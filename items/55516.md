## Summary

Keep open: current `main` still sends hook-dispatched isolated agent runs through the shared `nested` lane with default concurrency `1`, and the config/schema/docs still do not expose a `hooks.maxConcurrentRuns` or `CommandLane.Nested` concurrency setting.

## Evidence

- **Hook dispatch still uses the cron-to-nested path:** `dispatchAgentHook` forwards the hook `sessionKey` into `runCronIsolatedAgentTurn`, but it hard-codes `lane: "cron"`, matching the issue's reported execution path.
  - file: [src/gateway/server/hooks.ts:83](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server/hooks.ts#L83)
  - command: `nl -ba src/gateway/server/hooks.ts | sed -n '43,90p'`

- **Cron executor does not scope nested lane by session:** The isolated cron executor still calls `runEmbeddedPiAgent` with `lane: resolveNestedAgentLane(params.lane)`. It does not use `resolveNestedAgentLaneForSession(params.agentSessionKey)`, so distinct hook session keys do not get distinct global lanes on this path.
  - file: [src/cron/isolated-agent/run-executor.ts:154](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/cron/isolated-agent/run-executor.ts#L154)

  - file: [src/agents/lanes.ts:8](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/lanes.ts#L8)
