## Summary

Keep #70050 open. Current main still performs load-time session maintenance synchronously on the caller path when the store is over the configured entry cap, and `chat.history` remains gated during startup sidecar readiness. There is a config workaround (`session.maintenance.mode: "warn"` or a higher `maxEntries`) and current tests verify that config is honored, but the default startup-blocking behavior reported here has not been replaced with deferred/background/chunked maintenance.

## Best Possible Solution

Keep the issue open and fix the default startup path: make oversized-store maintenance deferred/background or chunked with event-loop yields, preserve an explicit low-friction disable/debug knob, and add regression coverage that proves gateway startup and `chat.history` availability are not blocked by pruning/capping a large `sessions.json`. Update session maintenance docs to match the final default behavior.
