## Summary

Keep open. Current `main` still fail-closes Windows allowlist shell wrappers only when `cmdInvocation` is true, even though PowerShell and `pwsh` are already recognized as shell-wrapper transports. The PR is addressing a real security gap, and the existing review discussion also shows the submitted patch needs a small test follow-up rather than closure.

## Evidence

- **Current policy still gates Windows wrapper blocking on `cmdInvocation`:** `evaluateSystemRunPolicy()` on `main` sets `windowsShellWrapperBlocked` only when all of `security === "allowlist"`, `shellWrapperInvocation`, `isWindows`, and `cmdInvocation` are true. That means non-`cmd.exe` Windows shell wrappers are not fail-closed by this guard today.
  - file: [src/node-host/exec-policy.ts:68](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/node-host/exec-policy.ts#L68)
  - command: `nl -ba src/node-host/exec-policy.ts | sed -n '52,75p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **PowerShell wrappers are first-class shell-wrapper transports in current main:** The shell-wrapper resolver explicitly includes `powershell` and `pwsh` in the wrapper sets/specs and extracts inline commands for the `powershell` wrapper kind. So current code already classifies PowerShell invocations as shell wrappers.
  - file: [src/infra/shell-wrapper-resolution.ts:16](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/infra/shell-wrapper-resolution.ts#L16)

- **Call site passes shell-wrapper state independently from `cmd.exe` detection:** `handleSystemRunInvoke()` derives `cmdInvocation` separately, then passes `shellWrapperInvocation: parsed.shellPayload !== null`. A Windows PowerShell wrapper can therefore reach policy evaluation with `shellWrapperInvocation === true` and `cmdInvocation === false`, which matches the PR's reported gap.
