## Summary

Current main still returns the freshly rotated device token in the `device.token.rotate` gateway RPC response. The behavior is source-backed, asserted by tests, consumed by Control UI, and exposed by the CLI. The related redaction PR #66794 is closed and unmerged in the provided context, so there is no merged implementation candidate to close against.

## Best Possible Solution

Keep this issue open and treat it as a gateway protocol/security design fix. The likely implementation path is to stop returning `entry.token` from normal `device.token.rotate` responses, add regression tests proving the fresh secret is absent or only available through an explicitly safer bootstrap/provisioning flow, and update Control UI, CLI output, and public docs to match the chosen contract.
