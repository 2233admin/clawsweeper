## Summary

Keep open. Current `main` fixes the narrower "no usage data" case, but it still treats explicit zero-usage snapshots as missing and can clear preserved session totals instead of retaining a stale display value. I found no evidence on `main` that the PR's `totalTokensEstimate` behavior, or an equivalent end-to-end fix, has already landed.

## Evidence

- **Zero-valued prompt/context snapshots are still discarded on main:** `derivePromptTokens` returns `undefined` when the summed prompt-side usage is `0`, and `deriveSessionTotalTokens` only accepts prompt overrides greater than `0`. That means explicit zero-usage snapshots are still treated as missing rather than as a value that can be preserved distinctly from freshness.
  - file: [src/agents/usage.ts:207](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/usage.ts#L207)
  - command: `nl -ba src/agents/usage.ts | sed -n '200,290p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Auto-reply persistence still clears totals on zero-only snapshots:** `persistSessionUsageUpdate` marks any present `lastCallUsage` object as a fresh context snapshot via `Boolean(params.lastCallUsage)`, but then writes `patch.totalTokens = totalTokens` and `patch.totalTokensFresh = typeof totalTokens === "number"`. For a zero-only `lastCallUsage`, `deriveSessionTotalTokens` returns `undefined`, so the stored total is cleared instead of being preserved as stale display state.
  - file: [src/auto-reply/reply/session-usage.ts:105](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/auto-reply/reply/session-usage.ts#L105)

- **Current regression coverage is only for missing usage, not zero usage:** The shipped test coverage on `main` explicitly verifies preservation when a provider returns no usage data at all (`#67667`).
