## Summary

Keep open. Current `main` only partially handles `authHeader`: it can inject an `Authorization` header onto the resolved model, but Anthropic-compatible runtime requests still go through the boundary-aware Anthropic transport that reads `options.apiKey` and emits `x-api-key` for non-OAuth flows. Onboarding verification is also still hardcoded to Anthropic-style `x-api-key` and never consults existing provider `authHeader` state, so the PR still targets behavior that is not implemented on `main`.

## Evidence

- **Anthropic verification probes still hardcode `x-api-key`:** `buildAnthropicHeaders()` always adds `x-api-key`, and `buildAnthropicVerificationProbeRequest()` always uses that helper for `/v1/messages` probes. There is no `authHeader` parameter or Bearer branch here.
  - file: [src/commands/onboard-custom-config.ts:252](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/commands/onboard-custom-config.ts#L252)
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Onboarding detect/verify path never threads `authHeader` into Anthropic probes:** `requestAnthropicVerification()` only accepts `{ baseUrl, apiKey, modelId }`, and both the unknown-endpoint detection path and explicit Anthropic verification path call it without any provider-config lookup.
  - file: [src/commands/onboard-custom.ts:120](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/commands/onboard-custom.ts#L120)
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Current runtime authHeader support is only header injection on the model:** `applyAuthHeaderOverride()` adds `Authorization: Bearer <apiKey>` to `model.headers` when `provider.authHeader === true`, and the run path applies that override before launching the attempt. This is only a header mutation; it does not change the Anthropic transport auth mode by itself.
  - file: [src/agents/model-auth.ts:767](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/model-auth.ts#L767)
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Anthropic requests on main still use the boundary-aware Anthropic transport:** For `anthropic-messages`, `createBoundaryAwareStreamFnForModel()` selects `createAnthropicMessagesTransportStreamFn()`, and `stream-resolution` uses that boundary-aware transport when the session streamFn is still the default/simple path.
  - file: [src/agents/provider-transport-stream.ts:75](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/provider-transport-stream.ts#L75)
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Boundary-aware Anthropic transport still emits `x-api-key` from `options.apiKey`:** The Anthropic transport merges `x-api-key` when `params.apiKey` is present, builds the non-OAuth client with `apiKey`, and resolves that `apiKey` directly from `options.apiKey`. So an `authHeader: true` provider still stays on the API-key auth path unless something also clears/swaps the runtime credential handling.
  - file: [src/agents/anthropic-transport-stream.ts:492](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/anthropic-transport-stream.ts#L492)
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)

## Risks / Open Questions

- Bearer-only Anthropic-compatible gateways can still fail on current `main` because runtime transport continues to emit `x-api-key` for non-OAuth Anthropic flows.
- Custom/onboarded Anthropic-compatible endpoints that rely on `authHeader: true` can still fail verification or auto-detection because onboarding probes always use Anthropic-style API-key headers.
