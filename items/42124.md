## Summary

Current `main` has a background-task ledger and delivery pipeline for OpenClaw-owned detached work, but it does not implement this PR's generic external task pub/sub API (`tasks.register/watch/unwatch/notify/list/remove`) or multi-watcher fan-out model. The existing `openclaw tasks` surface is a different feature.

## Evidence

- **Current docs describe tasks as an internal background-work ledger:** `docs/automation/tasks.md` says tasks track ACP, subagents, cron jobs, and CLI operations, and explicitly frames them as records for detached OpenClaw work rather than a generic external task registry.
  - file: [docs/automation/tasks.md:10](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/docs/automation/tasks.md#L10)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Current CLI surface does not match the PR API:** `docs/cli/tasks.md` lists `list`, `show`, `notify`, `cancel`, `audit`, `maintenance`, and `flow` subcommands. It does not document `register`, `watch`, `unwatch`, or `remove`.
  - file: [docs/cli/tasks.md:16](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/docs/cli/tasks.md#L16)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Implemented CLI behavior is policy-oriented, not event pub/sub:** `src/commands/tasks.ts` shows `tasksShowCommand` rendering an existing background task record, `tasksNotifyCommand` only updating `notifyPolicy`, and `tasksCancelCommand` canceling a tracked run. There is no generic task creation/watch/remove/event fan-out command path here.
  - file: [src/commands/tasks.ts:316](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/commands/tasks.ts#L316)

  - file: [src/gateway/server-methods.ts:71](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/gateway/server-methods.ts#L71)

There is no watcher list or generic external event registry in the current model.

## Risks / Open Questions

- `main` already has a substantial tasks/task-flow subsystem, so landing this PR as-is would likely need rework to fit the current ledger/delivery architecture instead of merging cleanly.
- The PR's gateway API and CLI names (`tasks.notify`, `tasks register/watch/...`) overlap with existing `openclaw tasks` semantics, which now mean background-task inspection and notify-policy changes.
