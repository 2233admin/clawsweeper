## Summary

Current main still does not implement the requested permanent delivery classification. The local media guard still emits the exact reported deterministic allowlist error, but the delivery queue permanent-error patterns do not include it; live send failures and recovery failures still call failDelivery for non-matching errors, which preserves retry behavior.

## Best Possible Solution

Keep this issue open. The narrow fix belongs in src/infra/outbound/delivery-queue-recovery.ts plus delivery queue policy/recovery tests: classify the exact local-media allowlist rejection as permanent so poisoned queued entries move directly to failed instead of burning retry budget. Enqueue-time validation and a lastErrorClass field can be follow-ups. Verify the Telegram `message is too long` path separately before adding that pattern.

## Risks / Open Questions

- A broad media-error regex could incorrectly skip retries for fixable local-media failures such as missing files, temporary staging paths, or allowlist changes; the implementation should match the deterministic allowlist rejection precisely, or classify LocalMediaAccessError code `path-not-allowed` where that structured error is available.
- The proposed Telegram `message is too long` pattern should be verified separately against current Telegram chunking and caption behavior before adding it, because current main already chunks normal Telegram text around a 4000-character limit and has some caption-specific fallback paths.
