## Summary

Keep this issue open. Current main still lacks the requested opt-in hard caps for wall-clock session lifetime and cumulative per-session token spend. The discussion contains a useful implementation sketch and an external Helm-doc workaround, but neither is shipped core behavior and the current code only covers daily/idle freshness, context-window compaction, usage display, and disk maintenance.

## Best Possible Solution

Keep this issue open and implement it as a core session-budget feature. Add optional config schema/types/docs for session duration and cumulative token limits, persist reliable per-session lifecycle and cumulative usage-budget state, enforce before costly LLM/tool work and on lifecycle checks, emit deterministic feedback/telemetry, and add focused regression tests for normal chat, cron/heartbeat, provider-owned CLI sessions, compaction/reset interactions, and budget-exceeded actions.

## Risks / Open Questions

- Budget-exceeded behavior still needs maintainer product/API judgment: stop, warn, compact, reset, or channel/runtime-specific fallback semantics.
- A correct cumulative token cap likely needs explicit per-session usage accounting rather than reusing `SessionEntry.totalTokens`, which current code treats as a context snapshot.
- Implementation should cover active long-running sessions, cron/heartbeat paths, provider-owned CLI sessions, compaction/memory flush ordering, and deterministic user/operator feedback.
