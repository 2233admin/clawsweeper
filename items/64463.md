## Summary

Keep this issue open. Current main still lacks the requested opt-in hard caps for wall-clock session lifetime and cumulative per-session token spend. Existing behavior covers daily/idle reset freshness, focused-thread auto-unfocus max age, per-run/cron timeouts, context-window compaction, usage display, and session disk maintenance, but none of those enforce `session.maxDurationMinutes` or `session.maxTokensPerSession`. The discussion includes a useful implementation sketch and an external Helm-doc workaround, but no shipped core behavior or canonical merged replacement.

## Best Possible Solution

Keep this issue open and implement it as a core session-budget feature. Add optional config schema/types/docs for session duration and cumulative token limits, persist reliable per-session lifecycle and cumulative usage-budget state, enforce before costly LLM/tool work and on lifecycle checks, emit deterministic feedback/telemetry, and add focused regression tests for normal chat, cron/heartbeat, provider-owned CLI sessions, compaction/reset interactions, and budget-exceeded actions.

## Risks / Open Questions

- Budget-exceeded behavior still needs maintainer product/API judgment: stop, warn, compact, reset, or channel/runtime-specific fallback semantics.
- A correct cumulative token cap likely needs explicit per-session usage accounting; current `SessionEntry.totalTokens` is deliberately a context snapshot, not cumulative spend.
- The implementation should cover active long-running sessions, cron/heartbeat paths, provider-owned CLI sessions, compaction/memory flush ordering, and deterministic user/operator feedback.
