## Summary

Keep open. Current main has partial persistent session labeling through `sessions.patch`, and Browser Control UI session selection can prefer `label`, but the reported WebChat/native session list path still relies on generated `displayName`/key and has no rename or fixed-label surface. The issue is fresh, concrete, and describes a real gap in how stable user-visible session names propagate across gateway/app clients after metadata-derived names change.

## Best Possible Solution

Keep the issue open and treat it as a core/app session UX gap. The best path is to make the existing persistent `label` field the canonical manual session name, or add an explicit manual-title field, then ensure `sessions.list` exposes a stable user-visible title consistently across Browser Control UI, native WebChat Swift models, Android parsing, and any Feishu/channel display path. Add tests for gateway list/patch precedence, browser selector behavior, and native decoder/UI fallback so generated `displayName` changes no longer overwrite a fixed user name.
