## Summary

Keep open. Current main has improved and documented the memory model, but it still does not implement the requested strict, single user-facing memory mode. The issue discussion asks for one contract covering builtin-only, qmd-only, and mixed behavior; current code still separates durable Markdown writes, builtin SQLite indexing, QMD backend selection, provider/search settings, and fallback behavior.

## Best Possible Solution

Keep this open as a focused maintainer design task. The best path is to define a single user-facing memory-mode contract, or an explicitly documented successor to `memory.type`, that covers durable write target, search backend, fallback policy, compaction flush behavior, index storage location, and migration/doctor guidance for users who already have both Markdown and SQLite memory artifacts. Likely code/docs areas are `src/config/types.memory.ts`, `src/config/zod-schema.ts`, `src/config/schema.help.ts`, `src/agents/memory-search.ts`, `extensions/memory-core/src/flush-plan.ts`, `extensions/memory-core/src/memory/search-manager.ts`, QMD/builtin manager tests, and the memory docs.

- Current docs reduce confusion, but teams can still observe different behavior when configs, embedding credentials, QMD availability, `memorySearch.enabled`, or `memoryFlush.enabled` differ.
