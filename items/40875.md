## Summary

PR #40875 should stay open. Current main still lacks heartbeat-specific fallback wording, so the docs gap is real, but the PR's proposed wording is not accurate enough for an automatic close/merge outcome: current fallback resolution skips unrelated configured fallbacks for a different-provider override, but still appends the configured primary when no explicit fallback override is supplied. The PR also adds a changelog entry without the required contributor thanks attribution.

## Best Possible Solution

Keep the PR open and ask for a narrow docs revision: document that heartbeat uses the normal model fallback candidate rules; no override and same-provider overrides use the configured chain, different-provider overrides skip unrelated configured fallbacks unless already in the configured chain, and the configured primary may still be tried at the end unless an explicit fallback override changes the candidate list. Also either remove the changelog entry if maintainers consider this docs-only clarification too small, or add the required `Thanks @liyishuai` attribution.
