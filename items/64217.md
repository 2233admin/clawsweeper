## Summary

Current main still only partially addresses prompt-cache hygiene. Inbound metadata has been stabilized, but `sessions_send` still injects requester/target-specific agent-to-agent text through `extraSystemPrompt`, that text still becomes provider-facing system instructions on the OpenAI Codex Responses path, and owner-only tools such as `cron` are still removed for non-owner or unknown senders. The prior ClawSweeper comment keeping this open matches current `main` at c7543701009ccdae83d38ee341e072c8a7c4af6e.

## Best Possible Solution

Keep this issue open. The best fix is to move `sessions_send` requester/target A2A metadata out of `extraSystemPrompt` and into the triggering message, provenance, or another non-prefix-critical turn structure; then add a regression test that checks same-session OpenAI Codex Responses requests keep stable `instructions` and tool schemas across direct and inter-session turns with the same `prompt_cache_key`. Separately, make an explicit product/security decision for owner-only tools: either keep a stable exposed schema with execution-time authorization denial, or document and test a deliberate cache-impact exception for source-based tool filtering.

## Risks / Open Questions

- Changing owner-only tool exposure for schema stability has a security boundary risk: any stable-schema design must preserve runtime authorization denial for non-owner and unknown senders, especially for `cron`, `gateway`, and `nodes`.
- A fix needs regression coverage that compares direct and `sessions_send` turns for the same target session on the OpenAI Codex Responses path; existing tests currently assert the behavior that causes the drift.
