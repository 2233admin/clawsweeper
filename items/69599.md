## Summary

Current `main` still routes the Control UI "New Session" affordance and `/new`/`/reset` through in-place reset behavior for the active `sessionKey`; I did not find evidence that the webchat fresh-start flows now create a distinct session identity equivalent to manually selecting/creating another session key.

## Evidence

- **Control UI New Session button still sends `/new`:** The chat toolbar wires `onNewSession` directly to `state.handleSendChat("/new", { restoreDraft: true })`, so the UI button is not using a separate session-creation path.
  - file: [ui/src/ui/app-render.ts:2295](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/ui/src/ui/app-render.ts#L2295)
  - command: `nl -ba ui/src/ui/app-render.ts | sed -n '2288,2304p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Local slash handling forwards `/new` and `/reset` on the current chat session:** The chat client dispatches `/new` and `/reset` by calling `sendChatMessageNow` with those literal commands; clearing history separately calls `sessions.reset` for the current `host.sessionKey`. I did not find a branch here that allocates a new session key or switches the UI to a new one.
