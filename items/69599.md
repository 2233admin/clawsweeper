## Summary

Keep #69599 open. Current main still wires the Control UI New Session and `/new`/`/reset` paths through the current chat session key, while the gateway reset path rotates the `sessionId` in place under the same canonical key. The independent fresh-session behavior exists via `sessions.create`, but the normal Control UI fresh-start affordances do not use it or switch the selector to the returned dashboard session key. Related #66533 was closed for heartbeat/selector pollution and treats this issue as the remaining follow-up; #47561 is a separate routing-mode durability bug.

## Best Possible Solution

Keep the issue open for maintainer product/API work. A conservative implementation would either wire the Control UI New Session button to `sessions.create`, switch `state.sessionKey` and the URL to the returned dashboard key, and refresh `sessions.list`; or keep `/reset` as same-key reset while adding/renaming a distinct independent chat-session action. Add focused UI/controller and gateway integration tests proving the selector key changes, the new key appears in the session list, and context/token state starts fresh.

## Risks / Open Questions

- Changing `/new` globally would affect documented same-key reset semantics across channels, so the least risky fix likely scopes independent creation to the Control UI New Session affordance or adds a separate explicit action.
- Current main does create a fresh `sessionId` and cleared token counters for the same key, so maintainers should decide whether the requested separate selector identity is product intent or whether the UI wording/docs should be clarified instead.
