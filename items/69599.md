## Summary

Current main still does not provide the requested parity between Control UI `/new`/`/reset`/New Session and direct session creation. The UI paths send `/new` or `/reset` against the current selected session key, and the gateway reset service rotates the session id in place under the same canonical key. The backend `sessions.create` path can create an independent dashboard session key, but that path is not used by the normal Control UI fresh-start affordances. Related context does not supersede this: #66533 was closed for the heartbeat selector-pollution bug and explicitly left this true-fresh-session semantics issue as the remaining follow-up; #47561 is a separate routing-mode durability bug.

## Best Possible Solution

Keep #69599 open. A conservative fix should make an explicit product decision: either wire the Control UI New Session path, and possibly `/new`, to `sessions.create`, switch the UI to the returned dashboard/webchat key, and refresh the selector; or preserve `/reset` as in-place reset while adding/renaming a separate independent "new chat/session" action. Add focused UI/controller and gateway integration tests proving the selected session key changes, the new key appears in `sessions.list`, and context/token state starts fresh.
