## Summary

Keep open. Current main still has the reported PI auth discovery gap: auth-profile to PI credential conversion only accepts inline `key`/`token` material, while proper SecretRef-backed profiles can persist only `keyRef`/`tokenRef`. Runtime auth has a resolver that can resolve those refs, but PI model discovery does not use it, so non-env refs such as keychain/exec remain plausible failures for OpenRouter picker/catalog availability.

## Best Possible Solution

Keep the issue open and fix the PI auth bridge generically: make PI credential discovery resolve `api_key` and `token` auth profiles through the existing auth-profile SecretRef resolver, or introduce an equivalent safe async discovery path, then add a regression test for a ref-only OpenRouter `api_key` profile. Verify the shared model catalog plus Discord/text model picker path sees OpenRouter as authenticated without persisting plaintext secrets.

- The eventual fix likely needs careful handling of synchronous PI discovery call sites and must preserve `allowKeychainPrompt: false` behavior to avoid unexpected prompts during catalog/model-picker reads.
