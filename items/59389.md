## Summary

Keep #59389 open. It is a recent regression report with a follow-up saying the delay still existed on 2026-04-10, and I did not find a direct #59389 fix, duplicate, or source/docs evidence proving current main resolves the reported 42-minute daily scheduled delivery delay. Current main has related cron and delivery hardening, including a busy-main-lane cron fix, but the remaining user-visible delay could still involve cron run timing, heartbeat contention, Bedrock/model latency, or outbound channel delivery state.

## Best Possible Solution

Keep the issue open and ask for `openclaw cron runs --id <jobId> --limit 20`, gateway logs around the scheduled and actual delivery timestamps, the cron job shape (`sessionTarget`, `wakeMode`, delivery mode/channel/to), heartbeat skip reasons, and provider/model timing. Maintainers should then reproduce on current main or add a focused regression test around the confirmed bottleneck, whether it is scheduler wake timing, main-session heartbeat contention, isolated-run execution, Bedrock latency, or outbound delivery queue recovery.
