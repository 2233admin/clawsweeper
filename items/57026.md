## Summary

Current `main` no longer requires `OLLAMA_API_KEY` for the ambient/local Ollama discovery path that would have caused this startup blocker. The Ollama provider now falls back to the non-secret `ollama-local` marker, the behavior is covered by tests, documented, and the changelog shows it shipped in `v2026.4.15`.

## Close Comment

Closing this as implemented on current `main` after a Codex review.

- `extensions/ollama/src/discovery-shared.ts:43` now falls back to the non-secret local marker `ollama-local` when `OLLAMA_API_KEY` is absent, instead of requiring the env var for the ambient/local Ollama path.
- `extensions/ollama/provider-discovery.test.ts:194` covers local Ollama auto-registration with no `OLLAMA_API_KEY`, and `extensions/ollama/provider-discovery.test.ts:227` covers the no-warning/no-failure path when Ollama is unreachable and not explicitly configured.
- `extensions/ollama/index.test.ts:302` keeps the stored `ollama-local` marker on the quiet ambient path, which matches the intended non-blocking local setup.
- `docs/providers/ollama.md:86` and `docs/providers/ollama.md:238` document that host-backed Ollama setups can use a placeholder/local marker instead of a real key.
- `CHANGELOG.md:555` includes this Ollama onboarding/auth behavior in release `v2026.4.15`.

If you still hit a startup failure on a newer build, please open a fresh issue with the exact Ollama config shape and whether `models.providers.ollama.apiKey` is an explicit SecretRef or only ambient discovery.
