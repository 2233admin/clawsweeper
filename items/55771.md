## Summary

Keep open. Current `main` does not implement `block`/`blockReason` for `before_model_resolve`, and the runtime still treats that hook as override-only plus fail-open on errors, which matches the issue report rather than resolving it.

## Evidence

- **Hook result type still lacks block fields:** `PluginHookBeforeModelResolveResult` only exposes `modelOverride` and `providerOverride`; there is no `block` or `blockReason` field in the current public contract.
  - file: [src/plugins/hook-before-agent-start.types.ts:14](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/plugins/hook-before-agent-start.types.ts#L14)
  - command: `nl -ba src/plugins/hook-before-agent-start.types.ts | sed -n '1,120p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Hook merger only combines overrides:** `mergeBeforeModelResolve` preserves the first defined `modelOverride` and `providerOverride` only. `runBeforeModelResolve()` uses that merger and has no terminal block policy analogous to `before_tool_call`.
  - file: [src/plugins/hooks.ts:222](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/plugins/hooks.ts#L222)

- **Agent setup swallows before_model_resolve failures and proceeds:** `resolveModelSelectionOverrides()` wraps `runBeforeModelResolve()` in `try/catch`, logs `before_model_resolve hook failed`, and then continues with normal model resolution. That means throwing from the hook does not block the model call on current `main`.
