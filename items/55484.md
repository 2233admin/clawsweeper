## Summary

Keep open. Current `main` still only supports ACP persistence as thread-bound `mode="session"`; it rejects non-thread persistent ACP spawns, exposes no `affinityKey` input, and generates a fresh random ACP session key per spawn instead of a deterministic affinity key for cron/orchestrator reuse.

## Evidence

- **`sessions_spawn` schema has no affinity contract:** The ACP tool schema exposes `runtime`, `agentId`, `resumeSessionId`, `model`, `cwd`, `thread`, `mode`, `sandbox`, `streamTo`, etc., but no `affinityKey` or other non-thread persistence selector.
  - file: [src/agents/tools/sessions-spawn-tool.ts:100](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/tools/sessions-spawn-tool.ts#L100)
  - command: `nl -ba src/agents/tools/sessions-spawn-tool.ts | sed -n '100,130p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **ACP mode parsing is still only `run` or `session`:** Current parsing accepts `params.mode` only when it is `run` or `session`; there is no separate persistent-affinity mode for ACP spawns from cron/orchestrator callers.
  - file: [src/agents/tools/sessions-spawn-tool.ts:191](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/tools/sessions-spawn-tool.ts#L191)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Non-thread ACP persistence is explicitly rejected:** `spawnAcpDirect` returns `thread_required` when ACP `mode="session"` is used without `thread=true`, with guidance to retry on a thread-capable channel or fall back to one-shot `mode="run"`. That is the opposite of the issue request for non-thread persistent affinity.
  - file: [src/agents/acp-spawn.ts:1069](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/acp-spawn.ts#L1069)

There is no deterministic key derived from requester identity, backend, cwd, and affinity key, so cron/orchestrator reuse cannot happen on `main` the way the issue describes.
