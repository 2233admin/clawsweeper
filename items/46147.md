## Summary

Current main already addresses the reported silent-hang path for `openclaw node run`: node-host connection failures are surfaced to stderr, insecure/non-loopback plaintext WS is rejected with explicit guidance, and a missing preauth handshake now times out instead of hanging indefinitely. The shipped `v2026.4.22` tag contains this behavior.

## Evidence

- **Node host now reports gateway connection failures:** `runNodeHost` wires `GatewayClient` errors and closes to stderr, so the foreground node host no longer fails silently after startup.
  - file: [src/node-host/runner.ts:211](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/node-host/runner.ts#L211)
  - command: `nl -ba src/node-host/runner.ts | sed -n '170,230p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Gateway client rejects unsafe/non-loopback plaintext WS with explicit recovery advice:** `GatewayClient.start()` fails closed for non-loopback `ws://` targets and tells the operator to use `wss://`, an SSH tunnel, or Tailscale instead of silently stalling.
  - file: [src/gateway/client.ts:230](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/client.ts#L230)

- **Missing WebSocket preauth handshake now becomes an explicit timeout:** If the socket opens but no `connect.challenge` arrives, the client raises `gateway connect challenge timeout ...` and closes the socket instead of waiting forever.
  - file: [src/gateway/client.ts:819](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/client.ts#L819)

## Close Comment

Closing as implemented on main after Codex review. Current OpenClaw already covers the silent-hang path this issue describes: `src/node-host/runner.ts:211` now prints node-host gateway connect failures to stderr, `src/gateway/client.ts:230` rejects unsupported plaintext non-loopback `ws://` targets with explicit SSH-tunnel/Tailscale guidance, and `src/gateway/client.ts:819` turns a missing WebSocket preauth challenge into a `gateway connect challenge timeout` error instead of hanging indefinitely. The supported SSH-tunnel flow is also documented in `docs/nodes/index.md:84`, and this behavior is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still see a live connection failure on a newer build, it should be tracked as a fresh issue with current version/build details and the exact stderr output now emitted by the node host.
