## Summary

Keep open. Current main still leaves `resolveConfigPath` without the proposed prototype-key guard, and a read-only import of the current source confirms it can return inherited prototype-chain values for blocked segments. The PR is small, coherent security hardening and should get maintainer review rather than cleanup closure.

## Best Possible Solution

Keep the PR open and either land the narrow guard in `src/shared/config-eval.ts` or ask the contributor to add regression tests in `src/shared/config-eval.test.ts` proving `__proto__`, `prototype`, and `constructor` segments return `undefined` while ordinary config paths still resolve normally.

## Evidence

- **Current implementation lacks guard:** `resolveConfigPath` splits the raw dot path and directly reads `(current as Record<string, unknown>)[part]` after only checking that the current value is a non-null object; it does not call `isBlockedObjectKey` for `__proto__`, `prototype`, or `constructor` segments.
  - file: [src/shared/config-eval.ts:20](https://github.com/openclaw/openclaw/blob/4ad8b613c9b60466224d9b3737fff7d63ada652a/src/shared/config-eval.ts#L20)
  - command: `nl -ba src/shared/config-eval.ts | sed -n '1,140p'`
  - sha: [4ad8b613c9b6](https://github.com/openclaw/openclaw/commit/4ad8b613c9b60466224d9b3737fff7d63ada652a)
- **Current behavior reproduces the concern:** Importing the current source with Node type stripping returned `{"proto":"function","constructor":"function","normal":true}` for `resolveConfigPath({}, "__proto__.toString")`, `resolveConfigPath({}, "constructor")`, and a normal config path, proving prototype-chain access remains possible on main.
  - command: `node --experimental-strip-types --input-type=module -e 'import("./src/shared/config-eval.ts").then(({resolveConfigPath})=>{const values={proto: typeof resolveConfigPath({}, "__proto__.toString"), constructor: typeof resolveConfigPath({}, "constructor"), normal: resolveConfigPath({browser:{enabled:true}}, "browser.enabled")}; console.log(JSON.stringify(values));})'`
