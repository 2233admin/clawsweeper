## Summary

Keep this PR open. Current main still does not expose `gateway.mcpLoopback.port` or any operator-facing way to pin the bundled MCP loopback port, even though the original CLI session-reset symptom is partly solved by stable `mcpResumeHash` canonicalization. The PR is stale because it patches the old eager gateway startup path; a rebased version needs to wire the config into the current lazy CLI-backend loopback startup path.

## Best Possible Solution

Keep this PR open, but require a rebase against current main. If maintainers still want stable operator-configured MCP URLs, add `gateway.mcpLoopback.port` to the gateway config type, strict Zod schema, generated schema/help/labels, and public gateway configuration docs, then thread the configured value into the current lazy CLI-backend `ensureMcpLoopbackServer(configuredPort)` path with targeted tests for configured-port binding and bind-failure diagnostics. If maintainers decide `mcpResumeHash` fully removes the product need, close only after recording that product decision or linking a canonical follow-up.

## Risks / Open Questions

- Merging the PR as-is would likely miss the current lazy `ensureMcpLoopbackServer()` path used by CLI backends, because the branch patches the old eager gateway startup path.
- Because stable `mcpResumeHash` already fixes the session-reset symptom, maintainers should confirm whether an operator-pinned MCP URL remains a desired config surface before adding a new public gateway key.
- A fixed port can collide or fail during restarts; the rebased implementation should preserve useful configured-port diagnostics in the current lazy startup flow.
- A predictable fixed local port should remain strictly loopback-bound and bearer-token protected; current low-level code has those protections, and the rebase should not weaken them.
