## Summary

Keep open. Current main still models memory-flush writes as one append-only daily memory target, so a flush prompt that asks for SESSION_HANDOFF.md or another explicit extra file would still be rejected by the write wrapper. The prior implementation attempt #66765 is closed and unmerged; related open issues #63664 and #68995 overlap the same restriction area but do not fully supersede this specific handoff-file request.

## Best Possible Solution

Keep this issue open and fix the memory-flush contract deliberately: add an explicit, validated additional write-target mechanism to the memory flush plan/config or plugin contract; thread it through the embedded runner and tool creation; allow full overwrite for SESSION_HANDOFF.md while preserving append-only behavior for the daily memory file; reject unrelated paths and path escapes; and add focused regression tests for allowed handoff writes, daily-memory append behavior, unrelated-path rejection, and traversal attempts.

## Risks / Open Questions

- Related open issues #63664 and #68995 touch the same write-policy area, so the final contract should be coordinated rather than patched narrowly in only the write wrapper.
