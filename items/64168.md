## Summary

Keep open. Current main does not provide strong evidence that the Telegram `/new` leak is fixed: the reset hook path still routes any `event.messages` produced by command hooks back to the originating channel, and the docs define `event.messages` as user-visible. The bundled `session-memory` hook itself now appears internal-only, but the reported `[note self-improvement (before reset): ...]` source is not identifiable in current source, and there is no Telegram-specific regression test proving that such notes are suppressed.

## Best Possible Solution

Keep the issue open and add a focused regression path: reproduce a Telegram `/new` reset with an enabled hook or fixture that emits an internal-looking self-improvement note, then either move that note to logging/internal storage or introduce a non-user-visible hook note channel. Add a Telegram or shared reset-hook test proving `/new` still produces the normal reset greeting but does not deliver internal notes unless a hook intentionally opts into user-visible replies.
