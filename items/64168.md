## Summary

Current main still leaves the reported Telegram `/new` leak path open. Bare `/new`/`/reset` can prepend runtime-loaded daily memory into the first reset prompt, startup context reads raw recent `memory/YYYY-MM-DD*.md` files, the bundled session-memory hook still persists assistant text from pre-reset transcripts into those files, and Telegram delivery uses the shared reply path without a guard for `[note self-improvement (before reset): ...]`-style output. The provided related WhatsApp report is a different channel-specific echo issue, not a canonical fix for this Telegram/startup-memory path.

## Best Possible Solution

Keep this issue open. Add a focused regression test for a Telegram/native bare `/new` or `/reset` where startup memory or session-memory contains `[note self-improvement (before reset): ...]`, then prevent internal hook/self-improvement notes from entering the startup prelude or suppress that artifact before user-visible reset delivery. Likely touch points are `src/auto-reply/reply/startup-context.ts`, `src/hooks/bundled/session-memory/transcript.ts`, and possibly the shared reply delivery sanitizer if the product decision is output-time suppression.
