## Summary

Keep open. Current main has partial restart resilience, but the specific Discord-visible recovery behavior is still not proven: restart recovery for marked main sessions replays internally with deliver:false, and outbound channel delivery only runs when delivery is explicitly enabled. The related Feishu restart-drop item is also still open for the same cross-channel recovery gap.

## Best Possible Solution

Keep the issue open and fix this as a generic external-channel restart recovery path: persist affected in-flight session route metadata when restart aborts or stale-lock cleanup marks a run, then on startup either resume with the original delivery route enabled or post a visible recovery message to the original Discord channel. Cover the behavior with a generic channel regression test and a Discord-oriented route/delivery test.
