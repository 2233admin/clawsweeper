## Summary

Keep #69249 open. Current main has partial restart hardening, but the Discord-visible failure described by the issue is not fully solved: direct startup recovery can mark and internally resume restart-aborted main sessions, while the only routed user-visible restart notice/continuation path is the restart-sentinel flow for managed restarts. The Discord inbound path records enough route metadata to make a fix plausible, but main-session restart recovery currently resumes with `deliver: false` and does not post a recovery/apology/resume message back into the Discord channel.

## Best Possible Solution

Keep the issue open and implement a channel-visible restart-abort recovery path. The likely fix belongs in startup/main-session recovery: persist or derive the interrupted session's last delivery context, enqueue an idempotent recovery notice or routed continuation for the original Discord target, and add regression coverage that a restart-aborted `agent:main:discord:*` session with no visible assistant follow-up either resumes with external delivery or posts a clear recovery message. Consider coordinating with the open Feishu restart-notification gap so the solution is generic across channel plugins rather than Discord-only.
