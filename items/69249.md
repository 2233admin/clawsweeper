## Summary

Keep open. Current main has useful managed-restart mitigations, but it still does not provide an automatic Discord-visible recovery message for an in-flight turn aborted by a gateway restart. The Discord abort path returns without fallback delivery, main-session restart recovery resumes with `deliver: false`, and the checked-in QA scenario only proves recovery on a subsequent user turn. Related #38836 remains open for the same generic channel restart-notification gap on another channel.

## Best Possible Solution

Keep #69249 open and fix this as a generic external-channel restart recovery path. Persist route metadata for in-flight user-visible turns when restart aborts or stale-lock cleanup marks a run, then on startup either resume with delivery enabled to the original route or post a clear recovery message to the original Discord channel. Add a generic channel regression test plus a Discord-specific route/thread delivery test.
