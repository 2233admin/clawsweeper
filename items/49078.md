## Summary

Keep open. Current `main` still models cron result routing as a single `delivery` object with only `announce`, `webhook`, or `none`, and the docs/runtime still describe only channel/webhook fallback delivery. There is no implemented `output` field, no multi-target routing list, no built-in file sink, and no post-run session injection target separate from the job's execution session.

## Evidence

- **Gateway cron schema has no `output` field and only three delivery modes:** `CronPayloadSchema` and `CronDeliverySchema` define job payload plus a single delivery object. Supported delivery modes are only `none`, `announce`, and `webhook`; there is no `output.targets`, file target, or session target list in the protocol.

There is no type for file output, multiple targets, or output-to-session injection after completion.

## Risks / Open Questions

- There is partial overlap with existing features: isolated jobs already have fallback channel/webhook delivery, and jobs can run in `current` or `session:<id>` sessions. That overlap does not satisfy the requested deterministic post-run output routing contract, but it may affect how maintainers want to scope a future implementation.
- Repository history available in this checkout appears shallow, so this review is based on current `main` code/docs rather than a full historical search for abandoned or superseded implementations.
