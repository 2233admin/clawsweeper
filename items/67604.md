## Summary

Keep open. The PR addresses a plausible cron-session cleanup gap, but the proposed implementation is not conservative enough for cleanup: it replaces current time-based run-session pruning with run-log-dependent eligibility, and the review thread surfaces concrete regressions that would cause some stale sessions to accumulate indefinitely.

## Evidence

- **Current main reaper is simple retention-based pruning:** `sweepCronRunSessions()` on current `main` deletes any `:cron:<job>:run:<id>` session whose `updatedAt` is older than the retention cutoff; it does not depend on run-log completion state.
  - file: [src/cron/session-reaper.ts:54](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/cron/session-reaper.ts#L54)
  - command: `nl -ba src/cron/session-reaper.ts | sed -n '1,170p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Current timer always runs the reaper in finally:** `onTimer()` calls `sweepCronRunSessions()` from the `finally` block after cron work, so the shipped behavior today is periodic retention cleanup of run sessions even when execution paths throw.
  - file: [src/cron/service/timer.ts:816](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/cron/service/timer.ts#L816)

- **Docs describe generic retention cleanup, not run-log-gated cleanup:** User-facing docs say `cron.sessionRetention` prunes isolated/completed cron run sessions after the retention window. They do not make pruning contingent on preserved run-log entries or only successful statuses.
  - file: [docs/gateway/configuration-reference.md:1011](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/docs/gateway/configuration-reference.md#L1011)

## Risks / Open Questions

- Failed cron runs could accumulate indefinitely if merged as reviewed, because terminal `error` statuses would not satisfy the new reclaim gate.
- Older stale sessions could stop being reclaimable once per-job run logs are rotated or missing, reintroducing storage growth under normal log-pruning behavior.
- The patch still appears to violate the repo’s English-only comment policy, adding avoidable review friction on a large behavior change.
