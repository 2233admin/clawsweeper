## Summary

Keep #71784 open. Current main still has the reported live-query failure shape: builtin memory search embeds the query once, remote OpenAI-compatible and Gemini embedding providers route through the guarded HTTP helper without a transport retry, and the readiness probe exercises the batch path instead of the live search path. Related #44166/#44167 are batch/reindex transport-retry work, while #71849 is a separate voice-call fast-context feature, so neither supersedes this issue.

## Best Possible Solution

Keep the issue open and fix the live builtin memory-search embedding path. The conservative path is to add a bounded retry/backoff for retryable transport failures around embedQueryWithTimeout or the shared single-embed remote HTTP helper, extend the classifier/tests to cover fetch failed, other side closed, ECONNRESET/UND_ERR_SOCKET, and TLS socket disconnects, and keep permanent provider/auth/config errors non-retryable. Dispatcher keep-alive tuning can be added if live validation confirms it is needed, but the user-visible reliability gap is the single-attempt live query path.
