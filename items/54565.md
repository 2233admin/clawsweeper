## Summary

Keep open. Current main still does not implement per-group WhatsApp debounce. The config type, strict schema, generated metadata, runtime monitor, and tests still only support channel/account/global inbound debounce, while group entries only cover mention/tool/prompt behavior. The existing debouncer has a per-item resolver seam, so this looks feasible in the WhatsApp plugin without a broad core redesign, but the requested behavior is not shipped yet.

## Best Possible Solution

Keep this issue open. Implement the WhatsApp case first by adding `debounceMs?: number` to the WhatsApp group config type, strict Zod schema, generated metadata, UI/help/docs, and regression tests. At runtime, resolve exact group then wildcard group debounce per inbound group message and feed it through the existing `createInboundDebouncer.resolveDebounceMs` seam, falling back to account/channel/global debounce for other chats. Include tests proving targeted group overrides do not affect other groups, and explicitly decide whether WhatsApp media messages should be batchable for the original image/image/text use case. Treat Telegram/Discord/other channels as separate follow-up design unless maintainers want a shared plugin contract.

## Risks / Open Questions

- The reported image/image/text workflow will not be fully solved by a group-level scalar alone unless WhatsApp media messages are explicitly allowed to participate in batching; current docs and runtime treat media as immediate-flush.
- The issue mentions Telegram, Discord, and other channels, but group config semantics differ by plugin. Shipping a shared cross-channel key without per-plugin design could create inconsistent behavior or an accidental public contract.
