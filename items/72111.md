## Summary

Keep open. Current main still treats a mixed `command` + `url` + `transport: "streamable-http"` MCP server as stdio because stdio resolution runs first and succeeds when `command` is present. The reported 30000ms timeout remains plausible because OpenClaw then waits for stdio JSON-RPC while the child process is actually serving HTTP on a port. Related issue #66940 only covered the streamable-http Accept header path for URL-only HTTP servers, not managed HTTP child-process lifecycle.

## Best Possible Solution

Keep the issue open. The best fix is either to add an explicit managed streamable-http child-process mode with port readiness probing, HTTP client connection, lifecycle cleanup, and regression coverage, or to reject mixed `command` + `url`/HTTP transport configs before runtime with a clear error and aligned docs. The current silent stdio precedence should not remain the observable timeout path.
