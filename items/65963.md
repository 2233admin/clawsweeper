## Summary

Keep this issue open. Current main has reduced the worst narrative-session leak with deterministic narrative session keys, `deleteSession`, and a stale artifact scrubber, but the main operator-facing requests remain unresolved: no supported `dreaming.model` config, no retention/burst controls, no `sessions purge --kind dreaming`, and no dreaming-specific hide/collapse behavior in the session picker or CLI list.

## Best Possible Solution

Keep this issue open for a memory-core-owned product fix: expose and document a supported dreaming execution/model override, wire it into narrative subagent runs and any phase execution paths, add bounded retention or purge-by-kind semantics for generated dreaming sessions/transcripts, and add targeted CLI/UI behavior to hide, collapse, or purge dreaming artifacts. #65589 can inform cost guardrails but is closed unmerged and does not supersede this issue.

## Risks / Open Questions

- Closing now would leave the reported cost-control problem unresolved because dreaming narrative subagent work still follows the default runtime model.
- The internal `MemoryDreamingExecutionConfig.model` type can look like partial implementation, but without manifest/docs/runtime wiring it is not a user-facing or effective config contract.
- Future work should build on the current cleanup and scrubber so retention/purge improvements do not reintroduce races with detached narrative sessions.
