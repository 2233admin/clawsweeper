## Summary

Keep open. Current main still implements `--fix-missing` by resolving one path from the persisted `sessionFile` and deleting the store entry when that single path is absent. Normal transcript readers use `resolveSessionTranscriptCandidates(...)` and prefer the `<sessionId>.jsonl` file before a stale generated `sessionFile`, so the reported mismatch can still produce false `prune-missing` cleanup plans. The issue also has a second-user confirmation, and the overlapping ClawSweeper report for #50248 was kept open for the same remaining gap.

## Best Possible Solution

Keep this issue open and fix the core session maintenance path. `sessions cleanup --fix-missing` should use the same candidate-resolution contract as session readers: resolve safe candidates for the entry, treat the entry as present when any current candidate exists, and only prune when no acceptable transcript candidate exists. Add focused regression coverage for a stale generated `sessionFile` whose `<sessionId>.jsonl` transcript exists, plus a custom `sessionFile` case to avoid breaking intentional overrides. Update the doctor missing-transcript check in the same change or track it explicitly so doctor does not continue recommending unsafe cleanup.

## Risks / Open Questions

- A fix must preserve legitimate custom `sessionFile` paths, forked transcript names, topic transcript names, and cross-agent absolute session paths that current candidate-resolution tests protect.
- If only `sessions cleanup` changes, `openclaw doctor` can still report recent sessions as missing through the same single-path check and steer users toward unsafe cleanup.
- Archived/deleted transcript variants need a deliberate policy decision: the issue examples include both active `<sessionId>.jsonl` files and archived `.deleted.*` files.
