## Summary

Keep open. Current main still only partially addresses the request: `agents.defaults.imageMaxDimensionPx` and OpenAI-compatible Chat Completions image budgets exist, but there is still no shared configurable image limit contract for sanitizer bytes, sharp input pixels, generic media caps, media-store/sandbox staging, gateway attachment parsing, or ACP attachment reads.

## Best Possible Solution

Keep this issue open as the canonical broad tracker. The best implementation is a backwards-compatible, documented image limit contract that threads configured byte, input-pixel, dimension, and per-request total limits through agent image sanitization, media/image-ops, generic inbound media, media-store/sandbox staging, gateway attachment parsing, ACP attachments, and OpenAI-compatible HTTP paths, while preserving existing narrower knobs as aliases or scoped overrides. Treat #40950 as only the narrower sandbox/media-store piece.

## Risks / Open Questions

- Closing now would leave several user-visible image byte and pixel limits unsupported by config despite partial documented knobs.
- The eventual fix needs bounded defaults and abuse controls because larger image limits affect memory, disk, network, and provider request cost.
- The config design should reconcile existing partial settings such as `agents.defaults.imageMaxDimensionPx`, `tools.media.image.maxBytes`, and `gateway.http.endpoints.chatCompletions.images.*` without breaking existing configs.
