## Summary

Current `main` does not contain this PR's Feishu fixes. The checked-out `main` at `01bc49c88c0cef9807b15df4d7486a0c5fe4afbd` still drops reaction synthetic events when Feishu omits `chat_type`, and it still replies using the synthetic reaction `message_id` rather than unwrapping back to the original `om_*` message. This PR is therefore not implemented on `main`, and there is not strong evidence to close it for any other allowed reason.

## Evidence

- **Missing `chat_type` still becomes `undefined` on lookup:** `parseFeishuMessageItem` only accepts `group`, `private`, or `p2p`; any missing/invalid `chat_type` still returns `undefined`, so current `main` does not infer chat type from `chat_id` prefixes as proposed in the PR.
  - file: [extensions/feishu/src/send.ts:270](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/extensions/feishu/src/send.ts#L270)

- **Reaction synthetic events still require recovered chat type:** `resolveReactionSyntheticEvent` still drops the event when neither the webhook payload nor the fetched message yields a normalized chat type, which matches the bug the PR is trying to fix.
  - file: [extensions/feishu/src/monitor.account.ts:143](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/extensions/feishu/src/monitor.account.ts#L143)
