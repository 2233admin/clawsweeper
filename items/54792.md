## Summary

Keep open. Current main has a clearer runtime/startup backstop and docs for `hooks.enabled` requiring `hooks.token`, but the shared schema/config-write validation requested here is still missing. `config.patch`, `config.apply`, `config.set`, and lower-level config persistence still rely on validators that do not reject `hooks.enabled=true` with no non-empty token before writing.

## Best Possible Solution

Keep this issue open. Add a shared cross-field config validation rule that rejects `hooks.enabled=true` unless `hooks.token` is a non-empty string, route it through `validateConfigObjectRaw` and `validateConfigObjectWithPlugins` so `config.set`, `config.apply`, `config.patch`, `writeConfigFile`, CLI writes, and plugin/runtime config writes reject the invalid state before persistence, and keep `resolveHooksConfig` as the startup/reload backstop. If `config.schema.lookup` is meant to guide clients before edits, extend the lookup/hint schema with dependency or warning metadata for `hooks.enabled`.

## Risks / Open Questions
