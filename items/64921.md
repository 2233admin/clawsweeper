## Summary

Keep open. Current main still resolves plugin text transforms through the indirect createRequire-based runtime helper, while the direct main-process active registry seam exists but is not used by resolveRuntimeTextTransforms(). The documented plugin contract and prompt application paths remain wired to this resolver, and the related #65331 is still an open, unmerged implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open and review or land #65331, or an equivalent patch, that makes resolveRuntimeTextTransforms() consult getActivePluginRegistry() directly in the main process and keeps the existing loadPluginRuntime() fallback for external plugin contexts. Add focused regression coverage that registers a plugin textTransform and proves it is applied through the CLI/provider prompt path.
