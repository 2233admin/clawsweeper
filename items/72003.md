## Summary

Keep open. The report is recent, concrete, and not safely closeable from current main: the WhatsApp inbound path appears intended to set `ChatType`, but the direct-context prompt injection still depends on `sessionCtx.ChatType` being exactly `direct`/`dm`, and current tests do not prove the full WhatsApp per-peer compiled prompt contains the delivery-context line. The issue's observation remains plausible because inbound metadata can still render `chat_type: "direct"` when the prompt-injection gate did not pass.

## Best Possible Solution

Keep the issue open and add a focused regression test that builds a WhatsApp direct `per-channel-peer` inbound/session path through prompt assembly and asserts the compiled system prompt contains `You are in a WhatsApp direct conversation. Your replies are automatically sent to this conversation.` If it fails, fix the root cause where `ChatType` is dropped or make direct-context injection use normalized/persisted session origin metadata instead of relying only on the raw `sessionCtx.ChatType` field. Also check Signal/Telegram direct sessions with the same test shape.
