## Summary

Current `main` already hard-times out wedged Telegram `getUpdates` requests and rebuilds the transport on polling/network recovery, which addresses the reported 90-110s stuck-poll behavior.

## Evidence

## Close Comment

This looks fixed on current `main`, so I would close this as implemented.

- Codex review of `main` found an explicit hard timeout for Telegram `getUpdates`: `extensions/telegram/src/request-timeouts.ts:6` sets it to `45000` ms, and `extensions/telegram/src/bot-core.ts:181`-`239` aborts the fetch when that timeout fires.
- The polling path already rebuilds the Telegram transport after stalls and recoverable network errors: `extensions/telegram/src/polling-session.ts:318`-`389` and `extensions/telegram/src/polling-transport-state.ts:31`-`45`.
- The transport itself now uses bounded undici keep-alive settings, including `keepAliveTimeout: 30000`, in `extensions/telegram/src/fetch.ts:38`-`58` and `293`-`364`.
- There is direct regression coverage for the stuck-poll case: `extensions/telegram/src/bot.fetch-abort.test.ts:72`-`90` asserts that a wedged `getUpdates` request is aborted after 45 seconds.
- The maintainer changelog also records this exact fix in `CHANGELOG.md:2337`: hard-timeout stuck `getUpdates` requests so wedged network paths fail over sooner.

I could not confirm a released version carrying this yet, so if anyone still sees the old 90-110s hangs they should retest on a build from current `main` and open a fresh issue with new logs if it still reproduces.
