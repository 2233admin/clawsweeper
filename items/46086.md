## Summary

Keep open. Current `main` still enforces multimodal memory only for providers whose adapter explicitly declares multimodal embedding support, and the built-in `local` adapter does not. The issue is neither fixed nor obsolete from the current code/docs/tests.

## Evidence

- **Config validation still rejects unsupported multimodal providers:** `resolveMemorySearchConfig` throws when `memorySearch.multimodal` is active and the selected provider adapter does not report `supportsMultimodalEmbeddings`; it also requires `memorySearch.fallback = "none"`.
  - file: [src/agents/memory-search.ts:390](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/agents/memory-search.ts#L390)
  - command: `nl -ba src/agents/memory-search.ts | sed -n '380,409p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Built-in local provider does not advertise multimodal support:** The built-in `local` memory embedding adapter defines `id: "local"`, transport `local`, and `create`, but no `supportsMultimodalEmbeddings` hook, so the validation path treats it as unsupported for multimodal memory.
  - file: [extensions/memory-core/src/memory/provider-adapters.ts:83](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/extensions/memory-core/src/memory/provider-adapters.ts#L83)
