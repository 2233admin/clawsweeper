## Summary

Current `main` now preserves outbound route/account metadata for spawned/background sessions and reuses the requester session’s stored route for automated announce delivery, which matches the reported WeChat-only failure mode where normal replies worked but subagent/cron pushes did not.

## Close Comment

Closing this as implemented on current `main` after a Codex review.

- OpenClaw now seeds spawned subagent sessions with the request delivery context so background sessions do not lose `channel` / `to` / `accountId` / `threadId` before announce delivery (`src/gateway/server-methods/agent.ts:648`, `src/gateway/server.agent.subagent-delivery-context.test.ts:79`).
- The subagent completion path now merges completion-only origins with the requester session's stored route (`lastChannel` / `lastTo` / `lastAccountId`) before attempting direct delivery, which is the failure mode described here for automation-only WeChat sends (`src/agents/subagent-announce-delivery.ts:464`).
- Current tests cover direct completion delivery with preserved route metadata and implicit route/account fallback for background sends (`src/agents/subagent-announce-delivery.test.ts:141`, `src/gateway/server-node-events.test.ts:894`, `src/gateway/server.agent.gateway-server-agent-a.test.ts:375`).
- Current docs also describe the repaired behavior for subagent completions and task delivery (`docs/help/faq.md:224`, `docs/automation/tasks.md:127`).
- Best in-repo proof of the implementation is commit `9ece33c50506ba031df382f69bde65dfda19bea5` on `main`.
