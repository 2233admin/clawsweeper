## Summary

Keep open. Current `main` does not implement the requested `#doc`/force-document behavior for WhatsApp media delivery. The WhatsApp outbound path still relies on MIME-kind auto-detection, and there is no `#doc` fragment handling in the media loader or WhatsApp outbound adapter.

## Evidence

- **No `#doc` support on current main:** Repo search found no `#doc` handling in WhatsApp outbound, media loading, outbound delivery, agent tools, or docs.
  - command: `rg -n "#doc" extensions/whatsapp src/media src/infra/outbound src/agents/tools docs -S`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Media loader does not strip local fragments or force document kind:** `loadWebMediaInternal()` strips only a leading `MEDIA:` prefix, then resolves the path/URL and derives kind from detected MIME. There is no fragment-based override to `document`.
  - file: [src/media/web-media.ts:398](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/media/web-media.ts#L398)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **WhatsApp outbound still auto-maps by loaded media kind:** `sendMessageWhatsApp()` sends a document only when loaded media kind is already `document`; images remain images. There is no option or branch to force an image upload through the document path.
  - file: [extensions/whatsapp/src/send.ts:117](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/whatsapp/src/send.ts#L117)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **WhatsApp normalization preserves only image/audio/video; everything else becomes document:** `normalizeWhatsAppLoadedMedia()` converts unknown/non-media kinds to `document`, but it has no input for `#doc` or any force-document flag.
  - file: [extensions/whatsapp/src/outbound-media-contract.ts:69](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/whatsapp/src/outbound-media-contract.ts#L69)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Existing tests cover only natural document kinds, not forced image-as-document:** WhatsApp send tests assert document behavior for an already-`document` media result (`file.pdf`), which confirms current behavior is MIME-kind driven rather than user-forced via fragment.
  - file: [extensions/whatsapp/src/send.test.ts:343](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/whatsapp/src/send.test.ts#L343)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Current public force-document knob is explicitly Telegram-only:** The message tool schema exposes `forceDocument`/`asDocument`, but the description scopes it to Telegram only, not WhatsApp.
  - file: [src/agents/tools/message-tool.ts:125](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/tools/message-tool.ts#L125)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)

## Risks / Open Questions

- Cross-referenced PR #42044 suggests prior work existed, but without the merged commit in this checkout and with no corresponding code on `main`, there is not enough evidence to close as implemented.
- A local path like `MEDIA:/path/to/image.png#doc` would still fail path resolution or remain MIME-detected, because fragment stripping for local media is absent.
