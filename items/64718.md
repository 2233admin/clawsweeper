## Summary

Keep open. Current main still lacks the sender-aware non-owner exec deny behavior, and this PR remains a plausible security fix candidate rather than obsolete cleanup. It also has an unresolved security concern around partial agent exec defaults and is paired with the same-author open hardening PR #64719, so maintainer/security review should decide the final boundary.

## Best Possible Solution

Keep this PR open for maintainer/security review. Revise the non-owner branch so user-injectable inline/session exec fields are ignored, no-override non-owner turns fail closed with security=deny and ask=always, partial agentExecDefaults cannot inherit permissive downstream defaults, and the behavior is coordinated with #64719's broader dangerous-tool policy. Add focused unit tests for no overrides, inline injection, persisted session overrides, partial agent defaults, and explicit safe operator defaults before merge.

## Risks / Open Questions

- Merging the PR as-is may still allow non-owner channel auto-reply exec when an operator has a partial agent tools.exec object that omits explicit security and ask, because downstream defaults remain full/off for gateway/node.
- The PR overlaps with #64719's broader dangerous-tool filtering; landing both independently could leave inconsistent override semantics for channel auto-reply tools.
- This is command-execution security behavior, so it needs maintainer/security review and regression coverage for no overrides, inline directive injection, persisted session overrides, partial agent defaults, and the intended operator override path.
