## Summary

Keep open. Current `main` still drops partial reindex progress on any full-reindex failure, only retries rate-limit/5xx embedding errors rather than bare transport failures like `TypeError: fetch failed`, and still logs memory sync failures via `String(err)` without preserving a stack/cause path.

## Evidence

- **Transport retry matcher still excludes bare fetch failures:** `isRetryableMemoryEmbeddingError` only matches rate-limit/429/5xx/cloudflare/token-budget text. It does not match `fetch failed`, socket resets, DNS failures, or other undici transport errors described in the issue.
  - file: [extensions/memory-core/src/memory/manager-embedding-policy.ts:83](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/memory-core/src/memory/manager-embedding-policy.ts#L83)
  - command: `nl -ba extensions/memory-core/src/memory/manager-embedding-policy.ts | sed -n '1,220p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Batch embedding path still uses that narrow retry policy:** Full indexing calls `runMemoryEmbeddingRetryLoop` from `embedBatchWithRetry` and `embedBatchInputsWithRetry`, so a bare transport failure still escapes the retry loop and aborts indexing.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:242](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/memory-core/src/memory/manager-embedding-ops.ts#L242)

- **Atomic reindex still deletes temp DB on any failure:** `runMemoryAtomicReindex` unconditionally removes the temp index files in the catch path before rethrowing, so partial progress is not resumable after an embedding failure.
  - file: [extensions/memory-core/src/memory/manager-atomic-reindex.ts:41](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/memory-core/src/memory/manager-atomic-reindex.ts#L41)
