## Summary

Current main still does not implement the requested send-time Feishu cross-bot broadcast. Existing Feishu broadcast support fans out one inbound human/platform event to multiple agents, but outbound Feishu replies are sent directly and are not synthesized back into other bot accounts. The linked discussion and related reports describe a concrete platform parity gap, so this issue should remain open.

## Best Possible Solution

Keep this issue open. Implement an opt-in Feishu-owned send-time synthetic inbound path keyed by configured peer bots/accounts, with per-group/account scoping, dedupe, loop limits, tests for outbound replies and multi-account behavior, and Feishu docs. Add a generic plugin/channel seam only if multiple bundled or third-party channels need the same capability.

## Risks / Open Questions

- Synthetic bot-to-bot delivery can create reply loops or token amplification unless it is opt-in, scoped, deduped, and guarded by consecutive-bot-turn limits.
- Existing Feishu inbound broadcast partially overlaps the multi-agent goal, so a future implementation should reuse or deliberately evolve that behavior instead of adding a confusing parallel config path.
- A cross-channel core abstraction may be premature; the first concrete implementation likely belongs in the Feishu plugin, with a generic SDK/core seam only if Telegram or another channel needs the same mechanism.
