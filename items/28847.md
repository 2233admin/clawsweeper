## Summary

Current `main` already covers several parts of #28847: multi-profile provider auth, cooldown/backoff after transient failures, model-aware rate-limit cooldowns, capped same-provider rotations, and probe throttling to reduce retry storms. But the requested Gemini project bucket behavior is still missing: there is no bucket/quota-group concept in auth config or runtime state, and `projectId` is only carried as credential metadata, not used to share cooldown across keys from the same project.

## Evidence

- **Multi-profile auth rotation exists:** Auth resolution already supports multiple profiles per provider, explicit ordering, round-robin ordering by last use, and moving cooldowned profiles to the end.
  - file: [src/agents/auth-profiles/order.ts:66](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/agents/auth-profiles/order.ts#L66)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Transient failure cooldowns already exist:** Transient failures mark auth profiles unavailable with cooldown windows; rate-limit failures can be model-scoped via `cooldownModel`.
  - file: [src/agents/auth-profiles/usage.ts:519](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/agents/auth-profiles/usage.ts#L519)

## Risks / Open Questions

- The issue body asks specifically for Gemini project bucket-aware cooldown and shared-bucket rotation; closing now would overstate what `main` implements.
- `docs/concepts/model-failover.md` documents multi-profile rotation and cooldowns, which may make the issue look solved at a glance even though bucket/project grouping is absent.
- The existing `projectId` field on OAuth credentials could be mistaken for implemented bucket support, but current runtime search results show it is not part of auth selection or cooldown aggregation.
