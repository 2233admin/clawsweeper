## Summary

Keep open. Current main still implements and documents direct ACP-bound follow-up routing: bound conversation/thread text is sent to the ACP session as prompt text, and the dispatch path calls the ACP runtime with the resolved prompt text rather than requiring the parent assistant to rewrite shorthand into an explicit child task. The related ACP cleanup items in the provided context cover narrower routing/transcript failures and do not supersede this parent-orchestration policy request.

## Best Possible Solution

Keep the issue open for maintainer product judgment. If this direction is accepted, implement an explicit parent-orchestrated ACP binding mode/default in the bound conversation/thread dispatch path, update ACP/subagent docs to distinguish continuity from verbatim relay, and add regression coverage proving shorthand thread follow-ups are rewritten before they reach the ACP child while explicit verbatim relay still passes through unchanged.

## Risks / Open Questions

- Changing this default would be a product behavior change because current docs and tests describe direct ACP-bound follow-ups as intentional routing.
- A parent-orchestrated default would need a clear design for which parent session interprets a thread-bound follow-up, how exact verbatim relay is requested, and how existing direct ACP thread UX remains available or migrates.
