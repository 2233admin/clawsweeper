## Summary

Keep open. Current main still treats ACP conversation/thread binding as direct routing to the bound ACP session, and the existing docs/tests describe that normal follow-up text lands in the ACP harness session rather than being parent-orchestrated and rewritten by default. The provided related items cover other ACP behavior such as timeouts, binding failures, status display, and transcript routing, but none supersede this parent-rewrite policy request.

## Best Possible Solution

Keep the issue open for maintainer/product judgment. A plausible fix would be a core/channel orchestration policy that distinguishes conversation-bound direct delivery from parent-owned ACP child follow-ups, plus docs/tests proving shorthand user replies are interpreted by the parent and rewritten into explicit child tasks unless the user requests verbatim forwarding.
