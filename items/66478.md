## Summary

Current main still does not normalize Feishu reaction emoji input: the shared Feishu `react` action trims the raw `emoji` value and forwards it unchanged into both add and remove/list reaction API paths. This PR remains a focused implementation candidate for the open linked bug, but it should be revised before landing to preserve the Feishu lazy runtime boundary and verify the mapping against Feishu's current `emoji_type` vocabulary.

## Best Possible Solution

Keep this PR open for maintainer review or land an equivalent Feishu-owned fix for #66406. The strongest path is a pure Feishu emoji normalizer backed by Feishu's current `emoji_type` vocabulary, applied before add and remove/list reaction calls, with tests for Unicode thumbs-up, pass-through type strings, mixed-case input, trimming, unknown input behavior, and remove/list filtering. Place the normalizer where it does not eagerly import the Feishu runtime SDK from `channel.ts`.
