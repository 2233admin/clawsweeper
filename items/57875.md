## Summary

Current `main` already adds Discord-specific block-stream coalescing that keeps short multi-paragraph replies buffered and merged into one outbound message instead of sending chunk-boundary fragments.

## Evidence

- **Discord now has provider-default block coalescing:** The Discord channel plugin advertises `blockStreamingCoalesceDefaults: { minChars: 1500, idleMs: 1000 }`, which means a 500-1000 character reply stays buffered and is not emitted as multiple block-stream messages.
  - file: [extensions/discord/src/shared.ts:109](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/extensions/discord/src/shared.ts#L109)

  - file: [src/auto-reply/reply/block-streaming.ts:203](https://github.com/openclaw/openclaw/blob/9ece33c50506ba031df382f69bde65dfda19bea5/src/auto-reply/reply/block-streaming.ts#L203)

- This was a code-and-history review only; I did not run a live Discord reproduction in the read-only checkout.
- If a user explicitly overrides `channels.discord.blockStreamingCoalesce` or global block-stream settings to much smaller thresholds, multi-message block delivery can still be configured intentionally.

## Close Comment

This looks implemented on current `main`, so I would close it.

- Codex review found Discord-specific block-stream coalescing on `main` at `extensions/discord/src/shared.ts:109`, with a default `minChars` of `1500` and `idleMs` of `1000`.
- The block-stream resolver in `src/auto-reply/reply/block-streaming.ts:203` now reads those channel defaults and merges chunked blocks with paragraph/newline-aware joiners instead of emitting each chunk separately.
- The coalescer behavior is covered by tests in `src/auto-reply/reply/reply-utils.test.ts:746`, which verify consecutive paragraph chunks are combined into one payload and still flush correctly at the end.
- The streaming docs were updated in `docs/concepts/streaming.md:68` to document the new default: Discord block streaming now gets a raised coalesce threshold specifically to avoid small fragmented replies.
- This evidence is on `main` in commit `9ece33c50506ba031df382f69bde65dfda19bea5`, dated 2026-04-24, which is after the latest release tag `v2026.4.23`, so there is no shipped release tag to cite yet.
