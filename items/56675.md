## Summary

Keep open. Current `main` still allows Slack draft-preview streaming and block streaming to run at the same time on the non-native path, so the mutual-exclusion cleanup requested in #56675 is not implemented. Some narrower sub-fixes from the issue are already present on `main` (thread target selection for native stream start, duplicate-stop guard), but the core stale-preview/duplicate-delivery class remains.

## Evidence

- **Draft preview still initializes on preview-only path:** `shouldInitializeSlackDraftStream` returns `previewStreamingEnabled && !useStreaming`, and `dispatchPreparedSlackMessage` uses that result to create a Slack draft stream. There is no block-streaming check in this path, so preview streaming still starts when native transport is off.

  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Block streaming is only force-disabled for native streaming:** `disableBlockStreaming` is set to `true` only when `useStreaming` is true. Otherwise Slack passes `false` when channel block streaming is explicitly enabled, or `undefined` to fall back to agent defaults. That means preview streaming and block streaming can still coexist on the non-native path.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:862](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/slack/src/monitor/message-handler/dispatch.ts#L862)

- **Partial-preview updates still run independently of block streaming:** `onPartialReply` stays enabled whenever `previewStreamingEnabled` is true and native streaming is not in use. There is no guard here for block-streaming activity, so draft previews continue updating even when block replies are being sent.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:869](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/slack/src/monitor/message-handler/dispatch.ts#L869)
