## Summary

Keep open. Current `main` adds plugin-accessible diagnostic events for per-call model lifecycle observation, but it does not add the requested `before_model_call` / `after_model_call` hook contract, and the existing docs still state `llm_input` / `llm_output` are not byte-for-byte provider-call captures.

## Evidence

- **Requested hook names are not on the public plugin hook surface:** `PluginHookName` and `PLUGIN_HOOK_NAMES` include `llm_input`, `llm_output`, `before_tool_call`, and `after_tool_call`, but no `before_model_call` or `after_model_call`.
  - file: [src/plugins/hook-types.ts:57](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/plugins/hook-types.ts#L57)

  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Docs still describe current model hooks as adapter-level observations:** The Codex harness docs explicitly say `llm_input` and `llm_output` are adapter-level observations, not byte-for-byte captures of internal request payloads.
  - file: [docs/plugins/codex-harness.md:532](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/docs/plugins/codex-harness.md#L532)

## Risks / Open Questions

- A maintainer could reasonably argue the new `onDiagnosticEvent` + `model.call.*` path satisfies the underlying observability goal, but the issue explicitly asks for plugin hooks and provider-call-level hook semantics.
- The current diagnostic events omit `requestPayload` and `responsePayload`, so closing as fully implemented would overstate what `main` ships today.
