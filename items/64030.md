## Summary

Keep this issue open. Current main no longer derives new cron job ids from the session key, but the user-visible bug remains: slash/backslash-bearing `session:` targets and `current` session keys are still rejected at cron validation/normalization time. Routing still preserves channel-native peer ids in session keys, docs still advertise current/custom session targeting, and the related implementation PR #64057 was closed unmerged.

## Best Possible Solution

Keep this issue open. Implement a cron-boundary fix that accepts opaque channel-native session keys for explicit `session:<id>` and resolved `current` targets, while keeping run-log filenames UUID-backed or otherwise encoded and retaining NUL/path traversal protections where filesystem paths are built. Add regression coverage for slash-bearing DingTalk-style session keys through normalize, create/update, current-session binding, persisted validation/manual-run, and run-log safety.
