## Summary

Current main already implements `node.list` as a local state aggregation path, not per-node network polling, so the reported offline-node/event-loop blockage no longer matches the code. The same implementation is present in the latest shipped release `v2026.4.22`.

## Evidence

- **`node.list` only aggregates pairing state plus connected sessions:** The handler reads device pairing, node pairing, and `context.nodeRegistry.listConnected()`, then returns the assembled catalog; there is no per-node network probe or 30s wait in this path.
  - file: [src/gateway/server-methods/nodes.ts:697](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/gateway/server-methods/nodes.ts#L697)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Connected nodes are returned from an in-memory registry:** `listConnected()` is a synchronous map read that returns the currently registered node sessions.
  - file: [src/gateway/node-registry.ts:103](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/gateway/node-registry.ts#L103)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Device pairing source is local JSON state:** `loadState()` for device pairing reads the persisted pairing files via `readJsonFile(...)`; it does not contact nodes over the network.
  - file: [src/infra/device-pairing.ts:143](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/infra/device-pairing.ts#L143)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Node pairing source is local JSON state:** `loadState()` for node pairing also reads persisted local files via `readJsonFile(...)`, not remote node I/O.
  - file: [src/infra/node-pairing.ts:134](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/infra/node-pairing.ts#L134)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Latest release contains the same `node.list` implementation:** Inspecting the tagged release shows the same `node.list` handler body and in-memory `listConnected()` implementation in `v2026.4.22`.

  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Close Comment

Closing from a Codex review because current OpenClaw no longer matches the failure mode described here. On `main`, `node.list` is implemented as local state aggregation in `src/gateway/server-methods/nodes.ts:697`, combining persisted pairing records with `context.nodeRegistry.listConnected()`. The connected-node lookup is an in-memory map read in `src/gateway/node-registry.ts:103`, and the pairing loaders read local JSON state in `src/infra/device-pairing.ts:143` and `src/infra/node-pairing.ts:134`. The protocol docs also describe `node.list` as returning known/connected node state in `docs/gateway/protocol.md:363`. I also verified that the latest shipped release, `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), contains that same implementation, so this looks already resolved on supported builds. If you can still reproduce this on a current release, please open a new issue with exact repro steps, gateway logs, and the node/plugin/channel details involved.
