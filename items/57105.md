## Summary

Current `main` no longer relies on passing `--confirm`/`--no-confirm` through to the target CLI. Host exec approval is now an out-of-band approval-record flow with documented `approval-pending`/`/approve` behavior, and the released `v2026.4.23` tree already contains that implementation and docs.

## Evidence

- **Approval state is out-of-band, not argv mutation:** `sanitizeSystemRunParamsForForwarding` strips user-supplied approval control fields, requires a real `runId`-backed approval record, and forwards the stored `systemRunPlan`/argv for approved node runs instead of appending CLI flags.
  - file: [src/gateway/node-invoke-system-run-approval.ts:78](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/node-invoke-system-run-approval.ts#L78)

  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Guard regression is covered by test:** The node-invoke approval-bypass test expects approval fields without `runId` to fail with a `params.runId` error before forwarding, which is incompatible with the reported `--confirm/--no-confirm` argv-injection failure mode.
  - file: [src/gateway/server.node-invoke-approval-bypass.test.ts:261](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/server.node-invoke-approval-bypass.test.ts#L261)
