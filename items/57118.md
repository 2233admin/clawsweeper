## Summary

Current `main` does not reproduce the issue as written. `config.patch` is a root-level JSON merge patch API with no `path` parameter, nested object patches are the documented way to change scalar leaves, and `hooks.internal.entries` is schema-open for dynamic hook IDs on current `main`.

## Evidence

- **`config.patch` has no `path` parameter:** The current protocol schema defines `config.patch` with `raw`, `baseHash`, `sessionKey`, `deliveryContext`, `note`, and `restartDelayMs` only.
  - file: [src/gateway/protocol/schema/config.ts:30](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/protocol/schema/config.ts#L30)
  - command: `nl -ba src/gateway/protocol/schema/config.ts | sed -n '20,48p'`
- **Docs show nested-object scalar leaf patching:** The gateway docs describe `config.patch` as JSON merge patch and show `requireMention: false` patched via nested objects, not via a `path` argument.
  - file: [docs/gateway/configuration.md:565](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/docs/gateway/configuration.md#L565)
  - command: `nl -ba docs/gateway/configuration.md | sed -n '560,588p'`
- **Server still intentionally rejects non-object top-level patch payloads:** `config.patch` parses `raw` and rejects non-object payloads with `config.patch raw must be an object`; the gateway test suite asserts that behavior for `raw: "null"`.
  - file: [src/gateway/server-methods/config.ts:487](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/server-methods/config.ts#L487)

- **Dynamic hook entry keys are allowed on current main:** `hooks.internal.entries` is defined as `z.record(z.string(), HookConfigSchema)` and the generated JSON Schema exposes `additionalProperties` for entry objects, so keys like `qmd-embedder` are accepted by schema.
  - file: [src/config/zod-schema.hooks.ts:96](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/config/zod-schema.hooks.ts#L96)
  - command: `nl -ba src/config/zod-schema.hooks.ts | sed -n '96,103p'; nl -ba src/config/schema.base.generated.ts | sed -n '20488,20518p'`

  - file: [src/agents/openclaw-gateway-tool.test.ts:536](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/openclaw-gateway-tool.test.ts#L536)

## Close Comment

Closing this based on a Codex review of current `main`: I couldn't reproduce the report against the current OpenClaw `config.patch` contract.

- `config.patch` does not accept a `path` parameter on current `main`; the RPC schema only defines `raw`, `baseHash`, `sessionKey`, `deliveryContext`, `note`, and `restartDelayMs` in [src/gateway/protocol/schema/config.ts](src/gateway/protocol/schema/config.ts:30).
- The documented behavior is a root-level JSON merge patch, and the docs show nested-object scalar leaf updates such as `requireMention: false` as the supported form in [docs/gateway/configuration.md](docs/gateway/configuration.md:565).
- Current server code and tests still intentionally reject non-object top-level patch payloads like `raw: "null"`, so the `raw must be an object` response matches the documented API rather than a current regression in [src/gateway/server-methods/config.ts](src/gateway/server-methods/config.ts:487) and [src/gateway/server.config-patch.test.ts](src/gateway/server.config-patch.test.ts:349).
- `hooks.internal.entries` is schema-open on current `main`: it is defined as `z.record(z.string(), HookConfigSchema)` and the generated schema exposes dynamic entry keys, so hook IDs like `qmd-embedder` are accepted in [src/config/zod-schema.hooks.ts](src/config/zod-schema.hooks.ts:96) and [src/config/schema.base.generated.ts](src/config/schema.base.generated.ts:20488).

If this is still happening through a helper or wrapper that adds `path=...` behavior, please open a fresh issue against that surface with the exact command path and the full request payload/response.
