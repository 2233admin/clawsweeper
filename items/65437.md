## Summary

Keep #65437 open. Current main still has the Slack room/group early-drop path described in the issue: Slack builds the control-command gate only from `channels.slack.allowFrom`/pairing-store owner auth and `channels.slack.channels.<id>.users`, then returns `null` for unauthorized roomish control commands before reply-stage `/elevated` authorization can evaluate `tools.elevated.allowFrom.slack`. The issue is recent, author is not protected, no protected labels are present, and the related Discord/tiered-auth items do not supersede this Slack-specific bug.

## Best Possible Solution

Keep this open and fix the Slack room/group `/elevated` path narrowly. The best implementation is to let Slack evaluate elevated authorization for `/elevated`-bearing messages before the generic control-command early drop, or add an elevated-specific authorizer to the inbound command gate, without widening authorization for unrelated control commands. Add regression coverage for a Slack channel/group message from a sender listed only in `tools.elevated.allowFrom.slack`, with no `channels.slack.allowFrom` and no `channels.slack.channels.<id>.users`, and update the public elevated/Slack docs to describe the resulting behavior. Registering `elevated.allowFromFallback` alone would not fix the reported room/group symptom unless the early inbound drop is also addressed.
