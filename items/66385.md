## Summary

Keep this PR open. Current main still does not implement `skills.allowExternalSkillsIn`: the skill loader has no external allowlist, the public config type and strict Zod schema omit the field, docs still describe strict realpath containment, and tests still assert escaped symlinked skills are rejected. The PR remains a focused implementation candidate, but it changes the skills trust boundary and should receive maintainer security/product review before merge or explicit rejection.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants trusted external symlinked skill roots, land a focused core skills/config change with safe realpath containment, generated config/docs artifacts, and tests covering default rejection, allowed external containment, prefix-boundary bypasses, canonical prefix resolution, empty entries, and target-workspace sync behavior. If maintainers do not want this trust expansion, close it only after explicit product/security judgment.

## Risks / Open Questions

- Loaded skills are exposed to prompt/runtime skill surfaces and may be copied into target workspaces by `syncSkillsToWorkspace`, so maintainer review should treat this as command/tool execution and data access surface.
- Because this is a public config key, `types.skills.ts`, `zod-schema.ts`, generated schema/baseline artifacts, docs, and tests must stay aligned before merge.
