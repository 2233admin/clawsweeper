## Summary

Keep this PR open. Current main still does not hydrate plugin-handled `send` media/path/file hints into shared attachment fields, and the PR is not superseded by the related Telegram/buffer-only work. The latest PR direction addresses review feedback with an opt-in contract, but that public plugin action contract and media preloading boundary need maintainer review before merge or close.

## Best Possible Solution

Keep this PR open for maintainer review or contributor refinement. The best path is a guarded public plugin action contract for send-media preloading, wired from `runMessageAction` into `hydrateAttachmentParamsForAction`, with tests proving both default URL/path forwarding and opt-in hydrated `buffer`/`contentType`/`filename`. Coordinate with #41779, #42674, and #57335, but do not close this PR as implemented or duplicate while current main lacks the plugin send media hydration path.

## Risks / Open Questions

- The PR expands a public plugin channel contract; maintainership should decide the exact `preloadSendMedia`/hydration semantics and require matching SDK docs/API checks before merge.
- Opt-in core preloading changes where remote/local media are fetched or read, so the implementation must keep using the existing sandbox and outbound media-access policy.
- Existing plugin send handlers such as Google Chat and Matrix already load URL/path media themselves; accidental broad opt-in could double-fetch media or break short-lived signed URLs.
