## Summary

Keep open. Current `main` still has hardcoded config-recovery notices with no `gateway.systemEvents.recoveryNotice` or equivalent policy, and the related items in the provided context do not supersede this request: #71289 narrowed whole-file rollback triggers, #49579 is a WhatsApp-specific banner filter PR, and #39307 asks for more visible context-reset signaling rather than suppression for customer-facing bots.

## Best Possible Solution

Keep the issue open for a core Gateway/config change. Implement a documented `gateway.systemEvents.recoveryNotice` policy, or an equivalent maintainer-approved name, in the config schema/types/docs and apply it consistently to config auto-recovery restart sentinels, startup/reload `enqueueConfigRecoveryNotice`, prompt-facing system events, and outbound restart notices while leaving actual recovery and restart continuations unaffected. Add focused tests for default/user, log-only, and off behavior.

## Risks / Open Questions

- A fix needs to cover both recovery-notice paths: the restart-sentinel `config-auto-recovery` message and the newer `Config recovery warning` system event path. Suppressing only one would leave the reported customer-facing leak partially intact.
- Changing defaults could hide an operator safety signal, so the default should likely preserve current behavior while allowing explicit log-only or off modes for customer-facing deployments.
