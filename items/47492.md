## Summary

Keep open. Current `main` has exec approvals and basic input provenance tags, but it does not implement a gateway-enforced sensitive-target policy that requires a verified human instruction lineage before running side effects.

## Evidence

- **No matching feature surface on main:** A repo-wide search on current `main` found no implementation or docs for `sensitiveTargets`, `requireInstructionProvenance`, `verified-human`, or `instruction provenance`.
  - command: `rg -n "sensitiveTargets|requireInstructionProvenance|verified-human|instruction provenance" src docs packages extensions VISION.md`
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Exec approval model only exposes ask/allowlist controls:** `ExecApprovalRequestPayload` and `ExecApprovalsDefaults` model command metadata plus `security`, `ask`, and allowlist-related settings. There is no field for verified-human provenance requirements or sensitive-target policy.
  - file: [src/infra/exec-approvals.ts:84](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/infra/exec-approvals.ts#L84)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Gateway approval handler does not enforce provenance policy:** `exec.approval.request` validates command/node/session fields and creates the approval record directly from command, host, session, and turn-source metadata. No branch checks for verified-human ancestry or target classification before registration.
  - file: [src/gateway/server-methods/exec-approval.ts:109](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/gateway/server-methods/exec-approval.ts#L109)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Existing provenance tags are too coarse for the request:** Input provenance currently supports only `external_user`, `inter_session`, and `internal_system`, with optional source session/channel/tool metadata. There is no stronger 'verified human requested this sensitive action' provenance primitive.
  - file: [src/sessions/input-provenance.ts:4](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/sessions/input-provenance.ts#L4)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Public config/docs still describe exec approvals as security + ask + allowlist:** The generated config schema exposes `tools.exec.security`, `tools.exec.ask`, and related allowlist controls only; the user docs for exec approvals show the same model. No public config surface exists for sensitive target patterns or verified-human provenance gating.
  - file: [src/config/schema.base.generated.ts:17453](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/config/schema.base.generated.ts#L17453)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Turn-source metadata is only used for routing approvals:** When an approval is pending, the shared handler checks `turnSourceChannel`/`turnSourceAccountId` only to decide whether there is a route to deliver the approval prompt. That is not the requested gateway policy proving a verified human instruction chain.
  - file: [src/gateway/server-methods/approval-shared.ts:165](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/gateway/server-methods/approval-shared.ts#L165)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)

## Risks / Open Questions

- Existing `ask: "always"` and allowlist settings can require human approval broadly, but they are not equivalent to the requested sensitive-target, provenance-aware guard.
- The request overlaps with broader taint/external-content hardening work, so future implementation may need coordination with ongoing security policy design rather than a narrow triage close.
