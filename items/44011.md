## Summary

Current `main` does not implement the typed message-hook `agentId` exposure requested by PR #44011, and outbound typed hook contexts still omit the session-scoped fields this PR is trying to thread through. The PR remains a live, relevant change rather than a cleanup-close candidate.

## Evidence

- **Typed message-hook context still lacks `agentId` on main:** `PluginHookMessageContext` in `src/plugins/hook-message.types.ts` includes `channelId`, `accountId`, `conversationId`, `sessionKey`, `runId`, `messageId`, `senderId`, and trace fields, but no `agentId` field.
  - file: [src/plugins/hook-message.types.ts:4](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/plugins/hook-message.types.ts#L4)

  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Mapper does not forward `agentId` into typed hook context:** `toPluginMessageContext` builds the exported message-hook context from `channelId`, `accountId`, `conversationId`, then conditionally adds `sessionKey`, `runId`, `messageId`, `senderId`, trace, and `callDepth`; there is no `agentId` mapping.
  - file: [src/hooks/message-hook-mappers.ts:220](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/hooks/message-hook-mappers.ts#L220)
