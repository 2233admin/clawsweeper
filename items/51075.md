## Summary

Keep open. Current `main` has added structured diagnostics/OTEL observability, but the gateway file logger itself still does not implement the issue’s requested schema changes: log lines are still emitted from raw `tslog` objects, subsystem loggers only auto-bind `subsystem`, and consumers still have to reconstruct `message` from numeric `"0"`, `"1"`, ... keys.

## Evidence

- **Gateway file logger writes raw tslog payloads without adding requested fields:** `buildLogger()` creates a hidden `tslog` logger and the file transport just serializes `logObj` with a computed `time`. There is no OpenClaw-side injection of `agent_id`, `session_id`, `channel`, or flattened `message` at write time.
  - file: [src/logging/logger.ts:153](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/logging/logger.ts#L153)
  - command: `nl -ba src/logging/logger.ts | sed -n '130,210p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Subsystem logger only auto-binds subsystem and passes message as a separate tslog arg:** `createSubsystemLogger()` lazily creates `getChildLogger({ subsystem: resolvedSubsystem })`. When metadata exists, `logToFile()` calls `method(meta, message)`, which preserves tslog’s positional-argument shape instead of emitting a flat top-level `message` field.
  - file: [src/logging/subsystem.ts:302](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/logging/subsystem.ts#L302)
  - command: `nl -ba src/logging/subsystem.ts | sed -n '295,380p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Consumers still reconstruct message from numbered keys:** `parseLogLine()` builds `message` by scanning numeric keys (`"0"`, `"1"`, ...) and joining them, which is direct evidence that the raw log-file schema still uses split positional fields. The CLI normalizes this only after reading the file.
  - file: [src/logging/parse-log-line.ts:12](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/logging/parse-log-line.ts#L12)
  - command: `nl -ba src/logging/parse-log-line.ts | sed -n '1,120p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Main now has a separate diagnostics observability path, but docs say it does not replace logs:** The diagnostics event types include structured `sessionId`, `sessionKey`, and `channel` fields, and the changelog shows recent work adding more diagnostic events. But `docs/logging.md` explicitly says diagnostics do not replace logs, so these additions do not fully implement the issue’s requested gateway log-line schema.
  - file: [src/infra/diagnostic-events.ts:12](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/infra/diagnostic-events.ts#L12)

  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)

## Risks / Open Questions

- `tslog` may still populate some internal metadata such as hostname, but the current repo code does not configure or test the requested `os.hostname()` behavior, so I could not treat that part as implemented.
- Some individual call sites may manually attach `channel` or `sessionId` metadata, but I found no automatic logger-context injection that would satisfy the issue’s 'every log line' requirement.
