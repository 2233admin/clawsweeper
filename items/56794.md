## Summary

Keep open. The March root-cause reference to `buildBundledChannelMaps()` is stale on current `main`, but the underlying bundled-vs-external channel collision is not clearly fixed. Current config validation still resolves bundled runtime schemas by channel id without consulting plugin disablement, and doctor preview warnings still flag any disabled plugin that declares the configured channel without checking whether another effective plugin owns that same channel.

## Evidence

- **Current channel config validation still keys bundled runtime parsing by channel id:** `ChannelsSchema` is top-level `.passthrough()`, but it still calls `normalizeBundledChannelConfigs()`, which iterates every configured channel, resolves a bundled runtime schema with `getDirectChannelRuntimeSchema(channelId)`, and applies it with `safeParse` without checking `plugins.entries.<id>.enabled` or external ownership.
  - file: [src/config/zod-schema.providers.ts:27](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/config/zod-schema.providers.ts#L27)
  - command: `nl -ba src/config/zod-schema.providers.ts | sed -n '1,170p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Bundled Feishu schema remains strict:** The bundled Feishu config schema uses `.strict()`, so if the bundled runtime schema is selected for `channels.feishu`, extra fields from an external replacement plugin can still be rejected as unknown properties.
  - file: [extensions/feishu/src/config-schema.ts:208](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/feishu/src/config-schema.ts#L208)

- **Doctor blocker scan still warns on any disabled channel-declaring plugin:** `scanConfiguredChannelPluginBlockers()` loads the manifest registry, walks every plugin with `plugin.channels.length > 0`, and emits a hit whenever that plugin is disabled and its channel matches configured channel ids. It never checks whether some other plugin is effective for the same channel.
  - file: [src/commands/doctor/shared/channel-plugin-blockers.ts:35](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/commands/doctor/shared/channel-plugin-blockers.ts#L35)
  - command: `nl -ba src/commands/doctor/shared/channel-plugin-blockers.ts | sed -n '1,170p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Core already has a multi-owner-aware channel presence policy, but doctor does not use it here:** `resolveConfiguredChannelPresencePolicy()` gathers all records declaring a channel and marks the channel effective when any owner evaluates effective. That is the logic needed for bundled-disabled/external-enabled takeovers, but the doctor blocker path above bypasses it.
  - file: [src/plugins/channel-presence-policy.ts:343](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/plugins/channel-presence-policy.ts#L343)
