## Summary

Keep open. Current `main` still handles Discord guild slash commands and inbound guild messages using only the local instance's `discordConfig.guilds`/channel allowlist, so a second instance on the same bot token can still reject a command with `This channel is not allowed.` before the intended instance handles it. The linked PR #55652 appears to be the unmerged fix: its patch adds a new shared `instance-claims` registry plus ownership checks in the Discord monitor paths, and that machinery does not exist on current `main`.

## Evidence

- **Slash-command deny path is still local-config only:** `dispatchDiscordCommandInteraction` resolves `guildInfo` from `discordConfig?.guilds`, then resolves `channelConfig`, then directly replies `This channel is not allowed.` when `channelConfig?.allowed === false` or policy denies. There is no prior cross-instance ownership check in this path.
  - file: [extensions/discord/src/monitor/native-command.ts:837](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/discord/src/monitor/native-command.ts#L837)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Concrete wrong-instance rejection still present:** Current `main` still contains the exact rejection branch the issue reports: `await respond("This channel is not allowed.");` in the guild slash-command path.
  - file: [extensions/discord/src/monitor/native-command.ts:887](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/discord/src/monitor/native-command.ts#L887)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Inbound guild-message path is also local allowlist only:** `preflightDiscordMessage` resolves `guildInfo` from `params.guildEntries`, then drops when the guild/channel is not in that local allowlist. There is no shared ownership/claim lookup before these deny branches.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:731](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/discord/src/monitor/message-handler.preflight.ts#L731)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **No claim-registry code exists on current main:** Searching `extensions/discord/src/monitor` for `instance-claims`, `resolveDiscordClaimOwnership`, `refreshDiscordClaims`, or `DISCORD_CLAIMS_PATH` returned no matches on current `main`, so the proposed shared-instance ownership mechanism is not present.
  - command: `rg --files extensions/discord/src/monitor | rg 'instance-claims' && rg -n "resolveDiscordClaimOwnership|refreshDiscordClaims|DISCORD_CLAIMS_PATH" extensions/discord/src/monitor/provider.ts extensions/discord/src/monitor/message-handler.preflight.ts extensions/discord/src/monitor/native-command.ts`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Linked PR patch is adding the missing fix, not already shipped:** The public patch for PR #55652 adds new files `extensions/discord/src/monitor/instance-claims.ts` and `.test.ts`, wires `resolveDiscordClaimOwnership(...)` into `native-command.ts` and `message-handler.preflight.ts`, and registers periodic claim refreshes in `provider.ts`. That strongly indicates the fix is still pending rather than already implemented on `main`.
