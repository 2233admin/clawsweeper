## Summary

Keep #70420 open. Current main still separates channel/provider liveness from stored conversation sessions: `sessions.list`/`sessions_list` is backed by persisted session stores, while Discord reconnect/health state is represented in runtime channel status. The recent #38596/#40413 fix improves repeated Discord health-monitor recovery loops, but it does not resolve the session-list liveness contract or add explicit docs warning operators not to use `sessions_list` as Discord socket health.

## Best Possible Solution

Keep this open for the channel-health owner to settle the contract. The conservative path is a docs/status clarification: session stores and `sessions_list` are conversation/history discovery surfaces, while Discord/channel connection health should use `openclaw status`, `openclaw health`, `channels.status`, or readiness/status surfaces. A larger behavior change would intentionally materialize a synthetic channel-health/session row on provider start/reconnect, with tests defining how it behaves for quiet Discord, Slack, Matrix, and Telegram workspaces.
