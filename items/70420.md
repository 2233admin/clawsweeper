## Summary

Keep open. Current `main` still leaves `sessions_list` backed by persisted session-store rows while Discord channel restarts and readiness are driven by runtime channel status, so the reported empty-after-restart shape remains possible and there is no main-side fix that re-materializes channel sessions on reconnect or fully resolves the contract gap called out in the issue.

## Evidence

- **`sessions_list` resolves from the session store, not channel runtime:** `resolveSessionKeyFromResolveParams` and the session-list path both delegate to `listSessionsFromStore`, which iterates persisted session entries from the gateway session store. That means `sessions_list` only sees materialized session rows, not live channel socket state.
  - file: [src/gateway/sessions-resolve.ts:65](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/gateway/sessions-resolve.ts#L65)
  - command: `nl -ba src/gateway/sessions-resolve.ts | sed -n '1,260p' ; nl -ba src/gateway/session-utils.ts | sed -n '1489,1515p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Health-monitor restart only restarts the channel runtime:** The health monitor handles an unhealthy channel by `stopChannel` then `startChannel`; there is no session-store materialization in that restart path.
  - file: [src/gateway/channel-health-monitor.ts:158](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/gateway/channel-health-monitor.ts#L158)
