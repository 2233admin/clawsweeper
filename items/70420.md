## Summary

Keep #70420 open. Current main still keeps `sessions.list`/`sessions_list` tied to persisted session-store rows, while Discord health-monitor reconnects only restart the channel runtime and Discord only writes session rows from inbound message processing. Recent transport-liveness fixes narrow false stale-socket restarts, but they do not recreate Discord session rows after a reconnect or explicitly document `sessions_list` as non-authoritative for Discord/channel liveness.

## Best Possible Solution

Keep the issue open for channel-health owner judgment. The conservative path is to document the contract clearly in `docs/gateway/health.md`, `docs/concepts/session-tool.md`, and any heartbeat/status guidance: use channel status/readiness/health probes for Discord connectivity and treat `sessions_list` as persisted session history only. If maintainers want `sessions_list` to be authoritative, add a generic, provider-neutral session materialization seam on successful channel start/reconnect and cover Discord plus at least one non-Discord channel in tests.
