## Summary

Keep #72262 open. Current main still lacks the requested WhatsApp `status=408 Request Time-out` troubleshooting runbook and channel troubleshooting table row, while the runtime/test surface does preserve that exact operator-facing error signature. The later issue discussion also corrected the earlier QEEK-AI claim and confirms the runbook was drafted but not integrated.

## Best Possible Solution

Keep the issue open and add the requested docs to current main: a dedicated `status=408 Request Time-out` accordion in `docs/channels/whatsapp.md` plus a concise row in `docs/channels/troubleshooting.md`. The runbook should explain the Baileys/WhatsApp Web socket meaning of 408, walk through `openclaw channels status --probe`, logs, doctor, and gateway status, distinguish transient reconnects from stale auth, DNS/proxy, host/runtime, and dependency problems, define healthy WhatsApp credential state, and document a backup-aware logout/login recovery path.
