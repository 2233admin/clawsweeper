## Summary

Keep open. Current main still has a 30s per-RPC gateway client timeout, only retries `chat.history` during startup when the server returns retryable `UNAVAILABLE`, and still logs `history failed: ...` for other errors. That means the reported first-launch timeout path is not clearly fixed on `main` or in release `v2026.4.23`.

## Evidence

- **Client still throws the reported timeout shape:** `GatewayClient` defaults `requestTimeoutMs` to 30_000 ms and rejects pending RPCs with `gateway request timeout for ${method}`. For `chat.history`, that is the exact error family described in the issue.
  - file: [src/gateway/client.ts:211](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/client.ts#L211)
  - command: `nl -ba src/gateway/client.ts | sed -n '200,260p;920,955p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Startup gate is implemented as retryable UNAVAILABLE, not timeout:** Gateway startup intentionally gates `chat.history` behind `STARTUP_UNAVAILABLE_GATEWAY_METHODS`, and request handling returns a retryable `UNAVAILABLE` error with `retryAfterMs: 500` when that gate is hit. If users are instead seeing `gateway request timeout`, that is a different path than the already-handled startup-unavailable case.
  - file: [src/gateway/server-methods.ts:117](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/server-methods.ts#L117)

- **Startup-gated methods stay blocked until sidecars finish:** `chat.history` remains unavailable until post-attach sidecars complete, then the startup gate is removed.
