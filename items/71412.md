## Summary

Current main still matches the reported bug. The stop-timeout path records the timeout as `running: true` and returns before clearing `store.aborts` or `store.tasks`; `startChannelInternal` still silently no-ops when that stale task entry remains; and the existing regression test still asserts that a second start does not occur after a stop timeout. The related PR #71456 was closed unmerged in the provided GitHub context, so this issue should remain open until that fix, or an equivalent patch, lands.

## Best Possible Solution

Keep #71412 open and land PR #71456 or an equivalent gateway lifecycle patch that clears the stale `store.aborts` and `store.tasks` entries on stop timeout, marks the account stopped rather than running, and updates the regression test to prove a subsequent `startChannel` registers a fresh task. Treat the `openclaw doctor` liveness expectation as a separate follow-up unless maintainers want it bundled.
