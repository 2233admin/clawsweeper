## Summary

Keep open. Current `main` does not implement the requested multi-account Discord skill gating, and the PR's proposed `anyConfig: ["channels.discord.token", "channels.discord.accounts"]` check is not safe because the existing config truthiness logic treats any object as satisfied, which would report Discord as ready even for an empty or tokenless `accounts` map.

## Evidence

- **Current bundled Discord skill still requires only the single-account token path:** The bundled Discord skill on `main` declares `requires.config = ["channels.discord.token"]`; there is no multi-account alternative in the shipped skill metadata.
  - file: [skills/discord/SKILL.md:4](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/skills/discord/SKILL.md#L4)

  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Current requirement parsing/evaluation has no `anyConfig` support:** On `main`, frontmatter parsing, requirement metadata, and runtime requirement evaluation only know about `bins`, `anyBins`, `env`, and `config`. The proposed feature is not already implemented.
  - file: [src/shared/frontmatter.ts:51](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/shared/frontmatter.ts#L51)

  - file: [src/shared/requirements.ts:1](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/shared/requirements.ts#L1)

Using `channels.discord.accounts` as a satisfaction check would therefore pass for `{}` or for account objects that exist without a token.
  - file: [src/shared/config-eval.ts:4](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/shared/config-eval.ts#L4)

That means the coarse `accounts` object check is insufficient to prove a usable Discord account is configured.
