## Summary

Keep open. Current main has addressed part of the report by stripping raw JSON/request_id-shaped payloads and by having a pre-reply transient HTTP retry path, but the Feishu-visible path can still surface provider-shaped `LLM error server_error: ...` text instead of the generic channel-safe fallback requested in #45050.

## Evidence

- **Shared formatter still emits provider-shaped server_error text:** `formatRawAssistantErrorForUi` parses API error payloads, drops the raw JSON/request_id from the displayed string, but returns `${prefix}${type}: ${message}`. For the reported Codex payload shape that becomes `LLM error server_error: An error occurred while processing your request.`, not a generic Feishu-safe fallback.
  - file: [src/shared/assistant-error-format.ts:206](https://github.com/openclaw/openclaw/blob/1531123d3547f9ff8ee8094590fc773305f018f2/src/shared/assistant-error-format.ts#L206)
  - command: `nl -ba src/shared/assistant-error-format.ts | sed -n '1,215p'`
  - sha: [1531123d3547](https://github.com/openclaw/openclaw/commit/1531123d3547f9ff8ee8094590fc773305f018f2)
- **Payload builder still appends formatted assistant error payloads:** When `lastAssistant.stopReason === "error"`, `buildEmbeddedRunPayloads` calls `formatAssistantErrorText` and pushes that `errorText` as an `isError` reply item. It does suppress raw/fingerprinted API payload text from assistant output, which fixes the raw JSON/request_id leak part, but it does not replace the formatted server_error with a generic message.
