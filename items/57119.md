## Summary

Keep open. Current `main` still treats `skills.entries.<key>.enabled=false` as a load-time visibility filter, not as filesystem or exec enforcement. Disabled skills are omitted from the skill registry/prompt and from sandbox skill sync, but host-side `read`/`write`/`edit` remain unrestricted unless `tools.fs.workspaceOnly=true`, and exec can still target skill files/scripts by path.

## Risks / Open Questions

- Closing this would hide a real core security-hardening gap for non-sandboxed runs and for sandbox configurations with `workspaceAccess: "rw"`.
- Current docs overstate what `skills.entries.<key>.enabled=false` does, so treating the report as obsolete would be misleading.
