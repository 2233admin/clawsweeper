## Summary

Current main still models `plugins.slots.*` as bare string plugin IDs across the public config type, runtime schema, generated schema, normalization, activation, validation, install/write, and docs surfaces. The requested `{ owner, claimed_at, claimed_by_version }` shape and slot takeover invariant remain a real core plugin/config contract RFC, not something already implemented or safely moved to ClawHub.

## Best Possible Solution

Keep the RFC open for maintainer design review. A concrete implementation path would add a backwards-compatible slot ownership union, normalize legacy string slots to a canonical owner view on read, update install/register/write paths to persist ownership metadata, add explicit conflict and force/allow-override semantics for slot takeover, and cover validation, generated schema/help, docs, activation, install/update/uninstall, defaults, and `"none"` behavior with focused tests.

## Risks / Open Questions

- This is a public config and plugin contract change; maintainers still need to choose the schema shape, migration order, provenance fields, and takeover/force semantics.
- Any implementation must preserve legacy bare-string configs, `"none"` disable behavior, default slots such as `memory-core` and `legacy`, generated schema/help alignment, install/update/uninstall behavior, and external plugin compatibility.
- The existing `plugins install --force` path is about replacing install targets, not authorizing slot takeover; reusing or extending that flag needs deliberate product/security review.
