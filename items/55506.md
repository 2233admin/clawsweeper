## Summary

Current `main` still has a plausible stale-bootstrap bug for reused chat sessions. Bootstrap workspace files are snapshotted per `sessionKey`, reused on later turns, and only cleared on session rollover/reset. Gateway `SIGUSR1` restarts are explicitly in-process, so that cache can survive a restart. Separately, `/context` prefers the last persisted run report, so `systemPromptReport.injectedWorkspaceFiles` can also look stale even before a fresh rebuild.

## Evidence

- **Per-session bootstrap snapshot cache:** `getOrLoadBootstrapFiles` memoizes workspace bootstrap files by `sessionKey` and returns the cached array on later calls instead of re-reading disk.
  - file: [src/agents/bootstrap-cache.ts:3](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/bootstrap-cache.ts#L3)

- **Normal runs use the session cache:** `resolveBootstrapFilesForRun` goes through `getOrLoadBootstrapFiles(...)` whenever a run has a `sessionKey`, which covers persistent channel sessions like Telegram DMs.
  - file: [src/agents/bootstrap-files.ts:229](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/bootstrap-files.ts#L229)

- **Cache invalidation only on rollover/reset:** The cache is cleared via `clearBootstrapSnapshotOnSessionRollover`, and the session path calls that only when there is a previous session id from a reset/expiry path. There is no invalidation on workspace file change.

  - file: [src/cli/gateway-cli/run-loop.ts:268](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/cli/gateway-cli/run-loop.ts#L268)

## Risks / Open Questions

- I did not run a live reproduction because this review is read-only and the task forbids mutating the checkout or creating test artifacts.
- A full external process restart would clear the in-memory map, but the code still leaves reused-session stale context plausible through in-process restarts and same-session caching.
- `/context` staleness alone does not prove the prompt was stale on the very latest turn, but the session bootstrap cache makes the underlying report materially credible.
