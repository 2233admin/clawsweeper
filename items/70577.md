## Summary

Keep open. Current `main` still has a real CLI/tool config-path divergence for `memory_search`: the CLI reloads fresh config before each command, while the tool binds the provided config snapshot at tool creation and reuses that snapshot for manager resolution. That matches the report closely enough that conservative cleanup should not auto-close it.

## Evidence

- **Memory tool binds a config snapshot once:** `createMemoryTool()` resolves `ctx` from `options.config` and returns `execute: params.execute(ctx)`, so the tool uses the captured `cfg`/`agentId` from creation time rather than reloading config per invocation.
  - file: [extensions/memory-core/src/tools.shared.ts:47](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/memory-core/src/tools.shared.ts#L47)
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **memory_search executes against captured cfg:** `createMemorySearchTool()` calls `getMemoryManagerContext({ cfg, agentId })` inside the executor, but `cfg` there is the closure-bound value from tool construction, not a live runtime reload.
  - file: [extensions/memory-core/src/tools.ts:183](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/memory-core/src/tools.ts#L183)
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **CLI reloads fresh config before memory search:** `loadMemoryCommandConfig()` calls `loadConfig()` and `resolveCommandSecretRefsViaGateway(...)`, and `runMemorySearch()` uses that freshly loaded `cfg` for `withMemoryManagerForAgent(...)`. This is a different path from the tool host.
  - file: [extensions/memory-core/src/cli.runtime.ts:93](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/memory-core/src/cli.runtime.ts#L93)
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Plugin tool plumbing already carries a live runtime config, but memory_search ignores it:** Plugin tool context includes `runtimeConfig`, and tool resolution supplies it from the active runtime snapshot. The memory tool factory only accepts `config`, not `runtimeConfig`, so it does not participate in that live-config path.
  - file: [src/plugins/tool-types.ts:8](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/plugins/tool-types.ts#L8)
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Comparable feature has explicit live-config handling and tests:** `active-memory` explicitly reloads runtime config via `api.runtime.config.loadConfig()` and has regression tests for live config disable/removal, which highlights that this repo already treats live-config drift as a real class of bug.
  - file: [extensions/active-memory/index.ts:1911](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/active-memory/index.ts#L1911)
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
