## Summary

Keep this issue open. Current main still compacts and retries by rebuilding the attempt from the original run prompt, with retry caps and replay-safety metadata but without the durable current-turn identity/original-message ledger requested here. The cron trigger path still passes a fixed cron prompt into the embedded runner, so the reported surface remains relevant. Related #59618 and #63216 are broader open compaction-resume reports, while closed #66128 only addressed one trigger path.

## Best Possible Solution

Keep this issue open and fix the embedded runner/subscriber compaction retry contract. Preserve a stable current-turn identity across compaction, carry retry attempt and original inbound message metadata into the resumed attempt, prevent cron/system prompt reconstruction from being treated as a fresh user turn, gate deterministic tool cascades from being re-executed after compaction, and add regression coverage for a cron/tool-failure cascade that compacts without re-injecting the original prompt as new input. Keep #59618 and #63216 cross-linked as related compaction-resume work; treat #66128 as only a trigger-path mitigation.
