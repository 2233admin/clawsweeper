## Summary

Keep this issue open. Current main still does not accept or use `plugins.entries.exa.config.webSearch.baseUrl`, and open PR #54939 is an explicit unmerged implementation candidate with closing syntax for this issue.

## Best Possible Solution

Keep this issue open and review PR #54939 as the implementation candidate. The fix belongs in the Exa plugin: add `webSearch.baseUrl` to the manifest schema and UI hints, plumb it through the Exa runtime via the trusted web-search endpoint path, include the resolved endpoint in cache-key behavior, update the Exa and web-search docs, and add Exa plugin/config tests proving the field is accepted and used. If maintainers intentionally do not want custom Exa endpoints, document that provider capability difference and return provider-specific validation guidance instead.

## Risks / Open Questions

- Adding configurable Exa endpoints is security-sensitive outbound behavior; the implementation should preserve guarded fetch behavior and avoid sending Exa API keys to unintended URLs without deliberate validation.
- If `baseUrl` is added, the resolved endpoint should be included in the search cache key so results from different proxies or gateways cannot share cached payloads.
