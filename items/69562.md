## Summary

Keep PR #69562 open. Current main still validates cron announce channels only, while `delivery.accountId` and `delivery.failureDestination.accountId` are accepted as non-empty strings and can flow into runtime delivery. The supplied PR context shows a focused fix adding account membership validation plus follow-up commits for failure destinations, default accounts, normalization, and inferred-channel edge cases. Related reports are adjacent cron delivery work, not a canonical duplicate of this account-id validation gap.

## Best Possible Solution

Keep #69562 open and review it as a focused gateway cron hardening PR. The best fix is to validate explicit cron announce account ids for both primary delivery and failure destinations against the resolved configured channel when that channel is concrete, include the plugin default account and normalized ids in the allowed set, skip checks for dynamic/inferred `last` routes where runtime channel is not known, and keep focused regression coverage for cron.add and cron.update.
