## Summary

Keep PR #69562 open. Current main still validates cron announce channels only; it accepts `delivery.accountId` and `delivery.failureDestination.accountId` as cron fields and later uses explicit account IDs at runtime, but the `cron.add`/`cron.update` gateway validation path does not check that those account IDs are configured. The PR is focused and its review thread shows follow-up commits addressing concrete edge cases, so it is not obsolete cleanup.

## Best Possible Solution

Keep the PR open for maintainer review. The best implementation path is a focused gateway create/update validation in `src/gateway/server-methods/cron.ts` that checks explicit primary and announce-mode failure-destination account IDs against the resolved channel plugin/configured accounts, includes default accounts, normalizes IDs consistently, and skips validation when the runtime channel is intentionally inferred from `last`. Keep regression coverage in `src/gateway/server.cron.test.ts` for both `cron.add` and `cron.update`.
