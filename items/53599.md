## Summary

Current main still does not provide the reported one-click cross-machine Chrome extension relay workflow. The browser plugin intentionally removed the legacy relay, rejects or migrates `driver: "extension"`, and documents replacements based on host-local Chrome MCP, raw CDP, or a browser-capable node host rather than a browser-only extension install on a different machine. The linked ClawHosters relay and confirming comments describe a distinct managed-hosting regression, not a stale duplicate or something already implemented.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. The best path is either to restore an explicitly opt-in, hardened relay inside the browser plugin with schema, docs, tests, and security review, or to add a narrow public browser transport/profile-driver API so the linked ClawHosters relay or a ClawHub plugin can integrate with the existing browser tool cleanly.

## Risks / Open Questions

- Restoring a WebSocket browser relay would reintroduce remote browser-control risk and needs explicit security review for authentication, origin validation, token handling, public exposure, and operator consent.
- The linked ClawHosters relay is useful prior art and a ClawHub plugin may be viable long term, but current main does not expose a clean browser profile-driver/transport API for integrating it into the existing browser tool without private internals or replacing the whole browser plugin.
- The documented Node Host proxy, `existing-session`, and remote CDP paths cover some remote-browser cases, but they do not satisfy the reported no-Node, no-special-Chrome-flags, Chrome-extension-only managed-hosting workflow.
