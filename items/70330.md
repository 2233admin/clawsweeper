## Summary

Keep open. Current main still leaves the reported WebChat session-continuity gap: Control UI WebChat connects with admin scope, the visible New Session action and local `/new`/`/reset` commands still flow through generic `chat.send`, and the gateway still parses reset slash text inside the agent path, archives the prior transcript, rotates the session id, and can strip `/reset some message` down to the post-reset message without a first-class reset provenance field in `chat.send` or the session reset RPC.

## Best Possible Solution

Keep this in OpenClaw core. Replace WebChat New Session and local WebChat `/new`/`/reset` handling with an explicit session-control RPC or an `allowSessionControl` flow carrying trigger metadata; persist an auditable reset/rotation event with old/new session ids and archive path; clear stale reset/draft/idempotency state across reconnect; surface reset archives and checkpoint recovery in Control UI; and add regression coverage for gateway restart/reconnect followed by a normal visible send so `agent:main:main` cannot silently rotate without provenance.
