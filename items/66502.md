## Summary

Keep open. Current `main` includes related hardening for symlinked dist entrypoints and doctor-based service repair, but it does not appear to cover the PR's specific case where the CLI is launched from a versioned pnpm global wrapper path like `.pnpm/openclaw@.../node_modules/openclaw/openclaw.mjs`. In that path shape, `resolveGatewayProgramArguments` still builds only version-pinned dist candidates, so the reported gateway service path drift across pnpm upgrades remains plausible on current `main`.

## Evidence

- **Current resolver still only derives dist candidates from the current path, its parent, and `.bin` shims:** `resolveCliEntrypointPathForService` falls back to `buildDistCandidates(resolvedPath, normalized)`. `buildDistCandidates` adds parent/base `dist/*` candidates and `.bin`-based package roots, but there is no logic to strip a `.pnpm/<pkg>@<version>/` segment and re-root to stable `node_modules/openclaw/...`.
