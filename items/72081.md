## Summary

Keep open. Current main intentionally avoids printing `#token=` in `openclaw dashboard` output, but the reporter's attached log shows the WSL/headless path where browser launch and clipboard both failed, leaving only a clean URL and SSH hint. That is still an actionable UX/auth recovery gap, and the fix should not simply reintroduce bearer tokens into default logs.

## Best Possible Solution

Keep the issue open and implement a safe headless/WSL recovery path: preserve clean default logs, but when browser launch and clipboard copy both fail, provide explicit remediation or an opt-in tokenized output mode with a clear warning, plus tests for the no-GUI/no-clipboard path. Also update the remote dashboard docs so they match the current security behavior.

## Evidence

That matches the reported WSL2/headless failure mode: the tokenized URL exists internally, but the user has no visible/copyable tokenized URL when both clipboard and browser opening fail.
