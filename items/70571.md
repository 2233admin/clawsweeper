## Summary

Keep open. Current `main` still enforces `safeBins` by executable name plus per-bin profile validation, so the underlying gap is not already implemented. The PR's compound-match path would weaken that safety model by bypassing `validateSafeBinArgv` for profiled binaries and by lowercasing full argv prefixes, and the existing PR discussion already called out those same P1 concerns. The PR also bundles a large unrelated README rewrite, which further argues against conservative maintainer cleanup here.

## Evidence

- **Current main requires profile validation for every safe-bin allow:** `isSafeBinUsage` on `main` only accepts a safe bin when the executable name is allowlisted, the resolved path is trusted, and `validateSafeBinArgv(argv.slice(1), profile, { binName: execName })` succeeds. There is no compound-command special case on `main`.

## Risks / Open Questions

- Merging the PR as-is would let compound `safeBins` entries bypass per-tool denied-flag and positional validation on profiled binaries.
- Lowercasing full argv prefixes for compound matching can conflate distinct short flags, widening what a `safeBins` entry approves.
- The bundled README rewrite creates unrelated review surface in a security-sensitive PR and increases the chance of accidental maintainer cleanup mistakes.
