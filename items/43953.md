## Summary

Keep open. Current `main` still drops native Feishu form fields before they reach the handler, and the PR as posted does not patch that production ingress path. The discussion also surfaced a plausible command-routing regression for select/form callbacks that still carry `action.value.command` or `action.value.text`.

## Evidence

- **Production ingress strips extra action fields:** `parseFeishuCardActionEventPayload` rebuilds `event.action` with only `value` and `tag`, so native Feishu fields like `form_value`, `input_value`, `name`, `option`, and `options` are discarded before `handleFeishuCardAction` runs.
  - file: [extensions/feishu/src/monitor.account.ts:186](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/monitor.account.ts#L186)
  - command: `nl -ba extensions/feishu/src/monitor.account.ts | sed -n '170,230p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Actual callback path goes through that parser:** The `card.action.trigger` webhook handler calls `parseFeishuCardActionEventPayload(data)` and passes its result into `handleFeishuCardAction`, so the parser behavior is on the real runtime path.
  - file: [extensions/feishu/src/monitor.account.ts:372](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/monitor.account.ts#L372)
  - command: `nl -ba extensions/feishu/src/monitor.account.ts | sed -n '360,392p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Current main handler still only falls back from `action.value`:** On the legacy path, `handleFeishuCardAction` builds the synthetic command from `buildFeishuCardActionTextFallback(event)`, and that fallback only inspects `action.value.text`, `action.value.command`, or stringified `action.value`. There is no handling for native Feishu form metadata on `main` today.
  - file: [extensions/feishu/src/card-action.ts:418](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/card-action.ts#L418)

- **Existing tests bypass the production ingress path:** The card-action tests construct `FeishuCardActionEvent` objects directly and call `handleFeishuCardAction` without exercising `parseFeishuCardActionEventPayload`, so tests like the ones proposed in the PR would not catch the parser dropping the new fields.
  - file: [extensions/feishu/src/bot.card-action.test.ts:47](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/bot.card-action.test.ts#L47)

- **Review discussion already identified behavioral risk:** The PR review thread flagged that routing `option`/`options` form actions into a structured JSON branch can bypass the existing plain-command extraction from `action.value`, which risks breaking downstream slash-command handling for mixed payloads. That concern was not resolved in the posted PR.
  - sha: [e4295e34af9b](https://github.com/openclaw/openclaw/commit/e4295e34af9bac12fd61d186a6b2aa8c8034611c)

## Risks / Open Questions

- Merging as-is would likely leave the user-visible bug unfixed because the webhook parser still strips the new Feishu fields before the handler sees them.
- Even after fixing ingress, form/select callbacks that also rely on legacy `action.value.command` extraction need a clear contract to avoid regressing existing slash-command flows.
- The test coverage discussed in the PR is too handler-local; without an ingress-path test, this class of bug can reappear unnoticed.
