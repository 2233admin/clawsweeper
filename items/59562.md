## Summary

Keep open. The CJK table portion is already tracked by #55512 and draft PR #55596, but current `main` still uses code-unit string length for table padding, so that bug is not implemented. The paragraph-break portion is not fully covered by that canonical CJK issue and current Telegram formatting preserves existing Markdown/softbreak structure rather than adding blank lines to single-newline model output.

## Best Possible Solution

Keep this issue open, or split it: use #55512/#55596 as the canonical CJK table-width track, and track the Telegram paragraph-spacing behavior separately. The CJK fix likely belongs in the shared Markdown table code renderer with display-width-aware padding plus tests for Chinese/Japanese/Korean cells across Telegram/Discord-style code rendering. The paragraph issue needs maintainer product judgment on whether to add a Telegram-specific prompt instruction, a formatter option, or a conservative parser-aware postprocess.
