## Summary

Keep open. Current `main` still implements webchat dictation with browser-native `SpeechRecognition`, documents Talk mode as browser microphone audio sent directly to a realtime provider, and restricts chat attachments to images. I found no server-side STT upload/transcription path for webchat voice input on `0e7250f37bf347c72790abf2a6692d5ec84e5ea1`.

## Evidence

- **Webchat dictation is still browser STT only:** `ui/src/ui/chat/speech.ts` defines STT exclusively through `SpeechRecognition` / `webkitSpeechRecognition`; `startStt()` instantiates that browser API and there is no alternate backend path in this module.
  - file: [ui/src/ui/chat/speech.ts:1](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/ui/src/ui/chat/speech.ts#L1)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Composer only exposes the browser dictation button:** The chat composer shows the mic button only when `isSttSupported()` is true and wires it to `startStt()` / `stopStt()`. There is no second server-STT toggle or recorder flow here.
  - file: [ui/src/ui/views/chat.ts:1256](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/ui/src/ui/views/chat.ts#L1256)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Control UI chat attachments are image-only:** The UI accepts only `image/*`, and `buildApiAttachments()` serializes outgoing attachments as `{ type: "image", ... }` only. That rules out recorded audio blobs being sent from webchat through `chat.send` today.
  - file: [ui/src/ui/chat/attachment-support.ts:1](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/ui/src/ui/chat/attachment-support.ts#L1)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Gateway chat attachment parsing is image-focused, not audio STT:** `src/gateway/chat-attachments.ts` models inline chat attachments as `ChatImageContent`, and the parser drops or rejects non-image MIME types instead of routing audio into `tools.media.audio`.
  - file: [src/gateway/chat-attachments.ts:18](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/gateway/chat-attachments.ts#L18)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Docs still describe browser-direct voice paths, not server-side webchat STT:** `docs/web/control-ui.md` says Talk mode mints a short-lived realtime client secret and the browser sends microphone audio directly to OpenAI; the mic button is described separately as a microphone dictation button. That matches browser-only voice behavior, not gateway-hosted STT for webchat.
  - file: [docs/web/control-ui.md:107](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/docs/web/control-ui.md#L107)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **No MediaRecorder-based fallback exists on current main:** Repo grep for `MediaRecorder` returned no matches in `ui`, `src`, `docs`, or `extensions`, so I found no implemented browser recorder upload path for server-side transcription.
  - command: `rg -n "MediaRecorder" ui src docs extensions --glob '!**/dist/**'`
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
