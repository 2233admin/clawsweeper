## Summary

Keep open. Current main supports canonical Gemma 4 refs and documents model allowlists, but the exact reported override (`Gemma4-26b-a4-it-gguf`) still maps to the generic disallowed-override reset path rather than a provider-specific diagnostic. The issue is recent, concrete, and none of the allowed close reasons fit.

## Best Possible Solution

Keep this open and tighten the model override UX: when clearing a stored override, include the rejected normalized provider/model and hint to use `/models`, add the model to `agents.defaults.models`, or use the correct provider-qualified ref. A focused regression test should cover an unsupported Gemma/GGUF-shaped override and verify the diagnostic points users toward `google/gemma-4-26b-a4b-it`, `openrouter/google/gemma-4-26b-a4b-it`, or the configured local provider ref as appropriate.
