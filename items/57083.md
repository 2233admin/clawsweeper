## Summary

Current main already handles OpenAI-compatible tool-call arguments as JSON strings on the wire and parses them into object-shaped internal tool-call arguments before use. The issue’s proposed root cause does not match the shipped transport contract.

## Close Comment

Closing this as already implemented on `main`.

- Codex review of current `main` and shipped `v2026.4.23` shows OpenClaw already treats OpenAI-compatible `function_call.arguments` as JSON strings on the wire (`src/agents/openai-ws-types.ts:22`).
- The streaming OpenAI-compatible transport parses those argument strings into internal object-shaped tool-call arguments before use (`src/agents/openai-transport-stream.ts:1361`).
- The response conversion path also `JSON.parse`s `function_call.arguments` into internal `toolCall.arguments` (`src/agents/openai-ws-message-conversion.ts:566`).
- The transport test suite explicitly covers this contract and round-trip behavior (`src/agents/openai-ws-stream.test.ts:669`).
- Because of that, the reported root cause does not match the current implementation; this closeout is based on a Codex review of the code and shipped release state.
