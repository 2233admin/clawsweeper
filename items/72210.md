## Summary

Current main still appears to emit the misleading `delivery.resolved.error` trace for bare `delivery.mode="none"` cron runs. The status path is correctly `not-requested`, but the delivery trace is still built from a failed synthetic resolution and then persisted into cron run logs. The linked fix PR #72219 remains open and unmerged.

## Best Possible Solution

Keep #72210 open until PR #72219 or an equivalent maintainer-approved change lands. The narrow fix should be in `buildCronDeliveryTrace()`: when `deliveryPlan.mode === "none"`, omit `delivery.resolved` or emit neutral skipped metadata while preserving `deliveryStatus: "not-requested"` and the existing explicit-target behavior that keeps agent-initiated `message` tool use possible. Add a focused regression test that inspects `result.delivery` or the emitted run-log entry for the no-deliver trace shape.
