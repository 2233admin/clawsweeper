## Summary

Keep #72210 open. Current main fixed the broader delivery-status confusion for `delivery.mode: "none"`, but the specific nested run-record diagnostic in this issue still appears: bare no-deliver cron runs synthesize a non-ok `resolvedDelivery` with an error, `buildCronDeliveryTrace` always serializes that as `delivery.resolved.error`, and the gateway appends `evt.delivery` directly to `cron/runs/<jobId>.jsonl`. The remaining issue is narrower than the related closed reports, which focused on `deliveryStatus`/false delivery failures rather than the nested diagnostic trace.

## Best Possible Solution

Keep the issue open and fix the cron run-record trace path. The narrow maintainer fix should make `buildCronDeliveryTrace` or its caller treat bare `delivery.mode: "none"` as a neutral skip: either omit `delivery.resolved` or emit a non-failure shape such as `{ ok: true, skipped: true, reason: "delivery.mode=none" }`, while preserving explicit no-deliver targets for agent message-tool context. Add a regression test around `runCronIsolatedAgentTurn` or cron run-log output that asserts no `delivery.resolved.error` is written when delivery was not requested.
