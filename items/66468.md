## Summary

Keep open. Current main at edbab0e2 still does not provide a live provider/model-attempt signal for the CLI TUI footer during fallback. The fallback loop records/logs attempts, the agent command path only learns the successful fallback after `runWithModelFallback` returns, and the TUI footer still renders from `sessionInfo` without handling a fallback/model-switch event. Related status-bar fixes address selected/default/persisted model display, not this in-flight fallback-attempt path.

## Best Possible Solution

Keep this issue open and implement a narrow typed fallback-attempt status path: emit an agent event before each actually attempted fallback candidate with runId, sessionKey, provider, model, attempt, and total; have the CLI TUI accept that stream with session/run filtering before the unknown-run drop when necessary; update `state.sessionInfo.modelProvider` and `state.sessionInfo.model`; refresh the footer immediately; and add regression coverage for fallback emission plus TUI state updates.

## Risks / Open Questions

- The existing web fallback notice events are useful precedent, but reusing only post-transition lifecycle notices would still leave the stale in-flight candidate window reported here.
