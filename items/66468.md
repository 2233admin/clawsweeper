## Summary

Current `main` does not appear to implement live footer updates for provider/model fallback attempts. The fallback runner persists the selected candidate before each retry, but it does not emit a dedicated model-switch agent event, and the TUI only refreshes session info after runs finish or error. On source inspection, the report still matches current behavior, so this should stay open.

## Evidence

- **Fallback loop has no live model-switch emission:** `runWithModelFallback(...)` iterates candidates and calls `runFallbackAttempt(...)`, but this loop does not emit any agent event for the candidate being attempted. A repo search also found no `model_switch` stream on current `main`.
  - file: [src/agents/model-fallback.ts:746](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/agents/model-fallback.ts#L746)
  - command: `nl -ba src/agents/model-fallback.ts | sed -n '735,875p'; rg -n "model_switch|emitAgentEvent" src/agents/model-fallback.ts`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Reply runner persists fallback selection, then emits only lifecycle start:** Before each fallback attempt, the reply runner calls `persistFallbackCandidateSelection(provider, model)`, which matches the doc claim that session readers can see the candidate about to run. Immediately after that, the visible agent event emitted here is `stream: "lifecycle"` with `phase: "start"`; there is no model/provider payload for the TUI to consume live.
  - file: [src/auto-reply/reply/agent-runner-execution.ts:909](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/auto-reply/reply/agent-runner-execution.ts#L909)

- **TUI only refreshes session info at terminal run states and handles no model-switch stream:** `finalizeRun(...)` and `terminateRun(...)` are the places that call `refreshSessionInfo?.()`. Separately, `handleAgentEvent(...)` filters agent events to known runs and only has explicit branches for `tool` and `lifecycle`, so there is no path that updates `state.sessionInfo.modelProvider/model` during fallback progression.
  - file: [src/tui/tui-event-handlers.ts:197](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/tui/tui-event-handlers.ts#L197)

That supports the issue's root cause: the data path exists in session state, but the TUI is not refreshing it live during fallback attempts.
