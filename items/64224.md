## Summary

Keep open. Current main still treats billing failures as auth-profile disabled state, not model-scoped state, so a LiteLLM proxy using one auth profile for multiple upstream models can still have sibling models blocked after one upstream billing-shaped failure. The related #66315/#66314 work fixed billing guidance for fallback-summary errors, but it did not change cooldown scope or allow disabling billing backoff.

## Best Possible Solution

Keep this issue open and implement a deliberate proxy-aware or model-scoped billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or add a documented supported way to disable billing backoff for user-managed proxy failover. The likely implementation belongs in auth-profile usage state plus model fallback, with a regression test using one LiteLLM auth profile serving two sibling models where a billing failure on one model does not block the other; update config schema/help/docs if a public opt-out is added.

## Risks / Open Questions

- A fix needs to distinguish true provider-key billing exhaustion from proxy/upstream-model billing failures; making every billing failure model-scoped could cause repeated retries against genuinely exhausted accounts.
- Allowing billingBackoffHours: 0 or a disable flag should be documented as an operator-managed failover mode so users understand the retry and cost implications.
