## Summary

Keep open. Current `main` appears to have fixed the specific cross-channel symptom behind the report: restart sentinels now capture delivery context from the initiating session, and internal `webchat` targets are not treated as deliverable outbound channels, so a dashboard-triggered restart no longer obviously falls back to Telegram by default. But the requested product surface still does not exist: there is no `restart-sentinel` config toggle on `main`, and the restart startup path still sends an outbound restart notice whenever a routable deliverable target survives restart.

## Evidence

- **Restart payload uses the initiating session's stored delivery context:** `extractDeliveryInfo()` loads the session store for the provided `sessionKey`, prefers that session's delivery context, and only falls back to the base session when the current entry is not routable. That means restart routing is derived from the active session, not a global last-used channel.
