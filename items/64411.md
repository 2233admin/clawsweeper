## Summary

Keep open. Current main has a configurable MCP connection timeout and idle cleanup, but the main reply setup path still awaits MCP catalog materialization, MCP tool calls have no per-call timeout/circuit breaker, bundle MCP tools are marked non-optional, and docs/config do not expose degraded/unavailable health state or failure-threshold controls.

## Best Possible Solution

Keep the issue open and implement a focused MCP/plugin reliability layer: bound catalog listing and tool invocation separately from connection setup, add per-server/tool timeout and failure-threshold config, cache degraded state to skip unhealthy optional MCPs quickly, surface diagnostics in status/logs/user-visible run metadata, and add regression tests showing a slow or disconnected MCP cannot stall an otherwise healthy channel reply path.
