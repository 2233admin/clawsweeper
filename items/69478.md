## Summary

Keep open. Current main still does not implement the reported deduplication. `enqueueSystemEvent` only skips the immediately previous identical text within a session, while exec approvals are still keyed by opaque approval IDs; duplicate approval requests with fresh IDs can coexist and be broadcast/delivered independently. The two related PRs in the issue timeline (#69731 and #69893) were closed unmerged, and their proposed non-consecutive system-event dedupe is not present in the checked-out main tree.

## Best Possible Solution

Keep #69478 open and fix it in the approvals path, with a focused regression test. The safest direction is to coalesce or reject pending exec approval requests using a stable intent key derived from agent/session plus the canonical system-run binding or command context, or to cancel pending approvals when failover replaces a run. Add tests proving two fresh approval IDs for the same pending exec intent do not produce two operator prompts, and consider whether `enqueueSystemEvent` also needs context-key-aware dedupe for non-approval system events.
