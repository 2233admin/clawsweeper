## Summary

Keep open. Current `main` still does not show the claimed `(runId, contextKey)` approval-event coalescing: system-event suppression is text-only, approval requests still mint fresh UUID-based ids/context keys, and approval delivery deduplicates only exact approval ids. I did not find strong evidence that the cross-referenced closed PRs shipped a real fix for this specific regression.

## Evidence

- **System-event queue only suppresses consecutive duplicate text:** `enqueueSystemEvent` normalizes `contextKey`, stores it, then suppresses only when `lastText === cleaned`. There is no compound-key dedup on `runId` or `contextKey` before enqueue.
  - file: [src/infra/system-events.ts:90](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/infra/system-events.ts#L90)
  - command: `nl -ba src/infra/system-events.ts | sed -n '80,115p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Approval requests still generate fresh UUID-based ids/context keys:** `createExecApprovalRequestContext` still calls `crypto.randomUUID()` and derives `contextKey` from that approval id (`exec:${approvalId}`), so retries naturally produce new ids/context keys unless some higher layer coalesces them.
  - file: [src/agents/bash-tools.exec-host-shared.ts:142](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/bash-tools.exec-host-shared.ts#L142)

- **Approval delivery runtime deduplicates only exact approval ids:** The live/replay approval runtime ignores duplicates only when `pending.has(request.id)` is true. That prevents replaying the same approval id twice, but it does not collapse semantically identical retries that arrive under a fresh id.

That is a different invariant from coalescing duplicate approval requests before approval is granted.
