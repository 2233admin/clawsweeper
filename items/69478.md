## Summary

Keep #69478 open. Current main still lacks the requested semantic dedupe for duplicate exec approval prompts under new approval IDs. The system-event queue only suppresses consecutive duplicate text, the approval request path generates a fresh UUID for each approval, and the approval manager/forwarder key pending state by approval ID rather than by run intent, command, runId, or contextKey. The related PRs in the provided timeline were closed unmerged, so they do not supersede or implement the fix on main.

## Best Possible Solution

Keep the issue open and fix it in the exec approval/request path, not only in generic system-event text dedupe. The maintainable fix is to introduce a stable pending-approval intent key derived from the normalized exec request, such as agentId/sessionKey plus systemRunBinding or command/cwd/env identity, and have the manager or request handler return/reuse the existing pending approval for the same unresolved intent. Add focused regression tests proving retries with a fresh generated approval ID do not produce multiple forwarded/native Telegram approval prompts, while distinct commands or distinct sessions still produce separate approvals.
