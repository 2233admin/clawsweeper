## Summary

Keep open. Current main still has the reported failure mode: runtime pricing data can reach `estimateUsageCost` with missing `cacheRead`/`cacheWrite` rates, and that path still returns `undefined` instead of defaulting those rates to `0`, which prevents the diagnostics OTel plugin from exporting a cost metric for that event.

## Evidence

- **Runtime pricing loader does not default cache prices:** `loadModelsJsonCostIndex` reads `models.json` with raw `JSON.parse`, and `buildProviderCostIndex` copies `model.cost.cacheRead` / `model.cost.cacheWrite` straight into `ModelCostConfig` without `?? 0`. The type says those fields should exist, but this runtime path does not validate or backfill them.
  - file: [src/utils/usage-format.ts:187](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/utils/usage-format.ts#L187)
  - command: `nl -ba src/utils/usage-format.ts | sed -n '173,245p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Cost estimator still fails on missing cache prices:** `estimateUsageCost` normalizes usage counters with `toNumber(...)`, but it multiplies by `cost.cacheRead` and `cost.cacheWrite` directly. If either price is `undefined`, `total` becomes non-finite and the function returns `undefined` rather than treating the missing cache price as zero.
  - file: [src/utils/usage-format.ts:371](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/utils/usage-format.ts#L371)
  - command: `nl -ba src/utils/usage-format.ts | sed -n '360,390p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **OTel exporter still drops the cost metric when cost is missing:** The diagnostics OTel service only emits `openclaw.cost.usd` when `evt.costUsd` is truthy. If `estimateUsageCost` returns `undefined` from the missing cache-price path above, no cost metric is exported for that model usage event.
  - file: [extensions/diagnostics-otel/src/service.ts:547](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/diagnostics-otel/src/service.ts#L547)
  - command: `nl -ba extensions/diagnostics-otel/src/service.ts | sed -n '520,580p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **No regression test covers the reported path:** The current `diagnostics-otel` tests for `model.usage` validate trace parenting only. There is no test asserting cost export behavior when cache pricing fields are absent or defaulted to zero.
  - file: [extensions/diagnostics-otel/src/service.test.ts:456](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/diagnostics-otel/src/service.test.ts#L456)

## Risks / Open Questions

- `src/config/types.models.ts` requires `cost.cacheRead` and `cost.cacheWrite`, so many validated config paths are safe; the remaining risk is unvalidated runtime pricing input such as raw `models.json` data, which is enough to keep the issue open.
- Even if cache prices were defaulted to zero, `extensions/diagnostics-otel/src/service.ts` still suppresses zero-valued counters/costs via truthy checks, so observability around zero values remains fragile.
