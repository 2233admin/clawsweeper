## Summary

Keep this issue open. Current main still parses Feishu/Lark DM root/thread metadata but routes normal direct messages by sender Open ID only; topic session routing, parent peers, thread bootstrap, and MessageThreadId remain group/topic-group scoped. The existing ACP DM/thread binding path is a partial workaround for explicitly bound sessions, not general per-topic DM session isolation.

## Best Possible Solution

Keep this issue open. Implement the feature in the bundled Feishu plugin with an explicit opt-in DM topic/session-scope config, preserve current sender-scoped defaults, derive DM conversation IDs from sender plus canonical topic/root/thread metadata, propagate that ID through routing, configured/runtime bindings, replies, thread context, sequential queueing, and debounce keys, then add focused DM topic tests plus Feishu schema/docs updates. If #63148 is still active, review or port that PR as the implementation candidate rather than closing the issue first.

## Risks / Open Questions

- Feishu/Lark direct-message topic event semantics still need verification before choosing canonical precedence among root_id, thread_id, parent_id, and message_id fallbacks.
- The default sender-scoped DM behavior should be preserved unless users explicitly opt into per-topic isolation, otherwise existing sessions, bindings, dynamic-agent creation, and ACP workflows could change unexpectedly.
- A fix should cover receive-lane and debounce partitioning, not only route session keys, so simultaneous DM topics do not still serialize or merge incorrectly.
