## Summary

Do not close. Current `main` still has an active code path where WhatsApp group `allowlist` mode with no configured `groups` becomes effectively open at the route level whenever sender allowlist state is present upstream, and WhatsApp inbound policy still populates that sender allowlist from DM `allowFrom` entries.

## Evidence

- **WhatsApp inbound policy still falls back from DM allowlist to group allowlist:** `resolveWhatsAppInboundPolicy` sets `groupAllowFrom` from `account.groupAllowFrom`, then falls back to `configuredAllowFrom`, then `[]`. That means DM `allowFrom` entries still feed group policy when `groupAllowFrom` is unset.
  - file: [extensions/whatsapp/src/inbound-policy.ts:92](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/whatsapp/src/inbound-policy.ts#L92)
  - command: `nl -ba extensions/whatsapp/src/inbound-policy.ts | sed -n '79,139p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Core group-policy logic still allows groups with no `groups` config when `hasGroupAllowFrom` is true:** `resolveChannelGroupPolicy` computes `senderFilterBypass` for `groupPolicy === "allowlist" && !hasGroups && hasGroupAllowFrom`, then treats that as `allowed`. This is the exact fail-open route-level behavior the issue is about.
  - file: [src/config/group-policy.ts:352](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/config/group-policy.ts#L352)

- **Current tests still encode the disputed behavior:** `src/config/group-policy.test.ts` explicitly expects `groupPolicy=allowlist` with no groups and `hasGroupAllowFrom: true` to return `allowed=true`. That is strong evidence the behavior is still intentional on current `main`, not fixed.
  - file: [src/config/group-policy.test.ts:93](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/config/group-policy.test.ts#L93)
