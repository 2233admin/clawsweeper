## Summary

Keep open. The issue’s exact `fetchWithSsrFGuard` callsites no longer exist on current `main`, but current QQ Bot token/API traffic still does not show a maintained QQ-specific SSRF/proxy policy path or user-facing network config, and there is no fake-IP regression coverage proving the reported Surge/Clash scenario is resolved.

## Evidence

- **QQ Bot token fetch on current main is a raw `fetch()` with no SSRF/policy parameter:** `TokenManager.doFetchToken()` posts to `https://bots.qq.com/app/getAppAccessToken` via plain `fetch(...)`; there is no `fetchWithSsrFGuard` call and no QQ-specific policy injection at this boundary.

## Risks / Open Questions

- Closing as `cannot_reproduce` would be weak because the old guarded-fetch implementation details changed, but there is no proof that the reported proxy environment now works on current main.
- Current main appears inconsistent: media fetches have explicit RFC2544 handling while token/API calls do not, which is close to the reporter’s underlying complaint even if the exact codepath changed.
- Without a QQ Bot network config surface or regression test, future maintainers cannot verify or preserve fake-IP proxy compatibility.
