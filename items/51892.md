## Summary

Keep open. Current `main` no longer leaks the `[[reply_to:...]]` tag into delivered iMessage text, but it still does not implement the native threaded-reply behavior requested in #51892. OpenClaw forwards `replyToId` only as `params.reply_to`, and current upstream `imsg` RPC/send code does not define or consume any reply-thread send parameter, so there is no strong evidence the requested native reply plumbing is implemented on `main`.

## Evidence

- **OpenClaw strips tags, then forwards `replyToId` only as `reply_to`:** `extensions/imessage/src/send.ts:147-159` strips inline directive tags from the outgoing message and, if `opts.replyToId` is present, sets `params.reply_to = resolvedReplyToId`. There is no `reply_to_guid` or equivalent native-thread field in the OpenClaw send request.
  - file: [extensions/imessage/src/send.ts:147](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/imessage/src/send.ts#L147)
  - command: `nl -ba extensions/imessage/src/send.ts | sed -n '1,220p'`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Literal reply tag symptom is outdated on current main:** `src/utils/directive-tags.ts:123-131` shows `stripInlineDirectiveTagsForDelivery()` removing `[[reply_to:...]]` tags from delivered text. That means the issue body's 'literal text prefix remains in the outbound message' description no longer matches current main, but this only fixes tag leakage, not native threaded replies.
  - file: [src/utils/directive-tags.ts:123](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/utils/directive-tags.ts#L123)

That is not evidence of native threaded-reply support.
