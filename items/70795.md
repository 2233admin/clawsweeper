## Summary

Keep open. Current main has partial coverage for `workspaceOnly=false` host reads, but the reported direct-session/tool-host policy mismatch is not proven fixed. The active read-tool assembly and workspace guard are still centered on a single workspace/sandbox root, and there is no exact regression test for a direct chat/session read against an approved non-workspace root.

## Best Possible Solution

Keep the issue open. Add a focused regression test for direct chat/session read-tool assembly with `tools.fs.workspaceOnly=false` and an approved non-workspace readonly root, then either thread the effective readonly roots through the topmost read-tool/tool-host path or document that sandboxed/direct sessions require explicit bind mounts and make the error reflect the effective policy.
