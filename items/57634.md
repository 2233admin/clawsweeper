## Summary

Current `main` already supports inline rich web content in WebChat/Control UI via hosted `[embed ...]` renders backed by managed HTML canvas documents, which covers the requested in-chat interactive-form use case even though the shipped contract is not a raw `type:"html"` message part.

## Evidence

- **Inline web content is a first-class WebChat/Control UI feature:** The docs say assistant messages can render hosted web content inline with `[embed ...]`, with sandbox modes including `scripts` for interactive embeds.
  - file: [docs/web/control-ui.md:182](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/docs/web/control-ui.md#L182)

- **Rich output contract is inline embed, not raw html message type:** The rich output protocol states `[embed ...]` is the only agent-facing rich render syntax for Control UI, renders in the assistant message surface, and block-form inline HTML embed shortcodes are not rendered.
  - file: [docs/reference/rich-output-protocol.md:18](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/docs/reference/rich-output-protocol.md#L18)

## Risks / Open Questions

- The implemented path is hosted `[embed ...]` plus managed HTML bundles, not an arbitrary raw `type:"html"` message payload rendered with `innerHTML`.
- The best identifying implementation commit found (`9ece33c50506ba031df382f69bde65dfda19bea5`) is on `main` but is not contained in a tagged release from the local checkout.

## Close Comment

Closing this as implemented on current `main` based on a Codex review.

- WebChat / Control UI already supports inline rich web content inside the assistant bubble via the hosted `[embed ...]` contract: `docs/web/control-ui.md:182` and `docs/reference/rich-output-protocol.md:18`.
- Managed canvas documents already accept inline HTML bundles (`entrypoint: { type: "html", value: ... }`) and expose them as hosted `/__openclaw__/canvas/documents/...` URLs: `src/gateway/canvas-documents.ts:16` and `src/gateway/canvas-documents.test.ts:60`.
- The UI tests cover the in-chat rendering path directly: `[embed ref="..."]` normalizes to an assistant-message canvas block and renders inside the assistant bubble: `ui/src/ui/chat/message-normalizer.test.ts:101` and `ui/src/ui/chat/grouped-render.test.ts:775`.
- The raw `type: "html"` / `innerHTML` approach from this issue is not the shipped API; current `main` intentionally uses hosted embeds instead of arbitrary inline HTML injection: `ui/src/ui/chat/tool-cards.test.ts:205`.
- Best implementation/proof commit on `main`: `9ece33c50506ba031df382f69bde65dfda19bea5`.
