## Summary

Keep open. Current main still implements memory search as one resolved provider/model/fallback/store per agent, with model filtering inside a single SQLite index and fallback handled by switching provider then rebuilding the store. It does not implement first-class multi-index embedding memory, `indexes[]`/`activeIndex` config, per-index ingestion state, observable active/fallback index routing, or runtime read failover across prebuilt vector spaces. Related #55590 is closed and does not supersede this more precise remaining design request.

## Best Possible Solution

Keep the issue open and design it as memory-core or memory plugin-contract work: add backward-compatible multi-index config, a canonical chunk source with per-index ingestion/backfill state, an active/fallback index router that never mixes vector spaces in one query, per-index status/metrics for coverage and lag, and tests/docs that preserve current single-index configs plus existing `memorySearch.fallback` full-reindex semantics.
