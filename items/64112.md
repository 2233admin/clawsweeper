## Summary

Keep this PR open. Current main still calls the plugin allowlist fallback for an unregistered primary command before proving that the command is a real bundled plugin CLI root, so an unknown command such as `openclaw run` can still get a misleading `plugins.allow` diagnostic. The PR remains a focused implementation candidate, and its latest patch addresses the prior review concerns around aliased plugin roots, metadata-probe failures, and `plugins.deny` during probing. Security review found no unrelated supply-chain-sensitive changes in the PR diff.

## Best Possible Solution

Keep this PR open and finish the maintainer review. The best code path is to land this PR, or apply an equivalent main-branch fix that only emits `plugins.allow` / `plugins.entries.<id>.enabled=false` diagnostics after resolving the missing primary command to a real bundled plugin CLI root or manifest-owned command alias, while keeping the `openclaw run` restrictive-allowlist regression and the metadata-probe failure/denylist safeguards.
