## Summary

Keep #72062 open. Current main does not implement `agents.list[].tools.deny_wrappers` or an equivalent per-agent exec script-path deny policy. The existing `tools.allow`/`tools.deny` policy is still a tool-name filter, while `tools.exec` approval/allowlist handling matches executable/binary policy and does not expose a documented per-agent glob deny over resolved interpreter script arguments. This is core exec-policy/security behavior, not an optional ClawHub plugin cleanup item, and the related approval-loop issues do not supersede it.

## Best Possible Solution

Keep the issue open for maintainer/product design. The likely core path is to add a typed per-agent exec wrapper/script deny surface, decide whether it belongs under `agents.list[].tools.exec` or beside `tools.deny`, resolve interpreter script operands safely before approval prompts, define glob and path-resolution semantics, make deny precedence explicit, document it on the public exec/config pages, and add focused tests for per-agent behavior, allowlist composition, and non-regression of existing exec approvals.
