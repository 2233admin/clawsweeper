## Summary

Keep #64016 open. Current main still allows queued non-interval wake requests to run immediately after an in-progress heartbeat, and the scheduler only applies the nextDueMs due-window guard to interval-triggered runs. The existing duplicate suppression is payload-level after the LLM call, not a run-level guard for the reported extra heartbeat executions.

## Best Possible Solution

Keep the issue open and add a scheduler-level regression test for an interval heartbeat that is still running when a scoped external wake arrives. The likely fix belongs in src/infra/heartbeat-runner.ts and/or src/infra/heartbeat-wake.ts: non-interval wake-triggered heartbeat runs should not execute another full heartbeat inside the current interval window unless there is a distinct event payload that must be drained, and schedule advancement semantics should be documented in the heartbeat scheduler tests.
