## Summary

Keep open. Current main does not give strong evidence that #57661 is fixed: the gateway resolver can reject missing session keys, but the agent-side `sessions_send` resolution path still falls back to a raw session key and the `agent` handler will create/persist a new session entry when that key has no existing store row. With the issue still recent and a confirming comment about non-existent-session sends timing out, there is not enough proof to close conservatively.

## Evidence

- **Agent tool swallows missing-key resolution failures:** `resolveSessionKeyFromKey()` catches `sessions.resolve` failures and returns `null`, and `resolveSessionReference()` then falls back to `resolveInternalSessionKey(...)` instead of surfacing a hard missing-session error.
  - file: [src/agents/tools/sessions-resolution.ts:282](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/tools/sessions-resolution.ts#L282)
  - command: `nl -ba src/agents/tools/sessions-resolution.ts | sed -n '282,428p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Gateway resolver itself does reject absent session keys:** `resolveSessionKeyFromResolveParams()` returns `No session found: <key>` when the canonical or legacy store key is absent, so the lower-level gateway path knows how to fail fast for missing sessions.
  - file: [src/gateway/sessions-resolve.ts:101](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/sessions-resolve.ts#L101)
  - command: `nl -ba src/gateway/sessions-resolve.ts | sed -n '100,130p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Agent handler turns unknown session keys into persisted sessions:** When `request.sessionKey` is present, the `agent` handler treats `!entry` as a new session, assigns `entry?.sessionId ?? randomUUID()`, and persists `store[primaryKey] = merged`, so an unresolved target key can become a new session instead of a clear not-found rejection.
  - file: [src/gateway/server-methods/agent.ts:620](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/server-methods/agent.ts#L620)
  - command: `nl -ba src/gateway/server-methods/agent.ts | sed -n '620,741p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Issue still has corroborating recent report:** The issue has a 2026-04-15 comment stating they also see timeouts when `session_send` targets a non-existent session, which weakens any close-as-fixed conclusion without a direct repro or targeted regression proof.
