## Summary

Keep #71609 open. Current main has partial hardening for URL/shared-token precedence and a bounded cached-device-token retry, but the browser Control UI still does not fully implement the requested stale device-token cleanup and non-looping recovery path. In particular, primary stored-token `AUTH_DEVICE_TOKEN_MISMATCH` is not cleared, `AUTH_DEVICE_TOKEN_MISMATCH` is not treated as non-recoverable, and a checked-in browser test still preserves reconnecting after an initial `AUTH_TOKEN_MISMATCH` with no retry.

## Best Possible Solution

Keep the issue open and fix the browser Control UI auth recovery path. The likely path is to clear stale cached device auth on `AUTH_DEVICE_TOKEN_MISMATCH` even when the stored device token was the primary credential, stop automatic reconnect loops once no safe retry remains, avoid burning the gateway auth rate limiter, keep explicit URL/shared-token precedence, and replace the `rotate/reissue` mismatch guidance with stale identity cleanup or reconnect instructions. Add focused browser tests for the scope-upgrade stale-token case and for the non-looping recovery behavior.
