## Summary

Keep open. Current `main` still uses the `/new` or `/reset` command timestamp for the memory filename/header, so the reported chronology bug is still plausible, and the PR discussion identified unresolved edge cases in the proposed fix.

## Evidence

- **Current main still stamps memory files from the command timestamp:** `saveSessionToMemory` sets `now = new Date(event.timestamp)`, then derives `dateStr`, the fallback time slug, and the `# Session:` header time from that same value. There is no session-created-at lookup on `main`, so a session created on one day and rotated with `/new` on another day would still be dated with the later command time.
  - file: [src/hooks/bundled/session-memory/handler.ts:84](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/hooks/bundled/session-memory/handler.ts#L84)

  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Main already has rotated-transcript fallback only for content:** `getRecentSessionContentWithResetFallback` falls back to the newest `.jsonl.reset.*` sibling when the active transcript is empty, and existing tests explicitly cover that rotated-session path. That makes the PR's goal credible, because content can already come from the old session while the filename/header on `main` still come from `event.timestamp`.
