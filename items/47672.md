## Summary

Keep open. Current `main` does not implement this PR’s change: the pre-doctor git-update path still prints a static update note and calls `runGatewayUpdate(...)` without passing an update progress controller, so the requested live per-step progress is still missing from `openclaw doctor`.

## Evidence

- **Doctor pre-update flow still uses only a static note:** `maybeOfferUpdateBeforeDoctor` still emits `note("Running update (fetch/rebase/build/ui:build/doctor)…", "Update")` and then calls `runGatewayUpdate` with only `cwd` and `argv1`; there is no `createUpdateProgress` import or `progress` wiring in current `main`.
  - file: [src/commands/doctor-update.ts:55](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/commands/doctor-update.ts#L55)
  - command: `nl -ba src/commands/doctor-update.ts | sed -n '40,90p'`
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Update runner only reports step progress when a progress hook is supplied:** `runStep` invokes `progress?.onStepStart?.(...)` and `progress?.onStepComplete?.(...)`; with no `progress` object passed from `doctor-update.ts`, no per-step progress output can be shown.
  - file: [src/infra/update-runner.ts:362](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/infra/update-runner.ts#L362)

- **Existing progress controller already exists and is used by `openclaw update`:** `createUpdateProgress` builds the spinner-based step reporter, and the normal update command creates `{ progress, stop }` from it before invoking the update flow. That contrast reinforces that the doctor path is the missing wiring point, not an already-implemented alternative.
