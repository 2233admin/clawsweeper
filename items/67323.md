## Summary

Keep open. The issue is recent, has multi-channel corroboration in the discussion, a maintainer already said it should stay open under umbrella issue #69208, and current `main` does not show a clear end-to-end fix for duplicate inbound user messages in session context.

## Evidence

- **Maintainer triage said not to close as MSTeams-only:** Issue discussion on 2026-04-20: BradGroux said the report is credible, likely broader than MSTeams, did not find a merged PR that clearly resolves #67323 end to end, and recommended keeping this issue open under umbrella #69208.
- **Cross-channel confirmations remain in the thread:** Issue discussion includes the 2026-04-16 report from `rrriiiccckkk` describing the same duplication pattern on Feishu and webchat, and a 2026-04-24 comment from `joeykrug` reporting the same symptom on Signal. That makes this a live cross-channel duplication family, not a stale Teams-only report.
- **Current Teams inbound path still records and dispatches inbound messages without a visible replay-rejection step:** On current `main`, the MSTeams handler enqueues the inbound system event with `activity.id` in the context key, then builds `ctxPayload` with `MessageSid: activity.id`, records inbound session state, and dispatches to the agent. I did not find a nearby inbound activity-id dedupe guard in this path.
