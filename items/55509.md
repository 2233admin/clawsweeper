## Summary

The report is about active memory-plugin prompt/capture behavior in OpenViking, which is outside OpenClaw core scope. OpenClaw intentionally delegates memory recall/prompt injection to the selected memory plugin.

## Evidence

- **Vision scope keeps memory behavior plugin-owned:** OpenClaw core stays lean, optional capability should usually ship as plugins, and memory is a dedicated plugin slot.

## Close Comment

Closing after a Codex maintainer review because the reported behavior is owned by the active memory plugin, not OpenClaw core. `VISION.md:54` and `docs/concepts/memory.md:31` make memory a plugin slot, and the core contract in `src/plugins/types.ts:2130` plus `src/plugins/registry.ts:1495` explicitly hands memory prompt injection to the selected memory plugin. This report names OpenViking v0.2.12, and `rg -n "OpenViking" .` returns no matches in this repo, so the requested freshness/timestamp/capture-policy changes belong in that plugin/skill repository rather than here. For reference, the bundled `memory-lancedb` plugin already treats recalled memories as untrusted historical data and avoids self-poisoning from model output in `extensions/memory-lancedb/index.ts:244` and `extensions/memory-lancedb/index.ts:632`. If this is still a problem in OpenViking, please reopen it in the OpenViking or ClawHub plugin repo.
