## Summary

Keep open. Current `main` still renders Slack outbound text as `mrkdwn` and only emits Block Kit when callers provide explicit `presentation` or `channelData.slack.blocks`; the proposed automatic Markdown -> Slack `rich_text` conversion is not implemented.

## Evidence

- **Slack markdown renderer still targets mrkdwn:** `markdownToSlackMrkdwn()` parses Markdown IR and renders Slack marker text, and `markdownToSlackMrkdwnChunks()` is the chunking path used for Slack text output.
  - file: [extensions/slack/src/format.ts:121](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/slack/src/format.ts#L121)
  - command: `nl -ba extensions/slack/src/format.ts | sed -n '115,170p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Agent guidance still asks for Slack mrkdwn:** Slack channel prompt hints explicitly say `text_markup: "slack_mrkdwn"` and instruct agents to use Slack mrkdwn rather than standard Markdown.
  - file: [extensions/slack/src/shared.ts:95](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/slack/src/shared.ts#L95)
  - command: `nl -ba extensions/slack/src/shared.ts | sed -n '88,110p'`

- **Outbound Slack blocks come only from explicit blocks/presentation/interactive payloads:** `resolveSlackBlocks()` merges `channelData.slack.blocks`, rendered `presentationBlocks`, and interactive blocks. There is no path that converts ordinary reply Markdown into Slack `rich_text` blocks.
  - file: [extensions/slack/src/outbound-adapter.ts:105](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/slack/src/outbound-adapter.ts#L105)
