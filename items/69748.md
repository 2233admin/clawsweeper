## Summary

Keep #69748 open. Current main still models Discord guild access as `requireMention?: boolean` plus `users`/`roles` sender allowlists, not the proposed `bypass`/`mention`/`deny` access-list policy. The related #39661 tracks a narrower sender-specific mention-policy gap, but this issue also asks for deny precedence, permission shortcuts, channel merge/inherit semantics, and backward-compatible migration, so it is not fully superseded.

## Best Possible Solution

Keep this open for bundled Discord plugin work. Add a typed Discord guild/channel access policy with `deny`, `bypass`, and `mention` entries, define precedence and channel inheritance/merge behavior, preserve existing `requireMention`/`users` as a migration/deprecation path, validate it in the strict schema, update docs and generated metadata, and cover user, role, permission, deny-first, and per-channel override cases in Discord preflight tests.

## Risks / Open Questions

- The replacement policy needs careful backward-compatible migration from `requireMention`/`users` so existing guilds do not become more permissive by accident.
- Permission shortcuts such as `perm:admin` need explicit Discord permission-source semantics and tests, since current role allowlists only match role IDs.
