## Summary

Keep this PR open. Current main still has the Telegram model-picker fallback bug: the routed agent is used to build the provider/model list, but the no-override current-model fallback still reads the global default instead of the routed agent default, so an agent-level default can fail to receive the picker checkmark.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land this fix or an equivalent one that computes the routed agent default in `resolveTelegramSessionState()`, uses it for the final no-override fallback, keeps the regression test for an agent default that differs from the global default, and verifies the Telegram extension lane before merge.

## Evidence

- **Current main fallback still uses global default:** After resolving the routed Telegram conversation and session entry, `resolveTelegramSessionState()` only calls `resolveDefaultModelForAgent()` for the stored-override default provider. If no override or stored model exists, it falls back to `runtimeCfg.agents?.defaults?.model`, ignoring `route.agentId`.
