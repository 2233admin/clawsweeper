## Summary

Keep this issue open. Current main still has no hard exec-boundary guard for model-issued OpenClaw self-management shell commands such as `openclaw config set ...` or `openclaw gateway restart` when host exec resolves to the documented default `security="full"` and `ask="off"`. The existing prompt guidance and gateway-tool config guard are useful partial hardening, but they do not block the raw exec path reported here.

## Best Possible Solution

Keep this open for a maintainer/security fix at the exec boundary. Add a hard runtime guard that blocks or requires explicit user approval for model-initiated OpenClaw self-management shell commands such as `openclaw config set`, `openclaw models set`, `openclaw gateway restart`, `openclaw update`, and uninstall equivalents unless they come from an explicit user-approved config-change action. Add regression coverage for ambiguous model-switch prompts that may show example commands but must not execute them automatically, and coordinate the broader destructive-command policy with #65307.

## Risks / Open Questions

- No live reproduction was attempted because reproducing the report would intentionally mutate OpenClaw config and restart the gateway; the source and docs are sufficient to show the enforcement gap.
- Deployments that explicitly tighten exec approvals or deny the exec tool may not reproduce, but current documented defaults still allow this class of model-issued host mutation.
- The fix should coordinate with #65307 so config mutation, model switching, gateway lifecycle, update, uninstall, and other self-management commands share one conservative policy instead of narrow one-off blocks.
