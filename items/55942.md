## Summary

Keep open. Current main has generic malformed-tool-call normalization and unknown-tool loop guards that could mitigate part of the symptom, but the NVIDIA provider still ships Kimi as a plain `openai-completions` model with no NVIDIA/Kimi-specific compat flags or regression coverage for this path. I do not have strong evidence that #55942 is fixed on current `main`.

## Evidence

- **NVIDIA/Kimi still has no provider-specific compat in catalog:** `buildNvidiaProvider()` still defines `moonshotai/kimi-k2.5` as a plain `openai-completions` model with no `compat` block, no replay/tool-schema hooks, and no string-content/tool-schema special handling.
  - file: [extensions/nvidia/provider-catalog.ts:13](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/nvidia/provider-catalog.ts#L13)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **String-content flattening is opt-in and defaults off:** OpenAI-completions string flattening only happens when `compat.requiresStringContent` is true; otherwise message arrays are sent through unchanged. The current NVIDIA/Kimi catalog entry does not opt into that compat path.
