## Summary

Keep open. Current `main` does not implement a custom TUI client display name: `openclaw tui` exposes no `--client-name`/`--username` flag, `TuiOptions` has no field for it, and the Gateway-backed TUI client still hardcodes `clientDisplayName: "openclaw-tui"`.

## Evidence

- **CLI surface has no custom-name flag:** `registerTuiCli()` defines `--local`, `--url`, `--token`, `--password`, `--session`, `--deliver`, `--thinking`, `--message`, `--timeout-ms`, and `--history-limit`, but no `--client-name` or equivalent option.
  - file: [src/cli/tui-cli.ts:10](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/cli/tui-cli.ts#L10)
  - command: `nl -ba src/cli/tui-cli.ts | sed -n '1,120p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **TUI option type has no display-name field:** `TuiOptions` includes connection/session/message fields only; there is no property that could carry a custom client display name into the TUI runtime.
  - file: [src/tui/tui-types.ts:1](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/tui/tui-types.ts#L1)

- **Gateway-backed TUI still hardcodes the display name:** `GatewayChatClient` constructs `GatewayClient` with `clientName: GATEWAY_CLIENT_NAMES.TUI` and `clientDisplayName: "openclaw-tui"`, so multiple TUI instances still present the same display name.
