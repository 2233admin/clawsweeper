## Summary

Keep open. Current `main` still buffers QQ Bot `kind: "tool"` text instead of delivering it immediately, and a later block/final reply clears the tool-only timeout before any buffered text is sent. The report's claim that QQ Bot `streaming.mode: "partial"` should imply preview/tool-progress delivery is not supported by current docs/schema, but the silent-while-tools-run behavior is still present and not fixed on `main`.

## Evidence

- **QQ Bot gateway still buffers tool text and returns without sending:** In the QQ Bot outbound dispatcher, `info.kind === "tool"` pushes `payload.text` into `toolTexts`, arms `toolOnlyTimeoutId`, and returns immediately. There is no immediate downstream send for text-only tool payloads.
  - file: [extensions/qqbot/src/engine/gateway/outbound-dispatch.ts:195](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/qqbot/src/engine/gateway/outbound-dispatch.ts#L195)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/outbound-dispatch.ts | sed -n '183,245p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **A later block response cancels the tool-only timeout:** When a block reply arrives, the same dispatcher sets `hasBlockResponse = true` and clears `toolOnlyTimeoutId`, which prevents the buffered tool-text fallback from firing before the final/block reply path continues.
  - file: [extensions/qqbot/src/engine/gateway/outbound-dispatch.ts:248](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/qqbot/src/engine/gateway/outbound-dispatch.ts#L248)
  - command: `nl -ba extensions/qqbot/src/engine/gateway/outbound-dispatch.ts | sed -n '247,257p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Tool-text fallback only exists for tool-only runs:** The only text send for buffered tool payloads is `sendToolFallback()`, which emits `toolTexts` after the timeout or in the final `finally` block when there was no block response. That matches the issue's described silent-until-final behavior.
  - file: [extensions/qqbot/src/engine/gateway/outbound-dispatch.ts:101](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/qqbot/src/engine/gateway/outbound-dispatch.ts#L101)

  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Core still treats tool results as a deliverable channel surface:** `dispatchReplyFromConfig()` forwards tool results to the channel dispatcher via `dispatcher.sendToolResult(deliveryPayload)` unless explicitly suppressed, so QQ Bot's buffering is a channel-specific choice rather than evidence that tool text is already fixed upstream.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:947](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/auto-reply/reply/dispatch-from-config.ts#L947)
