## Summary

Keep open. Current `main` still does not mark externally started session runs as busy until a `chat` event arrives, and the TUI ignores the `sessions.changed` lifecycle-start signal that the gateway emits. That means bridge/webhook/cron-triggered work can still look idle until the first streamed assistant delta or the final reply lands.

## Evidence

- **TUI only claims a run from chat events:** `handleChatEvent()` is where the TUI first records `activeChatRunId`; it does not set a run on agent lifecycle start. Busy status is only switched to `streaming` inside the `delta` branch.
  - file: [src/tui/tui-event-handlers.ts:305](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/tui/tui-event-handlers.ts#L305)
  - command: `nl -ba src/tui/tui-event-handlers.ts | sed -n '240,420p'`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Unknown lifecycle events are dropped:** `handleAgentEvent()` rejects events unless the run is already active/known. A system-injected run that emits lifecycle `start` before any `chat` delta is ignored, so the activity indicator cannot enter `running` from that signal.
  - file: [src/tui/tui-event-handlers.ts:400](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/tui/tui-event-handlers.ts#L400)

  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Current TUI event loop ignores sessions.changed:** The gateway client dispatch in `runTui()` handles only `chat`, `chat.side_result`, and `agent`. There is no branch for `sessions.changed`.
  - file: [src/tui/tui.ts:1034](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/tui/tui.ts#L1034)
  - command: `nl -ba src/tui/tui.ts | sed -n '1026,1044p'`

- **Gateway does emit a lifecycle-start session signal:** On lifecycle `start`, the gateway broadcasts `sessions.changed` with `sessionKey`, `phase`, and `runId`, which is the obvious signal an attached TUI could use before any assistant text is streamed.
  - file: [src/gateway/server-chat.ts:1004](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/gateway/server-chat.ts#L1004)

- **Tests encode the current limitation:** The TUI tests only expect lifecycle handling when `activeChatRunId` is already set, and they explicitly seed that run id from a prior chat event. There is no coverage for lifecycle-start-first external runs.
  - file: [src/tui/tui-event-handlers.test.ts:188](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/tui/tui-event-handlers.test.ts#L188)

  - file: [src/tui/tui.ts:585](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/tui/tui.ts#L585)

## Risks / Open Questions

- This is a code-path review, not a live interactive reproduction; however the current event wiring is specific enough that the behavioral gap is high-confidence.
- Some injected runs may emit a chat delta quickly, so the user may see a late streaming indicator instead of no indicator for the full run. The reported bug still stands for the missing start-of-run indicator.
