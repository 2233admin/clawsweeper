## Summary

Keep this PR open. Current main and the latest release do not implement an equivalent protected instruction-file mutation guard, and the PR remains a plausible core security-hardening candidate rather than conservative cleanup. The existing review discussion is useful: reviewers identified apply_patch, path-alias, and canonical-target bypass classes, and the PR author added follow-up commits addressing several of those areas.

## Best Possible Solution

Keep this PR open for maintainer security review or replace it with an equivalent core implementation. The best path is to define the explicit approval/product boundary for agent edits to bootstrap and instruction files, then implement one central mutation policy covering host write/edit, sandbox write/edit, apply_patch add/update/delete/move, normalized path aliases, symlink/hardlink canonical targets, and any intended exec/process scope. Add regression tests for those bypass classes and update docs so main-session memory/persona workflows remain clear.

## Risks / Open Questions

- Merging a partial guard could create a false sense of protection if any mutation path remains uncovered, especially apply_patch add/update/delete/move, sandbox bridge writes, path aliases, symlinks, hardlinks, or exec/process-created writes.
- The requested blanket block conflicts with documented main-session workflows that intentionally update SOUL.md and with memory flows that write MEMORY.md, so maintainers should define the approval path and exceptions before merge.
- The protected basename set is product policy, not just mechanics; maintainers should confirm whether CLAUDE.md, TOOLS.md, BOOT.md, TASKS.md, MEMORY.md, SOUL.md, and IDENTITY.md should all share identical runtime guard semantics.
