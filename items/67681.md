## Summary

Keep open. Current `main` clearly changed cron semantics: `--no-deliver` now disables runner fallback delivery, duplicate fallback delivery is suppressed when the agent already sent via the message tool, and docs explicitly describe that behavior. But there is not high-confidence evidence that the reported CLI-script path (`openclaw message send` inside an exec'd script) is fully covered, nor that OpenClaw now provides the stricter 'silent cron run' behavior the issue asks for.

## Evidence

- **CLI docs define current `--no-deliver` semantics narrowly:** The cron CLI docs now say isolated `--no-deliver` keeps output internal, but only by disabling runner fallback; they explicitly say it does not remove the agent's `message` tool when a chat route is available.
  - file: [docs/cli/cron.md:23](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/docs/cli/cron.md#L23)
  - command: `nl -ba docs/cli/cron.md | sed -n '18,40p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **User-facing automation docs match that behavior:** The scheduled-tasks docs say `delivery.mode = "none"` means no runner fallback delivery, and that isolated jobs can still send directly with the `message` tool; `announce` is skipped only when the agent already sent to the configured/current target.
  - file: [docs/automation/cron-jobs.md:137](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/docs/automation/cron-jobs.md#L137)

- **Current cron runtime keeps chat sending enabled for `delivery.mode="none"`:** `resolveCronToolPolicy()` enables the message tool for both `announce` and `none`, and `appendCronDeliveryInstruction()` adds no auto-delivery instruction when `deliveryRequested` is false.
