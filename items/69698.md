## Summary

Keep #69698 open. Current main still has the reported protocol mismatch: the persisted exec allowlist entry type can include internal `source` metadata, the gateway snapshot payload only redacts socket token/path metadata and does not strip allowlist `source`, while `exec.approvals.set` validates against a schema that rejects additional allowlist properties. The Control UI clones `exec.approvals.get` snapshots into its editable form and sends that form back unchanged on save, so a snapshot containing `source: "allow-always"` can still fail exactly as reported. The provided timeline links PR #69732 as the proposed fix, but it is open/unmerged in the supplied GitHub context, so the issue is not implemented on main yet.

## Best Possible Solution

Keep #69698 open until the transport redaction fix lands. The likely maintainer path is to land or supersede PR #69732 with a gateway/node snapshot sanitizer that strips internal allowlist-only fields such as `source` before Control UI receives them, plus regression coverage proving `exec.approvals.get` output can be sent back through `exec.approvals.set` without schema rejection while persisted internal metadata remains available to the store.
