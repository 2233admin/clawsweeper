## Summary

Keep this PR open. Current main still closes the WhatsApp socket without any debouncer-wide drain, and the PR targets the right failure mode, but the submitted one-shot `flushAll()` still does not quiesce late enqueues or active keyed work. The PR also needs focused regression coverage proving `reply`/`sendMedia` complete before socket close. Security review found no obvious supply-chain, CI, dependency, secret, or artifact-execution regression in the provided TS-only diff.

## Best Possible Solution

Keep this PR open for revision. The best implementation is a real debouncer drain/quiescence API in `src/auto-reply/inbound-debounce.ts` that repeatedly flushes current buffers and waits for active keyed chains until stable, then call it from `extensions/whatsapp/src/inbound/monitor.ts` after detaching inbound listeners and before `closeInboundMonitorSocket(sock)`. Add focused WhatsApp tests proving pending debounced messages can still send `reply` and `sendMedia` before close, including late-enqueue and in-flight keyed-task cases; keep the plugin SDK return shape and mocks aligned.
