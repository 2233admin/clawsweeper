## Summary

Keep open. Current main has auth-profile cooldowns and capped same-provider failover, but it does not implement the requested model-level circuit breaker with a dedicated `models.circuitBreaker` config, repeated-failure thresholding per provider/model, or persistent handling for timeout/network failures.

## Evidence

- **No circuit-breaker config or implementation found:** A repo-wide search for `models.circuitBreaker`, `circuitBreaker`, `failureThreshold`, and `cooldownMinutes` returned no matches on current `main`. That means the issue's proposed public surface is not present.
  - command: `rg -n "models\.circuitBreaker|circuitBreaker|failureThreshold|cooldownMinutes" src docs test package.json`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Current config is auth-profile cooldowns, not model circuit breaking:** The documented knobs are `auth.cooldowns.*`, including `overloadedProfileRotations` and `rateLimitedProfileRotations`. There is no documented model-level breaker config.
  - file: [docs/gateway/configuration-reference.md:719](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/docs/gateway/configuration-reference.md#L719)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Failure state is tracked on auth profiles, with model scoping only for rate limits:** `computeNextProfileUsageStats()` updates `errorCount`, `cooldownUntil`, and `disabledUntil` on the auth profile. `cooldownModel` is only populated when `reason === "rate_limit"`, so this is not a general provider/model circuit breaker.
  - file: [src/agents/auth-profiles/usage.ts:524](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/auth-profiles/usage.ts#L524)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Timeout failures are explicitly not persisted into failure state:** `maybeMarkAuthProfileFailure()` returns early when `reason === "timeout"`, so timeout/network-style failures from the report are not remembered across later runs.
  - file: [src/agents/pi-embedded-runner/run.ts:641](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/pi-embedded-runner/run.ts#L641)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Main already mitigates part of the problem via retry caps and fallback:** The failover docs and e2e tests show current behavior caps same-provider retries and can fall back immediately to another provider/model. That helps latency, but it is narrower than auto-disabling a failing model after repeated errors.
  - file: [src/agents/model-fallback.run-embedded.e2e.test.ts:754](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/model-fallback.run-embedded.e2e.test.ts#L754)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)

## Risks / Open Questions

- Some of the user pain is already partially addressed by current auth-profile cooldown and fallback behavior, so any future triage should distinguish that from the still-missing persistent per-model breaker.
- The issue text mentions 503s, network issues, and timeouts together; current main handles those classes differently, which makes a broad 'implemented' closure too risky.
