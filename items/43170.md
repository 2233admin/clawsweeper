## Summary

Keep open. Current main covers the Feishu configured-binding portion through the newer registry-backed conversation binding architecture, but the PR is not obsolete: `acp.defaultChannels` is still absent, catch-all ACP bindings are still rejected/skipped, and the bundled QQ Bot inbound path still does not resolve or initialize configured ACP bindings. The PR should be rebased or split rather than closed.

## Best Possible Solution

Keep this PR open only as stale-but-useful work to rebase or split. The best path is to port the useful pieces onto current main: add QQ Bot `bindings` provider support and inbound `resolveConfiguredBindingRoute`/`ensureConfiguredBindingRouteReady` handling through the current conversation-binding runtime SDK, add QQ Bot ACP binding docs/tests, and treat `acp.defaultChannels` plus catch-all bindings as a separate maintainer product/API decision because current schema and compiler intentionally require explicit `match.peer.id`.
