## Summary

Keep #64187 open. Current main still uses a plain rename-based SQLite atomic swap in memory-core, search/session/watch paths can still trigger that sync path, and the linked fix PR #71611 is open, unmerged, and not a safe superseding resolution yet.

## Best Possible Solution

Keep #64187 open as the canonical bug. A corrected fix should land in `extensions/memory-core/src/memory/manager-atomic-reindex.ts`, likely adapting the existing cron `renameWithRetry` pattern while preserving WAL/SHM suffix handling and unique backup paths, with focused tests for `EBUSY` retry and Windows-style fallback behavior before the issue closes.
