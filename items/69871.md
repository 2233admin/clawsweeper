## Summary

Current `main` still appears to route Z.AI image analysis through the generic media-understanding/OpenAI-compatible path, with no Z.AI-specific transport normalization in the image flow and no fix for preserving stale `baseUrl` values when the API surface stays `openai-completions`. The report looks actionable and should remain open.

## Evidence

- **Image tool execution path:** `runImagePrompt()` builds the media provider registry, resolves a provider, and then calls `describeImage`/`describeImages`; there is no dedicated Z.AI branch in the image tool itself.
  - file: [src/agents/tools/image-tool.ts:200](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/tools/image-tool.ts#L200)

  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Generic media runtime still drives image requests:** `resolveImageRuntime()` loads `models.json`, discovers the model from the Pi registry, and `describeImagesWithModelInternal()` ultimately calls `complete(model, context, ...)` on that discovered model. There is no Z.AI-specific image transport or endpoint override in this path.
  - file: [src/media-understanding/image.ts:130](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/media-understanding/image.ts#L130)
