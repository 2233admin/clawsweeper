## Summary

Current `main` still has the reported bridge-prune bug. `syncMemoryWikiBridgeSources` builds `activeKeys` from the current artifact list and then unconditionally prunes all prior `bridge` entries, so a transient `listActiveMemoryPublicArtifacts()` empty result would delete previously imported sources. The existing tests cover true source disappearance, but there is no regression test for capability loss with intact source files. The issue discussion also links an open fix PR, which further indicates this is not already resolved on `main`.

## Evidence

- **Unconditional prune remains on main:** `syncMemoryWikiBridgeSources()` reads public artifacts, builds `activeKeys` only from the current artifact list, and then always calls `pruneImportedSourceEntries()` with that set. If the plugin temporarily returns zero artifacts, `activeKeys` stays empty and the prune step still runs.
  - file: [extensions/memory-wiki/src/bridge.ts:224](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/extensions/memory-wiki/src/bridge.ts#L224)
  - command: `nl -ba extensions/memory-wiki/src/bridge.ts | sed -n '224,256p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Prune helper deletes every unmatched bridge entry:** `pruneImportedSourceEntries()` removes every state entry in the target group whose sync key is not in `activeKeys`; there is no guard for transient capability loss or for underlying source files still existing on disk.
  - file: [extensions/memory-wiki/src/source-sync-state.ts:98](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/extensions/memory-wiki/src/source-sync-state.ts#L98)

That matches intended stale-source pruning, but there is no companion test for the reported outage case where artifacts disappear while source files remain intact.
