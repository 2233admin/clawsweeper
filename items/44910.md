## Summary

Keep open. Current main still does not implement the specific #44910 fix: top-level FastAPI/Codex-style `{ "detail": "..." }` payloads are not recognized as raw API error payloads, so the reported JSON can still bypass formatting/suppression and be delivered as error text. The linked PR #45216 tracks a partial fix, but it remains open and is not present in the checked-out main.

## Evidence

- **Top-level detail payloads are not recognized:** `isErrorPayloadObject` only accepts `type === "error"`, request IDs, or a nested `error` object with `message`/`type`/`code`; it has no `record.detail` branch, so `{ "detail": "The 'gpt-5.3-codex' model is not supported..." }` returns null from `parseApiErrorPayload`.

  - sha: [b1b29a8fc210](https://github.com/openclaw/openclaw/commit/b1b29a8fc210a0772eb3728e03783890586e07f8)
- **Parsed API error info also ignores detail:** `parseApiErrorInfo` extracts only top-level `message` and nested `error.message`; it does not fall back to a top-level `detail` string, which is the shape called out in #44910 and in the linked PR discussion.
