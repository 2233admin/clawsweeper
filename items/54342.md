## Summary

Keep this issue open. Current main still blocks the central channel-less/headless ACP persistence path: `sessions_spawn(runtime:"acp", mode:"session")` still requires `thread:true`, the accepted spawn response still does not expose backend/agent ACP session identity, and `/hooks/agent` still runs through the isolated cron embedded-agent path rather than a hook-side ACP `runTurn` dispatch. Adjacent improvements exist for task tracking, `sessions_send`, and hook delivery suppression, but they do not close the reported workflow.

## Best Possible Solution

Keep this issue open and narrow it to a conservative headless ACP persistence contract. The likely product path is an explicit headless persistent ACP mode, or a narrow relaxation of the `mode:"session"` thread guard when `streamTo:"parent"` and cleanup semantics are defined; expose ACP backend/agent session identity through the spawn result or a documented status lookup; add regression coverage for channel-less hook-triggered ACP persistent spawns; and separately decide whether `/hooks/agent` should dispatch known ACP session keys to `acpManager.runTurn` or whether `sessions_send` should be the documented follow-up path.

## Risks / Open Questions

- Closing now would leave hook-triggered/headless ACP workflows without a supported persistent `mode:"session"` path for resume or multi-turn continuity.
- A future fix should not simply remove the thread guard; headless persistent ACP sessions need explicit lifecycle, cleanup, ownership, visibility, and timeout semantics.
- Exposing backend ACP resume identities or adding hook-to-ACP `runTurn` routing should preserve existing ACP policy checks and session visibility boundaries.
