## Summary

Keep #70894 open. Current `main` still routes `/hooks/agent` runs as isolated cron jobs and passes `forceNew: true` whenever `sessionTarget === "isolated"`; `resolveCronSession` only reuses an existing session when `forceNew` is not set. The linked PR #70895 is an open, unmerged proposed fix, so it is useful related work but not a reason to close the bug yet.

## Best Possible Solution

Keep #70894 open and review/merge an implementation like #70895: compute a `forceNewSession` value in `src/cron/isolated-agent/run.ts` that preserves fresh sessions for scheduled isolated cron runs but allows hook/webhook-sourced runs with stable, permitted `sessionKey` values to call `resolveCronSession` with `forceNew: false`. Add a regression test proving `/hooks/agent` or hook-provenance isolated runs reuse the session while ordinary isolated cron still forces a fresh run-scoped session.
