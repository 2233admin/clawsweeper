## Summary

#70894 describes a real, currently unresolved bug on main. `/hooks/agent` dispatch still creates an isolated cron job, and `runCronIsolatedAgentTurn` still passes `forceNew: true` for every isolated job, so `resolveCronSession` cannot reuse an existing session even when the hook session key policy accepts a stable `sessionKey`. The linked fix PR #70895 is open/unmerged, so this issue should remain open as the canonical bug until a fix lands.

## Best Possible Solution

Keep #70894 open until a fix lands on main. The best code path is the one the linked PR is aiming at: preserve forced fresh sessions for scheduled isolated cron jobs, but compute `forceNew` separately so hook/webhook-sourced isolated runs with an allowed stable `sessionKey` can reuse fresh session entries. Add regression coverage at the isolated-agent run layer and, ideally, a gateway hook test proving repeat `/hooks/agent` calls route to reusable session state.
