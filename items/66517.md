## Summary

Keep open: current `main` does not implement the proposed ratio-based `*Share` compaction-budget fields, and the PR discussion shows unresolved correctness concerns in the proposed wiring, so there is no strong basis to close it as already fixed or obsolete.

## Evidence

- **Current types expose only absolute compaction budgets:** On current `main`, `AgentCompactionConfig` includes `reserveTokens`, `keepRecentTokens`, and `reserveTokensFloor`, but no `reserveTokensShare`, `keepRecentTokensShare`, or `reserveTokensFloorShare` siblings.
  - file: [src/config/types.agent-defaults.ts:425](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/config/types.agent-defaults.ts#L425)
  - command: `nl -ba src/config/types.agent-defaults.ts | sed -n '380,450p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Config schema does not accept the proposed share fields:** The current Zod config schema defines `reserveTokens`, `keepRecentTokens`, `reserveTokensFloor`, and `maxHistoryShare`, but not the proposed `*Share` token-budget fields or `memoryFlush.softThresholdTokensShare`.
  - file: [src/config/zod-schema.agent-defaults.ts:169](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/config/zod-schema.agent-defaults.ts#L169)

- **Runtime reserve-floor resolution is still absolute-only:** `resolveCompactionReserveTokensFloor` on `main` reads only `cfg.agents.defaults.compaction.reserveTokensFloor` and otherwise falls back to `DEFAULT_PI_COMPACTION_RESERVE_TOKENS_FLOOR`; there is no share-based resolver in this file.
  - file: [src/agents/pi-settings.ts:43](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/agents/pi-settings.ts#L43)
