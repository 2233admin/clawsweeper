## Summary

Keep open. Current main still applies sender-scoped Telegram group auth to `channel_post` using a synthetic sender/channel identity, but the docs do not explicitly define that `groups[chatId].allowFrom` is unsupported or ignored on channel posts. That leaves the authorization semantics the issue reports still unclear.

## Evidence

- **`channel_post` still uses synthetic non-human sender identity:** The channel-post handler rewrites the update into a synthetic group message and sets `senderId` from `post.sender_chat.id`, else `post.from.id`, else the chat id. For channel posts this is not documented as a human sender id path, so per-chat sender allowlists are evaluated against channel/chat identity instead of a real human origin.
