## Summary

Keep open: current `main` still sends remote embedding requests without `encoding_format`, so this PR addresses behavior that is not implemented on `main` and the report is not obsolete.

## Evidence

- **Current outbound embedding request still omits `encoding_format`:** On current `main`, the remote embedding provider posts `{ model: client.model, input }` and does not include `encoding_format`. This is the exact call site the PR proposes to change, just moved to the newer `memory-host-sdk` path.
  - file: [src/memory-host-sdk/host/embeddings-remote-provider.ts:30](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/memory-host-sdk/host/embeddings-remote-provider.ts#L30)
  - command: `nl -ba src/memory-host-sdk/host/embeddings-remote-provider.ts | sed -n '1,120p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **OpenClaw already treats `float` as the default embeddings encoding:** The OpenAI-compatible gateway handler on current `main` normalizes requests so anything other than explicit `base64` is treated as `float`, and it returns numeric arrays in that mode. That makes the PR's proposed `encoding_format: "float"` consistent with existing behavior elsewhere in the codebase.
  - file: [src/gateway/embeddings-http.ts:311](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/gateway/embeddings-http.ts#L311)
  - command: `nl -ba src/gateway/embeddings-http.ts | sed -n '300,340p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Remote fetch parser expects float vectors, not base64 payloads:** The remote embedding fetch path parses `data[].embedding` as `number[]`. That matches the PR discussion and Greptile review that hard-coding `encoding_format: "float"` is internally consistent with current parsing.
  - file: [src/memory-host-sdk/host/embeddings-remote-fetch.ts:19](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/memory-host-sdk/host/embeddings-remote-fetch.ts#L19)

- **Current tests do not disprove the bug:** The existing remote-fetch test only asserts request bodies containing `input` and does not cover the provider layer adding `model` or `encoding_format`, so there is no current-main test evidence that the reported compatibility issue is obsolete.
  - file: [src/memory-host-sdk/host/embeddings-remote-fetch.test.ts:29](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/memory-host-sdk/host/embeddings-remote-fetch.test.ts#L29)
