## Summary

Keep #70149 open. Current main still routes the WhatsApp WebSocket login through `proxy-agent` via `resolveAmbientNodeProxyAgent()`, which is the specific path the reporter says times out; the existing regression coverage only proves an env proxy creates some agent, not that the explicit `HttpsProxyAgent`-working case is fixed. The related #61825 closure covers the broader missing-env-proxy issue, but this narrower proxy-agent-vs-HttpsProxyAgent failure is not safely implemented or superseded on main.

## Best Possible Solution

Keep the issue open and fix the WhatsApp plugin narrowly: resolve env/configured HTTP(S) proxy URLs into a Node `https.Agent` compatible with `ws` for Baileys' `agent` path, likely `HttpsProxyAgent`, while keeping undici dispatchers only for fetch/media paths. Add focused tests that assert the WebSocket agent constructor/path and prevent `proxy-agent` from being used for the reported WSS login case; then update WhatsApp docs/config if an explicit `channels.whatsapp.proxy` escape hatch is added. The WhatsApp proxy portion of #46356 can be mined, but should be split or narrowed before it becomes canonical.
