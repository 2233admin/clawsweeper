## Summary

Keep #64500 open. Current main still has the reported pair-awareness gap: ping-pong detection only reaches critical when both sides have stable no-progress result hashes, the global breaker counts the current tool/signature, and enforcement blocks only the current attempted call rather than both tools in the pair.

## Best Possible Solution

Keep the issue open. A maintainer fix should make tool-loop protection pair-aware: allow sustained A/B alternation to escalate without relying solely on identical result hashes, add a pair-scoped threshold before or alongside the current per-tool global breaker, and persist a short-lived session-level block for both tool signatures when a critical ping-pong or pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get alternation and for blocking both sides of the pair.
