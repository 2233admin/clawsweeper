## Summary

Keep open. Current `main` documents the tokenless Tailscale Serve trust model and lets operators disable it with `gateway.auth.allowTailscale: false`, but it does not implement optional secondary token/password auth on top of verified Tailscale identity, and there is no dedicated runtime warning for that mode.

## Evidence

- **No secondary-auth branch in gateway auth flow:** `authorizeGatewayConnectCore` accepts verified Tailscale Serve identity for `ws-control-ui` when `allowTailscale` is enabled and no explicit shared secret was supplied. After that it returns `method: "tailscale"`; there is no path that requires both Tailscale identity and token/password.
  - file: [src/gateway/auth.ts:461](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/auth.ts#L461)
  - command: `nl -ba src/gateway/auth.ts | sed -n '380,500p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Proposed trusted-proxy-plus-token model is not supported:** `trusted-proxy` mode rejects a configured shared token as mutually exclusive, so the issue's example of layering shared-secret auth on top of a trusted identity mode is not implemented.
  - file: [src/gateway/auth.ts:240](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/auth.ts#L240)

  - file: [src/config/types.gateway.ts:144](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/config/types.gateway.ts#L144)
