## Summary

Keep open: current `main` still does not implement a vi/vim mode for the TUI input editor, and the linked PR for this feature was closed without merging because the author concluded the correct implementation belongs upstream in `@mariozechner/pi-tui` rather than as an OpenClaw-side key interception layer.

## Evidence

- **Current editor wrapper only adds bespoke shortcuts, not a modal vi layer:** `CustomEditor.handleInput()` intercepts Alt/Ctrl/Escape keys and otherwise falls through to `super.handleInput(data)`. There is no vi state, normal/insert mode, or vi command parser on `main`.
  - file: [src/tui/components/custom-editor.ts:16](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/tui/components/custom-editor.ts#L16)
  - command: `nl -ba src/tui/components/custom-editor.ts | sed -n '1,140p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **TUI construction does not wire any vi-mode option:** `runTui()` instantiates `new CustomEditor(tui, editorTheme)` with no config or mode flag, so current `main` has no path to enable vi bindings.
  - file: [src/tui/tui.ts:536](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/tui/tui.ts#L536)
