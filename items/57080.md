## Summary

Current `main` fixes the underlying `status` multimodal-memory regression through capability-based memory embedding provider resolution, so this PR's CLI preload change is no longer needed.

## Close Comment

This looks obsolete on current `main`; Codex review found the underlying regression is already fixed.

- `status` still reaches memory config resolution through `src/commands/status.command.ts` -> `src/commands/status.scan.ts` -> `src/commands/status.scan-memory.ts`, so the original failure path still matters.
- The fix on `main` is different from this PR: `src/agents/memory-search.ts` now resolves embedding adapters through `src/plugins/memory-embedding-providers.ts`, which falls back to declared plugin capability providers via `src/plugins/capability-provider-runtime.ts` instead of requiring preloaded runtime plugins.
- There is explicit regression coverage in `src/agents/memory-search.test.ts` asserting Gemini multimodal memory works even when the runtime registry has not registered Gemini yet.
- Current CLI policy still keeps `status` off the proposed preload change (`src/cli/command-catalog.ts` and related tests), which is consistent with the newer capability-based fix making the preload unnecessary.
- The implementation proving this is on `main` in commit `68315792677f9628d1370d6869366442f1d3ecad`.
