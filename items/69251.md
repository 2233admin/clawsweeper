## Summary

Keep #69251 open. Current main still uses the reported POSIX `/bin/sh -lc` node shell wrapper, still sends that wrapper through `system.run.prepare`, and still fail-closes shell payloads whose first token resolves to a mutable/script-like file. The related #69308 discussion is useful context, but it only addressed non-interactive `SHELL` fallback and did not remove or compensate for this approval-binding path.

## Best Possible Solution

Keep the issue open and fix the node exec approval path directly. The likely product/code path is to make `system.run.prepare` produce a safe node-local binding for script/interpreter file operands hidden behind the `/bin/sh -lc` wrapper, or to make an explicit, well-tested trust-mode exception for `security=full` plus `ask=off`. Add regression tests for `/bin/sh -lc /tmp/script.sh`, `/bin/zsh /path/script.sh`, and `python3 /path/script.py`, while preserving the existing fail-closed cases for ambiguous mutable shell payloads.
