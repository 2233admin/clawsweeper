## Summary

Current main does not support the reported GBK/UTF-8-mix root-cause claim: OpenClaw’s transcript entrypoints explicitly write UTF-8, the current QQ Bot gateway decodes inbound frames as UTF-8, and transcript tests read JSONL back as UTF-8. But this report was filed against v2026.3.23, I could not verify that exact historical code path from the shallow checkout, and a real corruption bug could still exist upstream of transcript persistence. That is not strong enough evidence for an auto-close.

## Evidence

- **Current transcript helper forces UTF-8 on JSONL header creation:** `ensureSessionHeader()` writes the session header with `encoding: "utf-8"` before transcript appends proceed through `SessionManager`.
  - file: [src/config/sessions/transcript.ts:33](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/transcript.ts#L33)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Gateway transcript creation paths also force UTF-8:** The gateway chat/session transcript bootstrap paths both create JSONL headers with explicit `encoding: "utf-8"`.
  - file: [src/gateway/server-methods/chat.ts:1502](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/server-methods/chat.ts#L1502)

  - file: [extensions/qqbot/src/engine/gateway/codec.ts:16](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/qqbot/src/engine/gateway/codec.ts#L16)
