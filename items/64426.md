## Summary

Keep this open. Current main still only has opt-in generic tool-loop detection keyed on exact tool+params hashes; the first-class read wrapper and embedded read event path do not track same-turn path rereads, same-small-file-set reread ratios, or edit/write/test progress-aware suppression. The related #63352 remains open and exec-scoped, so it is complementary rather than a superseding canonical item.

## Best Possible Solution

Keep this open and implement a read-specific warning layer on the existing tool-loop detection path: normalize first-class read path identities per run/session, detect repeated same-file and same-small-file-set rereads within one turn, reset or suppress after edit/write/test or broadened-context progress, emit a model-visible soft nudge plus diagnostic event, and add focused tests and docs/config guidance.

## Risks / Open Questions

- The existing opt-in generic detector partially overlaps exact duplicate read calls, so a future fix should extend or coordinate with that path rather than introducing conflicting loop state.
- Read-reread heuristics can false-positive during legitimate debugging, so the implementation should stay warning-only at first and suppress or reset after clear progress such as edits, writes, tests, or broadened file context.
