## Summary

Keep #72080 open. Current main still has a plausible source-backed false-completion path for long-running ACP children: the bundled acpx backend defaults to a 120s prompt timeout, upstream acpx@0.6.1 can return a completed turn from observed session output after that timeout, OpenClaw marks the ACP task succeeded as soon as runTurn returns, and the parent delivery chain turns that into `Background task done`. Current transcript persistence also writes ACP dispatch text as `provider: "openclaw"`, `model: "acp-runtime"`, zero usage, and `stopReason: "stop"`, matching the report’s proxy-log shape rather than proving harness completion.

## Best Possible Solution

Keep the issue open and fix the ACP task lifecycle. The durable fix should distinguish preview/session-output from true terminal completion: avoid treating acpx timeout-with-observed-output as success for parent task delivery, or move OpenClaw to a runtime result/status path that waits for explicit ACP completion or confirms the harness is no longer active before emitting lifecycle `end`. Add a regression test that simulates an ACP child emitting initial text, exceeding the 120s timeout while still active, and then completing later; `sessions_yield` should wake only on the real terminal event.
