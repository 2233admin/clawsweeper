## Summary

Current `main` still misses `stream: "compaction"` emission in the embedded runner's overflow/timeout recovery path, so the underlying bug remains real. This PR should stay open rather than be auto-closed, because its proposed end-event payload diverges from the canonical compaction contract and would cause at least one current consumer to treat failed recovery compactions as completed.

## Evidence

- **Recovery path on main lacks compaction events:** `runOwnsCompactionBeforeHook` and `runOwnsCompactionAfterHook` only run plugin hooks around engine-owned timeout/overflow compaction. They do not call `emitAgentEvent()` or `params.onAgentEvent()`, so subscribers never see compaction lifecycle events in this path.
  - file: [src/agents/pi-embedded-runner/run.ts:705](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/pi-embedded-runner/run.ts#L705)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '690,760p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Canonical handler defines `completed` from actual result presence:** The normal subscribe-based compaction handler emits start/end events and computes `completed` as `hasResult && !wasAborted`, where `hasResult = evt.result != null`. That is the existing event contract other consumers already rely on.
  - file: [src/agents/pi-embedded-subscribe.handlers.compaction.ts:42](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/agents/pi-embedded-subscribe.handlers.compaction.ts#L42)
  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.compaction.ts | sed -n '1,220p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **The compaction result type allows `result` to be absent:** `CompactResult.result` is optional in the shared context-engine type, so a recovery-path implementation that derives `completed` from only `ok && compacted` can disagree with the canonical handler when an engine returns success without a populated `result`.
  - file: [src/context-engine/types.ts:15](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/context-engine/types.ts#L15)
  - command: `nl -ba src/context-engine/types.ts | sed -n '1,80p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **At least one current consumer still treats any end event as success:** `agent-runner-memory.ts` sets `memoryCompactionCompleted = true` on any `phase === "end"` compaction event, then increments session compaction count. The PR's new failed-recovery end events would therefore skew session accounting unless the consumer or event contract is tightened.
  - file: [src/auto-reply/reply/agent-runner-memory.ts:782](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/auto-reply/reply/agent-runner-memory.ts#L782)

- **PR discussion already surfaced the same contract risk:** The review thread on PR #67706 flagged both problems: `completed` in the patch diverges from `handleCompactionEnd`, and downstream code is not uniformly checking `completed` before treating `phase: "end"` as successful compaction.
