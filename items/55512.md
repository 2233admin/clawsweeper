## Summary

Keep open. Current `main` already converts Discord markdown tables before sending, so that half of the report is stale, but the shared code-table renderer still computes widths with `cell.text.length` and pads with that same count. That remains incompatible with full-width CJK characters, so the Telegram misalignment report is still valid and the issue is not fully implemented on `main`.

## Evidence

- **Shared table renderer still uses code-unit length for column widths:** `renderTableAsCode` derives each column width from `cell?.text.length` and pads rows with `widths[i] - (cell?.text.length ?? 0)`, which does not account for double-width CJK glyphs.
  - file: [src/markdown/ir.ts:509](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/markdown/ir.ts#L509)
  - command: `nl -ba src/markdown/ir.ts | sed -n '500,570p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Telegram send path uses the shared markdown table renderer:** Telegram resolves a table mode and passes it into `renderTelegramHtmlText`, which in turn calls `markdownToIR(..., { tableMode })`; Telegram therefore inherits the width bug from the shared renderer.
  - file: [extensions/telegram/src/send.ts:659](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/telegram/src/send.ts#L659)

  - file: [extensions/telegram/src/format.ts:75](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/telegram/src/format.ts#L75)
