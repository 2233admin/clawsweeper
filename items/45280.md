## Summary

Current `main` does not implement configurable per-call session isolation for the `voice-call` plugin. Classic auto-response still keys sessions by normalized caller phone number, realtime consult also falls back to phone-number scope when no explicit `call.sessionKey` is present, inbound call records do not persist a per-call session key, and the plugin config/docs expose no `sessionScope` option.

## Evidence

- **Classic auto-response still phone-scoped:** `generateVoiceResponse` normalizes `from` and hardcodes `sessionKey = \`voice:${normalizedPhone}\``; the `callId` parameter is present but not used for session routing.
  - file: [extensions/voice-call/src/response-generator.ts:197](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/voice-call/src/response-generator.ts#L197)

  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Realtime consult still phone-scoped by default:** `resolveVoiceCallConsultSessionKey` returns `call.sessionKey` only if already set; otherwise it derives `voice:${normalizedPhone}` and only falls back to `voice:${call.callId}` when no phone is available.
  - file: [extensions/voice-call/src/runtime.ts:87](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/voice-call/src/runtime.ts#L87)

## Risks / Open Questions

- Users can still leak conversation context across separate calls from the same phone number, especially for receptionist/booking/IVR use cases where each call should start clean.
