## Summary

Keep open. The current `main` branch still uses the same bundled SSE client path and 30s connection-timeout wrapper described in #70753, and I found no post-release fix on `main` for hosted HTTP/SSE timeout behavior.

## Evidence

- **Default remote HTTP MCP transport is still SSE over the bundled SDK client:** `resolveMcpTransport()` still defaults HTTP MCP servers to `SSEClientTransport`, wiring both request fetch and EventSource fetch through the bundled undici-backed fetch path. There is no new fallback or HTTP/2-specific handling on `main`.
  - file: [src/agents/mcp-transport.ts:78](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/agents/mcp-transport.ts#L78)
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **The runtime still matches the reported 30s startup-timeout symptom:** `connectWithTimeout()` still rejects with `MCP server connection timed out after ${timeoutMs}ms`, and session startup still calls it before listing tools for each configured bundle-MCP server. That matches the issue's reported gateway-boot failure mode.
