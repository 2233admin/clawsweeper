## Summary

Keep open. Current `main` does not implement `approvals.timeoutMs` or `OPENCLAW_EXEC_APPROVAL_TIMEOUT_MS`; it still uses a fixed exec-approval timeout, and the PR discussion already identified correctness gaps in the proposed implementation.

## Evidence

- **Current main still uses a fixed default timeout:** `DEFAULT_EXEC_APPROVAL_TIMEOUT_MS` is hardcoded to `1_800_000` and there is no config/env resolver in this file on current `main`. That means the requested configurable timeout is not implemented on `main`, even though the default is now much longer than the PR's original 120s baseline.
  - file: [src/infra/exec-approvals.ts:167](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/infra/exec-approvals.ts#L167)
  - command: `nl -ba src/infra/exec-approvals.ts | sed -n '160,176p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Gateway handler still falls back only to the constant:** The exec-approval RPC handler resolves `timeoutMs` from request params or `DEFAULT_EXEC_APPROVAL_TIMEOUT_MS`; it does not consult config. So `approvals.timeoutMs` is not already implemented server-side on `main`.
  - file: [src/gateway/server-methods/exec-approval.ts:146](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/gateway/server-methods/exec-approval.ts#L146)
