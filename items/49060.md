## Summary

Keep open. Current `main` still appears vulnerable to stdout protocol corruption in ACP mode: `serveAcpGateway()` wires ACP JSON-RPC over `process.stdout`/`process.stdin` but never calls `routeLogsToStderr()`, while the logging subsystem still defaults non-error console output to stdout unless that flag is set.

## Evidence

- **ACP server still writes protocol on stdout without stderr log routing:** `src/acp/server.ts:15` enters `serveAcpGateway()` and immediately loads config/bootstrap logic; `src/acp/server.ts:112` then binds `ndJsonStream()` to `process.stdout`/`process.stdin`. There is no `routeLogsToStderr()` import or call anywhere in this file.
  - file: [src/acp/server.ts:15](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/acp/server.ts#L15)
  - command: `sed -n '1,220p' src/acp/server.ts`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Logging still goes to stdout by default unless forced to stderr:** `src/logging/console.ts:116` shows `routeLogsToStderr()` only flips `loggingState.forceConsoleToStderr = true`. `src/logging/subsystem.ts:259` sends logs to stderr only when that flag is set or the level is error/fatal; otherwise `src/logging/subsystem.ts:264` uses `console.log`, which is stdout.
  - file: [src/logging/subsystem.ts:259](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/logging/subsystem.ts#L259)
  - command: `nl -ba src/logging/subsystem.ts | sed -n '252,266p'`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Existing stdio-safe entry points already apply the guard:** `src/mcp/tools-stdio-server.ts:26` explicitly calls `routeLogsToStderr()` before stdio transport setup, and `src/cli/completion-cli.ts:94` does the same before emitting completion output. ACP lacks the same protection.
  - file: [src/mcp/tools-stdio-server.ts:26](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/mcp/tools-stdio-server.ts#L26)
  - command: `sed -n '1,180p' src/mcp/tools-stdio-server.ts`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **CLI startup does not appear to cover `openclaw acp` automatically:** `src/cli/command-execution-startup.ts:36` routes logs to stderr only when `startupPolicy.suppressDoctorStdout` is true. `src/cli/command-startup-policy.ts:47` sets that from JSON output mode, and `src/cli/acp-cli.ts:12` defines `acp` without a `--json` output mode. So ACP server startup does not inherit the generic safeguard.
  - file: [src/cli/command-execution-startup.ts:36](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/cli/command-execution-startup.ts#L36)
  - command: `sed -n '1,220p' src/cli/command-execution-startup.ts`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **No current regression test covers ACP stderr routing:** `src/acp/server.startup.test.ts` exercises gateway startup ordering and auth forwarding, but there is no assertion that `serveAcpGateway()` calls `routeLogsToStderr()` or keeps stdout protocol-only.
  - file: [src/acp/server.startup.test.ts:121](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/acp/server.startup.test.ts#L121)
  - command: `sed -n '1,320p' src/acp/server.startup.test.ts`
