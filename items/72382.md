## Summary

Keep this issue open. Current main still permits the reported installer outcome: the Unix installer can warn that the npm global bin directory is missing from the user's original PATH, then print a successful install banner while post-install verification remains opt-in. The existing non-root smoke test also adds the npm global bin directory to PATH after running the installer, so it does not cover the reporter's immediate `openclaw: command not found` path.

## Best Possible Solution

Fix the Unix installer so default installer success is contingent on `openclaw` being discoverable for the user's next command, or change the success path to clearly stop short when the original shell PATH is missing the npm global bin directory. Add focused installer coverage that verifies npm global prefix/PATH behavior without a post-test PATH export, and update the public install troubleshooting docs if the intended workflow requires opening a new shell or sourcing shell startup files.
