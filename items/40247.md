## Summary

Keep this PR open. Current main still returns only the requested non-loopback bind host, while local gateway callers and tests still target 127.0.0.1. The PR remains a focused implementation candidate; #50751 is an overlapping open alternative, not a shipped superseding fix.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either rebase and land the server-side dual-bind fix with the PR's tailnet/custom regression tests plus docs/security expectation updates, or explicitly choose the #50751 client-side host-resolution approach and close this PR only after a shipped fix demonstrably covers internal calls, CLI probes, and sessions_spawn for gateway.bind=tailnet and gateway.bind=custom.

## Risks / Open Questions

- Dual-binding would intentionally change the documented direct-tailnet behavior where loopback does not work, so docs and operator/security expectations should be updated if this path lands.
- Adding a loopback listener for non-loopback configurations is local-only, but it does let local processes reach the gateway in tailnet/custom mode; maintainers should review that against gateway auth and pairing assumptions before merge.
- #50751 overlaps with a client-side host-resolution approach. Maintainers should choose one coherent gateway target/listener strategy or coordinate both to avoid inconsistent semantics.
