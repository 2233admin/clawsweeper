## Summary

Keep open. Current `main` still contains the reported Discord model-picker mismatch path, and the proposed PR is not on `main` and has unresolved correctness problems called out in review.

## Evidence

- **Current main still rereads the picker route session key and can emit the misleading warning:** `handleDiscordModelPickerInteraction` still sleeps, rereads `resolveDiscordModelPickerCurrentModel` from `route.sessionKey`, logs `model picker override mismatch`, and sends either `✅ Model set...` or `⚠️ Tried to set...` based on that reread. This is the exact behavior the PR says is wrong.
  - file: [extensions/discord/src/monitor/native-command-ui.ts:894](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/monitor/native-command-ui.ts#L894)

- **Current tests already encode the bound-thread false-mismatch scenario:** There is a current `main` test named `verifies model state against the bound thread session` that sets a bound thread target session key `agent:worker:subagent:bound` and asserts the mismatch log is emitted. That is strong evidence the reported bug still exists on `main`, not just in the PR description.
  - file: [extensions/discord/src/monitor/native-command.model-picker.test.ts:578](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/monitor/native-command.model-picker.test.ts#L578)

- **The proposed unconditional success confirmation is not proven safe by the dispatcher contract:** `DispatchDiscordCommandInteraction` returns `Promise<void>`, not a success/result type. In `dispatchDiscordCommandInteraction`, multiple guard paths for disabled channels, disallowed channels, DM policy, and unauthorized users call `respond(...)` and `return` without throwing. A caller that treats any resolved dispatch as success can show a false positive confirmation.
