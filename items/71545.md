## Summary

Keep open. Current main still uses a cached MCP session/client for `callTool` with no streamable-http lost-session detection, reconnect, metadata refresh, or retry path. The PR is recent, focused, and appears to address a real core MCP runtime gap rather than duplicating a shipped fix or ClawHub/plugin-scope request.

## Best Possible Solution

Keep the PR open for maintainer review. The likely best path is to land a hardened streamable-http reconnect implementation in `src/agents/pi-bundle-mcp-runtime.ts` that rebuilds the transport/client with the same JSON schema validator, shares one in-flight reconnect per server, retries the original tool call once, preserves retryability after failed reconnects, and keeps SSE/stdio behavior unchanged, with the PR's streamable-http regression tests covering those cases.
