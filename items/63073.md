## Summary

Keep open. Current main still hardcodes MiniMax usage polling to the CN host and does not pass configured MiniMax baseUrl or region context into usage fetching. This PR addresses a real remaining bug, but it overlaps with open #65054/#65059 and still needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review or consolidation with #65059. The best fix is a MiniMax-owned, region-aware usage endpoint resolver that preserves `GET /v1/api/openplatform/coding_plan/remains`, derives global/CN routing from the same credential/config sources used for MiniMax auth (`minimax`, `minimax-portal`, and `MINIMAX_API_HOST` where appropriate), defines the safe default host, applies an appropriate token-routing trust policy, updates the MiniMax docs, and adds regression tests for URL, method, global, CN, invalid/custom baseUrl, and API-key/OAuth configurations.

## Risks / Open Questions

- Merging this PR as-is may change default MiniMax usage polling from the documented/current CN host to the global host; #65059 instead states a CN fallback for backward compatibility, so default behavior needs maintainer judgment.
- The PR routes a usage bearer token to an origin derived from config/env through `fetchJson`; maintainers should decide whether this must be constrained to trusted MiniMax global/CN hosts or use the provider HTTP/SSRF policy path.
- There is overlapping open work in #65054/#65059, so leaving both PRs open duplicates review until a maintainer chooses one implementation or asks contributors to consolidate.
- No tests were run because this cleanup review was explicitly read-only; the decision is based on source, docs, tests, PR diff, and public issue/PR state inspection.
