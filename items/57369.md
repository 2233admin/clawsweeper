## Summary

Keep #57369 open. Current main still models session reset policy as only `daily` or `idle`, validates config with the same two literals, and documents only those modes. `resetByType` and `resetByChannel` reuse that same config shape, so the requested manual-only `mode: "never"` remains a real core session-config feature request rather than a duplicate, ClawHub item, or stale low-signal report.

## Best Possible Solution

Add a first-class `session.reset.mode: "never"` to the core session reset contract: update `SessionResetMode`, Zod/generated schema, config help, docs, and policy evaluation so automatic daily/idle expiry is disabled while explicit `/new`, `/reset`, and gateway `sessions.reset` still work. Cover base `session.reset`, `resetByType`, and `resetByChannel` with focused tests.
