## Summary

Current `main` already suppresses typing for silent `NO_REPLY` paths well enough to cover the reported Discord hold-pattern case, especially for unmentioned group listeners and silent partials.

## Evidence

- **Group-chat default now avoids eager typing for non-mentioned listeners:** `resolveTypingMode()` defaults group chats without a mention to `message`, while direct chats and mentioned group chats stay `instant`. That means sidecar agents in a multi-agent channel do not start typing at run start unless they emit renderable text.

## Close Comment

Closing as implemented on current `main` after a Codex review. OpenClaw now defaults unmentioned group listeners to `typingMode: "message"`, and that mode only starts typing on the first non-silent text delta instead of at run start (`src/auto-reply/reply/typing-mode.ts:16`, `src/auto-reply/reply/typing-mode.ts:70`). The typing controller also explicitly skips exact `NO_REPLY` and streamed `NO_REPLY` prefix fragments before starting typing (`src/auto-reply/reply/typing.ts:181`). Regression coverage matches this report: silent `NO_REPLY` partials and silent tool results do not forward and do not trigger typing (`src/auto-reply/reply/agent-runner.runreplyagent.e2e.test.ts:294`, `src/auto-reply/reply/agent-runner.runreplyagent.e2e.test.ts:497`). The changelog also records the shipped Discord-side cleanup for silent/NO_REPLY runs in `v2026.3.7` (`CHANGELOG.md:3083`). If this is still reproducible on a current release, please open a new issue with the exact channel/plugin config and a minimal repro.
