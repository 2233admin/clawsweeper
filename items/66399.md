## Summary

Current main still does not implement the central supervisor-level SIGTERM-first cancellation policy. The exec and CLI-agent paths still run through `ProcessSupervisor`, whose timeout/manual cancellation path calls `adapter.kill("SIGKILL")` directly; tests continue to assert immediate SIGKILL for no-output timeout, scoped replacement, and overall timeout. Main has adjacent safeguards in `killProcessTree`, adapter wait fallbacks, and session-lock cleanup, but those do not close this issue because the direct supervised child still gets hard-killed before it can run SIGTERM cleanup.

## Best Possible Solution

Keep this issue open. Implement a narrow supervisor-level cancellation change: for manual cancellation, scoped replacement, overall timeout, and no-output timeout, send SIGTERM first, start a bounded unref'ed grace timer, then escalate to SIGKILL if the run has not settled. Clear the escalation timer on settle/dispose, preserve the existing child/PTY SIGKILL wait fallbacks and `killProcessTree` safeguards, and update supervisor/adapter tests to assert SIGTERM-first behavior plus hard-kill escalation. Track #71710 separately for whether the agent process responds correctly to SIGTERM.

- The related #71710 agent-side SIGTERM handling problem may still need separate work after supervisor escalation lands; this issue should not be closed until the supervisor policy itself changes.
