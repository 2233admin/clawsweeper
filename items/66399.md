## Summary

Keep open. Current `main` still cancels timed-out supervisor runs by calling `adapter.kill("SIGKILL")`, and the child adapter still immediately `child.kill("SIGKILL")`s the tracked process. The repo now has a cross-platform post-kill wait fallback, so the drain-timeout part is at least partially addressed, but the graceful SIGTERM-first escalation requested by this issue is not implemented end-to-end.

## Evidence

## Risks / Open Questions

- The issue body's drain-timeout claim is partially outdated against current `main`: both child and PTY adapters now schedule a post-SIGKILL wait fallback. If maintainers retitle/refine the issue, the missing behavior is specifically graceful supervisor-driven escalation for the main exec process.
- Windows signal semantics remain platform-specific, so a future fix should be validated separately for plain child processes and PTY-backed runs.
