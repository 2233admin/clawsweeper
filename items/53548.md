## Summary

Current main still couples explicit `mode="session"` to `thread=true` in both native subagent and ACP spawn paths. Tests and docs still assert that contract, and the shipped #67790 work only improved diagnostics for unavailable thread bindings rather than implementing headless persistent sessions. The open #39372 PR remains a plausible implementation candidate, but the issue should stay open until a safe implementation lands or maintainers make an explicit product decision to keep the current contract.

## Best Possible Solution

Keep this issue open. The best implementation path is to decouple persistence from thread binding in `src/agents/subagent-spawn.ts` and `src/agents/acp-spawn.ts`, allow `mode="session"` without `thread`, create persistent native and ACP child sessions, return stable `childSessionKey` and label metadata for `sessions_send`, deliver bootstrap/completion output through the parent or originating chat when no thread binding exists, preserve current `thread=true` behavior, and update regression tests plus the public ACP/subagents docs. PR #39372 is the current implementation candidate, but it needs maintainer-quality ACP delivery and lifecycle handling before merge.

## Risks / Open Questions

- A fix should not simply delete the guards; native subagent and ACP lifecycle, cleanup, routing, completion delivery, and `sessions_send` follow-up semantics need regression coverage.
- ACP headless persistent sessions need careful delivery behavior so child output does not leak into the wrong requester conversation when no thread binding exists.
- Existing `mode="run"` plus `sessions_send` mitigates some native subagent workflows, but it does not satisfy explicit ACP persistent-session semantics and does not match the documented `mode="session"` contract requested here.
