## Summary

Keep open. Current main still performs auto-TTS inline before reply delivery: the dispatch paths await TTS payload generation before enqueueing/routing replies, and the speech-core implementation awaits provider synthesis before returning a payload with `mediaUrl`. The config/docs expose `messages.tts.mode` (`final`/`all`) but no async/background/non-blocking delivery mode. Related #42539 is adjacent Telegram delivery UX, not a canonical tracker for this queue-blocking async TTS request.

## Best Possible Solution

Keep the issue open and implement this in the core auto-reply/TTS delivery path rather than as a plugin-only workaround: add a `messages.tts.delivery`-style mode or equivalent policy, deliver visible text immediately, enqueue bounded background TTS jobs per session/channel, and route follow-up voice notes or edits through existing channel delivery/message-action seams. Add focused tests for Telegram block/final replies proving that text delivery and same-session inbound processing do not wait for slow TTS synthesis.
