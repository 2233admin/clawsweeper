## Summary

Current main no longer reproduces the reported 30s restart delay: gateway startup always supplies the gateway port to the lock code, and `acquireGatewayLock` now treats a free configured port as a dead owner immediately, with a darwin regression test covering the recycled/alive-PID case.

## Evidence

## Risks / Open Questions

- `src/infra/gateway-lock.ts` still does not register an immediate process-exit cleanup handler, so the lock file itself can still persist after an early crash; the reason to close here is that the reported 30s restart delay no longer reproduces on current main because the free-port check reclaims it immediately in the real gateway startup path.

## Close Comment

Thanks for the report. I reviewed current `main` with Codex and I can't reproduce the reported 30s restart delay anymore.

- `src/cli/gateway-cli/run.ts:656` passes the actual gateway port into `runGatewayLoop`, and `src/cli/gateway-cli/run-loop.ts:43` acquires the lock with that port.
- `src/infra/gateway-lock.ts:193` now checks whether that configured port is free before relying on PID/start-time heuristics. If the crashed gateway never bound the port, the stale lock owner is treated as dead immediately and the lock is reclaimed.
- `src/infra/gateway-lock.test.ts:249` has a regression test for the exact problematic shape: owner PID appears alive, platform is `darwin`, but the configured port is free, and lock acquisition succeeds.
- `src/gateway/server/http-listen.ts:25` and `docs/gateway/gateway-lock.md:17` show the runtime singleton guard is the socket bind / `EADDRINUSE` path now.

Because the reported restart delay no longer matches current `main`, I'm closing this as obsolete. If you can still reproduce a delayed gateway restart on a newer build, please open a fresh issue with exact restart steps, platform, and logs.
