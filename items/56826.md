## Summary

Current main already avoids the reported embedded WebSocket setup path for `openai-codex`: Codex traffic is normalized onto the ChatGPT backend HTTP transport, and the embedded runner now explicitly refuses the OpenAI WebSocket transport for Codex models.

## Evidence

- **Embedded runner blocks Codex WebSocket selection:** `shouldUseOpenAIWebSocketTransport()` only returns true for `provider === "openai"` with `modelApi === "openai-responses"`. The inline comment explicitly says `openai-codex` stays on the ChatGPT backend HTTP path instead of the public OpenAI Responses websocket endpoint.

## Close Comment

Closing as implemented on `main` after a Codex review.

- `src/agents/pi-embedded-runner/run/attempt.thread-helpers.ts:41` now refuses the embedded WebSocket transport unless the pair is exactly `provider=openai` plus `api=openai-responses`; the inline comment explicitly says `openai-codex` stays on the ChatGPT backend HTTP path.
- `src/agents/pi-embedded-runner/run/attempt.spawn-workspace.websocket.test.ts:53` covers the regression and asserts `shouldUseOpenAIWebSocketTransport(...)` is `false` for both `openai-codex/openai-responses` and `openai-codex/openai-codex-responses`.
- `extensions/openai/openai-codex-provider.ts:44` and `extensions/openai/openai-codex-provider.ts:129` normalize Codex traffic onto `openai-codex-responses` at `https://chatgpt.com/backend-api/codex`, so current OpenClaw no longer sends Codex embedded runs through the public OpenAI websocket endpoint.
- This behavior is already present in release `v2026.4.23` (`9b18ae877018a34933e27bb7b2986cb97e9602aa`).
