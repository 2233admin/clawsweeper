## Summary

Current `main` does not match the reported failure mode. The Web UI paste handler reads `props.onAttachmentsChange` from the chat view props, that callback is wired by the render layer into `state.chatAttachments`, attachments are forwarded into `chat.send`, and image-only turns are explicitly allowed in the reply gate.

## Close Comment

Closing after a Codex review of current `main` and the latest release code. The specific reported failure mode does not match the current implementation: the paste handler uses the chat view's internal `props.onAttachmentsChange` callback in `ui/src/ui/views/chat.ts:206`, and that callback is wired in `ui/src/ui/app-render.ts:2260` to update `chatAttachments`. Those attachments are then forwarded into `chat.send` in `ui/src/ui/controllers/chat.ts:213`, and image-only turns are explicitly allowed in `src/auto-reply/reply/get-reply-run.ts:412`. Querying `document.querySelector('openclaw-app').onAttachmentsChange` would still be expected to show `undefined`, because that callback is not a public custom-element property; it is an internal render prop. On current `main`, I do not see evidence for the reported undefined-callback delivery failure, so I’m closing this as no longer reproducible.
