## Summary

Keep open. Current `main` still has the negative/zero-threshold hole in memory-flush gating, especially for small-context windows or `reserveTokensFloor >= contextWindow - softThresholdTokens`, and the PR discussion materially improved the proposed fix instead of making it obsolete.

## Evidence

- **Small-context guard exists only in Pi compaction settings:** `applyPiCompactionSettingsFromConfig` already caps `reserveTokensFloor` against available context headroom for small windows, which shows the project has recognized this class of bug. That guard is local to Pi compaction settings and does not fix memory-flush gating.
  - file: [src/agents/pi-settings.ts:80](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/pi-settings.ts#L80)

  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Memory-flush plan still carries the raw reserve floor/default 20k:** The bundled memory-core flush plan still derives `reserveTokensFloor` directly from `agents.defaults.compaction.reserveTokensFloor` and otherwise falls back to `DEFAULT_PI_COMPACTION_RESERVE_TOKENS_FLOOR`, so small-window models can still enter the bad threshold state on current `main`.
  - file: [extensions/memory-core/src/flush-plan.ts:114](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/extensions/memory-core/src/flush-plan.ts#L114)

- **Core flush gate still disables itself when threshold goes non-positive:** `resolveMemoryFlushGateState` floors the inputs, computes `threshold = Math.max(0, contextWindow - reserveTokens - softThreshold)`, and then returns `null` when `threshold <= 0`. There is no clamp on `reserveTokensFloor`, so an oversized reserve still suppresses memory flush entirely on `main`.

Current `main` therefore still reproduces the edge case before a run starts.

Negative/zero thresholds therefore still shut off part of the flush path on `main`.
