## Summary

Keep open. Current `main` still resets any existing session file that lacks an assistant message, so the reported gateway-restart data-loss path remains plausible, and the PR's added regression test was also flagged for a Windows-incompatible `/tmp` temp-path assumption.

## Evidence

- **Current main still uses the broad `!hasAssistant` reset guard:** `prepareSessionManagerForRun()` on current `main` checks only for an assistant message, then truncates the session file with `fs.writeFile(..., "")` and clears in-memory state. A file containing `[session header, user message]` still satisfies this condition and would be wiped.
  - file: [src/agents/pi-embedded-runner/session-manager-init.ts:32](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/agents/pi-embedded-runner/session-manager-init.ts#L32)

  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **The reset path is still reached before any new assistant reply is appended:** Both embedded-runner and command transcript flows call `prepareSessionManagerForRun()` immediately after opening an existing session file and before appending any new assistant output. On a restart with an existing user-only turn, `hadSessionFile` is true and the broad reset condition still applies.
