## Summary

Keep this issue open. Current main still has flat `sessions`, `tasks`, `agents list`, `sessions_list`, and `/subagents list` surfaces, plus relationship metadata (`spawnedBy`, `parentSessionKey`, `childSessions`), but no user-facing CLI command that renders active agents and subagents as a hierarchy with JSON/watch/session/resource options.

## Best Possible Solution

Keep this open for a core inspection feature. A conservative path is to add a `sessions tree` or `agents tree` CLI command that reuses existing session/task/subagent metadata (`spawnedBy`, `parentSessionKey`, `childSessions`, status, runtime, tokens/cost), supports text and `--json` first, and defers `--watch` plus CPU/RSS resource accounting until the data contract is explicit.

## Risks / Open Questions

- The requested `--resources` option likely needs separate design because current rows expose runtime, token, and estimated cost metadata, but not reliable per-agent CPU/RSS ownership.
- The command name and scope need maintainer product judgment: `openclaw tree`, `openclaw sessions tree`, or `openclaw agents tree` could all be plausible, and the implementation should avoid creating a competing status model.
