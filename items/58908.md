## Summary

Keep #58908 open. Current main still emits only raw epoch-millisecond cron scheduling fields in the relevant structured logs/events, so the PR's requested human-readable `nextAt`/`nextRunAtMs` logging improvement has not shipped. The PR is useful but needs follow-up for the review-noted date-range guard before it is safe to merge.

## Best Possible Solution

Keep the PR open and ask the contributor to revise it: add a safe ISO formatting helper that returns `undefined` for non-finite and out-of-range timestamps, add regression tests for that edge case, and choose consistent structured field names while preserving the existing raw `nextAt` and `nextRunAtMs` fields. The linked feature issue #58574 remains the canonical product request until this lands.
