## Summary

Current main still does not implement the requested memoryFlush isolation from the main processing lane. The memory flush embedded run omits an explicit lane, the embedded runner defaults omitted lanes to `main`, and the queue docs still describe embedded runs as session-serialized and then globally queued on `main` by default. The related implementation candidate #53154 was closed unmerged, so there is no merged fix to close against.

## Best Possible Solution

Keep this issue open. The likely core fix belongs in `src/auto-reply/reply/agent-runner-memory.ts`: route memoryFlush through an explicit non-main background lane, or introduce a dedicated compaction lane with its own timeout and concurrency controls. Add regression coverage that captures the memoryFlush `runEmbeddedPiAgent` call and fails if it falls back to the default main lane, and decide whether same-session ordering should be preserved or relaxed for background compaction.

## Risks / Open Questions

- The current default `agents.defaults.maxConcurrent` behavior reduces some cross-session starvation compared with the reported v2026.3.x single-active-lane deployment, but it does not isolate memoryFlush and deployments can still configure lower main-lane concurrency.
- A simple global-lane override may still leave same-session messages blocked by the session lane, so maintainers need an explicit decision on whether background memory flush may bypass per-session serialization or only avoid process-wide main-lane starvation.
