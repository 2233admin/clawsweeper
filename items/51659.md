## Summary

Keep open. Current `main` adds Telegram-side fallback handling when a turn ends with no visible final reply, but the embedded runner still has return paths before the `embedded run done` log point, and the available tests only prove fallback delivery or `NO_REPLY` rewriting, not that the originally generated assistant response now reliably reaches Telegram. That is not strong enough evidence for a conservative close.

## Evidence

- **`embedded run done` is still not unconditional in the runner:** In `runEmbeddedPiAgent`, there is an earlier return for incomplete-turn handling immediately before the `embedded run done` debug log. Code inspection does not prove the reported missing-done path is fully eliminated.
  - file: [src/agents/pi-embedded-runner/run.ts:2119](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/pi-embedded-runner/run.ts#L2119)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '2100,2149p'`
- **Telegram now has an explicit no-visible-final-response fallback:** If `queuedFinal`, `sentFallback`, and `deliverySummary.delivered` are all false, Telegram dispatch synthesizes a fallback delivery from a `NO_REPLY` plan instead of silently ending the turn.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:1164](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/telegram/src/bot-message-dispatch.ts#L1164)
  - command: `nl -ba extensions/telegram/src/bot-message-dispatch.ts | sed -n '1162,1191p'`
- **Telegram tests cover the missing-visible-final path:** A dedicated test drives `dispatchReplyWithBufferedBlockDispatcher` to return `queuedFinal: false` and asserts that a DM turn is rewritten through silent-reply fallback.
  - file: [extensions/telegram/src/bot-message-dispatch.test.ts:2573](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/telegram/src/bot-message-dispatch.test.ts#L2573)
  - command: `nl -ba extensions/telegram/src/bot-message-dispatch.test.ts | sed -n '2570,2585p'`
- **Delivery tests verify `NO_REPLY` rewriting, not recovery of the original assistant text:** Direct Telegram delivery rewrites exact `NO_REPLY` into a visible message for direct sessions. That mitigates silent hangs, but it is not proof that the originally generated response content is preserved and delivered.
  - file: [extensions/telegram/src/bot/delivery.test.ts:296](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/extensions/telegram/src/bot/delivery.test.ts#L296)
