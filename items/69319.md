## Summary

Keep PR #69319 open. Current main still lacks a `skills.uninstall`/`skills.remove` Gateway RPC, while the existing skill lifecycle surface exposes ClawHub install and update only. The PR addresses a real remaining gap from #69299, but it should not be closed as implemented or superseded; it also needs the existing review concerns resolved before merge.

## Best Possible Solution

Keep the PR open or ask for a focused revision. The right product path is still a core Gateway `skills.uninstall` RPC because it mirrors the already-core `skills.install` ClawHub control-plane path and avoids each operator UI reinventing host filesystem deletion. Before merge, update the implementation to delete only ClawHub-tracked skill folders, map client slug errors to `INVALID_REQUEST`, keep the method in `operator.admin`, regenerate protocol clients/models, and retain backend plus gateway handler tests for installed, idempotent, malformed, traversal, legacy tracked slug, and non-ClawHub collision cases.

## Risks / Open Questions

- The PR as described should gate deletion on ClawHub provenance, such as origin metadata or lockfile membership, to avoid deleting manually managed or installer-managed skill folders with the same slug.
- Malformed slug errors should be classified as `INVALID_REQUEST` rather than `UNAVAILABLE` so operator clients do not retry bad input as an outage.
- The stale lockfile-only case may need clearer response semantics so callers can distinguish a pure no-op from lockfile cleanup.
