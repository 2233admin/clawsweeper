## Summary

Keep this issue open. Current main has optional/plugin-owned memory prompt guidance and Active Memory before eligible conversational replies, but it still does not provide the requested enforced recall-before-response rule check with block-on-failure semantics, especially for the reported subagent/workflow orchestration surface. The linked implementation PR #43825 is closed and unmerged.

## Best Possible Solution

Keep the issue open for maintainer design. The best path is an opt-in runtime policy or plugin/context-engine API that can run recall before selected model responses, inject recalled rule context, and optionally block or return a controlled failure when recall cannot run, with explicit coverage for embedded runs, CLI/subagents, automation, and existing Active Memory/memory plugin behavior.

## Risks / Open Questions

- A blanket forced recall step could add latency, cost, and surprising hidden personalization in automation or internal-worker paths, so the eventual design should be opt-in and scoped.
- A plugin-shaped solution may be preferable, but current `before_prompt_build` and Active Memory behavior is best-effort or conversationally scoped rather than an enforceable rule-checking gate.
- Resurrecting PR #43825 directly would still need maintainer design review, rebase, and validation across embedded runs, CLI/subagents, Active Memory, and memory plugin interactions.
