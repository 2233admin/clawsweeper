## Summary

Keep open. Current `main` has already shipped the error-classification cleanup from #67642, but the underlying `openai-codex` request path still uses the standard Node/undici fetch and proxy stack. The provider request surface exposes `proxy`, `tls`, and `allowPrivateNetwork`, but there is no browser-style TLS fingerprint/profile option or documented workaround in core for the reported Cloudflare JS challenge path behind mainland-China proxies.

## Evidence

- **OpenAI Codex client still uses the shared guarded fetch transport:** `openai-codex` traffic is created through the shared OpenAI client and passes `fetch: buildGuardedModelFetch(model)`, so any transport behavior is whatever the guarded Node fetch layer does today.
