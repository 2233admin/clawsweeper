## Summary

Keep this issue open. Current main still models `memorySearch.fallback` as a string provider id plus one shared `memorySearch.remote` object, and both immediate and runtime fallback paths continue to reuse that same remote config. The related PR #44878 proposed the requested object shape but is closed unmerged, while broader #55590 is closed, so this remains the narrow tracker for the missing capability.

## Best Possible Solution

Keep this issue open. A good fix should preserve the existing string fallback form for compatibility while adding an object fallback shape such as `{ provider, model, remote }`, then thread fallback-specific remote, model, headers/secrets, output dimensionality where needed, and provider defaults through config validation, `resolveMemorySearchConfig`, `createEmbeddingProvider`, runtime fallback activation, docs/help metadata, and focused tests for Gemini primary plus OpenAI-compatible fallback remote.
