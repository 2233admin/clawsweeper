## Summary

Current `main` already preserves paragraph newlines in the shared reply-cleanup path that would have affected all channel replies, and the outbound plain-text sanitizers used by channels like WhatsApp and Telegram do not collapse double newlines to single newlines. Regression tests cover the exact multi-paragraph case.

## Close Comment

This looks fixed on current `main`, so I'm closing it based on a Codex review.

- The shared reply-cleanup path now preserves newlines by collapsing only horizontal whitespace, not all whitespace, in [`src/auto-reply/reply/reply-inline.ts:42`](src/auto-reply/reply/reply-inline.ts:42).
- That helper is still invoked from the common reply pipeline in [`src/auto-reply/reply/get-reply-directives.ts:364`](src/auto-reply/reply/get-reply-directives.ts:364), which matches the report's cross-channel scope.
- Regression tests cover the reported failure mode directly: multi-line bodies keep `\n\n`, and `/status\nfirst paragraph\n\nsecond paragraph` stays as two paragraphs in [`src/auto-reply/reply/reply-inline.test.ts:11`](src/auto-reply/reply/reply-inline.test.ts:11) and [`src/auto-reply/reply/reply-inline.test.ts:17`](src/auto-reply/reply/reply-inline.test.ts:17).
- The shared plain-text outbound sanitizer used by channels such as WhatsApp and Telegram only compresses `3+` newlines to `2`, not `2` to `1`, in [`src/infra/outbound/sanitize-text.ts:45`](src/infra/outbound/sanitize-text.ts:45), with channel usage in [`extensions/whatsapp/src/outbound-base.ts:145`](extensions/whatsapp/src/outbound-base.ts:145) and [`extensions/telegram/src/outbound-adapter.ts:125`](extensions/telegram/src/outbound-adapter.ts:125).
