## Summary

Keep PR #70707 open. Current main still matches the reported gateway race: device removal and device-token rotation/revocation respond first, then defer socket disconnects with queueMicrotask; current request context only closes sockets; and the post-handshake RPC path rechecks shared gateway auth generation but has no equivalent invalidated-device-token gate. The provided Greptile review also treated the PR as a focused, well-tested security fix rather than obsolete or low-signal work.

## Best Possible Solution

Keep PR #70707 open and review or reimplement the same focused fix on current main: synchronously mark affected device-token clients invalid before responding in device.pair.remove, device.token.rotate, and device.token.revoke; make disconnectClientsForDevice set the same marker as defense in depth; reject later post-handshake RPCs from invalidated clients before handleGatewayRequest dispatch; and add regression tests for response-flush ordering plus pipelined RPC rejection.
