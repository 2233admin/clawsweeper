## Summary

Keep PR #70707 open. Current main still has the device-token rotate/revoke/remove response-before-disconnect pattern described by the PR, and I did not find an equivalent synchronous per-RPC invalidation guard on main. The PR discussion is relevant: the bot review summarized the same race and reported no blocking findings, which supports treating this as a plausible focused gateway hardening PR rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open and review it as a focused gateway hardening fix. The best implementation path is the PR's synchronous device-client invalidation plus a dispatch-time guard, with regression coverage proving invalidation happens before `respond(...)` for pair removal, token rotation, and token revocation.
