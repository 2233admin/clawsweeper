## Summary

Keep this issue open. Current main has Feishu topic session routing and thread-history lookup, but it still does not resolve a `thread_id`-only Feishu topic event to the root `om_*` message before choosing the outbound reply target. In the reported case, normal replies still fall back to the triggering `messageId`, which matches the bug instead of fixing it.

## Best Possible Solution

Implement the fix in the Feishu plugin. For topic-mode or `replyInThread` group events with `thread_id` but no usable `root_id`, `parent_id`, or `reply_target_message_id`, resolve the thread container to the earliest/root `om_*` message, cache `thread_id -> rootMessageId`, and use that root for ACP init error notices, normal replies, streaming cards, media replies, and dispatcher `rootId`/`replyToMessageId`. Add regression tests for the `thread_id`-only topic event that assert outbound reply anchoring, while preserving #32980 normal-group behavior where replies with `root_id` should target the triggering message.
