## Summary

Keep this issue open. Current main has useful external-content wrapping, web_fetch untrusted metadata, input provenance for selected user-message paths, plugin tool hooks, and tool-result persistence hooks. It still does not implement the central request: persistent taint/provenance through derived content, default content-derived execution denial, general exec risk classes, or provenance-aware secret/outbound enforcement.

## Best Possible Solution

Keep this issue open as the core security architecture umbrella unless maintainers split it into canonical implementation issues. A defensible implementation path is to add structured provenance/taint metadata for external inputs and tool outputs, preserve it through summaries and persistence, expose stable provenance/action/risk fields to before_tool_call and approval flows, and enforce content-derived exec, secret access, and outbound-action policy in core while coordinating narrower follow-ups such as #54373, #48519, #48503, #41140, #47492, and #53185.
