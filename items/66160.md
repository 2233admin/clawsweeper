## Summary

Keep this PR open. Current main still ignores `--custom-base-url` for LiteLLM non-interactive onboarding: the LiteLLM plugin uses the generic single-provider API-key helper, and that helper only calls `applyConfig(cfg)`, so the LiteLLM config path never sees `ctx.opts.customBaseUrl`. The PR is still a plausible focused bug-fix candidate, but it needs maintainer review because it also adds `/v1/models` discovery and the submitted diff does not add regression coverage.

## Best Possible Solution

Keep this PR open as a valid bug-fix candidate. The best maintainer path is to rebase it onto current main, add focused regression coverage for LiteLLM non-interactive `--custom-base-url`, and then decide whether to keep, narrow, or separately track the added `/v1/models` discovery behavior after resolving the logging, SSRF-policy, and capability-metadata review points.

## Risks / Open Questions

- The PR adds an outbound `/v1/models` probe during onboarding. It uses the repo SSRF guard, but maintainers should still review the private-proxy policy, redirect behavior, and whether model discovery belongs in this bug fix.
- The PR logs the raw LiteLLM base URL when discovery is unreachable; review comments already note that URL userinfo would be exposed if a user supplied credentials in the URL.
- The PR currently advertises optimistic capabilities for discovered models, while review discussion notes `/v1/models` can include IDs that may not support reasoning or image input.
- The diff shown for this PR does not include a focused regression test for LiteLLM non-interactive `--custom-base-url`, so the bug would remain easy to regress.
