## Summary

Keep #69570 open. Current main has useful partial mitigations, including a `/approve` shell-command guard and `/stop` clearing already queued followups for the target session, but the central async approval cascade is still present: each approval-gated exec creates its own detached completion watcher, and each completion can still enqueue a separate follow-up agent turn. The David-CKS comment adds a current deterministic repro and useful related context (#69386, #72062, #57568); those are adjacent, but this issue remains the canonical tracker for the broader approval-completion cascade.

## Best Possible Solution

Keep #69570 open and implement a core fix in the exec approval/followup path: track approval completion followups by session and originating task/turn, let `/stop` mark a cutoff that suppresses future completion followups from older approval ids, and coalesce multiple completions from the same logical task into one bounded followup. Add regression coverage for several approved execs completing after `/stop`, for user messages preempting old async completions, and for denied-approval retry loops. Treat #72062 (`deny_wrappers`) as a useful separate hardening feature, not the fix for this cascade.
