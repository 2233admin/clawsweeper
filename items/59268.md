## Summary

Current main does not implement per-macOS-node default chat sessions. The static `openclaw-macos` clientId hypothesis is only partly current because the handshake now includes instance/device identity, but the macOS WebChat default and chat send/history paths still resolve to the gateway-wide main session, so two Macs paired to the same gateway can still share the same default chat session by design/current behavior.

## Best Possible Solution

Keep the issue open for maintainer triage. A real fix would need a product decision and implementation that gives macOS WebChat a per-device/per-node default session key, plumbs that through the gateway hello/session defaults or client-side selection, updates the session picker labels, and adds multi-client regression coverage proving two macOS nodes on one gateway do not share default history unless explicitly selecting the same session.
