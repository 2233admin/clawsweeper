## Summary

Keep this issue open. Current main still emits the reported embedded `read tool called without path` warning for `read` tool-start events without `path` or `file_path`, and that warning still lacks the run/session/agent correlation needed to identify the malformed embedded caller. Adjacent fixes for heartbeat prompt guidance and canonical read-parameter validation do not resolve the Feishu reconnect/failover diagnostic gap described here.

## Best Possible Solution

Keep this open as a diagnostics and root-cause task. Add structured correlation to the `read tool called without path` warning using available `runId`, `sessionId`, `sessionKey`, and `agentId`, pass trigger/provider/channel/plugin context into the subscription if needed, then reproduce the Feishu timeout/reconnect/failover path and fix or reject the owner that constructs the malformed `read` call before it reaches the embedded tool-start handler.
