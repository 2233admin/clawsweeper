## Summary

Keep open. Current main still contains the reported embedded read-tool start warning, the warning still lacks the requested run/session correlation fields, and execution-time validation only rejects missing canonical `path` after the start event has already reached `handleToolExecutionStart`. The related alias-warning issues do not fully cover this report's malformed-caller/recovery-loop concern.

## Best Possible Solution

Keep the issue open and treat it as a diagnostics/root-cause bug. The best path is to add structured correlation to the `read tool called without path` warning or internal event (`runId`, `sessionId`, `sessionKey`, trigger, agent/channel/provider where available), reproduce the heartbeat/cron/Feishu timeout path, then either fix the malformed caller or reject/repair the bad read arguments before the embedded start handler emits the warning. If later evidence proves it is only an alias false positive, fold it into the existing alias-warning tracker; current evidence is not strong enough for that closure.
