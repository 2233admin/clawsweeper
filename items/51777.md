## Summary

Current `main` still appears to have the reported bug path, so this PR should stay open. In `src/flows/channel-setup.prompts.ts`, onboarding only calls `policy.setPolicy` when `nextPolicy !== current`. Feishu's adapter reports a missing config key as `"pairing"`, so a fresh config plus an explicit `"pairing"` choice is treated as a no-op. The config write path persists the pre-validation merge-patched candidate rather than schema-injected defaults, so that no-op does not materialize `channels.feishu.dmPolicy` into the file. No close reason applies.

## Evidence

That means an explicit selection can be dropped if `getCurrent` returns the same default value.

## Risks / Open Questions

- This PR is still useful because current `main` does not implement the fix, but its Feishu-only scope leaves the same default-equals-fallback persistence gap in other channel adapters that use `?? "pairing"`.
- There does not appear to be a direct regression test covering `maybeConfigureDmPolicies` persisting an explicit default choice from a missing source key, so the bug can regress silently.
