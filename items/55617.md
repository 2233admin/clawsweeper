## Summary

Keep open. Current `main` has not implemented the proposed high-priority-preserving truncation, and the PR’s stated basis is out of sync with the current prompt pipeline: skills are deduplicated by precedence, then sorted alphabetically before truncation. That means a suffix-based truncation would not reliably preserve workspace/project skills by source priority, so this is not a safe close as already-fixed or obsolete.

## Evidence

- **Current main still truncates by prefix:** `applySkillsPromptLimits` keeps `slice(0, maxSkillsInPrompt)` and, in compact mode, binary-searches the largest prefix that fits, then keeps `slice(0, lo)`. The PR is not implemented on current `main`.
  - file: [src/agents/skills/workspace.ts:678](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/skills/workspace.ts#L678)
  - command: `nl -ba src/agents/skills/workspace.ts | sed -n '667,716p'`
- **Prompt ordering is alphabetical before truncation:** `buildWorkspaceSkillsPrompt` compacts paths, then sorts `promptSkills` by `name` before calling `applySkillsPromptLimits`. Truncation therefore operates on alphabetical order, not source precedence order.
  - file: [src/agents/skills/workspace.ts:791](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/skills/workspace.ts#L791)
  - command: `nl -ba src/agents/skills/workspace.ts | sed -n '768,811p'`
- **Load precedence is only used to choose the winning duplicate:** `loadSkillEntries` merges sources in precedence order into a `Map`, then converts the merged values to an array and sorts by skill name. So precedence decides which duplicate survives, but not the later truncation order.
  - file: [src/agents/skills/workspace.ts:579](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/skills/workspace.ts#L579)
  - command: `nl -ba src/agents/skills/workspace.ts | sed -n '579,628p'`
- **Tests codify alphabetical prompt behavior on main:** There is an explicit test asserting that skills are sorted alphabetically regardless of entry insertion order, which conflicts with the PR description’s assumption that truncation sees a low→high priority list.
  - file: [src/agents/skills/compact-format.test.ts:274](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/skills/compact-format.test.ts#L274)
  - command: `nl -ba src/agents/skills/compact-format.test.ts | sed -n '274,286p'`
- **Docs describe precedence as duplicate resolution, not truncation priority:** The skills docs describe precedence for conflicting skill names and separately describe visibility/allowlists; they do not document budget truncation as preserving higher-precedence sources first.
  - file: [docs/tools/skills.md:22](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/docs/tools/skills.md#L22)
