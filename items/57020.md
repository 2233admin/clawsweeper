## Summary

Current `main` already rotates `sessionFile` when a reset creates a new `sessionId`, and the same implementation is present in shipped release `v2026.4.23`. The reported stale-transcript behavior is covered by an explicit regression test.

## Evidence

- **Resolver no longer reuses stale transcript after sessionId change:** `resolveAndPersistSessionFile()` now gates persisted `sessionFile` reuse on `baseEntry.sessionId === sessionId`; if the session id changed, it clears or replaces the old path before resolving the transcript file.

## Close Comment

Closing this as implemented on current `main`.

- Codex review of `src/config/sessions/session-file.ts:21` shows `resolveAndPersistSessionFile()` only reuses an existing `sessionFile` when the persisted `sessionId` matches the current one. If the session id changed, it resolves a fresh path instead of keeping the stale transcript.
- `src/auto-reply/reply/session.ts:453` and `src/auto-reply/reply/session.ts:720` still take the daily/idle stale-session path, mint a new `sessionId`, and then run that entry through the session-file resolver, which matches the reset flow described here.
- `src/config/sessions/sessions.test.ts:400` now has a regression test that explicitly asserts the same-session-key / new-session-id case rotates `sessionFile` and does not preserve the old path.
- This was a Codex review against the checked-out repo, and the same implementation is already present in shipped release `v2026.4.23` (`9b18ae877018a34933e27bb7b2986cb97e9602aa`).
