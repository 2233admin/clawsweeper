## Summary

Keep this PR open. Current main still retains tool-call history across idle heartbeat gaps and runs loop detection before recording the next call, so the linked false-positive bug is not implemented on main. The PR is a plausible focused fix candidate, but it still needs maintainer review for cutoff semantics, warning dedupe, generated config/docs alignment, and consolidation with the overlapping open fix attempt #40798.

## Best Possible Solution

Keep this PR open for maintainer review and consolidate it with #40144 and #40798. The best implementation should prune or scope loop history before detection at the real heartbeat or turn boundary, use activity/outcome-aware cutoff semantics that preserve slow-tool loop protection, reset or scope warning dedupe consistently, and add detect-then-record regression coverage plus aligned config/schema/docs updates.

## Risks / Open Questions

- The PR's stale cutoff is based on ToolCallRecord.timestamp, and recordToolCallOutcome currently records resultHash without refreshing timestamp; a slow tool call over the threshold could cause same-session loop history to be cleared too aggressively.
- Clearing toolCallHistory without also scoping or resetting toolLoopWarningBuckets can leave warning dedupe state from an earlier burst and suppress later warning-level loops after a reset.
- If staleThresholdMs becomes public config, generated schema/help/labels and user docs should be aligned with the TypeScript and Zod schema changes.
- There are multiple open fix attempts for #40144, so maintainers should choose one canonical implementation path before landing.
