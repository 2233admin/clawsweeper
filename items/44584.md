## Summary

Keep open. Current main has evidence of prompt-cache hardening around inbound metadata, but the combined issue’s CLI routing regression is not fixed: `openclaw agent --channel discord --to ...` still resolves the session from `--to` only, while `--channel` is treated as delivery metadata rather than as part of session lookup. For a Discord thread/channel target such as `channel:148...`, the current resolver path can still collapse to the agent main session instead of finding the existing live Discord thread session.

## Evidence

- **CLI session key ignores channel:** `agentViaGatewayCommand` resolves `sessionKey` with `cfg`, `agentId`, `to`, and `sessionId` only. The normalized `channel` is computed afterward and passed to the Gateway as delivery metadata, not used to find the session.
