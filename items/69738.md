## Summary

Keep open. Current `main` still derives the default cron store from a shared config/state root, and the repo docs explicitly require a unique `OPENCLAW_STATE_DIR` per gateway instance. That makes the reported duplicate-fire behavior still plausible for the described setup, and there is no code-level isolation or warning that would justify an auto-close.

## Evidence

- **Default cron store is still shared by default:** `resolveDefaultCronDir()` still returns `path.join(resolveConfigDir(), "cron")`, and `resolveDefaultCronStorePath()` still resolves to `<that>/jobs.json`. There is no built-in per-instance namespacing in the default path.
  - file: [src/cron/store.ts:26](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/cron/store.ts#L26)
  - command: `nl -ba src/cron/store.ts | sed -n '20,40p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Config-dir fallback still collapses to config file parent when state dir is unset:** `resolveConfigDir()` prefers `OPENCLAW_STATE_DIR`, but if that is unset it falls back to `path.dirname(OPENCLAW_CONFIG_PATH)`. Two configs in the same parent directory still resolve to the same default cron root.
  - file: [src/utils.ts:136](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/utils.ts#L136)

  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Gateway cron service uses that resolved store path at startup:** `buildGatewayCronService()` calls `resolveCronStorePath(params.cfg.cron?.store)` directly when cron is enabled. If two gateways land on the same default store path, both services can target the same cron store.
  - file: [src/gateway/server-cron.ts:147](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/server-cron.ts#L147)

That supports the existing workaround, but it does not mean the reported shared-store behavior has been fixed in code.

Follow-up comments converge on instance namespacing by config stem as the preferred fix, which makes this more than a vague support question.

## Risks / Open Questions

- The repo already documents unique `OPENCLAW_STATE_DIR` values for multiple gateways, so maintainers could still decide the shared-state setup is unsupported rather than a core bug.
