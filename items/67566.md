## Summary

Keep open. The report is recent, another user corroborated a tool-only failure on v2026.4.15, and current `main` does not show a specific fix that clearly resolves the `memory_search` tool path. The original empty-`meta` diagnosis is weakened by current code and by the follow-up comment, but that is not evidence the underlying bug is gone.

## Evidence

- **Issue discussion shows a still-plausible tool-only bug:** The original issue claims empty `meta` tables, but the follow-up comment reports `openclaw memory status --deep` and `openclaw memory search` both working while `memory_search` tool calls still fail under a proxy/TUN environment. That contradicts the reported root cause, but it also means the underlying bug was not explained away.
  - command: `Provided GitHub context for issue #67566 and comment 4269057467`
- **CLI and tool use different wrapper paths around memory search:** `runMemorySearch` calls `manager.search(...)` directly for the CLI path, while the tool path adds extra wrapper logic (`getMemoryManagerContext`, citations, active-memory session handling, supplement merging, and broad error catching). That matches the commenter's observation that CLI search can work while the tool still fails.
