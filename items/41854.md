## Summary

Current `main` does not implement Groq onboarding/auth-provider setup. The Groq extension on `7ac35b4f69009490180a35bf404638d4280a4865` is still audio-only, while onboarding choices and non-interactive auth flags are generated from provider plugin metadata that Groq does not provide.

## Evidence

- **Groq plugin is not a text provider:** The bundled Groq plugin only registers `groqMediaUnderstandingProvider`; it does not call `api.registerProvider(...)` or define any setup/auth wizard.
  - file: [extensions/groq/index.ts:1](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/groq/index.ts#L1)
  - command: `nl -ba extensions/groq/index.ts | sed -n '1,240p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Groq manifest lacks onboarding/auth choice metadata:** The Groq manifest has `providerAuthEnvVars` and media-understanding contracts, but no `providers` or `providerAuthChoices` entries, so there is no manifest-owned `groq-api-key` setup choice or `--groq-api-key` flag source.
  - file: [extensions/groq/openclaw.plugin.json:1](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/groq/openclaw.plugin.json#L1)

Core flags are empty on main, so without Groq manifest metadata there is no real `--groq-api-key` CLI option to forward.

## Risks / Open Questions

- The PR was authored against an older core-owned auth/onboarding architecture. If revived, it likely needs a rewrite into `extensions/groq` manifest/runtime provider seams rather than the old `src/commands/*` plumbing shown in the PR.
- Docs currently advertise Groq setup in `docs/providers/groq.md`, but the live plugin/runtime evidence above still lacks the actual onboarding/auth flow, so closing this PR as implemented would leave a real user-facing gap.
