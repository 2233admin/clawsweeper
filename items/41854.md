## Summary

Keep open. Current `main` still does not implement the PR's requested Groq text-inference onboarding/auth UX: Groq has bundled docs and a media-understanding plugin, but it does not contribute a text-provider setup flow, auth choice, or `--groq-api-key` flag in the current manifest-driven onboarding architecture.

## Evidence

- **Groq bundled plugin metadata is audio-only:** The bundled Groq manifest declares `providerAuthEnvVars` and `mediaUnderstandingProviders`, then ends without any `setup.providers` or `providerAuthChoices` metadata that would drive text-inference onboarding.
  - file: [extensions/groq/openclaw.plugin.json:1](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/extensions/groq/openclaw.plugin.json#L1)
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Groq plugin runtime only registers media understanding:** The Groq extension registers `groqMediaUnderstandingProvider` only; there is no provider-setup/onboarding registration for text inference.
  - file: [extensions/groq/index.ts:1](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/extensions/groq/index.ts#L1)
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Provider auth/setup choices are manifest-driven now:** Current `main` synthesizes provider auth choices from manifest `providerAuthChoices` and `setup.providers[].authMethods`. Without that metadata in the Groq manifest, no Groq onboarding choice is produced.
  - file: [src/plugins/provider-auth-choices.ts:145](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/plugins/provider-auth-choices.ts#L145)
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Interactive auth picker consumes provider setup flow contributions:** The auth-choice UI now pulls provider entries from `resolveProviderSetupFlowContributions(... scope: "text-inference")`, so Groq will not appear unless its plugin contributes a setup flow.
  - file: [src/commands/auth-choice-options.ts:26](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/commands/auth-choice-options.ts#L26)
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **CLI onboarding flags are also manifest-driven:** `register.onboard.ts` builds provider auth flags from `resolveManifestProviderOnboardAuthFlags()`. Since Groq has no qualifying manifest auth choice metadata, `--groq-api-key` is not available on current `main`.
  - file: [src/cli/program/register.onboard.ts:50](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/cli/program/register.onboard.ts#L50)
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **No Groq onboarding flag or auth-choice string exists on main:** A repository search for `groq-api-key` / `--groq-api-key` returned no matches in `src` or `docs`, confirming the requested surface is still absent.
  - command: `rg -n "groq-api-key|--groq-api-key" src docs`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Docs show partial/manual Groq support, not onboarding parity:** The public Groq docs already tell users to set `GROQ_API_KEY` and manually configure `agents.defaults.model.primary` to `groq/llama-3.3-70b-versatile`, which matches the PR description of partial support but does not replace the missing onboarding/auth flow.
  - file: [docs/providers/groq.md:19](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/docs/providers/groq.md#L19)
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)

- Closing this PR would discard a still-missing user-facing gap: Groq is documented and partially wired, but current main still lacks onboarding/auth-choice parity for text inference.
- The PR's code targets older hard-coded onboarding plumbing. Any eventual merge/follow-up likely needs rebasing onto the current plugin/manifest-driven provider setup flow instead of the previous command-local wiring.
