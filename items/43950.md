## Summary

Current main still loads local skills for runtime/listing, but `skills.search` and `openclaw skills search` remain ClawHub-only. The requested local registration or local search-index behavior is not implemented, and no canonical open replacement was identified in the provided related-item context.

## Best Possible Solution

Keep this issue open. The best path is a core-owned local discovery design: either extend `skills.search` and `openclaw skills search` with a local or combined source backed by `loadWorkspaceSkillEntries`, or add an explicit `skills.register` / `openclaw skills register <path>` flow that records safe local roots through the existing `skills.load.extraDirs` or managed-skill semantics. The implementation should include protocol/CLI/docs updates and tests for workspace and `~/.openclaw/skills` entries while preserving current ClawHub install/update behavior.
