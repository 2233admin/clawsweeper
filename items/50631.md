## Summary

Keep open. Current `main` still allows `gateway.auth.mode=none` with `gateway.tailscale.mode=serve`, so the reported exposure is not implemented/fixed on `main`. The PR also has a credible unresolved review point: it adds only a startup-time guard, while install/configure flows still accept the same config and can proceed without requiring a token.

## Evidence

That means `serve + none` still passes startup validation when bound to loopback.

## Risks / Open Questions

- Closing this PR would leave a plausible security bug unfixed on current `main`.
- Merging only the startup guard could create a partial fix where daemon install/configure succeeds but the service then fails on first boot.
- Current docs describe `auth.mode: none` as private-ingress-only, so the present behavior remains a security footgun until the validation story is aligned.
