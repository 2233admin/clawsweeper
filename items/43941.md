## Summary

Keep open. Current `main` still does not remove sandbox workspace directories during prune, so the reported leak is not implemented on `main`. The PR discussion also shows unresolved correctness issues around how workspace paths are derived from stored registry `sessionKey` values, so there is not strong evidence for an auto-close in any allowed category.

## Evidence

- **Current main prune flow does not delete workspace directories:** `pruneSandboxRegistryEntries` only calls `removeRuntime`, then removes the registry entry and optional browser bridge state. There is no `fs.rm(...)` or other workspace cleanup in the prune path, so the core bug described by the PR is still present on `main`.
  - file: [src/agents/sandbox/prune.ts:38](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/sandbox/prune.ts#L38)
  - command: `nl -ba src/agents/sandbox/prune.ts | sed -n '1,132p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Workspace layout is scope-dependent:** Sandbox workspaces are resolved from `cfg.workspaceRoot`, but `shared` scope uses the root directly while non-shared scopes use `resolveSandboxWorkspaceDir(workspaceRoot, scopeKey)`. Any prune cleanup must mirror this branching exactly or it will miss shared workspaces or delete the wrong directory.
  - file: [src/agents/sandbox/context.ts:41](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/sandbox/context.ts#L41)
  - command: `nl -ba src/agents/sandbox/context.ts | sed -n '38,46p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Registry entries already store the computed scope key:** Main sandbox registry updates persist `sessionKey: scopeKey`, not the raw session key. Browser registry updates do the same. That means a prune fix must treat stored `sessionKey` values as already-normalized scope keys rather than re-running `resolveSandboxScopeKey(...)` on them.
  - file: [src/agents/sandbox/context.ts:166](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/sandbox/context.ts#L166)
  - command: `nl -ba src/agents/sandbox/context.ts | sed -n '160,176p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Browser registry also uses scope keys:** Browser sandbox entries are written with `sessionKey: params.scopeKey`, matching the main registry behavior and reinforcing that cleanup code should not derive a second scope key from stored values.
  - file: [src/agents/sandbox/browser.ts:433](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/agents/sandbox/browser.ts#L433)
  - command: `nl -ba src/agents/sandbox/browser.ts | sed -n '420,450p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **PR discussion shows unresolved path-resolution risk on the latest reviewed head:** The PR thread records an initial review about wrong workspace-path derivation, an author reply claiming they switched to `resolveSandboxScopeKey(cfg.scope, sessionKey)`, and then a later review on head `dedcd229354efe23a9d92d5a31565ecf13f96f8d` pointing out that registry entries already store scope keys, so re-running `resolveSandboxScopeKey(...)` would still target the wrong folder for agent-scoped sandboxes. That leaves unresolved correctness concerns rather than a basis for closure.
  - file: [src/agents/sandbox/prune.ts:65](https://github.com/openclaw/openclaw/blob/dedcd229354efe23a9d92d5a31565ecf13f96f8d/src/agents/sandbox/prune.ts#L65)
  - sha: [dedcd229354e](https://github.com/openclaw/openclaw/commit/dedcd229354efe23a9d92d5a31565ecf13f96f8d)

## Risks / Open Questions

- There is no local prune-specific test coverage showing workspace deletion across `session`, `agent`, and `shared` scopes, so this area is easy to regress.
- Because browser and main registries both persist scope keys, a superficially plausible cleanup patch can still be wrong if it recomputes scope keys or treats shared scope like a slugged subdirectory.
