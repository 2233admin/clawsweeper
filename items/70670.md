## Summary

Keep open. Source review does not show a clear main-branch fix for this report, and the only explicit safeguard I found against editing injected bootstrap files is a memory-flush-specific prompt rule rather than a general runtime guard. The current code also weakens the reporter's exact reload theory because bootstrap context is snapshotted into the system prompt at run start, but there is no targeted proof on `main` that editing an injected file mid-turn is safe.

## Evidence

- **Bootstrap files are injected into the run prompt:** The embedded runner resolves bootstrap context for the run, builds the system prompt with `contextFiles`, and applies that prompt to the session. That means files such as `AGENTS.md`/`MEMORY.md` can be in prompt context during a turn, but the prompt is assembled up front rather than obviously reloaded after each edit.

  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Main-channel command path also injects workspace context:** The command/system-prompt path used for channel runs calls `resolveBootstrapContextForRun` and passes the injected files into `buildAgentSystemPrompt`, so the reported surface area is real on current `main`.
  - file: [src/auto-reply/reply/commands-system-prompt.ts:39](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/auto-reply/reply/commands-system-prompt.ts#L39)

## Risks / Open Questions

- This was a read-only source/history review only. I did not run a live reproduction, and `pnpm` is unavailable in this checkout, so there is no runtime proof that the loop is gone.
