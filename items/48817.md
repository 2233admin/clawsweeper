## Summary

Current `main` already pushes assistant output for `chat.send` runs over WebSocket `chat` events, including final message content, and has matching protocol/docs/e2e coverage. The reported `/status`-style missing-payload behavior no longer matches current `main`.

## Close Comment

Closing from a Codex review of current `main`. This behavior is implemented on `main` now: Gateway `chat.send` responses stream back over WebSocket `chat` events, and terminal events can carry the assistant message payload directly (`src/gateway/server-chat.ts:806`). The public schema also includes `message` on `ChatEvent` (`src/gateway/protocol/schema/logs-chat.ts:71`), the docs say `chat.send` streams via `chat` events (`docs/web/control-ui.md:152`), and current tests cover both unit and e2e cases where the final pushed event contains non-empty text (`src/gateway/server-chat.agent-events.test.ts:325`, `test/gateway.multi.e2e.test.ts:115`).

The strongest proof I found is commit `925d11d89054c5dd1ba31de5537d2ee3cb5cdd3b`, which is on current `main`, but I could not tie it to a released tag yet, so `fixedRelease` is left unset.
