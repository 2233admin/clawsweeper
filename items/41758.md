## Summary

Keep open. Current `main` does not already implement automatic zombie-Chrome cleanup before launch, and the submitted PR still has unresolved correctness/security blockers in review. The underlying problem remains plausible, but this patch is not in a mergeable state and now targets pre-browser-plugin file layout that would need rework.

## Evidence

- **Current launch path still hard-gates on a free CDP port:** On current `main`, `launchOpenClawChrome()` calls `ensurePortAvailable(profile.cdpPort)` before any process-cleanup logic. That means an occupied CDP port still fails launch immediately rather than attempting zombie-process recovery.

  - command: `nl -ba extensions/browser/src/browser/chrome.ts | sed -n '360,620p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Current recovery path is still manual `reset-profile`:** When HTTP responds on the port but the websocket is not usable and OpenClaw does not own the runtime, `ensureBrowserAvailableOnce()` throws a user-facing error telling the operator to run `action=reset-profile profile=<name>`. That is a manual workaround, not the auto-cleanup proposed by this PR.
