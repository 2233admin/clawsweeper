## Summary

Current `main` already handles the reported zombie-PID case: gateway lock acquisition now treats an existing PID as stale when the configured port is not listening, removes the stale lock, and retries. The same implementation and regression test are present in release `v2026.4.23`.

## Evidence

- **Port liveness check precedes PID trust:** `resolveGatewayOwnerStatus()` probes the configured port first and returns `"dead"` when the port is free, even if the PID still exists. That directly covers the reported 'PID alive but port not listening' case.
  - file: [src/infra/gateway-lock.ts:193](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/infra/gateway-lock.ts#L193)
  - command: `nl -ba src/infra/gateway-lock.ts | sed -n '186,214p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Stale lock is removed and startup retries:** When the prior lock owner resolves as `dead`, `acquireGatewayLock()` deletes the stale lock file and continues acquisition instead of hanging on the old PID.
  - file: [src/infra/gateway-lock.ts:299](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/infra/gateway-lock.ts#L299)

- **Normal gateway startup passes the resolved port into lock acquisition:** `openclaw gateway run` passes the configured port into `runGatewayLoop()`, and `runGatewayLoop()` calls `acquireGatewayLock({ port: params.lockPort })`, so the port probe is active on the exact startup path from the report.
  - file: [src/cli/gateway-cli/run.ts:655](https://github.com/openclaw/openclaw/blob/9b18ae877018a34933e27bb7b2986cb97e9602aa/src/cli/gateway-cli/run.ts#L655)

## Close Comment

Thanks for the report. A Codex review against current `main` found that this case is already handled.

- `src/infra/gateway-lock.ts:193` now probes the configured port before trusting the PID; if the port is not listening, the prior owner is treated as dead.
- `src/infra/gateway-lock.ts:299` removes that stale lock and retries acquisition instead of waiting indefinitely on the old PID.
- The normal `openclaw gateway run` path passes the resolved port into lock acquisition via `src/cli/gateway-cli/run.ts:655` and `src/cli/gateway-cli/run-loop.ts:43`, so this applies to the startup flow you reported.
- `src/infra/gateway-lock.test.ts:249` includes regression coverage for the exact scenario: owner PID alive, configured port free.
- This implementation is already present in release `v2026.4.23` (`9b18ae877018a34933e27bb7b2986cb97e9602aa`).
