## Summary

Keep open. Current `main` still serves cached cron state on non-force paths, so external edits are not reloaded for `cron.list`, `cron.status`, or manual `cron.run`. The PR's original store-mtime approach also does not cleanly match today's architecture because runtime state now lives in the `jobs-state.json` sidecar, so a store-file-only check would miss the most relevant manual recovery edits.

## Evidence

- **Cached fast path still skips reload on current main:** `ensureLoaded()` returns immediately whenever `state.store` is already populated and `forceReload` is not set. That means normal read/manual-run paths do not restat or reread disk before using cached cron data.

So `status`, `list`, and manual `run` all trust the cached in-memory store unless some other path forced a reload first.
  - file: [src/cron/service/ops.ts:74](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/cron/service/ops.ts#L74)
