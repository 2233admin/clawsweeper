## Summary

WebChat live streaming is already implemented on current `main` and was present in the `v2026.4.22` release; the issue's proposed `channels.web*` config shape does not apply because WebChat is an internal Gateway UI channel, not a configurable outbound channel.

## Evidence

- **WebChat is not a configurable outbound channel:** Docs explicitly say `webchat` is the internal WebChat UI channel and is not a configurable outbound channel, so `channels.web` / `channels.webchat` is not the architecture for this surface.
  - file: [docs/channels/channel-routing.md:16](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/docs/channels/channel-routing.md#L16)
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Gateway emits streamed chat deltas for WebChat:** The Gateway builds `chat` events with `state: "delta"` and broadcasts them to WebChat clients, then flushes any buffered delta before the final event.
  - file: [src/gateway/server-chat.ts:732](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/gateway/server-chat.ts#L732)
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **WebChat UI renders streamed partial text:** The UI stores `delta` payload text in `chatStream`, then replaces it with the final assistant message or persists the streamed partial when needed.
  - file: [ui/src/ui/controllers/chat.ts:451](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/ui/src/ui/controllers/chat.ts#L451)
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Tests cover WebChat streaming path:** There is an integration test asserting that agent events stream to WebChat clients when a run context is registered.
  - file: [src/gateway/server.agent.gateway-server-agent-b.test.ts:564](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/src/gateway/server.agent.gateway-server-agent-b.test.ts#L564)
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)

  - file: [docs/web/webchat.md:25](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/docs/web/webchat.md#L25)

## Risks / Open Questions

- If the reporter specifically wanted a user-configurable on/off or mode selector for WebChat to mirror `channels.<channel>.streaming`, that narrower config-parity request is not what current docs expose; current WebChat streaming appears to be built into the internal UI path instead.

## Close Comment

Closing from a Codex review: current `main` already implements live WebChat streaming. The Gateway emits `chat` `delta` events for WebChat and flushes them before the final message (`src/gateway/server-chat.ts:732`, `src/gateway/server-chat.ts:789`), and the WebChat UI consumes those deltas through `chatStream` and finalizes them into the chat transcript (`ui/src/ui/controllers/chat.ts:451`). The docs also clarify that `webchat` is an internal Gateway UI channel, not a configurable outbound `channels.web*` surface (`docs/channels/channel-routing.md:16`), so the requested config shape is not how this feature is modeled. This behavior is already present in the released `v2026.4.22` tag as well.
