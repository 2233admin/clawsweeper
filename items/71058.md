## Summary

Keep open. Current main still models Microsoft Teams as one bundled channel configuration with one Azure Bot credential set and one default account. The requested multi-bot, per-agent identity support is a concrete missing capability for the existing Teams plugin, not an already-shipped fix or a duplicate in the provided related context.

## Best Possible Solution

Keep the issue open and implement it in the bundled `extensions/msteams` plugin if maintainers want Teams parity with other multi-account channels. The likely path is backward-compatible `channels.msteams` single-object support plus an account-map form using existing channel account helpers, then thread `accountId` through credential resolution, monitor/JWT routing, outbound/probe/status/setup, SSO/token stores, and agent binding docs/tests.

## Risks / Open Questions

- Teams uses Bot Framework auth, proactive send, SSO/delegated tokens, and local stores; multi-bot support will need account-scoped credential, token, webhook, and storage handling to avoid cross-account identity bleed.
- The proposed array shape does not match the repo's current multi-account convention; maintainers should likely prefer `channels.msteams.accounts.<accountId>` plus `defaultAccount` and existing agent routing bindings.
- Setup, doctor, status, probe, docs, and tests need to move together because the current Teams plugin has default-account assumptions in both runtime and setup surfaces.
