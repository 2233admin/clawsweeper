## Summary

Keep open. Current main still models Microsoft Teams as one configured bot identity per gateway, even though generic routing can bind channel/account/peer scopes to agents. The requested user-visible capability, multiple Azure/Teams bot identities on one gateway with per-agent binding, is not implemented and is actionable in the bundled `@openclaw/msteams` plugin rather than a ClawHub-only close.

## Best Possible Solution

Keep #71058 open and implement it in the bundled `@openclaw/msteams` channel plugin. Prefer a backwards-compatible `channels.msteams.accounts`/`defaultAccount` design over an array, pass the resolved Teams account ID through inbound/outbound routing, support per-account agent binding through the existing binding/routing contracts or a documented account-level field, update setup/status/docs/schema/tests, and preserve the current single-bot config as the default legacy form.

## Risks / Open Questions

- A correct implementation needs careful Bot Framework JWT audience/app binding per bot identity so one webhook cannot accept tokens for the wrong configured app.
- Backwards compatibility must preserve the current single-object `channels.msteams` form and existing `MSTEAMS_*` environment-variable behavior.
- The issue proposes an array shape, but existing OpenClaw channel patterns favor `accounts` maps plus `defaultAccount`; maintainers should choose the repo-native shape even if it differs from the proposal.
