## Summary

Keep open. Current `main` still only supports global `session.maintenance`, so the requested per-agent override feature is not implemented on shipped code, and the PR discussion leaves unresolved behavior risks around shared/manual session-store targets and partial write-path adoption.

## Evidence

- **Current main resolves maintenance from global session config only:** `resolveMaintenanceConfig()` on `main` loads only `loadConfig().session?.maintenance` and has no `agentId` or per-agent merge path, so the shipped behavior does not match the PR feature.
  - file: [src/config/sessions/store-maintenance-runtime.ts:8](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/config/sessions/store-maintenance-runtime.ts#L8)
  - command: `nl -ba src/config/sessions/store-maintenance-runtime.ts | sed -n '1,30p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Agent config type has no per-agent maintenance field on main:** `AgentConfig` on `main` goes from `embeddedPi` to `sandbox` without any `maintenance` property, so operators cannot declare `agents.list[].maintenance` in the current typed config surface.
  - file: [src/config/types.agents.ts:70](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/config/types.agents.ts#L70)

  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Runtime schema on main does not accept per-agent maintenance:** `AgentEntrySchema` currently defines `id`, `default`, `name`, `groupChat`, `subagents`, `sandbox`, `params`, `tools`, and `runtime`, but no `maintenance` key, so the feature is not already implemented via schema support.
  - file: [src/config/zod-schema.agent-runtime.ts:812](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/config/zod-schema.agent-runtime.ts#L812)

The PR review correctly flagged that applying per-agent maintenance by `target.agentId` would need extra handling to avoid using one agent's policy for a shared or manually selected store.

That makes the open PR still active/incomplete work rather than a candidate for conservative auto-close.

## Risks / Open Questions

- If merged as discussed in the review, per-agent cleanup could apply the wrong policy to deduped shared stores or `--store` targets unless those cases are explicitly handled.
- The PR discussion also indicates normal runtime session-store writes were not fully adopted yet, so the feature may behave inconsistently between manual cleanup and ongoing writes until follow-up wiring lands.
