## Summary

Current main still has a meaningful macOS LaunchAgent restart ordering gap relative to #70801. `restartLaunchAgent` mitigates stale listeners with port cleanup before `kickstart -k`, but it does not implement the requested two-phase launchd stop/wait/start flow, and current tests explicitly assert the normal restart path uses kickstart with no bootout/bootstrap.

## Best Possible Solution

Keep #70801 open. The maintainer path should be a focused macOS launchd restart design: either implement a true two-phase stop/wait/start sequence in `src/daemon/launchd.ts` and mirror it in `src/daemon/launchd-restart-handoff.ts`, or explicitly document why the current stale-port cleanup is the intended replacement. If implemented, add tests that assert ordering for normal restart, installed-but-unloaded fallback, and detached self-restart.

## Evidence

- **restart path remains kickstart-first:** The normal macOS restart path resolves the service, optionally runs stale port cleanup, enables the LaunchAgent, then calls `launchctl kickstart -k`. `bootout`/`bootstrap` is only used after a not-loaded kickstart failure, not as a two-phase stop-wait-start restart sequence.

## Risks / Open Questions

- A future fix must preserve the installed-but-unloaded recovery behavior that currently bootstraps after `kickstart` reports the service missing.
- A future fix must handle the separate detached self-restart shell path, since it currently has its own kickstart/bootstrap ordering outside `restartLaunchAgent`.
- The existing stale-port cleanup may already reduce the observable race, so the product decision is whether that mitigation is sufficient or whether launchd-level two-phase ordering is still required.
