## Summary

Keep open. Current `main` does not implement the requested two-phase macOS LaunchAgent restart. `restartLaunchAgent` still uses `launchctl kickstart -k` as the primary path, and the existing unit tests explicitly assert that the normal restart path does not call `bootout` or `bootstrap`. The related PR #43961 was closed unmerged in the provided GitHub context, so it does not supersede this issue.

## Best Possible Solution

Keep this issue open and implement a deliberate macOS LaunchAgent two-phase restart in `src/daemon/launchd.ts`: stop or boot out the service, confirm the old runtime/PID or port is gone, then bootstrap/start or kickstart the service. Update `src/daemon/launchd.test.ts` to assert the new ordering and update `src/daemon/launchd-restart-handoff.ts` plus its tests if the self-restart handoff needs the same ordering.

## Risks / Open Questions

- Changing the happy path to bootout/wait/bootstrap will intentionally replace behavior that current unit tests assert, so tests need to be updated rather than treated as unchanged proof.
- The detached handoff path has its own shell-script restart sequence and should be handled explicitly so self-restart via the LaunchAgent still completes after the caller exits.
- The existing stale-port cleanup is only a mitigation before `kickstart`; it is not equivalent to the issue's requested launchd-owned two-phase stop and start.
