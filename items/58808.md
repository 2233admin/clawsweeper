## Summary

Keep this PR open. Current main still has the reported plugin SDK seam gap: channel agent-tool factories receive only config, while adjacent shared message-tool discovery/execution already receives trusted sender context. The PR is a narrow additive candidate for the DMWork sender-aware access-control use case and is not obsolete, implemented, or ClawHub-only.

## Best Possible Solution

Review and land this PR, or an equivalent core SDK change: extend ChannelAgentToolFactory/listChannelAgentTools params with optional requesterSenderId and senderIsOwner, thread options.senderId/options.senderIsOwner from createOpenClawCodingTools, and add a narrow regression test around a fake channel plugin agentTools factory. The change belongs in core because third-party channel plugins need a generic trusted-context seam; it should not be closed as ClawHub-only.
