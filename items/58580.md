## Summary

Keep #58580 open. The report is recent, concrete, and core-owned. Current main has stuck-session diagnostics, manual abort/reset/delete paths, and a narrow post-restart recovery path for lock-marked running sessions, but I did not find the requested health supervisor that detects repeated inbound messages without successful responses and auto-clears only the affected channel/session.

## Best Possible Solution

Keep the issue open and implement a gateway-owned session health supervisor. It should track per-session inbound attempts, in-flight state, and last successful response/delivery; after a configurable threshold with received messages and no success, emit a warning/diagnostic event, abort any active run if present, clear queued work for that session, and perform a targeted soft reset/prune that preserves routing metadata while clearing corrupted context/token/model override state. Add focused tests around auto-reply/Telegram-style dispatch, active-run false-positive avoidance, and session-store mutation so only the affected session is reset.
