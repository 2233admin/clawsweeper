## Summary

Keep open. Current `main` still hashes raw `exec` result details in loop detection, and those details include volatile fields (`durationMs`, `pid`, `startedAt`, `sessionId`) plus running-status text with session metadata, so identical repeated `exec` calls can still evade the no-progress/global-circuit-breaker logic this PR is trying to fix.

## Evidence

- **Current main still uses the generic fallback hash for exec results:** `hashToolOutcome()` has special handling for `process` poll/log, but all other tools fall through to `digestStable({ details, text })`. There is no `exec`-specific branch on current `main`, so raw `exec` details are still part of the result hash.
  - file: [src/agents/tool-loop-detection.ts:222](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/agents/tool-loop-detection.ts#L222)

  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Loop detection requires repeated identical result hashes:** `getNoProgressStreak()` only increments while successive matching calls keep the same `resultHash`; a changed hash breaks the streak, and `detectToolCallLoop()` only escalates to the global circuit breaker at 30 identical no-progress outcomes.
  - file: [src/agents/tool-loop-detection.ts:288](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/agents/tool-loop-detection.ts#L288)
