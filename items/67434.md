## Summary

Keep open. Current `main` now documents the strict-agentic `update_plan` auto-enable behavior and the `planTool: false` opt-out, so the documentation gap reported in the comments is partly addressed. But the runtime still only treats an attempt as planning-only when it has no non-`update_plan` tool activity; because attempt results aggregate all tool calls, a long burst of `update_plan` calls followed by a later `read` still bypasses the blocked/retry path. I did not find a dedicated default-on cap for consecutive `update_plan` calls.

## Evidence

This matches the reporter's observation that GPT-5/Codex strict-agentic runs can expose `update_plan` without an explicit `planTool: true`.
