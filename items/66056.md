## Summary

Keep open. Current main still has hardcoded English user-facing runtime and sanitizer error messages, no `messages.locale` or `messages.errorLocale` config/schema field, and no core locale catalog/formatter. The related pairing-locale PR #66055 was closed unmerged, and Control UI i18n is browser-local rather than gateway/channel runtime error localization.

## Best Possible Solution

Keep this issue open for maintainer product/design work. The likely core path is a small user-facing error catalog with English fallback, a documented `messages.locale` or `messages.errorLocale` config/schema/help surface, locale-aware formatting wired through sanitizer and reply-runner failure paths, and focused regression tests. If maintainers prefer plugin-owned localization, first add a structured error-key and locale-context hook so plugins do not need brittle English-string rewrites.
