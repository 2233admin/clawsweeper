## Summary

Keep open. Current `main` does not implement the credential-rejection hardening, and the PR as submitted still has unresolved merge blockers: it would introduce a raw secret-logging path for invalid per-job webhooks, leave legacy `notify + cron.webhook` jobs failing silently, and `openclaw doctor --fix` still rewrites legacy notify jobs to whatever `cron.webhook` contains without validating that target. There is also no automated test coverage for `normalizeHttpWebhookUrl` today.

## Evidence

- **Current main still accepts credential-bearing webhook URLs:** `normalizeHttpWebhookUrl` only checks the parsed protocol and then returns the trimmed URL. There is no rejection for `parsed.username` or `parsed.password`, so the hardening requested by the PR is not present on current `main`.

## Risks / Open Questions

- Merging the PR as-is would create a new secret leak in cron invalid-webhook warnings by logging raw `delivery.to` values with embedded credentials.
- Legacy `notify: true` jobs that still depend on `cron.webhook` would stop posting callbacks without a clear warning path.
- `openclaw doctor --fix` would continue migrating legacy jobs to unvalidated webhook targets, so the recommended repair path could break existing installs.
- Without dedicated validator tests, follow-up fixes for the above regressions would remain easy to accidentally regress again.
