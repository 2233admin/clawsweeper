## Summary

Keep this issue open. Current main still has a reachable /status path where channels.modelByChannel selects the active provider/model, but a stale positive sessionEntry.contextTokens value is treated as the authoritative contextTokensOverride before the freshly resolved channel model context window is used. The earlier ClawSweeper comment accurately describes the remaining gap, and I did not find a targeted regression test or source change that closes the channel-override case.

## Best Possible Solution

Keep this issue open and fix either the /status context resolution or the channels.modelByChannel application path so a channel-selected model ignores, clears, or recomputes stale sessionEntry.contextTokens when there is no explicit session model override. Add a regression test with a stale 1M session context value, a channel override to a smaller MiniMax-style model, and /status showing the active model's configured context window.
