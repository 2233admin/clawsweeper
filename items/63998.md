## Summary

Keep this issue open. Current main still has partial mitigations only: an opt-in active-transcript byte trigger, restart-aborted session recovery, and bounded bootstrap marker scanning. The reported default crash-restart/OOM failure mode is not fully fixed because restart/run/recovery paths can still materialize the full transcript before quarantine or rotation, and there is still no count-based circuit breaker for accumulated `openclaw:bootstrap-context:full` entries.

## Best Possible Solution

Keep this issue open and implement a core pre-open recovery guard for pathological active transcripts. The guard should stat/classify transcript files before `repairSessionFileIfNeeded`, `readSessionMessages`, or `SessionManager.open`, rotate/quarantine or compact into a bounded successor/degraded session, cap repeated bootstrap completion markers, reuse media offload/deduplication where applicable, and add focused regression tests for the crash-restart loop.

## Risks / Open Questions

- Current partial mitigations may help configured sessions but do not protect the default restart path from loading a pathological transcript before recovery.
- The restart recovery loop adds useful interrupted-run handling, but it can still read a large transcript and then re-enter normal run handling rather than quarantining or rotating first.
- A future fix must avoid full-file reads in the first recovery decision, preserve successful compaction/rotation behavior, and add regression coverage for repeated restart attempts and bootstrap marker accumulation.
