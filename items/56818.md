## Summary

Keep open. Current `main` still routes Feishu card-action callbacks through a synthetic `card-action-<token>` message id and feeds that id into the streaming reply path, so the reported `Invalid open_message_id` failure path still appears live rather than fixed on `main`.

## Evidence

- **Card actions synthesize a fake message id:** `buildSyntheticMessageEvent()` constructs the inbound callback as `message_id: \`card-action-${event.token}\`` instead of carrying any real Feishu message id from the callback payload.
  - file: [extensions/feishu/src/card-action.ts:119](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/feishu/src/card-action.ts#L119)
  - command: `nl -ba extensions/feishu/src/card-action.ts | sed -n '105,126p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Reply targeting still uses the synthetic callback message id:** Feishu reply routing derives `replyTargetMessageId` from `ctx.messageId`, then passes that value into `createFeishuReplyDispatcher()` as `replyToMessageId`. For card-action callbacks, `ctx.messageId` comes from the synthetic `card-action-...` event above.
  - file: [extensions/feishu/src/bot.ts:1034](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/feishu/src/bot.ts#L1034)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '1019,1045p'; nl -ba extensions/feishu/src/bot.ts | sed -n '1085,1096p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Streaming replies still send `replyToMessageId` as the Feishu reply target:** `createFeishuReplyDispatcher()` forwards `replyToMessageId` into `streaming.start()`, and `FeishuStreamingSession.start()` chooses reply mode whenever `replyToMessageId` is present, calling `client.im.message.reply({ path: { message_id: replyToMessageId } ... })`. If that id is `card-action-...`, it is not a real Feishu message id.
  - file: [extensions/feishu/src/reply-dispatcher.ts:287](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/feishu/src/reply-dispatcher.ts#L287)

  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Current tests codify the problematic card-action reply id flow:** The lifecycle test on current `main` explicitly expects card-action handling to call the reply dispatcher with `replyToMessageId: "card-action-tok-card-once"`. Separate dispatcher tests assert that `replyToMessageId` is passed through to `streaming.start()`.
  - file: [extensions/feishu/src/monitor.card-action.lifecycle.test-support.ts:180](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/feishu/src/monitor.card-action.lifecycle.test-support.ts#L180)
