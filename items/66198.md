## Summary

Keep open. Current main still routes Signal inbound text into raw model-facing fields (`BodyForAgent`/`BodyStripped`), while this PR wraps only legacy `formatInboundEnvelope(...)` body strings. The patch is focused and security-relevant, but it needs rework before merge.

## Best Possible Solution

Keep this PR open, but require a rework or replacement before merge. The fix should protect the actual Signal model-facing path, likely by deriving `BodyForAgent`/prompt supplemental context from wrapped untrusted user text or by using `UntrustedContext` consistently, while keeping `CommandBody`/`BodyForCommands` appropriate for command parsing. It should wrap pending history once or use stable marker IDs, keep generated message IDs outside untrusted blocks, and add focused Signal tests for direct messages, group messages, debounced bodies, quote-only context, and pending group history.

## Risks / Open Questions

- Merging as-is would create a false sense of Signal prompt-injection protection while raw `BodyForAgent` still feeds the normal model prompt.
- The submitted history wrapping happens in the render-time `formatEntry` path; because `wrapExternalContent` generates random marker IDs, repeated prompt assembly would make unchanged history bytes unstable.
- The submitted history body places generated `[id:...]` metadata inside the untrusted block, mixing trusted application metadata with attacker-controlled text.
- A correct fix must preserve raw command parsing fields while marking only model-facing user content or structured supplemental context as untrusted.
