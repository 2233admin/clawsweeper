## Summary

Current `main` routes heartbeat turns through the normal session initialization path, which regenerates a missing `sessionId` and persists a resolved `sessionFile` before history is read. I could not support the reported `heartbeat` -> `sessionId/sessionFile = null forever` behavior from the current code.

## Close Comment

Closing after Codex review because I could not reproduce this against current `main`.

Concrete evidence: heartbeat runs still go through `getReplyFromConfig()` (`src/infra/heartbeat-runner.ts:1062`), which calls `initSessionState()` (`src/auto-reply/reply/get-reply.ts:265`). That init path only reuses an existing session when `entry.sessionId` is truthy; otherwise it generates a new UUID (`src/auto-reply/reply/session.ts:427`, `src/auto-reply/reply/session.ts:489`) and then persists both `sessionId` and `sessionFile` via `resolveAndPersistSessionFile()` (`src/auto-reply/reply/session.ts:720`, `src/config/sessions/session-file.ts:18`). `chat.history` still reads from the persisted `entry.sessionId` (`src/gateway/server-methods/chat.ts:1891`), so the reported 'heartbeat-origin sessions stay null and history stays empty' path does not match the current mainline flow.

If this is still reproducible on a current build, please open a new issue with a minimized repro from current `main`, including the exact session key, whether any plugin/custom wrapper is involved, and a redacted `sessions.json` entry before/after one heartbeat turn.
