## Summary

Current main still has the reported scheduler shape: followup drains run until the queue is empty, each followup executes a full embedded agent turn on the same per-session FIFO lane as fresh inbound work, and new busy-session messages are appended as followups without a fairness cap or user-message priority. The related #65504 and #66638 items are adjacent responsiveness/notification work, but they do not supersede this specific followup-drain starvation bug.

## Best Possible Solution

Keep this open and fix it in the core auto-reply/session scheduler. The likely implementation is a bounded or fair followup drain, explicit priority or preemption for fresh user messages over accumulated followup/system-event work on the same session key, phase-aware aggregate timing for queue/preflight/provider/post-turn work, and a focused regression test that enqueues multiple followup turns plus a later fresh inbound message on the same session key and proves the fresh turn is not starved behind the entire backlog. Keep #65504 and #66638 linked as related responsiveness and notification-delivery work, but do not treat them as superseding this scheduler fairness bug.
