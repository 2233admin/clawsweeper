## Summary

Keep open. The discussion identifies a concrete gateway freeze mode from an oversized active session JSONL, and current main still routes embedded gateway runs and preflight compaction through whole-file transcript repair plus `SessionManager.open` without a default bounded guard. Main has useful update-channel docs and an opt-in transcript byte trigger, but that does not robustly fix the reported default gateway/channel starvation path.

## Best Possible Solution

Keep this issue open and implement bounded large-active-transcript recovery in the gateway embedded run path. Add a pre-open size guard before whole-file repair or `SessionManager.open`, then stream/yield repair, rotate/compact safely, or fail fast with an actionable recovery message while keeping HTTP/channel loops responsive. Add regression coverage for an oversized JSONL proving gateway/channel work is not starved, and document the operator setting plus recovery path on the public docs site.

## Evidence

- **Issue discussion gives a concrete root cause:** A commenter reports that after the gateway acquires `agent:main:main`, an 89 MB / ~40,000-line main session JSONL can monopolize CPU, stop logs, and freeze Telegram, WeChat, and Feishu handlers until the file is moved aside. That is specific enough to treat this as a live gateway/session recovery bug, not just a broad stability complaint.
- **Embedded gateway run still opens the active transcript after whole-file repair:** The embedded attempt path calls `repairSessionFileIfNeeded`, stats the session file, prewarms it, then calls `SessionManager.open(params.sessionFile)`. I did not find a size cap or fail-fast guard in this path before the repair/open sequence.

## Risks / Open Questions

- Closing now would leave default users dependent on manual transcript removal or explicit compaction/rotation configuration after an active JSONL grows large enough to stall the gateway.
- The opt-in byte threshold may still perform expensive repair/open work on the same oversized transcript before it can rotate or compact it.
- Related closed Windows gateway stability work such as #49865 does not supersede the large-active-transcript freeze described in this issue.
