## Summary

Keep open. Current main does not implement `OPENCLAW_WRAPPER` handling, the linked #69400 problem remains a core daemon install/update behavior gap rather than a ClawHub/plugin item, and the PR is still useful but needs changes before it is the best fix.

## Best Possible Solution

Keep the PR open and revise it into a complete core daemon fix: add an explicit, validated `OPENCLAW_WRAPPER` install contract for gateway and node services, persist/read it through the install plan's merged service environment or a documented `--wrapper` option, pass that environment into `resolveGatewayProgramArguments`/`resolveNodeProgramArguments`, validate regular file plus executability where supported, and cover reinstall/update persistence with focused tests. Do not close in favor of #39457; that issue is a broader post-update hook request and does not replace this narrower ProgramArguments wrapper support.

## Risks / Open Questions

- As submitted, the PR appears to read `process.env` directly; that would not preserve a wrapper already stored in the service env during update/reinstall unless the variable is also exported in the installer shell.
- The current service-env preservation path drops generic `OPENCLAW_*` keys, so the implementation likely needs an explicit `OPENCLAW_WRAPPER` service-env contract or a CLI flag persistence path, not only a program-args helper.
- The Greptile executable-bit comment is a reasonable hardening request for operator diagnostics, but it does not make the feature obsolete.
