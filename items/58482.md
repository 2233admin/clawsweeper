## Summary

Keep #58482 open. Current main has moved provider-specific batch logic into `extensions/{openai,google,voyage}/embedding-batch.ts`, but the requested AbortSignal cancellation behavior is still not implemented end to end: the shared batch execution contract has no `signal`, memory-core does not pass one to provider batch embeds, and OpenAI/Voyage/Gemini polling still uses uncancelable sleep/request paths. The PR is not merge-ready as-is because it targets older paths and the review discussion identified real wiring/listener issues, but the underlying bug is still a valid OpenClaw code change rather than implemented, duplicated, or out of scope.

## Best Possible Solution

Rework this PR or a canonical follow-up against the current extension-owned batch implementations: add `signal?: AbortSignal` to `MemoryEmbeddingBatchOptions` and `EmbeddingBatchExecutionParams`, propagate it from the memory-core caller through the OpenAI/Gemini/Voyage adapters and `run*EmbeddingBatches`, use a shared abort-aware sleep helper with already-aborted and listener-cleanup handling, pass `signal` into all batch upload/create/status/download HTTP calls, and add focused tests for cancellation behavior. Do not auto-close as implemented or duplicate until that current-main path exists or a newer canonical issue/PR tracks it.

## Risks / Open Questions

- The PR targets older `packages/memory-host-sdk/src/host/batch-openai.ts`, `batch-voyage.ts`, and `batch-gemini.ts` paths; current main now keeps provider batch code under `extensions/openai`, `extensions/voyage`, and `extensions/google`, so the patch likely needs a rebase/rewrite.
- A correct fix needs more than polling sleep cancellation: it should propagate the signal from memory-core into provider adapters, handle already-aborted signals, remove abort listeners on the normal resolve path, and pass the signal into upload/create/status/download HTTP requests.
- Regression tests should cover abort before polling sleep, abort during polling sleep, in-flight HTTP cancellation, and listener cleanup for at least one shared helper or each provider-specific path.
