## Summary

Current main fixes the local workspace/avatar-route side of this report, but it still deliberately does not render the issue's remote http(s) avatar example in Control UI. The CLI/config contract continues to accept remote avatar values, so a real product/security decision remains rather than a conservative close.

## Best Possible Solution

Keep this open for maintainer product/security judgment. Either implement remote avatars through an authenticated same-origin proxy/cache with CSP and privacy controls, or make remote avatars explicitly unsupported for Control UI by tightening CLI help, config schema/help, validation, and setup/UI feedback while preserving the current local workspace avatar route, auth handling, data/blob support, and oversized/outside-workspace diagnostics.

## Risks / Open Questions

- Directly rendering arbitrary remote avatar URLs would allow operator browsers to fetch third-party image URLs, which conflicts with the current CSP/privacy hardening.
- A simple 302 redirect approach like the closed related PR would not preserve the current same-origin image policy.
- Leaving CLI/config acceptance unchanged lets users configure http(s) avatars that Control UI intentionally will not render.
