## Summary

Keep #40024 open. Current main still has a reproducible code-path risk for the reported configuration: a configured custom provider with only `baseUrl` falls back to `openai-responses`, while the Chat Completions POST path is only selected when the resolved model API is `openai-completions`. Loopback provider POSTs also still require explicit private-network opt-in, so the reporter's “health check reaches proxy but no `/v1/chat/completions` POST” symptom is not conclusively fixed on main.

## Best Possible Solution

Keep the issue open and make the product contract explicit in code. A robust fix would either infer or select `openai-completions` for custom local `/v1` providers that do not support Responses, or validate `models.providers.<id>.api` plus `request.allowPrivateNetwork` early and fail with an actionable configuration error instead of timing out without a POST. Add a regression test for a `local-agent-proxy` provider at `http://127.0.0.1:3000/v1` covering the chosen transport path.
