## Summary

Keep this issue open. Current main still opens the memory DB with extension loading enabled and has an in-process cosine fallback, but the Windows sqlite-vec loader path reported here is not fixed: the loader still delegates to sqlite-vec without a Windows .dll suffix retry or post-load vec0 health probe, and indexing still skips chunks_vec when vector readiness fails.

## Best Possible Solution

Keep this issue open unless maintainers explicitly consolidate it into #65704 or a merged successor to #69059. The likely fix belongs in both sqlite-vec loader copies plus memory-core readiness: validate usable vec0 support after load, decide and test the Windows .dll suffix retry, preserve the in-process cosine fallback, and add focused mocked or Windows regression coverage for failed-load and load-success-but-no-usable-vec0 cases.
