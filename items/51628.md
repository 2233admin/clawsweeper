## Summary

Keep open. Current `main` appears to have fixed the transcript-side duplicate `delivery-mirror` append by propagating mirror idempotency keys and deduping transcript writes, but the delivery-queue recovery path still blindly replays pending entries and does not classify uncertain Telegram `sendMessage` failures as unsafe to resend. That leaves the user-visible duplicate-delivery part of #51628 unresolved on `main`.

## Evidence

- **Recovery still replays queued entries without Telegram-specific safety gating:** `recoverPendingDeliveries()` reloads each pending entry and immediately calls `drainQueuedEntry()`, which replays the stored delivery payload. The recovery params include the original `mirror` data and `skipQueue: true`, but there is no check on channel, `lastError`, or uncertain-send state before replay.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:112](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/infra/outbound/delivery-queue-recovery.ts#L112)
  - command: `nl -ba src/infra/outbound/delivery-queue-recovery.ts | sed -n '112,129p'; nl -ba src/infra/outbound/delivery-queue-recovery.ts | sed -n '402,421p'`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Queue storage records only a string `lastError`, not an unsafe-to-replay delivery state:** When a send fails, `failDelivery()` just increments `retryCount`, stamps `lastAttemptAt`, and saves a free-form `lastError`. There is no persisted state for "uncertain delivery already may have reached Telegram" that recovery could use to quarantine instead of replaying.
