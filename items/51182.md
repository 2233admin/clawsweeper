## Summary

Keep open. Current `main` has adjacent Telegram replay mitigations, but the inspected code does not give strong maintainer-grade proof that issue #51182 is fully fixed. The persisted update watermark is still written asynchronously, startup only performs a single persisted-offset confirmation probe, and the persisted sent-message cache is only consumed for reaction ownership filtering rather than outbound delivery idempotency.

## Evidence

- **Persisted offset writes are still async fire-and-forget:** `extensions/telegram/src/bot-core.ts:295` computes a safe watermark, then `extensions/telegram/src/bot-core.ts:329` dispatches `opts.updateOffset?.onUpdateId?.(safe)` through `Promise.resolve().then(...)` without awaiting shutdown/barrier semantics. `extensions/telegram/src/bot-core.ts:404` only triggers that async persistence after update completion.
  - file: [extensions/telegram/src/bot-core.ts:295](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/telegram/src/bot-core.ts#L295)
- **Startup confirmation is a single probe using last persisted id + 1:** `extensions/telegram/src/polling-session.ts:228` runs `bot.api.getUpdates({ offset: lastUpdateId + 1, limit: 1, timeout: 0 }, ...)` before the runner starts, and `extensions/telegram/src/polling-session.ts:244` immediately proceeds into the polling cycle. The inspected code does not show a stronger replay-drain or restart barrier for a stale-on-disk offset.
  - file: [extensions/telegram/src/polling-session.ts:228](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/telegram/src/polling-session.ts#L228)

I did not find a delivery-side dedupe check using this cache.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:883](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/telegram/src/bot-handlers.runtime.ts#L883)
