## Summary

Keep open. Current main still exposes `llm_input`, `llm_output`, and `after_tool_call` as void/observe-only hooks, and the embedded runner/adapter paths ignore returned prompt, response, block, or tool-result mutations. The open narrower PR #61485 overlaps the `llm_input`/`llm_output` portion, but it does not supersede this PR's broader adapter-side `after_tool_call` result-mutation proposal.

## Best Possible Solution

Keep this PR open unless maintainers explicitly reject the broader direction or land a canonical replacement. The best path is to review or land a focused mutable `llm_input`/`llm_output` implementation such as #61485, then split or redesign adapter-side `after_tool_call` result mutation as a separate plugin API decision that preserves exactly-once hook dispatch and defines ordering with persisted tool results.

## Risks / Open Questions

- This PR overlaps open #61485 for mutable `llm_input`/`llm_output`; maintainers should avoid landing two independent hook-contract implementations.
- Adapter-side `after_tool_call` result mutation conflicts with the current exactly-once dispatch model unless the API defines how it composes with the subscription handler and `tool_result_persist`.
- Prompt/systemPrompt/output/tool-result mutation is security-sensitive and needs review for `allowPromptInjection`, audit logging, payload cloning, result validation, failover/block semantics, and prompt-image re-resolution before merge.
