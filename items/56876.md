## Summary

Current `main` does not implement binary-document preprocessing on the `read` tool. OpenClaw still delegates `read` to the upstream pi tool, and the current upstream implementation only special-cases images; non-image files are still decoded as UTF-8 text, which leaves the reported `.pdf/.docx/.xlsx` failure mode unaddressed.

## Evidence

- **OpenClaw still builds `read` from the upstream tool:** `createOpenClawCodingTools()` creates `read` with `createReadTool(workspaceRoot)` and then wraps it; there is no OpenClaw-side document extraction or binary-file fallback added on top.
  - file: [src/agents/pi-tools.ts:473](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/pi-tools.ts#L473)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '430,520p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **The OpenClaw wrapper only adds paging and image handling:** `createOpenClawReadTool()` only applies adaptive paging, strips truncation metadata, normalizes read-image MIME, and sanitizes images. The sandbox read operations only return detected MIME when it starts with `image/`, so binary documents are not handled specially here.
  - file: [src/agents/pi-tools.read.ts:674](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/pi-tools.read.ts#L674)
  - command: `nl -ba src/agents/pi-tools.read.ts | sed -n '674,716p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Upstream `read` still decodes non-images as UTF-8 text:** Inspected the current upstream `pi-coding-agent` `read.ts`: the only MIME probe is `detectImageMimeType`; if that probe does not return an image MIME, the tool executes `const textContent = buffer.toString("utf-8")`. That matches the issue's raw-binary/garbled-text path for `.pdf`, `.docx`, and `.xlsx`.
  - command: `curl -Lsf https://raw.githubusercontent.com/badlogic/pi-mono/main/packages/coding-agent/src/core/tools/read.ts | nl -ba | sed -n '1,220p'`
- **Current tests do not cover binary-document reads:** The regression coverage in `pi-tools.create-openclaw-coding-tools.test.ts` verifies PNG image behavior and plain-text file behavior only. There is no proof on `main` that `read` preprocesses binary office/PDF documents.
  - file: [src/agents/pi-tools.create-openclaw-coding-tools.test.ts:463](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/pi-tools.create-openclaw-coding-tools.test.ts#L463)

- **PDF support exists as a separate tool, not a `read` fix:** OpenClaw registers a dedicated `pdf` tool via `createPdfTool()`, which reinforces that PDF handling is currently separate from the built-in `read` path rather than an implemented fix for this issue.
