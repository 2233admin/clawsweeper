## Summary

Keep #71973 open. Current main has the earlier broad prompt-cache work: the system prompt only includes timezone and has a stable/dynamic cache boundary. But the specific request is still real: heartbeat, cron, and post-compaction paths still append an exact wall-clock `Current time:` line into per-turn prompt content, and post-compaction explicitly does so unconditionally.

## Best Possible Solution

Keep the issue open as a focused optimization task. A maintainer-quality fix should preserve the stable system-prompt boundary, then audit heartbeat, cron, and post-compaction callers to either omit exact wall-clock time when not needed or move it into a later/runtime-only channel that does not disturb provider cacheable prefixes. It should add regression coverage proving timezone remains available, date-sensitive memory/startup flows still work, and recurring automation prompts do not unnecessarily include fresh timestamps.

## Risks / Open Questions

- The timestamp line appears to be intentional for date-sensitive startup, daily memory, reminder, and post-compaction behavior, so a fix should preserve those cases or make the exact-time injection conditional with tests.
