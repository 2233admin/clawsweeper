## Summary

Keep open. Current main supports static multi-agent routing, TUI/web agent listing or selection surfaces, and thread-bound subagent/session controls, but it does not implement the requested message-channel `/agent <name>` command that temporarily switches an ordinary chat session to another configured agent without changing global bindings or restarting the Gateway.

## Evidence

- **Issue request:** #43976 asks for `/agents` to list all available agents and `/agent <name>` to temporarily switch the current single-chat channel session to a different agent, with isolated memory/workspace and no global binding change or Gateway restart.
- **Current routing model is binding-driven:** The multi-agent docs describe inbound routing through `bindings`, with deterministic most-specific matching and fallback to the default agent. This covers configured routing, not a per-session runtime `/agent` switch command.
  - file: [docs/concepts/multi-agent.md:227](https://github.com/openclaw/openclaw/blob/1531123d3547f9ff8ee8094590fc773305f018f2/docs/concepts/multi-agent.md#L227)
