## Summary

Current main still does not implement the requested Gateway/channel `/agents` plus `/agent <name>` workflow for temporarily switching a single conversation to another configured agent without restart or global binding edits. Existing behavior covers static binding-based routing, thread/subagent focus, TUI-local agent switching, and Telegram per-topic config routing, but those are narrower than the issue request.

## Best Possible Solution

Keep this open for a maintainer-designed Gateway/channel command feature. The likely implementation should add authorized `/agents` and `/agent <id|main>` handling that lists configured agents, stores a temporary per-conversation route override, applies it before normal `bindings` routing, and defines clear reset/expiry semantics while preserving per-agent workspace, auth, memory, and session-store isolation. If maintainers prefer a plugin-first route, first add an explicit generic route-override API instead of asking channel plugins to mutate global bindings.

## Risks / Open Questions

- Closing this as implemented would conflate TUI-local `/agent` switching and `/focus` thread/session binding with the requested cross-channel temporary agent routing command.
- A correct implementation needs explicit authorization, per-conversation scope, expiry/reset semantics, and isolation of workspace, auth, memory, and session stores so one sender cannot unexpectedly move another user or group onto a different agent/tool policy.
