## Summary

Keep open. Current `main` does not contain the proposed `pruneOrphanedTranscripts` utility or its new unit coverage. Existing maintenance only prunes `sessions.json` entries, cleans archived transcripts, rotates `sessions.json`, and performs orphan cleanup as part of optional disk-budget enforcement, so this PR still adds distinct behavior.

## Evidence

- **No orphan-prune utility on main:** `src/config/sessions/store-maintenance.ts` on `main` ends its maintenance helpers with `capEntryCount()` and `rotateSessionFile()`; there is no `pruneOrphanedTranscripts` export or equivalent general orphan-transcript sweep.
  - file: [src/config/sessions/store-maintenance.ts:264](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/config/sessions/store-maintenance.ts#L264)

## Risks / Open Questions

- Without a non-budget orphan sweep, large orphan transcript sets can still persist on installs that do not configure or exceed `session.maintenance.maxDiskBytes`.
- Closing this PR as already implemented would be inaccurate because `main` only has partial, budget-triggered orphan cleanup, not the utility proposed here.
