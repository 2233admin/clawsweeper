## Summary

Keep open. Current main still does not support SecretRef/SecretInput values in top-level `openclaw.json` `env.vars`: the schema and public config type are string-only, runtime env collection applies plain strings directly, and the SecretRef runtime collectors/credential surface registry do not include `env.vars`. The related MCP env SecretRef work only covers `plugins.entries.acpx.config.mcpServers.*.env.*`, and the AWS Secrets Manager provider PR in the provided context was closed unmerged, so neither supersedes this request.

## Best Possible Solution

Keep this issue open and implement it as a core config/secrets change: widen `env.vars` and top-level env sugar to accept `SecretInput`, collect `env.vars.*` SecretRef assignments in the runtime secrets path, resolve before applying config env vars, preserve existing env safety filters/no-overwrite behavior, and update schema/docs plus focused tests for env/file/exec refs and unresolved-ref failures.

## Risks / Open Questions

- Implementing this affects global env injection, so resolution ordering must avoid persisting resolved plaintext into config or service metadata.
- Existing guardrails should be preserved: no overriding already-set env vars, dangerous env var filtering, non-portable key dropping, and unresolved `${VAR}` placeholder avoidance.
