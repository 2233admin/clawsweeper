## Summary

Keep this PR open. Current main still has the zero-output dispatch path the PR targets: an undefined resolver result becomes an empty replies array, queued counts are read, completion is recorded, and the function returns without queuing any visible fallback. The PR is a plausible focused fix, but it still needs maintainer review against sendPolicy suppression and canonical chat-type normalization before merge.

## Best Possible Solution

Keep this PR open and review/update it against current main. The best fix is a narrow dispatch-level fallback after final queued counts are known, only when user delivery is allowed, the turn is direct-style under canonical chat-type policy, there is no routed-origin delivery, no queued tool/block/final output, no block-stream/TTS output, and no intentional handled/no-text or plugin/suppressed-delivery branch. Add focused regression tests for direct empty-output fallback plus group/channel, sendPolicy-deny, routed-origin, block-stream/TTS, and handled-without-text no-op cases.
