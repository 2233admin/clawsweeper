## Summary

Current `main` does not expose Discord interaction response controls to `api.registerCommand(...)` handlers. Plugin commands still receive the generic `PluginCommandContext`, and Discord slash execution wraps the handler result in a single host-managed reply/follow-up path with one shared ephemeral setting, so the requested `defer`/`reply`/`followUp`/`editReply` control is not implemented.

## Evidence

- **Plugin command context is still generic:** `PluginCommandContext` only includes generic routing/session/binding fields plus `commandBody` and does not include any Discord responder surface such as `ctx.discord.respond` or interaction methods.
  - file: [src/plugins/types.ts:1772](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/plugins/types.ts#L1772)
  - command: `nl -ba src/plugins/types.ts | sed -n '1768,1848p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Command execution builds only that generic context:** `executePluginCommand(...)` constructs the handler context from generic fields (`senderId`, `channel`, `sessionKey`, `from`, `to`, thread metadata, binding helpers) and passes it straight to `command.handler(ctx)` with no Discord-specific responder injection.
  - file: [src/plugins/commands.ts:171](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/plugins/commands.ts#L171)
  - command: `nl -ba src/plugins/commands.ts | sed -n '170,255p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Discord slash plugin commands are post-processed by host reply delivery:** The Discord native command path calls `executePluginCommand(...)`, gets back a `ReplyPayload`, and then sends it through `deliverDiscordInteractionReply(...)`. The plugin handler does not receive the live interaction object or control over when/how replies are sent.
  - file: [extensions/discord/src/monitor/native-command.ts:1034](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/discord/src/monitor/native-command.ts#L1034)
  - command: `nl -ba extensions/discord/src/monitor/native-command.ts | sed -n '1032,1088p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Ephemeral behavior is still one shared host-level switch:** Discord delivery applies a single `responseEphemeral` flag to each outbound interaction message. That supports a global slash-command default, but not the mixed private/public follow-up flow requested in the issue.
  - file: [extensions/discord/src/monitor/native-command.ts:1286](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/discord/src/monitor/native-command.ts#L1286)
  - command: `nl -ba extensions/discord/src/monitor/native-command.ts | sed -n '1286,1338p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Discord already has responder APIs for interactive handlers, not command handlers:** `DiscordInteractiveHandlerContext` exposes `respond.acknowledge`, `respond.reply`, `respond.followUp`, and message-edit helpers, which highlights that this capability exists for interactive handlers but has not been extended to `registerCommand` handlers.
  - file: [extensions/discord/src/interactive-dispatch.ts:11](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/extensions/discord/src/interactive-dispatch.ts#L11)
