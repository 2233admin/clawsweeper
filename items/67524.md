## Summary

Keep open. Current `main` still rejects mixed `trusted-proxy` + shared-token config at startup and still fails closed for loopback `trusted-proxy` requests even when a token is present, so the deployment gap for internal cron/exec callers remains unresolved. The issue body's claim about a reachable runtime token fallback no longer matches current code, but the follow-up discussion reframed this as an open design/product gap rather than a closed bug.
