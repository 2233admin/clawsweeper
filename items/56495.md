## Summary

Keep open. Current `main` at `6b618f063516aa9ba85903aa4565c5a204c055b2` still has the bug shape described by the PR: WhatsApp `append` upserts go through `normalizeInboundMessage()` and `checkInboundAccessControl()` before the append-history age filter runs, and pairing suppression is still timestamp-only rather than append-aware.

## Evidence

- **Access control still runs before append replay filtering:** `handleMessagesUpsert` calls `normalizeInboundMessage(msg)` at line 648, then `maybeMarkInboundAsRead`, and only afterwards checks `upsert.type === "append"` and skips stale history at lines 656-663. That means append upserts still reach access control before replay filtering.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:648](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/whatsapp/src/inbound/monitor.ts#L648)

The function signature has no `isAppend` parameter here, so access control cannot distinguish replayed append messages from live messages.
