## Summary

Keep open. Current main does not show compaction itself creating a `ttsAuto: "always"` override, but the dashboard/chat path can still turn ordinary text into audio when effective TTS state comes from session state, local prefs, or config. The reporter's follow-up says this happened twice and a full reset fixes it, so a persisted-state/dashboard-TTS regression remains plausible and is not safely closeable from source inspection alone.

## Best Possible Solution

Keep the issue open. Triage by capturing the affected session entry (`ttsAuto`), local TTS prefs, and `messages.tts` config before and after compaction. If dashboard text conversations should never auto-synthesize unless explicitly requested, add a regression around dashboard `chat.send` plus compaction/block streaming and either bypass auto-TTS for webchat text sends or require `messages.tts.auto: "tagged"`, `/tts chat on`, or an explicit TTS directive/tool call before audio is generated.
