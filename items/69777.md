## Summary

Keep open. Current `main` does preserve `ttsAuto` across compaction, so the narrow theory from the triage comment is plausible, but there is not strong enough evidence to close: OpenClaw can still emit audio-only final replies whenever auto-TTS is enabled from session state, local prefs, or config, and the reporter’s follow-up that a full OpenClaw reset temporarily fixes it is consistent with persisted TTS state rather than a disproven bug.

## Evidence

- **Audio-only fallback still exists on current main:** When block streaming finishes without a final text payload, dispatch generates a TTS-only reply (`mediaUrl` + `audioAsVoice`) from accumulated text. That means users can legitimately see audio-only assistant replies even without an explicit per-message audio request, as long as TTS is effectively enabled.
