## Summary

Keep #53920 open. Current main still edits the tracked auth-monitor systemd template in place before copying it into the user systemd directory, and the template still hardcodes `/home/admin/openclaw/scripts/auth-monitor.sh`. The PR targets the right script path, but the latest review discussion leaves a real idempotency concern for users whose template was already mutated by the old setup flow.

## Best Possible Solution

Keep the PR open and ask for a narrow revision: copy the service template first, render only the installed user unit, set `ExecStart` to the current checkout's `scripts/auth-monitor.sh`, and make notification placeholder handling idempotent for both pristine commented placeholders and already-uncommented values from prior runs. If the PR stalls, implement that same fix directly in `scripts/setup-auth-system.sh` with a focused script-level regression test or explicit manual verification that setup does not dirty the repository template.
