## Summary

Keep open. Current `main` still rejects Feishu group inbound traffic before `requireMention` is evaluated unless the group passes `groupPolicy`/`groupAllowFrom`, so the reporter’s posted config (`groups.<chat>.requireMention: false` by itself) would still not create a group session on current code. That means this is not safely closable as fixed; at minimum there is still an unresolved v3.28-to-v4 compatibility or migration/docs gap.

## Evidence

- **Feishu group access still defaults to allowlist:** The Feishu config schema still defaults `channels.feishu.groupPolicy` to `"allowlist"`. `groups` is a separate per-group config map, while admission is controlled by `groupAllowFrom`/`groupPolicy`, so `groups.<chat>.requireMention=false` alone does not allow the group.
  - file: [extensions/feishu/src/config-schema.ts:163](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/feishu/src/config-schema.ts#L163)

- **Inbound handler still drops disallowed groups before mention logic:** In `handleFeishuMessage`, group messages are checked against `groupAllowFrom` first. If the chat is not allowed, the handler returns early with `group ... not in groupAllowFrom`; only after that does it call `resolveFeishuReplyPolicy` and apply `requireMention` gating.
  - file: [extensions/feishu/src/bot.ts:421](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/feishu/src/bot.ts#L421)
