## Summary

Current `main` already provides persistent Control UI authentication through browser-local device pairing and stored device tokens, so the report's premise that Control UI is token-only and requires re-entering a long token every new browser session is obsolete.

## Risks / Open Questions

- Current `main` solves this with paired-device tokens rather than a native WebAuthn/passkey biometric ceremony. Closing assumes the actionable request was persistent browser auth, not specifically Face ID/Touch ID UX.

## Close Comment

Closing based on a Codex review of current `main`. OpenClaw no longer relies only on pasted gateway tokens for Control UI access: the current docs describe one-time device pairing and say the browser/device is remembered after approval (`docs/web/control-ui.md:36`), the gateway protocol says successful pairing returns `hello-ok.auth.deviceToken` and that clients should persist it for future connects (`docs/gateway/protocol.md:517`), and the browser implementation stores that paired device token in local storage and reuses it on later connects (`ui/src/ui/device-auth.ts:10`, `ui/src/ui/gateway.ts:451`, covered by `ui/src/ui/gateway.node.test.ts:294`). So the reported problem of having to re-type the long token every new browser session is already addressed on `main`, even though the shipped solution is paired-device auth rather than a WebAuthn/FIDO2 passkey flow.
