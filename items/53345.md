## Summary

Keep this issue open. Current main already ships the Korean Control UI locale, registration, picker/docs, and tests, including in v2026.4.24, but the issue also asks for default AI-agent primary-language persistence and automatic Korean search localization. Those agent/search behaviors are not implemented on current main, and the paired same-author PR is closed and unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open, but narrow it to the remaining agent-side behavior. Preserve the shipped Korean Control UI locale, then design a tested primary-language response policy in the agent prompt or conversation layer with explicit precedence for user overrides and preservation of code/log/path/quote/tool-output text. Add Korean search localization only through provider-aware web-search parameters or configuration, not as an assumed global field.

## Risks / Open Questions

- Search localization cannot be implemented as one universal `search_lang: 'ko'` field because provider support and accepted parameters differ.
