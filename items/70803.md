## Summary

Keep open. Current `main` still has the normalization/runtime mismatch the PR describes: WhatsApp target normalization only accepts `@lid`, while the JID-to-E.164 runtime path already accepts `@hosted.lid`. The shipped release `v2026.4.23` still contains that mismatch, and current tests do not cover hosted LID normalization in `resolve-target.test.ts`.

## Evidence

- **Current normalizer only matches `@lid`:** On current `main`, `WHATSAPP_LID_RE` is `^(\d+)@lid$`, and both `isWhatsAppUserTarget` and `extractUserJidPhone` depend on it. That means `normalizeWhatsAppTarget("123456789@hosted.lid")` would fall through and return `null`.
  - file: [extensions/whatsapp/src/normalize-target.ts:6](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/whatsapp/src/normalize-target.ts#L6)

  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
