## Summary

Keep open. Current main still collapses the reported model-fallback cases to `unknown`: the failover reason unions do not include `empty_response`, `no_error_details`, or `unclassified`; `classifyProviderRuntimeFailureKind` returns `unknown` for the empty-signal, exact no-details, and catchall paths; fallback attempts persist `described.reason ?? "unknown"`; and existing tests/changelog explicitly lock the prior no-details fix as `unknown`, not the requested more specific labels.

## Best Possible Solution

Keep the issue open. The best fix is a scoped model-fallback diagnostic change: add explicit `empty_response`, `no_error_details`, and `unclassified` handling to the relevant failover/runtime-failure contracts or add a separate runtime-failure-kind field where changing `FailoverReason` is too broad; make all three inherit the current transient cooldown behavior; preserve sanitized raw previews for unclassified non-empty errors; and add regression coverage in the classifier, failover policy, model-fallback attempt logging, and fallback notice formatting tests.
