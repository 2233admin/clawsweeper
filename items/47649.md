## Summary

Current `main` already hardens Discord inbound attachment handling so failed large-file fetches no longer take down the channel listener/queue. The shipped `v2026.4.22` tree also contains this behavior.

## Evidence

## Risks / Open Questions

- No live Discord/Nitro reproduction was run; conclusion is based on current code and tests only.
- This fixes the listener-stall/crash class by degrading gracefully; it does not prove Discord CDN access will always succeed for every large Nitro attachment.

## Close Comment

Closing from a Codex review: current `main` already hardens this path well enough that large Discord attachment fetch failures should no longer take down the channel listener. `extensions/discord/src/monitor/message-utils.ts:383` catches attachment download errors and falls back to placeholder media metadata instead of bubbling the failure, `extensions/discord/src/monitor/message-handler.process.ts:177` applies dedicated attachment idle/total timeouts, and `extensions/discord/src/monitor/message-handler.queue.test.ts:723` verifies the per-channel queue recovers after a failed run instead of staying stuck. The same implementation is present in shipped release `v2026.4.22` (`a94763ee224e858e29dd177cd6f7f4daa438c2f1`). If you still see a listener outage on a newer build, please open a new issue with the exact OpenClaw version, attachment type/size, and gateway logs so it can be reproed against the current codepath.
