## Summary

Keep open. Current main has real tool-call plumbing and limited plan-only retry hardening, but it does not implement the issue’s requested broad enforcement path for local LiteLLM/vLLM models: no deterministic web_fetch/web_search routing, no retry with forced tool_choice when a required tool is merely described in text, and no general simulated-tool-call rejection/correction surface. The related comments are useful evidence of adjacent security/provenance and success-claim gaps, but they do not supersede this specific item.

## Evidence

- **Structured tool-call conversion only:** OpenAI Responses output is converted into an internal tool call only when the provider returns a structured `function_call` output item; assistant `message` output is treated as text. The final stop reason becomes `toolUse` only if the built content contains a `toolCall` block, so plain text such as a JSON-looking `{ "tool": "web_fetch" }` response is not a real tool invocation path.
