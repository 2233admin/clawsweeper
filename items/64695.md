## Summary

Keep this issue open. Current main still defaults the task registry and task-flow registry to SQLite-only stores that call `require("node:sqlite")`; there is no runtime availability probe, JSON fallback, or replacement backend. The scheduled sweep path now catches a failed sweep promise, which may reduce the original unhandled-rejection crash symptom, but the reported Homebrew runtime still cannot restore or persist task state and the stale-task maintenance update path still reaches the same SQLite requirement.

## Best Possible Solution

Keep this issue open. Implement runtime-safe storage selection for both the task registry and task-flow registry: detect unavailable `node:sqlite` before default store use, choose a maintainer-approved durable fallback or dependency, preserve existing SQLite behavior where available, and add regression tests that simulate missing `node:sqlite` across restore, update/upsert, and stale-task maintenance. The closed #64722 proposal is useful prior art but needs review or a superseding implementation before this can close.

## Risks / Open Questions

- A fallback store needs explicit durability, locking, and multi-process semantics so task state remains safe across restarts and overlapping gateway processes.
- Replacing `node:sqlite` with `better-sqlite3` would add native dependency and installer complexity, so that path needs maintainer review.
- Because scheduled sweeps can now swallow failures, a fix should include clear diagnostics plus regression coverage for restore, maintenance, and write/update paths when `node:sqlite` is unavailable.
