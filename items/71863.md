## Summary

Keep PR #71863 open. Current main still has the Signal daemon shutdown race described by #22676, while the PR directly changes the implicated lifecycle to async shutdown, await cleanup, add SIGKILL fallback, and add regression coverage.

## Best Possible Solution

Keep the PR open and review it as the focused fix for #22676. The right product/code path is to land the Signal plugin lifecycle change: async/idempotent daemon stop, SIGTERM-to-SIGKILL fallback, awaited monitor cleanup, logged fire-and-forget abort trigger, and the new regression tests.

## Evidence

- **Current main Signal daemon stop is still fire-and-forget:** `SignalDaemonHandle.stop` is typed as `() => void`, and `spawnSignalDaemon().stop()` only sends `SIGTERM` when the child is not killed/exited; it does not wait for `exit`/`close` and has no `SIGKILL` fallback.
  - file: [extensions/signal/src/daemon.ts:16](https://github.com/openclaw/openclaw/blob/6d60b035b4e7c1739f038808e041567448f58060/extensions/signal/src/daemon.ts#L16)

- **Current monitor cleanup does not await daemon exit:** `createSignalDaemonLifecycle.stop()` calls `daemonHandle?.stop()` synchronously, `onAbort` invokes it without awaiting, and the `finally` block also calls it without awaiting. That allows `monitorSignalProvider` to settle before the signal-cli child has actually exited.
  - file: [extensions/signal/src/monitor.ts:122](https://github.com/openclaw/openclaw/blob/6d60b035b4e7c1739f038808e041567448f58060/extensions/signal/src/monitor.ts#L122)
