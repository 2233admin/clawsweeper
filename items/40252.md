## Summary

Keep #40252 open. Current main still synthesizes `<media:image>` / `<media:${kind}>` for blank iMessage media sends and passes that value as the outbound RPC `text` alongside the attachment file. The related fix attempts #40341 and #46343 are closed unmerged in the provided GitHub context, and current tests still encode placeholder-based media-only cache behavior.

## Best Possible Solution

Keep the issue open and fix `extensions/imessage/src/send.ts` so media-only iMessage sends pass an empty `text` field plus `file`, while preserving real captions, reply IDs, directive cleanup for actual text, local media root handling, and the empty-payload guard. Add focused regression coverage for `sendMessageIMessage()` asserting media-only image/audio/video sends call the RPC with `text: ""` and `file`, then update delivery/echo-cache tests so dedupe no longer relies on user-visible placeholder text.
