## Summary

Keep open. Current `main` already provides the HTTP half of the request via `POST /tools/invoke`, and the reporter’s follow-up about `gateway.tools.allow` matches current docs/tests. But `sessions_spawn` is still intentionally denied over HTTP by default, and there is still no core CLI surface like `openclaw sessions spawn`, `openclaw tools invoke`, or equivalent documented command on `main`.

## Evidence

- **Direct HTTP tool invoke exists on main:** Gateway docs explicitly document `POST /tools/invoke` as a direct single-tool HTTP surface, with request body fields including `tool`, `args`, and optional `sessionKey`.
  - file: [docs/gateway/tools-invoke-http-api.md:11](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/docs/gateway/tools-invoke-http-api.md#L11)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **`sessions_spawn` remains HTTP-denied by default:** The same docs list `sessions_spawn` on the default hard deny list for `/tools/invoke`, and explain that `gateway.tools.allow` removes entries from that default deny list.
  - file: [docs/gateway/tools-invoke-http-api.md:104](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/docs/gateway/tools-invoke-http-api.md#L104)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Default deny list is enforced in code:** `DEFAULT_GATEWAY_HTTP_TOOL_DENY` includes `sessions_spawn` and `sessions_send`, matching the docs and reporter follow-up.
  - file: [src/security/dangerous-tools.ts:9](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/security/dangerous-tools.ts#L9)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Tests prove current HTTP behavior:** Gateway tests verify that `sessions_spawn` returns 404 over HTTP even when the agent policy allows it, and only works when `gateway.tools.allow` explicitly includes `sessions_spawn`.
  - file: [src/gateway/tools-invoke-http.test.ts:583](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/gateway/tools-invoke-http.test.ts#L583)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **No matching CLI command exists:** The CLI command tree shows `sessions` with only `cleanup`, and the registered `sessions` command implementation only supports listing plus `cleanup`. The low-level `gateway call` command targets Gateway RPC methods, not direct tool invocation.
  - file: [docs/cli/index.md:177](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/docs/cli/index.md#L177)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Registered `sessions` CLI has no spawn subcommand:** `registerStatusHealthSessionsCommands` defines `openclaw sessions` for listing and `openclaw sessions cleanup`; there is no `spawn` subcommand here.
  - file: [src/cli/program/register.status-health-sessions.ts:124](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/cli/program/register.status-health-sessions.ts#L124)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Existing generic gateway CLI is RPC-only:** `openclaw gateway call` is documented and registered as a low-level Gateway RPC helper taking a `<method>` and JSON params; it is not a documented `tools invoke` CLI.
  - file: [src/cli/gateway-cli/register.ts:417](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/cli/gateway-cli/register.ts#L417)
