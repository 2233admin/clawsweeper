## Summary

Keep open: current `main` still turns assistant auth failures into user-visible `HTTP 401: Invalid token` replies, and the bundled `zai` provider still has no runtime-auth refresh hook for the embedded runner to retry with a refreshed credential.

## Evidence

- **Raw HTTP errors are still surfaced verbatim:** `formatRawAssistantErrorForUi()` still returns plain `HTTP <code>: <text>` for non-JSON HTTP errors, so a provider error like `HTTP 401: Invalid token` remains user-facing instead of being normalized away.
  - file: [src/shared/assistant-error-format.ts:198](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/shared/assistant-error-format.ts#L198)
  - command: `nl -ba src/shared/assistant-error-format.ts | sed -n '178,214p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Embedded-run payload builder still emits assistant errors as reply text:** When the last assistant turn ends in error, `buildReplyPayloadsFromAttempt()` formats that assistant error and pushes it into `replyItems` with `isError: true`, which is the reply path Feishu uses.
  - file: [src/agents/pi-embedded-runner/run/payloads.ts:163](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/run/payloads.ts#L163)
  - command: `nl -ba src/agents/pi-embedded-runner/run/payloads.ts | sed -n '160,194p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Feishu inbound path still dispatches embedded-run replies directly:** The Feishu bot handler still calls `core.channel.reply.dispatchReplyFromConfig()` through `withReplyDispatcher()` for inbound messages, so the embedded runner's error payloads are what Feishu users receive.
  - file: [extensions/feishu/src/bot.ts:1219](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/feishu/src/bot.ts#L1219)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '1219,1244p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Retry-after-refresh only exists when runtime auth state exists:** `maybeRefreshRuntimeAuthForAuthError()` only retries after auth errors if `params.getRuntimeAuthState()` is already present. Without runtime-auth state, the 401 falls through to normal failover/surface-error handling.
  - file: [src/agents/pi-embedded-runner/run/auth-controller.ts:543](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/run/auth-controller.ts#L543)
  - command: `nl -ba src/agents/pi-embedded-runner/run/auth-controller.ts | sed -n '543,563p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Bundled Z.AI provider still lacks a runtime-auth refresh hook:** The current `zai` provider registration defines API-key auth methods, replay hooks, and usage hooks, but does not register `prepareRuntimeAuth`, so the embedded runner has no provider-specific way to mint or refresh a short-lived runtime token on 401.
  - file: [extensions/zai/index.ts:233](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/zai/index.ts#L233)
  - command: `nl -ba extensions/zai/index.ts | sed -n '233,314p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current Z.AI docs still describe static API-key auth:** The public provider docs still describe bundled `zai` auth as `ZAI_API_KEY` bearer auth, not a refreshable token flow. That matches the codepath above and shows no shipped fix for automatic token refresh on this provider surface.
  - file: [docs/providers/zai.md:9](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/providers/zai.md#L9)
  - command: `nl -ba docs/providers/zai.md | sed -n '9,16p'`
