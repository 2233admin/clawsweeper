## Summary

Keep open. Current `main` still intentionally serializes agent runs per session and queues or steers follow-ups instead of allowing multiple concurrent turns within the same session.

## Evidence

- **Design doc says per-session runs are serialized:** The queue design explicitly says inbound auto-reply runs are serialized, that `runEmbeddedPiAgent` enqueues by `session:<key>` to guarantee only one active run per session, and that per-session lanes guarantee only one agent run touches a given session at a time.
  - file: [docs/concepts/queue.md:10](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/docs/concepts/queue.md#L10)
  - command: `nl -ba docs/concepts/queue.md | sed -n '1,120p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Agent loop doc repeats single serialized run per session:** The agent loop documentation states that OpenClaw uses a single serialized run per session and that `runEmbeddedPiAgent` serializes runs via per-session and global queues.
  - file: [docs/concepts/agent-loop.md:15](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/docs/concepts/agent-loop.md#L15)
  - command: `nl -ba docs/concepts/agent-loop.md | sed -n '10,70p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Runtime code enforces a session lane before execution:** `runEmbeddedPiAgent` resolves a session lane, wraps execution in `enqueueSession(...)`, and then enqueues into a global lane. That is serialized execution, not parallel same-session turns.
  - file: [src/agents/pi-embedded-runner/run.ts:249](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/pi-embedded-runner/run.ts#L249)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '230,290p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Busy-session handling is queue/wait/interrupt, not concurrent turn start:** When a run is already active, queue policy returns `enqueue-followup`, and the queue-state handler waits for the previous active run to end before continuing. The only special case is `interrupt`, which aborts and waits; it still does not run two turns concurrently.
  - file: [src/auto-reply/reply/queue-policy.ts:11](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/auto-reply/reply/queue-policy.ts#L11)

- **UI/docs still describe queued follow-ups during an active run:** Current control UI docs say that while a run is active, normal follow-ups queue, and users can only use Steer to inject the follow-up into the running turn.
  - file: [docs/web/control-ui.md:174](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/docs/web/control-ui.md#L174)
