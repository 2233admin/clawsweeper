## Summary

Current main already persists `sessions_send` provenance on the transcript row and exposes that row in real time through the supported `session.message` subscription path; the remaining absence is only on the separate `chat` UI stream.

## Evidence

- **sessions_send stamps source session provenance:** The tool sends inter-session messages with `inputProvenance` including `sourceSessionKey`, `sourceChannel`, and `sourceTool: "sessions_send"`.
  - file: [src/agents/tools/sessions-send-tool.ts:283](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/tools/sessions-send-tool.ts#L283)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **provenance is a persisted transcript field:** `InputProvenance` includes `sourceSessionKey`, and user messages get a `provenance` object attached without changing the role.
  - file: [src/sessions/input-provenance.ts:12](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/sessions/input-provenance.ts#L12)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **docs say sessions_send persists inter-session provenance:** The transcript hygiene reference states that `sessions_send` writes `message.provenance.kind = "inter_session"` at transcript append time.
  - file: [docs/reference/transcript-hygiene.md:82](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/docs/reference/transcript-hygiene.md#L82)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **real-time transcript event forwards the full message:** `createTranscriptUpdateBroadcastHandler` emits `session.message` with the full `message` object from the transcript update, only adding OpenClaw meta like id/seq.
  - file: [src/gateway/server-session-events.ts:88](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/gateway/server-session-events.ts#L88)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **protocol documents the supported real-time path:** Gateway docs define `sessions.messages.subscribe` / `session.message` as the transcript/message subscription surface, and describe `chat` separately as a UI chat-update family.
  - file: [docs/gateway/protocol.md:341](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/docs/gateway/protocol.md#L341)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)

  - file: [ui/src/ui/app-gateway.ts:539](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/ui/src/ui/app-gateway.ts#L539)
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)

## Risks / Open Questions

- The `chat` event stream itself still does not embed provenance on its assistant/UI payloads, so third-party clients listening only to `chat` would need to switch to `session.message` or reload history on transcript updates.
- I could not tie this behavior to a tagged release from the available checkout, so `fixedRelease` is left null.

## Close Comment

Codex review: current `main` already covers the underlying request through the supported transcript subscription path. `sessions_send` stamps inter-session provenance (`src/agents/tools/sessions-send-tool.ts:283`), the transcript contract persists that provenance (`src/sessions/input-provenance.ts:12`, `docs/reference/transcript-hygiene.md:82`), and Gateway now exposes real-time transcript rows via `session.message` / `sessions.messages.subscribe` (`src/gateway/server-session-events.ts:88`, `docs/gateway/protocol.md:341`). The maintained UI already uses `session.message` to refresh live history for the active session (`ui/src/ui/app-gateway.ts:539`). Because the provenance-bearing real-time path is already implemented on `main`, I’m closing this. If you still need provenance duplicated on the separate `chat` UI event stream specifically, please open a narrower follow-up for that stream contract.
