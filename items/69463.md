## Summary

Keep open. Current main has broad observability support through diagnostic events, OTEL export, and `traceparent` propagation, but it does not implement this PR's configurable `diagnostics.sessionTracing` headers. I also cannot close it as ClawHub with high confidence because current provider/plugin seams support custom/provider-owned headers but do not expose the active `runId` plus `sessionKey` needed to reproduce the PR's behavior for existing providers.

## Best Possible Solution

Keep the PR open for maintainer review, but steer it toward a narrow, privacy-safe design: either add a vetted core diagnostics option for per-model-call correlation headers or extend the provider/plugin stream context with the minimal run/session correlation values needed for an external plugin to implement this. The implementation should be off by default, avoid leaking raw session keys by default, preserve caller headers, document the security tradeoff, and cover the wrapper/config behavior in focused tests.

## Risks / Open Questions

- The PR's proposed fallback of sending `sessionKey` can expose channel/user routing identifiers to third-party observability providers; a privacy-safe design should avoid that by default or make the tradeoff explicit.
- A generic all-provider header injector may conflict with provider/proxy expectations; any implementation should preserve caller header precedence and have focused tests around disabled-by-default behavior.
- The best path may be a smaller core or plugin-SDK seam rather than the exact PR shape, especially if maintainers prefer provider-owned observability plugins over global diagnostics config.
