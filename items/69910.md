## Summary

Keep #69910 open. Current main has the replay/outbound Ollama argument normalization, but the PR’s central inbound native response path is still missing: `buildAssistantMessage` copies `response.message.tool_calls[].function.arguments` directly into the assistant tool call. A provider-returned JSON string can still reach tool execution unchanged, and current tests cover object args plus replay normalization, not fresh native response string args. The provided PR context also shows the bot review concerns were addressed in the PR head with unsafe-integer-preserving parsing and malformed-input fallback.

## Best Possible Solution

Keep #69910 open and land or rework it as the focused fix: make native Ollama response `tool_calls[].function.arguments` accept string/unknown input, normalize it in `buildAssistantMessage` with the existing unsafe-integer-preserving fallback behavior, and keep regression tests for valid stringified args, malformed strings, object pass-through, and unsafe integer preservation.
