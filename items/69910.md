## Summary

Keep PR #69910 open. Current main already normalizes replayed Ollama tool-call arguments in `extractToolCalls()` and OpenAI-compatible payload messages, but `buildAssistantMessage()` still passes native Ollama response `tool_calls[].function.arguments` through unchanged. The current tests also do not cover string-valued Ollama tool-call arguments, while the PR adds the missing normalization and regression coverage, including the bot-reviewed unsafe-integer and malformed-JSON cases.

## Best Possible Solution

Keep the PR open or land an equivalent narrow patch on current main: normalize `toolCall.function.arguments` with the existing unsafe-integer-preserving `ensureArgsObject()` in `buildAssistantMessage()`, widen the local Ollama tool-call response type enough to represent provider reality, and add focused tests for valid JSON strings, object pass-through, unsafe integer preservation, and malformed-string fallback.
