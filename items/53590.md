## Summary

Keep this issue open. Current main has a startup warmup warning only when model resolution fully misses, and runtime failover can now classify invalid-model provider errors, but the reported OpenRouter configured-slug path is still not diagnosed at gateway startup or by doctor because OpenRouter dynamic resolution still accepts uncatalogued IDs as text-only models.

## Best Possible Solution

Keep this open and add non-fatal configured-model diagnostics for agents.defaults.model.primary and configured fallbacks. Gateway startup and openclaw doctor should warn when a provider catalog was successfully fetched and proves a configured ref is absent, while preserving OpenRouter dynamic pass-through when the catalog is unavailable or intentionally incomplete. Known retired or renamed slugs should be noted in changelog/docs when OpenClaw changes bundled defaults or compatibility behavior.

## Risks / Open Questions

- A fix that hard-fails every uncatalogued OpenRouter ID would regress OpenRouter's dynamic pass-through for newly published upstream models.
- Startup and doctor diagnostics should distinguish catalog-unavailable from catalog-fetched-and-model-absent so network failures do not become false invalid-model warnings.
