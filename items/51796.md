## Summary

Current OpenClaw already enforces fresh exec approval for every command when `ask` is `"always"`, rejects durable `allow-always` in that mode, and includes a regression test that specifically requires a second approval after an `allow-once` decision. The shipped `v2026.4.22` tag contains that behavior, so this issue appears fixed on current main and in the latest release.

## Evidence

- **Core policy: `ask="always"` always prompts:** `requiresExecApproval()` returns `true` immediately when `ask === "always"`, so prior durable approvals cannot suppress the next prompt.
  - file: [src/infra/exec-approvals.ts:779](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/infra/exec-approvals.ts#L779)

## Risks / Open Questions

- No live Control UI reproduction was run in this review because `pnpm` is unavailable in the checkout environment; this decision is based on static inspection of current main and the shipped `v2026.4.22` tag.
- The Control UI modal still renders an `Always allow` button unconditionally (`ui/src/ui/views/exec-approval.ts:85`), but the gateway now rejects that decision for `ask="always"`; that is a UX inconsistency, not the reported silent-approval bypass.

## Close Comment

Closing as implemented on current OpenClaw. Codex review of current `main` and the shipped `v2026.4.22` tag found that `ask: "always"` now hard-requires a fresh exec approval on every command (`src/infra/exec-approvals.ts:779`), durable `allow-always` is explicitly disabled for that mode (`src/infra/exec-approvals.ts:1020`, `docs/tools/exec-approvals.md:218`), and the gateway rejects `allow-always` resolutions for those requests (`src/gateway/server-methods/exec-approval.ts:360`). There is also a regression test that specifically asserts a second elevated command after `allow-once` produces a new approval request with a different id (`src/agents/bash-tools.exec.approval-id.test.ts:1134`). If you still hit this on `v2026.4.22` or newer, please open a new issue with exact reproduction steps, whether the command was `host=gateway` or `host=node`, and gateway logs from that run.
