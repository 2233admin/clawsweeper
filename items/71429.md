## Summary

Keep #71429 open. Current main has related generic hardening, but the specific Telegram normal auto-reply path is still not durably journaled before Telegram Bot API delivery, and sendChatAction failures are still treated as typing/feedback failures rather than a hard visible turn failure. This matches the existing maintainer discussion that explicitly left the item open.

## Best Possible Solution

Keep #71429 open. Add a focused Telegram regression that simulates a live Telegram turn, sendChatAction network failures, channel hot reload, and final Bot API delivery failure. Then either route normal Telegram auto-reply final payloads through a durable queue before Bot API send, or make delivery failure a visible hard turn failure with retry guidance in Telegram/UI. The existing outbound/session-delivery queue primitives are likely reusable, but same-surface Telegram deliverReplies needs explicit wiring and coverage.
