## Summary

Keep open. Current main at a35d259719cc7aecd1854fdc830ad6390b15f02a has related reload deferral and durable outbound primitives, but normal same-surface Telegram auto-replies still go through the in-process dispatcher to Telegram Bot API delivery without a write-ahead queue, and delivery/typing failures are not converted into a durable replay or hard visible turn failure.

## Best Possible Solution

Keep #71429 open. Add a focused Telegram regression that simulates a live turn, channel hot reload, sendChatAction failure, and final Bot API delivery failure; then either route normal Telegram final replies through an existing durable outbound/session-delivery queue before Bot API send, or mark the turn failed with visible Telegram/UI retry guidance.
