## Summary

Keep open. Current `main` already caches the memory tool runtime and the builtin `MemoryIndexManager` within a process, but it still defers builtin/local embedding-provider initialization until the first `memory_search`. Gateway startup warmup is limited to the `qmd` backend, so the reported cold-start class is not implemented away on `main` for the issue's stated builtin/local-embedding environment.

## Evidence

- **Tool runtime import is already cached:** `memory_search` does not re-import its runtime on every call within a process: `loadMemoryToolRuntime()` memoizes the dynamic import in `memoryToolRuntimePromise`, and manager lookup flows through that cached runtime (`extensions/memory-core/src/tools.shared.ts:17`, `extensions/memory-core/src/tools.shared.ts:19`, `extensions/memory-core/src/tools.shared.ts:91`).

  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Builtin manager caching already exists:** The builtin path already has module/global caching for `MemoryIndexManager.get()`: it resolves a singleton managed cache and reuses entries keyed by agent/workspace/settings/purpose, only bypassing cache for `purpose: "status"` (`extensions/memory-core/src/memory/manager.ts:64`, `extensions/memory-core/src/memory/manager.ts:155`).
