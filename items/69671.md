## Summary

Keep open. Current `main` still prepends the runtime startup prelude on bare `/new` and `/reset`, and the user-visible sanitization path still does not strip the startup-context markers or quoted-notes block, so the reported leak remains plausible and code-backed.

## Evidence

- **Bare reset/new still prepends startup context into the user prompt body:** `get-reply-run.ts` still builds `startupContextPrelude` for bare session reset/new turns and prepends it into `baseBodyForPrompt`, so the startup prelude is still stored on the user-message path the issue describes.
  - file: [src/auto-reply/reply/get-reply-run.ts:423](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/auto-reply/reply/get-reply-run.ts#L423)
  - command: `nl -ba src/auto-reply/reply/get-reply-run.ts | sed -n '410,520p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Startup-context markers match the issue report:** `startup-context.ts` still emits the exact visible markers from the issue: `[Startup context loaded by runtime]`, `[Untrusted daily memory: ...]`, `BEGIN_QUOTED_NOTES`, `END_QUOTED_NOTES`, and optional truncation text inside the generated prelude.
  - file: [src/auto-reply/reply/startup-context.ts:133](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/auto-reply/reply/startup-context.ts#L133)
  - command: `nl -ba src/auto-reply/reply/startup-context.ts | sed -n '120,220p;360,430p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Visible-history sanitizer still does not recognize startup-context sentinels:** `stripInboundMetadata` only fast-paths on inbound metadata sentinels plus the untrusted-context header, then strips those JSON blocks and active-memory prefixes. It has no handling for `[Startup context loaded by runtime]`, `[Untrusted daily memory: ...]`, or `BEGIN_QUOTED_NOTES`/`END_QUOTED_NOTES`.
  - file: [src/auto-reply/reply/strip-inbound-meta.ts:21](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/auto-reply/reply/strip-inbound-meta.ts#L21)
  - command: `nl -ba src/auto-reply/reply/strip-inbound-meta.ts | sed -n '1,320p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Control UI and gateway history still rely on that sanitizer:** Control UI `extractText()` and gateway `stripEnvelopeFromMessage()` both strip internal runtime context, then call `stripInboundMetadata()` for user-role text before rendering/sending visible history. There is no separate startup-context removal in either path.
  - file: [ui/src/ui/chat/message-extract.ts:11](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/ui/src/ui/chat/message-extract.ts#L11)
  - command: `nl -ba ui/src/ui/chat/message-extract.ts | sed -n '1,220p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Direct local repro against current source leaves the startup prelude visible:** Running `stripInboundMetadata()` from current `main` on a sample startup prelude returned the startup header, daily-memory marker, and quoted-notes fence unchanged, with only the trailing user text preserved after them. That matches the reported leak rather than a fixed state.
  - command: `node --input-type=module -e 'import { stripInboundMetadata } from "./src/auto-reply/reply/strip-inbound-meta.ts"; const input = `[Startup context loaded by runtime]\n[Untrusted daily memory: memory/2026-04-11.md]\nBEGIN_QUOTED_NOTES\n\`\`\`text\nsecret\n\`\`\`\nEND_QUOTED_NOTES\n\nhello`; console.log(stripInboundMetadata(input));'`
