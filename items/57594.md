## Summary

Keep open. Current `main` still appears to reject non-numeric Zalo proactive targets before send: the Zalo plugin advertises a numeric-only target heuristic, outbound resolution returns `Unknown target ...` for inputs that are not considered id-like, while the Zalo send and in-session reply paths themselves accept arbitrary non-blank string `chat_id` values.

## Evidence

- **Zalo plugin still uses numeric-only target detection:** The current Zalo channel plugin wires `targetResolver.looksLikeId` to `isNumericTargetId` with hint `<chatId>`.
  - file: [extensions/zalo/src/channel.ts:201](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/zalo/src/channel.ts#L201)
  - command: `nl -ba extensions/zalo/src/channel.ts | sed -n '198,204p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Numeric heuristic excludes hex-like chat ids:** `isNumericTargetId` only returns true for trimmed strings matching `^\d{3,}$`, so values like `3becaa50ae12474c1e03` are not treated as ids.
  - file: [src/plugin-sdk/reply-payload.ts:315](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/plugin-sdk/reply-payload.ts#L315)
  - command: `nl -ba src/plugin-sdk/reply-payload.ts | sed -n '314,320p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Outbound resolver turns non-id-like inputs into unknown-target errors:** `resolveMessagingTarget` only takes the direct-id path when `looksLikeTargetId(...)` is true; otherwise it falls back to directory lookup and returns `unknownTargetError(...)` on miss.
  - file: [src/infra/outbound/target-resolver.ts:357](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/infra/outbound/target-resolver.ts#L357)
  - command: `nl -ba src/infra/outbound/target-resolver.ts | sed -n '357,446p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Zalo send path itself accepts arbitrary string chat_id values:** The Zalo send helper only trims and requires a non-blank `chatId`, then sends it upstream as `chat_id` without numeric validation.
  - file: [extensions/zalo/src/send.ts:74](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/zalo/src/send.ts#L74)
  - command: `nl -ba extensions/zalo/src/send.ts | sed -n '74,123p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **In-session replies use raw Zalo chat ids:** Inbound Zalo handling records and replies using the raw `chatId`, which matches the report that existing sessions can reply even though proactive target resolution rejects the same id format.
  - file: [extensions/zalo/src/monitor.ts:555](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/zalo/src/monitor.ts#L555)
