## Summary

Keep open. Current `main` still has the delete-before-cleanup ordering and orphan-self reclaim path described in #57019, and I did not find evidence of a fix on `main`.

## Evidence

- **Release drops in-memory ownership before async file cleanup:** `releaseHeldLock` deletes the session key from `HELD_LOCKS` and only then starts the async `handle.close()` / `fs.rm()` cleanup, matching the reported race window.
  - file: [src/agents/session-write-lock.ts:151](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/session-write-lock.ts#L151)
  - command: `nl -ba src/agents/session-write-lock.ts | sed -n '127,173p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Same-PID lock without starttime is treated as orphan when map entry is missing:** `shouldTreatAsOrphanSelfLock` returns `!HELD_LOCKS.has(normalizedSessionFile)` for same-PID payloads that lack a valid `starttime`. That means the temporary gap created during async release still satisfies the orphan-self condition.
  - file: [src/agents/session-write-lock.ts:403](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/session-write-lock.ts#L403)
  - command: `nl -ba src/agents/session-write-lock.ts | sed -n '403,416p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Contended acquire still reclaims the file on orphan-self detection:** When `shouldTreatAsOrphanSelfLock` is true, `acquireSessionWriteLock` forces `stale: true`, adds `orphan-self-pid`, and removes the lock file with `fs.rm(lockPath, { force: true })` before retrying.
  - file: [src/agents/session-write-lock.ts:558](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/session-write-lock.ts#L558)

## Risks / Open Questions

- A same-process reacquire during async release can still lose the on-disk `.lock` file while the new holder believes it owns the lock.
- The current implementation is especially sensitive on paths where lock payloads do not carry a valid `starttime`, because the orphan-self branch depends only on the missing `HELD_LOCKS` entry.
