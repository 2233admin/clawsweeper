## Summary

Keep open. Current main still does not implement the requested prompt transition after visible response content finishes. It has a 30s TUI streaming watchdog mitigation from #67401, but normal TUI idle still depends on a later `final` chat event/lifecycle terminal path, and the gateway/agent lifecycle code still intentionally waits for async flush/finalization work before emitting that terminal event. The issue discussion remains actionable, including reports on v2026.4.9 and v2026.4.14 plus the named worker-agent configuration clue.

## Best Possible Solution

Keep this issue open as the TUI UX tracker. The best fix is an additive, tested completion model: either gateway/agent emits a content-complete or finalizing chat signal before lifecycle terminal cleanup, or the TUI state machine transitions from `streaming` to a distinct non-streaming finalizing state after visible deltas end while preserving active-run ownership. Add regression coverage for delayed lifecycle final events, async channel flushes, concurrent runs, hidden/tool-only deltas, and the named worker-agent path from the discussion.

## Risks / Open Questions

- A TUI-only silence heuristic could incorrectly mark idle while hidden thinking, tool-only output, delivery, or concurrent run work is still legitimate.
- The existing 30s watchdog is useful recovery for lost final events and should be preserved even after adding a more precise content-complete or finalizing transition.
- The named worker-agent reproduction from the discussion should be covered because it may exercise a different lifecycle/flush path than the default `main` agent.
