## Summary

Keep this PR open. Current main already supports numeric Telegram chat IDs when they are passed correctly as `delivery.to`, but it does not implement this PR's remaining compatibility path for a numeric target stored in `delivery.channel`. The PR is plausible unmerged work, but it needs maintainer review around recipient-inference semantics and the bot-raised target-confusion concern before it should land.

## Best Possible Solution

Keep the PR open and review it as a focused cron-delivery fix. The best path is a narrowed patch that either repairs legacy misplaced `delivery.channel` values in doctor/migration or performs runtime recovery while preserving intended allowFrom/implicit delivery semantics, with tests for single-channel recovery, multi-channel ambiguity, and allowFrom behavior.
