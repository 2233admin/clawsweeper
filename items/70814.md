## Summary

Keep open. Current main already supports explicit Telegram numeric announce targets in `delivery.to`, but it does not implement this PR's core recovery for numeric chat IDs mistakenly stored in `delivery.channel`. The PR still carries useful bug-fix work, though maintainers should review the allowFrom/security concern and the unrelated macOS formatting suppressions before landing.

## Best Possible Solution

Keep #70814 open. The best path is a narrow cron resolver fix that handles legacy or malformed `delivery.channel=<Telegram chatId>` only when unambiguous, preserves or explicitly re-applies allowFrom/authorization semantics, and keeps regression tests for the misplaced-channel case. The unrelated macOS formatting suppressions should be removed, split, or clearly justified before landing.
