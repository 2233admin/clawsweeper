## Summary

Keep open. Current main handles explicit Telegram cron targets when `delivery.to` is set, but it does not implement this PR's central fallback for numeric Telegram chat IDs that were stored in `delivery.channel` with no separate `delivery.to`. The PR is still relevant, though it needs maintainer review because the proposed inference path has an unresolved security concern around explicit-target/allowlist semantics and includes unrelated macOS SwiftFormat churn.

## Best Possible Solution

Keep the PR open and ask for a focused revision: implement numeric Telegram target recovery or fail-closed validation in `src/cron/isolated-agent/delivery-target.ts`, add the two resolver regressions for explicit `delivery.to` and misplaced numeric `delivery.channel`, preserve implicit allowlist semantics for inferred targets or document why explicit semantics are safe, and remove unrelated macOS formatting changes from the cron bug-fix branch.

## Risks / Open Questions

- Closing this PR would drop a still-relevant fix for stored cron jobs whose numeric Telegram chat ID is in `delivery.channel` instead of `delivery.to`.
- Landing the PR as-is may bypass implicit delivery allowlist behavior unless the inferred target path preserves implicit-mode policy or otherwise fails closed; this matches the Aisle security concern in the PR discussion.
- The PR also carries unrelated macOS SwiftFormat suppression edits, so maintainers may want the cron resolver fix split from formatting-only churn.
