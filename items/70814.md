## Summary

Keep open. Current main already handles explicit Telegram cron announce targets passed in `delivery.to`, but it does not preserve a numeric Telegram chat ID when that value is stored in `delivery.channel` with no separate `to`. The PR is a focused bug-fix candidate for that remaining path, but it still needs maintainer review because the current PR discussion includes a high-severity bot concern about target inference/allowlist behavior and the branch also contains unrelated macOS SwiftFormat-only changes.

## Best Possible Solution

Keep the PR open, but require maintainer review of the target-inference security model. The best path is a narrow cron resolver fix plus regression coverage for the misplaced numeric `delivery.channel` case, with allowlist behavior explicitly preserved or fail-closed, and unrelated macOS formatting changes removed or separated.
