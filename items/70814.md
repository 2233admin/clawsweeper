## Summary

Keep open. Current `main` still has the single-channel validation gap that lets non-channel `delivery.channel` values through, Telegram still treats numeric chat IDs as valid targets, and the cron resolver still only treats `delivery.to` as explicit. The PR is addressing a real bug, but the proposed `delivery.channel` -> `explicitTo` recovery changes the request into explicit mode, which bypasses the implicit-only `allowFrom` reroute covered by existing tests. That tradeoff still needs maintainer judgment, so this is not a conservative close candidate.
