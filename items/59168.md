## Summary

Current main still keys configured/custom model selection, allowlists, aliases, metadata overlays, per-model params, and runtime resolution by provider plus model id. The issue’s central request to separate an internal model profile key from the upstream API model id remains a real architectural feature request, not an implemented, duplicate, or stale close candidate.

## Best Possible Solution

Keep the issue open for maintainer design. The best path is a backward-compatible model-profile key contract that separates the stable OpenClaw selection key from the upstream API model id, then updates resolution, allowlists, aliases, UI/session persistence, per-model params, context/pricing metadata, provider payload tests, and docs. The exact field could be provider/name as proposed or a clearer explicit profile key, but current main has not implemented the separation.

## Risks / Open Questions

- Changing this would touch persisted model refs, /model and Control UI selection, allowlist semantics, per-model params, context metadata, fallback behavior, provider payloads, and docs, so it needs a deliberate migration rather than auto-close cleanup.
- The proposal’s provider/name choice may conflict with the current role of name as display text; a dedicated profile key or alias-backed design may be safer after maintainer design review.
