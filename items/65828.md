## Summary

Keep this PR open. Current main still has the MiniMax web-search metadata/onboarding mismatch the PR targets: runtime key resolution accepts `MINIMAX_API_KEY`, but MiniMax Search provider metadata, setup visibility, docs, and contract tests still center only the Coding Plan env vars. The PR is focused and not obviously security-sensitive, but the existing review concern about blindly auto-detecting generic MiniMax model keys as search-capable Token Plan keys is a real product decision that should be resolved before merge.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. Preserve the useful MiniMax metadata, docs, and onboarding alignment, but explicitly decide how generic `MINIMAX_API_KEY` should participate in web-search auto-detection: require explicit `plugins.entries.minimax.config.webSearch.apiKey`, add a search-capability/entitlement check if MiniMax exposes one, lower or constrain the blind env auto-detect path, or accept the false-positive risk by maintainer decision. Then add focused tests for MiniMax Search setup visibility and auto-detection under the chosen behavior.

## Risks / Open Questions

- Merging exactly as written may auto-route users who have a generic `MINIMAX_API_KEY` but no MiniMax Token Plan search entitlement to MiniMax Search, causing runtime failures and potentially bypassing another working provider with lower auto-detect priority.
- The PR updates contract metadata but does not add focused setup or auto-detection coverage for the `MINIMAX_API_KEY` behavior or the Token Plan false-positive edge case.
- Changing public docs to present `MINIMAX_API_KEY` as the preferred MiniMax Search env var should wait for maintainer agreement on the intended auto-detection semantics.
