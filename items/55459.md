## Summary

Keep open. Current `main` still has no `op`-specific cache mitigation, and the shipped 1Password exec-provider docs still show a plain `op read` example without the reported workaround. The issue is recent, coherent, and the repository only supports a manual config-side workaround today.

## Evidence

- **Resolver still passes exec args/env generically with no `op` special case:** `resolveExecRefs` builds a minimal child env from `passEnv` and `env`, then spawns the configured command with `args` exactly as provided. There is no auto-injected `OP_CACHE=false`, no command-name detection, and no arg rewriting.
  - file: [src/secrets/resolve.ts:697](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/secrets/resolve.ts#L697)

  - file: [docs/gateway/secrets.md:204](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/gateway/secrets.md#L204)
