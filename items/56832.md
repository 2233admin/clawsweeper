## Summary

Keep open. Current `main` has partial overlap handling for active reply runs, but the reported `/new` race is not clearly fixed: session reset still rotates session state before the active-run guard, and the user-facing error formatter still does not recognize the reported `{"detail":{"code":"deactivated_workspace"}}` payload shape.

## Evidence

- **Reset still happens before run-start overlap handling:** `getReply` initializes session state before it enters the later reply-run queue/overlap logic. That means `/new` can rotate session state first, which matches the reported reset-vs-previous-run race shape.
  - file: [src/auto-reply/reply/get-reply.ts:273](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/auto-reply/reply/get-reply.ts#L273)
  - command: `nl -ba src/auto-reply/reply/get-reply.ts | sed -n '250,390p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **`/new` rotation is immediate in session init:** When reset is triggered, `initSessionState` captures the previous entry and then mints a new `sessionId` in the reset path. There is no active-run abort/wait in this function before that rotation.
  - file: [src/auto-reply/reply/session.ts:459](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/auto-reply/reply/session.ts#L459)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '400,620p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Active-run wait is only a later mitigation:** Current `main` does have an overlap guard that aborts/waits and, if still busy, returns `Previous run is still shutting down`. But that logic lives later in `get-reply-run`/`get-reply-run-queue`, after session initialization, so it does not prove the reset race itself is fixed end-to-end.
  - file: [src/auto-reply/reply/get-reply-run-queue.ts:29](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/auto-reply/reply/get-reply-run-queue.ts#L29)
  - command: `nl -ba src/auto-reply/reply/get-reply-run.ts | sed -n '645,679p' && nl -ba src/auto-reply/reply/get-reply-run-queue.ts | sed -n '1,85p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Raw error formatter still misses the reported payload shape:** `parseApiErrorPayload` only recognizes payloads with top-level `type`, `request_id`, or an `error` object containing `message/type/code`. A payload shaped like `{"detail":{"code":"deactivated_workspace"}}` does not match, so `formatRawAssistantErrorForUi` falls through to returning the raw text unchanged.
  - file: [src/shared/assistant-error-format.ts:27](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/shared/assistant-error-format.ts#L27)
