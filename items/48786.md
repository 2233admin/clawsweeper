## Summary

Keep open. Current `main` still leaves the quoted-message fetch path unnormalized for Feishu mentions, so the reported `@_user_N` placeholders can still flow into reply context. The current-message ingress path normalizes mentions, but `getMessageFeishu()`/`parseFeishuMessageItem()` only parse raw body content and never read a `mentions` array for fetched messages.

## Evidence

- **Quoted fetch path still skips mention normalization:** `parseFeishuMessageItem()` returns `content: parseFeishuMessageContent(rawContent, msgType)` and does not call `normalizeMentions()` or read any mention metadata before `getMessageFeishu()` returns the fetched message.
  - file: [extensions/feishu/src/send.ts:260](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/extensions/feishu/src/send.ts#L260)
  - command: `nl -ba extensions/feishu/src/send.ts | sed -n '250,310p'`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Fetched message type omits mentions entirely:** `FeishuMessageGetItem` on current `main` has `message_id`, `chat_id`, `msg_type`, `body`, `sender`, and `create_time`, but no `mentions` field, so the quoted-message parser has no typed path to resolve `@_user_N` placeholders.
  - file: [extensions/feishu/src/send.ts:84](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/extensions/feishu/src/send.ts#L84)
