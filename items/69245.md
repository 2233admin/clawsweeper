## Summary

Keep PR #69245 open. Current main does not implement the proposed `openai-completions` cache-ttl eligibility path, and the PR still targets a real behavior gap: configured `contextPruning.mode: "cache-ttl"` is gated off unless the provider is covered by built-in Anthropic/Google/OpenRouter-style rules or a provider plugin override. This needs maintainer/product review rather than conservative cleanup, especially because a blanket `modelApi === "openai-completions"` rule may be broader than the existing provider-owned cache policy model.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is either to merge a scoped implementation that enables cache-ttl pruning for `openai-completions` when the transport/provider is known to support prompt caching, or to add an explicit model/provider compat flag such as cache-ttl eligibility so config-only OpenAI-compatible providers can opt in without requiring a full provider plugin. Include focused tests for positive `openai-completions` eligibility and negative unsupported-provider behavior.
