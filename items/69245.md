## Summary

Keep open. Current main still does not enable cache-ttl pruning for generic `openai-completions` providers: eligibility is limited to provider-plugin overrides, Anthropic-family routes, the Kilocode Anthropic model case, and Google prompt-cache eligibility. The PR is a focused, still-relevant change, but it needs maintainer product judgment because current docs and plugin contracts treat cache-TTL support as provider/model-specific rather than an automatic property of every OpenAI-compatible transport.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to decide the product contract first: if OpenClaw wants `cache-ttl` pruning for every OpenAI-compatible transport, merge a variant of this PR with the proposed tests plus a small docs update explaining that `api: "openai-completions"` is eligible when configured. If eligibility should remain tied to actual provider cache semantics, ask the contributor to target the relevant provider plugins through `isCacheTtlEligible` instead of adding a core-wide transport check.

## Risks / Open Questions

- Blanket `modelApi === "openai-completions"` eligibility would include native OpenAI, local OpenAI-compatible servers, and third-party proxies even when cache TTL behavior or cache-hit telemetry may differ by service/model.
- The existing provider hook is designed for provider-owned cache-TTL policy, so maintainers should decide whether this belongs in core as a generic transport rule or should remain an explicit per-provider/plugin opt-in.
