## Summary

Keep open. Current main still gates cache-ttl pruning through `isCacheTtlEligibleProvider`, and that function does not treat `modelApi === "openai-completions"` as eligible. The PR is a focused, recent change against a real current behavior gap, but it still needs maintainer judgment because OpenClaw currently models cache-TTL eligibility as provider/capability-specific rather than as a blanket transport-level property.

## Best Possible Solution

Keep the PR open for maintainer review. The likely path is either to accept the small eligibility expansion with focused tests and docs updates, or to revise it into a capability-gated design that enables cache-ttl pruning only for OpenAI-compatible providers/models with known prompt-cache support, such as explicit provider plugin eligibility or `compat.supportsPromptCacheKey`-style metadata.

## Risks / Open Questions

- A blanket `modelApi === "openai-completions"` rule may over-enable cache-ttl pruning for OpenAI-compatible servers that do not actually provide useful prompt caching or do not support stable prompt-cache keys.
- The current docs and plugin contract frame cache-ttl eligibility as provider/capability-specific, so accepting the PR may require a docs update or a narrower capability-gated implementation.
