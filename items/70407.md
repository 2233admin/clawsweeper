## Summary

Keep open. Current main still validates `memory-core` plugin config against an object-only `dreaming.storage` schema before CLI runtime config is returned, and I found no legacy/doctor migration that converts the reported string form to the object form.

## Best Possible Solution

Add a narrow compatibility migration before plugin schema validation and in `doctor --fix` for `plugins.entries.memory-core.config.dreaming.storage` when it is `"inline"`, `"separate"`, or `"both"`, rewriting it to `{ "mode": <oldValue>, "separateReports": false }`. Add regression coverage for config load/validation and doctor repair with the legacy string shape. If maintainers intentionally want to drop this internal knob, add a high-signal invalid-config hint and upgrade note instead of silently leaving the CLI-blocking schema error.
