## Summary

Keep #69451 open. Current main has useful mitigations, including session maintenance, store caching, preview tail reads, and history pagination surfaces, but the core complaint is still real: hot paths still materialize `sessions.json` as a whole, rewrite it as a whole, and load full transcript JSONL files before applying history limits or cursors. Related #71227 is a useful linked latency/skillsSnapshot subproblem, but it does not fully supersede this memory/RSS issue.

## Best Possible Solution

Keep the issue open and treat it as a core session-storage performance item. The best fix is to remove hot-path full-file materialization: shard or replace `sessions.json` with per-session metadata or SQLite, store large repeated fields such as `skillsSnapshot` out of line, apply list/history limits before row/message materialization, and add regression coverage or benchmarks with large session stores and transcript files. Link #71227 as the latency/skillsSnapshot subproblem rather than closing this as a duplicate.
