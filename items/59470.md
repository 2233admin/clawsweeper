## Summary

Keep open. Current main still dispatches plugin gateway handlers through an explicit `respond` callback and does not convert a handler's returned value into a gateway response, so the reported class of side-effect-success plus CLI timeout remains plausible and actionable.

## Best Possible Solution

Keep the issue open and resolve the product contract explicitly. The likely fix belongs around `src/gateway/server-methods.ts` and the plugin SDK types/docs: either support returned handler payloads as successful gateway responses, or keep the `respond` contract but add a guard/regression test that fails fast with a clear error when a plugin gateway handler completes without responding. Docs under `docs/plugins/*` should make the chosen contract explicit for plugin authors.

## Evidence

- **Issue repro has two independent plugin methods timing out after successful work:** The issue body reports both `discord-component-poc.send` and a lightweight `discord-component-poc.status` timing out at `openclaw gateway call`, while the send path completed and the callback path worked. There were no comments or maintainer labels changing the triage context.
- **Current handler type is callback-based, not return-value-based:** `GatewayRequestHandler` is typed as returning `Promise<void> | void`, with `respond` supplied in the handler options. This does not define a returned payload as the RPC result.

- **Dispatcher awaits the handler but ignores its returned value:** `handleGatewayRequest` selects the plugin/core handler, calls it with `respond`, and awaits the request scope wrapper; there is no post-handler response if the handler returns normally without calling `respond`.
  - file: [src/gateway/server-methods.ts:157](https://github.com/openclaw/openclaw/blob/4ad8b613c9b60466224d9b3737fff7d63ada652a/src/gateway/server-methods.ts#L157)
