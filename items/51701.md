## Summary

Keep open. Current `main` still matches the reported stale-session path: `resolveSession()` can return `isNewSession = true` while handing back the previous `sessionEntry`, and the CLI execution path still reuses that old entry's stored CLI session binding and bootstrap warning state. I did not find an equivalent landed fix or the PR's regression coverage on `main`.

## Evidence

- **`/new` still returns the previous session entry:** `resolveSession()` reads the stored `sessionEntry`, computes `isNewSession` from freshness, and returns both together. On a rollover, callers get `isNewSession = true` but still receive the old `sessionEntry` object.
