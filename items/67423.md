## Summary

Keep this issue open. Current main still has no split-provider auth contract that binds a selected provider entry to a specific auth profile. Runtime auth tries auth-profile order before provider-entry apiKey except for explicit api-key auth overrides, session auto-selection persists profiles from provider auth order, and status labels mirror that same profile-order path. The existing provider apiKey field is typed and documented as credential or SecretRef material, not as an auth-profile reference, so the requested behavior needs an explicit contract rather than cleanup closure.

## Best Possible Solution

Keep this issue open. Maintainers should define an explicit split-provider auth contract, likely a dedicated provider-entry auth profile reference or documented profile-reference semantics, then update runtime auth resolution, session auto-selection, status labels, docs, and focused regression tests for multiple provider entries sharing one upstream auth provider.
