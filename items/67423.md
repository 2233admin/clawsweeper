## Summary

Keep open. Current `main` still routes auth at provider scope, not per provider-entry `apiKey`, unless that entry explicitly sets `auth: "api-key"`. The reported split-provider OpenRouter case is therefore still plausible on `main`, and I found no in-tree fix or targeted coverage for it.

## Evidence

- **Explicit config apiKey only takes priority with auth=api-key:** `shouldPreferExplicitConfigApiKeyAuth` returns true only when the provider entry both sets `auth: "api-key"` and has an explicit `apiKey`, so a plain provider entry `apiKey` does not generally override profile routing.
  - file: [src/agents/model-auth.ts:186](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/agents/model-auth.ts#L186)
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Runtime auth still prefers auth-profile order before env/config apiKey:** `resolveApiKeyForProvider` resolves `auth.order` / auth profiles first and returns the first working profile before later falling back to env vars and then `models.providers.*.apiKey`. That matches the reporter's claim that profile routing can override a split provider entry's configured key.
  - file: [src/agents/model-auth.ts:463](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/agents/model-auth.ts#L463)
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Session auto-selection persists provider-wide auth override:** `resolveSessionAuthProfileOverride` computes `order = resolveAuthProfileOrder({ cfg, store, provider })` and persists `authProfileOverrideSource: "auto"`, so session stickiness is also driven by provider-level auth ordering rather than a provider-entry `apiKey` binding.
  - file: [src/agents/auth-profiles/session-override.ts:87](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/agents/auth-profiles/session-override.ts#L87)
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Auth order model is keyed only by provider/auth alias:** `resolveAuthProfileOrder` builds its order from `cfg.auth.order[provider]`, configured profiles filtered by `providerAuthKey`, and stored profiles for that provider. There is no model-aware or provider-entry-`apiKey` selector in this path.
  - file: [src/agents/auth-profiles/order.ts:66](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/agents/auth-profiles/order.ts#L66)
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Docs still describe auth-profiles first, provider apiKey last:** Current docs state: `auth-profiles.json` -> env vars -> `models.providers.*.apiKey`. That aligns with the code above and does not indicate a shipped fix for split provider entries sharing one upstream auth namespace.
  - file: [docs/gateway/config-tools.md:230](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/docs/gateway/config-tools.md#L230)
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **No targeted split-provider coverage found:** A repo search for issue-specific terms like `openrouter-minimax` and `openrouter:key-b` returned no matches, so I did not find a regression test or implementation explicitly covering the reported split-provider OpenRouter setup.
