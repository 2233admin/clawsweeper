## Summary

Keep this issue open. Current main still lacks a first-class `openclaw notify <message>` command and lacks a typed Gateway/Control UI event that renders visible notifications to connected UI sessions without creating an agent turn. The merged PWA/Web Push work from related PR #44590 is useful context, but it is a subscription-based browser push transport and test-send method, not the requested admin-scoped in-app UI broadcast surface.

## Best Possible Solution

Keep this issue open. The best path is to add an explicit admin-scoped Gateway method such as `ui.notify` or `notifications.broadcast`, a typed Gateway event, a Control UI toast/notification renderer for connected sessions, and a small `openclaw notify <message>` CLI wrapper. If maintainers want plugin ownership, first add the core UI/Gateway notification contract so an external plugin can implement the CLI without reaching into Control UI internals.

## Risks / Open Questions

- A narrow script workflow can use `openclaw gateway call push.web.test` after Web Push subscription setup, but that is subscription-based browser push and test-named, not the requested connected-session UI notification API.
- A production implementation needs maintainer product/security choices around admin scope, rate limiting, abuse/spam UX, toast persistence, and whether Web Push should be reused as a fallback transport.
