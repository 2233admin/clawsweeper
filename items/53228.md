## Summary

Keep this issue open. Current main has overlapping pieces, especially PWA/Web Push from #44590 and admin-scoped `chat.inject`, but it does not provide the requested intentional shell-facing `openclaw notify <message>` command or a generic connected-Control-UI notification/toast broadcast. The existing Web Push RPC is named and documented as a test/subscription surface, and the existing `notify` CLI subcommands are for task policy or node-local macOS notifications, not UI-session alerts.

## Best Possible Solution

Keep this open as a focused core CLI/Gateway/UI feature request. The likely product path is an intentional `openclaw notify <message>` or `openclaw gateway notify` command backed by a clearly named RPC, with operator/admin scope checks, typed UI notification events for connected Control UI/WebChat clients, optional Web Push fan-out for subscribed browsers, and docs covering auth, permissions, rate limits, and failure behavior. Reuse the existing Web Push plumbing where it fits, but do not treat `push.web.test` as the stable notification API without renaming or adding a real wrapper.

## Risks / Open Questions

- Overloading `push.web.test` as the long-term shell notification API would make test-only naming and permission semantics part of the product contract by accident.
- A real implementation needs explicit maintainer judgment on operator/admin scope, who may broadcast to UI clients, whether active WebSocket clients without Web Push permission should receive in-app toasts, and whether Web Push should be a fallback or the primary delivery path.
- Related session-broadcast work in the provided context overlaps conceptually, but it does not eliminate this issue because the requested user-facing CLI/UI notification surface remains unimplemented on current main.
