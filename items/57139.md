## Summary

Keep open. Current main still limits `sessions_history` to a visible live session key/session ID from the session store, while reset flows archive the prior transcript to `*.jsonl.reset.*`. There is still no agent-facing session tool that lists or reads those reset archives, so the reported gap remains.

## Evidence

- **Session tool docs expose no archived-session reader:** Current docs list `sessions_list` and `sessions_history`, and describe `sessions_history` as reading a specific session key or session ID from a prior list call. No archived-session listing or archive selector is documented.
  - file: [docs/concepts/session-tool.md:17](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/docs/concepts/session-tool.md#L17)
  - command: `nl -ba docs/concepts/session-tool.md | sed -n '1,80p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **`sessions_history` only proxies to `chat.history` for one resolved session key:** The tool schema takes `sessionKey`; execution resolves a visible session reference, then calls Gateway `chat.history` with that resolved key. There is no parameter or branch for archive enumeration or reset-archive selection.
  - file: [src/agents/tools/sessions-history-tool.ts:187](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/tools/sessions-history-tool.ts#L187)

- **Gateway chat history reads only the current store entry's transcript candidates:** `chat.history` loads the session entry for the requested key and calls `readSessionMessages(sessionId, storePath, entry?.sessionFile)`. `readSessionMessages` then picks the first existing transcript candidate for that current session entry; it does not scan `*.jsonl.reset.*` archives independently.
  - file: [src/gateway/server-methods/chat.ts:1891](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/server-methods/chat.ts#L1891)

- **`sessions.list` enumerates session-store entries, not archived transcript files:** The listing path starts from `Object.entries(store)` and filters/maps those live store rows. That means archived reset transcripts on disk are not surfaced unless they still have a live store entry, which reset/archive flows do not keep.

That confirms the archived data exists separately from the live session entry.
