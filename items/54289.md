## Summary

Keep open. Current main still implements the reported Gemini CLI OAuth client-config extraction behavior: it locates a local Gemini CLI install, reads oauth2.js or bundle files, regex-parses client ID and secret values, caches the result, and uses it as the fallback OAuth client configuration when explicit env vars are absent. The existing prompt/docs warn that Gemini CLI OAuth is unofficial, but they do not specifically disclose or gate reading OAuth client config from the local Gemini CLI installation.

## Best Possible Solution

Keep this issue open for maintainer security/product review. The likely fix belongs in the bundled Google plugin: remove automatic Gemini CLI source/bundle credential extraction by default, or require explicit extraction-specific opt-in before any filesystem discovery. Update extensions/google/oauth.credentials.ts, extensions/google/gemini-cli-provider.ts, Gemini OAuth tests, and the public Google/Gemini provider docs so the behavior and consent model are explicit. Review #41991 against that direction before landing diagnostic changes that preserve the same fallback.

## Risks / Open Questions

- Security severity and product policy are still maintainer calls: the trust model may make this hardening rather than a boundary-bypass vulnerability, but the shipped bundled behavior still matches the observable complaint.
- Removing automatic extraction outright may break existing Gemini CLI OAuth onboarding for users who rely on automatic client-config discovery from installed Gemini CLI layouts.
- A diagnostics-only PR that preserves the same fallback could make failures clearer while leaving the consent/security concern unresolved.
