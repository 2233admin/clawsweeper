## Summary

Keep open. Current `main` still has a concrete path that can call a non-channel plugin’s `register()` more than once during one gateway startup: the startup bootstrap loads `startupPluginIds` once with `preferSetupRuntimeForChannelPlugins` enabled when deferred channels exist, then reloads the same `startupPluginIds` after listen. Because that flag is part of the plugin-registry cache key, the second pass is a distinct cache miss, and non-channel plugins still register in `"full"` mode on both passes.

## Evidence

- **Gateway startup includes memory plugins in the startup plugin set:** `resolveGatewayStartupPluginIds()` treats memory plugins as gateway-startup plugins via `hasKind(plugin.kind, "memory")` and includes the explicit memory-slot plugin when enabled, so a memory plugin like the reported LanceDB plugin is in `startupPluginIds`, not just channel plugins.

  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **First startup load uses the startup plugin set with deferred-channel mode enabled:** Gateway bootstrap resolves `startupPluginIds` and calls `loadGatewayStartupPlugins(...)` with those ids. When any deferred configured channel exists, it sets `preferSetupRuntimeForChannelPlugins: true` for that load.
