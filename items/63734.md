## Summary

Current main does not implement the requested built-in L1-L4 command-text risk classifier for exec approvals. The shipped exec approval model is still policy/allowlist/durable-approval/safeBins based, with ACP-only readonly auto-approval kept separate. The existing plugin hook surface can block or require extra plugin approval for tool calls, but it does not provide a documented exec-policy API that can safely auto-allow host exec through the core approval policy, so this should stay open for maintainer product/security design.

## Best Possible Solution

Keep this issue open for maintainer decision. If pursued in core, implement it as an explicit opt-in exec policy layer with clear threat-model docs, config/UI affordances, and regression tests for safe reads, pipelines, wrappers, interpreters, Windows/POSIX parsing, and destructive-command denial. If maintainers prefer plugin-first classification, first add or document an exec-specific plugin API that can safely classify/block/require-approval/auto-allow while preserving host policy boundaries.

## Risks / Open Questions

- A broad built-in command classifier would materially change the exec security posture and needs maintainer/security design for shell parsing, POSIX and Windows behavior, pipelines, wrappers, aliases, interpreters, and destructive pattern semantics.
- A plugin-first solution likely needs a documented exec-policy hook that can classify, block, require approval, and optionally auto-allow without unexpectedly weakening the host approval policy.
