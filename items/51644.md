## Summary

Keep open. Current `main` already ships rapid same-sender inbound batching via `messages.inbound`, but the issue also asks for semantic long-message multi-topic splitting and a true preprocessing layer. I found evidence for the first half and no corresponding inbound topic-splitting implementation on current `main`.

## Evidence

- **Docs: rapid consecutive inbound messages already batch:** The user-facing docs say rapid consecutive messages from the same sender are batched into a single agent turn via `messages.inbound`, with per-channel debounce overrides.
  - file: [docs/concepts/messages.md:37](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/docs/concepts/messages.md#L37)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Core implementation: inbound debouncer buffers same-key items:** `createInboundDebouncer` groups items by key, appends later items to the existing buffer, and flushes the combined batch after the debounce timeout.
  - file: [src/auto-reply/inbound-debounce.ts:166](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/auto-reply/inbound-debounce.ts#L166)
