## Summary

Keep open. Current main no longer has the old `src/web` paths and the gateway stale-socket policy now ignores app-level `lastEventAt` without explicit transport liveness, but the WhatsApp plugin still has an extension-level quiet-connection watchdog that force-closes after `messageTimeoutMs` using only `lastInboundAt ?? startedAt`. There is still no WhatsApp presence-ping/liveness hook on the listener contract or heartbeat path, and tests still assert watchdog reconnect behavior rather than proving a successful socket ping suppresses quiet-channel restarts.

## Best Possible Solution

Keep the PR open and replace/rebase it with a WhatsApp-plugin patch under `extensions/whatsapp`. The best implementation is a narrow provider-proven liveness probe, likely backed by Baileys `sendPresenceUpdate("available")`, called from the heartbeat/watchdog path; refresh watchdog/status only when the probe succeeds, preserve forced reconnect when it fails, and add regression tests for both successful and failed ping behavior.
