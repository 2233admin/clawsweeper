## Summary

Keep open. Current `main` has partial channel-specific liveness UX (typing keepalives, status reactions, draft previews), but there is no generic gateway/channel-delivery silence watchdog keyed to last operator-visible output, no `channels.<id>.silenceWatchdog` config surface, and the closest core mechanism resets on internal phase changes rather than actual delivered output.

## Evidence

- **No generic silence-watchdog implementation surfaced:** Targeted search found no `silenceWatchdog`, `last_operator_visible_output`, or operator-visible silence tracker on current `main`; matches were unrelated polling/watchdog code and test strings.
  - command: `rg -n "silenceWatchdog|last_operator_visible_output|operator-visible silence|thresholdMs|maxBeacons|Still working" src docs extensions CHANGELOG.md`
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Current config surface is global status reactions, not per-channel watchdog messages:** The message config exposes `messages.statusReactions.{enabled,emojis,timing}` only. There is no per-channel watchdog message/threshold/max-beacon contract like the issue proposes.
  - file: [src/config/types.messages.ts:64](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/config/types.messages.ts#L64)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Closest core mechanism is reaction-based and resets on phase changes:** `createStatusReactionController()` only drives reaction emoji changes and resets its stall timers on each phase change. That does not track `last operator-visible output` and can be refreshed by internal lifecycle events instead of actual delivered channel output.
  - file: [src/channels/status-reactions.ts:212](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/channels/status-reactions.ts#L212)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Slack wiring refreshes status from reasoning/tool callbacks:** Slack feeds `onReasoningStream` into `statusReactions.setThinking()` and `onToolStart` into `setTool(...)`, so the existing stall UX is driven by internal reasoning/tool events, not by confirmed operator-visible deliveries.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:878](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/extensions/slack/src/monitor/message-handler/dispatch.ts#L878)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Existing long-run feedback is channel-specific, not a generic gateway watchdog:** Current `main` ships partial mitigations like Slack status reactions and Mattermost draft-preview streaming, which reduce silence on those channels but do not implement the requested cross-channel gateway watchdog.
  - file: [CHANGELOG.md:368](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/CHANGELOG.md#L368)
  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)

## Risks / Open Questions

- Some channels already provide enough feedback for parts of this UX problem, so future triage should distinguish between a generic gateway watchdog and channel-specific mitigations.
- A deeper runtime audit might find more per-channel signals, but the inspected code/docs/history do not show the requested generic, operator-visible-output-based watchdog.
