## Summary

Keep open. Current main still does not implement the PR's central cron behavior: `cron.retry` remains documented and wired for one-shot jobs, while recurring job errors still use the generic error-backoff floor and the later of natural next run/backoff. The PR is also not a conservative auto-close candidate because the provided review discussion flags substantive cron regressions and the branch includes unrelated gateway pairing/protocol generated-model changes.

## Best Possible Solution

Keep the PR open for maintainer handling, but ask for a trimmed cron-only patch or consolidation with the existing recurring-cron-retry tracking work. The best implementation should apply configured transient retry backoff to recurring cron/every jobs before the next natural schedule, fall back to ordinary recurring scheduling after the retry budget, preserve existing non-transient and #66019 safety behavior, update config docs/schema/help, and add focused regression tests for cron, every, exhausted budget, startup catch-up, and unresolved next-run cases.
