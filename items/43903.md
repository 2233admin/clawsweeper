## Summary

Keep open. Current main still has a single shared gateway token/password auth model, and token-authenticated Control UI sessions do not resolve a per-token label, user, or scope identity. Trusted-proxy and Tailscale remain identity-bearing alternatives, but they do not satisfy the requested simpler multi-token mode. The issue discussion is coherent and includes a corroborating multi-user deployment comment.

## Best Possible Solution

Keep this issue open for a deliberate core gateway design. A good implementation would add a backwards-compatible multi-token auth contract, resolve token metadata to a non-secret authenticated identity, carry label/scopes through HTTP and WebSocket auth results, feed that identity into Control UI sender/session metadata, preserve current single-token/device-pairing behavior, and update config schema, docs, and gateway auth/session tests.

## Risks / Open Questions

- This is a security-sensitive core gateway auth/session contract change, not a documentation-only cleanup.
- A future implementation needs constant-time matching across multiple tokens without revealing which token matched, careful SecretRef/env precedence, redaction in logs/config/status, rate-limit behavior that does not leak token inventory, and clear scope semantics.
- Trusted-proxy and Tailscale identity paths partly solve identity today, but they require external identity infrastructure and do not preserve the requested token-mode/local CLI/device-pairing workflow.
