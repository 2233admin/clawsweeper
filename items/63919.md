## Summary

Keep open. Current main has the HTTP /tools/invoke endpoint and a plugin-backed fallback for cataloged tools such as browser, but it still does not wire createOpenClawCodingTools into resolveGatewayScopedTools. The PR addresses a real remaining gap for direct exec/read/write/edit style tool invocation, tied to open issue #37131, but the current PR shape needs maintainer/security review before landing because coding tools are already before_tool_call-wrapped while the HTTP handler also runs the hook.

## Best Possible Solution

Keep #63919 open and ask for a revised, security-reviewed implementation rather than closing it. The likely path is a dedicated HTTP-safe coding-tool composition option: include coding tools only for surface="http", preserve gateway-tool precedence and the existing default deny list, avoid double before_tool_call wrapping or explicitly consume adjusted params on the HTTP path, and add regression coverage for an allowed non-denied coding tool, default-denied exec/apply_patch, MCP loopback exclusion, and hook/adjusted-param behavior. Coordinate the product scope with the still-open direct invocation request #37131.

## Risks / Open Questions

- Naively merging the current PR shape could double-run before_tool_call hooks and loop detection for HTTP coding tools.
- The wrapped coding-tool path can store adjusted params for HTTP toolCallIds without an obvious HTTP-side drain, so repeated large /tools/invoke calls need an explicit cleanup or unwrapped-tool design.
- Exposing exec/write/apply_patch over HTTP is a sensitive operator surface even with default deny; the final design should preserve loopback exclusion, default-deny behavior, owner-only policy, and clear docs.
