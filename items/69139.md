## Summary

Keep PR #69139 open. Current main still has the exact schema/handler behavior the PR targets: `AgentParamsSchema` is strict, has no root-level `paperclip` field or generic metadata slot, and the `agent` handler validates against it before dispatch. Main only relaxed opaque metadata for `wake` params, not `agent` requests, while the PR discussion includes a live repro confirming Paperclip heartbeats fail on this validation boundary.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is an additive gateway protocol change accepting opaque Paperclip metadata on `req agent` params, with a focused regression test in the gateway agent handler or protocol validator; if maintainers prefer a generic metadata envelope instead of a Paperclip-named field, update the PR and adapter/docs contract together.
