## Summary

Keep open. Current `main` still handles queue policy only at enqueue/active-run time, then sends the completed final reply without a last-moment pending-queue check. I did not find an implemented stale-reply suppression path matching #48814.

## Evidence

- **Queue policy acts before or during the run, not at final delivery:** `src/auto-reply/reply/get-reply-run.ts:596` checks lane size up front, and `:597-603` only applies `interrupt` there. The busy-session action is resolved at `:633-644`, so the existing queue modes are decided before final reply dispatch.

That path has no followup-queue/session-lane inspection before sending the final payload.
