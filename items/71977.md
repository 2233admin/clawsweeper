## Summary

Keep open. Current main has fixes for exact duplicate Feishu streaming-card finals and distinct finals that arrive before the stream goes idle, but it still starts a new FeishuStreamingSession for any non-identical final payload delivered after the previous streaming card has been closed. That matches the new report’s narrower multi-tool/distinct-final case more than an already-implemented fix.

## Best Possible Solution

Keep the issue open and add a focused Feishu regression for streaming=true/renderMode=card where a streaming card is closed and then several distinct final segments arrive. The likely fix belongs in extensions/feishu/src/reply-dispatcher.ts: retain enough closed-stream state for the current turn to suppress or safely fold post-close final fragments already represented by the completed streaming card, while preserving true separate final replies and media. If maintainers revive #56435’s in-place update approach, this issue should be covered by that work rather than closed before the behavior is actually on main.
