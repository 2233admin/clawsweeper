## Summary

Keep this issue open. Current main has ACP thread binding, output streaming, `/acp steer`, and native sub-agent steer controls, but ordinary messages in a bound ACP chat/thread still become serialized ACP turns. The ACP runtime/manager contract has no active-turn input path that can receive a Discord/thread message before the current `runTurn` completes, and native sub-agent steering remains restart-based rather than live same-turn input.

## Best Possible Solution

Keep this issue open. The best fix is to add a real ACP active-turn input capability outside the per-session actor queue, expose it through the ACP manager/runtime contract, and have bound chat/thread dispatch send ordinary follow-up messages to that path while a turn is active. Add regression coverage where a second Discord/thread-bound message reaches the active ACP runtime before the first `runTurn` completes. For native sub-agents, make the intended behavior explicit and either implement live same-turn steering in the embedded runner/session control path or document/test that chat replies remain queued or restart-based.
