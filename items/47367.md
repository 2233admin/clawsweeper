## Summary

Keep open. Current `main` does not show a fix for the reported WhatsApp QR handshake timeout: the login path retries only status `515`, treats `401` as logged-out cleanup, and returns all other failures such as `408` as terminal before any QR is shown.

## Evidence

- **408 still falls through as terminal login failure:** `waitForWhatsAppLoginResult()` only special-cases `515` restart and `401` logout; every other status returns `outcome: "failed"` with the formatted error, which matches the reported `408 Request Time-out WebSocket Error (Opening handshake has timed out)` behavior.
  - file: [extensions/whatsapp/src/connection-controller.ts:179](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/extensions/whatsapp/src/connection-controller.ts#L179)

  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **QR flow aborts on early login failure:** `startWebLoginWithQr()` races the QR promise against the login waiter; if the waiter fails before a QR arrives, it returns `WhatsApp login failed: ...` and resets the active login instead of retrying the socket/handshake.
