## Summary

Keep #71060 open. Current `main` has already hardened the `gateway restart` path so it uses `launchctl kickstart -k`, re-bootstraps when the service is not loaded, and hands off restarts invoked from the LaunchAgent process. However, the issue's `gateway stop` failure mode is still present in source and tests: the stop path waits only 10 x 100 ms, then falls back to `bootout` with warning text that it left the service unloaded. That is not fully covered by the older closed restart-focused reports in the provided related context.

## Best Possible Solution

Keep the issue open but narrow maintainer handling to the remaining `gateway stop` semantics: either avoid bootout for slow graceful stops, extend the confirmation budget to match gateway drain behavior, or re-bootstrap/record a recoverable disabled-but-loaded state after any bootout fallback. The already-shipped restart hardening and the docs warning against stop/start restart chains should be treated as partial mitigation, not a full close reason.
