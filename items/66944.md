## Summary

Keep open. Current main still lacks the requested native Control UI page contribution seam for plugins: navigation and tab rendering are static, bootstrap config carries no plugin page/import-map metadata, the manifest contract has no ui.pages field, and the plugin API only covers standalone HTTP routes/services/gateway methods for this surface. The discussion also includes a concrete Openclaw-Quest standalone-dashboard workaround and unresolved API/security design questions, so this remains valid plugin/API work rather than cleanup.

## Best Possible Solution

Keep this open as the canonical design and implementation track for a versioned Control UI plugin-page API. A strong solution would add manifest and/or SDK page declarations, gateway bootstrap metadata, static asset/auth routing, a browser Custom Element loader, navigation grouping, native tab rendering, docs/tests, and explicit security decisions for non-Lit Custom Elements and plugin-owned backend traffic.

## Risks / Open Questions

- Loading third-party plugin JavaScript inside the admin Control UI shell is security-sensitive and needs explicit design for CSP/sandboxing, auth scope, gateway client exposure, same-origin plugin HTTP/WS access, dependency/runtime sharing, unload behavior, and version compatibility.
- Closing as clawhub would be premature because the current plugin surfaces can host standalone pages but cannot make native Control UI tabs; VISION.md points missing plugin capabilities back to OpenClaw API design rather than external-only publication.
