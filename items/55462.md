## Summary

Keep open. Current `main` still routes Telegram `callback_query` events with the callback query ID as inbound `message_id`, not the source Telegram message's `message.message_id`, so the reported edit-targeting gap is still present.

## Evidence

- **Callback handler overrides inbound message id with callback id:** The Telegram `callback_query` handler builds a synthetic message from `callback.message`, but then calls `processMessage(..., { messageIdOverride: callback.id })`. That replaces the inbound message identifier with the callback query ID instead of the original Telegram message ID.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1771](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/telegram/src/bot-handlers.runtime.ts#L1771)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '1188,1785p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Session payload uses override as MessageSid:** Inbound Telegram session context sets `MessageSid` from `options?.messageIdOverride ?? String(msg.message_id)`, so the callback path above causes `MessageSid` to become the callback query ID rather than `callback.message.message_id`.
  - file: [extensions/telegram/src/bot-message-context.session.ts:342](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/telegram/src/bot-message-context.session.ts#L342)
  - command: `nl -ba extensions/telegram/src/bot-message-context.session.ts | sed -n '300,380p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Agent-visible conversation metadata is sourced from MessageSid:** The generic inbound metadata formatter emits `conversationInfo.message_id` from `ctx.MessageSid`/`ctx.MessageSidFull`. Because the Telegram callback path overrides `MessageSid`, agents see the callback ID in conversation metadata instead of the original Telegram message ID needed for `editMessage` targeting.
  - file: [src/auto-reply/reply/inbound-meta.ts:202](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/auto-reply/reply/inbound-meta.ts#L202)
  - command: `nl -ba src/auto-reply/reply/inbound-meta.ts | sed -n '178,240p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Current callback regression test does not cover original message_id forwarding:** The existing callback test only checks that the callback payload is routed as a message and that `answerCallbackQuery` is called. It does not assert that the original Telegram `message_id` is forwarded to the agent context.
  - file: [extensions/telegram/src/bot.create-telegram-bot.test.ts:536](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/telegram/src/bot.create-telegram-bot.test.ts#L536)
  - command: `nl -ba extensions/telegram/src/bot.create-telegram-bot.test.ts | sed -n '536,562p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)

## Risks / Open Questions

- Agents handling Telegram inline-button callbacks still lack the original target message ID in their inbound metadata, so workflows that need `editMessage(chatId, messageId)` remain unreliable or impossible without extra plugin-local workarounds.
- A fix here needs care: Telegram likely still needs callback-query dedupe/identity, so preserving callback ID may require a separate metadata field rather than replacing the message ID.
