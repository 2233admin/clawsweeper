## Summary

Keep PR #69567 open. Current main still builds base64 `AUTHORIZATION: basic ${auth_header}` values in custom checkout blocks without registering the derived value with `::add-mask::`, and the PR is a focused hardening change for that real gap. A sibling workflow, `.github/workflows/ci-check-testbox.yml`, appears to have the same pattern, so maintainers should review whether to include that block before merging rather than closing the PR.

## Best Possible Solution

Keep the PR open and review it as a plausible security hardening fix. The best code path is to register every generated `auth_header` with `echo "::add-mask::${auth_header}"` immediately after construction and before any Git command can use or emit it, including the same custom checkout pattern in `.github/workflows/ci-check-testbox.yml` if maintainers agree it is in scope. Then require the workflow sanity/zizmor lane and normal CI for the updated head before merge.
