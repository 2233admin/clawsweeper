## Summary

Current `main` does not give strong enough evidence to close #51143. The Telegram pipeline still installs one generic `bot.on("message")` handler per bot instance for group and supergroup traffic, while `defaultAccount` is only used to choose the implicit account when none is specified. The reporter's key diagnostic claim also does not match current code: the raw-update logger middleware is registered before the `bot.on("message")` handler, so its firing does not prove the grammY `message` filter failed. I also found existing tests that document intentional multi-account config asymmetry around top-level `groups` inheritance, which could still interact with the reported setup. Without a live repro on current `main` or a targeted fix/test proving the bug is obsolete, the conservative maintainer outcome is to keep this open.
