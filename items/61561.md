## Summary

Keep open. Current `main` still lacks the `iMessageLite` chat GUID normalization this PR is trying to add, so the underlying BlueBubbles satellite-message gap is not implemented on `main`. The PR also appears incomplete as written: OpenClaw documents `chat_guid:` allowlist entries for BlueBubbles, and current allowlist matching compares chat GUIDs exactly, so normalizing inbound webhook `chatGuid` values without canonicalizing allowlist `chat_guid:` entries could break matching for any config that stores the raw `iMessageLite;...` form.

## Evidence

- **Current main still returns raw chat GUID services unchanged:** `parseRawChatGuid` validates the three-part GUID and returns `${service};${separator};${identifier}` directly. There is no `iMessageLite -> iMessage` remap on current `main`. That means user-supplied `chat_guid:` targets also stay unnormalized.
