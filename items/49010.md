## Summary

Keep open. Current `main` does not implement message-level signing for stdio MCP transport, and the report is specific and still actionable. The transport still writes raw serialized JSON-RPC messages to stdio with no signing/verification envelope, and the public MCP server config only exposes transport basics such as `command`, `args`, `env`, `cwd`, `url`, `transport`, `headers`, and `connectionTimeoutMs`. `VISION.md` does say unsupported `mcporter` features should be opened upstream, but that is not one of the allowed close reasons here and is not strong enough to close this as maintainer cleanup.

## Evidence

- **Raw stdio transport has no signing hooks:** `src/agents/mcp-stdio-transport.ts:133` sends `serializeMessage(message)` directly to child stdin, and `src/agents/mcp-stdio-transport.ts:98` reads messages straight from `ReadBuffer` into `onmessage` with no signature, nonce, timestamp, or identity verification layer.
  - file: [src/agents/mcp-stdio-transport.ts:133](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/agents/mcp-stdio-transport.ts#L133)
