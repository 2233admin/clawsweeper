## Summary

Keep this PR open. Current main still lacks outbound Slack `replyBroadcast` / Slack API `reply_broadcast` support, so the PR is not obsolete. It remains an implementation candidate, but the provided PR discussion and diff show merge-readiness gaps around current message-tool params, upload/media behavior, and schema/dispatch alignment.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto current main, wire `replyBroadcast` through the current Slack-owned message-tool schema/dispatch/action/send path, emit Slack `reply_broadcast: true` only for eligible threaded `chat.postMessage` calls, explicitly support or reject upload/media semantics, update docs to use the current `message` tool contract (`action`, `message`, `threadId`/`replyTo`), and add focused Slack send/dispatch/discovery regression tests. Keep broader inbound/automatic mirroring behavior tracked separately in #63904 unless a maintainer chooses to merge the scopes.
