## Summary

Keep open. The underlying bug still exists on current `main`, but the proposed PR also appears to widen skill env injection before skill eligibility/filtering, which makes the current patch unsafe to auto-close or treat as already implemented.

## Evidence

- **Current main still blocks user-configured sensitive-pattern skill env vars:** `applySkillConfigEnvOverrides()` only seeds `allowedSensitiveKeys` from `primaryEnv` and `requiredEnv`; it does not add keys from `skillConfig.env` before `sanitizeSkillEnvOverrides()` re-checks blocked names. On current `main`, a configured `*_TOKEN`/`*_PASSWORD` key in `skills.entries.*.env` is still dropped unless it is already declared in metadata.
  - file: [src/agents/skills/env-overrides.ts:150](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/skills/env-overrides.ts#L150)
  - command: `nl -ba src/agents/skills/env-overrides.ts | sed -n '142,193p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **No-snapshot embedded runs load raw skill entries:** `loadWorkspaceSkillEntries()` returns unfiltered entries when there is no effective skill filter, so the embedded-runner path can receive the full discovered skill list rather than only eligible/selected skills.
  - file: [src/agents/skills/workspace.ts:847](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/skills/workspace.ts#L847)
  - command: `nl -ba src/agents/skills/workspace.ts | sed -n '836,853p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Env overrides are applied before prompt/eligibility filtering in embedded runs:** In the no-snapshot path, `attempt.ts` calls `applySkillEnvOverrides({ skills: skillEntries ?? [], ... })` immediately after loading entries, and only then resolves the skills prompt. That ordering matches the review concern that broadening sensitive-key allowance at injection time would affect all loaded skills for the run.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:599](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/pi-embedded-runner/run/attempt.ts#L599)

- **Eligibility currently trusts global process env:** `shouldIncludeSkill()` treats `process.env[envName]` as sufficient for `requires.env`, with no distinction between host env and skill-injected env. If the PR injects more sensitive names globally, unrelated skills can become eligible from another skill's configured secret.
  - file: [src/agents/skills/config.ts:97](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/skills/config.ts#L97)

- **PR discussion already surfaced unresolved regression risk:** The provided review discussion called out that allowing every `skills.entries.*.env` sensitive key would inject secrets even for disabled/unselected skills in the no-snapshot embedded-runner path, and also flagged duplicate-key first-wins leakage between skills. Current `main` code structure supports those concerns.
  - sha: [ba03176b7666](https://github.com/openclaw/openclaw/commit/ba03176b766676c61b84c42e8faa19128c41148f)

## Risks / Open Questions

- Merging the PR as-is would expand global `process.env` injection for sensitive keys before skill selection in embedded runs.
- Duplicate sensitive env names across skills would still share the existing first-wins global env behavior, so one skill could inherit another skill's credential.
