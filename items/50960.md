## Summary

Keep open. Current `main` still blocks user-configured `skills.entries.*.env` keys that match the generic sensitive-name patterns unless those keys are already declared in the skill metadata, so the reported bug is not implemented on `main`. The proposed PR also appears incomplete: the embedded runner applies skill env overrides before skill eligibility filtering, so broadly allowlisting every configured sensitive key would inject secrets for disabled or otherwise excluded skills.

## Evidence

- **Current sanitizer still blocks generic sensitive env names:** `sanitizeEnvVars` blocks `*_TOKEN`, `*_PASSWORD`, `*_SECRET`, etc., and `applySkillConfigEnvOverrides` only seeds `allowedSensitiveKeys` from `primaryEnv` and `requires.env` before calling `sanitizeSkillEnvOverrides`. It never adds arbitrary keys from `skillConfig.env`, so a configured key like `GOG_KEYRING_PASSWORD` would still be dropped on current `main`.

  - file: [src/agents/skills.test.ts:667](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/skills.test.ts#L667)

## Risks / Open Questions

- Closing this PR would discard a still-unfixed bug on `main` for user-configured sensitive-pattern skill env vars.
- Merging the PR as described would likely inject sensitive env vars for skills that are later filtered out, because overrides happen before eligibility checks.
- Even beyond the original bug, current `main` still lets one skill's injected env satisfy another skill's `requires.env`, so the area still needs maintainer attention rather than closure.
