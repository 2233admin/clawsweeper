## Summary

Keep #58411 open. Current main still has the exact slash-command/tool asymmetry for Discord: `sessions_spawn` exposes only `thread: true` for ACP binding, while `/acp spawn --bind here` has a separate current-conversation path. On Discord, the ACP `sessions_spawn({runtime:"acp", thread:true, mode:"session"})` path still selects child placement and creates/binds a child thread, even when the requester is already in a Discord thread.

## Best Possible Solution

Keep the issue open and add an explicit programmatic current-conversation ACP binding path for `sessions_spawn`, likely as an additive parameter such as `bindTo: "current"` or a typed placement option. The implementation should route that option through `spawnAcpDirect`, bind with `placement: "current"` against the requester conversation, preserve existing `thread:true` child-thread behavior for Discord, and add focused tests for Discord existing-thread/current-channel binding plus docs/tool-description updates.
