## Summary

Keep open: the report is thin, but current `main` still has a plausible code path where macOS Channels can render with no options, and I could not find strong evidence that the issue is already fixed or obsolete.

## Evidence

- **Gateway status uses loaded plugin list for channel UI:** `channels.status` reads `plugins = listChannelPlugins()` and then serializes `channelOrder` from `buildChannelUiCatalog(plugins)`. If the loaded channel registry is empty at request time, the payload's channel list is empty too.
  - file: [src/gateway/server-methods/channels.ts:162](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/gateway/server-methods/channels.ts#L162)
  - command: `nl -ba src/gateway/server-methods/channels.ts | sed -n '157,166p;289,296p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Loaded-plugin lookup has no bundled fallback for listing:** `listChannelPlugins()` returns only `listLoadedChannelPlugins()`. The test suite separately documents that direct lookup falls back to bundled plugins, but listing can still be empty when the runtime registry is malformed or not populated.
  - file: [src/channels/plugins/registry.ts:8](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/channels/plugins/registry.ts#L8)
  - command: `nl -ba src/channels/plugins/registry.ts | sed -n '1,40p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Registry test proves empty-list condition exists:** `registry.test.ts` explicitly expects `listChannelPlugins()` to return `[]` when the runtime registry has no `channels` field, while bundled fallback is only exercised through `getChannelPlugin()`.
  - file: [src/channels/plugins/registry.test.ts:28](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/src/channels/plugins/registry.test.ts#L28)
  - command: `nl -ba src/channels/plugins/registry.test.ts | sed -n '27,39p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **macOS Channels view trusts empty channelOrder:** The macOS UI uses `self.store.snapshot?.channelOrder ?? fallback`. Once a snapshot exists, an empty `channelOrder` does not fall back to the hard-coded channel list, so the sidebar can render no channel rows.
  - file: [apps/macos/Sources/OpenClaw/ChannelsSettings+ChannelState.swift:333](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/apps/macos/Sources/OpenClaw/ChannelsSettings%2BChannelState.swift#L333)
  - command: `nl -ba apps/macos/Sources/OpenClaw/ChannelsSettings+ChannelState.swift | sed -n '333,358p'`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Smoke tests do not cover empty/intermittent snapshot ordering:** The macOS smoke tests only build the view with nonempty canned channel snapshots/body construction and do not cover a snapshot whose `channelOrder` is empty.
  - file: [apps/macos/Tests/OpenClawIPCTests/ChannelsSettingsSmokeTests.swift:44](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/apps/macos/Tests/OpenClawIPCTests/ChannelsSettingsSmokeTests.swift#L44)
