## Summary

Keep open. Current main only partially addresses the reported class by stripping provider auth env from the default `openclaw acp` client bridge. The ACPX harness path used by `/acp spawn gemini` still lacks generic auth-contract metadata, per-agent Gemini/GenAI env scrubbing, and fail-closed auth preflight hooks, and the docs still treat harness auth as host-owned setup rather than documenting mixed OAuth/API-key drift risk.

## Best Possible Solution

Keep this issue open. The best path is an additive ACP auth-contract design in the OpenClaw control plane/plugin API, passed through `AcpRuntimeEnsureInput` and manager initialization/resume; then wire the ACPX Gemini route to declare OAuth expectations, scrub Gemini/Google GenAI API-key env vars when OAuth is required, fail closed on auth-contract mismatch, and document mixed OAuth/API-key host risks with focused ACP manager and ACPX Gemini tests.
