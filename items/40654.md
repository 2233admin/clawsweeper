## Summary

Current main still does not implement the PR’s requested behavior. `/acp close` closes the ACP session and unbinds bindings, but it does not mark ACP-created Discord child threads, emit archive-after-reply channel data, or archive the Discord thread after the reply. The PR remains a plausible implementation candidate, but it needs to be ported from the older `src/discord` / `src/channels/plugins/outbound/discord.ts` layout to the current `extensions/discord` plugin layout and reviewed for the archive command surface.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best maintainer path is to port it onto current main by storing ACP-created Discord child-thread ownership metadata during ACP thread binding, having `/acp close` request archive-after-reply only after close/unbind succeeds for the current owned Discord thread, implementing the post-reply archive in `extensions/discord`, and adding focused tests for ACP-created threads, manual/reused threads, non-Discord bindings, archive failures, degraded close notices, and outbound send overrides.

## Risks / Open Questions

- The archive side effect is persistent and user-visible; a port should fail closed unless the removed binding is the current Discord thread and has reliable ACP-created child-thread ownership metadata.
- The PR’s proposed `channelData.discord.archiveCurrentThreadAfterReply` should be kept as an internal command path or otherwise guarded. If arbitrary assistant/plugin outbound payloads can set that flag, they could archive Discord threads outside the intended `/acp close` flow.
- The review-fixed details need to survive the port: degraded close `runtimeNotice`, archive failure partial-success text, and injected `sendDiscord` override behavior.
