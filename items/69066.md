## Summary

Keep #69066 open. Current main does not implement the RFC’s separate internal service identity axis: gateway auth still exposes only user-facing modes, trusted-proxy loopback remains fail-closed, internal/CLI clients still authenticate through shared token/password/device auth, and docs/tests explicitly describe that behavior. The related items are narrower tactical bugs/PRs or broader RBAC/security topics, not a canonical replacement for this gateway-auth design discussion.

## Best Possible Solution

Keep the RFC open for maintainer product/security judgment. The best implementation path is an additive core gateway-auth change: introduce a documented `gateway.service` contract, token generation/storage/rotation, gateway-owned child env injection, `GatewayClient` service-token selection, an `authorizeGatewayConnect` service check before user auth, scoped backend/operator permissions, and regression tests covering trusted-proxy deployments with internal callers. Coordinate this with narrower trusted-proxy/local-backend bugs and RBAC follow-ups rather than auto-closing this umbrella design item.
