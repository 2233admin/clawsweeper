## Summary

Keep PR #69346 open. Current main has newer empty-response recovery and OpenRouter `/v1` self-healing, but it still does not implement the PR’s requested operator-facing diagnostic for zero-payload, zero-token empty streams. The existing terminal path still falls back to the generic “couldn't generate a response” text after retries, so the PR remains a plausible focused core UX fix that should be rebased and reviewed against the newer runner behavior rather than auto-closed.

## Best Possible Solution

Keep #69346 open. The best path is to rebase it onto current main and integrate the diagnostic after the existing empty-response retry path, preserving current retry/fallback behavior while adding provider/model/baseUrl guidance only for explicit zero-usage empty streams with regression coverage for nonzero usage, missing usage, side effects, error-stop, OpenRouter, and non-OpenRouter providers.
