## Summary

#70418 is a recent, concrete design proposal that is not implemented or superseded on current main. Current OpenClaw has `cacheRetention`, cache-ttl pruning metadata, prompt-cache diagnostics, and heartbeat keep-warm guidance, but no opt-in `cacheWarmer` config/schema/subsystem that performs pure background TTL refreshes without heartbeat/session/channel semantics. The linked #70417/#70602 context is adjacent heartbeat-cache alignment work and explicitly treats #70418 as orthogonal, so it should remain open for maintainer product/API direction.

## Best Possible Solution

Keep the issue open for maintainer design. The best path is either a core-owned, opt-in per-agent warmer that reuses the normal request builder through the active cache boundary, skips recent activity, discards responses, avoids session/channel delivery, and adds schema/docs/tests; or, if maintainers want this outside core, first define a plugin/runtime API that safely exposes those exact capabilities. Current plugin and heartbeat surfaces do not make this already solved.

## Risks / Open Questions

- The proposal needs maintainer product judgment because it adds recurring model spend and provider-specific behavior, even if it may reduce cache-write costs for long idle sessions.
- The open design questions in the issue still need validation, especially minimal Anthropic request shape, max token behavior, idle-skip policy, and how to avoid session/channel side effects.
