## Summary

Keep open. Current `main` still uses a fixed 2-hour stale-marker window for cron maintenance/manual recovery, while timeout policy already defines shorter effective job timeouts. The April 25 startup fix only covers interrupted runs on gateway restart, so this PR remains a distinct, still-relevant bugfix candidate rather than something already implemented or obsolete.

## Evidence

- **Current main still hard-codes a 2-hour stale-marker threshold:** `src/cron/service/jobs.ts` still defines `STUCK_RUN_MS = 2 * 60 * 60 * 1000` and clears `job.state.runningAtMs` only when `nowMs - runningAt > STUCK_RUN_MS`. No timeout-aware threshold is used in maintenance recompute.
