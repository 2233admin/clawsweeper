## Summary

Keep open. Current `main` still clears stale cron `runningAtMs` markers with a fixed 2-hour threshold, so the PR's timeout-aware cleanup is not implemented on `main` and still addresses a real gap for short-timeout jobs.

## Evidence

- **Fixed 2-hour stale-marker threshold still on main:** `normalizeJobTickState` still uses `STUCK_RUN_MS = 2 * 60 * 60 * 1000` and clears `runningAtMs` only when `nowMs - runningAt > STUCK_RUN_MS`. There is no per-job timeout-aware stale cleanup in current `main`.
  - file: [src/cron/service/jobs.ts:38](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/cron/service/jobs.ts#L38)

  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Per-job timeout policy exists separately:** `resolveCronJobTimeoutMs()` already computes explicit and default per-job timeouts, including positive `payload.timeoutSeconds` for `agentTurn` jobs, but `jobs.ts` does not use it for stale-marker recovery.
  - file: [src/cron/service/timeout-policy.ts:16](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/cron/service/timeout-policy.ts#L16)
