## Summary

Keep #70061 open. Current main still has the reported mismatch: Telegram DM topics intentionally produce `agent:main:main:thread:{chatId}:{topicId}` session keys under the default shared-DM route, while Active Memory only classifies `:group:`, `:channel:`, `:direct:`, `:dm:`, or exactly three-part main sessions as eligible direct chats. A threaded main DM key has more than three parts and no direct marker, so `resolveChatType()` returns `undefined`, `isAllowedChatType()` returns false, and the `before_prompt_build` hook exits before `maybeResolveActiveRecall()` can log/start/persist the recall run.

## Best Possible Solution

Fix this in `extensions/active-memory/index.ts` by classifying threaded direct-session suffixes on a main DM base key as `direct`, ideally with a small helper that understands `resolveThreadSessionKeys()` output rather than a broad provider fallback. Add a regression test using `sessionKey: "agent:main:main:thread:12345:99"`, `messageProvider: "telegram"`, `channelId: "telegram"`, and default `allowedChatTypes: ["direct"]`, asserting the recall sub-agent runs. Keep the existing `:direct:` threaded-session coverage because named/per-channel Telegram DM scopes still use that form.
