## Summary

Keep this issue open. Current main partially covers the request with batch embedding, exact-hash cache reuse, cache seeding during full reindex, status entry counts, and max-entry pruning, but it still does not implement fuzzy reuse for minor edits, queryable hit/miss/latency metrics, frequency-based cache warming, gradual model migration, or compressed vector storage. The existing ClawSweeper comment reached the same result, and current main at c7543701009ccdae83d38ee341e072c8a7c4af6e still supports keeping it open.

## Best Possible Solution

Keep this issue open, but narrow the implementation plan. Preserve the shipped batch embedding, exact-hash cache reuse, cache seeding, and pruning behavior; next add queryable cache metrics, then design opt-in fuzzy reuse, frequency-based warming, gradual provider/model migration or dual-write behavior, and vector compression with compatibility tests.
