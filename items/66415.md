## Summary

Keep open. Current `main` still does not add `required: []` on the non-strict OpenAI-compatible tool-schema path, and the current tests still encode that older behavior. The PR addresses a real gap, and the discussion shows the author also handled the review-found regression by restricting traversal to schema-bearing keywords.

## Evidence

- **Current non-strict path still skips empty `required` injection:** `normalizeOpenAIStrictToolParameters()` returns `normalizeToolParameterSchema(schema ?? {})` whenever `strict` is false. There is no non-strict recursive pass that adds `required: []` to object schemas on `main`.
  - file: [src/agents/openai-tool-schema.ts:62](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/agents/openai-tool-schema.ts#L62)
  - command: `nl -ba src/agents/openai-tool-schema.ts | sed -n '1,220p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Current WebSocket tests still expect non-strict schemas without `required: []`:** `convertTools()` regression coverage on `main` still expects `{ type: "object", properties: {} }` for parameter-free tools, leaves top-level `allOf` object entries without `required: []`, and preserves `type: "object"` schemas with properties but no empty `required` array. That matches the old behavior the PR is trying to change.

- **Current contract coverage only proves the strict path:** The runtime contract test asserts `required: []` parity for HTTP Responses and WebSocket only when the tool inventory resolves to `strict: true`; there is no equivalent non-strict contract proving the custom-provider/proxy case is already covered on `main`.

The author replied that this was fixed in follow-up commit `0d5cf97` by limiting traversal to schema-bearing keywords and adding regression coverage, which makes the PR discussion net-positive evidence rather than a reason to close it.
