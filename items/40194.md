## Summary

Current main still does not isolate Feishu group sessions by Feishu account/app when multiple Feishu bots in the same group route to the same agent. The Feishu inbound path passes accountId into routing, but the shared non-direct session-key builder drops accountId, so group peers collide on agent/channel/group/peerId. The issue has concrete reproduction data, no protected/maintainer labels, and no evidence in the provided related context that a closing PR has merged.

## Best Possible Solution

Keep this issue open and implement account-aware Feishu group session isolation. Prefer a Feishu-specific or generic opt-in account-scoped group mode using normalized accountId rather than raw app credentials, preserve legacy session continuity where feasible, document the behavior in Feishu/group-channel docs, and add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct session keys and correct reply account identity.
