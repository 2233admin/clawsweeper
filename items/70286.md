## Summary

Keep open. Current main still has the reported control flow: plugin-owned `inbound_claim` with `{ handled: true }` returns `queuedFinal: false`, and Telegram sends the silent-reply fallback when `queuedFinal` is false and no delivery has happened yet. The linked PR #70288 is relevant and includes useful `openclaw-codex-app-server` verification, but the fix is not present in this checkout or in `v2026.4.24`. Current tests also still assert the `queuedFinal: false` behavior for plugin-owned bindings, so this remains an active bug rather than cleanup-close material.

## Best Possible Solution

Keep #70286 open until the core behavior is fixed and covered. The likely product direction is to make plugin-owned async claim a distinct committed/claimed response state, or otherwise preserve an equivalent signal through `dispatchInboundMessageWithBufferedDispatcher`, so Telegram and other channels skip silent-reply fallback while still allowing future timeout/error feedback for claimed-but-silent plugins. PR #70288 is a useful starting point, but it should be validated against the current finalization path and updated tests before this issue closes.
