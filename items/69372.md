## Summary

Keep #69372 open. The reported Control UI canvas capability problem is real on current main, but the PR is not safely obsolete or already implemented: current UI still allows raw protected canvas/a2ui iframe paths when no scoped host is available, and the proposed reconnect-based refresh has a concrete risk of interrupting active chat/pending gateway work.

## Best Possible Solution

Keep the PR open, but require changes before merge: preserve the fail-closed iframe URL behavior and tests, then replace the timed websocket reconnect with a non-disruptive Control UI refresh path, likely an operator-safe capability refresh API or an explicit `hello.canvasCapabilityExpiresAtMs` plus refresh RPC. The fix should prove that active chat runs and pending requests are not reset during capability refresh.
