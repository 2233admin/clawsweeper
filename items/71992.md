## Summary

Current main still has the Control UI webchat race described in #71992: `handleChatEvent` appends the final/aborted assistant message to `chatMessages` before clearing `chatStream`, and the renderer still appends a live stream item whenever `stream` is non-null. The issue is recent, has an additional v2026.4.24 reproduction in the discussion, and the linked fix PR #72004 is open rather than merged.

## Best Possible Solution

Keep #71992 open until #72004 or an equivalent patch lands on main. The narrow Control UI fix should snapshot any fallback stream text, clear `chatStream`/`chatRunId`/`chatStreamStartedAt` before mutating `chatMessages` in both `final` and `aborted`, and add a regression test that observes the ordering at mutation time. Track separate persisted transcript/user-message duplicate behavior under the broader duplicate transcript work rather than closing this client-render race report prematurely.
