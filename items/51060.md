## Summary

Keep open. Current `main` still computes the Control UI message-footer `ctx` value from prompt-footprint tokens (`input + cacheRead + cacheWrite`) capped at 100%, while the session-level context UI uses `session.totalTokens / contextTokens`. The mismatch described in #51060 is still present in code and in the current tests.

## Evidence

- **Message footer still uses prompt footprint, not current session context:** `extractGroupMeta()` computes `promptTokens = input + cacheRead + cacheWrite` and then `contextPercent = min(round(promptTokens / contextWindow * 100), 100)`. It does not read `session.totalTokens` or any post-compaction session value.
  - file: [ui/src/ui/chat/grouped-render.ts:427](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/ui/src/ui/chat/grouped-render.ts#L427)
  - command: `nl -ba ui/src/ui/chat/grouped-render.ts | sed -n '388,436p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Current tests lock in the old footer behavior:** The UI test explicitly says it `includes cache tokens when rendering assistant context usage` and expects `44% ctx`; a second test expects `10% ctx` from `input/contextWindow`. That matches the issue's complaint that the footer is based on token footprint rather than current compacted context.
