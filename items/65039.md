## Summary

Keep this issue open. Current main still matches the reported DAVE-disabled Discord voice pre-Ready abort path: `/vc join` passes `voice.daveEncryption` through to `@discordjs/voice`, waits for `VoiceConnectionStatus.Ready` with a fixed 15s timeout, destroys the connection on failure, and only records a session after Ready. The existing DAVE receive recovery is post-Ready, so it does not cover a WebSocket close after Hello before UDP handshaking.

## Best Possible Solution

Keep this issue open until maintainers decide the intended non-DAVE Discord voice behavior. The likely implementation belongs in `extensions/discord/src/voice/manager.ts`: add explicit pre-Ready close/abort diagnostics and regression coverage for `entersState` failure before session activation, then either implement bounded retry/fallback to DAVE-enabled join or reject/deprecate `channels.discord.voice.daveEncryption=false` with clear operator guidance if Discord now requires DAVE. If maintainers consolidate with #39825 or #63098, preserve this issue's Hello-before-Ready trace in the canonical thread before closing it as duplicate.
