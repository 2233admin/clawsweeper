## Summary

Current `main` still contains the Feishu `@_all` bug this PR fixes. `checkBotMentioned()` returns `true` for any message whose raw content contains `"@_all"`, and the group `requireMention` gate trusts that result, so this remains a real behavior bug rather than an obsolete PR.

## Evidence

- **Current main still treats `@_all` as a bot mention:** `checkBotMentioned()` returns `true` immediately when `event.message.content` contains `"@_all"`, before inspecting the structured mentions array.
  - file: [extensions/feishu/src/bot-content.ts:236](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/extensions/feishu/src/bot-content.ts#L236)
  - command: `nl -ba extensions/feishu/src/bot-content.ts | sed -n '220,290p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **That result directly bypasses `requireMention` in groups:** Group message handling resolves `requireMention` and then drops only when `!ctx.mentionedBot`. Because `parseFeishuMessageEvent()` sets `mentionedBot` from `checkBotMentioned()`, the current `@_all` shortcut still causes mention-required group messages to be accepted.
