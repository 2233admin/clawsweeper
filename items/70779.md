## Summary

Keep open. Current main still matches the reported regression shape: custom OpenAI-compatible endpoints are normalized as not streaming-usage-compatible by default, but the OpenAI completions request builder still sends `stream_options.include_usage` unconditionally, and the stream pipeline only emits incremental user-visible text when upstream sends `choice.delta.content` chunks. The provided issue has concrete recent cross-surface logs and no canonical duplicate or shipped fix for the no-incremental Gemini-compatible streaming behavior.

## Best Possible Solution

Keep #70779 open. The likely maintainer path is to make the OpenAI-compatible completions request builder honor `supportsUsageInStreaming` when deciding whether to send `stream_options.include_usage`, or add an explicit per-model/provider escape hatch that actually removes the field for incompatible custom backends. Add focused regression coverage for a generic custom Gemini-style OpenAI-compatible stream showing incremental `text_delta` delivery when usage streaming is disabled, while preserving existing tests for known local/native providers that should still request usage.
