## Summary

Keep open. Current `main` does not implement a generic post-restart inbound catch-up path for live channel providers such as Discord. Discord startup wires live listeners and starts the gateway lifecycle, but there is no monitor-path history scan or replay after reconnect. Telegram is a partial exception: its polling path persists update offsets and preserves pending updates across restart windows, so the issue's cross-provider framing is broader than the code reality.

## Evidence

- **Discord startup registers listeners, then starts live lifecycle:** The Discord monitor allocates in-memory `guildHistories`, creates the message handler, registers monitor listeners, logs `client-start`, and immediately enters `runDiscordGatewayLifecycle(...)`. There is no startup step here that reads channel history or replays missed inbound messages.

## Risks / Open Questions

- The issue body overstates scope: Telegram long polling already has restart-safe inbound durability, so closing as 'cannot reproduce' would incorrectly hide the remaining Discord-style gap.
- Any future fix needs provider-specific replay semantics to avoid duplicate delivery, stale backlog reprocessing, or cross-channel behavior drift.
