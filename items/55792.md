## Summary

Current `main` does not implement a restart-time inbound backlog catch-up for Discord-style live gateway channels, so the issue remains actionable. The report’s proposed scope is broader than reality, though: Telegram already persists polling offsets and Matrix already persists sync cursors across restarts, so this should stay open as a narrower channel/provider reliability request rather than be closed.

## Evidence

- **Discord READY handler has no backlog fetch:** `createDiscordStatusReadyListener` handles READY by refreshing presence only; there is no startup history scan or missed-message replay hook in that path.
  - file: [extensions/discord/src/monitor/provider.startup.ts:86](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/monitor/provider.startup.ts#L86)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Discord monitor wires live listeners only:** Discord startup registers `DiscordMessageListener`, reaction listeners, and thread-update listeners against the live client event stream. That covers new gateway events, not offline backlog recovery after a process restart.
  - file: [extensions/discord/src/monitor/provider.startup.ts:257](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/monitor/provider.startup.ts#L257)

- **Discord history reads exist, but outside startup monitoring:** `readMessagesDiscord()` can fetch channel history via `Routes.channelMessages`, but repo search only found it used in action/runtime and tests, not in the Discord monitor startup path.
  - file: [extensions/discord/src/send.messages.ts:13](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/discord/src/send.messages.ts#L13)

- **Telegram already persists inbound cursor across restarts:** Telegram monitor loads a persisted update offset at startup and advances it after each processed update, so polling resumes from the saved update id instead of requiring a history scrape.
  - file: [extensions/telegram/src/monitor.ts:193](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/monitor.ts#L193)

## Risks / Open Questions

- This was a read-only source review; I did not run a live Discord restart reproduction with real credentials.
- I did not exhaustively inspect every bundled channel plugin, so the strongest conclusion is that Discord still lacks this behavior while Telegram and Matrix already have durable restart cursors.
