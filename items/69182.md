## Summary

Keep #69182 open. Current main enforces a hard timeout for active-memory, which addresses the narrower #70481-style overrun, but active-memory is still intentionally a blocking pre-prompt step and can still delay the main reply until its configured timeout. The issue’s core request for non-blocking behavior, lower/auto-tuned latency, or avoiding delay before first visible model output remains unresolved.

## Best Possible Solution

Keep this open as a focused active-memory product/performance item. Maintainers should decide whether to add a non-blocking/async recall mode, reduce or auto-tune default latency budgets, or otherwise stream the first visible reply before recall finishes; treat #70481 as the shipped hard-timeout fix, not as the canonical tracker for this remaining behavior.
