## Summary

Keep #69737 open. Current main has added some delivery-side redaction/formatting, but the reported class is not fully fixed: unclassified assistant errors can still be derived from `lastAssistant.errorMessage`, and multi-text assistant content is still joined by the shared visible-text extractors rather than selecting a single item.

## Best Possible Solution

Keep the issue open. The safest product path is a narrow delivery-layer helper that never posts raw `stopReason=error` text and returns a generic/sanitized user-facing error, plus a separate explicit policy for multi-text assistant messages in channel delivery, likely pick last non-empty final-answer text unless maintainers choose a config flag. Add focused regression tests around `handleAgentEnd` and `handleMessageEnd`/block replies for Slack-class delivery; consider splitting the multi-text policy into its own issue if ownership review should be separate.
