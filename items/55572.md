## Summary

Keep open: current `main` still rejects `api: "google-vertex"` in the config model-api enum, and the runtime transport tables also do not recognize `google-vertex` as a supported API id. The repo already has partial `google-vertex` provider-key and auth plumbing, so the report is coherent, but the requested capability is not implemented on `main` and the issue's proposed 3-line fix appears incomplete.

## Risks / Open Questions

- The issue author's proposed fix is likely too small: even if `google-vertex` were added to the schema enums, transport selection and inline-provider sanitization would still not recognize that API id.
- There is partial `google-vertex` provider-key support on `main`, so maintainers should clarify whether the intended solution is to document/configure that path or to add a first-class `google-vertex` API adapter end-to-end.
