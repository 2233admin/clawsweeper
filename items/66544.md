## Summary

Keep open. The PR targets a real symptom, but the proposed fix is too broad on current `main`: `buildGatewaySessionRow` would suppress any `origin.label === "heartbeat"`, and direct-session labels can legitimately come from a real sender/contact name with that exact value. Current code also does not carry a synthetic-heartbeat provenance flag into the gateway row, so the patch cannot distinguish internal heartbeat fallback IDs from real conversation labels.

## Evidence

- **Gateway row falls back directly to `origin.label`:** `buildGatewaySessionRow` computes `displayName` as `entry.displayName -> buildGroupDisplayName(...) -> entry.label -> origin.label`. The PR changes this exact fallback site, so a blanket string filter changes all sessions that rely on `origin.label`, not just synthetic heartbeat ones.
  - file: [src/gateway/session-utils.ts:1265](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/gateway/session-utils.ts#L1265)

  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Direct-chat labels legitimately come from sender names:** For direct chats, `resolveConversationLabel` prefers `SenderName` and otherwise `From`. A real contact/chat named `heartbeat` therefore produces `origin.label === "heartbeat"` without being synthetic.
  - file: [src/channels/conversation-label.ts:39](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/channels/conversation-label.ts#L39)

## Risks / Open Questions

- Merging the PR as written would hide legitimate direct-session names equal to `heartbeat`, not just internal heartbeat fallback senders.
- There is no targeted `buildGatewaySessionRow` regression test covering this edge case, so the proposed string filter is not defended by current test coverage.
