## Summary

Keep PR #69363 open. Current main still captures CLI session IDs with raw first-wins/last-wins `pickCliSessionId` logic and has no guard for transient Claude `hook_started`/`hook_response` SessionStart records. Main does include the narrower #61390 symptom fix for `No conversation found`, but that only classifies/recovers from stale IDs; it does not prevent storing the transient hook ID that this PR targets.

## Best Possible Solution

Keep this PR open or reimplement its approach on main: add a shared CLI session-id selection helper that treats Claude `system` records with `subtype: hook_started` or `hook_response` as fallback-only, apply it consistently in `parseCliJson`, `parseCliJsonl`, and `createCliJsonlStreamingParser`, and keep/add regression tests for hook-before-init and hook-after-init ordering. The existing #61390 recovery should stay as a fallback, not replace the root-cause parser fix.
