## Summary

Keep #69363 open. Current main still has the session-id selection behavior the PR is trying to fix: `parseCliJsonl` keeps the first `session_id`, while the JSON and streaming parsers allow later records to replace it, and none of those paths filter Claude `hook_started`/`hook_response` records. The related #61390 failover pattern is present on main, but that only recovers after a bad resume id is used; it does not prevent persisting a transient hook id in the first place.

## Best Possible Solution

Keep the PR open and land a rebased minimal parser fix: introduce a shared helper that treats Claude hook records as fallback-only session ids, use it in `parseCliJson`, `parseCliJsonl`, and `createCliJsonlStreamingParser`, and keep the focused regression tests for hook-before-init and hook-after-real-id ordering. This complements, rather than replaces, the shipped #61390 recovery path.
