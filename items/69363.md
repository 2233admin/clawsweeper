## Summary

Keep PR #69363 open. Current main still lacks the parser-level guard that ignores transient Claude CLI hook session IDs, while the shipped #61390/#65028 work only classifies `No conversation found` as `session_expired` after a bad ID has already been persisted. The PR remains a focused, plausible root-cause fix with reviewer support and targeted regression coverage in its diff.

## Best Possible Solution

Keep the PR open and land this fix or an equivalent parser-level change in `src/agents/cli-output.ts`: centralize session-id selection so non-hook records always win, hook records are only a fallback when no real id exists, and regression tests cover `parseCliJson`, `parseCliJsonl`, and `createCliJsonlStreamingParser`. The existing `No conversation found` recovery should remain as a fallback, not the primary fix.

## Evidence

- **Current parser still has unguarded session-id selection:** On current main, `parseCliJson` and the streaming parser use `pickCliSessionId(...) ?? sessionId`, while `parseCliJsonl` uses first-wins `if (!sessionId) sessionId = pickCliSessionId(...)`. There is no `nextCliSessionId` or hook-record filter, so a `hook_started`/`hook_response` record with `session_id` can still become the effective CLI session id depending on ordering.
