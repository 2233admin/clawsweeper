## Summary

Keep open: current `main` still hard-fails Discord startup when `oauth2/applications/@me` returns a non-OK HTTP response such as `429`. The token-derived fallback exists, but `fetchDiscordApplicationId()` only uses it for thrown transport errors, not for non-OK responses, so the reported failure mode remains plausible on current code.

## Evidence

- **Non-OK application lookup still returns undefined:** `fetchDiscordApplicationId()` returns the JSON `id` only when `res.ok` is true, then immediately returns `undefined` for all non-OK HTTP responses. The fallback to `parseApplicationIdFromToken(token)` only runs in the `catch` path.

  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Startup aborts on missing application id:** Discord provider startup calls `fetchDiscordApplicationId(...)` and throws `Failed to resolve Discord application id` when it gets a falsy result, matching the issue report's hard-fail behavior.
