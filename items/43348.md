## Summary

Keep open. Current main partially overlaps the proposal with `sessions_send`, `/acp cancel`, and `sessions.abort`, but it does not implement the proposed `acp_send` tool or `sessions_cancel` agent tool. The PR may still contain useful ACP lifecycle work, but it needs rebase/product alignment and security fixes before it can be merged.

## Best Possible Solution

Keep this PR open only as a candidate for rework: rebase it onto current main, decide whether an ACP-specific send tool is still needed beyond current `sessions_send` ACP handling, and if an agent cancel tool is desired, implement it as a guarded surface over the existing `sessions.abort`/ACP cancellation path with visibility/ownership checks, ACP target validation, docs/catalog updates, and focused tests.

## Risks / Open Questions

- Security: the PR's `sessions_cancel` path would allow cancellation by raw session key unless it reuses session visibility, ownership, and A2A policy guards before cancelling.
- Functional: `acp_send` needs an ACP-only target check or a safe fallback; otherwise non-ACP target output can be lost because normal delivery is intentionally disabled.
- Product/API: adding `sessions.cancel` beside current `sessions.abort` and `/acp cancel` would fragment the session-control API unless the naming and semantics are reconciled.
- Validation: the PR needs focused tests for agent tool registration, cancel authorization/visibility, ACP-only send targeting, and the existing `sessions.abort`/ACP cancellation interaction.
