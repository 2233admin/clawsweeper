## Summary

Current main still has the reported Google/Gemini CLI OAuth behavior: an environment-provided client ID can be used without a matching client secret, the token exchange only sends client_secret when present, and the browser callback remains fixed to localhost:8085. The existing manual fallback helps with listener failures but does not provide a configurable or dynamic redirect port.

## Best Possible Solution

Keep this open for the bundled Google plugin owner. Make the Gemini CLI OAuth client-secret contract explicit in code and docs, either by rejecting or warning when an env client ID is provided without a matching secret, or by documenting and testing public-client mode as intentional. Separately, add supported redirect URI/port configuration if compatible with Google OAuth registration, or document and regression-test the existing manual paste fallback as the intended recovery for localhost:8085 conflicts.

## Risks / Open Questions

- A dynamic port may be constrained by Google OAuth registered redirect URIs, so the practical fix may need configurable registered redirect URI support or explicit documentation and tests for the manual paste fallback.
