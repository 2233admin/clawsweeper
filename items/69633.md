## Summary

Keep #69633 open. Current main still has no SPA-side WebView2/native bridge, ready handshake, or handlers for the proposed recording, voice, and draft-text messages. The PR is a recent draft with pending tests and linked Windows-host work; because Windows companion apps are within the stated project roadmap and this touches the bundled Control UI, this needs maintainer product/implementation judgment rather than conservative cleanup.

## Best Possible Solution

Keep the PR open for maintainer judgment. The likely path is to either evolve it into a tested Control UI native-bridge integration wired into app lifecycle/state, with the message contract aligned to the Windows host PR, or have maintainers explicitly redirect the broader WebView2 bridge design to a different Windows-companion-app/API discussion.
