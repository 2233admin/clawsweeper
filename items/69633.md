## Summary

Keep #69633 open. Current main does not implement SPA-side WebView2 bridge support, and this PR is a recent draft tied to active Windows companion-app bridge work in openclaw/openclaw-windows-node#191 and #192. It should not be closed as implemented, duplicate, ClawHub, or not-actionable; the useful path is maintainer review plus changes because the current PR patch only adds an unimported module and has pending tests.

## Best Possible Solution

Keep the PR open as a draft/change-request candidate. The best path is to wire the bridge through the existing UI lifecycle or a small controller, add cleanup, validate WebView2 message payloads, add focused UI tests for no-op browser behavior and ready-handshake/message handling, and coordinate the message contract with openclaw-windows-node#191/#192. If maintainers decide WebView2 should not be hard-coded in the SPA, the alternative should be a small generic web/native bridge seam first, not closing this as ClawHub because current plugin surfaces do not mutate SPA-local chat UI state.
