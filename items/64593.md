## Summary

Keep open. The issue is recent, concrete, and still matches current main: managed Chromium is launched and stopped as a single tracked ChildProcess, while the reported normal-exit/container PID-1 descendant reaping problem is not covered by the existing startup-retry wait helper, stop path, tests, or Docker/browser docs. The report is also distinct from closed startup-failure issue #8611.

## Best Possible Solution

Keep this issue open and fix it in the bundled browser plugin/runtime lifecycle. The maintainer path should make managed Chromium container-safe across normal exits, stop/restart, and startup retry cleanup, likely through a supervised helper/init/subreaper or process-group/tree strategy that is safe when the gateway is PID 1. Add focused unit coverage plus a Docker/browser smoke that repeatedly exits managed Chromium and verifies no defunct chromium/chrome_crashpad processes remain, then document any supported container requirement or workaround.

## Risks / Open Questions

- Closing now would leave a plausible production PID-exhaustion bug untracked for container deployments where the gateway runs as PID 1 and adopts Chromium/crashpad descendants.
- A fix needs careful process-group or helper design so browser cleanup cannot accidentally signal the gateway process group.
- There is still no current Docker/browser regression proof that repeated managed Chromium normal exits leave no chromium/chrome_crashpad defunct processes.
