## Summary

Current main still does not implement automatic forwarding of original Node `agent.request` image attachments when `deliver: true`. Attachments are converted into agent input (`images` / `media://inbound` markers), but there is no protocol flag, command option, or delivery fallback that carries those original attachments into outbound IM delivery when the agent returns only text or no media.

## Best Possible Solution

Keep this issue open. The best implementation is an additive Gateway protocol/node-event change that preserves sanitized delivery-ready attachment references only when explicitly requested, carries that policy through `AgentCommandOpts`, and injects media into outbound payloads only under narrow conditions such as `deliver: true`, a resolved external target, channel media support, and no agent-supplied media. Add docs and regression tests for text-only replies, no-reply runs, agent-media override, unsupported channels, and attachment size/type enforcement.

## Risks / Open Questions

- Automatic sender-controlled media fanout needs explicit product and security design: opt-in semantics, target validation, size/type limits, outbound media access scoping, unsupported-channel behavior, and precedence when the agent also returns media.
- A plugin-only workaround is not cleanly sufficient today because the Node/Gateway path consumes attachments into agent input before outbound delivery sees delivery-ready media references.
