## Summary

Keep this issue open. Current main still has the reported BOM failure path: the shared frontmatter parser checks for `---` before stripping a leading UTF-8 BOM, skills delegate directly to that parser, and the local skill loader drops the skill when parsed frontmatter lacks `description`. The prior comment's distinction from closed #45432 remains correct: that related report was about write/edit tool BOM behavior, not skill discovery.

## Best Possible Solution

Fix the shared markdown frontmatter parser to tolerate one leading UTF-8 BOM before the opening `---`, then add a focused `parseFrontmatterBlock` regression test plus a skill loading or `skills list` path test proving a BOM-prefixed `SKILL.md` remains discoverable. Keep #45432 treated as a separate, already-closed write/edit-tool BOM report.

## Evidence

## Risks / Open Questions

- `parseFrontmatterBlock` is shared by skills and hooks, so the fix should strip only a single leading UTF-8 BOM before the opening delimiter check without broadening parsing semantics unexpectedly.
- Current tests do not cover BOM-prefixed markdown frontmatter or BOM-prefixed `SKILL.md` discovery.
