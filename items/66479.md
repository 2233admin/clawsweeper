## Summary

Keep open. Current `main` still appears to have the reported bug: the shared frontmatter parser requires `SKILL.md` to start with `---` at byte 0, so a leading UTF-8 BOM prevents frontmatter extraction, and the skill loader then drops the skill because `description` is missing.

## Evidence

- **Shared frontmatter parser does not strip BOM:** `extractFrontmatterBlock` normalizes newlines, then immediately checks `normalized.startsWith("---")`. A BOM-prefixed file therefore returns `undefined` before any YAML parsing.
  - file: [src/markdown/frontmatter.ts:183](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/markdown/frontmatter.ts#L183)
  - command: `nl -ba src/markdown/frontmatter.ts | sed -n '183,192p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Skills use the shared parser:** Skill frontmatter parsing is a thin wrapper around `parseFrontmatterBlock`, so the BOM behavior applies directly to skill discovery.
  - file: [src/agents/skills/frontmatter.ts:24](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/agents/skills/frontmatter.ts#L24)
  - command: `nl -ba src/agents/skills/frontmatter.ts | sed -n '24,25p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Missing parsed description causes discovery drop:** `loadSingleSkillDirectory` parses frontmatter, then returns `null` when `name` or `description` is missing. If BOM prevents frontmatter extraction, `description` is absent and the skill is skipped.
  - file: [src/agents/skills/local-loader.ts:60](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/agents/skills/local-loader.ts#L60)
  - command: `nl -ba src/agents/skills/local-loader.ts | sed -n '60,72p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Minimal runtime reproduction of current parser logic:** A BOM-prefixed frontmatter string evaluated with the same `startsWith("---")` check returns `false` on current checkout (`{"starts":false,"endIndex":33}`), matching the issue's root-cause claim.
  - command: `node -e "const content='\uFEFF---\nname: demo\ndescription: demo\n---\n'; const normalized=content.replace(/\r\n/g,'\n').replace(/\r/g,'\n'); const starts=normalized.startsWith('---'); const endIndex=normalized.indexOf('\n---',3); console.log(JSON.stringify({starts,endIndex,block: starts && endIndex !== -1 ? normalized.slice(4,endIndex) : undefined}))"`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **No current regression test for BOM-prefixed frontmatter:** `src/markdown/frontmatter.test.ts` covers several frontmatter shapes and a missing-frontmatter case, but no BOM-prefixed input. The skills workspace edge-case test also does not cover BOM handling.
  - file: [src/markdown/frontmatter.test.ts:5](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/markdown/frontmatter.test.ts#L5)
  - command: `nl -ba src/markdown/frontmatter.test.ts | sed -n '1,104p'`
