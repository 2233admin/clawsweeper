## Summary

Keep #39811 open. Current main has partial runtime mitigations for the silent-fallback symptom, but it still does not implement catalog-aware `openclaw config validate` for arbitrary configured model names. Related #39818 and #45294 are closed and cover overlapping runtime/reporting symptoms, but they do not supersede the remaining config-validation request.

## Best Possible Solution

Keep the issue open. The best path is to add catalog-aware validation or doctor/startup warnings for configured `agents.defaults.model.primary`, fallback refs, and provider-owned model refs where the provider catalog can be trusted, with clear warn-vs-error semantics for custom and dynamic providers. Separately, make fallback transitions visible through an appropriate channel/status path even when verbose mode is off, or split that remaining notification work into a narrower canonical follow-up.

## Risks / Open Questions

- The issue combines config-time model validation and user-visible fallback notification; maintainers may want to split the remaining work, but #39811 is still a valid tracker for the unresolved config-validation half.
