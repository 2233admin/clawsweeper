## Summary

Keep #71389 open. Current main still loads sharp through the bundled media-understanding-core image ops without a CPU preflight or a disabled/pass-through backend, and the core resize/conversion callers still propagate those calls. The related #47419 non-AVX2 Docker publishing cleanup is a different shipped-container regression and does not supersede this runtime sharp-load crash report.

## Best Possible Solution

Implement a real safe image backend path: add a documented `OPENCLAW_IMAGE_BACKEND=disabled|none` mode, preflight unsupported CPU/runtime conditions before importing sharp where possible, and make JPEG resize, HEIC conversion, and PNG resize degrade with explicit warnings or caller-level pass-through semantics. Add focused tests for backend disablement and sharp-load failure behavior; if SIGILL cannot be caught in-process, consider isolating sharp probing/work in a child process or documenting the unsupported runtime clearly.
