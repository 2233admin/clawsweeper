## Summary

Keep open. Current `main` still routes Discord exec approvals through a gateway bootstrap path that does not hydrate or pass a local TLS fingerprint, so the PR's reported self-signed `wss://` local-gateway failure is not shown as implemented on `main` and remains plausible.

## Evidence

- **Discord approvals still use shared approval-over-gateway helper:** The Discord monitor button context resolves approvals by calling `resolveApprovalOverGateway(...)`, so this flow still depends on the operator-approvals gateway client path rather than an inline Discord-specific TLS fix.
  - file: [extensions/discord/src/monitor/exec-approvals.ts:134](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/extensions/discord/src/monitor/exec-approvals.ts#L134)
  - command: `nl -ba extensions/discord/src/monitor/exec-approvals.ts | sed -n '119,145p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Shared approval resolver uses operator approvals gateway client:** `resolveApprovalOverGateway(...)` delegates to `withOperatorApprovalsGatewayClient(...)`, so any local-TLS fingerprint support has to exist in that client/bootstrap path on current `main`.
  - file: [src/infra/approval-gateway-resolver.ts:19](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/infra/approval-gateway-resolver.ts#L19)
