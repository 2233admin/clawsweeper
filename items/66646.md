## Summary

Keep open. Current main includes the classifier fix from #68700 and a later lock-scope narrowing, but it still does not implement the central requested fail-fast/session-busy behavior: session write-lock acquisition remains inside each model fallback attempt, and `runWithModelFallback()` intentionally continues across remaining candidates for unrecognized errors such as an unclassified `SessionWriteLockTimeoutError`.

## Best Possible Solution

Keep this issue open and implement a targeted non-model-failover path for `SessionWriteLockTimeoutError` or nested `OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` errors at the model fallback wrapper or agent attempt boundary. Add a regression with multiple model candidates where the run callback throws a session write-lock timeout, assert the callback is invoked exactly once, and surface a clear session-busy/infrastructure error. Queue/retry/backoff can be handled at the session lane or caller boundary; #69030 should remain a separate raw-inference follow-up.

## Risks / Open Questions

- Closing this based on #68700 alone would likely miss the remaining latency/cascade path: the lock error is no longer labeled as a provider timeout, but each remaining model candidate can still re-enter the same locked session attempt.
- A targeted fix must preserve the existing intentional fallback behavior for generic unknown provider errors, as covered by `src/agents/model-fallback.test.ts:361`.
- The related open raw-inference request #69030 may remove one trigger path for `infer model run`, but it is not a replacement for the core fallback/session-lock handling requested here.
