## Summary

Current main still treats MiniMax XML tool invocations as malformed assistant text to strip, not as tool calls to parse and execute. I found no execution-path parser for `<invoke ...>` / `</minimax:tool_call>` on `7ac35b4f69009490180a35bf404638d4280a4865`, so the report remains actionable rather than closable.

## Evidence

- **Current sanitizer still drops MiniMax XML:** `stripMinimaxToolCallXml()` explicitly removes `<invoke ...>...</invoke>` blocks and `</minimax:tool_call>` tags, with no conversion to structured tool calls in this code path.
  - file: [src/shared/text/assistant-visible-text.ts:310](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/shared/text/assistant-visible-text.ts#L310)
- **Unit tests codify stripping behavior:** `extractAssistantText` tests expect MiniMax XML-only assistant content to become an empty string and mixed content to retain only surrounding prose.
  - file: [src/agents/pi-embedded-utils.test.ts:35](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/pi-embedded-utils.test.ts#L35)
- **Session sanitization also strips MiniMax XML:** Session-tool text sanitization tests expect MiniMax XML payloads to be removed from recalled assistant text rather than surfaced as executed tool activity.
  - file: [src/agents/tools/sessions.test.ts:189](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/tools/sessions.test.ts#L189)
- **No parser found in current source tree:** A repo search for MiniMax/XML tool-call handling only found sanitizer/docs/test references, not an execution-path parser or dispatcher for `<invoke ...>` payloads.
  - command: `rg -n "<invoke|minimax:tool_call|\[TOOL_CALL\]|parse.*tool.*xml|xml tool" src extensions -g '!dist'`
