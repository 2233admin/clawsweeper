## Summary

Current `main` does not provide a Docker-native headless node workflow. `openclaw node install` still assumes a supervised OS service and fails on Linux when `systemctl --user` is unavailable, while `openclaw node run` still relies on the existing manual device-pairing flow and there is no dedicated `node pair` bootstrap path. The existing Docker docs cover a containerized gateway/CLI setup, not a containerized node host. The issue is therefore still actionable and should remain open.

## Evidence

- **`node install` still hard-fails when the service check cannot use systemd:** `runNodeDaemonInstall()` calls `service.isLoaded()` up front and surfaces `Node service check failed: ...` on error, which matches the report rather than bypassing service-manager checks for containers.
  - file: [src/cli/node-cli/daemon.ts:108](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/cli/node-cli/daemon.ts#L108)

  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Linux service install still requires user-systemd:** The Linux systemd path still throws `systemctl --user unavailable: unknown error` or `systemctl not available; systemd user services are required on Linux.` There is no Docker/container escape hatch here.
  - file: [src/daemon/systemd.ts:420](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/daemon/systemd.ts#L420)

- **Docs still frame `node install` as a background service and pairing as manual device approval:** The node CLI docs describe `openclaw node install` as installing a user service, say to use `openclaw node run` for foreground mode, and require pairing through `openclaw devices list` / `openclaw devices approve <requestId>`. There is no documented `node pair` or container-specific bootstrap flow.
  - file: [docs/cli/node.md:84](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/docs/cli/node.md#L84)

- **Docker docs cover the gateway container, not a node container:** `docs/install/docker.md` documents building and onboarding a Dockerized gateway plus `openclaw-cli` helper commands. It does not document running a headless node host inside Docker or a node-specific pairing bootstrap path.
  - file: [docs/install/docker.md:28](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/docs/install/docker.md#L28)

- **`node run` does not self-recover through pairing-required closes:** On WebSocket close, the client skips reconnect when auth failure detail is `PAIRING_REQUIRED`. That means a foreground node process in Docker still does not provide the lightweight bootstrap flow requested by the issue.
  - file: [src/gateway/client.ts:333](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/client.ts#L333)

- **Issue discussion identifies a plausible workaround direction, but it is not implemented as productized flow:** The existing comment is directionally useful: Docker Compose can replace the service-management part of `node install`. But current `main` still lacks the separate pairing/bootstrap command or documented node-container workflow that would make that workaround a supported path.

## Risks / Open Questions

- This may be partly a docs/UX gap rather than a large code change: if maintainers only want to support `node run` in containers, the missing piece could be a documented bootstrap/rerun flow instead of new runtime behavior.
- There may already be an out-of-repo community workflow, but nothing in current `main` docs or CLI surfaces it as supported OpenClaw behavior.
