## Summary

Keep #72214 open. It is a fresh non-maintainer report against v2026.4.24 with a concrete macOS/Node/undici reproduction, and current main still has plausible Telegram Bot API startup/probe paths that can sit on failing `api.telegram.org` fetches without a clear channel-level degraded diagnostic or regression proving gateway health/RPC stays responsive. Adjacent post-release Telegram changes remove one `getUpdates` offset-confirmation path and add polling leases, but they do not strongly prove this reported gateway-stall symptom is fixed.

## Best Possible Solution

Keep the issue open and treat it as a Telegram plugin runtime/gateway isolation bug. The best fix is to add a regression where Telegram Bot API startup, polling setup, shutdown, and health probes encounter a never-settling or ETIMEDOUT fetch, while `health`, `gateway status`, and `channels.status` remain responsive and Telegram is marked degraded with a clear diagnostic. The likely code paths are `extensions/telegram/src/polling-session.ts`, `extensions/telegram/src/probe.ts`, `extensions/telegram/src/fetch.ts`, and the gateway health/status handlers; also document a supported proxy/local Bot API workaround if that remains the operator path.
