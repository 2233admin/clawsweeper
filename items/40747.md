## Summary

Keep this PR open. Current main still requires `after_block_id` for `feishu_doc` insert and only supports insertion after an existing block, while the PR is a focused Feishu plugin implementation candidate for inserting at the beginning or at a root index.

## Best Possible Solution

Keep this PR open, or replace it with an equivalent maintainer patch in `extensions/feishu`: make `after_block_id` optional only for `action: "insert"`, add `index: Type.Optional(Type.Integer({ minimum: 0, ... }))`, default missing `after_block_id` to root insertion at `index ?? 0`, preserve the current after-block insertion branch, pass the index through both normal and batched insertion paths, and add focused Feishu doc tool tests plus skill docs for the new mode.

## Risks / Open Questions

- The PR should not merge as-is until `index` is constrained to a non-negative integer; otherwise floats or negative values can pass tool validation and fail later at the Feishu API boundary.
- The PR lacks targeted regression tests for index-only root insertion and for preserving the existing `after_block_id` path.
- The Feishu document skill docs would remain stale unless the new insert-by-index mode is documented alongside append and upload-image positioning.
