## Summary

Keep #72370 open. Current main intentionally prevents workspace hooks from overriding managed/bundled/plugin hook names, which explains the reported startup warning, but the reported diagnostic gap remains: ignored workspace hooks are dropped from resolved status, and a selected hook with no metadata events is still considered loadable by the CLI/status model even though the runtime loader skips registering it.

## Best Possible Solution

Keep the issue open and fix the hook diagnostics in core: preserve/report collision information in hooks list/check/status or doctor, mark enabled hooks with no metadata events as not loadable/actionable, and document the supported remediation path such as renaming workspace hooks or installing/linking them as managed hook packs when they intentionally replace managed behavior.

## Evidence
