## Summary

Keep #72370 open. Current main intentionally blocks workspace hooks from overriding managed/bundled/plugin hooks, and docs/tests cover that policy, so the reported rejection is real current behavior rather than already fixed. The issue also identifies an actionable core diagnostic gap: a managed hook with no `metadata.openclaw.events` can win the name collision, appear loadable in hook status because status does not treat empty events as blocking, and then be skipped by the loader with only a startup warning. That remaining behavior belongs in OpenClaw hook resolution/status/install validation, not in ClawSweeper cleanup.

## Best Possible Solution

Keep the issue open for a focused hook-system fix: preserve the documented no-override policy, but make collisions and no-event managed hooks visible and actionable in `openclaw hooks list/check/info` or install validation, and consider warning/erroring when an enabled managed hook with no events masks a workspace hook with the same name.
