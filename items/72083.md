## Summary

Keep #72083 open. Current main still has the reported Docker/source-mount mismatch: the runtime image ships both `dist/` and `extensions/`, production resolution prefers the built bundled plugin tree, and tests intentionally keep bundled plugins ahead of workspace duplicates. There is a global `OPENCLAW_BUNDLED_PLUGINS_DIR` escape hatch and explicit config paths can override bundled plugins, but the maintainer bind-mount workflow is not clearly documented and there is no plugin-by-plugin source-preference or ambiguity warning.

## Best Possible Solution

Keep the issue open for a core loader/docs fix. The best maintainer path is to document the supported Docker contributor workflow clearly, reconcile the precedence docs, and consider a narrow opt-in such as a documented `OPENCLAW_BUNDLED_PLUGINS_DIR=/app/extensions` workflow, a plugin-scoped source preference, or a one-time startup warning when `/app/extensions/<id>` is present but shadowed by `/app/dist/extensions/<id>`. The default packaged-image bundled precedence should remain conservative unless maintainers explicitly choose otherwise.

## Risks / Open Questions

- Changing default production precedence could affect bundled-plugin trust and runtime-dependency staging, so an opt-in override or warning is safer than silently preferring source in Docker images.
- Docs are currently inconsistent about scan order versus duplicate-id precedence, so a docs-only fix should reconcile those pages rather than just adding a workaround note.
