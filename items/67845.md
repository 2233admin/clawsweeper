## Summary

Keep open. Current `main` still lacks language-aware Telegram command menu registration for built-in commands: it registers one default command list, exposes no locale-specific config surface, and explicitly prevents custom commands from overriding built-ins. The request is coherent, and Telegram's upstream API does support per-language command menus.

## Evidence

- **Telegram menu sync uses a single default registration call:** `syncTelegramMenuCommands` deletes existing commands and then calls `bot.api.setMyCommands(retryCommands)` with only the command array. It does not pass `language_code`, `scope`, or any locale selection data.
  - file: [extensions/telegram/src/bot-native-command-menu.ts:255](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/telegram/src/bot-native-command-menu.ts#L255)
  - command: `nl -ba extensions/telegram/src/bot-native-command-menu.ts | sed -n '235,320p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Telegram config exposes only generic native-command toggles and flat custom commands:** `TelegramAccountConfig` has `commands?: ProviderCommandsConfig` and `customCommands?: TelegramCustomCommand[]`, but no per-language command map, locale selector, or language override for menu registration.
  - file: [src/config/types.telegram.ts:103](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/config/types.telegram.ts#L103)
  - command: `nl -ba src/config/types.telegram.ts | sed -n '90,125p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Built-in command names remain reserved even when custom commands are configured:** `registerTelegramNativeCommands` builds `reservedCommands` from all native command specs before resolving `customCommands`, so custom menu entries cannot reuse built-in names like `/help` or `/status` to supply localized descriptions.
  - file: [extensions/telegram/src/bot-native-commands.ts:511](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/telegram/src/bot-native-commands.ts#L511)
  - command: `nl -ba extensions/telegram/src/bot-native-commands.ts | sed -n '500,576p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Tests confirm native/custom name collisions are ignored:** The Telegram command-menu test asserts that a custom command named `status` is not registered when it collides with the native `/status` command, and the native English description remains in the menu.
  - file: [extensions/telegram/src/bot.command-menu.test.ts:122](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/telegram/src/bot.command-menu.test.ts#L122)

- **User-facing docs describe custom commands as non-overriding menu entries only:** The Telegram docs say custom commands cannot override native commands and are menu entries only, which matches the current implementation and means they are not a real workaround for localizing built-in Telegram menu items.
  - file: [docs/channels/telegram.md:349](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/docs/channels/telegram.md#L349)

- **Upstream Telegram API supports localized command menus:** The pinned `@grammyjs/types` API definition for `setMyCommands` includes optional `scope` and `language_code` arguments, so the requested behavior is supported by the Telegram API surface even though OpenClaw does not wire it up.
  - command: `curl -Lsf https://raw.githubusercontent.com/grammyjs/types/v3.26.0/methods.ts | rg -n "setMyCommands|language_code|scope" -n -C 3`
- **Issue discussion is concrete and internally consistent:** The reporter explicitly suggested keeping slash tokens stable while localizing the visible Telegram menu descriptions, which is a narrow and technically coherent request rather than an incoherent translation ask.

## Risks / Open Questions

- Closing this as implemented or cannot-reproduce would be incorrect: current main still exposes only one default Telegram command menu and cannot localize built-in command descriptions.
- There is a partial workaround for adding brand-new custom menu entries in any language, but it does not solve the reported gap for existing built-in Telegram commands.
