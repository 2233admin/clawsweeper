## Summary

Keep PR #69904 open. Current main does not implement the stale transient async notice filtering: system events have no stale policy, no session-advance discard helper, and formatted drains still consume all queued events. The PR is not obsolete, but the latest PR discussion identifies a real regression risk in the branch's early-return behavior for non-completion exec notices, so the right outcome is revision rather than cleanup closure.

## Best Possible Solution

Keep the PR open and ask for a targeted revision: add stale metadata/discarding for transient success/completion notices after session advance, thread session `updatedAt` into the relevant drains/preflight paths, keep durable blocked/error/timeout notices, and preserve cron plus non-completion exec notices such as `exec.started` and `exec.denied` with regression tests for the mixed-queue cases.
