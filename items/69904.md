## Summary

Keep PR #69904 open. Current main does not contain the proposed stale-event policy or filtering behavior, and the PR is a focused fix for real async notice paths. It also still has an unresolved correctness concern from review: exec-event heartbeats must not discard legitimate non-completion notices such as `Exec started` and `Exec denied`.

## Best Possible Solution

Keep the PR open and ask for a revised implementation that adds stale transient completion filtering in the system-event/session-drain path, marks only truly transient success/completion notices as drop-on-session-advance, and adds regression tests for mixed queues plus `exec.started`/`exec.denied` preservation.
