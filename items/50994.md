## Summary

Keep open. Current `main` still deletes an explicit empty cron `expr` during normalization, so the reported behavior is not implemented on `main`. The PR also needs maintainer review because the review-thread concern about direct service callers remains real: preserving `expr: ""` without an earlier rejection would let `createJob`/`applyJobPatch` carry an enabled cron schedule that computes no `nextRunAtMs` instead of failing fast.

## Evidence

- **Current main still strips empty cron expr:** `normalizeOptionalString()` trims empty/whitespace strings to `undefined`, and `coerceSchedule()` then deletes `expr` when `normalizedExpr` is falsy. That means `{ kind: "cron", expr: "" }` is still turned into a schedule with no `expr` on current `main`.
  - file: [src/cron/normalize.ts:71](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/cron/normalize.ts#L71)

  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Gateway path still normalizes before schema validation:** `cron.add` calls `normalizeCronJobCreate(...)` before `validateCronAddParams(...)`, and the cron schema still requires `expr: NonEmptyString`. On `main`, an explicit empty string is not preserved through normalization, so the original report is not resolved in the current request path.
