## Summary

Keep this issue open. Current main still treats Control UI/WebChat as an operator-selected session surface that defaults to the main session, while `identityLinks` canonicalization only fires when routing has a concrete direct peer id. Telegram DMs provide that peer id; default WebChat/Control UI does not. The docs still say both that `identityLinks` can link a person across channels and that WebChat defaults to the selected agent's main session, so the requested runtime mode or explicit docs clarification remains unresolved.

## Best Possible Solution

Keep the issue open for maintainer product/API judgment. The likely product path is an explicit Control UI/WebChat session mode or identity setting that can resolve a configured WebChat identity such as `webchat:default` through `session.identityLinks` and bind it to the same canonical direct key as linked Telegram DMs. If maintainers decide current operator semantics are intentional, update the public Session Management, Channel Routing, and WebChat docs to state that `identityLinks` applies to direct channel peers and that WebChat remains main-session/operator scoped unless the user manually switches sessions.
