## Summary

Keep open. Current `main` still defaults WebChat/Control UI to the selected agent's main session, `session.identityLinks` is only applied in direct-message session-key construction, and the docs still reflect the same ambiguity the issue reports rather than a shipped fix.

## Evidence

- **Control UI persists `main` as the default chat session:** UI settings load with `sessionKey: "main"` and `lastActiveSessionKey: "main"`, so a fresh WebChat/Control UI session starts from main rather than an identity-linked direct session.
  - file: [ui/src/ui/storage.ts:193](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/ui/src/ui/storage.ts#L193)
  - command: `nl -ba ui/src/ui/storage.ts | sed -n '189,210p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Gateway-side session defaulting only normalizes main-session aliases:** `applySessionDefaults` resolves aliases like `main` to the configured main session key and preserves explicit user selections, but it never consults `session.identityLinks` or derives a canonical direct session for WebChat.
  - file: [ui/src/ui/app-gateway.ts:207](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/ui/src/ui/app-gateway.ts#L207)
  - command: `nl -ba ui/src/ui/app-gateway.ts | sed -n '207,255p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **`identityLinks` is wired into DM routing, not WebChat startup/session selection:** `buildAgentPeerSessionKey` applies linked identities only when building direct-message session keys for non-`main` DM scopes; otherwise it returns the agent main session key. Coverage in `resolve-route.test.ts` exercises Telegram/Discord DM identity linking, not WebChat default-session selection.
  - file: [src/routing/session-key.ts:142](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/routing/session-key.ts#L142)
  - command: `nl -ba src/routing/session-key.ts | sed -n '130,223p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Current docs still say WebChat defaults to main while session docs broadly describe cross-channel identity linking:** `docs/channels/channel-routing.md` and `docs/platforms/mac/webchat.md` both say WebChat defaults to the selected agent's main session, while `docs/concepts/session.md` still says `session.identityLinks` lets the same person across channels share one session. That matches the reported expectation gap; it does not show a resolved behavior change on `main`.
  - file: [docs/channels/channel-routing.md:130](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/docs/channels/channel-routing.md#L130)
  - command: `nl -ba docs/channels/channel-routing.md | sed -n '130,145p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Reply-routing code treats WebChat as an admin/monitoring surface over existing sessions:** Session delivery logic explicitly preserves established external routes when `webchat` accesses a session created by Telegram/iMessage/etc., describing webchat/dashboard access as monitoring/admin access that must not overwrite the external route.
  - file: [src/auto-reply/reply/session-delivery.ts:103](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/auto-reply/reply/session-delivery.ts#L103)
