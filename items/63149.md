## Summary

Keep this issue open. Current main still has only generic lane caps and heartbeat wake coalescing, not a heartbeat-origin resource policy for the reported macOS local-gateway load path. Heartbeat runs still execute full agent turns, heartbeat admission checks main/session queue pressure but not the shared subagent lane, and subagent work still runs in the same gateway process. The related inference-guard PR was closed unmerged and no equivalent implementation is present on current main.

## Best Possible Solution

Keep this issue open as a core gateway and agent-scheduling performance bug. The best maintainer path is to add or validate heartbeat-origin resource policy: cap or serialize subagent spawns initiated by heartbeat turns, account for shared subagent-lane backlog before starting heartbeat runs, expose actionable lane/backlog/drain/event-loop signals, and add a focused regression or stress test with multiple heartbeat-enabled agents spawning background subagents while channel responsiveness remains bounded.
