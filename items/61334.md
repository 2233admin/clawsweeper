## Summary

Keep #61334 open. The original 4x/second observation was corrected by the reporter, but the reopened request is specifically for wake-on-next-job scheduling instead of the current fixed one-minute recheck. Current main still clamps cron timer sleeps to 60 seconds and logs `cron: timer armed` when it re-arms, so a daily-only installation can still produce one debug entry per minute.

## Best Possible Solution

Keep the issue open for maintainer/product judgment. A good fix would either implement a longer dynamic sleep until the earliest `nextRunAtMs` with a separate low-noise maintenance/watchdog path, or explicitly decide to keep the one-minute recovery tick and reduce/debug-gate the `cron: timer armed` log so normal daily schedules do not flood gateway logs.
