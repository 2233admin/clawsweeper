## Summary

Keep open. Current `main` still has no per-channel+recipient outbound serialization, and the PR discussion identifies real correctness gaps in the proposed lock placement/keying that are not disproven by current code or tests.

## Evidence

- **Current main has no per-target serialization:** `deliverOutboundPayloads` writes a queue entry and then immediately calls `deliverOutboundPayloadsWithQueueCleanup`; there is no mutex or serializer keyed by channel/account/recipient in this path.
  - file: [src/infra/outbound/deliver.ts:688](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/infra/outbound/deliver.ts#L688)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '688,750p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Existing claim is only per queue entry:** `withActiveDeliveryClaim` deduplicates by `entryId` only. Two concurrent sends to the same recipient but different queue entries still run independently.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:97](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/infra/outbound/delivery-queue-recovery.ts#L97)
  - command: `nl -ba src/infra/outbound/delivery-queue-recovery.ts | sed -n '97,109p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Queue I/O happens before any proposed serializer point:** `enqueueDelivery` performs async directory creation and file write before returning. That matches the review concern that a serializer acquired only after queue persistence would not guarantee FIFO by send initiation under concurrent callers.
  - file: [src/infra/outbound/delivery-queue-storage.ts:133](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/infra/outbound/delivery-queue-storage.ts#L133)
  - command: `nl -ba src/infra/outbound/delivery-queue-storage.ts | sed -n '133,160p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Account IDs are normalized elsewhere:** OpenClaw canonicalizes account IDs through `normalizeAccountId`, so a serializer key built from raw `params.accountId` can split logically identical accounts.
  - file: [src/routing/account-id.ts:35](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/routing/account-id.ts#L35)
  - command: `nl -ba src/routing/account-id.ts | sed -n '35,46p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Recipient strings are normalized by channel code:** Signal strips a leading `signal:` prefix in `parseTarget`, so raw `to` is not a canonical recipient key for all channels.
  - file: [extensions/signal/src/send.ts:62](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/signal/src/send.ts#L62)
  - command: `nl -ba extensions/signal/src/send.ts | sed -n '62,85p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Thread identity changes the real outbound target:** Discord rewrites threaded sends to `channel:${threadId}` in `resolveDiscordOutboundTarget`, so a serializer keyed only by `to` would over-serialize distinct threads and still not reflect the actual routed destination.
  - file: [extensions/discord/src/outbound-send-context.ts:29](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/discord/src/outbound-send-context.ts#L29)
  - command: `nl -ba extensions/discord/src/outbound-send-context.ts | sed -n '29,40p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **PR discussion already surfaced unresolved correctness concerns:** The PR review thread includes concrete comments about serializer placement after queue I/O and about raw account/recipient/thread key derivation. Those concerns line up with current `main` code paths and were not shown to be obsolete.

## Risks / Open Questions

- Out-of-order same-recipient deliveries remain plausible on current `main` when separate sessions send concurrently.
- Merging the PR as-is would likely need follow-up work around lock acquisition order and canonical target keying.
