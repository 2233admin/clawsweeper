## Summary

Keep PR #69701 open. Current main still has the exact loopback listen-host behavior the PR changes: when the configured bind host is 127.0.0.1 and ::1 is bindable, it returns both hosts with no Windows guard. Main does contain a separate #69674 changelog/source fix for reserving /healthz and /readyz before later request stages, but that does not change the Windows dual-bind decision described by this PR.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to either land the Windows-specific resolveGatewayListenHosts guard with focused win32 and non-Windows regression tests, or explicitly supersede it only after Windows verification shows the shipped probe-route change fully resolves the reported hang without changing listener host selection.
