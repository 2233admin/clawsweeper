## Summary

Keep PR #69701 open. Current main does not contain the Windows-specific bind-host guard proposed by the PR: `resolveGatewayListenHosts("127.0.0.1")` still probes `::1` and can return both `127.0.0.1` and `::1`, and gateway startup still listens on every returned host. Main has an adjacent #69674 health/readiness route fix, but that does not change loopback bind selection or prove the Windows dual-bind hang is obsolete.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is either to land the narrowly scoped `resolveGatewayListenHosts` Windows guard with the regression tests from the PR, or explicitly close the PR only after current main is proven on native Windows to avoid the reported HTTP hang without changing dual loopback binding.

## Evidence
