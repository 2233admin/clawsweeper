## Summary

Keep open. Current `main` already supports manual per-session labels in Control UI, and channel accounts already have an account `name` field for channel status surfaces, but there is no wiring that copies an account-level display name into new session labels/display names for the session selector. The requested config-driven account-to-session labeling feature is still unimplemented.

## Evidence

- **Channel accounts already expose a display name field, but only as `name`:** Shared channel account config defines `name?: string` with the comment `Optional display name for this account (used in CLI/UI lists)`, which matches account-status surfaces rather than session-label propagation.
  - file: [src/config/types.channel-messaging-common.ts:15](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/types.channel-messaging-common.ts#L15)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Session metadata stores conversation/thread labels and accountId separately:** Inbound session metadata derives `origin.label` from `resolveConversationLabel(ctx)` and stores `ctx.AccountId` separately as `origin.accountId`; there is no lookup of configured account `name`/display label here.
  - file: [src/config/sessions/metadata.ts:63](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/metadata.ts#L63)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Gateway session rows do not join account config names into session display names:** `buildGatewaySessionRow` resolves `displayName` from `entry.displayName`, group-display derivation, `entry.label`, or `origin.label`. It does not use `origin.accountId` to read a configured account name.
  - file: [src/gateway/session-utils.ts:1239](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/gateway/session-utils.ts#L1239)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Control UI session selector prefers explicit session label/displayName only:** The chat session selector resolves display text from `row.label` first, then `row.displayName`, then a parsed session-key fallback. No account-config display label is consulted.
  - file: [ui/src/ui/chat/session-controls.ts:425](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/ui/src/ui/chat/session-controls.ts#L425)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current workaround is manual session labeling in Control UI, not account-driven auto-labeling:** The Sessions view renders an editable `row.label` input and patches it through `props.onPatch(row.key, { label: value })`. The RPC schema likewise exposes `sessions.patch.label`, which confirms manual per-session labeling exists on current main.
  - file: [ui/src/ui/views/sessions.ts:516](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/ui/src/ui/views/sessions.ts#L516)
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
