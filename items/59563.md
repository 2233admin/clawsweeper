## Summary

Keep #59563 open. Current main has stable Feishu session-key routing and persists session metadata once a turn reaches generic session initialization, but it does not prove the reported disconnect-loss mode is fixed. The Feishu handler still marks an inbound message as processed before the later agent dispatch/session transcript path, and I did not find a durable inbound journal, reconnect backfill, or user-visible session-switch notice that would close the gap described in the issue.

## Best Possible Solution

Keep the issue open and fix the durability boundary: make Feishu inbound processing transactional by journaling the inbound event before dedupe finalization, marking it processed only after session store/transcript persistence succeeds, and adding reconnect recovery/backfill or an explicit user notice when a reconnect gap forces a new session. Add a regression test that simulates failure between Feishu dedupe finalization and generic session dispatch.

- Messages missed during a local gateway/WebSocket disconnect do not appear to be backfilled into OpenClaw session history.
