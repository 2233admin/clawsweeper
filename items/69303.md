## Summary

Keep open. Current `main` still has the reported asymmetric inbound dedupe behavior: normal completion commits the key, but the error path releases only the in-flight guard, so the same inbound `MessageSid` can be claimed again after a provider/runtime failure. The linked PR #71819 is the right active fix path, but it is open/unmerged in the provided GitHub context and its fix is not present on current `main`.

## Best Possible Solution

Keep #69303 open and land PR #71819 or an equivalent main-branch fix that preserves release-style retry for failures before any visible progress, but stamps or poisons the inbound dedupe key after externally visible dispatch or other side-effectful progress. Keep regression coverage for both safe pre-dispatch retry and replay rejection after visible progress.
