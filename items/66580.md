## Summary

Keep open. The heartbeat payload fix from PR #66580 is not implemented on current `main`, and the isolated-session queue mismatch described in issue #66487 still appears present. One unrelated `replay_invalid` classifier change from the PR is already on `main`, so the PR is mixed rather than obsolete.

## Evidence

- **Exec heartbeat prompt is still generic on main:** `buildExecEventPrompt` still takes only `opts` and tells the model the result is 'shown in the system messages above'; it does not accept or embed the queued exec event payload.
  - file: [src/infra/heartbeat-events-filter.ts:41](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/infra/heartbeat-events-filter.ts#L41)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Heartbeat runner still classifies exec events without passing payload text:** `resolveHeartbeatRunPrompt` still computes `hasExecCompletion` as a boolean over pending events and calls `buildExecEventPrompt({ deliverToUser: ... })`, so the actual exec completion text is not threaded into the heartbeat prompt body.
  - file: [src/infra/heartbeat-runner.ts:680](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/infra/heartbeat-runner.ts#L680)

  - file: [src/infra/heartbeat-runner.ts:558](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/infra/heartbeat-runner.ts#L558)

  - file: [src/infra/heartbeat-runner.ts:803](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/infra/heartbeat-runner.ts#L803)
