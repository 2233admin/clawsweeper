## Summary

Keep this issue open. Current main still has a real LiteLLM/OpenAI tool-schema compatibility gap for bundle MCP tools: bundle MCP schemas enter as raw MCP `inputSchema`, LiteLLM is an `openai-completions` provider without a schema hook, provider schema normalization runs before bundle MCP materialization, and the later generic OpenAI completions serialization path does not recursively sanitize array-valued `type` fields.

## Best Possible Solution

Keep this issue open. The best fix is a focused regression test that feeds bundle MCP-style schemas containing nested array-valued `type`, missing object metadata, and optional fields through bundle MCP materialization plus the LiteLLM/OpenAI completions request path, then add an OpenAI-compatible sanitizer or provider-family hook that runs after bundle tools are added and before transport serialization. It should preserve non-OpenAI provider behavior and improve diagnostics so the rejected tool and schema path are visible.
