## Summary

Current `main` already exposes the requested capability through the generic Gateway HTTP tool surface: `POST /tools/invoke` uses Gateway bearer auth, can invoke `sessions_spawn` when the operator explicitly allows that tool over HTTP, and `sessions_spawn` accepts the requested spawn arguments and returns an immediate accepted result with the child session key and run id while the child continues asynchronously.

## Risks / Open Questions

- This is not the exact proposed route shape: current main uses generic `POST /tools/invoke` instead of a dedicated `/api/sessions/spawn` endpoint.
- The current HTTP result returns `childSessionKey` and `runId`, not the exact proposed `202 { sessionKey, sessionId }` payload.
- Remote spawning is intentionally gated behind explicit `gateway.tools.allow: ["sessions_spawn"]`, so operators must opt in.

## Close Comment

Closing from a Codex review because current `main` already provides this capability through the Gateway HTTP tool surface added in `4630ce3d9e2f089062ccd386b0ae241325961d1a`. `POST /tools/invoke` uses the normal Gateway bearer auth (`docs/gateway/tools-invoke-http-api.md:11-25`), and when the operator explicitly opts in with `gateway.tools.allow: ["sessions_spawn"]`, the HTTP path can invoke `sessions_spawn` directly. The spawn tool accepts the requested inputs (`task`, `agentId`, `model`, `label`, `timeoutSeconds`) in `src/agents/tools/sessions-spawn-tool.ts:100-116`, and the underlying spawn path returns immediately with `status: "accepted"`, `childSessionKey`, and `runId` in `src/agents/subagent-spawn.ts:1150-1154` while the child continues asynchronously. The dedicated `/api/sessions/spawn` route from this issue is not present, but the reported capability gap on `main` is already covered well enough by `/tools/invoke` + `sessions_spawn`.
