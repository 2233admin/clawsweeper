## Summary

Keep this PR open. Current main still lacks the PR's central OpenAI-compatible `functions exec:0` / `functions.exec:0` tool-call ID and prefixed tool-name normalization, so the reported Kimi-style pairing failure remains plausible. The PR needs maintainer review rather than closure, especially because it also touches the node host exec workdir/approval path.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a narrow agent-side fix that normalizes only leading `functions ` versus `functions.` where pairing requires it, strips `functions.` / `functions ` prefixes before allowed-name checks and `sessions_spawn` redaction, preserves raw IDs for pending-state and replay-safe signed-thinking invariants, and includes regression tests for mixed dot/space IDs plus prefixed names. Split or clearly justify the node host exec workdir changes before merge.

- The PR currently mixes the tool ID/name normalization fix with node host exec workdir/approval behavior. That command-execution path needs explicit maintainer review or a split PR before merge.

- The provided verification is focused unit coverage; a live Kimi/OpenAI-compatible provider check would further reduce integration risk, but the lack of that check is not a reason to close the PR.
