## Summary

Keep open. Current `main` still has the TUI connect path bypass local TLS fingerprint resolution, while `GatewayClient` only activates the custom self-signed TLS verification path when `tlsFingerprint` is present. The status/probe path already resolves and forwards that fingerprint, so the report still matches current shipped code rather than an obsolete behavior.

## Evidence

- **TUI path still omits TLS fingerprint:** `ResolvedGatewayConnection` only includes `url`, `token`, `password`, and `allowInsecureLocalOperatorUi`, and `GatewayChatClient` forwards only those fields into `new GatewayClient(...)`. `resolveGatewayConnection()` never loads local TLS runtime state or returns a fingerprint.
  - file: [src/tui/gateway-chat.ts:48](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/tui/gateway-chat.ts#L48)

  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Gateway client needs `tlsFingerprint` for pinned local TLS:** `GatewayClient` only installs the custom TLS verification path for `wss://` when `this.opts.tlsFingerprint` is set; otherwise it connects with default Node TLS verification. That matches the reported failure mode against a self-signed local cert.
  - file: [src/gateway/client.ts:225](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/gateway/client.ts#L225)

  - file: [src/commands/gateway-status.ts:90](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/commands/gateway-status.ts#L90)
