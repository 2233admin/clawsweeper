## Summary

Keep open. Current main still has the reported TUI-specific TLS trust gap: the TUI local gateway path resolves the local WSS URL and auth, but never loads or passes the local gateway TLS fingerprint into `GatewayClient`. The status/probe path already does this, so the issue remains a real current-code mismatch rather than an implemented or duplicate cleanup candidate.

## Best Possible Solution

Keep this issue open. Implement the TUI trust handoff in `src/tui/gateway-chat.ts`: add optional `tlsFingerprint` to the resolved connection, load the local gateway TLS runtime only for configured local `wss://` targets with gateway TLS enabled and no URL override/remote URL, pass it into `GatewayClient`, and add focused tests in `src/tui/gateway-chat.test.ts` for propagation, unavailable-runtime fallback, URL override, and remote cases.
