## Summary

Keep open. Current `main` still resolves the systemd user-unit path from `HOME`/`USERPROFILE` only, while sudo systemctl operations target `SUDO_USER`'s user scope; the submitted PR's `os.userInfo({ username })` approach is also invalid and would not fix the mismatch.

## Evidence

- **Home resolution on main still ignores sudo caller:** `resolveHomeDir` returns only `HOME` or `USERPROFILE` and has no `SUDO_USER` fallback, so a sudo shell with `HOME=/root` still resolves to `/root`.
  - file: [src/daemon/paths.ts:8](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/daemon/paths.ts#L8)
  - command: `nl -ba src/daemon/paths.ts | sed -n '1,80p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Systemd unit path still uses resolved home directly:** `resolveSystemdUnitPathForName` builds the user unit path as `<home>/.config/systemd/user/<name>.service`, so the wrong home directory writes the unit into the wrong user tree.
  - file: [src/daemon/systemd.ts:47](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/daemon/systemd.ts#L47)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '1,80p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Sudo systemctl commands target the invoking user scope:** Under sudo, `execSystemctlUser` prefers `--machine <SUDO_USER>@ --user` and explicitly avoids bare `--user`, so the command scope and unit-file write path still diverge when `HOME=/root`.
  - file: [src/daemon/systemd.ts:374](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/daemon/systemd.ts#L374)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '367,398p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Main tests already encode sudo caller scope, not sudo home resolution:** The current test suite verifies restart commands target the sudo caller's user scope, but there is no corresponding mainline test covering unit-path home resolution under sudo.
  - file: [src/daemon/systemd.test.ts:844](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/daemon/systemd.test.ts#L844)
  - command: `nl -ba src/daemon/systemd.test.ts | sed -n '844,867p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **PR's proposed lookup is runtime-invalid:** A direct runtime check shows `os.userInfo({ username: "nobody" })` still returns the current user, so the PR's proposed `os.userInfo({ username: sudoUser })` call would not resolve another user's homedir.
  - command: `node -p 'const os=require("node:os"); JSON.stringify({current:os.userInfo().username, withUsername:os.userInfo({username:"nobody"}).username, sameHome:os.userInfo().homedir===os.userInfo({username:"nobody"}).homedir})'`

## Risks / Open Questions

- Closing this PR now would drop a still-real sudo/systemd unit-path bug from maintainer visibility unless a separate replacement lands first.
- Merging this PR as written would give false confidence: the proposed lookup still resolves the current effective user, so the original `/root/.config/systemd/user` mismatch remains.
