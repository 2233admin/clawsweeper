## Summary

Keep open. Current main partially addresses the `sessions_send` part by routing inter-session work to per-target-session `nested:<sessionKey>` lanes, but the broader request is still valid: cron and hook-dispatched isolated agent runs still route through the unscoped `nested` lane, that lane still defaults to `maxConcurrent: 1`, and there is no public config surface for nested-lane concurrency or hook lane selection.

## Best Possible Solution

Keep #57952 open and narrow the maintainer path: preserve the shipped per-session `sessions_send` lane behavior, then add a documented and tested concurrency strategy for cron/hook-dispatched isolated agent runs. The likely implementation should cover startup and hot reload, schema/types/help/docs, and regression tests for simultaneous cron/hook jobs. Maintainers should also decide whether `/hooks/agent` should accept a trusted `lane` field or whether hook concurrency should be controlled only by config.
