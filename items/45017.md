## Summary

Keep open. Current `main` still has the malformed-content crash surface this PR targets, and the PR remains useful: UI normalization and some extraction paths still dereference `content` array entries without checking for `null`/`undefined`/non-objects first.

## Evidence

- **Normalizer still trusts every content entry:** `normalizeMessage` still casts each `content` item to `Record<string, unknown>` and immediately reads `x.type` / `item.type` with no null/object guard, so malformed array members can still throw on current `main`. This affects both tool-message detection and content normalization.
  - file: [ui/src/ui/chat/message-normalizer.ts:249](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/ui/src/ui/chat/message-normalizer.ts#L249)

  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Rendered chat path still depends on normalizeMessage:** The dashboard render path still calls `normalizeMessage(message)` while building grouped chat output, so a malformed historical message can still break session rendering rather than being isolated.
  - file: [ui/src/ui/chat/grouped-render.ts:1379](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/ui/src/ui/chat/grouped-render.ts#L1379)
