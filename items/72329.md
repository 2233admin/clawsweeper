## Summary

Keep #72329 open. Current main accepts `compat.thinkingFormat: "qwen-chat-template"` in config/catalog normalization, but the OpenAI Completions request builder still only maps reasoning to OpenRouter `reasoning` or generic `reasoning_effort`; it never injects `chat_template_kwargs.enable_thinking` for Qwen-on-vLLM. Main does have a manual `params.chat_template_kwargs` / `extra_body` workaround and an automatic vLLM/Nemotron path, but that does not satisfy the issue's central request: make Discord `/think` and other session thinking controls dynamically affect Qwen chat-template kwargs.

## Best Possible Solution

Implement a narrow OpenAI Completions payload mapping for `compat.thinkingFormat: "qwen-chat-template"` that derives `chat_template_kwargs.enable_thinking` from the resolved OpenClaw thinking level, with precedence rules that preserve explicit `params.chat_template_kwargs` and `params.extra_body.chat_template_kwargs`. Add focused tests around `buildOpenAICompletionsParams` or the post-plugin wrapper for vLLM/Qwen and update the vLLM/provider docs once the behavior is shipped.
