## Summary

Keep open. Current main still routes markdown-table content into Feishu interactive cards, and the bundled Feishu send path still treats any nonzero card-send response as a generic failure with no ErrCode 11310-specific split/degrade fallback. The issue discussion also points to an upstream/plugin-side fix and an open OpenClaw PR, which supports that this is a real remaining gap rather than an obsolete report.

## Evidence

- **Markdown tables still select card rendering:** `shouldUseCard()` matches markdown tables, and outbound sends card messages when `renderMode` is `card` or `auto` with table content. That is the exact path the report says triggers Feishu's table-count limit.
  - file: [extensions/feishu/src/outbound.ts:61](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/outbound.ts#L61)
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Current card send path has no 11310 recovery:** `sendCardFeishu()` sends the interactive card and immediately delegates to the generic API success assertion. There is no branch for Feishu code `11310`, no message splitting, and no downgrade to post/text mode.
  - file: [extensions/feishu/src/send.ts:498](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/send.ts#L498)
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Nonzero Feishu responses still become generic errors:** `assertFeishuMessageApiSuccess()` throws on any nonzero code using only the provided error prefix and response message/code. This matches the issue's reported `Feishu card send failed: code 11310` behavior and shows no retry/fallback layer exists here.
  - file: [extensions/feishu/src/send-result.ts:9](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/send-result.ts#L9)
