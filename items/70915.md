## Summary

Keep open. Current `main` does not contain the requested ZAI vision-endpoint routing fix, and the PR’s proposed request-scoped config rewrite appears insufficient because media understanding resolves models through shared `models.json` discovery. The review discussion already identified that race, and the PR commit message says it was only documented, not fixed.

## Evidence

- **Current ZAI media-understanding path on main is still generic:** `zaiMediaUnderstandingProvider` on current `main` still delegates image requests straight to the shared `describeImageWithModel` / `describeImagesWithModel` helpers, with no ZAI-specific endpoint routing for vision models.
  - file: [extensions/zai/media-understanding-provider.ts:7](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/zai/media-understanding-provider.ts#L7)
  - command: `nl -ba extensions/zai/media-understanding-provider.ts | sed -n '1,80p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Shared image runtime resolves models from discovered registry, not a per-request transport override:** The media-understanding runtime first calls `ensureOpenClawModelsJson(params.cfg, params.agentDir)`, then loads discovery and resolves the model from `discoverModels(authStorage, params.agentDir)`. That means the effective base URL comes from shared model discovery state, not from a dedicated image-request baseUrl override seam.
  - file: [src/media-understanding/image.ts:139](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/media-understanding/image.ts#L139)
  - command: `nl -ba src/media-understanding/image.ts | sed -n '135,150p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **models.json is shared state keyed by config fingerprint:** `ensureOpenClawModelsJson` computes a fingerprint, consults a shared ready cache by `<agentDir>/models.json`, and may rewrite that file before discovery. The corresponding planning path serializes provider config into final `models.json` contents. This supports the review concern that cloning/mutating `cfg` per request is interacting with shared state rather than a request-local transport override.
  - file: [src/agents/models-config.ts:138](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/agents/models-config.ts#L138)
  - command: `nl -ba src/agents/models-config.ts | sed -n '138,190p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **User-facing docs still promise auto-resolved image understanding from configured Z.AI auth:** The public Z.AI docs say image understanding uses `glm-4.6v` and is auto-resolved from configured Z.AI auth, while the same page explicitly documents Coding Plan onboarding choices. That makes the underlying bug user-visible and still worth fixing, not something to close out.
  - file: [docs/providers/zai.md:142](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/docs/providers/zai.md#L142)

The pushed commit message says the race was documented in JSDoc, not eliminated, so the review discussion points to a real remaining flaw in the proposed fix rather than a resolved nit.

## Risks / Open Questions

- The underlying ZAI Coding Plan image-understanding bug still appears real on current `main`, so users can likely still hit 404s with `zai/glm-4.6v`-class vision models.
- Merging this PR as-is risks an intermittent fix because it depends on request-scoped config changes feeding a shared `models.json` discovery path instead of a true per-request endpoint override.
