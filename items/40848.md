## Summary

Keep #40848 open. Current main still does not expose or forward Telegram `entities`/`custom_emoji_id` through the shared `message` tool, and the issue has a concrete API-shaped request that fits the existing Telegram plugin-owned message-tool seam.

## Best Possible Solution

Keep the issue open and implement it in the Telegram plugin surface: add a typed/sanitized send-action `entities` schema contribution in `extensions/telegram/src/message-tool-schema.ts` or `channel-actions.ts`, parse it in `extensions/telegram/src/action-runtime.ts`, pass it through `sendMessageTelegram`, and update `extensions/telegram/src/send.ts` to choose Telegram `entities`/`caption_entities` instead of `parse_mode` where appropriate. Add focused tests for custom emoji text sends, UTF-16 offsets, HTML fallback interaction, chunking behavior, and media captions if supported.

## Risks / Open Questions

- A naive passthrough could conflict with the current HTML `parse_mode` rendering and fallback behavior, because Telegram specifies entities as an alternative to parse mode for message text.
- Offsets and lengths must be validated or documented as UTF-16 code units, especially for emoji/surrogate-pair text.
- Chunking and media-caption paths need explicit tests so entities are not applied to the wrong chunk or silently dropped.
