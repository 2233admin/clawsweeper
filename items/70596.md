## Summary

Keep open. `main` does not already implement builtin-local startup prewarm, and the proposed approach appears to introduce a real regression: a failed startup prewarm would poison the shared cached builtin manager for the rest of the process, which matches the unresolved P1 review concern on the PR.

## Evidence

- **Current main still skips builtin startup prewarm:** `startGatewayMemoryBackend` only initializes agents when the resolved backend is `qmd`; builtin memory is still skipped entirely, so this PR is not already implemented on `main`.
  - file: [src/gateway/server-startup-memory.ts:23](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/gateway/server-startup-memory.ts#L23)
  - command: `nl -ba src/gateway/server-startup-memory.ts | sed -n '1,43p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Default builtin managers are cached and reused:** `MemoryIndexManager.get` caches default-purpose managers by agent/workspace/settings key. A startup prewarm done through the normal default path would populate the same cached manager later reused by real `memory_search` requests.
  - file: [extensions/memory-core/src/memory/manager.ts:167](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/memory-core/src/memory/manager.ts#L167)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '155,184p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Failed provider initialization stays sticky on the cached manager:** `ensureProviderInitialized()` memoizes `providerInitPromise` and only clears it after successful initialization. If initialization rejects, the rejected promise remains on the manager; later calls re-await the same failure instead of retrying. `probeEmbeddingAvailability()` goes through that exact path.
  - file: [extensions/memory-core/src/memory/manager.ts:248](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/memory-core/src/memory/manager.ts#L248)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '238,271p'; nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '810,825p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Status-only managers intentionally bypass the cache:** The existing memory runtime tests explicitly assert that builtin managers fetched with `purpose: "status"` are not cached. That supports the PR's note that a status probe would not warm the real default manager path.
  - file: [extensions/memory-core/src/memory/search-manager.test.ts:646](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/memory-core/src/memory/search-manager.test.ts#L646)
  - command: `nl -ba extensions/memory-core/src/memory/search-manager.test.ts | sed -n '646,677p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **PR discussion already identified the same blocker:** The unresolved Codex review comment on PR #70596 flags a P1: calling `probeEmbeddingAvailability()` on the cached builtin manager can permanently break later searches after a transient startup failure because the failed init promise is retained.
  - sha: [8296e9c8b3c5](https://github.com/openclaw/openclaw/commit/8296e9c8b3c5ca9f3783f2aace1e42b7fe49c8bc)

## Risks / Open Questions

- Merging the PR as written could turn a transient local embedding startup failure into a persistent builtin memory failure until process restart.
- The PR also carries an unrelated `src/plugins/provider-runtime.test-support.ts` test change, so even aside from the blocker it is not a minimal isolated fix.
