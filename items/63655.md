## Summary

Keep this issue open. Current main still unconditionally sends `encoding_format: "float"` from the memory-lancedb plugin, which conflicts with the reporter's follow-up that ZhiPu rejects the `encoding_format` key entirely. The shipped Ollama/base64 fix is real but narrower than this issue and there is no current config or alternate embedding path that omits the parameter safely.

## Best Possible Solution

Keep this issue open and implement the compatibility fix in the memory-lancedb plugin: add an explicit embedding encoding/provider compatibility mode with an omit option, send provider-compatible requests without `encoding_format` when selected, normalize returned embeddings by accepting float arrays and decoding only string/base64 responses, and add regression tests for both forced-float providers and reject-unknown-parameter providers.

## Risks / Open Questions

- A fix must preserve the already-landed Ollama/OpenAI-compatible forced-float improvement while adding a path for providers such as ZhiPu that reject unknown `encoding_format` parameters.
- Because the OpenAI SDK injects and decodes base64 when the field is omitted, implementing an omit mode through the same `client.embeddings.create()` call may still send `encoding_format: "base64"`; the fix likely needs a narrow raw embeddings request or wrapper around SDK response handling.
