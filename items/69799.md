## Summary

Keep this issue open. Current main has useful mitigations, including app-owned macOS TCC surfaces for some node actions, a local-prefix installer option, stable Homebrew Node path resolution, and an optional gateway wrapper. It still does not make a bundled or dedicated OpenClaw-owned Node/Gateway runtime the supported default for macOS npm/global Gateway installs, so the requested TCC boundary remains unresolved.

## Best Possible Solution

Keep this as core macOS security and packaging architecture work. The best fix is to make the supported macOS Gateway/FDA runtime use a stable OpenClaw-owned identity by default, either a signed app-owned Gateway/runtime or a dedicated installed Node binary, with migration, doctor/onboarding, docs, and LaunchAgent reinstall behavior coordinated with #7227, #69561, and #22179. Treat the local-prefix installer and --wrapper as mitigations until maintainers explicitly make one the supported TCC boundary.

## Risks / Open Questions

- Closing this now would treat optional local-prefix or wrapper mitigations as equivalent to the requested supported default OpenClaw-owned TCC identity.
- Users on the documented npm/global macOS Gateway path can still grant macOS TCC permissions to a shared Node binary.
- Current docs describe app-owned TCC handling for some macOS node surfaces, while Gateway runtime docs and code still use an external CLI/runtime path; maintainers need to align the product boundary.
