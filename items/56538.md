## Summary

Keep #56538 open. Current main has a Windows code-page decoder for the node-host path, but the PR's targeted process/supervisor exec paths still decode captured output as UTF-8 strings or per-chunk Buffer.toString(), so the linked Windows GBK/CJK output bug is not fully implemented on main. The PR still needs maintainer review, cleanup, and regression tests rather than automated closure.

## Best Possible Solution

Keep the PR open and ask for a refreshed, focused core fix. The best implementation path is to make Windows output decoding a shared process helper used by runExec, runCommandWithTimeout, and supervisor child streaming paths; preserve UTF-8 fast paths and non-Chinese Windows code pages; handle unsupported TextDecoder labels safely; and add regression tests for CP936/GBK bytes plus split multibyte streaming chunks. The ACPX CLAUDE.md churn should be removed before maintainer review.
