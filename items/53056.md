## Summary

Keep this issue open. Current main still performs Slack ampersand escaping in the formatter and main outbound paths, while the linked fix attempts were closed unmerged after discussion showed that removing Slack escaping globally would conflict with Slack mrkdwn rules and still would not isolate the second encoding boundary.

## Best Possible Solution

Keep this issue open. Add focused Slack regression coverage for raw `R&D`/`B&A`, already-escaped `&amp;`, and numeric `&#38;` through normal send, slash/reply delivery, preview final edits, native streaming/tool-progress paths, and block fallbacks. Then fix the specific second encoding boundary while preserving Slack's required single escaping pass for literal `&`, `<`, and `>`.
