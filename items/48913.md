## Summary

Keep open. Current `main` still handles overloaded errors by rotating to another auth profile and then falling back or surfacing the error; it does not retry the same single profile/provider before failing.

## Evidence

- **No same-profile overload retry helper in run loop:** `maybeBackoffBeforeOverloadFailover()` only sleeps before overload failover, and the prompt path calls it immediately before `continue` after `advanceAuthProfile()` or before throwing fallback. There is no branch that retries the current profile in place.
  - file: [src/agents/pi-embedded-runner/run.ts:664](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/agents/pi-embedded-runner/run.ts#L664)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Single-profile configs cannot retry current profile:** `advanceAuthProfile()` starts at `getProfileIndex() + 1` and only advances to later candidates. If there is only one profile, it returns `false`; it never reuses the current profile.
  - file: [src/agents/pi-embedded-runner/run/auth-controller.ts:450](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/agents/pi-embedded-runner/run/auth-controller.ts#L450)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Failover policy only offers rotate/fallback/surface:** `resolveRunFailoverDecision()` returns `rotate_profile`, `fallback_model`, or `surface_error` for prompt/assistant overload failures. There is no decision type for retrying the same model/profile on overload.
  - file: [src/agents/pi-embedded-runner/run/failover-policy.ts:124](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/agents/pi-embedded-runner/run/failover-policy.ts#L124)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Docs still describe auth-profile rotation, not same-profile retry:** Current docs define `overloadedProfileRotations` as same-provider auth-profile rotations before model fallback, and `overloadedBackoffMs` as delay before retrying an overloaded provider/profile rotation.
  - file: [docs/gateway/configuration-reference.md:751](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/docs/gateway/configuration-reference.md#L751)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Current behavior documented as one auth-profile retry then fallback:** The failover docs say OpenClaw allows one same-provider auth-profile retry, then switches to the next configured model fallback. That matches profile rotation, not single-profile retry logic.
  - file: [docs/concepts/model-failover.md:219](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/docs/concepts/model-failover.md#L219)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Tests cover overload rotation/fallback, not same-profile retry:** Existing tests assert overload backoff before rotating profiles and that all-overloaded candidates surface an overload summary. I found no test proving retry of the same single profile before failure.
  - file: [src/agents/pi-embedded-runner.run-embedded-pi-agent.auth-profile-rotation.e2e.test.ts:931](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/agents/pi-embedded-runner.run-embedded-pi-agent.auth-profile-rotation.e2e.test.ts#L931)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
