## Summary

Keep open. Current main still has the bug boundary described in #67746: CLI logout calls the WhatsApp plugin's local logout helper directly instead of the gateway `channels.logout` RPC that stops the live runtime, and `channels remove` mutates config only without stopping/logging out the running WhatsApp account. The related PR #67981 correctly identified the narrow CLI logout fix but is closed/unmerged and explicitly did not address remove/config deletion semantics.

## Best Possible Solution

Keep #67746 open and implement a coordinated lifecycle fix. The narrow first step should make `openclaw channels logout --channel whatsapp` prefer gateway `channels.logout` when a local gateway is reachable, with local credential cleanup only as an explicit fallback. The broader remove path should either call the same live logout/stop plus credential cleanup for destructive delete, or document that `remove` is config-only and require a separate logout/reset command. Add regression coverage in `src/cli/channel-auth.test.ts` for the gateway logout RPC path and in `src/commands/channels.remove.test.ts` for the chosen remove/delete semantics, then update `docs/cli/channels.md`.

- A fix needs a maintainer decision on destructive semantics: whether `channels remove` without `--delete` should remain disable-only, and whether any remove/delete path should also clear WhatsApp credentials.
