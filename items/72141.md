## Summary

Current main already enforces the isolation contract for isolated cron agent turns: Gateway dispatches them through a dedicated `cron:<jobId>` key, each isolated run forces a fresh session id and transcript path, and forced-new cron session rollover whitelists only safe preferences while clearing stale transcript, routing, runtime, and ambient context fields. Tests and docs now cover that boundary.

## Best Possible Solution

Close this as already implemented on current main. Keep the current dedicated `cron:<jobId>` dispatch plus forced-new session sanitization, and use `current` or `session:<id>` only for jobs that intentionally need conversation continuity.
