## Summary

Keep open. Current `main` does not implement this PR's delivered-error handling, and the reported state remains reachable in the code path that returns `status: "error"` together with `delivered: true` for isolated cron runs.

## Evidence

- **`applyJobResult` still has no delivered-error special case:** Current `main` unconditionally writes `job.state.lastError = result.error` and still evaluates failure alerts for every `result.status === "error"`; there is no branch that clears error state or suppresses alerts when delivery succeeded.
  - file: [src/cron/service/timer.ts:421](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/cron/service/timer.ts#L421)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '410,530p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Delivered-error runs are still a real outcome:** The isolated cron runner builds a result with `status: hasFatalErrorPayload ? "error" : "ok"` and separately carries through `delivered`, so `status: "error"` with `delivered: true` is still a valid runtime combination.
  - file: [src/cron/isolated-agent/run.ts:822](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/cron/isolated-agent/run.ts#L822)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '812,930p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **No main-branch regression test for error+delivered state:** Current tests cover delivered-status persistence for successful runs and finished-event delivery metadata, but there is no test on `main` asserting the `status: "error"` plus `delivered: true` case discussed in this PR.
  - file: [src/cron/service.persists-delivered-status.test.ts:153](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/cron/service.persists-delivered-status.test.ts#L153)
  - command: `nl -ba src/cron/service.persists-delivered-status.test.ts | sed -n '130,247p'`

## Risks / Open Questions

- Closing this PR as already handled would drop still-relevant work: current `main` can still persist an error outcome on runs that also report successful delivery.
- The operator-facing behavior around delivered-error runs is still unsettled on `main` because failure-alert counting and alert emission remain tied to `result.status === "error"` without any delivered-status carveout.
