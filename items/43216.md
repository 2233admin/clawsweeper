## Summary

Keep this issue open. Current main has the 24-hour duplicate heartbeat suppression back in place, but the plugin-channel heartbeat fallback is still incomplete: `resolveHeartbeatDeliveryTarget()` recovers the stored session route, then still calls `resolveOutboundTarget()` and converts an unresolved plugin channel into `channel: "none", reason: "no-target"`. The existing tests cover session-layer recovery and normal registered-plugin heartbeat routing, but not the empty/unbootstrapped plugin-registry heartbeat case. The related PR #46203 is closed and unmerged in the provided context, so it does not supersede the remaining work.

## Best Possible Solution

Keep this issue open and narrow it to the remaining plugin-channel heartbeat fallback. The best fix is in `src/infra/outbound/targets.ts`: make `resolveHeartbeatDeliveryTarget()` handle `target: "last"` stored plugin routes end-to-end when session metadata already has a plugin channel and destination but the plugin registry is empty/unbootstrapped, while preserving current duplicate suppression. Add regression coverage that uses the real channel normalization path with an empty plugin registry and proves the recovered plugin route does not degrade to `no-target`.

There is no fallback that returns the recovered `resolvedTarget.channel`/`resolvedTarget.to` for the `target: "last"` stored-session plugin route.
