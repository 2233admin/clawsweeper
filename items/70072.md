## Summary

Keep open. Current main partially helps with `openclaw memory promote-explain` and verbose gateway logs, but it does not satisfy the central request: scheduled deep reports still write only aggregate counts, verbose scoring is not persisted as a workspace artifact, and there is no `openclaw memory dreaming doctor` command. The ranking path also filters failing entries before returning candidates, so the current deep report cannot explain which below-gate entries were rejected or why.

## Best Possible Solution

Keep #70072 open. The best path is a memory-core diagnostic layer that evaluates all short-term recall entries against deep thresholds, records per-candidate pass/fail reasons and component scores, renders a bounded threshold/candidate section in `memory/dreaming/deep/YYYY-MM-DD.md`, optionally writes structured verbose JSONL under `memory/.dreams/` when verbose logging is enabled, and adds either `openclaw memory dreaming doctor` or an equivalent aggregate diagnostic command with docs and tests.

## Risks / Open Questions

- The existing `promote-explain` and `rem-harness` paths are good building blocks, so implementation should extend the shared scoring/diagnostic model instead of creating a divergent scorer.
- Per-candidate reports can expose memory snippets; the final design should bound, redact, or make verbosity explicit so introspection does not over-share workspace memory content.
