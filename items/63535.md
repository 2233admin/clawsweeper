## Summary

This PR should remain open. Current main still does not implement stale cron session-row recovery in the reaper, so the idea is not obsolete, but the submitted patch remains incomplete: recovery is gated by retention pruning, it only targets canonical `:run:` keys while current cron persistence writes the visible mutable row under the stable base key, and the supplied coverage does not prove retention-disabled or defer-before-prune behavior. Security review found no supply-chain, CI, dependency, secret, network, or publish-surface changes in the supplied two-file diff.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is a rebased cron lifecycle/session recovery patch that runs independently of retention pruning, repairs the base persisted cron session row plus any legacy `:run:` rows, chooses a deliberate terminal status for interrupted work, and adds regression coverage for `sessionRetention: false`, stale versus active running sessions, base-key and legacy-run-key shapes, and short-retention defer-before-prune behavior. Then validate with the focused cron tests and the repo's changed-check lane when writable validation is allowed.

## Risks / Open Questions

- Merging the PR as-is would silently omit crash recovery for users who set `cron.sessionRetention` to `false`.
- Updating only `:run:` rows can leave the current base cron session row visible as `status: "running"`.
- A fixed two-hour orphan threshold may mark legitimate long-running cron work as done unless product behavior and status mapping are explicitly tested.
- The added test coverage in the PR does not prove one-sweep deletion deferral under short retention, because the sample rows are younger than the default 24-hour retention window.
