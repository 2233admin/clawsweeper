## Summary

Keep open. Current `main` still misses the skipped-result UI case this PR targets, so the underlying bug is not implemented on `main`. The PR remains directionally useful, but the review comment is valid: its skipped-case test uses an impossible payload shape and should be corrected before merge.

## Evidence

- **Current UI only surfaces payload failures when `res.ok === false`:** `runUpdate` on current `main` reads the `update.run` payload but only sets `lastError` inside `if (res && res.ok === false)`, so a business-level `status: "skipped"` result with `ok: true` still shows no feedback.
  - file: [ui/src/ui/controllers/config.ts:185](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/ui/src/ui/controllers/config.ts#L185)
  - command: `nl -ba ui/src/ui/controllers/config.ts | sed -n '170,215p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Browser client rejects only outer RPC failures, not business-level payloads:** `GatewayBrowserClient` resolves the request when the WS response frame has `ok: true` and rejects only when the outer frame has `ok: false`. That matches the PR description that transport-level failures reject but business-level payloads do not.
  - file: [ui/src/ui/gateway.ts:564](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/ui/src/ui/gateway.ts#L564)

- **Gateway `update.run` returns skipped as outer success:** The server responds with outer success via `respond(true, ...)` and sets payload `ok` to `result.status !== "error"`, so `status: "skipped"` produces a resolved response with payload `ok: true`.
