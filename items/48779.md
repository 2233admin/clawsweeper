## Summary

Keep open. On current `main` (`01bc49c88c0cef9807b15df4d7486a0c5fe4afbd`), `/status` and the per-reply usage footer still render tokens/cache/cost only; there is no implementation path that computes or displays API call counts for either surface.

## Evidence

- **`/status` output still ends at tokens/cache/cost:** The status card builds `usageCostLine` from token and cost data, then returns the final lines list with `usageCostLine`, `cacheLine`, and other status lines. There is no call-count field in this render path.
  - file: [src/status/status-message.ts:873](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/status/status-message.ts#L873)

- **Transcript fallback does not count API calls:** When `/status` backfills usage from the session log, it only restores `input`, `output`, `cacheRead`, `cacheWrite`, and a larger prompt/total token candidate. No request-count aggregation is computed here.
  - file: [src/status/status-message.ts:541](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/status/status-message.ts#L541)
