## Summary

Current `main` already routes Discord "Max reconnect attempts ... code 1005" gateway errors through lifecycle supervision and keeps listeners attached through disconnect, preventing the late uncaught emitter crash described in #57111 during stale-socket health-monitor restarts.

## Evidence

## Close Comment

Closing this as implemented on current `main`.

- Codex review checked current `main` and found a dedicated Discord gateway supervisor that classifies `Max reconnect attempts ... code 1005` as a handled `reconnect-exhausted` lifecycle event instead of leaving it as an uncaught emitter error (`extensions/discord/src/monitor/gateway-supervisor.ts:89`).
- The shutdown path now disconnects the gateway **before** removing lifecycle listeners, specifically so late gateway `error` events during restart/teardown are still caught and do not crash the process (`extensions/discord/src/monitor.gateway.ts:33`).
- Discord provider cleanup also disconnects the shared gateway and disposes the supervisor if startup/lifecycle exits early, covering the stale-socket restart window described here (`extensions/discord/src/monitor/provider.ts:1145`, `extensions/discord/src/monitor/provider.ts:1152`).
- Regression coverage now asserts that emitting `Max reconnect attempts ... code 1005` after cleanup does **not** throw, and that reconnect exhaustion is surfaced as a lifecycle error in tests (`extensions/discord/src/monitor/provider.test.ts:281`, `extensions/discord/src/monitor/gateway-supervisor.test.ts:107`, `extensions/discord/src/monitor/provider.lifecycle.test.ts:374`).
- This behavior is present in commit `68315792677f9628d1370d6869366442f1d3ecad` on `main`. It appears newer than release `v2026.4.23`, so there is no shipped release tag to point to yet.
