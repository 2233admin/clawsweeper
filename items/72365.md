## Summary

Keep open. #72365 is recent, non-maintainer-authored, and has concrete Docker/WSL2 reproduction data plus endpoint timings from current beta/main-era builds. Current main has some relevant mitigations, including optimistic local chat messages and a non-blocking `chat.send` ack, but I did not find source, test, changelog, or release evidence that fixes the reported gateway/control-plane unresponsiveness under chat plus expensive Control UI endpoints. Related #64253 is adjacent Windows gateway-load work, not a clear canonical duplicate for this Control UI/eager-endpoint regression.

## Best Possible Solution

Keep #72365 open for a focused maintainer investigation. Reproduce the reported Docker/WSL2 plugin set, capture event-loop delay/CPU profiles and RPC timings during chat sends, then fix the implicated path by making expensive Control UI endpoints bounded/deferred/cached or moving CPU-heavy work off the gateway event loop. Add regression coverage that proves `chat.send`, `chat.history`, websocket connect, and lightweight control requests remain responsive while synthetic slow `tools.effective`, usage, model catalog, and memory/wiki work is in flight.
