## Summary

Keep this issue open. Current main still only shows the Homebrew install command during skill onboarding and then proceeds to the brew-backed skill install path, which returns a `brew not installed` failure when no brew executable is available. The top-level shell installer can install Homebrew, but that is a separate install flow and does not fix the reported onboarding behavior. Related #14593 and #57555 are adjacent dependency-install UX reports, not merged fixes or canonical replacements for this request.

## Best Possible Solution

Keep this issue open. A conservative fix belongs in the skill dependency onboarding path: add an explicit macOS-only confirmation to install Homebrew before running brew-backed `skills.install`, re-detect `brew`, reuse the admin/failure-handling expectations from `scripts/install.sh`, and add focused tests/docs. If maintainers decide onboarding must not install Homebrew, change the prompt and docs so users are told this is a manual prerequisite instead of an install step.

## Risks / Open Questions

- Auto-installing Homebrew from onboarding would execute an external installer and may require administrator privileges, so the implementation needs explicit user consent, platform gating, failure handling, and clear non-interactive/Docker behavior.
- Changing only the top-level installer would not resolve users who enter the skill onboarding path after OpenClaw is already installed.
