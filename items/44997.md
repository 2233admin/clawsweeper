## Summary

Keep open. Current `main` does not implement this PR’s serialized read-modify-write fix. `agents add` still computes the new config from a stale snapshot, and the current `replaceConfigFile(baseHash=...)` path is still an unsynchronized read/check/write flow rather than a locked or atomic transform-on-fresh-state update.

## Evidence

- **`agents add` still precomputes from an old snapshot:** `agentsAddCommand` reads the config once, derives `nextConfig`, applies bindings, and then calls `replaceConfigFile`. That means the mutation is still computed outside any serialized critical section.
  - file: [src/commands/agents.commands.add.ts:113](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/commands/agents.commands.add.ts#L113)
  - command: `nl -ba src/commands/agents.commands.add.ts | sed -n '55,170p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Current replace path is optimistic, not serialized:** `replaceConfigFile` re-reads a snapshot and checks `baseHash`, but then simply delegates to `writeConfigFile`. There is no `withFileLock`, no retry loop here, and no transform applied under a lock to the freshest on-disk state.
  - file: [src/config/mutate.ts:145](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/config/mutate.ts#L145)

  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **`baseSnapshot` is not forwarded by the public writer:** `ConfigWriteOptions` includes `baseSnapshot`, but the exported `writeConfigFile` wrapper forwards only env/unset/runtime flags to `io.writeConfigFile`. So callers such as `replaceConfigFile` do not get an atomic write tied to the checked snapshot.
