## Summary

Keep open. Current `main` still has shared-auth websocket invalidation during reload/auth-generation changes, and the test suite explicitly preserves a case where `config.apply` disconnects an authenticated websocket even when it is just rewriting an unchanged SecretRef-backed token. I did not find strong evidence that the reported Feishu-triggered webchat disconnect is already fixed on `main`.

## Evidence

- **Feishu reload is a live hot-reload surface:** The Feishu plugin declares `reload: { configPrefixes: ["channels.feishu"] }`, and gateway hot reload restarts affected channels via stop/start. That means Feishu config/runtime churn still flows through the live reload path rather than an isolated background worker.
  - file: [extensions/feishu/src/channel.ts:603](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/feishu/src/channel.ts#L603)
  - command: `nl -ba extensions/feishu/src/channel.ts | sed -n '598,606p;1204,1216p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Hot reload also restarts the heartbeat runner:** Gateway hot reload calls `heartbeatRunner.updateConfig(nextConfig)` when the reload plan requests it. This matches the issue log pattern where Feishu restart activity is followed by `[heartbeat] started`.
  - file: [src/gateway/server-reload-handlers.ts:130](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/gateway/server-reload-handlers.ts#L130)

The websocket handshake and later request handling also close clients with `gateway auth changed` when the generation is stale.
