## Summary

Keep this issue open. Current main has useful pieces: subagent run records, completion announce delivery, persisted task records, task `state_changes` notification policy, and plugin hooks for spawn/end. It still does not implement the central request: framework-owned immediate user-facing spawn acknowledgements, native per-run progress lifecycle notifications for subagents, or a durable idempotent announce queue that survives gateway restart/channel outage independently of the parent session lane.

## Best Possible Solution

Keep this open as the umbrella runtime architecture request. The best path is to extend the existing subagent run registry, task ledger, and completion announce flow with runtime-owned lifecycle events for spawned/running/progress/completed/failed/cancelled, a framework-generated immediate requester-channel spawn acknowledgement, opt-in or policy-driven progress delivery that is not blocked by the parent session lane, and durable idempotent announce persistence/retry. Coordinate with #22358 for completion hook work and #54488 for session-lane starvation, but do not close this until the broader lifecycle delivery contract ships.

## Risks / Open Questions

- Closing this now would hide remaining user-visible gaps: immediate spawn acknowledgement, subagent progress delivery, and durable announce retry are still missing from current main.
- Any future channel-direct lifecycle delivery should stay limited to framework status events and continue using normal routing/idempotency safeguards, not become a bypass for arbitrary model output.
- The implementation should extend the existing subagent completion announce path and task ledger rather than replacing them wholesale, because they already cover completion routing, task records, and partial retry/resume behavior.
