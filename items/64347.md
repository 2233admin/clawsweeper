## Summary

Keep open. Current main still creates a fresh native subagent session key for every `sessions_spawn`, and neither the tool schema nor the subagent config exposes a warm-reuse/pool/reuse-main option. The related merged PR #64468 addressed bootstrap-marker persistence after clean `sessions_yield`; it was explicitly related to #64347 but did not add a warm session reuse path.

## Best Possible Solution

Keep this issue open as a real core/session-orchestration performance request. The best path is a maintainer-designed native `sessions_spawn` reuse mode, likely opt-in first, with locking, active-run checks, cleanup/archive behavior, docs, config/schema updates, and regression tests proving repeated spawns can reuse a safe warm session without breaking isolation or concurrent delegation.

## Risks / Open Questions

- A warm-reuse implementation needs explicit isolation, privacy, cleanup, and concurrency semantics; a naive stable key could mix unrelated tasks or collide with active child runs.
- Exact provider cost savings were not live-reproduced in this read-only review, but the requested reusable session-key path is plainly absent from current source/docs/config.
