## Summary

Current main still does not implement a warm-reuse path for native `sessions_spawn`. The native spawn code still creates a UUID-backed child session key for each spawn, the tool/config schemas expose no reuse key or native pooling option, and docs continue to describe native sub-agents as isolated `agent:<agentId>:subagent:<uuid>` sessions. Related #64468 fixed clean `sessions_yield` bootstrap marker persistence, and newer #72331 only made ACP-only fields tolerated/ignored for native sub-agent runtime; neither is a native warm-session reuse contract.

## Best Possible Solution

Keep this issue open. The maintainer path is to design an explicit native sub-agent reuse contract around `src/agents/subagent-spawn.ts`, `src/agents/tools/sessions-spawn-tool.ts`, subagent config/schema, and docs. The design should choose opt-in reuse versus automatic pooling, document isolation/history tradeoffs, handle concurrency locking, define archive/cleanup and follow-up routing semantics, and add focused tests for native `runtime="subagent"` reuse. Treat #64468 and #72331 as related cache/runtime cleanup, not as canonical resolution.

## Risks / Open Questions

- A native warm-reuse implementation could mix unrelated child history unless reuse is explicit, scoped, and documented.
- Automatic pooling needs locking/allocation semantics so concurrent spawns do not reuse an active child session unsafely.
- Archive/cleanup behavior and follow-up routing need a defined contract for reused native child sessions.
