## Summary

Current `main` does not show a fix for Slack DM history hydration with `userToken`. The inbound Slack monitor still runs on a bot-token Bolt client, DM context assembly excludes the room-history path entirely, and the code/docs still limit user-token preference to action/directory reads rather than inbound session hydration.

## Evidence

- **Monitor runtime is still bot-token based:** Slack monitor startup computes `resolveToken = account.userToken || botToken` only for auxiliary resolution, but the actual Bolt app and inbound client are created with `botToken`. That means inbound history/thread reads still flow through the bot-token client path.
  - file: [extensions/slack/src/monitor/provider.ts:170](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/slack/src/monitor/provider.ts#L170)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Inbound monitor context carries only botToken:** `SlackMonitorContext` stores `botToken` but no `userToken`, and helper reads like `conversations.info` explicitly pass `params.botToken`. This leaves no user-token-backed read path in the monitor context used for inbound session prep.
  - file: [extensions/slack/src/monitor/context.ts:34](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/slack/src/monitor/context.ts#L34)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **DM context assembly skips the history path:** `prepareSlackMessage` only builds pending history and `InboundHistory` when `isRoomish && ctx.historyLimit > 0`. Direct-message turns do not enter that path, so DM session context is not hydrated from Slack message history here.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:594](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/slack/src/monitor/message-handler/prepare.ts#L594)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Thread/history hydration still uses bot-token monitor client:** Thread starter media hydration uses `params.ctx.botToken`, and initial thread history uses `params.ctx.app.client`. Since the monitor client is bot-token-backed, this does not implement the reported user-token DM-history read behavior.
  - file: [extensions/slack/src/monitor/message-handler/prepare-thread-context.ts:130](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/slack/src/monitor/message-handler/prepare-thread-context.ts#L130)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Docs and action runtime still scope user-token preference to read actions, not inbound hydration:** Docs say user token can be preferred for `actions/directory reads`, and the action runtime prefers `userToken` only for explicit read operations. That matches the code split above and does not cover inbound DM session hydration.
  - file: [docs/channels/slack.md:421](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/docs/channels/slack.md#L421)
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Monitor subtree has no DM-history or userTokenReadOnly inbound handling:** A targeted search of `extensions/slack/src/monitor` found no `dmHistoryLimit` references and no monitor-side `userTokenReadOnly` handling; `userToken` appears only in provider setup for auxiliary token resolution.
  - command: `rg -n "dmHistoryLimit|userTokenReadOnly|userToken" extensions/slack/src/monitor -g '!**/*.test.ts'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
