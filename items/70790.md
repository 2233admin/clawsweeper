## Summary

Keep PR #70790 open. Current main has the adjacent toolsAllow forwarding/filtering fix and a /btw-specific empty-tools patch, but it still lacks the embedded-runner stream-resolution payload guard this PR proposes. OpenAI-compatible transport code can still serialize `tools: []` from empty tool contexts or replayed tool history, and related PR #47947 remains open rather than landed as a canonical replacement.

## Best Possible Solution

Keep this PR or a successor open, but rework it against current main as a focused provider-boundary guard: strip `tools: []` and orphaned `tool_choice` only where OpenAI-compatible payloads are serialized, preserve existing custom/provider stream semantics, and add regression tests for boundary-aware, custom, session-http-fallback, Responses, completions, and WebSocket-related cases. Coordinate it with #47947 so one canonical fix covers both empty tools and orphaned tool_choice.
