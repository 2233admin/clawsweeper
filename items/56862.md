## Summary

Keep open. Current `main` does not show a landed fix for the reported builtin-memory case: builtin search only blocks for a completely empty index, while dirty/stale indexes are still refreshed lazily after the search starts.

## Evidence

- **Search only forces sync for empty indexes:** `MemoryIndexManager.search()` synchronously calls `sync({ reason: "search", force: true })` only when `hasIndexedContent()` is false. If indexed content already exists, it immediately proceeds and merely schedules async sync via `startAsyncSearchSync(...)`. That matches the report's failure mode for stale/transitional existing indexes.
  - file: [extensions/memory-core/src/memory/manager.ts:300](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/memory-core/src/memory/manager.ts#L300)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '286,330p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Builtin managers start dirty on normal runtime startup:** `resolveInitialMemoryDirty()` returns `true` for any non-status manager with memory enabled, so normal runtime managers begin in a dirty state even when indexed metadata already exists.
  - file: [extensions/memory-core/src/memory/manager-status-state.ts:31](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/memory-core/src/memory/manager-status-state.ts#L31)
  - command: `nl -ba extensions/memory-core/src/memory/manager-status-state.ts | sed -n '20,35p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Dirty sync path can require full reindex:** `runSync()` computes `needsFullReindex` from stored meta vs current provider/model/providerKey/chunking/tokenizer and runs a safe/unsafe full reindex when they differ. But that work is only awaited inside `sync()`, not by the normal dirty-index search path above.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:976](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/memory-core/src/memory/manager-sync-ops.ts#L976)
  - command: `nl -ba extensions/memory-core/src/memory/manager-sync-ops.ts | sed -n '972,1026p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Reindex triggers cover the stale-index scenarios from the report:** `shouldRunFullMemoryReindex()` still treats provider/model/providerKey/chunking/tokenizer mismatches as full-reindex triggers, which is consistent with the report's 'works after forced reindex' behavior rather than evidence of a fix on `main`.
  - file: [extensions/memory-core/src/memory/manager-reindex-state.ts:76](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/memory-core/src/memory/manager-reindex-state.ts#L76)

- **Test coverage only proves empty-index bootstrap:** Current tests add coverage for bootstrapping a completely empty index on first search, but there is no nearby proof that search waits for a dirty existing index to finish reindexing before returning results.
  - file: [extensions/memory-core/src/memory/index.test.ts:481](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/memory-core/src/memory/index.test.ts#L481)
