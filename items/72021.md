## Summary

Keep #72021 open. Current main still has the reported behavior: short-term promotion computes a combined signal count from recall, daily, and grounded signals, then uses that combined count for the `minRecallCount` gate and score components. Existing tests even assert that a `recallCount: 0`, `dailyCount: 3` candidate can pass default ranking, so this is not implemented or obsolete.

## Best Possible Solution

Keep the issue open and fix it in `memory-core`: split real recall count from aggregate promotion signals. Use `recallCount` for `minRecallCount` or introduce separately named thresholds for daily/grounded signal support, update both rank and apply gates, align `promote-explain` and docs, and add regression tests showing daily/session-only entries do not satisfy a real recall gate while legitimate recall-backed entries can pass after score/threshold retuning.
