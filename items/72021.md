## Summary

Keep open. Current main still matches the reported behavior: short-term promotion computes a combined signal count from recall, daily, and grounded signals, uses that combined count for the `minRecallCount` gate and score averaging, and current tests explicitly expect daily-only signals to clear the default promotion gates. This is a real memory-core product semantics question, not implemented, not out of scope, and not safely auto-closeable.

## Best Possible Solution

Keep the issue open for memory-core maintainer judgment. If accepted, split the semantics so `minRecallCount` gates on real `recallCount`, keep combined daily/grounded support as a separate frequency/support metric or rename it clearly, recalibrate promotion scoring/defaults for recall-only entries, and update the existing daily-only promotion tests and user-facing docs to reflect the intended contract.
