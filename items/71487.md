## Summary

Keep #71487 open. Current main still has TTS backend/config/docs and a per-message browser-native Read aloud button, but no first-class Control UI Settings section that reflects/persists TTS enabled state, provider, default voice, or sample playback. The only issue discussion is an existing maintainer-authored Codex review that also explicitly kept this open for the same missing UI surface.

## Best Possible Solution

Keep the issue open and implement a first-class Control UI Settings Text-to-Speech panel. It should use existing tts.status, tts.providers, tts.enable, tts.disable, tts.setProvider, and tts.convert where possible; show configured versus unavailable providers; filter voices by selected provider; add sample playback; define a safe default voice persistence path; and cover the UI/RPC behavior with focused tests.
