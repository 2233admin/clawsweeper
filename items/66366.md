## Summary

Keep open. Current `main` still has the reported path: backgrounded exec exits enqueue raw untrusted summaries, those summaries are formatted as `System (untrusted): ...`, and exec-event heartbeats are explicitly prompted to relay that output back to the chat surface. There is a config workaround to suppress alert delivery, but no evidence on `main` that Discord now contains/sanitizes this by default or keeps it attached to the exec card path.

## Evidence

- **Background exec exits still enqueue raw `Exec ... :: output` system events:** `maybeNotifyOnExit()` builds `Exec ${status} (... ) :: ${output}` and enqueues it as an untrusted system event for the session, then requests an `exec-event` heartbeat.
  - file: [src/agents/bash-tools.exec-runtime.ts:279](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/agents/bash-tools.exec-runtime.ts#L279)

- **Queued untrusted events are materialized as `System (untrusted): [timestamp] ...` lines:** When system events are drained into the next run context, untrusted entries are prefixed with `System (untrusted)` and a timestamp, matching the issue’s visible message shape.
  - file: [src/auto-reply/reply/session-system-events.ts:85](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/auto-reply/reply/session-system-events.ts#L85)
