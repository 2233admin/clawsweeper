## Summary

Keep #69196 open. Current main still implements the Control UI microphone dictation button with browser SpeechRecognition/webkitSpeechRecognition, which is the exact path the reporter narrowed to `ERROR not-allowed` on Chrome/macOS despite granted microphone permission. The same-origin Permissions-Policy fix is present, but it does not add a fallback or visible error handling for this Web Speech denial.

## Best Possible Solution

Keep the issue open and fix the Control UI dictation path directly. A good resolution would add a robust fallback from browser audio capture to a gateway/provider STT path, or at minimum detect and surface SpeechRecognition `not-allowed` with actionable UI/docs so users are not left with a silent stop. Keep the same-origin Permissions-Policy header, but do not treat it as sufficient for this Web Speech failure.
