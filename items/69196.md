## Summary

Keep open. The issue is recent, non-maintainer-authored, and has concrete follow-up debugging showing Chrome's Web Speech API returns `not-allowed` on the local Control UI origin even though ordinary microphone permission is granted. Current `main` still implements the composer microphone as browser `SpeechRecognition`/`webkitSpeechRecognition` dictation, has only the same-origin microphone `Permissions-Policy` header covered by related #46921, and does not provide a plain audio/STT fallback or visible handling for the reported Web Speech `not-allowed` failure.

## Best Possible Solution

Keep the issue open and fix the Control UI dictation path directly. A good product/code path would surface Web Speech `not-allowed` errors in the composer, add a mocked `SpeechRecognition` regression test for denied startup/no transcript, and decide whether to add a fallback that records audio with `getUserMedia`/`MediaRecorder` and routes it through an existing provider STT capability when browser Web Speech is denied on local Chrome origins.

## Evidence

- **Reporter narrowed failure to Web Speech denial:** The issue comments state that the Mac mini Bluetooth microphone works elsewhere, Chrome microphone permission is granted, `window.SpeechRecognition` and `window.webkitSpeechRecognition` exist, but manual `webkitSpeechRecognition().start()` on `http://127.0.0.1:18789` fails with `ERROR not-allowed`. That makes this more specific than the already-addressed same-origin microphone header request in related #46921.
- **Current STT path depends on browser Web Speech only:** `isSttSupported()` only checks for `SpeechRecognition`/`webkitSpeechRecognition`, and `startStt()` constructs that browser recognition object and calls `recognition.start()`; there is no `getUserMedia`, `MediaRecorder`, or provider STT fallback in this dictation path.
