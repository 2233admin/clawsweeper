## Summary

Keep #69196 open. Current main still routes the Control UI microphone dictation button through browser SpeechRecognition/webkitSpeechRecognition, and the UI still clears STT errors without showing the reported not-allowed failure. The same-origin Permissions-Policy fix is present, but the reporter specifically narrowed this to Chrome Web Speech denial after microphone permission was granted and after a local microphone-header change.

## Best Possible Solution

Keep the issue open and fix the Control UI dictation path directly. A good resolution would add a robust fallback from browser audio capture to a gateway/provider STT path, or at minimum detect and surface SpeechRecognition not-allowed with actionable UI/docs so users are not left with a silent stop. Keep the same-origin Permissions-Policy header, but do not treat it as sufficient for this Web Speech failure.
