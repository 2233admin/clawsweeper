## Summary

Keep #70879 open. Current main still sends Feishu text/card/media messages through direct SDK create/reply calls, then immediately throws on non-zero API responses. I found Feishu-specific backoff detection only for typing-indicator reactions, not outbound message delivery, and existing generic retry helpers are available but not wired into the Feishu send path.

## Best Possible Solution

Keep the issue open and fix it in the bundled Feishu plugin, likely by adding a Feishu-owned bounded retry wrapper around message create/reply/card/media send operations. The wrapper should classify Feishu rate-limit/flow-control responses and thrown SDK/Axios shapes, respect retry-after hints when present, use exponential backoff with a small attempt cap, and add Feishu send tests proving retry on known codes and no retry on permanent errors.

## Risks / Open Questions

- Retrying outbound sends can duplicate messages if Feishu accepted a request but returned an ambiguous transport or internal error. The fix should use a strict Feishu rate-limit predicate and avoid broad regex retries for non-idempotent sends.
