## Summary

Keep #70879 open. Current main still lacks bounded retry/backoff in the Feishu message send path: text/card sends call the Feishu SDK once, API-level nonzero responses are converted directly to errors, and the existing tests explicitly preserve re-throw behavior for a rate-limit-shaped SDK error. The only Feishu backoff handling I found is for typing-indicator reactions, not outbound message delivery.

## Best Possible Solution

Keep the issue open and implement bounded Feishu outbound retry/backoff in the bundled Feishu plugin. The likely fix belongs around sendReplyOrFallbackDirect/sendFallbackDirect and card/text/media send helpers, using the existing retryAsync surface, recognizing thrown and non-throwing Feishu response codes such as 11232, 2200, 429, and known Feishu quota/rate-limit codes, honoring Retry-After when available, and adding unit tests for both reply/create/card paths and exhausted retry behavior.
