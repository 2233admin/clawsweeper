## Summary

Keep PR #69954 open. Current `main` still trusts the persisted `entry.sessionFile` path during `sessions cleanup --fix-missing` and deletes the store entry when that path is missing; it does not check the canonical `<sessionId>.jsonl` fallback described in #50248. The PR is a focused bug fix for a real remaining cleanup safety issue, though its bot/review feedback about repair-only `wouldMutate` accounting should be addressed before merge.

## Best Possible Solution

Keep the PR or an equivalent fix open. The best product path is to make cleanup check the canonical session-id transcript when a persisted `sessionFile` is missing, repair stale metadata when it finds that canonical transcript, and update summary accounting/tests so repair-only cleanup is visible to JSON callers.
