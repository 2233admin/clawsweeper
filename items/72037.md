## Summary

Keep #72037 open. Current main has moved the bundled ACPX runtime to `acpx@0.6.1`, but the default OpenClaw Codex ACP path still generates a wrapper that shells out through `npm exec --package @zed-industries/codex-acp...` or `npx`, and OpenClaw still does not bundle or stage `@zed-industries/codex-acp` as a runtime dependency. The related #60693 is broader and explicitly points to #72037 as the focused tracker; #71549 was closed unmerged and only covered the acpx version bump, not the reported npm/npx fallback.

## Best Possible Solution

Keep #72037 open as the focused tracker for the Codex ACP packaging regression. The likely fix belongs in the bundled ACPX/OpenClaw wrapper path: stage or depend on `@zed-industries/codex-acp` and its platform package so the generated Codex wrapper can launch a verified plugin-local binary directly, while preserving the isolated `CODEX_HOME` bridge and model/thinking override behavior. Add focused tests that the default Codex ACP command no longer resolves to `npm exec` or `npx`, plus package/lock/docs updates for the staged adapter dependency.
