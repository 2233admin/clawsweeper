## Summary

Current main still has the generic exec-approval forwarding fallback with `/approve <id>` placeholder text, so this PR is not obsolete. The PR is still useful, but as submitted it needs rework against current main because current approval rendering now has allowed-decision filtering, shared action descriptors, and native channel renderers that the diff must preserve.

## Best Possible Solution

Keep this PR open and rework it against current main. The implementation should update the generic fallback in `src/infra/exec-approval-forwarder.ts` to render concrete per-decision `/approve <actual-id> <decision>` lines, preferably through `buildExecApprovalActionDescriptors` or equivalent shared logic, while preserving allowed-decision filtering, markdown/plain-channel behavior, and native Telegram/shared interactive rendering. Add focused coverage in `src/infra/exec-approval-forwarder.test.ts` for markdown-capable fallback, non-markdown fallback, and `ask=always` without `allow-always`.

## Risks / Open Questions

- As submitted, the PR can advertise `allow-always` in fallback text even when current main disallows that decision for `ask=always`.
- The implementation should use or mirror the shared action-descriptor path so fallback text, native channel renderers, and allowed-decision filtering do not drift.
- The security-sensitive risk is functional rather than supply-chain: misleading approval text could steer users toward an unavailable decision, but the provided diff does not broaden CI, dependency, secret, install, or release-script surfaces.
