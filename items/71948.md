## Summary

Keep open. Current main still has the exact 30-minute Discord inbound-worker timeout path and generic user-facing reply described in #71948. Prior related fixes handled older listener, attachment, and gateway websocket timeout classes, but this new report is against v2026.4.24 and asks for richer timeout diagnostics/cancellation around the current inbound worker path, which is not implemented on main.

## Best Possible Solution

Keep #71948 open and implement a focused diagnostics improvement in the Discord plugin inbound worker. The best path is to include structured timeout context in logs and the fallback reply where safe: account id, queue key/session key, channel/message id, effective timeout value and whether it came from default or explicit config, active run count/queue depth if exposed, and the last known phase from reply planning/agent execution. Add regression tests around the timeout fallback text/log payload and avoid broad core changes unless a small plugin-safe queue/status API is needed.

## Risks / Open Questions

- The report combines several possible stall sources: session locking, qmd embedding/search, gateway websocket handshakes, subagent announcement, context overflow recovery, and Discord reconnect churn. A fix should avoid treating all of them as Discord transport failures without preserving underlying run/session state.
- Adding queue depth or stage diagnostics may require extending the Discord worker and possibly shared channel run-state surfaces; plugin/core boundaries should stay clean.
