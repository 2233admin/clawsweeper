## Summary

Keep open. Current main still has the reported Discord inbound worker timeout surface: the timeout is configurable and tested, but the user-facing Discord fallback remains the generic `Discord inbound worker timed out.` and timeout logs still only include channel/message context. Main now carries a resolved session key into timeout reply delivery for routing, but it still does not expose the requested run id, account/session context, queue depth, timeout source, or stuck-stage diagnostics to the Discord reply or timeout log.

## Best Possible Solution

Keep this open as Discord plugin diagnostics/runtime hardening. The likely fix belongs around `extensions/discord/src/monitor/inbound-worker.ts`, `extensions/discord/src/monitor/message-handler.process.ts`, and queue/status helpers: capture account id, queue/session key, timeout source, run id once the agent run starts, and best-known stage state; include safe context in the timeout log and Discord fallback reply; add focused tests for enriched timeout output; add a small queue-depth/status surface only if maintainers want queue depth in the timeout report.
