## Summary

Keep open. Current `main` does not implement native Synology Chat attachment support for this channel. The inbound webhook contract only accepts text fields, the inbound context carries only `body` text, and the docs/outbound path only support URL-based media delivery rather than uploaded Synology attachments.

## Evidence

- **Inbound webhook payload is text-only:** `SynologyWebhookPayload` contains `token`, `user_id`, `username`, `text`, and related metadata, but no attachment/file fields.
  - file: [extensions/synology-chat/src/types.ts:48](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/synology-chat/src/types.ts#L48)
  - command: `nl -ba extensions/synology-chat/src/types.ts | sed -n '40,80p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Webhook parser requires text and extracts no attachments:** `parsePayload` only resolves `token`, `user_id`, and `text`/`message`/`content`, and rejects requests missing `text`. There is no parsing path for image/PDF/CSV attachment metadata.
  - file: [extensions/synology-chat/src/webhook-handler.ts:294](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/synology-chat/src/webhook-handler.ts#L294)
  - command: `nl -ba extensions/synology-chat/src/webhook-handler.ts | sed -n '275,340p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Inbound context passed to replies is body-only:** `SynologyInboundMessage` only carries `body` plus sender metadata, and `buildSynologyChatInboundContext` maps that into `Body`/`RawBody`/`CommandBody` with no attachment objects.
  - file: [extensions/synology-chat/src/inbound-context.ts:5](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/synology-chat/src/inbound-context.ts#L5)
  - command: `nl -ba extensions/synology-chat/src/inbound-context.ts | sed -n '1,80p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Current docs describe only outbound URL-based media:** The Synology Chat docs say 'Media sends are supported by URL-based file delivery' and do not document inbound attachment handling.
  - file: [docs/channels/synology-chat.md:104](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/docs/channels/synology-chat.md#L104)
  - command: `nl -ba docs/channels/synology-chat.md | sed -n '100,130p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Outbound implementation is URL forwarding, not native attachment upload:** The plugin's `sendMedia` calls `sendFileUrl(...)`, and the agent prompt tells users to include a publicly accessible URL so the NAS can download and attach the file.
  - file: [extensions/synology-chat/src/channel.ts:293](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/synology-chat/src/channel.ts#L293)

## Risks / Open Questions

- The issue text is slightly ambiguous about inbound user-uploaded attachments versus outbound bot-sent attachments. Current `main` does not provide native uploaded-attachment handling in either direction; it only supports outbound file URLs.
