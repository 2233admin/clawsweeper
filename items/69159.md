## Summary

Keep open. Current main has related session-lock error classification work, but it does not implement this PR's central timeout wiring: embedded attempts still call `acquireSessionWriteLock` without `timeoutMs`, so the lock wait still falls back to the 10s default even when `openclaw agent --timeout` is higher. The PR also needs rebasing onto the current `SessionWriteLockTimeoutError` naming and the broader #13744 lock-scope/config discussion.

## Best Possible Solution

Keep the PR open, rebase it onto current main, adapt it to the existing `SessionWriteLockTimeoutError`/`OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` shape, pass the resolved run timeout into `acquireSessionWriteLock`, and add a focused regression test proving the embedded attempt forwards that timeout. If public programmatic handling is intended for plugin harness consumers, also export the error class or predicate from `openclaw/plugin-sdk/agent-harness-runtime`. Track lock-scope/configuration follow-up under #13744.
