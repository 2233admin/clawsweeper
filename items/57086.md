## Summary

Keep open. Current `main` still forwards a stored `sessionId` together with a canonical `sessionKey` on the gateway agent path, so the stale-session pinning described in PR #57086 is not implemented on `main`.

## Evidence

- **gateway agent still forwards stored session id after resolving canonical key:** In current `main`, the handler resolves and stores the canonical key, then keeps `resolvedSessionId = sessionId` and forwards both `sessionId: resolvedSessionId` and `sessionKey: resolvedSessionKey` to ingress. That is the exact stale-pinning shape the PR is trying to remove.
  - file: [src/gateway/server-methods/agent.ts:714](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/server-methods/agent.ts#L714)
  - command: `nl -ba src/gateway/server-methods/agent.ts | sed -n '680,760p'; nl -ba src/gateway/server-methods/agent.ts | sed -n '928,944p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **downstream session resolution still treats forwarded sessionId as authoritative:** `resolveSession()` uses `opts.sessionId?.trim()` before any freshness-based reuse of the stored entry. If gateway passes a stale `sessionId`, downstream rollover is bypassed even when a canonical `sessionKey` is also present.
  - file: [src/agents/command/session.ts:242](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/command/session.ts#L242)
