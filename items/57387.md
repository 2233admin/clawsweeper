## Summary

Keep open. Current main has session visibility and agent-to-agent allowlist controls, but it does not implement HMAC/token-based mutual authentication or receiver-side verification for `sessions_send` provenance. The related #50649 hook-system item is adjacent defense-in-depth, not a shipped substitute for cryptographic sender verification.

## Best Possible Solution

Keep this as a core security/design issue. The likely implementation belongs in the gateway/session protocol and `sessions_send` tool path: mint or derive per-session sender credentials, sign source/target/session/idempotency/message/timestamp metadata, verify before accepting `inter_session` provenance or A2A follow-up steps, and add config to require authentication per agent while preserving existing `tools.sessions.visibility` and `tools.agentToAgent` allowlist controls.

## Risks / Open Questions

- A plugin or hook-only solution would not fully address this report unless core/gateway delivery rejects unauthenticated `inter_session` provenance before the receiving agent sees it.
- A real fix needs a compatibility plan for existing unsigned internal task events and transcripts that currently rely on plain `inputProvenance` metadata.
