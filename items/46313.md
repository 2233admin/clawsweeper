## Summary

Keep open. Current `main` has a newer auth-profile architecture that repairs stale OAuth profile references at runtime and moves scheduling state to `auth-state.json`, but it does not persistently prune stale profile entries from `auth-profiles.json`. In fact, a current test explicitly asserts the stale profile remains on disk, so this PR is not implemented on `main` and should not be closed as obsolete.

## Evidence

- **Runtime repair is in-memory merge logic, not persisted pruning:** `mergeAuthProfileStores()` calls `reconcileMainStoreOAuthProfileDrift()`, which rewrites merged `profiles/order/lastGood/usageStats` for runtime use. The helper deletes the legacy profile from the merged runtime store only when the base store lacks it, but this operates on the loaded merged object, not on-disk cleanup.
