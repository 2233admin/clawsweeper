## Summary

Keep open. Current `main` still appears vulnerable to this report: OpenClaw’s ACP bridge/schema path requires `usage_update.used` to be a number, the bundled `acpx` runtime is still pinned to `0.5.3`, and I found no repo-side relaxation or null-handling coverage for `used: null` notifications.

## Evidence

- **OpenClaw bridge emits and types `usage_update` with numeric `used`:** `SessionUsageSnapshot` is declared as `{ size: number; used: number; }`, and the bridge sends `sessionUpdate: "usage_update"` with `used: sessionSnapshot.usage.used` and `size: sessionSnapshot.usage.size`. There is no local nullable path here.
  - file: [src/acp/translator.ts:127](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/acp/translator.ts#L127)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Docs confirm `usage_update` is still part of the current ACP bridge surface:** The current ACP docs say the bridge emits `session_info_update` and best-effort `usage_update` notifications from cached Gateway session snapshots.
  - file: [docs/cli/acp.md:49](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/docs/cli/acp.md#L49)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Bundled ACPX is still on `acpx` 0.5.3 and brings a second ACP SDK version:** `extensions/acpx/package.json` still pins `acpx` to `0.5.3`, while the workspace root depends on `@agentclientprotocol/sdk` `0.19.1`; `pnpm-lock.yaml` shows `acpx@0.5.3` resolves `@agentclientprotocol/sdk` `0.17.1`.
  - file: [extensions/acpx/package.json:7](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/acpx/package.json#L7)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Current ACP SDK schema still rejects nullable `used`:** Inspecting the published `@agentclientprotocol/sdk` `0.19.1` schema shows `zUsageUpdate` requires `size: z.number()` and `used: z.number()`, and `zSessionUpdate` includes `usage_update` via that strict schema. A `used: null` payload would still fail validation on the current main dependency set.
  - command: `curl -L --silent https://registry.npmjs.org/@agentclientprotocol/sdk/-/sdk-0.19.1.tgz | tar -xzOf - package/dist/schema/zod.gen.js | nl -ba | sed -n '2438,2486p'`
- **Published `acpx` runtime still forwards raw `session/update` notifications and only normalizes usage in its own projection layer:** The published `acpx` `0.5.3` runtime forwards `onSessionUpdate(notification)` as raw JSON-RPC `method: "session/update"`, while its separate runtime-event projection treats `payload.used` with `asOptionalFiniteNumber(...)`. That matches the reporter’s suspected schema/projection mismatch and I found no OpenClaw-side fix for it.
  - command: `curl -L --silent https://registry.npmjs.org/acpx/-/acpx-0.5.3.tgz | tar -xzOf - package/dist/runtime.js | nl -ba | sed -n '188,199p;515,528p'`
- **Tests only cover numeric `usage_update` values:** Current ACP translator tests assert `usage_update` notifications with numeric `used` values (`4096` and `1200`). I found no coverage for `used: null` or omitted `used` on this path.
  - file: [src/acp/translator.session-rate-limit.test.ts:369](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/acp/translator.session-rate-limit.test.ts#L369)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
