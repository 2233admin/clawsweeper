## Summary

Current main still has the reported gap: `tools.web.search.provider` is schema-validated as an arbitrary string, plugin-aware config validation does not check it against manifest-declared `webSearchProviders`, and runtime behavior still falls back when config names an unknown provider id. The issue remains actionable in core config/plugin validation rather than closeable as implemented, duplicate, stale, or ClawHub-only.

## Best Possible Solution

Keep this issue open. Add plugin-aware config validation for `tools.web.search.provider` in the config validation path, deriving accepted provider ids from bundled and installed plugin `contracts.webSearchProviders` metadata. Reject or clearly report provider typos, preserve warning-only behavior for stale missing-plugin configs, and add focused tests for built-in providers, manifest-declared plugin providers, invalid typos, and missing-plugin compatibility.
