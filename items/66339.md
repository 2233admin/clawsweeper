## Summary

Keep this issue open. Current main has QMD SQLite contention mitigations, including boot retry, read-only `busy_timeout`, same-process queues, embed file locking, and builtin fallback, but it still does not provide a source-backed fix for the reported gateway-vs-CLI runtime race. Gateway startup and `openclaw memory search` create/use memory managers in separate process paths, while QMD update serialization is a `globalThis` singleton and search only waits for the same manager instance's pending update.

## Best Possible Solution

Keep this open and fix it in the memory-core QMD owner path. The likely maintainer direction is per-agent cross-process coordination around QMD update/embed maintenance and foreground search, such as a shared update lock plus lock-aware bounded wait/skip/read behavior, with a regression test that models gateway background QMD maintenance and a separate CLI `memory search` contending for the same agent index. Preserve the existing same-process queue, read-only `busy_timeout`, embed lock, retry, and builtin fallback as mitigations, but do not treat them as the complete fix.

## Risks / Open Questions

- A future fix should avoid an unbounded cross-process lock that can deadlock or make foreground searches hang; it needs bounded wait, skip/read-only behavior, and stale-lock handling.
- No live macOS/QMD reproduction was run in this read-only review, but current source and tests still show the missing cross-process coordination needed for the reported runtime race.
