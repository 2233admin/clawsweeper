## Summary

Keep open. Current `main` has mitigations for QMD lock contention, but the code and tests still treat `SQLITE_BUSY` during boot/search as an expected runtime condition handled by retries and builtin fallback rather than a full coordination fix. That does not satisfy the issue’s requested behavior of avoiding lock contention during normal runtime.

## Evidence

- **Docs still describe background boot QMD work during normal runtime:** QMD runs `qmd update` + `qmd embed` on boot and periodically, and the docs explicitly say boot refresh runs in the background so startup is not blocked.
  - file: [docs/concepts/memory-qmd.md:52](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/docs/concepts/memory-qmd.md#L52)

- **Default config leaves boot sync asynchronous:** `memory.qmd.update.onBoot` defaults to enabled, while `memory.qmd.update.waitForBootSync` defaults to false; manager startup only awaits boot sync when that opt-in flag is true.
  - file: [src/config/schema.help.ts:1062](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/config/schema.help.ts#L1062)
