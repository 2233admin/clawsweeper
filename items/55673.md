## Summary

Current `main` already contains the webchat delivery/render bridge for session-bound cron agent turns after restart, and the same behavior is present in shipped release `v2026.4.22`.

## Evidence

- **Current-session cron bindings persist as session keys:** `sessionTarget="current"` is normalized into a persisted `session:<sessionKey>` binding, so restart-safe current-session jobs are stored against a concrete session key.
  - file: [src/cron/normalize.ts:540](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/cron/normalize.ts#L540)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Cron agent runs register and execute against the bound session:** Cron isolated/current-session agent runs register the run context with the bound `sessionKey` and pass that same key into the CLI/embedded agent runner, which is the prerequisite for session-scoped chat/session events.
  - file: [src/cron/isolated-agent/run-executor.ts:272](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/cron/isolated-agent/run-executor.ts#L272)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Gateway emits chat finals for session-keyed agent runs even without a chat.send link:** On lifecycle end, the gateway resolves `sessionKey` from the event/run context and emits a final chat event for visible runs even when there is no chat-linked client run. That is the bridge webchat needs for out-of-band cron completions.
  - file: [src/gateway/server-chat.ts:588](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/gateway/server-chat.ts#L588)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Webchat reloads history on final and session.message updates:** The Control UI/webchat handler reloads `chat.history` on final chat events, and also reloads on `session.message` updates when idle or after the active run clears.
  - file: [ui/src/ui/app-gateway.ts:456](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/ui/src/ui/app-gateway.ts#L456)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Regression tests cover transcript fanout and deferred webchat reload:** Tests assert that appended transcript messages broadcast `session.message` to subscribed operator clients, and that deferred `session.message` reloads are replayed after the active run terminates.
  - file: [src/gateway/session-message-events.test.ts:192](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/gateway/session-message-events.test.ts#L192)
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Shipped in the latest release tag:** The same `app-gateway` and `server-chat` logic is present in release tag `v2026.4.22`, so this behavior is already shipped, not only on current `main`.
  - command: `git show v2026.4.22:ui/src/ui/app-gateway.ts | sed -n '430,505p' ; git show v2026.4.22:src/gateway/server-chat.ts | sed -n '585,670p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Close Comment

Closing from a Codex review because this looks implemented on current `main`, and the same bridge is already present in shipped `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). Current-session cron jobs are persisted as concrete `session:<key>` bindings (`src/cron/normalize.ts:540`), cron agent runs register and execute against that bound session (`src/cron/isolated-agent/run-executor.ts:272`), the gateway emits final `chat`/`sessions.changed` updates for visible session-keyed runs even without a `chat.send` link (`src/gateway/server-chat.ts:588`), and webchat reloads `chat.history` on those final/session-message updates (`ui/src/ui/app-gateway.ts:456`). The regression coverage also exercises transcript append -> `session.message` fanout and deferred webchat reload behavior (`src/gateway/session-message-events.test.ts:192`, `ui/src/ui/app-gateway.node.test.ts:724`). If this is still reproducible on `v2026.4.22` or newer, please open a fresh issue with exact gateway logs and the stored cron job JSON so the current delivery path can be compared directly.
