## Summary

Keep #70197 open. Current main still loads the enabled-by-default bundled Bonjour gateway discovery plugin through @homebridge/ciao, and the lockfile still resolves ciao to 1.3.6. The published ciao 1.3.6 NetworkManager has the exact Windows `exec("arp -a | findstr /C:\"---\"")` path without `windowsHide` on a 15s poll. OpenClaw has disable workarounds, and upstream ciao 1.3.7 appears to contain the `windowsHide: true` fix, but current main has not consumed it or added a local patch.

## Best Possible Solution

Keep #70197 open as the canonical OpenClaw bug. The likely maintainer path is to update and stage the bundled Bonjour runtime dependency to `@homebridge/ciao` 1.3.7, or apply an equivalent local pnpm patch if a dependency bump is not acceptable. Then verify on Windows with Bonjour enabled that the gateway no longer spawns visible `cmd.exe`/`conhost.exe` windows during the ciao ARP polling loop. Keep the existing `discovery.mdns.mode="off"`, `OPENCLAW_DISABLE_BONJOUR=1`, and plugin-disable paths documented as workarounds, not as the fix.
