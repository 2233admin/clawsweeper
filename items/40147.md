## Summary

Keep #40147 open. Current main still implements the opposite of the PR's stated thread_id-only behavior: Feishu group session resolution treats either thread_id or root_id as thread context, ORs that into replyInThread, and the dispatcher then forces actual Feishu reply_in_thread delivery whenever threadReply is true. The latest release tag v2026.4.24 has the same routing. Related PR #39507 is not a clean superseding target because it proposes disabled replyInThread suppressing both thread_id and root_id, while #40147 intentionally makes root_id authoritative.

## Best Possible Solution

Keep #40147 open for maintainer review or explicitly consolidate it with #39507 after deciding the intended Feishu root_id semantics. A current-main fix should update extensions/feishu/src/bot-content.ts and extensions/feishu/src/reply-dispatcher.ts together, add regression coverage for thread_id-only disabled groups, root_id in normal groups, topic-group/topic-sender routing, and p2p root_id quote replies, then replace the current test expectation that disabled replyInThread still forces thread replies.
