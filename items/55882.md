## Summary

Keep open. Current main has queueing for inbound follow-up messages, restart recovery for interrupted main sessions, and a durable outbound delivery queue, but I did not find a general durable pending-deliverables/commitment mechanism that would guarantee a promised reply is resumed and sent after arbitrary task switching in the same session.

## Evidence

- **Active-run queueing is for incoming work, not promised outputs:** When a session is already active, queue handling decides whether to abort, enqueue a follow-up, or continue. In interrupt mode it clears the lane and aborts the active run; otherwise it routes later work through follow-up queue handling.
  - file: [src/auto-reply/reply/get-reply-run.ts:657](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/reply/get-reply-run.ts#L657)
  - command: `nl -ba src/auto-reply/reply/get-reply-run.ts | sed -n '640,715p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Queue policy only covers active-run message handling:** `resolveActiveRunQueueAction` only distinguishes `run-now`, `enqueue-followup`, and `drop` for active sessions and heartbeats. There is no separate durable commitment or deliverables state here.
  - file: [src/auto-reply/reply/queue-policy.ts:5](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/reply/queue-policy.ts#L5)
  - command: `nl -ba src/auto-reply/reply/queue-policy.ts | sed -n '1,80p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Follow-up drain batches queued messages while busy:** The follow-up queue drain builds prompts titled `[Queued messages while agent was busy]`, which is inbound backlog processing rather than a persisted queue of promised agent outputs.
  - file: [src/auto-reply/reply/queue/drain.ts:221](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/auto-reply/reply/queue/drain.ts#L221)
  - command: `nl -ba src/auto-reply/reply/queue/drain.ts | sed -n '145,230p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Durable queue is transport-level outbound delivery:** `enqueueDelivery` persists reply payloads before send and `ackDelivery`/`loadPendingDeliveries` recover delivery attempts, but this queue exists after payload creation. It is not a semantic task/commitment ledger for unfinished promised replies.
  - file: [src/infra/outbound/delivery-queue-storage.ts:133](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/infra/outbound/delivery-queue-storage.ts#L133)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Main-session recovery is restart-specific:** `buildResumeMessage()` and `resumeMainSession()` explicitly target turns interrupted by a gateway restart, not arbitrary preemption or task switching without restart.
  - file: [src/agents/main-session-restart-recovery.ts:70](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/main-session-restart-recovery.ts#L70)

- **Docs scope durable tasks to detached/background work:** The automation docs say the background task ledger tracks detached work such as ACP runs, subagent spawns, cron, and CLI operations. The docs do not describe a generic durable queue for ordinary in-session promised follow-ups.
  - file: [docs/automation/index.md:68](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/docs/automation/index.md#L68)
