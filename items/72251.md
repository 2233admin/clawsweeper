## Summary

Keep PR #72251 open. Current main still has the live-backup EOF race surface: backup creation makes one `tar.c()` call over the planned state tree with no volatile-file filter, retry loop, enriched EOF handling, logger notices, or `skippedVolatileCount` result/JSON/summary field. The PR is not superseded by the adjacent #67417 ENOENT race or #67990 broader backup-design proposal, and the author addressed the bot review findings in follow-up commits.

## Best Possible Solution

Keep PR #72251 open for maintainer review. The best path is a rebased, narrowly scoped backup-archiver fix on main that adds a built-in volatile-path filter, tar EOF retry with partial temp archive cleanup, enriched final errors, `skippedVolatileCount` stdout/JSON reporting, runtime log notices, focused regression tests, and any needed docs clarification about skipped live artifacts. Track #67417 and #67990 separately unless maintainers choose to consolidate the backup design work.
