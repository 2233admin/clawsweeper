## Summary

Keep open. Current main still recognizes `python: command not found` as an exec failure, but the final reply-shaping path suppresses ordinary non-timeout `exec`/`bash` failures at default verbosity and can leave assistant success text as the only user-visible reply. Existing tests still assert that default suppression behavior, including `Command not found`. The related open PR #60507 is a complementary destructive-command catalog, not a merged fix for post-exec fail-closed reply behavior.

## Best Possible Solution

Keep this issue open. The likely fix belongs in `src/agents/pi-embedded-runner/run/payloads.ts` and adjacent embedded-run coverage: treat failed `exec`/`bash` results as user-visible mutating failures at default verbosity, append or substitute a concise failed-exec warning for exit 127 / command-not-found, and suppress or override success-looking assistant text when the matching exec action failed. Add regression coverage in `src/agents/pi-embedded-runner/run/payloads.test.ts` or `payloads.errors.test.ts` for `python: command not found` in default local embedded mode. Keep #60507 separate unless it grows into the actual post-exec fail-closed implementation.
