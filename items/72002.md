## Summary

Keep #72002 open. Current main has the generic direct-chat prompt injection and the normal WhatsApp inbound path appears to populate `ChatType: "direct"`, but the report is recent, concrete, and specifically says the compiled trajectory prompt still lacks the delivery-context block. Source inspection also shows `inbound_meta.v2` can report `chat_type: "direct"` even when `sessionCtx.ChatType` is missing, so the observed metadata block is not enough to prove the direct-context gate ran. There is no WhatsApp-specific regression test for the compiled system prompt path.

## Best Possible Solution

Keep the issue open and add a focused regression around a WhatsApp `session.dmScope="per-channel-peer"` direct inbound turn that captures the compiled system prompt. The fix should make direct-context injection use a normalized/effective chat type derived from `sessionCtx.ChatType`, session origin metadata, or the direct session key as appropriate, while preserving webchat/internal suppression rules and adding coverage for WhatsApp, Telegram/Signal-style direct contexts, and the metadata-default-direct edge case.
