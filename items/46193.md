## Summary

Keep open. Current main does not implement the requested per-attempt fallback timeout: `runWithModelFallback` still waits on each candidate directly, the config schema has no `fallbackAttemptTimeoutSeconds` knob, and the main call sites only pass the existing fallback options. A hung primary attempt can still starve later fallback candidates.

## Evidence

- **Fallback loop still blocks on one candidate:** `runWithModelFallback` iterates candidates and directly awaits `runFallbackAttempt(...)` with no per-attempt timeout wrapper or abort-signal option in the public run options, so a never-settling candidate prevents the loop from reaching later fallbacks.
