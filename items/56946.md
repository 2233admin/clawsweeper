## Summary

Current `main` contains the bundled-plugin Jiti loader fix and no longer uses the reported channel-metadata discovery path that re-entered `bundled-plugin-metadata` during chat metadata initialization.

## Close Comment

This looks fixed on current `main`, so I'm closing it.

- Codex review of `src/plugins/loader.ts:442` and `src/plugins/jiti-loader-cache.ts:13` shows bundled-plugin Jiti loads are now created relative to the target plugin module (`jitiFilename: modulePath`), which removes the central-loader resolution path described in this report.
- `src/plugins/loader.jiti-filename.test.ts:50` adds a regression check for that exact behavior.
- The reported re-entry path through chat metadata is also gone on `main`: `src/channels/chat-meta-shared.ts:3` now reads from `src/channels/bundled-channel-catalog-read.ts:54`, which only parses bundled catalog/package metadata instead of calling `plugins/bundled-plugin-metadata`, and `src/plugins/contracts/boundary-invariants.test.ts:230` guards that boundary.
- `CHANGELOG.md:88` records this bundled-plugin Jiti startup-hang fix under `#70073`.

If you still hit a hang on a newer build, please open a fresh issue with the exact commit, whether it is a source or packaged run, and a minimal repro path.
