## Summary

Current `main` still reports dreaming from the deep-only resolver in `memory status`, so a light-only dreaming setup would still render as `off`. The issue remains actionable and should stay open.

## Evidence

- **CLI status still uses deep-only resolver:** `formatDreamingSummary` in the memory-core CLI still calls `resolveShortTermPromotionDreamingConfig`, so the displayed `Dreaming:` line is derived from the short-term promotion/deep config rather than phase-aware status.
  - file: [extensions/memory-core/src/cli.runtime.ts:212](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/memory-core/src/cli.runtime.ts#L212)
  - command: `nl -ba extensions/memory-core/src/cli.runtime.ts | sed -n '200,250p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Short-term dreaming helper is deep-backed:** `resolveShortTermPromotionDreamingConfig` directly wraps `resolveMemoryDeepDreamingConfig` and returns `enabled: resolved.enabled`, so disabling only `phases.deep.enabled` makes the summary read as off even if other phases remain enabled.
  - file: [extensions/memory-core/src/dreaming.ts:377](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/memory-core/src/dreaming.ts#L377)
