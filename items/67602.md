## Summary

Keep open. The report mixes up the delivery provider (`--channel slack`) with the Slack target (`--to channel:C...`), but current `main` still permits that invalid state, can still advertise `last|<channel-id>` in cron help when no plugins are loaded, lowercases/stores arbitrary `delivery.channel` values, and only fails later at outbound resolution with `Unsupported channel`. That makes the item a real CLI UX/validation gap rather than strong close material.

## Evidence

- **Issue discussion already isolated the misconfiguration:** Commenter `ngtk` explained that `--channel` is the delivery provider name (`slack`, `telegram`, etc.) and `--to` is the Slack channel ID, and suggested `openclaw cron edit <id> --channel slack --to C0AT2Q238MQ`. That aligns with the current docs/examples and narrows the report to a CLI guardrail problem rather than a Slack transport failure.
- **Current docs show the intended provider/target split:** The cron docs use `--channel slack --to "channel:C1234567890"` for Slack announce delivery, which matches the commenter’s workaround and shows the supported shape.
  - file: [docs/cli/cron.md:112](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/docs/cli/cron.md#L112)
  - command: `nl -ba docs/cli/cron.md | sed -n '86,116p'`
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **CLI help can still present `--channel` as `last|<channel-id>`:** `getCronChannelOptions()` falls back to `last|<channel-id>` when no channel plugins are loaded, and `cron add` uses that string directly in the `--channel` option help. That is exactly the misleading shape called out in the issue comments.
