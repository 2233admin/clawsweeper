## Summary

Keep PR #70884 open. Current main still lacks the trusted-proxy WebSocket scope-clearing documentation and HTTP-only `x-openclaw-scopes` clarification that this PR adds. Static inspection shows the PR matches current runtime behavior: HTTP helpers parse `x-openclaw-scopes`, while WebSocket authorization uses `connectParams.scopes` and clears self-declared scopes when device identity is missing for trusted-proxy/token/password paths, except the preserved local token/password `allowInsecureAuth` case. The provided PR context also shows the review feedback about `dangerouslyDisableDeviceAuth` wording and the `allowInsecureAuth` exception was addressed in the PR commits.

## Best Possible Solution

Keep the PR open for maintainer review and likely merge as a docs-only clarification. The best path is to update `docs/gateway/trusted-proxy-auth.md` and `docs/gateway/protocol.md` so they explicitly separate HTTP `x-openclaw-scopes` behavior from WebSocket handshake scopes, document device-less trusted-proxy scope clearing and missing-scope symptoms, and present `dangerouslyDisableDeviceAuth` only as a break-glass workaround with the existing local `allowInsecureAuth` token/password exception preserved.
