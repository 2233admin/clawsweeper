## Summary

Keep #69256 open. Current main still lacks a subagent-aware guard in direct cron session cleanup, so the PR is not obsolete or implemented elsewhere. However, the PR should not be auto-closed or considered safely done as-is: maintainer review comments identify a real follow-up requirement, because simply returning when descendants are active can skip `deleteAfterRun` cleanup permanently unless a second-stage cleanup path re-enters after descendants drain.

## Best Possible Solution

Keep the PR open and request a revision: direct cron cleanup should preserve the parent session while descendant runs are active, then re-attempt `sessions.delete` after those descendants drain. Add focused coverage in `src/cron/isolated-agent/delivery-dispatch.double-announce.test.ts` for structured and/or threaded direct delivery with active descendants, including the eventual cleanup path.
