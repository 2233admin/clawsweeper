## Summary

Keep open. The bug is not implemented on current main: direct structured/threaded cron delivery still deletes `deleteAfterRun` sessions without any descendant guard, while the text path has subagent-aware handling. The PR is plausible but needs revision because the review thread flagged missing regression coverage and a second-stage cleanup path; additionally, current tests/source indicate descendant checks for isolated cron runs should use `runSessionKey`, not `agentSessionKey` as shown in the PR patch.

## Best Possible Solution

Keep #69256 open and ask for an updated patch. The best fix should make direct structured/threaded cleanup subagent-aware using the run-scoped descendant key, preserve the parent session while descendants are active, arrange a later cleanup once descendants drain so `deleteAfterRun` still means delete after the run, and add regression tests covering structured and threaded direct delivery with active descendants plus the later cleanup path.

## Risks / Open Questions

- The PR as submitted may fix premature deletion only partially: an early return without a deferred retry can leave one-shot `deleteAfterRun` sessions undeleted after descendants finish.
- The PR patch shown in the provided context appears to count descendants under `agentSessionKey`, but current isolated cron subagent flow uses `runSessionKey`; that can miss descendants for default cron session keys.
- No tests were executed because this was a read-only cleanup review; the decision is based on current source/tests and the provided PR discussion/context.
