## Summary

Keep open. Current main still has the cron/headless exec-approval denial, but the denial is still converted into a normal tool-error result for the model loop rather than a cron-run fatal exit. I did not find a current run-level fast-fail path that guarantees no further provider calls after the first denial.

## Best Possible Solution

Keep the issue open and implement a cron/headless approval-denial terminal path. The likely fix belongs around the embedded run/tool boundary: classify `Cron runs cannot wait for interactive exec approval` or a structured exec-denial details code as fatal for cron, abort the current agent loop before another provider call, mark the cron execution `error`, and surface a compact structured summary with command, host, agent, effective security/ask/askFallback, and policy source. Add regression coverage that a scheduled cron exec denial produces one provider call at most, does not call `exec.approval.waitDecision`, and records a failed cron run.
