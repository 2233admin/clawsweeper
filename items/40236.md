## Summary

Current main still does not implement the requested cron job-level lifecycle contract. It has payload-level `timeoutSeconds`, one-shot `deleteAfterRun`, failure alerts, run-log/session retention, and plugin hooks with cron context, but not a `lifecycle` block with job-level timeout, max retry policy, cleanup by runs/time/status, or per-job start/complete/failure hook configuration. The related hook-focused issue and open PR are useful context but do not supersede the timeout and cleanup portions.

## Best Possible Solution

Keep this issue open as a product/API design item. The best path is an additive cron lifecycle contract that defines job-level timeout semantics, retry/cleanup policies such as max runs/TTL/status cleanup, and start/complete/failure hook delivery through the existing hook/plugin architecture, with protocol/schema, scheduler, CLI/UI, docs, and regression tests updated together. #37855 and PR #45465 can inform the hook portion, but the timeout and cleanup portions still need explicit implementation.

## Risks / Open Questions

- Running user-defined lifecycle scripts from cron is security-sensitive; any implementation should reuse existing plugin/internal hook trust, path validation, timeout, and isolation rules rather than adding an ad hoc script runner.
- The likely implementation crosses protocol, cron store normalization, scheduler semantics, CLI/UI, docs, and tests, and needs to preserve current `payload.timeoutSeconds`, `deleteAfterRun`, failure-alert, run-log, and session-retention behavior.
