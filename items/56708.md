## Summary

Keep open. Current `main` still treats Slack reactions as queued system events for a later turn, and there is no implemented config or wake path for reactions to trigger an immediate agent turn.

## Evidence

- **Slack reaction handler only queues a system event:** `registerSlackReactionEvents` formats the reaction text and calls `enqueueSystemEvent(...)`; it does not call `requestHeartbeatNow(...)` or any equivalent wake/turn trigger.
  - file: [extensions/slack/src/monitor/events/reactions.ts:45](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/extensions/slack/src/monitor/events/reactions.ts#L45)
  - command: `nl -ba extensions/slack/src/monitor/events/reactions.ts | sed -n '1,140p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **System events are explicitly for the next prompt:** Core system-event queue is documented as a lightweight queue of events to be prefixed to the next prompt, matching the issue's reported current behavior.
  - file: [src/infra/system-events.ts:1](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/infra/system-events.ts#L1)

  - file: [docs/cli/system.md:34](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/docs/cli/system.md#L34)
