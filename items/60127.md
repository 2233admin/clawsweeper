## Summary

Keep open. Current main documents OpenClaw as a single trusted-operator Gateway, with separate Gateways/profiles recommended for stronger isolation or tenants. Existing multi-agent, multi-account, pairing, and plugin controls are useful adjacent features but do not provide shared-server tenant isolation, tenant-scoped RBAC, quotas, or audit attribution. The KinthAI comment is useful external implementation evidence, but it also identifies missing core seams, so this needs maintainer product/security design rather than ClawHub or not-actionable cleanup.

## Best Possible Solution

Keep the issue open for maintainer product/security judgment. The concrete next step is a scoped design/RFC for tenant context propagation through Gateway requests, tenant-aware authorization/RBAC, per-tenant quotas/rate limits, tenant-scoped plugin/agent configuration, audit attribution, and threat-model/docs updates. If maintainers decide shared-server multi-tenancy is out of scope, that should be an explicit manual product decision with docs pointing users to separate Gateway/profile deployments and external managed services such as KinthAI.
