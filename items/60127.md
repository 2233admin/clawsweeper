## Summary

Current main has adjacent multi-agent isolation, operator scopes, control-plane auditing, and shared-inbox hardening, but it still does not implement the requested shared-server tenant model with tenant identity propagation, tenant-scoped RBAC, quotas, plugin enablement, or tenant-attributed audit boundaries. The issue discussion includes useful external implementation context from KinthAI that points to missing core primitives rather than proving this is cleanly plugin-only work.

## Best Possible Solution

Keep this issue open for maintainer product and security triage. If shared-server multi-tenancy is in scope, design tenant identity propagation through Gateway auth/request context and protocol, tenant-scoped state/config/plugins/skills/model auth, per-tenant quotas, tenant-attributed audit logs, and cross-tenant denial tests/docs. If out of scope, maintainers should document that hostile shared-Gateway tenancy is unsupported and split narrower issues only for specific plugin/API seams external managed-service implementations need.

## Risks / Open Questions

- This is broad product and security architecture work; maintainers may choose to explicitly keep shared hostile multi-tenancy out of scope rather than implement it.
- External services such as KinthAI show demand and possible shims, but current docs/source do not prove the existing plugin API is sufficient for clean tenant isolation.
- Current docs use both multi-agent sharing language and strict security disclaimers; any future direction should keep cooperative multi-agent use distinct from adversarial tenant isolation.
