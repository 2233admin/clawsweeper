## Summary

Keep open. Current `main` still does not sanitize bare leading `to=... {json}` tool-call prefixes, so the underlying leak is not clearly implemented away. At the same time, this PR's proposed start-anchored regex is over-broad and would also delete legitimate prose that begins with `to=` and contains JSON, which matches the existing review feedback on the PR.

## Risks / Open Questions

- The original user-visible leak still looks plausible on current `main`, because neither shipped sanitizer path handles raw `to=... {json}` prefixes.
- Merging the PR as written risks silently truncating legitimate assistant replies that start with `to=` and include JSON examples or documentation text.
