## Summary

Keep this issue open. Current main still implements Slack `read` as channel history or thread replies by `threadId`; it does not accept, forward, or filter by exact Slack `messageId`, so the reported thread-reply lookup gap remains.

## Best Possible Solution

Keep this issue open and fix it in the Slack plugin read path. Add `messageId` to Slack `read` dispatch, pass it through `action-runtime`, extend `readSlackMessages` to return/filter the exact Slack timestamp for both `messageId + threadId` and messageId-only channel reads, and add focused tests in the dispatcher, runtime, and helper. If Slack URL parsing feeds this path, also cover `pMESSAGE_TS?thread_ts=THREAD_TS` normalization.

## Risks / Open Questions

- A correct fix should avoid relying only on default reply/history page size; exact Slack timestamps may require constrained API parameters or pagination so older thread replies are not missed.
- If `messageId` equals the thread parent `threadId`, the current helper deliberately drops the parent from thread reads, so the new exact lookup should define and test that case.
