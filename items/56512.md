## Summary

Current main does not contain this PR's two fixes. The node CLI still lacks a registered `start` subcommand even though the daemon implementation exists, and the node-host runner still does not start the browser control server for remote-mode local proxying.

## Evidence

- **Node CLI router still omits `start`:** `src/cli/node-cli/register.ts` registers `run`, `status`, `install`, `uninstall`, `stop`, and `restart`, but there is no `node.command("start")` and no import of `runNodeDaemonStart`.
  - file: [src/cli/node-cli/register.ts:9](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/cli/node-cli/register.ts#L9)
  - command: `nl -ba src/cli/node-cli/register.ts | sed -n '1,220p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Daemon start logic already exists:** `src/cli/node-cli/daemon.ts` exports `runNodeDaemonStart()` and even renders `openclaw node start` in service start hints, so the gap is the router wiring described by the PR.
  - file: [src/cli/node-cli/daemon.ts:58](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/cli/node-cli/daemon.ts#L58)
  - command: `nl -ba src/cli/node-cli/daemon.ts | sed -n '1,260p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **macOS app still invokes `openclaw node start`:** `NodeServiceManager.start()` shells out with `["start"]`, and the macOS test suite still expects the CLI shape `[openclaw, "node", "start", "--json"]`. That expectation does not match the current node CLI router.
  - file: [apps/macos/Sources/OpenClaw/NodeServiceManager.swift:7](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/apps/macos/Sources/OpenClaw/NodeServiceManager.swift#L7)
  - command: `nl -ba apps/macos/Sources/OpenClaw/NodeServiceManager.swift | sed -n '1,220p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Node-host still does not start browser control server:** `src/node-host/runner.ts` loads config, plugin registry, credentials, and starts `GatewayClient`, but there is no import or call to `startBrowserControlServerFromConfig()` before the node host begins serving requests.
  - file: [src/node-host/runner.ts:146](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/node-host/runner.ts#L146)
  - command: `nl -ba src/node-host/runner.ts | sed -n '1,280p'`
  - sha: [6b618f063516](https://github.com/openclaw/openclaw/commit/6b618f063516aa9ba85903aa4565c5a204c055b2)
- **Browser control server is still gateway-owned on main:** Gateway startup launches plugin services in `src/gateway/server-startup-post-attach.ts`, and the browser plugin service is what starts `startBrowserControlServerFromConfig()` with `OPENCLAW_SKIP_BROWSER_CONTROL_SERVER` handling. That path is separate from node-host startup, so remote-mode node-host still has no local browser-control binder on current main.
  - file: [src/gateway/server-startup-post-attach.ts:285](https://github.com/openclaw/openclaw/blob/6b618f063516aa9ba85903aa4565c5a204c055b2/src/gateway/server-startup-post-attach.ts#L285)
