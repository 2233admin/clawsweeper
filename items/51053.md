## Summary

Current `main` does not implement this PR’s fix. `nodes list` still builds its paired inventory solely from `node.pair.list`, while `node.list` already reports the broader merged node catalog used by `nodes status`, so the reported mismatch remains plausible on current `main`.

## Evidence

- **`nodes list` still uses legacy pairing list as its source of truth:** The `list` command fetches `node.pair.list` unconditionally, derives `paired` from that response, and only consults `node.list` when `--connected` or `--last-connected` filters are active. There is no merge path that would include paired nodes visible only through `node.list`.
  - file: [src/cli/nodes-cli/register.status.ts:325](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/cli/nodes-cli/register.status.ts#L325)
  - command: `nl -ba src/cli/nodes-cli/register.status.ts | sed -n '312,420p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **`nodes status` reads the live merged node catalog:** The gateway `node.list` handler builds its response from `pairedDevices`, `pairedNodes`, and `connectedNodes`, then returns `listKnownNodes(catalog)`. That is the data source used by `nodes status`, so it can legitimately show nodes that are not present in `node.pair.list`.
  - file: [src/gateway/server-methods/nodes.ts:707](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/gateway/server-methods/nodes.ts#L707)
  - command: `nl -ba src/gateway/server-methods/nodes.ts | sed -n '704,724p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Merged node entries are marked paired from either device pairing or node pairing:** `buildEffectiveKnownNode` sets `paired: Boolean(devicePairing ?? nodePairing)`, and `createKnownNodeCatalog` includes IDs from both `pairedDevices` and `pairedNodes`. That means `node.list` can report a paired node even when `node.pair.list` has no corresponding paired entry.
  - file: [src/gateway/node-catalog.ts:91](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/gateway/node-catalog.ts#L91)
  - command: `nl -ba src/gateway/node-catalog.ts | sed -n '78,170p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Current CLI tests do not cover the reported mismatch:** The existing `nodes list` test only exercises filtering of entries already returned by `node.pair.list`; it does not cover the case where `node.list` contains a paired node and `node.pair.list` is empty.
  - file: [src/cli/program.nodes-basic.e2e.test.ts:65](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/cli/program.nodes-basic.e2e.test.ts#L65)
