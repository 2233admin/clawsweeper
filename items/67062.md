## Summary

Keep this issue open. Current main covers the reporter's example host class through the fixed `*.qq.com.cn` default, but it still does not expose or wire a supported `channels.qqbot.mediaAllowHosts` setting. QQ Bot media downloads still use a hardcoded SSRF hostname policy, while Microsoft Teams has the configurable `mediaAllowHosts` pattern the reporter cited.

## Best Possible Solution

Implement the feature in the bundled QQ Bot plugin: add `mediaAllowHosts` to the QQ Bot account/channel config type, Zod schema, plugin manifest, generated metadata, and docs; normalize configured hosts through the existing SSRF allowlist helpers; merge them with conservative QQ/Tencent defaults before inbound and outbound fallback media downloads; and add focused tests for defaults, custom host merging, HTTPS-only behavior, and continued private-network rejection.
