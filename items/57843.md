## Summary

Keep open. Current `main` still returns a bare `OutboundDeliveryResult[]` from outbound delivery, so hook-cancelled sends remain indistinguishable from transport failure and the PR’s core behavior is not implemented on `main`. Several current callers still treat an empty result array as failure/retryable state.

## Evidence

- **Outbound API unchanged on main:** `deliverOutboundPayloads` still returns `Promise<OutboundDeliveryResult[]>`, and the core loop simply `continue`s when a `message_sending` hook cancels a payload, so all-hook-cancelled deliveries still collapse to an empty array with no cancellation metadata.
  - file: [src/infra/outbound/deliver.ts:644](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/infra/outbound/deliver.ts#L644)

  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Gateway send still treats empty delivery as failure:** `send` still does `const results = await deliverOutboundPayloads(...)` and throws `No delivery result` when `results.at(-1)` is missing. That means an intentional hook cancellation still maps to gateway failure instead of a distinguished success/no-op outcome.
