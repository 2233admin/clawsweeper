## Summary

Keep open. Current main has useful mitigations for Gateway memory growth, but the central reported problem is still not proven fixed: session store clone/cache paths remain, transcript/history Gateway paths still materialize full files before limiting, and there is no multi-day RSS plateau evidence for the reported cron/channel/subagent workload.

## Best Possible Solution

Keep this issue open as the canonical broad Gateway RSS/session-accumulation tracker. The best maintainer path is to land or supersede the focused #45438 structuredClone/cache work, avoid retaining both serialized and cloned session-store state where possible, stream or tail/cap Gateway transcript reads before materializing full histories, preserve the shipped seqByRun and session-maintenance mitigations, and add targeted soak or regression evidence showing long-running Gateway RSS plateaus under cron jobs, channel traffic, and subagent sessions.
