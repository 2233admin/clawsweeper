## Summary

Keep open. Current main does not contain the linked PR’s claimed doctor Phase A change: `doctor` still gates daemon repair on `healthCommand`, and non-interactive doctor still gives that path a 3000 ms budget. The status paths also still rely on WebSocket probe success rather than treating a post-connect timeout/1006 as non-authoritative when local HTTP readiness is alive.

## Best Possible Solution

Keep #64400 open and fix the current doctor/status paths directly: use a lightweight liveness/status or readiness probe for doctor’s daemon-repair gate, avoid treating post-connect timeout/1006 as authoritative when local readiness is alive, and add focused regression tests for non-interactive 3000 ms behavior and loopback/no-shared-auth probe handling.
