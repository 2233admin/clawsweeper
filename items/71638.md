## Summary

Keep #71638 open. Current main still does not surface Telegram forward, via-bot, or reply provenance on MessageReceivedHookEvent.context.metadata. The metadata escape hatch exists and Telegram already derives some reply/forward context for prompts, but the canonical received-hook context and mappers only carry generic routing/sender metadata, and there is no Telegram via_bot/viaBot handling in the hook path.

## Best Possible Solution

Keep the issue open and implement a Telegram-owned additive fix: normalize bounded metadata.forward, metadata.viaBot, and metadata.replyTo from inbound Telegram msg, thread it through the canonical received-hook context into internal and plugin message_received mappers, apply it consistently across Telegram received-event emit paths, and add focused mapper plus Telegram emit-path tests.
