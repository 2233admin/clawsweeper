## Summary

Keep open. Current `main` still enforces session-store pruning by `updatedAt`/`maxEntries` without checking `SessionEntry.status`, and save-time enforcement preserves only the single caller-supplied `activeSessionKey`, not other concurrently running sessions. Load-time maintenance also prunes/caps oversized stores without any active/running-session preservation.

## Evidence

- **Enforce-mode save preserves only one active key:** `saveSessionStoreUnlocked()` builds `preserveSessionKeys` from only `opts.activeSessionKey`, then passes that single-key set into both stale pruning and max-entry capping. There is no scan for `entry.status === "running"` or any broader active-session preservation.
  - file: [src/config/sessions/store.ts:284](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/store.ts#L284)
  - command: `nl -ba src/config/sessions/store.ts | sed -n '240,320p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Pruning logic ignores session status:** `pruneStaleEntries()` deletes entries solely by `updatedAt < cutoff` unless the key is explicitly preserved, and `capEntryCount()` sorts/removes by recency alone. Neither function checks `SessionEntry.status`.
  - file: [src/config/sessions/store-maintenance.ts:165](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/store-maintenance.ts#L165)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Load-time enforcement has no active/running preservation:** `loadSessionStore()` applies `pruneStaleEntries()` and `capEntryCount()` whenever an enforced store exceeds `maxEntries`, with no `preserveKeys` and no status-aware exemption. An oversized store can therefore drop running sessions during load/restart.
  - file: [src/config/sessions/store-load.ts:131](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/store-load.ts#L131)

- **Running status is persisted and meaningful:** `SessionEntry.status` includes `"running"`, and restart recovery code relies on persisted `status === "running"` entries. That makes the lack of status-aware maintenance meaningful rather than theoretical.
  - file: [src/config/sessions/types.ts:154](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/types.ts#L154)

- **Current tests only cover one active key, not multiple running sessions:** The pruning tests exercise `getActiveSessionMaintenanceWarning()` with a single `activeSessionKey` and verify capping/pruning by recency, but there is no coverage for preserving multiple `status: "running"` sessions under `maxEntries` pressure.
  - file: [src/config/sessions/store.pruning.test.ts:88](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/config/sessions/store.pruning.test.ts#L88)

## Risks / Open Questions

- Concurrent subagent sessions can still be evicted when another session save triggers enforce-mode maintenance and `maxEntries` is exceeded.
- Gateway restart or any oversized-store load path can prune/cap running sessions because load-time maintenance does not preserve active entries.
- There is no targeted regression test proving that multiple `status: "running"` sessions survive count-based or age-based maintenance.
