## Summary

Keep open. Current `main` does not implement the reported `</arg_value>` suffix defense, and the PR as submitted is not safe to close out: its record-wide stripping approach would also mutate legitimate `write`/`edit` payload text, and it targets an older normalization shape that no longer exists on `main`. This still needs a rebased, narrower fix rather than cleanup closure.

## Evidence

- **Current main has no arg-suffix sanitizer in tool param validation:** `src/agents/pi-tools.params.ts` on `main` only contains required-param validation helpers and `wrapToolParamValidation`; there is no `stripXmlArgValueSuffix`, no XML suffix regex, and no parameter-normalization pass that would remove `</arg_value>` artifacts.

There is no sanitization step for `command`, `workdir`, `host`, `security`, `ask`, or `node` before execution-path logic runs.

## Risks / Open Questions

- The underlying Qwen/DashScope-style corruption still appears unhandled on current `main`, so affected providers may still break `exec` and file-path tools.
- Closing this PR would discard useful signal even though the submitted implementation needs rework rather than cleanup closure.
- Any eventual fix should be scoped to command/path-like fields in the current codepath; broad string sanitization risks corrupting legitimate file contents and edit replacements.
