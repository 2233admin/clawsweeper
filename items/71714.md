## Summary

Keep #71714 open. Current main has an ordinary Enter-to-send path, but the issue is recent, concrete, and Windows Chrome/Edge-specific, and the code still has an IME/composition guard that can bypass sending and allow the textarea default newline. There is no focused keydown regression coverage or shipped configurable send shortcut, so this should not be closed as implemented or non-reproducible from source inspection alone.

## Best Possible Solution

Add focused Control UI composer tests for ordinary Enter sending, Shift+Enter newline behavior, and IME/composition Enter not sending. Then verify the Windows Chrome/Edge path from the report and either refine the composition/keyCode 229 guard or land the broader configurable send-shortcut behavior in the WebChat/Control UI settings surface.
