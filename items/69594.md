## Summary

Keep open. Current `main` still has a code path where isolated cron runs can be marked `delivered` when the agent reports a matching `message` tool send, even though OpenClaw skips its outbound delivery path. That matches the reported `jobs.json` status plus missing outbound logs, and I did not find a current-main fix for delivery accounting.

## Evidence

- **Matching message-tool sends bypass cron outbound delivery:** `runCronIsolatedAgentTurn` sets `skipMessagingToolDelivery=true` when the agent reports a `message` tool send matching the resolved cron target, then passes that into delivery dispatch.
  - file: [src/cron/isolated-agent/run.ts:856](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/cron/isolated-agent/run.ts#L856)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '848,910p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Delivered state is set from the skip flag before any outbound send:** `dispatchCronDelivery` initializes both `delivered` and `deliveryAttempted` from `skipMessagingToolDelivery`, so a matching message-tool send is enough to produce `delivered=true` without OpenClaw's direct outbound adapter running.
  - file: [src/cron/isolated-agent/delivery-dispatch.ts:448](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/cron/isolated-agent/delivery-dispatch.ts#L448)
