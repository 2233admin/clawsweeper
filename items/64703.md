## Summary

Current main still misses the behavior this PR targets. `parseConfigValue` only treats plain integers/decimals as numbers, so unquoted exponent notation such as `1e5` falls through as a string on active `/config`, `/mcp`, `/debug`, and CLI MCP set paths. The final PR diff is focused and the old heartbeat review comments are stale against the final file list.

## Best Possible Solution

Keep this PR open for maintainer review. The likely best path is to land the focused `parseConfigValue` exponent-number support with colocated coverage for existing booleans, null, plain numbers, JSON, quoted strings, invalid JSON, and exponent notation, or explicitly document that unquoted exponent notation should remain a string if maintainers prefer the old parser policy.
