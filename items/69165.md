## Summary

Keep open. Current main has adjacent Telegram retry_after support for some send helpers, but it does not implement the requested outbound Telegram per-chat/topic queue, configurable rate/depth limits, or non-blocking queued drain semantics. The cron delivery path also deliberately skips the shared write-ahead delivery queue and can still mark a non-best-effort cron run as error when delivery throws.

## Best Possible Solution

Keep this open and implement it in the Telegram plugin/outbound layer: add a per-account, per-chat/topic outbound scheduler that all Telegram Bot API send paths use, serialize same-chat sends with configurable rate/depth limits, honor Telegram retry_after as the next eligible send time, apply bounded exponential backoff, and make best-effort/cron failure semantics explicit. Update Telegram config schema/docs and add tests for same-chat serialization, different-chat concurrency, 429 retry_after delay, queue overflow/drop policy, and cron bestEffort behavior.

## Risks / Open Questions

- A fix must preserve non-idempotent send safety: retrying unknown post-send network envelopes can duplicate visible Telegram messages, which current tests intentionally guard against.
- The implementation likely needs to cover multiple Telegram outbound surfaces, not just the message tool: cron direct delivery, interactive replies, media sends, stickers/polls/actions, and command/error replies may use different call paths.
