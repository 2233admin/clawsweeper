## Summary

Keep open. Current main has per-call Telegram retry handling that can respect `retry_after`, but the central request remains unimplemented: there is no shared per-chat outbound send queue, no per-chat/topic rate limiter, no queue-depth/drop policy, and important Telegram reply/cron paths still send directly or bypass the write-ahead queue.

## Best Possible Solution

Keep the issue open and implement this inside the bundled Telegram plugin/outbound layer: add a per-account/per-chat-or-topic scheduler that all Telegram outbound sends pass through, applies configurable rate limits and queue depth/drop policy, honors Telegram `retry_after` plus capped exponential backoff, and integrates cleanly with cron best-effort delivery and the existing write-ahead/retry plumbing. Add focused tests for concurrent same-chat sends, different-chat concurrency, 429 `retry_after`, repeated 429 backoff, queue overflow, and cron delivery status semantics.
