## Summary

Keep open. Current main has adjacent Telegram 429 handling, including per-request `retry_after` retry support and a grammY throttler on the long-lived Telegram bot, but it does not implement the requested outbound per-chat send queue, configurable queue depth, non-blocking submit semantics, or shared per-chat pacing for cron/tool/direct sends. The issue is also distinct from the related batching request #42246 and aligns with the still-open local ClawSweeper finding for #53895 about outbound retry-queue gaps.

## Best Possible Solution

Keep #69165 open and implement a Telegram-owned outbound send scheduler that is shared by direct `sendMessageTelegram` sends and bot reply paths where practical. It should serialize by chat/topic, honor Telegram `retry_after`, expose documented `channels.telegram` queue/rate/depth controls, integrate with `bestEffort` and the existing delivery recovery queue, and add focused tests for concurrent same-chat sends, different-chat concurrency, 429 retry_after delays, queue overflow/drop policy, and cron delivery error semantics. Coordinate with #42246 for optional batching and #53895 for transient-send durability rather than closing this as duplicate or implemented.

## Risks / Open Questions

- The existing grammY throttler on the long-lived bot may already reduce some interactive reply bursts; a fix should avoid double-throttling that path while still covering direct CLI/tool/cron sends.
- The existing write-ahead delivery queue and the related open #53895 retry-queue work overlap with durability, so the implementation should define one coherent outbound queue/retry contract rather than adding a second hidden queue.
