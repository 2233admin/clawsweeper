## Summary

Keep open. Current main still follows the reported Webchat add-attachment path: paste, file-select, and drop read files with `FileReader.readAsDataURL()`, store the resulting `dataUrl` in `chatAttachments`, and assign that array into Lit state before `chat.send` runs. The related shipped attachment work covers send/gateway normalization, not the immediate browser-side add failure described here.

## Best Possible Solution

Keep this open as a Control UI Webchat attachment-add bug. Add focused browser/UI regression coverage for selecting, pasting, and dropping a large image through `OpenClawApp`, then move large attachment payloads out of Lit reactive state, for example by storing Blob/object URLs or raw payloads in an id-keyed non-reactive store and converting to base64 only when `chat.send` is built.
