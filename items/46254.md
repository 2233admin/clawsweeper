## Summary

Current main does not support the reported causal path. Docker sandbox containers are intentionally persistent per sandbox scope and pruned on idle/age, while subagent concurrency limits are enforced by the in-process subagent lane and active-run registry rather than by counting running Docker containers.

## Close Comment

Closing after Codex review because the current `main` code does not match the reported failure mode. On current `main`, Docker sandbox containers are intentionally long-lived per sandbox scope rather than auto-removed after each subagent run (`docs/gateway/sandboxing.md:49`, `src/agents/sandbox/docker.ts:524`). Cleanup is handled by sandbox prune/recreate flows, with the documented default prune window at 24h idle / 7d max age (`docs/cli/sandbox.md:151`, `docs/cli/sandbox.md:181`, `src/agents/sandbox/prune.ts:24`). Also, `agents.defaults.subagents.maxConcurrent` is the in-process `subagent` queue-lane cap, not a count of running Docker containers (`docs/tools/subagents.md:328`, `src/process/command-queue.ts:235`). The active-child spawn gate is based on unfinished run records and excludes completed children once their descendants are done (`src/agents/subagent-spawn.ts:629`, `src/agents/subagent-registry-queries.ts:157`, covered in `src/agents/subagent-registry-queries.test.ts:212`). Because of that, the reported “containers stay up -> maxConcurrent slots exhaust” path does not reproduce against current `main`. If you still hit blocked subagent spawns on a newer build, please open a fresh issue with the current version plus `sessions_list`/status output and relevant gateway logs so maintainers can trace the active-run state directly.
