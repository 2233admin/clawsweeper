## Summary

Keep open. Current `main` still returns from `startDocsPiClient()` without closing already-open pipe handles on the `StdoutPipe()` and `StderrPipe()` error paths, so this PR is not implemented on `main` and the report is not obsolete. The underlying leak is real for failures before `process.Start()`, although the PR's `Start()` leak rationale is overstated on Go 1.24 because `exec.Cmd.Start()` closes `parentIOPipes` on its own failure path.

## Evidence

  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Go stdlib shows caller-owned pipe ends remain open until Start/Wait cleanup:** In Go 1.24, `StdinPipe`, `StdoutPipe`, and `StderrPipe` each allocate an `os.Pipe()` and append the caller-owned end to `c.parentIOPipes`. Those are only auto-closed from `Cmd.Start()`'s deferred cleanup when `Start()` is actually entered and fails; returning before `Start()` leaves the acquired parent ends open unless the caller closes them.

## Risks / Open Questions

- The PR description overstates the `process.Start()` branch: Go 1.24's `exec.Cmd.Start()` closes `parentIOPipes` when startup fails, so the actionable leak is the pre-`Start` `StdoutPipe()`/`StderrPipe()` branches.
- There is no current regression test covering low-FD partial initialization failures in `scripts/docs-i18n`, so a final fix should add one or otherwise prove the cleanup path.
