## Summary

Keep open. Current main has Slack Socket Mode reconnect and stale-health mitigations, including the #56508 orphaned-interval shutdown guard and v2026.4.24 provider-activity health changes, but it does not implement the core request in #58519: keepalive/ping-pong handling still runs through Slack Bolt/@slack/socket-mode in the gateway process, the Slack config surface still exposes no client/server ping timeout tuning, and message event handling still awaits the normal inbound agent dispatch path. Related #14248 remains the narrower open timeout-config request, while #58519 covers the broader event-loop-starvation and message-loss class.

## Best Possible Solution

Keep #58519 open as the canonical broad Slack Socket Mode event-loop-starvation issue. The best implementation path is to add a real transport resilience fix in the Slack plugin: expose SocketModeReceiver clientPingTimeout/serverPingTimeout config as the short-term mitigation tracked by #14248, and separately evaluate isolating Socket Mode keepalive/connection ownership from long agent-turn work or adding targeted event-loop yields in known synchronous hot paths. Add regression coverage around SocketModeReceiver option propagation and a simulated delayed/busy dispatch path so reconnect/ack/message delivery behavior is pinned.
