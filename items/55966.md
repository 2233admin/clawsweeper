## Summary

Keep open. Current `main` adds a small-audio guard that prevents later transcription attempts on 0-byte/tiny audio, but the Telegram inbound save path still writes directly to the final inbound media path via `saveMediaBuffer` with no temp-file/rename cleanup path for `ENOSPC`, so the reported storage-side bug is not fully fixed.

## Evidence

- **Telegram inbound media still saves through direct buffer persistence:** `downloadAndSaveTelegramFile` returns `saveMediaBuffer(...)` for both trusted local Telegram files and fetched remote Telegram files, so inbound Telegram media still depends on the generic direct buffer write path.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:188](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/telegram/src/bot/delivery.resolve-media.ts#L188)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '170,240p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Direct buffer persistence is not atomic:** `writeSavedMediaBuffer` writes straight to the destination path with `fs.writeFile(dest, params.buffer, ...)`; `saveMediaBuffer` then uses that helper directly. There is no temp-file-and-rename step or cleanup branch around failed writes here.
  - file: [src/media/store.ts:301](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/media/store.ts#L301)
  - command: `nl -ba src/media/store.ts | sed -n '290,320p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Atomic temp-file pattern exists elsewhere, but not in saveMediaBuffer:** `saveMediaSource` uses `tempDest` plus `fs.rename(tempDest, finalDest)` for URL downloads, which highlights that the inbound buffer path used by Telegram has not been migrated to the same atomic pattern.
  - file: [src/media/store.ts:378](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/media/store.ts#L378)
  - command: `nl -ba src/media/store.ts | sed -n '378,399p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Current main does guard against 0-byte/tiny audio at transcription time:** `assertMinAudioSize` throws `MediaUnderstandingSkipError("tooSmall", ...)` before provider transcription runs when audio is below `MIN_AUDIO_FILE_BYTES`, covering empty files too.
  - file: [src/media-understanding/runner.entries.ts:494](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/media-understanding/runner.entries.ts#L494)

  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)

## Risks / Open Questions

- The issue is only partially mitigated on current main: transcription no longer proceeds on empty/tiny audio, but the inbound media write path still lacks the atomic temp-file behavior requested in the report.
- This review was read-only and did not perform a live ENOSPC reproduction, so the keep-open call is based on current code paths and test coverage rather than rerunning the original failure on disk pressure.
