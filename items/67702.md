## Summary

Keep open. Current `main` does not implement a pre-update same-install gateway stop/block for git-mode updates, and the PR discussion surfaced unresolved correctness issues in the proposed approach rather than evidence that the problem is obsolete.

## Evidence

- **Current main only prepares restart after update work completes:** `updateCommand` waits until after the update result and post-update work, then checks whether the gateway service is loaded and prepares restart follow-up. There is no same-install preflight using `readCommand` and no pre-update `stop` call here.
  - file: [src/cli/update-cli/update-command.ts:1370](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/cli/update-cli/update-command.ts#L1370)
  - command: `nl -ba src/cli/update-cli/update-command.ts | sed -n '1368,1415p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Git update flow still mutates the checkout in place before any restart:** `runGatewayUpdate` runs `build`, `ui:build`, and `openclaw doctor` directly in the git checkout root. That matches the PR's premise that git-mode updates replace/use live install artifacts before the later restart path runs.
  - file: [src/infra/update-runner.ts:1122](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/infra/update-runner.ts#L1122)
  - command: `nl -ba src/infra/update-runner.ts | sed -n '1118,1195p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Main test suite lacks the PR's service-command coverage:** The current `update-cli` test mocks only `isLoaded` and `readRuntime` on the gateway service. It does not exercise `readCommand`/entrypoint matching or pre-update stop behavior, which indicates this PR's behavior is not already covered on `main`.
  - file: [src/cli/update-cli.test.ts:151](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/cli/update-cli.test.ts#L151)

That is evidence to keep the PR open for revision, not to close it as implemented or obsolete.

- The submitted PR likely needs follow-up changes for wrapper-command detection and restart-on-failure/skip handling before it is merge-ready.
