## Summary

Keep open. Current main has basic Telegram `apiRoot` support, but the reported endpoint-switch migration bug is still present: persisted offsets are not scoped to the Bot API endpoint, lower local-server update IDs can still be skipped before dispatch, and `deleteWebhook` 400 `Logged out` is still treated as a fatal setup error rather than a migration/cooldown condition.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin. Persist a normalized `apiRoot` or endpoint fingerprint with update-offset state, ignore/reset stored offsets when the endpoint changes, handle Telegram 400 `Logged out` during local Bot API migration without crash-looping, and add focused regression tests plus user-facing Telegram docs for local Bot API migration and offset reset behavior.

## Risks / Open Questions

- Closing now would leave a concrete migration failure where a valid local Bot API server can receive updates but OpenClaw skips them because the stored official-API offset is higher.
- The fix must preserve replay protection for normal restarts against the same endpoint while resetting or ignoring offsets only when the Bot API endpoint identity changes.
- `Logged out` handling should be narrowly scoped to the local Bot API migration/cooldown case so unrelated Telegram 400 auth or configuration errors are not hidden.
