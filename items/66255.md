## Summary

Keep this open. Current main has partial lazy-load mitigation for idle gateway startup, but the actionable crash surface remains: OpenClaw still pins @mariozechner/pi-coding-agent 0.70.2, the lockfile still includes the native Linux x64 GNU clipboard package through optional dependencies, upstream 0.70.2 still loads @mariozechner/clipboard on Linux display sessions unless TERMUX_VERSION is set, and OpenClaw still imports the top-level Pi package in gateway chat/session and embedded-run paths.

## Best Possible Solution

Keep this issue open. The best path is an OpenClaw-owned or upstream-supported opt-out that prevents @mariozechner/clipboard from loading before any top-level @mariozechner/pi-coding-agent import in gateway/background embedded paths, or a Pi upgrade that makes native clipboard loading explicit and lazy. Add focused regression coverage for Linux gateway request/chat/session/embedded paths with the guard enabled, and document the supported workaround instead of relying on renaming node_modules files or co-opting TERMUX_VERSION.

## Risks / Open Questions

- Current main may no longer crash during completely idle gateway boot because server-methods and embedded runtime imports are more lazy, so a fix should validate idle boot and the first gateway request/chat/session/cron/embedded run separately.
- A global clipboard disable could break intentional desktop clipboard behavior; the guard should be scoped to gateway/background paths or be explicit and documented.
- The native signal was not live-reproduced in this read-only checkout, but the dependency/import path plus two coredump reports and a confirmed TERMUX_VERSION workaround make the issue actionable.
