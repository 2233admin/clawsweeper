## Summary

Keep open. Current main still does not implement the proposed memory-lancedb smart extraction path, and the PR remains a scoped bundled-plugin feature rather than obsolete cleanup. The author addressed part of the early bot feedback, but the current PR still needs functional, integration, and security rework before it is merge-ready.

## Best Possible Solution

Keep this PR open for focused rework in the memory-lancedb plugin. The best path is to rebase onto current main, update the manifest and UI hints for smartExtraction, preserve current auto-capture cursor/live-config behavior, run candidate text through shouldCapture and captureMaxChars before any LLM call, treat existing memories as untrusted prompt data, make extraction failures observable or fall back safely, and add targeted config/parser/hook tests plus docs for the opt-in behavior.

## Risks / Open Questions

- Closing now would discard an active, scoped memory-lancedb feature that current main has not shipped and that has no canonical replacement in the provided related context.
- Merging as-is would likely reject user smartExtraction config because the strict plugin manifest schema is not updated with the new property.
- Merging as-is could bypass shouldCapture and captureMaxChars in the smart path, sending long, sensitive, or prompt-injection-like user messages to the extraction model and potentially storing them.
- Merging as-is could introduce a prompt-injection path by inserting existing stored memories into the extractor prompt as unescaped, user-controlled text.
- Merging as-is could make extraction outages hard to diagnose because chat-completion failures are caught and treated as an empty extraction result.
- The PR diff is based on an older auto-capture shape and needs careful rebase onto current main so live config, cursor tracking, and existing capture behavior are preserved.
