## Summary

Keep PR #59118 open. Current main still lacks the JSON parse error suppression proposed by the PR, and the raw short-error fallback can still surface the reported SyntaxError text. The PR is not ready to merge as-is because the provided discussion includes unresolved maintainer feedback about matcher breadth and misclassification risk, but it remains a focused, plausible fix for a real user-facing leak rather than cleanup-close material.

## Best Possible Solution

Keep the PR open for maintainer or author follow-up. The best implementation path is to classify malformed streaming JSON fragments closer to the transport/streaming layer when possible, then have `formatAssistantErrorText` and `sanitizeUserFacingText` consume that structured classification. If kept as last-mile sanitization, broaden coverage for the JSON.parse SyntaxError family, preserve raw API payload formatting precedence, and add regression tests for the #59076 observed string plus truncated-input and trailing-garbage variants.
