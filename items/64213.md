## Summary

Current main still does not implement the requested runtime fallback from a gateway catalog miss to explicit `models.providers.<provider>.models[].input` metadata. The resolver only accepts a catalog loader plus provider/model, returns false when no matching catalog entry exists, and the chat, agent request, and node-event attachment paths all depend on that result before parsing images. Existing tests still assert fail-closed catalog-miss behavior, while config types and docs continue to support explicit image-capable provider models. The related open PR #67766 is an implementation candidate for the same class of bug via #67759, but it is unmerged in the provided GitHub context and does not close this issue.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review or revise #67766, or land an equivalent focused fix that lets `resolveGatewayModelSupportsImages()` consult normalized explicit provider model metadata on catalog miss. The implementation should cover explicit provider refs, inferred unique configured providers, catalog miss plus config hit, text-only config, catalog-hit precedence, and all current attachment entry points: `chat.send`, agent requests, and node events.
