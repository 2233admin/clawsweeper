## Summary

Keep open. Current main still has Feishu group @mention detection wired around app-scoped open_id only, while the PR is a focused attempt to fix the still-open multi-bot bug in #40768. The PR is not ready to close as implemented or obsolete because main lacks union_id bot identity propagation, and the existing review feedback identifies additional command-stripping and debounce paths that the patch needs to cover before landing.

## Best Possible Solution

Keep the PR or canonical issue #40768 open, then rework the fix against current main: verify the Feishu bot identity API/source for a stable union_id or equivalent, store it through startup/retry/state/account handlers, pass it through receive/debounce/parse paths, update checkBotMentioned, normalizeMentions, mention-forward extraction, and resolveFeishuDebounceMentions to resolve the bot by union_id when open_id is app-scoped, and add focused regression tests for multi-bot group mention detection plus @Bot /model stripping.

## Risks / Open Questions

- The PR patch shown is against an older layout where mention logic lived directly in bot.ts; current main has split shared parsing into bot-content.ts and monitor.message-handler.ts, so it needs a careful rebase rather than a straight close or merge.
- The proposed union_id source needs dependency/API verification: the linked issue says bot/v3/info did not return union_id, while the PR assumes startup can fetch it. Maintainers should confirm the Feishu SDK/API contract before landing.
- Fixing only checkBotMentioned would leave known gaps in slash-command stripping and debounced multi-message dispatch, based on current code and existing review comments.
