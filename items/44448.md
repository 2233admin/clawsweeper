## Summary

Keep open. Current main still resolves `openclaw agent` session selection from `--to`, `--session-id`, explicit `sessionKey`, or `--agent`; the CLI `--channel`/`--reply-account` path is delivery metadata and is not used to look up an existing Discord session by delivery context (`channel` + `to` + `accountId`/thread). The related local review for #44450 reached the same conclusion, and #44448 is the earlier item, so I would not close this as a duplicate of the later report.

## Evidence

- **Reporter-provided runtime evidence identifies the missing lookup:** The issue body and follow-up comments report that local dist patches added `resolveAgentSessionKeyByDeliveryTarget`-style lookup by `channel` + `to` + `accountId`, normalized Discord targets, and warned that those manual compiled-bundle patches would be lost on update. That discussion matches the source gap still visible on current main.
- **CLI computes the session key before and without channel context:** `agentViaGatewayCommand` calls `resolveSessionKeyForRequest({ cfg, agentId, to: opts.to, sessionId: opts.sessionId })`; `opts.channel` is normalized only afterward and sent as a gateway param for delivery. No `channel`, `replyAccount`, or account id is included in session selection.
