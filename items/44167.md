## Summary

Keep open. Current `main` still does not implement retry handling for transient embedding transport failures, and the PR's behavior is not present after the memory-core move into `extensions/`.

## Evidence

- **Current batch embedding path still uses only the generic retry loop:** `embedBatchWithRetry()` on current `main` delegates directly to `runMemoryEmbeddingRetryLoop(...)` with `isRetryableMemoryEmbeddingError` and has no branch for transport failures or recursive batch splitting.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:234](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/memory-core/src/memory/manager-embedding-ops.ts#L234)
  - command: `nl -ba extensions/memory-core/src/memory/manager-embedding-ops.ts | sed -n '210,330p'`

- **Current retry classifier excludes fetch/network failures:** `isRetryableMemoryEmbeddingError()` only matches rate-limit / 5xx style strings (`429`, `5xx`, `cloudflare`, `tokens per day`). It does not match `fetch failed`, `econnreset`, `socket hang up`, or similar transient transport errors described by the PR.
