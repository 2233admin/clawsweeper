## Summary

Keep this issue open. Current main has mitigations for the large model-list path and OpenRouter onboarding is present, but the reported existing-config forced-wizard path still appears in the installer: it detects an existing config, logs that onboarding is skipped, then continues to `Starting setup` and executes `openclaw onboard` when a TTY is available. The prior ClawSweeper comment identified the same remaining behavior, and rechecking current main at edbab0e2 shows it has not been fixed.

## Best Possible Solution

Fix the installer or Umbrel setup entrypoint so an existing `OPENCLAW_CONFIG_PATH` or legacy config exits the onboarding path after doctor/dashboard handling instead of continuing into `openclaw onboard`. Add a focused regression test for the existing-config install path, then verify the packaged Umbrel wizard can filter large model catalogs and select OpenRouter.
