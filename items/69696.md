## Summary

Keep open. Current `main` still performs case-sensitive OpenRouter base-URL normalization in the active OpenRouter, Arcee, and embedded-runner paths, so the uppercase-host `/v1` case this PR fixes is not implemented on `main` and remains user-visible.

## Evidence

- **OpenRouter helper is still case-sensitive on main:** `normalizeBaseUrl` only trims and strips trailing slashes, then `normalizeOpenRouterBaseUrl` compares exact strings. An uppercase host like `https://OPENROUTER.AI/v1` will not match either canonical constant on current `main`.
  - file: [extensions/openrouter/provider-catalog.ts:21](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/openrouter/provider-catalog.ts#L21)
  - command: `nl -ba extensions/openrouter/provider-catalog.ts | sed -n '1,80p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Arcee path has the same exact-string normalization:** `normalizeArceeOpenRouterBaseUrl` uses the same trim-only normalization and exact comparisons, so the same uppercase-host miss exists there too.
  - file: [extensions/arcee/provider-catalog.ts:7](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/extensions/arcee/provider-catalog.ts#L7)
  - command: `nl -ba extensions/arcee/provider-catalog.ts | sed -n '1,80p'`

- **Embedded runner test-support path also remains case-sensitive:** The embedded-runner normalization helper likewise trims only and compares exact strings, matching the PR's third code path.
  - file: [src/agents/pi-embedded-runner/model.provider-runtime.test-support.ts:73](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-runner/model.provider-runtime.test-support.ts#L73)

- **Reasonable repro still shows the bad endpoint difference:** A direct runtime check showed `new URL('https://OPENROUTER.AI/v1').href` becomes `https://openrouter.ai/v1` (host lowercased, path unchanged). Fetching that `/v1` endpoint returned HTML, while fetching `https://openrouter.ai/api/v1/models` returned JSON. That matches the underlying failure mode the normalization is meant to prevent.
  - command: `node -e "console.log(new URL('https://OPENROUTER.AI/v1').href)"; node -e "fetch('https://OPENROUTER.AI/v1/models'...).then(...)"; node -e "fetch('https://openrouter.ai/api/v1/models'...).then(...)"`
- **PR discussion supports the patch as targeted and low-risk:** The existing Greptile review on the PR described the `.toLowerCase()` change as minimal, correctly scoped to comparison only, and backed by new regression tests, with no high-severity concerns noted.

## Risks / Open Questions

- Closing this PR would leave a real normalization gap on current `main` for uppercase-host OpenRouter legacy URLs.
- Current `main` also lacks the added regression coverage, so the edge case can regress silently across OpenRouter, Arcee, and embedded-runner paths.
