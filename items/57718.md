## Summary

Current main already contains explicit WhatsApp creds persistence and backup-recovery hardening for the reported startup false-restore loop, and the changelog shows it was shipped in the 2026.4.18 release line.

## Evidence

- **Restore only happens for unreadable creds:** `restoreCredsFromBackupIfNeeded()` returns early when `creds.json` exists and parses successfully, and only restores from `creds.json.bak` when the primary file is unreadable/corrupt.
  - file: [extensions/whatsapp/src/auth-store.ts:73](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/whatsapp/src/auth-store.ts#L73)
  - command: `nl -ba extensions/whatsapp/src/auth-store.ts | sed -n '73,116p'`
- **Startup waits for queued creds saves before bootstrap:** `createWaSocket()` drains the per-auth creds save queue before restore/bootstrap, specifically guarding the reconnect-time race that can cause false backup restores.
  - file: [extensions/whatsapp/src/session.ts:136](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/whatsapp/src/session.ts#L136)
  - command: `nl -ba extensions/whatsapp/src/session.ts | sed -n '121,146p'`
- **Creds writes are atomic:** `writeCredsJsonAtomically()` writes to a temp file, fsyncs, renames into place, reapplies mode, and syncs the directory, preventing truncated `creds.json` on interrupted saves.
  - file: [extensions/whatsapp/src/creds-persistence.ts:32](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/whatsapp/src/creds-persistence.ts#L32)
  - command: `nl -ba extensions/whatsapp/src/creds-persistence.ts | sed -n '32,59p'`
- **Regression tests cover this failure class:** Tests verify no restore on ordinary reads, restore only from corrupted creds, backups are not clobbered by corrupted creds, and failed atomic renames leave the prior `creds.json` valid.
  - file: [extensions/whatsapp/src/auth-store.test.ts:43](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/extensions/whatsapp/src/auth-store.test.ts#L43)
  - command: `nl -ba extensions/whatsapp/src/auth-store.test.ts | sed -n '43,79p'; nl -ba extensions/whatsapp/src/session.test.ts | sed -n '304,467p'`
- **Release notes match the reported bug:** The changelog explicitly calls out WhatsApp auth persistence and backup recovery hardening in 2026.4.18, plus earlier fixes for false backup restores and atomic `creds.json` writes.
  - file: [CHANGELOG.md:521](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/CHANGELOG.md#L521)

## Close Comment

Current `main` already contains the WhatsApp creds persistence and backup-recovery hardening that matches this report, so this looks fixed.

- Codex review of `extensions/whatsapp/src/auth-store.ts:73`, `extensions/whatsapp/src/session.ts:136`, and `extensions/whatsapp/src/creds-persistence.ts:32` found that startup now only restores from `creds.json.bak` when `creds.json` is unreadable, waits for pending queued saves before reopening the socket, and writes creds atomically through a temp-file + rename path.
- Regression coverage in `extensions/whatsapp/src/auth-store.test.ts:43`, `extensions/whatsapp/src/auth-store.test.ts:53`, `extensions/whatsapp/src/session.test.ts:304`, and `extensions/whatsapp/src/session.test.ts:436` covers the same failure mode: no restore on ordinary reads, restore only from corrupted creds, preserve a good backup, and keep the previous `creds.json` valid if an atomic rename fails.
- `CHANGELOG.md:521` records the shipped WhatsApp auth persistence and backup-recovery hardening in `v2026.4.18`, with related false-restore and atomic-write fixes also called out at `CHANGELOG.md:594` and `CHANGELOG.md:1015`.
