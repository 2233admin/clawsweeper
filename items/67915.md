## Summary

Keep open. Current `main` still has the unconditional `Outside allowed folders` fast-path ahead of the server `meta=1` availability check, so there is no strong basis to say this is fixed. The reporter’s exact startup-race explanation is weakened by newer bootstrap sequencing that waits for the initial config load before first gateway connect, but that is not enough to close the issue as unreproducible or implemented, especially with the linked open PR #67916.

## Evidence

- **UI still hard-denies local attachments before the server-side meta check:** `resolveAssistantAttachmentAvailability()` returns `unavailable` with reason `Outside allowed folders` immediately when `isLocalAttachmentPreviewAllowed()` is false. The async `assistant-media?meta=1` fetch only starts after that check passes, so current `main` still contains the behavior the report called out.
