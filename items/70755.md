## Summary

Keep open. The checked-out `main` still authenticates BlueBubbles webhook requests against `channels.bluebubbles.password`, the public docs still tell operators to publish that password on the webhook URL, and current tests still accept password-based webhook auth. The linked fix exists only in open PR #70757 and is not implemented on this checkout.

## Evidence

- **Runtime still uses API password as webhook auth secret:** `handleBlueBubblesWebhookRequest()` reads `guid`/`password` from the query or headers, then matches it against `target.account.config.password` for the `/bluebubbles-webhook` plugin route.
  - file: [extensions/bluebubbles/src/monitor.ts:185](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/extensions/bluebubbles/src/monitor.ts#L185)
  - command: `nl -ba extensions/bluebubbles/src/monitor.ts | sed -n '45,210p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)

That leaves the BlueBubbles password comparison as the effective network auth check for this surface.
