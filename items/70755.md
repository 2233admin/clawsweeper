## Summary

Current main still authenticates BlueBubbles webhooks with the configured BlueBubbles server/API password, and the docs still publish that same password in the public webhook URL. The linked fix PR #70757 exists but is still open and unmerged, so this is an active bug/security-hardening item rather than conservative cleanup.

## Best Possible Solution

Keep #70755 open and review PR #70757 or an equivalent focused fix. The durable fix should add a dedicated BlueBubbles inbound webhook secret, authenticate webhook requests against that secret instead of the server/API password, update setup/docs/secret metadata, and add regression coverage for rejecting inherited or missing webhook credentials.
