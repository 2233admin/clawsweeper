## Summary

Current main still exhibits the reported BlueBubbles credential-coupling bug: the webhook route is plugin-authenticated, accepts `guid`/`password` query/header tokens, and compares them to `channels.bluebubbles.password`. The BlueBubbles config/schema/secret docs do not expose a separate `webhookSecret`, and the public docs still tell operators to put the same password in the webhook URL. The linked fix PR #70757 is open and unmerged, so this issue should remain open until a fix lands or maintainers choose a canonical tracker.

## Best Possible Solution

Keep #70755 open until a fix lands. The likely product path is to add a distinct BlueBubbles `webhookSecret` credential surface, including per-account support, authenticate inbound webhooks only against that secret, keep `channels.bluebubbles.password` for BlueBubbles API access, and update setup flows, secret metadata, docs, generated config docs, and regression tests together. #70757 is the current proposed fix and should remain the canonical implementation discussion unless maintainers split it or replace it.
