## Summary

Current main still matches the reported failing sequence. The unsafe reindex path calls `resetIndex()`, `resetIndex()` drops `chunks_vec` and clears only `vector.dims`, and `syncMemoryFiles()` still prepares `DELETE FROM chunks_vec ...` whenever `vector.enabled && vector.available` before any later vector-table recreation. The related open PR #71611 addresses the safe-path Windows rename/EBUSY problem, not this sqlite-vec unsafe-reindex crash, and it is not merged in the inspected main checkout.

## Best Possible Solution

Keep #70559 open. Add a focused memory-core regression test for unsafe full reindex with sqlite-vec/vector availability true after `resetIndex()`, then fix the lifecycle by either preserving/purging `chunks_vec` during reset or resetting/recreating vector state before any prepared statements reference the vector table. If #71611 or a successor makes safe reindex reliable on Windows, link it as related, but it should not close this specific unsafe sqlite-vec crash until the product path is explicitly resolved.
