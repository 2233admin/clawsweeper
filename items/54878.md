## Summary

Keep this issue open. Current main still checks generic cross-agent visibility before honoring `tree` spawned-session lineage, so a parent session can still be blocked from reading its own ACP child session when the child key has a different `agent:` id such as `agent:codex:acp:<uuid>`. The issue discussion and closed unmerged PR attempts identify the right narrow fix, but that fix is not present on current main.

## Best Possible Solution

Keep this issue open and implement the narrow lineage fix in `src/plugin-sdk/session-visibility.ts`: for `tree` visibility, allow the requester session and targets present in the spawned set before applying generic cross-agent rejection. Add regression coverage for a parent such as `agent:main:main` reading a spawned `agent:codex:acp:<id>` child with default/tree visibility and no broad allowlist, preferably at both guard and `sessions_history` tool levels. Then review other session-tool callers for any pre-guard cross-agent checks that should also honor proven spawned lineage.

## Risks / Open Questions

- Closing now would leave ACP orchestration half-functional for parent-owned child history retrieval when the child session belongs to a different agent id.
- The eventual fix must only allow proven spawned-tree lineage before generic cross-agent policy; non-spawned cross-agent targets should remain governed by `tools.sessions.visibility` and `tools.agentToAgent`.
- `session_status` has an additional explicit-agent cross-agent gate before the shared guard, so maintainers should check whether the same lineage rule is expected there after fixing the shared guard.
