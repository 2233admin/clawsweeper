## Summary

Keep this PR open. Current main still has no ingress filter for operational alert text before delivery-mirror transcript persistence; it only has downstream replay/display hygiene for delivery-mirror entries. The PR remains a focused implementation candidate, and the author addressed the initial broad Spanish `Error en` false-positive review by narrowing the pattern in the provided PR context. Security review did not show CI, dependency, workflow, lockfile, publishing, or secret-handling changes, but the proposed filter is content-destructive and should receive maintainer review for false positives and ownership placement.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land or revise a narrow, regression-tested filter for non-user-facing operational alerts, or move suppression closer to the backup/git alert producer if maintainers do not want owner-specific strings in the generic transcript mirror layer. Broader delivery replay and duplicate work should remain tracked separately by #51628.

## Risks / Open Questions

- The alert strings look backup/git-owner-specific; maintainers should decide whether suppression belongs in the generic transcript mirror resolver or closer to the alert-producing owner boundary.
- Related delivery replay and duplicate behavior remains broader than this PR and is still tracked separately by open issue #51628.
