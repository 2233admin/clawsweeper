## Summary

Keep this issue open. Current main has useful adjacent reliability work for model fallback, LLM idle timeouts, Telegram network hardening, and polling-stall recovery, but it does not fully implement the reported four-agent Telegram scenario where unrelated LLM providers time out together while direct provider curl calls still succeed. Several concrete requested changes remain unresolved or only partially addressed: Telegram update sink concurrency still follows agent maxConcurrent, the LaunchAgent environment does not default UV_THREADPOOL_SIZE, LLM transport does not get a Telegram-style dnsResultOrder=ipv4first setting, and plain run timeouts still synthesize local timeout payloads rather than proving cross-agent failover recovery.

## Best Possible Solution

Keep this issue open as a current reliability investigation. The best maintainer path is to reproduce or instrument a four-agent Telegram workload on current main, measuring Telegram update sink pressure, command-lane wait time, session-lock hold/acquire time, LLM request connect/header/body/idle timing, abort reasons, provider retry behavior, and event-loop/threadpool pressure. Then fix the measured owner layer: decouple Telegram update handling from agent-run concurrency if polling/update processing is starved, add a generic configurable LLM network policy if Undici/DNS is implicated, or adjust timeout/fallback policy if fallback attempts still burn too much time. Add regression coverage for fallback timeout behavior plus a narrow concurrency/queue test that would fail for the reported all-agents-timeout pattern.
