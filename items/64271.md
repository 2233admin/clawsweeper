## Summary

Keep #64271 open. Current main still has the core behavior the issue reports: local managed browser launch failures are not tracked with a per-profile failure counter, cooldown, backoff, or circuit breaker, and at least one current regression test explicitly asserts that a failed lazy start clears the guard and can immediately launch again on the next call.

## Best Possible Solution

Implement the fix in the bundled browser plugin availability layer: add per-profile state for consecutive local managed launch/post-launch-CDP failures, apply bounded exponential backoff or a cooldown short-circuit after a small threshold, reset on successful CDP readiness and relevant profile/config changes, and surface a clear BrowserProfileUnavailableError that includes `browser.noSandbox: true` when applicable plus `browser.enabled: false` when the browser tool is not needed. Add focused tests around repeated ensureBrowserAvailable()/browser.request failures and update browser troubleshooting docs/changelog if the user-facing error changes.
