## Summary

Keep open. Current `main` does not implement this PR’s iOS permission-prompt behavior: Contacts still fast-fails on `.notDetermined`, Calendar still fast-fails on `.notDetermined`/no prompt path, and the iOS app bundle metadata still lacks the Contacts/Calendar usage-description keys this PR adds.

## Evidence

- **Contacts still does not prompt on first use:** `ContactsService.ensureAuthorization` returns `false` for `.notDetermined` with the explicit comment that prompts should not run during `node.invoke`, so fresh-install `contacts.search`/`contacts.add` still fail instead of requesting access.
  - file: [apps/ios/Sources/Contacts/ContactsService.swift:105](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/apps/ios/Sources/Contacts/ContactsService.swift#L105)
  - command: `nl -ba apps/ios/Sources/Contacts/ContactsService.swift | sed -n '101,124p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Calendar still has no permission request flow:** `CalendarService.events` and `CalendarService.add` only call `EventKitAuthorization.allowsRead/allowsWrite`; there is no `requestFullAccessToEvents`, `requestWriteOnlyAccessToEvents`, or other async permission bridge on current `main`. A `.notDetermined` status still throws `CALENDAR_PERMISSION_REQUIRED`.
