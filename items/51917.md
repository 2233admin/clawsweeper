## Summary

Current `main` has restart-sentinel support for a single explicitly-targeted session restart, but it does not implement automatic recovery of all unanswered channel sessions after a gateway restart, and drain-time inbound work is still rejected rather than queued/replayed.

## Evidence

- **Restart sentinel is single-payload, single-session state:** `RestartSentinelPayload` carries one optional `sessionKey`, and the persisted sentinel is a single `restart-sentinel.json` file. The default continuation is only built when a specific `sessionKey` is present.
  - file: [src/infra/restart-sentinel.ts:41](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/infra/restart-sentinel.ts#L41)
  - command: `nl -ba src/infra/restart-sentinel.ts | sed -n '30,120p'`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Startup recovery consumes one sentinel, not all interrupted/unanswered sessions:** Gateway startup calls `consumeRestartSentinel()` once, creates at most one startup task, and resumes only the sentinel's single `sessionKey`. There is no scan of session transcripts or channel sessions for unanswered messages.
  - file: [src/gateway/server-restart-sentinel.ts:307](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/gateway/server-restart-sentinel.ts#L307)
  - command: `nl -ba src/gateway/server-restart-sentinel.ts | sed -n '300,470p'`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Restart continuation must be provided by the initiating session/tool:** The gateway tool's restart flow asks callers to pass `sessionKey`/`continuationMessage` and writes a targeted sentinel from that context. This is scoped to the current task, not automatic resume of arbitrary interrupted channel sessions after restart.
  - file: [src/agents/tools/gateway-tool.ts:356](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/agents/tools/gateway-tool.ts#L356)
  - command: `nl -ba src/agents/tools/gateway-tool.ts | sed -n '350,410p'`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Drain window still rejects new work instead of queueing/replaying it:** During restart, the run loop calls `markGatewayDraining()`, and queue enqueues fail with `GatewayDrainingError`. The reply path converts that into a user-visible 'Gateway is restarting... try again' response, which matches the report's current workaround/problem rather than fixing it.
  - file: [src/cli/gateway-cli/run-loop.ts:166](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/cli/gateway-cli/run-loop.ts#L166)
  - command: `nl -ba src/cli/gateway-cli/run-loop.ts | sed -n '130,220p'`
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **No restart-resume config exists on current main:** Current gateway reload config only exposes `mode`, `debounceMs`, and `deferralTimeoutMs`; there is no `session.resumeAfterRestart` or similar config surface for automatic post-restart session recovery.
  - file: [src/config/types.gateway.ts:203](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/config/types.gateway.ts#L203)
  - command: `nl -ba src/config/types.gateway.ts | sed -n '200,230p'`
