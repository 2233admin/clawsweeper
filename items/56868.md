## Summary

Current `main` no longer uses the ACPX startup path from the report: the bundled `acpx` plugin now runs an embedded runtime instead of verifying a plugin-local `acpx.cmd`, and remaining Windows wrapper launches are resolved without shell parsing by default.

## Evidence

- **ACPX startup is now embedded, not external CLI verification:** The bundled ACPX service registers an embedded runtime backend and probes that runtime directly; there is no startup path here that shells out to a plugin-local `acpx.cmd` for `--version` verification.
  - file: [extensions/acpx/src/service.ts:95](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/acpx/src/service.ts#L95)
  - command: `nl -ba extensions/acpx/src/service.ts | sed -n '95,156p'`
- **Old command/version config surface is gone:** The current ACPX config schema contains runtime settings like `cwd`, `stateDir`, `probeAgent`, MCP bridges, and legacy-compat flags, but not the reported `command` / `expectedVersion` verification surface.
  - file: [extensions/acpx/src/config-schema.ts:81](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/acpx/src/config-schema.ts#L81)
  - command: `nl -ba extensions/acpx/src/config-schema.ts | sed -n '81,123p'`
- **Plugin manifest marks Windows wrapper knob as legacy compatibility only:** `strictWindowsCmdWrapper` is still accepted, but the manifest help text explicitly says the current embedded `acpx` runtime package uses its own Windows command resolution behavior and treats this field as compatibility-only.
  - file: [extensions/acpx/openclaw.plugin.json:110](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/acpx/openclaw.plugin.json#L110)
  - command: `sed -n '1,220p' extensions/acpx/openclaw.plugin.json`
- **Windows wrapper execution no longer depends on unquoted shell parsing by default:** The shared Windows spawn resolver unwraps `.cmd` / `.bat` launchers to Node or EXE entrypoints when possible, and only returns `shell: true` when a caller explicitly opts into shell fallback.
  - file: [src/plugin-sdk/windows-spawn.ts:203](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/plugin-sdk/windows-spawn.ts#L203)
  - command: `nl -ba src/plugin-sdk/windows-spawn.ts | sed -n '203,314p'`
- **Release evidence for shipped fix:** The changelog entry for `2026.4.5` says the ACPX runtime was embedded directly in the bundled `acpx` plugin, removing the extra external ACP CLI hop that the report depended on.
  - file: [CHANGELOG.md:1212](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/CHANGELOG.md#L1212)
  - command: `nl -ba CHANGELOG.md | sed -n '1188,1214p'`

## Risks / Open Questions

- `docs/tools/acp-agents-setup.md` still describes the older plugin-local `acpx --version` verification flow, so documentation may lag the code even though the reported runtime bug is no longer present on `main`.

## Close Comment

Current `main` already removes the failing startup path from this report, so this is being closed as implemented on `main` after Codex review.

- `extensions/acpx/src/service.ts:95` now starts an embedded ACPX runtime backend and probes that runtime directly; it no longer verifies a plugin-local `acpx.cmd` on startup.
- `extensions/acpx/src/config-schema.ts:81` no longer exposes the old `command` / `expectedVersion` verification config surface from the report.
- `extensions/acpx/openclaw.plugin.json:110` marks `strictWindowsCmdWrapper` as legacy compatibility only, reflecting the newer embedded runtime path.
- `src/plugin-sdk/windows-spawn.ts:203` now resolves Windows `.cmd` / `.bat` wrappers to direct Node/EXE entrypoints and only allows `shell: true` as an explicit opt-in fallback, which addresses the quoted-path failure mode described for profile paths with spaces.
- `CHANGELOG.md:1212` shows the ACPX runtime embed shipped in `v2026.4.5`, which is the first release where this specific startup path is gone.
