## Summary

Keep #70202 open. Current main supports multiple configured models, fallbacks, `/model` switching, and a Control UI chat-session model selector, and `openclaw configure --section model` can multi-select the `/model` allowlist. However, first-run `openclaw onboard` still runs a single auth choice and then a single default-model picker; it does not expose the multi-model allowlist/default setup requested in this issue. The optional automatic routing portion overlaps with #29874, but the onboarding multi-model UX remains unimplemented.

## Best Possible Solution

Keep this open and implement a bounded onboarding enhancement: after provider auth and primary/default selection in `runSetupWizard`, reuse or adapt `promptModelAllowlist`, `applyModelFallbacksFromSelection`, and `applyModelAllowlist` so first-run setup can select multiple allowed models while clearly marking one primary default. Update onboarding docs and wizard tests. Leave automatic task-complexity routing to a separate provider/plugin or ClawHub discussion unless maintainers explicitly want core routing policy.

## Risks / Open Questions

- Provider auth flows may already add several catalog/default-model entries internally, so an implementation should distinguish provider-owned catalog registration from the user-visible allowlist/fallback/default selection requested here.
- The UX should preserve current safety guidance: docs recommend a strong primary model for tool-enabled or untrusted-input agents, so a cost-saving setup should not silently route risky tasks to weaker models.
