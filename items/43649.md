## Summary

Keep open. The issue discussion has multiple affected Web UI users and a workaround of deleting the problematic session, which points at persisted chat history. Current main has size-budget safeguards for chat.history, but I did not find high-confidence evidence that it fixes this exact freeze. A malformed saved history content block can still reach the Control UI renderer and be dereferenced without an object guard, so the reported “load history, UI gets stuck until session deletion” remains plausible.

## Evidence

- **Issue discussion supports a persisted-history failure mode:** The report says the Web UI freezes on the sessions/chat page after loading historical messages, while deleting the session makes it usable again; comments also report Chrome/Edge affected with Firefox usable. That is consistent with a client-side history-render path rather than a generic gateway outage.
- **History is accepted into UI state with minimal shape validation:** loadChatHistory requests chat.history, filters visibility-only cases, then assigns the returned messages to state.chatMessages. It does not validate or normalize nested content blocks before render.
