## Summary

Current `main` at `fa976e5b93b3819f9e137fe08635764edee38290` still matches the reported bug: Slack interactive reply button actions are only queued as system events and acknowledged in Slack, with no visible wake or dispatch path for a new agent turn. The current tests explicitly encode that behavior, so there is not strong evidence to close this item.

## Evidence

- **Slack block-action handler still only enqueues + updates message:** For reply-button block actions, `handleSlackBlockAction` falls through to `enqueueSlackBlockActionEvent(...)` and then `updateSlackLegacyBlockAction(...)`; there is no adjacent wake, route resolution, or `dispatchReplyWithDispatcher(...)` call in this handler path.
  - file: [extensions/slack/src/monitor/events/interactions.block-actions.ts:862](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/slack/src/monitor/events/interactions.block-actions.ts#L862)
  - command: `nl -ba extensions/slack/src/monitor/events/interactions.block-actions.ts | sed -n '790,875p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Queued system events are consumed on later reply runs:** `drainFormattedSystemEvents()` drains queued system events from the session when a reply run is building prompt bodies, which supports the reporter's point that enqueueing alone does not itself trigger an immediate agent turn.
