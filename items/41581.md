## Summary

Keep this issue open. Current main still routes Telegram DM partial answer previews through forced message/editMessageText transport when streaming is active, while the lower-level draft stream still supports DM-preferring auto/sendMessageDraft behavior. The current tests and docs lock in the edit-message behavior, and the related attempted fixes #62207 and #62210 are closed and unmerged.

## Best Possible Solution

Keep this issue open. The best fix is a targeted Telegram dispatcher change that restores `auto`/sendMessageDraft transport for DM answer partial previews when safe, preserves final materialization and duplicate-final suppression, decides the intended reasoning-lane behavior, updates Telegram streaming docs, and adds regression tests proving smooth DM partial previews without duplicate final messages.

## Risks / Open Questions

- A naive switch back to `previewTransport: "auto"` could reintroduce the duplicate-final/materialization behavior called out by the dispatcher comment, so the fix needs lifecycle regression coverage.
- Reasoning and answer preview lanes share the dispatcher lane factory; the fix should deliberately decide whether reasoning previews should use draft transport or stay on message transport.
- No live Telegram DM smoke was run in this read-only review; the source, tests, docs, and related PR context are enough to keep the item open, not enough to prove a user-visible UX fix.
