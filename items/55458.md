## Summary

Current `main` still silently swallows `/status` memory probe failures, and the PR’s core idea is not implemented. The branch is stale against refactored status files, but that is a reason to rebase/split it, not to close it as obsolete.

## Evidence

- **Memory probe errors are still discarded on main:** `resolveSharedMemoryStatusSnapshot()` still calls `manager.probeVectorAvailability()` inside an empty `catch {}` and then returns status with no error field, so failed probes remain hidden from `/status`.
  - file: [src/commands/status.scan.shared.ts:210](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/commands/status.scan.shared.ts#L210)
  - command: `nl -ba src/commands/status.scan.shared.ts | sed -n '206,218p'`
- **No `probeError` field exists in the current memory snapshot type:** `MemoryStatusSnapshot` on `main` only extends `MemoryProviderStatus` with `agentId`; there is no field available to thread a probe failure through the status pipeline.
  - file: [src/commands/status.scan.shared.ts:29](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/commands/status.scan.shared.ts#L29)
  - command: `nl -ba src/commands/status.scan.shared.ts | sed -n '24,34p'`
- **Current status rendering has no path to display probe failures:** `buildStatusMemoryValue()` renders files/chunks, sources, plugin slot, vector, FTS, and cache state only; there is no branch that would surface a memory probe error to the user.
  - file: [src/commands/status.command-sections.ts:140](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/commands/status.command-sections.ts#L140)
  - command: `nl -ba src/commands/status.command-sections.ts | sed -n '140,190p'`
- **The PR is stale but not obsolete:** On current `main`, `src/commands/status.scan.ts` no longer owns the probe logic directly; it delegates memory resolution into `status.scan-memory.ts`/`status.scan.shared.ts`, and overview rows call `buildStatusMemoryValue()`. The proposed fix therefore needs rebasing onto the refactored files rather than closure.
  - file: [src/commands/status.scan.ts:6](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/commands/status.scan.ts#L6)
  - command: `nl -ba src/commands/status.scan.ts | sed -n '1,85p'`

## Risks / Open Questions

- Closing this PR would leave a live diagnostics gap on `main`: `/status` still hides failed memory probes.
- The PR bundles unrelated Fly/deploy/SDK/docs changes with the valid diagnostics fix, so maintainers likely need a narrower follow-up or a split PR.
- There is no current test evidence in `main` covering the memory probe error-display path, so the regression remains unguarded.
