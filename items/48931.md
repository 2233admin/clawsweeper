## Summary

Current main already implements the relevant Discord/gateway cleanup: stale-socket restarts are no longer driven by stale app-event timestamps for Discord-like channels, Discord reconnects clear stale heartbeat timers, and long Discord runs already use a typing keepalive loop.

## Close Comment

Closing as implemented on current `main` after a read-only Codex review. The reported restart pattern is addressed in the current gateway health policy: `src/gateway/channel-health-policy.ts:115` only treats a channel as `stale-socket` when it has explicit transport-activity timestamps, and `src/gateway/channel-health-policy.test.ts:139` now asserts that stale app-event timestamps alone do not stale-restart a connected Discord channel. On the Discord side, the monitor status shape in `extensions/discord/src/monitor/status.ts:1` does not publish `lastTransportActivityAt`, so a long model run with no inbound app events no longer trips the stale-socket path, and `extensions/discord/src/monitor/gateway-plugin.ts:257` also clears stale heartbeat timers before reconnecting to avoid zombie callback crashes. This was shipped in `v2026.4.22` per `CHANGELOG.md:278`, with an additional Discord heartbeat reconnect hardening note at `CHANGELOG.md:807`. If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with a diagnostics export and current gateway logs so maintainers can compare it against the post-fix behavior.
