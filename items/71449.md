## Summary

Keep this PR open. Current main does not implement the PR's central UX changes: the startup log still says plain `ready (...)`, there is no `sidecars settled` channel health breakdown after sidecars finish, Telegram polling cleanup still blocks retry before polling on recoverable `deleteWebhook` failures, and Feishu startup probing still only treats timeout/abort specially. The related closed startup PR #67921 covers bounded channel startup, not this logging/health-summary and channel-specific startup hardening; the related #56369 remains a broader open readiness bug rather than a duplicate close target for this PR.

## Best Possible Solution

Keep #71449 open for maintainer review. The best path is to preserve the legacy `ready (` marker while clarifying it means gateway listening, add the `sidecars settled` channel health summary using the existing runtime snapshot/health-policy machinery with deterministic output, and review the Telegram/Feishu changes as channel-owned startup hardening. It should not be closed as implemented or duplicate unless a separate landed commit adds those exact startup log and channel-hardening behaviors.
