## Summary

Keep PR #71449 open. Current main does not contain the PR’s central startup-readiness clarification: the gateway still logs the old `ready (...)` line, there is no `sidecars settled` channel-health summary, Telegram polling startup still blocks/retries foreground webhook cleanup before polling, and Feishu startup only logs timeout failures rather than the PR’s shorter probe budget plus transient network handling. The related #67921 cleanup only covers bounded/non-blocking channel startup, while #56369 remains open for a deeper startup readiness race, so this PR is neither implemented nor cleanly superseded.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to finish it as focused startup UX/runtime hardening: preserve the legacy `ready (` marker for tooling while clarifying gateway-listening semantics, add a deterministic `sidecars settled` summary from the existing channel health policy, time-bound Telegram webhook cleanup without blocking polling indefinitely, keep Feishu startup probes on the shorter probe budget with transient-network logging, and add/keep targeted regression coverage. Coordinate wording and behavior with the still-open #56369 readiness issue rather than closing this as already implemented.
