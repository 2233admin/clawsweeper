## Summary

Keep open. Current main still matches the reported root cause: the TUI watchdog is refreshed by chat deltas, but not by active tool events, and reconnect reloads history without clearing stale active-run/watchdog state. The linked fix PR #69125 was closed unmerged, and #69978 is a separate watchdog UX follow-up rather than a canonical fix for this bug.

## Best Possible Solution

Keep the issue open and implement or revive the substance of #69125: rearm the streaming watchdog for active run tool proof-of-life, clear stale active-run/watchdog state on reconnect before `loadHistory()`, and add focused TUI regression tests for both paths. Treat #69978 as a separate UX enhancement after the core desync bug is fixed.

## Evidence

## Risks / Open Questions

- The fix should decide whether hidden tool events still count as proof-of-life. If the rearm happens after the verbose-level filter, users with tool output suppressed may still hit the false-idle path.
- Reconnect reset likely needs a small event-handler/session seam because `clearStreamingWatchdog` is currently closure-local inside `createEventHandlers`. Add regression coverage for both active tool-event rearm and reconnect reset.
