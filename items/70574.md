## Summary

Keep open. Current `main` still lets a cross-agent spawn replace the parent requester account with a child-agent bound account, and completion routing still resolves delivery from the child session’s bindings using that rewritten account. The linked fix from PR #70607 is not present on `main`.

## Evidence

- **Cross-agent spawn still rewrites requester account from the child agent:** `resolveRequesterOriginForChild` still does a bound-account lookup when `targetAgentId !== requesterAgentId`, passes `agentId: params.targetAgentId`, and then returns `accountId: boundAccountId ?? params.requesterAccountId`. That means a child-agent binding can still override the parent session's account on cross-agent spawns.
  - file: [src/agents/spawn-requester-origin.ts:113](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/agents/spawn-requester-origin.ts#L113)
  - command: `nl -ba src/agents/spawn-requester-origin.ts | sed -n '94,132p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Completion routing still resolves against the child session key:** `resolveSubagentCompletionOrigin` normalizes `requesterOrigin`, builds `requesterConversation` from it, and calls `createBoundDeliveryRouter().resolveDestination` with `targetSessionKey: params.childSessionKey`. If `requesterOrigin.accountId` was rewritten to the child's account, routing will be evaluated against the child's bindings.
  - file: [src/agents/subagent-announce-delivery.ts:301](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/agents/subagent-announce-delivery.ts#L301)
  - command: `nl -ba src/agents/subagent-announce-delivery.ts | sed -n '276,316p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Bound router matches by channel plus account before conversation:** The bound delivery router first filters active bindings where `conversation.channel === requester.channel && conversation.accountId === requester.accountId`, then prefers an exact conversation match. A child-derived requester account therefore steers matching toward the child agent's bound conversation, which is the failure mode described in the issue.
  - file: [src/infra/outbound/bound-delivery-router.ts:34](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/infra/outbound/bound-delivery-router.ts#L34)
  - command: `nl -ba src/infra/outbound/bound-delivery-router.ts | sed -n '30,56p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Checked-in tests still encode target-agent account selection for cross-agent spawns:** `spawn-requester-origin.test.ts` still expects cross-agent resolution to select target-agent accounts such as `bot-alpha-room-a` and only preserves the caller account in the explicit same-agent scenario. That shows the parent-account invariant proposed in the issue is not implemented on current `main`.
  - file: [src/agents/spawn-requester-origin.test.ts:177](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/agents/spawn-requester-origin.test.ts#L177)

- **Linked fix has not landed on current main:** The issue timeline links PR #70607 (`fix(agents): preserve parent accountId in cross-agent subagent spawn routing`), but current `main` at `0e9156d2` still contains the child-account lookup and the tests above still assert the old behavior.
