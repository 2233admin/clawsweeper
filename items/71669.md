## Summary

Keep open. Current main documents and client-displays the insecure HTTP/device-identity failure, and the gateway enforces the intended rejection, but there is no startup/runtime server warning for the specific deployment shape requested by #71669. The related open #32473 is useful symptom evidence, not a narrower implementation tracker for this log warning.

## Best Possible Solution

Keep the issue open and implement a narrowly scoped gateway startup warning, likely in `logGatewayStartup` or the post-attach startup-log call path, using `controlUi.enabled`, `tlsEnabled`, bind hosts/origins, and `gateway.controlUi.dangerouslyDisableDeviceAuth`. The warning should point to `https://docs.openclaw.ai/web/control-ui#insecure-http` and have unit coverage in `src/gateway/server-startup-log.test.ts` without changing the existing connection rejection policy.

## Risks / Open Questions

- A startup warning must avoid false positives for localhost, HTTPS/TLS, Tailscale Serve, trusted-proxy, and break-glass `dangerouslyDisableDeviceAuth` paths; the server has to infer risk from config rather than a browser's actual secure-context result.
- Existing client-side/docs guidance reduces user confusion, but it does not help Docker/VPS operators who only inspect gateway logs.
