## Summary

Keep #57831 open. Current main still resolves `sessionTarget: "current"` to `session:<caller sessionKey>` without checking direct-message chat keys, `cron.update` still normalizes patches without caller session context, and runtime execution still trusts any `session:<id>` target after only path-safety validation. The PR discussion and Greptile review describe a real isolation gap and useful three-layer fix, though the provided review comments also indicate the PR likely needs revision for legacy/canonical direct-key classification before merge.

## Best Possible Solution

Keep the PR open and revise it into a robust cron session-target guard: classify unsafe direct-message session keys using the shared session-key/chat-type helpers without misclassifying custom named sessions, normalize both add and update paths with caller session context, add a runtime fallback for legacy dirty direct `session:` targets, and keep regression coverage in normalize/tool/gateway tests for direct, group, legacy, and custom named sessions.
