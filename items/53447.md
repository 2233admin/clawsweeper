## Summary

Keep this issue open. Current main still resolves the Discord native thread parent during preflight and uses it for routing/config binding, but ordinary reply model selection does not receive that parent channel id when `channels.discord.thread.inheritParent` is left at its default `false`. The related closed #57960 work addresses parent-channel route/binding inheritance, not this issue's specific `modelByChannel` fallback for Discord thread replies.

## Best Possible Solution

Implement a narrow Discord model-routing fix: carry the resolved native thread parent channel id into channel model override candidate resolution independently of `channels.discord.thread.inheritParent`, while preserving thread-specific override precedence. Add regression coverage for a Discord thread with no thread override but a parent channel override, plus a thread-specific override-wins case.
