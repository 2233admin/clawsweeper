## Summary

Current `main` no longer starts Telegram's native approval/gateway client during plugin registration. The Telegram plugin exposes native approvals through a lazy runtime adapter, the gateway now bootstraps approval handlers separately from channel startup, and approval-handler startup retries instead of crashing the process. OpenClaw also shipped an explicit startup fix for native approval handlers in `v2026.4.22`.

## Evidence

## Close Comment

Closing as `implemented_on_main` based on this Codex review. The reported startup path no longer matches current OpenClaw: Telegram's native approval runtime is now lazy-loaded through `extensions/telegram/src/approval-native.ts:91`, gateway approval bootstrap is started separately from channel account startup in `src/gateway/server-channels.ts:401`, and the bootstrap waits for runtime context registration and retries failed handler starts in `src/infra/approval-handler-bootstrap.ts:19`. On top of that, approval handlers now report ready after gateway authentication and replay pending approvals in the background (`src/infra/exec-approval-channel-runtime.ts:295`), which is also called out in the `v2026.4.22` changelog (`CHANGELOG.md:100`). There is regression coverage for this lifecycle in `src/gateway/server-channels.approval-bootstrap.test.ts:122` and for non-fatal replay failures in `src/infra/exec-approval-channel-runtime.test.ts:620`. Given that current `main` and the latest release already contain the startup-handling changes, this issue looks fixed.
