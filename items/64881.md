## Summary

Keep this issue open. Current main still has the prompt-side guard from the prior runtime-truthfulness work, but the value feeding it can still be derived from configuration alone: ambiguous `enabled && allowed` elevated state resolves to full access available, auto-reply and command prompt paths pass no provider/runtime `fullAccessAvailable` signal, and the resolver test still asserts that optimistic fallback.

## Best Possible Solution

Keep this issue open. The best fix is to introduce a first-class provider/runtime full-access capability signal, thread it through auto-reply and command prompt construction into `ExecElevatedDefaults`/`bashElevated`, change ambiguous resolver input to unavailable with a clear blocked reason, and add regression coverage for ambiguous input, blocked runtimes, provider/runtime plumbing, and final system-prompt guidance.
