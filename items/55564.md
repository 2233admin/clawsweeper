## Summary

Current `main` still appears to replay persisted outbound queue entries after gateway startup, with no stale-age discard or `gateway.delivery.*` TTL control. The issue remains actionable rather than closable.

## Evidence

- **Outbound sends are write-ahead queued before delivery:** `src/infra/outbound/deliver.ts:649` persists every outbound delivery via `enqueueDelivery(...)` before attempting the send, so undelivered messages survive a crash/restart.
  - file: [src/infra/outbound/deliver.ts:649](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/infra/outbound/deliver.ts#L649)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '645,690p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Gateway startup unconditionally triggers pending delivery recovery:** `src/gateway/server-runtime-services.ts:55` defines restart recovery, and `src/gateway/server-runtime-services.ts:116` calls it from scheduled-service activation after startup.
  - file: [src/gateway/server-runtime-services.ts:55](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/gateway/server-runtime-services.ts#L55)
  - command: `nl -ba src/gateway/server-runtime-services.ts | sed -n '50,130p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Recovery replays all pending entries by age order; no stale-message TTL/drop path exists:** `src/infra/outbound/delivery-queue-recovery.ts:341` says startup scans and retries pending entries; the implementation sorts by `enqueuedAt` and drains each entry unless blocked by retry backoff or max retries. There is no age-based discard check.
