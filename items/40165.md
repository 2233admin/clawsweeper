## Summary

Keep #40165 open. Current main has partial NO_REPLY hardening: auto-reply normalization can strip a trailing standalone NO_REPLY and the system prompt warns not to append it, but the gateway live chat/webchat projection path still does not strip suffix-position NO_REPLY from substantive text. Existing outbound payload tests also still preserve mixed text ending in NO_REPLY, so the requested gateway-level suffix cleanup is not implemented across the affected user-visible surfaces.

## Best Possible Solution

Keep this issue open. Implement a shared, conservative silent-token cleanup contract for gateway/live-chat display and outbound payload normalization: exact token suppresses, existing leading-fragment protections remain, and a trailing standalone NO_REPLY suffix is stripped from otherwise substantive assistant text. Add regression tests for `Task complete. NO_REPLY`, `Git is clean.\nNO_REPLY`, exact NO_REPLY suppression, malformed leading NO_REPLY output, and legitimate NO_REPLY mentions that should remain visible; then update the public docs to describe the broadened behavior.
