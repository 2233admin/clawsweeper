## Summary

Current `main` does not implement a Slack config or routing path that threads replies conditionally on tool use. Slack threading is still controlled only by `replyToMode`/`replyToModeByChatType` plus existing `thread.*` settings, so the requested behavior remains unimplemented.

## Evidence

- **No `replyOnToolUse` config or docs:** `rg -n "replyOnToolUse" src extensions docs` returned no matches on `main`, so there is no shipped config key, implementation hook, or documentation for the requested setting.
  - command: `rg -n "replyOnToolUse" src extensions docs`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Slack config surface lacks tool-use threading option:** `SlackThreadConfig` only exposes `historyScope`, `inheritParent`, `initialHistoryLimit`, and `requireExplicitMention`; `SlackAccountConfig` exposes `replyToMode` and `replyToModeByChatType`, but nothing conditional on tool execution.
  - file: [src/config/types.slack.ts:92](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/config/types.slack.ts#L92)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Runtime initializes threading from `replyToMode` only:** Slack monitor setup reads `slackCfg.replyToMode` and `slackCfg.thread.*`; there is no tool-use signal or conditional reply-thread routing knob during provider initialization.
  - file: [extensions/slack/src/monitor/provider.ts:174](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/slack/src/monitor/provider.ts#L174)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Reply routing logic is mode-based, not tool-based:** `resolveSlackThreadTs` and `createSlackReplyReferencePlanner` compute thread targets from `replyToMode`, incoming thread state, and whether a reply was already sent. No branch checks whether tools were invoked.
  - file: [extensions/slack/src/monitor/replies.ts:110](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/slack/src/monitor/replies.ts#L110)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Shared dispatch treats tool and conversational flows the same:** Slack dispatch uses one shared `hasRepliedRef` for both tool and auto-reply flows so `replyToMode=first` applies uniformly, which matches the report that routing does not differentiate tool-heavy turns from conversational ones.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:327](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/extensions/slack/src/monitor/message-handler/dispatch.ts#L327)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Docs and tests describe only static reply modes:** Slack docs list only `replyToMode`, `replyToModeByChatType`, and `thread.*`, and tests cover top-level replies staying in root for `off` and threading for `first`/`all`, again with no tool-use condition.
  - file: [docs/channels/slack.md:522](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/docs/channels/slack.md#L522)
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
