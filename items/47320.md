## Summary

Keep open. Current `main` still only exposes direct-child subagent listings. The `subagents` tool has no `depth`/`recursive` parameter, `/subagents list` is documented only without recursion, and both the controller-side filter and nested tests confirm main sees only depth-1 children while an orchestrator sees only its own depth-2 children.

## Evidence

- **No recursive/depth parameter on `subagents` tool:** The tool schema only accepts `action`, `target`, `message`, and `recentMinutes`; `list` always uses `listControlledSubagentRuns(controller.controllerSessionKey)` with no depth option.
  - file: [src/agents/tools/subagents-tool.ts:26](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/agents/tools/subagents-tool.ts#L26)

  - sha: [0e7250f37bf3](https://github.com/openclaw/openclaw/commit/0e7250f37bf347c72790abf2a6692d5ec84e5ea1)
- **Control scope is direct children only:** `listControlledSubagentRuns` filters runs whose latest `controllerSessionKey` exactly matches the current controller session key, so grandchildren are not returned to the root controller.
  - file: [src/agents/subagent-control.ts:145](https://github.com/openclaw/openclaw/blob/0e7250f37bf347c72790abf2a6692d5ec84e5ea1/src/agents/subagent-control.ts#L145)

## Risks / Open Questions

- There is existing descendant traversal logic in `src/agents/subagent-registry-queries.ts:301`, but it is not wired into the `subagents` tool or slash-command surface, so closing as implemented would be premature.
- Current `subagents` list does expose `pendingDescendants` and child-session metadata, which may partially reduce the pain, but it does not provide the requested direct visibility into depth-2 labels/status/runtime/tasks from depth-0.
