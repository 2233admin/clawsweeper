## Summary

Keep this issue open. Current main still routes embedded Anthropic Messages runs through the boundary-aware transport, and that transport still calls raw JSON.parse on each SSE data frame. I did not find source, test, docs, changelog, or local-history evidence that the reported bad-control-character streaming failure has been fixed. The related #68442 is useful adjacent signal, but it is open and covers a different webchat/tool-call freeze path, not a canonical duplicate of this pure-text Telegram failure.

## Best Possible Solution

Keep this issue open. Add focused regression coverage around the Anthropic Messages SSE parser and embedded runner path with long text, emoji/non-ASCII content, split UTF-8 boundaries, and an invalid raw-control-character-in-string frame matching the reported JSON.parse failure. If reproduced, harden parseAnthropicSseBody or its caller with a deliberately scoped repair or graceful-failure path that preserves terminal lifecycle delivery, then validate separately whether #68442 is fixed by the same parser change or needs its own UI/tool-call recovery work.
