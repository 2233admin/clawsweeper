## Summary

Keep open. Current main improves Telegram webhook request handling after startup, and the 2026.4.22 thinking bundle syntax failure appears obsolete in later source, but the reported Telegram startup failure is still actionable: webhook startup still performs outbound Telegram API initialization before the listener is bound, and a `setWebhook` failure closes the just-bound listener and throws, leaving the channel dependent on crash-loop restart rather than keeping local ingress available while advertisement is retried.

## Best Possible Solution

Keep the issue open and fix Telegram webhook startup in the bundled Telegram plugin: bind and keep the local listener alive before/while recoverable `setWebhook` advertisement is retried, or otherwise retry `setWebhook` in-place with backoff without closing the listener. Add a focused regression test where `setWebhook` fails once and `/healthz` or the configured webhook path remains reachable during the retry. Treat the 2026.4.22 invalid bundle as already obsolete unless release cleanup/yanking is still desired separately.

This matches the issue's observed state where a transient setWebhook failure leaves no local listener.
