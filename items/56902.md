## Summary

The reported duplicate-delivery bug is in the external `@tencent-weixin/openclaw-weixin` channel plugin, not in code shipped on `openclaw/openclaw` `main`. OpenClaw core already exposes generic inbound dedupe keyed by stable `MessageSid`, and the plugin SDK/docs make inbound handling plugin-owned.

## Close Comment

Thanks for the report. This Codex review found that the problem belongs in the external WeChat plugin rather than in OpenClaw core.

- `docs/channels/wechat.md:10` and `docs/channels/wechat.md:27` document `@tencent-weixin/openclaw-weixin` as an external plugin and say the WeChat-specific monitoring/runtime is owned there.
- `docs/plugins/sdk-channel-plugins.md:556` says inbound handling is channel-specific and owned by each channel plugin.
- OpenClaw core already has generic inbound dedupe keyed by stable `MessageSid` in `src/auto-reply/reply/inbound-dedupe.ts:58` and skips duplicates in `src/auto-reply/reply/dispatch-from-config.ts:270`.
- Bundled channel plugins follow that contract by forwarding platform message ids into `MessageSid`, for example `extensions/zalo/src/monitor.ts:602` and `extensions/telegram/src/bot-message-context.session.ts:342`.
- There is no Weixin implementation in this repository's `extensions/` tree on `main`; the checkout only contains docs/tests references for `openclaw-weixin`.

Because the reported defect is in plugin-owned inbound delivery, this should be tracked in the external plugin's own repository/maintainer queue rather than `openclaw/openclaw` core.
