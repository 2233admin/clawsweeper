## Summary

Keep #69326 open. Current main has generic support for the external WeChat channel plugin, but the browser WebChat path still sends `chat.send` with `deliver: false`, the server converts non-delivery WebChat turns to the internal `webchat` origin before consulting `deliveryContext`, and reply dispatch explicitly suppresses external routing from internal WebChat unless `ExplicitDeliverRoute` is true. That does not satisfy the reported expectation that browser-originated turns in a WeChat-backed session route replies back through `deliveryContext.channel=openclaw-weixin`.

## Best Possible Solution

Keep the issue open and decide the intended product behavior for browser WebChat on channel-scoped sessions. If maintainers want this relay, implement it generically by giving `chat.send` or the WebChat UI an explicit external-delivery path that derives `originatingChannel`, `originatingTo`, account, and thread from the session `deliveryContext`, sets `ExplicitDeliverRoute: true`, and then add regression coverage for a registered plugin channel such as `openclaw-weixin` without hardcoding WeChat-specific behavior in core.

## Risks / Open Questions

- Automatically relaying browser WebChat input to an external channel changes the current safety/product boundary for UI-originated messages, so this needs an explicit maintainer decision or visible UI/parameter semantics rather than a silent route change.
- The WeChat runtime is external, so any fix should remain channel-agnostic and use the existing plugin outbound contract instead of hardcoding `openclaw-weixin` in core.
