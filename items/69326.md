## Summary

Keep #69326 open. Current main still treats browser WebChat/Control UI sends as internal WebChat turns, and core Gateway routing explicitly prevents WebChat clients from inheriting a stored external deliveryContext even on channel-scoped sessions. The WeChat runtime is external, but the reported route block is in OpenClaw core behavior, so this is not safely closeable as plugin-owned cleanup.

## Best Possible Solution

Keep the issue open for maintainer/product handling. The likely path is an explicit WebChat/Control UI send-to-channel affordance for channel-scoped sessions, using stored deliveryContext or explicit originatingChannel/originatingTo only when the user intentionally targets the external conversation, with regression tests for a third-party plugin channel id such as openclaw-weixin and safeguards preserving the existing route-overwrite protections.

## Evidence
