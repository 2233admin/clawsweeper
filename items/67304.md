## Summary

Keep open. Current main still lacks `OPENCLAW_JITI_CACHE_DIR`, and the PR addresses a real plugin-loader operational gap rather than a ClawHub-only feature. The PR is not merge-ready as written because its unset fallback passes explicit `fsCache: true`, which can override Jiti's existing `JITI_FS_CACHE` / `JITI_CACHE` controls; it also needs focused tests and changelog/docs if maintainers accept the new env var.

## Best Possible Solution

Keep this PR open for contributor revision and maintainer review. The best implementation is a narrow resolver that returns a trimmed `OPENCLAW_JITI_CACHE_DIR` string only when non-empty and otherwise omits `fsCache` or returns `undefined` so Jiti keeps honoring `JITI_FS_CACHE` / `JITI_CACHE`; wire it through the shared plugin loader, `src/plugin-sdk/root-alias.cjs`, and the Matrix wrapper; add tests for unset, blank, custom path, and Jiti env-disable behavior; include effective `fsCache` in cache isolation if it can vary; and add the required changelog/docs entry if maintainers accept the env var.

## Risks / Open Questions

- Merging unchanged would force `fsCache: true` whenever `OPENCLAW_JITI_CACHE_DIR` is unset, potentially regressing deployments that use `JITI_FS_CACHE=false` or `JITI_CACHE=false` to avoid filesystem cache writes.
- The effective fsCache value can vary in tests or embedded runtimes, so loader-cache keys may need to include that value or otherwise avoid stale cached Jiti instances.
- A new user-facing env var needs focused tests and changelog/docs so operators can discover it and maintainers can preserve compatibility expectations.
