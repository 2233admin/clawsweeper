## Summary

Current `main` already exposes plugin-owned same-origin gateway HTTP routes and plugin gateway RPC methods, which cover the requested Mission Control proxy-path workaround without widening Control UI CSP.

## Evidence

## Close Comment

Closing from a Codex review because current `main` already has the pieces needed for this workflow. Plugins can now mount same-origin gateway HTTP endpoints with `api.registerHttpRoute(...)` (`src/plugins/types.ts:2037`, `docs/plugins/architecture-internals.md:529`) and register gateway RPC methods with `api.registerGatewayMethod(...)` (`src/plugins/types.ts:2047`). Gateway tests also show plugin routes are served before the Control UI SPA fallback (`src/gateway/server.plugin-http-auth.test.ts:596`), so a Mission Control integration can own a path like `/mission-control/api/...` on the gateway origin while the Control UI CSP remains locked to `connect-src 'self' ws: wss:` (`src/gateway/control-ui-csp.ts:35`). That means the blocker described here is already addressed on `main`, even though the solution is plugin-owned rather than a new `openclaw.json` key.
