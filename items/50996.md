## Summary

Keep open. Current `main` has session queue controls (`steer`, `interrupt`, `collect`), but it does not implement true mid-tool-call interruption for `process(action:"poll")`. The `process` tool still waits in its own timeout loop without checking the abort signal, so an active run can remain blocked until the poll timeout or process exit. That means the reported long-poll unresponsiveness is still plausible on current `main`.

## Evidence

- **`process poll` ignores abort signal:** The `process` tool `execute` signature receives `_signal` but does not use it; the parameter is explicitly ignored.
  - file: [src/agents/bash-tools.process.ts:132](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/bash-tools.process.ts#L132)
  - command: `nl -ba src/agents/bash-tools.process.ts | sed -n '1,260p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **`process poll` blocks in a timeout loop:** For `action: "poll"`, the tool sleeps in a `while` loop until the deadline or process exit (`setTimeout` up to the requested poll timeout, capped at 120000 ms) and does not check cancellation inside that loop.
  - file: [src/agents/bash-tools.process.ts:306](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/bash-tools.process.ts#L306)
  - command: `nl -ba src/agents/bash-tools.process.ts | sed -n '260,360p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Abort wiring exists, but only if tools honor the signal:** Tool execution is wrapped with a combined abort signal before `execute(...)` runs. Since `process poll` ignores that signal, aborting the session does not preempt the wait loop.
  - file: [src/agents/pi-tools.abort.ts:48](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/agents/pi-tools.abort.ts#L48)
  - command: `nl -ba src/agents/pi-tools.abort.ts | sed -n '48,68p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Queue `interrupt` only aborts the run, then waits for it to end:** Inbound queue interrupt mode calls `abortActiveRun(...)` and then waits for the active run to drain. If the current tool call does not abort promptly, the follow-up message still waits.
  - file: [src/auto-reply/reply/get-reply-run-queue.ts:29](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/auto-reply/reply/get-reply-run-queue.ts#L29)
  - command: `nl -ba src/auto-reply/reply/get-reply-run-queue.ts | sed -n '1,220p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **`steer` is only boundary-level, not mid-tool-call:** The queue design doc states that `steer` cancels pending tool calls only 'after the next tool boundary', which does not solve a long-running `process poll` already in progress.
  - file: [docs/concepts/queue.md:29](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/docs/concepts/queue.md#L29)
