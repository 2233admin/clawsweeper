## Summary

Keep open. Current `main` and the latest release still derive unfiltered `openclaw nodes list` output only from `node.pair.list`, so the stale-pairing inconsistency described in PR #50970 is not implemented on `main` and cannot be conservatively closed.

## Evidence

- **Unfiltered `nodes list` still only counts `paired` entries from `node.pair.list`:** `src/cli/nodes-cli/register.status.ts:325` loads `node.pair.list`, `:326` parses `{ pending, paired }`, and `:340-368` renders `filteredPaired` from that `paired` array. `node.list` is only fetched behind `hasFilters` at `:330-339`, so plain `openclaw nodes list` still has no merge path for live paired nodes missing from `node.pair.list`.
  - file: [src/cli/nodes-cli/register.status.ts:325](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/cli/nodes-cli/register.status.ts#L325)

  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)

## Risks / Open Questions

- Reviving the PR unchanged would make plain `nodes list` depend on `node.list`, which currently requires `operator.read` rather than the existing `operator.pairing` scope.
- A durable fix should preserve the existing old-gateway fallback pattern for `node.list` compatibility instead of hard-failing when that RPC is unavailable.
