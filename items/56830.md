## Summary

Keep open. Current `main` still shows the ambiguous Control UI/device-auth messaging this PR targets, while the gateway policy and tests still enforce the stricter localhost-only behavior. That means the report is still valid and the PR is not obsolete.

## Evidence

- **Current UI hint still implies HTTP plus allowInsecureAuth is a general fallback:** `overview.insecure.stayHttp` still says `If you must stay on HTTP, set {config} (token-only).`, and the overview renders it with `gateway.controlUi.allowInsecureAuth: true`, without the localhost-only caveat.
  - file: [ui/src/i18n/locales/en.ts:258](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/ui/src/i18n/locales/en.ts#L258)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Current formatted Control UI error still points users at insecure auth explicitly:** `CONTROL_UI_DEVICE_IDENTITY_REQUIRED` is still formatted as `device identity required (use HTTPS/localhost or allow insecure auth explicitly)`, which does not distinguish remote plain HTTP from localhost.
  - file: [ui/src/ui/connect-error.ts:75](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/ui/src/ui/connect-error.ts#L75)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Gateway policy still rejects remote Control UI when only allowInsecureAuth is set:** The connect policy keeps remote Control UI on plain HTTP rejected unless the client is local, and the adjacent test explicitly says `Remote Control UI with allowInsecureAuth -> still rejected.`
  - file: [src/gateway/server/ws-connection/connect-policy.ts:129](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/server/ws-connection/connect-policy.ts#L129)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Current handshake/UI path still uses the older short wording:** The WebSocket handler still sends `control ui requires device identity (use HTTPS or localhost secure context)` and closes with that same string, while current UI tests still only assert the older `device identity required` wording.
  - file: [src/gateway/server/ws-connection/message-handler.ts:653](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/server/ws-connection/message-handler.ts#L653)
