## Summary

Keep #69422 open. Current main still lacks the Google Chat thread-routing metadata needed for the reported streamed block and follow-up paths: inbound Google Chat stores `message.thread?.name` only as `ReplyToId`, not `MessageThreadId`; block reply payload/key/coalescer logic still only carries `replyToId`; queued follow-up routing reads `ctx.MessageThreadId`, so Google Chat follow-ups from threads still drain without a thread id. Related #70041 covers one long-reply symptom, but this issue is broader and remains actionable.

## Best Possible Solution

Keep this issue open as the broader canonical Google Chat thread-routing bug. The likely fix should make Google Chat thread identity a first-class routed value for inbound context, streamed block payload/coalescer identity, reply delivery, and follow-up queue drain, then add regression tests for long chunked replies and queued concurrent thread messages. Also align `messages.queue.byChannel.googlechat` between types and zod if that mitigation is kept.
