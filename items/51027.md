## Summary

Keep open. Current `main` does not implement the gateway alias rejection from this PR, and the PR discussion surfaced unresolved regressions in both halves of the change, so there is not strong evidence for an auto-close.

## Evidence

- **`main` still routes `sessions.patch` model strings through normal model resolution with no `list`/`status` special-case:** `applySessionsPatchToStore` only rejects empty model strings, then calls `resolveAllowedModelRef`; there is no current-main check rejecting `list` or `status` before alias/model resolution.
  - file: [src/gateway/sessions-patch.ts:401](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/gateway/sessions-patch.ts#L401)
  - command: `nl -ba src/gateway/sessions-patch.ts | sed -n '392,425p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Configured model aliases are first-class inputs today:** `buildModelAliasIndex` loads aliases from `cfg.agents.defaults.models`, and `resolveAllowedModelRefFromAliasIndex` resolves the raw string against that alias index. A hardcoded rejection of bare `list`/`status` in `sessions.patch` would bypass this existing alias path.
  - file: [src/agents/model-selection-shared.ts:273](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/model-selection-shared.ts#L273)

- **Current CLI intentionally avoids `node.list` on plain `nodes list`:** `register.status.ts` only calls `node.list` when `--connected` or `--last-connected` is active; the plain path uses `node.pair.list` alone. This matches the scope split where `node.pair.list` is under `operator.pairing` and `node.list` is under `operator.read`. The review-thread concern about the PR making `nodes list` require broader read scope is valid.
  - file: [src/cli/nodes-cli/register.status.ts:322](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/cli/nodes-cli/register.status.ts#L322)

That supports the later PR review comments that a count-only fallback from `node.list` would still leave row rendering/JSON based on a narrower source.
