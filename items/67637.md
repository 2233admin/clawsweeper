## Summary

Current `main` has a mock-only workaround for parity assertions via `/debug/requests`, but `QaBusMessage` still has no tool-trace field and the bus state still only records text/attachments/reactions. The requested message-level trace inspection is therefore not implemented on `main`.

## Evidence

- **QaBusMessage contract still has no tool-call field:** `QaBusMessage` only includes ids, text, attachments, reactions, and metadata; there is no `toolCalls` or equivalent trace payload in the public QA channel protocol.
  - file: [src/plugin-sdk/qa-channel-protocol.ts:24](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/plugin-sdk/qa-channel-protocol.ts#L24)
  - command: `nl -ba src/plugin-sdk/qa-channel-protocol.ts | sed -n '1,120p'`

- **QA bus state still constructs plain text messages:** `createMessage(...)` populates `QaBusMessage` with `text`, `attachments`, and `reactions` only, so tool traces are not being plumbed into stored bus messages.
