## Summary

Keep open. Current main still does not expose generalized tool-call traces on QaBusMessage or the QA bus transport surface. Main has mock-provider /debug/requests plannedToolName/plannedToolArgs assertions, which are useful adjacent coverage, but they do not satisfy the requested QaBusMessage trace field populated from actual agent/tool execution logs.

## Best Possible Solution

Keep this issue open. Implement a backwards-compatible QA trace surface, either optional structured tool-call traces on QaBusMessage or a dedicated scenario assertion API, populated from actual agent/tool execution logs rather than only mock-provider planned requests. Carry it through create, clone, snapshot, read, search, poll, and wait paths, and add focused qa-channel/qa-lab tests plus at least one scenario that fails when the agent fabricates success without calling the expected tool.
