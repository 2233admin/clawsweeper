## Summary

Current main still does not implement the requested config-backed session lane concurrency, automatic yield policy, or configurable lane-wait warning threshold. The related open PR #63864 appears to cover only the session-lane concurrency slice, so this issue remains the broader actionable tracker.

## Best Possible Solution

Keep this issue open as the broader product tracker. Review or rework #63864 for the session lane concurrency portion, wiring any accepted setting through AgentDefaults types, schema, defaults, generated metadata/docs, startup/reload lane application, and queue/runner tests. Separately design auto-yield in the embedded runner/tool loop, and either implement or split out the configurable lane-wait warning threshold, likely coordinated with #14747.

## Risks / Open Questions

- Increasing per-session concurrency can conflict with transcript/session mutation assumptions unless the implementation defines locking, ordering, and runner lifecycle semantics.
- Auto-yield is a separate runner/tool-loop design problem because forced interruption during tool execution differs from the existing explicit sessions_yield flow.
- If #63864 merges, the session-concurrency portion should be reassessed, but the auto-yield and lane-wait warning configurability still need either implementation here or narrower follow-up tracking.
