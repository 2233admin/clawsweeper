## Summary

Current main appears to have superseded the crash portion of PR #58216 through `waitForDiscordGatewayStop`: abort-triggered disconnects settle the promise before `gateway.disconnect()` can synchronously emit a Carbon gateway error, and the regression test covers that a stop event emitted during disconnect does not reject. However, the PR’s narrower requested behavior, suppressing/logging the expected `reconnect-exhausted` event differently inside `runDiscordGatewayLifecycle`, is not present on current main. Since the open PR may still contain useful log-suppression intent, and its own review comments also identify test/behavior gaps, it should not be auto-closed by this cleanup pass.

## Best Possible Solution

Do not merge PR #58216 as-is. The best maintainer path is a small current-main follow-up in `extensions/discord/src/monitor/provider.lifecycle.ts` and its tests: simulate abort-triggered `disconnect()` emitting `reconnect-exhausted` while the lifecycle handler is active, assert the lifecycle still resolves, and decide explicitly whether the expected event should be logged at info level instead of `runtime.error`. If maintainers consider the crash fix sufficient and accept the remaining error log, close manually as obsolete with that rationale.
