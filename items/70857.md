## Summary

Keep this issue open. Current main still has the reported `sessions.json.lock` write path: `updateSessionStore` and `updateSessionStoreEntry` both acquire `withSessionStoreLock` and then force `loadSessionStore(..., { skipCache: true })`, which bypasses the session-store cache and performs synchronous disk read/JSON parse while the lock is held. The fresh macOS corroboration and open PR #68554 remain directly relevant; PR #71903 is adjacent lock hardening, not a closure path for this active long-held store-lock report.

## Best Possible Solution

Keep this issue open and review or land #68554, or an equivalent maintainer-owned fix that removes the redundant `skipCache` disk rereads from `updateSessionStore` and `updateSessionStoreEntry` while preserving serialized writes and cache coherency. Add a focused regression or performance guardrail if feasible, then verify startup/RPC responsiveness with a large `sessions.json` and active channel/cron traffic before closing.
