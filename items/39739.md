## Summary

Current main still does not support `gateway.controlUi.allowedOrigins` scheme-level wildcards such as `chrome-extension://*`. The gateway origin checker only accepts the global `*` or an exact normalized `URL.origin`; Node normalizes Chrome extension origins to `"null"`, and current tests/docs do not cover or document extension wildcard support. Related PR #39867 attempted this but is closed unmerged; PR #38290 is still open and intentionally narrower, covering exact extension origins rather than wildcard matching.

## Best Possible Solution

Keep #39739 open for maintainer product/security judgment. If accepted, implement a narrowly scoped `chrome-extension://*` matcher in `src/gateway/origin-check.ts`, preferably by preserving parsed protocol/host for non-standard origins and matching only `chrome-extension:` rather than adding generic prefix wildcards. Add focused unit and WebSocket handshake tests for exact extension origins, `chrome-extension://*`, non-extension negatives, malformed origins, literal `Origin: null`, existing HTTP(S) exact matches, global `*`, host-header fallback, and loopback behavior. Update the Control UI/config docs if the wildcard becomes supported, and coordinate with PR #38290 without treating it as a replacement for this wildcard issue.

## Risks / Open Questions

- Origin allowlisting is security-sensitive; a generic trailing-star matcher could broaden trusted browser origins unless constrained to explicit, reviewed schemes and backed by negative tests.
- PR #38290 may eventually fix exact `chrome-extension://<id>` allowlisting, but that would still not solve the stable scheme-level wildcard request in this issue.
