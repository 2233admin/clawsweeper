## Summary

Keep #57297 open. Current main has provider-attempt diagnostics and WebChat audio embedding work, but it does not implement the requested explicit WebChat TTS route/delivery telemetry, correlation id, or user-visible final delivery status. The linked playback report #57296 is also still open in the provided context.

## Best Possible Solution

Keep the issue open and implement a narrow gateway/WebChat observability layer: emit structured generation, route, and delivery outcome events with a run/tool/idempotency correlation id; distinguish provider synthesis, WebChat embedding, assistant-media persistence, browser-playback/attachment fallback, and local-path denial errors; then surface the final TTS delivery state in the Control UI chat message.
