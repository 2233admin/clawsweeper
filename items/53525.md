## Summary

Current main still does not show enough evidence that the reported Qianfan/Baidu error-detail loss is fixed. The issue is concrete, recent, and remains an actionable bundled provider/transport bug rather than stale cleanup material.

## Best Possible Solution

Keep this issue open. The best fix is a narrow Qianfan provider adapter or shared OpenAI-compatible error-normalization seam that promotes Baidu/Qianfan non-2xx and streamed error payloads into assistant `errorMessage`, including `error.code/message/type`, top-level `code/message/requestId`, and streamed or legacy `error_code`/`error_msg`/`error_message` shapes. Add regression tests in `src/agents/provider-http-errors.test.ts` and `src/agents/openai-transport-stream.test.ts` so provider error chunks cannot silently produce empty assistant responses.
