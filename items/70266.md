## Summary

Keep #70266 open. Current main has assistant/avatar identity support for config, gateway identity resolution, avatar HTTP routes, and Control UI chat rendering, but the native macOS Talk Mode overlay still only renders the generic `TalkOrbView` and only consumes `ui.seamColor` from Talk config. The reporter's follow-up correctly distinguishes this from the closed web/Control UI identity issue #22199.

## Best Possible Solution

Keep the issue open and implement a targeted native macOS Talk Mode enhancement. The likely path is to resolve the active assistant identity through the existing gateway identity surface or an additive Talk config payload, load a resolvable avatar image safely, render it clipped inside the orb while preserving rings/pulse/thinking arcs, and fall back to the current gradient orb when no image avatar is available. Add focused macOS/parser coverage so this is not closed again against the web/Control UI avatar path.

## Risks / Open Questions

- A future implementation needs to preserve avatar auth/path safety. Existing `/avatar/<agentId>` handling is auth-gated for Control UI, so native macOS should avoid leaking local paths and should use a similarly authorized/resolved image path.
- The right product behavior may need a maintainer call for which identity to show when Talk Mode is attached to a session or non-default agent, but the current issue is still actionable as a focused macOS Talk overlay feature.
