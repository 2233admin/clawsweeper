## Summary

Current main still appears to have the reported product gap: stuck sessions are detected and emitted as diagnostics, but the diagnostic heartbeat does not invoke an abort/cancel/recovery path. Manual abort primitives and provider-specific fixes exist, including the related DeepSeek V4 replay fix, but they do not close this generic gateway self-recovery request.

## Best Possible Solution

Keep this issue open. The likely maintainer path is a core runtime change that wires the existing active-run/reply-run abort primitives into diagnostics through an explicit recovery policy such as `diagnostics.stuckSessionAbortMs`, emits a distinct recovery diagnostic event/log, releases queued work safely, and adds regression coverage for a processing session that never settles. Provider-specific fixes like the DeepSeek V4 replay patch should remain separate because they reduce one cause, not the generic self-healing gap.
