## Summary

Keep #71127 open: current main still implements stuck-session diagnostics as warning/event emission only, while abort paths exist only for explicit user/API interruption and are not called by the diagnostic heartbeat.

## Best Possible Solution

Keep the issue open and implement a conservative core recovery path: add an opt-in or carefully bounded `diagnostics.stuckSessionAbortMs`-style config, wire the diagnostic heartbeat to an existing session/run abort seam or a new narrow recovery hook, clear/release the affected session queue safely, emit a distinct abort diagnostic event/log, and cover it with focused diagnostics plus gateway/session-run tests. Provider-specific root-cause fixes should remain separate from this generic self-healing mechanism.
