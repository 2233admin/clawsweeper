## Summary

Current OpenClaw already implements Feishu agent routing via `bindings`, rewrites Feishu conversations onto the bound agent/session at runtime, and ships that behavior in `v2026.4.22`; the report matches older behavior from `openclaw 2026.3.13`, not current `main`.

## Risks / Open Questions

- The original report does not include the actual `bindings` config, agent IDs, or Feishu peer IDs, so the older 2026.3.13 failure mode cannot be reconstructed exactly from the issue alone.
- If the reporter used an `agentId` not present in `agents.list`, current routing intentionally falls back to the default agent.

## Close Comment

Closing from a Codex review because current OpenClaw already implements Feishu routing to specific agents, and that implementation is present in the shipped `v2026.4.22` tree as well. The current docs show Feishu multi-agent `bindings` examples in `docs/channels/feishu.md:329`, the binding compiler preserves the configured `agentId` in `src/channels/plugins/configured-binding-compiler.ts:96`, and the runtime route rewriter swaps Feishu conversations onto the bound agent/session in `src/channels/plugins/binding-routing.ts:71`. There is also explicit coverage for Feishu configured DM routing and bound conversation routing in `extensions/feishu/src/bot.test.ts:376` and `src/channels/plugins/binding-routing.test.ts:60`. This issue appears to match older behavior from `openclaw 2026.3.13`, not current `main` or `v2026.4.22`. If this still reproduces on a current build, please open a fresh issue with the exact `bindings` snippet, the agent IDs present in `agents.list`, and the Feishu DM/group IDs involved.
