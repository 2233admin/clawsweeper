## Summary

Keep open. Current `main` still routes WhatsApp outbound sends through `toWhatsappJid(...)`, which always normalizes phone numbers to `@s.whatsapp.net` and does not consult any LID mapping or Baileys LID lookup. The inbound path already has explicit LID->PN resolution, and current tests lock outbound behavior to PN JIDs, so there is no strong evidence this issue has been implemented or become obsolete on `main`.

## Evidence

- **Outbound send path still hardcodes PN JIDs:** `createWebSendApi().sendMessage`, `sendPoll`, and `sendComposingTo` all call `toWhatsappJid(...)` directly. `toWhatsappJid` preserves existing JIDs but otherwise always returns `${digits}@s.whatsapp.net`, with no auth-dir mapping lookup and no Baileys `lidMapping` usage.
