## Summary

Keep PR #70651 open. Current main still lacks Feishu card table-limit fallback handling on the outbound card paths, and this PR is a focused fix attempt for the real 230099/11310 delivery-loss bug. It is not implemented on main and is not cleanly superseded, but the PR still needs revision and regression tests before merge.

## Best Possible Solution

Keep PR #70651 open and request a narrow revision: inspect Axios `response.data` before any object serialization in `parseCardKitError`, add focused tests for the supported error shapes and both outbound fallback paths, and coordinate with the broader Feishu table-limit tracking so direct outbound and reply-dispatcher behavior do not diverge.
