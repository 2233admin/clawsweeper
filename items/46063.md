## Summary

Current `main` still loads workspace skills into the shared skills status report used by both the CLI and the web Skills tab. The only concrete follow-up in the issue thread points to malformed `SKILL.md` YAML/frontmatter rather than an OpenClaw regression.

## Close Comment

Closing this based on a Codex review of current `main`. I checked the current skills path and OpenClaw still loads `<workspace>/skills` into the shared `SkillStatusReport` (`src/agents/skills/workspace.ts:533`), serves that to the web UI via `skills.status` (`src/gateway/server-methods/skills.ts:68`), and both the CLI and Skills tab consume that same report (`src/cli/skills-cli.ts:27`, `ui/src/ui/views/skills.ts:102`). The loader intentionally skips skill folders whose `SKILL.md` frontmatter cannot be parsed or is missing required metadata like `description` (`src/agents/skills/local-loader.ts:60`), and the docs still require valid YAML frontmatter with `name` and `description` (`docs/tools/skills.md:144`, `docs/tools/creating-skills.md:26`). That matches the follow-up in this thread that malformed YAML headers were the cause, so I do not have evidence of an OpenClaw regression to keep open here. If a correctly formatted minimal `SKILL.md` under the active workspace still does not appear, please open a new issue with that sample plus `openclaw skills list --json` output and the exact OpenClaw version.
