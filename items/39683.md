## Summary

Keep PR #39683 open. Current main still has no Feishu/Lark-specific create/update validation for unresolved announce cron delivery targets, and the linked canonical bug #39599 remains open. The submitted patch is useful but still too narrow because it checks only raw `delivery.to` inside full-job validation, while current delivery resolution can derive targets from session/plugin routing and `applyJobPatch` runs validation for unrelated updates too.

## Best Possible Solution

Keep the PR open and request rework. The fix should validate Feishu/Lark announce delivery against the resolved target path at create/update time, accept explicit `delivery.to` and deterministic session/plugin resolution, avoid blocking unrelated updates to legacy jobs, keep Feishu/Lark-specific target hints near the channel/plugin surface, and add focused create/update regression coverage. If #40031 or a narrower follow-up lands that resolved-target validation first, this PR can then close as superseded.

## Risks / Open Questions

- Closing this PR now would drop a focused implementation thread while #39599 remains open and current main still lacks the requested fail-fast validation.

- Because validation runs after any `applyJobPatch`, the submitted raw-field guard could block simple remediation updates such as disabling legacy Feishu/Lark jobs that lack `delivery.to`.
- The broader related PR #40031 is still open and not a merged or clearly canonical replacement for this focused fix.
