## Summary

Keep #69735 open. Current main still passes native Ollama `response.message.tool_calls[].function.arguments` through unchanged in `buildAssistantMessage`, so string arguments can still reach downstream tool handling. Existing fixes/tests cover adjacent replay and parsing paths, and open PR #69910 is the narrow canonical fix path, but it has not landed on main.

## Best Possible Solution

Keep the issue open until the native Ollama inbound path is fixed. The best path is to land or recreate #69910 narrowly: normalize `response.message.tool_calls[].function.arguments` inside `buildAssistantMessage` with the existing unsafe-integer-preserving helper, keep object passthrough behavior, fall back safely for malformed/non-object strings, widen the local response type to reflect runtime inputs, and add regression tests for valid string args, malformed strings, unsafe integers, and object passthrough.
