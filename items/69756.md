## Summary

Keep open. Current `main` still routes oversized `chat.history` entries to `[chat.history omitted: message too large]`, and compaction summaries are stored in a `summary` field that `chat.history` sanitization does not truncate. That makes the reported post-compaction placeholder behavior still plausible on current code rather than obsolete or already fixed.

## Evidence

- **Compaction summaries are first-class transcript entries:** OpenClaw persists compaction output as `role: "compactionSummary"` with a raw `summary: string` field, so compaction data is not stored in the normal `content`/`text` fields.
