## Summary

Current `main` does not show a fix for this. The daemon install path still persists only durable service env (`~/.openclaw/.env` / config `env`) plus auth-profile env refs, while Discord channel SecretRef env tokens still resolve from the daemon process environment only. The Discord setup docs still tell users to `export DISCORD_BOT_TOKEN` and store `channels.discord.token` as an env ref before restarting, which remains unsafe for systemd unless the key is also made visible to the daemon via service env, `~/.openclaw/.env`, or `env.shellEnv`.

## Evidence

There is no path here that lifts channel SecretRef env refs like `channels.discord.token` into the unit environment.
