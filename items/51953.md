## Summary

Keep open. Current `main` still rejects password-mode handshakes unless `connectAuth.password` is present; token-only clients remain unsupported, and the same code is present in the latest release tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Evidence

- **Password mode still ignores token field on main:** `authorizeGatewayConnect` reads only `connectAuth?.password` in password mode and returns `password_missing` when that field is absent. There is no `connectAuth.token` fallback.
  - file: [src/gateway/auth.ts:491](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/gateway/auth.ts#L491)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **WS handshake path passes token-only auth through unchanged:** `resolveSharedConnectAuth` forwards `token` and `password` independently. A client that sends only `connectAuth.token` reaches shared-auth evaluation with `password` unset, so password mode falls into the missing-password branch.
  - file: [src/gateway/server/ws-connection/auth-context.ts:44](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/gateway/server/ws-connection/auth-context.ts#L44)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Current tests do not cover or prove a token fallback in password mode:** The password-mode tests only assert the `null` and wrong-`password` cases; there is no passing test for `connectAuth: { token: ... }` under `auth.mode: "password"`.
  - file: [src/gateway/auth.test.ts:315](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/gateway/auth.test.ts#L315)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Other code compensates by duplicating bearer secrets into both fields:** HTTP gateway auth constructs `connectAuth` as `{ token, password: token }`, which works around the server's password-only check rather than proving the underlying bug is fixed.
  - file: [src/gateway/http-utils.ts:127](https://github.com/openclaw/openclaw/blob/aa3a5f9ff7907c2a41d6259d24b529bda41f293a/src/gateway/http-utils.ts#L127)
  - sha: [aa3a5f9ff790](https://github.com/openclaw/openclaw/commit/aa3a5f9ff7907c2a41d6259d24b529bda41f293a)
- **Latest release tag still has the same password-only branch:** The release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` shows the same `const password = connectAuth?.password;` logic in `src/gateway/auth.ts`, so this was not shipped as fixed in `v2026.4.22`.
  - file: [src/gateway/auth.ts:492](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/gateway/auth.ts#L492)
