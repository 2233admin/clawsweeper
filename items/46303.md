## Summary

Keep open. Current main still lacks the central restart-time drain behavior: inbound debounce state is closure-local with no global flush surface, the SIGUSR1 run loop only drains active tasks/runs/runtime dependency installs, and there is no all-followup-queue drain waiter. The PR remains a plausible implementation candidate, but it is broad and security-sensitive enough to need maintainer review rather than cleanup closure.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best maintainer path is a narrower revision that adds a lifecycle-safe, controlled drain surface for active inbound debouncers; preserves public SDK compatibility; flushes debounced inbound work at the correct restart phase; waits for active turns, followup queues including dropped-summary state, and any channel worker queues that can receive flushed work; and adds focused restart-ordering and teardown-race tests.

## Risks / Open Questions

- The PR branch adds a mutable process-wide registry keyed by Symbol.for, which could let same-process plugin code interfere with restart flushing unless the final design exposes only controlled operations.
- The PR changes plugin-facing debounce return shapes and flushKey's apparent contract; current main exports createInboundDebouncer through public SDK subpaths, so compatibility needs an explicit plan.
- The proposed restart drain has channel lifecycle edge cases: some channels enqueue through worker queues, some teardown paths can still receive ingress while unregistering, and followup queues also carry dropped-summary state.
- The branch is large and recently active with merge/test-isolation fallout in the discussion, so keeping it open does not mean landing it unchanged.
