## Summary

Keep open. Current `main` still misses the Slack message-event liveness wiring described in the PR, so the underlying bug is real, but the proposed patch also appears to double-fire `trackEvent` for normal handled messages because `createSlackMessageHandler` already updates liveness.

## Evidence

- **Main still omits `trackEvent` when registering Slack message events:** `registerSlackMonitorEvents` passes `trackEvent` to reactions, members, channels, pins, and interactions, but not to `registerSlackMessageEvents`, so message and `app_mention` events are the only inbound Slack events not wired into the liveness callback on current `main`.
  - file: [extensions/slack/src/monitor/events.ts:18](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/slack/src/monitor/events.ts#L18)
  - command: `nl -ba extensions/slack/src/monitor/events.ts | sed -n '1,80p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Message-event handlers on `main` do not call the liveness callback:** `registerSlackMessageEvents` only checks `shouldDropMismatchedSlackEvent`, then either enqueues subtype system events or delegates to `handleSlackMessage`; there is no event-level `trackEvent` call in either the `message` or `app_mention` handler. That matches the PR's reported gap for subtype system events and DM-skipped mentions, which never reach `createSlackMessageHandler`.
  - file: [extensions/slack/src/monitor/events/messages.ts:148](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/slack/src/monitor/events/messages.ts#L148)
  - command: `nl -ba extensions/slack/src/monitor/events/messages.ts | sed -n '140,240p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Handled Slack messages already update liveness inside `createSlackMessageHandler`:** For accepted, non-duplicate messages, `createSlackMessageHandler` calls `trackEvent?.()` before thread resolution and enqueue. The existing tests assert exactly one callback for accepted messages and none for duplicates or dropped subtypes. That means adding another unconditional `trackEvent?.()` before `handleSlackMessage(...)` in the registrar would double-update liveness on the normal `message` and channel `app_mention` paths.
  - file: [extensions/slack/src/monitor/message-handler.ts:203](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/slack/src/monitor/message-handler.ts#L203)
  - command: `nl -ba extensions/slack/src/monitor/message-handler.ts | sed -n '200,280p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Current message-event tests do not cover the liveness gap yet:** The existing `messages.test.ts` cases verify routing for regular messages, channel/group messages, subtype system events, DM-skipped mentions, and channel mentions, but they do not assert any `trackEvent` behavior. That explains how the missing thread-through on current `main` escaped coverage and why additional tests are needed, but not necessarily with the PR's current callback placement.
  - file: [extensions/slack/src/monitor/events/messages.test.ts:198](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/slack/src/monitor/events/messages.test.ts#L198)
  - command: `nl -ba extensions/slack/src/monitor/events/messages.test.ts | sed -n '190,350p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Liveness callback updates both socket-health timestamps:** The shared Slack `trackEvent` callback sets both `lastEventAt` and `lastInboundAt`. Missing or duplicate calls therefore directly affect socket-health status, so this should stay open for a corrected fix rather than conservative cleanup.
  - file: [extensions/slack/src/monitor/provider.ts:292](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/slack/src/monitor/provider.ts#L292)
  - command: `nl -ba extensions/slack/src/monitor/provider.ts | sed -n '286,322p'`

## Risks / Open Questions

- Current `main` can still stale-flag a healthy Slack socket when inbound traffic consists only of message events that bypass the existing liveness callback branches.
- Merging the PR as currently described would likely add duplicate `trackEvent` writes for normal `message` and non-DM `app_mention` flows, increasing status churn without fixing the branch placement cleanly.
