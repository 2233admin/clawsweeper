## Summary

Current `main` already contains targeted fixes for this `profile="user"` / Chrome MCP attach path: it now surfaces `DevToolsActivePort` failures as a specific existing-session connectivity error instead of a generic tab-wait timeout, and the Chrome MCP session layer now resets poisoned existing-session sessions on timeout/transport failure so retries reconnect cleanly. Those fixes are covered by regression tests and shipped in releases after the report.

## Close Comment

Closing from a Codex review because current `main` already implements the needed fixes for this existing-session `profile="user"` path. OpenClaw now surfaces `DevToolsActivePort` attach failures as a specific Chrome MCP connectivity error in `extensions/browser/src/browser/server-context.availability.ts:126`, with regression coverage in `extensions/browser/src/browser/server-context.existing-session.test.ts:139`. It also resets poisoned Chrome MCP sessions on timeout/transport failure so retries reconnect cleanly instead of requiring a wedged session or gateway restart (`extensions/browser/src/browser/chrome-mcp.ts:308`, `extensions/browser/src/browser/chrome-mcp.test.ts:430`). These fixes were shipped in `v2026.4.20` and `v2026.4.22` per `appcast.xml:207` and the corresponding browser fix entries, so this report looks implemented on current OpenClaw rather than still actionable on `main`. If you can still reproduce the attach failure on `v2026.4.22` or newer, please open a new issue with current logs and exact reproduction details.
