## Summary

Keep open. Current main has improved workspace-skill loading, prompt compaction, and watcher-based refresh, but the linked restart-stale snapshot diagnosis is still plausible in current code: persisted session snapshots can have version 0, and a restarted process also starts its in-memory skills version at 0, so the refresh check can reuse a stale snapshot instead of rebuilding from the current skills inventory.

## Best Possible Solution

Keep this open until maintainers either add restart-safe skills inventory invalidation or confirm the current session-snapshot contract is intended. The strongest fix would compute a deterministic fingerprint of the visible skill roots, loaded skill metadata, filters, eligibility-relevant config, and prompt limits, store it with the session skillsSnapshot, and rebuild when it changes. Add regression coverage for an existing session across process restart with a workspace skill added while no watcher was active; if the remaining reports are instead frontmatter/filter/budget issues, add explicit diagnostics or docs guidance before manual closure.
