## Summary

Keep open. Current `main` still hard-fails a CDP socket close instead of reconnecting, so the underlying bug is not already fixed, and the proposed PR introduces a real callback-replay risk for non-idempotent CDP commands that needs maintainer review rather than auto-closure.

## Evidence

- **Current main still uses a single-attempt CDP socket flow:** `withCdpSocket` opens one WebSocket, rejects on `close`, then executes `fn(send)` exactly once. There is no reconnect or retry path on current `main`. That means the report is not implemented on `main`, and it also shows the current contract that the PR would change.
  - file: [extensions/browser/src/browser/cdp.helpers.ts:412](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/browser/src/browser/cdp.helpers.ts#L412)

  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Current tests still lock in failure on mid-call socket close:** The browser extension test suite explicitly expects `withCdpSocket(...)` to reject with `CDP socket closed` when the socket closes after a command is sent. That makes the linked bug still current on `main`, not obsolete or already resolved elsewhere.
  - file: [extensions/browser/src/browser/cdp.helpers.internal.test.ts:243](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/browser/src/browser/cdp.helpers.internal.test.ts#L243)

The PR review concern is valid: a retry loop that re-runs the full callback can duplicate side effects or leak orphaned tabs.

- **PR discussion already identified a concrete regression risk:** The provided PR review thread includes an unresolved P1/P2 concern that replaying `fn(send)` on retry can duplicate `Target.createTarget` and side-effectful `Runtime.evaluate` calls. Conservative cleanup should not auto-close a PR with a still-relevant bug report plus an unresolved correctness concern in the proposed fix.

## Risks / Open Questions

- The original cross-host/CDP transient-close reliability problem still appears to exist on current `main` because transport closes are still terminal in `withCdpSocket`.
- Merging this PR as written would broaden retry behavior by replaying full callbacks, which can duplicate non-idempotent CDP operations and create behavioral regressions.
