## Summary

Current `main` still keeps group/channel conversations isolated from DM sessions by design. OpenClaw has partial cross-session tools and optional memory plugins, but the requested group-to-DM context synchronization is not implemented as an automatic core behavior.

## Evidence

- **Session docs still describe isolated group sessions:** The current session docs say direct messages are shared by default, but group chats and rooms/channels are isolated. They also describe `session.identityLinks` only as a way to link the same person across channels so they share one DM session.
  - file: [docs/concepts/session.md:14](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/docs/concepts/session.md#L14)
  - command: `nl -ba docs/concepts/session.md | sed -n '1,120p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Channel routing docs still separate groups from DMs:** Current routing docs state DMs collapse to the agent main session by default, while groups and channels remain isolated per channel.
  - file: [docs/channels/channel-routing.md:24](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/docs/channels/channel-routing.md#L24)
  - command: `nl -ba docs/channels/channel-routing.md | sed -n '1,140p'`
  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **Routing code applies identity linking only to direct-message scopes:** `buildAgentPeerSessionKey()` only resolves `identityLinks` inside the `peerKind === "direct"` branch. Non-direct peers always return separate `agent:<id>:<channel>:<peerKind>:<peerId>` keys, so group/chat sessions do not merge with DMs.
  - file: [src/routing/session-key.ts:130](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/routing/session-key.ts#L130)

- **Cross-session support exists only as explicit tools:** OpenClaw does have `sessions_list`, `sessions_history`, and `sessions_send`, but the docs describe them as explicit agent tools for reading another session or sending a message to it, not automatic session-context synchronization.
  - file: [docs/concepts/session-tool.md:13](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/docs/concepts/session-tool.md#L13)

- **Cross-session recall is optional memory/plugin functionality:** Current docs describe transcript indexing via `memorySearch.experimental.sessionMemory` and the optional Honcho plugin for cross-session memory across channel switches. Those are workarounds/optional features, not the default session-routing behavior requested here.
