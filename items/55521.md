## Summary

Keep open. Current `main` does not implement the proposed early inbound Feishu `Typing` acknowledgment, and the PR as described would bypass existing Feishu reply gates and the `typingIndicator` opt-out semantics rather than matching current channel behavior.

## Evidence

- **No early ack on current main:** `handleFeishuMessage()` logs the received message and then proceeds into group/DM policy resolution; there is no `messageReaction.create(...Typing...)` call in this function on current `main`. A repo search only finds Feishu `Typing` reaction creation in the managed typing helpers, not in inbound message handling.
  - file: [extensions/feishu/src/bot.ts:365](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/feishu/src/bot.ts#L365)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '320,520p'; rg -n "messageReaction\.create\(|reaction_type: \{ emoji_type: \"Typing\" \}" extensions/feishu/src -S`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Inbound messages have silent-drop gates before reply work:** Current Feishu handling returns early for disabled groups, disallowed groups, disallowed senders, and mention-required messages that do not mention the bot. An immediate reaction inserted before these checks would acknowledge messages that OpenClaw intentionally ignores.
  - file: [extensions/feishu/src/bot.ts:417](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/extensions/feishu/src/bot.ts#L417)
