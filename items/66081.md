## Summary

Keep this PR open. Current main still finalizes `codex-cli` replies from stdout/JSONL only, while the PR is a plausible implementation candidate for the `--output-last-message` artifact path. It should not merge as-is because the current head has unresolved parser scoping, argument ordering, temp cleanup, and verbose logging review findings.

## Best Possible Solution

Keep this PR open and ask for a scoped revision: insert the Codex `--output-last-message` option before positional prompt args, restrict the new JSONL content fallback to `codex-cli` or a named dialect, read a non-empty artifact as the final reply while preserving stdout-derived session/usage/raw text, mask or abbreviate the artifact path in verbose logs, create and clean the artifact inside the existing guarded cleanup flow, rebase over current main, and retain focused tests for artifact precedence, empty-artifact fallback, and Codex JSONL message-content parsing.

## Risks / Open Questions

- If merged as-is, the new Codex JSONL text fallback can affect every JSONL backend, not just `codex-cli`, and may incorrectly turn status, diagnostic, or tool-result fields into final reply text.
- If merged as-is, appending `--output-last-message` after the positional prompt can make Codex argv parsing fragile for arg-mode runs; the flag should be inserted before prompt placement or via a backend-owned base-arg seam.
- The PR touches a command execution path and introduces a temp artifact file. There is no observed supply-chain expansion, but verbose logging should mask or abbreviate the generated path and artifact creation should live inside the existing guarded cleanup flow.
- The PR head appears based on an older runner return shape than current main; it should be rebased and verified so current `rawText` and `finalPromptText` behavior is preserved.
