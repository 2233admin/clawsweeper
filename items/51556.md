## Summary

Keep open. Current `main` still swallows non-`ENOENT` allowlist read errors, and the PR’s sync-only change would make sync and async behavior diverge rather than fully fixing the bug described in the discussion.

## Evidence

- **Async allowlist reader only rethrows `stat` failures:** `readAllowFromFileWithExists` rethrows non-`ENOENT` from `fs.promises.stat`, but after a successful stat it delegates the file read to `readJsonFileWithFallback` instead of handling read errors itself.
  - file: [src/pairing/allow-from-store-file.ts:185](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/pairing/allow-from-store-file.ts#L185)
  - command: `nl -ba src/pairing/allow-from-store-file.ts | sed -n '180,223p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **Shared async JSON helper still swallows non-`ENOENT` read failures:** `readJsonFileWithFallback` returns the fallback value with `exists: false` for any caught read error, not just `ENOENT`. That means async allowlist reads still hide `EACCES`/`EIO`-style `readFile` failures on current `main`.
  - file: [src/plugin-sdk/json-store.ts:17](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/plugin-sdk/json-store.ts#L17)

- **Sync allowlist reader still swallows the same classes of failures on `main`:** `readAllowFromFileSyncWithExists` returns `{ entries: [], exists: false }` for non-`ENOENT` errors from both `statSync` and `readFileSync`, so the bug the PR targets is still present in current `main`.

## Risks / Open Questions

- The underlying product contract is still unresolved: either both sync and async allowlist readers should propagate non-`ENOENT` read failures, or both should intentionally fall back cleanly.
- Merging the PR as written would change uncaught sync call sites while leaving async reads unchanged, creating a new inconsistency rather than a complete fix.
