## Summary

Keep this issue open. Current main fixes the Telegram phantom fallback half and shipped that in v2026.4.24, but the same issue still contains a separate Discord inbound blackout report. The maintainer discussion explicitly left the issue open for Discord logs/evidence, and current main does not provide issue-specific source, test, changelog, release, or discussion evidence proving the Docker Discord no-session/no-event behavior is fixed.

## Best Possible Solution

Keep this issue open for the Discord half. Treat Telegram as fixed in v2026.4.24, then collect sanitized Docker Discord startup logs, effective channels.discord config, channels status/probe output, and proof of whether MESSAGE_CREATE reaches DiscordMessageListener before preflight/session creation. If events do not reach the listener, investigate Discord gateway intents/listener registration; if they reach preflight and drop, inspect pairing, DM policy, guild/channel allowlists, requireMention, sender restrictions, replay/dedupe, and inbound worker behavior.
