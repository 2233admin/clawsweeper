## Summary

Keep this PR open. Current main still sends OpenAI Responses, Azure OpenAI Responses, and OpenAI chat completions directly without a provider request concurrency limiter or `LLM_MAX_CONCURRENT` config surface. The PR is relevant, but its current diff is incomplete and not mergeable as-is.

## Best Possible Solution

Keep this PR open for revision or maintainer replacement. A mergeable solution should add a tested provider request limiter that covers OpenAI Responses, Azure OpenAI Responses, and OpenAI chat completions; holds slots until streamed responses finish or abort; rejects queued work promptly on `AbortSignal`; parses invalid config safely; and documents the chosen configuration surface. If the same capability lands separately, close this PR as superseded at that point.

## Risks / Open Questions

- The PR should not merge as-is: the current diff does not cover all claimed SDK surfaces and the proposed limiter releases slots when the stream object is created, not when streaming finishes.
- Invalid or empty concurrency configuration and queued-call abort behavior need explicit tests; otherwise throttling can silently fail or leave cancelled work waiting.
- An env-only `LLM_MAX_CONCURRENT` knob may need maintainer judgment to fit the existing provider/config model and docs.
