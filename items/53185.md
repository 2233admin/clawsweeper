## Summary

Keep this issue open. Current main has substantial built-in `web_fetch` cleanup for hidden HTML and invisible Unicode, and web search providers wrap returned snippets as untrusted external content. The requested product/API surface is still missing: no `content-sanitizer.py` pipeline, no config-driven per-tool preprocessing hook, no external/plugin pre-model output interception path, no shared `web_search` sanitizer decision point, and no `threat_level` or HIGH/CRITICAL block semantics.

## Best Possible Solution

Keep this issue open but narrow it. Treat `web_fetch` hidden-content stripping as substantially implemented, then decide whether OpenClaw should add a trusted pre-model tool-result preprocessing API for `web_fetch`/`web_search`, extend bundled-only middleware with a documented external trust policy, or split broader taint/risk-class enforcement into #54508 while tracking the custom script/preprocessing hook here.

## Risks / Open Questions

- Closing as implemented would conflate the shipped `web_fetch` hidden-content cleanup with the still-missing `web_search`, custom preprocessing, and structured threat decision path.
- A script-based pre-model preprocessing hook is security-sensitive because it would execute configured code and rewrite tool output before model ingestion; it needs an explicit trust model, failure semantics, auditability, and runtime coverage.
- Related #54508 overlaps on broader taint/risk-class work, but it does not supersede this issue's concrete custom script/preprocessing API question.
