## Summary

Keep open. Current main has partial overlap: it can resolve an existing session by `sessionId` and send into the resolved session key, and it has limited checkpoint/thread branching paths. It does not expose the requested general `forkSession(id)` capability for an arbitrary existing session, nor a `continue: true`/most-recent-session API, so the central feature request is still actionable in core session/gateway surfaces.

## Best Possible Solution

Keep the issue open and implement this as a core session/gateway capability, likely by adding a documented `sessions.fork` operation that accepts a key or sessionId and creates a branched transcript-backed session, plus a `continue`/most-recent resolution option with clear agent/visibility scoping. Reuse `sessions.resolve` for exact ID lookup, preserve existing checkpoint branching semantics, and add protocol docs plus gateway tests for fork, resume-by-ID send, and continue-most-recent behavior.
