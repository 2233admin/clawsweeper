## Summary

Keep #71770 open. Current main has ordered model fallbacks, auth-profile cooldowns, provider usage/quota display, and fallback notices, but I did not find a supported path that proactively switches providers/models before a call based on configured token, quota, or cost thresholds, nor reverse-failover semantics when a quota window resets. The existing plugin hook can override a model before resolve, but the public event is prompt/attachment scoped and does not provide a complete core usage-routing policy surface.

## Best Possible Solution

Keep the issue open for maintainer product/design work in the model selection and fallback path. A solid implementation would add a documented config surface for token, quota, or cost thresholds; evaluate usage snapshots and local cost state before selecting each model candidate; integrate with `runWithModelFallback` without weakening existing cooldown/error fallback; emit user-visible and lifecycle notifications for threshold switches; and define reset/reverse-failover semantics. If maintainers prefer this as plugin-owned policy, the core work should first expose a deliberate usage-aware routing API rather than relying on the current prompt-only `before_model_resolve` hook.

## Risks / Open Questions

- Provider usage data is uneven across providers, so threshold semantics need clear handling for providers with no quota API, multiple windows, or only post-response token usage.
- Automatic switching can increase costs or surprise users unless fallback order, threshold units, notifications, and reverse-failover behavior are explicit and auditable.
- A simple plugin-level pre-resolve override could cover some custom routing, but it would not fully integrate with current fallback attempts, cooldown summaries, persisted fallback state, or reset-window recovery.
