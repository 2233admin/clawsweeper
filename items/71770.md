## Summary

Keep #71770 open. Current main has ordered model fallbacks, auth-profile cooldown rotation, provider usage/quota reporting, and local cost/session usage surfaces, but I found no supported config or runtime path that proactively switches models/providers before a request based on real-time token, quota, or cost thresholds. The request remains a concrete core model-routing feature rather than an already-shipped behavior or a pure ClawHub/plugin item.

## Best Possible Solution

Keep the issue open for product/design work in the core model selection and fallback path. A strong implementation would add a documented threshold policy surface, evaluate available provider usage snapshots and local session/cost data before selecting each candidate, integrate with `runWithModelFallback` without weakening existing cooldown/error fallback behavior, emit user-visible lifecycle notifications on threshold switches, and define reset/reverse-failover semantics per quota window. Provider plugins can keep owning usage snapshot fetch/auth hooks, but the generic routing policy likely belongs in core unless a deliberate plugin policy API is designed first.

## Risks / Open Questions

- Provider quota/usage APIs are uneven, so a future implementation needs clear behavior for providers without pre-call quota snapshots or only post-response token usage.
- Automatic switching can surprise users or increase spend unless thresholds, fallback order, reverse failover, and notifications are explicit and auditable.
- Related implemented work such as configurable rate-limit profile rotations (#52383) is reactive 429/cooldown handling and does not supersede proactive token/cost threshold routing.
