## Summary

Keep open. Current `main` still does not implement Mattermost `filePath`/`path` media fallbacks in `handleAction`, so the reported parity bug remains. The PR is directionally correct, but the review discussion surfaced a remaining blank-`media` edge case and missing regression tests, so there is not strong evidence for conservative cleanup closure.

## Evidence

- **Current Mattermost action path still only reads `params.media`:** On current `main`, Mattermost `handleAction` sets `mediaUrl` only from `params.media`; it does not check `filePath` or `path`.
  - file: [extensions/mattermost/src/channel.ts:193](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/mattermost/src/channel.ts#L193)
  - command: `nl -ba extensions/mattermost/src/channel.ts | sed -n '185,205p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Mattermost runtime only uploads when `mediaUrl` is present:** `sendMessageMattermost` only loads and uploads outbound media when `opts.mediaUrl` is truthy, so omitting the fallback in `handleAction` leaves local-path payloads unhandled on this path.
  - file: [extensions/mattermost/src/mattermost/send.ts:426](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/mattermost/src/mattermost/send.ts#L426)
  - command: `nl -ba extensions/mattermost/src/mattermost/send.ts | sed -n '395,470p'`

- **Sibling channels already implement the fallback parity:** Discord currently resolves `media` -> `path` -> `filePath`, and Google Chat resolves `media` -> `filePath` -> `path`, matching the PR's stated parity rationale.
  - file: [extensions/discord/src/actions/handle-action.ts:85](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/discord/src/actions/handle-action.ts#L85)
  - command: `nl -ba extensions/discord/src/actions/handle-action.ts | sed -n '80,92p'; nl -ba extensions/googlechat/src/actions.ts | sed -n '118,130p'`

  - file: [extensions/mattermost/src/channel.test.ts:399](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/extensions/mattermost/src/channel.test.ts#L399)

- **PR discussion identified an unresolved edge case:** The provided review discussion notes the proposed fallback chain still short-circuits on blank/whitespace `media`, preventing fallback to `filePath` or `path` in mixed payloads, and also calls out missing test coverage for the new paths.

## Risks / Open Questions

- The PR as discussed would still leave mixed payloads broken when `media` is present but blank, because the fallback chain keys off `typeof params.media === "string"` instead of normalized optional-string semantics.
- There is no current regression test on the Mattermost `handleAction` path for `filePath` or `path`, so even a correct fix would be easy to regress without adding coverage.
