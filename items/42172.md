## Summary

Current `main` still appears to have the reported bug. `src/gateway/server-methods/agent.ts` dispatches gateway `agent` runs straight into `agentCommandFromIngress(...)` without creating/registering a `chatAbortControllers` entry or wiring an `abortSignal`, while `chat.abort` only aborts runs present in that map. The `chat.send` path still has the expected registration and abort-signal plumbing, so the gap remains specific to `agent` RPC runs.

## Evidence

- **Agent RPC path still lacks abort registration:** `dispatchAgentRunFromGateway` calls `agentCommandFromIngress(params.ingressOpts, ...)` directly and never creates an `AbortController`, never inserts into `context.chatAbortControllers`, and never cleans up an entry afterward.
  - file: [src/gateway/server-methods/agent.ts:229](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/gateway/server-methods/agent.ts#L229)
  - command: `nl -ba src/gateway/server-methods/agent.ts | sed -n '229,305p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **chat.abort only works for tracked runs:** `chat.abort` looks up the run in `context.chatAbortControllers`; if no entry exists it returns `{ ok: true, aborted: false, runIds: [] }`. That means unregistered `agent` runs cannot be stopped through this RPC.
  - file: [src/gateway/server-methods/chat.ts:1956](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/gateway/server-methods/chat.ts#L1956)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '1950,2015p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **chat.send still uses the intended abort plumbing:** The `chat.send` handler creates an `AbortController`, stores it in `context.chatAbortControllers`, and passes `abortController.signal` into the reply pipeline, which is the behavior the issue says is missing from `agent` runs.
  - file: [src/gateway/server-methods/chat.ts:2241](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/gateway/server-methods/chat.ts#L2241)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '2240,2256p' && nl -ba src/gateway/server-methods/chat.ts | sed -n '2498,2508p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Abort support exists downstream but is not wired from gateway agent runs:** Agent command options still accept `abortSignal`, and `agent-command` still forwards `opts.abortSignal` into result/build execution paths. The missing piece is the gateway `agent` handler wiring.
  - file: [src/agents/pi-tools.ts:285](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/agents/pi-tools.ts#L285)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '260,320p' && nl -ba src/agents/agent-command.ts | sed -n '520,546p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)

## Risks / Open Questions

- Clients that use the gateway `agent` RPC instead of `chat.send` still cannot stop an in-flight run via `chat.abort`.
- Because the run remains active until completion, follow-up work on the same lane/session can stay blocked behind the stale run and produce confusing session state.
