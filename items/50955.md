## Summary

Keep open. Current `main` has some overlapping follow-up fixes, but it does not implement this PR’s core unification: native ACP `sessions_spawn(... thread:true, mode:"session")` still uses a different placement/conversation path than `/acp spawn`, so the Discord current-thread/current-surface cases described here are not fully superseded.

## Evidence

- **Native ACP spawn still picks placement from channel default, not current context:** `prepareAcpThreadBinding` resolves `placementToUse` from `resolveChannelDefaultBindingPlacement(policy.channel)` / capability fallback, without consulting the requester’s active thread/conversation when deciding `current` vs `child`. That means the native `sessions_spawn` ACP path is still structurally different from the `/acp spawn` path this PR is trying to unify.
  - file: [src/agents/acp-spawn.ts:610](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/src/agents/acp-spawn.ts#L610)

  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **/acp spawn still uses a separate current-vs-child decision path:** `bindSpawnedAcpSessionToThread` in the command path checks the active context (`threadId`, current conversation) and calls `resolveThreadBindingPlacementForCurrentContext`, which can choose `current` when already inside a thread. That separate logic is exactly the divergence the PR describes; it has not been unified on `main`.
