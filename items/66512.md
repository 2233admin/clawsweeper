## Summary

Keep open. Current `main` still installs a default systemd user unit that relies on systemd’s default `Type=simple`, and the gateway startup path only logs readiness; there is no `sd_notify`/`NOTIFY_SOCKET` integration on `main`. OpenClaw has added caller-side health polling for some flows like `openclaw update`, but that does not change the systemd readiness contract for generic `systemctl restart` consumers.

## Evidence

- **Canonical systemd unit still lacks notify semantics:** `buildSystemdUnit()` renders the service with `ExecStart`, restart policy, and timeouts, but no `Type=notify` or `NotifyAccess=` entry. On systemd that means the unit uses the default `Type=simple` behavior.
  - file: [src/daemon/systemd-unit.ts:67](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/daemon/systemd-unit.ts#L67)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Gateway readiness is only logged:** Gateway startup emits a `ready (...)` log line after plugin load, but this path does not send any systemd notification or other supervisor readiness signal.
  - file: [src/gateway/server-startup-log.ts:28](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/gateway/server-startup-log.ts#L28)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Docs still show the simple/default unit shape:** The Linux docs’ manual unit example includes the same service fields and likewise omits `Type=notify`, matching the current implementation rather than the requested readiness model.
  - file: [docs/platforms/linux.md:76](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/docs/platforms/linux.md#L76)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **No sd_notify implementation present on main:** A repo search found no `NOTIFY_SOCKET`, `sd_notify`, `READY=1`, or `WATCHDOG=1` handling in the current source tree.
  - command: `rg -n "Type=notify|NOTIFY_SOCKET|sd_notify|READY=1|WATCHDOG=1|NotifyAccess" src scripts docs test --glob '!docs/.vitepress/**' --glob '!**/dist/**'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Current mitigation is CLI-side polling, not systemd readiness:** `openclaw update` now waits for `waitForGatewayHealthyRestart()` after initiating the restart, which mitigates update-specific races, but this is a caller-side health wait and does not help external `systemctl restart && openclaw gateway health` flows.
  - file: [src/cli/update-cli/update-command.ts:781](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/cli/update-cli/update-command.ts#L781)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Health wait helper is an explicit post-restart probe:** The restart health helper polls runtime state and probes the WebSocket listener for up to 60 seconds; it confirms the project currently solves readiness in the CLI instead of via systemd `Type=notify`/`sd_notify`.
  - file: [src/cli/daemon-cli/restart-health.ts:17](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/cli/daemon-cli/restart-health.ts#L17)
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
