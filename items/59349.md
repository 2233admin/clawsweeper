## Summary

Keep open. Current main still binds exec approval requests and node exec follow-up events to sessionKey, not the originating sessionId. The code has coverage for draining stale system events that were already queued before /new, but I did not find an equivalent expectedSessionId/original session guard for late approval resolution or node exec events after the sessionKey has been rebound.

## Best Possible Solution

Keep the issue open and implement the proposed guard in the core approval/follow-up path: capture the sessionId mapped from sessionKey when the exec approval request is created, carry it through the approval/system.run follow-up contract, and before enqueueing or continuing any approval result compare it against the current sessionKey -> sessionId mapping. If the mapping changed after /new or /reset, log and drop the stale follow-up. Add a regression test where an approval is pending, /new rotates the sessionId under the same sessionKey, and the later allow/deny does not enqueue exec.denied or continuation text into the new transcript.
