## Summary

Current main still does not expose a `--thread-id` flag on `openclaw cron add/create` or `openclaw cron edit`, so the requested CLI enhancement is not implemented. Telegram topic delivery itself is supported today via `--to <chatId>:topic:<threadId>` and the cron delivery model already carries `delivery.threadId`, so the issue is partially mitigated but not resolved as filed.

## Evidence

- **`cron add/create` has no `--thread-id` option:** The command surface defines `--announce`, `--channel`, `--to`, `--account`, and `--best-effort-deliver`, but no `--thread-id`. The constructed `delivery` payload only includes `mode`, `channel`, `to`, `accountId`, and `bestEffort`.
  - file: [src/cli/cron-cli/register.cron-add.ts:100](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/cli/cron-cli/register.cron-add.ts#L100)

  - sha: [6944d7025d9c](https://github.com/openclaw/openclaw/commit/6944d7025d9c0ba4891cbe646ab5dab039d504ca)
- **`cron edit` has no `--thread-id` option either:** The edit command exposes `--announce`, `--channel`, `--to`, `--account`, and best-effort flags, but no `--thread-id`; its delivery patch builder only writes `mode`, `channel`, `to`, `accountId`, and `bestEffort`.
  - file: [src/cli/cron-cli/register.cron-edit.ts:69](https://github.com/openclaw/openclaw/blob/6944d7025d9c0ba4891cbe646ab5dab039d504ca/src/cli/cron-cli/register.cron-edit.ts#L69)

## Risks / Open Questions

- Closing as implemented would be inaccurate because the exact requested CLI flag parity with `openclaw message send --thread-id` is still missing.
- Closing as cannot_reproduce would also be weak: topic delivery is possible via `--to <chatId>:topic:<threadId>`, but the enhancement request for a dedicated `--thread-id` flag remains valid, and the slash-target validation gap mentioned later still appears open.
