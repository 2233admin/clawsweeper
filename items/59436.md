## Summary

Keep open. Current main still does not implement this PR's central optimization: `listSessionsFromStore()` builds every candidate row before `search`, `activeMinutes`, and `limit`, and row construction still does per-row transcript fallback plus a full-store child-session scan. The related issue #57715 is still open, #54269 was closed unmerged and only covered a narrower no-search limit case, and the PR discussion includes useful independent validation plus an addressed review nit.

## Best Possible Solution

Keep the PR open and have maintainers review/rebase it against current main. The best path is to land a behavior-preserving version that applies cheap filters, sort, `activeMinutes`, and `limit` before row enrichment, builds a request-scoped child-session index, preserves search semantics from store-entry fields, and adds regression tests that count/bound row builds or transcript fallback work for limited and searched `sessions.list` calls.
