## Summary

Keep open. Current `main` still leaves the skills snapshot version at `0` after process start, and the refresh gates in both session paths do not rebuild a cached `version: 0` snapshot after a gateway restart. I found no implementation on `main` that addresses the reported restart case; the shipped changelog entry only covers invalidation on `skills.*` config edits, not restart recovery.

## Evidence

- **Version state still resets to zero on current main:** `globalVersion` still initializes to `0`; `getSkillsSnapshotVersion()` returns `Math.max(globalVersion, local)`; and `shouldRefreshSnapshotForVersion()` returns `false` when both cached and next versions are `0`. That is the exact stale-after-restart condition from the report.
  - file: [src/agents/skills/refresh-state.ts:8](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/agents/skills/refresh-state.ts#L8)
  - command: `nl -ba src/agents/skills/refresh-state.ts | sed -n '1,120p'`
  - sha: [d3d9b5738f83](https://github.com/openclaw/openclaw/commit/d3d9b5738f83d92787f57db9186ddcebda60af9a)
- **Long-lived reply sessions still rely on the zero-version gate:** Reply-session refresh still depends on `shouldRefreshSnapshotForVersion(existingSnapshot?.version, snapshotVersion)` before rebuilding. After restart, an existing `skillsSnapshot.version === 0` with `snapshotVersion === 0` will not refresh.
  - file: [src/auto-reply/reply/session-updates.ts:145](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/auto-reply/reply/session-updates.ts#L145)
