## Summary

Keep this issue open. Current main still does not pass an OpenAI SDK `timeout` for the OpenAI Responses, Azure OpenAI Responses, or OpenAI Completions transports. The newer provider timeout seam reaches OpenClaw's guarded fetch, but the pinned OpenAI SDK still wraps custom fetch calls in its own 600000 ms default timeout unless the SDK client or request receives an explicit `timeout`.

## Best Possible Solution

Keep this issue open and implement the fix in the OpenAI-family transport owner path. The likely product direction is to forward the resolved provider request timeout, currently `models.providers.<id>.timeoutSeconds` -> `requestTimeoutMs`, into `OpenAI`/`AzureOpenAI` client construction and/or per-request SDK options for `openai-responses`, `azure-openai-responses`, and `openai-completions`. Add focused tests that mock or inspect the SDK constructor/request options for values above 600 seconds. If the separate exec-tool timeout report is still current, split it into its own issue rather than using it to close this OpenAI transport bug.
