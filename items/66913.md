## Summary

Current main still has the behavior this PR targets: `memory_forget` asks for a `memoryId` while showing only 8-character ID prefixes in the candidate list. The PR is a plausible focused fix, but it should stay open for maintainer review because exposing full deletion UUIDs in visible tool output is a real product/security tradeoff and the PR includes unrelated generated baseline churn.

## Best Possible Solution

Keep this PR open as the active implementation candidate. Maintainers should decide whether the candidate list should expose full UUIDs, use non-authoritative handles, or require explicit confirmation; then remove the unrelated generated baseline churn and run the relevant memory-lancedb typecheck/test lane before merge.
