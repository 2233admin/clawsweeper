## Summary

Keep #69740 open. Current main has not implemented the proposed centralized debug-env helper or production guard, and the provided related-item context does not contain a canonical superseding issue/PR for this debug-flag policy. The PR is not merge-ready as submitted because the stated security goal only covers one callsite and the branch also includes unrelated sessions/UI and install-smoke changes.

## Best Possible Solution

Keep the PR open for maintainer rework rather than auto-close. The best path is a narrow debug-env PR: put the helper in a shared boundary such as `src/infra/env.ts` and export it through `openclaw/plugin-sdk/runtime-env` for plugins, define the production/debug policy explicitly, migrate every intended `OPENCLAW_DEBUG_*` callsite or narrow the documented scope, add regression tests for production and non-production behavior, and remove or split the unrelated sessions/install-smoke changes.

## Risks / Open Questions

- Merging the PR as-is would create false confidence: the stated production debug-flag block is not applied to most listed callsites.
- A blanket `NODE_ENV=production` block is a product/security policy choice because documented production deployments use that environment and may still need explicit operator debug flags during incidents.
- The branch bundles unrelated sessions-limit UI/type changes, CSS formatting churn, and install-smoke timeout changes with the debug-env proposal; those should be split or dropped.
