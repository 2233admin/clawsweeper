## Summary

Keep open. Current `main` has partial diagnostics for Cloudflare/HTML failures, but it does not implement the requested transport-level mitigation or another clear fix for `openai-codex` behind challenged proxies. The report is recent, concrete, and still plausible on current code.

## Evidence

- **No TLS-profile/browser-fingerprint surface:** Provider request overrides expose only standard TLS fields (`ca`, `cert`, `key`, `passphrase`, `serverName`, `insecureSkipVerify`) plus proxy mode/url. There is no `tlsProfile`, browser emulation, or similar transport knob that would address the reported JA3/JA4 challenge path.
  - file: [src/agents/provider-request-config.ts:38](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/agents/provider-request-config.ts#L38)
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **HTTP transport still maps to plain Undici dispatcher settings:** `buildProviderRequestDispatcherPolicy()` converts provider request config into `direct`, `env-proxy`, or `explicit-proxy` dispatchers with ordinary `connect` / `proxyTls` options only. Nothing here changes the client fingerprint beyond normal proxy/TLS settings.
  - file: [src/agents/provider-request-config.ts:545](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/agents/provider-request-config.ts#L545)
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **Codex OAuth runtime still relies on Undici env-proxy bootstrap:** The bundled `openai-codex` runtime only calls `ensureGlobalUndiciEnvProxyDispatcher()` before delegating to PI OAuth helpers. That is proxy wiring, not a Cloudflare-challenge bypass or browser-like transport implementation.
  - file: [extensions/openai/openai-codex-provider.runtime.ts:1](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/extensions/openai/openai-codex-provider.runtime.ts#L1)
  - sha: [7e52223d3275](https://github.com/openclaw/openclaw/commit/7e52223d327555c611e8c91ee2500897e9eab0bf)
- **WebSocket path does not add proxy-specific mitigation:** The OpenAI WebSocket connection path applies request headers and direct TLS client options, but the only socket `agent` used here is the debug proxy agent. Current `main` does not apply provider `request.proxy` as a general WebSocket transport/bypass mechanism.
  - file: [src/agents/openai-ws-connection.ts:428](https://github.com/openclaw/openclaw/blob/7e52223d327555c611e8c91ee2500897e9eab0bf/src/agents/openai-ws-connection.ts#L428)

That fixes misleading messaging, not the underlying reachability problem.
