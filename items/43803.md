## Summary

Keep this issue open. Current main no longer makes every config.patch schedule a direct SIGUSR1 restart, but the specific browser profile path in this report still plans as restart-required because the bundled browser plugin registers the whole browser prefix as a restart prefix. That means browser.profiles.<name>.cdpUrl remains unresolved for the reporter's sandbox browser workflow.

## Best Possible Solution

Keep this issue open as the canonical tracker. Implement granular browser reload registration so safe profile-scoped paths such as browser.profiles.<name>.cdpUrl and likely browser.defaultProfile hot-reload or no-op, while preserving restart-required behavior for service-affecting browser fields. Add reload-plan and config.patch regression coverage for hybrid and hot modes, then update the public configuration docs to match the shipped behavior.
