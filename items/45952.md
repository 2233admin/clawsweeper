## Summary

Current `main` has partial mitigations for reconnect churn, but it does not implement a reconnect-safe outbound message queue or replay path for `chat.send`. The webchat transport still drops pending requests on socket close and reconnects without resending them, so the issue should remain open.

## Evidence

- **No outbound resend queue in the browser client:** `GatewayBrowserClient.request()` sends the frame directly on the live socket and rejects immediately when the socket is not open. There is no persistent client-side queue or resend logic for `chat.send` across reconnects.
  - file: [ui/src/ui/gateway.ts:621](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/ui/src/ui/gateway.ts#L621)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '621,632p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Pending requests are dropped on disconnect:** On WebSocket close, the client calls `flushPending(...)` to reject all in-flight requests, then schedules a reconnect. The reconnect path does not replay those dropped requests.
  - file: [ui/src/ui/gateway.ts:334](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/ui/src/ui/gateway.ts#L334)
  - command: `nl -ba ui/src/ui/gateway.ts | sed -n '334,376p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Chat queue is for busy-run deferral, not reconnect recovery:** `handleSendChat()` only queues messages when a run is already active; if disconnected it returns early. The reconnect drain in `connectGateway()` is limited to the existing `chatQueue` and only resumes it after seq-gap recovery, not after ordinary transport reconnects.
  - file: [ui/src/ui/app-chat.ts:323](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/ui/src/ui/app-chat.ts#L323)
  - command: `nl -ba ui/src/ui/app-chat.ts | sed -n '318,399p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)
- **Current UI blocks obvious disconnected sends but does not solve replay:** The chat view disables the textarea/button when disconnected and the Enter handler bails if `props.connected` is false. That reduces the old silent-loss path, but it is not the requested queue/ACK resend implementation.
  - file: [ui/src/ui/views/chat.ts:1046](https://github.com/openclaw/openclaw/blob/c1383680405879a3c3f43dceb20a694f54d3c13c/ui/src/ui/views/chat.ts#L1046)
  - command: `nl -ba ui/src/ui/views/chat.ts | sed -n '1038,1056p;1230,1252p'`
  - sha: [c13836804058](https://github.com/openclaw/openclaw/commit/c1383680405879a3c3f43dceb20a694f54d3c13c)

## Risks / Open Questions

- The original report's 'silent loss' symptom may be partially mitigated on current `main`, because disconnected compose is blocked and failed sends surface as request errors; that does not change the absence of reconnect-safe resend behavior.
- There is no focused regression test covering `chat.send` during a transport reconnect window, so closing this would be premature.
