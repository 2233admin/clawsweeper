## Summary

Keep #57752 open. Current main has substantial Kimi/Moonshot provider work, including a bundled Kimi Coding provider and user-facing docs, but I did not find source, tests, or changelog evidence that the reported `message_start`/`message_stop` event-order failure is fixed. The exact error is still classified as an invalid provider streaming response, which improves the displayed copy but does not prove Kimi/Moonshot streams are accepted.

## Best Possible Solution

Keep the issue open until maintainers either add a targeted fixture/live regression proving Kimi/Moonshot streams no longer fail on the reported `message_start` before `message_stop` sequence, or explicitly decide that such upstream SSE ordering remains unsupported and update docs/error guidance. The likely code path to investigate is the Kimi/Moonshot provider stream transport around `extensions/kimi-coding/stream.ts` and the Anthropic Messages stream parser used by the embedded runner.
