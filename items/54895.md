## Summary

Current main still lacks inter-session progress streaming for `sessions_send`. The issue remains concrete and actionable: `sessions_send` only supports fire-and-forget or synchronous wait semantics, `agent.wait` can still produce a wait timeout after accepted delivery, A2A announce remains completion-only, and gateway event routing is still connection/session-subscriber oriented rather than an agent-callable run-event stream.

## Best Possible Solution

Keep this issue open. The best implementation path is an additive core change: a permission-checked `agent.events` long-poll/read API backed by bounded per-run event buffering, plus either a `sessions_subscribe` tool or a `sessions_send` progress option. `sessions_send` should also distinguish successful delivery from wait timeout, and tests should cover long-running sends, event visibility, routing isolation, ACP/Codex behavior, and Telegram/topic-bound delivery.

## Risks / Open Questions

- A future run-event observation API needs explicit authorization checks so one session cannot inspect unrelated runs.
- Forwarded tool/assistant progress may expose sensitive tool args, results, or partial model output unless payloads are redacted or gated by verbosity policy.
- Messaging-channel progress forwarding needs throttling and delivery policy, especially for Telegram topics and other rate-limited channels.
