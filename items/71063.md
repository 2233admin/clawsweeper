## Summary

Keep open. Current `main` still has a plausible `EPIPE` path when a wrapped child process cannot `exec` its target, but this PR's `child.stdin.writable` guard and synthetic test do not match the observed runtime behavior, so there is not high-confidence evidence that the patch actually fixes the bug.

## Evidence

- **Current main writes to child stdin unconditionally:** `createChildAdapter` still writes and ends `child.stdin` whenever it exists; there is no pre-write guard and no async `error` listener on the stream before the initial write path.
  - file: [src/process/supervisor/adapters/child.ts:75](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/process/supervisor/adapters/child.ts#L75)
  - command: `nl -ba src/process/supervisor/adapters/child.ts | sed -n '70,100p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Linux path wraps commands in `/bin/sh`:** On Linux, child commands are wrapped through `/bin/sh -c '...; exec "$0" "$@"'`, so a missing target binary is handled inside the shell child rather than failing the outer spawn immediately.
  - file: [src/process/linux-oom-score.ts:110](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/process/linux-oom-score.ts#L110)
  - command: `nl -ba src/process/linux-oom-score.ts | sed -n '98,115p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Direct reproduction contradicts PR assumption:** A local Node reproduction of the wrapped missing-binary path showed `stdin.writable: true` and still produced `Error: write EPIPE`, so checking `child.stdin.writable` would not have prevented the observed failure in that runtime.
  - command: `node -e 'const {spawn}=require("node:child_process"); const c=spawn("/bin/sh",["-c","echo 1000 > /proc/self/oom_score_adj 2>/dev/null; exec \"$0\" \"$@\"","gemini"],{stdio:["pipe","pipe","pipe"]}); console.log(JSON.stringify({stdin:!!c.stdin,writable:c.stdin?.writable,pid:c.pid})); c.stdin?.write("x",err=>console.log("cb",err&&{code:err.code,message:err.message})); c.stdin?.end(); setTimeout(()=>{},200);'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **PR test covers a synthetic non-writable stub, not the reproduced path:** The PR test forces `stub.child.stdin.writable = false` on a `PassThrough` and asserts no throw/null callback. That matches the bot review follow-up, but it does not exercise the reproduced missing-binary case where stdin was still writable before the `EPIPE`.
  - file: [src/process/supervisor/adapters/child.test.ts:205](https://github.com/openclaw/openclaw/blob/84cc9bf9eac79fc4e7627c45848b20101f343626/src/process/supervisor/adapters/child.test.ts#L205)
  - sha: [84cc9bf9eac7](https://github.com/openclaw/openclaw/commit/84cc9bf9eac79fc4e7627c45848b20101f343626)

## Risks / Open Questions

- Merging this as-is could leave the real missing-binary crash path unfixed while adding a regression test that passes on an unrepresentative stub.
- The bug report itself still looks credible on current `main`, so closing or treating it as resolved would be premature.
