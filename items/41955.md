## Summary

Keep open. Current `main` still builds the webchat session dropdown from `sessions.list` rows plus the active session only, so configured agents without existing sessions are not surfaced. The PR’s original approach also needs rework because synthesized `agent:<id>:<mainKey>` entries are incorrect under global session scope: main aliases canonicalize to `global`, and the gateway intentionally drops non-default `agentId` when a canonical global session key is used.

## Evidence

- **Current dropdown logic still omits agents without sessions:** `resolveSessionOptionGroups` starts from `sessions?.sessions ?? []`, adds options from those rows, then adds only the current `sessionKey`. It does not synthesize options from `state.agentsList`.
  - file: [ui/src/ui/chat/session-controls.ts:473](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/ui/src/ui/chat/session-controls.ts#L473)

  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Configured agents are available separately via agents metadata:** `listAgentsForGateway` returns `{ defaultId, mainKey, scope, agents }` from configured agent state, including session scope. That data exists, but the dropdown builder above does not consume it for option synthesis.
  - file: [src/gateway/session-utils.ts:742](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/gateway/session-utils.ts#L742)
