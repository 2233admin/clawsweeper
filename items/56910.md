## Summary

Current `main` keeps core model selection intentionally simple (primary + fallbacks), while exposing a public plugin hook specifically for deterministic model/provider overrides before resolution. Repo vision also says optional capability should usually ship as plugins and that ClawHub is the canonical surface for community plugins, so this request fits a plugin/ClawHub solution rather than OpenClaw core.

## Risks / Open Questions

- This closes the core feature request, not the broader product discussion around documenting or standardizing routing patterns.
- A future built-in UX surface for routing indicators or savings analytics would be separate work from the already-supported plugin seam.

## Close Comment

This looks better scoped as a plugin/ClawHub feature than a new OpenClaw core feature.

- Codex review against current `main` found that core model selection is still intentionally simple: `primary` -> configured `fallbacks` -> provider auth failover (`docs/concepts/models.md:14`).
- Current `main` already exposes a public hook specifically for this kind of work: `before_model_resolve` is documented as the place to do model/provider override work (`docs/plugins/architecture.md:93`, `docs/tools/plugin.md:225`).
- The runtime actually executes that hook before model resolution and applies `providerOverride` / `modelOverride` from plugins (`src/agents/pi-embedded-runner/run/setup.ts:58`).
- The public hook contract includes the prompt and attachment metadata, which is enough to implement heuristic smart routing externally (`src/plugins/hook-before-agent-start.types.ts:7`).
- OpenClaw's vision explicitly says optional capability should usually ship as plugins and points community plugin distribution at ClawHub (`VISION.md:54`, `docs/tools/clawhub.md:10`, `docs/plugins/community.md:14`).
