## Summary

Keep open. Current `main` still has the formatter crash path for malformed cron rows, and the submitted PR as described would not fully close it because `truncate(job.name, ...)` remains unguarded.

## Evidence

- **Current formatter still assumes string `id` and `name`:** `printCronList` on current `main` still calls `pad(job.id, ...)` and `pad(truncate(job.name, ...), ...)`, while `pad` and `truncate` both require `string`. An undefined `id` or `name` still throws in the CLI formatter.
  - file: [src/cli/cron-cli/shared.ts:159](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/cli/cron-cli/shared.ts#L159)
  - command: `nl -ba src/cli/cron-cli/shared.ts | sed -n '150,310p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Cron store load is permissive:** `loadCronStore` reads `jobs` from JSON and returns them without schema validation, so malformed legacy rows can still reach runtime code paths before `doctor --fix` repairs them.
  - file: [src/cron/store.ts:153](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/cron/store.ts#L153)
  - command: `nl -ba src/cron/store.ts | sed -n '150,210p'`

- **Service code already defends against missing names elsewhere:** `sortJobs` and `listPage` explicitly coerce missing `name`/`id` to empty strings for sorting and filtering, which supports the report that malformed rows are handled defensively in some places but not in the CLI table formatter.
  - file: [src/cron/service/ops.ts:176](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/cron/service/ops.ts#L176)

That means malformed persisted rows are an expected compatibility case, not an incoherent report.
