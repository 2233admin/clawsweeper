## Summary

Keep open. Current `main` does not contain the PR's null-to-empty-string normalization for assistant tool-call messages, and the existing compatibility path only flattens array-form text content for models explicitly marked `compat.requiresStringContent`. That is not the same fix, and I found no merged regression coverage for `assistant.content === null` with `tool_calls`.

## Risks / Open Questions

- Closing as already implemented would drop a plausible real regression: current main still has no explicit null-to-empty-string normalization for assistant tool-call messages.
- The existing `compat.requiresStringContent` path is narrower than the PR and only addresses structured text arrays, so affected OpenAI-compatible providers may still reject payloads with `content: null`.
