## Summary

Keep open. Current `main` still lacks per-tool-call audit logs that say which tool-policy layer or rule allowed/denied the call. The runtime logs only tool start/end, while policy resolution exists only as internal filtering labels with no emitted matched-rule decision record.

## Evidence

- **Tool execution logging still only emits start/end:** `handleToolExecutionStart` and `handleToolExecutionEnd` log `embedded run tool start` / `embedded run tool end` with runId, tool, and toolCallId only; there is no policy decision or matched-rule field.
  - file: [src/agents/pi-embedded-subscribe.handlers.tools.ts:625](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/pi-embedded-subscribe.handlers.tools.ts#L625)
  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.tools.ts | sed -n '618,635p'; nl -ba src/agents/pi-embedded-subscribe.handlers.tools.ts | sed -n '1122,1136p'`
  - sha: [95b7a85f0607](https://github.com/openclaw/openclaw/commit/95b7a85f0607784d8e33b4ce7e42617174196490)
- **Policy layers are labeled internally but not audited:** `buildDefaultToolPolicyPipelineSteps` defines labels such as `tools.allow`, `tools.byProvider.allow`, `agents.<id>.tools.allow`, and `group tools.allow`, but `applyToolPolicyPipeline` just filters the tool list through those steps and only warns on unknown allowlist entries.
  - file: [src/agents/tool-policy-pipeline.ts:39](https://github.com/openclaw/openclaw/blob/95b7a85f0607784d8e33b4ce7e42617174196490/src/agents/tool-policy-pipeline.ts#L39)
