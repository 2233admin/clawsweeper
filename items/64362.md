## Summary

Keep open. Current main has useful restart-recovery mitigation for cleaned stale transcript locks, but I did not find the requested active-turn guard before dead-pid cleanup, and the newer open #71872 context says the same lost-response/session-history failure is still reproducible on ACP sessions. Closing #64362 would be premature without a source/test proof that Telegram/main-agent tool runs now deliver immediately after lock cleanup races.

## Best Possible Solution

Keep #64362 open until the lock cleanup path is made active-run/session-aware before removal, or until a maintainer lands a tested recovery that covers the Telegram main-agent tool-run delivery path and the ACP/sessions_history variant. A good fix should add regression coverage around dead-pid `.jsonl.lock` cleanup during an active tool turn, final channel delivery, and skipped/non-main session behavior.

## Evidence

- **Dead-pid cleanup still removes stale session locks directly:** `cleanStaleLockFiles` inspects `.jsonl.lock` files and, when `lockInfo.stale && removeStale`, removes the lock and logs `removed stale session lock`. The stale calculation includes `dead-pid`, `too-old`, and PID recycle reasons, but this path has no check that the associated session is actively processing a turn before removal.
