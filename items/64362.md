## Summary

Keep this issue open. Current main still has the conservative session-write-lock gap described in the report: same-PID lock files without verifiable starttime can be reclaimed when absent from the process-local held-lock map, and the existing recovery path does not prove immediate Telegram delivery after tool execution. The related PR #71903 remains the implementation candidate rather than shipped main behavior.

## Best Possible Solution

Keep this issue open and review, land, or supersede PR #71903 with a conservative session-write-lock fix. The fix should preserve live same-PID/no-starttime locks when ownership cannot be proven, retain safe cleanup for dead or verifiably recycled PIDs, and add regression coverage for the macOS/no-starttime tool-execution delivery path plus any `sessions_history` recovery behavior needed after stale lock cleanup.
