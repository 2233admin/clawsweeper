## Summary

Keep open. The issue remains concrete and plausible on current main: the Control UI shell still places `<openclaw-app>` in the page while registration depends on Lit decorator side-effect evaluation, and I did not find a source hardening change, explicit `customElements.define` fallback, or targeted served Chromium/Vivaldi regression proof that resolves the reported unregistered custom element failure.

## Best Possible Solution

Keep the issue open. The best path is to add a targeted served-Control-UI Chromium regression check that loads the actual `index.html`/built bundle, asserts `customElements.get('openclaw-app')`, captures module/CSP/service-worker/resource errors, and then hardens the bootstrap path based on the reproduced failure, likely with an explicit guarded registration or earlier observable bootstrap diagnostics if decorator/module evaluation is the failure point.
