## Summary

Keep this PR open. Current main still lacks peer-aware outbound account resolution in runMessageAction, and the PR is a focused fix attempt for a plausible multi-account Slack routing bug. It should not be merged as-is because the submitted fast path scans raw bindings, can include ACP bindings, and can misroute when multiple route bindings share the same peer under different agents/accounts.

## Best Possible Solution

Keep this PR open and request a revised implementation. The best path is to reuse or extend a typed outbound account resolver in src/infra/outbound/message-action-runner.ts that preserves explicit --account/defaultAccountId precedence, derives peer id/kind from the normalized outbound target, filters only route bindings via listRouteBindings, preserves resolved agent/account scoping, and adds regression tests for the reported Slack multi-account peer binding plus ACP-binding and same-peer collision cases.

## Risks / Open Questions

- Current main preserves the reported multi-account Slack failure mode when outbound sends omit an explicit account and rely on peer-scoped bindings.
- The PR as submitted can select ACP bindings because it scans raw cfg.bindings instead of listRouteBindings(cfg).
- The PR as submitted can misroute same-peer multi-account setups because it matches only channel plus peer.id before the existing agent-scoped fallback.
- No regression test is included for the reported Slack peer-binding case, ACP-binding exclusion, or same-peer collision behavior.
