## Summary

Keep this issue open. Current main still has no supported path to forward arbitrary inbound OpenAI-compatible HTTP headers such as `custom-token` into an agent run or expose them to a Python skill. Existing support is for fixed OpenClaw control headers and config-backed skill credentials, while related reports concern outbound provider headers and do not satisfy this inbound skill-context request.

## Best Possible Solution

Keep open for maintainer product/security design. If OpenClaw should support this, add an explicit allowlisted mapping from selected OpenAI-compatible inbound headers into per-run context or per-skill environment, with redaction, sandbox behavior, per-skill opt-in, docs, and gateway tests. If dynamic header passthrough is intentionally unsupported, document the supported alternative: configure fixed skill credentials through `skills.entries.<skill>.env` or `skills.entries.<skill>.apiKey`.

## Risks / Open Questions

- Forwarding arbitrary inbound HTTP headers into skill execution would be security-sensitive; a safe design likely needs explicit allowlists, redaction, per-skill opt-in, sandbox semantics, and tests.
- The issue text names `/v1/completions`, while current OpenClaw exposes `/v1/chat/completions` and `/v1/responses`; maintainers may need to clarify the intended compatibility endpoint before designing the feature.
