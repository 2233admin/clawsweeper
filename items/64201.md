## Summary

Keep open. Current main has related restart hardening, but the specific plugin-config reload crash-loop fix was not implemented: `createGatewayCloseHandler` still stops channel/plugin services before `configReloader.stop()` and closes the WebSocket/HTTP listeners only afterward. The linked PR #64255 identified this same shutdown-order fix but was closed unmerged, and current tests do not cover the needed ordering invariant.

## Best Possible Solution

Keep #64201 open. The best fix is a maintainer-reviewed gateway shutdown change that stops the config reloader as an early shutdown action and closes/bounds WebSocket/HTTP listeners before plugin/channel teardown, while preserving the current close timeouts so shutdown cannot hang. Add a focused `server-close.test.ts` ordering regression and, if feasible, a Docker/config-reload smoke that edits `plugins.allow` or `plugins.entries.*` and verifies the gateway returns without a restart loop.

## Evidence

This is the ordering #64255 proposed changing, so current main does not prove the reported race is fixed.
