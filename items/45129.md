## Summary

Keep open. Current `main` does not implement cron pre-hooks, and the proposed PR still has a blocking correctness gap: the new `hooks` payload is not persisted by cron job creation or patching, so the advertised CLI/API behavior would be a no-op.

## Evidence

- **Current cron job type has no hook field:** `CronJobBase` on current `main` ends at `failureAlert` and does not define any `hooks`/`pre` shape, so the feature is not present in the persisted/shared cron job contract.
  - file: [src/cron/types-shared.ts:1](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/cron/types-shared.ts#L1)

  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **Gateway protocol has no pre-hook schema:** Current cron gateway schema only includes the existing optional fields (`agentId`, `sessionKey`, `description`, `enabled`, `deleteAfterRun`) and no `hooks` field, so clients cannot send this on `main`.
  - file: [src/gateway/protocol/schema/cron.ts:70](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/gateway/protocol/schema/cron.ts#L70)
