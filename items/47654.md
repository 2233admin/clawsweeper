## Summary

Keep open. Current `main` at `0e7250f37bf347c72790abf2a6692d5ec84e5ea1` still fails closed on unknown config keys, so the requested strip-with-warning behavior is not implemented. The original Telegram `inlineButtons` repro is obsolete on `main`, but the broader config-validation change still needs design review rather than closure.

## Risks / Open Questions

- The specific Telegram `inlineButtons` example is already fixed on `main`, so any follow-up should be reviewed as a broader config-policy change, not as that exact schema gap.
- General unknown-key recovery would change current typo/error semantics for config write APIs and chat-config commands unless those write paths are handled separately.
- Current `main` documentation and changelog lean toward fail-closed config loading for safety, so closure as implemented or cannot-reproduce would be inaccurate.
