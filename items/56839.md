## Summary

Keep open. Current `main` still has a plausible path for this bug: cron delivery defaults announce jobs to `channel: "last"`, preserves raw `delivery.to`, and the WhatsApp target parser can coerce a provider-qualified value like `telegram:1234567890` into `+1234567890` instead of rejecting it. Recent cross-channel session-route hardening does not add channel/identifier validation for explicit cron targets.

## Evidence

- **Cron announce still falls back to `last` while preserving raw `to`:** `resolveCronDeliveryPlan` keeps `delivery.to` unchanged and defaults announce jobs without an explicit channel to `last`, which is the exact setup where a Telegram-prefixed target can later be reinterpreted through another channel.
  - file: [src/cron/delivery-plan.ts:29](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/cron/delivery-plan.ts#L29)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Tests confirm provider-qualified `to` values are preserved:** `src/cron/delivery.test.ts` explicitly expects `delivery.to` to remain `telegram:123`; there is no validation here that the target prefix matches the eventual announce channel.
  - file: [src/cron/delivery.test.ts:30](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/cron/delivery.test.ts#L30)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Delivery resolution reparses explicit targets using the selected/fallback channel plugin:** `resolveSessionDeliveryTarget` feeds `explicitTo` through `parseExplicitTargetForLoadedChannel(provider, raw)` where `provider` is the resolved channel or fallback `lastChannel`. If `lastChannel` is WhatsApp, a raw `telegram:...` target gets parsed as a WhatsApp target instead of being rejected.
  - file: [src/infra/outbound/targets-session.ts:32](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/infra/outbound/targets-session.ts#L32)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Cron delivery target resolution still allows `channel=last` to use session/fallback channel:** `resolveDeliveryTarget` sets `allowMismatchedLastTo` for `requestedChannel === "last"`, uses session state or channel selection as fallback, and then resolves the target against that channel. There is no guard that a provider-qualified `to` matches the chosen channel.
  - file: [src/cron/isolated-agent/delivery-target.ts:81](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/cron/isolated-agent/delivery-target.ts#L81)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **WhatsApp normalization can coerce foreign provider IDs into E.164:** `normalizeWhatsAppTarget` calls `normalizeE164(candidate)` for any non-JID target, and `normalizeE164` strips a leading `<provider>:` prefix before keeping digits. That means `telegram:1234567890` can normalize to `+1234567890` instead of failing validation.
  - file: [extensions/whatsapp/src/normalize-target.ts:57](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/extensions/whatsapp/src/normalize-target.ts#L57)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Gateway cron validation only checks configured channels, not channel/target consistency:** `assertConfiguredAnnounceChannel` verifies that `delivery.channel` names a configured channel, but it does not validate that `delivery.to` belongs to that channel namespace or format.
  - file: [src/gateway/server-methods/cron.ts:40](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/server-methods/cron.ts#L40)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)

## Risks / Open Questions

- This is based on conservative code-path inspection rather than a live repro, because the review must remain read-only.
- Recent session-routing changes on `main` reduce some cross-channel race cases, but they do not cover explicit provider-qualified cron targets being reparsed by a different channel plugin.
- If maintainers investigate, they should check both primary cron delivery and failure-alert delivery paths, because both route through the same target-resolution logic.
