## Summary

Keep open. Current `main` does not contain the DM-topic bound-session guard from #44900, and the inbound, native-command, and session-state paths still append a DM thread suffix unconditionally once a private-chat topic id is present. The related shipped Telegram fix on `main` (#48204) covers named-account DM topic base keys, not ACP-bound DM-topic follow-ups.

## Evidence

- **Route resolver still lacks a bound-DM-topic suffix guard:** `resolveTelegramConversationRoute()` on current `main` still returns only `route`, `configuredBinding`, and `configuredBindingSessionKey`; there is no `skipDmThreadSuffix`-style signal for callers after configured/runtime binding resolution.

On a bound DM topic, that means the already-bound base session key would still be rewritten with `:thread:<chatId>:<topicId>`.
