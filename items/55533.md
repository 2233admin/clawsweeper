## Summary

Keep open. Current main has added pieces of a compatibility strategy for external plugins, including a documented deprecation policy and host-version compatibility checks, but it does not fully address the issue's core complaint about root `openclaw/plugin-sdk` breakage. The public root export remains intentionally tiny, while the broken helpers named in the issue live on `openclaw/plugin-sdk/core` or other subpaths instead of being preserved behind a root-barrel compatibility shim.

## Evidence

  - file: [package.json:56](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/package.json#L56)

  - file: [src/plugin-sdk/index.ts:1](https://github.com/openclaw/openclaw/blob/d3d9b5738f83d92787f57db9186ddcebda60af9a/src/plugin-sdk/index.ts#L1)

## Risks / Open Questions

- Some source-loaded plugin paths may still see broader compatibility through `src/plugin-sdk/root-alias.cjs`, but the published package export for `openclaw/plugin-sdk` still points at the narrow `dist/plugin-sdk/index.js`, so packaged external plugins remain the important conservative case.
- The repo now documents a compatibility policy, but I did not find evidence that the specific root-barrel removals named in the issue were restored with a warning-based deprecation path.
