## Summary

Keep #68970 open. Current main still has the TUI reconnect/backfill weakness described by the reporter: the Gateway client detects sequence gaps, but the TUI gap handler only updates status, and reconnect performs one immediate history load with no delayed replay/backfill or session-message subscription. Web UI has stronger recovery paths, which matches the report that the same post-restart reply is visible there.

## Best Possible Solution

Keep the issue open and implement TUI reconnect/backfill parity: on gateway sequence gap and post-reconnect, safely reload or subscribe/backfill the current session transcript, likely in `src/tui/tui.ts` and `src/tui/tui-session-actions.ts`, while preserving active-run/concurrent-run behavior. Add targeted tests for `onGap` and post-reconnect heartbeat reply visibility.
