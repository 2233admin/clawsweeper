## Summary

Current main still does not implement the requested session-lane busy/timeout surfacing for `openclaw agent`. The issue body and existing Codex review identify a blocked same-session CLI dispatch path; current source still serializes runs through a per-session lane, logs lane wait details only diagnostically, sends an immediate Gateway `accepted` frame before dispatch settles, and the CLI waits for a generic final response without a typed lane-busy decode or `--force-queue`/queue-policy option.

## Best Possible Solution

Keep this issue open and fix the Gateway/CLI contract. Add a typed session-lane busy or lane-wait timeout result from the queue/agent dispatch path, propagate `lane`, `queueAhead`, and `waitedMs` through the Gateway final/error response, make `openclaw agent` print an actionable message, and add focused regression coverage for blocked same-session CLI dispatch. If queueing remains default behavior, add and document an explicit queue policy or force-queue option.
