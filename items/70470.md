## Summary

Keep #70470 open. This is a recent non-maintainer bug report against the bundled WhatsApp channel, and current main does not prove the read-receipt dependency is fixed. The monitor still awaits `sock.readMessages()` for accepted non-self inbound messages before enrichment, dedupe, and enqueueing, while existing tests cover normal/disabled/self-chat/blocked cases but not a rejected or hanging read-receipt path. The provided triage comment usefully identifies the read-receipt area, but its specific `shouldMarkRead` explanation is not fully established because blocked access returns `null` before `maybeMarkInboundAsRead()` runs.

## Best Possible Solution

Keep the issue open and ask for either logs or a focused regression. The likely maintainer path is to decouple or bound WhatsApp `readMessages()` so accepted inbound messages can still be enriched/enqueued even if read receipts fail or hang, and to add a WhatsApp monitor test proving Calendar/tool-triggering message processing is independent of read-receipt delivery. Also consider honoring `access.shouldMarkRead` for clarity, but validate it against the actual reported failure before closing.
