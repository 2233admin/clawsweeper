## Summary

Keep this issue open. Current main still exposes Telegram inline keyboard buttons on the agent/message send path, but does not expose native Telegram ReplyKeyboardMarkup, ReplyKeyboardRemove, resize_keyboard, or one_time_keyboard semantics. The related closed items are about inline buttons/callbacks and do not cover this separate reply-keyboard capability.

## Best Possible Solution

Keep the issue open and implement native Telegram reply keyboard support in the Telegram plugin send/action path. A conservative implementation would add a typed Telegram-owned reply_markup union supporting reply keyboard rows/buttons, ReplyKeyboardRemove, resize_keyboard, and one_time_keyboard, then thread it through extensions/telegram/src/action-runtime.ts, extensions/telegram/src/send.ts, tests, and docs. If maintainers want this portable across channels, define a generic quick-reply or persistent-keyboard contract first.
