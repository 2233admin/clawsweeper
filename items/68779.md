## Summary

Keep open. Current main contains useful partial hardening for existing-session browser flows, including longer manage-call budgets, some timeout propagation, and limited Chrome MCP session reset/retry behavior. It does not fully resolve #68779: the attach/readiness constants called out by the issue remain tight, existing-session action limitations are still documented and enforced, broad reconnect/retry after transport loss is not implemented, and a maintainer comment explicitly says to keep the item open with split acceptance criteria.

## Best Possible Solution

Keep #68779 open and narrow the acceptance criteria around the remaining product work: increase or make evidence-backed attach/readiness budgets for existing-session profiles, add reconnect/recovery after Chrome MCP transport or session loss rather than only clearing the cache, and add a focused regression test for a multi-step profile="user" flow where the selected page/session is refreshed without dropping the workflow. Keep the existing guidance and partial hardening as useful mitigation, but do not close until the reconnect and timeout-budget behavior is source-backed and tested.
