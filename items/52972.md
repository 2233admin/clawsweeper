## Summary

Current main still has the reported failure mode. The auto-reply guard appends the note whenever a reminder-like reply has `successfulCronAdds === 0` and no enabled same-session cron job, while the only production increment of `successfulCronAdds` is still the native `cron` tool with `action: "add"`. I found no current-main tracking for successful `exec`/`bash` commands that run `openclaw cron add --at ...`, and the existing regression tests only prove the native cron-tool path. The related PR #53024 directly targeted this bug but is closed and unmerged in the provided GitHub context, so this issue should remain open.

## Best Possible Solution

Keep this issue open. Implement PR #53024 or an equivalent current-main fix that treats successful `openclaw cron add` executions through `exec`/`bash` as successful reminder scheduling for the current turn, or ensure assistant-created reminders always use the native `cron.add` tool. Add regression coverage for successful exec/bash cron-add suppression, failed exec/bash cron-add still warning, and Telegram/auto-reply text that confirms a scheduled reminder without appending the false note.

## Risks / Open Questions

- A broad suppression based only on any existing cron job would hide genuine unbacked reminder promises; the safer fix is turn-local success telemetry or consistently routing assistant-created reminders through the native cron tool.
- The same-session cron-store fallback can mask some reproductions, so regression coverage should include an exec/bash-created one-time reminder path that lacks run-local telemetry.
