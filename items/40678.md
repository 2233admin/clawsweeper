## Summary

Keep open. Current main still has per-session TUI behavior and no opt-in observe/mirror mode: `/new` creates a fresh `tui-<uuid>` session, chat and side-result events return early unless their `sessionKey` matches the active TUI session, and history loading reads only `state.currentSessionKey`. Existing identity-link/session-sharing config can make channels share a session, but it does not provide the requested all-channel/all-session TUI activity view.

## Best Possible Solution

Keep this as a core TUI/Gateway UX issue. The best path is an explicit read-only `openclaw tui --observe` or `tui.mirrorChannels` mode that preserves `/new` isolation, uses existing read-scoped Gateway broadcasts plus session listing/preview/history APIs, labels source agent/session/channel, and only relaxes exact-session event filtering in that opt-in mode. Add regression tests for same-agent different-session events, channel-originated events, side-result visibility, and read-only behavior before closing.
