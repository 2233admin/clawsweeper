## Summary

Current `main` still does not inherit `tools.exec.host` / `tools.exec.security` / `tools.exec.ask` for chat-bash, so the underlying bug is not implemented. The PR also should not be auto-closed because its follow-up regex change narrows `!` alias routing and matches the review comment's regression concern for punctuation-prefixed shell commands.

## Evidence

- **Chat-bash on main still drops exec host/security/ask defaults:** `handleBashChatCommand()` forwards `timeoutSec`, `notifyOnExit`, and `notifyOnExitEmptySuccess` into `createExecTool(...)`, but not `host`, `security`, or `ask`, so chat-bash does not inherit the documented `tools.exec` policy fields on current `main`.

  - command: `nl -ba src/auto-reply/reply/bash-command.ts | sed -n '344,362p'`
  - sha: [d068cb960de4](https://github.com/openclaw/openclaw/commit/d068cb960de47d5587c5942f6beb86e0f124b449)
- **Exec policy only applies when passed as tool defaults:** `createExecTool()` resolves routing from `defaults?.host` and approval posture from `defaults?.security` / `defaults?.ask`; when those defaults are omitted, it falls back to generic approval defaults instead of the chat-bash caller's `cfg.tools.exec` settings.
  - file: [src/agents/bash-tools.exec.ts:1472](https://github.com/openclaw/openclaw/blob/d068cb960de47d5587c5942f6beb86e0f124b449/src/agents/bash-tools.exec.ts#L1472)
