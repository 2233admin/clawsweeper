## Summary

Keep open. Current `main` intentionally rejects loopback-source `trusted-proxy` auth, the tests and docs codify that behavior, and the PR discussion points toward a different audited service-identity design rather than merging a blanket loopback bypass.

## Evidence

- **Current auth path rejects loopback trusted-proxy sources:** `authorizeTrustedProxy` returns `trusted_proxy_loopback_source` for any loopback remote address, and `authorizeGatewayConnect` keeps `trusted-proxy` requests on that path instead of granting a local-direct bypass.
