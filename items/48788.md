## Summary

Keep open. Current `main` still has only ad-hoc filename parsing in separate places, not the centralized multi-encoding `Content-Disposition` utility requested here.

## Evidence

- **Core media fetch parser is local and limited:** `src/media/fetch.ts:63` has a private `parseContentDispositionFileName()` that handles `filename*=` by percent-decoding and `filename=` by regex, but it is scoped to `fetchRemoteMedia` and does not implement charset-aware recovery beyond that simple path.
  - file: [src/media/fetch.ts:63](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/src/media/fetch.ts#L63)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Feishu keeps a separate parser:** `extensions/feishu/src/media.ts:127` defines its own `decodeDispositionFileName()` with a separate regex path for `filename*=UTF-8''...` and plain `filename=...`. This is duplicated logic rather than a shared utility reused across adapters.
  - file: [extensions/feishu/src/media.ts:127](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/extensions/feishu/src/media.ts#L127)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **Tests cover only simple cases:** `extensions/feishu/src/media.test.ts:591` verifies extraction from `content-disposition: attachment; filename="clip.mp4"`, and nearby tests focus on upload-name sanitization/preservation. I did not find current-main tests for GBK, Shift-JIS, EUC-KR, or generalized multi-encoding recovery.
  - file: [extensions/feishu/src/media.test.ts:591](https://github.com/openclaw/openclaw/blob/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd/extensions/feishu/src/media.test.ts#L591)
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)
- **No current-main charset support surfaced in relevant code:** Repository search over `src`, `extensions`, and `test` found the two filename parsers above, but no relevant shared implementation for `GB18030`, `GBK`, `Shift-JIS`, `EUC-KR`, or similar multi-encoding filename decoding in the media/channel paths.
  - command: `rg -n "GB18030|GBK|Shift-JIS|EUC-KR|iconv-lite|RFC 5987|filename\*=|content-disposition" package.json pnpm-lock.yaml src extensions test -S`
  - sha: [01bc49c88c0c](https://github.com/openclaw/openclaw/commit/01bc49c88c0cef9807b15df4d7486a0c5fe4afbd)

## Risks / Open Questions

- `main` does appear to contain a narrower Feishu fix for preserving upload filenames, so maintainers should distinguish that shipped regression fix from this broader architectural follow-up.
- I could not inspect GitHub PR metadata with `gh` because the environment lacks `GH_TOKEN`, but the code on `main` is sufficient to determine that the requested centralized utility is not present.
