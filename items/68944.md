## Summary

Keep this issue open. The provided discussion shows the accidental #68614 cross-reference was acknowledged as unrelated, and current main still has a plausible post-challenge silent-failure path in the CLI Gateway client: `sendConnect()` is invoked from the `connect.challenge` event under a broad `handleMessage()` catch, and synchronous failures while assembling/signing the device connect frame are logged only as a debug parse error before any `connect` request is sent.

## Best Possible Solution

Keep the issue open and patch the CLI Gateway client. The best fix is to wrap the `sendConnect()` frame assembly/signing/send path in its own error handling that calls `onConnectError`, logs at an actionable level, closes or resets the socket cleanly, and does not mislabel non-JSON failures as parse errors. Narrow `handleMessage()` so only JSON parse/validation failures are swallowed as parse errors. Add a focused `src/gateway/client.test.ts` regression where device signing/public-key normalization or request-frame construction throws after `connect.challenge`, asserting that the CLI surfaces a connect error and does not leave an open socket with no `connect` frame.
