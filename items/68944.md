## Summary

Keep open. Current main/v2026.4.24 still has the same post-challenge client-side failure shape described in #68944: a `connect.challenge` can call into synchronous device signing from `handleMessage()`, any throw is swallowed as a debug-only parse error, and the client has already cleared its connect-challenge timeout before the signed `connect` request is emitted. The discussion also corrected the mistaken #68614 cross-reference and a later Linux report on 2026.4.22 describes a similar hang, so this is not safely implemented or superseded.

## Best Possible Solution

Keep #68944 open as the canonical fresh CLI gateway handshake report. The likely fix belongs in `src/gateway/client.ts`: wrap device payload construction/signing/public-key derivation in a narrow try/catch inside `sendConnect()`, report it through `onConnectError` and error logging, close the socket without reconnect churn, and narrow `handleMessage()` so non-JSON/signing failures are not mislabeled as parse errors. Add a focused regression test that forces `signDevicePayload` or public-key derivation to throw after `connect.challenge` and asserts a CLI-visible error instead of a missing `connect` frame or silent timeout.
