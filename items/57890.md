## Summary

Current `main` has a durable cron task ledger with structured statuses, but the cron-created records are system-scoped and the owner-visible task APIs used by agent/session surfaces explicitly filter to session-scoped tasks only. That means the repo does not show a clear agent-facing way for a later cron invocation to query whether a prior cron job completed or was interrupted, which is the core request in #57890.

## Evidence

- **Cron runs already create structured task records:** Cron execution creates a task record with `runtime: "cron"`, `sourceId` set to the cron job id, timestamps, and terminal updates to `succeeded`/`failed`/`timed_out`. This is real structured tracking, but it is just the ledger layer.
  - file: [src/cron/service/timer.ts:135](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/cron/service/timer.ts#L135)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '125,220p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Task model supports completion/interruption states:** The task registry has explicit structured fields for `sourceId`, `startedAt`, `endedAt`, `error`, and statuses including `succeeded`, `failed`, `timed_out`, `cancelled`, and `lost`.
  - file: [src/tasks/task-registry.types.ts:53](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/tasks/task-registry.types.ts#L53)
  - command: `nl -ba src/tasks/task-registry.types.ts | sed -n '1,120p'`

- **Cron task records are system-scoped, not session-scoped:** When cron creates its task ledger entry it passes `ownerKey: ""` and `scopeKind: "system"`. That means the record is not owned by the cron session itself.
  - file: [src/cron/service/timer.ts:142](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/cron/service/timer.ts#L142)
