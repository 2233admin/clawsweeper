## Summary

Keep open. Current main still injects WhatsApp channel connected/disconnected lifecycle notices into the resolved session system-event queue immediately, with no threshold, verbosity setting, reconnect coalescing, or surface-aware filter. The prior ClawSweeper comment remains accurate against current main; related #39406 and #41940 address different warning/health-monitor surfaces and do not supersede this request.

## Best Possible Solution

Keep this issue open and implement a tested channel lifecycle notification policy. A narrow practical path is a WhatsApp/account-level setting such as `statusEvents` and/or `notifyAfterMs`, with reconnect coalescing for outages that recover before the threshold while keeping serious stop/auth notices visible. Add regression coverage around the WhatsApp monitor plus system-event draining; if multiple channels need the behavior, expose an additive plugin SDK helper or generic channel lifecycle notification policy.
