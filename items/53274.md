## Summary

Current main still matches the central report: the browser client only creates device identity when `crypto.subtle` exists, and device identity fingerprinting still calls WebCrypto. The secondary `dangerouslyDisableDeviceAuth` scope-preservation and implied operator-scope requests are already implemented, but the Secure Context/LAN HTTP behavior remains open. Closed PR #53276 was an unmerged implementation candidate, and open #32473 overlaps but is less specific.

## Best Possible Solution

Keep this issue open. Maintainers should decide whether browser device identity should be supported over non-secure LAN HTTP; if yes, remove the `crypto.subtle` gate in `ui/src/ui/gateway.ts`, replace WebCrypto-only fingerprinting in `ui/src/ui/device-identity.ts` with an approved browser-safe hash path and direct dependency as needed, then update gateway/client tests and Control UI docs. A refreshed, narrower version of closed PR #53276 can start from the remaining device-identity work while leaving the already-shipped bypass and scope-normalization fixes intact.
