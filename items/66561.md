## Summary

Keep this issue open. The current main has adjacent fixes and diagnostics for OpenAI/Codex transport, session-lock timing, and model-call telemetry, but it still does not carry provider request-started/response-started state into the embedded attempt result or failover decision. A whole-attempt timeout still aborts the active session and is normalized to timeout/408, so the specific reported case, upstream SSE response had started before local abort, remains distinguishable only in diagnostics, not in the surfaced failover classification.

## Best Possible Solution

Keep this issue open and fix the embedded Pi runner/failover boundary. Track provider-call lifecycle separately from the whole-attempt timer, carry request-started/response-started or first parsed stream event state into EmbeddedRunAttemptResult and failover observation, and surface distinct reasons/statuses for no-response timeout, post-start local abort, parser/stream failure, and upstream explicit errors. Add a focused regression test that simulates a Codex SSE stream receiving at least one chunk before the enclosing timeout aborts it and asserts the logged/surfaced failure is not the generic no-response timeout/408 shape.
