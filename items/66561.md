## Summary

Keep open. Current `main` still intentionally classifies abort/stream-abort signals as `timeout`/408, and I did not find a landed change that distinguishes "provider stream started, then local abort" from a true no-response timeout. The discussion also surfaced a useful workaround around stale session overrides; current `main` has related session-override hardening, but that is orthogonal to the reported SSE misclassification.

## Evidence

- **Current main still surfaces generic timeout text for timed-out runs:** `run.ts` still emits a generic timeout payload when a run times out without assistant payloads, with only a coarse split between idle-timeout and general timeout. There is no separate surface here for "response started, then local abort/parser failure".
