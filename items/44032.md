## Summary

Keep open. The issue discussion identifies a real pain point, and current main still runs the memory flush synchronously before the foreground user turn. Main has partial mitigations now: incoming messages can be preserved as follow-up queue items, and the reply operation has a `memory_flushing` phase. But the requested core behavior is not implemented yet: the memory flush is not on a separate/background lane, the foreground turn does not start until it completes, and the direct steer path still rejects messages while the active embedded handle reports compacting.
