## Summary

Keep open. Current `main` still requires `scripts/docker/setup.sh` to build `openclaw:local` or pull a remote image before continuing, and there is no `--offline` or equivalent documented escape hatch for re-running setup against a pre-loaded local image on airgapped hosts.

## Evidence

- **No offline flag or arg parsing in setup entrypoint:** The script starts by reading environment variables (`OPENCLAW_IMAGE`, `OPENCLAW_SANDBOX`, etc.) and defining helpers, but there is no `$@` parsing or offline-mode branch at the top of the file. A direct `rg -n "offline" scripts/docker/setup.sh docs/install/docker.md` returned no matches.
  - file: [scripts/docker/setup.sh:1](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/scripts/docker/setup.sh#L1)
  - command: `nl -ba scripts/docker/setup.sh | sed -n '1,40p'; rg -n "offline" scripts/docker/setup.sh docs/install/docker.md`
  - sha: [af56926e2fc4](https://github.com/openclaw/openclaw/commit/af56926e2fc410809fcb0399435787c7fa8daef8)
- **Image acquisition is still unconditional on current main:** `setup.sh` always builds `openclaw:local` or pulls `OPENCLAW_IMAGE`, and exits on pull failure before the later setup steps run. There is no branch that first checks for an already-loaded local image and proceeds without network/build activity.
  - file: [scripts/docker/setup.sh:474](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/scripts/docker/setup.sh#L474)

- **The rest of setup is exactly the state the issue wants to preserve:** After the build/pull block, the script performs the data-directory ownership fix, onboarding, gateway config sync, and `docker compose up -d`. That matches the reporter's point that airgapped operators need the full setup routine, not just container startup.
  - file: [scripts/docker/setup.sh:491](https://github.com/openclaw/openclaw/blob/af56926e2fc410809fcb0399435787c7fa8daef8/scripts/docker/setup.sh#L491)
