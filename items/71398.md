## Summary

Keep open. Current main still treats a `secretref:/...` value as a configured plaintext string, not as a SecretRef, and QQ Bot then uses that SecretInput path for `channels.qqbot.clientSecret` / account `clientSecret`. I found no current-main parser, migration, validation error, release evidence, or duplicate that closes the reported upgrade regression. The existing Codex review comment also reached the same keep-open conclusion.

## Best Possible Solution

Keep #71398 open. The best fix is a maintainer decision plus implementation in the shared SecretInput migration/validation layer: either support/migrate the legacy `secretref:/...` form before QQ Bot runtime resolution, or reject it early with actionable doctor/config validation guidance. Align QQ Bot docs with the canonical SecretRef credential surface and add regression coverage for both `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`.
