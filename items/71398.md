## Summary

Keep open. Current main still does not resolve the reported `secretref:/...` string form for QQ Bot `clientSecret`; that value is accepted as a plain string and can continue to flow toward the QQ token request. The current code also leaves QQ Bot SecretRef support inconsistent: docs/schema mention SecretRef input, but QQ Bot has no channel secret contract collector and the runtime resolver test still expects unresolved structured refs to throw before runtime use.

## Best Possible Solution

Keep the issue open. The best fix is to make a maintainer decision on `secretref:/...` compatibility, then either migrate/resolve that legacy form through shared SecretInput/doctor logic or reject it with a clear validation error. In either path, add QQ Bot runtime SecretRef coverage for `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, add regression tests for default and named accounts, and align `docs/channels/qqbot.md` plus the SecretRef credential surface docs.
