## Summary

Keep this issue open. Current main still documents and generates Slack bot-level event subscriptions only, starts the Slack monitor around bot/app tokens, and routes normal inbound replies through the bot token. The configured user token remains a supplemental read token and a write fallback only when the bot token is unavailable, so the requested dedicated Slack user DM flow is not implemented.

## Best Possible Solution

Keep this open and implement it in the bundled Slack plugin: add user_events setup/docs for message.im, resolve and carry the configured user-token identity through monitor context, distinguish eligible user-level DMs from bot-level DMs, select the xoxp token for those replies only when userTokenReadOnly is false, and add regression tests for inbound routing, token selection, setup manifest output, docs, and self-loop prevention.
