## Summary

Keep open. Current `main` has partial mitigations for Chrome MCP existing-session on browser nodes, but I did not find high-confidence proof that the reported direct `browser.proxy` failure path is fixed end-to-end. The raw node-host proxy command still defaults to a 20s timeout, Chrome MCP still uses `--autoConnect` by default unless a custom endpoint/args are configured, and the supported-node docs still describe `user`/`existing-session` through a connected browser node.

## Evidence

- **Direct `browser.proxy` still uses a 20s default timeout on the node host:** `runBrowserProxyCommand()` resolves to `DEFAULT_BROWSER_PROXY_TIMEOUT_MS = 20_000` when callers do not pass `timeoutMs`, then wraps the dispatched browser request with that timeout. That is the exact low-level surface used in the issue's `openclaw nodes invoke ... browser.proxy ...` reproduction.
