## Summary

Keep open. Current `main` does not already implement this PR’s `resolveModel(...)` normalization, and the review thread surfaced a credible regression risk for valid nested same-provider model IDs. That is not strong close evidence in either direction, so this needs maintainer follow-up rather than conservative cleanup.

## Evidence

- **Current `resolveModel` still only applies static model-ID normalization:** On `main`, `resolveModel(...)` passes `modelId` through `normalizeStaticProviderModelId(normalizeProviderId(provider), modelId)` and then into `resolveModelWithRegistry(...)`. There is no same-provider prefix stripping step here, so a value like `provider="google"` plus `modelId="google/gemini-2.5-pro"` is not normalized to `gemini-2.5-pro` in this function.
