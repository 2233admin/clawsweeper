## Summary

Keep this issue open. Current main still emits pi-embedded lifecycle end events without `aborted` or `stopReason`, while the richer fallback in `agent-command.ts` is skipped after the subscribe path emits the terminal lifecycle event. The related fix PR #66574 is closed unmerged, and current tests still lock in field-less `data: { phase: "end" }` payloads.

## Best Possible Solution

Keep this issue open. The maintainer fix should update the pi-embedded subscribe lifecycle bridge so both the global `emitAgentEvent` payload and direct `onAgentEvent` payload for lifecycle end include `aborted` and `stopReason`, sourced consistently from the terminal attempt result or a typed terminal lifecycle metadata seam. Add regression coverage for natural completion, explicit abort, and timeout/fallback behavior, and preserve existing gateway session lifecycle status mapping.

## Risks / Open Questions

- A fix that derives `aborted` only from `stopReason` could misclassify timeouts or nonstandard stop reasons; the safest path should use explicit runner result metadata or extend terminal lifecycle meta through the subscribe bridge.
- Changing terminal lifecycle payloads should keep lifecycle error behavior and the agent.wait retry-grace handling intact, since gateway consumers already distinguish normal end, timeout, killed, and error states.
