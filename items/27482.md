## Summary

Keep open. Current `main` still limits chat/user-upload attachments to images; direct video upload to chat/LLM is not implemented end-to-end even though OpenClaw has provider-level video-understanding capability in other surfaces.

## Evidence

- **Chat picker only accepts images:** The Control UI attachment allowlist is hard-coded to `image/*`, and supported chat attachment MIME types are only `image/`.
  - file: [ui/src/ui/chat/attachment-support.ts:1](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/ui/src/ui/chat/attachment-support.ts#L1)
  - command: `nl -ba ui/src/ui/chat/attachment-support.ts | sed -n '1,80p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Paste/select/drop handlers are image-only:** Chat paste handling only collects clipboard items whose MIME starts with `image/`, and file select / drag-drop skip any file whose MIME is not accepted by the image-only attachment helper.
  - file: [ui/src/ui/views/chat.ts:214](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/ui/src/ui/views/chat.ts#L214)
  - command: `nl -ba ui/src/ui/views/chat.ts | sed -n '190,340p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **RPC payload always labels uploads as images:** When chat attachments are sent to `chat.send`, the UI serializes every attachment as `{ type: "image", mimeType, content }`; there is no video attachment shape on this path.
  - file: [ui/src/ui/controllers/chat.ts:223](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/ui/src/ui/controllers/chat.ts#L223)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '205,250p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Gateway parser drops non-image attachments:** `parseMessageWithAttachments` warns and drops attachments whose sniffed MIME is non-image or whose MIME cannot be recognized as image, so video payloads would not reach the model on the current chat attachment path.
  - file: [src/gateway/chat-attachments.ts:347](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/gateway/chat-attachments.ts#L347)
  - command: `nl -ba src/gateway/chat-attachments.ts | sed -n '300,430p'`
  - sha: [d3f6783b16ec](https://github.com/openclaw/openclaw/commit/d3f6783b16ecc10c67b101bf019ca4294453206a)
- **Tests codify non-image rejection:** Gateway tests explicitly assert that non-image payloads are dropped and logged, which matches the current implementation and confirms this is not an accidental omission in one layer.
  - file: [src/gateway/chat-attachments.test.ts:84](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/src/gateway/chat-attachments.test.ts#L84)

  - file: [docs/nodes/media-understanding.md:232](https://github.com/openclaw/openclaw/blob/d3f6783b16ecc10c67b101bf019ca4294453206a/docs/nodes/media-understanding.md#L232)
