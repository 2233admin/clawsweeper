## Summary

Keep open. Current `main` still reads entire cron run-log JSONL files into memory with uncapped `fs.readFile` in both per-job and all-jobs read paths, so the reported OOM/GC-pressure risk is still present and not implemented away on `main` or in release `v2026.4.23`.

## Evidence

- **Per-job read path is still uncapped:** `readCronRunLogEntriesPage` drains pending writes, then immediately does `fs.readFile(path.resolve(filePath), "utf-8")` with no prior `stat`, size cap, tail read, or queued prune.
  - file: [src/cron/run-log.ts:375](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/cron/run-log.ts#L375)
  - command: `nl -ba src/cron/run-log.ts | sed -n '371,412p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **All-jobs read path has the same pattern:** `readCronRunLogEntriesPageAll` enumerates every `runs/*.jsonl` file and reads each whole file with `fs.readFile`, then flattens all parsed entries in memory. The issue is not only unfixed for a single job; the aggregate path is also uncapped.
  - file: [src/cron/run-log.ts:423](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/cron/run-log.ts#L423)
  - command: `nl -ba src/cron/run-log.ts | sed -n '423,445p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Read path is user-reachable from cron runs RPC:** Gateway server method `cron.runs` calls `readCronRunLogEntriesPage` for a specific job and `readCronRunLogEntriesPageAll` for `scope === "all"`, so ordinary run-history listing still exercises the uncapped reads.
  - file: [src/gateway/server-methods/cron.ts:427](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/gateway/server-methods/cron.ts#L427)
  - command: `nl -ba src/gateway/server-methods/cron.ts | sed -n '427,474p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Pruning exists only on append path:** `appendCronRunLog` appends, then calls `pruneIfNeeded`, but the read functions do not call any size guard or capped reader. If pruning is skipped or fails earlier, reads still load the full file(s).
  - file: [src/cron/run-log.ts:152](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/cron/run-log.ts#L152)
  - command: `nl -ba src/cron/run-log.ts | sed -n '133,183p'`
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Latest release still contains the same uncapped read:** Tag `v2026.4.23` shows the same `fs.readFile(..."utf-8")` implementation in `readCronRunLogEntriesPage`, so no shipped release evidence supports closing as already fixed.
  - file: [src/cron/run-log.ts:375](https://github.com/openclaw/openclaw/blob/9b18ae877018a34933e27bb7b2986cb97e9602aa/src/cron/run-log.ts#L375)
