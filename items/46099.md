## Summary

Current main already implements the Discord READY/connected handling needed for this report, and the same lifecycle code and regression tests are present in release `v2026.4.22`, so the reported health-monitor restart loop is obsolete on shipped code.

## Close Comment

Closing as implemented on main. A Codex review of current `main` found the Discord monitor now waits for READY before treating the gateway as connected, polls `gateway.isConnected` after `Gateway websocket opened`, restarts startup sessions that never reach READY, and force-stops reconnects that open without becoming ready (`extensions/discord/src/monitor/provider.lifecycle.ts:180`, `extensions/discord/src/monitor/provider.lifecycle.ts:428`). The shipped release `v2026.4.22` already contains that lifecycle logic and its regression coverage for both the startup race and reconnect path (`extensions/discord/src/monitor/provider.lifecycle.test.ts:218`, `extensions/discord/src/monitor/provider.lifecycle.test.ts:473`). On that basis this specific connected-flag race appears fixed in released OpenClaw code.
