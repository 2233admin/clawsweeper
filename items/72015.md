## Summary

Keep open. Current main has a hard Active Memory timeout, which addresses one failure mode from related #65517, but it still intentionally runs a blocking embedded memory sub-agent in `before_prompt_build`, still inherits the current/session model by default when no explicit model is set, and QMD gateway startup still arms every configured QMD-enabled agent. The remaining reliability/defaults request is real and not fully implemented or superseded.

## Best Possible Solution

Keep the issue open as the canonical reliability/defaults follow-up. A good maintainer path would add an explicit Active Memory execution mode or short pre-prompt deadline, per-agent/session concurrency or reuse/drop behavior, stronger warnings or safer defaults for inherited slow models, and QMD boot controls such as global boot concurrency, jitter, lazy initialization, or an active-agent allowlist. Status/doctor should surface high-cost configurations once the runtime semantics are settled.

## Risks / Open Questions

- Part of the report is already fixed on current main: Active Memory now has a hard timeout path, so new work should avoid duplicating that narrower fix.
- Changing Active Memory defaults could affect operators who intentionally opted into blocking recall for better personalization.
- QMD boot throttling needs to preserve per-agent isolation, startup availability, and existing `waitForBootSync` semantics.
