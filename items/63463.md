## Summary

Keep this issue open. Current main has generic custom-provider behavior that should handle slash-containing model IDs when configured correctly, but the report is recent, concrete, and still lacks a maintainer-confirmed retry with the documented OpenAI-compatible provider shape.

## Best Possible Solution

Keep this issue open for maintainer triage. Ask the reporter to retry with models.providers.mlx.baseUrl set to http://127.0.0.1:8080/v1, api set to "openai-completions", and a non-empty local apiKey if required by the auth path. If it still fails, add a focused regression for mlx/mlx-community/Qwen3-30B-A3B-6bit that asserts custom-provider resolution and that the Chat Completions payload uses the bare model id; also consider adding a clearer diagnostic for custom OpenAI-compatible providers missing api or /v1.
