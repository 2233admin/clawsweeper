## Summary

Issue #57768 is a broad architecture request, but current OpenClaw direction explicitly keeps optional learning/improvement behavior out of core and routes it through plugins/skills. Main already exposes memory/context-engine plugin seams and ships opt-in pluginized feedback-to-behavior features, so this is better handled as ClawHub/plugin experimentation than a core roadmap item.

## Close Comment

Thanks for the thoughtful suggestion. After a Codex review of current `main`, this looks better scoped as ClawHub/plugin experimentation than an OpenClaw core issue.

- `VISION.md:54` says core stays lean and optional capability should usually ship as plugins.
- `VISION.md:68` and `VISION.md:101` say new skills should go to ClawHub first, and that heavy orchestration layers / default-core agent frameworks are not the current merge direction.
- `src/config/types.plugins.ts:24` already exposes separate plugin ownership for `memory` and `contextEngine`, which is the existing seam for trying alternative learning/improvement architectures.
- `docs/plugins/skill-workshop.md:16` already documents an opt-in plugin that turns user corrections and reusable procedures into durable skills, which is the closest current path for feedback-driven behavior improvement.
- `docs/concepts/memory.md:39` and `docs/concepts/memory.md:115` show long-term memory, dreaming, and related knowledge layers are already handled through pluginized memory surfaces rather than a single core RL pipeline.

Given the current repo vision, the maintainable path here is a plugin/skill or ClawHub prototype rather than expanding OpenClaw core architecture.
