## Summary

Keep #64103 open. Current main still exposes the exact ambiguous session `status` values (`running`, `done`, `failed`, `killed`, `timeout`) through gateway/session list surfaces, and there is still no `resumable`, `taskStatus`, or explicit status-semantics guidance in the session tool row shape. `sessions_send` itself can target an existing resolved session, so the remaining bug is the API/docs/prompt ambiguity that can steer orchestrators toward unnecessary `sessions_spawn` calls. Related open reports (#69202 and #71756) are adjacent lifecycle/compaction issues, not a canonical replacement for this status-field contract problem.

## Best Possible Solution

Keep the issue open and fix it as an agent-facing contract problem: add a backward-compatible `resumable` or explicit lifecycle/turn-state field to gateway/session list rows, document the semantics in the public session tools docs, and add default prompt/tool-description guidance telling orchestrators to prefer `sessions_send` for known sessions unless a real deletion/visibility failure occurs. Cover the behavior with focused tests for `sessions_list` row shape and prompt/tool descriptions; defer any breaking status rename to a versioned protocol migration.
