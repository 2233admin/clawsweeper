## Summary

Keep open. Current main still does not implement conditional SKILL.md activation based on tool or toolset availability. The shipped skill metadata, parser, docs, and eligibility pipeline only cover OS, binary, env, config, allowlist, and plugin-enabled gating. Related #52861 is broader performance work and does not supersede this narrower skill-spec/API request.

## Best Possible Solution

Keep this open as a core skill-spec/API design item. The best path is to define deterministic tool availability semantics after tool policy resolution, feed that context into skill eligibility or rebuild the skills prompt from effective tools, extend metadata parsing/types/evaluator with positive and fallback gates, and update docs plus focused tests for requires/fallback behavior. If maintainers do not want formal toolsets, narrow the design to explicit tool-name gates and cross-link #52861 for broader token-overhead work.
