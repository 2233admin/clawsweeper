## Summary

Keep open: the request is coherent and current `main` does not implement tool/toolset-based skill activation. OpenClaw’s skill metadata and eligibility pipeline still only gate on OS, bins, env, and config.

## Evidence

- **Docs only describe load-time gating by OS/bin/env/config:** The skills reference says OpenClaw filters skills at load time and lists supported `metadata.openclaw` gates as `always`, `os`, `requires.bins`, `requires.anyBins`, `requires.env`, `requires.config`, `primaryEnv`, and `install`; there is no tool/toolset activation field.
  - file: [docs/tools/skills.md:172](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/docs/tools/skills.md#L172)
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Published skill metadata type has no tool or toolset requirements:** `OpenClawSkillMetadata` only exposes `always`, `skillKey`, `primaryEnv`, `emoji`, `homepage`, `os`, `requires`, and `install`. `requires` is limited to `bins`, `anyBins`, `env`, and `config`.
  - file: [src/agents/skills/types.ts:19](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/skills/types.ts#L19)
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Frontmatter parser does not resolve any tool/toolset gating keys:** `resolveOpenClawMetadata()` reads only the manifest block, `requires`, `install`, and `os`, then returns the same limited metadata shape. No parser path exists for `requires_tools`, `fallback_for_tools`, `requires_toolsets`, or `fallback_for_toolsets`.
  - file: [src/agents/skills/frontmatter.ts:187](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/skills/frontmatter.ts#L187)
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Runtime eligibility check has no session tool availability input:** `shouldIncludeSkill()` decides eligibility from config plus `evaluateRuntimeEligibility()` using OS, remote platforms, bins, env, and config truthiness. It accepts no available-tool or toolset context, so the requested behavior is not implemented in the current load path.
  - file: [src/agents/skills/config.ts:73](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/src/agents/skills/config.ts#L73)
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Repo-wide search found no requested frontmatter keys on current main:** A search for `requires_toolsets`, `fallback_for_toolsets`, `requires_tools`, and `fallback_for_tools` returned no matches in the checked-out tree.
  - command: `rg -n "requires_toolsets|fallback_for_toolsets|requires_tools|fallback_for_tools" src docs skills extensions ui`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)

## Risks / Open Questions

- If this is implemented later, it likely needs session-level tool-policy awareness rather than only the current load-time host/config gating model.
- Because docs position skills as AgentSkills-compatible, adding new frontmatter fields may need a compatibility story or clear OpenClaw-specific namespacing.
