## Summary

Keep open. Current main still does not implement the PR's proposed provider-quota circuit breaker: there is no `quota_exhausted` reason or `agent:provider_tripped` hook on main, periodic usage-window errors still land in the existing `rate_limit` or `billing` lanes, and the documented behavior explicitly keeps retryable daily/weekly/monthly usage-window limits on the short cooldown path. The PR also has unresolved review findings in the provided discussion around HTTP 429 precedence, missing failure-reason scoring, and repeated single-provider probing, so it should not be treated as already shipped or safe to close as implemented.

## Best Possible Solution

Keep the PR open for maintainer judgment, but request a focused rework before merge: separate unrelated status-provider/TUI cost work, add a coherent quota-exhausted contract across failover/auth-profile types, classification priority, HTTP 429 message precedence, disabled backoff, and cooldown probing, then add regression tests for Gemini-style daily quota 429s and provider-disabled fallback behavior. The linked issue #64085 should remain the canonical product problem until that focused implementation lands.
