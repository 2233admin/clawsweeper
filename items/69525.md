## Summary

Keep #69525 open. Current `main` still forwards the source session/account fallback into the target channel path: the message tool writes `agentAccountId` into `params.accountId`, `runMessageAction` also falls back to `input.defaultAccountId`, and the outbound plugin dispatch passes that accountId through to the target plugin. Cross-provider sends are allowed by policy when `tools.message.crossContext.allowAcrossProviders` is true, but there is no source-vs-target provider account reset in the current code. The linked PR #69529 is relevant but is open/unmerged in the supplied GitHub context, so it does not make the issue closable.

## Best Possible Solution

Keep the issue open as the canonical bug until core preserves explicit target `accountId` values but suppresses source-session `agentAccountId`/`defaultAccountId` fallback when the selected target channel differs from `toolContext.currentChannelProvider` and across-provider sends are allowed. The fix likely belongs around `createMessageTool`/`runMessageAction`, with a regression test that simulates Telegram/source account `default` sending to an external plugin channel such as `openclaw-weixin` and asserts the plugin handler receives `accountId: undefined` unless the user explicitly provided one.

## Risks / Open Questions

- A correct fix needs to distinguish a user-explicit target accountId from the source session fallback; current `params.accountId` may already be populated by `createMessageTool` before `runMessageAction` sees it.
- The existing default-account tests will likely need a cross-provider companion case so same-provider defaults remain intact while different-provider sends omit the source account fallback.
