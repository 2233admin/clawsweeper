## Summary

Current `main` still propagates the caller/session `defaultAccountId` into outbound `message` sends before any cross-provider check, and the target channel plugin receives that `accountId` unchanged. I did not find an on-main fix between release `v2026.4.23` and `HEAD`, and the issue already has a linked open PR (`#69529`), so this is not a conservative close candidate.

## Evidence

- **Source session defaultAccountId still becomes outbound accountId on main:** `runMessageAction` resolves `accountId` as `readStringParam(params, "accountId") ?? input.defaultAccountId` and then writes it back into `params.accountId` with no provider-specific reset. That matches the reported leak path for cross-provider sends.
