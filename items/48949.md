## Summary

Keep open. Current `main` still has Feishu proxy handling only on the WebSocket path; the HTTP client path used for Feishu API calls still delegates to the Lark SDK default HTTP instance with only timeout/User-Agent wrapping, so there is not strong evidence that the reported `tenant_access_token` proxy failure is fixed on `main`.
