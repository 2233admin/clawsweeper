## Summary

Keep #71988 open. Current main still lacks the requested global `inbound_claim` broadcast after plugin-bound fallback, and the linked issue #48434 was closed specifically as superseded by this PR. The PR is not ready to close as implemented or duplicate, but it also should not land as-is because the proposed early return would bypass inbound dedupe cleanup and drops synthetic replies from handled broadcast claims.

## Best Possible Solution

Keep #71988 open as the active implementation review. The best path is a narrow core dispatch patch that broadcasts `inbound_claim` only for plugin-bound fallback cases, delivers any synthetic reply consistently with the targeted path, commits/releases inbound dedupe before every terminal return, and adds focused tests for missing-plugin/no-handler broadcast, ordinary core conversations not broadcasting, handled broadcast reply delivery, and dedupe cleanup.

## Risks / Open Questions

- Landing the PR as-is can leave an inbound dedupe key stuck as in-flight when a broadcast `inbound_claim` handler returns `{ handled: true }`.
- Landing the PR as-is can silently drop a synthetic reply returned by a handled broadcast `inbound_claim` listener.
- The PR currently updates existing assertions but does not add a focused positive regression test for handled broadcast behavior, dedupe cleanup, or synthetic reply delivery.
