## Summary

Keep open. Current `main` added bundled Z.AI image-understanding support, but the image tool still resolves explicit image models through a raw Pi model-registry lookup and does not apply the configured provider/baseUrl override path that the main model resolver uses. That means the reported `zai/glm-5v-turbo` routing bug is not clearly fixed on `main`.

## Evidence

- **Image runtime still trusts raw registry result:** `resolveImageRuntime()` writes/loads `models.json`, calls `discoverModels(...).find(...)`, then uses the returned model and its provider for auth without any follow-up merge of configured provider `baseUrl`/`api` overrides.
  - file: [src/media-understanding/image.ts:139](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/media-understanding/image.ts#L139)
  - command: `nl -ba src/media-understanding/image.ts | sed -n '130,170p'`
  - sha: [0e9156d20530](https://github.com/openclaw/openclaw/commit/0e9156d2053022d208c3eea1d0fa1e377ed0a154)
- **General model resolver has override logic that image runtime does not reuse:** The normal embedded-model path has `applyConfiguredProviderOverrides(...)` and calls it after `modelRegistry.find(...)`, explicitly merging configured provider/model metadata such as `baseUrl`, `api`, headers, and request settings before normalizing the model.
  - file: [src/agents/pi-embedded-runner/model.ts:309](https://github.com/openclaw/openclaw/blob/0e9156d2053022d208c3eea1d0fa1e377ed0a154/src/agents/pi-embedded-runner/model.ts#L309)
