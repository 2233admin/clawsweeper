## Summary

Current `main` does not implement the requested CLI surface. `openclaw pairing` only registers `list` for pending requests and `approve`; there is no `list-approved` or `revoke` subcommand. The underlying pairing store already has read/remove helpers, and there is a text-only `/allowlist` command that can list/remove store entries, but that is not the CLI requested here. `channels status --probe` still renders `allow:` from `account.allowFrom` on the status snapshot path and does not read the pairing allow-from store, so the requested merged status view is also not implemented.

## Risks / Open Questions

- A few channel plugins could theoretically inject custom snapshot fields, but there is no generic `channels status` merge from pairing-store approvals on the current main path.
- There is partial overlap with the text-only `/allowlist` command, so maintainers may want to consolidate UX rather than add new `pairing` subcommands, but the issue's explicit CLI request remains unmet.
