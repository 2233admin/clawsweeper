## Summary

Keep #40527 open. Current main partly invalidates the original no-handler diagnosis because the macOS node now advertises and handles `system.which`/`system.run`, but the remaining timeout class is still credible: the gateway starts the remote bin probe during node connect before sending the connect response, while the Swift client only starts its normal event listener after that response and drops unrelated frames while waiting.

## Best Possible Solution

Keep the issue open and fix the gateway/macOS node handshake path. The likely fix is to defer the initial `refreshRemoteNodeBins()` call until after the node has completed the connect response and started its listener, or to make the Swift client buffer/dispatch early `node.invoke.request` frames during connect. A complete fix should also decide whether co-located macOS app nodes should skip remote bin probing when the gateway already has local shell access, and add regression coverage for connect-time remote-bin probing.
