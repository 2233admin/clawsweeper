## Summary

Keep #70333 open. Current main mitigates one older restart-loop path by capturing the startup write hash, but it still materializes auto-enable-derived `enabled: true` keys and persists them during gateway startup and `doctor --fix`, which is the core behavior this issue asks to stop.

## Best Possible Solution

Keep the issue open and implement a runtime-only auto-enable path. Gateway startup and doctor should compute an effective activation config plus `autoEnabledReasons` for plugin/channel loading and diagnostics, but should not persist auto-enable-derived `channels.*.enabled` or `plugins.entries.*.enabled` keys. Update tests that currently assert materialized enabled flags, and keep the existing startup write-hash guard for other intentional config writes.

## Risks / Open Questions

- Changing this behavior needs care around restrictive `plugins.allow`: runtime activation must preserve the effective allowlist used for plugin loading without writing derived entries to `openclaw.json`.
- The existing startup write-hash guard should remain for legitimate startup writes such as generated gateway auth or control UI origin seeding; the fix should narrow auto-enable persistence rather than remove all startup write handling.
