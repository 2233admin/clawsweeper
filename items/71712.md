## Summary

Keep open. Current main has a broad owner-only `cron` tool and gateway `cron.add` path, but it does not implement the proposed narrowed agent-facing scheduling API: no playbook allowlist, no `activeUntil`/expiration field for recurring jobs, no per-agent quota, and no non-forgeable `createdBy` provenance on cron job definitions. The request is a core scheduler/security design issue, not a ClawHub-only extension item.

## Best Possible Solution

Keep the issue open as a core scheduler/security RFC. The likely implementation path is an additive gateway-owned scheduling surface that preserves unrestricted operator `openclaw cron add`, but gives agents only a narrower `schedule_task`-style path with registered playbook ids, gateway-injected creator/session provenance and timestamps, recurring `activeUntil`, per-agent active-job quotas, creation/update audit events, docs, and regression tests. Coordinate the recurring-expiration piece with #71711 and the broader provenance/harness work rather than closing this as ClawHub or already implemented.

## Risks / Open Questions

- This is a security/product design RFC, so maintainer direction is needed on the exact API and trust boundary before implementation.
- Part of the proposed invariant depends on recurring job expiration work tracked separately in #71711, which is still open.
- The current broad owner-only cron tool is useful for trusted operators, so a fix should add a narrower agent-facing path without breaking existing operator cron workflows.
