## Summary

Keep #72369 open. Current main still appears to route the bare `/new`/`/reset` startup prompt into the visible transcript path used by `chat.history` and Control UI WebChat. The related closed WebChat/system-scaffolding reports are adjacent, but they do not cover this specific reset-startup user-turn leak.

## Best Possible Solution

Keep the issue open. The best fix likely belongs in the core WebChat/session transcript path: preserve the startup instruction as internal runtime context for the agent, but avoid persisting or rendering it as an ordinary user-authored turn. A conservative implementation should add explicit metadata or a runtime-only transcript path for bare reset startup turns, render it as a lightweight session divider if needed, and add regression coverage for `chat.send` `/new` or `/reset` plus `buildChatItems`/`chat.history` display behavior.
