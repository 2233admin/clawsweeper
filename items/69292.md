## Summary

Keep open. Current `main` still lacks the PR's proposed doctor-side skip for already discovered manifest channel plugins, and the PR is a focused recent bug-fix candidate rather than obsolete cleanup. It also has substantive unresolved review concerns around workspace plugin activation and restrictive `plugins.allow` behavior, so it needs revision instead of closure.

## Best Possible Solution

Keep the PR open and request a revision. The likely maintainer path is a narrower doctor auto-enable fix that skips redundant `plugins.entries` only when the discovered plugin will actually load without that entry, preserves explicit trust/allowlist semantics for workspace and restrictive-allowlist configs, and adds regression tests for global installed plugins, workspace plugins, bundled plugins, and `plugins.allow` interactions.

## Risks / Open Questions

- The PR's current workspace-origin behavior may regress configured workspace channel plugins because workspace plugins are disabled by default without an allowlist or enabled entry.
- The restrictive `plugins.allow` path needs a deliberate policy decision: blindly skipping can leave a plugin excluded, while blindly writing `plugins.entries` can recreate the duplicate-plugin-id warning.
