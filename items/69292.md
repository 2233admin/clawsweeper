## Summary

Keep open. Current main still materializes `plugins.entries.<id>` for non-built-in configured channel plugin candidates, so this PR is not implemented or superseded. The PR addresses a real core doctor/config boundary rather than a ClawHub plugin request, but it likely needs revision because workspace-origin plugins are disabled by default unless explicitly enabled or allowlisted.

## Best Possible Solution

Keep #69292 open as the canonical core fix candidate. Revise it to separate global and workspace semantics: avoid redundant `plugins.entries` for truly auto-loaded global channel plugins, preserve required trust/allowlist state, and only skip workspace entries when the workspace plugin is already explicitly trusted. Add regression tests for global installed channels, workspace disabled-by-default behavior, bundled channel behavior, and restrictive `plugins.allow` cases.
