## Summary

Keep #39669 open. Current main still lacks a generic queued-message feedback seam for the busy-session followup path, and Feishu/Lark still only wires user-visible feedback through reply-start typing callbacks. Telegram/Discord/Slack have some channel-local initial status reactions, but that does not solve the cross-channel queued-followup request or the concrete Feishu gap.

## Best Possible Solution

Keep the issue open. The best fix is a generic, non-blocking queued-followup feedback seam in the active-run enqueue-followup path, such as an onQueued callback or typed receipt hook on GetReplyOptions and the shared dispatcher pipeline. It should fire before typing cleanup, swallow/log callback failures, let Feishu/Lark attach an immediate reaction or receipt, avoid duplicate signaling for channels with existing status reactions, and add focused tests proving it fires only for busy queued followups.
