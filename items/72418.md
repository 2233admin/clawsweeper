## Summary

Current main still contains the reported gateway behavior. A loopback, no-Origin client can self-declare the backend gateway client id/mode, authenticate with the shared token/password, and have `shouldSkipLocalBackendSelfPairing` return true. The handshake path then preserves self-declared scopes for that backend exception and skips device-pairing enforcement for unpaired backend devices. Existing tests explicitly preserve `operator.admin` for direct-local backend shared-token/password connects without device identity, while linked PR #72419 is still open in the provided GitHub context.

## Best Possible Solution

Keep #72418 open for maintainer security/gateway review. The likely product path is to remove or narrow the direct-local backend shared-token/password exception so backend pairing/scope bypass is not granted solely from client-declared `client.id` and `client.mode`; PR #72419 is the immediate candidate, but it should be reviewed against the intended backend handoff contract and covered with regression tests before merge.
