## Summary

Current `main` still has the bug this PR fixes. `resolveOutputPath()` treats any non-existent `--output` path without a trailing slash as a file path, so `openclaw backup create --output ~/Backups` would still write a gzip archive to `~/Backups` instead of `~/Backups/<timestamp>.tar.gz`.

## Evidence

- **Missing output paths still resolve as files:** In `resolveOutputPath()`, `fs.stat(resolved)` failures are swallowed and the function falls through to `return resolved`, so a non-existent path without a trailing slash is treated as an explicit file path rather than a destination directory.
  - file: [src/infra/backup-create.ts:109](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/infra/backup-create.ts#L109)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Archive writer always emits gzip tar output to that final path:** `createBackupArchive()` uses `outputPath` from `resolveOutputPath()`, creates only `path.dirname(outputPath)`, then writes the archive with `gzip: true`. If `outputPath` is `~/Backups`, the final file is created at that exact path with no `.tar.gz` suffix.
  - file: [src/infra/backup-create.ts:288](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/infra/backup-create.ts#L288)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **User-facing contract allows directory-style output:** The CLI still documents `--output <path>` as an `Archive path or destination directory` and gives `openclaw backup create --output ~/Backups` as the example for writing into a backup directory, which conflicts with current behavior when that directory does not exist yet.
  - file: [src/cli/program/register.backup.ts:23](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/cli/program/register.backup.ts#L23)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Latest tagged release contains the same resolver logic:** The same `catch {}` plus `return resolved` logic is present in `v2026.4.22`, so this is not already fixed in the latest release either.
  - file: [src/infra/backup-create.ts:109](https://github.com/openclaw/openclaw/blob/a94763ee224e858e29dd177cd6f7f4daa438c2f1/src/infra/backup-create.ts#L109)
  - command: `git show v2026.4.22:src/infra/backup-create.ts | nl -ba | sed -n '104,118p'`
  - sha: [a94763ee224e](https://github.com/openclaw/openclaw/commit/a94763ee224e858e29dd177cd6f7f4daa438c2f1)

## Risks / Open Questions

- Closing this PR would leave the reported backup-path bug unresolved on current `main`.
- Current user-facing docs/examples still imply directory semantics for `--output ~/Backups`, so the mismatch can continue to surprise users until the code is fixed.
