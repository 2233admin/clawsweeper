## Summary

Keep #69790 open. Current main has better normal in-process serialization and Telegram DM audio preflight support, but I did not find a shipped fix or regression test for the reported persistent session lock/synthetic inbound failure. The lock remains intentionally non-reentrant by default, embedded Pi attempts still acquire the session write lock without an explicit reentrant option, and an external/internal-dist `bot.handleUpdate()` path can still bypass the gateway session lane that normal inbound relies on.

## Best Possible Solution

Keep the issue open for maintainer triage. The best path is to either add a supported inbound-injection/plugin API that enters through the gateway session lane/reply-run registry and add a regression for existing Telegram DM sessions under active locks, or explicitly document that internal `dist/extensions/telegram/*` `bot.handleUpdate()` injection is unsupported and direct users to the built-in Telegram audio preflight/transcription path.
