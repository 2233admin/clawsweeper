## Summary

Keep open. Current `main` has partial support for Telegram topic icon emoji IDs in action/send paths, but it still does not accept `iconCustomEmojiId` in `TelegramTopicConfig` and there is no boot-time restore path that reapplies topic icons from config on gateway startup.

## Evidence

- **Topic config type still lacks the requested field:** `TelegramTopicConfig` on current `main` ends at `errorCooldownMs` and does not define `iconCustomEmojiId`, so the proposed config shape from the issue is not implemented.
  - file: [src/config/types.telegram.ts:223](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/config/types.telegram.ts#L223)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Config schema would reject the field:** `TelegramTopicSchema` is strict and only allows the existing topic keys; `iconCustomEmojiId` is absent here too, so the config cannot currently persist that value through validation.
  - file: [src/config/zod-schema.providers-core.ts:115](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/config/zod-schema.providers-core.ts#L115)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Action path already supports icon emoji IDs:** The Telegram action runtime already reads `iconCustomEmojiId` for `createForumTopic`, and the send layer passes it through to Telegram as `icon_custom_emoji_id`. This covers the issue's item 3, but not config persistence or restart restore.
  - file: [extensions/telegram/src/action-runtime.ts:598](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/telegram/src/action-runtime.ts#L598)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Edit API supports icon emoji IDs, but only when explicitly invoked:** `editForumTopicTelegram` accepts `iconCustomEmojiId` and sends `icon_custom_emoji_id` to Telegram, so the low-level capability exists.
  - file: [extensions/telegram/src/send.ts:1194](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/telegram/src/send.ts#L1194)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **No boot-time restore path found on current main:** A repo-wide search for `editForumTopic(` in `extensions/telegram/src` finds only two call sites: the send helper itself and one auto-topic-label rename call. There is no startup sweep that reads topic config and reapplies icons after restart.
  - command: `rg -n "editForumTopic\(" extensions/telegram/src`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **Existing runtime edit use is unrelated to restart restore:** The only direct runtime call edits the topic name for DM auto-labeling on first message, not forum-topic icon restoration during startup.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:1233](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/extensions/telegram/src/bot-message-dispatch.ts#L1233)
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
