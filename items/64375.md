## Summary

Keep this PR open. Current main still has the exact `applyJobResult` guard that limits `deleteAfterRun` deletion to `schedule.kind === "at"`, while the PR is a narrow candidate fix for the open linked bug. The overlapping #63871 is closed unmerged, so it is not a canonical shipped replacement.

## Best Possible Solution

Keep this PR open as the focused implementation candidate. The maintainer path is to rebase/review it, decide the durable-audit question separately, land this narrow guard change or an equivalent with regression coverage for successful `every` and `cron` deletion plus failed/skipped retention, then close #63770. The already-closed #63871 should remain non-canonical unless maintainers revive it.

## Risks / Open Questions

- Bot review comments raised a real broader hardening question: self-deleting jobs have no durable audit record before removal. That behavior already exists for one-shot jobs, but extending the flag to recurring schedules broadens the use case and may deserve a separate storage/audit follow-up.
- Docs and UI wording still describe `deleteAfterRun` as one-shot-oriented, so maintainers should decide whether recurring self-delete is intended product behavior and update user-facing text if this PR lands.
