## Summary

Current main still does not make Telegram interactive final/block replies durable across transient sendMessage failures. The shared outbound queue exists for callers that route through deliverOutboundPayloads, and Telegram direct outbound sends have bounded safe-send retry handling, but the interactive Telegram reply dispatcher still calls deliverReplies directly; deliverReplies ultimately calls bot.api.sendMessage and rethrows delivery failures without enqueueing the original reply for later drain.

## Best Possible Solution

Keep this open unless maintainers explicitly consolidate it into #50040. The fix should make Telegram interactive final/block replies durable by routing them through the shared outbound queue or adding a Telegram-owned persistent reply queue that preserves chat, topic/thread, reply/quote, account, session, media, and visibility metadata; separates permanent Telegram 4xx/auth/routing failures from retryable and ambiguous network failures; retries with bounded backoff/TTL; drains on startup and Telegram transport recovery; and adds regression coverage for deliverReplies/dispatcher behavior when sendMessage fails after the assistant answer has been produced.
