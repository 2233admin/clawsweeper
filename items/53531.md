## Summary

Current main still does not implement compaction-local same-provider auth-profile rotation/failover. Direct, manual, queued/context-engine, overflow, and timeout compaction paths still resolve or forward one auth profile for a compaction attempt, while normal prompt/assistant runs use a separate auth-controller candidate rotation path. The idempotency concern raised in the issue discussion remains a real design requirement for any fix.

## Best Possible Solution

Keep this issue open. The best fix is to extract/share auth-profile candidate ordering, cooldown skipping, failure classification, and failure marking for compaction; snapshot the transcript once; retry eligible same-provider profiles against that same snapshot; and commit exactly one successful summary/checkpoint/transcript rotation. Add focused regression tests for manual compaction, preflight budget compaction, overflow-triggered compaction, timeout-triggered compaction, context-engine delegation, and an idempotency case where the first profile fails after the request boundary and a later profile succeeds.

## Risks / Open Questions

- A naive retry loop could double-compact or apply a summary to a mutated transcript; the fix needs one captured compaction snapshot/attempt id and an at-most-one commit boundary.
- Manual `/compact`, preflight budget compaction, overflow recovery, timeout recovery, and context-engine-owned compaction enter through different paths, so a partial fix can leave the reported lockup in place on some surfaces.
