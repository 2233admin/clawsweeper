## Summary

Keep #40840 open. Current main still has a plausible delete-then-re-register race: `sessions.delete` performs active-runtime cleanup and removes the row, but normal agent teardown can still call `updateSessionStoreAfterAgentRun(...)`, which recreates `store[sessionKey]` when the row is missing. Newer `sessions.abort` and subagent kill surfaces help active-run control, but I did not find a hard-kill tombstone/epoch guard or regression coverage for the post-run persistence window. The related #12297 close also treated #40840 as the remaining canonical hard-kill semantics tracker.

## Best Possible Solution

Keep #40840 open. The best fix is a narrow core lifecycle change plus regression test: either make `sessions.delete` establish a durable per-session tombstone/epoch that post-run persistence checks before writing, or make `updateSessionStoreAfterAgentRun(...)` refuse to recreate a row when the backing store no longer contains that session for the same run generation. Document the final contract clearly: `sessions.abort` for active-work interruption, `sessions.delete` for store/transcript deletion, and any native hard-kill primitive if maintainers choose to expose one.
