## Summary

The request is for optional context-management behavior that OpenClaw already scopes to the plugin ecosystem rather than core. Current main has a first-class `context-engine` plugin slot, documents third-party context engines, and already points users to an existing community plugin (`lossless-claw`) for DAG-style context compaction; exec-output compaction is also already handled via the bundled `tokenjuice` plugin.

## Close Comment

Closing this as a plugin-scope request after Codex review.

- `VISION.md` says optional capability should usually ship as plugins, and new skills should go to ClawHub first instead of expanding core.
- OpenClaw already has a first-class `context-engine` plugin slot that owns context assembly, `/compact`, and related lifecycle hooks (`docs/concepts/context.md`, `docs/concepts/context-engine.md`, `docs/plugins/manifest.md`).
- Current docs already point users to `@martian-engineering/lossless-claw`, described as DAG-based conversation summarization with incremental compaction and reduced token usage (`docs/plugins/community.md`).
- The tool-output compaction part is also already on the plugin path via the bundled `tokenjuice` plugin for exec/bash results (`extensions/tokenjuice/package.json`, `extensions/tokenjuice/openclaw.plugin.json`).

If there is a core gap that the existing context-engine/plugin surfaces cannot support, please open a narrower follow-up with that specific limitation.
