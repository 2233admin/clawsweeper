## Summary

Keep open. Current `main` intentionally preserves the active run when a different run emits a `final`, and this PR's proposed behavior would conflict with that concurrent-run model by treating a foreign terminal event as completion of the active run.

## Evidence

- **Web UI controller preserves active run on foreign finals:** `handleChatEvent` appends a displayable `final` message from another run but returns `null` instead of clearing `chatRunId` or stream state, so the active run stays busy until its own terminal event arrives.
  - file: [ui/src/ui/controllers/chat.ts:439](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/ui/src/ui/controllers/chat.ts#L439)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '429,447p'`
  - sha: [7e49cc87f9e9](https://github.com/openclaw/openclaw/commit/7e49cc87f9e9cdad47d0572331d7a7a47281a83e)
- **Unit test locks in that behavior:** Current tests explicitly expect a foreign `final` with message content to append the message without clearing `chatRunId`, `chatStream`, or `chatStreamStartedAt`.
  - file: [ui/src/ui/controllers/chat.test.ts:112](https://github.com/openclaw/openclaw/blob/7e49cc87f9e9cdad47d0572331d7a7a47281a83e/ui/src/ui/controllers/chat.test.ts#L112)

## Risks / Open Questions

- The user-visible stuck-indicator report may still be real for a narrower case, but this PR's approach is too broad because it collapses concurrent-run and foreign-final handling together.
- `chat.inject` and other cross-run terminal events make premature busy-state teardown plausible, including hiding active progress and resuming queued work before the tracked run actually finishes.
