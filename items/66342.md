## Summary

Keep open. Current `main` does not support the PR's stated gateway-crash mechanism, and the proposed `throw`→`return` change would not actually stop Slack restart loops; it would also lose startup error state and the added regression test does not match the current Slack Bolt interop contract.

## Evidence

- **Gateway already catches Slack provider rejection:** `startAccount(...)` is wrapped in `task.catch(...)`, which records `lastError` instead of letting a rejection escape the gateway task. That contradicts the PR premise that a thrown Slack auth error becomes an unhandled rejection that kills the whole gateway.
  - file: [src/gateway/server-channels.ts:431](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/gateway/server-channels.ts#L431)
  - command: `nl -ba src/gateway/server-channels.ts | sed -n '431,436p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **A clean return still triggers channel auto-restart:** After the catch/finally chain, the channel manager unconditionally enters the restart path on task completion unless the account was manually stopped. Replacing Slack auth-error throws with `return` would still restart the Slack account rather than 'stop the provider gracefully'.
  - file: [src/gateway/server-channels.ts:445](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/gateway/server-channels.ts#L445)
  - command: `nl -ba src/gateway/server-channels.ts | sed -n '445,460p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Slack monitor runs through that managed startAccount path:** The Slack plugin's gateway hook returns `monitorSlackProvider(...)` directly from `startAccount`, so Slack startup/auth failures flow through the managed catch/restart path above.
  - file: [extensions/slack/src/channel.ts:559](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/slack/src/channel.ts#L559)
  - command: `nl -ba extensions/slack/src/channel.ts | sed -n '559,576p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Returning on startup auth failure drops status error state:** Slack status helpers clear `lastError` on connect and only populate it when an explicit disconnect/error payload is published. A startup `return` in `monitorSlackProvider` would skip the rejection path that sets `lastError`, matching the review concern that diagnostics would show a normal stop instead of an auth failure.
  - file: [extensions/slack/src/monitor/provider-support.ts:99](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/slack/src/monitor/provider-support.ts#L99)

- **Proposed regression test mock is inconsistent with current Bolt interop:** Current interop resolution requires `SocketModeReceiver`, and the repo's own Slack test helper mock includes it. The PR's added test mock in the provided diff only supplied `App` and `HTTPReceiver`, so it would fail during interop resolution before exercising `app.start()`.

## Risks / Open Questions

- Merging as written would still leave Slack on the channel-manager restart path, so it does not deliver the PR's stated 'stop cleanly' behavior.
- Startup auth failures would lose `lastError`/status visibility because the gateway records that state on rejection, not on a clean return.
- The new test does not currently validate the intended path because its Bolt mock shape is incomplete for socket-mode interop.
