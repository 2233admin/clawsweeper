## Summary

Keep open. Current `main` still has the inline-provider inconsistency the PR describes: `resolveModelWithRegistry` only accepts an inline match when `inlineMatch.api` is present, but `buildInlineProviderModels` still builds transport from raw `entry?.api` and does not apply Anthropic's provider-level default itself. Runtime config materialization applies provider defaults first, which mitigates some production paths, but the underlying gap is still present for raw callers and the requested fix is not implemented on `main`. The PR also now carries an unrelated Telegram retry commit with an existing duplicate-send risk review comment, so it is not a clean close candidate.
