## Summary

Keep #69529 open. Current main still lacks the main cross-provider account-id fix: `runMessageAction` still falls back to `input.defaultAccountId` before dispatch, writes that value into `params.accountId`, and sends it onward to plugin/core outbound handlers. The bundled gateway approval-schema fix and skill-description edits are also not present on main, so this PR still contains useful, unshipped work even though it should be cleaned up before landing.

## Best Possible Solution

Keep the PR open, but ask for maintainer-friendly cleanup before landing: split or clearly scope the unrelated skill-description changes, preserve explicit `accountId` and same-provider behavior, suppress only source-session defaults for allowed cross-provider sends in `src/infra/outbound/message-action-runner.ts`, add a direct regression test for the target plugin context, and add gateway schema/test coverage for optional approval `source` metadata.
