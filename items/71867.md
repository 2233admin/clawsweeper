## Summary

Keep #71867 open. Current main still runs `openclaw doctor` auth health against one resolved auth store, emits a generic `Model auth` note, and has no `--agent` option or multi-agent comparison. `openclaw models status --agent <id>` can inspect a specific store, but that does not fix the doctor output requested by the issue.

## Best Possible Solution

Keep the issue open and implement doctor-side agent attribution. A conservative path is to reuse the existing configured-agent resolution used by `models status --agent`, add an explicit `--agent <id>` or default-agent label to doctor auth health, include the agent id and auth store path in `Model auth` and cooldown notes, and add a multi-agent divergence warning only if doctor intentionally scans configured agent stores. Update `docs/cli/doctor.md` and add focused tests around `noteAuthProfileHealth`/doctor registration.

## Risks / Open Questions

- A fix should avoid making read-only doctor checks write inherited auth profiles into secondary agent stores; the current general store loader can write inheritance data when called with an explicit sub-agent dir in non-read-only paths.
- If doctor enumerates multiple agents, it needs deterministic ordering and non-interactive-safe behavior so keychain/OAuth prompts do not fan out unexpectedly.
