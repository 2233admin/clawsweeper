## Summary

Current main does not implement the requested provider trust boundary/configuration. Explicit `opts.images` are still stripped bluntly on every fallback retry, but prompt-detected image references are still loaded unconditionally in the embedded runner, which leaves the cross-provider image boundary issue represented by #43492/#57959 unresolved on main.

## Evidence

- **Explicit fallback images are stripped without provider/trust policy:** `runAgentAttempt` passes `undefined` for `images` and `imageOrder` whenever `isFallbackRetry` is true, for both CLI and embedded paths. This prevents explicit attachment forwarding, but it is not cross-provider-aware and has no trust configuration.
  - file: [src/agents/command/attempt-execution.ts:351](https://github.com/openclaw/openclaw/blob/1531123d3547f9ff8ee8094590fc773305f018f2/src/agents/command/attempt-execution.ts#L351)
  - command: `nl -ba src/agents/command/attempt-execution.ts | sed -n '232,560p'`
  - sha: [1531123d3547](https://github.com/openclaw/openclaw/commit/1531123d3547f9ff8ee8094590fc773305f018f2)
- **Fallback run options lack failure/trust context:** `ModelFallbackRunOptions` only carries `allowTransientCooldownProbe`; current main does not thread `previousFailureReason`, primary provider, partial execution, or an image trust policy into fallback attempts.
  - file: [src/agents/model-fallback.ts:66](https://github.com/openclaw/openclaw/blob/1531123d3547f9ff8ee8094590fc773305f018f2/src/agents/model-fallback.ts#L66)

There is no suppress flag in `RunEmbeddedPiAgentParams`, so local image references in prompt text can still be loaded even when explicit images were stripped.
