## Summary

Keep open. Current main still lacks a `chat.send` message body byte cap, and this PR is a focused gateway hardening candidate rather than obsolete stale cleanup. The PR needs maintainer review or author revisions for the byte-limit semantics called out in review before it should merge.

## Best Possible Solution

Keep this PR open as a useful gateway hardening candidate. Ask the author or a maintainer to revise the implementation so the 512 KiB policy is defined consistently, ideally with clear raw-vs-normalized byte semantics, no misleading schema character-count guard, updated docs, and focused boundary/multi-byte tests; then run the relevant changed gateway/protocol checks and merge if green.

## Risks / Open Questions

- The PR's schema guard is not equivalent to a UTF-8 byte cap because JSON Schema `maxLength` is character-oriented; it should be clarified, removed, or adjusted before merge.
- The PR currently measures the normalized string, so maintainers should decide whether the policy is raw request bytes or normalized message bytes and test that behavior explicitly.
- Because this is gateway input hardening, changed gateway/protocol checks should run before merge once revisions are made.
