## Summary

Keep open. Current `main` does not implement a `chat.send` 512 KB message cap, and the proposed PR still has unresolved correctness gaps called out in review.

## Evidence

- **Current sanitizer has no length check:** `sanitizeChatSendMessageInput` on `main` only NFC-normalizes, rejects null bytes, and strips control chars; it does not enforce any byte or character ceiling before accepting `chat.send` input.
  - file: [src/gateway/server-methods/chat.ts:634](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/server-methods/chat.ts#L634)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '620,665p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Current protocol schema leaves `message` unbounded:** `ChatSendParamsSchema.message` is still `Type.String()` with no `maxLength`, so there is no schema-level `chat.send` size guard on `main`.
  - file: [src/gateway/protocol/schema/logs-chat.ts:35](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/gateway/protocol/schema/logs-chat.ts#L35)

## Risks / Open Questions

- Large `chat.send` inputs on current `main` can still consume memory/CPU up to the broader 25 MB WebSocket payload ceiling.
- Merging the PR as written would document a 512 KB limit but still leave edge cases where schema and raw-input enforcement do not match that contract.
