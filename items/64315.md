## Summary

Keep open. Current main has adjacent status surfaces, but it does not implement optional horizontal auto-scrolling or marquee-style overflow behavior for long status text. Control UI topbar/status regions still use ordinary flex, wrapping, nowrap, or ellipsis behavior, and the TUI header/status/footer are assembled as fixed strings without terminal-width scrolling logic. Existing docs describe /status and /usage as the way to inspect richer status/token details, not an always-visible scrolling compact status surface.

## Best Possible Solution

Keep this open as a UI/product enhancement. The best path is an opt-in overflow status component for Control UI surfaces using measured overflow, reduced-motion handling, and hover/focus pause, plus a separate TUI width-aware status/header/footer renderer if terminal support is desired. If plugin-provided status text becomes part of the scope, define the plugin status contribution contract before wiring it into compact bars.
