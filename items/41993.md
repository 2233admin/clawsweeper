## Summary

Keep open. Current `main` still implements the fail-on-any-blocked-address behavior described in #41993, so the report is not implemented on `main` and has not been obsoleted by the latest release I could inspect.

## Evidence

- **Current SSRF guard still fails on any blocked DNS answer:** `assertAllowedResolvedAddressesOrThrow` iterates every resolved address and throws immediately when any entry is classified as blocked.
  - file: [src/infra/net/ssrf.ts:237](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/infra/net/ssrf.ts#L237)
  - command: `nl -ba src/infra/net/ssrf.ts | sed -n '228,396p'`
  - sha: [7ac35b4f6900](https://github.com/openclaw/openclaw/commit/7ac35b4f69009490180a35bf404638d4280a4865)
- **The throw happens before IPv4 preference/deduping:** `resolvePinnedHostnameWithPolicy` calls `assertAllowedResolvedAddressesOrThrow(results, params.policy)` before `dedupeAndPreferIpv4(results)`, so a mixed `[public IPv4, blocked IPv6]` answer set still rejects before the IPv4-preference logic can help.
  - file: [src/infra/net/ssrf.ts:365](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/infra/net/ssrf.ts#L365)

- **web_fetch still uses the pinned SSRF resolution path:** `fetchWithSsrFGuard` resolves the target hostname through `resolvePinnedHostnameWithPolicy` on the normal direct path, so this behavior is still on the `web_fetch` request path.
  - file: [src/infra/net/fetch-guard.ts:367](https://github.com/openclaw/openclaw/blob/7ac35b4f69009490180a35bf404638d4280a4865/src/infra/net/fetch-guard.ts#L367)

  - file: [src/infra/net/ssrf.ts:237](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/infra/net/ssrf.ts#L237)
