## Summary

Keep open. Current `main` still turns background `exec` completions into heartbeat-driven user replies when a chat route exists, and I found no cron-specific suppression or audience classification that would keep cron-owned completions internal.

## Evidence

- **Background exec completion still enqueues a system event and heartbeat wake:** `maybeNotifyOnExit` always formats an `Exec completed/failed ...` summary, enqueues it as a system event, and requests an `exec-event` heartbeat. There is no branch here for cron-owned sessions.
  - file: [src/agents/bash-tools.exec-runtime.ts:279](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/agents/bash-tools.exec-runtime.ts#L279)
  - command: `nl -ba src/agents/bash-tools.exec-runtime.ts | sed -n '276,308p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Heartbeat still asks the model to relay exec results to the user:** When a delivery channel/to is available, `canRelayToUser` is true, and exec-event prompts say `Please relay the command output to the user...` rather than keeping it internal.
  - file: [src/infra/heartbeat-runner.ts:841](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/src/infra/heartbeat-runner.ts#L841)
