## Summary

Keep this PR open. Current main still lacks the root `--log-file` option and `OPENCLAW_LOG_FILE` file-sink override, and the linked issue discussion explicitly scoped the shipped fix to config-backed `logging.file` only. The PR is a plausible implementation candidate, but its current shape still needs route-first handling and daemon-status service-env handling before merge.

## Best Possible Solution

Keep this PR open as the implementation candidate, but require revisions before merge: add the root `--log-file` parser/help surface, promote it before both route-first and Commander execution, add an explicit `OPENCLAW_LOG_FILE` resolver with precedence over `logging.file`, thread an env parameter through log-path resolution so daemon status reports the inspected service environment, and retain focused tests/docs for those boundaries. Maintainers should also decide whether the dedicated CLI flag is the desired product surface now that the linked `logging.file` bug is already closed.

## Risks / Open Questions

- The current PR shape promotes `--log-file` only in Commander preAction, so route-first commands such as `status`, `gateway status`, and routed config commands can still ignore the override.
- Daemon status needs log-file resolution to accept an explicit environment and use the inspected service environment; otherwise CLI env and service env can produce misleading `File logs` output.
- The user-selected log path is not a supply-chain or dependency risk, but it is a file-write surface and should remain constrained to the same local operator/config authority model as existing `logging.file`.
