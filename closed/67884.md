## Summary

Keep open. The proposed normalization strips provider-owned `user:` / `channel:` conversation identity prefixes from `message_received`/`message_sent` hook contexts, but current `main` and production channel resolvers treat those prefixes as canonical for some channels. The review comments correctly identify a regression risk, so this PR is not conservative maintainer-cleanup material.

## Evidence

- **Hook mapper paths are intentionally asymmetric on current main:** `toPluginMessageContext` currently passes `canonical.conversationId` through unchanged, while `toPluginInboundClaimContext` goes through `resolveInboundConversation()`, which preserves plugin-resolved canonical ids and only strips prefixes on the generic fallback path.
  - file: [src/hooks/message-hook-mappers.ts:220](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/hooks/message-hook-mappers.ts#L220)
  - command: `nl -ba src/hooks/message-hook-mappers.ts | sed -n '220,305p'`
  - sha: [fa976e5b93b3](https://github.com/openclaw/openclaw/commit/fa976e5b93b3819f9e137fe08635764edee38290)
- **Current tests already assert prefixed inbound canonical ids from a channel resolver:** `message-hook-mappers.test.ts` registers a `claim-chat` resolver that returns `channel:<id>` and `user:<id>`, and the existing expectations assert those prefixed values are preserved in inbound-claim context.
  - file: [src/hooks/message-hook-mappers.test.ts:53](https://github.com/openclaw/openclaw/blob/fa976e5b93b3819f9e137fe08635764edee38290/src/hooks/message-hook-mappers.test.ts#L53)

## Risks / Open Questions

- There may still be a real mismatch for non-resolver channels where outbound hook contexts keep channel-prefixed ids while inbound claim fallback strips them, so the underlying problem may need a narrower fix rather than closure.
