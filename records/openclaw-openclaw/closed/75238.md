## Summary

Close as duplicate/superseded. The reported Telegram topic symptom matches the all-channel group/channel visible-reply default already handled under #74876: v2026.4.27 made normal group/channel final replies private unless the agent uses the message tool or the operator opts back into automatic room posting. Current main documents and warns about that behavior, while the Telegram topic plumbing still preserves topic session and thread metadata when visible delivery is enabled.

## What This Changes

The issue asks OpenClaw to restore Telegram forum-topic assistant reply delivery after v2026.4.27 so generated replies are visibly sent to the originating group topic with the correct chat and thread identifiers.

## Best Possible Solution

Keep the canonical group/channel visible-reply policy from #74876 and use `messages.groupChat.visibleReplies: "automatic"` for deployments that want legacy automatic group/topic posting. New reports should focus only on Telegram delivery that still fails after that config or after an explicit message-tool send.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: process a group/channel turn without a visible-reply override and current main resolves it to `message_tool_only`, processes the agent response, and does not queue a normal final send.

## Solution Assessment

Yes. Consolidating this under #74876 is the narrowest maintainable outcome because the observed absence of `sendMessage` is explained by the documented all-channel group visible-reply default, while Telegram-specific topic routing remains covered in source and tests.
