## Summary

Keep open. Current main still does not surface session `label` or `displayName` through the `/status` status payload, active CLI sessions table, or TUI gateway-status summary. This PR remains a plausible implementation candidate for #51184, but it is stale against the current CLI render path and the review comments identify real resolver and terminal-sanitization issues that should be addressed before merge.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best implementation is to add `label` and `displayName` to the status payload/types, populate them from `SessionEntry`, update the current CLI report-data/session-row path and TUI gateway-status formatter, prefer a resolvable selector such as `label` or display both friendly name and key when needed, sanitize terminal-rendered friendly names with the existing text sanitization utilities, and add targeted tests for friendly labels plus control-character/ANSI input.

## Risks / Open Questions

- Merging the PR as-is would not update the active CLI `openclaw status` sessions table on current main because the render path has moved to `status.command-report-data.ts` and `status.command-sections.ts`.
- The proposed `displayName ?? label ?? key` fallback can show an unresolvable value when `displayName` differs from the stored `label`; gateway resolution accepts `key`, `sessionId`, or `label`, not `displayName`.
- Rendering friendly names without sanitization could allow newlines, control characters, or ANSI escapes from labels/thread metadata to affect CLI or TUI terminal output.
- There is no focused regression coverage proving friendly labels or unsafe friendly-name text in the status payload, CLI sessions table, or TUI gateway-status summary.
