## Summary

Keep open. Current main has partial mitigations for generic `/stop`/English stop commands and safer denied-exec copy, but it does not fully solve this report: normal-session exec denials still intentionally start an agent follow-up, the exact reported Chinese stop phrases are not recognized as high-priority stop commands, and the task-flow path does not implement the proposed consecutive-denial circuit breaker. The adjacent #69570 tracks async completion cascades, but it does not cover all denial-specific behavior in #69386.

## Best Possible Solution

Keep #69386 open until the denial-specific runaway path is fixed or deliberately consolidated into a canonical issue that covers it. A good fix would add a denial circuit breaker or suppression policy for repeated exec denials, prevent denied completion details from automatically spawning more agent work after user stop/deny intent is clear, expand shared/UI stop phrase handling for the reported Chinese variants, and ensure pending completion/follow-up queues can be cancelled or deprioritized when the user aborts. If maintainers want to consolidate, #69570 can track the broader async completion cascade, but it needs explicit denial-loop and stop-command acceptance criteria first.
