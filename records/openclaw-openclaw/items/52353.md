## Summary

Current main still does not provide the requested plugin-observable embedded-run timeout event. The run-deadline path logs and aborts, the outer runner synthesizes a user-facing timeout payload, and the existing lifecycle error emitter only covers assistant error stops without a structured `errorType: "timeout"` contract. Related timeout configurability work does not supersede this plugin event gap.

## Best Possible Solution

Keep this open and implement an additive embedded-run lifecycle timeout contract. Emit exactly one plugin-observable `lifecycle:error` event for final terminal embedded run/model timeout, with optional structured fields such as `errorType: "timeout"`, timeout kind, duration, run/session context, provider/model/profile context when sanitized, and recovery/snapshot metadata if available. Do not emit for intermediate timeout-compaction recovery or a silent idle-timeout retry that later succeeds. Add focused tests for event delivery and no-duplicate behavior, then document the optional lifecycle error fields in the public plugin runtime docs.

## Risks / Open Questions

- Adding the event in the raw timer handler could emit a terminal error before timeout compaction or the one silent same-model idle-timeout retry succeeds; the fix should emit only for final terminal timeout outcomes.
- The lifecycle event must avoid duplicate terminal events when the subscription handler already emits an assistant-error lifecycle event.
- Provider/model/profile context is useful for plugins, but the payload should preserve existing redaction behavior and avoid raw credential or provider-error leakage.
- Timeout classification should distinguish run deadline, model idle timeout, and compaction-grace timeout enough that recovery plugins do not retry too early or repeatedly.
