## Summary

Keep this issue open. Current main still has the embedded-run timeout path log and abort without emitting the requested plugin-observable lifecycle timeout event, and existing lifecycle error events do not expose `errorType: "timeout"` or equivalent structured timeout classification.

## Best Possible Solution

Keep this open and implement an additive embedded-run lifecycle error contract. The likely code path is the embedded attempt timeout/finalization boundary plus the existing lifecycle emitter: emit exactly one plugin-observable `lifecycle:error` event for terminal embedded run/model timeout with optional structured fields such as `errorType: "timeout"`, timeout kind, run/session ids, duration, provider/model/profile context when sanitized, and recovery/snapshot metadata if available. Add focused tests for run-deadline timeout, idle-timeout retry/final surfacing, compaction grace behavior, and `api.runtime.events.onAgentEvent()` delivery, then document the optional lifecycle error fields in the public plugin runtime docs.

## Risks / Open Questions

- Prematurely closing this would hide a real plugin API gap: timeout handling is visible to users, but not reliably visible to `api.runtime.events.onAgentEvent()` subscribers.
- A fix needs to avoid duplicate terminal lifecycle events because normal assistant errors already emit lifecycle terminal events through the subscription path.
- The timeout signal should distinguish terminal run timeout, same-model idle-timeout retry, and compaction grace timeout so recovery plugins do not retry too early or too often.
- Plugin-visible payloads should keep provider/model/profile context useful while preserving existing redaction behavior and avoiding raw credential or provider-error leakage.
