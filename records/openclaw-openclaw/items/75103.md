## Summary

Keep open. Current main still lacks process-starttime disambiguation in the Plugin SDK file-lock helper, while the PR applies the same raw `/proc` starttime pattern already used by adjacent lock code and adds focused regression coverage. I found no blocking functional or security issues in the patch from the available diff and local source review.

## What This Changes

The PR adds optional Linux process starttime persistence and comparison to `src/plugin-sdk/file-lock.ts`, plus unit tests for writer persistence, mismatched-starttime recovery, legacy fallback, and retry recovery.

## Best Possible Solution

Land the narrow Plugin SDK lock hardening after maintainer review and CI/Testbox validation, keeping the lock payload extension internal, optional, and compatible with legacy and non-Linux lock files.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to precreate a `.lock` file with the current PID, fresh `createdAt`, and a mismatched Linux `starttime`; current main treats the alive PID as non-stale until the age fallback, while the PR reclaims it.

## Solution Assessment

Yes. The patch is the narrowest maintainable fix because it reuses the existing raw `/proc` helper, matches adjacent lock semantics, avoids public SDK API changes, and preserves the current fallback when starttime evidence is missing.
