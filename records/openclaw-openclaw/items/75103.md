## Summary

Keep open. Current main still lacks Plugin SDK lock-owner start-time disambiguation, while this PR is a narrow implementation that matches the existing raw `/proc` start-time pattern used by adjacent lock helpers and includes focused regression coverage. I did not find a blocking correctness or security issue in the diff; the remaining action is maintainer review plus CI/Testbox validation.

## What This Changes

The PR adds an optional internal `starttime` field to Plugin SDK lock payloads, compares it with the live PID start time during stale-lock detection, and adds tests for persistence, mismatch recovery, legacy fallback, and spy cleanup.

## Best Possible Solution

Land the narrow Plugin SDK lock hardening after maintainer review and current-main validation, keeping the payload addition internal and optional while avoiding a broader shared lock refactor in this PR.

## Reproduction Assessment

Yes. A high-confidence reproduction is to precreate a `.lock` file for `acquireFileLock` with the current live PID, a fresh `createdAt`, and a mismatched Linux `starttime`; current main treats the PID as live and waits, while the PR reclaims the lock.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix: it reuses the existing raw `getProcessStartTime` helper, preserves SDK public signatures, and keeps legacy/non-Linux payload behavior on the existing fallback path.
