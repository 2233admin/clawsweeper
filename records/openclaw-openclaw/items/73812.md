## Summary

Keep open. The PR is useful and targets real current-main gaps, but it is not merge-ready because the new per-request deferral timeout is still lost when an explicit restart coalesces onto an older pending restart, and the user-facing fix still needs release-note coverage.

## What This Changes

The PR forces restart-sentinel continuation turns to use automatic source reply delivery, adds a per-request explicit restart deferral timeout for gateway-tool and chat restart paths, and extends restart continuation and deferral tests.

## Best Possible Solution

Revise the PR so restart-sentinel continuations explicitly request automatic source delivery, explicit restart timeout policy is centralized in the restart module, pending restart state stores and updates the effective per-request timeout during coalescing, and the changelog records the user-facing fix. The already-in-flight continuation behavior from #74424 should be handled here if it stays within the same restart continuation contract, or left as a clearly linked follow-up if maintainers want a separate change.

## Reproduction Assessment

Yes. Current main can be checked by dispatching a restart continuation from a Telegram-style group/topic session and observing no requested automatic source delivery, then by setting gateway.reload.deferralTimeoutMs to 0 with a nonzero pre-restart pending count. The PR-specific timeout gap is reproduced when an older pending restart without a timeout is followed by an explicit restart that coalesces with a per-request timeout.

## Solution Assessment

No as submitted. The automatic source-delivery override is the right narrow fix, but the timeout needs to be kept in pending restart state and shared through one exported constant so coalesced explicit restarts cannot silently keep the old indefinite wait.

- **[P1] Propagate deferral timeouts through coalesced restarts:** `src/infra/restart.ts:739-746`
  - body: The new timeout is read from the opts captured by the first scheduled restart. If a later explicit restart with deferralTimeoutMs coalesces onto an already-pending restart that had no per-request timeout, the coalesced branch only updates emit hooks, so the pending closure can still wait indefinitely when runtime config disables the timeout. Store the effective timeout with pending restart state and update it on coalescing.
