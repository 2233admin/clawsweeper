## Summary

Keep this PR open. Current main has not implemented AST-heuristic code chunking for memory sources: the shipped memory host still indexes Markdown plus enabled multimodal files, text indexing still calls `chunkMarkdown()`, and the public/config surface only exposes fixed token/overlap chunking. The PR remains a plausible implementation candidate, but it needs rebase/adaptation to the current `memory-core`/`memory-host-sdk` layout and should address the remaining watcher/data-scope review concerns before merge.

## Best Possible Solution

Keep the PR open as useful but not merge-ready. The best path is to rebase the idea onto the current memory-core/memory-host-sdk architecture, add a code-aware chunking strategy or equivalent host-SDK seam, update allowed-file and watcher behavior consistently, document the expanded indexing scope, and decide whether to fold it into #68019 only if that PR explicitly covers code-file declaration-boundary chunking.

## Risks / Open Questions

- The PR expands what can be sent to embedding providers from Markdown memory notes to source-code files under memory paths; maintainers should confirm this is intended, documented, and opt-in enough for the memory threat model.
- A later review comment in the provided context notes lowercase-only watcher globs for uppercase code extensions. If this PR is revived, that watch-mode stale-index edge case should be fixed in the rebased implementation.
- The PR targets older `src/memory/*` paths while current main has memory host internals under `packages/memory-host-sdk` and runtime manager code under `extensions/memory-core`, so applying it as-is would not match current architecture.
