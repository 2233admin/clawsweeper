## Summary

Keep open. Current main still does not support code-file memory entries or declaration-aware code chunking, but this branch needs maintainer review before merge because it targets the old memory layout, overlaps the broader open chunking work in #68019, and still has a concrete watcher parity bug plus a data-scope/privacy question.

## What This Changes

The PR adds code-file detection, regex declaration-boundary chunking, indexing dispatch, watcher updates for code extensions, and code-chunking tests under the old `src/memory` layout.

## Best Possible Solution

Land a current-main implementation that treats source-code memory files as an explicit supported input type in the memory host SDK and memory-core plugin, with discovery, watcher/filter parity, chunk-output reindex metadata, focused tests, user-facing docs for the expanded indexed data scope, and coordination with #68019's chunking strategy direction.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Watch uppercase code extensions:** `src/memory/manager-sync-ops.ts:390-392`
  - body: `detectCodeLanguage()` lowercases the extension, so `memory/example.TS` is accepted and tagged as code, but these watcher globs are built only from lowercase `CODE_EXTENSIONS` keys such as `*.ts`. On case-sensitive filesystems, edits or deletes to uppercase-extension code files will not mark memory dirty, leaving indexed chunks stale after startup. Use case-insensitive patterns or the current main directory-watch filter style so discovery and watch behavior match.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: No new dependency or CI execution surface is introduced, but the PR expands memory indexing to source files and needs an explicit data-scope review.

Concerns:

- **[medium] Expanded indexed-data scope lacks guardrails:** `src/memory/internal.ts:129`
  - body: The PR accepts recognized code extensions in `isAllowedMemoryFilePath()` and routes code chunks through the existing embedding path. With a remote embedding provider configured, source contents placed under memory paths can be sent off-host, but the branch does not add docs, opt-in language, or migration guidance for that expanded data scope.
