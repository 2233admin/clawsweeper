## Summary

Keep open. Current main has improved the misleading warning path, but it has not resolved the routed-provider execution problem: `apply_patch` is still only constructed for exact `openai` or `openai-codex` provider IDs, so Tailscale/PPQ-style provider routing can still omit the tool before policy filtering runs.

## What This Changes

The issue asks for `apply_patch` to be recognized and usable as a supported built-in tool in agent-routed OpenAI/Codex runs instead of being treated as unavailable or unknown during allowlist resolution.

## Best Possible Solution

Define the `apply_patch` capability contract for routed/OpenAI-compatible providers, then either add a trusted provider-alias or explicit capability override with focused tests and docs, or keep the strict provider gate and make diagnostics/docs explicit for routed-provider setups.

## Reproduction Assessment

Yes. A source-level reproduction is to resolve coding tools with `tools.exec.applyPatch` enabled and a non-exact routed provider ID; current code makes `applyPatchEnabled` false and omits `apply_patch` before policy filtering. I did not run a live Tailscale or PPQ gateway session in this read-only review.

## Solution Assessment

No. The workaround proposed in the issue targets `coreToolNames` classification, but current main shows the tool can be absent before the policy pipeline runs. The maintainable fix is an explicit routed-provider capability or trusted alias contract, or clearer diagnostics/docs if strict gating is intentional.
