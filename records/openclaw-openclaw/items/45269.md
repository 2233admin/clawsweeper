## Summary

Keep open. Current main has improved the misleading policy-warning path, but the routed-provider execution problem remains: `apply_patch` is still only constructed for exact `openai` or `openai-codex` provider IDs, so Tailscale/PPQ-style routed OpenAI-compatible setups can still lose the tool before policy filtering runs. Because this is a workspace-writing built-in, the remaining fix needs maintainer direction rather than an automatic cleanup close.

## What This Changes

The issue asks for `apply_patch` to be treated as an available core tool for eligible agent-routed runs instead of being dropped or warned as unknown/plugin-only during tool policy resolution.

## Best Possible Solution

Define the `apply_patch` capability contract for routed/OpenAI-compatible providers, then either add a guarded provider-alias or explicit capability override with tests and docs, or deliberately keep the strict provider gate and make diagnostics/docs explicit for routed-provider setups.
