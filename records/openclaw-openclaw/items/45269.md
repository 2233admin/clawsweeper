## Summary

Keep open. Current main has improved the noisy allowlist-warning path for gated core tools, but the routed-provider availability question has not been proven fixed: `apply_patch` is still created only for exact `openai` or `openai-codex` provider ids, while the discussion reports Tailscale and dynamic gateway routing. The related PR #45293 was closed unmerged after its author revised the diagnosis, and #43042 was closed as a duplicate pointing back here, so this remains the canonical item for maintainer triage.

## What This Changes

The issue asks for `apply_patch` to be recognized as a valid built-in tool during agent policy resolution so configured agent-routed runs can execute it instead of losing it as an unknown allowlist entry.

## Best Possible Solution

Define the supported capability contract for `apply_patch` under provider gateways: either preserve/recognize upstream OpenAI/OpenAI Codex identity for routed OpenAI runs, add a documented explicit override with tests and safety checks, or document that dynamic/non-OpenAI gateway routes cannot expose this tool.
