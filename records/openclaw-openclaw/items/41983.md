## Summary

Keep this PR open. Current main still aggregates known control-plane write clients by `deviceId+clientIp`, using `connId` only when both identity and IP are unknown, so the PR's intended per-connection isolation is not implemented or superseded. This is a real gateway policy/security tradeoff rather than a refactor-only cleanup.

## Best Possible Solution

Keep the PR open for maintainer policy review or author update. If maintainers want per-connection write budgets, the best path is to rebase onto current main, preserve bucket pruning and the hard cap, rename/update the stale keying test, add a regression test for same-device/IP multi-connection isolation, and update gateway docs/changelog to document the new rate-limit scope. If maintainers prefer the current aggregate `deviceId+clientIp` policy, close the PR manually with that product/security rationale.

## Risks / Open Questions

- Per-connection budgets would let multiple concurrent sessions from the same device/IP multiply the effective `config.apply`/`config.patch`/`update.run` budget, so maintainers should explicitly accept that gateway policy tradeoff before merge.
- The branch body is still the blank PR template and lacks author-provided reproduction, security-impact answers, and human verification.
- The PR should preserve current main's stale-bucket pruning and 10,000-entry cap when rebased or merged, since earlier review discussion identified memory growth as a risk for per-connection keys.
