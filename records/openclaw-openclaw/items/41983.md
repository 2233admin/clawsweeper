## Summary

Keep this PR open. Current main still keys known control-plane write clients by deviceId+clientIp, using connId only as a fallback when both identity and IP are unknown. The PR proposes a real gateway behavior and security-policy change, not an implementation already present on main or clearly superseded by a related item.

## Best Possible Solution

Keep the PR open for maintainer review or author update. The best path is to rebase onto current main, preserve bucket pruning and the hard cap, rename/update the stale keying test, add a regression test proving the intended same-device/IP multi-connection behavior, and update gateway docs/changelog if maintainers choose per-connection write budgets as the product/security policy.

## Risks / Open Questions

- Per-connection budgets would let multiple concurrent sessions from the same device/IP multiply the effective config.apply/config.patch/update.run budget, so maintainers should explicitly accept that product/security tradeoff before merge.
- A rebase must preserve current stale-bucket pruning and the 10,000-entry hard cap; dropping either would reintroduce the memory-growth concern raised in the PR discussion.
- The PR body is still the blank template and lacks author-provided reproduction, security-impact answers, and human verification.
