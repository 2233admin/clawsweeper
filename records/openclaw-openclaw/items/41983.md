## Summary

Keep this PR open. Current main still aggregates control-plane write budgets by `deviceId+clientIp` for known gateway clients, while this PR proposes a real gateway behavior change to isolate those budgets by `connId`. The work is not implemented or superseded on main, but it needs maintainer product/security review and a rebase that preserves current bucket pruning and hard-cap safeguards.

## Best Possible Solution

Keep this PR open for maintainer review or author update. The best path is to rebase onto current main, preserve bucket pruning and the hard cap, rename/update the stale keying test, add a regression test proving the intended same-device/IP multi-connection behavior, and update gateway docs/changelog if maintainers choose per-connection write budgets as the product/security policy.

## Risks / Open Questions

- Per-connection budgets would allow multiple concurrent sessions from the same device/IP to each receive a separate 3-per-60s budget for `config.apply`, `config.patch`, and `update.run`; maintainers should explicitly accept that security/product tradeoff before merge.
- A naive rebase could drop the current stale-bucket pruning or 10,000-entry hard cap and reintroduce unbounded memory growth under reconnection churn.
- The PR body is still the blank template and lacks author-provided reproduction, security-impact answers, and human verification.
