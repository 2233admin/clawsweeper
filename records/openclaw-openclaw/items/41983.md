## Summary

Keep this PR open. Current main still keys control-plane write budgets by `deviceId+clientIp` for identified or known-IP clients, so the PR's intended per-connection isolation is not implemented or clearly superseded. The PR is a focused gateway behavior change, but it needs maintainer policy review and a rebase that preserves current main's bucket-pruning and hard-cap safeguards.

## Best Possible Solution

Keep this PR open for maintainer policy review or an author update. If per-connection control-plane write budgets are desired, rebase onto current main, preserve the pruning and 10,000-entry cap, rename/update the stale keying test, add a same-device/IP multi-connection regression test, and update the gateway docs/changelog. If maintainers prefer the current aggregate `deviceId+clientIp` policy, close the PR manually with that product/security rationale.

## Risks / Open Questions

- Per-connection write budgets would let multiple simultaneous sessions from the same device/IP multiply the effective `config.apply`/`config.patch`/`update.run` budget; maintainers should explicitly accept or reject that policy change.
- The PR head predates current main's stale-bucket pruning and 10,000-entry cap; a rebase must preserve those safeguards to avoid unbounded growth under reconnection churn.
- The PR body is still the blank template and lacks author-provided reproduction, security-impact answers, docs/changelog updates, and human verification for a gateway control-plane behavior change.
