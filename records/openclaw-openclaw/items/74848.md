## Summary

Keep open. Current main still has a plausible native macOS node-only WSS stale-pin failure path for non-loopback, non-`.ts.net` hosts, and an open PR now explicitly targets this issue with closing syntax, so the issue should remain open until that PR lands or maintainers choose a different TLS trust policy.

## What This Changes

The issue reports that the macOS app's native node repeatedly disconnects from a remote WSS gateway with `cancelled` while the CLI node works against the same gateway.

## Best Possible Solution

Review and either land #75228 or replace it with an equivalent maintainer-approved patch that repairs stale pins for system-trusted public WSS certificate rotations while preserving fail-closed behavior for untrusted, self-signed, private-CA, and mismatched-store-key cases; close this issue only after that fix merges.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The reporter logs plus current source give a strong stale-pin mismatch hypothesis, but a deterministic reproduction still needs the reporter's stored macOS pin state or a macOS WSS setup where a non-`.ts.net` public certificate has rotated after the app stored the old fingerprint.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The linked PR's system-trust-gated auto-repair direction looks narrow and aligned with the observed failure, but maintainers should explicitly approve the TLS trust policy because it overlaps the security-sensitive pinning work in #50642.
