## Summary

Keep open. Current main still restricts stale TLS pin auto-repair to loopback and `*.ts.net` hosts, so the public WSS host described by the reporter can still hit the macOS app-only pinned-session reconnect path while the CLI node path avoids pinning unless a fingerprint is supplied. Open PR #75228 already targets this behavior with closing syntax, so this issue should stay open until that PR or an equivalent fix is reviewed and merged.

## What This Changes

The issue asks OpenClaw to fix a macOS companion app remote WSS node reconnect loop after a stale TLS pin/certificate rotation while `openclaw node run --tls` works against the same gateway.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS hosts, keeps self-signed/private-CA/untrusted mismatches fail-closed, and preserves focused Swift coverage for the trust-policy boundary.

## Reproduction Assessment

Yes for the policy gap: current main has a focused unit-level reproduction where a system-trusted `wss://gateway.example.com` pin mismatch returns `false` from `shouldAutoRepairStaleTLSPin`, and the pinning session cancels mismatches. A full live gateway certificate-rotation reproduction was not performed in this read-only pass.

## Solution Assessment

Unclear for the exact proposed patch until maintainers review the TLS policy change, but the direction in #75228 is the narrowest visible fix for the app-vs-CLI difference. Any accepted solution should gate repair on structured `.pinMismatch`, matching store key, `wss://`, and successful system trust while preserving fail-closed behavior for untrusted chains.
