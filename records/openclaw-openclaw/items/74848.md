## Summary

Keep open. Current main contains a partial post-report macOS stale TLS pin fix, but it only auto-repairs trusted loopback and `*.ts.net` pin mismatches; the reported custom WSS domain remains a plausible native macOS-node-only failure path while the app control channel and CLI node can still work through separate TLS/session paths.

## What This Changes

The issue reports that the macOS app’s remote node repeatedly disconnects from a WSS gateway with `cancelled` while the app control channel and CLI node work against the same gateway.

## Best Possible Solution

The desired end state is a macOS native node WSS trust flow that handles stale custom-domain certificates through an explicit, secure review/reset or configured fingerprint path, keeps CLI and app-node behavior understandable, and gives actionable UI/log output instead of a repeated `cancelled` loop.

## Reproduction Assessment

No. The logs and source give a strong code-level reproduction path with a stale stored WSS fingerprint, but the reporter’s exact host requires their certificate/keychain state or a controlled macOS direct-WSS setup with certificate rotation.

## Solution Assessment

Unclear. The current Tailscale/loopback auto-repair in #75038 is a narrow improvement, but a broader custom-domain repair needs maintainer/security agreement on certificate re-trust policy before implementation.
