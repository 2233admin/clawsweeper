## Summary

Keep open. Current main still leaves the reported public `wss://` macOS app-node stale TLS pin path unhandled, and open PR #75228 is an unmerged closing PR for the same behavior, so this issue should remain open pending review or rejection of that PR.

## What This Changes

The issue reports that the macOS app remote node repeatedly disconnects with a `cancelled` WSS gateway error while a CLI node connects to the same gateway successfully.

## Best Possible Solution

Review and either land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS hosts, keeps self-signed/private-CA/untrusted mismatches fail-closed, and preserves focused Swift coverage for the policy boundary.

## Reproduction Assessment

Unclear. The source-level reproduction path is strong: a stored pin mismatch on a trusted public `wss://gateway.example.com` still returns `false` in the current unit test, but a live repro would need the reporter's stale keychain pin state or a controlled public certificate rotation.

## Solution Assessment

Unclear until maintainers review the TLS trust boundary in #75228. The proposed direction is narrow and matches the app-vs-CLI difference if `systemTrustOk` is accepted as the repair gate; otherwise the safer path is to keep the current fail-closed policy and document a manual pin reset or SSH workaround.
