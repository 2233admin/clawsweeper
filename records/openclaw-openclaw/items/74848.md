## Summary

Keep open. Current main has a partial stale-TLS-pin repair after the report, but it is limited to loopback and Tailscale Serve names; the reported custom `wss://...xyz` direct gateway can still plausibly leave the macOS app node disconnected while the control channel and CLI node work.

## What This Changes

The issue requests fixing macOS app remote-direct node WebSocket disconnects that log `connect failed ... cancelled` even though the app control channel and `openclaw node run --tls` work against the same gateway.

## Best Possible Solution

Diagnose the native macOS app-node WSS path for custom remote gateways, then add a maintainer-approved certificate review/clear-pin or `gateway.remote.tlsFingerprint` flow that restores node connectivity without weakening first-use or rotated-certificate trust rules.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The logs and source give a strong code-level path via a stored custom-domain WSS pin mismatch, but a deterministic repro still needs the reporter's stored pin/certificate state or a macOS direct-WSS setup with a rotated non-`*.ts.net` certificate.

## Solution Assessment

Is this the best way to solve the issue? Unclear. Current main's Tailscale-only auto-repair is a cautious partial fix, but the reported custom-host case needs explicit maintainer policy for certificate rotation, user review, and documented fingerprint configuration.
