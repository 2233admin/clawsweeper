## Summary

Keep open. The report is recent and current main still has a macOS-native node-only WSS session path that can plausibly produce the reported `connect failed ... cancelled` while the app control channel and CLI node continue to work; I did not find a merged fix or exact canonical issue for this user-visible regression.

## What This Changes

The issue reports that in remote direct WSS mode the macOS app native node repeatedly disconnects with `cancelled` even though the app control channel and `openclaw node run --tls` connect successfully in the same environment.

## Best Possible Solution

Diagnose and fix the native macOS app node WSS trust/reconnect path so it follows the documented remote TLS fingerprint and trust behavior, gives actionable stale-pin remediation when a stored fingerprint no longer matches, and keeps the control-channel and CLI node behavior consistent without weakening gateway trust.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The logs plus source give a strong code-level hypothesis, especially a stale stored WSS fingerprint mismatch, but a deterministic repro still needs the reporter's macOS pin state or a local macOS direct-WSS setup with a rotated certificate.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The narrow implementation likely belongs in `GatewayTLSPinningSession`, `MacNodeModeCoordinator.buildSessionBox`, and related remote TLS config plumbing, but maintainers should first choose the expected first-enrollment and stale-fingerprint policy because the same surface is security-sensitive.
