## Summary

Keep open. Current main still limits macOS native-node stale TLS-pin auto-repair to loopback and `*.ts.net`, while the reported public WSS host remains a plausible failure path; the linked fixing PR is open and unmerged, so this issue should not be closed ahead of that review.

## What This Changes

The issue requests a fix for macOS app remote mode where the native app node repeatedly disconnects with `cancelled` on a remote WSS gateway while `openclaw node run --tls` works against the same gateway.

## Best Possible Solution

Review and land PR #75228 or an equivalent macOS fix that repairs stale pins for system-trusted public WSS certificate rotations, preserves strict behavior for untrusted/self-signed/private-CA gateways, and includes focused Swift coverage plus a macOS remote WSS smoke comparison against the CLI node path.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No for an end-to-end run in this checkout: deterministic reproduction needs a macOS app node with a stale stored TLS pin and a system-trusted public WSS certificate rotation. The current tests do give a high-confidence unit-level reproduction of the policy gap because public `gateway.example.com` remains non-repairing even when `systemTrustOk` is true.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers review the trust boundary in PR #75228. A narrow macOS-only repair gated on `pinMismatch`, matching store key, WSS, and `systemTrustOk` appears directionally maintainable if untrusted certificate paths stay fail-closed.
