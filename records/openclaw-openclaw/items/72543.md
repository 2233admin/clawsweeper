## Summary

Keep this issue open: current main still normalizes some `main` aliases, but the shared Control UI session option builder does not synthesize or pin the canonical main session when it is absent from `sessions.list` and the user is viewing another session.

## What This Changes

The issue requests that the Control UI chat session dropdown always include and normalize a stable canonical main-session entry for `agent:main:main`.

## Best Possible Solution

Add a shared Control UI helper that derives the canonical main key from `hello.snapshot.sessionDefaults.mainSessionKey`, prepends exactly one stable main-session option before discovered sessions, deduplicates any returned row for that key, and reuses the same normalization for current and persisted session keys.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resolveSessionOptionGroups` with an active non-main `sessionKey` and a `sessionsResult` that lacks `agent:main:main`; current code returns the supplied rows plus the active session only, so the canonical main option is absent.

## Solution Assessment

Yes. The narrow maintainable fix is in the shared Control UI option builder and defaults normalization, using gateway `sessionDefaults` rather than duplicating desktop/mobile dropdown logic or changing session-store semantics.
