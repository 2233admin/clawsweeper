## Summary

Keep this issue open. Current main still normalizes main aliases in settings, but the shared Control UI dropdown builder only adds sessions returned by `sessions.list` plus the active key, so the canonical main session can still be absent when the active session is a different key.

## What This Changes

The issue requests a stable pinned Control UI chat dropdown entry for the canonical main assistant session and consistent normalization of `main` aliases to that canonical key.

## Best Possible Solution

Add a shared Control UI normalization and option-building path that derives the canonical main key from `hello.snapshot.sessionDefaults.mainSessionKey`, prepends a stable main-session option before discovered sessions, keeps desktop and mobile selects aligned, and adds focused tests for the non-main active-session case.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resolveSessionOptionGroups` with active session `agent:main:subagent:x` and a `sessionsResult` that lacks `agent:main:main`; current code returns only the supplied rows plus the active key, so the canonical main option is absent.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if it is driven from Gateway `sessionDefaults` instead of a literal persona string, and the safest implementation is to inject through the shared option-group builder.
