## Summary

Keep this issue open. Current main normalizes `main` aliases to the Gateway-provided main session key, but the shared Control UI session dropdown builder still only renders `sessions.list` rows plus the active session key, so it can omit `agent:main:main` whenever the user is viewing another session and the store-backed list lacks the canonical main row.

## What This Changes

The issue requests that the Control UI chat session dropdown always include a stable canonical main-session option for `agent:main:main` and normalize `main` aliases to that key across selected and persisted settings.

## Best Possible Solution

Add the canonical main session through the shared Control UI option-group path, using `hello.snapshot.sessionDefaults.mainSessionKey` and agent identity metadata where available, so desktop and mobile chat dropdowns show the same stable main target without duplicating logic or changing Gateway session storage semantics.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resolveSessionOptionGroups` with an active non-main `sessionKey` and a `sessionsResult` that does not include `agent:main:main`; current main returns the supplied rows plus that active session, leaving the canonical main option absent.

## Solution Assessment

Yes. The narrow maintainable fix is to inject and label the canonical main option in the shared dropdown option builder and reuse the existing Gateway `sessionDefaults` normalization, rather than hard-coding a bundled identity or changing `sessions.list` storage behavior.
