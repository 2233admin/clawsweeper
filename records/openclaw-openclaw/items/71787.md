## Summary

Keep this PR open. Current main still accepts `context.open_chat_id` only when a flat operator open ID is present, while the Schema 2.0 nested `operator.user_id.open_id` path described in the linked bug still drops before dispatch. The PR is a focused parser/test/changelog fix, maintainer discussion already identified it as the better base than #71692, and I found no blocking review finding.

## What This Changes

The PR updates Feishu card-action callback normalization to read nested Schema 2.0 identity fields, adds lifecycle coverage for that callback shape, and expands the Feishu changelog attribution.

## Best Possible Solution

Land the narrow Feishu parser, lifecycle test, and changelog update after normal maintainer validation, while keeping instant card-update behavior from #71677 and unrelated install-stage security concerns out of this PR.

## Reproduction Assessment

Yes. A `card.action.trigger` payload whose only operator open ID is `operator.user_id.open_id` reproduces the current-main parser failure because `readString(operator.user_id)` returns undefined for the object and the required `openId` guard returns null before dispatch.

## Solution Assessment

Yes. Normalizing the Feishu callback shape at `parseFeishuCardActionEventPayload` is the narrowest maintainable fix because downstream card-action handling already expects a normalized `FeishuCardActionEvent`, and the PR keeps the existing token/openId/tag/action-value validation.
