## Summary

Keep this PR open. Current main still handles only the narrower Feishu `context.open_chat_id` fallback and still drops callbacks whose only operator identity is nested under `operator.user_id`, while the current PR head addresses that remaining gap and the prior dead-guard/changelog feedback appears resolved.

## What This Changes

This PR updates the Feishu card-action callback parser and lifecycle coverage to accept Schema 2.0 nested operator identity fields, and updates the Feishu changelog attribution.

## Best Possible Solution

Land this narrow Feishu parser, lifecycle test, and changelog fix after normal maintainer validation, while keeping unrelated instant-card-update behavior and install-stage security concerns out of this PR.

## Reproduction Assessment

Yes. A `card.action.trigger` payload whose only operator open ID is `operator.user_id.open_id` reproduces the current-main parser failure by leaving `openId` undefined, which causes the handler to log the malformed-payload message and return before dispatch.

## Solution Assessment

Yes. Normalizing the Feishu callback shape at the monitor parser boundary is the narrowest maintainable fix, because downstream card-action handling already expects a normalized `FeishuCardActionEvent` and the PR preserves the existing token/openId/tag/action-value validation.
