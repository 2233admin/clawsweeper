## Summary

Keep this PR open. Current main still does not satisfy the documented default-agent behavior for `models status`, the linked doctor false-positive bug is valid, and the latest branch targets the right command, probe, and doctor surfaces without an obvious introduced correctness bug. It still needs maintainer review because it intentionally changes which local auth/model store is inspected, exact-head CI is not fully green, and the PR is not yet mergeable.

## What This Changes

The branch centralizes `models status` agent-scope resolution, threads the resolved agent id and directory into auth probes, exempts the live compatibility agent directory from doctor orphan warnings, and adds command, doctor, and deadcode-parser regression coverage.

## Best Possible Solution

Land this PR, or a maintainer-equivalent patch, after review and green validation so `models status`, `models status --probe`, and doctor all follow one documented precedence: explicit `--agent`, then compatibility env override, then configured default agent, while doctor suppresses only the live compatibility store and continues warning on real orphan agent directories.

## Reproduction Assessment

Yes. The linked bug can be reproduced by configuring a non-main default agent, leaving `agents/main/agent` on disk, and running `openclaw models status`, `openclaw models status --probe`, and `openclaw doctor`; current main source and docs show the mismatch without needing a live provider call.

## Solution Assessment

Yes. The proposed shape is the narrow maintainable fix: centralize status scope resolution, pass that scope through the probe path, and add a targeted doctor exemption rather than changing unrelated `resolveOpenClawAgentDir()` call sites. Final acceptance still depends on maintainer review of the auth-store compatibility behavior and green CI.
