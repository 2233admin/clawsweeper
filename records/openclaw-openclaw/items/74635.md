## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and open implementation PRs already reference this issue, so the right maintainer cleanup is to review and consolidate the canonical protocol fix rather than close the issue early.

## What This Changes

The issue asks OpenClaw Gateway to stop rejecting Paperclip agent invocation payloads that include a root-level `paperclip` metadata field.

## Best Possible Solution

Land one additive Gateway protocol compatibility change through a canonical PR, with focused validator coverage, generated protocol artifacts, and changelog/docs follow-through if the wire contract changes; close duplicate PRs after that fix merges.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request with `message`, `idempotencyKey`, and a root-level `paperclip` object follows the report and will fail current `validateAgentParams` because the closed schema omits that field.

## Solution Assessment

Unclear. The reporter's optional `paperclip: Type.Unknown()` field is the narrowest compatibility shim, but the best long-term protocol shape may be a generic opaque adapter metadata namespace if maintainers want to avoid encoding an external integration name in the core schema.
