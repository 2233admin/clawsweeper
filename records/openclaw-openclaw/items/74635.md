## Summary

Keep open. The current gateway protocol still rejects a root-level `paperclip` property on `agent` requests, and an open implementation PR already targets this exact issue with closing syntax, so the issue should remain open until that PR is reviewed and either merged or replaced.

## What This Changes

The issue reports that Paperclip’s `openclaw_gateway` adapter sends a root-level `paperclip` field in agent invocation payloads, which current gateway validation rejects as an unexpected property.

## Best Possible Solution

Review the open target-repo PR that adds additive protocol tolerance and regression tests, then either merge that narrow fix or replace it with a maintainer-approved generic metadata field that keeps strict validation for unrelated unknown properties.

## Reproduction Assessment

Yes. A minimal `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` property follows the current validation path: `AgentParamsSchema` has no such field, `additionalProperties` is false, and the handler returns `INVALID_REQUEST`.

## Solution Assessment

Yes for keeping this issue open. The narrow additive schema/test fix proposed by the linked open PR is the smallest compatible repair, though maintainers may prefer a generic opaque metadata namespace before merging.
