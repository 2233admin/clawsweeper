## Summary

Keep open. Current main still rejects the reported root-level `paperclip` metadata on Gateway `agent` requests, and open PR #74974 explicitly links this issue as implementation work, so the issue should remain open until a canonical protocol fix lands or that PR is closed.

## What This Changes

Paperclip `openclaw_gateway` agent invocations need their root-level run metadata accepted by OpenClaw Gateway `agent` params or moved to a documented supported metadata path.

## Best Possible Solution

Land one maintainer-approved additive Gateway protocol compatibility change for adapter metadata in `agent` params, preserve strict rejection for unrelated root typos, add focused regression coverage, update generated protocol models and release notes/docs as needed, then close the linked Paperclip reports after merge.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the report and will fail current `validateAgentParams` because the schema is closed and omits that field.

## Solution Assessment

Unclear. The requested `paperclip: Type.Optional(Type.Unknown())` shim is the narrowest immediate compatibility fix, but a generic adapter metadata field may be the more maintainable protocol surface unless maintainers explicitly approve a vendor-named exception.
