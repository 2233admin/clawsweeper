## Summary

Keep open. Current main and the latest shipped release still reject a root-level `paperclip` field on gateway `agent` requests, and an open PR already targets this exact issue, so the issue should remain open pending review or merge of that PR.

## What This Changes

Allow Paperclip's root-level `paperclip` metadata on gateway `agent` request payloads, or route that context through an existing supported field, so Paperclip invocations are not rejected by strict validation.

## Best Possible Solution

Land #74974 or an equivalent additive protocol change that lets Paperclip metadata pass through `agent` validation without weakening unrelated strict validation, with regression coverage and generated client artifacts kept in sync. If maintainers prefer a generic metadata namespace, coordinate that as the accepted gateway contract before closing the issue.

## Reproduction Assessment

Yes. A current-main `agent` request containing `message`, `idempotencyKey`, and a root `paperclip` property follows the validated handler path and fails because `AgentParamsSchema` lacks that property while `additionalProperties` is false.

## Solution Assessment

Unclear. The reported `paperclip: Type.Optional(Type.Unknown())` approach is narrow and matches the existing wake-payload tolerance pattern, but maintainers may prefer a generic metadata field or `inputProvenance`-based contract to avoid baking a service name into the gateway protocol.
