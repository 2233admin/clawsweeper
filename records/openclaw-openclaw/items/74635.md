## Summary

Keep open. Current main still rejects root-level `paperclip` metadata on Gateway `agent` requests, and an open implementation PR references this issue with closing syntax, so the issue should remain open until the canonical protocol fix merges or that PR is closed.

## What This Changes

The issue asks OpenClaw Gateway agent invocation validation to tolerate Paperclip’s root-level metadata field instead of rejecting the payload as an unknown property.

## Best Possible Solution

Land one additive Gateway protocol compatibility change for adapter metadata in `agent` params, keep strict rejection for unrelated root fields, add focused regression coverage, regenerate protocol artifacts, update docs/changelog if the wire contract changes, and close this issue only after the accepted fix merges.

## Reproduction Assessment

Yes. A minimal Gateway `agent` request with valid `message` and `idempotencyKey` plus a root-level `paperclip` object follows the reported path into `validateAgentParams` and fails because the schema is closed and validation runs before dispatch.

## Solution Assessment

Unclear. The requested `paperclip: Type.Optional(Type.Unknown())` shim is the narrowest immediate compatibility fix, but a generic opaque adapter metadata field may better match the core-extension boundary unless maintainers explicitly approve a Paperclip-named protocol exception.
