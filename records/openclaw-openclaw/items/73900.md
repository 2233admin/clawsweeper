## Summary

Keep open. Current main still matches the reported routing leak, and the provided GitHub context shows open PR #73968 with closing syntax for this issue, so the issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

A main-session cron `systemEvent` heartbeat wake should route replies to the bound Telegram topic session instead of inheriting a global `heartbeat.to` DM target.

## Best Possible Solution

Ship a narrowly scoped fix that prevents cron-forced `target: "last"` wakes from inheriting explicit destination fields such as `to` and `accountId`, while preserving normal explicit heartbeat targeting and adding a regression for a Telegram forum-topic session with global `heartbeat.to` configured.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence static reproduction path: configure global `agents.defaults.heartbeat.to`, run a `sessionTarget: "main"` systemEvent cron job bound to a topic session, and the cron heartbeat override is shallow-merged before outbound delivery treats the inherited `to` as explicit. I did not run a live Telegram reproduction in this read-only review.

## Solution Assessment

Yes. Clearing or non-inheriting explicit destination fields for cron-supplied `target: "last"` is the narrowest maintainable direction; reviewing and landing #73968 or an equivalent patch is safer than changing all heartbeat `last` routing semantics.
