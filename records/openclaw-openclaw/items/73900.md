## Summary

Keep open. Current main still has the cron main-session path that forces `heartbeat.target = "last"` without clearing inherited destination fields, and the gateway adapter still shallow-merges that override over configured heartbeat defaults before outbound delivery treats `heartbeat.to` as explicit. The issue is also paired with open PR #73968 using closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that main-session cron `systemEvent` heartbeat wakes route replies back to the bound session/topic instead of inheriting a global heartbeat DM destination.

## Best Possible Solution

Land #73968 or an equivalent narrow fix that clears inherited explicit destination fields for cron-forced `target: "last"` main-session `systemEvent` wakes, with regression coverage proving a topic-bound session stays on its group/topic route even when global heartbeat defaults include a DM `to`.

## Reproduction Assessment

Yes. The issue provides concrete config, cron payload, session key, and observed logs, and current main statically reproduces the merge and delivery-resolution path without needing a live Telegram run.

## Solution Assessment

Yes. Clearing inherited explicit destination fields only for cron-forced `target: "last"` wakes is the narrowest maintainable direction; changing general heartbeat `target: "last"` semantics would be broader and riskier.
