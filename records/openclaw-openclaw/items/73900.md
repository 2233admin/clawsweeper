## Summary

Keep open. Current main still preserves an inherited global `heartbeat.to` when cron forces `heartbeat.target="last"`, so a topic-bound main-session `systemEvent` can still resolve delivery through the configured DM destination. The provided context also shows open PR #73968 uses closing syntax for this issue, so the issue should remain open until that PR is landed, replaced, or closed.

## What This Changes

Prevent main-session cron `systemEvent` heartbeat wakes that force `target: "last"` from inheriting global heartbeat destination fields and misrouting topic-bound reminder replies to DMs.

## Best Possible Solution

Land #73968 or an equivalent narrow fix that treats cron-forced `target: "last"` as session-bound routing by clearing inherited explicit destination fields, with regression coverage for a Telegram forum topic session and a global DM heartbeat default.

## Reproduction Assessment

Yes. The issue provides concrete config, cron payload, session key, and logs, and current main statically reproduces the route: cron passes only `target: "last"`, the gateway shallow-merges default `to`, and outbound resolution prefers that explicit destination over the session route.

## Solution Assessment

Yes. Clearing inherited explicit destination fields for cron-forced `target: "last"` wakes is the narrowest maintainable solution; changing all heartbeat `target: "last"` semantics would be broader and riskier.
