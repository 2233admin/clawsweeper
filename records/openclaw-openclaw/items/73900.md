## Summary

Keep open. Current main still matches the reported root cause: main-session cron systemEvent wakes force `heartbeat.target = "last"`, the Gateway cron adapter shallow-merges that override with configured heartbeat defaults, and outbound heartbeat routing treats an inherited `heartbeat.to` as an explicit destination before session-bound `last` routing. The report is concrete, recent, and privacy-sensitive, and I found no merged fix or open PR with closing syntax for this issue.

## Best Possible Solution

Keep this issue open and fix the main-session cron heartbeat handoff so cron-forced `target: "last"` does not inherit global explicit destination fields like `to` from heartbeat defaults. Add a focused regression covering a Telegram/forum-topic session with `agents.defaults.heartbeat.to` set to a DM and verify the reply route stays on the bound session/topic.

## Risks / Open Questions

- The reported behavior is a privacy/cross-context routing leak: a topic-bound reminder can be delivered to a DM when global heartbeat defaults include `to`.
- Changing generic heartbeat `target: "last"` semantics could affect explicit heartbeat configurations, so the fix should be scoped to cron/systemEvent override inheritance or explicitly distinguish inherited defaults from same-call explicit overrides.
- Adjacent open work around main-session systemEvent safety (#48883) and explicit heartbeat targeting (#41031) may overlap with the final design, so maintainer review should coordinate the route policy before implementation.
