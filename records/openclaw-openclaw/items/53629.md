## Summary

Keep this PR open. Current main still lacks Telegram Business handlers and outbound reply support, so the feature request is valid, but the proposed patch has concrete correctness and account-isolation defects that should block merge until a maintainer decides whether to salvage this branch or request a replacement implementation.

## What This Changes

The PR adds Telegram Business allowed update names, Business update context fields, business_connection and business_message handlers with an in-memory connection map and outbound API transformer, plus focused handler tests.

## Best Possible Solution

Ship a refreshed Telegram plugin implementation that handles Business connect and disconnect state per account, routes business_message through the existing DM authorization and inbound pipeline, updates dedupe and sequential keys for Business payloads, injects business_connection_id only for supported outbound methods using the correct user_chat_id mapping, and documents Telegram Business setup plus access-control behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Avoid pre-consuming the Business message dedupe key:** `extensions/telegram/src/bot-handlers.runtime.ts:1765-1767`
  - body: `handleInboundMessageLike` already calls `shouldSkipUpdate` for the same context. In a real bot run, the update-tracker middleware has registered the active update key before this handler runs, so this first call marks the key handled and the second call inside `handleInboundMessageLike` skips the Business message before it is processed.
  - confidence: 0.9
- **[P1] Scope Business connection mappings per Telegram account:** `extensions/telegram/src/bot-handlers.runtime.ts:97`
  - body: `businessConnectionMap` is shared at module scope, but OpenClaw supports multiple Telegram accounts. A Business message on one account can overwrite the same numeric chat ID for another account, causing the API transformer to inject the wrong `business_connection_id`.
  - confidence: 0.88
- **[P1] Remove Business mappings on disconnect:** `extensions/telegram/src/bot-handlers.runtime.ts:1744`
  - body: Business connection updates also report disabled connections via `is_enabled: false`, but this handler always stores the mapping. After a user disconnects the bot, outbound requests can keep using a stale `business_connection_id`.
  - confidence: 0.87
- **[P2] Use user_chat_id for connection routing:** `extensions/telegram/src/bot-handlers.runtime.ts:1740`
  - body: `BusinessConnection` exposes `user_chat_id` as the private chat identifier for the connection. Mapping `user.id` instead can route replies to the wrong target if those identifiers diverge, and it does not match the Telegram Business contract.
  - confidence: 0.78
- **[P2] Wire Business payloads into update keys:** `extensions/telegram/src/bot-updates.ts:24-34`
  - body: The patch adds Business fields to the context type, but `buildTelegramUpdateKey` still never reads `businessMessage` or `update.business_message`, and `sequential-key.ts` is not updated. Business updates without a usable `update_id` still miss semantic dedupe and can serialize on the generic `telegram:unknown` lane.
  - confidence: 0.82
- **[P3] Do not exclude Business chat actions:** `extensions/telegram/src/bot-handlers.runtime.ts:141`
  - body: The transformer skips `sendChatAction`, but the locked @grammyjs/types raw API includes `business_connection_id` for that method, and OpenClaw uses chat actions for typing state before replies. Business chats should receive the same account-scoped parameter for supported typing actions.
  - confidence: 0.73

## Security Review

Status: needs_attention

Summary: No dependency, workflow, or package metadata changes were found, but the patch introduces a security-sensitive account-isolation risk in Telegram Business outbound API mutation.

Concerns:

- **[medium] Shared Business map crosses Telegram account boundaries:** `extensions/telegram/src/bot-handlers.runtime.ts:97`
  - body: The module-scoped `businessConnectionMap` is shared by every Telegram account instance, while the API transformer mutates outbound payloads from that map. A Business connection observed on one account can affect outbound calls for another configured Telegram account using the same numeric chat ID.
  - confidence: 0.86
- **[low] Disconnected Business chats retain outbound authority:** `extensions/telegram/src/bot-handlers.runtime.ts:1744`
  - body: The handler ignores `is_enabled: false`, so stale Business connection IDs remain available after a disconnect and can keep being injected into outbound Telegram API calls.
  - confidence: 0.82
