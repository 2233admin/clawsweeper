## Summary

Keep this PR open. Current main has only partial Telegram Business groundwork in allowed-update fallback constants; it still lacks business_connection/business_message handlers, business update dedupe fields, outbound business_connection_id routing, docs, and business-specific tests. The PR is a useful implementation candidate for open issue #20786, but the existing P1 review comments and current dedupe semantics mean it needs a maintainer-guided refresh before merge.

## Best Possible Solution

Keep the PR open and ask for a current-main refresh or maintainer rewrite: add account-scoped Telegram Business connection tracking with disconnect cleanup, register business_message/business_connection handlers before the normal message handler, route business messages through the existing DM authorization and inbound pipeline without double-dedupe skipping, inject business_connection_id only for supported outbound methods, and add focused tests plus Telegram docs/changelog coverage.
