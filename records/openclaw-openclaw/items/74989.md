## Summary

Keep open. Current main still has the undefined-result crash path, and this PR is a focused implementation for the linked Telegram/plugin command regression; the remaining decision is maintainer selection among overlapping open fixes and whether to formalize void command handlers in the SDK contract.

## What This Changes

The PR normalizes nullish or non-object plugin command results to empty reply objects in the shared executor, generic command handler, and Telegram native command handler, with regression tests for those paths.

## Best Possible Solution

Land one canonical narrow fix that normalizes plugin command results at the shared executor boundary, keeps generic and Telegram native dispatch resilient to direct or mocked executor paths, and then close or supersede the overlapping PRs. If no-payload command handlers are supported behavior, align the plugin SDK type/docs with that decision.

## Reproduction Assessment

Yes. On current main, a plugin command handler resolving `undefined` can escape `executePluginCommand`, after which `handlePluginCommand` reads `result.continueAgent`; the linked v2026.4.27 Telegram report provides matching runtime logs.

## Solution Assessment

Yes, mostly. Normalizing at the shared executor boundary is the narrowest maintainable fix, and the caller-side guards protect direct or mocked runtime paths; the remaining maintainer decision is duplicate PR selection and SDK contract wording.
