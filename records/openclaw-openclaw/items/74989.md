## What This Changes

The PR normalizes nullish and non-object plugin command results in the shared command executor and Telegram native dispatch paths, with regression tests for shared, generic reply, and Telegram native command handling.

## Best Possible Solution

Land one canonical narrow fix that normalizes plugin command results at the shared executor boundary and keeps Telegram native and generic command dispatch resilient. If intentionally void command handlers are now supported API, align the SDK type/docs in the same PR or a focused follow-up.

## Reproduction Assessment

## Solution Assessment

Yes, mostly. Normalizing at `executePluginCommand` is the narrow shared fix, and the caller-side guards are reasonable defense for direct or mocked runtime paths; maintainers should decide whether the public SDK contract should also explicitly allow void/no-op command handlers.
