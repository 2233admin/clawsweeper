## Summary

Keep open. The PR addresses a real Telegram/plugin-command regression that is still present on current main, and the patch is a narrow implementation with focused regression coverage and green GitHub checks. Because #74838 and #74856 overlap the same #74800 fix, the next step is maintainer canonical-PR selection rather than closing this PR or routing an automated replacement.

## What This Changes

The PR normalizes nullish or non-object plugin command results at the shared executor and defensive dispatch paths, with tests for the executor, generic command handling, and Telegram native command handling.

## Best Possible Solution

Land one canonical narrow fix that normalizes plugin command handler output at executePluginCommand, keeps generic and Telegram dispatch robust against direct or mocked undefined results, then close or supersede the overlapping PRs and #74800. If no-payload handlers are intended supported behavior, follow with a small SDK contract/docs update.

## Reproduction Assessment

Yes. A plugin command handler that resolves undefined on current main can flow through executePluginCommand and crash when handlePluginCommand reads result.continueAgent; #74800 provides matching v2026.4.27 Telegram logs.

## Solution Assessment

Yes, mostly. Normalizing at the shared executor boundary is the narrowest maintainable fix, and the caller-side guards are reasonable defense for direct or mocked runtime paths; the remaining decision is canonical PR selection and possible contract wording follow-up.
