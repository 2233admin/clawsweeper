## Summary

Keep open. Current main still has the strict Control UI sessionKey gate that causes the linked live-chat drop, but the PR's runId-only fallback broadens event acceptance without proving the active run is still bound to the session that launched it.

## What This Changes

The PR changes Control UI chat event handling to accept active-run chat events by runId when sessionKey differs, adds delta/final regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the focused Control UI fix, but bind the runId fallback to the session that initiated the run or guarantee all session-change/history paths clear active run state before accepting mismatched-session events; then land it with the linked bug closing through the merge.

## Reproduction Assessment

Yes. A focused unit reproduction is state.sessionKey = "main" with state.chatRunId = "run-1" and a chat event using sessionKey "agent:main:main" plus runId "run-1"; current main returns null, while the PR tests exercise delta and final delivery.

## Solution Assessment

No, not quite yet. The runId fallback is the right narrow direction for the reported raw/canonical mismatch, but it should be session-bound or paired with complete active-run clearing so it cannot deliver an old run into a newly selected session.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Bind runId fallback to the launching session:** `ui/src/ui/controllers/chat.ts:724-727`
  - body: This accepts a chat event solely because its runId matches state.chatRunId, even when the event sessionKey differs from the current UI session. Normal switchChatSession clears chatRunId, but onPopState/applySessionSelection can change host.sessionKey directly and refreshChat/loadChatHistory clears stream state without clearing chatRunId, so an in-flight run from the previous session can be appended to the newly selected session. Track the session key captured when the run starts, or clear chatRunId on every session mutation path, before using runId as a cross-session unlock.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff expands live chat event acceptance from sessionKey equality to runId-only matching without proving the runId is still tied to the session that launched it.

Concerns:

- **[medium] RunId fallback is not session-bound:** `ui/src/ui/controllers/chat.ts:724`
  - body: A stale active chatRunId can survive direct sessionKey changes, and the new fallback would then accept a mismatched-session chat event into the current UI session if the old run completes later. That is a concrete cross-session content exposure risk in the Control UI event boundary.
  - confidence: 0.82
