## Summary

Keep open. Current main still resolves a single Telegram route for the receiving account, drops non-default named-account group fallback without an explicit binding, and has no Telegram account-forwarding or broadcast-to-account config surface. The existing stale label should not drive closure because the report is concrete, current behavior still matches the gap, and the remaining change needs maintainer product/security design.

## What This Changes

The issue requests opt-in Telegram multi-account routing so a group message received by one bot account can be forwarded or broadcast to other configured bot accounts while preserving each bot identity.

## Best Possible Solution

Design an additive, opt-in Telegram cross-account or broadcast routing model with loop prevention, per-account identity preservation, explicit allowlists, bot-authored message handling, docs, and focused multi-account group coverage; if maintainers decide not to support this pattern, document the supported alternative clearly.
