## Summary

Keep open. Current main still has the hardcoded media-store and sandbox-staging cap, so the user problem is real and not implemented, but the submitted PR head is stale and currently incorrect against the moved media/gateway surfaces.

## What This Changes

The PR adds a `media.maxBytes` config field and attempts to use it for inbound media storage, media serving, sandbox staging, schema types, labels, and help text while preserving the 5 MiB default.

## Best Possible Solution

Ship a refreshed, bounded `media.maxBytes` contract that is generated into schema/help metadata, initialized during current gateway startup, applied consistently to store defaults, sandbox staging, and chat sandbox preflight, and preserves intentional explicit per-caller or per-channel caps where those are meant to be independent.
