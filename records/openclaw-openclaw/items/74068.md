## Summary

Keep this PR open. Current main still lacks gateway-boundary validation for syntactically invalid cron schedules: `cron.add` and `cron.update` validate the request envelope, timestamp, and delivery shape, then call the cron service directly, where Croner parse errors can escape to the WebSocket catch-all as `UNAVAILABLE`. The linked bug #74066 remains open, and related PR #74101 was closed unmerged, so neither current main nor another landed PR supersedes this work. The PR diff is focused on the cron gateway handler and its validation test, with no security-sensitive workflow, dependency, lockfile, publishing, permission, or secrets changes visible in the reviewed diff.

## Best Possible Solution

Review and land this PR or an equivalent gateway-boundary fix. The fix should validate cron schedules for `cron.add` and `cron.update` before service mutation, defensively map Croner schedule parse failures to `INVALID_REQUEST`, preserve valid `cron`/`every`/`at` semantics, keep non-Croner service failures on the existing catch-all path, and retain focused coverage in `src/gateway/server-methods/cron.validation.test.ts`. After merge, linked issue #74066 can close through the PR.
