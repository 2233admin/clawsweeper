## Summary

Keep open for maintainer review. Current main still applies the one-shot `cron.retry.backoffMs` setting during recurring-job recompute after reload, while live recurring error scheduling and docs use the fixed recurring backoff schedule; this draft PR is a narrow, plausible service/test fix with no blocking review finding found in the provided diff.

## What This Changes

The PR changes cron service recompute logic so recurring errored jobs use the fixed recurring error backoff instead of `cron.retry.backoffMs`, and adds a regression test for a recurring errored job with custom one-shot retry config.

## Best Possible Solution

Land the scoped service/test change once the draft is ready and targeted validation is green, while keeping the broader recurring transient-retry policy discussion in its existing related issue/PR lane.

## Reproduction Assessment

Yes. Construct an enabled recurring job with `lastStatus: "error"`, a finite `lastRunAtMs`, and `cronConfig.retry.backoffMs` set to a long one-shot delay, then call `recomputeNextRuns`; current main uses the configured one-shot delay for the recurring backoff floor.

## Solution Assessment

Yes for the scoped reload/recompute bug. Removing the custom retry schedule from the recurring recompute path aligns it with live recurring scheduling, the pending-backoff check, and the documented one-shot-only `cron.retry` contract.
