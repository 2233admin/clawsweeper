## Summary

Keep open. Current main still suppresses `commentary`-phase assistant output before it can reach the embedded delivery callbacks that Discord uses for draft preview, block reply, partial reply, or final reply delivery. The related closed main-result latency report does not cover this separate early-commentary commit/durability request.

## What This Changes

Discord should commit early user-visible assistant commentary as its own visible message before long-running or restart-sensitive work, so generated status text is not delayed or lost on gateway restart.

## Best Possible Solution

Define a scoped delivery contract for approved user-visible status commentary, then route that contract through the existing embedded subscribe and Discord draft/dispatcher/restart-sensitive delivery paths with regression coverage. Raw `commentary` should remain suppressed unless maintainers explicitly decide it is safe to surface.
