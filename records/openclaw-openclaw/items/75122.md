## Summary

Keep open. Current main partially addresses the report with WhatsApp socket timing config, a generic outbound delivery queue/reconnect drain, and pruning for old unknown runtime-deps roots, but the central event-loop-blocking investigation is still not implemented: liveness warnings remain aggregate-only and do not attribute a blocking async resource or call path, auto-reply delivery still uses the direct WhatsApp reply path, and versioned runtime-deps accumulation is not fully covered.

## What This Changes

The issue requests investigation and fixes for WhatsApp 408 disconnects tied to long Gateway event-loop stalls, including blocker attribution, reconnect-safe outbound reply replay, exposed Baileys timing options, and runtime-deps garbage collection.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is source-attributed liveness warnings, removal or offloading of the blocking synchronous path, reconnect-safe outbound delivery semantics for WhatsApp replies, and explicit cleanup policy for stale versioned runtime-deps roots.
