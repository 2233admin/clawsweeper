## Summary

Keep open. Current main has liveness warnings, WhatsApp socket timing config, and startup pruning for stale runtime-deps roots, but it still does not attribute liveness warnings to a blocking async resource or guarantee outbound reply replay after a disconnect. The reported long event-loop stalls remain a real reliability investigation, not an implemented or clearly superseded item.

## What This Changes

The issue reports WhatsApp 408 disconnects caused by long Node event-loop stalls and asks to identify/log the blocking source, replay queued replies after reconnect, expose Baileys timing options, and garbage-collect stale plugin runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is to add low-noise source attribution for liveness warnings, identify and remove the blocking synchronous path, and design an idempotent reconnect-safe outbound delivery path while preserving the existing socket-timing and runtime-deps cleanup improvements.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness logs and deterministic 408 timing, but current repository tests do not provide a self-contained reproduction for the 33-100s event-loop blocks or dropped post-disconnect outbound replies.

## Solution Assessment

No automated close or narrow fix is the best path yet. Current main partially mitigates the report, but the remaining work needs maintainer-led performance tracing and delivery semantics review before a focused repair PR is safe.
