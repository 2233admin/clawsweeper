## Summary

Keep open. Current main partially addresses the report with WhatsApp socket timing config, generic outbound write-ahead recovery, and reconnect draining, but the central event-loop-blocking investigation is still unresolved: liveness warnings remain aggregate-only, automatic WhatsApp source replies are not durably queued, and stale versioned runtime-deps roots are still retained.

## What This Changes

The issue asks OpenClaw to identify event-loop blocking behind WhatsApp 408 disconnects, attribute liveness warnings to blocking async resources, replay dropped outbound replies after reconnect, expose Baileys timing options, and garbage-collect stale plugin runtime-deps trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is low-noise attribution for liveness stalls, removal of the synchronous blocker, idempotent reconnect-safe delivery for automatic WhatsApp replies, and safe cleanup of stale versioned runtime-deps roots while preserving the already-landed timing and generic delivery-queue improvements.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness logs and deterministic 408 timing, and current tests cover several partial mitigations, but the repository does not contain a self-contained reproduction for the reported 33-100s event-loop blocks or an exact dropped automatic reply after a 408 close.

## Solution Assessment

No, closing or handing this to a narrow automated repair is not the best path yet. Current main solves the timing and generic outbound-queue pieces, but the remaining work needs maintainer-led performance tracing and delivery-semantics scoping before a focused PR is safe.
