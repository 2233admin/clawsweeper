## Summary

Keep open. Current main still only partially addresses the report: WhatsApp socket timing, reconnect-aware send retries, a generic delivery queue, and unknown-root runtime-deps pruning exist, but liveness warnings still lack blocker attribution, WhatsApp auto-replies still use the direct delivery path, and versioned runtime-deps accumulation is not covered.

## What This Changes

The issue requests fixes for WhatsApp 408 disconnects tied to long Gateway event-loop stalls, including source attribution, reconnect-safe reply delivery, Baileys timing controls, and stale runtime-deps garbage collection.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The end state should add low-noise blocker attribution to liveness warnings, remove or offload the confirmed synchronous path, make WhatsApp reply delivery reconnect-safe with clear idempotency semantics, and define a safe retention policy for stale versioned runtime-deps roots.
