## Summary

Keep open. Current main partially addresses the report with WhatsApp socket timing controls, generic queued outbound delivery plus reconnect drain plumbing, and some runtime-deps cleanup/loop mitigations, but it still does not attribute liveness warnings to the blocking async resource, does not make the direct WhatsApp auto-reply path durable across reconnect, and still preserves stale versioned runtime-deps roots.

## What This Changes

The issue requests fixes for WhatsApp 408 disconnects caused by long Gateway event-loop stalls, including liveness source attribution, reconnect-safe reply replay, Baileys timing controls, and stale runtime-deps cleanup.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is to add low-noise blocker attribution to liveness warnings, remove or offload the confirmed synchronous path, make WhatsApp reply delivery reconnect-safe with clear idempotency semantics, and define a safe retention policy for stale versioned runtime-deps roots.

## Reproduction Assessment

Unclear. The reporter provided credible production liveness logs and deterministic 408 timing, and source inspection verifies several missing requested behaviors, but current main still lacks a self-contained reproduction for the 33-100s event-loop blocks or the exact dropped auto-reply-after-disconnect path.

## Solution Assessment

No automated close or narrow repair is the best path yet. The remaining work spans performance tracing, delivery semantics, and runtime-deps retention policy, so maintainer scoping should come before a focused fix PR.
