## Summary

Keep open. Current main has partial mitigations for the report, including WhatsApp socket timing knobs, generic outbound write-ahead queue/reconnect drain plumbing, and runtime-deps reinstall-loop improvements, but the issue's central asks are still not fully satisfied: liveness warnings do not attribute the blocking async resource, direct WhatsApp auto-replies still send through the immediate `msg.reply` path rather than the durable outbound queue, and versioned runtime-deps roots are intentionally preserved by the current pruning logic.

## What This Changes

The issue asks OpenClaw to investigate WhatsApp 408 disconnects tied to event-loop blocking, add blocker attribution, replay queued replies after reconnect, expose Baileys timing options, and garbage-collect stale versioned plugin runtime-dependency trees.

## Best Possible Solution

Keep this as an active WhatsApp/Gateway reliability investigation. The maintainable end state is to add low-noise blocker attribution to liveness warnings, remove or offload the confirmed synchronous path, make WhatsApp auto-reply delivery reconnect-safe with clear idempotency behavior, and define a safe retention policy for stale versioned runtime-deps roots.

## Reproduction Assessment

Unclear. The reporter provided credible production logs and timing correlations, and source inspection verifies several missing requested behaviors, but current main still lacks a self-contained reproduction for the 33-100 second event-loop blocks or the exact dropped auto-reply-after-disconnect path.

## Solution Assessment

No. Closing or routing this as a narrow automated fix is not the best path yet because the remaining work spans performance tracing, delivery semantics, and runtime-deps retention policy; maintainer scoping should come first.
