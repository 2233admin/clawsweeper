## Summary

Keep open. Current main still contains the reported early return before terminal chat finalization, the registry API still permits a shift miss, no current regression test covers the forced peek-success/shift-miss branch, and the linked implementation PR is closed without merge.

## What This Changes

The issue requests that gateway chat lifecycle finalization still emit a terminal `chat` event when the chat-run registry entry disappears between `peek()` and `shift()`.

## Best Possible Solution

Fix the gateway producer path so a shift miss after an already-resolved chat link falls back to the resolved `sessionKey` and client run id, then add a focused regression test that forces `peek()` to be truthy while `shift()` returns `undefined` during the same terminal lifecycle handler call.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a linked chat run, force `chatRunState.registry.shift(evt.runId)` to return `undefined` after `peek()` has returned a link, and emit a terminal lifecycle `end` or `error`; current main returns before broadcasting a final `chat` event.

## Solution Assessment

Yes. The narrow gateway fallback is the best fix because it repairs the producer-side terminal event instead of adding another UI/TUI recovery path after the final event has already been suppressed; the regression test should force the actual branch rather than pre-consuming the registry before the handler starts.
