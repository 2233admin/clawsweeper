## Summary

Keep open. Current main still has the terminal lifecycle branch that can return before `emitChatFinal` when a chat-linked `registry.shift()` misses, and the exact fix PR linked from the timeline is closed without merge. The natural runtime trigger still needs a focused regression because the default registry is synchronous, but the remaining repair is narrow and well bounded.

## What This Changes

The issue asks the gateway chat lifecycle finalizer to emit a final `chat` event via the resolved session/run fallback when `registry.peek()` found a chat link but `registry.shift()` later returns `undefined`.

## Best Possible Solution

Add a focused defensive fallback inside `finalizeLifecycleEvent`: when a chat link existed but `shift()` returns `undefined`, emit the terminal chat final using the already resolved `sessionKey` and client/event run id, then continue normal lifecycle cleanup and session persistence. Cover it with a gateway agent-event regression test and keep broader stuck-Stop scenarios such as #42011 separate unless the same test proves a shared root cause.

## Reproduction Assessment

Unclear for a high-confidence end-to-end live repro: the current handler contains the early-return branch, but the standard registry is synchronous. A focused unit reproduction is clear by injecting or wrapping the registry so `peek()` returns a link and `shift()` returns `undefined`, then asserting current main suppresses the final `chat` event.

## Solution Assessment

Yes, the requested fallback is the narrowest maintainable repair for this code path. The safer implementation should avoid a broad UI/TUI workaround and should assert that shift-miss finalization emits once and still performs terminal cleanup.
