## Summary

Keep open. Current main still contains the reported early-return branch, but the exact production interleaving is not yet proven because the current chat registry and handler are synchronous; the related closed PR provides useful direction but does not settle the issue or cover the branch correctly.

## What This Changes

The issue requests that Gateway chat lifecycle finalization emit a terminal `chat` event via the resolved session/run fallback when `registry.shift(evt.runId)` misses after a prior registry link lookup.

## Best Possible Solution

Either add a deliberately defensive fallback in `finalizeLifecycleEvent` that emits exactly one terminal chat final using the already resolved `sessionKey`/`eventRunId` when `shift()` returns undefined, or have a maintainer confirm the invariant is impossible and route the observed stuck-streaming symptom to the broader terminal-lifecycle work. A fix should preserve abort handling, lifecycle-error retry grace, session lifecycle persistence, and include a regression that truly exercises the shift-miss branch.

## Reproduction Assessment

Unclear. A focused unit test can force the branch by registering a chat link and making `registry.shift()` return undefined after `peek()` succeeds, but static inspection did not find a natural production path because the registry and handler are synchronous with no mutation point between the two calls.

## Solution Assessment

Unclear. The proposed fallback is the narrowest defensive hardening if maintainers want to protect this invariant, but the existing source PR test should not be copied as-is because it shifts before invoking the handler and does not reproduce the reported branch.
