## Summary

Keep open. Current main has improved some surrounding Codex discovery behavior, and the session rejection is now recognizable as `agents.defaults.models` allowlist enforcement, but the central generator problem is not fully resolved: explicit custom providers with models can still be serialized without a resolved `apiKey`, while the pinned PI registry still rejects non-built-in custom providers in that shape and drops custom-model loading for that `models.json`.

## Best Possible Solution

Keep this issue open. The best fix is to add a narrow generator guard or neutralization path for non-built-in custom providers whose models require PI request auth after env/profile/provider-specific resolution fails, with a regression test covering one invalid custom provider plus a valid Codex/plugin provider. Separately, improve the `sessions_spawn`/`sessions.patch` error so it points at `agents.defaults.models` and explains why `models list --all` can show models that are not session-allowlisted.

## Risks / Open Questions
