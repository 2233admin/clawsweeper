## Summary

Keep open. Current main still exposes only display-oriented outbound identity metadata to channel outbound adapters, while the delivery pipeline already has an agent id in session or mirror context before constructing the adapter callback context. The related fix PR was closed without merge, so the requested SDK gap remains unresolved.

## What This Changes

Request to add optional `agentId` metadata to `ChannelOutboundContext` and propagate it into channel outbound adapter callbacks for per-agent send attribution.

## Best Possible Solution

Expose `agentId?: string` as stable routing and attribution metadata on the public channel outbound context, populate it from existing session or mirror metadata for agent-driven sends, omit it when no agent is in scope, and keep display `identity` semantics unchanged.

## Repair Work Prompt

Implement the requested optional channel outbound `agentId` exposure for this issue, using #73072 only as source context because it is closed and unmerged. Add `agentId?: string` to `ChannelOutboundContext` as stable routing/attribution metadata distinct from display `identity`. Thread `agentId` through `ChannelHandlerParams`, `createChannelHandler`, `createPluginHandler`, and `createChannelOutboundContextBase` from `params.session?.agentId ?? params.mirror?.agentId`, matching the existing media-access precedence. Ensure `sendText`, `sendMedia`, `sendPayload`, `sendFormattedText`, `sendFormattedMedia`, and `renderPresentation` receive it, while sends with no agent in scope omit the key rather than setting a spurious `undefined` property where practical. Add focused regression tests in `src/infra/outbound/deliver.test.ts` for session propagation, mirror fallback, and absent-agent behavior. Update the channel plugin SDK docs to describe `agentId` as optional attribution/routing metadata and do not move it into `OutboundIdentity` or display fields. Changelog is only needed if repo policy treats this as user-facing plugin SDK behavior for the active release.
