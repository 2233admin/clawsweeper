## Summary

Keep open. Current main still lacks `agentId` on the plugin-facing outbound context, while the outbound session and mirror paths already carry an agent id upstream. The related implementation PR was closed unmerged, so it does not supersede or implement the request.

## What This Changes

The issue requests adding an optional stable `agentId` field to `ChannelOutboundContext` so channel plugins can attribute outbound sends to a specific agent.

## Best Possible Solution

Add a backward-compatible optional `agentId?: string` to the channel outbound context, thread it from session first and mirror second where delivery already has that metadata, update SDK API baseline/docs, and cover session, mirror fallback, and absent-id cases in outbound delivery tests.

## Reproduction Assessment

Yes. A read-only source trace on current main reproduces the gap: session and mirror metadata can carry `agentId`, but `ChannelOutboundContext`, `ChannelHandlerParams`, and the context builder do not expose it to adapter calls.

## Solution Assessment

Yes, with maintainer approval. Keeping stable agent routing metadata separate from display-only `identity` is the narrowest maintainable shape; the safer implementation is additive, omitted when absent, and explicitly documented as non-authorizing metadata.

## Risks / Open Questions

- The field changes a public plugin SDK contract, so maintainers should approve the semantics and precedence before automation attempts a replacement PR.

- The adapter context should omit `agentId` when absent rather than passing `agentId: undefined`, because plugins may enumerate context keys.
