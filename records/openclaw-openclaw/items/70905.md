## Summary

Keep this issue open. Current main still does not expose a stable `agentId` on `ChannelOutboundContext`, even though upstream outbound/session paths already carry agent identity and use it for media scoping and routing. This is an actionable public channel plugin SDK gap, not a ClawHub-only plugin request or duplicate of the related reports.

## Best Possible Solution

Add optional `agentId?: string` to `ChannelOutboundContext`, derive it in outbound delivery from the existing session/mirror context, thread it through `ChannelHandlerParams` and `createChannelOutboundContextBase`, cover every outbound adapter callback with tests, and document that `agentId` is stable routing/attribution metadata distinct from display `identity`.

## Risks / Open Questions

- The implementation should define a precedence rule for mismatched `session.agentId` and `mirror.agentId`; current media scoping uses `session` first, then `mirror`.
- Because this is public plugin SDK surface, it needs focused contract tests and docs rather than only internal plumbing.
- All outbound adapter surfaces should be covered: `sendText`, `sendMedia`, `sendPayload`, `sendFormattedText`, `sendFormattedMedia`, and `renderPresentation`; sends with no agent in scope should leave the field absent.
