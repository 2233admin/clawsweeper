## Summary

Keep open. Current `origin/main` still lacks a live fallback-attempt model signal, and the TUI footer still renders `sessionInfo.modelProvider` and `sessionInfo.model` without any agent-event path that updates those fields during the fallback loop. The related closed status-bar items cover selected/default model display, not the in-flight provider/model attempted during fallback.

## What This Changes

Show the provider/model currently being attempted in the TUI status bar during provider fallback instead of the last session/default model.

## Best Possible Solution

Add a typed fallback-attempt status path emitted immediately before each actually attempted candidate, carrying `runId`, `sessionKey`, provider, model, attempt, and total. Broadcast it through the existing agent-event path and consume it in the CLI TUI with correct session/run filtering so the footer updates before `sessions.list` catches up.
