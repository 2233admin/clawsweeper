## Summary

Keep open. Current main still does not report Feishu connection lifecycle or transport activity into the gateway runtime snapshot, so the intended behavior is not implemented on main. This PR remains relevant, but the provided PR head is incomplete against the current health-monitor contract and needs revision before it can safely merge.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is to consolidate with #39487/#39489 or pick one canonical PR, then implement Feishu status reporting end to end: pass a bound `ctx.setStatus` sink from `extensions/feishu/src/channel.ts` through `monitorFeishuProvider` and `monitorSingleAccount`, publish lifecycle fields from websocket/webhook start, abort, disconnect, and error paths, update `lastTransportActivityAt` from verified transport activity, optionally update `lastEventAt` for app events, and add focused tests proving Feishu snapshots drive the gateway health policy.

## Risks / Open Questions

- Merging the PR as provided would still leave real Feishu runs without gateway health lifecycle status because the status sink is not passed from the channel start path through `monitor.account.ts` to the transports.
- The PR focuses on `lastEventAt`, but current main's stale-socket policy requires meaningful `lastTransportActivityAt`; app-event timestamps alone will not make stale-socket detection work.
- Related open PRs #39487 and #39489 overlap this work, so maintainers should choose or consolidate a canonical implementation before merging duplicate fixes.
