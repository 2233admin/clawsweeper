## Summary

Keep open. Current main still does not report Feishu connection lifecycle or transport activity into the gateway health snapshot, and this PR remains relevant. The PR head is incomplete against the current contract, though: it changes only `monitor.ts`, `monitor.transport.ts`, and a new test, so the gateway start path and single-account monitor still do not pass a status sink through to real Feishu runs.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is to consolidate with #39487/#39489 or choose one canonical PR, then implement Feishu status reporting end to end: pass a bound `ctx.setStatus` sink from `extensions/feishu/src/channel.ts` through `monitorFeishuProvider` and `monitorSingleAccount`, publish lifecycle fields from websocket/webhook start, abort, disconnect, and error paths, update `lastTransportActivityAt` from verified transport activity, optionally update `lastEventAt` for app events, and add focused tests proving Feishu snapshots drive the gateway health policy.

## Risks / Open Questions

- Merging the PR as currently posted would still leave real gateway-managed Feishu runs without status reporting because the status sink is not passed from `channel.ts` through `monitor.account.ts` to the transports.
- The PR likely fails or needs substantial rebasing against current main: it passes `setStatus` into `monitorSingleAccount`, but current `MonitorSingleAccountParams` has no such field, and `monitor.transport.ts` has since been refactored around reconnect handling.
- Even if the callback reached the transports, `lastEventAt` alone does not satisfy the current stale-socket policy; a complete fix needs meaningful `lastTransportActivityAt` from verified transport activity.
- There is duplicate implementation work in open PRs #39487 and #39489, so maintainers should pick or consolidate a canonical approach before merging competing partial fixes.
