## Summary

Keep open. Current main still does not implement the PR's two intended behavior changes: restart-health still probes with only process environment gateway credentials, and rate-limit failover still prefers one same-provider auth-profile rotation before model fallback by default. The PR is not obsolete, but it needs a rebase and maintainer policy review.

## Best Possible Solution

Keep this PR open for maintainer follow-up. Rebase the restart-health part onto current main and wire it to the existing config-aware probe-auth helper with explicit warning handling. For the rate-limit part, either update the current failover policy, docs, and tests to skip same-provider rotation when a fallback model is configured, or close that portion as a rejected policy change while preserving the restart-health fix if desired.

## Risks / Open Questions

- The restart-health change is credential-sensitive: maintainers should decide config-vs-env precedence, local-vs-remote credential fallback, and how unresolved SecretRef warnings should surface.
- The PR head targets an older embedded runner shape, while current main has split failover policy and assistant failover logic into src/agents/pi-embedded-runner/run/*; it likely needs a rebase rather than direct merge.
- The rate-limit default is now documented and tested as one same-provider rotation before fallback. Changing that default should be an explicit product policy decision.
