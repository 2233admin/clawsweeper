## Summary

Keep this PR open. Current main has related Gateway timeout work, but it does not implement this PR's intended drift-based event-loop readiness preflight for `gateway probe`. The probe path still calls `probeGateway` directly and opens the WebSocket after setup without a readiness wait, while the existing event-loop yield only covers `callGateway`.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land a bounded, tested event-loop readiness helper for `probeGateway` and any affected Gateway RPC paths, or explicitly close the PR only after Windows/Node reproduction shows current main's budget and one-turn-yield changes fully solve the probe timeout without the drift-based preflight.
