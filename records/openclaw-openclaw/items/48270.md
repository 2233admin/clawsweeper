## Summary

Keep this PR open. Current main has a related `callGateway` one-turn event-loop yield, but it does not implement this PR’s shared drift-based readiness wait, and the `gateway probe` CLI still reaches `probeGateway` directly without that preflight. The PR is therefore not definitely implemented on main.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is either to land a bounded, tested readiness helper shared by `probeGateway` and `callGateway`, or to close the PR only after Windows/Node reproduction shows current main’s lazy-loading and one-turn RPC yield fully fix the `gateway probe` timeout without this change. Add focused tests around probe preflight behavior and timeout-budget handling if the helper lands.
