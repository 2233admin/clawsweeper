## Summary

Keep open. Current main still has the reported Telegram transport behavior: a fetch-failed error graph with a qualifying timeout or socket code can promote the resolver to sticky IPv4-only fallback, and that fallback policy forces IPv4/ipv4first even when the account configured dnsResultOrder: "verbatim". I did not find loopback IPv4 ECONNREFUSED suppression or regression coverage for the reported /etc/hosts override case.

## What This Changes

The issue asks Telegram networking to avoid arming sticky IPv4-only fallback when /etc/hosts intentionally maps api.telegram.org IPv4 to loopback while dnsResultOrder is verbatim and IPv6 is the intended route.

## Best Possible Solution

Add Telegram transport classifier or state-transition logic that recognizes loopback IPv4 ECONNREFUSED as an intentionally disabled IPv4 path in verbatim/IPv6-preferred configurations, prevents promotion to sticky IPv4 in that case, and covers the behavior with focused tests while preserving existing fallback for genuinely broken IPv6 routes.
