## Summary

Keep this PR open. Current main has newer iMessage self-chat, echo-cache, loop-rate, and reflected-content guards, but it still does not implement the PR's account-handle config, startup validation, or pre-policy sender-handle drop. The Greptile comment positively matches the intended two-layer guard, and the related open #63666 only adds reflected-content test coverage rather than superseding this account-handle protection.

## Best Possible Solution

Keep this PR open and review or refresh it against current main. The best implementation path is still iMessage-extension-owned: add an own-account handle field to the iMessage config type/schema, validate it against effective allowlists at monitor startup, pass a normalized handle into inbound decision processing, drop matching senders before `dmPolicy`/allowlist routing, feed that drop into loop-rate limiting, and add focused tests plus docs/config-reference updates as needed.
