## Summary

Keep open. Current main still appears to have the reported native persistent sub-agent duplicate-delivery path: `sessions_send` only suppresses A2A for parent-owned ACP sessions, while native sub-agent sends can return the child reply inline and then run the A2A reply/announce flow through the child session's stored `deliveryContext`. Related #53145 is useful overlap, but it is broader and does not make this native persistent-session report terminally redundant.

## What This Changes

The issue asks OpenClaw to prevent persistent native sub-agent `sessions_send` replies from appearing multiple times in the parent conversation and to avoid announce feedback through inherited sub-agent delivery context.

## Best Possible Solution

Fix the native parent-owned persistent sub-agent/session-send contract so a parent-visible child result has exactly one delivery path. The likely implementation should suppress duplicate same-run parent injection and avoid using the child session's inherited deliveryContext for parent-owned native sends, while preserving peer A2A and the existing ACP-owned skip semantics.

## Reproduction Assessment

Yes. The report gives concrete steps, and current source provides a high-confidence code-level reproduction path: a waited native sessions_send can return the child reply inline and still schedule A2A reply/announce work that resolves delivery from the target session's stored deliveryContext.

## Solution Assessment

No current-main solution exists for the reported native path. The narrow maintainable fix is to detect parent-owned native persistent sub-agent sends and choose one result path, rather than disabling peer A2A or announce delivery globally.
