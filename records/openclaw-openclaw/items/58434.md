## Summary

Keep open. Current main does not implement the PR's per-request `tool_deny` override for `/v1/responses`: the request schema is strict and has no `tool_deny` field, the OpenResponses handler never derives or forwards a deny override, and the agent tool pipeline only applies existing config/group/sandbox/subagent policies. The PR discussion is constructive: Greptile found the runtime approach sound and the one docs issue was addressed in the PR's follow-up commit, so this remains plausible focused core API work rather than an auto-close candidate.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land a narrow implementation that adds `tool_deny` to the strict OpenResponses schema, threads it through HTTP ingress and agent command options into the existing tool-policy pipeline as a deny-only per-request step, documents it on the public OpenResponses API page, and includes targeted tests for schema acceptance, group expansion, streaming/non-streaming plumbing, and non-persistence across follow-up requests.
