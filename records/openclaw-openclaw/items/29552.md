## Summary

Keep this issue open. Current main still does not implement the local gateway auth fallback that would address the reported pairing failure: local call credentials still prefer a stale OPENCLAW_GATEWAY_TOKEN over gateway.auth.token, callGatewayWithScopes sends one resolved credential set without a config-auth retry, and the existing AUTH_TOKEN_MISMATCH retry is scoped to cached device-token recovery. The directly related fallback PR #29676 is closed unmerged; open PR #72535 only improves newer-config warning text.

## Best Possible Solution

Keep this issue open for a maintainer auth-path decision. Either intentionally keep env-first local call precedence and document the manual stale-token recovery path more directly, or add a narrowly guarded local-only retry after AUTH_TOKEN_MISMATCH that retries once with config-first gateway.auth token/password when there is no remote mode, URL override, explicit token/password, or unresolved SecretRef. Warning-message cleanup should remain separate in #72535.
