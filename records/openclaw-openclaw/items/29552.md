## Summary

Keep this issue open. Current main has adjacent newer-config guards and clearer auth/device-token handling, but the reported gateway pairing failure still maps to an unresolved ordinary local Gateway auth path: local calls resolve one credential set, env credentials still win over `gateway.auth` for normal local call/probe paths, and `AUTH_TOKEN_MISMATCH` retry remains limited to trusted cached device-token recovery rather than retrying once with config-first `gateway.auth` credentials.

## What This Changes

Gateway pairing should recover or give actionable guidance when a version/update mismatch leaves local gateway credentials out of sync and produces `gateway token mismatch` failures.

## Best Possible Solution

Keep this open for maintainer Gateway-auth judgment. The best path is either to explicitly preserve env-first local call precedence and document stale-token recovery more directly, or add a narrowly guarded local-only retry after `AUTH_TOKEN_MISMATCH` that retries once with config-first `gateway.auth` token/password only when there is no remote mode, URL override, explicit token/password, or unresolved SecretRef. Treat #72535 as warning-message cleanup and any auth recovery as a separate focused Gateway-auth change with tests.
