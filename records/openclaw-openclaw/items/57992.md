## Summary

Keep #57992 open. Current main has some downstream error handling for failed subagent runs, but it does not prove or fully implement the requested behavior for transient Docker container creation failures during sandboxed `sessions_spawn`: the spawn path can still return `accepted` before the child run resolves its sandbox/container, Docker create/start has no retry or local log, and I found no targeted regression test or release evidence covering this failure mode. Related #39248 overlaps the non-main sandbox spawn area and remains open, but #57992 adds a distinct Docker-container failure/logging/error-state requirement that should not be closed as a duplicate without a canonical fix.

## Best Possible Solution

Keep the issue open and fix it in the sandboxed subagent execution path. Add a targeted regression test that forces Docker sandbox/container creation to fail after a `sessions_spawn` accept, then verify the child run transitions to failed, `subagents list` or session metadata exposes an actionable error, and gateway logs include the Docker failure. Consider one bounded retry around Docker create/start and keep #39248 linked as related non-main sandbox spawn coverage rather than treating it as a complete duplicate.
