## Summary

Keep open. Current main has useful partial mitigations for this class of failures, including event-loop diagnostics in readiness/diagnostic heartbeat, Feishu request timeouts and WebSocket ping/retry defaults, and a separate active-memory timeout fix. It does not yet prove the reported WSL2/Feishu all-HTTP timeout freeze is fixed, and it still lacks the requested supervisor-level recovery path for a process that is alive but unable to service timers or HTTP.

## Best Possible Solution

Keep this issue open as the broader gateway reliability thread. Treat #65517 as the fixed active-memory trigger, and continue this issue with a current-main diagnostics export or repro focused on WSL2/Feishu/all-HTTP failure. The best code direction is a supervisor-visible recovery path, such as systemd watchdog/readiness integration or an external health probe contract, plus Feishu post-start liveness/reconnect coverage if the Feishu transport remains implicated. If maintainers later confirm #65164 is the canonical all-HTTP failure tracker across platforms, close this as a duplicate then with that explicit comparison.

## Risks / Open Questions

- Closing as implemented would overclaim: current main adds observability and some narrower trigger fixes, but not a proven fix for the original WSL2 idle freeze or all outbound HTTP timeout cascade.
- A truly frozen Node event loop cannot execute the in-process diagnostic heartbeat or answer /readyz, so an external supervisor watchdog remains the missing recovery mechanism.
- #65164 overlaps on the all-outbound-HTTP symptom but differs in OS, Node version, startup timing, and WebSocket behavior, so it is not a high-confidence duplicate for this issue yet.
