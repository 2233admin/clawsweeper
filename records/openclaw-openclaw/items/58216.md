## Summary

Current main still lacks the Discord lifecycle guard this PR adds. A live `reconnect-exhausted` event emitted after abort but before lifecycle cleanup still returns `"stop"`, which `waitForDiscordGatewayStop` converts into a rejected lifecycle error. The latest PR diff is narrow, addresses the earlier review comments, and does not introduce security-sensitive or supply-chain changes.

## Best Possible Solution

Keep this PR open for maintainer review or landing. The best fix is the focused Discord plugin change in `extensions/discord/src/monitor/provider.lifecycle.ts`: when `event.type === "reconnect-exhausted"` and either `lifecycleStopping` or `params.abortSignal?.aborted` is true, log the expected shutdown event and return `"continue"` so `waitForDiscordGatewayStop` does not reject. Keep the repaired regression test that synchronously emits the live event during abort, then validate with the targeted Discord lifecycle test and the changed gate outside this read-only review.
