## Summary

Keep this PR open. The current-main gateway-status target path still needs hardening for metadata/link-local destinations, but this patch is incomplete because its string checks miss IPv4-mapped IPv6 literals for the same metadata range, a bypass already called out in the PR discussion.

## What This Changes

The PR adds gateway-status WebSocket hostname filtering plus a new security test file for metadata, link-local, wildcard, internal/local, localhost, loopback, LAN, public-hostname, and non-WebSocket URL cases.

## Best Possible Solution

Land a current-main gateway-status hardening that parses each explicit/configured target URL once, uses the shared SSRF/IP classifier or equivalent canonical IP-literal handling for mapped IPv6 and legacy forms, preserves intended loopback/LAN diagnostics, and adds regression coverage for mapped and hex IPv6 metadata URLs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only rejects exact 0.0.0.0, dotted 169.254.* hosts, and blocked hostname suffixes. Node normalizes `wss://[::ffff:169.254.169.254]:80` to `[::ffff:a9fe:a9fe]`, which bypasses those checks while still targeting the link-local metadata range. Use shared IP classification or equivalent canonical IP-literal handling and add mapped/hex IPv6 cases.
  - confidence: 0.94
