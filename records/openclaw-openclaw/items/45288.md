## Summary

Keep open. Current main has not implemented gateway-status SSRF hostname/IP filtering, but this PR should not merge as-is because the proposed string-prefix policy misses IPv4-mapped IPv6 metadata addresses and blocks documented `.local` gateway targets; the branch is also dirty against current main.

## What This Changes

The PR adds hostname/IP screening to `gateway-status` URL normalization and a new security test file for metadata/internal/wildcard blocks plus local/private gateway allow cases.

## Best Possible Solution

Keep the hardening effort, but implement a gateway-specific URL policy that parses IP literals with the shared IP helpers, rejects wildcard/link-local/cloud-metadata targets including IPv4-mapped and transition IPv6 forms, preserves documented `.local` and private-LAN gateway routes under the existing transport-security rules, and covers both explicit URLs and `gateway.remote.url`.

## Reproduction Assessment

Yes. Source inspection shows current main accepts the target strings into `resolveTargets()`, and a focused Node URL parsing check shows the PR’s IPv4-mapped metadata bypass without requiring a live network probe.

## Solution Assessment

No. The PR is directionally useful, but string-prefix hostname checks plus the generic blocked-hostname helper are not the best maintainable gateway policy; a context-specific parser should reuse shared IP classification and keep documented gateway routing semantics intact.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new block only checks dotted `169.254.` strings and `0.0.0.0`. `URL` canonicalizes `ws://[::ffff:169.254.169.254]:80` to an IPv6 hostname like `[::ffff:a9fe:a9fe]`, so the metadata endpoint bypasses this branch. Use the shared IP parser/SSRF helper and add IPv4-mapped/hex IPv6 cases before merging.
  - confidence: 0.94
- **[P2] Preserve documented .local gateway targets:** `src/commands/gateway-status/helpers.ts:110-112`
  - body: `isBlockedHostname()` rejects `.local`, but OpenClaw documents `.local` gateway routes for private LAN/Bonjour flows and TLS or explicit private-network opt-in paths. This would drop targets such as `wss://gateway.local:18789` or opted-in private LAN `ws://gateway.local:18789` instead of probing them.
  - confidence: 0.84
