## Summary

Keep this PR open. Current main still has the gateway-status target-normalization gap, so the hardening is not implemented there; however, the proposed patch is incomplete because IPv4-mapped IPv6 forms of the metadata address still bypass the new dotted-prefix checks. The existing review comments correctly identified this security hole, and the fix should be updated before merge.

## What This Changes

This PR adds gateway-status WebSocket target hostname checks and a new security test file to reject metadata/link-local/wildcard/internal hosts while preserving localhost, loopback, LAN IPs, and public hostnames.

## Best Possible Solution

Land a current-main gateway-status hardening that uses the existing shared SSRF host/IP classification for special-use and embedded-IPv4 literals, keeps the intended local/LAN gateway probing policy explicit, adds IPv4-mapped and hex IPv6 regression cases, and resolves the documented `.local` / `.internal` gateway-discovery policy before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Reject IPv4-mapped metadata addresses:** `src/commands/gateway-status/helpers.ts:103-106`
  - body: The new guard only rejects exact `0.0.0.0`, dotted `169.254.*`, and blocked hostname suffixes. Node canonicalizes `ws://[::ffff:169.254.169.254]:80` to `::ffff:a9fe:a9fe`, which does not match those checks and is not a blocked hostname, so the explicit target is still accepted. Use the shared SSRF IP classifier or equivalent IP-literal handling, and add mapped/hex IPv6 cases before this can be a complete CWE-918 fix.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The PR is a security fix, but the diff leaves a concrete SSRF bypass for IPv4-mapped IPv6 metadata endpoints.

Concerns:

- **[high] IPv4-mapped metadata URLs still bypass:** `src/commands/gateway-status/helpers.ts:103`
