## Summary

Keep open. Current main still matches the reported updater failure shape: dev preflight can skip over failed candidate lint runs and select a later clean commit, but the final result calculation still treats any earlier failed preflight lint step as blocking. The related #63627 close covered pre-commit hook OOMs, and #64818 is related preflight walkback hardening but does not replace this updater false-error path.

## What This Changes

`openclaw update --channel dev` should complete successfully when the updater walks back past commits whose preflight lint was SIGKILLed and finds a clean candidate.

## Best Possible Solution

Teach the dev updater to separate failed preflight attempts for discarded candidate commits from actual blocking failures after a clean candidate is selected, while keeping selected-candidate, rebase, install, build, UI build, and doctor failures fail-closed. Add a focused regression test for first-candidate lint failure followed by a clean second candidate.

## Reproduction Assessment

Yes. The issue log gives a direct reproduction shape, and current source still deterministically returns an error if any earlier failed preflight step remains in `steps` after a later candidate succeeds.

## Solution Assessment

Yes. The narrowest maintainable fix is in `runGatewayUpdate` result classification, not a global skip of dev preflight lint; memory/scoping improvements can be follow-up work if maintainers want to reduce SIGKILL risk.
