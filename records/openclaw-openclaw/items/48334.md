## Summary

Keep open. Current main does not implement the PR’s intended fix: cron isolated-agent finalization still persists context tokens from the nonblocking synchronous cache path, and status still treats stored positive contextTokens as authoritative with no stale-200k display repair. The PR is focused and not superseded; it needs review/rebase plus the existing review concerns addressed.

## Best Possible Solution

Keep this PR open and review/rebase it, or port the same fix onto main: cron isolated-agent persistence should resolve provider/model context windows through a blocking provider-aware path before storing contextTokens, and status should repair only the known stale DEFAULT_CONTEXT_TOKENS fallback while preserving explicit caps and validating stored values. Add/keep regression tests for cron persistence and status display repair, and make sure the status path remains read-only.

## Risks / Open Questions

- Closing this PR would discard a focused bug-fix candidate while current main still has the nonblocking cron persistence path and no status repair.
- Before merge, the status repair should preserve explicit context caps when an agent id cannot be parsed and should ignore invalid stored contextTokens such as 0, negative, or NaN.
- The async/status side of the PR should preserve the documented read-only status contract; current context cache loading can call ensureOpenClawModelsJson, which may write models.json.
