## Summary

Keep this PR open. Current main at f256eeba431b5f59e6085b7c11c9c0a309e408b7 still does not implement the PR’s central fix: isolated cron finalization persists context tokens via a nonblocking sync lookup, and status still treats any positive stored session contextTokens value as authoritative with no stale DEFAULT_CONTEXT_TOKENS display repair. The PR remains a focused bug-fix candidate, though its status-side implementation needs maintainer review to preserve the documented read-only status path and address the existing bot review comments.

## Best Possible Solution

Keep this PR open for maintainer review/rebase, or port the narrow fix onto current main: add a blocking provider/model context-window resolver suitable for cron persistence, use it when finalizing isolated cron sessions, add a read-only status display repair only for known stale DEFAULT_CONTEXT_TOKENS fallbacks, preserve explicit caps and invalid-value guards, and keep regression coverage for cron persistence plus status display repair.
