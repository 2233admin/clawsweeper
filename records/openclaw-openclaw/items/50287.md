## Summary

Keep open. Current main still does not enforce the requested guardrail for `models.providers.*.models[*].input`: core config validation accepts `audio` and `video`, while docs and catalog normalization still point to a narrower text/image deployment contract. Recent code also intentionally preserves configured audio/video modalities, so a maintainer should resolve that product contract before an automated fix PR tightens validation.

## Best Possible Solution

Keep this open and ask a maintainer to settle the model-input contract first. If `models.providers.*.models[*].input` is intended only for chat-model native inputs, tighten `ModelDefinitionSchema`, `ModelDefinitionConfig`, provider-catalog handling, and tests to reject unsupported audio/video before deploy. If audio/video are now intended, document and implement the actual runtime capability path instead, then update the issue as superseded by that broader contract change.
