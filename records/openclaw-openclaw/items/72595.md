## Summary

Keep open. Current main still has no Feishu/Lark channel proxy policy or mixed-proxy troubleshooting docs, and the linked same-author docs PR is open rather than merged. The report matches current code: Feishu WebSocket creation intentionally picks up ambient env proxy, while Feishu SDK HTTP calls use the SDK HTTP instance without a Feishu-owned direct/ambient policy switch.

## What This Changes

Request Feishu/Lark support for per-channel proxy bypass or NO_PROXY guidance so Feishu API/token/WebSocket traffic can go direct while model-provider traffic keeps using ambient proxy env.

## Best Possible Solution

Review the open docs PR as the immediate NO_PROXY guidance path, then decide whether the remaining runtime need should be a Feishu-owned channel/account proxy policy that consistently covers SDK HTTP, streaming-card token fetches, and WebSocket creation with focused tests and docs.

## Reproduction Assessment

Yes for a code-backed reproduction path: set gateway HTTP_PROXY/HTTPS_PROXY to a proxy that breaks Feishu/Lark official endpoints while provider traffic still needs it, then start Feishu in WebSocket mode and hit token/API/WebSocket startup. I did not run the live Windows network scenario here.

## Solution Assessment

Unclear. NO_PROXY documentation is the narrowest low-risk first step, but a channel-level proxy policy may be the better fix if maintainers want reliable per-channel behavior instead of relying on process env tuning.
