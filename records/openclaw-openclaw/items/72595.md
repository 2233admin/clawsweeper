## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting docs and a first-class Feishu proxy policy, and the same author has open PR #75050 covering the docs portion, so closing this issue before that PR/product decision would be premature.

## What This Changes

Request Feishu/Lark channel-specific proxy handling or documentation so provider traffic can keep ambient proxy settings while Feishu/Lark HTTP and WebSocket traffic can bypass them when needed.

## Best Possible Solution

Review and land #75050 if the docs guidance is acceptable, then decide whether a Feishu-owned proxyMode is warranted. If runtime support is approved, implement it inside the Feishu plugin config/account resolution and apply the same policy consistently to Lark SDK HTTP calls, streaming-card fetches if applicable, and WebSocket startup, with focused config and client tests.

## Reproduction Assessment

Unclear. The issue gives a concrete Windows mixed-proxy scenario, endpoint, and error string, and current-source review confirms the requested control surface is missing; high-confidence live reproduction still needs Feishu/Lark credentials plus a conflicting proxy environment.

## Solution Assessment

Unclear for the full issue. The open docs PR is a narrow, low-risk partial fix, but a channel-level proxy policy is a network-egress-sensitive product choice and should be implemented only after maintainers decide the public config contract.
