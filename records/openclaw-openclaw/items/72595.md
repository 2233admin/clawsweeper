## Summary

Keep open. Current main still has neither Feishu/Lark mixed-proxy troubleshooting docs nor a first-class Feishu proxy policy, and the same author has an open related docs PR that should be reviewed rather than closing this issue ahead of merge.

## What This Changes

Request to let Feishu/Lark channel traffic bypass or explicitly control proxy routing independently from model-provider HTTP(S)_PROXY env settings in mixed Windows proxy setups.

## Best Possible Solution

Review and, if acceptable, land #75050 as the immediate documentation fix. If maintainers want the fuller behavior change, add a Feishu-owned proxy policy that applies consistently to Lark SDK HTTP calls and WebSocket startup, with config-schema/docs/client tests, without changing global provider proxy behavior.

## Reproduction Assessment

Unclear. The issue gives a concrete Windows mixed-proxy setup and Feishu tenant-token endpoint error, but high-confidence live reproduction requires Feishu/Lark credentials plus a controlled proxy environment; current source review confirms the requested docs/config surface is missing.

## Solution Assessment

Unclear for the full request. The open docs PR is a narrow low-risk partial solution, while a channel-level proxyMode is likely the maintainable runtime fix only if maintainers accept the explicit egress-policy change.
