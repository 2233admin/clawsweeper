## Summary

Keep open. Current main still lacks both Feishu/Lark mixed-proxy troubleshooting guidance and a Feishu channel proxy policy, and the same author has an open paired docs PR (#75050), so maintainers should decide whether docs-only NO_PROXY guidance is enough or whether runtime Feishu proxy behavior should change.

## What This Changes

Request to let Feishu/Lark HTTP token/API and WebSocket traffic bypass or explicitly choose proxy routing per channel while model-provider traffic continues using ambient proxy env vars.

## Best Possible Solution

Keep this open while maintainers review #75050. If docs-only mitigation is accepted, land that PR and close afterward; otherwise implement a Feishu-owned proxy policy that applies consistently to the Lark SDK HTTP client, token/API fetch paths, and WebSocket client, with focused tests and docs.

## Reproduction Assessment

Yes. The report gives a concrete Windows mixed-proxy setup, affected Feishu token endpoint, observed error, and workaround; current source inspection confirms there is no channel-level policy to reproduce or avoid that split routing today.

## Solution Assessment

Unclear. NO_PROXY documentation is the lowest-risk mitigation, while a channel-level proxyMode is more complete; maintainers should choose before automation changes runtime network egress behavior.
