## Summary

Keep open. Current main still lacks Feishu/Lark mixed-proxy troubleshooting guidance and still has no Feishu-owned proxy policy; the issue is also paired with open docs PR #75050, so maintainers should decide whether docs-only NO_PROXY guidance is enough or whether runtime behavior needs a channel-level option.

## What This Changes

Request Feishu/Lark channel-specific proxy guidance or configuration so provider traffic can keep using ambient proxy env vars while Feishu HTTP token/API and WebSocket traffic can bypass or explicitly control that proxy.

## Best Possible Solution

Keep this open while maintainers review #75050. If docs-only NO_PROXY guidance is accepted, land that PR and close afterward; otherwise add a Feishu-owned proxy policy that consistently covers SDK HTTP token/API calls, fetch-based token paths, and WebSocket startup, with focused tests and docs.

## Reproduction Assessment

Yes. The issue gives a concrete Windows mixed-proxy setup, endpoint, env vars, and error, and current source inspection confirms Feishu HTTP and WebSocket routing still lack a shared per-channel direct/ambient policy.

## Solution Assessment

Unclear. Docs-only NO_PROXY guidance is the lowest-risk mitigation, but a channel-level proxyMode may be the more complete fix if maintainers want OpenClaw to own this mixed-routing behavior instead of leaving it to env configuration.
