## Summary

Keep open. The feature is not implemented on current main, and the PR is still useful but not merge-ready: it adds a config key in the old memory paths while the active package-backed SDK, OpenAI provider, batch path, config merge, docs/help, and tests still need a coordinated update. The related #41670 remains open and overlaps the provider-policy decision rather than superseding this PR.

## What This Changes

This PR adds an optional `memorySearch.remote.encodingFormat` config key, sends `encoding_format` on remote embedding requests, and adds base64 response decoding in the older `src/memory` direct and batch embedding paths.

## Best Possible Solution

Rework the feature onto current main after maintainers choose the policy. A complete implementation should add a documented `encodingFormat` contract only where supported, propagate it through resolved memory settings and cache identity as needed, update the package-backed SDK and bundled OpenAI direct/batch paths, decode and validate base64 float32 responses, document provider compatibility caveats, and cover omitted, `float`, `base64`, invalid config, direct, and batch behavior with focused tests.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: add `agents.defaults.memorySearch.remote.encodingFormat`, then follow `MemorySearchSchema` and `resolveMemorySearchSettings`; current main rejects or drops the key before the active provider paths can send it.

## Solution Assessment

No. The user-facing direction is plausible, but this PR is not the best current fix until it is rebased onto the package-backed memory host SDK and plugin provider architecture and reconciled with the documented provider-compatibility split and #41670.
