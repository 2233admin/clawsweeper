## Summary

Keep open. Current main still does not implement configurable `encodingFormat` for memory embedding requests, and the PR remains useful but not ready: it overlaps the narrower open #41670, targets paths that have since moved into the memory host SDK/OpenAI plugin architecture, and still has a config propagation defect in the supplied diff.

## What This Changes

The PR branch adds an optional `memorySearch.remote.encodingFormat` config key, sends `encoding_format` on remote embedding requests, updates config schema/types/labels, and decodes base64 embedding responses in direct and batch paths.

## Best Possible Solution

Port the feature onto the current memory host SDK and provider plugin architecture: add schema/type/help/generated config coverage, preserve the setting through resolved memory config, include `encoding_format` consistently in direct and batch OpenAI-compatible requests when configured, decode and validate base64 float32 responses in shared parsers, and add focused tests for omitted, `float`, `base64`, invalid config, direct, and batch cases.
