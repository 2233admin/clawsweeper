## Summary

Keep open. Current main still compares update versions through the generic SemVer helper, so numeric-only `-N` suffixes still behave as prereleases for update availability, but this PR is not ready to close or merge as-is because current main has since moved the parser into a shared ClawHub/plugin SemVer helper and the PR's numeric parsing has a low-severity precision concern.

## What This Changes

The PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and same-base prereleases.

## Best Possible Solution

Keep the PR open and adapt it to current main by applying correction-release ordering only to OpenClaw package update/status/startup/update-command comparisons, or by adding an explicit comparator mode that preserves generic ClawHub/plugin SemVer behavior. The final patch should compare numeric suffixes precision-safely and add focused regression coverage for correction-vs-stable, correction-vs-beta, revision ordering, and the affected update callers.

## Reproduction Assessment

Yes. The current source-level path is high confidence: update availability calls `compareSemverStrings()`, that function delegates to the generic SemVer parser, and the current regression test still asserts numeric `-N` as prerelease ordering.

## Solution Assessment

Unclear as-is. The requested behavior is the right direction for OpenClaw package versions, but the PR needs to be retargeted after the shared comparator extraction and should avoid lossy numeric suffix parsing.

## Review Findings

Overall correctness: patch is correct

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. Registry-provided versions above `Number.MAX_SAFE_INTEGER` can collapse adjacent revisions to the same JS number and produce the wrong update ordering; keep the suffix as a normalized string, use `BigInt`, or bound the accepted length before comparing.
