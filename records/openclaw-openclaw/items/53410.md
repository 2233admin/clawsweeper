## Summary

Keep open. Current main still compares update/status versions through the generic SemVer helper, which treats numeric `-N` suffixes as prerelease identifiers, so the false update-available behavior is not implemented away. The PR is useful source work, but it needs maintainer adaptation because current main moved the parser into a shared ClawHub/plugin SemVer helper and the proposed numeric revision parsing has a low-severity precision issue.

## What This Changes

This PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and same-base prereleases.

## Best Possible Solution

Keep this PR open and adapt it on current main so OpenClaw update/status/startup/update-runner comparisons use correction-release ordering for OpenClaw package versions while preserving generic ClawHub/plugin SemVer behavior. The adapted implementation should compare numeric suffixes without precision loss and cover correction-vs-stable, correction-vs-beta, revision ordering, and the affected update/status callers.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.78

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. Registry-provided versions above `Number.MAX_SAFE_INTEGER` can collapse adjacent revisions to the same JS number and produce the wrong update ordering; keep the suffix as a normalized string, use `BigInt`, or bound the accepted length before comparing.
  - confidence: 0.78
