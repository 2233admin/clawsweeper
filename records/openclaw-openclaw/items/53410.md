## Summary

Keep open. Current main still has the reported update-comparison gap: update checks delegate to the shared generic SemVer comparator, the current regression test still ranks `1.0.0-1` below `1.0.0-beta.1`, and OpenClaw’s release/version policy already treats `YYYY.M.D-N` as a stable correction release. The PR is useful, but it needs maintainer adaptation because current main moved the parser into a shared ClawHub-facing comparator and the proposed numeric revision parsing has a low-severity precision concern.

## What This Changes

This PR changes update-version comparison and regression tests so numeric-only `-N` suffixes rank as OpenClaw correction revisions newer than the same base stable release and same-base prereleases.

## Best Possible Solution

Keep the PR open and adapt it to current main by applying correction-release ordering only to OpenClaw update-version comparisons, or by adding an explicit mode to the shared comparator without changing generic ClawHub/plugin SemVer behavior. Regression coverage should include correction-vs-stable, correction-vs-beta, revision ordering, affected update/status callers, and precision-safe numeric suffix comparison, with changelog credit if maintainers accept the behavior.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.78

Full review comments:

- **[P3] Compare revision suffixes without lossy number coercion:** `src/infra/update-check.ts:405`
  - body: The new revision path stores an arbitrary numeric suffix with `Number.parseInt()`. For registry-provided versions above `Number.MAX_SAFE_INTEGER`, adjacent revisions such as `2026.3.23-9007199254740992` and `2026.3.23-9007199254740993` can collapse to the same JS number and produce the wrong update ordering. Keep the suffix as a normalized string, use `BigInt`, or bound the accepted length before comparing.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a low-severity update-ordering risk by parsing unbounded registry-controlled numeric revision suffixes into JS numbers.

Concerns:

  - body: The proposed `Number.parseInt()` revision comparison can lose precision for very large numeric suffixes, which can make update gating choose the wrong ordering for crafted or compromised registry version strings.
