## Summary

Keep open for maintainer review. Current main still has the silent doctor pre-update path, while this PR is a focused implementation that appears to wire the existing update progress callbacks correctly and addresses the earlier bot review concerns around non-TTY output and cleanup.

## What This Changes

The PR changes `src/commands/doctor-update.ts` so the doctor pre-update flow creates the shared update progress controller, passes its callbacks to `runGatewayUpdate`, gates spinner output on stdout TTY, and stops the controller in a `finally` block.

## Best Possible Solution

Land this focused doctor progress wiring after maintainer review, with narrow tests around callback handoff, non-TTY behavior, and thrown-error cleanup while leaving the shared update runner and progress helper API unchanged.

## Reproduction Assessment

Yes. In a git checkout on current main with interactive stdin, answering yes to the doctor update prompt reaches `src/commands/doctor-update.ts:55`, which prints only a static note and calls `runGatewayUpdate` without progress callbacks.

## Solution Assessment

Yes. Reusing `createUpdateProgress` is the narrowest maintainable fix because the update runner already exposes progress hooks and the regular update command already uses the same controller; focused tests are the main remaining hardening step.
