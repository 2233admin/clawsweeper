## Summary

Current main and the shipped release line no longer take the declaration-emission path that produced the reported TS2883 errors during `pnpm check`; the fix landed as a tsgo wrapper policy rather than the unmerged xai-specific annotation PR.

## What This Changes

The issue reports that `pnpm check` on OpenClaw 2026.4.6 fails during `pnpm tsgo` with TS2883 inferred-type portability errors in xai and bash tool files.

## Best Possible Solution

Keep the shipped tsgo no-declaration check policy as the fix for this `pnpm check` failure; explicit xai return annotations can remain optional cleanup unless a declaration-producing surface needs them.

## Reproduction Assessment

Yes for the original release: the report includes a concrete `pnpm check` log from 2026.4.6 with TS2883 diagnostics. On current main the same failure path is no longer expected because the check path disables declaration transforms before running tsgo; I did not run the broad gate because this was a read-only checkout review.

## Solution Assessment

Yes. For `pnpm check`, disabling declaration transforms in the no-emit tsgo wrapper is the narrow maintainable fix because it prevents this whole class of declaration-portability failures while still allowing explicit declaration builds such as plugin SDK dts generation to opt in.
