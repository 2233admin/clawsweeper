## Summary

Keep this PR open for normal review. Current main still has only the existing Feishu boolean/CardKit streaming path, so the feature is not obsolete, but the proposed patch still leaves a media-only segment delivery path outside the new segment queue and should be revised before merge.

## What This Changes

The PR adds an opt-in Feishu `streamingMode: "segment"` config, implements segment-based reply flushing, and updates the Feishu schema, generated channel metadata, docs, and dispatcher/config tests.

## Best Possible Solution

Land the Feishu segment streaming option after routing every segment-mode outbound payload, including media-only sends, through the same delivery ordering path, then verify the generated config metadata and targeted Feishu dispatcher/config tests while keeping CardKit streaming as the default.

## Reproduction Assessment

Yes. A focused unit test can defer the first `sendMessageFeishu` segment flush from `onToolStart`, then deliver a media-only payload; the PR branch can call `sendMediaFeishu` before the deferred text segment resolves.

## Solution Assessment

No. The feature direction is narrow and plugin-owned, but the current patch is not the best merge shape until media-only payloads share or await the segment delivery queue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Queue media-only sends behind segment flushes:** `extensions/feishu/src/reply-dispatcher.ts:663-664`
  - body: In segment mode, media-only payloads with `shouldDeliverText` false reach this branch and call `sendMediaReplies` without waiting for `segmentDeliveryQueue`. If a tool-start or idle partial flush is still sending, the attachment can arrive before the text segment, breaking the ordering guarantee this mode is adding. Queue or await media sends on the same segment delivery path.
  - confidence: 0.86
