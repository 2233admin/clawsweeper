## Summary

Keep this PR open, but do not merge it as-is. It addresses a real ACPX-managed Codex auth regression, but the current patch still has a security-sensitive copy fallback that can place Codex credentials under the recursively backed-up OpenClaw state directory.

## What This Changes

The PR bridges an existing Codex `auth.json` into the ACPX-managed isolated Codex home, documents that behavior, adds ACPX regression coverage, updates the changelog, and adjusts one Codex plugin test type.

## Best Possible Solution

Keep the host Codex auth reuse direction, but make the bridge credential-safe before merge: prefer symlink-only behavior, move any copy to a credential path with an explicit backup policy, or add a narrow backup exclusion with regression coverage proving copied Codex auth cannot enter default archives.

## Reproduction Assessment

Yes. Current main shows the isolated Codex ACP home lacks auth, and the linked bug reports direct ACPX working with normal Codex auth while OpenClaw-managed ACP fails; the PR finding is reproducible by forcing the symlink path to fail and then running the default backup path inspection.

## Solution Assessment

No. Bridging only `auth.json` while keeping Codex config hooks isolated is the right narrow product direction, but copying the credential into `<stateDir>/acpx/codex-home` is not the safest implementation while backups include state recursively.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep copied Codex auth out of backups:** `extensions/acpx/src/codex-auth-bridge.ts:281`
  - body: When symlinking fails, this fallback writes Codex `auth.json` into `<stateDir>/acpx/codex-home`. `openclaw backup create` archives the state directory recursively and currently only filters plugin `node_modules`, so routine backups can contain Codex credentials. Use symlink-only behavior, a credential-safe location, or an explicit backup exclusion with coverage before merging.
