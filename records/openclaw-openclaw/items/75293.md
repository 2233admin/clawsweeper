## Summary

Keep open: current main only works around the macOS failure by making the guest temp script mode `755`, while this PR aligns the write/chmod path with the same desktop-user transport used to execute the script and restores non-world-readable `700` permissions. I found no blocking review findings; live macOS Parallels validation should still gate merge.

## What This Changes

The PR changes the Parallels npm-update smoke harness so macOS update scripts are written and chmodded through the resolved desktop-user exec transport, restores `700` permissions, and adds a regression test for that transport shape.

## Best Possible Solution

Land this narrow PR after macOS Parallels npm-update smoke and the changed-file gate pass, keeping the Linux root-owned path separate and preserving non-world-readable temp scripts for provider credential material.

## Reproduction Assessment

Partly: the exact permission-denied regression existed in `af5a1fbd`, but current main changed the temp script mode to `755`, so that symptom is no longer expected on main. The remaining issue is source-verifiable, and the live reproduction/validation path is `pnpm test:parallels:npm-update -- --platform macos` on a Parallels host.

## Solution Assessment

Yes. Passing the resolved macOS exec args into the script writer is narrower and safer than the current `755` workaround because it keeps writer, chmod, and runner identities aligned while restoring `700` permissions.
