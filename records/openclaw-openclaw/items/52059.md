## Summary

Keep this PR open. Current main still does not implement `gateway.auth.scopes`: the config type/schema reject the key, resolved gateway auth does not carry configured scopes, and the WebSocket handshake still clears device-less token/password operator scopes without a configured regrant. The PR remains a plausible gateway feature, but it is stale against the current `auth-resolve.ts` and `ws-connection` split and still needs security-tight gating plus regression coverage.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is to rebase onto current main, add `gateway.auth.scopes` through the config type/schema/help/generated docs surfaces, carry a trimmed configured scope list through `auth-resolve.ts`, restore those scopes only for device-less sessions whose actual auth method is token or password, and add regression coverage for configured regrant, default cleared scopes, trusted-proxy non-elevation, and Tailscale non-elevation.

## Risks / Open Questions

- The PR touches gateway authentication and would grant operator scopes to sessions without paired device identity, so it needs focused tests for configured token/password regrant, absent scopes, trusted-proxy non-elevation, and Tailscale/header-auth non-elevation before merge.
- The current pull ref still contains the original patch shape, including user-facing help text with an internal `clearUnboundScopes` name and a regrant guard that is too broad for trusted-proxy sessions in the current auth model.
- The patch is stale against current main's `auth-resolve.ts` resolver split; applying it mechanically to `auth.ts` would miss the live resolver path.
