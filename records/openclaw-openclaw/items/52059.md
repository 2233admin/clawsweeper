## Summary

Keep this PR open. The underlying feature is still missing from current main, but the submitted patch is stale against the current gateway auth resolver split and has a concrete security issue because it regrants configured scopes through `sharedAuthOk`, which current main also sets for trusted-proxy auth.

## What This Changes

This PR adds optional `gateway.auth.scopes` config metadata and WebSocket handshake plumbing to regrant configured scopes to device-less token/password-authenticated connections.

## Best Possible Solution

Rebase the feature onto current main, add `gateway.auth.scopes` through the current config type, strict schema, labels/help, docs/generated config surface, and `auth-resolve.ts`, then restore configured scopes only when the actual auth method is `token` or `password`. The final patch should include focused gateway auth tests proving the default clear behavior remains and that trusted-proxy or Tailscale paths cannot receive token/password scopes.

## Reproduction Assessment

Yes. Current-main code and tests provide a high-confidence path: connect as an operator with a valid shared token or password and no device identity, request privileged scopes, then observe the connected session has cleared scopes and privileged RPCs fail with a missing-scope error.

## Solution Assessment

No. The requested direction may be maintainable, but this submitted patch is not the best current solution because it targets the old resolver shape and gates regranting on `sharedAuthOk`, which includes trusted-proxy auth; the safer fix is an explicit `authMethod === "token" || authMethod === "password"` gate after rebase.
