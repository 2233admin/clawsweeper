## Summary

Keep open. Current main still has no `gateway.auth.scopes` config surface, and the gateway tests still assert that device-less shared-token connections connect with cleared scopes and then fail privileged RPCs with `missing scope`. The PR is not implemented or obsolete; it remains a plausible gateway auth change, but needs a rebase plus focused security tests before it can land.

## Best Possible Solution

Keep this PR open for maintainer review or contributor follow-up. The best path is a rebased implementation that adds `gateway.auth.scopes` to the current config types, strict Zod schema, labels/help, and `auth-resolve.ts`; reapplies configured scopes only after unbound scopes are cleared and only for token/password auth; and adds regression coverage for configured scopes, absent scopes, and trusted-proxy/Tailscale non-elevation.

## Risks / Open Questions

- The change is security-sensitive because it grants operator scopes to sessions without paired device identity; it must be limited to authenticated token/password sessions and must not trust caller-declared scopes.
- The PR's described files predate current main's `auth-resolve.ts` split, so it likely needs a rebase into the current auth resolution path.
- No tests were executed because this was a required read-only review; the decision is based on source, tests, docs, provided PR discussion, and history inspection.
