## Summary

Keep this PR open. Current main still does not implement the PR’s central behavior: non-alpha under-cap PNGs still flow into JPEG optimization, and filename normalization for JPEG output is still limited to HEIC sources. The PR remains a useful implementation candidate, but it needs a rebase onto the active shared media loader and should address the existing PNG payload-validation concern before landing.

## Best Possible Solution

Rebase the useful parts onto `src/media/web-media.ts` and the current media tests. Preserve verified under-cap PNGs only after decoding or otherwise validating PNG bytes, normalize filenames whenever the actual optimized output is JPEG, and add regression coverage in the shared loader plus the WhatsApp-facing media path.
