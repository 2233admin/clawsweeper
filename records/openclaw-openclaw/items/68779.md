## Summary

Keep open. Current main has partial Chrome MCP hardening, but it does not satisfy the issue or the maintainer comment: attach/readiness budgets remain fixed in the implicated path, transport/timeouts still fail the active operation after resetting the cached session, and existing-session action limitations are still explicitly documented and enforced.

## What This Changes

The issue requests more reliable browser existing-session automation by relaxing timeout budgets, adding reconnection/recovery, exposing diagnostics/configuration, and covering multi-step user-profile flows.

## Best Possible Solution

Keep this issue open and implement a browser-plugin fix that gives Chrome MCP existing-session attach/readiness a defensible budget, refreshes or reconnects sessions without dropping supported multi-step workflows, preserves navigation safety, and adds regression coverage for a profile="user" flow that survives selected-page/session churn.
