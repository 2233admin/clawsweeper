## Summary

Keep this PR open. Current main still captures Telegram sticker `fileId` in ingress/session context, but `buildInboundUserContextPrefix()` does not project any sticker handle into the agent-visible conversation-info block. The PR is therefore not obsolete. The remaining review issue is contract alignment: the PR exposes `sticker_file_id`, while current Telegram sends accept `fileId` or `stickerId`, so maintainers should decide whether to rename the prompt key or add/document an accepted alias.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is a narrow fix that surfaces a replayable Telegram sticker handle in per-turn conversation info while aligning the exposed key with the accepted send contract, either by using `fileId`/`stickerId` in the prompt context or by adding a documented `sticker_file_id` alias to the shared schema and Telegram runtime. Keep the focused regression coverage for stable-system-metadata exclusion, direct webchat omission, and Telegram inclusion.

## Risks / Open Questions

- As written, a model may copy `sticker_file_id` into a Telegram sticker send call, but current main accepts `fileId` or `stickerId`; sticker replay can still fail unless the exposed prompt key and send contract are aligned.
- Exposing Telegram sticker file IDs to the model is a data-scope expansion. It should stay in untrusted per-turn conversation info and remain excluded from stable system metadata to avoid cache churn and misplaced trust.
