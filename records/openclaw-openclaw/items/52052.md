## Summary

Keep open. Current main still does not implement this PR's intended behavior: Telegram sticker file IDs are captured on inbound context, but `buildInboundUserContextPrefix()` does not surface a replayable `sticker_file_id` or equivalent in agent-visible conversation info. The unresolved review concern also still applies because the send path accepts `fileId`/`stickerId`, not `sticker_file_id`.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to update or land a narrow fix that exposes the replayable sticker handle in per-turn conversation info while aligning the surfaced key with the send tool contract, either by using an accepted key (`fileId`/`stickerId`) or by adding a documented `sticker_file_id` alias in both the shared schema and Telegram runtime. Keep regression tests for system-metadata exclusion, webchat omission, and Telegram inclusion.

- As written, the PR may not reliably enable sticker echo because it surfaces `sticker_file_id` while the current Telegram send tool accepts `fileId` or `stickerId`.
