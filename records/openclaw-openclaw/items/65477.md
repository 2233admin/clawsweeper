## Summary

Keep open. Current main still builds the per-response usage footer from token/cache/cost/session data only; the runner has the actual provider/model available, but does not pass that label into the footer formatter. Existing `/status` model display and `messages.responsePrefix` interpolation are separate surfaces, and the related implementation PRs are still open rather than merged.

## What This Changes

The issue requests showing the active provider/model identifier in the per-response `/usage` footer shown in messaging channels.

## Best Possible Solution

Land or consolidate a narrow implementation that formats the actual answering provider/model with the existing provider/model label helper, passes it into `formatResponseUsageLine`, preserves existing cost/cache/session behavior, and updates focused formatter, runner, and token-use documentation coverage.
