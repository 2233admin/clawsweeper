## Summary

Keep open. Current main still has the reported failure shape: `chunkMarkdown` has a fixed-width outer split that can cut between UTF-16 surrogate halves, while the embedding path forwards text without a lone-surrogate sanitizer. An open linked fix PR, #65783, explicitly uses closing syntax for this issue, so the issue should stay open until that PR is reviewed and merged or replaced.

## What This Changes

Fix memory indexing so chunking and embedding input handling do not emit lone UTF-16 surrogates that can permanently stall embedding reindexing.

## Best Possible Solution

Review, rebase, and land or replace PR #65783 with the same narrow shape: guard the outer coarse chunk boundary, sanitize lone surrogates immediately before embedding provider calls, and add regression tests for the mostly-Latin emoji boundary plus embed-boundary sanitizer behavior.

## Reproduction Assessment

Yes. A source-level reproduction is clear: choose a mostly-Latin line where `maxChars = chunking.tokens * 4` lands between an emoji high and low surrogate, for example 39 ASCII characters followed by an emoji with a 10-token chunk budget; the current outer loop slices at that boundary without checking the pair.

## Solution Assessment

Yes. The best fix direction is the linked PR's two-layer approach: preserve surrogate pairs in the root chunker path and also replace lone surrogates at the embedding boundary so future ingestion paths cannot send invalid UTF-8 payloads to providers.
