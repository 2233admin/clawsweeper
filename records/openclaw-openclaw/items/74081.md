## Summary

Keep open. Current main does not implement JSON-array env coercion, and the PR is a real behavior change to the config contract rather than obsolete cleanup; it needs maintainer review plus a write-back preservation fix before merge.

## What This Changes

The PR updates config env substitution so a config value that is exactly `${VAR}` can resolve to a `string[]` when the env value is a JSON string array, and adds boundary tests for that behavior.

## Best Possible Solution

If maintainers want this capability, land it as an explicit config-contract change: define strict whole-placeholder semantics, preserve authored env refs on unchanged write-back, document JSON-array env injection, add a changelog entry, and keep tests focused on substitution plus config write-back behavior.

## Reproduction Assessment

Yes. Current main can be checked by calling `resolveConfigEnvVars({ allowFrom: "${ALLOW_FROM}" }, { ALLOW_FROM: '["alice"]' })`, which follows the string-only branch; the PR's added tests encode the requested array result. The write-back issue is reproducible by pairing an authored string `${ALLOW_FROM}` with an incoming resolved array, which falls through the mismatched-type preservation path.

## Solution Assessment

No. The parser/test direction is useful, but applying array coercion globally without write-back preservation, docs/changelog coverage, and a maintainer decision on path scope is not the best merge shape.
