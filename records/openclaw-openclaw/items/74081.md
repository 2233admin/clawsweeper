## Summary

Current main does not implement this PR's intended behavior: `${VAR}` config substitution still returns strings only, and the public docs still document string substitution. The PR is narrow, but it changes the global config substitution contract, has failing checks, lacks docs/changelog coverage, and appears to need maintainer review around write-back preservation and the whitespace-tolerant opt-in.

## Best Possible Solution

Keep this PR open for maintainer review. If maintainers want this feature, the best path is to decide whether coercion should be global or limited to known list-typed config paths, update write-back preservation for whole-env-ref arrays, add docs/changelog coverage, resolve the whitespace behavior, and get targeted config tests plus the changed gate green.

## Risks / Open Questions

- Global whole-value coercion is a config-contract change: any exact `${VAR}` whose env value is a JSON string array can become `string[]`, not only known list-valued paths such as allowlists.
- The PR does not update `env-preserve`, so an env-backed list may be materialized into config on write-back instead of preserving the original `${VAR}` reference.
- Docs and changelog are missing for a user-visible config/env behavior change.
- PR checks are failing, including core and boundary checks.
- The whitespace-tolerant opt-in (`"  ${VAR}  "`) is intentional in the PR tests but remains a product/contract decision for maintainers.
