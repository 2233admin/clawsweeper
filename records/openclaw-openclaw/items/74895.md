## Summary

Keep this PR open for maintainer/author revision. The patch weakens Telegram runtime token resolution, while current main already has the safer read-only accessor pattern for the linked status/allowlist path and an existing regression test that requires unresolved non-env runtime SecretRefs to throw.

## What This Changes

The PR removes Telegram's strict non-env SecretRef check from `resolveRuntimeTokenValue`, so unresolved file/exec/vault bot token refs return `configured_unavailable` instead of throwing during token resolution.

## Best Possible Solution

Keep runtime Telegram token resolution strict for non-env SecretRefs, and keep read-only status/setup metadata on `inspectTelegramAccount` or config-accessor paths that can report `configured_unavailable` without dereferencing runtime credentials.

## Reproduction Assessment

Yes. The PR regression is reproducible by the existing `extensions/telegram/src/token.test.ts` case for unresolved non-env `botToken` SecretRefs, which would no longer throw after this diff; the linked status shape is also covered by the current `shared.test.ts` read-only accessor test.

## Solution Assessment

No. This is not the best fix because it changes the runtime resolver to solve a read-only inspection symptom; the narrower maintainable path is to keep runtime strict and ensure status/allowlist metadata never falls back to runtime credential resolution.
