## Summary

Keep open for maintainer review. Current main still lacks the guarded source-require-to-Jiti fallback for bundled public-surface source artifacts, while the PR is a focused implementation with no blocking static-review findings; the remaining decision is whether maintainers accept this loader execution-path behavior.

## What This Changes

The PR adds a guarded Jiti fallback for bundled source public-surface artifacts with emitted-style `.js` imports, extracts bundled entry load-profile emission, and updates focused plugin loader/runtime-deps/plugin tests.

## Best Possible Solution

Have a maintainer review the guarded fallback as the intended plugin loader boundary, keep the focused regression tests, and land this PR if that boundary is accepted; do not close it as implemented on main while the fallback is still absent from current main.

## Reproduction Assessment

Yes. Current main can resolve a requested `.js` public artifact to a source `.ts` file and then call `sourceArtifactRequire` directly; a bundled source artifact such as `extensions/anthropic/provider-policy-api.ts` imports `./config-defaults.js`, which is the emitted-style sibling import pattern the PR models.

## Solution Assessment

Yes. The proposed approach preserves boundary validation and the direct source-require fast path, then retries through the existing Jiti loader only for narrowly guarded module-not-found errors; a broader retry or removal of the fast path would be less targeted.
