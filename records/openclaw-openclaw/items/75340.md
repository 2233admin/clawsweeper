## Summary

Keep open. Current main still matches the reported packaged CLI/status failure mode: plain text status keeps the channel table on the fast metadata path, but that same path can skip both setup-runtime fallback and packaged runtime-deps registration before bundled channel setup/status modules are imported.

## What This Changes

Ensure packaged `openclaw status --usage` can show configured bundled channel rows without manual `NODE_PATH` or requiring `openclaw status --deep`.

## Best Possible Solution

Keep plain status fast and read-only, but make configured bundled channel rows available in packaged installs by either using lightweight manifest/setup metadata that does not import runtime-dependent modules, or by preparing the active staged bundled runtime-deps root before the existing bundled setup/status imports. Add focused regression coverage and a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: packaged `openclaw status --usage` with configured bundled channels whose dependencies exist only under the staged runtime-deps root follows a path that disables setup fallback and imports bundled setup/status code with runtime-deps installation disabled.

## Solution Assessment

Yes, directionally. The reporter's suggested runtime-deps registration path is plausible, but the best maintainable fix should stay scoped to status/read-only channel resolution and avoid expanding plain status into deep/live probing.
