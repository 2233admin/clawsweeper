## Summary

Keep this open. The report is recent, specific, and includes useful external evidence that the CLI stalls after the gateway sends `connect.challenge`, but current source inspection only proves the nominal handshake path and related post-release timeout/readiness mitigations; it does not prove the reported package.json walk loop is fixed or obsolete.

## What This Changes

The issue reports that CLI RPC commands can time out after receiving a gateway `connect.challenge` but before signing and returning the device nonce, apparently while repeatedly reading the installed `openclaw/package.json`.

## Best Possible Solution

A maintainer should reproduce the reporter's CLI command against current main or the next release with the full strace, identify whether the stall is from Gateway client readiness, version/package-root resolution, or bundled plugin runtime-deps loading, then either give a concrete state-repair/update answer or land a focused regression test and fix on the pre-auth CLI path.
