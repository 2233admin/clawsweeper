## Summary

Keep open. Current main still resolves media-understanding provider auth and then requires an API key before invoking the audio provider, so the reported no-auth/local STT failure is still plausible from source. There is also an open, unmerged closing PR, #75005, so the issue should remain open until that implementation candidate is reviewed, landed, or rejected.

## What This Changes

Allow media-understanding audio/STT providers that intentionally run locally or without credentials to execute without failing in the model-provider API-key resolver.

## Best Possible Solution

Review #75005 or an equivalent narrow patch that adds an explicit provider-owned synthetic/no-auth media execution hook, covers local/no-auth audio and video behavior in media-understanding tests, documents the plugin contract, and keeps unmarked remote providers fail-closed.

## Reproduction Assessment

Yes. The reported minimal reproduction maps directly onto current main: a media audio provider without a literal or resolvable API key reaches `resolveApiKeyForProvider` and `requireApiKey` before `transcribeAudio(...)` is called.

## Solution Assessment

Yes at the direction level. An explicit opt-in for provider-owned synthetic/no-auth media execution is the narrow maintainable path because it preserves the existing auth gate for normal remote providers while allowing local/self-hosted providers to declare their intent.
