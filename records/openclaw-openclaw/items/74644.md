## Summary

Keep open. Current main still resolves media-understanding audio/video provider auth before calling the plugin provider and requires a non-empty API key, while the open linked PR #75005 explicitly targets this bug with closing syntax and is not merged yet. The m13v comment independently confirms the same auth-gate failure mode, so this should stay open until that PR is reviewed, landed, or replaced.

## What This Changes

The issue requests an explicit local/no-auth execution path for media-understanding providers so plugin-owned STT providers can transcribe without auth-store API keys.

## Best Possible Solution

Review and land, close, or replace PR #75005. The desired end state is a narrow core/SDK media-provider auth seam that lets local or self-hosted providers opt into synthetic non-secret auth while preserving the existing fail-closed behavior for ordinary remote providers.

## Reproduction Assessment

Yes. The issue's plugin-only STT setup maps directly to the current code path: audio provider execution resolves model-provider auth and requires an API key before `transcribeAudio` is called.

## Solution Assessment

Yes, the best direction is an explicit opt-in synthetic/no-auth media-provider hook checked before the `requireApiKey` gate. That is narrower and safer than treating every missing key as local/no-auth.
