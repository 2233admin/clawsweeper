## Summary

Keep open. Current main still resolves media audio provider auth through the model-provider API-key path before calling the plugin's `transcribeAudio`, and the linked closing PR #75005 is open and unmerged, so this issue should stay open until that implementation is reviewed and landed or rejected.

## What This Changes

Allow explicitly local or no-auth `mediaUnderstandingProviders` audio/STT providers to run without requiring auth-store API key credentials.

## Best Possible Solution

Review and land a narrow media-provider opt-in for synthetic/no-auth execution, with tests proving local audio providers reach `transcribeAudio` without configured credentials while ordinary unmarked providers still require auth.

## Reproduction Assessment

Yes. The static source path gives a high-confidence reproduction: configure a plugin media audio provider with no literal/configured/auth-store key, then the runner reaches `requireApiKey` before invoking the provider's `transcribeAudio`.

## Solution Assessment

Yes in direction. An explicit opt-in synthetic/no-auth media-provider hook is the narrow maintainable fix because it avoids weakening the default auth gate for normal remote providers.
