## Summary

Keep open. Current main still resolves audio/video media-provider execution through the model-provider auth path and calls `requireApiKey` before invoking plugin media callbacks, while #75005 is an open closing PR that specifically attempts to fix this behavior.

## What This Changes

The issue asks for local or no-auth media understanding audio/STT provider plugins to run without being blocked by API-key credential resolution.

## Best Possible Solution

Review and land #75005, or an equivalent narrow fail-closed media-provider opt-in, with regression coverage proving marked local audio providers reach `transcribeAudio` without credentials while unmarked providers still fail closed.

## Reproduction Assessment

Yes. The source path is enough to reproduce with high confidence: configure an audio media provider without a literal/configured/auth-store credential and the runner reaches `requireApiKey` before invoking `transcribeAudio`.

## Solution Assessment

Yes in direction. An explicit synthetic/no-auth opt-in for media providers is the narrow maintainable fix because it preserves the existing auth gate for ordinary remote providers while allowing local providers to declare no-auth execution.
