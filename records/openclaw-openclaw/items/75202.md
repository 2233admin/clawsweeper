## Summary

Keep this PR open. Current main still has the reported shared MEDIA parser behavior and the linked bug has a concrete reproduction, but the proposed regex should not merge as-is because it can silently truncate valid media sources that contain normal punctuation after a file extension.

## What This Changes

The PR adds a MEDIA parser cleanup pass that truncates candidates at a file extension before JSON-looking trailing characters and adds a regression table case for a polluted image path.

## Best Possible Solution

Keep the fix in the shared `src/media` parser, but make suffix stripping distinguish serialized JSON wrappers from ordinary path or URL punctuation and cover both the polluted JSON case and valid punctuation-in-path cases.

## Reproduction Assessment

Yes. The linked bug provides a concrete single-line `MEDIA:/path/to/image.png"}],...` input, and current main's line-wide token capture plus outer-only cleanup explains how that becomes a polluted local media path.

## Solution Assessment

No, not as written. Parser-side hardening is the right layer, but the delimiter should be tightened so normal media source characters are preserved while serialized JSON tails are removed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve valid path punctuation after extensions:** `src/media/parse.ts:39`
  - body: The new regex treats `,` and `|` immediately after any file extension as serialized JSON and returns only the prefix. Since `MEDIA:` accepts path-or-url sources, valid values like `MEDIA:/tmp/render.png,final` or `MEDIA:https://example.com/a.png,final` would be silently rewritten to a different source and can deliver the wrong file or fail with ENOENT. Please tighten the delimiter to actual JSON wrapper syntax or add context before truncating.
  - confidence: 0.87
