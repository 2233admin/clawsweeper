## Summary

Keep this draft PR open: current main does not already implement the WebChat terminal backstop, and the proposed patch targets a real gap, but the diff introduces a timeout contract regression that should be fixed before merge.

## What This Changes

The PR adds a WebChat-only terminal timeout backstop for `chat.send`, creates missing transcript files before eager transcript updates, and adds a focused gateway regression test.

## Best Possible Solution

Keep the terminal backstop in the Gateway WebChat `chat.send` path, but preserve the established timeout semantics before merge: positive requested values should be honored, explicit zero should remain no-timeout behavior, and the regression test should cover both the positive timeout and zero/no-timeout cases.

## Reproduction Assessment

Yes. The PR's test reproduces a hung dispatch by waiting for the abort signal, and the review finding is reproducible by sending WebChat `chat.send` with `timeoutMs: 0` while the dispatch remains active: the proposed helper falls through to the 30s default instead of preserving the no-timeout sentinel.

## Solution Assessment

No, not as written. The Gateway-side backstop is a reasonable place to solve the stuck terminal event, but the timeout resolver should use the existing `timeoutMs: 0` contract instead of silently converting it to a finite WebChat timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve timeoutMs: 0 as no timeout:** `src/gateway/server-methods/chat.ts:1654`
  - body: `ChatSendParamsSchema` allows `timeoutMs: 0`, and `resolveAgentTimeoutMs` explicitly treats `overrideMs: 0` as the no-timeout sentinel. This helper ignores that explicit zero and falls through to the 30s WebChat default, so clients that intentionally disable the run timeout would now receive a terminal timeout error after 30 seconds. Please special-case zero consistently before arming the backstop.
  - confidence: 0.87
