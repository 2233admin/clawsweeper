## Summary

Keep this issue open. Current main still has targeted async cleanup guards and plugin cleanup callbacks, but no shared ResourceManager-style shutdown protocol across retry/tool, cleanup, and plugin/channel paths. The Feishu cleanup race cited in the issue still matches current code, and the related retry/tool PR remains open rather than merged.

## Best Possible Solution

Keep this open as the broad lifecycle-design tracker. Maintainers should first review or close PR #48512 for the retry/tool-call cluster, fix the Feishu HTTP cleanup path directly, and then decide whether a small shared core or Plugin SDK lifecycle contract is warranted. A good design should define stop-accepting-new-work, drain or cancellation behavior, timeout and forced-dispose semantics, tests for retry and server-close races, and clear owner boundaries before migrating call sites.
