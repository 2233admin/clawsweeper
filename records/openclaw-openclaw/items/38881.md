## Summary

Keep this PR open. Current main still does not implement the PR's intended final-reply model switch: imageModel is wired for media-understanding/image-tool fallback, and WebChat now routes text-only image attachments into MediaPaths for that preprocessing path, but the final runPreparedReply call still receives the normal selected provider/model. The PR is useful but not merge-ready because it is dirty against current main and the latest review comments identify unresolved routing semantics around media-understanding ordering, attachment type detection, URL media, unknown catalog entries, and imageModel fallbacks.

## What This Changes

The PR adds a resolveImageModelAutoSwitch helper in src/auto-reply/reply/get-reply.ts and a new Vitest file to switch the reply provider/model to agents.defaults.imageModel.primary when image media is detected and the selected model lacks vision support.

## Best Possible Solution

Keep this PR open for maintainer review or a carefully rebased replacement. The best fix should live in the reply/model-selection path, use current attachment normalization and type metadata, resolve imageModel primary plus fallbacks through the real alias/catalog helpers, switch before media-understanding or explicitly coordinate to avoid duplicate analysis, require positive evidence that the active model cannot accept images, respect heartbeat/session/channel/user overrides, and add focused regressions proving the final run receives the vision model when intended.

## Risks / Open Questions

- The PR head is currently dirty against main, so it needs rebase/conflict resolution before normal validation can be meaningful.
- The proposed switch happens after media-understanding in the PR diff, which can duplicate image analysis or produce competing image context when the final run is switched to a vision model.
- The PR's image detection is too broad and too narrow at the same time: it treats any MediaPath/MediaPaths entry as an image, misses MediaUrl/MediaUrls, and does not consult MediaTypes.
- The implementation only considers imageModel.primary, while current config supports imageModel.fallbacks and fallback-only image-model setups.
- The implementation treats an unknown current catalog entry as not vision-capable because modelSupportsVision(undefined) is false, which could override custom/new models that actually support image input.
