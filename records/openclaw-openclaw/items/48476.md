## Summary

Keep this PR open. Current main still does not implement the PR's intended launchd install resilience: `bootstrapLaunchAgentOrThrow()` performs a single `launchctl bootstrap` against `gui/<uid>` and immediately throws GUI-session guidance for unsupported GUI domains, with no duplicate-registration retry and no `launchctl load -w` fallback. The linked bug was closed by stale automation rather than a shipped fix. The PR is small and relevant, but it needs maintainer review because the proposed legacy `launchctl load` fallback changes macOS launchd domain/session behavior and has unresolved review concerns around sudo/root targeting and exit-code reliability.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase it onto current main, preserve or split out the low-risk duplicate-registration retry, and only keep the headless/sudo `launchctl load` fallback if it verifies the actual loaded domain/session and parses load diagnostics before reporting success. If maintainers decide headless LaunchAgent fallback is out of scope, narrow the PR to reinstall recovery and leave headless/sudo as documented guidance or a separate LaunchDaemon design discussion.
