## Summary

Keep open. Current main serializes Slack sends per account/token/recipient/thread, but it still does not add the requested configurable minimum-delay throttle needed to force distinct Slack timestamps; the related broad outbound rate-limiting PR is closed unmerged.

## What This Changes

Request a Slack-specific configurable outbound send throttle so rapid channel messages, thread replies, and DMs are spaced far enough apart for Slack timestamp ordering.

## Best Possible Solution

Add a focused Slack account-level throttle setting with strict schema/types/UI hints/docs and apply one shared delay helper consistently across Slack normal sends, DMs, thread replies, block/presentation sends, chunked text, media-caption follow-ups, and slash response_url follow-ups.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call sendMessageSlack twice rapidly, or send a chunked Slack reply with a fake Slack client, and observe that current main serializes calls without any timer-enforced minimum delay; live Slack display-order proof would be useful but is not required to confirm the missing throttle.

## Solution Assessment

Yes. A Slack-owned configurable delay in the Slack plugin send paths is the narrow maintainable fix; the closed broad multi-channel throttle PR was more invasive than this Slack timestamp-ordering issue requires.
