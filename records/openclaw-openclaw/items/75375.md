## Summary

Keep open. Current main already fires the internal startup hook on a delayed, unawaited timer, but the bundled boot-md handler still awaits BOOT.md agent work inside the gateway process, and that path bypasses the gateway agent RPC accepted/yield dispatch that exists specifically to avoid pre-turn runner event-loop monopolization. The report is recent, concrete, and distinct from the earlier boot-md delivery issue.

## What This Changes

The issue reports that enabling bundled boot-md can make gateway restarts appear offline because BOOT.md agent work runs during startup and monopolizes the gateway event loop.

## Best Possible Solution

Make boot-md launch BOOT.md work only after the gateway is ready and through a detached/background agent-run path with an event-loop yield and bounded cancellation/logging, while preserving per-agent workspace dedupe and the existing outbound message-tool behavior.

## Reproduction Assessment

Yes. Enable boot-md with a BOOT.md that performs a slow provider/tool/network operation, restart the gateway, and attempt UI/Telegram or health/ready probes while boot runs; current source shows the boot path awaits in-process agentCommand work that can monopolize the gateway event loop.

## Solution Assessment

No, current main is not the complete fix. The unawaited startup-hook timer helps with startup ordering, but the maintainable solution should reuse or extract the gateway agent dispatch pattern that accepts/yields before heavy runner setup and tracks/bounds detached work.
