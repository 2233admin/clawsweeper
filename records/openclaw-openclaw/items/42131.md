## Summary

Keep this PR open. The PR targets a real safe-bin trust/doctor bug: current main still lowercases configured `safeBins` in doctor while storing and comparing `safeBinTrustedDirs` with exact resolved casing, so the mixed-case false warning remains plausible on case-insensitive filesystems. This is not implemented on main, and the PR itself still needs maintainer review because the latest temp-dir filesystem probe may be too broad for mixed-sensitivity mounts.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best path is a shared, testable safe-bin path comparison helper that compares configured trusted dirs and resolved executable dirs consistently on genuinely case-insensitive paths, preserves distinct paths on case-sensitive or mixed-sensitivity volumes, and adds deterministic regression coverage for doctor/runtime mixed-case matching plus the case-sensitive sibling trust boundary.

## Risks / Open Questions

- Closing this PR now would discard a focused candidate for a current core exec-policy bug on case-insensitive filesystems.
- The PR's latest temp-dir-wide case-sensitivity detection can weaken the safe-bin trust boundary on mixed-sensitivity systems, for example when `os.tmpdir()` is case-insensitive but a configured trusted directory is on a case-sensitive volume.
- The added tests mirror the runtime probe, so they may not deterministically prove both the intended case-insensitive match and the case-sensitive sibling non-trust boundary.
