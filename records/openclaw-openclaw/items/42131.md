## Summary

Keep this PR open. Current main still stores safe-bin trusted directories with exact resolved casing while doctor lowercases configured safeBins before resolving them, so the reported mixed-case false warning remains plausible on case-insensitive filesystems. The affected surface is core exec safe-bin trust for `openclaw doctor` and runtime policy, not a ClawHub or external item. The PR is a focused implementation candidate, but its latest global temp-dir filesystem probe still needs trust-boundary review.

## Best Possible Solution

Keep this PR open for maintainer review or a small revision. The best fix is a shared, testable safe-bin path comparison helper that compares configured trusted dirs and resolved executable dirs consistently on genuinely case-insensitive paths, preserves distinct paths on case-sensitive or mixed-sensitivity volumes, and adds deterministic regression coverage for doctor/runtime mixed-case matching plus a trust-boundary test for case-sensitive siblings.

## Risks / Open Questions

- Closing this PR would drop a plausible core exec-policy bug fix that current main still does not implement.
- The latest PR implementation may weaken the safe-bin trust boundary on mixed-sensitivity systems where `os.tmpdir()` is case-insensitive but a configured trusted directory is on a case-sensitive volume; distinct sibling paths can collapse to one lowercase key.
- The added PR tests mirror the runtime filesystem probe, so they may not deterministically prove both the intended case-insensitive match and the case-sensitive sibling non-trust boundary.
