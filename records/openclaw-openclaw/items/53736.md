## Summary

Keep open. Current main still has no `tools.exec.denyPatterns` surface, and this PR is an active implementation touching exec policy plus public config schema rather than obsolete cleanup. It needs maintainer product/security review because the proposed regex contract overlaps the broader security-labeled denylist design in #71097 and still has a concrete segment-matching weakness for ordinary chained shell commands.

## What This Changes

The PR adds a global and per-agent `tools.exec.denyPatterns` config/schema field, threads it into exec defaults and chat-bash, rejects matching commands in `createExecTool`, and updates generated config artifacts plus deny-pattern tests.

## Best Possible Solution

Keep the PR open until a maintainer chooses the exec policy contract: either land a rebased, documented soft guardrail with explicit matching limits and segment-aware tests, or fold the work into #71097 so deny-pattern matching, actions, defaults, and approval behavior are designed together.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Match deny patterns against shell command segments:** `src/agents/bash-tools.exec.ts:260`
  - body: The deny guard tests each regex only once against `params.command.trim()`. With the documented patterns like `^claude\s`, a normal generated command such as `cd repo && claude --print hi` does not match even though the denied CLI still runs, so the guardrail misses the stated coding-CLI steering case. Reuse the existing shell-chain analysis or otherwise evaluate each executable segment before execution.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff adds an exec-policy guardrail, but its full-command regex matching can be bypassed by ordinary shell chaining and may be mistaken for stronger command blocking.

Concerns:

- **[medium] Deny patterns miss chained command segments:** `src/agents/bash-tools.exec.ts:260`
  - body: Because matching is applied once to the whole trimmed command string, start-anchored deny rules can be bypassed by `cd ... && denied-binary` or similar wrappers while still executing the denied command. That matters for a security-adjacent public config field advertised as blocking exec commands by regex.
  - confidence: 0.84
