## Summary

Current main still follows the reported duplication path: the main-session cron job queues the system event with a cron context key, and heartbeat still turns that same queued cron event into a scheduled-reminder prompt even when user delivery is disabled. The linked implementation PR remains the right next review target rather than closing this issue.

## What This Changes

The issue requests that main-session cron systemEvent jobs with delivery.mode set to none inject only the system event and suppress the redundant scheduled-reminder heartbeat wrapper.

## Best Possible Solution

Land PR #44973 or an equivalent narrow fix that marks main-session cron systemEvent injections and excludes only those already-injected events from cron reminder wrapper construction, with regression coverage for sessionTarget main, payload.kind systemEvent, and delivery.mode none.

## Reproduction Assessment

Yes. The report provides a concrete cron configuration, and current-main source inspection shows the same text can be both rendered as a queued system event and wrapped by the cron heartbeat prompt because of the `cron:` context key.

## Solution Assessment

Yes. The tag-and-skip direction is the narrowest maintainable fix because it deduplicates prompt construction for main-session injected cron events without changing delivery semantics or adding broad outbound dedupe.
