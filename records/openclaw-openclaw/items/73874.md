Keep open. The latest reporter update still reproduces the Windows Docker event-loop stall on v2026.4.29 after adopting the named runtime-deps volume, and current main only shows adjacent mitigations/diagnostics rather than evidence that the blocking startup or periodic work has been removed.

## What This Changes

The issue asks maintainers to investigate and fix a Windows 11 Docker Desktop gateway regression where HTTP/WS dispatch, Slack delivery, and probes stall behind long Node event-loop blocks after v2026.4.24.

## Best Possible Solution

Keep this as an active Windows Docker gateway regression investigation. The desired end state is a profiled root-cause fix that removes or offloads the blocking startup/periodic work, while preserving the named runtime-deps volume guidance and the existing gateway liveness diagnostics.

## Reproduction Assessment

Yes. The reporter provided a concrete Windows 11 Docker Desktop compose setup, version comparisons through v2026.4.29, and measured event-loop delay logs; this review did not independently run that Windows environment.

## Solution Assessment

Unclear. The current diagnostics, named-volume runtime-deps layout, and readiness/probe mitigations are useful, but the best fix should be based on a CPU/profiled blocking stack rather than only adding wider request timeouts or readiness waits.
