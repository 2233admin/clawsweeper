## Summary

Keep open. The report is recent, detailed, and the reporter's follow-up narrows the Docker bind-mount/runtime-deps part: the docs-recommended named volume improved startup but did not fix the post-ready HTTP/WS dispatch failures. Current main still has adjacent probe, handshake, Bonjour, and runtime-deps improvements, but I did not find source or test evidence that normal HTTP routes, internal WebSocket RPC, or Slack's runtime dispatch path are protected from a never-resolving handler or the reported event-loop starvation.

## Best Possible Solution

Keep this open as a current gateway/Docker regression investigation. Reproduce on Windows 11 Docker Desktop against v2026.4.23, v2026.4.26, and current main using both the original bind-mounted runtime-deps shape and the docs-recommended named-volume shape. If reproduced, split the work: add bounded diagnostics and timeout handling for normal HTTP stages and post-connect WS request dispatch in src/gateway, preserve the existing probe-first behavior, and separately handle Docker/config ownership guidance or doctor repair in Docker docs/config paths.
