## Summary

Keep open. The latest reporter update still shows the core Windows Docker event-loop starvation on v2026.4.27 after the runtime-deps named-volume mitigation, and current main still has unbounded HTTP stage and WebSocket request-dispatch awaits that do not prove normal routes, internal RPC, or Slack delivery survive the reported blocking stack.

## What This Changes

The issue asks maintainers to investigate and fix a Windows 11 Docker Desktop gateway regression where HTTP, WebSocket RPC, and Slack dispatch can starve or hang after startup.

## Best Possible Solution

Keep this as an active Windows Docker gateway starvation investigation. Maintainers should capture startup trace, liveness diagnostics, and a CPU profile during the 10s to 100s event-loop delay window, then land the narrow fix in the subsystem that is actually blocking the event loop or dispatch path.

## Reproduction Assessment

Yes. The reporter provided a concrete Windows 11 Docker Desktop compose/config path and reconfirmed the remaining failure on v2026.4.27 with named-volume runtime deps, event-loop delay metrics, Slack disconnects, and queued DM symptoms; this review did not independently live-run that environment.

## Solution Assessment

Unclear. Current main's runtime-deps volume, stale-lock, Slack timeout, probe, and diagnostics work are useful mitigations, but the best final fix should remove or offload the blocking work rather than only add wider waits or route-level bypasses.
