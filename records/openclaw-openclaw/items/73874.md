## Summary

Keep open. Current main has adjacent fixes for health/readiness probe ordering, model prewarm, WebSocket pre-auth handshake timeout, and runtime-deps lock diagnostics, but I did not find a source-backed fix for the broader Windows Docker Desktop bind-mount HTTP/WS dispatch deadlock described here. The report is recent, detailed, and covers non-probe HTTP routes, internal WebSocket RPC, Slack startup, bind-mount permissions, plugin runtime-deps, and config ownership behavior that are not fully covered by the existing probe-first fix.

## Best Possible Solution

Keep this issue open and route it as a current gateway/Docker regression investigation. Best next step is a maintainer reproduction on Windows 11 Docker Desktop comparing v2026.4.23, v2026.4.26, and current main with the reporter's bind-mount shape; collect `openclaw gateway diagnostics export`, `OPENCLAW_GATEWAY_STARTUP_TRACE=1`, readiness event-loop data, and WebSocket request traces. If reproduced, split fixes by owner: bounded HTTP/WS request-stage diagnostics or timeouts in `src/gateway`, Docker bind-mount/runtime-deps ownership guidance or setup repair in `docs/install/docker.md` and Docker scripts, and config/doctor ownership handling in `src/config` or doctor repair paths.
