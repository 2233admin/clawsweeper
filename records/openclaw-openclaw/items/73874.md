## Summary

Keep open. Current main has several adjacent mitigations for Docker runtime-deps storage, stale runtime-deps locks, Slack Socket Mode timeout tuning, event-loop diagnostics, probe ordering, and client-side handshake readiness, but I did not find source or shipped-release evidence that the remaining Windows Docker Desktop post-ready event-loop/HTTP/WS dispatch stall is fixed. The reporter also reconfirmed the core stall on v2026.4.27 after adopting the named-volume layout that fixed the install/runtime-deps symptoms.

## What This Changes

The issue reports a Windows 11 Docker Desktop regression where the Gateway reaches ready and accepts HTTP/WS connections but then stalls request dispatch, starving Slack Socket Mode and queued agent sessions despite runtime-deps and permission workarounds.

## Best Possible Solution

Keep this as an active gateway/Docker regression investigation. Maintainers should reproduce on Windows Docker Desktop with the named runtime-deps volume layout, capture startup trace plus a CPU profile around the liveness warning window, and then land a narrow fix in the blocking gateway/plugin/channel path rather than treating the already-fixed runtime-deps storage symptoms as the root cause.

## Reproduction Assessment

Yes, for triage purposes. The reporter provided a concrete Windows 11 + Docker Desktop setup, version comparison back to v2026.4.23, named-volume retest, and v2026.4.27 liveness logs showing a 103-second event-loop delay; this review did not independently run that Windows environment.

## Solution Assessment

Unclear. The current main changes are useful mitigations and diagnostics, but they do not look like the best final fix for the remaining dispatcher stall; the safer path is profiling the blocking stack first, then adding targeted offloading, timeout, or scheduling changes where the profile points.
