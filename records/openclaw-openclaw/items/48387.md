## Summary

Keep this PR open. Current main still ignores the outbound target mode inside the WhatsApp resolver and continues to reject explicit direct-message targets when a non-empty allowFrom list does not include the target. The PR is a focused implementation candidate, but it needs rebasing/porting to the current bundled plugin path and a maintainer decision on the intentional outbound authorization boundary change.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best path is to rebase/port the resolver and regression test to extensions/whatsapp/src, then have maintainers choose between this explicit-send bypass and the broader outbound-only allowTo policy before landing one coherent WhatsApp outbound authorization model.

## Risks / Open Questions

- The PR intentionally broadens explicit WhatsApp outbound delivery to any normalized direct target, so maintainers should confirm that explicit user-chosen sends are allowed to bypass allowFrom rather than requiring a separate outbound allowlist such as the open allowTo design in #49549.
- The PR patch in the provided context targets the older src/whatsapp path; current main owns this code under extensions/whatsapp/src, so the change needs a rebase/port before it can land cleanly.
