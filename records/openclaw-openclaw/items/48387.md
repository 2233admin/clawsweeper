## Summary

Current main still does not implement this PR’s intended behavior. Explicit WhatsApp outbound sends are still routed through the plugin resolver with config-derived allowFrom, and the resolver rejects direct E.164 targets outside a non-empty allowFrom list without a mode === "explicit" bypass. The PR is also stale against the current plugin layout because its patch targets src/whatsapp while main uses extensions/whatsapp/src. The related allowTo proposal in #49549 is an open alternative design, not a merged fix or definite supersession yet.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The maintainer path is to decide the WhatsApp outbound authorization model alongside #49549: either port this explicit-mode bypass and its regression test to extensions/whatsapp/src, or land an allowTo-style design and then close this PR as superseded once that replacement is merged.

## Risks / Open Questions

- The PR intentionally broadens the WhatsApp outbound authorization boundary: any caller using explicit mode could send to arbitrary valid direct WhatsApp numbers even when allowFrom is restrictive.
- The patch needs porting from the old src/whatsapp path to extensions/whatsapp/src before it can land on current main.
- The open allowTo proposal in #49549 may be the better product/security model, but it is still open and not a shipped or definitive replacement.
