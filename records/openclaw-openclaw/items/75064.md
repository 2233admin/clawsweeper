## Summary

Keep open. Current main still has the split-singleton risk this PR targets, and the PR is a narrow draft implementation with mirrored-runtime regression coverage and no concrete blocking review finding; it still needs maintainer review plus CI/Testbox validation before merge.

## What This Changes

The PR moves memory plugin state storage into `createPluginRuntimeStore` and adds mirrored-runtime regression coverage for memory-state plus Memory Wiki bridge/status artifact counting.

## Best Possible Solution

Make the active memory capability registry visible to canonical and mirrored runtime module instances while preserving the existing memory plugin SDK API, with regression coverage for Memory Wiki bridge import and status artifact counts. A host-owned capability service can remain a separate durability follow-up if maintainers want to remove this compatibility-layer global store later.

## Reproduction Assessment

Yes. The linked #74981 report provides concrete packaged Gateway steps and instrumentation, a second user confirms the same 2026.4.27 symptoms, and current main still has module-local memory state read by the bridge/status paths.

## Solution Assessment

Yes for the immediate regression. Reusing `createPluginRuntimeStore` is the narrowest existing mechanism for in-process duplicate-module sharing, while the broader host-owned capability service described in the PR body is a reasonable follow-up rather than a prerequisite for this small fix.
