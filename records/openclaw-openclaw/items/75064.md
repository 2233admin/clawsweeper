## Summary

Keep this draft PR open. Current main still stores the memory plugin registry in a module-local object, while Memory Wiki bridge/status read public artifacts through that registry; the PR is a narrow fix for the linked split-runtime report and I did not find a concrete blocking defect in the proposed diff, but it still needs maintainer review and changed-surface CI/Testbox validation before merge.

## What This Changes

The PR moves memory plugin registry storage behind `createPluginRuntimeStore` and adds mirrored-runtime regression tests for memory-state plus Memory Wiki bridge/status artifact counting.

## Best Possible Solution

Merge a narrow, backward-compatible fix that makes the active memory capability registry visible across canonical and mirrored runtime module instances, with regression coverage for Memory Wiki bridge/status and without changing the public memory plugin API. A host-owned memory capability service can remain a separate follow-up if maintainers want to replace the compatibility global with a more durable architecture.

## Reproduction Assessment

Yes. The linked report gives a concrete OpenClaw 2026.4.27 bridge-mode reproduction, and current main source confirms the plausible failure mode: `memory-core` and `memory-wiki` can import separate `memory-state` module instances that do not share the file-local registry.

## Solution Assessment

Yes for the immediate regression. Reusing `createPluginRuntimeStore` is the narrowest existing mechanism for sharing in-process state across duplicate module instances while preserving the existing memory plugin SDK surface; a host-owned capability service is broader follow-up work.
