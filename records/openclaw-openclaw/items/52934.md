## Summary

Current main still prefers ClawHub over npm for bare plugin specs, including community ClawHub packages. Existing mitigations such as explicit `npm:` installs, non-official warnings, private-package rejection, archive integrity checks, and update-time expected plugin id checks reduce exposure, but they do not prevent initial bare-name substitution. This remains a security/product design issue that should stay open.

## Best Possible Solution

Keep this open for a security/plugin-install design decision. The strongest fix would prevent non-official ClawHub packages from silently winning over trusted npm names, for example by requiring explicit `clawhub:` for community packages, letting npm win unless ClawHub provenance is official or trusted, adding durable trust/signature policy, or making first-time ClawHub source acceptance fail closed. Cover the chosen behavior in CLI and chat plugin install tests plus public plugin security docs.
