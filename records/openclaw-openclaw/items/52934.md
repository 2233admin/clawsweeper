## Summary

Current main still implements the reported ClawHub-first plugin install behavior for bare npm-safe plugin specs. A community ClawHub package can still satisfy a trusted bare name before npm is tried; existing warnings, archive verification, private-package rejection, source metadata, and update-time plugin-id checks reduce risk but do not fail closed for initial name substitution. This remains a security/product design issue, not cleanup-close material.

## Best Possible Solution

Keep this open for a security/plugin-install design fix. The best path is to prevent community ClawHub packages from silently winning over trusted npm names, either by requiring explicit clawhub: for non-official packages, letting npm win unless ClawHub provenance is official or trusted, adding durable trust/signature policy, or making install/update prompts fail closed until the operator explicitly accepts the source. Cover the chosen behavior in CLI and chat /plugin install tests plus public plugin security docs.
