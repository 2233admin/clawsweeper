## Summary

Keep open. Current main has not implemented the requested alias handling, and it also no longer supports the issue's premise that `file` or `filePath` are accepted read parameters at execution time. The remaining work is a contract decision: either restore aliases end to end, or make canonical `path` the only supported contract and align diagnostics/display surfaces.

## What This Changes

The issue requests updating the embedded read-tool start diagnostic so `file` and `filePath` are treated as path aliases instead of warning as missing path.

## Best Possible Solution

Set one coherent read-tool parameter contract across schema, validation, execution, diagnostics, display, and tests: either support `path`, `file_path`, `file`, and `filePath` everywhere, or enforce and document canonical `path` only while removing stale alias assumptions from non-execution surfaces.
