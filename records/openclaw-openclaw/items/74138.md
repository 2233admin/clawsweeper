## Summary

Current main still does not restore the legacy Feishu SDK facade that older npm-installed Feishu packages import, and two open unmerged PRs already specifically reference this issue as their target fix. The issue should stay open until maintainers review, consolidate, or land one compatibility repair.

## What This Changes

The issue reports a regression where Feishu multi-account message dispatch fails after upgrade because `createScopedPairingAccess` is missing from the Feishu SDK subpath used by the installed Feishu package.

## Best Possible Solution

Restore a narrow backward-compatible `openclaw/plugin-sdk/feishu` facade for the legacy named exports used by published Feishu packages, wire it through the SDK entrypoint/export generation path, and add a focused SDK subpath contract test so this public compatibility surface cannot be removed again accidentally.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: run or import an older npm-installed Feishu package that calls `createScopedPairingAccess` from `openclaw/plugin-sdk/feishu` against current main, where the Feishu SDK subpath is absent from the export registry.

## Solution Assessment

Yes. The narrowest maintainable solution is a compatibility facade plus contract coverage; a broader Feishu runtime rewrite would not address the SDK subpath regression as directly.
