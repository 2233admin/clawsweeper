## Summary

Current main still does not implement this PR’s intended behavior. The gateway run path resolves and forwards the effective port into lock acquisition, but the gateway lock payload type/schema and JSON writer still omit `port`, and gateway-lock tests still only cover port use for stale-owner probing, not persisted lock-file discovery. The PR remains a focused bug-fix candidate, with Greptile’s bounded-port suggestion already addressed in the PR discussion.

## Best Possible Solution

Keep this PR open. Rebase it onto current main, add an optional bounded `port` field to `LockPayload` and `LockPayloadSchema`, persist valid `opts.port` values when acquiring the lock, and land focused `src/infra/gateway-lock.test.ts` coverage for provided-port persistence and absent-port backward compatibility.
