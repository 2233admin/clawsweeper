## Summary

Keep #70050 open. Current main still performs default load-time session maintenance synchronously on the caller path for oversized stores, and `chat.history` remains startup-gated until sidecars are ready. The v2026.4.24 config fix gives a workaround (`session.maintenance.mode: "warn"` or higher `maxEntries`), but it does not replace the blocking default with deferred, chunked, or background maintenance.

## Best Possible Solution

Keep the issue open and fix the default startup path: make oversized session-store maintenance deferred, backgrounded, or chunked with event-loop yields; keep an explicit low-friction disable/debug knob; add regression coverage proving gateway startup and `chat.history` availability are not blocked by pruning/capping a large `sessions.json`; then update session maintenance docs to match the final default behavior.
