## Summary

Keep this open. Current main still performs session-store read, JSON parse, stale prune, and count cap synchronously inside `loadSessionStore`; later work reduced cap frequency and honors config, but it did not remove load-time maintenance from the startup-critical path reported here.

## What This Changes

The issue reports that gateway startup can block for roughly 20 minutes after the session maintenance default changed to enforce prune/cap work at session-store load time.

## Best Possible Solution

Keep the session-store size/OOM protection, but move expensive prune/cap cleanup out of the gateway startup critical path or make it budgeted/yielding after readiness. The WebUI timeout work in #44897 can be useful as a client resilience layer, but the root fix belongs in the session-store/gateway startup maintenance path.

## Reproduction Assessment

Partial. The repository has a high-confidence way to exercise the synchronous load-time maintenance path with oversized `sessions.json` fixtures, but not a high-confidence reproduction of the full 20-minute Windows hang or the stdin-unblock symptom.

## Solution Assessment

No. Current main is not the best final state because it still runs potentially expensive maintenance from a synchronous loader; a safer fix would defer or budget that work while keeping default cleanup and manual `openclaw sessions cleanup` behavior intact.
