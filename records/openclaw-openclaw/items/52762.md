## Risks / Open Questions

- The proposed `DedupeCache` change still performs a full TTL scan whenever the cache is over `maxSize`, so saturated high-throughput Discord servers may continue to pay O(n) work on new messages.
- The proposed `pruneMapToMaxSize()` rewrite changes runtime `NaN` limit behavior from no-op to deleting every entry, which can weaken request/dedupe guard tracking if malformed numeric config reaches the helper.
- `createDedupeCache()` and `pruneMapToMaxSize()` are exported through plugin SDK subpaths, so behavior changes affect bundled and third-party plugins, not only Discord.
