## Summary

Keep this PR open. Current main has not implemented the PR's stated lazy-prune change: `DedupeCache.check()` still calls the full prune path on every newly recorded key, and `pruneMapToMaxSize()` still uses the old repeated `map.keys().next()` loop. The PR is still plausibly relevant to Discord because the Discord inbound replay guard uses the shared claimable dedupe cache with a 5,000-entry memory cap, but the existing review comments identify correctness/performance risks that need rework before merge.

## Best Possible Solution

Keep this PR open for maintainer review and rework. The best path is an amortized or bounded-prune `DedupeCache` implementation that avoids the saturated-cache O(n) hot path, preserves existing invalid-limit behavior or intentionally normalizes it with tests, and adds regression coverage for the Discord 5,000-entry replay-guard scenario plus shared `pruneMapToMaxSize()` callers. If maintainers decide newer Discord non-blocking dispatch fully supersedes the user-facing delay report, that should be documented in a canonical follow-up before closing this PR.
