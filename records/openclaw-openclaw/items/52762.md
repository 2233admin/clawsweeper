## Summary

Keep open. Current main and the latest release still use eager dedupe pruning, so the performance problem this PR targets is not already implemented, but the proposed patch has blocking correctness and security-relevant semantic regressions that should be revised before merge.

## What This Changes

The PR changes `createDedupeCache().check()` to prune only after the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete a computed count of oldest entries in one forward pass.

## Best Possible Solution

Land or replace this with a focused shared-helper fix that removes per-message full TTL scans in both normal and saturated cache states, preserves or explicitly validates invalid numeric limit handling, and proves the shared helper plus Discord replay path with targeted tests and changed-gate validation.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists by filling a 5000-entry dedupe cache with unique keys under the Discord five-minute TTL and observing the PR still takes the full prune path on each overflow; a read-only JS simulation also reproduces the `NaN` map-pruning semantic change.

## Solution Assessment

No. The direction is useful, but this is not the best fix because it only delays the full scan until saturation and changes a public SDK helper's invalid-limit behavior; a bounded or amortized pruning strategy with stable limit semantics is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's 5-minute, 5000-entry replay guard, a busy server can keep the cache saturated, so every new unique message still scans about 5000 entries and the reported delay can continue.
  - confidence: 0.91
- **[P2] Preserve invalid-limit behavior in map pruning:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the loop never breaks, deleting every entry. Current behavior is effectively a no-op for `NaN`, and SDK guard callers can propagate malformed numeric input to this helper.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No dependency, CI, secret, or supply-chain change was found, but the shared map-pruning rewrite can weaken webhook guard state for invalid numeric limits.

Concerns:

- **[medium] Preserve webhook guard state for invalid limits:** `src/infra/map-size.ts:10`
  - body: The counted loop deletes every tracked entry when `maxSize` is `NaN`; in-repo webhook in-flight, fixed-window, and bounded-counter guards can normalize malformed `maxTrackedKeys` input to `NaN` before pruning, which can clear active request or rate-limit tracking.
  - confidence: 0.84
