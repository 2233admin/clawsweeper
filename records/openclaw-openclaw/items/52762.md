## Summary

Keep this PR open, but do not merge it as-is. Current main has not implemented the intended performance fix, and the proposed patch still leaves the saturated Discord replay-cache path doing repeated full scans while also changing shared map-pruning behavior in a way that can clear webhook guard state for invalid numeric limits.

## What This Changes

The PR changes the shared dedupe cache to prune only after exceeding `maxSize` and rewrites `pruneMapToMaxSize()` to delete old map entries with a counted forward pass.

## Best Possible Solution

Land or replace this with a focused shared-helper fix that removes per-message full scans in both normal and saturated cache states, preserves or explicitly validates invalid numeric limit handling, keeps public SDK helper semantics stable, and proves the infra helper plus Discord replay and webhook guard callers with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: `check()` still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's five-minute, 5000-entry replay guard, a saturated unique-message stream keeps triggering that O(n) scan on every new key, so the event-loop delay this PR targets can continue on busy servers.
  - confidence: 0.91
- **[P2] Preserve invalid-limit behavior in map pruning:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and the loop never breaks, deleting every entry. Current behavior leaves the map untouched, and SDK webhook guards can pass `NaN` after numeric normalization, so this can make in-flight or rate-limit guards forget active keys.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No CI, dependency, secret, or supply-chain change was found, but the shared map-pruning rewrite can weaken SDK webhook guard state for invalid numeric limits.

Concerns:

- **[medium] Preserve webhook guard state for invalid limits:** `src/infra/map-size.ts:10`
  - body: The counted loop deletes every tracked entry when `maxSize` is `NaN`; in-repo webhook in-flight and rate-limit guards can normalize malformed `maxTrackedKeys` input to `NaN` before pruning, which can clear active request or rate-limit tracking.
  - confidence: 0.84
