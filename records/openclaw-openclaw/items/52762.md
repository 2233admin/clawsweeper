## Summary

Keep open for maintainer review. Current main and the latest release still have the eager dedupe prune behavior, but this PR has blocking review findings: it still full-scans the cache after saturation and changes invalid-limit behavior in a shared helper used by webhook guards.

## What This Changes

The PR changes `createDedupeCache().check()` to call `prune()` only when the cache exceeds `maxSize` and rewrites `pruneMapToMaxSize()` to delete old entries with a counted forward pass.

## Best Possible Solution

Land or replace this with a focused shared-helper fix that removes per-message full scans in both normal and saturated cache states, preserves or deliberately validates invalid numeric limit behavior, keeps public SDK helper contracts stable, and proves the helper plus Discord replay path with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Avoid full-cache prune once the cache saturates:** `src/infra/dedupe.ts:57-59`
  - body: This still calls the full `prune(now)` path whenever `cache.size > maxSize`. With Discord's 5-minute, 5000-entry replay guard, a server above roughly 17 distinct inbound messages per second keeps the cache saturated, so every new committed message still scans about 5000 entries and the delay this PR targets can continue.
  - confidence: 0.91
- **[P2] Preserve invalid-limit behavior in map pruning:** `src/infra/map-size.ts:10-14`
  - body: When `maxSize` is `NaN`, `deleteCount` becomes `NaN` and `i++ >= deleteCount` never becomes true, so this loop deletes every entry. Current behavior is effectively a no-op for `NaN`; webhook guard callers can pass `NaN` after numeric normalization, so this can make in-flight or rate-limit guards forget active keys.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No supply-chain change was found, but the map-size diff introduces a security-relevant guard-state regression for malformed numeric limits.

Concerns:

- **[medium] Preserve webhook guard state on invalid limits:** `src/infra/map-size.ts:10`
  - body: The changed pruning loop deletes every tracked entry when `maxSize` is `NaN`. Because webhook guard helpers can propagate malformed numeric options into `pruneMapToMaxSize()`, this can clear in-flight or rate-limit state and weaken request protection.
  - confidence: 0.84
