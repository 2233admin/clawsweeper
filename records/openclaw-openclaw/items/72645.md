## Summary

Keep this PR open. Current main does not implement the proposed per-message `message.processing.*` or skill lifecycle `skill.execution.*` diagnostics/OTel surface, and the PR is still an active implementation candidate that needs maintainer API, privacy, cardinality, and scope review rather than conservative cleanup.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to decide whether first-class per-message and skill lifecycle telemetry belongs in the core diagnostics contract, then land a focused implementation with diagnostic contract tests, diagnostics-otel exporter tests, stability-recorder handling, public docs, and explicit privacy/cardinality guidance; otherwise close it after that product/API decision, not through automated cleanup.
