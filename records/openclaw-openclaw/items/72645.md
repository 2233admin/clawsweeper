## Summary

Keep open. Current main does not already implement the requested lifecycle telemetry, and this PR remains an active feature branch with concrete privacy and metric-cardinality blockers before it is mergeable.

## What This Changes

The PR adds `message.processing.*` and `skill.execution.*` diagnostics events from the embedded runner, exports corresponding OpenTelemetry spans and metrics in `diagnostics-otel`, adds a skill observability guide, and adjusts qa-lab dependency/test surfaces.

## Best Possible Solution

Keep the useful lifecycle telemetry direction, but land it only after maintainers settle the public event/attribute contract: metrics should use bounded stage/outcome labels, high-cardinality correlation IDs should stay off metrics, session identifiers should be removed, hashed, or made explicitly opt-in under a reviewed privacy policy, and docs/changelog should match the final contract.

## Reproduction Assessment

Yes. Static inspection of PR head `6582f290e888469701069c2203bb7b899c913423` shows the exported session identifiers and unbounded metric labels, while current-main search shows the requested lifecycle events are not already implemented.

## Solution Assessment

No. The proposed direction is useful, but this is not the best mergeable form until the telemetry contract avoids raw session IDs and keeps metrics low-cardinality.
