## Summary

Keep open. The cron warning direction matches a real current-main visibility gap, but the PR should not merge as-is because the new run-log `warnings` payload is not added to the public gateway protocol schema and the user-facing fix lacks the required changelog entry.

## What This Changes

The PR adds a configurable ghost-run warning and persisted run-log warning code for fast `ok` main-session `systemEvent` cron jobs using `wakeMode="next-heartbeat"`, updates config schema/help metadata and tests, and includes small provider/formatting cleanup.

## Best Possible Solution

Land a focused cron visibility fix after adding the `warnings` field to the gateway protocol contract, covering it with a small protocol/run-log test, and adding an Unreleased changelog entry. Any deeper guarantee that a `next-heartbeat` handoff actually produced an agent turn should remain separate product work.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a main-session `systemEvent` cron job with `wakeMode="next-heartbeat"` currently queues a heartbeat and returns `ok` immediately, while current run-log entries have no warning field to distinguish that handoff from confirmed processing.

## Solution Assessment

No, not as submitted. The warning-as-mitigation approach is the narrow maintainable fix for #63106, but the implementation needs protocol-schema and changelog follow-through before it is the best mergeable solution.
