## Summary

Keep this issue open. Current main still matches the reported Tailscale startup race window: the gateway enables Serve/Funnel and immediately performs a single `tailscale status --json` hostname lookup, with no retry/backoff around the post-enable status read. Verbose exec logging can still emit the confusing `Command failed: tailscale status --json` line before the caller suppresses the hostname lookup failure. The alternate `gateway.tailscale.binaryPath` config is also not present.

## Best Possible Solution

Keep this issue open and fix the Gateway Tailscale startup path. Add bounded retry/backoff around the post-Serve/Funnel hostname/status lookup, preferably in the shared Tailscale status helper or the Gateway startup hostname path, and add regression coverage where the first `tailscale status --json` call fails and a later retry succeeds without producing a misleading verbose startup failure. Treat `gateway.tailscale.binaryPath` as a separate config/schema/docs enhancement if maintainers want it.
