## Summary

Keep open. Current main still has the reported Gateway Tailscale startup path: after enabling Serve/Funnel, it immediately asks for the Tailnet hostname with a single `tailscale status --json` attempt. The alternate `gateway.tailscale.binaryPath` escape hatch is also not implemented in schema/types/docs.

## Best Possible Solution

Keep this issue open and fix the Gateway Tailscale startup path. Add bounded retry/backoff around the post-Serve/Funnel hostname/status lookup, preferably in the shared Tailscale status/hostname helper or Gateway startup path, and make the retry quiet until attempts are exhausted so verbose mode does not print a misleading transient `Command failed: tailscale status --json`. Add regression coverage where the first status call fails and a later attempt succeeds. Treat `gateway.tailscale.binaryPath` as a separate schema/docs/product enhancement if maintainers want explicit operator control.
