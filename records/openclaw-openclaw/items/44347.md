## Summary

Keep this issue open. Current main already preserves inbound Google Chat thread names and sends replies with `thread.name` plus `messageReplyOption`, so the threaded-reply half appears covered. The second requested capability is still missing: the Google Chat plugin remains service-account/webhook based, has no user OAuth or Workspace Events subscription configuration, and no code path for `chat.messages`/all-space-message delivery.

## Best Possible Solution

Keep this issue open, or split it so the completed threaded-reply behavior is separated from the remaining all-message delivery work. The remaining implementation should stay in `extensions/googlechat`: add a user-OAuth and/or verified Workspace Events subscription mode, extend config/schema/setup/docs for those credentials and scopes, and add focused tests that non-mention space message events can be accepted without weakening the existing service-account webhook security model.
