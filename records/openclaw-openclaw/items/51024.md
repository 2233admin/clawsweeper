## Summary

Keep open. Current main still clears stale cron `runningAtMs` markers with a fixed 2-hour threshold, while this PR’s intended change is to derive that cleanup window from each job’s effective timeout. Current restart and manual-run recovery reduce some stale-marker failures, but they do not implement the timeout-aware maintenance cleanup proposed here.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best path is to port the timeout-aware stale-marker helper and regression test onto current main, then explicitly decide whether default-timeout jobs should use the shorter effective timeout window or preserve the old 2-hour window unless `timeoutSeconds` is explicitly configured.

## Risks / Open Questions

- The functional review risk is scheduler concurrency: deriving stale cleanup from synthesized default timeouts may clear `systemEvent` jobs after about 11 minutes and default `agentTurn` jobs after about 61 minutes, so maintainers should decide whether that earlier recovery can duplicate a legitimately slow run.
- The PR appears supply-chain neutral from the provided diff, but it changes runtime recovery behavior in a shared cron path and should keep focused regression coverage when rebased.
