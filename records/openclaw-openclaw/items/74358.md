## Summary

Keep open. Current main still has source-backed Slack paths where the reporter's object-only `streaming.preview.toolProgress: false` config does not guarantee an in-flight preview: object config defaults to partial/native mode, top-level DM threading defaults can leave preview disabled, and `onPartialReply` is only wired for the non-native preview path. The latest discussion also treats open PR #56544 as a sibling flat-DM fix, not a complete fix for this disabled-tool-progress report.

## What This Changes

The issue requests restoring Slack in-flight preview text while hiding tool-progress lines when `streaming.preview.toolProgress` is `false`, especially for verbose multi-tool DM turns using object-only Slack streaming config.

## Best Possible Solution

Slack should treat `streaming.preview.toolProgress: false` as preview-on and progress-lines-off across object-only config, DM, threaded, and native/draft paths, with regression coverage that separates the flat-DM preview gate from the disabled-tool-progress behavior.

## Reproduction Assessment

Yes, for a high-confidence source-level reproduction path: object-only Slack streaming config defaults to partial/native behavior, default flat DMs have no thread target, and current dispatch can avoid creating a draft stream or partial-reply callback. A live Slack run would still be useful before closing the loop.

## Solution Assessment

Unclear. The issue body's line-level coupling is real but likely insufficient alone; the maintainable fix needs to separate tool-progress visibility from preview eligibility and reconcile the Slack native-streaming versus draft-preview path without accidentally duplicating #56544.
