## Summary

Keep open. Current main still aggregates descendant fallback replies, while this PR’s central behavior is to return only the freshest settled descendant reply. The PR is not implemented on main, and the latest review comments identify product risks that still need maintainer judgment before merge.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best product direction is a narrower cron fallback fix that suppresses stale same-child historical replies while preserving one latest usable reply per distinct child for fan-out runs, with regression coverage for same-child stale history, multi-child aggregation, silent replies, and a stricter descendant-only interim acknowledgement filter.

## Risks / Open Questions

- Merging the PR as-is can drop complementary fan-out cron results because it returns after the first usable descendant reply instead of preserving one usable result per distinct child.
- The PR applies `isLikelyInterimCronMessage` to arbitrary child-session output; that matcher is intentionally broad and can discard real short outputs containing phrases like `retrying now` or `should have your summary`.
- Closing as implemented would be incorrect because current main still has code and tests enforcing multi-child aggregation.
