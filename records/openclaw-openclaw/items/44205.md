## Summary

Keep this PR open. Current main does not implement the proposed freshest-only behavior: the shipped helper still aggregates usable fallback replies across distinct child sessions, and the current regression test asserts that multi-child output is joined. The remaining question is a maintainer/product decision about cron fallback semantics, not conservative cleanup closure.

## What This Changes

The PR branch changes cron isolated-agent fallback selection from multi-child aggregation to a freshest-only descendant reply, adds regression tests for that behavior, and adjusts the existing-value return cast in the shared global singleton helper.

## Best Possible Solution

Settle the intended cron fallback contract first. If the goal is stale-history suppression, keep one latest usable reply per distinct child session within the current requester run and add tests for same-child stale history, multi-child fan-out aggregation, silent replies, and real interim acknowledgements; if maintainers intentionally want freshest-only delivery, update the tests and user-facing subagent docs to make that product behavior explicit.

## Risks / Open Questions

- Freshest-only fallback can drop complementary results from fan-out cron runs where multiple distinct child sessions finish successfully and current main aggregates their latest outputs.
- Applying isLikelyInterimCronMessage to arbitrary child-session output may discard a real short result that happens to contain one of the interim hint substrings.
- The shared global singleton cast is unrelated to the cron behavior and current main already returns the existing global value with an explicit cast, so a rebase or narrower branch may be needed.
