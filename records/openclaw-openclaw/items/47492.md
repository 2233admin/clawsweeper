## Summary

Keep open. Current main has strong exec approval, allowlist, routing, and inter-session provenance primitives, but it does not implement the requested gateway-enforced verified-human instruction-provenance policy for sensitive targets. The related CaMeL PR is still open and is about taint from untrusted content, while this issue explicitly asks for a separate gate for clean/autonomous sessions that reach sensitive resources.

## Best Possible Solution

Keep this issue open for maintainer security/product design. The best path is to coordinate it with #54508 and the open CaMeL PR #39231, then define a generic core provenance contract and policy point for sensitive side-effect targets before adding config schema, gateway enforcement, approval UX, tests, and docs. It should not be bundled into a small drive-by fix PR without that design decision.
