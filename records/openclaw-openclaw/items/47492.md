## Summary

Keep open. Current main still has exec approvals, turn-source routing metadata, plugin approval hooks, and input-provenance annotations, but it does not implement a gateway-enforced verified-human instruction-provenance policy for sensitive target patterns. The related CaMeL and persistent-taint items are adjacent security work, but they do not make this issue obsolete because this request is about clean-session or autonomous sensitive actions that may have no untrusted-content taint.

## What This Changes

The issue requests a gateway-enforced policy that marks sensitive side-effect targets and requires actions against them to trace to a verified human instruction, with denial or stricter approval fallback when they do not.

## Best Possible Solution

Define a generic instruction-provenance contract and a gateway/tool-call policy point for sensitive side-effect targets, then wire it into exec and other high-risk surfaces with config schema, approval UX, protocol changes, tests, and docs once the security semantics are agreed.

## Reproduction Assessment

Not applicable as a feature/security design request. The high-confidence verification path is static: current source, schema, docs, and related issue context show annotations and approval guardrails, but no verified-human sensitive-target enforcement.

## Solution Assessment

Unclear for immediate implementation; a one-off exec-only patch would likely be too narrow. The maintainable solution is a coordinated core design with the approval, provenance, and CaMeL/taint surfaces so sensitive target policy applies consistently across side-effect tools.
