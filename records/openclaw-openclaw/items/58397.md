## Summary

Keep PR #58397 open. Current main has overlapping sessions_send announce repairs, including threadId propagation and stale-reply protections, but it does not implement the PR's central zero-timeout fail-closed announce planning. On current main, timeoutSeconds=0 still returns accepted with pending announce delivery and starts the background A2A flow; that flow can wait for the run, read history, resolve announce targets through sessions.list for lookup-backed channels, and run the ping-pong/announce agent steps. The PR remains potentially useful, but it needs rebase/update because current main has partial fixes and the final PR review notes an unresolved compatibility break in the changed runSessionsSendA2AFlow call shape.

## Best Possible Solution

Keep the PR open for maintainer/author follow-up. The best path is a current-main rebase that narrows the change to zero-timeout fail-closed announce planning, preserves the shipped threadId/accountId and parent-owned ACP skip behavior, makes the new announce plan compatible with all runSessionsSendA2AFlow callers or updates those callers/tests together, and adds focused regressions proving timeoutSeconds=0 does not depend on sessions.list, agent.wait, or chat.history for announce routing.
