## Summary

Keep this PR open. Current main still sends a DM pairing reply only when the pairing request is newly created, and public docs plus shared/channel tests explicitly describe and lock that no-repeat behavior. The submitted one-line guard change is not enough to land because it would also make `onCreated` run for repeat pending requests, breaking the callback contract used by channel integrations and leaving tests/docs inconsistent. Security pass: the diff is limited to pairing runtime code and does not touch dependencies, CI, package resolution, secrets, or release scripts, but it does broaden repeated exposure of still-valid DM pairing codes without a documented throttle or policy.

## Best Possible Solution

Keep this PR open only as an implementation candidate for a deliberate UX/product change. A complete version should resend the existing pending code when storage returns a non-empty code, preserve `onCreated` so it only runs for `created === true` or introduce a separately named reminder/reply callback, add or document any repeat-send cooldown policy, and update the shared pairing tests, affected channel tests, and the public pairing docs.

## Risks / Open Questions

- Merging the submitted one-line patch would make `onCreated` fire for repeat pending requests, creating duplicate pairing-request logs today and possibly repeated future one-time side effects in channel code.
- Repeatedly resending a still-valid pairing code to unapproved senders broadens code exposure and DM spam potential during the one-hour validity window unless maintainers define a cooldown or explicit repeat-reply policy.
- Current shared tests, Telegram/Slack/Signal/BlueBubbles channel tests, and public pairing docs all encode no-repeat behavior, so the PR would leave implementation, tests, and user-facing contract inconsistent.
