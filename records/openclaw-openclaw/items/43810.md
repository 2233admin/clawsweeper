## Summary

Keep this PR open, but it is not merge-ready. Current main intentionally sends DM pairing replies only when a request is newly created, while this PR would resend a valid pending code on every repeat request and also make creation-only callbacks fire for existing requests; the submitted one-line change conflicts with current tests, docs, and channel callback semantics.

## What This Changes

The PR changes the shared DM pairing challenge guard so existing pending requests with a code send another pairing reply instead of returning early.

## Best Possible Solution

If repeat reminders are desired, add an explicit resend/reminder contract that preserves `created` and `onCreated` for first-time request creation only, reuses only valid pending codes, applies a documented cooldown or suppression policy comparable to Matrix, and updates shared helper tests, affected channel tests, docs, and changelog together.

## Reproduction Assessment

Yes. Mock `upsertPairingRequest` to return `{ code: "ABCD", created: false }` in `src/pairing/pairing-challenge.test.ts`; current main returns `{ created: false }` without sending a reply, while this PR's `!code` guard would fall through because the code is truthy.

## Solution Assessment

No. The submitted one-line guard is not the best way to solve the UX request because it conflates reminder delivery with request creation; a named reminder path with cooldown and creation-only callback semantics is safer.
