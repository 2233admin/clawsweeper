## Summary

This PR should stay open because it targets a real remaining Telegram outbound-media proxy gap, but the branch is stale against current main. Current main has moved to the dispatcherAttempts/trusted explicit-proxy DNS transport contract and src/media/web-media.ts, while the PR still uses the older pinned-dispatcher fields and old WhatsApp media-loader layout.

## What This Changes

The PR threads Telegram proxy-aware transport options into remote outbound media prefetch for direct sends and bot reply delivery, extends shared outbound media load options, and adds Telegram send/reply regression tests.

## Best Possible Solution

Port the branch to the maintained contract by carrying TelegramTransport.sourceFetch, dispatcherAttempts, trusted explicit-proxy DNS state, and shouldRetryTelegramTransportFallback through direct sends, bot reply delivery, and native-command reply delivery into the shared web-media/fetchRemoteMedia path without weakening the SSRF guard.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to configure Telegram with a required proxy and send or reply with a remote media URL; current main reaches loadWebMedia without Telegram transport options in both direct-send and reply-delivery paths.

## Solution Assessment

No. The requested direction is right, but this branch is not the best mergeable solution until it is ported from the old pinned-dispatcher API and old WhatsApp media implementation to current main's dispatcherAttempts and src/media/web-media.ts contract.
