## Summary

Keep open. Current main and the latest release still leave outbound Telegram direct-send and reply-media prefetch on the generic media loader, while inbound Telegram media already uses the proxy-aware transport shape; the branch addresses a valid gap but is stale against current transport APIs and needs a port before merge.

## What This Changes

The PR threads Telegram proxy transport options through outbound remote media loading for direct sends and bot replies, extends shared media load option plumbing, adds Telegram regression coverage, and adjusts a WhatsApp test.

## Best Possible Solution

Port the fix onto current main using the current `TelegramTransport` contract: resolve or thread the transport into direct sends, bot reply delivery, and native-command reply delivery; extend shared outbound media options to carry dispatcher attempts and retry policy through `loadWebMedia` to `fetchRemoteMedia`; preserve existing SSRF guards and explicit-proxy DNS trust semantics; and cover both direct `mediaUrl` sends and delivered reply media with focused tests.
