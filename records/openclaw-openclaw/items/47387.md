## Summary

Keep this PR open. Current main still has the outbound Telegram media proxy gap this PR targets: API calls and inbound Telegram media downloads use proxy-aware Telegram transport, but outbound direct sends and reply delivery still prefetch remote media through the generic loader without Telegram transport options. The PR is stale against current transport names and should be rebased/ported rather than closed as already implemented.

## Best Possible Solution

Keep this PR open and port the central fix onto current main: use the current TelegramTransport shape, thread it from bot creation through message/native-command delivery into deliverReplies, resolve/reuse it for direct sendMessageTelegram mediaUrl prefetch, pass sourceFetch/dispatcherAttempts/shouldRetryTelegramTransportFallback/trusted explicit-proxy DNS through the outbound media loader, and add focused regression tests for Telegram direct sends and reply media paths.
