## Summary

Keep this PR open. Current main still does not route Telegram outbound remote media prefetch through the Telegram proxy-aware transport for either direct sends or reply delivery. The earlier PR review comments are still directionally correct against current main: inbound Telegram media downloads use the newer transport shape, but outbound media sends/replies still call the generic media loader without Telegram transport options.

## Best Possible Solution

Keep this PR open and port the central fix onto current main: resolve/reuse the current TelegramTransport for direct sendMessageTelegram media prefetch, thread the bot transport through message/native-command reply delivery into deliverReplies, extend the outbound media loader options as needed for dispatcherAttempts and shouldRetryTelegramTransportFallback, preserve trusted explicit-proxy DNS handling, and add focused regression tests for direct Telegram media sends and reply media paths.
