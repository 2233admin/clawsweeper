## Summary

Current main still matches the reported Telegram outbound final-reply bug. Telegram computes and documents a mediaMaxMb cap, and the direct send path honors it, but the final reply delivery path still does not thread mediaMaxBytes into deliverReplies, deliverMediaReply, or loadWebMedia. The linked fix PR #46425 was closed unmerged, so this issue should stay open.

## Best Possible Solution

Keep this issue open and implement the narrow parameter-threading fix: add mediaMaxBytes to Telegram delivery base options, deliverReplies, and deliverMediaReply; pass it as maxBytes to buildOutboundMediaLoadOptions; and cover normal final replies plus native command delivery with focused regression tests. #46425 is useful prior implementation context but was closed unmerged.
