Keep open. Current main still suppresses WhatsApp commentary with no opt-in live delivery path, so the PR is useful, but the proposed delivered-commentary delta helper can resend long already-delivered commentary segments and the branch is currently dirty against main.

## What This Changes

The PR adds opt-in WhatsApp live assistant commentary delivery with channel/account config, a generic commentary callback seam, text-only WhatsApp delivery, final-reply dedupe, abort/timeout handling, tests, docs, changelog, and Plugin SDK runtime exports.

## Best Possible Solution

Keep the opt-in, WhatsApp-only, text-only design, but fix the long-commentary dedupe math, add focused regression coverage, resolve the dirty branch/CI state, and then run normal maintainer validation before merging.

## Reproduction Assessment

Yes. The current-main gap is reproducible by source inspection: commentary-phase text returns before delivery, and no WhatsApp commentary config/callback symbols exist. The PR regression is reproducible at helper level with a delivered segment longer than the stored 8,000-character prefix followed by an extended current text.

## Solution Assessment

No as-is. The proposed architecture is a reasonable narrow seam for WhatsApp live commentary, but the delivered-commentary delta helper must honor the recorded delivered length for truncated stored snapshots before the PR is safe to merge.
