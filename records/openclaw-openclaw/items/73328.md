## Summary

Keep this issue open: current main still routes unlisted Claude 3.x/3.5 Bedrock model IDs through the Amazon Bedrock discovery default context window, and two open PRs already target the exact remaining work.

## What This Changes

Amazon Bedrock discovery should respect per-model context windows for Claude 3.x/4.x models instead of assigning the global 32000-token default to every discovered model.

## Best Possible Solution

Land one canonical Bedrock plugin-owned fix with regression coverage for the reported Claude 3.x/3.5 IDs and region-prefixed inference-profile variants. If maintainers choose a public override knob, keep the manifest schema, exported config type, validation, docs, UI hints, and discovery cache-key behavior aligned.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a reported active Claude 3.5 Sonnet Bedrock ID is absent from `KNOWN_CONTEXT_WINDOWS`, so discovery falls through to `defaults.contextWindow` at 32000 unless the user changes the global default.

## Solution Assessment

Unclear. The requested outcome is valid, but #73382 and #73959 implement different solution contracts; the narrow known-model/family resolver is probably the smaller behavioral fix unless maintainers explicitly want a new user-facing override surface.
