## Summary

Keep open. Current main still does not define or honor `messages.tts.filterEmoji`; the shared config contract, strict schema, generated schema, docs, auto-TTS path, agent `tts` tool, and Gateway `tts.convert` path all lack config-controlled emoji filtering. Existing emoji stripping is provider- or channel-local, so it does not satisfy the requested shared TTS behavior.

## What This Changes

The issue requests native support for a `messages.tts.filterEmoji` option that filters emoji before TTS synthesis, preferably enabled by default.

## Best Possible Solution

Add `messages.tts.filterEmoji?: boolean` to the public TTS config contract with a documented default, implement a shared TTS text-preparation helper that strips markdown and optionally strips emoji, and route automatic reply TTS, the agent `tts` tool, and Gateway `tts.convert` through it while preserving the visible chat reply text.
