## Summary

Current main does not implement the requested `messages.tts.filterEmoji` behavior. The TTS schema/type surface has no `filterEmoji` field, the agent `tts` tool accepts only `text`, `channel`, and `timeoutMs`, and the speech pipeline strips Markdown before synthesis but does not strip emoji. The report is recent, has no protected maintainer ownership, and appears narrow enough for a focused fix PR.

## Best Possible Solution

Keep this open and implement a narrow TTS config/runtime change: add `messages.tts.filterEmoji?: boolean` with a documented default, apply it only to text sent to speech providers and related spoken-text metadata, preserve visible chat text, and cover config/schema/docs plus tool and auto-TTS paths with focused tests.

The observable goal is that TTS-generated audio skips emoji by default, while users can set `messages.tts.filterEmoji: false` to preserve current spoken input.
