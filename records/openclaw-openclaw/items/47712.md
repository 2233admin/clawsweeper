## Summary

Keep open. Current main still does not define or honor `messages.tts.filterEmoji`: the strict config schema, public type, generated schema, docs, auto-TTS path, agent `tts` tool, and `tts.convert` path all lack shared config-controlled emoji filtering. Provider-local emoji stripping exists for Local CLI and Discord voice, but that does not satisfy the requested core TTS behavior.

## What This Changes

Add a `messages.tts.filterEmoji` option so TTS skips emoji before synthesis, with filtering enabled by default as requested.

## Best Possible Solution

Expose `messages.tts.filterEmoji?: boolean` as a documented TTS config field, default it consistently, and route all synthesis entry points through one shared spoken-text preparation helper while preserving existing visible replies and transcripts.
