## Summary

Keep open. Current main still routes `/v1/chat/completions` through the normal agent command path with generated session keys and persisted session metadata/transcripts, and there is no coded or documented passthrough/ephemeral mode for no-disk one-shot HTTP calls. The related open issues are relevant context, but none is a merged fix or exact canonical replacement for this narrower LLM-relay request.

## What This Changes

The issue requests a lightweight `/v1/chat/completions` mode that preserves Gateway auth, model routing, and fallback while skipping session creation, transcript/session metadata persistence, skills, tools, memory, and system-prompt injection for stateless calls.

## Best Possible Solution

Define a first-class, documented HTTP relay or ephemeral-session contract that cleanly separates provider routing/fallback/auth from agent session persistence, with explicit behavior for streaming, usage accounting, tools, system prompts, memory, and compatibility with existing OpenAI-compatible clients.

## Reproduction Assessment

Yes. The source-level path is high-confidence: call `/v1/chat/completions` without `user` or `x-openclaw-session-key`, and the gateway generates an `openai:<uuid>` agent session key before entering the normal agent-command persistence path.

## Solution Assessment

Unclear. The requested passthrough mode is plausible, but it is not yet proven as the best fix because it changes the endpoint contract more sharply than an ephemeral session flag and may need a shared provider-routing seam rather than a one-off shortcut.
