## Summary

Keep open. Current main still relies on the model successfully calling the `read` tool for the first-run BOOTSTRAP.md workflow, while embedded runs deliberately strip raw BOOTSTRAP.md from injected Project Context. The report is fresh, includes a concrete clean-install path on the latest release, and I did not find a current-main fix or clean-install/Ollama regression proof that resolves the reported UI/bootstrap failure.

## Best Possible Solution

Keep this issue open and reproduce the clean-install UI path on current main with `ollama/qwen3.5:9b` or another affected local model. The likely fix belongs in the first-run bootstrap path: either make the embedded bootstrap prompt robust when tool calling is unreliable, expose a clear blocking diagnostic when `read` is not callable, or add a fallback that injects/surfaces BOOTSTRAP.md safely for bootstrap-pending primary runs. Add a regression test around the current `runEmbeddedAttempt` bootstrap routing/context behavior plus a targeted Ollama/tool-call probe if feasible.
