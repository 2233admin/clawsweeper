## Summary

Keep this PR open. Current main still drops all-zero usage before it can reach status surfaces: subscription usage resolution requires `hasNonzeroUsage`, `getUsageTotals()` returns `undefined` for all-zero totals, and session persistence only updates token fields when usage is nonzero. The existing bot review comments correctly identified the same gap, and the PR’s small diff does not solve the end-to-end zero-usage case.

## Best Possible Solution

Keep the PR open for author or maintainer follow-up, or replace it with a complete fix on current main. The fix should track whether usage was observed independently from whether counters are nonzero, return zero-valued totals from `getUsageTotals()` when appropriate, update `updateSessionStoreAfterAgentRun()` so fresh zero usage does not become stale/no-usage, and add focused tests for all-zero provider usage through the subscription and session-store/status path.

## Risks / Open Questions

- Merging the PR as-is would likely leave the user-visible zero-usage `?` behavior unchanged while implying the bug is fixed.
- A complete fix has to preserve the distinction between "no usage observed" and "usage observed with zero values" across subscription aggregation and session persistence; changing only one guard risks regressing the existing stale-token protection for providers that truly omit usage.
- There is no regression test in the PR context for an all-zero provider usage snapshot flowing through to a fresh status or usage display.
