## Summary

Keep this PR open. Current main still lacks the requested coding-tool wiring for Gateway HTTP invoke, so the PR is not obsolete, but the latest branch still has blocking review concerns on a sensitive direct tool-execution surface.

## What This Changes

The branch adds an unwrapped coding-tools factory, appends those tools to the Gateway scoped resolver, and extends the HTTP invoke tests for allowlisted coding tools, hook execution, adjusted params, deny-list behavior, and loopback scope.

## Best Possible Solution

Land the direct coding-tool capability only after narrowing coding-tool construction to the HTTP surface, aligning the HTTP deny list and docs with canonical coding tool names, and using a raw factory that truly avoids wrapper hook execution for every tool it returns.

## Reproduction Assessment

Yes. A source-level reproduction is clear from the PR head: resolveGatewayScopedTools constructs coding tools even for loopback, and createOpenClawCodingToolsRaw maps after exec/process have already been spread by the deferred-description wrapper; I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. The direction is plausible, but the current patch is not the safest implementation because it widens loopback, leaves canonical mutating tools outside the HTTP deny list, and fails to unwrap exec/process reliably.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Deny canonical mutating tools before exposing coding tools:** `src/gateway/tool-resolution.ts:125-132`
  - body: This call makes the real coding tool names reachable through `/tools/invoke`, but the default HTTP deny list only contains legacy-style `fs_write`/`fs_delete`/`fs_move` names and does not deny the canonical `write`, `edit`, or `process` tools. With the normal pass-through policy, those file/process mutation tools become callable without an explicit `gateway.tools.allow` opt-in, contrary to the PR's stated security posture. Add the canonical names to the HTTP deny list and cover them with real resolver tests before wiring the factory in.
  - confidence: 0.9
- **[P2] Gate coding tools to the HTTP resolver surface:** `src/gateway/tool-resolution.ts:125-136`
  - body: `resolveGatewayScopedTools` is also used by MCP loopback with `surface: "loopback"`, but this constructs coding tools unconditionally. Loopback only excludes a small native set and does not apply the HTTP deny list, so any coding tool outside that set can appear on the loopback schema even though this PR is scoped to `/tools/invoke`. Build `codingTools` only when the resolved surface is `"http"` and add a resolver-level loopback regression.
  - confidence: 0.86
- **[P2] Unwrap before description wrappers hide the original tool:** `src/agents/pi-tools.ts:769`
  - body: `createOpenClawCodingTools()` applies deferred descriptions after hook wrapping, and that helper spreads `exec` and `process` into new objects. The new original-tool symbol is non-enumerable, so those spreads drop it; `unwrapToolBeforeToolCallHook()` then returns still-wrapped exec/process tools. Allowlisted HTTP calls to those tools will still run `before_tool_call` twice and retain adjusted params. Preserve the original pointer through wrappers or build the raw tool list before hook wrapping.
  - confidence: 0.88
