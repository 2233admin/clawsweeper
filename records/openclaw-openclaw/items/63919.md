## Summary

Keep this active PR open for revision. Current main still lacks the requested coding-tool wiring for `/tools/invoke`, and the branch addresses the earlier deny-list and hook double-fire concerns, but the latest diff is not merge-ready because the raw coding-tools call can re-enter plugin resolution on the HTTP path and the claimed `apply_patch` availability still does not work through this resolver. Review target: https://github.com/openclaw/openclaw/pull/63919

## What This Changes

The PR adds an unwrapped coding-tool factory, merges coding tools into Gateway HTTP `/tools/invoke` resolution, extends the default HTTP dangerous-tool deny list, and adds mocked HTTP invoke regression tests.

## Best Possible Solution

Revise the PR so the HTTP resolver gets a coding-only, unwrapped tool set that does not re-enter plugin resolution when `disablePluginTools` is requested; then decide and test the intended `apply_patch` behavior for direct HTTP calls, update the public deny-list docs, and run the targeted Gateway/security checks before merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on the PR head, any HTTP resolver call constructs `createOpenClawCodingToolsRaw`, which delegates into `createOpenClawTools` without `disablePluginTools`, conflicting with the existing plugin-reload e2e contract; allowlisting `apply_patch` still cannot surface the tool because no model provider is passed.

## Solution Assessment

No. The direction is useful, but the current implementation should not reuse the full agent tool factory inside the HTTP resolver; a narrower raw coding-tool seam or a properly threaded `disablePluginTools`/provider context would be safer and more maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep raw coding tools from loading plugins:** `src/gateway/tool-resolution.ts:132-139`
  - body: `createOpenClawCodingToolsRaw()` builds the full agent coding-tool set, and that factory appends `createOpenClawTools()` without `disablePluginTools`. This defeats the HTTP handler's known-core fast path and can re-run workspace/plugin tool resolution even for a core `/tools/invoke` request, which current `src/gateway/gateway.test.ts` explicitly guards against. Return only the coding primitives here or thread `disablePluginTools` through the raw factory.
  - confidence: 0.87
- **[P2] Make allowlisted apply_patch reachable:** `src/gateway/tool-resolution.ts:132-139`
  - body: The resolver does not pass any model provider into the raw factory, but `createOpenClawCodingTools()` only creates `apply_patch` when `isOpenAIProvider(options?.modelProvider)` is true. An operator can therefore allowlist `apply_patch` and still get a 404, even though this PR treats the HTTP deny list as the gate for coding tools. Resolve the intended provider context for direct HTTP calls or document/remove this tool from the promised surface and cover it with a regression.
  - confidence: 0.78
- **[P3] Update the HTTP deny-list docs:** `src/security/dangerous-tools.ts:16-21`
  - body: This adds `process`, `write`, and `edit` to the default HTTP deny list, but the public Tools Invoke docs still list the old defaults. Operators reading the docs will not know those canonical coding tools require `gateway.tools.allow`, so update the docs alongside the behavior change.
  - confidence: 0.9
