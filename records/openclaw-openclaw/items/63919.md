## Summary

Keep this PR open. The feature direction matches the open direct-tool-invocation request, and current main still lacks coding tools on the HTTP invoke resolver, but the latest patch still has blocking issues on a sensitive Gateway execution surface: canonical mutating/runtime coding tools are not in the default HTTP deny list, and the new raw factory does not actually unwrap exec/process after the deferred-description wrapper copies those tools.

## What This Changes

The PR adds a raw coding-tools factory, merges coding tools into Gateway-scoped tool resolution for `/tools/invoke`, and extends Gateway HTTP invoke tests with mocked coding-tool coverage.

## Best Possible Solution

Land the direct coding-tool capability only after the HTTP deny list and docs use the canonical coding tool names, the raw factory truly avoids hook wrapping for every returned tool, and regression coverage exercises the real factory/tool names rather than only mocked `read_file`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with the PR diff, `write`, `edit`, and `process` are merged into the HTTP resolver while the default deny list does not contain those names, and allowlisted `exec`/default `process` still carry the before-tool wrapper after deferred descriptions.

## Solution Assessment

No. The implementation is close, but the narrow maintainable fix is to align the deny list with canonical tools and build or preserve raw tools before any before-tool wrapper can be hidden by later object spreads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Deny canonical mutating tools before exposing coding tools:** `src/gateway/tool-resolution.ts:125-137`
  - body: This merge makes real coding-tool names reachable through `/tools/invoke`, but the default HTTP deny list still blocks legacy `fs_write`/`fs_delete`/`fs_move` names instead of canonical `write`, `edit`, and `process`. With the normal pass-through policy path, file mutation and process-control tools can become callable without the explicit `gateway.tools.allow` opt-in described by the PR/docs. Add the canonical names to the default deny list and cover them with resolver/HTTP tests using the real tool names.
  - confidence: 0.91
- **[P2] Preserve originals before description wrappers:** `src/agents/pi-tools.ts:768`
  - body: `createOpenClawCodingToolsRaw()` unwraps after `createOpenClawCodingTools()` has already applied deferred descriptions. That helper spreads `exec` and `process` into new objects, dropping the new non-enumerable original-tool symbol, so those two tools remain hook-wrapped on the HTTP path and still double-run `before_tool_call`. Build the raw list before hook wrapping or copy the original pointer through later wrappers, then add a real exec/process regression.
  - confidence: 0.88
