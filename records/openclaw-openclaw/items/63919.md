## Summary

Keep open. Current main still builds the Gateway /tools/invoke resolver from Gateway tools only, so this PR is not obsolete; it addresses a real gap supported by the HTTP deny-list/docs shape. The branch still needs maintainer/security iteration before merge because the proposed wiring routes hook-wrapped coding tools into a direct HTTP path that already runs hooks, and the surface can expose execution/file-mutation tools when explicitly allowlisted.

## What This Changes

The PR imports createOpenClawCodingTools into the Gateway tool resolver, merges coding tools into the HTTP-scoped /tools/invoke tool list with Gateway tools taking precedence, and adds a regression test with a stub coding tool.

## Best Possible Solution

Keep the capability alive, but land it only after the Gateway resolver uses an HTTP-safe coding-tool construction path or detects already hook-wrapped tools, drains or avoids adjusted-param retention on the HTTP path, preserves the loopback non-expansion guard, and adds regression coverage for allowlisted coding-tool availability, default deny behavior, owner/scoped auth semantics, single hook execution, adjusted-param cleanup, and Gateway-tool precedence.

## Risks / Open Questions

- The PR branch appears based on an older resolver shape and may need a rebase onto the current resolveSubagentToolPolicyForSession/isSubagentEnvelopeSession code before it can be reviewed as merge-ready.
- As proposed, HTTP coding tools would be created through createOpenClawCodingTools, whose tools are already wrapped with before_tool_call, while /tools/invoke also runs runBeforeToolCallHook before execution.
- Wrapped coding tools store adjusted params by toolCallId; current HTTP invoke code does not drain that cache after execution, unlike the embedded agent path.
- The PR hardcodes senderIsOwner: true when constructing coding tools, which should be checked against the current trusted-proxy/scoped-auth semantics even though the HTTP path applies owner-only filtering after resolution.
- The change can expose execution and file-mutation tools to authenticated HTTP callers when explicitly allowlisted, so secops/maintainer review is appropriate before merge.
