## Summary

Keep this PR open. Current main still lacks the native LINE sticker send path this PR adds: there is no `channelData.line.sticker` carrier, no `pushStickerMessage`/`createStickerMessage` helper, no outbound adapter branch for sticker dispatch, and no `STICKER:` directive parser. This is an active contributor PR paired with open tracker #64784, while the broader predecessor #65270 was closed unmerged in favor of this narrower replacement.

## Best Possible Solution

Keep the PR open for normal maintainer review. The best path is to validate the sticker helper and directive against the current LINE SDK/API contract, resolve the low-severity directive trust/design question, decide whether to split the coalescer fix, and then merge or request changes. After a send-side implementation lands, #64784 can continue tracking the broader inbound/catalog work that remains.

## Risks / Open Questions

- The PR adds a text directive that can be produced by model output; maintainers should decide whether line-anchored `STICKER:` directives are acceptable, need a trust gate, or should be structured-payload only.
- The PR combines a generic block-reply coalescer fix with a LINE-specific feature; maintainers may prefer landing the generic coalescer change separately.
- The proposed `supportsLineNativePayload` flag is LINE-specific but placed on a shared outbound adapter interface; this needs explicit maintainer agreement or a narrower contract shape.
- The PR has not been live-tested against a production LINE official account per its own verification notes, though it includes targeted unit coverage.
