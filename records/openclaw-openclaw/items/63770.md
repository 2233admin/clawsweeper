## Summary

Keep open: current main and the latest release still contain the at-only deletion guard, and the active linked PR is the correct implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue asks for explicit `deleteAfterRun: true` to delete successful `every` and `cron` jobs after their first run instead of only applying to `at` schedules.

## Best Possible Solution

Review and either land PR #64375 as the canonical narrow fix, with regression coverage and a user-facing changelog entry, or explicitly decide that recurring `deleteAfterRun` is unsupported and tighten the schema/docs/CLI wording so accepted inputs match runtime behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: create or patch an `every` or `cron` job with `deleteAfterRun: true`, then call the successful `applyJobResult` path; current main returns false because of the `kind === "at"` guard and the stored job is not removed.

## Solution Assessment

Yes, if maintainers accept the reported semantics. The narrow maintainable fix is to make `shouldDelete` depend on explicit `deleteAfterRun === true` plus successful completion, then add focused tests proving successful recurring jobs delete while failed recurring jobs remain for retry/inspection.
