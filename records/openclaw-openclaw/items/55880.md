## Summary

Keep this issue open. Current main has meaningful partial fixes for cron run delivery observability, staged package updates, and macOS launchd restart recovery, but the fresh 2026-04-28 report against v2026.4.26 describes the same post-upgrade LaunchAgent failure pattern after those paths already existed. The related closed cron issues cover recurring-job recovery after gateway restarts, not this broader macOS self-update, launchd recovery, provider-preflight, and user-visible notification reliability problem.

## Best Possible Solution

Keep the issue open and fix it as a targeted macOS self-update recovery bug: make post-update recovery share the safe doctor/gateway repair path for both installed-but-unloaded and service-not-installed LaunchAgent states, wait on a readiness signal long enough to avoid the startup race, and record failure-notification delivery attempts/results separately from the main cron run delivery state. Also verify whether missing optional provider or memory credentials can still block `openclaw update --yes --json`; if so, make that preflight warning-only unless the update path actually depends on those credentials.

## Risks / Open Questions

- Changing gateway recovery to reinstall or bootstrap services must preserve external-supervisor and user-managed gateway setups; the repair policy needs to remain explicit and conservative.
- Making notification delivery fully authoritative may require bounded waits or retry accounting so cron finalization does not hang on Discord/channel delivery.
- Provider credential preflight behavior needs a narrow reproduction before changing update semantics, because some doctor checks may be warnings while others intentionally block unsafe updates.
