## Summary

Keep open. Current main resolves the requested target agent and encodes it in the child session key, but the native and ACP task-run creation paths still do not pass that target as `agentId`; the task registry therefore falls back to the owner/requester session, which is the parent agent for cross-agent spawns. The reporter's `child_session_key` workaround remains valid, and the prior ClawSweeper comment is still directionally correct for the missing direct attribution field.

## What This Changes

The issue asks for spawned native and ACP subagent task rows to store the executing target agent in `task_runs.agent_id` so plugins can attribute active work correctly.

## Best Possible Solution

Plumb the resolved target agent id into future task-run creation for both native and ACP `sessions_spawn`, while keeping historical backfill or doctor repair as a separate product decision.

## Repair Work Prompt

Fix the `sessions_spawn` task attribution bug tracked by this issue. Current main resolves the target agent for native and ACP spawns, and SQLite already persists `TaskRecord.agentId`, but the native `registerSubagentRun` path and both ACP `createRunningTaskRun` paths omit explicit `agentId`, so cross-agent task rows fall back to the requester/parent agent. Implement a future-only fix: add `agentId` to `RegisterSubagentRunParams`, pass native `targetAgentId` from `spawnSubagentDirect`, include it in the native `createRunningTaskRun` call, and pass `agentId: targetAgentId` to both ACP `createRunningTaskRun` calls. Add focused regression coverage proving cross-agent native and ACP `sessions_spawn` task records, and preferably SQLite `task_runs.agent_id`, contain the target/executing agent rather than the parent. Do not add historical backfill, doctor repair, migrations, or broad task-registry inference changes unless a maintainer explicitly asks.
