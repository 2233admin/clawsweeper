## Summary

Keep open. The PR is maintainer-labeled and authored by a MEMBER, so it is not an auto-close candidate, and the patch has a blocking correctness issue: the new cancellation snapshot is written into the `agent:` dedupe namespace that the `agent` RPC already uses as its idempotency cache. The SDK event-filtering direction matches the documented split between normalized `run.events()` and `rawEvents()`, but the cancellation/wait fix needs a namespace or helper adjustment before merge.

## What This Changes

The PR modifies the SDK client to drop Gateway `chat` projection events from normalized run streams, adds cancellation lifecycle/wait snapshots for aborted Gateway runs, and updates SDK/gateway tests.

## Best Possible Solution

Keep the normalized-event filtering and cancellation lifecycle goal, but store aborted-run terminal state in a wait-only/non-idempotency location or extend the wait helper so `agent.wait` can observe cancelled chat/session runs without populating the `agent` RPC idempotency cache. Add coverage for aborting a `sessions.send` run and then starting an `agent` run with the same idempotency key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use a wait-only namespace for cancellation snapshots:** `src/gateway/server-methods/sessions.ts:1343`
  - body: The new `sessions.abort` snapshot is written as `agent:${firstAbortedRunId}`, but `agent` uses `agent:${idempotencyKey}` as its request cache and returns that payload before starting work. If a chat/session run is aborted and a later `agent` call reuses the same idempotency key, it can receive the synthetic timeout/cancel result instead of launching. Store this cancellation state outside the `agent:` idempotency namespace or make the wait helper read a separate cancellation namespace.
  - confidence: 0.9
