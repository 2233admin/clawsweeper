## Summary

Keep open. Current main still uses SKILL.md/frontmatter-based skills, ClawHub scan visibility, and generic plugin hooks; it does not implement the requested standardized skill.yaml permission manifest, install-time consent review, skills audit/trust grading, signing, or attestation. The recent maintainer comment and related open RFC/security threads make this active product/security design rather than conservative cleanup.

## What This Changes

The issue requests a standardized skill permission manifest with permission declarations, install-time review, audit/trust reporting, signing, and attestation support for skills.

## Best Possible Solution

A maintainer-designed, versioned skills-first capability manifest should land with parser/types, backward-compatible SKILL.md integration, direct install/update consent and audit display, ClawHub policy/display support, documented signing and attestation expectations, and a clear relationship to before_install and before_tool_call enforcement.

## Reproduction Assessment

Not applicable as a feature/security design request. The absence is high-confidence from static inspection of the current skills CLI, ClawHub installer, skill metadata types, plugin hook contract/docs, public skills docs, and focused searches for skill.yaml, audit, trust, signature, and attestation surfaces.

## Solution Assessment

Unclear whether the full proposed npm package model is the best direct implementation. The safer path is explicit maintainer RFC/design that can phase a small visible manifest vocabulary before deciding enforcement, trust grading, signing, and attestation policy.
