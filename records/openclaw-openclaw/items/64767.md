## Summary

Keep open. Current main still has the reported failure shape: key gateway/session readers synchronously materialize an entire session JSONL transcript into one string and split it with a regex before any response caps apply. The direct fix PR #64858 is closed and unmerged in the provided context, and the existing compaction byte guard is opt-in/preventive rather than a bounded reader for already-bloated transcripts.

## Best Possible Solution

Keep this issue open. The maintainer path is to add a bounded transcript-line iterator in the gateway/session transcript utilities and replace the full-string reads in readSessionMessages, readLatestSessionUsageFromTranscript, and nearby direct transcript scanners such as transcriptHasIdempotencyKey. Add focused regression coverage for files larger than the internal chunk size, CRLF and chunk-boundary cases, malformed lines, compaction markers, idempotency lookup, and usage aggregation. Keep the existing maxActiveTranscriptBytes/truncateAfterCompaction mitigation as preventive compaction behavior, but do not treat it as the fix for already-bloated transcripts or full-history readers.
