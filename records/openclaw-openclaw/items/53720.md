## Summary

Keep open. Current main still lacks the requested per-agent suspension contract, and this PR is a relevant implementation candidate, but it is not merge-ready because the advertised full suspension semantics still miss background schedulers and the new JSON CLI output can be polluted by human logs.

## What This Changes

The branch adds optional `agents.list[].enabled`, `openclaw agents disable/enable` commands, disabled status in agent list output, schema/docs/generated baseline/changelog updates, and routing/config regression tests.

## Best Possible Solution

Land a refreshed implementation that defines a typed, schema-validated `agents.list[].enabled?: boolean` contract, keeps disabled agents visible to management operations, consistently excludes them from routing plus cron/heartbeat/channel polling, preserves machine-readable CLI output, updates generated docs, and adds regression coverage for config preservation and scheduler suspension.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor disabled agents in cron and heartbeat:** `src/config/types.agents.ts:62-64`
  - body: The new contract says `enabled: false` fully suspends an agent, but this branch does not update `src/gateway/server-cron.ts` or `src/infra/heartbeat-runner.ts`. Those paths still derive work from raw configured agent entries, so cron jobs and heartbeat settings for disabled agents keep running.
  - confidence: 0.9
- **[P2] Route empty enabled sets to fallback:** `src/routing/resolve-route.ts:140-142`
  - body: Skipping disabled agents from the lookup can leave the map empty, but `pickFirstExistingAgentId` still returns the requested ID in the empty-map path. If every configured agent is disabled and a binding targets one, routing continues to that disabled agent instead of falling back.
  - confidence: 0.87
- **[P2] Keep disable and enable JSON output parseable:** `src/commands/agents.commands.disable.ts:58-60`
  - body: `agents disable --json` logs the config update before writing the JSON payload, and the enable command mirrors the same pattern. Because `runtime.log` writes to stdout, callers parsing stdout as JSON will fail; guard the human log the same way existing add/delete JSON paths do.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No dependency, workflow, permission, or secrets-handling regression was found, but the incomplete suspension control can leave background agent activity running after an operator disables an agent.

Concerns:

  - body: The PR documents `enabled: false` as stopping routing, cron, heartbeat, and channel polling, but cron and heartbeat are not enforced by the patch. Operators could rely on the new flag for incident response or governance while background runs continue.
