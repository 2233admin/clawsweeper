## Summary

Current main still does not support Workers AI model selection for `openclaw onboard --non-interactive --auth-choice cloudflare-ai-gateway-api-key`. The bundled Cloudflare plugin only accepts account ID, gateway ID, and API key, then writes an Anthropic Messages provider using the Claude Sonnet default and the `/anthropic` Gateway URL. The issue is concrete, recently re-reviewed by a maintainer/Codex comment, and still has no implemented equivalent in the Cloudflare onboarding path.

## Best Possible Solution

Keep this open and implement the requested support in `extensions/cloudflare-ai-gateway`: add a Workers AI endpoint/model selection mode for non-interactive onboarding, generate an OpenAI-compatible provider config for Workers AI model refs, keep the existing Anthropic default path backwards-compatible, update Cloudflare provider and CLI automation docs, and add focused tests for the generated config and flags.

## Risks / Open Questions

- The fix should preserve the existing Anthropic-through-AI-Gateway onboarding path and not silently reinterpret existing `cloudflare-ai-gateway` profiles as Workers AI profiles.
- Cloudflare documents more than one viable route for Workers AI through AI Gateway: provider-specific `/workers-ai/v1/chat/completions` and the unified compat route with `workers-ai/@cf/...` model ids. The implementation should choose one deliberately and document the resulting model ref shape. ([developers.cloudflare.com](https://developers.cloudflare.com/ai-gateway/usage/providers/workersai/))
- Cloudflare API key semantics differ between upstream Anthropic routing and Workers AI routing; tests should cover generated base URL, API compatibility, model id, auth profile metadata, and default model behavior.
