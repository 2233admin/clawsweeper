## Summary

Keep open. Current main at 5de284c2e30f915b769fac08dbeab45f57749702 still derives lastMessagePreview from transcript tail user/assistant messages without filtering diagnostic.* entries, and the current tests do not cover diagnostic preview pollution. The live PR patch is narrow, adds the diagnostic.* filter plus regression coverage and a changelog entry, and remains an active merge-ready candidate after maintainer follow-up. Related #66544 and the closed bug thread cover display-name/origin metadata paths, not this includeLastMessage preview path.

## Best Possible Solution

Keep this PR open for maintainer review and likely landing, or land an equivalent focused change that filters diagnostic transcript entries inside readLastMessagePreviewFromOpenTranscript with the included regression coverage and changelog credit. Do not close it as obsolete unless maintainers explicitly decide the includeLastMessage preview surface should allow diagnostic entries despite the current merge-ready branch.
