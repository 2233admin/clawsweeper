## Summary

Current main still has the read-image-to-Telegram media gap. Telegram delivery can send media once a reply payload has mediaUrl/mediaUrls, and the tool handler can queue structured tool media, but createOpenClawReadTool only normalizes/sanitizes the Pi read result and does not add details.media, details.path, or MEDIA text for the successfully read image path. The shared extractor still explicitly returns no media for Pi-style read image content with no path, and the regression test documents that empty result.

## Best Possible Solution

Keep this issue open and implement a trusted read-tool media bridge. The likely fix belongs in createOpenClawReadTool or trusted read-tool result handling: when a read call successfully returns image content for a validated path, attach structured media metadata such as details.media.mediaUrl/details.media.mediaUrls with trustedLocalMedia, preserving sandbox/workspace allowlist semantics. Add regression coverage for a Pi-style read image result with no MEDIA text and no details.path flowing into queued mediaUrls, plus negative coverage proving untrusted tools and alias tool names cannot attach local files.
