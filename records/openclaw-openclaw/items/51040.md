## Summary

Current main still does not make macOS Voice Wake respect the session selected in the macOS chat/session picker. Voice Wake forwards with default `sessionKey: "main"` and `channel: .webchat`; the overlay, wake fallback, and push-to-talk fallback paths pass only the wake trigger or no options. The merged trigger-based routing work in #30354 is related, but it is a separate gateway configuration layer and does not bind the current macOS picker selection to Voice Wake delivery.

## Best Possible Solution

Keep this issue open and fix the macOS Voice Wake send path. The best implementation is to make the macOS chat/session picker publish its active selected session target through an app-owned seam, resolve the selected session's delivery route at Voice Wake send time, pass the explicit `sessionKey` plus route fields into `VoiceWakeForwarder.ForwardOptions`, and add a regression test for a selected non-WebChat session target. Preserve the existing trigger-based gateway `voicewake.routing` behavior for cases with no explicit selected non-main target.
