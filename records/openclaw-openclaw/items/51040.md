## Summary

Current main still does not make macOS Voice Wake respect the selected session target. The macOS Voice Wake forwarder defaults to the main WebChat route, the overlay and fallback send paths pass only the wake trigger, and the active WebChat session state used by Talk Mode is not consulted by the Voice Wake path. Gateway-level voicewake.routing exists, but it is a separate trigger-based config layer, not a binding to the macOS session picker described in this issue.

## Best Possible Solution

Keep this issue open. Fix the macOS Voice Wake send path to resolve the selected session target at send time and pass the resulting sessionKey, channel, and to into VoiceWakeForwarder.ForwardOptions, or explicitly bind the macOS picker to Gateway voicewake.routing and document that behavior. Add a macOS regression test for a non-main selected session target.
