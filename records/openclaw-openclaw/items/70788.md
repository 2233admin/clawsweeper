## Summary

Keep open. Current main still generates a Windows Startup-folder `.cmd` launcher that uses `start "" /min cmd.exe /d /c`, writes no silent sidecar, and has tests that lock in the flashing launcher shape; the related Microsoft tracker is only a tracking PR, not a superseding implementation.

## What This Changes

Suppress the visible Windows Startup-folder gateway launcher flash by replacing the minimized `cmd.exe` handoff with a silent launcher while preserving service detection and cleanup.

## Best Possible Solution

Implement the fix in the core Windows daemon Startup-folder fallback: preserve the existing `OpenClaw Gateway.cmd` entry for detection, delegate login startup to a no-window launcher or equivalent hidden mechanism with robust path quoting, remove any sidecar on uninstall, and update focused fallback tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by forcing the `schtasks` fallback path and inspecting the generated Startup-folder script; the current test already does this and asserts the minimized-cmd command reported as flashing.

## Solution Assessment

Yes on the fix boundary. Replacing the Startup-folder `start /min cmd.exe` handoff in `src/daemon/schtasks.ts` is the narrow maintainable solution, while the specific `wscript` sidecar should be vetted for quoting and platform longevity.
