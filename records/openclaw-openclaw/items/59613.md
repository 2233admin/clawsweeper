## Summary

Keep open. Current main still matches the reported Docker sandbox behavior: the setup path defaults sandbox installs to `workspaceAccess: "none"`, the sandbox workspace mount remains read-only for that mode, and write-capable tools can still be exposed before the filesystem bridge rejects writes. The related #59718 is adjacent but narrower; it tracks writable custom bind mounts, not the default per-scope sandbox workspace contract reported here.

## Best Possible Solution

Keep this open for sandbox maintainer triage. The best fix is to make the `workspaceAccess: "none"` contract explicit and consistent: either make the per-scope sandbox workspace writable while keeping the real agent workspace hidden, or suppress mutating file tools and clearly document that the sandbox workspace is read-only. Coordinate separately with #59718 for the narrower writable-custom-bind authorization model.
