## Summary

Keep open. The PR addresses a real gateway shutdown reliability gap, but current main only partially overlaps with it and the PR diff would regress HTTP restart safety by letting shutdown finish while the HTTP listener can still be bound.

## What This Changes

The PR adds a 5s gateway shutdown timeout helper and tests, then wraps close-handler subsystem, WebSocket, and HTTP teardown calls in that timeout with optional warning logging.

## Best Possible Solution

Keep the current shutdown result and HTTP/WebSocket close semantics, then add a bounded shutdown-step helper only for subsystem stops that can block later cleanup. The helper should record timeout warnings, keep late promise settlement observed, and include focused tests for hung subsystem continuation and HTTP restart safety.

## Reproduction Assessment

Yes. The PR-introduced HTTP regression can be reproduced with a unit fake whose httpServer.close callback remains pending past 5s, then checking that close returns before the listener has closed.

## Solution Assessment

No. The direction is useful, but this diff is not the best fix because it treats HTTP close timeout as successful teardown; the safer solution is to adapt current main's shutdownStep and HTTP force-close model instead of racing every teardown step identically.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep HTTP close pending through forced connection close:** `src/gateway/server-close.ts:185-189`
  - body: This races httpServer.close() against the 5s subsystem timeout and lets shutdown complete when the timeout wins. The restart loop starts follow-up restart handling as soon as close() returns, so a listener that is still draining can keep the port bound and make the next gateway startup fail. Preserve the current grace period, force-close connections, and then wait for the listener or fail shutdown.
