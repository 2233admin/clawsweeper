## Summary

Keep open. Current main still preserves `source: "sessions"` internally but hard-codes surfaced built-in `memory_search` hits to `corpus: "memory"`, so the reported mislabeling remains. The KinthAI discussion usefully confirms downstream provenance impact, and open PR #72886 uses closing syntax for this issue while #71898 also targets the same bug, so the maintainer path is to review or choose one focused fix rather than close the report now.

## What This Changes

Fix `memory_search` so session-transcript hits with `source: "sessions"` are surfaced with `corpus: "sessions"` instead of `corpus: "memory"`.

## Best Possible Solution

Land one focused memory-core change that derives surfaced built-in result `corpus` from `result.source` after visibility filtering and result decoration, widens the local result type to allow `"memory" | "sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, wiki supplement behavior, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `memory.manager.search` so it returns a `MemorySearchResult` with `source: "sessions"`, execute `memory_search`, and inspect `details.results`; current main’s mapper still returns `corpus: "memory"` for that hit.

## Solution Assessment

Yes. Deriving the surfaced corpus from the existing `result.source` is the narrowest maintainable fix because the backend already records provenance and the public schema already accepts `corpus: "sessions"`; the broader index-time tagging redesign discussed in comments is not required for this bug.
