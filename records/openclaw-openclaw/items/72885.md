## Summary

Keep open: current main still preserves session provenance internally but flattens every surfaced built-in `memory_search` hit to `corpus: "memory"`. The issue is valid and already has open linked fixes, including #72886 with closing syntax and the smaller overlapping #71898, so it should stay open until one focused fix lands.

## What This Changes

The issue asks for `memory_search` to surface built-in session transcript hits with `corpus: "sessions"` instead of labeling them as durable memory hits.

## Best Possible Solution

Land one focused memory-core fix that derives the surfaced built-in result corpus from `result.source`, widens the surfaced result type to include `"sessions"`, and adds regression coverage for session-only, memory-only, and mixed result sets without changing ranking, visibility filtering, citation decoration, or recall tracking.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `memory.manager.search` so it returns a `MemorySearchResult` with `source: "sessions"`, execute `memory_search`, and inspect `details.results`; current main’s mapper still returns `corpus: "memory"` for that surfaced hit.

## Solution Assessment

Yes. Deriving the surfaced corpus from the existing `result.source` is the narrowest maintainable fix because the backend already records provenance and the public schema already accepts `corpus: "sessions"`; the broader KinthAI index-time tagging model is useful context but larger than needed here.
