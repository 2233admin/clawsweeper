## Summary

Keep this open: current main still preserves session hit provenance internally but flattens every surfaced built-in `memory_search` result to `corpus: "memory"`. The issue is also paired with open closing PR #72886, while the narrower #72926 is closed unmerged, so the remaining maintainer action is PR review or a narrower replacement rather than cleanup closure.

## What This Changes

The issue asks `memory_search` to surface session-transcript hits as `corpus: "sessions"` when the underlying search result has `source: "sessions"`.

## Best Possible Solution

Land a focused memory-core fix that widens the surfaced built-in result corpus type to `"memory" | "sessions"` and derives `corpus` from `result.source` after visibility filtering, with tools-level regression coverage for session-only, memory-only, and mixed result sets while leaving wiki supplements and source filtering unchanged.

## Reproduction Assessment

Yes. A high-confidence reproduction is to mock `memory.manager.search` to return a visible `MemorySearchResult` with `source: "sessions"` and call `memory_search`; current main maps that surfaced hit to `corpus: "memory"` at `extensions/memory-core/src/tools.ts:281`.

## Solution Assessment

Yes. Mapping surfaced corpus from the already-preserved `result.source` at the tool boundary is the narrowest maintainable fix; the open PR should be reviewed, or replaced with a smaller equivalent if its unrelated file churn is unacceptable.
