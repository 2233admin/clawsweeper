## Summary

Keep open. Current main has narrowed the original fallback scenario by treating user and legacy session model overrides as strict selections, but the requested delivery-hook provenance is still missing: `message_sending` receives outbound content plus channel/account/media routing only, while the provider/model-bearing `llm_output` hook is asynchronous observation and cannot enforce or rewrite delivery. The related discussion correctly points to an architectural API change, and open PRs such as #65149 and #39761 overlap with the same hook/provenance surface rather than superseding this issue.

## What This Changes

The issue requests gateway-level access to actual runtime provider/model identity at outbound delivery time so self-tagged model labels can be verified, corrected, blocked, or audited.

## Best Possible Solution

Define one additive, documented provenance contract for final outbound replies that carries the actual resolved provider/model/ref from the agent run into the delivery-time hook or a final-reply hook, then cover fallback, strict user override, auto fallback, queued delivery, and Telegram delivery behavior with focused tests. Coordinate this with the open hook-context PRs instead of landing a small local metadata patch that only covers one path.

## Reproduction Assessment

Yes for the remaining source-level issue: current `message_sending` hook inputs lack runtime provider/model while `llm_output` has that data only as non-blocking observation. The exact old legacy-override fallback steps should be revalidated because current main disables configured fallbacks for non-auto session overrides.

## Solution Assessment

No clear current fix is present. The best direction is a coordinated additive hook/provenance API; a three-line metadata patch to `applyMessageSendingHook` is probably too narrow unless it also proves the provider/model source is actual runtime identity across final reply, streaming, fallback, and queued delivery paths.
