## Summary

Keep #69110 open. Current main still does not pass actual runtime provider/model identity into `message_sending`, and there is no gateway enforcement that strips, overwrites, blocks, or alerts on agent-authored model tags that disagree with the runtime model. The related discussion is accurate: provider/model data exists in the agent/model telemetry and response-prefix paths, but it is not bridged into the outbound delivery hook/enforcement point. Related PR #65149 appears to target a plausible hook surface, but it is open and not present on main, so it does not supersede the issue yet.

## Best Possible Solution

Keep #69110 open as the canonical bug until main has a coordinated provenance/enforcement path. The likely fix belongs in the reply-to-delivery pipeline: carry resolved `providerUsed`/`modelUsed` from the agent run into a final outbound hook or `message_sending` metadata, then add focused tests for fallback from selected model to actual model and for Telegram/direct delivery hook payloads. If #65149 or a successor lands this comprehensively, close this issue then with the shipped commit/release evidence.
