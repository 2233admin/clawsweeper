## Summary

Keep open. Current main has some related truthful-model plumbing for response-prefix templating and session usage, but the reported delivery-hook gap remains: `message_sending` still receives channel/account/media routing data without the resolved runtime provider/model, and the typed hook event/context has no model provenance field.

## What This Changes

The issue requests gateway-level runtime model/provider identity to be passed to or enforced by outbound delivery hooks so visible model tags cannot silently diverge from the model that generated a reply.

## Best Possible Solution

Define one backwards-compatible provenance contract for final outbound replies, then thread resolved provider/model identity from the agent runner through final reply assembly into the chosen hook surface. The fix should cover normal final replies, followups, CLI-delivered agent replies, and channel direct-delivery paths, with tests proving hooks can see truthful runtime identity after fallback without trusting model-authored text.

## Reproduction Assessment

Partially yes. Source inspection gives a high-confidence reproduction for the missing `message_sending` metadata: register a hook and the event cannot contain runtime provider/model because the type and call site do not pass it. The exact live fallback sequence from the report is less certain on current main because user/legacy model overrides are now documented and tested as strict selections.

## Solution Assessment

No, the issue’s suggested three-line metadata patch is probably not the best complete fix. The current code shows model identity lives in the runner/final-reply layer, while `message_sending` is a lower delivery layer; a safer solution is a coordinated provenance hook or additive message context contract that avoids inconsistent behavior across reply paths.
