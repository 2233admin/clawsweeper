## Summary

Current main still has the reported product gap: `tools.fs.workspaceOnly` continues to guard `read` to the active workspace, while the skills prompt can still advertise managed/shared skill locations outside that workspace. The newer sandbox skill sync helps sandbox `workspaceAccess` modes, but it does not provide a general read-only skill-root exception for host workspace-only runs.

## Best Possible Solution

Keep the issue open. The likely maintainer path is a small core change in the filesystem policy and skill prompt/runtime boundary: either add a documented read-only skill-root exception for eligible skill files, or rewrite/sync prompt paths for workspace-only agents in a way covered by schema, docs, and regression tests for managed, personal, project, workspace, plugin, and extra skill roots.

- A fix should keep write isolation intact and limit any read exception to resolved, eligible skill roots rather than arbitrary host paths.
