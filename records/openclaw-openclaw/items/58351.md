## Summary

Keep open. Current main still allows the skills catalog to advertise managed/shared skill paths outside the agent workspace, while `tools.fs.workspaceOnly: true` wraps `read` with a strict workspace-root guard, so the reported failure remains plausible and source-backed.

## What This Changes

The issue requests that agents using `tools.fs.workspaceOnly: true` can still read globally installed skill files, either through a read-only allowlist, a skills-directory exemption, or workspace-relative prompt paths.

## Best Possible Solution

Keep the issue open and choose a maintainer-approved boundary that lets models read only the skill files advertised to them without broadening write access or exposing unrelated OpenClaw state. A good end state would align source, config schema/help, docs, and tests for managed skills, personal `.agents/skills`, extra dirs, plugin skills, and sandboxed/non-sandboxed runs.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: configure a managed skill outside the workspace, build the skills prompt so it lists that `SKILL.md`, then run the `read` tool with `tools.fs.workspaceOnly: true`; current main’s guard rejects the outside-workspace path before the read executes.

## Solution Assessment

Unclear. The report’s `allowRead` and auto-exempt options are viable but security-sensitive; the safest maintainable direction needs maintainer choice between explicit read-only allowlists and prompt/materialization behavior that keeps `workspaceOnly` strict.
