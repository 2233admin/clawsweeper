## Summary

Keep open, but do not merge as-is. Current main still lacks a generic `api.runtime.gateway` dispatcher, so the contribution is not redundant; however, the patch exposes the internal Gateway method bus as a public plugin runtime API without the authorization, caller identity, docs, and contract-test work that this boundary needs. The author did address Greptile's stale error-message note, and the underlying use case overlaps the still-open #39000 design discussion.

## What This Changes

The PR exports the internal Gateway dispatcher and adds `api.runtime.gateway.dispatchMethod<T>(method, params)` to plugin runtime types and Gateway runtime options, with a non-Gateway fallback error.

## Best Possible Solution

Define a narrow, documented plugin runtime contract for the sanitized-history and inter-plugin request/response use cases, with explicit allowed methods or capability checks, caller identity propagation, fallback behavior, public SDK docs, and contract tests coordinated with #39000.

## Reproduction Assessment

Not applicable as a feature/API PR rather than a bug report. The high-confidence review path is static: current main lacks `api.runtime.gateway`, and the PR diff adds a broad required runtime method without the surrounding authorization, identity, docs, and test-helper updates.

## Solution Assessment

No. Exposing the internal dispatcher directly is not the narrowest maintainable solution; a scoped or capability-checked helper, or a purpose-built sanitized history/inter-plugin service API, would preserve the existing Gateway trust model better.
