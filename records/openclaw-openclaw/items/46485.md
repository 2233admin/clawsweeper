## Summary

Keep open. Current main still does not expose `api.runtime.gateway.dispatchMethod`, and the motivating gap is real: plugin `getSessionMessages` still goes through raw `sessions.get` while `chat.history` has separate display-normalization and payload bounding. The PR is plausible, but exposing an arbitrary gateway RPC dispatcher is a public plugin API and security-sensitive authority change that needs maintainer review, tests, docs, and contract coverage before merge.

## Best Possible Solution

Keep this PR open for maintainer API/security review. The best path is either a constrained, documented `api.runtime.gateway.dispatchMethod` contract with explicit scope semantics, owner identity, tests, docs, and SDK baseline updates, or a narrower sanitized session-history helper that gives plugins `chat.history` semantics without exposing the whole gateway RPC surface.

## Risks / Open Questions

- Exposing a raw gateway dispatcher is broader than the current narrow `subagent` and `nodes` helpers and could let plugins invoke unrelated read/write RPCs under fallback synthetic scope or inherited request scope.
- The motivating use case may be better served by a dedicated sanitized session-history runtime helper instead of a general gateway method bus.
- The PR changes public plugin runtime types without corresponding docs, SDK baseline/contract tests, or test-runtime mock updates in the provided changed file list.
