## Summary

Keep open: current main still lacks the proposed plugin runtime gateway dispatcher, but the PR changes a public Plugin SDK and Gateway authorization boundary and is not ready to merge without maintainer API/security review.

## What This Changes

The PR exports the internal gateway dispatcher, wires it into plugin runtime options, and adds a required `PluginRuntime.gateway.dispatchMethod<T>(method, params)` fallback surface.

## Best Possible Solution

Land a reviewed plugin runtime contract that preserves the useful sanitized-history and cross-plugin request/response use cases while making caller identity, allowed methods/scopes, fallback behavior, docs, and contract tests explicit.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Add gateway to runtime test mocks:** `src/plugins/runtime/types.ts:62-65`
  - body: Adding a required `PluginRuntime.gateway` member makes `createPluginRuntimeMock` incomplete because its base object is typed as `PluginRuntime` and currently only provides `subagent` and `nodes` at the end. Add a default `gateway.dispatchMethod` mock or make the new runtime surface optional before changing the public type.
  - confidence: 0.9
- **[P2] Preserve plugin identity for dispatches:** `src/plugins/runtime/types.ts:62-65`
  - body: The registry proxy only wraps `runtime.subagent` calls in `withPluginRuntimePluginIdScope`; other runtime properties pass through unchanged. Exposing `runtime.gateway.dispatchMethod` this way lets service/event callers use fallback dispatch without `pluginRuntimeOwnerId`, so plugin-owned authorization and audit paths cannot attribute the caller.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff exposes a broad in-process Gateway RPC dispatcher to plugin code and needs explicit authorization plus plugin-identity semantics before merge.

Concerns:

- **[medium] Arbitrary Gateway RPC dispatch from plugins:** `src/plugins/runtime/types.ts:62`
  - body: The new method accepts any gateway method name and params, while the existing dispatcher can use the current request client or a synthetic operator client. That widens plugin authority from curated helpers to the Gateway method bus, including read and write-scoped methods, without an allowlist or capability contract.
  - confidence: 0.87
- **[medium] Missing plugin caller attribution:** `src/plugins/registry.ts:1978`
  - body: Current runtime proxy code attaches plugin identity only around `runtime.subagent`; a pass-through `runtime.gateway` helper would not automatically set `pluginRuntimeOwnerId`, weakening ownership checks and auditability for fallback dispatches.
  - confidence: 0.8
