## Summary

Keep open. Current main still does not expose `api.runtime.gateway.dispatchMethod` or an equivalent sanitized `chat.history` runtime helper to plugins. The PR targets a real plugin runtime API gap, but its broad arbitrary gateway-RPC dispatcher needs maintainer API/security review before it can be treated as the best fix.

## Best Possible Solution

Keep this PR open for maintainer review or replace it with a narrower plugin runtime contract. The likely path is either a constrained `api.runtime.gateway.dispatchMethod` with explicit scope, owner, tests, docs, and API baseline coverage, or a dedicated sanitized session-history helper that preserves `chat.history` semantics without exposing the whole gateway method bus.

## Risks / Open Questions

- The proposed `dispatchMethod(method, params)` API could let plugins invoke arbitrary read/write gateway RPCs under the current request or fallback synthetic operator context, which is broader than today’s narrow `subagent` and `nodes` helpers.
- A narrower sanitized-history helper may satisfy the motivating `chat.history` use case with less authority, but that product/API choice is not implemented on main.
- The PR as provided changes public runtime types without matching docs or contract tests in the changed file list.
