## Summary

Keep open. Current `main` and the latest release still appear affected: the TaskFlow SQLite migration copies legacy `owner_session_key` into `owner_key` but does not rebuild or drop the legacy NOT NULL column, while new flow inserts write only `owner_key`. The open implementation PRs #65767 and #63893 target this exact migration gap, so the issue should stay open until one canonical fix lands.

## What This Changes

The issue reports that `plugins/webhooks` `create_flow` fails on upgraded SQLite `flow_runs` tables because a legacy `owner_session_key` NOT NULL column remains while current code inserts only `owner_key`.

## Best Possible Solution

Land one canonical TaskFlow SQLite migration fix that rebuilds or otherwise normalizes legacy `flow_runs` tables, preserves existing rows, maps legacy owner data into `owner_key`, removes the blocking legacy NOT NULL column, and adds regression coverage for legacy schemas plus webhook-driven managed flow creation.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a legacy `flow_runs` table with `owner_session_key TEXT NOT NULL`: current `ensureSchema()` leaves that constraint in place, then `createManagedTaskFlow` persists through an insert that omits `owner_session_key`.

## Solution Assessment

Yes, the narrowest maintainable fix belongs in the TaskFlow SQLite migration/store layer. Changing the webhook request contract to require owner fields would contradict the documented route-bound session model and would not repair upgraded databases.
