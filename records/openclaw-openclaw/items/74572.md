## Summary

Keep open. Current main still has the reported Active Memory / Memory Core composition gap: Active Memory starts a lightweight embedded recall run with memory tools in `toolsAllow`, but the embedded runner only applies that allowlist after tool construction, while plugin tool materialization still receives policy allowlists that do not include the runtime `toolsAllow`. Open PR #74592 already uses closing syntax for this issue, so the issue should remain open until that PR is reviewed and merged or rejected.

## What This Changes

The issue requests that Active Memory lightweight embedded recall runs compose with Memory Core plugin tools and avoid persisting a failed placeholder assistant turn when the memory tool allowlist resolves empty.

## Best Possible Solution

Review and land or reject #74592. The desired end state is a generic fix where explicitly allowlisted plugin tools can be materialized for embedded runs before the final runtime allowlist filter, without hardcoding Memory Core IDs into core behavior.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to enable Active Memory and Memory Core, trigger an Active Memory pre-summarizer turn, and observe that current main applies the memory `toolsAllow` only after plugin tools have already been selected without that runtime allowlist.

## Solution Assessment

Yes for the triage route. Keeping this open behind #74592 preserves the contributor PR review and GitHub auto-close flow; the fix should remain generic to plugin tool materialization and embedded-run binding rather than special-casing Memory Core.
