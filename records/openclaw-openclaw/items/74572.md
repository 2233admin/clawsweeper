## Summary

Keep open. Current main still has the reported Active Memory and Memory Core composition shape, and open PR #74592 references this issue with closing syntax, so this issue should stay open until that PR is reviewed and merged or rejected. The related closed Active Memory reports cover different MCP over-inclusion and timeout paths, not this plugin-tool visibility failure.

## What This Changes

The issue reports that Active Memory lightweight embedded recall runs cannot see Memory Core plugin tools, causing the explicit memory tool allowlist to resolve empty and persist a failure placeholder turn.

## Best Possible Solution

Review and land or close #74592. The desired end state is a generic, extension-agnostic fix where Active Memory recall can materialize only explicitly allowlisted memory plugin tools before the final embedded-run filter, and unavailable memory tools do not leave a persistent user-visible placeholder in transcripts.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to enable active-memory and memory-core, trigger the Active Memory pre-summarizer, and observe that current main applies the memory toolsAllow after tool construction while runtime toolsAllow does not participate in plugin tool materialization.

## Solution Assessment

Yes for the triage route: keeping this issue open behind #74592 is the narrowest maintainable path. The implementation should stay generic to plugin-tool materialization and avoid hardcoding Memory Core behavior in core.
