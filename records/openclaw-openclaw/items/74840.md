## Summary

Keep open: the linked cron summary bug remains valid, and this patch does not solve the all-error-payload path. Cron still classifies a standalone error payload as fatal and then overwrites the final assistant text with the raw error; the prior #74807 ClawSweeper comment also identified that the repair needs a recovered-vs-unresolved mutating-tool distinction, which this PR does not add.

## What This Changes

This PR changes the isolated cron outcome fallback so `finalAssistantVisibleText` is considered before the first payload text when payload-derived summary or output text is missing.

## Best Possible Solution

Repair the cron/embedded-runner boundary so the final assistant-visible reply wins only for a proven recovered/self-corrected mutating-tool failure, while real unresolved failures remain fatal, and cover that boundary with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction is `resolveCronPayloadOutcome` with only `isError: true` payloads, `finalAssistantVisibleText`, and `preferFinalAssistantVisibleText: true`; current tests already assert that this returns the raw error, and the PR's fallback-only change does not reach the fatal override.

## Solution Assessment

No. The PR is not the best fix because it changes fallback text selection after payload summary extraction, but the reported path is decided by fatal structured-error precedence; the safer fix is to classify recovered versus unresolved mutating failures before producing or honoring a fatal cron payload.
