## Summary

Keep open. Current main has adjacent reliability work: per-session serialization, in-process inbound follow-up queueing, restart-specific resume, incomplete-turn retries, and durable transport retry after a reply payload exists. It still does not implement the requested durable per-session pending-deliverables/commitment queue that would resume a promised output after arbitrary task switching without a gateway restart.

## Best Possible Solution

Keep the issue open as a core session/turn-continuity bug. The best path is a typed per-session durable pending-deliverable record, possibly backed by Task Flow, written before preemptive task switches or interrupting operational work and cleared only when the deliverable is sent, explicitly deferred, cancelled, or superseded. Add regression coverage that simulates a Telegram or generic direct-chat long task, an unrelated high-priority interruption without gateway restart, and automatic resumption or visible deferral of the original promised output.
