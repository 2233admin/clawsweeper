## Summary

Keep open. The Gateway cleanup targets a real current-main leak path, but the PR still has a blocking correctness issue in the added tool-start cleanup: it can delete active tool metadata before the matching end event arrives.

## What This Changes

The PR centralizes Gateway chat buffer cleanup through `ChatRunState.clearRun`, tracks `bufferUpdatedAt` for raw-only buffers, expands stale-buffer maintenance, and adds TTL/max-entry sweeping for agent `toolStartData` with tests.

## Best Possible Solution

Keep the Gateway `clearRun` and raw-buffer timestamp direction, but make `toolStartData` cleanup active-run aware or lifecycle-driven so long-running tools keep their start args and duration until their end event; add a regression for a tool that runs longer than the TTL and still emits a complete `after_tool_call` event.

## Reproduction Assessment

Yes for the review finding. On the PR branch, start one tool, advance time beyond 30 minutes, start another tool so the global sweep runs, then end the first tool; `handleToolExecutionEnd` no longer has `startData`.

## Solution Assessment

No. The Gateway buffer cleanup is a narrow maintainable direction, but a global TTL for tool-start records is not the best fix because it cannot distinguish orphaned starts from active long-running tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve active tool starts beyond the TTL:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:656`
  - body: The new sweep runs on every tool start and deletes any `toolStartData` record older than 30 minutes, even if that tool is still running in another active run. When its matching end event later arrives, `handleToolExecutionEnd` has no `startData`, so `after_tool_call` gets empty params/no duration and start-arg-dependent bookkeeping can be skipped. Please only reap records once the owning run is inactive or otherwise known orphaned.
  - confidence: 0.87
