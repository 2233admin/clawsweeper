## Summary

Keep this PR open. Current main still does not implement the proposed `doctor.memory.remHarness` gateway RPC; REM previewing is present as a local `openclaw memory rem-harness` CLI surface, but not as a remote gateway/control-plane method. The PR remains an active, focused implementation candidate, with the main maintainer question being whether this REM-derived diagnostic payload should be exposed under `operator.read`.

## Best Possible Solution

Keep this PR open for normal maintainer review. If remote read-only REM diagnostics are desired, land a gateway implementation that registers `doctor.memory.remHarness`, scopes it deliberately under `operator.read`, routes through memory-core public/facade exports instead of deep internals, preserves CLI/REM lookback parity, bounds response size and work, and includes targeted gateway, facade, memory-core, docs, and changed-gate/build proof. If maintainers want REM previewing to remain CLI-only, close the PR only with that explicit product decision.
