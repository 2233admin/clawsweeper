## Summary

Keep this PR open. Current main still does not expose provider rate-limit headroom in `openclaw models status --probe`, so the PR is not obsolete or implemented. It remains a plausible implementation candidate, but it needs rebase/design/security review before merge because it adds a second live provider request path using credentials.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is a rebase onto current main, an explicit opt-in `--rate-limits` contract through both CLI parsing paths, rate-limit header capture through the existing provider runtime/transport seam or a narrow shared seam that preserves provider config and network policy, docs at `docs.openclaw.ai/cli/models`, and focused tests for option parsing, JSON/table output, provider base URL normalization, and transport failure cleanup.

## Risks / Open Questions

- Merging the PR as-is may send live provider credentials through a second direct HTTP path instead of the existing provider runtime/transport path, so proxy, TLS, attribution headers, custom provider config, and plugin-owned request behavior need review.
- Current main has both Commander and routed CLI surfaces for `models status`; the feature must be wired through both or it will be inconsistent.
- The PR creates a scriptable JSON/table contract and extra billable live provider calls, so docs and focused tests should accompany the implementation.
- The PR includes unrelated `.gitignore` churn for `openclaw-*.log`, which should be removed or justified separately.
