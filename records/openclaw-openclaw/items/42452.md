## Summary

Keep this PR open. Current main still does not expose provider rate-limit headroom in `openclaw models status --probe`, so the PR is not obsolete. It remains a relevant implementation candidate, but the current diff is stale against the routed CLI/docs surface and needs maintainer design/security review because it adds a second direct provider request path for live credentials.

## Best Possible Solution

Keep this PR open as a candidate, but require a rebase and design pass. The best path is an opt-in `--rate-limits` contract wired through both Commander and routed CLI parsing, documented at `docs.openclaw.ai/cli/models`, with header capture implemented through the existing provider runtime/transport path or a narrow generic seam that preserves provider config, auth, headers, proxy/TLS/private-network policy, and plugin-owned behavior. Add focused tests for option parsing/forwarding, table and JSON output, provider config normalization, and transport failure cleanup; remove or separately justify unrelated `.gitignore` churn.

## Risks / Open Questions

- Security/supply-chain review: the PR introduces an additional direct provider HTTP path that sends live provider credentials outside the existing runtime transport stack, so proxy, TLS, private-network, attribution/header, and plugin-owned request policies may be skipped unless a shared seam is used.
- Functional risk: adding only the Commander flag would miss the lean routed CLI path on current main, leaving the new feature unreachable or inconsistent on one CLI surface.
- Product/docs risk: the PR exposes a new scriptable JSON/table contract and makes extra billable provider calls, but the public `models` docs are not updated.
- Scope hygiene risk: the `.gitignore` addition for `openclaw-*.log` is unrelated to the stated feature and should be removed or justified separately.
