## Summary

Keep this PR open. Current main still does not expose provider rate-limit headroom in `openclaw models status --probe`, and the PR remains a relevant implementation candidate, but it is not merge-ready because the current diff adds a separate direct provider fetch path with unresolved transport/config and edge-case risks.

## Best Possible Solution

Keep this PR open as a candidate, but require a rebase onto current main and a safer design. The best path is to expose a stable opt-in `--rate-limits` JSON/table contract for `models status --probe`, add both Commander and routed CLI parsing, document the extra provider request at `docs.openclaw.ai/cli/models`, and capture rate-limit headers through the existing provider runtime/transport path or a narrow generic header-capture seam so provider config, auth, headers, proxy/TLS policy, and plugin-owned behavior are honored. Add focused tests for route parsing, option forwarding, table/JSON output, provider config normalization, and transport failure cleanup.

## Risks / Open Questions

- The PR sends real provider credentials through an additional direct HTTP request path; without a generic header-capture seam, it can miss provider-owned auth, headers, base URL normalization, proxy/TLS policy, and custom transport behavior.
- The provided review comments describe concrete edge bugs in the PR branch: timer cleanup on failed fetches and custom base URL path doubling for Anthropic/Google-style endpoints.
- The branch is stale against current main’s routed CLI parser and lazy probe import behavior; a straight merge could leave the new flag unreachable on one path or regress startup/import behavior.
- The `.gitignore` `openclaw-*.log` change is unrelated to the stated feature and should be removed or justified separately.
