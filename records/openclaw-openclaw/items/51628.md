## Summary

Keep this issue open. Current main still persists failed outbound Telegram deliveries in the write-ahead queue, then later recovery replays the saved payload through the normal delivery path with `skipQueue: true`; routed reply mirrors still do not provide a stable `idempotencyKey`, so recovery can append another `delivery-mirror` transcript entry if the content is no longer the latest equivalent assistant message. The KeepMyClaw comments are useful validation context, and #44666 fixed adjacent transcript dedupe behavior, but the recovery-policy gap remains.

## Best Possible Solution

Keep this issue open and fix the outbound recovery contract: persist or derive uncertain-delivery state for non-idempotent Telegram sends, quarantine or move uncertain `sendMessage` network failures instead of blindly replaying them, derive/preserve a stable delivery-mirror idempotency key for routed replies and recovery, and add regression coverage proving both no unsafe replay and no duplicate mirror append.

## Clownfish Work Prompt

Fix the Telegram delivery queue recovery duplicate-delivery bug described in this issue. Observable problem: a Telegram `sendMessage` network failure can leave a queued outbound entry pending; later recovery replays the non-idempotent send and can also append a duplicate `provider=openclaw, model=delivery-mirror` transcript entry because routed reply mirrors lack a stable idempotency key.

Expected fix boundary:
- In `src/infra/outbound/delivery-queue-storage.ts` / `src/infra/outbound/delivery-queue-recovery.ts`, persist or derive an uncertainty state for non-idempotent Telegram sends whose prior server-side delivery cannot be ruled out, and quarantine/move to failed/otherwise skip recovery replay for those entries instead of blindly calling `deliver`.
- Preserve safe recovery for channels and failure modes where replay is appropriate; do not broadly disable the delivery queue.
- In the routed reply/delivery mirror path, derive and preserve a stable delivery-mirror idempotency key that survives recovery replay without using private payload text as the only identity.
- Keep Telegram plugin ownership boundaries intact. Use `extensions/telegram` helpers only through existing public/local seams where appropriate; do not add core hardcoded bundled plugin policy beyond a generic seam if avoidable.
- Add focused regression tests proving: a Telegram `sendMessage` network failure persisted to the queue is not replayed on recovery; a recovery path with mirror metadata does not append a duplicate delivery-mirror transcript entry; existing safe recovery for non-Telegram or definitely-undelivered cases still works.
- Add a user-facing changelog fix entry with `Thanks @Pinghuachiu` and, if appropriate for the discussion context, `@Ryce` / `@Hollychou924`. Do not credit Codex/OpenClaw bot accounts.

Do not change unrelated channel behavior, package metadata, release files, or broad plugin APIs unless the tests show a required generic seam.
