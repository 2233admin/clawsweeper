## Summary

Keep this issue open. Current main still replays eligible queued outbound deliveries by passing the saved Telegram payload back through delivery, while the queue state only records retry/backoff error text and routed reply mirrors still lack a stable idempotency key. The related KeepMyClaw comments are useful validation context, but they do not replace the root-cause delivery recovery fix.

## Best Possible Solution

Keep this issue open and fix the outbound recovery contract: persist or derive uncertain-delivery state for non-idempotent Telegram sends, quarantine or require confirmation instead of replaying uncertain sendMessage failures, preserve a stable delivery-mirror idempotency key for routed replies, and add regression coverage for both no-replay and no-duplicate-mirror behavior.

## Risks / Open Questions

- User-visible duplicate Telegram replies can still happen when a sendMessage network failure leaves server-side delivery uncertain and recovery later replays the queued payload.
- The source session transcript can still diverge when a replayed delivery mirror lacks a stable idempotency key and an intervening assistant message defeats latest-equivalent fallback dedupe.
- A fix needs to preserve safe recovery for channels and failure modes where replay is appropriate, while treating non-idempotent Telegram sendMessage uncertainty conservatively.
