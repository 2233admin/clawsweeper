## Summary

Keep this draft PR open: current main does not implement the proposed chief orchestration stack, and the patch still has blocking API-boundary, packaging, and script correctness issues that need maintainer-led decomposition before merge.

## What This Changes

The PR adds autonomous chief/task orchestration, inbound receipt ledgers, Paperclip issue sync, Telegram task-continuity delivery, quality-guard/status-tag handling, runtime hardening, Windows-oriented scripts, package scripts, and tests.

## Best Possible Solution

Split the work into narrow owner-reviewed PRs: keep generic ledgers and continuation runners behind private or versioned runtime seams, keep Telegram-owned behavior inside the Telegram plugin, expose Paperclip only through an explicit reviewed boundary, and add docs/API-baseline coverage for any public contract that remains.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep Paperclip helpers out of the public SDK:** `src/plugin-sdk/channel-runtime.ts:29`
  - body: The new Paperclip helper reads the local claimed API key and sends it as a Bearer token to `params.apiUrl ?? DEFAULT_PAPERCLIP_API_URL`, but this line re-exports it through the plugin-facing `channel-runtime` compatibility shim. That turns an internal credentialed client into public plugin API; keep it behind a narrow core-owned or versioned runtime seam instead.
  - confidence: 0.86
- **[P2] Package the added receipt reconciliation script:** `package.json:1115`
  - body: `package.json` adds `reconcile:chief-receipt-state`, but the `files` whitelist only adds the archive and task-authority scripts. Published installs will not contain `scripts/reconcile-chief-receipt-state.ts`, so the new package script will fail outside a repo checkout.
  - confidence: 0.9
- **[P2] Load the requested config through supported APIs:** `scripts/archive-chief-task-ledger.ts:46-47`
  - body: The added scripts pass `configPath` into `loadConfig`, but the exported `loadConfig()` on current main takes no arguments. This should fail type checking and would not honor `--config`; set `OPENCLAW_CONFIG_PATH` before loading or use config IO with an explicit path.
  - confidence: 0.88
