## Summary

Keep open. Current main does not implement Feishu `ackReaction`, so this PR is not obsolete, but the patch is not mergeable as written because it regresses shared ack-reaction disable semantics and still lets unsupported cross-channel word aliases reach the Feishu reaction API.

## What This Changes

This PR adds Feishu account/channel `ackReaction` config, resolves it in the Feishu reply dispatcher, passes it into the typing reaction call, adds resolver tests, and documents the Feishu typing emoji setting.

## Best Possible Solution

Revise the Feishu-owned typing path so `ackReaction: ""` skips the typing reaction, unsupported global or identity values fall back to `Typing` or are explicitly skipped before calling Feishu, and schema/docs/metadata stay aligned. Maintainers should also decide how this fits with #69572 before merging reaction-specific customization.

## Reproduction Assessment

Yes. A focused source-level reproduction is clear: with the PR, `channels.feishu.accounts.main.ackReaction: ""` resolves to an empty string and then `addTypingIndicator()` converts it back to `Typing`; `messages.ackReaction: "eyes"` passes the `^\w+$` check and is sent to Feishu instead of preserving the previous `Typing` behavior.

## Solution Assessment

No. The requested Feishu parity is reasonable, but this patch is not the best implementation until it preserves the shared disable contract and validates or normalizes Feishu reaction names before the API call.

## Review Findings

  - confidence: 0.93

  - confidence: 0.86
