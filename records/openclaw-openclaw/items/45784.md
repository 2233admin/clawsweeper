## Summary

Keep this PR open. Current main still resolves default state/config directories as `$OPENCLAW_HOME/.openclaw`, so the reported nesting bug is not implemented on main. The PR is also not safe to merge as-is because it changes only `resolveConfigDir` and returns `OPENCLAW_HOME` directly for every explicit value, which would break the documented home-root contract and leave primary config I/O on the old `resolveStateDir` path. Related #45765 remains open, while #45793 and #46016 were closed unmerged.

## Best Possible Solution

Keep this PR open for maintainer judgment or author revision. The best fix is a narrow compatibility change that updates `resolveStateDir` and `resolveConfigDir` together, preserves `OPENCLAW_HOME` as a replacement home for normal service directories, avoids double nesting only when the explicit value already names a known state directory, and adds regression tests for both the fixed and preserved behaviors.

## Risks / Open Questions

- Merging this PR as-is would silently move `resolveConfigDir` consumers for `OPENCLAW_HOME=/srv/openclaw-home` from `/srv/openclaw-home/.openclaw` to `/srv/openclaw-home`.
- Because primary config I/O still goes through `resolveStateDir`, the PR can split `openclaw.json` from global `.env` and plugin root resolution.
- The diff has no regression tests for the intended `OPENCLAW_HOME=~/.openclaw` case or the existing `/srv/openclaw-home/.openclaw` compatibility contract.
