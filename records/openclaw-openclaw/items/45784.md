## Summary

Keep this PR open. Current main still treats OPENCLAW_HOME as a replacement home and resolves default config/state below $OPENCLAW_HOME/.openclaw, so the reported nesting problem is not implemented on main. The PR is also not safe to close as implemented or merge as-is: it changes only resolveConfigDir to return OPENCLAW_HOME directly for every explicit value, while the config I/O path still uses resolveStateDir/resolveConfigPath and existing tests/docs preserve the home-root contract. Related #45765 remains open; the narrower #45793 and #46016 attempts were closed unmerged, so there is no shipped or active canonical fix to make this PR obsolete.

## Best Possible Solution

Keep this PR open for maintainer judgment or author revision. The best code path is a narrow, tested compatibility fix that keeps OPENCLAW_HOME as a replacement home for normal service directories, but avoids double nesting when the explicit value already names a known state directory; it must update resolveStateDir and resolveConfigDir together and cover config I/O, dotenv, plugin roots, and docs/tests consistently.

## Risks / Open Questions

- Merging this PR as-is would silently move resolveConfigDir consumers for general OPENCLAW_HOME values such as /srv/openclaw-home from /srv/openclaw-home/.openclaw to /srv/openclaw-home.
- Because the PR changes src/utils.ts only, primary config path resolution through src/config/paths.ts would remain on the old state-dir behavior, creating potential split paths between openclaw.json, global .env, plugins, secrets, media, cron, and other resolveConfigDir consumers.
- The diff has no regression tests for either the reported OPENCLAW_HOME=~/.openclaw case or the existing OPENCLAW_HOME=/srv/openclaw-home -> /srv/openclaw-home/.openclaw contract.
