## Summary

Keep open. Current main still has the Signal audio-preflight gap: Signal now preserves audio attachment metadata, but it does not call the shared audio transcription preflight before text-dependent mention/command handling, and it does not mark a preflight transcript on the inbound context. The existing maintainer comment also explicitly kept this issue open after the MediaPath work as only a partial fix.

## What This Changes

The issue requests a Signal audio preflight hook so inbound Signal voice notes can be transcribed before agent dispatch, ideally through the existing provider-backed transcription path or a documented configurable hook.

## Best Possible Solution

Implement a Signal-owned provider-backed voice-note preflight after Signal access checks and attachment limit enforcement, but before mention/command decisions that depend on message text. Reuse the existing transcribeFirstAudio helper, feed the transcript into body/mention/command handling, preserve media metadata, and set MediaTranscribedIndexes so downstream media understanding does not transcribe the same attachment again.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with a Signal group requiring mentions and an audio-only attachment, current main reaches the no-mention return before fetching attachments or producing a transcript, so the agent only sees media metadata or a placeholder rather than speech text.

## Solution Assessment

Yes. The best current fix is a narrow Signal provider-backed preflight using the existing plugin SDK media-runtime helper; adding a new shell-command hook should be a separate product/security design if maintainers still want that flavor later.
