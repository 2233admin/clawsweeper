## Summary

Keep this PR open. Current main still lacks the intended WhatsApp credential symlink rejection and auth-directory mode hardening, but the PR targets stale `src/web/*` paths after WhatsApp moved into `extensions/whatsapp`. Security review found no unexplained workflow, dependency, lockfile, package-script, or release-surface changes in the provided PR diff; the remaining issue is that the useful hardening needs a current-main port before it can land.

## Best Possible Solution

Keep this PR open and ask for a current-main adaptation. The best implementation belongs in `extensions/whatsapp/src/{creds-files.ts,auth-store.ts,session.ts}`: add a shared credential-file reader/check that rejects symlinked `creds.json` and `creds.json.bak` before all presence, status, identity, age, and backup reads; create the WhatsApp auth directory with `0o700` and best-effort chmod existing directories; preserve the current atomic save flow; and add focused regression tests for symlinked primary creds, symlinked backups, normal files, and directory mode behavior.
