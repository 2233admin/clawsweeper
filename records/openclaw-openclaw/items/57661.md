## Summary

Keep open. Current main has substantial sessions_send/A2A delivery work, but this issue reports frequent timeouts on v2026.3.28 and a later commenter narrowed one reproducible case to sending to a non-existent session. The current code still uses the caller-provided wait timeout directly, returns timeout from agent.wait, and for key-shaped unresolved session inputs can fall through to starting a new agent run instead of producing an immediate target-not-found error. The related #64622 timeout-hardening PR remains open rather than shipped.

## Best Possible Solution

Keep the issue open and narrow it into a focused sessions_send timeout bug. The best fix path is to add regression coverage for non-existent key-shaped targets and multi-target/A2A sends, then make sessions_send either resolve targets strictly and return an immediate not-found error when appropriate, or document/implement intentional new-session creation separately from sends to existing sessions. If timeout tuning is needed, it should be scoped to true cross-agent A2A waits and coordinated with the still-open #64622 work rather than closing this report as already fixed.
