## Summary

Keep open. Current main still has the risky source path: doctor’s generic legacy config normalizer will move reported Signal top-level values into `channels.signal.accounts.default` when named accounts exist, and Signal does not currently provide a named-account promotion filter to keep shared defaults at the channel level.

## What This Changes

The issue asks doctor `--fix` to stop corrupting Signal multi-account configs by inventing a settings-only `accounts.default` entry from shared top-level Signal defaults.

## Best Possible Solution

Keep the issue open and make Signal’s doctor promotion distinguish true legacy account identity fields from channel-level shared defaults, with a regression test using the reported two-account shape so doctor leaves `cliPath`, `dmPolicy`, and `groupPolicy` inherited at `channels.signal` rather than creating a phantom account.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path on current main: feed the reported config shape through doctor’s legacy normalization and the helper still selects the shared Signal keys and writes `accounts.default`.

## Solution Assessment

No. The current generic promotion behavior is not the best way to solve Signal multi-account migration because it converts inherited channel defaults into a runnable account; a Signal-specific promotion filter or equivalent core contract is the safer narrow fix.
