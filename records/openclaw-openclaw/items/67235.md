## Summary

Current main now implements the requested per-agent auth-store behavior for `models status --probe --agent <id>`. The status command forwards the resolved agent context into the probe pipeline, the probe target builder opens `ensureAuthProfileStore(agentDir)`, probe execution uses the same agent context, and a regression test covers requested-agent versus default-store planning.

## What This Changes

The issue reports that `models status --probe --agent <id>` should use the target agent's auth profile store and auth order instead of the default agent store.

## Best Possible Solution

Keep the current threaded agent-context implementation and let users pick up `v2026.4.29-beta.1` or a later release; no additional source change is needed for this reported mismatch.

## Reproduction Assessment

Yes. The report has a high-confidence code-level reproduction path: before the fix, `--agent` resolution stopped before probe planning; current main can be verified by tracing `agentDir` through `modelsStatusCommand()` to `runAuthProbes()` and `buildProbeTargets()`, with regression coverage for the requested-agent store.

## Solution Assessment

Yes. Threading the already-resolved `agentId`, `agentDir`, and `workspaceDir` through the probe pipeline is the narrow maintainable fix and preserves default-store behavior when `--agent` is omitted.
