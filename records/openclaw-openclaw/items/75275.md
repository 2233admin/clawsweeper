## Summary

Keep open. This is a fresh stable-release Gateway availability regression with detailed live evidence, and current main does not contain a post-v2026.4.29 source change in the implicated bundled runtime-deps, loader, or Telegram startup path. The current code still has a fallback path where Gateway startup continues after prestage failure and per-plugin runtime-deps staging can invoke a package-manager install during plugin loading.

## What This Changes

The issue reports that OpenClaw 2026.4.29 reaches Gateway ready with Telegram enabled, then runtime-deps package-manager activity and sustained Gateway CPU starve the event loop and make loopback RPC/WebSocket probes time out.

## Best Possible Solution

Make bundled runtime-deps staging idempotent and convergent for packaged Gateway startup: if the package-level runtime root already has the generated manifest and satisfied dependency sentinels, no late package-manager install should run after readiness. If staging is required, it should complete or fail before the Gateway is advertised ready, with regression coverage for the Telegram-enabled packaged path and without changing dependency versions or plugin boundary contracts.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the live failure: the report gives exact upgrade/start commands, version, OS, logs, process tree, runtime-deps lock details, and repeated sampling. A deterministic local test still needs to be added around the post-ready runtime-deps/materialization path.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no proposed patch yet. The likely best direction is to fix runtime-deps staging/materialization or Gateway readiness ordering, not to mask the symptom by increasing RPC or WebSocket handshake timeouts.
