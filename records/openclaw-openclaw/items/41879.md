## Summary

Current main still matches the reported defaults-only heartbeat scope bug. `agents.defaults.heartbeat` is documented as global behavior, but enablement and scheduling still treat only per-agent heartbeat blocks as explicit scope and otherwise fall back to the default agent. The zero-config disable expectation conflicts with current docs, so that part needs maintainer product/docs judgment rather than cleanup closure.

## Best Possible Solution

Keep this issue open as the canonical tracker. Align heartbeat scope policy across `src/infra/heartbeat-summary.ts`, `src/infra/heartbeat-runner.ts`, scheduling, runtime skip checks, and regression tests: per-agent heartbeat blocks should keep explicit opt-in scope; defaults-only heartbeat should apply to the configured agent set or the docs should be changed to document the default-agent fallback. Separately decide whether zero-config heartbeat remains documented default behavior or changes to disabled-by-default, then update docs/tests accordingly.

## Risks / Open Questions

- Fixing defaults-only scope will increase heartbeat activity for users who configured `agents.defaults.heartbeat` with multiple listed agents but were unknowingly getting only the default agent scheduled.
- The issue's zero-configuration disable expectation conflicts with current docs describing a default heartbeat interval, so maintainers should decide that policy explicitly.
- The enablement helper, scheduler, summary/status, and runtime skip paths share this policy and should be updated together to avoid a partial fix.
