## Summary

Current `main` still matches the reported defaults-only multi-agent bug. `isHeartbeatEnabledForAgent()` only treats per-agent `agents.list[].heartbeat` blocks as explicit heartbeat configuration, then falls back to enabling only the default agent. The scheduler uses the same default-agent-only fallback, and existing tests still assert that a non-default agent is disabled even when `agents.defaults.heartbeat` is configured. The zero-config expectation needs product clarification because docs currently describe heartbeats as enabled by default, but the defaults-only multi-agent issue is still real.

## Best Possible Solution

Keep this issue open as the canonical tracker. The likely fix belongs in the heartbeat owner path: align `isHeartbeatEnabledForAgent()`, `resolveHeartbeatAgents()`, and scheduler/summary coverage so `agents.defaults.heartbeat` applies to the configured agent set when no per-agent heartbeat blocks exist, while any `agents.list[].heartbeat` block keeps the current explicit-agent scoping. Add regression coverage for defaults-only multi-agent enablement and scheduling, then make a separate product/docs decision on zero-config default heartbeat behavior.

## Risks / Open Questions
