## Summary

Keep open. Current main still ignores `agents.defaults.heartbeat` when deciding heartbeat scope for a multi-agent defaults-only config: both the summary helper and scheduler fall back to the default agent only, and the regression test still asserts that behavior. The zero-config part needs product/docs clarification because current docs describe heartbeats as enabled by default, but the defaults-only multi-agent bug remains real.

## Best Possible Solution

Keep this issue open as the canonical tracker. The best fix is to align `isHeartbeatEnabledForAgent()`, `resolveHeartbeatAgents()`, and heartbeat summary/scheduler coverage with the documented contract: `agents.defaults.heartbeat` applies to the configured agent set when no per-agent heartbeat blocks exist, while the presence of any `agents.list[].heartbeat` block scopes heartbeats to only those agents. Add regression coverage for defaults-only multi-agent enablement and scheduling, then separately decide whether zero-config default heartbeat should remain documented/default-on or become opt-in.

## Risks / Open Questions

- Changing defaults-only scope would increase heartbeat activity for users who configured `agents.defaults.heartbeat` and have multiple agents; that appears to match docs, but should be called out in release notes.
- The issue's zero-configuration expectation conflicts with current docs that say heartbeats are enabled by default. Maintainers should decide that behavior separately from the defaults-only multi-agent bug.
