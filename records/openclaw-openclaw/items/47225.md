## Summary

Keep this PR open. Current main still does not implement a context-engine return contract or stream boundary that lets a context engine select the model-visible tool schema subset for a turn. Main now forwards `availableTools: Set<string>` into `assemble()`, which partially addresses tool awareness, but it is prompt-guidance metadata rather than the `toolsOverride` behavior this PR proposes.

## Best Possible Solution

Keep this PR open for maintainer review or a revised patch. The best implementation would add a backwards-compatible context-engine contract for per-turn schema selection, pass enough live tool information for engines to choose safely, deliberately define empty-subset behavior, filter at the model-request boundary without broadening executor policy, and add focused runner/helper tests plus docs for plugin authors.

## Risks / Open Questions

- Closing this PR now would discard a still-missing plugin/context-engine API for per-turn model-visible tool schema selection.
- The proposed implementation still needs maintainer design review for empty override semantics, name-based versus object-based filtering, and how schema visibility relates to executor availability.
- Current main's static `tools.allow` / `tools.byProvider` policy is not a full substitute because it changes the available runtime tool set rather than letting a context engine choose schemas per turn.
