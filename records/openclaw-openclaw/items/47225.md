## Summary

Keep open. Current main still has no result-side tool visibility override, and this PR is useful but incomplete: it adds a shared context-engine result field while only PI consumes it, treats an explicit empty override as absent, and does not give engines a usable live-schema selection contract.

## What This Changes

The PR adds optional `AssembleResult.toolsOverride`, a PI stream wrapper that filters model-visible tools by name, and PI runner wiring that applies the wrapper when the override is non-empty.

## Best Possible Solution

Define an additive per-turn tool visibility API for context engines with explicit empty-subset semantics, a practical live tool-name or schema selection input, enforcement in both PI and Codex harnesses, docs, and focused tests, while keeping executor authorization unchanged.

## Reproduction Assessment

Yes. Source review is enough: current main lacks `toolsOverride`, the PR diff ignores empty arrays, and the Codex harness still passes full `toolBridge.specs`.

## Solution Assessment

No. The direction is reasonable, but this patch is not the best complete solution because it changes a shared result type without complete selection inputs or enforcement across active harnesses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor explicit empty toolsOverride arrays:** `src/agents/pi-embedded-runner/run/attempt.ts:2128`
  - body: The guard only wraps the stream when `toolsOverride.length > 0`, so a context engine cannot intentionally hide all tool schemas for a pure-chat turn. Check field presence instead, or document and warn that an empty array is a no-op.
  - confidence: 0.96
- **[P2] Apply tool overrides across active harnesses:** `src/context-engine/types.ts:10-16`
  - body: `toolsOverride` is added to the shared context-engine result type, but only the PI runner consumes it. The Codex app-server harness also calls `assemble()` and still starts threads with full `toolBridge.specs`, so the same engine would behave differently there.
  - confidence: 0.88
- **[P2] Expose a usable live-tool selection contract:** `src/context-engine/types.ts:16`
  - body: The result type asks plugins to return `AnyAgentTool[]`, but `assemble()` receives only `availableTools?: Set<string>`. Engines cannot build the advertised filtered schema list from the live registry without a separate private tool source.
  - confidence: 0.9
