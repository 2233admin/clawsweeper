## Summary

Keep open. Current main still honors `[[reply_to_current]]`, still recommends it in the agent prompt, and only provides global/group visible-reply controls rather than the requested per-agent directive guard and per-agent auto-send suppression. The same-author split issues for the narrower directive and auto-send pieces remain open, so this broader report should stay available for maintainer coordination.

## What This Changes

The issue requests optional per-agent runtime config to disable `[[reply_to_current]]` and suppress automatic assistant-text delivery so visible output must go through explicit message-tool calls.

## Best Possible Solution

Define one coherent contract: add a per-agent directive policy that can reject or rewrite `reply_to_current` while preserving explicit `reply_to:<id>` and message-tool threading, and decide how per-agent visible-delivery policy should layer on the existing `messages.visibleReplies` model. Update config types/schema/help/docs and cover directive parsing, reply plumbing, source delivery, streaming/block replies, and Feishu-style explicit message-tool threading.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: `[[reply_to_current]]` is parsed into `currentMessageId`, tests expect explicit tags to work even when implicit threading is denied, and direct chats remain automatic unless the global `messages.visibleReplies` workaround is enabled.

## Solution Assessment

Unclear. The requested runtime guard is valid, but the best implementation needs maintainer design because adding a per-agent `autoReply` beside `messages.visibleReplies` could create overlapping public config semantics.
