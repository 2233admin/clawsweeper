## Summary

Keep open. Current main still honors `[[reply_to_current]]`, still prompts agents to prefer it, and only provides global/group visible-reply controls rather than the requested agent-scoped directive guard plus per-agent automatic-send suppression.

## What This Changes

Request per-agent outbound controls to disable `[[reply_to_current]]` processing and suppress automatic assistant-text delivery so only explicit message-tool sends reach channels.

## Best Possible Solution

Define one coherent agent-scoped outbound policy contract: a directive policy that can reject or rewrite `reply_to_current` while preserving explicit `reply_to:<id>` and message-tool threading, plus clear per-agent visible-delivery layering over the existing global/group visible-reply settings. Then update config types, schema/help/docs, parser/reply plumbing, delivery tests, and prompt guidance together.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by running reply threading on assistant text containing `[[reply_to_current]]` with a current message id; current main maps it to that message id, and direct chats still default to automatic delivery unless the global visible-reply setting is changed.

## Solution Assessment

Unclear. The requested runtime guard is a plausible direction, but the exact public config shape needs maintainer design because per-agent `autoReply` or directive policy semantics overlap existing global/group visible-reply controls.
