## Summary

Keep open. Current main has a useful global `messages.visibleReplies` path for suppressing automatic source replies, but it does not provide the requested per-agent control, and `[[reply_to_current]]` remains accepted, resolved to the current inbound message, tested as honored, and recommended in the agent prompt.

## What This Changes

The issue requests per-agent runtime configuration to disable `[[reply_to_current]]` and to disable automatic channel sending of assistant text so selected agents must use explicit message tool sends.

## Best Possible Solution

Add a typed, documented per-agent delivery policy that composes with existing `messages.visibleReplies`: selected agents can require explicit message tool sends, and selected agents can block `[[reply_to_current]]` while preserving `[[reply_to:<id>]]` and explicit `message` tool threading. The same policy should adjust prompt guidance and surface a concise next-turn warning when the disabled directive is attempted.

## Reproduction Assessment

Yes. Source and existing tests provide a high-confidence reproduction path: parse or deliver assistant text containing `[[reply_to_current]]` and current main resolves it to the current message id; config schema inspection shows the proposed per-agent keys are not accepted.

## Solution Assessment

No, current main is not the full solution. `messages.visibleReplies: "message_tool"` is a useful global workaround for auto-send, but it is not per-agent and does not disable or warn on `[[reply_to_current]]`; the durable fix belongs in core delivery/directive config rather than SOP-only guidance.
