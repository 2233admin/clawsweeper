## Summary

Keep open. Current main still honors `[[reply_to_current]]` and only has global/group visible-reply controls, so the requested per-agent directive guard and per-agent auto-send suppression are not implemented.

## What This Changes

Request optional per-agent runtime config to disable `[[reply_to_current]]` and suppress automatic assistant-text delivery so only explicit message tool calls produce visible output.

## Best Possible Solution

Define one coherent contract: add a per-agent delivery policy that extends the existing visible-reply/source-delivery model, and add a separate per-agent directive policy that rejects or rewrites `reply_to_current` while preserving explicit `reply_to:<id>` and message-tool threading. Update config types/schema/help/docs and cover directive parsing, reply plumbing, source delivery, streaming/block replies, and Feishu-style explicit message-tool threading.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: `[[reply_to_current]]` is parsed into `currentMessageId`, tests expect explicit tags to work even when implicit threading is denied, and direct chats remain automatic unless the global `messages.visibleReplies` workaround is enabled.

## Solution Assessment

Unclear. The requested runtime guard is directionally valid, but adding `agents.list[].autoReply` beside `messages.visibleReplies` may not be the cleanest long-term contract without maintainer design input.
