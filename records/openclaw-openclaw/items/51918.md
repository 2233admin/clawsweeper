## Summary

Keep open. Current main still has adjacent collect/followup/steer/interrupt behavior, hard /stop and sessions.abort controls, and /btw side questions, but it does not implement owner-priority or priority-flagged soft in-turn delivery that preserves the active run and lets the agent absorb owner context before the final response. The supplementary-context comment is valid and is not covered by the existing queue modes.

## Best Possible Solution

Keep this as a core agent-turn/message-queue feature request. The best path is a maintainer design for an explicit owner or priority soft-steer mode: public config/API shape, safe in-turn delivery boundary, streaming and non-streaming behavior, authorization metadata preservation, Telegram group/topic routing, and focused regression tests for collect, steer, interrupt, /stop, /btw, and followup interactions.

## Risks / Open Questions

- Soft preemption needs explicit semantics for preserving pending tool calls versus canceling or superseding them, especially because existing steer and interrupt behavior are intentionally different.
- Owner-priority handling must preserve sender authorization, owner checks, Telegram group/topic routing, account/thread metadata, and should not elevate non-owner or relayed messages.
- Streaming and non-streaming runs diverge today, so a new in-turn delivery path needs tests for both to avoid dropped, duplicated, or delayed followups.
- Existing /stop, interrupt, steer, steer-backlog, collect, and /btw behavior overlap with this request, so docs and tests should define precedence before implementation.
