## Summary

Keep open. Current main still has adjacent hard-stop, interrupt, steer, collect, followup, and BTW side-question behavior, but it does not implement owner-priority or priority-flagged soft in-turn delivery that preserves the active run and lets the agent absorb new owner context before the final response.

## Best Possible Solution

Keep this as a core agent-turn/message-queue feature request. The best path is a small explicit design for owner-priority or priority-flagged followups: supported config/API surface, safe tool-boundary injection that preserves the active run, authorization/routing metadata preservation, and focused tests for default collect mode, Telegram group/topic routing, streaming and non-streaming runs, /stop, interrupt, steer, steer-backlog, and /btw interactions.

## Risks / Open Questions

- Soft preemption needs explicit semantics for preserving pending tool calls versus canceling them, because existing steer cancels pending tool calls and interrupt aborts the active run.
- Owner-priority handling must preserve sender authorization, owner checks, Telegram group/topic routing, and account/thread metadata without elevating non-owner messages.
- Streaming and non-streaming runs differ today; a new soft in-turn delivery path needs coverage for both so followups are not duplicated, dropped, or delayed until finalization.
- Existing /stop, interrupt, steer, steer-backlog, collect, and /btw behavior overlap with this request, so docs and tests should define precedence clearly.
