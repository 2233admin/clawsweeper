## Summary

Keep open. Current main still implements session persistence as a string-resolved filesystem JSON store with path-scoped locking, not as a backend-driver contract or plugin-registerable adapter. The related comments and storage PRs show real demand, but the remaining work is a maintainer API/product design decision rather than a conservative cleanup close.

## What This Changes

Request to replace the path-only JSON session store with a pluggable session-store backend contract for file, SQLite, Redis, Convex, custom adapters, and multi-instance deployments.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK contract with the existing file store as the compatibility default, or narrow the roadmap to built-in local storage such as directory-backed or SQLite sessions and document why external session backends remain out of scope.
