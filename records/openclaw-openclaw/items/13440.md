## Summary

Keep open. Current main still exposes a path-only, filesystem-backed session store and plugin helper surface; it does not implement the requested configurable session-store backend/adapter contract. The issue also has recent maintainer/bot keep-open reviews and adjacent open directory-backed work, but that work is narrower than Redis/Convex/SQLite/custom multi-instance storage.

## What This Changes

The issue requests a pluggable session-store backend contract so session state can use file, SQLite, Redis, Convex, custom, or similar stores instead of only `sessions.json`.

## Best Possible Solution

Make an explicit maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK capability with the current file store as the compatibility default, or narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external backends remain out of scope.

## Reproduction Assessment

Not applicable as a feature/API request. The current-main verification path is high confidence: inspect the config type/schema, resolver, load/save/lock implementation, plugin runtime surface, and docs to confirm there is no adapter backend seam.

## Solution Assessment

Unclear. A backend adapter contract is a plausible direction, but the best solution depends on maintainer choices about core versus plugin ownership, backend credentials, migration guarantees, recovery semantics, and overlap with the directory-backed store work.
