## Summary

Keep open. Current main still uses a path-only, file-backed `sessions.json` store with direct filesystem JSON reads/writes and a sidecar write lock; the plugin-facing surface exposes those file-store helpers rather than a backend driver or adapter registration contract. The discussion and related database-backed work show real demand, but the requested Redis/Convex/SQLite/custom multi-instance session-store seam remains a maintainer product/API decision.

## What This Changes

Request to add a pluggable session store adapter and config backend model for file, SQLite, Redis, Convex, and custom stores instead of only a `sessions.json` path.

## Best Possible Solution

Make a maintainer product/API decision: either define a supported SessionStoreAdapter-style core or SDK capability with the current file store as the compatibility default, or explicitly narrow the roadmap to a built-in local storage model such as directory-backed or SQLite-backed sessions and document why external session backends remain out of scope.
