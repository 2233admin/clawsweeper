## Summary

Keep open. Current main still treats `session.store` as a string path to a gateway-owned `sessions.json`, and the load/save/update path remains direct JSON file IO protected by the existing file-lock queue. The plugin SDK exposes session-store helper functions, not a backend registration or adapter API, so Convex/SQLite/Redis-style pluggable session stores remain a real design item rather than something implemented or superseded by the related memory-backend discussions.

## Best Possible Solution

Keep this open for maintainer design. The best path is to decide whether session persistence should gain a generic core/SDK `SessionStoreAdapter` seam with the current file store as the default, or whether narrower work such as directory-backed or SQLite-backed session storage should land first. Any implementation should update config types/schema/help, docs, migration/doctor behavior, plugin SDK boundaries, and focused session-store concurrency tests.
