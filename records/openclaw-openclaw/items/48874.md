## Summary

Current main has adjacent routing, session listing, session creation, deletion/transcript archiving, compaction restore, and memory search capabilities, but it does not implement the RFC’s central multi-session architecture: DMs still collapse to main by default, session.mainKey is treated as legacy/ignored in runtime routing, the session lifecycle/API lacks first-class active/archive/pin/tag/channel-binding fields and archive/restore/pin methods, Control UI New Session still dispatches /new, and memory remains plugin/agent-workspace scoped rather than a new public RAG knowledge-base layer. The issue should stay open for maintainer design/splitting, with PR #73725 treated as a narrow WebChat create/switch slice rather than full resolution.

## Best Possible Solution

Keep this issue open as an umbrella/design thread, or split it into smaller maintainer-owned issues: canonical session identity and dmScope/mainKey routing policy; first-class active/archive/pin/tag lifecycle plus archive/restore/list APIs; Control UI create/archive/restore/pin workflows; and a scoped shared memory/RAG contract. Review and either land or close PR #73725 as the narrow WebChat create/switch slice, while treating Artyomkun’s linked global-architecture fork as useful prior art rather than a completed OpenClaw implementation.
