## Summary

Keep this PR open. Current main still has the gateway dedupe cleanup under-eviction bug: the loop condition depends on `params.dedupe.size` while the loop deletes entries. The PR's production change targets the right code path, but its added test only overflows by one entry, which current main already handles because it already sorts by entry timestamp. The PR also overlaps with open PR #45979, so maintainers should choose one canonical patch.

## Best Possible Solution

Keep this PR open as a plausible fix candidate, but request a regression test that overflows `dedupe` by more than one entry and asserts the final size is exactly `DEDUPE_MAX`. Then review it alongside #45979 and land one canonical patch that snapshots the excess keys or precomputes `excess` before deleting while preserving timestamp-based eviction.
