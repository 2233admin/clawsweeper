## Summary

Keep this issue open. Current main still matches the reported Slack failure mode: block-action payloads carry `threadTs`, but session-key resolution drops it and enqueues a passive system event, so a click can land on the base Slack session while the pending prompt is on the thread-suffixed session. The linked open PRs are implementation candidates, not shipped fixes.

## What This Changes

Slack interactive button clicks should wake or dispatch exactly one agent turn for the originating Slack session and thread, with replies delivered back to that same thread.

## Best Possible Solution

Keep this issue as the canonical tracker until one Slack-only fix lands that forwards `threadTs` into system-event session-key resolution, reuses the existing thread-session helper, proactively wakes or dispatches the correct session exactly once, preserves Slack reply delivery context, and adds regression coverage for thread, duplicate-click, and busy-session behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Slack `block_action` with `container.thread_ts`: current main parses that value but resolves the system-event session without it, which matches the reported base-session versus thread-session mismatch.

## Solution Assessment

Yes for the requested direction, but the implementation should be one reconciled patch rather than another parallel approach. The narrow maintainable path is to keep the fix inside the Slack plugin boundary, use the existing `resolveThreadSessionKeys` seam, and choose one canonical wake/dispatch path from the open PRs.
