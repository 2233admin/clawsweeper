## Summary

Keep this PR open. Current main still lacks the transient Telegram sendChatAction cooldown/retry suppression this PR targets; the handler only implements the existing 401 backoff/suspension path, and current tests still assert repeated 500 errors throw without any cooldown. The provided PR discussion also says a maintainer pushed a narrow repair so this branch remains the canonical contributor path, while the related alternative PR is closed and unmerged.

## Best Possible Solution

Keep this PR open for maintainer review and landing, or request targeted changes if maintainers disagree with the cooldown semantics. The likely correct implementation belongs in the Telegram plugin handler: classify recoverable network, 429, and 5xx sendChatAction failures there, suppress redundant API calls during cooldown, preserve the existing 401 suspension behavior, and keep transient failures rejecting so the shared typing guard can trip during sustained outages.
