## Summary

Keep this issue open. Current main still only maps Claude CLI permission mode for YOLO exec policy; when `tools.exec.security` is `allowlist`, the Claude backend leaves `--permission-mode` unset and does not translate or enforce `exec-approvals.json`, while the native gateway exec path still evaluates that allowlist and denies misses. The referenced #67666 was a Codex CLI resume sandbox fix and does not cover Claude CLI allowlist forwarding.

## What This Changes

Request that OpenClaw forward per-agent exec allowlist policy into Claude CLI backend sessions so Claude Code-internal exec follows the same allowlist semantics as native OpenClaw exec.

## Best Possible Solution

Keep this open for maintainer/security design, then implement a Claude CLI policy bridge that preserves OpenClaw allowlist semantics or clearly documents any unsupported cases, with regression coverage for allowed and denied commands under per-agent allowlist policy.
