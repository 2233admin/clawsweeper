## Summary

Keep this PR open. Current main still has the Telegram model-picker no-override fallback bug: the callback resolves the routed agent and builds an agent-aware provider list, but the final session-state fallback still reads the global `agents.defaults.model`, so the current-model checkmark can be wrong when the routed agent has its own default. The PR is a narrow, appropriate fix with regression coverage; it needs normal maintainer review, rebase/validation, and likely no separate replacement work.

## What This Changes

This PR changes Telegram model-picker callback session reconstruction to fall back through `resolveDefaultModelForAgent({ cfg, agentId })` and adds a regression test for an agent-level default that differs from the global default.

## Best Possible Solution

Land this PR or an equivalent rebased patch that computes the routed agent default inside `resolveTelegramSessionState()`, returns that `provider/model` for the no-override fallback, keeps the agent-default-vs-global-default regression test, and validates the Telegram extension before merge.
