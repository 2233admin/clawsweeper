## Summary

Keep open. Current main still has a plausible Windows Scheduled Task restart path for ordinary config/SIGUSR1 reloads, the linked restart-drain work was explicitly called out as not covering this stale-lock/EADDRINUSE mode, and the recent supervised-lock loop guard is systemd-specific rather than Windows/schtasks coverage.

## What This Changes

Editing OpenClaw config while a Windows gateway is running should not leave a stale gateway lock or EADDRINUSE restart loop during hot reload.

## Best Possible Solution

Make ordinary config/SIGUSR1 restarts on Windows deterministic: either keep them in-process even when Scheduled Task markers are present, or add a Windows-specific scheduled-task handoff with health-checked startup, stale-lock recovery, bounded retries, and tests. Preserve fresh-process behavior for update restarts and align docs with the actual Windows service path.

## Reproduction Assessment

Yes, at a code-path level. The report gives concrete Windows 11 steps, and current main can model the risky path by setting win32 Scheduled Task markers, triggering a config/SIGUSR1 restart, and observing the supervised full-process restart path rather than the unmanaged in-process fallback.

## Solution Assessment

No current solution is complete. PR #71465 improved restart draining, but the remaining Windows Scheduled Task stale-lock/EADDRINUSE recovery needs a targeted fix in the gateway restart/lock path with Windows-specific coverage.
