## Summary

Current main still has the reported eager gateway aggregation path: `loadCombinedSessionStoreForGateway` discovers every agent session store, reads each `sessions.json`, and callers apply agent/session/label/limit filters afterward. Recent session maintenance and clone-reduction work are useful mitigations, but they do not implement the requested lazy per-agent or indexed lookup path, so this issue should stay open.

## Best Possible Solution

Keep this issue open. The best fix is to add a targeted per-agent and/or indexed session lookup path for gateway session resolution, make `sessions.list` honor `agentId` and `limit` before reading unrelated stores, use streaming or pagination for true all-agent listings, and reduce cache cloning without exposing mutable shared state. Add regression coverage with multiple large per-agent stores proving agent-scoped listing/resolution does not read unrelated `sessions.json` files.
