## Summary

Current main still has the reported eager gateway aggregation path. `loadCombinedSessionStoreForGateway` resolves every agent session-store target and loads each `sessions.json`; gateway `sessions.list`, sessionId/label resolution, transcript lookup, and usage paths still call that combined loader before filters can narrow scope. Existing cleanup and clone-reduction work mitigates growth, but it does not implement the requested lazy per-agent or indexed lookup path.

## Best Possible Solution

Keep this issue open and fix the gateway session lookup path. The best implementation is a targeted per-agent and/or indexed session-store API that lets `sessions.list`, `sessions.resolve`, transcript lookup, and usage paths avoid reading unrelated `sessions.json` files when the request has a key or agent scope, while preserving broad disk-only discovery for true all-agent listings. Add regression coverage with multiple large/instrumented stores proving agent-scoped calls do not read unrelated stores, and keep cleanup maintenance as a separate mitigation rather than the main fix.

## Clownfish Work Prompt

Fix the gateway session-store OOM caused by eager all-agent aggregation. In current main, `loadCombinedSessionStoreForGateway` reads every discovered `sessions.json`, and `sessions.list` plus sessionId/label resolution filter only after that. Add a target-scoped session-store lookup/listing path so requests with an explicit session key or `agentId` load only the relevant agent store(s), and avoid unrelated store reads in `sessions.list`, `sessions.resolve`, transcript-key lookup, and `sessions.usage` where the request is scoped. Preserve disk-only/retired agent visibility for true all-agent listings (#32804 behavior) and do not expose mutable cached store objects to callers. If all-agent pagination/indexing is too broad for one PR, keep all-agent behavior unchanged and document the remaining follow-up clearly in code/tests. Add regression tests with multiple per-agent stores and mocked/instrumented `loadSessionStore` proving agent-scoped listing/resolution does not read unrelated `sessions.json` files. Related refs: this issue, #45438 for clone pressure, and closed unmerged PR #51362 as prior context. Add a user-facing changelog fix entry with `Thanks @mband0` if behavior changes.
