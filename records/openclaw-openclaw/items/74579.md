## Summary

This should stay open: current main still has the same native Ollama and agent prompt/context plumbing that could produce the reported behavior, and the report is fresh, concrete, and release-specific. The code confirms runtime context can be appended as prompt-local system context for normal turns, while native Ollama receives the system prompt plus any resolved tools; there is not enough evidence that current main has fixed the observed local-model failure.

## What This Changes

The issue reports that native Ollama models in a clean TUI session on OpenClaw 2026.4.26 ignore the user prompt and answer injected runtime/context metadata instead.

## Best Possible Solution

Reproduce with the reporter's diagnostics or a local Ollama daemon, then patch the owner boundary that actually causes the model to treat runtime metadata as the task. The end state should keep the user's visible prompt as the clear final task, keep runtime context private and bounded for local models, ensure `compat.supportsTools:false` is resolved for the active model entry, and add focused coverage around native Ollama payload shaping plus `/context detail`/bootstrap context reporting.
