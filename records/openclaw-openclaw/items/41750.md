## Summary

Current main still does not recover from an unhealthy Chrome/Chromium process that continues to own the managed CDP port. The bundled browser plugin detects CDP readiness and cleans stale Chromium Singleton lock files, but a busy port still fails before any owner-scoped listener cleanup, and existing tests explicitly cover fail-closed behavior rather than killing the stale listener.

## Best Possible Solution

Keep this issue open. Implement the recovery in the bundled browser plugin: before launching a local managed profile, probe the loopback CDP endpoint with a short timeout; if the port is occupied but unhealthy, inspect the listener process and terminate only OpenClaw-owned Chrome/Chromium processes for that profile, wait for the port to clear, then launch fresh. Add focused tests for unhealthy owned listener cleanup, healthy pre-existing browser reuse, attachOnly/remote refusal to kill, and unrelated listener refusal.

## Risks / Open Questions

- A fix must not kill unrelated services on the same port. It should terminate only local, non-attachOnly, OpenClaw-managed Chrome/Chromium processes matching both the expected `--remote-debugging-port` and managed `--user-data-dir`.
- The report uses "zombie" colloquially; the robust trigger should be failed CDP health/readiness plus owner checks, not RSS or Unix zombie-state heuristics alone.
