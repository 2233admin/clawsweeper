## Summary

Current main still fails closed when the managed browser CDP port is occupied by an unhealthy listener. The browser plugin probes CDP reachability and handles stale Chromium Singleton lock files, but it does not inspect and terminate an OpenClaw-owned Chrome/Chromium process that still owns port 18800 after becoming unresponsive. The related attempted fix in #41758 is closed unmerged, so there is no canonical implemented fix on main.

## Best Possible Solution

Keep this issue open and implement owner-scoped recovery in the bundled browser plugin. Before launching a local managed profile, probe the loopback CDP endpoint with a short timeout; when the port is occupied but unhealthy, inspect the listener process and terminate only OpenClaw-owned Chrome/Chromium processes for that profile, wait for the port to clear, and then launch fresh. Add regression tests covering unhealthy owned listener cleanup, healthy pre-existing browser reuse, attachOnly/remote refusal, and unrelated listener refusal.

## Risks / Open Questions

- A recovery fix must not kill unrelated services on the same loopback port; it should only terminate local OpenClaw-managed Chrome/Chromium processes that match both the expected `--remote-debugging-port` and managed `--user-data-dir`.
- The report uses "zombie" colloquially, so the robust trigger should be failed CDP health/readiness plus ownership checks, not RSS size or Unix zombie-state heuristics alone.
- Cross-platform listener inspection and termination need focused tests for macOS/Linux/Windows behavior, attachOnly/remote refusal, healthy existing-browser reuse, and unrelated listener refusal.
