## Summary

Keep open. Current main still does not recover the reported stale managed Chrome/CDP-port case: the browser plugin probes CDP health, handles stale Chromium Singleton lock files, and restarts tracked child processes, but it still has no owner-scoped cleanup for an untracked OpenClaw-managed Chrome/Chromium process that remains bound to port 18800 while unresponsive. The detailed report is still actionable, and the related attempted fix in #41758 is closed unmerged.

## Best Possible Solution

Keep this issue open and implement owner-scoped recovery in the bundled browser plugin. Before launching a local managed profile, probe the loopback CDP endpoint with a short timeout; when the port is occupied but unhealthy, inspect the listener and terminate only OpenClaw-owned Chrome/Chromium processes for that profile, wait for the port to clear, and launch fresh. Add regression tests for unhealthy owned listener cleanup, healthy pre-existing browser reuse, attachOnly/remote refusal, and unrelated listener refusal.
