## Summary

Keep this PR open, but do not merge it as-is. The author addressed the earlier wrapped Slack WebClient error concern, but the rebased failed-delivery accounting now drops delivered `tool` counts before calling the shared visible-dispatch helper, so delivered tool-only Slack replies can be misclassified as undelivered.

## What This Changes

The PR adds bounded Slack `chat.postMessage` retries, unwraps Slack WebClient request errors for retry classification, adjusts Slack reply-delivery accounting using failed counts, and adds focused Slack tests.

## Best Possible Solution

Update the PR so failed-count correction is generic over `ReplyDispatchKind`: subtract failed counts for `tool`, `block`, and `final`, pass a full corrected count record into `hasVisibleInboundReplyDispatch`, and add a Slack regression test for a delivered tool-only reply path. Keep the bounded retry wrapper and wrapped-error unwrapping if maintainers accept the Slack retry tradeoff.

## Reproduction Assessment

Yes. A source-level reproduction is a Slack turn with `counts.tool > 0`, no block/final, and no queued final: current main's helper marks it visible, while the PR head passes only block/final counts and treats it as undelivered.

## Solution Assessment

No, not as-is. The retry direction and wrapped-error fix are narrow, but the accounting fix should preserve the full shared dispatch-count contract instead of adapting only block/final counts.
