## Summary

Keep this issue open. Current main still deliberately returns `INVALID_REQUEST` / `unknown agent id` when `tools.catalog` or `tools.effective` is called with an agent ID that is not configured, and the report still lacks the failing `agentId`, `sessionKey`, redacted config, tool RPC payloads, and logs needed to tell stale local state from a real tool-policy regression. The configured `tools.profile: "full"` should not by itself reduce the runtime tool list to only messaging/session tools, so the remaining bug report is plausible but not closeable yet.

## Best Possible Solution

Keep the issue open and ask for a fresh reproduction on current OpenClaw with the failing `agentId`, `tools.catalog` and `tools.effective` request params/responses, `openclaw agents list`, redacted `agents.list` and `tools` config, active session key, selected channel/plugin/provider, web-search provider config, install method/path, and gateway logs. If the data shows stale local state, improve Control UI or doctor diagnostics; if a valid full-profile session still gets only messaging tools, trace `resolveEffectiveToolPolicy` and the runtime tool filtering across agent, provider, group/channel, session, and web-search provider policy.
