## Summary

Keep #64752 open. Current main still handles Telegram `message_reaction` updates as authorized passive system-event notifications only; the reaction path enqueues context but does not call the normal Telegram dispatch path or any heartbeat/wake helper, and the issue has recent live reproduction evidence through OpenClaw 2026.4.21.

## Best Possible Solution

Keep #64752 open. Implement a narrow Telegram-owned fix in the bundled Telegram plugin: after an authorized added reaction is enqueued, honor a documented opt-in reaction wake setting and request a session-scoped agent/heartbeat turn for the resolved route. Add regression tests proving reaction-only updates wake the agent when enabled and remain passive context events when disabled; leave richer reaction semantics/custom emoji normalization to #47677 unless maintainers choose to fold both into one design.

## Risks / Open Questions

- Telegram reaction updates do not include `message_thread_id`, so a wake implementation needs a deliberate routing policy for forum topics to avoid waking the wrong topic-scoped session.
- The broader semantic reaction design, custom emoji support, and richer target-message context overlap with open #47677; the narrow wake bug should either stay scoped here or be explicitly superseded by that broader tracker.
