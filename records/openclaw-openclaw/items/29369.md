## Summary

Keep this issue open. Current main still does not expose or honor a spawn-time `tools` allowlist on `sessions_spawn`; tool policy remains config/session scoped, and adding per-call tool policy needs maintainer security semantics before an implementation PR is queued.

## Best Possible Solution

Keep this open for maintainer design. If approved, implement a narrow `sessions_spawn.tools` policy object that is validated in the tool schema, propagated through both native and ACP spawn paths, stored or passed as trusted runtime metadata, and applied as a final narrowing layer in the effective tool-policy pipeline with deny-wins behavior. Add targeted tests and docs for precedence and failure behavior.

## Risks / Open Questions

- Spawn-time tool policy is a security boundary; allowing the call to broaden capabilities would be risky. A safe design should be final narrowing-only with deny-wins semantics unless maintainers explicitly choose otherwise.
- Native subagent and ACP runtime paths have separate schemas and spawn params; implementing only one would create inconsistent `sessions_spawn` behavior.
- Docs and tests need to pin precedence against `tools.allow`, `agents.list[].tools`, provider/group policy, sandbox policy, and `tools.subagents.tools`.
