## Summary

Keep this issue open. Current main still does not expose or honor a spawn-time `tools` allowlist on `sessions_spawn`; schema, native spawn, ACP spawn, docs, and the effective tool-policy pipeline still use config/profile/sandbox/subagent policy layers rather than a per-call policy object.

## What This Changes

Add a spawn-time tools allowlist to `sessions_spawn` so sub-agent tool permissions can be configured per call instead of only through agent profiles.

## Best Possible Solution

Keep this open for maintainer design. If approved, add a narrow `sessions_spawn.tools` policy object that is schema-validated, propagated through both native and ACP spawn paths, treated only as trusted runtime metadata, and applied as a final deny-wins narrowing layer in the effective tool-policy pipeline. Update docs and focused tests for precedence, invalid entries, empty allowlists, and native/ACP parity.

## Risks / Open Questions

- A per-call tool allowlist is security-sensitive: if implemented as a grant instead of a final narrowing layer, it could widen child-agent tools beyond configured profile, provider, sandbox, group, or subagent policy.
- Native and ACP spawn paths would need parity; implementing only one path would leave inconsistent behavior for the same `sessions_spawn` surface.
- Unknown tool names and empty allowlists need fail-closed behavior consistent with the existing explicit allowlist guard.
