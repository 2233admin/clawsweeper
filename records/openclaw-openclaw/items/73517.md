## Summary

Keep open. Current main still has the repeated task-maintenance lookup shape reported in the CPU profile, and open PR #74406 is already linked as the intended fix, so this issue should remain open until that PR or an equivalent change lands.

## What This Changes

The issue reports that gateway task registry maintenance can sustain high CPU by repeatedly loading and cloning session state while reconciling stale task records from `runs.sqlite`.

## Best Possible Solution

Land #74406 or an equivalent narrow task-maintenance change that creates a per-pass backing-session lookup context, caches session-store reads and duplicate CLI chat-type derivations, and preserves the fresh recheck around recovery hooks and changed tasks.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same session store or channel session key, run maintenance, and count repeated `loadSessionStore` and `deriveSessionChatType` calls. The exact production CPU profile cannot be fully replayed without the reporter's original task registry database.

## Solution Assessment

Yes, the linked per-maintenance-pass cache direction is the narrowest maintainable fix for the reported hot path because it avoids changing task schema, retention policy, startup repair, or automatic database quarantine semantics. Broader quarantine/rebuild behavior would be follow-up product scope.
