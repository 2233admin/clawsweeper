## Summary

Keep open. Current main still has the repeated task-maintenance lookup shape from the reported CPU profile, and open PR #74406 is the active closing implementation candidate rather than a merged fix.

## What This Changes

The issue asks OpenClaw to prevent task registry maintenance from sustaining high CPU on stale or large `runs.sqlite` state by avoiding repeated session-store loads, duplicate chat-type derivation, and unsafe handling of pathological task registries.

## Best Possible Solution

Review and land PR #74406 or an equivalent narrow task-maintenance change that reuses backing-session store reads and duplicate CLI chat-type derivations within one maintenance pass while preserving the fresh recheck around recovery hooks and changed tasks. Broader malformed-database quarantine should remain in the separate task-registry corruption track such as #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same session store or channel session key, run `runTaskRegistryMaintenance`, and count repeated `loadSessionStore` and chat-type derivation calls. The exact production CPU profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes, the per-maintenance-pass cache direction in PR #74406 is the narrowest maintainable fix for the reported hot path because it avoids task schema, retention, startup repair, and automatic database quarantine changes.
