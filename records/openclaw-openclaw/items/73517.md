## Summary

Keep open. Current main still has the repeated task-maintenance lookup shape reported in the CPU profile, and open PR #74406 is already linked as the intended fix, so this issue should remain open until that PR or an equivalent change lands.

## What This Changes

The issue asks for task registry maintenance to stop burning CPU on stale `runs.sqlite` data by avoiding repeated session-store and chat-type lookups during stale task reconciliation.

## Best Possible Solution

Land #74406 or an equivalent narrow task-maintenance change that creates a per-pass backing-session lookup context, caches session-store reads and duplicate CLI chat-type derivations, and preserves the fresh recheck around recovery hooks and changed tasks.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records that share the same session store or channel session key, run maintenance, and count repeated `loadSessionStore` and `deriveSessionChatType` calls. The exact production CPU profile still depends on the reporter's private original task registry database.

## Solution Assessment

Yes, the linked per-maintenance-pass cache direction is the narrowest maintainable solution for the reported hot path because it avoids task schema, retention policy, startup repair, and automatic database quarantine changes. Broader quarantine/rebuild behavior would be separate product scope.
