## Summary

Keep open. Current main still has the repeated task-maintenance lookup shape described in the CPU profile, and open PR #74406 is still the targeted fix candidate rather than a reason to close this issue before merge.

## What This Changes

Task registry maintenance should avoid sustained gateway CPU use by reusing repeated backing-session and chat-type lookups while reconciling stale task records from runs.sqlite.

## Best Possible Solution

Review and land #74406, or an equivalent narrow task-maintenance change, so each maintenance pass reuses backing-session store reads and duplicate CLI chat-type derivations while preserving the fresh recheck around recovery hooks and changed tasks. Broader malformed-database quarantine should remain tracked separately in #71689.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records that share the same session store or channel session key, run maintenance, and count repeated `loadSessionStore` and `deriveSessionChatType` calls. The exact high-CPU production profile still depends on the reporter's private task registry database.

## Solution Assessment

Yes. The linked per-maintenance-pass cache direction is the narrowest maintainable fix for the reported hot path because it avoids task schema, retention, startup repair, and automatic database quarantine changes.
