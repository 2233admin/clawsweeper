## Summary

Keep open. Current main still has the repeated task-maintenance lookup shape reported in the CPU profile, and open PR #74406 is already linked with closing syntax as the intended fix, so this issue should remain open until that PR or an equivalent change lands.

## What This Changes

Task registry maintenance should avoid repeatedly loading or cloning session stores and deriving chat types for stale task records from a large or stale `runs.sqlite`, preventing sustained gateway CPU hot-loops.

## Best Possible Solution

Land #74406 or an equivalent narrow task-maintenance change that creates a per-pass backing-session lookup context, caches session-store reads and duplicate CLI chat-type derivations, and keeps the fresh recheck around recovery hooks or changed tasks.

## Reproduction Assessment

Yes for the code-level hot path: seed multiple stale subagent or CLI task records sharing the same session store or channel session key, run maintenance, and count repeated `loadSessionStore` and `deriveSessionChatType` calls. The exact production CPU magnitude depends on the reporter's original private `runs.sqlite` state.

## Solution Assessment

Yes for the linked PR direction: a per-maintenance-pass lookup cache is the narrowest maintainable fix for the reported hot path. Broader database quarantine or retention policy changes would be separate product scope.
