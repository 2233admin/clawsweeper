## Summary

Keep this PR open. Current main still does not implement the requested NetBird Gateway bind mode: the config type/schema reject `netbird`, bind resolution has no NetBird path, and Gateway TLS remains driven only by explicit `gateway.tls`. The PR is a relevant implementation candidate with clear user interest, but it still needs maintainer review because the patch has blocking origin/status/link-surface defects and a security-sensitive network exposure heuristic.

## What This Changes

The PR adds a `gateway.bind: "netbird"` mode across Gateway config/schema, CLI/status/onboarding surfaces, NetBird `wt*` interface detection, bind resolution, auto-generated self-signed TLS, tests, and changelog.

## Best Possible Solution

Refresh this PR against current main, implement NetBird support in the current shared Gateway bind/link/status/origin paths, seed or explicitly require the actual NetBird HTTPS origin, load the generated TLS fingerprint anywhere auto-TLS is used, and land only after maintainers accept the NetBird interface trust boundary. If core should avoid service-specific VPN bind modes, keep the user problem tracked while designing a generic VPN bind profile/plugin seam.

## Security Review

Status: needs_attention

Summary: No dependency, workflow, lockfile, or package-publishing change was found, but the patch changes Gateway network exposure and relies on a weak NetBird interface attribution heuristic.

Concerns:

- **[medium] Strengthen NetBird interface attribution:** `src/infra/netbird.ts:11`
  - body: The new detector treats any non-internal interface whose name starts with `wt` as NetBird before binding the Gateway and enabling TLS. On hosts with another `wt*` interface or non-default NetBird naming, this can expose the Gateway on an unintended private network, so the implementation should use stronger attribution or make the trust boundary explicit in config/docs.
