## Summary

Keep open. Current main still derives the packaged bundled runtime-deps stage key from `path.resolve(packageRoot)`, while the bundled plugin loader and bundled channel path can arrive at equivalent installs through different lexical roots. The linked PR #75048 remains open with closing syntax for this issue, so this should stay open until that fix or an equivalent replacement lands.

## What This Changes

The issue reports that OpenClaw 2026.4.27 can derive multiple `plugin-runtime-deps` stage hashes for the same bundled install, causing bundled channel/plugin `dist` imports to fail with `ENOENT` under multi-instance or equivalent-root deployments.

## Best Possible Solution

Review and land #75048, or an equivalent narrow replacement, so the packaged runtime-deps stage key hashes the canonical filesystem package root and has regression coverage for equivalent package-root views. After that, keep the disabled/unconfigured bundled channel discovery noise as a separate follow-up only if it still reproduces.

## Reproduction Assessment

Yes. A focused reproduction is to expose the same packaged OpenClaw root through two equivalent paths, such as a real path and symlink/junction path, then compare `resolveBundledRuntimeDependencyInstallRoot`; current main still hashes the lexical `path.resolve` form, so the stage keys can diverge.

## Solution Assessment

Yes. Canonicalizing the package root at the hash seam is the narrowest maintainable fix because it converges the shared stage directory without broad channel-loader rewrites or new public API.
