## Summary

Keep open. Current main still derives the packaged bundled runtime-deps stage key from the lexical `path.resolve(packageRoot)`, and the open linked PR that targets this exact hash divergence remains unmerged, so this issue should wait for that fix or an equivalent replacement to land.

## What This Changes

Make packaged bundled runtime-deps staging use one stable per-version install root across equivalent package-root paths so multi-instance gateways can load bundled channel dist chunks reliably.

## Best Possible Solution

Review and land #75048, or an equivalent narrow replacement, so the stage-key hash uses the canonical filesystem path for the package root and includes focused regression coverage for equivalent package-root views. Keep adjacent stale-runtime-path reports such as #75045 tracked separately if they do not collapse to the same hash fix.

## Reproduction Assessment

Yes. A focused reproduction is to expose the same packaged OpenClaw root through two lexical paths, such as a real path and symlink or junction path, then compare `resolveBundledRuntimeDependencyInstallRoot` outputs; current main still hashes the lexical package-root string, so the mechanism remains reproducible without needing the full Windows PM2 setup.

## Solution Assessment

Yes for the linked PR direction, but current main has not applied it. Canonicalizing at the runtime-deps hash seam is narrower and more maintainable than changing bundled channel loading or adding an operator workaround.
