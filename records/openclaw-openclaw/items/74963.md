## Summary

Current main still has the lexical runtime-deps stage-key hash, and the open closing PR for this exact report has not merged in the provided GitHub context. The issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that OpenClaw 2026.4.27 can derive different plugin-runtime-deps stage hashes for the same physical install across multi-instance or equivalent-path setups, causing bundled channel and plugin ENOENT failures under mismatched dist roots.

## Best Possible Solution

Land #75048 or an equivalent narrow fix that canonicalizes the packaged package root before computing the runtime-deps stage key, then keep focused coverage for equivalent physical package roots and disabled/unconfigured bundled-channel discovery noise.

## Reproduction Assessment

Yes. A focused reproduction is to expose the same packaged OpenClaw root through two equivalent filesystem paths, such as real and symlink/junction views, then compare the external bundled runtime-deps install roots; current main still hashes the lexical path so the divergence remains explainable without a live Windows PM2 run.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable fix because it corrects the hash seam used to name the external stage root instead of broadening bundled channel loading, rewriting discovery, or adding an operator workaround.
