## Summary

Keep this PR open. Current main still does not clearly implement the PR's Slack-specific behavior: Slack builds its own reply dispatcher and calls `deliverReplies()`, whose current `sendMessageSlack()` call sites do not run `message_sending`. The earlier shared fix in #70118 covers the `dispatchInboundMessageWith*` wrapper paths, but Slack uses a prebuilt dispatcher with plain `dispatchInboundMessage()`, so the claimed coverage gap remains plausible. The PR is also a draft and mixes a real hook-coverage fix with a new bundled Slack addressee guard that needs maintainer product/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The strongest path is to land or adapt the Slack `message_sending` coverage fix with focused tests for text, blocks, media, and relevant streaming/fallback paths. Treat the addressee guard as a separate product decision: either move it to a ClawHub/plugin follow-up, or make it a generic official Slack safety plugin with account-scoped auth, docs, schema parity, and clear bundled-plugin ownership.

## Risks / Open Questions

- The PR mixes a broadly useful Slack hook-coverage fix with a new bundled addressee-guard plugin that looks deployment-specific; maintainers may want these split before merge.
- The new plugin declares `logPath` in its schema but the diff does not implement file logging, which would create config/UI drift if shipped as-is.
- The proposed hook placement covers `deliverReplies()` send paths, but Slack native streaming/preview edit paths should be checked so `message_sending` coverage is not still partial or double-fired.
- The addressee guard reads a global `SLACK_BOT_TOKEN`; for multi-account Slack installs it may need an account-scoped credential path or SDK helper before becoming a bundled plugin.
