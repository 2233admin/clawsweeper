## Summary

Keep open. Current main has reporting-oriented provider usage snapshots and `/status`/`session_status` visibility for Anthropic 5h/week windows, but it does not implement the requested pace-aware runtime policy: no Anthropic rate-limit header capture into persistent runtime state, no projected pace tiers, no `agents.defaults.pace` config, no prompt injection or tier notifications, and no proactive subagent/non-interactive throttling before model calls. Related open trackers such as #13615, #38775, and #71770 are useful design context, but they do not make this specific Anthropic pace-aware autonomous-agent request obsolete.

## Best Possible Solution

Keep this open as a pace-aware autonomous-agent design item. The best path is to consolidate it with #13615, #38775, and #71770, then decide whether pace state is core-owned or exposed through a new plugin/provider policy API. A complete design should cover Anthropic headers versus usage polling, per-profile persistence, provider fallback behavior, status/session context, and clear subagent/non-interactive throttling rules.
