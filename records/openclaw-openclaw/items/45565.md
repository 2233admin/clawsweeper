## Summary

Keep this issue open. Current main has a narrow `messages.suppressToolErrors` boolean and heartbeat-only `suppressToolErrorWarnings`, but it does not implement a configurable target such as `lifecycleTarget`, `internalWarningTarget`, or category-based `messageRouting` for gateway/tool/lifecycle warnings. Related #39406 overlaps on warning-delivery policy but does not supersede this dedicated-channel routing request.

## Best Possible Solution

Keep open for maintainer product/design review, then implement a generic internal notification routing policy rather than a Discord-only special case. The likely direction is a documented `messages` config surface with safe defaults, optional suppression, category routing for lifecycle/tool/system notices, schema/help/docs coverage, and delivery through existing outbound target resolution with tests for active-channel fallback, dedicated-channel routing, and mutating tool safety.
