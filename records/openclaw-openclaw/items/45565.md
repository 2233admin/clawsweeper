## Summary

Keep open. Current main still does not implement a configurable dedicated target for gateway lifecycle or internal tool-warning messages: the only relevant public knob is suppression for some tool errors, while warning payloads are still ordinary error reply payloads delivered through the normal current/originating conversation route.

## What This Changes

The issue requests a `messages.*` configuration surface to route or suppress gateway lifecycle warnings and internal tool-failure notifications away from the active conversation channel.

## Best Possible Solution

Add a generic core message-routing policy under `messages.*` for internal warning categories, with strict schema/types/help/docs, validated target resolution through the existing channel routing layer, backward-compatible active-channel fallback, and regression coverage for lifecycle, tool-error, and system-notice delivery alongside the existing suppression behavior.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but the source-level check is high-confidence: current main rejects the proposed config keys and still emits warnings as normal `isError` reply payloads on the existing conversation route.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the exact proposed key names because no maintainer design exists yet; the maintainable direction is a shared, category-aware routing policy with validated targets, not a Discord-only special case or a blanket suppression default.
