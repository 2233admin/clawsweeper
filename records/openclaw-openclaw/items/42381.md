## Summary

Keep this PR open. Current main still lacks a remote node-host guard for `openclaw models auth login`, the linked bug is still the canonical open report, and this PR remains a focused implementation candidate rather than obsolete work.

## Best Possible Solution

Keep this PR open, rebase it onto current main, and review or land a focused remote node-host auth-write guard in `src/commands/models/auth.ts` with regression coverage in `src/commands/models/auth.test.ts`. Prefer the shared `src/gateway/net.ts:isLoopbackHost` helper for gateway host classification, and decide explicitly whether the guard should apply only to OAuth/device-code flows or to every provider auth method that writes local credentials. After merge, #42291 can close as implemented.

## Risks / Open Questions

- The PR-local loopback helper shown in the provided diff is narrower than the existing shared `src/gateway/net.ts:isLoopbackHost` helper and may falsely block valid loopback forms such as `127.0.0.2`, `[::1]`, or IPv4-mapped loopback addresses.
- The proposed guard runs before auth method selection, so it may block API-key, token, device-code, or custom provider auth methods unless maintainers intentionally want a broad remote-node auth-write block rather than OAuth-only behavior.
