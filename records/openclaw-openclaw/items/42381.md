## Summary

Keep this PR open. Current main still has no remote node-host guard in `openclaw models auth login`, the linked bug remains open, and this PR is a focused implementation candidate rather than obsolete work. It needs maintainer review/rebase before landing, mainly around loopback classification and whether the guard should apply to OAuth-only flows or every provider auth method that writes local credentials.

## Best Possible Solution

Keep this PR open, rebase it onto current main, and review/land a focused remote node-host auth-write guard in `src/commands/models/auth.ts` with regression coverage in `src/commands/models/auth.test.ts`. Prefer the shared `src/gateway/net.ts:isLoopbackHost` helper for gateway host classification, and decide explicitly whether the guard applies only to OAuth/device-code flows or to every provider auth method that writes local credentials. After merge, the linked bug can close as implemented.

## Risks / Open Questions

- The PR-local loopback helper shown in the provided diff is narrower than the existing shared helper and may falsely block valid loopback forms such as `127.0.0.2`, `localhost.`, `[::1]`, or IPv4-mapped loopback addresses.
- The proposed guard runs before auth method selection, so it may block API-key, token, device-code, or custom provider auth methods unless maintainers intentionally want a broad remote-node auth-write block rather than OAuth-only behavior.
- No supply-chain or CI permission regression is visible in the provided PR diff; the main remaining review risk is functional scope for an auth-sensitive command.
