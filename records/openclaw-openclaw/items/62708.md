## Summary

Keep open. Current main still declares, writes, clears, copies, and reads `claudeCliSessionId`, including an active Claude CLI history fallback with test coverage, so the cleanup is not implemented and needs a maintainer compatibility decision before removing old persisted-session support.

## What This Changes

The issue requests removing the legacy `claudeCliSessionId` field from `SessionEntry` and the related fallback, dual-write, passthrough, persistence, and test coverage.

## Best Possible Solution

Decide whether OpenClaw is ready to retire compatibility for session entries that only contain `claudeCliSessionId`; if yes, remove the field consistently from the session type, CLI helper fallback and dual-write paths, gateway/session copy paths, Claude history fallback, and tests without adding a startup migration.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a static refactor request. The current-main verification path is high confidence: `rg claudeCliSessionId` plus focused source/test inspection shows the field still exists and has intentional fallback coverage.

## Solution Assessment

Unclear until maintainers make the compatibility call. If they accept losing resume/history import for entries that only have `claudeCliSessionId`, the requested removal is the right direction, but the implementation must also cover `src/gateway/cli-session-history.claude.ts`, which the issue body did not list.
