## Summary

Keep open. The latest head addresses the earlier mediaUrls and schema-marker feedback, but the main delivery path still will not expose the new Matrix presentation hooks because the registered Matrix channel facade only forwards text, media, and poll methods. The new sendPayload path also still accepts arbitrary Matrix extraContent, which can override generated Matrix event fields.

## What This Changes

The PR adds Matrix MessagePresentation rendering and sendPayload support to the Matrix runtime adapter, adds outbound tests for metadata/media forwarding, and introduces a Matrix presentation metadata docs page.

## Best Possible Solution

Revise the PR so the registered Matrix channel facade exposes presentationCapabilities, renderPresentation, and sendPayload, either directly or through an expanded shared runtime delegate helper; restrict Matrix event metadata forwarding to a validated com.openclaw.presentation payload; add a core/facade delivery regression test; and wire the new docs page into the Matrix docs navigation or existing Matrix page.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to follow deliverOutboundPayloads into the registered Matrix channel facade: current main and the PR diff leave the new hooks off that facade, so a normal Matrix presentation reply will fall back to plain text instead of attaching com.openclaw.presentation.

## Solution Assessment

No, not yet. The runtime-level renderer, fallback text, schema marker, and mediaUrls handling are directionally useful, but the mergeable fix needs facade delivery wiring and a narrower Matrix metadata contract before it is the maintainable solution.
