## Summary

Keep open. Current main still lacks Matrix MessagePresentation metadata, so the contribution targets a real gap, but the latest diff is not mergeable because the new runtime hooks are not exposed through the registered Matrix channel facade and the proposed sendPayload path can forward arbitrary Matrix event-content fields.

## What This Changes

The PR adds Matrix outbound MessagePresentation rendering, Matrix extraContent forwarding through sendPayload, media/sendPayload unit coverage, and a new Matrix presentation metadata docs page.

## Best Possible Solution

Revise the PR so the Matrix channel facade or runtime delegate helper forwards presentationCapabilities, renderPresentation, and sendPayload into the registered outbound adapter; constrain sendPayload to validated com.openclaw.presentation metadata only; add a delivery-path regression test plus docs navigation and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to trace a Matrix ReplyPayload with presentation through deliverOutboundPayloads into createPluginHandler: current main reads renderPresentation/sendPayload from the registered Matrix outbound facade, but that facade only forwards sendText, sendMedia, and sendPoll, so the metadata path is never reached.

## Solution Assessment

No. The overall direction is appropriate and matches the downstream native Matrix client context in the discussion, but the current patch wires the hooks only into the runtime adapter and leaves Matrix extraContent too broad; the narrower maintainable fix is facade forwarding plus a whitelisted presentation metadata contract.
