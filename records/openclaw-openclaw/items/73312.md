## Summary

Keep open. Current main does not implement Matrix MessagePresentation metadata, and the PR is useful but still not mergeable because the new presentation hooks are only on the Matrix runtime adapter, Matrix event extra content is forwarded too broadly, and the docs/changelog surfaces are incomplete.

## What This Changes

The PR adds Matrix outbound MessagePresentation rendering, com.openclaw.presentation event metadata forwarding, outbound tests for presentation/media payloads, and a new Matrix presentation metadata docs page.

## Best Possible Solution

Revise this PR so the registered Matrix channel facade forwards presentationCapabilities, renderPresentation, and sendPayload, or extend createRuntimeOutboundDelegates to support those hooks; add a delivery/facade regression proving MessagePresentation reaches Matrix events; validate and whitelist only com.openclaw.presentation metadata; and wire the docs nav plus a single-line changelog entry before merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to trace normal Matrix delivery through src/infra/outbound/deliver.ts into the registered outbound adapter from extensions/matrix/src/channel.ts; the PR adds hooks to matrixOutbound, but the facade still exposes only sendText, sendMedia, and sendPoll.

## Solution Assessment

No. The fallback rendering and metadata shape are a reasonable direction, but this is not the best mergeable solution until the facade/core delivery path is wired and the Matrix metadata forwarding contract is constrained to validated presentation content.
