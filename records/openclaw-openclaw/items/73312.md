## Summary

Keep this PR open. Current main still lacks Matrix MessagePresentation event metadata, so the PR is not obsolete, but the latest diff still misses the registered Matrix channel facade wiring that core delivery uses and exposes a too-broad Matrix event-content forwarding path.

## What This Changes

The PR adds Matrix runtime presentation rendering and sendPayload handling, media URL coverage, and a new Matrix presentation metadata docs page.

## Best Possible Solution

Revise this PR so the Matrix channel facade forwards presentationCapabilities, renderPresentation, and sendPayload, or extend the shared runtime delegate helper to support those hooks; add a core delivery or channel facade test proving MessagePresentation reaches Matrix events; and constrain forwarded Matrix event metadata to a validated com.openclaw.presentation payload.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to follow core delivery through src/infra/outbound/deliver.ts into the registered Matrix channel facade: current main and the PR diff leave the new runtime hooks off that facade, so normal delivery cannot call them.

## Solution Assessment

No, not yet. The runtime-level approach and fallback rendering are directionally right, but the best maintainable fix must include facade/core delivery wiring and a narrower Matrix metadata forwarding contract before merge.

## Review Findings

Overall correctness: patch is incorrect

  - body: These hooks are only added to matrixOutbound, but core delivery reads renderPresentation and sendPayload from the registered channel facade.
