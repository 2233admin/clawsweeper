## Summary

Keep open. The PR still has merge-blocking wiring and security issues: it adds Matrix presentation hooks to the runtime adapter, but normal delivery uses the registered Matrix channel facade, which still exposes only the legacy send methods; the new sendPayload path also forwards arbitrary Matrix extraContent into event content.

## What This Changes

The PR adds Matrix runtime MessagePresentation rendering, forwards Matrix extraContent for com.openclaw.presentation, adds outbound tests, and adds a Matrix presentation metadata docs page.

## Best Possible Solution

Revise the PR so the Matrix channel facade forwards presentationCapabilities, renderPresentation, and sendPayload, or extend createRuntimeOutboundDelegates to support those hooks; add a delivery or facade regression test proving MessagePresentation reaches Matrix events; whitelist validated com.openclaw.presentation metadata only; and wire the docs navigation plus required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to trace normal Matrix delivery from src/infra/outbound/deliver.ts through the registered outbound handler in extensions/matrix/src/channel.ts; the PR's hooks are added to matrixOutbound but are not exposed by that registered facade.

## Solution Assessment

No. The runtime renderer and fallback direction are reasonable, but the proposed implementation is not the best mergeable solution until the registered delivery path is wired and the Matrix extraContent contract is constrained to validated presentation metadata.
