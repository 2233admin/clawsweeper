## Summary

Keep open. The PR targets a real Matrix presentation-metadata feature, but the latest head still is not mergeable: the new Matrix hooks live on the runtime outbound object while the registered Matrix channel facade does not expose them to core delivery, and the proposed sendPayload path forwards Matrix event-content fields too broadly.

## What This Changes

The PR adds Matrix MessagePresentation rendering and custom event metadata under com.openclaw.presentation, a Matrix presentation docs page, and Matrix outbound tests for rendering, extraContent forwarding, and media URL handling.

## Best Possible Solution

Revise this PR so the registered Matrix channel facade, or a generalized runtime delegate helper, exposes presentationCapabilities, renderPresentation, and sendPayload to core delivery; constrain Matrix sendPayload to validated com.openclaw.presentation metadata only; add a facade or delivery regression test proving normal Matrix replies carry the metadata; and wire the docs nav plus the required changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to trace deliverOutboundPayloads through the registered Matrix channel adapter: core reads renderPresentation and sendPayload from that facade, while the PR only adds those hooks to matrixOutbound behind a facade that still forwards sendText, sendMedia, and sendPoll.

## Solution Assessment

No. Rendering a plain-text fallback plus additive Matrix metadata is a reasonable direction, but the proposed implementation is not the best mergeable solution until the registered delivery path is wired and the Matrix extraContent contract is narrowed to validated presentation metadata.
