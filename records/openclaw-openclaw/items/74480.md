## Summary

Keep this PR open. Current main still passes the raw Active Memory `timeoutMs` into the embedded runner while the outer watchdog and prompt hook already include setup grace, so the PR addresses a real remaining inconsistency; I found no blocking correctness or security issue in the proposed patch, but it still needs maintainer review and targeted CI before merge.

## What This Changes

The PR adds a shared Active Memory embedded-recall timeout helper and uses `timeoutMs + setupGraceTimeoutMs` for the embedded runner, watchdog, and prompt hook, with regression-test expectations updated accordingly.

## Best Possible Solution

Land a narrow Active Memory fix that applies the same setup-grace-aware timeout at every embedded recall boundary, keep the strengthened unit test, and then have maintainers decide whether #73306 is fully resolved or whether broader timeout-family issues remain tracked by #73801 or another canonical issue.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists: configure a tiny Active Memory `timeoutMs`, set setup grace higher, and make the embedded runner take longer than the raw timeout but less than the grace-extended budget; current main passes only the raw timeout to the runner even though the outer hook can wait longer.

## Solution Assessment

Yes. The proposed helper is the narrowest maintainable fix for this boundary because it centralizes the existing effective timeout arithmetic without changing provider routing, memory search behavior, circuit-breaker policy, or the configured timeout clamp.
