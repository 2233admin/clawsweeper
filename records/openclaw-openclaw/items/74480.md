## Summary

Keep this PR open. Current main still passes the raw Active Memory `timeoutMs` into the embedded runner while the outer watchdog and hook registration already include setup grace, so the PR addresses a real remaining timeout-boundary inconsistency. I did not find a definite functional or security regression in the proposed patch from read-only review; it needs normal maintainer review and targeted CI before merge.

## What This Changes

The PR adds a shared Active Memory embedded-recall timeout helper and uses `timeoutMs + setupGraceTimeoutMs` for the embedded runner, outer watchdog, hook timeout, and related regression tests.

## Best Possible Solution

Land a narrow Active Memory fix that applies the setup-grace-aware timeout consistently at every embedded recall boundary, keep the regression test focused on the plugin-to-runner handoff, then decide whether #73306 is fully resolved or should stay open for any remaining timeout family.
