## Summary

Keep open. Current main contains the related text-only Discord intent gate, but the default voice-enabled path still requests voice-state events, installs the Discord voice adapter, loads the voice manager, and registers the ready listener; I found no source, test, docs, changelog, related-item, or history evidence that the reported enabled-voice idle CPU spin has been fixed.

## What This Changes

The issue reports sustained idle CPU in the embedded Discord gateway when Discord voice is enabled by default, even after the separate text-only `GuildVoiceStates` intent fix.

## Best Possible Solution

Keep this open and investigate in the Discord plugin. Reproduce/profile `openclaw-gateway` on Linux with embedded Discord and `channels.discord.voice.enabled=true`, isolate the hot loop, then either patch the specific voice runtime path or make an explicit maintainer-approved default/runtime change with focused regression coverage and docs/changelog updates.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for current main: the report gives concrete v2026.4.26 observations and an A/B workaround, and source inspection confirms the implicated enabled-voice path remains active, but I did not live-profile current main with real Discord bot accounts.

## Solution Assessment

Is this the best way to solve the issue? Not applicable for a proposed patch; closing via the #73709 text-only intent fix would not solve this enabled-voice report. The narrowest maintainable path is to profile the enabled voice runtime first, then patch the actual loop or deliberately change the voice default with maintainer approval.
