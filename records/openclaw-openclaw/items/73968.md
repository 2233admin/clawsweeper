## Summary

Keep this PR open, but do not merge it as-is. The proposed helper is aimed at the right routing field problem, but it only sanitizes the direct gateway `runHeartbeatOnce` wrapper; the normal queued/fallback cron wake path can still merge `target: "last"` with inherited `heartbeat.to`/`accountId` and preserve the cross-context delivery leak reported in #73900.

## What This Changes

The PR adds a gateway cron helper that omits inherited heartbeat `to` and `accountId` when `runHeartbeatOnce` receives `target: "last"`, plus a regression test and changelog entry.

## Best Possible Solution

Apply the destination-field omission at the shared cron wake override boundary used by both direct and queued wakes, or in the heartbeat wake runner's targeted merge when the request is a cron-forced `target: "last"` wake, then add regression coverage that runs the actual main-session cron job path rather than only invoking the adapter dependency directly.

## Reproduction Assessment

Yes. The source path is high-confidence: a main-session cron job marks itself active, direct heartbeat execution can skip as `cron-in-progress`, the queued wake merges `{ target: "last" }` with configured heartbeat defaults, and delivery resolution still honors inherited `to`/`accountId`.

## Solution Assessment

No. The proposed helper is directionally correct for one wrapper call, but the narrow maintainable fix must cover both `runHeartbeatOnce` and queued `requestHeartbeatNow` cron wakes without changing general explicit heartbeat destination semantics.
