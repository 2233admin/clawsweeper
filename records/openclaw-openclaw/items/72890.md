## Summary

Keep open. Current main still decides Anthropic adaptive-thinking support from Claude model-id substrings, while configured custom `anthropic-messages` model metadata has no adaptive-thinking capability that reaches thinking-level validation or the Anthropic request payload path.

## What This Changes

The issue requests a config-declared adaptive-thinking capability for custom Anthropic Messages provider models whose opaque IDs do not match OpenClaw's built-in Claude model allowlist.

## Best Possible Solution

Add a typed config-backed adaptive-thinking capability for configured `anthropic-messages` models, consume it in thinking-profile validation and the Anthropic transport option/payload resolver, keep the Claude model-id list as fallback, and document the custom proxy example without customer-specific prefixes.

## Reproduction Assessment

Yes. A focused unit test can configure a custom `models.providers.my-proxy` model with `api: "anthropic-messages"`, `id: "zyb-high"`, `reasoning: true`, and a non-off thinking level; current main will miss the adaptive gate and build a budget-token thinking payload instead of `thinking.type: "adaptive"`.

## Solution Assessment

Yes. A typed config/compat capability is the narrowest maintainable solution because it supports opaque proxy IDs without hardcoding corporate prefixes or forcing otherwise simple `models.providers` users to write a provider plugin.
