## Summary

Keep open. Current main still gates Anthropic adaptive-thinking payloads on Claude model-id substrings, and configured custom-provider metadata cannot currently advertise adaptive support for an opaque `anthropic-messages` proxy model.

## What This Changes

The issue requests a config-declared adaptive-thinking capability so custom Anthropic-compatible proxy model IDs can opt into adaptive thinking without hardcoded model ID prefixes.

## Best Possible Solution

Add a typed config-backed adaptive-thinking capability for configured `anthropic-messages` models, consume it in thinking-profile validation and Anthropic transport option/payload resolution, preserve the Claude model-id list as fallback behavior, and document a generic custom proxy example without customer-specific prefixes.

## Reproduction Assessment

Yes. A focused unit test can configure `models.providers.my-proxy` with `api: "anthropic-messages"`, an opaque model id such as `zyb-high`, `reasoning: true`, and a non-off thinking level; current main will miss the adaptive gate and build a budget-token thinking payload instead of `thinking.type: "adaptive"`.

## Solution Assessment

Yes. A typed config/compat capability is the narrowest maintainable path because it supports opaque proxy IDs without hardcoding corporate prefixes or requiring an otherwise simple custom provider to become a provider plugin.
