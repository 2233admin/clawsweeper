## Summary

Keep open. Current main still routes Anthropic adaptive-thinking payloads through hardcoded Claude model-id substring checks, and the typed custom-provider model config/schema/docs do not expose a way for an opaque Anthropic-compatible proxy model id to declare adaptive-thinking support. The earlier ClawSweeper comment is still consistent with current main; related open PRs address adjacent Claude-name matching and effort issues, but they do not supersede this config-backed custom-provider request.

## What This Changes

The issue requests a config-declared adaptive-thinking capability for custom Anthropic-compatible provider model IDs so opaque proxy model names can use `thinking.type: "adaptive"` and `output_config.effort`.

## Best Possible Solution

Add a typed model compat/config capability that lets configured Anthropic Messages models declare adaptive-thinking support, then consume that same normalized capability in thinking-profile validation, Anthropic transport option resolution, payload shaping, schema/help output, and public docs.

## Reproduction Assessment

Yes. A focused unit reproduction can configure `models.providers.my-proxy` with `api: "anthropic-messages"`, an opaque reasoning-capable model id such as `zyb-high`, and a thinking level; current main will use the non-adaptive budget-token branch because `supportsAdaptiveThinking(model.id)` only matches Claude-name substrings.

## Solution Assessment

Yes. The narrow maintainable direction is a typed config/compat capability consumed by the existing model catalog and Anthropic transport paths; hardcoding customer prefixes or relying only on a custom provider plugin would preserve the upgrade-fragile behavior reported here.
