## Summary

Keep open. Current main still gates Anthropic adaptive-thinking payloads on Claude model-id substrings, while custom provider model config exposes generic reasoning and OpenAI-style effort metadata but no typed Anthropic adaptive-thinking override. The existing ClawSweeper comment reached the same keep-open conclusion, and the relevant source/docs surfaces still match that finding.

## What This Changes

Allow custom Anthropic-compatible provider models to declare adaptive-thinking support so opaque proxy model IDs can receive adaptive thinking payloads without patching model-id allowlists.

## Best Possible Solution

Add a generic, typed model compat/config capability for Anthropic adaptive thinking, have thinking profile resolution advertise and validate adaptive levels from that metadata, and have the Anthropic Messages transport consume the same metadata when deciding between adaptive effort and budget-token thinking.

## Reproduction Assessment

Yes. A high-confidence unit reproduction can configure a custom anthropic-messages model with an opaque id and reasoning enabled, then request a thinking level; current main will not advertise adaptive for that catalog entry and the transport will take the budget-token branch because supportsAdaptiveThinking(model.id) returns false.

## Solution Assessment

Yes. A typed config/compat flag wired through schema, thinking-profile validation, gateway/session surfaces, and Anthropic transport is the narrow maintainable fix; hardcoding customer proxy prefixes would preserve the reported upgrade-fragile behavior.
