## Summary

Keep open for normal maintainer review. Current main still keeps PID-alive lock owners without start-time evidence, and the PR is a narrow patch for true-legacy owner records with focused tests and changelog coverage; I did not find a blocking correctness or security issue in that stated scope.

## What This Changes

The PR adds mtime-based stale expiry for PID-alive bundled runtime-deps lock owners missing both starttime and createdAtMs, adds regression tests for stale/fresh legacy locks and createdAt-bearing locks, and adds an Unreleased changelog entry.

## Best Possible Solution

Land the narrow true-legacy fallback with its regression tests after maintainer scope confirmation and exact-head checks, then close or link the paired #74948 report. Keep broader recovery for pre-starttime but createdAt-bearing Docker locks as a separate, explicit maintainer decision because it has a higher active-install safety tradeoff.

## Reproduction Assessment

Yes. A direct helper call on current main with isAlive(pid) returning true, no starttime, no createdAtMs, and stale lock mtimes follows the live-PID branch to return false, so the stale lock is not removed.

## Solution Assessment

Yes for the stated true-legacy case. The PR uses stale mtime evidence only when modern owner fields are absent, preserving current behavior for active starttime-bearing and createdAt-bearing locks.
