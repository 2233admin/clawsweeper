## Summary

Keep open. The amended PR targets the right legacy Docker lock edge case, but its fallback is still too broad: any PID-alive lock with no stored starttime and stale mtimes can be removed, including current active locks on platforms where starttime is unavailable but `createdAtMs` exists.

## What This Changes

The PR adds a mtime-based stale fallback for PID-alive runtime-deps locks that lack starttime evidence and updates bundled runtime-deps tests for the legacy Docker lock case.

## Best Possible Solution

Keep the legacy mtime fallback, but narrow it so it only applies to true legacy owner records that lack both trustworthy starttime evidence and `createdAtMs`; add regression coverage for a PID-alive, createdAt-bearing, no-starttime lock with stale mtimes staying held.

## Reproduction Assessment

Yes. The original issue has a direct function-level reproduction with a live PID, no starttime, no `createdAtMs`, and stale lock mtime; the introduced regression is likewise reproducible by adding `createdAtMs` to that no-starttime live-PID owner while keeping stale mtimes.

## Solution Assessment

No. The direction is right, but the current PR condition is broader than the stated legacy lock shape and can violate the active-lock safety invariant on platforms without starttime support.
