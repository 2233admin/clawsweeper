## Summary

Keep open. Current main still keeps a PID-alive runtime-deps lock held when both start-time evidence and createdAt evidence are absent, while this PR adds a narrow true-legacy mtime fallback plus focused tests. I did not find a blocking correctness or security issue in the proposed patch, but it still needs maintainer review, final CI, and scope confirmation because createdAt-bearing stale Docker locks remain intentionally out of scope.

## What This Changes

The PR adds a legacy-only mtime fallback to `shouldRemoveRuntimeDepsLock()` for PID-alive owner records missing both `starttime` and `createdAtMs`, plus regression tests for stale true-legacy locks and createdAt-bearing live locks.

## Best Possible Solution

Land the narrow true-legacy fallback if CI is clean and maintainers agree the scope is no-starttime/no-createdAt lock recovery. If operators are still blocked by stale createdAt-bearing Docker locks, track that as a separate design because it needs a safe way to distinguish old container incarnations from active non-Linux installs.

## Reproduction Assessment

Yes. The focused reproduction is a direct call to `shouldRemoveRuntimeDepsLock()` with a live PID, stale lock mtime, and no `starttime` or `createdAtMs`; current main returns held, while the PR test asserts stale locks expire and fresh ones stay held.

## Solution Assessment

Yes for the true no-createdAt legacy shape. The PR is intentionally narrow and avoids using age alone for createdAt-bearing live locks, which is the safer current invariant for non-Linux and active-install cases.
