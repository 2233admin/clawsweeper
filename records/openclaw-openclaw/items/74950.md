## Summary

Keep open. Current main still keeps a PID-alive runtime-deps lock held when both starttime and createdAt evidence are absent, while the latest PR head adds a narrow stale-mtime fallback for that true legacy shape and addresses the earlier createdAt-bearing live-lock regression. I found no new actionable code-review findings; the remaining action is normal maintainer review, CI completion, and deciding whether createdAt-bearing pre-starttime Docker locks need a separate follow-up.

## What This Changes

The PR adds a legacy-only mtime fallback in `shouldRemoveRuntimeDepsLock()` for PID-alive bundled runtime-deps locks without `starttime` or `createdAtMs`, plus regression tests for stale true-legacy locks and createdAt-bearing live locks.

## Best Possible Solution

Land the narrow legacy-lock predicate and tests if CI finishes cleanly, preserving active-lock safety for starttime-matched and createdAt-bearing locks. If operators also have stale Docker locks with `createdAtMs` but no `starttime`, track that as a separate design that can distinguish old container incarnations without expiring current non-Linux installs by age alone.

## Reproduction Assessment

Yes. The focused reproduction is a direct call to `shouldRemoveRuntimeDepsLock()` with a live PID, no `starttime`, no `createdAtMs`, and stale lock mtimes; current main returns held, while the PR test asserts stale locks expire and fresh ones do not.

## Solution Assessment

Yes for the true no-createdAt legacy shape. The latest predicate is the narrowest maintainable fix I see in this PR because it uses existing mtime data only when both stronger owner fields are absent and adds a regression test for the prior createdAt-bearing live-lock hazard.
