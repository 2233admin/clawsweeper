## Summary

Keep open. Current main still keeps live-PID runtime-deps locks before consulting stale mtimes, and this PR is an active, narrow implementation for the true-legacy no-starttime/no-createdAt lock shape. I found no blocking correctness or security issue in the supplied diff; the remaining work is maintainer review, CI/Testbox validation, and scope confirmation around createdAt-bearing stale Docker locks.

## What This Changes

The PR adds a true-legacy mtime fallback to bundled runtime-deps lock expiry and regression tests for stale/fresh no-starttime/no-createdAt locks plus createdAt-bearing live locks.

## Best Possible Solution

Land the narrow true-legacy fallback after maintainer review and green validation if the intended scope is no-starttime/no-createdAt lock recovery. Keep createdAt-bearing stale Docker lock recovery as a separate design decision unless maintainers choose a safe cross-platform disambiguation rule.

## Reproduction Assessment

Yes. The focused reproduction is a direct call to `shouldRemoveRuntimeDepsLock()` with a live PID, no `starttime`, no `createdAtMs`, and stale lock mtimes; current main returns held, while the PR adds tests expecting stale true-legacy locks to expire and fresh ones to stay held.

## Solution Assessment

Yes for the true-legacy shape. The PR is narrow because it only uses mtime when both stronger owner fields are absent, while preserving the existing safety invariant for createdAt-bearing live locks on platforms without process start-time evidence.
