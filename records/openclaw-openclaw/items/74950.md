## Summary

Keep open. The legacy Docker lock edge is valid against current main, but the PR’s fallback is too broad and can remove a lock still owned by a live, same-incarnation installer after the stale-age threshold.

## What This Changes

The PR adds createdAtMs and lock/owner-file mtime fallback staleness checks inside `shouldRemoveRuntimeDepsLock()` for PID-alive runtime-deps locks and updates the bundled runtime deps tests for the legacy Docker lock case.

## Best Possible Solution

Land a narrower predicate that expires only legacy PID-alive locks with no trustworthy process start-time and no `createdAtMs`, using stale lock-dir/owner-file mtime, while preserving live same-incarnation and active createdAt-bearing locks.

## Reproduction Assessment

Yes. A direct predicate check on current main with a live PID, no start-time evidence, no `createdAtMs`, and an old `lockDirMtimeMs` follows the PID branch and returns `false`, matching the stale legacy Docker lock scenario.

## Solution Assessment

No. The PR targets the right edge case, but the best fix should limit mtime fallback to true legacy owner files instead of expiring any live PID lock after `createdAtMs` or mtime crosses the stale threshold.

## Review Findings
