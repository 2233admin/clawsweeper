## Summary

Keep this issue open. Current main still lets the store-only `/allowlist` path skip the config-write authorization branch and then mutate the pairing allowlist store, while the linked focused fix PR #72361 is open and unmerged.

## What This Changes

The issue reports that `/allowlist add|remove --store` can mutate the persistent pairing allowlist store without honoring channel or account `configWrites=false`.

## Best Possible Solution

Land PR #72361 or an equivalent narrow patch that applies the existing config-write authorization before every pairing-store allowlist mutation, with regression coverage for store add and store remove when `configWrites=false`.

## Reproduction Assessment

Yes. The source path is enough for high-confidence reproduction: `/allowlist add|remove --store` sets `shouldUpdateConfig` false, skips `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`.

## Solution Assessment

Yes. Reusing the existing config-write policy before the store mutation is the narrowest maintainable fix unless maintainers intentionally add and document a separate permission for pairing-store edits.
