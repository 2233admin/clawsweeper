## Summary

Keep this issue open. Current main still lets store-targeted `/allowlist add|remove` skip the `configWrites` denial path before mutating the pairing allowlist store, and open PR #72361 is already linked as the focused implementation candidate.

## What This Changes

Require `/allowlist add|remove --store` to honor channel and account `configWrites` before mutating the persistent pairing allowlist store.

## Best Possible Solution

Apply the existing `resolveConfigWriteDeniedText`/`authorizeConfigWrite` policy before any persistent pairing-store allowlist mutation, using the same origin channel/account semantics as config writes, and add focused regression tests for store add/remove with `configWrites=false`.

## Reproduction Assessment

Yes. A focused code-path check on current main is enough: `/allowlist add dm --store <id>` sets `shouldUpdateConfig=false`, skips the denial check, and reaches `updatePairingStoreAllowlist(...)`.

## Solution Assessment

Yes. Reusing the existing config-write authorization is the narrowest maintainable fix unless maintainers intentionally define a separate permission surface for pairing-store allowlist edits.
