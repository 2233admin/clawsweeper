## Summary

Keep this issue open. Current main still routes store-targeted `/allowlist add|remove` around the existing `configWrites` denial check before mutating the pairing allowlist store, and open PR #72361 is already the focused, unmerged fix candidate linked with closing syntax.

## What This Changes

The issue requests that `/allowlist add|remove --store` honor channel/account `configWrites` before mutating the persistent pairing allowlist store.

## Best Possible Solution

Land PR #72361 or an equivalent narrow patch that applies the existing config-write authorization policy before every pairing-store allowlist mutation, with focused regression tests for store add and remove when `configWrites=false`. If maintainers intentionally want pairing-store edits to use a separate permission, add that explicit policy and update the docs instead of leaving `--store` as an undocumented exception.

## Reproduction Assessment

Yes. A high-confidence source-path reproduction exists on current main: with an owner-authorized sender on a pairing-capable channel where `configWrites=false`, `/allowlist add dm --store <id>` skips the policy branch and reaches the pairing-store update call.

## Solution Assessment

Yes. Reusing `resolveConfigWriteDeniedText` or the underlying `authorizeConfigWrite` policy before store mutation is the narrowest maintainable fix unless a separate pairing-store write permission is deliberately introduced and documented.
