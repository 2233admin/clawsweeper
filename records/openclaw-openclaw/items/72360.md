## Summary

Keep this issue open. Current main still lets store-targeted `/allowlist add|remove` skip the existing `configWrites` authorization branch before mutating the pairing allowlist store, and open PR #72361 already references this issue as the focused fix candidate.

## What This Changes

The issue asks for `/allowlist add|remove --store` to honor channel/account `configWrites` before mutating the persistent pairing allowlist store.

## Best Possible Solution

A merged narrow fix, whether PR #72361 or an equivalent patch, should enforce the existing config-write policy before any pairing-store allowlist mutation and add focused regression coverage for `configWrites=false` store add/remove paths.

## Reproduction Assessment

Yes. The current-main code path is enough to reproduce with an owner-authorized sender on a pairing-capable channel: `/allowlist add dm --store <id>` skips the config-write denial branch and reaches the store update call.

## Solution Assessment

Yes. Reusing `resolveConfigWriteDeniedText`/`authorizeConfigWrite` before store mutations is the narrowest maintainable fix unless maintainers intentionally define a separate permission for pairing-store edits.
