## Summary

Keep this issue open. Current main still lets `/allowlist add|remove --store` skip the `resolveConfigWriteDeniedText(...)` / `configWrites` gate and then call the pairing-store mutation path. The provided GitHub context also shows open PR #72361 with closing syntax for this issue, so the right maintainer path is to review and land or close that PR rather than close this tracking issue early.

## Best Possible Solution

Keep this issue open until #72361 or an equivalent focused fix is merged. The fix should apply the same config-write authorization to store-only `/allowlist add|remove --store` mutations before `updatePairingStoreAllowlist(...)` runs, and add regression tests for origin and target account/channel `configWrites=false` cases. If pairing-store edits are intentionally a separate permission surface, the product/docs path should define and document that separate gate instead of relying on the current bypass.
