## Summary

Keep this issue open. Current main still lets the store-targeted `/allowlist` mutation path skip the existing config-write authorization check, and the linked closing PR is open and unmerged in the provided issue context, so this should remain tracked until that PR or an equivalent security fix lands.

## What This Changes

The issue reports that `/allowlist add|remove --store` should honor channel/account `configWrites=false` before mutating the persistent pairing allowlist store.

## Best Possible Solution

Land PR #72361 or an equivalent narrow patch that applies the existing config-write authorization before every pairing-store allowlist mutation, with focused regression coverage for both add and remove `--store` requests when origin or target `configWrites=false`. If maintainers intentionally want store edits governed by a separate permission, that permission should be explicit and documented instead of silently bypassing `configWrites`.

## Reproduction Assessment

Yes. Current main reproduces by source inspection: an owner-authorized `/allowlist add|remove ... --store` sets `shouldUpdateConfig` false, skips the branch with `resolveConfigWriteDeniedText(...)`, and reaches `updatePairingStoreAllowlist(...)`.

## Solution Assessment

Yes. Reusing the existing `resolveConfigWriteDeniedText`/`authorizeConfigWrite` policy for store-targeted allowlist mutations is the narrowest maintainable fix; a separate permission would need product and docs follow-through.
