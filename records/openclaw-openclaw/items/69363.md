## Summary

Keep PR #69363 open. Current main still has the exact parser behavior the PR targets: `parseCliJsonl` keeps the first session_id it sees, while the JSON and streaming parsers still let later records overwrite session IDs, and there is no current main guard for Claude `hook_started`/`hook_response` records or regression test for transient hook IDs.

## Best Possible Solution

Keep the PR open and land or adapt its shared `nextCliSessionId(parsed, current, backend)` approach in `src/agents/cli-output.ts`, with regression tests covering `parseCliJson`, `parseCliJsonl`, and `createCliJsonlStreamingParser` so Claude hook session IDs are fallback-only and never replace real conversation IDs.
