## Summary

Keep open. Current main has partial lock-based main-session restart recovery, but the timeout/failure path still only counts a failed recovery attempt; it does not attach/import the existing transcript into a replacement session or provide a durable usable-history fallback.

## What This Changes

The issue asks OpenClaw to recover or expose main-session context after a gateway-restart recovery timeout instead of silently creating a fresh, contextless session while old transcript files remain on disk.

## Best Possible Solution

Fold this into the Gateway graceful-restart recovery track: wait for Gateway readiness before resume attempts, make timeout state explicit, avoid unsafe hidden replay for channel-backed sessions, and add either a bounded transcript-backed recall/import path or a clear user-visible recovery failure that links the old session history.

## Reproduction Assessment

Yes. A focused source-level reproduction is to create a running main-session store row with `abortedLastRun: true` and a resumable transcript tail, mock `callGateway` to time out, and observe `recoverRestartAbortedMainSessions` returning `failed` without attaching the old transcript to a replacement session.

## Solution Assessment

No, current main is only a partial fix. The existing synthetic resume RPC is useful, but the durable context-loss problem still needs readiness/retry semantics plus a transcript-backed fallback or explicit recovery UX; the workspace-first workaround in the discussion is operationally helpful but not a core recovery fix.
