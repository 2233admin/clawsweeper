## Summary

Keep open. Current main has a partial main-session restart recovery path, but a Gateway `agent` RPC timeout still counts as a failed recovery without a transcript-backed import/fallback or explicit user-visible recovery result, and the broader restart-recovery track remains open.

## What This Changes

The issue reports that main-session recovery after a Gateway restart can time out and leave prior transcript context orphaned instead of being retried or imported into the replacement session.

## Best Possible Solution

Treat this as part of the graceful Gateway restart recovery design: preserve the old main-session binding while recovery is pending, retry after Gateway readiness, and if resume is not accepted, surface an explicit recovery result with a bounded transcript-backed history/import path that respects transcript integrity rules.

## Reproduction Assessment

Yes, at source level: mark a running main-session row with `abortedLastRun: true`, give it a resumable transcript tail, and make the Gateway `agent` call time out; recovery returns a failed attempt without transcript import. A live reproduction should still use the reported Windows/Telegram restart timing to confirm the race in the field.

## Solution Assessment

No, current main is not the complete fix. The existing implementation is a useful partial retry path, but the durable context-recovery problem needs a maintained fallback or explicit failure UX aligned with #57425.
