## Summary

Keep open. Current main has partial main-session restart recovery, but a timeout in the bounded Gateway resume RPC still leaves recovery as failed without a transcript-backed fallback or explicit recovered-history path, which matches the reported context-loss class.

## What This Changes

The issue reports that main-session restart recovery times out after a gateway restart and then creates a fresh session without transcript-backed context recovery.

## Best Possible Solution

Treat this as part of the Gateway graceful-restart recovery track: preserve the old main-session binding while recovery is pending, retry after Gateway readiness, and if resume cannot be accepted, surface an explicit recovery result plus a bounded transcript-backed history/import path instead of losing usable context.

## Reproduction Assessment

Yes, at source level: create a running main-session row with `abortedLastRun: true` and a resumable transcript tail, then make the Gateway `agent` RPC time out; `recoverRestartAbortedMainSessions` returns `failed` without importing transcript history into a replacement session. A full live reproduction should use the reported Windows/Telegram restart timing.

## Solution Assessment

No for closing as solved. The current 10-second synthetic resume path is useful partial recovery, but the durable context-recovery problem still needs a fallback or explicit UX aligned with the broader #57425 restart-recovery design.
