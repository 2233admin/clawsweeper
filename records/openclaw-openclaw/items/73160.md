## Summary

Keep open. Current main still has only partial main-session restart recovery: a stale-lock marker can trigger a bounded resume attempt, but a Gateway `agent` RPC timeout is still counted as failed without attaching/importing the existing transcript into the replacement active session.

## What This Changes

The issue requests reliable main-session recovery after Gateway restart timeouts, including retry/fallback behavior and access to the prior transcript-backed context instead of silently starting fresh.

## Best Possible Solution

Fold this into the graceful Gateway restart recovery design: keep the interrupted main-session binding recoverable while Gateway readiness settles, make retry state explicit, and add a bounded transcript-backed attach/import or clear user-visible failure path when live resume cannot be accepted.

## Reproduction Assessment

Yes. At source level, a running main session with `abortedLastRun: true` and a resumable transcript tail calls `callGateway` with a 10 second timeout; if that call times out, recovery reports `failed` and leaves the transcript detached from any replacement active session.

## Solution Assessment

No. The current RPC-only resume path is a useful partial fix, but it is not the best complete solution because it has no durable context fallback after repeated Gateway timeouts.
