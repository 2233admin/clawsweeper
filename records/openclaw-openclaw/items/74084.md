## Summary

Keep open. Current main does not implement the proposed bindings-array or content-fingerprint cache fallback, and the PR contains focused code and tests for the rebuilt-cfg workload. It is still an open, unmerged routing hot-path change with a failed core check and an unresolved FIFO-vs-LRU eviction note, so it needs maintainer review before landing.

## What This Changes

This PR adds secondary evaluated-bindings cache lookup by bindings array identity and SHA-256 content fingerprint in `src/routing/resolve-route.ts`, plus tests for rebuilt configs with identical or changed binding content.

## Best Possible Solution

Keep the PR open, resolve the eviction semantics by either refreshing fingerprint hits or naming the policy accurately, add or retain focused rebuilt-cfg coverage, capture small performance evidence for large binding sets, and land only after the routing checks and changed gate are green.

## Reproduction Assessment

Yes. Source inspection shows current main rebuilds the evaluated-bindings by-channel cache for fresh cfg objects, and the PR adds a focused rebuilt-cfg test for identical binding content; I did not run tests because this was a read-only review.

## Solution Assessment

Unclear. The cache boundary is narrow and plausible, but the PR still needs CI proof, perf evidence, and a decision on FIFO versus true LRU before it can be treated as the best routing hot-path fix.
