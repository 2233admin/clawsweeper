## What This Changes

The PR enables `tools.loopDetection.enabled` by default, updates tests/config help/types/docs metadata, normalizes some exec outcome hashes, and changes generic repeat detection to warn or block based on consecutive identical tool arguments.

## Best Possible Solution

Land one narrow, rebased canonical fix that enables loop detection by default, preserves explicit opt-out, keeps current main's stable exec outcome hashing and run-scoped history, updates every user-facing config/docs surface consistently, and gates any critical blocking on accumulated no-progress evidence rather than repeated arguments alone.

Yes. Current main has a source-level and unit-test reproduction: repeated identical calls with omitted config still resolve through `enabled: false`, and the existing default test asserts no loop is detected.

## Solution Assessment

No as-is. Enabling the guard by default is the right direction, but this PR should be narrowed so generic repeats do not hard-block while results are changing and the documentation is fully synchronized.
