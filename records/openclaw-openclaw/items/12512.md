## Summary

Keep open. Current main still exposes eligible skills through a model-facing `<available_skills>` catalog and tells the agent to read and follow the selected `SKILL.md`; I did not find instruction isolation, role locking, per-skill role/memory/tool capability metadata, or behavioral monitoring. The prior April 26, 2026 maintainer/Codex comment remains accurate against current main SHA `b85edb3f0cf6c2a8bec06537af9211a044266a6c`.

## Best Possible Solution

Keep this open as core security hardening. Define a trusted/untrusted precedence model for `SKILL.md`, add enforceable per-skill capability metadata for role, memory, and tool influence, apply it through prompt assembly/read handling/tool policy/memory/sandbox behavior, and add malicious-skill regression tests plus docs. ClawHub moderation and scanning can supplement this boundary, but should not replace an explicit core boundary for installed skills.
