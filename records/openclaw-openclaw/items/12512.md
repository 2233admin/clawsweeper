## Summary

Keep open. Current main still exposes skills through a model-facing catalog and instructs the model to read and follow selected SKILL.md files; existing controls cover skill visibility, allowlists, install/generation scanning, sandboxing, and tool policy, but not the requested trusted/untrusted instruction isolation, role locking, or per-skill memory/tool capability enforcement. The April 26 and April 29 issue comments also kept this open, and the current relevant source still matches that assessment.

## What This Changes

Request to add instruction isolation and per-skill capability boundaries for SKILL.md so untrusted skill instructions cannot override agent role, memory, or tool policy.

## Best Possible Solution

Define and document a trusted/untrusted skill precedence model, extend skill metadata and runtime policy with role, memory, and tool-influence controls, enforce that boundary through prompt assembly plus runtime policy, and add malicious SKILL.md regression coverage around the affected prompt and skill paths.
