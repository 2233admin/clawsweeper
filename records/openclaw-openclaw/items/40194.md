## Summary

Keep open. Current main still builds Feishu group session keys from channel + group peer id without the Feishu account/app identity, so multiple Feishu apps in the same group can still collide when routed to the same agent.

## Best Possible Solution

Keep this issue open and implement account-aware Feishu group session isolation. Prefer a Feishu-owned or additive routing/session-key option that includes a normalized accountId/app identity for non-default multi-account group routes while preserving legacy single-account/default-account continuity, then add regression tests and update Feishu/channel routing docs.

## Clownfish Work Prompt

Fix the Feishu same-group multi-account session collision reported in https://github.com/openclaw/openclaw/issues/40194. Current main passes accountId through route resolution but group session keys are still `agent:<agentId>:feishu:group:<chatId>` for default Feishu group scope, so multiple Feishu apps/accounts in the same group can collide when routed to the same agent. Implement a conservative account-aware isolation path for Feishu group sessions, preferably Feishu-owned or via an additive generic routing option. Preserve existing single-account/default-account continuity where feasible, do not globally change all channel group key shapes unless the compatibility story is explicit, and do not break Feishu broadcast cross-account dedupe. Add a regression test with two Feishu accounts/apps in the same group routed to the same agent proving distinct session keys and correct account-specific dispatch identity. Update Feishu/channel-routing docs for the new multi-account group behavior. Add a user-facing changelog entry if repo policy requires one, crediting @tontonbinary; do not add unrelated refactors or change unrelated channels.
