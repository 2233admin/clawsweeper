## Summary

Keep this issue open. Current main still passes Feishu accountId through routing and reply dispatch, but the group session key itself is accountless, so multiple Feishu bot accounts in the same group can still share the same same-agent transcript/session bucket.

## Best Possible Solution

Keep this issue open. Implement account-aware Feishu group session isolation in the Feishu plugin, likely by appending a normalized accountId/appId-derived suffix for non-default account group routes while preserving legacy default-account continuity. Add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct session keys and correct outbound account identity, then update the Feishu/channel-routing docs for the compatibility behavior.

## Risks / Open Questions

- Changing the generic group key shape globally would break existing transcript continuity; a Feishu-specific compatibility path or explicit account-scoped group mode is safer.
- Account-scoped bindings can route different Feishu accounts to different agents, but they do not solve the reported same-agent, same-group transcript collision.
- Feishu broadcast cross-account dedupe is separate from this single-agent session isolation bug and should not be treated as a fix.
