## Summary

Current main still appears vulnerable to the reported Feishu multi-bot group collision: Feishu passes accountId into route resolution and replies, but the group session key used for transcript/session storage is still keyed only by agent, channel, peer kind, and group peer id. Multiple Feishu app accounts in the same group routed to the same agent can therefore share one session bucket.

## Best Possible Solution

Keep this issue open. Implement account-aware Feishu group session isolation in the Feishu plugin, probably by appending a normalized accountId/appId-derived suffix for non-default account group routes while preserving legacy default-account continuity. Add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct session keys and correct outbound account identity, then update Feishu/channel-routing docs for the compatibility behavior.

## Risks / Open Questions

- Changing the generic group key shape globally would break existing transcript continuity; a Feishu-specific compatibility path or explicit account-scoped group mode is likely safer.
- Account-scoped bindings can route different Feishu accounts to different agents, but they do not solve the same-agent, same-group transcript collision reported here.
- Feishu broadcast cross-account dedupe is a separate duplicate-dispatch safeguard and does not fix the single-agent session isolation problem.
