## Summary

Keep this issue open. Current main still carries Feishu accountId through routing and reply dispatch, but the group session key itself remains account/app agnostic, so multiple Feishu apps in the same group can still converge on the same agent group session when they resolve to the same agent.

## Best Possible Solution

Keep this issue open. Implement account-aware Feishu group session isolation in the Feishu plugin or through an additive routing/session-key option, likely appending a normalized accountId or appId for non-default Feishu group routes while preserving legacy default-account continuity. Add a regression test with two Feishu accounts in the same group routed to the same agent, and update Feishu/group routing docs to document the compatibility behavior.

## Risks / Open Questions

- A blanket change to generic group session key shape could break transcript continuity for existing group channels, so the fix likely needs a Feishu-specific compatibility path or an additive opt-in/account-scoped session-key seam.
- Account-scoped bindings can route different Feishu accounts to different agents, but they do not isolate multiple accounts in the same group when those accounts intentionally resolve to the same agent/session.
