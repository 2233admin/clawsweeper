## Summary

Keep #58713 open. Current main still routes `sessions_send` into the resolved existing target session, then runs reply-back and announce steps against the requester/target session keys. The config schema exposes only enable/allow and ping-pong limits for agent-to-agent messaging; there is no ephemeral A2A session option or cleanup lifecycle. `sessions_spawn` is documented as isolated, but it remains a spawned sub-agent workflow rather than the requested peer-to-peer `sessions_send` mode.

## Best Possible Solution

Keep the issue open for maintainer product/API design. A conservative implementation would add an explicit `sessions_send` option or `session.agentToAgent` policy for isolated peer exchanges, create a deterministic temporary session with no prior transcript, return the result to the requester without writing follow-up turns into either main session, enforce existing A2A allowlists/visibility, and clean up or retain the temporary transcript according to a documented lifecycle. Tests should cover schema acceptance/rejection, existing default behavior, ephemeral routing, cleanup, and Feishu/channel announce behavior.
