## Summary

Keep open. Current main has partial mitigations for this fresh reliability report, but it still preserves the central behavior being reported: Active Memory runs a blocking pre-reply embedded recall turn, still inherits the current/primary model unless configured otherwise, and QMD startup can still arm multiple agents when memory search is explicitly or default-enabled. The newest discussion also adds useful timing evidence and a related PR pattern for keychain caching, but the linked realtime PR is not a replacement for this broader issue.

## Best Possible Solution

Keep this issue open and either implement or split focused reliability work: make Active Memory opportunistic/nonblocking by default with a short deadline, add per-agent/session recall concurrency or dedupe/drop behavior, warn or choose a safer dedicated recall model instead of silently inheriting heavy chat models, and add QMD boot controls such as max concurrent boot updates, jitter, lazy initialization, or active/default-agent-only policy. Status/doctor/docs should also surface expensive Active Memory and broad QMD-on-boot configurations; the keychain/setup timing thread can be tracked as a narrower follow-up if maintainers prefer.

- Multi-agent operators who explicitly enable memory search broadly can still arm multiple QMD managers and boot updates at gateway startup.
- The newest timing data suggests remaining setup/auth overhead inside embedded recall runs that is only partially related to the currently open realtime keychain PR.
