## Summary

Keep this open. Current main has partial mitigations for QMD startup laziness, hard timeout behavior, and repeated Active Memory timeouts, but it still uses a blocking pre-reply Active Memory hook, still inherits the current/session model by default, and the paired draft implementation PR remains open and unmerged.

## What This Changes

Request safer Active Memory and QMD defaults so recall does not overload multi-agent gateways or add long pre-reply latency.

## Best Possible Solution

Keep the issue open until #73667 or a replacement lands a maintainable default profile: Active Memory should fail open with a strict user-visible deadline or nonblocking path, avoid unsafe inherited-model defaults or warn clearly, bound stacked recall work, and keep QMD startup lazy or explicitly concurrency-limited for multi-agent gateways.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: enable Active Memory for an eligible direct session without config.model and the before_prompt_build hook awaits maybeResolveActiveRecall/runEmbeddedPiAgent before the main prompt proceeds; QMD startup behavior is reproducible by opting into memory.qmd.update.startup with multiple memory-enabled agents.

## Solution Assessment

No for current main. The current implementation is only a partial mitigation; the narrow maintainable path is to review and land or supersede #73667 rather than closing this before the blocking/default-model behavior is resolved.
