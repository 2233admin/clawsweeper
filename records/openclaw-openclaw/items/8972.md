## Summary

Keep this issue open. Current main still resolves Discord guild channel allowlist entries as exact id/name/slug keys, thread-parent fallbacks, or the literal `*` fallback. A key such as `ws-*` is stored as an ordinary channel key and is not evaluated as a glob, while the category alternative and additive override semantics remain separate open design items.

## Best Possible Solution

Keep this issue open and implement a documented Discord-owned channel pattern form only after maintainers settle the access-control semantics. The likely implementation should preserve exact id/name/slug and thread-parent precedence, keep literal `*` as the broad fallback, define ordering for multiple pattern matches, update exported types/Zod schema/docs/status/audit behavior, and add regression tests proving matching dynamic Discord channels are accepted without per-channel config edits. Keep category inheritance in #60582 and additive override semantics in #73387 unless maintainers choose to merge those designs.
