## Summary

Keep this PR open. Current main and the latest release still use direct `fs.rename` in memory atomic reindex, so the intended Windows EBUSY fix is not implemented. The proposed direction matches the existing cron-store retry/fallback pattern, but the submitted branch is not mergeable because it breaks template literals, drops WAL/SHM suffix handling, and removes the unique backup UUID.

## What This Changes

The PR modifies `memory-core` atomic reindex file swaps to add a `renameWithRetry` helper with EBUSY retry/backoff and EPERM/EEXIST copy/unlink fallback.

## Best Possible Solution

Revise this PR, or replace it with a clean maintainer patch, by porting the tested cron-store retry/fallback behavior into `memory-core` atomic reindex while preserving SQLite sidecars and backup UUIDs. Add focused regression tests for EBUSY retry, EPERM/EEXIST fallback, and sidecar moves, and keep #64187 open until a corrected implementation lands.
