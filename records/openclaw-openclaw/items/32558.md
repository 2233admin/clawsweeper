## Summary

Keep this PR open. Current main still lacks the Microsoft Teams Graph upload-session fallback this PR is trying to add; OneDrive and SharePoint uploads still use direct `/content` PUTs, and the existing tests only cover the direct path. The PR remains useful but needs maintainer review/rebase because its patch predates current User-Agent requirements and the review thread flags a stall-count off-by-one.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. Rebase it onto current main, preserve OpenClaw User-Agent behavior, fix the stall-count guard, confirm the upload-session threshold, and land a focused implementation in `extensions/msteams/src/graph-upload.ts` with OneDrive and SharePoint large-file regression tests.

## Risks / Open Questions

- The PR branch predates current OpenClaw User-Agent coverage in `graph-upload.ts`; a rebase must preserve `buildUserAgent()` on Graph session creation and direct upload requests.
- The upload-session URL is preauthenticated and should remain unlogged and used only after a successful Graph `createUploadSession` response.
- The stall-count guard in the PR implementation currently allows one more stalled response than `UPLOAD_SESSION_MAX_STALLS` implies.
- The 4 MiB threshold should be confirmed against the observed Teams/Graph failure mode before landing; current Microsoft Graph docs describe resumable uploads for large files and specific byte-range requirements. citeturn1view0
