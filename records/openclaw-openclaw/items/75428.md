## Summary

Close as implemented on current main. The issue targets v2026.4.29, and current main already contains post-release fixes for the same embedded-run prep hot paths: media generation capability discovery is no longer on the reply critical path, targeted provider hook lookups are cached, and packaged runtime-deps activation skips reinstall work when the stage is already materialized.

## What This Changes

The issue reports a v2026.4.29 regression where Weixin and Feishu replies spend about 59 seconds in embedded-run preparation, especially `system-prompt` and `stream-setup`.

## Best Possible Solution

Keep the current main implementation: preserve manifest-first media capability checks, cached targeted provider hook resolution, and runtime-deps materialization short-circuiting, then let the fix ship in the next release after v2026.4.29.

## Reproduction Assessment

Yes for the reported symptom: the issue includes a concrete embedded-run prep trace, and current source maps those labels to the exact runner stages. A live Weixin/Feishu replay is not needed to verify that the current-main hot-path fixes cover the slow prep work shown in the trace.

## Solution Assessment

Yes. The current main fixes are narrower and more maintainable than a channel-specific workaround because they remove broad plugin/provider discovery and repeated runtime-deps work from shared embedded-run preparation.
