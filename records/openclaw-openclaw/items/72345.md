## Summary

Keep open. Current main still routes operational `voicecall` CLI commands through a process-local runtime that starts the webhook listener, while the Gateway already exposes `voicecall.*` RPC methods that the CLI could delegate to; the related #58115 fix covers in-process runtime reuse, not this separate CLI/Gateway process collision.

## What This Changes

The issue asks for `openclaw voicecall call` and related CLI actions to avoid starting a second voice-call webhook runtime when the Gateway already owns the voice-call webhook port.

## Best Possible Solution

Make the voice-call plugin CLI Gateway-aware: delegate operational call/control/status commands through the existing `voicecall.*` Gateway RPC methods when a Gateway is available or explicitly selected, preserve a deliberate standalone fallback, and ensure diagnostic/read-only commands do not start or retain long-lived voice runtime resources.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: Gateway service startup calls `ensureRuntime()`, a separate CLI `voicecall call` process also calls `ensureRuntime()`, runtime creation starts the webhook server, and the default listener is `127.0.0.1:3334`.

## Solution Assessment

No, current main is not the best solution because it keeps CLI operational commands tied to local runtime creation. The narrow maintainable fix is plugin-local Gateway RPC delegation using the existing SDK helper, not a broad core runtime registry.
