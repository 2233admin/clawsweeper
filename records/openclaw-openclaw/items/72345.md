## Summary

Keep open. Current main still routes `openclaw voicecall call` and the other operational voicecall CLI commands through a process-local runtime, and that runtime still starts its own webhook listener on the configured bind/port before placing or managing calls. The Gateway already exposes `voicecall.*` RPC methods and the plugin SDK exposes a CLI-to-Gateway helper, so the remaining work is a narrow CLI delegation fix rather than a cleanup close.

## What This Changes

The issue asks for `openclaw voicecall call` to avoid starting a second local voice-call webhook runtime when the Gateway already owns the webhook port, preferably by delegating to the running Gateway or by producing an actionable diagnostic.

## Best Possible Solution

Add a Gateway-aware path for voice-call CLI operational commands that calls the existing `voicecall.*` Gateway RPC methods through `openclaw/plugin-sdk/gateway-runtime` when the user selects Gateway/remote operation, while keeping local runtime startup for explicit standalone use and documenting the behavior.

## Reproduction Assessment

Yes. The source-level path is high-confidence: a Gateway process can bind the default voice webhook, then a separate CLI process reaches `ensureRuntime()` -> `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(3334, 127.0.0.1)`. No live Twilio call is needed to verify the port-collision control flow.

## Solution Assessment

Yes. Delegating through the existing Gateway RPC methods is the narrowest maintainable direction because the Gateway already owns the live runtime and the plugin SDK already exposes a CLI Gateway helper; a cross-process runtime registry would be broader than needed.
