## Summary

Keep open. Current main still routes `openclaw voicecall call` and the other operational voice-call CLI commands through the plugin's process-local runtime, while Gateway startup also creates that runtime and binds the configured webhook listener. The previously shipped #58115 fix covers in-process runtime reuse, but this issue describes a separate CLI process colliding with the Gateway-owned listener, and the existing `voicecall.*` Gateway RPC methods make this a focused repair candidate.

## What This Changes

The issue asks for the voice-call CLI to avoid `EADDRINUSE` when the Gateway daemon already owns the voice-call webhook port, preferably by delegating calls to the running Gateway or producing a clearer diagnostic.

## Best Possible Solution

Add a Gateway-aware path for the operational `voicecall` CLI commands that delegates to the existing `voicecall.*` Gateway RPC methods when a Gateway target is selected, while preserving local runtime startup only for explicit standalone use. The fallback path should also translate webhook bind failures into a clear message that points users at Gateway delegation rather than surfacing raw `EADDRINUSE`.

## Reproduction Assessment

Yes. The source-level path is high-confidence: a Gateway process starts the voice-call webhook, and a separate CLI process has no shared `globalThis` runtime, so `voicecall call` reaches `ensureRuntime()` -> `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(3334, 127.0.0.1)`.

## Solution Assessment

Yes. Delegating through the already-registered `voicecall.*` Gateway RPC methods is the narrowest maintainable fix; adding a cross-process singleton registry or moving voice-call runtime ownership into core would be broader than needed.
