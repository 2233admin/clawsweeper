## Summary

Keep open. Current main still routes `openclaw voicecall call` and the other operational voice-call CLI commands through a process-local voice-call runtime that starts its own webhook listener, while the Gateway already exposes matching `voicecall.*` RPC methods. The prior #58115 fix addressed in-process runtime reuse, but this separate CLI process collision remains a valid, narrow plugin bug.

## What This Changes

The issue reports that `openclaw voicecall call` starts a local voice-call runtime and collides with a running Gateway-owned voice-call webhook listener on `127.0.0.1:3334`.

## Best Possible Solution

Add a Gateway-aware path in the voice-call plugin CLI for operational commands, using the existing `voicecall.*` Gateway RPC methods when selected or when a running Gateway can be safely detected, while preserving explicit standalone local runtime behavior and emitting an actionable diagnostic for Gateway-owned webhook port collisions.

## Reproduction Assessment

Yes. The source-level path is high-confidence: Gateway service startup calls `ensureRuntime()` and binds the webhook, while a separate CLI process reaches `ensureRuntime()` -> `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(3334, 127.0.0.1)`.

## Solution Assessment

Yes. A plugin-local CLI delegation through the existing plugin-facing Gateway RPC helper is the narrowest maintainable repair; a core cross-process runtime registry or startup-time migration would be broader than this bug requires.

## Repair Work Prompt

Fix the voice-call CLI/Gateway webhook port collision reported in https://github.com/openclaw/openclaw/issues/72345. Keep the repair plugin-local unless a genuinely missing public SDK seam is discovered. Current `extensions/voice-call/src/cli.ts` calls `ensureRuntime()` for `call`, `start`, `continue`, `speak`, `dtmf`, `end`, and `status`; that creates `createVoiceCallRuntime()`, which starts `VoiceCallWebhookServer` on the configured serve address. When the Gateway already owns `127.0.0.1:3334`, the CLI fails with EADDRINUSE. Use the existing Gateway RPC surface registered in `extensions/voice-call/index.ts` (`voicecall.initiate`, `voicecall.start`, `voicecall.continue`, `voicecall.speak`, `voicecall.dtmf`, `voicecall.end`, `voicecall.status`) and the plugin-facing helper exported from `openclaw/plugin-sdk/gateway-runtime` where appropriate. Preserve standalone local-runtime behavior for explicit local use, but add a Gateway-aware/delegating path or explicit flag for operational commands and an actionable EADDRINUSE diagnostic that points users at the Gateway-backed command path. Add focused tests in `extensions/voice-call/index.test.ts` or a new colocated CLI test proving Gateway delegation does not call `createVoiceCallRuntime()` and that the local fallback error is clearer. Update `docs/cli/voicecall.md`, `docs/plugins/voice-call.md`, `extensions/voice-call/README.md`, and `CHANGELOG.md` if the user-facing CLI behavior changes. Do not move voice-call ownership into core and do not change unrelated webhook/security behavior.
