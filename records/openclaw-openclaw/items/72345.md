## Summary

Keep open. Current main still routes `openclaw voicecall call` and related CLI actions through a process-local voice-call runtime, and runtime creation still starts a webhook listener on the configured bind/port before placing calls. A Gateway-owned runtime and existing `voicecall.*` RPC methods are present, so the report remains valid and narrow enough for a focused repair.

## What This Changes

Make the voice-call CLI work when a running Gateway already owns the voice-call webhook listener, either by delegating to the Gateway or by producing a clearer conflict diagnostic.

## Best Possible Solution

Add a Gateway-aware path in the voice-call plugin CLI that delegates operational commands through the existing `voicecall.*` Gateway RPC methods when the user selects Gateway/remote operation, while keeping local runtime startup for explicit standalone use and adding an actionable `EADDRINUSE` diagnostic for the fallback path.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: one Gateway process starts the voice-call service and binds the webhook, while a separate CLI process reaches `ensureRuntime()` -> `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(3334, 127.0.0.1)`.

## Solution Assessment

Yes. Delegating through the existing Gateway RPC methods is the narrowest maintainable fix; adding a cross-process runtime registry or moving voice-call ownership into core would be broader than needed.

## Repair Work Prompt

Fix the voice-call CLI/Gateway webhook port collision reported in https://github.com/openclaw/openclaw/issues/72345. Current `openclaw voicecall call` and related operational commands start a local `VoiceCallRuntime`, which tries to bind the webhook port even when the Gateway already owns it. Implement a Gateway-aware CLI path in the voice-call plugin, preferably using the public `openclaw/plugin-sdk/gateway-runtime` helper and the existing `voicecall.*` Gateway RPC methods. Keep standalone local-runtime behavior available for explicit local use, avoid adding bundled plugin IDs or voice-call policy to core, and add a clearer `EADDRINUSE` fallback diagnostic if local runtime startup still collides. Cover at least `call`/`start`, and include the management commands that operate on existing calls (`continue`, `speak`, `dtmf`, `end`, `status`) if the chosen flag/mode applies to them. Update `extensions/voice-call/index.test.ts` for the delegated path and local fallback, update `docs/cli/voicecall.md`, `docs/plugins/voice-call.md`, and `extensions/voice-call/README.md` if command behavior or flags change, and add a changelog fix entry with reporter credit. Related context: #58115 fixed only in-process runtime reuse; #47697 and #53126 describe the same remaining CLI/Gateway collision. Do not move voice-call runtime ownership into core or introduce a broad cross-process runtime registry.
