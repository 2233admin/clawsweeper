## Summary

Keep open. Current main still has the reported voice-call CLI/gateway port-conflict path: runtime-owning `voicecall` CLI actions create a process-local voice-call runtime, and runtime creation starts a webhook listener on the configured/default `127.0.0.1:3334`. The related #58115 work fixed in-process runtime reuse, but it does not solve a separate CLI process colliding with a Gateway-owned webhook port.

## Best Possible Solution

Implement a gateway-aware voice-call CLI path. Runtime-owning commands should delegate to the existing `voicecall.*` Gateway RPC methods when the user selects a running Gateway target or has remote Gateway mode configured, while preserving the current local-runtime behavior for standalone use. Add regression coverage proving delegated CLI commands do not call `createVoiceCallRuntime()` and do not start a webhook listener, plus a clearer local-runtime `EADDRINUSE` error if startup still collides.

## Clownfish Work Prompt

Fix the voice-call CLI/Gateway webhook port conflict tracked by https://github.com/openclaw/openclaw/issues/72345. Current `openclaw voicecall` runtime-owning actions call `ensureRuntime()` in `extensions/voice-call/src/cli.ts`; when the Gateway already owns the voice-call webhook listener on `127.0.0.1:3334`, a separate CLI process can hit `EADDRINUSE`. Implement a gateway-aware CLI path for `smoke --yes`, `call`, `start`, `continue`, `speak`, `dtmf`, `end`, and `status`: when a Gateway target is explicitly selected via existing gateway client options or configured remote Gateway mode, call the existing `voicecall.*` Gateway RPC methods instead of creating the local runtime. Preserve current standalone local-runtime behavior when no Gateway target is selected/configured. Use `openclaw/plugin-sdk/gateway-runtime` through the voice-call local barrel if extension code needs `callGatewayFromCli`/`addGatewayClientOptions`; do not import core `src/**` internals from extension production code. Add focused regression tests showing delegated CLI commands call the expected `voicecall.*` RPC methods and do not call `createVoiceCallRuntime()`/`ensureRuntime()`. Add a clearer local-runtime `EADDRINUSE` error if startup collides despite local mode. Update `docs/cli/voicecall.md` or `docs/plugins/voice-call.md` if user-facing flags/options change, and add a single-line changelog fix entry thanking @serrurco; mention related #47697, #53126, and #58115 where useful. Do not change provider auth defaults, webhook security policy, or unrelated voice-call runtime behavior.
