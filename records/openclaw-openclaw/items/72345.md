## Summary

Keep open. Current main still routes operational `voicecall` CLI commands through a process-local voice-call runtime that starts the webhook server, so a separate CLI process can still collide with a Gateway-owned `127.0.0.1:3334` listener. The shipped shared-runtime work for #58115 covers in-process plugin contexts, but it does not add Gateway RPC delegation for this CLI path.

## What This Changes

The issue requests that `openclaw voicecall call` and related CLI commands avoid starting a parallel voice-call webhook runtime when the Gateway already owns the webhook port.

## Best Possible Solution

Add a plugin-local Gateway-aware CLI path for operational voice-call commands, keep read-only setup and dry-run smoke commands from creating or retaining long-lived runtime handles, and retain local runtime startup only for explicit standalone/live fallback with an actionable port-collision diagnostic.

## Reproduction Assessment

Yes for the original bug: with the Gateway voice-call runtime already bound to the default webhook address, `openclaw voicecall call` reaches `ensureRuntime()` and `createVoiceCallRuntime()`, which calls `webhookServer.start()` and `listen(3334, 127.0.0.1)`. The newer setup hang has credible reporter output but still needs a focused current-main CLI-exit test.

## Solution Assessment

Yes. Delegating CLI operations through the existing `voicecall.*` Gateway RPC methods is the narrowest maintainable fix; a core runtime registry or bundled-plugin special case would be broader than needed.
