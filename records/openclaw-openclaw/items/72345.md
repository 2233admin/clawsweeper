## Summary

Keep open. Current main still wires `openclaw voicecall call` through the voice-call plugin's process-local runtime, and runtime creation still starts a webhook listener on the configured bind/port, so a separate CLI process can still collide with a Gateway-owned listener. The existing `voicecall.*` Gateway RPC methods and public gateway-runtime SDK helpers make this a focused repair candidate.

## What This Changes

The issue requests a gateway-aware `voicecall` CLI path, or at least a clearer diagnostic, so `openclaw voicecall call` does not fail with `EADDRINUSE` when the Gateway already owns the voice-call webhook port.

## Best Possible Solution

Add a gateway-delegating path for voice-call CLI operational commands using the existing `voicecall.*` Gateway RPC surface, while preserving explicit standalone local runtime startup. The fallback local-runtime path should turn `EADDRINUSE` on the configured voice-call bind/port into an actionable message that points users to the Gateway-backed mode.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a Gateway process starts the voice-call webhook, while a separate CLI process has no shared `globalThis` runtime and reaches `createVoiceCallRuntime()` -> `webhookServer.start()` -> `listen(3334, 127.0.0.1)`.

## Solution Assessment

Yes. Delegating CLI operations through the already-registered `voicecall.*` Gateway RPC methods is the narrowest maintainable fix; a new cross-process runtime registry or moving voice-call ownership into core would be broader than needed.

## Repair Work Prompt

Fix the voice-call CLI/Gateway webhook port collision tracked by https://github.com/openclaw/openclaw/issues/72345, and cover the same remaining CLI conflict described by #47697 and #53126. Implement this in the voice-call plugin boundary: make `openclaw voicecall` operational commands able to delegate to a running Gateway through the existing `voicecall.*` RPC methods, using the public `openclaw/plugin-sdk/gateway-runtime` helpers such as `callGatewayFromCli`/`addGatewayClientOptions` where appropriate. Preserve the current local runtime behavior for explicit standalone use, but when that fallback hits `EADDRINUSE` on the configured voice-call bind/port, report an actionable Gateway-owned-webhook message instead of the raw listen error. Do not move voice-call-specific behavior into core and do not add bundled plugin ids to core. Add focused tests that prove the Gateway-delegating CLI path does not call `ensureRuntime()` and maps call/start/continue/speak/dtmf/end/status to the existing RPC method names, plus a regression test for the clearer local-runtime `EADDRINUSE` diagnostic. Update `docs/cli/voicecall.md` and `docs/plugins/voice-call.md` if a flag or documented workflow changes. Changelog entry is expected because this is a user-facing CLI bug fix; credit the reporter or relevant contributor, not Codex.
