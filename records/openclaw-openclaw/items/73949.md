## Summary

Keep open. Current main still lacks the abort-time live `reconnect-exhausted` suppression that this PR adds, and the patch is narrow to the Discord plugin lifecycle/test surface with no blocking functional or security findings found in the diff review.

## What This Changes

The PR adds a Discord provider lifecycle guard that treats abort-time live `reconnect-exhausted` events as clean shutdown, adds a focused regression test, and records the user-facing changelog credit.

## Best Possible Solution

Land this PR, or an equivalent narrow Discord plugin fix, with the behavior kept in `extensions/discord/src/monitor/provider.lifecycle.ts`, focused coverage in `provider.lifecycle.test.ts`, and the existing changelog credit preserved; no core health-monitor change is needed for this bug.

## Reproduction Assessment

Yes. Against current main, a live `reconnect-exhausted` event delivered after `abortSignal.aborted` but before lifecycle cleanup reaches `handleGatewayEvent`, returns `stop` and is converted by `waitForDiscordGatewayStop` into a `DiscordGatewayLifecycleError`.

## Solution Assessment

Yes. The PR fixes the narrow point where the live event is classified into the stop/reject path, uses the existing abort-signal contract, and leaves disallowed-intents, fatal errors, and non-shutdown reconnect exhaustion behavior intact.
