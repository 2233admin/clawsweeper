## Summary

Keep open. Current main still relies on process-local numeric skills snapshot versions and does not compare a restart-stable config/root fingerprint, so a persisted version-0 snapshot can be reused after an offline `skills.entries.<name>.enabled=false` change; the related draft PR #71497 is an unmerged fix candidate, not shipped behavior.

## What This Changes

The issue requests durable invalidation of persisted skills snapshots so disabling a skill in `skills.entries` is reflected in existing long-lived sessions after restart or offline config edits.

## Best Possible Solution

Add restart-stable skills snapshot validity based on the effective inputs that make a snapshot correct, preferably a deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots/entries. The end state should be that existing sessions rebuild their snapshot on the next turn after a skill is disabled, including after gateway restarts and edits made while the gateway was stopped.

## Reproduction Assessment

Yes. Persist an existing session with `skillsSnapshot.version: 0`, add `skills.entries.<name>.enabled=false` while the gateway is stopped, restart, and send another message in the same session; current main computes version 0 again and the source predicate reuses the stale snapshot.

## Solution Assessment

Yes for the issue's durable inputs-hash direction; it directly models snapshot validity across restarts. The narrower startup-version seeding direction in #71497 may cover the restart/offline edit symptom, but it should explicitly test disabled-skill snapshots and may be less precise than comparing effective snapshot inputs.
