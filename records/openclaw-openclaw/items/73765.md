## Summary

Keep open. Current main still has the stale persisted skills snapshot gap: disabled-skill filtering works when a snapshot is rebuilt, but existing version-0 session snapshots can be reused after restart/offline config edits because refresh only checks the process-local snapshot version and the agent skill filter.

## What This Changes

Existing long-lived sessions should stop advertising skills after `skills.entries.<name>.enabled=false`, including when the config edit happens while the gateway is stopped or before a restart.

## Best Possible Solution

Refresh persisted skills snapshots from restart-stable inputs. A deterministic fingerprint over effective skills config, agent skill filter, and resolved skill roots/entries would directly prevent stale prompt reuse; if maintainers prefer the narrower startup seed in #71497, it should be rebased, linked to this issue, and covered with the disabled-skill regression before merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist a session with `skillsSnapshot.version: 0`, add `skills.entries.<name>.enabled=false` while the gateway is stopped, restart, and send another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes, restart-stable invalidation is the right direction; the current main implementation is not sufficient because it depends on an in-memory version bump. A snapshot-input hash is the most maintainable durable fix, with startup version seeding as a possible narrower mitigation.
