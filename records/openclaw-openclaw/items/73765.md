## Summary

Keep open. Current main still reuses persisted skills snapshots using only a process-local numeric version and the agent skill filter, so a disabled skill can remain in an existing session after an offline config edit or gateway restart.

## What This Changes

Refresh existing session skill snapshots when a later `skills.entries.<skill>.enabled: false` config change should remove a skill from the active prompt and resolved skill set.

## Best Possible Solution

Persist and compare a deterministic skills snapshot input fingerprint before reusing a session snapshot. The fingerprint should cover the effective skills config, agent skill filter, and resolved skill roots/entries closely enough to catch disabled skills, stale paths, and installed-skill changes without depending on watcher timing.

## Reproduction Assessment

Yes. A source-level reproduction is to persist a session with `skillsSnapshot.version: 0`, edit `openclaw.json` to disable a previously snapshotted skill while the gateway is stopped, restart, then send another turn on the same session; current main computes version 0 again and reuses the old snapshot.

## Solution Assessment

Yes. A restart-stable fingerprint is the narrowest maintainable direction because it models the actual snapshot inputs; a startup-only version bump would be broader, rebuild more than necessary, and still would not distinguish which inputs changed.
