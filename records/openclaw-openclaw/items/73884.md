## Summary

Keep this PR open. Current main still allows and uses a 30000 ms Telegram polling stall threshold while the grammY runner long-poll window is 30 seconds, so the PR's intended 60000 ms safety floor is not implemented on main; the patch is narrow and I found no concrete correctness or security issue in the diff.

## What This Changes

The PR raises Telegram pollingStallThresholdMs minimums from 30000 ms to 60000 ms across runtime clamping, Zod validation, generated config metadata, docs, UI hints, types, and affected tests.

## Best Possible Solution

Rebase this focused PR on current main, keep the 60000 ms lower bound aligned across runtime clamping, config validation, generated metadata, docs, UI hints, and tests, then land it if maintainers accept the stricter Telegram config contract.

## Reproduction Assessment

Yes. The source path is enough to reproduce the issue class: configure Telegram pollingStallThresholdMs as 30000 on current main, let a getUpdates long poll run slightly past grammY's 30 second window, and the liveness detector can force a stall restart because the threshold has no buffer.

## Solution Assessment

Yes, with a maintainer compatibility decision. Raising the floor to 60000 ms in both runtime and validation is the narrowest maintainable fix for unsafe low thresholds; changing grammY's long-poll timeout or weakening the watchdog would have broader behavior impact.
