## Summary

Keep this PR open. Current main still does not implement the intended behavior for an enabled alternative Feishu/Lark plugin unless that plugin declares a preferOver relationship, and the PR also still needs adaptation to the current split auto-enable files plus the allowlist review concern.

## Best Possible Solution

Keep this PR open, but require it to be rebased and adapted to the current split files, likely src/config/plugin-auto-enable.shared.ts plus focused tests. The best fix should skip bundled channel auto-enable only when another channel owner is actually effective/trusted under the same activation rules, including plugins.enabled, allow, deny, and explicit disable. Add regression coverage for the openclaw-lark Feishu case and a restrictive plugins.allow case, or make the replacement relationship explicit through manifest channelConfigs.feishu.preferOver and test that path.
