## Summary

Keep this PR open. Current main still uses the global `agents.defaults.model` as the no-session-override fallback for Telegram model-list callbacks, so an account bound to an agent with its own configured model can still show the wrong check mark. The PR remains a narrow implementation candidate with a matching regression test and no security-sensitive drift in the provided diff.

## Best Possible Solution

Review and land this focused PR after normal validation. The fix should stay in the Telegram plugin runtime: compute the bound agent default once with `resolveDefaultModelForAgent({ cfg: runtimeCfg, agentId: route.agentId })`, reuse it for stored override provider resolution, and use it for the no-session-override model-list current selection. Keep the new regression test for a Telegram-bound agent whose explicit model differs from `agents.defaults.model`.
