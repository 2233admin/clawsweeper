## Summary

Keep this PR open for maintainer review. Current main still lacks the reverse lookup needed for the reported malformed hex-suffixed tool names, but the proposed patch broadens dispatch normalization in a way that can coerce an unmarked unknown tool name into a different allowlisted tool, so it should not merge as-is.

## What This Changes

The PR adds an allowlist-backed reverse lookup in the embedded runner tool-call normalizer for underscore-stripped, dot-prefixed, dropped-leading-character, and hex-suffixed malformed tool-name variants.

## Best Possible Solution

Land a narrow embedded-runner normalization fix that recovers only allowlisted malformed tool names when there is strong structural evidence, preserves existing delimiter and alias behavior, and fails closed on exact, prefix, underscore-stripped, dropped-leading-character, and cross-plugin collisions with colocated regression tests for both recovery and non-recovery cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require a structural signal before tail-only recovery:** `src/agents/pi-embedded-runner/run/attempt.tool-call-normalization.ts:148-151`
  - body: The new exact fallback accepts `allowedName.substring(1)` with no prefix, suffix, id, or hash signal. With only `msearch` allowlisted, a raw unknown name like `search` would now be rewritten to `msearch` and dispatched, whereas current main leaves it unknown for the loop guard. Please remove tail-only exact recovery or require separate evidence that the provider actually dropped the first character.
