## Summary

Keep this issue open. Current main still treats normal chat `/model` selections as session-persistent state, including mixed-content messages, and there is no `/ask` or `--once` chat syntax that runs one response on a different text model while preserving the session default for later user turns, queued followups, heartbeats, or cron work.

## Best Possible Solution

Keep the issue open and implement a core chat/run feature, not a ClawHub/plugin workaround. Add a first-class per-turn model override such as `/model <provider/model> --once <message>` or `/ask <provider/model> <message>` that reuses the existing model resolver, allowlist, auth-profile, and runtime validation, but carries the selected provider/model only on the current user response. Add focused tests proving the session entry is not mutated, queued followups are not retargeted, later user turns use the previous session/default model, and heartbeat/cron behavior remains isolated. Update the model and slash-command docs at the same time.
