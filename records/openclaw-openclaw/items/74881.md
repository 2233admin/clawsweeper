## Summary

Keep this PR open. The linked Windows Gmail setup bug is valid on current main, but this branch is not merge-ready because it misses the documented CLI setup/run path, leaves one resolved `.cmd` path on direct `spawn`, adds a shadowable `where.exe` lookup, and changes a deprecated public SDK barrel without the required baseline follow-through.

## What This Changes

The PR adds a Windows-only executable resolver for `gcloud`, `tailscale`, and gateway-watcher `gog` calls, adds resolver tests, and exports `resolveSystemBin` from the deprecated plugin SDK `infra-runtime` barrel.

## Best Possible Solution

Keep the fix focused on the Windows Gmail helper-command blocker: resolve PATHEXT shims for all documented setup, run, and gateway watcher paths; launch `.cmd`/`.bat` results through an existing Windows-safe spawn seam; avoid executing a shadowable lookup helper; and remove the unrelated SDK export unless it is made an intentional SDK change with generated baseline and docs updates.

## Reproduction Assessment

Yes. The linked bug has a concrete native PowerShell reproduction, and current main still shows bare `gcloud`, `tailscale`, and `gog` invocations on the documented Gmail setup/run paths; the PR-specific gaps are statically reproducible from the latest provided diff.

## Solution Assessment

No. Resolving helper commands is the right direction, but this implementation is incomplete and broadens security/API surface unnecessarily; the safer fix is a shared non-executing resolver plus Windows-safe spawn wiring across both Gmail code paths.
