## Summary

Keep this PR open. The linked Windows Gmail setup bug is real and current main still has bare helper-command paths, but the latest PR does not fully cover the documented CLI setup/run path, still directly spawns a possible `.cmd` watcher process, and carries an unrelated public SDK export that appears to create API-baseline drift.

## What This Changes

The PR adds a Windows-only helper-command resolver, applies it to `gcloud`/`tailscale` in Gmail setup utilities and `gog` in the gateway watcher, adds resolver tests, and exports `resolveSystemBin` from the deprecated plugin SDK `infra-runtime` barrel.

## Best Possible Solution

Keep the branch focused on the Windows Gmail helper-command fix: resolve PATHEXT shims for all documented setup/run/watcher paths, route long-running `.cmd`/`.bat` launches through an existing Windows-safe spawn seam, avoid executing a PATH-resolved `where.exe`, and remove or separately justify the SDK API export with the required baseline/docs updates.

## Reproduction Assessment

Yes. The linked bug has concrete native PowerShell reproduction, current main still shows bare `gcloud`, `gog`, and `tailscale` helper paths, and the PR-specific gaps are statically reproducible from the latest diff plus the documented CLI flow.

## Solution Assessment

No. The resolver direction is reasonable, but this implementation is not the best fix because it misses `src/hooks/gmail-ops.ts`, leaves a direct `.cmd` spawn in the watcher, adds an avoidable `where.exe` execution step, and changes the public SDK surface outside the Windows Gmail fix.
