## Summary

Keep open. The linked Windows Gmail setup bug is still valid on current main, and the PR moves in the right direction, but the branch is not mergeable as-is because it misses the documented CLI gog path, can still direct-spawn a resolved .cmd shim, and introduces a command-resolution trust concern around bare where.exe.

## What This Changes

The PR adds a Windows-only helper resolver for gcloud/tailscale/gog PATHEXT shims, wires it into Gmail setup utilities and the gateway Gmail watcher, adds resolver tests, and exports resolveSystemBin from the deprecated infra-runtime SDK barrel.

## Best Possible Solution

Keep the Windows Gmail fix narrow: use a non-executing PATH/PATHEXT resolver or extend the existing executable-path helper, apply it to gcloud, tailscale, and both Gmail CLI and gateway gog paths, route any .cmd/.bat launch through the existing Windows-safe spawn seam, and leave SDK barrel exports out unless they are treated as an intentional public SDK change.

## Reproduction Assessment

Yes. #54470 provides concrete native Windows setup steps and current main still has bare gcloud, tailscale, and gog calls in the affected flow; the PR-specific gaps are statically reproducible from the latest diff and call graph.

## Solution Assessment

No. The resolver direction is reasonable, but this implementation is not the best fix until it covers the documented CLI gog path, avoids direct .cmd spawn, avoids executing PATH-resolved where.exe, and removes or properly promotes the SDK export.
