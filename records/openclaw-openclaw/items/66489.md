## Summary

Keep open. Current main still throws the reported local-media allowlist error, but the delivery-queue permanent-error classifier does not match that string, so recovery paths continue through retry/backoff before `failed/`. The broader local-media staging work around #69183 and the relative `MEDIA:` normalizer reduce some ways to hit this, but they do not implement the requested delivery-queue safety net.

## What This Changes

Classify `Local media path is not under an allowed directory` as a permanent delivery error so poisoned queued deliveries move to `failed/` without exhausting retries.

## Best Possible Solution

Add a precise permanent classification for the local-media allowlist rejection in the delivery-queue policy, preferably using structured `LocalMediaAccessError` code `path-not-allowed` where available and the exact persisted message as the recovery fallback. Cover both the pure classifier and a queued recovery entry moving straight to `failed/`; leave enqueue-time validation, `lastErrorClass`, and Telegram overlong-message handling as separately verified follow-ups unless a PR proves them together.
