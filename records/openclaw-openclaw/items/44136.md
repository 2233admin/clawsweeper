## Summary

Keep open. Current main still does not implement this PR’s central priority rule: GitHub Copilot runtime auth returns a token-derived baseUrl and the runner applies it without checking whether the model already has an explicit custom baseUrl. The PR’s intent is still useful, but the patch targets an older runner shape and should be ported to the current plugin/auth-controller paths.

## Best Possible Solution

Keep this PR open as a useful bug-fix candidate, but port the behavior to the current GitHub Copilot plugin/runtime auth architecture. The best fix should preserve explicit custom Copilot baseUrls, still apply token-derived Business/Enterprise endpoints when no explicit custom value exists, update runtime and effective models, handle refresh/profile rotation source changes, preserve lazy runtime imports, and add regression coverage for custom proxy, default-to-token endpoint, context-window-capped effectiveModel, refresh, and profile rotation.

## Risks / Open Questions

- As submitted, the PR is not directly mergeable against current main because it edits the older direct runner token-flow shape instead of the current `extensions/github-copilot` and `src/agents/pi-embedded-runner/run/auth-controller.ts` flow.
- Security/supply-chain review: the PR does not add dependencies, workflows, lockfile changes, secret handling, or package-publishing changes, but its top-level Copilot token import would weaken the lazy runtime boundary and should not be carried forward in a port.
- A naive `model.baseUrl !== default` guard can still mis-handle auth profile rotation if a previous token-derived endpoint is mistaken for an explicit custom endpoint; the port should track configured/default/token-derived source carefully.
