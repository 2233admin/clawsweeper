## Summary

Keep open. Current main does not implement `messages.tts.autoEmotion`, and the PR is an active contributor feature branch, but the latest PR head still has a concrete ElevenLabs correctness bug and an explicit maintainer product-boundary question before merge.

## What This Changes

This PR adds opt-in `messages.tts.autoEmotion`, maps inferred emotions into provider-specific TTS controls across speech-core and provider adapters, and updates config schema/help/labels, generated schema, docs, and focused tests.

## Best Possible Solution

Revise the ElevenLabs explicit-setting detection so only `voiceSettings.style` or `voiceSettings.stability` suppress auto-emotion, add a regression test for a similarityBoost-only config, then have the requested maintainers decide whether this provider-aware mapping belongs in shared speech-core before merge.

## Reproduction Assessment

Yes. Configure ElevenLabs with `messages.tts.autoEmotion.enabled: true` and only `voiceSettings.similarityBoost`, then synthesize happy text; the PR predicate returns explicit=true before the auto `style`/`stability` override can be built.

## Solution Assessment

No, not as-is. The opt-in mapping approach may be acceptable if maintainers approve the boundary, but the current ElevenLabs guard is broader than both the PR's provider mapping and the documented opt-out behavior.

## Review Findings

Overall correctness: patch is incorrect
