## Summary

Keep open: this active contributor PR still has a concrete ElevenLabs behavior mismatch that can block the new opt-in feature for valid configs, and the author has requested focused maintainer review on the product boundary.

## What This Changes

The PR adds opt-in `messages.tts.autoEmotion` inference in speech-core, maps inferred emotions into provider-specific TTS controls, and updates provider adapters, config schema/help/labels, generated schema, docs, and focused tests.

## Best Possible Solution

Keep the PR open and revise the ElevenLabs explicit-setting detection so only `voiceSettings.style` or `voiceSettings.stability` suppress auto-emotion, then add a regression test for similarityBoost-only ElevenLabs config and let the named maintainers decide whether shared speech-core is the right home for this provider-aware mapping.

## Reproduction Assessment

Yes. Configure ElevenLabs with `messages.tts.autoEmotion.enabled: true` and only `voiceSettings.similarityBoost` in the raw provider, persona, or request override, then synthesize happy text; the PR predicate returns explicit=true before building the auto `style`/`stability` override.

## Solution Assessment

No, not as-is. The opt-in shared mapping approach may be maintainable if maintainers approve the product boundary, but the ElevenLabs guard is broader than the provider fields the PR actually owns and broader than the docs describe.
