## Summary

Keep open. Current main still has the redundant SQLite preflight in the builtin memory manager, and this PR is a focused implementation for the linked open performance issue; it is not obsolete. The diff looks functionally sound from code inspection, but the branch is currently not mergeable and still lacks direct cache hit/miss/invalidation coverage.

## What This Changes

The PR adds an in-memory nullable cache for `hasIndexedContent()` in memory-core and updates chunk write/delete, reset, and reindex paths to populate or invalidate it.

## Best Possible Solution

Rebase or refresh the PR against current main, keep the cache local to the builtin `MemoryIndexManager` with invalidation on every chunk mutation and DB-handle swap path, and add focused coverage for cache hit/miss and invalidation behavior before landing.

## Reproduction Assessment

Yes. Current main reproduces the code-level performance issue because repeated builtin `search()` calls still enter `hasIndexedContent()` and execute SQLite probes when no cache field exists.

## Solution Assessment

Yes. A per-manager tri-state cache with explicit invalidation on writes/deletes/resets is the narrowest maintainable fix for this hot preflight path; the safer landing path is to rebase it and add targeted tests.
