## Summary

Keep this PR open. Current main still leaves the parent `/home/node/.cache` directory outside the ownership fix in the optional Docker browser-install path, and the proposed one-line Dockerfile change is the narrow, correct fix. Related PR #43846 was closed unmerged as the duplicate child, while open PR #61167 does Docker Compose/script env plumbing and does not replace this Dockerfile ownership change.

## What This Changes

This PR changes the optional Docker browser-install block to chown `/home/node/.cache` instead of only `/home/node/.cache/ms-playwright`.

## Best Possible Solution

Land the focused parent-cache ownership fix, preferably with a small assertion in `src/dockerfile.test.ts` that the browser-install block chowns `/home/node/.cache`, then validate the browser-enabled Docker build path if feasible.

## Reproduction Assessment

Yes. The current Dockerfile runs `mkdir -p /home/node/.cache/ms-playwright` as root before `USER node`; Docker will create the parent `.cache` directory as root while the existing chown only covers the child directory.

## Solution Assessment

Yes. Widening ownership to `/home/node/.cache` is the narrowest maintainable Dockerfile fix for the reported runtime write failure; the safer addition is regression coverage, not a broader Docker setup rewrite.
