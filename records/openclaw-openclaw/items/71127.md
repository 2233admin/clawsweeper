## Summary

Keep open. Current main has added a conservative stuck-session recovery helper, but active embedded/reply runs remain warning-only by default, and the requested configurable abort/recovery contract is still not present. The open PR #73243 specifically closes this issue, so the right path is to review or land that PR rather than close this issue before merge.

## What This Changes

The issue requests diagnostics-driven recovery for sessions stuck in `processing`, including aborting the in-flight run, releasing the queue, and emitting a recovery signal without requiring an external gateway restart.

## Best Possible Solution

Land a reviewed diagnostics recovery implementation that adds a separate configurable abort threshold, wires the heartbeat to abort and drain active reply/embedded runs only after that threshold, emits structured recovery telemetry, and keeps the existing stale-lane cleanup safeguards and docs/schema/tests aligned.
