## Summary

Keep open. Current main has conservative stale-lane recovery, but active stuck runs are still warning-only and there is no configurable abort threshold; the remaining work is already represented by open closing PR #73243 for maintainer review.

## What This Changes

The issue requests runtime self-recovery for sessions stuck in `processing`, including a separate diagnostics abort threshold, active-run cancellation, queue release, and structured recovery diagnostics.

## Best Possible Solution

Review and either land or revise #73243 so diagnostics keeps the existing warning threshold, adds a separate conservative abort threshold, uses established reply/embedded-run cleanup primitives, emits structured recovery diagnostics, updates config/docs, and preserves healthy long-running work until the abort threshold is crossed.

## Reproduction Assessment

Yes. The report has production logs, the discussion adds a deterministic v2026.4.25 Telegram restart-window reproduction, and current-main source confirms active stuck work is still not aborted by diagnostics.

## Solution Assessment

Yes in direction, but not as current main stands. A separate abort threshold reviewed through #73243 is narrower and safer than overloading `stuckSessionWarnMs` or forcing immediate active-run cancellation at the warning point.
