## Summary

Keep open. Current main has diagnostic warning plus conservative stale-lane recovery, but active embedded runs, active reply work, and active lane tasks remain observe-only by default, and the open closing PR has not landed.

## What This Changes

The issue asks for diagnostics to abort and release sessions stuck in `processing` after a configurable threshold instead of only logging repeated stuck-session warnings.

## Best Possible Solution

Land or revise a narrow core diagnostics fix that keeps the warning threshold separate from a conservative abort threshold, uses existing run abort/drain primitives, emits structured recovery diagnostics, and updates config docs plus focused tests.

## Reproduction Assessment

Yes. The report has production logs, RayWoo added a deterministic v2026.4.25 Telegram restart-window reproduction, and current-main source inspection confirms active stuck work is still warning-only by default.

## Solution Assessment

Yes in direction, but not yet implemented on main. A separate `stuckSessionAbortMs` recovery path is safer than overloading `stuckSessionWarnMs` or aborting immediately at the warning threshold.
