## Summary

Keep open. Current main still has warning-triggered conservative stale-lane recovery, but active embedded runs, active reply work, and active lane tasks are intentionally observe-only without a public abort threshold; the remaining fix is already represented by open closing PR #73243.

## What This Changes

The issue requests configurable diagnostics-driven recovery that aborts stuck `processing` sessions, releases their session lane, and emits structured abort diagnostics without requiring an external gateway restart.

## Best Possible Solution

Review, revise, or land #73243 so diagnostics keeps the warning threshold, adds a separate conservative abort threshold, uses existing reply/embedded-run cleanup primitives, emits structured recovery diagnostics, updates config/docs, and preserves healthy long-running work until the abort threshold is crossed.

## Reproduction Assessment

Yes. The issue has production log evidence, a later deterministic v2026.4.25 Telegram restart-window reproduction, and current-main source confirms active stuck work is still not aborted by diagnostics.

## Solution Assessment

Yes in direction, but current main is not the complete solution. A separate abort threshold reviewed through #73243 is narrower and safer than overloading `stuckSessionWarnMs` or cancelling active work at the warning point.
