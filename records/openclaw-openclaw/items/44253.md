## Summary

Keep open. Current main still has no `selfDeny` or delegate-only tool policy key, and the active tool pipeline applies agent/global deny rules before the subagent policy layer, so a subagent allowlist cannot restore tools removed from the parent or target agent. The request is concrete, but it changes a security-sensitive tool-permission invariant and should get maintainer design review rather than cleanup closure.

## What This Changes

Request adding per-agent `tools.selfDeny` or an equivalent delegate-only policy so an orchestrator can pass tools to subagents while hiding or blocking those tools from its own model turn.

## Best Possible Solution

Define a small, documented policy primitive that removes listed tools only from the current agent's model-visible tool set after normal allow/deny resolution, while preserving the normal resolved tool pool for child sessions and keeping `deny` absolute. The implementation should update typed config, generated schema/help, runtime filtering, docs, and focused policy tests without letting subagents escalate tools denied by global, provider, group, sandbox, or explicit subagent policy.

## Reproduction Assessment

Not applicable as a runtime crash, but the current limitation has a high-confidence static reproduction path: configure an agent deny for `exec` and a subagent allow for `exec`, then follow the current pipeline where agent deny runs before the final subagent filter. The docs and policy code both confirm the subagent allowlist cannot add back the removed tool.

## Solution Assessment

Unclear pending maintainer design review. The proposed `selfDeny` direction is plausible and narrow, but because it changes tool-permission inheritance semantics, the safest solution is an explicit self-only final visibility filter rather than any subagent escalation or allowlist override of earlier denies.
