## Summary

Keep open. Current main still does not perform the requested boot-time sweep for fresh session lock files owned by the current process. Gateway startup calls `cleanStaleLockFiles()`, that helper removes only locks already classified as stale by PID/age parsing, and the self-owned orphan logic is only applied during later acquisition contention. The supplied GitHub context also shows open PR #49660 with closing syntax for this issue, so the issue should remain open until that PR is reviewed and landed or closed.

## Best Possible Solution

Keep this issue open and review PR #49660. The best fix is a narrow startup/cleanup change that makes `cleanStaleLockFiles()` or the startup sidecar classify self-owned lock files with no matching in-memory `HELD_LOCKS` entry as `orphan-self-pid`, log/report that reason, preserve genuinely active in-process locks, and add regression coverage for both direct cleanup and gateway startup behavior.
