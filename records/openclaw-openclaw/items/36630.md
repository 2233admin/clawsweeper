## Summary

Keep open. Current main still lacks native Signal quote send parameters and structured inbound quote IDs, so the PR is useful source work, but the current head is not mergeable: it still has plugin-boundary violations, failing null/blank reply-threading semantics, formatted-send bypasses, quote-id validation risk, and media/queue edge-case regressions.

## What This Changes

The PR adds Signal inbound quote-context parsing, outbound Signal quote metadata plumbing, shared outbound `quoteAuthor`/`replyToId:null` handling, and regression tests across Signal and shared reply delivery paths.

## Best Possible Solution

Land a smaller Signal-owned implementation that keeps production plugin imports behind existing SDK surfaces, updates shared reply-threading semantics before widening public payload types, routes formatted/direct/queued Signal sends through the same quote-author rules, validates quote IDs as decimal timestamps, and proves direct plus group quote behavior with focused tests and a live Signal smoke.

## Reproduction Assessment

Yes. Static reproduction is high confidence from the current PR head: the added null/blank reply-threading tests disagree with unchanged implementation, production Signal helpers deep-import core internals, formatted Signal sends ignore quote metadata, and inbound quote IDs accept `Number(...)` coercions.

## Solution Assessment

No. The feature direction is useful, but this branch is not the best merge vehicle until the shared threading, SDK boundary, formatted-send, queue recovery, media fallback, and quote-id validation issues are corrected.

Full review comments:

- **[P1] Update threading before allowing null reply IDs:** `src/auto-reply/reply-payload.ts:28`
  - body: This widens `replyToId` to include `null`, but `resolveReplyThreadingForPayload` still uses a truthy check. The new tests asserting null suppression, blank normalization, and trimming will fail, and production auto-threading can still overwrite an explicit do-not-reply payload.
  - confidence: 0.96
- **[P1] Move quote-context imports behind SDK surfaces:** `extensions/signal/src/monitor/quote-context.ts:1-2`
  - body: This production Signal helper imports `kindFromMime` and `normalizeE164` via relative `../../../../src/**` paths, which violates the bundled plugin boundary and can break standalone plugin packaging. Use the existing `openclaw/plugin-sdk/media-runtime` and `openclaw/plugin-sdk/text-runtime` exports instead.
  - confidence: 0.97
- **[P1] Move reply-delivery types behind the SDK:** `extensions/signal/src/monitor/reply-delivery.ts:1`
  - body: This production Signal helper imports `ReplyPayload` from a core `src/**` file. Signal plugin code should use the public reply SDK surface so bundled and third-party plugin builds do not depend on core layout internals.
  - confidence: 0.96
- **[P1] Forward quote metadata through formatted Signal sends:** `src/infra/outbound/deliver.ts:1133`
  - body: Adding `quoteAuthor` to the shared send overrides does not make default Signal text replies quoted, because the formatted Signal adapter ignores `replyToId` and `quoteAuthor` when it calls `sendMessageSignal`. Generic Signal text sends with reply metadata still post as top-level messages.
  - confidence: 0.9
- **[P2] Treat blank Signal reply IDs as unset:** `extensions/signal/src/monitor/reply-delivery.ts:12-13`
  - body: `normalizeReplyToId` maps whitespace-only strings to `null`, which this module treats as explicit reply suppression. Blank payload IDs should behave like `undefined` so inherited quote context can still apply; reserve `null` for an intentional do-not-reply override.
  - confidence: 0.88
- **[P2] Persist quote authors in queued deliveries:** `src/infra/outbound/deliver.ts:379`
  - body: `quoteAuthor` is now part of outbound delivery params, but the write-ahead queue still serializes only `replyToId` and `replyToMode`. A recovered Signal group reply will replay without the author, causing quote metadata to be dropped after a crash/retry.
  - confidence: 0.86
- **[P2] Require decimal-only inbound quote IDs:** `extensions/signal/src/monitor/quote-context.ts:67-68`
  - body: `Number(trimmed)` accepts inputs like `1e3` and `0x10`, so malformed Signal quote payloads can normalize to a different message id and author-cache key. Inbound quote IDs should use the same decimal-only timestamp validation as outbound quote sends.
  - confidence: 0.91
- **[P2] Fall back to mediaUrl when mediaUrls is empty:** `extensions/signal/src/monitor.ts:359-360`
  - body: Using `mediaUrls ?? ...` treats an explicit empty array as final, so `{ mediaUrls: [], mediaUrl: "..." }` drops the single-media fallback that the previous shared reply-payload helper preserved.
  - confidence: 0.89
