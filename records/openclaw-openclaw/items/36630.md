## Summary

Keep this PR open. Current main at d22a851253b0225e72847dc6b1a7b74d8035e214 still does not implement the PR's central bidirectional Signal quote-reply behavior: Signal sends have no native quote options or quote RPC params, Signal outbound paths do not pass reply metadata into formatted/direct sends, monitor replies are sent without quote metadata, and inbound quote context still lacks a structured ReplyToId. The PR remains a real implementation candidate, but its current diff still needs maintainer cleanup for conflict markers, extension-boundary violations, and quote-attribution risks before merge.

## Best Possible Solution

Keep this PR open and rework it as a smaller Signal-owned quote-reply change: add plugin-safe quote timestamp/author plumbing through a SDK seam or local Signal facade, route formatted/direct/monitor text and media sends through quote-aware delivery, surface inbound Signal quote ids as ReplyToId with quote body/sender, remove conflict markers and core deep imports, and add focused regression coverage for group quote-author lookup, formatted sends, monitor replies, media/chunk/sendPayload consumption, debounce, and queued recovery.
