## Summary

Keep open. Current main still lacks the requested complete Signal quote-reply behavior, but this PR head is not merge-ready because it contains unresolved conflict markers, production plugin-boundary violations, and several concrete reply/quote metadata regressions that affect shared outbound and Signal inbound behavior.

## What This Changes

The PR wires Signal quote-reply metadata through outbound send/delivery and inbound Signal quote context, widens reply payload typing across channel surfaces, and adds Signal/outbound regression tests.

## Best Possible Solution

Keep the useful Signal-owned quote-reply work, but make the branch mergeable by resolving conflict markers, moving production helper imports behind SDK or Signal-local seams, hardening quote id and quote-author consumption, preserving existing media/debounce/thread semantics, and validating the touched Signal and outbound surfaces.

Overall confidence: 0.91

Full review comments:

- **[P1] Remove conflict markers from the Signal test:** `extensions/signal/src/monitor/event-handler.mention-gating.test.ts:5-9`
  - body: The test header still contains merge-conflict markers and a duplicate import. TypeScript cannot parse this file, so the branch cannot pass the Signal test/typecheck lanes until the conflict is resolved.
  - confidence: 0.99
- **[P1] Move quote-context imports behind plugin SDK surfaces:** `extensions/signal/src/monitor/quote-context.ts:1-2`
  - body: This new production module imports core internals through ../../../../src paths, which violates the bundled plugin boundary and can break standalone plugin builds. Use plugin-sdk exports or Signal-local helpers instead.
  - confidence: 0.94
- **[P1] Move reply-delivery type imports behind a public surface:** `extensions/signal/src/monitor/reply-delivery.ts:1`
  - body: This production helper imports ReplyPayload from a core src path instead of the plugin SDK reply-runtime surface. That couples the Signal plugin to host internals and crosses the extension boundary.
  - confidence: 0.92
- **[P2] Clear quoteAuthor when reply consumption clears replyToId:** `src/infra/outbound/message-plan.ts:10`
  - body: Adding quoteAuthor to shared send overrides means chunk planning can clear an implicit replyToId while leaving the old author attached to later chunks. Clear quoteAuthor whenever the matching reply target is consumed.
  - confidence: 0.8
- **[P2] Require decimal-only Signal quote ids:** `extensions/signal/src/monitor/quote-context.ts:67-68`
  - body: normalizeSignalQuoteId uses Number(trimmed), which accepts non-decimal strings such as 1e3 and 0x10. Signal quote timestamps should be decimal-only so malformed payloads cannot alias to a different message id.
  - confidence: 0.84
- **[P2] Preserve mediaUrl when mediaUrls is empty:** `extensions/signal/src/monitor.ts:344-345`
  - body: The replacement media list uses nullish coalescing, so payloads with mediaUrls: [] and a fallback mediaUrl now resolve to an empty list and drop the attachment. Preserve the prior fallback behavior when mediaUrls is present but empty.
  - confidence: 0.86
- **[P2] Do not treat blank Signal reply ids as suppression:** `extensions/signal/src/monitor/reply-delivery.ts:12-13`
  - body: normalizeReplyToId maps whitespace-only strings to null, which resolveSignalReplyDelivery treats as an explicit do-not-reply override. Blank strings should be unset so inherited reply context can still apply.
  - confidence: 0.82
- **[P2] Consume inherited Signal reply state only when inherited was sent:** `extensions/signal/src/monitor.ts:361-365`
  - body: markSignalReplyConsumed is called with effectiveReplyTo, including explicit per-payload targets. An explicit first payload can consume the inherited reply state even though that inherited target was never used, causing later payloads to lose their expected inherited quote.
  - confidence: 0.78
- **[P2] Do not drop debounced quote-only batches:** `extensions/signal/src/monitor/event-handler.ts:482-487`
  - body: When a debounce flush contains multiple quote-only entries, combinedText is empty and the handler returns before dispatching or recording the quote context. Quote-only support should preserve those batches instead of dropping them.
  - confidence: 0.86
- **[P2] Choose quote context from the latest debounced message:** `extensions/signal/src/monitor/event-handler.ts:491-501`
  - body: The debounce merge selects the latest entry that has quoteTarget, so a quoted message followed by a plain message can still surface stale ReplyTo* context. Use the latest message's quote state so plain follow-ups clear the quote.
  - confidence: 0.8
