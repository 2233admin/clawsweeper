## Summary

Keep open. Current main already uses a safer generated Linux systemd unit default, but the requested doctor/status hygiene visibility is still missing: runtime inspection does not collect KillMode, TasksCurrent, or MemoryCurrent, and the human/JSON status surfaces only expose the existing runtime summary plus extra-service scan results. The issue discussion also adds useful direction toward doctor warnings and a careful KillMode tradeoff, so this is still actionable rather than stale cleanup.

## What This Changes

Add Linux systemd gateway hygiene diagnostics to doctor/status and evaluate safer KillMode behavior for generated or repaired gateway units.

## Best Possible Solution

Add a Linux-only systemd hygiene layer that reads KillMode, TasksCurrent, MemoryCurrent, and MainPID through the existing service runtime path, surfaces those values in gateway status/OpenClaw status and doctor, and emits a focused warning with systemctl/systemd-cgls/restart guidance when an existing unit still uses KillMode=process with suspicious cgroup growth. Keep the current control-group default unless maintainers explicitly choose a mixed-mode policy change.

## Repair Work Prompt

Implement Linux systemd gateway hygiene diagnostics for this issue. Current main already renders `KillMode=control-group`; do not change that default or migrate existing units silently. Add a typed runtime/audit surface that reads `KillMode`, `TasksCurrent`, `MemoryCurrent`, and `MainPID` from `systemctl --user show <unit>` for the managed gateway unit, parses numeric values safely, carries them through the existing service runtime/status structures, and surfaces them in `openclaw gateway status --deep`/JSON and `openclaw status --deep` where the gateway service overview is shown. Add a doctor warning path, preferably reusing the service runtime/audit flow, that warns when an existing Linux systemd gateway unit has `KillMode=process` plus suspicious task or memory growth while the gateway is otherwise reachable. Include concise remediation hints: `systemd-cgls --user-unit openclaw-gateway.service`, `systemctl --user show openclaw-gateway.service -p KillMode -p TasksCurrent -p MemoryCurrent -p MainPID`, and `systemctl --user restart openclaw-gateway.service`. Keep thresholds conservative and documented in code/tests; avoid noisy per-PID classification unless needed. Add or update CLI docs for the new status/doctor output and add a single-line changelog entry with thanks to @openclawq and @Hollychou924 because this is user-facing CLI diagnostics. Do not touch unrelated daemon restart behavior, provider/channel code, or bundled plugin surfaces.
