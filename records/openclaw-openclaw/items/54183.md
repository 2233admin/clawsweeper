## Summary

Keep this PR open. Current main has shipped durable outbound queue/reconnect-drain behavior for the persistent WhatsApp message-loss reports, but it still does not expose a `channels.whatsapp.retry` config surface, and maintainer discussion already treats configurable immediate retry as adjacent rather than identical. The branch remains potentially useful, but it needs maintainer product/architecture review and a rebase against current queue semantics before merge.

## What This Changes

Adds a WhatsApp `retry` config surface and shared exponential-backoff wrapper for text/media/poll sends, with WhatsApp schema/generated docs/tests and shared retry-policy helper updates.

## Best Possible Solution

Decide whether OpenClaw still wants a user-facing immediate WhatsApp send retry knob after the shipped durable queue work. If yes, rebase onto the current plugin-local outbound path, use focused SDK retry/error subpaths, align the retry classifier with queue/reconnect-drain behavior, and document exactly which transient failures retry immediately versus defer to the queue.
