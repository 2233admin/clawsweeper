## Summary

Keep open. Current main still has the plugin-loader behavior described in the report: bundled runtime dependency aliases are applied to plugin Jiti loads, and the alias resolver still prefers conditional `import` targets over `require`, which can map `ws` to its ESM wrapper instead of the CommonJS constructor. The existing ClawSweeper comment is still accurate, and #74312 remains useful background rather than a duplicate.

## What This Changes

Preserve Node-style CommonJS `require("ws")` constructor semantics for custom plugins loaded through the OpenClaw plugin loader.

## Best Possible Solution

Make bundled runtime dependency alias resolution preserve CommonJS semantics for plugin `require()` by selecting the `require` export for sync Jiti aliases when available, falling back to `import` only when no require-compatible target exists, and add a `ws`-shaped conditional-export regression test.

## Repair Work Prompt

Fix the plugin loader compatibility bug from https://github.com/openclaw/openclaw/issues/74547. Current main builds bundled runtime dependency Jiti aliases from conditional exports using `import`, `node`, `default` order, so plugin `require("ws")` can be redirected to the `ws` ESM wrapper instead of the CommonJS constructor. Preserve CommonJS `require()` semantics for plugin loads by making the bundled runtime dependency alias resolver choose the `require` condition when building sync Jiti aliases, falling back to the existing import/default behavior only when no require-compatible target exists. Add a regression test using a tiny `ws`-shaped package with `exports['.'].require = './index.js'` and `exports['.'].import = './wrapper.mjs'`; verify the root alias resolves to the CommonJS target while existing root/subpath/wildcard/scoped alias behavior still passes. Keep the change in the generic plugin loader/alias layer, do not add a `ws`-specific special case or plugin-owned workaround, and do not change unrelated plugin SDK resolution. Add a user-facing changelog fix entry crediting `Thanks @aderius` if the repo's active changelog section requires one for plugin compatibility fixes.
