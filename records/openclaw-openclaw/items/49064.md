## Summary

Keep this PR open. Current main has strong underlying usage infrastructure, a nested `openclaw gateway usage-cost` command, chat `/usage cost`, Control UI usage views, and Gateway RPCs for `usage.cost` / `sessions.usage`, but it does not implement the PR’s intended top-level `openclaw usage` command with date ranges, session ranking, by-model summaries, and JSON output. The PR is still an implementation candidate, but it needs to be rebased/refit to the current usage surfaces and should address the existing review blockers before maintainer consideration.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it onto current main and implement the top-level CLI as a thin wrapper over the existing `usage.cost` / `sessions.usage` infrastructure, with production-module tests for Commander options, date conflicts, local/UTC date semantics, complete aggregation before display limiting, shared `formatTokenCount`, and JSON output. If maintainers decide the nested `gateway usage-cost` surface is enough, that should be a product decision or a separate superseding thread, not an automatic cleanup close.

## Risks / Open Questions

- As submitted, the PR can report incorrect custom date ranges outside UTC and silently let `--days` override more explicit range flags.
- The proposed session/model breakdown appears to cap the scanned sessions before computing totals, which can make top-spend and by-model results wrong for large histories.
- The PR’s test approach, per the provided diff, reimplements helper logic instead of testing the production module, so it may not catch regressions in the actual CLI implementation.
- There is product overlap with current `gateway usage-cost`, `/usage cost`, Control UI usage, and `sessions.usage`; landing should avoid creating a divergent second aggregation path.
