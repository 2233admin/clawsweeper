## Summary

Keep open. Current main has useful partial mitigations, but it still leaves same-provider shared-profile billing fallback chains blocked until near expiry and does not clear stale usage state on configure-driven plugin re-auth.

## What This Changes

The issue asks OpenClaw to recover promptly from persisted provider billing/cooldown state after billing top-up or successful provider re-auth, without requiring manual edits to auth-state.json.

## Best Possible Solution

Treat same-provider shared-profile billing fallback chains as having no effective alternate billing fallback for recovery probes, and clear provider/profile-scoped stale usage state after successful configure/onboarding auth refresh while preserving unrelated profile order and cooldown history.

## Reproduction Assessment

Yes. The issue provides step-by-step Anthropic and GitHub Copilot reproductions, and current source/tests corroborate the remaining paths: billing-disabled primaries with fallbacks are skipped far from expiry, and configure-based plugin auth lacks the direct-login stale-lockout cleanup.

## Solution Assessment

Yes, with a narrower version of the proposed direction. Targeted billing probes plus provider-scoped cooldown clearing after successful auth refresh are more maintainable than removing all persisted cooldown state, because they preserve backoff protection while fixing recovery.

## Repair Work Prompt

Fix the persistent provider cooldown recovery bug tracked by https://github.com/openclaw/openclaw/issues/70903. Current main already keeps active disabledUntil/cooldownUntil windows immutable, probes billing-disabled primaries when there are no fallback candidates, and clears stale lockouts for direct `models auth login`; do not undo those protections. Add the missing recovery behavior: (1) when all fallback candidates for a billing-disabled provider effectively share the same provider/profile and there is no useful cross-provider fallback, allow a throttled probe instead of waiting until near expiry; (2) after successful configure/onboarding/plugin provider auth refresh, clear stale provider/profile usage state for the affected provider/profile, matching the direct login cleanup but without wiping unrelated providers or profile ordering. Coordinate with open PR #74909 and do not duplicate its failure-policy-only change unless it has not landed and the edit is necessary for this issue.
