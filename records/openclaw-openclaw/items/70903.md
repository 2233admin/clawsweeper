## Summary

Keep open. Current main has partial mitigation, but it still leaves the reported recovery problem active when a billing-disabled primary has configured fallbacks and when configure/plugin auth refresh writes fresh credentials without clearing stale persisted usage state.

## What This Changes

The issue requests prompt recovery from persisted provider cooldown or disabled auth-state entries after billing top-up or successful provider re-auth, without manual JSON edits and gateway restarts.

## Best Possible Solution

Keep persisted cooldowns for unresolved provider-health failures, but add targeted recovery: detect same-provider/shared-profile billing chains as having no useful billing fallback, probe or clear them promptly after recovery, and clear provider/profile-scoped stale usage state after successful configure/plugin auth refresh while preserving unrelated provider state.

## Reproduction Assessment

Yes. The issue and follow-up comment give concrete Anthropic billing and GitHub Copilot re-auth scenarios, and current tests/source still encode the remaining billing skip and missing plugin-auth cleanup paths.

## Solution Assessment

Yes. The narrowest maintainable solution is targeted recovery and auth-refresh cleanup, not removing persisted cooldowns wholesale, because the existing disabled/cooldown state still protects unresolved provider failures and fallback health.
