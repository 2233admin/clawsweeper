## Summary

Keep open. Current main contains adjacent recovery work, but the reported user problem is still present for same-provider-only billing fallback chains and for successful `openclaw configure --section model` re-auth flows that only upsert credentials without clearing stale provider usage state.

## What This Changes

The issue asks OpenClaw to recover from persisted provider auth cooldown/disabled state after billing recovery or successful re-auth without requiring manual `auth-state.json` edits and gateway restarts.

## Best Possible Solution

Finish shared provider auth recovery in core: treat same-provider fallback chains that all depend on the same disabled profile as having no effective billing fallback for probe decisions, and make successful provider re-auth through configure/plugin auth-choice flows clear stale cooldown/disabled usage state for the affected profile(s). Add focused regressions for Anthropic same-profile billing recovery and GitHub Copilot configure re-auth clearing a `format` cooldown, while preserving cross-provider fallback preference and probe throttles.

## Repair Work Prompt

Fix openclaw/openclaw#70903 on a narrow branch. Current main only partially recovers provider auth cooldowns: billing probes run immediately only when there are no fallback candidates, so same-provider-only fallback chains sharing a disabled profile still remain blocked until near expiry; successful provider re-auth through `openclaw configure --section model` upserts credentials without clearing stale `usageStats` cooldown/disabled state.

Expected implementation boundary:
- In `src/agents/model-fallback.ts`, adjust the billing cooldown probe decision so fallback candidates are considered effective only when they can actually escape the disabled billing profile. A same-provider chain whose candidates resolve to the same provider/profile set should be treated like no effective fallback for billing recovery probing. Preserve existing cross-provider fallback preference and throttle behavior; do not probe every same-provider candidate in one run.
- In the configure/plugin auth-choice path (`src/commands/configure.gateway-auth.ts`, `src/commands/auth-choice.apply.ts`, `src/plugins/provider-auth-choice.ts`, or a small shared helper), clear stale cooldown/disabled usage state for the affected provider/profile(s) after successful provider auth/upsert. Reuse `clearAuthProfileCooldown` or a shared helper rather than duplicating JSON surgery. Preserve unrelated profile ordering, `lastGood`, unrelated providers, and unrelated cooldown history.
- Keep core generic: no Anthropic- or Copilot-specific hard-coding beyond tests/fixtures. No startup/load-time migration that wipes auth state globally.
- Add a changelog fix entry with Thanks @mattglover11 and @rocke2020. Update docs only if the user-visible recovery command/semantics change.

Regression coverage to add:
- Anthropic same-provider Opus fallback chain where primary and fallback share `anthropic:default` with `disabledReason: "billing"` far from expiry should allow one throttled recovery probe rather than skipping all models.
- A cross-provider fallback chain should still prefer the healthy cross-provider fallback instead of repeatedly probing billing-disabled primary far from expiry.
- `openclaw configure --section model` / plugin auth-choice successful re-auth should clear stale `cooldownUntil`/`cooldownReason` or `disabledUntil`/`disabledReason` for the affected profile, including a `format` cooldown case, without clearing unrelated profiles.
