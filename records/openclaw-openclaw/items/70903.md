## Summary

Keep open. Current main contains partial recovery work, but it still leaves the core user problem: billing-disabled shared-profile fallback chains can be skipped until the persisted disabled window nears expiry, and configure/plugin auth refresh does not clear stale usage state the way direct models auth login now does. The open related PR #74909 addresses adjacent non-auth failure poisoning, not the billing recovery and configure cleanup paths tracked here.

## What This Changes

The issue asks OpenClaw to recover promptly from persisted provider/profile cooldowns after billing restoration or successful re-auth instead of blocking users until auth-state timestamps expire or are manually edited.

## Best Possible Solution

Keep persisted cooldowns for real unresolved provider health failures, but treat same-provider/shared-profile billing fallbacks as having no real alternate billing route and clear provider/profile-scoped stale usage state after successful configure/onboarding auth refresh while preserving unrelated provider state.

## Reproduction Assessment

Yes. The issue provides concrete Anthropic and GitHub Copilot reproductions, and current source plus existing tests show the remaining behavior: far-from-expiry billing-disabled primaries with configured fallbacks are skipped, and configure/plugin auth refresh lacks direct-login stale-lockout cleanup.

## Solution Assessment

Yes, the best direction is targeted recovery rather than deleting persisted cooldowns wholesale. Same-provider billing probing plus provider-scoped cleanup after successful auth refresh fixes the lockout while retaining backoff protection and normal cross-provider fallback preference.
