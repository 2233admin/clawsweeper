## Summary

Keep open. Current main has partial mitigations, but the reported recovery failure is still present for shared-profile same-provider billing fallback chains and for configure-based provider re-auth paths that leave stale persisted usage state behind.

## What This Changes

The issue asks OpenClaw to recover automatically from persisted provider cooldown/disable state after billing recovery or successful provider re-authentication.

## Best Possible Solution

Keep the persisted cooldown mechanism, but add targeted recovery: treat same-provider shared-profile fallback chains as having no effective billing fallback for throttled recovery probes, and clear provider/profile-scoped stale usage state after successful configure/onboarding auth refresh while preserving unrelated profile history.

## Reproduction Assessment

Yes. The issue provides concrete Anthropic and GitHub Copilot reproductions, and current source/tests corroborate both remaining paths: billing-disabled same-provider fallbacks are skipped, and configure/plugin auth refresh does not clear stale persisted usage state.

## Solution Assessment

Yes, but only as a targeted fix. Probing recoverable billing disables and clearing stale state after successful re-auth is narrower than removing all persisted cooldowns, which would weaken retry-storm protection.
