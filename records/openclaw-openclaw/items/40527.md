## Summary

Keep open. Current main still queues the remote skill bin probe during node-role connect before the gateway sends `hello-ok`, while the Swift app starts normal event dispatch only after connect succeeds. The issue discussion also narrows the remaining failure to connect-time probe timing because `system.which` now works manually on app-backed macOS nodes.

## What This Changes

The issue asks OpenClaw to stop remote skill binary probes from timing out when a macOS app-backed node connects from the same host as the gateway, while preserving remote macOS skill support.

## Best Possible Solution

Make the gateway/app-node handshake robust so remote bin probes cannot be delivered before app-backed nodes are ready to process `node.invoke.request`; preserve remote macOS skills and treat loopback/co-located suppression as a secondary policy only if the handshake fix does not remove the timeout spam.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/40527. Current main queues `refreshRemoteNodeBins` during node-role connect before the gateway sends `hello-ok`; the Swift `GatewayChannelActor` waits for the connect response and starts normal event dispatch only after `sendConnect` returns, so an automatic remote bin probe can send `node.invoke.request` before app-backed nodes process events. Implement a robust handshake/probe ordering fix so initial remote macOS bin probes cannot time out solely because they are sent during connect. Preserve remote macOS skill support: do not remove `system.run`/`system.which` from the macOS app, do not blanket-disable remote skills, and do not rely on user `denyCommands` as the fix. Prefer moving/defering the initial gateway probe until after `hello-ok` is sent and the node is ready, or explicitly dispatch/queue pre-response Swift events safely; add loopback/co-located suppression only if it is still needed after the handshake fix. Add focused tests for the gateway ordering and Swift node session early-event behavior, plus any MacNodeModeCoordinator coverage if command advertisement changes. Update the active changelog with a single-line user-facing fix entry crediting `Thanks @Adam-Researchh` and `Thanks @tavaresgmg` if the code change lands.
