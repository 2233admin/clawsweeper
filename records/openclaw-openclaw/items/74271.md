## Summary

Keep open. Current main still lacks the requested state-dir `.env` shadowing warning, and an open unmerged PR explicitly references this issue with closing syntax, so this issue should remain open until that PR is reviewed and landed or replaced.

## What This Changes

Add redacted doctor/status/gateway-status diagnostics when `~/.openclaw/.env` contains `OPENCLAW_GATEWAY_TOKEN` that can override a configured local `gateway.auth.token`.

## Best Possible Solution

Keep this issue open as the canonical tracker until the linked PR or a replacement lands a shared, redacted diagnostic that reads the trusted state-dir `.env`, detects local mode plus configured `gateway.auth.token` plus non-empty `OPENCLAW_GATEWAY_TOKEN`, and surfaces consistent remediation in doctor/status/gateway-status with docs and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: configure local `gateway.auth.token` while `OPENCLAW_GATEWAY_TOKEN` is present in the CLI environment, then credential resolution chooses env-first for call/probe while the gateway service/auth path can remain config-first; current tests already encode that split.

## Solution Assessment

Yes as a product direction. A warning is the narrowest maintainable fix because it preserves intentional env-based auth and avoids printing or deleting secrets; the implementation should be state-dir-scoped and shared across diagnostic surfaces rather than an arbitrary process-env warning only.
