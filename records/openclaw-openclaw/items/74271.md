## Summary

Current main still does not implement the requested guardrail. OpenClaw loads the trusted state-dir `.env` and still has credential paths where a non-empty `OPENCLAW_GATEWAY_TOKEN` can affect local call/probe behavior while config-backed auth/status paths remain separate; existing diagnostics cover macOS `launchctl` overrides, service-token drift, and SecretRef availability, but not the specific `~/.openclaw/.env` token plus `gateway.auth.token` local-mode conflict.

## Best Possible Solution

Keep this open and add a shared read-only diagnostic helper that detects local gateway mode with configured `gateway.auth.token` plus non-empty `OPENCLAW_GATEWAY_TOKEN` in the trusted state-dir `.env`, redacts all secrets, and surfaces consistent guidance in `openclaw doctor`, `openclaw status`/`status --all`, and `openclaw gateway status`. The warning should explain that the env token can override config credentials, point to `~/.openclaw/.env`, suggest removing or syncing the env token, and suggest restarting the gateway afterward.
