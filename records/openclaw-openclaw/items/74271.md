## Summary

Keep open. Current main still has the reported credential-precedence split: CLI/probe paths can consume `OPENCLAW_GATEWAY_TOKEN` loaded from the state-dir `.env`, while gateway auth resolves token config first, and I did not find the requested redacted state-dir `.env` warning in doctor/status/gateway status. The issue also has open closing PR #74433, so repository policy keeps this issue open until that PR lands or is replaced.

## What This Changes

Warn users when local gateway mode has `gateway.auth.token` configured but `~/.openclaw/.env` contains a non-empty `OPENCLAW_GATEWAY_TOKEN` that can shadow the config token.

## Best Possible Solution

Keep this issue as the canonical report and resolve it by landing or replacing PR #74433 with a shared, redacted diagnostic that reports the state-dir `.env` source and remediation consistently across doctor, status, and gateway status without printing token values.

## Reproduction Assessment

Yes. A high-confidence reproduction path is local gateway mode with `gateway.auth.token` in `~/.openclaw/openclaw.json` and a different non-empty `OPENCLAW_GATEWAY_TOKEN` in `~/.openclaw/.env`; current dotenv loading and credential precedence explain the CLI/probe versus gateway-service mismatch.

## Solution Assessment

Yes, the requested warning is the narrow maintainable direction: preserve env override behavior for intentional users, but add a redacted diagnostic and restart guidance. The safest implementation should be a shared helper rather than separate one-off warnings in each command.
