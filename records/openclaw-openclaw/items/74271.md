## Summary

Keep open. Current main still has the local gateway env/config credential split and the state-dir dotenv reader, but doctor/status/gateway-status do not yet emit the requested redacted warning. The open closing PR #74433 is the implementation candidate, so this issue should remain open until that PR lands or is replaced.

## What This Changes

Request a redacted diagnostic in doctor/status/gateway status/install flows when a non-empty state-dir OPENCLAW_GATEWAY_TOKEN shadows a configured local gateway.auth.token.

## Best Possible Solution

Keep this as the canonical report and resolve it through PR #74433 or a replacement that adds one shared, redacted diagnostic: read the state-dir .env directly, compare a non-empty OPENCLAW_GATEWAY_TOKEN with configured local gateway.auth.token, explain env precedence, and suggest removing or syncing the env key plus restarting the gateway without printing secrets.

## Reproduction Assessment

Yes. A high-confidence reproduction path is local gateway mode with gateway.auth.token set in ~/.openclaw/openclaw.json and a different non-empty OPENCLAW_GATEWAY_TOKEN in ~/.openclaw/.env; current main's dotenv loading and precedence tests explain the resulting auth disagreement.

## Solution Assessment

Yes for the requested direction: a warning is safer than deleting or rewriting a credential that may be intentional. Current main does not yet implement it, and the maintainable fix should be a shared redacted helper used by doctor/status surfaces rather than separate ad hoc checks.
