## Summary

Keep this PR open. The underlying security gap still appears present on current main, but the branch is stale against the current MCP handler split and carries a supply-chain-sensitive axios release-age policy change that should not be bundled with the MCP authorization fix.

## What This Changes

The PR splits gateway dangerous-tool constants, filters plugin-tools MCP list/call with a universal deny list plus gateway.tools overrides, adds MCP security-boundary tests, and adds axios to pnpm minimumReleaseAgeExclude.

## Best Possible Solution

Rebase the MCP hardening onto current main by applying a universal dangerous-tool deny set in the active MCP handler/server flow for both listTools and callTool, preserving ownerOnly and before_tool_call behavior, keeping HTTP-only entries separate, documenting gateway.tools behavior for the MCP bridge, and leaving dependency release-age policy out of this PR.
