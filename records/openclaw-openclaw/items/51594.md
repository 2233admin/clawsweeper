## Summary

Keep open. Current main and the latest release still have the strict literal minute check, so the PR addresses a real cron stagger bug, but the patch is not merge-ready because its comma-list handling misses composite Croner-valid expressions and it adds unsupported numeric-prefix step behavior. The diff is limited to cron helper code and tests, with no concrete security or supply-chain concern.

## What This Changes

The PR adds a minute-field zero detector and tests so top-of-hour stagger detection recognizes step, list, range, range-step, and wildcard minute fields for 5-field and 6-field cron expressions.

## Best Possible Solution

Tighten the helper to match Croner default parser semantics: recurse through comma-separated minute entries, support Croner-valid wildcard, list, range, and range-step forms, remove `0/N` behavior unless proven accepted under OpenClaw's Croner options, and add regression coverage through helper, create, normalize, and doctor default-stagger paths before landing and closing #51587.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step coverage:** `src/cron/stagger.ts:23`
  - body: OpenClaw uses Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless sloppy range parsing is enabled. This helper and its new test should not lock in `0/15` as supported OpenClaw cron syntax; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
