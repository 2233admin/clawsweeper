## Summary

Keep open. The underlying cron stagger bug is still present on current main, and this PR is useful, but the patch is not merge-ready because the helper misses Croner-valid mixed comma entries and adds support for numeric-prefix stepping that OpenClaw's default Croner configuration rejects.

## What This Changes

This PR adds a minute-field helper and tests so top-of-hour cron stagger detection treats wildcard, step, list, range, and range-step minute fields as including minute zero.

## Best Possible Solution

Revise and land this PR or a narrow replacement so the helper follows Croner-compatible minute-field semantics for both 5-field and 6-field expressions, including recursive comma-list entries, while avoiding unsupported numeric-prefix stepping; then use the merged fix to resolve #51587.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Recurse through comma-separated minute entries:** `src/cron/stagger.ts:28`
  - body: `minuteFieldIncludesZero` returns from the comma branch before checking range or step syntax inside each list item. Croner parses list entries recursively, so expressions like `0-5,30 * * * *` and `*/15,45 * * * *` fire at minute zero but this helper returns false, leaving those jobs without the default stagger this PR is meant to add.
  - confidence: 0.94
- **[P3] Remove unsupported numeric-prefix step coverage:** `src/cron/stagger.ts:23`
  - body: OpenClaw constructs Croner with default options, and Croner 10 rejects numeric-prefix stepping such as `0/15` unless sloppy range parsing is enabled. The helper and tests should not lock in `0/15` as supported OpenClaw cron syntax; use Croner-valid forms such as `*/15` or `0-59/15` instead.
  - confidence: 0.86
