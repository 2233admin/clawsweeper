## Summary

Keep open. Current main and the latest release still have the reported literal-minute check, so the PR is not obsolete; however, the PR branch still misses Croner-valid composite comma-list entries and documents unsupported `0/N` syntax, so it should not merge as-is.

## What This Changes

The PR adds a minute-field helper and tests so cron top-of-hour stagger detection treats step, list, range, range-step, and wildcard minute fields as firing at minute zero.

## Best Possible Solution

Tighten the helper so it mirrors Croner-compatible minute-zero detection for both 5-field and 6-field cron expressions, including recursive comma-list entries, and avoid recognizing syntax that OpenClaw's Croner configuration rejects. After that lands with regression coverage through the helper and persisted default-stagger paths, the linked bug can close.

## Reproduction Assessment

Yes. Source inspection on current main is a high-confidence reproduction: `isRecurringTopOfHourCronExpr` only accepts literal minute `"0"`, so nonliteral minute fields that include zero are not eligible for the documented default stagger.

## Solution Assessment

No. The PR is aimed at the right narrow helper, but the implementation should recurse through comma-separated entries and remove `0/N` support so it matches OpenClaw's actual Croner contract.
