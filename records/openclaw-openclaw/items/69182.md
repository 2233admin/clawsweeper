## Summary

Keep #69182 open. Current main fixes the narrower #70481-style hard-timeout overrun, but active-memory is still intentionally a blocking pre-prompt step and can still delay the main reply until its configured timeout. The issue’s core request for non-blocking behavior, lower/auto-tuned latency, or avoiding delay before first visible model output remains unresolved.

## Best Possible Solution

Keep this open as a focused active-memory performance/product item. Maintainers should decide whether to add a non-blocking or async recall mode, stream the first visible reply before recall completes, reduce or auto-tune default latency budgets, or explicitly document the blocking tradeoff as intentional. Treat #70481 as the shipped hard-timeout fix, not as the canonical tracker for this remaining behavior.
