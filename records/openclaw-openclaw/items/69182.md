## Summary

Current main still matches the core complaint: Active Memory remains an intentionally blocking `before_prompt_build` recall pass, so eligible chat turns can wait for the configured recall budget before prompt construction and first model output. Recent work fixed narrower timeout-enforcement, setup-budget, and partial-timeout behavior, but it did not make Active Memory non-blocking, lower the default latency budget, or add an automatic low-latency mode.

## Best Possible Solution

Keep this open as the focused Active Memory latency/design follow-up. Maintainers should decide whether to add an explicit non-blocking or background mode, reduce or auto-tune the default timeout, or keep blocking semantics while adding a clearer low-latency preset and docs that set expectations for first-output delay.

## Risks / Open Questions

- Changing Active Memory to non-blocking could change prompt-context semantics because recalled memory currently has to arrive before the main prompt is built.
- Reducing or auto-tuning the default timeout is a product tradeoff between first-token latency and recall quality; current docs explicitly recommend different budgets by query mode.
- Recent timeout fixes may make the worst stalls less severe, but closing this as implemented would overstate the evidence because the configured blocking recall window still remains.
