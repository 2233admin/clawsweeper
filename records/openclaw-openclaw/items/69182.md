## Summary

Keep open. Current main has useful Active Memory timeout mitigations, but the central report remains valid: eligible chat turns still wait for Active Memory in the foreground `before_prompt_build` path before the main reply can start. Related #75020 was already closed as a duplicate of this issue, so this should remain a canonical tracker rather than be closed during cleanup.

## What This Changes

The issue asks Active Memory to stop delaying first visible chat replies by its full foreground timeout, for example through non-blocking recall, smaller/tuned defaults, or fail-open behavior.

## Best Possible Solution

Keep this issue as the canonical Active Memory foreground-latency tracker for the ~5s blocking symptom. The desired end state should preserve the existing hard-deadline, partial-summary, cache, and circuit-breaker safeguards while making normal chat responsiveness explicit: either a non-blocking/background recall mode, a narrower fail-open foreground budget, or measured docs/defaults that avoid surprising first-token delays.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: enable Active Memory for an eligible direct/webchat session and send a user turn; current source shows `before_prompt_build` waits on `maybeResolveActiveRecall`, which waits for the embedded recall run or its timeout path before the main reply can proceed.

## Solution Assessment

Unclear. The issue names several possible fixes, and current main's timeout mitigations are useful but not the complete solution; maintainers should choose the product shape before an implementation PR changes foreground recall behavior.
