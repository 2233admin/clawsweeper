## Summary

Keep open. Current main has fixed the worse timeout-overrun class from related reports, but it still intentionally runs Active Memory as a foreground `before_prompt_build` pass, so eligible chat replies can still wait up to the configured Active Memory timeout before first visible model output. The linked duplicate review for #75020 also treats this issue as the canonical tracker for the remaining latency/product decision.

## What This Changes

The issue asks OpenClaw to prevent the active-memory plugin from delaying chat replies by waiting on its blocking pre-reply memory recall step up to the configured timeout.

## Best Possible Solution

Keep this issue as the canonical Active Memory foreground-latency tracker and decide the intended product contract: either preserve blocking recall with clearer fast defaults and profiling, or add a plugin-owned/background recall mode that cannot delay first visible output while keeping existing blocking behavior opt-in.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: enable Active Memory for an eligible direct chat, make the recall subagent slower than `config.timeoutMs`, and current main will wait in `before_prompt_build` until the recall returns, times out, or the circuit breaker skips later turns.

## Solution Assessment

Unclear. Making Active Memory fully non-blocking is not a drop-in fix because its current purpose is to inject memory before the main answer; a maintainer should choose between semantic change, new mode, timeout defaults, or live profiling first.
