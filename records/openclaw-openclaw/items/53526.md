## Summary

Keep open. Current main and the latest release still keep implicit session resets on the daily default, while this PR attempts a real behavior change but has a blocking guard bug already identified in review discussion and needs maintainer acceptance of longer shared-channel context retention.

## What This Changes

The PR changes implicit group and thread session reset defaults to idle with a 10080-minute window while leaving direct/DM sessions on the daily default, and updates reset-policy tests.

## Best Possible Solution

If maintainers accept the behavior change, implement the type-aware implicit default in `src/config/sessions/reset-policy.ts`, keep explicit base/type/channel overrides authoritative, add exact `10080` coverage for no-config and unrelated-override group/thread paths, and update the user-facing session docs plus changelog.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope the idle default check to this reset type:** `src/config/sessions/reset.ts:114-115`
  - body: The new `impliedDefaultMode` uses `!hasExplicitReset`, but `hasExplicitReset` becomes true when any `resetByType` entry exists. A config with only `dm` or `direct` configured therefore still resolves group/thread sessions to `daily`, so the primary fix does not apply for a common override shape.
  - confidence: 0.93
- **[P3] Assert the exact group idle window:** `src/config/sessions/sessions.test.ts:151`
  - body: The updated test only checks that group `idleMinutes` is not `45`, which would also pass for `undefined` or another wrong value. Pin the expected `10080`-minute default so the behavior this PR adds is actually covered.
  - confidence: 0.82
