## Summary

Keep open. Current main already transcribes Telegram voice/audio messages and frames machine-generated transcripts for the agent, but it does not expose the requested structured voice-note/message-type flag. Telegram still collapses native `message.audio` and `message.voice` into the same generic audio placeholder, and the shared inbound context/metadata surfaces have no `is_voice`, `message_type`, or equivalent field.

## Best Possible Solution

Keep this issue open and implement an additive, typed inbound message-origin/modality field. Populate Telegram from `msg.voice` before audio/voice collapse, expose it through the shared agent context and cache-safe dynamic inbound context block, document it, and add focused Telegram plus shared inbound metadata tests. Consider coordinating with #40838 because both ask for richer Telegram-native inbound context fields.
