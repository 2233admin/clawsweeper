## Summary

Keep open. Current main still collapses Telegram voice notes into generic audio/transcript context, and neither the shared inbound context type nor rendered agent metadata exposes a voice-note or native message-type marker. Related #40838 is broader Telegram metadata work and can inform naming, but it does not supersede this narrower request.

## What This Changes

Expose an agent-visible inbound message-origin or modality field so Telegram voice notes remain distinguishable from typed text or regular audio after transcription.

## Best Possible Solution

Add a backwards-compatible generic inbound message-origin field to the shared context, populate it from Telegram before voice is normalized to audio, render it in the dynamic conversation-info metadata for agents, and document the public field with focused regression coverage for voice notes versus regular audio/text.
