## Summary

Keep this issue open. Current main still has the reported contract mismatch: the strict per-agent schema omits `verboseDefault` and `elevatedDefault`, while defaults, resolver/status surfaces, generated per-agent metadata, and an open closing PR show the work is not landed yet.

## What This Changes

The issue requests that per-agent `agents.list[]` entries accept durable `verboseDefault` and `elevatedDefault` overrides instead of rejecting them during strict config validation.

## Best Possible Solution

The desired end state is one aligned config contract: either accept and document both per-agent keys through schema, public types, generated metadata, docs, and focused regression tests, or deliberately remove the remaining per-agent-looking runtime/type surfaces. Given the existing per-agent verbose resolver surface, schema and docs alignment remains the smaller maintainable fix.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `agents.list` is parsed through strict `AgentEntrySchema`, and that schema does not declare the two reported keys. I did not run the CLI because this review is read-only, but the validation path is direct.

## Solution Assessment

Yes, with one caveat. Adding the missing per-agent schema/type/help/docs/generated metadata plus parser-level regression tests is the narrow fix for `verboseDefault`; the same change should explicitly confirm and thread the intended `elevatedDefault` per-agent contract.
