## Summary

Keep this issue open. Current main still has the reported schema mismatch: `agents.list[]` is validated by a strict `AgentEntrySchema` that omits `verboseDefault` and `elevatedDefault`, while adjacent defaults, resolver/status code, docs examples, and an open closing PR show this contract is still actively being resolved.

## What This Changes

Request to make per-agent `verboseDefault` and `elevatedDefault` consistent across config validation, resolver/runtime behavior, SDK/types, generated metadata, and docs.

## Best Possible Solution

The end state should make one contract explicit: either support these as per-agent overrides through `AgentEntrySchema`, public types, resolved config, generated schema/help metadata, docs, and focused regression tests, or intentionally keep them global-only and remove the remaining per-agent-looking surfaces. Given the existing per-agent verbose resolver/test pattern, schema and docs alignment looks like the smaller path.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `agents.list` is parsed with strict `AgentEntrySchema`, and that schema does not declare the two reported keys. I did not run `openclaw config set` because this review is read-only, but the validation path is direct.

## Solution Assessment

Yes, with one implementation caveat. Adding the missing per-agent schema fields plus generated metadata/docs/tests is the narrow maintainable fix for `verboseDefault`; the same PR should confirm and thread the intended `elevatedDefault` per-agent contract rather than only relaxing validation.
