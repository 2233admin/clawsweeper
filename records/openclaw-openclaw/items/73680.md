## Summary

Keep this issue open. Current main still has the reported contract mismatch: `agents.list[]` is validated by a strict `AgentEntrySchema` that accepts `thinkingDefault`, `reasoningDefault`, and `fastModeDefault` but still omits `verboseDefault` and `elevatedDefault`, while defaults/docs/runtime surfaces continue to reference at least part of that default-resolution path. An open closing PR already targets the issue, so the right cleanup path is maintainer review of that PR rather than closing or queueing duplicate automation.

## What This Changes

The issue requests that per-agent `verboseDefault` and `elevatedDefault` be accepted and documented consistently, or that the runtime/type surfaces stop implying per-agent support.

## Best Possible Solution

Land or revise the existing fix PR so the contract is explicit across `AgentEntrySchema`, public config types, generated schema/help metadata, runtime resolution for both fields, docs, and focused regression tests. If maintainers decide `elevatedDefault` should remain global-only, the per-agent-looking runtime/type surfaces should be removed instead of accepted silently.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `agents.list` is parsed through a strict `AgentEntrySchema`, and that schema does not declare the two reported keys; the generated schema mirrors that omission. I did not run `openclaw config set` because this review is read-only, but the validation path is direct.

## Solution Assessment

Yes for the preferred direction, with one caveat. Adding the missing per-agent schema/docs/test coverage is the narrow maintainable fix for `verboseDefault`; `elevatedDefault` should also be threaded through the actual resolved-agent contract or deliberately kept global-only with the misleading surfaces removed.
