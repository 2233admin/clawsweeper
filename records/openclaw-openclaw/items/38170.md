## Summary

Keep this PR open. Current main still does not implement the requested Telegram workflow: `download-file` is absent from Telegram action discovery/dispatch/runtime, placeholders are still bare `<media:...>` tags, and the only current `file_id` exposure is a sidecar fallback when media was not downloaded. The PR is still useful context for #8843, but it targets removed pre-plugin paths and its download implementation would need a security-aware port to the current Telegram plugin runtime.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but require a refresh against current main. The refreshed work should live in `extensions/telegram/src/channel-actions.ts` and `extensions/telegram/src/action-runtime.ts`, reuse the existing Telegram token/account, transport, proxy, custom `apiRoot`, media-size, SSRF, and media-store helpers, expose `file_id` through a parser-safe inbound placeholder/history contract, update exact placeholder consumers, and add focused Telegram action/body/history tests plus docs.

## Risks / Open Questions

- The PR's download action is security-sensitive: it downloads remote Telegram media and writes local files, but the proposed diff bypasses current Telegram transport/proxy/custom-api-root/SSRF/mediaMaxMb/media-store helpers.
- The PR changes placeholder syntax to include attributes; exact placeholder consumers such as audio detection and inbound-audio dispatch need a parser-safe update, as review comments already flagged.
- The PR currently modifies removed pre-plugin paths, so merging it as-is would not affect current Telegram runtime and would likely conflict with the extension architecture.
- The proposed `/tmp/tg_${fileId.slice(0, 16)}${ext}` output path can collide across different Telegram files with the same prefix.
