## Summary

Current main still does not implement the Telegram workflow this PR targets. Telegram does not advertise or dispatch `download-file`, the Telegram runtime has no `downloadFile` alias/handler, media placeholders remain bare `<media:...>` tags, and the only current `file_id` exposure is a sidecar fallback (`<media:video> [file_id:...]`) rather than the parser-safe placeholder attribute/history contract proposed here. The PR also targets removed pre-plugin paths and needs to be ported into `extensions/telegram` before it can be considered mergeable.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but require a refresh against current main. The refreshed implementation should live in `extensions/telegram/src/channel-actions.ts` and `extensions/telegram/src/action-runtime.ts`, expose `download-file` through Telegram message-tool discovery, implement downloads by reusing the existing Telegram token/account, transport, SSRF, custom apiRoot, size-limit, and media-store helpers, expose `file_id` through a parser-safe inbound placeholder/history contract, update exact placeholder consumers such as audio detection, and add Telegram action/body/history tests plus user-facing Telegram docs. If maintainers decide the feature should not be bundled, close it with an explicit product decision rather than as already implemented.

## Risks / Open Questions

- The PR is stale against current main: it modifies old core Telegram action paths that no longer exist, so it cannot land without a plugin-layout port.
- The proposed download implementation in the PR context bypasses the current Telegram proxy/network fallback and media SSRF/download helper, which could regress deployments that require Telegram proxying or custom Bot API roots.
- The proposed deterministic `/tmp/tg_${fileId.slice(0, 16)}${ext}` output path can collide and overwrite files for different Telegram file IDs with the same prefix, as the existing review comment noted.
- Embedding attributes in media placeholders will require updating exact-tag consumers such as inbound-audio detection and adding focused tests so audio/tts routing does not regress.
