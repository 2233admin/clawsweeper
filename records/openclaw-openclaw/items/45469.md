## Summary

Keep open. Current main still has an unbounded GatewayClient reconnect path for ordinary WebSocket closes; the existing hardening covers request timeouts, reconnect-timer cleanup, and selected terminal auth pauses, but it does not add a retry ceiling or caller-specific reconnect policy for generic network/restart closes.

## What This Changes

The issue requests a bounded or configurable reconnect policy for `GatewayClient.scheduleReconnect()` so permanently unreachable gateways do not trigger indefinite reconnect attempts and resource churn.

## Best Possible Solution

Define an explicit reconnect policy for GatewayClient that preserves indefinite recovery where it is documented, gives UI or supervisor-backed clients a finite terminal path or threshold callback, and covers retry exhaustion, reset after `hello-ok`, auth-pause behavior, and caller defaults in focused tests.

## Reproduction Assessment

Yes. A high-confidence reproduction is to fake an ordinary WebSocket close such as code 1006 or 1012, advance fake timers, and observe current main repeatedly creating replacement sockets because there is no retry counter or cap.

## Solution Assessment

Unclear. The issue identifies a real unbounded path, but the suggested blanket max-retry default is not clearly the best solution because node-host recovery intentionally relies on long-lived reconnects; a caller-specific policy or threshold callback is safer.
