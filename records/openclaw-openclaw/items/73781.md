## Summary

Keep open. Current main still leaves the prepared embedded Pi settings path without a retry override, tool-loop detection remains default-off, Qwen remains on the OpenAI-compatible tool-call path, and #74434 plus #74503 are open closing PRs rather than merged fixes.

## What This Changes

The issue requests that the runtime stop replaying a prior failed tool call and allow the agent to reason or choose a new action after failed or empty exec results.

## Best Possible Solution

Land one canonical agent-runtime fix that disables Pi agent-level auto-retry only for OpenClaw embedded sessions without persisting changes to user Pi settings, preserves prepared compaction overrides, and adds focused regression coverage for failed exec followed by a changed or no-op next assistant action. If maintainers want the repeated-thinking guard too, it should be wired into the shared context or completions path that reaches Qwen traffic.

## Reproduction Assessment

Unclear. The report gives a concrete Qwen/Feishu/failed-exec replay pattern and the linked PRs identify a plausible Pi SDK retry root cause, but this read-only review did not run the live provider path and current tests do not cover that exact sequence.

## Solution Assessment

Unclear. The narrow retry-disablement direction in #74434 looks closest to the root cause, while #74503 covers a broader thinking-loop case, but neither has landed and previous review notes indicate both need revision before merge.
