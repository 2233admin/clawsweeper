## Summary

Keep this PR open. Current main has adjacent mechanisms (`/queue interrupt`, `sessions.steer`, and provider-supplied thread starter/history prompt context), but it does not implement this PR's concrete `/interrupt <message>` chat command or its persistent cross-session thread context store. The PR also still needs a maintainer-quality refresh because the provided review context flags routing, registry, webchat delivery, and thread-context race/keying issues that are directly relevant to the proposed behavior.

## Best Possible Solution

Keep the PR open and refresh it rather than auto-close it. The strongest path is to split or update the work: implement `/interrupt <message>` by reusing the existing `sessions.steer`/abort seams while preserving accountId, threadId, internal/webchat delivery semantics, and native command registration; then handle cross-session thread context either by extending the existing thread starter/history pipeline or by adding a locked, normalized, documented persistent store with focused tests for Slack, Telegram topics, webchat/internal surfaces, and concurrent cron deliveries.

## Risks / Open Questions

- The PR does not appear to add third-party dependencies, workflow changes, lockfile changes, package scripts, or supply-chain-sensitive code execution paths based on the provided file list and diff summary; the main risk is runtime correctness and data scoping.
