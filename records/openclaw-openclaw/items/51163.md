## Summary

Keep this PR open. Current main still has adjacent mechanisms, including queue mode `interrupt` and the lower-level `sessions.steer` RPC, but it does not ship the PR's two concrete changes: a chat-facing `/interrupt <message>` command and a persistent cross-session thread-context store for cron/isolated-agent replies. The supplied PR discussion also still contains material routing, native command registration, thread/account preservation, and serialized-write concerns, so this is not conservative cleanup closure material.

## Best Possible Solution

Keep the PR open and refresh it. The best path is to implement `/interrupt <message>` on top of the existing `sessions.steer`/abort seams, register it in the chat command registry, and preserve accountId, threadId, and internal/webchat delivery semantics. The cross-session context work should either integrate with the existing provider thread starter/history pipeline or add a documented, normalized, serialized persistent store with focused tests for Slack, Telegram topics, webchat/internal routing, and concurrent cron deliveries.

## Risks / Open Questions

- Closing this PR would drop an unmerged implementation candidate for linked feature work that current main does not yet implement.
- Merging the PR as-is could misroute interrupted replies on multi-account, threaded, or internal/webchat surfaces if accountId, threadId, and delivery mode are not preserved end to end.
- The proposed persistent thread-context store can inject prior task summaries into later prompts; key normalization and serialized writes need careful review to avoid cross-conversation context leakage or nondeterministic context loss.
