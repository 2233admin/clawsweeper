## Summary

Keep this PR open. Current main has adjacent interruption primitives (`/queue interrupt`, Gateway `sessions.steer`, and Control UI `/redirect`), but it still does not implement the PR’s two intended changes: a Gateway/chat-facing `/interrupt <message>` command and a persistent cross-session thread-context store for cron/isolated-agent follow-up replies.

## Best Possible Solution

Keep this PR open and refresh it against current main. The best path is to split or tighten the two features: implement `/interrupt <message>` on top of the existing `sessions.steer`/abort seams with chat command registry and docs coverage, preserving accountId, threadId, internal/webchat delivery mode, and subagent cleanup; then implement cross-session thread context with normalized provider keys, serialized writes, scoped prompt injection, and focused Slack, Telegram, webchat/internal, and concurrent cron-delivery tests.

## Risks / Open Questions

- Closing this PR would drop an open implementation candidate for behavior that current main does not ship end to end.
- Merging as-is could misroute redirected replies on multi-account or threaded surfaces because the PR requeue path does not preserve all delivery context.
- The proposed persistent thread-context store injects prior run summaries into future prompts, so key normalization, serialized writes, TTL/cap pruning, and prompt-leakage boundaries need careful review.
