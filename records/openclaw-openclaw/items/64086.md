## Summary

Keep open. Current `origin/main` still lacks the requested `PluginStatusProvider`/`registerStatusProvider` API, registry storage, SDK docs, and TUI footer consumption, while the issue is paired with open PR #72557 using resolving syntax. Adjacent plugin session status/debug lines exist, but they are projected to verbose status/reply surfaces, not a supported TUI footer contribution path.

## What This Changes

The issue requests a plugin API that lets plugins register small TUI footer status providers for context-engine stats, provider health, and similar plugin-owned indicators.

## Best Possible Solution

Use #72557 as the active implementation candidate, but close this issue only after main provides an approved plugin-facing status provider contract and a supported path for the TUI footer to consume those contributions, coordinated with #53774 for broader footer customization.
