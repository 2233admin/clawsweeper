## Summary

This should stay open: current main still matches the reported failure mode because `readPageSummaries` reads every wiki markdown page with an unbounded `Promise.all`, and `compileMemoryWikiVault` may invoke that read burst multiple times in one compile. The fix is narrow and belongs in the bundled `memory-wiki` plugin, with an existing concurrency helper available through the plugin SDK.

## What This Changes

The issue requests bounded filesystem read concurrency for memory-wiki compile/ingest so iCloud/FileProvider-backed vaults do not hit EDEADLK, hangs, or partial zero-length reads under bursty `fs.readFile` load.

## Best Possible Solution

Add a bounded worker limit inside the memory-wiki page read path, preferably using the existing SDK concurrency helper and a conservative default such as 8, with a focused regression test that proves peak concurrent `fs.readFile` calls stay under the cap while page ordering and compile output remain stable.

Preserve deterministic page ordering and existing compile output.
