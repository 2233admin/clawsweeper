## Summary

Keep open. Current main still lacks the media-specific outbound sanitizer, but this branch is dirty against main and has blocking regressions in the non-media error paths plus an unsafe recursive cause-chain walk.

## What This Changes

The branch adds a safe media-fetch error message/helper, replaces MediaFetchError text at attachment hydration, broadcast, and message_sent failure boundaries, and adds focused sanitizer tests.

## Best Possible Solution

Land a rebased media-specific sanitizer that maps MediaFetchError to one stable channel-safe message at outbound action and hook boundaries, keeps formatErrorMessage as the fallback for all non-media failures, and uses a cycle-safe classifier with regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: failed loadWebMedia produces detailed MediaFetchError text, attachment hydration forwards it, and current broadcast/message_sent paths format that error instead of substituting a stable media-safe message.

## Solution Assessment

No, not as proposed. The direction is right, but the patch should preserve the shared non-media error formatter and make the MediaFetchError cause-chain detector cycle-safe before merge.

  - confidence: 0.91

  - confidence: 0.89
- **[P2] Guard cause-chain traversal against cycles:** `src/media/fetch.ts:40-41`
  - body: `isMediaFetchError` recursively follows `Error.cause` on arbitrary caught outbound errors. A cyclic cause chain from a plugin or adapter would overflow before the failure can be handled; traverse iteratively with a seen set like `formatErrorMessage`.
  - confidence: 0.82
