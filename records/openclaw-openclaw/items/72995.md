## Summary

Keep open. Current main still supports the reported failure class: managed `web_search` does not propagate the agent tool cancellation signal into the web-search runtime or Gemini provider contract, and abort-shaped provider failures still bypass normal tool error logging.

## What This Changes

The issue reports intermittent instant `web_search` aborts with the Gemini provider and no corresponding `gateway.err.log` entry.

## Best Possible Solution

Add a backwards-compatible web-search execution context that carries `AbortSignal` and abort diagnostics through the managed tool wrapper, runtime, provider SDK type, provider helper APIs, and Gemini implementation; log unexpected provider `AbortError` failures safely while preserving true user/session cancellation as cancellation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear for the intermittent live Gemini abort itself because it is timing-dependent and this read-only review did not run live calls. The supporting source path is high-confidence: a provider `AbortError` can still reach the PI adapter path that rethrows before normal tool error logging.

## Solution Assessment

Is this the best way to solve the issue? Unclear as a final root-cause fix because the suspected session-level abort race is plausible but not proven. The maintainable direction is an additive provider execution-context seam plus focused cancellation/logging tests, rather than a Gemini-only timeout tweak.
