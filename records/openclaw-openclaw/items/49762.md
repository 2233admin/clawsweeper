## Summary

Keep this PR open. Current main still renders the per-response usage footer as token/cache/cost/session metadata only; the existing diagnostic duration is not passed into the footer formatter. The PR’s core feature remains unimplemented, but the submitted diff needs rework for the review comments before it is merge-ready.

## Best Possible Solution

Keep the PR open for author or maintainer rework: add response duration to the intended usage-footer mode, preserve existing token/cache/cost formatting when duration is absent, decide whether duration belongs only in `full` mode or in both modes, sample or label the timing consistently, and add focused runner/formatter tests plus docs if the user-visible footer changes.

## Risks / Open Questions

- The submitted formatter refactor changes the cost separator from ` · est` to ` / est` even when durationMs is omitted, contradicting the PR’s backward-compatibility claim.
- Passing durationMs into the formatter unconditionally would also change `/usage tokens`, even though current docs and behavior keep that mode leaner than `/usage full`.
- The proposed sampling point may include post-generation delivery drains, so the label needs either an earlier measurement point or wording that makes clear it is end-to-end wall time.
- The feature needs targeted tests for formatter output, `/usage tokens` versus `/usage full`, and cost separator preservation.
