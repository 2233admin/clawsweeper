## Summary

Keep open. The issue is current and well narrowed: current main still routes Anthropic OAuth Claude CLI runs through the Claude live stdio path with `--replay-user-messages`, while the direct Anthropic HTTP converter already groups consecutive internal `toolResult` messages. There is no source, test, changelog, or related-item evidence that the Claude live-session replay regression has been fixed on main.

## What This Changes

The issue requests a fix for a 2026.4.x regression where replay after multi-tool assistant turns can submit Anthropic `tool_use` blocks without the immediately following grouped `tool_result` blocks, especially on Claude CLI/OAuth live sessions.

## Best Possible Solution

Add a focused regression for a Claude live stdio session that has a retained assistant turn with 2+ tool calls followed by split `toolResult` records, then fix the live-session/replay boundary so a later user prompt cannot submit orphan Anthropic `tool_use` blocks. Keep the direct Anthropic HTTP converter's existing grouping behavior intact, and split the separate `auth-profiles.json.order.anthropic` reset note into its own issue if it is still reproducible.

## Reproduction Assessment

Yes. The reporter gives a high-confidence fresh-session reproduction with version bisection: start an `anthropic:claude-cli` OAuth session, produce one multi-tool assistant turn, then send a simple follow-up and observe Anthropic rejecting orphan `tool_use` ids; current main still matches the described Claude live stdio route.

## Solution Assessment

Unclear at implementation level, but the first repair target is clear. The direct Anthropic converter is already doing the suggested grouping, so the narrow maintainable fix should start in `src/agents/cli-runner/claude-live-session.ts` and its replay/stdin tests rather than broad transcript rewrites.
