## Summary

Keep open. The reporter and later source-level comment narrowed this from a generic compaction failure to a current `claude-cli`/OAuth live-session replay regression, and current `main` still routes that surface through warm Claude stdio with `--replay-user-messages` while only writing the next user prompt to stdin. The direct Anthropic HTTP converter already groups consecutive `toolResult` messages, so the remaining work belongs in the Claude CLI live-session/replay boundary, not as a duplicate direct-transport fix.

## What This Changes

The issue reports that multi-tool Anthropic/Claude CLI session replay can submit assistant `tool_use` blocks without the immediately following grouped `tool_result` blocks after a later user message.

## Best Possible Solution

Add a focused regression for the `claude-cli` live stdio path with an assistant turn containing multiple tool calls and split tool-result records, then fix that boundary so follow-up user turns cannot replay retained `tool_use` blocks without all matching results adjacent/grouped. Preserve the direct Anthropic HTTP converter’s existing grouping behavior, and only change transcript compaction or pruning if the regression proves the live replay state is being rebuilt from pruned OpenClaw session data.

## Reproduction Assessment

Yes. The reporter provides a high-confidence fresh-session reproduction on `anthropic:claude-cli` OAuth: a multi-tool assistant turn followed by any later user message causes Anthropic to reject orphan `tool_use` IDs, and the current source path matches that route.

## Solution Assessment

No for the issue body’s first suggested direct-converter fix: current `main` already groups consecutive `toolResult` messages in the direct Anthropic HTTP transport. The narrower maintainable fix is in the `claude-stdio` live-session/replay path or in the transcript state it hands to Claude Code.
