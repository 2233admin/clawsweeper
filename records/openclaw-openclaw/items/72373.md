## Summary

Keep open. Current main still forwards OpenShell sandbox exec command text into the shared SSH `/bin/sh -c` wrapper without generated-command or malformed-shell preflight, and generic repeated same-tool/same-args loops still return warning-level results until separate no-progress breakers apply. The open loop PRs overlap the repeated-call half, but they do not cover the OpenShell command-validation gap.

## What This Changes

Request OpenShell to reject malformed or generated shell command strings before shell execution and make repeated identical tool calls hard-abort after a threshold.

## Best Possible Solution

Add a focused OpenShell/SSH exec preflight that rejects the reported non-runnable placeholder commands and malformed shell syntax before remote shell invocation, then resolve the generic repeated-call hard-abort behavior through the existing loop-detection work so one canonical threshold policy ships with tests and docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: commands like `workflow install <name>` or an unclosed quote currently flow from OpenShell into `buildExecRemoteCommand` and `/bin/sh -c`; generic repeated calls can be reproduced from the existing loop-detection tests, which still assert warning-level behavior.

## Solution Assessment

Yes. The narrow maintainable direction is OpenShell/SSH exec preflight plus coordinated loop-detector escalation, not a broad heuristic filter across all host exec modes or a separate parallel loop policy.

## Repair Work Prompt

Fix openclaw/openclaw issue #72373 narrowly. Current main forwards OpenShell sandbox exec command text through `extensions/openshell/src/backend.ts` into `buildExecRemoteCommand` and then `/bin/sh -c` without rejecting generated placeholders or malformed shell syntax first. Add a focused preflight in the OpenShell/shared SSH exec path before remote shell invocation, preferably before creating the SSH session when feasible. Reject the reported hazards with a clear tool-visible error: literal generated placeholders such as `<name>` or `<workflow-id>` in command/argument positions, unclosed single or double quotes, trailing unescaped backslash escapes, unterminated backticks, and unterminated command substitutions. Preserve legitimate shell syntax such as input redirection, here-docs, and ordinary commands. Add regression tests for both rejected examples and accepted valid shell syntax in the OpenShell/SSH helper tests. Do not broaden host exec YOLO filtering, do not change unrelated sandbox behavior, and do not change generic loop-detection defaults or policy unless intentionally superseding the existing #58726/#60248/#62775 work with maintainer-approved scope. If loop detector code is touched, coordinate with those refs and update the existing loop tests/docs. Changelog is only needed if the resulting behavior is user-facing in a release note-worthy way; keep any entry single-line with proper human attribution.
