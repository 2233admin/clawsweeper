## Summary

Keep open. Current main still forwards OpenShell exec text into the shared SSH `/bin/sh -c` wrapper without generated-placeholder or malformed-shell preflight, and the generic same-tool/same-args loop detector remains warning-only unless a later no-progress/global breaker reaches its own threshold. Recent post-report loop commits improved exec outcome hashing and run scoping, but they do not implement the requested fail-fast OpenShell validation or configurable hard-abort for repeated malformed/generated calls. The related #58726 remains an open narrower loop-abort report, but it does not cover the OpenShell command-preflight half of this issue.

## Best Possible Solution

Keep this issue open and implement a focused runtime hardening change: add shared exec/OpenShell command preflight before SSH command construction for obvious generated placeholders and malformed shell structure; make repeated identical failed or no-progress generic calls capable of becoming a clear critical block at the configured threshold; add regression tests for OpenShell/shared SSH command construction and before-tool-call blocking; and separately lint or rewrite bundled skill/help examples that look copy-executable while containing placeholders.

## Risks / Open Questions

- A generated-command preflight needs careful scoping so legitimate shell syntax, redirection, comparisons, command substitution, and literal angle brackets in data are not rejected incorrectly.
- A generic repeat hard-abort should probably be failure/no-progress gated and configurable to avoid blocking legitimate repeated status checks or polling workflows.
- The current global no-progress breaker can still allow many repeated malformed exec attempts before a critical block, especially when outputs vary enough to avoid stable no-progress hashing.
