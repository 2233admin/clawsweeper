## Summary

Keep open. Current main still forwards OpenShell/SSH sandbox exec command text to `/bin/sh -c` without generated-command preflight, and generic repeated same-tool/same-args loops remain warning-only until the later global no-progress breaker. The related #58726 issue covers the loop-abort half, but it does not cover malformed OpenShell command validation.

## What This Changes

The issue requests fail-fast validation for malformed/generated OpenShell commands and hard-abort behavior for repeated identical failing tool loops.

## Best Possible Solution

Add a shared sandbox exec preflight before the SSH `/bin/sh -c` wrapper for obvious generated-command hazards, then escalate repeated failed or no-progress generic tool calls at the configured critical threshold while keeping progressing repeats warning-only. Update focused tests and any bundled skill/help examples that present placeholders as copy-executable commands.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: an OpenShell sandbox exec command containing a literal placeholder or unbalanced shell syntax reaches `buildExecRemoteCommand` and `/bin/sh -c`, and the loop tests encode generic repeats as warning-only below the global breaker.

## Solution Assessment

No. Current main is only a partial mitigation because it has unknown-tool and global no-progress breakers, but not the requested malformed OpenShell command preflight or critical-threshold generic failed/no-progress abort.
