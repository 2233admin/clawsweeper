## Summary

Keep open. Current main still lacks the requested OpenShell generated-command preflight, and generic repeated tool calls remain warning-level until the separate no-progress breaker fires. The related loop issue and open loop PRs cover the hard-abort policy half, but they do not cover the OpenShell command-validation gap.

## What This Changes

Add OpenShell preflight validation for malformed/generated shell commands and hard-abort repeated identical tool loops after the configured threshold.

## Best Possible Solution

Add a focused OpenShell/SSH exec command preflight before remote shell construction, with tests for the reported generated-command hazards, and converge the generic-repeat hard-abort behavior through the existing loop-detection issue/PR track instead of creating a second policy fork.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: commands such as workflow install <name> flow from OpenShell into buildExecRemoteCommand and /bin/sh -c without preflight, while current loop tests still assert generic repeats are warning-only below the global breaker.

## Solution Assessment

Yes. The narrow maintainable direction is OpenShell-owned command preflight plus coordinated loop-detector escalation through the canonical loop work; a repo-wide heuristic across all exec modes would be riskier.
