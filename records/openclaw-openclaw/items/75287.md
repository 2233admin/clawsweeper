## Summary

Keep open. Current main still has deterministic paths that match the report: arbitrary non-builtin command roots can enter plugin CLI registration, channel log commands still use the broad `channels` startup plugin policy, and model/session gateway paths can still start provider catalog work under polling.

## What This Changes

The issue asks for invalid CLI commands, channel-log reads, and gateway model/session/control polling to avoid expensive plugin/provider loading and to exit without hot child processes.

## Best Possible Solution

Land a focused performance fix that validates unknown command roots from core descriptors and manifest command metadata before runtime plugin loading, keeps `channels logs` on a plugin-free metadata/log-file path, and serves model/session polling from cached, single-flight, or manifest-backed provider catalog data instead of repeatedly materializing broad provider runtime hooks.

## Reproduction Assessment

Yes. The issue provides concrete commands, process samples, plugin-load output, gateway timing logs, CPU samples, and event-loop diagnostics; current source inspection confirms deterministic paths that can still produce the reported plugin-loading behavior.

## Solution Assessment

Not applicable as a patch. The best repair should be narrower than a loader redesign: fail arbitrary unknown CLI roots before full plugin CLI registration, keep known plugin command activation compatible, and make gateway catalog/polling paths avoid broad uncached provider runtime materialization.
