## Summary

Keep open. Current main still contains a plausible and partly intentional path that loads plugin runtimes for an unrecognized CLI primary command, and the gateway model/session paths still start model-catalog/provider augmentation work under polling; the only hot-path file change after the reported commit is an unrelated proxy-config read change.

## What This Changes

The issue requests that unknown CLI commands fail fast without plugin/provider loading or lingering hot children, and that gateway model/session/log/control surfaces avoid repeated expensive provider/plugin loads under polling.

## Best Possible Solution

Land a focused performance/CLI fix that validates unknown command roots from core descriptors and manifest command metadata before runtime plugin loading, exits cleanly on unavailable commands, and keeps model/session/control polling on cached or lightweight provider metadata instead of repeated broad provider runtime materialization.

## Reproduction Assessment

Yes. The issue provides concrete commands, process samples, logs, CPU/event-loop diagnostics, and current source inspection shows deterministic paths that match the reported plugin loads; live process verification should be part of the repair PR.

## Solution Assessment

Not applicable as a patch. The best repair should be narrower than a loader redesign: fail arbitrary unknown CLI roots before full plugin CLI registration, preserve known plugin CLI activation, and make gateway model catalog/provider augmentation single-flight, cached, or manifest-backed for polling paths.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75287 with a narrow performance/CLI repair. Observable bugs: `openclaw foo` or another arbitrary unknown root must fail fast without provider catalog loads, full runtime plugin loads, or a lingering hot child process; gateway `models.list` / `sessions.list` / Control UI polling must not repeatedly cold-load provider plugins or saturate the event loop when no agent run is active. Keep the fix bounded: do not redesign the plugin loader, do not remove legitimate plugin CLI command support, and do not make core depend on bundled plugin ids. Prefer manifest command aliases/activation metadata to decide known plugin command roots before full runtime CLI registration; arbitrary unknown roots should reach a normal unknown/unavailable-command error without `loadOpenClawPlugins`. For gateway catalog work, keep model/session polling on a cheap cached/single-flight or manifest-backed metadata path and avoid repeated `cache:false` provider runtime materialization. Preserve the existing `channels logs` manifest/log-file behavior. Add regression tests for unknown CLI command no-plugin-load/clean-exit behavior and for repeated model/session list calls avoiding broad provider reloads, including empty/error catalog cases.
