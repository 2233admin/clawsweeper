## Summary

Keep open. The report is recent, detailed, and was captured on a commit that already included the prior #73853/#73847 provider-registry cache work; current `main` has no relevant changes after that reported commit. Source inspection shows partial protections exist, but `models.list` can still drive the gateway model-catalog path into provider catalog augmentation, which is consistent with the reported provider-plugin load bursts and event-loop pressure.

## What This Changes

Investigate and reduce repeated provider-plugin catalog/runtime loading and gateway event-loop saturation when model/session/log surfaces are hit while the Control UI is polling.

## Best Possible Solution

Keep this open and land a focused gateway/plugin performance fix that makes model/session/list surfaces serve from cheap metadata or a bounded single-flight cache, prevents repeated broad provider runtime loading under polling, and keeps channel log/status commands on manifest/log-file paths.

## Reproduction Assessment

Yes. The report gives a concrete source commit, gateway command, Control UI polling behavior, log excerpts, CPU samples, liveness diagnostics, and CLI commands; the current source still contains a plausible model-catalog-to-provider-plugin hot path to exercise.

## Solution Assessment

Not applicable as a patch, but the best fix should be narrower than another broad plugin-loader refactor: make the gateway model catalog and visible configured-model views avoid provider runtime materialization on the polling path, with regression tests for repeated calls and empty/error catalog cases.
