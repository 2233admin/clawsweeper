## Summary

Keep open. Current main still rejects the described remote plain-HTTP Control UI path and documents the secure-context requirement, but gateway startup logging does not yet emit the requested operator warning; the linked implementation PR is still open and unmerged, so the issue should remain open until maintainers land, replace, or reject that work.

## What This Changes

Request a gateway startup warning when Control UI is enabled and remote plain-HTTP access will be rejected because device auth still requires a secure browser context.

## Best Possible Solution

Land a narrow gateway startup-log change, likely by reviewing and cleaning up #73163 or replacing it with an equivalent focused PR, that warns only when resolved Control UI, bind-host, TLS, and device-auth configuration make remote plain-HTTP browser connections likely to fail. Keep the existing security enforcement unchanged and cover warning/suppression cases in startup-log tests.
