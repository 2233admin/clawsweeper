## Summary

Keep open. Current main still rejects non-secure, non-localhost Control UI sessions without device identity and documents the workaround, but it does not emit the requested gateway startup warning; an open unmerged PR already targets this exact issue with closing syntax, so the issue should stay open until that PR lands or is closed.

## What This Changes

The issue requests a gateway startup warning when Control UI is enabled and plain HTTP remote clients will be rejected because browser device identity is unavailable outside a secure context.

## Best Possible Solution

Land a narrow gateway startup warning in the startup logging path, with regression tests for warning and suppression cases, while preserving the existing Control UI device-identity rejection behavior and leaving broader policy changes to the related issues.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: call `logGatewayStartup` on current main with Control UI enabled, non-loopback HTTP bind, and TLS disabled; no warning is emitted, while the connection policy and auth tests still reject device-less remote Control UI connections.

## Solution Assessment

Yes. Adding startup-log guidance is the narrowest maintainable fix for this issue because it improves operator diagnostics without weakening the existing auth policy; the linked PR should be reviewed, rebased if needed, and either landed or replaced.
