## What This Changes

The PR changes Google Chat webhook auth to accept issuer/email variants without email claims, adds auth rejection diagnostics and tests, and falls back sender email extraction to event.user.email.

## Best Possible Solution

A rebased or replacement Google Chat fix should preserve the shipped per-account diagnostics, validate any issuer-only or add-on auth change against real Google token shapes and google-auth-library behavior, keep add-on sub binding strict, and add focused coverage for the sender-email fallback if that behavior is still desired.

Yes for the blocking PR defects: google-auth-library@10.6.2 source shows the mocked issuer-only payloads are rejected before the PR's checks run, and the PR diff shows diagnostics always logging through targets[0]. Live Google Chat delivery was not exercised.

## Solution Assessment

Full review comments:

- **[P2] Verify issuer-only tokens through the real issuer gate:** `extensions/googlechat/src/auth.ts:128-132`
  - body: The new issuer-only acceptance checks run after verifyIdToken(), but google-auth-library 10.6.2 rejects issuers outside accounts.google.com, https://accounts.google.com, or the universe domain before returning a ticket. Tokens with iss set to the Chat service account or Workspace Add-on service account never reach this branch, so the mocked tests do not prove the intended production 401 fix.
  - confidence: 0.88
- **[P2] Log failures through the checked target:** `extensions/googlechat/src/monitor-webhook.ts:108`
  - body: logAuthFailure() always selects targets[0] as the logger even though the collected rejection reasons may belong to later accounts on the same webhook path. In multi-account deployments this routes diagnostics through the wrong account runtime sink and weakens the per-account troubleshooting this change is meant to add.
  - confidence: 0.83
