## Summary

Keep open. Current main partially handles the adjacent stale-token case by refusing transcript fallback when `totalTokensFresh=false`, but it still trusts a persisted fresh `totalTokens` after compaction and the compaction-count update path leaves old totals intact when no trustworthy `tokensAfter` is available. The provided related context also identifies open PR #72285 as the active focused implementation candidate, so this issue should not be closed from current-main evidence.

## What This Changes

The issue asks `/status` to stop showing stale pre-compaction context totals for sessions that have already compacted, preferring a trustworthy post-compaction snapshot or unknown instead.

## Best Possible Solution

Review and land #72285 or an equivalent narrow fix that makes compaction token state authoritative: use the latest trustworthy post-compaction `tokensAfter` when available, and otherwise invalidate or hide pre-compaction totals so `/status` renders unknown rather than stale over-budget context.

## Reproduction Assessment

Yes. A source-level reproduction is clear: use a compacted session entry with `compactionCount > 0`, `contextTokens: 128000`, stale `totalTokens: 151000`, `totalTokensFresh: true`, and no post-compaction `tokensAfter`; current `buildStatusMessage` will render that stale total as current context.

## Solution Assessment

No, current main is not sufficient because it still allows stale fresh totals to survive compaction. The safer solution is to enforce freshness at the compaction boundary and have status prefer only a latest trustworthy post-compaction snapshot or unknown.
