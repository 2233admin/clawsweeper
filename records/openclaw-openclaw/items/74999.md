## Summary

Keep open: current main does not implement namespaced `/models` grouping, and this PR is useful but not merge-ready because the new scope is not carried through the interactive Telegram model-list path.

## What This Changes

The PR adds `/models <provider>/<group>` parsing and grouped text output for namespaced model catalogs, with tests for aggregator grouping, scoped drilldown, flat `all`, and local-model non-grouping.

## Best Possible Solution

Keep the namespaced grouping direction, but make the model-list channel hook scope-aware or fall back to text grouped/scoped output when `modelPrefix` is present; then add Telegram coverage and update the slash-command docs.

## Reproduction Assessment

Yes. A high-confidence static repro is a Telegram `/models litellm` or `/models litellm/deepseek` catalog with namespaced models: the PR computes scoped `models`, but Telegram list rendering is reached before grouped text and its pagination callbacks encode only the provider.

## Solution Assessment

No, not as-is. The text-surface implementation is close, but the maintainable fix needs to thread namespace scope through interactive list rendering or intentionally bypass that renderer for grouped/scoped views.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Thread namespace scope through interactive model lists:** `src/auto-reply/reply/commands-models.ts:552`
  - body: The new scoped `models` array is passed into `buildModelsListChannelData` with only `provider`, and Telegram is the current implementation of that hook. That means `/models litellm` on Telegram returns before the group text is built, and `/models litellm/deepseek` can render page 1 of the scoped list while Prev/Next callbacks reopen the unscoped provider list. Please make the channel data scope-aware or fall back to the text grouped/scoped response for these paths.
  - confidence: 0.88
