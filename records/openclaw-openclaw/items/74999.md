## Summary

Keep this PR open for author/maintainer follow-up. The namespaced grouping direction is useful, but the current head still routes Telegram `/models <provider>` and scoped `/models <provider>/<group>` through the existing provider-only interactive list path before the new grouped text path can run, so the main user-facing Telegram surface does not preserve namespace scope.

## What This Changes

The PR adds provider/model namespace parsing and grouped text output for `/models <provider>/<group>`, with tests for aggregator grouping, scoped listings, flat `all`, and local-model non-grouping.

## Best Possible Solution

Keep the namespace grouping feature, but make the channel list adapter and Telegram callback data scope-aware, or intentionally bypass interactive list rendering for grouped/scoped views. Then add Telegram coverage for top-level grouped catalogs and scoped pagination, plus a docs update for the new drilldown syntax.

## Reproduction Assessment

Yes. A static high-confidence repro is Telegram `/models litellm` with multiple namespaced groups: the PR returns provider-only channel data before grouped text is built. A scoped list with more than one page also reproduces scope loss because Telegram pagination callbacks encode only the provider.

## Solution Assessment

No, not as currently patched. The text fallback implementation is close, but the maintainable fix needs to thread namespace scope through the interactive channel contract and Telegram callback handling, or avoid that renderer for the grouped/scoped cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Thread namespace scope through interactive model lists:** `src/auto-reply/reply/commands-models.ts:577-585`
  - body: The PR computes a scoped `models` array, but still calls `buildModelsListChannelData` with only `provider`. Telegram implements this hook and returns here before grouped text is built, so `/models litellm` shows the old flat keyboard; scoped lists can also lose the group on pagination because callbacks are still `mdl_list_<provider>_<page>`. Please make the channel data/callback path scope-aware or fall back to text output for grouped/scoped views.
