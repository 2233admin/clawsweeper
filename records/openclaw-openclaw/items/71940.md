## Summary

Keep open. Current main still uses the CDP screenshot path for normal browser viewport captures, so this PR is not obsolete; however, source tracing at PR head found lifecycle defects in the new Hyprland capture helper that should be fixed before merge.

## What This Changes

The branch adds an opt-in `browser.hyprlandCapture` configuration for headed Linux/Hyprland viewport screenshots, a `grim`/`hyprctl` capture helper, route/docs/schema/test updates, and a public `openclaw/plugin-sdk/resolve-system-bin` subpath.

## Best Possible Solution

Revise this PR in place: preserve normal process shutdown while cleaning the virtual output, remove the virtual output on every setup failure after creation, update the PR body to target #64317, keep the opt-in config/docs/tests aligned, and rerun the focused browser/config/SDK checks plus the changed gate before merge.

## Reproduction Assessment

Yes. #64317 gives a concrete Omarchy/Hyprland/Wayland headed Chromium screenshot timeout path, and the current patch defects are reproducible by source tracing the new setup and signal cleanup paths at PR head.

## Solution Assessment

No. The opt-in `grim` fallback is a plausible narrow solution for the reported Hyprland timeout, but this patch is not the best mergeable form while it can swallow stop signals, leak virtual outputs on setup failure, and carry stale PR metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve shutdown after signal cleanup:** `extensions/browser/src/browser/hyprland-capture.ts:84-93`
  - body: After Hyprland capture is used, these `SIGINT`/`SIGTERM` listeners run cleanup but do not let the signal terminate the process. Node removes the default signal behavior once a listener is installed, so a normal systemd or Docker stop can leave the gateway process alive until a later signal or force kill.
  - confidence: 0.88
- **[P2] Remove the virtual output if setup fails:** `extensions/browser/src/browser/hyprland-capture.ts:106-112`
  - body: `createHeadlessOutput()` can succeed before `getActiveWorkspaceId()` or the workspace move throws, but cleanup is not registered until afterward and `_state` is still null. The route then falls back to CDP while leaving the new Hyprland output behind; remove `outputName` before rethrowing setup failures.
  - confidence: 0.94
