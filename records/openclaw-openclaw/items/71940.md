## Summary

Keep open. Current main still uses the CDP screenshot path and has no Hyprland/grim fallback, so the PR is not obsolete; however, the latest PR head still has concrete merge blockers in setup cleanup and Plugin SDK API baseline drift, and its body still points at the unrelated Windows Gmail issue.

## What This Changes

The PR branch adds an opt-in Hyprland/Wayland `grim` viewport screenshot fallback for headed browser viewport screenshots, browser config/schema/docs/tests, and a new `openclaw/plugin-sdk/resolve-system-bin` subpath.

## Best Possible Solution

Revise this PR in place: keep the opt-in Hyprland fallback, make virtual-output cleanup symmetric for every setup/capture failure, regenerate the Plugin SDK API baseline or avoid adding the public SDK subpath, update the body to target the Hyprland bug instead of #54470, then run the focused browser/config/SDK checks plus the changed gate in Testbox.

## Reproduction Assessment

Yes. #64317 provides concrete Omarchy/Hyprland/Wayland steps and current main can be source-verified to use only CDP for normal viewport screenshots; the setup-cleanup defect is reproducible by tracing a successful `create-output` followed by failure before `_state` or exit cleanup is registered.

## Solution Assessment

No. The feature direction is plausible and current main lacks it, but the proposed patch is not yet the best mergeable solution because it can leak a virtual output during setup failure, leaves the Plugin SDK API baseline stale, and carries a stale closing-issue/body mismatch.
