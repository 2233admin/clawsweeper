## Summary

Keep this PR open. The Windows resolver direction is useful for the linked Gmail setup bug, but the branch still misses the CLI `webhooks gmail setup/run` `gog` invocations, directly spawns a resolved `.cmd` shim in the long-running watcher path, and still carries the Hyprland capture setup-cleanup leak noted in the prior review. The branch also now mixes a Windows Gmail fix with the earlier Hyprland browser feature despite the current title/body describing only the Windows work.

## What This Changes

The PR adds a Windows executable resolver for `gcloud`, `gog`, and `tailscale`, threads it into some Gmail setup/watcher paths, and also carries an opt-in Hyprland/grim browser screenshot fallback with config/schema/docs plus a new Plugin SDK resolver subpath.

## Best Possible Solution

Revise the PR in place or split it: keep a narrow Windows Gmail change that routes every `gcloud`, `gog`, and `tailscale` setup/run/watcher path through one Windows-safe invocation helper, including long-running `gog watch serve`, and keep the Hyprland feature separate unless it is intentionally part of this PR. If the Hyprland changes stay here, make output removal symmetric for every post-create setup failure before landing.

## Reproduction Assessment

Yes. The linked Windows bug gives a concrete `openclaw webhooks gmail setup --account ...` reproduction, and source tracing shows that path still reaches a bare `gog` invocation in `src/hooks/gmail-ops.ts`; the Hyprland leak is reproducible by a mocked create-output success followed by monitor or dispatch failure.

## Solution Assessment

No. Resolving `gcloud` and `tailscale` before `runCommandWithTimeout` is the right direction, but the proposed branch is not the narrowest complete fix because it misses the CLI `gog` setup/run path, directly spawns resolved `.cmd` shims in one watcher path, and carries an unrelated browser feature with an open lifecycle defect.
