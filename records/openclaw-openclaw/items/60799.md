## Summary

Keep this issue open. Current main still converts a top-level plain-array cron store to an empty store, so `cron list` can report zero jobs and the first `cron add` can persist only the new in-memory job. The proposed fix PR #61113 was closed unmerged, and current main/latest release still lack plain-array load or load-add-save regression coverage.

## Best Possible Solution

Add a small cron-store normalization seam in `src/cron/store.ts` used by both `loadCronStore()` and `loadCronStoreSync()` that accepts both `{ version: 1, jobs: [...] }` and legacy top-level `[...]` stores. Add focused regression tests for async load, sync load, and load-add-save preservation of existing legacy-array jobs; then consider a doctor repair/warning so affected users can persist the canonical envelope safely.

## Evidence

The sync loader repeats the same guard.

## Risks / Open Questions

- Closing this now would hide a current data-loss path for users whose `jobs.json` is a legacy top-level array: the scheduler can load zero jobs and the next mutation can overwrite the store from empty in-memory state.
- A doctor-only repair would be insufficient unless the runtime loader also accepts top-level arrays, because startup/list/add paths hit `loadCronStore()` before doctor can normalize anything.
