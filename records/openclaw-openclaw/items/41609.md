## Summary

Keep open. Current main still has the reported direct inbound Feishu interactive-card parsing gap: live `im.message.receive_v1` events call `parseMessageContent()`, and that parser has no `interactive` or `interactive_card` branch. The richer interactive-card parser exists only on the fetched/read message path in `send.ts`, and the adjacent card-action callback path still only reads `action.value`, leaving dropdown/form fields to the separate open callback work.

## Best Possible Solution

Keep this issue open as the canonical direct inbound Feishu interactive-card parsing gap. The best fix is a focused Feishu plugin change that shares the existing card-content extraction with `parseMessageContent()`, handles both `interactive` and `interactive_card` live message types plus merged/forwarded submessages, preserves safe fallbacks when Feishu only supplies a placeholder, adds direct inbound regression tests, and coordinates with #42908 for callback selection/form fields rather than mixing unrelated callback work into the same patch.
