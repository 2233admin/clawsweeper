## Summary

Keep open. Current main does not show a blanket Control UI/WebChat exec block, but it still has a real contract mismatch: `tools.byProvider` is resolved from the selected model provider/model, while config help still describes it as channel/surface-scoped. The reported `allowlist miss` also remains a separate host exec approval-layer denial, so this needs maintainer direction rather than cleanup closure.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor a `tools.byProvider.webchat` policy after elevated sender rules are configured, or whether the remaining `exec denied: allowlist miss` is an intentional additional exec restriction.

## Best Possible Solution

Align the public contract with the implementation. The conservative end state is to clarify schema help/docs that `tools.byProvider` keys model providers/models, document that WebChat/Control UI surface policy is separate, and keep host exec allowlist behavior under `tools.exec.security` and approvals. If WebChat-specific tool policy is desired, add a separate explicit surface/channel policy seam without bypassing exec approvals.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with a Control UI/WebChat session and a selected model provider other than `webchat`, `tools.byProvider.webchat` is not consulted by `resolveEffectiveToolPolicy()`, and `tools.exec.security=allowlist` can still deny an unsatisfied command as an allowlist miss.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a bypass for host exec allowlists would not be the best solution; the safer options are contract/docs correction or a deliberately designed surface-policy seam.
