## Summary

Keep open. Current main still shows the same unresolved contract mismatch: Control UI identifies as the WebChat surface, but `tools.byProvider` is resolved from the selected model provider/model, and generated schema help still describes the key as channel/surface-scoped. The reporter's retest shows Control UI exec can work with `tools.exec.security = "full"`, so this is not a blanket WebChat block; the remaining work needs maintainer product/security judgment around docs/schema clarification versus a deliberate surface-policy seam.

## What This Changes

The issue asks whether WebChat / Control UI exec should be allowed when elevated sender rules and `tools.byProvider.webchat` allow `exec`, or whether the remaining `allowlist miss` is an additional policy bug.

## Best Possible Solution

Align the public contract with the implementation. The maintainable end state is for docs, schema help, and generated schema metadata to state that `tools.byProvider` keys model providers/models; if maintainers want WebChat/Control UI-specific grants, add a separate message-surface policy contract that still leaves `tools.exec.security`, allowlists, and approvals in force.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while using a selected model provider other than `webchat`; `resolveEffectiveToolPolicy()` will not apply that policy, and with `tools.exec.security=allowlist` plus no matching allowlist or durable approval, gateway exec can still throw `exec denied: allowlist miss`.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as a way to open host exec from the browser would not be the safest fix; the narrower maintainable path is docs/schema clarification unless maintainers explicitly design a separate surface-policy seam.
