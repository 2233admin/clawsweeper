## Summary

Keep open. Current main still has a real contract mismatch: Control UI presents as WebChat, but `tools.byProvider` is resolved from the selected model provider/model, and the exec allowlist denial is enforced by a separate host-exec security layer. The reporter’s retest narrows this away from a blanket Control UI exec block and toward allowlist-mode plus policy semantics.

## What This Changes

The issue asks whether WebChat/Control UI exec should honor `tools.byProvider.webchat` plus elevated sender config, or whether the remaining `exec denied: allowlist miss` is intentional allowlist behavior.

## Best Possible Solution

Align the public contract with implementation first: schema help, generated metadata, Control UI copy, and docs should consistently say `tools.byProvider` keys the selected model provider/model. If maintainers want surface-specific WebChat/Control UI grants, add a separate explicit surface-policy contract with tests that preserve host exec allowlist and approval layers.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure `tools.byProvider.webchat` while the session’s selected model provider is something else, then run in `tools.exec.security=allowlist` without a matching allowlist or durable approval; current main will not apply the `webchat` provider policy and can still throw an allowlist miss.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as permission to open host exec from the browser is not the safest narrow fix; the safer maintainable path is docs/schema clarification unless maintainers explicitly approve and design a separate surface-policy seam.
