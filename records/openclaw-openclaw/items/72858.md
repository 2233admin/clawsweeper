## Summary

Keep open. Current main still has a contract mismatch: runtime `tools.byProvider` is resolved from the selected model provider/model, while config help and some docs can still read like channel/surface policy; separately, `exec denied: allowlist miss` is produced by the host exec allowlist layer, not by WebChat-specific tool availability.

## What This Changes

The issue asks OpenClaw to clarify or fix why WebChat/Control UI exec remains denied with an allowlist miss after elevated sender and provider tool policy settings are configured.

## Best Possible Solution

Align the public contract with implementation first: document and schema-help `tools.byProvider` as model-provider/model scoped, keep host exec allowlist behavior under `tools.exec.*` and host approvals, and only add a separate WebChat/channel surface policy seam if maintainers explicitly want that capability without bypassing approvals.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with a Control UI/WebChat session using a selected model provider other than `webchat`, `tools.byProvider.webchat` is not consulted by `resolveEffectiveToolPolicy()`, and `security=allowlist` can still deny an unmatched command as an allowlist miss.

## Solution Assessment

Unclear. Treating `tools.byProvider.webchat` as an exec bypass would not be the safest fix; the narrower maintainable options are docs/schema clarification or a deliberately designed surface/channel policy layer that still composes with host approvals.
