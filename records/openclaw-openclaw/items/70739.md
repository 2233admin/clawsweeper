## Summary

Keep open. The PR addresses a real gateway streaming gap and follows the existing SSE-comment pattern, but the latest diff still has a concrete timer-sanitization bug and leaves a security-sensitive stream-lifetime/concurrency decision for maintainers before merge.

## What This Changes

The PR adds a shared SSE heartbeat helper with tests and wires it into the streaming `/v1/chat/completions` and `/v1/responses` gateway endpoints.

## Best Possible Solution

Land a hardened SSE-comment heartbeat only after normalizing the interval to a finite bounded value, keeping the backpressure/error cleanup, restoring timers in the new tests, and deciding whether this endpoint also needs a stream TTL or concurrent-stream cap as part of the gateway threat model.

## Reproduction Assessment

Yes. The reported current-main gap is source-reproducible: both streaming branches can go quiet because they only write real model events, and the proposed helper would add bytes during those quiet windows; the interval bug is locally reproducible with Node showing non-finite intervals become 1ms timers.

## Solution Assessment

Unclear as-is. SSE comments are the right transport-level direction and match existing session-history precedent, but the proposed helper needs interval validation and a maintainer decision on stream lifetime/concurrency before it is the safest mergeable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Validate the heartbeat interval before scheduling:** `src/gateway/http-common.ts:147`
  - body: `Math.max(1000, options?.intervalMs ?? 30_000)` leaves `NaN`, `Infinity`, and `-Infinity` unchanged. Node coerces those durations to a 1ms interval, so any future caller that passes a malformed numeric override can turn each open SSE stream into a high-frequency write loop despite the intended clamp. Normalize with `Number.isFinite` and fall back to the default before calling `setInterval`.
  - confidence: 0.9
- **[P3] Restore real timers after heartbeat tests:** `src/gateway/http-common.test.ts:301-303`
  - body: The new heartbeat suite enables fake timers in `beforeEach` but never restores real timers. That leaves the rest of this test file running under fake timers, which is brittle for any later async or timer-based assertions added to `http-common.test.ts`. Add an `afterEach` that clears pending timers and calls `vi.useRealTimers()`.
  - confidence: 0.68

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it does introduce a stream-lifetime security posture change that should be reviewed before merge.

Concerns:

- **[medium] Heartbeat bypasses proxy idle caps without stream limits:** `src/gateway/openai-http.ts:639`
  - body: The PR starts heartbeats unconditionally on the authenticated streaming endpoints. Backpressure handling prevents unbounded heartbeat buffering, but current gateway code does not show a maximum stream lifetime or per-subject/IP concurrent-stream cap, so internet-exposed or `auth.mode: none` deployments lose proxy idle timeout as a fallback resource limiter for quiet streams.
  - confidence: 0.74
