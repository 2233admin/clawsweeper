## Summary

Keep open. Current main still sends session-memory slug generation through shared embedded-run auth failure handling without the existing local policy, so a billing-classified helper error can still disable the shared auth profile and trigger provider-wide skip behavior.

## What This Changes

The issue reports that session-memory LLM slug generation can turn an Anthropic claude-cli HTTP 400 extra-usage error into a shared billing cooldown for the whole auth profile.

## Best Possible Solution

Use the existing local auth-profile failure policy on the slug-generator embedded helper, add focused regression coverage for that call, and leave the broader dedicated slug-model/congestion work under #33962 or PR #65134.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: send a billing-classified slug-helper prompt error through `generateSlugViaLLM`; because the helper omits the local policy, the embedded runner can mark the shared auth profile with a billing cooldown.

## Solution Assessment

Yes for a narrow repair. Passing `authProfileFailurePolicy: "local"` at the slug-generator call site is the smallest maintainable fix and avoids changing global billing classification semantics used by normal model fallback.
