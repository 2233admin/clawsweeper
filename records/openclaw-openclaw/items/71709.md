## Summary

Keep open. Current main still lets the session-memory slug helper run through shared embedded-run auth failure handling, so a billing-classified helper prompt error can still write a profile-wide billing disable window; an existing local failure-policy seam can address the bug, but an open broader slug-generator PR makes maintainer routing useful before creating another repair PR.

## What This Changes

The issue requests that slug-generator internal helper failures stop marking a shared Claude CLI auth profile as billing-disabled for five hours.

## Best Possible Solution

Keep this open until maintainers either land the broader slug-generator replacement in #65134 or make a focused fix that passes `authProfileFailurePolicy: "local"` from `generateSlugViaLLM` and adds regression coverage for billing-classified helper errors not mutating shared auth state. Treat any broader change to the Anthropic extra-usage billing classifier as a separate policy decision.

## Reproduction Assessment

Yes. A high-confidence source reproduction path exists: have the slug-generator embedded run receive a prompt error whose text classifies as billing, and because the helper call lacks the local policy, the runner can mark the active auth profile as billing-disabled in shared state.

## Solution Assessment

Yes for the narrow repair: use the existing local auth-profile failure policy on the slug-generator helper and test that shared auth state is not updated. The open #65134 alternative may also solve it by removing the embedded runner from slug generation, but that broader direction needs maintainer choice.
