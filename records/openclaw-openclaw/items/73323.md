## Summary

Keep this issue open. Current main contains useful mitigations for pricing opt-out, Telegram transport fallback, polling liveness, and handshake/readiness diagnostics, but the fresh discussion and current code do not prove that the broader cross-subsystem Gateway degradation is fixed; pricing still runs from the initial runtime-services path when enabled, remote pricing fetches still use 60s timeouts, and the channel stop-timeout path still preserves the blocked task state described in the Telegram restart failures.

## What This Changes

Investigate and fix chronic Gateway runtime degradation across pricing fetches, Telegram polling/send operations, and slow Gateway RPCs on Windows 11 with Node 24/25.

## Best Possible Solution

Keep this as the canonical broad Gateway runtime-degradation tracker until maintainers validate current main or the next release on an affected long-lived Windows Gateway, land or replace the pricing-startup subset in #73486 or #75067, repair the stopChannel timeout recovery path tracked by #71412 or an equivalent fix, and then split any remaining process-wide RPC/event-loop degradation into a narrower follow-up only after it is isolated.

## Reproduction Assessment

Yes. The discussion provides high-confidence reproduction evidence from multiple affected hosts, repeated logs, direct-network counterexamples, and an A/B pricing-skip test; this review did not run a live Windows Gateway.

## Solution Assessment

No complete solution is currently in place. The current implementation has useful mitigations, but the narrowest maintainable path is staged: isolate pricing startup, fix channel stop-timeout recovery, then live-validate any remaining Gateway RPC/event-loop degradation.
