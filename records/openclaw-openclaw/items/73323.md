## Summary

Keep this issue open. Current main has useful adjacent mitigation for pricing opt-out, startup/liveness diagnostics, and Telegram transport cleanup, but I did not find source-backed proof that the reported chronic cross-subsystem Gateway degradation is fixed. Pricing still uses 60s OpenRouter/LiteLLM fetch timeouts, the refresh still starts from the initial runtime-services path, Telegram stall handling remains reactive, and the normal RPC slowdown symptoms are not addressed by a narrow pricing toggle alone. The discussion is fresh, detailed, and corroborated by multiple related reports and an open narrower pricing-startup PR (#73486).

## Best Possible Solution

Keep this open as the canonical broad Gateway runtime degradation tracker. Review, land, or supersede #73486 for the pricing-startup subset, then run a long-lived Windows 11 + Node 24/25 Gateway with Telegram enabled on current main or the next release. Compare `models.pricing.enabled` true and false, collect `diagnostics.stability`, event-loop delay/utilization, handle/socket counts, Undici/Telegram transport state, and timings for `models.list`, `doctor.memory.status`, `node.list`, and Telegram send/poll cycles. Split only if the pricing startup issue is fixed but the chronic runtime/RPC degradation still reproduces.
