## Summary

Keep this issue open. Current main has adjacent mitigations for pricing startup and model prewarm, but it does not have a source-backed fix for the broader Windows gateway degradation described here: repeated 60s pricing fetch timeouts, Telegram polling/send stalls, and slow normal gateway RPCs. The report is fresh, detailed, supported by two follow-up reproductions/comments, and not superseded by the pricing-only PR or the closed pricing-bootstrap issue.

## Best Possible Solution

Keep this open for gateway and Telegram maintainer triage. The best path is to reproduce or instrument a long-running Windows gateway with Telegram enabled, compare behavior with models.pricing.enabled false, collect event-loop delay, handle/socket counts, undici dispatcher state, and RPC timing during stalls, then fix the shared runtime/network degradation or split out a narrower canonical follow-up only after the remaining scope is proven. Treat #72027 and #53639 as adjacent pricing work, not a full resolution for this broader issue.
