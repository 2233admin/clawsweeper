## Summary

Keep open. Current main supports Claude CLI execution when `agentRuntime.id` explicitly selects `claude-cli`, but it still does not infer Claude CLI transport from `agents.defaults.cliBackends` alone for canonical `anthropic/*` PI or subagent dispatch. That remaining behavior is a maintainer-level transport and billing contract decision, not a conservative cleanup close.

## What This Changes

The issue asks OpenClaw to make PI, subagent, and related model dispatch honor configured Claude CLI backends so Anthropic OAuth runs use the same Claude CLI transport as chat instead of direct API.

## Best Possible Solution

Define one supported contract and make it consistent: either keep Claude CLI selection explicit through `agentRuntime.id: "claude-cli"` with docs/doctor diagnostics for `cliBackends`-only configs, or add a deliberate opt-in policy that lets eligible Anthropic PI/subagent/model paths route through a configured Claude CLI backend with regression coverage for auth forwarding, fallback behavior, and subagent spawn.

## Reproduction Assessment

Yes for the OpenClaw routing behavior. With canonical `anthropic/*`, `agents.defaults.cliBackends.claude-cli`, and no `agentRuntime.id`, current source leaves `cliExecutionProvider` as `anthropic` and falls through to embedded PI for native subagent execution; live Anthropic billing was not reproduced in this read-only pass.

## Solution Assessment

Unclear. Inferring CLI transport from `cliBackends` alone is not obviously the safest fix because current docs define `agentRuntime` as the executor-selection contract and `cliBackends` as backend configuration; a maintainer should choose explicit docs/doctor enforcement or an opt-in routing change.
