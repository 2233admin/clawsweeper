## Summary

Keep open. Current main has an opt-in loop detector and a critical global no-progress breaker, but the reported `session_status`/generic same-tool same-args path still warns first, remains disabled by default, and only blocks identical stable outcomes at the global threshold of 30 rather than the requested configurable low hard-abort threshold.

## What This Changes

The issue asks OpenClaw to hard-abort an agent run after a small configurable number of identical tool calls instead of only returning loop warnings.

## Best Possible Solution

Implement a conservative shared loop-detector fix: when `tools.loopDetection` is enabled, repeated identical same-tool/same-args calls with stable no-progress outcomes, including `session_status`, should escalate to a critical before-tool-call block at a documented configurable threshold. Keep the block in the shared agent tool hook, align schema/docs/tests, and avoid process-kill semantics unless maintainers explicitly choose a stronger policy.

## Reproduction Assessment

Yes. The issue logs show repeated `session_status` warnings at 20 and 30, and current tests/source provide a focused reproduction: enable loop detection, repeat a generic same-args tool with identical output, and observe warning-only behavior until the global threshold of 30.

## Solution Assessment

Yes, with one caveat. The best narrow fix belongs in `src/agents/tool-loop-detection.ts` plus the existing before-tool-call critical block path; changing default-on behavior or forcing a process/session kill is a separate product-policy decision that should be explicit.

## Repair Work Prompt

Fix the enabled loop-detection hard-abort gap for repeated identical tool calls. Current main warns for generic same-tool/same-args repeats and only blocks stable no-progress generic calls at `globalCircuitBreakerThreshold` 30; `session_status` is not a known poll tool. Implement a conservative shared fix in the agent loop detector so enabled repeated identical no-progress calls, including `session_status` with identical args/output, escalate to a `critical` before-tool-call block at a documented configurable threshold. Prefer using or extending existing threshold semantics (`warningThreshold`, `criticalThreshold`, `globalCircuitBreakerThreshold`) rather than adding broad process-kill behavior. Do not globally enable loop detection or force a default threshold of 5 unless the docs/schema/tests are updated deliberately and the false-positive risk is addressed. Add focused unit and before-tool-call e2e coverage showing: disabled detection still does nothing, warning still occurs below threshold, generic/session_status no-progress repeats block at the configured critical threshold, changing outputs do not block, and existing known poll/global breaker behavior remains intact. Update `docs/tools/loop-detection.md`, `docs/gateway/config-tools.md`, schema help/generated docs if config semantics change, and add a changelog entry only if the behavior is user-facing. Do not fold in OpenShell malformed command validation from #72373 unless explicitly scoped by a maintainer.
