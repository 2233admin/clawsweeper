## Summary

Keep open. Current main still has the reported reconnect flash, and this PR is useful net-new work, but the patch is not merge-ready because current auth/device revocation closes can arrive as WebSocket code/reason-only `4001` frames while the PR only clears the sticky prior-session state when a structured error detail is present.

## What This Changes

The PR adds a sticky Control UI `hasEverConnected` state across gateway host, Lit/view state, render gating, tests, and changelog so transient WebSocket reconnects keep webchat mounted while structured auth or pairing failures restore the credentials gate.

## Best Possible Solution

Keep the sticky prior-session direction, but make explicit auth/session revocation closes clear the sticky state immediately, either by mapping known `4001` auth/device close reasons in the UI client path or by propagating structured close details from the gateway client, with focused no-error close tests.

## Reproduction Assessment

Yes. The linked bug has concrete tab-refocus steps on current main, and the PR-specific defect is reproducible after a successful hello by emitting `onClose({ code: 4001, reason: "gateway auth changed", error: undefined })`, which leaves `hasEverConnected` true and suppresses the credentials gate.

## Solution Assessment

No, not as written. A sticky prior-hello flag is a maintainable fix for transient reconnect flashes, but the implementation must also treat current server auth/device revocation close frames as re-auth failures before it is the narrow safe solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clear sticky state on auth revocation closes:** `ui/src/ui/app-gateway.ts:408-409`
  - body: After a successful hello, `GatewayBrowserClient` passes server close frames to `onClose` with only code and reason. Current server revocation paths use `4001, "gateway auth changed"` and `4001, "device removed"`, so this branch never clears `hasEverConnected` for those explicit re-auth cases and keeps cached Control UI content mounted. Please treat those close code/reason pairs as auth failures or propagate structured details, then cover the no-error close case.
  - confidence: 0.86
