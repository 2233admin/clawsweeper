## Summary

Keep open. Current main still does not implement the requested native no-third-party E2EE browser chat transport. Browser chat remains authenticated Gateway WebSocket RPC with plaintext `chat.send`/`chat.history` payloads; WebRTC support is for provider-backed realtime voice/Talk; browser device identity is an auth/pairing mechanism, not a chat encryption layer. The prior owner-posted Codex review also kept this open, and I did not find a merged/shipped related item that supersedes that conclusion.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. A good path is to define an additive Gateway protocol for encrypted chat envelopes and key negotiation, browser-held per-device/per-agent key lifecycle, isolated session/channel routing, transcript/storage semantics, relay or NAT fallback behavior, Control UI/WebChat UX, docs, and regression tests. If maintainers want plugin-first delivery, first design the missing plugin/SDK seams so an external browser-native E2EE transport can integrate without bypassing Gateway pairing, auth scopes, session isolation, or transcript rules.
