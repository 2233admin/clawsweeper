## Summary

Keep open. Current main still does not implement the requested native no-third-party E2EE browser chat transport. The shipped browser/chat surfaces are authenticated Gateway WebSocket RPCs with plaintext `chat.send`/`chat.history` payloads; browser device identity signs the connection handshake, and WebRTC support is scoped to provider-backed Talk/realtime voice, not an encrypted browser chat layer.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. A good path is an additive Gateway protocol for encrypted chat envelopes and key negotiation, browser-held per-device/per-agent key lifecycle, isolated session/channel routing, transcript/storage semantics, NAT/relay fallback behavior, Control UI/WebChat UX, docs, and regression tests. If maintainers prefer plugin-first delivery, first define the missing core/plugin seams so an external browser-native E2EE transport can integrate without bypassing Gateway pairing, auth scopes, session isolation, or transcript rules.

## Risks / Open Questions

- Closing as implemented would conflate authenticated Gateway WebSocket chat, PWA/device pairing, and provider-backed Talk WebRTC with the requested browser-held E2EE chat transport.
- Closing as ClawHub now would skip unresolved core protocol and security design: encrypted envelopes, key lifecycle, per-agent/per-device routing, transcript semantics, auth scopes, and relay/fallback behavior.
- Related native chat/session-browser work appears to cover ordinary WebChat/session UX, not the E2EE/no-third-party browser transport requested here.
