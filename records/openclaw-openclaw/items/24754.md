## Summary

Keep this issue open. Current main has browser Control UI/WebChat, PWA/Web Push, WebRTC-backed Talk, and configurable DM session isolation, but it still lacks the requested browser-held E2EE chat transport: encrypted message envelopes, browser key negotiation, per-device/per-agent encrypted channels, and transcript/storage semantics for ciphertext.

## Best Possible Solution

Keep this open for maintainer-led product/security design. The best path is an additive Gateway/browser chat design covering encrypted envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, isolated routing, transcript/storage semantics, relay/fallback behavior, Control UI/WebChat UX, docs, and contract tests. If maintainers prefer plugin-first delivery, define the missing Gateway/plugin seams first so an external E2EE browser transport can integrate without bypassing core auth, routing, and transcript policy.
