## Summary

Keep this issue open. Current main still has authenticated Gateway WebSocket chat, browser device pairing/auth, WebChat/Control UI, PWA/Web Push, and provider-backed Talk/WebRTC voice, but not the requested native browser-held E2EE chat transport with per-agent/per-device encrypted channels and no third-party dependency. The prior maintainer Codex review in the issue discussion also kept it open, and the relevant chat/WebChat/Control UI/protocol paths have not changed since that review point.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. The best path is an additive Gateway/browser chat design covering encrypted envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, isolated routing, transcript/storage semantics, relay/fallback behavior, Control UI/WebChat UX, docs, and tests. If maintainers prefer plugin-first delivery, define the missing core/plugin seams first so an external E2EE browser transport can integrate with Gateway pairing, auth scopes, routing, and transcript policy.

## Risks / Open Questions

- Closing as implemented would conflate authenticated Gateway WebSocket chat and provider-backed Talk/WebRTC voice with the requested browser-held E2EE chat transport.
- Closing as ClawHub-only would skip unresolved core product/security architecture questions: encrypted chat envelopes, browser-held key negotiation, per-device and per-agent key lifecycle, transcript/storage rules, routing/auth scopes, and NAT/relay fallback behavior.
- Existing plugin surfaces support channels, HTTP routes, services, and realtime voice providers, but the requested first-party browser chat UX appears to need Gateway protocol and Control UI integration before a clean external implementation could prove the design without bypassing core auth/routing/transcript rules.
