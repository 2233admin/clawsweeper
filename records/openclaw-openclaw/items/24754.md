## Summary

Keep open. Current main still does not provide the requested native no-third-party E2EE browser chat transport. The shipped browser chat path is authenticated Gateway WebSocket RPC with plaintext chat payloads; WebRTC is scoped to provider-backed Talk/realtime voice, and browser device keys are used for connection identity/pairing rather than chat envelope encryption. The prior maintainer Codex review also explicitly kept this open, and I did not find current-main evidence that supersedes it.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. The likely path is an additive Gateway/browser chat protocol for encrypted envelopes and key negotiation, browser-held per-device/per-agent key lifecycle, isolated session/channel routing, transcript/storage semantics, NAT/relay fallback behavior, Control UI/WebChat UX, docs, and regression tests. If maintainers want plugin-first delivery, first define the missing core/plugin seams so an external E2EE browser transport can integrate without bypassing Gateway pairing, auth scopes, session isolation, or transcript rules.
