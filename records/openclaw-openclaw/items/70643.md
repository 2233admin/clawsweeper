## Risks / Open Questions

- A fix that locks only the final temp-file rename would still leave stale read/merge/write windows in replaceConfigFile and mutateConfigFile.
- The include-file mutation fast path in src/config/mutate.ts also writes config-adjacent files and needs an explicit lock-order decision so root-config locking does not introduce deadlocks or leave includes unprotected unexpectedly.
- Config writes have sensitive existing behavior around runtime snapshot refresh, SecretRef/env restoration, owner-display-secret persistence, backup rotation, destructive-write rejection, and last-known-good recovery; regression coverage should pin those interactions under the new lock.
- Lock timeout handling should fail clearly, avoid orphaned .lock sidecars, and preserve the existing re-entrant behavior for nested config writes.
