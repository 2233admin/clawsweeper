## Summary

Keep this PR open. Current main still does not implement cron procedural playbook memory, and the PR remains a plausible implementation candidate, but it needs revision before merge because the proposed hook is not on the real scheduled/startup/manual cron outcome paths and the recovery classifier gap from review still applies.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require revision before merge: move signal recording into a shared cron outcome helper used by scheduled ticks, startup catch-up, and manual `cron run`; persist and propagate `lastErrorKind`; add tests for those real paths plus concurrent write behavior; define privacy and retention semantics for the new local memory file; and consider splitting the autonomy roadmap doc from the runtime change.

## Risks / Open Questions

- Functional readiness: the proposed recorder is wired to `executeJob`, while scheduled ticks, startup catch-up, and manual cron runs use other current-main execution paths.
- Functional correctness: recovery counts can be dropped when the original failure was classified by `errorKind`, because current job state does not persist `lastErrorKind`.
- Privacy and retention: the PR adds durable local storage for job ids, job names, and raw error text without a settled retention or operator-data policy.
- Concurrency: current cron can run multiple due jobs in parallel via `cron.maxConcurrentRuns`; the PR's fresh file-backed load/save per signal could lose updates without serialization or shared state.
- Scope hygiene: the PR bundles a large autonomy roadmap doc with the runtime change, which maintainers may want split before focused review.
