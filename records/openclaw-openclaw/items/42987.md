## Summary

Keep this PR open. Current main still does not implement cron procedural playbook memory, and the PR is a plausible implementation candidate with concrete review gaps to address rather than obsolete cleanup.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to revise it so playbook signal recording lives in a shared cron outcome helper used by scheduled ticks, startup catch-up, and manual cron run; persist and propagate lastErrorKind; add tests for those real paths and concurrent writes; document privacy and retention semantics for the local memory file; and consider splitting the autonomy roadmap doc from the runtime change.

## Risks / Open Questions

- Functional readiness: the proposed recorder placement misses scheduled ticks, startup catch-up, and manual cron run paths that are active on current main.
- Functional correctness: recovery counts can be dropped when the original failure was classified by errorKind, because current cron job state does not persist that value.
- Privacy and retention: the PR adds durable local storage for job ids, job names, and raw error text, so maintainers should settle local data handling semantics before merge.
- Concurrency: current cron can run multiple due jobs in parallel via cron.maxConcurrentRuns, so per-signal file load/save needs serialization or a shared state strategy.
- Security review: the provided diff does not show CI, dependency, package publishing, or external code execution changes, but the new persistence surface should be reviewed as sensitive operational data handling.
