## Summary

Keep this PR open. Current main still does not implement cron procedural playbook memory, and the PR is not obsolete as implemented-on-main. It remains a plausible implementation candidate, but it needs a rebase and product decision because current main now has cron_changed and prompt-hook surfaces that may support a plugin/ClawHub implementation instead of a new core memory layer.

## Best Possible Solution

Keep this PR open for maintainer follow-up. Best path: rebase onto current main, split the autonomy roadmap docs from the runtime change if needed, decide whether procedural cron memory belongs in core or in a plugin using cron_changed plus prompt hooks, and if it stays core, record from the shared cron outcome path, persist lastErrorKind, serialize memory writes, add privacy/retention docs, and update tests to cover scheduled, startup catch-up, and manual runs.

## Risks / Open Questions

- Functional readiness: the proposed wiring only in executeJob would miss scheduled ticks, startup catch-up, and manual runs on current main.
- Functional correctness: recovery counts can be silently dropped unless lastErrorKind is persisted and propagated with lastError.
- Concurrency/persistence: current cron supports maxConcurrentRuns, so per-signal load/save needs serialization or another shared-state strategy.
- Privacy/data handling: durable memory of job ids, job names, and raw error text needs explicit local retention and redaction semantics.
- Product boundary: current main now has cron_changed and prompt hooks, so maintainers should decide whether this should be core or a plugin/ClawHub feature before merging.
