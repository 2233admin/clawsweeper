## Summary

Keep open. Current main still has a credible MiniMax Anthropic-compatible usage propagation gap: MiniMax API-key and portal providers route through `anthropic-messages`, but the Anthropic stream transport only copies native `input_tokens`/`output_tokens` usage fields into the assistant snapshot; downstream Usage page aggregation can normalize aliases only after a nonzero usage object reaches persisted session data.

## What This Changes

MiniMax Anthropic-compatible token usage should be captured and shown on the Control UI Usage page instead of persisting zero token counts while message counts still aggregate.

## Best Possible Solution

Fix usage capture at the Anthropic-compatible transport or immediate MiniMax provider wrapper boundary so MiniMax streaming responses produce a normalized, nonzero assistant usage snapshot before session persistence. Keep the Usage page transcript-backed rather than inferring token totals from message counts or provider identity.

## Reproduction Assessment

Yes. The issue includes concrete release, endpoint, raw nonzero usage, and zero persisted JSONL usage; source tracing shows that a missed transport usage snapshot would produce exactly that symptom while message counts still aggregate.

## Solution Assessment

Yes. A narrow transport or provider-wrapper normalization fix with regression coverage is the maintainable path; a UI-only workaround would hide the symptom while leaving session logs, status, and cost summaries wrong.

## Repair Work Prompt

Fix MiniMax Anthropic-compatible token usage propagation for the Control UI Usage page. Current MiniMax providers default to `anthropic-messages`, but `src/agents/anthropic-transport-stream.ts` only copies native Anthropic usage fields into assistant usage; MiniMax may provide OpenAI-style `prompt_tokens`, `completion_tokens`, and `total_tokens`, leaving persisted session usage at zero.
