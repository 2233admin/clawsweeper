## Summary

Keep open: current main still has the MiniMax Anthropic-compatible usage propagation gap, and PR #75136 is the active open implementation candidate that explicitly targets this issue. The source path matches the reporter's analysis: MiniMax routes through `anthropic-messages`, the transport still copies only native Anthropic usage keys from `message_start`/`message_delta`, and downstream aggregation can only preserve usage it receives.

## What This Changes

The issue reports that MiniMax Anthropic-compatible runs persist zero token usage, so Control UI Usage shows MiniMax token totals as 0 while message counts still increment.

## Best Possible Solution

Review, rebase if needed, and land PR #75136, or an equivalent narrow transport-boundary fix that normalizes MiniMax-compatible usage before assistant snapshots and session persistence. The Control UI should continue to consume transcript/session usage rather than infer token counts from provider identity or message count.

## Reproduction Assessment

Yes. The issue gives exact setup details and raw nonzero provider usage, and current main can be reproduced with a deterministic Anthropic SSE fixture containing `prompt_tokens`/`completion_tokens` or `message_stop` usage that the transport currently ignores.

## Solution Assessment

Yes. Normalizing compatible-provider usage at the Anthropic stream transport boundary is the narrow maintainable solution; a Control UI-only workaround would leave session logs, status summaries, and cost totals wrong.
