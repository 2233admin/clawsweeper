## Summary

Keep open. Current main still lets the Discord self-message guard fail open when bot identity lookup returns no id, and replay/followup paths still do not carry enough trusted Discord identity metadata to re-check own-bot messages. The newer /new cleanup clears queued state for reset sessions, but that only addresses one symptom after a contaminated message has already reached history or followup state. The related implementation PR #46338 is closed unmerged, so it is not a shipped fix.

## Best Possible Solution

Keep this issue open. Implement a Discord-owned fix that makes bot identity reliable or fail-closed, carries trusted Discord author/bot identity through pending history and queued followup metadata, re-applies own-bot filtering before replay/followup execution, and keeps the existing /new queue cleanup covered by regression tests for failed fetchUser("@me"), pending-history replay, and queued followups.

## Risks / Open Questions

- A fix must preserve intentional allowBots=true and allowBots="mentions" behavior for other bots while fail-closing OpenClaw's own Discord bot account.
- Replay filtering should use trusted Discord ids, not display labels, to avoid false positives from bot renames, relay bots, or PluralKit-style identity handling.
- If trusted identity is added to shared history/followup metadata, the change should be backwards-compatible for plugin SDK consumers.
