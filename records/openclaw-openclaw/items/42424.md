## Summary

Keep open. Current main still has the core Discord self-message gap: bot identity lookup can fail to an undefined `botUserId`, both live self-filters are conditional on that id, and pending history/followup replay still lacks trusted Discord author/bot identity for a second self-filter. The `/new` queue cleanup path is present, but that only addresses one symptom. The related implementation PR #46338 is closed unmerged, so it does not supersede this issue.

## Best Possible Solution

Keep this issue open. Implement a Discord-owned fix that makes bot identity reliable or fail-closed, carries trusted Discord author/bot identity through pending history and queued followup metadata, re-applies self-authored-message protection before replay/followup execution, and keeps the current `/new` queue cleanup covered by regression tests.

## Risks / Open Questions

- A fix needs to preserve intentional Discord `allowBots=true` and `allowBots="mentions"` behavior while still fail-closing OpenClaw's own bot account.
- Replay/followup filtering should use trusted Discord ids, not sender labels, to avoid false positives for relay bots and false negatives after bot renames.
- PluralKit relay handling is explicitly special-cased today, so the fix should not collapse PluralKit member identity into generic bot suppression.
