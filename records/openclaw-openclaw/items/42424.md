## Summary

Keep open. Current main still has the Discord self-message invariant split across conditional live-ingress checks, while bot identity lookup can still fail to an undefined id and replay/followup paths still lack trusted Discord author/bot identity for a second guard. The current /new reset cleanup does clear queues for reset sessions, but that only addresses one symptom after contamination has already reached history or followup state. The related implementation PR #46338 is closed unmerged, so it is not a canonical shipped fix.

## Best Possible Solution

Keep this issue open. Implement a Discord-owned fix that makes bot identity reliable or fail-closed, carries trusted Discord author/bot identity through pending history and queued followup metadata, re-applies own-bot filtering before replay/followup execution, and keeps the existing /new queue cleanup covered by regression tests.

## Risks / Open Questions

- A fix must preserve intentional `allowBots=true` and `allowBots="mentions"` behavior for other bots while fail-closing OpenClaw's own Discord bot account.
- Replay/followup filtering should use trusted Discord ids rather than display labels, so bot renames, relay bots, and PluralKit handling do not create false positives or false negatives.
- PluralKit is explicitly special-cased in current preflight behavior, so the Discord-owned fix should keep PluralKit identity semantics separate from generic bot suppression.
