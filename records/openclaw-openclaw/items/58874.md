## Summary

Keep PR #58874 open. Current main still hardcodes `sourceChannel` to the internal/webchat channel for subagent completion provenance, and the existing tests verify delivery routing but do not assert the `inputProvenance.sourceChannel` behavior this PR is trying to fix. The PR should not be closed as implemented or obsolete; it needs maintainer review and likely a small revision to use the normalized/resolved requester origin plus regression coverage.

## Best Possible Solution

Keep the PR open and revise it into a narrow, tested fix: set completion `sourceChannel` from the normalized/resolved requester origin (`targetRequesterOrigin?.channel ?? INTERNAL_MESSAGE_CHANNEL` or an equivalent local variable after fallback resolution), add regression coverage that asserts `inputProvenance.sourceChannel` for non-webchat and normalized requester origins, and explicitly decide whether descendant-wake provenance should remain internal or receive requester-origin threading.
