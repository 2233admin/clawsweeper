## Summary

Keep open: this is a protected maintainer/MEMBER PR and current main still has the fixed guard behavior, so cleanup should not close it. The latest diff targets the right runtime surface, but it introduces a P2 onboarding regression by lowering the shared guard constant that custom-provider onboarding uses as its default context window.

## What This Changes

The PR changes context-window guard thresholds to 4k/8k floors with 10%/20% derived thresholds, tracks reference tokens for capped runtime contexts, removes fixed embedded-runner guard overrides, and updates focused tests, local-model docs, and changelog text.

## Best Possible Solution

Keep the centralized dynamic guard resolver and removal of fixed runner overrides, but split custom-provider onboarding defaults from guard safety floors before merge. A small follow-up in this PR should preserve or explicitly choose an onboarding contextWindow default that leaves real prompt budget, add a regression test for newly added custom providers, and then rely on normal maintainer CI review.

## Reproduction Assessment

Yes. On current main, resolving an 8k model context and passing it to evaluateContextWindowGuard reproduces the original fixed-threshold block because the hard minimum is still 16k; the existing guard tests also assert that behavior. The PR-introduced onboarding regression is reproducible by following applyCustomApiConfig for a new custom provider after the constant is lowered to 4k.

## Solution Assessment

No, not as-is. Centralizing dynamic threshold resolution is the right narrow direction, but the patch should decouple the guard floor from custom-provider onboarding defaults so a threshold change does not create 4k default model configs with 4096 max output tokens.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep custom-provider defaults above the guard floor:** `src/agents/context-window-guard.ts:5`
  - body: Lowering `CONTEXT_WINDOW_HARD_MIN_TOKENS` to 4_000 also lowers `DEFAULT_CONTEXT_WINDOW` in `src/commands/onboard-custom-config.ts` because that file imports this constant. New custom-provider onboarding would now write `contextWindow: 4000` while `maxTokens` remains 4096, leaving effectively no prompt budget and contradicting the documented 200000 default; split the onboarding default/minimum from the guard floor or set it explicitly.
  - confidence: 0.88
