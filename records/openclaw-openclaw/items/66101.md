## Summary

Keep this PR open. Current main still does not implement the proposed main-specific host-level OpenClaw self-inspection exception, and the PR remains a focused implementation candidate, but its current patch changes a command-execution safety boundary and has blocking bypasses for mutating OpenClaw commands and platform/pipeline detection.

## What This Changes

The PR adds a main-agent host-level OpenClaw command exception in `src/agents/bash-tools.exec.ts` plus path tests for allowed read-only self-inspection and blocked mixed or mutating forms.

## Best Possible Solution

Define the policy for whether `main` should have this narrow full-trust self-inspection exception, then implement it with parser/segment-based detection, cross-platform OpenClaw binary recognition, a strict read-only argv allowlist, explicit rejection of all mutating flags and shell composition, and regression coverage for Linux paths, pipes, doctor flags, and memory status flags.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Detect OpenClaw commands after pipes:** `src/agents/bash-tools.exec.ts:178-181`
  - body: `MAIN_HOST_LEVEL_OPENCLAW_COMMAND` only treats start, `;`, newline, `&&`, and `||` as command boundaries. A command such as `echo ok | openclaw update` does not match the trigger, so `assertMainHostLevelOpenClawBoundary` is skipped and a mutating OpenClaw command can run under full/off.
  - confidence: 0.92
- **[P1] Match host OpenClaw paths across platforms:** `src/agents/bash-tools.exec.ts:178-185`
  - body: The absolute-path patterns are hard-coded to `/Users/...`, so Linux paths such as `/home/user/.npm-global/bin/openclaw` or `/home/user/.openclaw/scripts/ops-host-openclaw.sh` do not trigger the boundary. That lets main bypass the new restriction on Linux hosts.
  - confidence: 0.9
- **[P1] Reject mutating doctor modes:** `src/agents/bash-tools.exec.ts:219-222`
  - body: The doctor allowlist rejects only the exact `--fix` token. Current CLI registration also exposes `--repair`, `--force`, and `--generate-gateway-token`, with `--fix` as an alias for repair, so `main` can still run mutating doctor operations through this exception.
  - confidence: 0.96
- **[P1] Reject mutating memory status flags:** `src/agents/bash-tools.exec.ts:228-230`
  - body: The PR allows `openclaw memory status` with arbitrary trailing arguments. Current docs describe `memory status --fix` as repair work and `--index` as reindexing, so this read-only exception would permit mutating memory operations.
  - confidence: 0.95

## Security Review

Status: needs_attention

Summary: The diff touches a host command-execution boundary and currently permits concrete mutating OpenClaw commands and guard-bypass forms.

Concerns:

- **[high] Mutating doctor commands can pass the allowlist:** `src/agents/bash-tools.exec.ts:219`
  - body: The proposed doctor guard only blocks exact `--fix`, but current CLI behavior includes other mutating modes such as `--repair`, `--force`, and `--generate-gateway-token`. This weakens the new main-host safety boundary.
  - confidence: 0.96
- **[high] Mutating memory status flags can pass the allowlist:** `src/agents/bash-tools.exec.ts:228`
  - body: The proposed `memory status` allowance accepts arbitrary arguments, including documented mutating flags such as `--fix` and reindexing via `--index`.
  - confidence: 0.95
- **[high] Boundary trigger can be skipped by path and pipeline forms:** `src/agents/bash-tools.exec.ts:178`
  - body: The trigger regex does not match single-pipe command boundaries and hard-codes macOS `/Users` absolute paths, so Linux absolute OpenClaw invocations and pipe-right OpenClaw commands can avoid the new enforcement.
  - confidence: 0.9
