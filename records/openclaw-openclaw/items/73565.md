## Summary

Keep open: current main still has direct transcript `appendMessage` sinks that do not route through the existing redaction guard, and the provided GitHub context lists PR #73563 as the active open closing implementation candidate. Because this is security-sensitive and already paired with an open fix PR, it should stay open until that PR or an equivalent maintainer-approved fix lands.

## What This Changes

Session transcript append paths should pass persisted message payloads through the existing redaction guard before writing JSONL transcripts.

## Best Possible Solution

Review and land PR #73563 or an equivalent maintainer-approved patch that routes every scoped bare transcript append sink through `guardSessionManager` with config where applicable, while preserving idempotency, transcript update behavior, parentId continuity, and `logging.redactSensitive: "off"` semantics with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: send token-bearing text through the config transcript append, CLI text-turn transcript, gateway inject helper, or Codex mirror while transcript redaction is enabled, then inspect the JSONL transcript; current main shows those paths can persist via direct `appendMessage` without the redaction guard.

## Solution Assessment

Yes. Reusing `guardSessionManager` is the narrowest maintainable direction because it applies the existing before-write hook, transcript redaction, and tool-result guard infrastructure instead of adding a second transcript scrubber.
