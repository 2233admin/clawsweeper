## Summary

Keep this issue open. Current main still has transcript persistence paths that call `SessionManager.appendMessage` without `guardSessionManager`, and the open unmerged PR #73563 is the paired implementation candidate with closing syntax for this issue.

## What This Changes

The issue requests applying existing session transcript redaction guards to bare `appendMessage` persistence paths in core session management, CLI/ACP attempt execution, gateway injection, and the bundled Codex mirror.

## Best Possible Solution

Land or replace the paired implementation so every transcript append sink uses the existing guard/redaction path, threads config where available, and preserves parentId, idempotency, and transcript event behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction path exists: send token-bearing text through the config transcript append, CLI/ACP turn persistence, or gateway inject helper with transcript redaction enabled; current main writes through bare `SessionManager` while the guarded path is where redaction runs.

## Solution Assessment

Yes. Reusing `guardSessionManager` at the identified entry points is the narrowest maintainable direction because it applies the existing hook, redaction, and tool-result guard infrastructure instead of adding another scrubber.
