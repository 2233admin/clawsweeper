## Summary

Keep open. Current main already covers unique approval ID prefix lookup, but it still does not implement reply-only `/approve <decision>` handling. The PR remains useful, but the proposed extractor is not merge-ready because it can submit a truncated or wrong approval ID on an authorization path.

## What This Changes

The branch adds reply-to `/approve <decision>` parsing with ID extraction from `ReplyToBody`, updates exec and plugin approval prompt hints, and adds parser and forwarder tests.

## Best Possible Solution

Merge the reply-to approval UX after the extractor reads the canonical full approval ID line, trims the full value, and adds regression coverage for whitespace-containing exec IDs plus plugin approval messages whose title or description contains an earlier `ID:` line. Keep the existing main prefix lookup rather than adding a second resolver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Capture the canonical full ID line:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: `APPROVAL_ID_RE` captures only `\S+` from the first line named `ID:`. Approval IDs are generic non-empty strings, and plugin approval text can contain freeform lines before the canonical request ID, so reply-only approvals can submit a truncated or spoofed prefix and resolve the wrong pending request. Capture the canonical full ID line and trim it before resolving.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it changes approval authorization handling and can misresolve reply-extracted approval IDs.

Concerns:

- **[medium] Reply extraction can misroute approvals:** `src/auto-reply/reply/commands-approve.ts:40`
  - body: The new extractor uses the first non-space token from the first `ID:` line in the replied message. Because exec approval IDs are generic strings and plugin request text can contain earlier `ID:` lines, a reply-only approval can be routed to the wrong pending request.
  - confidence: 0.86
