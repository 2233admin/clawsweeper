## Summary

Keep this PR open. Current main does not implement the proposed protection: `resolveGatewayRuntimeConfig` still only rejects Tailscale Funnel unless auth mode is password, still permits Tailscale Serve on loopback with `auth.mode="none"`, and current tests explicitly assert that serve+none resolves successfully. The PR is a focused security fix candidate, but it needs rebase/update and should address the install-flow gap raised in review before landing.

## Best Possible Solution

Keep this PR open as the implementation candidate for the paired bug report. Rebase it onto current main, change the current serve+none success test into a rejection test, add or update install/doctor/service-repair validation so `gateway install` refuses the same configuration before writing a service, and keep the narrow runtime startup guard with regression coverage for serve+none, funnel+non-password, and serve+token/password/trusted-proxy allowed paths.
