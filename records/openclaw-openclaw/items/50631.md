## Summary

Keep this PR open. Current main still does not implement the PR’s intended guard: `gateway.auth.mode="none"` with `gateway.tailscale.mode="serve"` still resolves successfully, then auth.mode `none` accepts callers before Tailscale identity auth is considered. The same-author linked security issue remains open, so this PR is still an implementation candidate rather than obsolete cleanup.

## Best Possible Solution

Keep this PR open as the implementation candidate for the linked security issue. Rebase it onto current main, add the Serve plus explicit `auth.mode="none"` rejection with targeted tests, and decide whether install/doctor/service-repair validation should reject or repair that invalid remote-exposure config before writing a managed service. Preserve valid local loopback `none`, Serve with real auth or verified Tailscale identity, and Funnel’s password-only rule.
