## Summary

Keep this issue open. Current main still masks the OpenAI Codex device-pairing code in remote mode even though the helper has the real user code, and an open closing PR already targets the same bug.

## What This Changes

Request that `openclaw models auth login --provider openai-codex` surface the device-pairing code in SSH or remote terminal sessions.

## Best Possible Solution

Land #74258 or an equivalent narrow OpenAI plugin change that prints `Code: <userCode>` through the user-facing prompter note in remote mode, keeps persistent `runtime.log` free of the code, and updates regression coverage for both surfaces.

## Reproduction Assessment

Yes. The current source and existing unit test provide a high-confidence reproduction path: set `isRemote: true` for the OpenAI Codex device-code auth method and the prompter note is expected to show the masking placeholder instead of the real user code.

## Solution Assessment

Yes for the linked #74258 direction, not for current main. Surfacing the short-lived code in the interactive prompter note while keeping it out of `runtime.log` is the narrowest maintainable fix for the reported SSH terminal workflow.
