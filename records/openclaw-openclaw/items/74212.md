## Summary

Keep open. Current main still masks the OpenAI Codex device-pairing code in remote/SSH mode, the latest tagged release has the same behavior, and open PR #74258 already uses closing syntax for this exact bug.

## What This Changes

The issue asks OpenClaw's OpenAI Codex device-pairing auth flow to show the device code in SSH/remote terminals instead of replacing it with a local-device placeholder.

## Best Possible Solution

Land or replace the narrow OpenAI plugin fix that prints `Code: <userCode>` in the remote prompter note, keeps persistent runtime logs code-free, and updates the provider unit tests so both terminal visibility and log hygiene are enforced.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction is to run the OpenAI Codex device-code auth method with `isRemote: true` and a mocked `userCode`; current main sends the placeholder to `prompter.note` and never exposes the code in the SSH-visible prompt.

## Solution Assessment

Yes. The best fix is provider-local in `extensions/openai/openai-codex-provider.ts`: include `userCode` in the terminal note for remote mode while leaving `ctx.runtime.log(...)` URL-only; open PR #74258 matches that boundary.
