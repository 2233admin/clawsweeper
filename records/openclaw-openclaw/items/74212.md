## Summary

Keep this issue open. Current main still masks the OpenAI Codex device-pairing code on remote/SSH paths, and open PR #74258 already uses closing syntax for this bug, so the issue should stay open until that PR or an equivalent provider-local fix lands.

## What This Changes

The issue requests that `openclaw models auth login --provider openai-codex` surface the OpenAI Codex device-pairing code in SSH/remote sessions instead of replacing it with a placeholder.

## Best Possible Solution

Land PR #74258 or an equivalent provider-local patch that always includes `Code: <userCode>` in the `prompter.note` for the OpenAI Codex device-code flow, updates the regression test to require that terminal-visible code in remote mode, and keeps `runtime.log` limited to the verification URL.

## Reproduction Assessment

Yes. The source-level path is high confidence: SSH-related env marks the auth method as remote, the current provider code masks `userCode` in that branch, and the existing unit test explicitly asserts the masked note.

## Solution Assessment

Yes, the narrowest maintainable fix is in `extensions/openai/openai-codex-provider.ts` and its colocated test. Changing core remote detection or writing the code to persistent runtime logs would be broader than needed for the reported failure.
