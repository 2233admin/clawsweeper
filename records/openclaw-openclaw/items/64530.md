## Summary

Current main still does not implement the requested durable reopen/startup recovery seam. The code has session-file cleanup, replay-time pairing repair, live pending-tool guards, and restart-aborted resume logic, but none of those paths append idempotent parentId-valid synthetic recovery rows to an already-persisted transcript tail.

## What This Changes

The issue requests first-party reopen/startup transcript compensation that preserves existing rows and append-only repairs unresolved tail tool calls plus persisted orphan user tails.

## Best Possible Solution

Add a narrow startup/reopen transcript recovery seam that inspects only the active tail, preserves all existing rows, appends parentId-valid synthetic error toolResult rows for eligible unresolved tool calls, appends one explicit assistant recovery marker for persisted orphan-user tails, and proves idempotence with focused unit plus embedded-runner coverage.

## Repair Work Prompt

Implement the linked issue as a narrow OpenClaw runtime/session transcript repair. Add a startup/reopen seam that inspects only the active transcript tail, preserves all existing rows, and appends recovery rows through SessionManager or an equivalent parentId-aware API. It should append synthetic error `toolResult` rows for eligible unresolved tail tool calls, preserve partial real tool results, append one explicit assistant recovery marker for a persisted orphan user tail, be idempotent across repeated reopen/startup, and never re-execute tools or rewrite deep history. Keep the fix in core agent/session runtime code, not plugins. Coordinate with the existing live pending-tool guard and replay sanitizer so repair rows are not duplicated. Treat `stopReason: "error"` and `stopReason: "aborted"` conservatively because current code intentionally avoids synthesizing tool results for those cases; add explicit tests for the chosen policy. No changelog entry is required unless the maintainer wants this user-facing recovery behavior called out.
