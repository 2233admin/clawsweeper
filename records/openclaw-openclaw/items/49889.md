## Summary

Current main still has only partial, internal Telegram delivery observability. The Telegram lane code distinguishes preview-finalized, preview-retained, sent, and skipped internally, and preview-finalized now bridges to an internal message:sent hook, but there is still no public/log/operator delivery outcome taxonomy for sent vs preview-finalized vs preview-retained vs hook-canceled vs transport-failed vs intentional skip.

## Best Possible Solution

Keep this open and define an additive Telegram delivery-observability contract: expose a structured final delivery outcome for fresh sends, preview-finalized edits, preview-retained/ambiguous retention, hook-canceled skips, transport failures, and intentional skips; emit it consistently through internal hooks and plugin hooks or a companion delivery event; add dispatch/lane tests for each path; document the fields in Telegram and hook docs.
