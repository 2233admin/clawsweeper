## Summary

Keep open. Current main still lacks dangerous-tool denial for external channel auto-reply providers, but this PR is security-sensitive and the latest patch still has blocking policy-scope and runtime-authorization issues that need maintainer/security review before merge.

## What This Changes

The PR adds a default dangerous-tool deny list to message-provider filtering, passes group allow overrides into that filter, adds `exec` to the owner-only core tool-name constant, and expands related unit tests.

## Best Possible Solution

Revise the PR so dangerous filesystem and exec tools are denied for untrusted external channel auto-reply contexts while preserving internal/trusted providers, heartbeat/event runs, explicit operator policy semantics, and channel memory-flush read/write behavior. If `exec` owner-only hardening stays in scope, enforce it through `src/agents/tool-policy.ts` or exec tool metadata with runtime authorization tests, and review it alongside #64718.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope the default deny list to untrusted channel providers:** `src/agents/pi-tools.message-provider-policy.ts:37-39`
  - body: The new fallback applies `DEFAULT_DENIED_TOOLS` to every non-empty `messageProvider`. Current main uses internal provider values such as `webchat`, `heartbeat`, `cron-event`, and `exec-event`, so this strips `exec` and file tools from trusted/internal runs outside the stated external channel auto-reply threat model.
  - confidence: 0.9
- **[P1] Preserve memory flush tools for channel sessions:** `src/agents/pi-tools.message-provider-policy.ts:37-39`
  - body: Memory-triggered runs are reduced to `read` and `write` before this filter, and the memory runner carries the originating channel as the message provider. Denying both tools for Telegram/Discord/Slack-style providers leaves the flush with no usable tools, silently breaking channel memory persistence unless a separate config allow override happens to restore them.
  - confidence: 0.88
- **[P2] Enforce exec owner-only where runtime policy checks it:** `src/agents/tools/owner-only-tools.ts:1`
  - body: Adding `exec` to `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES` only updates the exported constant. Runtime filtering is driven by `src/agents/tool-policy.ts` fallbacks or `tool.ownerOnly`, and the exec tool sets neither, so non-owner runs that are not removed by message-provider filtering can still receive `exec`.
  - confidence: 0.92
