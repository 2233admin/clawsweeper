## Summary

Keep open: current main still has the post-challenge silent-failure shape described in the thread. Recent timeout, retry, and server-side handshake hardening helps other Gateway connection failures, but it does not surface synchronous `sendConnect()` frame-building/signing failures before the `connect` request is emitted.

## What This Changes

The issue reports Gateway-backed CLI commands hanging after receiving `connect.challenge` because the client never emits the signed `connect` reply.

## Best Possible Solution

Make the Gateway client fail fast and loudly when the post-challenge `connect` frame cannot be assembled or emitted: isolate JSON parse errors from event handling, wrap the `sendConnect()` frame construction/signature/public-key path, close/reset the socket cleanly, and propagate the exact connect error through CLI Gateway calls with focused regression coverage.

Implement a narrow fix in `src/gateway/client.ts` and, if needed for CLI propagation, `src/gateway/call.ts`: keep JSON parse errors scoped to JSON parsing/validation, wrap the `sendConnect()` frame construction/signing/public-key normalization/request-send path in explicit error handling, route failures through `onConnectError` with an actionable message, log them as connect failures rather than parse errors, and close/reset the socket so Gateway-backed CLI commands fail promptly instead of hanging until the outer timeout.
