## Summary

Keep open for maintainer review. Current main still has the reported gap in the native Codex conversation-binding path: `runBoundTurn` waits for collector completion after `turn/start` without checking whether the returned turn is already terminal, while the main Codex harness already handles that protocol shape. The PR is narrow, aligns with the existing collector semantics, and I found no blocking correctness or security finding in the diff.

## What This Changes

The PR routes terminal Codex `turn/start` responses through the existing bound conversation collector and adds a regression test for synchronous completion.

## Best Possible Solution

Land a narrow conversation-binding fix that treats terminal `turn/start` responses the same way as `turn/completed` notifications while preserving the existing dynamic-tool and approval boundaries, with the regression locked in under `extensions/codex/src/conversation-binding.test.ts`.

## Reproduction Assessment

Yes. Current main can be reproduced with a mocked Codex app-server where `turn/start` returns a `completed` turn and emits no later `turn/completed`; the bound path then waits until the collector timeout.

## Solution Assessment

Yes. Routing the terminal response through the existing collector completion path is the narrowest maintainable fix and matches the already-established main Codex harness pattern.
