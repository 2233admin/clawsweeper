## Summary

Keep open. Current main only strips historical OpenAI-compatible reasoning replay for Gemma 4 and selected provider-owned cases, while the reported Qwen/oMLX-style custom OpenAI-compatible path remains uncovered. The issue discussion gives a clear two-turn failure pattern, and the related fix PRs are closed unmerged.

## What This Changes

The issue reports that Qwen/oMLX OpenAI-compatible follow-up requests replay prior `reasoning_content` in conversation history, causing strict JSON parsing failures and empty responses after the first turn.

## Best Possible Solution

Add targeted replay-history hygiene for affected Qwen-style local or self-hosted OpenAI-compatible backends so historical provider reasoning blocks are removed or downgraded before follow-up requests, while preserving same-turn tool continuations and providers that explicitly require replayed reasoning.

## Solution Assessment

Yes for the repair direction. The narrow maintainable fix is replay policy/hygiene for affected Qwen-style OpenAI-compatible history, not generic JSON escaping or unconditional stripping across all providers.
