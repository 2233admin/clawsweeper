## Summary

Keep open. Current main has the useful #73580 timing/config mitigation for Baileys 408 and handshake timeout loops, but the reported startup failure class is still terminal when `waitForWaConnection` sees a pre-listener close: the controller closes the socket and rethrows, and the monitor only retries the special auth-unstable startup case.

## What This Changes

The issue asks for WhatsApp Web to remain connected after successful QR pairing instead of exiting on an immediate WebSocket ETIMEDOUT/408 startup close.

## Best Possible Solution

Keep the issue open and finish the WhatsApp startup reliability path: preserve the current `web.whatsapp.*` timing controls, then route retryable pre-listener 408/ETIMEDOUT closes through the existing reconnect policy with regression coverage and terminal-status safeguards.
