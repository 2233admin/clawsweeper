## Summary

Keep open. Current main still lacks Tavily as a web_fetch provider, and this PR is the live implementation attempt, but it should not merge yet because the provider returns the wrong payload shape for core web_fetch normalization and fetch-only config overrides leak into Tavily search.

## What This Changes

The PR registers Tavily as a bundled web_fetch provider, adds Tavily webFetch config and secretref/docs coverage, implements the provider through Tavily Extract, and extends Tavily tests.

## Best Possible Solution

Keep the feature in this PR, but make the runtime provider satisfy the existing web_fetch fallback contract by returning a top-level text payload and add an integration-style test through createWebFetchTool. Also split fetch-specific credential/base-url resolution from Tavily search so webFetch overrides do not alter web_search or tavily_search behavior.

## Reproduction Assessment

Yes. A focused static reproduction is to force the web_fetch provider fallback path and observe that core reads only payload.text, while the proposed Tavily provider returns results[].rawContent/content, producing an empty final text field.

## Solution Assessment

No. The product direction is reasonable, but the implementation should map Tavily Extract output into the existing web_fetch provider contract and use fetch-scoped resolvers instead of changing shared Tavily search resolution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Return a web_fetch text payload from the provider:** `extensions/tavily/src/tavily-fetch-provider.ts:33-50`
  - body: Core provider fallback normalization only reads the top-level `text` field from the provider payload before wrapping and returning the final `web_fetch` result. This provider returns the raw Tavily Extract shape (`results[].rawContent`/`content`), so selecting Tavily would produce an empty `text` response even when Tavily returned page content. Map the extracted content to the existing provider payload shape, and cover it through `createWebFetchTool`.
  - confidence: 0.91
- **[P2] Keep fetch overrides out of search resolution:** `extensions/tavily/src/config.ts:68-80`
  - body: `resolveTavilyApiKey` and `resolveTavilyBaseUrl` are also used by `runTavilySearch`, so preferring `webFetch.apiKey`/`webFetch.baseUrl` here makes a fetch-only override change Tavily web_search and tavily_search behavior. Add fetch-specific resolution for the new provider instead of changing the shared search resolver.
  - confidence: 0.87
