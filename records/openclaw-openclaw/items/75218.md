## Summary

Keep open. Current main does not yet provide Tavily as a native web_fetch provider, so the PR is useful, but the proposed provider does not satisfy the current web_fetch fallback payload contract and has a fetch-specific baseUrl override leaking into Tavily search routing.

## What This Changes

The PR registers a Tavily web_fetch provider, adds Tavily webFetch manifest/config/secret/docs surfaces, passes format into Tavily Extract, and extends Tavily tests.

## Best Possible Solution

Keep the provider in the Tavily plugin boundary, but have it return the web_fetch provider payload shape that core expects, especially a root text field derived from the selected Tavily Extract result, and add an integration test through createWebFetchTool. If webFetch.baseUrl is exposed, extraction should use that override without changing existing Tavily search calls.

## Reproduction Assessment

Yes. Configure Tavily as the web_fetch provider and force provider fallback, such as with readability disabled; current main will call the provider and then normalize only rawPayload.text, while this branch returns only results/rawContent/content.

## Solution Assessment

No. The plugin boundary and additive manifest/docs direction are reasonable, but the implementation must map Tavily Extract into the existing web_fetch fallback contract and isolate fetch-only config before it is merge-ready.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Return the text field expected by web_fetch:** `extensions/tavily/src/tavily-fetch-provider.ts:50`
  - body: The web_fetch fallback path does not return provider payloads directly; current main normalizes provider output by reading `payload.text`. This provider returns Tavily Extract's `results` with `rawContent`/`content`, so the final `web_fetch` response wraps an empty string even when Tavily extracted page content. Map the selected extract result to a root `text` field before returning and cover it through `createWebFetchTool`.
  - confidence: 0.94
- **[P2] Keep fetch baseUrl out of search calls:** `extensions/tavily/src/config.ts:79-80`
  - body: `resolveTavilyBaseUrl` is shared by both `runTavilySearch` and `runTavilyExtract`, but this change makes it prefer `webFetch.baseUrl`. Setting the documented fetch-only override would unexpectedly route Tavily search requests to the extract base URL. Use a separate extract base resolver or pass the fetch override only from the web_fetch/extract path.
  - confidence: 0.88
