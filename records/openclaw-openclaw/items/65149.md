## Summary

Keep this PR open. Current main still lacks the requested final-reply payload plugin hook or an equivalent final-delivery seam that can modify finalized payloads with resolved provider/model/usage context before the usage footer is appended. The PR is a plausible implementation candidate, but it changes public plugin API behavior and should get maintainer API review, docs, focused tests, and coordination with the related model-identity report before merge.

## What This Changes

This PR adds a `final_reply_payloads` plugin hook to the hook type registry and runner, then calls it from final reply assembly with resolved provider, model, response-usage mode, and usage-line context before appending the usage footer.

## Best Possible Solution

Finish a supported plugin API change that gives plugins stable final-delivery access to resolved provider/model and usage context, with documented hook semantics, targeted tests for usage-footer ordering and fallback-selected versus actual runtime model, and a deliberate decision on how it relates to #69110.
