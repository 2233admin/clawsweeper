## Summary

Keep this PR open. Current main still does not implement the timestamp plus last-message preview UI: the backend can return `lastMessagePreview` when explicitly requested, but the Control UI does not request it, does not type it locally, and does not render it in the Sessions table. The PR remains directionally useful for #47120, but the submitted diff also has unresolved review/security concerns, especially the unrelated package override and unbounded preview reads.

## Best Possible Solution

Keep the PR open as an implementation candidate for #47120, or resolve it only after a maintainer chooses and lands a canonical patch such as #48396. The best implementation should remove the unrelated package override, request `includeLastMessage` only for a bounded/capped Sessions list, add `lastMessagePreview` to the UI row type and session event field handling if needed, render a normalized/truncated preview with a clear timestamp treatment, and add focused controller/view tests for the request parameters and rendered preview.

## Risks / Open Questions

- Closing this PR as implemented would be incorrect because current main still does not request, type, or render session previews in Control UI.
- Merging this PR as posted would mix a small UI feature with an unexplained package-resolution override, increasing supply-chain/reproducibility risk and likely frozen-install risk if the lockfile is not updated.
- Always enabling `includeLastMessage` without a bounded/capped list can trigger transcript-tail file reads across every listed session when the Sessions limit is unbounded.
- Rendering transcript previews without normalization/truncation can create oversized rows and reduce Sessions table scanability.
