## Summary

Keep open. Current main still has a reachable Android/Termux sharp native-module failure path: sharp is still the media-understanding image backend, the lockfile has no Android sharp/libvips optional package, update/install paths remain generic, and resize/HEIC/web-media optimization still load the sharp-backed image ops when raw pass-through is not safe.

## What This Changes

The issue asks OpenClaw to stop breaking sharp-backed image processing on Android/Termux after updates, either through Termux-native rebuild repair, a native-module fix command, or graceful raw-image fallback.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image loading and sanitization: preserve original image inputs only when MIME, byte size, and header-derived dimensions prove they are already safe; otherwise omit the image or return a clear backend-unavailable error. Keep Termux-specific native rebuild automation as a separate packaging decision.
