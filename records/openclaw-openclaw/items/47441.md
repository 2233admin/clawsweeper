## Summary

Keep open. Current main still has a reachable Android/Termux sharp native-module failure path: sharp remains the shipped image-processing backend, sharp 0.34.5 still has no android-arm64 optional package, and OpenClaw's update/doctor runtime-dependency repair path is generic package staging rather than a Termux-native rebuild or graceful image fallback. The existing header-metadata fast path helps already-safe common image blocks, but resize-required images, HEIC conversion, and web-media optimization still depend on the sharp-backed backend.

## What This Changes

The issue requests an Android/Termux-safe fix for sharp image-module failures after OpenClaw updates, either through rebuild automation, a repair command, or graceful raw-image fallback.

## Best Possible Solution

Add a bounded fallback in the shared image loading and sanitization surface: when the sharp-backed backend is unavailable, preserve the original only for image inputs whose MIME, byte size, and header-derived dimensions prove they are already safe; otherwise return a clear backend-unavailable omission or error. Keep Termux-specific native rebuild automation as a separate maintainer-scoped packaging decision.
