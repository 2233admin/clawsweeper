## Summary

Keep open. Current main still has a reachable Android/Termux sharp failure path for image inputs that need resize, HEIC conversion, PNG alpha resizing, or web-media optimization; the newer header-metadata fast path only lets already-safe common images pass through without loading sharp.

## What This Changes

The issue asks OpenClaw to stop breaking sharp-backed image processing on Android/Termux after updates, via Termux-aware native repair, a fix-native-modules utility, or a graceful raw-image fallback.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image ingestion and sanitization: preserve original image inputs only when MIME, byte size, and header-derived dimensions prove they are already safe; otherwise omit the image or return a clear backend-unavailable error. Keep Termux-specific native rebuild automation as a separate packaging decision.
