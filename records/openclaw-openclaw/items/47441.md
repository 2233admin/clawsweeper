## Summary

Keep open. Current main still has a reachable Android/Termux sharp-unavailable path: the media-understanding runtime still ships sharp, the locked sharp dependency has no android-arm64 optional package, and resize/HEIC/PNG-alpha optimization paths still call the sharp-backed plugin backend outside the macOS sips path. The prior issue discussion already converged on the same narrow fallback repair while leaving Termux-native rebuild automation as broader packaging work.

## What This Changes

The issue requests Android/Termux-safe handling for sharp native-module failures after OpenClaw updates, either by Termux-aware rebuild tooling or by gracefully preserving safe raw images when sharp cannot load.

## Best Possible Solution

Add a bounded backend-unavailable fallback in shared image ingestion/sanitization: when the sharp-backed backend cannot load, preserve the original only for normalized image MIME types whose byte size and header-derived dimensions are already within provider limits; otherwise omit the image or return a clear backend-unavailable error. Keep Termux-specific native rebuild/update tooling as a separate packaging decision.

## Reproduction Assessment

Yes for the current-main application path: force the bundled sharp import/load to fail, then process a non-GIF web/local image that defaults to optimization, an oversized or over-dimension image block, HEIC input, or PNG alpha resizing. The exact Android/Termux device setup was not live-tested here.

## Solution Assessment

Yes for the narrow graceful fallback: it is the smallest maintainable repair for the user-visible image failure without taking on platform-specific toolchain rebuilding. Termux rebuild automation can still be considered later, but it should not block the shared image fallback.
