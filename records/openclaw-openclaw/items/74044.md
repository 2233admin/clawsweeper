## Summary

Keep this issue open. The report is fresh, specific, and still not proven fixed on current main; the current Discord code has adjacent startup/reconnect mitigations, but the default Discord path still enables voice and `GuildVoiceStates`, and there is no live Raspberry Pi arm64 validation with a real Discord bot token showing the gateway CPU wedge is gone.

## What This Changes

The issue reports that enabling the Discord channel on Raspberry Pi arm64 with OpenClaw 2026.4.26 wedges the gateway near 100% CPU after Discord provider startup, while disabling Discord restores gateway health.

## Best Possible Solution

Validate current main or the next build on the reported Raspberry Pi arm64 setup with `channels.discord.enabled=true` and a real Discord bot token. If it still wedges, keep the fix inside the Discord plugin startup, gateway lifecycle, or voice surface, add a targeted regression test for the narrowed failure mode, and document `channels.discord.voice.enabled=false` only as a confirmed workaround if live testing proves it avoids the spin.

## Reproduction Assessment

Yes for the reported 2026.4.26 setup because the issue gives concrete Pi arm64, Node, Discord-token startup, probe, CPU, and Discord-disabled recovery checks; unclear for current main because reproducing it requires the same live hardware and Discord credentials.

## Solution Assessment
