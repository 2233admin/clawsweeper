## Summary

Keep this PR open, but the submitted patch should not merge as-is. Current main still has no launcher-level termination fix for the Windows hang reported in #74378, but an unconditional root-launcher `process.exit` creates a concrete stdout/stderr truncation risk for piped or redirected CLI output and needs a narrower strategy plus Windows validation.

## What This Changes

This PR adds a final `process.exit(process.exitCode ?? 0)` to `openclaw.mjs` after the launcher help or built-entry import path completes.

## Best Possible Solution

Fix the Windows lingering-process symptom at the launcher or CLI cleanup boundary without unconditionally hard-exiting before streams can drain, then add launcher coverage or documented smoke proof for Windows `version`/`status`, piped or redirected output, exit-code preservation, and foreground long-running commands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid forcing exit before CLI output can flush:** `openclaw.mjs:317`
  - body: Calling `process.exit(...)` at the launcher boundary applies to every completed CLI command. Node documents that `process.exit()` can terminate while stdout/stderr writes are still pending, so JSON/help/blob output that is piped or redirected can be truncated; use a narrower termination path or drain the streams before hard exit.
  - confidence: 0.86
