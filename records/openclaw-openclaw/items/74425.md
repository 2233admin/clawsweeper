## Summary

Keep this PR open, but do not merge the current patch as-is. The linked Windows regression is concrete, yet the patch adds a global hard exit at the launcher boundary, which conflicts with Node's documented stdout/stderr behavior and can truncate command output after otherwise successful CLI completion.

## What This Changes

The PR adds a final `process.exit(process.exitCode ?? 0)` call to `openclaw.mjs` after launcher help fast paths or the built CLI entry import completes.

## Best Possible Solution

Fix the Windows lingering-process symptom at the leaking handle, command cleanup boundary, or a carefully drained launcher termination path while preserving `process.exitCode`; add proof for Windows `version` and `status`, piped or redirected output, and an intentional foreground command such as `gateway`.

## Reproduction Assessment

Unclear. The linked regression provides concrete Windows npm-installed steps for `version` and `status`, but this read-only Linux review could only verify the current code and test contracts, not reproduce the Windows handle leak live.

## Solution Assessment

No. A global `process.exit()` after every launcher path is broader than the observable bug and conflicts with Node's stdout/stderr flush guidance; a narrower cleanup or explicit stream-drain path is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Drain output before forcing launcher exit:** `openclaw.mjs:317`
  - body: Calling `process.exit(...)` at the launcher boundary applies to every completed CLI command. Node documents that this can terminate while stdout/stderr writes are still pending, so help, JSON, or other piped output can be truncated; use a narrower termination path or explicitly drain the streams before a hard exit.
  - confidence: 0.86
