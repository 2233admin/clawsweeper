## Summary

Keep this PR open. It targets a real open Windows CLI hang report and current main still falls through after the root launcher imports the built entry, but the proposed unconditional launcher-level exit likely regresses the documented root `--version` / `-v` fast path because that path is started asynchronously inside `dist/entry.js` and is not awaited before the outer wrapper would exit.

## What This Changes

The PR adds an unconditional final `process.exit(process.exitCode ?? 0)` to root `openclaw.mjs` after the launcher help or built-entry import path completes.

## Best Possible Solution

Revise the launcher-boundary termination fix so it only forces exit after all entry-owned async fast paths and CLI cleanup have settled, then land it with focused launcher/version coverage plus Windows npm-installed smoke proof for the reported `status` and version command hangs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Await the version fast path before forcing launcher exit:** `openclaw.mjs:244`
  - body: `dist/entry.js` handles root `--version` by calling `tryHandleRootVersionFastPath(process.argv)` without awaiting it; that helper prints and exits later from `resolveVersion().then(...)`. With this new unconditional outer `process.exit()` after the import, the wrapper can terminate before the version promise writes output, regressing `openclaw --version` / `-v`. Make the fast path awaited or keep the forced exit behind an entry result that confirms async fast paths are done.
  - confidence: 0.9
