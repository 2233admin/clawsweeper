## Summary

Keep open. Current main still lacks the llama.cpp runtime-context discovery that #73664 needs, and this PR is the focused implementation path, but the latest diff is not ready because it reads the wrong `/props` shape for current llama.cpp. The earlier Greptile index-based concern appears addressed by the latest `flatMap` rewrite, so the remaining blocker is the dependency-contract mismatch.

## What This Changes

The PR adds self-hosted OpenAI-compatible discovery of llama.cpp runtime context via `/props`, maps `/v1/models` `meta.n_ctx_train` to `contextWindow`, preserves explicit configured `contextWindow`, and expands provider setup tests.

## Best Possible Solution

Keep the PR open and revise the parser to read `default_generation_settings.n_ctx` from llama.cpp `/props`, optionally retaining a top-level `n_ctx` fallback if maintainers want tolerance for nonstandard servers. The regression test should use the documented nested `/props` response while preserving the existing explicit-config precedence and guarded-fetch behavior.

## Reproduction Assessment

Yes. A unit fixture matching current llama.cpp, with `/v1/models` returning `meta.n_ctx_train: 262144` and `/props` returning `{ default_generation_settings: { n_ctx: 65536 } }`, would leave this patch's `contextTokens` undefined because it reads only `data.n_ctx`.

## Solution Assessment

No. The feature direction is the right narrow fix for #73664, but the proposed implementation reads the wrong dependency field; the safer solution is to parse the documented nested runtime context field and update the test fixture accordingly.
