## Summary

Keep open. Current main has ACP task-record cleanup and closes completed oneshot ACP turns, but I did not find a lifecycle guard that closes or quarantines a persisted ACPX session when a parent-owned ACP task flow is already terminal, nor a repair path that prevents a stale persistent ACPX record from being resumed and routed back into a Telegram requester thread. The provided context also correctly separates this from closed #73202, which was about live npx adapter resolution rather than stale ACP session/task-flow delivery.

## Best Possible Solution

Keep this issue open and fix it in the ACP owner path: when a parent-owned ACP child task or mirrored task flow reaches a terminal state, reconcile the ACP session metadata with the ACPX persisted record, close or mark orphaned sessions non-resumable when no active binding/consumer remains, unbind or quarantine stale channel bindings, and add doctor/tasks audit repair plus regression coverage for Telegram requester delivery not being hijacked by a stale ACPX session.
