## Summary

Keep open. Current main and the latest shipped tag still reject the reported Anthropic `sk-ant-api03-...` value on the `models auth paste-token --provider anthropic` path, and the referenced fix PR is closed unmerged rather than a landed fix.

## What This Changes

The issue requests that Anthropic `paste-token` accept standard `sk-ant-api03-...` API keys, or clearly route API-key users to the separate Anthropic API-key auth path.

## Best Possible Solution

Widen the shared Anthropic token validator to accept both `sk-ant-oat01-` setup tokens and `sk-ant-api03-` API keys, add focused validator and command-path regression coverage, and update Anthropic token-path wording if needed so API-key users are not given a setup-token-only error.

## Reproduction Assessment

Yes. The current source gives a high-confidence reproduction without a real secret: an Anthropic `paste-token` value starting with `sk-ant-api03-` reaches a validator that only accepts `sk-ant-oat01-` and returns the reported error.

## Solution Assessment

Yes. A narrow validator/test update is the most maintainable fix if `paste-token` should store both Anthropic credential families; the safer alternative is an explicit command error that routes API keys to the existing `api-key` method if maintainers want `paste-token` to remain setup-token-only.
