## Summary

Keep open. Current `main` still rejects the reported Anthropic `sk-ant-api03-...` value on the `models auth paste-token --provider anthropic` path, the latest shipped tag has the same validator, and the linked fix PR is closed unmerged rather than landed.

## What This Changes

The issue requests that Anthropic paste-token authentication accept standard `sk-ant-api03-...` API keys or clearly route those users to a separate API-key auth path.

## Best Possible Solution

Widen the Anthropic paste-token validation and user-facing message to accept both `sk-ant-oat01-` setup tokens and `sk-ant-api03-` API keys, with focused validator and command-path regression coverage. If maintainers intentionally want separate flows, the end state should be an actionable rejection that points API-key users to the existing Anthropic API-key auth method instead of the current misleading prefix error.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction without a real secret: an Anthropic paste-token value starting with `sk-ant-api03-` reaches a validator that only accepts `sk-ant-oat01-` and returns the reported error.

## Solution Assessment

Yes, a narrow validator/message update plus tests is the best current fix if paste-token remains a generic manual credential path. The safer alternate is explicit CLI redirection to the existing `api-key` method if maintainers decide API keys should not be stored through `paste-token`.
