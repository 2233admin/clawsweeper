## Summary

Keep open. Current main still rejects the reported `sk-ant-api03-...` value on the `models auth paste-token --provider anthropic` path, the separate Anthropic API-key auth method does not fix that command’s failure behavior, and the linked implementation PR was closed unmerged.

## What This Changes

Allow `openclaw models auth paste-token --provider anthropic` to handle standard Anthropic `sk-ant-api03-...` API keys, or clearly route those users to the separate API-key auth path instead of failing with a setup-token prefix error.

## Best Possible Solution

Preserve the existing Anthropic setup-token path, then add explicit provider-aware handling for `sk-ant-api03-...`: either persist it as an Anthropic API-key profile with matching config mode, or fail with a clear instruction to use `openclaw models auth login --provider anthropic --method api-key`. Cover the validator/command path with focused regression tests and update the user-facing docs or changelog as appropriate.

## Repair Work Prompt

Current main rejects standard `sk-ant-api03-...` Anthropic API keys on `openclaw models auth paste-token --provider anthropic` because `validateAnthropicSetupToken` only accepts `sk-ant-oat01-`. Implement one narrow provider-aware fix: preserve existing `sk-ant-oat01-` setup-token behavior, and for `sk-ant-api03-...` either create/store a proper Anthropic `api_key` auth profile with matching `auth.profiles` mode, or return a clear provider-specific error that tells users to run `openclaw models auth login --provider anthropic --method api-key`. Do not silently store API keys as token-mode profiles unless you prove the Anthropic config-default/runtime semantics remain correct. Use #72124 as useful prior art, but re-evaluate its token-mode approach before copying it. Add regression coverage for the shared validator/classifier and the real `modelsAuthPasteTokenCommand` call path.
