## Summary

Keep open. Current `main` still sends Anthropic `paste-token` input through a setup-token-only validator, so a `sk-ant-api03-...` API key deterministically receives the reported `Expected token starting with sk-ant-oat01-` error. The repository already has a separate Anthropic API-key auth method, so the remaining fix is narrow and should not be treated as implemented or closed by the broader open relaxation PR.

## What This Changes

The issue reports that `openclaw models auth paste-token --provider anthropic` rejects standard Anthropic `sk-ant-api03-...` API keys and only accepts `sk-ant-oat01-...` setup tokens.

## Best Possible Solution

Keep the setup-token validator strict for `sk-ant-oat01-...`, but make the `paste-token` UX handle `sk-ant-api03-...` intentionally: either store it as an Anthropic `api_key` profile through the existing API-key helper or reject it with a direct command pointer to `openclaw models auth login --provider anthropic --method api-key --set-default`. Avoid broad length-only token acceptance.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: any non-empty Anthropic pasted value starting with `sk-ant-api03-` reaches the Anthropic setup-token validator in `modelsAuthPasteTokenCommand` and fails before profile storage.

## Solution Assessment

No, the current behavior is not the best way to solve the issue because it rejects a recognizable API-key family without using or pointing clearly to the existing API-key auth method. The safest fix is prefix-specific handling that preserves the token/API-key mode split.
