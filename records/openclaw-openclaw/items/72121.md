## Summary

Keep open. Current main and the latest shipped release still reject `sk-ant-api03-...` on the reported `models auth paste-token --provider anthropic` path because that path calls a validator that only accepts `sk-ant-oat01-`; the targeted implementation PR is closed unmerged. The bug is narrow and actionable, but the broader open #50975 approach should not be treated as the safest fix without maintainer review because it relaxes validation beyond the reported API-key prefix.

## What This Changes

Accept or clearly route standard Anthropic `sk-ant-api03-...` API keys when users run `openclaw models auth paste-token --provider anthropic`.

## Best Possible Solution

Update the shared Anthropic paste-token validator to handle the documented `sk-ant-api03-` API-key prefix with focused tests, or deliberately keep the token/API-key paths separate and make the CLI error/help text route API-key users to the existing Anthropic API-key auth method.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence without a real secret: an Anthropic paste-token value starting with `sk-ant-api03-` reaches `validateAnthropicSetupToken`, which only allows `sk-ant-oat01-` and returns the reported error.

## Solution Assessment

Yes, if `paste-token` remains a generic token paste path, a narrow accepted-prefix expansion plus regression tests is the smallest maintainable fix. If maintainers want semantic separation, the safer alternative is a targeted rejection that points `sk-ant-api03-...` users to the existing `api-key` auth method instead of the current setup-token-only message.
