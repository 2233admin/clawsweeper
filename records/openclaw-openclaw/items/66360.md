## Summary

Keep open. Current main still does not provide a `session.maintenance` size or line cap for active transcript `.jsonl` files; the shipped `maxActiveTranscriptBytes` compaction guard is a useful partial mitigation, but it lives under agent compaction settings, requires `truncateAfterCompaction`, and is not the requested maintenance cleanup path for transcript files.

## What This Changes

Add first-class transcript `.jsonl` size or line maintenance controls so long-lived sessions cannot grow active transcript files without bound.

## Best Possible Solution

Add an explicit, documented maintenance path for active transcript `.jsonl` files, or make a maintainer decision that the shipped compaction byte guard is the supported answer and update this issue/docs accordingly. If implemented, it should preserve session integrity, avoid reusing deprecated `rotateBytes`, keep defaults non-destructive, and include schema/help/docs/tests for the new config surface.
