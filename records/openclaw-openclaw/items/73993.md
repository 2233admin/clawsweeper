## Summary

Keep open. Current main still has the single accepted-offset Telegram polling behavior that the linked silent-drop report is about, while this PR is a relevant active implementation that adds a fetched-offset durability barrier and a separate completed watermark; I did not find a concrete blocking correctness or security issue in the diff.

## What This Changes

The PR changes Telegram polling to persist fetched/accepted update offsets separately from completed handler watermarks, migrates the offset store to a v3 snapshot, threads completed-offset callbacks through the bot/tracker, adds restart recovery logging, and covers the behavior with Telegram polling/store/tracker tests plus a changelog entry.

## Best Possible Solution

Review and land this PR or an equivalent Telegram plugin-local durability fix that stores fetched/accepted offsets separately from completed-safe handler progress, restarts from the completed watermark, preserves duplicate/replay guards, and keeps regression coverage for interrupted two-update polling batches. The Greptile monotonicity note is a reasonable defensive polish item, but the current tracker call path already requests completed watermarks monotonically.

## Reproduction Assessment

Yes for the restart durability race: current main source shows accepted offsets are persisted before handler completion and restored as a skip watermark, while grammY runner advances its fetch offset after getUpdates. The no-crash production variant is still unclear without runtime logs or a narrower failing sequence.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix for the confirmed race because it stays inside the Telegram plugin, adds a durability barrier at getUpdates, and separates completed handler progress from accepted/fetched offsets rather than rewriting the core channel loop.
