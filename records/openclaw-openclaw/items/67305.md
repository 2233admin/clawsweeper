## Summary

Keep this open. Current main still exposes only the existing public queue modes and the generic followup queue drains after debounce/quiet time, not after a session/run readiness signal. The subagent announce queue has useful busy-parent deferral prior art, but that remains an internal announce path rather than an operator-selectable `messages.queue` mode for ordinary chat/status completion delivery.

## What This Changes

The issue requests an operator-selectable readiness-aware queue collection mode that buffers status/completion notices until a session is ready and then flushes or coalesces them safely.

## Best Possible Solution

Add a documented public `messages.queue` mode or option that waits on a generic session/run readiness signal, coalesces eligible queued notices into the next safe response slot, and defines a clear fallback so buffered updates cannot starve indefinitely.
