## Summary

Keep open. Current main has related duplicate-message and blank-prompt guards, but the reported memory-flush plus compaction replay path is not clearly fixed: memory-triggered maintenance runs still flow through the same orphaned-user prompt merge path, and the existing dedupe windows do not cover the roughly nine-minute repeat described here.

## What This Changes

Reports that a single user message can be replayed as a fresh delivery after memory flush and hook-timed compaction.

## Best Possible Solution

Add a regression-backed guard at the memory/maintenance compaction boundary so hook-triggered or memory-triggered runs cannot merge or enqueue the prior user prompt as a fresh user turn, while preserving legitimate active-run user follow-ups and provider role-order repair for real user/manual turns.

## Reproduction Assessment

Unclear for the exact production incident: the report gives a concrete timeline, but the referenced session JSONL is private. There is enough current-main code evidence for a focused synthetic regression using a memory-triggered embedded run with a trailing user leaf and compaction/runtime context.

## Solution Assessment

No, current main does not yet look like the best solution. The safer fix is a narrow trigger-aware idempotency guard around orphaned-user repair or memory-maintenance prompt submission, rather than widening generic prompt dedupe windows across all channels.
