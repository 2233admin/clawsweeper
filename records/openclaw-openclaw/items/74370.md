## Summary

Keep open. Current main still has the memory-flush, compaction-hook, transcript-rotation, and follow-up queue surfaces described by the report, but I did not find a current-main guard that clearly prevents a hook-triggered compaction from surfacing the prior user prompt as a fresh turn several minutes later. Existing duplicate protections are narrower than the reported path: compaction duplicate pruning is time-windowed, queue/inbound dedupe depends on provider message ids or active queue state, and the memory flush run is silent but does not provide cross-boundary prompt idempotency.

## What This Changes

The issue reports that one user message was re-delivered as a new turn after memory flush and hook-timed compaction completed.

## Best Possible Solution

Add a regression-backed idempotency guard at the memory flush, compaction, and follow-up/session boundary so maintenance or hook-triggered compaction cannot re-submit the previous visible user prompt as a new turn, while preserving legitimate queued follow-ups, provider redeliveries with the same message id, and session rotation.

## Reproduction Assessment

Unclear: the issue gives a concrete timeline and likely code path, but the referenced session JSONL is private and not available in the checkout. A high-confidence reproduction should be built as a synthetic test that runs a silent memory flush, compaction/hook event, session rotation, and delayed same-content prompt delivery.

## Solution Assessment

Yes, a narrow idempotency fix in the memory-flush/compaction/follow-up boundary is the best current direction. Broad compaction-policy changes or extending the 60 second transcript duplicate window alone would be riskier because they could suppress legitimate repeated user requests.
