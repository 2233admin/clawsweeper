## Summary

Keep open. Current main still has global cron management for the reported surfaces: jobs can carry agent/session metadata and cron.add stamps caller metadata, but cron.list, cron.update, cron.remove, cron.run, and cron.runs do not receive or enforce trusted caller scope. The issue discussion includes concrete production reports and a recent maintainer comment keeping the ownership model open for follow-up.

## What This Changes

Request per-agent or trusted caller-scope cron isolation so agents in multi-user deployments cannot globally list, update, remove, run, or inspect other agents’ scheduled jobs.

## Best Possible Solution

Keep this issue open for a maintainer-designed cron ownership model. The likely path is to thread trusted caller agent/session scope from createCronTool or the Gateway request context into cron.list, cron.update, cron.remove, cron.run, and cron.runs; enforce default job visibility/mutation by owner; preserve an explicit admin/operator all-jobs mode; handle existing global jobs safely; and update docs/tests alongside the protocol/service/UI surfaces. Related issue #35447 should be considered during consolidation, but this issue has the richer production context and should not be closed now.
