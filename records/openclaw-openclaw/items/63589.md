## Summary

Keep open. Current main and the latest release still write only the basic WhatsApp inbound log payload; the participant and reply fields are computed for the queued inbound message but are not included in `inboundLogger.info()`. The linked implementation PR is useful source context, but it was closed unmerged and did not land this logger change.

## What This Changes

The issue requests richer WhatsApp gateway inbound log entries that include group participant identity and quoted-reply context fields.

## Best Possible Solution

Add the already-computed WhatsApp participant and quoted-reply scalar fields to the inbound logger payload, keep message delivery and mention gating unchanged, and add focused regression coverage for the log line itself.

## Reproduction Assessment

Yes. A unit test can emit a group `extendedTextMessage` with `contextInfo.quotedMessage` through the existing monitor inbox harness and assert the `inboundLogger.info()` payload contains `senderJid`, `senderE164`, `senderName`, and `replyTo*`; current main would fail that assertion because the log object lacks those keys.

## Solution Assessment

Yes. The narrowest maintainable fix is an additive enrichment of the existing `inboundLogger.info()` object from `inbound.participantJid`, `inbound.senderE164`, `msg.pushName`, and `enriched.replyContext`, without changing access control, group mention gating, delivery behavior, or raw message logging.
