## Summary

Keep open. The PR addresses the likely Anthropic 400 mechanism, but as written it removes the runtime-only transcript override and would make the full pre-compaction memory-flush directive a normal visible/persisted user prompt, regressing the shipped transcript-hygiene behavior for memory flushes.

## What This Changes

The PR removes the explicit empty `transcriptPrompt` from memory-flush runs, updates the memory-flush tests to expect it to be absent, and adds a changelog entry for the Anthropic 400 fix.

## Best Possible Solution

Keep the PR open and adjust the implementation so memory flushes submit provider-valid non-empty content without exposing the full housekeeping prompt as a visible user turn. Add focused regression coverage for both the Anthropic empty-message path and the prior transcript/chat.history leak.

## Reproduction Assessment

Yes. The source path on current main is high-confidence: memory flush passes `transcriptPrompt: ""`, runtime-context splitting turns that into `prompt: ""`, and the runtime-only branch submits that empty prompt to the active session.

## Solution Assessment

No. Removing the override is not the best fix because it trades the Anthropic 400 for a regression of the runtime-only transcript contract; the safer path is a narrow runtime-only submission fix or sanitized non-empty maintenance prompt that remains hidden from user-facing history.
