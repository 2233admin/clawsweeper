## Summary

Keep open. Current main still has the Feishu HTTP server cleanup pattern described here: `stopFeishuMonitorState()` is synchronous, calls `server.close()`, then immediately deletes or clears `httpServers`. The webhook abort cleanup path in `monitor.transport.ts` has the same immediate `server.close()`/delete pattern. The issue is also paired with open PR #48588, which uses GitHub closing syntax for this issue, so the right cleanup path is to review/land or replace that PR before closing.

## Best Possible Solution

Keep this issue open and review/land #48588, or replace it with an equivalent maintainer-owned PR that makes Feishu HTTP server shutdown awaitable in both `monitor.state.ts` and `monitor.transport.ts`, force-closes active connections where appropriate, adds timeout/error handling, and adds focused regression coverage. Close this issue only after that fix merges and ships or is otherwise verified on main.
