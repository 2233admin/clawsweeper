## Summary

Keep open. Current main has mitigated the fixed Control UI polling subproblem, but the core `sessions.list` scaling issue remains: the gateway still builds full session rows for the store before applying search, activeMinutes, and limit, and row construction still does per-row child-session resolution. Related PRs #58550 and #59436 are still open implementation paths rather than shipped fixes.

## Best Possible Solution

Keep #58534 open as the umbrella reliability issue until a maintainer lands a core sessions-list/store fix. The best path is to compare/finish #59436's early filtering and child-index approach with #58550's broader SQLite store proposal, add regression coverage proving `limit` and filters avoid work for non-returned rows, and keep #59317 closed as the already-fixed Control UI polling subproblem.
