## Summary

Keep this issue open. Current main still exposes default memory-core recall/read tooling, plus adjacent memory flush, promotion, Active Memory recall, and optional memory-lancedb store/forget tools, but it does not implement the requested built-in bounded memory CRUD surface with hard per-store character budgets, over-limit rejection with current usage/entries, substring replace/remove, duplicate handling for the default file-backed path, or credential-pattern screening. Related #70540 is still an open draft first slice for `memory_add`, not shipped on main and not a complete replacement for this request.

## Best Possible Solution

Keep this open for memory-core design and implementation. The best product direction is a structured model-facing memory write surface with config-backed per-store character budgets, atomic add/list/replace/remove behavior, duplicate detection, credential-pattern screening, and over-limit responses that include current usage plus entries for consolidation. If #70540 lands as `memory_add` only, extend it or split the remaining hard-budget CRUD work into a canonical follow-up before closing this issue.
