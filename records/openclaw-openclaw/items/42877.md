## Summary

Keep this issue open. Current main at 2ccdbc7dd93bfcb08e100a479af79d8d8a211d97 still provides memory recall/read tools, bootstrap injection caps, and append/promotion flows, but it does not implement the requested model-facing bounded memory CRUD tool with hard per-store character budgets, over-limit rejection with current entries/usage, substring replace/remove, duplicate detection, or credential-pattern screening. Related #70540 is an open draft and only covers a first memory_add slice, so it is not a shipped implementation or full superseding fix.

## Best Possible Solution

Keep this open for memory-core design and implementation. The best path is a structured model-facing memory write surface with config-backed per-store character budgets, atomic add/replace/remove/list semantics, duplicate detection, credential-pattern screening, and over-limit responses that include current usage and entries for consolidation. If hard limits are required, add a shared enforcement guard so generic file writes, memory flush, promotion, and dreaming cannot bypass the same budget. #70540 can be reviewed as a first slice, but it should not close this issue unless the remaining hard-budget CRUD behavior is implemented or split into a canonical follow-up.
