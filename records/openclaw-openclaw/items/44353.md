## Summary

Keep this issue open. Current main has fixed the broad provider-error fallback loop for configured fallback chains, and the Bedrock provider now discovers inference profiles, but the exact reported configuration uses `agents.defaults.subagents.model.fallbacks`. The current subagent spawn path still resolves only a model string and the later fallback override resolver does not read subagent-specific fallback lists, so the reported subagent-specific gap remains plausible.

## Best Possible Solution

Keep the issue open and fix the subagent path: when `agents.defaults.subagents.model` or `agents.list[].subagents.model` is an object with `fallbacks`, preserve that fallback list for the child session or teach `resolveEffectiveModelFallbacks` to resolve subagent-origin auto overrides. Add a focused regression test using the reported Bedrock-style primary plus a cross-provider fallback to prove the spawned subagent attempts the fallback after a provider-level error.
