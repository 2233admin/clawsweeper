## Summary

Keep this issue open. Current main fixes the broad provider-error fallback loop when a fallback chain is actually present, but the reported configuration stores fallbacks under `agents.defaults.subagents.model.fallbacks`, and current subagent spawn/fallback resolution still appears to drop that subagent-specific fallback list.

## Best Possible Solution

Keep the issue open and narrow it to subagent fallback propagation. The best fix is to make automatic subagent runs preserve fallbacks from `agents.defaults.subagents.model.fallbacks` and per-agent `agents.list[].subagents.model.fallbacks`, while keeping explicit `sessions_spawn.model` user overrides from unexpectedly inheriting fallback chains. Add focused regression coverage proving a spawned subagent with a Bedrock-style primary attempts the configured cross-provider fallback after a provider-level error.

## Clownfish Work Prompt

Fix the remaining subagent fallback propagation gap for this issue. Current main already falls back on generic provider errors when a candidate chain exists, but spawned subagents configured with `agents.defaults.subagents.model: { primary, fallbacks }` or per-agent `agents.list[].subagents.model: { primary, fallbacks }` resolve only the primary model string and lose the fallback list. Implement the fix in the agent/subagent model resolution boundary so automatic subagent model overrides carry the configured subagent fallback chain into `runWithModelFallback`. Preserve existing explicit user override semantics: an explicit `sessions_spawn.model` should remain a user override and must not unexpectedly inherit config fallbacks unless existing code already does so for user overrides. Prefer a small helper that resolves subagent model fallback overrides by target agent/config rather than hard-coding provider ids or Bedrock strings in core. Add regression tests for default subagent fallbacks and per-agent subagent fallbacks, including a Bedrock-style primary such as `amazon-bedrock/anthropic.claude-opus-4-5-20251101-v1:0` with `github-copilot/claude-sonnet-4` fallback, proving the fallback candidate is attempted after an unrecognized provider-level error. Do not broaden provider classifiers unless needed for the test; the main bug is the missing candidate chain. Update `docs/tools/subagents.md` only if the behavior text needs clarification. No changelog entry is required unless maintainers want user-facing release notes for the fallback fix.
