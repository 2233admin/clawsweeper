## Summary

Keep this issue open. Current main has the generic provider-error fallback loop and Bedrock provider hooks/discovery, but the reported subagent config path still drops `agents.defaults.subagents.model.fallbacks` and `agents.list[].subagents.model.fallbacks` before `runWithModelFallback()` builds candidates, so the remaining issue is subagent fallback propagation rather than provider-error classification.

## What This Changes

Configured model fallbacks should be attempted when a Bedrock-primary subagent run hits a provider-level error instead of terminating after the primary failure.

## Best Possible Solution

Keep the issue open and narrow the fix to subagent fallback propagation: preserve fallback arrays from `agents.defaults.subagents.model.fallbacks` and `agents.list[].subagents.model.fallbacks` for automatic native subagent runs, pass them into the model fallback runner, and keep explicit `sessions_spawn.model` overrides strict. Coordinate with #72984 because it touches the same subagent model-resolution surface.
