## Summary

Keep this issue open. Current main still resolves Discord routing from static binding inputs before mention detection, so a Discord thread that inherits a parentPeer binding can still route to the bound parent-channel agent even when the message explicitly mentions another agent/bot identity. The existing ClawSweeper comment is still accurate on current main; related #13304 covers broader content-based routing and is not a canonical replacement for this Discord mention override request.

## Best Possible Solution

Keep this issue open. A maintainer should define explicit precedence for Discord bot mentions and per-agent mentionPatterns, then implement mention-based route selection in the Discord preflight/route-resolution path before mention gating returns the effective route. Add regression coverage for a Discord thread inheriting a parentPeer binding while mentioning another configured agent, and update the channel-routing, Discord, and multi-agent docs with the final ordering.

## Risks / Open Questions

- Mention override precedence is still a product decision: it needs to be defined against /focus runtime thread bindings, configured ACP/channel bindings, static peer/parentPeer bindings, and default fallback.
- Native Discord mentions identify bot user IDs, while per-agent routing also depends on OpenClaw agent IDs, Discord account identity, and text mentionPatterns; the mapping must be explicit before implementation.
- A fix should preserve existing guild/channel access checks and mention-gating safety; mention-based routing should not become a bypass around authorization or group policy.
