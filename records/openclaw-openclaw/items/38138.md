## Summary

Keep this issue open. Current main still performs Discord route selection from static channel/account/peer/parentPeer inputs before mention detection, so a thread inheriting a parentPeer binding can still route to the parent-channel agent while explicit mentions and per-agent mentionPatterns only gate the already selected route. Related #35821 attempted this behavior but was closed unmerged, and broader #13304 is not a canonical replacement for this Discord-specific override.

## What This Changes

Add a Discord mention-based routing override so an explicit agent bot mention or per-agent mention pattern can select that agent ahead of static parent/channel bindings.

## Best Possible Solution

Keep this open for maintainer product/contract review. The likely fix belongs in the Discord-owned inbound route path or an additive generic routing seam: detect explicit target-agent mentions before mention gating, define precedence against /focus runtime bindings, configured ACP/channel bindings, parentPeer inheritance, and route-cache behavior, then add regression coverage for a Discord thread inheriting one bound agent while mentioning another configured agent. Update the channel routing and Discord docs once the contract is settled.
