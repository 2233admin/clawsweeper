## Summary

Keep open: current `main` and the latest release still route Slack outbound text through formatters that encode raw ampersands to `&amp;`, while the linked fix PRs were closed unmerged and the discussion still needs a safe exactly-one-escaping decision backed by live Slack rendering proof.

## What This Changes

The issue requests that Slack outbound messages containing `&` stop rendering as visible double-encoded text such as `&#38;`.

## Best Possible Solution

Keep the report open until Slack rendering is verified directly, then update the Slack-owned formatter, send, preview, and slash-response paths to produce one safe entity-escaping layer while preserving intentional Slack angle tokens and literal-entity safety.
