## Summary

Keep this issue open. Current main still sends Slack outbound text through helpers that escape raw ampersands to `&amp;`, the two linked fix PRs are closed unmerged, and the Slack tests still assert the current ampersand-escaping behavior rather than covering the reported visible `&#38;` regression.

## Best Possible Solution

Keep this issue open. Add focused Slack regression coverage for raw `R&D`/`B&A`, already-escaped `&amp;`, and numeric `&#38;` inputs across normal send, slash replies, preview final edits/readback, and tool-progress previews; then fix the duplicate-encoding boundary inside `extensions/slack` while preserving Slack's required single escaping pass.
