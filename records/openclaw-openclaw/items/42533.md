## Summary

Keep this issue open. Current main still matches the reported macOS speech-input failure mode: the PTT, Talk Mode, Voice Wake runtime, and Voice Wake tester paths all tap the hardware input format and append raw buffers directly to `SFSpeechAudioBufferRecognitionRequest`, with no shared mono/downmix or `nativeAudioFormat` conversion layer for multi-channel input devices.

## Best Possible Solution

Keep this issue open and fix the macOS app speech capture pipeline. Add a shared macOS voice-input helper that converts or downmixes multi-channel `AVAudioPCMBuffer` input from the hardware tap into a Speech-compatible mono/native format before appending it, wire it through `VoicePushToTalk`, `TalkModeRuntime`, `VoiceWakeRuntime`, and `VoiceWakeTester`, then add focused conversion coverage plus manual validation on a >2-channel macOS input device.
