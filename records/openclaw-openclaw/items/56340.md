## Summary

Keep this PR open. Current main already covers the Codex auto-WebSocket exclusion with a newer endpoint-aware guard, but it does not implement the PR’s central soft/hard before_tool_call blockMode contract or the multi-shell live .profile loading fallback. The PR is not directly merge-ready against current main because the WebSocket portion would need to preserve the newer baseUrl endpoint classification.

## Best Possible Solution

Do not close as implemented. Rebase this PR and keep only the still-missing pieces: the documented soft/hard before_tool_call blockMode contract, ownership-aware approval semantics, focused tests, and any still-needed live profile fallback hardening. Preserve current main’s endpoint-aware OpenAI WebSocket guard instead of reverting it to the older provider/api-only predicate.

## Risks / Open Questions

- The PR touches security-sensitive tool execution policy. Before landing, maintainers should re-verify that lower-priority plugins cannot convert another plugin’s soft block into an approval bypass and that hard blocks remain terminal by default.
- The PR also touches live test environment/profile loading. Review should confirm .profile sourcing remains scoped to explicit live-test flows and does not leak host credentials into normal CI/test runs.
- The WebSocket part is already solved differently on current main. A rebase must preserve the newer modelBaseUrl endpoint-classifier guard so custom, local, Azure, or ChatGPT backend endpoints do not drift onto the hardcoded public OpenAI WebSocket path.
