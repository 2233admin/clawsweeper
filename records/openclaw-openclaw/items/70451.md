## Summary

Keep open: current main still leaves the hooks CLI report path on full runtime plugin diagnostics, so the Windows hang described here remains plausible even after the broader CLI bootstrap mitigation for #64306.

## What This Changes

The issue reports that Windows `openclaw hooks` commands hang or are killed because hook command handling still reaches plugin registry/runtime loading.

## Best Possible Solution

Fix the hooks CLI to build hook reports from workspace/bundled/managed hook discovery plus plugin manifest or metadata-snapshot hook data, avoiding `buildPluginDiagnosticsReport` and full runtime module loading for `list`, `info`, `check`, `enable`, and `disable`.

## Repair Work Prompt

Fix the Windows hooks CLI hang reported here. `src/cli/hooks-cli.ts` currently builds hook reports by calling `buildPluginDiagnosticsReport`, which reaches `loadOpenClawPlugins` with `loadModules: true`; change `openclaw hooks list/info/check/enable/disable` and bare `hooks` to derive plugin-managed hook entries from manifest/metadata snapshot data or a bounded hook-specific metadata helper without importing plugin runtime modules. Preserve documented behavior: workspace hooks remain opt-in, bundled/managed hooks still appear, plugin-managed hooks appear as `plugin:<id>`, and plugin-managed hooks still cannot be enabled/disabled through the hooks CLI. Add focused regression coverage proving the hooks commands/report builder do not call the full diagnostics/runtime loader while still showing plugin-managed hooks. Do not change plugin runtime activation semantics, plugin install/update behavior, or global CLI preload policy. Add a changelog fix entry if the user-visible CLI behavior changes.
