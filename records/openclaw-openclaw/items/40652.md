## Summary

Keep open. Current `main` still only partially satisfies the requested recovery guidance: direct Gateway startup suggests `openclaw doctor --fix` but omits the `openclaw.json.bak` fallback, daemon start/restart preflight still points at generic `openclaw doctor`, and the related implementation PRs #40664, #40706, and #47127 are closed and unmerged.

## What This Changes

Gateway invalid-config startup errors should suggest `openclaw doctor --fix` and, if still blocked, recovery from `openclaw.json.bak`.

## Best Possible Solution

Centralize the invalid-config recovery hint used by direct Gateway startup validation and daemon lifecycle preflight, include `openclaw doctor --fix` plus a cautious `openclaw.json.bak` restore fallback, and cover both surfaces with focused tests without changing automatic rollback policy.

## Reproduction Assessment

Yes. A deterministic source-level reproduction exists: the direct startup recovery test expects the incomplete message, and daemon start/restart preflight constructs the generic doctor hint when `readConfigFileSnapshot()` returns invalid.

## Solution Assessment

Yes. The narrowest maintainable solution is a shared message/helper update with targeted assertions for direct startup and daemon preflight; this should remain a UX/error-text fix rather than expanding automatic config recovery behavior.
