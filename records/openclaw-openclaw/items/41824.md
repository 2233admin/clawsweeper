## Summary

Keep this issue open. Current main has improved the ACP/Telegram stream-completion path with final-only projection and a terminal done-event guard, and it has native embedded-runner handling for mutating tool failures, but it still does not implement the central requested safety property: a Telegram/ACP final answer that claims a file/message action completed is not checked against an observed completed mutating action before being delivered.

## What This Changes

The issue requests stronger Telegram/ACP harness behavior so agents cannot report file/message actions as complete when those actions did not execute.

## Best Possible Solution

Keep this open for maintainer design. The best path is a runtime/harness-boundary verified-action shape: record completed mutating tool/message/file events where OpenClaw can observe them, expose that evidence to a final-answer gate for ACP/Telegram, and start with an opt-in or narrowly scoped warning/revision/block behavior. Regression coverage should simulate a Telegram/ACP final reply claiming a file/message send with no matching completed action and assert that the user sees a correction, block, or explicit warning.
