## Summary

Keep open. Current main has materially better Telegram/ACP ordering safeguards than the reported 2026.3.8 build, including Telegram final-only handling, ACP final-only projection, terminal-done enforcement, and a Claude ACP completion fix. Those mitigations do not implement the central requested safety property: preventing or flagging a final Telegram/ACP answer that claims a file/message action completed when no corresponding tool/action actually ran.

## Best Possible Solution

Keep this issue open and decide a product-level verified-action shape before implementation. The likely fix belongs at the agent runtime/harness boundary, not only Telegram delivery: track completed mutating tool/message/file events, expose them to a final-answer gate for ACP/Telegram, and make the guard opt-in or narrowly scoped at first. Regression coverage should simulate a Telegram/ACP final reply claiming a file/message send with no matching completed action and prove the user sees a correction, block, or explicit warning.
