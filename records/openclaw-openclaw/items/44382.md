## Summary

Keep this PR open. Current main already shipped a narrower TUI readability fix for light terminals, but it does not implement this PR's central feature set: five named built-in TUI themes, a `--theme` CLI option, runtime `/theme`, a palette registry, or `setTheme()`/`getThemeName()` APIs. The PR also still needs maintainer review because the branch mixes the TUI feature with unrelated package-resolution/security-sensitive lockfile changes.

## Best Possible Solution

Keep this PR open as a feature candidate, but require a rebase and split before serious merge review. The TUI theme work should be isolated from package/security/protocol changes, aligned with the current `OPENCLAW_THEME=light|dark` implementation, and covered with focused tests for named theme selection, startup validation, runtime `/theme` behavior, editor/select-list/markdown recoloring, and legacy theme exports. If #38982 becomes the canonical picker implementation, use it to supersede only the overlapping picker flow, not the broader named-theme system.
