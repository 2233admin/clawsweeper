## Summary

Keep this PR open. Current main has newer launchd restart hardening, but it still does not implement the PR's central bounded EIO-only `launchctl bootstrap` retry; several bootstrap paths remain single-attempt. The PR is stale against current code and needs rebase/rescope, but it remains a focused launchd bug-fix candidate. The existing review discussion was useful: the test gap called out by Greptile was addressed in the PR head commit `082ddf0e0d77d3f4145675390c680d5eae34812b`.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase it onto current main and rescope around a shared bounded EIO-only `launchctl bootstrap` retry used by `bootstrapLaunchAgentOrThrow`, `repairLaunchAgentBootstrap`, install activation, restart fallback recovery, and detached handoff/update bootstrap paths. Separately decide whether PID SIGKILL escalation is still needed under the current kickstart-first model; if retained, add stronger PID identity protection and focused tests for retry success/exhaustion, errno 125 and non-EIO non-retry behavior, doctor repair, install activation, restart fallback, and shell handoff paths.

## Risks / Open Questions

- The PR is stale against current main because launchd restart behavior is now kickstart-first with newer restart/update hardening; applying it directly could regress newer behavior.
- The proposed SIGKILL escalation is correctness-sensitive. A liveness guard reduces risk, but PID reuse between the guard and kill remains a maintainer-review concern unless stronger process identity checks are added or escalation is dropped.
- A TypeScript-only retry helper would still miss generated shell paths such as detached launchd handoff and macOS update restart helpers unless those surfaces are updated too.
