## Summary

Keep open. Current main still leaves the bundled `memory-lancedb` plugin on vector-only LanceDB retrieval: it creates/opens a LanceDB table without an FTS/BM25 index, exposes no hybrid config, passes only embedding vectors into search, and has tests asserting `vectorSearch`. The existing BM25 + vector implementation is for the separate default memory engine, not LanceDB parity. PR #7636 attempted this exact feature but was closed unmerged, and the April 26 review comment already reached the same conclusion.

## Best Possible Solution

Keep this open as a focused bundled `memory-lancedb` parity enhancement. The best implementation belongs in `extensions/memory-lancedb`: add typed hybrid config and manifest/docs entries, create or maintain an FTS index on `text`, pass original query text through recall/forget/CLI/hooks and duplicate checks, use verified LanceDB hybrid search or a tested local fusion fallback, preserve vector-only fallback, and add targeted tests for config validation, FTS setup, exact keyword recall, fallback behavior, and query-text propagation.
