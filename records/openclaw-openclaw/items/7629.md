## Summary

Keep open. Current main still leaves the bundled memory-lancedb plugin on vector-only retrieval: table initialization creates/opens the LanceDB table without an FTS/BM25 index, the config and manifest expose no hybrid-search knobs, recall/search paths embed query text and pass only vectors into MemoryDB.search, and tests continue to assert vectorSearch. The separate default memory-core engine has BM25 + vector hybrid search, but that does not provide parity for users who select the LanceDB memory plugin. PR #7636 attempted this feature and is closed unmerged, so it is not a superseding implementation.

## Best Possible Solution

Keep this open as a focused bundled memory-lancedb parity enhancement. The best fix belongs in extensions/memory-lancedb: add typed hybrid config and manifest/docs entries, create or maintain an FTS index on text, pass normalized query text through recall/forget/CLI/auto-recall/duplicate paths, use the verified pinned LanceDB JS FTS/vector/rerank API or a tested local fusion fallback, preserve vector-only fallback, and add targeted tests for config validation, FTS setup, exact keyword recall, fallback behavior, and query-text propagation.

## Clownfish Work Prompt

Implement focused hybrid BM25 + vector search for the bundled memory-lancedb plugin.

Observable problem: current `extensions/memory-lancedb` embeds recall/search text and calls `table.vectorSearch(vector)` only, so exact keyword matches such as IDs, env vars, code symbols, and names are missed. The default memory-core path already has BM25 + vector hybrid retrieval, but users selecting `plugins.slots.memory = "memory-lancedb"` do not get that parity.

Expected boundary:
- Work primarily in `extensions/memory-lancedb` and `docs/plugins/memory-lancedb.md`.
- Add typed config, manifest schema, and UI hints for an optional `hybrid` block, likely including `enabled`, `vectorWeight`, `textWeight`, and any minimal fallback/reranker knobs needed after verifying LanceDB JS support.
- Verify the pinned `@lancedb/lancedb` 0.27.2 TypeScript API before coding. Do not assume Python `query_type="hybrid"` maps directly to JS. Use supported JS APIs such as FTS indexes, fullTextSearch/search fts, vector search, rerank, or a tested local fusion fallback.
- Create or maintain an idempotent FTS index on the `text` column when hybrid is enabled. Preserve startup/load behavior when FTS setup fails by logging and falling back to vector-only search.
- Thread the normalized original query text into MemoryDB.search for `memory_recall`, `memory_forget`, CLI `ltm search`, auto-recall, and duplicate checks as appropriate. Avoid changing memory-core behavior.
- Preserve current vector-only behavior when hybrid is disabled or unavailable, including existing minScore/duplicate-threshold semantics unless tests justify a narrow adjustment.
- Add tests for config validation/defaults, manifest schema, FTS index setup/failure fallback, query-text propagation, exact keyword/BM25 recall participation, vector-only fallback, and existing auto-recall/capture paths.
- Update user-facing docs and add a changelog entry under the active version because this is a user-facing enhancement. Credit the original reporter with `Thanks @joeykrug`; include any PR contributors as appropriate.

Do not change the @lancedb/lancedb dependency version, package manager metadata, memory-core hybrid behavior, or unrelated plugin/gateway surfaces without explicit maintainer approval.
