## Summary

Keep open. Current main resolves Gemini web_search SecretRefs correctly only when the caller's config matches the active runtime source snapshot; stale existing tool/session config can still fall back to the raw input config and pass an unresolved plugin-scoped SecretRef into the Gemini provider runtime.

## What This Changes

The issue asks for managed Gemini web_search execution to use the resolved active secrets runtime snapshot instead of raw plugin-scoped SecretRefs, including for existing sessions after reload.

## Best Possible Solution

Make managed web_search execution prefer the active resolved runtime snapshot for active Gateway/session tool calls, or otherwise re-resolve stale source configs before provider createTool runs, while keeping explicit isolated-config callers testable and documented by a narrow regression test.

## Reproduction Assessment

Yes. A focused unit test can activate a resolved runtime snapshot, then call runWebSearch with a stale/non-matching config containing plugins.entries.google.config.webSearch.apiKey as a SecretRef and assert the provider receives the resolved key rather than the raw ref.

## Solution Assessment

No current fix on main fully solves the issue. The narrow maintainable fix belongs in the generic web_search runtime/snapshot selection path, with a Gemini-shaped regression fixture only to prove plugin-scoped SecretRefs no longer leak to provider runtime code.
