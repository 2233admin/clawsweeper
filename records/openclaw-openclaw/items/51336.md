## Summary

Keep this issue open. Current main still formats generic overload and fallback rate-limit messages without provider/model context, even though the runner and payload paths already pass provider/model into `formatAssistantErrorText`. Open PR #55851 is an unmerged implementation candidate with closing syntax for this issue, so the issue should remain open until that PR lands or maintainers choose a narrower replacement.

## Best Possible Solution

Review and either land or close PR #55851. If maintainers close that PR, implement a narrower equivalent on current main: pass sanitized active provider/model into transient overload/rate-limit formatting, preserve existing provider-specific raw rate-limit detail handling, and update formatter, payload, lifecycle, and Telegram-facing coverage so generic 529/429 messages identify the upstream provider that actually failed.
