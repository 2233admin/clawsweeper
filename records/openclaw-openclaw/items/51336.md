## Summary

Current main still surfaces Anthropic overload and generic 429/rate-limit paths through provider-agnostic user-facing copy, even though the active provider/model is already available at the runner and formatter boundary. The issue should stay open because open PR #55851 is an unmerged implementation candidate with closing syntax for this issue.

## Best Possible Solution

Review and either land or close PR #55851, or implement a smaller equivalent on current main: pass sanitized active provider/model context into the transient overload/rate-limit formatter, preserve existing provider-specific raw rate-limit detail handling, and update formatter, payload, lifecycle, and Telegram-facing tests so generic 529/429 messages identify the upstream provider that actually failed.
