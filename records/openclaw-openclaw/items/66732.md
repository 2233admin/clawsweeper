## Summary

Current main still leaves the reported Telegram-visible false reminder note possible: the reminder guard appends the exact note when successful cron-add telemetry is zero, telemetry still only counts direct `cron.add` success, and the Telegram direct/media/payload send paths forward text without a note-specific suppressor. This is a valid narrow behavior bug rather than a close candidate.

## What This Changes

The issue asks OpenClaw to stop Telegram from showing a false unscheduled-reminder note after a reminder was actually scheduled.

## Best Possible Solution

Keep this issue open and fix the user-visible contradiction at the Telegram delivery boundary, or make reminder-success telemetry cover every supported successful reminder creation path before the guard runs. The guard should continue protecting normal unscheduled commitments on other channels.
