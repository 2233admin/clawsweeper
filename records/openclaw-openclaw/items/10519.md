## Summary

Keep this issue open. Current main still renders Telegram fenced code blocks with static `<pre><code>` tags, the shared Markdown IR drops the fence language before Telegram rendering, and the Telegram formatter test still asserts output without a `language-*` class. The related WebChat copy-button work is a separate UI renderer path and does not satisfy Telegram Bot API HTML output.

## Best Possible Solution

Keep this issue open and implement Telegram fenced-code language preservation. The narrowest maintainable path is to carry the first fence info token through the shared Markdown IR/render-aware chunking path, or add a Telegram-specific fence-aware renderer if maintainers want to avoid expanding the shared IR contract. The fix should emit `class="language-js"` for fenced blocks with a language, omit the class for indented or unlabeled blocks, sanitize the attribute, and preserve existing Telegram HTML escaping, file-reference wrapping, chunk splitting, and plain-text fallback behavior.
