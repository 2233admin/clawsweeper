## Summary

Keep open. Current main has not implemented this route-precedence change, and the PR addresses a real gateway misdelivery path from #68378, but the proposed same-channel target-replacement branch can still persist a mixed route by pairing a new `to` with the old session `accountId`.

## What This Changes

The PR changes gateway agent delivery-context merging so current request routes can override stale session routes, forwards the resolved target into `runContext.currentChannelId`, adds gateway regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the PR’s route-precedence direction, but make same-channel target replacement atomic: when `to` changes, persist only the request-supplied coupled route fields or clear omitted `accountId` and `threadId`, then add a regression for a target change with no `accountId`. The `currentChannelId` propagation can stay with that narrowed fix.

## Reproduction Assessment

Yes. A focused unit case can start with `deliveryContext: { channel: "discord", to: "channel:stale", accountId: "discord-old" }`, invoke `agent` with `deliver: true`, `channel: "discord"`, `to: "channel:fresh"`, and no `accountId`, then assert the persisted route does not retain `discord-old`.

## Solution Assessment

No. The PR solves the central stale-channel precedence bug, but the same-channel target replacement path is not the safest maintainable solution until it stops carrying old account metadata onto a new target.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear stale account when replacing a route target:** `src/gateway/server-methods/agent.ts:946`
  - body: When the request changes `to` on the same channel but omits `accountId`, this branch writes the new target with `sessionDelivery.accountId`. The immediate explicit turn avoids that fallback, but the session is persisted with a mixed route, so a later implicit delivery can send to the new target using the old account. When the target changes, `accountId` should come from the request or be cleared.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The PR has a concrete message-routing safety concern: a target replacement can persist a stale accountId with a new destination.

Concerns:

- **[medium] Mixed route can persist stale accountId:** `src/gateway/server-methods/agent.ts:946`
  - body: A same-channel request that changes `to` without providing `accountId` stores the new target with the old session accountId. In multi-account channel setups, that can route later replies through the wrong account or credential scope.
  - confidence: 0.86
