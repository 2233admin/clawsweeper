## Summary

Keep open. Current `main` still separates successful phase report/event writes from best-effort detached Dream Diary prose, and the reported rem-backfill/rollback symptoms map directly to current backfill date formatting and marker-normalization behavior.

## What This Changes

The issue reports that memory-core dreaming phase artifacts continued after April 26, 2026 while `DREAMS.md` stopped receiving timestamped Dream Diary entries, and that rem-backfill/rollback can leave inconsistent diary structure.

## Best Possible Solution

Keep this open and reconcile it with #72546 and open PR #70332. The desired end state is an idempotent Dream Diary migration/backfill/rollback path for legacy or partially marked `DREAMS.md`, plus clearer cron narrative failure handling so successful phase reports cannot silently imply a healthy diary when detached prose generation failed.

## Reproduction Assessment

Partly yes. The malformed rollback/header symptom and date-only backfill heading are high-confidence static reproductions from current code and tests; the stale multi-day cron diary symptom still needs live cron/subagent logs to prove the exact failure cause.

## Solution Assessment

No proposed patch is attached, and current `main` is not sufficient. The narrow maintainable fix is to make diary marker insertion legacy-aware, revisit backfill heading compatibility deliberately, and add focused tests before changing the broader detached cron narrative contract.
