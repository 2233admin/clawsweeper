## Summary

Keep open. Current main still has a credible split path where phase reports and completion events can be written while Dream Diary prose is detached best-effort work, and the reported rollback/header corruption maps to current marker-normalization behavior for a non-empty unmarked `DREAMS.md`.

## What This Changes

The issue reports that memory-core dreaming phase artifacts continue after April 26, 2026 while `DREAMS.md` stops advancing, and that `rem-backfill`/rollback can leave inconsistent Dream Diary formatting or managed-block structure.

## Best Possible Solution

The desired end state is an idempotent Dream Diary migration/backfill/rollback path for legacy or partially marked `DREAMS.md`, plus clearer cron narrative failure handling so phase reports cannot silently imply a healthy diary when detached prose generation fails.

## Reproduction Assessment

Partly yes. The rollback/header symptom has a high-confidence static reproduction path from `ensureDiarySection()` plus `removeBackfillDiaryEntries()` on a non-empty unmarked diary; the multi-day stale diary symptom needs live cron/subagent logs to prove the exact detached failure cause.

## Solution Assessment

No proposed patch is attached, and current main is not sufficient. The narrow maintainable solution should make diary marker insertion legacy-aware, add rollback/backfill regression coverage, and then decide whether cron narrative failures should become surfaced status or retry/backfill work.
