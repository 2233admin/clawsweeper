## Summary

Keep open. Current main still has a plausible split path where phase reports and completion events are written synchronously while Dream Diary prose is detached best-effort work, and the reported backfill/rollback symptoms map to current marker normalization and date-formatting behavior.

## What This Changes

The issue reports that memory-core dreaming continued writing phase reports while DREAMS.md stopped receiving dated diary entries, and that rem-backfill/rollback can leave malformed or inconsistent Dream Diary sections.

## Best Possible Solution

Keep the issue open and make the Dream Diary recovery path idempotent for legacy or partially marked DREAMS.md files, align backfilled diary headings with the intended public format, and make cron narrative failures visible or retryable enough that successful phase reports do not imply a healthy diary append.

## Reproduction Assessment

Partly yes. The rollback/header symptom has a high-confidence static reproduction from removeBackfillDiaryEntries on a non-empty unmarked DREAMS.md, while the multi-day stale diary symptom needs live cron/subagent logs to prove the exact failure mode.

## Solution Assessment

No proposed patch is attached, and current main is not sufficient. PR #70332 may cover adjacent hardening, but the safest solution needs explicit scoping across this report, #72546, and the open diary-hardening PR before an automated repair is queued.
