## Summary

Keep open for maintainer follow-up. Current main and the v2026.4.27 release contain strong fixes for the likely stale-diary cause: the Dream Diary subagent timeout was raised from 15s to 60s, and detached cron narratives are now concurrency-limited to avoid cascading lock contention and timeouts. But the report also includes recovery/backfill structure symptoms; current code still intentionally writes date-only grounded backfill headings, and I did not find exact coverage for repairing a duplicated empty Dream Diary managed-section block.

## What This Changes

The issue reports that memory-core dreaming can keep producing phase reports while `DREAMS.md` stops receiving Dream Diary entries, and that `rem-backfill`/rollback can leave diary formatting or managed-block structure inconsistent.

## Best Possible Solution

Confirm the reporter's install on v2026.4.27 or newer. If diary advancement is fixed after upgrade, close or split the stale-diary part against the shipped timeout/concurrency fixes; if the recovery symptom remains, add a narrow memory-core repair that canonicalizes duplicate Dream Diary managed sections and documents whether grounded backfill headings should remain date-only.

## Reproduction Assessment

Unclear. The historical reproduction is plausible and matches the already-fixed timeout/concurrency failure family, but I do not have a high-confidence current-main reproduction for the duplicated empty managed header block or for stale diary output after the v2026.4.27 fixes.

## Solution Assessment

Unclear. The current timeout and detached-concurrency fixes look like the right solution for stale diary writes after phase reports, but closing the entire report would be premature because the recovery/backfill structure portion is not fully covered by current tests or docs.
