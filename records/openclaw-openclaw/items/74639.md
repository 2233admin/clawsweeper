## Summary

Keep open. The current code still allows the reported observable split: phase reports and `memory.dream.completed` events are written before the Dream Diary narrative path runs, and the narrative path is best-effort and can return without appending `DREAMS.md` on timeout, error, or empty output. Current tests also intentionally keep grounded backfill headings date-only, so at least part of the recovery symptom is still a live product/format question rather than an implemented fix.

## What This Changes

The issue reports that memory-core dreaming continued writing phase reports while `DREAMS.md` stopped receiving timestamped Dream Diary entries, and that `memory rem-backfill`/rollback can leave malformed diary structure.

## Best Possible Solution

Keep the issue open and resolve it in memory-core by making Dream Diary lag observable and recoverable: either append a clearly marked deterministic fallback entry when phase reports are written but narrative generation fails, or surface the failed narrative state in status/doctor so users can repair it. The backfill/rollback path should also normalize duplicate or empty Dream Diary marker blocks while preserving real diary content, and docs/tests should state whether grounded backfill headings are intentionally date-only.

## Reproduction Assessment

Yes, partially. Current code and tests provide a high-confidence focused reproduction for phase reports advancing without a diary entry when the narrative subagent times out or returns no text; the exact reporter install cause still needs logs to distinguish timeout, model failure, empty response, or marker corruption.

## Solution Assessment

Unclear. The current implementation is intentionally best-effort for narrative writes and intentionally date-only for grounded backfill headings, so the best fix needs maintainer agreement on whether to add fallback diary entries, status/doctor repair, a heading-format change, or only marker normalization.
