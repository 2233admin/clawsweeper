## Summary

Keep this PR open. Current main already includes the basic Feishu tool-account precedence change, but it still lacks the PR's final review-driven behavior: only trust inherited session account context for Feishu-origin tool contexts. The remaining cross-channel account-id collision guard is plausible, scoped to the Feishu plugin, and worth review rather than closing as already implemented.

## Best Possible Solution

Keep this PR open for maintainer review or replacement. Rebase it narrowly onto current main, preserve the existing explicit account > enabled contextual Feishu account > configured default behavior, then add the missing `messageChannel` gate so contextual account IDs only override `channels.feishu.defaultAccount` for Feishu-origin tool contexts. Add targeted Feishu tests for non-Feishu fallback, Feishu-origin contextual routing, explicit overrides, unknown account fallback, and normalized account IDs.

## Risks / Open Questions

- Closing this PR as implemented would lose the remaining guard against non-Feishu sessions whose `agentAccountId` happens to match a Feishu account ID, which can route tool actions to the wrong Feishu tenant.
- The behavior affects data access scope across Feishu tenants, so the patch should keep focused regression coverage for explicit `accountId`, Feishu-origin contextual routing, non-Feishu fallback, unknown-account fallback, and normalized account IDs.
