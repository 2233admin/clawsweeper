## Summary

Keep open: current main still builds the Sessions checkpoint UI/API from persisted `compactionCheckpoints` metadata only, so historical sessions with valid checkpoint files but missing store metadata can still return an empty list. The closed related PR #68883 targeted the same gap and was closed unmerged for contributor workflow noise, not because the behavior landed.

## What This Changes

The issue asks the Web UI Sessions list checkpoint drawer to display historical compaction checkpoints that still have `.checkpoint.<uuid>.jsonl` files on disk even when the session-store checkpoint array is empty.

## Best Possible Solution

Add a gateway-side fallback or reindex path that discovers valid `{sessionId}.checkpoint.<uuid>.jsonl` files for the requested session when persisted checkpoint metadata is absent, exposes those entries consistently through list/get/branch/restore, and covers historical disk-only sessions with regression tests.
