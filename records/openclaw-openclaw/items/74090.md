## Summary

Keep open. Current main has the relevant overflow diagnostics, bounded auto-compaction, memory promotion preview/apply primitives, remote REM-harness preview, and session archive/delete support, but it still does not connect repeated overflow pressure to a deduped promote-then-rotate lifecycle suggestion or operator confirmation flow.

## What This Changes

The issue requests a lifecycle suggestion that detects repeated session context overflow, previews memory promotion candidates, and lets an operator confirm promote-and-rotate actions.

## Best Possible Solution

Approve a lifecycle design first: persist per-session overflow pressure, emit a deduped structured suggestion with a stable id and dry-run candidate count, expose it through operator-visible CLI/UI surfaces, and require explicit confirmation before either promotion or session rotation. Coordinate the memory-write side with #66287, #67363, and #72887 so the trigger does not bypass needed review or distillation safeguards.

## Reproduction Assessment

Yes. The existing overflow loop test provides a high-confidence simulation path for repeated overflow, and current source inspection shows the path compacts/retries or returns `context_overflow` without a promote-then-rotate suggestion; I did not execute tests because this review was required to stay read-only.

## Solution Assessment

Unclear pending maintainer design approval. The issue's opt-in suggestion and confirmation model is safer than automatic promotion/rotation, but the exact persistence window, stable correlation id, preview source, UI/CLI surfaces, and main-session rotation semantics should be settled before implementation.
