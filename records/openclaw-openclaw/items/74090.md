## Summary

Keep open. Current main contains the building blocks for overflow diagnostics, bounded compaction, memory-promotion preview/apply, pre-compaction memory flushing, transcript rotation after compaction, and session archive/delete, but it still does not wire repeated overflow pressure into a deduped promote-then-rotate lifecycle suggestion with dry-run candidates and an operator confirmation flow.

## What This Changes

Request to surface an operator-visible promote-then-rotate suggestion, with precomputed memory-promotion candidates and opt-in actions, when repeated session context overflow pressure is detected.

## Best Possible Solution

Approve a small lifecycle design that persists per-session overflow pressure, emits one deduped structured suggestion with a stable id and dry-run candidate count, surfaces it in operator-visible CLI/UI/log paths, and requires explicit confirmation before promotion or session rotation.

## Reproduction Assessment

Yes. The existing overflow loop test provides a high-confidence simulated reproduction of repeated overflow, and source inspection shows the current outcome is compaction/retry or a terminal context_overflow error without a promote-then-rotate suggestion.

## Solution Assessment

Unclear pending maintainer design approval. The issue's suggestion-and-confirmation model is safer than automatic promotion/rotation, but the state storage, main-session rotation behavior, memory review boundary, and UI actions should be decided before implementation.
