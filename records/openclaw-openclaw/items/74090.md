## Summary

Keep this open. Current main has the ingredients the issue calls out: overflow diagnostics and capped auto-compaction, memory promotion preview/apply, rem-harness preview plumbing, and session archive/delete primitives. It still does not connect repeated session overflow to a deduped promote-then-rotate lifecycle suggestion, operator notification, CLI prompt, dismissal state, or confirmation flow.

## Best Possible Solution

Keep this issue open and have maintainers approve the first lifecycle design: track repeated overflow per session, emit a deduped structured suggestion with a candidate count from a public memory preview seam, surface it in `openclaw sessions` and `openclaw memory status`, and require explicit operator confirmation before promotion or session rotation.
