## Summary

Keep open. Current main still has no post-report browser launcher/config/docs change that proves the reported v2026.4.26 Chrome namespace/sandbox startup failure is fixed, and the report says the failure persisted after configuring executablePath and noSandbox. The remaining decision depends on confirming whether this was native Windows, WSL2-to-Windows Chrome, or a Linux/container managed-launch path and what config actually resolved at runtime.

## What This Changes

The issue requests a fix for browser automation startup failing with Chrome namespace/sandbox errors on Windows/local and Ubuntu/container setups, blocking `openclaw browser --browser-profile openclaw start`.

## Best Possible Solution

Make the browser path deterministic for the confirmed host mode: either fix managed Chromium launch/doctor diagnostics for the failing Linux/container case, or guide Windows/WSL split-host installs to remote CDP, node-host browser control, or host-local Chrome MCP with clear config and verification while keeping no-sandbox opt-in.
