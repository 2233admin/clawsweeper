## Summary

Keep open. Current main has explicit browser no-sandbox support and Linux/WSL troubleshooting docs, but the reported v2026.4.26 failure occurred after the reporter says noSandbox and executablePath were configured, and there is no relevant post-report browser launcher/config change to close this as fixed. The next decision depends on confirming the exact host mode and resolved browser config.

## What This Changes

The issue reports that managed browser startup fails with Chrome sandbox/namespace errors on Windows/Ubuntu-container paths, blocking browser automation even after configuring executablePath and noSandbox.

## Best Possible Solution

Clarify the supported environment and resolved browser profile first, then make browser startup deterministic for that mode: either repair managed launch/doctor diagnostics for the failing Linux/container path or route Windows/WSL split-host setups to remote CDP, node-host browser control, or Chrome MCP with explicit, actionable configuration while keeping no-sandbox opt-in.
