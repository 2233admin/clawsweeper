## Summary

Keep open. Current main has the pre-tool choke point, trusted policy surface, session reset lifecycle, and plugin approval plumbing, but it still does not implement the requested framework-owned session PIN gate: no `session.auth.pin` config/schema field, no core authenticated bit on `SessionEntry`, and no trusted TypeScript verifier that can unlock tool execution outside prompt/model context.

## Best Possible Solution

Keep this open for maintainer design. The likely path is to add a documented session-auth config using existing secret/redaction conventions, persist a framework-owned authenticated flag on `SessionEntry`, clear it on `/new` and `/reset`, enforce it in the trusted pre-tool path before ordinary plugin hooks, and add a native command/gateway/internal verifier that compares PIN attempts in TypeScript outside model context. Cover config/schema validation, before-tool blocking, reset lifecycle, transcript/log redaction, and security docs.

## Risks / Open Questions

- A prompt-only or ordinary external-plugin-only implementation would not satisfy the requested prompt-injection-resistant guarantee because untrusted content could still steer the model around the policy contract.
- A future implementation needs careful SecretRef/redaction handling so configured PINs and user PIN attempts do not leak into transcripts, logs, diagnostics, tool results, or model context.
- The verifier must define clear lifecycle and UX semantics for `/new`, `/reset`, aborts, background/system events, and approval-style channel flows before it becomes a safe core security primitive.
