## Summary

Keep this PR open. Current main still has the two gaps the PR targets: Slack block-action system events are queued without waking the routed session, and the Slack system-event resolver still cannot receive `threadTs`, so thread interactions resolve to the base channel/session path. The patch is useful but overlaps active Slack interaction work in #61502, #57308, and #67751, so it needs maintainer reconciliation rather than cleanup closure.

## What This Changes

The PR adds a wake hint to `enqueueSystemEvent`, passes Slack block-action `threadTs` into system-event session resolution, derives thread-aware Slack session keys with `resolveThreadSessionKeys`, and adds focused wake/thread-routing tests.

## Best Possible Solution

Keep this PR available as source work while maintainers settle the canonical Slack interaction path: wake the routed session only after a non-duplicate interaction system event is queued, derive the same thread-aware session key used by normal Slack routing, preserve reply delivery context, and cover block actions, reply buttons/selects, duplicates, and busy-session retry behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a Slack block action with `container.thread_ts` reaches `enqueueSlackBlockActionEvent`, current tests show `threadTs` is payload-only, the resolver call omits `threadTs`, and `enqueueSystemEvent` has no wake path on current main.

## Solution Assessment

Unclear as merge-ready, but the direction is narrow and maintainable. The wake-after-successful-enqueue and `resolveThreadSessionKeys` approach matches existing contracts, but the patch should be rebased and reconciled with #61502, #57308, and #67751 before maintainers land it.
