## Summary

Current main still does not implement Discord trusted webhook ingress. Discord preflight resolves webhook IDs, but bot-authored webhook messages are still controlled only by `allowBots`/`allowBots="mentions"`, PluralKit identity handling, and bound-thread echo suppression. There is no `trustedWebhooks` guild config, no trusted webhook allowlist check, and no `mapToUser` sender mapping path.

## Best Possible Solution

Keep this issue open for the Discord plugin owner and security review. If accepted, implement it inside `extensions/discord` with typed and zod-validated guild `trustedWebhooks` config, a trusted webhook check before the default bot-message drop, explicit authorization rules for any `mapToUser` mapping, preserved echo guards, and focused docs/config/test coverage for allowed, denied, mention-only, mapped-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` would change effective sender identity, so implementation needs explicit security review for configured guild users, command/tool authorization, auditability, and channel allowlists.
- Trusted webhook ingress must preserve existing own-message and bound-thread webhook echo suppression to avoid self-triggering loops.
- Closing this as covered by generic HTTP hooks would lose the reporter's same-channel Discord webhook workflow.
