## Summary

Keep this issue open. Current main still lacks Discord trusted webhook ingress: webhook IDs are resolved, but bot-authored webhook messages still pass only through broad `allowBots`/`allowBots="mentions"`, PluralKit, and bound-thread echo suppression. There is no guild-level `trustedWebhooks` config, no trusted allowlist check, and no `mapToUser` sender remapping path.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it in `extensions/discord` with typed and zod-validated guild `trustedWebhooks` config, a preflight trusted-webhook allowlist before the default bot-message drop, explicit `mapToUser` authorization against configured guild users, preserved echo guards, docs/config metadata, and focused tests for allowed, denied, mention-only, mapped-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` would change effective sender identity and needs explicit authorization rules, auditability, and interaction with guild/channel allowlists and tool policy.
- Trusted webhook ingress must preserve own-message and bound-thread webhook echo suppression to avoid self-triggering loops.
- Closing this as covered by generic HTTP hooks would lose the reporter's same-channel Discord webhook workflow.
