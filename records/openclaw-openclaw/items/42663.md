## Summary

Keep this issue open. Current main still lacks the requested Discord trusted webhook ingress: webhook IDs are resolved for existing PluralKit/echo-suppression paths, but bot-authored webhook messages still depend on broad `allowBots` or `allowBots="mentions"`, and there is no guild-level `trustedWebhooks` config or `mapToUser` sender remapping. The related implemented HTTP hook feature does not satisfy the reporter's same-channel Discord webhook workflow.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it in `extensions/discord` with typed and zod-validated guild `trustedWebhooks` config, a trusted-webhook allowlist check before the default bot-message drop, `mapToUser` authorization against configured guild/channel users, preserved echo guards, public docs/config metadata, and focused preflight tests for allowed, denied, mention-only, mapped-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` would affect effective sender identity, so an implementation needs explicit authorization against configured guild/channel users and clear audit behavior.
- Trusted webhook ingress must preserve own-message and bound-thread webhook echo suppression to avoid loops.
- Generic HTTP hooks are related automation, but they do not preserve the reporter's Discord same-channel webhook workflow.
