## Summary

Keep open. Current main still requires adapters to provide `shouldContinueAutoSelection` or auto-selection aborts, so the PR addresses a real remaining plugin-contract gap for third-party memory embedding adapters. The branch should not merge as-is because its remote-provider guard checks the adapter id instead of the adapter transport, which can incorrectly suppress failures from third-party local adapters.

## What This Changes

The PR changes `extensions/memory-core/src/memory/embeddings.ts` so auto provider selection treats common auth/config errors from adapters without `shouldContinueAutoSelection` as skippable failures instead of aborting.

## Best Possible Solution

Keep the PR direction, but update the fallback to respect explicit adapter guidance first, apply auth/config soft-skipping only to non-local transports, and add focused tests for third-party remote and local adapters that omit `shouldContinueAutoSelection`.

## Reproduction Assessment

Yes. A focused unit reproduction is an auto-select remote adapter that throws a missing-key or unauthorized error and omits `shouldContinueAutoSelection`; current main defaults to aborting, while the PR would continue. The PR’s introduced edge case is a third-party adapter with `transport: "local"`, id other than `local`, and no hook throwing an auth-like error.

## Solution Assessment

No, not as written. The heuristic is a narrow maintainable solution, but the guard should use `adapter.transport !== "local"` rather than `adapter.id !== "local"` and tests should lock that contract down.
