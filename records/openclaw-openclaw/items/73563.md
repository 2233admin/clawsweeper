## Summary

Keep open. Current main still has the transcript-redaction gap this PR targets, and the PR is a relevant implementation candidate, but the latest patch still has blocking correctness and SDK-boundary issues before it should merge.

## What This Changes

The PR wraps several raw session transcript append paths with `guardSessionManager`, threads config through some callers, centralizes transcript update-mode handling, adds a `session-transcript-guard` Plugin SDK subpath, and adds redaction/idempotency/event tests.

## Best Possible Solution

Land a corrected, maintainer-reviewed security fix that applies the shared transcript guard to every raw transcript append path, threads active config through all production callers that have it, reports blocked guard writes accurately, preserves single transcript update semantics, and either keeps the guard internal or deliberately documents it as a supported generic Plugin SDK contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Thread config through outbound mirror calls:** `src/config/sessions/transcript.ts:165-166`
  - body: Adding `config` to the transcript helper is not enough while the two production outbound mirror callers still omit their active cfg. Those delivery-mirror writes will keep using default redaction, so `logging.redactSensitive="off"` and custom redact patterns are silently ignored on delivered-message transcript mirrors.
  - confidence: 0.88
- **[P2] Handle blocked guard appends before success:** `src/config/sessions/transcript.ts:288-289`
  - body: After this helper wraps the session manager, `appendMessage` can return `undefined` when a before-write hook blocks the message or the guard drops it. The helper still returns `{ ok: true, messageId }`, so callers can observe success with no persisted transcript entry and an undefined id. Check the append result before returning success.
  - confidence: 0.84
- **[P2] Keep guard out of public SDK exports:** `package.json:829-832`
  - body: The new subpath is added to the package export map and SDK entrypoints, but its source comment says it is bundled-extension-only and not for third-party plugins. In this repo every `plugin-sdk` entrypoint becomes an importable public contract, so this promotes internal SessionManager plumbing as public API unless it is explicitly documented and accepted as a generic SDK seam.
  - confidence: 0.82
