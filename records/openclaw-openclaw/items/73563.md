## Summary

Keep this PR open. The underlying transcript-redaction gap is still present on current main and the PR is a relevant implementation candidate, but the latest head is not merge-ready because it still has blocked-write success semantics, incomplete active-config threading, and an unresolved public SDK/security-surface decision.

## What This Changes

The PR wraps several session transcript append paths with `guardSessionManager`, adds update-mode/config plumbing and redaction tests, and exports `guardSessionManager` through the agent harness runtime barrel.

## Best Possible Solution

Land a corrected, maintainer-reviewed security fix that applies the shared transcript guard to every raw transcript append path, preserves existing transcript update semantics, threads active runtime config through all production mirror callers, handles guard-blocked writes explicitly, and either keeps the guard internal to bundled code or documents/version-locks it as a supported Plugin SDK contract.

## Reproduction Assessment

Yes. Current main can be inspected to find the raw `SessionManager.open(...).appendMessage(...)` paths, and the PR regression can be reproduced with a `before_message_write` blocker against `appendExactAssistantMessageToSessionTranscript`, which should not report success when no JSONL entry is written.

## Solution Assessment

No. The direction is right, but this is not yet the best fix because it leaves success reporting wrong for blocked appends, skips active redaction config in outbound mirror callers, and promotes internal guard plumbing through a public SDK barrel without the matching product/API decision.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle blocked guard appends before returning success:** `src/config/sessions/transcript.ts:290-291`
  - body: After this helper wraps the session manager, `appendMessage` can return `undefined` when `before_message_write` blocks the message or the guard drops it. The helper still returns `{ ok: true, messageId }`, so callers can observe success even though no transcript entry was persisted. Check the append result and return a failure or explicit blocked result instead.
  - confidence: 0.88
- **[P2] Thread config through outbound transcript mirrors:** `src/config/sessions/transcript.ts:168`
  - body: Adding `config` to the transcript helper is not enough while the production outbound mirror call sites still omit their active `cfg`. Those paths will use default redaction, so `logging.redactSensitive="off"` and custom `logging.redactPatterns` are silently ignored. Pass the active config from both outbound mirror callers.
  - confidence: 0.85
- **[P2] Keep the transcript guard out of the public runtime barrel:** `src/plugin-sdk/agent-harness-runtime.ts:129`
  - body: Exporting `guardSessionManager` from `openclaw/plugin-sdk/agent-harness-runtime` promotes internal SessionManager redaction/update plumbing as public Plugin SDK API. Use a narrow internal/bundled seam, or document and version this as a supported SDK contract before syncing the baseline.
  - confidence: 0.8

## Security Review

Summary: The diff improves a real transcript redaction gap, but it still affects security-sensitive persistence behavior and public SDK exposure.

  - body: The PR adds config-aware transcript helpers, but production outbound mirror callers still do not pass their active config.
