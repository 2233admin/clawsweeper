## Summary

Keep this PR open. Current main still has the Discord duplicate-token startup path the PR is trying to fix: token resolution can give the default account `DISCORD_BOT_TOKEN` while a named account uses the same configured token, the inbound replay key is scoped by account id, and gateway startup still starts every listed Discord account. The PR is a plausible implementation candidate, but its current diff appears to miss the actual gateway account-start path because Discord's plugin config still exposes raw `listDiscordAccountIds()` to `src/gateway/server-channels.ts`.

## Best Possible Solution

Keep the PR open for maintainer review, but request changes before landing: move the token-deduplication into the Discord plugin's startup account surface or another gateway-visible hook so duplicate-token accounts are not started, preserve status/diagnostic behavior for the dropped account, and add a regression test proving one monitor starts when default env fallback and a named config account resolve to the same token.
