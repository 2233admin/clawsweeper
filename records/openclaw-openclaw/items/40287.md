## Summary

Keep this PR open. Current main still does not register or hydrate Slack `file_shared` events, so the central large-file/multi-file Socket Mode fallback requested by the PR is not implemented. Main has since gained adjacent Slack file improvements, including inbound fileId placeholders, forwarded attachment extraction, and secure `download-file` handling, but those only help after a normal Slack message/attachment reaches the message pipeline. The PR remains a plausible implementation candidate, though it must be ported from the removed `src/slack` layout into `extensions/slack` and tightened before merge.

## Best Possible Solution

Keep this PR open as the canonical implementation candidate, but require a rebase or replacement port into `extensions/slack`: add and register a `file_shared` event handler under `extensions/slack/src/monitor/events`, hydrate the matching Slack message through a bounded event-time-anchored `conversations.history` lookup, dedupe Slack retries and normal `message`/`file_share` delivery, preserve current account/team/channel/user authorization and Slack media SSRF protections, update `extensions/slack/src/setup-shared.ts` and `docs/channels/slack.md` for the `file_shared` bot event, and add focused Slack plugin tests for large-file, multi-file, duplicate-delivery, busy-channel, and app-mention interactions.
