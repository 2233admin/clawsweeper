## Summary

Keep this issue open. Current main still verifies only the first Docker apt key fingerprint before dearmoring the whole downloaded key file, and open PR #74254 is already the direct implementation candidate for this supply-chain hardening gap.

## What This Changes

The issue requests hardening the Docker CLI install path so the downloaded Docker apt signing-key file is rejected unless it contains exactly one public key before fingerprint validation and dearmor.

## Best Possible Solution

Keep this issue open until PR #74254 or an equivalent narrow Dockerfile hardening change lands, with a guard that rejects any Docker apt key file containing other than exactly one primary pub record before fingerprint comparison and dearmor.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a key file whose first fpr matches the pinned fingerprint but also contains an extra primary pub key would pass the current awk ... exit comparison and then be dearmored wholesale into the apt keyring.

## Solution Assessment

Yes. The requested exactly-one-primary-key validation is the narrow maintainable fix because GnuPG distinguishes primary public keys from subkeys in --with-colons output; PR #74254 appears to target that boundary but still needs review and merge.
