## Summary

Keep this issue open. Current main still validates only the first `fpr` record from the downloaded Docker apt keyfile and then dearmors the entire file, while the directly linked PR #74254 remains open as the narrow implementation candidate for the requested single-primary-key guard.

## What This Changes

The issue requests hardening the Dockerfile so Docker apt GPG validation rejects keyfiles containing more than one primary public key before the keyfile is trusted.

## Best Possible Solution

Review and land PR #74254 or an equivalent narrow Dockerfile change that counts exactly one primary `pub` key before fingerprint comparison and before `gpg --dearmor`, with focused coverage for the official Docker key path and a multi-primary-key rejection case.

## Reproduction Assessment

Yes. The current-main reproduction path is source-level and high confidence: `Dockerfile:245` exits after the first `fpr` record, while `Dockerfile:250` dearmors the full downloaded keyfile, so an additional primary key would not be rejected by the existing guard.

## Solution Assessment

Yes. The best direction is the narrow PR #74254 approach or equivalent: reject anything other than exactly one primary `pub` key before comparing the expected fingerprint and before writing the trusted APT keyring.
