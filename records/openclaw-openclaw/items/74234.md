## Summary

Current main still has the reported behavior: the Docker CLI install path verifies only the first `fpr` record from the downloaded Docker apt key file, then dearmors the entire file into the repository keyring. APT documentation says that when `Signed-By` names a keyring without a fingerprint, all keys in that keyring are selected, so an appended extra public key would not be excluded by the current validation.

## Best Possible Solution

Keep this open and fix the Docker CLI install block narrowly: validate that the downloaded key material contains exactly one primary public key whose primary fingerprint matches `OPENCLAW_DOCKER_GPG_FINGERPRINT`, or otherwise pin the selected apt signer fingerprint in the `Signed-By` configuration. Add a targeted Dockerfile test that would fail for an appended second public key, and avoid unrelated Docker base image, package, or release workflow changes.
