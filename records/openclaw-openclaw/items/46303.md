## Summary

Keep this PR open. Current main at db40ec404a9178294dba3121758ff59459983a11 still lacks the PR's central behavior: restart drains active command/runtime work, but does not flush active inbound debounce buffers or wait for the process-global followup queues before SIGUSR1 teardown. The PR remains a plausible implementation candidate, though it needs maintainer review for lifecycle ordering, SDK compatibility, and the security-sensitive global registry design.

## Best Possible Solution

Keep this PR open as an implementation candidate, but request a narrower maintainer-reviewed revision: add a lifecycle-safe drain surface for active inbound debouncers, preserve or version the public SDK contract, order restart draining so flushed work can run, wait for active turns plus followup queues including dropped-summary state, cover channel worker queues, and add focused restart/teardown race tests.

## Risks / Open Questions

- Closing this PR would discard a current-main bug-fix candidate for restart-time loss of closure-local debounce buffers and process-global followup queues.
- The proposed global debouncer registry is security-sensitive: strong process-global references can retain stale closures/timers, and same-process plugin code may interact with restart-time handles unless the API is narrow and teardown is robust.
- The PR changes plugin-facing debounce contracts through SDK re-exports; third-party plugins may rely on the current `{ enqueue, flushKey }` shape and `flushKey: Promise<void>` behavior.
- The PR discussion includes unresolved lifecycle concerns around ingress shutdown ordering, late enqueues, dropped-summary followup state, and channel-specific worker queues such as Discord.
