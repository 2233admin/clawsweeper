## Summary

Keep open. Current main still does not implement the PR's intended restart-time inbound debounce flush and followup queue drain, but the PR branch is not ready to merge because it introduces concrete restart lifecycle, public SDK compatibility, and availability/security issues.

## What This Changes

The PR adds a process-global inbound debouncer registry and flush path, a followup-queue drain waiter, SIGUSR1 restart integration, and bundled channel teardown/test updates to reduce message loss during gateway reloads.

## Best Possible Solution

Revise the branch around a backward-compatible restart-drain design: keep public SDK signatures stable, mirror the followup queue drain predicate including dropped summaries, keep live debouncers discoverable until ingress is closed, and close the external enqueue window at restart start through an explicit internal-drain bypass or transport pause.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count dropped summaries while waiting for followup drain:** `src/auto-reply/reply/queue/drain-all.ts:21-22`
  - body: `scheduleFollowupDrain` keeps draining while `queue.items.length > 0 || queue.droppedCount > 0`, but this waiter only counts queued items plus an active drain sentinel. A queue with only summarized dropped messages can report `drained: true` and let restart close before the summary followup is sent.
  - confidence: 0.91
- **[P2] Keep idle live debouncers registered until ingress closes:** `src/auto-reply/inbound-debounce.ts:83-84`
  - body: This pre-close flush can unregister any debouncer idle for five minutes even though the server may still be accepting channel ingress. A live but quiet channel can then receive a late message and buffer it on a handle no longer visible to the global restart flush.
  - confidence: 0.86
- **[P2] Gate external enqueues before restart drain:** `src/cli/gateway-cli/run-loop.ts:424`
  - body: The branch waits through debounce flush, active-task drain, and followup drain before setting `gatewayDraining`. During that window inbound users can still enqueue new command work that the active-task snapshot may not wait for, prolonging restarts and reintroducing restart-time loss or cost spikes.
  - confidence: 0.82
- **[P2] Preserve flushKey return type for SDK compatibility:** `src/auto-reply/inbound-debounce.ts:235-240`
  - body: `createInboundDebouncer` is re-exported through public plugin SDK subpaths, and this changes `flushKey` from the current `Promise<void>` shape to `Promise<boolean>`. Third-party plugins or shared test helpers typed against the old contract can fail to type-check on upgrade; keep `flushKey` compatible and expose delivery counts through an additive method.
  - confidence: 0.78
