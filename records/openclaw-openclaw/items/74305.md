## Summary

Keep this issue open. Current main has several related ACPX/Codex improvements after v2026.4.26, including bundling @zed-industries/codex-acp 0.12.0, preserving configured Codex ACP command arguments through the local wrapper, and propagating sessions_spawn model/thinking options. However, the reported live failure is specifically a configured codex-acp command using gpt-5.5 plus model_reasoning_effort, and current coverage does not live-verify that combination or guarantee a specific validation error when the adapter rejects it. Upstream codex-acp also still has recent gpt-5.5/model availability reports, so this is not a safe implemented-on-main close without a current-main live smoke or a targeted fix.

## What This Changes

The issue asks for ACPX Codex worker runs to succeed, or fail with a specific validation error, when codex-acp is configured with model and model_reasoning_effort overrides.

## Best Possible Solution

Keep the issue open until maintainers either live-reproduce against current main with the bundled acpx plugin and Codex auth, or add a targeted OpenClaw fix that validates/translates unsupported Codex ACP model/reasoning overrides before the adapter returns a generic internal error. The likely fix boundary is the acpx plugin wrapper/runtime layer plus a regression covering both -c model and -c model_reasoning_effort configured command args.
