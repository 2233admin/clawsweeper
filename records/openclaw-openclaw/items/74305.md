## Summary

Keep open. Current main still appends Codex ACP runtime model/thinking `-c` startup config without checking whether the configured command already contains those keys, and latest release `v2026.4.29` has the same helper. The reporter also confirmed the workaround of removing the configured `-c model` / `-c model_reasoning_effort` args, while open PR #74339 is already explicitly paired to this issue as the targeted fix candidate.

## What This Changes

The issue asks for ACPX Codex worker runs to succeed, or fail with a clear validation error, when configured Codex ACP model/thinking overrides are present.

## Best Possible Solution

Review and land PR #74339, or an equivalent narrow ACPX runtime fix that preserves configured Codex adapter args while skipping runtime startup `-c` keys already present in the command, with tests for full and partial overlap; close this issue only after that reaches main or a shipped release.

## Reproduction Assessment

Yes. The issue gives concrete operator steps, and current main statically shows the same path: configured Codex ACP `-c` args can be preserved, then `sessions_spawn` model/thinking options append another `-c model` / `-c model_reasoning_effort` pair before session startup.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the ACPX Codex command-normalization/runtime override path, because core `sessions_spawn` should continue forwarding generic ACP model and thinking options while the Codex ACP command builder owns startup config arguments.
