## Summary

Keep open. Current main still leaves the shared message tool's `channel` schema as an undescribed optional string, so the requested tool-schema guidance is not implemented; prompt text and fallback routing help some cases but do not fix the schema metadata gap, and the related implementation PRs are closed unmerged.

## What This Changes

Request to add a TypeBox description, and possibly enum guidance, to the shared message tool's `channel` parameter so agents pass a provider or channel plugin id instead of a destination conversation/channel id.

## Best Possible Solution

The message tool should keep `channel` optional, describe it as the messaging provider or channel plugin id such as `slack` or `discord`, clearly distinguish it from destination targets, and add a focused schema assertion; enum support should only be added if it can include dynamic plugin ids without blocking third-party channels.

## Repair Work Prompt

Fix the shared message tool schema guidance for this issue. In `src/agents/tools/message-tool.ts`, keep `channel` optional but add a TypeBox description explaining that it expects a messaging provider or channel plugin id such as `slack` or `discord`, not a destination conversation/channel/user id. Add a focused assertion in `src/agents/tools/message-tool.test.ts` that the serialized message tool schema exposes this description. Do not add a hard-coded enum unless it is generated from the registered/configured channel values and remains compatible with third-party channel plugins. Do not change runtime channel fallback behavior or Slack target normalization in this PR. No changelog is expected for a schema metadata/test-only repair unless maintainers decide to call out the agent UX fix.
