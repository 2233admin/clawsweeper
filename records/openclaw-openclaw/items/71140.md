## Summary

Keep open. Current main still has the Telegram media send paths and native reaction primitives, but the normal outbound reply path still does not parse `[[react...]]` directives, does not synthesize `channelData.telegram.reaction`, rejects reaction-only Telegram replies as no-text/no-media, and the issue is paired with open draft PR #71141 using closing syntax.

## What This Changes

The issue requests shared outbound normalization plus Telegram delivery/adapter support so `[[react_to_current:...]]` and `MEDIA:` replies become native Telegram reactions and media sends instead of leaked or dropped text directives.

## Best Possible Solution

The end state should be a shared reply directive parser that emits structured reaction metadata, outbound normalization that preserves it in channel data, and Telegram delivery/adapter code that applies the native reaction to the resolved reply target before optional text/media delivery, with focused parser, payload, delivery, and adapter tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `[[react_to_current:...]]` is not recognized by the inline directive parser, remains visible text, and reaction-only Telegram payloads reach the no-text/no-media rejection path instead of native reaction handling.

## Solution Assessment

Yes. The issue’s shared-parser plus Telegram-specific delivery/adapter wiring is the narrow maintainable fix; a Telegram-only string special case would leave the shared outbound normalization contract inconsistent.
