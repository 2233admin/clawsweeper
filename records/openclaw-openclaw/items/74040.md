## Summary

Keep open. Current main still treats any replay-history length as prompt content, so blank user-message entries can bypass the empty-submission guard. The draft PR is also not landable as written because it reassigns a `const promptSubmission` and sets `skipPromptSubmission` before the intended fallback greeting would be submitted.

## Best Possible Solution

Keep this PR open as draft and revise the fix in the embedded runner prompt-submission boundary. The best path is a typed content-aware helper, covered by targeted tests for blank string and blank multi-block user messages, that either skips cleanly or injects/submits a startup prompt without changing unrelated provider, active-memory, or plugin hook behavior.
