## Summary

Keep open. Current main still uses a length-only prompt-submission guard, so blank replay-history entries can still make an empty visible prompt look submit-worthy. The PR is a plausible narrow fix direction, and its second commit addressed the earlier skipPromptSubmission concern, but the current diff is still draft and not landable because it assigns to the existing const promptSubmission binding and adds no targeted coverage for blank string or blank multi-block user messages. The security pass found no supply-chain-sensitive changes: the PR only touches the embedded runner source file.

## Best Possible Solution

Keep this PR open as draft and revise it into a typed, content-aware prompt-submission fix. The best patch should move the substantive-content check into attempt.prompt-helpers.ts, use it from the embedded runner submission gate, decide explicitly whether blank startup/reset cases should skip or receive the existing safe startup prompt, and add focused tests for blank string user content, blank multi-block text content, images, legitimate non-empty replay history, and runtime-only context. Do not touch provider selection, plugin hooks, CI, dependencies, or unrelated transcript repair behavior.
