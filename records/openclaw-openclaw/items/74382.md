## Summary

Keep open. Current main still has a plausible onboarding UX bug: the dependency picker can show the keyless local `openai-whisper` entry, then the same setup flow can prompt for `OPENAI_API_KEY` or `ELEVENLABS_API_KEY` for separate env-gated skills. The issue is also already paired with open closing PRs #74407 and #74891, so it should stay open until maintainers review and land or close those PRs.

## What This Changes

The issue asks for `openclaw onboard` to stop contradicting the keyless `openai-whisper` install hint by later asking for OpenAI Whisper or ElevenLabs API keys without clear scoping.

## Best Possible Solution

The best outcome is to land one narrow onboarding fix that keeps local `openai-whisper` keyless, scopes OpenAI/ElevenLabs prompts to the actual env-gated skills, and adds regression coverage for the mixed keyless/keyed missing-skill path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `buildWorkspaceSkillStatus` with missing local `openai-whisper` plus missing env-gated `openai-whisper-api` and `sag`, then run `setupSkills`; the current code renders install options first and then prompts for every missing `primaryEnv`.

## Solution Assessment

Yes, a narrow onboarding UX fix is the right solution boundary. Changing the skill metadata would be less maintainable because the local and API-backed Whisper skills are already distinct; the bug is in how the setup flow presents and gates credential prompts.
