## Summary

Keep open. Current main still loses the queued exec-event delivery context for the reported configuration: explicit Slack heartbeat targets do not become turn-source routing, so the Slack thread id can be dropped before outbound delivery. The related draft PR #71582 is still the active implementation candidate rather than a reason to close this issue.

## What This Changes

Ensure Slack async exec completion relays stay in the originating Slack thread, including when heartbeat is configured with an explicit Slack target/to.

## Best Possible Solution

Preserve or prefer queued exec-event deliveryContext, including Slack threadId, in the generic heartbeat/outbound resolver even when heartbeat.target and heartbeat.to are explicit. Add a focused regression for explicit Slack heartbeat targets and keep ordinary scheduled heartbeat stale-thread suppression intact.

## Reproduction Assessment

Yes. The reporter provided a concrete Slack async exec reproduction, and current main shows the same failure at the resolver level: the event thread exists, but explicit heartbeat target resolution does not pass a turn-source channel/to alongside it.

## Solution Assessment

Yes for the direction, but not yet implemented on main. The narrow maintainable fix belongs in generic heartbeat delivery resolution rather than Slack send, because Slack only receives the threadId that core preserves.
