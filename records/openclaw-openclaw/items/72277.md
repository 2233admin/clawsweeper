## Summary

Keep open. Current main still preserves the exec-event delivery context up to heartbeat preflight, but explicit Slack heartbeat targets do not treat that context as the active turn source, so the originating Slack thread can still be dropped before outbound delivery.

## What This Changes

The issue requests that Slack async exec completion relays stay in the originating channel thread when heartbeat delivery is configured with an explicit Slack target.

## Best Possible Solution

Preserve queued exec-event deliveryContext, including Slack threadId, through the generic heartbeat/outbound resolver even when the configured heartbeat target is an explicit Slack channel/to. Add a focused regression for explicit Slack heartbeat targets while keeping stale-thread suppression for ordinary scheduled heartbeats.

## Reproduction Assessment

Yes. The reporter provided a concrete Slack async exec reproduction, and current main shows a resolver-level reproduction path: the event threadId exists but is not converted into an active turn-source route for explicit heartbeat targets.

## Solution Assessment

Yes for the requested direction; there is no patch under review here. The narrow maintainable fix belongs in generic heartbeat delivery target resolution rather than Slack send code, because Slack can only thread the relay if core preserves delivery.threadId.
