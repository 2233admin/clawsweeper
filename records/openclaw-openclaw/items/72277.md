## Summary

Keep open. Current main still drops the queued exec-event thread route when heartbeat delivery is configured with an explicit Slack target/to, and the open draft PR #71582 remains the active implementation candidate for this routing gap.

## What This Changes

Ensure Slack async exec completion relays stay in the originating Slack thread when the agent has an explicit Slack heartbeat target.

## Best Possible Solution

Preserve the queued exec-event deliveryContext, including Slack threadId, through generic heartbeat delivery target resolution even when heartbeat.target and heartbeat.to are explicit. Keep ordinary scheduled-heartbeat stale-thread suppression intact and add focused regression coverage for explicit Slack heartbeat target routing.

## Reproduction Assessment

Yes. The reporter provided a concrete Slack async exec reproduction, and current main shows the same route loss at the resolver level: the event can carry threadId, but explicit heartbeat targets do not pass a parsed turn source into heartbeat session delivery resolution.

## Solution Assessment

Yes for the proposed direction, but it is not implemented on main. The narrow maintainable fix is in the generic heartbeat/outbound routing path, not in Slack send, because Slack already uses threadId when core preserves it.
