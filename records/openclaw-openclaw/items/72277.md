## Summary

Keep open. Current main still drops the queued exec-event thread route for explicit Slack heartbeat targets, and the related draft PR #71582 remains the active implementation candidate rather than a merged fix.

## What This Changes

The issue asks for Slack async exec completion relays to stay in the originating channel thread even when heartbeat delivery is configured with an explicit Slack target and `to`.

## Best Possible Solution

Preserve or prefer queued exec-event `deliveryContext`, including Slack `threadId`, in the generic heartbeat/outbound resolver when the wake is scoped to an event, while keeping ordinary scheduled heartbeat stale-thread suppression intact.

## Reproduction Assessment

Yes. The reporter provided a concrete Slack async exec marker and config, and current main reproduces the failure at the resolver level: the event context exists, but explicit heartbeat target resolution does not carry the thread route into outbound delivery.

## Solution Assessment

Yes for the direction, but not yet on main. The narrow maintainable fix belongs in generic heartbeat delivery resolution because the Slack adapter already sends in-thread when core passes a `threadId`.
