## Summary

Current main still appears to match the reported Slack explicit-heartbeat-target edge case. Exec completion events carry a deliveryContext, but heartbeat target resolution only merges that turn-source route for target "last"; with an explicit Slack heartbeat target/to, the threadId can be dropped before Slack send sees it. The related open PR #71582 remains the active implementation candidate, so this issue should stay open pending review/landing or maintainer closure of that PR.

## Best Possible Solution

Keep this issue open and route it to #71582 or an equivalent focused fix. The implementation should make exec-event wakes prefer or carry the queued event deliveryContext, including Slack threadId, even when heartbeat.target is an explicit Slack target/to, while preserving stale-thread clearing for ordinary scheduled heartbeats. Add a focused regression test for explicit Slack heartbeat target/to plus queued exec-event deliveryContext with threadId.
