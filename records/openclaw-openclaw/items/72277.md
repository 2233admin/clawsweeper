## Summary

Keep open. Current main still matches the reported failure mode: async exec completion events carry the originating deliveryContext, but explicit heartbeat targets do not treat that queued context as the full turn source, so Slack can receive no threadId and post the relay at channel root. The open draft PR #71582 remains the active implementation candidate, not a reason to close this issue before merge.

## What This Changes

The issue reports that Slack async exec completion replies started from a channel thread escape to the channel root when heartbeat delivery is configured with an explicit Slack target.

## Best Possible Solution

Land or replace #71582 with a narrow fix that preserves queued exec-event deliveryContext as the delivery source for explicit heartbeat targets, including Slack threadId, while keeping scheduled heartbeat behavior from inheriting stale session threads. Add focused coverage for explicit Slack heartbeat targets and async exec completion relay.

## Reproduction Assessment

Yes. The issue gives concrete Slack thread reproduction steps, and current source provides a deterministic code-level reproduction: enqueue an exec-event deliveryContext with a Slack threadId, configure heartbeat with explicit Slack target/to, and resolve delivery without a threadId.

## Solution Assessment

Yes for the proposed direction, but it is not implemented on main. The maintainable fix belongs in generic heartbeat/outbound target resolution because Slack already preserves thread_ts when core passes a valid threadId.
