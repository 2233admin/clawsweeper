## Summary

Keep this issue open. Current main still has arbitrary lane plumbing, but ordinary inbound channel auto-reply and follow-up runs do not get a configured browser lane or per-channel/per-account lane assignment, so missing `lane` values still fall back to the process-wide `main` lane.

## Best Possible Solution

Keep this issue open. The best path is a core scheduler/config change: add a validated custom-lane concurrency surface, apply it during gateway startup and reload, route inbound channel/account/agent runs to an execution lane before `runEmbeddedPiAgent()` enters the global queue, and cover docs/tests for interactions with `agents.defaults.maxConcurrent`, cron/subagent lanes, queue modes, and browser-heavy workloads. Browser-tool auto-detection can be considered later, but pre-run lane assignment is the safer first implementation.

## Risks / Open Questions

- Closing this now would drop a still-missing core scheduling/config capability: slow browser-heavy inbound sessions can still consume the same process-wide main-lane capacity as fast text replies.
- The eventual fix needs maintainer design around lane naming/validation, config reload behavior, fairness, provider rate limits, per-session serialization, and whether routing belongs under agent defaults, channel/account bindings, or a queue policy surface.
