## Summary

Keep this issue open. Current main still supports arbitrary internal lane strings, but it does not expose a configured browser lane or per-channel/per-account lane routing for ordinary inbound channel sessions. Channel-triggered auto-reply runs still enter `runEmbeddedPiAgent()` without a `lane`, so `resolveGlobalLane()` defaults them into the process-wide `main` lane.

## Best Possible Solution

Keep this issue open. The best implementation path is a core scheduler/config change: add a validated custom-lane concurrency surface, apply it on gateway startup and reload, route inbound channel/account/session runs to an execution lane before `runEmbeddedPiAgent()` enters the global lane, and cover interactions with per-session serialization, `agents.defaults.maxConcurrent`, cron/subagent lanes, queue modes, and browser cleanup. Browser-tool auto-detection can be considered later, but pre-run lane assignment is the safer first fix.
