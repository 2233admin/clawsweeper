## Summary

Keep this issue open. Current main still has arbitrary lane plumbing, but inbound channel primary and follow-up agent runs do not have a validated custom/browser/per-channel lane config surface and still default to the process-wide main lane when no explicit lane is passed.

## Best Possible Solution

Keep this issue open for a maintainer-owned core lane-routing design. Add a validated custom lane-pool config surface, apply it during gateway startup and config reload, route inbound primary and follow-up channel runs into the selected lane before runEmbeddedPiAgent enqueues global work, and document/test precedence across channel, account, binding, agent, and workload classification. Browser-tool auto-detection can follow after explicit pre-run routing exists.
