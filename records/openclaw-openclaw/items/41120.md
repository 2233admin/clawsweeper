## Summary

Keep open. Current main still has the low-level arbitrary lane plumbing, but inbound channel auto-reply and queued follow-up runs do not have a browser lane or per-channel/per-account lane routing surface; missing lane values still fall back to the process-wide main lane.

## Best Possible Solution

Keep this issue open and implement a core scheduling/config path: add a validated custom-lane concurrency surface, apply it during gateway startup and reload, add lane assignment for inbound channel/account/agent routing before runEmbeddedPiAgent() enters the global queue, and cover primary auto-reply plus queued follow-up tests and docs. Browser-tool auto-detection can be considered later, but explicit pre-run routing is the safer first fix.
