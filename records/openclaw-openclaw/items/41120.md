## Summary

Current main still lacks the requested dedicated browser/workload lane routing. The queue accepts arbitrary lane strings, but gateway startup only configures main, cron, cron-nested, and subagent concurrency, and inbound auto-reply/follow-up runs still call the embedded runner without a configured lane, so they fall back to the process-wide main lane.

## Best Possible Solution

Keep this issue open. The best implementation path is a core lane-routing surface: add validated custom lane pools, apply their concurrency during gateway startup/reload, route inbound channel/account/agent or workload-classified sessions into the selected lane before runEmbeddedPiAgent enqueues global work, and cover both primary auto-reply and queued follow-up paths. Browser-tool auto-detection can be a later enhancement, but explicit pre-run routing is the key missing capability.
