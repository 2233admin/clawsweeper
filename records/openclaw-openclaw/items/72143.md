## Summary

Keep open. Current main still only partially addresses the report: successful code-0 direct fallback content is prefix-free, but denied and other non-success resume-failure fallbacks still expose the internal prefix, and the session resume path still has no retry/backoff or internal success-after-fallback logging.

## What This Changes

The issue asks the exec approval follow-up fallback to retry agent session resume, remove the internal resume-failure prefix from user-facing direct delivery, log resume failures internally, and preserve command result content.

## Best Possible Solution

Land a small agents-focused fix: add bounded retry/backoff around the agent resume/follow-up call, remove the resume-failure prefix from every direct fallback delivery path, and emit sanitized internal logging only after resume attempts are exhausted. Keep the change in the exec approval follow-up module and its colocated tests rather than folding it into broader delivery policy work.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists by making `callGatewayTool` reject in the denied resume-failure path; the current test still expects the internal prefix in the Telegram message. Source inspection also shows one resume attempt and no retry/backoff.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix: retry the resume path briefly, then silently direct-deliver sanitized result content and log internal resume metadata. Current main is not sufficient because it only suppresses the prefix for successful code-0 fallback content.
