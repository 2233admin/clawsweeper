## Summary

Keep open. Current main still only partially addresses the report: successful code-0 direct fallback deliveries avoid the internal prefix, but denied and other non-success resume-failure fallbacks can still show it, the resume handoff is still single-shot, and successful direct fallback after a resume failure is not logged with the requested diagnostics.

## What This Changes

The issue requests retry/backoff for exec approval session-resume follow-up, removal of the internal fallback prefix from user-facing direct delivery, and internal-only diagnostics when fallback is needed.

## Best Possible Solution

Land a focused agents fix in `src/agents/bash-tools.exec-approval-followup.ts`: retry the `agent` resume/follow-up call with bounded short backoff, silently preserve sanitized command status content for every direct fallback result kind, and emit one sanitized internal warning after resume attempts are exhausted. Keep broader delivery-ready refactors separate.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists by making `callGatewayTool` reject in the denied resume-failure path; the current colocated test expects Telegram content to include the internal prefix. Source inspection also shows there is no retry/backoff around the resume call.

## Solution Assessment

Yes, the requested focused fix is the narrowest maintainable direction. Current main’s success-only prefix suppression is incomplete because it leaves denied/non-success fallback messages prefixed and does not add retry/backoff or internal diagnostics.
