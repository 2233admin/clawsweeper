## Summary

Keep open. Current main only partially addresses the report: successful code-0 direct fallback no longer shows the internal prefix, but denied and non-success fallback deliveries still do, session resume is attempted once with no backoff, and successful direct fallback after resume failure is not logged internally.

## What This Changes

The issue requests retry/backoff for exec approval follow-up session resume failures, removal of the internal resume-failure prefix from direct fallback messages, internal-only logging, and preservation of command result content.

## Best Possible Solution

Land a focused agents fix in `src/agents/bash-tools.exec-approval-followup.ts`: add bounded retry/backoff around the session-resume `agent` call, remove the resume-failure prefix from every direct fallback message, and log a sanitized internal warning only after resume attempts are exhausted. Keep broader delivery policy or task-registry changes separate.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction exists by making `callGatewayTool` reject in the denied resume-failure path; the current colocated test expects the direct Telegram content to include the internal prefix. Source inspection also shows exactly one resume attempt before fallback.

## Solution Assessment

No. Current main is only a partial solution because it suppresses the prefix for successful code-0 completions but leaves denied/non-success fallback messages prefixed and does not add retry/backoff or internal logging.
