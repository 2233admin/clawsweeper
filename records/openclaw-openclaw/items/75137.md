## Summary

Keep open. The report is fresh, concrete, and still distinct from the Gateway/plugin-runtime CPU cluster: it names the separate `openclaw-tui` process in local TUI mode, while current `main` only shows adjacent TUI status/watchdog and dependency changes, not a proven idle CPU fix.

## What This Changes

The issue reports that `openclaw-tui` consumes roughly 89-99% CPU while idle or during message turns in `openclaw chat` / `openclaw tui --local`.

## Best Possible Solution

Keep the issue open, collect `perf top`, `strace`, or a V8 CPU profile from an affected idle TUI process, then fix the smallest proven loop in `src/tui` or upstream `pi-tui`. If profiling instead shows Gateway/runtime-deps starvation, preserve this issue's TUI-process evidence and link it to the Gateway tracker rather than closing it as the same bug prematurely.
