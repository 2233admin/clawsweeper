## Summary

Keep open. Current main and v2026.4.26 still apply symmetric jitter after a finite `retryAfterMs`, so the reported lower-bound violation is not implemented on main; the paired #68541 remains open and #68600 is closed/unmerged, so this PR is still the active implementation candidate pending maintainer review.

## What This Changes

The PR changes `retryAsync` to use positive-only jitter when a server-supplied `retryAfterMs` can be honored, and adds regression coverage for adversarial, boundary, capped, and non-integer retry-after delays.

## Best Possible Solution

Land or replace this with a focused retry helper fix that preserves `Retry-After` as a lower bound whenever the configured cap can honor it, keeps ordinary exponential backoff behavior unchanged, and locks the contract with targeted unit tests for adversarial jitter, cap equality, cap overflow, and fractional retry-after values.
