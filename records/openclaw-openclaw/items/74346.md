## Summary

Keep open. Current main still treats any live recorded PID as an active bundled runtime-deps lock owner, so deterministic PID reuse inside Docker can block the new gateway process. The related runtime-deps reports are adjacent, but this same-version Docker lock predicate bug is not already fixed or clearly superseded.

## What This Changes

The issue requests fixing bundled plugin runtime-deps lock staleness so Docker restarts do not treat a reused container PID as a live lock owner.

## Best Possible Solution

Keep this issue open and land a narrow runtime-deps lock hardening change: record Linux process starttime when acquiring the bundled runtime-deps lock, treat a live PID with a mismatched recorded starttime as stale, preserve legacy lock handling, and add focused tests for Docker-style PID reuse without weakening active-install protection.

## Repair Work Prompt

Fix OpenClaw issue #74346: bundled plugin runtime-deps install locks can survive Docker gateway restarts because `shouldRemoveRuntimeDepsLock()` treats a reused live PID as the original owner. Keep the fix scoped to runtime-deps lock ownership. In `src/plugins/bundled-runtime-deps.ts`, extend the lock owner payload written by both sync and async lock acquisition paths to include Linux process starttime when available, preferably by reusing `getProcessStartTime`/`isPidAlive` from `src/shared/pid-alive.ts` instead of duplicating `/proc` parsing. Update `readRuntimeDepsLockOwner()` and `shouldRemoveRuntimeDepsLock()` so a lock is removable when the PID is dead or when a stored starttime is present and differs from the current starttime for that PID. Preserve backward compatibility with existing `{pid, createdAtMs}` owner files and do not make active long-running installs expire by age alone; keep the intent of the existing active-lock test. Add focused tests in `src/plugins/bundled-runtime-deps.test.ts` for live same PID with mismatched starttime (remove), live same PID with matching starttime (keep), legacy owner files, dead PID, and ownerless stale behavior. Do not broaden this into cross-version runtime-deps cleanup for #73520. Add a user-facing changelog fix entry if current repo policy expects one for Docker/runtime-deps fixes; no docs change is required unless adding a short workaround. Validate with the targeted bundled runtime-deps test and targeted formatting; use Testbox for the changed gate before merge.
