## Summary

Keep open. Current main still lacks the requested actual-runtime-model HTTP response metadata on the Gateway OpenAI-compatible endpoints, and the existing discussion correctly frames the remaining work as a public API-contract decision rather than cleanup-close material.

## What This Changes

The issue requests an `X-Actual-Model` response header, or equivalent transport metadata, that exposes the model that actually processed each Gateway OpenAI-compatible HTTP request.

## Best Possible Solution

Define and document an additive Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming behavior, then wire authoritative provider/model data from the execution path into that contract with focused tests.
