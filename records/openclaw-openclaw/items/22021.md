## Summary

Keep open. Current main still lacks an actual-runtime-model transport surface on the Gateway OpenAI-compatible HTTP endpoints: the handlers echo request-side model ids, the existing selected-model callback remains below the HTTP layer, and the streaming paths flush headers before a fallback-selected model can be represented as a normal response header. The prior maintainer review in the discussion reached the same API-contract concern, and the relevant files have not changed since that review.

## What This Changes

The issue requests an `X-Actual-Model` response header, or equivalent transport metadata, so clients can see which backend model processed each Gateway HTTP request.

## Best Possible Solution

Define an additive Gateway actual-model metadata contract for `/v1/chat/completions` and `/v1/responses`, covering both non-streaming and streaming behavior, then wire the existing actual selected provider/model data into that contract with focused tests and public docs.
