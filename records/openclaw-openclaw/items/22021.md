## Summary

Current main still does not implement `X-Actual-Model` or an equivalent actual runtime provider/model field on the Gateway OpenAI-compatible HTTP responses. The Chat Completions and OpenResponses handlers continue to preserve the request-side compatibility `model` value in response bodies and SSE chunks, while the actual selected model callback exists only on internal reply/channel paths. Related open items #51441 and #7006 cover broader resolved-model observability, and open PR #54985 is adjacent outbound LLM request-context header work, but none supersedes this transport-response request.

## Best Possible Solution

Keep this open. Maintainers should define a Gateway HTTP contract for actual runtime provider/model metadata, thread selected provider/model from fallback execution into `src/gateway/openai-http.ts` and `src/gateway/openresponses-http.ts`, preserve the existing compatibility `model` response semantics, explicitly define non-streaming and SSE behavior, document it on the public Gateway HTTP API docs, and add focused tests for both compatibility endpoints.

## Risks / Open Questions

- A plain HTTP response header is straightforward for non-streaming responses but not for streaming once SSE headers have been flushed; maintainers need to define the streaming contract explicitly.
- Changing the existing response body `model` field to the backend runtime model could break clients relying on the documented agent-target compatibility semantics.
- There is overlap with broader resolved-model observability work in #51441, #7006, and adjacent outbound-header work in #54985, so naming and scope should be kept consistent.
