## Summary

Keep open. Current main still lacks an actual-runtime-model response metadata contract for the OpenAI-compatible Gateway endpoints: the handlers use the request-side agent target in response bodies/chunks, while the actual selected model callback remains below the HTTP layer. The related open PR #54985 is adjacent request-context work, but it does not implement the requested `X-Actual-Model` response surface or resolve the streaming contract problem.

## What This Changes

Request to expose the model that actually processed each Gateway OpenAI-compatible HTTP request, ideally via an `X-Actual-Model` response header, for runtime transparency and fallback debugging.

## Best Possible Solution

Define and document a Gateway actual-model metadata contract for both `/v1/chat/completions` and `/v1/responses`, covering non-streaming and streaming behavior, then wire selected provider/model data from `onModelSelected` or equivalent run metadata with focused tests and docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: inspect the current Gateway HTTP handlers and observe that no actual-model response header or SSE/body metadata is emitted, while response bodies and chunks use the request-side `model` value.

## Solution Assessment

Unclear. A response header is plausible for non-streaming responses, but it is not sufficient by itself for streaming fallback cases because headers are flushed before late model selection; the maintainable fix needs an explicit streaming metadata contract.
