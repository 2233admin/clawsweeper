## Summary

Keep open. Current main still leaves the reported metadata-free different-name QMD conflict unresolved: verified path/pattern matches can be rebound, but name-only conflicts still fall through to a generic skipped-add warning while searches continue to target the configured managed collection name. The existing discussion and open PR #72297 are aligned with that remaining gap.

## What This Changes

The issue asks OpenClaw to detect and reconcile or clearly surface QMD collection conflicts when the same path/pattern already exists under a different collection name.

## Best Possible Solution

Keep this issue open until a merged memory-core change either safely reconciles verified same-path/pattern different-name conflicts or surfaces a targeted manual repair hint for unverified name-only conflicts, with regression coverage that prevents destructive removal without metadata.

## Reproduction Assessment

Yes. The issue provides concrete QMD stderr for the `life` versus `life-main` conflict, and current tests model the metadata-free add-conflict path that still produces the skipped-add warning.

## Solution Assessment

No for current main. The narrow maintainable direction is metadata-gated rebind plus a clear manual repair hint for unverified conflicts; current main still leaves the user with a generic skipped-add warning and later searches against the missing configured collection.
