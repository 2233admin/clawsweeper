## Summary

Keep open. Current main still handles only same-name conflicts or different-name conflicts with verified path/pattern metadata; the metadata-free stale-name case described here still falls through to a skipped-add warning while searches keep targeting the configured managed collection name. The linked open PR #72297 is the focused implementation candidate for the remaining gap.

## What This Changes

A configured QMD memory collection should not fail searches when QMD already has the same path/pattern registered under a different collection name.

## Best Possible Solution

Review and land a narrow #72297-style change that keeps destructive collection removal limited to verified path/pattern matches, but turns metadata-free different-name conflicts into a clear manual repair hint naming the stale QMD collection so later searches do not fail opaquely.

## Reproduction Assessment

Yes. The report includes concrete QMD stderr, and current regression coverage models the same metadata-free conflict class by expecting the skipped-add warning after a name-only collection listing.

## Solution Assessment

Yes for the linked PR direction, not for current main. The maintainable fix is a targeted warning/manual repair path for unverified stale-name conflicts, while preserving metadata-gated automatic rebinding for safe cases.
