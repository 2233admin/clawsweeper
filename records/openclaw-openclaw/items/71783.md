## Summary

Keep open. Current main still cannot resolve the reported metadata-free different-name QMD conflict: it only rebinds another collection name when QMD list output exposes verified path and pattern metadata, and an open PR already targets the remaining warning/manual-repair behavior.

## What This Changes

The issue asks OpenClaw to handle or clearly report a QMD path/pattern conflict where the existing collection is registered under a different name than the configured managed collection.

## Best Possible Solution

Keep the issue open until PR #72297 or an equivalent memory-core change lands: preserve automatic rebind only for verified path/pattern matches, and add a targeted warning or manual repair hint for stderr-only conflicts such as `life` versus `life-main`.

## Reproduction Assessment

Yes. The report includes concrete QMD stderr for a `life` versus `life-main` path/pattern conflict, and current tests model the metadata-free add-conflict path that still produces a skipped-add warning.

## Solution Assessment

Yes for the linked implementation direction; no for current main. The narrow maintainable solution is metadata-gated rebind plus clear repair guidance for unverified conflicts, not blind removal of whatever collection name appears in stderr.
