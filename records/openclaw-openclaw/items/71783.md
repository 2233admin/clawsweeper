## Summary

Keep this issue open. Current main still leaves the metadata-free different-name QMD conflict unresolved: it only auto-rebinds when QMD exposes verifiable path/pattern metadata, while name-only conflict output still falls through to the skipped-add warning and searches continue to target the configured managed name. Open PR #72297 remains the active focused implementation candidate.

## Best Possible Solution

Keep the issue open and review #72297 or an equivalent narrow memory-core change. The fix should safely repair only when QMD exposes verifiable path/pattern metadata, and otherwise parse the stale collection name only to surface a clear manual repair hint such as removing the conflicting QMD collection before retrying boot/search.
