## Summary

Keep open. Current main still has the reported metadata-free, different-name QMD path/pattern conflict path: verified same-path conflicts can rebind, but a name-only QMD collection listing plus an add conflict still falls through to a skipped add while searches continue to target the configured collection name. Open PR #72297 is the active implementation candidate, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that QMD boot-time collection registration handle or clearly surface a same-path/pattern collection that already exists under a different name, such as `life-main` when OpenClaw searches `life`.

## Best Possible Solution

Land PR #72297 or an equivalent memory-core change that parses QMD's stale conflict name from add stderr, surfaces a targeted manual repair hint when `collection list` lacks metadata, keeps automatic rebind limited to verified same-path/pattern matches, and preserves regression coverage for the metadata-free conflict path.

## Reproduction Assessment

Yes. The issue includes concrete QMD stderr for the `life` versus `life-main` conflict, and current tests model the same metadata-free list plus add-conflict path that still produces a skipped-add warning on current main.

## Solution Assessment

No for current main; yes for the linked implementation direction. The narrow maintainable solution is a metadata-gated automatic rebind plus an explicit repair hint for stderr-only conflicts, not blindly removing a different-name collection based only on stderr.
