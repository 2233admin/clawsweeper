## Summary

Keep this PR open. The underlying Azure embeddings request is still not implemented on current main, and the proposed patch is useful source work, but it does not yet satisfy native Azure behavior because the request auth, endpoint detection, and API-version configuration remain incomplete.

## What This Changes

The PR updates the memory-lancedb direct OpenAI-compatible embedding client to detect Azure-looking base URLs and add Azure-specific request headers and query parameters.

## Best Possible Solution

Implement Azure-aware memory embeddings with a parsed-host helper shared with existing Azure surfaces, use an Azure request path that sends `api-key` without a bogus bearer header, expose an optional `embedding.apiVersion` in config/manifest/docs, and add focused Azure and non-Azure request-shaping tests for both direct and provider-backed embedding paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use Azure auth instead of adding api-key to the OpenAI client:** `extensions/memory-lancedb/index.ts:174-179`
  - body: The patch still constructs the generic OpenAI client with `apiKey`. In openai@6.34.0 that client always adds `Authorization: Bearer <apiKey>`; adding `defaultHeaders: { "api-key": apiKey }` does not replace it. Azure requests from this branch can therefore carry a bogus bearer token instead of using the Azure auth shape.
  - confidence: 0.87
- **[P2] Expose the Azure API version through config:** `extensions/memory-lancedb/index.ts:179`
  - body: The patch hardcodes `api-version=2024-02-01`, but memory-lancedb config and the plugin manifest reject any `apiVersion` override. Azure deployments that require a different supported API version would have no config path short of forking the plugin.
  - confidence: 0.9
- **[P2] Detect Azure endpoints by hostname and shared suffixes:** `extensions/memory-lancedb/index.ts:173`
  - body: `baseUrl?.includes(".openai.azure.com")` misses valid Azure host families already recognized elsewhere, including `*.services.ai.azure.com` and `*.cognitiveservices.azure.com`, and it is not constrained to the parsed hostname. Those endpoints would still use OpenAI-style auth/query handling.
  - confidence: 0.88
