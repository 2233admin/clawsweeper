## Summary

Keep open. Current main still has no `allowStateAccess` or equivalent state-only permission, and `agent_end` remains behind the raw-conversation `allowConversationAccess` gate even though the linked hardware/display plugin only wants lifecycle state. This needs maintainer product and security direction rather than an automatic cleanup close.

## What This Changes

Request a fine-grained state-only plugin hook permission or API so lifecycle/status plugins can observe agent state without full conversation-content access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and define a minimal state-only lifecycle contract for external plugins, either by documenting and hardening filtered agent-event subscriptions or by adding a sanitized hook/event shape with only run/session/state fields. The chosen path should update config schema/help, plugin hook docs, SDK overview, and loader/host-hook tests.

## Security Review

Status: needs_attention

Summary: This issue requests a security-relevant permission split; any fix must preserve the raw conversation gate while exposing only minimal lifecycle state.

Concerns:

- **[medium] Keep state access away from raw agent_end messages:** `src/plugins/hook-types.ts:272`
  - body: `agent_end` currently includes `messages`, so state-only access must not allow the existing payload for non-bundled plugins without `allowConversationAccess`.
  - confidence: 0.95
- **[medium] Do not classify content-bearing hooks as state-only:** `src/plugins/hook-types.ts:198`
  - body: `before_agent_reply` carries `cleanedBody` and `before_tool_call` carries tool params, so a state API should use sanitized state events rather than broadening those hook payloads.
  - confidence: 0.9
- **[low] Document safe event-subscription filtering:** `src/plugins/host-hook-runtime.ts:227`
  - body: `registerAgentEventSubscription` can subscribe to multiple agent streams, including assistant/tool streams; if it becomes the state path, docs and tests should steer status plugins to lifecycle-only or sanitized state data.
  - confidence: 0.82

## Risks / Open Questions

- A state-only implementation could accidentally expose `agent_end.messages`, `before_agent_reply.cleanedBody`, or `before_tool_call.params` if it loosens existing hooks instead of using a sanitized payload.
- The existing agent-event subscription seam may be the right answer for lifecycle-only state, but it is not yet documented and tested as the replacement for this hardware/status use case.
- The report's claim that `before_agent_reply` and `before_tool_call` require `allowConversationAccess` does not match current main; the remaining confirmed gate is `agent_end`, and maintainers should decide whether separate state signals are needed for those phases.
