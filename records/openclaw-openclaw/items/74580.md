## Summary

Keep open. Current main still treats `agent_end` as a raw conversation hook, blocks non-bundled plugin registration without `allowConversationAccess`, and has no `allowStateAccess` or documented lightweight turn-ended plugin API. The linked OpenClaw Interaction Bridge is useful evidence of a real plugin-author use case, but the fix changes a security-relevant permission boundary and needs maintainer product/security direction rather than automatic cleanup.

## What This Changes

The issue requests a granular state-only plugin hook permission or equivalent lifecycle signal so external plugins can observe agent turn completion without full conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document a minimal lifecycle-only plugin contract for turn completion. The durable end state should include a sanitized payload with session/run/state fields only, clear cadence semantics, schema/help/docs updates, and focused loader/SDK contract tests proving that conversation content remains gated.

## Reproduction Assessment

Yes. A high-confidence focused reproduction is a non-bundled plugin that registers `agent_end` without `plugins.entries.<id>.hooks.allowConversationAccess=true`; current loader tests and registry code show the hook is blocked, while exact searches show no state-only alternative permission on main.

## Solution Assessment

Unclear. `allowStateAccess` on the existing `agent_end` payload is not the safest fix because that payload includes raw `messages`; a sanitized turn-ended event or a documented lifecycle subscription path would be safer, but maintainers need to choose the public plugin API boundary.

## Risks / Open Questions

- Relaxing `agent_end` directly would expose `messages` to plugins that were meant to receive only lifecycle state.
- Classifying `before_agent_reply` or `before_tool_call` as state-only would be unsafe because their payloads include reply text or tool parameters.
- The existing `registerAgentEventSubscription` seam may be technically usable, but the issue raises unresolved product questions about whether it is the documented lightweight per-turn status API plugin authors should rely on.
- Open PR #71399 may become relevant to the final design, but it is currently open and internal-hook focused rather than a merged solution for this external plugin permission request.
