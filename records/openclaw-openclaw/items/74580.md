## Summary

Keep open. Current main still treats `agent_end` as a conversation-content hook for non-bundled plugins, and there is no `allowStateAccess` flag or documented lightweight turn-ended contract that satisfies the narrowed state-only request. The linked OpenClaw Interaction Bridge example and follow-up clarification make this a real plugin API and security-boundary question rather than stale cleanup.

## What This Changes

Request a fine-grained state-only permission or lifecycle signal so non-bundled plugins can observe agent/turn completion without granting raw conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document a minimal lifecycle/turn-ended contract that exposes only safe fields such as `runId`, `sessionKey`, phase, success/error, and duration. The contract should include tests proving no prompt, response, tool params, or message arrays reach state-only plugins.

## Reproduction Assessment

Yes. A non-bundled plugin that registers `agent_end` without `plugins.entries.<id>.hooks.allowConversationAccess=true` is blocked by the current registry path, and `src/plugins/loader.test.ts` codifies that behavior.

## Solution Assessment

Unclear. The requested `allowStateAccess` flag is plausible, but the current `agent_end` payload contains raw messages, so the safer solution is a sanitized lifecycle-only event or a documented event-subscription path rather than reclassifying the existing hook payload.
