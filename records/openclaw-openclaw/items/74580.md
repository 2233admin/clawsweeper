## Summary

Keep open. Current main still classifies `agent_end` as a conversation-content hook, still blocks non-bundled `agent_end` registration unless `allowConversationAccess` is enabled, and does not expose an `allowStateAccess` permission or documented lightweight turn-ended contract. The existing `registerAgentEventSubscription` seam is relevant and was correctly noted in discussion, but it is currently documented as a generic workflow-monitor subscription path, not as the narrow state-only `agent_end` replacement requested here.

## What This Changes

Request a state-only plugin hook permission or equivalent lifecycle signal so non-bundled plugins can observe agent turn completion without full conversation access.

## Best Possible Solution

Keep raw-content hooks behind `allowConversationAccess` and add or document a sanitized lifecycle/turn-ended contract that exposes only fields such as `runId`, `sessionKey`, terminal phase, success/error, and duration. If maintainers choose the existing event-subscription seam, clarify per-run versus per-turn behavior and add contract tests proving no conversation text is delivered.

## Reproduction Assessment

Yes. The current limitation is reproducible by loading a non-bundled plugin that registers `agent_end` without `allowConversationAccess`; `src/plugins/loader.test.ts` already encodes that registration as blocked.

## Solution Assessment

Unclear. A new `allowStateAccess` flag is plausible, but the safest shape is likely a sanitized lifecycle-only event or documented subscription path rather than reclassifying the existing `agent_end` payload that includes `messages`.
