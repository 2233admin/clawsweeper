## Summary

Current main still appears to contain the reported ACP bridge/runtime boundary bug: the debug ACP client creates a metadata-free bridge session, the bridge default still uses an `acp:<uuid>` Gateway session key, and ACP reply dispatch treats ACP-shaped keys without stored ACP metadata as stale `ACP_SESSION_INIT_FAILED` instead of falling through to normal Gateway-backed generation.

## What This Changes

The issue asks for `openclaw acp client` bridge sessions to stream assistant chunks instead of failing first prompt handling with `acp_session_init_failed`.

## Best Possible Solution

Keep this open and fix the ACP bridge/runtime boundary. Default Gateway-backed `openclaw acp` bridge sessions should use a key shape that ACP runtime dispatch does not claim as a metadata-backed ACPX session, while preserving explicit `_meta.sessionKey`, `_meta.sessionLabel`, `--session`, `loadSession`, `/acp spawn`, configured ACP bindings, and `sessions_spawn({ runtime: "acp" })` behavior. Add a regression for the default first-prompt path proving `agent_message_chunk` output instead of `ACP_SESSION_INIT_FAILED`, then update `docs/cli/acp.md` and the changelog if the visible default key wording changes.

## Repair Work Prompt

Fix the default Gateway-backed `openclaw acp` bridge first-prompt failure from this issue. Current main creates metadata-free bridge sessions through `createAcpClient`/`AcpGatewayAgent.newSession`, gives them the fallback key `acp:<uuid>`, Gateway canonicalizes that into an ACP-shaped session key, and ACP runtime dispatch handles it as stale `ACP_SESSION_INIT_FAILED` instead of allowing normal Gateway-backed generation.

Expected fix boundary: make default Gateway-backed ACP bridge sessions use a bridge-specific key shape that `isAcpSessionKey` and ACP runtime dispatch do not claim as an ACPX/runtime session. Preserve explicit `_meta.sessionKey`, `_meta.sessionLabel`, CLI `--session`, `loadSession`, `/acp spawn`, configured ACP bindings, and `sessions_spawn({ runtime: "acp" })` behavior. Avoid a broad change that weakens real ACP runtime detection or stale-binding cleanup.

Add focused regression coverage for the default metadata-free `newSession` plus first `prompt` path proving assistant `agent_message_chunk` output and no `ACP_SESSION_INIT_FAILED`. Update `docs/cli/acp.md` and add a changelog fix entry if the visible default session key wording changes. Do not touch unrelated provider/channel behavior.
