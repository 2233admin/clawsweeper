## Summary

Current main still has the reported ACP bridge failure path. The debug ACP client creates a metadata-free session, `AcpGatewayAgent.newSession` still defaults that session to an `acp:<uuid>` Gateway key, the Gateway passes the canonicalized key into the ACP reply-dispatch hook, and ACP resolution still treats ACP-shaped keys without `SessionAcpMeta` as stale `ACP_SESSION_INIT_FAILED` instead of falling through to normal Gateway-backed generation.

## Best Possible Solution

Keep this open and fix the ACP bridge/runtime boundary. Preferred direction: make default Gateway-backed `openclaw acp` bridge sessions use a bridge-specific key shape that `isAcpSessionKey` and ACP runtime dispatch do not claim, while preserving explicit `_meta.sessionKey`, `--session`, `/acp spawn`, and `sessions_spawn({ runtime: "acp" })` behavior. Add a focused regression for the default first-prompt path proving assistant `agent_message_chunk` output instead of `ACP_SESSION_INIT_FAILED`, then update `docs/cli/acp.md` and the changelog if the default key wording changes.

## Clownfish Work Prompt

Fix the default ACP bridge session-init failure described by this issue. Current main behavior: `openclaw acp client` calls `client.newSession({ cwd, mcpServers: [] })`; `AcpGatewayAgent.newSession` falls back to `acp:${sessionId}`; Gateway canonicalizes/routes that ACP-shaped key through chat dispatch; ACP `reply_dispatch` resolves it as an ACP runtime session; because no `SessionAcpMeta` exists, `AcpSessionManager.resolveSession` returns stale `ACP_SESSION_INIT_FAILED`, so the client can see echoed input plus `end_turn`/error instead of assistant `agent_message_chunk` output. Implement the smallest fix at the bridge/runtime boundary. Prefer preventing default Gateway-backed ACP bridge sessions from using a key shape that `isAcpSessionKey`/ACP runtime dispatch claims, unless after reading the code you determine valid metadata persistence is the safer intended contract. Preserve explicit `_meta.sessionKey`, `--session`, `/acp spawn`, and `sessions_spawn({ runtime: "acp" })` semantics for real ACP runtime sessions. Add focused regression coverage for the default `createAcpClient`/`AcpGatewayAgent.newSession` first-prompt path proving a Gateway-backed assistant chunk is emitted and ACP runtime stale handling is not invoked. Update `docs/cli/acp.md` if the default bridge session key wording changes. Add a user-facing changelog fix entry under the active version with `Thanks @ninaopenclaw`; do not credit Codex/OpenClaw. Do not change unrelated ACPX runtime behavior, provider auth/env stripping, or broad session-store semantics.
