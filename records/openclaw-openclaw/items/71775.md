## Summary

Keep open. Current main still performs the documented 4:00 AM daily session rollover that matches the reported overnight context loss, and the remaining question is whether maintainers want a scheduled-reset continuity change or whether this specific WeChat case is caused by external plugin/session-key churn. Related PR #48235 is an open, unmerged implementation candidate for the generic continuity-hint path, so closing this issue now would be premature.

## What This Changes

The issue reports overnight WeChat conversation context loss after resuming around 5 AM without using `/new`, `/reset`, or `/compact`.

## Best Possible Solution

Keep the documented daily reset policy unless maintainers choose a product change, but address the user-visible amnesia either by landing a reviewed continuity mechanism for automatic daily/idle resets or by proving and fixing a WeChat/session-key stability regression with focused logs and tests.

## Reproduction Assessment

Unclear. The generic overnight rollover is reproducible from current source and test coverage by sending the next message after the 4:00 AM boundary, but the WeChat-specific regression lacks the plugin/config/session-store evidence needed for high-confidence reproduction.

## Solution Assessment

Unclear. The narrowest likely core direction is the continuity-hint approach tracked by #48235, while a WeChat-specific fix should wait until maintainers can separate core scheduled reset behavior from plugin or account routing drift.
