## Summary

Keep open. Current main does not implement the PR's intended JWT-expiry short-circuit for stale OpenAI Codex OAuth credentials, and the proposed change appears narrow, provider-local, and consistent with the existing OAuth manager locking/persistence design. I found no blocking correctness or security findings in the diff.

## What This Changes

The PR adds an OpenAI Codex OAuth refresh short-circuit that trusts a cached access-token JWT `exp` claim when it remains beyond the shared refresh margin, caps the corrected `expires` value at 30 minutes, and adds focused provider tests for JWT and fallback cases.

## Best Possible Solution

Land a provider-local fix if CI confirms it: keep the JWT-expiry correction inside the OpenAI plugin, continue relying on the shared OAuth manager for locking and persistence, and avoid adding core provider-specific policy beyond the existing SDK margin constant.

## Reproduction Assessment

Yes. The PR body and related reports give a concrete stale-`expires` plus valid-JWT access-token scenario, and current main confirms that once the stored profile is considered unusable it enters the OpenAI Codex HTTP refresh path without checking the JWT `exp` claim.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix I found: reuse the existing Codex JWT expiry helper and shared OAuth refresh margin, cap the bypass window, and let the existing OAuth manager persist the corrected credential.
