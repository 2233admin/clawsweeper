## Summary

Keep this PR open. Current main still does not choose outbound accounts from the target peer, so the reported Slack multi-account failure mode remains plausible, but the submitted patch is not safe to merge because it scans raw bindings and loses agent/account scope when multiple bindings share a peer.

## What This Changes

The PR adds a peer-id-based account lookup in `runMessageAction` before the existing default-account and agent-keyed binding fallback.

## Best Possible Solution

Revise this into a typed outbound account resolver near `runMessageAction`: keep explicit `--account` and `defaultAccountId` precedence, use route-only bindings, derive peer id/kind from the normalized target where possible, choose a peer-bound account only when it is unambiguous or properly scoped, and add targeted regression coverage for the Slack multi-account case plus ACP and same-peer collision cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Filter route bindings before peer account lookup:** `src/infra/outbound/message-action-runner.ts:849-851`
  - body: The new lookup iterates `cfg.bindings`, but that array contains both route bindings and `type: "acp"` bindings. ACP bindings are for persistent ACP routing and the existing account fallback intentionally filters them with `listRouteBindings`; a matching ACP entry can now set `params.accountId` and send through an account route logic should ignore.
  - confidence: 0.92
- **[P1] Preserve agent scope when peer matches are not unique:** `src/infra/outbound/message-action-runner.ts:862-866`
  - body: This fast path matches only `channel` plus `peer.id` and then skips the existing agent-keyed fallback. If two agents/accounts bind the same Slack channel, sends from the second agent will choose whichever binding appears first in `cfg.bindings`, causing the same cross-account misrouting the PR is meant to fix.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff touches outbound credential/account selection and can cross account boundaries by selecting non-route or first-match peer bindings.

Concerns:

- **[medium] Outbound sends can select the wrong account boundary:** `src/infra/outbound/message-action-runner.ts:849`
  - body: Because the new lookup scans raw bindings and ignores agent/account disambiguation for peer collisions, it can choose an account that the route binding system would not select. That can send under the wrong Slack bot identity or fail closed with permission errors.
