## Summary

Keep this PR open. Current main still has the raw iOS chat attachment path, so the PR addresses a real linked bug, but the latest head still has a correctness issue: the new chat policy is documented and tested as a 1600 px long-edge cap while it delegates to a width-only transcoder path, leaving portrait images above the stated pixel budget.

## What This Changes

The PR adds iOS chat image upload processing through `ChatImageProcessor` and `JPEGTranscoder`, updates `OpenClawChatViewModel` to store processed JPEG attachments, adds alpha flattening in the transcoder, and adds tests for resize, byte budget, metadata stripping, invalid data, and alpha handling.

## Best Possible Solution

Land a corrected version that applies the chat pixel cap to the actual oriented long edge, keeps the metadata-stripping and alpha-flattening behavior, adds portrait-image coverage that fails for 1600x2133-style output, and validates the OpenClawKit Swift tests plus an iPhone composer/send path before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clamp the actual long edge:** `apps/shared/OpenClawKit/Sources/OpenClawKit/ChatImageProcessor.swift:54-56`
  - body: `maxDimension` is documented and tested as a longest-edge cap, but this passes it as `maxWidthPx`. `JPEGTranscoder` only scales from oriented width, so portrait photos can still come out around 1600x2133, or avoid resizing entirely when width is already <=1600. Please compute the scale from `max(orientedWidth, orientedHeight)` or add a true long-edge transcoder mode, with portrait coverage.
