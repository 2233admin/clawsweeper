## Summary

Keep open. Current main still has the original iOS chat attachment problem, but this PR should not merge as-is because the branch documents a 1600 px longest-edge cap while the implementation still calls a width-based transcoder path; the contributor’s own 1600x2133 device evidence matches that source-level mismatch.

## What This Changes

The PR adds iOS chat attachment preprocessing that re-encodes selected images as resized JPEGs with source metadata stripped, runs the work off the MainActor, extends JPEGTranscoder alpha handling, and adds OpenClawKit Swift tests.

## Best Possible Solution

Land a corrected version that keeps image processing centralized in JPEGTranscoder, adds a true long-edge mode or equivalent adapter calculation, covers portrait and narrow-tall fixtures, preserves metadata-stripping and alpha-flattening tests, and removes or rewrites the misleading post-processing size guard.

## Reproduction Assessment

Yes. A high-confidence source reproduction is any portrait or narrow-tall image where oriented width is at or below the width cap, or where width clamping yields an output such as 1600x2133; the current PR path can exceed the claimed 1600 px longest edge.

## Solution Assessment

No, not in this revision. Reusing JPEGTranscoder off the MainActor is the right maintainable direction, but the branch needs a true longest-edge contract before it solves the stated iOS composer behavior cleanly.
