## Summary

Keep this PR open. Current main still has the raw iOS attachment path that #68524 reports, and the PR is aimed at the right product boundary, but the current head is not merge-ready because it promises a 1600px long-edge cap while still delegating to a width-only transcoder contract; it also leaves a misleading dead safety-net comment and lacks the required changelog entry. The earlier alpha/metadata review threads appear addressed in the current diff and discussion.

## What This Changes

The PR branch routes iOS chat image attachments through a new `ChatImageProcessor`, re-encodes them as resized JPEGs with source metadata stripped, adds JPEG alpha flattening in `JPEGTranscoder`, and adds Swift unit tests for image processing behavior.

## Best Possible Solution

Land a corrected branch that gives chat uploads a true longest-edge policy while keeping the shared transcoder contract deliberate and PhotoCapture stable, preserves the metadata-strip and alpha-flatten coverage, adds portrait/narrow-tall regression tests and changelog coverage, and leaves #68524 linked until the fixing PR merges.

## Reproduction Assessment

Yes. A synthetic portrait or narrow-tall image whose oriented width is <=1600 and height is >1600 exercises the current width-only path; the PR passes `maxDimension` into that API, so the output can exceed the advertised long-edge cap. I did not run tests because this was a read-only review, but the source-level path is direct.

## Solution Assessment

No, not yet. Reusing `JPEGTranscoder` off the MainActor is the right maintainable direction, but the branch needs a true long-edge adapter/API and matching portrait coverage before it solves the stated issue cleanly.
