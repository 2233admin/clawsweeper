## Summary

Keep this PR open. Current main still has the iOS composer bug that this PR targets, and the branch is useful active work, but the current head still clamps oriented width instead of the actual long edge; the contributor’s own device evidence showing 1600x2133 outputs supports that code-level finding. This needs author or maintainer correction and validation, not cleanup closure or an automated replacement PR.

## What This Changes

The PR adds an iOS chat attachment processor that re-encodes selected images through JPEGTranscoder off the MainActor, strips source metadata, stores attachments as JPEGs, adds alpha flattening in JPEGTranscoder, and adds Swift tests for resize, payload budget, metadata stripping, invalid data, and alpha handling.

## Best Possible Solution

Land a corrected version that implements a true long-edge image-transcode policy, keeps metadata stripping and alpha flattening centralized in the JPEGTranscoder path, adds portrait and tall/narrow image coverage, validates the OpenClawKit Swift tests plus an iPhone camera-roll composer/send round trip, and leaves the broader configurable media-limit work tracked in #67031.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Clamp the actual long edge:** `apps/shared/OpenClawKit/Sources/OpenClawKit/ChatImageProcessor.swift:54`
  - body: `maxDimension` is documented and tested as a longest-edge cap, but this passes it as `maxWidthPx`, which only constrains oriented width. Portrait images can still return around 1600x2133, and tall images whose width is already <=1600 may avoid resizing entirely, so add a true long-edge transcode mode or compute the scale from `max(width, height)` with portrait coverage.
  - confidence: 0.96
- **[P3] Remove the unreachable 5 MB post-check:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatViewModel.swift:1160-1163`
  - body: `processForUpload` maps `JPEGTranscodeError.sizeLimitExceeded` into an error before returning, so `processed.count > 5_000_000` cannot be reached through the current call path. The comment also describes a best-effort return path that the adapter does not implement, which can mislead future maintenance of the attachment gate.
  - confidence: 0.86
