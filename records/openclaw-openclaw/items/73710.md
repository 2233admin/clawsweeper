## Summary

Keep open. The PR addresses a real iOS composer bug tracked by #68524, and current main still sends raw attachment bytes, but the latest head still maps the chat policy's `maxDimension` to a width-only transcoder parameter. Portrait photos can remain taller than the promised 1600 px cap, matching the contributor's 1600x2133 device output, so the branch needs a targeted correction before merge.

## What This Changes

The PR adds a Swift chat image processor that re-encodes attachments through `JPEGTranscoder`, updates chat attachment storage to use processed JPEG data, adds alpha flattening in the transcoder, and adds OpenClawKit tests for resize, byte budget, metadata stripping, invalid data, and alpha handling.

## Best Possible Solution

Land a corrected version that clamps the actual oriented long edge, preserves the metadata-stripping and alpha-flattening behavior, adds portrait-image coverage that fails for 1600x2133-style output, and then validates the OpenClawKit Swift tests plus a real-device composer/send path before merging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Clamp the actual long edge:** `apps/shared/OpenClawKit/Sources/OpenClawKit/ChatImageProcessor.swift:54`
  - body: `maxDimension` is documented and tested as a long-edge cap, but this passes it to `JPEGTranscoder.maxWidthPx`. That helper scales from `orientedWidth`, so portrait camera photos can still come out around 1600x2133 instead of fitting within 1600 px on the long edge. Please compute the scale from `max(orientedWidth, orientedHeight)` or add a true long-edge transcoder mode, and add portrait coverage before merge.
  - confidence: 0.9
