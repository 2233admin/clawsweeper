## Summary

Current main does not implement the requested WhatsApp next-step guidance, so this should not be closed as obsolete. The PR is useful but not merge-ready because the new onboarding note is gated only on link success and can tell users to verify inbound routing even after setup leaves inbound DMs disabled or unapproved.

## What This Changes

This PR adds WhatsApp onboarding next-step copy, WhatsApp hello-world docs and CLI examples, setup tests/helpers, and a changelog entry for the WhatsApp onboarding gap.

## Best Possible Solution

Keep the PR open and update it so onboarding always gives a hello-world path, but derives the runtime note from the final WhatsApp config: outbound CLI guidance can stay explicit, while inbound guidance should only be shown when the selected DM policy and allowlist or pairing state make that path routable.

## Reproduction Assessment

Yes. The proposed logic can be reproduced by a finalize flow where WhatsApp linking succeeds and the user selects separate-phone setup with `dmPolicy: "disabled"`; the patched helper already queues that selection, and current inbound access control blocks that inbound route.

## Solution Assessment

No. The product direction is right, but the implementation should gate or tailor the next-steps note based on the final DM access policy instead of using link success alone.
