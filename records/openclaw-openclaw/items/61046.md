## Summary

Keep this PR open. Current main still has the provider-prefix display bug the PR targets, but the submitted patch is not merge-ready because it resolves bare configured rows under the inferred provider while leaving alias indexes keyed under the default provider.

## What This Changes

The PR changes `resolveConfiguredEntries` to infer a unique configured provider for bare model refs in configured model keys and fallbacks, and adds focused unit tests for bare refs, ambiguity, and explicit provider prefixes.

## Best Possible Solution

Land a narrow models-list fix that applies the same bare-ref provider inference to configured rows, fallback rows, alias index keys, and alias fallback resolution, with regression coverage for bare configured keys that have aliases and fallback values that reference those aliases.

## Reproduction Assessment

Yes. A config with `models.providers.google.models: [{ id: "gemini-2.0-flash-001" }]` plus a bare `agents.defaults.models` key reproduces the current-main provider drift by source inspection; adding `alias: "Gemini"` to that bare key reproduces the PR’s alias-index regression.

## Solution Assessment

No, not as submitted. Inferring the unique configured provider is the right direction, but the alias index must be built or translated with the same effective provider so aliases and alias fallbacks stay aligned with the displayed model key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep aliases keyed to the inferred provider:** `src/commands/models/list.configured.ts:76-80`
  - body: When a bare `agents.defaults.models` key has an alias, this change parses the configured row with the inferred provider, but `aliasesByKey` is still populated from an alias index built with `DEFAULT_PROVIDER`. The row can show as `google/<model>` while its alias remains stored under `openai/<model>`, dropping the alias tag and letting alias fallbacks resolve to the wrong provider.
  - confidence: 0.86
