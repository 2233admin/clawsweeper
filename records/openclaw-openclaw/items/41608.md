## Summary

Keep open. Current main still routes Slack inbound messages with raw Slack IDs, while configured route-binding peer IDs are only trimmed and compared/indexed literally. A binding that uses `channel:C...` will not match an inbound Slack peer ID of `C...`, so the route can still fall back to the default agent and produce `agent:main:...` session keys. The related fix PR #42162 is closed unmerged in the provided context.

## Best Possible Solution

Normalize Slack route-binding peer IDs without changing canonical session key shape. A conservative fix would let target-style binding IDs such as `channel:C123`, `user:U123`, `<@U123>`, and raw IDs match the raw IDs supplied by Slack inbound events, then add focused regression tests for Slack channel/group and DM bindings. If maintainers want raw IDs as the documented form, add config help or doctor warnings while preserving compatibility for existing target-style configs.
