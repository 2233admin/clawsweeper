## Summary

Current main still appears to miss Slack route bindings whose configured peer ID uses target-style forms such as `channel:C...`. Slack inbound routing passes raw Slack IDs into the generic matcher, while the route-binding path only trim-normalizes configured peer IDs and then compares them literally. The related normalization PR #42162 is closed and unmerged in the supplied context, so this issue still needs maintainer follow-up rather than cleanup closure.

## Best Possible Solution

Keep this issue open. A conservative fix should normalize Slack route-binding peer IDs before generic route matching, accepting raw IDs plus target-style forms such as `channel:C123`, `user:U123`, and `<@U123>`, while preserving canonical raw Slack IDs in session keys. Add focused Slack route/prepare regression tests and update config help/docs if raw provider IDs remain the preferred form.
