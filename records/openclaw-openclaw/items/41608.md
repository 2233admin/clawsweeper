## Summary

Keep this issue open. Current main still passes raw Slack IDs into route resolution, while route binding peer IDs are only trimmed and then compared literally. A binding with `peer.id: "channel:C..."` still will not match an inbound Slack peer ID of `C...`, so the route can fall back to the default agent and produce `agent:main:...` session keys. The related fix PR #42162 is closed unmerged in the provided context, and current tests cover raw Slack IDs plus Slack target parsing separately, not the reported prefixed route-binding shape.

## Best Possible Solution

Fix Slack route-binding normalization without changing canonical session key shape. A conservative path is to normalize Slack `bindings[].match.peer.id` values so target-style forms like `channel:C123`, `user:U123`, `<@U123>`, and raw IDs match the raw IDs supplied by inbound Slack events, then add focused regression tests for Slack channel/group and direct-message bindings. If maintainers prefer raw IDs as the documented form, add config help or doctor warnings while preserving compatibility for existing target-style configs.
