## Summary

Keep open. Current main at 20e21173715ef488850023a90f72585a01cc379f still does not expose WhatsApp message delete/revoke: discovery advertises only react/poll, the WhatsApp action adapter routes only react, and the runtime throws for every non-react action. Related #38723 is closed as duplicate, while #70813 is still open and unmerged as the focused implementation candidate.

## What This Changes

Add a WhatsApp message-tool delete/revoke action so agents can retract their own sent WhatsApp messages through the bundled WhatsApp plugin.

## Best Possible Solution

Keep this issue open as the canonical bundled WhatsApp plugin request until #70813 or a similar focused PR lands. The best fix belongs in the WhatsApp plugin action path: advertise and accept the agreed public action name, route through the gateway-owned WhatsApp runtime, call Baileys with an own-message delete key, enforce fromMe/target authorization and group participant handling, and update focused WhatsApp action/send tests plus user-facing docs/changelog.

## Risks / Open Questions

- Message delete/revoke is a destructive channel mutation; the implementation needs strict bot-owned/fromMe behavior and existing outbound target authorization so it cannot become a delete-others/admin-delete surface.
- There is a public action naming question: this issue asks for delete/revoke, while #70813 uses edit/unsend terminology. Maintainers should settle whether delete is the public action, an alias, or intentionally documented as unsend.
- Live WhatsApp server behavior, including delete windows and group-key requirements, still needs implementation-time validation; source inspection only proves OpenClaw rejects the action before transport.
