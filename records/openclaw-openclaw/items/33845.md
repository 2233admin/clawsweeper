## Summary

Keep open. Current main still keeps Slack thread participation in process memory only, so the user-visible restart bug is real and this PR is not obsolete. The latest branch is focused, but it still introduces a blocking local-file hydration issue and now overlaps with the merged plugin state-store foundation plus the older open Slack persistence PR, so it needs maintainer direction before merge.

## What This Changes

This PR persists the Slack thread participation cache to a JSON file, adds persistence/reset test helpers, and expands Slack cache tests for restart survival, TTL, clear, corrupt/oversized files, capacity, and file-mode behavior.

## Best Possible Solution

Land one focused Slack restart-survival implementation that uses the approved plugin keyed store or an explicitly approved hardened Slack-local store, preserves the in-memory hot path if needed, and consolidates or closes the overlapping Slack persistence PR. The final patch should reject non-regular state files before hydration, clean up failed temp writes, cover restart/TTL/clear/corrupt/oversized/capacity/storage-failure cases, and keep the Slack API barrel free of test-only helpers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject non-regular state files before reading:** `extensions/slack/src/sent-thread-cache.ts:103-110`
  - body: `hydrateFromDisk` still calls `statSync` and then `readFileSync` on the persistence path without checking `lstat`/`isFile`. A symlink, FIFO, or device node in the state path can bypass the intended regular-file size guard or block the first Slack event while the gateway reads it. Reject non-regular files before reading, or move this onto the shared keyed-store primitive.
  - confidence: 0.82
- **[P3] Delete temp files when rename fails:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: `persistToDisk` writes a UUID temp JSON file and then renames it into place, but if `renameSync` throws the scheduled persist path swallows the error and leaves that temp file behind. Repeated locked-path or permission failures can accumulate stale `*.tmp` files containing Slack thread identifiers; wrap the rename so the temp path is removed on failure.
  - confidence: 0.83

## Security Review

Status: needs_attention

Summary: The PR introduces local filesystem persistence for Slack thread participation; no dependency or CI supply-chain change was found, but the custom store still has concrete file-handling issues.

Concerns:

- **[medium] Hydration follows non-regular state files:** `extensions/slack/src/sent-thread-cache.ts:103`
  - body: The PR uses `statSync` and `readFileSync` on the final persist path without rejecting symlinks or special files, which can turn a tampered local state path into a local DoS or unintended read before Slack message handling continues.
  - confidence: 0.82
- **[low] Failed atomic rename leaves state temp files:** `extensions/slack/src/sent-thread-cache.ts:217`
  - body: If the final rename fails, the temp JSON file is not removed and the scheduled write swallows the error, allowing repeated failures to leave Slack thread identifiers in leftover temp files.
  - confidence: 0.83
