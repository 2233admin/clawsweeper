## Summary

Keep open. Current main still builds the published gateway runtime from `node:24-bookworm-slim` and the default runtime package list does not install `python3`; the only Python escape hatch is an opt-in build argument for local rebuilds, so users pulling the official image still hit the reported workspace `.py` script regression. The related open PRs address adjacent fs-safe fallback or a `python` symlink and do not restore `python3` in the published gateway image.

## What This Changes

The issue reports that the published gateway Docker image no longer includes `python3`, breaking user-authored workspace Python scripts that worked in earlier images.

## Best Possible Solution

Restore baseline `python3` in the published gateway runtime image, add a Dockerfile/runtime-image regression assertion, and update Docker docs plus changelog. If maintainers intentionally keep the gateway image interpreter-free, the docs and `.py` execution failure should make that breaking change explicit and actionable for official-image users.

## Reproduction Assessment

Yes. The issue gives direct `docker run` comparisons between v2026.4.27 and v2026.4.23, and current main plus the upstream Node slim Dockerfile confirm the official runtime path still does not install `python3` by default.

## Solution Assessment

Yes. Re-adding baseline `python3` is the narrowest compatibility restoration for workspace `.py` scripts in the published image; the current opt-in apt package hook is useful for local rebuilds but does not fix users pulling `ghcr.io/openclaw/openclaw:<version>`.
