## Summary

Keep open. Current main still builds the gateway runtime from the slim Node image and the default runtime apt package list still omits `python3`; the optional apt hook only helps local image builds, and the related open PRs address adjacent `python` symlink or fs-safe fallback behavior rather than restoring user-authored workspace `.py` script execution in pulled official images.

## What This Changes

Restore official gateway Docker image support for `python3` so workspace and skill `.py` scripts that worked before the v2026.4.x slim-image switch do not fail at runtime.

## Best Possible Solution

The desired end state is a published gateway image that can run baseline `python3` workspace scripts again, with a Dockerfile regression assertion and user-facing Docker docs/changelog aligned to that runtime contract.

## Reproduction Assessment

Yes. The issue gives direct `docker run` comparisons between affected and last-known-good release images, and current main source still predicts the missing-`python3` result because the runtime base is slim and the default apt list omits `python3`.

## Solution Assessment

Yes. Re-adding baseline `python3` to the official gateway runtime is the narrowest compatibility restoration for pulled images; a docs-only apt-package recipe helps custom builds but would not fix users consuming release tags.
