## Summary

Keep open. Current main still builds the gateway runtime from the slim Node image and does not install `python3` in the default runtime package set, while the related open work only addresses internal `fs-safe` helper behavior or a `python` symlink when `python3` already exists. The reported user-authored workspace `.py` script breakage therefore remains a real, separately actionable Docker runtime contract issue.

## What This Changes

The issue asks OpenClaw to restore, document, or clearly diagnose `python3` availability for user-authored workspace Python scripts in the gateway Docker image after the v2026.4.x slim-runtime switch.

## Best Possible Solution

Restore a clear Docker runtime contract for workspace Python. The strongest compatibility fix is to install baseline `python3` in the published gateway runtime image and update Docker docs, Dockerfile tests, and the changelog; if maintainers intentionally keep the image interpreter-free, the docs and `.py` execution diagnostics should make that breaking change explicit and actionable.

## Reproduction Assessment

Yes. The issue provides direct `docker run` commands comparing v2026.4.27 against v2026.4.23, and current main source confirms the default runtime image and package list still omit `python3`.

## Solution Assessment

Yes, restoring `python3` in the gateway runtime image is the narrowest way to preserve the pre-slim workspace script behavior. A docs-only escape hatch would be safer for image size but would leave the reported upgrade regression in place for users of the published image.
