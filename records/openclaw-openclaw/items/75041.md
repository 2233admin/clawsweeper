## Summary

Keep open. The prior ClawSweeper keep-open assessment still matches current main: the gateway runtime still uses the slim Node base, the default runtime package list still omits `python3`, and the optional apt-package hook only helps locally built images. The linked open PRs are related but do not restore user-authored workspace `.py` script execution in pulled official images.

## What This Changes

The issue asks OpenClaw to restore or explicitly document Python support for user-authored workspace `.py` scripts in the published gateway Docker image after v2026.4.x images stopped including `python3`.

## Best Possible Solution

Restore baseline `python3` in the published gateway runtime image, add a Dockerfile regression assertion plus image smoke proof, and update Docker docs and the changelog. If maintainers intentionally keep the gateway image interpreter-free, the docs and missing-`python3` failure path should make that breaking change explicit and actionable.

## Reproduction Assessment

Yes. The issue gives direct `docker run` comparisons between affected and last-known-good published tags, and current main source still predicts the missing-`python3` result because the runtime base is slim and the default package list omits `python3`.

## Solution Assessment

Yes. Re-adding baseline `python3` to the default runtime package list is the narrowest compatibility restoration for pulled official images; a docs-only apt-package workaround helps custom builds but does not fix users who upgrade release tags.
