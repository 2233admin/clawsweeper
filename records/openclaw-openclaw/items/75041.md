## Summary

Keep open. Current main and the latest shipped release still build the gateway runtime from the slim Node image and the default runtime apt package list still omits `python3`; the existing apt-package hook only helps local rebuilds. The related open PRs are adjacent fs-safe fallback and `python` symlink work, but neither restores user-authored workspace `.py` scripts in the pulled gateway image.

## What This Changes

Restore or clearly document Python availability for user-authored workspace `.py` scripts after the v2026.4.x gateway Docker image moved to a slim base without `python3`.

## Best Possible Solution

Restore baseline `python3` in the published gateway runtime image, add a Dockerfile regression assertion plus image smoke proof, and update Docker docs and the changelog. If maintainers intentionally keep the published image interpreter-free, then the docs and `.py` execution failure path should make that breaking contract explicit and actionable.

## Reproduction Assessment

Yes. The issue provides direct `docker run` comparisons between affected and last-known-good published tags, and current main plus `v2026.4.29` source still predict the missing-`python3` result for the gateway runtime image.

## Solution Assessment

Yes. Re-adding baseline `python3` is the narrow compatibility restoration for workspace `.py` scripts; the current build-arg hook is useful for custom local images but does not fix users pulling official release tags.
