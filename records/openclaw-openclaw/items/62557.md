## Summary

Keep open. Current main still handles Linux update restarts through the OpenClaw-managed systemd user-service path, while system-level OpenClaw gateway units are documented and discovered elsewhere. The reporter's setup is a valid supported supervisor shape, and the updater still lacks system-scope restart detection or accurate system-scope guidance.

## What This Changes

Request that `openclaw update` detect system-wide systemd gateway services and restart them, or print the correct `sudo systemctl restart ...` guidance instead of only using user-scope systemd commands.

## Best Possible Solution

Teach the Linux package-update restart path to recognize active system-scope OpenClaw gateway units using the existing service-discovery concepts, then preserve managed user-service behavior while adding explicit system-scope restart handling or guidance.

Fix https://github.com/openclaw/openclaw/issues/62557. `openclaw update` on Linux currently prepares restarts only through the managed user-level systemd service path and the detached helper emits `systemctl --user restart ...`. Add a conservative system-scope path for active OpenClaw gateway units such as `/etc/systemd/system/openclaw.service` or `openclaw-gateway.service`: detect the relevant system unit using existing `src/daemon/inspect.ts`/systemd discovery concepts, then either restart it only when already authorized or print exact `sudo systemctl restart <unit>` guidance. Do not install or rewrite system units, do not change service ownership policy, and do not add hidden privilege escalation. Preserve existing managed user-service behavior and update focused tests/docs. Add a changelog fix entry crediting `Thanks @robinchoice`.
