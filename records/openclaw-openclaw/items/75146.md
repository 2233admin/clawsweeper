## Summary

Keep open. Current main still has the mid-turn retry failure mode because the recovery result has no mid-turn provenance and the run loop retries with the original prompt; this PR targets that gap with a narrow source marker, continuation prompt, and focused regression coverage.

## What This Changes

The PR marks mid-turn precheck recovery results, retries those paths with a continuation prompt instead of the original user prompt, and adds run-loop regression tests for truncation and compaction recovery.

## Best Possible Solution

Land the narrow PR after the normal changed gate if maintainers agree with the continuation-prompt approach, preserving turn-start preflight retries while preventing mid-turn recovery from resubmitting the original user request.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static and testable: enable mid-turn precheck, trigger a post-tool-result precheck recovery, and observe current main continuing through runEmbeddedPiAgent with params.prompt again; the PR adds mocked coverage for the truncation and compaction variants.

## Solution Assessment

Yes. Tagging mid-turn recovery and using a continuation prompt is a narrow fix against the current runner interface; a native Pi continue path could be considered later, but it would be a broader integration change than this regression requires.
