## Summary

Keep open. Current main still does not pass a per-request timeout to bundled MCP `Client.callTool` calls, while this PR is an active implementation candidate for #60967. The latest PR head addresses the earlier 30s default regression with a separate resolved request timeout and 60s fallback, but it remains unmerged and has failing checks that need normal maintainer review before landing.

## What This Changes

This PR threads a resolved bundled MCP request timeout through transport config/runtime and passes it as `RequestOptions.timeout` to `Client.callTool`, with tests for the 60s fallback and explicit `connectionTimeoutMs` propagation.

## Best Possible Solution

Keep this PR open as the active implementation candidate for #60967. The best path is normal maintainer review: inspect the failing checks, decide whether to require a focused runtime test for the third `callTool` argument, then land this PR or an equivalent patch with a user-facing changelog entry if the timeout semantics are accepted.
