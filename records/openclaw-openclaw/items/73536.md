## Summary

Keep open. Current main still has the reported bundled MCP per-call timeout gap, but the latest PR head introduces a compatibility regression by treating every explicit `connectionTimeoutMs` value as the tool-call request timeout even though that field is documented as connection-only.

## What This Changes

The PR adds a resolved MCP request timeout, propagates it through MCP transport/session state, passes it as the third `Client.callTool` argument, and adds config-resolution tests.

## Best Possible Solution

Keep the PR open and adjust the timeout contract before merge: either add a documented public `mcp.servers.<name>.requestTimeoutMs` field with schema/docs/tests, or preserve the existing 60s request behavior for all connection-only configurations while allowing an explicit request-timeout opt-in. Then pass that resolved request timeout to `Client.callTool` and add a changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure a bundled MCP tool call that runs longer than 60 seconds and current main cannot override the MCP SDK request timeout because `callTool` receives no `RequestOptions.timeout`. The PR regression is also source-reproducible by setting `connectionTimeoutMs: 10000` and observing the resolved request timeout become 10 seconds on PR head.

## Solution Assessment

No. Passing a third-argument timeout to `Client.callTool` is the right SDK seam, but reusing the connection-only config field for explicit request budgets is not the safest compatibility path. A separate `requestTimeoutMs` config field, or a compatibility fallback that never shortens existing request budgets, is more maintainable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Do not reuse connection timeouts for tool-call requests:** `src/agents/mcp-transport-config.ts:74`
  - body: This returns any explicit `connectionTimeoutMs` as the per-call request timeout. Please use a separate request timeout field or a compatibility fallback that does not shorten existing connection-only configs.
