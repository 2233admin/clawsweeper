## Summary

Keep this PR open. Current main still does not pass MCP RequestOptions.timeout to bundled MCP Client.callTool calls, so the linked #60967 bug remains valid. The PR is the active focused implementation candidate, but its current patch still has a real default-timeout regression, a wrong config-key comment, and no focused test for the third callTool argument.

## Best Possible Solution

Keep this PR open as the active implementation candidate for #60967, but update it before merge: preserve the SDK's 60s request default for unconfigured servers or add a dedicated documented request/tool timeout field, fix the JSDoc config key to connectionTimeoutMs, add a focused test proving Client.callTool receives the expected RequestOptions.timeout, and include changelog/docs coverage if the public config semantics change.

## Risks / Open Questions

- As patched, unconfigured bundled MCP servers would get a 30,000 ms per-call timeout from connectionTimeoutMs, which is a regression from the MCP SDK's 60,000 ms request default for tools that currently finish in 31-59 seconds.
- The patch comment references mcp.servers.<name>.timeout even though the current public config field is mcp.servers.<name>.connectionTimeoutMs.
- There is no focused regression test that captures the third Client.callTool argument, so the intended RequestOptions.timeout propagation is not pinned.
- Docs and public config types currently describe connectionTimeoutMs as connection-only; reusing it for tool calls needs either deliberate docs/changelog coverage or a separate request timeout field.
