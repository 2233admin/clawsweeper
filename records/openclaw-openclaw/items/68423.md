## Summary

Keep this issue open. Current main still has the exact polluted-backup gap described: both async and sync `maybeRecoverSuspiciousConfigRead` parse `${configPath}.bak`, verify only that the backup fingerprint has `gatewayMode`, and copy it over the live config without checking for redacted secret placeholders. The existing redacted-placeholder guard protects last-known-good promotion/recovery, not the legacy `.bak` restore path, and the issue is paired with an open unmerged PR that uses closing syntax.

## What This Changes

Request to refuse redacted-placeholder-polluted config backups during recovery and backup creation, with deeper backup history and config validation support for detecting pollution.

## Best Possible Solution

Keep this issue open while PR #68462 is reviewed, or replace it with an equivalent maintainer-owned patch. The fix should validate parsed `.bak` contents before both async and sync restore copy operations, add regression coverage for polluted backup refusal, and separately decide whether source validation in `maintainConfigBackups`, backup retention policy, and backup-scanning CLI behavior belong in the same patch or follow-ups.
