## Summary

Keep this PR open for maintainer review. Current main still contains the WhatsApp group route bypass path, while this PR targets the implicated fallback and adds focused regression coverage; I did not find a concrete blocking defect in the updated diff.

## What This Changes

This PR changes WhatsApp inbound policy so only explicit `groupAllowFrom` contributes to the group route sender-bypass check, disables the allowFrom fallback at that call site, and adds regression tests for the no-groups allowlist case.

## Best Possible Solution

Land a narrow WhatsApp-owned fix that prevents DM allowFrom entries from enabling the no-groups group route bypass, keeps explicit `groupAllowFrom` behavior intact, and preserves sender fallback only where a group route is otherwise explicitly admitted.

## Reproduction Assessment

Yes. The current-main reproduction is a code-level path: configure WhatsApp `groupPolicy: "allowlist"` with DM `allowFrom` and no `groups`/`groupAllowFrom`; the fallback makes `hasGroupAllowFrom` true and the group route is allowed. The PR adds a focused unit test for that route-level failure mode.

## Solution Assessment

Yes for the reported route bypass. Disabling fallback only when computing the route-level `hasGroupAllowFrom` input is narrower than changing the shared fallback contract, and it leaves explicit group sender allowlists working.
