## Summary

Keep open. Current main still has the macOS companion behavior described in the report: signed builds default Sparkle update checks/downloads on, the About settings expose one default-on toggle that drives both checks and downloads, and the Swift config reader only audits/preserves suspicious update-channel-only clobbers rather than restoring the prior config before returning it. Core TypeScript config I/O has recovery for this clobber shape, but that does not fully close the companion-app data-loss path.

## Best Possible Solution

Keep this open for the macOS companion/update owner area. Split automatic checking from automatic downloading/installing in persisted state and UI, make downloads or install-on-quit prompt-only or opt-in, and add signed-build coverage that beta update selection requires explicit user opt-in. Separately, route companion config reads/writes through the core config I/O path or port the same last-known-good recovery into OpenClawConfigFile, with a regression test that an update.channel-only clobber is restored before the app or gateway consumes it.

## Risks / Open Questions

- Closing now would hide a remaining macOS companion data-loss path: the core config reader can restore suspicious clobbers, but the Swift companion reader still returns the clobbered root.
- The update UX still conflates automatic checks and automatic downloads behind one default-on setting, which does not match the reporter's expectation that installs should be explicit.
- There is no current macOS policy/test proving signed companion builds cannot select beta updates unless the user explicitly opts in, even though the current appcast points at stable releases.
