## Summary

Keep open. Current main has useful core config recovery and the current appcast is stable-oriented, but the macOS companion still does not satisfy the report: signed builds default Sparkle checks and downloads to on, and the Swift config reader still audits/preserves suspicious clobbers rather than restoring the user config before returning it.

## Best Possible Solution

Keep this open for the macOS companion/update owner area. The best path is to split automatic checks from automatic downloads/install in UI and persisted state, make downloads or install-on-quit prompt-only or opt-in, and add a signed-build policy/test that beta appcast/update selection requires explicit user opt-in. Separately, port the core last-known-good recovery behavior into the Swift config loader or route companion config reads/writes through the core config I/O path, with a regression test that an `update.channel`-only clobber is restored before the app or gateway consumes it.

## Risks / Open Questions

- Closing now would hide an app-specific data-loss path: the core config reader can recover, but the macOS companion's Swift reader still returns the clobbered config after auditing it.
- The auto-update behavior still needs macOS product judgment because automatic checks, automatic downloads, and install-on-quit are separate user expectations, while the current UI ties checks and downloads together behind one default-on toggle.
- The original beta install is less likely with the current stable appcast, but there is still no macOS policy/test proving signed companion builds cannot select beta updates unless the user explicitly opts in.
