## Summary

Keep open. Current main supports loading shared skills from `~/.openclaw/skills` and `~/.agents/skills`, but `openclaw skills install` still only targets the active agent workspace `skills/` directory. The request is distinct from the adjacent workspace-selection issues because it asks for an intentional shared/global install target.

## Best Possible Solution

Add an explicit opt-in global target for native ClawHub skill installs, most likely `openclaw skills install <slug> --global` targeting the managed shared root `~/.openclaw/skills`. Keep current workspace behavior unchanged, reject ambiguous combinations with `--agent`, print the resolved target path, cover CLI and installer behavior with focused tests, and update the CLI/skills/ClawHub docs. A config default such as `skills.install.defaultTarget` can be a follow-up if maintainers want policy beyond the explicit flag.
