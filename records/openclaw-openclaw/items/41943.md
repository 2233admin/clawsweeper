## Summary

Keep open. Current main still does not implement the PR's intended Feishu/Lark Axios HTTP proxy tunneling for `HTTPS_PROXY`/`HTTP_PROXY`; Feishu HTTP requests only get timeout injection, while proxy support in this plugin is currently WebSocket-only. The PR targets a real bundled Feishu plugin gap, but the patch should be revised for current main with NO_PROXY-aware target selection, type-safe Axios option forwarding, cache behavior, and focused regression tests before merge.

## Best Possible Solution

Keep this PR open and ask for a revision against current main: implement Feishu HTTP proxy support in `extensions/feishu/src/client.ts` with target-aware `HTTPS_PROXY`/`HTTP_PROXY` selection, preserve `NO_PROXY` behavior via the existing proxy-env helpers, handle proxy state in the client cache deliberately, add focused Feishu HTTP proxy/bypass tests, and keep the Axios option forwarding type-safe.

## Risks / Open Questions

- Merging the PR as-is could route Feishu/Lark HTTP requests through a proxy even when `NO_PROXY` should bypass the target host.
- A proxy agent captured at client construction needs a clear cache strategy, otherwise runtime proxy-env changes can leave cached Feishu clients using stale routing.
- The current PR shape needs a type-safe adapter because the Feishu wrapper currently accepts Lark SDK request options, while Axios-specific fields such as `httpsAgent` and `proxy` are not represented in the local code path.
- No focused Feishu HTTP proxy or bypass tests exist on current main.
