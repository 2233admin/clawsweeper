## Summary

Keep open. Current main still does not implement Feishu/Lark Axios HTTP proxy tunneling for `HTTPS_PROXY`/`HTTP_PROXY`: the Feishu HTTP wrapper injects timeouts only, while existing proxy support is limited to Feishu WebSocket and broader opt-in runtime proxy paths. This PR remains aimed at a real bundled Feishu plugin gap, but it needs revision for current main, NO_PROXY-aware routing, type-safe Axios option forwarding, and focused tests before it should merge.

## Best Possible Solution

Keep this PR open and ask for a revision: implement Feishu HTTP proxy support in `extensions/feishu/src/client.ts` with target-aware `HTTPS_PROXY`/`HTTP_PROXY` selection, preserve `NO_PROXY` behavior, decide how proxy env changes interact with the client cache, add focused Feishu HTTP proxy/bypass tests, and use a type-safe Axios option adapter aligned with the current Lark SDK and plugin SDK proxy helpers.

## Risks / Open Questions

- Merging the PR as-is could route Feishu/Lark HTTP requests through a proxy even when `NO_PROXY` should bypass the target host.
- The current PR shape is authored against older Feishu client code and does not align with current shared proxy helpers or the existing async `resolveAmbientNodeProxyAgent()` WebSocket path.
- A direct Axios-option injection needs type-safe handling because the Lark SDK `HttpRequestOptions` type does not currently declare `httpsAgent` or `proxy` fields.
- The current main test suite lacks regression coverage for Feishu HTTP proxy and bypass cases.
