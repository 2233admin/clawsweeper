## Summary

Keep open. Current main still does not implement the PR's intended Feishu/Lark Axios HTTP proxy tunneling for HTTPS_PROXY/HTTP_PROXY, so this PR is not obsolete. The change targets a real bundled Feishu plugin gap, but it needs revision before merge for NO_PROXY-aware routing, cache behavior, type-safe Axios option forwarding, dependency/package hygiene, and focused tests.

## Best Possible Solution

Keep this PR open and ask for a revision against current main: implement Feishu HTTP proxy support in extensions/feishu/src/client.ts with target-aware HTTPS_PROXY/HTTP_PROXY selection, preserve NO_PROXY behavior via the existing proxy-env helpers, handle proxy state in the client cache deliberately, keep Axios option forwarding type-safe, declare or avoid any new direct runtime dependency, and add focused Feishu HTTP proxy/bypass tests.

## Risks / Open Questions

- Merging the PR as-is could route Feishu/Lark HTTP requests through a proxy even when NO_PROXY should bypass the target host.
- A proxy agent captured at client construction needs a cache strategy, otherwise cached clients can use stale routing after proxy-env changes.
- Axios-specific options need a type-safe bridge because the local wrapper is typed as Lark.HttpRequestOptions, which does not declare httpsAgent or proxy.
- If the implementation imports https-proxy-agent directly from Feishu plugin code, extensions/feishu/package.json should declare the runtime dependency or the code should reuse an existing SDK helper.
- No focused Feishu HTTP proxy or bypass regression tests exist on current main.
