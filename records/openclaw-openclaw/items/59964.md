## Summary

Keep this PR open, but it is not mergeable as-is. Current main already has last-known-good startup and hot-reload recovery for globally invalid configs, but it does not include the PR's manual `openclaw config recover` command; the PR also introduces concrete recovery, notification, and docs defects that need rework against the current recovery path.

## What This Changes

The PR adds a proposal doc plus code for `openclaw config recover`, `.bak` backup candidate validation/restoration, broader config auto-recovery, startup recovery, and reload-time recovery notifications.

## Best Possible Solution

Rework the PR around current main's last-known-good recovery APIs: keep plugin-local invalidity isolated, resolve `$include` and env substitution before candidate validation, scope recovery notices to a trusted/current context, add focused CLI/gateway/config tests, and remove the proposal document from user docs.

## Reproduction Assessment

Yes for the PR review findings: the stale notification context, invalid dry-run success path, raw candidate validation, and committed non-English proposal are directly reproducible from the PR diff. I did not mutate the checkout or run tests under the read-only constraint.

## Solution Assessment

No. The current PR is not the best solution because it duplicates older `.bak` recovery logic instead of building on current last-known-good recovery and introduces notification/privacy and validation regressions.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope recovery alerts to the triggering write:** `src/gateway/config-reload.ts:112-124`
  - body: `lastWriteEvent` is retained and reused for every later recovery alert. A filesystem-only corrupt edit after an unrelated config write can therefore send recovery details to the previous session/thread; clear or match the write context before notifying.
  - confidence: 0.9
- **[P2] Resolve includes before validating recovery candidates:** `src/config/io.ts:1004`
  - body: Backups can be valid only after `$include` expansion and env substitution, but this path validates the raw parsed root. Split configs documented by OpenClaw will be marked invalid and unavailable for recovery.
  - confidence: 0.87
- **[P2] Clear invalid selections before reporting dry-run success:** `src/config/io.ts:2889-2890`
  - body: When `--from` names an invalid backup, the result still exposes `selectedSource`/`restoredPath`. `runConfigRecover` treats `selectedSource` as the dry-run success signal, so an invalid requested source can report that it would recover.
  - confidence: 0.92
- **[P2] Remove the proposal from user docs:** `docs/PR-PROPOSAL.md:3-5`
  - body: `docs/PR-PROPOSAL.md` is a PR design note, mostly Chinese, and says it should be rebased before submit. The repo docs are user-facing English source, so this should stay in the PR discussion or be removed.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR adds recovery-notice delivery logic that can leak recovery details to a stale chat/session context.

Concerns:

- **[medium] Recovery notice can leak to stale session:** `src/gateway/config-reload.ts:112`
  - body: The reload path stores the last config write event and reuses its `sessionKey` and `deliveryContext` for later recovery notifications. A later invalid direct edit with no matching write can send config recovery details to an unrelated conversation.
  - confidence: 0.9
