## Summary

Keep open. Current main still dispatches `before_model_resolve` from inside each embedded fallback candidate attempt, so an unconditional routing plugin can still rewrite every fallback candidate back to its selected provider/model. Main has the partial `runId`/`jobId`/current candidate metadata mitigation, but the public hook context still lacks `attempt`/`isFallbackRetry`, and docs still do not state a per-fallback-candidate contract.

## What This Changes

The issue asks OpenClaw to make `before_model_resolve` compatible with model fallback, either by firing it once per logical run or by exposing enough retry metadata for routing plugins to yield during fallback attempts.

## Best Possible Solution

OpenClaw should choose and ship an explicit hook contract. The strongest end state is a regression-tested model-routing path where a plugin-selected primary model can fail over without being re-overridden; if the maintained contract stays per-candidate, `PluginHookAgentContext` should expose retry metadata and the public docs/types should say so clearly.
