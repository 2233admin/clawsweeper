## Summary

Keep open. Current main still has Feishu reply, streaming-start fallback, and outbound card paths that can send table-heavy markdown through interactive-card delivery without a 230099/11310 table-limit classifier or retry to the existing plain-text table-conversion path. The issue is also paired with open implementation PRs, including a closing PR, so it should stay open until maintainers review and consolidate that work.

## What This Changes

The issue requests Feishu fallback or table conversion when interactive-card delivery hits Feishu's table-count limit for markdown-table-heavy responses.

## Best Possible Solution

Land or rework a focused Feishu plugin fix that classifies CardKit table-limit failures such as 230099/11310, retries affected reply, streaming-start fallback, and outbound card sends through the existing plain-text/table-conversion path or a safe table-rendering strategy, preserves threading and mentions, rethrows unrelated card failures, and adds regression coverage for each delivery surface.
