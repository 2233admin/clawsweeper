## Summary

Keep open. Current main still reloads history for the current in-memory TUI session after an in-process WebSocket reconnect, but a fresh `openclaw tui` startup derives its session from CLI/default inputs and has no persisted last-active TUI session to resume. The `tui-<uuid>` path is intentional `/new` isolation and should be preserved while adding an explicit resume path.

## What This Changes

The issue asks for TUI reconnects or restarts to resume the prior conversation session, or clearly warn/carry context when starting a fresh session.

## Best Possible Solution

Add an explicit TUI resume contract: persist the last active TUI session key per relevant gateway/local/agent scope, use it on fresh startup only when no explicit session was requested, visibly report fallback or fresh-session creation, and keep `/new` as an isolated new-session command.

## Repair Work Prompt

Fix the TUI fresh-process session resume bug tracked by https://github.com/openclaw/openclaw/issues/73546. Current main preserves in-process reconnect history for the existing `currentSessionKey`, but a new `openclaw tui` process derives its session only from `--session`/defaults and loses the last active TUI session. Implement a conservative resume contract in the TUI layer: remember the last active TUI session key per relevant gateway/local/agent scope, use it on startup only when no explicit `--session` was provided, and show a clear system/status message when resuming or when falling back to a fresh/default session. Preserve existing `/new` behavior that creates isolated `tui-<uuid>` sessions and do not turn `/new` into a shared reset. Cover gateway and local embedded mode if they share the affected startup path. Add focused tests for explicit session opt-out, default resume, `/new` isolation, and reconnect/history loading. Update TUI docs if user-visible startup/resume behavior changes. Do not change unrelated gateway reset/archive semantics or Control UI session persistence.
