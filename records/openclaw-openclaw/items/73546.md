## Summary

Keep open. Current main still starts `openclaw tui` without `--session` from the configured main/global key, and the fresh-process startup path does not persist or recover the last active `tui-<uuid>` session. The related #41168 report is useful context, but this issue is the narrower actionable TUI continuity bug.

## What This Changes

The issue requests that TUI reconnects or restarts resume the previous conversation, or visibly warn/carry context when starting a fresh session is unavoidable.

## Best Possible Solution

Add a TUI-owned resume contract: persist the last active TUI session key in scoped local state, update it when the TUI selects or switches sessions, use it on startup only when no explicit `--session` was supplied and the key is valid for the current agent/scope, and show a visible resumed/fallback/fresh-session status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: start TUI with no `--session`, create or switch to a `tui-<uuid>` session, exit, relaunch with no `--session`, and observe startup resolving back to main/global before loading history for that selected key.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the TUI startup/session-action boundary using existing session listing/history APIs plus scoped local state; changing gateway reset/archive semantics would be broader unless a regression proves the TUI boundary is insufficient.
