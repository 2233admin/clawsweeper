## Summary

Keep this PR open. Current main still does not implement the proposed non-interactive onboarding flags for trusted-proxy gateway auth, trusted proxy IPs, trusted-proxy allowUsers, or Control UI allowed origins. The underlying config, runtime auth model, and interactive configure flow already support those fields, so the PR remains a plausible focused onboarding parity change rather than obsolete or out of scope.

## Best Possible Solution

Keep this PR open, or replace it with an updated focused PR, to add non-interactive onboarding support for `gateway.auth.mode="trusted-proxy"`, `gateway.trustedProxies`, `gateway.auth.trustedProxy.userHeader`, `requiredHeaders`, `allowUsers`, and `gateway.controlUi.allowedOrigins`. The implementation should align with current runtime guards, explicitly handle token/password cleanup, make a deliberate `allowLoopback` decision, and include targeted tests plus docs tied to the trusted-proxy security model. Coordinate with #69066 for broader trust-model wording, but do not close this PR as implemented or superseded.

## Risks / Open Questions

- The PR as submitted spreads existing `gateway.auth` into trusted-proxy auth, which can preserve an existing token; current startup validation rejects trusted-proxy mode with a configured token, so the updated PR should explicitly clear mutually exclusive token/password fields when switching modes.
- The PR docs/example include `127.0.0.1` in trusted proxies but the diff does not add an `allowLoopback` flag; current main rejects loopback-source trusted-proxy auth unless `gateway.auth.trustedProxy.allowLoopback = true`, so maintainers need an explicit product/security decision there.
- Because this is a non-interactive path for a security-sensitive gateway auth mode, final review should verify proxy IP parsing, header normalization, requiredHeaders semantics, allowUsers behavior, Control UI allowed-origin handling, and docs warnings against the current trusted-proxy model.
