## Summary

Keep this issue open. Current main supports per-agent tool-level allow/deny, but the requested path-scoped allow/deny policy is still missing from the config schema and runtime file-tool enforcement.

## What This Changes

Request adding optional per-agent tool deny lists plus path allow/deny glob policies for write-capable tools, with validation and runtime scope-violation failures.

## Best Possible Solution

Keep the existing deny-first tool policy, then design one generic, schema-backed path policy for file-capable tools that uses canonical path checks, has clear host/sandbox semantics, and explicitly decides whether command execution is handled by exec sandboxing/approvals rather than brittle shell redirection parsing.
