## Summary

Keep open. Current main still lacks a QA bus surface for actual agent tool-call traces; the available planned-tool debug data is mock-provider-specific and does not satisfy the requested scenario assertion contract. The remaining work needs maintainer judgment on whether this belongs in the public QA message contract or a narrower qa-lab assertion API.

## What This Changes

The issue requests an optional tool-call trace on QA bus messages so qa-lab scenarios can assert that agents actually invoked expected tools instead of only matching response text.

## Best Possible Solution

Add a backwards-compatible QA trace surface sourced from actual tool-start events with sanitized tool names and arguments, then carry it through QA bus create, clone, snapshot, read, search, poll, and wait paths. The chosen surface should be documented as either a public QaBusMessage field or a narrower qa-lab assertion API, with focused qa-lab/qa-channel coverage and at least one scenario that fails when expected tool use is fabricated.
