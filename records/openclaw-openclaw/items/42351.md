## Summary

Current main still does not implement the requested Feishu handler for `im.chat.access_event.bot_p2p_chat_entered_v1`. The issue remains actionable: Feishu defaults to the WebSocket monitor path, that path registers a fixed handler map, and neither source nor tests/docs mention the reported access event. The related implementation PR #42434 is closed unmerged, so it is not a canonical fix to close against.

## Best Possible Solution

Keep this issue open. Add a narrow Feishu-owned no-op or low-noise handler for `im.chat.access_event.bot_p2p_chat_entered_v1` in `extensions/feishu/src/monitor.account.ts`, plus regression coverage proving the handler is registered on the WebSocket monitor path and does not stop the Feishu channel or Gateway transport. Maintainers should also verify whether the restart comes from Lark SDK retry behavior, event pressure, or an OpenClaw supervisor path before adding any broader generic restart handling.
