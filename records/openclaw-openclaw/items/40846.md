## Summary

Keep open. Current main still does not implement automatic macOS Gateway sleep prevention, so the PR is not obsolete, but the patch should not merge as-is because the helper launch has unresolved lifecycle, async error-handling, and command-resolution issues already confirmed in review discussion.

## What This Changes

The PR modifies `src/cli/gateway-cli/run-loop.ts` to add a Darwin-only `spawnCaffeinate()` helper that runs `caffeinate -i -w <pid>`, starts it when `runGatewayLoop` begins, and kills it during final cleanup.

## Best Possible Solution

Land this only after the Gateway lifecycle and command-execution edges are tightened: use `/usr/bin/caffeinate` or a validated absolute resolver, attach child `error` and useful exit handling before `unref()`, start the helper inside the same cleanup-protected scope as the lock, preserve the non-macOS no-op path, add run-loop tests, and document the default macOS power behavior if maintainers want it enabled.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Handle caffeinate spawn errors:** `src/cli/gateway-cli/run-loop.ts:36-39`
  - body: `spawn()` reports a missing or inaccessible executable through the child process `error` event, so the surrounding `try/catch` does not protect Gateway startup. Add an `error` listener before `unref()` so a broken helper launch logs and degrades instead of crashing with an unhandled child-process error.
  - confidence: 0.96
- **[P2] Start caffeinate inside the cleanup scope:** `src/cli/gateway-cli/run-loop.ts:64`
  - body: The helper is started before `acquireGatewayLock()`, but cleanup is only registered in the later `finally`. If lock acquisition throws, the helper can be left running outside the intended cleanup path; start it after the lock is acquired or wrap lock acquisition and helper startup in the same cleanup scope.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces a Gateway startup command-execution path that should avoid PATH-based executable resolution before merge.

Concerns:

- **[medium] Avoid PATH-based caffeinate execution:** `src/cli/gateway-cli/run-loop.ts:36`
  - body: `spawn("caffeinate", ...)` lets the launch environment decide which executable runs during Gateway startup. Because wrappers, dev shells, and launchd/service environments can affect PATH, this fixed macOS helper should use `/usr/bin/caffeinate` or another validated absolute path with explicit error handling.
  - confidence: 0.8
