## Summary

Keep open. Current main still has the Control UI terminal-event ordering that can expose both the committed assistant message and the live stream item, the latest release tag has the same ordering, and open PR #72004 is the direct closing implementation candidate for this issue.

## What This Changes

The issue requests a Control UI WebChat fix so assistant final or aborted messages render once instead of being duplicated when live stream state overlaps committed chat history.

## Best Possible Solution

Review and land PR #72004, or an equivalent narrow Control UI change that snapshots fallback stream text, clears chatStream/chatRunId/chatStreamStartedAt before mutating chatMessages in both final and aborted paths, and adds an order-sensitive regression test. Keep backend transcript dedupe tracked separately under #69208 or narrower linked issues.

## Reproduction Assessment

Yes. The issue has concrete cross-platform reports, and current main shows a source-level reproduction path: process an active-run final or aborted event while chatStream is populated, then renderChat can observe both the committed history item and the still-populated stream item.

## Solution Assessment

Yes for the client-side duplicate-render race. Clearing stream state before appending the final or aborted assistant message is the narrowest maintainable fix; backend transcript dedupe is a separate solution for the adjacent server-side reports.
