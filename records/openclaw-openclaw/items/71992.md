## Summary

Current main still has the Control UI webchat race described in this issue: `handleChatEvent` writes the final/aborted assistant message into `chatMessages` before clearing `chatStream`, while the renderer still passes both `chatMessages` and the live stream into `buildChatItems`. The provided GitHub context also shows an open PR with closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review/land PR #72004 or an equivalent patch that snapshots fallback stream text, clears `chatStream`/`chatRunId`/`chatStreamStartedAt` before mutating `chatMessages` in both `final` and `aborted`, and adds an order-sensitive regression test. Coordinate the separate backend/transcript duplicate cases under #69208 rather than closing this focused Control UI rendering race early.
