## Summary

Keep open. Current main still lacks a cold-start guardrail for short isolated cron agent turns, and an open PR already references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests warning, timeout-budget, or documentation guardrails for short isolated cron `agentTurn` jobs that omit `lightContext` and `toolsAllow`.

## Best Possible Solution

Land a narrow guardrail that preserves explicit timeout semantics while warning on create/update for risky isolated cron `agentTurn` jobs with short explicit timeouts and no lightweight context or tool allowlist, with public cron docs explaining the cold-start budget tradeoff.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create an isolated `agentTurn` cron job with `timeoutSeconds: 60`, no `lightContext`, and no `toolsAllow`; current main starts the timeout before embedded workspace/runtime/bootstrap setup, though this review did not run a live 60-second job.

## Solution Assessment

Unclear until PR #74893 is reviewed, but the direction is sound: a targeted warning and docs update is safer than silently inflating explicit user timeouts or changing scheduler-wide timeout semantics.
