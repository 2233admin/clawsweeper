## Summary

Keep open. Current main still preserves short explicit cron agent-turn timeouts, starts the deferred cron timer from an embedded-runner signal that fires before workspace/plugin/bootstrap setup, and documents light context/tools allowlists without the requested cold-start timeout caveat. The linked open PR #74893 already targets this issue, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue asks for a guardrail or documentation so short-timeout isolated cron agentTurn jobs without `lightContext` or a restricted `toolsAllow` do not spend the whole timeout budget on cold-start/bootstrap before doing useful work.

## Best Possible Solution

Review and land or revise PR #74893 so OpenClaw adds a narrow tested warning/guardrail for risky short-timeout isolated agentTurn jobs, plus docs that explain the bootstrap budget tradeoff, while preserving explicit user timeout semantics.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: create an isolated agentTurn cron job with a short explicit `timeoutSeconds`, no `lightContext`, and no restricted `toolsAllow`; current main preserves that timeout and starts it before bootstrap-heavy setup completes.

## Solution Assessment

Yes at the direction level. A creation/update warning plus docs is the narrowest maintainable fix; silently inflating explicit timeouts would change user-specified execution semantics and needs more product judgment.
