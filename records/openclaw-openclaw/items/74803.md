## Summary

Keep open. Current main has fixed the older cron-lane wait/deadlock class, but this report is a different cold-start budget gap: an explicit short `timeoutSeconds` still applies once execution is signaled, that signal fires before embedded runner workspace/bootstrap setup, and the docs expose `lightContext`/`toolsAllow` without the requested warning.

## What This Changes

The issue requests safer handling or documentation for short-timeout isolated `agentTurn` cron jobs that omit `lightContext` and `toolsAllow`.

## Best Possible Solution

Add a narrow, tested guardrail around isolated cron `agentTurn` creation/update for short explicit timeouts without `lightContext` or a restricted `toolsAllow`, and update the cron docs to explain cold-start/bootstrap budget and the recommended workaround without silently changing user-specified timeout semantics.
