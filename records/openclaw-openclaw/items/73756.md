## Summary

Keep this issue open. Current main still has the reported Telegram fail-open shape: `dmPolicy: "open"` returns before sender allowlist evaluation, account-level `allowFrom` can replace a restrictive top-level allowlist, and existing schema/docs/audit coverage mostly makes public access explicit rather than resolving the conflicting effective-policy case.

## Best Possible Solution

Keep this open for Telegram-owned access-control work. Define fail-closed effective-policy semantics when a restrictive top-level allowlist conflicts with account-level public DM settings, preserve an explicit intentional-public-bot path, add regression coverage for `mergeTelegramAccountConfig`, `enforceTelegramDmAccess`, direct event authorization, schema/setup/doctor/security/status surfaces, and coordinate with related PR #38683 rather than closing this issue as already handled.
