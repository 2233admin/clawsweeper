## Summary

Keep this PR open. Current main still has the Feishu outbound observability gap the PR targets: inbound handling logs only the dispatcher completion summary, while the reply dispatcher sends text, cards, and media without logging successful delivery results or Feishu message IDs. The PR is not obsolete, but it should be revised before merge to address the review concern that text/card fallback sends can be logged as replies even when Feishu fell back to a top-level send.

## Best Possible Solution

Keep the PR open and ask for a small rebase/revision: add success logging on the current reply-dispatcher text/card/media paths, preserve tests for direct send and reply/thread context, and make the logged mode reflect the actual send path. The cleanest fix is likely to have the Feishu send helpers return whether they used `reply` or fallback `send`, then have the dispatcher log that result instead of deriving mode only from `replyToMessageId`.

## Risks / Open Questions

- Merging the PR as described could produce misleading `reply ok` logs for text/card sends that actually fell back to direct top-level delivery after a withdrawn or missing reply target.
- The new log lines include Feishu chat IDs, reply target IDs, and message IDs. That is useful operational telemetry, but reviewers should confirm it is acceptable for normal runtime logs and does not introduce broader data exposure than existing Feishu logs.
