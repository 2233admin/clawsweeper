## Summary

Keep open. Current main still routes `openclaw system event --mode now` through the normal wake/heartbeat path: the CLI exposes no priority or bypass flag, the wake RPC accepts only mode/text, and the heartbeat runner intentionally skips when the target session lane is busy. That matches the reported delayed-alert failure mode rather than resolving it.

## Best Possible Solution

Keep the issue open for a maintainer design pass. The likely direction is an explicit priority/bypass contract on the system-event CLI/RPC surface that can surface an immediate operator/TUI alert without mutating an active session unsafely; prompt transcript injection may still need to wait for the session lane while a separate UI alert path bypasses it.

## Risks / Open Questions

- A naive bypass that writes directly into the session transcript could race active agent runs or corrupt session state, which is exactly why the current heartbeat path avoids interrupting a busy session lane.
- The wake schema accepts unknown fields for forward compatibility, so a caller could send `priority` today and have it silently ignored unless the CLI/RPC/runtime contract is made explicit and tested.
- System events feed prompt-visible `System:` lines with trust handling; any immediate alert path needs to preserve untrusted-event downgrades and avoid creating a spoofing or prompt-injection shortcut.
