## Summary

Keep open. The PR targets a real current-main mismatch for multi-account Discord skill readiness, but the proposed Discord gate is too broad and would create false-ready output when `channels.discord.accounts` exists without any usable account token.

## What This Changes

The PR adds an `anyConfig` OR requirement to skill requirement parsing/evaluation/formatting/tests and changes the bundled Discord skill to accept either `channels.discord.token` or `channels.discord.accounts`.

## Best Possible Solution

Keep the OR-requirement direction only if it can express a real configured Discord token, add regression coverage for both empty `accounts` and token-bearing multi-account configs, and update the skills metadata docs if `anyConfig` becomes a public frontmatter contract.

## Reproduction Assessment

Yes. Current main can reproduce the original bug with a config that only has `channels.discord.accounts.<accountId>.token`, and the PR's new false-positive path is reproducible by evaluating `channels.discord.accounts` against an empty object.

## Solution Assessment

No. A generic `anyConfig` can be a reasonable seam, but using the `accounts` container as the Discord readiness signal is not the narrowest correct fix because it does not prove any account token exists.
