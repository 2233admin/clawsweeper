## Summary

Keep open. Current main does not implement the proposed `openclaw profile export/import` surface, so this PR is not obsolete; however, the latest branch still has a blocking privacy/security issue because its profile sanitizer is narrower than the shared sensitive-config matcher already used on main.

## What This Changes

The PR branch adds a new `openclaw profile` export/import CLI, profile archive infrastructure, docs, command/CLI/infra tests, and adjacent gateway/plugin test fixture updates.

## Best Possible Solution

If core profile portability is accepted, land it as a documented archive format that reuses the shared sensitive-key and sensitive-URL semantics, exports only portable plugin install metadata, preserves non-destructive import behavior, and has regression coverage for sanitizer misses, traversal rejection, dry-run reporting, and import safety.

## Reproduction Assessment

Yes. A focused unit check can call `buildSafeProfileConfig` with `plugins.entries.<id>.config.privateKey`, `encryptKey`, `serviceAccount`, or `serviceAccountRef`; those keys pass the PR sanitizer even though current main's shared matcher classifies them as sensitive.

## Solution Assessment

No as currently proposed. The feature direction may be viable, but the sanitizer should reuse `isSensitiveConfigPath` or mirror that full contract with regression tests before a privacy-safe export surface can merge.
