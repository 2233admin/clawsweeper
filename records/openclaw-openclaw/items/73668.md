## Summary

Keep open. Current main still does not implement this PR's intended privacy-safe `openclaw profile export/import` workflow. The PR head adds that new CLI/docs/infra surface, but it remains unmerged and security-sensitive. Greptile's secret-stripping finding still appears valid after the latest force-push because dictionary entry names can still be reclassified as dictionary-container keys during recursive sanitization.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to decide whether privacy-safe profile portability belongs as a new core `openclaw profile` command or should be folded into the existing backup/migration model, then fix the sanitizer collision, clarify dry-run output, validate plugin install records, add focused regression tests, document the trusted-archive safety model, and drop the unrelated plugin SDK test edit.

## Risks / Open Questions

- The PR's privacy promise is not yet safe: a dictionary-entry name collision can leak token/API-key/auth/env-like fields into an exported profile archive.
- The new import path writes config, workspace markdown files, and plugin install records from a tar payload, so it needs explicit trusted-source UX, size/shape limits, provenance/schema validation, rollback behavior, and threat-model review.
- Dry-run import reporting still appears misleading because `filesWritten` is populated for would-write files even when `opts.dryRun` is true.
- The PR includes an unrelated plugin SDK facade-loader test change; that should be removed or justified before review.
