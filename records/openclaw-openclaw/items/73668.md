## Summary

Keep open. Current main does not implement `openclaw profile export/import`, and the latest PR head still has blocking privacy-safety gaps in the new archive exporter, even though earlier plugin-index and dictionary-collision issues appear addressed.

## What This Changes

The PR adds a new `openclaw profile export/import` CLI surface, profile archive infrastructure, docs, and CLI/command/infra tests for privacy-safe local profile portability.

## Best Possible Solution

If core profile portability is accepted, land it with a sanitizer that reuses the shared sensitive-key matcher and sensitive-URL redaction semantics, keeps the portable plugin-install projection, preserves traversal and non-destructive import protections, and adds regression tests for private keys, service-account objects, and credential-bearing URLs.

## Reproduction Assessment

Yes. A unit-level `buildSafeProfileConfig` input with `plugins.entries.demo.config.privateKey`, `encryptKey`, `serviceAccount`, or `mcp.servers.remote.url` containing userinfo or `?token=` will serialize those values because the PR only filters a narrow key set and clones primitive strings unchanged.

## Solution Assessment

No as currently proposed. The feature direction may be viable, but a privacy-safe archive exporter should call or faithfully mirror `isSensitiveConfigPath` plus the shared sensitive URL redactor before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reuse shared sensitive matching for profile export:** `src/infra/profile-portability.ts:360-372`
  - body: The profile sanitizer only drops token/password/API-key-style names. Fields such as `privateKey`, `encryptKey`, `serviceAccount`, and `serviceAccountRef` are already treated as sensitive on main, but they can survive under portable plugin, skill, MCP, agent, or tool config and be written into the archive. Reuse the shared matcher or mirror its full contract with regression coverage.
  - confidence: 0.91
- **[P1] Redact credential-bearing URLs before exporting profiles:** `src/infra/profile-portability.ts:399-400`
  - body: The sanitizer clones primitive strings unchanged, so an exported `mcp.servers.<id>.url` can still include URL userinfo or sensitive query params such as `token` or `client_secret`. Current main already has sensitive-URL path detection and redaction helpers for this case; profile export should use them or omit those values.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: Profile export can still place credential material into an archive that is documented as privacy-safe.

Concerns:

- **[high] Private keys can survive profile sanitization:** `src/infra/profile-portability.ts:360`
  - body: The PR's hard-coded key filter misses current-main sensitive patterns such as private keys, encryption keys, and service-account payloads, so arbitrary portable config under plugins, skills, MCP, tools, or agents can leak those credentials into the archive.
  - confidence: 0.91
- **[high] MCP URLs can leak embedded credentials:** `src/infra/profile-portability.ts:399`
  - body: The exporter keeps primitive URL strings unchanged even when they contain userinfo or sensitive query parameters. OpenClaw already treats MCP server URLs as sensitive URL paths and has shared redaction helpers for this exact shape.
  - confidence: 0.88
