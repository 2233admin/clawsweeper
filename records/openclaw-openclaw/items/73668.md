## Summary

Keep open. Current main still does not implement this PR's intended privacy-safe `openclaw profile export/import` workflow, and the latest PR head remains product- and security-sensitive. The PR adds the missing surface locally, but it is not obsolete or already shipped, and the sanitizer collision noted in review discussion is still present at head `08fdefb66cf2f8e5663d6bd453a6f644ad9d510f`.

## Best Possible Solution

Keep this PR open for maintainer product and security review. The best path is to decide whether privacy-safe profile portability belongs as a new core `openclaw profile` command or should be folded into backup/migration, then fix the sanitizer collision with explicit dictionary-depth state, add a regression test for colliding entry ids, validate plugin install records and trusted-archive behavior, correct dry-run output, update the CLI docs/index if accepted, and split unrelated browser/Signal/plugin-loader test changes out of this PR.

## Risks / Open Questions

- The PR head still appears capable of leaking secret-like fields from portable config when a dictionary entry id collides with one of the internal dictionary-container key names.
- The import path writes plugin install records from an archive without a visible schema/provenance validation layer; that is sensitive because install records can influence later plugin reinstall behavior.
- Dry-run import currently reports would-write files in `filesWritten`, which is misleading for JSON consumers and operators reviewing a safe preview.
- The branch mixes unrelated browser, Signal installer, channel catalog, doctor, and plugin SDK test changes into a profile portability PR; those should be split or justified before maintainer review.
