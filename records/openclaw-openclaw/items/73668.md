## Summary

Keep open. Current main still lacks the requested profile portability surface, and the latest branch is not merge-ready because privacy-sensitive config sanitization misses key patterns that the existing config redaction layer already treats as secrets.

## What This Changes

The PR adds `openclaw profile export/import`, profile archive creation/import infrastructure, docs, CLI/command/infra tests, and adjacent plugin/UI test fixes.

## Best Possible Solution

If core profile portability is accepted, land it as a documented privacy-safe archive format that reuses the shared sensitive-key semantics, has regression tests for sanitizer misses, keeps plugin install metadata as a portable projection, and preserves non-destructive import behavior without unrelated branch churn.

Full review comments:

- **[P1] Omit privateKey and encryptKey during profile export:** `src/infra/profile-portability.ts:360-372`
  - body: The sanitizer only drops a narrow hard-coded key set, so `plugins.entries.<id>.config.privateKey` or `encryptKey` survives into the archive even though the shared config-sensitive matcher treats those names as secrets. Because profile export includes arbitrary plugin config, this can still leak credentials from a privacy-safe archive; reuse the shared sensitive-path logic or add these patterns with regression coverage.
