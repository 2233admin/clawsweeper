## Summary

Keep this PR open. Current main still does not provide the proposed privacy-safe `openclaw profile export/import` surface, and the latest PR diff remains product- and security-sensitive rather than obsolete or already implemented. The prior review discussion was useful: the reported sanitizer bypass is still visible in the latest diff, and import still trusts archive-provided plugin install records without schema/provenance validation.

## Best Possible Solution

Keep this PR open for maintainer review. If privacy-safe profile portability belongs in core, fix the sanitizer with explicit dictionary-depth state, add a regression test for colliding dictionary entry ids, validate plugin install records before import, separate actual writes from dry-run planned writes in JSON, and keep unrelated test trust changes out of the profile portability patch where possible.

## Risks / Open Questions

- Merging as-is can leak secret-like config fields into a supposedly privacy-safe archive when a dictionary entry id collides with sanitizer container key names such as `entries`, `servers`, `models`, `cliBackends`, or `providerOptions`.
- Importing raw plugin install records from an archive can affect future plugin reinstall/update behavior and should validate schema and provenance before becoming trusted local state.
- Dry-run import reports planned writes in `filesWritten`, which can mislead JSON consumers that treat the field as actual mutation proof.
- The PR is XL and adds an untrusted archive import/export surface; it needs maintainer product and security review, not conservative cleanup closure.
