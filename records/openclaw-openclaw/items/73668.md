## Summary

Keep open. Current main does not implement a top-level profile export/import command, and the existing backup and migrate surfaces do not cover the same OpenClaw-to-OpenClaw privacy-safe portability flow. The PR is useful source work, but it should not merge as-is because the archive code copies the full machine-managed plugin index payload into a privacy-safe profile archive.

## What This Changes

The PR adds `openclaw profile export/import`, profile archive creation/import infrastructure, CLI/docs/tests, and bundled-plugin test setup adjustments.

## Best Possible Solution

If core profile portability is accepted, land it as a documented privacy-safe archive format that exports sanitized config, allowed workspace markdown, and a portable plugin install-record projection only. The plugin index should be rebuilt or refreshed on import instead of copying cached machine-specific registry state, with regression coverage proving no local paths, secrets, sessions, credentials, or stale cache paths enter the archive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Sanitize plugin install records before archiving:** `src/infra/profile-portability.ts:645-646`
  - body: The export path copies `plugins/installs.json` verbatim into the profile archive. That file is the machine-managed plugin index, not just portable install specs: it can include `sourcePath`/`installPath` in install records and cached `manifestPath`/`rootDir` values. A privacy-safe export can therefore leak local paths and import stale machine-specific registry state; write a sanitized portable install-record payload and rebuild or refresh the target index instead.
  - confidence: 0.9
