## Summary

Keep open. This PR is still open and unmerged, and current `main` does not implement the intended privacy-safe `openclaw profile export/import` workflow. The existing `--profile` support only selects an isolated state/config directory, while current backup and migration docs cover full-state backup or cross-system migration rather than a narrower privacy-safe profile archive. The prior Greptile and ClawSweeper comments remain relevant: the latest PR head still carries profile archive/import code that needs maintainer product and security review before it could be merged.

## Best Possible Solution

Keep this PR open for maintainer product and security review. If privacy-safe profile portability belongs in core, fix the sanitizer with explicit dictionary-depth state, add a regression test for colliding entry ids, validate plugin install records on import, clarify dry-run JSON semantics, split unrelated bundled-plugin test-trust plumbing if possible, and then run the appropriate changed gates before merge.

## Risks / Open Questions

- Merging the PR as-is could export secret-like fields when dictionary entry ids collide with container key names such as `entries`, `servers`, `models`, `cliBackends`, or `providerOptions`.
- Importing raw plugin install records from an archive can affect later plugin reinstall/update behavior and should validate schema/provenance before becoming trusted local state.
- The PR head reports dry-run would-write entries under `filesWritten`, which can mislead automation that treats that field as actual writes.
- The branch includes bundled-plugin trust/test plumbing outside the core profile portability feature and should receive separate supply-chain/test-environment review.
