## Summary

Keep open. Current main does not implement the PR's intended privacy-safe `openclaw profile export/import` workflow: the core command catalog and lazy registry expose `backup` and `migrate`, but no `profile` command, and the existing `--profile` code only selects an isolated state directory. The PR remains a plausible implementation candidate, but the latest head still needs maintainer product and security review before merge.

## Best Possible Solution

Keep this PR open for maintainer product and security review. If privacy-safe profile portability belongs in core, fix the sanitizer with explicit dictionary-depth state, add a regression test for colliding entry ids, validate plugin install records before import, clarify dry-run JSON semantics, split unrelated bundled-plugin test plumbing out of the branch, then run the appropriate changed gates before merge.

## Risks / Open Questions

- Merging as-is could leak secret-like config fields from exported profile archives when dictionary entry ids collide with container key names such as `servers`, `entries`, `models`, `cliBackends`, or `providerOptions`.
- Importing plugin install records from an archive affects later plugin reinstall/update behavior and should validate the record schema/provenance before becoming a trusted local index.
- The import JSON result currently reports dry-run would-write entries under `filesWritten`, which can mislead automation that treats that field as actual writes.
- The branch includes bundled plugin trust/test setup changes that are not part of profile export/import and should be reviewed separately for supply-chain and test-environment implications.
