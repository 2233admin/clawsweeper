## Summary

Keep open. Current main still lacks the proposed profile export/import surface, and the latest PR head remains blocked by a concrete privacy/security bug in the export sanitizer despite addressing earlier review feedback.

## What This Changes

The PR adds `openclaw profile export/import`, a profile archive creation/import layer, docs, CLI/command/infra tests, and adjacent gateway/plugin test fixture updates.

## Best Possible Solution

If core profile portability is accepted, land it as a documented archive format that reuses the shared config redaction contract, exports only portable plugin install metadata, preserves non-destructive import semantics, and carries focused regression tests for secret stripping, sensitive URLs, traversal rejection, workspace file allowlisting, and plugin install projection.

## Reproduction Assessment

Yes. A unit-level `buildSafeProfileConfig` input containing `plugins.entries.demo.config.privateKey`, `encryptKey`, or `serviceAccount` will survive the PR sanitizer, while current main's shared matcher classifies those paths as sensitive.

## Solution Assessment

No as currently proposed. The narrowest maintainable fix is to reuse `isSensitiveConfigPath` plus existing sensitive-URL redaction semantics, or mirror that full contract with regression coverage, before writing profile config into an archive.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Reuse shared sensitive matching before profile export:** `src/infra/profile-portability.ts:360-372`
  - body: The profile sanitizer only drops token/password/API-key style names, but profile export writes arbitrary portable plugin, skill, MCP, and agent config into the archive. Keys that main already treats as sensitive, such as `privateKey`, `encryptKey`, and `serviceAccount`, can survive into a supposedly privacy-safe export. Reuse the shared sensitive matcher/redactor or mirror its full contract with regression tests before archiving config.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR can leak credentials into profile archives because its new export sanitizer is narrower than OpenClaw's existing sensitive config redaction contract.

Concerns:

- **[high] Profile archive can include private keys:** `src/infra/profile-portability.ts:360`
  - body: The new profile export path omits common token/password fields but misses private-key, encryption-key, and service-account patterns that current main already treats as sensitive. Since profile export includes arbitrary portable plugin/skill/MCP/agent config, those credentials can be written into the archive.
  - confidence: 0.9
