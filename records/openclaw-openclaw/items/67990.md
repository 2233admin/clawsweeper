## Summary

Keep this issue open. Current main has first-class backup creation, verification, config-only mode, workspace opt-out, top-level canonical path dedupe, and a narrow hard-coded filter for installed plugin `extensions/*/node_modules`, but it still does not implement the central proposal: user-configurable exclusion rules, general recursive filtering, service-aware gateway quiescence, restore/full-mode command surface, or live-file retry/skip handling. Related #72251 remains an open partial for one live tar EOF race and does not supersede the broader design.

## What This Changes

The issue proposes a broader backup design with configurable exclusion rules, service-aware archiving, symlink-safe traversal, tar.gz archives, and distinct config-only/full backup modes.

## Best Possible Solution

Keep this issue as the broader backup design tracker, preserve the shipped create/verify/config-only behavior and current plugin dependency pruning, then split maintainer-approved follow-ups for configurable exclusion policy, live-file resilience, explicit quiesce/service semantics, and restore/import disaster-recovery behavior coordinated with #13616, #67417, #54242, and #72251.

## Risks / Open Questions

- The remaining work spans multiple design decisions, so a single automated fix PR would risk mixing exclusion policy, service lifecycle, restore semantics, and archive consistency into one hard-to-review change.
- Implicit gateway stop/start during backup affects availability and operator trust; it likely needs an explicit product decision such as an opt-in quiesce mode.
- Backup coverage includes credentials and state, so any configurable exclusions or restore/import path needs a security review for secrets handling, archive integrity, and safe defaults.
- Open #72251 may change the live-file race slice and should be reconciled before designing retry/filter behavior for this broader tracker.
