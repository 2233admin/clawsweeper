## Summary

Keep this issue open. Current main still does not preserve session memory on all requested lifecycle exits: bundled `session-memory` remains registered for manual `command:new`/`command:reset`, lazy daily/idle rollover emits only typed `session_end`/`session_start` plugin hooks, and session-store maintenance prunes stale entries without a pre-delete memory/internal hook. Open PRs #61675 and #71792 cover the daily/idle half, but the prune path remains unaddressed.

## Best Possible Solution

Keep this issue open as the broader tracker, or split only if maintainers create a canonical prune-specific follow-up. Review, land, or replace #61675/#71792 for daily/idle rollover, then add a maintainer-approved prune path that dispatches a documented pre-delete lifecycle or session-memory-compatible hook with the removed `SessionEntry` and stable transcript path before store maintenance deletes/archive-cleans the entry.

## Risks / Open Questions

- Important transcript context can still be lost when stale sessions are pruned before any reset/compaction-triggered memory capture runs.
- A naive prune fix could run LLM-backed memory extraction or plugin hooks while holding the session-store write lock, increasing latency and failure coupling for ordinary store writes.
- Daily/idle reset work is already represented by open PRs #61675 and #71792, so a new implementation should avoid duplicating or conflicting with those candidates.
