## Summary

Keep open. Current main still matches the reported behavior: runtime auth, session auto-selection, and status labels resolve canonical provider auth-profile order before the selected provider entry's config credential, while the schema/docs define `models.providers.*.apiKey` as credential material rather than an auth-profile reference.

## What This Changes

Request that split provider entries sharing the same upstream auth provider use the selected provider entry's intended credential instead of an arbitrary canonical `auth.order` profile.

## Best Possible Solution

Define an explicit provider-entry auth binding that preserves existing `apiKey` credential semantics, then apply that single rule in runtime auth resolution, session auto-selection, status labels, schema/help/docs, and focused split-OpenRouter regression tests.
