## Summary

Keep this PR open. Current main already shares the unhandled-rejection handler registry through `globalThis`, but `installUnhandledRejectionHandler()` still registers a new `process.on("unhandledRejection", ...)` listener every time it is called. The PR's intended idempotency fix remains relevant, though it should be rebased onto the current handler structure and covered by a focused listener-count regression test.

## Best Possible Solution

Keep the PR open, or replace it with an equivalent maintainer patch, by adding a shared installed guard to `installUnhandledRejectionHandler()` in `src/infra/unhandled-rejections.ts`. Preserve the current global handler Sets, make repeated installer calls a no-op, and add focused regression coverage that repeated calls do not increase `process.listenerCount("unhandledRejection")`.
