## Summary

Keep open. Current main has a per-command resetHookTriggered fallback guard, but it does not implement this PR’s intended bounded dedupe for repeated reset-hook emissions for the same prior session. The PR also has unresolved functional review concerns that should be addressed before it can be considered obsolete or merge-ready.

## Best Possible Solution

Keep the PR open and rework it against current src/auto-reply/reply/commands-reset-hooks.ts. The best fix is a narrowly scoped dedupe that registers before the first await, avoids process-wide suppression of valid later resets, and includes regression tests for duplicate re-entry plus valid subsequent resets. No dependency, CI, or publishing changes are needed.

## Risks / Open Questions

- The PR’s current dedupe approach can still allow concurrent duplicate emissions if the key is registered after await triggerInternalHook.
- A process-wide session-id dedupe can skip legitimate later /new or /reset hooks if the same prior session id is reused or remains stable.
- Closing this PR as implemented would likely drop useful test coverage and a plausible remaining bug because current main does not contain equivalent prior-session duplicate suppression.
