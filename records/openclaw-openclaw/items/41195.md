## Summary

Keep this issue open. Current main restores and checks Feishu dedupe state for the same `message_id`, but the reported failure is replayed Feishu history arriving with fresh message IDs after WebSocket reconnect. The current receive, claim, transport, and dispatch paths still have no reconnect replay window, connection-start timestamp gate, `create_time` cutoff, or `ignoreOfflineMessages` config surface.

## Best Possible Solution

Fix this in the bundled Feishu plugin. Add a conservative per-account WebSocket reconnect/startup replay guard that records incoming IDs while suppressing stale replay events, likely using `create_time` relative to the active connection time and optionally an explicit `ignoreOfflineMessages` or replay-window config. Add focused tests for fresh-ID stale replays being skipped, fresh post-connect messages still dispatching, and existing same-ID persistent dedupe continuing to work.

## Evidence

- **Issue discussion scopes the remaining bug to fresh IDs:** The issue body says restoring `feishu-dedup.json` is insufficient because Feishu can assign new message IDs to replayed messages; a commenter also characterized the behavior as a Feishu plugin reconnect/dedupe problem. The prior ClawSweeper comment on this issue reached the same keep-open conclusion.
- **Persistent dedupe is keyed by message ID:** Feishu persistent dedupe normalizes a single message ID and records/checks it via `tryRecordMessagePersistent` and `hasRecordedMessagePersistent`; this catches same-ID repeats but not the same user message replayed under a new ID.
