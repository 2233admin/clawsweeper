## Summary

Keep this issue open. Current main contains meaningful Feishu duplicate protections, including persistent same-`message_id` dedupe warmed from disk and early in-flight message claims, but the specific reported failure remains: replayed Feishu WebSocket history with fresh `message_id` values is still treated as fresh inbound work. The issue discussion and related context also frame this as a bundled Feishu plugin reconnect behavior, not a ClawHub/off-repo item.

## Best Possible Solution

Fix this in the bundled Feishu plugin. Add a conservative per-account WebSocket reconnect/startup replay guard that can suppress stale replayed inbound events even when Feishu assigns fresh `message_id` values, likely using `create_time` relative to connection start plus an explicit opt-in/offline-message policy if user-visible behavior changes. Keep the existing persistent same-ID dedupe, and add focused tests for fresh-ID stale replays being skipped, fresh post-connect messages still dispatching, same-ID dedupe still working, and debounce batching not merging suppressed replay content.
