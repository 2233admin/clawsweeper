## Summary

Current main still leaves the fresh-message-ID Feishu replay path open. Feishu now has same-message-ID persistent dedupe, warmup, and in-flight claims before debounce, but WebSocket reconnect/startup does not attach a replay-grace or timestamp/offline-message policy, so old Feishu events replayed under new IDs can still reach dispatch.

## Best Possible Solution

Keep this open and fix it in the bundled Feishu plugin, not core. The best path is a maintainer-approved per-account WebSocket replay policy that preserves current same-ID persistent/in-flight dedupe while suppressing old create_time events replayed after connect/reconnect under fresh IDs. The policy should be explicit in config/docs and covered by focused tests for stale fresh-ID replay suppression, fresh post-connect dispatch, same-ID dedupe, and debounced batches containing suppressed replay entries.
