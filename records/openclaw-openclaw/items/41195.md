## Summary

Keep this issue open. Current main still has Feishu duplicate protections for same-`message_id` repeats and restart-warmed persistent dedupe, but I did not find a WebSocket reconnect/startup replay guard that suppresses stale Feishu history when the replay arrives with fresh `message_id` values. The prior ClawSweeper keep-open assessment still matches current main at 7a23b2d945cc2d2e5c452e4641213e71de8dcc66.

## Best Possible Solution

Fix this in the bundled Feishu plugin. Add a conservative per-account WebSocket startup/reconnect replay guard that can suppress stale replayed inbound events even when Feishu assigns fresh `message_id` values, likely based on `message.create_time` relative to connection start plus an explicit policy for offline messages if user-visible behavior changes. Keep the existing persistent same-ID dedupe and add focused tests for fresh-ID stale replays being skipped, fresh post-connect messages still dispatching, same-ID dedupe still working, and debounce batching not merging suppressed replay content.
