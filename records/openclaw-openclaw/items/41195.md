## Summary

Keep this issue open. Current main has persistent Feishu same-message-ID dedupe and in-flight same-ID processing claims, but the reported WebSocket replay case with fresh Feishu `message_id` values is still not guarded by startup/reconnect grace, `create_time` filtering, or an offline-message policy.

## Best Possible Solution

Fix this in the bundled Feishu plugin by keeping the existing persistent same-ID dedupe and adding a conservative per-account WebSocket startup/reconnect replay guard for fresh-ID stale events, likely based on `message.create_time` relative to connection start plus an explicit offline-message policy/config. Add focused tests for fresh-ID stale replay suppression, fresh post-connect dispatch, same-ID dedupe, and debounce batching around suppressed replay content.
