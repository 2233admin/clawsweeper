## Summary

Keep open. Current main still has the reported key-only drain cleanup, the latest release still contains the same behavior, and the issue is paired with an open same-author closing PR that should be reviewed before this issue is resolved.

## What This Changes

The issue reports that auto-reply follow-up drain cleanup can delete a replacement queue after `/stop` or session reset because final cleanup deletes `FOLLOWUP_QUEUES` by key without checking queue identity.

## Best Possible Solution

Review #68839 as the canonical implementation candidate. If it is correct, land the identity guard and regression test so GitHub closes this issue from the merged PR; if not, request a revised auto-reply queue fix that prevents late drain cleanup from deleting a replacement queue.
