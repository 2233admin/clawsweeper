## Summary

Keep PR #38290 open. Current main still normalizes incoming browser origins through WHATWG `URL.origin` and compares that value directly to lowercased `gateway.controlUi.allowedOrigins`, so exact `chrome-extension://<id>`, `app://localhost`, and similar opaque-origin schemes still cannot match. The PR remains a focused gateway allowlist bug fix; scheme wildcard support is separate in #39739.

## Best Possible Solution

Keep PR #38290 open and land it after rebase, or land an equivalent narrow patch in `src/gateway/origin-check.ts` that symmetrically normalizes incoming `Origin` headers and configured `gateway.controlUi.allowedOrigins`. Use `URL.origin` for standard origins and a `protocol//host` fallback only for opaque-origin schemes where `URL.origin` is `"null"`. Keep/add unit coverage plus a gateway WebSocket handshake regression for accepted and rejected extension origins, and leave wildcard pattern support to #39739.

## Risks / Open Questions

- Origin handling is security-sensitive; the final patch should preserve literal `Origin: null` rejection, exact matching semantics, global `*` behavior, host-header fallback behavior, and token/device auth behavior.
- The PR branch is older than current main and should be rebased with focused gateway unit and WebSocket regression tests before landing.
- Scheme-level wildcard behavior such as `chrome-extension://*` remains a separate product/security decision tracked by #39739.
