## Summary

Keep this PR open. Current main still normalizes browser Origin values with URL.origin and compares allowlist entries to that value directly, so exact chrome-extension://, app://, and tauri:// origins still collapse to "null" and cannot match. The PR remains a narrow gateway origin-allowlist fix; scheme wildcards remain separate in #39739.

## Best Possible Solution

Keep this PR open and land it after rebase, or land an equivalent narrow gateway patch. Normalize incoming Origin headers and configured gateway.controlUi.allowedOrigins symmetrically: use URL.origin for standard origins, fall back to protocol//host only for opaque-origin schemes, keep literal Origin: null rejected, and leave scheme wildcard support to #39739. Add focused unit tests plus a WebSocket handshake regression for accepted and rejected extension origins.
