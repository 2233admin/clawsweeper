## Summary

Keep open. Current main still lacks the requested BlueBubbles trainer/supervised owner-approval mode, the issue is fresh and actionable, and the safest next step is BlueBubbles maintainer/product review because the feature changes private-message delivery and correction logging semantics.

## What This Changes

The issue requests an optional BlueBubbles `trainerMode` with `training` and `supervised` owner-approval flows, iMessage owner commands, pending-draft TTL, correction logging, and optional AI-send tagging.

## Best Possible Solution

Keep the work BlueBubbles-owned after maintainer approval: add schema-backed `trainerMode`, `trainerNotifyNumber`, and optional `agentTag`; capture outbound DM replies before BlueBubbles sends them; maintain pending draft state with TTL; parse owner approval/custom-reply commands; log corrections intentionally; update docs/config metadata; and cover the DM routing, expiry, and failure modes with focused tests. Avoid core changes unless implementation proves a reusable channel-approval seam is needed.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence current-main verification path: the requested config/API names are absent, the public BlueBubbles docs do not mention the mode, and the current DM delivery path sends directly to the resolved BlueBubbles target.

## Solution Assessment

Unclear until BlueBubbles owner/product review. The proposed plugin-local boundary is the right starting point because this is specific to BlueBubbles/iMessage DM delivery, but the human-gated reply and correction-log semantics should be approved before code is queued.
