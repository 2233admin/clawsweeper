## Summary

Keep this issue open. Current main has useful partial support for config-injected and forward-compatible provider models, but it still does not implement the RFC's central invariant: a single canonical resolved-model descriptor consumed consistently by runtime, status/list, pickers, model-selection consumers, capability gates, and docs.

## Best Possible Solution

Keep this issue open until maintainers either accept the shared resolved-model descriptor direction or split it into explicit phase issues. The best implementation path is to define a canonical resolution result carrying identity, provider transport/API, capabilities, provenance, availability/auth state, and unsupported reason, then migrate runtime resolution, `models status`, `models list`, Gateway/UI pickers, auto-reply/model selection, thinking/reasoning gates, validation, and docs to consume it consistently.
