## Summary

Keep this issue open. Current main has useful partial support for config-injected provider models, including catalog visibility, `models list` forward-compat rows, and provider dynamic runtime hooks, but it still has separate lightweight catalog rows, runtime `Model<Api>` resolution, CLI row availability, status auth/probe logic, Gateway/UI picker catalogs, and thinking capability resolution. That does not yet implement the RFC's central invariant: one canonical resolved-model descriptor with identity, transport, capabilities, provenance, availability, and unsupported reason consumed consistently across status, list, runtime, pickers, gates, and docs.

## Best Possible Solution

Keep this issue open until maintainers either land a shared resolved-model descriptor or intentionally split the RFC into tracked phases. The implementation should define one descriptor/result that carries identity, transport/API, capabilities, provenance, availability/auth state, and unsupported reason, then migrate runtime resolution, `models status`, `models list`, Gateway/UI model pickers, auto-reply model selection, thinking/reasoning gates, and docs to consume it consistently.
