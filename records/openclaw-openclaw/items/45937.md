## Summary

Keep this PR open. Current main still does not recognize the MiniMax 2062 Chinese rate-limit sample, and the PR remains a plausible fix candidate, but its current diff also carries unresolved false-positive risks from the generic `请稍后重试` phrase and from matching arbitrary non-error assistant text.

## Best Possible Solution

Keep this PR open for revision or maintainer review. The best path is a precise MiniMax/CN failover classifier, preferably provider-owned via `classifyFailoverReason` or a conservative shared matcher, with the generic retry-only phrase omitted and tests covering the MiniMax 2062 positive case plus Chinese retry-phrase negative cases.

## Risks / Open Questions

- Adding standalone `请稍后重试` would classify generic Chinese retry suggestions from auth, billing, maintenance, or timeout errors as `rate_limit`.
- Matching any short non-error assistant content risks retrying or falling back from a valid answer that merely discusses rate limits, high demand, or usage limits.
- The PR does not preserve a concrete failover reason for text-only provider payloads, which can degrade cooldown/backoff behavior.
- Security review: no supply-chain or CI risk is apparent from the provided diff because it only touches TypeScript runtime classification helpers and no workflows, dependencies, lockfiles, scripts, or secrets paths.
