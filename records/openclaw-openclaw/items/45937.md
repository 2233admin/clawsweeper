## Summary

Keep this PR open. The MiniMax/CN-provider rate-limit problem is still not fully implemented on current main, but the branch has blocking classifier bugs: it adds a generic Chinese retry phrase as a standalone rate-limit signal and treats short successful assistant text as a provider failure candidate.

## What This Changes

The branch adds Chinese rate-limit patterns to `src/agents/pi-embedded-helpers/failover-matches.ts` and broadens `isFailoverAssistantError` to classify short non-error assistant text as failover-worthy.

## Best Possible Solution

Implement a precise MiniMax/CN failover classifier, preferably in MiniMax's provider-owned `classifyFailoverReason` hook or a conservative shared matcher. It should match the 2062/rate-limit payload, omit generic retry-only text, only treat non-error text as failover when it is an explicit provider error payload, preserve the failover reason, and add positive MiniMax 2062 plus negative Chinese retry-phrase regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Restrict non-error text matching to real provider errors:** `src/agents/pi-embedded-helpers/errors.ts:1047`
  - body: This branch applies rate-limit and overload matchers to any short `end_turn` assistant text. A normal successful answer that mentions `rate limit`, `usage limit`, or `high demand` can be treated as a failed provider turn, causing retry/fallback instead of returning the response.
  - confidence: 0.91
- **[P2] Remove generic retry text from rate-limit patterns:** `src/agents/pi-embedded-helpers/failover-matches.ts:21`
  - body: `请稍后重试` means “please retry later” and is commonly appended to non-rate-limit failures. Since rate limits are classified before timeout, billing, and auth paths, this can force unrelated Chinese provider errors into `rate_limit`.
  - confidence: 0.9
- **[P2] Preserve the matched failover reason for text-only errors:** `src/agents/pi-embedded-helpers/errors.ts:1048`
  - body: The new text-content branch returns only `true`, while the runner still derives `assistantFailoverReason` from `errorMessage`, which is empty for the targeted `end_turn` case. Those retries can be emitted as `unknown` and skip reason-specific backoff or cooldown behavior.
  - confidence: 0.84
