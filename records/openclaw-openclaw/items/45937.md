## Summary

Keep this PR open. Current main has some generic Chinese failover patterns, but it still does not implement the MiniMax 2062 rate-limit recognition this PR targets, and the PR’s current diff has unresolved false-positive concerns around generic retry text and non-error assistant responses.

## Best Possible Solution

Keep this PR open for revision or maintainer review. The strongest path is a precise MiniMax/CN classifier, preferably in MiniMax's provider-owned `classifyFailoverReason` hook or a conservative shared matcher, omitting generic retry-only text, preserving the failover reason for explicit provider error payloads, and adding positive MiniMax 2062 plus negative Chinese retry-phrase regression tests.

## Risks / Open Questions

- The standalone `请稍后重试` pattern can misclassify Chinese auth, billing, maintenance, or timeout messages as rate limits.
- The proposed non-error assistant text matching can retry or fallback a valid short answer that merely discusses rate limits or high demand.
- The proposed text-only path returns a boolean without preserving `rate_limit` versus `overloaded`, which can degrade cooldown/backoff behavior.
