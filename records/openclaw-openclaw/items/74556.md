## Summary

Keep this PR open for maintainer handling. The patch addresses a real Codex app-server state-isolation gap on current main and I did not find a concrete blocking correctness or supply-chain issue, but the protected `maintainer` label, XL size, and security-sensitive Codex auth/home boundary make this an explicit human review and merge decision.

## What This Changes

The PR isolates local Codex stdio app-server `CODEX_HOME`/`HOME` per agent, adds Codex migration/doctor/docs coverage for native Codex assets, forwards Codex auth profiles into Codex harness runs, and adds Slack/Signal untrusted system-event coverage.

## Best Possible Solution

Land the PR after maintainer review confirms the intended Codex state boundary and changed-surface validation remains green, preserving the stdio-only isolation, manual-review migration treatment for native plugins/hooks/config, and channel trust-boundary tests.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: a local stdio Codex app-server launch inherits the gateway process environment because the bridge does not set `CODEX_HOME` or `HOME` and the transport copies `process.env`. A gateway process with personal Codex roots in its environment can therefore leak those roots into the app-server child.

## Solution Assessment

Yes. The proposed approach is the narrow maintainable fix: isolate only OpenClaw-owned local stdio launches, leave WebSocket app-server ownership external, preserve explicit env overrides while reserving the managed isolation variables from `clearEnv`, and provide doctor/migration guidance instead of auto-activating native Codex executable state.
