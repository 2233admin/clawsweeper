## Summary

Keep this PR open. Current main and v2026.4.26 still lack an equivalent protected instruction-file mutation guard, so there is no conservative cleanup close available. The proposed patch is a focused security-hardening attempt, but it still needs maintainer security/product review because its guard is not centralized around resolved mutation targets and it conflicts with documented main-session memory workflows.

## What This Changes

The PR adds an instruction-file guard module, wraps host and sandbox write/edit tools, adds protected-basename checks to apply_patch, and adds tests for protected instruction-file mutations.

## Best Possible Solution

Keep the PR open for maintainer security review or replace it with an equivalent core implementation. The desired end state is one central agent-file mutation policy that checks normalized, resolved, canonical targets for host write/edit, sandbox write/edit, apply_patch add/update/delete/move, and any intended process-backed write scope, with an explicit operator approval or session exception model for legitimate persona and memory updates plus regression tests and docs updates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve write guards against the workspace target:** `src/agents/pi-tools.instruction-file-guard.ts:50-53`
  - body: The wrapper realpaths the raw `path` argument, so relative workspace paths and sandbox container paths are checked against the gateway process cwd instead of the actual tool target. An alias such as `notes.md` inside the agent workspace can miss this check and then be resolved by the underlying write/edit tool to the protected file. Reuse the same workspace/container resolution path before checking canonical targets.
  - confidence: 0.87
- **[P2] Check resolved patch targets before writing:** `src/agents/apply-patch.ts:154-163`
  - body: The apply_patch guard checks hunk strings before `resolvePatchPath` and before the final file operation. That leaves the patch path policy separate from the actual mutation target, especially when workspace-only is disabled or a sandbox bridge follows aliases. Enforce the protected-file check on the same resolved target that will be written, moved, or removed.
  - confidence: 0.82
- **[P2] Preserve approved main-session memory updates:** `src/agents/pi-tools.instruction-file-guard.ts:17-25`
  - body: Adding `memory.md` to an unconditional protected set makes all agent writes to `MEMORY.md` fail, but the workspace template still tells main sessions they can update `MEMORY.md` freely. Add the promised operator approval/session exception model, or migrate the documented memory workflow, before merging this behavior change.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch is security-sensitive and has no dependency, lockfile, workflow, or supply-chain changes, but the proposed protection is still bypassable because it is not enforced on resolved mutation targets.

Concerns:

- **[medium] Alias paths can bypass protected-file checks:** `src/agents/pi-tools.instruction-file-guard.ts:50`
  - body: The write/edit guard checks a raw tool argument and apply_patch checks raw hunk text, while the actual mutation path resolves workspace-relative, sandbox, and alias forms later. A protected-file mutation policy should run on the canonical target that will be written or removed.
  - confidence: 0.84
