## What This Changes

The PR exports a CLI progress spinner eligibility helper that skips readline-backed spinners when `process.stdin.isRaw` is already true, and adds focused progress tests for raw TUI input and normal TTY commands.

## Best Possible Solution

Keep this PR open for maintainer review and CI. If maintainers agree with the raw-stdin boundary, land the small progress helper plus focused tests so CLI progress preserves normal TTY spinners while avoiding clack/readline ownership inside an active TUI.

## Reproduction Assessment

Yes. The code path is high-confidence from source: Crestodian TUI input reaches the status operation, status calls `statusCommand`, and current CLI progress creates a clack spinner for default TTY progress without checking whether stdin is already raw.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix I found: guard only the readline-backed spinner on `process.stdin.isRaw`, while keeping normal TTY spinner behavior and leaving non-spinner progress modes untouched.
