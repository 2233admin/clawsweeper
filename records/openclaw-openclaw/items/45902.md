## Summary

Keep this PR open. Current main still does not implement the central Telegram behavior: `/model` in a Telegram topic/thread updates only the active session, there are no parent-chat future-thread default fields or helpers, and new Telegram thread sessions are not seeded from a parent future default. The proposed branch also remains too broad for cleanup closure because it mixes the Telegram change with unrelated memory/gateway work, local E2E tooling, and unresolved review concerns.

## Best Possible Solution

Keep this PR open, but request a scoped revision or replacement focused on Telegram model inheritance. The runtime fix should persist a same-chat parent future-thread model default from every `/model` path, seed only genuinely new Telegram forum-topic and DM-topic sessions from that parent default, avoid cross-chat leakage for main-scoped DM keys, cover Telegram `direct_messages_topic` payloads if needed, and add focused tests for concurrent writes, parent-key resolution, and metadata pre-create edge cases. Split memory, gateway, recovery-script, and live E2E tooling into separate maintainer-reviewed PRs.
