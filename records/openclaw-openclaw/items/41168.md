## Summary

Current main still does not implement the central requested behavior. With no explicit `--session`, the TUI still resolves to the configured global/main session and loads history for that key; it does not choose the latest user-active non-heartbeat session. Current session storage has `lastInteractionAt` and system-event paths preserve it separately from `updatedAt`, but the public TUI/session-list surface still exposes only `updatedAt`, so a safe latest-user-active startup policy is still missing. The storage-bloat side has meaningful mitigations through compact skill snapshots and session maintenance, but that does not close the unresolved TUI startup bug.

## Best Possible Solution

Keep this issue open. Implement a TUI startup resume policy for absent `--session`: after agent metadata is available, list sessions for the current agent, exclude heartbeat/cron/system-only rows, choose the latest real user-active session using `lastInteractionAt` or an equivalent public session-list field, then load that session history. Add TUI/gateway regression tests and update the TUI docs. If storage growth still reproduces on Windows/npm-global installs after current compact skill snapshots and session maintenance, split that into a focused session-retention or skill-snapshot storage bug with concrete session files, sizes, config, and reproduction steps.

## Risks / Open Questions

- Changing the no-`--session` TUI default is user-visible and needs an explicit resume policy plus regression coverage.
- Selecting by `updatedAt` alone would risk choosing heartbeat, cron, exec-event, or other system-maintenance activity rather than a real user conversation.
- The storage-growth claim should be split or revalidated after the TUI startup behavior is fixed, because current main already has compact skill snapshots and session cleanup but may still create large per-session snapshots depending on configuration.
