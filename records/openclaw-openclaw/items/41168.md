## Summary

Current main still leaves the reported behavior unresolved. The no-`--session` TUI path defaults to the configured agent main session, then loads that history on connect; the session refresh path only updates metadata for the already-selected key. The storage-bloat half is also not fixed on main: `skillsSnapshot` is still part of `SessionEntry`, and session-store/disk-budget serialization still stringifies the full store. Open PR #48425 is a relevant storage-focused implementation, but it is not merged and does not address the TUI latest-active-session selection behavior.

## Best Possible Solution

Keep this issue open. Split the work cleanly: implement the TUI no-explicit-session startup policy so it selects the latest real user-active session for the selected agent, using `lastInteractionAt` or an equivalent user-activity signal and falling back to the configured main session when no candidate exists; preserve explicit `--session`, global scope, and manual `/session` behavior. Handle the sessions.json `skillsSnapshot` bloat through PR #48425 or a canonical storage PR, then link the two outcomes before closing this issue.
