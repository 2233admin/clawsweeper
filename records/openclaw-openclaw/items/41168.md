## Summary

Current main still does not implement the central requested behavior. With no explicit `--session`, the TUI resolves to the configured main/global session and loads that history; it does not select the latest user-active non-heartbeat session. Current session APIs expose `updatedAt` ordering, but not enough user-activity metadata for the TUI to safely distinguish heartbeat/system updates from a real user conversation. The storage portion has cleanup and compact skill snapshot mitigations, but the reported TUI startup behavior remains open.

## Best Possible Solution

Keep this issue open. Implement a no-`--session` TUI startup resume policy: after connecting and loading agent metadata, query sessions for the current agent, choose the latest user-active non-synthetic session using `lastInteractionAt` or an equivalent API field, explicitly exclude heartbeat/cron/system-only rows, then load that session history. Add gateway/TUI regression tests and document the startup behavior. If storage growth still reproduces, split that into a focused session-retention or skill-snapshot storage bug with Windows/npm-global reproduction data.
