## Summary

Current main still does not implement the central TUI startup behavior. With no explicit `--session`, the TUI resolves to `global` or the configured agent main session and then loads that history; it does not inspect stored sessions or choose the latest user-active non-heartbeat session. The storage side has current mitigations through compact skill snapshots and session maintenance, but that does not close the unresolved TUI resume bug.

## Best Possible Solution

Keep this issue open. Add a TUI startup resume policy for absent `--session`: after agent metadata is available, list sessions for the current agent, expose or derive a real user-activity timestamp such as `lastInteractionAt`, exclude heartbeat/cron/system-only rows, choose the latest real user-active session, and load that history. Add TUI/gateway regression coverage and update the TUI docs. If storage growth still reproduces afterward, split it into a focused session-retention or skill-snapshot storage issue with concrete session files, sizes, config, and reproduction steps.
