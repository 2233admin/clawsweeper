## Summary

Keep open. Current main still initializes a no-`--session` TUI launch to the current agent's main/global session and then loads that key's history; it does not scan sessions for the latest user-active non-heartbeat session. Storage maintenance now bounds old session rows, but new sessions still persist a per-session `skillsSnapshot`, so the storage-bloat half needs a real fix or narrower follow-up rather than closure.

## Best Possible Solution

Keep this issue open. Implement a TUI startup resume policy for the no-`--session` case after agent metadata is known: list sessions for the selected agent, expose or derive `lastInteractionAt`, filter heartbeat/cron/system-only rows, choose the latest real user-active session, and then load that history. Preserve explicit `--session` and global behavior, add TUI/gateway regression coverage, update the TUI docs, and separately verify whether per-session `skillsSnapshot` storage needs dedupe or a narrower retention follow-up.

## Risks / Open Questions

- Changing the absent-`--session` startup target is user-visible; explicit `--session`, global scope, workspace-derived agent selection, missing stores, and main-session fallback need a compatibility policy.
- A resume policy based only on `updatedAt` could reproduce the bug because heartbeat, cron, exec notifications, or bookkeeping may mutate rows without a user interaction.
- The storage-growth report combines two behaviors; after fixing TUI resume, maintainers should decide whether `skillsSnapshot` dedupe, snapshot-size reduction, retention tuning, or a separate storage issue is the right follow-up.
