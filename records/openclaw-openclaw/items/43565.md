## Summary

Keep open. Current main still lacks `tools.fs.allowedRoots`, and this PR is the active implementation attempt for the feature, but the branch introduces blocking filesystem-boundary regressions in `apply_patch` deletion, sandbox bind-mount allowlist handling, and default `apply_patch` availability.

## What This Changes

This PR adds a `tools.fs.allowedRoots` config field and threads additional filesystem roots through FS path guards, host read/write/edit, sandbox/media consumers, `apply_patch`, schema/help text, and regression tests.

## Best Possible Solution

Land a corrected, rebased version that keeps one canonical filesystem policy path, preserves pinned fs-safe helpers for every root-contained mutation, resolves sandbox/container paths through the bridge or mount table before comparing configured roots, preserves current `apply_patch` defaults, and keeps config/schema/help/tests aligned across read/write/edit, `apply_patch`, and media tools.

## Reproduction Assessment

Yes. Source inspection against current main and PR head reproduces both the missing `allowedRoots` feature on main and the PR regressions; targeted tests should cover delete, bind-mount allowlist, and default-enable behavior after fixes.

## Solution Assessment

No. The feature direction is useful, but this branch is not the best mergeable solution until it restores pinned deletion, validates allowed roots after sandbox mount resolution, and keeps default `apply_patch` enablement unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore pinned removal for apply_patch deletes:** `src/agents/apply-patch.ts:311`
  - body: The workspace-only delete path validates with `assertSandboxPath()` and then calls `fs.rm()` on the resolved string. Current main uses `removePathWithinRoot()`, so this separates validation from deletion and can be redirected outside the matched root if a parent is swapped between the check and the operation. Use the pinned removal helper with `sandboxResult.matchedRoot` and `sandboxResult.relative`.
  - confidence: 0.9
- **[P2] Resolve sandbox paths before comparing allowed roots:** `src/agents/pi-tools.read.ts:601`
  - body: In sandbox runs, this wrapper checks the container path against host-configured `additionalRoots` before `SandboxFsBridge.resolvePath` maps bind mounts. A bind such as `/host/shared:/shared` with `allowedRoots: ["/host/shared"]` still rejects `/shared/file`, so the new setting fails for the Docker bind-mount case.
  - confidence: 0.87
- **[P2] Preserve default apply_patch enablement:** `src/agents/pi-tools.ts:360-361`
  - body: This branch requires `tools.exec.applyPatch.enabled` to be explicitly true before creating `apply_patch`. Current main keeps `apply_patch` available for OpenAI models unless it is explicitly disabled, so merging this would silently remove the tool from normal coding profiles with no `applyPatch` config.
  - confidence: 0.86
