## Summary

Keep this open. Current main still has the reported provider-namespace hazard: even though configured model parsing has an exact-provider escape path, the agent runtime path immediately re-normalizes the resolved default through `normalizeModelRef`, and `normalizeProviderId` still maps `kimi-coding` to `kimi`.

## What This Changes

The issue asks OpenClaw to preserve provider-namespaced custom model refs such as `kimi-coding/k2p5` instead of collapsing the provider to `kimi` during model selection.

## Best Possible Solution

Keep the issue open and land a narrow model-selection fix that preserves exact configured provider IDs such as `kimi-coding` through default resolution, runtime execution, allowed-model checks, and status display, while retaining documented `kimi`/`kimi-code` compatibility for the bundled Kimi provider.

## Risks / Open Questions

- Removing `kimi-coding` from `normalizeProviderId` globally could regress existing Kimi auth/plugin alias behavior; an exact-configured-provider-first fix may be safer than a broad alias deletion.
- The fix needs to cover both configured defaults and stored/session model overrides; current exact parsing alone is not enough because later runtime code re-normalizes the provider.
- Kimi docs now present `kimi/kimi-code` as canonical and `kimi/k2p5` as legacy, so the repair should not make canonical onboarding or legacy Kimi compatibility worse.
