## Summary

Keep this issue open. Current main still collapses the reported `kimi-coding` provider namespace to `kimi`, and the runtime model-selection path can re-normalize configured defaults and stored overrides after the partial exact-provider parsing path. PR #74315 is an open closing candidate, so the issue should remain open until that PR or a safer variant lands.

## What This Changes

The issue asks OpenClaw to preserve custom provider namespaces such as `kimi-coding/k2p5` during model resolution so status and runtime selection use the configured provider directly instead of falling back from `kimi/k2p5`.

## Best Possible Solution

Preserve exact configured provider IDs through model parsing, catalog lookup, runtime execution, allowlist validation, stored overrides, and status output, while retaining the intended `kimi` and `kimi-code` compatibility behavior for the bundled Kimi provider.

## Reproduction Assessment

Yes. The deterministic source path is clear on current main: `normalizeProviderId("kimi-coding")` returns `kimi`, `normalizeModelRef` applies that mapping, and `agent-command` uses that helper for default and stored model selections; the existing tests also assert the collapsing behavior.

## Solution Assessment

Unclear. PR #74315 is a plausible narrow fix, but maintainers should decide whether global alias removal is the intended compatibility contract or whether a config-aware exact-provider-first resolver is safer.
