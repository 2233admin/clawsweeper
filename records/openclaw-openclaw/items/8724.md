## Summary

Keep this issue open. Current main still has adjacent run-level, provider request, and reset-on-chunk idle timeout controls, but it does not provide the requested typed global/per-model non-resetting generation timeout, and the proposed per-model key would still be rejected by the strict agent defaults schema.

## What This Changes

The issue requests configurable global and per-model generation timeouts so long-running model output can be aborted and optionally routed to fallback models.

## Best Possible Solution

Add a backwards-compatible generation-timeout contract at agent defaults and per-model catalog entries, resolve the effective non-resetting budget for each selected fallback candidate, enforce it during active generation with timeout classification/logging, and keep first-token timeout/provider-skip behavior as separate work.

## Repair Work Prompt

Add a backwards-compatible typed generation timeout surface for agent generation, including a global default under `agents.defaults` and per-model overrides under `agents.defaults.models[provider/model]`. Resolve the effective timeout for each selected primary/fallback candidate and enforce it as a non-resetting total generation budget during active model generation; do not repurpose provider HTTP request timeouts or the reset-on-chunk idle watchdog. Classify/log generation timeout aborts so existing fallback handling can treat them as timeout failures. Keep first-token timeout and provider-level skip behavior explicitly out of scope. Add focused tests for schema acceptance/rejection, effective timeout resolution, non-resetting generation timeout enforcement, fallback timeout classification, and preservation of existing idle/provider timeout behavior. Do not change dependency pins, provider-specific defaults, or unrelated model routing policy. Changelog entry is expected because this is user-facing config behavior; credit the issue reporter per repo changelog rules.
