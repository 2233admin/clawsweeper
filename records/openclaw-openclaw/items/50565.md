## Summary

Keep open. Current main no longer shows the exact old `feishu: tool call` strings, but the central bug remains: embedded tool lifecycle logging is still not attributed to the originating gateway channel, and the available channel/provider identity is not threaded into the tool subscription logger.

## Best Possible Solution

Keep this issue open and fix it with a narrow, generic channel-aware logging change: thread the existing originating channel/provider identity into the embedded subscription/tool-handler context, derive the gateway channel subsystem when available, fall back to `agent/embedded`, and add regression coverage proving non-Feishu channel tool logs are not attributed to Feishu.

Fix the logging attribution bug from https://github.com/openclaw/openclaw/issues/50565. Current main still logs embedded tool lifecycle events through a static embedded logger instead of the originating gateway channel. Thread the existing `messageChannel`/`messageProvider` identity from the embedded agent run path into `subscribeEmbeddedPiSession` and the tool-handler context, derive a generic channel subsystem logger such as `gateway/channels/<channel>` when present, and fall back to `agent/embedded` when no channel is known. Do not hard-code bundled channel IDs in core, do not change tool execution/permission semantics, and do not add raw secret-bearing tool-argument logging. Add focused regression tests that prove a Telegram-origin run logs tool lifecycle activity under the Telegram channel subsystem and does not attribute non-Feishu tool calls to Feishu.
