## Summary

Keep open. Current main no longer has the old Feishu-specific `tool call` log text, but the remaining attribution bug is still present: embedded tool lifecycle debug logs use a static `agent/embedded` logger because the subscription layer is not given the originating channel identity.

## What This Changes

The issue asks for tool call lifecycle log entries to be attributed to the originating channel subsystem instead of a hardcoded or static logger.

## Best Possible Solution

Add a typed channel-origin or logger seam from `runEmbeddedAttempt` into `subscribeEmbeddedPiSession`, derive a safe channel subsystem logger for message-channel runs, keep `agent/embedded` as fallback, and cover tool start/end attribution with focused regression tests.

## Repair Work Prompt

Fix tool lifecycle log subsystem attribution for embedded runs tracked by https://github.com/openclaw/openclaw/issues/50565. Current main still routes `embedded run tool start/end` through the static `agent/embedded` logger in `subscribeEmbeddedPiSession`, even though `runEmbeddedAttempt` has `messageChannel` / `messageProvider`. Add a typed optional channel-origin or logger seam from `runEmbeddedAttempt` through `buildEmbeddedSubscriptionParams` into `subscribeEmbeddedPiSession`; derive a safe `gateway/channels/<channel>` logger for valid message-channel runs and keep `agent/embedded` for channel-less/internal runs. Add focused tests proving tool start/end attribution uses the channel logger when a message channel is supplied and falls back otherwise. Preserve existing sanitized logging: do not add raw tool args, exec commands, tool output, or sensitive result payloads. Likely no docs are needed; add a single-line changelog Fixes entry only if the active changelog treats diagnostics/logging attribution as user-facing, crediting @xucheng and @Hollychou924 per repo policy.
