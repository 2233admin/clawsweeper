## Summary

Keep open. Current main still drops provider-stream session identity, and this PR is the focused implementation candidate paired with a concrete downstream glueclaw transcript-bleed report. The patch is broadly the right additive API direction, with one non-blocking documentation correction before merge.

## What This Changes

The PR adds optional `sessionId` and `sessionKey` fields to `ProviderCreateStreamFnContext` and threads them into provider stream registration from embedded attempt, compaction, and BTW paths.

## Best Possible Solution

Land the additive optional fields if maintainers accept the API shape, correct the sessionId wording, add focused passthrough tests for attempt/compaction/BTW, document the trusted provider-plugin boundary, and then resolve the linked createStreamFn issue.

## Reproduction Assessment

Yes. Current-main source inspection shows session identity exists in the embedded, compaction, and BTW flows but is not included in the createStreamFn plugin context; the linked glueclaw report gives a concrete multi-conversation transcript-bleed scenario.

## Solution Assessment

Yes, with a small docs/test adjustment. Adding optional `sessionId` and `sessionKey` to the existing provider createStreamFn context is the narrowest generic core seam for custom provider transports; the related #54725 prepareExtraParams work is adjacent but not a replacement.
