## Summary

Keep #57600 open. Current main still lacks the PR's binary-document read fallback, #57612 is still open/unmerged in the provided context, and the related issue #35406 remains open. This is a plausible focused read-tool fix rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open or explicitly supersede it only after a canonical landed change covers the same read-tool behavior. The best implementation path is a small, tested read-wrapper change in src/agents/pi-tools.read.ts that uses the existing MIME helper for both host and sandbox reads, returns a structured fallback for unsupported binary document formats, and aligns with any native DOCX/PDF extraction follow-up tracked by #35406/#57612.

## Risks / Open Questions

- The PR's approach appears to double-read files after the base read, so maintainers should review performance, path consistency, and host/sandbox behavior before merge.
- The fallback includes PDF, while OpenClaw already has PDF extraction for input_file/document-extract flows; the final behavior should avoid blocking intended PDF extraction surfaces.
- #57612 is narrower native DOCX extraction and does not cover the full .pdf/.doc/.xls/.ppt/.xlsx/.pptx guard proposed here, so closing #57600 now could drop still-relevant generic read-tool work.
