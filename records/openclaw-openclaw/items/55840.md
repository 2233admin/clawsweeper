## Summary

Current main intentionally removed the Chrome extension relay and migrates legacy configs to Chrome MCP `existing-session`. That is a documented replacement path, but it does not restore the requested `driver: "extension"` relay, bundled extension files, integrated port 18792 relay, or extension-specific Chrome debugger behavior. This needs maintainer product/security judgment rather than conservative cleanup closure.

## Best Possible Solution

Keep this issue open for a browser-plugin maintainer decision. The concrete paths are either to intentionally restore and harden an extension-relay driver in the bundled browser plugin, or to define a narrow browser-driver/plugin API and move an extension relay implementation to ClawHub with docs that clearly explain the Chrome MCP migration limits.

## Risks / Open Questions

- Closing as implemented would conflate the Chrome MCP migration with the requested restoration of extension-specific relay capabilities such as integrated port 18792, Chrome debugger controls, and extension auto-reattach behavior.
- Restoring this path would touch privileged browser control, Chrome debugger/CDP authority, auth/token handling, and packaged browser assets, so it needs explicit browser/security maintainer review.
- A ClawHub/plugin answer may be viable only if maintainers accept replacing the whole bundled browser plugin or add a narrower browser-driver extension API; current main does not show a simple driver-registration surface.
