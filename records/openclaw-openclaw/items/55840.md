## Summary

Keep this issue open. Current main still treats Chrome Extension Relay as removed: `driver: "extension"` is rejected/migrated away, the relay lifecycle helper is an intentional no-op, docs point users to Chrome MCP `existing-session`, and only icon remnants remain under `assets/chrome-extension` without extension runtime files.

## Best Possible Solution

Keep this issue open for browser-plugin maintainer follow-up. The best path is either to intentionally restore and harden an optional Chrome Extension Relay driver in the bundled browser plugin, with schema/docs/tests/package assets/security review, or to design a documented browser-driver/plugin API so an external relay plugin can provide the missing behavior with clear migration guidance.

## Risks / Open Questions

- Closing as implemented would conflate Chrome MCP `existing-session` with the requested Chrome Extension Relay behavior, including extension debugger control, auto-reattach, video control, and chat automation use cases.
- Restoring this would reintroduce privileged existing-browser control, relay authentication, packaged extension assets, and Chrome debugger/CDP authority, so it needs explicit browser/security maintainer judgment.
- A ClawHub plugin could be a future direction only if the browser-driver/plugin seam is made clear enough; current main exposes browser config facades and full plugin replacement paths, but no simple public driver-registration API for adding `driver: "extension"` to the bundled browser plugin.
