## Summary

Current main still does not restore Chrome Extension Relay support. The browser plugin keeps the old relay lifecycle as an intentional no-op, the profile config/schema/API reject `driver: "extension"`, doctor/docs migrate legacy configs to Chrome MCP `existing-session`, and no current plugin API appears to let an external relay driver register cleanly. Related #53599 and #72727 confirm the same remaining gap, while #19289 tracks a broader browser-driver plugin interface rather than a shipped replacement.

## Best Possible Solution

Keep this open for browser-plugin maintainer product/security review. The best path is either an explicit, opt-in, hardened restoration of the Chrome Extension Relay driver with docs/tests/security review, or a stable browser backend/plugin API that lets external implementations such as the ClawHosters relay integrate without private internals, plus migration docs that call out where Chrome MCP `existing-session` is not equivalent.

## Risks / Open Questions

- Restoring this as-is would reintroduce a privileged Chrome extension/debugger relay, WebSocket authentication, origin validation, and remote-browser control surface that needs explicit security review.
- Chrome MCP `existing-session` does not cover all reported old-relay capabilities, especially extension-specific video/debugger control, auto re-attach behavior, chat automation expectations, and cross-machine browser control without a node host.
- A ClawHub/plugin solution is not currently sufficient unless OpenClaw first exposes a stable browser profile-driver or transport API; otherwise external relays must rely on private browser internals.
