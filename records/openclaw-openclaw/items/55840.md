## Summary

Current main still does not restore Chrome Extension Relay support. The browser plugin keeps the relay startup helper as an intentional no-op, config/profile creation still rejects `driver: "extension"`, docs/changelog describe Chrome MCP `existing-session` as the migration path, and existing-session remains narrower than the requested extension-specific debugger, auto-reattach, video-control, chat-automation, and integrated port-18792 relay behavior. Related open issues #53599 and #19289 are useful context but do not fully supersede this issue.

## Best Possible Solution

Keep this issue open for browser-plugin maintainer review. The clean product direction is either an explicit decision to restore and harden an optional Chrome Extension Relay driver, or a documented browser-driver/plugin API that lets an external relay plugin implement the old extension-specific capabilities with clear migration docs explaining where Chrome MCP `existing-session` is not equivalent.

## Risks / Open Questions

- Restoring this feature would reintroduce privileged existing-browser control through Chrome debugger/CDP authority, relay authentication, WebSocket/origin handling, and packaged extension assets, so it needs explicit browser/security maintainer judgment.
- Chrome MCP `existing-session` is not equivalent to the requested extension relay behavior: current docs list feature limits and current code has no `extension` driver path.
- Related #53599 and #72727 overlap on port-18792/remote relay availability, and #19289 overlaps on a browser-driver plugin API, but none fully resolves this issue’s local extension-specific video/chat/auto-reattach restoration request.
- A ClawHub/plugin route may be viable later, but current main still uses a closed browser driver set and does not expose a simple third-party browser-driver registration seam.
