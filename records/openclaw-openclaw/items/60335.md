## Summary

Keep this open. Current main still has the reported Slack behavior: text-only thread replies can fall back to hydrated thread-starter media, and the linked fix PR is still open and unmerged, so the issue should stay open until that PR is reviewed, refreshed, landed, or explicitly superseded.

## What This Changes

Slack thread replies should include only media attached to the current reply, not re-download or re-attach the thread starter's media as current-turn media.

## Best Possible Solution

Refresh and land the Slack fix so thread replies keep parent text/history context but never populate MediaPath or MediaPaths from starter.files; direct media attached to the reply should continue to work, and the existing parent-file ID filter should remain as a defense for inherited Slack event files.

## Reproduction Assessment

Yes. A static current-main reproduction path is clear: a thread reply with no own media, a starter with files, and starter text reaches resolveSlackThreadContextData, hydrates starter.files, and then prepareSlackMessage exposes those paths in the media payload.

## Solution Assessment

Yes. The linked direction is the narrowest maintainable fix: stop hydrating thread-starter media for replies while retaining starter text and direct reply media; the current file-ID filter alone is insufficient because it only filters files already present on the reply event.
