## Summary

Keep this PR open, but it is not merge-ready. Current main still has unbounded tool-schema fields, so the hardening goal is valid, but the proposed branch introduces several schema caps that contradict current runtime and docs contracts, and its browser change targets a path that has moved to the browser plugin.

## What This Changes

The PR adds TypeBox maxItems, minimum, maximum, and description metadata across built-in agent tool schemas for browser, canvas, gateway, memory, message, node, session, subagent, and web-fetch tools.

## Best Possible Solution

Rebase the hardening onto current main, apply bounds only where they are backed by runtime clamps, provider API limits, protocol schemas, or user-facing docs, move browser changes to the plugin-owned schema, and add focused contract tests that keep schema hints aligned with those limits.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Raise node timeout bound to cover max recordings:** `src/agents/tools/nodes-tool.ts:113-114`
  - body: The new 120s timeoutMs cap is lower than valid media durations: nodes still allow 300000 ms recordings, and screen_record can run for that full duration. A caller recording near the max can no longer set a gateway timeout long enough for the operation to complete.
  - confidence: 0.93
- **[P2] Keep sessions_list compatible with 200-row lookups:** `src/agents/tools/sessions-list-tool.ts:36`
  - body: Capping sessions_list.limit at 50 rejects a pattern OpenClaw still uses internally: announce-target resolution asks sessions.list for 200 rows, and the gateway protocol has no maximum. Larger installs can lose the ability to find older or less-active sessions.
  - confidence: 0.9
- **[P2] Preserve long-running subagent timeouts:** `src/agents/tools/sessions-spawn-tool.ts:44-46`
  - body: The runtime and docs still allow any non-negative runTimeoutSeconds value, with 0 meaning no timeout. Adding maximum: 3600 makes legitimate subagent jobs intended to run past an hour fail schema validation before reaching the existing runtime behavior.
  - confidence: 0.91
- **[P2] Use the active browser schema and backend snapshot cap:** `src/agents/tools/browser-tool.schema.ts:98`
  - body: The branch edits the old browser schema path and lowers snapshot limit to 1000, while current main registers the plugin-owned browser schema and the active backends still clamp at 2000. A rebased patch should harden the active schema without reducing supported snapshot size.
  - confidence: 0.84
- **[P2] Permit Discord media channel type:** `src/agents/tools/message-tool.ts:329`
  - body: Discord's GuildMedia channel type is supported by the current Discord runtime as forum-like. Capping channel management type at 15 blocks valid channel-create/thread behavior for that supported channel type.
  - confidence: 0.88
