## Summary

Keep this PR open. Current main still has unbounded tool-schema fields that match the PR's hardening goal, so it is not obsolete or implemented. The stale branch is not merge-ready as-is because several proposed caps conflict with current runtime-supported behavior and the browser schema now lives under the bundled browser plugin path.

## Best Possible Solution

Keep the PR open, but require a rebase and a narrower runtime-aligned revision: add maxItems/maximum/maxLength only where current runtime, provider docs/types, protocol schemas, or existing constants support the value; preserve intentional no-timeout and long-running workflows; update the browser plugin-owned schema path; and add focused schema/contract tests so future caps stay aligned with runtime behavior.

## Risks / Open Questions

- Merging the stale PR as-is could reject valid tool calls for 200-row session lookup, long/no-timeout subagent runs, long node media recordings, large browser snapshots, or newer Discord channel types.
- The browser tool schema has moved to the bundled browser plugin path, so the PR's browser changes need a rebase and ownership update before review.
- Schema-only hardening without focused contract tests can drift from runtime/provider limits and silently block valid workflows.
