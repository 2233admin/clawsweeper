## Summary

Keep this PR open. Current main still has unbounded tool-schema parameters that match the PR's hardening goal, so the work is not implemented or obsolete. The stale PR is not merge-ready as-is because several proposed bounds conflict with current runtime, docs, or provider-supported behavior and could reject valid tool calls.

## Best Possible Solution

Keep the PR open, but require a rebase and a narrower runtime-aligned revision: add maxItems/maximum/maxLength only where backed by current runtime, provider, protocol, or docs limits; preserve intentional long-running and no-timeout workflows; update the browser-plugin-owned schema path; and add focused schema/contract tests that fail when schema bounds drift from runtime behavior.

## Risks / Open Questions

- Merging the stale PR as-is could turn valid tool calls into schema-validation failures for long node media recordings, long/no-timeout subagent runs, 200-row session lookups, 2000-node browser snapshots, or Discord GuildMedia workflows.
- The hardening is schema-only; without contract tests tied to runtime/provider limits, future caps can drift from actual behavior.
- The browser tool now lives under the bundled browser plugin path on current main, so the PR needs a rebase/ownership update before it can be evaluated for merge.
