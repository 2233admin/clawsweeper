## Summary

Keep open. Current main still has the unsafe combination described: `sessionTarget: "current"` is persisted as a live `session:<key>`, one-shot `at` jobs default to `deleteAfterRun: true`, session-bound cron runs use the requested persistent key, and direct-delivery cleanup still deletes `params.agentSessionKey` without proving that key is cron-owned or isolated.

## Best Possible Solution

Keep this open and land a focused cron fix that limits delete-after-run session deletion to cron-owned disposable run sessions. Preserve explicit `session:*` execution semantics, but make direct-delivery cleanup skip live/current/custom persistent session keys unless the run can prove the key is isolated/cron-owned. Add a regression for a non-default Signal-style direct DM key created through `sessionTarget: "current"` or explicit `session:<live-key>`, and keep the existing #67807 isolated cleanup cases green.

## Clownfish Work Prompt

Current main resolves `sessionTarget: "current"` to a persistent `session:<live session key>`, one-shot `at` jobs default to `deleteAfterRun: true`, and direct delivery cleanup calls `sessions.delete` on `params.agentSessionKey`. Implement a focused fix so delete-after-run session cleanup only deletes cron-owned disposable run sessions, not live/current/custom persistent `session:*` conversation keys. Preserve the existing #67807 behavior for isolated cron direct deliveries, including text, silent, threaded, structured, and structured-silent direct delivery cleanup. Add regression coverage for a non-default live DM-style key such as `agent:ops:signal:default:direct:recipient` created through `sessionTarget: "current"` or explicit `session:<live-key>` with `deleteAfterRun: true`, verifying direct delivery does not call `sessions.delete` for that live key and does not rebind/delete the live conversation. Do not change job-store one-shot deletion semantics, delivery routing, provider/channel adapters, or unrelated retention policy.
