## Summary

Keep open. Current main still has the unsafe combination described: `current` cron session targets are persisted as `session:<live key>`, explicit session targets run on that persistent key, only `isolated` forces a fresh run session, and direct-delivery delete-after-run cleanup still calls `sessions.delete` on `params.agentSessionKey` without proving the key is cron-owned.

## What This Changes

The issue requests preventing one-shot `deleteAfterRun` cron jobs created from a live non-default DM from deleting or rebinding that live conversation session when `sessionTarget` resolves to `current` or an explicit `session:<live key>`.

## Best Possible Solution

Land a focused cron runtime fix that limits delete-after-run session deletion to cron-owned disposable run sessions. Preserve explicit `session:*` execution semantics, skip direct-delivery cleanup for live/current/custom persistent session keys unless the run can prove the key is isolated or cron-owned, and add regression coverage for a non-default Signal-style direct DM key created through `current` or explicit `session:<live-key>`.

## Repair Work Prompt

Fix the cron delete-after-run live-session deletion bug from issue #69467. Current main normalizes `sessionTarget: "current"` to `session:<live session key>` and explicit `session:*` jobs run on that persistent key, but `dispatchCronDelivery` direct cleanup deletes `params.agentSessionKey` whenever `deleteAfterRun` is true. Implement a narrow ownership guard so delete-after-run direct delivery cleanup only deletes cron-owned disposable run sessions, while preserving explicit `session:*` execution semantics and preserving #67807 isolated cleanup behavior.

Expected boundary:
- Do not convert all `current` or `session:*` jobs to isolated; those are documented persistent-session modes.
- Make direct-delivery cleanup skip live/current/custom persistent session keys unless the run can prove the key is isolated or cron-owned.
- Keep isolated `deleteAfterRun` direct cleanup working for text, silent, threaded, structured, and structured-silent delivery paths.
- Add a regression for a non-default Signal-style direct DM key such as `agent:agent-a:signal:default:direct:recipient`, covering `sessionTarget: "current"` after normalization or explicit `session:<live-key>`, and assert cleanup does not call `sessions.delete` for that live key.
- Add or update run/session identity tests if the fix passes an explicit ownership flag from `runCronIsolatedAgentTurn` into `dispatchCronDelivery`.
