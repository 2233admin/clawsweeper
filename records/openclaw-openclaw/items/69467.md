## Summary

Keep open. Current main still composes three unsafe behaviors: `current` can persist a live chat session key, `session:*` targets are passed into the cron agent-turn runner as the run session key, and delete-after-run direct cleanup deletes that same key. The detailed issue logs therefore still match current source behavior, while the related open PR #57831 targets the same root but is not merged and has prior review blockers.

## What This Changes

Prevent one-shot `deleteAfterRun` cron agent turns created from live non-default DM sessions from deleting or rebinding the live conversation when `sessionTarget` resolves to `current` or a concrete `session:*` key.

## Best Possible Solution

Land a narrow cron/session safety fix that prevents delete-after-run cleanup from deleting user-facing `session:*` chat keys. Direct-message `current` bindings should either fall back to an owned isolated cron run session or reuse the live session without cleanup, with runtime fallback for legacy dirty jobs and regression tests for non-default Signal/DM direct delivery plus `delivery.mode: "none"` message-tool sends.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: create an `agentTurn` one-shot cron job from a direct session with `sessionTarget: "current"`, let normalization persist `session:<live key>` and default `deleteAfterRun`, then direct-delivery cleanup will call `sessions.delete` on that resolved session key.

## Solution Assessment

Yes for the requested product direction: live user DM sessions must not be treated as cron-owned disposable sessions. The related PR #57831 is directionally aligned, but the safer final implementation should use canonical session-shape parsing, preserve safe custom/group sessions, and guard the delete path itself.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because cron cleanup can delete or rebind a user-facing direct-message session key; no PR diff was reviewed here.

Concerns:

- **[medium] Cron cleanup can target live chat sessions:** `src/cron/isolated-agent/delivery-dispatch.ts:476`
  - body: When a one-shot cron job resolves to a direct `session:*` key, delete-after-run cleanup calls `sessions.delete` on that key without proving it is cron-owned. That can cause data availability/integrity loss for the live DM transcript binding.
  - confidence: 0.9
- **[medium] Persistent direct-session targets broaden cron data scope:** `src/cron/normalize.ts:590`
  - body: `current` and explicit `session:*` cron targets can bind scheduled work to a human direct-message session, giving scheduled runs access to and writeback into live conversation context unless the target is downgraded or cleanup is disabled.
  - confidence: 0.86
