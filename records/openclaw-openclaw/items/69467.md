## Summary

Keep open. Current main still has the reported failure path: `current` can persist as a concrete live session key, non-main `agentTurn` cron jobs run through the detached cron runner, and `deleteAfterRun` direct-delivery cleanup deletes `agentSessionKey` without checking whether that key is cron-owned or a live chat binding. The broader ephemeral-session request in #60745 is related, but it does not supersede this narrower live-session data-continuity bug.

## What This Changes

The issue requests that one-shot cron reminders with `deleteAfterRun` stop deleting or rebinding live non-default DM sessions when `sessionTarget` resolves to `current` or `session:<live-dm-key>`.

## Best Possible Solution

Keep the issue open and make `deleteAfterRun` cleanup distinguish cron-owned run sessions from persistent live conversation sessions. Isolated cron sessions should still be cleaned up, one-shot job definitions should still auto-delete after success, and `current`/`session:<live-channel-key>` runs should either reuse the live session without deleting it or run in a separate disposable working session while preserving the live delivery target.

## Reproduction Assessment

Yes. The issue body gives a concrete live-log timeline, and current main provides a deterministic code-level reproduction: create a non-main `agentTurn` one-shot from a live DM with `sessionTarget: "current"`, let it normalize to `session:<live-key>`, then observe direct-delivery `deleteAfterRun` cleanup call `sessions.delete` on that live key.

## Solution Assessment

No, current main is not the best way to solve it because cleanup is keyed to `agentSessionKey` regardless of session ownership. The narrow maintainable fix is to limit direct cleanup to cron-owned isolated run keys or otherwise skip cleanup for live `current`/`session:*` bindings while adding regression coverage.
