## Summary

Keep open. The issue's reported path still exists on current main: `current` is normalized to `session:<caller-session-key>`, session-bound cron jobs are run against that unwrapped key, only `sessionTarget === "isolated"` forces a new cron session, and `deleteAfterRun` direct-delivery cleanup still calls `sessions.delete` on `params.agentSessionKey` without checking whether that key is a live chat binding. Related PR #67807 fixed skipped cleanup for direct deliveries, but that makes cleanup more reliable; it does not add the missing guard for live/current session-bound reminders.

## Best Possible Solution

Keep the issue open and fix it in the cron/session owner path. `deleteAfterRun` session cleanup should only delete cron-owned isolated run sessions, or one-shot reminders targeting `current`/`session:<live-key>` should run in an ephemeral working session while preserving delivery context. Add a regression around a non-default live DM key with one-shot `at`, `deleteAfterRun: true`, direct delivery and `delivery.mode: "none"` plus message-tool delivery, asserting `sessions.delete` is not called for the live key and the live session binding is preserved.
