## Summary

Keep open. The direction is useful, and the latest PR commit addresses the earlier Greptile system-event concern, but the patch is not merge-ready because the new `PluginHookAgentContext` sender/chat fields are only wired through the early reply hook path and the harness helper; current CLI and embedded-agent hook contexts still omit them for other agent hooks.

## What This Changes

The PR adds optional sender/chat fields to plugin agent hook context, adds `SenderSessionKey` to `MsgContext`, and gives that sender session key priority during reply session initialization.

## Best Possible Solution

Keep the additive SDK shape, but centralize or consistently reuse agent hook context construction so sender/chat fields flow through `get-reply`, embedded Pi runtime hooks, CLI runtime hooks, cron reply interception, and harness helpers, with focused tests for at least one reply hook and one runtime lifecycle/LLM hook.

## Reproduction Assessment

Yes. A static reproduction is enough: register a plugin hook such as `llm_input` or `agent_end` on a channel message with `SenderId`, and the current embedded/CLI hook context builders still have no path to include `senderId` or `chatId`.

## Solution Assessment

No, not as currently scoped. Adding optional fields is the right compatibility direction, but the implementation should wire the same normalized identity fields through all producers of `PluginHookAgentContext` or deliberately narrow the public contract and PR title/body.
