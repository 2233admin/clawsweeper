## Summary

Keep open. Current main still applies the memory-only `toolsAllow` after the full embedded-run tool registry is constructed, and the run still pays fixed system-prompt, resource-loader, and agent-session setup before streaming. There are partial mitigations already in place, but they do not implement the requested fast path or remove the setup floor described in the report.

## What This Changes

Request a `runEmbeddedPiAgent` fast-path opt-in so active-memory and other memory-only embedded runs can avoid full tool registry construction and other fixed setup overhead.

## Best Possible Solution

Add a generic embedded-run lightweight construction path that can build only explicitly allowed tools, including plugin-owned memory tools, while preserving sandbox, tool-policy, auth, and plugin boundaries. Treat resource-loader or provider-session reuse as a separate benchmarked design step rather than folding it into an unreviewed boolean flag.

## Reproduction Assessment

Yes for the source-level root cause: current main constructs the full embedded tool registry before applying the memory-only allowlist, and active-memory uses that path on each eligible turn. The exact timing numbers from the report still need live benchmark proof on a representative workspace.

## Solution Assessment

Unclear. A fast-path concept is directionally right, but the safest maintainable fix is likely a narrower construction-time tool allowlist or lightweight embedded profile first, with benchmark coverage, before adding resource-loader or stream warm pools.
