## Summary

Keep open. Current main appears to avoid the exact wrong-header path for an Anthropic API-key-shaped value because `sk-ant-api` is sent through `x-api-key`, but OpenClaw still accepts and can prioritize Anthropic `type: "token"` profiles and lacks a targeted warning or docs guidance for Enterprise/API-only accounts.

## What This Changes

The issue requests Anthropic auth-profile validation, diagnostics, and documentation so Enterprise API keys are stored as `api_key` profiles rather than `token` profiles.

## Best Possible Solution

Preserve Anthropic setup-token and Claude CLI token support, but add provider-owned diagnostics for Anthropic token profiles whose stored/resolved value looks like an Anthropic API key, guide users to `api_key`/`key`, and document the difference in the public auth/model docs.

## Reproduction Assessment

Partially yes. Current-main source gives a high-confidence static reproduction for the remaining gap: an Anthropic `type: "token"` profile is still eligible and can be selected, while no doctor/status/docs warning flags API-key-shaped token material; the live Enterprise 529 loop needs a real Enterprise credential.

## Solution Assessment

Yes for keeping this open as a narrow cleanup, but no to removing the global `token` schema. The safer fix is an Anthropic-owned warning/doctor/status/docs path that does not break supported setup-token credentials.
