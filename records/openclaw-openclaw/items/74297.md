## Summary

Keep open. Current main appears to avoid the exact wrong-header behavior for `sk-ant-api` credentials, but the reported cleanup is still valid: Anthropic `type: "token"` profiles remain accepted/selectable and there is no targeted doctor/status warning or explicit docs note telling Enterprise/API-only users to store Anthropic API keys as `type: "api_key"` with `key`.

## What This Changes

The issue requests Anthropic auth-profile validation, diagnostics, and documentation so Enterprise/API-key accounts are not configured with `type: "token"`.

## Best Possible Solution

Preserve supported Anthropic setup-token and Claude CLI auth, but add a provider-owned diagnostic for Anthropic token profiles whose stored value looks like an Anthropic API key, and document that Anthropic Enterprise/API-only accounts should use `type: "api_key"` with `key`.

## Reproduction Assessment

Partially yes. Static current-main checks reproduce the accepted/selectable Anthropic `type: "token"` profile and the missing targeted warning/docs path; the live 529 loop needs an Anthropic Enterprise credential and was not reproduced here.

## Solution Assessment

Yes, a narrow provider-owned warning plus docs update is the best direction. Removing `token` from the global auth-profile schema would be too broad because Anthropic setup-token auth is still an intentional supported path.
