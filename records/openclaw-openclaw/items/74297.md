## Summary

Keep open. Current main appears to have fixed the likely wrong-header transport half by treating `sk-ant-api`-shaped values as Anthropic API keys, but the requested diagnostic and docs cleanup remains real: Anthropic `token` profiles are still valid, can be preferred ahead of `api_key` profiles, and static token profiles are not surfaced by auth health or doctor.

## What This Changes

The issue requests validation, diagnostics, and documentation for Anthropic Enterprise API credentials mistakenly stored as `type: "token"` in `auth-profiles.json`.

## Best Possible Solution

Preserve Anthropic setup-token support, but add an Anthropic-specific diagnostic in auth health, doctor, or model status when an Anthropic token profile contains an API-key-shaped credential, and document the exact `api_key`/`key` profile shape for API-only and Enterprise accounts.

## Reproduction Assessment

Yes for the remaining repo behavior: current source accepts an Anthropic `type: "token"` profile, can order it before `api_key`, and classifies a non-expiring token as static. The provider-side 529 loop still requires live Enterprise credentials to reproduce end to end.

## Solution Assessment

Not applicable as a patch review. The maintainable fix is a narrow provider-owned warning and docs update, not removing the supported Anthropic setup-token path or changing generic token semantics.
