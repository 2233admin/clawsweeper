## Summary

Keep open. Current main appears to avoid the exact wrong-header failure for API-key-shaped Anthropic credentials, but Anthropic `type: "token"` profiles are still accepted, can be selected ahead of API-key profiles, and have no targeted doctor/status warning when the stored value looks like an Anthropic API key. The remaining work is narrow provider-owned diagnostics and docs, not removal of supported setup-token auth.

## What This Changes

The issue asks OpenClaw to validate or warn about Anthropic Enterprise API keys stored as `type: "token"` and to document that API-only Anthropic accounts should use `type: "api_key"`.

## Best Possible Solution

Preserve Anthropic setup-token and Claude CLI token support, but add an Anthropic-owned diagnostic in doctor/status for token profiles whose inline value or resolved SecretRef looks like an Anthropic API key, and update the public auth/provider docs to show Enterprise/API-only accounts using `type: "api_key"` with `key` or `ANTHROPIC_API_KEY`.

## Reproduction Assessment

Partially yes. Static reproduction on current main is high-confidence: store an Anthropic `type: "token"` profile with an API-key-shaped value and the parser/order/health paths accept it without a targeted warning; live 529 reproduction still needs an Enterprise credential.

## Solution Assessment

Not applicable as a patch review. Removing `token` from the schema would be the wrong fix because Anthropic setup-token auth is intentionally supported; the safer path is a provider-specific warning plus docs while keeping current x-api-key routing for API-key-shaped values.
