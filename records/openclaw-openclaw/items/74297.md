## Summary

Keep open. Current main still accepts Anthropic `type: "token"` auth profiles and still supports Anthropic setup-token as an intentional bearer-token path, while the docs/status paths do not specifically warn when an API-key-shaped Anthropic credential is stored as a token. The current Anthropic transport appears to route non-`sk-ant-oat` credentials through `x-api-key`, so the exact wrong-header 529 path may already be avoided, but the requested validation, diagnostic, and Enterprise/API-key documentation cleanup remains unimplemented.

## What This Changes

The issue requests Anthropic auth-profile validation, diagnostics, and documentation so API-key-shaped Enterprise credentials are not stored or treated as `type: "token"`.

## Best Possible Solution

Preserve Anthropic setup-token and Claude CLI token support, then add an Anthropic-specific diagnostic/docs path for token profiles containing direct API-key-shaped credentials. The end state should tell Enterprise/API-only users to store Anthropic API keys as `type: "api_key"` with `key`, while leaving provider-neutral token semantics intact.

## Reproduction Assessment

Unclear for the exact 529 loop because it requires a live Anthropic Enterprise account and current main’s transport routes non-`sk-ant-oat` credentials as `x-api-key`. There is a high-confidence source reproduction of the remaining cleanup gap: current main still accepts token profiles and the docs/status paths lack the requested Anthropic API-key-shaped token warning.

## Solution Assessment

Yes for the conservative direction: the best fix is a targeted Anthropic diagnostic and documentation update, not removal of `type: "token"` from the global schema. A safer implementation should reuse existing auth-status/doctor/profile validation surfaces and keep credential material redacted.
