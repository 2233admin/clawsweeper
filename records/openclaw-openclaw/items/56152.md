## Summary

Keep open. The PR addresses a plausible macOS `/tmp` vs `/private/tmp` local-media false negative, but current `main` has not implemented that variant-aware behavior, and the PR patch has unresolved security review findings on the local filesystem boundary. It needs a revised implementation rather than cleanup closure.

## Best Possible Solution

Revise the PR instead of closing it: keep canonical media-path enforcement when `fs.realpath(mediaPath)` succeeds, only use the lexical media path when realpath fails, allow both lexical and canonical variants for configured roots as needed, preserve the workspace-sibling guard against canonical variants, and add tests for `/tmp`↔`/private/tmp`, symlink escape outside the allowed root, and workspace-sibling symlink cases.

There is no `resolvePathVariants` helper or lexical/canonical root cross-check on main.

## Risks / Open Questions

- The proposed lexical-plus-realpath cross-product can permit symlink traversal out of an allowed local media root when the media path realpath succeeds outside that root.
- The proposed `resolved` value changes the workspace-sibling restriction from canonical path behavior to lexical path behavior, which may miss symlinked paths into `workspace-*` siblings.
- Current `main` may still reject some macOS `/tmp` and `/private/tmp` representations when one side of the media/root realpath lookup fails; that user-facing false negative still needs a safe fix.
