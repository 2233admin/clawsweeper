## Summary

Keep this PR open. Current main still does not implement MCP resources for either the channel MCP server or the plugin-tools MCP server, so it cannot be closed as implemented. The PR contains relevant unmerged product work, but it needs a rebase onto the current shared MCP stdio server structure and a tighter security pass before it is mergeable.

## Best Possible Solution

Keep the PR open and ask for a current-main rebase. The best implementation path is to add resource support through the current MCP server abstractions, preferably by extending `createToolsMcpServer` only as needed, wire channel resources with the SDK 1.29 resource API, and replace bespoke config masking with `redactConfigObject`/schema-backed redaction or a narrow safe projection. Add regression tests for `resources/list`, `resources/read`, unknown URI errors, channel resource behavior, and redaction of headers, env, URL userinfo/query secrets, `pass`/`auth_token`, and arbitrary catchall fields such as `apiKey` or `clientSecret`. Update `docs/cli/mcp.md` once the shipped resource contract is final.

## Risks / Open Questions

- The PR is stale against current main: plugin-tools MCP serving now delegates through `src/mcp/tools-stdio-server.ts`, so adding resource handlers directly in `plugin-tools-serve.ts` would duplicate or bypass the shared server path unless rebased thoughtfully.
- The proposed `openclaw://config/mcp` resource is security-sensitive. It should use the existing config redaction path or a strict safe projection, not bespoke field-by-field masking, because MCP server entries allow arbitrary extra keys.
- The PR's local URL query redaction list diverges from `src/shared/net/redact-sensitive-url.ts` and misses sensitive names like `pass`, `passwd`, and `auth_token`.
- Exposing plugin and MCP config over MCP expands the trust boundary; docs/tests should make clear what is exposed, what is omitted, and how secrets are redacted.
