## Summary

Keep this PR open. Current main still does not provide MCP resource reads for the channel or plugin-tools servers, but the branch is stale against current main and has blocking security/handler regressions that need maintainer review before merge.

## What This Changes

The PR adds MCP resource registration for channel status/conversations and plugin tool/config inspection, wires resource handlers into the plugin-tools server, and adds plugin-tools resource tests.

## Best Possible Solution

Rebase the resource feature onto the current MCP helper structure, keep tool execution behind the existing handler boundary, serialize config resources through the shared config redaction path, and add regression coverage for arbitrary secret-like MCP server fields plus sensitive URL query keys before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use shared redaction for MCP config resources:** `src/mcp/plugin-resources.ts:105-143`
  - body: `openclaw://config/mcp` clones each server config and masks only `headers`, `env`, and selected URL pieces before returning the rest of the object. MCP server entries allow arbitrary keys, so fields such as `apiKey`, `authToken`, or `clientSecret` would be exposed to any MCP client reading this resource; route this payload through the existing config redaction helper instead.
  - confidence: 0.92
- **[P2] Preserve the current plugin-tool handler seam:** `src/mcp/plugin-tools-serve.ts:77-86`
  - body: The branch's plugin-tools server still builds its own tool map and calls `tool.execute` directly. Current main routes this server through `createPluginToolsMcpHandlers`, which filters `ownerOnly` tools and enforces `before_tool_call` approvals, so the resource rebase needs to keep that helper instead of restoring the older direct execution path.
  - confidence: 0.86
