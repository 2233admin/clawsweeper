Keep open. Current main still validates `edit.edits` as direct `{ oldText, newText }` objects before edit recovery runs, while the reported DeepSeek/OpenAI-compatible path can still plausibly deliver a malformed entry shape that current coverage does not exercise.

## What This Changes

The issue reports that the `edit` tool rejects both stringified and native-array `edits` payloads on the DeepSeek/OpenAI-compatible route, making edits unusable from that environment.

## Best Possible Solution

Add a focused regression that exercises the DeepSeek/OpenAI-compatible tool-call path with a native `edits` array, capture the raw parameters received by OpenClaw, then fix only the boundary that creates or receives the malformed wrapper while keeping `{ oldText, newText }` as the public edit contract.
