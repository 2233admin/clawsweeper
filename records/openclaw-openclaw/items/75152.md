## What This Changes

The issue reports that the `edit` tool rejects both stringified and native-array `edits` payloads when routed through a DeepSeek/OpenAI-compatible environment, making edit unusable from that setup.

## Best Possible Solution

Add a focused regression that exercises the DeepSeek/OpenAI-compatible tool-call path with a native `edits` array, capture the raw parameters received by OpenClaw, then fix only the confirmed boundary while preserving `{ oldText, newText }` as the public edit contract.
