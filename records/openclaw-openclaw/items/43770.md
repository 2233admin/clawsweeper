## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main still rejects arbitrary top-level plugin-entry keys, while the supported and tested runtime path is `plugins.entries.<id>.config`; broad passthrough would accept misplaced settings that plugin validation, normalization, and runtime delivery do not consume.

## Best Possible Solution

Request changes and keep this PR open for maintainer review. The best path is to preserve the explicit plugin-entry shape, keep plugin-owned settings under `plugins.entries.<id>.config`, and add or adjust regression coverage around base validation, manifest `configSchema` validation/defaulting, normalization, and `api.pluginConfig` delivery. If maintainers want legacy top-level compatibility, implement an explicit migration or normalization path with warnings and tests instead of accepting arbitrary passthrough keys.

## Risks / Open Questions

- Merging as-is could turn actionable `Unrecognized key` errors into accepted but ignored plugin settings, because normalization and runtime delivery still consume only `entry.config`.
- Broad top-level passthrough expands the core config surface outside plugin-owned `configSchema` validation, which can hide typos or misplaced credentials instead of producing clear diagnostics.
- There is a separate possible contract mismatch around documented `plugins.entries.*.apiKey` and `env` metadata versus the strict current schema; that should be handled explicitly if maintainers want those fields supported, not by raw passthrough.
