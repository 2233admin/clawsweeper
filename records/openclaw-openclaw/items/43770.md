## Summary

Keep open. The PR is not implemented on current main and the proposed `.passthrough()` change would make misplaced top-level plugin settings validate without being consumed by plugin schema validation, normalization, or runtime `api.pluginConfig` delivery.

## What This Changes

The PR changes `PluginEntrySchema` from `.strict()` to `.passthrough()` and adds a regression test asserting top-level plugin-entry keys such as `mode`, `userId`, `autoRecall`, and `oss` are accepted and preserved.

## Best Possible Solution

Keep the plugin-entry envelope explicit and route plugin-owned settings through `plugins.entries.<id>.config`, where manifest validation, defaults, runtime delivery, docs, and secret-handling semantics align. If maintainers want compatibility for legacy top-level keys, add a deliberate migration or doctor/fix path with warnings and tests instead of raw passthrough.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Keep plugin-owned keys under config:** `src/config/zod-schema.ts:154`
  - body: Switching to `.passthrough()` makes keys like `plugins.entries.openclaw-mem0.mode` validate, but current validation and runtime still consume only `entry.config`: plugin-aware validation validates `entry?.config ?? {}`, normalization preserves only the known envelope plus `config`, and the loader passes only the validated nested config as `api.pluginConfig`. This turns a loud schema error into settings that validate but are not applied.
  - confidence: 0.91
