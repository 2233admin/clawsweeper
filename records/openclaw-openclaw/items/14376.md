## Summary

Keep open. Current main has partial cron error observability and generic retry/backoff, but recurring cron still lacks the requested reason-aware guard layer: no billing/auth/format circuit breaker, no Retry-After-aware recurring scheduling, no job-level cooldown/disabled guard fields, and no UI/client surface for those guard states.

## Best Possible Solution

Keep this issue open until recurring cron has an explicit reason-aware guard layer. The implementation should classify terminal run errors once, persist and expose guard state, respect Retry-After where available, auto-pause or circuit-break recurring billing/auth/format failures with safe configurable defaults and one alert, keep timeout/network/rate-limit jobs enabled with bounded backoff, and coordinate with provider/auth-profile cooldowns across timer policy, protocol schema, UI/client surfaces, docs, and regression tests.
