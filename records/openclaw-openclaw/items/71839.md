## Summary

Keep this PR open. Current main still passes an implicit Telegram draft reply target and still tells assistants to prefer `[[reply_to_current]]`, so the PR is not obsolete; however, the patch should preserve explicit native reply tags when a streamed preview is finalized, and the unrelated QA-lab mock-provider change still needs maintainer scope review.

## What This Changes

The PR removes the Telegram draft stream `replyToMessageId`, softens assistant reply-tag guidance, updates related tests, and broadens QA-lab mock OpenAI thread-memory matching to instructions context.

## Best Possible Solution

Land a narrow Telegram change that stops implicit draft-preview quoting and updates assistant guidance, while preserving explicit native reply tags by routing explicit `replyTo*` finals through a send path instead of an edit-only preview finalization path. Keep the QA-lab mock-provider adjustment only if maintainers agree it belongs with this PR, otherwise split it.

## Reproduction Assessment

Yes. Current main can be checked directly: a Telegram message with `message_id` and no text creates a draft stream with `replyToMessageId`, and the PR removes that option; the review finding follows from the existing lane-delivery path that edits active text previews without calling `deliverReplies`.

## Solution Assessment

No, not as-is. Removing routine draft quoting and softening the prompt are the right direction, but the patch needs one more guard so explicit `[[reply_to_current]]` or `[[reply_to:<id>]]` replies still produce native Telegram replies when a preview was already streamed.
