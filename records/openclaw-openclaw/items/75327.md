## Summary

This PR should stay open for maintainer judgment because it changes the current group/channel visible-reply contract rather than only fixing a missing implementation detail. Current main intentionally keeps normal group/channel final replies private by default and makes visible output go through the message tool, with a newer merged fix covering the silent-delivery failure when that tool is unavailable.

## What This Changes

The PR adds `WasMentioned` to source reply delivery-mode resolution so mentioned group/channel turns override `message_tool_only` to `automatic`, with resolver and visibility-policy tests.

## Best Possible Solution

Keep the current tool-only group/channel contract unless maintainers explicitly decide @mentions should be a visible-final-reply exception. If that exception is accepted, implement it with clear precedence for explicit config/requested modes, update Discord/Slack/channel coverage, and change docs plus changelog together.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: current main has tests and source paths showing a mentioned Discord guild message still pre-resolves `sourceReplyDeliveryMode` as `message_tool_only`; the user-visible result depends on the model using `message(action=send)` under the documented contract.

## Solution Assessment

Is this the best way to solve the issue? No: the PR’s resolver override changes the documented group/channel privacy contract and can override explicit tool-only delivery requests; the safer current-main direction is to keep the message tool available and fall back only when tool delivery is unavailable.
