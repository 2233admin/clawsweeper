## Summary

Keep open. Current main still lacks a config-backed exec path denylist, and the linked implementation PR #75101 is open and unmerged, so this issue should stay open for maintainer security/product review rather than being closed early.

## What This Changes

Request a `tools.exec.denyPathPatterns` config-level denylist, with possible deny actions, to block exec commands whose arguments target sensitive file paths before execution.

## Best Possible Solution

Review and land PR #75101 or an equivalent security-reviewed implementation that adds the config schema/types/help/docs and enforces a shared pre-exec deny-path gate before allowlist or approval-backed execution in gateway and node host paths, with focused tests for literal argv paths, tilde/home handling, cwd resolution, shell payload scanning, and documented v1 bypasses.

## Reproduction Assessment

Yes for the implementation gap: current-main source and docs have no `denyPathPatterns` schema, docs, or runtime gate, so the requested policy cannot be configured. The private token-leak incident itself cannot be reproduced from public artifacts.

## Solution Assessment

Yes on the broad direction: a shared exec-policy gate is the maintainable place to enforce this defense-in-depth control. The exact config shape, especially action modes beyond hard block, should be settled in security/product review before merge.
