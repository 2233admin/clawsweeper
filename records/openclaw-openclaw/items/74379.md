## Summary

Keep open. Current main still has no `tools.exec.denyPathPatterns` or `denyPathAction` config surface, and the gateway/node exec paths enforce approvals, allowlists, safe bins, and strict inline-eval checks without a path-pattern deny gate. The request is security-sensitive and should get maintainer/security design review before implementation.

## What This Changes

Add a config-level exec path denylist and action mode that blocks or gates exec commands whose literal path-like arguments match configured sensitive glob patterns.

## Best Possible Solution

Define a security-reviewed `tools.exec` path policy, then enforce it before direct or approval-backed gateway and node exec runs. The change should update config types/schema/help/generated docs, document v1 literal-argv limitations, and add focused tests for path-like argv matching, tilde/home normalization, safeBins and allowlist interaction, and node/gateway parity.
