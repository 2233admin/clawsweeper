## Summary

Keep this open. Current main still routes browser Talk through realtime voice provider config, not the reported OpenAI plugin `stt` block, so the reported unconfigured-provider outcome remains consistent with the code and docs rather than an implemented or stale report.

## What This Changes

The issue asks for Control UI Talk mode to use an OpenAI-compatible local Whisper STT endpoint configured under `plugins.entries.openai.config.stt` instead of reporting the OpenAI realtime voice provider as unconfigured.

## Best Possible Solution

Define the browser Talk realtime contract explicitly. If local Whisper Talk is in scope, add or document a dedicated realtime voice provider or gateway relay path; otherwise improve diagnostics and docs so batch `tools.media.audio` or plugin `stt` configuration is not mistaken for Talk realtime setup.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure only `plugins.entries.openai.config.stt`, call `talk.realtime.session`, and current main builds no matching realtime provider config before the resolver reaches the unconfigured OpenAI realtime provider branch.

## Solution Assessment

No. Treating an OpenAI-compatible batch transcription endpoint as an implicit duplex Talk provider would blur two different contracts; a dedicated realtime provider/relay path or a clearer unsupported-config diagnostic is the safer maintainable direction.
