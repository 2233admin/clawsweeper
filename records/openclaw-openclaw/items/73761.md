## Summary

Keep this open. Current main still does not route `plugins.entries.openai.config.stt` or `tools.media.audio` into browser Talk realtime provider resolution, so the reported `Realtime voice provider "openai" is not configured` path remains plausible and source-backed. The remaining decision is product/API scope: either make this unsupported batch-STT configuration fail with a clearer diagnostic, or add/document a dedicated realtime voice provider/plugin path for local Whisper Talk.

## What This Changes

The issue asks for Control UI Talk mode to use an OpenAI-compatible local Whisper STT endpoint configured under `plugins.entries.openai.config.stt` instead of reporting the OpenAI realtime voice provider as unconfigured.

## Best Possible Solution

Define the browser Talk realtime contract explicitly. At minimum, surface a clear diagnostic that `plugins.entries.openai.config.stt` and `tools.media.audio` configure batch audio transcription, not Control UI Talk realtime; if local Whisper Talk is desired, implement or document it through a dedicated realtime voice provider plugin path rather than overloading the OpenAI batch STT config.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure only `plugins.entries.openai.config.stt`, start Control UI Talk, and current main builds no matching realtime provider config before reaching the unconfigured OpenAI realtime provider branch.

## Solution Assessment

Unclear. Treating an OpenAI-compatible batch transcription endpoint as an implicit duplex Talk realtime provider is not clearly the narrowest maintainable fix; explicit diagnostics plus a real realtime voice provider/plugin path is safer unless maintainers choose a compatibility bridge.
