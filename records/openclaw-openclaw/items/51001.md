## Summary

Keep this PR open. Current main still does not implement the PR’s central user-visible behavior: configurable fixed-delay retries for recurring cron jobs before the next natural scheduled run. The existing recurring error path only applies a backoff floor and chooses the later of the natural schedule and the backoff, so long-cadence jobs can still wait until the next natural run. The PR remains an implementation candidate for the linked open feature request, but it needs maintainer review on the unresolved semantics and the gateway/protocol side effects before landing.

## Best Possible Solution

Keep the PR open as a candidate implementation, or supersede it with a focused maintainer-owned version that adds per-job recurring retry policy end to end: cron types/state, create/update normalization, gateway schema and generated clients, CLI `cron add/edit` flags, docs, and regression tests. Before landing, settle the remaining semantics for one-shot jobs, clearing/recomputing retry state on edits, half-configured retry policies, and preserving cadence after retry success or exhaustion.
