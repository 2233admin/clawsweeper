## Summary

Current main has implemented substantial parts of the request: `openclaw config validate` exists with tests/docs, config writes validate before commit, and existing `${ENV}` placeholders are restored during write-back. The central remaining behavior is still present: bare `openclaw config` is intentionally implemented and documented as the configure wizard entrypoint, not help on already configured systems. The value-mode secret-safe UX also still needs maintainer product judgment for new secret writes.

## Best Possible Solution

Keep this issue open and narrow it to the remaining product decision: whether bare `openclaw config` should show help when the config is already valid/complete while preserving `openclaw configure` and `openclaw config --section ...` as wizard paths. If maintainers choose to change behavior, update `src/cli/config-cli.ts`, the configure/config docs, and focused CLI tests. Separately decide whether value-mode secret writes need warning/guidance or stricter SecretRef-first UX for known secret paths.
