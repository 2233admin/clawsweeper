## Summary

Keep this PR open, but it still needs changes before merge. Current main has the reset/queue bug the PR targets, and the proposed direction is right, but the patch only aborts reset-triggered active work when the session command lane has depth; active reply operations can exist before or outside that lane, so a reset can still wait instead of interrupting.

## What This Changes

The PR adds reset-triggered active-run queue-policy handling, suppresses steer/followup handling for reset turns, extends the lane clear/abort path for resets, passes reset context into the agent runner, and adds queue-policy regression tests.

## Best Possible Solution

Update this PR or land a narrow equivalent so reset-triggered turns use interrupt semantics for all same-session active reply work, independent of whether the command lane currently has queued or active depth, while preserving heartbeat drops and normal non-reset queue modes. Add integration coverage around reset tails during an active run and keep the public queue semantics documented.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Abort reset runs even without lane depth:** `src/auto-reply/reply/get-reply-run.ts:820-822`
  - body: The reset abort path is still gated on `laneSize > 0`. Same-session active work can be registered in the reply-run registry before the embedded command lane has depth, so `resetTriggered` will make the policy return `run-now` but `resolvePreparedReplyQueueState` still receives the original non-interrupt queue mode and only waits. That leaves reset turns delayed instead of aborting the active reply operation. Treat reset-triggered runs as interrupting in the wait/abort path, not only when the command lane reports depth.
  - confidence: 0.78
