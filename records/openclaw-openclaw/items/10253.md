## Summary

Keep this issue open. Current main still has no hook/webhook-specific lifecycle policy for on-complete cleanup, delayed cleanup, archive/delete behavior, transcript retention, or completed-hook visibility. Hook agent requests still run through isolated cron-style execution, but non-cron hook session keys persist as ordinary agent session rows and are not covered by the cron run-session reaper.

## Best Possible Solution

Keep this open. The best path is a core hooks.sessionCleanup policy with a non-destructive default, config types/schema/help/docs, runtime finalization for direct /hooks/agent, mapped hooks, and the Gmail preset, plus focused tests for agent:<id>:hook:gmail:* persistence, cleanup/archive behavior, transcript retention, and sessions list visibility. Maintainers should decide the exact destructive-mode defaults before implementation.

## Risks / Open Questions

- A cleanup/delete feature touches operator audit data and transcript retention, so defaults and destructive modes need explicit maintainer agreement.
- The policy must define precedence with session.maintenance, cron.sessionRetention, and subagent archiveAfterMinutes to avoid surprising cleanup of unrelated sessions.
- Delayed cleanup needs a durable or clearly best-effort design; an in-memory timer alone could be lost on gateway restart.
- Current session rows do not expose a hook kind, so hiding completed hook sessions may require session metadata/protocol/UI decisions rather than only deleting rows.
