## Summary

Keep open. Current main still handles Feishu `card.action.trigger` through a handler that falls through without returning a callback payload, while the action path is typed as `Promise<void>` and awaits command dispatch; the shipped #71670 fix only addressed Schema 2.0 parsing, not instant card updates.

## What This Changes

Add Feishu card-action instant callback responses so buttons can immediately return a disabled or transitional card/toast while synthetic command processing continues asynchronously, optionally controlled by an `ocf1` disable-state field.

## Best Possible Solution

Implement a Feishu-plugin-owned, typed instant callback response path for eligible card actions: keep validation and replay-token dedupe intact, return a valid transitional card/toast immediately, dispatch the synthetic command asynchronously, and document any new backward-compatible `ocf1` disable-state contract.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A static reproduction on current main is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without returning any callback payload while `handleFeishuCardAction` awaits the synthetic message path.

## Solution Assessment

Is this the best way to solve the issue? Yes, if the fix stays inside the Feishu plugin and uses the SDK callback-response path rather than adding core behavior. The safer shape is an opt-in typed transitional response that preserves malformed-payload rejection, token dedupe, and command authorization boundaries.
