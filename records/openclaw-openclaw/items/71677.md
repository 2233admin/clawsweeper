## Summary

Keep open. Current main still accepts and routes Feishu card-action callbacks without returning an immediate callback response payload, so the requested disabled/loading card UX is not implemented; the related Schema 2 parsing work does not supersede this instant-update request.

## What This Changes

Request Feishu interactive card button callbacks to immediately return a transitional disabled/loading card or toast response while command handling continues asynchronously.

## Best Possible Solution

Implement this inside the Feishu plugin: for eligible structured quick and approval-confirm actions, validate and claim the callback token, launch synthetic command handling in the background with token completion in its `finally` path, and immediately return a typed disabled/loading card or toast response through the existing SDK callback path. If a disable-state field is added to `ocf1`, document and test it as a backward-compatible Feishu card interaction extension.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event and observe that it resolves without returning a callback response while `handleFeishuCardAction` still awaits the synthetic command path.

## Solution Assessment

Yes, the issue's requested handler-return approach is the narrow maintainable direction because the SDK already uses handler return values for card callback updates. The fix should stay Feishu-plugin-owned and preserve the current validation, stale/wrong-user checks, duplicate-token dedupe, and command authorization flow.
