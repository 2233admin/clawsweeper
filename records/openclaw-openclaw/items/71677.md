## Summary

Keep open. Current main still has no Feishu instant card callback response path: the registered `card.action.trigger` handler falls through without returning a payload, and the action implementation is still a `Promise<void>` path that awaits synthetic command dispatch for quick actions. The related Schema 2.0 parsing fix does not cover this UX feature.

## What This Changes

Support Feishu interactive-card callbacks returning an immediate transitional disabled-card or toast response so button clicks give feedback before the agent finishes processing.

## Best Possible Solution

Implement this inside the bundled Feishu plugin as a typed, backward-compatible instant callback response path: validate and claim the callback token first, return a valid transitional disabled-card/toast response for eligible structured buttons, dispatch the synthetic command asynchronously, and document any new `ocf1` disable-state contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to invoke the registered `card.action.trigger` handler with a valid structured quick-action event on current main and observe that it resolves with no callback payload while the quick-action branch awaits `dispatchSyntheticCommand`.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix if it stays plugin-owned and preserves Feishu validation, token dedupe, and authorization checks. A core API change is not justified unless implementation proves a missing generic plugin seam.
