## Summary

Keep open. Current main still accepts Feishu card actions, validates/dedupes them, and then completes the handler without returning an instant callback payload; the requested disabled/loading card response is not implemented and remains a narrow Feishu plugin enhancement.

## What This Changes

Add Feishu interactive-card callback support that immediately returns a transitional disabled/loading card response while the synthetic command continues asynchronously.

## Best Possible Solution

Implement this inside the Feishu plugin: claim and validate the callback token first, return a typed transitional callback card/toast for eligible quick/confirm actions, dispatch the synthetic command asynchronously, and add a backward-compatible ocf1 disable-state field only if the renderer and parser both need it.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A static reproduction is to invoke the registered card.action.trigger handler with a valid structured quick-action event and observe that it resolves without a callback response while handleFeishuCardAction awaits the synthetic message path.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable fix is Feishu-plugin-owned and can use the existing SDK callback-response return path without adding a core API, provided it preserves malformed-payload rejection, replay-token dedupe, and command authorization boundaries.
