## Summary

Keep this PR open. Current main still does not include the proposed HarmonyOS app, HarmonyOS gateway client ID, platform docs, labeler coverage, or mobile-node policy. The PR remains the live implementation candidate after related issue #43991 was closed, but it is an XL first-party platform change with security-sensitive auth/storage/build surfaces and unresolved review findings that need maintainer review before merge.

## What This Changes

The PR adds a new HarmonyOS ArkTS app under apps/HarmonyOS with DevEco/Hvigor project files, secure preferences and device auth, gateway session/auth handling, onboarding/connect/chat/settings UI, screenshots/docs, and a new openclaw-harmonyos gateway client ID validation case.

## Best Possible Solution

Keep this PR open for explicit maintainer product and security review. If OpenClaw wants first-party HarmonyOS support, split or harden the work: accept a HarmonyOS platform/client policy, add the gateway client ID and mobile-node/talk policy with tests, align setup-code/bootstrapToken auth with iOS/Android, replace shell-based hvigor execution with deterministic tooling, remove developer-local docs/config drift, add platform docs and labeler coverage, and require DevEco plus real-device validation. If first-party support is not desired, maintainers should say so and incubate the HarmonyOS app outside core until the platform contract is agreed.
