## Summary

Keep open. Current main still does not implement the HarmonyOS app, HarmonyOS gateway client ID, platform docs, or mobile-node policy this PR proposes. This PR remains a possible implementation candidate, but it is an XL native-app change with gateway auth, credential storage, build tooling, and unresolved functional/security-sensitive review findings, so it should not be closed by conservative cleanup.

## Best Possible Solution

Keep this PR open as a HarmonyOS implementation candidate. Maintainers should decide product fit first. If accepted for core, the work should be split or hardened: add any gateway client ID/mobile-node policy with tests and docs, align setup-code/bootstrapToken auth with iOS/Android, fix the secure storage and connection-state review findings, and replace the unsafe hvigor wrapper/dependency flow before merge. If first-party support is not desired yet, incubate the HarmonyOS app outside the core repo until the platform contract is agreed.

## Risks / Open Questions

- The PR is an XL native-app contribution touching gateway auth, token/device identity handling, secure storage, platform permissions, dependency/build tooling, and docs; it needs explicit maintainer product and security review.
- The proposed build wrapper installs dependencies and executes a shell command built from forwarded CLI arguments, which is not acceptable without safer invocation and deterministic dependency handling.
- The HarmonyOS setup-code/auth path appears out of sync with current bootstrapToken pairing semantics and could fail normal authenticated onboarding.
- A core platform policy decision remains: whether HarmonyOS should become a first-party companion app, what gateway client ID/platform behavior it gets, and whether talk/mobile-node capability should include it.
