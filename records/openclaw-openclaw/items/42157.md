## Summary

Current main still has the Telegram webhook restart-loss window described here. The webhook path validates the request and body, sends 200 OK to Telegram, then runs `bot.handleUpdate` in a detached in-memory async task with no durable inbound queue, dequeue, or startup replay. The related contributor PRs #40056, #42185, and #42523 are useful prior art but are closed unmerged, so there is no canonical merged fix to close against.

## Best Possible Solution

Keep this issue open. The best implementation belongs in `extensions/telegram`: add a durable inbound webhook queue that stores validated raw updates before returning 200, processes them through the existing Telegram bot pipeline, removes entries only after full update processing including deferred work completes, and replays pending entries on startup before accepting new webhook traffic. The closed contributor PRs #40056, #42185, and #42523 should be treated as useful prior art for a maintainer-reviewed fix.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after returning 200 and before or during in-memory `bot.handleUpdate` completion.
- A future fix needs careful at-least-once semantics: validate guards and Telegram secret before persistence, durably enqueue before ACK, dequeue only after all synchronous and deferred Telegram work settles, replay pending entries on startup, and handle duplicate updates safely.
