## Summary

Keep open. Current main still acknowledges Telegram webhook deliveries before processing completes, with no durable enqueue/replay path for inbound webhook payloads. The linked contributor PRs #40056, #42185, and #42523 are useful prior art, but all are closed unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/telegram`: add a durable inbound webhook queue that stores validated raw Telegram updates before returning 200, processes them through the existing bot pipeline, removes entries only after processing and deferred work settle, and replays pending entries on startup. The closed contributor PRs #40056, #42185, and #42523 should be treated as useful prior art for a maintainer-reviewed implementation with regression tests and docs updates.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after returning 200 and before or during `bot.handleUpdate` completion.
- A future durable queue needs careful at-least-once semantics: persist only authenticated validated payloads, replay on startup before accepting new webhook traffic, delete only after all Telegram deferred work settles, and tolerate duplicate updates.
