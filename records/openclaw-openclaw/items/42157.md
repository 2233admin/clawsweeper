## Summary

Keep open. Current main still acknowledges Telegram webhook deliveries before processing them and does not persist or replay inbound webhook payloads across gateway restarts. The related contributor PRs #40056, #42185, and #42523 are useful prior art but are closed unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/telegram`: add a durable inbound webhook queue that stores validated raw updates before returning 200, processes them through the existing bot pipeline, removes entries only after full processing including deferred work, and replays pending entries on startup before accepting new webhook traffic. The closed contributor PRs #40056, #42185, and #42523 should be treated as useful prior art for a maintainer-reviewed implementation with regression tests and docs updates.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after returning 200 and before or during in-memory `bot.handleUpdate` completion.
- A future fix needs careful at-least-once semantics: validate guards and Telegram secret before persistence, durably enqueue before ACK, dequeue only after synchronous and deferred Telegram work settles, replay pending entries on startup, and tolerate duplicate updates.
