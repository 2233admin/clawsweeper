## Summary

Keep this issue open. Current main still represents explicit user aborts and restart-interrupted work with the same `abortedLastRun` marker, while startup recovery still treats a running main session with that marker as permission to inject a synthetic resume turn. The prior ClawSweeper comment remains accurate on the current checkout.

## What This Changes

The issue reports that explicit stop/no-action user instructions should terminate work and prevent later internal continuation messages.

## Best Possible Solution

Persist explicit user aborts separately from restart-interruption recovery. `/stop`, `chat.abort`, and recognized no-action abort phrases should mark the session as terminal user-stopped or recovery-suppressed, while stale-lock recovery should use a restart-specific marker that is the only marker eligible for synthetic resume.
