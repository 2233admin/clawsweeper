## Summary

Keep open. The PR does not fix the claimed Windows/Docker crash path because its new helper is not called, while current main still routes secrets/config auth writes through existing synchronous writers with unguarded chmod behavior.

## What This Changes

The PR adds `src/shared/fs-atomic-fix.ts` with a synchronous temp-file writer that catches `chmodSync` failures before renaming the temp file into place.

## Best Possible Solution

Revise this PR or replace it with a focused change on the active secrets persistence path: preserve parent-directory creation, handle the expected Windows-mounted filesystem chmod/replace failure where callers actually write auth/config files, keep unexpected credential-permission failures visible, and add targeted regression coverage.

## Reproduction Assessment

Yes for the PR defect: current-main search shows the helper has no callers, while `runSecretsApply` still calls `writeTextFileAtomic`. A targeted test can mock `fs.chmodSync` throwing EPERM on the active writer path and show this PR does not affect that path.

## Solution Assessment

No. Adding a disconnected helper is not the narrow maintainable fix; the change needs to update the existing writer or a shared helper actually used by it, while preserving directory creation and narrowing the chmod fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Wire the helper into active writers:** `src/shared/fs-atomic-fix.ts:7`
  - body: This adds `writeTextFileAtomicSync`, but nothing imports or calls it. Current secrets apply still uses `writeTextFileAtomic` in `src/secrets/shared.ts`, whose `chmodSync` remains unguarded, so merging this leaves the reported Windows/Docker crash unchanged.
  - confidence: 0.98
- **[P2] Preserve parent directory creation:** `src/shared/fs-atomic-fix.ts:8-9`
  - body: The new writer builds and writes the temp file before ensuring the destination directory exists. The existing writer calls `ensureDirForFile(pathname)`, so using this helper for new auth/env files would regress to `ENOENT` when the parent directory has not been created yet.
  - confidence: 0.94
- **[P2] Narrow the chmod fallback to EPERM:** `src/shared/fs-atomic-fix.ts:14-15`
  - body: The catch suppresses every `chmodSync` failure. If this is used for auth/config files, non-Windows or non-EPERM permission hardening failures would be hidden while the write still proceeds.
  - confidence: 0.84
