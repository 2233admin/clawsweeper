## Summary

Keep open. The report is recent, concrete, and still plausibly current: it includes Windows/Node 24 Gateway CLOSE_WAIT/FIN_WAIT_2 evidence, a follow-up comment saying CLOSE_WAIT still appears on a newer version, and a newer open Windows Gateway crash report cross-references it. Current main has related WebSocket shutdown, pre-auth, and client-stop hardening, but the checked source and tests prove narrower paths, not the reported long-running live Gateway socket accumulation or Windows crash pattern.

## Best Possible Solution

Keep this issue open until maintainers reproduce or disprove the leak on current main/latest release on Windows with Node 24. The best fix path is to capture Gateway logs plus netstat/socket-state snapshots during a long-running WebSocket workload, determine whether the leak is in the core Gateway WS lifecycle, a client reconnect path, or a channel/plugin transport such as Mattermost, then add focused regression coverage for the implicated close/terminate path. If #71699 is confirmed to share the same root cause, make that the canonical bug only after the relationship is proven.
