## Summary

Keep open. Current main still has the exact gap this PR targets: updateLastRoute enters the session-store lock but still calls loadSessionStore without skipCache, while the adjacent locked session-store writers already bypass the cache. The PR is a small, targeted implementation candidate, and the Greptile review/comment matches the live code path I inspected.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a current-main version of the updateLastRoute cache-bypass fix, preferably using the same locked-writer style as the adjacent functions and adding a focused regression test that primes the session-store cache, simulates an external preserved-mtime write, then verifies updateLastRoute does not clobber that data.
