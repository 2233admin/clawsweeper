## Summary

Keep open. Current main still writes a local-offset top-level `time` field while serializing the original log object unchanged, so a nested `_meta.date` Date continues to become a UTC `Z` string in raw JSONL logs. The directly related fix PR is closed, unmerged, and has zero changed files, so this remains a narrow logging repair candidate.

## What This Changes

The issue reports that raw log JSON contains `_meta.date` in UTC while the top-level `time` field uses the local-offset log formatter, causing inconsistent timestamp displays during log inspection.

## Best Possible Solution

Normalize `_meta.date` only in the copied JSONL file-log payload so it equals the top-level `time` string for the same log timestamp, keep existing host-local transport semantics, avoid mutating the original tslog object, and add regression coverage for both fields.
