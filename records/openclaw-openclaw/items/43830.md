## Summary

Keep open. Current main has several redaction mechanisms, but they are scoped to logs, persisted transcripts, diagnostic/support exports, and selected tool/result delivery surfaces; it does not contain the requested built-in, configurable automatic sanitization layer for final agent output. The directly related open PRs (#45783 and #72733) are implementation candidates that need maintainer security/product review rather than premature closure.

## What This Changes

Request to add configurable automatic sanitization for sensitive data in agent responses, tool outputs, and related delivery paths.

## Best Possible Solution

Maintainers should choose one security design for automatic sensitive-data handling, then land a focused implementation with explicit config semantics, coverage for final replies, block streaming, tool-result summaries, persisted transcripts, and provider-bound traffic, plus docs that distinguish core guarantees from plugin-extensible policy.

## Reproduction Assessment

Not applicable as a feature request, but the implementation gap is high-confidence: current main has no `outputSanitization` or `privacy` config surface and no built-in final-response sanitizer in the delivery path.

## Solution Assessment

Unclear. The requested direction is valid, but the best fix needs maintainer security/product judgment because the open proposals choose different boundaries and defaults; a narrow, reviewed core guardrail or explicitly supported plugin path would be safer than merging competing broad designs.
