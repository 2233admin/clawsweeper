## What This Changes

The issue requests adding MiniMax manifest `providerAuthAliases` so `minimax-cn` reuses `minimax` auth and `minimax-portal-cn` reuses `minimax-portal` auth.

## Best Possible Solution

Review and land #72798, or apply an equivalent maintainer patch that adds the two MiniMax manifest aliases and focused manifest regression coverage. The core resolver should remain manifest-backed so the plugin trust filtering introduced for auth aliases stays intact.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `minimax-cn` and `minimax-portal-cn` are runtime hook aliases, but the manifest has no `providerAuthAliases`, and auth lookup only remaps aliases from manifest metadata before checking env vars or profiles.

## Solution Assessment

Yes. The narrow manifest metadata fix is the best path because it aligns MiniMax with the documented provider contract and sibling provider manifests without weakening the core auth-alias trust model.
