## Summary

Keep open. Current main still matches the reported behavior: ACTION_ASSIST parses as a promptless assist launch, but MainViewModel sends every assistant launch to Chat, and VoiceTabScreen has only manual mic/talk permission flows with no assistant-driven one-shot auto-activation state. The request is narrow, current code has the needed destination surface, and no open PR appears to specifically close this issue.

## What This Changes

Route Android ACTION_ASSIST launches without a text prompt to the Voice tab and auto-enable or request microphone capture once, while keeping ASK_OPENCLAW text prompts routed to Chat.

## Best Possible Solution

Implement the behavior in the Android app by special-casing promptless source="assist" launches in MainViewModel, adding a one-shot pending Voice auto-activation state consumed by VoiceTabScreen, reusing the existing Voice tab permission flow for RECORD_AUDIO, and preserving current ASK_OPENCLAW text-prompt behavior as Chat/draft-first. Add focused unit or Compose coverage around routing, one-shot consumption, stale chat state cleanup, and permission-grant continuation; update the Android docs if the visible assistant entry behavior changes.

## Repair Work Prompt

Implement the Android assistant gesture routing requested by openclaw/openclaw#65414. On current main, `ACTION_ASSIST` parses as `AssistantLaunchRequest(source="assist", prompt=null, autoSend=false)` but `MainViewModel.handleAssistantLaunch` routes every assistant launch to Chat. Change only the Android app surface needed for this behavior: promptless `source == "assist"` launches should request `HomeDestination.Voice`, clear stale chat launch state, and set a one-shot pending Voice auto-activation flag; App Actions / `ASK_OPENCLAW` text prompts must continue routing to Chat as today and must not become auto-send. In `VoiceTabScreen`, consume the one-shot flag exactly once from a `LaunchedEffect`; if RECORD_AUDIO is already granted, call `viewModel.setMicEnabled(true)`, otherwise request mic permission and enable the mic only if that grant corresponds to this pending assistant launch. Do not change manifest roles, gateway config, global voice wake behavior, or unrelated assistant hardening. Add focused tests for MainViewModel routing/state cleanup and one-shot consumption or a small UI seam as appropriate; update `docs/platforms/android.md` if the user-visible assistant entry text changes. Be aware of adjacent open PR openclaw/openclaw#61298, which hardens assistant launch persistence but does not implement this Voice routing and is stale against current ASK_OPENCLAW draft behavior.
