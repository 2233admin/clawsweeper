## Summary

Keep open. Current main still routes promptless Android ACTION_ASSIST launches into Chat, while the Voice tab only supports manual mic/Talk activation and has no one-shot assistant-launch auto-activation state. The issue is recent, concrete, unprotected, and no open PR match was found, so it is a narrow repair candidate rather than cleanup to close.

## What This Changes

The issue requests routing promptless Android ACTION_ASSIST launches to the Voice tab with one-shot mic auto-activation while preserving prompt-based ASK_OPENCLAW/App Actions routing to Chat.

## Best Possible Solution

Implement the Android-side routing change where the assistant request is handled: promptless source="assist" launches should request HomeDestination.Voice and set a one-shot pending mic activation flag, while prompt-bearing ASK_OPENCLAW/App Actions launches continue through the Chat draft/auto-send path. VoiceTabScreen should consume that flag exactly once, request RECORD_AUDIO when needed, and enable the mic only after a grant; update Android docs/changelog if behavior changes.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: Intent.ACTION_ASSIST parses to source="assist" with no prompt, MainViewModel.handleAssistantLaunch currently selects HomeDestination.Chat, and VoiceTabScreen has no assistant-origin auto-activation path.

## Solution Assessment

Yes. The issue’s proposed direction is the narrowest maintainable solution: keep parsing and manifest/App Actions stable, add the special-case routing and one-shot state in MainViewModel/VoiceTabScreen, and avoid gateway or config schema changes.
