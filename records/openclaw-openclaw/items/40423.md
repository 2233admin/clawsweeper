## Summary

Keep this PR open. Current main still does not implement or document `OPENCLAW_AGENT_ID` or `OPENCLAW_SESSION_KEY` for the requested `exec`, `acp-client`, and `tui-local` child-process paths; those paths only expose the existing `OPENCLAW_SHELL` markers. The related #72980 is open and narrower, so it does not supersede this broader PR.

## Best Possible Solution

Keep this PR open for rebase and maintainer review. The best implementation is a shared or consistently mirrored runtime env resolver that injects stable `OPENCLAW_AGENT_ID` plus an approved optional `OPENCLAW_SESSION_KEY`, threads explicit agent/session context through exec direct/sandbox/PTY and gateway approval follow-up execution, updates the current ACP helper in `src/acp/client-helpers.ts`, passes current TUI agent/session state into `createLocalShellRunner`, and adds focused tests for explicit-agent precedence, session-key fallback, missing/legacy context, env override semantics, and session-key exposure policy.

## Risks / Open Questions

- Exposing `OPENCLAW_SESSION_KEY` through process environment is security-sensitive because env values can leak via process inspection, shell tracing, logs, crash reports, and subprocess inheritance; this needs explicit maintainer policy before landing.
- The submitted PR patch is stale against current ACP layout: current spawn-env logic lives in `src/acp/client-helpers.ts`, while the supplied diff modifies the older helper location in `src/acp/client.ts`.
- The supplied review context identified an explicit-agent propagation gap in gateway approval follow-up execution; any revived patch should cover that path with tests.
- Env override semantics need a clear contract: `OPENCLAW_SHELL` is currently forced after env merging, while related discussion for session markers mentions caller-supplied env winning on conflict.
