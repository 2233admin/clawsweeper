## Summary

Keep open and do not automerge yet. The PR adds a useful `openclaw proxy validate` surface that current `main` still lacks, and exact-head GitHub check runs are green, but the changed proxy validation engine still has a security-sensitive false-pass path: any HTTP 403 on a denied destination is treated as proof that the proxy blocked it even though the fetch result cannot distinguish a proxy denial from a reachable origin returning 403.

## What This Changes

The PR adds `openclaw proxy validate`, a proxy-validation runtime, CLI/runtime tests, proxy docs updates, and an Unreleased changelog entry for operator-managed outbound proxy preflight checks.

## Best Possible Solution

Keep the proxy validation command in the CLI/security area, but revise denied checks so success requires a transport failure or an unambiguous proxy-generated deny signal. If HTTP status denial remains supported, the result contract should carry enough response-source or proxy-denial metadata, with regression tests proving ordinary origin 403 responses fail validation.

## Reproduction Assessment

Yes. The injected `fetchCheck` seam can return `{ ok: false, status: 403 }` for a denied URL, and `runProxyValidation` reports overall success even though that result shape can also represent a reachable origin returning 403.

## Solution Assessment

No. The command and docs direction are appropriate, but accepting a bare HTTP 403 as proof of proxy denial is not the narrowest safe solution for a security preflight; the validator should fail closed unless it can verify the proxy blocked the destination.
