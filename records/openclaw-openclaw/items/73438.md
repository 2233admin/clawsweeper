## Summary

Keep open. This is an open draft PR from a MEMBER with the protected `maintainer` label, and current `main` still lacks the requested `openclaw proxy validate` command. The latest head addresses the earlier disabled-proxy and docs-alignment feedback, but it still needs maintainer/security follow-up because denied checks accept a bare HTTP 403 as proof of proxy denial even though that response could come from a reachable origin.

## What This Changes

This draft PR adds `openclaw proxy validate`, a proxy validation engine, CLI/runtime tests, Network Proxy documentation, and a changelog entry.

## Best Possible Solution

Keep the validation command in the proxy owner area, but make denied checks fail unless the block signal is unambiguous, or explicitly document and configure any accepted proxy-deny statuses with their false-positive tradeoff. Also update the `openclaw proxy` CLI reference so the public command surface and security guide stay aligned.

## Reproduction Assessment

Yes. The added unit test path with a denied check returning `{ ok: false, status: 403 }` reproduces the concern: current `runDeniedCheck` marks that as passing without proving the 403 came from the proxy.

## Solution Assessment

No. The overall command boundary is appropriate, but the current denied-response implementation is not the safest maintainable solution for a security preflight; it should require a transport failure or a proxy-specific denial signal, or make the accepted-status policy explicit and auditable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Do not accept bare 403 as a proxy denial:** `src/infra/net/proxy/proxy-validation.ts:246-252`
  - body: `runDeniedCheck` marks any HTTP 403 as a successful denied check, but that status can also come from the denied origin after the proxy allowed the request. A loopback or metadata service returning 403 would make validation pass while the destination is reachable, so require a transport failure or an unambiguous proxy-deny signal before passing this check.
  - confidence: 0.86
- **[P3] Update the proxy CLI reference:** `src/cli/proxy-cli.ts:57-68`
  - body: This adds a public `openclaw proxy validate` subcommand, but `docs/cli/proxy.md` still lists the proxy commands without `validate` or its options. Add the command there so the CLI reference matches the new surface, not only the Network Proxy guide.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds security-sensitive outbound proxy validation, and the denied-check semantics can produce a false pass for reachable destinations returning HTTP 403.

Concerns:

- **[medium] Bare 403 can mask reachable denied destinations:** `src/infra/net/proxy/proxy-validation.ts:246`
  - body: The validator treats any HTTP 403 as an explicit proxy denial, but the response source is not checked. If the proxy allows a denied target and that origin returns 403, the validation result incorrectly passes, weakening the preflight guarantee.
  - confidence: 0.86
