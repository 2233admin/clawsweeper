## Summary

Keep open. Current main has meaningful HTTP/env proxy plumbing for model-provider requests, but the issue title explicitly asks for SOCKS5/HTTP support and SOCKS5 is not implemented for the core model-provider request path. The provided related reports cover different scopes and do not provide a canonical duplicate for the remaining SOCKS5 model-provider work.

## Best Possible Solution

Keep this issue open and narrow the implementation target: add SOCKS5 support to `models.providers.*.request.proxy` for model-provider transports, likely by extending the guarded dispatcher/proxy-agent path, then update schema validation, docs, redaction/secret handling, and provider transport tests for OpenAI/Anthropic/Google-compatible streams.
