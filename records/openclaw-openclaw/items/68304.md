## Summary

Keep open. Current main still lacks the adapter-level `isSilentReplyText` guard that this PR adds to `sendMessageTelegram`; higher-level Telegram delivery paths normalize many exact `NO_REPLY` replies, but direct adapter calls can still reach the Telegram API. The PR is narrow, but it needs maintainer review because it is linked to open #38603 and the proposed guard should account for Telegram inline buttons or prove that case is unreachable.

## What This Changes

Adds an `isSilentReplyText` import and early return in `extensions/telegram/src/send.ts` so `sendMessageTelegram` returns a suppressed result for bare `NO_REPLY` text when no media URL is set.

## Best Possible Solution

Land a narrow Telegram adapter defense-in-depth fix after reviewing the inline-button case and adding direct unit coverage; keep any broader preview-buffering behavior under #38603 if it still reproduces after the adapter guard.

## Risks / Open Questions

- The proposed early return ignores `opts.buttons`; if a legitimate structured Telegram message uses `NO_REPLY` as fallback text with inline buttons and no media, this patch would suppress the buttons unless the guard is tightened or the case is proven impossible.
- Current main has higher-level Telegram delivery tests, but no direct `sendMessageTelegram` unit test for adapter-boundary `NO_REPLY` suppression.
- The adapter guard is useful defense in depth, but it does not by itself solve pre-sentinel preview text if the visible flash in #38603 comes from draft preview updates rather than the final send path.
