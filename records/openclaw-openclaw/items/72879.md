## Summary

Keep this open. Current main still has a native Google path that can serialize Gemini `functionCall` replay without a `thoughtSignature` whenever the in-memory tool-call block lacks one; the current Google replay hook only handles ordering/sanitization, while the issue has a recent native-Google Gemini reproduction and matches Google's documented validation rule for current-turn function calls.

## What This Changes

The issue reports a native Google Generative AI Gemini tool-calling regression where replayed `functionCall` parts can be sent without required `thought_signature` metadata after the earlier 2026.4.24-era fix.

## Best Possible Solution

Add a provider-scoped native Google replay repair that preserves real same-model signatures byte-for-byte, recovers or merges native signatures when they are available, and uses Google's documented dummy-signature behavior only for transferred or deterministic current-turn function-call history that cannot have a real signature. Keep the existing cross-provider real-signature stripping boundary and add focused regression coverage for signed replay, missing native signatures, transferred/fallback history, and parallel function calls.

## Reproduction Assessment

Yes. The issue includes a concrete native Google Gemini 3 thinking/tool-calling reproduction, and current source inspection shows the same unsigned `functionCall` payload shape can still be built when a replayed tool-call block lacks `thoughtSignature`.

## Solution Assessment

Yes for keeping this open and routing a focused fix PR. The narrowest maintainable solution is native Google signature repair in the provider replay/transport boundary, not a broad transcript sanitizer or cross-provider real-signature reuse.
