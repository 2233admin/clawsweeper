## Summary

Keep this open: current main still leaves the reported native Google missing-signature path uncovered. The implementation round-trips a `thoughtSignature` when it is already present on the native Google tool-call block, but it does not merge or re-inject a missing native Google signature the way the Google OpenAI-compatible path does, and the issue comments include a concrete 2026.4.27 native-Gemini reproduction.

## What This Changes

The issue reports a regression where Gemini tool-calling turns through the native Google Generative AI provider can fail with a 400 because `functionCall` replay parts are missing `thought_signature`.

## Best Possible Solution

Fix the native Google provider path so Gemini tool-call signatures are preserved or recovered for same provider/model replay without leaking provider-specific signatures across model/provider boundaries. The repair should add focused native Google transport tests for missing or chunked signatures, already-signed function calls, parallel calls, and cross-provider stripping.

Yes. The issue has a concrete 2026.4.27 CLI reproduction with the exact Google 400, and current-main inspection shows the same missing-signature path remains possible when a native Google tool-call block lacks `thoughtSignature`.

## Solution Assessment
