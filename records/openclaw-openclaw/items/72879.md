## Summary

Keep this open: current main still has a native Google `google-generative-ai` gap for the reported missing `thoughtSignature` replay case. The shipped code preserves signatures that are already present on native tool-call blocks and the OpenAI-compatible Google path has a reinjection helper, but native Google serialization still emits an unsigned `functionCall` whenever the in-memory block lacks `thoughtSignature`, matching the current 2026.4.27 reproduction in the issue discussion.

## What This Changes

The issue reports a Gemini native Google Generative AI regression where thinking-mode tool-calling turns can replay `functionCall` parts without the required `thoughtSignature` after the earlier 2026.4.24-era fix.

## Best Possible Solution

Keep the issue open and implement a narrow native Google provider fix so Gemini tool-call signatures are preserved or recovered for trustworthy same-provider/current-turn replay, while retaining the existing cross-provider stripping behavior. Cover the repair in native Google transport tests rather than treating the older sanitization or OpenAI-compatible fixes as sufficient.

## Reproduction Assessment

Yes. The issue has a concrete 2026.4.27 CLI reproduction and exact 400 error, and current-main source inspection shows the same omission path when a native Google `toolCall` block lacks `thoughtSignature`; I did not run a live request because no Google key is configured here.

## Solution Assessment

Yes. The best direction is a scoped native Google transport/replay repair with regression tests, not closing against #23457 or the already-present same-part round-trip behavior from #66949; a broader transcript-policy change would risk cross-provider signature leakage.
