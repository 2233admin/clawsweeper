## Summary

Keep this open. Current main still has the reported Mattermost streaming contract mismatch: docs advertise canonical Mattermost preview streaming modes, but the Mattermost schema/type/metadata omit `streaming` and the inbound runtime still wires draft-preview delivery unconditionally. An open PR already targets the narrow schema/runtime gap, so this should remain a tracker rather than be closed or queued as duplicate repair work.

## What This Changes

The issue asks Mattermost to honor the documented `channels.mattermost.streaming` preview-mode config and address draft-preview notification/noise UX from the current streaming behavior.

## Best Possible Solution

Keep this as the Mattermost streaming-contract tracker until PR #73252 or an equivalent Mattermost-local fix lands. The desired end state is root and per-account `streaming` schema/type/metadata support, runtime preview-mode resolution, legacy `blockStreaming` compatibility, an `off` path that bypasses draft-preview callbacks/finalization, focused schema/runtime/docs coverage, and then a separate decision on whether final notifications need an additional Mattermost UX option.

## Reproduction Assessment

Yes for the documented config mismatch: current main documents `channels.mattermost.streaming`, while the strict Mattermost schema/type/metadata omit it and the inbound monitor wires preview streaming unconditionally. The Mattermost notification behavior has reporter evidence, but no live client notification run was performed in this read-only review.

## Solution Assessment

Yes, the narrow maintainable direction is a Mattermost-local implementation of the existing channel streaming contract, with PR #73252 as the visible candidate. A separate `finalNotification` flag should wait until `streaming.mode` can actually disable or select preview behavior.
