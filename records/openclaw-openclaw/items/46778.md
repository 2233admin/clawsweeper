## Summary

Current main still dedupes Feishu inbound messages by `message_id`. Prior fixes cover same-ID Feishu replays and race windows, but this issue specifically reports identical private-message content arriving with different `message_id` values, so it should not be closed as implemented or stale.

## Best Possible Solution

Keep this issue open and add a Feishu-owned fallback dedupe key for `im.message.receive_v1` deliveries where the same sender/content/timestamp arrives with different `message_id` values. The fix should stay inside the Feishu plugin unless an existing generic dedupe helper is reused, and it should preserve the existing same-message-id dedupe path.
