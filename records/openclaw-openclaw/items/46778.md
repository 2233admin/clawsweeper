## Summary

Keep open. Current main still deduplicates Feishu inbound delivery by `message_id` at the early processing claim, debounce filtering, persistent store, and final handler, so the reported same-message/different-`message_id` duplicate path remains valid and narrow.

## What This Changes

Requests a Feishu plugin fallback dedupe path for duplicate private-chat messages where the same logical message arrives with different `message_id` values.

## Best Possible Solution

Add a Feishu-local duplicate fingerprint that complements the existing `message_id` fast path and records only tightly matched duplicate deliveries, leaving core plugin/runtime dedupe contracts unchanged.

## Reproduction Assessment

Yes. A focused unit reproduction can enqueue two Feishu `im.message.receive_v1` text events with the same account, sender, chat/thread, content, and `create_time` but different `message_id` values; current main treats both ids as fresh.

## Solution Assessment

Yes. A Feishu-owned fallback fingerprint is the narrowest maintainable fix because the observed failure is service-specific delivery behavior and the current plugin already owns inbound event parsing, debounce, and persistent dedupe calls.
