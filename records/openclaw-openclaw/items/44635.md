## Summary

Keep this issue open. Current main has named browser profiles and shipped `browser.profiles.<name>.userDataDir` for Chrome MCP `existing-session` attachment, but it still does not implement the requested agent-scoped browser configuration or per-agent default profile binding. The proposed `agents.list[].browser.userDataDir` shape remains outside the strict agent schema, managed browser storage is still derived from browser profile name, and the browser tool only uses an explicit tool-call `profile` or the global browser default.

## Best Possible Solution

Keep this open and treat it as a browser/core configuration feature. The best path is a deliberate agent-scoped browser policy contract, likely under `agents.defaults.browser` and `agents.list[].browser`, that can bind an agent to a named browser profile or browser policy. The browser plugin should receive the active agent id and derive the default profile from that context when the tool call omits `profile`; any managed custom `userDataDir` support should be schema-backed, documented, tested, and safe around profile reset/delete behavior. Coordinate with broader per-agent browser isolation/proxy work rather than closing this as covered by existing-session `userDataDir`.

## Risks / Open Questions

- Closing this now would hide a real privacy/security boundary: agents can still share the same browser profile unless callers manually choose different profiles or change the global default.
- The existing `browser.profiles.<name>.userDataDir` support may look like a fix, but it is limited to Chrome MCP existing-session attachment and does not provide managed per-agent storage isolation.
- A future implementation needs careful reset/delete semantics if custom managed `userDataDir` paths are allowed, so OpenClaw does not trash user-owned browser data unexpectedly.
