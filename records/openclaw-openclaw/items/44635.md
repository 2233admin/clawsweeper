## Summary

Keep this issue open. Current main supports named browser profiles and `browser.profiles.<name>.userDataDir` for Chrome MCP existing-session attachment, but it still does not support `agents.list[].browser.userDataDir`, per-agent browser defaults, or per-agent CDP/profile isolation. The follow-up discussion describes real isolation needs for skills and remotely hosted CDP instances, so this remains a browser/core configuration gap rather than stale cleanup.

## Best Possible Solution

Keep the issue open as a browser/core configuration feature. The best path is to design an agent-scoped browser selection contract, likely an `agents.defaults` and `agents.list[]` binding to a named browser profile or browser policy, then update the browser tool/server so omitted `profile` values resolve from active agent context. If custom managed `userDataDir` or per-profile CDP/flag policy is added, it should come with schema, docs, tests, redaction/SSRF safeguards, and clear reset/delete behavior.

## Risks / Open Questions

- Closing this now would blur a real privacy/security boundary: browser profile selection is global or manually requested, not automatically scoped to the active agent.
- The existing `browser.profiles.<name>.userDataDir` support can look like the requested feature, but it applies to Chrome MCP existing-session attachment and not to managed per-agent browser storage.
- There are related browser isolation and proxy reports, including #37487, but no clear canonical maintainer-designated replacement that fully supersedes this issue.
