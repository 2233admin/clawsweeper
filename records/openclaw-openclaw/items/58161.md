## Summary

Keep this issue open. Current main has embedded Pi bundle MCP tool execution, but it still lacks the client capability and request-handler path needed for MCP elicitations, so the reported dynamic approval use case is not implemented or superseded.

## What This Changes

Add MCP elicitation support to the embedded Pi bundle MCP runtime so plugin-backed MCP tools can request runtime user approval instead of relying on static configuration.

## Best Possible Solution

Add a maintained embedded Pi bundle MCP elicitation bridge that advertises the supported MCP capability, registers an elicitation/create handler, routes interactive prompts through OpenClaw's existing approval surfaces with deny/cancel/timeout semantics, fails closed in non-interactive runs, and documents the supported behavior for plugin authors.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is any bundle MCP server that needs elicitation/create during callTool: current main does not advertise the elicitation capability or install a handler, so that server cannot obtain a dynamic user response through embedded Pi.

## Solution Assessment

Unclear until maintainer design is settled. The requested direction is likely the right substrate, but the narrowest maintainable fix should reuse existing plugin approval plumbing rather than adding plugin-specific static policy or a separate approval system.
