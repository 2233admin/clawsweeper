## Summary

Keep open: current main exposes bundle MCP tools in embedded Pi, but the Pi MCP client path still does not advertise or handle MCP `elicitation/create` requests. Existing plugin-approval plumbing and the separate Codex elicitation bridge make this a real runtime/API gap, not a ClawHub-only plugin request.

## What This Changes

Add MCP elicitation support to the embedded Pi bundle MCP client so plugin-backed MCP tools can request dynamic runtime approval instead of relying on coarse static destructive-action settings.

## Best Possible Solution

Add a generic embedded Pi bundle MCP elicitation bridge in core: advertise elicitation only when OpenClaw can service it, register an `elicitation/create` handler, route approval-style prompts through plugin approvals with clear server/tool identity, return correct `accept`/`decline`/`cancel`/timeout outcomes, fail closed in non-interactive contexts, and document the supported interactive and non-interactive behavior. Coordinate this with the broader managed MCP registration work in #57355 without making plugin-specific assumptions.
