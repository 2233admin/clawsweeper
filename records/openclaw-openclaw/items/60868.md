## Summary

Current main still lacks the branchâ€™s `.error`/`.cause` chain-message collector and chain-wide overflow handling, so the PR is not obsolete; however, the latest patch has a blocking security/correctness issue because raw nested error messages can now be surfaced outside the existing redaction path.

## What This Changes

The PR adds a shared `.error`/`.cause` message collector, uses it for chain-aware context-overflow detection in model fallback, embedded overflow compaction, and auto-reply classification, and adds regression tests plus a small MiniMax test ordering edit.

## Best Possible Solution

Revise the branch so raw chain candidates are used only for classification, while diagnostics, logs, reply text, stored failure messages, and any displayed error text continue through `formatErrorMessage()` or equivalent redaction. Preserve the useful chain-wide `.error`/`.cause` detection and the billing/rate-limit precedence coverage already added by the newer tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Redact chain messages before surfacing them:** `src/agents/failover-error.ts:218`
  - body: `collectErrorChainMessages()` returns direct nested `message` strings, and the new runtime call sites reuse those strings as overflow/error text. That bypasses the existing `formatErrorMessage()` redaction path, so an inner provider error containing `Authorization: Bearer ...`, an API key, or another secret can leak into logs or user-facing failure replies. Keep raw chain candidates separate from display text, or redact collected messages before returning them for those surfaces.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete secret-redaction regression by exposing raw nested error-chain messages outside the existing formatter/redaction path.

Concerns:

- **[high] Raw nested error messages can leak secrets:** `src/agents/failover-error.ts:218`
  - body: Provider and transport errors may include bearer tokens, API keys, or serialized request metadata in nested `.error`/`.cause.message` fields. The branch collects those raw strings and uses them in runtime diagnostics/replies instead of consistently passing display text through `formatErrorMessage()` or `redactSensitiveText()`.
