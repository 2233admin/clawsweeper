## Summary

Keep open. Current main has retry_after-aware retry for individual Telegram send calls and a shared write-ahead delivery recovery queue, but it still lacks the requested per-chat outbound pacing queue, Telegram queue config/depth/drop policy, and non-blocking queued submit semantics. Related #42246 remains complementary batching/digest work, not a duplicate of transport-layer rate limiting.

## What This Changes

Add a Telegram outbound message queue that serializes sends per chat/topic, applies configurable rate limiting, honors Telegram 429 retry_after, and defines queue depth/drop/backoff behavior.

## Best Possible Solution

Keep this open for a Telegram plugin design pass. The best path is a plugin-owned outbound pacing layer keyed by account/chat/topic that reuses the existing retry_after-aware retry runner, adds documented Telegram queue config and depth/drop behavior, and explicitly decides how queued or non-blocking submission interacts with shared outbound delivery results, hooks, cron status, and bestEffort handling. Keep #42246 as the separate batching/digest discussion.

## Risks / Open Questions

- A naive queue could change user-visible delivery ordering, cron completion semantics, and bestEffort error behavior across shared outbound delivery paths.
- Per-chat pacing needs a clear key model for account/chat/topic and a bounded memory/durability policy so bursts do not become silent loss or unbounded delay.
- Telegram send retries are non-idempotent; any new retry/drain layer must preserve the existing care around ambiguous post-send network failures to avoid duplicate visible messages.
