## Summary

Keep open. Current main has retry_after-aware retry for individual Telegram API calls and a shared write-ahead delivery recovery queue, but it still sends Telegram outbound messages directly and does not implement the requested shared per-chat/topic pacing queue, depth/drop config, or non-blocking queue-submit semantics.

## What This Changes

Add a Telegram outbound send queue that serializes sends per chat/topic, respects Telegram 429 retry_after with backoff, exposes queue tuning config, and prevents burst delivery failures from marking completed cron work as failed.

## Best Possible Solution

Implement a Telegram plugin-owned outbound pacing layer keyed by account/chat/topic that wraps all relevant Telegram outbound paths, reuses the existing retry_after-aware retry runner, adds documented queue config and depth/drop behavior, and preserves the shared delivery contract without blind duplicate retries.

## Reproduction Assessment

Not applicable as a pure bug reproduction, but there is a high-confidence static verification path. Current main shows direct awaited Telegram sends, per-call retry_after retry, and inbound-only sequencing, with no per-chat outbound pacing queue or config.

## Solution Assessment

Unclear. The requested per-chat pacing layer is the right product direction for this failure mode, but the safest implementation needs maintainer design on queue ownership, non-blocking delivery results, and duplicate-safe retry classification.
