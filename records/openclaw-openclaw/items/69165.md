## Summary

Keep open. Current main has retry_after-aware Telegram retry for individual send helper calls and a generic write-ahead delivery recovery queue, but it does not implement the requested per-chat outbound pacing queue, Telegram outbound queue config, queue depth/drop policy, or non-blocking queued submit semantics. The related #42246 batching request remains complementary rather than a duplicate.

## Best Possible Solution

Keep this issue open for a Telegram plugin design pass. The best path is a plugin-owned outbound pacing layer keyed by account/chat/topic that reuses the existing retry_after-aware retry machinery, defines queue depth/drop and best-effort semantics, documents Telegram outbound queue config, and preserves or explicitly updates the shared outbound delivery contract. Keep #42246 as the separate batching/digest discussion.

## Risks / Open Questions

- A direct implementation could accidentally conflate the existing write-ahead delivery recovery queue with the requested per-chat transport pacing queue, causing duplicate sends or stale recovery behavior.
- The requested non-blocking submit behavior changes delivery result and cron success/error semantics; maintainers should decide how queued Telegram sends report message IDs, failures, and dropped messages before implementation.
- This overlaps but does not duplicate #42246; batching and per-chat transport throttling should be designed so they compose rather than racing or double-buffering messages.
