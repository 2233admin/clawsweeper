## Summary

Keep open. Current main still parses `[[reply_to_current]]` as shared delivery metadata and has no per-agent directive policy in the strict agent config schemas, so the requested runtime guard is not implemented.

## What This Changes

The issue requests a per-agent config/runtime directive to disable processing of `[[reply_to_current]]` so accidental agent output cannot trigger reply threading in channels such as Feishu DM.

## Best Possible Solution

Add a documented per-agent reply-directive policy, resolve it from the active session agent, and apply it at the shared directive parsing boundary before channel adapters receive `replyToId`; keep Feishu-specific outbound behavior unchanged except that disabled agents no longer produce a reply id from `reply_to_current`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is `applyReplyThreading` with a payload containing `[[reply_to_current]]` and a `currentMessageId`; current tests assert this resolves to `replyToId`, including explicit-tag paths that survive implicit-deny or channel off-mode handling.

## Solution Assessment

Yes, the requested direction is a reasonable narrow fix if implemented as a shared parser policy rather than a Feishu-only workaround. The safer variant gates only `reply_to_current` for the selected agent and leaves explicit id, media, and audio directives intact.
