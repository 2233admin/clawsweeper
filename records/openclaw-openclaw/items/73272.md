## Summary

Current main still honors `[[reply_to_current]]` through the shared reply directive parser and outbound threading paths, and the strict per-agent config schema has no `directives.replyToCurrent` policy. The broader same-author #72627 overlaps, but this issue is a narrower, actionable runtime-guard request and should stay open for a focused fix PR.

## What This Changes

Request a per-agent configuration directive that prevents selected agents' accidental `[[reply_to_current]]` output from being converted into reply metadata.

## Best Possible Solution

Add a documented, backward-compatible per-agent directive policy and enforce it at the shared reply directive parsing/threading boundary for final payloads, streaming chunks, and outbound normalization, while preserving Feishu transport behavior and explicit message-tool replies.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to pass text containing `[[reply_to_current]]` plus a `currentMessageId` through `parseInlineDirectives` or `applyReplyThreading` and observe that current main produces `replyToId` without considering the active agent.

## Solution Assessment

Yes. For this narrower issue, the best path is a shared agent-scoped parser/threading policy; a Feishu-only transport workaround would leave the runtime-owned directive behavior unchanged for other channels and payload paths.
