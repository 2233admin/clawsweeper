## Summary

Keep open. Current main still has no per-agent directive policy for `[[reply_to_current]]`: shared parsing and reply delivery paths continue to convert the tag into reply metadata, and the strict agent config schema has no `directives.replyToCurrent` field. The related #72627 is broader and still open, so this remains a valid focused repair slice rather than an implemented or duplicate cleanup close.

## What This Changes

Requests an opt-in per-agent config to disable runtime processing of accidental `[[reply_to_current]]` output before it becomes reply metadata.

## Best Possible Solution

Add a backward-compatible, opt-in per-agent directive policy in the shared agent config and reply-directive runtime, apply it consistently before `reply_to_current` becomes `replyToCurrent` or `replyToId`, preserve explicit `[[reply_to:<id>]]` and message-tool replies, and document the config surface.

## Reproduction Assessment

Yes. A source-level reproduction is to pass text containing `[[reply_to_current]]` plus a `currentMessageId` through `parseInlineDirectives`, `applyReplyThreading`, streaming directives, or outbound payload planning and observe reply metadata emitted without any agent-policy check.

## Solution Assessment

Yes, scoped narrowly. The requested opt-in runtime guard is the narrow maintainable fix for accidental tag processing; auto-send suppression belongs to the broader #72627 discussion unless maintainers expand this task.
