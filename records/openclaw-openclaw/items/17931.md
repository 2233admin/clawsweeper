## Summary

Keep open. Current main still does not implement automatic read-only skill directory mounts. Docker `ro`/`none` workspaces are mounted read-only now, but non-rw sessions still stage skills by copy, `rw` sessions still expose workspace skills through the writable workspace mount, and the #22669 bind-precedence fix only supports a manual custom-bind workaround rather than generating per-skill read-only overlays.

## Best Possible Solution

Keep this open for maintainer/secops direction. A strong fix would derive eligible skill source directories from the existing skill loader, overlay read-only skill binds after broader Docker workspace mounts for `workspaceAccess: "rw"`, avoid writable duplicate skill state for `ro`/`none` when source mounts are available, define SSH/OpenShell behavior, update config-hash and fs-bridge mount modeling, and revise sandbox/skills docs plus tests across all workspace access modes.
