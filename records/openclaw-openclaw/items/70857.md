## Summary

Keep open. Current main and the latest release still contain the reported session-store write pattern: session-store mutations acquire the store lock, then force a `skipCache: true` load that bypasses the in-process cache and synchronously reads/parses `sessions.json` while the lock is held. The fresh Windows/macOS corroboration and open PR #68554 make this active implementation work, not cleanup-close material.

## What This Changes

The issue reports slow Windows startup and reply latency associated with long-held session-store write locks on `sessions.json.lock`.

## Best Possible Solution

Keep this issue open and review or land #68554, or an equivalent maintainer-owned current-main fix that removes redundant locked `skipCache` disk rereads from `updateSessionStore` and `updateSessionStoreEntry` while preserving cache coherency, external-writer safety, ACP metadata preservation, and the current `clone: false` memory behavior. Update the stale-cache regression test to match the intended contract, consider a focused large-store/performance guardrail, then validate startup, `node.list`, `sessions.list`, cron activity, and first-turn reply latency with a large session store.
