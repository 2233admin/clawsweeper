## Summary

Keep this issue open. Current main has transcript-backed token and cost summaries plus auth-profile rotation/fallback routing, but completed model-call usage is still exposed and aggregated by provider/model/session surfaces, not by the resolved auth profile that paid for each call. The existing burn0 discussion is useful as an interim per-request cost workaround, but it does not replace native OpenClaw auth-profile attribution.

## Best Possible Solution

Keep this issue open until OpenClaw records the resolved authProfileId for each completed model call in safe metadata, exposes it through transcript/usage/status or diagnostics, aggregates cost by auth profile, and emits a clear user-facing profile rotation/fallback event. burn0/#55379 can remain an external cost workaround, but it is not a native replacement for OpenClaw auth-profile attribution.

## Risks / Open Questions

- Adding auth-profile attribution needs careful redaction: profile ids may be safe diagnostic metadata, but credential material must not enter transcripts, logs, status payloads, UI, or plugin hooks.
- The change would affect persisted transcript readers and gateway/UI usage consumers, so it should be additive and backward-compatible.
- Resolved-profile attribution must handle fallback retries, cooldown probes, user-pinned profiles, provider aliases, plugin-owned transports, and CLI harnesses consistently.
