## Summary

Keep open. The current PR is useful, but it is not a complete fix for the linked session-memory poisoning report because it only removes known chat-template special tokens; the reported raw <tool_call> XML, role-marker, and NO_REPLY artifacts would still flow into memory files.

## What This Changes

The branch exports the existing LLM special-token replacement helper, applies it while reading session-memory transcript text, adds focused transcript regression tests, and adds a changelog entry.

## Best Possible Solution

Session-memory should have a hook-owned sanitizer at the transcript read or write seam that neutralizes every model-artifact class in the report, with tests for special tokens, raw tool-call XML, role-marker scaffolding, NO_REPLY-only turns, clean prose preservation, and the final memory-file output.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence synthetic path: a session JSONL assistant message containing raw <tool_call>, role-marker, or NO_REPLY text is extracted as text and written under Conversation Summary; this PR would only replace special-token literals in that string.

## Solution Assessment

No. Reusing the existing special-token helper is a good subset, but it is not the narrowest maintainable complete fix for #69943 unless the session-memory sanitizer also strips or elides the other persisted artifact classes before memory reinjection.
