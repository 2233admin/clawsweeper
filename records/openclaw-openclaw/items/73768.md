## Summary

Keep open. The patch is aimed at the right session-memory seam and fixes known chat-template literals, but it would under-fix the linked bug because raw `<tool_call>` XML, orphan role-marker text, and `NO_REPLY` artifacts can still flow into saved memory and be re-injected. Maintainers should reconcile this with the broader open sanitizer PR #69961 before merge.

## What This Changes

The PR exports `replaceLlmSpecialTokenLiterals`, applies it to string and text-block content returned by the session-memory transcript reader, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land one canonical session-memory hardening path that sanitizes or elides all model artifacts identified in #69943 before memory persistence or reuse, with tests for special tokens, tool-call XML, orphan role markers, `NO_REPLY`/garbage turns, and clean-content preservation.

## Reproduction Assessment

Yes. A synthetic session JSONL message containing raw `<tool_call>...</tool_call>`, bare role-marker lines, or `NO_REPLY` still follows the same `getRecentSessionContent` to `entryParts.push(sessionContent)` path and is outside this PR's token-literal helper.

## Solution Assessment

No. Reusing the existing token-literal helper is a clean subset fix, but it is not the best complete fix for the linked bug because the reported poisoning loop includes non-token artifacts that remain untreated; #69961 is the closer canonical direction.
