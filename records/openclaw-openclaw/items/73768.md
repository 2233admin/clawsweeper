## Summary

Keep open. Current main still persists raw session-memory transcript text, and this PR is a useful narrow step for chat-template tokens, but it does not cover the full linked poisoning vector: raw tool-call XML, orphan role markers, and `NO_REPLY` artifacts would still be written into `memory/` and re-injected. The earlier changelog-drop concern appears resolved in the latest diff/current main, so the remaining blocker is fix scope, especially with broader PR #69961 already open for the same bug.

## What This Changes

The PR exports the existing LLM special-token literal sanitizer, applies it to string and text-block content returned by the session-memory transcript reader, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land one canonical session-memory hardening path that sanitizes or elides all model artifacts identified in the linked bug before memory persistence or reuse: chat-template tokens, plain tool-call XML, orphan role markers, exact malformed silent tokens, and dominated garbage turns. Keep the bug linked until that canonical fix, tests, and changelog entry match the actual covered behavior.

## Reproduction Assessment

Yes. A synthetic session JSONL turn containing raw `<tool_call>...</tool_call>`, role-marker lines, or `NO_REPLY` without special-token literals still follows the same `getRecentSessionContent` to `entryParts.push(sessionContent)` path and is not removed by this PR's helper.

## Solution Assessment

No. Reusing the existing token-literal helper is a maintainable subset fix, but it is not the best complete fix for the linked issue because the reported poisoning loop includes several non-token artifacts that remain untouched.
