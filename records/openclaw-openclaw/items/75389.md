## Summary

Keep open. The PR is a useful implementation candidate for #75380, but it is not merge-ready because it wires default diagnostic caps into a writer whose cap behavior permanently drops later JSONL events instead of rotating to a fresh active file, and the new config/API surface is not reflected in docs, generated config metadata, or the changelog.

## What This Changes

The PR adds default/env/config max-byte caps for cache-trace and Anthropic payload JSONL writers, plus focused tests that accept the new settings.

## Best Possible Solution

Land a focused bounded-rotation implementation for queued diagnostic JSONL files, preserving the existing symlink/hardlink/file-permission protections, then document and generate the new cache-trace and payload-log retention settings with changelog coverage for #75380.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: fill a cache-trace or payload JSONL file to the configured cap, then the PR's call into `getQueuedFileWriter(..., { maxFileBytes })` will return without appending future events instead of rotating.

## Solution Assessment

No. As submitted, the PR addresses unbounded growth only by suppressing future diagnostics after the cap; the maintainable fix for #75380 is bounded rotation or an explicitly accepted retention policy, with the public config/docs surface kept in sync.
