## Summary

Keep open. The PR is a useful narrow attempt for #75380, but current evidence shows it caps the diagnostic JSONL sinks by dropping future writes after the active file reaches the threshold, while the linked issue asks for bounded rotation/retention that keeps diagnostics flowing. The latest head adds generated base schema coverage, but still leaves the public config/help/docs/changelog surface incomplete.

## What This Changes

The PR adds default/env/config max-byte handling for cache-trace and Anthropic payload JSONL logging, updates base config schema/types, and adds focused tests for the new settings.

## Best Possible Solution

Land a focused bounded-rotation implementation for queued diagnostic JSONL files, preserving the existing symlink, hardlink, directory-mode, file-permission, and redaction protections, then publish the cache-trace and payload-log retention settings through schema metadata, docs, changelog, and focused regression tests.

Yes. Source inspection gives a high-confidence path: fill a cache-trace or payload JSONL file to the configured cap, then the PR's `getQueuedFileWriter(..., { maxFileBytes })` call reaches the writer's cap branch and future diagnostic lines are skipped instead of rotating.
