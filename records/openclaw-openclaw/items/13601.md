## Summary

Keep this issue open. Current main has durable JSONL-backed cron run history plus status/delivery/query filtering through `cron.runs` and the Control UI, but the issue’s requested product surface is still only partially implemented: no timestamp range filters, no `openclaw cron logs` or `cron history` command surface with status/time/follow/export options, no success-rate/failure aggregates in `cron list`, and no manual prune/vacuum/history-retention-days workflow. The related status UX issue #16799 and PR #16831 do not supersede this because they targeted `openclaw status`; the PR was closed unmerged in the provided context.

## Best Possible Solution

Keep this issue open until maintainers either narrow or implement the remaining cron history product surface. A complete path would add timestamp range fields to `ReadCronRunLogPageOptions`, `CronRunsParamsSchema`, server handlers, UI state, and CLI options; expose a clearer `openclaw cron logs` or `openclaw cron history` command with status/time/follow/export behavior; add success-rate and failure aggregates to `cron list` or a dedicated summary; and decide whether manual prune/vacuum/history-retention-days belongs in core or should be documented out of scope.
