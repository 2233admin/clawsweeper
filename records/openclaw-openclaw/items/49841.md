## Summary

Keep this PR open. Current main still does not implement a Nacos-backed config source, and there is no current plugin or bundle API that can replace the core gateway config load path before startup. The PR is also not merge-ready as-is because it adds a new env-driven outbound HTTP config path and has substantive unresolved review concerns around reload, write semantics, CLI coverage, and source/runtime snapshot preservation.

## Best Possible Solution

Keep this PR open for maintainer product review. The best implementation path is a generic, tested core config-source seam with explicit Nacos support only if maintainers want remote config in core: it should preserve source/runtime snapshots, define persistent vs in-memory write behavior, cover all CLI/gateway readers consistently, validate Nacos long-poll/auth/error handling against real Nacos behavior, update docs/changelog, and pass changed config/gateway checks. If maintainers decide remote config backends belong outside core, the better follow-up is to design a plugin API first rather than closing this as already satisfiable through today’s plugin surface.
