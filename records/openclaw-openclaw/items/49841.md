## Summary

Keep this PR open. Current main does not implement a Nacos config source, so the PR is not obsolete, but the proposed branch is still unmerged, merge-conflicted, and has blocking gateway/config correctness issues plus a remote-config security review question before it can land.

## What This Changes

The PR adds an opt-in Nacos-backed gateway config source with file/Nacos adapters, env-based source resolution, Nacos long-poll reload subscription, CLI/doctor/startup wiring, docs, changelog, and tests.

## Best Possible Solution

If remote config belongs in core, land a source seam where file remains the default and Nacos is an explicit, validated adapter used consistently by startup metadata reads, runtime reads, writes, reloads, docs, and security guardrails. If Nacos should remain outside core, define the missing pre-start config-source plugin API first and move the adapter to plugin space.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route startup metadata reads through the active source:** `src/gateway/server-startup-config.ts:205-206`
  - body: `loadGatewayStartupConfigSnapshot()` initializes the Nacos source, but immediately calls `readConfigFileSnapshotWithPluginMetadata()`, whose top-level implementation still delegates straight to `createConfigIO()`. A Nacos-only gateway can therefore start from local/default config while the later reloader is subscribed to Nacos. Make the plugin-metadata snapshot path source-aware before enabling this mode.
  - confidence: 0.91
- **[P2] Notify subscribe reloaders after Nacos writes:** `src/config/io.ts:2501`
  - body: In Nacos mode `writeConfigFile()` updates the runtime snapshot and returns, while subscribe-mode reloaders do not register the existing in-process write listener. Successful `config.set`/`config.patch` calls can update `config.get` without reapplying hot-reloadable services until an external Nacos edit or restart occurs.
  - confidence: 0.88
- **[P3] Describe standalone CLI Nacos writes accurately:** `docs/gateway/configuration.md:744`
  - body: The docs say CLI `config set` updates the in-memory snapshot in Nacos mode, but `writeConfigFile()` now throws in a standalone process with no runtime snapshot or refresh handler. Update the docs/CLI wording or route standalone writes through a running gateway session.
  - confidence: 0.84
