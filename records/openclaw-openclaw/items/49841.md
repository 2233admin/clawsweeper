## Summary

Keep this PR open. Current main still has no Nacos config source, and the PR is an open, dirty, broad gateway/config change with a blocking reload-semantics bug plus unresolved remote-config security guardrails.

## What This Changes

The PR adds an optional Nacos-backed gateway config source with file/Nacos source resolution, Nacos HTTP fetch/long-poll subscription, gateway reload/startup/CLI/doctor wiring, docs, changelog, and tests.

## Best Possible Solution

If maintainers want remote config in core, land a generic config-source seam with file as the default and Nacos as an explicit, validated adapter that preserves current read/write/reload semantics, documents persistence limits accurately, includes changelog credit, and has reviewed auth/transport guardrails plus real Nacos validation. If that scope belongs outside core, define the pre-start config-source plugin API first and move Nacos support to plugin space.

## Reproduction Assessment

Yes. Diff inspection gives a focused reproduction: enable Nacos source, call a gateway config write that changes a hot-reloadable setting, and observe that the Nacos write branch returns without notifying the subscribe-mode reloader. A live Nacos smoke is still needed before shipping.

## Solution Assessment

No. The proposed abstraction is useful, but this PR is not the best mergeable form yet because it bypasses the existing write notification contract in subscribe mode and introduces remote config security semantics before the supported guardrails are defined.
