## Summary

Keep this open. Current main still rejects arbitrary direct top-level fields under `skills.entries.<skill>`, and the linked implementation PR is closed unmerged; however, current docs now explicitly define `skills.entries.<skill>.config` as the custom-field bucket, so the remaining decision is a config-contract/product choice rather than something safe to auto-close.

## What This Changes

Allow `skills.entries.<skill-name>` to persist arbitrary custom top-level fields such as `api_endpoint` and `user_api_key` without runtime config validation rejecting the update.

## Best Possible Solution

Pick one public contract and align schema, generated schema, docs, tests, and UI behavior around it: either support direct custom top-level skill-entry fields with explicit validation/redaction guardrails, or keep the stricter contract and route custom skill data to `skills.entries.<skill>.config` while routing Tavily users to `plugins.entries.tavily.config.webSearch.*`.

## Reproduction Assessment

Yes. The current source and existing unit test give a high-confidence reproduction path: parsing a config with `skills.entries.custom-skill.url` as a direct top-level field fails, while the same custom data under `config` is accepted; no test command was run because this review is read-only.

## Solution Assessment

Unclear. The direct-field fix proposed by #74024 is narrow mechanically, but current docs intentionally describe nested `config` as the custom-field bucket, so maintainers should decide the public config contract before another implementation PR changes it.

- Allowing arbitrary direct top-level fields under each skill entry would weaken typo detection unless known fields remain strongly validated.
