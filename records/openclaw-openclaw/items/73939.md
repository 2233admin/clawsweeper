## Summary

Keep this issue open. Current main still rejects custom top-level fields under `skills.entries.<skill>`, and the GitHub context shows open unmerged PR #74024 already references this issue with closing syntax as the implementation candidate. The older write-then-reload rollback symptom is partly stale because `config.patch` now validates before writing, but the requested direct custom-field capability is not implemented on main.

## Best Possible Solution

Keep this issue open while maintainers review, land, or close PR #74024. If direct custom skill fields are accepted, the fix should update `SkillEntrySchema`, generated schema output, focused schema tests, docs, and secret/redaction metadata for custom fields. If maintainers prefer the stricter contract, docs and UI messaging should clearly route custom skill data to `skills.entries.<skill>.config` and route Tavily users to `plugins.entries.tavily.config.webSearch.*`.
