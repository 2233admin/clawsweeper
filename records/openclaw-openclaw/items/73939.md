## Summary

Keep this open. Current main still rejects unknown direct fields under `skills.entries.<skill>` and the generated schema, runtime schema, docs, and focused tests now consistently describe the stricter nested-`config` contract; the linked fix PR #74024 is closed unmerged, so the remaining work is an explicit maintainer decision on whether to broaden the public config shape.

## What This Changes

The issue asks OpenClaw to allow custom per-skill fields such as `api_endpoint` and `user_api_key` directly under `skills.entries.<skill-name>` without Gateway runtime validation rollback.

## Best Possible Solution

Make an explicit maintainer decision for the skills config contract. If direct top-level custom fields are allowed, update the Zod schema, generated schema, exported types, docs, UI/schema hints, focused validation tests, and secret-handling coverage together; if the current strict shape is intentional, document the migration path more clearly and route custom data to `skills.entries.<skill>.config` or plugin-owned config such as Tavily’s `plugins.entries.tavily.config.webSearch.*`.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: `OpenClawSchema.safeParse` rejects an unknown direct field under `skills.entries.custom-skill`, the current focused test asserts that rejection, and Gateway save/reload paths call the same validation layer.

## Solution Assessment

Unclear. Allowing unknown top-level fields would satisfy the reported examples, but it changes the public config contract and secret-handling surface; the safer maintainable path needs maintainer confirmation before automation changes schema behavior.
