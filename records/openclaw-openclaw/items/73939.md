## Summary

Keep this issue open. Current main no longer matches the reported write-then-`reload-invalid-config` rollback path because `config.patch` validates the merged config before writing, and the generated schema/runtime schema both reject unknown top-level `skills.entries.<skill>` fields. But the requested capability, persisting direct custom fields such as `api_endpoint` and `user_api_key` under `skills.entries.<skill>`, is still not implemented as written. The current contract is `enabled`, `env`, `apiKey`, and `config`, while Tavily specifically is now plugin-owned and documented under `plugins.entries.tavily.config.webSearch.*`.

## Best Possible Solution

Have a maintainer decide the skill config contract explicitly. The conservative code path is to keep direct `skills.entries.<skill>` fields strict, document the supported escape hatches more clearly, and route Tavily users to `plugins.entries.tavily.config.webSearch.*`. If OpenClaw wants custom per-skill config as a first-class feature, add it under a deliberate payload such as `skills.entries.<skill>.config`, with schema, redaction, UI, docs, and runtime consumption behavior aligned.

## Risks / Open Questions

- Loosening `skills.entries.<skill>` to allow arbitrary top-level fields would create a config-contract and redaction question, especially for secret-like custom names that are not in current schema metadata.
- Tavily is plugin-owned on current main; making `skills.entries.tavily.apiKey` drive Tavily behavior would conflict with the documented plugin config path unless a deliberate compatibility bridge is added.
- External skills may expect different config shapes, so maintainers should decide whether the supported escape hatch is `skills.entries.<skill>.config`, `env`, `apiKey` with `primaryEnv`, or a new versioned skill config API.
