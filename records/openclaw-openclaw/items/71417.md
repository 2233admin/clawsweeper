## Summary

Keep open. Current main still has the reported session-resume UX gap: there is no fresh-session flag, no force-fresh Gateway protocol field, no resume banner, and no structured resumed metadata, while both embedded and Gateway paths still reuse a fresh stored session when the caller targets an agent without an explicit fresh intent. The adjacent open PR #72932 is useful context but, based on the provided PR summary, does not replace the missing fresh-session control or resume visibility requested here.

## What This Changes

The issue requests a way for `openclaw agent` calls without `--session-id` to avoid or clearly report implicit reuse of the persisted main agent session.

## Best Possible Solution

Preserve resume-by-default compatibility, but add an explicit fresh-session control and make implicit resume observable in both Gateway and `--local` paths through stderr diagnostics and structured JSON metadata, with matching docs and regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a fresh `agent:<id>:main` session entry, run `openclaw agent --agent <id> -m ...` without `--session-id`, and observe current embedded/Gateway resolution select the stored `sessionId`; I did not run the live CLI because this review is read-only.

## Solution Assessment

Yes. The reporter's Option B shape is the narrowest maintainable fix: add `--new-session`/`--fresh`, plumb force-fresh intent through local and Gateway paths, and emit resume visibility, while avoiding the compatibility break of changing the default to fresh.
