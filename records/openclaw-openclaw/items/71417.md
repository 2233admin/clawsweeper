## Summary

Keep this issue open. Current main still lacks an `openclaw agent` fresh-session control or resume banner, omitted/blank `--session-id` with `--agent` still resolves to the agent main session, fresh stored sessions are reused in both Gateway and embedded paths, and omitted delivery channel still falls back to `last`. The reporter's follow-up is concrete and recent, and no merged implementation or maintainer close direction is present in the provided discussion.

## Best Possible Solution

Keep the issue open. Implement the reporter's backward-compatible Option B: add `--new-session` with optional `--fresh` alias, extend `AgentCliOpts` and `AgentParamsSchema`, plumb a force-new flag through Gateway and local embedded execution, short-circuit session reuse in `resolveSession` and the Gateway handler when set, emit a concise stderr resume banner when an existing session is reused, update `docs/cli/agent.md` and `docs/tools/agent-send.md`, and add focused tests for omitted, blank, explicit existing `--session-id`, and force-new behavior across Gateway and `--local`.

## Risks / Open Questions

- Silent resume can keep unrelated old transcript context and inflate token usage for operator tests.
- Changing the default to fresh would be a compatibility break for scripts that intentionally rely on auto-resume, so the safer path is likely an opt-in flag plus an explicit resume banner.
- A resume banner may affect scripted stderr consumers; this needs a deliberate suppressibility or always-on decision from maintainers.
