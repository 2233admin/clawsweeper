## Summary

Keep open. Current main still reuses a fresh stored agent main session when `openclaw agent --agent <id>` omits a real `--session-id`, and there is no `--new-session`/`--fresh` flag, Gateway protocol field, resume banner, or docs coverage. The adjacent explicit `--session-id` remapping behavior is partially fixed, and #72932 is an open related PR, but neither proves the remaining fresh-session UX gap is resolved on current main.

## What This Changes

The issue requests a safer `openclaw agent` session UX: either default fresh sessions, add an explicit fresh-session flag, or warn/annotate when an existing session is resumed implicitly.

## Best Possible Solution

Preserve resume-by-default compatibility for existing multi-turn workflows, add an explicit fresh-session flag with an alias, plumb the intent through local and Gateway execution, and make implicit resume observable in stderr and structured JSON metadata with docs and regression tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: create or seed a fresh `agent:<id>:main` store entry, run `openclaw agent --agent <id> -m ...` without a real `--session-id`, and both local and Gateway resolution can select the stored session id; I did not run the live CLI because this review was read-only.

## Solution Assessment

Yes. The reporter’s backward-compatible Option B is the narrowest maintainable direction: add a fresh-session knob and observable resume behavior while preserving scripts and multi-turn workflows that intentionally rely on agent main-session continuity; flipping the default should be a separate product/deprecation decision.

## Repair Work Prompt

Fix openclaw/openclaw#71417. Current main still implicitly resumes a fresh stored `agent:<id>:main` session when `openclaw agent --agent <id> -m ...` omits a real `--session-id`. Add a backward-compatible explicit freshness control: register `--new-session` with `--fresh` as an alias on `openclaw agent`; plumb a boolean through `AgentCliOpts`, Gateway `AgentParamsSchema`, `agentViaGatewayCommand` params, local `AgentCommandOpts`, and session resolution. When true, bypass stored-session reuse and allocate a new session id under the selected session key. When false and an existing session is reused, make resume observable with a one-line stderr diagnostic and structured JSON metadata such as `resumed`, `sessionKey`, and last activity/start timestamp, without polluting JSON stdout. Coordinate with related PR #72932 and preserve the existing explicit `--session-id` behavior; do not flip the default to fresh. Update `docs/cli/agent.md`, `docs/tools/agent-send.md`, focused parser/local/Gateway/session tests, and `CHANGELOG.md` with user-facing credit to @mayank6136. Do not change unrelated Gateway startup, Bonjour, provider, or channel behavior.
