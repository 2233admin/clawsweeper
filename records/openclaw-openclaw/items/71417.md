## Summary

Keep open. Current main still has the reported CLI/session UX gap: `openclaw agent --agent <id>` without explicit fresh intent can reuse the stored fresh `agent:<id>:main` session, the CLI/protocol expose no fresh-session flag, and the docs do not describe a resume banner or structured resumed metadata. The reporter's follow-up comments add useful real-world reproduction detail and favor the backward-compatible flag-plus-banner path; related PR #72932 is adjacent but remains open and does not supersede this issue.

## What This Changes

The issue asks for `openclaw agent` to provide a fresh-session control or visible resume notice instead of silently reusing the persisted agent main/last session when no fresh session is requested.

## Best Possible Solution

Keep the compatibility-preserving resume default, add an explicit fresh-session flag through both Gateway and local embedded paths, surface implicit resume in stderr and JSON metadata, and update CLI docs plus a user-facing changelog entry with reporter credit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: seed a fresh `agent:<id>:main` session entry, run `openclaw agent --agent <id> -m ...` without session controls, and the local or Gateway resolver selects the stored `sessionId`; I did not run the live CLI because this review was read-only.

## Solution Assessment

Yes. The reporter's Option B is the narrowest maintainable path: preserve scripts that rely on resume-by-default while adding an explicit fresh-session knob and visible resume diagnostics instead of silently flipping the default behavior.
