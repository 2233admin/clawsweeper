## Summary

Keep open. Current main still exposes only compaction metadata to typed `after_compaction` plugin hooks, while the generated summary is available in the compaction result and is already used for internal lifecycle metadata. A prior PR for the same idea was closed unmerged as a dirty branch, so the remaining work is real and narrow rather than implemented or superseded.

## What This Changes

The issue requests adding the generated compaction summary text and optional summary token count to the `after_compaction` plugin hook payload.

## Best Possible Solution

Add an optional summary-text field to the public typed plugin hook contract, forward it through every `after_compaction` emitter that already has a generated summary, document the field, and cover direct, engine-owned, overflow, harness, and wired auto-compaction paths with focused tests. Add summary-only token metadata only if it can be computed with an existing reliable tokenizer or estimator without changing compaction behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: enable or mock `after_compaction`, force a compaction path with a generated summary, and observe that the current hook payload contains counts/session file but no summary text; the existing tests already assert that old shape.

## Solution Assessment

Yes, the narrow additive contract change is the best direction. It should use optional fields and runtime `typeof summary === "string"` guards, avoid exposing pre-compaction message content in this PR, and keep the existing compaction algorithms unchanged.
