## Summary

Keep this issue open. Current main still exposes only the source-channel `Timestamp` through the shared inbound context, agent-visible inbound metadata, SDK inbound helpers, and plugin/internal hook payloads. The current `receivedAt`/`receivedAtMs` matches are local timing/debug fields in specific plugins or harness code, not a cross-channel Gateway receipt timestamp contract that agents can use for latency accounting.

## Best Possible Solution

Keep this issue open. Add an additive shared receipt timestamp, such as `ReceivedAtMs` or `GatewayReceivedAtMs`, to `MsgContext`/`FinalizedMsgContext`, SDK inbound envelope/direct-DM helpers, and inbound hook payloads. Stamp it once at channel ingress or finalization when absent, allow buffered plugins to preserve the first receipt time, render a stable ISO `receivedAt` or `gatewayReceivedAt` in agent-visible inbound metadata, and cover the metadata builder, hook mapper, SDK helper, and one buffered or long-poll-style channel path with focused tests.

## Risks / Open Questions

- Closing now would lose a real observability gap for long-poll and buffered channel paths: current agent-visible latency calculations still have only the source-channel timestamp.
- A future fix should be additive because `Timestamp` is already used as the source-channel timestamp in public SDK and hook contracts.
- Receipt time is per-message volatile data, so it should be rendered in dynamic inbound metadata and hook payloads rather than stable system prompt metadata that affects prompt caching.
