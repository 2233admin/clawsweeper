## Summary

Keep this issue open. Current main still carries only the source-channel message timestamp through the shared inbound context, agent-visible conversation metadata, direct-DM helper, generic channel turn builder, and message hook contracts; Telegram has an internal `receivedAtMs`, but it is used for buffering/debug timing and is not exposed as shared inbound metadata.

## What This Changes

Add a Gateway receipt timestamp to inbound metadata so agents can distinguish source/channel delivery latency from OpenClaw processing latency.

## Best Possible Solution

Add a backward-compatible shared receipt timestamp stamped once at inbound Gateway/channel ingress, preserve `Timestamp` as the source-channel message time, expose the new value through agent-visible inbound metadata and typed hook/SDK payloads, and document the semantics for buffered paths.
