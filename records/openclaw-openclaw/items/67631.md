## Summary

Keep open. Current main still documents and implements the macOS app and CLI as sharing the `ai.openclaw.gateway` LaunchAgent, with manual escape hatches (`--attach-only`, disable marker, and `OPENCLAW_SERVICE_REPAIR_POLICY=external`) but no automatic terminal-doctor guard that detects the app LaunchAgent before installing or restarting the gateway service. The issue wording is partly imprecise because the app autostart job is `ai.openclaw.mac`, not the gateway label, but the underlying ownership conflict remains actionable.

## Best Possible Solution

Keep this issue open and implement a conservative macOS gateway ownership guard. Terminal `openclaw doctor --fix` and gateway service repair should detect an installed or loaded `ai.openclaw.mac` app LaunchAgent and either skip install/start/restart with a clear note, automatically honor an external-supervisor/attach-only policy for app-owned setups, or route both app and CLI through one canonical gateway owner. Add regression coverage around `maybeRepairGatewayDaemon`, launchd service detection, and the app attach-existing fallback, then update the macOS gateway and doctor docs with safe Mac app guidance.
