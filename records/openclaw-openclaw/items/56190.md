## Summary

Keep open. Current main still lacks `${VAR:-default}` fallback support, and the PR is useful but not mergeable as-is because defaulted refs are still treated as unresolved refs and skipped by config env-var injection.

## What This Changes

The PR adds a brace scanner and updates config env substitution, env-ref write-back preservation, and unit tests to support `${VAR:-default}` fallbacks in config strings.

## Best Possible Solution

Land fallback support after the parser/helper API can separate default-resolvable refs from required unresolved refs, or resolve fallback-bearing `env.vars` values before the existing unresolved-ref skip. Add a regression test where `env.vars.API_KEY` is `${UPSTREAM:-demo}` and another config field consumes `${API_KEY}`.

## Reproduction Assessment

Yes. A high-confidence reproduction path is `env.vars.API_KEY = "${UPSTREAM:-demo}"` with another config field set to `${API_KEY}`; the PR still reports the `env.vars` value as an env reference, so the injection path skips it before substitution.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not the best merge shape until defaulted refs bypass the unresolved-ref skip or are resolved before `applyConfigEnvVars`; a distinct required-ref helper or token kind would be safer.
