## Summary

Keep this issue open. Current main has Telegram group allowlists, owner-derived Telegram exec approvers, native Telegram approval delivery, private routing for some sensitive owner-only group commands, and explicit session-level approval skips such as `/elevated full`, but it still does not implement the requested opt-in trusted-owner/group/tool-class approval bypass with audit labeling and destructive-operation fallback behavior.

## Best Possible Solution

Keep this issue open for maintainer design. A complete implementation should live primarily in the Telegram plugin/config surface, with a generic approval-policy seam only if the host approval pipeline needs one. It should require resolved numeric owner IDs, allowlisted group/topic scope, configured tool classes, an audit reason such as `trusted-owner-bypass`, destructive-operation fallback behavior, schema/docs updates, and focused Telegram approval-path tests.
