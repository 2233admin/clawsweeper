## Summary

Keep this issue open. Current main still loads and watches the personal `~/.agents/skills` source without a source-level opt-out, while the strict public config surface only supports extra dirs, watcher toggles, prompt limits, allowlists, and per-skill disables after discovery. The related #64639 discussion was closed as a duplicate of this older canonical issue, so this remains valid narrow follow-up work rather than cleanup to close.

## What This Changes

Request a config option to disable scanning/loading/watching `~/.agents/skills` for Claude Code coexistence, with possible extension to `<workspace>/.agents/skills`.

## Best Possible Solution

Add a backwards-compatible skills load setting that filters disabled built-in source roots before filesystem discovery, prompt construction, status/snapshot exposure, and watcher registration. Keep defaults unchanged, document the source ids and examples, and leave broader on-demand skill routing as a separate design item.

## Reproduction Assessment

Yes. The current code and tests provide a high-confidence reproduction path: place a skill under a mocked `~/.agents/skills`, build the workspace skills prompt, and it is included; separately, `ensureSkillsWatcher` registers the home `.agents/skills` target with no source-specific config guard.

## Solution Assessment

Yes for the short-term coexistence issue. A source-level opt-out is narrower and more maintainable than dozens of `skills.entries.*.enabled: false` entries; allowlists and prompt limits can reduce visibility after discovery but do not stop loading or watching the unwanted source.
