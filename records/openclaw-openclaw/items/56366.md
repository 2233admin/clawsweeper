## Summary

Current main still always scans `~/.agents/skills` and `<workspace>/.agents/skills`; there is no source-level disable, ignore list, or `scanAgentsSkills` option in the typed config or zod schema. Existing per-skill disables, per-agent allowlists, and prompt limits reduce exposure after loading, but they do not solve the requested coexistence/control over the personal Claude Code skill directory.

## Best Possible Solution

Keep this issue open as actionable core skills/config work. The likely implementation belongs in the skills loader and watcher, with a typed and zod-validated `skills.load` option that can exclude personal and possibly project `.agents/skills` roots, plus docs and regression tests proving excluded roots neither load into `<available_skills>` nor participate in watcher targets.

## Risks / Open Questions

- A fix should cover both initial loading and watcher refresh paths, otherwise disabled roots could still trigger refresh churn or reappear after a snapshot rebuild.
- The exact public config shape needs maintainer choice: a narrow `scanAgentsSkills` toggle is lower risk, while a general ignore/source policy is more flexible but affects config schema, docs, generated help, and tests.
