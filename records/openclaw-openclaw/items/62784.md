## Summary

Keep this open. Current main has partial Slack Socket Mode resilience work, including a 15s default Slack SDK pong timeout and bounded reconnect backoff with jitter, but the central reported startup behavior remains: all configured accounts for a channel are still launched concurrently and the Slack start path does not desynchronize Socket Mode connection startup.

## What This Changes

The issue requests Slack Socket Mode startup/reconnect hardening so many agent-specific Slack apps do not create synchronized pong-timeout storms that can crash the gateway.

## Best Possible Solution

Add a Slack-owned, abortable startup desynchronization before opening Socket Mode when more than one enabled Slack Socket Mode account is configured, while preserving the existing generic channel concurrency, ping-timeout tuning, and reconnect backoff. Cover it with focused Slack tests and document/changelog only if a user-facing config knob is added.

## Reproduction Assessment

Unclear for the full crash: reproducing SIGKILL needs multiple live Slack apps and host pressure. Yes for the triggering code path: current main still starts all accounts concurrently through Promise.all and has no Slack Socket Mode startup stagger.

## Solution Assessment

No, current main is only a partial mitigation for this report. The narrow maintainable fix is Slack-specific startup jitter for multi-account Socket Mode, not core connection pooling or a broad channel-manager concurrency change.
