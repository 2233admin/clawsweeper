## Summary

Keep this issue open. Current main still sends gateway-executed plugin message actions through an explicit 10,000 ms caller timeout, the CLI send path does not expose or pass a longer timeout for media uploads, and PR #74169 is still an open unmerged implementation candidate linked with closing syntax.

## What This Changes

The issue asks OpenClaw to send a roughly 10 MB file with `openclaw message send --channel wecom --media` without failing with `gateway timeout after 10000ms`.

## Best Possible Solution

Keep the issue open until PR #74169 is reviewed, revised, landed, or replaced with a generic core fix that gives gateway-executed plugin message sends a suitable timeout boundary and focused regression coverage. A targeted message-action or media-send timeout is safer than broadly raising every Gateway RPC timeout without checking hung-call behavior.

## Reproduction Assessment

Yes for the core timeout boundary: a gateway-executed plugin `send` action that takes longer than 10 seconds before responding will hit the explicit wrapper timeout. The reporter supplied concrete CLI steps; I did not live-test the external WeCom upload in this read-only review.

## Solution Assessment

No for the linked PR as described: changing the default in `src/gateway/call.ts` is unlikely to cover a path that passes `timeoutMs: 10_000` explicitly from `src/infra/outbound/message-action-runner.ts`. The better fix should address the message-action gateway caller timeout directly and add a regression test around gateway-executed media sends.
