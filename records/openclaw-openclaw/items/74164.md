## Summary

Keep open. Current main still routes CLI `message send` for gateway-executed plugin channels through an explicit 10,000 ms timeout, the latest release contains the same boundary, and the linked closing PR remains open rather than merged.

## What This Changes

The issue reports that sending a roughly 10 MB file through the WeCom channel via `openclaw message send --media` times out at 10,000 ms instead of completing.

## Best Possible Solution

Keep this issue open until #74169 is landed or replaced with a targeted fix for gateway-executed plugin message sends. The end state should remove or deliberately raise the explicit `message.action` timeout for media-capable sends, ideally with focused regression coverage for plugin sends that take longer than 10 seconds.

## Reproduction Assessment

Yes for the core timeout boundary. The reporter's CLI command plus current code shows gateway-executed plugin sends wait on `message.action` under an explicit 10,000 ms timeout; a delayed gateway plugin action or the reported 10 MB upload exercises the same failure path.

## Solution Assessment

No for the linked PR as described in the provided context, because changing only the generic gateway-call default would not affect this path while the message-action runner passes an explicit 10,000 ms timeout. A narrower maintainable fix should address the message-action/media-send timeout boundary or add an intentional config/CLI seam for it.
