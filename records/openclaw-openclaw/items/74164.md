## Summary

Keep this issue open. Current main still has the gateway-executed plugin message-action path bounded by an explicit 10,000 ms timeout, and the open linked PR #74169 is an implementation candidate rather than a reason to close this issue before it is merged or replaced.

## What This Changes

The issue requests that `openclaw message send --media` via the WeCom plugin send a roughly 10 MB file successfully instead of failing with `gateway timeout after 10000ms`.

## Best Possible Solution

Keep this issue open until PR #74169 is reviewed, revised, landed, or replaced with a generic core fix that gives gateway-executed plugin media sends enough time without broadly weakening unrelated Gateway RPC timeout behavior.

## Reproduction Assessment

Yes, for the core timeout boundary: the reported CLI command maps to `messageCommand` -> `runMessageAction` -> gateway-executed plugin `message.action`, and current source still applies a 10,000 ms wrapper timeout. A live WeCom upload was not run, but the code path matches the reported error string and timing.

## Solution Assessment

No, the currently linked PR direction is not clearly the best fix because changing the generic gateway default may not affect the explicit timeout passed by `message-action-runner.ts`. A narrower fix at the message-action gateway boundary, with regression coverage for media/plugin sends, is more maintainable.
