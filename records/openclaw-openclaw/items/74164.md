## Summary

Keep this issue open. Current main still routes Gateway-executed plugin message sends through an explicit 10,000 ms wrapper timeout, the CLI media-send path does not expose or pass a longer timeout, and the linked fix PR is still open rather than merged.

## What This Changes

The issue reports that `openclaw message send --channel wecom --media <about-10MB-file>` fails with `gateway timeout after 10000ms` instead of sending the file successfully.

## Best Possible Solution

Keep the issue open until the Gateway-executed plugin message-action timeout is fixed in the core send path, with focused coverage proving slow media/plugin sends can outlive the current 10-second wrapper without masking genuinely hung Gateway calls.

## Reproduction Assessment

Yes for the core timeout boundary: a Gateway-executed plugin `send` action whose `message.action` RPC takes longer than 10 seconds will use the explicit `10_000` ms wrapper timeout shown in current main. The exact live WeCom upload was not reproduced because it needs external service credentials and a real target.

## Solution Assessment

No current merged solution exists. The linked PR is the right place to review the fix, but the safer implementation likely needs to address the explicit message-action timeout boundary rather than only the generic Gateway call default.
