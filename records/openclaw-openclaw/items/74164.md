## Summary

Keep open. Current main still sends gateway-executed plugin message actions through an explicit 10,000 ms timeout, and the issue is already paired with open, unmerged PR #74169 rather than an implemented fix.

## What This Changes

The issue requests that `openclaw message send --channel wecom --media <large-file>` complete successfully instead of failing with `gateway timeout after 10000ms`.

## Best Possible Solution

Keep this issue open until the linked PR is revised and landed, or replaced with a targeted fix for gateway-executed plugin message sends. The fix should cover media sends that exceed 10 seconds and include focused regression coverage for the explicit message-action timeout path.

## Reproduction Assessment

Yes for the core timeout boundary. The provided CLI command plus current source show a WeCom gateway plugin send can wait on `message.action` under a fixed 10,000 ms caller timeout; a delayed plugin upload exercises the same failure mode.

## Solution Assessment

No for a fix that only changes the generic gateway-call fallback timeout, because `message-action-runner` currently passes an explicit `timeoutMs: 10000`. The safer solution is to adjust the message-action timeout boundary or add an intentional CLI/config seam for long-running plugin media sends.
