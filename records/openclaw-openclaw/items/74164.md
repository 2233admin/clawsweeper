## Summary

Keep open. Current main still has a 10,000 ms timeout on gateway-executed plugin message actions, the CLI media send path does not pass a longer timeout or expose a timeout flag, and the linked closing PR remains open rather than merged.

## What This Changes

The issue requests that large media sends through the WeCom channel complete successfully instead of failing with a gateway timeout after 10,000 ms.

## Best Possible Solution

Keep this issue open until #74169 is landed, revised, or replaced with a targeted generic fix for gateway-executed plugin message sends. The fix should address the explicit message.action timeout path for media sends without adding WeCom-specific core policy, and it should add focused regression coverage for a gateway plugin send that takes longer than 10 seconds.

## Reproduction Assessment

Yes for the core timeout boundary. The reporter's CLI command plus current main shows gateway-executed plugin sends wait on message.action under an explicit 10,000 ms timeout; a delayed gateway plugin action or the reported 10 MB upload exercises that path.

## Solution Assessment

No for a generic default-only timeout bump as the complete solution. The narrow maintainable fix should adjust the explicit message-action/media-send timeout boundary or provide an intentional scoped override, then prove gateway-executed plugin sends no longer inherit the 10-second limit.
