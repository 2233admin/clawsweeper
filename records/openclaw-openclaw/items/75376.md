## Summary

Keep open. Current main already loads full SKILL.md bodies on demand and caches/bounds/filters the skill catalog, but source, docs, and tests still show the available-skills metadata list is injected whenever eligible skills exist, so the requested semantic on-demand catalog injection remains unsolved.

## What This Changes

The issue requests semantic on-demand skill loading so ordinary chat turns can skip injecting and scanning the full available-skills catalog.

## Best Possible Solution

If maintainers want this direction, design an opt-in skill-catalog selection mode that preserves existing allowlist, metadata-gating, slash-command, snapshot invalidation, and prompt-cache contracts, with docs explaining the current allowlist/budget workaround and the new fallback behavior.

## Reproduction Assessment

Yes for the observable prompt behavior: current source and tests show eligible skills produce an `<available_skills>` prompt block and the system prompt tells the model to scan it. The exact 10-20% token/latency estimate was not reproduced by a benchmark in this read-only review.

## Solution Assessment

Unclear. The requested direction is narrower than loading every SKILL.md body, but semantic preselection is a core prompt/product behavior change and should be designed before implementation rather than queued as a mechanical cleanup.
