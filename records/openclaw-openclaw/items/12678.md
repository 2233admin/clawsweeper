## Summary

Current main still lacks the requested capability-manifest, explicit grant/revoke, pre-model default-deny filtering, central execution-boundary authorization, and durable high-risk audit model for skills/tools. The issue discussion already distinguishes existing name/profile-based tool policy and the @vainplex/openclaw-governance before_tool_call plugin from the requested core/plugin-API capability layer, and the latest maintainer comment says to keep this open.

## Best Possible Solution

Keep this open as a core/plugin-API security hardening item. A maintainable path is staged: add manifest/skill metadata for required and optional sensitive capabilities, persist grants and revocations, surface CLI/UI/doctor diagnostics, filter ungranted high-risk tools before model exposure, enforce the same decision at every execution boundary, and write durable audit records for high-risk decisions. Treat @vainplex/openclaw-governance as useful dynamic before_tool_call prior art layered after static grants, not as the canonical grant system.

## Risks / Open Questions

- Closing now would conflate adjacent name-based tool policies, owner-only gates, HTTP default denies, exec approvals, and before_tool_call governance with the requested static capability/grant/audit system.
- A future implementation needs one authorization decision reused by model-exposed tools, direct HTTP tools, MCP/loopback tools, native harness paths, and plugin-owned command paths so pre-exposure filtering is not bypassable.
- The linked @vainplex/openclaw-governance plugin is useful prior art for dynamic policy, but it cannot replace a core/plugin-API grant model because it runs after tools are already available rather than declaring and granting capabilities before exposure.
