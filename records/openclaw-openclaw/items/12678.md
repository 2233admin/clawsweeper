## Summary

Keep this issue open. Current main has meaningful adjacent controls, but it still does not implement the requested capability-declaration, grant/revoke, default-deny, central enforcement, and durable audit layer for skills/plugins/tools. The issue discussion also explicitly distinguishes the existing @vainplex/openclaw-governance before_tool_call plugin and related tool-policy PRs from the missing core/plugin-API grant model.

## Best Possible Solution

Keep this open as a core/plugin-API security hardening item. The best path is a staged design: add manifest/skill metadata for required and optional sensitive capabilities; add persisted grants/revocations with CLI/UI/doctor diagnostics; filter ungranted high-risk tools before model exposure; enforce the same decision at every execution boundary; and write durable audit records for high-risk capability decisions. Treat @vainplex/openclaw-governance as useful dynamic before_tool_call prior art that can layer on top of static grants, not as the canonical grant mechanism.

## Risks / Open Questions

- Closing now would conflate adjacent name-based tool policy, owner-only gating, sandboxing, Gateway RPC scopes, and before_tool_call governance with the requested capability manifest/grant/audit system.
- A future implementation needs one central authorization decision that covers model-exposed tools plus direct HTTP/RPC/MCP/non-LLM invocation paths, otherwise pre-exposure filtering could be bypassed.
- A default-deny rollout needs migration, diagnostics, and operator consent UX so existing trusted skills/plugins are not silently broken without a clear grant path.
