## Summary

Keep this issue open. Current main has meaningful adjacent controls: name-based tool allow/deny policy, sender/group policy, sandbox and exec approval knobs, owner-only gates, and plugin before_tool_call governance. It still does not implement the requested manifest-declared sensitive capability taxonomy, persisted grants/revocations, default-deny high-risk authorization before tool exposure, execution-boundary enforcement, or durable audit records for capability decisions.

## Best Possible Solution

Keep this open as a core/plugin-API security hardening item. A good implementation path is staged: add manifest and skill metadata for required/optional sensitive capabilities, add persisted grant/revoke state and diagnostics, filter ungranted high-risk tools before model exposure, enforce the same decision at execution boundaries, and record durable audit entries. Treat @vainplex/openclaw-governance as valuable dynamic before_tool_call prior art that can layer on top of static grants, not as the canonical grant mechanism.

## Risks / Open Questions

- Closing now would conflate adjacent name-based tool policy and hook-time governance with the requested manifest/grant/enforcement/audit permission model.
- A future default-deny rollout needs migration, diagnostics, and compatibility handling so existing trusted plugins and skills are not broken without a clear operator path.
- The implementation needs one central authorization model that covers model-facing tools and direct HTTP/RPC/MCP/non-LLM invocation paths, otherwise pre-exposure filtering could be bypassed.
