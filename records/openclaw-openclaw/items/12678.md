## Summary

Keep this issue open. Current main has meaningful adjacent controls, including name-based tool allow/deny policy, group/sender policy, trusted bundled pre-tool policies, normal before_tool_call governance, sandbox/elevated controls, and Gateway HTTP tool deny defaults. It still does not implement the requested manifest/skill-declared sensitive capability taxonomy, persisted grants/revocations, default-deny filtering before high-risk tool exposure, central execution-boundary authorization for all invocation paths, or durable capability audit records. The @vainplex/openclaw-governance plugin discussed in the thread is useful before_tool_call prior art, but the discussion correctly distinguishes it from the missing core/plugin-API grant layer.

## Best Possible Solution

Keep this open as a core/plugin-API security hardening item. A good staged path is: add manifest and skill metadata for required/optional sensitive capabilities, add persisted grant/revoke state and diagnostics, filter ungranted high-risk tools before model exposure, enforce the same decision at every tool/API invocation boundary, and record durable audit entries. Treat @vainplex/openclaw-governance as valuable dynamic before_tool_call prior art that can layer on top of static grants, not as the canonical grant mechanism.

## Risks / Open Questions

- Closing now would conflate adjacent name-based tool policy and hook-time governance with the requested capability manifest, grant, enforcement, and audit model.
- A future default-deny rollout needs migration, diagnostics, and compatibility handling so existing trusted skills/plugins do not break without an operator path.
- The implementation needs one central authorization model covering model-facing tools plus direct HTTP/RPC/MCP/non-LLM invocation paths, otherwise pre-exposure filtering could be bypassed.
