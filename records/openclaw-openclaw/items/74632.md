## Summary

Keep open. Current main still has no `envelope` field in the Gateway session create/patch schemas, no `SessionEntry` storage for a session-owned runtime policy, and no create/patch handler branch that persists or enforces one. The same author has open PR #74799 as the implementation candidate, so this issue should remain open while maintainers review that PR and decide the remaining network/callback scope.

## What This Changes

Requests additive Gateway protocol support for a session-owned runtime policy envelope on `sessions.create` and `sessions.patch`, with tool/path/network enforcement, audit events, and an optional external authorization callback.

## Best Possible Solution

Keep this issue as the design and acceptance tracker while reviewing PR #74799 for additive protocol/schema/storage/enforcement. If merged, close this as implemented or split any remaining network/callback work into narrower follow-up issues with explicit security-owner review.

## Reproduction Assessment

Yes. A Gateway RPC request that includes `envelope` on `sessions.create` or `sessions.patch` would fail the current AJV/TypeBox validation because both schemas enumerate accepted fields and set `additionalProperties: false`.

## Solution Assessment

Unclear. The narrow `sessions.create`/`sessions.patch` envelope is a plausible core protocol path, but network access and external callback semantics need a lower-layer fail-closed design rather than tool-input scanning alone.
