## Summary

Keep open. Current main still lacks a first-class per-session runtime envelope on the Gateway session RPCs: the typed schemas reject unknown `envelope` fields, the create/patch handlers only persist existing metadata/runtime override fields, and the session store type has no envelope field. The related open PR #74799 is the active implementation candidate and should be reviewed rather than sweeping this issue closed.

## What This Changes

Add an optional per-session runtime envelope to `sessions.create` and `sessions.patch`, persist it on sessions, enforce it for tool/path/network/bash scope, and optionally add a gateway-side external authorization callback.

## Best Possible Solution

Review and land PR #74799 or a successor that adds an additive protocol/store/runtime envelope with fail-closed enforcement before execution and after parameter rewrites, plus Gateway schema tests, runtime enforcement tests, and docs. Treat network access and external authorization callbacks as explicit security design follow-ups unless they can be enforced with a real sandbox/proxy boundary in the same change.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to call `sessions.create` or `sessions.patch` with `envelope`; current main validates against schemas with `additionalProperties: false` and no `envelope` property, so the field is rejected before persistence or enforcement.

## Solution Assessment

Unclear as a final design, but the core direction is appropriate. A first-class session envelope belongs in the Gateway protocol and session store; network access and external callback behavior should be reviewed separately because they define a security boundary that should not rely on best-effort string matching.
