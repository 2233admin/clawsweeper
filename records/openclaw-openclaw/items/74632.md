## Summary

Keep open. Current main still lacks a session-owned runtime envelope in the Gateway session RPC contract, and the same-author open PR #74799 is the active implementation candidate rather than a reason to close or queue duplicate work.

## What This Changes

The issue requests an additive per-session runtime policy envelope on `sessions.create` and `sessions.patch`, with persisted tool/path/network constraints enforced on all tool calls for that session.

## Best Possible Solution

Keep the issue open while maintainers review #74799 or replace it with an accepted narrower design. The durable end state should be an additive protocol field with typed schema/docs, persisted `SessionEntry` state, pre-execution enforcement that survives policy/hook/approval rewrites, focused create/patch/tool-blocking tests, and an explicit separate decision for network enforcement and external callbacks if they cannot be enforced reliably in the same layer.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: add `envelope` to a `sessions.create` or `sessions.patch` payload and current main rejects it because the TypeBox/AJV schemas omit the field and disable additional properties.

## Solution Assessment

Unclear. A session-owned envelope is a plausible narrow boundary for the reported gap, but the exact enforcement layer, audit event semantics, and network/callback scope need maintainer security review; #74799 is the active candidate to evaluate.
