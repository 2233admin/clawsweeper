## Summary

Keep this PR open. Current main has added a generic plugin-owned cleanup path for plugin-created sessions, but it still has timestamped memory-core dreaming narrative session keys, still uses that timestamped key as the idempotency key, and still lacks this PR's pre-run cleanup before narrative generation. The PR is substantive, but its gateway change now needs maintainer review because it narrows the current generic plugin-owner cleanup seam to a memory-core/dreaming-prefix special case.

## Best Possible Solution

Keep this PR open for a rebase onto current main. Preserve the generic `pluginRuntimeOwnerId` cleanup seam and request-scoped forced synthetic client behavior in gateway, move the stable session key, timestamped idempotency key, and pre-run cleanup behavior into `extensions/memory-core`, and add focused tests for stale-context cleanup, request-scoped fallback, wrong-plugin/wrong-session denial, and existing plugin-owned cleanup callers.

## Risks / Open Questions

- Merging the PR as-is may regress the newer generic plugin-owned cleanup contract by allowing synthetic admin only for memory-core dreaming sessions, leaving other plugin-owned cleanup callers back on the `operator.admin` failure path.
- The PR adds bundled plugin identity and a `dreaming-narrative-*` prefix rule to gateway core, which conflicts with the repository's extension-agnostic core boundary unless maintainers explicitly want that exception.
- Stable per-workspace/per-phase session keys should be reviewed for overlapping same-workspace dream runs and for behavior when pre-cleanup fails but the next run proceeds with reused context.
