## Summary

Keep open. Current main still has the core whole-file session metadata and transcript materialization paths reported here, although it also has useful mitigations such as store maintenance, cache reuse, bounded title preview reads, and limit-before-row-enrichment for session lists. The open session-storage cluster, especially #51946 plus #71227 and #72165, should be handled before closing this production memory/RSS report.

## What This Changes

The issue asks OpenClaw to replace synchronous whole-file session metadata and transcript reads with lazy, streamed, paginated, or bounded session-storage behavior so large gateway histories do not drive high RSS.

## Best Possible Solution

Keep this issue open as part of the session-storage performance cluster. The maintainable end state is to land or supersede the directory-backed/per-session metadata work for `sessions.json`, then add cursor-aware or streaming transcript readers and summary/index fields so history and list surfaces do not hydrate unrelated large files.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: use a large `sessions.json` and large JSONL transcript, then hit session load/update or history APIs; current main still synchronously parses/stringifies the full store and reads complete transcripts before limiting history responses.

## Solution Assessment

No single suggested item is sufficient by itself. The best path is staged: first remove whole-store metadata hot paths, then replace full-transcript readers with bounded cursor-aware readers while preserving migration, maintenance, and transcript compatibility.
