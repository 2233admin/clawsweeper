## Summary

Keep #69451 open. Current main has pruning, rotation, cache, and tail-preview mitigations, but the reported storage architecture problem remains: session metadata is still read/parsed/cloned and written as one `sessions.json`, and full transcript JSONL files are still materialized before limits/cursors are applied in history and memory-index paths. Related #71227 is adjacent latency/skillsSnapshot evidence, not a complete supersession of this broader memory/RSS issue.

## Best Possible Solution

Keep the issue open as a core session-storage performance item. The best fix is to remove hot-path full-file materialization: shard or replace `sessions.json` with per-session metadata or SQLite, store repeated large fields such as `skillsSnapshot` out of line, apply list/history limits before row/message materialization, stream or tail-read transcript history, and add large-store regression coverage. Link #71227 as the latency/skillsSnapshot subproblem rather than closing #69451 as duplicate.
