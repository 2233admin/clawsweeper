## Summary

Keep open. Current main still preserves the reported memory/performance shape: `sessions.json` is read and parsed as one monolithic object, update paths still re-read and stringify the whole store, and gateway history surfaces still materialize full transcript message arrays before applying limits or cursors. Recent main has useful mitigations for cache cloning, pruning, caps, and bounded previews, and open PR #51946 targets the directory-backed metadata-store side, but the central storage/history issue is not implemented on main.

## What This Changes

The issue asks OpenClaw to avoid reading and parsing entire session metadata and transcript files in gateway startup/history paths by moving toward lazy, streaming, sharded, or bounded session storage.

## Best Possible Solution

Keep this issue open and route it through the active session-storage performance cluster. Maintainers should review, land, or explicitly supersede #51946 for the directory-backed/per-session metadata side, then follow with a narrower transcript/history change that provides cursor-aware or streaming reads for `readSessionMessages` callers while preserving existing transcript compatibility, maintenance behavior, and migration safety.
