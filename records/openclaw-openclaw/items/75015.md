## Summary

Keep open for maintainer review. Current main still has the silent group/channel fallback without a warning, while this PR is a narrow one-file warning slice for the broader linked regression and I did not find a concrete blocking bug in the patch.

## What This Changes

The PR adds a createSubsystemLogger-based one-shot runtime warning in `source-reply-delivery-mode.ts` when a group/channel turn uses the default visible-reply mode because neither group nor global `visibleReplies` is configured.

## Best Possible Solution

Land the resolver-level warning if maintainers accept the wording and CI proof, while keeping #74876 open or split for the remaining documentation, config-editing, and migration-policy work.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `resolveSourceReplyDeliveryMode` with empty config and `ChatType` set to `group` or `channel`; current tests and dispatch coverage show it becomes `message_tool_only` and suppresses the automatic final reply.

## Solution Assessment

Yes for this PR's warning slice. Placing a one-shot warning in the shared resolver is narrow, channel-agnostic, and only fires when the affected fallback is actually used; it is not the full solution for the broader linked issue.
