## Summary

Keep open. The PR targets a plausible listener-retention problem, but the proposed one-line cleanup is not mergeable because it removes the Discord gateway emitter's only post-dispose `error` listener; current code and tests intentionally keep that listener to suppress late gateway errors that would otherwise become uncaught EventEmitter errors.

## What This Changes

The PR adds a single `emitter.off("error", onGatewayError)` call in `createDiscordGatewaySupervisor.dispose()` for Discord gateway supervisor cleanup.

## Best Possible Solution

Revise the Discord gateway supervisor cleanup so disposal releases heavy lifecycle state without leaving the emitter without an `error` sink, then cover both properties in focused gateway-supervisor and provider-cleanup tests.

## Reproduction Assessment

Yes. Applying the PR's added `off()` would leave no `error` listener after dispose; the current tests already reproduce the expected late-error path, and a local Node EventEmitter check confirms that a late `error` emission without a listener throws.

## Solution Assessment

No. Fully detaching the listener is not the narrowest maintainable fix because it trades a memory-retention issue for a process-crash path; a safer fix would swap to a lightweight post-dispose sink or equivalent cleanup design.
