## Summary

Current main still matches the report: post-update plugin sync loads tracked install records, updates ClawHub records without checking explicit plugin disablement, and turns any plugin update error into a `post-update-plugins` failure. The item should stay open for a targeted code/test fix.

## Best Possible Solution

Keep this open and add an effective-enable-state filter in the post-update plugin updater before ClawHub/npm/marketplace network calls. Explicitly disabled tracked plugins should be skipped or reported as non-fatal warnings, while enabled plugin update failures remain fatal. Add regression coverage for a disabled ClawHub install record returning HTTP 429 and update the CLI docs if the failure policy changes.

## Risks / Open Questions

- A disabled ClawHub plugin can still make a successful core update report failure and skip restart follow-up when the registry returns 429 or another transient error.
- A fix must preserve fail-closed behavior for enabled plugin updates, integrity drift, and real install failures rather than broadly downgrading all plugin-sync errors.
- The updater also migrates disabled plugin config entries when package ids change, so skip logic should preserve disabled-but-configured state and avoid losing install metadata.
