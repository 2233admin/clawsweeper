## Summary

Keep this PR open. Current main still does not implement the PR’s intended behavior: malformed own `required` values can still pass through the shared tool schema normalization path, and the Gemini cleaner still preserves nested non-array `required` values. The PR remains a focused implementation candidate for the Bailian GLM-5 schema rejection described in the linked report, but it needs rebase/adaptation to the current schema-normalization layout and a maintainer call on the Gemini key-copy hardening note.

## Best Possible Solution

Keep this PR open and request a rebase onto current main. The best fix is to strip own non-array `required` fields in the shared tool-parameter schema normalization path before provider-specific cleaning, preserve valid `required` arrays, add regression coverage for top-level OpenAI-compatible/Bailian-style schemas and nested Gemini schemas, and explicitly resolve whether Gemini dynamic-key hardening belongs in this PR or a follow-up.
