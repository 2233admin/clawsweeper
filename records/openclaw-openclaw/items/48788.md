## Summary

Keep this issue open. Current main still has only narrow/private filename handling in shared fetch code plus Feishu-local mojibake recovery; it does not yet provide the requested centralized, exported, multi-encoding Content-Disposition filename utility across channel adapters. Open PR #71517 remains the active partial implementation candidate and explicitly leaves broader channel migration as follow-up.

## Best Possible Solution

Keep this issue open. Review and either land or close PR #71517 on its merits, then use this issue to track the remaining central filename utility scope: exported decoder contract, RFC 5987 charset/language handling, Latin-1/Windows-1252 mojibake recovery, explicit GB18030/GBK/Shift-JIS/EUC-KR coverage, path-safe basename handling, tests, and migration of relevant channel download paths.
