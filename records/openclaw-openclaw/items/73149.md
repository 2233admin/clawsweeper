## Summary

Keep open. Current main covers the HEARTBEAT_OK heartbeat-ack subset and normal NO_REPLY silence, but the broader cron/system-event no-op sentinel suppression requested here is still absent, and open PR #73785 is explicitly paired with this issue but has not merged.

## What This Changes

Request to make cron, heartbeat, and system-event no-op replies silent at runtime so sentinel outputs such as HEARTBEAT_OK, NO_NEW_AUDIO, and SESSION_WATCHDOG_OK are not delivered visibly.

## Best Possible Solution

Land or replace a narrow core delivery-boundary fix that centralizes no-op sentinel detection for heartbeat/system-event runs, suppresses exact known no-op outputs, preserves meaningful exec or cron summaries, and adds regression coverage; structured noReply plumbing can remain a follow-up if sentinel matching is too limited.

## Reproduction Assessment

Yes. The discussion gives a concrete HEARTBEAT_OK delivery-gap trace, and current-main code inspection shows the remaining broader sentinel path by the absence of NO_NEW_AUDIO and SESSION_WATCHDOG_OK suppression.

## Solution Assessment

Yes for the immediate bug class. The narrowest maintainable fix is delivery-boundary suppression of exact no-op sentinels for heartbeat/system-event runs while keeping substantive summaries deliverable; a first-class structured noReply flag is useful but larger than needed to resolve this report.
