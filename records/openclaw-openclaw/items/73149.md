## Summary

Keep open. Current main has narrow HEARTBEAT_OK heartbeat-ack suppression and generic NO_REPLY suppression, but it still does not implement the broader job-defined no-op/silent delivery contract requested here; the issue also has an open, unmerged closing PR, so it should remain the canonical tracker until that PR or an equivalent fix lands.

## What This Changes

The issue requests a runtime-level silent or no-reply delivery mode for cron, heartbeat, and system-event no-op outputs so sentinel strings and short no-op summaries are not posted to user channels.

## Best Possible Solution

Land, revise, or replace PR #73785 with a narrow delivery-boundary fix that suppresses exact no-op sentinel outputs for heartbeat/system-event runs while preserving meaningful cron and exec summaries; a structured `noReply` or `suppressDelivery` result flag can remain a follow-up if maintainers want a broader contract.
