## Summary

Keep open. Current main covers the narrower `HEARTBEAT_OK` heartbeat-ack path and the existing `NO_REPLY` silent-token path, but it still lacks the broader cron/system-event no-op sentinel suppression requested here; open PR #73785 is explicitly paired with this issue and should be reviewed, landed, or replaced before the issue closes.

## What This Changes

The issue requests first-class cron/heartbeat/system-event no-op delivery semantics so successful sentinel replies such as `HEARTBEAT_OK`, `NO_NEW_AUDIO`, and `SESSION_WATCHDOG_OK` do not become visible channel messages.

## Best Possible Solution

Land or replace a narrow core fix that centralizes no-op sentinel suppression at the heartbeat/system-event delivery boundary, keeps substantive exec or cron summaries deliverable, and adds regression coverage for the requested sentinel cases; a structured `noReply` result flag can remain a follow-up if maintainers want a broader API.

## Solution Assessment

Yes for the immediate bug. Reviewing and landing PR #73785, or an equivalent focused replacement, is the narrowest maintainable path before considering a larger first-class structured no-reply API.
