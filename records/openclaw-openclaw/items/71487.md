## Summary

Keep open. Current main still has backend and CLI TTS capability, plus gateway RPCs, but no Control UI Settings consumer or first-class Settings panel for those controls, so the discoverability gap remains and is a bounded repair candidate.

## What This Changes

Add a Control UI Settings Text-to-Speech section with enable/disable, provider and default voice selection, and optional sample playback backed by existing TTS preferences and provider APIs.

## Best Possible Solution

Implement a dedicated Text-to-Speech panel in Control UI Settings that reads gateway TTS status/provider inventory, toggles the existing local auto preference, selects the preferred provider, persists a provider-appropriate default voice through a generic contract, and plays a short sample through provider-backed conversion.

## Reproduction Assessment

Yes. Source inspection reproduces the gap: Quick Settings props/rendering and app-render wiring have no TTS state or actions, and an exact ui/src search for TTS gateway method consumers returns no matches.

## Solution Assessment

Yes. A focused Settings panel over existing tts.status, tts.providers, tts.enable, tts.disable, tts.setProvider, and tts.convert is the narrowest maintainable direction; add only the minimal generic voice-list or voice-persistence RPC needed for provider-neutral default voice selection.
