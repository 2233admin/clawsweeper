## Summary

Keep this open. Current main still exposes TTS through CLI/runtime/gateway RPCs and generic config editing, but the Control UI Quick Settings path has no TTS props, rendered Settings card, or app wiring for status/provider/voice/sample playback. The existing Codex review comment reached the same conclusion, and current main does not close the requested UX gap.

## What This Changes

Add a first-class Control UI Settings surface for enabling/disabling TTS, choosing the default provider, choosing a provider-filtered default voice, and optionally playing a sample.

## Best Possible Solution

Ship a discoverable Text-to-Speech panel in Control UI Settings that uses the existing gateway TTS status/provider/mutation/sample paths, adds a small voice-list/default-voice persistence seam if needed, keeps provider-specific details behind registry/runtime contracts, and covers the flow with focused UI and gateway tests.

## Repair Work Prompt

Implement the requested first-class Text-to-Speech section in Control UI Settings for this issue.

Expected boundary:
- Add a discoverable TTS panel/card in Quick Settings or another obvious Control UI Settings surface.
- Read current state from gateway TTS methods, at minimum `tts.status` and `tts.providers`.
- Provide an enable/disable toggle wired to `tts.enable` / `tts.disable`.
- Provide a provider selector wired to `tts.setProvider`, showing configured vs unavailable providers from gateway data.
- Provide a provider-filtered voice selector. Prefer an existing safe gateway-accessible voice source; if needed, add a small additive gateway method that wraps `listSpeechVoices` without exposing provider secrets to the browser.
- Persist a selected default voice through the agreed TTS preference/config contract. Be explicit in code/tests whether this writes local TTS prefs or `messages.tts.providers.<provider>` config.
- Add a sample/play action using `tts.convert` with the selected provider/voice and authenticated media playback.
- Keep core extension-agnostic: do not hardcode bundled provider ids/defaults in UI/core beyond provider registry data returned by the gateway.
- Do not change provider-specific voice defaults, provider auth behavior, or unrelated Talk/realtime voice controls.
- Update docs/changelog for the new user-facing Settings UI; add a changelog entry crediting `@wen0531` if this repo's active changelog policy applies.
