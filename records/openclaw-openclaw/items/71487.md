## Summary

Keep open. Current main still has TTS backend RPCs, CLI/docs, provider status, and provider voice metadata, but the Control UI Settings surfaces do not expose a first-class TTS enable toggle, provider selector, default voice picker, or sample playback path.

## What This Changes

Add a first-class TTS section in Control UI Settings with an enable toggle, default provider selector, provider-filtered voice picker, and optional sample playback.

## Best Possible Solution

Implement a focused Control UI Text-to-Speech settings panel that reads tts.status/providers, toggles TTS with tts.enable/disable, changes provider with tts.setProvider, exposes provider-filtered voices through the safest available voice inventory path, persists the selected default voice through an agreed TTS preference/config contract, and plays a short provider-backed sample through tts.convert.

## Reproduction Assessment

Not applicable as a bug reproduction, but the missing feature is high-confidence to verify: inspect the current Control UI Settings source and run the targeted ui/src TTS RPC search. Current main has backend TTS RPCs and docs, but no Settings UI controls for them.

## Solution Assessment

Yes, a first-class Control UI Settings panel backed by existing TTS RPCs plus a narrow default-voice persistence seam is the maintainable direction. The generic config editor and /tts command path do not solve the discoverability problem described here.
