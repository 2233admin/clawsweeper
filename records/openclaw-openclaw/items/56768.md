## Summary

Keep open. Current main still prints a bare Control UI URL from `openclaw gateway status`, while `openclaw dashboard` has separate token-fragment handling for clipboard/browser use. The PR remains a plausible implementation candidate for the #45471 token_missing UX, but it needs a current-main rework and security review because main now has explicit tests that dashboard tokens must not be logged in terminal output.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rebase/redesign against current main. The safest path is to share token-aware dashboard URL resolution without logging secrets, preserve SecretRef non-embedding, pass the runtime service port, avoid token resolution when Control UI is disabled, add gather/print JSON redaction tests, and either update the changelog attribution or choose a product decision that `gateway status` should stay non-tokenized and direct users to `openclaw dashboard`.
