## Summary

Keep open. Current main still rebuilds the Control UI session dropdown from `sessions.list` but then unconditionally adds the currently selected `sessionKey`, so a deleted active direct session can still appear even after the Gateway response no longer includes it. The prior targeted fix PR #48190 was closed unmerged.

## Best Possible Solution

Land a focused Control UI fix that treats `sessions.list` as authoritative for normal direct sessions: after a fresh sessions refresh, stop rendering a missing active direct session and normalize the active selection/settings to a valid fallback such as the Gateway main/default session. Preserve the current missing-key fallback for ephemeral subagent and cron sessions, and add regression coverage for both cases.
