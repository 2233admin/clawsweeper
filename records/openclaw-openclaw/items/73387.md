## Summary

Keep this issue open. Current main still implements Discord guild `channels` as an implicit allowlist: an unmatched channel under a non-empty guild `channels` map resolves to `allowed: false`, and the active preflight path drops that message even when `groupPolicy` is `open`. Docs and regression coverage still describe/assert the current deny behavior, and the linked related items are adjacent rather than a canonical replacement for this narrower semantic change.

## Best Possible Solution

Keep open for bundled Discord plugin work. Update Discord channel-policy resolution so `guilds.*.channels` entries act as additive per-channel overrides when `channels.discord.groupPolicy` is `open`, with unlisted channels inheriting guild-level defaults. Preserve deny-by-default semantics for `groupPolicy: "allowlist"`, keep explicit `enabled: false` channel entries blocking, document precedence, and add regression tests for listed overrides, unlisted inherited channels, explicit denial, and the relevant Discord command/reaction surfaces.

## Risks / Open Questions

- Changing this behavior can widen Discord guild access for existing configs that rely on `guilds.*.channels` as an implicit allowlist, so the fix needs an explicit compatibility/migration decision.
- The resolver is reused by message preflight, reactions, native commands, and agent-component authorization, so a partial fix could leave inconsistent Discord surfaces.
