## Summary

Keep this issue open. Current main still implements the documented Discord behavior the report is asking to change: once a guild has a non-empty `channels` map, unmatched guild channels resolve to a denied channel config, and the Discord routing paths then drop those messages or commands even when the account-level `groupPolicy` is `open`. The related open items are broader or adjacent, not a clean canonical replacement for this narrower additive-overrides request.

## Best Possible Solution

Keep open for bundled Discord plugin work. Update Discord channel-policy resolution so `guilds.*.channels` entries are additive per-channel overrides when `channels.discord.groupPolicy` is `"open"`: unmatched channels should inherit guild-level defaults such as `requireMention`, sender restrictions, tools, skills, and prompts. Preserve implicit allowlist semantics for `groupPolicy: "allowlist"`, keep explicit `enabled: false` denies blocking, document the precedence rules, and add regression tests for listed overrides, unlisted inherited channels, and explicit denial across the main Discord ingress paths.

## Risks / Open Questions

- Changing this behavior can widen Discord guild access for existing configs that relied on a non-empty `guilds.*.channels` map as an implicit allowlist, so maintainers need an explicit compatibility and migration decision.
- The implementation needs to preserve current `groupPolicy: "allowlist"` behavior and explicit channel denies such as `enabled: false` while allowing unmatched channels to inherit guild defaults only in open-mode routing.
- Because several Discord entry points consume the same resolved channel config, the fix should cover message preflight, reactions/listeners, native commands/components, and voice ingress together or document any intentional differences.
