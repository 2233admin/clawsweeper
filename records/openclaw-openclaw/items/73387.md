## Summary

Keep open. Current main and the latest tagged release still treat a non-empty Discord guild `channels` map as an implicit channel allowlist, so the requested additive override behavior is not implemented; the change is valid but needs maintainer compatibility and access-control review before an automated fix should widen message processing.

## What This Changes

The issue asks for Discord guild `channels` entries to behave as additive per-channel overrides so unlisted channels inherit guild-level `groupPolicy`, `requireMention`, and sender defaults instead of being denied.

## Best Possible Solution

Keep this as bundled Discord plugin work and define compatible semantics explicitly: unlisted channels should inherit guild defaults for open-guild override use cases, while explicit `enabled: false`, wildcard/parent matches, and `groupPolicy: "allowlist"` denial behavior remain intentional and documented.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves that channel as `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear until maintainer review. Additive overrides are a plausible narrow product fix, but the safest implementation should preserve explicit-deny and allowlist-mode behavior or introduce an explicit compatibility mode instead of globally removing the fallback deny.
