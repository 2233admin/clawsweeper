## Summary

Keep open. Current main still implements and documents Discord guild `channels` as an implicit channel allowlist, and the related items are either broader open feature requests or closed unmerged PRs rather than a merged canonical fix.

## What This Changes

The issue requests changing Discord guild `channels` semantics so channel entries act as additive per-channel overrides while omitted channels inherit guild-level defaults.

## Best Possible Solution

Implement this in the bundled Discord plugin with a documented compatibility boundary: omitted channels should inherit guild defaults for additive override use cases, while explicit channel disables and strict allowlist-mode access remain enforceable across message, native command, reaction, component, and voice ingress paths.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves that channel to `{ allowed: false }` and the preflight path drops it.

## Solution Assessment

Unclear until maintainer compatibility review. The additive override model is the right direction for the reported open-guild use case, but the safest implementation should distinguish override-only entries from strict channel allowlisting rather than globally removing fallback-deny semantics.
