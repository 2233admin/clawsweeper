## Summary

Keep open. Current main still resolves unmatched Discord guild channel entries to a denied channel config and the preflight path still drops those messages, so the requested additive override behavior is not implemented; the related open items are broader and the exact older PRs/issues are closed or unmerged rather than canonical fixes.

## What This Changes

The issue requests changing Discord guild `channels` entries from an implicit allowlist into additive per-channel overrides so unlisted channels inherit guild-level defaults, especially under `groupPolicy: "open"`.

## Best Possible Solution

Keep this in the bundled Discord plugin and define explicit compatibility semantics: unlisted channels should inherit guild defaults for open-guild override use cases, while explicit `enabled: false` channel disables and strict allowlist-mode denial remain preserved, with docs and regression tests updated across message preflight, voice, reactions, and component/native command auth.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the channel to `{ allowed: false }` and then preflight drops it.

## Solution Assessment

Unclear. Additive inheritance is the narrow behavior for the reported `groupPolicy: "open"` case, but applying it globally would be a security-sensitive compatibility change; the safer direction is to preserve explicit-deny and allowlist-mode semantics unless maintainers approve a broader migration.
