## Summary

Keep open. Current main still treats a non-empty Discord guild `channels` map as an implicit channel allowlist, and the requested additive-override behavior remains blocked across message preflight and adjacent Discord access paths. Prior related PRs attempted similar semantics but are closed unmerged, while the open related issues are broader or pattern-focused rather than canonical fixes for this exact behavior.

## What This Changes

The issue requests changing Discord guild `channels` config semantics so entries act as additive per-channel overrides instead of making every unlisted channel implicitly denied.

## Best Possible Solution

Keep this as bundled Discord plugin work: represent unmatched channels distinctly from explicit denies, let unlisted channels inherit guild defaults for the `groupPolicy: "open"` additive-override use case, preserve explicit disabled and allowlist-mode denial semantics, and update Discord docs plus focused regression tests across preflight, native commands/components, and voice access.

## Reproduction Assessment

Yes. A source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves the channel to `{ allowed: false }` and preflight drops the message.

## Solution Assessment

Unclear until maintainer compatibility review. The additive-override model is the right user-facing direction for the reported open-guild use case, but the safe implementation should be scoped so it does not globally remove allowlist or explicit-deny behavior.
