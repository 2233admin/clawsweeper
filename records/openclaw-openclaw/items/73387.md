## Summary

Keep open. Current main still implements the Discord guild `channels` map as an implicit channel allowlist: an unlisted channel resolves to `allowed: false`, and the preflight path drops the message before guild-level `requireMention` defaults can apply. The related items are adjacent broader access-policy or pattern-matching requests, not a merged canonical fix for this narrower behavior.

## What This Changes

The issue requests Discord guild `channels` entries to behave as additive per-channel overrides so unlisted channels inherit guild-level defaults instead of being denied.

## Best Possible Solution

Keep this as bundled Discord plugin work: add an explicit additive-override path for open guild routing where unlisted channels inherit guild defaults, while preserving explicit channel disables and existing allowlist-mode denial semantics, then update Discord docs and focused regression coverage for listed overrides, unlisted inheritance, wildcard entries, and explicit denial.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Discord guild config with `groupPolicy: "open"`, guild-level `requireMention: false`, and a non-empty `guilds.*.channels` map that omits the incoming channel; current main resolves that channel to `{ allowed: false }` and preflight drops it.

## Solution Assessment

Unclear until maintainer compatibility review. The additive override model is a plausible narrow fix for open guild routing, but the safest implementation needs an explicit security stance so existing allowlist and explicit-deny configurations do not loosen unexpectedly.
