## Summary

Keep open. Current main still has the restart-sensitive shape reported here: the iMessage monitor creates a fresh in-memory sent-message cache on startup, subscribes to `watch.subscribe` without a rowid/cursor fence, and the current self-chat regression tests intentionally dispatch an `is_from_me=true` self-chat message when the echo cache is empty. Related #62761 should be handled with the same cursor/startup-fence fix, but this narrower self-message echo-loop report is still valid.

## What This Changes

Prevent legacy iMessage gateway restarts from treating self-authored Messages DB rows as fresh inbound messages when the in-memory echo cache is empty.

## Best Possible Solution

Add a legacy iMessage per-account restart fence/cursor in the plugin: record or persist a safe rowid high-water mark, pass the resume point to `watch.subscribe` when supported, skip already-existing startup rows, and update the cursor only after messages are safely handled or intentionally dropped. Preserve the current real self-chat behavior by applying the fence to startup/replay rows rather than blanket-dropping all `is_from_me` self-chat traffic.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by constructing an `is_from_me=true` self-chat payload with matching `sender`, `chat_identifier`, and `destination_caller_id` plus an empty echo cache; current tests assert that this dispatches, and the monitor creates that empty cache on every restart.

## Solution Assessment

Yes, the best direction is the report's startup rowid fence expanded into the cursor/high-water path already needed by #62761. Persisting only the echo cache is heavier and less complete, while dropping all `is_from_me` self-chat messages would conflict with existing self-chat behavior.
