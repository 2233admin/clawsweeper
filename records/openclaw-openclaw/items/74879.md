## Summary

Keep open. Current main still runs shell-completion cache generation during onboarding, and that path still uses a synchronous 30-second `completion --write-state` subprocess without the plugin-skip environment used by the update path, so the reported slowdown remains plausible and actionable.

## What This Changes

The issue reports a regression where onboarding and shell completion are much slower in 2026.4.26+ than in 2026.4.14 on Debian-based Docker installs.

## Best Possible Solution

Keep automatic onboarding completion work bounded or off the critical path: use the existing core-only completion-cache path for automatic onboarding, or defer full plugin-command cache generation until after onboarding while preserving explicit full plugin completions for manual `openclaw completion --write-state` runs.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart with no completion profile reaches `setupWizardShellCompletion`, calls `ensureCompletionCacheExists`, and spawns `completion --write-state` with a 30-second synchronous timeout. The exact reported Docker timing was not live-measured.

## Solution Assessment

No for current main; it still keeps completion-cache generation on the onboarding path. The adjacent open PR #72910 may be a partial timeout fix, but the narrower maintainable fix is to avoid full plugin-command completion generation during automatic onboarding.
