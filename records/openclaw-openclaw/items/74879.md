## Summary

Keep open. Current main and the latest release still put completion-cache generation on the onboarding path, and the shared doctor/onboarding helper still runs a synchronous full `openclaw completion --write-state` with a 30-second timeout. The related open PR #72910 is relevant but not merged and appears to extend the timeout/progress behavior rather than remove full completion generation from the onboarding critical path.

## What This Changes

The issue reports a regression where Docker onboarding and shell completion became much slower in OpenClaw 2026.4.26+ than in 2026.4.14.

## Best Possible Solution

Keep the automatic onboarding and doctor paths bounded and fast: use the existing skip-plugin/core-only completion refresh precedent or defer full plugin-command cache generation outside the onboarding critical path, while preserving explicit full `openclaw completion --write-state` behavior for manual user runs. Coordinate this with #72910 so timeout/progress changes and the onboarding fix do not compete.

## Reproduction Assessment

Yes for the code path: a fresh QuickStart run with no installed completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists`, which synchronously spawns `openclaw completion --write-state` through the doctor helper with a 30-second timeout. The exact reported multi-minute Docker timing was not live-verified here.

## Solution Assessment

Unclear that the adjacent open PR is the complete solution. Raising the timeout can help slow-but-successful cache writes, but the narrower maintainable onboarding fix is to use a skip-plugin/core-only automatic cache refresh or defer full plugin completion generation off the critical path.
