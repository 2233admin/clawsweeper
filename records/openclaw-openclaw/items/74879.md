## Summary

Keep open: current main and the latest release still run synchronous completion-cache generation from onboarding, and that path still uses the full `completion --write-state` command with a 30-second bound. The related PR is open and overlapping, but it is not a merged fix for the onboarding path.

## What This Changes

The issue reports a regression where OpenClaw 2026.4.26+ onboarding on Docker/umbrelOS takes minutes and shell completion often times out, whereas 2026.4.14 completed quickly.

## Best Possible Solution

Keep automatic onboarding completion work off the slow full plugin-command path: use the existing core-only cache generation mode or defer full plugin-command cache generation outside onboarding, while preserving explicit full `openclaw completion --write-state` for manual refreshes.

## Reproduction Assessment

Yes for a source-level reproduction path: a fresh QuickStart with no completion profile reaches `setupWizardShellCompletion`, then `ensureCompletionCacheExists`, then the synchronous 30-second `completion --write-state` subprocess. The exact reported Docker wall-clock timing remains unmeasured here.

## Solution Assessment

Unclear for the adjacent PR: increasing the timeout and progress output may reduce ETIMEDOUT reports, but the narrower maintainable solution for this onboarding bug is to avoid full plugin-command completion generation on the automatic onboarding path.
