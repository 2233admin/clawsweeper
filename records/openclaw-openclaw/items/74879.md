## Summary

Keep open. Current main still routes QuickStart onboarding through automatic shell-completion cache generation, and that shared doctor/onboarding helper still runs `openclaw completion --write-state` synchronously with a 30-second timeout while preserving normal plugin-command discovery; the related open PR #72910 only raises the timeout and does not make onboarding fast or core-only.

## What This Changes

The issue reports a recent regression where Docker onboarding on 2026.4.26+ takes minutes and shell completion often times out around 30 seconds, instead of completing within a few seconds as in 2026.4.14.

## Best Possible Solution

Automatic onboarding and doctor health work should stay bounded and core-only: reuse or centralize the update path's skip-plugin completion refresh for automatic setup paths, preserve explicit full plugin-command cache generation for manual `openclaw completion --write-state`, and coordinate that with #72910 so the timeout/progress change and fast-onboarding fix do not compete.

## Reproduction Assessment

Yes for the code path: a fresh QuickStart run with no completion profile reaches `setupWizardShellCompletion`, which synchronously generates the completion cache through the 30-second doctor helper. The exact multi-minute Docker timing is still not live-verified from this checkout.

## Solution Assessment

Unclear for the related open PR as the complete solution: raising the timeout helps slow-but-successful cache writes, but it does not make automatic onboarding fast. The narrower maintainable fix is to use a skip-plugin/core-only cache refresh or defer full plugin completion generation outside the onboarding critical path.
