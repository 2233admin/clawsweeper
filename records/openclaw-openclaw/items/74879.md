## Summary

Keep open: current `main` and the latest release still put fresh onboarding on the synchronous shell-completion cache path, and that path still runs full `completion --write-state` with the 30-second timeout and plugin command registration enabled by default. The related PR #72910 is relevant to completion-cache timeouts, but its stated scope is update and doctor refresh behavior, so it does not prove the onboarding slowdown is fixed.

## What This Changes

The issue reports a regression where onboarding and shell-completion setup became much slower in versions 2026.4.26 and newer compared with 2026.4.14.

## Best Possible Solution

Keep full plugin-command completion generation for explicit `openclaw completion --write-state`, but move automatic onboarding cache refresh to a bounded core-only path or defer full cache generation outside the onboarding finish path, with focused regression coverage for QuickStart and existing completion-cache behavior.

## Reproduction Assessment

Yes for a high-confidence source-level reproduction path: a fresh QuickStart with no completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists`, which synchronously spawns `completion --write-state` with a 30-second timeout. The exact Docker/umbrelOS runtime duration still needs live timing if maintainers want platform-specific proof.

## Solution Assessment

Unclear because this issue has no patch and the linked PR appears to solve only a neighboring timeout/progress problem. Raising the timeout may reduce `ETIMEDOUT`, but the narrower maintainable fix for onboarding speed is to avoid or defer full plugin-command completion generation on the automatic onboarding path.
