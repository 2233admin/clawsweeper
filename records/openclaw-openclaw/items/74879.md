## Summary

Keep open. Current main still runs shell-completion cache generation during onboarding, and that helper still uses a synchronous 30-second `completion --write-state` subprocess without the plugin-skip environment used by the update path. The related open PR #72910 covers the same completion-cache timeout area, but it is not merged into current main and may not remove onboarding's slow critical path.

## What This Changes

The issue reports a regression where OpenClaw 2026.4.26 and newer take minutes to complete onboarding and often time out shell completion on a Docker-based umbrelOS install, compared with 2026.4.14.

## Best Possible Solution

Keep automatic onboarding completion work bounded: use a core-only completion-cache path or defer full plugin-command cache generation outside onboarding, while preserving explicit full plugin completions for manual `openclaw completion --write-state` runs.

## Reproduction Assessment

Yes for the source path: a fresh QuickStart with no shell completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists` and then the synchronous 30-second completion-cache spawn. The exact reported Docker timing was not live-measured in this read-only review.

## Solution Assessment

Unclear that #72910 is the best complete fix. Raising the timeout may reduce `ETIMEDOUT` on slow machines, but the narrower maintainable fix for onboarding speed is to avoid full plugin-command completion generation on the automatic onboarding path.
