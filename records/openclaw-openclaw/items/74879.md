## Summary

Keep open: current main still puts fresh QuickStart onboarding on the synchronous completion-cache generation path, and that helper still runs the full `completion --write-state` subprocess with the 30-second timeout and plugin command loading enabled by default. The related open PR is relevant but appears partial for this onboarding-specific regression, so this needs maintainer coordination rather than cleanup closure.

## What This Changes

The issue asks to restore fast onboarding and shell-completion responsiveness after a regression reported between 2026.4.14 and 2026.4.26+ on Docker/umbrelOS.

## Best Possible Solution

Keep full plugin-command completion generation for explicit `openclaw completion --write-state`, but make automatic onboarding use a bounded core-only cache path or defer full plugin-command cache generation until after onboarding completes.

## Reproduction Assessment

Yes for a source-level reproduction path: fresh QuickStart with no installed completion reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists` and synchronously spawns `completion --write-state`. The exact Docker/umbrelOS wall-clock duration still needs live timing proof.

## Solution Assessment

Unclear because this issue has no patch and the related PR appears to address timeout/progress symptoms rather than the onboarding blocking path. Raising the timeout may reduce `ETIMEDOUT`, but avoiding or deferring full plugin-command generation is the narrower onboarding-speed fix.
