## Summary

Keep open. Current main still puts shell-completion cache generation on the onboarding path, and the helper used there still runs a synchronous 30-second `completion --write-state` subprocess without the plugin-skip environment used by the update path; the adjacent open PR #72910 is related but not merged and does not clearly remove onboarding's slow critical path.

## What This Changes

The issue reports that onboarding and shell-completion setup became much slower in recent Docker-based OpenClaw releases, with 2026.4.14 as the last known good version and 2026.4.26+ as the reported regression range.

## Best Possible Solution

Keep onboarding and doctor completion work bounded: use the existing core-only completion-cache path or defer full plugin-command cache generation outside automatic onboarding, while preserving explicit full plugin completions for manual `openclaw completion --write-state` runs.

## Reproduction Assessment

Yes for the source path: a fresh QuickStart with no shell completion profile reaches `setupWizardShellCompletion`, which calls `ensureCompletionCacheExists` and then the synchronous 30-second completion-cache spawn. The reported Docker timing was not live-measured in this read-only review.

## Solution Assessment

Unclear that the adjacent open PR is the best complete fix. Increasing the timeout can reduce `ETIMEDOUT` on slow machines, but the narrower maintainable fix for onboarding speed is to avoid full plugin-command completion generation on the automatic onboarding path.
