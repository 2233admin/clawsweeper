## Summary

Current main still has the macOS browser-proxy auth gap this PR targets: local `gateway.auth.token` and `gateway.auth.password` are only read as strings, while the browser proxy consumes that local config. The PR remains relevant, but it adds a security-sensitive `/usr/bin/security` keychain subprocess path and should stay open for maintainer review before merge.

## What This Changes

This PR updates the macOS `GatewayEndpointStore` local auth resolution to accept plain strings or keychain-backed `source: exec` SecretRef dictionaries for `gateway.auth.token` and `gateway.auth.password`, injects a keychain runner, adjusts async endpoint refresh, and adds focused `GatewayEndpointStoreTests` including `localConfig()` coverage.

## Best Possible Solution

Keep this PR open for maintainer review and land it, or land an equivalent patch. The fix should remain scoped to the macOS app `GatewayEndpointStore` local auth path, resolve supported local `gateway.auth.token` and `gateway.auth.password` keychain SecretRefs through a small injected resolver seam, preserve env and launchd precedence, keep remote auth behavior unchanged unless separately designed, cover both helper and `localConfig()` browser-proxy paths, and bound/review the keychain subprocess behavior.

## Risks / Open Questions

- The PR touches a credential-resolution path and adds local subprocess execution via `/usr/bin/security`; that should be explicitly reviewed for timeout, fallback, and keychain prompt behavior before merge.
- The final PR diff still has `process.waitUntilExit()` without a bounded timeout; `localConfig()` uses the sync resolver for the browser-proxy path, so a stuck keychain command could block endpoint resolution.
- OpenClaw's documented SecretRef runtime model generally resolves secrets eagerly into an in-memory snapshot rather than lazily on request paths, so the app-local resolver seam is a product/security exception worth maintainer approval.
