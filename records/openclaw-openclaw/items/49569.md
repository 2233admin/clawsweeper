## Summary

Keep this PR open. Current main still lacks 429 retry_after handling for Telegram command-menu sync, so the intended bug fix is not implemented on main. The PR is useful, but the live head is not land-ready: it misses structured grammY retry_after parsing, appears based on an older persistent command-hash cache shape that differs from current main, and one live check run is failing.

## Best Possible Solution

Keep this PR open, but require a rebase onto current main and a narrower implementation that preserves or explicitly revisits the current process-local cache decision. The Telegram fix should parse grammY parameters.retry_after first, include wrapped response/error.parameters shapes or reuse the existing Telegram error helper shape handling, keep bounded retry behavior clear, add regression tests for structured retry_after and description fallback, and rerun the failing check before maintainers consider landing it.
