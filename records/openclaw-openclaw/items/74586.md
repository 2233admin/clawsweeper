## Summary

Keep this open. The report is fresh, has a detailed transcript-based reproduction plus an independent confirmation, and current main still shows the same Active Memory watchdog signal being propagated into embedded tool execution, the same `Aborted` tool wrapper behavior, and the same timeout-result path for empty or `NONE` output.

## What This Changes

Active Memory should stop prematurely aborting embedded `memory_search` calls, classify completed `NONE` recall runs as empty instead of timeout, and prevent prior prompt-error entries from leaking into later recall prompts.

## Best Possible Solution

Keep Active Memory's bounded pre-reply deadline, but separate deadline enforcement from normal embedded memory tool dispatch until the deadline actually expires; then classify a completed `NONE` recall as `empty`, preserve real partial summaries only for true timeouts, and filter stale `openclaw:prompt-error` custom entries out of fresh recall prompts.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path using Active Memory with persisted transcripts plus a direct `memory_search` control, and a second user confirmed the same behavior on a nearby release/configuration.

## Solution Assessment

Unclear because there is no patch yet. The narrow maintainable fix should live in the Active Memory watchdog/embedded-runner boundary and prompt-error replay filtering, not in model tuning or a generic timeout-doctor workaround.

Preserve the hard Active Memory deadline, but do not let the watchdog signal prematurely abort normal embedded `memory_search`/`memory_recall`/`memory_get` tool calls before the deadline actually fires.
