## Summary

Keep open. Current main still has source paths matching the reported Active Memory embedded-run abort, timeout classification, and prompt-error carryover behavior, and the issue has persisted-transcript evidence plus an independent confirmation.

## What This Changes

Active Memory should not abort embedded memory recall tool calls, should classify completed `NONE` recall as empty rather than timeout, and should prevent prior prompt-error metadata from leaking into later recall context.

## Best Possible Solution

Keep Active Memory bounded by a hard outer deadline, but separate deadline enforcement from ordinary memory tool dispatch until the deadline actually wins; classify completed empty or `NONE` recall as `empty`; filter `openclaw:prompt-error` custom entries from future LLM-boundary context; and cover these paths with focused Active Memory and embedded-runner tests.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path using Active Memory with persisted transcripts plus a direct `memory_search` control, and a second user independently confirmed the same behavior on v2026.4.27.

## Solution Assessment

Unclear because there is no proposed patch on this issue. The narrow maintainable direction is in the Active Memory embedded recall and embedded-runner abort/prompt lifecycle, not in model tuning or a generic timeout-doctor workaround.
