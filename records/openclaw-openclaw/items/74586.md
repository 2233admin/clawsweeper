## Summary

Keep open. The report is fresh, well evidenced, and current main still has the same risk shape: Active Memory forwards one watchdog abort signal into the embedded runner and tool wrapper, the wrapper can surface `Aborted` tool results, the timeout recovery path still maps empty or `NONE` partial output to `status=timeout`, and prompt-error custom entries are persisted without a matching LLM-boundary filter for that custom type.

## What This Changes

Active Memory embedded recall should stop aborting embedded `memory_search` calls prematurely, classify completed empty or `NONE` recalls correctly, and prevent stale prompt-error metadata from leaking into subsequent recall prompts.

## Best Possible Solution

Keep Active Memory's hard wall-clock deadline, but separate deadline enforcement from ordinary memory tool execution until the deadline actually expires; treat a completed `NONE` recall as a clean empty result, and filter prior prompt-error custom metadata out of fresh Active Memory recall prompts.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with Active Memory enabled, persisted transcripts, direct chat, and direct `memory_search` as a control, and another user confirmed the same behavior on v2026.4.27.

## Solution Assessment

Unclear because there is no proposed patch yet. Current main is not the complete solution: the watchdog signal still reaches the embedded tool wrapper, empty or `NONE` timeout recovery still records `status=timeout`, and prompt-error filtering is not specific enough for the reported leak.
