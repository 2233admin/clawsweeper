## Summary

Keep open. Current main has partial Active Memory timeout recovery, but the source still leaves the reported embedded abort/timeout classification and stale prompt-error paths plausible, and the issue has a fresh independent confirmation on the latest stable release.

## What This Changes

The issue reports Active Memory embedded recalls aborting memory_search tool calls, returning status=timeout with no summary, misclassifying completed NONE responses, and carrying prior prompt-error entries into later recall runs.

## Best Possible Solution

Keep Active Memory's hard deadline, but separate deadline enforcement from normal memory tool execution until the deadline actually expires, classify a completed NONE recall as a clean empty result, and prevent prior openclaw:prompt-error custom entries from entering a fresh Active Memory recall prompt.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction path with Active Memory enabled, persisted transcripts, direct chat, and direct memory_search as a control; a second user confirmed the same behavior on v2026.4.27.

## Solution Assessment

No current-main solution is sufficient yet. The existing partial-timeout recovery can salvage non-empty transcript text, but it does not fully address the embedded memory tool abort path, empty/NONE timeout classification, or prompt-error state leakage.
