## Summary

Keep open. Current main still shares Active Memory's watchdog abort signal with the embedded run and tool wrapper path, and the latest embedded-run cleanup timeout change only bounds cleanup/lane wedging; it does not prove the reported embedded memory_search aborts, clean NONE completion being classified as timeout, or stale prompt-error replay are fixed.

## What This Changes

Active Memory embedded recall should stop aborting embedded memory_search calls, avoid timeout status after a clean embedded model completion, and prevent prior prompt-error entries from leaking into the next recall run.

## Best Possible Solution

Keep the issue open and fix the plugin-owned recall cancellation boundary: memory tools should receive a live signal until the configured recall budget actually expires, completed NONE or empty replies should record the existing non-timeout empty semantics, and stale openclaw:prompt-error custom entries should be filtered or retired before the next Active Memory recall context is built.
