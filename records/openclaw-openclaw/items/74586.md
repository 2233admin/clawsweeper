## Summary

Keep open. The report is fresh, detailed, and current main still has the same sensitive path: Active Memory races its own watchdog against `runEmbeddedPiAgent` while passing that watchdog `AbortSignal` into the embedded runner and tool wrappers, and the wrapper can surface exactly the reported `Aborted` tool result. Existing tests cover timeout and partial-timeout behavior, but not the reported case where the model produces `NONE` before the configured budget and the run is later classified as a timeout.

## What This Changes

Active Memory embedded recall should stop aborting `memory_search` tool calls, avoid timeout classification after a clean embedded model completion, and prevent prior `prompt-error` entries from leaking into the next recall run.

## Best Possible Solution

Fix the Active Memory and embedded-runner cancellation boundary so recall tool calls receive a live signal until the configured recall budget actually expires, completed `NONE`/empty replies are recorded as a non-timeout result under existing Active Memory semantics, and stale `openclaw:prompt-error` custom entries do not enter the next recall run's context.
