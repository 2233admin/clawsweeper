## Summary

Keep open. Current main still has the Active Memory watchdog signal forwarded into the embedded recall run, the tool wrapper can still surface the exact `Aborted` tool result, and the timeout recovery path still maps empty or `NONE` recovered output to `status=timeout`; the fresh report also has a persisted transcript plus an independent confirmation.

## What This Changes

The issue reports that Active Memory embedded recall aborts `memory_search` tool calls and records timeout status even when the embedded model completes with `NONE`.

## Best Possible Solution

Keep Active Memory's hard deadline, but separate deadline enforcement from normal memory tool dispatch until the deadline actually expires; completed `NONE` or otherwise empty recall should classify as `empty`, true deadline expiry should remain `timeout` or `timeout_partial`, and prior `openclaw:prompt-error` custom entries should be isolated from fresh Active Memory recall prompts.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction path with persisted Active Memory transcripts and a direct `memory_search` control, and a second user independently confirmed the same pattern on OpenClaw 2026.4.27.

## Solution Assessment

Unclear because there is no proposed patch. The maintainable fix should stay narrow to Active Memory and the embedded-runner/tool-abort boundary, not model tuning or a generic AI timeout-doctor workaround.
