## Summary

Keep this issue open. Current main still does not expose Discord slash interaction response controls to native `api.registerCommand` handlers. Discord native commands are deferred before dispatch, plugin command handlers only receive the generic `PluginCommandContext`, and the returned `ReplyPayload` is delivered afterward through one channel-level `responseEphemeral` setting. The existing Discord `respond` helper is available for component/modal interactive handlers, not native plugin commands.

## Best Possible Solution

Keep this open and design an additive plugin-command responder seam that preserves OpenClaw authorization and route/session resolution. The implementation should avoid exposing raw Discord objects, decide whether the public API is generic or channel-scoped, model the lifecycle on the existing Discord interactive `respond` helper, support defer/reply/followUp/edit semantics with per-message ephemeral flags, and add focused Discord native-command tests plus plugin docs.

## Risks / Open Questions

- Adding `ctx.discord` directly to the generic core command context could violate the extension-agnostic core boundary; a generic responder seam or channel-owned typed augmentation needs maintainer design review.
- Discord interaction lifecycle rules make double-response, already-deferred, expired-interaction, and mixed ephemeral/public follow-up behavior easy to regress without focused tests.
- Exposing raw Discord or Carbon interaction objects would couple third-party plugins to channel internals and could bypass OpenClaw auth, routing, delivery accounting, or session-state expectations.
