## Summary

Keep this issue open. Current main still does not expose Discord slash interaction response controls to native `api.registerCommand` handlers: command handlers receive the generic `PluginCommandContext`, Discord native commands are deferred before dispatch, and plugin command results are delivered afterward as a single `ReplyPayload` under the configured slash-command ephemeral mode. The existing Discord `respond` helper is only wired to component/modal interactive handlers, not native plugin commands.

## Best Possible Solution

Keep this open and design an additive, typed Discord responder seam for native `api.registerCommand` handlers. Inject it only after OpenClaw authorization and route resolution, model it on the existing Discord interactive `respond` helper, avoid exposing raw Discord objects, support defer/reply/followUp/edit semantics with per-message ephemeral flags, and add focused Discord native command tests plus public plugin docs.

## Risks / Open Questions

- A responder seam that exposes raw Discord/Carbon interaction objects could couple third-party plugins to Discord internals and bypass OpenClaw auth, routing, expiry, or delivery accounting.
- Discord interaction lifecycle rules make double-response, deferred-response, expired-interaction, and mixed ephemeral/public follow-up behavior easy to regress without focused tests.
- The current global `responseEphemeral` path is simple but insufficient for the requested per-message visibility; an implementation needs to preserve existing command authorization and route/session behavior.
