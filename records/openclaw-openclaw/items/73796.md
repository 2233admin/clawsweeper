## Summary

Keep open. Current main still has the reported browser CLI and media parser mismatch: browser screenshot/snapshot output can emit `MEDIA:~/...`, while the shared parser intentionally rejects and strips home-dir-prefixed media paths. The previously linked implementation candidate, #73798, is closed and unmerged, so there is no current-main fix to close this against.

## What This Changes

Fix browser screenshot delivery so captured browser images under the user home directory are emitted in a form the channel media pipeline accepts.

## Best Possible Solution

Keep the parser's home-dir rejection intact and fix the browser CLI producer: machine-readable `MEDIA:` lines should use absolute `result.path` / `result.imagePath`, while human-readable display output can continue using `shortenHomePath`. Add browser CLI regression tests for screenshot and AI snapshot image output.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: for a screenshot saved under `$HOME`, `shortenHomePath` can emit `~/.openclaw/...`, and `splitMediaFromOutput` rejects and strips that `MEDIA:` line before outbound delivery.

## Solution Assessment

Yes, the narrow producer-side fix is the best path. It restores browser screenshot delivery without changing the shared parser's intentional traversal and home-dir protection.
