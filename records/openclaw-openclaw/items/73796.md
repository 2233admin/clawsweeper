## Summary

Keep this issue open. Current main still rejects inline `MEDIA:~/...` directives in the shared media parser, while browser screenshot CLI output can still emit `MEDIA:` lines using `shortenHomePath`. The earlier structured-media work helps browser tool results, but it does not fix the specific final-reply path where the agent emits a tilde-prefixed `MEDIA:` line.

## Best Possible Solution

Keep this issue open and fix the shared inline media path. The conservative implementation is to normalize `~` and `~/` before `isValidMedia` rejects the candidate, or stop machine-readable browser `MEDIA:` output from using `shortenHomePath`; then add focused tests for `splitMediaFromOutput`, `parseReplyDirectives`, and the outbound reply-media path so Telegram receives a sendable media URL while traversal remains blocked.
