## Summary

Keep open. Current main still has the reported producer/validator mismatch: browser screenshot and AI snapshot CLI output can emit home-shortened `MEDIA:~/...` paths, while the shared media parser rejects those paths before Telegram receives a media payload. The earlier focused fix PR is closed and unmerged, so the user-facing bug remains actionable.

## What This Changes

Browser screenshot delivery to chat should work when screenshots are saved under the user’s OpenClaw media directory instead of being dropped because the emitted `MEDIA:` path uses `~/`.

## Best Possible Solution

Fix the browser plugin producer: emit absolute `result.path` and `result.imagePath` values only for machine-readable `MEDIA:` lines, keep `shortenHomePath` for human-readable path summaries, and leave the shared parser’s traversal/home-dir rejection intact.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current browser inspect output can produce `MEDIA:~/...`, current parser tests reject that shape, and final reply delivery depends on the parser producing `mediaUrls`.

## Solution Assessment

Yes for the producer-side repair; it is narrower and safer than expanding `~` in the shared parser because the browser route already returns absolute saved media paths and parser rejection protects sensitive home-dir paths.
