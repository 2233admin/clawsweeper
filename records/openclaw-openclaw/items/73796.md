## Summary

Keep open. Current main still has the producer/parser mismatch: the browser CLI emits machine-readable `MEDIA:` lines through home-shortened display formatting, while the shared media parser rejects home-dir-prefixed media sources; the prior fix PR is closed and unmerged.

## What This Changes

The issue requests that browser screenshots captured through the browser plugin be delivered to chat instead of being lost when the assistant repeats a home-dir-shortened `MEDIA:` directive.

## Best Possible Solution

Change only the browser CLI's machine-readable `MEDIA:` lines to emit absolute paths for screenshot and AI snapshot images, keep human-readable display paths shortened, and add focused regression tests. Leave `src/media/parse.ts` home-dir rejection intact.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: stub the browser CLI screenshot or AI snapshot result to a path under the effective home directory, observe the emitted home-shortened `MEDIA:` line, then pass that line through `splitMediaFromOutput` and see that no media URL is produced.

## Solution Assessment

No for the issue's parser-normalization preference; the safer fix is the producer-side absolute-path emission used by the closed candidate PR. That solves the browser screenshot delivery bug without relaxing the parser's traversal and home-dir protections.
