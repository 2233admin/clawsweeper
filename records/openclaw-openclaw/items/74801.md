## Summary

Keep open. Current main has moved past the exact old cron/script rotation mechanics because file logs now rotate in-process, but the Linux systemd journald storage-pressure concern remains unaddressed: service output still reaches stdout/stderr, generated systemd units do not set a journald retention or stream policy, and Linux docs still point users at journalctl without retention guidance.

## What This Changes

Request bounding or documenting Linux systemd journald storage for OpenClaw service logs because the old rotation path watched a file that did not reflect service output.

## Best Possible Solution

Keep this open until OpenClaw has an explicit Linux service logging policy that covers bounded file logs and bounded systemd service output, likely starting with documented or doctor/setup guidance for journald retention before any default unit stream or vacuum behavior changes.

## Reproduction Assessment

No high-confidence live reproduction was run. Static checks show current Linux service output can still reach journald and that no OpenClaw journald retention guidance exists; a full reproduction would install/run the current gateway as a Linux systemd user unit, generate logs, and compare journalctl disk usage with the rotating file log.

## Solution Assessment

Unclear. The report lists several plausible fixes, but the narrowest maintainable path likely starts with docs or doctor/setup guidance for journald retention rather than silently vacuuming journald or changing generated service streams by default.
