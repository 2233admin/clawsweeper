## Summary

Keep open. Current main has replaced the old cron/script-shaped file-log rotation concern with in-process rotating file logs, but it still leaves the reported Linux systemd journald storage-pressure path unresolved: generated user units do not redirect or cap stdout/stderr, runtime hints and docs treat journalctl as the Linux service log surface, and no inspected docs give journald retention guidance.

## What This Changes

The issue asks OpenClaw to address systemd journald growth because the old log-rotation job watched the file log while service output is stored in journald.

## Best Possible Solution

Choose an explicit Linux service logging policy and implement it consistently: likely document supported journald retention settings and/or add a doctor/setup hint, while preserving current rotating file logs and avoiding automatic host-level journal vacuuming unless maintainers intentionally approve that boundary.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available by inspecting the generated Linux systemd unit and runtime hints: stdout/stderr are not redirected or capped, Linux service logs are expected in journalctl, and systemd documentation confirms the default journal path.

## Solution Assessment

Unclear. The issue lists plausible remedies, but automatic journal vacuuming is a host-level operation; the narrowest maintainable first step is likely docs or doctor guidance unless maintainers decide the generated unit should redirect service output.
