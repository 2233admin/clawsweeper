## Summary

Keep open. Current main has moved from the exact cron-file-rotation behavior described in the report to in-process rotating file logs, but the Linux systemd/journald storage-pressure concern remains unaddressed: service output still reaches stdout/stderr, the generated systemd unit does not redirect or cap it, and the docs do not mention journald retention settings.

## What This Changes

The issue asks OpenClaw to handle or document journald retention for systemd user-service installs instead of relying only on OpenClaw file-log rotation.

## Best Possible Solution

Choose an explicit Linux service logging policy: document supported journald retention settings for systemd installs, add safe setup/doctor guidance if desired, and only change generated service output behavior if maintainers want OpenClaw to own that part of the operator experience.

## Reproduction Assessment

No high-confidence live reproduction was run. Source inspection gives a plausible reproduction path: install the gateway as a Linux systemd user service, emit normal console/subsystem logs, then compare the OpenClaw file log with `journalctl` disk usage under the service unit.

## Solution Assessment

Unclear until maintainers choose the policy boundary. A docs or doctor recommendation is likely safer as a first step than automatically vacuuming journald, because journald retention is host-level behavior and service stream redirection can affect diagnostics.
