## Summary

Keep open. Current main no longer matches the old cron-script details exactly because file logs now rotate in-process, but the Linux systemd journald storage-pressure concern remains unaddressed: service output still reaches stdout/stderr, Linux service hints point operators to journalctl, and neither the generated unit nor the docs provide journald retention or redirection guidance.

## What This Changes

The issue requests a fix or documentation for Linux systemd installs where OpenClaw logs continue flowing to journald while file-log rotation does not cap journal storage growth.

## Best Possible Solution

Choose an explicit Linux service logging policy and then implement it in the narrowest place: user-facing docs/setup guidance, a safe doctor/status recommendation, generated systemd unit behavior, or a combination that preserves rotating file logs and recent service diagnostics.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path for the remaining issue: run OpenClaw as the generated Linux systemd user service with console logging enabled, and stdout/stderr can continue accumulating in journald while file-log rotation remains separate.

## Solution Assessment

Unclear. The issue lists plausible fixes, but the best solution needs maintainer policy judgment because vacuuming journald is host-level behavior and changing service stdout/stderr handling can alter support diagnostics.
