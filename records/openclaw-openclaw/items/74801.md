## Summary

Keep open. Current main has fixed the obsolete cron/script part by using in-process rolling file logs, but the Linux systemd journald storage-pressure concern remains unresolved: service output still goes to stdout/stderr, the generated unit has no journal or stream policy, and docs/tests cover file-log rotation rather than journald retention.

## What This Changes

Update OpenClaw’s Linux/systemd logging behavior or docs so gateway service output is bounded in journald, not only in the gateway file log.

## Best Possible Solution

Keep this open until maintainers define an explicit Linux service logging policy that bounds both rolling file logs and service-manager logs, likely starting with user-facing journald retention guidance or doctor/setup hints before changing generated service stream behavior by default.

## Reproduction Assessment

No high-confidence live reproduction was run in this review. Static source checks show current systemd service output can still reach journald and no journald retention guidance exists, but live proof would require running the current gateway under a Linux systemd user unit, generating logs, and comparing `journalctl` disk usage against the rolling file log.

## Solution Assessment

Unclear. The report’s file-log rotation concern is already solved differently on main, but the remaining journald pressure needs a maintainer choice; docs or doctor guidance is safer than silently vacuuming journald or redirecting service streams without an explicit policy.
