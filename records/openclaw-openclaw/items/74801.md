## Summary

Keep open. Current main has moved past the reported old cron/truncate behavior and now rotates file logs in-process, but the Linux systemd journald pressure concern remains: service output is still emitted to stdout/stderr, the generated systemd unit has no stream or journal retention policy, and docs point users to `journalctl` without retention guidance.

## What This Changes

The issue asks OpenClaw to address Linux systemd logging pressure by aligning file-log rotation, journald retention guidance, or service logging behavior.

## Best Possible Solution

Keep the issue open until maintainers choose an explicit Linux service logging policy, likely starting with low-risk docs or doctor guidance for journald retention and only changing service stream behavior if that is intentional for systemd installs.

## Reproduction Assessment

No. Static checks show current systemd installs can still feed journald and lack retention guidance, but a high-confidence reproduction would run the current gateway as a Linux systemd user service, generate logs, and compare `journalctl --disk-usage` with the rolling file log behavior.

## Solution Assessment

Unclear. The report lists multiple possible fixes, and a docs/doctor recommendation is likely safer than silently vacuuming journald or redirecting service streams by default, but that needs maintainer policy judgment.
