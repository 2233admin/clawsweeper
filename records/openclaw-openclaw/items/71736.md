## Summary

Keep open. Current main still only exposes a generic Control UI descriptor registry and gateway projection, with no Control UI browser consumer or broader slot model; the issue discussion and active same-author follow-up stack also treat the current surface as partial.

## What This Changes

Add data-driven Control UI plugin contribution slots for plugin chat modes, approval cards, event classifiers, input guards, and status surfaces using safe host-owned descriptor rendering.

## Best Possible Solution

Keep this as the broad Control UI contribution-slot design thread until maintainers choose an additive, versioned, host-owned descriptor model with typed schemas, gateway projection, Control UI consumers, plugin disablement semantics, sanitization against arbitrary browser code, and fixture coverage for enabled/disabled plugins plus plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request, not a bug report. The verification path is source/docs/test inspection against current main plus the linked discussion, which shows a narrower generic descriptor foundation but no full contribution-slot implementation.

## Solution Assessment

Unclear. The direction is plausible, but the maintainable first slice likely needs maintainer product/API judgment; a manifest-first host-owned component catalog for panels/status surfaces looks safer than a broad mutable runtime registry.
