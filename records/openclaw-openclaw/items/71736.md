## Summary

Keep open. Current main has a generic plugin Control UI descriptor registry and gateway projection, but it still does not implement the broader contribution-slot model requested here: plugin-projected chat modes, input guards, event classifiers, status surfaces or panels, native pages, or browser-side rendering behavior. The active discussion and related open PRs explicitly treat the merged foundation as partial, not terminal.

## What This Changes

Request an RFC-level plugin SDK and Control UI contribution surface for data-driven plugin chat modes, approval card families, event classifiers, input guards, status surfaces, and safe host-owned rendering.

## Best Possible Solution

Keep this issue as the broad Control UI contribution-slot design thread until maintainers choose the supported slot vocabulary and projection model. The eventual implementation should be additive and host-owned: typed descriptor schemas, gateway projection, Control UI consumers, disablement semantics, sanitization against arbitrary browser code, and fixture coverage for enabled/disabled plugins and plugin-originated actions.

## Reproduction Assessment

Not applicable; this is an RFC/feature request rather than a bug report. The verification path is source/docs/test inspection against current main plus the active related discussion, which shows a narrower generic descriptor foundation but no full Control UI contribution-slot implementation.

## Solution Assessment

Unclear. The requested direction is plausible, but the best maintainable solution needs maintainer API judgment; a manifest-first, host-owned component catalog for a first slice such as panels/status surfaces may be safer than a broad mutable runtime registry.
