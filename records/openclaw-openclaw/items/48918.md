## Summary

Keep this issue open. Current main still lacks a supported skill-scoped user preferences/conventions mechanism: the loader, discovery path, prompt formatter, watcher, docs, and Claude skill materialization all center on `SKILL.md`, while `skills.entries.*.config` is not rendered as natural-language skill context. The issue discussion remains actionable and specifically supports a markdown companion-file approach.

## Best Possible Solution

Keep the issue open and make a small maintainer-approved core skill-loading contract for skill-scoped user preferences. A likely direction is a markdown companion file such as `preferences.md`, loaded or surfaced only with the matching skill, lower precedence than `SKILL.md`, with docs and tests for precedence, token limits, watcher refresh, prompt exposure, sandbox/Claude materialization, and interaction with existing `skills.entries.*` config.
