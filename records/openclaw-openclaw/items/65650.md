## Summary

Keep open. Current main still lacks a durable pre-dispatch inbound message store for accepted Slack turns; the active Slack path goes from in-memory debounce into direct reply dispatch, while the existing restart drain, restart sentinel, and outbound delivery queues cover different surfaces. Open PR #49431 remains the clearest implementation candidate to review, land, split, or supersede.

## What This Changes

The issue requests durable SQLite-backed inbound message persistence and replay around gateway SIGTERM/restart so accepted Slack messages are not lost mid-processing.

## Best Possible Solution

Keep this issue open until OpenClaw has a durable pending-inbound store at the channel/gateway ingress boundary before active dispatch, with startup replay, shutdown/restart handling, idempotent delivery semantics, and targeted regression coverage for an accepted Slack turn interrupted before final reply delivery.

## Reproduction Assessment

Yes. The report gives a concrete Slack Socket Mode path, and current-main code inspection confirms accepted Slack events can move from in-memory debounce into active dispatch without a durable pending-inbound replay record.

## Solution Assessment

No, current main is not the best solved state yet. The narrow maintainable direction is to review, land, split, or supersede #49431 with a generic pending-inbound/replay seam rather than relying on outbound queues or restart-sentinel continuations.
