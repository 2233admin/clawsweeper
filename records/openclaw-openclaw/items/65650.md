## Summary

Keep open. Current main still lacks a durable pre-dispatch inbound Slack/gateway message store; it relies on in-process debounce/direct dispatch plus graceful restart drain and separate session/outbound recovery queues. The related open PR #49431 remains the clearest implementation candidate to review, land, split, or supersede.

## What This Changes

Wire a durable SQLite-backed inbound message store into active gateway/Slack ingress so accepted in-flight messages survive SIGTERM and restart.

## Best Possible Solution

Keep this issue open and use it to review, land, split, or supersede #49431. The best fix is a durable pending-inbound store at the channel/gateway ingress boundary before active dispatch, with startup replay, shutdown/restart handling, idempotent Slack event keys, and targeted regression coverage for an accepted Slack turn interrupted before final reply delivery.

## Risks / Open Questions

- A SIGTERM or hard process exit after Slack has accepted an event can still lose the turn because the accepted inbound message is not persisted before active dispatch.
- The existing restart drain, restart sentinel, task registry, outbound delivery queue, and session-delivery recovery are easy to confuse with the requested fix, but they cover different surfaces and do not replay the original accepted Slack event.
- Any eventual durable replay implementation needs idempotency around Slack event IDs/thread routing so recovery does not duplicate already-delivered turns or send replies to the wrong thread.
- PR #49431 is size XL and unmerged, so maintainers may need to split or reshape it before landing.
