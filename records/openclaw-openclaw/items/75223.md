## Summary

Keep open: current main already has an equivalent scoped-discovery change for the probe target builder, but the plain `models status` path still loads the auth store through the no-external-profiles helper and therefore still misses runtime-only external CLI credentials. The related #75272 probe-hardening PR is adjacent, but its body explicitly excludes external CLI credential discovery, so it does not supersede this PR.

## What This Changes

The PR updates CLI model status and probe paths to pass scoped external CLI auth discovery options into auth profile store loading, with regression tests for both paths.

## Best Possible Solution

Refresh and land the remaining narrow status-path fix against current main, preferably using the current `externalCliDiscoveryForConfigStatus` helper while preserving the existing scoped probe behavior and adding/keeping focused regression coverage for CLI status and probe target construction.

## Reproduction Assessment

Yes. Current main still provides a source-level reproduction for the status half: `models status` loads the store without external CLI overlays even though the documented credential semantics require scoped discovery for runtime-only CLI credentials. The original probe store-load regression is already partially fixed on main, so remaining verification should focus on plain status and on keeping probe coverage during rebase.

## Solution Assessment

Yes, with a current-main refresh. Passing the config-derived external CLI scope into status-store loading is the narrowest maintainable fix; the probe-side change should be reconciled with main's newer `externalCliDiscoveryScoped` implementation rather than reintroducing older plumbing.
