## Summary

Keep open. Current main still lacks the requested channel-chat workflow to list configured agents and temporarily switch a single conversation to another configured agent without editing bindings or restarting the Gateway. The matching `/agents` chat surface is still thread-bound subagent/session state, while configured-agent selection exists in Gateway client surfaces such as TUI/WebChat and static routing bindings.

## What This Changes

Request a session-scoped `/agents` and `/agent <name>` command surface for single-chat channels that temporarily reroutes the current conversation to another configured agent while preserving each agent's workspace and memory isolation.

## Best Possible Solution

Add an authorized Gateway/channel routing feature that lists configured agents in external chat, stores a temporary per-conversation configured-agent override before static binding fallback, provides reset or expiry semantics, and preserves the selected agent's workspace, auth, and memory boundaries. If maintainers want this plugin-led, first expose a documented core route-override API for configured agent ids.
