## Summary

Keep open. Current main still does not implement `neverReply` or an equivalent no-LLM/no-outbound group observation mode. Mention gating stores unmentioned group messages for context, but mentioned messages still proceed to reply; `sendPolicy: "deny"` and `visibleReplies: "message_tool"` suppress delivery paths while still running the agent. The plugin-based alternative discussed in the PR is still plausible, especially the linked `openclaw-listen-only` plugin, but current main still does not broadcast unbound `inbound_claim` hooks, so replacing this PR with a plugin path requires a maintainer API/product decision first.

## What This Changes

The PR adds a `neverReply` channel/account/default config option, schema/help/docs updates, SDK export, tests, and group-message suppression hooks across the group-capable channel handlers so group messages can be recorded as context without LLM calls or replies.

## Best Possible Solution

Keep this PR open for maintainer architecture and security review. The best path is either a focused core `neverReply` implementation with schema/docs/tests and consistent account-scoped pending-history behavior across supported channels, or an explicit maintainer decision to broaden the plugin hook seam and move listen-only behavior into a plugin such as `https://github.com/flowolforg/openclaw-listen-only`.
