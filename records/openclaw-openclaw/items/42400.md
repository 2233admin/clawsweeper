## Summary

Keep open. Current main still lacks a `neverReply` or equivalent no-LLM/no-outbound group mode, and the related plugin-hook path is not currently enough to replace the PR because unbound `inbound_claim` handlers are still not broadcast. The PR is a real implementation candidate, but the remaining decision is maintainer architecture and security review across many channel handlers rather than cleanup closure.

## What This Changes

This PR adds a `neverReply` channel/account/default config option with schema/help/docs/SDK export changes, resolver and wiring tests, and per-channel group-message suppression gates intended to observe allowed group traffic without invoking the agent or sending replies while leaving DMs unchanged.

## Best Possible Solution

Choose one maintainer-approved architecture and make it complete: either land a focused core `neverReply` policy with consistent account-scoped pending-history semantics, docs, generated config metadata, and representative channel tests, or first extend the plugin hook seam so a listen-only plugin can safely own the behavior outside core.
