## Summary

Keep open. Current main still does not implement the `neverReply` config, and the existing plugin `inbound_claim` path does not supersede it because main only routes plugin-bound conversations. The proposed patch is still not merge-ready: several documented history-retention paths return before writing pending history, and Telegram accepts a per-group/topic safety config that the resolver cannot read.

## What This Changes

The PR adds a `neverReply` channel/account/default config intended to suppress group replies while optionally retaining group messages as pending context across group-capable channels.

## Best Possible Solution

Decide the architecture first. If the core config path is accepted, make each channel use a tested shared pattern that writes, reads, and clears pending history consistently before suppressing replies; if the plugin path is preferred, land the generic hook broadcast seam and keep listen-only behavior in an external plugin such as `openclaw-listen-only`.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction for the PR regressions: enable `neverReply` for Discord, LINE, Matrix, or Mattermost group traffic and the new branch returns before the documented pending-history write path is reached.

## Solution Assessment

No, not as currently written. The requested behavior is useful, but this implementation repeats per-channel gates and already diverges from the documented retention semantics; a shared core helper pattern or a maintainer-approved plugin seam would be safer.
