## Summary

Keep open. Current main and the latest v2026.4.27 tag still start the default-enabled model-pricing refresh from initial Gateway runtime services before post-attach sidecars and channels complete, so the default startup path described in the report is not yet fixed. The config opt-out is useful mitigation, but the direct ordering fix remains the open, unmerged #73486, with #73323 tracking broader runtime degradation.

## What This Changes

The issue reports that Gateway startup still waits roughly 75 seconds for OpenRouter/LiteLLM pricing fetch timeouts before channels and sidecars become ready on macOS, even after the earlier model-pricing manifest reuse fix.

## Best Possible Solution

Move default pricing refresh out of the pre-ready Gateway startup path so channel and sidecar readiness cannot wait on optional OpenRouter/LiteLLM catalog fetches. Keep `models.pricing.enabled: false` as a documented workaround, and leave #73323 to track any remaining long-running process/network degradation after the startup ordering fix lands.

## Reproduction Assessment

Yes. The issue provides a detailed restart timeline and current source gives a focused reproduction path: run Gateway with default pricing enabled while OpenRouter/LiteLLM pricing fetches hang or time out, then verify whether `gateway ready` and channel startup wait for the pricing timeout.

## Solution Assessment

Yes, the narrow direction in #73486 is the best current fix: defer optional pricing refresh until after sidecars/channels are ready without disabling pricing globally. A shorter timeout or opt-out alone would mitigate symptoms but would not remove pricing from the readiness-critical startup path.
