## Summary

Keep open. Current main and latest release still start the default-enabled model-pricing refresh from early Gateway runtime setup before the sidecar/channel readiness boundary, so the reported startup-readiness bug is not resolved by the shipped manifest-reuse fix or by the config opt-out workaround.

## What This Changes

The issue reports macOS Gateway startup/readiness blocking for about 75 seconds while OpenRouter and LiteLLM pricing fetches time out despite the earlier manifest-reuse fix.

## Best Possible Solution

Land or adapt #73486 so optional model-pricing refresh starts only after the post-attach sidecars-ready path and scheduled-service activation, while keeping `models.pricing.enabled: false` as an operator mitigation and leaving broader runtime/network degradation to #73323.

## Reproduction Assessment

Yes. A high-confidence path is to make the OpenRouter and LiteLLM pricing fetches hang under default config, restart the Gateway, and verify whether channel/sidecar readiness waits; current source still starts pricing before readiness and retains 60s fetch timeouts.

## Solution Assessment

Yes. For this issue's narrow scope, the best fix is to move optional pricing refresh fully behind Gateway/channel readiness; the shipped opt-out helps affected users but is not the best default behavior because remote pricing enrichment should not compete with startup readiness.
