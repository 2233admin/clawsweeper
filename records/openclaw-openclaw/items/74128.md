## Summary

Keep open. Current main still starts the default model-pricing refresh from early Gateway runtime setup before the post-attach sidecars/channel readiness boundary; the shipped opt-out is useful mitigation but does not resolve the default boot-readiness problem reported here.

## What This Changes

The issue reports that macOS Gateway boot still waits roughly 75 seconds on default OpenRouter/LiteLLM pricing fetch timeouts after the earlier partial fix, delaying readiness despite fast standalone fetches.

## Best Possible Solution

Land or adapt PR #73486 so optional pricing refresh starts only after sidecars/channels have reached the ready path, keep `models.pricing.enabled: false` as a documented mitigation, and leave the broader runtime/network degradation investigation in #73323 for symptoms that continue after deferral.

## Reproduction Assessment

Yes. A high-confidence path is to run default pricing with OpenRouter/LiteLLM fetches stalled to their 60s timeout during Gateway restart and observe whether channel/sidecar readiness waits; the issue supplies deterministic macOS logs and current source still preserves the risky ordering.

## Solution Assessment

Yes. Deferring the optional pricing refresh until after readiness is the narrowest maintainable fix for this issue; the shipped config opt-out helps affected users but does not protect the default startup path.
