## Summary

Keep open. The report is concrete, recent, and still points at a narrow browser CLI regression: the `open` and `tabs` handlers only produce JSON if the browser parent option survives lazy dispatch, and current tests do not exercise that real lazy path for these commands.

## What This Changes

The issue asks for `openclaw browser --json open` and `openclaw browser --json tabs` to emit machine-readable JSON on stdout instead of human-readable text on stderr.

## Best Possible Solution

Add regression coverage that drives `browser --json open` and `browser --json tabs` through `registerBrowserCli` and the lazy browser command registration path, then preserve the selected parent JSON mode so these commands emit structured stdout JSON while keeping text-mode output unchanged.

## Repair Work Prompt

Fix the browser CLI JSON regression reported at https://github.com/openclaw/openclaw/issues/74574. In `v2026.4.26`, `openclaw browser --json open <url>` and `openclaw browser --json tabs` print human-readable text through the normal log path instead of JSON on stdout, while `browser evaluate` still works. Keep the fix scoped to browser CLI/lazy option propagation. Add regression tests that drive `registerBrowserCli` through the real lazy browser command path for both `browser --json open` and `browser --json tabs`; do not rely only on `createBrowserManageProgram` direct-handler tests. Ensure parent `--json` and `--browser-profile` survive lazy reparse, `open` writes the `BrowserTab` payload as JSON, `tabs` writes `{ "tabs": [...] }`, and text-mode output remains unchanged.
