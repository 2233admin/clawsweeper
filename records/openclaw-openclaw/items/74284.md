## Summary

Keep open. Current main still lets heartbeat runs without an explicit `heartbeat.model` use stored session model overrides, while docs and tests preserve auto-fallback persistence until reset; the heartbeat-specific stale fallback path remains valid and is not fully covered by #51677.

## What This Changes

The issue asks OpenClaw to stop autonomous heartbeat sessions from continuing to use a session-persisted stale model or fallback after `agents.defaults.model.primary` changes and the gateway restarts.

## Best Possible Solution

Add a narrow heartbeat/default-model-change rule that ignores or clears stale auto-sourced fallback overrides when the stored fallback was selected for a previous effective default, while preserving explicit heartbeat models, user overrides, legacy user overrides, and normal unchanged-default fallback persistence.

## Repair Work Prompt

Fix the heartbeat stale model follow-up described in openclaw/openclaw#74284. Observable bug: the autonomous heartbeat session (`agent:main:main`) can keep using a model/fallback selected when the session was created even after `agents.defaults.model.primary` changes and the gateway restarts. Scope the fix narrowly to heartbeat/default-model-change behavior: when a heartbeat run has no explicit `heartbeat.model`, do not let an auto-sourced stale fallback override pin the run to an old configured default. Preserve user-selected overrides (`modelOverrideSource: "user"` and legacy overrides without a source), preserve explicit `agents.*.heartbeat.model`, preserve channel model overrides where appropriate, and avoid changing unrelated session reset or fallback policy. Likely implementation areas: `src/auto-reply/reply/get-reply.ts`, `src/auto-reply/reply/model-selection.ts`, possibly a small helper near session/model override handling. Add focused regression coverage for a heartbeat run where the session store contains an auto fallback override/fallbackNotice from an old primary and config now has a different `agents.defaults.model.primary`; assert the new default is selected and the stale override is not used. Also keep or extend tests proving user overrides and unchanged auto fallback behavior still work. Add a user-facing changelog entry if repo policy expects one for this bug fix, credited with `Thanks @brtkwr`. Do not alter provider fallback classification, auth profile rotation, or explicit `heartbeat.model` semantics.
