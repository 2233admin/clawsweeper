## Summary

Keep open. Current main still lets heartbeat runs without an explicit `heartbeat.model` apply a stored session model override, including auto-fallback overrides selected for an older default, and the issue is now paired with open closing PR #74932 rather than resolved on main.

## What This Changes

The issue requests that heartbeat runs for `agent:main:main` stop reusing stale auto-fallback model overrides after `agents.defaults.model.primary` changes across gateway restarts.

## Best Possible Solution

Land or supersede #74932 with a narrow heartbeat model-selection rule that clears or skips only stale direct auto-sourced fallback overrides when the recorded selected default no longer matches the current configured default, while preserving explicit heartbeat models and normal override contracts.

## Reproduction Assessment

Yes. Seed a heartbeat session entry with `modelOverrideSource: "auto"`, fallback notice metadata for an older default, and no explicit `heartbeat.model`; current main's heartbeat and reply selection path will apply the stored override after the default changes.

## Solution Assessment

Yes. The maintainable direction is a focused heartbeat/default-change guard in model selection, not a broad startup migration or a general removal of auto-fallback persistence.
