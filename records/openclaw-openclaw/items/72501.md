## Summary

Keep this PR open. Current main still lacks poll-answer session routing, and the PR is aimed at the right Telegram plugin boundary, but the latest diff still has a security-sensitive trusted prompt-context regression and a non-idempotent send failure mode that should be fixed before merge.

## What This Changes

The PR adds a Telegram poll registry, records public poll metadata when polls are sent, registers `poll_answer` handling, routes vote events to the originating session, and adds Telegram tests.

## Best Possible Solution

Keep the registry-backed Telegram plugin design, but revise the PR so poll vote notifications preserve the untrusted external-content boundary, registry persistence is best-effort after a successful send, focused tests cover those behaviors, and the user-facing fix has a single-line changelog entry.

## Reproduction Assessment

Yes. On current main, sending a public Telegram poll can return a `pollId`, and Telegram/grammY support `poll_answer`, but source inspection shows no main-branch handler or registry lookup; for the PR findings, the changed lines directly show the missing `trusted: false` and awaited post-send registry write.

## Solution Assessment

No, not yet. The owner boundary and persistent registry approach are maintainable, but the proposed implementation should not promote vote-derived text into trusted prompt context or fail an already-sent Telegram poll because local state persistence failed.
