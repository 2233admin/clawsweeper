## Summary

Keep open. Current main still lacks usable inbound Telegram poll-answer routing, so the PR targets a real gap, and the current diff no longer includes the earlier CODEOWNERS overwrite. It is not merge-ready because the new handler routes group poll answers without chat-origin proof from Telegram's payload, and the poll-send path can report failure after Telegram already sent the poll if the registry write fails.

## What This Changes

The PR adds a Telegram poll registry, records public poll metadata after sending polls, registers a `poll_answer` handler that enqueues vote events into routed sessions, explicitly includes `poll_answer` in allowed updates, and adds focused tests.

## Best Possible Solution

Revise the PR so public poll metadata is persisted durably, registry persistence failures after a successful Telegram send are logged but non-fatal, and group `poll_answer` updates only enqueue session events after explicit sender authorization or membership proof appropriate for an update that lacks source-chat context.

## Reproduction Assessment

Yes. Current main can be checked by sending or inspecting a public Telegram poll path: poll sends return a `pollId`, but there is no `bot.on("poll_answer")` handler to route the later vote. The PR-specific defects are reproducible by making the registry path unwritable after a successful send and by invoking the new group handler with a stored group poll plus a voter payload that has no chat-origin proof.

## Solution Assessment

No. Adding persistent poll-answer routing is the right direction, but this implementation should not fail a completed send on registry-write errors and should tighten group voter authorization before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Prove group voters before enqueueing poll answers:** `extensions/telegram/src/bot-handlers.runtime.ts:1024-1097`
  - body: Telegram `PollAnswer` updates identify the voter but not the chat where a non-anonymous vote was cast. This handler authorizes against the stored origin chat and then enqueues for that group session, so an open allowed group has no proof that the voter is actually a member of that group. Require explicit sender allowlist authorization or verify group membership before routing group poll-answer events.
  - confidence: 0.82
- **[P2] Keep registry failures from failing sent polls:** `extensions/telegram/src/action-runtime.ts:456-466`
  - body: The poll has already been sent when this awaited registry write runs. If state directory creation or the atomic write fails, the action rejects and reports failure even though Telegram delivered the poll, which can cause duplicate retries and a false user-facing error. Wrap and log this write so the send result is still returned.
  - confidence: 0.9
