## Summary

Keep this PR open. Current main does not implement the proposed Telegram unknown-group action surface: there are no `unknownGroupAction`, `unknownGroupMessage`, or `unknownGroupCooldownMs` config keys, and current runtime behavior only logs/skips blocked group messages or sends fixed native-command auth replies. The PR is a plausible implementation candidate, but the review context and current code show it still needs maintainer review and fixes before it could safely land.

## Best Possible Solution

Keep the PR open for maintainer review. If the feature is accepted, implement it inside the Telegram plugin with strict config/schema/docs coverage, a shared unknown-group helper used by both normal messages and native command auth, correct wildcard-vs-unknown classification, `buildTelegramThreadParams` for forum-topic warnings, bounded cooldown behavior, and targeted tests for ignore/warn/leave plus command and General-topic cases.

## Risks / Open Questions

- Closing this PR as implemented would be incorrect because current main has no equivalent config, docs, tests, or runtime behavior.
- The PR's proposed unknown-group classification needs tightening so wildcard-configured groups or sender-allowlist failures are not treated as truly unknown chats.
- The warn/leave path should share existing Telegram thread-parameter helpers to avoid sending `message_thread_id=1` in General forum topics.
- Native slash commands are authorized before normal message handling, so unknown-group behavior needs integration there as well, not only in message handlers.
- The proposed cooldown tracker and `warn` without message behavior from the review comments are quality/operability issues to address before shipping, though they are not supply-chain concerns.
