## Summary

Keep open. Current main still exposes cron schedules as one of `at`, fixed `every`, or cron expression, and the runtime next-run computation does not evaluate user/job state to shorten an interval. The external zeroclaw cross-reference is not a merged OpenClaw fix, and the related OpenClaw items in the provided context cover different scheduling or concurrency concerns rather than this conditional-interval feature.

## Best Possible Solution

Keep this open for a focused scheduler design. The best path is an additive cron contract or plugin API seam that defines safe predicate evaluation, state sources, next-run recomputation on run completion and relevant state changes, protocol/CLI/tool schema validation, docs, and regression tests. Do not close as ClawHub until the plugin surface can express conditional intervals without ad hoc mutation of core cron state.

## Risks / Open Questions

- A safe design needs to define where `state.*` comes from, when state changes trigger rescheduling, and what predicate language is allowed so cron config does not become an unsafe arbitrary-expression surface.
- Existing plugin hooks can observe and update cron jobs, but using a plugin to rewrite schedules would be a workaround rather than the requested public cron contract; maintainers should decide whether this belongs in core schedule schema or a new plugin API seam.
