## Summary

Keep open. Current main still lacks the proposed plugin-management Gateway RPC family, and the PR is a plausible implementation direction, but it should not merge as-is because `plugins.update` drops the existing documented update selection semantics for npm package specs and version selectors.

## What This Changes

The PR adds admin-scoped Gateway `plugins.*` RPC params, handlers, plugin-management service helpers, scope/rate-limit wiring, generated Swift protocol models, docs, and tests for plugin list, inspect, doctor, registry refresh, install, update, uninstall, enable, and disable operations.

## Best Possible Solution

Keep the PR open, preserve the shared service-layer direction, and update the RPC contract/service so remote plugin updates match the existing documented update selector behavior or deliberately document a narrower API with separate tests before maintainer approval.

## Reproduction Assessment

Yes for the review finding: with an installed npm plugin whose record is keyed by plugin ID, call `plugins.update` with a documented npm spec/version selector such as `@openclaw/voice-call@beta`; the PR passes that string as a literal plugin ID and `updateNpmInstalledPlugins` reports a skipped no-record outcome. The original feature request itself is not a bug report, and current-main absence was verified by source search.

## Solution Assessment

No, not yet. The service extraction and admin-scoped Gateway handlers are a maintainable direction, but the update path should reuse the existing CLI selection logic/spec override semantics or expose an intentionally different typed selector before this becomes the best implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Preserve update selection semantics:** `src/plugins/management-update.ts:40-42`
  - body: `plugins.update` now passes `params.id` directly as a plugin ID. That bypasses the existing `openclaw plugins update <id-or-npm-spec>` selection helper, so documented calls like `@openclaw/voice-call@beta` are treated as a literal plugin ID and return a successful skipped outcome instead of updating the tracked package with a spec override. Reuse the selection helper or add an explicit version/spec selector to the RPC contract.
