## Summary

Keep open. Current main still lacks remote plugin-management RPCs, so the PR is not obsolete, but the latest head has a concrete error-classification bug in install failure handling and is missing the required changelog entry for a user-facing feature.

## What This Changes

The branch adds typed Gateway plugin-management RPC schemas, handlers, scopes, rate limits, shared plugin-management services, regenerated Swift models, docs, and tests for plugin list, inspect, doctor, registry, install, update, uninstall, enable, and disable operations.

## Best Possible Solution

Land the feature only after the install error mapping is corrected, the required changelog entry is added, CI is green, and maintainers explicitly accept the admin-scoped remote plugin-management boundary.

## Reproduction Assessment

Yes for the review finding: call `plugins.install` with `source: "path"` and a missing Gateway-host path. The path resolver returns an uncoded `path not found` failure, and the PR maps that to `UNAVAILABLE` instead of an invalid request.

## Solution Assessment

No as-is. The service-layer approach is a maintainable direction for sharing CLI plugin operations with Gateway handlers, but the PR should preserve client-error semantics for uncoded local install failures before it is the best mergeable solution.
