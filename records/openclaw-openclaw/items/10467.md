## Summary

Keep open. Current main still exposes `sessions_spawn` as a single fixed `subagent` lane: the tool schema and execution path do not accept or forward a caller-selected lane, native and ACP spawn dispatches hard-code `AGENT_LANE_SUBAGENT`, and config/docs/tests still model sub-agent concurrency as one scalar cap. The recent maintainer-side and ClawSweeper comments are still accurate against current main.

## What This Changes

The issue requests an optional `sessions_spawn.lane` parameter plus per-lane sub-agent concurrency configuration while preserving the default `subagent` lane when omitted.

## Best Possible Solution

Implement a backward-compatible `sessions_spawn.lane` API that defaults to `subagent`, validates lane names, forwards the resolved lane through native and ACP spawn paths, applies per-lane sub-agent concurrency at Gateway startup and reload, and updates schema, docs, changelog, and regression tests without changing existing callers.

## Reproduction Assessment

Yes. The feature gap is reproducible by static inspection on current main: no `lane` schema field exists, no execution path reads or forwards a lane, and both native and ACP child dispatches send the fixed `subagent` lane.

## Solution Assessment

Yes. An additive lane parameter is the narrowest maintainable direction because the Gateway `agent` protocol and command queue already support arbitrary named lanes; the repair should reuse that infrastructure instead of adding a separate scheduler.
