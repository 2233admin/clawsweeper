## Summary

Keep this issue open. Current main still has the exact feature gap: `sessions_spawn` exposes no caller-selectable lane, native and ACP child dispatch both hard-code the `subagent` lane, and config/docs/tests still describe one scalar global subagent concurrency cap. The adjacent open per-agent lane PR is related infrastructure work, but it does not implement `sessions_spawn.lane` for sub-agent runs.

## What This Changes

Add multi-lane concurrency support to `sessions_spawn` so callers can choose named sub-agent queue lanes with compatible per-lane concurrency limits.

## Best Possible Solution

Add an additive `sessions_spawn.lane` API that defaults to the existing `subagent` lane, validates lane names, threads the resolved lane through native and ACP child dispatch, and exposes compatible per-lane subagent concurrency configuration with docs and regression tests preserving existing callers.
