## Summary

Keep open. The PR addresses a plausible resilience goal, but it is not merge-ready: the two named gateway startup maintenance paths already catch and log failures on current main, while the registry-loader change flips a generic fail-fast contract used outside gateway startup and would leave existing tests expecting `throwOnLoadError: true` stale or failing.

## What This Changes

The PR changes gateway startup maintenance from `Promise.all` to `Promise.allSettled` with per-task logging, and changes runtime registry loading to stop throwing on plugin load errors.

## Best Possible Solution

Keep the resilience work scoped to a proven gateway startup crash path: preserve the generic `ensurePluginRegistryLoaded` fail-fast behavior, add a targeted gateway-startup test for any remaining uncaught task failure, and preserve useful error stacks in any new isolation logging.

## Reproduction Assessment

Yes. A high-confidence reproduction path is static and test-backed: apply the PR diff and run the runtime registry/CLI plugin registry tests that currently assert `throwOnLoadError: true`; for behavior, use a configured plugin that throws during load and call `ensurePluginRegistryLoaded` for that configured scope.

## Solution Assessment

No. The `Promise.allSettled` change is mostly redundant for the named helpers on current main, and the `throwOnLoadError: false` change is too broad for a gateway-startup-only fix; a narrower gateway-specific isolation path with tests would be safer.
