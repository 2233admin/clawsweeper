## Summary

Keep open. The report is fresh, concrete, and Slack-specific; current main has shared MEDIA parsing/staging and Slack upload primitives, but I found no Slack monitor regression coverage proving that every thread-reply/fallback path consumes raw MEDIA directives before delivery. The most suspicious remaining gap is that the Slack monitor delivery helper only sends media when the payload already has media fields, so an unnormalized payload can still be posted as text.

## What This Changes

Slack thread replies containing standalone MEDIA directives should be converted into uploaded Slack file attachments instead of appearing as plain text or disappearing.

## Best Possible Solution

Add Slack-specific regression coverage for thread replies with raw MEDIA directives, then ensure the Slack monitor reply/fallback path always runs directive parsing and local media normalization before sendMessageSlack while preserving thread_ts and existing media access policy.

## Reproduction Assessment

Yes. The issue provides concrete manual steps, and the code supports a focused unit reproduction by feeding a Slack thread reply containing MEDIA:/path into the monitor delivery path and asserting sendMessageSlack receives mediaUrl plus threadTs rather than raw text.

## Solution Assessment

Unclear. The existing shared parser/stager and Slack upload primitive are the right building blocks, but the best fix should close the Slack-specific unnormalized delivery gap rather than adding a second upload mechanism or weakening local media policy.
