## Summary

Keep open. Current main at a820a307dfeb950fa633c0ae6fb2386a35a91991 still only deduplicates direct cron delivery by executionId derived from jobId plus runStartedAt, so the scheduled-slot replay gap described by this PR is not implemented on main. The live PR is focused on cron delivery code/tests plus one timer outcome field and does not introduce supply-chain-sensitive changes. Remaining maintainer review should decide whether the proposed slot key also needs threadId and refresh the branch against current main if needed.

## Best Possible Solution

Keep this PR open for maintainer review and likely merge after resolving the threadId key question, rebasing or refreshing checks against current main, and preserving the regression test for the same scheduled slot with different runStartedAt values plus the startup catch-up deliveryAttempted propagation.
