## Summary

Keep open. Current main still formats the response usage footer’s input count from raw `usage.input`, while cache-heavy runs expose the missing prompt-token-total behavior that PR #58917 is trying to fix for #58875. The PR is not merge-ready as-is because it carries unrelated Telegram/cron/provider-test changes and prior review comments flagged correctness risks, but the central footer fix is not implemented or superseded on main.

## Best Possible Solution

Keep #58917 open as useful but require a focused rebase: update only the response usage footer path to display prompt-token totals for cached-token-heavy normalized usage, keep cost estimation on raw usage fields, add regression tests for both `input + cacheRead` display and no double counting after `normalizeUsage`, and split unrelated Telegram/cron/provider-test changes into separate PRs if still needed.
