## Repair Work Prompt

Fix the reply-threading bug tracked by https://github.com/openclaw/openclaw/issues/66540. In `src/auto-reply/reply/reply-payloads-base.ts`, parser-default `replyToCurrent: false` must not prevent `applyReplyThreading` from assigning the implicit `currentMessageId`/`replyToId` when reply threading is otherwise allowed. Preserve intentional non-threaded command replies from `src/auto-reply/reply/commands-core.ts`; if `false` is still needed as a command sentinel, split that explicit opt-out from parser-default directive output rather than treating every false value as an opt-out. Add a regression test in `src/auto-reply/reply/reply-plumbing.test.ts` for a normal payload with `replyToCurrent: false`, no explicit `replyToId`, `replyToMode: "all"` or `"first"`, and a `currentMessageId`; it should receive the implicit `replyToId`. Keep or extend `src/auto-reply/reply/commands-core.send-policy.test.ts` so command replies remain non-threaded. Check the streaming/directive path in `src/auto-reply/reply/streaming-directives.ts` and the followup/queue path before deciding whether helper-only change is sufficient. Do not change channel transport behavior or directive syntax. Add a user-facing changelog fix entry if the repair changes shipped behavior, crediting the reporter and prior relevant contributors according to repo changelog rules.
