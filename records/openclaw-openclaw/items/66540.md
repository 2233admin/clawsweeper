## Summary

Current main still contains the reported `replyToCurrent === false` veto in the shared reply-threading helper, and the linked fix PRs in the provided context are closed/unmerged. The issue should stay open until the helper and affected queue/streaming paths distinguish a default-false parse result from an intentional opt-out and add regression coverage.

## Best Possible Solution

Keep this issue open. The best fix is a small shared reply-threading change that treats parser/default `replyToCurrent: false` as absence of a tag, while preserving any intentional command/non-threading opt-out via an explicit policy or separate field. Add a regression test for `applyReplyThreading` with `replyToCurrent: false`, `replyToMode: "all"` or `"first"`, and `currentMessageId`, plus a focused queue/streaming test if that path is the user-visible reproducer.

## Risks / Open Questions

- A blanket removal of the false check could regress command replies, because `commands-core` deliberately clears `replyToId` and sets `replyToCurrent: false`, with a test asserting command replies are non-threaded.
- `resolveFollowupDeliveryPayloads` currently calls `applyReplyThreading` without a `currentMessageId`, so the queued/followup path may need an additional reply-target handoff beyond the shared-helper change.
