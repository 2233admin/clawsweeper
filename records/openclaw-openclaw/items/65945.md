## Summary

Keep open. Current main still treats Node `agent.request` image attachments as agent input, while delivery sends only agent-produced reply payloads; there is no protocol field, command option, or delivery hook that preserves original Node attachments for automatic outbound IM forwarding. The prior ClawSweeper comment remains accurate against the newer main SHA, and related #7095 covers multimodal agent input rather than this outbound forwarding behavior.

## What This Changes

The issue requests an opt-in or fallback path that forwards Node-provided image attachments to the target IM channel when `agent.request` uses `deliver: true`, even if the agent returns only text or no media.

## Best Possible Solution

Keep the issue open until maintainers approve the contract. A good implementation would add an explicit opt-in field, carry sanitized delivery-ready media references through the gateway command path, and inject them only when `deliver: true`, the target is resolved and authorized, the channel supports the media type, and the agent did not provide its own media.
