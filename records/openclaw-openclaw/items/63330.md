## Summary

Keep this PR open. Current main still does not expose the proposed `api.runtime.followup.enqueueFollowupTurn()` plugin runtime surface and does not include the bundled `gateway-restart` plugin. Main has adjacent mechanisms, including next-turn injections, subagent helpers, system events/heartbeat, and durable restart-continuation/session-delivery internals, but those do not fully replace a plugin API for injecting a user-role prompt into an arbitrary existing session and waking it. The PR also adds a security-sensitive restart and shell-command execution path, so it needs maintainer design/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer design and security review. The best path is to decide whether OpenClaw should expose a generic plugin runtime followup API; if yes, port it onto current main's durable session-delivery/restart-continuation infrastructure with strict session-context preservation. The bundled `gateway-restart` plugin should either be split from the API work or aligned with the existing owner-only gateway restart policy, restart sentinel, dependency metadata, docs, and focused tests.

## Risks / Open Questions

- The PR adds a plugin tool that can run configured shell commands and terminate the gateway process; it must be owner-only, respect `commands.restart`, and have clear operator opt-in semantics before merge.
- The proposed API is built around followup queue semantics while current main has a durable session-delivery queue for restart continuations; merging without rebasing the design could lose callbacks across crashes/restarts or duplicate newer infrastructure.
- Reconstructing an existing session turn is context-sensitive; existing review comments flagged workspace, account, group policy, streaming break, elevated execution, and transcript-path preservation risks.
- The proposed gateway-restart plugin writes a single marker and deletes it after enqueue; review discussion flagged durability and concurrency concerns that should be resolved against the current restart-sentinel/session-delivery model.
- The branch is large and stale against current main, with unrelated qa-lab/a2ui/lockfile churn in the PR history; it should be split or rebased before normal review.
