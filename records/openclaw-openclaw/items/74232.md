## Summary

Keep open. Current main still silently drops fully sanitized unknown or malformed assistant tool-call turns, so the PR targets a real gap, but the latest diff is not safe to merge because it persists synthetic toolResult entries for assistant tool calls that were removed from the transcript.

## What This Changes

The PR adds WARN diagnostics and detailed sanitizer reports for dropped assistant tool calls, then changes the session write guard and tests to persist synthetic error toolResult messages for fully dropped unknown or malformed calls.

## Best Possible Solution

Keep the structured diagnostics and WARN logging idea, but do not persist toolResult messages unless the matching assistant tool call remains in transcript order and the active transcript policy allows synthetic results. If immediate model feedback is required, implement it in a provider-safe layer that cannot create orphan tool results, and cover all-invalid, mixed valid/invalid, policy-disabled, beforeMessageWrite, thinking-replay, and existing stream-guard cases.

## Reproduction Assessment

Yes. The PR’s own updated unit cases reproduce the problematic shape by appending a fully invalid assistant tool-call turn and expecting a lone persisted toolResult; current replay-repair tests show that such orphan toolResult entries are invalid and dropped.

## Solution Assessment

No. The PR fixes observability directionally, but persisting a synthetic toolResult after dropping the originating assistant turn is not the narrowest safe solution; logging-only plus a provider-safe feedback path would avoid invalid transcript state.
