## Summary

Keep open. Current main still has the reported scheduler fairness problem: busy inbound work is appended to the same followup FIFO, the drain runs until the backlog is empty, and each followup turn then enters the same per-session FIFO lane. Related open work such as #57239 may overlap with responsiveness, but it does not reference this issue with closing syntax and does not bound or prioritize the followup drain itself.

## What This Changes

The issue asks OpenClaw to prevent followup queue draining from monopolizing a session lane and starving later inbound Discord, WhatsApp, or system-event messages.

## Best Possible Solution

Add core scheduler fairness to the auto-reply followup path: bound consecutive followup turns or drain elapsed time, give fresh user/control inbound work a way to run before accumulated system/followup backlog on the same session key, and add regression coverage proving fresh inbound is not starved while preserving per-class ordering and collect-mode safety.

## Repair Work Prompt

Fix the session-lane starvation reported in openclaw/openclaw#54488. Work in the core auto-reply/session scheduler, not channel-specific plugins. Add a bounded/fair followup drain so accumulated followup/system-event work cannot monopolize a session key indefinitely, and give later fresh user/control inbound work a deterministic way to run before the entire old followup backlog. Preserve ordering within each priority class and preserve collect-mode route/auth grouping, dedupe, cap/drop/summarize behavior, cross-channel routing metadata, and existing stop/clear semantics. Do not hard-code Discord, WhatsApp, Telegram, or bundled plugin ids into core. Check overlap with #57239 and #52026 before choosing the final shape, but do not treat fast-lane-only behavior as sufficient. Add focused regression tests that enqueue multiple followups, enqueue later fresh inbound on the same session key while a drain is active, and prove the fresh turn is not starved behind the whole backlog. Add a user-facing changelog fix entry in the active version with `Thanks @Flakedict`.
