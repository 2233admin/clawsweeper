## Risks / Open Questions

- The API would expose the final post-policy per-attempt tool list and schemas, including tools that are never called, to loaded plugins; maintainers should confirm this fits the trusted-plugin boundary.
- The proposed contract passes `parameters` by reference. If those are provider-facing schema objects, hook handlers could accidentally mutate later provider/session behavior unless the final implementation clones, freezes, or clearly documents read-only handling.
- Fire-and-forget per-attempt emissions can repeat on retries; consumers may need documented dedupe expectations or a stable fingerprint.
- Because this is an API/contract feature with SDK baseline and docs impact, it needs maintainer design review rather than an automated close or a separate replacement PR.
