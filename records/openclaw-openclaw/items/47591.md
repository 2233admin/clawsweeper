## Summary

Keep this issue open. Current main has partial mitigations for typing, tool-progress previews, outbound retries, and crash-recovery delivery queueing, but it still does not implement the requested gateway-level silence watchdog or per-turn response-delivery confirmation state machine. The related #65919 is narrower active-run presence work and does not cover end-of-turn delivery confirmation, retry dedupe, final-failure alerting, or no-response detection.

## Best Possible Solution

Keep this open as a core/runtime design item. Align active-run presence with #65919, then design a generic delivery-confirmation contract around the shared reply/outbound pipeline: turn/response IDs, delivered/failed/no-response states, channel ACK capability, retry dedupe, final-failure alerting, preserved redelivery, per-channel config, docs, and regression coverage for final replies, message-tool sends, chunked delivery, preview/final transitions, and final-only channels.

## Risks / Open Questions

- A broad gateway-level watchdog could violate documented final-only channel semantics unless it is gated by per-channel capability/config and silent-reply policy.
- Delivery confirmation needs careful idempotency design: current queue recovery can replay payloads after failure, while the proposal also wants immediate retry and delayed-ACK dedupe across chunks and tool sends.
- Operator-visible alerting in groups/channels may leak internal runtime status unless ownership, mention, command, and final-only policies are preserved.
