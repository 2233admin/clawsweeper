## Summary

Keep open. Current main does not implement the PR’s central MCP-backed skill readiness gate, and the PR remains a recent, unique implementation candidate. The live PR patch still needs maintainer review because the readiness logic appears to allow a server-level callable grant to override an explicit skill-level non-callable record, and the new smoke script is an opt-in command-execution surface that should be reviewed before merge.

## Best Possible Solution

Continue review rather than close. Rebase the PR onto current main, keep the MCP readiness gate if maintainers want this core behavior, add explicit tests for skill-level denial precedence, owner scoping, server fallback, and missing/stale readiness, and either justify or remove the unrelated package-boundary/test-alias changes. The smoke script should remain opt-in and documented before any automation uses it.

## Risks / Open Questions

- Merging the current readiness logic could advertise a skill explicitly marked `callable: false` if its server record is `callable: true`.
- The smoke script executes `mcporter` from PATH against user config when `--verify-callable` is passed; keep it explicitly opt-in and verify the correct cwd for global config before wiring it into automation.
- The PR includes package-boundary and test-alias changes that are not part of the MCP readiness feature, which increases review scope and could mask type-boundary behavior if left unexplained.
