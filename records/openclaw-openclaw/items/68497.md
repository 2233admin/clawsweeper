## Summary

Keep open. Current main exposes `window.openclawSendUserAction`, but it still only posts to native iOS/Android bridge handlers; a normal browser has no Gateway-bound action transport. The canvas WebSocket is live-reload-only, canvas HTTP rejects non-GET/HEAD requests, and the broader related #66983 remains open rather than superseding this narrower action-bridge request.

## What This Changes

Expose a browser fallback for the canvas A2UI user-action bridge so `window.openclawSendUserAction` can route web page actions back to the Gateway over WebSocket or HTTP.

## Best Possible Solution

Define the browser action bridge as a typed core canvas/gateway contract: accept only authenticated or capability-scoped browser requests, validate the A2UI user-action payload, route it into the same agent-action path native nodes use, return `openclaw:a2ui-action-status` to the page, and document the security model alongside focused gateway/canvas tests.
