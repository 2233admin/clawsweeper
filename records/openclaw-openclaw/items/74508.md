## Summary

Keep open. Current main still stops Telegram draft previews at the Telegram character limit, so the PR is addressing a real remaining Telegram streaming gap. The latest head appears to address the earlier offset-reset and archived-preview cleanup concerns, but it introduces a blocking retry-state bug in the new 429 handling: after a rate-limited send returns false, the draft loop retries the same text, while the PR has already marked that text as sent, so the retry can be skipped and the preview update lost.

## What This Changes

This PR updates Telegram draft streaming to split overflowing previews into continuation messages, label reasoning continuations, suppress partial reasoning-tag answer leakage, add shared edit/typing throttles and 429 backoff, and cover the behavior with Telegram tests.

## Best Possible Solution

Keep the overflow-chaining direction, but revise the branch so retryable 429 failures do not poison duplicate-send state, then add a regression test where a rate-limited draft send retries and actually sends the same pending preview text.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P2] Reset sent-text markers before rate-limit retries:** `extensions/telegram/src/draft-stream.ts:454-461`
  - body: The new 429 path returns `false`, so `createDraftStreamLoop` requeues the same text for a later flush. But `lastSentText` and parse mode were already updated before the failed Telegram call; on retry, the duplicate check returns `true` and clears the pending update without sending or editing it. Clear those markers on retryable failure, or only set them after a successful send.
