## Summary

Keep open: current main still stops Telegram draft previews when rendered text exceeds the Telegram character cap, and this draft PR addresses a real user-visible overflow problem. The latest PR head fixes the earlier duplicate-marker retry issue, but the new 429 backoff path can still clobber newer pending preview text, so the branch needs revision before merge.

## What This Changes

This draft PR updates Telegram draft streaming to split and chain overlong previews, label reasoning continuation messages, suppress partial reasoning-tag answer leakage, add shared edit/typing throttles with 429 backoff, refactor retry_after extraction, and extend Telegram tests.

## Best Possible Solution

Keep the overflow-chaining direction, but revise the rate-limit retry path so a 429 backoff cannot overwrite newer accumulated preview text, then add a regression where an update arrives during the 429 sleep and verify the latest preview is retried rather than stale content.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve newer pending text during 429 backoff:** `extensions/telegram/src/draft-stream.ts:466-469`
  - body: The 429 catch sleeps while the draft loop still treats this send as in flight. If `update()` receives newer accumulated reasoning during that sleep, `createDraftStreamLoop` later sees `sent === false` and restores the older `text` from this failed attempt over the newer `pendingText`, so a rate-limited stream can retry stale preview content and miss the final update. Preserve newer pending text on retry or move the backoff into loop state, and add a regression with an update arriving during the 429 sleep.
  - confidence: 0.86
