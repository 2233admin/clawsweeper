## Summary

Keep open. Current main still has the reported stop-on-overflow behavior, and this PR is useful but not merge-ready because the fork-resume prompt fix does not cover the actual file-backed runtime path and the required changelog entry is missing.

## What This Changes

The PR changes Telegram draft streaming to split overflowing reasoning previews across messages, adds shared throttling/backoff behavior, suppresses partial reasoning-tag answer flicker, and adjusts CLI argument construction for fork-session resumes.

## Best Possible Solution

Rebase the useful Telegram overflow work onto current main, keep the Telegram and CLI fixes either fully integrated or split into separate PRs, add coverage through the actual dispatch/execute paths, and include the required changelog entry before merge.

## Reproduction Assessment

Yes. For the reported Telegram behavior, current main can be reproduced with a small maxChars draft-stream unit path: send an initial fitting preview, then update past the limit and observe the stream stop rather than send a second message. For the PR-specific prompt gap, source inspection shows the runtime never creates a system prompt file when useResume is true.

## Solution Assessment

No. Overflow chaining is the right direction, but the current PR does not fully solve the fork-session resume prompt bug for file-backed/default backends, and it needs rebase plus changelog work before it is the narrow maintainable fix.
