## Summary

Keep open. Current main still stops Telegram draft streaming once the rendered preview exceeds the Telegram limit, so the PR targets a real remaining behavior, but the current draft patch is not mergeable because it routes intentionally completed overflow chunks through the stale-preview cleanup contract and can also drop a requested lane rotation during 429 backoff.

## What This Changes

The PR updates the Telegram plugin to chain overlong reasoning draft previews into continuation messages, add continuation rendering plus edit-rate throttling/backoff, suppress partial reasoning-tag leakage, and expand Telegram streaming tests.

## Best Possible Solution

Revise the Telegram draft-stream implementation so completed overflow chunks have an explicit retained lifecycle distinct from stale/superseded previews, and make 429 backoff preserve both the newest pending text and any requested message rotation. Cover the fix with draft-stream unit tests plus a dispatch regression that finishes a long reasoning stream and verifies all intended chained messages remain visible.

## Reproduction Assessment

Yes. The current-main overflow behavior is reproducible by driving createTelegramDraftStream with a rendered preview over maxChars; the PR-specific cleanup regression can be reproduced with a dispatch test that overflows reasoning, finalizes the turn, and observes the completed chunk being deleted through archivedReasoningPreviewIds.

## Solution Assessment

No. The PR is aimed at the right Telegram plugin surfaces, but the current patch is not the best solution because it reuses stale-preview cleanup semantics for retained overflow chunks and suppresses lane rotation during 429 backoff without replaying it later.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve chained reasoning chunks during cleanup:** `extensions/telegram/src/draft-stream.ts:379-382`
  - body: The overflow path reports intentionally completed reasoning chunks through onSupersededPreview, but Telegram dispatch treats reasoning previews from that callback as archived cleanup targets and deletes them during final cleanup. A long reasoning stream would briefly chain and then lose completed chunks except the active last one, so this needs a distinct retention signal or cleanup exclusion plus a dispatch regression.
  - confidence: 0.91
- **[P2] Defer lane rotation after rate-limit backoff:** `extensions/telegram/src/draft-stream.ts:543-549`
  - body: When a 429 backoff is active, forceNewMessage returns without recording that rotation is still needed. Call sites such as onReasoningStream then clear splitReasoningOnNextStream and enqueue the next block, so after the backoff the retry edits the old message instead of starting the requested new one.
  - confidence: 0.84
