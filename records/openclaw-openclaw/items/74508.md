## Summary

Keep open. Current main still stops Telegram draft previews when the rendered preview exceeds the Telegram limit, and this branch targets the right user-visible Telegram reasoning stream path. The latest patch is not ready to merge because the new overflow offset can survive normal fresh-message rotations and cause later independent previews to be sliced or suppressed.

## What This Changes

The PR updates Telegram draft streaming to split oversized previews across chained messages, adds explicit reasoning-continuation rendering and concurrent-lane throttle scaling, suppresses partial closing-tag answer flicker, and adds colocated tests.

## Best Possible Solution

Revise the branch so overflow-internal rotations preserve the active split offset, while public fresh-message rotations reset it for a new logical preview; keep the continuation renderer and chunk-draining behavior covered by targeted Telegram draft-stream tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reset overflow offsets when starting a fresh preview:** `extensions/telegram/src/draft-stream.ts:330`
  - body: After an overflow chain, `textBaseOffset` remains nonzero. Existing callers use `forceNewMessage()` to start unrelated fresh previews, but this patch leaves that offset in place, so the next `update()` runs through `trimmed.slice(textBaseOffset)` and can drop the first part of the new message or suppress it entirely if it is shorter than the old offset. Split the internal overflow rotation from the public fresh-message reset, and add a regression that overflows once, calls `forceNewMessage()`, then sends a new short update.
  - confidence: 0.9
