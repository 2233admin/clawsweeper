## Summary

Keep open: current main still has a concrete OpenAI-compatible request-shaping gap for strict Mindie-style backends. The schema and model normalizer support generic compat flags, but the Chat Completions request builder still sends `stream_options.include_usage` unconditionally, and related #50374 tracks a different message-key sanitization problem.

## What This Changes

The issue requests a fix for Mindie/OpenAI-compatible local model requests failing with HTTP 422 because OpenClaw sends request parameters the strict backend rejects.

## Best Possible Solution

Make the OpenAI Chat Completions transport honor `compat.supportsUsageInStreaming: false` by omitting `stream_options`, preserve streamed usage for known or explicitly compatible providers, and keep tool-related mitigation on schema-backed generic compat flags rather than hardcoding Mindie IDs.
