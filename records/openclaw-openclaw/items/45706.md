## Summary

Keep open: current main still has an actionable OpenAI-compatible request-shaping gap for strict Mindie-style backends. The config/schema path can represent streaming-usage compatibility, but the Chat Completions payload builder ignores the opt-out and still sends the field that this class of backend is reported to reject.

## What This Changes

The issue reports Mindie/OpenAI-compatible chat completions failing with HTTP 422 because the backend rejects unsupported request parameters.

## Best Possible Solution

Add generic OpenAI-compatible request shaping that honors compat.supportsUsageInStreaming: false by omitting stream_options, preserves usage reporting for native/known-compatible or explicitly opted-in endpoints, and documents the supported compat keys without hardcoding Mindie provider IDs.

## Repair Work Prompt

Fix the strict OpenAI-compatible streaming request-parameter bug tracked by https://github.com/openclaw/openclaw/issues/45706. Current `buildOpenAICompletionsParams()` always sends `stream_options: { include_usage: true }` for streaming Chat Completions, even when `models.providers.<provider>.models[].compat.supportsUsageInStreaming` is false or defaulted false for strict custom endpoints. Implement generic request shaping, with no Mindie-specific provider IDs: omit `stream_options` when resolved compat says streaming usage is unsupported, preserve it for native/known-compatible endpoints and explicit `compat.supportsUsageInStreaming: true`, and keep the local/proxy token-usage behavior from PR #68746 for compatible backends. Add focused regression coverage for `compat.supportsUsageInStreaming: false` omitting `stream_options` and for compatible/opted-in endpoints still sending it. Do not solve broader message-key sanitization from #50374 in this PR. Update docs/changelog if user-visible config guidance changes; changelog entries must be single-line with contributor attribution.
