## Summary

Keep this issue open. Current `main` still starts outbound delivery recovery on non-minimal Gateway startup, loads every pending delivery queue JSON entry, and treats old `retryCount=0` entries without `lastAttemptAt` as immediately eligible with no pending-age cutoff or replay-count cap. The latest release baseline shown in the repository has the same outbound recovery behavior, so this has not been fixed on `main`.

## Best Possible Solution

Implement a conservative outbound recovery policy in `src/infra/outbound`: expire or move stale pending deliveries to `failed/` before replay, preserve recent crash recovery, and add focused tests for old `retryCount=0` entries without `lastAttemptAt`, `maxRecoveryMs` interaction, startup recovery, and reconnect drains. If operator control is desired, add documented config/schema for max pending age and possibly max replay count.
