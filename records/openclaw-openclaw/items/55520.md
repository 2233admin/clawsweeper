## Summary

Keep this issue open. Current main still does not implement the reported provider-aware fallback or reclassification for OpenAI-compatible streams: bare `reasoning_content` remains a thinking block, `delta.content` remains visible text, and existing recovery retries after reasoning-only turns rather than converting answer-like reasoning into text.

## Best Possible Solution

Keep the issue open and implement a provider-aware stream classification path. The safest direction is explicit provider/plugin compat metadata or typed upstream fields, with regression tests for both mirrored samples: answer-like bare `reasoning_content` with empty `content`, and provider output where `content` is known to be internal reasoning. Avoid a global bare-reasoning-to-text fallback unless the provider contract proves it is user-visible answer text.

## Risks / Open Questions

- Closing now would hide a still-current loss path for OpenAI-compatible providers that return answer-like text only in bare `reasoning_content`, `reasoning`, or `reasoning_text` fields.
- A blanket fallback from all bare reasoning fields to visible text would risk exposing private chain-of-thought for providers where those fields are truly internal reasoning.
- The mirror case where `delta.content` contains internal-looking reasoning needs provider-specific evidence or typed upstream metadata; text-shape heuristics alone would be unsafe.
