## Summary

Keep open. Current main has important partial mitigations for this class of failures, but it does not implement the reported fallback/reclassification behavior. OpenAI-compatible stream parsing still maps non-empty `delta.content` to `text` and bare `reasoning_content`/`reasoning`/`reasoning_text` to `thinking`; the newer retry path can recover reasoning-only turns, and Z.AI now disables thinking by default, but neither proves that a provider returning the visible answer only in `reasoning_content` will be delivered as text or that text containing internal reasoning will be reclassified safely.

## Best Possible Solution

Keep open. The best path is a targeted provider-aware stream-classification fix with regression coverage for the exact mirrored samples: empty `content` plus answer-like `reasoning_content`, and content that is known provider reasoning rather than final answer. Z.AI should keep using its thinking controls; any generic OpenAI-compatible fallback should be opt-in or based on documented provider fields/types so real reasoning is not leaked.

## Risks / Open Questions

- Closing now could hide a still-current loss/leak path for OpenAI-compatible providers that put visible answer text into bare reasoning fields rather than `content` or typed `reasoning_details` visible-output records.
- A naive fallback that always exposes `reasoning_content` as text would risk leaking private chain-of-thought for providers where that field is truly internal reasoning; the fix needs provider/model-aware evidence and tests.
