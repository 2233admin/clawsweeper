## Summary

Keep this PR open. Current main still only partially covers the intended CLI help cleanup: `cron add --at` mentions `--tz`, but `cron edit --at` does not, and both current `--tz` help strings still describe cron expressions only. The PR is useful and narrow, but should not merge unchanged because its proposed `(default: UTC)` wording is wrong for recurring cron schedules, which default to the Gateway host timezone when `tz` is omitted.

## What This Changes

The PR changes the `cron add` and `cron edit` CLI help strings so `--tz` is described as applying to cron expressions and offset-less `--at` datetimes, and updates `cron edit --at` help text for discoverability.

## Best Possible Solution

Keep the PR open and request or apply a narrow wording refresh: `cron edit --at` should mention offset-less datetimes with `--tz`, and both `--tz` help strings should cover `--cron` plus offset-less one-shot datetimes while preserving the two defaults: cron uses the Gateway host local timezone when omitted, while offset-less `--at` is UTC unless `--tz` is supplied.
