## Summary

Keep open. Current main has several runtime-deps, mirror, and manifest-cache mitigations, including a post-release fix for reinstall loops, but the issue thread still contains concrete post-mitigation reports and current code retains reachable bundled plugin loading, runtime-deps lock, mirror preparation, and request-time registry paths that need affected-host proof before this can be closed.

## What This Changes

The issue asks OpenClaw to fix a packaged plugin-loader/runtime-deps hot loop that repeatedly mirrors, parses, or stages bundled plugin runtime artifacts and starves the gateway event loop.

## Best Possible Solution

Keep this as the canonical plugin-loader/runtime-deps starvation tracker until maintainers retest an affected packaged host on current main or the next package. If it still reproduces, add focused timing/cache-hit instrumentation around plugin registry loading, bundled runtime-deps materialization, dist-mirror locks, chat-completions preparation, and channel startup, then split or close against narrower proven fixes as appropriate.
