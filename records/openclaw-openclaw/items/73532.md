## Summary

Keep open. Current main contains several targeted mitigations for the reported manifest, FileHandle, runtime-deps, and mirror paths, but the issue thread has fresh residual reports on v2026.4.27 and a current-main chat-completions trigger, and source inspection still shows reachable exact-cache-key loader and runtime-deps/mirror lock paths that need affected-host proof before closure.

## What This Changes

The issue reports a bundled plugin loader/runtime-deps hot loop that repeatedly mirrors/parses bundled plugin runtime state, saturates CPU, grows memory, and starves Gateway Telegram, WebSocket, and OpenAI-compatible request handling.

## Best Possible Solution

Keep this as the canonical plugin-loader/runtime-deps starvation tracker until maintainers retest an affected packaged host on current HEAD or the next package with timing/cache-hit instrumentation. If it still reproduces, land the smallest proven fix in the plugin loader/runtime-deps/request-time tool path and then close satellites such as #74860 or #75032 only when their distinct behavior is covered or intentionally split.
