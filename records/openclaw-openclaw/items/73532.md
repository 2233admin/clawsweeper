## Summary

Keep open. Current main contains several targeted runtime-deps and manifest-cache mitigations, but the issue discussion still has post-mitigation reports, including a local-main chat-completions spike, and current code still has reachable runtime-deps lock, runtime dist-mirror preparation, package-level materialization, and request-time manifest registry paths. This remains the canonical tracker until an affected packaged host is retested on current main or the next release.

## What This Changes

The issue reports a bundled plugin/runtime-deps loader hot loop where repeated runtime dist mirror preparation, runtime-deps staging, and JSON5 manifest parsing starve the Gateway event loop and break Telegram, WS status probes, and chat requests.

## Best Possible Solution

Keep this as the canonical runtime-deps/plugin-loader starvation tracker through an affected-host retest on current main or the next package. If it still reproduces, split or fix the smallest proven hot path with timing/cache-hit instrumentation around runtime-deps materialization, dist-mirror locks, manifest registry loading, chat-completions preparation, and channel startup.

## Reproduction Assessment

Yes for the affected released range: the issue gives detailed Linux, Docker, and Raspberry Pi setups with plugin counts, V8/perf stacks, CPU/RSS symptoms, and v2026.4.24-v2026.4.27 behavior. It is unclear for exact current main because this review did not run an affected host and newer mitigations after v2026.4.29 have not been proven against the same reproducer.

## Solution Assessment

Unclear. The current-main fixes are sensible targeted mitigations, but the best solution is not to close this tracker until the startup, repeated-staging, and request-time chat-completions paths are quiet on an affected packaged host.
