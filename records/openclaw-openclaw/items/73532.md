## Summary

Keep open. The report is recent, detailed, and still actionable against current main. The current tree has useful plugin metadata snapshot, manifest parsing, and runtime mirror mitigations, but I did not find a direct source, test, changelog, or history proof that the reported steady-state plugin-loader hot loop is fixed. The broader #72338 is related but still open and does not supersede the narrower plugin-loader profile and ruled-out-trigger data here.

## Best Possible Solution

Keep this issue open as the focused plugin-loader hot-loop tracker. The best next step is targeted instrumentation around resolvePluginLoadCacheContext, getCachedPluginRegistry/setCachedPluginRegistry, prepareBundledPluginRuntimeDistMirror, loadPluginManifestRegistry/loadPluginManifest, and steady-state gateway RPC/provider/tool call sites. If confirmed, normalize compatible cache keys and add a per-process debounce or memo for runtime mirror preparation, plus a regression test or profiler proof that repeated post-startup calls do not re-mirror bundled runtime roots or reparse manifests unnecessarily.
