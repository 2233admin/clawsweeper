## Summary

Keep open. Current main has several real mitigations for the runtime-deps and manifest hot paths, but the latest issue discussion reports residual v2026.4.27/current-main request-time stalls and repeated staging patterns, and the current code still has reachable plugin-tool and exact-cache-key runtime loading paths that need affected-host proof before closure.

## What This Changes

The issue reports a packaged Gateway plugin-loader hot loop where bundled runtime-deps mirror preparation and JSON5 manifest parsing repeatedly saturate CPU, starve the event loop, and block Telegram/RPC/OpenAI-compatible request handling.

## Best Possible Solution

Keep this as the canonical runtime-deps/plugin-loader starvation tracker until maintainers retest a packaged affected host on current main or the next package. If it still reproduces, add focused timing/cache-hit instrumentation around plugin registry loading, request-time plugin tool creation, runtime-deps materialization, dist-mirror locks, and channel startup, then land the smallest proven fix with packaged Linux/Docker evidence.

## Reproduction Assessment

Yes for the released packaged range: the issue and comments provide concrete Linux/Docker/Raspberry Pi setups, plugin counts, profiles, logs, and v2026.4.24-v2026.4.27 behavior. It is unclear for exact current main because this read-only review did not run an affected-host package reproduction and the latest main-specific report predates some later runtime-deps/mirror commits.

## Solution Assessment

Unclear. Current main contains sensible mitigations, but they are not yet a proven end-to-end fix for the current request-time and repeated-staging reports; maintainer retest plus narrower root-cause selection is safer than closing or handing this to a one-shot repair branch.
