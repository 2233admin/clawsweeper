## Summary

Keep open. Current main contains several runtime-deps and manifest-cache mitigations for the reported hot paths, but the provided thread has fresh post-v2026.4.27 and current-main reports of residual event-loop stalls, repeated staging, and a deterministic `POST /v1/chat/completions` trigger; the current code still has reachable request-time plugin tool and exact-cache-key registry load paths that need affected-host proof before closure.

## What This Changes

Investigate and fix the plugin loader and bundled runtime-deps hot loop that saturates the Gateway event loop during startup or agent request handling.

## Best Possible Solution

Keep this as the canonical runtime-deps/plugin-loader starvation tracker until maintainers retest a packaged Linux/Docker affected host on current main or the next package with plugin-load cache-hit, runtime-deps staging, dist-mirror lock, event-loop delay, and request-time tool-construction timing enabled; then land the smallest proven fix or split narrower follow-ups such as #74860 or #75032 only after the root cause is isolated.

## Reproduction Assessment

Yes for the affected packaged range: the report and comments include concrete Linux/Docker/Raspberry Pi setups, profiles, logs, plugin counts, and v2026.4.24-v2026.4.27 behavior. Exact current HEAD reproduction is not yet high-confidence because this read-only pass did not run the affected host scenario and the latest main-specific report predates 54e6e3d7.

## Solution Assessment

Unclear. The current mitigations are directionally correct for mirror scans, manifest parsing, and file-handle closure, but the post-mitigation chat-completions and repeated-staging reports mean closure or a one-shot repair branch is not the safest path yet.
