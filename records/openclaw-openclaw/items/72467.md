## Summary

Keep this PR open. Current main still has the reported Discord audit mismatch, and the PR fixes the central top-level `allowFrom` case, but the branch should be adjusted before merge because it counts `discordCfg.allowFrom` from a merged account config instead of the effective account allowlist resolver, which can create an audit false negative for legacy account overrides.

## What This Changes

The PR updates the Discord security audit to count top-level `allowFrom` in `ownerAllowFromConfigured`, adds a regression test, and adds a changelog entry.

## Best Possible Solution

Update the audit to compute configured owner allowlists through the same effective Discord account allowlist path used by runtime authorization, append pairing-store entries as the audit already does, keep the regression coverage, and then land after normal maintainer validation.

## Reproduction Assessment

Yes. On current main, a config with native Discord commands, a configured guild, no guild/channel users list, and only top-level `channels.discord.allowFrom` still reaches the warning branch because `ownerAllowFromConfigured` ignores that field.

## Solution Assessment

No, not quite. The PR is the right narrow area, but the implementation should use the effective account allowlist resolver rather than the merged `discordCfg.allowFrom` field so multi-account legacy override precedence stays aligned with runtime behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use the effective account allowlist for audit suppression:** `extensions/discord/src/security-audit.ts:173-178`
  - body: For non-default accounts, `params.account.config` is a merged config. If root `channels.discord.allowFrom` is set but an account explicitly overrides with legacy `dm.allowFrom: []`, this new read still counts the inherited root list even though `resolveDiscordAccountAllowFrom` gives the account legacy field precedence and runtime owner auth sees no owner list. That creates a false negative for `channels.discord.commands.native.no_allowlists`; compute the configured owner list through the same effective resolver used by native-command auth, then append `storeAllowFrom`.
  - confidence: 0.84
