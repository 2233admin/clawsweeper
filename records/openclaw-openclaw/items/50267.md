## Summary

Keep this PR open/reworkable rather than closing. Current main has the newer provider-request policy infrastructure, but the image-understanding direct `complete()` path still does not visibly apply OpenRouter attribution headers, and the OpenRouter plugin’s normal `wrapStreamFn` hook is not installed by this path. The PR also needs rework because its provider-only header lookup would apply OpenAI hidden attribution headers to custom OpenAI-compatible image routes, which the current request policy intentionally gates by endpoint.

## Best Possible Solution

Rework the PR on current main to apply direct-completion attribution through the shared endpoint-aware request policy, likely `resolveProviderRequestPolicyConfig`/`resolveProviderRequestHeaders` with `capability: "image"` and `transport: "media-understanding"`, and add focused tests for OpenRouter image headers plus custom-base-url non-injection. Keep the PR open until that rework or a merged canonical replacement lands.

## Risks / Open Questions

- The PR as written would use provider-only `resolveProviderAttributionHeaders`, so OpenAI/OpenAI-Codex image models with custom proxy base URLs could receive hidden `originator`/`User-Agent` attribution headers that current main intentionally gates to native hosts.
- The PR targets an older file path; current main uses `src/media-understanding/image.ts`, so it needs a rebase and probably a different implementation point.
- There is no current regression test proving OpenRouter image-tool requests get attribution headers while custom proxy routes do not.
