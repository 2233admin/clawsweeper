## Summary

Keep open. Current main has broader fallback improvements for thrown/unrecognized errors and related OpenRouter no-endpoints failures, but the reported HTTP-200 business rejection can still return as an embedded error-only result that the result classifier ignores, allowing the fallback loop to record `candidate_succeeded` before trying the next model.

## Best Possible Solution

Keep this issue open and land a focused fallback regression plus classifier fix. The fix should convert an error-only embedded result containing `success:false` / `CE-011` / `ń”üµŁóĶ«┐ķŚ«` provider business rejection text into a failover classification before `candidate_succeeded` is recorded, while preserving the existing safeguards for visible replies, aborts, and outbound side effects.
