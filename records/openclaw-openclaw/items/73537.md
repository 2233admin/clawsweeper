## Summary

Keep this issue open. Current main has beta-first release lanes, release validation, update status, dry-run, staged package update verification, and update hardening, but it still does not provide the requested per-release production-readiness / early-adopter / known-issues indicator in GitHub release notes, release metadata, or update surfaces. The open update-review PR and broader update-trust issues overlap, but they do not supersede this narrower release-label request.

## Best Possible Solution

Keep this issue open for release-process/product follow-up. A good path is to define a documented release-readiness enum, require release operators to set it in GitHub release notes or release metadata for each release, and optionally surface that value through `openclaw update status`, a future update-review command, or Control UI update surfaces. Coordinate broader update-risk UX with #61973, #65011, and #65302.
