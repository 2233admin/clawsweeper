## Summary

Keep open. Current main has beta-first release flow, stronger release validation, staged update verification, dry-run/status update surfaces, and the latest GitHub release notes, but it still does not expose the requested per-release production-ready / early-adopter / known-issues indicator. The overlapping pre-upgrade risk PR is open and unmerged, and it would not by itself define a release-owner readiness label.

## What This Changes

Request to add a visible per-release production-readiness or stability indicator to release notes, release metadata, or update surfaces.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, require release operators to set it from one source of truth for each release, and render it in GitHub release notes first, with update/status CLI visibility added only if maintainers want users to consume it from installed clients.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: release docs, update docs, update CLI registration, release metadata checks, latest changelog/release body, and the related open PR all show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the taxonomy and criteria. The requested direction is plausible, but the maintainable solution should start with release metadata/process policy rather than a heuristic-only update command.
