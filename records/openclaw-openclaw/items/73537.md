## Summary

Keep this open. Current main has stronger release validation, beta-first release flow, post-publish checks, and update status/dry-run surfaces, but it still does not provide the requested visible per-release production-ready / early-adopter / known-issues indicator in release notes, release metadata, or update UI/CLI surfaces. The related update-risk PR is still open and does not supersede this narrower release-readiness metadata request.

## What This Changes

Request a visible per-release production-readiness stability indicator, such as production-ready, early-adopter, or known-issues, in release notes, GitHub release metadata, or update surfaces.

## Best Possible Solution

Define a documented release-readiness enum and criteria in the release process, make it a required source-of-truth field for each published release note or release metadata record, and expose it consistently in update-facing surfaces after the policy is settled.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: docs:list, exact terminology searches, release workflow inputs, release docs, changelog, update CLI registration, and update docs show no visible per-release readiness label.

## Solution Assessment

Unclear until release owners choose the taxonomy and source of truth. The requested direction is plausible, but the maintainable path should start with release policy/metadata rather than a heuristic CLI-only assessment.
