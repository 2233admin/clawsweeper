## Summary

Keep open. Current main still rejects the reported symlinked `~/.openclaw` layout, and the issue is already paired with open implementation PRs, so it should remain open until a reviewed fix lands.

## What This Changes

Allow `openclaw exec-policy preset yolo` to work when the default `~/.openclaw` directory is a trusted first-level symlink while preserving deeper exec-approvals symlink hardening.

## Best Possible Solution

Land a reviewed fix that permits exactly the trusted first-level `.openclaw` symlink shape under the user home on supported platforms, validates the symlink target safely, keeps the final approvals file and deeper path components protected, and adds focused regression coverage plus docs if the supported layout changes.

## Reproduction Assessment

Yes. The source path is high-confidence: a symlinked `.openclaw` below the resolved home root is encountered by `assertNoSymlinkPathComponents()` and rejected before `saveExecApprovals()` can write the approvals file.

## Solution Assessment

Unclear between the two open PRs, but the requested direction is sound if implemented narrowly. The maintainable fix should be the smallest safe home-boundary relaxation with owner/mode validation and continued rejection of deeper symlinks.
