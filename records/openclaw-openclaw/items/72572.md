## What This Changes

The issue requests support for a trusted, user-owned first-level `~/.openclaw` symlink under the normal home directory while continuing to reject unsafe deeper symlinks in the exec approvals path.

## Best Possible Solution

Land one reviewed implementation that permits exactly one safe, user-owned, non-group/other-writable first-level `.openclaw` symlink and keeps the existing destination-file and deeper-symlink protections, with focused store tests and docs if the supported layout changes.
