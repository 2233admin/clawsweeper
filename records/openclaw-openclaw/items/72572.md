## Summary

Keep open. Current main still rejects a symlinked `~/.openclaw` path component during exec approvals writes, and the issue is already paired with open closing PRs, so it should remain open until one fix is reviewed and merged.

## What This Changes

Support `openclaw exec-policy preset yolo` when the OS-home `~/.openclaw` directory is a trusted symlink while preserving deeper exec-approvals symlink protections.

## Best Possible Solution

Land one focused fix that allows only a trusted first-level `~/.openclaw` symlink under the OS home, validates the real target ownership and writability, keeps rejecting deeper symlinks and a symlinked `exec-approvals.json`, and adds regression tests for the GNU Stow-style layout.

## Reproduction Assessment

Yes. A symlinked `HOME/.openclaw` followed by `openclaw exec-policy preset yolo` deterministically reaches `saveExecApprovals()` and the current component walk rejects `.openclaw` before the write.

## Solution Assessment

Yes in direction, with a security-sensitive implementation caveat. The narrow maintainable solution is to allow exactly the trusted home-boundary `.openclaw` symlink while preserving the existing deeper traversal and destination-file protections.
