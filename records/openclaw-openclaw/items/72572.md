## Summary

Keep open. Current main still rejects the reported symlinked `~/.openclaw` shape, and open PRs #72650 and #72713 reference this issue with closing syntax, so the issue should remain open until one implementation is reviewed and merged.

## What This Changes

Support `openclaw exec-policy preset yolo` when the default `~/.openclaw` directory is a trusted first-level symlink while keeping deeper exec-approvals symlink traversal blocked.

## Best Possible Solution

Land one reviewed implementation that permits exactly one trusted, user-owned, non-group/other-writable first-level `.openclaw` symlink and continues rejecting chained or deeper symlinks plus symlinked approval-file destinations, with focused store tests and docs if the supported layout changes.

## Reproduction Assessment

Yes. Create a home where `~/.openclaw` is a symlink and run `openclaw exec-policy preset yolo`; the source path calls `saveExecApprovals()`, then `ensureDir()`, which rejects the symlinked `.openclaw` component before writing.

## Solution Assessment

Yes in direction, but not yet implemented on main. The narrow owner/mode-checked first-level `.openclaw` exception is the maintainable fix shape; broader state-dir or macOS changes should be included only if review shows they are necessary and equivalently hardened.
