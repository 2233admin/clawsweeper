## Summary

Keep open. Current main still has the reported compatibility gap: the OpenAI transport normalizes tool schemas without passing model compat, and the bundled Fireworks Kimi K2.5 router has no catalog compat entry for stripping JSON Schema `not`.

## What This Changes

The issue asks OpenClaw to apply `compat.unsupportedToolSchemaKeywords` during OpenAI-compatible tool schema normalization and add the Fireworks Kimi K2.5 router compat metadata for unsupported `not`.

## Best Possible Solution

Thread `ModelCompatConfig` through the OpenAI tool-schema normalization API used by Responses and Completions, add the Fireworks manifest compat entry for the affected Kimi router, and lock both paths with focused unit tests while keeping Fireworks-specific metadata in the Fireworks plugin.

## Reproduction Assessment

Yes. A focused unit reproduction can build an `openai-completions` model with `compat.unsupportedToolSchemaKeywords: ["not"]` and a tool schema containing `{ "not": {} }`; current `buildOpenAICompletionsParams` reaches the normalizer without `modelCompat`, so the unsupported keyword remains.

## Solution Assessment

Yes. The narrow maintainable fix is to pass the raw/extracted model compat into schema normalization and add provider-owned Fireworks catalog metadata, rather than adding Fireworks-specific branching in core or broadening `getCompat` for unrelated transport policy fields.
