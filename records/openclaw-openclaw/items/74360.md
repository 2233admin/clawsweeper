## Summary

Keep this PR open. Current main still has the broad trusted web-tools private-network policy, and this PR is a plausible fix for #74357, but the latest patch adds public Plugin SDK exports without updating the generated SDK API baseline. Because this is a security-sensitive implementation PR paired with an open issue, it needs maintainer/security review and a PR-branch update before merge.

## What This Changes

The PR splits web-tool SSRF handling into strict trusted endpoints and explicit self-hosted helpers, switches SearXNG to the self-hosted helper, exports those helpers through the Plugin SDK, and adds SSRF policy tests.

## Best Possible Solution

Land a maintainer-approved SSRF fix that keeps public web-search providers strict by default, preserves a validated self-hosted private-network path for SearXNG-style endpoints, updates generated SDK baseline/docs for any public helper exports, and lets #74357 close only after merge.

  - body: These new integration tests assert that trusted endpoints block private targets, but they inherit `HTTP_PROXY`/`HTTPS_PROXY` from the process. In an environment with a proxy configured, `withTrustedWebToolsEndpoint` enters trusted env-proxy mode and can call the mock fetch instead of taking the pinned-DNS block path, making the tests fail for environment reasons. Clear or stub the proxy env in this test file before asserting the SSRF policy behavior.
  - confidence: 0.78
