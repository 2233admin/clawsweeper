## Summary

Keep open. Current main does not already include the guardrails plugin, and the PR is an active, security-sensitive bundled-plugin proposal with concrete review findings on credential handling and dispatch availability that should be resolved before any merge decision.

## What This Changes

The PR adds a bundled `extensions/guardrails` plugin using `before_dispatch` with blacklist, HTTP, and dynamic-import connectors, plus config schema, tests, runtime asset packaging, release checks, and lockfile updates.

## Best Possible Solution

Maintainers should decide whether guardrails belongs as an official bundled security plugin or as an external ClawHub/native plugin using the existing hook seam. If it lands in core, it should first support the existing secret pipeline, bound initialization waits with deterministic fallback, align hook/docs/changelog, and complete dependency/security/package validation.

## Reproduction Assessment

Yes for the review defects: manifest/runtime inspection shows SecretRefs would be ignored for provider keys, and a custom HTTP provider or import module whose `init()` never settles would reproduce the unbounded `before_dispatch` wait. For the feature itself, reproduction is not applicable beyond confirming current main lacks the bundled plugin.

## Solution Assessment

No, not as-is. The plugin-shaped approach uses the right extension seam, but merging a bundled security plugin without SecretRef-aware credentials, bounded init fallback, and maintainer/secops ownership review is not the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Make provider keys SecretRef-aware:** `extensions/guardrails/openclaw.plugin.json:26-28`
  - body: The new HTTP connector takes provider API keys from plugin config, but the manifest declares `http.apiKey` and channel override `apiKey` fields as plain strings with no `configContracts.secretInputs` or sensitive UI hints, and the resolver only accepts strings. Bundled plugin credentials would have to live as plaintext config and would not be resolved/redacted through the existing SecretRef pipeline.
  - confidence: 0.93
- **[P2] Bound HTTP provider initialization waits:** `extensions/guardrails/index.ts:177-178`
  - body: If a registered HTTP provider's `init()` never settles, this path awaits `entry.initPromise` on every matching `before_dispatch` event with no timeout. That blocks dispatch indefinitely instead of returning the configured fallback, so provider initialization should be covered by the same timeout/fallback semantics as checks.
  - confidence: 0.84
- **[P2] Bound import connector initialization waits:** `extensions/guardrails/index.ts:321-322`
  - body: If an import connector's module load or `init(args)` never settles, the handler waits on `entry.initPromise` before the per-check timeout wrapper exists. A bad custom checker can therefore hang external-channel dispatch instead of honoring `timeoutMs`/`fallbackOnError`.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces credential-bearing outbound checks and trusted dynamic code execution, and the current PR head does not wire provider keys into the existing SecretRef/sensitive config pipeline.

Concerns:

- **[high] Plaintext provider API keys:** `extensions/guardrails/openclaw.plugin.json:26`
  - body: `http.apiKey` is declared as a plain string with no SecretRef contract or sensitive UI hint, including per-channel override keys. That would push guardrail provider tokens into normal plugin config instead of the existing secret-resolution/redaction path.
  - confidence: 0.93
- **[medium] Trusted local code execution surface:** `extensions/guardrails/src/import-connector.ts:23`
  - body: The import connector executes operator-specified local modules through `jiti`. That may be acceptable for trusted deployments, but as a bundled plugin it needs explicit maintainer security review, docs, and threat-model clarity around absolute paths and hot reload.
  - confidence: 0.86
- **[medium] Unbounded init can stall inbound dispatch:** `extensions/guardrails/index.ts:177`
  - body: HTTP and import connector initialization promises are awaited from `before_dispatch` without a timeout, so a hung provider/module initialization becomes an availability issue at a high-risk ingress boundary.
  - confidence: 0.86
