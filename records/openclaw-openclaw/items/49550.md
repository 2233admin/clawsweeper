## Summary

Keep open. Current main already implements a safer version of the raw JSON `api_error` half of this PR, including MiniMax-style `unknown error, 520` payloads and regression coverage for billing/auth/format shadowing. It does not fully implement the PR's explicit HTTP 520 status handling: 520 is still absent from the transient HTTP status set and the HTTP-status classifier has no 520 branch. The PR is also stale against the safer current classifier and should not be merged as-is.

## Best Possible Solution

Rework this PR or replace it with a smaller PR that preserves current main's safer transient-signal `api_error` classifier, then adds only the missing HTTP 520 status support in `TRANSIENT_HTTP_ERROR_CODES` and `classifyFailoverClassificationFromHttpStatus`, with focused tests for `isTransientHttpError("520 ...")`, `classifyFailoverReasonFromHttpStatus(520)`, and the existing non-transient api_error guard cases.
