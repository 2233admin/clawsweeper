## Summary

Keep open. Current main implements the safer raw JSON api_error behavior, including the MiniMax-style `unknown error, 520` payload and billing/auth/format guards, but it still does not implement the PR's explicit HTTP 520 status handling. The PR remains directionally useful, but should be rebased/reworked to add only the missing 520 status support without weakening the current classifier.

## Best Possible Solution

Keep the PR open for maintainer follow-up, or replace it with a smaller patch that preserves current main's api_error transient-signal classifier and adds only HTTP 520 support in `TRANSIENT_HTTP_ERROR_CODES`, `classifyFailoverClassificationFromHttpStatus`, and focused tests for the 520 status paths.
