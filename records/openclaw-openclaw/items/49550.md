## Summary

Keep this PR open. Current main already has the safer MiniMax-style JSON `api_error` handling from merged #49611, including transient-signal and billing/auth/format guards, but it still does not implement this PR's explicit HTTP 520 status handling in the transient status set or direct HTTP status classifier.

## Best Possible Solution

Rebase or replace this PR with a narrow patch that adds 520 to `TRANSIENT_HTTP_ERROR_CODES`, classifies direct HTTP 520 as `timeout`, and adds focused tests for `isTransientHttpError`, `classifyFailoverReasonFromHttpStatus(520)`, and structured `resolveFailoverReasonFromError({ status: 520 })`, while preserving current main's #49611 transient-signal `api_error` guard and non-transient/billing/auth regression coverage.
