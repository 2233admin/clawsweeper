## Summary

Keep open. The report is recent and concrete, and current main still has a source-backed timeout plumbing gap in the browser snapshot path: the browser tool exposes a top-level timeout, and the gateway/node proxy can enforce one when supplied, but the snapshot action/client path still omits it and falls back to a fixed 20s client timeout.

## What This Changes

The issue requests that browser snapshot actions return page content or an accessibility tree instead of timing out after browser open/navigate succeeds.

## Best Possible Solution

Keep this open and fix the browser plugin snapshot path so per-call timeoutMs and cancellation are handled consistently across the agent tool, host client, gateway browser.request proxy, node proxy, and /snapshot execution, with focused regression tests around host and node snapshot calls.

## Reproduction Assessment

Yes, for triage: the issue gives a clear open/navigate/snapshot sequence, affected version, OS, gateway status, and repeated cross-URL failure. I did not run the macOS browser flow live, but current source inspection provides a concrete path that can reproduce timeout handling failure by passing timeoutMs to snapshot and observing it is dropped.

## Solution Assessment

Yes, the narrow maintainable fix is in the browser plugin snapshot timeout/cancellation plumbing. Changing core gateway policy, browser profile defaults, or generic tool timeouts would be broader than the evidence supports.
