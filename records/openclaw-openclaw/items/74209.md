## Summary

Keep open. Current main contains useful Bonjour and discovery-startup mitigations, but the central bundled-plugin/default-startup policy remains unresolved: Bonjour is still default-enabled at startup, bundled default sidecars are still selected for gateway startup, and runtime-deps staging still includes default-enabled non-provider plugins.

## What This Changes

The issue asks for upgrades and gateway startup to avoid enabling or initializing broad unused bundled plugins, especially Bonjour, in a way that can block startup or normal operation.

## Best Possible Solution

Define an explicit configured-only or minimal-profile startup policy for upgrades and recovery, preserve explicit enables/disables and allowlists, keep Bonjour bounded and non-blocking, and document the operator recovery path for disabling unused bundled plugins and restoring provider registration.

## Reproduction Assessment

Yes for the remaining policy shape: current main statically shows default-enabled Bonjour, default bundled startup inclusion, and default-enabled runtime-deps staging. The exact small-host Linux watchdog/restart churn is environment-specific and was not live-reproduced here.

## Solution Assessment

No. Current main's Bonjour restart cap and discovery advertise timeout are good narrow mitigations, but they do not fully solve the requested default bundled-plugin/minimal-profile startup behavior.
