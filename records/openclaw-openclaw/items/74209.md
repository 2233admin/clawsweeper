## Summary

Keep open. Current main contains the narrow Bonjour restart-cap mitigation, but it still keeps Bonjour in the enabled-by-default startup path and preserves broader default-enabled bundled plugin/runtime-dependency behavior; the maintainer follow-up and later macOS data point both state that the default/minimal plugin policy remains unresolved.

## What This Changes

The issue asks OpenClaw to prevent upgrades and gateway startup from enabling or initializing broad optional bundled plugins, especially Bonjour, in a way that can block startup or make unused plugins affect operators.

## Best Possible Solution

Keep this issue as the canonical maintainer tracking item for the default bundled-plugin startup policy. The desired end state is a gateway startup and recovery path that loads/stages only configured or genuinely required plugins by default, keeps Bonjour and other optional discovery/provider/channel services non-blocking or opt-in when appropriate, and documents an operator-safe minimal profile with targeted tests across Bonjour, startup planning, runtime-deps staging, status/health, and disabled-channel behavior.

## Reproduction Assessment

Unclear. The issue has detailed logs and a plausible upgrade path, and current-main source checks reproduce the remaining default-on/default-enabled policy, but the exact startup timeout and task-lock churn have not been live-reproduced on the affected Debian/npm-global host from this read-only review.

## Solution Assessment

No. The landed c39ca49c71 watchdog cap is a useful narrow mitigation for Bonjour flapping, but it is not the full solution to the broader default bundled-plugin/minimal-profile startup problem; related open PRs such as #74778 and #74820 should be reviewed as narrow Bonjour follow-ups, not treated as closing the policy issue.
