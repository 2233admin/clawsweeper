## Summary

Keep open. Current main contains a narrow Bonjour restart-loop mitigation and runtime-dependency staging improvements, but the core report is broader: Bonjour is still enabled and startup-activated by default, gateway startup still stages/loads selected default-enabled bundled plugin surfaces, and maintainer discussion explicitly says the default bundled-plugin/minimal-profile policy remains unresolved.

## What This Changes

The issue asks OpenClaw to prevent default bundled plugins, especially Bonjour, from making upgraded gateways unusable by making optional startup plugins minimal, opt-in, or fully non-blocking.

## Best Possible Solution

Keep the issue as the policy tracker until maintainers decide whether optional bundled startup services such as Bonjour should become opt-in/minimal after upgrade, or stay default-on with stronger guarantees that failures cannot delay readiness, churn restarts, or stage unused dependencies. Narrow Bonjour fixes can land separately, but the durable end state should include startup planning, runtime-dependency selection, operator recovery docs, and a safe/minimal profile or equivalent migration path.

## Reproduction Assessment

Unclear for an exact live reproduction: the issue body has detailed Debian/npm-global upgrade steps and logs, and current main verifies the default-on startup path, but I did not reproduce the full constrained-host upgrade failure locally. The current source checks do give a high-confidence way to verify the remaining policy gap.

## Solution Assessment

No, current main is not the complete best solution. The c39ca49c71-style watchdog cap is a useful narrow mitigation, but the broader fix needs maintainer policy around default bundled-plugin activation and safe startup/recovery behavior.
