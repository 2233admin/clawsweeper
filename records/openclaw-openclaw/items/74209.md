## Summary

Keep open. Current main contains meaningful mitigations for Bonjour churn and bundled runtime-dependency staging, but it still keeps Bonjour enabled-by-default/startup-loaded and still treats default-enabled bundled plugins as active unless operators explicitly constrain them. The reporter follow-ups, maintainer comment, and later macOS data point all frame the remaining problem as broader default/minimal-profile policy, not only the already-landed Bonjour restart cap.

## What This Changes

The issue reports that upgrading through the 2026.4.26 plugin set can make the gateway unusable because many bundled/default plugins, especially Bonjour, may load or repair dependencies during startup until the operator trims the install to a minimal configured plugin set.

## Best Possible Solution

Keep the issue open until maintainers choose and implement the intended startup policy: either a configured-only/minimal profile after upgrade, or proof that default-on bundled services such as Bonjour can never delay readiness, restart the gateway, or stage unused dependency trees on constrained hosts. The narrow Bonjour fixes can continue separately, but the broad end state should align manifest defaults, Gateway startup planning, runtime-deps selection, status/health reporting, and operator recovery docs.

## Reproduction Assessment

Partially yes. The narrow Bonjour probing/announced flap has a high-confidence unit reproduction in extensions/bonjour/src/advertiser.test.ts, and the reporter supplied concrete upgrade logs; however, the full packaged Linux upgrade/minimal-profile failure was not replayed in this read-only sweep.

## Solution Assessment

No for closing this issue. Current main is a good partial mitigation for one Bonjour churn loop, but the best complete solution still needs maintainer policy and implementation across plugin activation/defaults and runtime-deps startup planning.
