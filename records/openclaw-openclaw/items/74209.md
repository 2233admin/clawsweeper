## Summary

Keep open. Current main includes a narrow Bonjour watchdog mitigation, but the broader reported upgrade/default-plugin problem remains: Bonjour is still default-enabled and startup-loaded, default-enabled startup plugins are still admitted by the gateway planner, and bundled runtime-dependency selection still includes default-enabled non-provider plugins. The maintainer comment and later macOS data point both explicitly say the minimal/configured-only policy question is unresolved.

## What This Changes

Request that upgrades avoid enabling or loading a broad unused bundled plugin set by default, and that Bonjour or other optional plugin failures never block gateway readiness or normal message handling.

## Best Possible Solution

Keep the narrow Bonjour watchdog mitigation, then decide and implement the intended upgrade/startup policy: either make optional bundled services such as Bonjour opt-in/minimal after upgrade, or prove default-on plugins are non-blocking and only configured/needed runtime dependencies affect readiness, status, and health output. The linked Bonjour PRs can be reviewed separately, but the end state should include operator recovery documentation or a safe minimal profile.

## Reproduction Assessment

Yes for the remaining policy behavior: source inspection shows Bonjour remains default-enabled and startup-loaded, default-enabled startup plugins remain accepted, and runtime-deps selection still includes default-enabled non-provider plugins. The exact Debian restart/resource churn was not live-reproduced here, but the issue body has detailed logs and recovery evidence.

## Solution Assessment

No current main is not the complete solution. The landed Bonjour sliding-window cap is a useful narrow mitigation, but the requested configured-only/minimal default policy and unused-plugin isolation still need maintainer direction.
