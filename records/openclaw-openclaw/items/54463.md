## Summary

Keep this open. Current main still has no OpenClaw-side QMD traversal policy, exclusion knob, version gate, or regression coverage that would make the reported symlink-loop failure impossible across supported QMD installs; the existing docs still describe the case as a workaround-only runtime limitation.

## What This Changes

QMD-backed memory indexing should avoid `ENAMETOOLONG` from symlink loops in workspace-visible temporary monorepos and keep docs aligned with runtime symlink handling.

## Best Possible Solution

Keep this as an active QMD traversal robustness item. Maintainers should decide whether OpenClaw should require and verify a safe QMD version, update docs around that contract, add an OpenClaw-side pre-update exclusion/version guard, or coordinate an upstream QMD fix; any accepted path should include a regression case for a workspace-visible monorepo symlink loop.
