## Summary

Keep open. Current main still ships the WhatsApp plugin on Baileys 7.0.0-rc.9, the local Baileys patch does not touch the auth/login handshake paths discussed in the report, and the current QR login path still waits on that same Baileys socket after logging `WhatsApp QR received.` Later WhatsApp reliability work improves transport liveness and timeout configuration, but the provided maintainer PR context explicitly says it does not replace this separate QR-pairing report.

## Best Possible Solution

Keep the issue open and route it to WhatsApp plugin maintainers. The best path is to decide the dependency/backend strategy for Baileys QR pairing, then implement a focused WhatsApp-owned fix with regression coverage for QR login failure/retry/error surfacing and live Docker/macOS ARM64 QR-pairing proof. Existing related work such as #47710 and #63939 should be reviewed as adjacent candidates, not treated as closing evidence until a fix lands.
