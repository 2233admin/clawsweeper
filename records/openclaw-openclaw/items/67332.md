## Summary

Keep this PR open. Current main still lacks the requested owner-only task/team tools, so the branch is not obsolete, but the proposed patch has blocking correctness and authorization review issues and remains stacked on open #67331.

## What This Changes

This PR adds owner-only `task` and `team` agent tools, an in-memory team registry, owner-only registry wiring, task/team tests, and the filesystem permission/error hardening changes from its dependency branch.

## Best Possible Solution

Keep the PR as an implementation candidate, but require changes or split it: land or separate #67331 first, then wire task/team tools through existing owner-scoped task helpers, repo schema helpers, explicit action validation, intentional team persistence semantics, and focused authorization/error-mapping tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Import Type from the installed typebox package:** `src/agents/tools/task-tool.ts:1`
  - body: The new task tool imports `Type` from `@sinclair/typebox`, and `team-tool.ts` does the same, but current `package.json` only declares `typebox` and the repo has migrated imports to that package. This will fail type resolution/build unless the imports are changed to `typebox`.
  - confidence: 0.93
- **[P1] Use owner-scoped task access for the task tool:** `src/agents/tools/task-tool.ts:140`
  - body: The tool is advertised as owner-only, but its get/list/update/output paths read and mutate raw task registry records by caller-supplied lookup or ownerKey. Current main already has `task-owner-access` helpers that enforce session-scoped ownerKey matching; use those with the caller's session owner instead of raw registry access so one owner-scoped run cannot inspect or change unrelated task records by ID.
  - confidence: 0.84
- **[P2] Use flat enum schemas for tool inputs:** `src/agents/tools/task-tool.ts:31-39`
  - body: `Type.Union(Type.Literal(...))` generates `anyOf`, which the repo's `stringEnum` helper explicitly avoids because some providers reject those schemas. Replace the task action/runtime/notifyPolicy unions, and the matching team action union, with `stringEnum`/`optionalStringEnum`.
  - confidence: 0.91
- **[P2] Classify allowed-mount escapes as workspace boundaries:** `src/agents/fs-permission-denied.ts:22-26`
  - body: Several new sandbox bridge call sites wrap errors containing `Sandbox path escapes allowed mounts`, but the classifier only recognizes `escapes sandbox root` and outside-root messages. Those permission errors will get `reason=unknown` instead of `workspace_boundary`, breaking callers that branch on the structured reason.
  - confidence: 0.9
- **[P2] Validate task action before output lookup:** `src/agents/tools/task-tool.ts:232-233`
  - body: After the explicit create/get/list/stop/update branches, every other action falls through to the output lookup. Direct callers such as `/tools/invoke` call `execute` without TypeBox validation, so a typo like `ouptut` can return task output or a misleading not-found error instead of `unsupported action`.
  - confidence: 0.88
- **[P2] Return ToolInputError for missing team updates:** `src/agents/tools/team-tool.ts:88-93`
  - body: `updateTeam` throws a plain `Error` when the team ID is unknown, and the tool does not translate it. `/tools/invoke` maps only `ToolInputError`/authorization errors to 4xx, so an ordinary missing team update becomes a 500 internal failure instead of a correct client error.
  - confidence: 0.87
