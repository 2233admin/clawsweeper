## Summary

Keep this PR open. Current main still has the Feishu mention-cascade behavior, and the PR is a valid focused attempt to remove it, but the latest branch still drops the stable-ID/card conversion path needed for explicit mention-forward replies and already has active maintainer handoff discussion.

## What This Changes

The PR removes Feishu inbound `mentionTargets` from reply dispatch, changes mentioned users to sanitized agent prompt context, adds no-auto-mention regression tests for text/card/streaming-card replies, and adds a changelog entry.

## Best Possible Solution

Land a Feishu-owned fix that disables unconditional inherited mention propagation for fallback, error, ack, ordinary, card, and streaming replies while preserving intentional explicit mentions through stable allowlisted `{openId, name}` context and a safe explicit-only text/card conversion path.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce the bug: incoming Feishu mentions become `ctx.mentionTargets`, those targets are passed into reply dispatch, and the send helpers turn them into outbound Feishu mention markup.

## Solution Assessment

No. The PR is directionally correct for the cascade, but the maintainable fix also needs to preserve explicit mention semantics without reintroducing automatic prefixing.

## Review Findings

  - body: After dispatcher-side mention metadata is removed, explicit mention forwarding has to come from model-visible context. This hint only exposes sanitized display names, while post/rich-text mentions can render as `@name` with the stable `open_id` kept internal, so duplicate or renamed users cannot be converted safely to notifying mentions.
  - confidence: 0.86

Feishu card mentions rely on the `<at id=...></at>` conversion path, so explicit mention replies in card/streaming mode can render as plain text unless an explicit-only rewrite replaces the removed automatic prefixing.
  - confidence: 0.84
