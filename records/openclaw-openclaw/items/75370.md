## Summary

Keep open. Current main still lacks the requested Poppler fallback, so the PR is not obsolete, but this branch is not merge-ready: it introduces unsafe temp-file handling for user PDFs, unbounded synchronous external-process execution, and a rendering path that can violate the existing pixel budget.

## What This Changes

The PR adds synchronous `pdftoppm` availability detection and a page-rendering fallback in `extensions/document-extract/document-extractor.ts` when loading `@napi-rs/canvas` fails.

## Best Possible Solution

Keep the PR open for revision only if maintainers accept a Poppler fallback: implement it inside `document-extract` with private restrictive temp files or file descriptors, argv-based process execution, timeout/abort and output bounds, pixel-budget-faithful rendering, focused regression tests, docs, and a changelog entry. If that external binary contract is not accepted, the safer end state is to surface a clear degradation/error path instead of silent text-only output.

## Reproduction Assessment

Yes. The linked issue gives a concrete headless Linux setup without `@napi-rs/canvas`, and current source shows that path returns empty images; the PR defects are statically reproducible from the added lines that write `/tmp`, call `execSync`, and force a 72 DPI minimum.

## Solution Assessment

No. A Poppler fallback is a plausible owner-module fix, but this implementation is not the best way because it adds unsafe temp-file handling, unbounded synchronous execution, and pixel-budget drift; a bounded `execFile`/spawn implementation with private temp storage and docs/tests would be the safer path.

Full review comments:

- **[P2] Honor the pixel budget in the pdftoppm path:** `extensions/document-extract/document-extractor.ts:180`
  - body: For any page where `resolveRenderPlan()` returns `scale < 1`, this line forces the fallback back to 72 DPI. The code then subtracts only `plan.pixels`, so large pages can generate images above `request.maxPixels`, defeating the existing guard and risking oversized prompt payloads.

  - body: This writes the user's PDF to a predictable path under `/tmp` with default `writeFileSync` permissions before invoking Poppler. On shared Linux hosts that can expose PDF contents or allow local race/symlink attacks; use a private temp directory/file with restrictive mode and clean it up.
  - confidence: 0.88

  - body: The renderer calls `pdftoppm` through `execSync` with no timeout or abort path while processing untrusted PDFs. If Poppler hangs or takes a long time, this blocks the Gateway event loop; use argv-based `execFile`/spawn with a timeout and output bounds.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces external PDF rendering with unsafe temp-file and process-execution handling that needs revision before merge.

Concerns:

- **[high] PDF contents written to shared temp paths:** `extensions/document-extract/document-extractor.ts:183`
  - body: The fallback writes user-supplied PDF bytes to predictable `/tmp/openclaw-pdf-*` paths with default file permissions, then deletes them later. Sensitive PDFs can be exposed on shared hosts, and predictable global temp paths introduce local race/symlink risk.
  - confidence: 0.88
- **[medium] Unbounded synchronous Poppler execution:** `extensions/document-extract/document-extractor.ts:189`
  - body: The render path invokes `pdftoppm` via `execSync` without timeout or abort handling. A hostile or malformed PDF can keep the Gateway process blocked while the external renderer runs.
  - confidence: 0.86

## Risks / Open Questions

- The branch writes user-supplied PDFs to predictable shared `/tmp` paths using default permissions before cleanup, which can expose document contents on multi-user hosts and creates local race/symlink risk.
- The branch runs `pdftoppm` through synchronous `execSync` without timeout or abort handling, so a malformed or expensive PDF can block the Gateway event loop.
- The fallback renderer ignores `resolveRenderPlan()` whenever `plan.scale < 1` by forcing at least 72 DPI, so generated images can exceed the configured pixel budget while accounting says they did not.
- The PR adds a user-facing fallback but does not update `CHANGELOG.md`, `docs/tools/pdf.md`, or focused tests for the new optional Poppler dependency contract.
