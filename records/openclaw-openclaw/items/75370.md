## Summary

Keep open. Current main still has the canvas-missing PDF image fallback bug, so the PR is not obsolete, but the proposed patch is not merge-ready because it adds external rendering for untrusted PDFs with a pixel-budget bypass, synchronous shell execution, and missing release/docs coverage.

## What This Changes

The PR changes `extensions/document-extract/document-extractor.ts` to detect `pdftoppm` and render selected PDF pages through temporary files when `@napi-rs/canvas` cannot load.

## Best Possible Solution

If maintainers accept Poppler as an optional fallback, keep it inside the `document-extract` plugin but make it bounded: use argv-based asynchronous process execution, avoid shell/PATH surprises, enforce the existing pixel/output/time budgets exactly, document the optional system dependency, add regression tests for the canvas-missing path, and add the required changelog entry.

## Reproduction Assessment

Yes. The original bug is reproducible on current main by sending a low/no-text PDF through extraction fallback without `@napi-rs/canvas`; the extractor catches the missing canvas module and returns no images, and the linked issue provides a concrete Ubuntu 24.04 setup. The PR defects are statically visible in the latest head at the 72-DPI floor and `execSync()` call.

## Solution Assessment

No. A Poppler fallback is a plausible owner-module fix, but this implementation is not the best way to solve it because it can exceed `request.maxPixels`, blocks the Gateway event loop while invoking an external renderer, and omits the docs/changelog/test coverage expected for a new system dependency path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Honor the render pixel budget:** `extensions/document-extract/document-extractor.ts:175`
  - body: When `resolveRenderPlan()` chooses `scale < 1`, this `Math.max(72, ...)` still renders at full 72 DPI in the Poppler path. Large or low-budget pages can produce more pixels than `request.maxPixels` while the loop subtracts only `plan.pixels`, so derive the Poppler output size from the render plan without the 72-DPI floor.
  - confidence: 0.94
- **[P2] Run Poppler without blocking the event loop:** `extensions/document-extract/document-extractor.ts:188-190`
  - body: This uses `execSync()` for every rendered PDF page. A malformed or expensive PDF can block the Node event loop for up to 30 seconds per page, and the interpolated command still routes paths through shell parsing; use `execFile` or `spawn` with argv, timeout/abort handling, and output bounds instead.
  - confidence: 0.9
- **[P3] Document the optional Poppler dependency:** `extensions/document-extract/document-extractor.ts:149-151`
  - body: The fallback only works when `pdftoppm` is installed and on PATH, but the PR does not update the PDF docs that currently describe the canvas-only image-rendering path. Add user-facing install/behavior notes for the optional Poppler renderer.
  - confidence: 0.85
- **[P3] Add the required changelog entry:** `extensions/document-extract/document-extractor.ts:149`
  - body: This is a user-facing PDF extraction behavior change, but the PR only changes plugin source. Add an entry under the active `CHANGELOG.md` section and credit the contributor per repo policy.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces external rendering of untrusted PDFs and still needs safer process execution plus hard resource-bound enforcement before merge.

Concerns:

- **[medium] Synchronous shell execution of external PDF renderer:** `extensions/document-extract/document-extractor.ts:188`
  - body: The patch invokes `pdftoppm` through `execSync()` with an interpolated shell command and bare binary lookup. Even with a timeout, hostile or malformed PDFs can block the Gateway event loop per page, and argv-based execution would avoid shell parsing of temp paths.
  - confidence: 0.9
- **[medium] Pixel budget bypass amplifies untrusted PDF rendering:** `extensions/document-extract/document-extractor.ts:175`
  - body: The Poppler fallback keeps a 72-DPI floor even when the render plan chose a smaller scale to satisfy `maxPixels`, allowing oversized PNG renders and larger memory or prompt payloads for attacker-supplied PDFs.
  - confidence: 0.92
