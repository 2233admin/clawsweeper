## Summary

Keep open. Current main still lacks the requested canvas-missing PDF image fallback, and the linked bug remains valid, but this draft PR is not merge-ready because it loses low-volume extracted text, still bypasses the render pixel budget in the Poppler path, and runs an external PDF renderer synchronously through a shell.

## What This Changes

This PR changes the bundled document-extract plugin to invoke `pdftoppm` from temporary files when `@napi-rs/canvas` cannot load, while refactoring PDF text and image extraction helpers.

## Best Possible Solution

Keep the work in the document-extract plugin if maintainers accept Poppler, but implement it as a bounded renderer: preserve partial extracted text, use argv-based async process execution, enforce time/output/pixel limits, clean temp files, add focused fallback tests, document the optional system dependency, and add the required changelog entry.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: a low-text PDF reaches image rendering, `loadCanvasModule()` throws when `@napi-rs/canvas` is absent, and the extractor returns no images.

## Solution Assessment

No. A Poppler fallback may be the right direction, but this implementation is not the best fix until it preserves text, avoids synchronous shell execution, honors the render budget, and documents/tests the new external renderer contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Preserve low-volume extracted text:** `extensions/document-extract/document-extractor.ts:241`
  - body: The refactor returns `""` whenever extracted text is below `minTextChars`, so PDFs with a little text now lose that text before image rendering and also return empty text when no renderer is available. Keep the extracted text for the final `{ text, images }` result while using the threshold only to decide whether images are needed.
  - confidence: 0.94
- **[P2] Honor the render pixel budget:** `extensions/document-extract/document-extractor.ts:175`
  - body: When `resolveRenderPlan()` chooses `scale < 1`, this `Math.max(72, ...)` still renders through Poppler at full 72 DPI. Large or low-budget pages can produce more pixels than `request.maxPixels` while the loop subtracts only the smaller planned pixel count.
  - confidence: 0.93
- **[P2] Run Poppler asynchronously without a shell:** `extensions/document-extract/document-extractor.ts:187-190`
  - body: This invokes `pdftoppm` via `execSync()` for every rendered page. A malformed or expensive PDF can block the Node event loop for up to 30 seconds per page, and argv-based `execFile` or `spawn` would avoid shell parsing of generated paths.
  - confidence: 0.9
- **[P3] Document the optional Poppler dependency:** `extensions/document-extract/document-extractor.ts:149-151`
  - body: The fallback only works when `pdftoppm` is installed and on `PATH`, but the PR does not update the PDF docs that currently describe the canvas-only image-rendering path. Add user-facing install and behavior notes for the optional renderer.
  - confidence: 0.86
- **[P3] Add the required changelog entry:** `extensions/document-extract/document-extractor.ts:149-151`
  - body: This is a user-facing PDF extraction behavior change, but the PR only changes plugin source. Add an entry under the active `CHANGELOG.md` section and credit the contributor per repo policy.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces external rendering of untrusted PDFs and still needs safer process execution plus hard resource-bound enforcement before merge.

Concerns:

- **[medium] Synchronous shell execution of external PDF renderer:** `extensions/document-extract/document-extractor.ts:187`
  - body: The patch invokes `pdftoppm` through `execSync()` with a command string. Even with a timeout, hostile or malformed PDFs can block the Gateway event loop per page, and argv-based execution would avoid shell parsing of generated file paths.
  - confidence: 0.9
- **[medium] Pixel budget bypass amplifies untrusted PDF rendering:** `extensions/document-extract/document-extractor.ts:175`
  - body: The Poppler fallback keeps a 72-DPI floor even when the render plan chose a smaller scale to satisfy `maxPixels`, allowing oversized PNG renders and larger memory or prompt payloads for attacker-supplied PDFs.
  - confidence: 0.92
