## Summary

Keep open. Current main still lacks the requested `pdftoppm` fallback, and the linked bug is concrete, but this PR is not merge-ready because the latest head still bypasses the render pixel budget and introduces synchronous shell execution for untrusted PDF rendering; it also lacks the required user-facing changelog/docs/test coverage.

## What This Changes

The PR modifies `extensions/document-extract/document-extractor.ts` to try `pdftoppm` when `@napi-rs/canvas` cannot load, rendering selected PDF pages through a temporary file and child process.

## Best Possible Solution

Keep the fallback inside the `document-extract` plugin boundary if maintainers accept Poppler: remove the DPI floor so the existing pixel budget remains hard, invoke `pdftoppm` through argv-based asynchronous process execution with timeout/output bounds, document the optional system dependency, add focused tests for the canvas-missing path, and add the required changelog entry.

## Reproduction Assessment

Yes. The original bug is statically reproducible from current main by following the low-text PDF path until `loadCanvasModule()` fails and returns `images: []`; the PR defects are likewise visible in the latest head at the DPI floor and `execSync` call.

## Solution Assessment

No. A Poppler fallback is a plausible owner-module fix, but this implementation is not the best solution because it can exceed `request.maxPixels` and can block the Gateway event loop while shelling out to an external renderer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Honor the render pixel budget:** `extensions/document-extract/document-extractor.ts:175`
  - body: When `resolveRenderPlan()` returns `scale < 1`, this `Math.max(72, ...)` still renders at full 72 DPI in the fallback path. Large or low-budget pages can therefore produce far more pixels than `request.maxPixels` while the loop subtracts only `plan.pixels`, so remove the 72-DPI floor or derive the Poppler output size from the plan exactly.
  - confidence: 0.94
- **[P2] Run Poppler without blocking the Gateway event loop:** `extensions/document-extract/document-extractor.ts:188-190`
  - body: This uses synchronous shell execution for every rendered page. A malformed or expensive PDF can block the Node event loop for up to 30 seconds per selected page, and the command string still routes temp paths through shell parsing; use `execFile`/`spawn` with argv, timeout/abort handling, and output limits instead.
  - confidence: 0.88
- **[P3] Add the required changelog entry:** `extensions/document-extract/document-extractor.ts:149`
  - body: This is a user-facing PDF extraction behavior change, but the PR only changes plugin source. Add an entry under the active `CHANGELOG.md` section and credit the contributor per repo policy.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR introduces external PDF rendering for untrusted inputs and still needs safer process execution plus resource-bound enforcement before merge.

Concerns:

- **[medium] Synchronous shell execution of external PDF renderer:** `extensions/document-extract/document-extractor.ts:188`
  - body: `pdftoppm` is invoked through `execSync()` with an interpolated shell command. Even with a timeout, hostile or malformed PDFs can block the Gateway event loop per page, and argv-based execution would avoid shell parsing of temp paths.
  - confidence: 0.88
- **[medium] Pixel budget bypass amplifies untrusted PDF rendering:** `extensions/document-extract/document-extractor.ts:175`
  - body: The fallback keeps a 72-DPI floor even when the render plan chose a smaller scale to satisfy `maxPixels`, allowing oversized PNG renders and larger prompt/memory payloads for attacker-supplied PDFs.
  - confidence: 0.92
