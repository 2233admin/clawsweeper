## Summary

Keep open. Current main still has the reported pre-approval path: unpaired node reconnect creates a pending node-pair request but keeps declared commands live, and `node.invoke` authorizes from that live session command list. The provided context also shows open, unmerged PR #65169 from the same author with closing syntax for this issue, so this issue should remain open until that PR or an equivalent fix is reviewed and landed or explicitly rejected.

## What This Changes

The issue reports that an `operator.write` caller can invoke declared commands such as `system.run` on a reconnecting device-paired node before node-pair approval exists, despite the documented pairing gate.

## Best Possible Solution

Keep this issue open and review PR #65169 as the implementation candidate. If the documented node-pairing gate is intended, land that PR or an equivalent fix that publishes no live node commands until approved node-pair state exists and adds regression coverage around `node.invoke`; if maintainers decide current trusted-operator behavior is intentional, close the PR explicitly and update pairing/security docs plus tests together.
