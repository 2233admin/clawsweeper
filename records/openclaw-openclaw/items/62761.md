## Summary

Keep open. Current main contains the later `is_from_me` outbound-row mitigation, but it still does not implement the requested legacy iMessage restart replay cursor: `watch.subscribe` is called without a `since_rowid`/rowid/start fence, and targeted search found no per-account persisted cursor state. The related closed reports explicitly consolidated their remaining replay work onto this issue, so this remains the canonical tracker.

## What This Changes

The issue requests legacy iMessage cursor persistence so update, crash, or gateway restarts do not replay already-processed Messages DB rows as new inbound messages.

## Best Possible Solution

Implement the replay fence in the legacy iMessage plugin: persist a per-account last processed rowid or startup high-water marker, pass it to `watch.subscribe` as `since_rowid` when available, advance it only after safe handling or intentional drops, and cover restart, self-chat, and iCloud-sync replay cases with focused tests.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: start the legacy iMessage monitor and inspect the `watch.subscribe` request, which currently contains only `attachments`; the issue and #72322 provide live restart/iCloud-sync steps for the observable replay.

## Solution Assessment

Yes. The narrow durable fix is a plugin-owned rowid/high-water cursor passed to upstream `watch.subscribe`; the existing `is_from_me` filter is useful but does not solve backlog replay after restart.
