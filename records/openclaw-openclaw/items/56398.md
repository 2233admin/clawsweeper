## Summary

Keep open. The reported approval-flow bug still exists on current main, but the PR targets the older inline fast-fail location; current main now runs the no-route decision through the shared approval helper, so this needs maintainer review or an adapted patch rather than cleanup closure.

## What This Changes

This PR adds a two-phase guard to the exec approval no-route fast-fail path and adds/renames gateway server-method tests for two-phase pending behavior and single-phase fast-fail behavior.

## Best Possible Solution

Preserve the intended behavior on current main by adding the equivalent two-phase no-route guard at the shared pending-approval helper, keeping single-phase fast-fail intact, and adding a regression test for two-phase exec approvals with no approval clients, no forwarding, and no turn-source route. Maintainers should decide whether to land that as a narrow adapted fix or via the broader #56581 consolidation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `exec.approval.request` with `twoPhase: true`, no approval clients, no accepted forwarding delivery, and no turn-source route; current main reaches the shared no-route expiry branch before sending the two-phase `accepted` response.

## Solution Assessment

Yes for the original base, but not as-is for current main. The proposed `!twoPhase` guard is the narrow behavior change, but it now belongs in `handlePendingApprovalRequest` with regression coverage rather than only in the older `exec-approval.ts` branch.
