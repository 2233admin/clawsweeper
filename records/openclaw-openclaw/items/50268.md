## Summary

Keep open. Current main has a partial configure-focused preservation path from #70793, but the provider API-key path can still pre-clobber `agents.defaults.model.primary` via provider `applyConfig`, and `openclaw onboard --auth-choice ...` still calls auth setup with `setDefaultModel: true` without opting into preservation. The Keep My Claw comment is a useful restore workaround, but it does not fix the core config overwrite path.

## Best Possible Solution

Keep this open and fix the generic provider auth flow so provider setup can merge provider config and allowlist entries without clobbering an existing `agents.defaults.model.primary` unless the caller explicitly requested a new default. The fix should cover both configure and rerun onboarding, and add regression coverage with a real-provider-style `configPatch` that includes `agents.defaults.model.primary`.

## Clownfish Work Prompt

Fix the provider-onboarding default-model clobber described in #50268. Current main partially fixed configure in #70793, but real provider auth methods still return config patches generated by provider `applyConfig`, and those patches can already overwrite `agents.defaults.model.primary` before `preserveExistingDefaultModel` runs. Also, `openclaw onboard --auth-choice ...` still calls `applyAuthChoice` with `setDefaultModel: true` and no preservation option.

Expected boundary:
- Preserve an existing `agents.defaults.model.primary` when configure or rerun onboarding is adding provider auth/config, unless the user explicitly uses an intentional default-setting path such as `openclaw models auth login --provider <id> --set-default` or `openclaw models set <model>`.
- Keep first-run onboarding behavior: if no primary exists, provider setup may choose the provider default.
- Do not remove provider config or allowlist additions; the new provider's model should remain available.
- Add regression coverage for a provider auth method whose `configPatch` already contains `agents.defaults.model.primary`, not only the mocked `defaultModel` path.
- Include an xAI/OpenRouter-style scenario with an existing Bedrock/Anthropic primary and a newly added provider default.
- Update `docs/cli/onboard.md` or `docs/concepts/models.md` if the rerun-onboard behavior becomes documented. Add a changelog Fixes entry if this PR changes user-visible behavior, crediting `Thanks @rixcorp-oc`.

Likely files: `src/plugins/provider-auth-choice.ts`, `src/plugins/provider-api-key-auth.ts`, `src/wizard/setup.ts`, `src/commands/auth-choice.test.ts`, `src/commands/auth-choice.apply.plugin-provider.test.ts`, `src/wizard/setup.test.ts`, `docs/cli/onboard.md`, `docs/concepts/models.md`. Avoid provider-specific special cases in core; use the generic auth/default-model seam.
