## Summary

Keep open. Current main still has the clobber path for real plugin provider auth methods: the provider config patch can write a new primary model before the preservation helper reads the previous primary, so the configure preservation contract from #70793 is not guaranteed for xAI/OpenRouter-style providers. The Keep My Claw comment is a useful recovery workaround, but it does not remove the OpenClaw config mutation bug.

## What This Changes

The issue requests that onboard/configure add a second provider without silently replacing an existing `agents.defaults.model.primary`.

## Best Possible Solution

Preserve the pre-auth primary model through the plugin provider auth application path and base preservation decisions on that original value, while keeping explicit default-setting commands authoritative and adding regression coverage for provider patches that already write `agents.defaults.model.primary`.

## Repair Work Prompt

Fix the provider-auth default-model preservation bug tracked by https://github.com/openclaw/openclaw/issues/50268. Observable bug: when configure/onboard adds an API-key provider such as xAI or OpenRouter, the provider `configPatch` can write `agents.defaults.model.primary` before the preservation helper reads the previous primary, so an existing Bedrock or other primary is silently replaced despite `preserveExistingDefaultModel: true`. Expected boundary: keep explicit default-setting flows such as `openclaw models auth login --set-default` authoritative, and keep no-existing-primary setup able to choose the new provider default. For configure-style preservation, carry the original pre-auth primary/model object through the plugin provider auth path and restore/preserve that original value even when the provider patch already contains a primary. Add regression coverage for a real xAI/OpenRouter-style provider auth result with both `configPatch` and `defaultModel`, including fallback preservation. Likely files: `src/plugins/provider-auth-choice.ts`, `src/plugins/provider-auth-choice-helpers.ts` if needed, `src/commands/auth-choice.test.ts`, `src/commands/auth-choice.apply.plugin-provider.test.ts`, `src/plugins/provider-auth-choice-helpers.test.ts`, `extensions/xai/onboard.test.ts`, and `extensions/openrouter/onboard.test.ts`. Do not change bundled provider default models, provider IDs, auth-profile storage semantics, or explicit `--set-default` behavior. Add a user-facing changelog fix entry in the active version with reporter credit, likely `Thanks @rixcorp-oc`. Validation: run targeted tests first, then `pnpm check:changed` in Testbox per repo policy.
