## Summary

Keep open. Current main still does not expose ACP child-process PID metadata through the `sessions_spawn` result, persisted ACP session metadata, or `session_status`, and the issue is already paired with open unmerged PR #74756, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

Request to expose process metadata, especially OS PID correlation, for local ACP child sessions created through `sessions_spawn(runtime="acp")` and related status/control surfaces.

## Best Possible Solution

Review #74756 or a replacement PR, settle an additive ACP process-metadata contract that can represent available and unavailable adapter/harness PIDs, persist it in ACP session metadata, and expose it through spawn/status surfaces before closing this issue.

## Reproduction Assessment

Yes. The missing field is reproducible by static inspection of current main: the ACP spawn result type, accepted response construction, tool return path, and persisted metadata all omit PID/process metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right narrow direction, but maintainers should decide whether the best schema is one `pid`, separate adapter and child-agent PIDs, or status-only metadata before landing the paired PR.
