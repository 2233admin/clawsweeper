## Summary

Current main still lacks a first-class process-correlation surface for ACP children spawned by `sessions_spawn`. The spawn response and `session_status` tool expose logical session/run state, while ACP metadata and runtime contracts persist ACP/backend/session identity rather than typed host process handles; related orphan/stderr issues overlap but do not supersede this narrower observability gap.

## What This Changes

The issue requests PID/process metadata or a safe kill/close control path for local ACP processes created by `sessions_spawn(runtime="acp", ...)`.

## Best Possible Solution

Define an additive ACP process-metadata contract, have ACPX populate adapter and child-agent process identifiers when available, persist explicit availability/unavailability in ACP session metadata, and expose that through `sessions_spawn`, `session_status`, and ACP status surfaces with scoped cancellation/termination behavior documented separately.

## Risks / Open Questions

- ACPX 0.6.1 appears to have an optional record-level PID string in status summaries, but current OpenClaw does not expose a typed adapter/child-agent PID contract and this does not prove child-agent PID availability.
- Exposing host PIDs or adding kill/close controls needs an explicit permission and security model so agents cannot misuse host process metadata or terminate unrelated processes.
- The implementation may need an upstream ACPX change if OpenClaw needs both adapter PID and child-agent PID rather than only a backend record-level process hint.
