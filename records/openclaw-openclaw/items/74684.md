## Summary

Keep open. Current main still lacks a first-class PID or process-correlation surface for ACP children spawned through `sessions_spawn`, and open PR #74756 is explicitly linked with closing syntax, so this issue should stay open until that PR or an equivalent fix is reviewed and merged.

## What This Changes

The issue requests process metadata, such as ACP adapter and child-agent PIDs or a scoped kill/close surface, for `sessions_spawn(runtime: "acp")` child sessions.

## Best Possible Solution

Review #74756 or a replacement that defines an additive ACP process-metadata contract, has ACPX populate available adapter and child-agent process identifiers, persists explicit availability or unavailability, and exposes that through `sessions_spawn` and ACP status surfaces with separately documented cancellation/termination behavior.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: on current main, the ACP spawn result, persisted ACP metadata, and session status tool paths have no typed PID/process fields to return for a running ACP child.

## Solution Assessment

Unclear. Exposing process metadata is the right direction, but the safest implementation needs maintainer agreement on the API/security model and whether a single PID is sufficient before the linked PR lands.
