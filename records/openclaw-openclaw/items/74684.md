## Summary

Keep open. Current main still lacks typed ACP process/PID metadata in the ACP spawn result, persisted ACP session metadata, runtime handle/status types, and the agent-facing `session_status` path; the issue is also paired with open unmerged PR #74756, so it should remain open until that PR or an equivalent replacement lands.

## What This Changes

Expose OS process metadata for ACP sessions created by `sessions_spawn(runtime="acp")` so operators can correlate child sessions with adapter or child-agent processes.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that represents available and unavailable adapter/child-agent identifiers, persists them with ACP session state, exposes them through `sessions_spawn` and ACP status surfaces, and keeps cancellation semantics separate from raw PID exposure.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path for this API-shape bug: the ACP spawn result, runtime contracts, persisted metadata, docs, and status tool output all omit typed process metadata.

## Solution Assessment

Unclear. First-class process correlation is the right direction, but maintainers should settle the typed schema, visibility/security model, and ACPX source of truth before treating the paired PR as final.
