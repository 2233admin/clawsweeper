## Summary

Keep open. Current main still does not expose ACP adapter or child process PID metadata through sessions_spawn, persisted ACP session metadata, or session_status, and the issue is paired with open unmerged PR #74756, so it should remain open until that PR or an equivalent replacement lands. https://github.com/openclaw/openclaw/issues/74684

## What This Changes

The issue asks for first-class process/PID metadata or a control surface to correlate running ACP sessions spawned by sessions_spawn(runtime="acp") with their local OS processes.

## Best Possible Solution

Land #74756 or a replacement additive ACP process-metadata contract that carries available/unavailable adapter and harness process identifiers through the ACP runtime handle/status, persists them on the session record, exposes them in sessions_spawn and status surfaces, and pairs observability with a first-class close/kill path rather than requiring manual ps matching.

## Reproduction Assessment

Yes. Static inspection of current main is enough to reproduce the missing surface: the ACP spawn result type and return object, runtime handle/status contract, persisted SessionAcpMeta, and status display path all omit typed PID/process metadata.

## Solution Assessment

Unclear. The requested direction is valid, but the best final fix should settle the stable schema and security model for exposing process identifiers and cancellation before treating the paired PR as complete.
