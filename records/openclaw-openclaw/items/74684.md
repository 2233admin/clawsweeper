## Summary

Keep open. Current main still lacks typed ACP process/PID metadata in the ACP spawn result, runtime handle/status contracts, persisted ACP session metadata, and status surfaces; the issue is also paired with open unmerged PR #74756, so it should remain open until that PR or an equivalent replacement lands.

## What This Changes

Expose first-class process/PID metadata for ACP sessions spawned through `sessions_spawn` so operators can correlate running ACP harness processes with OpenClaw session records.

## Best Possible Solution

Review and either land #74756 or replace it with an additive ACP process-metadata contract that represents available and unavailable adapter/harness process identifiers, persists the metadata, exposes it through `sessions_spawn` and status surfaces, and keeps cancellation or kill controls as explicit APIs rather than implicit raw-PID behavior.

## Reproduction Assessment

Yes. Static inspection of current main is a high-confidence reproduction path because the ACP spawn result, tool return path, runtime/status types, persisted session metadata, and docs all omit first-class PID/process correlation metadata.

## Solution Assessment

Unclear. Exposing process correlation is the right direction, but maintainers should decide whether the final shape is a single optional PID or a typed process-metadata object that can represent adapter and child harness processes safely.
