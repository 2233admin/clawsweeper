## Summary

Keep open. Current main still does not expose the source Telegram callback message ID to the generic agent session path: callback clicks fall through as synthetic messages with `messageIdOverride: callback.id`, the inbound context stores that override as `MessageSid`, and the agent metadata/threading helpers consume `MessageSid` rather than a separate callback source message ID. The issue also has an open PR with explicit closing syntax, so the best maintainer path is to review and land or close that implementation candidate rather than auto-close this issue early.

## Best Possible Solution

Keep this issue open until PR #55548 or an equivalent maintainer-reviewed fix lands. The likely product/code path is to add a typed callback source message id to the Telegram callback fallback, propagate it through `TelegramMessageContextOptions` and `MsgContext`, include it in agent-visible conversation metadata, and ensure edit/reply tooling prefers the source message id for callback-triggered turns without breaking existing `MessageSid` semantics.
