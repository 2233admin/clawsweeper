## Summary

Keep open. The latest PR head still contains the two previously identified workflow blockers despite the follow-up comment, and current main has no equivalent hosted-installer checksum guard. The live source/host drift remains observable, so this remains useful work but the patch is incorrect as-is.

## What This Changes

The PR adds a GitHub Actions workflow that fetches https://openclaw.ai/install.sh, compares its SHA-256 digest with scripts/install.sh, and runs on selected pushes, pull requests, a daily schedule, and manual dispatch.

## Best Possible Solution

Keep the drift guard idea, but revise the workflow so PR-time validation does not compare an in-flight installer source change against the already-deployed CDN script. Run the byte-for-byte source/host check after merge, on schedule, and by manual dispatch; separate schedule and push concurrency; keep #73837 open until the hosted installer is actually redeployed and the guard proves future drift.

## Reproduction Assessment

Not applicable as a feature/CI PR, but the CI failure mode has a high-confidence proof path: the latest workflow still triggers on scripts/install.sh pull requests, and the live SHA-256 check shows source and hosted bytes differ. That means an installer-source PR would fail before deployment for the same reason this new workflow is currently failing.

## Solution Assessment

No. A read-only scheduled/main checksum guard is a reasonable direction, but the current PR trigger and concurrency design are not the narrowest maintainable implementation; PR validation should use workflow-sanity or a PR-produced expected artifact instead of checking proposed source bytes against the live host.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Limit PR-time source-host comparisons:** `.github/workflows/check-hosted-installer-sync.yml:11`
  - body: This workflow runs on pull requests that change scripts/install.sh, but a PR checkout contains the proposed installer while openclaw.ai still serves the last deployed installer. Because the job fails on any hash mismatch, future installer-source PRs will fail until after merge and deployment. Remove this path from pull_request validation or make PR validation compare against a PR-produced expected artifact.
  - confidence: 0.97
- **[P2] Separate scheduled checks from push cancellation:** `.github/workflows/check-hosted-installer-sync.yml:22-23`
  - body: schedule and push runs on main both use refs/heads/main for github.ref, so cancel-in-progress can let a main push cancel the daily drift check. Include github.event_name in the concurrency group, or otherwise avoid cancelling scheduled drift checks from push runs.
  - confidence: 0.91
