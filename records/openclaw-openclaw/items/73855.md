## Summary

Keep this PR open. Current main does not include the proposed hosted-installer drift workflow, and the linked hosted-installer problem remains observable: `scripts/install.sh` differs from `https://openclaw.ai/install.sh`. The PR is a plausible implementation candidate, but it needs revision before merge because its `pull_request` trigger compares an in-flight `scripts/install.sh` change against the already-deployed hosted script and is already failing on the PR head.

## Best Possible Solution

Keep this PR open as a focused implementation candidate for hosted installer drift, but revise it before merge: remove `scripts/install.sh` from the `pull_request` trigger or make PR validation compare against an expected artifact rather than the live CDN, separate scheduled concurrency from push concurrency, and land it with workflow-sanity/owner review. The linked installer issue should stay open until the hosted script is redeployed and the guard proves future source/host drift is caught.

## Risks / Open Questions

- The proposed `pull_request` trigger includes `scripts/install.sh`, so legitimate installer-source PRs compare PR bytes against the already-deployed hosted script and can fail until after deployment.
- The proposed concurrency group uses only workflow and ref; schedule and push runs on `refs/heads/main` can cancel each other, reducing the reliability of the daily drift alert.
- The new scheduled workflow depends on live `openclaw.ai` availability and could create noisy CI failures during CDN/site outages, although it uses read-only permissions and does not execute the downloaded script.
