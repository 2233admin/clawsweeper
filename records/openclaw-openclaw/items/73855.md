## Summary

Keep this PR open. Current main does not include a hosted-installer checksum workflow, and live verification still shows `scripts/install.sh` differs from `https://openclaw.ai/install.sh`. The PR is a plausible implementation candidate for the linked installer drift problem, but it needs revision before merge because the proposed `pull_request` trigger would compare in-flight installer changes against the already-deployed CDN copy and can deadlock legitimate installer PRs.

## Best Possible Solution

Keep this PR open, but revise it before merge: remove `scripts/install.sh` from the `pull_request` trigger or make PR validation compare against an expected artifact instead of the live CDN, separate scheduled concurrency from push concurrency, and land the workflow with CI/workflow owner review. The linked installer issue #73837 should remain open until the hosted script is redeployed and a guard proves future source/host drift is caught.

## Risks / Open Questions

- If merged as-is, the `pull_request` trigger on `scripts/install.sh` can fail legitimate installer changes because PR bytes are compared to a CDN copy that only updates after merge/deployment.
- The proposed concurrency group can let `schedule` and `push` runs on `refs/heads/main` cancel each other, reducing the reliability of the daily drift alert.
- A live-CDN checksum workflow can create noisy failures during openclaw.ai/CDN outages unless maintainers decide whether this should be required CI, a scheduled alert, or part of the deploy pipeline.
- The linked installer issue remains open, so this PR should stay available as an implementation candidate rather than being closed before the hosted script is redeployed and guarded.
