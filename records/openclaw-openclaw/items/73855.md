## Summary

Keep open. The PR is still a useful focused CI idea, but the live GitHub diff at head `64534f91b08ae0afe1919bcf20474f5257d153b0` still contains the two previously identified workflow blockers, so it is not mergeable as-is.

## What This Changes

The PR adds a new GitHub Actions workflow that compares `scripts/install.sh` against the hosted `https://openclaw.ai/install.sh` checksum on push, pull request, schedule, and manual dispatch events.

## Best Possible Solution

Keep the checksum guard, but make it mergeable by running the live source-vs-host comparison only after merge, on schedule, or by manual dispatch; pull-request validation should avoid comparing in-flight installer changes to deployed bytes, and the concurrency group should distinguish event names. The linked installer bug should remain open unless the installer source and hosted deployment issue are fixed separately.

## Reproduction Assessment

Yes for the PR-introduced CI failure mode: the current PR head still has the problematic trigger and concurrency lines, the source and hosted installer hashes differ, and the new workflow is failing on the PR head. This is not a user-facing bug reproduction path, but it is enough to reproduce the merge blocker.

## Solution Assessment

No. A read-only hosted-installer checksum guard is a reasonable mechanism, but this implementation is not the best mergeable form because it compares PR-head installer bytes against already-deployed bytes and lets scheduled checks share cancellation with main pushes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Limit live-host comparisons on pull requests:** `.github/workflows/check-hosted-installer-sync.yml:11`
  - body: This workflow runs on pull requests that change `scripts/install.sh`, but a PR checkout contains the proposed installer while `openclaw.ai` still serves the last deployed installer. Because the job fails on any hash mismatch, future installer-source PRs will fail until after merge and deployment; remove this path from `pull_request` validation or skip the live checksum comparison for pull_request events.
  - confidence: 0.97
- **[P2] Separate scheduled checks from push concurrency:** `.github/workflows/check-hosted-installer-sync.yml:22-23`
  - body: `schedule` and `push` runs on `main` both use `refs/heads/main` for `github.ref`, so `cancel-in-progress: true` can let a main push cancel the daily drift check. Include `github.event_name` in the concurrency group or otherwise avoid cancelling scheduled checks from push runs.
  - confidence: 0.91
