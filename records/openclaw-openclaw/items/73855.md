## Summary

Keep open. The drift guard is useful, but the latest PR head still contains the same two blocking workflow issues already reported: pull requests that edit `scripts/install.sh` would compare proposed source bytes against the already-hosted installer, and push/schedule runs still share the same cancellable concurrency group. Current main also has no equivalent checksum guard, and the live hosted installer still differs from `scripts/install.sh`, so this PR needs revision rather than cleanup closure.

## What This Changes

The PR adds a GitHub Actions workflow that fetches `https://openclaw.ai/install.sh`, hashes it, compares it with `scripts/install.sh`, and runs on pushes, pull requests, a daily schedule, and manual dispatch.

## Best Possible Solution

Keep the checksum guard idea, but revise it so byte-for-byte source/host comparison runs after merge, on a schedule, and by manual dispatch, while PR-time validation checks only workflow shape or a PR-produced expected artifact. Keep the related installer bug open until the hosted installer is actually redeployed and the guard proves the hosted bytes match source.

## Reproduction Assessment

Not applicable as a CI-feature PR, but the failure mode has high-confidence proof: the latest PR workflow still runs on `scripts/install.sh` pull requests, the live source and hosted hashes currently differ, and the PR head’s new check is already failing.

## Solution Assessment

No. A read-only hosted-installer drift guard is a reasonable direction, but the current PR trigger and concurrency design are not the narrowest maintainable implementation; it should avoid comparing proposed installer source changes against the already-deployed CDN script and should isolate scheduled runs from push cancellation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Limit PR-time source-host comparisons:** `.github/workflows/check-hosted-installer-sync.yml:11`
  - body: This workflow runs on pull requests that change `scripts/install.sh`, but a PR checkout contains the proposed installer while `openclaw.ai` still serves the last deployed installer. Because the job fails on any hash mismatch, future installer-source PRs will fail until after merge and deployment. Remove this path from `pull_request` validation or make PR validation compare against a PR-produced expected artifact.
  - confidence: 0.97
- **[P2] Separate scheduled checks from push cancellation:** `.github/workflows/check-hosted-installer-sync.yml:22-23`
  - body: `schedule` and `push` runs on main both use `refs/heads/main` for `github.ref`, so `cancel-in-progress: true` can let a main push cancel the daily drift check. Include `github.event_name` in the concurrency group, or otherwise avoid cancelling scheduled drift checks from push runs.
  - confidence: 0.91
