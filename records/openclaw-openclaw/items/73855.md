## Summary

Keep open. Current main does not have an equivalent hosted-installer checksum guard, but the PR still has blocking workflow behavior: the latest PR files API shows the workflow still compares PR-time installer edits against the already-hosted script, and its schedule/push concurrency still shares one cancellation group. The author's follow-up comment says those issues were fixed, but the current PR diff and failing workflow run for the same head SHA do not match that claim.

## What This Changes

The PR adds a GitHub Actions workflow that fetches `https://openclaw.ai/install.sh`, compares its SHA-256 digest with `scripts/install.sh`, and runs on push, pull request, daily schedule, and manual dispatch.

## Best Possible Solution

Keep the drift-guard idea, but revise the workflow so byte-for-byte source/host comparison runs after merge, on schedule, and by manual dispatch rather than on installer-source pull requests. Include the event name or another discriminator in the concurrency key, and do not close #73837 until the hosted installer is redeployed or that bug is otherwise resolved.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature PR, but yes for the introduced CI failure mode: the current PR diff still triggers on `scripts/install.sh` pull requests, the live source and hosted hashes differ, and the PR head's new workflow run is failing.

## Solution Assessment

Is this the best way to solve the issue? No. A read-only hosted-installer drift check is a reasonable addition, but this implementation should not compare in-flight installer source changes against the already-deployed host and should not let main pushes cancel scheduled drift checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Limit PR-time source-host comparisons:** `.github/workflows/check-hosted-installer-sync.yml:11`
  - body: This workflow runs on pull requests that change `scripts/install.sh`, but a PR checkout contains the proposed installer while `openclaw.ai` still serves the last deployed installer. Because the job fails on any hash mismatch, future installer-source PRs will fail until after merge and deployment. Remove this path from `pull_request` validation or make PR validation compare against a PR-produced expected artifact.
  - confidence: 0.97
- **[P2] Separate scheduled checks from push cancellation:** `.github/workflows/check-hosted-installer-sync.yml:22-23`
  - body: `schedule` and `push` runs on main both use `refs/heads/main` for `github.ref`, so `cancel-in-progress` can let a main push cancel the daily drift check. Include `github.event_name` in the concurrency group, or otherwise avoid cancelling scheduled drift checks from push runs.
  - confidence: 0.92
