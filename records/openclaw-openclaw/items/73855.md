## Summary

Keep open. Current main still has no hosted-installer checksum workflow, and live verification still shows `scripts/install.sh` differs from `https://openclaw.ai/install.sh`. The PR is a plausible implementation candidate for #73837, but the proposed workflow needs revision before merge because its `pull_request` trigger checks in-flight installer edits against the already-deployed hosted script and its schedule/push concurrency can cancel the daily drift signal.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for hosted installer drift, but revise it before merge: remove `scripts/install.sh` from the `pull_request` trigger or make PR validation compare against a PR-produced expected artifact, separate schedule/push concurrency, and keep the workflow read-only and non-executing. #73837 should remain open until the hosted installer is actually redeployed and the drift guard proves future source/host mismatches are caught.

## Risks / Open Questions

- The proposed `pull_request` path filter includes `scripts/install.sh`, which can block legitimate installer PRs because the CDN is expected to update after merge/deployment, not during PR validation.
- The proposed concurrency group omits `github.event_name`, so scheduled and push runs on `refs/heads/main` can cancel each other and suppress the daily drift signal.
- This is CI infrastructure that fetches live network content on a schedule; even though the PR only hashes the download and uses read-only permissions, it should receive workflow-owner review before merge.
