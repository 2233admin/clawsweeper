## Summary

Keep open. Current main runs `message_sending` for normal Telegram `deliverReplies`, but Telegram preview-stream finalization can edit or materialize the final preview and return before `deliverReplies` runs. The existing dispatch test still asserts that `deliverReplies` is not called when a final answer stays in the streamed preview, so the reported bypass remains valid for the default preview-streaming path.

## What This Changes

Telegram `message_sending` hooks should run for outgoing Telegram replies even when preview streaming is enabled.

## Best Possible Solution

Add a Telegram preview-delivery hook adapter so final preview edit/materialization is treated as outbound delivery: run `message_sending` exactly once before committing the final preview text, honor content rewrites and cancellation, preserve existing fallback behavior, and keep broader all-channel outbound policy work in the related architecture issues.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is the current `streaming: partial` short text reply path: `deliverLaneText` finalizes by editing the preview, and the existing dispatch test asserts `deliverReplies` is not called for that case.

## Solution Assessment

No, current main is not the best solution because it only hooks normal delivery. The narrow maintainable fix is Telegram-local wiring around preview finalization, with tests for cancel, rewrite, fallback, and no double hook execution.
