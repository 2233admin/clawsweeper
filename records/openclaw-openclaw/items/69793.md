## Summary

Keep this issue open. Current main still builds no-text Telegram media body text from the Telegram message shape before saved media MIME can correct it, so a photo-shaped message with saved non-image media can still reach the model as `<media:image>`. The related broad PR #69893 is closed unmerged, while #59948 is adjacent context rather than a fix for this MIME/body mismatch.

## What This Changes

Telegram inbound body construction should use actual saved media content types for no-text placeholders instead of always presenting photo-shaped or non-audio media as `<media:image>`.

## Best Possible Solution

Add a Telegram-local placeholder classifier for saved media that maps `allMedia[].contentType` to non-misleading media tags when saved media exists, while keeping Telegram message-shape placeholders only for missing/failed downloads and covering the edge cases in focused body-builder tests.

## Reproduction Assessment

Yes. A deterministic unit-level reproduction is to call `resolveTelegramInboundBody()` with no user text, a `msg.photo`, and `allMedia: [{ path, contentType: "application/octet-stream" }]`; current code keeps `<media:image>` because `rawBody` is set from `resolveTelegramPrimaryMedia()` before looking at saved media MIME. I did not run tests because this review is read-only.

## Solution Assessment

Yes, deriving no-user-text placeholders from saved `allMedia[].contentType` is the narrowest maintainable fix for this issue. The hard-coded `msg.photo` MIME fallback discussed in the issue is not the best solution, and a broad cross-channel refactor is optional follow-up rather than required here.
