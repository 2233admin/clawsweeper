## Summary

Keep this issue open. Current main still derives the Telegram photo placeholder from the Telegram message shape before considering saved media content type, so a no-caption photo-like message with non-image saved media can still expose `<media:image>` in the model-visible body. The related implementation PR #69893 was closed unmerged, so no fix has landed.

## Best Possible Solution

Keep this issue open and land a focused Telegram plugin fix. Derive the no-text media placeholder from saved `allMedia[].contentType` when media exists, preserve message-shape placeholders for failed downloads with no saved media, preserve audio preflight transcript behavior, and add regression tests for non-image, image, video, mixed media, and audio cases. Use #69893 only as prior art because it was closed unmerged and bundled unrelated work.
