## Summary

Keep open. Current main still has the Telegram inbound placeholder mismatch: photo messages seed `rawBody` from `resolveTelegramPrimaryMedia()` as `<media:image>`, and `resolveTelegramInboundBody()` preserves that placeholder even when `allMedia` contains a non-image or unknown `contentType`. The related PR #69893 was closed unmerged in the provided GitHub context, so it does not supersede or implement the issue.

## Best Possible Solution

Keep the issue open and implement a Telegram inbound placeholder helper that derives `<media:image>`, `<media:video>`, `<media:audio>`, or `<media:document>` from resolved media `contentType`/media kind after download. Add focused tests for photo messages with `application/octet-stream` or document-like media, mixed media, and multiple images, and consider reusing the content-type-based placeholder patterns already present in Discord and Mattermost.
