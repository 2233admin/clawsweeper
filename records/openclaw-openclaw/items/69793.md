## Summary

Keep this issue open. Current main still derives Telegram no-text photo body text from the Telegram message shape before consulting saved media MIME, so a photo-shaped message with saved non-image media can still surface `<media:image>` to the model. The related broad PR #69893 is closed unmerged, and a read-only search found no open PR currently tracking this exact fix.

## What This Changes

Telegram inbound no-user-text media placeholders should be derived from the actual saved media `contentType` instead of always using `<media:image>` for photo-shaped or non-audio media.

## Best Possible Solution

Keep this open and land a focused Telegram plugin fix. When saved media exists and there is no user text, derive the placeholder from `allMedia[].contentType`; keep Telegram message-shape placeholders only for failed downloads with no saved media; preserve audio transcript behavior; avoid a hard-coded photo MIME fallback; and add regression tests for image, non-image/document, video, mixed-media, and audio cases.
