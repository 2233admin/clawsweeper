## Summary

Current main still does not implement Feishu missed-message recovery after gateway sleep or reconnect. This PR is a concrete Feishu-plugin implementation attempt, but the provided review discussion and current source show it is not mergeable as-is and the underlying user-visible gap remains.

## Best Possible Solution

Keep this PR open for maintainer judgment, but require a replacement implementation before merge: wire recovery into the Feishu monitor lifecycle, use `createFeishuClient` with the correct chat/message list APIs, reconstruct events from top-level message fields, reuse shared `chatHistories`, sequential routing, and durable dedupe, add config/permission docs, and cover startup/reconnect backfill with focused Feishu plugin tests.

## Risks / Open Questions

- Merging this PR as-is could introduce a polling loop that fails at runtime because the recovery module uses incorrect Lark client/API assumptions and is not integrated with the monitor lifecycle.
- The proposed mention filter can process unrelated messages containing `@`, which creates spam/privacy risk in group chats.
- No live Feishu tenant replay test was run in this read-only review; the eventual implementation should validate startup/reconnect backfill behavior against the real Feishu API.
