## Summary

Keep this PR open. Current main still lacks Feishu missed-message recovery on startup or WebSocket reconnect, and this PR is a focused attempt at that behavior, but the patch is not mergeable as-is because it is not wired into the monitor lifecycle and has compile/runtime and policy risks.

## Best Possible Solution

Keep the PR open for rework rather than close it. The best implementation is a Feishu-owned recovery path inside the existing monitor lifecycle: use `createFeishuClient` with `client.im.chat.list` and `client.im.message.list`, wire startup/reconnect recovery through the shared `chatHistories`, use persistent dedupe plus sequential processing, respect group policy and structured mention data, document required Feishu scopes/config, and add focused regression tests plus live Feishu API proof where feasible.

## Risks / Open Questions

- Merging the PR as-is could fail compile because `ResolvedFeishuAccount` has no `client` property and `dedup.ts` does not export `hasRecordedMessage`.
- The proposed polling path could create privacy-sensitive or unwanted replies by scanning group history every 30 seconds and treating any `@` character as a bot mention.
- The patch adds no dependency, workflow, lockfile, or package-publishing changes, so the security review found no supply-chain surface expansion beyond the new Feishu history-polling behavior.
- No tests or builds were run because this was a read-only review; the decision is based on source, docs, provided PR discussion, and the PR patch.
