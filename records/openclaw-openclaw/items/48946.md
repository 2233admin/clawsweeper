## Summary

Keep this PR open. Current main still lacks Feishu missed-message recovery on startup or WebSocket reconnect, and this PR is a focused attempt at the linked Feishu gap, but the patch is not mergeable as-is because it is not wired into the monitor lifecycle and conflicts with current Feishu client, event, dedupe, and history handling.

## Best Possible Solution

Keep this PR open for rework or supersede it with a safer Feishu-owned implementation. The recovery path should hook startup/reconnect in the Feishu monitor lifecycle, use createFeishuClient with current client.im.chat.list and client.im.message.list contracts, reconstruct FeishuMessageEvent from top-level message-list fields, dispatch through the existing dedupe/processing-claim/sequential queue/shared chatHistories path, respect structured mention and group-policy rules, document required scopes, and add focused regression tests plus live API proof where feasible.
