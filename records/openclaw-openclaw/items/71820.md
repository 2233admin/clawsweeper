## Summary

Keep open. Current main still lacks the BlueBubbles reply-context API fallback, and this PR is an active, mergeable implementation with the earlier SSRF, account-inheritance, changelog, and docs review feedback addressed; I found one narrow P3 cache-amortization gap for prefixed reply ids that should be fixed before final maintainer review.

## What This Changes

This PR adds an opt-in BlueBubbles reply-context API fallback with config/types/docs/tests/changelog updates, routes the fetch through the typed BlueBubbles client, and promotes attachment download failures to runtime error logging.

## Best Possible Solution

Land the BlueBubbles-owned fallback after preserving cache hits for part-index-prefixed reply ids and letting exact-head CI finish, keeping the behavior default-off, typed-client/SSRF-routed, documented, and separated from any future default-on product decision.

## Reproduction Assessment

Yes. Main can be reproduced by processing an inbound BlueBubbles reply with `replyToId` but no `replyToBody`/`replyToSender` after the reply cache is empty; current main only checks RAM. The new P3 finding is reproducible with a `p:0/<guid>` reply id because the helper fetches/cache-writes the bare guid while the caller's next cache lookup still uses the prefixed id.

## Solution Assessment

Yes, with one small repair. The PR's default-off, extension-owned, typed-client fallback is the narrow maintainable solution; the safer final shape is to canonicalize or alias prefixed reply ids consistently so the promised cache amortization actually works.

## Repair Work Prompt

On openclaw/openclaw PR #71820, fix the BlueBubbles reply-context fallback cache-amortization gap for part-index-prefixed reply ids. Preserve the current default-off config, typed `BlueBubblesClient`/SSRF path, docs, and changelog intent. The observable bug: `fetchBlueBubblesReplyContext` strips `p:0/<guid>` to `<guid>` before fetching and writes only `<guid>` into `rememberBlueBubblesReplyCache`, while `processMessageAfterDedupe` later calls `resolveReplyContextFromCache` with the original prefixed `replyToId`, so repeated prefixed replies miss RAM and re-fetch. Implement the smallest BlueBubbles-owned fix, such as sharing a canonicalization helper with the caller/cache lookup or storing an alias for the original id. Add a regression test proving two sequential `p:0/<guid>` fallback lookups reuse the populated cache or make only one API/client request. Do not change the default, broaden core APIs, or move behavior out of the BlueBubbles plugin.
