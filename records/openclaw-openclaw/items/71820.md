## Summary

Keep this PR open. Current main still resolves inbound BlueBubbles reply context only from the process-local cache and does not include the opt-in API fallback, config surface, or fallback tests. The overlapping #73241 is closed unmerged and its typed-client/security ideas appear to have been folded into this PR, so it is useful implementation work rather than a superseded PR.

## What This Changes

The PR adds an opt-in BlueBubbles reply-context HTTP API fallback for cache misses, with config/schema/types, tests, changelog, cache write-back/deduping, plus attachment download error visibility and log-sanitization changes.

## Best Possible Solution

Keep this PR open for maintainer/security review. The best path is to land one BlueBubbles-owned implementation, either this PR or a maintainer-directed split, that uses the typed `BlueBubblesClient`, preserves SSRF protections and config inheritance, validates reply IDs, writes successful fetches back to the reply cache, covers dedupe/failure/account-merge cases with tests, and keeps the changelog/docs aligned.

## Risks / Open Questions

- The new fallback is a webhook-triggered network fetch keyed by message data, so maintainer review should verify typed-client SSRF policy, redirect behavior, reply-id/path validation, timeout behavior, and best-effort failure swallowing on the final branch.
- Promoting attachment download failures to `runtime.error` is useful, but the final merged shape must keep all GUIDs, URLs, passwords, tokens, and Authorization headers sanitized before default-level logging.
- This PR bundles three separable wins: reply-context fallback, attachment error visibility, and log redaction/changelog updates. Maintainers may prefer splitting if review or release risk is easier that way.
