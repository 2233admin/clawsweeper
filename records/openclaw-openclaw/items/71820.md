## Summary

Keep this PR open. Current main still has no BlueBubbles `replyContextApiFallback` config, helper, docs, tests, or inbound API fetch on reply-context cache misses. The existing shipped work only normalizes webhook reply metadata, uses a process-local bounded reply cache, and lazy-refreshes outbound Private API status for reply sends/effects. The PR remains an active implementation candidate, but it still needs maintainer review because it adds a new webhook-triggered network fetch and currently carries unrelated formatting-only file changes outside BlueBubbles.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land this BlueBubbles-owned opt-in fallback after CI and scope cleanup, or explicitly choose a canonical competing PR if #73241 supersedes it. The implementation should stay behind channel/account config, preserve account inheritance, use the existing BlueBubbles SSRF policy resolver, keep cache-miss/failure/dedupe tests, and remove unrelated formatting churn before merge.
