## Summary

Keep open. Current main still lacks the BlueBubbles reply-context API fallback, and this PR remains useful, but it is not merge-ready: the new helper normalizes `p:N/<guid>` reply IDs before caching while the caller continues later cache and short-id lookups with the raw prefixed ID. The GitHub API also reports the head as dirty against the base branch, so it needs rebase plus a small correctness fix before maintainer review can finish.

## What This Changes

The PR adds an opt-in BlueBubbles inbound reply-context API fallback with config/schema/types, a typed-client fetch helper, cache write-back/deduping tests, setup-surface tests, attachment download error visibility, and a changelog entry.

## Best Possible Solution

Land one BlueBubbles-owned implementation after making the normalized reply ID consistent across fetch, cache write-back, later cache lookup, and short-id resolution; keep the typed-client SSRF path, opt-in config inheritance, failure swallowing, and focused BlueBubbles tests.

## Reproduction Assessment

Yes. A code-level reproduction is available with an inbound reply whose ID is `p:1/msg-0`: current tests show that shape is preserved, while the PR helper strips it to `msg-0` before caching and the caller still performs later lookups with the raw prefixed ID.

## Solution Assessment

No, not quite yet. Routing the fallback through `BlueBubblesClient` with opt-in config is the right direction, but the cache-key normalization bug should be fixed before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reuse the normalized reply ID for cache lookups:** `extensions/bluebubbles/src/monitor-reply-fetch.ts:157`
  - body: For `p:N/<guid>` reply IDs, this helper strips the prefix and stores the fetched context under the bare GUID, but `processMessageAfterDedupe` continues to call `resolveReplyContextFromCache` and `getShortIdForUuid` with the raw prefixed ID. That means the promised cache write-back does not amortize for the supported prefixed shape, and later replies keep refetching instead of getting the short ID/cache hit. Normalize the ID consistently at the caller/cache boundary or store aliases for both forms.
  - confidence: 0.86
