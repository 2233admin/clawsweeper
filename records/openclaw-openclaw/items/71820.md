## Summary

Keep open: current main still lacks this fallback, and the active PR still has one P2 correctness bug where prefixed BlueBubbles reply IDs are normalized for fetch/cache write-back but not for later cache or short-ID lookups.

## What This Changes

The PR adds an opt-in BlueBubbles reply-context HTTP fallback, config/schema typing, reply-cache write-back tests, attachment download error visibility, log redaction, and a changelog entry.

## Best Possible Solution

Land one BlueBubbles-owned implementation after making the normalized reply ID consistent across fetch, cache write-back, later cache lookup, and short-ID resolution while keeping the typed-client SSRF path, opt-in config inheritance, failure swallowing, and focused BlueBubbles tests.

## Reproduction Assessment

Yes. A code-level reproduction is an inbound reply whose ID is `p:1/msg-0`: the PR strips that to `msg-0` before caching, while the caller continues later lookups with `p:1/msg-0`.

## Solution Assessment

No, not yet. Routing the fallback through `BlueBubblesClient` with opt-in config is the right direction, but the cache-key normalization bug should be fixed before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reuse the normalized reply ID for cache lookups:** `extensions/bluebubbles/src/monitor-reply-fetch.ts:157`
  - body: For `p:N/<guid>` reply IDs, this helper strips the prefix and stores the fetched context under the bare GUID, but `processMessageAfterDedupe` continues to call `resolveReplyContextFromCache` and `getShortIdForUuid` with the raw prefixed ID. That means the promised cache write-back does not amortize for the supported prefixed shape, and later replies keep refetching instead of getting the short ID/cache hit. Normalize the ID consistently at the caller/cache boundary or store aliases for both forms.
  - confidence: 0.86
