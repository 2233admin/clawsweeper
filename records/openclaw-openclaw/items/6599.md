## Summary

Keep open. Current main still lacks a proactive fallback-chain test command: the slash `/models` path only routes provider/list/add flows, the CLI exposes status/list/set/fallback-management/auth probes but no fallback-chain test, and the existing live probe path is auth-profile oriented rather than a primary-skipping fallback walk. The direct implementation PR #43342 is closed unmerged, while #63145 is a broader per-model health-check request rather than a replacement for this targeted command.

## What This Changes

Add a `/models test-fallback` command that proactively verifies configured model fallbacks by skipping the primary model and optionally walking the full fallback chain.

## Best Possible Solution

Ship a shared fallback-chain probe that uses OpenClaw's real model/auth resolution, intentionally starts from configured fallback candidates instead of the primary, reports the responding provider/model and per-candidate failures, supports first-fallback and all-fallback modes, and documents how it differs from auth-profile probes and the broader #63145 per-model health-check idea.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is static inspection of the command registration, slash parser, docs, and fallback runtime on current main, which shows the requested command is absent.

## Solution Assessment

Yes, a first-class fallback-chain probe is the narrowest maintainable direction; it should not be hidden inside `models status --probe`, which currently means auth-profile probing. If maintainers later merge it with #63145, that should be an explicit product decision rather than an accidental broadening.
