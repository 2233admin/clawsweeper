## Summary

Current main still does not implement the requested per-model or alias-scoped context profiles. The available controls are global, per-agent, or model-capability based, but there is no config surface that independently controls workspace file injection, tool visibility, and skills visibility for a selected `/model` alias or provider model entry.

## Best Possible Solution

Keep this issue open. The best path is a backward-compatible selected-model prompt policy, likely on `agents.defaults.models` entries or named model profiles, with schema/types/docs/tests and runtime wiring through model selection, bootstrap-file resolution, skills prompt assembly, effective tool assembly, `/context` reporting, and `/model` status. Preserve `compat.supportsTools: false` as a model capability fallback while adding explicit user-controlled prompt visibility.

## Risks / Open Questions

- `compat.supportsTools: false` already means model capability/runtime compatibility; a future `tools: false` visibility profile should not silently conflate user prompt policy with provider capability detection.
- Per-agent `skills` and `tools` policies exist, so any per-model context profile needs a clear precedence order with per-agent policy, global tool policy, runtime tool allowlists, and capability-derived suppression.
- If multiple aliases for the same upstream API model need different prompt policies, the implementation may need to coordinate with #59168 or an equivalent internal-key design.
