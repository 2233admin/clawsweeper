## Summary

Keep open. Current main still does not implement per-model or alias-scoped context profiles for workspace injection, tool prompt visibility, or skills prompt visibility. Existing controls are useful but partial: `contextInjection` is global, model alias entries only carry alias/params/streaming, `compat.supportsTools` is capability metadata, and provider/model tool policy does not cover workspace files or skills. The issue discussion's token-waste concern still matches current behavior because no-tools models can have callable tools removed while the normal prompt and skills path are still built unless a separate runtime `toolsAllow` path is used.

## Best Possible Solution

Keep this issue open. The best path is a backward-compatible selected-model prompt policy on model aliases or named model profiles, with config schema/types/docs/tests and runtime wiring through model selection, bootstrap-file resolution, skills prompt assembly, effective tool assembly, `/context` reporting, and model status surfaces. Preserve `compat.supportsTools` as a capability fallback while adding explicit user-controlled prompt visibility.

## Risks / Open Questions

- Implementation needs a clear precedence order among explicit context profiles, `compat.supportsTools`, `tools.byProvider`, per-agent policies, runtime `toolsAllow`, and `disableTools` so capability detection is not confused with user prompt policy.
- Alias-scoped profiles may interact with #59168 because current model catalog keys are still tied to provider/model identifiers.
- A hook or plugin can customize some prompt behavior, but the requested user-facing config, schema validation, bootstrap-file selection, skills prompt suppression, and model-status reporting need core wiring.
