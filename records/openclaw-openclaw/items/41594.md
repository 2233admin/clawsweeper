## Summary

Current main at 5de284c2e30f915b769fac08dbeab45f57749702 still does not implement the requested per-model or alias-scoped context profiles. Existing controls are useful but partial: workspace bootstrap injection is global/default-scoped, model alias entries only allow alias/params/streaming, provider model compat only exposes capability metadata such as supportsTools, and tool/skill filtering is handled through separate global/per-agent/runtime paths rather than one selected-model prompt policy.

## Best Possible Solution

Keep this issue open. The best path is a backward-compatible selected-model prompt policy on model aliases or named model profiles, with config schema/types/docs/tests and runtime wiring through model selection, bootstrap-file resolution, skills prompt assembly, effective tool assembly, /context reporting, and /model/status surfaces. Preserve compat.supportsTools as a capability fallback while adding explicit user-controlled prompt visibility.

## Risks / Open Questions

- A future implementation needs a clear precedence order between contextProfile.tools, compat.supportsTools, tools.byProvider, per-agent tool policies, runtime toolsAllow, and disableTools so capability detection is not conflated with user prompt policy.
- If aliases for the same upstream API model need distinct prompt policies, #59168 or an equivalent internal-key design may be a prerequisite for clean configuration semantics.
- The current partial workarounds can reduce some tool surface, but they do not solve per-alias workspace file selection or skill prompt visibility, so closing now would lose a real remaining product request.
