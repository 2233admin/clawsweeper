## Summary

Keep open. Current `main` still exposes the requested compaction token settings as absolute numeric values only across the public type, validation schema, generated schema, docs, runtime resolver, and focused tests. This is a valid narrow enhancement with a clear implementation surface and no protected-label or linked-open-PR blocker in the provided issue context.

## What This Changes

Request support for percentage strings in compaction token settings so `reserveTokensFloor` and `keepRecentTokens` scale with the active model context window.

## Best Possible Solution

Add a small typed token-budget value contract that preserves existing numeric behavior, accepts documented percentage strings for the selected compaction settings, resolves them against the active model or effective context window at runtime, and updates generated schema, docs, changelog, and focused tests together.

## Reproduction Assessment

Not applicable as a feature request, but there is a high-confidence static verification path: current type definitions, Zod validation, generated schema, docs, and runtime resolver all accept numbers only and contain no percentage parser for these settings.

## Solution Assessment

Yes. Explicit percentage-string support is the narrowest maintainable direction because it keeps existing numeric configs backward-compatible; auto-scaling existing absolute numbers would silently change established behavior.
