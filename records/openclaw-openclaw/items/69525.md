## Summary

Keep open. Current main still resolves the message action accountId from `input.defaultAccountId` before cross-provider policy runs, the allowed cross-provider policy returns without clearing it, and plugin dispatch forwards that value to the target channel action. Related PR #69529 targets the bug, but it is still open, dirty, and bundles unrelated changes.

## What This Changes

The issue asks OpenClaw core message sending to stop passing a source session accountId to a different target provider during allowed cross-provider sends.

## Best Possible Solution

Land a narrow outbound core fix in `runMessageAction`: preserve explicit `accountId`, keep same-provider defaults and agent binding fallback, but suppress only the implicit source `defaultAccountId` when `allowAcrossProviders` is enabled and `toolContext.currentChannelProvider` differs from the resolved target channel. Add a focused regression in the message-action runner plugin-dispatch or context tests, then review or split PR #69529.

## Reproduction Assessment

Yes. A focused unit scenario can invoke `runMessageAction` with `defaultAccountId`, `crossContext.allowAcrossProviders: true`, a source `currentChannelProvider`, and a different target channel, then assert the target plugin `handleAction` receives `ctx.accountId` as undefined rather than the source default.

## Solution Assessment

Yes, the narrow core fix is the best direction. Plugin-side validation is a useful defensive workaround, but core should not send a source-provider account id into an unrelated target-provider channel action.
