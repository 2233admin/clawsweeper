## Summary

Current `main` still matches the reported root cause: `runMessageAction` resolves the target channel, then unconditionally falls back to the source session `input.defaultAccountId` when no explicit `accountId` is provided, and that value is passed into send/plugin dispatch. Cross-provider sends are documented and policy-allowed when `tools.message.crossContext.allowAcrossProviders` is true, but the current tests only cover the allow/deny policy and do not cover clearing the source account id for the target provider. The linked PR #69529 is open and unmerged in the provided context, so it is not shipped evidence.

## Best Possible Solution

Keep the issue open and land a narrow core fix in `src/infra/outbound/message-action-runner.ts`: preserve an explicitly provided `accountId`, but do not inherit `input.defaultAccountId` when the resolved target `channel` differs from `input.toolContext.currentChannelProvider` under `allowAcrossProviders`. Add a regression test in the outbound message-action runner/plugin-dispatch area that sends from one provider context to another and asserts the target plugin receives `accountId: undefined`, plus a companion assertion that an explicit `accountId` is still honored.
