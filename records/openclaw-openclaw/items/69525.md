## Summary

Current main still appears to have the reported leak: `runMessageAction` applies `input.defaultAccountId` before cross-provider policy, the allowed cross-provider path returns without clearing it, and plugin dispatch forwards that `accountId`. Open PR #69529 targets this but is still open and bundles unrelated changes, so this issue should stay open.

## Best Possible Solution

Keep this issue open and land a narrow fix in the outbound message-action path: preserve explicit `accountId`, suppress only `input.defaultAccountId` when `allowAcrossProviders` is enabled and `toolContext.currentChannelProvider` differs from the target channel, and add a regression test in the current split message-action runner tests. Review or split #69529 rather than closing this before a merged fix.
