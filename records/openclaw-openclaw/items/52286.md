## Summary

Keep open: current main still has the reported Telegram forum-topic routing gap for message-tool file/media sends when the model supplies the current sender's positive Telegram user ID as the target. Same-chat topic threading is implemented, but there is no current-main guard that remaps that sender-user target back to the originating supergroup chat plus topic thread.

## What This Changes

The issue requests that Telegram message-tool file/media sends from forum topics stay in the originating topic even when the model supplies the sender user ID as the target.

## Best Possible Solution

Keep this as the canonical Telegram forum-topic file/media routing bug and implement a narrow current-conversation attachment route or Telegram-specific guard that maps accidental current-sender user-ID targets back to the originating group chat plus MessageThreadId while preserving an explicit way to intentionally DM a user.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Telegram forum-topic turn with currentChannelId set to the negative supergroup chat and currentThreadTs set, followed by message(action=send, media=..., target=<positive sender user id>); the current parser and auto-threading path leave that as a direct chat with no topic thread.

## Solution Assessment

No. The existing same-chat auto-threading is the right base behavior for explicit group/topic targets, but it does not solve the reported sender-user-ID case; the safer solution is a deterministic same-conversation attachment path or an explicit Telegram intent guard rather than broad target rewriting.
