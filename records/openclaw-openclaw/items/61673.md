## Summary

Keep open. Current main still only supports static Docker sandbox bind arrays: config validation accepts `binds?: string[]`, merge logic combines global and per-agent arrays, and container creation appends those strings during `docker create`. The requested pre-container host resolver is a security-sensitive core/plugin API question, not something that is already implemented or safely moved to ClawHub without a missing core seam.

## What This Changes

The issue requests a host-side pre-container-creation hook that resolves Docker sandbox bind mounts dynamically from session/channel/user context.

## Best Possible Solution

Design a first-class pre-create sandbox policy seam in core if maintainers accept the capability. It should receive trusted run/session/channel context before config hashing and backend creation, merge or replace static binds explicitly, validate all returned binds through the existing bind security policy, fail closed on timeout or invalid output, and update config/schema/docs/tests; a typed plugin hook may be safer than a raw host executable if the project wants this as an extension API.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives high-confidence proof of the current limitation: strict config/schema/docs only expose static `docker.binds`, and the Docker create path consumes only the resolved `cfg.binds` array.

## Solution Assessment

Unclear. The proposed `resolveBinds` executable is a plausible operator workflow, but it may not be the safest design because it introduces gateway-privileged host command execution; a typed plugin/runtime policy hook with bounded context and existing validation may be the narrower maintainable API.

## Security Review

Status: needs_attention

Summary: The request changes sandbox isolation and would add host-side pre-create execution, so it needs explicit security design before implementation.

Concerns:

- **[medium] Constrain host-side bind resolution:**
  - body: A resolver that runs before container creation would execute with gateway host privileges and decide which host paths enter the sandbox. The design must constrain the executable or plugin source, trust and minimize session context, enforce timeouts, validate returned binds with the existing blocklists, and account for container reuse to avoid broadening file access across users.

## Risks / Open Questions

- The accepted API shape is undecided: external executable, typed plugin hook, or another host-side policy seam have different security and deployment tradeoffs.
- Dynamic bind resolution affects gateway-privileged host execution and container filesystem isolation, so implementation should not proceed without explicit maintainer/security design review.
- Container reuse and config hashing would need a precise policy for session-specific bind outputs; otherwise a session could reuse a container created with another session's mounts.
