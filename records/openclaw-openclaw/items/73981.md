## Summary

Keep open. Current main still has the reported streaming behavior: `EmbeddedBlockChunker` is fence-aware only, strips paragraph separators in the flush-on-paragraph path, and has no table-span parser or table-streaming integration coverage. The PR is a relevant implementation candidate, but it should not merge as-is because the proposed patch can exceed `maxChars` when preserving paragraph separators and leaves valid GFM pipe tables undetected.

## What This Changes

This PR preserves paragraph/newline separators in `EmbeddedBlockChunker`, adds table-span detection to avoid splitting detected Markdown pipe tables during block streaming, and updates streaming docs, changelog, unit tests, and integration tests.

## Best Possible Solution

Land one canonical streaming chunker fix that preserves consumed paragraph/newline separators, keeps detected Markdown tables intact when feasible, preserves channel `maxChars` guarantees, covers GFM table edge cases, and includes focused chunker plus `subscribeEmbeddedPiSession` regression tests.

## Reproduction Assessment

Yes. Current main can be reproduced by feeding `EmbeddedBlockChunker` multi-paragraph text or text/table/text with `flushOnParagraph`; the source and existing tests show paragraph separators are consumed and table spans are not considered.

## Solution Assessment

No, not as-is. The PR is pointed at the right shared chunker surface, but the patch needs the `maxChars` accounting and table-parser edge cases fixed before it is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Include separator length in maxChars check:** `src/agents/pi-embedded-block-chunker.ts:218-221`
  - body: When a paragraph break starts within `paragraphLimit`, the PR emits through `paragraphBreak.index + paragraphBreak.length` but only checks `paragraphBreak.index - start <= paragraphLimit`. A buffer like `abcdefghij\n\nRest` with `maxChars: 10` would emit a 12-character chunk even though `maxChars` is the channel cap.
  - confidence: 0.9
- **[P2] Accept one-hyphen table delimiters:** `src/markdown/table-spans.ts:50-53`
  - body: The delimiter regex requires three or more hyphens per cell. GFM delimiter cells consist of hyphens with optional colons, so valid tables using `| - | - |` are not protected and can still be split mid-table.
  - confidence: 0.86
- **[P2] Handle single-column pipe tables:** `src/markdown/table-spans.ts:42-48`
  - body: After trimming the leading and trailing pipes, a row like `| Alpha |` has one cell, so `cells.length >= 2` rejects valid single-column pipe tables. Those tables remain invisible to the span parser and can still split across chunks.
  - confidence: 0.88
