## Summary

Current main still appears to preserve the reported behavior: successful compaction can be tracked before a retry, but a later mixed fallback exhaustion with verbose details off still falls through to the generic external `/new` failure copy instead of a compaction/timeout-aware recovery message.

## What This Changes

Improve embedded-agent failure reporting so a successful auto-compaction followed by retry timeout and exhausted or unavailable fallbacks does not collapse to the generic `/new` recovery text.

## Best Possible Solution

Keep the issue open and add a targeted core fix that preserves structured successful-compaction state through fallback-summary failures, then formats a specific external-chat recovery when the retried model times out and fallback candidates are unavailable or exhausted.
