## Summary

Keep open. The PR targets a valid Moonshot provider bug that current main still does not handle, and the functional diff is narrow with exact-head CI green; before merge it still needs normal maintainer review plus the required user-facing changelog entry.

## What This Changes

The PR adds a Moonshot payload helper that injects empty `reasoning_content` on assistant messages with `tool_calls` when Moonshot thinking is enabled, plus regression tests for enabled, disabled, and pre-existing reasoning-content cases.

## Best Possible Solution

Land a focused Moonshot-only outgoing payload repair that preserves existing `reasoning_content`, adds the empty placeholder only when final Moonshot thinking remains enabled for assistant tool-call replay, keeps generic OpenAI-compatible replay unchanged, and includes the required Unreleased changelog fix entry with contributor credit.

## Reproduction Assessment

Yes. Current main source inspection plus #71491/#70563 gives a high-confidence path: a Moonshot thinking-enabled replay can contain assistant `tool_calls` without top-level `reasoning_content`, while Moonshot's thinking docs require preserving that field through multi-step tool calls. citeturn1view0

## Solution Assessment

Yes, with the changelog fix and maintainer validation. The provider wrapper is the narrowest maintainable place for this compatibility repair and mirrors the existing DeepSeek V4 payload-patch pattern; preserving real reasoning through compaction would be better for quality, but the placeholder is a pragmatic rejection guard once history is already missing it.
