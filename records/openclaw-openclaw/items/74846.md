## Summary

Keep open for maintainer review. Current main still lacks a Moonshot-specific replay backfill for assistant tool-call messages missing `reasoning_content`, while the PR is focused on the affected provider payload boundary, matches the current Kimi tool-call contract, and has no blocking review finding from this read-only pass.

## What This Changes

The PR adds a Moonshot payload helper that inserts empty `reasoning_content` on assistant messages with `tool_calls` when Moonshot thinking is enabled, plus focused regression tests for enabled, disabled, and pre-existing reasoning content cases.

## Best Possible Solution

Land a focused Moonshot-only payload repair at the final outgoing request boundary: preserve existing `reasoning_content`, add an empty-string placeholder only for assistant messages with tool calls when final Moonshot thinking remains enabled, keep generic OpenAI-compatible replay semantics unchanged, and add changelog/docs only if maintainers consider the provider bug user-facing enough.

Yes. Current main source inspection plus the concrete #71491 report gives a high-confidence reproduction path: a Moonshot thinking-enabled replay can contain assistant `tool_calls` without top-level `reasoning_content` before the provider request, which the upstream Kimi docs say is rejected.
