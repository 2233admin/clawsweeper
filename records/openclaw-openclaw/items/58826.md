## Summary

Keep open. Current main has token/cost observability, provider quota display, model failover after provider-side limit errors, and optional tool-loop detection, but it does not provide the requested built-in token/request/cost budget enforcement at request, session, daily, provider, or per-agent scope. Related #42475 covers an overlapping cost-budget subset and #9912 covers iteration caps, but neither fully supersedes this broader token/request/provider-quota request. Related PR #45879 is closed unmerged and its proposed TokenMetabolism surface is not present on main.

## Best Possible Solution

Keep this issue open unless maintainers explicitly choose #42475 as the canonical budget-control umbrella and migrate the missing token/request/provider-quota requirements there. The likely implementation belongs in the gateway/model-dispatch path with a typed config contract, per-agent/global overrides, usage accounting integration, and docs for warn/block/fallback behavior; loop detection can remain an adjacent guardrail rather than the budget mechanism.

## Risks / Open Questions

- Closing this as a duplicate of #42475 would lose request-count, token-count, provider-quota, and fallback/warn action requirements that are broader than per-agent dollar budgets.
- Existing loop detection helps with one failure mode but is disabled by default and does not provide budget accounting or daily/session caps.
- A real implementation needs product design around the enforcement point, accounting source of truth, provider quota freshness, warning versus block versus fallback actions, and config schema compatibility.
