## Summary

This should stay open. Current main still configures the Google Gemini CLI backend with `--skip-trust` and JSON prompt flags, without mapping OpenClaw's YOLO exec policy to Gemini CLI's documented `--approval-mode yolo` mode; the reporter's follow-up is therefore actionable for the tools/memory failure. The separate Matrix CPU saturation symptom still needs logs or a reproducible fixture before it can be called fixed or fully diagnosed.

## What This Changes

The issue reports constant CPU saturation, failed file-tool execution, and empty long-term memory files in a Matrix setup using the Google Gemini CLI provider on Node v25.9.

## Best Possible Solution

Add a provider-local Google Gemini CLI permission-mode normalizer that follows the same OpenClaw exec-policy boundary as the Claude CLI backend, preserves explicit user backend args, and then ask for Matrix CPU logs or a retest if resource exhaustion persists after tools and memory work again.
