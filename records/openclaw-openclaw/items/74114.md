## Summary

Keep open. The PR addresses a real linked bug in #72021, and current main has not implemented the requested recall-only gate, but the proposed patch conflicts with current tested and documented daily/grounded promotion behavior. It needs maintainer policy judgment and matching tests/docs before merge, not conservative cleanup closure.

## What This Changes

This PR changes memory-core short-term promotion to compare `minRecallCount` against `recallCount` alone in both ranking and apply filtering, while leaving aggregate `signalCount` for scoring.

## Best Possible Solution

Define explicit memory-core promotion semantics that separate organic recall evidence from automated daily/session and grounded evidence where needed, then update the gate logic, regression tests, and memory docs together so #72021 is fixed without silently disabling supported dreaming/backfill paths.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: applying the PR diff makes current daily-only (`recallCount: 0`, `dailyCount: 3`) and grounded-only (`groundedCount: 3`) promotion tests fail the default `minRecallCount` gate before ranking or applying.

## Solution Assessment

No. The recall-only comparison is a narrow fix for one reported symptom, but as-is it is not the best maintainable solution because it breaks current daily/grounded promotion semantics without a policy-compatible replacement or matching tests/docs changes.
