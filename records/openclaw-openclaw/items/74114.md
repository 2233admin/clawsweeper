## Summary

Keep open. Current main still uses aggregate signal counts for memory-core promotion gates, and this PR is a narrow proposed fix for linked issue #72021, but the patch conflicts with existing tested and documented daily/grounded promotion paths and needs maintainer policy judgment before merge.

## What This Changes

The PR changes memory-core short-term promotion filtering so `minRecallCount` is compared against `recallCount` only in both ranking and apply paths while leaving aggregate `signalCount` for scoring.

## Best Possible Solution

Define the promotion threshold policy explicitly: separate true user recall evidence from automated daily/session evidence where needed for #72021, while preserving a supported threshold path for grounded backfill and any intended daily/session promotion flow, or deliberately retiring those paths with matching tests and docs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: applying this PR makes the current daily-only and grounded-only promotion tests fail because their candidates have `recallCount: 0` and would no longer clear the default `minRecallCount=3` gate.

## Solution Assessment

No, not as-is. The recall-only gate is a plausible narrow piece of the #72021 fix, but the maintainable solution must also reconcile daily/session and grounded promotion semantics instead of silently disabling currently tested/documented paths.
