## Summary

Keep open for maintainer review. Current main and the latest release still omit `standardFontDataUrl`, so the underlying PDF fallback bug is real, but this PR should not merge as-is because its `path.sep` trailing separator breaks PDF.js validation on Windows and the changelog entry contains a stray marker.

## What This Changes

The PR changes the bundled `document-extract` PDF fallback to resolve `pdfjs-dist/standard_fonts` from the installed package root, forward it as `standardFontDataUrl`, add a local legacy type shim and regression assertion, and add a changelog entry.

## Best Possible Solution

Revise this PR or land #74355/equivalent so the plugin resolves `pdfjs-dist` from the installed package root, passes a plain filesystem path ending in literal `/`, keeps the type shim, adds real standard-font PDF coverage, removes the stray changelog marker, and then lets the linked bug close through the merged PR.

## Reproduction Assessment

Yes. The reported bug has a high-confidence path: run the fallback extractor against a PDF using a standard font while current main calls PDF.js without `standardFontDataUrl`; the PR-specific regression is reproducible by evaluating the branch on Windows where `path.sep` is `\` and PDF.js requires a trailing `/`.

## Solution Assessment

No as written. Resolving from `pdfjs-dist/package.json` is the right direction, but the branch should append a literal `/` for PDF.js, remove the changelog marker, and ideally use the real-PDF fixture coverage from the overlapping candidate before merge.

## Review Findings

Build the directory with `path.join` if needed, but append a literal `/` for the value passed to PDF.js.
