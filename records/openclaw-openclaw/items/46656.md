## Summary

Keep open. Current main still lacks the requested WebChat/Control UI inline-button path: the Gateway chat schema has no interactive payload, no `chat.callback` method is registered or scoped, WebChat config only exposes history sizing, final/history projection carries display content rather than controls, and the Control UI message model/render path has no button item.

## What This Changes

Add full WebChat/Control UI inline-button support, including an enablement config path, WebSocket button payloads, callback RPC handling, and Control UI rendering.

## Best Possible Solution

Implement an additive WebChat interactive-control path using the shared `presentation` contract: preserve sanitized final-only button/select blocks through chat events and history, render them in Control UI with clicked or disabled state, and add a scoped `chat.callback` RPC that initially routes selected values back into the same session. Treat webhook routing as a separate hardened follow-up unless maintainers explicitly include it in the first design.

## Security Review

Status: needs_attention

Summary: No PR diff was reviewed, but the requested WebChat callback/webhook design is security-sensitive and needs maintainer trust-boundary review before implementation.

Concerns:

- **[medium] Define callback routing trust boundaries:**
  - body: A WebChat callback RPC would convert authenticated browser clicks into agent input. The contract should bind callbacks to session and message ownership, validate callback values, define replay/disabled-state behavior, and preserve operator scopes before shipping.
  - confidence: 0.88
- **[medium] Harden webhook egress separately:**
  - body: The proposed `callbackWebhookUrl` would let UI clicks trigger Gateway-originated HTTP requests. That path needs explicit SSRF/egress policy, authentication, retry, and audit semantics, so it should not be bundled into a first UI button implementation without design approval.
  - confidence: 0.82
