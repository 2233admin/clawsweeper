## Summary

Keep this issue open. Current main still buffers QQBot `kind: "tool"` text in `toolTexts`, returns before normal delivery, and only flushes that text through the fallback path when no block/final reply arrives. QQBot `streaming.mode` still controls block/official C2C streaming, not immediate tool-progress delivery, and the public streaming docs still omit QQBot from supported tool-progress preview surfaces.

## What This Changes

Request that QQBot/gateway intermediate tool or progress text be delivered immediately when partial streaming is enabled instead of being buffered and discarded before the final block reply.

## Best Possible Solution

Implement the behavior in the QQBot plugin boundary: deliver useful `kind: "tool"` progress text immediately or through QQBot's existing C2C streaming/preview mechanism when partial streaming is active, suppress or dedupe it correctly once the final block arrives, preserve media fallback behavior, and document any public config surface that is added.
