## Summary

Keep this PR open. Current main at f256eeba431b5f59e6085b7c11c9c0a309e408b7 still does not implement the proposed `before_identity_resolve` hook or an equivalent dynamic direct-message peer identity seam. The PR remains a plausible implementation candidate, but the review discussion also identifies real follow-up work around blank hook results, secondary Slack/WhatsApp routing paths, multi-plugin execution, and session-isolation semantics that should be resolved through maintainer review rather than auto-closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused additive core/plugin contract: add a typed identity-resolution hook, expose a narrow SDK helper, thread an optional resolved peer id through route and session-key construction, integrate all direct-message and secondary routing paths for Slack, MS Teams, and WhatsApp, and add contract/channel tests for normalization, blank results, identityLinks fallback, multi-plugin priority/short-circuit behavior, and session-isolation safety.

## Risks / Open Questions

- Merging the PR unchanged could split direct-message history when secondary Slack system-event paths or WhatsApp broadcast fan-out rebuild session keys from raw sender ids instead of the plugin-resolved canonical id.
- A blank or whitespace `canonicalPeerId` from a higher-priority hook could suppress lower-priority valid results unless empty values are normalized before first-result-wins precedence.
- Running every registered identity hook after a winning result can add avoidable latency and side effects if plugins call external identity services.
- The feature changes session-isolation semantics: a buggy or compromised plugin could collapse distinct users into the same session, so plugin contract wording and regression tests need to be explicit.
