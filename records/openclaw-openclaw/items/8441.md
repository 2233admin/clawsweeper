## Summary

Keep this issue open. Current main still does not implement first-class `skills.entries.<skill>.model` or `skills.entries.<skill>.thinking`: the type/schema/docs only expose `enabled`, `apiKey`, `env`, and custom `config`, strict validation still rejects unknown per-skill top-level keys, and skill command invocation has no carrier for model/thinking defaults. The closed related PR #8455 was not merged, and #40536 was closed while pointing back to this broader request.

## Best Possible Solution

Implement the feature in core skills config and invocation plumbing: add validated optional `model` and `thinking` fields to `SkillConfig`/`SkillEntrySchema`, carry resolved per-skill defaults through `SkillCommandSpec` or an equivalent invocation context, apply them as one-turn defaults with documented precedence against explicit directives/session overrides/agent defaults, add schema and invocation/model-selection tests, and update the public skills config docs plus changelog.

## Clownfish Work Prompt

Implement per-skill `model` and `thinking` defaults for `skills.entries.<skill>` on current main. The observable request is this issue: users should be able to configure `skills.entries.<skill>.model` and `skills.entries.<skill>.thinking` so direct skill invocation uses those as one-turn defaults, similar in spirit to cron job model/thinking overrides.

Expected boundary:
- Add optional first-class `model` and `thinking` fields to `SkillConfig` and `SkillEntrySchema`; keep the existing `config` bag for arbitrary skill-owned settings.
- Carry resolved per-skill defaults through `SkillCommandSpec` or an equivalent skill invocation context from `src/agents/skills/command-specs.ts` into `src/auto-reply/reply/get-reply-inline-actions.ts`.
- Apply these as per-turn defaults for model selection/thinking, not persistent session mutations unless existing directive handling intentionally persists them. Define and test precedence: explicit same-turn user directives should win over skill defaults; existing agent/session defaults should remain fallback behavior.
- Reuse existing model resolution/allowlist/fallback/auth-profile and thinking normalization/policy helpers. Do not broaden directive parsing syntax, and do not bypass model allowlists by injecting provider/model directly.
- Tool-dispatched skill commands that bypass the model should not need model/thinking changes unless a current code path actually runs a model turn.
- Update `docs/tools/skills-config.md` and `docs/tools/skills.md`; add a user-facing changelog entry if this repo’s active changelog section expects one, crediting `Thanks @tsukhani` and optionally commenters whose feedback is reflected.
