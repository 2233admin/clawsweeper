## Summary

Keep this PR open, but do not merge it as-is. Current main still has the stale cron cache behavior the PR targets, while the branch only invalidates on `jobs.json` and can miss current split-store runtime edits in `jobs-state.json`; it also records the cache mtime in an order that can bless a stale read.

## What This Changes

The branch adds mtime-based cache invalidation to `CronService.ensureLoaded()` and new Vitest coverage for manual cron recovery and deleted-job reloads.

## Best Possible Solution

Refresh the PR so cron cache reuse invalidates on both backing files, records observed mtimes after successful reads, preserves non-file handling, and adds focused coverage for external `jobs-state.json` recovery before `cron.run` and `cron.list`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Watch jobs-state.json before returning cached cron data:** `src/cron/service/store.ts:34-36`
  - body: Current main stores runtime fields such as `runningAtMs` in the derived `jobs-state.json` sidecar. This fast path only stats `state.deps.storePath`, so clearing a stuck runtime marker in the sidecar can leave `jobs.json` unchanged and keep returning the stale in-memory job.
  - confidence: 0.92
- **[P2] Record cron mtimes after reading the store:** `src/cron/service/store.ts:42-48`
  - body: The new cache hit depends on `state.storeFileMtimeMs`, but the load path records an mtime captured before `loadCronStore()` reads the file. If an external write's mtime becomes visible before the new contents are read, the service can cache old contents with the new mtime and skip future reloads.
  - confidence: 0.78
