## Summary

Keep open. Current main still fails closed when a trusted-proxy request has no configured identity header, the exported config/schema/docs do not include a default or fallback user field, and the discussion already identifies this as distinct from the closed local-direct password fallback work. The requested behavior changes an auth trust boundary, so it needs maintainer/security approval rather than cleanup closure or an automated repair lane.

## What This Changes

The issue requests a `trustedProxy.defaultUser` or `fallbackUser` option so trusted-proxy auth can accept trusted-proxy requests when the configured identity header is absent, especially for Cloudflare WebSocket upgrade paths.

## Best Possible Solution

Keep this open for maintainer/security review. If accepted, add an explicit trusted-proxy fallback identity option across exported config types, zod/generated schemas, docs, audit guidance, and auth/control-ui tests, with strict rules for when it can apply and whether `allowUsers` must include the fallback identity. If rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to an auth/proxy path that reliably forwards identity headers on WebSocket upgrades.

## Reproduction Assessment

Yes for the OpenClaw behavior: a trusted-proxy auth call with a trusted source but no `userHeader` reproduces the current `trusted_proxy_user_missing` rejection, and the unit test asserts that. The Cloudflare-specific edge limitation was not live-reproduced here, but the repository-side missing fallback path is clear.

## Solution Assessment

Unclear. A `defaultUser` could solve the reporter's deployment, but it is not automatically the best fix because it authorizes requests with no per-request identity; the maintainable path needs explicit security semantics or a documented unsupported Cloudflare configuration.
