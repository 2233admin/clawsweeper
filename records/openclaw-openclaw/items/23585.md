## Summary

Keep open. Current main still fails closed when the trusted-proxy identity header is absent, the config/schema contract has no `defaultUser` or `fallbackUser`, and the discussion already identifies this as distinct from the closed local-direct password fallback work. The requested fallback changes an authentication trust boundary and needs maintainer/security approval rather than conservative cleanup closure.

## What This Changes

Add a trusted-proxy default or fallback user option that authenticates requests from configured trusted proxies when the configured identity header is absent.

## Best Possible Solution

Keep the issue open until maintainers decide the trusted-proxy contract. If accepted, add an explicit fallback identity option across exported config types, zod and generated schemas, docs, security audit guidance, and gateway/control-ui auth tests with precise fail-closed rules. If rejected, document Cloudflare Zero Trust IP-whitelist WebSocket paths without forwarded identity as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers.
