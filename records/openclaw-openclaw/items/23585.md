## Summary

Keep open. Current main still fails closed when the trusted-proxy identity header is absent, and the config/types/schema/docs/tests do not expose a default or fallback identity option. The linked discussion also separates this Cloudflare/WebSocket missing-header case from the closed local-direct password fallback and Control UI pairing work, so this remains a security-sensitive product decision rather than conservative cleanup.

## What This Changes

Request support for a trusted-proxy default or fallback user when a request comes from a trusted proxy but the configured identity header is missing, especially for Cloudflare Tunnel WebSocket upgrades.

## Best Possible Solution

Keep the issue open for maintainer/security review. If accepted, add an explicit trusted-proxy fallback identity contract across exported config types, zod and generated schemas, docs, security audit guidance, and gateway/control-ui auth tests, with precise fail-closed rules. If rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers on WebSocket upgrades.
