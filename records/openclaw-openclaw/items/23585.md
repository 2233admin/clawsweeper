## Summary

Keep open. Current main still does not support a trusted-proxy default/fallback user when the identity header is absent: auth returns `trusted_proxy_user_missing`, config/schema/docs expose only identity-header based fields, and tests assert the fail-closed behavior. The discussion also distinguishes this from the now-closed local-direct fallback work, so the remaining decision is a security-sensitive trusted-proxy product choice rather than cleanup closure.

## Best Possible Solution

Keep this issue open for maintainer/security review. If accepted, implement an explicit trusted-proxy fallback identity option across the exported config type, zod and generated schemas, config help/docs, security audit guidance, and auth tests. The semantics should require trusted proxy source checks, required headers, loopback policy, and browser-origin policy to pass before fallback applies, and should define whether `allowUsers` must include the fallback identity. If rejected, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to password auth, Tailscale Serve, or a proxy path that reliably forwards identity headers on WebSocket upgrades.
