## Summary

Keep open. Current main still performs the source-local realpath containment check before skill-name merging, and escaped symlinked skill entries still emit a warn-level skills log. The reported personal `~/.agents/skills` symlink-to-bundled-skill case is therefore still current, even though bundled copies may load from their normal source. The provided context also shows an open implementation candidate with closing syntax, so this issue should remain open until that candidate or an equivalent fix is reviewed and merged or explicitly rejected.

## Best Possible Solution

Keep this issue open. Review the linked implementation candidate or land an equivalent focused skill-loader change that preserves realpath containment while suppressing or downgrading the non-actionable warning when a symlinked personal skill resolves outside `~/.agents/skills` but the same skill is already available from a trusted bundled or managed source. Add a regression test for a personal `.agents/skills` symlink to a bundled skill.
