## Summary

Keep open. The PR is useful source work, but it is not mergeable as the complete fix because it edits the obsolete `pi-extensions` context-pruning path while current `main` runs the `pi-hooks` pruner, and that active pruner still has CJK underestimation gaps for soft-trim eligibility and serialized tool-call arguments.

## What This Changes

The PR branch adds a CJK/non-Latin weighted string-length estimator to the old `pi-extensions` context pruner, wires it into message and tool-result estimates including tool-call arguments and soft-trim eligibility, and adds estimator unit tests.

## Best Possible Solution

Port the CJK-aware estimation behavior into `src/agents/pi-hooks/context-pruning`, reuse the shared `src/utils/cjk-chars.ts` helper for all text-bearing estimates including serialized tool-call arguments, keep soft-trim measurement and extraction in coherent units, and add focused regressions for CJK tool results, CJK tool-call arguments, CJK Extension B, and ASCII behavior.

## Reproduction Assessment

Yes. A source-level reproduction is a CJK-heavy prunable `toolResult` below raw `softTrim.maxChars` but above the weighted budget, or an assistant `toolCall` with CJK arguments; current `main` still measures those two active-pruner paths with raw `.length`.

## Solution Assessment

No. The PR direction is useful, but this patch is not the best solution because it edits an obsolete path and its weighted soft-trim eligibility change is not paired with weighted head/tail extraction semantics.
