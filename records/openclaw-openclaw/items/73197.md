## Summary

Keep open. The PR contains useful stability work, but it is not merge-ready because the Gmail watcher backoff reset runs on `spawn`, which fires before an immediately crashing child exits, so the intended crash-loop mitigation stays effectively fixed at the original short delay. The branch also needs rebase/current-main reconciliation for media cleanup and should not drop unrelated i18n registry coverage.

## What This Changes

The PR wraps subagent output polling errors, adds warning logs for best-effort config fallbacks, changes Gmail watcher restarts to use exponential backoff, clears a media cleanup interval on server close/error, and deletes the i18n registry test.

## Best Possible Solution

Repair the branch before merge: reset Gmail restart backoff only after the child has survived a real healthy window or readiness signal, keep or move the i18n registry coverage, reconcile the media cleanup hunk with current gateway maintenance code, and add focused tests for the crash-loop backoff plus the guarded subagent/config fallback paths.

## Reproduction Assessment

Yes for the blocking PR bug: a child process that immediately exits emits `spawn` before `exit`, and the proposed logic resets `restartBackoffMs` on that `spawn` event before the exit handler schedules the next restart. That gives a high-confidence reproduction path for the ineffective backoff without needing live Gmail or `gog`.

## Solution Assessment

No. The subagent/config/media intentions are mostly narrow, but the Gmail watcher implementation does not solve the crash loop as written; the safer fix is to reset the backoff only after `gog` stays alive for a meaningful interval or reports readiness.

- **[P1] Reset Gmail backoff only after a healthy run:** `src/hooks/gmail-watcher.ts:96-98`
  - body: The new `spawn` listener runs as soon as the OS creates the child, before `gog` can fail. Move the reset behind a healthy-uptime timer or readiness signal so persistent crashes actually back off.
