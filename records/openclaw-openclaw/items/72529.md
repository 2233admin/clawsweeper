## Summary

Current main already implements the requested cron routing through the generic persistent-session target. `sessionTarget: "session:<id>"` and `openclaw cron --session session:<id>` accept full agent session keys, including subagent keys, and the runner reuses that session instead of forcing a fresh isolated session.

## What This Changes

The issue requests cron agent-turn jobs to route directly into persistent subagent sessions instead of only main or isolated sessions.

## Best Possible Solution

Keep the generic persistent-session cron target and direct users to `openclaw cron --session "session:<full-subagent-session-key>"`, plus `--agent <id>` for non-default agent profiles, instead of adding a subagent-only alias or using `--session-key` with isolated jobs.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is source and test inspection: create an agent-turn cron job with `sessionTarget: "session:agent:main:subagent:<uuid>"` and current main resolves that to the persistent session key while only literal `isolated` forces a fresh session.

## Solution Assessment

Yes. The existing generic `session:<id>` target is the narrowest maintainable solution because subagent sessions are already normal agent session keys, and adding a `subagent` spelling would duplicate core routing semantics.
