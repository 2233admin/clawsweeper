## Summary

Current main and shipped releases already solve the request through the generic persistent cron session target. A separate subagent-specific target is unnecessary because `session:<full-session-key>` accepts subagent keys such as `agent:<agentId>:subagent:<uuid>` and the runner reuses that persistent session instead of forcing isolated-session rollover.

## What This Changes

Request cron agent-turn jobs route directly into existing persistent subagent sessions instead of only main or isolated sessions.

## Best Possible Solution

Keep the generic persistent-session cron target and direct users to `--session "session:<full-session-key>"` for subagent sessions, with `--agent <agentId>` when the target session belongs to a non-default agent. Avoid adding a subagent-only alias unless a future report shows a real missing API beyond the existing session-key route.

## Reproduction Assessment

Not applicable as a bug reproduction, but there is a high-confidence verification path: create or inspect a cron agent-turn job with `sessionTarget: "session:agent:<agentId>:subagent:<uuid>"` and confirm the Gateway passes the unwrapped key to the runner while the runner does not force a fresh isolated session.

## Solution Assessment

Yes. The generic `session:<id>` target is the narrowest maintainable solution because it covers subagent, named, and other persistent sessions without baking subagent-specific policy into cron core.
