## Summary

Current main already supports the requested routing through the generic persistent cron session target. A cron agent-turn job can use `sessionTarget: "session:<full-session-key>"` or `openclaw cron --session "session:<full-session-key>"`; subagent session keys use the same `agent:<agentId>:subagent:<uuid>` shape, so a separate `subagent` target spelling is not needed.

## What This Changes

The issue requests a way for cron agent-turn jobs to run in an existing persistent subagent session instead of creating an isolated cron session.

## Best Possible Solution

Keep the generic persistent-session cron target as the supported interface. Users should schedule subagent wakeups with `--session "session:<full subagent session key>"`; for non-default agents, also pin `--agent <agentId>` so the job uses the intended agent configuration.

## Reproduction Assessment

Not applicable as a feature request, but the requested behavior has a high-confidence verification path: inspect a cron job with `sessionTarget: "session:agent:<agentId>:subagent:<uuid>"` and confirm the scheduler passes that full key to the runner instead of creating a new `cron:<jobId>` session.

## Solution Assessment

Yes. The existing `session:<id>` target is the narrowest maintainable solution because it works for all persistent session keys, including subagent keys, without adding a subagent-specific cron alias or special-case core behavior.
