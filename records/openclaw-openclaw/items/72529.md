## Summary

Current main already supports the requested cron-to-persistent-subagent-session flow through the generic `sessionTarget: "session:<id>"` target. A subagent session key such as `agent:<agentId>:subagent:<uuid>` is a valid persistent session id for that path, while `--session-key` is not the routing knob for binding cron execution to an existing session.

## What This Changes

Request cron agent-turn jobs be able to run inside an existing persistent subagent session created by `sessions_spawn` with `thread: true` and `mode: "session"`.

## Best Possible Solution

Keep the generic persistent-session cron target and route users to `openclaw cron --session session:<full-session-key>` using the subagent `childSessionKey`; adding a separate `subagent` target spelling would duplicate an existing session-key abstraction.

## Reproduction Assessment

Not applicable as a feature request; the high-confidence verification path is source/test inspection on current main. Existing tests cover creating/running a cron job with `sessionTarget: "session:project-alpha-monitor"` and prove the runner receives that persistent session key.

## Solution Assessment

Yes. The current implementation is the narrowest maintainable solution because subagent sessions are already ordinary persistent session keys, so cron does not need subagent-specific routing logic.
