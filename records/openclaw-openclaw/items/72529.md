## Summary

Current main already implements the requested cron routing through the generic persistent-session target: cron agent-turn jobs accept `sessionTarget: "session:<id>"`, the Gateway strips that prefix and passes the persistent key to the runner, and isolated-only fresh-session behavior is only forced for `sessionTarget: "isolated"`. The issue's `--session-key` observation is accurate, but that flag is not the routing knob; the supported path is `openclaw cron ... --session session:<full-session-key>`.

## What This Changes

Request to let scheduled cron agent-turn jobs run inside an existing persistent subagent session instead of only the main session or a fresh isolated session.

## Best Possible Solution

Keep the shipped generic persistent-session target and document/use `--session session:<full-session-key>` for subagent session keys; avoid adding a subagent-specific alias unless a separate API gap appears. Users should continue treating `--session-key` as metadata/grouping and set `--agent <id>` when they need a specific agent workspace/config for the run.

## Reproduction Assessment

Yes. A high-confidence verification path exists in current main: create or inspect a cron job with `sessionTarget: "session:<key>"` and observe the Gateway passes `<key>` as the runner `sessionKey`; existing tests cover that route without requiring a live provider run.

## Solution Assessment

Yes. The generic `session:<id>` target is the narrowest maintainable solution because it works for subagent keys and other persistent session keys without hard-coding subagent-specific policy into cron.
