## Summary

Keep this PR open, but do not merge it as-is. The reported Bun bootstrap bug is real and current main still lacks Bun-aware direct-miss detection, but the proposed specifier-only check weakens the existing direct-import boundary and can mask a transitive Bun resolution failure that reuses the same relative specifier.

## What This Changes

The PR updates `openclaw.mjs` so `isDirectModuleNotFoundError` recognizes Bun `ERR_MODULE_NOT_FOUND` errors by `err.specifier` and relative-specifier message text.

## Best Possible Solution

Amend the Bun path so it only returns true when `err.specifier` matches and Bun's `err.referrer`, when present, points to the launcher module. Add focused launcher coverage for a Bun-shaped direct optional miss and a Bun-shaped transitive miss before landing.

## Reproduction Assessment

Yes. The linked bug report gives concrete Bun steps and an error shape, and source inspection confirms current main only handles Node-style `err.url` or absolute-path messages.

## Solution Assessment

No. The direction is right, but the implementation should use Bun's `referrer` field to preserve the helper's direct-miss contract instead of relying on specifier/message text alone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require Bun referrer before swallowing missing imports:** `openclaw.mjs:65`
  - body: This helper is used to hide only direct launcher import misses. Bun's ResolveMessage also carries `referrer`; matching only `err.specifier` means a module loaded by the launcher that fails on the same relative specifier can be mistaken for the top-level miss and hidden behind fallback behavior. Please require the Bun referrer to be the launcher before returning true and cover that transitive case.
  - confidence: 0.86
