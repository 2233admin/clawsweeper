## Summary

Keep open. Current main still only supports TTL-gated context pruning, and the linked rolling-pruning PR is open source work rather than a merged fix. The older matching issue is closed as not planned, so this issue should remain available until a current-main implementation lands or maintainers choose the canonical config shape.

## What This Changes

Add turn-based or rolling context pruning so old tool results can be pruned during continuously active sessions without waiting for a cache-TTL idle window.

## Best Possible Solution

Implement an additive rolling pruning path in the existing context-pruning runtime and config contract, with schema/types/generated schema, docs, changelog, and focused tests proving cache-ttl behavior is preserved while rolling mode can prune on every context event.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with `cache-ttl` and a recent `lastCacheTouchAt`, `context-pruning/extension.ts` exits before `pruneContextMessages`, and the existing test asserts the immediate second pass after a prune returns undefined.

## Solution Assessment

Unclear as written. A rolling mode is the narrowest likely fix, but maintainers should choose the public config shape and decide whether #13570's threshold bypass remains separate before merging #43238 or a replacement.
