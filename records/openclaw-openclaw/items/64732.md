## Summary

Keep open: current main partially addresses this class for `openclaw help` and the reserved `tools` namespace, but the generic fallback still treats unregistered non-plugin primary tokens as plugin IDs under a restrictive `plugins.allow`. The maintainer discussion already made this the canonical tracker for the broader command-token/plugin-ID confusion, and source evidence shows the remaining `tool`-style case is still valid.

## What This Changes

The issue asks OpenClaw to stop treating built-in, framework, or synthetic CLI command tokens as plugin IDs when `plugins.allow` is restrictive, especially for `openclaw tool` and `openclaw help`.

## Best Possible Solution

Add an explicit CLI command-classification layer before the plugin allowlist fallback: framework/core/reserved non-plugin names should never suggest adding themselves to `plugins.allow`, manifest CLI aliases should resolve to their owning plugin ID, runtime slash aliases should explain chat-only usage or the related CLI command, and truly unknown commands should fall through to Commander-style unknown-command handling.

## Reproduction Assessment

Yes. The issue gives a concrete restrictive `plugins.allow` config and command sequence, and current source still routes an unregistered primary token such as `tool` into the missing-plugin fallback; `help` is partially fixed by a fast path, but the remaining non-plugin-token case is reproducible by code inspection.

## Solution Assessment

Yes. The best fix is a narrow classification/diagnostic change in the CLI startup path, not adding pseudo command IDs to `plugins.allow` or restoring an undocumented `openclaw tool` command surface.

## Repair Work Prompt

Fix the CLI startup allowlist diagnostic bug tracked by https://github.com/openclaw/openclaw/issues/64732. Current `main` partially fixes `openclaw help`, but unregistered non-plugin primary tokens such as `tool`, `run`, and reserved chat/framework names can still flow through `resolveMissingPluginCommandMessage()` and receive misleading `plugins.allow excludes "<token>"` guidance. Implement a small explicit command-classification step before the plugin allowlist fallback. Preserve existing behavior for real plugin IDs and manifest-backed CLI aliases: if the owning plugin is missing from `plugins.allow`, point at the owning plugin ID; if the parent plugin is disabled, keep the existing disabled-entry guidance; runtime slash aliases should say they are chat-only or point to the related CLI command. Non-plugin framework/core/reserved names and truly unknown tokens should not suggest adding themselves to `plugins.allow`; they should either use a clear non-CLI/unknown-command message or fall through to Commander unknown-command handling. Do not resurrect an undocumented `openclaw tool` command unless maintainers explicitly ask; current docs use `openclaw infer image generate` for image generation. Add focused Vitest coverage around `help`, `tool`, `run`, `subagents`, a real plugin ID such as `browser`, and a manifest alias such as `wiki`/`dreaming`. Add a user-facing changelog entry with reporter credit, likely `Thanks @efe-arv`, and include related duplicate refs if appropriate.
