## Summary

Keep open. Current main still has a credible core tool-policy mismatch: required plugin tools are materialized, but the default coding profile can filter them down to the core `group:memory` names, which explains why only `memory_search` and `memory_get` remain visible.

## What This Changes

The issue reports that after upgrading to OpenClaw 4.26, a MemOS plugin registering multiple required tools via `api.registerTool()` exposes only `memory_search` and `memory_get` to the agent.

## Best Possible Solution

Define the supported contract for required plugin tools under tool profiles, especially memory-slot plugins. A durable fix should add an explicit safe policy/group mechanism or clear doctor/docs opt-in guidance, then cover a memory plugin that registers `memory_search`, `memory_get`, and additional required tools under the default coding profile.

## Reproduction Assessment

Yes for the core policy path: a plugin registering required tools beyond `memory_search` and `memory_get` under `tools.profile: "coding"` will be materialized and then filtered by the profile allowlist. The exact MemOS/Windows setup was not live-run during this read-only review.

## Solution Assessment

Unclear because there is no proposed patch yet. Exposing every required plugin tool by default would address the symptom but is not obviously the best fix because some reported tools are mutating or publishing capabilities; the safer direction is an explicit maintainer-approved policy contract.
