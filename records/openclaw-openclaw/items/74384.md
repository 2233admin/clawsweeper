## Summary

Keep open. Current main still has a credible OpenClaw policy path for the reported behavior: non-optional plugin tools are materialized, then the default coding profile can filter third-party tools down to explicitly allowed names such as `memory_search` and `memory_get`. The related hook-access workaround is useful context but does not resolve this tool-visibility contract.

## What This Changes

The issue reports that a third-party memory plugin's required `api.registerTool()` tools are no longer exposed to the agent after upgrading to OpenClaw 4.26, leaving only `memory_search` and `memory_get` available.

## Best Possible Solution

Define and document the intended contract between required plugin tools and tool profiles. A durable fix should either add a safe plugin-owned policy mechanism for active memory plugins, or make the explicit opt-in path and diagnostics clear, with regression coverage for a memory plugin that registers more than `memory_search` and `memory_get`.

## Reproduction Assessment

Yes for the core mechanism: a synthetic enabled memory plugin that registers `memory_search`, `memory_get`, and one extra required tool under `tools.profile: "coding"` should exercise the same `createOpenClawCodingTools()` and policy pipeline path. The exact external MemOS plugin on Windows was not live-verified here.

## Solution Assessment

Unclear because there is no proposed patch and the open question is the product/security contract. A blanket expose-all change would be risky; a maintainer-approved active-memory/plugin policy slot or explicit opt-in contract is safer.
