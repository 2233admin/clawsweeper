## Summary

Keep open. Current main still has the reported failure shape: plugin tools are materialized, then filtered through the active tool profile, and the default/coding memory surface only names `memory_search` and `memory_get`. The #73748 workaround covers conversation hooks, but this issue is specifically about agent-visible tools registered with `api.registerTool()`.

## What This Changes

The issue reports that after upgrading to OpenClaw 4.26, a third-party MemOS memory plugin's required `api.registerTool()` tools beyond `memory_search` and `memory_get` register but are not exposed to the agent.

## Best Possible Solution

Define the intended contract for required third-party plugin tools under tool profiles, especially memory-slot plugins. A durable fix should either add an explicit safe policy/group mechanism for selected memory-plugin tools, or provide clear doctor/docs guidance for opting into the full plugin tool set, with regression coverage for a memory plugin that registers more than `memory_search` and `memory_get`.
