## Summary

Keep open. Current main still has a credible OpenClaw bug/policy mismatch: required third-party plugin tools are materialized, but the default coding profile can filter them back down to the two core memory tool names. The related #73748 workaround covers conversation-hook access, not model-visible tool exposure.

## What This Changes

The report asks OpenClaw to restore exposure of third-party tools registered with `api.registerTool()` after the 4.26 upgrade, where only `memory_search` and `memory_get` remain visible to the agent.

## Best Possible Solution

Define the contract between required plugin tools and restrictive tool profiles, then align implementation, docs, diagnostics, and tests. A durable fix should either provide a maintainer-approved profile-stage opt-in for memory-like plugins or make the `tools.alsoAllow` / plugin-id allow path explicit enough for plugin authors and users to configure safely.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the core mechanism: register a synthetic plugin with `memory_search`, `memory_get`, and one additional required tool, then run tool assembly with `tools.profile: "coding"` and observe the policy pipeline keep only the profiled names. The exact external Windows MemOS install was not live-run in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? Unclear because there is no proposed patch and the open point is product/security policy. A broad change that exposes all required third-party tools through restrictive profiles is risky; a maintainer-approved memory/plugin policy or explicit opt-in path is safer.
