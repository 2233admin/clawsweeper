## Summary

Keep open. Current main still tails `logs.tail` or the configured local file fallback and does not prefer the active Linux systemd journal or print service provenance before following logs. The reporter's validated prototype comment is useful, but that behavior is not present in the checked-out source, tests, or docs, and no PR mentioning this issue was found.

## What This Changes

The issue requests that Linux `openclaw logs --follow` prefer the active gateway systemd journal after side-by-side cutover and print CLI, service, and log-source provenance with mismatch warnings before tailing.

## Best Possible Solution

Implement a Linux-only local CLI path that detects the active `openclaw-gateway.service`, tails the active service journal when available, prints clear provenance and mismatch warnings before following, and falls back to file logs only with an explicit warning when service-aware logging is unavailable.

## Repair Work Prompt

Fix the Linux local `openclaw logs --follow` behavior for this issue. Implement a focused change in `src/cli/logs-cli.ts` and supporting helpers/tests as needed: when running on Linux with the implicit local gateway connection and an active `openclaw-gateway.service` is available, prefer tailing the active service journal, ideally `journalctl --user _PID=<pid>`, instead of falling back to `readConfiguredLogTail` for pairing-required/scope/close/timeout local RPC failures. Before following, print provenance in text mode: CLI version, active service ExecStart, active service PID, service version when it can be determined without hanging, exact log source, and a clear mismatch warning when CLI/service/source disagree. Preserve explicit `--url` behavior and Control UI / gateway `logs.tail` file-log behavior. Keep file-tail fallback available only when service-aware journal access is unavailable and make that fallback explicit. Design `--json` provenance carefully with typed metadata/notice events so existing log event consumers are not broken unnecessarily. Add focused tests in `src/cli/logs-cli.test.ts`; add or reuse daemon/systemd helper tests only if new helpers are introduced. Update `docs/cli/logs.md` and `docs/logging.md` for the Linux journal preference and fallback semantics. Do not change unrelated logging formats or global service install behavior.
