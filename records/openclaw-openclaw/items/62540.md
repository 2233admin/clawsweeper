## Summary

Keep this issue open. Current main recognizes provider 429s as rate-limit failures and can rotate auth profiles or continue through configured model fallback, but it does not implement the requested durable paused-for-rate-limit state, provider-hint retry scheduling, persisted retry metadata, or automatic continuation of the same pending model call. The open #49251 is closely related, but this issue has more specific acceptance criteria that remain useful unless maintainers choose the broader queueing issue as canonical.

## What This Changes

Request automatic pausing and retrying of active OpenClaw runs on provider 429/rate-limit errors, including retry-delay parsing, persisted retry metadata, retry scheduling with backoff/jitter, user-visible status, and automatic resumption without a new prompt.

## Best Possible Solution

Add a first-class rate-limit pause/resume lane in the agent run orchestration: classify 429s with provider retry hints, persist the pending retry metadata, expose retry status and time, schedule a bounded wake with jitter/backoff, retry the pending model call, and fall back to the existing auth-profile/model fallback path when retries are exhausted or unsafe.

## Reproduction Assessment

Yes. By code inspection, a synthetic provider 429 follows the existing `rate_limit` classification into auth-profile rotation or model fallback, and no current path persists `retry_at`, exposes a rate-limit pause state, or schedules a wake for the same pending call.

## Solution Assessment

No. Current main’s fallback/profile-rotation behavior is useful immediate recovery, but it is not the requested durable pause and automatic retry at the provider-supplied delay; a narrower implementation should extend the run orchestration/status boundary rather than only changing error text.
