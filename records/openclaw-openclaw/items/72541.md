## Summary

Keep open. Current main still has the reported false-negative shape: accepted `sessions_spawn` results are available as structured tool results and are classified as side effects, but the embedded-runner incomplete-turn classifier only exempts silent replies and committed user-visible messaging delivery, not successful spawn-only relay turns.

## What This Changes

The issue requests that gateway completeness checks treat a successful zero-payload `sessions_spawn` turn as terminal success for pure-relay agents instead of surfacing an incomplete-turn warning.

## Best Possible Solution

Teach the embedded-runner completion classifier to recognize a current-attempt accepted `sessions_spawn` result with a non-empty child session key as completed work, while preserving the existing warning path for failed spawns and empty turns without successful tool completion.

## Reproduction Assessment

Yes. A focused unit reproduction can construct a zero-payload attempt with `lastAssistant.stopReason: "stop"`, `sessions_spawn` side-effect metadata, and an accepted structured tool result, then verify current `resolveIncompleteTurnPayloadText` still returns the warning text.

## Solution Assessment

Yes. The narrowest maintainable fix is a structured success exemption in the incomplete-turn helper, backed by regression tests, rather than changing `sessions_spawn` semantics or disabling empty-turn detection globally.

Fix https://github.com/openclaw/openclaw/issues/72541. Current main treats a zero-payload turn that successfully calls `sessions_spawn` and then stops as an incomplete turn.
