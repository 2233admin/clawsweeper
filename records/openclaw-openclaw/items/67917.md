## Summary

Keep open. Current main still hard-blocks normal OpenAI Codex browser OAuth when the TLS preflight classifies a certificate error, and an in-tree regression test asserts that behavior. The device-code path and adjacent proxy/auth fixes are useful context, but they do not prove the reported macOS custom-CA browser-OAuth and runtime-fetch regression is fixed.

## What This Changes

The issue asks OpenAI Codex OAuth precheck and runtime requests to work with a valid custom TLS/CA setup instead of failing with certificate and fetch errors.

## Best Possible Solution

Keep the issue open until OpenAI Codex browser OAuth no longer treats the speculative TLS preflight as a hard blocker when the real upstream OAuth flow can proceed, and until the runtime transport has a documented, tested story for effective custom CA/proxy/env inheritance on macOS.
