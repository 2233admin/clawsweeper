## Summary

Keep open. The report is current and concrete, and current main still allows parallel subagent starts on a shared gateway lane with no model-warmup queue or worker offload. The central subagent/concurrency files are unchanged from v2026.4.29, so there is no implemented-on-main evidence for the requested fix.

## What This Changes

The issue reports gateway event-loop saturation and 1012 restart when three heavy thread-bound `sessions_spawn` subagents start in parallel, requesting nonblocking model startup, spawn throttling, better diagnostics, and tunable stuck-session thresholds.

## Best Possible Solution

Profile v2026.4.29 and current main under the reported parallel `sessions_spawn` workload, identify the blocking startup phase, then keep Gateway RPC/liveness responsive by adding a bounded subagent startup/model-prep queue, targeted yielding, caching, or worker/process isolation at the measured hotspot.

## Reproduction Assessment

Unclear. The issue gives concrete steps and logs, and code review confirms current main still permits the reported three-way subagent burst, but I did not have a live Ollama cloud/gateway setup to reproduce the event-loop delay and 1012 restart end to end.

## Solution Assessment

Not applicable because there is no patch. The proposed directions are plausible, but the best fix should first distinguish gateway prep starvation from provider/model backend latency and then throttle or offload only the responsible phase.
