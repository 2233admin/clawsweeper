## Summary

Keep open. Current main still has the reported WebChat ordering and refresh races, and this active contributor PR is a coherent implementation candidate, but the proposed reset path still leaves stale tool-stream artifacts visible until history reload succeeds, so it should not merge as-is.

## What This Changes

The PR changes Control UI WebChat and Gateway history projection to stabilize transcript timestamps/keys, reconcile stale history loads, defer unrelated finals during active streams, reset chat views after session resets, trim cumulative stream prefixes, and render assistant stream failures as visible errors.

## Best Possible Solution

Keep this PR open as the implementation candidate, have the author clear tool-stream state in the synchronous reset path before reloading history, then rerun the focused WebChat/Gateway tests and let required CI finish on the latest head.

## Reproduction Assessment

Yes. Current main has source-level reproduction paths for stale chat.history replacing newer local state, unrelated final messages appending during an active stream, and cumulative stream text being duplicated at tool boundaries; the reset-path finding can be reproduced by resetting after a tool/streamed response while delaying or failing chat.history.

## Solution Assessment

No, not as-is. The overall approach is scoped to the right Control UI/Gateway surfaces, but the reset implementation should also call the existing tool-stream reset path before history reload so stale stream/tool artifacts cannot survive the visible clear.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Clear tool-stream state during synchronous resets:** `ui/src/ui/app-chat.ts:649-650`
  - body: The new bare /new and /reset path clears `chatMessages` after `sessions.reset`, but it leaves `chatStreamSegments`, `chatToolMessages`, and the tool stream maps untouched until `loadChatHistory` succeeds. `buildChatItems` renders those stream/tool artifacts independently of `chatMessages`, so after a reset with prior tool output the active pane can still show stale stream/tool cards, and they persist if `chat.history` fails. Reset the tool stream in this path before reloading history.
