## Summary

Keep open. Current main still has the WebChat history/stream reconciliation behavior this PR is trying to fix, but the proposed patch introduces two concrete correctness regressions around stream-prefix trimming and deferred-message deduplication, so it needs author revision and maintainer review rather than cleanup closure.

## What This Changes

The PR changes Gateway chat-history projection/session transcript timestamp metadata plus Control UI chat keying, stale history merging, active stream preservation, deferred assistant finals, stream-prefix trimming, and regression tests.

## Best Possible Solution

Revise the PR so stream text has a single representation boundary, either raw cumulative text or already-trimmed suffix but not both, and preserve distinct deferred finals using stable message/run metadata instead of text-only signatures; then land after targeted UI/gateway tests, lint/format checks, UI build proof, and a real WebChat refresh smoke when feasible.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Avoid trimming committed stream prefixes twice:** `ui/src/ui/controllers/chat.ts:826`
  - body: The PR stores a trimmed suffix in `state.chatStream` for each delta, but render/final/tool-boundary paths trim `chatStream` against the same committed segments again. If the true suffix starts with the same text as the committed prefix, for example committed `A` and cumulative stream `AA reply`, the second trim drops legitimate output from the live bubble and any fallback final message. Keep `chatStream` cumulative until render/final time, or make downstream paths treat it as already trimmed.
  - confidence: 0.87
- **[P2] Preserve repeated deferred final messages:** `ui/src/ui/controllers/chat.ts:309`
  - body: `queueDeferredChatMessage` dedupes deferred finals only by role plus rendered text/content. Two unrelated background finals that both say something like `Done` while an active run is streaming collapse into one queued message, so the second result is never flushed. Use stable message/run metadata when present, or keep repeated finals instead of content-only deduping.
  - confidence: 0.81
