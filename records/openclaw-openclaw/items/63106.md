## Summary

Keep open. Current main still records the main-session `systemEvent`/`next-heartbeat` handoff as `ok` immediately after enqueueing and requesting a heartbeat, with no short-duration ghost-run warning, run-log annotation, or configurable threshold. The newer #72677 PR is open, unmerged, and explicitly targets this issue, so the issue should remain open until that PR lands or maintainers choose a different fix path; the earlier #63111 PR was closed unmerged.

## What This Changes

Main-session `systemEvent` cron jobs using `wakeMode: "next-heartbeat"` can finish in a few milliseconds and be logged as `ok` without evidence that the agent processed the event, and the request asks for a configurable ghost-run warning/run-log signal.

## Best Possible Solution

Review and land #72677 or an equivalent maintainer-approved change that gives this handoff a scoped warning/run-log signal, exposes the threshold through cron config/schema/help, and adds gateway/run-log regression coverage. If the related payload-loss report proves the root cause is deeper, the durable end state should distinguish queued heartbeat handoff from confirmed agent processing instead of relying only on elapsed time.
