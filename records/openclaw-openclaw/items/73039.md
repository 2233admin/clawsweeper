## Summary

Keep open. Current main has not implemented this memoization, and the live PR still keys web-tool memoization on a runtime metadata object that is cloned on every read, so active runtime web providers miss the intended warm cache; the PR also lacks the required changelog entry for a user-facing performance change.

## What This Changes

The PR adds a WeakMap-chain memoization helper with unit tests and wraps image, media generation, PDF, and web tool factory construction inside createOpenClawTools.

## Best Possible Solution

Revise the web-tool memoization to key on stable runtime-provider fingerprints, or expose a stable readonly runtime metadata snapshot, then add focused coverage through the real runtime metadata getter and include an Unreleased changelog entry for the performance change.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: set active runtime web metadata, call createOpenClawTools twice, and the production getter returns a new structured clone each time, causing refs: [config, runtimeWebTools] to miss the WeakMap-chain cache for web tools.

## Solution Assessment

No. Process-lifetime memoization is a plausible narrow direction, but keying web tools on the cloned runtime metadata object preserves invalidation by defeating warm-cache hits for active runtime providers; stable scalar fingerprints or a stable snapshot are safer.
