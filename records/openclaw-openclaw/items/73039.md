## Summary

Keep open. Current main has no factory memoization, and this PR targets a plausible agent hot path, but the latest diff still keys web-tool caching on `runtimeWebTools` object identity even though current main returns defensive clones. The earlier review discussion caught stale-provider and cache-growth risks, and the author addressed those in shape, but the remaining key design defeats warm hits for active runtime web providers.

## What This Changes

The PR adds a `memoizeToolFactory` helper with unit tests and wraps the image, media generation, PDF, web_search, and web_fetch factory calls inside `createOpenClawTools`.

## Best Possible Solution

Keep the factory memoization direction, but make the key stable and complete: use scalar fingerprints for runtime web provider metadata, keep object refs only for stable objects such as config and sandbox bridge, and add focused `createOpenClawTools` coverage proving both provider-change invalidation and warm-cache hits.

## Reproduction Assessment

Yes. With active runtime web metadata, `getActiveRuntimeWebToolsMetadata()` returns a fresh structured clone on each call, so repeated `createOpenClawTools` calls on the PR pass distinct `runtimeWebTools` refs into the memoizer for web_search and web_fetch.

## Solution Assessment

No. Memoizing these factories is a narrow maintainable approach, but keying web tools on cloned runtime metadata object identity preserves invalidation by sacrificing the warm-cache behavior the PR is meant to provide.
