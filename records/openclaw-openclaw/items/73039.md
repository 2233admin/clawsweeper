## Summary

Keep open. Current main still constructs these tools directly, and the latest PR head still has a concrete correctness/performance issue: the web-tool memoization keys on a freshly cloned runtime metadata object, so the runtime web path misses the cache across turns instead of amortizing the expensive factory work.

## What This Changes

The PR adds a WeakMap-chained `memoizeToolFactory` helper with unit tests and wraps the image, media generation, PDF, and web tool factory calls inside `createOpenClawTools`.

## Best Possible Solution

Revise the web-tool cache key to use stable scalar runtime-provider fingerprints, or expose a stable readonly runtime metadata snapshot, then add focused coverage through `createOpenClawTools` and the real runtime metadata getter so provider-change invalidation and warm-turn cache hits are both proven.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use a stable runtime web-tools fingerprint:** `src/agents/openclaw-tools.ts:257-269`
  - body: `getActiveRuntimeWebToolsMetadata()` returns a structured clone on every call, so putting `runtimeWebTools` in `refs` gives the web factory wrappers a new WeakMap key each turn whenever runtime metadata is active. That leaves `createWebSearchTool` and `createWebFetchTool` effectively unmemoized for the runtime-provider path this PR targets; key on stable scalar provider metadata or expose a stable snapshot instead.
