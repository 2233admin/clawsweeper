## Summary

Keep open. Current main still resolves and returns assistant identity emoji from the gateway, but the Control UI bootstrap/state/chat rendering path does not carry an emoji field into assistant chat group footers, so the reported v2 UI regression remains observable in source. The earlier #45898 fix attempt is closed unmerged and targeted the agent selector dropdown rather than the transcript sender label described in the issue discussion.

## What This Changes

The issue asks to restore the configured agent identity emoji in the Control UI chat transcript where v1 rendered it inline with the agent name.

## Best Possible Solution

Carry the assistant identity emoji through the bootstrap/RPC normalization and app state, then render it inline with the assistant sender name for streaming and persisted chat groups while preserving current avatar override behavior and the existing sessions-table emoji handling.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock bootstrap or agent.identity.get with an emoji, render assistant chat groups, and observe that the current app state and render props discard that value before .chat-sender-name is produced.

## Solution Assessment

Yes. A narrow Control UI propagation and rendering fix is the best path because the gateway already resolves emoji; changing only selector dropdowns or avatar fallback behavior does not restore the reported transcript sender label.
