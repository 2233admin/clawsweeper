## Summary

Keep this PR open. Current main accepts the `agents.list[].sandbox.sessionToolsVisibility` config shape, but the runtime session visibility path still resolves the sandbox clamp from `agents.defaults` only, and the affected session tools are not wired with `requesterAgentIdOverride`. The PR remains a focused implementation candidate for access-control-sensitive runtime behavior, with useful discussion around requester identity, guard routing, custom `session.mainKey`, and `session_status`.

## Best Possible Solution

Keep this PR open as the focused implementation candidate unless #59170 supersedes it first. The best path is to review or rework the PR so `sessions_list`, `sessions_history`, `sessions_send`, and `session_status` consistently derive the effective requester agent, apply `agents.list[].sandbox.sessionToolsVisibility` before falling back to `agents.defaults`, preserve A2A checks and spawned-tree restrictions, handle custom `session.mainKey`, and update focused tests/docs.

- Current main has schema acceptance without matching runtime behavior, so users can configure `agents.list[].sandbox.sessionToolsVisibility` today but the session tools do not honor it.
- Open PR #59170 overlaps broader per-agent `tools.sessions.visibility` and `tools.agentToAgent` scoping; whichever PR lands should reconcile config precedence and tests to avoid two divergent per-agent visibility models.
