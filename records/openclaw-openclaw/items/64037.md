## Summary

Keep this PR open. Current main still does not expose inbound media metadata on `PluginHookAgentContext` for `before_agent_reply`, and handled hooks still return before the normal inbound media staging path. The proposed change addresses a real plugin API gap, but the branch is stale against the current hook type split and needs tests, docs, and explicit maintainer API/security review before merge.

## What This Changes

The PR adds `mediaPaths`, `mediaUrls`, and `mediaTypes` to the `before_agent_reply` plugin hook context and populates them from inbound message media metadata.

## Best Possible Solution

Land a rebased additive plugin hook API in the current hook contract file, preferably with normalized defensive-copy media fields or a structured media attachment shape, plus public docs and focused tests that define path, URL, MIME type, and sandbox/staging semantics.
