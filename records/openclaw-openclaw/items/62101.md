## Summary

Keep open. The BlueBubbles bug still appears valid on current main, but this PR should not merge as-is because it puts E.164 coercion in a shared allowFrom schema used by other channel plugins, which would regress QQBot numeric sender allowlists during normal config parsing.

## What This Changes

The PR changes shared allowFrom config parsing to turn numeric entries into +E.164 strings, updates BlueBubbles handle normalization for all-digit inputs, and adds tests for both behaviors.

## Best Possible Solution

Keep the shared allowFrom schema lossless and solve the issue inside the BlueBubbles-owned config/runtime path: normalize numeric or all-digit BlueBubbles phone handles to +E.164, preserve chat_id/chat_guid/email entries, and add cross-channel regression coverage proving QQBot numeric allowlists remain unchanged.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: BlueBubbles dmPolicy=allowlist with allowFrom [19175551234] stringifies to 19175551234 while incoming senders remain +19175551234, and the PR's shared transform would make QQBot allowFrom [12345] parse as +12345 while QQBot compares against sender 12345.

## Solution Assessment

No. The BlueBubbles runtime guard is directionally useful, but the schema transform belongs in BlueBubbles-specific parsing or normalization, not in the shared AllowFromEntrySchema used by other channel plugins.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Keep E.164 coercion out of the shared allowFrom schema:** `src/channels/plugins/config-schema.ts:19`
  - body: `AllowFromEntrySchema` is used by QQBot and other channel plugins, not just BlueBubbles. Coercing every numeric entry to `+${n}` means a QQBot config like `allowFrom: [12345]` parses to `+12345`, but QQBot normalizes sender IDs to bare strings before comparison, so legitimate numeric sender IDs no longer match. Move this coercion into the BlueBubbles-owned schema/runtime path and keep the shared schema lossless.
  - confidence: 0.92
