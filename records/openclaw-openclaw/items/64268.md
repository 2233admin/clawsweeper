## Summary

Keep open. Current main still emits `llm_output` once after the embedded attempt with aggregate usage, while per-assistant-message handling only records/commits usage and does not emit a per-call output hook. The related PR #45990 is still open and unmerged, so it is an implementation candidate rather than shipped behavior.

## What This Changes

Request per-LLM-call plugin observability for agentic loops, either by emitting `llm_output` per iteration with aggregate semantics preserved or by adding a new per-call hook.

## Best Possible Solution

Land an additive, documented per-call observability contract in the embedded runner/subscription path. It should expose per-call usage plus provider/model/run/session/agent identifiers, preserve or clearly flag aggregate compatibility, and include focused multi-round tests plus plugin hook documentation.
