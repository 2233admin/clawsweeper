## Summary

Keep open. Current main has adjacent WebChat history, active-branch, and duplicate-send protections, but I could not prove that the specific restart/compaction migration path that replays an old transcript into a new session is fixed. If that replay still writes old rows as fresh transcript messages, the current transcript-update fanout and WebChat history reload path would still surface those persisted duplicates.

## What This Changes

Prevent session restart or migration from replaying prior session transcript rows as new WebChat messages while keeping prior context available through `chat.history`.

## Best Possible Solution

Add a focused regression for context-overflow/restart recovery or migration where an old transcript remains readable through `chat.history` without being appended or broadcast as fresh `session.message` rows, then patch the narrow writer/recovery/import path that performs the replay.

## Reproduction Assessment

Unclear. The reporter supplied credible steps and transcript timing, but this review found no current-main test or live verification for the exact restart/compaction migration plus WebChat reconnect path.

## Solution Assessment

Unclear.
