## Summary

Current main now covers the reported replay/duplication path: WebChat history is projected from the active transcript branch, compaction transcript rotation prunes duplicate replayed user-tail entries, and restart recovery resumes with a single continuation turn rather than broadcasting old history as fresh messages. The proof is on current main only; no tag in this checkout contains the proof commit.

## What This Changes

The issue requests that session restart or migration stop replaying old transcript history into WebChat as duplicate visible messages while keeping history available through `chat.history`.

## Best Possible Solution

Keep the current-main approach: history consumers should read the active transcript branch through `chat.history`, compaction rotation should prune duplicate replay tails, and restart recovery should resume through one continuation turn instead of rebroadcasting prior history. If a newer build still reproduces the problem, the follow-up should include a fresh transcript slice with sessionId/sessionKey rotation details.

## Reproduction Assessment

Yes, at the focused source/regression level: current tests cover active-branch history projection, duplicate compaction-tail pruning, and WebChat reload reconciliation. A full live context-overflow restart was not run in this read-only pass.

## Solution Assessment
