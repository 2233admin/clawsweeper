## Summary

Keep this open. The report is recent and has concrete replay timestamps plus related issue/PR context, but current main does not show a complete fix or regression test for the specific context-overflow restart/migration path that re-writes old transcript history into a same-key new session.

## What This Changes

The issue reports that after a session restart or migration with a new sessionId and the same sessionKey, old conversation history is replayed into WebChat as duplicate visible messages and redundant transcript rows.

## Best Possible Solution

Add a focused current-main regression around context-overflow restart or session migration where old history remains available through `chat.history` without being appended again as live transcript messages, then fix the replay/write path with stable transcript ids or idempotency rather than only suppressing UI rendering.

## Reproduction Assessment

Unclear. The issue gives concrete timestamps and a plausible workflow, but this read-only review did not run a live context-overflow restart; existing tests cover related WebChat reload and `chat.send` races, not the specific migration replay path.

## Solution Assessment

Unclear. A broadcast-handler-only fix is unlikely to be sufficient because current main already Set-dedupes subscribers; the safer fix is to locate the path that re-appends old history and make that replay/import idempotent or file-only.
