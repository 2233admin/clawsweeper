## Summary

This issue should stay open. Current main still supports the explicit `models.providers.*.request.proxy.mode = "env-proxy"` workaround, but it does not implement the requested startup warning when proxy env vars are present and remote providers lack request proxy configuration. The related implementation candidate, #68023, is closed and unmerged, so it is useful prior design context rather than shipped behavior.

## Best Possible Solution

Implement a non-breaking startup warning in or near `logGatewayStartup`: detect configured proxy environment variables with the existing proxy-env helpers, inspect configured model providers for remote HTTP(S) targets, skip local/private/non-remote providers and providers with explicit `request.proxy`, and emit one concise warning that points users to `models.providers.<provider>.request.proxy.mode = "env-proxy"`. Add focused startup-log tests for the warn, no-env, all-configured, and local-provider cases, using #68023 as prior design context but not as shipped code.

## Risks / Open Questions

- The warning needs careful remote-target classification so loopback, `.local`, bare-host, LAN/private, and explicitly proxied providers do not produce noisy or misleading startup messages.
- The implementation should remain warning-only; auto-enabling env proxy would change security and routing semantics that current guarded fetch tests intentionally preserve.
- The warning should avoid printing proxy URLs, credentials, or provider secrets; provider ids and the config remediation path are enough.
