## Summary

Keep this open. Current main still has explicit model-provider `env-proxy` support, but the gateway startup logger does not implement the requested warning for proxy environment variables plus remote providers without explicit request proxy configuration; the directly related implementation PR is closed and unmerged.

## What This Changes

Request a non-breaking startup warning when `HTTP_PROXY`/`HTTPS_PROXY` is present and remote model providers are not explicitly configured with `models.providers.<provider>.request.proxy.mode = "env-proxy"`.

## Best Possible Solution

Add a warning-only gateway startup check that reuses existing proxy-env helpers, inspects configured model providers for remote HTTP(S) targets without explicit request proxy configuration, emits one actionable remediation message, and adds focused startup-log tests for warn, no-env, all-configured, and local/non-remote cases.

## Repair Work Prompt

Implement the startup warning requested by openclaw/openclaw#67060. Add warning-only logic in or near `logGatewayStartup` that detects configured proxy env (`HTTP_PROXY`/`HTTPS_PROXY` and lowercase variants as appropriate), inspects `cfg.models.providers` for remote HTTP(S) provider targets, skips local/loopback/private/LAN/tailnet/`.local`/NO_PROXY-bypassed targets and providers with any explicit `request.proxy`, and warns once with an actionable remediation such as `models.providers.<provider>.request.proxy.mode = "env-proxy"`. Reuse existing proxy-env and network classification helpers where they are lightweight; do not import broad plugin/runtime registries into gateway startup. Do not auto-enable env proxy or otherwise change provider routing semantics. Use #68023 only as prior design context because it is closed unmerged. Add focused tests in `src/gateway/server-startup-log.test.ts` for: warning when proxy env plus remote unconfigured provider exists; no warning with no proxy env; no warning when all remote providers have request proxy configured; no warning for local/non-remote providers. Add a user-facing active-version changelog entry if maintainers expect startup warning changes to be called out, with `Thanks @wnlen` and, if materially reusing the prior PR design, `Thanks @skylee-01`.
