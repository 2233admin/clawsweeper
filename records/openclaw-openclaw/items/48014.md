## Summary

Keep this PR open. Current main has shipped Microsoft Teams passwordless support for certificate and managed identity under `authType: "federated"`, which likely resolved the linked enterprise-auth issue #40855, but it does not implement this PR’s specific `defaultAzureCredential` auth type, DefaultAzureCredential token provider, local `az login`/credential-chain behavior, or the setup reset behavior described in the PR body. The PR is also stale against the current Teams SDK migration and needs maintainer review/rebase rather than cleanup closure.

## Best Possible Solution

Keep the PR open as a possible implementation candidate, but ask for a rebase onto the current Teams SDK architecture. The best version would either add a clearly named DAC mode to the current `secret`/`federated` config model, or explain why current managed-identity support supersedes it, while resolving the separate identity-client-id concern and preserving the setup reset fix if still applicable.

## Risks / Open Questions

- The PR targets the older `@microsoft/agents-hosting`/`MsalTokenProvider` shape, while current main has migrated the Teams plugin to `@microsoft/teams.apps` `App` plus `createMSTeamsTokenProvider`; a direct merge would need a substantial rebase.
- Security/supply-chain pass: the provided diff’s new dependency on `@azure/identity` is purpose-aligned and current main already ships that dependency pinned in the msteams plugin, with no visible CI/workflow/secret/lifecycle-hook changes. The main remaining security-sensitive review point is credential selection: avoid broad fallback or wrong-principal auth when bot app IDs and managed identity client IDs differ.
- The setup-wizard reset fix in the PR may still be relevant: current `setup-core` writes appId/appPassword/tenantId over existing msteams config without clearing an existing federated auth mode.
