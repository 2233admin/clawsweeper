## Summary

Keep this PR open. Current main implements Microsoft Teams passwordless auth through the existing `authType: "federated"` certificate and managed-identity paths, but it still does not implement this PR's explicit `defaultAzureCredential` auth type, DAC credential-chain behavior, or local Azure CLI credential flow. The PR is also security-sensitive and still needs maintainer review/rebase against the current Teams SDK architecture.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto the current `@microsoft/teams.apps` implementation, decide whether OpenClaw wants a separate `defaultAzureCredential` mode in addition to `federated` managed identity, design identity-client-id selection explicitly, document supported Azure/local credential behavior, and carry forward the setup-reset fix if it still reproduces on current setup-core.

## Risks / Open Questions

- The PR touches authentication, token acquisition, and the dependency lockfile. Those are security-sensitive surfaces, but the dependency addition is consistent with the stated DAC feature rather than an unrelated supply-chain change.
- The unresolved review concern about pinning DAC managed/workload identity client IDs to the bot app ID remains material: bot app registrations and user-assigned managed identities can be distinct, so the rebased implementation needs an explicit identity-selection model.
- DAC fallback behavior is a product/security decision for Teams: local `az login` convenience and multi-source fallback can be useful, but production webhook auth should avoid silently authenticating as an unintended principal.
- The PR's setup reset fix likely still needs to be carried forward, but adapted to current main's `secret`/`federated`, `certificatePath`, `useManagedIdentity`, and `managedIdentityClientId` names.
