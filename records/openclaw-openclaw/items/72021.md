## Summary

Keep open. Current main still uses aggregate `signalCount` for the promotion gate, v2026.4.27 has the same behavior, and #74114 is an open unmerged PR with closing syntax for this issue, so the right next step is maintainer review of that PR rather than cleanup closure.

## What This Changes

The issue asks short-term memory promotion to stop treating daily/session/grounded signals as real user recalls for `minRecallCount` and to revisit scoring so genuinely recalled entries remain promotable.

## Best Possible Solution

Define the product meaning of `minRecallCount` for deep promotion. If it means real user recalls only, land the active PR with matching promotion tests and docs plus a separate decision on score calibration; if aggregate evidence is intentional, document that semantics and track any ranking/noise tuning separately.

## Reproduction Assessment

Yes for the mixed-signal gate: current source and tests show a `recallCount: 0`, `dailyCount: 3` candidate can clear the gate because `minRecallCount` is checked against aggregate `signalCount`. The reporter's exact score-calibration numbers would need their short-term recall store or a new fixture to reproduce precisely.

## Solution Assessment

Unclear. #74114 is the narrow active fix for the gate, but it changes currently tested daily/grounded semantics and preserves aggregate score denominators, so maintainers should decide the intended semantics and update tests/docs before merging.
