## Summary

Keep open. Current main still gates `minRecallCount` on aggregate `signalCount`, the latest release carries the same behavior, and #74114 is the open unmerged closing-syntax PR for the reported fix path.

## What This Changes

The issue asks OpenClaw to stop daily/session short-term signals from satisfying promotion recall gates as if they were real user search recalls.

## Best Possible Solution

Use #74114 as the active implementation thread: decide whether `minRecallCount` should mean real user recalls only, then land a recall-only gate with updated tests/docs and any needed score calibration, or document that aggregate signals intentionally satisfy the threshold.

## Reproduction Assessment

Yes for the mixed-signal gate: current source and tests show a `recallCount: 0`, `dailyCount: 3` candidate can clear the default promotion gates. The exact reporter score examples need their recall-store fixture to reproduce numerically.

## Solution Assessment

Unclear. The linked PR is the narrow active fix for the gate, but maintainers need to decide how that interacts with current daily-only and grounded-only promotion behavior before merging.
