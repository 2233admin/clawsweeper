## Summary

Keep open. Current main still lacks a durable transcript append helper on the public plugin runtime surface, and the reporter's linked `openclaw-plugin-cron-context` workaround is good evidence that the existing in-memory queue does not cover the use case. The request is valid-looking, but it needs maintainer API and security design before implementation.

## What This Changes

Request to expose a Plugin Runtime API for durable cross-session transcript appends, primarily so cron output can be persisted into target channel session history.

## Best Possible Solution

Design an additive, narrow plugin runtime transcript-write API that wraps the existing session manager path, keeps assistant-only delivery-mirror semantics by default, requires explicit trusted-plugin capability or config opt-in, preserves idempotency and transcript update events, and adds SDK docs plus contract/runtime tests. Separately decide whether built-in isolated cron announce delivery should mirror into target channel sessions without requiring a plugin.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks `api.runtime.agent.session` transcript append support while the internal durable helper and ephemeral `enqueueSystemEvent` fallback are both present.

## Solution Assessment

Unclear. A persistent transcript helper is plausible, but exposing the internal helper directly with arbitrary session keys and user-role writes is not yet proven to be the narrowest secure API.

## Security Review

Status: needs_attention

Summary: The requested API is security-sensitive because it would let trusted plugin code persist content into other sessions' transcript history.

Concerns:

- **[medium] Gate durable transcript writes by plugin capability:** `src/plugins/runtime/types-core.ts:142`
  - body: Adding this under `api.runtime.agent.session` would let plugin code write durable content into any session key. That is stronger than the current in-memory system-event queue and should require explicit trusted-plugin permission plus audit semantics.
  - confidence: 0.88
- **[medium] Keep transcript writes assistant-only and sanitized:** `src/config/sessions/transcript.ts:224`
  - body: The internal exact append helper rejects non-assistant messages, but the requested `role?: "assistant" | "user"` shape would allow plugins to forge user turns unless maintainers define role limits, attribution, and transcript-shape validation first.
  - confidence: 0.84

## Risks / Open Questions

- The proposed role-flexible shape is broader than the existing assistant-only delivery-mirror helper and could enable forged user turns if adopted directly.
- Adjacent open transcript-redaction work (#73563) suggests the final API should align with transcript sanitization before implementation.
