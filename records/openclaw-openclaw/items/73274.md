## Summary

Keep open. Current main still lacks a durable transcript-append helper on the public plugin runtime surface, and the reporter's linked `openclaw-plugin-cron-context` workaround is useful evidence that `enqueueSystemEvent` is not enough for cross-session persistence. This is a plausible SDK gap, but the requested write surface is security-sensitive and needs maintainer API design before implementation.

## What This Changes

The issue requests an additive public Plugin Runtime API that lets plugins persist assistant transcript messages into another existing session by `sessionKey`.

## Best Possible Solution

Design a narrow, additive plugin runtime transcript-write API that wraps the host-owned session manager path, defaults to assistant-only delivery-mirror semantics, requires explicit trusted-plugin capability or operator opt-in, preserves idempotency and transcript update events, and uses the same redaction/guard hooks as other transcript writes. Maintainers should also decide whether isolated cron `announce` delivery should mirror to target channel sessions directly, without requiring a plugin workaround.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence static check is that current main lacks `api.runtime.agent.session` transcript append support while the internal durable helper, ephemeral system event path, and durable-but-once next-turn injection path are all present.

## Solution Assessment

Unclear. A persistent transcript helper is plausible, but directly exposing the internal helper with arbitrary session keys or user-role writes is not the narrowest secure solution; an assistant-only, opt-in mirror helper is the safer starting point.
