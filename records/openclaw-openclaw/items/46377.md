## Summary

Keep this PR open. Current main still has the reported native PDF provider security gap, but the submitted patch should not merge unchanged because it forces strict guarded fetch semantics that bypass the shared provider proxy path and is stale against the current 120s native PDF timeout behavior.

## What This Changes

The PR updates native Anthropic and Gemini PDF provider calls in `src/agents/tools/pdf-native-providers.ts` to use guarded fetches, move Gemini auth to `x-goog-api-key`, and release guarded-fetch resources.

## Best Possible Solution

Revise the native Anthropic and Gemini PDF calls to use the shared provider HTTP path, or an equivalent helper, while preserving timeout, provider transport normalization, proxy selection, private-network policy, audit context, cleanup, and Gemini header-based auth; add focused regression tests for SSRF blocking, proxy-mode selection, timeout preservation, and absence of the Gemini API key in the URL.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve provider proxy routing in guarded PDF fetches:** `src/agents/tools/pdf-native-providers.ts:69`
  - body: The new `withStrictGuardedFetchMode` path forces strict pinned-dispatch behavior instead of the shared provider HTTP helper. In HTTP(S)_PROXY-only deployments, native Anthropic/Gemini PDF calls can fail or bypass operator proxy routing while other provider calls still work through the proxy, so this needs to preserve the provider helper's proxy-selection semantics.
  - confidence: 0.9
- **[P2] Keep the native PDF timeout on guarded requests:** `src/agents/tools/pdf-native-providers.ts:69`
  - body: Current main gives these native provider calls a 120s timeout and tests assert an `AbortSignal`. The guarded fetch call added here passes neither `timeoutMs` nor `signal`, so rebasing this patch as written can reintroduce indefinite native PDF hangs that current main already fixed.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff addresses a real SSRF/key-leak issue, but it changes security-sensitive provider egress in a way that can bypass operator proxy controls.

Concerns:

- **[medium] Strict guarded fetch bypasses provider proxy routing:** `src/agents/tools/pdf-native-providers.ts:69`
  - body: Using `withStrictGuardedFetchMode` directly bypasses the shared provider HTTP helper that conditionally uses trusted env-proxy mode. That can route these provider requests outside operator-managed HTTP(S)_PROXY egress controls, weakening the deployment's network security boundary.
  - confidence: 0.88
