## Summary

Keep this PR open. Current main still has the core reported behavior: native Anthropic and Gemini PDF provider calls use raw `fetch()` against configurable provider URLs, and Gemini still places the API key in the request URL. The PR is still relevant, but it needs a rebase/revision against the newer timeout and provider-transport code before it is a safe landing candidate.

## Best Possible Solution

Keep this PR open and revise it before merge: rebase onto current main, route native Anthropic/Gemini PDF provider requests through the guarded provider HTTP path while preserving the 120s timeout and proxy behavior, move Gemini auth out of the URL, add provider-specific audit contexts and cleanup, and add focused SSRF/header regression tests.

## Risks / Open Questions

- Current main still allows configured native PDF provider base URLs to be fetched without the application-level SSRF guard.
- Current main still sends the Gemini API key in the request URL for native PDF analysis.
- The PR branch no longer applies cleanly to current main and would need a rebase around the timeout and Google transport-normalization changes.
- The direct strict-guard approach can regress proxy-only/provider-proxy deployments unless revised to preserve the current provider HTTP helper semantics.
- Focused regression tests are still missing for private/internal base URL blocking, Gemini header auth/no query key, release cleanup, and provider-specific audit context.
