## Summary

Keep this PR open. Current main still does not implement the intended metadata/user-message separator: prompt construction still joins inbound metadata and user text with blank lines, and the sanitizer/test-helper surfaces have no support for the proposed `---openclaw:user-msg---` marker. The PR remains a focused bug-fix candidate, but it needs maintainer review and updates against current prompt composition and sanitizer coverage before merge.

## Best Possible Solution

Keep this PR open as a focused bug-fix candidate. The best path is to rebase it onto current main, adapt separator insertion to the current `runPreparedReply` and media-only body flow, align prompt-composition helpers/tests, make separator stripping tightly scoped to the injected marker next to stripped metadata, cover all generated inbound metadata block types, and run the changed OpenClaw gates before merge.

## Risks / Open Questions

- Closing now would drop a still-relevant focused bug-fix candidate; current main does not implement the explicit separator behavior the PR proposes.
- Merging without further review could leak the internal separator into web/API/TUI/session surfaces if sanitizer coverage misses any visible path.
- Separator removal is a user-content integrity boundary; it should remove only the synthetic marker adjacent to stripped OpenClaw metadata and preserve legitimate user text.
