## Summary

Keep open. Current main still leaves the reported behavior partially unsolved: Discord direct raw runner failures retain generic failure copy, diagnostics observe active stuck sessions by default instead of marking or rotating them, and overlapping recovery PRs need maintainer selection before this can be closed.

## What This Changes

The issue requests unambiguous Discord session health/routing and user-facing timeout or partial-success state after an embedded agent turn times out after producing side effects.

## Best Possible Solution

Add a shared terminal timeout/session-health outcome in the core agent/session runner that can mark, rotate, or otherwise make the affected session unroutable before later dispatch, and surface channel-appropriate timeout copy warning that prior side effects may have completed. Keep Discord regression coverage, but avoid a Discord-only workaround for a shared runner lifecycle problem.

## Solution Assessment
