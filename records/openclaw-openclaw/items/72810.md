## Summary

Keep open. Current main still has the same core gap: Discord direct pre-reply runner failures can surface generic failure copy, CLI overall timeouts become timeout failover errors without a session-health or partial-success notice, and stuck-session recovery remains observe-only for active embedded/reply/lane work by default. The related open #73581 covers broader stuck-lane recovery, but it does not fully replace this Discord user-facing timeout and routing ambiguity.

## What This Changes

The issue asks OpenClaw to make Discord-routed agent timeout outcomes unambiguous by marking or rotating wedged sessions and distinguishing partial-success timeouts from total failures.

## Best Possible Solution

Add a shared agent/session health outcome for terminal embedded-agent timeouts that can mark the affected session failed or rotate it before later routing, and surface timeout-specific copy warning that previous side effects may have completed. Coordinate this with the broader stuck-lane policy in #73581, with Discord direct and group/channel coverage, instead of adding a Discord-only workaround.

## Reproduction Assessment

Yes for a source-level reproduction: current tests show Discord direct raw runner failures still get generic copy, and stuck-session recovery tests show active embedded runs remain observe-only by default. A live Discord/Claude CLI 900s run would still be useful before landing recovery semantics that abort or rotate active sessions.

## Solution Assessment

Unclear until maintainers choose the product policy for active timeout recovery. The narrowest maintainable solution is a shared timeout/session-health transition plus channel-appropriate user copy, not unconditional active-run abort and not a Discord-only routing patch.
