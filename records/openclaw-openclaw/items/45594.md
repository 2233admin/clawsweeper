## Summary

Keep this PR open. Current main still advertises `cron.retry.retryOn: ["server_error"]`, but it does not model `server_error` as a `FailoverReason`, cannot expose it through cron state/protocol, and the retry gate still only matches raw 5xx text. The PR remains a focused implementation candidate, with the existing review discussion pointing to a concrete 4xx-precedence concern to preserve during review/rebase rather than a reason to close it.

## Best Possible Solution

Keep this PR open. Rebase it onto current main, add `server_error` as a first-class failover and cron protocol reason, map it to fallback status 500, make cron retry detection consult the failover classifier before raw regex fallback, keep transport/server errors out of auth-profile health poisoning, and preserve 4xx precedence with focused failover, cron protocol, and one-shot cron retry regression coverage.
