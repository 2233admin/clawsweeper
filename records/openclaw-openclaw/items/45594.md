## Summary

Keep this PR open. Current main still does not implement the PR's intended behavior: cron config and docs allow `retryOn: ["server_error"]`, but `server_error` is not a `FailoverReason`, cannot be exposed as `lastErrorReason`, and the one-shot cron retry gate only regex-matches raw 5xx text. The provided PR discussion also shows a real review concern around preserving 4xx precedence; that is a reason to review/rebase carefully, not to close the PR as obsolete.

## Best Possible Solution

Keep this PR open. Rebase it onto current main's `classifyFailoverSignal`/`failover-matches` structure, add `server_error` as a first-class failover and cron protocol reason, map it to a 500 fallback status, keep it out of auth-profile health poisoning like transport failures, and make cron retry detection consult the classifier before raw regex fallback. Preserve 4xx precedence and add focused failover, cron protocol, and one-shot cron retry regression coverage.
