## Summary

Keep open. Current main has same-provider auth-profile rotation once distinct profiles exist, but it still lacks a supported way for `models auth login` to create separate OpenAI Codex OAuth profiles for the same email/team subscription. The earlier alias PR was closed unmerged, and the remaining profile-targeting work is still open through #40402/#49315 rather than shipped.

## What This Changes

The issue requests support for multiple OpenAI Codex OAuth auth profiles under the same provider/account, with team/org or alias separation and same-provider rotation on rate limits.

## Best Possible Solution

Resolve this through the existing named-login-profile track: add a provider-safe `--profile-id` or equivalent alias/discriminator to `models auth login`, thread it through typed provider auth context before persistence, make OpenAI Codex OAuth/device-code honor it or derive team/account ids ahead of plain email, and add regression coverage for two same-email Codex OAuth profiles plus rate-limit rotation across them.

## Reproduction Assessment

Yes for the supported-surface gap: current main can be checked by inspecting `models auth login` options and the Codex identity helper, which show same-email OAuth results still map to the same profile id. A live two-team OAuth account was not required to prove the missing CLI/profile discriminator.

## Solution Assessment

Yes, the narrow maintainable direction is the profile-targeting work already tracked by #40402/#49315, with provider-owned persistence semantics fixed before landing. A broad fallback rewrite is not needed because same-provider rotation already exists once multiple distinct profiles are present.
