## Summary

This PR should stay open for maintainer and author revision, not close: current main still does not implement a Feishu outbound/subagent streaming bridge, but the proposed patch is not safe to merge as-is because it introduces duplicate delivered-run streaming/final delivery, drops Feishu thread identifiers, changes block-streaming fallback behavior, and adds unthrottled card replacement updates.

## What This Changes

The PR adds a `ChannelPlugin.createOutboundDispatcher` hook, wires Feishu delivered and subagent agent runs into reply-dispatcher streaming callbacks, adds Feishu `blockStreaming` config, and changes streaming-card snapshot replacement behavior.

## Best Possible Solution

Revise this into a narrow, SDK-safe outbound streaming design: scope live streaming to native subagent runs that lack an inbound dispatcher, preserve Feishu reply/thread transport identifiers, avoid duplicate final delivery, keep block-chunk behavior backward-compatible, coalesce or throttle replacement updates, and add focused Feishu plus agent-command regression coverage and public SDK/docs updates if the hook remains plugin-facing.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Limit outbound streaming to subagent runs:** `src/commands/agent.ts:738-771`
  - body: The bridge is created for `opts.deliver === true`, but delivered main-agent runs still finish through normal outbound delivery. On Feishu that can emit a streamed card and then send the final payload again for the same turn, so the bridge should be limited to subagent runs or final delivery must be explicitly coordinated.
  - confidence: 0.9
- **[P1] Pass the Feishu thread id through:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher reduces `threadId` to a boolean and drops the actual thread/root id. In thread-only contexts without `replyToId`, `FeishuStreamingSession.start` has no concrete topic target and can post streamed subagent output as a top-level group message.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:313-314`
  - body: `account.config?.blockStreaming !== false` enables block chunks when the config is unset, while current Feishu behavior and tests suppress internal block payloads unless card streaming can safely consume them. This can send fragmented internal block replies to default accounts.
  - confidence: 0.86
- **[P2] Throttle replaceContent updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent()` bypasses the existing 100 ms update throttle, but the new model/tool/reasoning paths call it repeatedly, including a per-character model-name animation. That can burst Feishu CardKit updates and trigger rate-limit or sequence failures.
  - confidence: 0.82
- **[P2] Use a public SDK type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: The Feishu plugin adds a relative import from `../../../src/auto-reply/types.js`, which crosses from extension production code into core internals. Plugin-facing channel work should use an exported SDK/runtime facade so bundled and third-party plugin contracts do not depend on private source paths.
  - confidence: 0.78
