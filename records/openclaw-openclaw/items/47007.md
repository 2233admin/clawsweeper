## Summary

Keep this PR open. Current main still does not implement the proposed Feishu outbound/subagent streaming bridge: Feishu has normal reply-dispatcher streaming cards, but native subagent runs are still documented and wired as post-run completion/announce delivery. The PR is plausible product work, but the existing review comments identify concrete regressions that need a revised patch before merge.

## Best Possible Solution

Keep the PR open for maintainer review and request a revised implementation: add a narrow SDK-safe outbound/subagent streaming seam only if core agrees it is needed, scope it to subagent-lane runs, preserve Feishu reply/thread transport identifiers, avoid duplicate final delivery, throttle card updates, and add focused Feishu plus agent-command regression tests for subagent streaming and main-agent non-regression.

## Risks / Open Questions

- Merging the PR as provided could duplicate streamed/final Feishu replies if the bridge is created for normal delivered agent runs as well as subagent runs.
- The proposed Feishu outbound dispatcher drops the actual thread id in the reviewed diff, which could stream outside the intended Feishu topic/thread.
- The proposed replace-style card updates need throttling or coalescing to avoid Feishu CardKit rate or sequence failures.
- The proposed hook is a new plugin-facing channel seam, so it needs SDK/API/docs/tests rather than a Feishu-only private addition.
