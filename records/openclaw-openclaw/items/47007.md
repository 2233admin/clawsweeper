## Summary

Keep this PR open. Current main still does not implement the proposed Feishu outbound/subagent character-by-character streaming bridge. Main has Feishu inbound reply streaming and newer Feishu subagent hooks for thread/completion routing, but native subagent runs still flow through subagent lifecycle/announce or post-run outbound delivery rather than a channel streaming dispatcher. The PR also has unresolved review findings that need a revised patch before it is safe to merge.

## Best Possible Solution

Keep the PR open for maintainer review and request a revised implementation: add an SDK-safe outbound/subagent streaming seam only if core accepts that direction, scope it to subagent-lane runs, preserve Feishu reply/thread identifiers, avoid duplicate final delivery, throttle/coalesce card updates, respect extension boundary rules, and add focused Feishu plus agent-command regression tests.

## Risks / Open Questions

- Merging the submitted PR as-is could duplicate Feishu streamed/final replies because the bridge is created for normal delivered agent runs while deliverAgentCommandResult still performs final outbound delivery.
- The PR's Feishu outbound dispatcher reduces threadId to a boolean replyInThread flag in the provided diff, risking streams outside the intended Feishu topic/thread.
- The proposed replace-style card updates and model-name animation could bypass the current Feishu streaming-card throttle and hit CardKit rate or sequence limits.
- The PR adds a new plugin-facing ChannelPlugin seam without the SDK/docs/API contract follow-through expected for third-party plugin compatibility; the provided diff also deep-imports a core auto-reply type from Feishu extension code.
