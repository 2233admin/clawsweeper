## Summary

Keep this PR open for maintainer review: current main does not implement the PR's intended native Feishu subagent character streaming, and the proposed patch still has blocking delivery, routing, plugin-boundary, and Feishu streaming-card regressions that match the prior review discussion.

## What This Changes

The PR adds a `ChannelPlugin.createOutboundDispatcher`/`OutboundDispatcher` seam, wires Feishu outbound agent-command output into `createFeishuReplyDispatcher`, changes Feishu streaming-card replacement/status behavior, and adds Feishu `blockStreaming` config handling.

## Best Possible Solution

Revise this as a narrow, SDK-safe native subagent outbound streaming path: scope it to subagent runs that actually lack an inbound dispatcher, preserve `replyToId`/`threadId`/`rootId` routing, coordinate with final delivery to prevent duplicates, keep block replies opt-in, coalesce replacement-card updates, expose any new public contract through the plugin SDK/docs, and cover Feishu plus agent-command regressions.

## Reproduction Assessment

Yes. The blocking issues are reproducible by code inspection of the PR diff against current main: delivered runs satisfy the new bridge guard while final delivery still runs, and Feishu thread routing is reduced to a boolean before streaming starts.

## Solution Assessment

No. The proposed approach is not the best solution as-is because it broadens a public channel contract and reuses Feishu reply streaming without preserving routing and delivery semantics; a subagent-only bridge in the current refactored command/delivery path is safer.

## Review Findings

Overall confidence: 0.9

Full review comments:

- **[P1] Limit the bridge to subagent runs:** `src/commands/agent.ts:738`
  - body: This guard also creates the streaming bridge for ordinary `--deliver` main-agent turns. Those turns still go through final outbound delivery, so Feishu can show a streamed card and then receive the final reply again for the same run.
  - confidence: 0.9
- **[P1] Preserve Feishu thread routing targets:** `extensions/feishu/src/channel.ts:496`
  - body: The dispatcher turns `threadId` into a boolean and drops the actual topic/root id. In thread-only contexts without `replyToId`, the streaming card has no concrete thread target and can be posted at the top level instead of the originating topic.
  - confidence: 0.88
- **[P2] Keep block streaming opt-in:** `extensions/feishu/src/reply-dispatcher.ts:432-433`
  - body: `account.config?.blockStreaming !== false` enables block delivery when the setting is unset. That changes the existing Feishu behavior that drops internal block chunks unless the streaming-card path can safely consume them, so default accounts can receive fragmented internal chunks.
  - confidence: 0.86
- **[P2] Throttle replacement card updates:** `extensions/feishu/src/streaming-card.ts:361-374`
  - body: `replaceContent` bypasses the existing card update throttle while the new model/status/reasoning paths can call it repeatedly, including per-character model-name animation. This can burst Feishu CardKit updates and trigger rate or sequence failures.
  - confidence: 0.82
- **[P2] Use an SDK type for model selection:** `extensions/feishu/src/reply-dispatcher.ts:10`
  - body: Feishu production code imports `ModelSelectedContext` through a relative `../../../src/**` path, crossing the bundled plugin boundary. This type needs to come from an SDK or local facade before the plugin-facing contract expands.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds no dependency or CI execution surface, but it introduces a concrete Feishu routing/privacy risk by losing thread targets for streamed output.

Concerns:

- **[medium] Preserve Feishu thread targets:** `extensions/feishu/src/channel.ts:496`
  - body: The outbound dispatcher converts `threadId` to `Boolean(params.threadId)` and does not pass the actual Feishu topic/root id to the streaming session. In topic or thread-only contexts, subagent output can be posted outside the intended thread and exposed to the broader chat.
