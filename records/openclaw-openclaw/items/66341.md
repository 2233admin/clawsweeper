## Summary

Keep this PR open. Current main still requires `operator.admin` for cron mutations and does not persist or enforce a creator-scope ceiling, so the PR is not obsolete or implemented. The branch remains security-sensitive and should be reviewed as a focused cron authorization change, with unrelated browser/QA-lab changes split out.

## Best Possible Solution

Keep this PR open for maintainer review, but ask for a focused cron-only branch. The best path is to split unrelated browser/QA-lab changes, lower only the intended cron surface or add explicit ownership/caller-scope subset guards for update/remove/run, persist creator scopes at add time, enforce the ceiling in both embedded and CLI/MCP tool resolution, and add regression coverage proving write-scoped callers cannot execute or modify admin-created cron jobs to gain admin tools.
