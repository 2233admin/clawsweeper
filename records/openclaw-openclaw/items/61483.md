## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the Android `http.request` node command, and the closed feature issue #65440 points here as the canonical implementation, but the patch introduces a remotely invokable Android HTTP proxy without enough destination policy and has a default-off toggle path that does not re-advertise the command until reconnect.

## What This Changes

The PR adds an Android-only `http.request` node invoke command, an Android Settings opt-in, gateway Android allowlist support, and Android/gateway tests for forwarding HTTP requests through a connected Android node.

## Best Possible Solution

Keep this canonical PR open, but land only after the mobile HTTP proxy has a reviewed destination policy, appropriate gateway risk/approval semantics, a working capability refresh or reconnect flow for the Settings toggle, Android platform docs, changelog credit, and targeted Android/gateway tests.

## Reproduction Assessment

Yes. The PR diff directly shows `validateUrl` accepting any `http` or `https` URL before `openConnection()`, and the toggle setter only writes prefs while the node's declared commands are built at connect time, so both reported review paths are reproducible by source inspection.

## Solution Assessment

No. The feature direction is coherent, but the submitted implementation is not the best merge path until it reuses or mirrors OpenClaw's SSRF/private-destination policy and fixes the default-off toggle's current-session behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Block private HTTP destinations before proxying:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:45`
  - body: The new handler accepts any `http`/`https` URL once the scheme matches, then opens it from the phone. With `http.request` allowlisted for Android, an agent can ask the node to fetch loopback, LAN, link-local, or metadata-style hosts reachable from the device. Add DNS/IP destination checks, or an explicitly reviewed allow policy, before opening the connection.
  - confidence: 0.91
- **[P2] Reconnect after enabling HTTP access:** `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:798-800`
  - body: The setter only writes the preference, but the gateway stores the node's declared command list from `buildNodeConnectOptions()` when the node connects. Since HTTP access defaults off, toggling it on in Settings will not add `http.request` to the current node session, so gateway `node.invoke` still rejects it as not declared until the user manually reconnects.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a remotely invokable Android HTTP proxy without concrete destination-level SSRF/private-network controls, so security review is not cleared.

Concerns:

- **[high] Missing SSRF destination filtering:** `apps/android/app/src/main/java/ai/openclaw/app/node/HttpHandler.kt:45`
  - body: `validateUrl` only checks the scheme before the Android device opens the connection. That allows gateway-driven requests to loopback, private LAN, link-local, metadata, multicast, or other special-use destinations reachable from the phone unless a reviewed denylist or allow policy is added.
  - confidence: 0.92
- **[medium] HTTP proxy command is treated as a normal Android default:** `src/gateway/node-command-policy.ts:100`
  - body: The gateway diff adds `http.request` to the Android platform default allowlist. Even with the Android-side opt-in, this makes arbitrary URL fetches a normal declared node command rather than a high-risk command with explicit policy or approval handling.
  - confidence: 0.78
