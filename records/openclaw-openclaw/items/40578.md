## Summary

Keep open: current main still does not provide per-agent isolation for `memory-lancedb`, and this draft branch is not merge-ready because it is stale against the current prompt-build recall path and has a scoped memory isolation bug around `agentId` handling.

## What This Changes

The PR adds an opt-in `agentScoping` setting to `memory-lancedb`, adds an `agentId` LanceDB column/migration, and changes memory tools plus lifecycle hooks to store, recall, and forget memories by the current agent context.

## Best Possible Solution

Land a rebased implementation that scopes the current `before_prompt_build` auto-recall path and all tool/auto-capture paths, uses one validated or encoded owner key consistently for store/search/delete, preserves legacy shared rows deliberately, logs migration failures, updates docs/config metadata, and adds A/B isolation regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Validate agent IDs before scoped memory access:** `extensions/memory-lancedb/index.ts:428-455`
  - body: When `agentScoping` is enabled, the patch stores `ctx.agentId` raw but strips disallowed characters before search/delete filters. IDs outside the allowed set can store rows that are never recalled, or collapse into another agent's allowed ID. Validate/reject or encode the ID once and use the same value for writes, reads, and deletes.
  - confidence: 0.92
- **[P2] Port scoping to the prompt-build recall hook:** `extensions/memory-lancedb/index.ts:591-601`
  - body: Current main moved LanceDB auto-recall to `before_prompt_build`, and tests assert the legacy `before_agent_start` hook is not registered. Rebase this patch onto the current hook path, or the automatic recall path will remain unscoped or the PR will reintroduce stale hook behavior.
  - confidence: 0.9
- **[P3] Unwrap factory tools in live coverage:** `extensions/memory-lancedb/index.ts:354`
  - body: The patch changes memory tools to factories, but PR-head live coverage still treats `registeredTools[i].tool` as an object with `.execute`. Invoke the factory with a tool context before calling `.execute`, or the live store/recall/forget smoke will fail immediately.
  - confidence: 0.86
- **[P3] Log agentId migration failures:** `extensions/memory-lancedb/index.ts:119-122`
  - body: The migration catch block swallows schema/addColumns failures even though scoped reads depend on the new column. Surface a warning through plugin logging so operators can diagnose why enabled scoping is not isolating existing tables.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention because the diff changes a memory data-isolation boundary and currently lets raw and filtered agent IDs diverge across scoped writes, reads, and deletes.

Concerns:

- **[high] Agent ID normalization can cross memory scopes:** `extensions/memory-lancedb/index.ts:428`
  - body: Scoped writes store raw `ctx.agentId`, while scoped reads/deletes filter using a stripped value. Malformed or unexpected IDs can become unrecoverable or collide with another agent's allowed ID, undermining the isolation boundary this PR is meant to add.
