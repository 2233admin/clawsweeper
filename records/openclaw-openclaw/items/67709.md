## Summary

Keep this issue open. Current main has a good partial fix: `chat.history` no longer synchronously loads the model catalog to compute the missing thinking default, and there is a regression test for that. The remaining startup-shape concern is still real enough to track because Control UI chat refresh still starts `models.list` on initial chat load, `models.list` still awaits the full gateway model catalog, and the browser RPC client still has no per-request timeout for a slow in-flight request.

## What This Changes

The issue requests that WebChat/Control UI startup remain responsive instead of waiting minutes on cold `chat.history` and `models.list` RPCs.

## Best Possible Solution

Finish the cold-start responsiveness fix by making the initial chat surface independent from full model catalog discovery. `chat.history` should stay catalog-free; the chat model picker can load in the background, use a cheap configured/default view, or fail fast with a retryable placeholder while preserving existing configured-model visibility once catalog discovery completes.

## Repair Work Prompt

Fix the remaining cold-start responsiveness path for issue https://github.com/openclaw/openclaw/issues/67709. Current main already has `chat.history` avoiding `loadGatewayModelCatalog()`; keep that behavior. Focus on the remaining `models.list`/Control UI chat startup path: the initial chat view should render history and allow sending without waiting for full model catalog discovery. Do not change provider/plugin ownership boundaries or weaken configured/default model visibility semantics. Likely approaches include backgrounding `refreshChatModels`, adding a bounded/placeholder model-picker load for startup, or adding a cheaper configured-view path in `models.list` that does not require full provider catalog discovery when config is sufficient. Add regression coverage with a deferred or never-resolving model catalog/model list request proving chat history/load/send readiness is not blocked while model loading remains pending. Changelog only if the user-facing startup behavior changes enough to mention; credit the issue reporter if adding an entry.
