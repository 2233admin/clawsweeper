## Summary

Keep this PR open. Current main only gives Feishu `/stop` and `/btw` out-of-band lanes; it does not implement this PR’s broader command-queue bypass for `/new`, `/reset`, `/status`, `/help`, `/model`, and other command-only Feishu text messages. The PR is stale against the current handler split and needs maintainer review, but it is still a plausible implementation candidate for the linked Feishu regression rather than cleanup material.

## Best Possible Solution

Keep this PR open and review it as a candidate fix for the Feishu command fast-path regression tracked by #42803. The best path is to rebase or rewrite the idea onto `createFeishuMessageReceiveHandler` and `getFeishuSequentialKey`, decide which command classes should bypass the Feishu per-chat lane, preserve authorization before destructive side effects where feasible, and add regression tests for active-run immediacy plus ordering/drop behavior for `/stop`, `/new`, `/reset`, `/status`, and read-only commands.

## Risks / Open Questions

- The PR targets the older `monitor.account.ts` receive-handler shape, while current main routes Feishu message handling through `monitor.message-handler.ts` plus `sequential-key.ts`, so it likely needs a rebase or rewrite before maintainer review can be meaningful.
- Routing all detected text commands around the same-chat lane can reorder command effects relative to buffered or already queued user text; read-only commands and destructive lifecycle commands probably need different semantics.
- The proposed destructive cleanup path is security-sensitive because unauthorized `/stop`, `/new`, or `/reset` attempts could discard pending buffered text before command authorization unless the implementation gates or defers that side effect.
- Adding `dropKey` changes the shared inbound debouncer surface exported through the plugin SDK/runtime, so it needs focused contract coverage beyond the Feishu-specific tests.
