## Summary

Keep this open. Current main has adjacent fixes for Telegram media delivery once a media URL exists, but the ordinary assistant reply path still does not fully implement the reported behavior: local markdown image targets are explicitly not promoted, and the main agent reply/block-reply paths do not propagate Telegram's markdown-image extraction opt-in into final/block payload normalization.

## What This Changes

Ensure Telegram ordinary assistant replies preserve and deliver image attachments from text plus local or generated markdown-image media instead of delivering only the text.

## Best Possible Solution

Thread the channel markdown-image opt-in through ordinary assistant final and block reply normalization, then promote only safe local/generated markdown image destinations through the existing agent-scoped media access checks. Add parser, agent-runner, outbound payload, and Telegram dispatch/delivery regression coverage for text plus local image delivery without double-sending.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current parser tests explicitly assert local markdown image targets produce no mediaUrls, and the ordinary runReplyAgent/buildReplyPayloads call path does not pass the Telegram extraction flag into final payload normalization.

## Solution Assessment

No, current main is not the best solution yet. The maintainable fix should reuse the existing channel opt-in and local media access boundary rather than lifting arbitrary local markdown paths globally.
