## Summary

Keep open. Current main fixes adjacent Telegram media-delivery paths, but it still leaves the central local/generated markdown-image promotion gap: Telegram opts into markdown-image extraction while the shared parser only lifts remote HTTPS markdown image destinations, and current tests explicitly keep local markdown image targets as text with no mediaUrls. The remaining fix is security-sensitive because it changes when assistant-authored local paths become outbound attachments.

## What This Changes

Report requests that ordinary Telegram assistant replies preserve image attachments, especially local/generated markdown image content, across markdown parsing, block-stream final payload suppression, and Telegram lane delivery.

## Best Possible Solution

Add a narrow reply-pipeline fix that promotes only safe local/generated markdown image destinations into media payloads for Telegram, runs those paths through the existing agent-scoped media normalizer, preserves the current block-stream and Telegram lane safeguards, and adds regression coverage for local markdown images, streamed text plus final media, and Telegram text-plus-media delivery.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: with extractMarkdownImages enabled, current parser tests still assert local markdown image targets remain text with mediaUrls undefined, which matches the reported text-only Telegram result before media delivery is reached.

## Solution Assessment

Unclear. The requested direction is correct at a high level, but the best fix should not broadly lift all local markdown paths in parse.ts; it should reuse the existing reply media access boundary and prove Telegram does not double-send or drop media after streaming.
