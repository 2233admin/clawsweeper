## Summary

Keep this issue open. Current main still does not promote local/generated markdown image destinations into reply media, which is central to the reported Telegram ordinary-assistant text-plus-image loss; the remaining fix needs maintainer review because it touches the local file media trust boundary.

## What This Changes

Ensure Telegram ordinary assistant replies that include text plus local/generated image markdown or media payloads deliver the image attachment instead of only the text.

## Best Possible Solution

Add guarded local/generated markdown-image support through the existing agent-scoped media normalization path, then cover ordinary Telegram final replies with text plus local image media across block-stream and delivery paths without changing unrelated channel behavior.

## Reproduction Assessment

Yes. A high-confidence parser-level reproduction is to call markdown image extraction on a local path such as `![x](/root/.openclaw/media/tool-image-generation/example.png)`; current tests and source show this produces no mediaUrls.

## Solution Assessment

The report's direction is broadly right, but lifting all local markdown destinations blindly is not the safest solution. The maintainable fix should reuse the existing workspace/sandbox/local-root media access checks and add focused Telegram regression coverage.
