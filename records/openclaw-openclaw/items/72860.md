## Summary

Keep open. Current main has fixed several adjacent Telegram media-loss paths, but the remaining reported local/generated Markdown-image path is still not implemented: Telegram opts into Markdown-image extraction, while the shared parser only promotes validated remote HTTP(S) Markdown images and the current tests still assert local Markdown-image targets remain text with no media URLs.

## What This Changes

The issue asks OpenClaw to preserve image attachments in ordinary Telegram assistant replies that contain text plus local/generated Markdown-image content, matching the behavior already seen for direct sends and routeReply media payloads.

## Best Possible Solution

Add a narrow reply-pipeline implementation that promotes only safe local/generated Markdown-image destinations into media payloads for Telegram, routes them through the existing reply media normalizer/access boundary, and keeps the current block-stream and lane-delivery safeguards covered by focused regression tests.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is to call the current parser/buildReplyPayloads path with extractMarkdownImages enabled and a local Markdown image target; existing tests assert mediaUrls stays undefined, matching the text-only Telegram outcome before media delivery is reached.

## Solution Assessment

Unclear. The requested direction is valid, but broadly lifting all local Markdown-image paths in parse.ts would not be the best fix; maintainers should first define the safe generated/local media boundary and then reuse the existing media normalization path.
