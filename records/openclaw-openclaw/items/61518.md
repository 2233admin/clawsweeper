## Summary

Keep this PR open for revision. Current main still does not implement the central `web_fetch` HTTP(S)_PROXY behavior, and the PR is useful, but it is not ready as-is because proxy selection is tied to the initial URL scheme and can miss scheme-changing redirects; the bundled cron silent-reply/forum-topic pieces are already present on main.

## What This Changes

The PR adds strict `web_fetch` env-proxy dispatcher selection and regression tests for HTTP(S)_PROXY, plus cron silent-reply/forum-topic delivery suppression docs and tests.

## Best Possible Solution

Land a narrow web-fetch-only fix that opts strict guarded web fetch into env-proxy when HTTP_PROXY or HTTPS_PROXY could be used on any redirect hop, preserves explicit dispatcherPolicy overrides, keeps SSRF DNS pinning/direct fallback, relies on EnvHttpProxyAgent for NO_PROXY handling, and adds protocol/redirect regression coverage.

## Reproduction Assessment

Yes: on current main, `web_fetch` reaches `fetchWithWebToolsNetworkGuard` without any env-proxy dispatcher policy, and the existing test still asserts HTTP_PROXY does not produce an `EnvHttpProxyAgent`. The PR-specific gap can be reproduced with an `http://` URL redirecting to `https://` while only HTTPS_PROXY is set.

## Solution Assessment

No: the requested direction is right, but this PR’s implementation is not the best final fix because it chooses env-proxy from only the initial URL protocol. A safer implementation should select env-proxy whenever either HTTP(S) proxy env could apply, or move that decision per-hop into the guarded fetch path, with redirect tests.
