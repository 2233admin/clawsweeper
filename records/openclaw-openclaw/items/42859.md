## Summary

Keep this PR open. Current main still injects gateway client identity into `MsgContext` sender fields and prompt assembly still renders those fields as `Sender (untrusted metadata)`, so main has not implemented the PR’s intended behavior. The PR is still directionally useful, but its blanket removal is too broad because `SenderId` is also used by command authorization and sender-bound session lifecycle ownership checks.

## Best Possible Solution

Request a narrower fix: preserve trusted gateway caller identity for command authorization, run attribution, delivery context, and sender-bound ownership checks, while preventing synthetic Control UI/internal gateway client metadata from being rendered into prompt-visible untrusted sender metadata. Add paired regression tests proving Control UI prompts omit the noisy sender block while CLI or other gateway callers still retain identity for command auth and binding ownership behavior.
