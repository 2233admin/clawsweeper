## Summary

Keep this PR open. Current main still injects Control UI gateway client identity into `MsgContext` sender fields, and prompt assembly still renders those fields as `Sender (untrusted metadata)`. The PR is not obsolete, but its current blanket removal is too broad because `SenderId` also feeds command authorization and sender-bound session/focus ownership checks.

## Best Possible Solution

Request a narrower fix: keep trusted gateway caller identity available for command authorization, run attribution, and sender-bound ownership checks, but prevent synthetic Control UI/internal gateway client metadata from being rendered into prompt-visible untrusted sender metadata. Add paired regression tests proving Control UI prompts omit the noisy sender block while CLI or other gateway callers still retain identity for command auth and binding ownership behavior.
