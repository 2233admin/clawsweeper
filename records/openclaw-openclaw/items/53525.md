## Repair Work Prompt

Fix the Qianfan provider error-surfacing bug reported at https://github.com/openclaw/openclaw/issues/53525. Current main routes Qianfan through `api: "openai-completions"`; when Baidu/Qianfan returns provider errors, the session can record `stopReason: "error"` with no `errorMessage` and no content. Implement a narrow, generic OpenAI-compatible provider error normalization path that supports Baidu-style payloads such as `{ error_code, error_msg }`, `{ error_message }`, and `{ msg }` without hard-coding policy outside the relevant shared/provider seam. Also handle streamed top-level error objects without `choices` in `processOpenAICompletionsStream`, while preserving legitimate usage-only chunks. Keep existing sanitization/truncation behavior and do not dump raw provider payloads, request headers, credentials, host internals, or full bodies into assistant content. Add focused regression tests for non-2xx provider error payload formatting and streamed top-level Qianfan-style error chunks. Add a single-line changelog fix entry in the active release section if behavior changes; credit the reporter and useful analysis (`Thanks @yeiei and @Hollychou924`) if the entry is added.
