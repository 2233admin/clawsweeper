## Summary

Keep open: current main still rejects the requested cache-trace patch through the agent-facing gateway tool, and open PR #73098 is the active implementation candidate rather than a reason to close before merge.

## What This Changes

Allow the agent-facing gateway config patch/apply path to enable cache-trace observability when content capture is explicitly disabled, without allowing sensitive cache-trace payload capture.

## Best Possible Solution

Review and land PR #73098, or an equivalent narrow change, so `diagnostics.cacheTrace.enabled` can be patched only in a safe no-content-capture configuration with regression tests for safe enablement and blocked content capture.

## Reproduction Assessment

Yes. Static inspection of current main is enough: the reported patch changes `diagnostics.cacheTrace.enabled` and the three `include*` leaves, and the gateway tool guard rejects every changed path outside `ALLOWED_GATEWAY_CONFIG_PATHS`.

## Solution Assessment

Yes. The narrow maintainable direction is to conditionally allow safe cache-trace enablement while continuing to block content-capture fields; unconditionally allowing the subtree would be riskier because the capture flags default to `true`.
