## Summary

Keep open. Current main still routes live builtin memory search through a single query embedding call with timeout only, while batch indexing has the retry loop. The reported transport errors are still outside the memory embedding retry classifier on main, and the related open PR #44167 is batch/reindex-focused rather than a live-query fix.

## Best Possible Solution

Keep this issue open and land a narrow memory embedding transport fix: classify transient socket/fetch failures, apply bounded retry/backoff to the live query embedding path used by `openclaw memory search` and recall, and add focused unit coverage. If #44167 lands first, reuse its transport classifier but still add query-path retry coverage.

## Clownfish Work Prompt

Fix the live builtin memory-search embedding transport retry bug described in this issue. Current main retries batch embeddings but `embedQueryWithTimeout()` still calls `provider.embedQuery()` once, and OpenAI-compatible `fetchRemoteEmbeddingVectors()` uses one guarded `postJson()` call. Add a conservative transient transport-error classifier covering `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, socket hangups, and TLS socket disconnects, then apply bounded retry/backoff to the live query embedding path used by `openclaw memory search` and active recall. Preserve immediate failure for auth/config/validation/4xx errors, keep SSRF guarded fetch behavior intact, and do not add a config knob or dependency/dispatcher-wide tuning unless tests or live proof require it. Coordinate with #44167: if its batch classifier/split helper has landed, reuse it; if not, keep this PR focused on query retry and compatible with that work. Add or update focused Vitest coverage in memory-core and memory-host-sdk as needed. Add a single-line changelog fix entry with reporter attribution; do not change dependency pins, lockfiles, or unrelated memory/QMD concurrency behavior.
