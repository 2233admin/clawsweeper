## Summary

Keep open. Current main still leaves live memory-search query embeddings as a single remote provider call, while existing retry handling is limited to batch embedding/indexing paths and still does not classify the reported socket/TLS transport errors as retryable.

## What This Changes

The issue requests a fix for intermittent transient socket/TLS failures in live `openclaw memory search` query embedding calls across remote embedding providers.

## Best Possible Solution

Extend the existing memory embedding retry policy to cover transient socket/TLS transport failures and apply a bounded retry/backoff to live query embeddings, with focused tests proving query search retries transient failures while preserving immediate failure for non-retryable provider errors.

## Reproduction Assessment

Yes. The issue gives a concrete remote-provider `openclaw memory search` loop and two independent follow-up confirmations; current-main source inspection verifies that the live query embedding path remains single-shot.

## Solution Assessment

Yes. A scoped live-query retry using the existing memory embedding retry helper is the narrowest maintainable first fix; global Undici pool tuning or a new config knob should wait for proof that the scoped retry is insufficient.

## Repair Work Prompt

Fix the live memory-search query embedding retry gap for https://github.com/openclaw/openclaw/issues/71784. Current main retries batch embedding/indexing through `runMemoryEmbeddingRetryLoop(...)`, but `embedQueryWithTimeout()` in `extensions/memory-core/src/memory/manager-embedding-ops.ts` still calls `provider.embedQuery(text)` once, so `openclaw memory search` can fail on transient `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, or TLS socket disconnect errors. Keep the first PR narrow: extend the existing retry classifier in `manager-embedding-policy.ts` to cover transport/socket failures, apply the existing bounded retry/backoff helper to live query embeddings, and preserve immediate failures for auth/config/schema/non-retryable HTTP errors. Add focused tests for the classifier and the live query path; do not change global Undici dispatcher behavior, SSRF guard policy, provider config schema, QMD behavior, or introduce a new user-facing retry knob. #44166 and PR #44167 are related batch-indexing context only; do not make this issue depend on that PR unless the final patch intentionally shares helper code. Because this is a user-facing reliability fix, add a single-line CHANGELOG entry in the active Fixes section with appropriate non-bot reporter credit such as Thanks @kevinheinrichs, @heithmurray, and @kaufmanchang.
