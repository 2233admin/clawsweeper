## Summary

Keep open. Current main still routes live builtin memory search through a single query embedding call with only a timeout wrapper; the existing retry loop is used for batch paths, and its classifier still excludes the reported fetch/Undici/TLS socket failures. The later confirmations on 2026.4.26 and OpenRouter make this current and actionable, while related PR #44167 is open but scoped to batch/reindex behavior and does not close the live-search gap.

## What This Changes

The issue requests reliable live memory-search embedding by retrying transient remote-fetch socket failures such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, and TLS disconnects.

## Best Possible Solution

Add a conservative transient transport-error classifier and bounded retry/backoff for the live query embedding path used by builtin memory search, while preserving fail-fast behavior for auth, config, schema, and HTTP 4xx errors. Keep dispatcher tuning and user-facing retry config as follow-up unless targeted evidence shows the narrow retry is insufficient.

## Reproduction Assessment

Yes. The issue body gives a repeatable `openclaw memory search` loop with remote embedding providers, and follow-up comments confirm the same transport failure class on upgraded installs and another provider; static inspection shows current main still has the single-shot live query embedding path.

## Solution Assessment

Yes. A narrow retry-on-transient-transport fix around live query embeddings is the smallest maintainable first step; broad Undici pool tuning or new config knobs should wait until there is proof that the retry layer is insufficient.

## Repair Work Prompt

Fix the live builtin memory-search embedding retry gap tracked by https://github.com/openclaw/openclaw/issues/71784. Current main calls `MemoryManager.search()` -> `embedQueryWithTimeout()` and only applies a timeout around `provider.embedQuery(text)`, while batch paths use `runMemoryEmbeddingRetryLoop()`. Add a conservative transient transport-error classifier for fetch/Undici/socket failures such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, `socket hang up`, `terminated`, and `Client network socket disconnected before secure TLS connection was established`, and use bounded retry/backoff for live query embeddings. Preserve fail-fast behavior for auth/config/schema errors and HTTP 4xx/non-transient provider errors. Coordinate with open PR #44167, which handles adjacent batch/reindex transport retry work; avoid duplicating its batch split policy unless the branch has already landed or the fix must share the classifier. Likely files: `extensions/memory-core/src/memory/manager-embedding-policy.ts`, `extensions/memory-core/src/memory/manager-embedding-ops.ts`, related tests, and only touch `packages/memory-host-sdk` if the retry belongs lower than memory-core after inspection. Add a single-line changelog Fixes entry with reporter credit for the user-visible reliability fix. Do not add a config knob, provider-specific branches, or broad Undici dispatcher tuning unless focused tests prove the narrow retry is insufficient.
