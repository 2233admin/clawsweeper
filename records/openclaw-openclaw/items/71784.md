## Summary

Keep open. Current main still routes live builtin memory search through a single query embedding call without retrying the reported transport/socket failures, and later issue comments confirm the same failure class after 2026.4.26 and with OpenRouter.

## What This Changes

The issue requests reliable live memory search by retrying or otherwise hardening transient remote embedding transport failures in the builtin memory search path.

## Best Possible Solution

Add a narrow transient transport-error classifier and bounded retry/backoff for live query embeddings, sharing semantics with the existing batch retry loop while preserving fail-fast behavior for auth, configuration, schema, and non-retryable HTTP failures. Keep dispatcher tuning and a user config knob as follow-up work unless the focused fix proves insufficient.

## Repair Work Prompt

Fix the live builtin memory search reliability bug reported in https://github.com/openclaw/openclaw/issues/71784. Current main still runs `embedQueryWithTimeout()` as a single `provider.embedQuery(text)` call, while batch embedding has a retry loop and the retry classifier omits transient transport failures. Implement a narrow bounded retry/backoff for live query embeddings, preferably by reusing or extending the existing memory embedding retry policy. Classify only transient transport/socket failures such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, `socket hang up`, `terminated`, and `Client network socket disconnected before secure TLS connection was established`; do not retry auth/config/schema errors or non-retryable HTTP responses. Keep the change in memory-core/remote embedding seams, avoid provider-specific branching unless tests prove it is necessary, and do not add a config knob or broad Undici dispatcher tuning in this first PR. Coordinate with related #44166/#44167 so batch split/reindex behavior is not duplicated or regressed. Add focused unit coverage for query retry success, retry exhaustion, and non-retryable failure behavior. Changelog: add a user-facing fix entry if runtime behavior changes, crediting the reporter/contributor per repo rules.
