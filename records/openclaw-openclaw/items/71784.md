## Summary

Keep this issue open. Current main still sends live builtin memory-search query embeddings through a single guarded remote HTTP request, while existing memory embedding retries cover batch/rate-limit style paths and do not classify the reported transient transport errors. The recent production retest on 2026.4.26 also says the same symptom still reproduces, and the related open PR/issue track batch reindex resilience rather than this live query-search path.

## Best Possible Solution

Keep this issue open and fix it in the memory query embedding path, not only the batch reindex path. A conservative maintainer fix should add a narrow transient transport-error classifier for memory embedding HTTP calls and apply bounded retry/backoff to embedQueryWithTimeout() or the shared single-embed remote HTTP helper, covering fetch failed, other side closed, ECONNRESET, EPIPE, UND_ERR_SOCKET, and TLS socket disconnects while preserving immediate failures for auth/config/4xx errors. Add regression coverage in memory-core and memory-host-sdk for live query embeddings, then live-validate whether guarded-fetch dispatcher keep-alive tuning is also needed.
