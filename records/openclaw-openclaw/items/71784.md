## Summary

Keep open. Current main still routes live builtin memory search through a single query embedding call with no retry, while the remote OpenAI/Gemini paths use the guarded remote fetch helper and the memory retry classifier still omits the reported fetch/socket/TLS failure strings. Later issue comments confirm the same class on upgraded installs and OpenRouter, and the related open batch/reindex PR does not close this live-query surface.

## What This Changes

Make live builtin memory search resilient to intermittent provider-agnostic remote embedding transport failures during per-query embedding.

## Best Possible Solution

Add a conservative transient transport-error classifier and bounded retry/backoff for live query embeddings, ideally sharing semantics with the batch retry path while preserving existing fail-fast behavior for configuration, authentication, schema, and non-retryable HTTP errors. Keep dispatcher keep-alive tuning and user-facing retry config as follow-up work unless focused tests or live proof show the narrow retry is insufficient.

Yes. The issue gives concrete provider config and a repeatable `openclaw memory search` loop, and two later comments confirm the same failure class on upgraded/provider-diverse installs; static inspection of current main shows the live query path remains single-shot.

## Solution Assessment

Yes. A narrow retry for transient transport/socket failures around live query embeddings is the smallest maintainable first fix; broad Undici pool tuning or a new config knob should wait for targeted proof after the retry behavior is covered.

## Repair Work Prompt

Fix the live builtin memory search transient embedding transport failure tracked by https://github.com/openclaw/openclaw/issues/71784. Current main retries batch embeddings but `MemoryManager.search()` still calls `embedQueryWithTimeout()` once, so `openclaw memory search` and the `memory_search` tool can fail on transient remote provider socket errors such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, and `Client network socket disconnected before secure TLS connection was established`.

Scope the fix narrowly: add or reuse a conservative transient transport classifier for memory embeddings and apply bounded retry/backoff to the live query embedding path. Coordinate with #44166 and open PR #44167 so batch and query paths do not grow divergent classifiers; extend/adapt the existing retry helper if appropriate. Preserve fail-fast behavior for auth/config/schema errors and non-retryable HTTP 4xx responses. Do not add provider-specific branches, new config knobs, cache format changes, endpoint changes, dispatcher keep-alive tuning, or broad core imports from plugin code unless maintainer review explicitly asks for them.

Likely files: `extensions/memory-core/src/memory/manager-embedding-ops.ts`, `extensions/memory-core/src/memory/manager-embedding-policy.ts`, `extensions/memory-core/src/memory/manager-embedding-policy.test.ts`, `extensions/memory-core/src/memory/manager-embedding-timeout.test.ts`, possibly `packages/memory-host-sdk/src/host/embeddings-remote-fetch.ts` and tests if the retry belongs below providers, plus provider tests for OpenAI/Gemini if call-shape coverage changes. Add a user-facing changelog fix entry if runtime behavior changes; credit the reporter/contributors, not Codex/OpenClaw bot accounts.
