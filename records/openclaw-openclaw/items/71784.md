## Summary

Keep open. Current main still leaves the live memory-search query embedding path as a single guarded remote fetch wrapped only by a timeout, while the existing embedding retry loop applies to batch indexing. The follow-up comments confirm the same transport-failure class after upgrades and across OpenRouter/Gemini/OpenAI-style providers, and the related open PR covers batch reindexing rather than this live-query surface.

## What This Changes

The issue asks OpenClaw to make live memory search resilient to intermittent remote embedding transport failures such as `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, and TLS socket disconnects.

## Best Possible Solution

Add a narrow transient transport-error classifier and bounded retry/backoff for live query embeddings, preferably by reusing the existing memory embedding retry helper. Preserve immediate failures for auth, config, schema, and non-retryable HTTP errors; treat dispatcher keep-alive tuning or new user config as follow-up only if focused tests or live proof show the retry is insufficient.

## Reproduction Assessment

Yes. The issue provides a concrete remote-provider `openclaw memory search` loop, and the later comments add post-upgrade/provider-diverse confirmations; source inspection on current main verifies that the live query embedding path remains single-shot.

## Solution Assessment

Yes. A focused live-query retry using the existing memory embedding retry policy is the narrowest maintainable first fix; broad Undici pool changes should wait for evidence that the scoped retry does not cover the observed socket failures.
