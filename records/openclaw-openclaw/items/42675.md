## Summary

Keep this PR open. Current main still allows Discord startup to continue with an undefined bot identity when `fetchUser("@me")` rejects, and the proposed PR is a focused but stale implementation candidate tied to the still-open security/regression tracker #42219 rather than work already implemented on main.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a current-main rewrite in `extensions/discord/src/monitor/provider.startup.ts`: fail Discord startup when `fetchUser("@me")` rejects or returns no normalized bot id, add focused regression tests for both cases, and then handle any defense-in-depth mention-gate cleanup under the linked security/regression tracker #42219.
