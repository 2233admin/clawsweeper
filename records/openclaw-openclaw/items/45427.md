## Summary

Current main already carries resolved sender names through the inbound context. Discord resolves a platform-side sender name, stores it as `SenderName` alongside `SenderId`, and prompt metadata now prefers that name in the `sender` field plus a structured sender block with `id` and `name`. The relevant prompt fix shipped in `v2026.3.2`.

## Best Possible Solution

Keep the shipped SenderName-based inbound context and prompt metadata contract. Add a separate alias only if a concrete SDK/plugin consumer needs an exact `sender_display_name` field; otherwise the current implementation avoids another duplicate identity field.
