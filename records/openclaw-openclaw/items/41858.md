## Summary

Keep open. Current main shipped a root-and-filter mitigation for the linked FD-exhaustion report, but it has not definitely implemented this PR's intended depth-limit watcher hardening. The active skills, builtin memory, and QMD memory watchers still call chokidar without `depth` or `followSymlinks`, and the memory docs still promise recursive `extraPaths` behavior that a shallow cap could break.

## Best Possible Solution

Keep this PR open as a focused hardening candidate, or replace it with a rebased patch against `src/agents/skills/refresh.ts` and `extensions/memory-core/src/memory/manager-sync-ops.ts` after maintainers choose the intended depth, symlink, and `extraPaths` contract. Add regression tests that assert the chokidar options and update the public memory docs if recursive watch behavior changes; consider whether QMD needs matching watcher hardening.

## Risks / Open Questions

- A naive memory depth cap can silently miss documented recursive `memorySearch.extraPaths` content, including multimodal extra directories, unless the boundary and fallback behavior are designed and documented.
- The PR is stale against current main because the active memory watcher now lives in `extensions/memory-core/src/memory/manager-sync-ops.ts`, not `src/memory/manager-sync-ops.ts`.
- The PR does not settle `followSymlinks`, broad `extraPaths`, or QMD watcher behavior, which are part of the remaining resource-exhaustion surface.
