## Summary

Current main still should not close this issue. The raw-stream write guard now treats only explicit truthy values as enabled, so `0`, `off`, `false`, empty, and unset are disabled only when those values are what the gateway process actually sees. The reported LaunchAgent-managed path remains actionable because service env can also come from durable `.env`/config sources and generated service-env wrappers, normal restart still kickstarts the existing job without rewriting that wrapper, and the public docs still do not explain raw-stream disable values or LaunchAgent/env-source precedence.

## Best Possible Solution

Keep this issue open. The best fix is a narrow docs plus regression pass: document exact enabled and disabled values for `OPENCLAW_RAW_STREAM`, explain how CLI flags, LaunchAgent inherited env, durable `$OPENCLAW_STATE_DIR/.env` / `~/.openclaw/.env`, config `env.vars`, and generated service-env wrappers interact, and add focused tests for raw-stream disabled values plus LaunchAgent install/restart env behavior. If live macOS verification confirms stale wrapper or durable env overrides `launchctl unsetenv`/`0` unexpectedly, update the service-env refresh/status/readback path so operators can identify and clear the effective source.

## Risks / Open Questions

- Raw-stream logs can include prompts, assistant output, and tool output, so unclear disable/readback behavior is a privacy and operator-trust risk.
- The remaining ambiguity is source precedence rather than the truthy parser itself: post-disable writes could come from launchctl inherited env, durable `.env` or config env, a generated service-env wrapper, or a gateway process that did not actually restart into the expected environment.
