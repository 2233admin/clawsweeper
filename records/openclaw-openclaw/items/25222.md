## Summary

Keep this issue open. Current main still queues same-session messages during active runs without an immediate channel-visible busy/status acknowledgement, cancel/new-session guidance, or stable busy reason payload. Related implementation work exists, but it remains open/unmerged and overlapping design should be reconciled before this canonical UX request closes.

## Best Possible Solution

Keep this issue open as the canonical UX request. Implement a generic active-session busy-reply contract around queue policy, runReplyAgent, and dispatch: configurable channel-safe status text, optional /stop and /new guidance, stable busy reason metadata, and focused tests for queue policy, runner return payloads, dispatch, and at least one channel path. Evaluate #39117 as an implementation candidate and fold or duplicate-close #50600 only after maintainers choose the canonical design.
