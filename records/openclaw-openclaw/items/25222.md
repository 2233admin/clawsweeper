## Summary

Keep this issue open. Current main still treats active same-session followups as queued work with no immediate channel-visible busy/status reply, no busy reason payload, and no automatic guidance for /stop or /new on the busy path. The related PR #39117 remains open/unmerged and #50600 overlaps rather than superseding this canonical UX request.

## Best Possible Solution

Keep this issue open as the canonical UX request. Reconcile #39117 and #50600, then implement a generic active-session busy-reply contract around queue policy, runReplyAgent, and dispatch: configurable channel-safe status text, optional /stop and /new guidance, stable busy reason metadata, and focused coverage for queue policy, runner return payloads, dispatch, and at least one channel path.
