## Summary

Keep this issue open. Current main still queues or serializes ordinary same-session messages during an active run instead of returning the requested immediate channel-visible busy/status reply with cancel/new-session guidance. The existing `/stop`, `/new`, typing/status-adjacent behavior, and queued follow-up handling are useful pieces, but they do not implement the central requested UX.

## Best Possible Solution

Keep this issue open. The best path is a generic, configurable active-session busy-reply path around `resolveActiveRunQueueAction`, `resolvePreparedReplyQueueState`, and `runReplyAgent`, with channel-safe text/action guidance and focused runner plus bridge regression tests, especially for Telegram same-topic serialization.
