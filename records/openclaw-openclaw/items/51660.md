## Summary

Keep open. Current main still has no supported `tools`-level umask, file mode, dir mode, or `filePermissions` policy for agent-created user-facing output. Existing code and docs cover tool containment and sensitive-state hardening, but host write/edit, apply_patch, workspace-safe writes, sandbox bridge writes, and exec-created files do not share a configurable output-permission policy.

## Best Possible Solution

Keep this open and design a core, documented user-output file creation policy, likely under `tools.fs` or a narrow `tools.fileCreation` block. It should validate octal values, apply to host write/edit, apply_patch, workspace-safe writes, sandbox bridge writes, and exec-created workspace output where feasible, while preserving restrictive modes for OpenClaw-managed sensitive state. Maintainer/secops should settle the config boundary and exec/ownership semantics before implementation.

## Risks / Open Questions

- A global `process.umask` change could accidentally loosen sensitive OpenClaw-managed files such as config, credentials, auth profiles, transcripts, sessions, audit output, and plugin state.
- A partial fix limited to the host write tool would not solve the reported Docker/NAS workflow for exec-created files, apply_patch, sandbox bridge writes, or workspace-safe writes.
- The issue's environment also mentions UID/GID ownership (`nobody:users` vs `node:node`); umask and mode settings alone may not solve ownership on bind-mounted NAS shares.
- The config shape needs strict octal validation and clear docs distinguishing user-facing output from internal security hardening.
