## Summary

Keep open. The PR is not merge-ready: it introduces a block-scoped `initializedPid` reference that would fail TypeScript compilation, and the currently pinned `acpx@0.6.1` runtime does not expose PID on the handle/status fields this patch reads, so the intended `sessions_spawn` PID would still be omitted.

## What This Changes

The PR adds optional PID fields to ACP runtime/session types, tries to propagate ACPX delegate PIDs into session metadata/status, and includes PID in ACP `sessions_spawn` accepted results with new unit tests.

## Best Possible Solution

Define an additive, typed ACP process-metadata contract, have ACPX populate the real process identifier fields it can support, then propagate those fields through ACP session metadata, `sessions_spawn`, and status surfaces with tests that exercise the actual ACPX wrapper behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Hoist initializedPid before returning it:** `src/agents/acp-spawn.ts:1464`
  - body: `initializedPid` is declared with `const` inside the initialization `try` block, but both accepted-result returns are after that block. TypeScript will fail with `Cannot find name 'initializedPid'` until the variable is declared in the outer scope and assigned inside the `try`.
  - confidence: 0.98
- **[P2] Read PID from the actual acpx runtime surface:** `extensions/acpx/src/runtime.ts:517`
  - body: The pinned `acpx@0.6.1` public `ensureSession` handle does not include `pid`; its internal record PID is only reflected in the status summary string. Wrapping `delegate.ensureSession()` with `resolvePidFromHandle()` therefore leaves the real `sessions_spawn` path without a PID even after this change.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff intentionally exposes host process IDs to tool-visible results and persisted session metadata, which needs maintainer confirmation of the ACP process-metadata security model.

Concerns:

- **[medium] Confirm PID exposure policy:** `src/agents/acp-spawn.ts:1464`
  - body: Returning and persisting OS PIDs gives agents process-correlation metadata that can be combined with other tools to inspect or signal host processes. This may be acceptable for ACP debugging, but the PR should land with an explicit scoped contract for which PID is exposed and to whom.
