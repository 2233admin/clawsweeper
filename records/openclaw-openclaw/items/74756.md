## Summary

Keep this PR open, but it is not merge-ready. The linked bug is real on current main, and the PR moves in a useful direction, but the branch has a TypeScript scope blocker, depends on PID fields that the pinned ACPX runtime does not expose through its public handle/status contract, and needs explicit maintainer policy for exposing host process IDs.

## What This Changes

The PR adds optional PID fields to ACP runtime/session metadata and attempts to propagate ACPX delegate PIDs into `sessions_spawn` accepted results with unit coverage.

## Best Possible Solution

Define a documented additive ACP process-metadata contract, update ACPX or its wrapper to expose the real adapter and child-agent identifiers that contract promises, then propagate those fields through ACP session metadata, `sessions_spawn`, status surfaces, docs, and tests that exercise the actual ACPX runtime behavior.

## Reproduction Assessment

Yes. Spawn an ACP child with `sessions_spawn({ runtime: "acp" })` on current main and inspect the accepted result/status; source and docs confirm no PID field is produced and operators are still pointed at manual `ps` correlation.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it does not compile as written, mocks PID output that `acpx@0.6.1` does not publicly return, and skips the process-metadata/security contract needed before exposing host PIDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Hoist initializedPid before returning it:** `src/agents/acp-spawn.ts:1467`
  - body: `initializedPid` is declared inside the initialization `try` block, but both accepted-result returns are after that block. TypeScript cannot resolve the name at these return sites, so this branch will fail to compile until the variable is declared in the outer scope and assigned inside the `try`.
  - confidence: 0.98
- **[P2] Use a real ACPX PID source:** `extensions/acpx/src/runtime.ts:520`
  - body: The pinned `acpx@0.6.1` public `ensureSession` handle and `getStatus` result do not expose top-level or `details.pid` values, so these tests pass by mocking delegate output real ACPX-backed spawns will not return. Expose the PID in ACPX or read a supported runtime surface before propagating it.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: Security review needs attention because the patch intentionally exposes host OS process IDs in agent-visible and persisted ACP metadata, though it does not add new dependencies or CI execution paths.

Concerns:

- **[medium] Confirm host PID exposure policy:** `src/agents/acp-spawn.ts:1467`
  - body: Returning and persisting host PIDs lets agents correlate OpenClaw sessions with local OS processes and may combine with other tools or future kill controls. That can be acceptable for ACP debugging, but it should land with a scoped contract for which PID is exposed, who can see it, and how stale or reused identifiers are handled.
  - confidence: 0.74
