## Summary

Keep this PR open for rework. The requested PID surface is valid and current main still lacks it, but the branch reads PID fields that the pinned ACPX runtime does not expose and also needs an explicit host-PID visibility contract before merge.

## What This Changes

The PR adds optional PID fields to ACP runtime handles, statuses, session metadata, `sessions_spawn` accepted results, and related unit tests.

## Best Possible Solution

Add a typed ACPX process-metadata contract first, update OpenClaw to consume that published contract deliberately, and then propagate the documented PID semantics through the plugin SDK runtime types, ACP control-plane metadata, `sessions_spawn`, `session_status`, docs, changelog, and integration-style coverage.

## Reproduction Assessment

Yes. Source inspection against current main reproduces the missing PID surface, and inspection of `acpx@0.6.1` reproduces why this PR would not populate PID for real ACPX-backed sessions.

## Solution Assessment

No. The direction is useful, but this implementation is not the narrowest maintainable fix because it depends on non-existent ACPX fields; the safer path is a typed ACPX contract plus explicit OpenClaw visibility and lifetime semantics.
