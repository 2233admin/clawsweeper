## Summary

Keep open. Current main still lacks the requested ACP process-correlation surface, but this PR is not merge-ready because it has a TypeScript scope blocker, reads PID data from a surface that `acpx@0.6.1` does not currently expose as typed handle/status metadata, and needs an explicit API/security decision before host PIDs are returned to agents.

## What This Changes

The PR adds optional `pid` fields to ACP runtime/session metadata and attempts to propagate ACPX delegate PIDs through the ACP manager into `sessions_spawn` results with unit coverage.

## Best Possible Solution

Define a documented additive ACP process-metadata contract, update ACPX or its wrapper to expose the real adapter and child-agent identifiers that contract promises, then propagate those fields through ACP session metadata, `sessions_spawn`, status surfaces, docs, and tests that exercise the actual ACPX runtime behavior.

## Reproduction Assessment

Yes. The linked bug is reproducible by spawning an ACP child with `sessions_spawn(runtime="acp")` on current main and inspecting the returned result/status; source inspection confirms no PID field is produced, and the PR's compile and dependency-surface failures are visible from the diff and pinned `acpx@0.6.1` runtime output.

## Solution Assessment

No. The general direction is useful, but this patch is not the best fix because it does not compile, depends on mocked PID fields that the pinned ACPX runtime does not expose, and skips the process-metadata/security contract needed before exposing host PIDs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P1] Hoist initializedPid before returning it:** `src/agents/acp-spawn.ts:1467`
  - body: `initializedPid` is declared inside the initialization `try` block but both accepted-result returns are after that block. TypeScript cannot resolve the name at these return sites, so the branch will fail to compile until the variable is declared in the outer scope and assigned inside the `try`.
  - confidence: 0.98
- **[P2] Read PID from the actual ACPX runtime surface:** `extensions/acpx/src/runtime.ts:520`
  - body: The pinned `acpx@0.6.1` `ensureSession` handle and `getStatus` result do not expose typed `pid` fields, so these wrapper tests only pass because they mock impossible delegate output. Real ACPX-backed spawns will still omit the requested PID unless ACPX exposes the process metadata or the wrapper reads a real supported source.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The PR intentionally exposes host OS process IDs in agent-visible ACP spawn/status metadata, so it needs an explicit process-metadata policy before merge.

Concerns:

- **[medium] Confirm host PID exposure policy:** `src/agents/acp-spawn.ts:1467`
  - body: Returning and persisting host PIDs lets agents correlate OpenClaw sessions with local OS processes and may combine with other tools or future kill controls. That can be acceptable for ACP debugging, but it should land with a scoped contract for which PID is exposed, who can see it, and how stale/reused identifiers are handled.
