## Summary

Keep open. Current main still does not implement the Discord adapter guard that this PR proposes, and the shared reply prompt path can still prepend `[Thread starter - for context]` on existing sessions whenever Discord supplies `ThreadStarterBody`. The open linked bug #41355 and maintainer branch update make this PR the active implementation candidate rather than an obsolete PR.

## Best Possible Solution

Keep this PR open for maintainer review and landing, or land an equivalent Discord adapter fix. The desired implementation is to compute the effective Discord session key, omit `ThreadStarterBody` when that effective session already exists, preserve first-turn starter context, and keep the regression test that proves existing thread sessions omit the starter while new sessions still receive it. After merge, close #41355 through the PR's closing reference.
