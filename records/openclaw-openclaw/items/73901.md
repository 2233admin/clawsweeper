## Summary

Current main still lacks the requested approval gate. The agent-facing gateway tool continues to fail closed on non-allowlisted config mutations, and widening that boundary needs explicit maintainer/security design rather than cleanup closure.

## What This Changes

Requests an optional operator approval gate, plus optional expanded allowlist, for agent-facing `gateway config.patch` changes outside the current strict allowlist.

## Best Possible Solution

Add a first-class config mutation approval policy: keep strict rejection as the default, classify config paths as always allowed, promptable, or never promptable, show an operator-readable diff, bind approval to the exact base hash and patch, add an optional reviewed allowlist, and cover Telegram/Discord/native approval delivery plus gateway-tool and protocol tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: run the agent-facing `gateway` tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current code computes changed paths and throws before forwarding the RPC, and tests encode those blocked cases.

## Solution Assessment

No, not as a direct reuse of exec approvals. The maintainable solution is a config-specific approval policy and protocol surface, because config writes need path classification and exact diff/base-hash semantics that exec/plugin approvals do not currently model.
