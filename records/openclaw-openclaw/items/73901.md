## Summary

Current main does not implement the requested operator approval gate for agent-facing `gateway` `config.patch` outside the allowlist. The core RPC can write validated config for admin-scoped callers, but the model-facing gateway tool still fails closed on paths outside `ALLOWED_GATEWAY_CONFIG_PATHS`, and the current approval protocol is scoped to exec/plugin approvals rather than config mutation approvals.

## Best Possible Solution

Keep this open for maintainer/security design. The best path is a first-class config mutation approval policy: classify paths as always allowed, promptable, or never promptable; show an operator-readable diff; bind approval to the exact base hash and patch; preserve the strict default; optionally add a reviewed config allowlist; and update Telegram/Discord/native approval delivery plus gateway-tool and protocol tests.

## Risks / Open Questions

- This request sits on the GHSA-related model-to-operator trust boundary; broadening `config.patch` without a path policy could let a model alter auth, plugin loading, tools, MCP servers, hooks, or sandbox settings after one generic approval.
- Exec approvals are not a per-user auth boundary per docs; reusing that UX for config writes needs explicit semantics for who can approve, what exact diff is approved, expiry/replay behavior, and which paths remain never-promptable.
