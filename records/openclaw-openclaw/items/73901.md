## Summary

Keep open. Current main still intentionally rejects agent-driven `config.patch` mutations outside a narrow allowlist, and the issue asks to change a GHSA-related model-to-operator trust boundary rather than describe behavior that is already implemented. The request is concrete, but it needs maintainer security/product design before code work.

## What This Changes

The issue requests an operator approval path for agent-driven `gateway config.patch` mutations outside the current allowlist, with optional configured allowlist expansion for approved paths.

## Best Possible Solution

Add a first-class config mutation approval policy: keep rejection as the default, classify paths as always allowed, promptable, or never promptable, show an operator-readable diff, bind approval to the exact base hash and patch, support an explicit reviewed allowlist, and cover Telegram/Discord/native approval delivery plus gateway-tool and protocol tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call the agent-facing gateway tool with `action: "config.patch"` for a protected path such as `plugins.allow` or `memory.qmd.command`; current main rejects it before RPC forwarding, and guard tests encode those blocked cases.

## Solution Assessment

No. Reusing exec approvals for every non-allowlisted config patch is not the safest maintainable solution; this needs config-specific approval semantics with path classification, exact diff/base-hash binding, and explicit never-promptable paths.
