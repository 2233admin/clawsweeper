## Summary

Keep this open. Current main still does not provide native Control UI pages or tabs from plugins: navigation and page rendering are built-in, bootstrap config has no plugin page/import-map metadata, plugin manifests do not declare `ui.pages`, and the adjacent `registerControlUiDescriptor` API is limited to descriptor metadata for session/tool/run/settings surfaces. The issue remains active product/API work with concrete Openclaw-Quest context and security decisions still needed.

## What This Changes

Requests a plugin UI extension system that lets plugins declare native Control UI pages or tabs, load Custom Elements at runtime, and share the main UI theme and gateway client instead of using standalone plugin pages.

## Best Possible Solution

Keep this as the canonical design track for a versioned Control UI plugin-page API. The end state should define manifest or SDK page declarations, safe plugin asset serving, bootstrap metadata, navigation grouping, browser Custom Element loading, gateway/client access rules, docs, and contract/UI tests with the security model documented before external plugin authors depend on it.

## Security Review

Status: needs_attention

Summary: No patch was reviewed, but the proposed feature is security-sensitive because it would execute plugin-owned browser code inside the authenticated operator Control UI.

Concerns:

- **[high] Define the Control UI plugin code trust boundary:** `src/gateway/control-ui-contract.ts:5`
  - body: A native plugin page loader would share the Control UI DOM, theme, and likely gateway auth/client context with third-party plugin JavaScript. The design should explicitly cover CSP, allowed entry URLs, operator scopes, dependency/runtime sharing, data access, unload behavior, and whether untrusted plugins may use this surface.
