## Summary

Keep open. Current main does not implement the PR's heartbeat/session activity controls or progress-prompt guidance, and the PR branch still has blocking correctness and authorization-sensitive stop-semantics issues that need maintainer review before any merge.

## What This Changes

This PR adds Control UI heartbeat/session stop controls, a new `sessions.activity` RPC, broader `sessions.abort` cleanup for queued/subagent/ACP/cron work, stream/progress handling changes in Control UI and OpenClawKit, and system-prompt guidance for milestone/progress updates.

## Best Possible Solution

Split and rebase the work: keep ordinary generation Stop on `chat.abort`, design session activity/autonomy stop against #39127 with protocol docs and authorization tests, and land the progress-prompt guidance separately with focused prompt coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep normal chat abort on chat.abort:** `ui/src/ui/controllers/chat.ts:337`
  - body: The regular WebChat stop path now sends `sessions.abort` for active generation aborts. That handler also clears queues, stops subagents, cancels ACP sessions, and disables session-bound cron jobs, so clicking Stop on one answer can pause unrelated session automation. Keep ordinary generation stop on `chat.abort`, or put the broader cleanup behind an explicit autonomy-stop action.
  - confidence: 0.9
- **[P2] Use snapshots as replacements when no delta is present:** `ui/src/ui/controllers/chat.ts:208`
  - body: When a non-prefix assistant snapshot arrives without `delta`, this fallback appends it to the current stream. Gateway snapshots are authoritative in that case, so rewritten or replayed text can be duplicated or corrupted in the UI and in locally committed abort partials.
  - confidence: 0.86
- **[P2] Exclude dropped followups from pending size:** `src/auto-reply/reply/queue/state.ts:43`
  - body: `droppedCount` tracks discarded queue entries, not pending work. Including it makes `followupQueued > 0` and `canStop: true` after the queue is actually empty, causing the stop control to appear for work that cannot be stopped.
  - confidence: 0.91
- **[P2] Return cleanup state after timeout errors:** `src/gateway/server-methods/sessions.ts:1030-1040`
  - body: By this point queues, subagents, ACP sessions, and cron jobs may already have been cleaned up. If the embedded-run wait times out, the handler returns only `UNAVAILABLE`, so clients cannot reconcile the work already stopped and may keep stale UI state or retry unnecessarily.
  - confidence: 0.75
