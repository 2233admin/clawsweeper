## Summary

Keep open: the contributor’s goal is real, but the PR is not mergeable as written because it makes the shared `sessions_spawn` contract reject documented ACP default-agent calls before ACP runtime resolution can run. Current `main` still preserves optional `agentId` plus an opt-in strict `requireAgentId` path, so the remaining decision is contract/product direction rather than conservative cleanup.

## What This Changes

The PR makes `sessions_spawn.agentId` required in the shared tool schema/runtime reader and updates spawn-related tests and mocks to pass explicit agent IDs.

## Best Possible Solution

Preserve ACP `acp.defaultAgent` compatibility while improving explicit target reliability for native subagents, either by strengthening the existing `requireAgentId` config path and prompt guidance or by adding runtime-specific schema behavior that requires `agentId` only where omission is not a supported contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction is `sessions_spawn({ runtime: "acp", task: "..." })` with `acp.defaultAgent` configured: the PR's required shared reader throws before `spawnAcpDirect` can resolve the default.

## Solution Assessment

No. A global shared-schema requirement is not the narrowest maintainable fix because ACP intentionally supports omitted `agentId`; a runtime-specific or config-backed strict path is safer.
