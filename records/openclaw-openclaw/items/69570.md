## Summary

Keep this issue open. Current main still creates a separate follow-up `agent` turn for each async exec approval completion, keyed by the individual approval id. `/stop` and queue controls now provide partial mitigation, but I did not find task-level batching, stale completion discard, completion suppression after stop, or guaranteed user-message priority over pending approval-completion follow-ups. The 2026-04-26 confirmation comment is recent evidence that this remains an active runtime/product bug family, related to #69386 and #72062.

## Best Possible Solution

Keep this issue open as the canonical async exec approval completion-cascade bug. The best product/code direction is a core follow-up scheduler or circuit breaker around exec approval completions: coalesce multiple completions for the same session/task when safe, let `/stop` and newer user input preempt or cancel pending completion follow-ups, suppress stale completions after a task is abandoned, and preserve existing per-approval audit/idempotency guarantees. Wrapper-denial policy work should stay separate under #72062 because it reduces one trigger path but does not fix the completion cascade.
