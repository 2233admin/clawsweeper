## Summary

Keep this issue open. Current main still routes approved async exec completions into separate follow-up agent turns keyed by individual approval ids; I found partial queue clearing and stop handling, but not task-level batching, stale-completion suppression, or priority cancellation for already-scheduled approval completion follow-ups.

## What This Changes

The issue reports that approved async exec completions can create many delayed follow-up agent turns, blocking later user input and lacking batching, stale suppression, or cancellation.

## Best Possible Solution

Keep this as the canonical async exec approval completion-cascade bug. The durable fix should add a core scheduler or circuit breaker that can coalesce safe same-session completions, let /stop and newer user input preempt pending completion follow-ups, suppress stale completions after abandonment, and preserve direct diagnostics plus per-approval audit/idempotency.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is multiple same-session gateway exec calls that require approval and then finish: current main schedules one async follow-up agent turn per approval id. The issue discussion also includes a recent deterministic production reproduction on OpenClaw 2026.4.21.

## Solution Assessment

No, the current per-approval follow-up path is not the best solution for the reported behavior. Pausing approval-pending turns as in #70953 would reduce approval flooding, but this issue still needs explicit completion coalescing/cancellation semantics for already approved async commands.
