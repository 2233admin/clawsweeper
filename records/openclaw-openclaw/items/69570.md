## Summary

Keep #69570 open. Current main still starts a separate async agent follow-up for each approved or denied exec result, and current tests assert that approval resolution auto-continues the same session without a second user turn. The existing /approve shell-command guard addresses a related loop class, but it does not solve this issue's queued completion/follow-up cascade or pending-followup cancellation problem.

## Best Possible Solution

Keep the issue open and fix it in core approval/session orchestration: batch or squash exec approval completions per session/logical task, make /stop and chat.abort cancel pending exec approval follow-ups and completion events, prioritize new user input over stale completion turns, and add a circuit breaker for repeated deny/approval variants. #72062 may reduce wrapper-policy surface area, but it is not a substitute for this core queue/follow-up fix.
