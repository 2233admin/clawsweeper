## Summary

Keep open. Current main still routes pre-send `message_sending` through the typed plugin hook runner only, while user-defined hook-directory handlers live in the internal hook registry; the open PR #73570 references this issue with closing syntax, so the issue should stay open until that PR lands or maintainers replace it.

## What This Changes

Make the outbound pre-send `message_sending` lifecycle fire for user-defined hook-directory handlers, not only typed plugin hooks.

## Best Possible Solution

Land or replace the linked implementation with a canonical pre-send internal hook bridge that preserves plugin modification/cancel ordering, supports or migrates the existing `message_sending` HOOK.md event spelling, and adds focused docs/tests for the internal pre-send event semantics.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: hook-directory metadata registers literal event keys, internal dispatch only reaches `message` and `message:<action>` listeners, and the current pre-send outbound path only invokes plugin `runMessageSending`.

## Solution Assessment

Unclear. Bridging the pre-send path is the right direction, but the final fix should explicitly handle `message_sending` versus `message:sending` compatibility and pre-delivery ordering rather than only adding a notification that existing hook-directory listeners may not receive.
