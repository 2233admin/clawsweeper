## Summary

Keep open. Current main still has a plugin-only `message_sending` pre-delivery path for the inspected outbound surfaces, while user-defined HOOK.md handlers are registered in the internal hook registry. The open PR #73570 explicitly references this issue with closing syntax, so this issue should remain open until maintainers land, replace, or close that PR.

## What This Changes

User-defined hooks registered for `message_sending` should fire before outbound message delivery, matching the plugin hook surface.

## Best Possible Solution

Land or replace PR #73570 with a pre-delivery internal hook bridge that preserves plugin modification/cancel ordering, defines `message_sending` versus `message:sending` compatibility, documents the supported user-hook semantics, and adds focused coverage for shared outbound delivery plus inbound reply dispatch paths.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: HOOK.md event strings are registered literally, internal dispatch reaches only `message` and `message:<action>` listeners, and the current pre-send delivery path only invokes plugin `runMessageSending`.

## Solution Assessment

Unclear. Bridging the pre-send path is the right area, but the implementation should be reviewed for event-name compatibility, pre-delivery ordering, and whether user hooks should only observe or also modify/cancel outbound content.
