## Summary

Keep open. Current main still dispatches the shared outbound pre-send path only through the plugin hook runner, while file-based internal hooks are registered in a separate registry and only receive `type` / `type:action` events. The open PR #73570 references this issue with closing syntax, so the issue should remain open until that PR or a replacement lands.

## What This Changes

User-defined HOOK.md hooks that declare the pre-send `message_sending` event should fire before outbound message delivery, matching the plugin pre-send hook surface.

## Best Possible Solution

Review and land PR #73570 or a replacement that defines the canonical user-facing pre-send internal hook event, preserves pre-delivery ordering, handles `message_sending` versus `message:sending` compatibility or documentation clearly, and adds focused tests for shared outbound delivery plus inbound reply dispatch.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: HOOK.md event strings are registered directly, internal dispatch only reaches `message` and `message:<action>` listeners, and current pre-send delivery code only calls plugin `runMessageSending`.

## Solution Assessment

Unclear. Bridging the pre-send path is the right area, but a complete fix should also settle event-key compatibility and the observe-versus-modify/cancel contract for user-defined hooks.
