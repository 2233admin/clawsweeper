## Summary

Keep open. Current main still has the reported gap: typed plugin `message_sending` hooks can rewrite or cancel outbound delivery, but standalone/internal HOOK.md hooks are not dispatched for a pre-send message event. The existing `message:sent` bridge proves the dual-dispatch pattern exists, while `message_sending` remains plugin-runner-only in the shared outbound path and the inbound/webchat before-deliver path.

## Best Possible Solution

Implement and document an internal pre-send hook surface in the shared outbound delivery path and the inbound/webchat before-deliver path. Decide whether standalone hooks should listen on `message:sending`, `message_sending`, or both; define whether they can observe only or mutate/cancel; then add regression tests proving a user hook from `hooks/` fires before delivery.

## Risks / Open Questions

- Maintainers need to decide the public internal event key contract: existing internal events use colon keys such as `message:sent`, while the reporter attempted plugin-style `message_sending`; supporting only one spelling may leave some user hooks broken.
- Internal hook handlers currently return `void`, while typed plugin `message_sending` can rewrite or cancel delivery. If the user-hook feature is meant to provide guardrails rather than observation only, mutation/cancel semantics need an explicit contract and regression coverage.
- There are multiple outbound surfaces, including shared outbound delivery, inbound/webchat before-deliver dispatch, and Telegram-specific delivery. A fix should cover the relevant delivery paths consistently or document any intentional exceptions.
