## Summary

Keep open. Current main still has a plugin-only `message_sending` pre-send path for the inspected outbound delivery surface, while user-defined HOOK.md handlers are registered in the separate internal hook registry. Open PR #73570 explicitly references this issue with closing syntax, so the issue should remain open until maintainers review, land, or replace that implementation candidate.

## What This Changes

User-defined HOOK.md hooks registered for `message_sending` should fire before outbound message delivery, matching the existing plugin hook interception point.

## Best Possible Solution

Add a canonical internal pre-send hook bridge with explicit compatibility for existing HOOK.md event naming or a documented migration path, preserve plugin modification/cancel ordering, and add focused coverage for the shared outbound delivery path plus any intended inbound reply dispatch path.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: HOOK.md events are registered literally, internal dispatch only reaches `message` and `message:<action>` listeners, and the current pre-send delivery path only invokes plugin `runMessageSending`.

## Solution Assessment

Unclear. Bridging the pre-send path is the right area, but the final fix should address underscore-versus-colon event compatibility, documented semantics, and ordering before maintainers treat the issue as solved.
