## Summary

Keep open. Current main still has a real gap: typed plugin `message_sending` hooks run before delivery, but user/internal hooks are not bridged for a `message:sending` or `message_sending` event. The related older hook requests are closed, and no canonical open implementation candidate was present in the provided context.

## Best Possible Solution

Implement and document an internal pre-send hook surface, ideally in the shared outbound delivery path and the inbound/webchat before-deliver path. Add regression tests proving a user hook registered from `hooks/` fires before delivery, decide whether `message:sending`, `message_sending`, or both are accepted, and define whether internal hooks can mutate/cancel or are observe-only.

## Risks / Open Questions

- The implementation needs a clear public contract for the internal event key. Current internal hooks use colon names such as `message:sent`, while the typed plugin hook is `message_sending`; supporting only one spelling could leave existing user attempts broken.
- Internal hook handlers currently return `void`, while typed plugin `message_sending` can rewrite or cancel delivery. If maintainers want user hooks to provide guardrails rather than observe only, the fix needs explicit mutation/cancel semantics and tests across both delivery paths.
