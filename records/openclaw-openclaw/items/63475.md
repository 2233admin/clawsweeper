## Summary

Keep this issue open. Current main still starts Feishu accounts through the gateway's concurrent per-account path, and the existing sequential Feishu preflight only covers the aggregate no-accountId monitor path. The two linked closing PRs are open and unmerged, so the issue should remain open for review rather than being closed as implemented.

## What This Changes

The issue reports Feishu multi-account gateway startup firing startup bot identity and token-related probes concurrently, causing one account to time out and recover later in background retry.

## Best Possible Solution

The end state should keep the fix inside the Feishu plugin: serialize or otherwise throttle startup bot-identity probing across concurrent gateway `startAccount` calls, preserve background recovery behavior, and add regression coverage for the actual gateway per-account startup path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure multiple Feishu accounts and start the channel through gateway whole-channel startup, which maps account IDs through `Promise.all` and reaches `fetchBotIdentityForMonitor` in each per-account Feishu task; live Feishu credentials would be needed to reproduce the external timeout itself.

## Solution Assessment

Yes, the narrow maintainable solution is Feishu-owned serialization or throttling around startup identity probes, not a core-wide channel startup policy change. The existing open PR #63481 is the closest implementation candidate, but it should be reconciled with PR #67466 before merge.
