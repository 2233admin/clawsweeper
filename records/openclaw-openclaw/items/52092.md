## Summary

Keep this issue open. Current main has meaningful cron catch-up, concurrency, and session-store mitigations, but it still does not implement the reported Gateway heap-headroom/config behavior: generated Gateway service environments and systemd ExecStart paths do not set `NODE_OPTIONS` or `--max-old-space-size`, and `GatewayConfig` has no `nodeMaxOldSpaceMb`-style contract. The open #42853 remains the closest implementation candidate; #55334/#69404 address adjacent session-store OOM pressure, not this complete cron-triggered service heap issue.

## Best Possible Solution

Keep this issue open as the cron-triggered Gateway heap/OOM tracker. The best maintainer path is to review and land a narrow service heap-headroom/config solution, likely a revised #42853 or equivalent, while preserving the existing cron concurrency/catch-up/session-maintenance safeguards and coordinating broader session-store payload work under #55334.
