## Summary

Keep open. Current main has useful mitigations for oversized active transcripts and async/bounded trajectory recording, but it does not prove the reported gateway event-loop starvation class is fixed: QQBot heartbeats still run on the Gateway event loop, and compaction transcript rotation still has synchronous full-transcript JSON serialization before the async write.

## What This Changes

The issue reports large-session compaction and trajectory persistence blocking the Node event loop long enough to drop QQBot WebSocket heartbeats and make cron jobs fail.

## Best Possible Solution

Keep this issue open and land a focused liveness fix that makes large-session compaction and transcript rotation nonblocking for the Gateway event loop, with a synthetic large-transcript/event-loop-delay regression test and QQBot/cron validation notes. Recovery-only heartbeat reconnects should remain secondary to preventing long stalls.

## Reproduction Assessment

Yes. The issue gives a concrete field reproduction and liveness signature, and current code inspection shows the QQBot heartbeat timer shares the Gateway event loop; a high-confidence validation path is a large active transcript plus compaction while observing `eventLoopDelayMaxMs` and QQBot close 4009/1006 behavior.

## Solution Assessment

No. Current main's configured byte threshold, async trajectory appends, and reconnect handling are useful partial mitigations, but they are not the best complete fix for this report because they do not prove compaction/session serialization is off the Gateway event loop or bounded enough to protect WebSocket heartbeats and cron scheduling.
