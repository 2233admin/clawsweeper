## Summary

Keep open. Current main still has no skill-specific before/after/invoke/success/fail hook in either the typed plugin hook surface or the internal hook catalog; only tool-call hooks and skill-install hooks are present. The discussion’s enterprise permission-check use case makes this a security-sensitive plugin API design item rather than stale cleanup or ClawHub-only work.

## What This Changes

Add a skill execution lifecycle hook surface, or at least an execution log, so plugins or scripts can observe and potentially act before and after skill invocation.

## Best Possible Solution

Add an additive, documented typed plugin hook surface for explicit skill invocation: a pre-invocation decision hook that can allow, block, or require approval with a reason, plus a post-invocation observation hook carrying bounded redacted metadata such as skill name, invocation source, agent/session/run context, dispatch mode, outcome, duration, and failure summary. Keep internal script hooks secondary unless maintainers decide they need parity after the plugin contract is secure.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug reproduction; this is a feature/API gap. Static inspection of the hook catalog, hook runner, public docs, skill command runtime, skill metadata, and related tests gives high confidence that the requested skill execution boundary is absent.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the exact proposed JSON script-hook shape. A typed plugin API with explicit blocking, approval, timeout, and redaction semantics is the safer maintainable solution, especially for the enterprise permission-check use case raised in the discussion.
