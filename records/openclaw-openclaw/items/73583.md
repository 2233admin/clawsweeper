## Summary

Keep open. Current main still lacks the Telegram direct status path, the Telegram suppressed-source fallback wiring, and follow-up failure replies, so this PR is not obsolete. The PR branch has a concrete P2 correctness/privacy bug: its suppressed-source fallback sends raw reply payloads instead of applying the normal dispatcher normalization first.

## What This Changes

The branch adds Telegram direct `/status` delivery, Telegram group/topic suppressed-source reply fallback wiring, a core suppressed-source callback, follow-up failure replies, topic-aware sequential keys, and focused regression tests.

## Best Possible Solution

Keep the PR active, but require the fallback path to preserve normal reply normalization before channel delivery, add a regression for suppressed fallback payloads containing exact silent tokens, and have a maintainer explicitly approve the Telegram group visibility behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Normalize suppressed replies before fallback delivery:** `src/auto-reply/reply/dispatch-from-config.ts:1053-1056`
  - body: The new suppressed-source fallback calls `onSuppressedSourceReply` with the raw payload instead of using the normal dispatcher normalization. An exact `NO_REPLY` final still has sendable text here, and heartbeat stripping, response prefixes, sanitization, and channel transforms are also skipped before Telegram can post the fallback visibly. Please normalize or filter the payload with the same rules used by `createReplyDispatcher` before invoking the channel callback.
