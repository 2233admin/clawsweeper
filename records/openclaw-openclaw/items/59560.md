## Summary

Keep open. Current main still lacks a WhatsApp per-contact policy surface, and the newer related #73740 has already been closed as the narrower duplicate of this broader tracker; 3sken’s comment also preserves a real workaround that shows the same user need remains unsolved in config.

## What This Changes

The issue requests root-level and account-level WhatsApp per-contact rules for `allowInbound`, `allowOutbound`, and `allowAutonomousReply`, with account-level field overrides.

## Best Possible Solution

Design this as a WhatsApp-owned plugin/config change with a typed per-contact policy model, predictable root/account merging, enforcement before inbound side effects, explicit outbound checks, autonomous-reply delivery guards, docs, generated config metadata, and focused tests.

## Reproduction Assessment

Not applicable as a runtime bug report; yes, the product gap is source-verifiable. Current schema and routing code show no per-contact inbound/outbound/autonomous policy, and allowed inbound DMs proceed into normal handling.

## Solution Assessment

Unclear until a maintainer chooses the public config shape. The maintainable direction is hard WhatsApp access control in config/schema/runtime, not memory, prompts, or the dummy/silent-agent workaround described in the related discussion.
