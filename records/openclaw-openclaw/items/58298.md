## Summary

Keep open. Current main has Feishu message-tool send support and can send a generated card from shared `presentation`, but Feishu buttons/selects still degrade to markdown text inside the card and the send/thread-reply schema does not expose a raw `card` payload. The proposed raw `card` field conflicts with the current message-presentation direction, but the underlying agent-authored interactive Feishu card/button capability is still not implemented.

## Best Possible Solution

Keep the issue open, but steer implementation toward the existing `presentation` contract rather than adding a provider-native `card` field to the shared message tool. The likely fix belongs in the Feishu plugin: render `MessagePresentation` button/select blocks into Feishu Schema 2.0 `action` elements with structured `ocf1` values, route callbacks through the existing card-action path, and add tests proving agent-authored buttons return actionable inbound data. If maintainers want raw Feishu schema authoring instead, consolidate that with the open Feishu-specific card tool work rather than widening the generic message schema.
