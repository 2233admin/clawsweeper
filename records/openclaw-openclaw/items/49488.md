## Summary

Keep open. Current main and the latest shipped Android app still use minSdk 31, so this PR is not obsolete or implemented; however, the proposed API 26 support floor still has concrete API 26-28 compatibility gaps and an explicit maintainer concern that this is broader than the current branch covers.

## What This Changes

The PR lowers the Android app minSdk from 31 to 26 and adds API-level fallbacks or gates for dynamic color, foreground service startup, DNS discovery, location, notification listener access, thermal status, and motion-related permission/UI reporting.

## Best Possible Solution

Decide the Android support floor explicitly. If API 26 support is accepted, land a comprehensive compatibility pass that guards every API 29+ surface, keeps runtime command advertisement aligned with actual invocability, updates Android docs to the new floor, and records focused emulator or device proof for the core connection, settings, permissions, discovery, notification, and node-command paths.

  - confidence: 0.86

NodeRuntime still advertises and invokes motion commands from MotionHandler sensor availability, while MotionHandler requires ACTIVITY_RECOGNITION, which is API 29+, so API 26-28 nodes with sensors can expose commands that always fail.
  - confidence: 0.88
