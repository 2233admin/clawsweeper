## Summary

Keep this PR open, but do not merge it as-is. The direction is useful and the discussion shows earlier foreground-service and DNS compatibility issues were addressed, but current main still ships `minSdk = 31`, the PR branch is merge-conflicting, and the latest diff still leaves API 26-28 crashes/capability mismatches plus a location cancellation regression.

## What This Changes

The PR lowers the Android app minSdk from 31 to 26 and adds API-level fallbacks or guards for dynamic colors, foreground service startup, DNS discovery, location, notification access, thermal status, and motion-related UI/permission reporting.

## Best Possible Solution

Decide the Android support floor explicitly. If API 26 support is accepted, rebase this PR and land a comprehensive compatibility pass that guards every API 29+ runtime/UI surface, keeps advertised node capabilities aligned with invocable commands, preserves existing location cancellation behavior, updates Android docs to the new floor, and includes focused API 26, 28, 29, 30, and current-device validation.

## Reproduction Assessment

Yes. Apply the PR support floor, then open Settings on API 26-28 for the unguarded assistant-role path, run a location request that times out before a late callback on API 30+, or connect an API 26-28 device with motion sensors and observe `motion.*` advertised while invocation is gated by an API-29 permission.

## Solution Assessment

No. The direction may be valid and earlier foreground-service/DNS review comments were addressed, but this branch is not the best mergeable solution until it is rebased and the remaining API-level guards, runtime capability alignment, and location cancellation regression are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guard assistant-role APIs below Android Q:** `apps/android/app/build.gradle.kts:66`
  - body: Lowering `minSdk` makes API 26-28 devices supported, but Settings still computes and launches the assistant-role flow through `RoleManager` without an SDK guard. `RoleManager` is API 29+, so opening Settings or resuming it on Oreo/Pie can crash before the user reaches the rest of the app; hide or guard the assistant role block below Q.
  - confidence: 0.88
- **[P2] Preserve cancellation-safe location resume:** `apps/android/app/src/main/java/ai/openclaw/app/node/LocationCaptureManager.kt:107`
  - body: The new API 30+ location branch changes the existing cancellation-safe callback from `cont.resume(location) { _, _, _ -> }` to plain `cont.resume(location)`. That reopens the late-callback timeout crash fixed by the current main implementation; keep the cancellation handler when using `getCurrentLocation`.
  - confidence: 0.84
- **[P2] Apply the motion SDK gate to runtime advertising:** `apps/android/app/src/main/java/ai/openclaw/app/tools/Features.kt:9`
  - body: The new `Features.Motion` gate is only applied to UI and permission reporting. `NodeRuntime` still advertises motion commands from sensor availability while `MotionHandler` requires the API-29 `ACTIVITY_RECOGNITION` permission, so API 26-28 devices with sensors can expose commands that cannot succeed. Gate the runtime availability callbacks as well.
  - confidence: 0.89
