## Summary

Keep this PR open. Current main still treats `workspaceAccess: "none"` as read-only for Docker mounts, local and remote FS bridge writes, memory flush gating, tests, and docs, so the PR’s intended change is not implemented or obsolete. The PR is a plausible focused implementation candidate, but it still needs maintainer review because it broadens sandbox write behavior and should be reconciled with the remote/OpenShell bridge path.

## Best Possible Solution

Review and either land a corrected version of this PR or replace it with a maintainer-owned patch that makes `workspaceAccess: "none"` writable only for the isolated sandbox workspace across Docker mounts, local FS bridge, remote/OpenShell FS bridge, memory flush gating, config-hash recreation, docs, and regression tests. Use the open related behavior threads #37634 and #36525 for product context, and include targeted or live proof that writes stay confined to the sandbox workspace.
