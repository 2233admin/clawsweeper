## Summary

Keep PR #69245 open. Current main does not implement the requested `openai-completions` cache-TTL eligibility, and the PR addresses a real runner gate rather than a cleanup-only change. However, the blanket `modelApi === "openai-completions"` rule likely needs maintainer/product review or narrowing because current docs and plugin APIs treat prompt-cache support as provider/compat-specific, not universal across every OpenAI-compatible endpoint.

## Best Possible Solution

Keep the PR open, but do not merge it as a blanket rule without maintainer review. The best path is to narrow the behavior to verified cache-capable OpenAI-compatible providers or introduce/use an explicit provider/model compat signal, then add focused tests for the supported providers and update prompt-caching/session-pruning docs if the eligibility contract changes.

## Risks / Open Questions

- Blanket eligibility for all `openai-completions` providers may enable TTL pruning for local/proxy providers that do not actually have prompt-cache semantics or prompt-cache-key support.
