## Summary

Keep open. Current main still does not resolve `delivery.to: "env:DISCORD_BACKUPS_CHANNEL"` before cron Discord delivery, so the env-var target remains a literal outbound target and the reported failed/unknown delivery state is still actionable.

## Best Possible Solution

Implement explicit, documented cron delivery target env-var or SecretRef resolution before outbound target normalization, then add tests proving an env-backed Discord channel resolves, delivery returns `delivered: true`, and cron state persists `ok` plus `lastDeliveryStatus: "delivered"`. If env targets are not intended, validation and docs should reject this configuration clearly instead of treating it as a literal Discord target.
