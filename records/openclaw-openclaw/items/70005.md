## Summary

Current main partially mitigates the reported dreaming failure mode by filtering transcript noise and blacklisting chat-scaffolding tags such as `assistant`, but it still does not implement the requested semantic-importance/material-density selection step. REM reflection generation remains count-shaped, and there is no `dreaming.promptStyle` or selection-strategy config surface.

## Best Possible Solution

Keep this open and fix it inside `memory-core` with a conservative internal semantic-importance/material-density filter before REM reflections and Dream Diary narrative inputs. Preserve the existing heartbeat/cron/scaffolding filters, prefer snippets with explicit decisions/preferences/conclusions/unusual events over repeated routine chatter, and only add a public `dreaming.promptStyle` or selection strategy config if maintainers explicitly want that user-facing surface.

## Clownfish Work Prompt

Fix the dreaming material-selection problem described in this issue. Current main filters transcript/scaffolding noise, but REM still builds themes from count-ranked concept tags and can feed low-importance repeated material into Dream Diary narratives.

Expected fix boundary:
- Work primarily in `extensions/memory-core/src/dreaming-phases.ts` and `extensions/memory-core/src/dreaming-phases.test.ts`.
- Add a conservative internal semantic-importance/material-density filter before REM reflections and REM narrative input.
- Prefer snippets that contain explicit decisions, purchases/selections, preferences, conclusions, durable rules, non-obvious findings, or unusual events.
- De-prioritize routine chatter, role labels, heartbeat/cron/system artifacts, generic status, repeated low-signal tags, and short one-off scaffolding.
- Preserve existing session transcript sanitization and the REM tag blacklist.
- Do not add a public `dreaming.promptStyle` or strategy config unless maintainers explicitly request it.
- Do not make a prompt-only change; the reporter already tested that and found it insufficient.
- Changelog is not required unless the PR changes user-facing config/docs/output semantics materially.

Add regression coverage showing that repeated low-signal/scaffolding entries do not dominate REM reflections or narrative data over a high-signal decision/preference/conclusion snippet. Keep the algorithm deterministic and prompt-cache friendly: stable ordering, no model calls, no network, no nondeterministic scoring.

Related refs: this issue, plus the existing REM harness/diagnostic surface around `previewRemDreaming` and `memory rem-harness`.
