## Summary

Keep open. Current main mitigates the concrete `assistant` token and transcript-noise symptoms, but REM material selection is still count-shaped and Dream Diary REM input still comes from candidate truths or the first recent snippets rather than a semantic-importance filter. The issue is recent, unlabeled, non-maintainer-authored, and I found no merged implementation or canonical duplicate that closes the remaining request.

## What This Changes

The issue requests better `/dreaming` material selection so REM/Dream Diary summaries prioritize meaningful decisions, preferences, conclusions, and unusual events over high-frequency transcript or scaffold tokens.

## Best Possible Solution

Add an internal memory-core semantic-importance/material-density selection layer before REM reflections and REM Dream Diary narrative input. It should prefer snippets with decisions, preferences, conclusions, and unusual events over routine role/scaffold chatter or repeated generic tags, preserve existing transcript hygiene and stop-word filters, and avoid adding a public selection config unless maintainers intentionally choose that surface.

## Reproduction Assessment

Yes, for the code-level failure mode: a focused unit case can build repeated short-term recall entries and observe `previewRemDreaming` ranking repeated concept tags by count/strength. A full live reproduction of the reporter's 10-day corpus is not available in this read-only review.

## Solution Assessment

Yes, the narrowest maintainable fix is an internal semantic-density filter/scorer in `memory-core` before REM reflection and narrative input selection. A prompt-only change is not enough because the current frequency-shaped material has already been selected before the narrative prompt runs.
