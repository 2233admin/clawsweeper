## Summary

Keep open. Current main has useful partial coverage for compaction visibility and recovery, including `/compact`, `/status`, optional `notifyUser` notices, Control UI status toasts, compaction checkpoints, restore/branch APIs, plugin hooks, and context-engine extension points. It still does not implement the requested confirmable preview/edit/reject flow or the proposed `auto` / `notify-only` / `require-confirmation` policy surface, so the central product request remains unresolved.

## What This Changes

The issue requests user-visible notifications, preview/edit/approval controls, rollback/re-expansion, configuration, and developer hooks for automatic context compaction, trimming, and message dropping.

## Best Possible Solution

Define a scoped product design for compaction transparency that builds on existing `notifyUser`, checkpoints, hooks, and context-engine seams: add a clear policy model, channel/UI preview and approval UX, timeout behavior for automatic overflow recovery, and docs/tests for the supported surfaces.

## Reproduction Assessment

Not applicable as a feature request. The current-main check is source-backed: docs and config show partial notification/recovery surfaces, while targeted searches show the requested preview/confirmation policy is absent.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but the best maintainable solution should probably be split into a narrow policy/API design plus channel/UI follow-through rather than attempted as one large behavior change.
