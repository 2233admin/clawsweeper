## Summary

Keep this PR open. Current main still does not implement the PR's central lock-loss protection: the watchdog can force-release an in-process session write lock without notifying or aborting the active run, and the embedded runner does not install a guard that rejects SessionManager writes after lock ownership is lost. The discussion shows a plausible core agents fix with follow-up commits addressing bot review, not a ClawHub-only workaround or an implemented-on-main change.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best core direction is a stable lock-loss path in `src/agents/session-write-lock.ts` and `src/agents/pi-embedded-runner/run/attempt.ts`: notify the active run before watchdog force release, abort both the run controller and active session, reject later transcript writes after lock ownership is lost, cover rewrite paths, and add focused regression tests for watchdog callbacks, lock ownership checks, lost-lock write rejection, cleanup ordering, and active-session abort behavior.

## Risks / Open Questions

- The PR depends on dependency-private `SessionManager` methods (`_persist` and `_rewriteFile`); maintainers should verify the current `@mariozechner/pi-coding-agent` contract or prefer a stable upstream/public seam if available.
- Per-write fsync on hot transcript paths may have latency or throughput impact and should be scoped or measured before merge.
- Lock-loss teardown ordering still deserves review, especially around pending tool-result flushes, active session abort, resource cleanup, and final session lock release.
- Recent main changes reduce some long-held-lock pressure, but they do not provide equivalent post-watchdog lock-loss notification or write rejection.
