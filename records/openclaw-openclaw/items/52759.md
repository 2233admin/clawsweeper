## Summary

Keep open. Current main still ships tool-loop detection disabled by default, so the central loop-detection change is not implemented, but this PR is dirty against main, overlaps several open default-on PRs, and mixes in a read-offset change whose original run-abort issue is already handled at the adapter boundary. The PR needs maintainer reconciliation and a rebase rather than cleanup closure.

## What This Changes

The PR flips `tools.loopDetection.enabled` to true by default, updates related tests/docs/config metadata and generated baselines, and wraps adaptive `tools.read` execution to return an out-of-range-offset diagnostic.

## Best Possible Solution

Land a focused, rebased loop-detection default decision only after preserving run-id scoping and aligning tests, schema help, generated SHA artifacts, and current docs. If maintainers want read-specific EOF clamping beyond the adapter diagnostic behavior, handle it as a separate read-tool PR with partial-output regression coverage.
