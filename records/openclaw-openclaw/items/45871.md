## Summary

Keep this PR open. Current main still caps `session.agentToAgent.maxPingPongTurns` at 5 in runtime clamping, Zod validation, generated schema/help, and public docs, so the PR's intended 0-20 behavior is not implemented. The PR discussion usefully clarifies that this is the `sessions_send` reply-back loop, not delegation depth, and the proposed cap aligns with the existing loop detector critical threshold, but the branch needs a current-main refresh before it is mergeable.

## Best Possible Solution

Keep this PR open for maintainer review unless a maintainer chooses a different canonical PR. Rebase it onto current main, raise both the runtime clamp and Zod schema ceiling to 20, regenerate/update config schema/help artifacts, update current English docs only, drop obsolete localized-doc edits, add focused tests for accepting 20 and rejecting or clamping above 20, and include a brief performance/cost caution if maintainers want the docs to call out higher-turn usage.

## Risks / Open Questions

- The PR is stale against current main: it edits `docs/zh-CN/**`, but localized docs are not maintained in this repository and no `docs/zh-CN` directory exists on current main.
- The PR does not update `src/config/schema.base.generated.ts`, so schema/help consumers would still advertise `maximum: 5` unless generated artifacts are refreshed.
- Raising the opt-in cap to 20 can increase agent runtime, model context, and tool cost on constrained hosts; the default staying at 5 helps, but docs/help text may need a short caution.
- There are other open attempts and issue threads for the same cap increase, so maintainers should pick one canonical implementation path rather than let several stale PRs drift.
