## Summary

Keep this PR open. Current main still caps `session.agentToAgent.maxPingPongTurns` at 5 in runtime clamping, config validation, generated schema/help, and public docs, so the PR's intended 0-20 behavior is not implemented yet. The PR discussion usefully clarifies this is the `sessions_send` reply-back loop and that 20 aligns with the existing loop detector critical threshold, but the branch is stale against current main and needs a focused refresh before maintainer review.

## Best Possible Solution

Keep this PR open for maintainer review or ask the contributor to refresh it. The best implementation is a focused current-main patch that raises the runtime clamp and Zod schema ceiling to 20, regenerates config schema/help artifacts, updates current English docs only, drops obsolete localized-doc edits, and adds boundary tests for accepting 20 and rejecting or clamping values above 20. A brief docs/help caution about higher-turn cost and resource use would address the review discussion without changing the default behavior.

## Risks / Open Questions

- The PR is stale against current main: `docs/tools/index.md` is now much shorter and no longer contains the exact hunk shown in the PR context, while localized `docs/zh-CN/**` docs are not maintained in this repo per `docs/AGENTS.md`.
- The branch does not update `src/config/schema.base.generated.ts`, so schema/help consumers would still advertise and enforce a generated maximum of 5 unless generated artifacts are refreshed.
- The current tests exercise ping-pong behavior with `maxPingPongTurns: 2` and loop detection thresholds, but there is no focused boundary test proving config value 20 is accepted and values above 20 are rejected or clamped.
- Raising the opt-in cap to 20 can increase runtime, model context, and cost on constrained hosts; keeping the default at 5 limits blast radius, but docs/help may need a short caution.
