## Summary

Current main still does not implement a user-facing configurable exec approval timeout. The PR remains a plausible implementation candidate, but it is security-sensitive, diverged from current main, and still needs maintainer decisions on the public config shape, validation bounds, docs/default wording, propagation, and tests. The provided discussion also includes an April 26 maintainer-authored Codex review explicitly keeping this PR open for follow-up.

## Best Possible Solution

Keep this PR open for maintainer review, or explicitly supersede it under #51287 if maintainers choose a different public config shape. A landing path should choose the public key, add bounded validation, update generated schema/help/docs, thread the resolved timeout through agent, gateway, node, and transport waits, align the documented default with current main, and add focused propagation and expiry tests.

## Risks / Open Questions

- Configurable exec approval timeouts affect a security guardrail; unbounded positive integers could leave approvals pending far longer than intended.
- The PR text/help describes a 120000 ms default, while current main and docs use a 1,800,000 ms, 30-minute default.
- The branch has diverged from current main; for example, current src/cli/nodes-cli/register.invoke.ts is now a short generic node invoke command rather than the older approval path referenced by the PR context.
- The PR context does not show focused tests for schema acceptance, bounds, config precedence, agent/gateway/node propagation, expiry behavior, or transport timeout alignment.
