## Summary

Current main still lacks a user-facing configurable exec approval timeout. The PR targets a real remaining request, but it is not implemented on main, the proposed key is absent from the strict config schema, the runtime still sends the fixed approval timeout, and #51287 remains open for the same product decision.

## Best Possible Solution

Keep this PR open for maintainer review or explicitly supersede it under #51287 if maintainers choose a different public key. A landable fix should pick one supported config surface, use bounded validation, align help/docs with the current 30-minute default, thread the resolved timeout through agent, gateway, node, and transport waits, and add focused propagation and expiry tests.

## Risks / Open Questions

- Exec approval timeout is security-sensitive; the PR's proposed schema uses a positive integer without an upper bound, which could leave host-exec approvals pending for unexpectedly long periods.
- The PR body/help text says the default is 120000 ms, but current main and v2026.4.26 use 1800000 ms.
- Current `src/cli/nodes-cli/register.invoke.ts` no longer matches the PR patch shape, so landing this branch as-is would need a rebase and renewed runtime review.
- No current evidence shows focused tests for schema acceptance, bounds, config precedence, agent/gateway/node propagation, expiry behavior, and transport-timeout alignment.
