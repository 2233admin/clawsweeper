## Summary

Keep this open. Current main still routes OpenRouter replay policy through the passthrough-Gemini hook, which does not enable strict9 tool-call-id sanitization for Mistral-family model IDs, while direct Mistral strict9 behavior already exists and docs still describe model-id based Mistral strict9 behavior.

## What This Changes

Restore strict9 tool-call-id sanitization for Mistral/Codestral model IDs routed through proxy providers such as OpenRouter after provider replay policy moved into plugins.

## Best Possible Solution

Add a provider-owned replay-policy path, likely through a shared replay helper composed by OpenRouter, that recognizes Mistral-family proxy model IDs and selects strict9 tool-call-id sanitization while preserving existing OpenRouter Gemini thought-signature behavior and adding focused regression coverage.

## Repair Work Prompt

Fix the current OpenRouter/proxy Mistral strict9 regression. The observable bug is that Mistral-family model IDs routed through OpenRouter or similar proxy providers do not select `sanitizeToolCallIds: true` with `toolCallIdMode: "strict9"`, causing Mistral backends to reject tool-call IDs. Keep the fix in the current provider-owned replay-policy/plugin SDK surface; do not resurrect `src/agents/provider-capabilities.ts` or move provider-specific policy back into core. Prefer a shared replay helper/composition if it keeps OpenRouter Gemini passthrough behavior and Mistral-family strict9 behavior in one tested contract. Preserve existing OpenRouter Gemini thought-signature cleanup and non-Mistral negative behavior. Add regression tests for OpenRouter `mistral-large-latest`, `mistralai/devstral-2512:free`, and a Codestral-style ID resolving strict9, plus a non-Mistral OpenRouter model that does not use strict9; keep direct Mistral strict9 tests green.
