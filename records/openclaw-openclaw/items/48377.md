## Summary

Keep this PR open. Current main still does not implement create-with-content for `feishu_doc`: the create schema has no `content` field, and the create runtime branch only calls `createDoc`. The PR remains a focused implementation candidate for the linked open Feishu bug, but it still needs maintainer review around tests, empty-content semantics, response shape, and whether `writeDoc` is the right helper for freshly-created documents.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for the linked Feishu bug. The maintainer-quality path is to add `content` to the create schema, populate the new document through a tested helper, define empty-content behavior, return a clean create response, avoid unnecessary Feishu API calls if practical, and update the Feishu doc skill documentation to match the final schema and permission behavior.

## Risks / Open Questions

- The PR currently needs a focused regression test proving `feishu_doc` create with `content` creates the document and inserts converted markdown blocks in one tool call.
- The proposed implementation uses a truthiness check for `p.content`, so maintainers should decide whether an explicitly provided empty string or whitespace-only string should be treated as provided, rejected, or equivalent to omitted content.
- Reusing `writeDoc` is plausible but clears the newly-created document first; the PR strips `blocks_deleted` from the response, but the extra clear/list calls and final response shape still need maintainer review.
- The Feishu doc skill documentation is already stale around create parameters and requester permission wording, so the final supported create schema should be documented after the behavior is settled.
