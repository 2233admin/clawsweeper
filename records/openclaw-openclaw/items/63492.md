## Summary

Keep open. Current main still treats session persistence as local filesystem state: `session.store` is a path, `sessions.json` is loaded and atomically rewritten on disk, and transcripts are resolved to local `*.jsonl` files. The prior ClawSweeper review and the later maintainer comment both point in the same direction: this is useful but still requires core design work, not cleanup closure.

## What This Changes

Request a pluggable external session storage backend so multiple OpenClaw gateway instances can share session metadata and transcripts without sticky routing.

## Best Possible Solution

Design a core storage contract that covers both session metadata and transcript history, keep the filesystem backend as the default, add config/schema/docs and migration/doctor support, prove one multi-instance-safe backend, then expose a stable plugin-facing backend registration API if additional stores should live outside core.

## Reproduction Assessment

Not applicable as a feature request, but the current gap is high-confidence from source and docs inspection: current main resolves a filesystem `sessions.json` path and local transcript files rather than a replaceable external backend.

## Solution Assessment

Unclear because there is no proposed patch here. The requested direction is plausible, but the maintainable route is a versioned core design first rather than adding ad hoc Redis/MongoDB fields directly to existing path-based helpers.
