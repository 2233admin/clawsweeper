## Summary

Keep this issue open. Current main still does not provide first-class browser-tab A2UI canvas node support: there is no explicit `web` node platform policy, device-less `role: "node"` sessions still reject even with shared auth, browser-origin node auto-approval is excluded, and the related #68497 action-bridge request is narrower than browser node registration and pairing/auth design.

## Best Possible Solution

Keep this open as the canonical design and implementation request for browser tabs acting as A2UI canvas node surfaces. A good fix should add an explicit `web` node platform policy with canvas/A2UI commands, define a secure browser node auth/pairing story for reverse-proxied deployments, add regression tests for browser Origin handling and node-role auth, and document the supported web canvas-node pattern. Coordinate with #68497 for browser action delivery, but do not treat it as a replacement.

## Risks / Open Questions

- The requested shared-token or trusted-proxy browser node bypass is security-sensitive: without a scoped device identity, bootstrap token, origin check, and command-scope model, a browser-origin client could register an unauthorized node surface.
- A first-class `web` platform should avoid inheriting the broad `unknown` fallback accidentally; canvas-only defaults would be safer than copying all unknown commands.
- #68497 may be useful follow-up work, but treating it as a duplicate would lose the distinct platform allowlist, node registration, and pairing/auth work requested here.
