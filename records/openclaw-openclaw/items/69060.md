## Summary

Keep open. Current main still lacks the proposed stable `x-openclaw-session` header, and the PR is a narrow, plausible implementation for the QMD/session-warm part of the related latency report. It should not be cleanup-closed; the remaining work is normal maintainer review of the public HTTP session contract, with docs/regression coverage before landing.

## What This Changes

This PR changes `src/gateway/http-utils.ts` so OpenAI-compatible chat requests can use a new `x-openclaw-session` header to derive an agent-scoped stable `openai-session:<hint>` session key between the OpenAI `user` field and random UUID fallback.

## Best Possible Solution

Land the additive scoped-session header if maintainers accept the API shape, with tests for header priority, same-hint reuse, agent scoping, and docs that distinguish scoped hints from the full session-key override. Keep #68920 open for the separate light-context/voice-mode prompt assembly work.

## Reproduction Assessment

Yes. Current main can be checked from source and with a focused `/v1/chat/completions` test: repeated calls with the proposed header should produce the same `agent:<id>:openai-session:<hint>` key, while main currently has no branch that can do that. I did not run tests because this review was explicitly read-only.

## Solution Assessment

Yes for the QMD/session-warm subproblem. An additive agent-scoped header is narrower than changing `x-openclaw-session-key` semantics or requiring every client to set the OpenAI `user` field, but it should ship with docs/tests and does not solve the broader light-context latency request.
