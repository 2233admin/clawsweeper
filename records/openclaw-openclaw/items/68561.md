## Summary

Keep open. Current main has fixed adjacent cron configuration boundaries and already provides `cron run` for manual triggering, but the specific `ANNOUNCE_SKIP` ergonomics gap remains: exact assistant `ANNOUNCE_SKIP` replies are still suppressed globally, and the cron CLI/service/docs do not warn when a main-session `systemEvent` prompt tells the agent to produce that token.

## What This Changes

The issue asks for cron documentation, diagnostics, validation, and possible dry-run behavior around `ANNOUNCE_SKIP` in main-session cron payloads.

## Best Possible Solution

Keep the issue open and make a narrow cron ergonomics fix: document that `ANNOUNCE_SKIP` is an internal exact-output control token, warn or diagnose when a main-session cron `systemEvent` instructs the agent to reply with it, and point users to `cron run` plus run history for manual testing instead of adding a separate dry-run command unless maintainers choose that product surface.
