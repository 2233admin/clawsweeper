## Summary

Keep open. Current main still has the reported Telegram typing outage surface: the low-level keepalive loop has no tick-level failure breaker, the auto-reply typing controller rethrows start failures without a max-failure threshold, and `sendTypingTelegram` still uses the normal channel retry profile unless a caller overrides it. The issue discussion also corrected the framing: normal Telegram reply dispatch has a higher-level guard, but expensive cosmetic retries and bypass paths remain unresolved, with #55886 and #56096 needing maintainer triage before spawning duplicate repair work.

## What This Changes

The issue asks for Telegram typing keepalive and `sendChatAction` failures during network outages to be bounded so cosmetic typing calls cannot keep retrying and make the gateway unresponsive.

## Best Possible Solution

Land one canonical fix that keeps typing indicators best-effort: bound repeated keepalive failures, make Telegram typing fail cheaply without changing normal message delivery retries, preserve the existing 401 suspension behavior, and add regression coverage for both shared typing lifecycle and Telegram-specific retry/cooldown behavior.

## Reproduction Assessment

Yes. The issue provides concrete network-outage steps and production log signatures, and current main still contains the source-level path that can repeatedly invoke expensive Telegram typing failures under outage conditions.

## Solution Assessment

Yes, the narrow maintainable direction is to bound typing keepalive failures and make Telegram typing retries cosmetic. The immediate path should be maintainer review or expansion of #55886 together with #56096 rather than a separate duplicate repair PR.
