## Summary

Keep this open. Current main has partial typing safeguards, but the reported gap is not fully fixed: the generic keepalive loop still has no error-aware breaker, the auto-reply typing controller uses that loop with rethrowing starts, and Telegram standalone typing still inherits the normal channel retry profile unless a caller/config overrides it. The two related fix PRs were closed unmerged.

Keep this issue open and land one focused bug-fix PR that makes typing keepalive failures bounded and makes Telegram typing best-effort/cheap to fail, while preserving normal Telegram message delivery retries and existing typing mode behavior.

## Clownfish Work Prompt

Fix the remaining typing keepalive/network-failure bug tracked by https://github.com/openclaw/openclaw/issues/45759. Current main still has no error-aware breaker in `createTypingKeepaliveLoop`, `src/auto-reply/reply/typing.ts` uses that loop with `rethrowOnError: true`, and `sendTypingTelegram` still inherits the normal Telegram retry profile. Implement a focused fix that: (1) ensures async keepalive tick failures cannot become unhandled interval rejections and stop or trip after a small number of consecutive failures, while preserving restart/cleanup semantics; (2) makes Telegram typing (`sendChatAction`) best-effort and cheap to fail by default, without changing `sendMessageTelegram`/media delivery retry behavior; (3) preserves the existing Telegram 401 sendChatAction backoff circuit breaker; (4) adds focused tests for failure counting/reset/stop behavior and Telegram typing retry defaults. Related closed unmerged attempts are #45836 and #47018; use them as context, not as code to blindly copy. Add a user-facing changelog fix entry with appropriate reporter/commenter thanks, and do not change unrelated channel retry policy or global Telegram timeout semantics unless a narrow per-typing seam is required.
