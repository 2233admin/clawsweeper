## Summary

Keep open. Current main still has a one-shot remote media fetch on the media-understanding path used by Discord audio preflight; the report’s `(0/1)` interpretation is slightly imprecise because that log count is attachment successes over selected attachments, but there is no retry/backoff before the audio provider or CLI transcription step.

## What This Changes

Add retry-with-backoff for transient Discord audio/media-understanding fetch failures before transcription, while failing fast on non-recoverable errors.

## Best Possible Solution

Keep this issue open and implement a narrow shared retry policy for media-understanding remote attachment fetches, preferably around the shared `fetchRemoteMedia`/`MediaAttachmentCache` boundary so Discord preflight and the normal audio pass benefit together. Retry only transient network/read failures and likely recoverable 5xx responses with bounded backoff+jitter, fail fast for 4xx, maxBytes, SSRF blocks, and caller aborts, then add focused tests plus a user-facing changelog fix entry crediting @gabrielexito-stack.
