## Summary

Keep open. Current main still uses a one-shot remote media fetch for normal media-understanding attachments, and open PR #74553 is the closing implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue requests bounded retry/backoff for transient remote-media fetch failures before Discord audio attachments enter the transcription pipeline.

## Best Possible Solution

Review and land PR #74553, or replace it with an equivalent narrow shared fix at the fetchRemoteMedia/MediaAttachmentCache boundary with focused tests for transient fetch/read failures, 5xx retry, and fail-fast carve-outs.

## Reproduction Assessment

Yes. A source-level reproduction is clear: route a remote audio attachment through MediaAttachmentCache.getBuffer, make the first fetch fail transiently, and current main makes one fetchRemoteMedia call before surfacing MediaFetchError.

## Solution Assessment

Yes. The shared fetch/cache boundary is the narrowest maintainable place to fix this because it covers Discord audio preflight and normal media-understanding consumers without adding Discord-specific retry policy.
