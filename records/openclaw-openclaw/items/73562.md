## Summary

Keep this open. Current main still has the closed-stdin prompt failure shape, and the linked implementation PR is open and unmerged, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

`plugins uninstall <id>` should fail cleanly when confirmation stdin closes before an answer, while preserving piped `y`/`n` input and `--force`.

## Best Possible Solution

Review and land PR #73566 or an equivalent narrow fix that makes the shared prompt helper settle on close/EOF, has `plugins uninstall` convert that specific condition into exit 1 with interactive or `--force` guidance, and adds focused prompt plus uninstall regression tests. Keep the stale-entry cleanup in #73551 separate unless a maintainer intentionally broadens the same PR.

## Reproduction Assessment

Yes. Current main reaches the vulnerable `promptYesNo()` await in the non-forced uninstall path, and the underlying Node readline behavior reproduces exit 13 for closed stdin while preserving piped input behavior.

## Solution Assessment

Yes at the design level. Handling EOF in the shared prompt primitive and catching that specific condition in `plugins uninstall` is narrower and safer than a blanket non-TTY gate because piped `y`/`n` answers should continue to work.
