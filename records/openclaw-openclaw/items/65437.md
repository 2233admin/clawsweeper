## Summary

Keep open. Current main still has the Slack room/group early-drop path reported here: `/elevated` is detected as a control command, Slack authorizes that command gate only from Slack account/channel allowlists, and a blocked room/group message returns before `tools.elevated.allowFrom.slack` or the elevated fallback hook can be evaluated. The related Discord/tiered-auth reports are useful context but do not supersede this Slack-specific bug.

## Best Possible Solution

Keep this open and fix the Slack room/group `/elevated` path narrowly. Either let `/elevated` messages from senders in `tools.elevated.allowFrom.slack` reach `resolveElevatedPermissions`, or register/add a Slack elevated-specific fallback/authorizer without widening access to other control commands. Add a regression test for a Slack channel/group sender listed only in `tools.elevated.allowFrom.slack`, and update the public elevated/Slack docs to describe the resulting behavior.
