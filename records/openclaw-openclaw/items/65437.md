## Summary

Keep open. Current main still blocks Slack room/group text control commands before elevated authorization can evaluate `tools.elevated.allowFrom.slack`, and the related Discord/tiered-auth items do not supersede this Slack-specific authorization bug.

## What This Changes

The issue asks for Slack `/elevated` activation to work, or to have a documented fallback, when the sender is listed only in `tools.elevated.allowFrom.slack`.

## Best Possible Solution

Fix the Slack room/group `/elevated` path narrowly so a sender listed in `tools.elevated.allowFrom.slack` can reach elevated permission resolution without gaining access to unrelated Slack control commands. Add a Slack regression test for the room/group text-command path and update the public elevated or Slack docs to describe the Slack behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: Slack room/group message, channel itself allowed, no `channels.slack.allowFrom` and no `channels.slack.channels.<id>.users`, sender listed only in `tools.elevated.allowFrom.slack`, and text `@bot /elevated on`; current `prepareSlackMessage` computes `commandAuthorized=false` and returns `null` before `resolveElevatedPermissions`.

## Solution Assessment

Unclear until a maintainer chooses the seam. The safest direction is an elevated-specific Slack authorizer or a channel elevated fallback that applies only to `/elevated`, not adding `tools.elevated.allowFrom.slack` as a general Slack command authorizer.
