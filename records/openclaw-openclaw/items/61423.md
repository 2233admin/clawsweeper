## Summary

Keep open. Current main only exposes the existing toolbar New Session button as a confirmed `/new` reset path, while this PR still adds useful named session creation through `sessions.create`; however the latest patch still has a concrete reconnect bug where it can derive the new session’s agent from stale session-list state.

## What This Changes

The PR adds a Control UI chat-header New Session button, a naming modal, session-key helpers/tests, and a `sessions.create` flow that switches to the created session.

## Best Possible Solution

Land a named session-creation UI that uses `sessions.create`, preserves the entered label, switches to the created key, and derives the target agent only from fresh connection-specific state or a server-confirmed session context.

## Reproduction Assessment

Yes. By code inspection, reconnect or switch gateways while `sessionsResult` still contains an old `agent:<id>:...` row, then confirm the PR's New Session modal before a fresh sessions list is loaded; the PR can accept the stale row as trusted agent context.

## Solution Assessment

No. The PR is on the right product path, but the current implementation should be tightened so session-list trust is cleared, versioned, or refreshed for the active connection before creating the new session.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Require fresh session context before creating sessions:** `ui/src/ui/app.ts:146-148`
  - body: `resolveNewSessionAgentId` trusts `sessionsResult` whenever it contains the active key, but `connectGateway` does not clear or version that list across reconnects and the Chat tab does not necessarily reload sessions on hello. In that window, this can create the new session under an agent from a previous connection instead of the active gateway/session context. Clear/version `sessionsResult` on reconnect or require a fresh sessions load for the current client before accepting membership here.
