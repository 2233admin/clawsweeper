## Summary

Keep open. Current main has the `beforeDeliver` seam and inbound `message_sending` wiring from the related work, but I did not find a canonical fail-closed structured-output validator at the native dispatcher to delivery handoff; structured payload handling is still split between dispatcher hooks, outbound normalization, and channel/plugin-specific structured payload detection.

## What This Changes

Request a universal pre-send validation gate for structured operator-facing reply payloads before native reply dispatcher delivery, while preserving simple replies and existing message hooks.

## Best Possible Solution

Define a shared structured `ReplyPayload` validation contract for core-owned fields such as `presentation`, `interactive`, and delivery metadata, decide how plugin-owned `channelData` participates, then run that validation before dispatcher delivery for native replies and any equivalent outbound paths with focused tests for cancellation, simple-text preservation, and hook ordering.

## Reproduction Assessment

Not applicable as a traditional runtime bug. Code inspection gives a high-confidence path audit: current main has pre-delivery hook seams and tests, but no canonical structured payload validator to exercise with a failing structured payload.

## Solution Assessment

Unclear. The dispatcher `beforeDeliver` seam is a plausible narrow location, but the best fix needs an agreed structured payload contract and should avoid channel-by-channel policy while preserving plugin-owned payload compatibility.
