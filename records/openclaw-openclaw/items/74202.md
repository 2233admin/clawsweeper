Full review comments:

- **[P1] Add the custom model fields to the manifest schema:** `extensions/amazon-bedrock/openclaw.plugin.json:61`
  - body: `normalizeBedrockAuthConfig()` always returns `awsBedrockCustomSelected`, and the exported `BedrockSetupOptions` type also allows `awsBedrockCustomModelBaseId`, but the manifest keeps `additionalProperties: false` and does not declare either field. Any normalized config round-tripped through manifest validation will fail or lose data, so add these properties or remove them from the exported contract.
  - confidence: 0.94
- **[P1] Mark Bedrock credential fields as sensitive:** `extensions/amazon-bedrock/openclaw.plugin.json:104-122`
  - body: The new API key, secret key, and session token hints are missing `sensitive: true`. The setup wizard only avoids unmasked prompts when this flag is set, and secret/redaction tooling relies on sensitive metadata, so these credentials can be handled as normal config fields if merged as-is.
  - confidence: 0.9
- **[P2] Align the surface with a shipped pi-ai resolver:** `docs/providers/bedrock.md:356-358`
  - body: The docs now say this surface matches pi-ai's resolver contract, but current OpenClaw pins `@mariozechner/pi-ai` without these Bedrock auth fields and the linked upstream pi-ai PR is closed unmerged. Merging now would expose setup/config keys that the runtime dependency does not consume.
  - confidence: 0.86
- **[P3] Preserve explicit empty Bedrock regions consistently:** `extensions/amazon-bedrock/bedrock-auth-config.ts:91`
  - body: All other defaults in this helper use nullish coalescing, but `awsRegion` uses `||`, so an explicitly provided empty string is replaced with `us-east-1`. If empty strings should be rejected, let schema/validation reject them; otherwise use `??` here to preserve caller-provided values consistently.
  - confidence: 0.74
