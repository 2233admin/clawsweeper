## Summary

Keep open. Current main still does not provide native audio attachments from the `read` tool into the active model turn. The read wrapper and upstream read dependency are image-only, the pinned session content contract still only allows text/image content blocks, and the OpenAI/Google/Anthropic tool-result transports forward text/images but not audio. Existing audio features cover inbound media understanding, STT/TTS, realtime voice, and related headless Google `@file` work, but they do not solve `read("*.mp3")` returning a multimodal audio block to the model.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is to add a stable audio content-block contract for read/tool results, teach the read path to detect and guard supported audio MIME types, forward audio only through capable providers such as Google/Gemini and OpenAI, return a clear unsupported-model/tool-result message otherwise, and add focused regression coverage across read plus provider conversion. Treat #46601/#46607 as related but narrower Google headless `@file` work, not the canonical fix for this issue.
