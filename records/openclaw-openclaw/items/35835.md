## Summary

Current main still does not implement audio-file multimodal parity for the read tool. The read wrapper and provider tool-result transports remain text/image-oriented, while existing audio support covers STT, TTS, inbound media understanding, realtime voice, and the narrower Google headless @file work rather than read("*.mp3") returning native audio to the active model turn.

## What This Changes

Add read tool support for audio files as native multimodal attachments, with clear unsupported-model handling.

## Best Possible Solution

Keep this open for maintainer design. The best path is to add a stable audio content-block contract for read/tool results, teach the read path to detect and guard supported audio MIME types, forward audio only through capable providers such as Google/Gemini and OpenAI, return a clear unsupported-model message otherwise, and add focused regression coverage across read plus provider conversion. Treat #46601/#46607 as related but narrower Google headless @file work, not the canonical fix for this issue.
