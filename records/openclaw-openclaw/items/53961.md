## Summary

Keep open. Current main still lacks deliveryStatus tracking and structured failure diagnostics, so the underlying work is not obsolete, but this draft PR has correctness issues around JSON output on strict delivery errors and false failure reporting for intentional hook cancellations.

## What This Changes

This PR modifies command delivery code and tests to track deliveryStatus, log diagnostic delivery failures, include delivery status in --json --deliver output, and cover delivery success, preflight, thrown-error, partial-failure, zero-result, and no-payload paths.

## Best Possible Solution

Keep this PR as a useful implementation candidate, but sequence it with #57843, preserve the single parseable --json contract on strict delivery errors, and consolidate the JSON status surface with #57755 before maintainer review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Emit JSON before rethrowing delivery failures:** `src/agents/command/delivery.ts:330`
  - body: The changed `if (opts.json && !deliver)` suppresses the existing early JSON envelope for `--json --deliver`. In strict delivery mode, `deliverOutboundPayloads` rethrows before the later status envelope can run, so scripts can receive no parseable JSON even though the CLI docs reserve stdout for JSON output.
  - confidence: 0.9
- **[P2] Do not mark hook cancellations as failed delivery:** `src/agents/command/delivery.ts:426`
  - body: `deliverySucceeded = results.length > 0` treats an all-cancelled `message_sending` hook as a failed send. Current outbound delivery intentionally continues without adding a result when hooks return cancel, so expected cancellations such as thread ownership would produce false failure status/logs.
  - confidence: 0.86
