## Summary

Keep this PR open, but do not merge it as-is. Current main does not already implement the assistant-conclusion freshness gate, so the work is not obsolete; however, the latest PR head still has a concrete target-matching bug in the config-key freshness heuristic, and the branch is stale relative to the current prompt-build helper surface.

## What This Changes

The PR adds an opt-in assistant-conclusion freshness gate, replay metadata for diagnostic tool results, prompt-build integration, and focused agent/session tests for plugin-state and config-key freshness checks.

## Best Possible Solution

Rebase the PR onto current main, keep the existing prompt-build hook/queued-injection/heartbeat behavior intact, fix config target extraction to prefer the most specific config key while ignoring filename tokens like `openclaw.json`, and add focused regression tests for same-root different-plugin paths plus `openclaw.json` prompt wording.

## Reproduction Assessment

Yes. A focused helper check can reproduce the main defect: ask about `plugins.entries.qqbot.enabled` with recent evidence for `plugins.entries.other.enabled`; the PR's root collapse plus nested-path matcher can incorrectly classify that unrelated evidence as fresh.

## Solution Assessment

No. The feature direction is plausible and default-off, but this implementation's target-hint heuristic is not precise enough for the stale-evidence problem it is trying to solve; the safer fix is to parse and compare the most specific config target before allowing reuse.
