## Summary

Keep open. Current main still caps active remote `configRemote` and explicit non-loopback gateway probe targets to 1500ms even when the CLI receives a larger `--timeout`, and the existing tests still assert that cap. The earlier local-loopback fix does not cover this remote path.

## What This Changes

Fix `gateway probe` so healthy remote gateway targets do not false-negative because active non-loopback probes ignore the caller-supplied timeout budget.

## Best Possible Solution

Remote probe targets should have a timeout policy that matches the user-facing budget for active configured/explicit remote gateways, while preserving intentional short caps for inactive local loopback and SSH setup paths unless maintainers choose to document and expose a different effective per-target budget.
