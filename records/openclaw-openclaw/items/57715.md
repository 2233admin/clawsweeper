## Summary

Keep open. Current main still has the reported hot-path shape: `listSessionsFromStore()` builds full `GatewaySessionRow` objects before applying `search`, `activeMinutes`, and `limit`; each row can trigger transcript fallback reads; and child-session resolution still scans the store per row. The linked fix PRs #59436 and #68878 are open/unmerged, so the issue is not implemented or superseded yet.

## Best Possible Solution

Keep #57715 open as the canonical performance bug until a focused PR lands. The best code path is to pre-filter, sort, and apply `limit` before full row construction wherever semantics allow, build a request-scoped child-session index instead of scanning the store per row, and add regression coverage proving pre-limit rows do not trigger transcript fallback reads. After one of #59436 or #68878 is merged and verified on main, close this issue from that PR.
