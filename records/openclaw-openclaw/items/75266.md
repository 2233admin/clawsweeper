## Summary

Keep this PR open, but it should not merge as-is: the boolean-false heartbeat direction is useful, yet the branch drops the existing `heartbeat.skipWhenBusy` config surface and adds a local-only gate script. Current `main` still documents, validates, types, and consumes `skipWhenBusy`, so the PR needs a rebase/fix before maintainer approval.

## What This Changes

The PR adds `false` support for `agents.defaults.heartbeat` and per-agent `heartbeat` across config schemas, generated schema/types, heartbeat runtime helpers, WhatsApp heartbeat handling, tests, and a new gate script.

## Best Possible Solution

Preserve the new `heartbeat: false` capability while rebasing onto current `main`, keeping all existing heartbeat object fields such as `skipWhenBusy` in the Zod schema, exported type, generated schema, tests, and runtime behavior. Drop the local-only gate script or replace it with a repo-portable validation path if the project actually wants checked-in gates.

## Reproduction Assessment

Yes for the blocking regression: current `main` already has a schema test path that parses `heartbeat: { timeoutSeconds: 45, skipWhenBusy: true }`, and the PR’s strict object schema omits `skipWhenBusy`, so that valid config would be rejected after the branch change. The local-only gate issue reproduces by running the added script anywhere other than the contributor’s machine path.

## Solution Assessment

No, not as written. Supporting `false` is a reasonable fix direction, but the implementation must be rebased to preserve the existing `skipWhenBusy` config contract and remove the hard-coded local gate before it is the narrowest maintainable solution.
