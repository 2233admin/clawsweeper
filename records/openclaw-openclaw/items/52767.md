## Summary

Keep open. Current main still dispatches `gateway:startup` on each gateway startup, the bundled `boot-md` handler still invokes `runBootOnce` for every configured agent, and `runBootOnce` still runs whenever `BOOT.md` exists and is non-empty. The recent boot-session isolation mitigates the long-lived main-session replay concern, but it does not add durable idempotency or reset semantics for the repeated execution reported here.

## Best Possible Solution

Keep the issue open for maintainer handling. Decide whether `BOOT.md` is a repeatable startup checklist or an at-most-once boot hook; if it is at-most-once, add a durable per-agent workspace guard with explicit reset/re-run semantics and regression tests proving repeated `gateway:startup` does not re-run the same completed `BOOT.md`. If it should remain repeatable, update the public docs and UX to make that contract clear and steer one-time initialization to `BOOTSTRAP.md`, while preserving the current boot-session isolation.

## Risks / Open Questions

- Closing now would leave a recent production stall report unresolved while current main can still re-run a non-empty `BOOT.md` on each configured gateway startup.
- A naive sentinel could break the currently documented repeatable startup-check workflow, so the fix needs explicit reset/re-run semantics and a per-agent workspace or content/config-generation key.
- The current fresh boot-session and mapping-restore mitigation reduces old main-session history replay risk, but repeated boot work, model/tool cost, and startup pressure remain possible.
