## Summary

Keep open. Current main still has the false-negative Windows status path because runtime reads preflight with a broad `schtasks /Query` before the task-specific lookup; this PR remains relevant, but it needs maintainer review and rebase/port handling before merge.

## What This Changes

The PR removes the broad Windows `schtasks /Query` preflight from daemon runtime status reads, handles thrown task-scoped query failures through fallback logic, and adds focused startup-fallback regression coverage.

## Best Possible Solution

Land or port the narrow daemon fix on current main: Windows runtime status should skip only the broad availability preflight, keep install/stop/restart availability checks intact, preserve Startup fallback and unknown-status handling for failed or thrown task-scoped queries, and keep regression coverage for the single-call and thrown-query cases.

Yes. A focused mocked reproduction is clear: make the global `schtasks /Query` preflight fail while the task-scoped `/Query /TN <task> /V /FO LIST` would succeed or while a Startup fallback exists; current main returns before reaching the task-scoped read.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because it changes only runtime status reads, leaves lifecycle preflights in place, and routes thrown task-scoped query errors through the existing fallback/non-zero branch; it should be rebased or ported onto current main before merge.
