## Summary

Keep open. Current main has stronger Volcengine provider/catalog support, including the `volcengine-plan` auth alias and documented DeepSeek model rows, but I did not find source, docs, tests, or release evidence proving the reported Volcengine Lite UUID-style key 401 is fixed. The current runtime still accepts any non-empty key, attaches it unchanged to the Volcengine provider configs, and sends OpenAI-compatible requests through the generic OpenAI client path.

## Best Possible Solution

Keep this issue open until a maintainer can reproduce against a Volcengine Lite account using the UUID-style key. The best fix path is a focused Volcengine completion/auth regression, preferably with a mocked transport plus an optional live smoke; then either document the required key, endpoint, and model pairing or add a Volcengine-specific auth/request adaptation in `extensions/volcengine` if the vendor requires something beyond the current OpenAI-compatible bearer-key path.
