## Summary

Keep this issue open. Current main still resolves a non-auto pinned harness id before the config-backed policy and gives that pinned policy `fallback: "none"`, so a configured same-scope `fallback: "pi"` is bypassed on pinned/plugin-runtime paths. The current tree has improved the `claude-cli` alias path, but that does not resolve the pinned harness fallback gap.

## What This Changes

The issue asks OpenClaw to either honor configured agent runtime fallback for pinned explicit harnesses or document and warn that pinned runtimes always fail closed.

## Best Possible Solution

Define the pinned-runtime contract explicitly. For config-created pins, either carry the same-scope fallback policy into selection, preferably by storing or passing the policy at pin time, or document that persisted pins always fail closed; pair the behavior with focused tests and a clear warning when PI fallback can change the billed backend.

## Reproduction Assessment

Yes. A focused reproduction is to select or run with `agentHarnessId: "codex"`, no registered `codex` harness, and config `agents.defaults.agentRuntime: { id: "codex", fallback: "pi" }`; current main reaches the pinned resolver first and throws the disabled-PI-fallback error.

## Solution Assessment

Unclear. Option A is plausible but not automatically the safest fix because it changes a billing-sensitive fail-closed brake and may conflict with session pinning semantics; a maintainer should choose between policy-carrying fallback support and stronger docs/warnings.
