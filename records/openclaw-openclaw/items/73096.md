## Summary

Keep this issue open. Current main still resolves a pinned `agentHarnessId` before the config-backed runtime policy and hardcodes that pinned policy to `fallback: "none"`, while the docs/config path allow same-scope `fallback: "pi"` for explicit plugin runtimes. The report is valid, but the right fix needs maintainer judgment because changing it can alter billing-sensitive runtime fallback behavior.

## What This Changes

The issue asks OpenClaw to either honor configured agent runtime fallback for pinned harness ids or make the fail-closed pin behavior explicit in docs and diagnostics.

## Best Possible Solution

Define the pinned-runtime contract explicitly: either a pinned explicit runtime with same-scope `fallback: "pi"` falls back to PI with a clear warning, or persisted pins always fail closed and the docs/schema/diagnostics say so plainly. Add focused coverage for `agentHarnessId` plus configured fallback and keep CLI backend aliases out of embedded-harness pinning.

## Reproduction Assessment

Yes. A focused unit check can call harness selection or an embedded run with `agentHarnessId: "codex"`, no registered `codex` harness, and config fallback `pi`; current main reaches the pinned policy first and throws the disabled-PI-fallback error.

## Solution Assessment

Unclear. A mechanical parameter pass-through would satisfy the config knob, but it may not be the safest behavior because the existing hardcoded `none` is a fail-closed billing brake; the maintainable solution needs an explicit product decision plus tests and warning/docs updates.
