## Summary

Keep this issue open. Current main still resolves a non-auto pinned `agentHarnessId` before the config-backed policy and returns `fallback: "none"`, so a configured same-scope `fallback: "pi"` can still be bypassed on pinned/plugin-runtime paths. The current tree has improved the `claude-cli` alias path, but that does not resolve the pinned embedded-harness policy gap.

## What This Changes

The issue asks OpenClaw to either honor configured embedded/agent runtime fallback for pinned harness ids or make the fail-closed pinned behavior explicit enough to avoid surprising billing-sensitive routing.

## Best Possible Solution

Define the pinned-runtime contract explicitly. Either carry fallback policy with config-derived pins and add a clear warning/docs/tests for intentional PI fallback, or keep pins fail-closed and update docs/schema/status wording so users know `fallback` does not apply after a harness id is pinned.

## Reproduction Assessment

Yes. A focused reproduction is to select or run with `agentHarnessId: "codex"`, no registered `codex` harness, and config `agents.defaults.agentRuntime: { id: "codex", fallback: "pi" }`; current main reaches `resolvePinnedAgentHarnessPolicy` first and throws the disabled-PI-fallback path.

## Solution Assessment

Unclear. Honoring the configured fallback is plausible, but not automatically the best fix because the current hardcoded `none` is also the fail-closed brake for a billing-sensitive backend switch; maintainers should choose the runtime contract before a narrow patch lands.
