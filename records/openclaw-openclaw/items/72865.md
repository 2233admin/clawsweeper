## Summary

Keep open. Current main has related Anthropic assistant-prefill hygiene, but the Amazon Bedrock Converse path still does not appear to strip a normal trailing assistant/prefill message before sending a thinking-enabled Claude request; adjacent open PRs cover related assistant-tail cases but are not merged proof for this Feishu/Bedrock regression.

## What This Changes

The issue reports a 2026.4.25 regression where a new Feishu/Lark session using Bedrock Claude Opus 4.6 fails on the first user message because the outgoing conversation is rejected as assistant message prefill.

## Best Possible Solution

Add a canonical assistant-tail guard for Bedrock Claude thinking requests, preferably by composing the existing shared Anthropic prefill payload wrapper into the Amazon Bedrock provider or by landing an equivalent generic pre-provider guard that preserves tool-use turns. If the Feishu/Lark greeting is persisted as a normal assistant row, the channel/plugin side should also mark it as transcript-only or avoid leaving it as replayable model history, with regression coverage for Bedrock Converse plus the Feishu reset/new-session path.

## Reproduction Assessment

Yes: the report gives a concrete version/provider/model/channel sequence, and the linked Feishu/Lark issue describes the same greeting-as-assistant-tail mechanism. I did not live-run Bedrock credentials here, but current source inspection shows the Bedrock path lacks the prefill stripping used by native Anthropic/Cloudflare routes.

## Solution Assessment

Unclear as a single-source fix because the failure spans Bedrock request shaping and Feishu/Lark greeting persistence. A Bedrock wrapper using the existing shared prefill stripper is the narrowest OpenClaw-side fix, while the overlapping open PRs should be reviewed for whether they also cover the first-message Feishu case without over-dropping valid assistant turns.
