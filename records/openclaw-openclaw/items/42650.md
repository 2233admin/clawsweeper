## Summary

Keep this issue open. Current main has memory search, promotion, dreaming/backfill review output, active-memory recall injection, LanceDB memory tools, and the memory-wiki companion layer, but it still lacks the requested first-class operator flows for reviewing, editing, non-destructively forgetting/tombstoning, superseding, and resolving conflicts over canonical durable memory records.

## Best Possible Solution

Keep this issue open and implement it with the memory owner modules alongside #42646, #42648, and #42651. The best path is a canonical durable memory record/status model plus operator-facing review/list, edit, non-destructive forget/tombstone or superseded transitions, and conflict-resolution flows exposed through CLI/RPC or UI, with docs and tests. Memory-wiki should remain a provenance-rich companion layer unless maintainers explicitly decide it owns active-memory maintenance semantics.
