## Summary

Keep this issue open. Current main still closes the long-lived WhatsApp inbound listener through `sock.ws?.close?.()` via `closeInboundMonitorSocket`, the shared WhatsApp socket close helper also raw-closes the underlying WebSocket, and the pinned Baileys version exposes `end(error)` with broader socket cleanup. The paired fix PR #52443 is closed unmerged and no open PR currently references this issue.

## Best Possible Solution

Fix this in the WhatsApp plugin by widening the inbound and shared close-helper socket types to include `end(error?: Error)`, preferring `sock.end(new Error("OpenClaw listener close"))` or a context-appropriate error when available, preserving `sock.ws?.close?.()` as a compatibility fallback, and updating teardown tests to cover both Baileys-capable and fallback sockets. Use PR #52443 as prior art, but implement against the current split helper structure rather than the older inline monitor close path.
