## Summary

Current main still does not keep configured run-side typing presence alive through a long quiet active run until run completion plus reply dispatch idle. The remaining gap is narrow and already has an open targeted implementation PR, so this issue should stay open for maintainer review rather than be closed.

## What This Changes

Request to add lifecycle-owned active-run presence so already-enabled typing, reaction, or progress surfaces stay alive until the run is complete and reply dispatch is idle, without changing default disclosure or final-only delivery behavior.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #71828 with an equivalent narrow fix that refreshes the internal run-side typing TTL on active keepalive ticks, preserves stale-presence cleanup, avoids new default progress/disclosure surfaces, and adds regression coverage for active runs that outlive the TTL.
