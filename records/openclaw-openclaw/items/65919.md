## Summary

Current main still does not keep configured run-side typing presence alive through a long quiet active run until run completion plus reply dispatch idle. The issue should stay open because the remaining gap is concrete and PR #71828 is already open as the targeted implementation candidate.

## What This Changes

Request to keep already-enabled active-run presence surfaces such as typing, reactions, or channel progress alive until both the run is complete and reply dispatch is idle, without changing default disclosure or final-only delivery behavior.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #71828 with an equivalent narrow fix that refreshes internal run-side typing TTL on active keepalive ticks, preserves stale-presence cleanup, avoids new default progress or disclosure surfaces, and adds regression coverage for active runs that outlive the TTL.
