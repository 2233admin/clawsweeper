## Summary

Keep open. Current main still has the reported pre-Ready failure surface: Discord `/vc join` passes `channels.discord.voice.daveEncryption=false` through to `@discordjs/voice`, waits for `VoiceConnectionStatus.Ready` with a fixed 15s initial timeout, and destroys the connection on any `entersState` abort; the existing DAVE receive recovery only runs after a connection is Ready, so it does not address a Hello-before-Ready close. Related open reports #39825 and #63098 overlap, while merged #41536 covers later receive/reply recovery rather than this earlier handshake path.

## What This Changes

Investigate and fix Discord `/vc join` aborting before `Ready` after the voice WebSocket `Hello`, especially when `channels.discord.voice.daveEncryption` is disabled.

## Best Possible Solution

Keep the report open and decide the Discord voice join policy explicitly: either make DAVE-disabled joins work reliably, reject or warn on unsupported non-DAVE joins with a clear diagnostic, or add a documented configurable Ready timeout/retry path. The fix should stay in the Discord plugin voice manager/config/docs/tests and be validated against the related handshake-abort reports separately from post-Ready receive recovery.
