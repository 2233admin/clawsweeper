## Summary

Keep open. Current main still treats Telegram reactions as context notifications only: it enqueues standard emoji reaction system events but does not wake an agent turn, normalize Telegram custom emoji, or expose per-emoji reaction semantics. The issue is also paired with open PR #47687, which uses closing syntax for this work, and open bug #64752 tracks the narrower wake-trigger failure, so this should not be closed before that implementation path is resolved.

## What This Changes

Request first-class Telegram reaction input support so reaction-only events can wake the agent, normalize standard and custom emoji, express per-emoji operator semantics, and provide durable observability.

## Best Possible Solution

Review #47687 against the narrower #64752 repro and either land a maintained Telegram-channel implementation or replace it with a narrower PR that adds a documented, typed reaction semantics/wake contract, standard and custom emoji normalization, targeted tests, and docs updates without hardcoding site-specific emoji policy into core.
