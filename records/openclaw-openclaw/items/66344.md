## Summary

Keep this PR open for maintainer review. Current `main` still lacks table-aware block-streaming chunk protection, but the PR patch has two concrete correctness defects that can still split valid or incrementally streamed Markdown tables; #73981 is also open as a competing replacement path, so this needs maintainer choice rather than automatic cleanup.

## What This Changes

The PR adds a Markdown table span parser, combines table spans with code-fence spans in `EmbeddedBlockChunker`, and adds parser/chunker unit coverage for table-safe streaming chunks.

## Best Possible Solution

Land one canonical table-aware block-streaming fix that keeps incomplete streaming tables buffered, protects valid no-edge-pipe Markdown tables, adds `subscribeEmbeddedPiSession` integration coverage, updates the streaming docs/changelog as needed, and then close or supersede the duplicate implementation PRs after the canonical SHA lands.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Defer table-tail emission while streaming:** `src/agents/pi-embedded-block-chunker.ts:455`
  - body: When `drain({ force: false })` receives a growing table that already exceeds `maxChars`, `parseTableSpans` can report the current buffer tail as `table.end`. Returning that end emits partial rows immediately, so later `append()` calls continue the same table in a new message and the table-splitting bug remains for incremental streams.
  - confidence: 0.9
- **[P2] Protect no-edge-pipe table rows:** `src/markdown/table-spans.ts:76-86`
  - body: Valid two-column Markdown rows without outer pipes, such as `A | B`, contain only one pipe. The level-1 expansion requires `countPipes(...) >= 2`, so it protects only the delimiter row and leaves the header/body rows available for chunk splits.
  - confidence: 0.88
