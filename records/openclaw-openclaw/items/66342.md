## Summary

Close as implemented on current main. The Slack auth crash class is already handled by shipped gateway supervision plus the Slack Bolt startup-auth fix, and the proposed clean-return patch is no longer the best implementation because it would hide terminal Slack failures behind the channel manager’s clean-exit path.

## What This Changes

The PR adds a Slack monitor regression test and changes Slack socket startup auth/max-attempt failures from thrown errors to clean returns.

## Best Possible Solution

Keep the shipped gateway-supervision and Slack Bolt startup-auth implementation. If maintainers still want Slack auth failures to pause without any retry, that should be an explicit terminal channel failure/no-restart state rather than a clean monitor return.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve terminal Slack failure status:** `extensions/slack/src/monitor/provider.ts:502-509`
  - body: These returns make `monitorSlackProvider` resolve successfully for permanent auth failures and exhausted startup retries. The current gateway manager treats a non-manual resolved channel task as a clean exit, records `channel exited without an error`, and schedules the normal auto-restart path, so an invalid Slack token loses its concrete auth error and still restarts instead of being marked as a terminal Slack failure. Keep a handled failure signal or explicit no-restart state before exiting.
  - confidence: 0.88
