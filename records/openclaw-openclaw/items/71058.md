## Summary

Keep open. Current main still models Microsoft Teams as one top-level bot credential and one runtime webhook/JWT context, while the issue discussion describes a real multi-agent identity gap. The related duplicate was closed in favor of this issue, and the open Microsoft tracker PR lists this work as unresolved rather than implementing it.

## What This Changes

Add multi-bot Microsoft Teams support so one Gateway can configure multiple Azure/Teams bot identities and route them to specific agents.

## Best Possible Solution

Implement this inside the Microsoft Teams plugin using the existing account-map pattern: keep the current top-level config as the implicit default account, add accounts/defaultAccount support, select the correct account from authenticated Teams/Bot Framework context, carry accountId through inbound routing, outbound replies, setup/status, and docs, and leave agent routing in bindings.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from current-main source and docs: Teams has one credential/config/runtime account, and the multi-agent docs omit Teams from the account-map channel list.

## Solution Assessment

No for the exact proposed array plus agentBinding shape. The safer maintainable direction is to reuse channels.msteams.accounts/defaultAccount and the existing bindings resolver, with a Teams-specific auth/account-selection design reviewed by maintainers.
