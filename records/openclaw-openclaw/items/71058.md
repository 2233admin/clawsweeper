## Summary

Keep open. Current main still exposes Microsoft Teams as one top-level bot account, and the issue discussion describes a real multi-agent identity gap rather than a stale or already-implemented request. The closed related feature #72876 is duplicative, while open draft PR #74163 is a maintainer tracker, not an implementation that supersedes this work.

## What This Changes

Add support for multiple Microsoft Teams/Azure bot identities on one OpenClaw Gateway, with each bot account routable to a specific agent.

## Best Possible Solution

Implement this inside the bundled Microsoft Teams plugin using the existing account-scoped channel pattern: add channels.msteams.accounts plus defaultAccount, preserve today’s top-level config as the implicit default account, select the account from validated Bot Framework bot identity/webhook context, carry accountId through routing, replies, pairing, setup/status/docs, and add generic SDK seams only if the plugin cannot express the behavior through current contracts.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature gap. The limitation is high-confidence from current-main inspection: Teams has one credential/config/runtime account, and the multi-agent docs omit Teams from the multi-account channel list.

## Solution Assessment

No for the exact proposed array plus agentBinding shape. The maintainable path should reuse the existing accounts/defaultAccount and bindings pattern already used by multi-account channels, with a Teams-specific account-selection/auth design reviewed by maintainers.
