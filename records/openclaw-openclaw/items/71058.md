## Summary

Current main does not support multiple Microsoft Teams bot identities on one Gateway. The Teams bundled plugin still models `channels.msteams` as a single top-level config, setup resolves only the default account, and runtime startup/probe paths use one credential set and one webhook listener. The issue is recent, contributor discussion is minimal but coherent, and the existing account-binding/plugin surfaces suggest this is a real implementable Teams plugin enhancement rather than something to auto-close.

## Best Possible Solution

Keep open. Implement this in the bundled `extensions/msteams` plugin by adding a backwards-compatible multi-account config shape such as `channels.msteams.accounts` plus `defaultAccount`, migrating/retaining the current single-object form as the default account, starting/probing each enabled Teams bot account, scoping credentials/JWT validation/webhook routing/state by account, and documenting how to bind `msteams:<accountId>` to agents with the existing agent routing commands.

## Risks / Open Questions

- A correct implementation will touch Teams webhook auth/audience validation, Graph/delegated token storage, status/probe, outbound routing, pairing/allowlists, docs, and setup, so it should not be treated as a small schema-only change.
- The issue proposes an `agentBinding` field, but OpenClaw already has account-scoped agent routing; maintainers should decide whether to reuse `agents bind --bind msteams:<accountId>` instead of adding a parallel binding concept.
