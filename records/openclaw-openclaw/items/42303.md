## Summary

Current main still does not implement this PR's intended opt-in Linuxbrew support for Docker builds. The PR remains relevant to #14593, but it is stale against the current Docker setup layout and is not safe to merge as-is because the proposed Dockerfile fetches and executes Homebrew HEAD without an integrity policy, pollutes non-brew images with unconditional Homebrew env/PATH, and edits the root docker-setup.sh body that is now only a wrapper.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but do not merge it as-is. The best path is to choose one canonical fix for #14593: safe opt-in Linuxbrew support in the current Dockerfile plus scripts/docker/setup.sh with conditional runtime env handling, docs, tests, Docker proof, and installer integrity policy; or an alternate product path such as apt fallback or hiding brew-only installs in Linux containers. Once that lands on main, close or supersede the remaining implementation PRs against the shipped fix.

## Risks / Open Questions

- Merging this PR as-is would add a Docker build-time network execution path from Homebrew HEAD without a pinned revision, checksum, or explicit installer integrity policy.
- The proposed Dockerfile sets Homebrew ENV/PATH for every image, including non-brew builds where those paths do not exist.
- The proposed shell quoting around the fetched Homebrew installer is likely to fail when the installer script contains single quotes.
- The PR edits root docker-setup.sh, but current main routes setup through scripts/docker/setup.sh, so the setup wiring is stale.
- The OPENCLAW_SKIP_BUILD failure path in the PR points users at a top-level openclaw build command that the current CLI command registry does not provide.
- Closing or merging this PR without resolving #14593 would leave Docker users with brew-only skill installs that still fail or are hidden only by alternative unmerged PRs.
