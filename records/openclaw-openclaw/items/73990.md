## Summary

Keep open. Current main still only derives status/session context usage from provider usage, cost, or model snapshots; a contentful transcript with no usable usage telemetry can still fall through as null or zero, so local OpenAI-compatible sessions can keep showing misleading 0% context.

## What This Changes

Request a fallback that estimates local/OpenAI-compatible session context usage from transcript content when provider usage telemetry is absent or zero.

## Best Possible Solution

Add a generic transcript-derived prompt-token fallback in the Gateway/session usage extraction and status path: keep valid provider telemetry authoritative, estimate visible transcript text only when no positive context total exists, and cover local/openai-completions no-usage transcripts with focused regression tests.

## Reproduction Assessment

Yes. The existing `src/gateway/session-utils.fs.test.ts` fixture creates a non-empty user/assistant transcript with no usage telemetry and asserts that the extractor returns null; downstream status/session code then drops missing or zero token snapshots.

## Solution Assessment

Yes. A narrow extractor/status fallback is the best direction because provider telemetry should remain authoritative whenever present, while local no-telemetry transcripts need a conservative generic estimate rather than provider-specific special cases.
