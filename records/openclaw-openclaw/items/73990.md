## Summary

Keep open. Current main still has the reported gap: transcript fallback only works when a usable usage/model snapshot exists, while a contentful transcript with no provider usage telemetry still returns no snapshot and downstream status/session rows cannot recover a nonzero context value. The related open #75357 is narrower transport work and does not supersede this broader no-telemetry fallback request.

## What This Changes

Request a conservative transcript-derived context estimate for local/OpenAI-compatible sessions when provider usage telemetry is missing so status does not stay at 0% or empty usage.

## Best Possible Solution

Add a generic transcript fallback that preserves valid provider usage first, then estimates a conservative prompt/context token count from visible transcript content only when no positive usage-derived snapshot exists, with regression coverage for contentful no-telemetry local/OpenAI-compatible transcripts.

## Reproduction Assessment

Yes. Current main has a deterministic fixture that writes user and assistant transcript content with no usage telemetry and asserts the extractor returns null; the status/session paths then drop the missing snapshot before formatting context usage.

## Solution Assessment

Yes. The requested extractor/session-status fallback is the narrowest maintainable solution because provider transports and usage normalization remain authoritative whenever they report usable token telemetry, while content-only transcripts get a conservative last-resort estimate.
