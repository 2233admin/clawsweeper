## Summary

Keep open. Current main still treats a contentful transcript without usable provider usage telemetry as having no transcript usage snapshot, so local/OpenAI-compatible sessions can still collapse to zero-ish context displays instead of getting a conservative transcript-derived estimate.

## What This Changes

The issue asks OpenClaw to estimate context usage from transcript text for local/OpenAI-compatible sessions when provider usage telemetry is absent or zero.

## Best Possible Solution

Implement a generic Gateway/status transcript fallback that preserves valid provider usage first, then estimates a conservative prompt/context token count from visible transcript text only when no positive usage-derived context total exists, with focused regression coverage for no-telemetry local/OpenAI-compatible transcripts.

## Reproduction Assessment

Yes. Current main has a deterministic unit-test shape: a user/assistant transcript with no usage telemetry returns `null` from `readLatestSessionUsageFromTranscript`, and the status/session row paths then have no positive fallback total to display.

## Solution Assessment

Yes. The narrowest maintainable fix is in the transcript extractor/session-status fallback path; provider adapters should remain authoritative when they return usable token telemetry, and the fallback should only cover absent or non-positive telemetry with transcript content present.
