## Summary

Keep open. Current main still has the no-telemetry gap described here: transcript fallback works when assistant usage telemetry exists, but a contentful local/OpenAI-compatible transcript with no usable usage snapshot still returns null and is then dropped by status/session display paths. The related closed context-usage issues covered telemetry extraction and metadata fallback, but not transcript-derived estimation when telemetry is absent.

## What This Changes

The issue requests a conservative transcript-derived context estimate for local/OpenAI-compatible sessions when provider usage telemetry is missing so status/session context displays do not stay at zero.

## Best Possible Solution

Add a generic transcript usage fallback in the Gateway/status path that preserves valid provider-reported usage first, then estimates a conservative prompt/context token count from visible transcript content only when no positive usage-derived context total exists, with regression coverage for local/OpenAI-compatible no-telemetry transcripts.

## Reproduction Assessment

Yes. The current fixture in `src/gateway/session-utils.fs.test.ts` creates a nonempty user/assistant transcript with no usage telemetry and asserts null, and the status/session-row paths then drop that missing snapshot before formatting context usage.

## Solution Assessment

Yes. A narrow extractor/status fallback is the maintainable direction because provider transports and OpenAI-compatible normalization should remain authoritative whenever they report usable token telemetry.
