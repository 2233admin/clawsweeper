## Summary

Keep open. Current main still persists auto-enable-derived enabled flags during gateway startup and doctor repair. A previous #67436 fix appears to suppress one self-write reload loop by recording the startup write hash, but it does not implement the issue's requested runtime-only auto-enable behavior or keep openclaw.json canonical.

## Best Possible Solution

Keep this issue open and implement a runtime-only auto-enable path: gateway startup should not call writeConfigFile for autoEnable.config, doctor should report auto-enable reasoning without marking it as a persisted repair, and plugin/channel loading should consume the auto-enabled runtime snapshot plus autoEnabledReasons. Add regression tests proving openclaw.json is not rewritten when channels.whatsapp.enabled, channels.telegram.enabled, or plugins.entries.openai.enabled are absent, while the configured runtime components still activate.

## Risks / Open Questions

- The restart-churn symptom may be reduced or absent on current main because #67436 added a startup write-hash guard, but the persistence behavior reported in #70333 is still present.
- Changing auto-enable to runtime-only touches startup, doctor, plugin activation, and restrictive plugins.allow behavior, so it needs focused regression coverage for configured built-in channels and provider plugins.
