## Summary

Keep open. Current main still writes inferred auto-enable changes from startup and doctor paths; the existing hash guard mitigates self-induced reload churn, but it does not implement the requested runtime-only canonical config behavior, and current tests still assert the persistence path.

## What This Changes

Make channel/provider auto-enable runtime-only so startup and doctor do not reintroduce inferred `enabled` flags for configured WhatsApp, Telegram, and OpenAI entries in `openclaw.json`.

## Best Possible Solution

Decide the canonical config contract first. If runtime-only auto-enable is the intended direction, split startup and doctor into an in-memory activation overlay plus read-only diagnostics, while leaving explicit user enables and true legacy migrations as the only paths that rewrite config.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-backed: a configured channel/provider without `enabled` still produces `applyPluginAutoEnable()` changes that startup and doctor pass into write-capable mutation paths; existing tests assert those writes.

## Solution Assessment

No. The current hash guard is a good mitigation for the reload loop, but it is not the narrowest solution for the requested non-persistent canonicalization; a runtime activation overlay would better match the report if maintainers choose that contract.
