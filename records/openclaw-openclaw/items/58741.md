## Summary

Keep open. Current main has generic session reset and maintenance controls that can mitigate Feishu session growth, but it does not implement the requested Feishu-specific idle/max-age lifecycle surface with Discord-style channel overrides and enforced expiry.

## Best Possible Solution

Keep the issue open and narrow it. For ordinary Feishu session accumulation, document or recommend `session.resetByChannel.feishu` together with `session.maintenance` in the Feishu/session docs. For Discord-style parity, add a Feishu `threadBindings` schema/UI/docs surface at root and account scope, enforce expired Feishu binding records before runtime routing, and add focused tests for idle timeout and max-age expiry.

## Risks / Open Questions

- The issue mixes two related scopes: ordinary Feishu session-store retention and Discord-style thread-bound session lifecycle. Maintainer clarification would help choose whether this should be a docs/config guidance issue or a Feishu thread-binding parity implementation.
- Generic `session.resetByChannel.feishu` plus `session.maintenance` may be enough for some deployments, but it does not provide a Feishu-specific hard max-age override or automatic expiry for every Feishu session key.
