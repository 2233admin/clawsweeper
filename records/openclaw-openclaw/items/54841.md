## Summary

Keep open. Current main still does not enforce the configured Feishu webhookPath: the handler records the configured path for status and rate-limit identity, then processes requests through generic guards, body reading, signature validation, challenge handling, and dispatch without comparing the incoming URL path. PR #54904 is open and unmerged with the targeted fix, so this issue should remain open until that PR or an equivalent Feishu plugin patch lands.

## What This Changes

The issue asks Feishu webhook mode to reject requests whose URL path does not match the configured webhookPath before parsing, signature validation, or dispatch.

## Best Possible Solution

Land PR #54904 or an equivalent Feishu plugin patch that normalizes the configured webhookPath, parses the incoming request pathname, returns 404 for mismatches before body parsing/signature validation/dispatch, preserves query-string acceptance on the configured path, and adds focused regression coverage.
