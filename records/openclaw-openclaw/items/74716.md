## Summary

Keep open. Current main does not implement node-hosted MCP sessions, the PR is an active implementation with a protected `maintainer` label, and the latest head fixes the earlier pending-close race but still has a timeout path that can abandon a node-hosted MCP open after the gateway has already sent the open frame.

## What This Changes

The PR adds additive gateway protocol/session plumbing for node-hosted MCP servers, including node pairing approval/storage, registry session routing, a NodeMcpClientTransport, generated Swift protocol models, tests, and docs.

## Best Possible Solution

Keep the protocol slice on this branch, make every sent MCP open closeable on timeout as well as on caller cancellation, then validate the targeted gateway/protocol tests, protocol generation checks, and exact-head CI before maintainer merge review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Close MCP opens that time out:** `src/gateway/node-registry.ts:347-350`
  - body: `openMcpSession()` sends `node.mcp.session.open`, but the timeout handler only deletes the pending and active maps before resolving `TIMEOUT`. After that, `closeMcpSession()` cannot send `node.mcp.session.close` because the active entry is gone, so a slow node can finish launching a permission-sensitive MCP process with no live gateway client. Send a close frame on timeout, or keep the session closeable until the node acknowledges shutdown.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff introduces privileged node-hosted MCP session plumbing, and the open-timeout path can abandon a node-side session after the gateway sent the open request.

Concerns:

- **[medium] Timed-out MCP opens are not closed on the node:** `src/gateway/node-registry.ts:347`
  - body: When an MCP open times out, the gateway drops registry state without sending `node.mcp.session.close`; because process ownership and macOS permissions live on the node side, a slow launch can continue without a client consuming or closing it.
  - confidence: 0.86
