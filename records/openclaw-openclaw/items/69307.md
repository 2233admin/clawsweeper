## Summary

Current main still lacks a modifying plugin hook between assistant reply text generation and TTS directive parsing. Existing hook surfaces either observe model output, claim/own dispatch, or run during outbound delivery after TTS has already been applied, so the requested per-message voice-routing API remains valid and open.

## Best Possible Solution

Keep this issue open and design an additive, documented modifying plugin hook that runs after assistant reply text is available and before TTS directive parsing on the default reply paths. The implementation should update SDK types, hook runner wiring, docs, and targeted tests, while preserving existing static TTS config precedence and directive stripping behavior.

## Risks / Open Questions

- Adding this hook creates a new public plugin contract; hook naming, event fields, merge order, timeout behavior, and result semantics should be decided intentionally.
- The hook would expose and allow modification of assistant reply text before TTS, so non-bundled plugin access should be gated consistently with raw conversation hooks or another explicit trust policy.
- The TTS path has final, tool, block-streaming, and synthetic TTS-only reply cases; a partial implementation could miss the reported per-message routing path.
