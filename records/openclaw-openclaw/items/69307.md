## Summary

Current main still does not provide a modifying plugin hook between assistant reply generation and TTS directive parsing. The existing hook surfaces are either too early, observational, claiming/dispatch-owning, or downstream of TTS, so the requested per-message voice-routing seam remains valid.

## What This Changes

The issue requests an additive modifying plugin hook that can transform assistant reply text after LLM output and before TTS directive parsing so plugins can inject per-message voice directives or skip TTS.

## Best Possible Solution

Keep this open and implement an additive, documented modifying hook, named `before_tts` or a maintainer-approved equivalent, that runs after reply text is available and before `maybeApplyTtsToReplyPayload` calls `parseTtsDirectives`. The fix should update SDK hook types, runner wiring, default reply dispatch integration, docs, and targeted tests without changing existing static TTS config precedence or `message_sending` delivery behavior.

## Repair Work Prompt

Implement the feature requested in openclaw/openclaw#69307. Add an additive modifying plugin hook, preferably named `before_tts` unless existing maintainer conventions point to a better name, that runs after assistant reply text is available and before `maybeApplyTtsToReplyPayload`/`parseTtsDirectives` on default auto-reply paths. The hook should allow sequential text transformation for per-message voice directive injection and a clear per-message TTS skip result. Update `src/plugins/hook-types.ts`, `src/plugins/hooks.ts`, public SDK exports/baseline as needed, and wire the hook in `src/auto-reply/reply/dispatch-from-config.ts` before TTS application while preserving existing TTS auto/off behavior, modelOverrides policy, directive stripping, reply_dispatch behavior, and downstream message_sending semantics. Add focused tests for hook registration/merge behavior and dispatch ordering proving injected `[[tts:voice=...]]` reaches TTS and skip avoids TTS for that message. Update `docs/plugins/hooks.md` and `docs/tools/tts.md`; add a changelog entry only if repo policy treats this plugin API addition as user-facing. Do not introduce bundled-plugin-specific IDs or make core depend on a specific TTS provider.
