## Summary

Current main still lacks a modifying plugin hook between assistant reply text generation and TTS directive parsing. Existing hooks are either too early, observational, claiming/dispatch-owning, or downstream of TTS, so the requested per-message voice-routing seam remains valid and should stay open.

## What This Changes

The issue requests an additive modifying plugin hook that can transform assistant reply text after LLM output and before TTS directive parsing so plugins can inject per-message voice directives or skip TTS.

## Best Possible Solution

Add a documented, additive modifying hook, named `before_tts` or a maintainer-approved equivalent, that runs after reply text is available and before TTS directive parsing on default reply paths. Keep static TTS config precedence, directive policy, `reply_dispatch`, and downstream `message_sending` behavior intact.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature request. The missing capability is high-confidence from source inspection: there is no registered hook, runner method, dispatch integration, or docs entry for a pre-TTS text transform.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable solution: VISION favors extending the plugin API when useful plugin behavior cannot be built with current seams. A ClawHub plugin alone cannot implement this until core exposes the missing pre-TTS hook.

## Repair Work Prompt

Implement the feature requested in openclaw/openclaw#69307. Add an additive modifying plugin hook, preferably named `before_tts` unless existing maintainer conventions point to a better name, that runs after assistant reply text is available and before `maybeApplyTtsToReplyPayload` / `parseTtsDirectives` on default auto-reply paths. The hook should allow sequential text transformation for per-message voice directive injection and a clear per-message TTS skip result. Update `src/plugins/hook-types.ts`, `src/plugins/hooks.ts`, public SDK exports/baseline as needed, and wire the hook in `src/auto-reply/reply/dispatch-from-config.ts` before TTS application while preserving existing TTS auto/off behavior, modelOverrides policy, directive stripping, `reply_dispatch` behavior, and downstream `message_sending` semantics. Update `docs/plugins/hooks.md` and `docs/tools/tts.md`; add a changelog entry only if repo policy treats this plugin API addition as user-facing. Do not introduce bundled-plugin-specific IDs or make core depend on a specific TTS provider.
