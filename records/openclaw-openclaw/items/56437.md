## Summary

Keep this issue open. Current main has strong generic model fallback behavior, prompt/context injection, fallback logging/trajectory events, and cron payload fallback plumbing, but it does not implement the requested native `fallbackRunbook` config field or fallback-only system prompt injection. The related model-ID validation request is also only partially covered: cron payload `model` is validated, while configured fallback refs are still treated as explicit operator intent and are not prevalidated against provider catalogs at config load/doctor time.

## Best Possible Solution

Keep this open and treat it as a core failover/config design item. A good implementation would add a typed, documented fallback-only prompt field to default and per-agent model config, inject it only for non-primary fallback attempts while preserving normal workspace context, include tests for embedded, CLI, cron, and per-agent fallback paths, and add doctor/models-status diagnostics that warn on fallback refs missing from known provider catalogs without blocking valid dynamic provider setups.

## Risks / Open Questions

- A fallback runbook would be advisory prompt text, not a hard security control; hard constraints such as tool access and config mutation should remain enforced by tool policy, permissions, and config guards.
- Provider catalog validation can be incomplete or offline for some plugins/providers, especially dynamic OpenRouter-style catalogs, so doctor/config validation likely needs warnings or probe-backed diagnostics rather than assuming every unknown fallback ID is invalid.
