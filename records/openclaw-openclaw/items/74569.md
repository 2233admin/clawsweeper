## Summary

Current main already implements the Matrix restart protections requested here: Matrix has a file-backed sync cursor, only trusts restart backlog replay after a clean persisted shutdown, drops pre-startup room history when the cursor is not restart-safe, and durably dedupes inbound `roomId|event_id` before session recording. The Matrix release notes also show the durable restart dedupe fix shipped in `v2026.3.22`; the broader related reports about outbound stale queue replay and multi-channel duplicate injection remain distinct.

## What This Changes

The issue asks OpenClaw to stop treating old Matrix room events as fresh inbound messages after a gateway restart or host reboot by restoring sync state and/or durably deduping Matrix `event_id`s.

## Best Possible Solution

Keep the shipped Matrix restart design: persist the Matrix `/sync` cursor, only trust restart backlog replay after a clean sync-store persist, drop old pre-startup history when the cursor is not safe, and keep durable `roomId|event_id` dedupe around handler/session/reply dispatch. If someone can still reproduce on `v2026.3.22` or newer, it should come back as a fresh regression with event IDs, timestamps, and Matrix account/storage state details.
