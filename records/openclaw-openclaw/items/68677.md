## Summary

Keep this PR open. Current main still lacks the resize-result cache proposed here: historical session images still flow through `sanitizeSessionMessagesImages`, `sanitizeContentBlocksImages` still calls `resizeImageBase64IfNeeded`, and that helper still decodes metadata with `getImageMetadata` before its no-op early return. The linked bug issue is closed and prior PR #64514 was closed unmerged, but the observable fix is not present on main or in v2026.4.26. The PR is a focused implementation candidate and the author followed up on the only concrete bot review nit.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to review or adapt its full-payload-keyed, byte-bounded resize cache in `src/agents/tool-images.ts`, keep the repeated-call/cache-bound regression coverage, consider the cache lifetime/privacy tradeoff, and run the normal changed gates before merge.
