## Summary

Keep open. Current main still has the runtime-deps lock behavior described by the PR and the paired bug, while the PR branch is a focused implementation candidate with an unresolved correctness review about comparing start-time values in different units/resolutions. ## What This Changes

The branch modifies bundled runtime-deps lock ownership to persist a process start-time marker, treat live-PID locks with a different start time as stale, and add tests for matching, mismatched, and missing start-time evidence.

## Best Possible Solution

Land a narrow runtime-deps lock hardening fix, in this PR or an equivalent replacement, that records a comparable process-start identity for both sync and async lock writers, reuses or aligns with the existing `getProcessStartTime()` tick-based contract, preserves legacy and non-Linux behavior when start-time evidence is unavailable, and closes #74346 after regression coverage and changed-lane validation pass.
