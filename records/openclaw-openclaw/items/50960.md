## Summary

Keep this PR open. Current main still does not implement the PR's central behavior: `skills.entries.*.env` keys that match the sensitive-name sanitizer, such as `GOG_KEYRING_PASSWORD` or `CUSTOM_SERVICE_TOKEN`, are only allowed when the skill metadata declares them as `primaryEnv` or `requires.env`. The PR remains a plausible fix for the linked skill/exec env bug, but it needs maintainer review because the submitted diff widens a sensitive env-var bypass and the existing run path can apply skill env overrides before all prompt eligibility filtering is complete.

## Best Possible Solution

Keep this PR open as a useful implementation candidate for #31583, but ask for a focused revision: allow explicitly configured sensitive-pattern keys only for the skill env scope that should be active, preserve the hard host-env blocks, add regression coverage for `GOG_KEYRING_PASSWORD`/`CUSTOM_SERVICE_TOKEN`, and add tests for disabled/unselected skills plus duplicate sensitive env keys before landing.
