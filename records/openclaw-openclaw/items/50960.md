## Summary

Keep this PR open. Current main still builds the skill sensitive-env allowlist from skill metadata only, then sanitizes `skills.entries.*.env` without adding those configured keys, so `GOG_KEYRING_PASSWORD` and `CUSTOM_SERVICE_TOKEN` style values remain blocked unless declared by the skill. The PR targets real unresolved work for #31583, but the submitted host-env change still needs maintainer review around active/eligible skill scope and duplicate sensitive env names before it is safe to merge.

## Best Possible Solution

Keep this PR open as an implementation candidate for #31583. The best revision is a focused skills env fix that allows explicitly configured sensitive-pattern keys only for the active/eligible skill scope, preserves hard dangerous-host-env blocks, isolates or rejects duplicate sensitive env names, and adds regression coverage for `GOG_KEYRING_PASSWORD`, `CUSTOM_SERVICE_TOKEN`, disabled/unselected skills, duplicate keys, and host exec inheritance.

## Risks / Open Questions

- Closing this PR now would leave #31583 unresolved for skills like `gog` that need undeclared sensitive-pattern values from `skills.entries.*.env`.
- Merging the current approach as-is could inject sensitive `*_TOKEN` or `*_PASSWORD` values for disabled, blocked, or prompt-ineligible skills into the host process before final skill prompt filtering.
- Duplicate sensitive env names still need an explicit isolation or rejection policy; the current process-wide first-wins env state can make one configured skill inherit another skill's credential when they share a key.
