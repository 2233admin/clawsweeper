## Summary

Current main still deliberately excludes pure timeout failures from auth-profile cooldown persistence, and the configured fast-escalation knobs only cover overloaded and rate-limit paths. The issue remains valid, though a focused fix should avoid treating every isolated transport timeout as permanent auth failure.

## Best Possible Solution

Keep this open and implement a narrow timeout-specific failover policy: count consecutive model/profile timeouts in the active run, or add a model-scoped short-lived timeout circuit breaker, so configured fallback models are reached quickly without poisoning shared auth health for one-off transport stalls.
