## Summary

Current main still does not implement a config-backed default for `openclaw tui --deliver`. The CLI still hard-defaults `--deliver` to false, passes `Boolean(opts.deliver)` into `runTui`, `runTui` still resolves `deliverDefault` as `opts.deliver ?? false`, and normal sends use that fixed startup value. The config type, strict schema, generated config surfaces, and TUI docs still do not expose `tui.deliver`; the related implementation PRs #33198 and #33208 are closed unmerged, and #66991 only covered the stuck-spinner lifecycle failure.

## Best Possible Solution

Keep this issue open and implement a narrow config contract for the TUI delivery default: add documented `tui.deliver?: boolean`, make omitted CLI input use `config.tui?.deliver ?? false`, keep `--deliver` as true override and add/verify `--no-deliver` as false override, update generated config docs/schema surfaces, and add focused CLI/runtime/send-path tests. Reconcile the existing public TUI docs that mention `/deliver` and a settings toggle.
