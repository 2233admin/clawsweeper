## Summary

Keep open. Current main still hard-defaults TUI delivery to false from the CLI/runtime path, has no `tui.deliver` (or equivalent) public config key in the config types/schema/help surfaces, and the earlier implementation PRs for this request are closed unmerged.

## What This Changes

The issue requests a config-backed default for the TUI `--deliver` behavior so users do not need to pass `openclaw tui --deliver` on every invocation.

## Best Possible Solution

Keep this issue open and implement a narrow public TUI config contract: add optional `tui.deliver?: boolean`, resolve omitted CLI input as `config.tui?.deliver ?? false`, preserve absent-config behavior as false, preserve `--deliver` as an explicit true override, add `--no-deliver` as an explicit false override, and update focused tests/docs/config metadata without changing unrelated delivery semantics.
