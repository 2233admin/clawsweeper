## Summary

Keep open: current main supports one Gateway control plane with operator/node roles and separate isolated Gateway profiles, but it does not implement Gateway federation. The request would change core Gateway protocol, authentication, routing, and state ownership, so it needs maintainer architecture/security review rather than a cleanup close.

## What This Changes

Request support for Gateway-to-Gateway federation using mutual WebSocket links, a new Gateway role, and routed cross-Gateway messages.

## Best Possible Solution

Keep this as an architecture/product issue until maintainers define the federation model: topology, trust and auth, protocol versioning, routing semantics, state ownership, failure behavior, and docs/tests. If approved, land it as incremental Gateway protocol/server changes with explicit compatibility coverage.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is source/docs inspection: there is no accepted `gateway` role, no `gw.send` method, and current docs describe isolated Gateways rather than federation.

## Solution Assessment

Unclear. The proposed new role and method are a useful sketch, but the maintainable solution needs a broader Gateway protocol/security design before implementation.
