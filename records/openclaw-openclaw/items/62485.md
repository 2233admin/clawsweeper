## Summary

Keep open. Current main still has the static `createConfigIO` import in the CLI banner config helper, so the reported help-path problem is not implemented on main; however this PR is not mergeable as-is because its dynamic-import cache assumes a `.then()` callback can run synchronously and then caches `undefined`, leaving configured banner tagline modes ignored.

## What This Changes

This PR adds a `banner-config-lite.runtime.ts` dynamic-import boundary and rewrites `readCliBannerTaglineMode` to lazily resolve CLI banner tagline config instead of statically importing `createConfigIO`.

## Best Possible Solution

Revise the lazy banner-config boundary so help generation avoids loading the heavy config graph while preserving `cli.banner.taglineMode` semantics when config is already available, and add focused tests for both `openclaw tui -h` help behavior and configured `off`/`default` tagline modes.

## Reproduction Assessment

Yes for the PR regression: the dynamic-import timing check shows the `syncResolved` assumption is false, and source inspection shows the cache is set before the import settles. The original CLI hang has a clear command path and current main still matches the reported static-import implementation, but I did not run the potentially hanging command in this read-only review.

## Solution Assessment

No. The `*.runtime.ts` boundary direction is reasonable, but this implementation is not the best fix because it permanently caches the fallback value and still uses a process-global cache that ignores the passed `env`.

## Review Findings

Overall correctness: patch is incorrect

  - confidence: 0.96

  - confidence: 0.86
