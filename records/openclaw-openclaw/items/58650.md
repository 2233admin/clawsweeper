## Summary

Keep #58650 open. Current `main` still has the reply-runtime behavior this PR targets: `/reasoning on` with block replies can leave `assistantTexts` populated from streamed/block content, and final payload assembly still prefers any non-empty `assistantTexts` over `fallbackAnswerText` from `lastAssistant`. The PR discussion and review comments show a focused bug fix with follow-up handling for chunk dedupe and message-baseline edge cases, but those changes are not present on current `main`.

## Best Possible Solution

Keep the PR open and land a rebased, focused version or equivalent main-branch fix: pass the current assistant-message baseline through the attempt result into `buildEmbeddedRunPayloads`, append `fallbackAnswerText` only for `/reasoning on` when current-message assistant text does not already cover it, and keep the append/dedupe/off-mode/multi-chunk/baseline regression tests.
