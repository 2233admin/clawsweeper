## Summary

Keep this PR open. Current main still lacks the requested configurable `node.invoke` default, and the branch is a useful implementation candidate, but it is not merge-ready because the advertised config key would not pass strict runtime schema validation and the configured default would not reach node-side timeout enforcement.

## What This Changes

The branch adds a `gateway.nodes.invokeTimeoutMs` TypeScript config field, a configurable default timeout on `NodeRegistry`, and startup wiring from gateway config into the node registry.

## Best Possible Solution

Finish this as a complete config contract change: add `gateway.nodes.invokeTimeoutMs` to the strict schema, help/labels/generated metadata, and public docs; validate it as a positive integer with any maintainer-approved bound; resolve the effective timeout once before sending `node.invoke.request`; preserve per-call precedence; and add focused config and gateway/node invoke tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Add schema support for the new timeout config:** `src/config/types.gateway.ts:390`
  - body: `GatewayNodesConfig` exposes `invokeTimeoutMs`, but `gateway.nodes` is validated by a strict zod object that this PR does not update. A user who follows the new example config will hit an unrecognized-property validation failure before gateway startup, so the feature is not usable until the schema and config metadata/docs are updated.
  - confidence: 0.96
- **[P2] Pass the resolved timeout to the node request:** `src/gateway/node-registry.ts:143-144`
  - body: The fallback default is applied only after sending `node.invoke.request`, so the payload still has `timeoutMs: undefined` when the caller omits a per-call timeout. Swift node sessions use their local 30,000 ms default in that case, meaning a longer configured gateway default can still time out node-side; resolve the effective timeout before building the payload and use it in both places.
  - confidence: 0.94
- **[P2] Reject non-positive configured invoke timeouts:** `src/gateway/server.impl.ts:488-490`
  - body: The startup guard accepts any number, so `invokeTimeoutMs: 0` or a negative value becomes the registry default and makes every defaulted invoke time out immediately or rely on timer clamping. Validate this as a positive integer in the config schema/startup path before assigning it.
  - confidence: 0.88
