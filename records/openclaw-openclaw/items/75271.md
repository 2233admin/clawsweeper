## Summary

Current main already prevents the user-visible Anthropic prefill loop. The repaired JSONL may still contain a trailing assistant record, but the embedded runner strips trailing non-tool assistant turns from the outbound Anthropic replay context before provider submission, and the same behavior is covered by a regression test and present in the v2026.4.29 tag.

## What This Changes

The issue reports that embedded agent session repair can leave repaired JSONL transcripts ending with an assistant message, which then causes Anthropic Opus requests to be rejected as unsupported assistant prefill.

## Best Possible Solution

Keep the shipped outbound replay guard as the product behavior: it preserves stored transcript history while preventing stale assistant tails from being sent as Anthropic prefill. A separate session-janitor/quarantine feature can be filed if maintainers want on-disk cleanup beyond provider replay safety.

## Reproduction Assessment

Yes. The historical failure shape is a replay context ending `[user, assistant]`; current main has a focused regression test that exercises that shape for Anthropic outbound replay and verifies the provider receives only the user turn.

## Solution Assessment

Yes. Stripping only the outbound replay copy is the narrow maintainable fix for the Anthropic 400 loop because it avoids mutating user-visible transcript history while preventing the provider-invalid prefill request.
