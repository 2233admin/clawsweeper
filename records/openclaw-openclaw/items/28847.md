## Summary

Keep open. Current main has adjacent rate-limit classification, API-key list rotation, and per-profile/model cooldown behavior, but it still does not implement the central requested Gemini quota-bucket model: explicit project/bucket metadata, bucket-level cooldown state, and fanout from one failed key/profile to all keys/profiles in the same quota bucket.

## Best Possible Solution

Keep this open for maintainer design. The best path is to add an explicit quotaBucket/quotaGroup field for provider auth profiles or provider-owned auth metadata, persist bucket-level health alongside profile usage state, and update Gemini rate-limit handling so a 429/RESOURCE_EXHAUSTED cools every key/profile in the same project bucket before selecting another bucket. The linked Keymanager sidecar and aashari fork are useful prior art, but current OpenClaw should not rely on log tailing or config rewrites for this behavior.
