## Summary

Keep open. Current main has adjacent Gemini/API-key list support, 429/RESOURCE_EXHAUSTED classification, profile cooldowns, model-scoped cooldowns, and rotation caps, but it still does not implement the central Gemini quota-bucket/project grouping: there is no explicit quotaBucket/quotaGroup config or credential metadata, no bucket-level persisted health, and no fanout that cools sibling keys/profiles in the same Google project after one quota failure.

## What This Changes

Request provider API-key pools with bucket/project-aware cooldown and backoff after Gemini 429/quota errors to avoid retry storms.

## Best Possible Solution

Keep this open for maintainer design. The best path is to add an explicit provider-owned quotaBucket/quotaGroup field for auth profiles or stored credential metadata, persist bucket-level health alongside existing profile usage state, and update Gemini rate-limit handling so a 429/RESOURCE_EXHAUSTED cools every key/profile in the same project bucket before selecting a different bucket. The sidecar and fork are useful prior art, but core should not depend on log tailing or auth-file rewrites for this behavior.
