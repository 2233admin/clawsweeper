## Summary

Current main still does not implement the requested native imsg `reply_to_guid` send path. The legacy iMessage plugin forwards sanitized reply metadata as `reply_to` and strips visible directive tags, which fixes the leaked-text symptom from #39512, but it does not prove native iMessage threaded replies through imsg. BlueBubbles has a separate native reply-threading path via Private API, so the remaining work should be clarified as legacy imsg-specific or closed only after an imsg send contract exists and OpenClaw targets it.

## Best Possible Solution

Keep this issue open. First verify and document the actual imsg outbound native-reply contract. If imsg supports it, update `extensions/imessage/src/send.ts` to send the documented field, strip delivered text correctly, and add a fake-client regression test asserting both the JSON-RPC params and sanitized text. If public imsg still treats `reply_to_guid` as read-only metadata, coordinate or pin the required imsg release/API before changing OpenClaw.
