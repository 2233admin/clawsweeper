## Summary

Keep this issue open. Current main still strips visible reply tags and sends sanitized iMessage reply metadata as `reply_to`, but it does not pass `reply_to_guid` to the legacy imsg JSON-RPC send call. The remaining blocker is not generic OpenClaw reply routing; it is the legacy imsg outbound native-threading contract, which should be verified before changing OpenClaw.

## Best Possible Solution

Keep this issue open for maintainer follow-up. First verify the actual imsg outbound native-reply contract and the minimum imsg release that supports it, if any. If supported, update `extensions/imessage/src/send.ts` to send the documented reply field, keep delivered text stripped of directive tags, and add a fake-client regression test that asserts the exact JSON-RPC `send` params. If public imsg only exposes `reply_to_guid` as read-only metadata, document the legacy limitation and steer native threaded replies to BlueBubbles or upstream imsg work.
