## Summary

Keep open. Current main still does not implement a native `openclaw doctor --dry-run`: the doctor command has no flag, the typed options cannot propagate one, docs omit it, and several doctor repair paths can still write config or perform runtime/service repairs. The related global agent dry-run issue and the open `--check-config` PR are different scopes, so they do not supersede this focused doctor safety request.

## What This Changes

Request to add `openclaw doctor --dry-run` so doctor can validate and preview recommended repairs without writing config or performing repair side effects.

## Best Possible Solution

Add a first-class doctor dry-run contract: register and type `--dry-run`, propagate it through repair mode, prompts, config flow, and health contributions, render the same repair plan without prompts or writes, suppress non-config side effects, define exit-code behavior, and update the public doctor docs and changelog.

## Repair Work Prompt

Implement native `openclaw doctor --dry-run` for the doctor command only. Register the flag in `src/cli/program/register.maintenance.ts`, add `dryRun` to `DoctorOptions`, and thread it through doctor repair mode, prompter/config flow, and health contributions. In dry-run mode, run validation and collect/render the same repair recommendations, but do not prompt, call `replaceConfigFile`, install bundled plugin runtime dependencies, save cron state, install/start/restart gateway services, or perform other doctor repair side effects. Define and test the requested non-zero exit behavior when issues or would-change repairs are found. Keep #41418 global agent tool dry-run and #47302 `--check-config` separate; do not implement global tool-call interception or deep config validation as part of this PR. Update `docs/cli/doctor.md` and add a user-facing changelog entry with appropriate reporter credit, unless a maintainer says to omit the changelog. Preserve existing `--fix`, `--repair`, `--yes`, and `--non-interactive` behavior outside dry-run.
