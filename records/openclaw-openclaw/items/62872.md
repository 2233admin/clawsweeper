## Summary

Keep open. Current main still has the documented peer-session `sessions_send` A2A follow-up path: non-skipped sends run bounded ping-pong turns and then an announce step, and the tool schema still has no per-call way to disable that flow. The merged ACP parent-owned child skip is relevant but intentionally narrower than the persistent peer-session behavior reported here.

## What This Changes

The issue requests preventing `sessions_send` between persistent sessions from creating duplicate A2A ping-pong and announce messages.

## Best Possible Solution

Keep this as the canonical tracker for persistent-session duplicate behavior. The desired end state is a context-aware `sessions_send` A2A lifecycle, or an explicit per-call A2A disable/max option, that prevents unwanted persistent peer follow-up turns while preserving the parent-owned ACP child skip and any intentional peer announce workflow with regression coverage.
