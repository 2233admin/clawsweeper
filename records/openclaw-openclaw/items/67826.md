## Summary

Keep open for maintainer review. Current main has already superseded the setup allowFrom direction by making Telegram setup numeric-ID-only, while the PR still preserves/test-locks setup @username resolution; the separate runtime target resolver hardening is not implemented on main and needs an explicit contract decision.

## What This Changes

The PR widens Telegram setup allowFrom and target resolver inputs to tolerate null, undefined, and non-string values, switches token-file inspection to `normalizeOptionalString`, and adds a setup-surface test for malformed allowFrom entries.

## Best Possible Solution

A mergeable fix should preserve numeric-only Telegram allowFrom setup, drop the stale setup username-resolution pieces, and either keep the resolver contract string-only with caller-boundary validation or add explicit, focused Telegram defensive coercion without widening the public SDK contract accidentally.
