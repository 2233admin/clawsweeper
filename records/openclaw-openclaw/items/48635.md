## Summary

Keep this PR open. Current main still has the same user-facing gaps the PR targets: the per-response `/usage` footer formats token/cache/cost data only, and `agents.defaults` still has no `responseUsageDefault` config surface. The PR remains a plausible implementation candidate for #12578 and part of the model-footer requests in #40215/#65477, but it needs rebasing and correctness cleanup before it can land.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase onto current main, move the formatter changes to `src/auto-reply/reply/agent-runner-usage-line.ts`, remove the log artifact, add typed/schema/generated-doc support for `agents.defaults.responseUsageDefault`, preserve an explicit session-level off override, wire the actual model and resolved run thinking level into the footer under a tested mode contract, and update `/usage` docs/tests.

## Risks / Open Questions

- As drafted, falling back to `cfg.agents.defaults.responseUsageDefault` with `??` can make `/usage off` ineffective for a session, because current `/usage off` deletes `responseUsage` from the session entry.
- The PR reports thinking from `activeSessionEntry?.thinkingLevel`, but current main resolves the actual run thinking level onto `followupRun.run.thinkLevel`; defaults or provider fallback can otherwise be omitted or misreported.
- The PR needs a maintainer/product decision and tests for whether model/thinking metadata belongs only in `full` mode or also in `tokens` mode.
- The added root log file is unrelated to the feature and includes generated runtime path details; it should be removed before merge.
