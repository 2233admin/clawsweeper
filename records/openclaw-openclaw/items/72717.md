## Summary

Keep this open. Current main still has a digest-assisted, Markdown-backed `wiki_search` path with no SQLite/FTS cache, no config surface for an index, and no compile/apply refresh hook for an indexed wiki cache; the related shared-backend crash report is adjacent but does not supersede this performance feature.

## What This Changes

Request an internal SQLite FTS5 cache for `wiki_search`, refreshed by wiki compile/apply writes and used before falling back to Markdown scanning.

## Best Possible Solution

Add the index inside the bundled `memory-wiki` plugin as a read-optimized cache under the wiki cache area, refreshed by compile/apply/ingest, with ranked FTS results feeding the existing result shape and the current digest/Markdown path retained as fallback.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from current source because wiki search reads compiled digests and Markdown files, while config/docs expose no SQLite/FTS index path.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution if kept inside `memory-wiki`. Moving wiki content into `memory/main.sqlite` or core memory search would cross ownership boundaries and broaden the change unnecessarily.
