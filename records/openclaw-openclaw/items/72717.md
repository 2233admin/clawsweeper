## Summary

Keep this open. Current main still implements memory-wiki search as digest-assisted local Markdown reads plus optional shared-memory supplement results, and it does not contain the requested SQLite/FTS-backed wiki_search cache. The request is recent, concrete, plugin-owned, and narrow enough for a focused repair PR.

## What This Changes

Add an internal SQLite FTS5 cache for memory-wiki content so wiki_search can query ranked indexed wiki pages and refresh the cache after wiki compile/apply writes, with file-scan fallback.

## Best Possible Solution

Implement a memory-wiki-owned read cache under the wiki cache directory: Markdown pages and compiled digests remain the source of truth, compile/apply update or rebuild the FTS index, wiki_search uses ranked FTS results when fresh, and the existing digest/file-scan path remains the fallback.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature/performance request. We do have a high-confidence verification path: current-main source and docs show digest-assisted Markdown reads and no SQLite/FTS memory-wiki implementation.

## Solution Assessment

Yes, the requested direction is a maintainable narrow solution if it stays inside memory-wiki and reuses existing SQLite helper patterns with fallback. It should not expand memory/main.sqlite or move wiki-specific ranking into core memory search.
