## Summary

Keep this open. Current main still has no SQLite or FTS-backed wiki index in the memory-wiki plugin; `wiki_search` uses digest-assisted candidate selection and Markdown file reads, and the config/docs expose only `shared` and `local` search backends. The request is recent, concrete, unpaired by an open implementation PR in the provided context, and narrow enough for a focused memory-wiki implementation.

## What This Changes

The issue requests adding a SQLite FTS5-backed cache for `wiki_search` so wiki content can be searched through an indexed layer with ranked results and file-scan fallback.

## Best Possible Solution

Add the SQLite FTS cache inside the bundled memory-wiki plugin as a read-optimized artifact under the wiki cache area, refresh it from compile/apply/ingest paths, query it before the existing local Markdown search, and keep the current digest/file scan behavior as the fallback.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from current source because memory-wiki has no SQLite symbols or config surface and `wiki_search` still reads digest candidates plus Markdown files.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution if implemented within memory-wiki. Reusing or expanding `memory/main.sqlite` would blur ownership between the active memory plugin and wiki-specific ranking/provenance behavior.
