## Summary

Keep this open. Current main still lacks a SQLite/FTS-backed memory-wiki search cache: `wiki_search` uses compiled digest candidates plus Markdown reads, while compile/apply refresh JSON/JSONL artifacts only. The request is recent, concrete, plugin-owned, and narrow enough for a focused repair PR.

## What This Changes

Request adding an internal SQLite FTS5 cache for `wiki_search`, refreshed by memory-wiki compile/apply writes and used before falling back to Markdown scanning.

## Best Possible Solution

Add a memory-wiki-owned SQLite/FTS cache under the wiki cache or plugin state area, refreshed by compile/apply/ingest paths, queried by `wiki_search` for ranked wiki results, and backed by the existing digest/Markdown path as fallback. Keep the work inside the memory-wiki plugin unless a genuinely shared SDK seam becomes necessary.

Not applicable as a bug reproduction; this is a feature/performance request. The missing capability is high-confidence from source and docs because current `wiki_search` uses digest-assisted Markdown reads and memory-wiki exposes no SQLite/FTS backend.

## Solution Assessment

The observable gap is that `wiki_search` currently reads compiled digests and Markdown pages instead of querying an indexed wiki cache.
