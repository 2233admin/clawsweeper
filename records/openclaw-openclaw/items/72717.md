## Summary

Keep this open. Current main still implements memory-wiki search as a compiled-digest-assisted Markdown reader, with no SQLite or FTS-backed wiki cache, and the request is concrete enough for a narrow plugin-owned follow-up.

## What This Changes

Request to add an internal SQLite FTS5 cache for memory-wiki `wiki_search`, refreshed by wiki compile/apply writes and falling back to Markdown scans when unavailable.

## Best Possible Solution

Add a memory-wiki-owned SQLite/FTS cache under the wiki cache area, with Markdown pages and compiled digests remaining the source of truth, index refreshes tied to compile/apply/import flows, staleness detection, and graceful fallback to the current local search path. Keep this separate from `memory/main.sqlite` and avoid moving wiki-specific behavior into core memory search.

## Reproduction Assessment

Not applicable as a bug reproduction: this is a feature/performance request. The missing capability is verifiable from current source and docs because `wiki_search` uses digest-assisted Markdown reads and no SQLite/FTS wiki index exists.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution if implemented as an internal memory-wiki cache with fallback. The broader companion-DB decision/exchange schema should remain out of scope unless maintainers choose a separate product design for durable decision records.

## Repair Work Prompt

Implement an internal SQLite FTS5 cache for the bundled memory-wiki plugin to speed up `wiki_search` while preserving current behavior. Scope the change to memory-wiki-owned code: add a cache under `.openclaw-wiki/cache/` or an equivalent wiki cache path, index wiki page title/body/metadata/path with enough stored metadata to reconstruct ranked results, refresh/rebuild the cache from `compileMemoryWikiVault` and after `applyMemoryWikiMutation`/import paths that compile, and have `searchMemoryWiki` query the FTS cache first when fresh. Preserve Markdown pages and existing compiled digests as the source of truth, detect missing/stale/unavailable SQLite or FTS5 support, and fall back to the current digest-assisted Markdown search without user-visible failure. Keep `memory/main.sqlite` and core memory search schemas unchanged. Preserve existing search modes (`auto`, `find-person`, `route-question`, `source-evidence`, `raw-claim`) and claim/provenance metadata in results. Update docs and changelog for the user-facing performance/cache behavior; credit the issue author in the changelog if adding an entry. Do not add broad core/plugin API seams unless unavoidable.
