## Summary

Keep open. Current main still routes google-antigravity through the pi-ai Cloud Code Assist transport, and the pinned @mariozechner/pi-ai 0.70.6 tarball still defaults the Antigravity User-Agent version to 1.21.9 unless PI_AI_ANTIGRAVITY_VERSION is set. The auth-cache half is only partially mitigated: same-process saves update an active runtime snapshot, but an externally refreshed auth-profiles.json is still masked by an active runtime snapshot until an explicit snapshot replacement path such as secrets.reload runs.

## What This Changes

The issue asks to fix google-antigravity by moving off a deprecated Antigravity user-agent version and by refreshing or invalidating stale runtime auth snapshots after credential refresh.

## Best Possible Solution

Keep this open and fix both halves deliberately: choose a provider-owned Antigravity version strategy for the pinned Cloud Code Assist transport, then add a targeted runtime auth snapshot refresh or invalidation path so successful credential refresh or explicit reload lets the active gateway observe fresh credentials without restart.

## Reproduction Assessment

Yes for a source-level reproduction: current main still pins pi-ai 0.70.6, whose Antigravity transport defaults to User-Agent antigravity/1.21.9, and active runtime auth snapshots are returned before disk is re-read. Live reproduction of the sandbox rejection still needs maintainer OAuth credentials.

## Solution Assessment

Unclear until maintainers choose the Antigravity version strategy. The narrowest maintainable path appears to be an upstream pi-ai fix or OpenClaw provider-local compatibility shim plus a generic auth snapshot refresh seam, not a startup-only migration or hard-coded core recovery policy.
