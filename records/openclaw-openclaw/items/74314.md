## Summary

Keep open. Current main still pins the pi-ai transport version whose Antigravity user-agent default is 1.21.9, google-antigravity remains an active route through that transport, and runtime auth snapshots can still mask refreshed on-disk credentials except for same-process save or explicit secrets-runtime refresh paths.

## What This Changes

Fix google-antigravity by moving off the deprecated Antigravity API user-agent version and ensuring refreshed OAuth credentials become active without a gateway restart.

## Best Possible Solution

Keep the issue open until maintainers choose the Antigravity version strategy and add a generic runtime auth snapshot refresh or invalidation path that lets refreshed auth-profiles.json state recover without restarting the gateway.

## Reproduction Assessment

Yes for the source-level failure path: current main still routes google-antigravity/google-gemini-cli models through the pinned pi-ai transport with the 1.21.9 Antigravity default, and active runtime auth snapshots are preferred before disk reads. A full live sandbox reproduction still needs maintainer Google OAuth credentials.

## Solution Assessment

No current-main solution is complete. The narrow maintainable direction is a provider/dependency fix for the Antigravity user-agent plus a generic auth snapshot reload or invalidation mechanism, not a one-off hidden core migration.
