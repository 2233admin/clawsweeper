## Summary

Keep open. Current main still has both repository-side causes: OpenClaw pins the pi-ai transport version whose Antigravity default User-Agent is still 1.21.9, and runtime auth snapshots can continue serving in-memory credentials until a same-process save or explicit secrets reload replaces them.

## What This Changes

The issue asks OpenClaw to restore google-antigravity by updating or configuring the Antigravity User-Agent version and refreshing stale runtime auth snapshots after OAuth token refresh without requiring a gateway restart.

## Best Possible Solution

Keep this in the OpenClaw provider/auth backlog and land a maintainer-reviewed fix that chooses an Antigravity version strategy for the pinned Cloud Code Assist transport, then adds a targeted runtime auth snapshot refresh or invalidation path so refreshed OAuth credentials are observed by the active gateway without a full restart.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the repository-side causes: inspect the current pi-ai pin, the published 0.70.6 transport default, and the runtime snapshot precedence path. The external Google sandbox rejection still needs a maintainer live OAuth smoke to reproduce end-to-end.

## Solution Assessment

Is this the best way to solve the issue? Not applicable as a patch because this is an issue report, but the requested direction is broadly right: a manual environment override or gateway restart is not the durable fix, while a provider-owned version strategy plus auth snapshot refresh path is the maintainable boundary.
