## Summary

Keep this issue open. Current main still routes google-antigravity through the pinned @mariozechner/pi-ai Cloud Code Assist transport, and the upstream v0.70.6 provider source still defaults the Antigravity User-Agent to 1.21.9 unless PI_AI_ANTIGRAVITY_VERSION is set. The auth snapshot side is only partially addressed: same-process saves refresh an active snapshot and secrets.reload can replace runtime snapshots explicitly, but current main still serves active runtime auth snapshots ahead of disk, so external auth-profiles.json refresh is not automatically observed without a snapshot replacement path.

## What This Changes

The issue reports that google-antigravity fails with a deprecated Antigravity API version and that runtime auth snapshots can keep stale failed credentials after token refresh.

## Best Possible Solution

Keep the issue open and fix both observable paths deliberately: choose a maintainable Antigravity version strategy for the pinned Cloud Code Assist transport, then add a targeted runtime auth snapshot refresh or invalidation mechanism so successful credential refresh or an explicit reload makes the active gateway observe fresh credentials without a full restart.

## Reproduction Assessment

Partially yes. The stale Antigravity default is reproducible from current source and upstream v0.70.6 source without live credentials, and the external-refresh snapshot problem has a clear code-level reproduction because active runtime snapshots are returned before disk state; the full live sandbox failure still needs maintainer OAuth smoke proof.

## Solution Assessment

No. Current main only provides partial mitigation through same-process save refresh and explicit secrets.reload; it does not resolve the stale upstream Antigravity default or automatically observe externally refreshed auth-profiles.json in an active gateway.
