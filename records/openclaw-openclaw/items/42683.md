## Summary

Current main still lacks the PR’s intended iMessage chat.db/GUID reply-context fallback. The live plugin only forwards reply context when `imsg` already supplies `reply_to_*`, while this PR targets the removed `src/imessage` tree and would need to be ported to `extensions/imessage`. Keep it open as a useful stale implementation candidate, or supersede it with a maintainer PR tied to the still-open #10872 tracker.

## Best Possible Solution

Keep this PR open as a useful stale implementation candidate, or replace it with a maintainer PR that ports the idea into `extensions/imessage/src/monitor`. The final fix should resolve GUID/thread-origin metadata into the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` flow, prefer or validate `thread_originator_guid` per #10872, keep `chat.db` reads opt-in through configured `dbPath`, skip local DB reads for remote-host sessions, fail closed on permissions/schema/locking errors, and add regression coverage for GUID-only replies plus failure cases.
