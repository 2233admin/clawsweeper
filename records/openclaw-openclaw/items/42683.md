## Summary

Keep this PR open. Current main still lacks the intended iMessage chat.db reply/quote-context fallback; it only forwards `reply_to_*` data when imsg already supplies resolved text. The PR is stale because it targets the removed `src/imessage` tree, but the underlying fix remains useful and is related to the still-open #10872 tracker.

## Best Possible Solution

Keep this PR open as a useful stale implementation candidate, or supersede it with a maintainer PR that ports the idea into `extensions/imessage/src/monitor`. The final fix should resolve GUID/thread-origin metadata into the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` flow, prefer or validate `thread_originator_guid`, preserve the local-only `remoteHost` protection, stay within plugin-safe imports or SDK seams, and add regression coverage for GUID-only replies plus missing-permission/schema failure cases. The remaining product gap is also tracked by #10872.

## Risks / Open Questions

- Closing as implemented would leave the GUID-only/chat.db iMessage inline-reply path unresolved on current main.
- The PR is not merge-ready as-is because it targets the removed `src/imessage` tree rather than the bundled plugin in `extensions/imessage`.
