## Summary

Keep this PR open. Current main still does not implement the PR’s intended chat.db/GUID fallback for iMessage inbound reply context, and the PR is still a useful implementation candidate even though it targets the removed `src/imessage` tree rather than the active bundled plugin under `extensions/imessage`.

## Best Possible Solution

Keep this PR open as a useful stale implementation candidate, or supersede it with a maintainer PR that ports the idea into `extensions/imessage/src/monitor`. The final fix should resolve GUID/thread-origin metadata into the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` flow, prefer or validate `thread_originator_guid` per #10872, keep chat.db reads opt-in through configured `dbPath`, skip local DB reads for remote-host sessions, fail closed on permissions/schema/locking errors, and add regression coverage for GUID-only replies plus failure cases.
