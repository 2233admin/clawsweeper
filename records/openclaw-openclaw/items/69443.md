## Summary

Keep open. Current main still lets a raw `agent` Gateway RPC omit `sessionKey`, default the run to the main session, and treat a session-key-shaped `to` value as an outbound delivery target instead of the transcript target. The built-in subagent announce path already uses `sessionKey`, so this is a narrow raw-RPC/session-target ambiguity rather than an external WeChat plugin closeout.

## What This Changes

The issue asks for subagent callback RPCs that name a WeChat session key to route into that exact session instead of the main session.

## Best Possible Solution

The end state should make raw `agent` RPC session targeting explicit and safe: a session-shaped target must resolve to the intended session or fail before the run starts, while ordinary channel delivery targets keep their current semantics.

## Reproduction Assessment

Yes. A source-level reproduction is an `agent` RPC with `to` set to an `agent:<agentId>:<channel>:direct:<peer>` key and no `sessionKey`; current main defaults the transcript target before treating `to` as explicit delivery metadata.

## Solution Assessment

No. The current behavior is not the best solution because it silently routes the run to the main session; a narrow fix should either promote the session-shaped `to` into `sessionKey` under well-defined conditions or reject it with a clear instruction to use `sessionKey`.

## Repair Work Prompt

Fix the raw Gateway `agent` RPC behavior reported in https://github.com/openclaw/openclaw/issues/69443. Current main defaults to the main session when `params.sessionKey` is omitted, even if `params.to` is an `agent:<agentId>:...` session key; it then treats that value as an outbound delivery target.
