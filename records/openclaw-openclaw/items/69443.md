## Summary

Current main still does not treat a canonical `agent:<agentId>:...` value in low-level `agent` RPC `to` as the session selector. With no explicit `sessionKey` or `sessionId`, that value is forwarded as a delivery/turn target, then the agent command's normal `to` session derivation collapses direct-chat shaped values to the agent main session, matching the reported WeChat callback misrouting. Related #69576 fixed a different delivery-mirror account-scoping leak and does not cover this RPC boundary.

## Best Possible Solution

Keep this issue open and fix the core gateway `agent` RPC boundary in a plugin-agnostic way: when no explicit `sessionKey` or `sessionId` is provided and `to` is a canonical agent session key, route the run to that session or reject with a clear validation error. Preserve normal delivery-target semantics for ordinary `to` values and add a regression test for the reported session-key-in-`to` path.

When `openclaw gateway call agent` is invoked without explicit `sessionKey` or `sessionId` and `params.to` is a well-formed canonical agent session key such as `agent:main:openclaw-weixin:direct:<peer>`, the gateway should route the agent run to that session instead of letting the lower agent command derive `agent:main:main` from `to`. Add regression coverage in the gateway agent RPC tests for a channel-session-key-shaped `to` with no explicit `sessionKey`/`sessionId`, and ensure explicit `sessionKey`, explicit `sessionId`, `replyTo`, and ordinary delivery `to` behavior remain unchanged.
