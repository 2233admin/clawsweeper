## Summary

Keep #69443 open. Current main has related delivery-context hardening for subagent sessions, but it does not implement the reported `agent` RPC behavior where `to` is a full OpenClaw session key. The current code still treats `sessionKey` as the session selector and `to` as an external delivery/derivation input, which can collapse a direct session-like `to` value back to the main session path.

## Best Possible Solution

Keep the issue open and make an explicit maintainer decision for the `agent` RPC contract. The best code path is either to support session-key-shaped `to` by promoting it to `sessionKey` before session resolution, or to reject it with a clear validation error and document that callbacks must use `sessionKey`. Add a core regression test using a stub channel and a key like `agent:main:openclaw-weixin:direct:user@im.wechat` so the behavior is plugin-agnostic.
