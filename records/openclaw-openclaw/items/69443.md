## Summary

Keep open. Current main still has a plausible generic Gateway/session-routing bug: when the `agent` RPC receives a canonical session key in `to` but no `sessionKey`, the handler does not promote that value to the target session, and the downstream session resolver treats it as a direct-chat target that collapses back to the main session. The built-in subagent announce path already uses `sessionKey`, so this is narrower than all subagent delivery, but the reported manual RPC path remains actionable.

## What This Changes

The issue requests that a subagent Gateway `agent` RPC callback addressed to a WeChat canonical session key route to that WeChat session instead of the agent main session.

## Best Possible Solution

Keep the fix generic in the Gateway/session layer: when `agent` RPC receives a canonical `agent:<id>:...` value in `to` without `sessionKey` or `sessionId`, either route it as the explicit session and use stored delivery metadata, or fail fast with a targeted validation error. Cover the WeChat-shaped direct key with regression tests and leave WeChat-specific runtime code in the external plugin.

## Reproduction Assessment

Yes. A focused static reproduction is clear: call `agent` with `to: "agent:main:openclaw-weixin:direct:...@im.wechat"` and no `sessionKey`; current main leaves `resolvedSessionKey` unset, then the runner derives a direct-chat session from `to` and collapses it to `agent:main:main`.

## Solution Assessment

No for the reported workaround as a long-term API contract: using `to` as a session selector is ambiguous because `to` is also the external delivery target. The maintainable fix is a narrow compatibility/validation guard in core while preserving `sessionKey` as the canonical RPC field.
