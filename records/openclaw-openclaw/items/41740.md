## Summary

Current main still has the reported gap. Discord approval buttons resolve decisions through the shared operator-approvals Gateway client, and that client constructs GatewayClient without tlsFingerprint. The normal callGateway path still has separate local self-signed TLS fingerprint resolution, so the approval path remains inconsistent. The related fix PRs #41741 and #46232 are useful prior art, but both were closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open. The best fix is to resolve the local gateway TLS fingerprint in the shared operator-approval Gateway bootstrap/client path, pass it through to GatewayClient, and add focused regression coverage for the local wss:// self-signed case plus override/remote cases so TLS validation is not weakened. The closed #41741 and #46232 remain useful prior art but should not close this issue unless an equivalent tested fix lands on main.
