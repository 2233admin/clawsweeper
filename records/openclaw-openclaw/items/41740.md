## Summary

Current main at 28ff82dcdae7f7fb11491deefe48bd82b68eb0bb still leaves the reported gap open. Discord approval buttons resolve through the shared operator-approval Gateway client; that client constructs GatewayClient without tlsFingerprint, while callGateway has the local wss:// self-signed TLS fingerprint resolution logic. Prior PRs #41741 and #46232 are useful implementation context, but both are closed unmerged and no equivalent fix is present on main.

## Best Possible Solution

Keep this issue open and fix the shared operator-approval Gateway client/bootstrap path. The fix should resolve the local Gateway TLS runtime under the same conditions as callGateway, pass tlsFingerprint into GatewayClient, and add focused regression tests for local self-signed wss plus CLI/env override and configured remote cases. #46232 is closer prior art than the Discord-only #41741, but either should be adapted to current main rather than closing this issue before a tested fix lands.
