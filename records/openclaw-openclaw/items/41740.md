## Summary

Keep open. Current main still routes Discord exec approval decisions through the shared operator-approvals GatewayClient path, and that path constructs GatewayClient without a tlsFingerprint. GatewayClient only uses pinned self-signed WSS handling when tlsFingerprint is supplied, while the existing callGateway path has local TLS runtime fingerprint loading that operator approvals do not reuse. The related fix PRs #41741 and #46232 were closed unmerged, so this remains an actionable bug.

## Best Possible Solution

Keep this issue open. The best fix belongs in the shared operator approvals gateway bootstrap/client path: resolve the same local gateway TLS runtime fingerprint used by callGateway when gateway.tls.enabled is true, the resolved target is local wss://, and no explicit CLI/env URL override or remote URL is in effect; pass that fingerprint into GatewayClient; add focused regression coverage for resolveGatewayClientBootstrap or createOperatorApprovalsGatewayClient plus the Discord approval button/native approval runtime path. The closed PRs #41741 and #46232 are useful prior art but should not close this issue unless an equivalent tested fix lands on main.
