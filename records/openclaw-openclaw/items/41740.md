## Summary

Current main still omits the local TLS fingerprint on the shared operator-approvals GatewayClient path used by Discord exec approvals. The existing callGateway path has the expected local self-signed TLS fingerprint logic, but the approval resolver/bootstrap path does not reuse it. The related fix PRs #41741 and #46232 were closed unmerged, so this remains an actionable bug.

## Best Possible Solution

Keep this issue open. The best fix belongs in the shared operator approvals gateway bootstrap/client path: resolve the same local gateway TLS runtime fingerprint used by callGateway when gateway.tls.enabled is true, the resolved target is local wss://, and no explicit CLI/env URL override or remote URL is in effect; pass that fingerprint into GatewayClient; add focused regression coverage for resolveGatewayClientBootstrap or createOperatorApprovalsGatewayClient plus the Discord approval button/native approval runtime path. The closed PRs #41741 and #46232 are useful prior art but should not close the issue unless an equivalent tested fix lands on main.
