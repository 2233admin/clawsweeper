## Summary

Current main still has the reported bug: Discord approval buttons and the native approval runtime both use the shared operator-approvals GatewayClient path, and that path constructs GatewayClient without a tlsFingerprint. The existing callGateway path has the local self-signed TLS fingerprint logic, but the operator approvals bootstrap/client path does not. The related fix PRs #41741 and #46232 were closed unmerged, so this remains actionable rather than closeable.

## Best Possible Solution

Keep this issue open. The best fix belongs in the shared operator approvals gateway bootstrap/client path: resolve the same local gateway TLS runtime fingerprint used by callGateway when gateway.tls.enabled is true, the resolved target is local wss://, and no explicit CLI/env URL override or remote URL is in effect; pass that fingerprint into GatewayClient; add focused regression coverage for resolveGatewayClientBootstrap or createOperatorApprovalsGatewayClient plus the Discord/native approval paths. The closed PRs #41741 and #46232 are useful prior art but should not close this issue unless an equivalent tested fix lands on main.
