## Summary

Keep open. Current main still allows CLI fallback retries to reuse a stored CLI backend session while sending prompt text again; the embedded runner has explicit orphaned-user repair, but the CLI path only clears missing/expired sessions and does not repair a resumed session after timeout or rate-limit fallback.

## What This Changes

The issue requests CLI fallback retry repair so resumed CLI backend sessions do not append duplicate user turns after timeout or rate-limit fallback.

## Best Possible Solution

Add a CLI-owned fallback retry repair that avoids resubmitting the same prompt into an already-mutated resumed backend session, while preserving the task through a fresh/reseeded session or a backend-confirmed transcript repair. Cover timeout and rate-limit retries with focused CLI regression tests.

## Repair Work Prompt

Fix the CLI fallback retry duplicate-user-turn bug tracked by this issue. Current main still reuses stored CLI backend sessions during fallback retries and still sends prompt text in resume mode; add a CLI-owned repair so timeout/rate-limit fallback after possible prompt submission does not append the same user turn again. Stay within the agent command and CLI runner surfaces; do not change embedded runner behavior except as needed for shared helpers. Prefer a backend-aware reset/reseed or transcript-confirmation path over blindly skipping the prompt, because skipping can lose the task when the first attempt failed before the backend persisted it. Consider Claude CLI's transcript check, Codex/Gemini resumeArgs behavior, and stored cliSessionBinding persistence. Add focused regression coverage for a resumed CLI session whose first attempt fails after prompt submission and whose fallback does not reuse/resubmit into the same backend session. Mention #44188 as the origin context and #63696 as adjacent embedded-only work, not as a replacement. Changelog is probably not needed unless the user-visible retry behavior changes materially beyond bug repair.
