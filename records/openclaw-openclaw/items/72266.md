Keep this PR open for maintainer review. Current main does not implement the per-session emotion-mode surfaces, and the branch contains relevant gateway/TTS/status work, but it is still an unmerged stacked PR with an unresolved display-sanitizer correctness issue around markdown reference links.

## What This Changes

The branch adds per-session emotion mode across TTS reply metadata, the `/emotions` directive, `voice.md` workspace prompting, gateway history/live/status display, session protocol schemas, Swift generated models, docs, and tests.

## Best Possible Solution

Keep the PR open, fix or explicitly disposition the markdown-reference sanitizer behavior, then review and land the four emotion-mode PRs in stack order with gateway/TTS/status changed-gate coverage.

## Reproduction Assessment

Yes. The PR-head sanitizer has a source-level reproduction path: a line-start assistant message like `[fast][1] before choosing.` in off/on mode reaches `isFollowedByMarkdownReferenceLink`, which returns false because there is no earlier non-whitespace on that line, so `[fast]` is eligible for stripping.

## Solution Assessment

No, not yet. Threading a per-session emotion mode through gateway display and status surfaces is the right direction, but the sanitizer should not corrupt valid markdown reference-link text, or maintainers should explicitly accept #72821 as a follow-up before merge.
