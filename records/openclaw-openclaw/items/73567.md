## Summary

Keep open. The PR targets the right QQBot boundary, but the new active config provider relies on `getRuntimeConfig()`, and the current gateway reload path does not refresh that process snapshot for external `bindings` edits because they are classified as no-op reloads. That means the linked CLI binding scenario can still route against stale bindings unless the PR also refreshes or injects the accepted runtime config for no-restart changes.

## What This Changes

The PR adds a QQBot active runtime-config provider, wires it into per-inbound context building and outbound dispatch, adds QQBot regression tests, updates the changelog, and removes a redundant Codex plugin test cast.

## Best Possible Solution

Land the QQBot per-inbound lookup together with a generic runtime-config refresh for accepted no-restart binding edits, plus a regression that drives an external `agents bind` config write through gateway reload and then verifies the next QQBot inbound routes to the new peer binding.

## Reproduction Assessment

Yes. The linked bug gives concrete QQBot CLI and C2C steps, and source inspection confirms the CLI writes the config file while `bindings` reload is treated as no-op before runtime snapshot activation.

## Solution Assessment

No. The QQBot boundary change is the right shape, but using `getRuntimeConfig()` alone is not sufficient until the gateway refreshes that snapshot for no-op external binding changes or provides QQBot with the reloader's accepted config.
