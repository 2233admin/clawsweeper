## Summary

Keep open. Current main still distinguishes some failure classes, but it does not implement the requested dead-key policy: generic HTTP 401/403 and `invalid_api_key`/`permission_error` payloads stay in the `auth` lane, and `auth` is persisted as a cooldown rather than a disabled credential state. The merged #60404 work shortened `auth_permanent` lockouts, but it did not move ordinary dead-key 401s out of cooldown or make auth-profile cooldowns honor provider `Retry-After` headers.

## What This Changes

Auth-profile failover should immediately skip or disable dead-key 401 credentials while keeping 429 and 529 provider failures on transient cooldown/failover paths.

## Best Possible Solution

Define and implement a conservative auth-health policy that promotes high-confidence dead credentials, such as expired/revoked/invalid API-key 401 payloads, out of the transient cooldown lane while preserving transient handling for 429 and 529. The change should align classifier coverage, auth-profile state persistence, doctor/status guidance, model-fallback behavior, and public docs.
