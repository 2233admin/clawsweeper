## Summary

Keep open. Current main has adjacent failover improvements and a newer `auth_permanent` lane, but it still deliberately treats generic HTTP 401 and `invalid_api_key` as the short `auth` lane, which then uses cooldown state rather than the requested persistent dead-key disable/re-enable behavior.

## Best Possible Solution

Implement a precise auth-profile failure policy for dead credentials: classify high-confidence invalid/expired API-key responses into a durable disabled state or explicit `auth_permanent` lane, skip those profiles without re-probing every cooldown window, provide a clear operator re-enable/rekey path, and keep 429/529 on transient cooldown/rotation behavior with targeted regression tests for scheduler-dispatched `/v1/chat/completions` failover.
