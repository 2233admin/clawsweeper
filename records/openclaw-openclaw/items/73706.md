## Summary

Keep this PR open. Current main still drops the canonical shared outbound session key before `message_sending`, and the PR is a focused implementation candidate for that bug plus the native-command redirect session-key mismatch; the only concrete patch concern I found is a non-blocking doc overstatement around `runId`.

## What This Changes

The PR threads the canonical outbound `sessionKey` into shared `message_sending` hook contexts, aligns routed native-command replies with the agent runtime session key, and adds tests plus JSDoc for session-key versus policy-key semantics.

## Best Possible Solution

Land the narrow shared-outbound session-key plumbing and native redirect alignment after normal maintainer review, then either tighten the `runId` wording or track outbound `runId` threading under the broader hook-correlation request.

## Reproduction Assessment

Yes. A focused source-level reproduction on current main is `deliverOutboundPayloads` with a session and enabled `message_sending` hooks: the hook context is built without `sessionKey`; the native redirect mismatch is likewise visible by comparing `dispatch-from-config.ts` to `get-reply.ts`.

## Solution Assessment

Yes for the reported `sessionKey` plumbing and native redirect mismatch: the PR uses the existing canonical key and avoids falling back to `policyKey`. It is only partial for the broader `runId` contract, which should be narrowed or implemented separately.
