## Summary

Keep open. Current main has runtime-backed health snapshots, but plain `openclaw health --json` can still return a cached snapshot captured before channel startup, and Telegram still normalizes missing lifecycle state to `running=false` even when a probe succeeds. The latest discussion confirms the mismatch on `2026.4.27`, and open PR #55550 is overlapping but unmerged and Telegram-specific.

## What This Changes

The issue asks for `openclaw health --json` to report Telegram runtime health consistently with successful probes, `status --deep`, and `doctor`, or to document a narrower meaning for `running`.

## Best Possible Solution

Land a focused shared fix that refreshes or invalidates public health cache after channel startup/restart and keeps health JSON built from the same resolved account/runtime snapshot contract as `channels.status`. If `running` intentionally remains narrower than operator-visible health, add a separate machine-readable health/liveness field and document the distinction instead of letting probe success contradict `running=false`.

## Reproduction Assessment

Yes. Configure Telegram, restart the gateway, wait for `status --deep` or `doctor` to show Telegram OK, then compare plain `openclaw health --json`; the reporter confirmed that path on `2026.4.27`, and source shows the startup cache can be primed before `startChannels()` publishes runtime state.

## Solution Assessment

No. Current main's runtime-snapshot merge is a useful partial fix for refreshed payloads, and PR #55550 may help Telegram probe-derived summaries, but the narrowest maintainable solution should handle the shared cache invalidation/runtime snapshot contract so Slack and other channel variants do not keep diverging.
