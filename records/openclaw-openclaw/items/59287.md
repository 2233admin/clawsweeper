## Summary

Keep open. Current main has a partial improvement for fresh health snapshots, but the startup/cache timing path that can serve a pre-channel-start `running=false` snapshot from plain `openclaw health --json` still exists, and the reporter confirmed the mismatch on the latest release.

## What This Changes

The issue asks for `openclaw health --json` to stop reporting Telegram as `running=false` when the runtime is healthy and status/doctor probes show Telegram OK.

## Best Possible Solution

Refresh or invalidate the public health cache after channel startup completes, or otherwise prevent plain `health --json` from serving a channel summary captured before runtime state is populated. Keep the shared resolved-account/runtime snapshot path and add regression coverage that compares health JSON with `channels status --probe --json` for Telegram and the reported Slack credential-source variant.

## Reproduction Assessment

Yes. Configure Telegram, restart the gateway, let `status --deep` or doctor show Telegram OK, then compare plain `openclaw health --json` while the fresh startup health cache can still contain the pre-channel-start runtime snapshot; current source still has that ordering.

## Solution Assessment

No. The current-main snapshot-builder change is a good part of the fix for refreshed payloads, but it is not sufficient while startup can prime and serve a stale cached health snapshot before `startChannels()` publishes runtime state.
