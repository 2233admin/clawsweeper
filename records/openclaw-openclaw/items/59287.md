## Summary

Keep open. Current main still has the reported health/status mismatch class: plain `openclaw health --json` is a cache-eligible gateway health RPC, and the health summary still builds channel summaries from runtime-safe fields rather than the richer per-account snapshot path used by `channels.status`. Runtime-backed `running` can be correct after channel startup, but a cached pre-channel-start health snapshot can still report `running=false`, and Telegram/Slack credential source fields can still fall back to `none` even when the probe succeeds.

## Best Possible Solution

Keep this issue open. The best fix is to make gateway health use the same runtime-plus-resolved-account snapshot contract as `channels.status`, or refresh/replace the primed health cache after channel startup completes. Add regression coverage comparing `health --json`, live/probe health, and `channels status --probe --json` for configured running Telegram, including `running`, `lastStartAt`, and `tokenSource`; include analogous Slack credential-source coverage because the discussion reports the same class there.
