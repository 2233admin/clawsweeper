## Summary

Keep open. Current main still has the reported health/status mismatch class. Runtime-backed `running` can be correct after channel startup, but plain `openclaw health --json` remains cache-eligible and can serve a pre-channel-start snapshot. The health summary also still bypasses the richer per-account snapshot builder used by `channels.status`, so Telegram `tokenSource` and the Slack credential-source variant can still fall back to `none` even when probes succeed.

## Best Possible Solution

Keep this issue open and fix the shared gateway health contract. The best implementation is to make health build channel account summaries through the same runtime-plus-resolved-account snapshot path used by `channels.status`, or otherwise carry safe credential-source metadata into runtime-backed health snapshots, and refresh or replace the primed health cache after channel startup completes. Add regression coverage comparing `health --json`, live/probe health, and `channels status --probe --json` for configured running Telegram, plus analogous Slack credential-source coverage from the issue discussion.
