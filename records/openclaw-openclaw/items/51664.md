## Summary

Keep this issue open. Current main still wires WebChat code-block Copy directly to `navigator.clipboard.writeText` and ignores rejected writes, the renderer still emits that affected button path, and the latest release tag still has the same behavior. The related fallback PRs were closed unmerged, so they are useful context but not a fix on main.

## Best Possible Solution

Keep the issue open and implement a shared Control UI clipboard helper for WebChat copy actions. The helper should try `navigator.clipboard.writeText`, handle missing or rejected Clipboard API calls, fall back to a browser-supported temporary-textarea/`document.execCommand('copy')` path where appropriate, and report success or failure. Wire code-block Copy through it, reuse it for adjacent Control UI copy buttons if practical, add focused tests for rejected Clipboard API calls and fallback behavior, then do an Edge-on-Windows smoke check.
