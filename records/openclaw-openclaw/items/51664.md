## Summary

Keep this issue open. Current main still wires WebChat code-block Copy directly to `navigator.clipboard.writeText(code)` with an empty rejection handler, the markdown renderer still emits that affected `.code-block-copy` path, focused coverage only checks that the button renders, and latest release `v2026.4.26` has the same behavior. The related fallback PRs #51746 and #52010 were closed unmerged, so they are useful context but not a shipped fix.

## Best Possible Solution

Keep the issue open and implement a shared Control UI clipboard helper for WebChat copy actions. The helper should try `navigator.clipboard.writeText`, handle missing or rejected Clipboard API calls, fall back to a temporary textarea plus `document.execCommand('copy')` where appropriate, and return success/failure. Wire WebChat code-block Copy through it, add focused tests for rejected Clipboard API calls and fallback behavior, and do an Edge-on-Windows smoke check before closing.
