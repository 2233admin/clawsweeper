## Summary

Keep open. The PR addresses real remaining behavior on current main: bundled channel entry load failures are still logged as one flat warning, while the lock-timeout throw side has a stable timeout message that can support structured classification. The patch is small and CI is green, with one low-severity parser edge worth maintainer follow-up.

## What This Changes

This PR adds file-local bundled runtime-deps lock-timeout detection and structured warning metadata in `src/channels/plugins/bundled.ts`, plus new tests for lock-timeout and generic import-error warning behavior.

## Best Possible Solution

Land the narrow observability fix after maintainer review, ideally with the parser adjusted to split at ` (waited=` so structured `lockDir` remains reliable, while keeping broader runtime-deps recovery work tracked separately under #73874.

## Reproduction Assessment

Yes. The PR includes a focused reproduction path by mocking `loadChannelPluginModule` to throw the formatted lock-timeout error and verifying `getBundledChannelPlugin` returns undefined while the warning carries structured metadata; current main source confirms the old flat-string branch.

## Solution Assessment

Mostly yes. File-local catch-side classification is a narrow maintainable fix that preserves existing skip behavior, but parsing `lockDir` using the waited marker would be safer than `indexOf(" (")`.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.86

Full review comments:

- **[P3] Parse the lock path using the waited marker:** `src/channels/plugins/bundled.ts:298-299`
  - body: `indexOf(" (")` stops at the first space-parenthesis sequence after the prefix, so a valid install path such as `/tmp/OpenClaw (prod)/...` would emit a truncated `lockDir` while still classifying the timeout. Since this PR adds `lockDir` specifically for structured logging, parse up to the stable ` (waited=` marker or otherwise avoid returning a misleading partial path.
  - confidence: 0.84
