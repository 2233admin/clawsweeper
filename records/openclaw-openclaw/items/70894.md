## Summary

Keep open. Current main still routes approved `/hooks/agent` session keys into isolated cron runs with `forceNew: true`, and the issue is already paired with open, unmerged PR #70895 using closing syntax.

## What This Changes

The issue asks for webhook-triggered agent runs with an approved stable `sessionKey` to preserve conversation context across calls instead of starting a fresh isolated session every time.

## Best Possible Solution

Review and land PR #70895, or an equivalent narrow change, so webhook-sourced isolated runs with policy-approved stable session keys can reuse fresh sessions while ordinary scheduled isolated cron jobs continue to start fresh sessions.

## Reproduction Assessment

Yes. Repeated approved `/hooks/agent` calls with the same `sessionKey` flow into an isolated cron job, and current main still passes `forceNew: true` before `resolveCronSession` can reuse an existing stored session.

## Solution Assessment

Yes. The narrow maintainable fix is to derive `forceNew` from both isolated cron target and hook/webhook provenance, rather than making all isolated cron jobs reusable or weakening hook session-key policy.
