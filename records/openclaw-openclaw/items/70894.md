## Summary

Keep open. Current main still sends approved `/hooks/agent` calls through the isolated cron runner with `forceNew: true`, so stable, policy-allowed `sessionKey` values still cannot preserve webhook conversation state; open PR #70895 remains the paired fix candidate and should be reviewed, landed, or replaced before this issue closes.

## What This Changes

Webhook `/hooks/agent` runs should reuse fresh sessions for policy-approved stable `sessionKey` values instead of forcing every invocation into a new agent session.

## Best Possible Solution

Review and land PR #70895 or a narrow equivalent that derives `forceNew` from hook/webhook provenance instead of `sessionTarget` alone, while keeping normal scheduled isolated cron runs fresh and adding regression coverage for both paths.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: send repeated approved `/hooks/agent` requests with the same stable `sessionKey`, and the current path forwards that key into an isolated cron job that always calls `resolveCronSession` with `forceNew: true`.

## Solution Assessment

Yes. The narrow maintainable fix is to allow only hook/webhook-sourced isolated runs with approved stable keys to reuse fresh sessions, rather than making all isolated cron runs reusable or changing hook session-key policy.
