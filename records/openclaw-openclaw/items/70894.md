## Summary

Current main still appears to reproduce the reported webhook session-continuity bug. `/hooks/agent` policy-checks and forwards a caller/session-config session key with `externalContentSource: "webhook"`, but the isolated runner still passes `forceNew: input.job.sessionTarget === "isolated"` into `resolveCronSession`; that resolver only reuses prior history when `forceNew` is false. The issue is also paired with open PR #70895 using closing syntax, so it should stay open until maintainers review, land, or replace that fix.

## Best Possible Solution

Keep this issue open and review/land PR #70895 or a narrow equivalent. The fix should preserve forced fresh sessions for true scheduled isolated cron jobs while allowing policy-approved webhook/hook-sourced isolated turns with stable session keys to reuse fresh existing sessions, with regression coverage for both paths.
