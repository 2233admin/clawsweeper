## Summary

Keep open. Current main has meaningful reliability work around Discord outbound delivery: normal Discord sends use REST, retry 429/5xx failures, and shared outbound delivery has a write-ahead queue with startup recovery. However I did not find the requested Discord reconnect/READY-triggered drain equivalent to WhatsApp. The only reconnect drain hook currently selects WhatsApp queue entries, while isolated cron delivery intentionally bypasses the write-ahead queue and relies on a short in-process transient retry loop. That makes the exact requested parity with WhatsApp still unimplemented or at least not proven implemented for Discord reconnect windows.

## Best Possible Solution

Keep this issue open until Discord has either a proven reconnect-window reproduction on current main or a targeted implementation. If the bug still reproduces, the likely fix is a Discord-owned reconnect/READY drain using the existing delivery-queue drain helper for Discord entries, plus focused tests that preserve cron idempotency and do not replay already-successful sends. If live/current-main testing shows REST delivery is unaffected by gateway reconnects, document that behavior and close with that evidence.

## Risks / Open Questions

- Discord ordinary message delivery is REST-based on current main, so a maintainer should first confirm whether WebSocket reconnect still creates a real outbound failure mode before adding new queue behavior.
- Cron direct delivery intentionally bypasses the write-ahead queue to avoid duplicate proactive sends; any durable Discord reconnect queue must preserve that idempotency protection.
