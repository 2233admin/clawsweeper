## Summary

Keep open. Current main has a shared write-ahead outbound delivery queue and startup recovery, but Discord still has no owner-side reconnect drain that replays Discord queue entries after the gateway reaches READY/RESUMED; the same-day ClawSweeper discussion identified the same remaining gap, and the current source confirms it at c4a4c189f11e1a41b9245b10c3169a22dc5b16ca.

## What This Changes

Request that Discord outbound deliveries failed during a gateway WebSocket reconnect be queued and replayed after reconnect, matching WhatsApp reconnect behavior.

## Best Possible Solution

Keep the shared delivery queue in core and add a Discord-owned reconnect drain in the Discord plugin lifecycle, scoped by channel and account, using the existing plugin SDK delivery-queue runtime. Update focused tests and user-facing changelog/docs only if the behavior or config surface changes.

## Reproduction Assessment

Yes for a focused source-level regression: enqueue or fail a queued delivery with channel="discord", drive the Discord gateway lifecycle through READY/reconnect, and observe that no drainPendingDeliveries call currently occurs. A live Discord disconnect reproduction was not run.

## Solution Assessment

Yes, a Discord-plugin reconnect drain using the existing delivery-queue runtime is the narrowest maintainable fix. A core migration or broad new config surface is unnecessary unless implementation discovers a missing generic SDK hook.
