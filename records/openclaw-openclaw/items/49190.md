## Summary

Keep open. Current main still has only `ok | error | skipped` cron run statuses, unresolved/non-best-effort announce delivery failures still return `status: "error"`, and Control UI/protocol filters do not accept or render `warning`. Related #64413 is adjacent delivery-reason normalization, not a replacement for this status/UI split.

## Best Possible Solution

Implement the status split in the cron-owned surface: add either a first-class `warning` run status or equivalent execution/delivery outcome fields that surface as warning in API/UI; update cron result production, `applyJobResult`, run-log filtering/persistence, gateway protocol schemas/generated clients, Control UI labels/styles/filters, docs, and focused tests. Coordinate reason-code wording with #64413 without treating it as a replacement.

## Clownfish Work Prompt

Fix this issue by implementing a cron warning outcome for execution-success plus delivery-failure cases. Keep the change scoped to the cron-owned status/delivery surface.

Expected behavior: when an isolated cron agent turn executes successfully and final announce/direct delivery fails or cannot resolve a target, record and expose a warning outcome rather than an execution error. True execution/model/tool failures must remain `error`; skipped provider preflight remains `skipped`; successful execution with successful or not-requested delivery remains `ok`. Delivery diagnostics such as `lastDeliveryStatus`, `lastDeliveryError`, `lastDelivered`, run-log `deliveryStatus`, and delivery traces should remain populated.

Likely implementation areas: update `CronRunStatus` and cron state/run outcome types in `src/cron/types.ts`; update delivery-failure return paths in `src/cron/isolated-agent/delivery-dispatch.ts`; add warning handling to `applyJobResult` in `src/cron/service/timer.ts` so delivery warnings do not increment `consecutiveErrors`, do not trigger execution failure backoff, and still schedule future runs like successful executions; update run-log parsing/filtering in `src/cron/run-log.ts`; update gateway protocol schemas in `src/gateway/protocol/schema/cron.ts` and any generated clients/types required by the repo; update Control UI types, status filters, labels, and styling in `ui/src/ui/types.ts`, `ui/src/ui/controllers/cron.ts`, `ui/src/ui/views/cron.ts`, `ui/src/i18n/locales/en.ts`, and relevant CSS. Do not hand-edit generated non-English locale bundles unless running the repo's i18n sync is part of the PR.

Tests: update existing tests that currently assert `status: "error"` for delivery-target/direct delivery-only failures, add `applyJobResult` coverage proving warning does not increment execution errors/backoff/failure alerts, add protocol validator coverage for warning status and filters, add run-log filter coverage, and add Control UI/controller coverage for warning labels/filters. Coordinate terminology with #64413 but do not try to solve full delivery reason normalization in this PR.

Changelog: add a single-line user-facing fix entry under the active version with `Thanks @NorthernDream` and include @Hollychou924 if using their implementation-path comment materially. Do not change unrelated cron retry, provider preflight, or channel delivery policy behavior.
