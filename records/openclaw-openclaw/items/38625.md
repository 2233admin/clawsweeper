## Summary

Current main still only partially overlaps the Crab fleet-monitoring request. WebSocket `hello-ok` exposes `server.version` and admin-scoped `update.run` exists, but `/health` and `/healthz` still return only liveness status, there is no `POST /admin/update` HTTP route, and the typed update RPC still rejects target-version fields.

## Best Possible Solution

Keep this issue open until maintainers either implement the requested HTTP API for Crab-style fleet tooling or explicitly document a WS-only management direction. A conservative path would add `version` to `/health` and `/healthz` without forcing config load, update probe tests/docs, then decide whether `POST /admin/update` should exist as an authenticated and rate-limited wrapper over `update.run`. If version pinning is accepted, extend the typed update schema and runner target contract rather than adding a one-off HTTP-only parameter.

## Risks / Open Questions

- Adding `POST /admin/update` is security-sensitive because it would expose host package update and restart orchestration over HTTP; it should reuse existing admin scopes, token auth, rate limiting, audit logging, restart coalescing, and update-runner semantics.
- Adding `version` to unauthenticated `/health` and `/healthz` may intentionally trade fleet observability for version disclosure; maintainers should decide that product/security policy explicitly.
- The CLI/update runner has internal tag/dev-target concepts, but public `update.run` does not expose version pinning. Adding `target_version` only to HTTP would split the control-plane contract.
