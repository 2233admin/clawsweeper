## Summary

Keep open. Current main still leaves Feishu OpenChat/topic-group session classification unresolved: outbound bare `oc_` targets default to direct sessions, the inbound classifier only treats `group`/`topic_group` chat types as group, and the session store does not merge direct/group variants for the same chat id. The safe fix needs maintainer judgment because prior merged work explicitly found that `oc_` can also represent p2p chats, and the related open PR already has blocking review feedback about that contract.

## What This Changes

The issue asks OpenClaw to use one Feishu session for `oc_` OpenChat/topic-group chats across proactive outbound sends and inbound webhook messages instead of splitting them into direct and group session keys.

## Best Possible Solution

Keep the issue open and resolve the Feishu-owned chat classification contract first. The maintainable end state is one shared Feishu classifier used by inbound routing, mention-forward detection, and outbound session routing, with explicit `chat:`/`group:` targets preserved and any legacy duplicate-session cleanup handled by doctor/fix repair rather than startup migration.

## Reproduction Assessment

Unclear for the full live Feishu duplicate without a current event payload from the reporter, but the static reproduction path is clear: current main routes a bare outbound `oc_` target as direct and can separately record group-shaped Feishu sessions. The missing piece is a verified discriminator for OpenChat topic groups that does not misclassify true p2p `oc_` chats.

## Solution Assessment

No, the suggested blanket `oc_` prefix correction is not the best fix because it conflicts with prior shipped behavior that treats bare `oc_` as ambiguous. A safer fix should use a verified Feishu signal, then apply the result consistently across inbound, outbound, and session metadata paths.
