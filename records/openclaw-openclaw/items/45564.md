## Summary

Keep open. Current main partially addresses the accidental-reset problem for the WebChat New Session button, but the broader request remains open: typed `/new` and `/reset` are still intentionally sent without confirmation, the gateway reset handler still acknowledges bare reset commands immediately, and there is no `session.confirmWipe`/confirm-steps style config. The related WebChat-only issue #45800 is now closed and PR #45863 is closed unmerged, so it should not be treated as the canonical tracker for the remaining cross-command behavior.

## Best Possible Solution

Keep this issue open for maintainer product review. If accepted, implement one central confirmation flow for destructive bare `/new` and `/reset` commands, likely in the gateway reset-command path with UI/channel helpers only where needed; preserve `/reset soft`, ACP-bound reset behavior, and reset-with-tail semantics deliberately. Add focused tests for first command creating a pending confirmation, second command within the timeout executing, timeout/cancel behavior, WebChat typed commands, and at least one channel/native-command path. Update slash-command/config docs if a `session.confirmWipe` style setting is added.
