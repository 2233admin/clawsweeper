## Summary

Keep open. Current main still lacks cursor pagination fields on both requested Gateway RPCs: `sessions.list` has strict params without `cursor`, slices only by `limit`, and returns no `nextCursor` or `total`; `agents.files.list` only accepts `agentId` and returns a fixed bootstrap/MEMORY file set. The referenced cursor helper work in #75208 is still an open, unmerged PR, so this needs maintainer sequencing and scope review rather than cleanup closure.

## What This Changes

Request to add cursor-based pagination fields to `agents.files.list` and `sessions.list`, reusing the cursor helper pattern proposed in #75208.

## Best Possible Solution

Settle the shared cursor contract from #75208 first, then add additive cursor pagination to `sessions.list` across schema, runtime, SDK/client types, generated artifacts, docs, and focused tests. For `agents.files.list`, either add the same fields for protocol consistency after maintainer approval or explicitly document why the fixed-size file set remains unpaginated.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence API-shape reproduction: the strict schemas reject `cursor` on both RPCs, and `sessions.list` only truncates with `limit` without returning `nextCursor` or `total`.

## Solution Assessment

Unclear for the full requested scope. Cursor pagination is a maintainable direction for `sessions.list`, but applying it to `agents.files.list` should wait for maintainer scope and sequencing decisions because that RPC is currently fixed-scope and the proposed cursor helper is still in #75208.
