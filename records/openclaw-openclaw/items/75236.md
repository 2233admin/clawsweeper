## Summary

Keep open. Current main still has only limit-based `sessions.list` behavior and no pagination fields on `agents.files.list`; the related cursor helper from #75208 is not on main, and the `agents.files.list` scope needs maintainer judgment because it currently returns a fixed bootstrap/memory file set rather than an unbounded workspace listing.

## What This Changes

Add cursor pagination fields to `agents.files.list` and `sessions.list`, following the `artifacts.list` cursor pattern proposed in #75208.

## Best Possible Solution

Settle the shared cursor contract from #75208 first, then add cursor pagination to `sessions.list` across schema, runtime, SDK/client types, generated artifacts, docs, and focused tests. Apply the same fields to `agents.files.list` only if maintainers want protocol consistency despite its fixed-size file set, or explicitly document why it remains unpaginated.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence API-shape reproduction: current strict schemas reject `cursor` on both RPCs, and `sessions.list` only slices by `limit` without returning `nextCursor` or `total`.

## Solution Assessment

Unclear for the exact requested scope. Cursor pagination is a maintainable direction for `sessions.list`, but applying it to `agents.files.list` should wait for maintainer scope and sequencing decisions because that RPC is currently fixed-scope and #75208 is still open.
