## Summary

Keep open. Current main still lacks cursor pagination for `sessions.list`, and the `agents.files.list` schema also lacks the requested pagination fields, although that RPC currently returns a fixed bootstrap/memory file set rather than arbitrary workspace files. Because the request depends on the still-open artifact pagination work in #75208 and changes public Gateway protocol shape, maintainer sequencing and scope should come before an automated repair.

## What This Changes

Request to add cursor-style `cursor`, `limit`, `nextCursor`, and `total` pagination to `agents.files.list` and `sessions.list`, following the artifact pagination pattern proposed in #75208.

## Best Possible Solution

Settle the shared cursor contract from #75208 first, then add cursor pagination to `sessions.list` with schema, runtime, SDK/client, docs, and focused tests. Apply the same fields to `agents.files.list` only if maintainers want protocol consistency despite its current fixed-size file set, or document why that RPC intentionally remains unpaginated.

## Reproduction Assessment

Yes for the missing API shape: current schemas reject `cursor` on both RPCs because `additionalProperties` is false, and `sessions.list` returns only a limited first page with no `nextCursor` or total. This is a feature/API gap rather than a runtime crash reproduction.

## Solution Assessment

Unclear for the exact requested scope. Cursor pagination is a maintainable direction for `sessions.list`, but applying it to `agents.files.list` needs maintainer judgment because the current implementation is fixed to bootstrap/memory files and not an unbounded workspace scan.
