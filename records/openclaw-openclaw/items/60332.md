## Summary

Keep this issue open. Current main still displays `/status` and `session_status` compactions from the persisted session counter only, while third-party `ownsCompaction` context engines can do proactive/background compaction through `afterTurn` or maintenance without any plugin-facing count/stat signal for core to persist.

## Best Possible Solution

Keep this open and add an additive plugin-facing accounting path: for example, let context engines return/report per-session compaction deltas from `afterTurn` or maintenance, or add an optional stats API. Persist those deltas through the existing `incrementRunCompactionCount` session-store path and add regression coverage proving an `ownsCompaction` engine’s proactive compaction is reflected in both `/status` and `session_status`.
