## Summary

Keep open. Current main still resolves PDF tool execution through the registry-only media helper, so the config-defined custom-provider failure described by linked #65742 is not implemented on main. This PR is still a plausible focused fix, but the proposed fallback should address the existing normalized provider-alias review concern before landing.

## Best Possible Solution

Keep this PR open or have a maintainer take it over. The best fix is a PDF/media model resolver that preserves the current normalized registry lookup, falls back to `cfg.models.providers` using `findInlineModelMatch`-equivalent normalized provider/model matching, and adds regression coverage for both a registry-miss custom provider and a provider-alias custom provider.

## Risks / Open Questions

- Landing the current PR head verbatim may still fail for alias-key custom providers because the proposed inline fallback matches provider strings exactly instead of normalizing them.
- Closing this PR without a landed replacement would leave the PDF custom-provider registry-miss gap effectively orphaned, because #65742 was closed as superseded by this PR.
- Related broader work such as #53738 should be coordinated with this fix, but it is not a shipped replacement for the narrow PDF failure.
