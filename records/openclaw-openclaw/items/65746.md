## Summary

Keep this PR open. Current main still has the registry-only PDF model lookup that the PR is trying to fix, but the submitted patch drops existing normalized registry behavior and only matches inline provider models by exact provider id, so it needs a narrow repair before merge.

## What This Changes

The PR adds a PDF-only fallback from runtime registry lookup to `models.providers` inline custom provider models and adds a regression test for config-defined custom provider model resolution.

## Best Possible Solution

Land a config-aware PDF model resolver that keeps the existing normalized registry lookup first, then falls back to `models.providers` using the same normalized provider and provider-scoped model matching semantics as the embedded runner, with focused tests for local custom providers, provider aliases, and provider-prefixed model ids.
