## Summary

Keep open. Current main still does not implement a `/restore` command, and this PR is a relevant implementation candidate, but the latest head still has blocking transcript ownership and reset-destination bugs that need revision before merge.

## What This Changes

This PR adds a built-in `/restore [number]` command, archived transcript listing/restoration helpers, sessionKey transcript headers in selected creation paths, focused tests, and slash-command docs.

## Best Possible Solution

Revise the branch so restore moves the selected archive into `resetResult.entry.sessionFile`, validates archive ownership before listing and restoring, handles keyless legacy archives through an explicit recovery policy, scans until enough matching archives are collected under a bounded strategy, and has a single reset-hook owner with tests for persisted session files, thread/topic sessions, keyless/corrupt/long headers, and multi-channel isolation.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: mock `performGatewaySessionReset` to return an `entry.sessionFile` that differs from `resolveSessionTranscriptPathInDir(newSessionId, sessionsDir)`, or place a keyless/corrupt archive from another channel in the sessions directory and call `listRecentArchives` for the current channel.

## Solution Assessment

No. The feature direction is reasonable, but this PR is not yet the safest maintainable implementation because it bypasses the reset service's persisted destination and treats unknown archive ownership as a match.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Restore into the reset entry transcript file:** `src/gateway/session-restore.ts:246-252`
  - body: `performGatewaySessionReset` persists the active transcript path in `resetResult.entry.sessionFile`, and reset can preserve an existing `currentEntry.sessionFile`. This derives a fresh path from `newSessionId`, so restore can move the archive to a file the session store does not read while reporting success. Use and validate `resetResult.entry.sessionFile` as the destination.
  - confidence: 0.94
- **[P1] Require an owned archive header before listing it:** `src/gateway/session-restore.ts:198-203`
  - body: `readSessionKeyFromHeader` returns `null` for missing keys, parse failures, truncation, and read errors, and this condition accepts `null` for any requested `sessionKey`. That can expose or restore another channel's transcript. Skip unreadable/keyless archives unless a deliberate legacy recovery policy proves ownership.
  - confidence: 0.92
- **[P2] Apply the scan limit after sessionKey filtering:** `src/gateway/session-restore.ts:191-193`
  - body: The scan ceiling is applied before the `sessionKey` filter, so newer archives from other channels can consume the whole window and make valid current-channel history disappear. Bound total work separately, but continue until enough matching archives are collected.
  - confidence: 0.86
- **[P2] Let one restore path emit reset hooks:** `src/auto-reply/reply/commands-restore.ts:103-116`
  - body: The handler emits reset command hooks here, then `performSessionRestore` immediately calls `performGatewaySessionReset`, which triggers the internal command hook path again. Hook consumers can see duplicate side effects or inconsistent pre-reset snapshots; route restore through a single reset hook owner.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: Security review needs attention: the archive ownership filter can cross the sessionKey confidentiality boundary for archived chat transcripts.

Concerns:

- **[high] Cross-channel transcript restore via unknown archive ownership:** `src/gateway/session-restore.ts:201`
  - body: Archive header parsing collapses missing `sessionKey`, parse failures, truncation, and read errors to `null`, and `listRecentArchives` treats `null` as visible to any requested `sessionKey`. In a multi-channel instance, an authorized user in one channel can be offered another channel's archived transcript when that archive is legacy, corrupt, or has a long header.
  - confidence: 0.92
