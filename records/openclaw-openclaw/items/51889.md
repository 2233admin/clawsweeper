## Summary

Keep open. Current main still does not provide the requested chat restore command, and the latest PR branch still needs revision before merge because restore can write to a transcript path the session store does not read and can cross the sessionKey isolation boundary for keyless or unreadable archives.

## What This Changes

The PR adds a built-in `/restore [number]` chat command, archive listing/restoration helpers, sessionKey transcript headers, docs, and focused tests for archived session restoration.

## Best Possible Solution

Rework the branch into a first-party restore flow that restores into the validated `resetResult.entry.sessionFile`, enforces a private and explicit legacy-archive policy, scans archives until it has enough matching entries under a bounded policy, lets one reset path own hook emission, and adds coverage for multi-channel isolation, persisted sessionFile paths, topic/thread sessions, corrupt or long headers, and archive-list drift.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Restore into the reset entry session file:** `src/gateway/session-restore.ts:248-252`
  - body: `performGatewaySessionReset` has already persisted the active entry and may preserve an existing `entry.sessionFile`. Deriving `targetPath` from the new session id and topic can move the archive into a file the session store does not read, so `/restore` can report success while the next turn starts from the reset-created transcript. Use and validate `resetResult.entry.sessionFile` as the restore destination.
  - confidence: 0.93
- **[P1] Do not accept keyless archives for every channel:** `src/gateway/session-restore.ts:201`
  - body: `readSessionKeyFromHeader` returns `null` for missing keys, parse failures, truncated headers, and read errors, and this condition accepts `null` for any requested sessionKey. In a multi-channel instance, an authorized user in one channel can be offered or restore another channel's archive. Distinguish legacy no-key archives from unreadable headers and require a safe ownership policy before showing them.
  - confidence: 0.91
- **[P2] Continue scanning until enough matching archives are found:** `src/gateway/session-restore.ts:193`
  - body: The scan ceiling is applied before sessionKey filtering, so newer archives from other channels can consume the whole scan window and make matching current-channel history disappear. Apply `limit` after filtering, or use a bounded scan policy that still searches until enough matching archives are found.
  - confidence: 0.86
- **[P2] Avoid firing reset hooks twice during restore:** `src/auto-reply/reply/commands-restore.ts:104-113`
  - body: This calls `emitResetCommandHooks`, then `performSessionRestore` immediately calls `performGatewaySessionReset`, which also triggers command and before-reset hooks. Hook consumers can observe duplicate reset side effects or inconsistent pre-reset snapshots; let one reset path own hook emission for restore.
  - confidence: 0.82
