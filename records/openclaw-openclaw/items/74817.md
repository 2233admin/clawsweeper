## Summary

Keep open. Current main still has no Feishu `allowBots`/`ignoreOtherBots`-style policy or sender-type filter, but the reporter's follow-up also says Feishu may not deliver bot-to-bot events at all, so the remaining work needs maintainer product judgment rather than an automatic close. The `@all` half is already intentionally documented and tested as not being a bot mention.

## What This Changes

The issue requests Feishu handling for broadcast `@所有人` mentions and a way to avoid multi-bot group responses or bot-to-bot loops when `requireMention:false` is configured.

## Best Possible Solution

Treat this as two decisions: keep broadcast-only `@all` documented as a Feishu/platform limitation unless payload evidence proves OpenClaw can detect it, and decide whether the bundled Feishu plugin should add a Slack/Discord-style `allowBots` policy for real app-authored inbound events or leave relay behavior to an external plugin/fork.

## Reproduction Assessment

Unclear. Static inspection shows a synthetic Feishu group event with `sender_type:"app"` would not be filtered before dispatch, but the reporter's later research says Feishu does not deliver bot-to-bot receive events; broadcast `@all` behavior is reproducible in current tests as intentionally not a bot mention.

## Solution Assessment

Unclear. A narrow Feishu-local `allowBots`/bot-sender policy is the smallest maintainable fix if app-authored events reach OpenClaw, while the linked relay design is a broader optional feature that should not be added without maintainer product approval.
