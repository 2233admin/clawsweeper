## Summary

Keep open. Current main still intentionally excludes Feishu broadcast-only @all from bot mention detection, has no documented or schema-backed ignore-other-bots policy, and the reporter's follow-up reframes the bot loop as a Feishu platform limitation plus optional relay feature rather than a cleanup-closeable core bug.

## What This Changes

The issue requests Feishu handling for group-wide @all mentions and a configurable way to avoid or replace bot-to-bot echo behavior when requireMention is false.

## Best Possible Solution

Keep #75001 as the focused @all semantics thread, then make an explicit Feishu product decision on the bot-to-bot side: either add a compatibility-preserving app-sender suppression policy if Feishu delivers those events, or treat synthetic relay/shared-history behavior as a separate Feishu plugin/fork feature with clear safety boundaries.

## Reproduction Assessment

Unclear. Static source and tests reproduce the current @all suppression and missing ignore-other-bots surface, but the bot-to-bot echo loop needs a live Feishu im.message.receive_v1 payload or log because the reporter later says Feishu suppresses bot-to-bot delivery at the platform layer.

## Solution Assessment

Unclear. A narrow ignoreOtherBots flag is only useful if Feishu actually delivers bot-authored inbound events, while synthetic Bot Relay is a larger product and trust-boundary choice; the @all half should be handled through #75001.
