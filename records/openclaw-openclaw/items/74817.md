## Summary

Keep open. Current main still has a real Feishu gap for the multi-bot echo-loop half of the report: Feishu events expose `sender_type`, but inbound group dispatch does not filter `sender_type: "app"` before running a turn when mention gating is disabled. The `@all` half is already treated as a documented platform/semantics limitation rather than implemented behavior, so this should stay open for a narrow Feishu fix plus clearer docs.

## What This Changes

Request Feishu group handling to respond appropriately to `@所有人`/`@all` when possible and add a built-in way to ignore other bot messages when `requireMention` is false.

## Best Possible Solution

Add Feishu-local bot-authored message suppression, preferably matching the existing Slack/Discord `allowBots` default-false model with account/group inheritance, and document that broadcast-only `@all` is not a bot mention unless Feishu sends a payload that identifies the bot directly.

## Repair Work Prompt

Fix Feishu multi-bot echo loops from bot-authored group messages when `requireMention` is false. Mirror the existing Slack/Discord safety model where bot-authored inbound messages are ignored by default, with an explicit opt-in such as `allowBots` if maintainers need bot-to-bot workflows. Use Feishu's `event.sender.sender_type === "app"` signal, preserve current handling of the bot's own messages, and keep the change inside the Feishu plugin unless a generic SDK seam is clearly needed. Add account/group config schema support and docs if an opt-in is added. Do not make broadcast-only `@all` count as a bot mention unless the Feishu event payload explicitly identifies this bot; instead keep or improve the documented limitation for `@所有人`/`@all`. Add regression tests for dropping other app-authored group messages with `requireMention:false` and for the explicit opt-in path. Add a user-facing changelog fix entry with `Thanks @shao-zhijie`.
