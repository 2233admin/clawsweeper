## Summary

Keep open. Current main still does not expose the requested Feishu bot-message suppression policy, and the `@all` half is already tracked by the narrower open #75001. Current main has adjacent Feishu broadcast dispatch code, but it is not documented as a Feishu relay/ignore-bots feature and does not fully resolve the issue as reported.

## What This Changes

This issue requests Feishu support for `@all` delivery semantics plus a built-in way to prevent or route multi-bot group replies when `requireMention` is false.

## Best Possible Solution

Keep #75001 as the canonical `@all` behavior discussion, then make a Feishu product decision for multi-bot groups: either productize/document Feishu broadcast as the supported multi-agent route, or add a Feishu-owned app-sender policy by carrying `sender_type` into `FeishuMessageContext`, adding explicit account/group config with a compatibility-preserving default, and covering dispatch, schema, docs, and tests.

## Reproduction Assessment

Partly. Source and tests give a high-confidence static reproduction for broadcast-only `@all` not setting `mentionedBot`, but the bot-to-bot echo loop lacks a live Feishu event payload and the reporter's follow-up says the Feishu platform may not deliver bot-to-bot receive events at all.

## Solution Assessment

Unclear. A narrow `sender_type: "app"` filter would be maintainable if Feishu actually delivers app-authored inbound events, but the relay/broadcast direction and the open #75001 `@all` semantics need maintainer product direction before a cleanup close or automated fix.
