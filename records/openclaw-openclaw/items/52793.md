## Summary

Current main still has the reported coarse filesystem policy: `tools.fs.workspaceOnly` is a single boolean shared by read/write/edit and folded into apply_patch, with only a separate `tools.exec.applyPatch.workspaceOnly` override for apply_patch. There is no typed, schema-accepted, or runtime-threaded per-operation fs policy that can allow reads/search outside the workspace while keeping write/edit/apply_patch workspace-contained.

## Best Possible Solution

Keep this open. Implement a backward-compatible fs policy extension in core: preserve `tools.fs.workspaceOnly` as the coarse boolean, add an object/per-operation form or explicit read-only external roots, update config types, strict schema, generated schema/help, and public docs, then thread the richer policy through `src/agents/tool-fs-policy.ts`, `src/agents/pi-tools.ts`, plugin tool context, and media local-root handling as needed. Add regression coverage proving read/search-style access can reach allowed external roots while write/edit/apply_patch remain workspace-contained.
