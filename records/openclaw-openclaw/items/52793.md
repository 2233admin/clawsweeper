## Summary

Current main still has the reported coarse filesystem policy: `tools.fs.workspaceOnly` is a single boolean, the strict config schema only accepts that boolean, and runtime still threads one `workspaceOnly` flag through read, write, edit, media/local path handling, and apply_patch containment. There are active related implementation candidates, especially #52951 (`tools.fs.roots`) and #43565 (`tools.fs.allowedRoots`), so the best path is maintainer/security review rather than cleanup closure or a new autonomous fix PR.

## Best Possible Solution

Keep this issue open. Maintainers should review the active filesystem-policy PRs, especially #52951, and decide the preferred core contract: per-operation `workspaceOnly`, explicit external read/search roots, or access-mode roots. The implementation should preserve the legacy boolean, keep write/edit/apply_patch confined unless explicitly and safely widened, update config types/schema/generated docs/public docs, and add focused tests for read/search access outside the workspace while mutation tools remain contained.
