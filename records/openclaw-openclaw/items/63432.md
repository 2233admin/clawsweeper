## Summary

Current main now contains a targeted Control UI/WebChat attachment fix for this RangeError path: attachment bytes are kept out of Lit state and optimistic chat messages, object URL previews are used for display, and send-time serialization still forwards the payload to `chat.send`. The changelog explicitly records that fix as covering the related attachment stack-overflow report and referencing this report; it is on main but not in the latest shipped release yet.

## What This Changes

The issue asks for iPad Safari over Tailscale Control UI image attachment or pasted-image sends to succeed instead of failing with `RangeError: Maximum call stack size exceeded` while text-only sends work.

## Best Possible Solution

Keep the current side-store/object-URL attachment implementation. If iPad Safari over Tailscale still reproduces after a build containing the fix, open a fresh report with a full stack trace so maintainers can target the remaining post-resize path specifically.
