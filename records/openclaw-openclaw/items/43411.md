## Summary

Keep this PR open. Current main still does not implement the PR's central additive behavior for config-derived `logging.redactPatterns`; it only has additive handling for newer safety-boundary tool payload paths. The PR is still a focused redaction fix candidate, though it likely needs a current-main rebase and a docs/schema contract decision.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is to decide one public contract for config-derived `logging.redactPatterns`, implement a shared resolver that preserves default token/API-key/PEM patterns where intended, wire it through `resolveRedactOptions`, `redactToolDetail`, log tailing, and persisted transcript redaction, preserve explicit caller-supplied pattern semantics if that remains the API contract, then align docs/schema help and add regression tests proving custom config patterns do not drop default redaction across affected sinks.
