## Summary

Keep open. Current main still treats bound ACP follow-ups as direct ACP prompt text: bound conversation dispatch retargets the ACP session, `tryDispatchAcpReply` derives the prompt from the inbound message fields, and regression coverage asserts the raw follow-up text reaches `runtime.runTurn`. The public ACP docs also describe direct bound follow-up routing. A same-author PR, #69675, is open for docs/policy clarification, so this needs maintainer review rather than cleanup closure or a separate automated fix PR.

## Best Possible Solution

Keep this issue open and review the paired open PR #69675 first. The best path is a maintainer decision on the intended ACP semantics, then a focused change that aligns runtime routing, accepted-note wording, docs, and regression tests so parent-owned shorthand follow-ups are rewritten when appropriate while explicit interactive/verbatim ACP relay remains available.
