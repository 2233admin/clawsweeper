## Summary

Keep this issue open. Current main still builds the TUI Gateway URL from explicit URL/env/config/default port paths and does not consult the active running Gateway runtime port before selecting the local loopback target. The provided GitHub context also shows open target-repo PR #73338 with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review, land, or supersede #73338 with a small target-repo fix. The implementation should provide a trustworthy active local Gateway runtime-port source, make default TUI loopback resolution use it, and add regression coverage proving a Gateway active on 48789 makes openclaw tui resolve ws://127.0.0.1:48789 without weakening explicit URL, remote mode, auth, or stale-lock behavior.
