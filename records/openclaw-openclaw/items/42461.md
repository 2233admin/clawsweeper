## Summary

Keep this issue open. Current main still builds the TUI default Gateway URL from explicit URL/env/config/default-port resolution and does not consult an active running Gateway port before selecting the local loopback target. The provided context also shows open target-repo PR #73338 with closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review, land, or supersede #73338 with a small target-repo fix. The likely best implementation is to record a trustworthy active local Gateway runtime port, make default TUI loopback resolution use it before falling back to config/defaults, and add regression coverage proving a Gateway active on 48789 makes openclaw tui resolve ws://127.0.0.1:48789 while explicit URL and remote config still win.
