## Summary

Current main still appears to have the reported TUI custom-port regression. The TUI resolves its Gateway URL from explicit URL/env/config/default port paths, while the Gateway CLI can run with a runtime --port that is not persisted into lock metadata for the TUI to consume. The provided context also shows open target-repo PR #73338 with closing syntax for this issue, so this issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review, land, or supersede #73338 with a small fix that records a trustworthy active local Gateway runtime port, makes default TUI loopback resolution consult it before falling back to config/defaults, and adds regression coverage proving an active Gateway on 48789 makes openclaw tui resolve ws://127.0.0.1:48789 while explicit URL/env/remote config still win.
