## Summary

Keep this PR open. Current main still creates response-prefix context with only the agent identity, so `{model}`, `{provider}`, and `{thinkingLevel}` are only populated after the normal model-selection callback runs. The fast-abort path returns before that normal reply path, and the same-author linked bug report #45314 is still open. A newer draft PR #45454 also targets the same bug, but nothing has landed on main yet.

## Best Possible Solution

Keep this PR open until maintainers choose a canonical fix for #45314. The landed implementation should ensure stop/abort replies have a populated response-prefix context before dispatcher normalization, include focused regression coverage for templated abort prefixes, and either fix or explicitly scope follow-up work for session overrides, channel overrides, and catalog-derived thinking defaults.
