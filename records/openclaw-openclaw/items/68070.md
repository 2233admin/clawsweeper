## Summary

Keep this PR open, but do not treat the current patch as merge-ready: current main still lacks immediate non-retryable handling for parameter-validation failures, and the proposed `retryable` field is not consumed by the current Pi/OpenClaw runtime as a loop stop condition.

## What This Changes

The PR marks Pi tool parameter validation errors as non-retryable and propagates retryability metadata through the tool-definition adapter error-result payload.

## Best Possible Solution

Keep the PR active, but revise it toward a documented and tested validation-error contract that the runtime actually honors for missing or malformed tool parameters, while preserving ordinary tool-error behavior and existing loop guards.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Use a runtime-consumed stop signal for validation failures:** `src/agents/pi-tool-definition-adapter.ts:151`
  - body: Adding `retryable: false` to the JSON payload does not stop the Pi/OpenClaw loop. The pinned Pi runtime only uses `AgentToolResult.terminate` to end a tool batch, and the OpenClaw loop-control paths do not consume tool-result `retryable`, so a model that repeats the malformed call can still loop until the outer cap or optional loop detector fires.
  - confidence: 0.82
