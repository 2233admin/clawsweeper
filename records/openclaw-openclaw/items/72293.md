## Summary

Keep open. Current main still has only a partial fix: persisted session lifecycle state can map `stopReason: "aborted"` to a killed run, but parent-facing `agent.wait` and the subagent registry still classify a plain lifecycle `end` with only that stop reason as successful completion. The issue is also paired with an open linked fix PR, so the right next step is maintainer review of that PR or an explicit replacement, not closure.

## What This Changes

The issue requests that empty-output aborted subagent terminal turns be classified and announced as parent-visible failures instead of silently completing as done.

## Best Possible Solution

Land the linked fix PR or an equivalent narrow core change that treats final assistant `stopReason: "aborted"` as a terminal failed/killed outcome across lifecycle production, `agent.wait`, subagent registry completion, persisted session state, and announce formatting, with regression tests for an empty-content aborted assistant turn producing a parent-visible failure.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current `agent.wait` would classify it as `ok`, and the registry listener would complete the subagent run with `outcome.status: "ok"`.

## Solution Assessment

Yes, the best direction is a narrow lifecycle classification fix that maps aborted stop reasons into the existing failed/killed/error surfaces. Auto-retry policy should stay out of the initial fix unless maintainers explicitly want a broader provider-retry behavior change.
