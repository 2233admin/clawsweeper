## Summary

Keep this open. Current main still has a source-level path where a terminal lifecycle event with `stopReason: "aborted"` but no `aborted: true` is classified as successful by `agent.wait` and the direct subagent registry, while only session persistence maps that stop reason to a killed/aborted state. The linked fork PR remains open and unmerged, so the issue should stay open while that fix is reviewed, landed, or replaced.

## What This Changes

Treat empty-content aborted subagent terminal turns as parent-visible failures instead of successful done states, so status, `abortedLastRun`, and completion announcement behavior stay consistent.

## Best Possible Solution

Land or replace a narrow core fix that classifies final assistant `stopReason: "aborted"` as a failed or killed terminal outcome across lifecycle production, `agent.wait`, subagent registry completion, persisted session state, and parent-visible announce formatting, with regression tests for empty-content aborted assistant turns and stopReason-only lifecycle events.

## Reproduction Assessment

Yes. A source-level reproduction is to emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main records `agent.wait` as `ok` and the subagent registry as a successful completion.

## Solution Assessment

No current-main solution is complete. The narrow maintainable fix is to propagate the aborted terminal classification through all consumers while preserving the existing retry-grace behavior for transient abort markers.
