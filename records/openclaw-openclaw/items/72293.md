## Summary

Keep this issue open. Current main still treats a lifecycle end carrying only an aborted stop reason as successful in the parent-facing wait and direct subagent registry paths, even though session persistence can now map that stop reason to a killed/aborted-last-run state. The linked fork PR remains open and unmerged, so this issue should stay open while that patch is reviewed, replaced, or landed.

## What This Changes

The issue reports that aborted empty-content subagent terminal turns are classified as successful completion and fail to notify the parent.

## Best Possible Solution

Land or replace a narrow core fix that treats final assistant `stopReason=aborted` as a terminal failure/killed outcome consistently across lifecycle production, `agent.wait`, direct subagent registry completion, persisted session state, task/announce formatting, and regression tests for empty-content aborted assistant turns.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a lifecycle `end` event with `stopReason=aborted` and no `aborted: true`; current main persists that as killed in one helper but `agent.wait` returns `ok` and the subagent registry records an ok completion.

## Solution Assessment

No. Current main is only a partial solution; the right fix is the narrow classification change across all consumers, and the linked PR direction is plausible but needs maintainer review or replacement because the registry path still needs coverage.
