## Summary

Keep this issue open. Current main has partial progress, but the parent-visible paths are still inconsistent: lifecycle metadata can carry `stopReason: "aborted"` and persisted session state maps that to killed, while `agent.wait` and the subagent registry still treat a plain lifecycle `end` with only that stop reason as successful. The issue is also paired with an open, unmerged closing PR in a fork, so the right next step is maintainer review of that PR or a narrow replacement.

## What This Changes

The issue requests that aborted empty-output subagent runs be surfaced as failed or killed terminal outcomes, with a parent-visible announcement instead of being silently treated as done.

## Best Possible Solution

Land a narrow core fix that treats final assistant `stopReason: "aborted"` as a terminal failure or killed outcome across lifecycle emission, `agent.wait`, subagent registry completion, persisted session state, and announce formatting, with regression tests for an empty-content aborted assistant turn producing a parent-visible failed completion.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main records the stop reason but still returns `ok` from `agent.wait` and completes the subagent registry outcome as `ok`.

## Solution Assessment

No, current main is not yet the best solution because it only fixes the persisted session-state half. The narrow maintainable direction is to classify the same aborted terminal state consistently at both parent-facing consumers, while retaining retry grace for transient abort markers.
