## Summary

Keep this open. Current main still only partially handles final assistant `stopReason: "aborted"`: persisted session state maps it to a killed run, but the lifecycle producer, `agent.wait`, and direct subagent registry completion can still classify a stopReason-only abort as a successful `end`/`ok` outcome. The timeline also links an open, unmerged fork PR that uses closing syntax for this issue, so the maintainer path is to review, land, or replace that PR rather than close early.

## What This Changes

The issue asks OpenClaw to treat empty-content aborted subagent terminal turns as parent-visible failed/aborted completions instead of marking them done and dropping the announce.

## Best Possible Solution

Land or replace a narrow core fix that consistently treats final assistant `stopReason: "aborted"` as a parent-visible failure/killed outcome across lifecycle emission, `agent.wait`, direct subagent registry completion, announce formatting, and regression tests for empty-content aborted assistant turns. Avoid adding a new public status union unless maintainers explicitly want that contract change.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to emit a lifecycle `end` event with `stopReason: "aborted"` and no `aborted: true`; current main persists that as killed in the session helper but `agent.wait` and the subagent registry still classify it as `ok`.

## Solution Assessment

No, current main is not the best solution because it only fixes persisted session state. The safer fix is to classify stopReason-only aborts as an existing failure/killed outcome through the lifecycle consumer paths and announce a failure/no-result completion instead of auto-retrying by default.
