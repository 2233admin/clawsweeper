## Summary

Keep this PR open. Current main still lacks bot-specific inbound debounce, and the latest PR branch is a useful focused start, but the proposed feature remains incomplete because no production channel path passes bot-origin state into debounce resolution. The earlier generated-schema drift comments appear addressed by later PR commits, so the remaining blocker is feature reachability rather than metadata parity.

## What This Changes

The PR adds `messages.inbound.botDebounceMs` to config types, schema, generated metadata, and help labels, updates `resolveInboundDebounceMs` to prefer it when `senderIsBot` is true, and adds resolver unit tests.

## Best Possible Solution

Keep this PR open and either land it explicitly as preparatory config/API work with docs/changelog and a tracked follow-up, or complete the product path in the same branch by threading reliable per-message bot-origin state through the shared debounce helper and direct channel resolver call sites, with focused tests for Slack, Discord, Telegram, WhatsApp, and any other supported bot-aware channels.
