## Summary

Keep open. Current main still leaves same-model OpenAI Responses replay without a general tool-id normalizer, and #74850 is an open, unmerged PR that targets this exact bug with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Normalize and cap OpenAI Responses tool-call identifiers before replay so subsequent function_call and function_call_output payloads use provider-valid call_id values and fc-shaped item ids.

## Best Possible Solution

Review and land #74850 or an equivalent focused patch that normalizes Responses replay ids before pi-ai conversion, preserves canonical call_|fc_ reasoning pairs, and adds regression coverage for overlong call ids plus non-fc item ids.

## Reproduction Assessment

Yes. The issue provides a no-network onPayload payload-capture repro, and source inspection confirms same-model pi-ai 0.70.6 tool calls skip normalization before the converter splits raw ids into Responses payload fields.

## Solution Assessment

Yes. A narrow Responses-specific normalizer before provider conversion is the best direction; globally enabling the generic OpenAI sanitizer would conflict with documented reasoning replay preservation behavior.
