## Summary

Keep open. Current main still has a plausible same-provider OpenAI Responses replay gap: OpenAI-owned replay policy keeps Responses tool ids untouched, OpenClaw’s OpenAI helper only downgrades one missing-reasoning `|fc_*` case, and the pinned `@mariozechner/pi-ai` converter still splits same-model ids directly into `function_call.id` and `call_id`. The report is concrete, dependency-scoped, and I found no open PR paired with it.

## What This Changes

The issue asks OpenClaw to normalize OpenAI Responses replay tool identifiers so stored `{call_id}|{response_item_id}` values cannot be replayed as overlong or malformed `function_call` and `function_call_output` ids.

## Best Possible Solution

Add a Responses-specific replay id normalization step before the pi-ai payload conversion that deterministically sanitizes and caps `call_id`, gives `function_call.id` a valid `fc_` shape when replaying an item id, and rewrites the matching tool result to the same normalized call id without breaking reasoning-paired replay.

## Repair Work Prompt

Fix this issue by adding OpenAI Responses-specific replay identifier normalization before payload conversion. Current main disables generic tool-call-id sanitization for OpenAI Responses and pi-ai 0.70.6 only normalizes cross-provider/model tool ids, so same-provider stored ids such as `{long call_id}|{long/non-fc item id}` can replay as invalid `function_call.call_id`, `function_call.id`, and `function_call_output.call_id`.

Expected boundary: keep the fix in the OpenAI Responses replay/transcript hygiene path, likely `src/agents/pi-embedded-helpers/openai.ts` plus the `sanitizeSessionHistory` composition in `src/agents/pi-embedded-runner/replay-history.ts`; adjust `extensions/openai/replay-policy.ts` only if a typed policy knob is genuinely needed. Do not apply the existing generic strict sanitizer wholesale, because current OpenAI Responses reasoning replay depends on preserving the two-part `call_id|fc_id` shape when replayable reasoning is present.

Implementation goals: deterministically sanitize/cap the Responses `call_id`, ensure replayed `function_call.id` is a valid `fc_`-shaped item id when present, and rewrite the matching `toolResult.toolCallId`/`toolUseId` to the same normalized call id. Preserve canonical `call_123|fc_123` behavior and missing-reasoning downgrade behavior.

Add regression tests for an assistant tool call id shaped like `call_${long}|notfc_${long}` followed by the matching tool result under `openai-responses`, verifying the sanitized history and/or pi-ai payload no longer contains overlong call ids or non-`fc_` item ids. Update `docs/reference/transcript-hygiene.md` and `CHANGELOG.md` if production behavior changes; changelog entry should credit `@flashosophy`, not Codex.

Do not change dependency versions, package manager metadata, broad provider replay policy for non-OpenAI providers, or unrelated transcript repair behavior.
