## Summary

Keep open. Current main still has OpenAI Responses replay paths that can pass same-model overlong or malformed tool ids through to `function_call.call_id`, `function_call.id`, and `function_call_output.call_id`; PR #74850 is open and references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The bug report asks OpenClaw to normalize OpenAI Responses replay tool identifiers so stored overlong or malformed `call_id|item_id` pairs are not sent back to Responses providers after tool use.

## Best Possible Solution

Land PR #74850 or an equivalent targeted OpenAI Responses replay normalizer that preserves canonical `call_*|fc_*` reasoning pairs, rewrites malformed or overlong ids before payload conversion, and keeps regression coverage for both persisted history and live stream continuation paths.

## Reproduction Assessment

Yes. The report includes a no-credential payload-capture reproduction, and current source inspection shows the same-model replay path still skips normalization before splitting ids into Responses payload fields.

## Solution Assessment

Yes. A Responses-specific normalizer immediately before pi-ai/OpenClaw Responses payload conversion is the narrow maintainable fix; broad generic tool-id sanitization would risk breaking valid reasoning replay pairs.
