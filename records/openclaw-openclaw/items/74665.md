## Summary

Keep open. Current main still has a same-model OpenAI Responses replay path that can carry raw overlong or malformed tool ids into `function_call` and `function_call_output` payload fields, and open PR #74850 already references this issue with closing syntax as the implementation candidate.

## What This Changes

The issue requests normalization of OpenAI Responses replay `call_id` and `fc_*` item ids so follow-up requests after tool use do not send overlong or malformed provider payload identifiers.

## Best Possible Solution

Land PR #74850, or an equivalent focused patch, that normalizes Responses `call_id` and `fc_*` item ids at the Responses replay conversion boundary while preserving valid canonical same-model reasoning replay pairs and adding regression coverage for overlong and malformed ids.

## Reproduction Assessment

Yes. The issue includes a no-credential payload-capture reproduction, and current source inspection shows the same-model path preserves raw ids before direct split into Responses payload fields.

## Solution Assessment

Yes. A Responses-specific boundary normalizer is the narrowest maintainable fix; enabling broad generic OpenAI tool-id sanitization would risk breaking valid native reasoning replay pairs.
