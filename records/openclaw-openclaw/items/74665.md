## Summary

Keep open. Current main still has a credible OpenAI Responses replay id-normalization gap, and the issue is now paired with open closing PR #74850, so repository policy says the issue should remain open until that PR lands or is rejected.

## What This Changes

The issue requests normalization of OpenAI Responses replay tool identifiers so malformed or overlong stored `call_id|item_id` values are not sent back as invalid `function_call.id` or `call_id` fields after tool use.

## Best Possible Solution

Land #74850 or an equivalent narrow fix that normalizes OpenAI Responses replay ids before the pi-ai payload converter splits them, while preserving canonical same-model `call_*|fc_*` reasoning pairs and adding regression coverage for overlong call ids plus non-`fc_` item ids.

## Reproduction Assessment

Yes. The issue provides a credential-free payload-capture reproduction, and current-main source plus the pinned pi-ai 0.70.6 converter show the same-model path can bypass normalization and split raw ids into the outgoing Responses payload.

## Solution Assessment

Yes, the linked fix direction is the narrowest maintainable path: keep OpenAI Responses reasoning replay intact but add Responses-specific id normalization at the OpenClaw replay boundary. The issue should stay open until the linked closing PR merges or maintainers choose a replacement implementation.
