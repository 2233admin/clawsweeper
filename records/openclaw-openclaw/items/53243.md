## Summary

Keep this PR open for maintainer review. The linked bug is real and current main does not implement the requested default increase or byte-aware visible truncation, but the branch is stale against the current shared chat display projection and still has a serialized-byte budgeting defect that can preserve raw text only to have it replaced by the oversized placeholder later.

## What This Changes

This PR raises the old `chat.history` per-text cap to 64,000 characters, adds raw UTF-8 byte-aware truncation before oversized placeholder replacement, and adds gateway regression tests for long ASCII and emoji-heavy assistant messages.

## Best Possible Solution

Implement this behavior in the shared chat display projection path, choose and document the intended default/config semantics, budget visible truncation against serialized history bytes before placeholder replacement, and cover both `chat.history` and related session-history/UI paths with focused regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Budget truncation against serialized history bytes:** `src/gateway/server-methods/chat.ts:425-427`
  - body: The fallback trims candidates with `Buffer.byteLength(candidate, "utf8")`, but the later oversized gate measures `JSON.stringify(message)`. Escape-heavy text can stay under this raw text budget, expand past the 128 KiB single-message cap when serialized, and still be replaced entirely by the placeholder. Budget the prefix against the JSON-encoded message/string bytes so these messages degrade to visible truncation instead of omission.
