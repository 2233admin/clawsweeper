## Summary

Keep open: the user problem is still partially valid on current main, but this branch is not merge-ready because it targets the old gateway-side truncation location and its byte fallback does not match the serialized byte guard. Merged configurable-limit work does not supersede the requested default/visible-truncation behavior.

## What This Changes

This PR raises the old `chat.history` text cap, adds raw UTF-8 byte-aware fallback truncation, and adds gateway regression tests for long ASCII and emoji-heavy assistant text.

## Best Possible Solution

Implement the behavior in the shared chat display projection path, preserve long assistant text when the fully serialized history entry fits the hard cap, truncate UTF-8-heavy and JSON-escape-heavy text against the same serialized byte budget used by the oversized guard, and align the chosen default/config/docs/tests with that contract.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to write a transcript with a 50,000-character ASCII assistant text and call default `chat.history`; current main projects it through the 8,000-character default before byte caps. The PR-specific defect is also reproducible with JSON-escape-heavy text that passes raw UTF-8 budgeting but exceeds serialized message bytes.

## Solution Assessment

No. The PR addresses the right problem, but the best fix should live in `src/gateway/chat-display-projection.ts` and budget truncation against serialized history bytes, with a maintainer decision on the default/config behavior.
