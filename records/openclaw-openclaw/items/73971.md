## Summary

Keep open: current main still has the reported UX mismatch. The Control UI labels config array rows one-based, while config validation failures are summarized from canonical zero-based Zod paths, so a fourth visible model/fallback row can still be reported as index 3.

## What This Changes

The issue asks for Control UI model-list validation errors to use numbering that matches the visible row labels.

## Best Possible Solution

Keep canonical ConfigValidationIssue.path values zero-based for API/tests, but add a display-layer validation formatter for Control UI config-save/apply errors that maps array path segments to the same human row labels users see, ideally while preserving the raw path in error details.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make the fourth entry in a Control UI config array invalid, such as models.providers.<id>.models or agents.defaults.model.fallbacks, then save; the UI row badge is #4 while the validation path is emitted with numeric index 3.

## Solution Assessment

Yes. The best fix is a narrow display-formatting change for config validation summaries rather than changing Zod path semantics or the canonical ConfigValidationIssue.path contract.
