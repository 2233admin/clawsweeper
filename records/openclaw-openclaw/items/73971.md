## Summary

Keep open. Current main still has the reported mismatch: the Control UI labels config array entries as one-based, while the config validation path surfaced in the gateway error summary preserves zero-based array segments. The provided related issue match for `#4` is unrelated, and the mirror cross-reference is not a canonical OpenClaw fix.

## What This Changes

The issue asks OpenClaw to make the model/provider entry numbering shown in the UI match the numbering used in backend validation errors.

## Best Possible Solution

Keep internal config issue paths stable for APIs and tests, but add a human-facing validation path formatter that renders array positions one-based, or otherwise annotate the message so UI `#4` and validation output refer to the same entry without changing config semantics.

## Reproduction Assessment

Yes. A focused reproduction is to submit config with an invalid fourth `models.providers.<provider>.models` entry through the Control UI: the form labels it `#4`, while current validation formatting will surface a path containing `.3`. I did not run tests because this review is read-only, but the source path is direct.

## Solution Assessment

Yes. The narrowest maintainable fix is to improve human-readable config validation summaries while preserving raw `details.issues[].path` for code and diagnostics; changing the UI to zero-based numbering would make the UX worse.
