## Summary

Keep open. Current main still has a real numbering mismatch: the Control UI labels array entries as one-based `#4`, while the validation path carried into gateway error summaries remains the zero-based array index, so an invalid fourth model entry can be reported as `models...3...`. This is narrow enough to fix without changing the underlying config schema or raw machine-readable issue path contract.

## What This Changes

The issue requests consistent numbering between Control UI model-provider list entries and the gateway validation error shown for the same entry.

## Best Possible Solution

Add a shared user-facing config-issue path formatter that displays array positions consistently with the Control UI row labels, use it in validation summaries and Control UI issue/error rendering, and keep raw validation paths unchanged for API consumers.

## Reproduction Assessment

Yes. A fourth invalid entry under `models.providers.<provider>.models` will render as `#4` in the Control UI array, while the validation path is built from the zero-based segment `3` and is surfaced in the gateway config error summary.

## Solution Assessment

Yes. The narrow maintainable fix is display-layer normalization for user-facing errors, not changing Zod paths or config issue metadata that other callers may depend on.
