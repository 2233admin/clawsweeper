## Summary

Keep open. Current main still has the reported UX mismatch: the Control UI presents config array entries as one-based rows, while validation errors are formatted from raw zero-based schema paths and surfaced back to the user. The existing ClawSweeper comment matches the current code, and the provided related #4 hit is only the reporter's row label, not a duplicate issue.

## What This Changes

The issue asks for model configuration validation errors to use numbering that matches the Control UI row labels so users do not confuse a zero-based backend index with a one-based UI row number.

## Best Possible Solution

Keep this open until a narrow patch adds human-facing config error formatting that aligns array entries with Control UI labels, while preserving raw zero-based paths in structured validation details for compatibility.

## Reproduction Assessment

Yes. A malformed value in the fourth item of a config array should be displayed as row #4 in the form, while current source inspection shows the returned validation path will still contain the raw `.3` segment in user-facing error text.

## Solution Assessment

Yes, the narrowest maintainable fix is a display-only formatter for config validation messages. Structured ConfigValidationIssue.path should remain raw, and tests should prove that human text changes without breaking machine-readable details.
