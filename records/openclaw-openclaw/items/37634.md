## Summary

Current main still leaves `workspaceAccess: "none"` in the read-only/write-blocked path even though that mode uses the isolated sandbox workspace rather than the main agent workspace. The issue should stay open as the focused tracker while the related open fix PR #46300 is reviewed, landed, or superseded with a complete sandbox permission review.

## What This Changes

The issue requests that `workspaceAccess: "none"` hide the main agent workspace while keeping the isolated sandbox `/workspace` writable for tools and skill I/O.

## Best Possible Solution

Keep this issue open as the focused tracker. The best path is to review, land, or supersede #46300 with a complete fix that makes `workspaceAccess: "none"` writable only for the isolated sandbox workspace across Docker/browser mounts, local and remote filesystem bridges, memory flush gating, docs, and regression tests, with `@openclaw/secops` review of the permission semantics.
