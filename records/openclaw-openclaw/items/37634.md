## Summary

Keep this issue open. Current main at b85edb3f0cf6c2a8bec06537af9211a044266a6c still gives `workspaceAccess: "none"` an isolated sandbox workspace, but the managed Docker/browser mount, local filesystem bridge, remote filesystem bridge, memory-flush gate, docs, and tests still treat that workspace as read-only or non-writable. The related open PR #46300 is a plausible implementation candidate, but it is not merged.

## Best Possible Solution

Keep this issue open as the focused tracker. The best path is to review, land, or supersede #46300 with a complete fix that makes `workspaceAccess: "none"` writable only for the isolated sandbox workspace across Docker/browser mounts, local and remote filesystem bridges, memory flush gating, config-hash recreation, docs, and regression tests, with `@openclaw/secops` review of the permission semantics.
