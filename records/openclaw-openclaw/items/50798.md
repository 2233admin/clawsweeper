## Summary

Current main has adjacent ACP/thread delivery fixes, but it does not implement this request. `sessions_send` still has no `visible` parameter and still uses internal delivery, while the generic outbound `message` path still resolves an outbound session route and records session metadata for Discord channel/thread targets. The existing bound-thread routing helps Bot A route Bot B's visible post to the ACP session, but it does not stop the original proxy send from creating the `agent:main:discord:channel:<thread_id>` stub that this issue reports.

## Best Possible Solution

Keep this issue open and implement a narrow, tested proxy-only visible delivery path for ACP-bound Discord threads. The least surprising direction is an explicit `message`/gateway send option that posts through the selected Discord account while bypassing outbound session entry creation, guarded so ordinary message mirroring behavior remains unchanged. Docs should explain the ACP coordinator/worker use case and relate it to the already-fixed visible reply behavior.

## Clownfish Work Prompt

Implement a narrow proxy-only visible delivery path for ACP-bound Discord thread coordination. The observable bug is that a coordinator using `message(accountId: "acp")` to post visibly into a worker ACP Discord thread also creates or updates `agent:main:discord:channel:<thread_id>` session metadata, causing route pollution. Prefer an explicit, default-off option such as `proxyOnly` on the message/gateway send path rather than changing all Discord outbound sends. When `proxyOnly` is true, deliver the Discord message through the selected account to the requested thread/channel, but skip outbound mirror route session creation/transcript metadata for the coordinator. Guard the behavior so normal `message` sends continue to create/mirror sessions as before. Add tests proving the proxy message is sent and no main Discord session entry is recorded, plus tests proving default behavior is unchanged. Update ACP/Discord docs and add a changelog fix/feature entry credited to @ruan330 if the PR is user-facing. Do not touch unrelated ACP reply duplication fixes or broaden session binding policy beyond this path.
