## Summary

Keep this PR open. Current main still reads the first existing transcript candidate and extracts usage without checking whether a parseable session header belongs to the requested session id, so the stale-transcript bug described by the PR remains present. The PR is focused, security-low-risk, and the only recorded review concern is a test fixture model-name style nit.

## What This Changes

This PR adds a first-line session-header id check to `readLatestSessionUsageFromTranscript` and a regression test expecting `null` when a stale transcript belongs to a different session.

## Best Possible Solution

Land this focused gateway fix after the small fixture cleanup and normal CI, preserving legacy/custom transcripts that lack a parseable session header while rejecting transcripts whose parseable session header id clearly belongs to another session.
