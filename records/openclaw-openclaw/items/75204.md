## Summary

Keep open. Current main still reproduces the reported split-state behavior: `OPENCLAW_STATE_DIR` is honored by the shared state-dir resolver, but the exec approvals file and socket defaults still resolve through hard-coded `~/.openclaw` paths, and the related fix PRs were closed unmerged. The remaining fix is real, but it touches host exec approval policy and socket-token persistence, so legacy migration/read semantics need maintainer or secops judgment before an automated repair.

## What This Changes

Request that the default exec approvals file and socket location honor `OPENCLAW_STATE_DIR` instead of continuing to use `~/.openclaw`.

## Best Possible Solution

Route only the default exec approvals file and default socket path through `resolveStateDir()`, preserve explicit configured socket paths, define a legacy read/migration path for existing home-based approval files, and update the CLI/docs plus focused regression tests so doctor, gateway, node host, and local policy reporting agree.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: with `OPENCLAW_STATE_DIR` set to a non-default directory, `resolveStateDir()` returns the override while `ensureExecApprovals()` and `saveExecApprovals()` still write via `resolveExecApprovalsPath()` to `~/.openclaw/exec-approvals.json`.

## Solution Assessment

No current solution is in place. The narrow maintainable fix is to make the default exec approval file and socket state-dir-backed while preserving existing home-based policy state; a docs-only or doctor-only change would not stop the split-state write.
