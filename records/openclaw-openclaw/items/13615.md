## Summary

Keep this issue open. Current main still has adjacent reactive provider 429/cooldown/fallback handling, provider usage readouts, and inbound/control-plane rate limiters, but it does not implement the requested configurable outbound API rate limiting, queueing, token/cost budget enforcement, warning/status controls, or `openclaw limits` CLI. The existing maintainer comment also keeps this open as the canonical tracker, with #13686 closed unmerged as prior art and narrower related issues still open.

## What This Changes

The issue requests configurable outbound API rate limiting, throttling, token/cost budgets, usage visibility, and `openclaw limits` CLI controls.

## Best Possible Solution

Keep this open as the canonical core design tracker. The best path is a maintainer-reviewed, opt-in outbound policy layer covering model-provider dispatch first, then web-search/tool and channel/plugin boundaries as explicit follow-ups, with schema/docs/status/CLI support for RPM/TPM/RPD windows, queue/backpressure controls, 429 retry/backoff behavior, warning thresholds, and token/cost budgets. Related issues such as #42475, #60450, #45771, and #14206 should remain narrower inputs unless a maintainer consolidates them.
