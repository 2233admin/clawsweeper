## Summary

Keep open. Current main does not contain the PR's intended doctor coverage for gateway lock inspection/repair: doctor still only wires session-lock health, there is no `inspectGatewayLock`/`noteGatewayLockHealth` implementation, and the docs only describe session lock cleanup. The PR remains a plausible implementation candidate for the linked stale gateway lock issue, but it needs review fixes before merge because the submitted diff has stale-detection and active-lock deletion risks.

## Best Possible Solution

Keep this PR open and revise it: add a doctor gateway-lock contribution and tests, but base inspection/removal on the same owner-status logic used by `acquireGatewayLock` so live owners are preserved, PID reuse is detected, and stale-without-repair plus stale-with-repair paths are covered. After that, run the targeted doctor/gateway-lock tests and then the appropriate changed gate.

- Closing the PR now would lose a focused implementation candidate for the still-open stale gateway lock doctor coverage request.
