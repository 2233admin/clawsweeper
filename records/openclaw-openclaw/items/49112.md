## Summary

Keep this PR open. Current main still does not provide `openclaw doctor` coverage for gateway lock inspection or repair, and the linked bug #49036 remains a real product gap. The submitted PR is not obsolete, but it needs revision before merge because its inspection/removal semantics are weaker than the current gateway lock owner-status logic and the branch is stale against the current doctor contribution flow.

## Best Possible Solution

Keep this PR open and revise it. The best path is to add a gateway-lock doctor contribution on the current `src/flows/doctor-health-contributions.ts` surface, implement inspection/removal using the same owner-status semantics as `acquireGatewayLock`, add stale-without-repair and stale-with-repair tests, and update the doctor/gateway docs once behavior is safe.

## Risks / Open Questions

- The submitted PR should not merge as-is because `doctor --fix` could remove an active gateway lock after the age threshold when the owner is still alive.
- A PID-only liveness check would miss Linux PID reuse cases that current gateway startup already handles with start-time and argv validation.
- The PR modifies the older direct `src/commands/doctor.ts` flow, while current main routes doctor health through `src/flows/doctor-health-contributions.ts`, so it needs a rebase and wiring update.
