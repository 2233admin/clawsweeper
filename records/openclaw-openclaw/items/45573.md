## Summary

Keep open. Current main still leaves ordinary non-forum Telegram group messages without an `updateLastRoute`, so the core inbound metadata write remains background-only on the path most relevant to the reporter's restart-correlated loss; the linked fix PR is closed unmerged and no current-main change resolves the regular-group persistence gap.

## What This Changes

The issue requests durable Telegram group chat session persistence so group conversations survive gateway restarts and appear correctly in session listing/history tools.

## Best Possible Solution

Implement a narrow Telegram/core session change that makes ordinary non-forum Telegram group inbound metadata durable before dispatch continues while preserving one session per regular group, separate forum-topic sessions, and the intentional no-last-route behavior for bound ACP topic routes.

## Reproduction Assessment

Unclear for the full live Telegram restart data-loss sequence. The current-main code path that makes the report plausible is reproducible with a regular non-forum Telegram group message: no resolved topic means no `updateLastRoute`, and `recordInboundSession` returns without awaiting the metadata write.

## Solution Assessment

Unclear whether the old PR's `updateLastRoute`-only approach is the complete fix. The safer path is a regression-tested change that either supplies a safe regular-group route update or awaits the no-route metadata path without changing documented group/topic session keys.
