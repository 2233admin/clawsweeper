## Summary

Keep open. Current main does not show an OpenClaw path that strips Telegram or Discord token lines from the trusted state-dir `.env`, but it also has not implemented the requested migration hardening: the migration guide still lacks a `.env` token-integrity step, and doctor does not have a scoped migration preflight for expected channel env credentials.

## What This Changes

The issue requests investigation of reported rsync migration loss of Telegram and Discord bot token lines from the state-dir `.env`, plus doctor and migration-documentation hardening.

## Best Possible Solution

Add a migration-guide checklist that tells operators to verify copied state-dir `.env` keys without printing token values and to run `openclaw status --all` or channel-specific probes after cutover. If doctor is extended, scope it to explicit configured credential sources or enabled channel state rather than warning on every absent optional default env var.

## Reproduction Assessment

No high-confidence reproduction path is available from current main alone. The issue gives concrete incident steps, but source inspection shows only `.env` reads and credential inspection, not an OpenClaw mutation or filter that would strip those two lines during rsync.

## Solution Assessment

Unclear for the full requested solution. Documentation plus status/probe guidance is the lowest-risk path now; a blanket doctor check for missing default Telegram/Discord env vars is not the best fix unless it is scoped to cases where OpenClaw can know the token is expected.
