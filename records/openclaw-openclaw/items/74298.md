## Summary

Keep open. Current main does not show an OpenClaw path that strips Telegram or Discord env keys, and the related Telegram runtime false-positive problem is now covered, but the requested migration hardening is still missing: the migration guide lacks a state-dir `.env` integrity checklist, and doctor does not have a targeted expected-token preflight beyond existing configured SecretRef/channel diagnostics.

## What This Changes

The issue asks OpenClaw to investigate loss of env-backed Telegram and Discord bot tokens during state-dir rsync migration, add doctor preflight checks for expected channel env keys, and document `.env` migration verification.

## Best Possible Solution

Leave this open until a maintainer chooses the migration-hardening boundary: document state-dir `.env` verification in the migration guide and only add doctor warnings where config or SecretRefs make a token expectation explicit, avoiding generic warnings for optional absent channel env vars.

## Reproduction Assessment

No high-confidence reproduction path is established. The report gives production rsync steps, but this review did not live-reproduce the transfer anomaly; source inspection only verifies that current OpenClaw would load and inspect those keys when present.

## Solution Assessment

Unclear. A docs checklist is a narrow maintainable fix, but a generic doctor check for missing channel env vars is not the best default unless it is scoped to explicit configured expectations to avoid false positives.
