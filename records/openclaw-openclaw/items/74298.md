## Summary

Keep open. Current main does not show an OpenClaw path that strips Telegram or Discord token keys from a trusted state-dir `.env`, but the requested migration hardening is still not implemented: the migration guide does not include a token-integrity check, and doctor/status coverage is conditional rather than a dedicated migration preflight.

## What This Changes

The issue asks OpenClaw to investigate missing Telegram/Discord bot token env keys after an rsync migration, add doctor preflight coverage for expected env credentials, and document `.env` migration verification.

## Best Possible Solution

Keep this issue open for a scoped migration-hardening decision: document state-dir `.env` verification and post-cutover channel probes, and add doctor/status diagnostics only where OpenClaw can infer that a credential is expected from configured channel state or an explicit SecretRef.

## Reproduction Assessment

No, there is not a high-confidence reproduction for the reported rsync line stripping. The high-confidence check is source/docs inspection: current main preserves these keys when present, while migration docs and preflight coverage remain incomplete.

## Solution Assessment

Unclear for the broad doctor request. A targeted docs update plus narrowly scoped diagnostics for configured credentials would be maintainable; a blanket optional-env-var warning would be noisy and could mislead users.
