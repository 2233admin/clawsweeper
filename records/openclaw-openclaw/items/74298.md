## Summary

Keep open. Current main does not show an OpenClaw path that strips `TELEGRAM_BOT_TOKEN` or `DISCORD_BOT_TOKEN` from the state-dir `.env`, and the channel inspectors accept those env keys when present. The remaining request is still real migration hardening: the migration guide does not yet call out `.env` token/key verification after copy, and a blanket doctor warning for optional channel env vars needs maintainer judgment to avoid noisy false positives.

## What This Changes

The issue asks OpenClaw to investigate a migration incident where Telegram and Discord bot token entries disappeared from a copied state-dir `.env`, add `openclaw doctor` preflight coverage for expected channel env keys, and document `.env` migration verification.

## Best Possible Solution

Keep this open for a targeted migration-hardening pass: document state-dir `.env` key-presence verification in the migration guide, recommend `openclaw status --all` plus `openclaw channels status --probe` after cutover, and add doctor/status diagnostics only where config or SecretRef metadata proves a channel credential is expected.

## Reproduction Assessment

No, not for the reported rsync behavior. The current source review gives high confidence that OpenClaw accepts those keys when present and does not strip them, but a live rsync/production-target reproduction would require external filesystem mutation outside this read-only review.

## Solution Assessment

Unclear as requested. A blanket doctor check for missing Telegram/Discord env vars is not the best default, but a config-informed diagnostic plus explicit migration-doc verification would be a maintainable way to address the observed failure mode.
