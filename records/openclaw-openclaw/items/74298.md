## Summary

Keep open. Current main does not show an OpenClaw path that rewrites or strips `TELEGRAM_BOT_TOKEN` or `DISCORD_BOT_TOKEN` from the trusted state-directory `.env`, and the channel inspectors/status surfaces accept those env keys when present. The requested migration hardening is still not fully implemented: the migration guide lacks a state-dir `.env` token-integrity checklist, and doctor has only targeted channel/SecretRef checks rather than a migration preflight for expected channel env keys.

## What This Changes

The issue asks OpenClaw to harden VPS migrations against missing Telegram and Discord bot token env vars by investigating the rsync report, adding doctor preflight coverage, and documenting `.env` verification.

## Best Possible Solution

Add migration-focused guidance that tells users to verify expected state-dir `.env` keys, run `openclaw status --all` or channel probes after cutover, and document what missing-token output looks like. Add doctor logic only where OpenClaw has an explicit expectation, such as configured env SecretRefs or enabled channel accounts, without warning on every absent optional channel env var.

## Reproduction Assessment

No high-confidence reproduction is available for the reported rsync line stripping. Current-main source inspection shows OpenClaw accepts and preserves these keys when present, but the production incident lacks a deterministic repo-level reproducer or retained before/after files.

## Solution Assessment

Unclear for a generic doctor check: warning on missing channel env names alone is not the safest fix because it would create false positives. A narrower docs/status checklist plus targeted doctor warnings for explicit configured token expectations is the maintainable direction.
