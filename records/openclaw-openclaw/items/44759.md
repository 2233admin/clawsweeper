## Summary

Current main still queues trusted browser screenshot media for channel delivery. The implementation has moved from the older text-only `MEDIA:` path to structured `details.media.mediaUrl`, but `browser` remains trusted for local tool-result media, screenshot results still carry the local path, and pending tool media is merged into the next visible reply or flushed as media-only. The Pinchy comment is a useful corroborating production use case, and the open broad sanitization items do not yet implement the narrow per-tool screenshot visibility control on main.

## Best Possible Solution

Keep this issue open and implement a core tool-result media visibility policy. The narrowest useful fix is a `browser screenshot` visibility option such as `private: true` that keeps the image block available to the model but prevents `details.media.mediaUrl` from entering pending channel delivery. A broader config-level per-tool default could sit in the same pending-media pipeline, with tests proving suppressed screenshots remain model-visible and generated media still auto-delivers when intended.
