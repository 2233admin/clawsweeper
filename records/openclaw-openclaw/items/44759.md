## Summary

Keep this issue open. Current main still lets `browser screenshot` expose a screenshot file path as trusted tool-result media and queue it for outbound reply delivery, even though the implementation has moved from the older `MEDIA:` text path toward structured `details.media.mediaUrl`.

## Best Possible Solution

Implement a core tool-result media visibility policy, with a narrow browser screenshot opt-out such as `private: true` and/or a config-level per-tool default. The screenshot result should still include the image block for model vision, but `extractToolResultMediaArtifact`/`handleToolExecutionEnd` should skip queuing media marked private. Add regression coverage around the browser tool, image result helper, trusted media filtering, pending media attach, and lifecycle flush paths.

## Risks / Open Questions

- Sensitive browser pixels can still be attached to visible channel replies when browser screenshot results enter the pending tool-media pipeline.
- The older `MEDIA:`-tag detail in the issue body is no longer the only path; current main also uses structured `details.media.mediaUrl`, so a fix must cover both structured and legacy media extraction.
- A fix should preserve intentional automatic delivery for generated media tools such as `image_generate`, `video_generate`, music, and TTS while adding a suppressible/private policy for inspection tools like browser screenshots.
