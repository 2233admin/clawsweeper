Keep open. Current main has useful adjacent guardrails, but I do not see a source- and test-backed fix for the combined Telegram topic, approved exec followup, and context-overflow retry path; the followup still resumes through the normal Gateway agent path and overflow recovery still retries after compaction.

## What This Changes

The issue reports that Telegram async exec approval followups can leak HEARTBEAT_OK or internal orchestration text into user-visible topic chat after context-overflow compaction retries.

## Best Possible Solution

Add a first-class internal/no-op delivery state for async exec and heartbeat-style followups that survives session resume and context-overflow compaction retries, then cover Telegram topic delivery with an approved exec followup that compacts and must not render HEARTBEAT_OK or internal orchestration prompt text while still relaying real exec output.
