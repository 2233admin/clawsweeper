## Summary

Keep open: current main still has the early-abort response-prefix context gap, and multiple open PRs already target the issue with closing syntax. This should stay open until maintainers choose, land, or supersede a canonical fix.

## What This Changes

The issue requests that early stop or abort replies resolve dynamic `responsePrefix` template variables such as `{model}` and `{thinkingLevel}` instead of sending raw placeholders.

## Best Possible Solution

Land or supersede the linked fix PRs with one narrow implementation that supplies safe runtime/session model and thinking context to fast abort and routed command replies, while preserving normal `onModelSelected` override behavior and adding regression tests for both paths.

## Reproduction Assessment

Yes. Configure a `messages.responsePrefix` containing `{model}` or `{thinkingLevel}`, trigger `stop` or `/stop` while a run is active, and the fast-abort branch sends before model-selection context is populated while the resolver preserves missing variables literally.

## Solution Assessment

Unclear which proposed implementation is the best without maintainer review. The safest direction is to derive prefix context from actual session/runtime metadata where available, not merely hide dynamic prefixes or leak placeholders.
