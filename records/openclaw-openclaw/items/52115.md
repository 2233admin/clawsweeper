## Summary

Current main still appears to have the reported long-running gateway freshness gap: agent-facing `memory_search` uses the cached default memory manager, dirty searches start sync in the background, and existing rows can be queried before created/deleted memory files are reflected. This is valid, narrow, and suitable for a focused repair PR.

## What This Changes

The issue requests freshness-aware agent-facing `memory_search` results after memory files are created, renamed, or deleted during a long-running gateway session.

## Best Possible Solution

Make builtin `MemoryIndexManager.search()` freshness-aware for dirty memory/session state by awaiting the relevant pending or newly-started sync before querying rows, while keeping clean searches fast and covering create/update/delete first-search behavior with regression tests.

## Repair Work Prompt

Fix the current `memory_search` freshness gap for long-running gateway sessions. Agent-facing `memory_search` uses the default cached builtin `MemoryIndexManager`; when memory files or enabled session state are dirty, the first search currently starts sync asynchronously and can query stale rows, including deleted paths. Implement a narrow fix in memory-core so dirty or already-pending search sync is awaited before querying rows, while preserving the clean-index fast path and avoiding broad QMD or plugin API changes. Add regression coverage for first search after memory file create/update/delete and adjust the async search helper test expectations. Changelog is expected for this user-facing bug fix; credit @FicheallADa. Do not change unrelated memory ranking, provider selection, QMD fallback, or CLI command behavior except where needed to keep CLI tests aligned.
