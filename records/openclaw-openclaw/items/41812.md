## Summary

Keep open. Current main still lacks catalog feedback in `models set`, so the PR is not obsolete, but the current patch should not merge because it replaces the existing conflict-safe config mutation path with a stale direct write after an async catalog load.

## What This Changes

The PR changes `models set` to load the configured model catalog, warn when the selected provider/model is absent from that catalog, and then persist the selected model as the default.

## Best Possible Solution

Implement catalog feedback inside the existing model-command mutation discipline: keep a conflict-checked `replaceConfigFile`/`updateConfig` flow, preserve source-config and include handling, make the warning-versus-rejection policy explicit for configured/custom/stale-catalog models, and add focused tests for invalid catalog feedback plus concurrent-write safety.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve the conflict-safe config write path:** `src/commands/models/set.ts:29`
  - body: The patch loads config, awaits `loadModelCatalog`, then writes an `updated` object derived from that earlier snapshot with `writeConfigFile(updated)`. Because `writeConfigFile` rereads the current config and computes persistence from that newer runtime snapshot to the stale object, config changes made while the catalog is loading can be treated as deletions. Keep this on the `updateConfig`/`replaceConfigFile` path or carry the validated snapshot/base hash through the catalog check.
  - confidence: 0.88
