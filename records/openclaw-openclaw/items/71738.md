## Summary

Keep this PR open. Current main still has the Matrix thread-history and fallback-reply behavior the branch targets, so the work is not obsolete; the branch should not land as-is because its main-room pagination can skip unread events and its thread watermark handling can hide fresh messages after sub-queue eviction.

## What This Changes

This PR scopes Matrix room history and read actions by thread, adjusts Matrix thread send/edit relation metadata, forwards `threadId` through message actions, and adds focused regression tests.

## Best Possible Solution

Land a reworked Matrix-only PR that keeps thread scope through room-history, read, send, and edit paths; aligns pagination cursors with the events actually returned; validates thread sub-queue liveness before watermark writes; updates the Matrix history docs; and keeps focused regression coverage.

## Reproduction Assessment

Yes. The current-main behavior is reproducible by source inspection: thread relation fallback metadata is unconditional, threaded edits write `m.in_reply_to` on the outer replace relation, room-history APIs have no thread scope, and Matrix reads have no thread endpoint path.

## Solution Assessment

No, not yet. The requested direction is maintainable and Matrix-spec-aligned, but this implementation needs the pagination and thread watermark fixes before it is the safest way to solve the issue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve unread events when over-fetching main timeline:** `extensions/matrix/src/matrix/actions/messages.ts:188`
  - body: When main-room reads filter thread events, this can fetch extra pages, collect more non-thread events than `limit`, then drop the surplus with `slice(0, limit)` while returning `res.end` from the last fetched page. Those dropped events are now behind `nextBatch`, so callers paginating with that token skip readable main-room history. Keep the cursor aligned to the returned events or avoid fetching past the requested window.
  - confidence: 0.88
- **[P2] Reject stale consumes for evicted thread queues:** `extensions/matrix/src/matrix/monitor/room-history.ts:410`
  - body: `consumeHistory` only validates the room generation before writing a thread watermark. If a thread sub-queue is evicted and later recreated while an older trigger is still in flight, the late consume can advance the new thread watermark past fresh messages and hide them from the agent. Track thread queue liveness/generation or skip the write when the scoped sub-queue no longer matches the snapshot.
  - confidence: 0.86
- **[P3] Clear legacy room watermarks during room eviction:** `extensions/matrix/src/matrix/monitor/room-history.ts:133`
  - body: `clearRoomWatermarks` now only recognizes JSON watermark keys, but `computePendingHistory` still reads legacy `agentId:roomId` keys. After upgrading, evicted rooms can leave legacy entries in the capped watermark map, reducing available capacity for active rooms. Also delete the legacy room key when clearing a room.
  - confidence: 0.8
