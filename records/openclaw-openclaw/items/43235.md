## Summary

Keep this issue open. Current main still defaults ordinary inbound agent turns to the shared `main` command lane, and the requested per-agent isolation is not implemented yet. The issue is also explicitly paired with open PR #73991 using closing syntax, so the remaining work should be handled by reviewing, landing, or closing that PR rather than closing this issue now.

## What This Changes

Request support for per-agent or per-agent-group command lanes so long-running work for one agent cannot starve other agents on the shared main lane.

## Best Possible Solution

Keep the issue open until PR #73991 or a replacement lands a narrow, opt-in per-agent lane resolver and config contract that preserves the existing shared-lane default, updates schema/docs/tests, and covers the Feishu multi-bot bottleneck reported here.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to run multiple inbound agent messages without an explicit lane, such as the reported Feishu @all setup; current source shows those runs resolve to `CommandLane.Main`, and the issue comment includes matching `lane=main` wait diagnostics.

## Solution Assessment

Yes in direction, but not yet proven for the open PR. The maintainable solution belongs in the core command queue/agent config lane resolution path because the bottleneck is the shared gateway `main` lane, while channel-specific workarounds would not isolate other inbound surfaces.
