## Summary

Keep this PR open for maintainer review. Current main still has the heredoc/enforced-command mismatch the PR is trying to address, but the proposed branch changes a security-sensitive host exec approval path too broadly and has a concrete P1 regression for timeout fallback approvals.

## What This Changes

The PR changes gateway exec approval follow-up so heredoc-required approvals execute raw `params.command` instead of the rebuilt allowlist `enforcedCommand`, while leaving non-heredoc approvals on the enforced command path.

## Best Possible Solution

Land a narrow gateway approval fix that preserves raw heredoc command text only after explicit `allow-once` or `allow-always`, keeps the rebuilt enforced command for allowlist timeout fallback approvals, and adds focused regression tests for both heredoc branches plus the unchanged non-heredoc path.

## Reproduction Assessment

Yes. A focused unit test can drive `processGatewayAllowlist` with `requiresHeredocApproval`, a timed-out approval decision, and `askFallback: "allowlist"`; the PR head then selects raw `params.command` where current secure behavior should keep `enforcedCommand`.

## Solution Assessment

No. The PR fixes the likely heredoc mangling at the wrong granularity by branching on `requiresHeredocApproval` alone; the safer implementation should branch on an explicit allow decision and leave fallback approvals on the enforced wrapper.
