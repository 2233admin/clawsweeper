## Summary

Keep this issue open. Current main still does not implement owner takeover or human handoff for WhatsApp DM auto-reply: the requested config/state terms are absent, outbound non-self `fromMe` DMs are dropped before auto-reply state could be updated, and the related implementation PR #58919 was closed unmerged.

## Best Possible Solution

Keep this open and implement a focused, default-off owner handoff path, preferably WhatsApp-first unless maintainers explicitly want a shared cross-channel abstraction. The fix should detect manual outbound `fromMe` DM user messages before the current access-control drop, persist per-account/per-thread pause/cooldown state, skip auto-reply dispatch while paused, support explicit/automatic resume according to documented config, and add targeted WhatsApp tests/docs. PR #58919 is useful prior art but not shipped evidence.

## Clownfish Work Prompt

Implement a focused WhatsApp DM owner takeover / human handoff for auto-reply on current main. Observable problem: when the human owner sends a manual outbound WhatsApp DM on the linked account, OpenClaw keeps auto-replying to that same thread. Current code passes `isFromMe` into `checkInboundAccessControl`, then drops non-self outbound DMs before hooks/session metadata/auto-reply state can see them. Treat PR #58919 as prior art only; it closed unmerged and should not be copied blindly.

Scope: add a default-off typed WhatsApp config surface for owner handoff/takeover with cooldown and explicit/auto resume semantics, using naming consistent with existing channel config/schema/docs. Detect outbound `fromMe` DM user-content messages before the current access-control return-null path, update per-account/per-thread pause/cooldown state, and gate `dispatchWhatsAppBufferedReply` while the thread is paused. Support explicit resume only if there is a clear owner-gated command path; otherwise document the supported auto-resume behavior and leave explicit resume as a follow-up. Do not change unrelated channel access policies, do not move WhatsApp-owned behavior into core unless a generic helper is demonstrably needed, and do not make core depend on WhatsApp internals.

Update docs and config schema/help as needed. Add a user-facing changelog entry under the active version with `Thanks @Maliq-dlt`; include `Thanks @Hamza1s34` too if the implementation materially uses PR #58919’s approach. Keep generated config metadata aligned if the config surface changes.
