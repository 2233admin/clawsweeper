## Summary

Keep open. Current main still has the Windows ACL command-resolution and workspace env-policy gaps this PR targets, while the PR head is open, mergeable, CI-green, and security-sensitive enough to need maintainer/security review rather than automated closure or replacement.

## What This Changes

The PR adds SYSTEMROOT/WINDIR to the host env and workspace dotenv policy artifacts, exports the shared Windows system-root default, resolves Windows ACL helper/reset commands through validated System32 paths, and adds regression tests.

## Best Possible Solution

Land one maintainer-approved Windows hardening that blocks untrusted workspace Windows root overrides through the shared env policy, validates SystemRoot/WINDIR before resolving System32 helper binaries, and coordinates the ACL, reg.exe, taskkill.exe, and shell-resolution surfaces into one coherent result.

## Reproduction Assessment

Yes. On current main, a unit seam can pass env: { SystemRoot: ".\\fake-root" } to inspectWindowsAcl and observe .\fake-root\System32\icacls.exe/whoami.exe resolution, while workspace dotenv currently lacks SYSTEMROOT/WINDIR blocking; the PR adds tests for those exact paths.

## Solution Assessment

Yes for the ACL surface: reusing getWindowsInstallRoots keeps the fix narrow and aligned with the existing validator. The safer landing path is maintainer/security review plus consolidation with the related Windows env hardening PRs rather than an automated replacement branch.
