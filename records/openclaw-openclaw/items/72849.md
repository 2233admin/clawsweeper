## Summary

Keep open. Current main still has the missing-id cron identity gap in the runtime load and outcome paths, and the open paired PR #72853 is the implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Cron jobs loaded from persisted configuration without an `id` should receive stable identities so runtime state cannot be misapplied to the first id-less job.

## Best Possible Solution

Land #72853 or an equivalent maintainer-approved cron fix that assigns and persists stable ids for missing or blank persisted job identities during store load/hydration, rejects missing job ids in timer/state-application paths, preserves doctor repair behavior, and adds regression coverage for multiple id-less persisted jobs plus sidecar state merge.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load two persisted cron jobs with neither `id` nor `jobId`, then let an outcome carry `jobId: undefined`; current `applyOutcomeToStoredJob()` can match the first job whose `id` is also undefined.

## Solution Assessment

Yes. Load-time identity canonicalization plus defensive timer/state guards is the narrow maintainable solution; doctor-only repair is useful but insufficient because the Gateway can still load and execute malformed rows before doctor is run.
