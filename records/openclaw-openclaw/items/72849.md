## Summary

Keep open. Current main still leaves the missing cron-job `id` path live, including both state merge and outcome lookup surfaces, and the issue is already paired with open PR #72853 using closing syntax; maintainers should review, land, or close that PR before resolving this issue.

## What This Changes

The issue requests stable identity backfill for persisted cron jobs missing `id` and defensive guards so runtime state cannot match or write through an undefined job id.

## Best Possible Solution

Land one canonical cron fix that assigns stable ids for missing or blank persisted job identities during load/hydration, persists the canonical shape, refuses undefined job ids in timer/state application paths, and covers the regression with focused store/timer tests plus a short docs/changelog note.

## Reproduction Assessment

Yes. A high-confidence source reproduction is to load multiple persisted jobs with no `id`, let one run produce an outcome with an undefined job id, and observe that `applyOutcomeToStoredJob()` can match the first undefined-id entry; current code inspection shows that path remains live.

## Solution Assessment

Yes. The narrowest maintainable solution is load-time identity canonicalization plus defensive timer/state guards, which matches the existing open implementation candidate rather than requiring a broader cron redesign.
