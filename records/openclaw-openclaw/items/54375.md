## Summary

Keep open. Current `main` intentionally keeps DBus/user-bus transport failures on the generic service-loaded error path, and this PR would reintroduce a false-negative state that lifecycle callers can treat as a successful not-loaded service.

## What This Changes

The PR changes `isSystemdServiceEnabled` so DBus/user-bus unavailable `systemctl is-enabled` failures return `false` instead of throwing.

## Best Possible Solution

Keep DBus/user-bus transport failures on the generic service-loaded error path. If broader non-fatal behavior is desired, introduce a typed unknown transport state and update install, status, reset, uninstall, and daemon lifecycle tests together.

## Reproduction Assessment

Yes. Current unit tests model direct and exhausted machine-scope DBus/user-bus failures and assert rejection, and caller inspection shows a `false` result is consumed as a successful not-loaded state in `openclaw uninstall --service`.

## Solution Assessment

No. The one-line guard is too broad because it collapses an unknown transport failure into disabled/not loaded; the existing install-scoped helper is the narrower maintainable behavior unless the service-loaded API is redesigned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve DBus probe failures as errors:** `src/daemon/systemd.ts:635`
  - body: Returning `false` here turns an exhausted DBus/user-bus transport failure into "service not loaded." `openclaw uninstall --service` treats `false` as a successful not-loaded state and returns before stop/uninstall, so an installed service can remain after a temporary bus outage. Keep this path as an error or introduce a distinct unknown state and update lifecycle callers/tests.
