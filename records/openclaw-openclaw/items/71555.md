## Summary

Keep this open. Current main still reaches the pinned Pi `SessionManager.open()` path for embedded runs and compaction, and that upstream implementation still reads/parses the same JSONL file once to extract the header and again from `setSessionFile()` before context rebuilding. The newer byte-threshold compaction and mid-turn precheck work are useful mitigations, but they do not remove the initial large-file open cost reported here.

## What This Changes

Avoid gateway crashes by preventing large embedded Pi session JSONL transcripts from being whole-file read and parsed twice during context rebuild.

## Best Possible Solution

Fix the Pi session-open boundary so OpenClaw either consumes an upstream Pi release that reads the header once and reuses parsed entries, or adds a narrow OpenClaw helper using public Pi APIs to avoid the duplicate full-file parse. Add focused coverage with a large JSONL/oversized tool-result fixture showing embedded runs and preflight compaction recover or fail cleanly without reopening the transcript twice.

## Reproduction Assessment

Unclear. The duplicate whole-file parse path is reproducible by source inspection of `@mariozechner/pi-coding-agent@0.70.6` plus current OpenClaw call sites, but this read-only review did not run the reporter's large transcript or a synthetic gateway crash scenario.

## Solution Assessment

Not applicable as a proposed patch. The issue identifies the right boundary; the maintainable fix is to remove the duplicate parse at session open rather than add more later-stage truncation or precheck logic.
