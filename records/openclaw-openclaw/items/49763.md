## Summary

Keep this issue open. Current main still lets Ctrl+O replace an active busy TUI status with non-busy tool-toggle text, which stops and removes the Loader spinner; no open PR was found that specifically fixes or closes this issue.

## Best Possible Solution

Keep the issue open and fix the TUI status model in src/tui/tui.ts so transient tool-output toggle notices do not replace an active busy state. A conservative implementation should preserve the current Loader whenever activeChatRunId or pendingOptimisticUserMessage is present, while still toggling chatLog.setToolsExpanded, and add a focused regression test for Ctrl+O during an active run.
