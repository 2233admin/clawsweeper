## Summary

Keep open. Current main still lacks the requested public doctor-check extension surface: doctor health contributions are core-only and statically resolved, the plugin API has no doctor-check registration method, and the existing plugin doctor contract only covers compatibility/config repair. The issue is recent and source-anchored by follow-up discussion, but the repair portion needs maintainer API and trust-policy direction before implementation.

## What This Changes

Request a persistent doctor extensibility API so operators or plugins can add custom environment checks and potential fixes that survive npm upgrades.

## Best Possible Solution

Keep this as a core/plugin-SDK design issue. Add an additive doctor-check contribution contract with plugin attribution, deterministic ordering, duplicate-id handling, focused tests, and docs; start with read-only plugin checks or explicitly gate repair support behind a clear interactive and unattended-fix trust policy.

## Reproduction Assessment

Not applicable as a runtime bug reproduction because this is a feature/API request. The verification path is high-confidence source inspection: current main has no public way for a plugin or persistent local extension to register arbitrary doctor health checks or fixes.

## Solution Assessment

Unclear until maintainers decide the API and safety policy. A narrow plugin-SDK doctor contribution seam appears to be the maintainable direction, but plugin-owned fixes, CLI flags, duplicate IDs, and host-wide config access need explicit design first.
