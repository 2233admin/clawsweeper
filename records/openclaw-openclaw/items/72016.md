## Summary

Keep open. Current main still has a static core-only doctor health contribution list and only narrow plugin doctor compatibility contracts; there is no public plugin SDK API for arbitrary doctor checks or plugin-owned fix actions. The issue is recent, actionable, and the discussion correctly frames this as a core/plugin-SDK extension-point decision rather than something an external plugin can solve with today’s APIs.

## What This Changes

The issue requests upgrade-safe doctor command extensibility so custom environment-specific setup checks and optional fixes can be registered without patching OpenClaw source.

## Best Possible Solution

Keep this as a core/plugin-SDK design item. Add a narrow documented doctor-check contribution contract with plugin source attribution, deterministic ordering, duplicate-id handling, tests, and docs; start with read-only plugin checks unless maintainers explicitly approve the fix contract and unattended repair semantics.

## Risks / Open Questions

- A plugin-contributed doctor fix API would run near local config/state repair paths, so maintainers need to decide source attribution, duplicate-id behavior, prompting, and unattended --repair/--yes policy before implementation.
- The motivating multi-gateway/channel-overlap check may require host-wide profile/config discovery beyond the active gateway config, which should be explicitly scoped.
- Existing plugin CLI commands, security audit collectors, and compatibility migrations are adjacent surfaces; the design should avoid duplicating them while still integrating with openclaw doctor.
