## Summary

Keep open. Current main still has a core-only doctor health contribution list and narrow plugin doctor compatibility contracts, but no public plugin SDK API or CLI controls for custom doctor checks. The issue discussion is recent and source-anchored, and the requested `--fix` extensibility raises public API and local repair-authority questions that need maintainer design review before implementation.

## What This Changes

Expose an extensible doctor-check API so operators or plugins can add persistent environment-specific diagnostics and optional fixes that survive npm upgrades.

## Best Possible Solution

Keep this as a core/plugin-SDK design item. Add a narrow, additive doctor-check extension contract with plugin source attribution, deterministic ordering, duplicate-id policy, tests, and docs; start with read-only checks, then add fix support only after prompting and unattended-mode semantics are explicitly settled.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence verification path is current-main source inspection, which shows no public way for a plugin or persistent local extension to register arbitrary doctor health checks.

## Solution Assessment

Unclear until maintainer direction lands. A plugin-SDK doctor contribution seam is the maintainable direction, but `fix` authority, CLI controls, SDK location, and host-wide config access should be designed before a PR starts.
