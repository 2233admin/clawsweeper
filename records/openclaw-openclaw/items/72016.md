## Summary

Keep open. Current main still has a core-only doctor health contribution list plus narrow plugin compatibility migrations, but no public plugin SDK registration API or doctor CLI controls for custom checks. The discussion is recent, source-anchored, and points to a real core/plugin-SDK design gap rather than stale support cleanup.

## What This Changes

Expose an extensible doctor-check API so operators or plugins can add persistent environment-specific diagnostics and optional fixes that survive npm upgrades.

## Best Possible Solution

Keep this as a core/plugin-SDK design item. Add a narrow, additive doctor-check extension contract with plugin source attribution, deterministic ordering, duplicate-id policy, tests, and docs; start with read-only checks, then add repair support only after prompting and unattended-mode semantics are settled.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence verification path is source inspection on current main, which shows no public way for a plugin or persistent local extension to register arbitrary doctor health checks.

## Solution Assessment

Unclear until maintainer direction lands. A narrow plugin-SDK doctor contribution seam is the maintainable direction, but fix authority, CLI controls, SDK location, and host-wide config access should be designed before implementation.
