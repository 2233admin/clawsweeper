## Summary

Keep open. Current main still has a core-only doctor health contribution loop plus narrower plugin compatibility/repair hooks, but no public plugin-SDK registration API for persistent custom doctor checks or plugin-owned doctor fixes. The issue discussion is recent, source-anchored, and raises API/security choices that need maintainer direction before implementation.

## What This Changes

Request a persistent doctor extensibility API so plugins or local extensions can add custom environment checks and optional fixes that survive npm upgrades.

## Best Possible Solution

Treat this as a core/plugin-SDK design item. Add a small additive doctor-check contribution contract with plugin source attribution, deterministic ordering, duplicate-id policy, contract tests, and docs; start with read-only checks, then add repair support only after prompt, logging, and unattended-mode semantics are settled.

## Reproduction Assessment

Not applicable as a feature/API request. The high-confidence verification path is source inspection on current main, which shows no public way for a plugin or persistent local extension to register arbitrary checks into the main doctor health flow.

## Solution Assessment

Unclear until maintainer direction lands. A narrow plugin-SDK doctor contribution seam is the likely maintainable direction, but fix authority, CLI controls, SDK subpath placement, and host-wide config access should be designed before implementation.
