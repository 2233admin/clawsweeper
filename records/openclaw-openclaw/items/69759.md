## Summary

Keep open. Current main still counts every session file ending in `.jsonl` for `openclaw memory status`, while the indexing path filters session files through the usage-counted transcript helper that excludes compaction checkpoint snapshots. The reported denominator mismatch remains valid on current main.

## What This Changes

The issue asks for `openclaw memory status` to exclude compaction checkpoint transcript snapshots from the session-source file total so the displayed indexed count matches the actual indexing corpus.

## Best Possible Solution

Align the memory status session scan with the same usage-counted transcript corpus used by indexing, preferably by exposing/reusing `isUsageCountedSessionTranscriptFileName` through the memory host runtime contract. Add focused CLI status coverage for a normal transcript plus a compaction checkpoint snapshot, and add a user-facing changelog fix crediting @xinzf.

## Reproduction Assessment

Yes. A current-main source check gives a high-confidence reproduction path: place a normal session `.jsonl` and a `<session>.checkpoint.<uuid>.jsonl` in the session directory, then `scanSessionFiles` will count both while indexing lists only usage-counted transcripts.

## Solution Assessment

Yes. The narrow maintainable fix is to make the status scan use the same session artifact predicate as indexing; indexing checkpoint snapshots would broaden the corpus and does not match the existing design intent.
