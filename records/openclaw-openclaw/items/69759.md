## Summary

Keep open. Current main still counts every top-level session file ending in `.jsonl` for `openclaw memory status`, while the session indexing corpus is filtered through the usage-counted transcript predicate that excludes compaction checkpoint snapshots. The reported `Indexed: 3/4 files` mismatch remains possible when a checkpoint transcript is present.

## What This Changes

The issue requests that `openclaw memory status` exclude compaction checkpoint transcript snapshots from the session file denominator so the indexed/total file count matches the actual indexed corpus.

## Best Possible Solution

Make a narrow memory-core fix that aligns `scanSessionFiles` with the same usage-counted session transcript corpus used by indexing, preferably by exposing/reusing the existing predicate through the memory host runtime contract. Add focused `memory status` coverage for a normal transcript plus a `.checkpoint.<uuid>.jsonl` snapshot, and add a user-facing changelog fix crediting @xinzf.
