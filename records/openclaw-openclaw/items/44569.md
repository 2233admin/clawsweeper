## Summary

Keep this PR open. Current main still does not forward config `env.vars` into sandbox `exec` environments, so the PR is not implemented or obsolete. The bug is real, but the submitted patch needs maintainer revision because it changes the sandbox env contract and should preserve `applyConfigEnvVars` precedence and placeholder guards.

## Best Possible Solution

Keep the PR open and revise it into a guarded sandbox env propagation patch: derive an effective config env map that mirrors `applyConfigEnvVars` semantics, layer it below sandbox env and per-call params, add regression tests for sandbox forwarding, precedence, blocked keys, existing env precedence, and unresolved placeholders, and update the public sandbox/env docs if maintainers decide config `env.vars` is part of the sandbox exec contract.

## Risks / Open Questions

- Merging the PR as written could make sandbox exec receive config values that gateway exec would intentionally skip when an existing process env value is already set.
- Merging the PR as written could forward literal unresolved `${VAR}` placeholders into sandbox commands, diverging from `applyConfigEnvVars` behavior.
- Forwarding config `env.vars` into sandbox exec expands which configured secrets can reach sandboxed commands; that may be desired, but it needs an explicit contract, tests, and docs because current docs describe sandbox env isolation.
- Adjacent sandbox env work such as #40290 should be coordinated so `env.vars`, `sandbox.docker.env`, sanitization, and admin override semantics do not drift.
