## Summary

Keep this PR open. Current main still does not forward config `env.vars` into sandbox `exec` environments, so the PR is not obsolete or implemented; however, the proposed direct `collectConfigRuntimeEnvVars` forwarding needs maintainer review because it changes sandbox secret exposure and does not preserve `applyConfigEnvVars` runtime guards.

## Best Possible Solution

Keep this PR open and revise it into a guarded sandbox env propagation patch: derive an effective config env map that mirrors `applyConfigEnvVars` semantics, layer it below sandbox/backend env and per-call params, add regression tests for forwarding, precedence, blocked keys, existing env precedence, and unresolved placeholders, and update the public sandbox/env docs if maintainers decide config `env.vars` is part of the sandbox exec contract.

## Risks / Open Questions

- Merging the PR as written could make sandbox exec receive config values that host/gateway exec would intentionally skip when an existing runtime env value already exists.
- Merging the PR as written could forward literal unresolved `${VAR}` placeholders into sandbox commands, diverging from `applyConfigEnvVars` behavior.
- Forwarding config `env.vars` into sandbox exec expands configured secret exposure to sandboxed commands; that may be the desired contract, but current docs describe sandbox env isolation.
- The PR body's sanitizer rationale overstates current exec-time coverage: Docker create sanitizes configured container env, but `docker exec` argument construction forwards non-`PATH` env values it is given.
