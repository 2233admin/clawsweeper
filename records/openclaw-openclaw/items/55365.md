## Summary

Keep this issue open. Current main still validates browser navigation for URL shape, scheme, SSRF policy, and proxy mode, but it does not reject `http(s)` URLs with `username` or `password` userinfo before forwarding the same raw URL into Playwright/Chrome MCP or tab creation. The provided related items do not supersede this: #51822 is the same hardening pattern for cron webhooks, while the browser SSRF and proxy reports cover different browser surfaces.

## Best Possible Solution

Add a narrow browser-navigation guard in `extensions/browser/src/browser/navigation-guard.ts` after URL parsing: reject `http:`/`https:` URLs with `parsed.username` or `parsed.password`, return a redacted URL in the error text, and point users to `openclaw browser set credentials`, cookies, or headers. Cover `assertBrowserNavigationAllowed`, `/navigate`, and `/tabs/open` so credential-bearing URLs are neither forwarded to Playwright/Chrome MCP nor surfaced raw in route errors; keep CDP endpoint userinfo behavior unchanged.

## Risks / Open Questions

- Credential-bearing browser navigation URLs can still reach Playwright/Chrome MCP and may be echoed by downstream runtime errors before OpenClaw can redact them.
- Adding the guard should preserve remote CDP URL userinfo support, which is documented separately for `browser.cdpUrl` and already tested in CDP helpers.
