## Summary

Keep this issue open. Current main still validates browser navigation URLs for shape, protocol, SSRF policy, proxy mode, and hostname resolution, but it does not reject http(s) URL userinfo before passing the original URL into Chrome MCP, Playwright, or CDP tab creation. Related PR #51822 is useful prior art for cron webhook URLs, but it is open, separate, and does not cover the browser tool surface.

## Best Possible Solution

Add a narrow browser-navigation guard in extensions/browser/src/browser/navigation-guard.ts for http: and https: URLs: reject parsed.username or parsed.password early, build the error from a redacted URL, and direct users to `openclaw browser set credentials`, cookies, or explicit headers. Add regression coverage for assertBrowserNavigationAllowed, /navigate, /tabs/open, and Playwright create/navigate paths while preserving existing CDP endpoint userinfo behavior. #51822 can be used as prior art, but it should not be treated as the canonical browser fix.

## Risks / Open Questions

- Credential-bearing page-navigation URLs can still reach Chromium, Playwright, Chrome MCP, or CDP tab creation and may be echoed by downstream error paths before OpenClaw can redact them.
- A fix must distinguish page-navigation URLs from remote CDP endpoint URLs, where userinfo support is intentional and covered by existing CDP tests/docs.
