## Summary

Keep this PR open. Current main already covers the logging-config side in a newer implementation, but the active local embedding provider still lacks the PR's VRAM retry and temporary CPU fallback behavior; the branch also needs a small cleanup fix before it is safe to land or replace.

## What This Changes

The PR updates logging config/file-date handling and adds local memory embedding GPU VRAM retry, one-operation CPU fallback, and targeted tests.

## Best Possible Solution

Land a refreshed implementation that keeps the shipped logging changes, ports the local embedding VRAM retry and one-operation CPU fallback to packages/memory-host-sdk/src/host/embeddings.ts, preserves contextSize behavior, and covers retry serialization plus CPU-init cleanup failures in targeted tests.

## Reproduction Assessment

Yes. The PR body and tests give a mockable VRAM initialization failure path, and current main still has a single local embedding initialization path that rethrows failures; the remaining review finding is reproducible by making the CPU fallback createEmbeddingContext call reject.

## Solution Assessment

No, not as written. The embedding direction is maintainable, but it needs to be rebased onto the package implementation and wrap CPU fallback initialization so cpuModel/cpuLlama are disposed even when context creation fails.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Dispose CPU fallback resources when init fails:** `src/memory-host-sdk/host/embeddings.ts:212`
  - body: If the CPU fallback model loads but `cpuModel.createEmbeddingContext()` rejects, this helper throws before returning the `dispose` callback, so `cpuModel` and `cpuLlama` are never released. Repeated fallback failures can leak native model/runtime resources; wrap CPU context creation in a cleanup path that disposes both objects on failure.
  - confidence: 0.89
