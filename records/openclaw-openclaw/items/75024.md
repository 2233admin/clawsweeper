## Summary

Keep this PR open. Current main still only promotes Feishu `root_id` into the thread root context, while downstream topic routing depends on that root context; the proposed change is a narrow Feishu plugin fix with no blocking static-review finding.

## What This Changes

The PR updates Feishu message parsing to fall back from `root_id` to `reply_target_message_id`, `parent_id`, and `upper_message_id`, adds `upper_message_id` to the event type, and adds parser coverage for the fallback order.

## Best Possible Solution

Land a narrow Feishu plugin fix that preserves existing `root_id` and `thread_id` precedence, uses reply/parent ancestors only as fallbacks for missing roots, and keeps coverage around both parser normalization and topic-session routing.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu group topic event with `groupSessionScope: "group_topic"` or `"group_topic_sender"`, no `root_id`/`thread_id`, and a populated reply or parent message id; current main leaves `ctx.rootId` undefined, so downstream topic routing cannot use that ancestor as the session root.

## Solution Assessment

Yes, with one suggested strengthening. Normalizing the fallback at `parseFeishuMessageEvent` is the narrowest maintainable boundary because downstream routing and reply context already consume `ctx.rootId`; adding a dispatch-level regression test would make the PR stronger but I do not see a blocking correctness bug.
