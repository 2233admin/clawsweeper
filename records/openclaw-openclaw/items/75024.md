## What This Changes

The PR updates Feishu message parsing to derive `ctx.rootId` from root/reply/ancestor fields, adds `upper_message_id` to the event type, and adds parser fallback tests.

## Best Possible Solution

Land the Feishu plugin fix after adding the required Unreleased changelog entry; a dispatch-level regression for topic-session reuse would be useful but is not required to understand the narrow parser boundary.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu group/topic reply event with no `root_id` or `thread_id` but with `reply_target_message_id` or `parent_id`; current main leaves `ctx.rootId` unset in `parseFeishuMessageEvent`, so group-topic routing cannot use that ancestor as the session root.

## Solution Assessment

Yes, with the changelog gap fixed. Normalizing the fallback inside `parseFeishuMessageEvent` is the narrowest maintainable solution because downstream session routing and reply dispatch already consume `ctx.rootId`.
