## Summary

Keep this PR open for maintainer review. Current main still leaves `ctx.rootId` unset when Feishu message events omit `root_id`, and the PR is a narrow Feishu plugin patch that targets the parser boundary downstream routing already consumes; I did not find a concrete blocking correctness or security issue in the diff.

## What This Changes

The PR changes Feishu message parsing to fall back from `root_id` to `reply_target_message_id`, `parent_id`, and `upper_message_id`, extends the event type with `upper_message_id`, and adds parser regression tests for the fallback order.

## Best Possible Solution

Land a narrow Feishu plugin fix that preserves existing `root_id` and `thread_id` precedence, normalizes the best available reply ancestor only when the root is missing, and keeps regression coverage around parser fallback plus any maintainer-requested dispatch or debounce behavior.

## Reproduction Assessment

Yes. A focused reproduction is a Feishu group/topic reply event with no `root_id` or `thread_id` but with `reply_target_message_id` or `parent_id`; on current main, source inspection shows `parseFeishuMessageEvent` leaves `ctx.rootId` unset, so topic routing cannot use that ancestor as the session root.

## Solution Assessment

Yes, with optional strengthening. Normalizing the fallback in `parseFeishuMessageEvent` is the narrowest maintainable boundary because downstream routing and reply dispatch already consume `ctx.rootId`; a dispatch-level or debounce-key regression test would make the proof stronger.
