## Summary

Keep this PR open. Current main still persists `memoryFlushCompactionCount` as the post-increment count after a flush-triggered compaction, while the dedup gate still skips when that value equals `compactionCount`. The latest release tag still has the same behavior, the linked bug #12590 remains open, and the duplicate repair #69138 is closed, so this PR has not been superseded or implemented on main.

## Best Possible Solution

Keep this PR open for maintainer review and landing, or apply the same narrow fix directly: preserve the pre-increment `memoryFlushCompactionCount`, update the persisted-counter expectation from `2` to `1`, keep the consecutive-cycle regression test, retain the changelog credit, run the changed checks, and let #12590 close through the normal merge flow.
