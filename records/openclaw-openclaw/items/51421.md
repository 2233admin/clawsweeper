## Summary

Keep this PR open. Current main still contains the post-increment `memoryFlushCompactionCount` reassignment that the PR removes, the dedup gate still skips when the flush count equals the compaction count, and current tests still assert the old persisted value. The PR remains a focused implementation candidate for the open memoryFlush reliability bug rather than something already implemented on main.

## Best Possible Solution

Keep this PR open for maintainer review and landing as the canonical fix path. The best implementation is the narrow behavior change already proposed: preserve the pre-increment `memoryFlushCompactionCount`, update the persistence test and consecutive-cycle regression test, keep changelog credit, then let the linked bug close through the normal merge flow.
