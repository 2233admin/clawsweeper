## Summary

Keep open. Current main still sends a bare session-reset startup prompt, optionally preceded by runtime-loaded daily memory, without explicitly telling the model that the reset turn contains no user-provided calendar, email, or JSON facts and that real-world claims must be verified before action.

## What This Changes

The issue asks OpenClaw to prevent session-reset startup turns from fabricating structured or calendar-like data and treating it as user-provided input.

## Best Possible Solution

Add a built-in reset/startup guardrail that treats bootstrap files, daily memory, and workspace context as context only, makes clear that the reset turn has no user-provided structured facts unless a reset-tail message exists, keeps the first reply brief or quiet when appropriate, and requires tool verification before acting on calendar/email/task data.

## Reproduction Assessment

Unclear. There is a high-confidence static path showing current main can send the startup prompt plus optional daily memory to the model on reset, but the fabricated calendar JSON needs a provider transcript or live Xiaomi reproduction to prove end-to-end.

## Solution Assessment

Yes. A narrow guardrail in the reset prompt and startup prelude, backed by focused reset/gateway tests, is the best repo-level fix; the reporter's AGENTS.md workaround is useful evidence but should not be required in every workspace.
