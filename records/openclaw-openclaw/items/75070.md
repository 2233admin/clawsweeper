## Summary

Keep open. Current main still lacks an explicit session-reset anti-fabrication guardrail, and the issue is paired with open PR #75127 using closing syntax, so cleanup should wait for that PR or an equivalent fix to land.

## What This Changes

The issue requests that bare session reset startup avoid fabricated calendar/email/JSON-style data and require a short greeting, silence, or tool verification instead.

## Best Possible Solution

Review and land PR #75127, or an equivalent narrow fix, so all bare reset prompt variants explicitly forbid presenting invented data as real and require tool verification before acting on calendar, email, task, or structured-data facts.

Unclear. The prompt path is statically reproducible by following bare `/new` or `/reset` into `resolveBareSessionResetPromptState`, but the actual fabricated-data response is model/provider-dependent and the issue provides no transcript logs.
