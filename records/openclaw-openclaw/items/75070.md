## Summary

Keep open. Current main still has model-run reset/startup paths that send a fresh-session prompt and may prepend runtime-loaded daily memory, but the reset prompt does not explicitly tell the model that the reset turn contains no user-supplied calendar/email/JSON data or require verification before acting on real-world facts. Bare chat `/new` and `/reset` are already acknowledged without invoking the model, so the remaining gap is the gateway/model-run reset surface.

## What This Changes

The issue asks OpenClaw to prevent empty or bootstrap session-reset turns from causing fabricated user data, and to reply with a short greeting or no-op unless the user supplied real reset-tail input.

## Best Possible Solution

Harden the reset/startup model-run contract in the reset prompt and any runtime startup prelude so bootstrap files, daily memory, and workspace context are clearly context only, not user-provided facts; the model should greet briefly or stay quiet unless there is reset-tail input or pending BOOTSTRAP work, and it should verify real-world calendar/email/task facts with tools before acting.

## Reproduction Assessment

Unclear. There is a high-confidence static path on current main showing bare gateway/model-run resets can send the reset startup prompt plus optional daily memory, but the fabricated JSON/calendar output is provider-specific and the report lacks a transcript or live credentials.

## Solution Assessment

Yes, a narrow guardrail in the reset prompt/prelude with regression tests is the best repo-level fix. The reporter's AGENTS.md workaround is useful evidence, but this should not require every workspace to carry its own reset-safety wording.
