## Summary

Keep open. Current main still forwards caller-supplied Firecrawl scrape URLs to the Firecrawl API without a target guard, so the security hardening targets a real remaining gap. The proposed patch should not merge as-is because the latest diff imports `SsrFBlockedError` from a plugin SDK subpath that current main does not export, which would break Firecrawl client module loading before the guard can run.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper, calls it from `runFirecrawlScrape`, and adds tests for allowed and blocked scrape URL inputs while the PR body also describes cache-cap behavior.

## Best Possible Solution

Revise this as a narrow Firecrawl security fix: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF helpers and errors from exported plugin SDK subpaths, keep focused regression tests for the client and scrape caller surfaces, and align the PR body with the actual shared cache behavior and Firecrawl hosting policy.
