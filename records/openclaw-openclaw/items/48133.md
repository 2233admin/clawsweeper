## Summary

Current main still does not implement the PR's central Firecrawl SSRF fix: direct `firecrawl_scrape` calls forward the caller-supplied URL to Firecrawl without local URL/hostname validation. The cache-size concern is already covered by the shared 100-entry `writeCache` cap, but the security behavior remains missing. The PR should stay open, but it needs security/import review before landing because the submitted diff references a non-exported SDK subpath and uses a static hostname check with an acknowledged DNS-resolution limitation.

## Best Possible Solution

Keep this PR open and revise it into a narrow Firecrawl security fix: validate the direct `firecrawl_scrape` and Firecrawl API facade URL before cache lookup or API forwarding, use an exported SDK SSRF helper such as `openclaw/plugin-sdk/ssrf-runtime` or `security-runtime`, add direct Firecrawl tests for metadata, localhost, private IP, malformed, credential-bearing, and non-HTTP URLs, and either remove the cache-cap claim or align it with the existing shared 100-entry `writeCache` behavior. Security review should decide whether literal-host blocking is sufficient for the remote Firecrawl model or whether the DNS-resolution limitation needs a stronger design or explicit documentation.
