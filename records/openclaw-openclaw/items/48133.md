## Summary

Keep open: current main still has the Firecrawl scrape SSRF gap and this PR is a plausible fix direction, but the latest patch has a blocking SDK import failure and needs maintainer/security review before merge.

## What This Changes

The PR adds a Firecrawl scrape URL validation helper and tests to block non-HTTP(S), localhost/private, and metadata scrape targets before forwarding caller-supplied URLs to Firecrawl.

## Best Possible Solution

Revise this as a narrow Firecrawl security fix: validate caller-supplied scrape targets in `runFirecrawlScrape` before cache lookup and API forwarding, import SSRF helpers from exported SDK subpaths, decide and document the hosted versus self-hosted Firecrawl SSRF policy, keep focused regression tests, and align the PR body with the actual shared cache behavior.
