## Summary

Keep this issue open. Current main still routes the Control UI New Session button and typed `/new`/`/reset` through `chat.send` on the active session key, while the independent dashboard-session creation path remains `sessions.create`. Gateway tests also preserve in-place reset semantics for `chat.send`, so the reported Control UI create-vs-reset gap is still real and narrow.

## Best Possible Solution

Implement a Control UI-owned fresh-session path: the New Session button and Control UI typed `/new` or `/reset` should call `sessions.create` for the active agent, switch the UI to the returned key with the existing session-switch machinery, refresh sessions/history/avatar/commands, and preserve non-UI gateway `chat.send` reset semantics unless maintainers intentionally change that contract.
