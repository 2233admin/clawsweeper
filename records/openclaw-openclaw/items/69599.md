## Summary

Keep this issue open. Current main still routes the Control UI New Session button and typed `/new`/`/reset` through `chat.send` on the active UI session key, while the independent fresh dashboard-session path exists separately as `sessions.create`. The prior ClawSweeper comment remains accurate against current `main` and the issue is narrow enough for a focused UI-owned fix.

## What This Changes

Control UI `/new`, `/reset`, and New Session should create and switch to a fresh independent session identity like direct dashboard session creation instead of reusing the active main session key.

## Best Possible Solution

Add a Control UI-owned create-and-switch flow: call `sessions.create` for the active agent, pass the current session as parent when valid, switch with `switchChatSession` to the returned key, refresh session/history/avatar/commands state, and leave generic gateway/channel reset behavior unchanged unless maintainers make a broader product decision.

## Reproduction Assessment

Yes. Current main statically reproduces the issue: New Session and typed `/new`/`/reset` call `chat.send` with the active `state.sessionKey`, while direct fresh session creation is the separate `sessions.create` RPC.

## Solution Assessment

Yes. The narrowest maintainable fix is to change the Control UI affordances to use `sessions.create` and then switch to the returned session key, while keeping gateway `chat.send` reset behavior in place for non-UI callers.
