## Summary

Keep open. Current main still does not expose or thread `gatewayUrl`/`gatewayToken` through `sessions_spawn` or `sessions_send`; native subagent and ACP dispatch still call the current gateway, while the existing safe gateway override helper is only wired into other gateway-backed tools.

## Best Possible Solution

Keep this open for maintainer design. The best path is to define a native remote target contract for `sessions_spawn` and `sessions_send`, reuse the existing safe gateway override/credential-resolution seam, explicitly decide remote credential/workspace/transcript ownership, and add targeted tests proving the child work executes on the remote gateway rather than only routing metadata locally.

## Risks / Open Questions

- Remote session delegation is security-sensitive because it changes where model credentials, filesystem access, command execution, transcripts, and tool effects occur.
- A naive implementation could leak local gateway credentials to a remote target or bypass the existing allowlisted gateway override validation in `src/agents/tools/gateway.ts`.
- The product contract still needs maintainer design for remote model/account selection, workspace ownership, transcript storage, delivery/announce routing, and failure cleanup.
- No live two-gateway reproduction was run in this read-only review; the keep-open decision rests on current source and docs showing the requested session-tool plumbing is absent.
