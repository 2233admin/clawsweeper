## Summary

Keep this PR open. Current main still does not provide custom Feishu account selection in the onboarding wizard, and the PR is useful source work, but the proposed diff does not yet complete the feature because the selected account ID is not threaded into the Feishu config-writing flow and the fresh-install guard would misfire.

## What This Changes

The PR changes Feishu setup/account code to export the default account resolver, normalize setup account IDs, and prompt for a custom Feishu account ID during onboarding.

## Best Possible Solution

Land a corrected Feishu-owned setup change that uses the framework-resolved accountId through resolveAccountIdForConfigure, finalize, runEditFlow, runNewAppFlow, credential writes, and policy writes; preserve default first-account behavior; and cover the second-account path with focused tests.

## Repair Work Prompt

Repair the Feishu custom-account onboarding work from https://github.com/openclaw/openclaw/pull/68455. Preserve the PR's intent: first Feishu setup should keep using the default account without a confusing prompt, while configuring an additional Feishu bot should let the user choose/create a named account and write credentials, domain, dmPolicy, allowFrom, and group policy under `channels.feishu.accounts.<id>` instead of the top-level/default account. Prefer the existing setup framework contract: `resolveAccountIdForConfigure`/generic account picker may be reused, but the resolved `accountId` must be passed through `finalize`, `runEditFlow`, and `runNewAppFlow` into every `patchFeishuConfig`/policy write. Do not add core special cases, workflow/dependency changes, or unrelated Feishu runtime refactors. Add focused tests in `extensions/feishu/src/setup-surface.test.ts` for fresh setup staying default, existing setup choosing a new account, and the selected account receiving credentials/policies. Changelog is probably not needed unless maintainers want a user-facing note for Feishu onboarding.
