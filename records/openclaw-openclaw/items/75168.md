## Summary

Keep open. Current main still has unguarded asynchronous channel lifecycle logger dereferences, and the provided timeline shows an open implementation PR, #75172, linked to this issue, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

Prevent gateway channel lifecycle handlers from crashing when a channel logger entry is missing during asynchronous startup, exit, cleanup, or restart handling.

## Best Possible Solution

Land #75172 or an equivalent narrow gateway fix after review, with defensive logger guards plus regression coverage for a channel that is active in the registry while missing from the logger/runtime maps.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a deferred or re-pinned channel plugin that starts after `channelLogs` is built, then resolves or rejects in `startAccount`; the async continuation reaches `log.error?.(...)` with `log` undefined.

## Solution Assessment

No, not as a close outcome yet. The reported `log?.error?.(...)` guard is the right defensive part, but the maintainable fix should also confirm the missing logger/runtime-map path is either handled or covered by regression tests before this issue closes.
