## Summary

Keep open. Current main and the latest release still contain the unguarded async channel lifecycle logger dereferences, and the provided GitHub context shows open PR #75172 already references this issue with closing syntax, so the issue should remain open until that implementation is reviewed and merged or closed.

## What This Changes

The issue reports a gateway crash caused by unguarded per-channel logger dereferences during async channel lifecycle error handling.

## Best Possible Solution

Review and land #75172, or an equivalent narrow gateway fix, with regression coverage for a channel plugin that is present in the active registry while its `channelLogs` entry is missing during startup, cleanup, exit, restart, or stop-timeout handling.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a channel manager started with a missing `channelLogs[channelId]` entry and a channel task that rejects or exits, which reaches `log.error?.(...)`/`log.info?.(...)` after `log` is `undefined`.

## Solution Assessment

Yes. Guarding the async lifecycle call sites as `log?.error?.(...)`, `log?.info?.(...)`, and `log?.warn?.(...)` is the narrow crash fix; map synchronization can be reviewed separately if maintainers want a broader cleanup.
