## What This Changes

Fix a gateway crash where async channel lifecycle handlers can dereference `log.error` or `log.info` after `channelLogs[channelId]` is missing.

## Best Possible Solution

Review #75172 as the active implementation candidate, but make sure the final fix both guards all async channel lifecycle logging against a missing logger and preserves valid logger/runtime entries for channels introduced by deferred plugin reloads, with a focused regression test in `src/gateway/server-channels.test.ts`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a deferred channel plugin that becomes active after `channelLogs` is built, then exits or rejects in `startAccount`, causing an async continuation in `startChannelInternal` to evaluate `log.error` or `log.info` when `log` is undefined.

## Solution Assessment

No, not as only `log?.error?.(...)` everywhere. That guard is necessary to stop the crash, but the more maintainable fix should also account for deferred channels missing logger/runtime entries rather than only suppressing the resulting TypeError.
