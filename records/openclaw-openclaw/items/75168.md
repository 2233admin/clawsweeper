## Summary

Keep open. Current main and the latest release still contain the unguarded per-channel logger dereferences in the async channel lifecycle path, while open PR #75172 already references this issue with closing syntax and is the active implementation candidate to review before closing the issue.

## What This Changes

The issue reports a gateway crash when async channel lifecycle handlers dereference an undefined per-channel logger after channel/plugin lifecycle state changes.

## Best Possible Solution

Land #75172 or an equivalent narrow gateway fix that guards every async lifecycle logger dereference against a missing channel logger, adds a focused regression test for logger-map eviction/missing entries, and checks whether the deferred plugin reload path also needs logger/runtime map synchronization.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an active channel plugin id that is absent from `channelLogs`, followed by a resolving, rejecting, cleanup-failing, or auto-restart channel task in `startChannelInternal`, which reaches `log.error?.(...)` or `log.info?.(...)` with `log` undefined.

## Solution Assessment

Unclear. The reporter’s `log?.error?.(...)` direction is the narrow defensive fix for the crash, but the final patch should also verify that deferred channel plugin reload cannot leave related runtime maps inconsistent.
