## Summary

Keep open. Current main still has no Active Memory single-shot mode: the config type/schema expose query and prompt tuning but no execution mode, eligible prompt builds still await the Active Memory recall path, and that path still calls the embedded Pi agent with memory tools. The hard-timeout sibling from #72347 is implemented, but this separate latency-architecture request remains valid.

## What This Changes

The issue requests an opt-in Active Memory mode that performs one memory search and one model completion without the embedded agent loop.

## Best Possible Solution

Add an opt-in Active Memory execution mode in the active-memory plugin, keep the existing embedded-agent mode as the default, and implement the new path with one bounded memory search plus one simple model completion, with docs and regression coverage for config validation, no tool loop, timeout behavior, and summary/NONE parsing.

## Reproduction Assessment

Yes. A source-level reproduction is enough for the missing capability: current main rejects or ignores a proposed mode field because the schema/type lack it, and eligible prompt builds always route through runEmbeddedPiAgent rather than a direct single-shot branch.

## Solution Assessment

Yes, the requested opt-in mode is the narrowest maintainable direction for this specific issue because it preserves the existing recall-heavy agent-loop behavior while giving latency-sensitive preference-only use cases a bounded one-search/one-completion path. The safer implementation is a small plugin/runtime memory-search seam plus simple-completion reuse, not cross-plugin deep imports.
