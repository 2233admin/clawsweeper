Keep this open. Current `main` still appears to have the reported false-positive path in memory-wiki lint, and the issue is already paired with open PR #73722 using GitHub closing syntax, so it should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests that `openclaw wiki lint` resolve existing memory-wiki pages when Obsidian links use page titles, slug variants, fragments, or imported source-path suffixes.

## Best Possible Solution

Land PR #73722 or an equivalent focused memory-wiki change that normalizes lint lookup keys and link targets in `extensions/memory-wiki/src/lint.ts`, preserves imported source provenance, and adds regression coverage in `extensions/memory-wiki/src/lint.test.ts`.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create an existing page whose title, slug, fragment-stripped Obsidian link, or source-path suffix differs from its exact extensionless relative path/basename, then `collectBrokenLinkIssues` will not find it in the current target set.

## Solution Assessment

Yes. The narrowest maintainable solution is to fix the memory-wiki lint resolver and add focused lint tests; the already-open PR #73722 appears aimed at exactly that boundary, so the issue should wait on PR review rather than a separate repair lane.
