## Summary

Keep open. Current main and the latest release still compare raw memory-wiki lint targets against only extensionless relative paths and basenames, so title, slug, fragment-bearing, and imported source-path variants can still be reported as broken; open PR #73722 is already the paired closing implementation candidate.

## What This Changes

`openclaw wiki lint` should resolve Obsidian wikilinks through title, slug, fragment-stripped, path/basename, and imported source-path suffix variants without rewriting imported source evidence.

## Best Possible Solution

Land PR #73722, or an equivalent memory-wiki-owned fix, that expands lint lookup normalization, strips fragments and queries consistently, preserves imported provenance, and adds focused regression coverage while leaving broader markdown-aware extraction work to #70986.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a page titled `Alpha Database` stored as `syntheses/alpha-db.md` and linked as `[[Alpha Database#Evidence]]` leaves a fragment-bearing target that current lint will not match against its `syntheses/alpha-db` and `alpha-db` keys.

## Solution Assessment

Yes. The narrow maintainable fix is in the memory-wiki lint resolver and colocated tests; rewriting imported source evidence would damage provenance and should not be the solution.
