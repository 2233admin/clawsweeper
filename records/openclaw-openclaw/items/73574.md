## Summary

Keep this open: current main still compares wiki lint targets against only exact extensionless relative paths and basenames, while an open linked PR explicitly closes this issue and implements the requested normalization with regression coverage. Repository policy says not to close an issue while its closing PR remains open.

## What This Changes

The issue requests that `openclaw wiki lint` normalize Obsidian link targets by title, slug, fragment-stripped path, basename/path variants, and imported source-path suffixes before reporting `broken-wikilink`.

## Best Possible Solution

Land PR #73722 or an equivalent focused memory-wiki lint change that indexes normalized page/title/slug/source-path keys, strips fragments and query suffixes before lookup, and keeps compile, ingest, rendering, and search behavior unchanged.

## Reproduction Assessment

Yes. A vault with a page titled `Alpha Database` and a link like `[[Alpha Database#Evidence]]` has a high-confidence source-level reproduction because current main stores the fragment-bearing Obsidian target and compares it against only `syntheses/alpha-db` and `alpha-db`.

## Solution Assessment

Yes. The linked PR's direction is the narrowest maintainable fix: keep the change inside the memory-wiki lint resolver and add a regression test for the reported link variants rather than rewriting imported evidence or changing compile/render behavior.
