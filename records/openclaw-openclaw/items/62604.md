## Summary

Keep open. Current main handles clipboard entries that expose an `image/*` item, but it does not harden the fallback where the clipboard exposes a `data:image/...;base64,...` payload as text, so the browser can still insert that payload into the composer. The send/draft restoration observation is already aligned with current code, and the installed-package source-map concern is real but should stay secondary to the paste hardening.

## What This Changes

The issue requests safer Control UI image paste handling, clearer send/draft failure behavior, and better guidance or traceability for debugging installed hashed Control UI bundles.

## Best Possible Solution

Harden the Control UI composer paste path so pasted image data URLs are either converted into normal attachments or blocked with a clear local error before they become draft text, add focused paste regression tests, and keep draft/attachment restoration in the existing send pipeline. Treat installed-bundle hotpatching as unsupported unless maintainers explicitly choose to change package/source-map policy.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a paste event whose clipboard has no `image/*` item but does have a text/plain `data:image/...;base64,...` payload; current `handlePaste` returns without preventing the browser's default textarea insertion.

## Solution Assessment

Yes for the paste bug: the narrowest maintainable fix is in `ui/src/ui/views/chat.ts` with tests in `ui/src/ui/views/chat.test.ts`, plus any send serialization assertions needed. Shipping source maps or changing npm package contents is a separate product/security decision and should not be bundled into the first repair unless requested.
