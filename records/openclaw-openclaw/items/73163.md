## Summary

Keep open. The PR is a focused implementation for the linked Control UI startup-warning issue and is mostly correct, but it should avoid false-positive warnings when gateway auth or Control UI enablement is supplied through resolved runtime overrides rather than raw config; exact-head CI was also still partly in progress during review.

## What This Changes

The PR adds a gateway startup warning for non-loopback plain-HTTP Control UI access that requires a secure browser context, plus warn/quiet regression tests and a changelog entry.

## Best Possible Solution

Land the startup warning after threading resolved runtime state into the predicate and after exact-head CI is green, preserving the existing Control UI auth rejection policy and keeping broader LAN/HTTP auth behavior in separate product/security discussions.

## Reproduction Assessment

Yes. Current main can be verified statically: non-secure browsers skip device identity, the gateway rejects remote Control UI operator sessions without device identity, and startup logging currently has no warning; the PR edge case is reproducible by comparing resolved runtime overrides with the raw cfg-only warning predicate.

## Solution Assessment

Mostly yes. Startup logging is the right narrow boundary for operator guidance, but the predicate should use the already resolved Control UI enabled state and auth mode rather than only `cfg.gateway` so quiet cases match actual runtime behavior.

## Review Findings

Overall correctness: patch is correct
