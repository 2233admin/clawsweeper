## Summary

Keep open. Current main still lacks the requested startup warning, and this PR is the right narrow surface with tests and changelog coverage, but the latest head still derives the warning from raw config instead of the resolved gateway runtime state, so CLI/server overrides can produce a misleading warning.

## What This Changes

The PR adds a gateway startup warning for non-loopback plain-HTTP Control UI access that will require secure-context device identity, adds warning/quiet regression tests, and adds a changelog entry.

## Best Possible Solution

Thread the resolved Control UI enabled state and resolved auth mode into the startup logging boundary, keep the existing auth rejection behavior unchanged, add focused quiet-case tests for CLI/server overrides, and then land after exact-head checks finish green.

## Reproduction Assessment

Yes. Current main can be verified statically: non-secure browser contexts skip device identity, the gateway rejects remote Control UI sessions without device identity, and startup logging has no targeted warning; the PR regression is reproducible by using runtime auth/controlUiEnabled overrides that differ from cfgAtStart.

## Solution Assessment

Mostly yes. Startup logging is the narrow maintainable place for this operator guidance, but the predicate should evaluate resolved runtime state rather than only cfg.gateway so the warning matches actual startup behavior.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Use resolved runtime state for the warning:** `src/gateway/server-startup-log.ts:66-72`

Pass the resolved Control UI enabled state and auth mode into the logger and cover those override cases.
