## What This Changes

The PR adds a gateway startup warning for non-loopback plain-HTTP Control UI access, suppresses it for loopback/TLS/disabled UI/device-auth-bypass cases, and adds startup-log tests plus related package-acceptance and unit-fast-path test expectation updates.

## Best Possible Solution

Land an updated narrow startup-log change, through this PR or a replacement, that warns when the resolved Gateway listener makes remote plain-HTTP Control UI connections likely to fail while preserving the existing auth policy and documented exceptions.

## Reproduction Assessment

Yes. Static reproduction is high confidence: call `logGatewayStartup` on current main with Control UI enabled, `bindHost: "0.0.0.0"`, and `tlsEnabled: false`; no Control UI secure-context warning path exists, while connect-policy code still rejects the corresponding device-less remote Control UI connection.

## Solution Assessment

Yes, with a small review caveat. A startup warning after bind/TLS/config resolution is the narrowest maintainable fix for #71669, but maintainers should decide whether to refine the predicate for trusted-proxy or HTTPS-terminated reverse-proxy deployments before landing.
