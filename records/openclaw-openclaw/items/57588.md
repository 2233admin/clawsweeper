## Summary

Keep open. Current main still contains the reported permissive bearer-auth fallback behavior: HTTPS enforcement is opt-in, `shouldAttachAuth` is optional, and the helper retries a 401/403 by setting `Authorization: Bearer ...` when no predicate is supplied. The helper does not appear in the current documented/exported SDK subpath catalog and no in-repo production caller was found, which reduces demonstrated exploit reach, but it does not satisfy any close reason because the source and tests still encode the requested hardening gap.

## Best Possible Solution

Keep the issue open for a narrow core hardening decision: either remove the unused helper, keep it internal and explicitly mark it unsupported, or change it to fail closed by default with HTTPS required and an explicit `shouldAttachAuth`/allowlist contract, then update the current tests to cover the fail-closed behavior. The related fetch-guard redirect-header issue should remain separate because it covers a different request path.
