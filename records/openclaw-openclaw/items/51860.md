## Summary

Keep open. Current main partially addresses the later TMPDIR failure by using a durable macOS service temp directory, but the central request remains unimplemented: macOS gateway service management is still LaunchAgent/gui-domain only, there is no `gateway install --system`, and docs still say headless macOS needs a custom LaunchDaemon that OpenClaw does not ship.

## Best Possible Solution

Keep this issue open and implement an explicit macOS system-service mode if maintainers want boot-without-login support: add `openclaw gateway install --system`, generate `/Library/LaunchDaemons/ai.openclaw.gateway.plist` with safe ownership/permissions and `UserName`, use the existing durable state temp path, teach status/restart/stop/uninstall/doctor to target the system launchd domain, and document the operational/security tradeoffs. If maintainers decide not to support this in core, the issue should be closed by explicit product decision rather than swept as already implemented.

## Risks / Open Questions

- The remaining work is security-sensitive because a system LaunchDaemon path needs root-owned plist permissions, `UserName` handling, state/home ownership checks, system-domain `launchctl` status, and uninstall/doctor behavior.
- The stale TMPDIR symptom should not be re-fixed in isolation; current main already has a durable macOS service TMPDIR path, so this issue should stay focused on first-class LaunchDaemon install/status support.
