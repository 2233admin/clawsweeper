## Summary

Keep this issue open. Current main has WhatsApp reconnect/append handling, proxy/timing support, and generic OpenShell sandbox docs, but it still does not prove or document the reported Docker-to-k3s WhatsApp inbound failure. The discussion remains plausible as a deployment/network event-delivery problem rather than an already-shipped OpenClaw fix.

## Best Possible Solution

Keep this open as a WhatsApp/OpenShell deployment triage item. Reproduce or rule out Baileys WebSocket/event delivery inside the nested k3s pod, then either land targeted OpenShell/k3s networking guidance or add WhatsApp plugin diagnostics/runtime support if proxy, socket, or health visibility is missing for that environment.
