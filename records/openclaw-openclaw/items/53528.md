## Summary

Keep this PR open. Current main does not implement the PR's intended `jobs.json` watcher or an equivalent idle reload path: cron force-reloads from disk only inside timer ticks, and `armTimer` still returns without scheduling a timer when there are no enabled jobs or next run times. The linked #52772 report is closed, but the specific startup-empty/direct-file-write problem is not clearly implemented on main.

## Best Possible Solution

Keep the PR open, but rework it before merge: implement a core cron-owned watcher or bounded idle poll that runs through the existing locked store/recompute/timer path, wakes or re-arms the scheduler after external changes, suppresses self-write invalidations, handles missing-file and rename semantics, and adds targeted regression tests for direct `jobs.json` writes after empty startup.

## Risks / Open Questions

- The PR should not merge as-is: provided review comments identify watcher correctness gaps, including no scheduler wake after invalidation, missing-file startup behavior, self-write invalidation, and file-replacement watcher loss.
- Current docs say externally edited `jobs.json` schedules are reloaded, but current code only proves reload on timer/manual paths; startup with no jobs still appears to have no idle polling or watch trigger.
- A safe implementation needs regression coverage for empty or missing `jobs.json` at startup, external write or create while running, self-writes via `persist`, rename-based saves, and timer re-arming.
