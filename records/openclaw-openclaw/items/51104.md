## Summary

Keep this issue open. Current main at a820a307dfeb950fa633c0ae6fb2386a35a91991 still lacks Mattermost WebSocket protocol ping/pong keepalive; the Mattermost connector only handles open/message/close/error plus the bot update_at REST poll. The secondary gateway stale-event threshold is configurable now, but that does not prove Mattermost socket liveness and Mattermost does not publish lastTransportActivityAt. PR #57621 remains the relevant open implementation candidate in the provided GitHub context, so this should stay open for review/landing rather than be closed.

## Best Possible Solution

Keep this issue open and review or land PR #57621, or an equivalent Mattermost-owned patch in extensions/mattermost/src/mattermost/monitor-websocket.ts that adds WebSocket ping(), listens for pong, terminates after missed pong cycles so runWithReconnect reconnects, cleans up timers on close/error/abort, and adds focused monitor-websocket tests. After that merges, close this issue and related Mattermost half-dead WebSocket reports such as #41837 or #44160 as implemented or duplicates of the landed fix.

## Risks / Open Questions

- Leaving the current code unchanged can still allow a Mattermost WebSocket to appear connected while receiving no inbound events until some separate fallback fires or an operator restarts/reloads the channel.
- The existing update_at poll detects bot-account modification cycles, not NAT/proxy/TCP half-dead sockets; treating it as equivalent to ping/pong would miss the reported failure mode.
- PR #57621 should still be reviewed for timer cleanup, abort/close/error behavior, fake WebSocket coverage, and config-surface fit before landing.
