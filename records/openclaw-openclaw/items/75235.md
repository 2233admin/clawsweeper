## Summary

Keep open. Current main still matches the reported failure class: assistant-only failed turns can be filtered out before Anthropic transport, `buildAnthropicParams` can still send the converted message list without a non-empty guard, and the auto-reset path does not recognize Anthropic's `messages: at least one message is required` schema error, so the session can fall through to the generic external failure instead of being reset.

## What This Changes

The issue requests recovery for assistant-only or zero-user corrupted session transcripts that repeatedly trigger Anthropic `messages: at least one message is required` failures, plus clearer reset confirmation UX.

## Best Possible Solution

Add a focused corruption-recovery path for Anthropic-compatible schema failures caused by zero replayable user messages or leading assistant-only error transcripts. The durable end state should archive the bad transcript, rotate or drop the poisoned active session, avoid replaying assistant-only failed history into the replacement session, return the existing friendly corruption-reset message, and keep any reset readiness UX aligned with the documented session command model.

## Reproduction Assessment

Yes. A source-level reproduction is clear: load a transcript containing only assistant `stopReason: "error"` sentinel turns, run through Anthropic replay, observe those turns filtered before conversion, and hit the runner path where the resulting schema error is not classified as corruption.

## Solution Assessment

Yes for a narrow auto-reset or read-time repair guard that treats assistant-only/no-user Anthropic replay as corrupt. A generic empty Anthropic-message fallback alone is not the best fix because it can mask the provider 400 while leaving the poisoned session and unclear reset UX in place.
