## Summary

Keep this open. Current main still has a source-level path that matches the report: a profile with both `cdpPort: 18800` and `cdpUrl: "http://127.0.0.1"` will prefer the non-empty URL, parse the missing port as HTTP port 80, and then surface the launch failure as `Failed to start Chrome CDP on port 80`. The docs say local managed profiles should not set `cdpUrl`, but the current schema allows the mixed config and the launcher does not fail early with an actionable config error.

## What This Changes

The issue asks OpenClaw to make `openclaw browser start` work for a macOS managed Chrome profile whose config contains both `cdpPort: 18800` and a no-port loopback `cdpUrl`.

## Best Possible Solution

Normalize or validate mixed browser profile endpoint config so OpenClaw no longer silently launches/probes port 80 for a local managed loopback profile that also declares `cdpPort`. The maintainable end state is a regression test plus either a compatibility normalization to the explicit profile port or an early, actionable config/doctor error telling users to remove `cdpUrl` for managed profiles.

## Reproduction Assessment

Yes. The source path is high-confidence: the reported config’s no-port loopback `cdpUrl` is parsed as port 80 and overwrites the explicit profile `cdpPort` before the managed Chrome launch error is constructed.

## Solution Assessment

Unclear for the exact policy, but the current behavior is not the best solution. The safest implementation should be narrow: handle no-port loopback `cdpUrl` plus explicit `cdpPort` for managed profiles, without changing remote CDP default-port semantics.
