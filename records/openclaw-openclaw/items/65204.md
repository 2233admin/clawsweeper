## Summary

Keep open. The reported config combines a managed `openclaw` profile `cdpPort: 18800` with `cdpUrl: "http://127.0.0.1"`; current `main` still lets the profile `cdpUrl` branch overwrite the port with the URL parser's default HTTP port 80, matching the reported Chrome bind failure. Related browser-CDP fixes cover readiness, SSRF, and retry behavior, but not this missing-port `cdpUrl` plus `cdpPort` conflict.

## Best Possible Solution

Keep this issue open. The browser plugin should either validate this conflicting managed-profile config with an actionable error or resolve it conservatively by honoring `cdpPort` for local managed profiles when `cdpUrl` is a bare loopback host without an explicit port. Add a regression test in `extensions/browser/src/browser/config.test.ts` for the reporter's config shape, and consider a doctor/fix rule that removes stale local managed `profiles.<name>.cdpUrl` values while preserving true remote CDP profiles.
