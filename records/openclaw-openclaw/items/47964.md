## Summary

Keep open. Current main still only normalizes `mode` and `apiKey` plus static SecretRef fields when loading auth profiles, while downstream OAuth eligibility and PI conversion require canonical `access`, `refresh`, and `expires`; the reported alias-shaped stored credential failure remains plausible and is distinct from adjacent stale-profile/default-profile fixes.

## What This Changes

The issue asks maintainers to confirm or implement compatibility handling for older OpenAI Codex OAuth auth-profile field names that are present in `auth-profiles.json` but not usable at runtime.

## Best Possible Solution

Define a narrow auth-profile compatibility policy for Codex OAuth credentials: either normalize specific known legacy aliases on read with canonical fields winning conflicts, or keep the schema strict and add a clear doctor/models-status diagnostic for present-but-unusable OAuth profiles, with focused regression coverage for the chosen behavior.

## Reproduction Assessment

Yes. A high-confidence source-traced reproduction is to load an `openai-codex` OAuth profile containing only alias token fields such as `access_token`/`refresh_token` or `accessToken`/`refreshToken`; the loader leaves those fields untouched, and downstream canonical checks treat the profile as unusable. I did not create a live fixture because this review was read-only.

## Solution Assessment

Unclear until maintainer policy is set. The narrowest maintainable fix likely belongs in the auth-profile loader or diagnostics path, not in Codex refresh, profile rotation, or PI discovery special-casing.
