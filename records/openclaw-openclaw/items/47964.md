## Summary

Current main does not clearly implement the requested compatibility path. The auth-profile loader normalizes a few legacy fields (`mode`, `apiKey`, SecretRef-shaped key/token values), but I did not find normalization for legacy OAuth token field names such as `access_token`/`refresh_token` or `accessToken`/`refreshToken`. The PI runtime conversion still requires canonical `access`, `refresh`, and positive numeric `expires`, so a profile with older OAuth field names can remain present but unusable. Related closed reports cover adjacent Codex auth/profile routing and refresh issues, but not this specific stored-field compatibility case.

## Best Possible Solution

Keep this issue open and implement a narrow auth-profile compatibility fix: normalize known legacy OAuth field aliases on load or provide a clear doctor/models-status warning when a stored OAuth profile is present but unusable due to schema drift. The fix should stay in the auth-profile owner path, add regression tests for `openai-codex` legacy OAuth shapes, and avoid changing Codex refresh/rotation behavior.
