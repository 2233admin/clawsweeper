## Summary

Keep this issue open. Current main still lacks a generic queued-followup feedback hook in the busy `enqueue-followup` path, and Feishu/Lark still only starts its typing reaction from reply-start callbacks that the queued path never reaches. Telegram, Discord, and Slack have some channel-level status reaction behavior, but that does not solve the reported Feishu/Lark gap or provide the shared channel seam requested here. Related work exists in #39117, and #39676 was closed unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep this issue open and route it through the existing maintainer review of #39117, or split a focused follow-up if that PR does not land. The best implementation is a generic, non-blocking queued-followup feedback seam in the active-run `enqueue-followup` path, such as an `onQueued`/queued-receipt callback on `GetReplyOptions` or the shared reply dispatcher. It should fire only when a followup is actually enqueued, swallow/log callback failures so queue reliability is preserved, wire Feishu/Lark to an immediate reaction or receipt, avoid duplicate signals for channels that already set status reactions, and add focused tests for busy queued followups.
