## Summary

Keep open: current main still has the reported ACP background-task behavior, and open PR #74103 still references this issue with closing syntax but is unmerged and already has a needs-changes review. The issue should remain open until that PR, or an equivalent sanitized/tested fix, lands.

## What This Changes

The issue requests that successful ACP background-task completions derive a useful `terminalSummary` from non-blocked progress output and raise the `progressSummary` cap from 240 characters to about 1200.

## Best Possible Solution

Land PR #74103 after addressing the needs-changes review, or replace it with an equivalent narrow ACP manager patch that copies cleaned successful progress into `terminalSummary`, preserves blocked-outcome handling, raises the cap only as an intentional UX choice, and adds regression coverage for useful summaries plus suppressed control/internal text.

## Reproduction Assessment

Yes. The static source path is high-confidence: non-thought ACP output populates `taskProgressSummary`, ordinary non-blocked success returns `{}`, `markBackgroundTaskTerminal` stores a null terminal summary, and terminal delivery emits the bare completion label.

## Solution Assessment

Unclear as currently proposed in the linked PR. Falling back from progress output and raising the cap is the right boundary, but the maintainable fix should strip silent-control tokens and include regression tests before the issue is closed.
