## Summary

Keep open: current main and the latest release still have the reported ACP background-task summary behavior, and open PR #74103 references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks ACP background-task completions to return a useful `terminalSummary` from successful `progressSummary` output and to raise the progress-summary cap from 240 characters to about 1200.

## Best Possible Solution

Review and land PR #74103 after it preserves blocked-outcome handling, strips internal/control-only text from successful progress fallbacks, raises the bounded progress cap, and adds regression coverage for both useful successful summaries and suppressed internal details.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The static current-main path is enough: non-thought ACP output populates `taskProgressSummary`, ordinary non-blocked success returns `{}`, terminal finalization stores a null summary, and delivery formats only `terminalSummary`.

## Solution Assessment

Is this the best way to solve the issue? Yes on the boundary: falling back from cleaned successful progress and increasing the bounded cap is the narrow maintainable fix, with sanitization and tests required before merge.
