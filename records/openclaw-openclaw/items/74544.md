## Summary

Keep open. Current main still has the reported diagnostic behavior: `memory status --deep` waits for the live embedding probe before rendering, that probe can take the normal 600-second local batch timeout on cache miss, and the existing progress call is silent for non-TTY callers. The issue is also now paired with open PR #75106 using closing syntax, so the right cleanup path is to review and land or close that PR rather than closing this issue early.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid silent multi-minute embedding-probe waits and to return a bounded, parseable embedding status or timeout while preserving vector/FTS/cache visibility.

## Best Possible Solution

Keep this issue open while maintainers review PR #75106 or an equivalent focused fix: bound the `memory status --deep` embedding readiness probe with a status-specific budget, emit non-TTY progress or a parseable timeout, keep vector/FTS/cache/index state visible, and preserve the longer batch timeout for real indexing work.

## Reproduction Assessment

Yes for the CLI diagnostic bug: current source shows `memory status --deep` awaits `probeEmbeddingAvailability()`, whose cache-miss path runs a live batch ping with the local 600-second timeout and no non-TTY progress fallback. The Windows/model-specific provider timeout itself is not independently reproduced in this read-only review.

## Solution Assessment

Yes, the linked PR direction appears to be the narrow maintainable solution if its implementation review passes: add a status-specific probe budget and progress fallback instead of lowering provider-wide batch timeouts. This issue should stay open until that PR lands or maintainers choose a different fix.
