## Summary

Keep open. Current main still permits the reported long silent diagnostic wait, and this issue is paired with open PR #75106 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `openclaw memory status --deep` to complete or fail with observable progress and a bounded diagnostic error instead of silently waiting up to the embedding batch timeout.

## Best Possible Solution

Review and land PR #75106 if its budget and output contract are acceptable, or land an equivalent memory-core fix that bounds the status-only embedding probe, emits non-TTY progress, and reports a distinct timeout state while leaving real indexing batch budgets intact.

## Reproduction Assessment

Yes for the primary diagnostic bug: current source shows `memory status --deep` awaits a live embedding batch ping that can use the 600-second local batch timeout before output is rendered. The provider-specific Windows timeout still needs reporter or maintainer live logs if it persists after the diagnostic budget fix.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix for this issue: bound only the status probe, expose a parseable timeout, and add progress output without changing indexing behavior. The deeper local-provider timeout can be investigated separately if it remains observable.
