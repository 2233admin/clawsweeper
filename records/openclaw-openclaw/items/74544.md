## Summary

Keep open. Current main still matches the corrected report: deep memory status waits for a live embedding probe before rendering output, a cache miss uses the normal batch embedding path, and the local/builtin batch timeout remains 600 seconds with no non-TTY progress fallback. The related memory-status and doctor reports overlap, but they do not fully supersede this issue’s specific silent diagnostic wait.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid silent multi-minute embedding-probe waits and to expose a bounded, parseable status or error when the main agent’s local embedding batch times out on Windows.

## Best Possible Solution

Separate diagnostic readiness from indexing workload behavior: deep status should quickly report vector, FTS, cache, and embedding-probe state with a short status-specific timeout and visible progress/error output, while preserving the longer provider batch timeout for actual indexing or embedding work. If the shorter probe still fails for this Windows/local model case, handle that as a follow-up provider-runtime investigation with DEBUG logs.

## Repair Work Prompt

Fix openclaw/openclaw issue #74544. Current main routes `openclaw memory status --deep` through `runMemoryStatus`, which awaits `manager.probeEmbeddingAvailability()` before printing output; the builtin manager probes with `embedBatchWithRetry(["ping"])`, and the local adapter/batch timeout is 600_000 ms. Non-TTY progress is also silent because the deep probe does not request a log fallback. Preserve the longer batch timeout defaults for real indexing/embedding work; do not globally lower local provider batch timeouts.
