## Summary

Keep open. Current `main` still makes `openclaw memory status --deep` wait for a live embedding readiness probe before printing the status report, and that probe uses inline batch timeout defaults that can be 600 seconds for local/self-hosted providers. The related open reports cover adjacent status/probe ambiguity, but none supersede this specific silent hang/readiness-timeout behavior.

## What This Changes

The issue asks for `openclaw memory status --deep` to complete or return a parseable timeout/error instead of hanging silently when basic memory status is already ready.

## Best Possible Solution

Add a diagnostic-specific budget for the `memory status --deep` embedding readiness probe, report a clear timeout/unavailable result in both human and JSON output, and keep the longer indexing/batch defaults for actual indexing work separate from status readiness.
