## Summary

Keep open. Current main still lets `memory status --deep` call the live embedding probe directly, and that probe still reaches the 600s local batch timeout path before returning an embeddings failure. The linked fix PR #75106 is open and explicitly closes this issue, so the issue should remain open until that PR, or an equivalent fix, lands.

## What This Changes

The issue asks for `openclaw memory status --deep` to stop waiting silently on the full local embedding batch timeout and to report bounded, parseable progress or timeout state.

## Best Possible Solution

Review and land PR #75106, or an equivalent memory-core change, so `memory status --deep` has a short diagnostic budget, non-TTY progress output, and a distinct timeout result. Keep broader live-probe semantics tracked in #47235 if provider flapping or misleading readiness remains after the command is bounded.

## Reproduction Assessment

Yes. The issue has detailed reporter evidence, and current-main source shows the reproducible path: deep status awaits `probeEmbeddingAvailability()`, which calls `embedBatchWithRetry(["ping"])` and can inherit the 600s local batch timeout.

## Solution Assessment

Yes, at the issue level. A CLI-level diagnostic budget plus visible progress is the narrowest maintainable fix for a status command; deeper provider warmup failures can be investigated separately without making readiness checks block for the full embedding batch window.
