## Summary

Keep open. Current main still routes `openclaw memory status --deep` through a live embedding readiness probe before rendering the status report, and a cache miss still uses the normal batch embedding timeout path that can wait 600 seconds for local providers. The reporter's correction narrows the issue from an infinite hang to a long silent diagnostic wait, but that behavior remains current and is not superseded by the adjacent doctor/status reports.

## What This Changes

The issue asks for `openclaw memory status --deep` to avoid a silent multi-minute embedding probe wait and to make the underlying main-agent embedding timeout diagnosable when normal status already shows vector and FTS ready.

## Best Possible Solution

Keep the issue open and land a focused memory-core diagnostic fix: `memory status --deep` should return a parseable embedding-probe timeout or unavailable result within a diagnostic budget, keep vector/FTS/index state visible, and reserve the longer batch timeout defaults for actual indexing or embedding work. If the reporter's local model still times out after that, collect DEBUG output as a separate provider/runtime investigation.
