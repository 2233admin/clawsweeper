## Summary

Keep open. Current main still has the source-backed long `memory status --deep` embedding probe path with no non-TTY progress fallback, and the issue is now paired with open PR #75106 using closing syntax, so it should remain open until that PR is reviewed and merged or rejected.

## What This Changes

The issue reports that `openclaw memory status --deep` can wait for the local embedding batch timeout with no progress output, then return an embeddings timeout while vector and FTS remain ready.

## Best Possible Solution

Review and land a maintainer-approved version of #75106, or close that PR with an equivalent maintainer fix, so the deep status probe has a bounded diagnostic budget and visible progress for non-TTY callers while indexing keeps the longer batch timeout. Keep the broader probe semantics/flapping work in #47235 if residual provider-readiness ambiguity remains after the CLI budget fix.

## Reproduction Assessment

Yes. The reporter supplied a corrected run showing completion only after the internal 600s embedding batch timeout, and current main shows `--deep` awaiting `probeEmbeddingAvailability()`, which calls `embedBatchWithRetry(["ping"])` under the local 600s batch timeout while non-TTY progress without fallback is a no-op.

## Solution Assessment

Yes for the CLI diagnostic bug. The linked PR direction is narrow and maintainable because it bounds only the status probe, preserves the longer indexing timeout, adds a parseable timeout state, and documents the behavior; reviewers still need to decide whether the underlying provider timeout belongs in this issue or in the broader related issue.
