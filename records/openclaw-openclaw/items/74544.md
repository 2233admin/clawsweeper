## Summary

Keep open. Current main still lets the deep memory-status path await the live embedding probe that can inherit the 10-minute local batch timeout, and the linked fix PR #75106 is open and unmerged, so this issue should remain tracked until that PR or a replacement lands.

## What This Changes

The issue asks for `openclaw memory status --deep` to stop blocking silently on long embedding-probe timeouts, report progress or a bounded failure, and preserve enough diagnostics to investigate the underlying Windows/local embeddings timeout.

## Best Possible Solution

Land a narrow status-only diagnostic budget and non-TTY progress fallback for `memory status --deep`, render probe timeout distinctly from provider unavailability, and keep the longer batch timeout for actual indexing work. Broader live-probe flapping or misleading readiness semantics should stay tracked under #47235 if still reproducible after this fix.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. The reporter supplied concrete Windows/OpenClaw version steps and a follow-up showing the command returns only after the 600s embedding batch timeout, and current main source still has the same direct deep-probe path that can wait on the local batch timeout.

## Solution Assessment

Is this the best way to solve the issue? Yes as a direction: bounding only the diagnostic status probe and adding progress output is narrower than changing indexing/provider batch budgets globally. The linked PR still needs normal review and merge before this issue can close.
