## Summary

Keep open. Current main still matches the reported failure mode: session maintenance can prune/cap/evict idle topic or thread session rows, and the only built-in preservation path is the active session key supplied by the caller.

## Best Possible Solution

Keep the issue open and fix core session maintenance. Add a shared protected-session predicate based on existing session thread/conversation resolution plus stored metadata such as `chatType`, `origin.threadId`, and `forkedFromParent`, then apply it consistently to stale pruning, entry-count capping, disk-budget store-entry eviction, load-time maintenance, and `openclaw sessions cleanup` preview/apply paths. Add regression tests for Telegram forum topics, generic `:thread:` sessions, and provider ids that contain `:topic:` but are not thread sessions.

## Risks / Open Questions

- The report's claim that `maxDiskBytes` defaults to 100MB does not match current source/docs; disk-budget eviction requires `maxDiskBytes` to be configured, but prune/cap paths still have default retention controls.
- A fix should not use a raw `:topic:` substring alone: current tests show some provider conversation ids can contain embedded `:topic:` without being thread sessions, so the existing session conversation/thread parsing helpers should drive the predicate.
