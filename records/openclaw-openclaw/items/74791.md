## Summary

Keep open. Current main does not implement these pairing env overrides, and the PR is useful source work, but the patch is not merge-ready because it accepts malformed integer prefixes and changes security-sensitive pairing TTL/cap behavior without aligned docs or maintainer approval.

## What This Changes

This PR adds module-load environment overrides for DM pairing pending request TTL and max count in `src/pairing/pairing-store.ts`, plus a new Vitest resolver test file.

## Best Possible Solution

If operator-tunable DM pairing limits are desired, land a stricter version with whole-integer validation, documented defaults and bounds in pairing/security docs, changelog coverage, and tests for both resolver behavior and store behavior under overridden TTL/max values.

## Reproduction Assessment

Yes. The parser issue is reproducible by source inspection of the PR diff: `Number.parseInt` accepts malformed integer prefixes, and the new fractional test explicitly expects that behavior.

## Solution Assessment

No, not as written. The direction may be acceptable after maintainer security review, but this implementation should reject malformed values and align documentation before it changes the pairing access-control posture.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject malformed integer environment values:** `src/pairing/pairing-store.ts:71`
  - body: The new parser uses `Number.parseInt`, so values like `OPENCLAW_PAIRING_PENDING_TTL_MINUTES=3.7` or `3abc` are accepted as `3` instead of falling back. The PR describes these knobs as integer-only and says malformed or fractional input should use the default; silently truncating a pairing TTL/max changes the security window for a misconfigured deployment.
