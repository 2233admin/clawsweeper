## Summary

Keep this issue open. Current main still silently drops or forgives malformed workspace SKILL.md metadata instead of surfacing parser/validation diagnostics, and there is no `openclaw skills validate` or `skills doctor` command. The existing CLI/UI status paths only expose successfully loaded skills plus requirement status, so the core user problem remains actionable in this repository.

## Best Possible Solution

Keep the issue open and implement a structured skill-load diagnostics path: collect per-SKILL.md validation results during workspace discovery, expose them through `skills.status` or a dedicated validation method, add a CLI validation/debug command or extend `skills check`, render warnings in the Skills UI, and update `docs.openclaw.ai/tools/creating-skills` plus `docs.openclaw.ai/cli/skills` with malformed frontmatter examples and recovery steps.

## Risks / Open Questions

- Implementing this should preserve current compatibility intentionally: current main permits a missing `name` by falling back to the directory name, even though docs mark `name` as required.
- The cleanest fix likely needs a structured diagnostics contract shared by loader, CLI, gateway protocol, and UI rather than only logging warnings, so JSON consumers and the web view can show unloaded SKILL.md records safely.
