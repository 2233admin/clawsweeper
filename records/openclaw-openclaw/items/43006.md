## Summary

Keep open. Current main still has a single resolved memory-flush plan and a one-shot gate; it does not implement progressive 80%/90%/95% staged reminders, stage-specific prompts, persisted stage state, or a forced-summary handoff.

## Best Possible Solution

Keep this issue open for maintainer design. A complete implementation likely belongs in the memory-flush/core plugin contract: add staged threshold configuration, a staged decision/result type, per-stage dedup state, stage-specific prompts, a defined forced-summary or compaction handoff, and matching config schema, docs, and focused tests.

## Risks / Open Questions

- A progressive design would add extra silent model turns and needs maintainer decisions about latency, prompt noise, duplicate durable-memory writes, and what the forced-summary stage should do.
- The current memory plugin API is too narrow for a third-party plugin to express staged flushes cleanly without an additive core/plugin contract.
