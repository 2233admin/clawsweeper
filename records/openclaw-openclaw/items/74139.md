## Summary

Keep open. The linked bug is valid on current main and this PR is aimed at the right guard, but the patch is broader than the reported `llm-task`/config-allowlist false positive because it also removes the existing fail-closed runtime `toolsAllow` + `disableTools` diagnostic.

## What This Changes

The PR adds an early `null` return in `buildEmptyExplicitToolAllowlistError` whenever `disableTools` is true and updates the colocated guard test to expect no error for that disabled-tools case.

## Best Possible Solution

Land a narrowed fix that treats intentional no-tool embedded `llm-task` runs as valid when the only explicit allowlist sources are inherited policy/config sources, while preserving the runtime `toolsAllow` conflict error and coordinating any separate `modelRun` behavior with #73031.

## Reproduction Assessment

Yes. A direct guard call with a non-empty config allowlist source, zero callable tools, `toolsEnabled: true`, and `disableTools: true` reproduces the current false-positive shape; the `llm-task` embedded runner path reaches that shape by passing `disableTools: true`.

## Solution Assessment

No, not as written. The current PR solves the reported symptom by skipping every disabled-tools guard case, but the narrower maintainable fix is to skip only inherited/config allowlist false positives and keep explicit runtime `toolsAllow` conflicts fail-closed.
