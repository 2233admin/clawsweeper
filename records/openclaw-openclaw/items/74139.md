## Summary

Current main still contains the false-positive path this PR targets: `llm-task` starts embedded runs with `disableTools: true`, the embedded runner creates no callable tools but still gathers explicit allowlist sources, and the guard currently returns an error when `disableTools` is true. The PR is a narrow, plausible fix for #74019 and is not implemented on main, so it should stay open for review/landing rather than be closed.

## Best Possible Solution

Keep this PR open and review/land the narrow guard change if CI passes. The right fix is to treat `disableTools: true` as an intentional no-tool embedded run while preserving the fail-closed error for explicit allowlists that resolve to no tools because tools are unavailable or missing.
