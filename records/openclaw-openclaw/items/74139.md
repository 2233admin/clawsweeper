## Summary

Keep open. The main-branch bug this PR targets is still present, but the branch should not merge as-is because its unconditional `disableTools` bypass also drops the current fail-closed behavior for explicit runtime `toolsAllow` conflicts.

## What This Changes

This PR changes the empty explicit tool allowlist guard to return `null` whenever `disableTools` is true and updates the colocated guard test to expect no error in that case.

## Best Possible Solution

Land a narrowed version that treats `llm-task`'s intentional no-tool embedded run as valid when only parent/config allowlists are present, while preserving fail-closed behavior for runtime `toolsAllow` conflicts and coordinating the adjacent `modelRun` case with #73031.

## Reproduction Assessment

Yes. A direct guard call with a non-empty config allowlist source, no callable tools, `toolsEnabled: true`, and `disableTools: true` reproduces the current false-positive error, and the `llm-task` embedded runner path reaches that shape.

## Solution Assessment

No, not as written. The fix direction is right for #74019, but the implementation is broader than necessary because it bypasses the guard for every disabled-tool run instead of only the config-derived allowlist case.
