## Summary

Keep open. The reported llm-task false positive is real on current main, but this PR is broader than the safe fix because it suppresses the existing runtime `toolsAllow` + `disableTools` fail-closed guard.

## What This Changes

The PR adds an early null return in `buildEmptyExplicitToolAllowlistError` whenever `disableTools` is true and changes the colocated unit test to expect no guard error for that disabled-tools case.

## Best Possible Solution

Land a narrowed fix that permits llm-task's intentional no-tool embedded run when only parent/config allowlist sources are present, while preserving the existing fail-closed behavior for runtime `toolsAllow` conflicts and coordinating the separate `modelRun` case with #73031.

## Reproduction Assessment

Yes. A direct guard call with a non-empty config allowlist source, no callable tools, `toolsEnabled: true`, and `disableTools: true` reproduces the reported false-positive error; the llm-task embedded-runner path reaches that shape in current main.

## Solution Assessment

No, not as written. The fix direction is right for #74019, but returning `null` for every disabled-tools run is broader than necessary and bypasses an existing runtime allowlist conflict check.
