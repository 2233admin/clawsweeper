## Summary

Keep open. The PR’s cleanup target still exists on current main, but the proposed type-only refactor should not merge as-is because it asserts the Discord setup resolver output as the setter input shape even though those shapes differ.

## What This Changes

The PR adds a local Discord guild/channel allowlist entry type and replaces the Discord setup apply path’s `resolved as never` cast with a cast to that explicit entry array type.

## Best Possible Solution

Type the Discord setup allowlist boundary around the real resolver output, map resolved `guildId`/`channelId` values into the config writer’s `guildKey`/`channelKey` shape, define the unresolved fallback explicitly, and add a focused Discord setup test for a resolved guild/channel entry.

## Reproduction Assessment

Yes for the review finding. Existing source and tests show the resolver returns `guildId`/`channelId`; passing that object shape into the current setter causes `entry.guildKey` to be absent and the setter to use the `*` fallback.

## Solution Assessment

No. The proposed cast is narrower-looking but not sound; the maintainable fix is a small mapper or shared typed boundary before calling `setDiscordGuildChannelAllowlist`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Map resolver output before asserting the allowlist entry type:** `extensions/discord/src/setup-core.ts:291`
  - body: `resolveDiscordGroupAllowlist` returns `resolveDiscordChannelAllowlist` results with `guildId`/`channelId`, but this assertion passes them to `setDiscordGuildChannelAllowlist` as `{ guildKey, channelKey? }`. The setter reads `entry.guildKey` and falls back to `*`, so selected Discord setup entries can still be stored under the wildcard instead of the selected guild/channel. Map the resolver output to the setter shape before applying it.
  - confidence: 0.87
