## Summary

Keep open. Current main still has the Slack channel-monitor bug: `dmHistoryLimit` is accepted and documented, but the active Slack monitor derives only `historyLimit` and keeps pending-history recording, prompt history injection, structured `InboundHistory`, and clearing behind the `isRoomish` path that excludes 1:1 DMs. The prior fix PR was closed unmerged, and related #64427 is broader cross-session Slack history hydration rather than a clean supersession of this narrower monitor-wiring issue.

## What This Changes

The issue asks OpenClaw to wire Slack `dmHistoryLimit` into 1:1 DM prompt-history assembly so non-threaded DM replies can receive prior context.

## Best Possible Solution

Land a focused Slack plugin fix that computes an effective history limit for roomish versus direct-message contexts, carries the DM limit through the monitor context, records pending 1:1 DM history when enabled, uses that effective limit for prompt body history, `InboundHistory`, and dispatch cleanup, and adds Slack monitor regression coverage plus a user-facing changelog entry. Keep cross-session Slack history hydration and cross-channel parity as separate follow-up items.

Current main accepts and documents `channels.slack.*.dmHistoryLimit`, but `monitor/provider.ts` derives only `historyLimit`, `SlackMonitorContext` cannot carry a DM limit, and `message-handler/prepare.ts` plus `dispatch.ts` gate pending-history recording, prompt history injection, structured `InboundHistory`, and cleanup on `isRoomish && ctx.historyLimit > 0`, which excludes 1:1 Slack DMs. Scope the PR to the Slack plugin monitor path.
