## Summary

Keep open. Current main still does not provide a persisted job-level cron lifecycle contract for timeout, retry/cleanup policy, or per-job start/complete/failure hooks; the existing implementation covers payload-level timeouts, one-shot deletion, failure alerts, run/session pruning, and global plugin observation only. The related PR #45465 remains open and only addresses part of the hook story, while the proposed hook strings need maintainer API and security review before implementation.

## What This Changes

Add cron job-level lifecycle configuration for maximum run time, retry/auto-cleanup policy, and start/complete/failure hooks.

## Best Possible Solution

Design one additive cron lifecycle contract that spans Gateway protocol schemas, stored job normalization, scheduler timeout semantics, cleanup policy, hook execution boundaries, CLI/UI/docs, and focused regression tests, while preserving current payload timeout, delivery, failure-alert, retention, and plugin cron_changed behavior.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is directly verifiable on current main by inspecting the cron protocol schema, stored job types, timeout policy, cleanup path, hook dispatch, CLI/docs, and related open PR context.

## Solution Assessment

No, this is not ready for an automated narrow fix. The requested shape spans persistence, protocol, scheduler semantics, hook execution security, and overlapping unmerged hook work, so the best next step is maintainer design/review before implementation.
