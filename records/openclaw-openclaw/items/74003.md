## Summary

Current main still matches the reported bug: before_tool_call plugin approvals are requested without trusted turn-source routing metadata, while the gateway already accepts those fields and fail-closes with `decision: null` when no approval route is present. The linked fix PR is closed and unmerged, and the reporter's follow-up correctly narrows the fix away from model-generated tool params.

## What This Changes

This issue asks OpenClaw to forward the originating chat/channel routing context when a `before_tool_call` hook returns `requireApproval`, so plugin approvals can be delivered back to Telegram and other chat channels.

## Best Possible Solution

Add a runtime-only route context seam for before_tool_call approvals, populate it from the trusted channel/account/target/thread values already available to the agent run, forward those fields to `plugin.approval.request`, and keep the gateway no-route behavior fail-closed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: a channel-originated `before_tool_call` approval currently calls `plugin.approval.request` without route fields, the gateway no-route path returns `decision: null`, and the agent surfaces the reported error.

## Solution Assessment

Yes, with the corrected trusted-source constraint. The narrow maintainable fix is to forward runtime hook context, not to read route metadata from model-generated tool params or plugin-mutated params.
