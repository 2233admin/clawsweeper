## Summary

Keep this open. Current main still omits trusted turn-source route context from before_tool_call plugin approval requests, while the gateway already accepts those fields and fail-closes with a null decision when no approval route is present. The only linked repair PR is closed and unmerged, and the reporter's correction is important: the safe fix should use trusted runtime context, not model/tool params.

## What This Changes

Forward trusted originating-channel route metadata for before_tool_call plugin approval requests so plugin approval prompts can be delivered to the source chat.

## Best Possible Solution

Extend HookContext and createOpenClawCodingTools to carry the same trusted turn-source route data used by exec approvals, forward it to plugin.approval.request, add regression tests for routed and unrouted plugin approvals, and keep the gateway no-route path fail-closed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: before_tool_call requireApproval reaches requestPluginToolApproval without turnSource* fields, then the gateway no-route branch returns decision:null and the agent maps that to the reported error.

## Solution Assessment

Yes, with the corrected trust boundary. The original args.params/baseParams direction is not the best fix; the narrow maintainable solution is trusted runtime route context propagated into the hook ctx and forwarded from there.
