## Summary

Current main still matches the reported bug: the before_tool_call approval helper sends plugin/tool/session fields but no turn-source route metadata, while the gateway already accepts those fields and returns an immediate null decision when no approval route exists. The related fix PR #74150 is closed and unmerged, so it does not resolve or supersede this issue.

## What This Changes

The issue requests that before_tool_call plugin approval requests forward originating channel route metadata so plugin approval prompts can be delivered back to the source chat.

## Best Possible Solution

Implement a narrow agent-side propagation fix: add trusted turn-source fields to the before_tool_call hook context, populate them from createOpenClawCodingTools runtime route options, forward them to plugin.approval.request, and add regression tests proving routed approvals include the route fields while no-route requests still fail closed.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: a channel-backed run with a before_tool_call requireApproval currently sends plugin.approval.request without route fields, and the gateway no-route branch returns decision:null, which the agent maps to the reported error.

## Solution Assessment

Yes, if solved through trusted runtime context. The original args.params/baseParams direction is not the best fix because those values are model/tool inputs; route metadata should be carried through HookContext or an equivalent runtime-only seam.
