## Summary

Current main still matches the reported bug. `requestPluginToolApproval` sends `plugin.approval.request` without the turn-source route fields, while the gateway plugin-approval handler accepts those fields and the shared approval path expires immediately with `decision: null` when no approval client, forwarder, or turn-source route exists. The issue is narrow and actionable; it should stay open and is a good focused fix-PR candidate.

## Best Possible Solution

Keep this issue open and fix the before_tool_call plugin approval caller so `plugin.approval.request` receives trusted turn-source routing metadata (`turnSourceChannel`, `turnSourceTo`, `turnSourceAccountId`, and `turnSourceThreadId`) consistently with exec approvals and the Codex app-server plugin approval bridge. Add a regression test proving same-chat route metadata is forwarded and preserve the existing no-route fail-closed behavior.
