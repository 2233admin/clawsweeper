## Summary

Keep this PR open for maintainer review. Current main still mutates cron jobs in place before cron expression validation can fail, while this PR moves `cron.update` to a staged-job flow and adds service-level regression coverage for the disabled-job edit and failed-enable paths from #74459. The overlapping #74068 is a gateway-only prevalidation approach; this PR appears to address the underlying service atomicity issue more directly, so it should not be closed as implemented or obsolete.

## What This Changes

The PR changes `src/cron/service/ops.ts` so `cron.update` stages and validates a cloned job before replacing stored state, and adds regression tests for invalid disabled cron edits and failed enables against already-invalid disabled jobs.

## Best Possible Solution

Land one canonical fix that makes cron service updates atomic: build the full post-patch job, validate any schedule that can throw before committing it to stored state, preserve legacy tolerance for already-invalid disabled jobs, and keep gateway error responses as `INVALID_REQUEST`. If this service-level patch is accepted, use it to resolve #74459 and then supersede the gateway-only alternative.
