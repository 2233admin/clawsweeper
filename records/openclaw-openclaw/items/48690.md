## Summary

Keep open. Current main still has the core behavior this PR targets: lane tasks are awaited directly with no task-level timeout, and the linked same-author bug report #48488 remains open. The branch is not obviously ready to land unchanged because its timeout fallback needs to preserve caller runtime budgets and cancellation semantics, but it is still a relevant implementation candidate rather than obsolete or implemented on main.

## Best Possible Solution

Keep this PR open, but require a rework before landing: add automatic lane recovery in `src/process/command-queue.ts` or an equivalent queue health mechanism, preserve caller-specific timeout budgets, document/handle zombie-task overlap, and keep regression coverage for lane unblock, reset generation handling, disabled/custom timeouts, and diagnostics. Once that lands, close the paired #48488 as implemented.
