## Summary

Keep open. Current main still supports the reported installer outcome: the curl installer can warn that the original shell PATH is missing the npm global bin directory, skip verification by default, and still print a successful install banner, while the existing non-root smoke verifies only after manually exporting the npm global bin path. The existing ClawSweeper comment reached the same conclusion and I did not find a current-main fix or paired open PR in the provided context.

## What This Changes

The issue asks for the curl installer on Ubuntu/Linux to leave `openclaw` available after a reported successful install instead of ending with `openclaw: command not found`.

## Best Possible Solution

Tighten the Unix installer contract so a default successful install is not reported as ready when the user's next shell command cannot find `openclaw`. Either make the CLI discoverable through a bin directory already on the original PATH, or change finalization/verification so the installer clearly reports the required PATH action instead of an unconditional success banner; add focused coverage for the original-PATH case and keep the public troubleshooting docs aligned.
