## Summary

Keep open. Current main still allows the Unix npm installer to warn that the npm global bin directory is missing from the user's original PATH, resolve the binary inside the installer process anyway, and then print the installed-successfully banner; that matches the reporter's post-install `openclaw: command not found` failure mode closely enough to remain actionable.

## What This Changes

The issue reports that the Unix curl installer prints a successful install on Ubuntu 26.04 after npm/PATH warnings, but `openclaw` is not available in the next shell.

## Best Possible Solution

Make the Unix installer's normal success path match what the user can actually run next: either ensure the CLI is discoverable from the next shell after npm prefix redirection, or make the missing-original-PATH case an action-required/non-success outcome with focused installer coverage and matching troubleshooting docs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: run the npm-method installer with the npm global bin directory absent from the parent shell PATH and `OPENCLAW_VERIFY_INSTALL` unset; current main can warn about the PATH gap and still continue to the success banner.

## Solution Assessment

Yes. A narrow installer fix in `scripts/install.sh` with targeted installer and smoke coverage is the best direction; docs-only guidance would not address the misleading default success path for new installs.
