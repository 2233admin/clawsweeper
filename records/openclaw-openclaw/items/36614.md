## Summary

Keep open. Current main still has the write-side contamination path for `session.dmScope: per-channel-peer`: routing builds isolated direct-message session keys and marks those routes with `lastRoutePolicy: "session"`, but the shared recorder still writes `deliveryContext` to whatever update target the caller supplies, and several direct-message channel paths still pass the main session key as that target. Telegram and WhatsApp already show the intended policy-based shape, and the related fix PRs for this issue are closed unmerged while merged #38418 only addressed an adjacent read-side WebChat stale-route symptom.

## Best Possible Solution

Keep this issue open and finish the write-side routing fix. The conservative implementation is to make every direct-message last-route writer choose its target through `route.lastRoutePolicy` and `resolveInboundLastRouteSessionKey`, so `lastRoutePolicy: "session"` writes delivery metadata to the scoped inbound session while `lastRoutePolicy: "main"` preserves existing main-session behavior. Add focused regression coverage for the central recorder contract and representative iMessage, Slack, Signal, LINE, Matrix, Mattermost, and Discord component paths, while preserving the existing Telegram and WhatsApp policy-based behavior.
