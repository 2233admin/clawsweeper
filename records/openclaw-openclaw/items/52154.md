## Summary

Keep open. Current main does not implement `commands.shellProfile` or an equivalent per-command shell profile source/load path. The strict commands schema and exported config type do not contain the key, exec still builds shell invocations without a profile input, and the docs only cover `env.shellEnv` startup environment import plus `tools.exec.pathPrepend`, which do not cover aliases/functions from a custom profile file. The PR is still a plausible implementation candidate, but it touches a security-sensitive command execution surface and needs maintainer review/rework rather than conservative cleanup close.

## Best Possible Solution

Keep this PR open for maintainer product and security review. The best implementation path is to add an explicit, validated config surface for command profile loading, wire it through all intended exec surfaces, document exact host/shell scope, and keep allowlist/approval behavior from being bypassed by profile-defined functions or aliases.
