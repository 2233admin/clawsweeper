## Summary

Keep open. Current main does not have the proposed `diagnostics.stuckSessionAbortMs` config surface, so the PR is not obsolete, but the patch is not safe to merge because its abort path clears only queued work, marks diagnostics idle, and can leave the active lane slot wedged.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs` config/schema/help/reload handling plus heartbeat logic and tests for clearing a stuck session lane and resetting diagnostic state after the abort threshold.

## Best Possible Solution

Rework the feature through the existing stuck-session recovery runtime: resolve the abort threshold independently from warning emission, pass an explicit active-abort signal only after that threshold, release lanes through the established abort/drain/reset path, make queued-message handling intentional, and add heartbeat integration coverage plus aligned config docs/generated schema checks.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Release the active lane before marking the session idle:** `src/logging/diagnostic.ts:678-680`
  - body: A stuck session blocks later messages because its lane has an active task. `clearCommandLane` only rejects queued entries; it leaves the active task slot in `activeTaskIds`, so with `maxConcurrent=1` the lane can remain blocked while diagnostics now reports the session as idle. Route this through the existing abort/reset recovery path before resetting state.
  - confidence: 0.91
- **[P2] Evaluate abort threshold independently:** `src/logging/diagnostic.ts:676`
  - body: The abort branch is nested under `ageMs > stuckSessionWarnMs`, so a valid configured abort threshold below the warn threshold cannot fire at the configured time. Either enforce `abort >= warn` during resolution or evaluate the abort condition independently from warning emission.
  - confidence: 0.87
