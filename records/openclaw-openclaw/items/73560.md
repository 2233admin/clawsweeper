## Summary

Keep open. Current main has conservative stuck-session recovery but not the PR's configurable force-abort threshold, and the proposed patch still has blocking correctness issues around active lane release and threshold evaluation.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs` config/schema/help/reload handling plus diagnostic heartbeat logic and tests intended to clear a stuck session lane and mark the session idle after a configurable abort threshold.

## Best Possible Solution

Implement the abort threshold as an explicit policy on the existing stuck-session recovery runtime: resolve abort timing independently from warning emission, pass an active-abort signal only after that threshold, release lanes through the abort/drain/reset path, and update generated config docs plus heartbeat/recovery integration tests.

## Reproduction Assessment

Yes. The active-lane issue is reproducible from current queue tests: one active task plus one queued task leaves `clearCommandLane` rejecting only the queued entry, while `resetCommandLane` is the helper that releases the active slot. The threshold issue is reproducible by configuring an abort threshold below the warning threshold and advancing the heartbeat before the warning threshold.

## Solution Assessment

No. The PR solves a real stuck-session problem, but direct heartbeat queue clearing bypasses the recovery runtime and does not release active lane work; the safer fix is to extend `recoverStuckDiagnosticSession` with explicit abort-threshold semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Route aborts through the lane reset path:** `src/logging/diagnostic.ts:678-680`
  - body: `clearCommandLane` only rejects queued entries and leaves the active task slot occupied. With `maxConcurrent=1`, a stuck active task can keep blocking the session lane after diagnostics marks the session idle, so later messages may remain wedged while recovery is reported as complete.
  - confidence: 0.91
- **[P2] Evaluate the abort threshold independently:** `src/logging/diagnostic.ts:676`
  - body: The abort check is nested under `ageMs > stuckSessionWarnMs`, so a valid configured abort threshold below the warning threshold cannot fire at the configured time. Either enforce the ordering during config resolution or check abort timing separately from warning emission.
  - confidence: 0.87
