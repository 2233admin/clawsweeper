## Summary

Keep this issue open. Current main still only supports `systemEvent` and `agentTurn` cron payloads, and the scheduler, Gateway protocol, CLI, agent-facing cron tool, Control UI, and docs have no direct `exec` or shell payload path. The discussion also makes this the canonical tracker: related duplicates were closed toward this issue, and PR #51276 was closed unmerged rather than shipped.

## Best Possible Solution

Keep this issue open as the canonical core cron direct-exec tracker. The best implementation path is an additive `exec` or shell cron payload across cron types, Gateway schema/validators, create/update normalization, runtime dispatch, CLI add/edit, Control UI, agent-facing cron tool guidance, public docs, and focused tests, using a controlled process runner integrated with existing run history, task records, failure alerts, timeout/cancellation, security policy, and delivery behavior.

## Risks / Open Questions

- A native cron exec runner would add a privileged host-command path and needs explicit security design for approvals, sandbox policy, shell selection, environment handling, working directory, output capture/truncation, retries, timeouts, cancellation, logging, and delivery.
- Closing this now would drop the canonical direct-exec cron tracker while duplicate reports have already been closed toward it and the only implementation PR is closed unmerged.
- Broader related requests such as #29907 and #37131 overlap with conditional pipelines or direct tool invocation, but they do not replace this narrower cron direct-exec payload request.
