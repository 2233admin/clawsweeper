## Summary

Keep this issue open. Current main still does not provide a zero-LLM cron exec payload: cron payload types and Gateway schemas only allow systemEvent or agentTurn, isolated/current/session job validation still requires agentTurn, detached execution still calls the isolated agent runner, and the CLI/UI/docs expose only the agent-mediated path. The related implementation PR #51276 is closed unmerged, while recent comments add production and Windows datapoints rather than resolving the request.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The best path is an additive cron payload kind for direct command execution that updates cron types, Gateway protocol/schema validation, job normalization/validation, runtime dispatch, run history, retry/failure accounting, delivery, CLI add/edit/run surfaces, Control UI, docs, and focused tests, with an explicit secops review of the host-command policy.

## Risks / Open Questions

- A direct cron exec runner is security-sensitive because persisted schedules could launch host commands without an interactive LLM turn; the design needs explicit approval, sandbox, host routing, cwd/env, shell selection, output truncation, cancellation, timeout, retry, and delivery policy.
- Closing this now would orphan the canonical tracker after duplicate shell/exec cron requests were consolidated here and the related implementation PR was closed unmerged.
- The existing exec tool and exec-approval docs do not implement cron payload persistence, Gateway validation, scheduler dispatch, run history, or channel delivery for zero-LLM scheduled command execution.
