## Summary

Keep this issue open. Current main at 7a23b2d945cc2d2e5c452e4641213e71de8dcc66 still has no cron direct-exec payload: the core cron payload type, Gateway validator, runtime dispatch, CLI, Control UI, agent-facing cron tool, and public docs all expose only `systemEvent` and `agentTurn`. Related PR #51276 is closed unmerged, and the discussion makes this the canonical tracker for several duplicate direct-exec cron requests.

## Best Possible Solution

Keep this issue open as the canonical core direct-exec cron tracker. The best path is an additive `exec` or shell payload across cron types, Gateway schema/validators, normalization, runtime dispatch, CLI add/edit, Control UI, agent-facing cron tool guidance, docs, and focused tests, implemented with a controlled process runner integrated with existing task/run history, failure alerts, delivery behavior, timeout/cancellation, and exec security policy.

## Risks / Open Questions

- A native cron exec runner would be a privileged host-command surface and needs explicit design for approvals, sandbox/security policy, shell selection, environment handling, working directory, output capture/truncation, timeouts, cancellation, logging, retries, and delivery.
- Closing this now would orphan the canonical direct-exec cron tracker after multiple duplicate reports were already closed toward it and the only implementation PR was closed unmerged.
- Recent cron timeout and concurrency fixes reduce some pain, but they still run through `agentTurn` and do not satisfy the zero-LLM stdout-to-channel use case.
