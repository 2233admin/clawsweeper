## Summary

Keep this issue open. Current main still has no zero-LLM cron exec payload: cron payload types, Gateway validators, job validation, runtime dispatch, CLI add/edit surfaces, Control UI, and docs all remain limited to `systemEvent` for main-session jobs and `agentTurn` for isolated/current/session jobs. The related implementation PR #51276 is closed unmerged, while duplicate shell/exec requests were consolidated here.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The best path is an additive, security-reviewed cron payload kind for direct command execution that updates cron types, Gateway protocol/schema validation, normalization, validation, runtime dispatch, run history, retry/failure accounting, delivery, CLI add/edit/run surfaces, Control UI, docs, and focused tests while preserving existing `systemEvent` and `agentTurn` behavior.

## Risks / Open Questions

- Direct cron exec is security-sensitive because persisted schedules could launch host commands without an interactive model turn; it needs explicit policy for approvals, sandbox/host routing, cwd, env, shell selection, output truncation, cancellation, timeout, retry, and delivery.
- A partial implementation could bypass existing exec approval semantics or create a second command-execution policy surface unless it is deliberately tied to the documented exec/approval model.
- Closing this now would orphan the canonical tracker after related shell/exec cron requests were consolidated here and #51276 closed unmerged.
