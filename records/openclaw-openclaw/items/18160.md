## Summary

Keep this issue open. Current main still does not implement a zero-LLM cron exec payload: the cron type union, Gateway schema, validator, runtime dispatch, CLI add/edit surface, Control UI form, and docs remain limited to `systemEvent` and `agentTurn`. The attempted implementation in #51276 is closed unmerged, while several duplicate direct-exec cron requests were already consolidated toward this thread, so this remains the canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The likely implementation should be an additive `exec` or shell cron payload spanning cron types, Gateway schemas, normalization/validation, runtime dispatch, CLI add/edit, Control UI, agent-facing cron tools if applicable, docs, and focused tests, with a clear host-command security policy and integration with existing run history, retries, failure alerts, timeout/cancellation, and delivery behavior.

## Risks / Open Questions

- A native cron exec runner is security-sensitive because it would add scheduled host command execution. It needs an explicit policy for approvals/sandboxing, shell selection, cwd/env handling, output truncation, timeouts, cancellation, run logs, retries, alerts, and channel delivery.
- Closing this issue now would orphan the canonical tracker after duplicate direct-exec cron requests were already consolidated here and the implementation PR was closed unmerged.
- Recent cron reliability and delivery improvements reduce some agentTurn pain, but they do not satisfy the zero-LLM stdout-to-channel use case requested here.
