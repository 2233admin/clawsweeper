## Summary

Keep open. Current main still does not implement a zero-LLM cron exec payload, and the latest release tag has the same limitation; the request remains a valid core cron feature with unresolved host-command security policy rather than a closeable duplicate or ClawHub-only extension.

## What This Changes

The issue requests an additive cron payload mode that runs a configured shell command directly, captures stdout/stderr, records status, and optionally delivers output without starting an LLM agent turn.

## Best Possible Solution

Keep this as the canonical core tracker and land an additive, owner-reviewed cron command payload with Gateway protocol/schema updates, scheduler dispatch, run history/accounting, delivery behavior, CLI and Control UI support, docs, focused tests, and a documented host-command security model.

## Reproduction Assessment

Yes. A high-confidence current-main check is to validate or create a cron job with `payload.kind: "exec"`; the schema has no such variant, validation rejects non-`agentTurn` isolated jobs, and detached runtime dispatch has no direct execution branch.

## Solution Assessment

Yes, the requested additive core cron payload is the right direction for deterministic scheduled shell work because external cron or shim workarounds cannot satisfy OpenClaw's cron store, run history, retry accounting, and channel delivery contract. The exact implementation should wait for maintainer and secops policy review.
