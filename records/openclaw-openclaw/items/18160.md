## Summary

Keep this open. Current main still has no zero-LLM cron exec payload, and the issue discussion now treats this as the canonical tracker after duplicate shell/exec cron requests were consolidated here and the related implementation PR was closed unmerged.

## What This Changes

Add a direct cron execution mode that runs a shell command on schedule, captures stdout/stderr, and optionally delivers the result through OpenClaw without creating an LLM agent turn.

## Best Possible Solution

Keep this as the canonical core tracker and land an additive, owner-reviewed cron payload kind for deterministic command execution with explicit host-command policy, Gateway schema support, runtime dispatch, run-log/error accounting, delivery behavior, CLI/UI/docs coverage, and focused regression tests.

## Risks / Open Questions

- A direct cron exec runner is security-sensitive because persisted schedules could run host commands without an interactive LLM turn; approval, sandbox, host routing, cwd/env, output truncation, cancellation, timeout, retry, and delivery policy need maintainer/secops design.
- The implementation spans public protocol, scheduler runtime, run history, delivery, CLI, Control UI, docs, and tests, so a narrow code-only patch could easily leave inconsistent surfaces.
- The linked ollama-exec-shim workaround may help some users in the interim, but it does not close the core OpenClaw cron observability and delivery request tracked here.
