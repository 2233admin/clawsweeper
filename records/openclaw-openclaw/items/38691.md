## Summary

Keep this PR open. Current main still hard-disables the Android Voice Tab manual mic path when `SpeechRecognizer.isRecognitionAvailable(context)` is false, and there is still no configured OEM speech-recognizer fallback. The PR targets a real remaining gap with useful Xiaomi validation, but it needs a rebase onto the current Android package path and a no-backend retry safety fix before it is mergeable.

## Best Possible Solution

Keep this PR open for revision. Rebase it onto `apps/android/app/src/main/java/ai/openclaw/app/voice/MicCaptureManager.kt`, preserve the generic configured-service fallback using the device `voice_recognition_service` setting plus `ComponentName.unflattenFromString` and `SpeechRecognizer.createSpeechRecognizer(context, component)`, and add a no-backend safety path that disables manual mic or avoids retrying when no usable recognizer exists. Add focused Android/Robolectric coverage around the recognizer creation/error seam if practical; otherwise keep the Xiaomi manual proof and explain why platform mocking is not reliable.

## Risks / Open Questions

- Merging the PR as-is can regress devices with no usable speech recognizer backend: after the hard stop is removed, `ERROR_CLIENT` and most other errors would still schedule repeated restarts instead of disabling the mic.
- The PR targets the old `ai.openclaw.android.voice` package path, while current main uses `ai.openclaw.app.voice`, so it needs a rebase before it can apply cleanly.
- There is no focused `MicCaptureManager` recognizer fallback regression test, so the no-backend safety behavior could regress without either a small test seam or clearly documented manual device proof.
