## Summary

Keep #38691 open. Current main still hard-disables the Android Voice Tab manual mic path when `SpeechRecognizer.isRecognitionAvailable(context)` is false, and there is still no generic configured OEM speech-recognizer fallback. The PR has concrete Xiaomi validation and uses an Android-supported targeted recognizer API, but it needs revision before merge because the current error path would retry `ERROR_CLIENT`, and the PR targets the old package path.

## Best Possible Solution

Keep #38691 open for revision. Rebase it onto `apps/android/app/src/main/java/ai/openclaw/app/voice/MicCaptureManager.kt`, add a generic configured OEM recognizer fallback for the manual Voice Tab mic path using `Settings.Secure.getString(..., "voice_recognition_service")`, `ComponentName.unflattenFromString`, and `SpeechRecognizer.createSpeechRecognizer(context, component)`, and preserve a clean disable/no-retry path when no usable backend exists. Compile-check the settings-key detail against SDK 36 and add focused Android/Robolectric coverage if the recognizer creation seam can be tested without brittle platform mocking.

## Risks / Open Questions

- The PR targets the old `apps/android/app/src/main/java/ai/openclaw/android/voice/MicCaptureManager.kt` path; current main uses `ai.openclaw.app.voice`.
- There is no focused `MicCaptureManager` speech-recognizer regression test in the Android test tree, so a small test seam or Robolectric coverage would reduce regression risk if feasible.
