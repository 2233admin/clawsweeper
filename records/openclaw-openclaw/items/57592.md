## Summary

Keep open. Current main still renders the TUI as text/Markdown only for assistant messages and explicitly stringifies tool-result image blocks as placeholders. I found no terminal image protocol detection, no `tui.inlineImages`-style config, and no Kitty/iTerm2/Sixel rendering path in the TUI code or public TUI docs. Related PR #49022 is an open implementation attempt, but the feature is not on main and the issue remains useful as product tracking until a correct TUI implementation lands.

## Best Possible Solution

Keep the issue open. The best product path is a focused core TUI implementation that carries assistant/tool image content through the TUI render model, detects or configures terminal image support for iTerm2/Kitty/Sixel, falls back to the current placeholder on unsupported terminals, documents the setting/behavior, and adds regression tests for image-capable and fallback terminals.
