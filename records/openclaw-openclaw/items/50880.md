## Summary

Keep open. Current main still has the reported steer-to-followup behavior in the active-run path: steer injection is gated on streaming, active non-streaming embedded handles reject queued steering, and active `steer` mode still resolves to followup. The supplied GitHub context also shows focused maintainer PR #73365 is open and unmerged, so this should wait for review or replacement of that PR rather than close early.

## What This Changes

Fix `messages.queue.mode: "steer"` so inbound Discord messages during active tool execution are delivered into the current run at the documented steering boundary instead of silently falling back to followup, with diagnostics for rejected steer attempts.

## Best Possible Solution

The desired end state is lifecycle-aware steer delivery: eligible same-session messages should reach the embedded steering queue while a run is active, rejected steer attempts should produce useful diagnostics, deliberate followup fallback should remain intact, and docs/tests should consistently describe the actual injection boundary.
