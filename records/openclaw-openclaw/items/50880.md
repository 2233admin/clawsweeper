## Summary

Keep this issue open. Current main still has a streaming-only steer injection gate, still falls active `steer` mode back to the followup queue when that gate is not met, and Discord still serializes same-session inbound jobs before the reply pipeline can attempt steering. Open PR #73365 is useful related work, but it is unmerged and does not by itself prove the reported Discord path is fixed on main.

## Best Possible Solution

Keep this issue open and route it to #73365 or a narrow replacement. The best fix should make active steer injection lifecycle-aware in core, add diagnostic proof for rejected steer attempts, and either fix or explicitly regression-test the Discord same-session inbound path so a `messages.queue.mode: "steer"` message can reach `queueEmbeddedPiMessage` during an active run.
