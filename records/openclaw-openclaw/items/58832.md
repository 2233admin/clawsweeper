## Summary

Keep open. Current main still routes top-level Slack DMs with `replyToMode="all"` through the message timestamp as a routed thread id, and the latest `v2026.4.27` snapshot has the same behavior. The existing Slack test suite even asserts the reported `:thread:<messageTs>` DM session key, so this is not implemented cleanup.

## What This Changes

The issue requests that top-level Slack direct messages with isolated DM session scope keep using the persistent direct-message session key instead of appending `:thread:<message ts>` when reply threading is enabled.

## Best Possible Solution

Slack should keep ordinary top-level direct-message roots on the base direct session key for continuity, while preserving separate Slack reply/thread delivery metadata for actual thread replies and assistant-root DM cases.

## Reproduction Assessment

Yes. A source-level reproduction is `resolveSlackRoutingContext` or `prepareSlackMessage` with `isDirectMessage: true`, `replyToMode: "all"`, a message `ts`, and no real thread reply; current tests assert the resulting session key contains `:thread:<messageTs>`.

## Solution Assessment

Yes. The narrow maintainable fix is in the Slack routing/threading boundary: only thread-scope the session key for actual thread replies, while leaving outbound reply targeting and `MessageThreadId` semantics intact where Slack needs them.
