## Summary

Keep open. Current main has prompt-cache retention, cache-ttl pruning, and heartbeat keep-warm behavior, but it does not implement the proposed dedicated Cache TTL Warmer or a `cacheWarmer` config/schema/docs/runtime surface. The issue is still a maintainer design decision because the proposed background model-call path would need explicit product, cost, provider, and state-boundary decisions.

## Best Possible Solution

Keep this issue open for maintainer design. If approved, implement a narrow opt-in cache-maintenance path or explicit plugin API seam that can reuse the same request prefix as a real user turn through the cache-control boundary, append a bounded minimal marker, consume/discard the response, avoid tool execution and all transcript/channel/memory writes, document `cacheRetention` and cost behavior, and include focused scheduler/request-shape tests plus an explicit live Anthropic validation plan.

## Risks / Open Questions

- A silent background provider-call loop could create unexpected Anthropic spend, rate-limit pressure, and confusing failure modes unless defaults, opt-in behavior, telemetry, and failure handling are explicit.
- Reusing heartbeat directly would likely violate the requested boundaries because current heartbeat can read HEARTBEAT.md, process task blocks, strip or deliver `HEARTBEAT_OK`, send typing/outbound payloads, update heartbeat task state, and consume system events.
- The proposed minimal Anthropic marker request needs live validation for accepted message shape, `max_tokens` behavior, cache TTL refresh semantics across markers, and behavior after compaction/session reset.
- The current plugin API does not clearly expose a safe way for an external plugin to build and send the exact next-turn provider request while avoiding transcript/channel state, so a core implementation or new plugin API seam needs maintainer design first.
