## Summary

Keep open. Current main supports workspace skills and has recent prompt-budget and grouped-directory loader improvements, but the gateway and CLI session refresh path still depends on process-local version state rather than a durable inventory fingerprint, so a persisted version-0 snapshot can remain stale across restart-time skill changes.

## What This Changes

The issue reports that valid workspace skills under `~/.openclaw/workspace/skills/` are only partially reflected in the agent `<available_skills>` context.

## Best Possible Solution

Keep this open until #71497 or an equivalent fix lands and proves gateway, CLI, and cron skill snapshots rebuild after restart when the effective workspace skill inventory changes. The durable end state should compare an inventory fingerprint or otherwise provide restart-safe invalidation while preserving source precedence, containment checks, eligibility gates, and prompt-size limits.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an existing persisted `skillsSnapshot` with `version: 0`, workspace skills changed while OpenClaw is stopped, then a restart where `getSkillsSnapshotVersion()` returns 0 and the gateway/CLI refresh check treats the old snapshot as current.

## Solution Assessment

No. Current main's version-only refresh is not durable for restart-time inventory drift; the safer fix is durable inventory-based invalidation or an equivalent restart-safe snapshot version contract, not unboundedly listing every skill in the prompt.
