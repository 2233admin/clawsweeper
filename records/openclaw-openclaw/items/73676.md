## Summary

Keep open. Current main still has unguarded `process.cwd()` calls in the CLI startup, dotenv, main-module detection, and required-home fallback paths, and the issue is already paired with open implementation PRs that use closing syntax, so it should remain open until a maintainer selects and lands a fix.

## What This Changes

The issue requests that CLI startup handle a deleted current working directory instead of crashing with Node’s `ENOENT: uv_cwd` error.

## Best Possible Solution

Add one shared safe-cwd startup helper or seam that covers main-module detection, CLI/runtime dotenv loading, and required-home fallbacks while preserving cwd-local `.env` precedence whenever cwd is available.

## Reproduction Assessment

Yes. The reported create/cd/remove/run path maps directly to Node throwing from `process.cwd()`, and current main still calls `process.cwd()` during startup before command-specific handling can recover.

## Solution Assessment

Yes, keeping this open for maintainer review of the linked implementation PRs is the right outcome. The maintainable fix should cover all current startup raw-cwd surfaces together, rather than only guarding the first observed CLI dotenv call.
