## Summary

Keep open. Current main still has unguarded `process.cwd()` calls on the entrypoint, CLI dotenv, infra dotenv, and home-dir fallback paths, and the issue is already paired with open implementation PRs, including one with closing syntax, so it should not be closed or re-queued before maintainer review.

## What This Changes

The issue requests CLI startup handling for a deleted current working directory so `openclaw tui` reports a clear error or falls back instead of crashing with `ENOENT: uv_cwd`.

## Best Possible Solution

Land one shared safe-cwd startup path that covers main-module detection, CLI and infra dotenv loading, and required home-dir fallbacks, with targeted tests that mock `process.cwd()` throwing `ENOENT`/`uv_cwd` while preserving normal cwd-local `.env` precedence.

## Reproduction Assessment

Yes. The reported create/cd/remove/run path maps directly to Node throwing from `process.cwd()`, and current main still calls `process.cwd()` during top-level startup and dotenv probing before command-specific recovery can run.

## Solution Assessment

Unclear which open PR is the final best patch, but keeping the issue open is the right triage outcome. A narrow one-call-site fix would leave other startup cwd callers exposed; the maintainable fix is a shared helper or startup seam covering all current raw-cwd surfaces together.
