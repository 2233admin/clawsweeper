## Summary

Keep this issue open. Current main still has unguarded `process.cwd()` calls in the early CLI startup path, including before the full CLI error-handling path is installed, so the deleted-cwd `uv_cwd` report is still actionable. The provided related items cover other CLI crash classes, not this deleted-current-directory failure.

## Best Possible Solution

Add a small safe-cwd startup helper used by the entrypoint, `isMainModule` callers, and CLI dotenv probing. On `ENOENT`/`uv_cwd`, either fall back to a stable directory such as the user's home/state dir or print a clear early diagnostic asking the user to `cd` to an existing directory. Preserve documented `.env` precedence when cwd exists, and add focused regression coverage by injecting/mocking `process.cwd()` failures in entry/run-main tests.

## Evidence

- **Current main still calls process.cwd during entrypoint main-module detection:** `isMainModule` defaults `cwd` to `process.cwd()`, and `src/entry.ts` calls it without passing a cwd during top-level entrypoint detection. If the process cwd has been deleted, this can throw before `runMainOrRootHelp` installs its later startup catch.
  - file: [src/infra/is-main.ts:32](https://github.com/openclaw/openclaw/blob/bdfb408ce6eeb724a638c106104dbbcae50b612c/src/infra/is-main.ts#L32)
