## Summary

Keep this issue open. Current main still has unguarded `process.cwd()` calls in the CLI startup path, including the entrypoint main-module check and CLI dotenv probing/loading. The existing ClawSweeper comment is consistent with the source evidence, and the related items describe different CLI crash classes rather than a canonical fix for deleted-current-directory `uv_cwd` handling.

## Best Possible Solution

Keep this open and fix it in shared CLI startup code. Add a small safe cwd helper used by entrypoint main-module detection, `isCliMainModule` callers, and CLI dotenv probing/loading; on `ENOENT`/`uv_cwd`, either fall back to a stable trusted directory or print a clear early diagnostic asking the user to `cd` to an existing directory. Add focused regression tests by injecting `process.cwd()` failures around `isMainModule`, run-main dotenv probing, and `loadCliDotEnv`.
