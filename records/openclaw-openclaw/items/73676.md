## Summary

Keep this issue open. Current `main` still has unguarded startup-path `process.cwd()` calls that can throw `ENOENT: uv_cwd`, and the issue is paired with open PR #74994, which uses GitHub closing syntax and targets the same four startup surfaces.

## What This Changes

Handle a deleted current working directory during CLI startup so commands like `openclaw tui` do not crash with `ENOENT: uv_cwd` and instead recover or report a clear error.

## Best Possible Solution

Land PR #74994 or an equivalent shared safe-cwd startup fix that covers main-module detection, CLI and infra dotenv loading, and required-home fallbacks while preserving cwd-local `.env` precedence whenever cwd is valid.

## Reproduction Assessment

Yes. The reported create/cd/delete/run path maps directly to Node throwing from `process.cwd()`, and current `main` still calls `process.cwd()` in startup paths before command-specific recovery can run.

## Solution Assessment

Yes for keeping the issue open until the linked closing PR merges. A one-call-site guard would leave other startup callers exposed, so the maintainable fix is a shared helper or startup seam covering all listed raw-cwd surfaces together.
