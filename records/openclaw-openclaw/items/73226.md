## Summary

Keep this open. Current main still contains the WebChat exclusion reported here, and the latest release contains the same behavior; however, nearby tests and routing guards intentionally treat WebChat as an internal/admin surface in several paths, so the safe outcome is maintainer product/security review rather than an automatic close or blind one-line change.

## What This Changes

The issue reports that `session.dmScope: per-channel-peer` is ignored for WebChat because the runtime policy session-key resolver excludes the `webchat` channel and falls back to `agent:main:main`.

## Best Possible Solution

Maintainers should explicitly decide whether WebChat direct turns participate in secure DM mode. If yes, add a scoped WebChat runtime-policy/session-key path with regression coverage that preserves established external delivery routes; if no, document WebChat as an internal/admin exception in the WebChat, session, and security docs so `dmScope` expectations are clear.

## Reproduction Assessment

Yes. A direct WebChat context on `agent:main:main` with a sender id reaches `resolveRuntimePolicySessionKey`; `resolvePolicyChannel` returns `undefined` for `webchat`, and the current unit test asserts the result stays `agent:main:main`.

## Solution Assessment

Unclear. Removing `channel !== "webchat"` is the obvious local change, but current tests intentionally preserve WebChat main-session and external-route behavior, so the best fix depends on the intended WebChat trust boundary and should include targeted tests either way.
