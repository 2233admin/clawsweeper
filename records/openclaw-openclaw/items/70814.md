## Summary

Keep this PR open. Current main already supports numeric Telegram chat IDs when they are passed as `delivery.to` and rejects target IDs misfiled as `delivery.channel` during normal gateway add/update, but it does not implement this PR's runtime compatibility path for persisted/direct jobs with a numeric value in `delivery.channel`. The PR is plausible but needs maintainer review because the new inference path changes recipient semantics and has an unresolved allowFrom/target-confusion concern.

## What This Changes

The PR changes cron delivery-target resolution to treat a non-channel `delivery.channel` value as `delivery.to` when exactly one announce channel is configured, adds resolver regressions for numeric targets, logs inference failures, and includes unrelated macOS SwiftFormat hunks.

## Best Possible Solution

Keep the PR open and narrow it to the cron-delivery behavior. The desired end state is either a doctor/migration repair for legacy misplaced `delivery.channel` values or runtime recovery that preserves the intended allowFrom and implicit-delivery semantics, with tests for single-channel recovery, multi-channel ambiguity, and allowFrom behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve allowFrom checks for inferred targets:** `src/cron/isolated-agent/delivery-target.ts:107`
  - body: The new recovery path promotes a raw non-channel `delivery.channel` value into `explicitTo`. That makes `resolveSessionDeliveryTarget()` choose explicit mode, while the allowFrom reroute below only runs for implicit delivery. In single-channel configs with allowFrom, a misplaced target can now be delivered outside the allowlist instead of being rerouted or blocked, so the patch should keep inferred targets under the intended policy or add coverage proving this is acceptable.
