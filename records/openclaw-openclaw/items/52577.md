## Summary

Keep open. Current main has stronger Discord gateway readiness, reconnect status, delivered-event dedupe, and queued inbound processing, but it still only processes MESSAGE_CREATE events that Carbon delivers. I found no READY/reconnect catch-up pass that fetches recent Discord messages and replays missed messages from the startup/reconnect gap.

## Best Possible Solution

Implement this in the Discord plugin as a READY/reconnect catch-up pass. Track per-account/channel last-seen message cursors, fetch a small bounded recent-message window through Discord REST after startup READY and runtime reconnect READY, dedupe against the same replay key shape used by live MESSAGE_CREATE events, and feed eligible messages through the existing preflight/createDiscordMessageHandler path. Add regression coverage for startup gap, runtime reconnect gap, live/replayed duplicate delivery, and authorization-preserving replay.
