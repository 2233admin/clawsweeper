## Summary

Keep open. Current main still serializes the whole session store, keeps both reported fields on `SessionEntry`, and has an open PR with closing syntax for the same session-serialization problem, so this should not be closed before that PR is reviewed or replaced.

## What This Changes

The issue requests reducing `sessions.json` bloat from persisted `skillsSnapshot` and `systemPromptReport` data across agent runs.

## Best Possible Solution

Review and land #48425 or an equivalent narrow replacement that centralizes session-store disk serialization, keeps disk-budget measurement aligned with the write path, excludes transient `skillsSnapshot` data from disk, and deliberately defines the minimal persisted shape needed for `systemPromptReport` warning dedupe.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: populate a session with `skillsSnapshot` or `systemPromptReport`, then save it through the current session-store path; current main serializes the entire entry into `sessions.json`.

## Solution Assessment

Unclear. A shared serialization filter is the narrow maintainable boundary for `skillsSnapshot`, but the best treatment for `systemPromptReport` depends on whether maintainers want the full report persisted or only compact dedupe metadata.
