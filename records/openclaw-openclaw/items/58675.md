## Summary

Keep open. Current main still lacks Copilot `/models` discovery, so the PR is useful, but the latest branch introduces a security-sensitive SSRF policy gap for the new discovery fetch and still has incomplete/failing public checks.

## What This Changes

The PR adds a GitHub Copilot provider-owned `/models` discovery helper, wires discovered models and IDE headers into catalog resolution, deep-merges Copilot implicit headers with configured headers, and updates discovery/implicit-provider tests.

## Best Possible Solution

Keep the provider-owned discovery design, but change the discovery fetch policy to preserve the model-provider private-network contract: validate/allowlist the configured hostname without skipping private-network checks, merge in `models.providers.github-copilot.request.allowPrivateNetwork` only when explicitly enabled, and add a regression test for private base URLs before rerunning the targeted Copilot/provider discovery checks and CI.

## Reproduction Assessment

Yes for the review finding: configure the PR branch with `models.providers.github-copilot.baseUrl` pointing to a private or loopback HTTP endpoint and no `request.allowPrivateNetwork`; the code path builds an allowed-hostname SSRF policy and attempts discovery with the Copilot bearer token. Not applicable as a current-main bug reproduction because this is a feature PR; the main-branch gap is source-verifiable from the empty Copilot catalog.

## Solution Assessment

No, not as currently proposed. The provider-owned, additive `/models` discovery approach is the right shape, but the fetch policy should honor the existing private-network opt-in before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Honor private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:91`
  - body: The discovery fetch now targets `models.providers.github-copilot.baseUrl`, but this policy is built with `allowedHostnames`; current SSRF code skips private-network checks for allowed hostnames. A configured `http://127.0.0.1` or metadata-style base URL can therefore receive the Copilot bearer token without `request.allowPrivateNetwork`. Build the policy from a hostname allowlist plus the provider request opt-in instead.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The raw-fetch concern was addressed, but the replacement SSRF policy still bypasses private-network opt-in for configured Copilot discovery base URLs.

Concerns:

- **[high] Discovery can bypass provider private-network opt-in:** `extensions/github-copilot/discovery.ts:91`
  - body: `ssrfPolicyFromHttpBaseUrlAllowedHostname(baseUrl)` marks the configured target hostname as allowed, and the current SSRF implementation skips private-network checks for allowed hostnames. Because the PR sends `Authorization: Bearer <copilotToken>` to that discovery URL, private or loopback configured base URLs should require `models.providers.github-copilot.request.allowPrivateNetwork` before the request is allowed.
