## Summary

Keep open. Current main does not implement Copilot chat model discovery, and this PR is still useful, but the latest head still has a blocking security issue: the new authenticated discovery fetch can bypass the existing private-network opt-in for configured Copilot base URLs.

## What This Changes

This PR adds GitHub Copilot `/models` discovery, merges discovered models and IDE headers into provider catalog resolution, deep-merges Copilot headers during implicit provider merging, updates raw-fetch lint allowlisting, and adds discovery/contract tests.

## Best Possible Solution

Keep the provider-owned additive discovery design, but build the discovery fetch policy from a hostname allowlist that does not skip private-network checks, merge in `models.providers.github-copilot.request.allowPrivateNetwork` only when explicitly enabled, and add a regression test for configured private/LAN base URLs without opt-in.

## Reproduction Assessment

Yes. On PR head, configure `models.providers.github-copilot.baseUrl` to a private or metadata-style HTTP endpoint without `models.providers.github-copilot.request.allowPrivateNetwork`; source inspection shows catalog passes that URL to `discoverCopilotModels`, which sends the Copilot token with an `allowedHostnames` policy that skips private-network checks.

## Solution Assessment

No. The discovery feature is the right general shape, but the current policy choice is not the safest maintainable implementation because it bypasses the existing provider request private-network contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Honor private-network opt-in for discovery fetches:** `extensions/github-copilot/discovery.ts:91`
  - body: `discoverCopilotModels` can receive `models.providers.github-copilot.baseUrl`, but this policy uses `allowedHostnames`. Current SSRF handling skips private/internal IP checks for allowed hostnames, so a configured LAN, metadata, or explicitly disallowed loopback URL can receive the Copilot bearer token without `request.allowPrivateNetwork`; build a hostname allowlist policy and merge the provider request opt-in instead.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The diff still introduces a security-sensitive authenticated Copilot discovery fetch that can bypass the provider private-network opt-in.

Concerns:

- **[high] Discovery can bypass private-network opt-in:** `extensions/github-copilot/discovery.ts:91`
  - body: The discovery request sends `Authorization: Bearer <copilotToken>` to a configured base URL while using `ssrfPolicyFromHttpBaseUrlAllowedHostname(baseUrl)`. In current SSRF code, `allowedHostnames` skips private/internal address checks, so private targets should require the provider request opt-in before receiving credentials.
  - confidence: 0.88
