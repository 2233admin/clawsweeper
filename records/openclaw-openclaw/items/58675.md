## Summary

Keep open. The feature is still useful and current main does not implement Copilot chat model auto-discovery, but the refreshed PR head has a blocking security issue: the new discovery path sends a Copilot bearer token and configured headers through raw `fetch()` to a user-configured `baseUrl`, bypassing OpenClaw's SSRF-guarded provider transport.

## What This Changes

The PR adds GitHub Copilot `/models` discovery, wires discovered models and IDE headers into the provider catalog, deep-merges Copilot implicit headers with configured headers, and extends discovery, implicit-provider, and contract tests.

## Best Possible Solution

Keep the provider-owned discovery feature, but route the `/models` request through `fetchWithSsrFGuard` or an equivalent plugin SDK helper, thread provider request policy including `allowPrivateNetwork`/proxy/TLS where appropriate, remove the new raw-fetch allowlist entry, and add regression coverage for blocked private URLs plus explicit private-network opt-in.

## Reproduction Assessment

Yes for the blocking review finding: configure `models.providers.github-copilot.baseUrl` to a private or loopback URL with a token available, and the PR path calls raw `fetch()` on `${baseUrl}/models` with Copilot and configured headers. The feature gap itself is source-verifiable rather than a bug reproduction, because current main has no chat model discovery helper.

## Solution Assessment

No, the current PR shape is not merge-ready because the new network call bypasses the established SSRF-guarded provider transport. After that is fixed, the overall plugin-local, additive discovery approach is the narrow maintainable direction.
