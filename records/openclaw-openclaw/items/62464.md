## Summary

Keep open. Current main treats timeout-shaped MiniMax/provider failures as model-fallback signals, but it still attempts each fallback candidate once and throws `FallbackSummaryError` after the chain is exhausted; the only same-model idle-timeout retry is explicitly disabled when a fallback chain is configured.

## What This Changes

The issue requests retry/backoff behavior for MiniMax request timeouts so transient failures do not exhaust the MiniMax fallback chain and fail the whole user request.

## Best Possible Solution

Add a generic, bounded timeout retry path for model runs before a timeout-shaped candidate is treated as failed, ideally reusing the existing idle-timeout retry machinery and avoiding MiniMax-specific model lists or hard-coded provider policy.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction path: configure a fallback chain, make each candidate return a timeout-shaped `FailoverError`, and current main records each candidate failure once before throwing `FallbackSummaryError`. A live MiniMax replay was not needed for this triage because the reported logs match the deterministic fallback loop and observer code.

## Solution Assessment

No. Current main’s fallback-on-timeout behavior is useful but not the best final behavior for this report because it does not retry transient timeouts before exhausting the chain; a narrow generic retry for safe timeout cases is preferable to adding more MiniMax fallback models.

## Repair Work Prompt

Fix the timeout retry gap reported in https://github.com/openclaw/openclaw/issues/62464. Current main classifies timeout-shaped provider failures as model-fallback signals, but when a fallback chain is configured each candidate is attempted once and same-model idle-timeout retry is disabled by the `!fallbackConfigured` gate in `src/agents/pi-embedded-runner/run.ts`. Implement a generic, bounded retry for safe timeout/idle-timeout cases before a candidate is marked failed or before the fallback chain is exhausted. Do not hard-code MiniMax provider ids, do not add more default fallback models, and do not broaden retries for context-overflow, explicit external aborts, auth/billing failures, or cases that would replay unsafe tool side effects beyond the existing retry semantics. Prefer reusing the existing `MAX_SAME_MODEL_IDLE_TIMEOUT_RETRIES` / `handleAssistantFailover` machinery if it can be made to work per fallback candidate. Add focused tests that demonstrate a configured fallback chain retries a timed-out candidate before falling back or failing, while preserving current terminal `FallbackSummaryError` behavior after the bounded retries are exhausted. Update `docs/concepts/model-failover.md` or `docs/concepts/retry.md` only if the user-visible retry policy changes. Changelog only if the runtime behavior change is user-facing; if added, credit the issue reporter `@jaydenlichong29-create` and the PR author per repo changelog rules.
