## Summary

Keep this PR open. Current main still does not implement the requested 20s cap for broad `$HOME` or `/` `find` scans, and the PR remains a focused implementation candidate rather than cleanup noise. It needs review changes before merge because the guard touches exec timeout behavior and still appears incomplete across node/gateway approval routes and some parser cases called out in review.

## Best Possible Solution

Keep this PR open and request changes before merge. The best path is to rebase onto current main, factor a shared broad-`find` timeout classifier used by local exec, gateway approval follow-up, and node execution planning, parse all `find` start roots, resolve relative roots against command `workdir`, support `~`, `$HOME`, and `${HOME}`, preserve explicit timeout and real traversal-limiting escape hatches only when they apply to the actual `find`, and add focused regression tests for those routes and parser cases.

## Risks / Open Questions

- Security review: this PR is in an exec command parsing and timeout path. The patch does not introduce new third-party code execution, CI permissions, dependency resolution, publishing, or secret-handling changes, but parser mistakes could either leave broad scans uncapped or prematurely cap legitimate workspace searches.
- The PR patch appears to apply the guard only in the main local foreground execution flow. Current main has separate gateway approval follow-up and node `system.run` timeout paths that would need the same classifier to avoid inconsistent behavior.
- The PR still needs parser hardening for multiple `find` roots and `${HOME}` forms, plus regression coverage for local, gateway approval, node, relative workdir, and prompt guidance behavior.
