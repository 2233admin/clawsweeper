## Summary

Keep this issue open. The report is new, concrete, and targets `sessions_send`/multi-agent routing on the latest shipped release, while current `main` still routes the tool through one-shot Gateway RPC calls plus `agent.wait`/history reads that can return the exact timeout/error shapes described. The related closed reports cover adjacent Gateway handshake/RPC fixes, but they do not prove that this multi-agent `sessions_send` failure mode is fixed or duplicated.

## Best Possible Solution

Keep the issue open and investigate the `sessions_send` path end to end on Windows with multiple agents: session key/label resolution, `agent` RPC start, active run registration, lifecycle/dedupe waiter notification, and reply history readback. The likely fix should add stale/late completion diagnostics or recovery where the missed signal occurs, plus a focused regression test around repeated `sessions_send` waits or Gateway close handling.
