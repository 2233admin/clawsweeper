## Summary

Keep this open. Current main still routes `sessions_send` through a one-shot Gateway `agent` RPC followed by `agent.wait`, and those paths can still return the reported `status: "timeout"` or surface `gateway closed (1000 normal closure)` errors. The related closed Gateway issues cover adjacent CLI/browser/cron handshake and teardown failures, not a proven fix for this newer Windows multi-agent session routing/wait failure.

## What This Changes

The issue reports intermittent `sessions_send` timeouts and `gateway closed (1000 normal closure)` errors in multi-agent Windows deployments even while Gateway health checks and direct channel messages continue to work.

## Best Possible Solution

Investigate the `sessions_send` flow end to end on a Windows multi-agent setup, then fix the narrow missed-signal point with targeted diagnostics and regression coverage for repeated waits, recoverable Gateway closes, session-state refresh, or reply readback as the repro indicates.

## Reproduction Assessment

No: we do not have a high-confidence deterministic reproduction of the full Windows multi-agent failure. Source inspection shows current-main paths that can still emit the reported timeout and Gateway close shapes, which supports keeping the issue open but does not prove the root cause.

## Solution Assessment

Unclear: there is no proposed patch, and the current adjacent Gateway readiness and orphan-recovery work does not clearly solve this `sessions_send` wait/routing mode. A safe fix should be driven by a focused repro or new diagnostics at the session resolution, run registration, lifecycle/dedupe, transport-close, or reply-readback boundary.
