## Summary

Keep this open. Current main still has only adjacent version metadata and future-config downgrade guards; it does not provide the requested migration source/target version preflight, and the machine-migration docs still allow installing OpenClaw, copying state, and running doctor without requiring same-version alignment first. The later comment linking #74297 and #74298 is useful context because it narrows this to a shared cutover preflight bug rather than a broad migration feature request.

## What This Changes

Request to prevent silent state incompatibility during blue/green machine migrations by requiring version alignment in docs and adding a doctor-side migration version compatibility check.

## Best Possible Solution

Add a conservative, read-only migration-version preflight in doctor and the migration guide: require same-version alignment before copying state, report source/current/target version skew using existing version parsing plus `meta.lastTouchedVersion` or backup manifest metadata where available, and leave automatic schema rewrites to owner-specific doctor/fix paths.

## Reproduction Assessment

Yes. The high-confidence reproduction for the product gap is to inspect current main: the documented machine migration path has no version alignment step, `doctor` exposes no migration source/target option, and searches for likely preflight symbols return no implementation.

## Solution Assessment

Yes for the first fix. A read-only doctor warning/preflight plus required migration docs is the narrowest maintainable path; startup-time migrations or broad automatic schema rewrites would be higher risk and should not be part of the initial repair.

The observable bug is that blue/green machine migration can import state from a different OpenClaw version without any doctor-side preflight or required docs alignment. Add docs requiring source and target OpenClaw versions to be aligned before copying `~/.openclaw` or workspace state.
