## Summary

Keep open. Current main still lacks the requested source/target migration version preflight: the migration guide does not require version alignment before copying state, `openclaw doctor` has no migration source/target option, and the existing future-version guard only protects selected older-binary destructive actions. The related discussion correctly ties this to #74297 and #74298 while identifying existing version stamps and backup manifest metadata as ingredients, not a shipped fix.

## What This Changes

The issue asks OpenClaw to detect and document source/target version mismatches during machine or blue/green state migration.

## Best Possible Solution

Add a conservative, read-only migration-version preflight to `openclaw doctor` and the migration guide: require same-version alignment before copying state, warn or fail when imported state was written by a newer source than the running target binary, and use existing version parsing plus `meta.lastTouchedVersion` or backup manifest/runtime-version metadata where available. Leave automatic schema migrations and startup-time state rewrites out of the first fix.

## Reproduction Assessment

Yes. The product gap is reproducible by inspecting current main: the migration steps do not require version alignment, the doctor command has no migration source/target option, and the existing future-version guard is limited to newer-config destructive-action protection.

## Solution Assessment

Yes. A doctor-side read-only preflight plus explicit docs is the narrowest maintainable fix; it addresses the cutover boundary without adding broad automatic schema rewrites or startup/load-time migrations.
