## Summary

Keep this open. Current main has adjacent version-drift safeguards, but they only warn/block when an older binary touches config written by a newer binary; they do not provide the requested source/target migration preflight, and the machine-migration docs still omit required version alignment before copying state.

## What This Changes

Add migration guidance and doctor-side checks so BLUE/GREEN state transfers detect source/target OpenClaw version mismatch before silently accepting incompatible state.

## Best Possible Solution

Add a read-only migration version preflight to doctor and update the migration guide to require same-version alignment before copying state. The first fix should warn or block on clear downgrade/unsupported skew using existing version parsing plus available metadata, without adding automatic startup migrations or rewriting imported state.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: inspect the current machine-migration docs and doctor CLI wiring, then search for source/target migration preflight terms; current main has no such option or documented alignment step.

## Solution Assessment

Yes, a conservative doctor preflight plus docs is the narrowest maintainable fix. Automatic schema rewrites during startup or broad hard-fail behavior would be riskier and should stay out of the first repair.
