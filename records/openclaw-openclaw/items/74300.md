## Summary

Keep this issue open. Current main has adjacent pieces, including config last-touched version stamping, a future-config guard for older binaries, cron/auth-profile doctor migrations, and backup manifest runtime-version metadata, but it does not implement the requested source/target migration version preflight. The machine-migration docs still tell users to install OpenClaw on the new machine, copy state, then run doctor, without requiring version alignment before transfer, and `openclaw doctor` has no `--migration-target-version` or equivalent compatibility check.

## What This Changes

Require and check OpenClaw source/target version alignment before machine or VPS state migrations, including migration docs and a doctor preflight path.

## Best Possible Solution

Keep this open and add a conservative migration-version preflight: document same-version alignment as required before copying `~/.openclaw`/workspace state between machines, and add a read-only `openclaw doctor` option that reports source/current/target version mismatch using existing version parsing plus `meta.lastTouchedVersion` or backup manifest metadata when available. Do not add automatic schema migrations or startup-time state rewrites as part of the first fix.
