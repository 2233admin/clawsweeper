## Summary

Keep this open. Current main still lacks the requested cross-environment migration version preflight: the machine-migration guide copies state before any same-version check, `openclaw doctor` exposes no source/target version option, and the existing future-config guard only blocks selected older-binary actions after config metadata is already present.

## What This Changes

Requests migration documentation and doctor preflight behavior that detect source/target OpenClaw version skew before accepting copied state.

## Best Possible Solution

Add a conservative doctor-side migration preflight and update the migration guide so users align versions before copying state, while reusing existing version parsing and `meta.lastTouchedVersion` or backup manifest metadata where available. Leave automatic state rewrites or startup-time migrations out of the initial fix.

## Reproduction Assessment

Yes for the missing guard and docs: current main can be inspected to show the migration guide has no same-version step and doctor has no migration-version option. The exact production auth/cron cascade is not live-reproduced here, but the requested preflight gap is directly observable.

## Solution Assessment

Yes, a narrow read-only doctor preflight plus required version-alignment docs is the best first fix. It addresses the cutover boundary without introducing risky automatic schema migrations or startup-time rewrites.
