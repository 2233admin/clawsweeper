## Summary

Keep this open. Current main still has version metadata and a future-config downgrade guard, but the machine-to-machine migration path does not require source/target version alignment and `openclaw doctor` has no migration source/target version preflight; the discussion also preserves this as a distinct root cause from the adjacent auth and token-transfer reports.

## What This Changes

The issue requests a migration preflight and documentation update so moving OpenClaw state between machines requires compatible source and target versions before copied state is accepted.

## Best Possible Solution

Add a read-only migration-version preflight to doctor and the machine-migration guide: require same-version alignment before copying state, warn or fail when imported state was last written by a newer OpenClaw than the running target binary, and leave broad automatic schema rewrites out of the first fix.

## Reproduction Assessment

Yes for the actionable gap: current docs and CLI/type wiring can be inspected directly and show no version-alignment step or doctor migration-version option. The production auth/plugin cascade is incident-specific, but the missing preflight is reproducible from source and docs.

## Solution Assessment

Yes, the best first fix is a conservative doctor warning/error plus docs update using existing version comparison helpers and available metadata. A broad startup migration or automatic state rewrite would be riskier and outside the narrow bug boundary.
