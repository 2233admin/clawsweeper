## Summary

Keep open. Current main still lacks the sender-aware auto-reply exec deny default, and the PR is useful but not yet safe to merge because its non-owner branch can still fail open with partial agent exec defaults and overlaps with #64719's broader dangerous-tool policy.

## What This Changes

The PR adds a `senderIsOwner` input to auto-reply exec override resolution, passes the command owner flag at the directive call site, and expands unit coverage for non-owner exec default behavior.

## Best Possible Solution

Revise the PR so non-owner auto-reply ignores user-controlled inline/session exec fields and either merges `security=deny` plus `ask=always` into partial operator defaults or requires a fully explicit operator policy before granting exec, then coordinate that boundary with #64719 before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Require explicit policy before allowing non-owner exec defaults:** `src/auto-reply/reply/get-reply-exec-overrides.ts:20`
  - body: The non-owner branch returns any non-null `agentExecDefaults` object as-is. If an agent config only sets `host`, `node`, or another exec field while leaving `security`/`ask` unset, `createOpenClawCodingTools` still passes unset values through and `createExecTool` falls back to host defaults of `security=full` and `ask=off`. That leaves unauthenticated auto-reply senders with permissive exec despite this hardening.
  - confidence: 0.9
