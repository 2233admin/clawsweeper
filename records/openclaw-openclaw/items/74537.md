## Summary

Close as implemented. Current main and release v2026.4.27 already stop bare `/reset` and `/new` after reset hooks acknowledge the command, so the reset lifecycle can complete without falling through into an empty provider/model request. The lossless-claw no-op context matches the same hook-then-empty-tail path fixed by the shared auto-reply reset handler.

## What This Changes

The issue asks OpenClaw to avoid surfacing an empty-message LLM/provider error when `/reset` or `/new` is issued on an already fresh Telegram session with lossless-claw active.

## Best Possible Solution

Keep the shared reset-command behavior shipped in v2026.4.27: reset hooks still run, bare commands acknowledge locally without model execution, and tailed reset commands continue into the agent path. If a fresh report reproduces on v2026.4.27 or later, it should include new logs showing whether the failing model call comes from OpenClaw core or a plugin-owned summarization path.

## Reproduction Assessment

Yes. The pre-fix path is source-level reproducible: reset session init strips a bare reset to an empty body, and before the fix the command handler could fall through after reset hooks. Current main has a direct regression test for the fixed bare `/new` path and source proof for `/reset` through the same branch.

## Solution Assessment

Yes. The current shared auto-reply handler is the narrow maintainable fix because it stops the empty command tail before model execution without adding Telegram-specific or lossless-claw-specific behavior, while preserving `/reset <message>` and `/new <message>`.
