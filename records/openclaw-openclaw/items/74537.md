## Summary

Current main already stops bare `/reset` and `/new` before the normal agent/model execution path, which addresses the empty-message provider call described in the report. The fix is covered by targeted reset-command tests and shipped in `v2026.4.27`.

## What This Changes

Avoid making an empty LLM request and surfacing a provider 400 when `/reset` or `/new` is sent to an already-fresh session, especially with Telegram and lossless-claw active.

## Best Possible Solution

Keep the shipped command-handler fix: bare `/reset` and `/new` should fire reset hooks, send a local acknowledgement, and stop before agent/model execution; only reset commands with a non-empty tail should dispatch that tail to the model.

## Reproduction Assessment

Yes. The issue gives concrete steps and provider logs, and current-main code/tests provide a high-confidence verification path: bare `/reset` or `/new` returns an acknowledgement and does not invoke `runEmbeddedPiAgent`.

## Solution Assessment

Yes. Handling this at `maybeHandleResetCommand` is the narrowest maintainable fix because it preserves reset hooks and tail dispatch while preventing the empty body from reaching any provider.
