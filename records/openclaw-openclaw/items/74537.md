## Summary

Keep open. Current main acknowledges bare `/reset` and `/new` without normal command fallthrough, but the reset lifecycle paths still emit empty-history `before_reset` payloads and `session_end` events, so the reported lossless-claw/provider 400 path is still plausible and not implemented away.

## What This Changes

The issue asks OpenClaw to avoid triggering model-backed reset/session-end lifecycle work when `/reset` or `/new` is run against an already-fresh session with no transcript messages.

## Best Possible Solution

Define and test the empty-transcript lifecycle contract so already-fresh resets do not trigger model-backed wrap-up work, while preserving useful `before_reset` and `session_end` notifications or providing an explicit no-op/empty-history signal for plugins that need cleanup.

## Reproduction Assessment

Yes for triage: the issue gives concrete Telegram steps and provider/cooldown logs, and current source still shows empty reset lifecycle payloads can be emitted. I did not run a live Telegram/lossless-claw reproduction in this read-only review.

## Solution Assessment

Unclear. Skipping the failing empty-history model work is the right outcome, but a direct core skip of all lifecycle hooks may be too broad; the safer fix needs maintainer choice of whether to guard `before_reset`, `session_end`, plugin-side model work, or add a typed empty-transcript signal.
