## Summary

Keep open. Current main still forwards `--history-limit 0` into `chat.history`, and the PR’s runtime change is a narrow TUI-side fix with regression coverage; the remaining merge blocker is the missing public TUI docs and required changelog entry.

## What This Changes

The branch adds a `historyLimit === 0` early return in TUI history loading and a regression test verifying that no gateway history request is made.

## Best Possible Solution

Merge the TUI-side zero guard after documenting that `--history-limit 0` disables initial history loading and adding the required user-facing changelog entry; keep the gateway `chat.history` positive-limit contract unchanged.

## Reproduction Assessment

Yes. Current main parses `--history-limit 0`, passes it to TUI `loadHistory`, and the gateway `chat.history` schema rejects limits below 1; the PR’s added test covers the desired no-gateway-call path.

## Solution Assessment

Yes for the runtime behavior. Handling zero in the TUI before calling `chat.history` is narrower than relaxing the gateway protocol for all clients, but docs and changelog should be added before merge.
