## Summary

Keep this PR open. Current main still forwards `--history-limit 0` into `chat.history`, whose gateway schema rejects zero, while this PR applies a narrow TUI-side no-history branch with focused regression coverage and no blocking review findings.

## What This Changes

The PR adds a TUI zero-history path that clears local chat state, marks history loaded, refreshes session info without calling `chat.history`, and covers that path with a Vitest regression test.

## Best Possible Solution

Merge the narrow TUI-side guard after maintainer review, preserving the gateway contract that `chat.history` limits start at 1 while allowing the TUI flag value `0` to mean no local history load.

## Reproduction Assessment

Yes. A code-trace reproduction is high confidence: `openclaw tui --history-limit 0` parses to numeric zero, current main forwards that zero to `chat.history`, and the gateway schema rejects `limit` below 1.

## Solution Assessment

Yes. Handling zero at the TUI session-loading boundary is the narrowest maintainable fix because it avoids weakening the shared gateway RPC contract and keeps no-history behavior local to the client option.
