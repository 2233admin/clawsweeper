## Summary

Keep open. Current main still has no supported Telegram parse-mode config at either `channels.telegram.parseMode` or `channels.telegram.accounts.<id>.parseMode`, and the main send, media-caption, draft-preview, and lower-level delivery paths still force Telegram HTML rendering with plain-text fallback only after HTML parse failures. The prior #10758 custom-emoji/raw-HTML item is narrower and closed, so it does not supersede this broader config request.

## Best Possible Solution

Keep this issue open and implement the feature in the bundled Telegram plugin/config surface once maintainers decide the allowed modes. A complete fix should add a typed Telegram parse-mode setting, decide whether per-account overrides are supported, wire it consistently through sends, media captions, edits, draft previews, and delivery helpers, then update strict schema, exported config types, generated config metadata/help, public Telegram docs, and focused Telegram/config tests. Treat MarkdownV2, raw HTML, custom emoji entities, and escaping as explicit product/security choices rather than incidental side effects.
