## Summary

Keep this PR open. It is maintainer-authored, carries the protected `maintainer` label, and current `main` still has the shared resolver behavior the PR targets: `resolveGroupToolPolicy` includes caller-provided `groupId` in group-policy candidates before session/spawned context. The PR is also security-sensitive because it changes group-scoped tool authorization across agents, Gateway HTTP/loopback tools, and media read policy, so it needs explicit maintainer review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer security review. The likely path is to review and land the resolver hardening plus gateway/session-context propagation if the trust model is correct, or ask for a smaller maintainer-approved variant that derives group trust only from verified server-side session or ingress metadata before changing shared tool policy.
