## Summary

Keep open. Current main still does not implement the requested built-in L1-L4 command risk classifier for exec approvals, and the new comment is an active implementation offer rather than evidence that the feature shipped. The shipped path remains policy, allowlist, durable approval, safe-bin, and strict inline-eval based.

## Best Possible Solution

Keep this issue open for maintainer product/security design. If pursued in core, add an explicit opt-in exec risk policy layer before approval resolution, with documented threat model, config/UI behavior, and tests for safe reads, dangerous denials, pipelines, wrappers, interpreters, obfuscation, and Windows/POSIX parsing. If maintainers prefer plugin-first classification, first add or document an exec-specific plugin policy hook that can classify, block, require approval, and safely auto-allow without weakening host exec approval boundaries; firefox1918's offered L4 detector would be a useful candidate PR or prototype for that path.

## Risks / Open Questions

- A broad heuristic command classifier is security-sensitive: shell parsing, wrappers, interpreters, PATH resolution, aliases, pipelines, Unicode/ANSI obfuscation, and Windows/POSIX differences can all create false allows or false denials.
- A plugin can currently block or add an extra approval before a tool call, but using the existing generic hook surface as the whole answer would not implement safe-command auto-approval inside host exec approvals.
- The related RubberBand work was closed unmerged per the provided context, so treating that as shipped would leave the core request unresolved.
