## Summary

Current main still does not implement the requested built-in L1-L4 command risk classifier for exec approvals. The active exec path remains policy, allowlist, durable-approval, safe-bin, and strict inline-eval based, and firefox1918's L4 detector offer is useful follow-up context rather than shipped code.

## What This Changes

The issue requests a built-in exec command risk classifier that auto-approves safe commands, uses normal or enhanced approval for intermediate risk, and auto-denies dangerous commands.

## Best Possible Solution

Keep this open for maintainer product/security design. If pursued in core, add an explicit opt-in exec risk policy layer before approval resolution with documented threat model, config/UI behavior, and regression coverage for safe reads, dangerous denials, pipelines, wrappers, interpreters, obfuscation, and Windows/POSIX parsing. If maintainers prefer plugin-first classification, first document or add an exec-specific policy hook that can classify, block, require approval, and safely auto-allow without weakening host exec approval boundaries; firefox1918's L4 detector could be reviewed as a prototype for that path.

## Risks / Open Questions

- A built-in heuristic classifier would materially change host-exec security posture; shell parsing, wrappers, interpreters, PATH resolution, aliases, pipelines, obfuscation, and Windows/POSIX differences can all create false allows or false denials.
- The existing generic plugin hook can block or require extra approval before a tool call, but it does not provide an exec-specific safe auto-allow decision through the host approval boundary.
- The new L4 detector offer may be valuable, but maintainers need to decide whether a dangerous-command detector belongs in core, in a plugin, or behind a new exec policy API before implementation is safe to automate.
