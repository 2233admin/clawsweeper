## Summary

Keep open. Current main has cron scheduling, task-ledger completion tracking, failure alerts, and a narrow reply-time reminder guard, but it still does not implement the requested native commitment/deferred-task primitive with `commit add/list/done`, generic commitment records, overdue escalation, or startup/session surfacing for non-reminder promises. The related PENDING.md items are workaround/plugin-pattern discussions and do not supersede this core product request.

## Best Possible Solution

Keep this issue open and scope a core design around a first-class commitment record with due time, linked cron/task run, done/cancel/list/status operations, reconciliation from cron/task outcomes, overdue or failed-run escalation, and startup/session surfacing for uncompleted commitments. PENDING.md or skill-style patterns can remain optional workarounds, but they should not be treated as a replacement for the requested native primitive.

## Evidence

- **Issue scope:** The issue asks for a first-class commitment primitive that creates a one-shot schedule, tracks completion, marks done, escalates missed work, and surfaces overdue commitments. The reporter's follow-up describes a real deployment where a promise was written to a ledger but no cron was created, which is the exact gap the requested primitive would close.
- **No native commit CLI or gateway method:** The core CLI descriptor catalog lists built-in groups through `tasks` and has no `commit` or `commitment` command. The gateway method list exposes `cron.list/status/add/update/remove/run/runs`, but no commitment namespace.
  - file: [src/cli/program/core-command-descriptors.ts:6](https://github.com/openclaw/openclaw/blob/db40ec404a9178294dba3121758ff59459983a11/src/cli/program/core-command-descriptors.ts#L6)
  - command: `rg -n 'commit|commitment|deferred' src/gateway src/cli src/tasks src/cron src/auto-reply docs/cli docs/automation docs/tools --glob '!**/*.test.ts'`
  - sha: [db40ec404a91](https://github.com/openclaw/openclaw/commit/db40ec404a9178294dba3121758ff59459983a11)
- **Cron remains the scheduling surface:** `openclaw cron add` creates a scheduled job with schedule, session target, wake mode, payload, and optional delivery. That covers timed execution, but not a separate commitment record, `done` operation, or overdue commitment lifecycle.
  - file: [src/cli/cron-cli/register.cron-add.ts:66](https://github.com/openclaw/openclaw/blob/db40ec404a9178294dba3121758ff59459983a11/src/cli/cron-cli/register.cron-add.ts#L66)

- **Tasks are a ledger, not a commitment scheduler:** The tasks docs explicitly say background tasks are the activity ledger, not the scheduler. The CLI task commands support list, audit, maintenance, show, notify, cancel, and flow inspection, but not creating or completing a promised future commitment.
  - file: [docs/automation/tasks.md:11](https://github.com/openclaw/openclaw/blob/db40ec404a9178294dba3121758ff59459983a11/docs/automation/tasks.md#L11)

- **Cron/task completion substrate exists only after a cron run exists:** Cron execution creates running task records and later completes or fails those task records by run ID. This is useful substrate for an implementation, but it only applies once a cron job was actually created.
  - file: [src/cron/service/timer.ts:160](https://github.com/openclaw/openclaw/blob/db40ec404a9178294dba3121758ff59459983a11/src/cron/service/timer.ts#L160)

- **Reminder guard is intentionally narrow:** The reply guard only matches reminder/ping/follow-up/check-back wording, appends a warning if no `cron.add` succeeded, and suppresses the warning when a session-related cron already exists. It does not create commitments, list them, mark them done, escalate overdue work, or catch generic promises such as reports or weekend fixes.
  - file: [src/auto-reply/reply/agent-runner-reminder-guard.ts:8](https://github.com/openclaw/openclaw/blob/db40ec404a9178294dba3121758ff59459983a11/src/auto-reply/reply/agent-runner-reminder-guard.ts#L8)

- **Related items do not supersede:** The provided related context for #69621 and #69637 concerns a PENDING.md workspace-pattern workaround and docs-only PRs. Those are not a native commitment primitive and do not replace the product-level API and runtime behavior requested here.

## Risks / Open Questions

- A native commitment primitive needs product design for false positives in natural-language promise detection.
- Escalation and channel notification behavior needs explicit authority and consent boundaries.
- Implementation should reuse cron run history, failure alerts, task-ledger reconciliation, and the existing reminder guard instead of adding a parallel scheduler.
