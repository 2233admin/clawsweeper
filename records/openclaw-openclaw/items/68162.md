## Summary

Keep this open. The report is recent, corroborated by concrete released-version logs, and current main still has no targeted Control UI reconnect/page-refresh regression proving the same backing gateway session survives without a reset/new-session rollover.

## What This Changes

The issue requests that Control UI WebChat resume the existing gateway session across WebSocket reconnects, browser minimize/restore, and page refresh instead of creating a new session.

## Best Possible Solution

Add an explicit session-continuity contract for Control UI reconnects: reconnecting or remounting with the same selected session should reload history and send follow-ups against the same backing session unless the user intentionally resets or the configured reset policy legitimately expires it.
