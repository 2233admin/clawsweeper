## Summary

Keep open. The PR adds a plausible OpenAI `/models` discovery primitive, but current main still lacks the provider-catalog hook and Control UI/model-list plumbing that would make it user-visible, and the linked design issue still needs maintainer direction before this should land as a public plugin surface.

## What This Changes

The PR adds `discoverOpenAIModels`, tests for OpenAI-compatible `/models` response mapping and caching, and exports the helper from the OpenAI plugin barrel.

## Best Possible Solution

Settle the provider dynamic-catalog contract in #74481 first, then either merge this helper after small cleanup or fold it into the real OpenAI provider catalog path with docs, cache scoping, Control UI/model-list refresh behavior, and focused tests.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. Static inspection of the branch diff and current main is enough to verify that the helper exists on the PR branch but is not wired into current catalog resolution.

## Solution Assessment

Unclear. The helper is a reasonable first slice and mirrors Mantle, but the best end-state depends on the provider catalog API decision and may be cleaner if landed with the actual OpenAI provider hook instead of as a standalone public export.
