## Summary

Keep open. Current main still uses the builtin SQLite database as a retrieval/index store over Markdown memory files and session chunks, not as the requested typed durable record model with canonical memory rows, revisions, lifecycle status, confidence, validity windows, and normalized entities. The adjacent memory-wiki and memory-lancedb plugins add useful related capabilities, but they do not replace the default memory-core SQLite storage contract or satisfy this issue’s SQLite schema request.

## Best Possible Solution

Keep this issue open. The best path is to design a versioned durable-memory contract in memory-core and the memory-host SDK: SQLite migrations and contract tests for canonical memories rows, memory_revisions, explicit type/status/confidence/timestamp/validity/entity fields, and docs explaining how the Markdown index, memory-wiki, and optional memory plugins interact with the new source-of-truth model. Coordinate that with #42647, #42648, and #42650 so provenance, write behavior, and review/forget/conflict flows share one schema.
