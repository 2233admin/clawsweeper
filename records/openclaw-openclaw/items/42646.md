## Summary

Keep open. Current main still implements SQLite as a retrieval index over Markdown memory files, not the requested typed durable SQLite record model with canonical memory rows, revisions, lifecycle statuses, confidence, validity windows, and normalized entities. Same-author related memory MVP issues for provenance, write pipeline, and review/forget/conflict flows remain open and should be coordinated with this schema work.

## Best Possible Solution

Keep this issue open. The best path is a versioned durable-memory storage contract owned by memory-core and the memory-host SDK: add migrations and contract tests for canonical memories rows, memory_revisions, explicit type/status/confidence/timestamp/validity/entity fields, and docs explaining how Markdown/index backends interact with the new source-of-truth model. Coordinate this with #42647, #42648, and #42650 so provenance, write behavior, and operator flows share one schema.
