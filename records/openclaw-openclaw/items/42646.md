## Summary

Keep open. Current main still implements the builtin SQLite memory backend as a search/index database over Markdown memory files and session chunks, not the requested typed durable SQLite record model with canonical memory rows, revision history, lifecycle statuses, confidence, validity windows, and normalized entities. The adjacent memory-wiki plugin adds provenance-rich wiki artifacts, but its docs explicitly say it sits beside the active memory plugin rather than replacing memory-core storage. Same-author related Memory MVP issues for provenance, write pipeline, and review/forget/conflict flows remain open and should be coordinated with this schema work.

## Best Possible Solution

Keep this issue open. The best path is to design and implement a versioned durable-memory storage contract in memory-core and the memory-host SDK: migrations and contract tests for canonical memories rows, memory_revisions, explicit type/status/confidence/timestamp/validity/entity fields, and docs explaining how Markdown/index/wiki backends interact with the new SQLite source-of-truth model. Coordinate this with #42647, #42648, and #42650 so provenance, write behavior, and operator review/forget/conflict flows share one schema.
