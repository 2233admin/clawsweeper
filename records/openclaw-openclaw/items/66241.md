## Summary

Keep open. Current main still does not persist the Discord component/modal registry across gateway restarts; this PR remains a plausible implementation candidate, but it touches Discord routing and authorization state and needs maintainer review or rework against the existing state-dir, lock, and atomic JSON patterns. Related earlier PRs #40288 and #47818 are closed/unmerged, and #70626 is an open draft helper rather than an implemented Discord fix.

## Best Possible Solution

Keep this PR open for maintainer follow-up. Rework the Discord registry persistence to use the active OpenClaw state directory and existing locked/atomic JSON patterns, or adapt it after #70626 if that helper lands. The final change should preserve route overrides and `allowedUsers`, validate persisted records before loading, prune expired entries, bound growth, and add focused restart round-trip, malformed-file, expired-prune, and authorization/routing regression tests.

## Risks / Open Questions

- Current main still loses Discord component and modal registry state across gateway restarts, so visible Discord buttons/forms can expire before their logical TTL.
- This PR persists Discord routing and authorization metadata; dropping or corrupting `sessionKey`, `agentId`, `accountId`, or `allowedUsers` can route to the wrong agent/session or relax user-scoped component authorization.
- The PR introduces local disk persistence and path handling in an interaction-adjacent path, so it needs maintainer review for state directory consistency, file locking, atomicity, malformed data handling, and event-loop blocking behavior.
