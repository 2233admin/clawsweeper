## Summary

Keep open, but the PR should not merge as-is. Current main deliberately keeps trusted-proxy identity on an explicit proxy path, rejects token mixing at startup, requires `allowLoopback` for loopback proxy identity headers, and documents password-only local fallback; this PR cuts across those security contracts while #69066 is still open for the broader auth design.

## What This Changes

The PR changes gateway auth to add trusted-proxy local shared-secret fallback, including token/password fallback for loopback requests, altered local-direct detection, implicit same-host loopback proxy header handling, and related auth tests.

## Best Possible Solution

Keep the current explicit security model unless maintainers deliberately change it: password fallback for direct internal clients, `allowLoopback` for same-host reverse proxies, and any token/service fallback handled through a coordinated startup/config/docs/RFC decision with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require explicit opt-in for loopback proxy headers:** `src/gateway/auth.ts:301-302`
  - body: This exception accepts trusted-proxy identity from a loopback source whenever the Host is non-local and forwarding context is present. A local process can set those headers itself, so this makes same-host loopback trusted-proxy auth implicit instead of gated by `trustedProxy.allowLoopback`. Keep the loopback guard fail-closed unless the explicit config opt-in is set.
  - confidence: 0.9
- **[P2] Preserve raw loopback local-direct detection:** `src/gateway/auth.ts:151-153`
  - body: `isLocalDirectRequest` now depends on `resolveRequestClientIp`, which intentionally returns `undefined` when the socket address is in `trustedProxies` and no forwarded client header is present. With loopback in `gateway.trustedProxies`, direct `127.0.0.1` requests can stop being local-direct, affecting shared locality checks beyond this fallback path.
  - confidence: 0.86
- **[P2] Align token fallback with startup validation:** `src/gateway/auth.ts:390-396`
  - body: The fallback accepts `auth.token` in trusted-proxy mode, but current startup validation rejects trusted-proxy configs that also set a shared token. As written, the advertised token fallback is not usable through normal gateway config unless the validation and docs contract change with it.
  - confidence: 0.88
