## Summary

Keep open. Current main does not implement variant model keys: aliases resolve to the configured key and the runtime model id remains the selected string, so a suffixed key can still reach provider resolution/API handling unchanged. The requested behavior is valid, but the proposed global colon delimiter is unsafe because colon is already meaningful in real model ids.

## What This Changes

The issue requests allowing multiple configured variants of the same provider model by stripping a custom suffix from model keys before API calls.

## Best Possible Solution

Add an explicit, documented variant mapping for `agents.defaults.models` so a unique config key/alias can point to a separate provider model id while keeping per-variant params. The implementation should preserve exact model ids by default and update model listing, `/model`, WebChat picker selection, params lookup, and provider runtime resolution together.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure an alias on a suffixed key, select that alias, and the resolver returns the suffixed model ref; runtime fallback then constructs the provider model using that exact `modelId`. I did not run live provider calls because this review is read-only, but the current source path is sufficient to reproduce the mismatch.

## Solution Assessment

No. Stripping everything after `:` is not the best fix because `:` is part of real provider model ids. A safer solution is an explicit config field or variant schema that separates the unique OpenClaw config key from the provider-facing model id.
