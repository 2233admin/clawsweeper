## Summary

Current main still does not implement the central OpenAI/OpenAI-compatible sampling-penalty passthrough request. Config accepts and merges arbitrary params, and native Ollama now has a provider-owned params path, but the shared Pi/OpenAI transport path still only first-class forwards temperature/maxTokens plus existing wrapper-specific fields. The prior implementation PRs referenced in the discussion are closed unmerged, and the newer related issues are broader rather than a clean superseding canonical item.

## Best Possible Solution

Keep this issue open as the canonical narrow request for OpenAI/OpenAI-compatible sampling-penalty passthrough. The best fix is a focused transport change that normalizes finite numeric snake_case and camelCase sampling params from existing merged `params` sources, forwards only supported keys for OpenAI Responses, OpenAI-compatible Chat Completions, Azure, and WebSocket payloads, preserves existing payload wrappers and `extra_body` behavior, adds tests, and documents the supported scope.

## Clownfish Work Prompt

Implement OpenAI/OpenAI-compatible generation sampling parameter passthrough for this issue. Current main already accepts and merges arbitrary `params`, but `src/agents/pi-embedded-runner/extra-params.ts` only maps `temperature` and `maxTokens` into stream options, and the OpenAI Responses/Chat Completions/WebSocket payload paths do not first-class forward `frequency_penalty`, `presence_penalty`, `repetition_penalty`, or config-level `top_p`/`top_k`.

Expected boundary:
- Add a focused helper in `src/agents/pi-embedded-runner/extra-params.ts` or a small adjacent helper to collect finite numeric sampling params from existing merged `params` sources.
- Support snake_case and camelCase aliases where appropriate: `frequency_penalty`/`frequencyPenalty`, `presence_penalty`/`presencePenalty`, `repetition_penalty`/`repetitionPenalty`, `top_p`/`topP`, and `top_k`/`topK`.
- Forward only supported request keys for OpenAI/OpenAI-compatible transport families: `openai-responses`, `openai-codex-responses`, `azure-openai-responses`, `openai-completions`, and the OpenAI WebSocket payload path. Do not blanket inject these fields into Anthropic/Google/native Ollama paths; native Ollama already owns its params in `extensions/ollama/src/stream.ts`.
- Preserve existing `onPayload`, `extra_body`, `chat_template_kwargs`, OpenRouter/Z.AI/vLLM wrappers, provider hook wrappers, and store-stripping behavior. If `extra_body` collides with generated request keys, keep the existing documented collision precedence.
- Add focused Vitest coverage for config merge, alias normalization, invalid/non-finite rejection, Responses payload injection, Chat Completions payload injection, WebSocket payload injection, `onPayload` chaining, and non-compatible-provider non-injection.
- Update user-facing docs for supported model params, preferably `docs/gateway/config-agents.md` and the OpenAI/provider docs. Add a changelog entry under the active version because this is user-facing behavior; credit the original reporter and useful commenters per repo changelog policy.

Do not change provider auth, model discovery, unrelated request policy, dependency versions, lockfiles, or native Ollama behavior.
