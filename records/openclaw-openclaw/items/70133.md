## Summary

Keep open. Current main still serializes Feishu same-chat message handling behind a promise chain with no timeout, lease expiry, abort, or forced release path, and the queued task still awaits the full agent dispatch before later same-chat messages can start. The related #54409 remains open and explicitly tracks a different collect-mode batching failure, so it does not supersede this queue-starvation report.

## Best Possible Solution

Keep this issue open and implement a narrow Feishu queue starvation fix: add a bounded release or timeout guard for same-chat dispatch tasks, preserve normal same-key ordering when tasks settle promptly, keep `/stop` control and `/btw` out-of-band lanes working, and add regression tests proving a never-settling first P2P task cannot block later same-chat messages indefinitely. Coordinate with #54409 only if the implementation also changes collect-mode behavior.
