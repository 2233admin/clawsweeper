## Summary

Keep open. Current main still has the reported Feishu same-chat starvation path: normal messages are serialized behind a per-chat Promise chain before they can reach the shared gateway active-run queue, and the latest release tag still contains that behavior. The related #54409 is overlapping but intentionally distinct, and open PR #43512 only addresses control-command bypasses, not ordinary P2P messages stuck behind a long dispatch.

## What This Changes

The issue requests that Feishu same-chat P2P message handling stop blocking later messages indefinitely when an earlier dispatch hangs or takes minutes.

## Best Possible Solution

Add a Feishu-owned stale-release or bounded bypass for ordinary same-chat message dispatch so later messages can reach the shared gateway turn queue while preserving existing control-command and /btw out-of-band lanes, dedupe claims, and normal fast-path ordering. Cover the starvation case with regression tests and keep broader collect-mode redesign tied to #54409 if it requires a larger queue architecture change.

## Reproduction Assessment

Yes. The provided logs give a credible live sequence, and current main has a deterministic unit-level reproduction: hold the first same-key Feishu dispatch promise and enqueue a second normal P2P event; the second task remains behind previous.then(task, task) until the first resolves.

## Solution Assessment

Yes, the requested timeout or release guard is the right direction, but the implementation should be Feishu-local unless a generic channel seam is proven necessary. PR #43512 is not sufficient because it only bypasses control commands and leaves ordinary P2P messages serialized behind the same queue.
