## Summary

Keep open. Current main still clears the ClawHub request timeout as soon as fetch returns headers, then reads the skill archive body with `response.arrayBuffer()` without an active timeout or signal. The linked implementation candidate #52140 was closed unmerged, and the latest release still contains the same native ClawHub archive body-read pattern.

## Best Possible Solution

Keep this issue open. The best fix is to keep an abort timeout alive through native ClawHub archive body consumption, or stream the archive through a signal-aware path, add a stalled-mid-stream regression test for `downloadClawHubSkillArchive`, and then verify Telegram still emits progress plus final or error delivery while ordinary same-chat follow-ups queue behind the active turn.
