## Summary

Keep open. Current `main` still clears the ClawHub fetch timeout when headers arrive, then reads the skill archive body with `response.arrayBuffer()` without an active body timeout. The linked fix candidate #52140 is closed and unmerged, and the latest release contains the same native ClawHub archive body-read pattern.

## Best Possible Solution

Keep this issue open and fix the native ClawHub archive body-read path. The best PR would keep an abort or idle timeout active through archive body consumption, preferably via a small shared helper used by skill and package archive downloads, add a stalled-mid-stream regression test for `downloadClawHubSkillArchive`, and verify that Telegram still delivers a final or error message when the install fails instead of hanging.

## Clownfish Work Prompt

Fix this issue by making native ClawHub skill archive downloads timeout while consuming the response body, not only while waiting for response headers. Work from current `main`. The observable bug is that Telegram-triggered skill installation can appear silent/unresponsive when the ClawHub archive server returns headers and then stalls mid-stream. The expected fix boundary is `src/infra/clawhub.ts` plus focused tests; prefer a small shared helper for ClawHub archive body reads so `downloadClawHubSkillArchive` is covered and `downloadClawHubPackageArchive` does not keep the same class of bug. Preserve existing auth, URL construction, integrity calculation, temp-file cleanup, and public API shapes. Do not rework Telegram queueing semantics; same-chat follow-ups may still queue behind an active turn, but the install should resolve with success or error. Add a regression test that simulates a `Response` body that stalls after headers and proves `downloadClawHubSkillArchive` rejects/cleans up on timeout or idle timeout. Add a user-facing changelog Fixes entry with `Thanks @benbenzhuyi`; include `Thanks @oliviareid-svg` too if the prior #52140 approach materially informs the implementation.
