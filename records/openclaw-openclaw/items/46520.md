## Summary

Keep open. Current main still canonicalizes `tauri://localhost` and `electron://app` through `URL.origin`, which Node reports as the opaque string `null`; the allowlist then compares against `null` instead of the configured custom-scheme origin. The closed related PRs in the supplied context were not merged, and the latest issue comment matches current behavior.

## Best Possible Solution

Keep this issue open and land a narrow gateway fix in `src/gateway/origin-check.ts`: when `URL.origin` is `null` but `protocol` and `host` are present, compare a reconstructed `protocol//host` origin against the explicit allowlist. Add `tauri://` and `electron://` accepted/rejected cases in `src/gateway/origin-check.test.ts`, and verify literal `Origin: null` remains rejected.

## Risks / Open Questions

- Closing now would leave Tauri/Electron-style desktop clients choosing between a broken exact allowlist and the documented `allowedOrigins: ["*"]` security downgrade.
- The eventual fix should preserve rejection of literal `Origin: null` and only reconstruct hosted custom-scheme origins for exact allowlist matching.
