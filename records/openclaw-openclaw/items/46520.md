## Summary

Keep open. Current main still normalizes hosted custom-scheme origins through `URL.origin`, so exact `gateway.controlUi.allowedOrigins` entries such as `tauri://localhost` cannot match; open PR #73511 is already the active closing implementation candidate and should be reviewed or closed before this issue is resolved.

## What This Changes

The issue requests that Control UI gateway origin validation accept explicitly configured custom-scheme origins such as `tauri://localhost` and `electron://app` without requiring wildcard `allowedOrigins`.

## Best Possible Solution

Land a narrow gateway origin normalization fix that preserves exact `scheme://host` matching for hosted custom schemes, keeps literal `Origin: null` and malformed origins invalid, keeps unlisted custom origins rejected, and adds focused `tauri://` / `electron://` regression tests.

## Reproduction Assessment

Yes. A read-only Node check reproduces `URL.origin === "null"` for `tauri://localhost` and `electron://app`, and current main passes that value directly into the exact allowlist comparison.

## Solution Assessment

Yes. The active PR direction is the narrowest maintainable solution: fix custom-scheme normalization before the existing allowlist check without changing wildcard, host-header fallback, loopback, or literal-null behavior.
