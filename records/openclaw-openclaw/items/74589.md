## Summary

Keep open. Current main still has the Anthropic-compatible transport boundary that can convert replay history to an empty `messages` array and send it to the provider, and this issue is already paired with open closing PR #74731, so it should remain open until that PR is reviewed and merged or rejected.

## What This Changes

Investigate and guard MiniMax M2.7 Anthropic-compatible requests that intermittently become `messages: []` after tool-call chains, causing MiniMax 400/2013 `chat content is empty` errors.

## Best Possible Solution

Review and land the focused Anthropic-compatible payload guard in #74731 if it preserves valid tool-result continuations and prevents zero-message provider requests. Keep the broader blank replay-history work tracked separately by #73926 and #73929.

## Reproduction Assessment

Partially. There is a high-confidence code-level reproduction path by passing empty or fully filtered replay messages through the current Anthropic-compatible payload builder and observing `messages: []`; the full live MiniMax tool-chain failure remains intermittent and not minimized.

## Solution Assessment

Yes. A final guard at the Anthropic-compatible payload boundary is the narrowest maintainable fix for this MiniMax failure mode, while broader embedded-runner prompt hygiene should stay in the separate linked work.
