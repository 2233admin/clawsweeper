## Summary

Keep open. The Windows LSP shim failure is real on current main and this PR is pointed at the right shared resolver, but the patch is not merge-ready because it resolves against `process.env` instead of the merged LSP server environment and it is missing the required user-facing changelog entry.

## What This Changes

The PR wires bundled LSP server spawning through the shared Windows spawn resolver and passes the materialized command, argv, shell, and windowsHide settings to `child_process.spawn`.

## Best Possible Solution

Keep the shared resolver approach, but build the merged LSP spawn env once, pass it to both `resolveWindowsSpawnProgram` and `spawn`, add focused LSP runtime coverage for Windows `.cmd` lookup through config env, and add a single-line `CHANGELOG.md` Fixes entry before merge.

## Reproduction Assessment

Yes. Current main statically reproduces the reported path because bundled LSP servers flow to a direct `spawn(config.command, ...)` with no Windows `.cmd` resolver; the PR body also reports manual Windows 11 before/after proof. The PR-specific env mismatch is reproducible by putting the LSP shim only in `config.env.PATH`.

## Solution Assessment

No, not as written. Reusing `resolveWindowsSpawnProgram` is the narrow maintainable direction, but the resolver must see the same environment used by the child process and the user-facing fix needs changelog and targeted regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Resolve against the merged LSP env:** `src/agents/pi-bundle-lsp-runtime.ts:72-75`
  - body: `resolveWindowsSpawnProgram()` defaults to `process.env`, but this spawn later gives the child `{ ...process.env, ...config.env }`. If an LSP config supplies `PATH` or `PATHEXT`, the resolver can miss the `.cmd` shim or resolve a different wrapper than the child environment intends, leaving Windows startup broken or launching the wrong target. Build the merged env once and pass it to both the resolver and `spawn`.
  - confidence: 0.88
- **[P3] Add the changelog entry:** `CHANGELOG.md:12`
  - body: This is a user-facing Windows LSP fix, but the diff does not add a `CHANGELOG.md` entry under the active Fixes section. OpenClaw policy requires user-facing `fix` changes to be listed so the repair is not missing from release notes.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff touches process execution and currently lets the Windows resolver choose a command target from a different environment than the child process will receive.

Concerns:

- **[medium] Resolver can use the wrong PATH:** `src/agents/pi-bundle-lsp-runtime.ts:72`
  - body: Because the PR does not pass the merged LSP env into `resolveWindowsSpawnProgram`, Windows PATH/PATHEXT lookup can use `process.env` while the spawned LSP server receives `{ ...process.env, ...config.env }`. That mismatch can select an unintended wrapper or entrypoint in command-execution code.
  - confidence: 0.78
