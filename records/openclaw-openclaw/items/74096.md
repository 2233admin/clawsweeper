## Summary

Keep open. Current main still builds image/video/music provider arrays before the shared media resolver can return the explicit model config, and the provided context shows #74097 is an open paired PR with closing syntax for this issue.

## What This Changes

The issue requests deferring image/video/music runtime provider listing until the media tool model-config resolver actually needs fallback candidate resolution.

## Best Possible Solution

Review and land #74097, or an equivalent narrow patch, so resolveCapabilityModelConfigForTool accepts a provider thunk, calls it only after explicit modelConfig is incomplete, updates the image/video/music callers, and adds regression coverage for both complete and incomplete config paths.

## Reproduction Assessment

Yes for the eager lookup: current source shows all three media tools synchronously call listRuntime*GenerationProviders before resolveCapabilityModelConfigForTool can short-circuit on explicit config. The exact production reload timing on current main would need live timing because the related cache fix changes part of the cost profile.

## Solution Assessment

Yes. The proposed provider-thunk approach is the narrowest maintainable fix because it keeps fallback candidate behavior intact while avoiding plugin capability resolution when explicit media model config is already complete.
