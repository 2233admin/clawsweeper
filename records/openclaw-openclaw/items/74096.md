## Summary

Keep open. Current main still eagerly lists image, video, and music capability providers before the shared resolver can short-circuit on an explicit complete modelConfig, and open PR #74097 is already linked with closing syntax as the narrow implementation candidate. Per policy, this issue should stay open until that PR merges or is closed.

## Best Possible Solution

Keep this issue open and treat #74097 as the active implementation candidate. The best fix is the narrow lazy-provider-listing change described there: make resolveCapabilityModelConfigForTool accept a provider thunk, call it only after explicit modelConfig fails to resolve, update the image/video/music callers, and add regression coverage proving the thunk is not invoked for complete explicit modelConfig but is invoked for incomplete config.
