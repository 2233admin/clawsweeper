## Summary

Keep open. Current main still lacks the Mattermost-specific tool threading context and send-action fallback needed for proactive `message` sends without an explicit `replyTo` to inherit the active Mattermost thread. The issue is already paired with open closing PRs that target the message-tool and cron/heartbeat cases, so maintainers should review or supersede those rather than close this issue now.

## What This Changes

The issue requests that Mattermost proactive message sends and cron/heartbeat wake replies inherit the active thread context when no explicit reply target is supplied.

## Best Possible Solution

Review, rebase, land, or replace the linked Mattermost PRs with a Mattermost-owned threading tool context and guarded send-action fallback, plus regression tests for explicit `replyTo`, implicit same-thread inheritance, cross-channel suppression, `first`/`all`/`off` behavior, and any remaining cron/heartbeat wake routing gap.
