## Summary

Keep open. Current main still lacks the Mattermost message-tool thread-context fallback and still persists last-route delivery context only for direct Mattermost sessions, while open PRs #52120 and #52236 explicitly target the two reported cases with closing syntax. The issue should not be closed before those PRs are reviewed and merged or superseded.

## What This Changes

Requests that Mattermost proactive message-tool sends and cron/heartbeat wake replies inherit the active thread context for thread-scoped sessions.

## Best Possible Solution

Mattermost should provide thread-aware message-tool context and persist routable delivery context, including threadId, for thread-scoped channel/group sessions, with regression coverage for both proactive message sends and cron/heartbeat wake delivery.
