## Summary

Keep this PR open. Current main still exposes `buffer`/`filename`/`contentType` on the shared `message.send` schema, but the active Telegram plugin path does not hydrate inline buffers for `send`, does not count `buffer` as media in the shared send runner, and does not forward inline buffer fields to the Telegram transport. The PR is stale against current pluginized paths and still needs a pre-decode base64 size guard, but it remains a relevant implementation candidate for #41779 rather than something current main has already made obsolete.

## Best Possible Solution

Keep this PR open as a useful but stale implementation candidate for #41779. A mergeable version should refresh onto current main, move the Telegram-specific changes into `extensions/telegram/src/action-runtime.ts`, `extensions/telegram/src/channel-actions.ts`, and `extensions/telegram/src/send.ts`, coordinate shared `send` media hydration with #63578, add focused regression coverage for buffer-only sends and data-URL normalization, and enforce a pre-decode base64 size bound before constructing a `Buffer`.
