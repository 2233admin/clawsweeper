## Summary

Keep this PR open. Current main still exposes `buffer`, `filename`, `contentType`, and `mimeType` on the shared message send schema, but the active Telegram pluginized send path only hydrates and forwards URL/path media. The PR remains a relevant implementation candidate for the Telegram buffer attachment bug, but it needs a refresh for current pluginized paths and a stronger pre-decode base64 size guard before it is mergeable.

## Best Possible Solution

Keep this PR open as a useful implementation candidate. A mergeable version should refresh onto current main, coordinate generic `send` media hydration with #63578, update the active Telegram plugin action/runtime/transport paths, add focused regression tests for buffer-only sends and data-URL normalization, and enforce a pre-decode base64 size bound before constructing a Buffer.
