## Summary

Keep open. Current main still has the reported bypass shape: inbound dedupe keys on route plus `MessageSid`, `chat.send` maps each client idempotency key to `MessageSid`, and the Control UI guard only coalesces duplicate submits in the same UI path before a second run id exists. The related #72314 work is useful adjacent idless-message dedupe, but it is open and does not directly cover two identified Control UI/WebChat dispatches that both have ids.

## What This Changes

The issue requests that Control UI/WebChat send exactly one inbound turn per browser message instead of dispatching duplicate turns under different sender and run-id identities.

## Best Possible Solution

Keep the issue open, collect metadata for one duplicate pair, then either remove the upstream double dispatch or add a narrowly scoped Control UI/WebChat guard that does not change normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains how two different run ids would bypass dedupe, but there is no current-main duplicate pair or log that identifies the second dispatch path.

## Solution Assessment

Unclear. A secondary content fingerprint may be useful defense in depth, but the narrowest maintainable fix is to first trace why two runtime-distinct dispatches happen and guard or remove that path.
