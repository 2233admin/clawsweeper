## Summary

Keep open. Current main still supports the reported bypass shape: inbound dedupe is keyed by provider/route/session/message id, `chat.send` uses the request idempotency key as `MessageSid`, and the related open PR #72314 is adjacent missing-id work rather than a fix for two runtime-distinct sends with different ids.

## What This Changes

The issue requests a fix for Control UI WebChat messages being injected twice into the same session under different sender identities and message ids, causing duplicate agent replies.

## Best Possible Solution

Keep the issue open, collect metadata for one duplicate pair, and then either remove the extra upstream dispatch or add a narrow backend guard that catches the same Control UI/WebChat fan-out without changing normal repeated-message behavior.

## Reproduction Assessment

Unclear. The reporter gives plausible v2026.4.24 steps and current source explains why two different run ids would bypass dedupe, but the review did not find a current-main duplicate pair or logs that name the second dispatch path.

## Solution Assessment

Unclear. A secondary content fingerprint may be useful defense in depth, but the safer maintainable fix is to first trace why two runtime-distinct dispatches happen and guard or remove that path.
