## Summary

Current main addresses the reported Control UI duplicate-send path by coalescing duplicate in-flight submits before a second `chat.send` run id is generated. The fix is covered by regression tests, documented in WebChat behavior docs, and included in v2026.4.27.

## What This Changes

The issue requests that a single Control UI/WebChat user message produce only one inbound agent turn, instead of duplicate turns with distinct sender/run identities.

## Best Possible Solution

Keep the shipped Control UI submit coalescing behavior and leave broader content-fingerprint inbound dedupe work to the separate idless-retry path tracked by #72314.

## Reproduction Assessment

Yes: the current regression test reproduces the relevant duplicate in-flight Control UI submit path by issuing two same-prompt sends before acknowledgement. Current main now collapses that to one `chat.send` request and one optimistic user message.

## Solution Assessment

Yes: coalescing at the Control UI boundary is the narrowest fix for this report because it prevents a second run id and sender context from reaching gateway dispatch, while preserving provider-message-id inbound dedupe semantics.
