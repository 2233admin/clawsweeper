## Summary

Keep this PR open. Current main does not contain the proposed guardrail, and the latest PR head still has blocking security and correctness issues in the fallback scanner, vault persistence, and new API-key config surface.

## What This Changes

The PR adds an opt-in reversible sensitive-data guardrail that scans user prompts with a local OpenAI-compatible model or regex fallback, replaces detected secrets with `[VAULT_N]` tokens before model transit, restores tokens for assistant/tool paths, and adds config, UI metadata, docs, and a vault unit test.

## Best Possible Solution

Keep the PR open and require a hardened security design before merge: block-aware regex fallback coverage with tests, no durable plaintext vault unless it has a reviewed lifecycle or protected storage path, SecretRef/sensitive handling for the local model API key, and a maintainer decision on whether the feature should be a generic guardrail hook/contract rather than embedded-runner-specific policy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: send a normal multi-line PEM/OpenSSH/PGP private key through `scanSensitive` in regex-only mode; because the scanner splits on newlines before applying full-block regexes, the complete key is not redacted before model transit.

## Solution Assessment

No. The current PR is not the best way to solve the issue yet because it embeds broad security policy and reversible secret storage directly in the runner while leaving concrete fallback, storage, and credential-config gaps; a hardened implementation behind a reviewed guardrail contract would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scan private key blocks before line splitting:** `src/security/patterns.ts:97-109`
  - body: The private-key regexes match full PEM/OpenSSH/PGP blocks, but `scanSensitive` splits the prompt by newline and then runs every regex on each single line. A normal multi-line private key therefore yields no complete finding in regex-only mode, so the key can still be sent to the cloud model despite the fallback claiming private-key coverage. Run block patterns against the full text before line splitting, or use block-aware scanning.
  - confidence: 0.95
- **[P2] Avoid leaving reversible vault files in session storage:** `src/agents/pi-embedded-runner/run.ts:923-929`
  - body: This writes the token-to-secret map as raw JSON beside the session file. `0600` helps with cross-user reads, but the file still survives session end, backups, and support collection outside the existing SecretRef audit/scrub lifecycle. Keep the vault in memory unless persistence has a reviewed lifecycle, or integrate it with protected storage, cleanup, and audit handling.
  - confidence: 0.86
- **[P2] Treat the local model API key as a secret input:** `src/config/zod-schema.ts:1029`
  - body: `securityGuardrail.localApiKey` is added as a plain string, so it cannot use the existing SecretRef path and is not registered as sensitive for config redaction/UI metadata. This is a new API-key-bearing config field in a security feature, so it should use `SecretInputSchema.optional().register(sensitive)` and the matching resolved runtime type, like other `apiKey` fields.
  - confidence: 0.84
