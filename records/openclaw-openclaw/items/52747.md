## Summary

Keep open. The PR targets a real current-main ACP control-plane gap, but the proposed timeout design is not merge-ready because it can release the keyed session lane while the timed-out operation continues, can undercount live runtimes during timed idle eviction, and accepts timer values that Node turns into near-immediate timeouts. Prior Aisle and ClawSweeper review comments already flagged the same concurrency and resource-accounting concerns.

## What This Changes

This PR adds configurable `acp.sessionLane.taskTimeoutMs`, wraps ACP session actor queue operations with a timeout, passes it through manager session and idle-eviction paths, updates config schema/help/generated metadata, and adds manager coverage for a hung initialize unblocking a later initialize.

## Best Possible Solution

Keep a session-lane deadline feature, but revise it so deadlines cannot break per-session serialization: add cooperative cancellation where ACP runtime contracts support it, fence or retire stale timed-out operations before later same-session work can observe side effects, keep runtime handles counted until close is confirmed or tracked as closing, clamp and unref timers, and cover the behavior with focused manager/config regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve serialization after lane timeouts:** `src/acp/control-plane/session-actor-queue.ts:31-39`
  - body: `KeyedAsyncQueue` advances same-key work when the promise returned from this wrapper settles, but the timeout rejects that promise while `params.op()` continues in the detached chain. Later initialize/status/cancel/close work for the same ACP session can then run concurrently with stale side effects, so the timeout needs cancellation/fencing or must keep the queue tail locked until the old operation is safe to retire.
  - confidence: 0.94
- **[P1] Keep evicted runtimes counted until close finishes:** `src/acp/control-plane/manager.core.ts:1892-1899`
  - body: Idle eviction clears `runtimeCache` before `runtime.close()` and now runs under the lane timeout. If close hangs and the timeout releases the lane, the live runtime is no longer counted by `maxConcurrentSessions`; clear only after confirmed close or track a closing state that still counts.
  - confidence: 0.9
- **[P2] Clamp session-lane timeouts to Node's timer range:** `src/config/zod-schema.ts:519`
  - body: The schema accepts any nonnegative integer, but Node sets delays above `2147483647` ms to `1` ms. A large configured timeout would therefore fail ACP lane tasks immediately, so reject or cap oversized values before scheduling the timer.
  - confidence: 0.91
