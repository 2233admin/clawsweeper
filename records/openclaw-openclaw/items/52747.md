## Summary

Keep this PR open. Current main still has the reported ACP lane hang risk, but this patch is not merge-ready because the timeout settles the queue-visible promise while the original actor operation keeps running, and it also applies that timeout to runtime eviction paths whose cache accounting assumes close completion.

## What This Changes

The PR adds configurable `acp.sessionLane.taskTimeoutMs`, wraps ACP session actor queue tasks with a timer, wires the setting through config schema/help/generated metadata, and adds a regression test for a hung initialize task unblocking a queued initialize.

## Best Possible Solution

Keep the ACP session-lane deadline work, but revise the design so initialize/status/cancel/close/idle-eviction deadlines do not break per-session serialization, stale operations cannot mutate newer session state, runtime cache/concurrency accounting remains accurate until close is confirmed or fenced, timer values are bounded to Node-safe delays, timers are unref'd, and the new config is documented with focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve serialization after a lane timeout:** `src/acp/control-plane/session-actor-queue.ts:31-39`
  - body: `KeyedAsyncQueue` advances same-key work when the promise returned from this wrapper settles, but this timer rejects that queue-visible promise while `params.op()` keeps running. The next same-session task can then start and mutate/cache/close the same ACP session concurrently with the timed-out operation, so the timeout needs cancellation/fencing or a caller-visible timeout that does not release the lane until the old operation is safe to retire.
  - confidence: 0.94
- **[P1] Keep evicted runtimes counted until close is confirmed:** `src/acp/control-plane/manager.core.ts:1890-1906`
  - body: Idle eviction still clears `runtimeCache` before awaiting `runtime.close()`, but the new timeout can now let eviction continue after a hung close while the runtime is still alive. Because `maxConcurrentSessions` is enforced from the cache size, this can orphan a runtime and undercount active sessions; clear the cache only after close succeeds or track a closing state that still counts.
  - confidence: 0.89
- **[P2] Clamp timeout values before scheduling timers:** `src/config/zod-schema.ts:519`
  - body: The config schema accepts any nonnegative integer, and the generated schema allows up to `Number.MAX_SAFE_INTEGER`, but Node clamps delays above the signed 32-bit timer range to `1` ms with `TimeoutOverflowWarning`. A large configured timeout would therefore fail lane tasks immediately, so validation or resolution should reject or cap values above Node's safe timer maximum.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies or workflow execution paths, but it introduces ACP concurrency and resource-accounting risks in runtime-control code.

Concerns:

- **[high] Timeout can break per-session serialization:** `src/acp/control-plane/session-actor-queue.ts:31`
  - body: Rejecting the queue-visible promise while the underlying actor operation continues can let later same-session work run concurrently with stale side effects from the timed-out task.
  - confidence: 0.94
- **[high] Idle eviction can orphan runtime accounting:** `src/acp/control-plane/manager.core.ts:1890`
  - body: A timed-out idle close can leave the runtime process/session alive after its cache entry was removed, weakening `maxConcurrentSessions` enforcement and resource cleanup.
  - confidence: 0.88
- **[medium] Large timeout values can force immediate failures:** `src/config/zod-schema.ts:519`
  - body: Because Node clamps oversized timer delays to `1` ms, the permissive config schema can turn an intended long timeout into near-immediate ACP task failure.
  - confidence: 0.9
