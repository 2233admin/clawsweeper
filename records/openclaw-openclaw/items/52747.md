## Summary

Keep open. The underlying stuck ACP session-lane problem is real and current main still has no lane deadline, but this PR's timeout design releases the queue-visible promise while the underlying actor operation can keep running, which can break per-session serialization and runtime accounting. Maintainer review should decide the ACP cancellation, fencing, and cleanup semantics before merge. https://github.com/openclaw/openclaw/pull/52747

## What This Changes

The PR adds configurable `acp.sessionLane.taskTimeoutMs`, wires it through ACP session actor execution and config schema/help/generated metadata, and adds manager coverage for a hung initialize task unblocking a queued initialize.

## Best Possible Solution

Keep a session-lane deadline feature, but revise it so deadlines do not break per-session serialization: add cooperative cancellation where runtime contracts support it, fence or retire stale timed-out operations before later same-session work observes side effects, keep runtime handles counted until close is confirmed or represented as closing, clamp timer values to Node's supported range, and cover initialize/status/cancel/close/idle-eviction paths with focused tests.

## Reproduction Assessment

Yes. A mocked ACP runtime whose `ensureSession()` never settles reproduces the stuck lane, and the PR's own test shape demonstrates that a later same-session initialize is blocked without a timeout. Source inspection then shows why the proposed early rejection changes the serialization boundary.

## Solution Assessment

No. Releasing the promise that `KeyedAsyncQueue` uses as the same-key serialization boundary is not the narrowest safe fix; the safer design needs cancellation or fencing plus explicit runtime accounting before the lane is released.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve serialization after lane timeouts:** `src/acp/control-plane/session-actor-queue.ts:32-40`
  - body: `KeyedAsyncQueue` advances same-key work when the promise returned from this wrapper settles, but the timeout rejects that promise while `params.op()` continues in the detached chain. Later initialize/status/cancel/close work for the same ACP session can then run concurrently with stale side effects, so the timeout needs cancellation/fencing or must keep the queue tail locked until the old operation is safe to retire.
  - confidence: 0.94
- **[P1] Keep evicted runtimes counted until close finishes:** `src/acp/control-plane/manager.core.ts:1892-1899`
  - body: Idle eviction clears `runtimeCache` before `runtime.close()` and this PR runs that path under the lane timeout. If close hangs and the timeout releases the lane, the live runtime is no longer counted by `maxConcurrentSessions`; clear only after confirmed close or track a closing state that still counts.
  - confidence: 0.9
- **[P2] Clamp session-lane timeouts to Node's timer range:** `src/config/zod-schema.ts:520`
  - body: The schema accepts any nonnegative integer, but Node sets delays above `2147483647` ms to `1` ms. A large configured timeout would therefore fail ACP lane tasks immediately, so reject or cap oversized values before scheduling the timer.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The diff does not add dependency or workflow supply-chain changes, but it does introduce ACP runtime-control concurrency changes that can weaken per-session serialization and runtime resource accounting.

Concerns:

- **[high] Timeout can break per-session serialization:** `src/acp/control-plane/session-actor-queue.ts:32`
  - body: Rejecting the queue-visible promise while the underlying actor operation continues can let later same-session work run concurrently with stale side effects from the timed-out task.
  - confidence: 0.94
- **[high] Idle eviction can orphan runtime accounting:** `src/acp/control-plane/manager.core.ts:1892`
  - body: A timed-out idle close can leave the runtime process/session alive after its cache entry was removed, weakening `maxConcurrentSessions` enforcement and cleanup tracking.
  - confidence: 0.88

  - body: Because Node sets oversized timer delays to `1` ms, the permissive config schema can turn an intended long timeout into near-immediate ACP task failure.
  - confidence: 0.9
