## Summary

Keep this PR open. Current main at 7e5c3753f6f797d4faf7e22eefe5a11f5a3a17ed still does not implement the proposed chat-channel session switching: there is no loaded `/sessions` handler, `/session` only accepts `idle` and `max-age`, and session config/store types do not include `session.historyLimit` or per-SessionKey switchable history. The PR remains an implementation candidate, but it needs maintainer review for the session-state and security-sensitive restore semantics raised in review.

## Best Possible Solution

Keep this PR open for normal maintainer review or a successor implementation. The best path is a focused chat-session switching change that registers `/sessions`, supports `/session <number|sessionId|back>`, persists bounded per-SessionKey history, preserves or deliberately clears every session-scoped runtime/security field, handles in-flight run and lifecycle-hook semantics, updates docs/config schema, and adds regression coverage for disabled history, LRU eviction, transcript paths, snapshot restore, and command visibility.
