## Summary

Keep open. Current main appears to have addressed part of the reverse-proxy challenge-auth path by resolving remote-mode node-host credentials and by returning device-pairing request IDs, but it still does not provide first-class SSH transport support for `openclaw node run` or `openclaw node install`. The current documented node-host SSH workflow is still an external tunnel plus `--host 127.0.0.1 --port <localPort>`.

## Best Possible Solution

Keep this issue open and split the fix boundary: preserve the current remote-token/device-pairing improvements, then design a reviewed node-host SSH transport path that reuses the existing SSH tunnel helper without weakening WebSocket auth or private-network safeguards. The implementation should cover foreground `node run`, installed services, docs, and tests, and should be reconciled with the related open SSH remote-config work in #26250.
