## Summary

Keep open. Current main has improved the node-host credential and device-pairing path, but the central request remains unresolved: node-host foreground and installed-service flows still build a direct WebSocket connection from host/port/TLS settings, while SSH tunnel lifecycle exists in gateway probing and the public node docs still document a manually managed tunnel.

## What This Changes

The issue requests first-class SSH transport support for `openclaw node run` and installed node services, plus corrected challenge-auth behavior for remote node pairing behind a TLS reverse proxy.

## Best Possible Solution

Implement a reviewed node-host SSH transport that reuses the existing SSH port-forward helper for both foreground and installed-service flows, connects `GatewayClient` to the local loopback tunnel endpoint, preserves remote token/password and device-pairing semantics, updates node docs, and coordinates endpoint validation with #26250.

## Reproduction Assessment

Yes for the remaining SSH-transport gap: current source and docs show node-host foreground and service flows accept only direct host/port/TLS settings and require users to manage SSH tunnels externally. The reverse-proxy challenge-auth half is only source/test assessed here, not live-reproduced against nginx/podman.

## Solution Assessment

No, the current state is not the best complete solution because node hosts still depend on external tunnel management. The maintainable solution is a node-host-owned transport path with explicit tunnel lifecycle, loopback endpoint validation, and credential/device-pairing coverage.
