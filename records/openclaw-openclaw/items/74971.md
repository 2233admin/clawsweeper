## Summary

Keep open. Current main still has a source-backed config-load path from model-default materialization into bundled provider policy public-artifact loading, and that loader still prepares bundled runtime dependencies by default before gateway startup reaches normal runtime activation.

## What This Changes

Fix the gateway config-load regression where sealed Docker deployments can fail with EACCES because provider policy/default normalization tries to create a bundled plugin runtime-deps staging directory.

## Best Possible Solution

Fix the control-plane boundary so config validation/default materialization can load lightweight provider policy artifacts without creating or repairing bundled runtime-dependency roots. Runtime-dependency staging should remain in startup, plugin activation, doctor, and explicit plugin dependency repair paths where a sealed stage directory can fail with an actionable runtime error.

## Reproduction Assessment

Yes. The issue provides concrete deployment steps and an exact stack, and current main source confirms the same path from config load through provider policy public-artifact loading into runtime-deps preparation.

## Solution Assessment

Yes. The narrowest maintainable fix is to make provider policy public-artifact loading opt out of runtime-deps installation during config/default loading, then add regression coverage proving config materialization does not mkdir or install runtime deps for those lightweight policy artifacts.
