## Summary

Keep open. Current main still has a source-backed path from config load into bundled provider public-artifact loading, and that loader defaults to preparing bundled runtime dependencies under the configured stage directory before config load completes. The report is recent, concrete, and not paired with an implementation PR in the provided context.

## What This Changes

The issue requests that gateway config loading on v2026.4.27 not fail with EACCES when a sealed Docker deployment makes `/var/lib/openclaw/plugin-runtime-deps` non-writable.

## Best Possible Solution

Keep the issue open and fix the control-plane boundary so config validation/default materialization can load lightweight provider policy artifacts without creating or repairing bundled runtime-dependency roots. Runtime-dependency staging should remain in startup, doctor, plugin activation, and explicit `plugins deps` repair paths, with a clear failure there if the stage root is intentionally sealed.

## Reproduction Assessment

Yes. The issue provides a concrete Docker/WSL2 repro and exact EACCES stack, and current source inspection confirms the same path from config load through provider policy public-artifact loading into runtime-deps preparation. I did not execute the live container repro because the review was read-only.

## Solution Assessment

Yes, keeping this open is the right outcome. The narrowest maintainable fix is to make provider policy public-artifact loading opt out of runtime-deps installation during config defaults, then add a regression test proving config materialization does not call the runtime-deps installer for those lightweight policy artifacts.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/74971. On v2026.4.27/current main, gateway config loading can call bundled provider policy surfaces during `applyModelDefaults`; `src/plugins/provider-public-artifacts.ts` loads `provider-policy-api.js` without `installRuntimeDeps: false`, so the public-surface loader prepares bundled runtime deps under `OPENCLAW_PLUGIN_STAGE_DIR` and sealed Docker deployments fail with EACCES before config load completes. Keep the fix narrow: make config/default provider policy artifact loading read-only, likely by passing `installRuntimeDeps: false` from `tryLoadBundledProviderPolicySurface`, or an equivalent generic seam that ensures `materializeRuntimeConfig`/`applyModelDefaults` cannot mkdir/install runtime deps. Do not disable runtime-deps staging for plugin activation, gateway startup, doctor, `plugins deps`, or public artifacts that explicitly need runtime dependencies. Add regression coverage using a packaged public artifact fixture, mock, or non-writable-stage simulation proving provider policy loading during config defaults does not call `prepareBundledPluginRuntimeRoot`/the runtime-deps installer. Preserve existing public-surface loader behavior for callers that do not opt out. Add a single-line CHANGELOG fix entry under the active version with `Thanks @eurojojo`; no docs change is needed unless the runtime-deps contract changes.
