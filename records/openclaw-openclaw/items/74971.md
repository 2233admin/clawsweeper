## Summary

Keep open. Current main still lets config/default materialization load bundled provider policy artifacts without opting out of runtime-dependency preparation, so a sealed stage root can still hit the reported EACCES before gateway startup. The report is concrete, recent, and narrow enough for a focused repair PR.

## What This Changes

The issue asks OpenClaw to stop failing config load on sealed Docker gateways when provider policy/default normalization tries to create `/var/lib/openclaw/plugin-runtime-deps`.

## Best Possible Solution

Keep the issue open and fix the control-plane boundary: config validation/default materialization should load lightweight provider policy artifacts without creating or repairing bundled runtime-dependency roots, while startup, plugin activation, doctor, and explicit dependency repair continue to stage runtime deps where appropriate.

## Reproduction Assessment

Yes. The issue provides concrete sealed Docker steps and an exact stack, and current main source confirms the same path from config load through provider policy loading into runtime-deps preparation.

## Solution Assessment

Yes. The narrowest maintainable fix is to make provider-policy public artifact loading opt out of runtime-deps installation during config/default loading, rather than changing global staging behavior or requiring sealed deployments to make the stage root writable during config reads.
