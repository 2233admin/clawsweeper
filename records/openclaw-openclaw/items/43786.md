## Summary

Keep open. Current main still blocks the requested non-loopback `gateway.auth.mode="none"` deployment shape in both runtime config validation and CLI preflight, and the exact earlier fix PR was closed unmerged. Because accepting this request would deliberately change the Gateway auth boundary and a broader gateway-auth RFC is still open, this should go to maintainer security/product review rather than automatic cleanup.

## What This Changes

Request to allow `gateway.auth.mode: "none"` for trusted internal deployments without requiring a trusted proxy or shared-secret gateway auth.

## Best Possible Solution

Maintainers should decide whether private-ingress/no-auth non-loopback operation is an approved Gateway mode. If approved, implement it as an explicit dangerous opt-in with aligned runtime validation, CLI preflight, docs, security audit warnings, and tests; if not approved, keep the current guardrail and document trusted-proxy or shared-secret deployment alternatives.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `resolveGatewayRuntimeConfig` with `gateway.bind="lan"` and `gateway.auth.mode="none"` is currently tested to throw `refusing to bind gateway`, and the CLI preflight mirrors that policy.

## Solution Assessment

Unclear. The requested direct `auth.mode=none` allowance is the user-visible fix, but it is not clearly the best maintainable solution until maintainers settle the Gateway auth boundary and how it relates to the broader #69066 service-identity discussion.
