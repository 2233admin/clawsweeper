## Summary

Keep open. The PR is a useful replacement/follow-up for #68622 and it fixes the earlier caller-supplied `roundOneReply` trust-boundary problem, but the current head is still not merge-ready: GitHub reports it dirty against `main`, an existing Codex review found a delegated fast-completion reply-baseline bug, and the branch predates current-main `sessions_send` safeguards for ACP-child A2A skipping, inter-session prompt marking, and target-scoped nested lanes.

## What This Changes

The PR adds a typed plugin `sessions_send` hook with SDK exports/API baseline updates, invokes hooks before default core handling, strips caller-supplied delegated `roundOneReply`, accepts plugin-owned delegated dispatch metadata, and adds sessions tool regression tests.

## Best Possible Solution

Land a rebased implementation that carries the `sessions_send` hook and trust hardening onto current main, preserves ACP-child A2A skipping plus inter-session provenance and per-target lane behavior, exports/documents the full public SDK hook contract, and then supersedes or closes #68622 after merge.

## Reproduction Assessment

Yes. The PR head and current `run-wait` code provide a high-confidence static reproduction for the delegated fast-completion bug: capture the baseline after a plugin has already produced the assistant reply, then the wait/read helper filters that reply as unchanged.

## Solution Assessment

No. The hook direction and trust-boundary change are plausible, but the best fix needs a clean rebase plus an earlier delegated baseline strategy, preservation of current-main A2A/provenance/lane behavior, and SDK/docs alignment before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Snapshot delegated runs before handing control to hooks:** `src/agents/tools/sessions-send-tool.ts:450-455`
  - body: In delegated mode, this reads the baseline only after `runSessionsSend` returns. A plugin can start and complete the delegated run before returning `waitRunId`, so the baseline can capture the new assistant reply and `waitForAgentRunAndReadUpdatedAssistantReply` will report `reply: undefined`; capture the baseline before plugin dispatch or require/use a trusted plugin reply without filtering it as the baseline.
  - confidence: 0.9
- **[P1] Preserve the ACP-child A2A skip on rebase:** `src/agents/tools/sessions-send-tool.ts:386-388`
  - body: Current main skips `runSessionsSendA2AFlow` when the requester is the parent of a background ACP child and reports `delivery.status="skipped"`; this branch always marks delivery pending and always starts A2A. Rebasing this as-is would reintroduce the parent-child ping-pong path that current main already fixed.
  - confidence: 0.86
- **[P2] Keep inter-session provenance and target lanes:** `src/agents/tools/sessions-send-tool.ts:491-496`
  - body: This rebuilt `sendParams` block uses the raw message and `AGENT_LANE_NESTED`, while current main annotates inter-session messages and scopes nested lanes with `resolveNestedAgentLaneForSession(resolvedKey)`. Preserve those current-main fields so inter-session prompts remain marked and different target sessions do not block each other.
  - confidence: 0.84
- **[P3] Export the nested sessions_send hook types:** `src/plugin-sdk/core.ts:102-106`
  - body: The new public event/result types expose `PluginHookSessionsSendTask` and `PluginHookSessionsSendCancelTarget`, but `plugin-sdk/core` only re-exports the context/event/result names. Export the nested task and cancel-target types from the same SDK surface so plugin authors can name the full contract directly.
  - confidence: 0.74
