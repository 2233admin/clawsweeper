## Summary

Keep open. Current main still awaits lane tasks directly with no task-level timeout or `taskTimeoutMs` enqueue option, and the diagnostic recovery path intentionally skips active lane tasks. The provided GitHub context also shows open, unmerged PR #48690 with closing syntax for this issue, so cleanup policy should leave this issue open until that PR or a maintainer replacement lands.

## Best Possible Solution

Keep this issue open and review PR #48690 or land a maintainer-authored equivalent that adds a configurable active-task timeout or recovery path in `src/process/command-queue.ts`, with regression tests proving a never-settling task rejects, clears the active slot, notifies waiters, and lets queued session and cron work drain.
