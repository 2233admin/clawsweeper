## Summary

Keep open. Current main does not yet implement the proactive pre-attempt cron fallback stop, but the PR should not merge as-is because the proposed guard can suppress valid fallbacks for jobs that rely on `agents.defaults.timeoutSeconds` instead of payload-level cron timeouts.

## What This Changes

The PR adds an optional pre-attempt stop hook to model fallback, threads cron deadline metadata into isolated cron execution, and adds tests for skipping fallback attempts when remaining cron budget is too low.

## Best Possible Solution

Land a rebased, narrow fix that preserves current deferred isolated-timeout cleanup semantics, computes the fallback skip threshold from the real per-attempt execution budget, keeps the generic hook only where needed, and locks the behavior with focused model-fallback plus isolated-cron tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Base the fallback budget on the attempt timeout:** `src/cron/isolated-agent/run.ts:986`
  - body: `resolveCronJobTimeoutMs(params.job)` falls back to the 60-minute cron safety ceiling when `payload.timeoutSeconds` is unset, while the model attempt itself uses `prepared.context.timeoutMs` from `resolveAgentTimeoutMs` and `agents.defaults.timeoutSeconds`. With a 5-10s agent default, this guard still requires 30s of remaining outer budget and skips fallbacks that could complete within the real attempt timeout. Compute the threshold from the prepared attempt timeout or pass that value through the executor.
  - confidence: 0.88
