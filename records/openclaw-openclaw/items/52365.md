## Summary

Keep open. The production failure mode is still plausible on current main, but this branch should not merge as-is because the fallback guard uses the outer cron safety timeout instead of the prepared per-attempt execution budget, and the deadline plumbing is stale against current deferred timeout semantics.

## What This Changes

The PR adds a model-fallback pre-attempt stop hook, threads cron deadline metadata into isolated cron execution, skips low-budget fallback attempts, and adds cron/model-fallback regression tests.

## Best Possible Solution

Land a rebased, narrow fix that keeps the generic pre-attempt stop seam, derives the fallback skip threshold from the prepared per-attempt execution budget, computes the propagated cron deadline from the current execution-start lifecycle, and preserves explicit stop errors when no real candidate failure exists.

## Reproduction Assessment

Yes. The PR body includes a concrete production log where the primary consumes nearly all cron budget and the fallback is killed at 0ms; current-main source inspection confirms there is still no pre-attempt fallback veto in the cron path.

## Solution Assessment

No. The proposed direction is useful, but the current patch is not the best mergeable solution because it bases the budget on the outer cron safety timeout and does not integrate with current deferred isolated-timeout semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Base the fallback budget on the attempt timeout:** `src/cron/isolated-agent/run.ts:988-992`
  - body: This uses resolveCronJobTimeoutMs(params.job), which falls back to the 60-minute agent-turn safety timeout when payload.timeoutSeconds is unset. The actual model attempt uses prepared.context.timeoutMs, so a job with a 5-10s agents.defaults.timeoutSeconds can incorrectly require 30s remaining and skip a fallback that still fits the real attempt budget.
  - confidence: 0.88
- **[P2] Start the propagated deadline when execution starts:** `src/cron/service/timer.ts:89-93`
  - body: Current main defers isolated cron timeouts until onExecutionStarted so lane wait does not consume execution budget. Computing deadlineAtMs before executeJobCore lets queue wait drain the propagated fallback budget and can skip fallbacks before the job has had real execution time.
  - confidence: 0.84
- **[P2] Preserve the stop reason when no candidate failed:** `src/agents/model-fallback.ts:915-916`
  - body: When beforeAttempt stops after only cooldown/auth skips, lastError is still undefined. This break falls through to the synthesized fallback summary, dropping the hook's explicit stop message that cron budget checks are supposed to surface.
  - confidence: 0.78
