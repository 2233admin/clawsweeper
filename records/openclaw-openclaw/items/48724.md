## Summary

Keep open. Current main has not implemented the PR’s core security hardening: Docker setup still prints the gateway token, the default Compose ports are still published without a loopback host binding, the CI workflow still uses mutable action tags, and the added secure Compose file is not present. The PR is also not ready as-is because review comments identified portability and usability regressions in its diff.

## Best Possible Solution

Keep the PR open but require a rebase and focused rewrite: move token masking to scripts/docker/setup.sh with tests, decide whether default Docker publishing should remain LAN/Tailscale-friendly or become loopback-only and update docs accordingly, pin or intentionally leave CI actions under a documented policy using current action versions, and drop or fix docker-compose.secure.yml before considering merge.

## Risks / Open Questions

- Security review: this PR touches CI action execution and Docker network exposure, so it should not be merged without updating the action pins to current refs and validating the intended Docker exposure model.
- The PR’s added docker-compose.secure.yml would ship user-specific paths and a host-specific VPN IP unless repaired, making the new hardening file non-portable and likely broken outside the author’s machine.
- Masking the token in stdout is still worthwhile, but the health-check command needs a self-contained safe workflow, such as reading from the generated env file or telling the user how to export the token without printing it.
