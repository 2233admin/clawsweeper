## Summary

Keep this PR open, but it still needs changes before merge. The patch adds a delivery-confirmation result to the block reply pipeline, yet current production callback, dispatcher, and Telegram delivery contracts still return no sent-media metadata, so the new path is not wired through the surface that needs it.

## What This Changes

The PR changes the block reply pipeline to accept an optional `{sentMediaUrls}` result from `onBlockReply` and adds unit coverage for confirmed, empty-confirmed, and legacy media tracking behavior.

## Best Possible Solution

Keep the delivery-aware seam, but make it end-to-end: extend the public callback and dispatcher delivery contracts to return confirmed sent media, have Telegram report the media URLs it actually delivered, preserve final media when block delivery skipped it, dedupe media when block delivery confirmed it, and cover both core payload and Telegram regression paths with tests plus a changelog entry.

## Reproduction Assessment

Yes. The linked bug has concrete Telegram steps, and the source-level path is clear: block streaming emits mixed text plus media, final payload construction consults `getSentMediaUrls()`, and incorrect tracking either strips an unsent attachment on main or, under this PR, lacks production delivery confirmation for already-sent media.

## Solution Assessment

No. The optional result type is a reasonable building block, but this PR only consumes it inside the pipeline and tests; the maintainable fix needs to propagate confirmed delivery metadata through the production callback, dispatcher, and Telegram delivery layer.
