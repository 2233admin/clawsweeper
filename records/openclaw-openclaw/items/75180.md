## Summary

Keep this PR open. It targets a real Telegram mixed text+media delivery bug, but the current patch still uses text presence as a proxy for whether media was delivered; that can preserve the same final media after a successful mixed block send and resend the attachment.

## What This Changes

The PR changes block reply sent-media tracking so only media-only block payloads are recorded for final-reply media dedupe, and updates the block pipeline unit expectation.

## Best Possible Solution

Make block media tracking delivery-aware rather than text-shape-aware: media that was actually sent by the block path should still suppress final media, while media that only appeared beside previewed or text-only streamed content should survive into the final reply. Add regression coverage across buildReplyPayloads and the Telegram lane/media delivery path.

## Reproduction Assessment

Yes. A source-level reproduction is a successful block-streamed payload with both text and media followed by final payload construction for the same text+media reply; the PR leaves getSentMediaUrls empty for that successful mixed block send, so the final media can be delivered again.

## Solution Assessment

No. The current PR fixes the reported drop for one path by not tracking mixed media, but that is not the narrowest maintainable fix because it loses the duplicate guard for mixed media that was actually delivered.
