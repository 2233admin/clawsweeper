## Summary

Keep open. The PR adds a useful delivery-confirmation seam, but current main and the PR diff do not wire that seam through the production reply dispatcher or Telegram delivery path, so real callbacks still return void and the new media tracking falls back to media-only heuristics. The branch also lacks the required changelog entry for a user-facing reply/media fix.

## What This Changes

The PR changes the block reply pipeline to accept an optional `{sentMediaUrls}` result from `onBlockReply` and adds unit coverage for confirmed, empty-confirmed, and legacy media-only tracking behavior.

## Best Possible Solution

Keep the delivery-aware seam, but make it real end-to-end: extend the callback/dispatcher contract to return confirmed sent media from production delivery, have Telegram block delivery report the media URLs it actually sent, preserve final media when block delivery skipped it, dedupe media when block delivery confirmed it, and cover that behavior with core payload and Telegram regression tests plus a changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is a production-style void `onBlockReply` that enqueues or sends a mixed text+media block and then builds the final payload for the same reply; under this PR the mixed block uses the legacy media-only fallback, so `getSentMediaUrls()` stays empty and final media dedupe cannot know the block path already delivered the attachment.

## Solution Assessment

No. The optional result type is a reasonable building block, but this PR only consumes it in the pipeline and tests; it does not connect the confirmation result to the production callback, dispatcher, or Telegram delivery path that would make the behavior reliable.
