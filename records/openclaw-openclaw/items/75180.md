## Summary

Keep open. The updated PR moves toward delivery-aware media tracking, but the new confirmation result is not wired through the production block-reply callback path, so current Telegram mixed text+media block sends can fall back to untracked media and then resend the same final attachment. It also still lacks the required user-facing changelog entry.

## What This Changes

The PR changes the block reply pipeline to accept an optional `{sentMediaUrls}` result from `onBlockReply` and adds unit coverage for confirmed, empty, and legacy media tracking behavior.

## Best Possible Solution

Thread delivery-aware media confirmation through the real block-reply delivery path, not only the test-only factory callback: update the callback contract, propagate confirmed sent media from the dispatcher/channel path, preserve media that was not sent, suppress media that was sent, add regression coverage across final payload construction and Telegram lane delivery, and add a single-line Unreleased Fixes changelog entry.

## Reproduction Assessment

Yes. A source-level reproduction is a production-style void `onBlockReply` that successfully sends or enqueues a mixed text+media block, followed by final payload construction for the same text+media reply; under this PR, `getSentMediaUrls()` stays empty for that mixed block and the final attachment can be delivered again.

## Solution Assessment

No. The optional result type is a reasonable seam, but this PR does not connect it to the production callback/dispatcher path that Telegram uses, so the actual behavior still depends on the legacy fallback rather than delivery confirmation.
