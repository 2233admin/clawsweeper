## Summary

Keep this PR open. Current main still lacks the proposed provider/auth/config surface, and the PR branch is not merge-ready because the new core stream path misses existing Google transport behavior, leaves at least one runtime surface unable to use the keyless auth mode, and moves Google SDK execution across the current plugin dependency boundary.

## What This Changes

Adds a new `google-genai` provider path backed by `@google/genai`, a `google-genai-sdk` auth mode, Vertex project/location config, live tests, docs, and embedded-runner routing for the new stream function.

## Best Possible Solution

Revise this work so unified Google GenAI and Vertex auth live within the bundled Google plugin/provider architecture or replace it with narrower fixes for the current `google-vertex` ADC/proxy path, while preserving existing transport parity, guarded network policy, dependency ownership, docs, and focused regression coverage.

  - body: `model.vertexai` is read here, but provider-level `models.providers.google-genai.vertexai` is never copied into the runtime model by the existing resolver.   - confidence: 0.9

  - body: This new mode returns auth with no API key, but only the embedded runner checks were updated. `/btw` still exempts only `aws-sdk` before calling `requireApiKey`, so a `google-genai-sdk` model fails before the new stream can run.
  - confidence: 0.91
- **[P1] Declare the SDK where core imports it:** `src/agents/google-genai-stream.ts:1`
  - body: This core file imports `@google/genai`, but the root package does not declare that dependency; it is currently owned by `extensions/google/package.json`. In a published install, the core `dist/agents` import should not rely on a plugin-local dependency path.
  - confidence: 0.86
- **[P2] Forward tools and run options to the SDK:** `src/agents/google-genai-stream.ts:96-100`
  - body: The request passed to `generateContentStream` only includes `model`, `contents`, and an optional system instruction. It drops `context.tools`, tool choice, max-token/temperature controls, and the abort signal, so tool-dependent runs cannot call functions and caller limits/cancellation are ignored.
  - confidence: 0.92
- **[P2] Route SDK calls through provider request policy:** `src/agents/google-genai-stream.ts:40-54`
  - body: Creating the SDK client directly here bypasses the existing provider transport policy used by the current Google stream, including guarded fetch, configured proxy/dispatcher behavior, timeout handling, and request capture. The new network path should preserve those controls or remain inside a plugin-owned runtime seam that explicitly handles them.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff introduces direct Google SDK network execution from core and crosses the current plugin-local dependency boundary.

Concerns:

- **[medium] Direct SDK calls bypass provider network policy:** `src/agents/google-genai-stream.ts:40`
  - body: `new GoogleGenAI(...)` and `ai.models.generateContentStream(...)` use the SDK's internal network stack instead of OpenClaw's guarded provider fetch path, so configured proxy/dispatcher, timeout, capture, and network policy controls may not apply to this provider.
  - confidence: 0.84
- **[medium] Core import relies on plugin-local dependency:** `src/agents/google-genai-stream.ts:1`
  - body: The new core stream imports `@google/genai`, but current main declares that package only in `extensions/google/package.json`; this changes dependency ownership and can break published package resolution unless the dependency is deliberately moved or the runtime stays plugin-owned.
  - confidence: 0.87
