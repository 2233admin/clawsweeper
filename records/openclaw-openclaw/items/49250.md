## Summary

Current main still does not implement the requested prompt/heartbeat race UX. The Control UI queue is tied to locally owned `chatRunId` state, heartbeat/other-run deltas can still be ignored while a prompt-owned run is active, and the backend follow-up queue only preserves execution ordering without exposing the requested visible queued-prompt/cancel-inject state in the UI. Related #49252 is closed and PR #49261 was closed unmerged, so neither is an open canonical fix for the remaining behavior.

## Best Possible Solution

Keep this issue open and implement a session-scoped active-run plus queued-follow-up surface shared by Gateway chat events and the Control UI. When a heartbeat is active and a user submits a prompt, the UI should keep rendering heartbeat/tool progress, show the prompt as queued or pending with cancel/steer/inject controls, and drain or inject it after the heartbeat reaches a safe boundary. Related #49252 and closed-unmerged PR #49261 should be treated as context, not as the remaining tracker.
