## Summary

Keep open. Current main still intentionally blocks normal Gateway startup when `gateway.mode=local` is missing, while `--allow-unconfigured` and Crestodian cover only bootstrap/repair paths. The reported no-provider WebUI crash and Raw mode failure still lack concrete reproduction data, so this is not safe to close as implemented or stale.

## What This Changes

The issue requests that Gateway and WebUI startup tolerate incomplete configuration and missing provider/model setup instead of crashing or blocking startup.

## Best Possible Solution

Keep the existing recovery/bootstrap paths, then have maintainers decide whether normal loopback Gateway startup should get a safer configless mode or keep the `gateway.mode` guard. Split any remaining work into focused regressions for no-provider WebUI startup and the exact Raw mode snapshot state once those are reproducible.

## Reproduction Assessment

Partially. The missing-`gateway.mode` Gateway startup block is high-confidence from source and tests, but the no-provider WebUI crash and Raw mode failure do not have enough config, route, browser, or log detail to reproduce confidently.

## Solution Assessment

Unclear. A broad automatic startup relaxation is not clearly the best fix because the current guard is documented and tested as intentional; the safer path is maintainer triage plus targeted UI regressions only after concrete reproduction.
