## Summary

Keep open. Current main has bootstrap/recovery paths and safer configured-model fallbacks, but normal Gateway startup still intentionally blocks missing or clobbered `gateway.mode`, and the reported no-provider WebUI crash plus Raw mode failure lack concrete reproduction data. That combination is not a policy-valid cleanup close.

## What This Changes

The issue requests that Gateway and Control UI start usefully without complete configuration or a configured model/provider, and adds that Raw mode is currently not working.

## Best Possible Solution

Define an explicit maintainer-approved startup contract: either keep the current guarded normal Gateway startup and improve setup/recovery messaging, or add a loopback-only degraded/bootstrap Gateway and Control UI state with regression coverage for missing config, no configured provider, and Raw-mode-safe snapshots.

## Reproduction Assessment

Unclear. The Gateway guard portion is source- and test-reproducible on current main, but the WebUI no-provider crash and Raw mode failure do not have enough concrete steps or logs for a high-confidence reproduction.

## Solution Assessment

Unclear. A blanket removal of the startup guard is not clearly the best fix because current code and docs treat missing `gateway.mode` as suspicious config damage; a narrower degraded bootstrap path would need maintainer product/security approval first.
