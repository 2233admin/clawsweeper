## Summary

Keep this PR open. The work targets a real memory/plugin-runtime gap, but the current head introduces a blocking plugin-tool regression during in-flight loads and is dirty/diverged from current main, so it needs author or maintainer follow-up before merge.

## What This Changes

The PR adds direct memory-wiki fallback paths for memory-core tools, exports a memory-wiki API subpath, changes plugin tool resolution toward fresh non-activating snapshot loads, preserves more memory plugin state during snapshots, and adds targeted memory/plugin tests plus a local pre-PR script.

## Best Possible Solution

Keep the PR open, rebase it onto current main, preserve current memory tool live-config behavior, and update the tool-registry path so an in-flight snapshot load falls back to a compatible active registry when one exists before degrading.

## Reproduction Assessment

Yes. A source-level reproduction is to have a compatible active plugin registry, make isPluginRegistryLoadInFlight return true for the same load options, and call resolvePluginTools; the PR head returns [] because src/plugins/tools.ts returns undefined before active-registry compatibility is checked.

## Solution Assessment

No. Fresh non-activating snapshot loads may be a reasonable direction for stale tool closures, but dropping all plugin tools during concurrent equivalent loads is not the narrowest maintainable behavior; the safer path is to reuse a compatible active registry as the fallback.
