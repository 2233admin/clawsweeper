## Summary

Keep open. Current main has partial context-window capping, but it still preserves precedence in important paths: configured model context can beat discovery, per-agent session persistence can return the override directly, and Anthropic maxTokens can forward a requested value above the model cap.

## What This Changes

The issue requests resolving effective context and max token limits as the minimum across agent config, provider model config, model discovery, and defaults.

## Best Possible Solution

Add a shared token-limit resolver that collects positive finite candidates from agent defaults/per-agent config, provider/model config, discovered runtime metadata/cache, and fallback defaults, then chooses the conservative minimum while preserving provider-qualified disambiguation and explicit long-context opt-ins. Apply the same pattern to output maxTokens where request params meet model/provider limits, with regression tests and docs updates for the contract.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: construct config where a provider model contextWindow is larger than discovered runtime metadata, or pass per-agent contextTokens through session persistence, and current code returns the higher override before lower model limits in those paths.

## Solution Assessment

Yes, with care. The proposed minimum-based behavior is the right safety default, but it should be implemented through normalized candidate collection that preserves the existing provider-qualified collision fixes and Anthropic context1m exceptions.
