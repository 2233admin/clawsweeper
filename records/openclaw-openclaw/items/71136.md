## Summary

Keep open. Current main still has partial capping but not the requested conservative minimum behavior: several paths return configured or per-agent values before lower model/discovery limits, and direct Anthropic maxTokens can still exceed the model cap.

## What This Changes

The issue requests resolving effective contextTokens and maxTokens as the minimum of agent config, provider/model config, discovery metadata, and defaults.

## Best Possible Solution

Add a shared token-limit resolver that gathers positive finite candidates from agent defaults/per-agent config, configured provider/model metadata, discovered runtime/cache metadata, and fallback defaults, then chooses the conservative effective minimum while preserving provider-qualified disambiguation and explicit long-context opt-ins. Apply the same candidate-based approach to output maxTokens at transport/request boundaries and update tests/docs to make the contract clear.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: set a higher agent or configured model context value with lower discovered/runtime model metadata, or pass an Anthropic maxTokens request above model.maxTokens, and current main returns the higher configured/requested value in those paths.

## Solution Assessment

Yes, with guardrails. The minimum-based behavior is the right narrow safety fix, but it should be implemented through normalized candidate collection rather than replacing provider-qualified discovery lookup or Anthropic long-context exceptions with a bare global minimum.
