## Summary

Keep open. Current main still has precedence-based token-limit resolution in several active paths, including session persistence, context-window guarding, model metadata normalization, docs, and at least one provider maxTokens resolver, so the requested conservative minimum behavior is not implemented end to end.

## What This Changes

The issue requests resolving effective contextTokens and maxTokens as the minimum positive value available from agent config, provider/model config, model discovery/runtime metadata, and defaults.

## Best Possible Solution

Add a shared positive-finite token-limit candidate resolver for contextTokens and maxTokens, use it in the context resolver, context-window guard, session-store persistence, embedded-runner model normalization, model merge/runtime metadata, and provider request paths, then update tests and public config docs to define the conservative minimum contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: configure or pass a contextTokens/maxTokens value above a lower model/discovery/runtime limit, and current main still has paths that return the higher configured/requested value before the lower cap is considered.

## Solution Assessment

Yes. The requested minimum-of-available-limits direction is the narrow maintainable fix, but it should be centralized through shared candidate collection rather than scattered Math.min calls so provider-qualified lookup and context1m exceptions stay correct.
