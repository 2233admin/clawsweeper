## Summary

Keep this PR open. Current main still emits the bare `path-not-allowed` message and lacks the proposed regression coverage, so the PR is not obsolete; the remaining maintainer decision is whether exposing configured filesystem roots in a shared error message is acceptable.

## What This Changes

The PR adds a bounded allowed-roots formatter to `assertLocalMediaAllowed`, appends that hint to both `path-not-allowed` throw sites, and adds short-list and overflow regression tests.

## Best Possible Solution

Keep the focused diagnostic improvement open, but land it only after maintainers accept or adjust the path-disclosure behavior; a safer version would keep the existing error code and rejected path while gating full root lists to operator-local diagnostics or using sanitized root labels where errors can cross channel/tool boundaries.

## Reproduction Assessment

Yes. On current main, creating one allowed root and calling `assertLocalMediaAllowed('/var/elsewhere/file.png', [allowedRoot])` reaches the final `path-not-allowed` throw with only the rejected path in the message; the workspace fast-path branch has the same bare shape.

## Solution Assessment

Unclear. The PR is the narrowest code change for operator-facing diagnostics, but because the same `LocalMediaAccessError.message` can be serialized through RPC/tool error handling, maintainers should explicitly decide whether raw configured roots belong in that shared message.
