## Summary

Keep open. Current main still drops cron wake targeting before the call reaches CronService, while the lower Gateway cron adapter already knows how to resolve agentId/sessionKey when those fields are supplied. The related hook-routing issue and open PR are adjacent webhook work, not a fix for the direct cron tool/Gateway wake path.

## What This Changes

Request optional agentId/sessionKey support on the cron tool wake action so wake events can target a non-default agent/session instead of always falling back to main.

## Best Possible Solution

Land a focused additive plumbing fix that threads optional agentId and sessionKey through the cron tool wake schema/execution path, Gateway WakeParams/handler, CronServiceContract, CronService.wake, ops.wakeNow, and timer.wake, reusing the existing Gateway cron target resolver and keeping no-target behavior unchanged.

Yes. A high-confidence code-level reproduction exists: invoke the cron tool wake action with agentId/sessionKey and current main will accept the extra fields at the outer schema boundary but forward only mode/text before enqueueing an unscoped system event and heartbeat.

## Solution Assessment

Yes. The narrow maintainable fix is to plumb the optional target fields through existing cron wake contracts and add regression coverage at each boundary, not to create a new routing mechanism.
