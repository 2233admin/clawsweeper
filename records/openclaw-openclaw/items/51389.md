## Summary

Keep this PR open. Current main has broadened model fallback behavior, including several HTTP 400/422-classified paths and even unrecognized errors while candidates remain, but it does not implement the PR's intended configurable `fallbackOnErrors` surface. The config type, strict Zod schema, generated schema/help, resolver plumbing, and `runWithModelFallback` parameters on main still have no `fallbackOnErrors` support.

## Best Possible Solution

Keep the PR open for maintainer/product review. The best path is to rebase or replace it with a current-main implementation that adds `fallbackOnErrors` only if OpenClaw wants this public config surface, extends the strict config schema and generated help/docs, wires every chat fallback call path, preserves current default behavior, fails fast on disallowed custom statuses, and adds focused tests for `default`, `all`, custom arrays, queued follow-ups, and any image/PDF/non-chat model scope decision.
