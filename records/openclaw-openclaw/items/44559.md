## Summary

Keep this issue open. Current main still implements native Windows gateway management as a Scheduled Task plus per-user Startup-folder fallback, not as a native service or non-interactive Windows supervisor with evidence that it survives the reported PowerShell/session lifecycle. The newer Windows hardening on main improves fallback/restart behavior, but it does not remove `/SC ONLOGON` or `/IT`, and the docs still recommend WSL2 for the fuller/stable path while describing native Windows as Scheduled Task/login-item based.

## Best Possible Solution

Keep this open for Windows gateway service design. The best fix is either to add a supported native Windows supervisor mode that keeps the Gateway reachable after closing the initiating PowerShell window, with regression/manual Windows proof, or to explicitly document native Windows as login-session scoped and steer headless/persistent installs to WSL2/systemd. If the intended scope is only terminal-close survival rather than logout/headless survival, the fix should prove that `openclaw gateway install` starts through Task Scheduler or a detached launcher that is not tied to the PowerShell process tree.
