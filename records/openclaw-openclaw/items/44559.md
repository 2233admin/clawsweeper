## Summary

Keep this issue open. Current main still creates the native Windows gateway Scheduled Task as a logon-oriented task with `/SC ONLOGON`, `/RL LIMITED`, and `/RU <user> /NP /IT` when a task user is available, and the documented fallback is a per-user Startup-folder login item. The issue discussion remains aligned with current source/docs, and the related Microsoft tracker PR is closed unmerged. An adjacent open PR (#68149) may be relevant to Windows ScheduledTasks behavior, but it is not merged and still needs maintainer/security review rather than making this issue obsolete.

## Best Possible Solution

Keep this open for maintainer product/design handling. The best path is to decide the supported native Windows persistence model, then either add a tested service/non-interactive Scheduled Task mode that survives closing PowerShell/session with documented privilege and security tradeoffs, or explicitly document native Windows gateway install as login-session scoped and point headless users to WSL2/systemd. #68149 is a relevant implementation candidate to review, but it needs targeted Windows validation and security review before it can close this issue.
