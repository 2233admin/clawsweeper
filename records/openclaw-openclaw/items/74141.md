## Summary

Keep open. Current main still has the reported `session_status` resolution gap, and the issue is paired with open implementation PRs that use closing syntax, so it should remain open until maintainers review and land one canonical fix or choose a different approach.

## What This Changes

The issue asks for `session_status` to resolve `sessionKey="current"` to the requester session for channel-plugin agents instead of throwing `Unknown sessionKey: current` and causing retry latency.

## Best Possible Solution

Land one narrow `session_status` fix that falls back from explicit literal `current` to the requester session only after literal key and sessionId lookup fail, with regression coverage for channel-style requester keys and the existing literal-current precedence tests left intact.

## Reproduction Assessment

Yes. By code inspection, a channel-style requester with explicit `sessionKey="current"` and no matching literal/current/requester store entry follows the reported path to `Unknown sessionKey: current`; the reporter also supplied concrete gateway logs for that path.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix: keep the shared resolver semantics intact, add the `session_status` fallback after existing literal-current resolution, and validate the visibility/self-target handling with focused tests.
