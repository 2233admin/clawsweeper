## Summary

Keep open. Current main still has the reported contract mismatch in the `session_status` resolution path, and two open unmerged PRs already explicitly target the fix, so cleanup should happen by reviewing and landing or closing the canonical PR rather than closing this issue first.

## What This Changes

Make `session_status` resolve explicit `sessionKey="current"` to the caller session for channel-plugin agents without breaking literal `current` session key or sessionId precedence.

## Best Possible Solution

Keep this issue open until one canonical fix PR lands. The implementation should add a narrow explicit-`current` requester fallback after literal `current` key/sessionId lookup, preserve visibility checks as self, and add a focused channel-style requester regression test.

## Reproduction Assessment

Yes. The report gives a concrete channel-style requester key and logs, and current main has a static reproduction path where explicit `sessionKey="current"` can miss store/gateway resolution and fall through to `Unknown sessionKey`.

## Solution Assessment

Yes. The proposed final fallback in `session-status-tool.ts` is the narrowest maintainable fix because it preserves existing literal `current` precedence while making the documented alias work for channel-plugin requesters.
