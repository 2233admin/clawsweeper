## Summary

Keep open. Current main still exposes the reported `session_status` mismatch for explicit `sessionKey="current"` from channel-plugin requester sessions, and open PRs #74142 and #72306 already target the issue with closing syntax, so the issue should stay open until one canonical fix lands or both PRs are closed.

## What This Changes

The issue requests that `session_status` resolve explicit `sessionKey="current"` to the requester session for channel-plugin agents instead of throwing `Unknown sessionKey: current`.

## Best Possible Solution

Keep the issue open until one canonical fix PR lands. The end state should add a narrow explicit-`current` requester fallback in `session-status-tool.ts` after literal `current` key/session-id precedence, preserve self-visibility handling, and add a focused channel-style requester regression test.

## Reproduction Assessment

Yes. The report gives a concrete channel-style requester key and gateway log sequence, and current main has a static path where explicit `sessionKey="current"` can miss store/gateway resolution and fall through to `Unknown sessionKey`.

## Solution Assessment

Yes. The proposed final fallback after literal `current` resolution is the narrowest maintainable fix because it preserves existing literal-session behavior while making the documented alias work for channel-plugin requesters.
