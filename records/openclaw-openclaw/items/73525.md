## Summary

Keep open. Current main still has the reported split between the fast `openclaw status` Channels table and the live `channels.status` account payload, and the linked closing PR #73662 remains open rather than merged.

## What This Changes

`openclaw status` should populate its Channels table consistently with configured/running channels shown by `openclaw channels status` instead of rendering a header-only table.

## Best Possible Solution

Resolve this by reviewing and landing #73662 or an equivalent narrow fix that augments missing visible Channels rows from the existing gateway status snapshot while preserving the fast read-only path and unreachable-gateway fallback behavior.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, a second commenter confirmed the same behavior across Discord and Telegram, and current main shows the source-level split that reproduces the empty table whenever read-only discovery yields no rows while live `channelAccounts` has running accounts.

## Solution Assessment

Yes for the direction: filling missing `openclaw status` rows from the already-fetched gateway snapshot is the narrow maintainable fix. The specific #73662 state/detail mapping should still be reviewed before merge.
