## Summary

Keep open. Current main still has a real status-rendering gap: the default status path can build the visible Channels table from read-only discovery only, while the dedicated channels status command renders live gateway account rows.

## What This Changes

The issue requests that `openclaw status` populate its Channels table when configured/running channels are visible through `openclaw channels status`.

## Best Possible Solution

Resolve this by landing #73662 or an equivalent narrow fix that fills missing Channels rows from the existing gateway `channels.status` snapshot while preserving the fast default status behavior and unreachable-gateway fallback.

## Reproduction Assessment

Yes. The issue has concrete CLI steps, another commenter confirmed the same split with Discord and Telegram, and current main still has the code path where default status can render no rows while `channels.status` has live account data.

## Solution Assessment

Not applicable as a merged solution yet. The narrow status overview/channel-table merge proposed by #73662 appears to be the right direction, but it is still open and unreviewed on main.
