## Summary

Keep open. Current main still has the reported split: plain `openclaw status` builds the visible Channels table from read-only discovery without merging the gateway channel-account snapshot, while `openclaw channels status` renders those gateway account rows. The issue also has open closing PR #73662, so the right maintainer path is to review or land that PR or an equivalent fix before closing.

## What This Changes

The issue requests that `openclaw status` populate its Channels table with configured/running channel accounts when `openclaw channels status` can see them.

## Best Possible Solution

Preserve the fast status contract while making the Channels table truthful when the Gateway already reports running channel accounts; #73662 or an equivalent narrow fix should add that bridge with coverage for plain status, `--deep`, and unreachable-Gateway fallback behavior.

## Reproduction Assessment

Yes. The issue gives concrete command steps, a second user confirmed the same Discord/Telegram behavior, and current-main source/tests show the same split between the read-only status table and gateway-backed channels status output.

## Solution Assessment

Unclear until #73662 is reviewed. Synthesizing missing rows from the existing gateway snapshot is a plausible narrow solution, but maintainers should confirm the chosen augment behavior preserves richer read-only rows and handles missing or unreachable gateway cases correctly.
