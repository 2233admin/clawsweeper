## Summary

Keep open. Current main still has the split reported here: `openclaw channels status` renders live `channels.status` account rows, while default `openclaw status` builds the visible Channels table from read-only config/plugin discovery and only uses the live payload for warning overlays. The #72993 changelog entry is adjacent but narrower: it keeps status fast and shows configured Telegram metadata, not full parity with the live account snapshot.

## Best Possible Solution

Fix the CLI status owner path so, when the gateway is reachable, the visible `openclaw status` Channels table is populated from the same `channels.status` account snapshot used by `openclaw channels status`, while retaining the current read-only config/plugin-discovery table as the unreachable-gateway fallback. Add a regression test where `channels.status` returns Telegram account rows and default `openclaw status` renders a non-empty Channels table from them.
