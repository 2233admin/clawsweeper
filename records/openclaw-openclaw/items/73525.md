## Summary

Keep open. Current main still has a real split between the two surfaces: `openclaw channels status` renders the live `channels.status` gateway payload, while `openclaw status` fetches that payload only for issues and builds the visible Channels table separately from config/read-only plugin discovery. That is not strong enough for an implemented-on-main close, especially for a same-day regression report against 2026.4.26.

## Best Possible Solution

Keep this issue open and fix the CLI status owner path. The best implementation is to make `openclaw status` build its visible Channels rows from the same `channels.status` gateway snapshot used by `openclaw channels status` when the gateway is reachable, while retaining the current read-only config/plugin-discovery table as the fallback for unreachable gateways. Add a regression test where `channels.status` returns Telegram account rows and `openclaw status` renders at least one Channels table row instead of a header-only table.
