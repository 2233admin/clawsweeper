## Summary

Keep open. Current main still routes direct and mapped hook-agent requests through an isolated cron-style job, which forces a fresh session id even when the caller supplies the same stable hook session key. The reusable session helper exists, but the hook payload, mapping, transform, and strict config schema surfaces still lack an explicit persistent hook-session mode; the related PRs are closed unmerged and remain prior art rather than an implemented fix.

## What This Changes

The issue requests persistent multi-turn webhook hook sessions when callers provide a stable session key.

## Best Possible Solution

Add an explicit `sessionMode: "isolated" | "persistent"` option for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep isolated behavior as the default, update config types/schema/generated docs/user docs, and only pass `forceNew: false` for the explicit persistent hook-agent path so stable hook session keys can reuse fresh existing session ids without weakening cron isolation.
