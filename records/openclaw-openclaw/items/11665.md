## Summary

Keep open. Current main still routes hook-agent requests through an isolated cron-style job, so a stable hook `sessionKey` reaches the session resolver with `forceNew: true` and cannot reuse the stored transcript session. The related implementation PRs are useful prior art, especially #27193, but they are closed unmerged and do not supersede the remaining work.

## What This Changes

The issue requests multi-turn webhook hook support by reusing an existing transcript session when a stable hook `sessionKey` is supplied.

## Best Possible Solution

Add a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. Keep isolated mode as the default, propagate the mode through dispatch, and only pass `forceNew: false` for explicit persistent hook-agent runs with stable allowed session keys.
