## Summary

Keep open. Current main still routes direct and mapped hook-agent requests through the isolated cron-style runner, so a stable hook `sessionKey` still reaches session resolution with `forceNew: true` and cannot reuse an existing transcript session id. The closed related PRs, especially #27193, are useful prior art but were not merged and do not supersede the remaining work.

## What This Changes

Allow webhook hook agent runs with a stable `sessionKey` to reuse an existing transcript/session id for multi-turn conversations instead of always starting a fresh isolated session.

## Best Possible Solution

Add a documented, backward-compatible hook option such as `sessionMode: "isolated" | "persistent"` for direct `/hooks/agent` payloads, hook mappings, and transform results. The implementation should default to isolated, propagate the explicit mode through dispatch, and set `forceNew: false` only for persistent hook-agent runs with allowed stable session keys.

## Reproduction Assessment

Yes. A direct or mapped hook with a stable `sessionKey` is normalized and dispatched to a synthetic isolated cron job, which calls `resolveCronSession` with `forceNew: true`; the existing tests cover that forced-new resolution creates a new session id despite a fresh stored entry.

## Solution Assessment

Yes. The explicit `sessionMode` opt-in is the narrowest maintainable direction because it enables the requested multi-turn webhook behavior without weakening default hook or cron isolation; automatic reuse based only on key shape would be less safe.
