## Summary

Keep open. The contribution is aimed at a real linked google-vertex ADC regression, but it does not fix the affected chat-completion path: current main still resolves google-vertex ADC to a synthetic marker and sends it through the Google transport auth header builder, while the new Vertex SDK helper is not called from that path.

## What This Changes

The PR adds Google ADC credential parsing, authorized_user refresh-token exchange with an in-memory access-token cache, tests for that helper, and a new async Google Vertex GoogleGenAI factory that injects a Bearer token.

## Best Possible Solution

Finish the google-vertex fix in the actual Google chat/provider auth path: convert authorized_user ADC into a Bearer token or otherwise feed ADC through GoogleAuth before request headers are built, avoid sending the synthetic ADC marker as x-goog-api-key, preserve non-authorized_user ADC fallback behavior, and add regression coverage for the linked Docker authorized_user setup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the ADC helper into the chat path:** `extensions/google/google-genai-runtime.ts:32-34`
  - body: The new createGoogleVertexGenAI helper is never used by the google-vertex chat-completion path identified in the linked bug. Current main's transport still calls getEnvApiKey(model.provider) and buildGoogleHeaders(), so authorized_user ADC users continue through the existing marker/API-key-shaped path while this helper only sits unused in google-genai-runtime.ts. Please wire the token/header logic into the actual google-vertex stream path or the PR will not close the reported regression.
  - confidence: 0.93
- **[P2] Fall through for other ADC credential types:** `extensions/google/adc-credentials.ts:46`
  - body: loadAdcCredentials throws on every ADC type except service_account and authorized_user. If createGoogleVertexGenAI is used as the Vertex ADC entry point, valid ADC configurations such as workload/workforce identity credential configuration files will fail before the Google SDK can handle them. The helper should special-case authorized_user and delegate unknown non-service-account shapes to the existing SDK/GoogleAuth path.
  - confidence: 0.79
