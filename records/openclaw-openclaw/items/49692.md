## Summary

Keep this issue open. Current main improved the restart wording and can drain work that was already accepted into a command queue, but new inbound agent work is still rejected during the gateway drain window before it can be queued and replayed. The remaining auto-retry behavior requested here is not implemented on main.

## Best Possible Solution

Keep this issue open. The best core fix is to accept drain-window inbound agent tasks into preserved FIFO command lanes without starting new work until restart reset drains them, then update the current rejection tests and add regression coverage for multiple messages, per-session/channel ordering, and SIGUSR1 replay. Treat durable full stop/start replay and provider-attribution copy as separately scoped follow-ups.

## Risks / Open Questions

- Closing now would lose the remaining core behavior gap: auto-retry of messages rejected during the gateway drain window is still missing.
- Durable full stop/start replay is a larger design than the in-process SIGUSR1 drain queue and likely needs separate persistence, idempotency, duplicate-delivery, and FIFO ordering decisions.
- The provider-overload attribution request added in comments is adjacent UX work and should be split or tracked separately rather than treated as resolved by the drain-message copy change.
