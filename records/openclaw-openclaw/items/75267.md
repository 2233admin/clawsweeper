## Summary

Keep open. Current main still has the provider-scoped picker leak this PR targets, but the proposed patch introduces actionable regressions around fallback scoping and initial selection, so it needs changes before merge.

## What This Changes

The PR changes default model picker filtering and initial-value handling, and adds regression tests for provider-scoped alias/default-model and allowlist picker behavior.

## Best Possible Solution

Keep the PR focused, but adjust it so configured supplements are hidden only when a preferred-provider scope is actually active, and invalid initialValue fallback prefers a displayed model option rather than a prepended action entry. Add regression coverage for those edge cases before normal maintainer validation.

## Reproduction Assessment

Yes. On current main, configure an alias such as sonnet that resolves to claude-cli/claude-sonnet-4-6, then call promptDefaultModel with preferredProvider: "openai", allowKeep: false, includeManual: false, ignoreAllowlist: true, and an OpenAI catalog row; the current flow can append and preselect the configured Claude value outside the OpenAI rows.

## Solution Assessment

No, not as-is. The direction is narrow and appropriate, but the patch should use the existing hasPreferredProvider distinction for supplement hiding and should choose a displayed model value, not the first option, when clamping initialValue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate supplement hiding on active provider scope:** `src/flows/model-picker.ts:801-806`
  - body: When preferredProvider is set but no catalog rows match it, maybeFilterModelsByProvider leaves the picker unscoped. This new guard still hides the current configured supplement in that case, making a custom/non-catalog current model unselectable. Please only enforce this provider check when hasPreferredProvider is true.
  - confidence: 0.87
- **[P2] Prefer a model option when clamping initialValue:** `src/flows/model-picker.ts:824`
  - body: options[0] can be "Enter model manually" or a provider setup action because those entries are prepended before model rows. If the configured model is hidden, defaulting initialValue to options[0] can send the user into an action path even though valid model choices are displayed. Please fall back to the first displayed model option instead.
  - confidence: 0.84
