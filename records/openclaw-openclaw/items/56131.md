## Summary

Current main still does not expose reset archive chains through `sessions_history`; the tool delegates to `chat.history`, which reads only the current resolved session entry and active transcript candidates. The related #60409 is still open and describes only a partial fallback, so this should stay open for maintainer follow-up.

## What This Changes

Request that `sessions_history` and related session recovery surfaces discover prior `.jsonl.reset.*` fragments and preserve enough provenance for context recovery after a session reset.

## Best Possible Solution

Keep this issue open until the session retrieval contract can return or clearly advertise reset archive fragments for a logical session key, with focused tests for reset archive discovery and a separate maintainer decision on whether task/exec provenance belongs in the same change.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to reset a session so the old transcript becomes `<sessionId>.jsonl.reset.<timestamp>`, then query `sessions_history` for the logical key and observe that current main reads only the new active entry unless an archived file is already explicitly stored as the entry's `sessionFile`.

## Solution Assessment

Yes at the retrieval-layer direction level, but unclear as a single patch scope. The narrow maintainable fix should add archive discovery/metadata to the session history path first, then split exec completion provenance if that needs a separate task/runtime contract.
