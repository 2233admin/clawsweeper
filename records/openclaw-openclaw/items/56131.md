## Summary

Keep open. Current main still archives reset transcripts as `.jsonl.reset.*`, but `sessions_history` goes through `chat.history`, and `chat.history`/`readSessionMessages` only read the current resolved transcript candidates. There is no archive-chain discovery, `includeArchived`/`followResets` option, or reset-archive metadata in the tool response. The related open PR #60409 is a partial implementation candidate, but it is not merged and its described fallback does not cover the full logical reset-chain/discoverability request.

## Best Possible Solution

Keep this issue open. The best path is to implement a gateway-owned reset-archive discovery helper used by `chat.history`, `sessions_history`, `sessions_list` previews, and the HTTP history surface as appropriate, with tests for active primary plus archived prior fragments, missing primary fallback, multiple archives, truncation metadata, and post-reset system-event provenance. Review or expand #60409 if it is still the active candidate, but make sure the merged fix covers the logical session-chain recovery described here.
