## Summary

Keep open. Current main still returns sqlite-vec load success without a Windows retry or post-load vec0/function usability probe, and memory-core still marks vectors available based on that loader result. The related Windows issue #65704 was closed as superseded by this issue, while #69059 is still open/unmerged and #65156 still tracks the overlapping silent-load/no-functions failure mode.

## What This Changes

Fix Windows memory indexing so sqlite-vec loads and registers correctly, creates/populates `chunks_vec`, and reports vector availability instead of degrading memory indexing/search.

## Best Possible Solution

Land one narrow memory-host sqlite-vec readiness fix: validate usable sqlite-vec support after load, handle the Windows suffix/path retry if confirmed, keep the source and package loader copies aligned, preserve the in-process cosine fallback with actionable diagnostics, and add regression coverage for both hard load failure and load-success-without-usable-vec0 cases. After that merges, close this issue and reconcile #65156/#69059 as implemented or superseded as appropriate.
