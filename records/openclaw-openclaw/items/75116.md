## Summary

Keep open. Current main still lacks the requested per-server `injectCallerContext` contract and behavior, and the issue is paired with open closing PR #75118, which should be reviewed or closed before the issue is resolved.

## What This Changes

The issue requests an opt-in per-server `mcp.servers.<name>.injectCallerContext` flag that forwards OpenClaw caller identity headers to trusted remote bundled MCP servers without changing default behavior.

## Best Possible Solution

Review PR #75118 for the trust boundary and implementation details, then land a shared bundle-MCP/config-layer change if accepted: remote-only, default-off, non-overwriting caller-header merge, stripped downstream control flag, placeholder env population when any merged server opts in, and matching schema/docs/tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a remote `mcp.servers.<name>.url` entry with the proposed flag and inspect the generated bundle MCP config: current main has no explicit schema/type support, does not merge caller headers in the bundle path, and only emits `OPENCLAW_MCP_*` placeholders for the loopback runtime.

## Solution Assessment

Yes, pending maintainer approval of the trust boundary. A per-server default-off flag is the narrow maintainable direction; a global flag would create broader accidental identity leakage risk, and the paired PR is the active implementation candidate.
