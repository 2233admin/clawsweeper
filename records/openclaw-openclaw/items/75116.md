## Summary

Keep open. Current main already forwards caller context headers to OpenClaw's own loopback MCP server, but there is no implemented per-server `injectCallerContext` behavior for user-configured or bundle-provided remote MCP servers.

## What This Changes

The issue requests an opt-in per-MCP-server `injectCallerContext` flag that forwards OpenClaw caller identity headers to trusted remote MCP servers without overwriting user-supplied headers.

## Best Possible Solution

Add the option in the shared MCP config/bundle merge layer so all CLI backend modes receive the same remote-only, non-overwriting header merge, strip the control flag from emitted downstream config, populate caller placeholder env when any merged server opts in, and document the trust boundary.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: a remote `mcp.servers.<name>.url` entry is merged through without caller-context headers, while the caller placeholder env is only created for the OpenClaw loopback MCP runtime.

## Solution Assessment

Yes, pending maintainer signoff on the trust boundary. A per-server default-off flag is narrower and safer than a global switch because it avoids silently leaking caller identity to unrelated third-party MCP endpoints.
