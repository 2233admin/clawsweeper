## Summary

Keep open. Current main still has unguarded OpenAI Completions tool payload emission, and open PR #74738 is already linked as the implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenAI-compatible chat-completions requests to honor `compat.supportsTools: false` by omitting `tools`, `tool_choice`, and the tool-history `tools: []` fallback.

## Best Possible Solution

Land PR #74738 or an equivalent narrow change that checks model tool support inside `buildOpenAICompletionsParams()`, suppresses `tools`, `tool_choice`, and the history fallback when disabled, and keeps regression coverage for both non-empty tools and replayed tool history.

## Reproduction Assessment

Yes. The issue includes a concrete payload-capture repro, and static inspection of current main shows the same unguarded `context.tools` and tool-history fallback assignments in the OpenAI Completions payload builder.

## Solution Assessment

Yes. The narrow maintainable fix is to guard the transport payload builder itself with the existing model-tool-support helper, because relying only on the embedded runner leaves direct or future transport callers able to send unsupported tool fields.
