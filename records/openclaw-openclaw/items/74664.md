## Summary

Keep open: current main still accepts and documents `compat.supportsTools: false`, but the current `@mariozechner/pi-ai` OpenAI Completions adapter used by OpenClaw still emits `tools` from `context.tools` and the tool-history fallback without consulting that flag. OpenClaw's normal embedded-runner tool registration gate does not fully cover the dependency payload edge reported here, and no open PR search result specifically references this issue as a fix.

## What This Changes

The issue asks OpenAI-compatible chat-completions requests to omit tool payload fields when a model is configured with `compat.supportsTools: false`, including the tool-history `tools: []` fallback.

## Best Possible Solution

Add a narrow OpenAI Completions compatibility guard in the existing embedded stream wrapper layer, or consume an upstream `pi-ai` fix if one lands first, so `compat.supportsTools === false` removes tool-related request fields even when `context.tools` or replayed tool history reaches the transport. Keep the existing OpenClaw-level tool gating and add payload-capture regression coverage for both non-empty tools and the `tools: []` history fallback.
