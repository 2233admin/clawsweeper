## Summary

Keep open: current main still has the reported OpenAI-compatible completions payload bug, and an open closing PR already targets the exact issue. The issue should remain open until that PR or an equivalent narrow fix is merged.

## What This Changes

The issue asks OpenAI-compatible chat-completions requests to honor `compat.supportsTools: false` by omitting `tools`, `tool_choice`, and tool-history `tools: []` payloads.

## Best Possible Solution

Land a narrow fix in `buildOpenAICompletionsParams()` that uses the existing `supportsModelTools(model)` helper to suppress `tools`, `tool_choice`, and the tool-history `tools: []` fallback when `compat.supportsTools === false`, with regression coverage for non-empty tools and replayed tool history while preserving default behavior when the flag is true or omitted.

## Reproduction Assessment

Yes. The issue provides a concrete payload-capture repro, and current main statically follows the same path: a model with `compat.supportsTools: false` and non-empty `context.tools` still reaches the unguarded `params.tools = convertTools(...)` branch.

## Solution Assessment

Yes in principle. The narrowest maintainable fix is to guard the OpenAI completions transport builder itself, because the runner already filters normal tool registration but direct or boundary-aware transport callers can still pass tools through.
