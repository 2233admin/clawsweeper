## Summary

Keep open: current main still ignores `compat.supportsTools === false` in the OpenAI-compatible chat-completions payload builder, and the linked open PR is the right implementation candidate to review rather than a reason to close this issue early.

## What This Changes

OpenAI-compatible chat-completions models configured with `compat.supportsTools: false` should omit `tools`, `tool_choice`, and the tool-history `tools: []` fallback from outbound payloads.

## Best Possible Solution

Apply the narrow transport-layer fix: make `buildOpenAICompletionsParams()` check `supportsModelTools(model)` before emitting `tools`, `tool_choice`, or the tool-history empty-tools fallback, and add regression tests for non-empty tools plus replayed tool history while preserving default tool behavior when the flag is true or omitted.

## Reproduction Assessment

Yes. The issue has a concrete payload-capture reproduction, and static inspection of current main shows the same branch will still emit tool payload fields whenever `context.tools` is present or tool history needs fallback.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix because the runner already gates normal tool registration; the missing guard belongs in the completions payload builder as defense for direct callers and any unfiltered context path.
