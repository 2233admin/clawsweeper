## Summary

Keep open: current main still accepts `compat.supportsTools: false` but the OpenAI Completions payload builder does not honor it when constructing `tools`, `tool_choice`, or the tool-history fallback. The linked open PR explicitly closes this issue, so this issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

The issue requests that OpenAI-compatible chat-completions payloads omit `tools`, `tool_choice`, and the tool-history `tools: []` fallback when a model sets `compat.supportsTools: false`.

## Best Possible Solution

Land a focused fix in the OpenAI Completions transport that reuses the existing `supportsModelTools()` semantics at payload-construction time, suppresses both normal tool payloads and the tool-history fallback when disabled, and adds regression coverage for non-empty `context.tools` plus replayed tool history while preserving default tool behavior when the flag is true or omitted.

## Reproduction Assessment

Yes. The issue provides a credential-free payload-capture repro, and current main source still has the unguarded `context.tools` and `hasToolHistory()` branches that would produce the reported payload shape; no runtime test was executed because this review was read-only.

## Solution Assessment

Yes. Guarding inside `buildOpenAICompletionsParams()` is the narrowest maintainable fix because upstream runner filtering already exists but direct or future call paths can still pass tools into the transport builder.
