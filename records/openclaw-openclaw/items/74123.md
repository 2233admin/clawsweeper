## Summary

Keep open. Current main still has the sandbox resolver ordering that can corrupt canonical `media://inbound/<id>` image references before managed-media hydration, and open PRs #74186 and #74278 already target the same report with closing syntax, so the issue should stay open until a maintainer lands, rejects, or consolidates those fixes.

## What This Changes

The issue reports that image understanding fails for canonical `media://inbound/<image>` references because they are resolved as malformed workspace `media:/inbound/...` paths instead of OpenClaw-managed inbound media files.

## Best Possible Solution

Land or replace with one narrow fix that recognizes canonical `media://inbound/<id>` before sandbox bridge path resolution, preserves inbound media ID validation and sandbox path boundaries, and adds focused regression coverage for both the sandbox resolver and any generic URI fall-through guard that maintainers choose to keep.

## Reproduction Assessment

Yes. The source-level reproduction path is high-confidence: sandboxed image input reaches `resolveSandboxedBridgeMediaPath`, the resolver delegates to the bridge before fallback, and the sandbox filesystem resolver treats non-absolute `media://inbound/...` as workspace-relative input. I did not run tests because this review is read-only.

## Solution Assessment

Yes, a shared sandbox media resolver fix is the narrowest maintainable solution for the reported sandbox-shaped corruption. A defensive `loadWebMedia` guard may also be useful, but it should not replace handling the URI before the sandbox bridge rewrites it.
