## Summary

Keep open. Current `main` and the latest release still contain the resolver ordering that can corrupt a canonical `media://inbound/...` reference before managed-media hydration, and two open implementation PRs already reference this issue with closing syntax, so the issue should remain open until one fix lands or the PRs are closed.

## What This Changes

The issue asks for `media://inbound/<image-file>` image references to resolve to OpenClaw-managed inbound media instead of malformed workspace paths like `media:/inbound/...`.

## Best Possible Solution

Land or replace the existing implementation PRs with one narrow fix that recognizes canonical `media://inbound/<id>` before sandbox bridge path resolution, preserves inbound media ID validation and sandbox/workspace boundaries, and adds focused sandbox resolver plus image-tool regression coverage. A generic `loadWebMedia` stranded-scheme guard is useful as defense in depth, but it should not be the only fix if the sandbox resolver can still corrupt the URI first.

## Reproduction Assessment

Yes. The source path is high-confidence: sandboxed image inputs reach `resolveSandboxedBridgeMediaPath`, that resolver delegates to the bridge before fallback, and standard `path.resolve` maps `media://inbound/example.png` to `/workspace/media:/inbound/example.png`.

## Solution Assessment

Yes for the current routing decision. Keeping this issue open while maintainers review #74186 and #74278 is the narrowest maintainable path; after one correct PR lands, this can close as implemented.
