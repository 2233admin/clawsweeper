## Summary

Keep open. Current main still has the resolver ordering that can turn an intact `media://inbound/<id>` image reference into a workspace-relative `media:/inbound/...` path before managed-media hydration, and open PRs #74186 and #74278 are already implementation candidates tied to this issue.

## What This Changes

The issue requests that inbound images referenced as `media://inbound/<file>` resolve through OpenClaw managed media storage instead of being treated as an invalid workspace path.

## Best Possible Solution

Land or replace with one narrow fix that recognizes canonical `media://inbound/<id>` before sandbox bridge/workspace resolution can rewrite it, preserves inbound media ID validation and sandbox boundaries, and adds focused regression coverage for the sandboxed image-tool path plus any generic `loadWebMedia` stranded-URI guard the maintainer chooses to keep.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists from source: call the sandboxed image tool with `media://inbound/<id>`, which reaches `resolveSandboxedBridgeMediaPath`; the bridge path resolver can treat that non-absolute string as cwd-relative and produce the reported `media:/inbound` path.

## Solution Assessment

Yes. The best primary fix is a shared sandbox media resolver change that handles `media://inbound` before bridge resolution; a generic `loadWebMedia` URI guard is useful defense-in-depth but is not sufficient if the sandbox bridge already rewrote the URI.
