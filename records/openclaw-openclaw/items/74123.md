## Summary

Keep open. Current main still has the sandbox resolver ordering that can corrupt a canonical `media://inbound/<id>` reference before managed-media hydration, and two open PRs already target this issue with closing syntax, so the issue should stay open until a maintainer lands or rejects the fix path.

## What This Changes

The issue requests that `media://inbound/<image-file>` image inputs resolve to OpenClaw-managed inbound media instead of a malformed workspace path such as `media:/inbound` under the workspace.

## Best Possible Solution

Land or replace with one narrow fix that recognizes canonical `media://inbound/<id>` before sandbox bridge path resolution, preserves existing inbound ID validation and workspace boundaries, and adds focused sandbox resolver plus sandboxed image-tool regression coverage coordinated with #74186, #74278, and #63497.

## Reproduction Assessment

Yes. The current source gives a high-confidence path: a sandboxed image input reaches `resolveSandboxedBridgeMediaPath`, the resolver delegates to the bridge before fallback, and the sandbox filesystem resolver treats non-absolute `media://inbound/...` as workspace-relative input.

## Solution Assessment

Yes, a shared sandbox media resolver fix is the narrowest maintainable solution for the reported sandbox-shaped corruption. A broader `loadWebMedia` URI guard may be useful defense-in-depth, but it should not replace the resolver fix if sandbox bridge resolution has already rewritten the URI.
