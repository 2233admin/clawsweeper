## Summary

Keep open. Current main still routes sandboxed `media://inbound/<id>` image inputs through direct sandbox path resolution before managed-media hydration, which can turn the URI into a malformed workspace path like `/workspace/media:/inbound/<id>`. Related #63476 and open PR #63497 cover the broader image-tool `media://inbound` problem, but that PR is still unmerged and explicitly does not change sandbox/workspace rules, so it does not close this sandbox resolver gap.

## Best Possible Solution

Keep this issue open. The best fix is a narrow shared sandbox media resolver change that recognizes canonical `media://inbound/<id>` references, validates them with the existing inbound-media ID rules, and maps them to the sandbox-visible `media/inbound/<id>` claim-check file before direct bridge resolution can reinterpret the URI as `media:/inbound`. Add focused resolver coverage plus sandboxed image-tool regression coverage, and coordinate with #63476/#63497 so the non-sandbox and sandbox fixes converge without widening local media access.
