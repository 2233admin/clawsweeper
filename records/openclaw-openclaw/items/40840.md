## Summary

Keep open. Current main still has the reported lifecycle gap: `sessions.delete` deletes the store row after cleanup, while the normal agent-run finalizer can later write `store[sessionKey] = merged` from a stale pre-run session snapshot and recreate the same key.

## What This Changes

The issue asks for `sessions.delete` to behave as a reliable session termination boundary, or for OpenClaw to document that it is only a cleanup/reset primitive when normal agent runs can still re-register the same session key.

## Best Possible Solution

Add a deletion fence around successful session deletion, preferably where the post-run session-store updater writes back results: if the run began from an existing key and the current store no longer contains that key at finalization time, skip the stale write or return a structured failure instead of recreating the row. Keep existing cleanup behavior, add focused regression coverage for the stale snapshot case, and document weaker semantics only if maintainers intentionally decline hard-delete behavior.

## Repair Work Prompt

Fix the session deletion resurrection race reported in https://github.com/openclaw/openclaw/issues/40840. Current main lets `updateSessionStoreAfterAgentRun` recreate a session key from a stale pre-run `sessionStore` snapshot after `sessions.delete` has successfully removed that key. Add a conservative deletion fence so a post-run finalizer for an existing session does not recreate a key that is absent from the current on-disk store at finalization time. Prefer the smallest fix in `src/agents/command/session-store.ts`; only adjust `src/gateway/server-methods/sessions.ts` or `src/gateway/session-reset-service.ts` if a tombstone/generation helper is needed. Preserve normal creation of brand-new explicit sessions and existing ACP metadata preservation. Add regression coverage that writes a session row, passes a stale in-memory snapshot to the finalizer, deletes/removes the row from disk before finalization, and verifies the key stays deleted.
