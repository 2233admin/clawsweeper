## Summary

Keep open. Current main still has the loose `channels status --timeout` path: `channelsStatusCommand` uses raw `Number(...)`, forwards the value to gateway transport and request params, and has no focused invalid-timeout regression test. The PR remains useful, though it should be rebased/adapted to current main rather than applied verbatim.

## Best Possible Solution

Keep the PR open and ask for a rebase/update: import `parseTimeoutMsWithFallback`, compute the fallback as `opts.probe ? 30_000 : 10_000`, pass `{ invalidType: "error" }`, and add a dedicated `channelsStatusCommand` option-validation test for invalid timeout values outside the SecretRef fallback describe block. Then run the changed command/test lane before maintainer review.
