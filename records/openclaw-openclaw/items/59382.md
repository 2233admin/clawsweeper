## Summary

Keep this PR open. Current main still implements `channels status --timeout` with raw `Number(...)` parsing and forwards that value to the gateway request, so the PR's intended CLI-level validation is not implemented. The branch remains a narrow implementation candidate; the provided diff is limited to the command and focused tests, with no workflow, dependency, package, secret, or release-surface changes.

## Best Possible Solution

Keep this PR open for maintainer review and landing, or apply the same focused change on main: route `channelsStatusCommand` through `parseTimeoutMsWithFallback(opts.timeout, opts.probe ? 30_000 : 10_000, { invalidType: "error" })`, keep the dedicated invalid-timeout command-flow coverage, decide the actual CLI `--probe` default behavior, then run the normal changed gate before merge.
