## Summary

Keep this PR open. Current main still parses `openclaw channels status --timeout` with raw `Number(...)`, so invalid values are not validated through the shared timeout helper yet. The proposed change remains a focused implementation candidate, and the provided PR diff is limited to the command path plus focused tests with no security-sensitive workflow, dependency, lockfile, install, release, or secret-handling changes.

## Best Possible Solution

Review and land this PR, or apply the equivalent focused change on main: update `src/commands/channels/status.ts` to use `parseTimeoutMsWithFallback(opts.timeout, fallback, { invalidType: "error" })`, preserve the 10s and 30s defaults, and keep the focused option-validation tests. That closes the current command validation gap while preserving contributor credit.
