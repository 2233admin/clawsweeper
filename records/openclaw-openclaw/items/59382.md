## Summary

Keep this PR open. Current main still uses raw `Number(...)` parsing for `channels status --timeout`, then forwards that value into the gateway status request instead of validating with the shared CLI timeout parser. The PR remains a narrow implementation candidate, and the provided diff is limited to the command and its focused tests with no observed security-sensitive file surface.

## Best Possible Solution

Review and land this PR, or apply the equivalent focused change on main: use `parseTimeoutMsWithFallback(opts.timeout, opts.probe ? 30_000 : 10_000, { invalidType: "error" })` in `src/commands/channels/status.ts`, keep command-level invalid-timeout coverage, settle the intended `--probe` default at the CLI boundary, and validate with the normal changed gate before merge.
