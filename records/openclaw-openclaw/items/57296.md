## Summary

Keep #57296 open. Current main has partial backend work for WebChat audio payloads and separate UI support for MEDIA-derived audio attachments, but the two do not line up end-to-end: gateway persists/broadcasts local TTS audio as raw `content` blocks with `type: "audio"`, while the Control UI renderer only renders audio controls for normalized `attachment` items produced from `MEDIA:` text/attachment content. I did not find a current UI path that renders those raw audio content blocks as audible or downloadable media.

## Best Possible Solution

Keep the issue open and implement one end-to-end WebChat TTS delivery contract. The conservative fix is either to have gateway emit a Control UI-renderable attachment/MEDIA representation backed by `__openclaw__/assistant-media`, or to teach the Control UI to render `type: "audio"` content blocks and provide a history-safe URL/download fallback. Add a regression that drives a TTS/tool media reply through `chat.send`/`chat.history` into the Control UI renderer and asserts an audible/downloadable `<audio controls>` surface.
