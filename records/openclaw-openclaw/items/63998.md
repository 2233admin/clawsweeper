## Summary

Keep this open. Current main still has only partial mitigations: the active transcript byte guard is opt-in, while repair, restart recovery, and gateway history paths can still materialize a pathological JSONL transcript before any bounded quarantine or recovery decision. The new Linux confirmation strengthens the report rather than making it stale.

## What This Changes

The issue asks OpenClaw to prevent oversized session JSONL transcripts from causing crash-restart OOM loops by bounding transcript loading/replay, capping repeated bootstrap markers, deduplicating repeated media, and degrading gracefully.

## Best Possible Solution

Implement a core startup-safe transcript recovery guard that stats and classifies active JSONL files before repair, restart recovery, readSessionMessages, or SessionManager.open; rotate, quarantine, or compact pathological transcripts into a bounded successor without first loading the whole file; add a total-count circuit breaker for repeated full-bootstrap markers; preserve existing media offload behavior where possible; and cover the recovery path with focused regression tests and docs.

## Reproduction Assessment

Yes. The issue gives concrete transcript size, message counts, marker counts, restart cadence, and logs, and a second reporter confirmed matching OOM restart-loop symptoms on Linux on 2026-04-30.

## Solution Assessment

Yes for the recommended repair direction, not for the current implementation. The narrow maintainable fix is startup-safe transcript classification/quarantine plus a bootstrap marker cap; the existing opt-in compaction guard and related open PRs are useful but do not protect the restart path by default.
