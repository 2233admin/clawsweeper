## Summary

Keep this open: current main has useful partial mitigations for oversized active transcripts, successor transcripts, and duplicate retry cleanup, but the reported crash-restart/OOM loop is not fully fixed. Default configuration still leaves the active transcript byte guard disabled, and restart/repair/history paths can still materialize a pathological JSONL transcript before any bounded quarantine or recovery decision.

## What This Changes

The issue asks OpenClaw to prevent bloated session transcripts from causing gateway crash-restart OOM loops by adding transcript size guards, bootstrap replay circuit breakers, media deduplication, and graceful degradation.

## Best Possible Solution

Implement a core startup-safe transcript recovery guard that stats and classifies active JSONL files before repair, restart recovery, `readSessionMessages`, or `SessionManager.open`; rotate, quarantine, or compact pathological transcripts into a bounded successor without first loading the whole file; add a total-count circuit breaker for repeated full-bootstrap markers; preserve existing media offload/dedup behavior where possible; and document the recovery path with focused regression coverage.
