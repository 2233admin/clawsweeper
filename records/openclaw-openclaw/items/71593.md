## Summary

Keep open. Current main still exposes a closed SecretRef provider model limited to `env`, `file`, and `exec`, while the linked closing implementation PR is still open and unmerged. The prior discussion also added a concrete plugin-config motivation, so the issue remains a valid API/security/product follow-up rather than cleanup-closeable.

## What This Changes

Request to add a plugin/SDK SecretRef provider seam so plugins can define secret sources, and to broaden SecretRef object support across core secret-bearing config fields such as skills env, plugin config, and channel credentials.

## Best Possible Solution

Land or replace PR #72548 with a versioned, documented plugin SecretRef provider contract that preserves strict built-in env/file/exec behavior, keeps config/SDK/Gateway schemas aligned, honors plugin enablement and origin boundaries, and either includes or separately tracks the remaining SecretRef field-coverage gaps called out by the issue and the engram authToken comment.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection: current main rejects non-env/file/exec SecretRef sources in config, SDK, Gateway wire schemas, and resolver dispatch, and only supports manifest-declared plugin config SecretInput paths.

## Solution Assessment

Unclear until maintainer review of PR #72548. A plugin-defined provider seam matches the documented plugin capability model, but the proposed split that defers broader field coverage needs maintainer product and security approval before this issue can be considered solved.
