## Summary

Keep open. Current main still has a closed SecretRef provider/source model limited to env, file, and exec; the existing plugin config secretInputs mechanism helps declared plugin-owned paths but does not let plugins register new SecretRef provider sources, and the open implementation candidate #72548 is still unmerged.

## What This Changes

The request is to add a plugin-defined SecretRef provider seam and broader SecretRef acceptance across core config schema surfaces so users can add secret backends and coverage without editing core schemas.

## Best Possible Solution

Review and land #72548 or a narrower maintainer-approved replacement that adds a versioned, documented plugin SecretRef provider contract, then follow through on the remaining SecretInput coverage for skills env values, generic plugin config sensitive paths, and channel credential schemas where appropriate.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence verifiable by static inspection: current type, zod, Gateway protocol, docs, and resolver dispatch only support env, file, and exec, and the engram comment gives a concrete user-visible plugin config case.

## Solution Assessment

Unclear until maintainer review of the active API proposal. A plugin-defined SecretRef provider seam fits OpenClaw's plugin-first direction, but #72548 needs security/versioning review and may need follow-up coverage work because it explicitly defers part of the issue's scope.
