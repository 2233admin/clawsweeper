## Summary

Keep this issue open. Current main still has a closed SecretRef source/provider model and no plugin-defined SecretRef provider contract; the open implementation PR #72548 is explicitly paired with this issue but is unmerged, so the issue should stay open until that PR is reviewed/landed or replaced. The broader requested field-coverage work is also not fully solved on main.

## What This Changes

Request support for plugin-defined SecretRef providers so users can add secret backends without repeatedly changing core schemas, including broader coverage for secret-bearing config fields such as channel credentials, skill env values, and plugin config paths.

## Best Possible Solution

Review #72548 as the current implementation candidate, or replace it with a narrower design that either adds a versioned plugin SecretRef provider contract across manifest metadata, SDK/runtime loading, config and Gateway schemas, resolver dispatch, docs, and tests, or explicitly documents exec as the supported universal provider path and opens a separate follow-up for remaining SecretInput field coverage.
