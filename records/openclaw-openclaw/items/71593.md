## Summary

Keep open. Current main still exposes SecretRefs and secret provider configs as a closed env/file/exec model, plugin config coverage is limited to declared SecretInput paths rather than a provider-registration seam, and the GitHub context shows open PR #72548 is already the closing implementation candidate that should be reviewed before this issue is closed.

## What This Changes

The issue asks for a plugin-defined SecretRef provider contract so users can mask OpenClaw config fields without changing core schemas for every provider or credential field.

## Best Possible Solution

Review and either land #72548 or request a narrower replacement that adds a versioned, documented plugin SecretRef provider contract, then explicitly decide how to track the remaining config-field coverage for plugin configs, skills env values, and channel credential schemas before closing this issue.

## Reproduction Assessment

Not applicable as a feature/API request. The gap is high-confidence by static inspection: current main accepts and resolves only env/file/exec SecretRef sources, and the engram comment supplies a concrete plugin-config field that still needs broader SecretInput coverage.

## Solution Assessment

Unclear until maintainer review of #72548. A plugin-defined SecretRef provider seam matches the plugin-first architecture, but the current implementation candidate is broad and secret-handling sensitive, and the broader field-coverage half may need a separate tracked design decision.
