## Summary

Keep this open. Current main has a solid configured SecretRef foundation, but it still does not provide lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools, and the requested `get` path needs a trusted redaction plus owner/session-scope contract before it can safely return live credential material.

## What This Changes

Add a secret broker plugin/tool surface for lazy `secret.get`, `secret.has`, and `secret.list` access over Bitwarden, Keychain, 1Password, and Vault, with transcript redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract on top of the existing SecretRef provider foundation, including `has`/`list` metadata semantics, redacted or non-persistent `get` results, and a pre-execution owner/direct-session gate; then implement the first backend through the maintainer-approved bundled path or after extending the plugin API safely.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, but lazy agent-callable `secret.*` tools and a safe redaction/session-scope API are absent.

## Solution Assessment

Unclear. The requested generic plugin direction is directionally aligned with OpenClaw's plugin vision, but a plain external plugin is not sufficient until core exposes a trusted redacted-result and session-scope contract; the safer path is either a bundled trusted implementation or a plugin-API extension first.

## Security Review

Status: needs_attention

Summary: This issue needs security attention because it proposes returning live credential material through agent tools before OpenClaw has a safe trusted-result and session-scope contract for that surface.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools return ordinary tool content, while pre-model tool-result rewriting is bundled-only, so core needs a trusted redacted or non-persistent result contract before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:81`
  - body: OpenClaw supports group and channel sessions with authorization behavior that is separate from context visibility. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions, not only redacted after a tool result exists.
  - confidence: 0.88
