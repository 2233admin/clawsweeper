## Summary

Keep this issue open. Current main has a substantial SecretRef system for configured credentials, including exec-backed 1Password/Vault-style providers, but it does not implement lazy agent-facing `secret.get`/`secret.has`/`secret.list` tools, and the requested non-transcript secret handling is a core/trusted-plugin API and security-design question rather than a safe ClawHub close today.

## What This Changes

Request a first-class secret broker plugin surface with `secret.get`, `secret.has`, and `secret.list` over Bitwarden, macOS Keychain, 1Password, and Vault, with transcript redaction and session scope guards.

## Best Possible Solution

Define a trusted secret capability contract that reuses the existing SecretRef provider foundation, adds explicit redacted or non-persistent secret result semantics, and enforces session trust/scope policy before exposing any Bitwarden, Keychain, 1Password, or Vault-backed agent tools.
