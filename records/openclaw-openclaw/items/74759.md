## Summary

Keep this open. Current main has SecretRef-backed configured credential resolution, including exec providers suitable for 1Password/Vault-style integrations, but it does not provide lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools. The issue discussion already narrowed the remaining work to a product/security decision because external plugins cannot currently use the trusted pre-model tool-result middleware needed for safe secret redaction.

## What This Changes

Add a first-class secret broker tool surface for named secrets over Bitwarden, macOS Keychain, 1Password, and Vault, with redaction and session-trust guards.

## Best Possible Solution

Define a trusted secret capability contract that reuses the existing SecretRef provider foundation, adds explicit redacted or non-persistent secret result semantics, and gates tool availability/execution by session trust before exposing Bitwarden, Keychain, 1Password, or Vault-backed tools.

## Reproduction Assessment

Not applicable as a feature and security-design request. The gap is high-confidence from source inspection: current main has eager configured SecretRef resolution, while the requested lazy `secret.*` agent tools are absent.

## Solution Assessment

Unclear; the original generic plugin direction is not sufficient by itself because external plugins cannot currently perform pre-model secret redaction. The safer path is a maintainer-approved choice between a trusted bundled plugin and a plugin API extension that first defines redaction and session-scope semantics.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credentials through agent tools before core has a safe result and session-trust contract for that surface.

Concerns:

- **[high] Define redacted secret tool results before exposing get:** `docs/plugins/building-plugins.md:179`
  - body: `secret.get` would intentionally return credential material. Current external plugin tools return normal tool content, while pre-model tool-result rewriting is restricted to trusted bundled plugins, so core needs a safe result contract before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:112`
  - body: OpenClaw has distinct group/channel sessions and group authorization behavior. A secret broker needs an owner-DM or equivalent trust check before the tool is offered or executed, not only transcript redaction after the fact.
  - confidence: 0.86
