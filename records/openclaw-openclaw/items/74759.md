## Summary

Keep this open. Current main has configured SecretRef resolution with env/file/exec providers and a constrained `secrets.resolve` RPC, but it still lacks lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools and the trusted redaction/session-scope contract needed to return live credential material safely. The follow-up discussion correctly identifies the bundled-only tool-result middleware as the design blocker and asks for maintainer direction.

## What This Changes

Request a first-class secret broker plugin with agent-callable `secret.get`, `secret.has`, and `secret.list` over Bitwarden, Keychain, 1Password, and Vault, with transcript redaction and trusted-session guards.

## Best Possible Solution

Define a versioned trusted secret capability contract on top of the existing SecretRef provider foundation: `has/list` return only metadata, `get` returns redacted or non-persistent secret results, and tool exposure/execution is blocked for non-owner or group/channel sessions before backend calls. Then implement the first backend either as a maintainer-approved bundled trusted plugin or after adding the plugin-API trust extension.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from targeted source/docs inspection: configured SecretRefs exist, but lazy agent-callable `secret.*` tools and a safe redaction/session-scope API are absent.

## Solution Assessment

Unclear. The requested generic plugin direction is not sufficient by itself because external plugins cannot currently register the trusted pre-model result middleware; the safer path is maintainer approval for a bundled implementation or an additive plugin-API trust contract first.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credential material through agent tools before OpenClaw has a safe result-redaction and trusted-session contract for that surface.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools return normal tool content, while pre-model tool-result rewriting is restricted to bundled plugins, so core needs a trusted redacted or non-persistent result contract before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:81`
  - body: OpenClaw has group and channel sessions with separate authorization and context behavior. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions, not only redacted after the result is produced.
  - confidence: 0.88
