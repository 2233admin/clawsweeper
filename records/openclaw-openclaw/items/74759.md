## Summary

Keep this open. Current main has SecretRef-backed configuration resolution, including exec providers that can wrap tools like 1Password or Vault, but it still lacks lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools and the trusted redaction/session-scope contract needed to expose live credentials safely.

## What This Changes

The issue requests a first-class secret broker plugin exposing `secret.get`, `secret.has`, and `secret.list` over backends such as Bitwarden, macOS Keychain, 1Password, and Vault without persisting returned secret values.

## Best Possible Solution

Define a trusted secret capability contract on top of the existing SecretRef provider foundation: `has`/`list` name semantics, redacted or non-persistent `get` results, and a pre-execution session-scope gate. Then implement the first backend either as a maintainer-approved bundled trusted plugin or after extending the plugin API to express that trust boundary for ClawHub plugins.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, while lazy agent-callable `secret.*` tools and the safe redaction/session-scope API are absent.

## Solution Assessment

Unclear. The requested plugin shape is reasonable, but an ordinary external plugin is not the best complete solution because pre-model result rewriting and trusted tool policy are bundled-only today; a maintainer-approved trust contract or bundled implementation should come first.

  - body: OpenClaw supports group and channel sessions where trigger authorization is not a universal redaction boundary. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions, not only redacted after a result is produced.
