## Summary

Keep this open. Current main has a mature configured SecretRef foundation, but it still lacks the requested lazy agent-callable secret broker tools and needs a maintainer security/product decision before exposing live credential material through tools.

## What This Changes

Request a first-class secret broker plugin with `secret.get`, `secret.has`, and `secret.list` over Bitwarden, macOS Keychain, 1Password, and Vault-style backends.

## Best Possible Solution

Define a trusted secret capability contract over the existing SecretRef provider foundation, including metadata-only `has`/`list`, redacted or non-persistent `get` results, and an owner/direct-session pre-execution gate; then implement the first backend through the maintainer-approved bundled path or after extending the plugin API safely.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, but lazy agent-callable `secret.*` tools and a safe redaction/session-scope API are absent.

## Solution Assessment

Unclear. The requested generic plugin direction is not sufficient by itself because external plugins cannot currently register the trusted pre-model tool-result middleware; the safer path is either a bundled trusted implementation or a core plugin-API extension first.

## Security Review

Status: needs_attention

Summary: This issue needs security attention because it proposes returning live credential material through agent tools before OpenClaw has a trusted result and session-scope contract for that surface.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools return normal tool content, while pre-model tool-result rewriting is bundled-only, so core needs a trusted redacted or non-persistent result contract before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:81`
  - body: OpenClaw supports group and channel sessions whose authorization controls are separate from context visibility. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions, not only redacted after a result exists.
  - confidence: 0.88
