## Summary

Keep this open. Current main has a strong SecretRef provider foundation, but it still does not provide lazy agent-callable `secret.get`/`secret.has`/`secret.list` tools, and the proposed surface needs a maintainer decision because external plugins cannot currently provide the trusted pre-model redaction and pre-execution session-scope gates required for live credential material.

## What This Changes

Add a first-class secret broker plugin/tool surface for resolving named secrets from Bitwarden, macOS Keychain, 1Password, or Vault with redaction and session-scope guards.

## Best Possible Solution

Define a trusted secret capability contract on top of the existing SecretRef provider foundation, including `has`/`list` metadata semantics, redacted or non-persistent `get` results, and an owner/direct-session gate before the tool is offered or executed; then implement the first backend through the maintainer-approved bundled path or after extending the plugin API safely.

## Reproduction Assessment

Not applicable as a feature and security-design request. The current-main gap is high-confidence from source and docs inspection: configured SecretRefs exist, but lazy agent-callable `secret.*` tools and a safe redaction/session-scope API are absent.

## Solution Assessment

Unclear. The requested generic plugin direction is not sufficient by itself because external plugins cannot currently register the trusted pre-model result middleware or trusted pre-tool policy; the safer maintainable path is a bundled trusted implementation or a core plugin-API extension first.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because it proposes returning live credential material through agent tools before OpenClaw has a trusted result and session-scope contract for that surface.

Concerns:

- **[high] Define redacted secret result semantics before get:** `src/plugins/registry.ts:369`
  - body: A `secret.get` tool would intentionally return credential material. Current external plugin tools produce normal tool content, while pre-model tool-result rewriting is bundled-only, so core needs a trusted redacted or non-persistent result contract before any backend returns secrets.
  - confidence: 0.93
- **[high] Gate secret tools by trusted session scope:** `docs/channels/groups.md:83`
  - body: OpenClaw supports group and channel sessions where allowlists are trigger authorization, not a universal context redaction boundary. Secret tools should be hidden or blocked before execution for group/channel and otherwise untrusted sessions.
  - confidence: 0.88
- **[medium] Decide bundled trust versus plugin API trust first:** `src/plugins/registry.ts:1566`
  - body: Both trusted tool-result middleware and trusted pre-tool policies are bundled-only today. Shipping this as an external plugin without a new trust contract would leave the most important safety controls unavailable to the plugin.
  - confidence: 0.9
