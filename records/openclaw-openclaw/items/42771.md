## Summary

Keep this PR open. Current main still installs Docker Playwright browsers under `/home/node/.cache/ms-playwright`, docs still tell Docker users to persist that cache path, and the same-author paired bug #42819 remains open. The PR is a plausible in-repo fix, but the current patch still needs Dockerfile correctness fixes before merge.

## Best Possible Solution

Keep this PR open for maintainer review and contributor fixes. The best path is to rebase onto current main, set `/opt/playwright` as the Docker Playwright browser registry path, create it with `node:node` ownership for default builds, recursively chown after browser preinstall, avoid deleting apt lists inside the BuildKit cache mount, create the Chromium symlink only after a successful install, update both Docker and browser-control docs, and add focused Dockerfile assertions for path, ownership, symlink, and cache behavior.

## Risks / Open Questions

- Closing this PR now would drop a plausible implementation candidate for the still-open paired Docker bug #42819.
- Merging the current patch as-is can leave `/opt/playwright` unwritable by `USER node` in default builds where `OPENCLAW_INSTALL_BROWSER` is unset.
- The patch still deletes apt lists inside a named BuildKit cache mount, which can degrade rebuild caching without reducing the final image layer.
- Docs have drifted since the PR was opened: current main also documents Docker Playwright persistence in `docs/tools/browser-control.md`.
