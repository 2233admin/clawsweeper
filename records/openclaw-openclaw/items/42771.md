## Summary

Keep this PR open. Current main still installs Docker Playwright browsers under `/home/node/.cache/ms-playwright`, docs still document that cache path, and the same-author paired bug #42819 remains open. The PR is a plausible in-repo implementation candidate, but its current patch still needs Dockerfile correctness fixes and docs reconciliation before merge.

## Best Possible Solution

Keep this PR open for maintainer review and contributor fixes. The best path is to rebase onto current main, set `/opt/playwright` as the Docker browser registry path, create it with `node` ownership in all builds, recursively chown after preinstall, remove apt-list deletion from the BuildKit cache mount, create the Chromium symlink only after a successful install, update `docs/install/docker.md` and `docs/tools/browser-control.md`, and add focused Dockerfile assertions for path, ownership, symlink, and cache behavior.

## Risks / Open Questions

- Closing this PR now would drop a plausible implementation path for the open paired Docker browser-path bug #42819.
- Merging the current patch as-is can leave `/opt/playwright` unwritable by `USER node` when `OPENCLAW_INSTALL_BROWSER` is unset, because the setup step creates it without `node:node` ownership.
- The patch still deletes `/var/lib/apt/lists/*` inside a named BuildKit apt cache mount, which can degrade rebuild caching without reducing the committed image layer.
- The docs patch is stale against current main because Docker Playwright guidance now lives in `docs/tools/browser-control.md` as well as `docs/install/docker.md`.
- No current test asserts the requested path, ownership, symlink, or cache behavior.
