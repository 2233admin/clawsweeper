## Summary

Keep open. Current main still classifies API internal-server errors as `timeout`, and timeout reasons are explicitly ignored for auth-profile cooldown, so the reported same-provider retry/cooldown failure is not fully fixed. The targeted fix PR #45625 was closed unmerged.

## Best Possible Solution

Keep this issue open and land a narrow failover-classification fix: treat Anthropic/Pi SDK `api_error: Internal server error` as an overload/profile-cooldown-worthy transient so same-provider profiles are skipped quickly and cross-provider fallback can run, while preserving generic 500/server-error timeout behavior. Add focused classifier and auth-profile/fallback regression coverage; only adjust cron docs/changelog if the user-facing behavior changes.

## Clownfish Work Prompt

Fix the cron LLM outage fast-fail issue by making Anthropic/Pi SDK internal-server API errors cooldown-worthy without broadening generic 500 behavior. Current main classifies `LLM error api_error: Internal server error` as `timeout`; `timeout` is dropped by auth-profile failure policy, so same-provider profiles keep rotating and cron can burn the full `timeoutSeconds` window. Start from the analysis in PR #45625, but adapt it to the current split helpers (`failover-matches.ts` and `errors.ts`). Implement a narrow classifier so Anthropic-style/Pi SDK `api_error` internal-server payloads are treated as `overloaded` or an equivalent auth-profile cooldown reason. Preserve existing behavior for raw `500 Internal Server Error`, generic service-unavailable text, and plain status prose unless the message carries the precise API-error transient signal. Add regression coverage for plain `LLM error api_error: Internal server error`, JSON api_error transient payloads, and a negative generic 500 case. Add or adjust auth-profile/model-fallback coverage if needed to prove the reason marks cooldown and moves to cross-provider fallback quickly. Do not change provider SDK dependencies, retry budgets, cron schedule semantics, or broad scheduler behavior. Changelog only if maintainers consider the failover behavior user-facing; credit the reporter/contributor per repo policy.
