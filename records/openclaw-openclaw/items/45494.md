## Summary

Keep open. Current main still routes the reported Anthropic/Pi SDK `api_error: Internal server error` shape through the timeout path, and timeout classifications are deliberately excluded from auth-profile cooldown, so repeated same-provider stalls remain plausible before cron can apply its post-run backoff or failure alert handling. The focused fix PR #45625 is closed unmerged, and adjacent open PRs #45594 and #74909 touch nearby server-error/cooldown semantics without resolving this exact path.

## What This Changes

The issue requests faster failure, retry/backoff, and optional delivery surfacing for cron agent jobs when LLM provider API errors persist across fallback attempts.

## Best Possible Solution

Define a narrow provider-outage failover contract for Anthropic/Pi SDK `api_error: Internal server error`: classify it as a transient provider/server outage in a way that bounds same-provider retries or profile rotation, reaches cross-provider fallback promptly, and lets the existing cron backoff/failure-alert machinery run after the agent turn returns. The fix should preserve generic raw 500, CDN/proxy, transport-timeout, auth, billing, and format behavior, and should include focused classifier plus runner-policy regression coverage.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the focused failure path: follow `LLM error api_error: Internal server error` through the current classifier, observe `timeout`, then observe timeout is dropped by auth-profile failure policy. The full live provider-outage cron schedule timing was not reproduced in this read-only pass.

## Solution Assessment

Is this the best way to solve the issue? No for current main: the existing classifier/cooldown behavior does not fast-fail the reported SDK error shape. The narrowest maintainable fix is in failover classification plus runner/profile policy, with cron scheduler changes only if that lower-level run still cannot return promptly.
