## Summary

Keep open. Current main still has no Feishu outbound minimum-interval or pacing configuration, and the burst-prone Feishu reply, comment reply, message-tool, media, and streaming-card paths still call Feishu APIs directly or only use unrelated streaming-update coalescing. The related open trackers cover broader cross-provider rate limiting and reactive Feishu retry/backoff, but they do not supersede this proactive Feishu message-spacing request.

## What This Changes

Add a configurable Feishu channel message rate-limiting/minimum-interval setting, ideally also reducing bursts caused by segmented LLM replies.

## Best Possible Solution

Add a Feishu-owned per-account outbound pacing layer, resolved from top-level and per-account config, and apply it consistently around visible outbound sends that can burst Feishu API calls. Keep broad cross-provider rate-limit policy in #13615 and reactive Feishu retry/backoff in #70879 unless maintainers intentionally expand the scope.

## Repair Work Prompt

Implement a narrow Feishu plugin fix for https://github.com/openclaw/openclaw/issues/14206. Add a configurable Feishu outbound message pacing/minimum-interval setting that can be set at `channels.feishu` and overridden per `channels.feishu.accounts.<id>`, with a conservative disabled/default behavior unless the existing config style clearly supports an opt-in default. Keep the implementation Feishu-owned; do not add broad core/provider rate-limit policy and do not implement reactive retry/backoff, which belong to #13615 and #70879.

Expected boundary: add schema/type/manifest/docs support, resolve the setting through existing account config merging, and introduce a per-account pacing helper used by burst-prone visible outbound delivery paths: normal chunked replies, structured-card replies, comment-thread replies, message-tool outbound text/cards, and media send calls after upload where applicable. Be careful with streaming-card behavior: preserve the existing streaming update coalescing and only pace message/card send operations where it prevents Feishu API bursts without making live updates unusable.

Add focused Vitest coverage for config parsing/inheritance and the affected dispatcher/outbound/send surfaces using fake timers or injected delays. Update `docs/channels/feishu.md`, `extensions/feishu/openclaw.plugin.json`, and `CHANGELOG.md` with a user-facing Feishu entry credited to the reporter. Do not touch unrelated channel rate-limit policy, core plugin APIs, dependency versions, or bundled Feishu auth behavior.
