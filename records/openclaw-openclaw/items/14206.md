## Summary

Keep this issue open. Current main still lacks a Feishu outbound minimum-interval or pacing configuration, and the normal reply, comment reply, message-tool outbound, media, and streaming-card paths still send directly to Feishu APIs aside from unrelated typing/backoff and streaming-update throttles. The existing broader rate-limit tracker #13615 and retry/backoff tracker #70879 are related, but they do not supersede this Feishu-specific proactive pacing request.

## What This Changes

The issue requests configurable outbound message rate limiting for the bundled Feishu channel, including a minimum interval between Feishu sends and possible interaction with response chunking.

## Best Possible Solution

Add a Feishu-owned, per-account outbound pacing layer that is configurable at top level and per account, documented in the Feishu channel docs and manifest/schema, and used consistently by all Feishu message delivery surfaces that can burst API calls. Keep #13615 for broader cross-provider rate limiting and #70879 for retry/backoff after rate-limit errors unless a maintainer explicitly expands the scope.
