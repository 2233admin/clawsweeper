## Summary

Current main still does not expose WhatsApp forwarding metadata to plugin inbound hooks. The WhatsApp path can locate Baileys `contextInfo`, but `isForwarded` and `forwardingScore` are not extracted into the normalized WhatsApp inbound message, not carried into the finalized inbound context, and not emitted by `inbound_claim` or `message_received`; related PR #46291 is still open, so this issue should remain open for maintainer review rather than cleanup closure.

## What This Changes

Expose WhatsApp `contextInfo.isForwarded` and `contextInfo.forwardingScore` to plugin `inbound_claim` and `message_received` hook payloads.

## Best Possible Solution

Land a narrow additive WhatsApp fix, preferably by reviewing/adapting #46291, that extracts `contextInfo.isForwarded` and `contextInfo.forwardingScore`, normalizes `isForwarded` when Baileys reports true or the score is at least 1, carries both through `WebInboundMessage` and `FinalizedMsgContext`, and exposes them through `CanonicalInboundMessageHookContext` metadata for `inbound_claim` and `message_received` with focused tests and plugin-hook docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: construct or trace a Baileys inbound message with `extendedTextMessage.contextInfo.forwardingScore` / `isForwarded`; current main can find the `contextInfo` object but has no typed or mapped path that reaches plugin hook events.

## Solution Assessment

Yes, the requested additive pass-through is the narrow maintainable direction. The fix should stay channel-owned for WhatsApp extraction and use the existing generic hook metadata surface rather than adding broad core WhatsApp policy.
