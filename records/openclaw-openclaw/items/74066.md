## Summary

Keep this issue open. Current main still routes malformed cron expressions through `context.cron.add(...)` and `context.cron.update(...)` without a cron-pattern validation/catch boundary, so Croner parse errors can still escape to the WS catch-all and become `UNAVAILABLE`. Open PR #74068 uses closing syntax for this issue and is the right implementation candidate to review/land before closing.

## Best Possible Solution

Keep this issue open and review/land PR #74068 or an equivalent narrow gateway-side validation fix. The fix should validate `schedule.kind === "cron"` expressions for `cron.add` and `cron.update` before service mutation, return `ErrorCodes.INVALID_REQUEST` with sanitized user-facing text, preserve valid cron/every/at behavior, and add regression tests for bad field count, out-of-range fields, invalid timezone, and defensive Croner parse failures.
