## Summary

Keep open. Current main still resolves `openclaw agent --channel discord --to ...` before channel/account delivery metadata is applied, and the shared resolver still only derives from `to`/session id/session key/agent id. The related default-agent `--to` work and delivery-context persistence work are present, but they do not add the requested delivery-context lookup for an existing Discord channel/thread session.

## Best Possible Solution

Implement delivery-context session routing for `openclaw agent` when channel/target metadata is provided and no explicit session selector should win. Normalize Discord raw numeric and `channel:<id>` targets through the shared channel target parser, search the relevant session store for matching `deliveryContext`/`lastChannel`/`lastTo`/`lastAccountId`/`lastThreadId`, preserve explicit `--session-id`/`--session-key` precedence, then update CLI docs to describe the supported behavior.

## Clownfish Work Prompt

Fix the unresolved CLI session routing bug for `openclaw agent --channel discord --to ...`. Current main resolves the session key in `src/commands/agent-via-gateway.ts` before channel/account delivery metadata is applied, and `src/agents/command/session.ts` cannot look up existing sessions by delivery context. Implement a narrow, generic resolver extension that preserves explicit `--session-key` and `--session-id` precedence, uses shared channel target parsing/normalization for channel targets, and for Discord handles both raw numeric IDs and `channel:<id>` targets. Search the appropriate agent session store for an existing session whose normalized delivery context or legacy last-delivery fields match `(channel, to, accountId, threadId)` before falling back to the main/default session behavior. Do not add Discord-specific ids or policy to core beyond generic channel parser seams; use existing plugin/SDK target parsing surfaces. Add regression tests for raw Discord numeric `--to`, `channel:<id>`, account-specific matching, and explicit session selector precedence. Update `docs/cli/agent.md` and `docs/tools/agent-send.md` if the supported behavior changes. Changelog entry expected because this is user-facing CLI behavior; credit the reporter from this issue, not Codex.
