## Summary

Keep open. Current main partially moved the server-side parser away from the old hardcoded 5 MB cap by using `agents.defaults.mediaMaxMb`, but the Control UI upload request is still not implemented end to end: images still hit a fixed 6 MiB cap, the bootstrap payload does not expose an upload policy to the browser, paste/select/drop paths read full files before validation, and the docs do not explain the effective upload or WebSocket/base64 ceiling.

## Best Possible Solution

Keep this issue open and implement a focused Control UI upload-policy change. The conservative path is to use the existing `agents.defaults.mediaMaxMb` limit as the server policy, make the image hydration cap coherent with that resolved policy or clearly clamp it with a documented reason, expose the resolved upload limit and practical WebSocket/base64 ceiling through `/__openclaw/control-ui-config.json`, prevalidate paste/file-select/drop before `FileReader.readAsDataURL`, document the behavior on https://docs.openclaw.ai/web/control-ui, and add targeted Gateway/UI tests for defaults, raised-limit acceptance, oversized rejection, bootstrap exposure, and browser-side rejection before send.
