## Summary

Keep this issue open. Current main still has a hardcoded 5,000,000 byte attachment ingestion cap on the Control UI/WebChat `chat.send` path, while `gateway.controlUi` has no upload-size configuration and the browser UI has no configured prevalidation surface. The request is recent, concrete, core-owned Control UI/Gateway behavior, and the discussion includes an active contributor proposal rather than a merged or superseding implementation.

## Best Possible Solution

Keep the issue open and implement a documented, default-preserving upload policy in the core Gateway/Control UI path. A conservative first PR should add a validated `gateway.controlUi.maxUploadSizeMb` or shared chat attachment limit, wire it into config types/schema/help metadata, pass the resolved byte cap to `chat.send` attachment parsing, expose it through the Control UI bootstrap/runtime config for browser prevalidation, document the practical WebSocket ceiling, and add tests for default rejection, raised-limit acceptance, oversized rejection, and UI prevalidation.

## Risks / Open Questions

- A fix that only changes the parser cap may still fail for larger raw files unless the 25 MiB WebSocket frame limit and base64 inflation are handled or documented.
- Raising limits increases browser and Gateway memory pressure because the current Control UI path reads the full file as a data URL and sends base64 over WebSocket.
