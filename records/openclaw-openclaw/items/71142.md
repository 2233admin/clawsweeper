## Summary

Keep open. Current main only partially addresses configurable Control UI upload sizing: `chat.send` now uses the shared `agents.defaults.mediaMaxMb` cap, but image uploads still hit a separate 6 MiB parser guard, Control UI bootstrap/client code does not expose or prevalidate an effective upload policy, and the docs do not describe the configured upload limit or WebSocket/base64 ceiling.

## What This Changes

Request to make the Control UI upload size limit configurable so browser uploads can align with larger media understanding and channel media limits.

## Best Possible Solution

Choose one public upload policy, thread the resolved raw-byte limit through gateway parsing and Control UI bootstrap, add browser prevalidation before data URL reads, keep server-side MIME/byte checks authoritative, and document the effective raw upload cap plus transport ceiling on the Control UI docs page.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `agents.defaults.mediaMaxMb` above 6 MiB and send a `chat.send` image attachment larger than `MAX_IMAGE_BYTES`; current parser code and tests show it still rejects with `image exceeds size limit`.

## Solution Assessment

Unclear because there is no patch yet and commenters are still asking maintainers to choose the public config shape. The narrowest maintainable direction appears to be reusing the existing shared media cap unless maintainers explicitly want a Control-UI-specific override.
