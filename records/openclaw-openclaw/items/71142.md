## Summary

Keep open. Current main has a configurable shared attachment parser cap, but the Control UI request is still not fully implemented: image uploads above 6 MiB are still rejected server-side, the current bootstrap contract does not expose an upload policy, browser file handlers still read files before any configured-size check, and the public docs do not describe the effective upload or WebSocket/base64 ceilings. Recent related PR work appears useful but partial, so this still needs maintainer direction rather than cleanup closure.

## What This Changes

The issue requests a configurable Control UI upload-size limit so browser uploads can accept larger media consistently with agent media limits and channel behavior.

## Best Possible Solution

Choose the public upload policy surface, then implement the remaining Control UI path end to end: align server image acceptance with the approved configured limit, expose raw-byte and practical WebSocket/base64 ceilings through bootstrap, reject oversize paste/file/drop inputs before FileReader, document the behavior on https://docs.openclaw.ai/web/control-ui, and keep targeted Gateway/UI regression tests. If #72612 is merged first, follow it with the server image-cap and docs work rather than closing this issue early.

## Reproduction Assessment

Yes. Configure agents.defaults.mediaMaxMb above 6 and submit a Control UI image between 6 MiB and the configured parser cap: chat.send reaches parseMessageWithAttachments with the raised maxBytes, but the image-specific MAX_IMAGE_BYTES rejection still fails the request.

## Solution Assessment

Unclear until a maintainer chooses the public config boundary. Current main is not sufficient, and #72612 alone appears to solve only the browser bootstrap/pre-read guard slice, not the server image cap or docs contract.
