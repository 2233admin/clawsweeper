## Summary

Keep open. Current main narrowed the original 5 MB problem by routing chat attachment parsing through `agents.defaults.mediaMaxMb`, but the Control UI upload experience is still not configurable/effective end to end: image attachments still hit the separate 6 MiB `MAX_IMAGE_BYTES` guard, the bootstrap contract does not expose an upload policy to the browser, paste/file/drop handlers still read files before any size precheck, and the Control UI docs do not explain the effective limit or WebSocket/base64 ceiling.

## What This Changes

Request a configurable or otherwise consistent Control UI upload-size limit so larger phone images and other supported media can be uploaded through browser chat without diverging from backend media limits.

## Best Possible Solution

Choose a single public upload policy, thread the resolved effective byte limit through Gateway parsing and the Control UI bootstrap contract, reject oversized paste/file/drop inputs before `FileReader.readAsDataURL`, keep server-side MIME and byte checks authoritative, and document both the configured raw-byte cap and the WebSocket/base64 practical ceiling on the Control UI docs page.

## Reproduction Assessment

Yes. On current main, configure `agents.defaults.mediaMaxMb` above 6 MiB and submit an image larger than `MAX_IMAGE_BYTES` through Control UI; static source and the current regression test show `parseMessageWithAttachments` rejects it at the fixed image cap even though the shared parser cap is higher.

## Solution Assessment

Unclear because there is no patch yet and the issue thread is still waiting for maintainer direction on the public config shape. The narrowest maintainable path appears to be reusing the existing shared media cap unless maintainers explicitly want a separate Control UI override.

## Risks / Open Questions
