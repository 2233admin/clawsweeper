## Summary

Keep this issue open. Current main has moved past the old hardcoded 5 MB `chat.send` parser cap by using `agents.defaults.mediaMaxMb`, but the user-visible Control UI request is not fully implemented: images still hit a fixed 6 MiB image cap, the bootstrap contract does not expose an upload limit to the browser, paste/file/drop paths read and base64-encode files before any configured-size validation, and the docs do not explain the effective limit or WebSocket/base64 ceiling. The discussion by mayank6136 and hclsys correctly narrows the remaining work; related open PR #44814 targets an older `webui`/`agent.request` shape and does not supersede this current Control UI `chat.send` gap.

## Best Possible Solution

Keep this issue open for a focused Control UI upload-policy change. The best path is to choose the effective policy surface, preferably either the existing shared `agents.defaults.mediaMaxMb` or a new explicitly documented `gateway.controlUi.maxUploadSizeMb`; make the image parser/hydration cap coherent with that policy; expose the resolved raw-byte and practical transport limits through `/__openclaw/control-ui-config.json`; prevalidate paste, file-select, and drop before `FileReader.readAsDataURL`; document the defaults and WebSocket/base64 ceiling on the public Control UI docs page; and add targeted Gateway/UI tests for default behavior, raised-limit acceptance, oversized rejection, bootstrap exposure, and browser-side rejection before send.

## Risks / Open Questions

- Closing now would lose a real remaining product gap: high-resolution images above 6 MiB still reject before the configured shared attachment cap can help.
- Adding only a server-side knob would leave the browser reading and base64-encoding oversized files before the Gateway rejects them.
- The final design still needs maintainer choice between reusing `agents.defaults.mediaMaxMb` as the one source of truth or adding a Control UI-specific `gateway.controlUi` upload policy.
- The implementation should account for the WebSocket payload ceiling and base64 inflation so operators do not configure limits the transport cannot reliably carry.
