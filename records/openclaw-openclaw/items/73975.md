## Summary

Keep this PR open for normal maintainer review. Current main still lacks the Discord announce-session lookup opt-in, while the PR targets the existing session-backed announce path and adds focused coverage for the open Discord multi-account routing bug; I found no blocking functional or security findings in the diff.

## What This Changes

The PR opts Discord runtime and manifest metadata into session-backed A2A announce target lookup, adds regression tests for `deliveryContext.accountId` and `lastAccountId`, and records the Discord fix in the changelog.

## Best Possible Solution

Land the narrow Discord opt-in with the paired runtime/manifest metadata and regression coverage, then let the linked Discord routing issue close through the PR merge path while keeping the already-closed duplicate and reset-metadata PR as context.

## Reproduction Assessment

Yes. On current main, a Discord A2A target parsed from `agent:main:discord:channel:...` returns before `sessions.list` because the Discord plugin lacks the opt-in flag, so a stored `deliveryContext.accountId` or `lastAccountId` is not forwarded to the announce `send`.

## Solution Assessment

Yes. Opting Discord into the existing session lookup path is narrower and safer than changing the core default for every plugin, and it matches how Slack and WhatsApp already preserve account-aware announce routing.
