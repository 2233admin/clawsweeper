## Summary

Keep open. Current main still depends on `node:sqlite` FTS5 for the builtin memory keyword path, degrades to vector-only when FTS is unavailable, and does not surface FTS readiness through `openclaw doctor`; the still-open linked PR is an implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue requests restoring or clearly surfacing the broken memory-search keyword/BM25 fallback when the runtime `node:sqlite` build lacks FTS5.

## Best Possible Solution

Keep the issue open until maintainers choose and land a fix path: at minimum thread FTS availability into doctor and user-facing guidance, and separately decide whether restoring keyword/BM25 on affected Node builds warrants a vetted alternate SQLite backend or documented runtime requirement.

## Reproduction Assessment

Yes. The issue and comments provide concrete logs across macOS and Linux, and current main still reproduces the code path conceptually: an FTS5 table creation failure is caught, `fts.available` becomes false, and hybrid keyword results are skipped.

## Solution Assessment

Unclear. The open doctor PR is a narrow, maintainable visibility improvement, but it does not restore keyword/BM25 fallback; adding a replacement SQLite driver or changing backend strategy needs maintainer product and dependency review.
