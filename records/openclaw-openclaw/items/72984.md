## Summary

Keep open. Current main still does not implement this PR's intended subagent runtime model resolution, and the PR remains a plausible implementation candidate. The main blocker is review of the trust boundary around using session lineage markers for model/provider selection, plus an explicit decision on the behavior-changing subagent model precedence.

## Best Possible Solution

Keep this PR open as an active implementation candidate. The best path is to revise or review the fix so subagent runtime/read-side defaults are resolved from trusted server-owned session lineage, decide and document the intended precedence between agents.defaults.subagents.model and agents.list[].model, and land with the proposed regression coverage once the security concern is resolved.

## Risks / Open Questions

- Merging as-is could make model/provider selection depend on spawnDepth or subagentRole while those markers are accepted through the public sessions.patch protocol for subagent/acp-shaped keys; this needs a trusted/server-owned lineage invariant or a narrower detection rule.
- The precedence reorder from agent primary before global subagent default to global subagent default before agent primary is user-visible for existing configs and needs maintainer approval plus release-note or docs treatment.
- The provided PR file list includes src/config/validation.ts and extensions/qa-lab/runtime-api.ts changes outside the stated subagent model-resolution path; they should be split, justified as merge fallout, or reviewed separately before landing.
