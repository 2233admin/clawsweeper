## Summary

Keep open. Current main still does not implement the PR's intended subagent runtime model resolution: the configured subagent default is not used as a read-side fallback, the reply runtime still starts from the parent/default agent model before session state is considered, and existing tests still assert the old precedence. The PR remains a plausible implementation candidate, but it needs maintainer review for the bot-raised trust-boundary concern, the precedence behavior change, and the non-subagent files in the PR scope.

## Best Possible Solution

Keep this PR open as an active implementation candidate. The best path is to revise or review the fix so subagent runtime/read-side defaults are resolved from trusted server-owned session lineage, decide and document the intended precedence for `agents.defaults.subagents.model` versus `agents.list[].model`, split or justify unrelated files, and land with the proposed regression tests once the security concern is resolved.

## Risks / Open Questions

- Merging as-is would make model/provider selection depend on `spawnDepth` or `subagentRole`, while current session patch APIs allow clients to set those fields for subagent/acp-shaped keys; the fix should use trusted server-owned lineage or tighten mutation rules first.
- The precedence change from agent primary before global subagent default to global subagent default before agent primary is behavior-changing for existing configs and needs explicit maintainer approval plus docs/release-note treatment.
- The PR contains non-subagent `src/config/validation.ts` and `extensions/qa-lab/runtime-api.ts` changes that should be split, justified as required merge fallout, or reviewed independently.
