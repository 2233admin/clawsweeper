## Summary

Keep open. Current main still lacks the gateway/reply fallback this PR targets, so the work is not obsolete, but the current PR head still has alias-resolution regressions in both new fallback paths and needs revision before merge.

## What This Changes

The PR changes subagent configured model precedence and adds gateway/reply runtime fallback helpers and tests so subagent sessions use configured subagent defaults, plus small config-validation and qa-lab type-export adjustments.

## Best Possible Solution

Revise the PR so gateway and reply runtime fallbacks reuse the alias-aware subagent spawn resolver or a shared alias-aware helper, add alias regression coverage for both fallback paths, and settle the precedence change with matching tests, docs, and changelog.

## Reproduction Assessment

Yes. Configure agents.defaults.models with an alias such as gpt, set a subagent default to that alias, and exercise the new reply helper or an empty-model subagent session fallback; the PR parses it as <defaultProvider>/gpt instead of the configured provider/model.

## Solution Assessment

No. The direction is useful, but the current patch is not the best fix because it bypasses the existing alias-aware subagent spawn model resolution and leaves a user-visible precedence change without fully coordinated coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Resolve aliases in the reply-time subagent default:** `src/auto-reply/reply/directive-handling.defaults.ts:56`
  - body: `resolveSubagentConfiguredModelSelection` can return a configured alias such as `gpt`; the existing spawn path resolves that through the alias index, but this helper feeds the raw value to `parseModelRef`. In that configuration a subagent can boot and later persist `<parent-provider>/gpt` instead of the configured provider/model. Reuse the alias-aware spawn resolver or a shared alias-aware helper here.
  - confidence: 0.91
- **[P2] Resolve aliases in the gateway fallback:** `src/gateway/session-utils.ts:1149`
  - body: The new `resolveSessionModelRef` fallback has the same alias gap when `entry.model` is empty: a configured subagent alias is parsed as a bare model under the default provider rather than resolved through the alias index. That makes the race fallback disagree with the spawn-time model selection already covered by current tests.
  - confidence: 0.89
