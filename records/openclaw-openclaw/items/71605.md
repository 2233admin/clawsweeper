## Summary

Keep this open. Current main still only partially mitigates the report: timeout fallback now uses a fresh embedded session, but the CLI still waits on the final Gateway `agent` response, accepted `runId` frames are intentionally hidden from that caller, `agent.wait` is not yet a reply-payload replacement, and non-timeout Gateway failures still fall back to embedded execution without a daemon/state-dir guard.

## What This Changes

The issue requests reliable external `openclaw agent` invocation while a daemon is running by fixing Gateway WS `agent` timeout behavior and embedded fallback lock contention, or by documenting/refusing the unsafe mode clearly.

## Best Possible Solution

Make the external-agent path explicit and reliable: have the CLI receive the accepted Gateway `runId` quickly, wait through `agent.wait` or an additive terminal RPC that can return the final reply payload, avoid embedded fallback after the Gateway has accepted ownership of a run, and keep the #62981 fresh-session mitigation for true pre-accept timeout fallback cases.

## Reproduction Assessment

Unclear. The issue provides concrete reproduction steps and current source still contains the relevant wait/fallback mechanics, but this read-only review did not run a live daemon on macOS to reproduce the intermittent 60-second Gateway timeout.

## Solution Assessment

No. Current main is only a partial solution: timeout fallback uses a fresh session, but the main CLI still hides accepted `runId` frames behind `expectFinal: true` and still lacks a clear safe fallback contract for non-timeout Gateway failures while a daemon may own the same state directory.
