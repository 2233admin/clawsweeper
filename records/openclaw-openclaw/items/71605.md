## Summary

Keep this open. Current main has partial mitigation for the accepted-frame starvation path, but the CLI still waits on the final Gateway `agent` response, still treats Gateway failure as permission to run embedded against the same state, and embedded runs still acquire the same session transcript lock with the 10s default wait. The related #62981 covers the lock-contention subset, but it does not cover the broader Gateway dispatch/CLI integration behavior reported here.

## What This Changes

The issue asks OpenClaw to make external `openclaw agent`/Gateway WS invocation reliable by fixing Gateway `agent` dispatch timeouts, preventing embedded fallback from fighting the daemon for session locks, or documenting the limitation.

## Best Possible Solution

Keep this as an active Gateway/CLI integration bug. The desired end state is a CLI flow that treats the Gateway `agent` RPC as a fast accept/runId operation, waits explicitly through `agent.wait` for terminal output when needed, and prevents same-state embedded fallback from silently competing with a running Gateway daemon by either using an explicit alternate state directory or failing with a clear operator-facing message. The public `openclaw agent` and agent-send docs should describe the supported external-client pattern and the fallback limits.

## Repair Work Prompt

Fix the Gateway/CLI external agent invocation bug tracked by openclaw/openclaw#71605, coordinating with #62981 for the lock-contention subset. Current `openclaw agent` calls `callGateway({ method: "agent", expectFinal: true })`, so it ignores the accepted `{ runId, status: "accepted" }` response and only resolves on the final response; on Gateway failure it falls back to embedded execution against the same state/session and can contend with the daemon's session lock. Implement a narrow fix: make the CLI submit path resolve on the fast accepted Gateway response, then use `agent.wait` with the requested timeout to wait for terminal output when the CLI needs the reply. Preserve JSON/plain output, delivery behavior, idempotency/runId behavior, and existing fallback metadata. Do not fall back to embedded after a Gateway run has been accepted. For pre-accept Gateway connection/request failures, prevent silent same-state embedded fallback when a running Gateway is using the same state/session directory: either use an explicit existing isolated-state mechanism if one exists, or fail with a clear message that embedded fallback is unavailable while the Gateway owns that state. Update `docs/cli/agent.md` and `docs/tools/agent-send.md` to document the supported Gateway accept/wait behavior and fallback limitation. Add focused tests for accepted-then-wait CLI behavior, no embedded fallback after accepted run, and lock/contention-safe fallback handling. Do not change bundled plugin/provider/channel policy or unrelated Gateway starvation fixes in this PR.
