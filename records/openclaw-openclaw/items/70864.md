## Summary

Keep open: current main does not implement this scoped mention-pattern policy, and the PR branch still has merge-blocking contract bugs. The latest head fixes prior Slack/Matrix review items, but several account-backed channels now expose account-scoped `mentionPatterns` without passing the resolved account policy into runtime evaluation, and IRC's public key is split between incompatible schema shapes.

## What This Changes

The PR adds a shared mention-pattern policy resolver/config surface, exposes global/agent/provider scoped modes in schemas/docs/SDK runtime, and gates configured regex mention matching across bundled channel handlers while preserving native mentions.

## Best Possible Solution

Before merge, align the config contract with runtime behavior: every channel that exposes account-level `mentionPatterns` should either pass the resolved account policy into `resolveMentionPatternsEnabled` or not expose that account-scoped key, and IRC should use one compatible shape for its existing `mentionPatterns` key or introduce a non-conflicting policy key.

## Reproduction Assessment

Yes. A named BlueBubbles, LINE, Mattermost, Nextcloud Talk, Signal, Telegram, WhatsApp, Zalo, or IRC account with account-scoped `mentionPatterns.mode: "deny"` and a matching global pattern will still evaluate regex mentions wherever the handler calls the resolver without `providerPolicy`.

## Solution Assessment

No. The shared resolver is a maintainable direction, but this PR is not yet the best implementation because schema exposure and runtime evaluation disagree for named accounts and for IRC's existing key.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass account mention policies into channel handlers:** `extensions/bluebubbles/src/monitor-processing.ts:1089-1094`
  - body: `bluebubblesAccountSchema` now accepts `channels.bluebubbles.accounts.<id>.mentionPatterns`, and account resolution already merges that into `account.config`, but this new resolver call only reads the top-level `channels.bluebubbles.mentionPatterns`. Named account `mode`, `allowIn`, or `denyIn` settings are silently ignored; pass `account.config.mentionPatterns` as `providerPolicy` here and apply the same fix to the other account-backed handlers using this call shape.
  - confidence: 0.9
- **[P2] Keep IRC mentionPatterns schema compatible:** `src/config/zod-schema.providers-core.ts:1226`
  - body: This changes core IRC `mentionPatterns` from the existing `string[]` trigger-pattern field into the new policy object, but the IRC plugin schema and exported types still define the same key as `string[]`. That leaves one public config key with incompatible shapes across validation/metadata/runtime and can reject existing IRC configs or the new policy shape depending on which schema is used. Use a compatible policy key or update all IRC schemas/types together without breaking the existing array contract.
  - confidence: 0.84
