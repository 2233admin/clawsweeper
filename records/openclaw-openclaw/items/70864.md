## Summary

Keep open. The PR is directionally useful and current main does not already provide the scoped mention-pattern policy, but the latest diff appears to fail open for required-mention group channels when configured patterns are denied.

## What This Changes

The PR adds scoped mention-pattern policy controls across bundled channel runtimes, config/schema/help metadata, docs, SDK exports, changelog, and tests.

## Best Possible Solution

Keep the shared resolver approach, but update the PR so deny mode suppresses configured pattern evidence without disabling required-mention enforcement on pattern-only channels. Add targeted tests for at least iMessage plus one runtime path such as BlueBubbles or Signal, then land the source PR after CI/Testbox stays green.

## Reproduction Assessment

Yes. On the PR head, the added iMessage test path uses a group with `requireMention: true`, global `mentionPatterns: ["@openclaw"]`, and provider policy `{ mode: "deny" }`; the diff expects dispatch with `effectiveWasMentioned=false`, which demonstrates the fail-open behavior.

## Solution Assessment

No. The shared resolver and SDK exposure are the right shape, but the implementation should keep required mention gating fail-closed when configured patterns are present but denied, while still preserving native mentions and authorized command bypasses.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep deny mode from bypassing required mentions:** `extensions/bluebubbles/src/monitor-processing.ts:1089-1097`

Keep the policy from counting regex matches without turning required mention gating off.
