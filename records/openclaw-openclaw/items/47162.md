## Summary

Keep open. Current main still lacks Python and shell coverage in the core skill scanner, so the PR is not obsolete; however, the proposed scanner changes still leave concrete bypasses that should block merge until revised.

## What This Changes

The PR extends `src/security/skill-scanner.ts` and its tests to scan `.py`, `.sh`, and `.bash` files, add Python and shell detection rules, and add per-line-rule include/exclude extension guards.

## Best Possible Solution

Revise this PR, or replace it with an equivalent focused scanner-hardening patch, so Python and shell coverage includes extension-based files plus extensionless shebang or manifest entrypoints, Python command-exec matching catches variable payloads without reintroducing noisy subprocess false positives, and regression tests cover the bypasses and false-positive boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Scan extensionless shell entrypoints:** `src/security/skill-scanner.ts:51`
  - body: The patch only adds `.sh` and `.bash` to the scannable extension set, while `isScannable()` still relies on `path.extname()` and both directory and forced-entry scans skip non-scannable files. Common shebang entrypoints such as `bin/install` or `postinstall` remain skipped, so the new shell rules miss a common package-script shape.
  - confidence: 0.93
- **[P2] Match variable Python command payloads:** `src/security/skill-scanner.ts:189`
  - body: The `os.system`/`os.popen`/`os.exec*` arm requires the first argument to be a quoted literal. A script can assign the command to a variable and call `os.system(cmd)`, which evades the new critical Python command-exec rule despite being the same dangerous behavior.
  - confidence: 0.88
