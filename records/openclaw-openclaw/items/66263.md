## Summary

Keep open. Current main still leaves the Codex-native compaction accounting gap: the native app-server compaction result has no post-compaction token snapshot, gateway manual compaction clears stored totals without recording an explicit stale/unknown state, and transcript fallback can still recover pre-compaction assistant usage as fresh. The merged related #64669 work addresses cumulative token projection, but not this compaction-boundary path.

## What This Changes

The issue asks for Codex native app-server compaction to capture `tokensAfter` when available and otherwise mark post-compaction context usage unknown or stale so status and transcript fallback do not revive pre-compaction token totals.

## Best Possible Solution

Implement the Codex-native compaction boundary at the plugin/gateway seam: capture a trustworthy post-compaction usage snapshot when the app-server emits one, persist an explicit stale/unknown context state when it does not, and make transcript-backed status/listing ignore pre-boundary usage for current context percentages while preserving cost/cumulative accounting separately.
