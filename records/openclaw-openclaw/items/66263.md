## Summary

Current main still leaves the reported Codex native compaction accounting gap open. The native compaction path reports success without a post-compaction token snapshot, the generated app-server compacted notification has no token field, and session/status fallback can still recover full-transcript usage as fresh after manual compaction clears stored totals. The merged related #64669 work addresses cumulative token projection but does not implement this issue's compaction boundary or unknown/stale post-compaction state.

## What This Changes

The issue requests Codex native compaction to record reliable post-compaction token usage when available and otherwise preserve an unknown/stale context-usage state across transcript and status fallbacks.

## Best Possible Solution

Implement Codex-native compaction accounting at the plugin/gateway boundary: capture a reliable post-compaction usage event when the app-server provides one, persist an explicit unknown/stale state when it does not, record a compaction boundary that transcript fallback honors, and keep status/UI displays from showing precise fresh percentages until a trustworthy post-compaction snapshot exists.
