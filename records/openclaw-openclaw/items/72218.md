## Summary

Keep this PR open, but not merge-ready. Current main still has the heartbeat final-payload bypass the PR targets, while the branch can strip heartbeat ack text in message mode before the heartbeat runner applies `ackMaxChars` suppression.

## What This Changes

This PR changes `buildReplyPayloads` so heartbeat final payloads pass through `HEARTBEAT_OK` sanitation, keeps Bun socket error expansion out of heartbeat runs, and adds focused payload-builder tests.

## Best Possible Solution

Keep the narrow payload-layer fix, but preserve heartbeat ack semantics: exact OK-only heartbeat final payloads should be suppressed before channel delivery, while heartbeat replies that include `HEARTBEAT_OK` plus short ack text must still reach heartbeat-mode `ackMaxChars` logic or an equivalent covered branch.

## Reproduction Assessment

Yes. The original bug has a direct code-path reproduction through `buildReplyPayloads({ isHeartbeat: true, payloads: [{ text: "HEARTBEAT_OK" }] })` on current main, and the PR regression follows from `HEARTBEAT_OK all good` being stripped in message mode before heartbeat-mode suppression runs.

## Solution Assessment

No, not as written. Removing the bypass is the right direction, but applying message-mode stripping to all heartbeat final payloads is not the safest fix because it can erase the token before heartbeat-specific ack suppression sees it.
