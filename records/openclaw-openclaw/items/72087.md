## Summary

Keep this open. The report has a strong differential reproduction, a second macOS reproduction on the latest shipped release, and current main still shows the same entry/bootstrap and OpenAI Codex image-provider boundaries without a clear fix for the entry-path-only failure.

## What This Changes

The issue asks for `openclaw infer image generate --model openai/gpt-image-2` through the packaged entrypoint to behave the same as direct `runCli()` and direct OpenAI Codex OAuth image-provider calls.

## Best Possible Solution

Add a focused fix in the entry/bootstrap or shared provider-network path so packaged CLI invocation preserves the same HTTP dispatcher/auth behavior as direct `runCli()` and direct provider calls, then lock it with an entrypoint-level regression rather than another provider-only unit test.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: the report gives an entrypoint command plus a lower-layer success matrix on Linux, and a second commenter reproduced the same failure on macOS with v2026.4.27.

## Solution Assessment

Is this the best way to solve the issue? Yes. The maintainable fix should target the shared entry/main bootstrap behavior and prove `dist/entry.js`-style invocation, because the provider and guarded HTTP helper paths already have direct evidence of working independently.
