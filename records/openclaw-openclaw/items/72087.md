## Summary

Keep open. Current main still has a distinct `src/entry.ts` main-module bootstrap before `runCli`, `infer` remains a network-capable CLI path with proxy/global Undici setup, and the OpenAI image provider still supports the Codex OAuth `openai/gpt-image-2` path separately. I found provider and Google CLI coverage, but no current source/test/release evidence proving the reported Linux `dist/entry.js`/`openclaw.mjs` Codex OAuth image-generation failure is fixed.

## Best Possible Solution

Keep this open and investigate the `src/entry.ts` main-module path against a Linux host with OpenAI Codex OAuth and dual-stack DNS. The best fix should make `openclaw infer image generate --model openai/gpt-image-2` behave the same as direct `runCli(argv)` and direct provider calls, without weakening proxy/SSRF behavior, then add focused regression coverage around entry bootstrap plus a live/manual Codex OAuth image-generation smoke.
