## Summary

Keep this open. The report gives a strong differential reproduction across packaged entry invocation versus direct `runCli()` and provider calls, current `main` still has the same distinct entry/main bootstrap boundary, and the merged adjacent OpenAI Codex OAuth transport fix does not touch the entrypoint or OpenAI image-generation path.

## What This Changes

The issue asks OpenClaw to fix the packaged CLI entry/main path so `openclaw infer image generate --model openai/gpt-image-2` with OpenAI Codex OAuth behaves like direct `runCli()` and direct provider calls.

## Best Possible Solution

Make packaged `openclaw.mjs`/`dist/entry.js` invocation preserve the same network and auth behavior as direct `runCli()` and direct OpenAI image-provider calls, then cover that boundary with a regression test and an optional live Codex OAuth smoke when credentials are intentionally available.

## Solution Assessment
