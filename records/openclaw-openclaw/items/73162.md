## Summary

Keep open for maintainer review. Current main and the latest tagged release still cap Slack Socket Mode reconnects at 12 attempts, while this PR's one-line change matches the existing provider contract where `maxAttempts: 0` means unlimited and non-recoverable Slack auth errors still fail fast.

## What This Changes

The PR changes Slack Socket Mode reconnect policy by setting `maxAttempts` from `12` to `0` in `extensions/slack/src/monitor/reconnect-policy.ts`.

## Best Possible Solution

Land a narrow Slack-owned reconnect-policy fix that keeps the gateway retrying transient Socket Mode disconnects indefinitely, preserves fail-fast handling for permanent credential errors, and leaves disconnected-status hardening as a separate follow-up if maintainers want it.

## Reproduction Assessment

Yes. The current-main source and linked bug discussion give a high-confidence reproduction path: trigger 12 consecutive Slack Socket Mode disconnect/start failures and the provider exits through the finite reconnect cap while the gateway remains otherwise usable.

## Solution Assessment

Yes. Changing the policy constant is the narrowest maintainable fix because the provider already defines `0` as the unlimited sentinel, already logs `∞`, and already separates permanent auth errors from transient socket failures.
