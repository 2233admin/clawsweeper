## Summary

Keep open. Current main has mitigations for this bug family, including a package-update guard when running inside the gateway service and the merged launchd split-brain repair from #72616, but it does not yet deliver a working built-in auto-update handoff for managed macOS package installs. The gateway auto-updater still invokes `openclaw update --yes --channel ... --json` through the piped command runner, package updates from the gateway service now fail early instead of applying, and the service refresh path still runs `gateway install --force` before the prepared detached restart script is spawned.

## What This Changes

The issue requests a safe macOS auto-update handoff so a gateway-owned update subprocess cannot lose stdout/stderr and crash when launchd bootout stops the parent gateway before restart recovery is active.

## Best Possible Solution

Define a supported auto-update handoff for managed package installs that runs outside the live gateway service dependency chain, keeps stdio/logging independent of the parent gateway, and preserves the current safety guard against replacing the active dist tree from the managed service process.

## Reproduction Assessment

Yes for the remaining current-main behavior: with gateway service markers present, a package update is rejected before it can apply, so the built-in gateway auto-updater cannot complete a managed package update. The original bootout/EPIPE crash path is not reproduced directly on current main because that guard stops the package update earlier.

## Solution Assessment

No. The current behavior is a useful safety mitigation, but it is not the best final fix because it avoids the outage by preventing the advertised package auto-update path from completing instead of providing a detached, recoverable update handoff.
