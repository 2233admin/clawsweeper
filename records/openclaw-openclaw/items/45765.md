## Summary

Keep open: current main still deterministically appends another `.openclaw` segment when `OPENCLAW_HOME` already resolves to a state-directory basename, and the open related PR has not landed.

## What This Changes

The issue requests that an explicit `OPENCLAW_HOME` ending in `.openclaw` write config and state under that directory instead of `$OPENCLAW_HOME/.openclaw`.

## Best Possible Solution

Land a focused core path-resolution fix that bypasses the extra `.openclaw` append only when the explicit home path already names a known state directory, covers both `resolveStateDir()` and `resolveConfigDir()`, preserves ordinary parent-home semantics, and adds regression tests for positive and negative cases.

## Reproduction Assessment

Yes. A static source trace is enough: set `OPENCLAW_HOME` to a path whose basename is `.openclaw`, then both the state and config helpers append another `.openclaw` segment on current main.

## Solution Assessment

No. Current main is not fixed, and the open PR's direct-use approach needs maintainer review because the narrower maintainable fix should cover both helpers while preserving non-state-dir `OPENCLAW_HOME` behavior.
