## Summary

Keep open. Current main has a related Feishu WebSocket repair from PR #72411, but it still does not fully resolve this issue's reported startup failure: OpenClaw passes `wsConfig` through a cast even though the resolved SDK constructor does not expose that field, and the monitor only retries when `WSClient.start()` rejects. In @larksuiteoapi/node-sdk 1.62.0, `start()` begins connection work without awaiting the handshake; real startup failure is surfaced through `onError` or the SDK's higher-level channel wrapper. The Lark-domain workaround from the discussion also remains relevant because setup/docs still default Feishu/Lark accounts to WebSocket mode.

## Best Possible Solution

Keep this issue open. The best fix is to align the Feishu WebSocket supervisor with @larksuiteoapi/node-sdk 1.62.0's real lifecycle contract: use `onReady`/`onError` or the SDK's `LarkChannel` connection wrapper so OpenClaw can distinguish initial handshake success from retry exhaustion, remove or replace the unsupported `wsConfig` constructor workaround unless upstream documents it, add a regression test where `start()` returns before asynchronous connection failure, and document or enforce the Lark international domain/webhook limitation if WebSocket remains unsupported there.

- No live Feishu/Lark WebSocket handshake was run in this read-only review, so the exact split between platform/domain mismatch and SDK startup behavior still needs maintainer or reporter validation.
