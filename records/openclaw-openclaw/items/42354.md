## Summary

Keep open. Current main has the heartbeat-config shape from the related PRs, but the resolved @larksuiteoapi/node-sdk@1.62.0 constructor does not expose or consume `wsConfig`, and OpenClaw still treats `WSClient.start()` returning as proof that the first WebSocket handshake succeeded. The Lark-domain workaround from the issue discussion also remains relevant because docs/config still default Feishu/Lark to WebSocket mode.

## Best Possible Solution

Keep this issue open. The best fix is to align the Feishu WebSocket transport with the SDK lifecycle by waiting for `onReady`/`onError` or reusing the SDK `LarkChannel` handshake wrapper, then add a regression test where `start()` returns before asynchronous initial connection failure. Maintainers should also decide and document the Lark international path: either force/validate webhook mode when WebSocket is unsupported or provide a clear early error with the required config.
