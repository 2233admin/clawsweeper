## Summary

Keep open. Current main partially carries the heartbeat-config idea from #45674 via the merged #72411 work, but the Feishu WebSocket monitor still treats `WSClient.start()` as a handshake/connection promise even though the resolved SDK reports first-connect success or failure through callbacks. The Lark-domain workaround from the discussion also remains relevant because docs/config still default Feishu/Lark to WebSocket mode.

## Best Possible Solution

Keep this issue open. The best fix is to align the Feishu WebSocket transport with @larksuiteoapi/node-sdk 1.62.0 by waiting for `onReady`/`onError` or reusing the SDK's `LarkChannel` handshake wrapper, then add a regression test where `start()` returns before asynchronous connection failure. Maintainers should also decide whether Lark international should force or document webhook mode when WebSocket is unsupported, and remove or replace the unsupported `wsConfig` workaround unless upstream documents it.

## Risks / Open Questions

- No live Feishu/Lark WebSocket handshake was run in this read-only review, so the exact split between platform/domain mismatch and SDK lifecycle handling still needs maintainer or reporter validation.
- Closing now could mark the default Feishu/Lark WebSocket startup path fixed while OpenClaw still cannot observe SDK asynchronous initial-connect failure or retry exhaustion.
- The current `wsConfig` field may be ignored by the resolved SDK constructor, so relying on it could leave the original startup failure unresolved for some deployments.
