## Summary

Keep open. Current main still lacks the requested direct single-job cron read path: the CLI inspection path pages through `cron.list`, the gateway method registry/scopes/protocol/server handler do not expose `cron.get`, and the agent cron tool has no `get` action. The prior implementation PR #67677 is closed unmerged, so it is useful prior art rather than shipped behavior.

## What This Changes

This issue requests an additive single-job cron read API through `openclaw cron get <id>` and gateway RPC `cron.get`, returning one stored cron job by id with a clean not-found error.

## Best Possible Solution

Add the direct core read API as an additive contract: define and validate `cron.get` params, register it in gateway methods and read scopes, implement a server handler backed by `CronService.getJob(id)`, expose `openclaw cron get <id>`, update the agent cron tool if agent workflows remain in scope, and cover the new path in focused CLI/protocol/server/tool tests and docs without changing scheduling, execution, delivery, or `cron list` behavior.
