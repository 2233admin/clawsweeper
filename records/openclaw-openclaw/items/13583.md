## Summary

Keep this issue open. Current main has adjacent hooks for pre-tool blocking, final-answer revision, reply dispatch, and outbound message cancellation, but it still does not implement the requested fail-closed pre-response responseGate with mandatory tool-call accounting, typed validators, two-phase planning, bounded retry/fallback behavior, or audit records. The discussion also identifies useful complementary plugin work while preserving the core gap: existing plugins can gate tools or validate/filter output, but cannot mechanically prevent final response delivery until required tools and validators pass.

## What This Changes

Request to add a pre-response enforcement layer that mechanically blocks final answers until mandatory tool calls and configured validators pass, preferably through a two-phase planning/validation/response protocol with auditability.

## Best Possible Solution

Keep this issue open as the canonical response-gate policy-as-code design item. The best maintainer path is a versioned core/plugin API for mandatory tool-call accounting, typed response validators, fail-closed final-response gating, bounded retry/fallback behavior, and audit records; domain-specific policy packs can then live in plugins or ClawHub.
