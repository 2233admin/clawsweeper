## Summary

Keep this issue open. Current main still has adjacent mechanisms for pre-tool blocking, final-answer revision, reply dispatch, and outbound message cancellation, but it does not implement the requested fail-closed responseGate with mandatory tool-call accounting, typed validators, two-phase planning, bounded retry/fallback behavior, or audit records. The discussion is useful because it identifies complementary plugin work while preserving the core gap: plugins can gate tools or filter/cancel output, but cannot yet mechanically prevent final response delivery until required tools and validators pass.

## What This Changes

Request to add a pre-response enforcement layer that blocks final answers until mandatory tool calls and configured validators pass, ideally through a two-phase planning, validation, response protocol with auditability.

## Best Possible Solution

Keep this open as the canonical response-gate policy-as-code design item. The desired end state is a versioned core/plugin API for mandatory tool-call accounting, typed response validators, fail-closed final-response gating, bounded retry/fallback behavior, and audit records, with domain-specific policy packs implemented as plugins or ClawHub packages.
