## Summary

Keep open. The report is fresh and concrete, and current main still routes both affected commands through active gateway probe/RPC paths. Some post-release changes made default `status` lighter, but I found no code, test, or release evidence proving the WSL2 hang for both `openclaw status` and `openclaw cron list` is fixed. The related #73381 is useful context, but it covers the gateway `agent` sendPolicy path, not the inspected `status` or `cron.list` methods.

## Best Possible Solution

Keep this issue open. Ask the reporter for targeted diagnostics: `openclaw status --debug --timeout 5000`, `openclaw cron list --timeout 5000 --json`, gateway logs around the hang, whether either command eventually exits with `gateway timeout`, and whether behavior changes after gateway restart or with a clean config/device token.
