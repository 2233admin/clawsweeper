## Summary

Keep open. The report is fresh, concrete, and corroborated by another user seeing `openclaw status` slow/time out after upgrading to 2026.4.26. Current main has post-release mitigations for the `status` reachability path, but I did not find code, tests, changelog, or release evidence proving the combined `openclaw status` plus `openclaw cron list` hang/slow path is fixed. `cron list` still traverses the Gateway `cron.list` RPC and delivery-preview resolution, and the related #73381 remains an agent send-policy issue rather than a proven duplicate for these commands.

## Best Possible Solution

Keep this issue open. Reproduce against current main or the next release with `openclaw status --debug --timeout 5000`, `openclaw status --json --timeout 5000`, and `openclaw cron list --json --timeout 5000`, plus redacted Gateway logs around the slow call. If the root cause is the status probe/auth path, extend the current reachability fallback; if it is `cron.list` delivery preview or scheduler read behavior, bound or degrade that Gateway handler and add a regression that exercises the CLI-to-Gateway path rather than only direct service reads.
