## Summary

Keep open. The PR targets a real Feishu inbound-context gap that current main still has, but the branch is dirty against main and the proposed history-media merge is not ready as-is because it enforces the media cap after downloads and needs rebase-aware integration with current audio preflight indexing.

## What This Changes

The PR preserves raw Feishu message content on fetched message metadata and prepends best-effort media resolved from recent pending group-history entries to the inbound Feishu media payload.

## Best Possible Solution

Rebase the PR onto current main, keep the rawContent plumbing, merge pending-history media before audio preflight or recompute media indexes afterward, enforce the item/download cap before resolving rich post assets, and add focused Feishu tests for a pre-mention image plus a multi-image post cap case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Cap rich history media before downloading:** `extensions/feishu/src/bot.ts:802-805`
  - body: The 10-item guard runs after `resolveFeishuMediaList` returns, but that helper downloads every embedded image/media in a rich `post` before returning. A single pending post can still fetch far more than 10 assets, defeating the latency and cost bound this change promises.
  - confidence: 0.91
- **[P2] Keep audio indexes aligned after prepending history media:** `extensions/feishu/src/bot.ts:815-827`
  - body: On current main, audio preflight computes `MediaTranscribedIndexes` from the triggering-message media list before this history block would prepend older media. If a triggering audio message also has pending history media, the final `MediaPaths` order shifts and the transcribed index can point at the wrong item unless the merge happens before preflight or the index is recomputed.
  - confidence: 0.78
