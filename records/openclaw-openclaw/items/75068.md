## Summary

Keep open. Current main and the latest release still preflight-reject Brave `llm-context` `freshness`, `date_after`/`date_before`, and `ui_lang`; Brave's current LLM Context contract supports `freshness` including custom date ranges, and an open related PR already targets the freshness/date portion of the bug. https://github.com/openclaw/openclaw/issues/75068

## What This Changes

The issue asks OpenClaw to stop falsely rejecting Brave `web_search` `llm-context` freshness/date filters, and to clarify or adjust `ui_lang` and `count` behavior for that mode.

## Best Possible Solution

Keep this issue open as the canonical bug report, then review or replace PR #51005 so the Brave plugin unblocks validated freshness/date filters for `llm-context`, adds focused provider tests, and updates the Brave Search and web-search docs to match the settled parameter contract.

## Reproduction Assessment

Yes. A Brave `web_search` call in `llm-context` mode with `freshness`, `date_after`, or `date_before` deterministically hits local `unsupported_*` returns in `executeBraveSearch` before any Brave request is sent.

## Solution Assessment

Unclear. Removing the freshness/date preflight guards is the narrow maintainable fix, but `ui_lang` should probably remain rejected or be documented separately unless Brave formally supports it, and `count` can be handled as a distinct improvement.
