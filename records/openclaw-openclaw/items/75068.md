## Summary

Keep open. Current main still rejects Brave `llm-context` freshness and date filters before the Brave request is sent, and the latest release commit still contains the same guards. The linked open PR is the active implementation path for the documented freshness/date portion, while `ui_lang` is still a contract/product decision because Brave's LLM Context docs do not list it.

## What This Changes

The issue requests allowing Brave `llm-context` web_search calls to use documented freshness and date-range filters, and asks maintainers to decide whether `ui_lang` and `count` should pass through or be clarified as unsupported.

## Best Possible Solution

Keep this issue open until maintainers review, land, or replace #51005 for the documented freshness/date behavior, with tests proving URL parameters and cache keys. Treat `ui_lang` as unsupported unless Brave documents it or maintainers explicitly approve passthrough, and decide separately whether `count` should be included in the same Brave LLM Context fix.

## Reproduction Assessment

Yes. With any configured Brave API key value, calling Brave `llm-context` web_search with `freshness`, `date_after`/`date_before`, or `ui_lang` deterministically returns the local `unsupported_*` response before any network request is made.

## Solution Assessment

Unclear for the whole report. The best narrow fix is to unblock validated freshness/date filters and update docs/tests; `ui_lang` passthrough is not clearly the best solution because the upstream endpoint documentation does not include that parameter.
