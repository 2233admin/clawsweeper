## Summary

Keep open. Current main and latest release still locally reject Brave `llm-context` `freshness`, `date_after`/`date_before`, and `ui_lang` before sending a request; the dependency contract supports freshness/range filtering and `count`, while `ui_lang` remains undocumented. Related PR #51005 is open and targets the freshness/date part, so this should stay open for maintainer review rather than be closed or assigned to a competing repair lane.

## What This Changes

The issue requests that Brave `web_search` in `llm-context` mode stop returning local `unsupported_*` errors for freshness, date range, and `ui_lang` parameters, with optional `count` parity.

## Best Possible Solution

Keep this issue open as the canonical bug report until PR #51005 or a narrow replacement lands. The desired end state is a Brave plugin runtime, tests, and docs contract that pass supported freshness/date filters through to LLM Context, handle `count` if maintainers want parity with Brave's documented parameter set, and explicitly document or preserve the `ui_lang` boundary.

## Reproduction Assessment

Yes. Configure Brave `web_search` for `llm-context` mode and call it with `freshness`, `date_after`, or `date_before`; current main deterministically returns the local `unsupported_*` response before any Brave request is sent. `ui_lang` also reproduces locally for valid region locales like `en-US`, though Brave-side support is less certain because the official docs omit it.

## Solution Assessment

Unclear. Removing the freshness/date preflight guards and converting date filters into Brave `freshness` ranges is the narrow maintainable direction; `ui_lang` should probably remain rejected or be reworded unless Brave documents support, and `count` can be a small follow-up if maintainers include it in scope.
