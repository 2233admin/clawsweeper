## Summary

Keep this PR open. Current main still does not implement the requested media-server hardening headers, and the PR remains a focused security-hardening candidate rather than obsolete. It needs rework for the current media handler shape plus maintainer review on the exact CSP and cache policy.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase or rework the hardening onto `createMediaRequestHandler`, add shared assertions for success and error responses, and make an explicit media-server CSP/cache policy decision that accounts for current active-content attachment handling and one-shot hosted URL fallback behavior.

## Risks / Open Questions

- Merging the PR verbatim would target the old `attachMediaRoutes` shape rather than current `createMediaRequestHandler`.
- The proposed `Content-Security-Policy` includes `style-src 'unsafe-inline'`; current active-content attachment handling may allow a stricter media-specific CSP, so this should be an explicit maintainer decision.
- The proposed `Cache-Control: no-store` may make one-shot hosted media URLs more fragile for channel paths that upload from URL and fall back to the original media URL on upload/fetch errors, such as Tlon.
- The PR’s provided test coverage asserts the new headers on the success path only; error responses should also be covered if the intended contract is all media responses.
