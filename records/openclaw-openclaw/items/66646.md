## Summary

Keep open. Current main has partial fixes for stale lock cleanup, raw `infer model run`, and timeout misclassification, but the central active-lock cascade is still present: `agentCommand()` runs each candidate through `runWithModelFallback()`, the embedded attempt acquires the session write lock inside that candidate attempt, and `runWithModelFallback()` still continues to later candidates for unrecognized errors such as direct `SessionWriteLockTimeoutError`.

## Best Possible Solution

Keep this issue open and add a targeted non-model-fallback path for `SessionWriteLockTimeoutError` or nested `OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` at the fallback wrapper or agent-attempt boundary. Add a regression in `src/agents/model-fallback.test.ts` with multiple candidates where the run callback throws a session lock timeout, asserting the callback runs once and the original/session-busy infrastructure error surfaces. Keep the existing stale-lock cleanup and raw `infer model run` improvements as complementary fixes.
