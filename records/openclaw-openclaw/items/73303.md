## Summary

Keep open. Current main and the latest tagged release only bound the restart drain to a five-minute default; they still intentionally allow the Gateway to wait on active work before `server.close`, which matches the reported multi-minute pre-start gap under LaunchAgent supervision.

## What This Changes

The issue asks OpenClaw to investigate and mitigate multi-minute macOS LaunchAgent Gateway restart delays in v2026.4.26, including shutdown drain behavior, plugin/channel teardown visibility, and possible workarounds.

## Best Possible Solution

Define and implement the intended Gateway restart policy: either make operator/service-manager restarts fast by default with explicit cancellation/progress semantics, or expose a clear safe-drain versus force-restart path while preserving reply delivery guarantees for active work.

## Reproduction Assessment

Yes for the likely mechanism: current code and fake-timer tests show restart requests can wait until the 300-second default deferral expires while active work remains. The exact LaunchAgent machine and external plugin mix from the report were not live-reproduced.

## Solution Assessment

No. The current shipped behavior is a partial mitigation and documented knob, but it still permits multi-minute restart delays; the maintainable fix needs an explicit product decision on restart semantics.
