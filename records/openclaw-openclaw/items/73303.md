## Summary

Keep open. Current main still allows a restart to spend up to 300 seconds draining active tasks, embedded runs, and bundled runtime dependency installs before `server.close` runs, which matches the reported multi-minute pre-start gap. The latest shipped tag did not remove the behavior, and current main only bounds it rather than resolving the operational restart latency.

## What This Changes

Request to investigate macOS LaunchAgent Gateway restarts that can wait several minutes after SIGTERM before a new process starts, with possible plugin/channel teardown and model-switch latency involvement.

## Best Possible Solution

Keep this open for a Gateway lifecycle policy decision: manual/service-manager restarts should either become fast by default with explicit cancellation/progress semantics, or expose a clearly documented safe-drain versus force-restart path while preserving reply delivery and active-run protection.

## Reproduction Assessment

Yes for the likely mechanism: current main tests and code show that SIGTERM-with-restart-intent and config/SIGUSR1 restarts can wait for active work until the configured/default deferral timeout. The exact macOS LaunchAgent environment from the report was not live-reproduced.

## Solution Assessment

No. The current bounded 300-second default and `deferralTimeoutMs` knob are a partial mitigation and diagnostic surface, but they are not the narrowest complete solution for users seeing manual restarts appear stuck for several minutes.
