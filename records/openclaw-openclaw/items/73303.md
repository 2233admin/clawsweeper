## Summary

Keep open. Current main still allows a managed Gateway restart to wait on active tasks, embedded runs, or bundled runtime-deps installs for the default 300000ms budget before the old process exits, which still matches the reported multi-minute LaunchAgent pre-start delay. Recent runtime-deps work may reduce one source of waits, but it does not change the restart lifecycle policy behind this report.

## What This Changes

The issue asks OpenClaw to diagnose and fix multi-minute macOS LaunchAgent Gateway restart delays seen around v2026.4.24-v2026.4.26.

## Best Possible Solution

Define the operator restart lifecycle policy explicitly, then implement a clear safe-drain versus prompt/force-restart path with progress logging and documented timeout semantics. Config reload can preserve conservative delivery guarantees, but an operator/service-manager restart should not silently look stuck for several minutes without an intentional policy choice.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the likely mechanism: current code and fake-timer tests show restart requests can wait until the 300-second default expires while active work remains. The exact LaunchAgent machine and external plugin mix from the report were not live-reproduced.

## Solution Assessment

Is this the best way to solve the issue? No. The current bounded drain prevents an infinite hang, but it still permits the reported multi-minute restart delay and needs a clearer operator restart policy or force path.
