## Summary

Keep open. Current main still has the reporter's two delivery paths for Discord thread-bound ACP `mode: "session"`: ACP bootstrap delivery sends inline to the bound thread, while `sessions_spawn` still registers the ACP run with completion announce tracking enabled. The native subagent path has a suppression guard for this same direct-delivery case, but the ACP wrapper does not.

## Best Possible Solution

Fix ACP `sessions_spawn` tracking so thread-bound `mode: "session"` runs that already deliver inline do not also request completion announce delivery. A narrow implementation would expose or infer the ACP inline-delivery plan and register the tracked run with `expectsCompletionMessage: false` only for that case, preserving run-mode and `streamTo: "parent"` behavior. Add regression coverage for Discord ACP session-mode initial and follow-up turns showing inline delivery stays enabled and no second announce is dispatched.
