## Summary

Keep open. Current main still publishes the reported default piped installer command, and the issue is paired with open closing PR #73830, so it should remain open until that PR is reviewed, replaced, landed, or closed. The installer has relevant `/dev/tty` safeguards, but there is not enough current-main proof that the WSL2 interactive pipe hang is fixed end to end.

## What This Changes

The issue reports that the documented piped `install.sh` command can hang on WSL2 Ubuntu and corrupt the streamed script when installer interaction consumes stdin.

## Best Possible Solution

Keep this issue open while #73830 is reviewed. The durable end state should make the public install path safe for interactive installer behavior, either by ensuring all interactive subprocesses use the terminal instead of the script stream or by documenting an invocation that separates the script source from prompt input, with smoke coverage aligned to that path.

## Reproduction Assessment

Unclear for the full WSL2 hang: the report gives a concrete command, platform, and failure mode, and current docs still publish that command, but this read-only pass did not run the mutating installer. Existing smoke coverage uses `--no-prompt`, so it does not reproduce the interactive path.

## Solution Assessment

No for closing now. The issue is paired with an open closing PR, and the proposed docs-only `bash -s --` approach needs review because it does not inherently separate stdin from the streamed script.
