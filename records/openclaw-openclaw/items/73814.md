## Summary

Keep open. Current main still documents the reported `curl | bash` installer path, the live hosted installer still matches the reporter's `warn_shell_path_missing_dir` line-number evidence, and the linked closing PR #73830 is open and unmerged, so this issue should remain open until that fix or a stronger installer-side fix lands.

## What This Changes

The issue reports that the documented piped installer command can hang on WSL2 Ubuntu and eventually execute a truncated `warn_shell_path_missing_dir` function name because interactive installer work may consume the `curl | bash` script stream.

## Best Possible Solution

Keep the issue open until maintainers either land #73830 with any needed follow-up coverage or ship an installer-side hardening that proves the hosted `install.sh` cannot consume its own script stream. The end state should align every public installer command, the installer help text, and the hosted artifact, then verify the documented flow in a disposable Linux/WSL TTY smoke.

## Reproduction Assessment

Unclear: the reporter provided concrete WSL2 Ubuntu steps and the live hosted script still matches the reported line-number evidence, but this review did not execute the installer because it would mutate the environment. A high-confidence reproduction should run the documented command in a disposable WSL/Linux TTY and capture whether prompts or child commands read from the script pipe.

## Solution Assessment

Unclear: `bash -s --` in #73830 is a plausible mitigation for the documented command, but current main still has several plain `| bash` examples and no interactive smoke proving the live hosted path is fixed. The safer solution is to align docs plus installer/help examples and add a targeted TTY smoke or harden all prompt/child-process stdin handling in the script.
