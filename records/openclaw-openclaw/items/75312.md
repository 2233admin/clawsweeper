## Summary

Keep open. Current main still matches the reported failure shape: the memory-wiki shared search path resolves an active memory manager and calls `.search(...)` without a structural guard or fallback, and the related items cover different search work rather than superseding this bug.

## What This Changes

Make `wiki_search` gracefully fall back when shared memory search resolves an unavailable, malformed, or failing active manager for `corpus="memory"` or `corpus="all"`.

## Best Possible Solution

Add a narrow defensive fallback in `extensions/memory-wiki/src/query.ts`: validate that a resolved active memory manager has a callable `search`, catch shared-memory search failures, return wiki results already computed for `corpus="all"`, and return no memory results when shared memory is unavailable. Cover malformed-manager and throwing-search cases in `extensions/memory-wiki/src/query.test.ts`.

## Reproduction Assessment

Yes. Configure memory-wiki with `search.backend="shared"` and `corpus="all"` or `"memory"`, mock `getActiveMemorySearchManager` to return a truthy manager without `.search`, then call `searchMemoryWiki`; current main reaches the direct `.search` call and throws the reported TypeError shape.

## Solution Assessment

Yes. The best fix is at the `memory-wiki` consumer boundary because both bundled and third-party memory plugins flow through the active memory facade, and a bad shared-memory result should not erase wiki-only results.
