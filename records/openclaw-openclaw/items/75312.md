## Summary

Keep this issue open. Current main still has the reported failure shape: `memory-wiki` enables shared memory search for `corpus="memory"` or `"all"`, resolves a truthy active memory manager, and calls `.search(...)` without a structural guard or local fallback, so a malformed or incompatible active memory plugin can abort `wiki_search` after wiki results were already computed.

## What This Changes

Fix `wiki_search` so shared-backend `memory`/`all` searches do not throw when the active memory manager lacks or fails `.search`, and instead degrade to wiki-only or empty memory results.

## Best Possible Solution

Add a narrow defensive fallback in `extensions/memory-wiki/src/query.ts`: validate that the resolved active memory manager has a callable `search`, catch shared-memory search failures, and preserve already-computed wiki results for `corpus="all"` while returning no memory results for unavailable shared memory. Cover both malformed-manager and throwing-search cases in `extensions/memory-wiki/src/query.test.ts`.

## Reproduction Assessment

Yes. A deterministic unit reproduction is to configure `memory-wiki` with `search.backend="shared"` and `corpus="all"`, mock `getActiveMemorySearchManager` to resolve a truthy manager without `.search`, and call `searchMemoryWiki`; current main reaches line 1217 and throws the same TypeError shape.

## Solution Assessment

Yes. The narrowest maintainable fix is at the `memory-wiki` consumer boundary, because built-in and third-party active memory plugins both flow through the SDK facade and a malformed runtime manager should not make wiki-only results disappear.
