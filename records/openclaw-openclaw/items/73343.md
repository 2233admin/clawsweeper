## Summary

Keep open. Current main still has no sender/channel identity path into `before_tool_call` or shell execution environments; the only adjacent support is native plugin-tool context, which does not cover exec, skill scripts, MCP tools, or hook-provided env injection. The request is valid but security-sensitive and extends public SDK/core contracts, so it needs maintainer design rather than cleanup closure.

## What This Changes

Expose trusted channel sender and channel identity to plugin, skill, and tool execution, preferably through an opt-in env contribution path for shell execution.

## Best Possible Solution

Add an additive, opt-in plugin-SDK/core seam that carries normalized sender, channel, account, and conversation identity into tool-hook context, then provides a typed exec-env contribution path that flows through existing gateway/node/sandbox env validation. Document the privacy/security model and add focused tests for hook context, env merging, native plugin tools, and skill/script execution.
