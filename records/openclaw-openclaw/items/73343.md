## Summary

Keep open. Current main still does not expose sender/channel identity through the generic tool-hook result, exec environment construction, or bundle MCP call path, even though sender identity is available earlier in the run and reaches some OpenClaw-owned tools. The linked open PRs are relevant implementation context, but the remaining work is a public Plugin SDK and privacy/security design decision.

## What This Changes

Request to expose channel sender identity and channel metadata to skill, shell tool, and MCP execution so downstream services can attribute actions to the channel user.

## Best Possible Solution

Add an additive, opt-in Plugin SDK design that carries normalized requester identity into tool hook context, supports explicit exec-env contribution, supports MCP caller metadata where appropriate, validates reserved env keys, documents privacy/redaction behavior, and includes focused regression coverage for hooks, exec hosts, and bundle MCP tools.

## Reproduction Assessment

Yes. A high-confidence code-inspection reproduction exists by tracing MsgContext.SenderId into run.senderId and tool creation, then observing that PluginHookToolContext, PluginHookBeforeToolCallResult, exec env construction, and SessionMcpRuntime.callTool do not carry that identity to generic plugin/tool execution.

## Solution Assessment

Unclear pending maintainer design. A plugin-driven opt-in seam is safer than automatic global OPENCLAW_* env injection, but the maintainable fix should cover hook context, exec env, MCP metadata, docs, and privacy controls together.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it asks to expose channel user identity to executable and third-party tool surfaces.

Concerns:

- **[medium] Design identity exposure as opt-in:**
  - body: Sender and channel identifiers may be personal or tenant-scoped data. Passing them through env vars or MCP metadata can expose them to subprocess logs, node hosts, sandbox runtimes, and third-party MCP servers, so any implementation should require explicit plugin/admin opt-in, documented fields, reserved-key validation, and redaction guidance.
