## Summary

Keep this issue open. Current main still does not provide the requested sender-identity-to-exec-environment seam: inbound/message surfaces carry sender identity, and native plugin-owned tool factories have a partial requesterSenderId context, but before_tool_call has no sender/channel/account identity and no env contribution result, while exec environment construction only uses process env, tool params.env, and sandbox env.

## Best Possible Solution

Design an additive core/plugin-SDK seam: carry normalized sender/channel/account identity into tool-hook context, then add a typed exec-env contribution path or documented before_tool_call env merge that works for gateway, sandbox, node-host, and skill/script execution. Reuse existing env sanitization, make identity exposure opt-in and auditable, and update plugin/skills docs plus focused tests for the security model.

## Risks / Open Questions

- Exposing sender identifiers to shell environments can leak user identity to arbitrary tools and scripts, so any implementation should be opt-in, auditable, and use the existing env validation path instead of bypassing blocked keys such as PATH.
- Because native plugin-owned tools already receive requesterSenderId, the best fix should extend or reuse that trusted context deliberately rather than adding a second inconsistent identity shape.
