## Summary

Keep this issue open. Current main still lacks the requested sender/channel identity path into skill or shell tool execution environments. SenderId exists upstream and is partially available to native plugin-owned in-process tools, but before_tool_call has no sender/channel/account identity or env contribution result, and exec environment construction still merges only process env, tool params.env, and sandbox env. The earlier ClawSweeper comment remains accurate after reviewing current main.

## Best Possible Solution

Design an additive core/plugin-SDK seam: carry normalized sender/channel/account identity into tool-hook context, then add a typed exec-env contribution path or dedicated exec env provider that merges through existing host/sandbox env sanitization. Document the security model and add focused tests for before_tool_call, exec host modes, skills, and native plugin tool context consistency.

## Risks / Open Questions

- Exposing sender identifiers through shell environments can leak user identity to arbitrary tools, scripts, sandbox/node hosts, approval records, or logs, so the implementation should be opt-in, auditable, and use existing env validation paths.
- Because requesterSenderId already exists for plugin-owned in-process tools, a fix should avoid introducing a second inconsistent identity shape and should cover gateway, sandbox, node-host, skills, and native/MCP hook relay paths deliberately.
