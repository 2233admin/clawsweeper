## Summary

Keep this PR open. Current main still has unbounded direct OpenAI Responses, Azure OpenAI Responses, and OpenAI-compatible chat-completions calls, so the proposed feature remains relevant, but the submitted patch is incomplete and should not merge as-is.

## What This Changes

The PR adds a `src/utils/providerLimiters.ts` helper configured by `LLM_MAX_CONCURRENT` and wraps two OpenAI/Azure Responses SDK calls in `src/agents/openai-transport-stream.ts`.

## Best Possible Solution

Land a revised implementation, either on this branch or as a replacement PR, that adds a tested OpenAI-compatible provider request limiter at the transport boundary, covers Responses and chat-completions paths, holds slots for the full stream lifetime, handles aborts and bad config safely, and documents the supported control surface.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P2] Wrap chat completions with the limiter:** `src/agents/openai-transport-stream.ts:1222-1224`
  - body: The PR body says `chat.completions.create` is limited, but the completions transport still calls `client.chat.completions.create(...)` directly. Requests through this path bypass `LLM_MAX_CONCURRENT` entirely.
  - confidence: 0.98
- **[P2] Hold limiter slots until streamed responses finish:** `src/agents/openai-transport-stream.ts:905-910`
  - body: The new wrapper only surrounds `client.responses.create(...)`. For streaming responses the promise resolves when the stream object is created, so the slot is released before `processResponsesStream` consumes the stream, defeating the concurrency cap under long-running streams.
  - confidence: 0.95
- **[P2] Fall back when LLM_MAX_CONCURRENT is invalid:** `src/utils/providerLimiters.ts:24`
  - body: `Math.max(1, Number(process.env.LLM_MAX_CONCURRENT || '2'))` yields `NaN` for a typo such as `LLM_MAX_CONCURRENT=two`. Since `active >= NaN` is always false, the limiter stops limiting instead of using the safe default.
  - confidence: 0.97
- **[P2] Reject queued work when the caller aborts:** `src/utils/providerLimiters.ts:19-23`
  - body: Queued entries do not carry or observe the caller's `AbortSignal`, so an already-cancelled request can remain in the queue until a slot opens and then start after the caller has given up.
  - confidence: 0.9
- **[P2] Format the new limiter and wrapped call sites:** `src/utils/providerLimiters.ts:4-36`
  - body: The new helper and first wrapped call site are flush to column 0 or under-indented. The repository format gate will reject this patch until the file and call sites are run through the repo formatter.
  - confidence: 0.92
