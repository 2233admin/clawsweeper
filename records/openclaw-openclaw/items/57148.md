## Summary

Keep open. Current main still has the unbounded subagent completion-result injection described here: the selected child output is assigned directly to the task-completion internal event, formatted into the parent prompt between untrusted-child-result markers, and passed through both queued and direct announce delivery as `internalEvents`. Existing `toolResultMaxChars` coverage is real, but it applies to live tool results, not subagent completion findings. The related #6662 cleanup appears to have changed how this is rendered, and #41312 covers retained tool-result pressure, but neither implements a subagent result-size cap.

## Best Possible Solution

Keep this issue open and implement a focused subagent completion-result budget in the announce path before `internalEvents[].result` is rendered or forwarded. The cleanest product shape is likely a documented config key aligned with existing `agents.defaults.contextLimits` and per-agent overrides, or an explicitly named `agents.defaults.subagents` limit if maintainers prefer subagent-owned tuning. Reuse `truncateToolResultText`, add tests for direct announce, queued announce, nested child-completion findings, and fallback delivery, and update `docs/tools/subagents.md`, `docs/reference/token-use.md`, and the config docs/schema/help together.

## Risks / Open Questions

- Adding the cap needs care around where it applies: prompt injection, nested child-completion aggregation, queued announces, and external fallback delivery may want slightly different behavior.
- A fix should avoid hiding all useful completion signal on failures/timeouts; preserving head/tail context via the existing tool-result truncation helper looks safer than a plain slice.
