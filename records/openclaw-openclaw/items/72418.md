## Summary

Keep this open. Current main still contains the reported direct-local backend self-pairing path, tests explicitly preserve backend requested admin scopes for direct-local shared-token/password connects without device identity, and the same contributor has an open, unmerged closing PR (#72419). Per the review rules, that PR is an implementation candidate, not a reason to close this issue before merge or maintainer rejection.

## Best Possible Solution

Keep this issue open while maintainers review #72419 or decide that the documented trusted-loopback backend exception is intended. If hardening is accepted, the best fix belongs in the gateway WebSocket handshake/auth path: bind backend self-pairing to a server-side credential or verifiable backend identity, preserve legitimate direct-local helper flows, and add focused regression tests for shared token/password, device-token, browser-Origin, and remote/locality cases.
