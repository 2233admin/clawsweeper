## Summary

Keep open. Current main and latest release still contain the reported backend shared-token self-pairing exception, and #72419 is an open unmerged closing PR, so this should remain a maintainer security/product review item rather than a cleanup close.

## What This Changes

Request to harden gateway WebSocket backend self-pairing so loopback callers with a shared gateway token cannot self-declare the `gateway-client` backend identity and skip device pairing.

## Best Possible Solution

Keep this issue open until maintainers either land a narrowed backend identity boundary through #72419 or explicitly accept and document the existing loopback shared-token backend exception with matching tests.

## Reproduction Assessment

Yes. The helper-level reproduction is high confidence: declare `gateway-client`/`backend` on loopback without an Origin header, authenticate with a valid token/password, and `shouldSkipLocalBackendSelfPairing` returns true; `message-handler` then uses that value to preserve scopes and skip not-paired approval.

## Solution Assessment

Unclear. A dedicated backend credential or server-side identity check may be the narrower hardening path, but the current docs reserve this exception for internal control-plane RPCs and #72419 is still unmerged, so maintainer security/product review should choose the final contract.
