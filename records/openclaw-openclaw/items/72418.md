## Summary

Keep this issue open. Current main still preserves the reported loopback shared-token backend self-pairing path, the docs describe it as an intentional single-operator trust-model exception, and an open closing PR from the same contributor is still pending maintainer/security review.

## What This Changes

The issue reports that a loopback client with a valid shared gateway token can self-declare the backend gateway client identity, skip device pairing, and preserve requested operator scopes without a registered device token.

## Best Possible Solution

Keep this open while maintainers review #72419 and decide the gateway trust-model direction. If the exception is accepted, close with an explicit security rationale; if hardening is desired, bind backend self-pairing to a server-side backend credential or identity and add focused regression coverage for spoofed loopback backend clients without registered device identity.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: a declared backend gateway client on loopback, without browser Origin, with valid token/password shared auth causes `shouldSkipLocalBackendSelfPairing` to return true; a full live WebSocket exploit run was not performed because this review was read-only.

## Solution Assessment

No for cleanup closure. Current main preserves the reported path and the paired PR is still open, so the best route is maintainer security review followed by either an explicit trust-model rejection or a targeted hardening change.
