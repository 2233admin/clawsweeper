## Summary

Keep this open because current main still contains the reported loopback backend self-pairing path, the behavior is documented as an intentional gateway trust-model exception, and open PR #72419 is still an unmerged closing implementation candidate that does not remove the shared-token/password condition.

## What This Changes

The issue reports that a loopback client with a valid gateway shared token can self-declare the backend `gateway-client` identity and skip device pairing without a registered device token.

## Best Possible Solution

Maintain this as an open security/product review item until maintainers decide whether the documented backend exception is acceptable. If hardening is desired, bind backend self-pairing to a server-side backend credential or identity and add focused regression coverage for spoofed loopback backend clients without registered devices.

## Reproduction Assessment

Yes. The helper-level reproduction is high-confidence from current source and tests: a declared backend `gateway-client` on loopback, without a browser Origin, with valid token/password shared auth makes `shouldSkipLocalBackendSelfPairing` return true. A live WebSocket run was not performed because this review was read-only.

## Solution Assessment

No for cleanup closure. Current main intentionally preserves the behavior, and PR #72419 is the active implementation candidate but does not remove the challenged shared-token/password condition, so maintainer review is the safer route.
