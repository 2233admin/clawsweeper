## Summary

Keep open. Current main has manual browser profiles, opt-in sandbox browser isolation, and session tab cleanup, but omitted-profile host browser tool calls still route through the shared default `openclaw` profile, and Chrome MCP sessions are cached per profile rather than per agent.

## What This Changes

The issue asks OpenClaw to give each agent an isolated Chrome process, CDP port, user profile, and browser tool lifecycle instead of sharing one browser instance across agents.

## Best Possible Solution

Define one supported isolation model for browser automation, then implement it consistently across host, sandbox, node-proxy, and Chrome MCP paths with docs and tests. The likely end state is either automatic per-agent/per-session managed profiles with unique user data dirs and CDP ports, or clear product guidance that sandbox browser isolation is the supported multi-agent path.

## Reproduction Assessment

Yes. A high-confidence reproduction path is two non-sandbox agents with the browser tool enabled, both omitting `profile`, then opening/listing tabs: code inspection shows both resolve to the shared default `openclaw` profile unless explicitly configured otherwise.

## Solution Assessment

Unclear. The requested per-agent Chrome instance is directionally appropriate for isolation, but the best implementation needs maintainer design because OpenClaw already has manual profiles, sandbox-scoped browsers, node proxies, and existing-session Chrome MCP behavior that must remain compatible.
