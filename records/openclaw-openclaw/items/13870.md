## Summary

Keep open with high confidence. Current main only displays client-supplied displayName metadata or the deviceId and still lacks an operator-owned label field, rename protocol/schema/handler, CLI command, and Control UI action. The related discussion keeps this as the canonical paired-device naming workflow, with #23916 covering broader metadata display rather than the rename feature.

## What This Changes

Add user-set human-friendly labels for paired devices so the CLI and Control UI can identify devices without relying on long cryptographic IDs or reconnect metadata.

## Best Possible Solution

Add an additive paired-device label workflow with a durable operator-owned label separate from reconnect metadata, expose it through Gateway protocol, CLI, and Control UI, document the precedence rules, and keep node.rename plus broader metadata display as separate surfaces.

## Repair Work Prompt

Implement paired-device label management end to end. Treat the first workClusterRefs URL as the canonical issue and keep #23916 as broader metadata-display follow-up, not part of this fix.

Expected boundary:
- Add an additive `device.pair.rename` Gateway method with protocol schema/types/validators, method registration, and pairing scope mapping.
- Persist an operator-owned paired-device label or override separate from reconnect/client metadata. The operator label should take display precedence over client-supplied `displayName`; reconnect metadata must not overwrite it. Existing client `displayName` behavior should continue when no operator label is set.
- Add server handler authz matching the existing paired-device management model: paired-device token callers can rename their own device; admin/shared authorized callers can manage other devices according to existing device management semantics.
- Add `openclaw devices rename --device <id> --name <displayName>` with JSON output and safe CLI rendering.
- Add a Control UI rename action for paired devices in the Devices panel.
- Update `docs/cli/devices.md` and `docs/gateway/protocol.md`; add a user-facing changelog entry if the active changelog section expects one, crediting @koome23 and @shokk and not Codex.

Do not change node.rename semantics, node pairing records, token rotation/revocation semantics, broad devices-list metadata display from #23916, dependency versions, or generated/vendor files unless the repo's normal protocol generation scripts require them. Include focused tests for storage precedence, protocol/server validation, authz, CLI behavior, UI action wiring, and reconnect metadata not clobbering the operator label.
