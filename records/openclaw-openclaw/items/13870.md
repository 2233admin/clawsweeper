## Summary

Keep open. Current main still lacks the end-to-end paired-device naming workflow requested here: storage only has client-supplied display metadata, the Gateway protocol and method registry have no paired-device rename method, and neither the CLI nor Control UI exposes a rename action. The discussion and related items support keeping this as the canonical operator-owned paired-device label request, with #23916 covering broader metadata display and #57424 already folded into this area.

## What This Changes

Add user-set human-friendly labels for paired devices so the CLI and Control UI can identify devices without relying on long cryptographic IDs or reconnect metadata.

## Best Possible Solution

Add an additive paired-device label workflow with durable operator-owned label storage, explicit display precedence over client metadata, a device.pair.rename Gateway method, CLI and Control UI rename actions, docs, and focused tests while leaving node.rename and broader metadata display as separate surfaces.

Expected boundary:
- Add an additive `device.pair.rename` Gateway method with protocol schema/types/validators, method registration, and pairing-scope mapping.

- Add a Control UI rename action for paired devices in the Devices panel.
- Update `docs/cli/devices.md` and `docs/gateway/protocol.md`; add a user-facing changelog entry in the active section if required, crediting @koome23 and @shokk and not Codex.

Do not change `node.rename` semantics, node pairing records, token rotation/revocation semantics, broad devices-list metadata display from #23916, dependency versions, or generated/vendor files unless normal protocol generation requires it. Include focused tests for storage precedence, protocol/server validation, authz, CLI behavior, UI action wiring, and reconnect metadata not clobbering the operator label.
