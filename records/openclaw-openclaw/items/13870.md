## Summary

Keep open. Current main still does not provide an editable, durable paired-device label workflow: it only stores client-supplied `displayName` metadata, exposes no paired-device rename protocol or handler, and has no CLI or Control UI action to set a friendly name. The related #23916 remains broader display-metadata work, while closed #57424 appears to have been folded into this naming problem rather than superseding it.

## What This Changes

Request user-assigned human-friendly names for paired devices in the Control UI and `openclaw devices list`, with a CLI rename command.

## Best Possible Solution

Add an additive paired-device label workflow: persist an operator-owned label separate from client `displayName`, define display precedence as operator label then client `displayName` then raw `deviceId`, expose a typed `device.pair.rename` Gateway method with authorization, add CLI and Control UI rename actions, and document the behavior without merging it into broader automatic metadata display.
