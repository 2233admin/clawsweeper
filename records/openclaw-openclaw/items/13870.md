## Summary

Keep open. Current main still lacks the requested operator-owned paired-device rename workflow: it can display client-supplied displayName metadata, but there is no persisted label override, no device.pair.rename protocol/schema/handler, no openclaw devices rename command, and no Control UI rename action. The existing maintainer comment and @shokk's collision note both support keeping this as the canonical paired-device naming item, separate from broader metadata display in #23916.

## What This Changes

Add human-friendly, user-assignable names or labels for paired devices in the CLI and Control UI instead of relying on long device IDs or reconnect-supplied metadata.

## Best Possible Solution

Implement paired-device label management end to end: add an additive device.pair.rename method/schema/handler, persist an operator-owned label or override with documented precedence over reconnect metadata, expose openclaw devices rename --device <id> --name <displayName>, add a Control UI rename action, cover storage/protocol/CLI/UI/authz behavior with focused tests, and update devices/protocol docs while keeping node.rename and broader #23916 metadata display separate.

## Risks / Open Questions

- Reusing displayName directly for operator labels would let reconnect metadata overwrite user labels; the fix needs a separate operator-owned label or explicit precedence rule.
- The new gateway method must preserve existing paired-device management authorization: non-admin paired-device callers can manage only their own device unless they have operator.admin.
- CLI and Control UI rendering must continue sanitizing client-controlled fields and should avoid exposing token secrets or approved-scope internals.
- The change crosses storage, protocol, gateway handlers, CLI, UI, docs, and tests, so schema/docs/test drift is the main implementation risk.
