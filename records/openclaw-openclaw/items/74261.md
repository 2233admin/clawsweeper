## Summary

Keep open. Current main still has no WhatsApp group-creation agent tool, active-listener method, send/runtime API, gateway method, or docs surface; the request also is not a clean ClawHub-only close because an external plugin cannot safely create groups through the already-linked WhatsApp session without a WhatsApp-owned runtime seam and policy decision. The newest discussion strengthens the need for maintainer product/security review because Baileys-based group creation can carry account-abuse risk in repeated fixed-recipient workflows.

## What This Changes

The issue requests a WhatsApp-owned `whatsapp_create_group` agent tool that creates a WhatsApp group from a group name and participant phone-number list via the linked WhatsApp/Baileys session.

## Best Possible Solution

If maintainers approve the capability, implement it inside the WhatsApp plugin behind an explicit owner/trusted-requester gate: normalize participant numbers to WhatsApp JIDs, add a narrow active-listener/send/runtime method that calls Baileys `groupCreate`, redact identifiers in logs, record outbound activity, test the new path, and document the operational risk. If maintainers reject automated group invites, document that WhatsApp group creation is intentionally unsupported.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high confidence: source, docs, and runtime-surface inspection show login/send/poll/reaction/typing only, with no WhatsApp group-create path.

## Solution Assessment

Unclear until maintainer product/security review. The proposed Baileys-backed tool is technically plausible, but the maintainable version needs WhatsApp-owned policy gates and runtime seams; a broad core helper or ClawHub-only answer would not safely access the existing linked session.
