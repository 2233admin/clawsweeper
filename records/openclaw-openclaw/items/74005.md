## Summary

Keep open. Current main still leaves the reported class of proxy-only OpenAI provider failure plausible: OpenAI Chat Completions uses OpenClaw's guarded provider fetch path, and that guard does not automatically trust inherited HTTPS_PROXY/HTTP_PROXY in strict mode unless explicit provider proxy config or managed OpenClaw proxy mode is active. The report is recent, detailed, and the fix boundary is security-sensitive, so this should stay with maintainers rather than be swept closed.

## What This Changes

The issue asks for OpenAI Chat Completions provider traffic to honor HTTPS_PROXY in proxy-only sandboxes, or to emit an actionable diagnostic when that proxy path is bypassed.

## Best Possible Solution

Preserve the strict SSRF default, then make the model-provider proxy contract explicit. Maintainers should either add a narrow, reviewed trusted-env-proxy path for public native model endpoints in proxy-only deployments, or improve docs/doctor/runtime errors so users are directed to `models.providers.openai.request.proxy.mode: "env-proxy"` or managed `proxy.enabled` instead of expecting ambient HTTPS_PROXY alone to be sufficient.

## Reproduction Assessment

Yes. The issue provides concrete HTTPS_PROXY-only sandbox steps and proxy-log evidence, and current source/tests independently show why ordinary OpenAI provider requests without explicit `request.proxy` or managed proxy mode remain on strict guarded fetch instead of inherited env-proxy routing.

## Solution Assessment

Unclear. Automatically honoring inherited HTTPS_PROXY for this path would address the reported deployment, but current tests document a deliberate SSRF/DNS-pinning invariant, so the safest fix needs maintainer security/product judgment before implementation.
