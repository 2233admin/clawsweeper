## Summary

Keep open. Current main still sends OpenAI-family provider requests through the guarded provider fetch path without auto-selecting trusted env-proxy mode, while the SSRF guard explicitly preserves strict DNS-pinned behavior unless the managed proxy lifecycle or an explicit env-proxy request mode is active. This is not implemented on main; the open PR #72480 is the narrow implementation candidate for the same provider guarded-fetch proxy gap.

## What This Changes

The issue reports that OpenAI requests to api.openai.com fail in HTTPS_PROXY-only environments because the provider request path does not traverse the env-proxy dispatcher.

## Best Possible Solution

Keep this issue open until PR #72480 or an equivalent generic provider transport fix lands. The desired end state is for trusted model-provider HTTP transports to honor operator env proxy routing when appropriate, without adding OpenAI-specific special cases or weakening SSRF protections for untrusted fetches.

## Reproduction Assessment

Yes. The reporter provides a concrete HTTPS_PROXY-only gateway reproduction, and current-main code plus tests show the relevant provider fetch path remains strict unless explicit env-proxy configuration or managed proxy mode is active.

## Solution Assessment

Unclear until maintainer/security review of #72480 completes. The best direction appears to be the generic `buildGuardedModelFetch` fix proposed there, not an OpenAI-id-specific SDK workaround.
