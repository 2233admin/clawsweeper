## Summary

Keep open. Current main still leaves this failure class plausible: OpenAI completions is routed through OpenClaw's guarded provider fetch, and that guard intentionally ignores inherited proxy env in strict mode unless explicit provider proxy config or managed proxy mode is active. The linked prior fixes and the xAI comparison do not prove the bare `api.openai.com` completions path works in an HTTPS_PROXY-only sandbox.

## What This Changes

The issue asks OpenClaw to make `openai-completions` requests to `api.openai.com` work in HTTPS_PROXY-only environments, or emit a clear diagnostic when proxy routing is bypassed.

## Best Possible Solution

Preserve the strict SSRF default, then make the provider-proxy contract explicit. Maintainers should either add a narrowly trusted provider transport path for public native model endpoints in proxy-only deployments, or improve docs/doctor/runtime errors so users are directed to `models.providers.*.request.proxy.mode: "env-proxy"` or managed `proxy.enabled`.

## Reproduction Assessment

Yes. The report gives concrete HTTPS_PROXY-only sandbox steps and proxy-log evidence, and current source/tests independently show why provider requests without explicit `request.proxy` or managed proxy mode stay on strict guarded fetch instead of inherited env-proxy routing.

## Solution Assessment

Unclear. Automatic inherited-env behavior would address the reported deployment, but the current strict-mode test documents a deliberate SSRF/DNS-pinning invariant; the safer solution needs maintainer security/product judgment first.
