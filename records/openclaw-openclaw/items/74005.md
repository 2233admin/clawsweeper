## Summary

Keep open. Current main and v2026.4.26 both route OpenAI-family embedded turns through OpenClaw's boundary-aware provider transport, but that transport still calls `fetchWithSsrFGuard` in strict mode. The SSRF guard intentionally ignores plain `HTTPS_PROXY`/`HTTP_PROXY` env proxy settings unless the managed OpenClaw proxy lifecycle or an explicit trusted/env-proxy request mode is active, which matches the reporter's HTTPS_PROXY-only failure mode rather than disproving it.

## Best Possible Solution

Keep this issue open. The best next step is a maintainer decision on proxy semantics: either make provider HTTP transports honor inherited env proxy settings for trusted/public model endpoints without weakening SSRF guarantees, or document/doctor the required `request.proxy.mode: "env-proxy"` or `proxy.enabled=true` configuration for HTTPS_PROXY-only sandboxes. Any code fix should be generic in the provider transport/fetch guard layer, not OpenAI-id-specific.
