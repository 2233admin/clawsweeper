## Summary

Keep open. Current main still leaves the reported proxy-only OpenAI provider failure class plausible: native OpenAI Chat Completions uses OpenClaw's guarded provider fetch, and that guard does not automatically trust inherited HTTP_PROXY/HTTPS_PROXY in strict mode unless explicit provider proxy config or managed OpenClaw proxy mode is active. The report is recent, detailed, and the fix boundary affects SSRF/proxy policy, so it needs maintainer judgment rather than cleanup closure.

## What This Changes

The issue asks for OpenAI Chat Completions provider traffic to honor HTTPS_PROXY in proxy-only sandboxes, or to emit an actionable diagnostic when that proxy path is bypassed.

## Best Possible Solution

Preserve the strict SSRF default while making the model-provider proxy contract explicit. Maintainers should either add a reviewed trusted-env-proxy path for public native model endpoints in proxy-only deployments, or improve docs/doctor/runtime diagnostics so users are directed to models.providers.openai.request.proxy.mode: "env-proxy" or managed proxy.enabled without implying ambient HTTPS_PROXY alone is sufficient.

## Reproduction Assessment

Yes. The issue provides concrete HTTPS_PROXY-only sandbox steps and proxy-log evidence, and current source/tests independently show why ordinary OpenAI provider requests without explicit request.proxy or managed proxy mode remain on strict guarded fetch instead of inherited env-proxy routing.

## Solution Assessment

Unclear. Automatically honoring inherited HTTPS_PROXY for this path would address the reported deployment, but current tests document a deliberate SSRF/DNS-pinning invariant, so the safest implementation needs maintainer security/product judgment first.
