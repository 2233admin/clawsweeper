## Summary

Keep open. This is a fresh, narrower report after the #71026 cleanup: it describes a cron runtime failure, not only stale status display. Current main still has a real `claude-cli` OAuth dead-end if the external CLI bootstrap is skipped/refused: `claude-cli` has no pi-ai OAuth refresh adapter, profile resolution returns null, and the explicit profile path throws the exact reported `No credentials found for profile "anthropic:claude-cli"` error. Existing coverage verifies external sync/status pieces, but not the stale `anthropic:claude-cli` plus fresh macOS Keychain path inside isolated cron.

## Best Possible Solution

Keep this issue open. Add a focused regression around isolated cron/provider auth with an expired persisted `anthropic:claude-cli` OAuth profile and mocked fresh Claude CLI keychain credentials, then adjust the auth runtime so `claude-cli` either re-reads/adopts external CLI credentials before returning null or surfaces a clear diagnostic when the external bootstrap was skipped/refused. Also consider separating the Claude CLI credential cache by `allowKeychainPrompt`/source so no-keychain runtime probes cannot poison the longer external-sync path.
