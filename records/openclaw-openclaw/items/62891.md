## Summary

Keep this open. Current main added a partial restart mitigation for overdue isolated agent-turn jobs, but cron activation and hot cron reload still do not wait for Discord-style channel transport readiness, and normal timer ticks can still execute channel-delivered cron work before the target channel is connected.

## What This Changes

Request to prevent cron and LLM-triggering scheduled work from firing during startup or SIGUSR1/config reload before Discord-style channel transports are ready.

## Best Possible Solution

Add a bounded, delivery-aware readiness gate at the gateway/cron boundary for startup activation and hot cron restart. It should wait for the relevant configured channel transport to become healthy, skip the wait for jobs that cannot deliver through a channel, time out with clear diagnostics, and cover full startup plus config.patch/SIGUSR1 reload behavior.

## Repair Work Prompt

Fix the cron/channel readiness race described in openclaw/openclaw#62891. Current main only defers overdue isolated agent-turn startup catch-up; it still starts cron during gateway activation and config hot reload without a delivery-aware channel transport readiness gate. Implement a bounded gate at the gateway/cron boundary for automatic startup activation and hot cron restart/config.patch paths. The gate should use existing channel runtime snapshot/health/readiness signals where possible, wait only when due automatic cron work can deliver through a messaging channel, skip webhook/no-delivery/manual cron.run paths, ignore disabled or unconfigured channels, and time out with clear diagnostics instead of blocking indefinitely. Do not hard-code Discord-only behavior in core; use generic channel health/readiness concepts and keep plugin boundaries intact. Add focused tests for full startup, hot reload/SIGUSR1 config.patch cron restart, asynchronous Discord-style readiness, missed-job catch-up, regular due timer jobs, webhook-only/no-delivery jobs, and disabled/unconfigured channels. Update cron docs and add a changelog fix entry with Thanks @Stache73 if behavior changes are user-facing.
