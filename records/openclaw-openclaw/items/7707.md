## Summary

Keep this issue open. Current main still has adjacent source labels, session visibility filtering, Active Memory untrusted framing, Memory Wiki provenance, and optional memory-lancedb prompt-injection mitigations, but it does not implement persistent memory trust metadata, trust-policy config, review/expiration gates, or trust-weighted retrieval for the default memory path.

## Best Possible Solution

Keep this open for maintainer/product/security design. A complete solution should add structured source trust metadata at ingestion, persist it through memory-core, QMD, session-memory, Active Memory, Memory Wiki bridge, and SDK contracts, document trust-policy config, enforce retrieval-time filtering/downweighting/review gates, support expiration by trust tier, preserve provenance through compaction/dreaming flows, expose trust state where useful in UI/docs, and add regression coverage across the default and optional memory paths. HMAKT99's AKF comment is useful prior art to evaluate, not shipped behavior.

## Risks / Open Questions

- Closing would lose a real security/product design gap: durable memory can still be searched, promoted, compacted, or bridged without persisted source trust tiers that retrieval can enforce.
- A complete fix likely needs SDK/config/schema and memory backend changes, not only a bundled plugin, because current memory result and config contracts do not carry trust fields.
- Adjacent prompt framing, session visibility checks, and provenance-rich wiki pages reduce risk in specific flows but do not prevent trust laundering through durable memory, session indexing, compaction, dreaming, or bridge paths.
