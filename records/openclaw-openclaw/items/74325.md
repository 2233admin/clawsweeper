## Summary

Keep open. Current main has several partial mitigations for the reported startup stall, but it still does not prove that the two-phase restart freeze is gone: channel startup remains part of the sidecar readiness path, the post-ready work is still outside the startup trace, and a fresh related report now tracks bundled runtime-mirror blocking that only has partial unreleased fixes.

## What This Changes

The issue asks OpenClaw to stop blocking the Gateway event loop for roughly 75 seconds during restart, especially inside `sidecars.channels` and an immediate post-`gateway ready` freeze that starves local WebSocket handshakes.

## Best Possible Solution

Keep this open until a maintainer gets a current-main or next-release restart trace from the reporter, ideally with after-ready instrumentation enabled. The end state should be a Gateway restart where core readiness is not blocked by bundled channel/runtime-deps work, local WebSocket upgrades do not starve during post-ready work, and the trace clearly attributes any remaining long synchronous chunk to a specific channel, plugin runtime mirror, hook, or provider-discovery path.

## Reproduction Assessment

Yes for the reported 2026.4.26 environment: the issue includes deterministic restart steps, exact startup trace output, and repeated plugin-side event-loop-lag measurements. No high-confidence current-main reproduction is available yet because the reporter has not rerun after the v2026.4.27 and unreleased runtime-mirror changes.

## Solution Assessment

Unclear. Current main has the right direction for part of the problem, but the narrowest maintainable next step is to instrument the after-ready window and rerun on current main before choosing between channel startup, runtime-mirror, hook, or provider-discovery changes.
