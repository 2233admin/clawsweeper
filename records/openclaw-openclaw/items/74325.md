## Summary

Keep open. Current main has partial mitigations for the 2026.4.26 startup path, but it still eagerly waits for channel startup before `gateway ready`, still stops startup event-loop tracing at `ready`, and there is no current-main proof that the reported post-ready WebSocket starvation window is gone.

## What This Changes

The issue reports deterministic gateway restart event-loop blocking in `sidecars.channels` plus a second post-ready freeze that causes local plugin WebSocket handshakes to time out.

## Best Possible Solution

Keep this report open until maintainers collect a current-main restart trace that includes after-ready attribution, then either fix the measured blocker directly or link each remaining phase to a canonical tracker such as lazy channel startup or bundled runtime mirror work.

## Reproduction Assessment

Yes for the reported release and host: the issue includes concrete 2026.4.26 restart steps, startup trace timings, and repeated plugin-side event-loop-lag observations. Current-main reproduction is not high-confidence yet because this read-only review did not run the reporter's WSL2 packaged restart scenario.

## Solution Assessment

Unclear. Current main's prewarm and runtime-mirror mitigations are useful, but they do not prove the combined issue is fixed; the safer path is after-ready tracing plus a current-main rerun before deciding whether the remaining fix belongs in channel lifecycle, runtime mirror, or another post-ready path.
