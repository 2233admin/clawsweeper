## Summary

Keep open. Current main has partial mitigations beyond the reported 2026.4.26 build, but it still eagerly waits for channel startup before readiness, still disables startup event-loop tracing at `ready`, and related open trackers/PRs do not fully cover this issue's combined restart plus post-ready WebSocket starvation evidence.

## What This Changes

The issue reports deterministic gateway restart event-loop blocking during `sidecars.channels` plus a second freeze immediately after `gateway ready`, causing local plugin WebSocket handshakes to time out.

## Best Possible Solution

Keep this report open until maintainers capture a current-main restart trace with after-ready attribution, then either fix the measured blocker directly or explicitly link each resolved phase to the lazy-channel and runtime-mirror trackers after their fixes land and verify the CoClaw WebSocket recovery case.

Current-main reproduction is still unclear because this read-only review did not run the reporter's WSL2 packaged restart scenario.

## Solution Assessment
