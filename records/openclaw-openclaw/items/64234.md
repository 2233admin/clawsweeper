## Summary

Keep this PR open for maintainer review, but it is not ready to merge. Current main does not implement the requested privacy/encryption surface, and the PR branch has multiple blocking correctness and security-control gaps: the new config key is not accepted by the strict runtime schema, at-rest encryption is only a helper/test module rather than wired into session storage, and messaging-channel system-prompt privacy is not threaded through the embedded runner path.

## What This Changes

The PR adds a `privacy` config type, PII redaction helpers and tests, prompt/tool/user/TTS/media hook wiring, AES-256-GCM session-encryption helpers, and a `openclaw/plugin-sdk/privacy-runtime` export while also reformatting many generated/runtime files.

## Best Possible Solution

Keep the PR open for explicit maintainer/security design review, but require the branch to be narrowed and fixed before merge: decide how it relates to #45783, split PII redaction from at-rest transcript encryption if useful, add runtime schema/docs/API-baseline coverage, wire encryption into actual session read/write paths, and cover the embedded messaging paths with regression tests.

## Reproduction Assessment

Yes. The key failures are reproducible by static path checks: `privacy` is absent from the strict runtime schema, `privacy.atRest` has no production encryption call sites, and the embedded messaging prompt builder never receives `params.config?.privacy`. A focused test can assert `OpenClawSchema.safeParse({ privacy: { enabled: true } })` currently fails on the PR branch.

## Solution Assessment

No. The proposed direction is useful, but the current implementation is not the narrowest maintainable solution because several advertised privacy controls are inert or unavailable; a safer path is to first land a schema-backed, tested text-redaction control plane, then add at-rest encryption as a separately wired session-storage change.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Add the privacy config to the runtime schema:** `src/config/types.openclaw.ts:138`
  - body: This adds `privacy` to the TypeScript config type, but the strict `OpenClawSchema` used by config validation has no matching root key. Any normal `openclaw.yml` that sets `privacy.*` will be rejected before the new redaction or encryption hooks can run, making the feature unusable from supported config.
  - confidence: 0.96
- **[P1] Wire at-rest encryption into session storage:** `src/privacy/types.ts:98-99`
  - body: `privacy.atRest.enabled` is documented as encrypting session JSONL files, but the diff only adds the encryption helper and tests; no production session read/write path calls `encryptSessionFile` or decrypts before replay. After the schema issue is fixed, enabling this setting still leaves transcripts written in plaintext.
  - confidence: 0.94
- **[P1] Thread privacyConfig through embedded prompts:** `src/agents/system-prompt.ts:990`
  - body: The new masking call depends on `params.privacyConfig`, but only the CLI helper passes it. Messaging channels build prompts through `buildEmbeddedSystemPrompt`, which still has no privacy parameter and is called without `params.config?.privacy`, so hostname/path/context-file redaction silently does nothing for the main channel execution path.
  - confidence: 0.93
- **[P2] Use async PBKDF2 for session encryption:** `src/privacy/session-encrypt.ts:95-97`
  - body: `deriveKey` calls `pbkdf2Sync` with 210k iterations inside async encryption/decryption APIs. Encrypting or decrypting a session, especially a directory of sessions, will block the gateway event loop and stall unrelated messages and UI work; use the async `pbkdf2` API instead.
  - confidence: 0.9
- **[P2] Always close the header file descriptor:** `src/privacy/session-encrypt.ts:140-143`
  - body: If `fd.read()` throws after `fs.open()` succeeds, the descriptor is never closed because `fd.close()` is after the read and not in a `finally`. A long-running gateway that probes many files can leak descriptors until process I/O starts failing.
  - confidence: 0.88
- **[P2] Mask repo paths that contain spaces:** `src/privacy/payload-redact.ts:200-206`
  - body: The repo masking regex captures only up to whitespace (`[^\s|]+`), so a runtime line like `repo=C:/Users/Alice Smith/project` is only partly replaced and the remainder of the identifying path stays in the prompt. Capture up to the field separator instead of stopping at spaces.
  - confidence: 0.86
