## Summary

Keep open. Current main still lacks WhatsApp per-group `allowFrom`: the public config type, strict schema, generated metadata, docs, inbound access gate, and command authorization path all resolve only account/channel-level `groupAllowFrom` with `allowFrom` fallback. The related WhatsApp prompt, multi-account inheritance, and deny/blocklist items do not supersede this narrower per-group sender-authorization request.

## What This Changes

Add optional per-group `allowFrom` to WhatsApp root and account-scoped `groups` entries so each group can override the channel/account `groupAllowFrom` sender authorization list.

## Best Possible Solution

Implement an additive WhatsApp plugin change that adds `allowFrom` to root and account-scoped group entries, treats a defined group value as an override, preserves existing `groupAllowFrom`/`allowFrom` fallback when undefined, updates generated config metadata and docs, and covers both inbound replies and commands.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: `channels.whatsapp.groups.<jid>.allowFrom` is rejected by the strict schema, and the runtime never reads a matched group entry when deciding sender authorization.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable solution because it matches existing channel prior art and avoids changing the separate WhatsApp multi-account inheritance policy tracked elsewhere.
