## Summary

Keep open: the PR targets a real Slack route-contamination bug on current main and the latest branch addresses the earlier per-account dispatch-test gap, but the dispatch-side fix still targets the base route session for non-main threaded DMs instead of the prepared turn session.

## What This Changes

The branch changes Slack DM last-route persistence in prepare and dispatch to use `route.lastRoutePolicy` instead of always writing `route.mainSessionKey`, and adds focused dmScope regression tests.

## Best Possible Solution

Keep the policy-based prepare change, but make dispatch resolve non-main last-route writes from the prepared turn session rather than the base route session. Add a threaded Slack DM regression that proves `lastRoutePolicy: "session"` updates the thread-suffixed session while `lastRoutePolicy: "main"` still preserves main-DM owner pin behavior.

## Reproduction Assessment

Yes. A high-confidence seam reproduction is a Slack DM with non-main effective `dmScope` and a thread context, such as `replyToMode="all"` or an incoming DM thread, where `prepared.ctxPayload.SessionKey` can be `route.sessionKey:thread:<id>` while the PR dispatch branch still updates `route.sessionKey`.

## Solution Assessment

No. The PR’s route-policy direction is the right narrow fix, but dispatch should mirror the prepared turn session for `lastRoutePolicy: "session"`; using `route.sessionKey` remains wrong for supported threaded DM routing.

## Review Findings

Resolve the target from the prepared turn session and add a threaded DM regression.
