## Summary

Current main still leaves the reported native Windows Gmail setup failure actionable. The documented Gmail setup command still routes to the same setup path, dependency checks can recognize Windows PATHEXT shims, but the actual Gmail helper execution still invokes bare `gcloud`, `gog`, and `tailscale` paths that are not resolved to `.cmd`/`.ps1` shims before spawning. The existing Windows command shim remains limited to npm/npx/corepack/pnpm/yarn-style package-manager commands, and no current test covers Gmail helper resolution on win32.

## Best Possible Solution

Keep this issue open. Add a Windows-aware helper-tool resolver for Gmail setup and runtime paths before spawning `gcloud`, `gog`, and `tailscale`, preferably reusing `src/infra/executable-path.ts` or a narrowly extended shared exec shim while preserving argv-based no-shell execution. Add win32 regression tests for PATH/PATHEXT `.cmd` shims across setup, run, and watcher flows.

## Risks / Open Questions

- Native Windows was not live-tested from this Linux checkout, but the reporter provided concrete PowerShell evidence and current main still shows the command-resolution gap that matches the reported `spawn gcloud ENOENT` failure.
- A fix should preserve the current no-`shell: true` hardening. Broad shell execution would increase command-injection exposure for argv-based helpers.
- Coverage should include setup, run, and gateway watcher paths; the direct `spawn("gog", ...)` flows can otherwise keep failing after `gcloud` setup is repaired.
