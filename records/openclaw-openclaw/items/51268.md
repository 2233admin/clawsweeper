## Summary

Keep open. Current main still lacks an ACP device-identity opt-out, and this PR implements the right core mechanism by threading an explicit null into the existing GatewayClient contract, but the public CLI documentation still omits the new flag and the final flag spelling should be settled before merge.

## What This Changes

The PR adds an ACP server `noDeviceIdentity` option, registers `openclaw acp --skip-device-identity`, passes `deviceIdentity: null` to `GatewayClient`, and adds startup tests for null versus undefined device-identity handling.

## Best Possible Solution

Merge this PR or an equivalent maintainer patch after the final flag name is chosen, the ACP CLI reference at `https://docs.openclaw.ai/cli/acp` is updated with the same name and security meaning, and the server option, Commander registration, standalone parser/help, and null-versus-undefined tests remain aligned.

## Reproduction Assessment

Yes. Current-main code inspection gives a high-confidence reproduction path: `openclaw acp` never passes `deviceIdentity`, and `GatewayClient` auto-loads one unless it receives explicit null; the PR's startup tests exercise the intended null and undefined branches.

## Solution Assessment

Yes for the core implementation direction: explicit `deviceIdentity: null` is the narrowest way to use the existing GatewayClient and gateway policy contracts. The proposed patch is not merge-ready until docs and final flag naming are aligned.
