## Summary

Keep open. Current main still does not implement either central change from this PR: Gemini OAuth credential extraction still returns null when `gemini` is absent from PATH, and the workspace bootstrap code still has no `WORKING.md` constant, validation entry, loader entry, tests, or docs. The PR also targets an older `extensions/google-gemini-cli-auth` layout while current main owns Gemini CLI OAuth in the bundled `google` plugin, so it needs a rebase or split before maintainers can evaluate it cleanly.

## Best Possible Solution

Keep this PR open for maintainer review, but ask for a rebase and likely split. The Gemini fix should be ported to the current `extensions/google/oauth.credentials.ts` implementation with targeted tests that restore all mutated environment state. The `WORKING.md` feature should be decided as agent-bootstrap product behavior first, then implemented in `src/agents/workspace.ts` with validation, loading, subagent/cron semantics, docs, and tests aligned.

## Risks / Open Questions

- The PR combines two unrelated surfaces: Google Gemini CLI OAuth credential discovery and agent workspace bootstrap semantics. Splitting would make review and ownership cleaner.
- The OAuth test in the PR still has a review-identified PATH restore bug that could leak a broken PATH into later tests if landed as-is.
- The `WORKING.md` behavior needs a maintainer decision for subagent and cron sessions before implementation, because current filtering intentionally limits bootstrap files for those session types.
- The proposed APPDATA fallback expands credential extraction to a user-writable npm global package tree; that may be acceptable under the local CLI trust model, but it is a real security review point.
