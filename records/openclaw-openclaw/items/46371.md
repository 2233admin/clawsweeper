## Summary

Keep this PR open. Current main still does not implement either central change: Gemini CLI OAuth still returns null when `gemini` is absent from PATH, and workspace bootstrap still has no `WORKING.md` support. The PR also needs a rebase/port because the Gemini OAuth code now lives in the bundled `google` plugin, not the older `extensions/google-gemini-cli-auth` path used by the PR.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but ask for a rebase and likely split. Port the Gemini fix to `extensions/google/oauth.credentials.ts` with targeted tests that restore all mutated environment state, then decide and implement `WORKING.md` in `src/agents/workspace.ts` with validation, loading, subagent/cron semantics, docs, and tests aligned.

## Risks / Open Questions

- The provided bot review identified a real test isolation issue in the PR: the new OAuth test mutates `process.env.PATH` without restoring it.
- The OAuth change expands credential discovery to `%APPDATA%\npm\node_modules` even when `gemini` is absent from PATH; that should get explicit maintainer security review against the local Gemini CLI trust model before landing.
- The `WORKING.md` behavior needs a product decision for subagent and cron sessions because current bootstrap filtering intentionally limits context for those sessions.
- The PR bundles two unrelated surfaces, Google Gemini CLI OAuth and agent workspace bootstrap semantics, which makes ownership and review harder.
