## Summary

Keep this issue open. Current main has strong in-memory replay repair for orphaned and displaced tool results, and it already calls a session-file repair hook before loading sessions, but that on-disk hook does not apply tool-use/tool-result pairing repair, so the persistence gap reported here still appears actionable.

## What This Changes

The issue requests repairing orphaned or displaced persisted tool-result/tool-use transcript pairings on session file load and rewriting the session file so corruption does not survive restarts.

## Best Possible Solution

Extend the existing session-file repair path to persist the same conservative pairing invariants already enforced during replay: drop truly orphaned tool results, move displaced matching results next to their assistant tool calls, synthesize only storage-safe missing results, keep backups, and document the persistent repair behavior.

## Reproduction Assessment

Yes. A focused source-level repro is a valid session JSONL file with a header plus an orphaned or displaced `toolResult`; current file repair will parse and keep it unchanged unless one of the existing malformed-line/empty-content branches fires, while replay sanitization would repair only the in-memory copy.

## Solution Assessment

Yes. The issue’s repair-on-load direction is the narrowest maintainable fix because the load and compaction paths already call `repairSessionFileIfNeeded`; adding a new opt-in config should be a maintainer product choice, not a prerequisite for fixing the persisted corruption loop.
