## Summary

Keep open. Current main still has the direct buffer-backed Discord `message.send` false-success gap: the public send schema accepts `buffer`/`filename`/MIME fields, but ordinary `send` is excluded from attachment hydration, the shared runner and Discord bridge only promote path/URL media references, and Discord upload only runs when `mediaUrl` is present. The path/filePath multipart route appears implemented and tested, so the remaining work is narrowed to direct buffer delivery or explicit rejection; open PRs #57335 and #63578 target this area but are not merged.

## What This Changes

The issue requests that Discord `message.send` upload attachments for local path, filePath, or buffer inputs instead of returning success with an empty Discord attachments array.

## Best Possible Solution

Review or consolidate #57335 and #63578 into one narrow fix that makes ordinary Discord `message.send` align with its advertised schema by hydrating and delivering `buffer`/`filename`/`contentType` through the existing attachment and multipart upload path, or by rejecting unsupported attachment fields with a clear error and regression coverage.
