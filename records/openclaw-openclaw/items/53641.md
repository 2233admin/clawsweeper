## Summary

Keep open. Current main still has the direct buffer-backed Discord send gap: the public send schema accepts attachment buffer fields, but ordinary send hydration exits before processing them and the Discord bridge/runtime only reaches multipart upload through media path or URL fields. Related PRs #57335 and #63578 target this area but are still open in the provided GitHub context.

## What This Changes

The issue reports Discord `message.send` returning success and a message id while attachment inputs are not uploaded to Discord.

## Best Possible Solution

Make ordinary Discord `message.send` either hydrate and deliver `buffer`/`filename`/`contentType`/`mimeType` through the shared attachment pipeline and Discord multipart upload path with regression coverage, or fail fast when those fields are unsupported for `send`.
