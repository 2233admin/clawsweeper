## Summary

Keep open. Current main has adjacent memory flush, session-memory hooks, timeout-triggered compaction, and memory-core transcript ingestion, but it still does not implement the requested repeated-timeout pending session capture and later recovery into daily memory. The linked #46930 is a separate gateway log-level issue, not a canonical tracker for this work.

## Best Possible Solution

Keep this open and design a generic repeated-timeout recovery path: after timeout exhaustion leaves user turns unprocessed, persist a bounded redacted pending transcript artifact with stable session/message metadata, then recover it after model availability returns through memory-core or daily-memory maintenance with dedupe, retention cleanup, docs, and regression coverage.
