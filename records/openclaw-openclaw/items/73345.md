## Summary

Keep open; the PR is focused and useful, but it is not ready to merge because adding `thread` as a delivery destination without adding it to the hook replay scope can make idempotent retries replay the wrong run for a different thread/topic.

## What This Changes

The PR adds optional `thread` parsing to `/hooks/agent` payloads, forwards it as `CronJob.delivery.threadId`, and adds gateway hook coverage for payloads with and without `thread`.

## Best Possible Solution

Revise this PR so `thread` is treated as part of the delivery destination everywhere the hook request is scoped: normalize it, forward it to cron delivery, include it in replay dedupe scope, add a regression where the same idempotency key with a different thread dispatches separately, and document the public webhook field before merge.

## Review Findings

Reusing an idempotency key while changing only `thread` can return the first runId and skip delivery to the new topic/thread; include the normalized thread in the replay scope and cover that case.
