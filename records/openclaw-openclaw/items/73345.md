## Summary

Keep open. Current main still lacks `/hooks/agent` thread propagation, and the PR is a useful narrow implementation, but it is not merge-ready because it makes `thread` part of delivery without making replay dedupe thread-aware; the earlier ClawSweeper review called out the same concrete bug, and the source pass here confirms it.

## What This Changes

The PR adds optional `thread` parsing to `/hooks/agent`, forwards it into `CronJob.delivery.threadId`, and adds gateway hook tests for payloads with and without `thread`.

## Best Possible Solution

Revise this PR so `thread` is treated as part of the delivery destination everywhere the hook request is scoped: normalize it, forward it to cron delivery, include it in replay dedupe scope, cover same-idempotency-key/different-thread dispatch, and document the webhook field before merge.

## Reproduction Assessment

Yes. A source-level reproduction is two `/hooks/agent` requests with the same idempotency key and same delivery target except for `thread`; because the PR's replay scope omits thread, the second request can return the first runId and skip dispatch.

## Solution Assessment

No. Forwarding `thread` to `CronJob.delivery.threadId` is necessary but incomplete; the safer fix is to include the normalized thread in replay scope and align the public webhook docs and value contract.
