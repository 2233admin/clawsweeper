## Summary

Keep open. Current main still keeps tool-call interception in the plugin hook runner, while managed/internal HOOK.md hooks remain a void-returning internal event system with no tool_call event type, no blocking or parameter-rewrite return contract, and no bridge from tool execution. The issue also has fresh maintainer signal keeping it as the canonical tracker; related #17008 is closed as a duplicate and #29590 was closed unmerged as only a partial after-tool-call bridge attempt.

## Best Possible Solution

Keep this issue open for maintainer product/API design. A complete fix should add typed managed/internal before and after tool-call events, define blocking and parameter-rewrite return semantics, validate or reject unsupported HOOK.md event names, bridge all relevant tool execution paths without double-firing existing plugin hooks, and add focused tests plus docs. If maintainers decide tool-call interception should stay plugin-only, close later with that explicit product decision and point users to plugin hooks and ClawHub packaging.
