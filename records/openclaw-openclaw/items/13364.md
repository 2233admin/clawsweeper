## Summary

Keep open. Current main still keeps managed/internal HOOK.md hooks on a void-returning internal event registry, while tool-call interception remains wired through plugin hooks only. The issue also has fresh maintainer signal keeping it as the canonical tracker; related #17008 and #67085 are closed as duplicates, and #29590 was closed unmerged as only a partial after-tool-call bridge attempt.

## Best Possible Solution

Keep this issue open as the canonical managed/internal tool-call hook design item. A complete fix should add typed managed before/after tool-call events, define blocking, approval, timeout, failure, and parameter-rewrite semantics, bridge all relevant tool execution paths without double-firing existing plugin hooks, validate or warn for unsupported HOOK.md event names, and update focused tests plus the public hooks docs. If maintainers decide tool-call interception must remain plugin-only, close later with that explicit product/API decision and point users to plugin hooks and plugin packaging.

## Risks / Open Questions

- A partial bridge could double-fire plugin hooks or miss one of the tool execution paths unless before and after dispatch are designed against the existing plugin runner wiring.
- Blocking tool execution from managed HOOK.md scripts is security-sensitive API work: fail-open/fail-closed behavior, timeouts, parameter rewrite precedence, errors, and audit data need explicit maintainer design.
- The current HOOK.md metadata accepts arbitrary event strings, so any fix should also decide whether unsupported tool event names should be validated, warned, or made reachable.
