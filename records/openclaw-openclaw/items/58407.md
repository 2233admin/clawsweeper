## Summary

Keep this issue open. Current main has Discord thread parent metadata, thread starter context, model-only parent fallback, and optional parent transcript inheritance, but it does not fetch or inject parent-channel pinned messages into Discord thread sessions; the existing pin support is only exposed as manual message actions. The stale label is not enough to close this because the report is recent and the requested feature remains unimplemented.

## What This Changes

Request automatic, configurable inheritance of a Discord thread's parent-channel pinned messages into the thread session context.

## Best Possible Solution

Add an opt-in Discord plugin feature that, for thread messages with a parent channel id, fetches a bounded set of parent-channel pins, sanitizes them, and exposes them as untrusted supplemental context or explicit metadata for the thread turn without enabling parent transcript inheritance or treating pins as system prompt authority.

## Reproduction Assessment

Yes for the missing behavior at code level: the current Discord thread context builder can be inspected to show no parent pin lookup or pin payload. A live Discord pin scenario would still be useful after a patch to verify permissions, rate limits, and formatting end to end.

## Solution Assessment

Unclear for the reporter's system-context option, because pinned messages should not become trusted system instructions. The narrow maintainable solution is an opt-in Discord-plugin implementation that injects pins through the existing untrusted supplemental context/metadata path.
