## Summary

Keep open: current main still only dedupes this path when a stable MessageSid/clientRunId is present, the user transcript update path still emits rows without an idempotency key, and #72290 remains open as the paired implementation candidate for this regression.

## What This Changes

The issue reports an intermittent 2026.4.24 regression where duplicate inbound user messages reach session history across WebChat and WeChat, causing duplicate assistant replies.

## Best Possible Solution

Review #72290 as the canonical implementation candidate, then land or revise a narrow Gateway/core fix that suppresses retry duplicates with unstable IDs while preserving intentional repeated same-text messages across channels.

## Reproduction Assessment

Yes for a focused code-level reproduction path: current main returns no inbound dedupe key without MessageSid and emits user transcript updates without transcript idempotency metadata. The live intermittent WSL/WebChat/WeChat scenario was not rerun.

## Solution Assessment

Unclear until #72290 is reviewed. The transcript idempotency part is narrow, but the fallback inbound key must be checked against cross-channel false-positive cases before calling it the best fix.
