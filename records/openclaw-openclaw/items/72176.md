## Summary

Keep open. Current main still lacks a fallback inbound dedupe key when provider retries arrive without a stable MessageSid, and the user transcript append path still does not carry the chat.send client run id as an idempotency key; the issue is also paired with open fixing PR #72290, so closing before that PR lands would hide active unresolved work.

## What This Changes

The issue asks OpenClaw to stop intermittently delivering the same inbound user message twice, with duplicate assistant replies, after the 2026.4.24 regression across webchat and WeChat.

## Best Possible Solution

Review #72290 as the canonical implementation candidate, then land or revise a narrow core fix that adds conservative inbound retry dedupe and transcript-level idempotency with tests for unstable IDs, duplicate retries, and intentional repeated messages.

## Reproduction Assessment

Yes. The code-level reproduction is high-confidence: a retry context without stable MessageSid produces an invalid inbound dedupe claim and continues dispatch, while repeated chat.send transcript updates with distinct request ids append user rows without a transcript idempotency key; the live intermittent channel path was not exercised here.

## Solution Assessment

Unclear until #72290 is reviewed. The likely best direction is narrow idempotency at the inbound and transcript layers, but any content-based fallback should be bounded by provider, route, peer, timestamp, and tests to avoid suppressing legitimate repeated messages.
