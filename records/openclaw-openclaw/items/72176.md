## Summary

Keep open. Current main still relies on stable inbound provider message IDs or a reused `chat.send` idempotency key for this class of duplicate suppression, while the user transcript update path still emits rows without an idempotency marker; the linked fixing PR #72290 is open and uses closing syntax, so this issue should stay open until that PR is reviewed, revised, or landed.

## What This Changes

The issue reports intermittent duplicate user-message delivery after upgrading to 2026.4.24, across WebChat and WeChat, with duplicated transcript rows and assistant replies.

## Best Possible Solution

Review #72290 as the canonical implementation candidate, then land or revise a narrow Gateway/core fix that adds stable transcript idempotency for user turns and suppresses retry duplicates with unstable IDs while preserving intentional repeated same-text messages across channels.

## Reproduction Assessment

Yes for a focused code-level reproduction path: current main returns no inbound dedupe key without `MessageSid`, only dedupes `chat.send` by exact reused idempotency key, and emits user transcript rows without transcript idempotency metadata. The live intermittent WSL/WebChat/WeChat path was not rerun in this read-only review.

## Solution Assessment

Unclear until #72290 is reviewed. The transcript idempotency part is narrow, but the fallback inbound key strategy must be checked against cross-channel false positives before calling it the best fix.
