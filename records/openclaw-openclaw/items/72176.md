## Summary

Current main still does not cover the reported retry shape: inbound dedupe is keyed on `MessageSid`/`clientRunId`, and the WebChat user transcript append path still writes user rows without an idempotency key. This issue should stay open because the linked fixing PR is open and uses closing syntax for this exact regression.

## What This Changes

The issue requests a fix for intermittent duplicate user message and assistant reply delivery after upgrading to 2026.4.24, especially when retries reach WebChat or WeChat with unstable request/message ids.

## Best Possible Solution

Review PR #72290 as the canonical fix candidate, then land or revise a narrow regression fix that dedupes retry deliveries with unstable ids while preserving intentional repeated messages and adding focused tests for inbound dedupe and user transcript idempotency.

## Reproduction Assessment

Yes for a code-level reproduction path: current main only dedupes `chat.send`/inbound delivery when the same `clientRunId`/`MessageSid` is reused, and user transcript appends do not carry a user idempotency key. The live intermittent channel repro was not run here.

## Solution Assessment

Unclear until the linked PR is reviewed. User transcript idempotency by `clientRunId` is a narrow maintainable fix, but any content-based fallback key should be checked against intentional repeated messages across channels, accounts, threads, and timestamps.
