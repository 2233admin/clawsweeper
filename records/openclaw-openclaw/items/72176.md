## Summary

Keep open. The issue has an open, unmerged fixing PR (#72290) with closing syntax, and the current main checkout still lacks the two concrete protections described there: fallback inbound dedupe when MessageSid is unavailable/unstable and transcript idempotency on the WebChat user-message append path.

## Best Possible Solution

Review #72290 as the canonical implementation candidate. A good fix should add a conservative fallback inbound dedupe key for retry paths without stable MessageSid, add `clientRunId`/idempotency to user transcript updates, and include regression tests proving duplicate same-timestamp/content deliveries do not create duplicate transcript rows or assistant replies while legitimate repeated messages still pass.
