## Summary

Keep open. The cache-leak problem remains valid on current main, but this PR should not merge as-is because it bundles the narrow cache caps with broad tool-schema changes and a security-sensitive web_search rewrite that breaks the current provider and credential routing model.

## What This Changes

This PR branch adds oldest-entry caps to four agent runtime cache Maps and also changes multiple agent tool schemas while replacing the registry-backed web_search tool with inline provider clients.

## Best Possible Solution

Land a narrow cache-only implementation on current main that caps the four intended Maps with oldest-entry eviction only on new-key insertion, preserves existing web_search provider/runtime/SecretRef behavior, and adds focused regression coverage for cache size and refresh semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The PR replaces the registry-backed `web_search` tool with a hard-coded provider list. Current main supports bundled and runtime providers through `resolveWebSearchDefinition()`/`runWebSearch()`, including Firecrawl, Tavily, Exa, DuckDuckGo, Ollama, SearXNG, and custom runtime providers, so configured setups outside this list would stop working or fall back incorrectly.
  - confidence: 0.93
- **[P1] Use plugin-owned SecretRefs for web_search keys:** `src/agents/tools/web-search.ts:553-563`
  - body: The new credential resolvers read legacy tool config fields or env vars instead of the current plugin-owned `plugins.entries.<plugin>.config.webSearch.*` paths and resolved SecretRefs. Existing users with provider credentials stored in plugin config would get missing-key errors even though their provider is configured.
  - confidence: 0.9
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92`
  - body: This eviction call runs before every `workspaceFileCache.set`, including stale-refresh writes for a file path already in the map. At capacity, refreshing an existing file evicts an unrelated oldest entry and then overwrites the same key, reducing effective capacity and causing avoidable disk churn.
  - confidence: 0.88
- **[P2] Preserve sub-second node clip durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding `minimum: 1000` to `durationMs` rejects values from 0 to 999 ms that the existing schema accepted up to the 300000 ms maximum. That is an unrelated behavior change for camera clips and contradicts the PR's cache-only/no-behavior-change scope.
  - confidence: 0.82
- **[P3] Apply the model cache cap consistently:** `src/agents/context.ts:52-58`
  - body: Only the discovered-model path enforces the new cap, while `applyConfiguredContextWindows` writes to the same cache without a guard. A large configured model list can push the map past the cap, then the next discovered insert can evict configured entries.
  - confidence: 0.76

## Security Review
