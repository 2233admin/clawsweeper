## Summary

Keep this PR open. Current main still does not implement the PR's central cache-cap behavior for bootstrap snapshots, workspace file contents, cache-trace writers, or the model context-token cache. The PR is still a plausible implementation candidate, but it should not be treated as obsolete or ready cleanup because the supplied diff also bundles unrelated tool-schema tightening and a large security-sensitive `web_search` rewrite that bypasses current runtime/plugin provider and SecretRef contracts.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up, but ask for it to be split. The best path is a small current-main PR that only caps the four intended maps, evicts only on new-key insertions, and adds focused regression coverage for bootstrap snapshots, workspace file cache, cache-trace writers, and model context cache behavior. The tool-schema hardening and `web_search` provider rewrite should be removed from this PR or handled separately with explicit provider, credential, and security review.

## Risks / Open Questions

- Merging the current PR as-is could regress non-hard-coded web_search providers, including bundled providers such as Firecrawl, Exa, Tavily, DuckDuckGo, Ollama, and SearXNG, and custom runtime providers registered through the plugin registry.
- Merging the current PR as-is could break plugin-owned `plugins.entries.<plugin>.config.webSearch.*` credential and SecretRef resolution by routing through legacy/env-only provider logic.
- The PR bundles unrelated schema validation changes; at least one review comment identifies a quiet behavior change for sub-second `durationMs` values.
- The cache-cap idea still needs implementation polish: eviction should only run when inserting new keys, and model context cache behavior should avoid displacing configured entries unexpectedly.
