## Summary

Keep this PR open. Current `main` still does not cap the four global cache maps targeted by the PR, so the central memory-growth fix is not obsolete. The PR also remains too broad to treat as ready cleanup: it bundles unrelated tool-schema validation changes and a large `web_search` network/credential routing rewrite that conflicts with the current plugin/runtime provider architecture called out in review comments.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up, but ask for it to be split. The best path is a small current-main PR that only caps the four intended maps, evicts only when inserting new keys, and adds focused regression tests for bootstrap snapshots, workspace file cache, cache-trace writers, and model context cache behavior. The schema hardening and `web_search` provider rewrite should be removed from this PR or handled separately with explicit provider/credential/security review.

## Risks / Open Questions

- The PR is not a narrow cache fix: the provided diff changes 17 files with 2,644 additions and includes broad tool-schema hardening unrelated to the stated four cache maps.
- The PR's `src/agents/tools/web-search.ts` rewrite is security-sensitive because it changes network provider selection and credential routing; current main routes this through plugin/runtime provider contracts, SecretRefs, and registry candidates.
- The narrow cache-cap idea still needs careful implementation details, especially evicting only on new-key insertions and keeping configured model context entries from being displaced unexpectedly.
