## Summary

Keep open. The memory-growth problem is valid on current main, but this PR should not merge as-is because it combines the cache caps with unrelated web_search provider/credential rewrites and broad tool-schema validation changes that create concrete regressions. A narrow cache-only replacement is the right maintainer path.

## What This Changes

The PR adds oldest-entry caps to four agent-runtime Map caches and also changes multiple agent tool schemas plus the web_search provider implementation.

## Best Possible Solution

Land a narrow cache-only implementation on current main that caps the intended Maps with oldest-entry eviction only on new-key insertions, preserves web_search provider/SecretRef contracts and existing tool schemas, and adds focused regression tests for cap and refresh semantics.

## Reproduction Assessment

Yes. Current main has module-level Maps in the reported cache paths with write sites that do not enforce maximum size; repeated distinct session keys, workspace paths, trace file paths, or model IDs reproduce unbounded growth at the source level.

## Solution Assessment

No. The cache-cap direction is appropriate, but this PR is not the best implementation because it mixes the memory fix with unrelated provider, credential, and schema behavior changes; a cache-only replacement is safer and easier to validate.

- **[P1] Route web_search through the provider registry:** `src/agents/tools/web-search.ts:25`
  - body: The PR replaces the registry-backed resolveWebSearchDefinition()/runWebSearch() path with inline provider handling. Custom runtime providers and bundled providers outside the hard-coded list can stop working or fall back incorrectly.
  - confidence: 0.93
- **[P1] Use plugin-owned SecretRefs for web_search keys:** `src/agents/tools/web-search.ts:553-563`

  - confidence: 0.9
- **[P2] Evict workspace cache only for new keys:** `src/agents/workspace.ts:92`
  - body: The eviction call runs before every workspaceFileCache.set, including refreshes for a file path already in the map. At capacity, refreshing an existing file evicts an unrelated oldest entry and reduces effective capacity.
  - confidence: 0.88
- **[P2] Preserve sub-second node clip durations:** `src/agents/tools/nodes-tool.ts:165-170`
  - body: Adding minimum: 1000 to durationMs rejects 0-999 ms values that current main accepts up to the existing 300000 ms maximum. That is an unrelated behavior change for camera clips.
  - confidence: 0.82
- **[P3] Apply the model cache cap consistently:** `src/agents/context.ts:52-58`
  - body: Only the discovered-model path enforces the new model-cache cap, while applyConfiguredContextWindows writes to the same cache without a guard. A large configured model list can push the map past the cap and later have configured entries evicted.
  - confidence: 0.76
