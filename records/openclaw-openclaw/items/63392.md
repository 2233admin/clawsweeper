## Summary

Current main still does not implement per-agent backup, restore, or clone. The backup CLI only exposes whole-install create/verify flows, the archive planner and manifest are whole-state/workspace scoped, and the docs still describe full local-state backup rather than `--agent`/`--as` behavior. Related #41111 and #59280 cover generic restore work, but they do not supersede this narrower per-agent granularity request.

## Best Possible Solution

Keep this issue open as a valid feature request. The best path is to settle the generic restore command and manifest safety model, then add per-agent scoping: `backup create --agent <id>`, restore with optional `--as <id>`, scoped workspace plus `agents/<agentId>` payloads, config-entry rewrite for `id`/`name`/`workspace`/`agentDir`, docs, and focused tests.

## Risks / Open Questions

- Per-agent restore and clone would extract archives, rewrite config, and move files under state/workspace paths, so it needs the existing traversal, containment, no-overwrite, and dry-run protections carried forward.
- The requested archive can include auth profiles and session transcripts; clone behavior needs explicit policy for sensitive credentials, provider state, and historical conversations.
- Open generic restore work (#41111/#59280) is adjacent. A per-agent implementation should share the same manifest and safety model rather than creating a second incompatible restore format.
