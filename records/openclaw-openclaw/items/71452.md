## Summary

Keep open. Current main still has the reported message-action mismatch: the shared text formatter hard-caps rendered rows at 25, while read, pins, and search commands expose larger limits and still route through that formatter. Provider-side pagination is only partial and does not solve the CLI rendering contract.

## What This Changes

The issue requests pagination and requested-limit support for message read, list-pins, search, and related chat listing paths instead of a hardcoded 25-result cap.

## Best Possible Solution

Implement a narrow message action contract fix: make the formatter limit configurable from the requested action limit, preserve the 25-row default when no limit is supplied, propagate list-pins limits through provider actions where supported, expose next-page metadata consistently, and add focused regressions for read, list-pins, and search rendering above 25 rows.

## Reproduction Assessment

Yes. A focused unit reproduction can call formatMessageCliText with read, list-pins, or search payloads containing more than 25 rows and observe that current main truncates rendered output through messages.slice(0, 25).

## Solution Assessment

Yes. The maintainable solution is to thread the existing limit through the formatter and provider dispatch layers, with additive pagination metadata rather than a broad rewrite or provider-specific workaround.
