## Summary

Keep open. Current main still has the shared message CLI renderer hard-capped at 25 rows, while the `read`, `pins`, and `search` CLI paths expose caller limits and still flow through that capped formatter. Provider-side pagination exists in pieces, especially for MS Teams channel and pin listing, but it does not remove the observable CLI output cap or provide a consistent next-page contract.

## What This Changes

Add pagination and caller-controlled limits for message read, pinned-message listing, and message search instead of the hardcoded 25-row rendered output cap.

## Best Possible Solution

Implement an additive message action contract update: thread the caller's requested display limit into `formatMessageCliText` and `renderMessageList`, keep the default 25-row output when no limit is supplied, propagate limits through read/list-pins/search providers where supported, expose next-page metadata consistently, and document any provider caps.

## Reproduction Assessment

Yes. A focused unit check can pass `formatMessageCliText` a `read`, `list-pins`, or Discord-style `search` result with more than 25 message rows and observe truncation from `messages.slice(0, 25)`; I did not run tests because this review was required to stay read-only.

## Solution Assessment

Yes. The narrow maintainable solution is to thread existing caller limits and additive pagination metadata through the shared formatter/action boundary, preserving provider-specific caps rather than rewriting provider implementations wholesale.
