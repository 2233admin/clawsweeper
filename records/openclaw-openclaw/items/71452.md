## Summary

Keep open. Current main still has the reported mismatch: the shared message formatter hard-caps text output at 25 rows, while the CLI exposes `--limit` for read, pins, and search, and the provider-side pagination that exists does not remove the renderer cap or provide a shared cursor contract. The existing comments are aligned with the current code state, including a concrete contributor plan, but no fix is present on main yet.

## What This Changes

The issue requests configurable limits and pagination for message read, pinned-message listing, message search, and Microsoft Teams chat/channel listing instead of a hardcoded 25-row display cap.

## Best Possible Solution

Land a focused message-tool contract fix that makes CLI text rendering honor an explicit requested limit, keeps the current 25-row default when no limit is supplied, threads limits through read/pins/search paths, and adds an additive optional cursor or next-page metadata field for providers that can expose it.

## Repair Work Prompt

Fix the message CLI/message-tool pagination mismatch tracked by https://github.com/openclaw/openclaw/issues/71452. Current main hard-caps `renderMessageList` with `messages.slice(0, 25)` while `message read`, `message pins`, and `message search` expose `--limit`. Make the formatter accept and honor an explicit requested display limit while preserving the existing 25-row default when no limit is provided. Thread the limit through read, pins, and search formatting call sites; if `formatMessageCliText` needs caller options, add a narrow typed parameter from `src/commands/message.ts` rather than a broad runner rewrite. Where provider payloads already expose pagination metadata, surface a consistent optional next-page token/cursor in an additive way without breaking existing payload consumers. For Microsoft Teams, preserve existing `@odata.nextLink` loops and `$top` behavior; do not add dependencies or perform broad provider rewrites. Add focused Vitest coverage showing read, list-pins, and search can render more than 25 rows when a higher limit is requested and still cap at 25 by default. Add a changelog entry because this changes user-visible CLI behavior, crediting the reporter/commenter per repo rules.
