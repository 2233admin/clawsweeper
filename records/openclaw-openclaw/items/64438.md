## Summary

Keep open. Current main still supports remote memory embeddings and local/QMD ranking paths, but it does not provide a Cohere-style remote reranker config, schema, HTTP client, builtin integration, QMD integration, or plugin-facing reranker provider contract. The request is valid, but it needs maintainer design because it would add a new outbound HTTP surface for private memory snippets.

## What This Changes

Request support for remote Cohere-compatible reranker endpoints in memory search, including builtin and QMD behavior, fallback, batching, and timeout configuration.

## Best Possible Solution

Add an additive remote reranker design for memory search: typed config/schema/help/docs, SecretRef-aware credentials, guarded HTTP client for a Cohere-compatible rerank contract, builtin post-retrieval reranking with vector/hybrid fallback on failure, and a clear QMD strategy. If maintainers prefer plugin ownership, first expose a documented memory reranker provider API so Qwen/Cohere/Jina adapters can live outside core without replacing the whole memory backend.

## Risks / Open Questions

- Remote reranking would send memory snippets to an operator-configured HTTP service, so SSRF/private-network policy, SecretRef handling, proxy behavior, logging, and timeout defaults need explicit design before implementation.
- QMD integration is not just a config add: maintainers need to decide whether OpenClaw should post-process QMD results with an external reranker or require upstream QMD support for remote rerankers.
- The current plugin SDK exposes memory embedding providers, not a generic memory reranker provider, so a plugin-first implementation likely needs a new additive contract rather than a standalone external plugin today.
