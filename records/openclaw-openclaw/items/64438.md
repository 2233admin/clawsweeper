## Summary

Keep open. Current main still has remote embedding support and local/QMD ranking paths, but it does not implement a Cohere-style remote reranker config, schema, HTTP client, builtin memory integration, QMD integration, or plugin-facing reranker provider contract. The request is valid and concrete, but it needs maintainer design because it adds a new outbound HTTP path for private memory snippets.

## What This Changes

Request support for remote Cohere-compatible reranker endpoints in memory search, including builtin and QMD behavior, fallback, batching, and timeout configuration.

## Best Possible Solution

Add an additive remote reranker design for memory search: typed config/schema/help/docs, SecretRef-aware credentials, a guarded Cohere-compatible rerank HTTP client, builtin post-retrieval reranking with vector/hybrid fallback on failure, and an explicit QMD strategy. If maintainers prefer plugin ownership, first expose a documented memory reranker provider API so Qwen/Cohere/Jina adapters can live outside core without replacing the whole memory backend.

## Risks / Open Questions

- Remote reranking would send private memory snippets to an operator-configured HTTP service, so SSRF/private-network allowance, SecretRef handling, proxy behavior, header redaction, logging, and timeout defaults need explicit design before implementation.
- QMD integration needs maintainer direction: OpenClaw could post-process QMD results externally, or require upstream QMD support for remote rerankers instead.
- The current plugin SDK exposes memory embedding providers, not a generic memory reranker provider, so a plugin-first solution likely needs a new additive contract before external adapters can cover this cleanly.
