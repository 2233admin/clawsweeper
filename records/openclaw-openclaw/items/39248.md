## Summary

Keep this issue open. Current main still classifies native `sessions_spawn` child keys as non-main sandbox sessions, starts the child through the non-blocking accepted path, and does not show a Docker-backed regression proof that the child either initializes with transcript/token progress or fails with an actionable sandbox startup error. Related #57992 overlaps on sandboxed subagent zombies but is still open and is not a clear superseding canonical item.

## Best Possible Solution

Keep this open and fix the native `sessions_spawn` sandbox startup path generically: under `agents.defaults.sandbox.mode: "non-main"`, a Docker-backed child should either start and record normal session/transcript/token progress or transition the subagent/background task/session surfaces to an error state with an actionable sandbox startup error in logs. Related #57992 can share the same fix if the implementation covers both deterministic default-sandbox stalls and transient Docker container creation failures.
