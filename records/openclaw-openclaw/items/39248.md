## Summary

Keep this issue open. Current main still makes native `sessions_spawn` children non-main sandbox candidates under `agents.defaults.sandbox.mode: "non-main"`, still dispatches the child `agent` run through the non-blocking accepted-ack path, and I did not find a regression/live proof that a Docker-backed child initializes, records transcript/token progress, and surfaces sandbox startup failure cleanly. Related #57992 overlaps on sandboxed subagent zombies, but it is still open and is not a clear canonical replacement for this deterministic default-sandbox repro.

## Best Possible Solution

Keep this open and add a targeted regression or live Testbox proof for native `sessions_spawn` under `agents.defaults.sandbox.mode: "non-main"`. The fix should prove that the child Docker-backed run starts and records transcript/session progress, or that Docker sandbox initialization failures transition the subagent/background task to an error state and surface an actionable error in logs and `subagents`/session status instead of leaving an accepted child run with zero messages/tokens.
