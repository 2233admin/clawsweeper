## Summary

Keep open. Current main still lacks the proposed `doctor:node-runtime` contribution, and the branch is useful, but the latest patch is not merge-ready: it misses an already-EOL Node release line that still passes the runtime guard and it still leaks Windows home paths in pasteable Doctor output. Earlier review feedback for the Linux/macOS home-prefix bug and the Node 22 comment date appears addressed.

## What This Changes

This PR adds a `doctor:node-runtime` health contribution, a Node runtime diagnostics module, and tests for version manager detection, EOL/advisory warnings, and summary formatting.

## Best Possible Solution

Keep the Doctor runtime contribution, but make the advisory logic cover every Node line that can pass OpenClaw's runtime guard and is known EOL or in maintenance, then reuse the shared home-path display helper or match both path separators before merge.

## Reproduction Assessment

Yes. `buildNodeRuntimeWarnings()` with Node 23 and a date after 2025-06-01 returns no advisory despite the upstream Node schedule marking v23 EOL; `buildNodeRuntimeSummary()` with `USERPROFILE=C:\Users\Alice` and a backslash `execPath` does not shorten the home path.

## Solution Assessment

No. The feature direction is reasonable, but the current implementation uses an incomplete static release schedule and a local home-shortening implementation instead of the existing cross-platform helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Warn on Node 23 EOL instead of dropping the advisory:** `src/commands/doctor-node-runtime.ts:129-136`
  - body: `runtimeSatisfies()` accepts Node 23 because it is above the 22.14 minimum, so users on that line reach this Doctor check. Node v23 reached EOL on 2025-06-01, but this table only includes major 22, which makes `buildNodeRuntimeWarnings()` return no advisory for an already-EOL runtime. Please add the known non-LTS/EOL line or otherwise warn for known EOL majors below the recommended runtime.
  - confidence: 0.91
- **[P2] Handle backslash home paths when shortening execPath:** `src/commands/doctor-node-runtime.ts:242-243`
  - body: On Windows, `USERPROFILE` and `process.execPath` normally use `\`, but this boundary check only accepts `${homeDir}/`. A version-managed path under `C:\Users\Alice\...` will therefore print the full username in copy-pasteable Doctor output instead of a home-relative path; reuse `shortenHomePath()` or accept a `\` boundary too.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No dependency, workflow, or supply-chain change was found, but the new always-on Doctor summary has a low-severity privacy leak on Windows home paths.

Concerns:

- **[low] Windows Doctor output can disclose usernames:** `src/commands/doctor-node-runtime.ts:242`
  - body: The summary formatter only shortens paths with a `/` separator after the home directory. Windows `USERPROFILE` and `process.execPath` commonly use backslashes, so support-ready Doctor output can include `C:\Users\<name>` even though a shared helper already avoids this.
  - confidence: 0.84
