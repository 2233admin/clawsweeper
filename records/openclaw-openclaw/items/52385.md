## Summary

Current main still does not enforce internal-only `sessions_send` delivery for agent-to-agent Slack-bound sessions. The initial injected target run is internal (`deliver: false`, internal channel), but the A2A follow-up announce flow can still call the gateway `send` method for the resolved surface channel/thread when the announce reply is anything other than `ANNOUNCE_SKIP`. Current docs and tests also describe/assert visible A2A announce delivery, so this needs maintainer product-contract review rather than cleanup closure.

## Best Possible Solution

Keep this open and have a maintainer decide the A2A delivery contract. The likely product fix is to make inter-agent `sessions_send` internal-only by default, or add an explicit `silent`/internal-only API option with safe defaults, then update the A2A announce path, protocol/docs, and tests so Slack channel-bound and thread-bound targets cannot receive visible internal coordination traffic while the target session transcript still receives the message.
