## Summary

Keep open. Current main has useful post-release hardening, but it still preserves the two risky startup paths from the report: native slash-command deploy can retry rate-limited deploys, and bot identity still hard-fails startup on `fetchUser("@me")` failures. The issue is concrete, recent, and not the same SecretRef mismatch tracked by #74765.

## What This Changes

The issue reports a Discord v2026.4.29 startup regression involving OAuth/application and bot identity fetch timeouts plus repeated native slash-command deploy HTTP 429s.

## Best Possible Solution

Discord startup should avoid amplifying Discord REST trouble: prefer configured or parsed IDs where possible, skip or cool down native command deployment after a startup 429 without making multiple deploy attempts, and keep the gateway usable with existing slash commands when identity preflight is temporarily rate-limited or timing out.

## Reproduction Assessment

Partially yes. The deploy side is source- and test-reproducible because current tests expect three rate-limited deploy attempts; the `users/@me` timeout needs the reporter's Discord/network conditions, but the hard-fail startup path is visible in current code.

## Solution Assessment

No. Current main's post-release changes are useful but not sufficient because they quiet repeated deploy logs and improve queued REST retry handling without removing the deploy retry amplification or relaxing the hard startup dependency on `fetchUser("@me")`.

Current main already has an application ID bypass and collapsed deploy 429 logging, but `deployDiscordCommands` still retries rate-limited deploy failures up to three times and `fetchDiscordBotIdentity` still hard-fails startup on `fetchUser("@me")` timeout/failure. Make startup avoid amplifying Discord REST trouble: treat native slash-command deploy 429s as a single non-fatal skip/cooldown that leaves existing commands active, avoid repeating deploy while identity/application preflight is failing or rate-limited, and use configured/parsed IDs where possible. Do not broaden into SecretRef handling, #74765, unrelated Discord routing, or global REST scheduler policy unless a small seam is required.
