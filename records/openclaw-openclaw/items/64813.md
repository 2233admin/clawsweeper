## Summary

Keep this PR open. Current main now has a safer generic `params.extra_body`/`extraBody` path for OpenAI-compatible Completions payloads, which partially covers the attribution use case, but it still does not implement the direct `params.user` passthrough this PR proposes. The PR is also unsafe as-is because its wrapper injects `user` into every provider payload without an API-family guard or tests.

## Best Possible Solution

Keep this PR open and request changes, or replace it with a maintainer-owned patch. The best implementation is a guarded static `params.user` payload wrapper limited to provider API families that actually support the field, with string normalization, tests proving injection for supported OpenAI/OpenRouter-style payloads, tests proving no mutation for unsupported providers, and docs clarifying whether users should prefer direct `params.user` or the newer `params.extra_body.user` escape hatch. Keep the broader dynamic metadata/trace design tracked under #63901.

## Risks / Open Questions

- Merging this PR as-is could send an unexpected top-level `user` key to non-OpenAI payloads such as Anthropic or Gemini-native routes, causing provider rejections or undefined request semantics.
- A global `agents.defaults.params.user` value could be propagated to providers the operator did not intend to receive that attribution identifier.
- Closing this PR now would still leave direct `params.user` support unimplemented on main; the existing `extra_body` path is useful but narrower and uses a different config shape.
