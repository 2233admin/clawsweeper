## Summary

Keep open. Current main has file-backed memory, hybrid search, short-term promotion, optional LanceDB storage, and a provenance-rich Memory Wiki layer, but it still lacks the requested conservative durable write resolver with canonical create/merge/update/conflict outcomes, required source attachment, entity normalization, and non-destructive revision history.

## What This Changes

The issue requests a Memory MVP write pipeline that classifies candidate memories, attaches sources, dedupes or merges near-duplicates, routes contradictions to review, and preserves provenance plus revisions before durable writes.

## Best Possible Solution

Design and implement the durable memory resolver as a coordinated memory host/plugin contract: typed canonical records, mandatory source references, normalized subjects/entities, near-duplicate and contradiction lookup, explicit create/merge/update/conflict-for-review outcomes, and append-only revision/provenance history, with Memory Wiki, Markdown, LanceDB, and indexing layers staying as supporting surfaces.

## Reproduction Assessment

Not applicable as a feature architecture request. The high-confidence verification path is source/docs inspection: current main exposes search, append-only flush, promotion, and wiki claims, but no canonical durable write resolver or storage schema.

## Solution Assessment

Unclear for a concrete patch because no implementation is under review. The requested direction is valid, but the maintainable solution should be a coordinated schema and resolver design rather than a one-off change to Markdown flush, short-term promotion, or Memory Wiki metadata.
