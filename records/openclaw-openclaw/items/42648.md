## Summary

Keep open. Current main still provides memory search/indexing, append-style flush and promotion, an optional LanceDB store with high-similarity duplicate checks, and a provenance-rich memory-wiki companion, but it does not provide the requested resolver-backed durable write pipeline with required source attachment, canonical records, merge/update/conflict outcomes, entity normalization, or revision history. The related memory schema, provenance, review/conflict, and CLI/operator-surface issues remain open, so this should stay as active product architecture work.

## What This Changes

Build a conservative memory write pipeline that classifies candidate memories, attaches sources, checks duplicates and conflicts, resolves create/merge/update/review outcomes, and preserves provenance plus revision history before durable writes.

## Best Possible Solution

Design this as a first-class durable resolver layer coordinated with #42646, #42647, #42650, and #42651: typed candidates, required source references, canonical SQLite records, entity normalization, duplicate and contradiction lookup, explicit create/merge/update/conflict-for-review outcomes, and non-destructive revision history, while keeping memory-wiki as a companion knowledge layer rather than the active write-path replacement.
