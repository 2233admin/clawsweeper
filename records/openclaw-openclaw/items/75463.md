## Summary

Keep open. Current main still has the bare/prefixed in-flight race described in the PR, and the branch is a narrow functional repair with focused regression coverage. I found no blocking correctness or security issue, but the user-facing BlueBubbles fix should carry a changelog note before merge.

## What This Changes

The PR changes BlueBubbles reply-context fallback to cache a validated `p:n/<guid>` alias when that prefixed lookup joins an existing bare in-flight fetch, and adds a concurrency regression test.

## Best Possible Solution

Land the narrow in-flight alias caching repair with its regression test, then keep the BlueBubbles reply-context fallback documentation and changelog aligned while avoiding any broader cache or API behavior changes.

## Reproduction Assessment

Yes. A high-confidence reproduction path is two concurrent `fetchBlueBubblesReplyContext()` calls for the same account and bare id, with the bare `msg-race` call starting first and `p:0/msg-race` joining the shared in-flight promise; current main returns the existing promise before caching the prefixed alias.

## Solution Assessment

Yes for the functional bug. Caching the validated prefixed alias in a continuation on the shared in-flight promise is the narrowest maintainable fix because it preserves single-request dedupe while recording each caller's alias after a successful fetch; the remaining safer addition is a changelog update before merge.
