## Summary

Current main has partial mitigation for this failure class, but the exact stale versioned runtime-deps cleanup requested by the report is not implemented. Package updates now stop/restart managed services through the updated install, and startup prunes stale `openclaw-unknown-*` runtime-deps roots, but current code deliberately leaves versioned roots such as `openclaw-2026.4.25-*` in place and I did not find a regression covering the reported `2026.4.24` to `2026.4.26` stale-root crash-loop.

## Best Possible Solution

Keep this issue open. The best path is a maintainer-owned fix that either safely removes or fully ignores stale `openclaw-{version}-{hash}` runtime-deps roots during package updates/startup, preserves lock safety and `OPENCLAW_PLUGIN_STAGE_DIR` path-list semantics, and adds a regression for an old `2026.4.24` versioned root surviving beside a new install. Docs should be updated if cleanup behavior becomes part of the supported contract.
