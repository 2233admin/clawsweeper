## Summary

Keep open. Current main still prunes only `openclaw-unknown-*` runtime-deps roots and has a regression test that explicitly preserves stale versioned roots, so the reported stale `openclaw-{version}-*` update crash-loop cleanup is not implemented yet.

## What This Changes

The issue requests automatic cleanup or safe ignoring of stale versioned `plugin-runtime-deps/openclaw-{version}-*` roots after `openclaw update` so Gateway does not crash-loop on old staged runtime code.

## Best Possible Solution

Define a runtime-deps ownership policy that either prunes stale OpenClaw-owned versioned roots or ignores incompatible old roots during startup/update/doctor, while preserving locked roots and read-only path-list layers. Add regression coverage with an old versioned root beside the active version and an `openclaw-unknown-*` root.

## Reproduction Assessment

Yes for the current-main cleanup gap: source inspection and the existing focused pruning test show stale versioned roots survive pruning. The full launchd crash-loop was not live-reproduced during this read-only review.

## Solution Assessment

Unclear whether the issue body's update-only deletion sketch is the best final implementation. A safer fix would share ownership-aware cleanup or ignore logic across Gateway startup, package-update recovery, and the manual deps command, with explicit lock and layered-root tests.
