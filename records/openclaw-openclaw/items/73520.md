## Summary

Keep open. Current main still does not implement the requested stale versioned `plugin-runtime-deps` cleanup: startup and `plugins deps --prune` only prune `openclaw-unknown-*` roots, and the focused regression test still asserts an old `openclaw-{version}-*` root is preserved. The adjacent update restart and runtime-deps lock fixes reduce related failure modes, but they do not close this issue's versioned-root cleanup/ignore policy.

## What This Changes

The issue requests that OpenClaw update/startup clean up or safely ignore stale versioned `plugin-runtime-deps/openclaw-{version}-{hash}` roots so Gateway restarts after package updates do not crash-loop on old runtime mirrors.

## Best Possible Solution

Implement an ownership-aware runtime-deps root policy in the plugin/runtime-deps owner area: either remove stale `openclaw-{version}-{hash}` roots that are provably OpenClaw-owned, inactive, and unlocked, or make startup/update resolution explicitly ignore incompatible old versioned roots. The final behavior should preserve path-list layer semantics, lock safety, package-update hard restarts, and include a regression with an old 2026.4.24 root beside the current install root.

## Reproduction Assessment

Yes for the current-main cleanup gap: source inspection shows only `openclaw-unknown-*` roots are pruned, and the regression test asserts an old versioned root survives. The complete macOS launchd crash-loop was not live-reproduced during this read-only review.

## Solution Assessment

Unclear whether the issue body's update-only deletion sketch is the best final implementation. A safer maintainable fix is a locked, ownership-aware cleanup or ignore path that works during both package update and Gateway startup without breaking staged-root layering.
