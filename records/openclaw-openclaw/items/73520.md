## Summary

Keep open. Current main still only prunes stale `openclaw-unknown-*` runtime-deps roots and still preserves stale versioned `openclaw-{version}-*` roots in the focused regression test, so the reported update crash-loop class is not implemented on main. The package-update restart path has adjacent mitigation for the old-process reload part, and related runtime-deps PRs/issues cover locks and timeout amplification, but none is a canonical fix for stale versioned roots.

## What This Changes

The issue requests automatic cleanup or ignoring of stale versioned `plugin-runtime-deps/openclaw-{version}-*` roots after `openclaw update` so the Gateway does not crash-loop on old staged runtime code.

## Best Possible Solution

Keep this open and implement an ownership-aware runtime-deps root policy in the plugin/runtime-deps owner area: either remove stale `openclaw-{version}-{hash}` roots that are provably OpenClaw-owned and unlocked, or make startup/update resolution ignore incompatible old versioned roots. The final behavior should preserve `OPENCLAW_PLUGIN_STAGE_DIR` path-list layering, lock safety, package-update hard restarts, and add a regression with an old `2026.4.24` root beside the current install root.

## Reproduction Assessment

Yes for the current-main cleanup gap: source inspection shows only `openclaw-unknown-*` roots are pruned, and the focused test asserts an old versioned root survives. The complete macOS launchd crash-loop was not live-reproduced during this read-only review.

## Solution Assessment

Unclear whether the issue body's update-only deletion sketch is the best final implementation. A safer maintainable fix is a locked, ownership-aware cleanup or ignore path that works during both package update and Gateway startup without breaking staged-root layering.
