## Summary

Keep open. Current main still has only the adjacent unknown-root pruning path, and the focused regression test still asserts that stale versioned `openclaw-{version}-*` runtime-deps roots survive pruning. The issue is recent, labeled In Progress, and the related PRs in the discussion address adjacent timeout/lock amplification rather than the stale versioned-root cleanup policy itself.

## What This Changes

The issue requests that `openclaw update` and Gateway startup recover from stale versioned `plugin-runtime-deps/openclaw-{oldVersion}-*` roots left after package updates.

## Best Possible Solution

Implement an ownership-aware runtime-deps root policy in the plugin/runtime-deps owner area: remove or ignore stale `openclaw-{oldVersion}-{hash}` roots only when they are provably OpenClaw-owned and unlocked, preserve path-list layering, and add a regression with an old versioned root beside the current install root after an update.

## Reproduction Assessment

Yes for the current-main cleanup gap: the pruning helper can be checked with a stale `openclaw-2026.4.25-*` root, and the existing test asserts that versioned root remains. The exact five-hour macOS launchd crash-loop was not live-reproduced during this read-only review.

## Solution Assessment

Unclear whether the issue body's update-only deletion sketch is the best final fix. A safer solution is an ownership-aware cleanup or resolution-ignore policy that works across update, Gateway startup, locks, and staged-root path-list semantics.
