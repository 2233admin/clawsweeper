## Summary

Keep open. Current main still has the reported stale-versioned-root cleanup gap: the shared prune helper only considers `openclaw-unknown-*` runtime-deps roots, Gateway startup and `openclaw plugins deps --prune` both use that helper, and the focused regression test still asserts that a stale `openclaw-2026.4.25-*` root survives pruning. Adjacent update/restart hardening reduces some failure modes but does not implement cleanup or ignore handling for old versioned roots.

## What This Changes

Clean or ignore stale versioned `plugin-runtime-deps/openclaw-{version}-*` roots after update/startup so old OpenClaw runtime mirrors cannot crash the Gateway.

## Best Possible Solution

Keep this open until runtime-deps owners implement one shared policy that prunes stale OpenClaw-owned versioned roots or ignores incompatible old roots during startup/update resolution, with regression coverage for an old versioned root beside the active install root and preserved handling for locks, `openclaw-unknown-*` roots, and path-list staging.

## Reproduction Assessment

Yes for the current-main cleanup gap: source inspection and the existing pruning test show stale versioned roots survive. The full launchd crash-loop was not replayed during this read-only review.

## Solution Assessment

Unclear whether the issue body's update-only deletion sketch is the best final fix. A safer solution is an owner-reviewed cleanup or ignore policy shared by Gateway startup, package-update recovery, and the CLI/cache-prune surface.
