## Summary

Current main still has only partial mitigation for this update crash-loop class: package updates now stop/restart the managed service and startup prunes stale `openclaw-unknown-*` roots, but versioned roots such as `openclaw-2026.4.24-*` are still retained and there is no regression covering the reported stale-versioned-root case.

## What This Changes

The issue requests automatic cleanup or safe ignoring of stale versioned `plugin-runtime-deps/openclaw-{version}-{hash}` roots after `openclaw update` to prevent Gateway crash-loops.

## Best Possible Solution

Keep this issue open. The best fix is to make startup/update either safely ignore or remove stale versioned `openclaw-{version}-{hash}` runtime-deps roots while preserving active locks, multi-root stage-dir lookup, and the current version’s install root, then add a regression for an old `2026.4.24` root surviving beside a newer install.
