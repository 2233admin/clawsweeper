## Summary

Keep this issue open. Current main still has the reported active-provider fast path gated by `!hasExplicitPluginConfig(params.cfg?.plugins)`, `plugins.entries` still makes that helper return true, and the linked implementation PR #73794 is still open/unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open while maintainers review and land #73794 or an equivalent core fix. The likely fix belongs in `src/plugins/capability-provider-runtime.ts`: reuse active providers for non-memory/non-speech capability lookups even when `cfg.plugins.entries` is non-empty, keep memory and speech reconciliation unchanged, and add a regression test in `src/plugins/capability-provider-runtime.test.ts` for an active non-speech provider with explicit plugin entries.
