## Summary

Keep open. Current main does not implement the workflow seams this PR is trying to add, and the PR head still has concrete merge-blocking issues in attachment config propagation and default scope routing for dynamic session actions.

## What This Changes

This PR adds generic Plugin SDK workflow seams for session actions, host-mediated attachments, scheduled session turns, finalize retry, priority injections, Gateway/Swift protocol updates, advanced workflow fixtures, and QA hardening.

## Best Possible Solution

Keep the generic workflow-seam direction, but repair the PR branch so attachment delivery uses the same registry-scoped config for lookup and send execution, dynamic session actions get a nonempty default operator scope path for CLI/default callers, and focused regression tests cover both cases before the broader SDK/Gateway validation continues.

## Reproduction Assessment

Yes. The config bug is reproducible by sending an attachment from a registry configured with a non-default session store/config: route lookup uses params.config, while sendMessage falls back to global config. The default-scope bug is reproducible by invoking plugins.sessionAction through the CLI/default Gateway path without explicit scopes; the call connects with [] and is rejected before the action handler enforces requiredScopes.

## Solution Assessment

No. The broad additive SDK/Gateway design is plausible, but this PR head is not the best mergeable solution until the scoped-config propagation and dynamic method scope routing are fixed with targeted tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass the scoped config into attachment delivery:** `src/plugins/host-hook-workflow.ts:161-170`
  - body: `sendPluginSessionAttachment` resolves the delivery route with `params.config`, but the actual `sendMessage` call omits `cfg: params.config`. In custom session-store, multi-workspace, or isolated plugin contexts, lookup and delivery can use different configs, causing failed delivery or delivery through the wrong channel/account. Pass the same config into `sendMessage` and assert it in the workflow test.
  - confidence: 0.88
- **[P2] Keep session actions on a nonempty default scope:** `src/gateway/method-scopes.ts:290-291`
  - body: `plugins.sessionAction` is classified as a dynamic method, so CLI/default callers with no explicit scopes use `resolveLeastPrivilegeOperatorScopesForMethod` and connect with `[]`; `authorizeGatewayMethod` then rejects the method before the action handler can enforce its `requiredScopes`. Give this dynamic method a nonempty default operator scope or let CLI callers fall back to the default operator scopes.
  - confidence: 0.82

## Security Review
