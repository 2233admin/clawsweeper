## Summary

Keep open. Current main does not implement the workflow-seam stack, and the PR head still has a concrete P2/security-sensitive config propagation bug in attachment delivery before it is mergeable.

## What This Changes

This PR adds generic Plugin SDK workflow seams for session actions, host-mediated session attachments, scheduled session turns, finalize retry, priority injections, Gateway/Swift protocol updates, advanced workflow fixtures, and QA hardening.

## Best Possible Solution

Keep the PR open, pass the registry-scoped config through the actual sendMessage call, add a regression assertion for cfg propagation, then continue the broader SDK/Gateway review and validation on the repaired head.

## Reproduction Assessment

Yes. On PR head, use sendSessionAttachment from a registry with a non-default session store/config; route lookup uses params.config, but outbound delivery omits cfg and falls back to global runtime config.

## Solution Assessment

No. The generic workflow-seam direction is appropriate, but the current patch is not mergeable until attachment delivery uses the same scoped config for both route lookup and send execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass registry config to attachment sends:** `src/plugins/host-hook-workflow.ts:161-170`
  - body: `sendPluginSessionAttachment` now resolves the delivery route with `params.config`, but the actual `sendMessage` call omits `cfg: params.config`. In custom session-store, multi-workspace, or isolated plugin contexts, route lookup and delivery can use different configs, causing failed delivery or sending through the wrong configured channel/account. Pass the same config into `sendMessage` and assert it in the workflow test.
  - confidence: 0.88
