## Summary

Keep open. Current main does not implement the workflow SDK/Gateway seams this PR adds, and the latest PR head still has one concrete P2 delivery/config bug in the new host-mediated attachment path.

## What This Changes

This PR adds Plugin SDK workflow seams for typed session actions, host-mediated session attachments, scheduled session turns, finalize retry support, priority injections, Gateway/Swift protocol updates, advanced workflow fixtures, and QA hardening.

## Best Possible Solution

Keep the PR open, pass the registry-scoped config through the actual attachment send path, add regression coverage that asserts sendMessage receives cfg, then complete the public SDK/Gateway review and exact-head validation before landing.

## Reproduction Assessment

Yes. On PR head, call sendSessionAttachment from a registry using a non-default session store/config; route lookup uses params.config, but outbound delivery falls back to global runtime config because cfg is omitted from sendMessage.

## Solution Assessment

No. The overall generic workflow-seam direction is appropriate, but the current patch is not the best mergeable solution until attachment delivery uses the same registry-scoped config for both route lookup and send execution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Pass registry config to attachment delivery:** `src/plugins/host-hook-workflow.ts:161-170`
  - body: `sendPluginSessionAttachment` now resolves the delivery route with `params.config`, but the actual `sendMessage` call omits `cfg: params.config`. In custom session-store, multi-workspace, or isolated plugin contexts, that can route with one config and deliver with the process-global config, causing failed delivery or a send through the wrong channel/account. Pass the same config into `sendMessage` and assert it in the workflow test.
  - confidence: 0.88
