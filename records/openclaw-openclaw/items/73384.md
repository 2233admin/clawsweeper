## Summary

Keep open. Current main does not already include the workflow SDK/Gateway seams, so this PR is not obsolete, but the current head still has confirmed blocking issues in attachment config propagation and default CLI scope routing for `plugins.sessionAction`.

## What This Changes

This PR adds generic Plugin SDK workflow seams for session actions, host-mediated session attachments, scheduled session turns, finalize retry support, Gateway/Swift protocol updates, advanced workflow fixtures, and related QA hardening.

## Best Possible Solution

Keep the consolidated workflow-seam PR open, repair the two confirmed regressions on the PR branch, add focused regression assertions for config forwarding and default session-action scopes, then continue normal maintainer review of the additive SDK/Gateway surface.

## Reproduction Assessment

Yes. Static current-head reproduction is high-confidence: use `sendSessionAttachment` with a non-default registry config and observe `sendMessage` receives no `cfg`, and call `plugins.sessionAction` through CLI default scope selection to see an empty scope list rejected before action dispatch.

## Solution Assessment

No. The generic workflow-seam direction is appropriate and current main does not implement it, but the present patch is not the best merge candidate until scoped outbound config and dynamic method scope defaults are fixed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Pass scoped config into attachment delivery:** `src/plugins/host-hook-workflow.ts:161-170`
  - body: `sendPluginSessionAttachment` now uses `params.config` to find the session route, but the actual `sendMessage` call omits the same config. In custom-store or multi-workspace contexts, route lookup can use one config while delivery falls back to the global runtime config, causing failed delivery or artifacts sent via the wrong channel/account. Pass `cfg: params.config` to `sendMessage` and assert it in the attachment test.
  - confidence: 0.9
- **[P2] Avoid zero-scope defaults for session actions:** `src/gateway/method-scopes.ts:290-291`
  - body: Classifying `plugins.sessionAction` while leaving it without a fixed least-privilege scope makes `callGatewayCli` connect with `[]` scopes by default. The server special-case rejects session actions unless at least one operator scope is present, so CLI/Gateway callers that do not pass explicit scopes cannot invoke plugin actions at all. Return a non-empty default scope for this dynamic method or keep it on the normal CLI default-scope path.
  - confidence: 0.86
