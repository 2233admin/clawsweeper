## Summary

Keep open. Current main still supports OpenAI-compatible `input_type` forwarding for memory embeddings but does not expose or forward Jina's separate `task` field, and the issue is now paired with open PR #74444 using closing syntax, so the issue should remain open until that implementation is reviewed and landed or replaced.

## What This Changes

The issue requests configurable Jina v5 embedding task parameters, such as query/document task fields or an extra embedding request body, so OpenAI-compatible Jina embedding requests can use task-specific adapters.

## Best Possible Solution

Add a narrow typed memory embedding task seam for OpenAI-compatible providers, forward the query/document task values to inline and batch embedding requests, include them in cache identity, and update config schema, docs, tests, and changelog while preserving existing `inputType` behavior.

## Reproduction Assessment

Yes. The gap is reproducible by static inspection on current main: the strict `memorySearch` schema has no task fields, and both inline and batch OpenAI-compatible embedding payload builders omit `task`.

## Solution Assessment

Yes for keeping this issue open. The best implementation direction is a typed query/document task surface reviewed through #74444 or a narrow replacement, rather than closing the issue before the linked PR merges.
