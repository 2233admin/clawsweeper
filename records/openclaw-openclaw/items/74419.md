## Summary

Keep open. Current main still only forwards OpenAI-compatible `input_type` for memory embeddings and does not expose or send Jina's `task` request field; this issue is also already paired with open closing PR #74444, so it should remain open until that PR lands or is replaced.

## What This Changes

Add a memory-search configuration surface that can forward Jina Embeddings v5 task values to OpenAI-compatible embedding requests, ideally with separate query and document settings.

## Best Possible Solution

Review and land #74444 or an equivalent typed implementation that carries task values through config types, schema, resolved memory settings, OpenAI-compatible request bodies, batch indexing, cache identity, SDK-facing options, docs, changelog, and focused tests without weakening existing `input_type` behavior.

## Reproduction Assessment

Yes. A code-level reproduction is available: configure a Jina v5 memory embedding task field or inspect request construction, and current main has no accepted config key or request-body path for `task`; no live Jina call is needed to verify this deterministic gap.

## Solution Assessment

Yes, the typed query/document task direction is the best narrow fix for OpenClaw core. A generic `extraBody` passthrough would solve the immediate API shape too, but it is broader and should only be used if maintainers explicitly want that extension surface.
