## Summary

Keep open. Current main still has no configurable Jina/OpenAI-compatible `task` field in the memory embedding config or OpenAI embedding request bodies, and the issue is explicitly paired with open PR #74444 through closing syntax, so it should remain the tracking issue until that PR or an equivalent replacement lands.

## What This Changes

Request adding a configurable `task` parameter for Jina v5/OpenAI-compatible memory embeddings, preferably split between query and document embeddings.

## Best Possible Solution

Land a narrow implementation through the linked PR or an equivalent replacement that adds typed query/document task fields through config types, schema, resolved config, provider create options, inline and batch OpenAI embedding request bodies, cache identity, docs, changelog, and focused tests.

## Reproduction Assessment

Yes. A source-level reproduction is enough: current main accepts and forwards only the existing `inputType` family, and both OpenAI embedding request builders omit `task`, so a Jina v5 `retrieval.query` or `retrieval.passage` adapter cannot be configured.

## Solution Assessment

Yes. The typed `queryTask`/`documentTask` direction is the narrowest maintainable fix for asymmetric Jina behavior, as long as it is wired through both query and indexing paths and included in cache identity.
