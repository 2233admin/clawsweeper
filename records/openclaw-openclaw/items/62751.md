## Summary

Keep open. Current main emits `session:patch` for the `sessions.patch` RPC path, but the reported `/model` and `session_status` model-change paths still mutate session entries directly without dispatching the internal hook, so the observable hook gap remains.

## What This Changes

Request that session model changes made through `/model` and agent-internal model mutation paths emit a `session:patch` hook event.

## Best Possible Solution

Centralize explicit session model mutations so `sessions.patch`, `/model`, and `session_status` all emit the same non-blocking `session:patch` event after a successful store update, with focused regression coverage for the slash-command and tool paths.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: run a `/model` directive or `session_status` call with a `model` parameter and mock `triggerInternalHook`; current code updates the session store through direct model-override paths, while only the `sessions.patch` RPC handler dispatches the hook.

## Solution Assessment

Yes, Option A is the best direction. Reusing the existing `session:patch` pipeline or a shared event helper is narrower and more consistent with the documented hook contract than adding a second `session:model_changed` surface first.
