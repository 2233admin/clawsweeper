## Summary

Keep open. Current main still lacks recursive or depth-bounded subagent listing: the tool and slash-command paths list only direct controlled children, while nested workers are exposed only through descendant counts or child-session summaries. The issue is recent, corroborated by a second user comment, and narrow enough for a focused backward-compatible repair.

## What This Changes

The issue requests `depth` and `recursive` options for subagent listing so a parent agent can inspect nested workers with hierarchy metadata.

## Best Possible Solution

Add a list-only, bounded traversal mode that defaults to todayâ€™s direct-child behavior, accepts finite `depth` plus `recursive`, returns flat rows with `depth` and `parentSessionKey`, and documents the orchestrator workflow without broadening session control permissions.

Yes. A high-confidence static reproduction path exists: seed a main-owned orchestrator run plus an orchestrator-owned worker run, then the current main list path returns only the orchestrator while the worker appears only as descendant summary metadata.

## Solution Assessment

Yes. The requested `depth`/`recursive` direction is the narrow maintainable solution if it reuses the existing descendant traversal helpers, remains bounded, and preserves direct-owner control semantics for non-list actions.
