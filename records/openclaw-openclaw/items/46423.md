Keep this PR open. Current main still does not implement transient-specific recurring retry before the next natural schedule, so the underlying bug remains valid, but the PR is not merge-ready because it mixes the cron fix with unrelated guardrail allowlist changes and extends the one-shot retry config contract into recurring runtime behavior without a matching maintainer decision.

## What This Changes

The PR changes recurring cron/every error handling to classify transient failures and retry on a backoff path, adds cron restart/regression coverage, and includes unrelated test and guardrail updates from branch maintenance.

## Best Possible Solution

Land a focused cron scheduler fix that retries transient recurring failures before the next natural schedule while preserving permanent-error cadence, manual force-run cadence, restart catch-up behavior, and an explicit documented retry contract. The final diff should keep targeted cron tests and remove unrelated guardrail, Discord, script, and CI-drift changes.
