## Summary

Keep open. Current main still has a credible native Google transport path where provider-prefixed configured Google model ids can be preserved into the REST resource path, and a second user confirmed the same configuration shape after upgrading; overlapping open PRs already target the same normalization class.

## What This Changes

Reports 404 Model Not Found when Google Gemini model refs such as `google/gemini-3.1-flash-lite-preview` work in the main session but fail during sub-agent spawn or session reset.

## Best Possible Solution

Land one narrow Google/model-resolution fix that strips only redundant active-provider prefixes before native Google model matching and URL construction, preserves valid nested ids for wrapper providers, and adds regression coverage for configured Google model entries plus sub-agent spawn/reset inheritance.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction is to configure `models.providers.google.models[].id` as `google/gemini-3-flash-preview` or `google/gemini-3.1-flash-lite-preview` with `api: google-generative-ai`, resolve the bare inherited Google model, and observe that current URL construction can produce `models/google/<model>`.

## Solution Assessment

No. Current main is not the best solution because it still preserves redundant same-provider prefixes into the native Google transport path; the safer fix is scoped prefix stripping coordinated with the existing open prefix-normalization PRs.
