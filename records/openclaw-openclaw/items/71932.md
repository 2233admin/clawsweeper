## Summary

Keep open. Current main still has a credible Google Generative AI path where a configured Google model id like `google/gemini-...` is preserved into native Google URL construction as `models/google/gemini-...`, matching the 404 class confirmed in the issue discussion. Overlapping open PRs already target this prefix-normalization area, so the right next step is maintainer consolidation rather than closure or a new autonomous fix branch.

## What This Changes

Fix Google Gemini sub-agent and session-reset requests so provider-prefixed `google/...` model ids do not produce Google API 404s.

## Best Possible Solution

Keep this issue open and land one focused Google-owned normalization fix that strips only redundant active-provider prefixes before Google model matching and native transport URL construction, while preserving valid nested ids for wrapper providers such as OpenRouter and NVIDIA-style model refs.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction is to configure `models.providers.google.models[].id` as `google/gemini-3-flash-preview` or `google/gemini-3.1-flash-lite-preview`, resolve it through the Google provider, and observe that current URL construction would build `models/google/<model>`.

## Solution Assessment

No. Current main is not the best solution because it still preserves the redundant same-provider prefix into the Google native transport; the safer path is a scoped same-provider prefix strip coordinated with #41398 and #41517.
