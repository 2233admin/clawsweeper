## Summary

Keep this open. Current main still has a credible Google native transport path where an inline configured Google model id such as `google/gemini-3.1-flash-lite-preview` can be preserved into the resolved model and then sent as `models/google/...`; the newer reporter comment provides a concrete matching Google provider config, and overlapping prefix-normalization PRs remain open rather than merged.

## What This Changes

The issue reports that Google Gemini 3.1 Flash-Lite works in the main session but sub-agent spawn or session reset fails with a 404, likely because Google model ids are normalized differently on that path.

## Best Possible Solution

Keep the issue open and land one consolidated, Google-owned normalization fix that strips only redundant active-provider prefixes before Google model matching and native Google transport URL construction, while preserving valid cross-provider nested ids and adding regression coverage for sub-agent spawn, session reset, and inline `models.providers.google.models[].id` entries written as `google/<model>`.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to configure `models.providers.google.models[].id` as `google/gemini-3.1-flash-lite-preview`, resolve it through the Google provider for a sub-agent or reset path, and observe current code can preserve that id until the native Google URL builder constructs `models/google/gemini-...`.

## Solution Assessment

No. Current main is not the best solution because it only normalizes some explicit model refs; the maintainable fix is a scoped same-provider prefix strip at Google/provider resolution or transport boundaries, coordinated with the two open prefix PRs.
