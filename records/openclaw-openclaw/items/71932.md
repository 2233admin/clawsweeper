## Summary

Keep open. Current main still has a credible Google native transport path where a configured inline Google model id shaped as `google/<model>` is matched but then carried forward as `model.id`, so the Google Generative AI URL builder would request `models/google/<model>`; two related prefix-normalization PRs remain open and should be consolidated rather than closing this report.

## What This Changes

Fix Google sub-agent and session-reset model handling so provider-prefixed Gemini model ids such as `google/gemini-3.1-flash-lite-preview` are sent to the Google Generative AI API as valid model resource names.

## Best Possible Solution

Land one consolidated Google-owned normalization fix that strips only redundant same-provider prefixes before Google model matching and native Google transport URL construction, while preserving cross-provider nested model ids. Add regression coverage for `sessions_spawn`, session reset, and inline `models.providers.google.models[].id` values written as `google/<model>`.

## Reproduction Assessment

Yes. A high-confidence code-path reproduction is to configure an inline Google provider model with id `google/gemini-3-flash-preview` or `google/gemini-3.1-flash-lite-preview`, resolve it through the Google provider, and observe that current URL construction would build `models/google/<model>`.

## Solution Assessment

No. Current main is not a complete solution; the best fix is a scoped same-provider prefix normalization path coordinated with #41398 and #41517, not another broad normalization that could rewrite valid wrapper-provider ids.
