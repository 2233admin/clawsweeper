## Summary

Keep this issue open. Current main still rejects BOM-prefixed markdown frontmatter before skill parsing, so a Windows-edited `SKILL.md` can still disappear from `openclaw skills list`; the related #45432 report covers a separate write/edit-tool BOM failure, and #74163 is an open maintainer tracker rather than a focused implementation fix.

## What This Changes

The issue asks OpenClaw to keep discovering Windows-edited skills whose `SKILL.md` begins with a UTF-8 BOM before YAML frontmatter.

## Best Possible Solution

Update the shared markdown frontmatter parser to tolerate one leading UTF-8 BOM before the opening `---`, then add focused parser and workspace skill-loading coverage proving a BOM-prefixed `SKILL.md` remains discoverable while normal non-BOM behavior stays unchanged.
