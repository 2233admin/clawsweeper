## Summary

Keep this issue open. Current main and the latest release still reject BOM-prefixed markdown frontmatter before skill parsing, so a Windows-edited `SKILL.md` can still be dropped from discovery; #45432 is a separate write/edit-tool BOM report, and #74163 is a tracker rather than a focused implementation PR.

## What This Changes

The issue asks OpenClaw skill discovery to tolerate a leading UTF-8 BOM in `SKILL.md` frontmatter so Windows-edited skills remain visible in `openclaw skills list`.

## Best Possible Solution

Update the shared markdown frontmatter parser to tolerate one leading UTF-8 BOM before the opening `---`, then add focused parser and skill-loading coverage proving a BOM-prefixed `SKILL.md` remains discoverable. Keep the write/edit-tool BOM behavior tracked separately from #45432.

## Repair Work Prompt

Fix Windows BOM-prefixed `SKILL.md` discovery. Current main rejects content beginning with `\uFEFF---` in `src/markdown/frontmatter.ts`, which causes `src/agents/skills/local-loader.ts` to drop otherwise valid skills because `description` is missing after parsing. Change the shared frontmatter parser to tolerate exactly one leading UTF-8 BOM before the opening `---`; do not strip arbitrary invisible characters, change closing delimiter semantics, or modify the separate write/edit-tool BOM behavior from #45432. Add a `parseFrontmatterBlock` regression test for `\uFEFF---` and a skill-loading/status/list-path test proving a BOM-prefixed `SKILL.md` remains discoverable with its name and description. Add a user-facing changelog fix entry crediting `Thanks @ianpor`. Validate with targeted tests first, then the changed gate in Testbox before handoff.
