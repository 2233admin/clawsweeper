## Summary

Keep open. Current main and the latest release expose adjacent subagent hooks only; the requested post-completion/pre-announcement extension point is still missing. `subagent_ended` is observation-only and too narrow, `subagent_delivery_target` only changes routing, and the announce flow builds and delivers the completion event without a public plugin interception point. The issue discussion also notes that related delivery fixes improved symptoms without covering this API gap.

## What This Changes

Add a public extension hook that fires after a spawned subagent completes and before its completion announcement is surfaced to the parent/requester, with bounded completion context and optional announcement enrichment, suppression, or fallback handling.

## Best Possible Solution

Keep this open for maintainer API design. The best path is either a public typed plugin hook at the post-subagent-completion/pre-announcement point or a canonical lifecycle/delivery design that explicitly covers the same scope: bounded child/run metadata, approved completion output or safe child-history access, timing/usage where available, delivery status/error visibility where appropriate, and clear return semantics for announcement enrichment, suppression, or filtering, with docs and ordering/regression tests.

## Risks / Open Questions

- The requested hook would expose child completion data and possibly transcript/history access to plugins, so it needs a bounded privacy and permission contract before implementation.
- Allowing plugins to suppress or modify completion announcements could create silent-delivery regressions unless return semantics, timeouts, error handling, and ordering are designed carefully.
- The request overlaps with broader lifecycle/delivery architecture work such as #66638; a narrow hook should not accidentally hard-code a contract that conflicts with a later event-delivery model.
