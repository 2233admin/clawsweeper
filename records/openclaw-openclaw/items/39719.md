## Summary

Keep this open. Current main still reuses persisted `skillsSnapshot` entries using only snapshot version and agent skill filter, while neither the persisted nor runtime snapshot records the workspace/source-root identity needed to detect an iCloud-to-local workspace move. Related #73765 tracks the same stale-cache family for config-based skill disabling, but it does not fully supersede this workspace-migration path issue.

## Best Possible Solution

Implement workspace-aware skill snapshot invalidation. Add normalized workspace/source-root provenance or a stable inputs hash to `SkillSnapshot` and `SessionSkillSnapshot`, populate it when building snapshots, compare it before reusing persisted `sessions.json` entries, invalidate on `agents.defaults.workspace` and per-agent workspace changes, and add a regression test with an old iCloud absolute `SKILL.md` path that must be rebuilt against the new local workspace.

## Clownfish Work Prompt

Fix stale skill snapshots after workspace migration. Current main resolves the active workspace in `src/agents/agent-command.ts` but reuses `sessionEntry.skillsSnapshot` using only snapshot version and agent skill filter, while `SkillSnapshot`/`SessionSkillSnapshot` do not record workspace or skill-source provenance. Add normalized workspace/source-root provenance or a stable inputs hash to the runtime and persisted skill snapshot shape, populate it in `buildWorkspaceSkillSnapshot`, and refresh persisted snapshots when the current workspace/source roots/config inputs differ. Include workspace config changes such as `agents.defaults.workspace` and per-agent `workspace` in invalidation or hash inputs. Keep core generic; do not add iCloud-specific path rules or one-off session migrations. Add a regression test that seeds a session with a stale absolute `SKILL.md` path from an old iCloud-style workspace and verifies the next turn rebuilds the snapshot against the new workspace. Also cover version-0 restart behavior and `skills.entries.*.enabled` compatibility with related #73765 where feasible. Changelog is not required unless the maintainer wants a user-facing bugfix note.
