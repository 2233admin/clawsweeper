## Summary

Keep open. Current main still rejects the Gateway `agent` RPC on `sendPolicy=deny` before checking whether the request is actually delivering externally, and the paired closing PR #73500 is still open and unmerged. This should wait for that PR or an equivalent focused fix to land.

## What This Changes

The issue asks for non-delivery Gateway agent invocations to bypass outbound `session.sendPolicy=deny` enforcement while preserving the block for explicit delivery requests.

## Best Possible Solution

Land #73500 or an equivalent focused change in `src/gateway/server-methods/agent.ts` that enforces `sendPolicy=deny` only when `request.deliver === true`, with regression coverage proving non-delivery calls run and explicit delivery calls remain blocked.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: call the Gateway `agent` method for a session whose `resolveSendPolicy` result is `deny` while omitting `deliver` or setting it false; current main returns the policy error before computing delivery intent.

## Solution Assessment

Yes. The narrow maintainable fix is to guard the Gateway policy check with `request.deliver === true`, mirroring the embedded agent command path while preserving the outbound-delivery safety check.
