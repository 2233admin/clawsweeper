## Summary

Keep open. Current main still has no Gateway system-events recovery-notice policy, and the recovery notice remains prompt-visible and channel-visible on the relevant recovery paths.

## What This Changes

Add a documented `gateway.systemEvents.recoveryNotice` config policy that can keep current recovery notices user-visible, log them only, or suppress them for customer-facing bots.

## Best Possible Solution

Implement a Gateway-owned config policy with the current behavior as the default, apply it consistently to prompt-visible recovery events and out-of-band recovery delivery, keep restart continuations unaffected, and document the operator-facing modes.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: a config auto-recovery sentinel flows through `loadRestartSentinelStartupTask` into both `enqueueSystemEvent` and `deliverRestartSentinelNotice`, while startup/reload recovery directly enqueues `Config recovery warning` for the main session.

## Solution Assessment

Yes, with tight scope. A Gateway config knob is the narrow maintainable solution if it gates both recovery-notice implementations instead of adding another banner-specific prompt filter.
