## Summary

Keep open. Current main still has no `gateway.systemEvents.recoveryNotice` policy, and the recovery paths still enqueue prompt-visible system events and, for restart-sentinel recovery, outbound channel delivery. The related #71289, #49579, and #39307 context does not supersede this general operator-controlled recovery-notice policy request.

## What This Changes

The issue requests a Gateway config knob to suppress or downgrade config-recovery notices for customer-facing bots while preserving current behavior by default.

## Best Possible Solution

Add a Gateway-owned, documented config policy with current behavior as the default, gate both prompt-visible recovery system events and outbound restart-sentinel recovery delivery, and leave last-known-good recovery plus restart continuations unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: trigger last-known-good config recovery or a `config-auto-recovery` restart sentinel, then observe that current main queues `System:` prompt content and, when routed, sends an outbound channel notice without any config-mode check.

## Solution Assessment

Yes. A general `gateway.systemEvents.recoveryNotice` policy is the narrow maintainable direction because banner-specific filters like #49579 do not cover all recovery notices or customer-facing channel personas.
