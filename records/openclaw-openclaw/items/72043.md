## Summary

Keep open. Rechecking current main at bb3a0c9 shows the requested Gateway recovery-notice policy is still missing: config recovery notices are still prompt-visible, restart-sentinel recovery notices are still delivered to routed channels, and the related WhatsApp banner filter and plugin-local recovery narrowing do not cover this general customer-facing bot case.

## What This Changes

The issue requests a `gateway.systemEvents.recoveryNotice` configuration policy to control whether automatic Gateway config-recovery notices are injected into prompts, delivered out-of-band, logged only, or suppressed.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates config-recovery notices across both prompt-visible system events and out-of-band restart-sentinel channel delivery, while leaving actual config recovery and restart continuation delivery unchanged.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available on current main: whole-file recovery calls `enqueueConfigRecoveryNotice`, and `config-auto-recovery` restart sentinels flow through `scheduleRestartSentinelWake`, which enqueues a system event and sends the outbound notice when routed.

## Solution Assessment

Yes. The proposed `gateway.systemEvents.recoveryNotice` policy is the narrow maintainable solution if it covers both recovery-notice implementations; adding more banner-specific prompt filters like #49579 would not solve the general operator-controlled policy need.
