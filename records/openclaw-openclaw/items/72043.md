## Summary

Keep this issue open. Current main still lacks the requested `gateway.systemEvents.recoveryNotice` config contract, and the recovery notices remain hardwired into prompt-visible system events and, for restart-sentinel recovery, outbound channel delivery.

## What This Changes

Add a `gateway.systemEvents.recoveryNotice` config knob to control whether config recovery notices are shown to the agent/user, logged only, or suppressed for customer-facing bots.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that covers both restart-sentinel recovery notices and startup/reload `Config recovery warning` events, while leaving actual recovery and restart continuations unchanged.

## Reproduction Assessment

Yes. A code-level reproduction is clear: a `config-auto-recovery` restart sentinel with a session route flows through `loadRestartSentinelStartupTask`, which enqueues the message as a system event and sends the outbound notice; drained system events are then prepended to the next agent prompt.

## Solution Assessment

Yes, with one scope refinement. The proposed `gateway.systemEvents.recoveryNotice` knob is a narrow maintainable fix if it gates both restart-sentinel notices and the newer startup/reload config-recovery warning path rather than adding another banner-specific filter like #49579.
