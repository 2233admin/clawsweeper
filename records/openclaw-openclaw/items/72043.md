## Summary

Keep open. Current main still lacks the requested Gateway config policy, and both recovery-notice paths remain prompt-visible without a mode gate; related work narrows recovery triggers or filters one WhatsApp banner, but does not solve the general customer-facing recovery notice problem.

## What This Changes

Request adding `gateway.systemEvents.recoveryNotice` with `user`/`log`/`off` modes to control Gateway recovery notices without changing actual recovery or restart continuations.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that gates recovery notices across prompt-visible system events and out-of-band channel delivery, supports log-only and suppressed modes, and leaves config recovery plus restart continuation delivery unchanged.

## Reproduction Assessment

Yes. A code-level reproduction is clear: a `config-auto-recovery` restart sentinel flows through `loadRestartSentinelStartupTask`, which unconditionally enqueues a system event and sends the outbound notice; startup/reload recovery also directly enqueues `Config recovery warning` into the main session.

## Solution Assessment

Yes, with one scope refinement. A `gateway.systemEvents.recoveryNotice` knob is the narrow maintainable fix if it covers both recovery-notice implementations, rather than adding more banner-specific prompt filters like #49579.
