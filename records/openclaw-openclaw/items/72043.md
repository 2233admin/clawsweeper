## Summary

Keep open: current main still lacks the requested recovery-notice config policy, and recovery notices remain prompt-visible and channel-visible on the relevant Gateway paths. The related items are adjacent context, not a merged or canonical replacement for this narrower config-policy request.

## What This Changes

Request adding a `gateway.systemEvents.recoveryNotice` config option to control whether last-known-good config recovery notices are user-facing, log-only, or suppressed.

## Best Possible Solution

Add a Gateway-owned, documented config policy with a current-behavior default that controls recovery-notice delivery across prompt system events, out-of-band channel notices, and log-only or suppressed modes while leaving actual recovery and restart continuations unchanged.

## Reproduction Assessment

Yes. A code-level reproduction is clear: a `config-auto-recovery` restart sentinel with a session route flows through `loadRestartSentinelStartupTask`, which enqueues the message and sends the outbound notice; drained system events are then prepended to the next agent prompt.

## Solution Assessment

Yes, with one scope refinement. The proposed `gateway.systemEvents.recoveryNotice` knob is a narrow maintainable fix if it covers both restart-sentinel recovery notices and `Config recovery warning` notices rather than adding another banner-specific filter like #49579.
