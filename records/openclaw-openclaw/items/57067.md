## Summary

Keep open. Current main still makes the Agents page primary navigation a single native dropdown, and the Default button only stages an unsaved config draft, so both parts of the reported UX regression remain observable in source. The related closed #61887 overlaps the visibility complaint, while open #54178 is adjacent selector friction rather than a complete fix for this issue.

## What This Changes

The issue requests that the Control UI Agents page show agents directly instead of only inside a dropdown and that Default button changes persist after refresh.

## Best Possible Solution

Implement visible desktop agent navigation on the Agents page, keep an accessible compact/mobile selector if needed, and make the Default action persist through the existing `agents.list[].default` config contract with state refresh and focused UI tests.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: render the Agents page with multiple agents and it exposes a single `.agents-select`; click Default and the current handler only stages a dirty config draft, so a refresh before Save loses it.

## Solution Assessment

Yes. A visible list/sidebar plus an immediately persisted default action is the narrowest maintainable direction for the reported workflow; retaining a mobile dropdown is fine as long as desktop visibility and persistence are fixed.
