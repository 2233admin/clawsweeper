## Summary

Keep this open. Current main still lacks a Copilot-owned `compat.supportsEagerToolInputStreaming: false` setting on the static Claude catalog, default Copilot model builder, and forward-compatible synthetic model path, while the pinned pi-ai dependency defaults that flag to true and injects `tools[].eager_input_streaming`. The built-in boundary-aware Anthropic transport omits the field when selected, but that is not positive evidence that every shipped Copilot/pi-ai path avoids the dependency default reported in v2026.4.29 logs.

## What This Changes

The issue reports that GitHub Copilot Claude models still send pi-ai's `eager_input_streaming` tool field on v2026.4.29 and asks for Copilot Anthropic model metadata to disable eager tool input streaming.

## Best Possible Solution

Add a provider-local Copilot compat helper or normalization path so every `github-copilot` model using `anthropic-messages` carries `compat.supportsEagerToolInputStreaming: false`, covering manifest/static catalog entries, `buildCopilotModelDefinition`, and `resolveCopilotForwardCompatModel`, with focused regression tests and no core/pi-ai behavior change.

## Reproduction Assessment

Yes. The report provides a concrete v2026.4.29 CLI command and matching gateway errors, and source plus dependency inspection confirms the missing Copilot metadata allows pi-ai's default eager tool streaming behavior on any path that reaches pi-ai. I did not run the live Copilot command because no Copilot/GitHub token variable was present in env or `~/.profile`.

## Solution Assessment

Yes. A Copilot-owned compat flag or normalization hook is the narrowest maintainable fix because the rejection is specific to GitHub Copilot's Anthropic-compatible proxy; changing core Anthropic defaults or the pi-ai dependency would be broader and riskier.
