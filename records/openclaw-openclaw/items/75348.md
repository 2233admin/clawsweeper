## Summary

Keep this open. Current main still lacks a Copilot-owned `compat.supportsEagerToolInputStreaming: false` setting for Claude Copilot models in the static catalog, default builder, and dynamic model resolver, while pi documents that Anthropic-compatible models default to sending per-tool `eager_input_streaming` unless that compat flag is disabled. ([pi.dev](https://pi.dev/docs/latest/models))

## What This Changes

The issue reports that GitHub Copilot Claude models still reject `tools[].eager_input_streaming` on v2026.4.29 and asks for Copilot Anthropic model metadata to disable eager tool input streaming.

## Best Possible Solution

Add a provider-local Copilot compat helper or normalization path so every `github-copilot` model using `anthropic-messages` carries `compat.supportsEagerToolInputStreaming: false`, covering manifest/static catalog entries, `buildCopilotModelDefinition`, and `resolveCopilotForwardCompatModel`, with focused regression tests and no core/pi-ai behavior change.

## Reproduction Assessment

Yes. The report provides a concrete v2026.4.29 CLI command and matching gateway errors, and current source plus the pi-ai compatibility contract confirm that missing Copilot metadata can leave eager tool streaming enabled.

## Solution Assessment

Yes. A Copilot-owned compat flag or normalization hook is the narrowest maintainable fix because the rejection is specific to GitHub Copilot's Anthropic-compatible proxy; changing core Anthropic defaults or patching pi-ai would be broader and riskier.
