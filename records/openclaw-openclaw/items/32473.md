## Summary

Keep open. Current main still rejects non-localhost plain-HTTP Control UI sessions without device identity, the browser still skips device identity when WebCrypto is unavailable, docs describe that as the current security policy, and open PR #56830 uses closing syntax while changing only messaging rather than the underlying policy.

## What This Changes

Requests that Control UI support or better handle Docker/VPS/LAN access where non-localhost plain HTTP fails with `control ui requires device identity` because browser device identity is unavailable outside HTTPS or localhost.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment. Review #56830 as the current messaging-only candidate, use #53274 for any actual remote-HTTP device-identity design change, and use #71669 if maintainers want startup warning/operator diagnostics without changing the security policy.
