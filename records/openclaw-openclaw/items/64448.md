## Summary

Keep open. Current main still leaves the reported Feishu image duplicate-send path unresolved: image media is uploaded and then sent/replied without a stable provider `uuid`, and the shared outbound context still has no idempotency key for Feishu to consume. The related Feishu rate-limit issue is adjacent delivery reliability work, but it does not supersede this image reconnect/dedupe report.

## What This Changes

The issue requests a fix for occasional duplicate Feishu image sends when WebSocket reconnects during the media upload/send flow.

## Best Possible Solution

Add a backwards-compatible outbound send idempotency field through the shared direct-send/action context, have the Feishu plugin pass it as Lark/Feishu `uuid` on image `im.message.create` and `im.message.reply`, and cover reconnect/retry duplicate simulation while preserving intentional repeated sends.

## Repair Work Prompt

Fix openclaw/openclaw#64448. Current main can duplicate Feishu image delivery when a WebSocket reconnect/retry re-enters the image send path because `sendMediaFeishu` uploads the image and then calls `sendImageFeishu` without a stable Feishu `uuid`. Add a backwards-compatible outbound idempotency field through the shared direct-send/action context, have the Feishu plugin pass it as `uuid` on `client.im.message.create` and `client.im.message.reply` for image sends, and add focused regression tests showing the same logical outbound send reuses the same UUID while intentional separate sends do not collapse. Keep the fix scoped to Feishu media/idempotency and the minimal shared context seam; do not implement broad Feishu rate-limit retry/backoff from #70879 unless the maintainer explicitly expands scope. Likely files: `src/channels/plugins/outbound.types.ts`, `src/cli/send-runtime/channel-outbound-send.ts`, `extensions/feishu/src/outbound.ts`, `extensions/feishu/src/media.ts`, and their colocated tests. Changelog only if the final change is user-facing behavior, with required attribution. Do not touch dependency versions, lockfiles, or unrelated channel retry policy.
