## Summary

Current main still leaves the reported Feishu image duplicate-send failure plausible and unresolved: the image path uploads media and then sends/replies without a stable provider `uuid`, and the shared outbound contexts do not expose a send idempotency key for Feishu to consume.

## What This Changes

The issue reports intermittent duplicate Feishu image sends when WebSocket reconnect occurs during the media upload/send path and asks for stable idempotency or retry protection.

## Best Possible Solution

Keep this issue open and implement a narrow Feishu delivery fix: add an additive stable outbound send idempotency key through the shared direct-send/action contexts, map it to Feishu/Lark `uuid` on `im.message.create` and `im.message.reply` for image sends, and cover duplicate retry/reconnect simulation without blocking intentional separate sends.

## Repair Work Prompt

Fix the Feishu duplicate outbound image send risk reported here. Current main routes image sends through `extensions/feishu/src/media.ts` (`sendMediaFeishu` -> `uploadImageFeishu` -> `sendImageFeishu`) and never passes a stable Feishu/Lark message `uuid`; shared outbound contexts also lack a provider send idempotency key. Implement a backward-compatible, additive per-send idempotency field through the narrow send surfaces that can reach Feishu direct sends and message-tool sends, then map it to `uuid` on `client.im.message.create` and `client.im.message.reply` for image messages. Do not use a mediaUrl-only in-flight lock that would suppress intentional later sends of the same image. Do not change Feishu auth, WebSocket reconnect policy, SDK versions, dependency patches, or unrelated channel behavior. Add focused regression coverage that simulates duplicate retry/reconnect invocation with the same per-send key and verifies Feishu receives the same `uuid`, while separate sends get separate keys. Update user-facing changelog with a single-line fix entry thanking @lonewolfqql if the fix changes shipped behavior. Related refs: this issue and #70879 only as adjacent reliability context, not as a duplicate.
