## Summary

Current main at 7e5d6dba still lacks the missing-plugin-id guard: `plugins enable <id>` and `plugins disable <id>` mutate config from the raw CLI argument before any registry lookup. This PR remains a valid implementation candidate for the linked bug #73551; related PRs #73932 and #74016 are closed and unmerged, while #73623 is an overlapping open candidate that maintainers should choose or consolidate.

## Best Possible Solution

Keep this PR open for maintainer review as an implementation candidate for #73551. The best fix is one focused landing that validates enable/disable ids against the discovered plugin snapshot before `enablePluginInConfig` or `setPluginEnabledInConfig`, exits nonzero with a clear not-found error, skips config writes and registry refreshes on misses, preserves exact-id behavior for valid plugins, and leaves cleanup of already-stale entries to a separate doctor/fix path.
