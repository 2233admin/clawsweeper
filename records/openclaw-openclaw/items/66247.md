## Summary

Keep this PR open. Current main still uses Telegram's plain per-chat/per-topic sequentializer, so ordinary follow-up messages in a busy topic cannot reach core immediate queue handling until the adapter lane drains. The PR remains a relevant unmerged implementation candidate; its current diff has addressed the earlier queued-alias and media-only review points, but it still needs maintainer review for embedded-only active runs, per-instance config resolution, and the new public SDK runtime export.

## Best Possible Solution

Keep this PR open and finish the Telegram-specific busy-session bypass, or land an equivalent current-main implementation. The best fix should preserve existing `/stop`, `/status`, `/btw`, and approval lanes; use canonical queue-mode precedence and aliases; honor per-instance config; detect both reply-run and embedded-only active states; cover text and media-only immediate turns; keep ordinary collect/followup serialization; and avoid widening the SDK contract more than the Telegram plugin actually needs.
