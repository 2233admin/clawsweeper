## Summary

Keep this PR open. Current main still registers Telegram's plain per-chat/per-topic sequential key and has no busy-session control lane for ordinary messages in immediate queue modes. The PR remains a relevant unmerged implementation candidate, but the supplied review thread still identifies embedded-run and per-instance config concerns that should be resolved before merge.

## Best Possible Solution

Keep the PR open and finish the Telegram-specific busy-session bypass, or land an equivalent current-main implementation. The best path is to preserve existing control lanes, match canonical queue-mode precedence and aliases, honor per-instance config, detect both reply-run and embedded-only busy states, cover text and media-only immediate turns, prove ordinary collect/followup serialization still holds, and keep any SDK seam narrow and documented.
