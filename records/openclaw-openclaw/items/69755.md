## Summary

Keep open. Current main still emits a direct `Run: openclaw doctor --non-interactive` restart follow-up without checking whether the delivery surface can complete approvals, and the related doctor-on-restart PR is still open rather than shipped.

## What This Changes

The issue asks for post-restart/config-patch doctor follow-up messages to distinguish recommended checks from commands that are actionable in the current approval-constrained surface.

## Best Possible Solution

Adjust the restart sentinel doctor follow-up wording or payload context so constrained surfaces present the doctor check as a recommended terminal/approvals-capable follow-up, while preserving the actual doctor command for places where it is actionable.

## Reproduction Assessment

Yes. Current main statically reproduces the bad loop: config/update/restart sentinel payloads include the unconditional doctor hint, and the formatted message is delivered to heartbeat/outbound surfaces without an approval-capability check. The exec approval path can produce the reported `approval-timeout (allowlist-miss)` when the command is attempted under a constrained policy.

## Solution Assessment

Yes, the issue's suggested direction is the right boundary. The narrow maintainable fix is in restart sentinel follow-up copy/context handling, not in weakening exec approvals or changing doctor semantics.

Fix https://github.com/openclaw/openclaw/issues/69755 on current main. The observable problem is that restart/config-patch/update notices tell users to `Run: openclaw doctor --non-interactive` even when the notice is delivered through heartbeat/chat/approval-constrained surfaces where that command may hit exec approval timeout/allowlist constraints. Keep the fix narrowly in restart sentinel follow-up messaging/context handling. Prefer wording such as a recommended follow-up to run doctor in a terminal or approvals-capable OpenClaw UI when the current surface may not be able to complete approvals; preserve the concrete command and profile-aware formatting. Do not weaken exec approvals, do not make doctor more permissive/mutating, and do not add channel-specific core ids. Check related open PR #64684 before editing so the fix does not duplicate its auto-doctor implementation. Likely update `formatDoctorNonInteractiveHint` or add a small typed option used by config/write/update/gateway restart sentinel callers, update affected tests, and add a changelog fix entry crediting `Thanks @1yihui`.
