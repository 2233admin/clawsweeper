## Summary

Keep this PR open. Current main still supports configured Telegram proxies and standard HTTP(S) proxy env routing through the Telegram transport, but it does not implement the PR's Telegram-specific `OPENCLAW_TELEGRAM_PROXY` override. The submitted patch is also stale against the current plugin layout because it targets removed `src/telegram/*` paths instead of `extensions/telegram/*`.

## Best Possible Solution

Keep open and ask for a narrow refresh against the current Telegram plugin. If maintainers want a Telegram-only override, implement it under `extensions/telegram` with shared proxy resolution and precedence `channels.telegram.proxy` / per-account proxy > `OPENCLAW_TELEGRAM_PROXY` > standard `HTTPS_PROXY`/`HTTP_PROXY`, wire it through send, monitor, probe, audit, and chat lookup paths, and add focused tests plus docs. If the existing `channels.telegram.proxy`, standard process proxy env support, and global proxy feature are considered sufficient, close only after that explicit product decision is documented.

## Risks / Open Questions

- The PR cannot merge as-is because it targets removed `src/telegram/*` files; a replacement patch needs to be scoped to `extensions/telegram/*` and the plugin SDK boundary.
- A Telegram-specific env proxy can carry Telegram bot-token API traffic through an operator-provided proxy, so a refreshed patch needs explicit docs and tests for precedence, `NO_PROXY`, credential exposure/redaction expectations, blank values, and malformed URL behavior.
- Current explicit proxy behavior tends to fail closed on dispatcher construction errors while standard env proxy can fall back to direct transport; maintainers should choose the intended behavior for `OPENCLAW_TELEGRAM_PROXY` before landing.
