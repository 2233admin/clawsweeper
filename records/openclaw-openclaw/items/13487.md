## Summary

Keep open. Current main still lacks the requested Discord multi-bot precedence because route selection happens before Discord target signals are interpreted, mention/reply checks only compare against the running handler's bot ID, and the shared mention-gating result has no default-owner suppression state. Related PRs #47393 and #67460 overlap parts of the area but are open partials, not a merged implementation of this issue.

## What This Changes

Request deterministic Discord routing precedence so explicit bot mentions win first, reply-target bot authors win second, and the per-channel default owner responds only when no targeted signal applies.

## Best Possible Solution

Implement target selection inside the Discord plugin before shared default routing: derive a configured Discord bot-user-ID to OpenClaw account/agent map, choose explicit mentioned bot accounts first, choose the referenced-message bot author second, and fall back to the per-channel default owner only when no targeted signal exists. Add focused Discord preflight, shared mention-gating, and route resolver coverage proving targeted cases suppress the default owner.
