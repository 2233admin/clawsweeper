## Summary

Keep open. Current main still does not implement deterministic Discord multi-bot target precedence: preflight resolves the normal route before target-signal handling, mention/reply metadata is only compared with the currently running bot user ID, and there is no configured bot-id-to-agent selection or default-owner suppression path for another targeted bot.

## What This Changes

Request deterministic Discord routing where explicit bot mentions win over reply-target bot metadata, both suppress the per-channel default owner, and only untargeted messages fall back to the default owner.

## Best Possible Solution

Implement target selection inside the Discord plugin before shared default routing: map configured Discord bot user IDs to OpenClaw accounts/agents, select explicit mentioned bot accounts first, select the referenced-message bot author second, and fall back to the channel default owner only when no targeted signal exists. Add focused preflight/routing tests proving targeted cases suppress the default owner.

## Reproduction Assessment

Yes. GTZhou supplied a concrete live guild-channel reproduction, and current-main static inspection shows the same failure path: route selection precedes target evaluation and mention/reply checks compare only against the running botUserId.

## Solution Assessment

Yes. The narrow maintainable solution is Discord-owned target selection plus a small generic suppression seam only if shared mention-gating must express default-owner suppression; hardcoding Discord identities into core routing would cross the plugin boundary.
