## Summary

Keep open. Current main already covers the Feishu part through the newer registry-backed configured-binding path, but this PR is not fully implemented on main: `acp.defaultChannels` is absent, catch-all ACP bindings without `match.peer.id` are still rejected/skipped, and QQ Bot still lacks configured ACP binding provider/inbound route readiness handling. The PR discussion also contains useful resolved review notes and the companion QQ Bot implementation link `sliverp/qqbot#156`, so the remaining work is still salvageable rather than obsolete.

## Best Possible Solution

Keep this PR open as stale-but-useful work to rebase or split. Preserve main's generic configured-binding registry, keep the already-shipped Feishu path, and port the remaining QQ Bot support by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-runtime`, with route/provider tests and QQ Bot docs. Treat `acp.defaultChannels`, catch-all bindings, and ACP auto-enable from bindings as a separate maintainer product/security decision.

## Risks / Open Questions

- Landing this PR as-is would likely conflict with main's newer registry-backed configured-binding architecture and the current `openclaw/plugin-sdk/conversation-runtime` SDK path.
- `acp.defaultChannels` and the proposed `acpx` auto-enable behavior are broad routing controls: a small config change could route every conversation on a channel/account through an external ACP CLI runtime, so they need explicit product and security review.
- QQ Bot integration needs to fit its bridge/engine split without bypassing access control, mention gating, route policy, or configured-binding readiness failure handling.
- Security pass found no workflow, dependency, lockfile, or package-publishing files in the provided PR file list, but the runtime-routing and plugin auto-enable changes expand external code execution reach.
