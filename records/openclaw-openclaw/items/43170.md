## Summary

Keep open. Current main has superseded the narrow Feishu/type-widening part with the generic configured-binding registry and shipped Feishu ACP route readiness, but it still does not implement the PR's central remaining behavior: `acp.defaultChannels`, catch-all ACP bindings without `match.peer.id`, QQ Bot configured ACP binding/provider support, or acpx auto-enable from ACP bindings alone. The PR discussion remains useful, including the resolved reverse-lookup review and the companion QQ Bot pointer `sliverp/qqbot#156`, so this is stale but salvageable rather than cleanup-closeable.

## Best Possible Solution

Keep this PR open for maintainer follow-up, ideally as a rebase or split. Preserve current main's generic configured-binding registry and shipped Feishu path, then port the QQ Bot portion by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-runtime`, with route/provider tests and QQ Bot docs. Treat `acp.defaultChannels`, catch-all ACP bindings, and acpx auto-enable from bindings as a separate product/security decision.
