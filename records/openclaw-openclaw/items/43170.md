## Summary

Keep open. Current main has superseded the Feishu/type-widening part with a newer generic configured-binding registry and Feishu route readiness path, but this PR is not fully implemented on main: `acp.defaultChannels` is absent, catch-all ACP bindings without `match.peer.id` are still rejected/skipped, and QQ Bot has no configured ACP binding provider or inbound configured-binding route readiness handling. The PR discussion also contains useful resolved review notes and the companion QQ Bot implementation pointer `sliverp/qqbot#156`, so the work is stale-but-salvageable rather than obsolete.

## Best Possible Solution

Keep this PR open as stale-but-useful work to rebase or split. Preserve main's generic configured-binding registry and shipped Feishu path, then port the remaining QQ Bot support by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-runtime`, with route/provider tests and QQ Bot docs. Treat `acp.defaultChannels`, catch-all ACP bindings, and ACP auto-enable from bindings as a separate maintainer product/security decision.
