## Summary

Keep open. Current main does not implement PR #70607: the cross-agent path still resolves the target agent's bound account. But the PR is not safe conservative cleanup either: current tests explicitly cover target-agent bound routing for cross-agent native and ACP spawns, while also preserving the caller account for same-agent spawns. The one-character PR change conflicts with those contracts and has an unresolved review concern, so this needs maintainer behavior judgment rather than auto-closing.

## Best Possible Solution

Keep the PR open for maintainer judgment, but do not land it as-is. If the intended invariant is that completion requester origins always preserve the parent account, implement a targeted change that preserves same-agent caller-account behavior and updates the existing native/ACP tests that currently assert target-agent bound routing. If the current target-agent binding behavior is intentional, close or revise #70574 with that documented routing contract instead.

## Risks / Open Questions

- The current implementation and tests conflict with the linked issue's parent-origin invariant; resolving this may require changing existing native and ACP account-routing contracts.
- The PR's one-character `!==` to `===` change appears too broad because it would re-enable binding resolution for same-agent spawns, a path current main explicitly protects.
- Any fix touches shared requester-origin construction used by both native `sessions_spawn` and ACP spawning, so regressions should cover same-agent, cross-agent, thread-bound, and multi-account channel cases.
