## Summary

Keep this PR open for maintainer protocol review. The Paperclip rejection is still real on current main, but this branch exposes a vendor-named root `agent` parameter in the public Gateway protocol and overlaps an existing open PR for the same compatibility break.

## What This Changes

The PR adds an optional opaque `paperclip` field to Gateway `AgentParamsSchema`, regenerates Swift `AgentParams` models, and adds validator tests for Paperclip metadata while retaining strict rejection of unrelated unknown fields.

## Best Possible Solution

Land one additive, documented, generic opaque adapter metadata contract for Gateway `agent` params, regenerate affected client models, preserve strict rejection for unrelated root fields, keep Paperclip regression coverage, add release-note coverage, and then consolidate the related Paperclip reports.

## Reproduction Assessment

Yes. A Gateway `agent` request with valid `message` and `idempotencyKey` plus a root `paperclip` field has a high-confidence rejection path on current main because the schema is closed, AJV preserves unknown fields, and `handleAgent` validates before runtime dispatch.

## Solution Assessment

No. The PR likely fixes the immediate Paperclip payload, but a vendor-named root field is not the narrowest maintainable core protocol shape without explicit maintainer approval; a generic opaque adapter metadata field is safer.
