## Summary

Keep this PR open for maintainer review. The reported Gateway rejection is real on current main, but the patch makes a Paperclip-specific root field part of the public Gateway agent protocol; the safer direction is a generic opaque adapter metadata seam or an explicit maintainer decision to accept this compatibility shim.

## What This Changes

The PR adds an optional opaque root-level `paperclip` field to `AgentParamsSchema` and a new validator test covering Paperclip metadata acceptance plus continued rejection of unrelated unknown agent params.

## Best Possible Solution

Implement one additive, generic Gateway agent metadata field such as a documented `adapterMeta` or `pluginParams` record, keep strict rejection for unrelated root fields, add regression coverage for Paperclip-style metadata, and then consolidate the duplicate reports/PRs after the accepted fix merges.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction: an `agent` request with `message`, `idempotencyKey`, and a root `paperclip` field fails `validateAgentParams` because the schema is closed, and #74635 supplies the live Paperclip gateway steps and error text.

## Solution Assessment

No. The patch solves the immediate payload shape, but the best maintainable protocol fix is a generic namespaced opaque metadata field or an explicit maintainer-approved exception, not a vendor-named root property in core Gateway protocol.
