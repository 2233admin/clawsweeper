## Summary

Keep this PR open for maintainer review. The underlying Paperclip compatibility bug is real on current main, but this branch changes a public Gateway protocol field shape and overlaps the still-open #69139 implementation path, so it needs API-owner consolidation rather than cleanup closure.

## What This Changes

This PR adds optional opaque root-level `paperclip` metadata to Gateway `AgentParamsSchema`, mirrors it into generated Swift Gateway models, and adds validator tests for Paperclip metadata while preserving rejection of unrelated unknown root fields.

## Best Possible Solution

Land one additive, documented, generic opaque Gateway agent metadata field such as `adapterMeta` or `pluginParams`, regenerate affected client models, keep strict rejection for unrelated root fields, add Paperclip regression coverage against the accepted contract, and close the related Paperclip reports after that fix merges.

## Reproduction Assessment

Yes. A Gateway `agent` request with `message`, `idempotencyKey`, and root `paperclip` has a high-confidence rejection path on current main because the schema is closed, AJV preserves unknown fields, and dispatch validates before runtime handling; #74635 provides the matching live error string.

## Solution Assessment

No. The PR unblocks the immediate payload shape, but a vendor-named root field is not the narrowest maintainable core protocol shape unless a maintainer explicitly approves that exception; a generic opaque adapter metadata field is safer.
