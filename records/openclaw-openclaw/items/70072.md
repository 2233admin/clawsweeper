## Summary

Keep open. Current main still exposes deep-dreaming promotion only as aggregate counts plus logger-only verbose details, and it has adjacent single-candidate `promote-explain` tooling but no per-run rejected-candidate report, structured verbose artifact, or aggregate dreaming doctor diagnostic. The request remains distinct from the closed calibration issue it references.

## What This Changes

The issue requests deep-dreaming introspection: per-candidate ranking and rejection reasons in reports, optional structured verbose scoring output, and an aggregate `openclaw memory dreaming doctor` diagnostic.

## Best Possible Solution

Add a shared memory-core deep-candidate evaluation API that can return bounded considered candidates with score components, threshold pass/fail state, promotion/apply status, and rejection reasons. Use that surface for the scheduled deep report, optional structured verbose output, and an aggregate CLI diagnostic, with docs and focused tests covering both passing and rejected candidates.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is an additive diagnostics request. Current-main inspection gives a high-confidence verification path: the report writer only receives aggregate lines, verbose scoring goes to the logger, and no aggregate dreaming doctor command is registered.

## Solution Assessment

Yes. A shared evaluation surface is the best maintainable direction because the report, JSONL trace, and doctor command need the same scoring and gate-reason data; duplicating threshold logic in each caller would risk drift from actual promotion behavior.
