## Summary

Keep open: current main still does not allow host-local HTML/XML/CSS documents through the validated-text host-read path, and the latest PR is close but still leaves a security-sensitive XML sniff path outside full text validation.

## What This Changes

The PR expands the shared media host-read guard and regression tests so host-local HTML, XML, and CSS attachments can be sent when they pass validated-text checks, while keeping plain text and renamed binary payloads blocked.

## Best Possible Solution

Keep the validated-text approach, but remove XML/text formats from the buffer-verified document allowlist and ensure XML sniffs are accepted only through the full-buffer text-validation branch. Add a regression for extensionless or .txt XML-prologue content with a binary tail, then rerun the targeted media/outbound tests and the changed gate before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep XML out of the buffer-verified allowlist:** `src/media/web-media.ts:107`
  - body: Adding `application/xml` to `HOST_READ_ALLOWED_DOCUMENT_MIMES` lets host reads that do not enter the declared text-alias branch return before `isValidatedHostReadText` runs. `file-type@22.0.1` sniffs `application/xml` from a `<?xml ` prefix, so an extensionless or `.txt` file with an XML prologue and binary tail can bypass the full-buffer text validation this PR is trying to require.
  - confidence: 0.88
