## Summary

Keep open: current main still does not implement the requested host-read HTML/XML/CSS behavior, and the latest PR head has a blocking XML validation regression plus a host-read XML validation concern that should be fixed before merge.

## What This Changes

The PR extends the host-read media guard to allow validated HTML, XML, and CSS document attachments, adds shared media and outbound attachment regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the change focused on a validated-text path for HTML/XML/CSS/XML: permit absent sniff or an XML/text sniff only after full-buffer isValidatedHostReadText, avoid treating application/xml as a buffer-verified document type, keep .txt blocked, and retain regression coverage for real files plus renamed ZIP and opaque binary payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Allow XML sniffs through the text validator:** `src/media/web-media.ts:121`
  - body: Adding text/xml to the declared text-alias set routes .xml files into this branch, but file-type@22.0.1 sniffs an XML prologue as application/xml. Because the branch only accepts when sniffedContentType is absent, the new XML acceptance case throws before it can pass validation.
  - confidence: 0.96
- **[P2] Keep XML out of the buffer-verified allowlist:** `src/media/web-media.ts:106`
  - body: application/xml is now accepted by the sniffed-document allowlist before isValidatedHostReadText runs for extensionless or non-alias host reads. file-type recognizes XML from a '<?xml ' prefix, so a file with that prefix and an opaque binary tail can bypass the full text validation this PR is trying to require.
  - confidence: 0.82
