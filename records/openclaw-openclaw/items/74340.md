## Summary

Keep open. The PR fixes one direct successful-read path, but it is still incomplete because a successful frozen-result refresh can call completion capture without an outcome and still fall through to raw tool text. The linked #74073 production repro and overlapping #73783/#74113 work point to the same subagent announce/capture root area, so this should be resolved as one canonical fix rather than merged as-is.

## What This Changes

The PR adds an ok-status guard before the subagent output selector falls back to raw text and adds a regression test for successful tool-only histories.

## Best Possible Solution

Converge on one subagent announce/capture fix that prevents raw tool results from becoming successful completion or frozen-result text across direct announce, initial freeze, and refresh paths, while preserving timeout/error partial-progress behavior and aligning docs/tests with the final contract.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a successful tool-only history: the PR covers direct `readSubagentOutput(..., { status: "ok" })`, while the remaining gap is the `refreshFrozenResultFromSession` path that captures without an outcome and can still select `snapshot.latestRawText`.

## Solution Assessment

No. The PR is a useful partial fix, but the `outcome?.status === "ok"` guard is too narrow because successful refresh/capture calls can still arrive without that outcome; a safer fix should either carry/derive the outcome there or suppress raw tool-result fallback for success-like completion capture more centrally.
