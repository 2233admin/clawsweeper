## Summary

Keep this PR open for author or maintainer revision. The branch fixes the direct `readSubagentOutput(..., { status: "ok" })` path, but current code still has an outcome-less frozen-result refresh path that can call the same selector without the new guard and write raw tool output into `frozenResultText`; related open work also covers the same compaction/raw-output bug family.

## What This Changes

The PR adds an `ok`-status guard in `selectSubagentOutputText` so successful tool-only histories do not fall back to raw tool result text, and adds a regression test for that direct read path.

## Best Possible Solution

Revise or supersede this branch with one consolidated subagent announce fix that prevents raw tool/toolResult fallback for successful completion captures across direct announce, initial freeze, and refresh paths, while preserving timeout/error partial-progress behavior and aligning docs/tests with the chosen contract.

## Reproduction Assessment

Yes. A source-level reproduction is a successful tool-only history: the PR's new test covers the direct `readSubagentOutput(..., { status: "ok" })` path, while the remaining failing path is the outcome-less `refreshFrozenResultFromSession` capture that can still select `snapshot.latestRawText`.

## Solution Assessment

No. The PR is a useful partial fix, but an `outcome?.status === "ok"` guard is not the best complete solution because successful capture/refresh calls can still arrive without that outcome; the safer fix should cover the capture helper or selector contract used by frozen-result refreshes too.
