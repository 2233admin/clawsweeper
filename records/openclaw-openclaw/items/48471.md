## Summary

Current main still lacks the requested first-class one-line local bootstrap for daemon install/refresh, dashboard auth, Telegram bot setup, and non-interactive Telegram owner access. Some pieces exist, including daemon-aware non-interactive onboarding, dashboard token URLs, interactive Telegram allowlist setup, and a Telegram username resolver, but they are not composed into one supported bootstrap surface.

## Best Possible Solution

Keep this open as a first-run CLI/product gap. The best implementation is a first-class local bootstrap surface, either by extending `openclaw onboard --non-interactive` or adding a dedicated bootstrap command that composes daemon install/refresh, dashboard URL output with auth bootstrap, Telegram token setup, Telegram username-to-ID resolution, `dmPolicy: "allowlist"`, numeric owner `allowFrom`, and `commands.ownerAllowFrom`, with public docs and automated tests. If unattended DM pairing remains unsupported, document that explicitly and route automation through allowlist/preapproval instead of bypassing the trust model.
