Current main and the latest release still contain the unguarded gateway channel logger dereferences, while this PR addresses the crash path with defensive logger guards, deferred-plugin logger/runtime backfill, regression coverage, and no blocking review finding found in the read-only sweep.

## What This Changes

The branch guards async gateway channel lifecycle logger calls, backfills channel logger/runtime maps after deferred plugin reload, marks secrets-state system events untrusted, and adds regression coverage plus a changelog entry.

## Best Possible Solution

Land this PR or an equivalent patch after normal exact-head validation, preserving both the defensive async logger guards and the deferred plugin logger/runtime map backfill so late-loaded channel plugins start with consistent lifecycle dependencies.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a deferred channel plugin that appears in the active registry while `channelLogs[channelId]` is missing, then exits or rejects during `startChannelInternal`, causing current main to dereference `log.error?.(...)` with `log` undefined.

## Solution Assessment

Yes. The core fix is the narrow maintainable path: guard all async logger continuations that can run after map state changes, and synchronize logger/runtime maps after deferred plugin reload rather than only hiding the crash.
