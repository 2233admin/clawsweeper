## Summary

Keep this PR open. Current main still passes channel message bodies and thread-context bodies into reply prompt assembly after only tag/newline sanitization, so the security hardening is not obsolete. The submitted PR is directionally useful, but its current trust-classification approach still needs maintainer review because it appears to use reply-routing metadata and an incomplete trusted-provider allowlist for internal heartbeat/cron-event prompts.

## Best Possible Solution

Keep this PR open and request a focused revision: derive wrapping from the actual ingress/source provider rather than reply-routing OriginatingChannel, treat heartbeat and cron-event as trusted internal providers alongside exec-event/node/cli, preserve the raw queue summary and thread-context wrapping improvements, and add regression tests for channel bodies, thread context, routed heartbeat delivery, and internal-only scheduler prompts.
