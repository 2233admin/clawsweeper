## Summary

Keep open. Current main still activates the top-level managed proxy through `global-agent`, and the Discord explicit proxy path remains only a workaround rather than a fix for `proxy.enabled=true` changing the TLS/SNI behavior observed by the reporter.

## What This Changes

Make top-level `proxy.enabled=true` route Discord gateway/API and related channel HTTP traffic through a local HTTP forward proxy without changing TLS verification to the proxy hostname.

## Best Possible Solution

Keep the top-level managed proxy contract, but replace or constrain the unsafe Node core proxy hook so HTTPS and WSS use HTTP CONNECT while preserving the original upstream TLS server name. Add regression coverage with a local CONNECT proxy and HTTPS/WSS targets, then update the network proxy docs only if the supported surface intentionally narrows.

## Reproduction Assessment

Yes. The report includes a concrete OpenClaw config and a minimal `global-agent` reproduction with the same certificate hostname mismatch, and current main still runs that `global-agent` path for top-level proxy startup.

## Solution Assessment

No patch is proposed here, and the current per-channel/per-provider workaround is not the best repo-level solution. The maintainable fix should preserve central egress routing or explicitly narrow the documented contract with maintainer approval.
