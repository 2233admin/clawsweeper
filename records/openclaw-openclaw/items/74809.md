## Summary

Keep open. Current main still implements and documents the top-level managed proxy by forcing global-agent for Node core HTTP/HTTPS, and Discord only uses its explicit HttpsProxyAgent path when a Discord-specific proxy is configured. The related Discord proxy work explains the reporter's workaround, but it does not resolve the reported top-level proxy TLS path.

## What This Changes

The issue reports that top-level `proxy.enabled=true` with a local HTTP forward proxy breaks Discord TLS/SNI, while owner-specific explicit proxy settings work.

## Best Possible Solution

Keep the issue open and repair the managed proxy contract by replacing or constraining the unsafe global-agent path with CONNECT-based routing that preserves upstream TLS server identity. Add regression coverage with a local HTTP CONNECT proxy and HTTPS/WSS targets, then update `docs/security/network-proxy.md` if the supported surface changes.

## Reproduction Assessment

Yes. The issue includes a minimal global-agent reproduction, and current main still activates the same global-agent path for `proxy.enabled=true`; I did not run a live Discord/Surge test in this read-only review.

## Solution Assessment

Yes, keeping this open for maintainer-directed repair is the best current path. The existing Discord-specific proxy workaround is useful but narrower than the documented top-level proxy contract, and an automated fix should wait until maintainers choose how to preserve managed egress coverage.
