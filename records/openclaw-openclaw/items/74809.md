## Summary

Keep open. Current main still implements the reported top-level managed proxy path by forcing `global-agent` for Node `http`/`https`, and the docs still promise top-level HTTP, HTTPS, and WebSocket routing; the Discord-specific explicit proxy path matches the reporter's workaround but does not repair the process-wide `proxy.enabled=true` TLS failure path.

## What This Changes

The issue reports that top-level `proxy.enabled=true` with a local HTTP proxy causes Discord TLS hostname verification failures through the process-wide `global-agent` route, while owner-specific explicit proxy settings work.

## Best Possible Solution

Keep the top-level managed proxy contract, but replace or constrain the unsafe Node-core proxy hook so HTTPS and WSS traffic use HTTP CONNECT while preserving TLS verification against the original destination. Add regression coverage with a local CONNECT proxy and real HTTPS/WSS targets, then update the network proxy docs if the supported surface changes.

## Reproduction Assessment

Yes. The issue includes concrete config, logs, and a minimal `global-agent` repro, and current main still starts the same forced global-agent path; a live Surge/Discord run was not performed in this read-only review.

Unclear. The requested outcome is correct, but the exact implementation should be chosen by maintainers because the fix touches the security-sensitive process-wide managed proxy guardrail and must not reduce egress coverage.
