## Summary

Keep open. Current main still routes the top-level managed proxy through forced `global-agent` Node `http`/`https` hooks, while the shipped Discord-specific proxy path only applies when a Discord proxy is configured. The report has a concrete reproduction and the existing tests/docs still preserve the unsafe top-level contract without proving TLS hostname/SNI correctness through a local HTTP CONNECT proxy.

## What This Changes

The issue requests that top-level `proxy.enabled` route Discord gateway/API traffic through a local HTTP forward proxy without changing TLS hostname verification to the proxy host.

## Best Possible Solution

Keep the top-level managed proxy contract, but replace or constrain the unsafe `global-agent` path so Node core HTTPS and WebSocket traffic use an HTTP CONNECT proxy while preserving upstream TLS server identity. Add a regression test with a local CONNECT proxy and HTTPS/WSS target that verifies the proxy sees the upstream host and certificate validation remains tied to the destination, then update the public network proxy docs if supported coverage changes.

## Reproduction Assessment

Yes. The issue includes a minimal `global-agent` reproduction using the same env variables that current main still injects, and current source still calls `bootstrapGlobalAgent()` for top-level proxy activation.

## Solution Assessment

No current fix is present. The Discord and Telegram owner-specific proxy paths are valid workarounds, but they do not solve the documented `proxy.enabled=true` top-level proxy behavior.
