## Summary

Keep open. Current main still has reset-time preservation mitigations, but it does not run an agentic memory-flush turn against the outgoing session before manual `/new` or `/reset`, daily rollover, or idle rollover replaces and archives that session.

## What This Changes

Request to run a bounded agentic memory-flush turn before manual and automatic session resets destroy the outgoing session.

## Best Possible Solution

Keep this as the canonical maintainer design thread for reset-time memory persistence. The end state should be a bounded shared reset-flush path that can run the existing memory capability against `previousSessionEntry` before teardown when policy allows, while keeping `session-memory` as a raw audit fallback and documenting manual, daily, and idle behavior consistently.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: follow bare `/new` or `/reset` through `initSessionState` and `maybeHandleResetCommand`, or stale daily/idle rollover through `evaluateSessionFreshness`, and the outgoing session is replaced or archived without an agentic memory-flush call.

## Solution Assessment

Unclear because this is a feature/design issue rather than a patch. The requested direction is plausible, but the best implementation needs maintainer decisions on blocking versus background execution, timeout policy, config shape, and whether daily/idle resets should run the same flush by default.
