## Summary

Keep open. The requested Telegram `allowH2: false` and per-resolver dispatcher isolation are already present in current main and in `v2026.4.29`, but the fresh report says the observable Windows multi-bot stall still reproduces on that release, so the remaining problem is not safely closeable as implemented.

## What This Changes

The issue requests fixing Windows Telegram long-poll stalls and web_fetch timeouts by applying the existing HTTP/2-disabling dispatcher behavior to Telegram polling and considering isolated per-bot undici pools.

## Best Possible Solution

Keep this issue open and reproduce on Windows with the reported multi-bot setup against `v2026.4.29` and current main, collecting Telegram network diagnostics around `getUpdates`, web_fetch timeouts, DNS family selection, proxy env, and account/token resolution; any fix should target the newly identified remaining path rather than reapplying the already-present HTTP/2 disable.

## Reproduction Assessment

No. The report gives concrete steps, but a high-confidence reproduction requires Windows, Node 24, five live Telegram bot accounts, and the reporter's network conditions; this review can only verify that the proposed dispatcher code is already present.

## Solution Assessment

No for the stated fix, unclear for the remaining bug. Applying `allowH2: false` is not a remaining solution because current main and `v2026.4.29` already do it; the maintainable next step is live diagnosis of the 4.29-specific stall path.
