## Summary

Keep this issue open. Current main has same-file edit serialization through the pinned Pi edit tool, so the related locking problem is partly addressed, but OpenClaw still has no distinct noop result for a request that finds the file already in the desired final state. The current wrapper either returns the underlying success, converts selected post-write failures back to success, or rethrows/annotates mismatch errors.

## Best Possible Solution

Implement a real post-state edit classifier: return normal success when this request writes, a distinct noop result when the desired final state is already satisfied after waiting/rechecking, and preserved errors for true mismatches. Add focused regression coverage for concurrent canonical edits[], already-up-to-date state, true failure, and non-identical concurrent edits; separately decide whether OpenClaw should accept flat oldText/newText compatibility input.
