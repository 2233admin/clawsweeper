## Summary

Keep #72262 open. Current main still lacks the requested WhatsApp `status=408 Request Time-out` runbook in both target docs files, and the latest issue comment explicitly acknowledges the earlier QEEK-AI claim was only a chat spec, not an integrated docs change. OpenClaw’s WhatsApp runtime/tests do preserve the exact `status=408` / `Request Time-out` signature, so the docs gap is real and actionable in this repo.

## Best Possible Solution

Add the requested docs on main: a dedicated `WhatsApp disconnected with status=408 Request Time-out` subsection in `docs/channels/whatsapp.md` and a concise `status=408 Request Time-out` row in `docs/channels/troubleshooting.md`. The runbook should explain the observable Baileys/WhatsApp Web socket failure, walk through `channels status --probe`, logs, doctor, and gateway status, distinguish transient reconnects from stale auth/network/proxy/runtime issues, define healthy WhatsApp credential state, and document a backup-aware `channels logout` then `channels login` recovery path.
