## Summary

Keep this issue open. Current main still has retention-based cron run pruning and one-shot `deleteAfterRun` cleanup paths, but it does not expose or implement the requested completion-based ephemeral mode for recurring cron jobs or hook mappings.

## Best Possible Solution

Add an explicit `ephemeral` or completion-cleanup option for isolated cron jobs and hook agent mappings, wire it through the cron/hook schemas, CLI/config normalizers, runtime dispatch, and docs, then delete only the intended run session after success or error using the existing session deletion/transcript archival path. Cover recurring cron jobs, hook mappings, failed runs, disabled delivery, and static versus templated session keys with focused tests.

## Evidence

## Risks / Open Questions

- An implementation needs to avoid deleting shared or static hook session keys that users intentionally reuse; cleanup should be explicit and scoped to the intended ephemeral run session.
- Transcript handling needs a clear policy for success, error, delivery-disabled, and templated `sessionKey` cases so session deletion does not silently discard useful debugging data.
