## Summary

Keep open. Current main still uses FIFO command-lane draining and has no priority API, so the feature is not already implemented, but this PR is not merge-ready because the starvation guard does not actually protect low-priority work from a steady stream of high-priority entries and the branch is dirty against current queue APIs.

## What This Changes

This PR adds command-queue priority levels, optional enqueue priority, priority-aware queue scanning with low-to-normal promotion, and process command-queue tests for the new scheduling behavior.

## Best Possible Solution

Rebase or replace the branch on current main, keep the existing command-queue timeout/snapshot/reset and lane APIs intact, and make the starvation guard eventually select aged low-priority work even when newer high-priority entries keep arriving.

## Reproduction Assessment

Yes. A high-confidence reproduction follows from the PR code: enqueue a low-priority entry, let it age past the threshold, then keep enqueueing high-priority entries; the aged low entry only becomes Normal, so `pickNextIndex` still selects High entries first.

## Solution Assessment

No. In-place priority scanning fits the existing lane queue shape, but the proposed low-to-normal promotion does not satisfy the stated starvation guarantee and the branch must be rebased onto the current `CommandQueueEnqueueOptions` and lane model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Guarantee starvation promotion over high-priority work:** `src/process/command-queue.ts:133-134`
  - body: After `STARVATION_PROMOTION_MS`, a low-priority entry is only promoted to `Normal`. A lane with a steady stream of `High` entries can still bypass that aged low-priority entry forever, which breaks the PR's starvation guarantee; promote aged work above high priority or use an age-based score and add a regression test with repeated high-priority arrivals.
  - confidence: 0.9

Rebase this on current main and add `priority` to `CommandQueueEnqueueOptions` while still storing `taskTimeoutMs`, otherwise stuck lane tasks lose their current timeout escape hatch.
  - confidence: 0.86
