## Summary

Keep open. Current main still drains the command queue FIFO and exposes no priority option, while the PR adds a focused priority scheduling API with tests and no definite code-review findings; the remaining decision is maintainer acceptance of the shared queue semantics plus normal CI gating.

## What This Changes

The PR adds CommandPriority and an optional enqueue priority, selects queued work by effective priority with a 30s starvation promotion, and adds command queue tests for ordering, default priority, onWait, and task timeout behavior.

## Best Possible Solution

Land the in-place priority scheduling after maintainers approve the generic queue API and CI gates are acceptable, keeping the behavior in the shared process queue rather than introducing special-case urgent lanes.

## Reproduction Assessment

Yes. Current main can be reproduced by enqueueing blocked work and a later urgent task in one lane: the lane drains with FIFO shift() and the options type has no priority field, so the urgent task cannot jump ahead.

## Solution Assessment

Yes. The PR’s scan-and-splice approach is the narrowest maintainable solution I found because it preserves enqueue timestamps, lane state, timeout accounting, and existing API defaults while adding priority selection.
