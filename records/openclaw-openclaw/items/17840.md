## Summary

Keep open. Current main still implements reaction notifications as queued system events, and the heartbeat wake primitive is available separately, but there is no documented opt-in config, per-message flag, or bundled channel reaction path that turns an accepted reaction into an agent wake/turn. The closed overlaps #56708 and #11747 do not supersede this broader cross-channel request, while #38714 and #73681 are narrower adjacent issues.

## What This Changes

The issue requests an opt-in cross-channel mechanism so accepted reaction events can wake the agent and trigger an agent turn instead of only being queued as passive system events.

## Best Possible Solution

Keep this issue open as the canonical cross-channel product/API request. A maintainer should choose the opt-in contract, such as a config-level `reactionTriggersTurn` mode or a narrower per-message `awaitReaction` flag, then wire accepted reaction events to `runtime.system.requestHeartbeatNow` only after existing channel access and reaction notification filters pass. The implementation should update config schemas/generated metadata, channel docs, and focused tests across the reaction-capable bundled plugins.

## Risks / Open Questions

- A broad reaction-triggered wake mode can create noisy or looping agent turns if it does not preserve the current passive default and strict opt-in behavior.
- Reaction wake behavior must run only after each channel's existing auth, allowlist, bot-authored-message, and notification filters pass; otherwise low-friction emoji input becomes a broader agent-triggering surface.
- A one-channel fix could fragment config semantics unless the maintainer first chooses a shared contract and updates schemas, generated metadata, docs, and tests consistently.
