## Summary

Keep this issue open. Current main still synthesizes `<media:image>` / `<media:${kind}>` for blank iMessage media sends and passes that value as the outbound imsg JSON-RPC `text` parameter. The linked fix PRs #40341 and #46343 are closed unmerged, and the latest release tag still contains the same behavior.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/imessage/src/send.ts`: stop synthesizing visible `<media:...>` fallback text for outbound media-only sends, preserve user captions and `replyToId`, keep the guard that rejects sends with neither text nor media, add a direct regression test that blank text plus media sends RPC `text: ""` with `file`, and update deliver/echo/self-chat tests so dedupe no longer depends on visible placeholder text.

## Risks / Open Questions

- No live iMessage device send was run in this read-only review; the source path deterministically passes the placeholder as RPC `text`, but the eventual fix should verify imsg accepts empty `text` when `file` is present.
- The fix should be scoped to outbound sends. Inbound iMessage attachment normalization still uses media placeholders as agent-facing context and echo/dedupe support, so removing placeholders wholesale would risk regressions.
