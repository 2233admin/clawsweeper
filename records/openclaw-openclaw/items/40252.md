## Summary

Keep this issue open. Current main still synthesizes `<media:image>` / `<media:${kind}>` for blank iMessage media sends, assigns that value to the outbound message, and sends it as the imsg JSON-RPC `text` field together with the attachment file. The related fix PRs #40341 and #46343 are closed unmerged, and the latest release tag still contains the same behavior.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/imessage/src/send.ts`: stop synthesizing visible `<media:...>` fallback text for outbound media-only sends, preserve user captions and `reply_to` metadata for text-plus-media sends, keep the media-or-text guard, add a direct regression test that blank text plus media sends RPC `text: ""` with `file`, and update deliver/echo/self-chat tests so dedupe no longer depends on visible placeholder text.

## Risks / Open Questions

- No live iMessage device send was run in this read-only review; the source path deterministically sends the placeholder as RPC text, but the eventual fix should verify imsg accepts an empty `text` field when `file` is present.
- The fix should be scoped to outbound sends. Inbound iMessage attachment normalization still uses media placeholders as agent-facing context and should not be removed wholesale.
- Removing the outbound placeholder will require updating echo/self-chat dedupe tests so they rely on message IDs or attachment-aware matching instead of visible placeholder text.
