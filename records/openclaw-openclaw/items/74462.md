## Summary

Keep open. Current main still has no durable `contextSafety` config contract, and the code paths that validate normal config reads remain strict enough to reject both `session.contextSafety` and a top-level `contextSafety`; however, the referenced runtime support is not present in the current source or release tag, so this needs maintainer product/architecture confirmation rather than a schema-only cleanup close.

## What This Changes

Request to add a supported durable `contextSafety` config path, matching TypeScript types, schema metadata, docs, and a decision on whether the top-level key should remain a legacy alias.

## Best Possible Solution

Decide the intended product contract first. If context safety is supported, implement it end to end with schema, types, generated metadata/help, docs, config validation tests, and runtime tests proving the durable config is consumed; if it is not supported, remove or document the stale/generated alias and point users at the existing session reset, maintenance, and compaction controls.

## Reproduction Assessment

Yes for the reported schema rejection: current main can be verified statically because the strict `SessionSchema` and strict root schema omit the requested keys and validation routes through `OpenClawSchema.safeParse`. The claimed runtime support is unclear because the named runtime symbol and slash commands are absent from current source and the latest release tag.

## Solution Assessment

Unclear/no for the suggested schema-only direction: it is not the best fix until the runtime contract is confirmed. The safer path is an end-to-end product decision that either adds a consumed durable config surface or removes the unsupported alias from generated/runtime artifacts.
