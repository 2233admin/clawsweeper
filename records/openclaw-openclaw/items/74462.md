## Summary

Keep open. Current main still rejects `session.contextSafety` and top-level `contextSafety`, and the public `SessionConfig` type omits that field; however, current source also has no `contextSafety` runtime symbol or the named rollover/handoff commands, while the supported context-pressure controls now live under `agents.defaults.compaction`. That makes this a real contract/product question rather than a safe schema-only cleanup.

## What This Changes

The issue requests a durable config schema, type, metadata, and docs path for `contextSafety` settings that the reporter says generated runtime artifacts already read.

## Best Possible Solution

Decide the intended public contract first. If context safety is supported, add it end to end: schema, `SessionConfig` type, generated schema/help/labels metadata, docs, validation tests, and runtime tests proving the durable config is consumed. If the intended surface is the existing `agents.defaults.compaction` controls instead, document or diagnose that path clearly and avoid accepting a misleading `contextSafety` alias.

## Reproduction Assessment

Yes: a static current-main reproduction is clear from source. The normal validator routes through strict `OpenClawSchema.safeParse`, `SessionSchema` and the root schema omit `contextSafety`, and exact source search finds no matching runtime consumer or slash commands.

## Solution Assessment

Unclear: the suggested schema/type addition is plausible only if maintainers confirm `contextSafety` is an intended runtime contract. The safer path is either an end-to-end implementation or a docs/diagnostic clarification that points users to the existing `agents.defaults.compaction` controls.
