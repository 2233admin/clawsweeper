## Summary

Keep open. Current main still rejects the requested durable config keys, and the current source plus latest release tag do not show the claimed `contextSafety` runtime or named rollover/handoff commands, so this needs maintainer product/architecture confirmation rather than closure or a schema-only cleanup.

## What This Changes

The issue requests a durable `contextSafety` config path, preferably under `session.contextSafety` with matching schema, TypeScript types, config metadata, docs, and possible top-level legacy alias support.

## Best Possible Solution

Decide the intended product contract first. If context safety is supported, implement it end to end with `session.contextSafety` schema, exported types, config labels/help/generated metadata, docs, validation tests, and runtime tests proving the durable config is consumed; then separately decide whether top-level `contextSafety` should migrate as a legacy alias. If it is not intended, document the supported compaction/session controls and remove or explain any generated-runtime artifact that suggests otherwise.

## Reproduction Assessment

Yes for the schema rejection: the strict session/root schemas and validator path provide a high-confidence source-level reproduction. The runtime-support half is unclear because current source and the latest release tag do not contain the reported `contextSafety` symbols or commands.

## Solution Assessment

Unclear. Accepting the key in schema alone is not the best fix unless maintainers confirm or add the runtime contract; the safer path is an end-to-end config/runtime/docs/test change or a documentation cleanup that points users to existing compaction controls.
