## Summary

Keep open for maintainer review. Current main still has the reported abort/timeout payload leak path, and this PR is aimed at the right payload assembly surface, but the current diff also suppresses all `stopReason: "aborted"` assistant output even though the existing external-abort contract says user-stopped runs should continue normally and let partial assistant output carry the turn.

## What This Changes

The PR updates embedded run payload assembly to treat aborted assistant turns as formatted error surfaces, suppress aborted answer/reasoning text, and add regression tests for aborted timeout leakage of partial text, reply tags, and reasoning.

## Best Possible Solution

Keep the timeout-leak fix at `buildEmbeddedRunPayloads`, but narrow the suppression condition so it applies to timeout/internal abort delivery rather than every `stopReason: "aborted"` turn. The builder likely needs explicit `timedOut`/`externalAbort` context or a narrower caller-provided suppression flag, plus a regression test that preserves user-stop partial delivery.

## Reproduction Assessment

Yes. For the original bug, call `buildEmbeddedRunPayloads` on current main with non-empty `assistantTexts`, `lastAssistant.stopReason: "aborted"`, and a timeout-shaped `errorMessage`; current main can emit the partial text. For the PR finding, use an external-abort-like aborted assistant with partial answer text and observe that the proposed builder suppresses it without knowing whether the abort was user-initiated.

## Solution Assessment

No as-is. The PR targets the right final payload assembly point for timeout leakage, but blanket `lastAssistantAborted` suppression is broader than the current external-abort contract; the safer fix is to distinguish timeout/internal aborts from user-stopped runs before dropping answer and reasoning payloads.
