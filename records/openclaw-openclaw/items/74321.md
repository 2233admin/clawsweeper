## Summary

Keep open. Current main still treats the reported permanent delivery failures as retryable in key paths: recovery only recognizes a narrow route/membership regex set, live send failures still call the generic queue failure path, and the existing Telegram chunking mitigation does not cover auth/not-found/provider-size failures.

## What This Changes

The issue asks OpenClaw to classify permanent outbound delivery errors such as Telegram message-too-long, auth, and not-found failures, stop retrying them, and move them to failed delivery handling while preserving transient retries.

## Best Possible Solution

Add a maintainable permanent-delivery-error classification path that covers high-confidence provider failures on both first live send and recovery replay, preferably with channel-owned classification if maintainers want to reverse provider-specific regex drift in core. Preserve bounded transient retry/backoff, keep Telegram chunking, avoid treating every 400 as permanent, and add regression coverage for message-too-long, auth, and not-found cases.

## Reproduction Assessment

Yes. A focused static/unit reproduction is clear: on current main, the classifier regex set cannot match `400: Bad Request: message is too long`, `401 Unauthorized`, or generic `404 Not Found`, and live send failures still enter failDelivery rather than permanent failed/ handling.

## Solution Assessment

Unclear until maintainer direction. The issue’s requested behavior is right, but the best implementation choice is still open between a fast contained pattern add plus first-failure classification and the broader channel-owned classifier seam raised in discussion.
