## Summary

Keep open. Current main still classifies only a small shared regex set as permanent, so the reported Telegram `message is too long`, generic unauthorized/auth, and generic not-found delivery failures can still fall through to retry/backoff; first live-send failures also still record non-abort errors with `failDelivery` before any permanent-error handling.

## What This Changes

The issue requests permanent delivery-error classification for provider failures such as Telegram message-too-long, auth, and not-found errors so doomed queue entries stop retrying and stale messages are not replayed later.

## Best Possible Solution

Add permanent-delivery-error classification that runs on both first live send failure and recovery replay, with Telegram-owned handling for message-too-long, payload-size, auth, and not-found cases where possible, while preserving transient retry/backoff for 429, 5xx, and network errors. Cover the classifier, queue cleanup, live-send failure path, and Telegram chunking fallback with focused regression tests.

## Reproduction Assessment

Yes. A queued delivery whose send function throws `Call to 'sendMessage' failed! (400: Bad Request: message is too long)` still misses `isPermanentDeliveryError` and goes through `failDelivery`; a live first send failure follows the same non-abort `failDelivery` path.

## Solution Assessment

Unclear until maintainers choose the boundary. Extending the shared pattern list is the narrowest immediate fix, but the cleaner long-term solution is an additive channel-owned outbound classifier seam because the current shared core list already contains provider-specific strings.
