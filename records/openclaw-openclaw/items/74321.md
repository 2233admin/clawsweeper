## Summary

Keep open. Current main has useful mitigations, including bounded recovery retries and Telegram text chunking, but it still does not fully implement the requested permanent-error handling: the live send failure path records non-abort errors with `failDelivery`, and the recovery classifier still misses the reported `message is too long`, `401 Unauthorized`, and generic `404 Not Found` classes. The related fix PR was closed unmerged, and the newer overlapping issue is broader rather than a shipped replacement.

## What This Changes

Classify high-confidence permanent outbound delivery failures so Telegram message-too-long, auth, and not-found errors are failed instead of retried, while preserving transient retry behavior and Telegram long-text splitting.

## Best Possible Solution

Add a backwards-compatible delivery-error classification path that applies to both live sends and recovery replay. Prefer an optional channel-owned classifier on the outbound adapter or SDK contract so Telegram, Matrix, and other plugins own provider-specific permanent strings, while core preserves generic bounded retry/backoff and moves classified permanent failures to `failed/` with targeted tests.

## Reproduction Assessment

Yes. A source-level reproduction is clear: mock a Telegram send failure with `Call to 'sendMessage' failed! (400: Bad Request: message is too long)` and current main will not match it as permanent; on first live failure it calls `failDelivery`, and during recovery it only dead-letters errors matched by the current regex list.

## Solution Assessment

Unclear. A small regex plus live-path classification patch would address the immediate bug, but the discussion and repository boundary rules point toward a channel-owned classifier seam as the more maintainable solution for provider-specific permanent errors.
