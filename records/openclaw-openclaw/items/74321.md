## Summary

Keep open. Current main bounds recovery and has Telegram text chunking, but it still does not classify the reported permanent delivery failures on live send or recovery, so message-too-long, auth, and generic not-found errors remain retry candidates until generic max-retry handling.

## What This Changes

The issue asks OpenClaw to classify permanent outbound delivery failures such as Telegram message-too-long, auth, and not-found errors, move them to failed instead of retrying, and split oversized Telegram text where possible.

## Best Possible Solution

Add permanent delivery-error classification that is used by both live send cleanup and recovery. Prefer a channel-owned classifier seam if maintainers accept the plugin/API change; otherwise implement a contained high-confidence pattern fix for the named permanent classes, preserve transient retry/backoff, keep Telegram chunking, and add regression coverage for message-too-long, auth, not-found, and ambiguous 400 cases.

## Reproduction Assessment

Yes. The static reproduction is high confidence: throw `Call to 'sendMessage' failed! (400: Bad Request: message is too long)` from live delivery or recovery, and current main routes it through failDelivery because the classifier has no matching permanent pattern.

## Solution Assessment

Unclear until maintainer direction. Classifying permanent errors and halting retries is the right product behavior, but the cleaner implementation likely needs a channel-owned classifier seam rather than continuing to accumulate Telegram and Matrix strings in shared core.
