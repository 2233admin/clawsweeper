## Summary

Keep open. Current main has important mitigations: outbound recovery is bounded, permanent errors can be moved to failed/ during recovery, and Telegram text is chunked. But the requested behavior is not fully implemented because live send failures are still recorded with failDelivery before classification, and the permanent-error patterns do not cover the reported message-too-long / 401 / 404 classes.

## What This Changes

Classify permanent outbound delivery failures, especially Telegram message-too-long/auth/not-found errors, so they halt instead of being retried and replayed later.

## Best Possible Solution

Keep this open and implement a narrow fix that classifies high-confidence permanent outbound delivery errors both on the first live send failure and during recovery. Preserve bounded retry/backoff for transient errors, keep Telegram chunking, add regression tests for message-too-long/auth/not-found cases, and avoid classifying all 400 responses as permanent.
