## Summary

Keep open: current main still mitigates replay, Gateway display, and chat.send attachment paths, but it does not add the requested storage-layer claim-check for Discord/channel image turns before session JSONL persistence. The issue discussion includes concrete live forensics, and #43982 is adjacent chat.history projection work rather than a persistence fix.

## What This Changes

Store inbound Discord/channel images in session transcripts as durable media references and metadata instead of inline base64 image content blocks.

## Best Possible Solution

Add a backward-compatible transcript claim-check for inbound/user image content: persist compact media reference and metadata instead of raw `image.data`, hydrate only bounded recent references during model-context construction, preserve old inline-base64 transcripts, and reuse the existing media store/media URI contract where possible.

## Reproduction Assessment

Yes. The discussion includes concrete Discord DM forensics from v2026.3.24, and current main has a traceable code path where media refs hydrate into base64 image blocks before `activeSession.prompt` while the central append guard does not rewrite those blocks before persistence.

## Solution Assessment

Yes. A central persistence-layer claim-check is the narrowest maintainable fix; replay pruning, display redaction, and Gateway-only offload paths reduce symptoms but do not remove duplicate inline base64 from stored transcripts.

## Repair Work Prompt

Implement a storage-layer fix for https://github.com/openclaw/openclaw/issues/1210. Current main still persists inbound/user image content blocks with inline base64 in session JSONL; replay pruning and chat.history redaction are not enough. Add a backward-compatible transcript claim-check so new inbound/user image content is persisted as compact media reference/metadata instead of raw `image.data`, while current-turn model calls can still receive bounded image data. Prefer a central session persistence or embedded-runner seam over Discord-only special cases; reuse existing media store / `media://inbound/<id>` / `MediaPath` contracts where possible.
