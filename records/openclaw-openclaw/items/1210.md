## Summary

Keep open. Current main has replay-time pruning, Gateway history display redaction, and WebChat attachment offload mitigations, but it still does not implement the requested storage-level claim-check for session JSONL: user image blocks can still be constructed as `{ type: "image", data }` and the guarded append path persists non-tool-result messages without replacing image data with durable media references.

## What This Changes

The issue requests replacing inline base64 Discord image blocks in session transcripts with compact media reference metadata and bounded hydration from disk.

## Best Possible Solution

Add a shared, backward-compatible transcript claim-check for inbound image content: persist compact media reference and metadata instead of raw `image.data`, hydrate only bounded recent references during model-context construction, keep legacy inline-base64 transcripts readable, and align Gateway history/export display with the compact representation.
