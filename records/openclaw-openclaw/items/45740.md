## Summary

Keep this issue open. Current main still has the reported gh-issues prompt-injection surface: raw issue bodies and review-comment JSON are interpolated into spawned sub-agent prompts, while sessions_spawn only offers sandbox selection plus config-level sub-agent tool policy, not a per-call tool allow/deny scope that this skill can set dynamically. The discussion and linked PRs are useful partial mitigation context, but they do not show the current main behavior is fixed.

## Best Possible Solution

Keep this issue open for maintainer/security-hardening judgment. The likely path is to decide a narrow first mitigation for gh-issues, such as escaping/wrapping issue bodies and review-comment text as untrusted data before child prompt construction, plus a broader runtime decision on whether sessions_spawn should support per-call tool profiles or a safer unattended-mode default. Review the linked open fork PR as a partial prompt-level mitigation, but do not treat it as resolving the runtime hardening gap unless maintainers explicitly accept that scope.
