## Summary

Keep this open. Current main still embeds untrusted GitHub issue bodies and review-comment data directly into gh-issues child-agent task prompts, while the available runtime controls are sandbox selection and config-level subagent tool policy rather than a first-class untrusted-data or per-spawn tool-scope boundary. The earlier target-repo PR was closed unmerged, and the fork PR discussed on the issue is only a partial prompt-warning layer.

## What This Changes

The issue requests hardening gh-issues automation so untrusted GitHub issue and review comment bodies are not injected directly into tool-enabled child-agent prompts.

## Best Possible Solution

Define a maintainer-approved mitigation boundary: escape and wrap gh-issues issue/comment content as untrusted data before child task construction or route it through a structured summary/data channel, then decide whether unattended gh-issues spawns should require sandboxed target agents or first-class per-spawn tool profiles. Preserve notification behavior and add regression coverage for hostile issue/comment content plus any sessions_spawn policy changes.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: the gh-issues skill template substitutes `{body}` and `{json_array_of_actionable_comments}` directly into child task strings. There is not yet a current-main PoC showing the model crosses a security boundary after receiving that content.

## Solution Assessment

Unclear because there is no merged patch. The prompt-warning/fence approaches discussed on the issue are useful first layers, but the stronger maintainable direction is structured untrusted-data handling plus an explicit maintainer decision on sandbox/tool-scope defaults for unattended subagents.
