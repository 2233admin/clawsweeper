## Summary

Keep open. Current main still has a default UTF-8 stream decode path that decodes each chunk independently, and the issue is paired with open PR #51303 using closing syntax, so this should remain open until that PR lands or is superseded.

## What This Changes

The issue reports U+FFFD replacement characters in longer Chinese WebChat group-chat responses, suspected to come from UTF-8 multibyte sequences being decoded across chunk boundaries incorrectly.

## Best Possible Solution

Resolve through PR #51303 or a narrow replacement that makes default UTF-8 output decoding boundary-safe, preserves the Windows legacy-codepage fallback, adds split-CJK coverage for the default decoder and child adapter wiring, and verifies any remaining JSONL/socket or truncation surface needed for the reported WebChat path.
