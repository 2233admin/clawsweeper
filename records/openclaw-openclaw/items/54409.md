## Summary

Current main still has the Feishu inbound path serializing same-chat normal messages before they reach the gateway reply pipeline, so `messages.queue.mode = "collect"` can miss the active-run window and run later messages individually. The related #67793 and #70133 reports are useful context, but this issue remains a focused, valid Feishu collect-mode bug rather than cleanup material.

## Best Possible Solution

Keep this issue open and fix the Feishu inbound message path so normal same-chat followup messages can reach `dispatchReplyFromConfig` while the first run is still active, allowing the existing gateway `collect` queue to batch them. Preserve Feishu dedupe, inbound debounce, first-message ordering, abort/control lanes, and `/btw` behavior, and add a focused Feishu regression test that simulates an active run followed by multiple same-chat messages under collect mode.

Current main serializes normal same-chat Feishu messages through `createSequentialQueue` before `handleFeishuMessage` reaches `dispatchReplyFromConfig`, so followup messages arrive after the first run is no longer active and gateway `messages.queue.mode = "collect"` treats them as separate `run-now` turns.
