## Summary

Keep open. Current main still lacks Telegram edited-message dispatch, so the PR is not obsolete, but the proposed debounce path is blocked by the update tracker lifecycle: the delayed flush calls the shared inbound path only after the same update key has already been completed and cached as seen, so live edited-message updates would be skipped.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-message debounce, exposes a test timing knob, adjusts the Telegram test harness idle callback, and adds focused edited-message tests.

## Best Possible Solution

Keep the PR open and revise the Telegram edited-message path so update dedupe is accepted while the grammY update is still active, or so the delayed flush uses a pre-accepted event that does not re-dedupe after the timer. Add a middleware-aware regression test that exercises the tracker begin/finish lifecycle around the debounce.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level: a live grammY update enters `bot.use`, the PR handler schedules a timer and returns, `finishUpdate` records the key as completed, and the timer later calls `shouldSkipUpdate` on that same key.

## Solution Assessment

No. The requested handler/debounce direction is the right narrow fix for #72873, but the current implementation should move or split the dedupe check so the delayed dispatch is not dropped as a duplicate.

The current PR schedules `edited_message` dispatch behind a debounce timer, then calls `handleInboundMessageLike`, whose first step is `shouldSkipUpdate`; in live grammY middleware the update has already finished and its key is in `recentUpdates`, so the timer flush is skipped. Add or adjust tests so at least one edited-message case exercises the bot middleware/update-tracker begin/finish lifecycle rather than only calling the handler directly. Do not add `edited_channel_post`, transcript replacement, config migrations, dependency changes, or core Telegram-specific policy. Run targeted Telegram tests and the changed gate in the normal OpenClaw validation lane; add a changelog only if maintainers expect user-facing PR changelog entries for this fix branch.
