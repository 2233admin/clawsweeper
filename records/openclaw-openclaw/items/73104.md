## Summary

Keep open. The PR addresses a real Telegram bug, but the current head still routes the delayed edited-message flush through the original update context after the production middleware has already marked that update as completed, so the final debounced dispatch can be skipped; it also still needs the required user-facing changelog entry.

## What This Changes

The PR adds a Telegram `edited_message` handler with per-message debounce, test timing support, focused edited-message tests, and a test-harness idle callback cleanup.

## Best Possible Solution

Keep the PR open and repair the debounce/update-tracker interaction so the update is accepted by middleware before the handler returns, while the delayed final message dispatch does not re-check the same completed update key. Add a middleware-path regression test and an Unreleased Telegram changelog entry, then land this PR or an equivalent narrow replacement for #72873.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static: in the real bot pipeline the update-tracker middleware finalizes the update when the handler returns, then the debounce timer calls the shared inbound path with the same context and the recent update cache can skip it.

## Solution Assessment

No. The debounce handler is the right direction, but reusing the original `ctxForDedupe` for delayed dispatch is not the narrowest maintainable fix because it collides with the existing update tracker; separate update-level dedupe from the delayed final message dispatch.
