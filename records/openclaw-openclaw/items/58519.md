## Summary

Keep this open. Current main has useful Slack Socket Mode hardening, including a 15s default pong timeout and configurable Socket Mode ping/pong options, but it still runs Slack Socket Mode through Bolt's receiver in the gateway process and has no worker-thread transport isolation. The latest issue comment reports the same production flap on v2026.4.26, so the central starvation/message-loss problem is not proven fixed.

## What This Changes

Prevent Slack Socket Mode keepalive from being starved by heavy gateway/agent turns so pong timeouts, reconnect flapping, and silent Slack message loss do not occur.

## Best Possible Solution

Keep the issue open for maintainer design on durable Slack transport isolation or gateway event-loop starvation mitigation. Preserve the current timeout tuning, but decide whether Slack Socket Mode should run in an isolated worker/process, whether long synchronous agent/context paths need cooperative yields, and how delivery/ack paths should avoid logging success when transport state is stale.
