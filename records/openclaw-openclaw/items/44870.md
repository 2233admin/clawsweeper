## Summary

Keep open. Current main still has the reported custom-provider onboarding gap: custom OpenAI-compatible setup verifies and persists Chat Completions only, while OpenAI Responses support exists in config/runtime and docs only through manual configuration or Azure-specific custom onboarding paths.

## What This Changes

The issue reports that custom OpenAI-compatible model verification returns HTTP 503 for an endpoint whose model list works, and requests parity with the successful Anthropic-compatible custom setup path.

## Best Possible Solution

Add first-class custom-provider onboarding support for OpenAI Responses endpoints: an interactive selection or detection path, non-interactive flag parsing, verification against `/responses`, persistence as `models.providers.<id>.api: "openai-responses"`, and aligned docs/tests while preserving the current Chat Completions default.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: choose custom OpenAI-compatible onboarding and OpenClaw builds a `/chat/completions` probe, then persists `openai-completions`; there is no custom Responses option to select or parse.

## Solution Assessment

Yes. A narrow onboarding/config/docs/test change is the best path because the runtime and config schema already support `openai-responses`; a broader provider runtime rewrite is unnecessary for the reported failure.
