## Summary

Keep open. Current main still validates non-Azure custom OpenAI-compatible endpoints with Chat Completions and writes `api: "openai-completions"`, while the reporter's follow-up points to a backend that appears to require OpenAI Responses-style requests. The runtime/config layer can already represent `openai-responses`, so this looks like a narrow setup/probe support gap rather than a stale or out-of-repo report.

## What This Changes

The issue requests that custom OpenAI-compatible model setup validate and configure a Responses-capable backend instead of failing with HTTP 503 when the backend does not support the Chat Completions probe.

## Best Possible Solution

Add first-class custom-provider support for OpenAI Responses endpoints: expose a Responses compatibility choice, probe `POST /responses`, store `api: "openai-responses"` for that choice, and keep the existing Chat Completions path as the default for generic OpenAI-compatible backends.

## Repair Work Prompt

Fix the custom provider setup gap reported here. Current behavior: when a user configures a custom OpenAI-compatible endpoint such as `https://example.com/v1` with model `gpt-5.4`, OpenClaw verifies non-Azure OpenAI-compatible endpoints via `POST /chat/completions` and stores `api: "openai-completions"`, so Responses-only proxies can fail with HTTP 503 even though `/v1/models` works. Add first-class custom OpenAI Responses support without changing the existing Chat Completions default. Expected boundary: update the interactive custom provider wizard and non-interactive custom flags if needed to allow a Responses compatibility choice, add a `POST /responses` verification probe with a Responses payload, store `api: "openai-responses"` for that choice, and keep Azure OpenAI behavior and Anthropic behavior intact. Update targeted tests in the custom onboarding/config area, add docs for when to choose Responses vs Chat Completions, and add a user-facing changelog fix thanking @86jkuncle. Do not change bundled provider defaults, model catalogs, or runtime transport semantics beyond what is needed to make custom setup select the existing Responses API path.
