## Summary

Keep open. Current main has channel health checks, session listing, manual reset/delete, run timeouts, and session-store maintenance, but I did not find a session-level health monitor that detects repeated inbound messages without successful replies and automatically resets only the affected stuck session. The issue discussion also includes a corroborating operator report, so this is not just a stale title-search match.

## What This Changes

The issue requests a gateway session health check that logs and automatically clears or resets a channel session after prolonged unresponsiveness despite incoming messages.

## Best Possible Solution

Add an explicit session-health/recovery design that records last successful visible reply and consecutive failed or silent inbound turns per session, then performs a conservative configurable recovery action for only that session after proving there is no active run. The implementation should reuse the existing reset primitive, preserve channel routing metadata, emit clear logs/diagnostics, and coordinate with the narrower stale-running recovery tracked in #59878.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No live reproduction was run in this read-only review, and the report does not include the exact corrupted sessions.json, logs, or config from the incident. Static inspection does confirm the missing auto-recovery mechanism, and a focused regression test could seed a stale session row plus repeated failed dispatches.

## Solution Assessment

Is this the best way to solve the issue? Unclear as written: automatically clearing sessions is directionally useful, but the safest maintainable solution needs maintainer-approved semantics for reset versus prune versus notify and a false-positive guard for legitimate long-running sessions.
