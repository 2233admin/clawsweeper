## Summary

Keep open. Current main still exposes shared poll fields on the LLM-facing `message` tool and still rejects `action="send"` when non-empty or true poll-looking fields are present, before Feishu/Telegram send handling can run. The zero-default mitigation is present, but it does not cover the reported model auto-fill case. Open PR #46455 remains the direct implementation candidate and is still unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and review, land, or replace PR #46455. The durable fix should sanitize accidental poll creation params before the send guard or narrow the LLM-facing schema so poll creation fields are only visible for poll-capable actions/channels, while preserving explicit `action="poll"` validation and adding regression coverage for Feishu and Telegram send-with-attachment calls with non-empty auto-populated poll fields.
