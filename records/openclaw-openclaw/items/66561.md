## Summary

Keep this issue open. Current main has adjacent Codex transport normalization and model-call diagnostics, but the embedded runner still treats an enclosing attempt timeout as a generic timedOut/timeout condition without carrying request-started or response-started state into failover classification, observation logs, or the timeout payload path. The linked workaround/fix context does not close the root report because #41282 covers Codex base URL/SSE routing and #66599 was a closed, unmerged narrow AbortError proposal.

## What This Changes

The issue reports that OpenAI Codex SSE responses can begin receiving upstream bytes, then be locally aborted by the embedded run and surfaced as a generic timeout/408 failover.

## Best Possible Solution

Track provider-call lifecycle separately from the whole-attempt timer and propagate request-started/response-started or first-stream-chunk state into `EmbeddedRunAttemptResult`, failover decisions, observation logging, and user-facing error synthesis so post-start local aborts are not reported as plain no-response timeout/408 failures.
