## Summary

Keep this PR open. Current main still sends top-level metadata for `openai-codex-responses`, and the PR's builder-only strip is not enough because the live send path re-merges `turnState.metadata` immediately before `client.responses.create(...)`. The PR is aimed at a real, active Codex OAuth product path, but it needs an amendment at the final payload boundary or the provider turn-state policy.

## Best Possible Solution

Keep the PR open and ask for an amendment: ensure the final payload sent to `client.responses.create(...)` cannot contain top-level metadata for `openai-codex-responses`, while preserving metadata for regular `openai-responses` and `azure-openai-responses`. The clean fix likely belongs at the final `mergeTransportMetadata` call site or in OpenAI provider turn-state/payload policy, with a regression test that exercises the final send payload rather than only the builder output.
