## Summary

Keep open. Current main still routes Telegram delete-message API errors through the shared logging/rethrow path, while the action runtime reports every successful call as `deleted: true`; the related fix PRs are closed unmerged and a current GitHub search found no open PR referencing this issue.

## What This Changes

Make Telegram message deletion fail soft for benign Telegram delete no-op 400s and surface a structured warning instead of treating them as hard failures.

## Best Possible Solution

Patch the Telegram plugin only: classify exact benign delete no-op errors, use the existing `shouldLog` seam to avoid ERROR-level noise for those cases, return a structured warning from `deleteMessageTelegram`, surface `deleted: false` through the action runtime, and add focused regression tests plus a user-facing changelog entry credited to the reporter.

## Reproduction Assessment

Yes. The reported `openclaw message delete --channel telegram ...` path maps to the documented Telegram delete action, and the behavior can be reproduced without live Telegram credentials by injecting a Telegram API whose `deleteMessage` rejects with one of the reported benign 400 messages.

## Solution Assessment

Yes. A narrow plugin-local fail-soft path is the best way to solve this; the safer alternative to broad 400 handling is exact delete-error classification with tests that non-benign errors still rethrow.

## Repair Work Prompt

Fix the Telegram plugin bug from https://github.com/openclaw/openclaw/issues/73726. Current main still hard-fails benign Telegram `deleteMessage` 400s through `deleteMessageTelegram`. Keep the fix plugin-local: add an exact helper/classifier for benign delete no-op errors such as `message to delete not found`, `message can't be deleted` / `message cant be deleted`, `MESSAGE_ID_INVALID`, and only delete-specific forbidden strings after confirming they are safe; do not soften broad Telegram 400s, auth failures, missing credentials, rate limits, or unrelated permission errors. Update `extensions/telegram/src/send.ts` so `deleteMessageTelegram` returns `{ ok: true } | { ok: false; warning: string }`, suppresses shared ERROR logging for benign delete no-ops via the existing `requestWithDiag(..., { shouldLog })` seam, logs at verbose level if appropriate, and rethrows non-benign errors. Update `extensions/telegram/src/action-runtime.ts` so the `deleteMessage` action returns `{ ok: true, deleted: true }` on actual delete and a warning result with `deleted: false` on soft delete. Add focused tests in the Telegram test files for all benign strings, non-benign rethrow, API-error log suppression, and action-runtime warning propagation. Add a single-line user-facing `CHANGELOG.md` fix entry credited to `@Avicennasis`. Do not move behavior into core and do not change unrelated Telegram send/reaction behavior.
