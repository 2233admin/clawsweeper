## Summary

Keep open. Current main still lets Telegram delete-message API failures pass through the diagnostic request wrapper without delete-specific fail-soft handling, and the related fix PRs for this report are closed unmerged rather than superseding the issue.

## What This Changes

Telegram delete-message actions should treat benign already-gone or non-deletable message responses as non-fatal and surface a warning instead of logging and propagating an error.

## Best Possible Solution

Patch this inside the Telegram plugin: add a narrow benign-delete classifier, pass a `shouldLog` predicate so those known delete outcomes do not hit the API error logger, return a warning result from `deleteMessageTelegram`, surface `deleted: false` through the action runtime, and add focused tests for benign and non-benign delete failures.

## Reproduction Assessment

Yes. A stale or non-deletable Telegram `messageId` makes `api.deleteMessage` reject; current main routes that rejection through `requestWithDiag`, logs it through the API wrapper, and rethrows because there is no delete-specific catch.

## Solution Assessment

Yes. A narrow Telegram-plugin fix is the maintainable boundary; it should mirror the existing reaction fail-soft shape while also using the existing `shouldLog` hook to suppress ERROR logging before the warning result is returned.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/73726 in the Telegram plugin only. In `extensions/telegram/src/send.ts`, add a narrow helper/classifier for benign `deleteMessage` failures such as `message to delete not found`, `message can't be deleted`, `MESSAGE_ID_INVALID`, and `MESSAGE_DELETE_FORBIDDEN`. Use the existing `createTelegramRequestWithDiag` `options.shouldLog` hook so those benign delete outcomes do not emit `telegram deleteMessage failed` at ERROR before the catch runs. Widen `deleteMessageTelegram` to return `{ ok: true } | { ok: false; warning: string }`, catch only the benign delete cases, log at verbose level if useful, and rethrow non-benign auth, permission, network, rate-limit, or unknown failures. Update `extensions/telegram/src/action-runtime.ts` so the delete action surfaces `{ ok: false, deleted: false, warning }` when the helper returns a warning and still returns `{ ok: true, deleted: true }` on actual success. Add focused regression tests in the Telegram test files for all benign strings, non-benign rethrow/log behavior, and action-runtime warning propagation. Add a single-line user-facing changelog fix entry with `Thanks @Avicennasis`. Do not change core channel abstractions, plugin SDK contracts, Telegram auth behavior, or unrelated draft-stream/native-command delete paths.
