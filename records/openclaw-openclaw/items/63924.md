## Summary

Keep open. Current main already carries the Feishu schema direction and now has a fresh `readBestEffortConfig()` helper, but this PR still bundles unrelated fixes and would introduce blocking regressions in provider identity handling and skill path containment. It needs maintainer-guided rebase/splitting rather than cleanup closure.

## What This Changes

The PR replaces Feishu config schema generation, changes health snapshots to use best-effort config reads, normalizes BytePlus/Volcengine coding-plan provider IDs, and allows symlinked skill paths to fall back to unresolved paths.

## Best Possible Solution

Split the work. Keep the Feishu schema change as already handled on main, make any health fix a focused current-main change with a pinned-runtime regression test, leave coding-plan provider IDs distinct while relying on manifest auth aliases, and handle symlink aliasing separately with reviewed canonical-root escape coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep plan provider ids distinct:** `src/agents/model-selection.ts:94-100`
  - body: The PR collapses `volcengine-plan` and `byteplus-plan` into their base providers in generic normalization. Current manifests register those plan providers as separate catalog/runtime providers with coding-specific base URLs; only auth should alias to the base provider. This can make provider-qualified context and runtime lookups search the wrong provider key.
  - confidence: 0.89
- **[P1] Preserve realpath containment for skills:** `src/agents/skills/workspace.ts:212-219`
  - body: When a skill path's realpath escapes the configured root, this fallback returns the unresolved path because the textual path is under the root. Subsequent loading can follow the symlink to an outside `SKILL.md`, weakening the current fail-closed skill root boundary and contradicting existing symlink-escape tests.
  - confidence: 0.91

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete skill path containment concern; no workflow, lockfile, dependency-source, or publishing metadata changes were found in the provided diff.

Concerns:

- **[medium] Symlink fallback bypasses skill root containment:** `src/agents/skills/workspace.ts:212`
  - body: Returning the unresolved path after the resolved realpath escapes lets a symlink under the configured skills root load content from outside that root. Skills are operational instructions, so this boundary should stay fail-closed unless a separate canonical alias design is reviewed and covered by escape tests.
  - confidence: 0.91
