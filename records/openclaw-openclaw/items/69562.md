## Summary

Keep #69562 open. Current main still accepts `delivery.accountId` and `delivery.failureDestination.accountId` structurally but only validates announce channels during `cron.add`/`cron.update`; it does not verify that an explicit cron announce account belongs to the resolved channel. The PR is a focused gateway bug-fix attempt with review discussion addressing failure-destination coverage, default-account compatibility, normalized IDs, and inferred-channel behavior, so it remains useful rather than obsolete.

## Best Possible Solution

Keep the PR open and review it as a real cron/gateway bug fix. The best path is to add account-id validation for `cron.add` and `cron.update` using the channel plugin account contract (`listAccountIds`, resolved default account, normalized ids), cover both `delivery.accountId` and `delivery.failureDestination.accountId`, preserve `channel: "last"` and inferred-channel semantics, and avoid broad plugin registry loading in gateway hot paths where a lightweight resolver is available.
