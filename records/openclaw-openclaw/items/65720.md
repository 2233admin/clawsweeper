## Summary

Keep this issue open. Current main still serves the Control UI from the Gateway HTTP server on the Gateway process and port, and the only static fallback I found is the normal SPA route fallback while that process is alive. There is restart/recovery handling, but no crash-time same-port static maintenance server or gateway.serveStaticOnCrash-style config surface.

## Best Possible Solution

Keep this open for maintainer product/design judgment. The best path is either to document supervised restart as the supported resilience strategy, or to design a core Gateway/service-manager fallback that can safely serve a minimal maintenance page on the configured Gateway port while the real Gateway is down, with typed config/schema/help, docs, and tests for port handoff, custom fallback content, restart recovery, auth/origin behavior, and information disclosure.

## Risks / Open Questions

- A same-port fallback after the Gateway process exits cannot be implemented as an in-process Control UI toggle alone; it likely needs a supervisor/helper process or service-manager design that owns port handoff.
- A fallback maintenance page would need explicit auth, origin, caching, and information-disclosure rules because the Control UI is an authenticated admin surface.
- The requested behavior overlaps service reliability/product strategy, so a narrow autonomous fix risks creating unsupported lifecycle semantics.
