## Summary

Keep open. Current main still serves the Control UI from the Gateway HTTP process on the Gateway port, and the only static fallback is the normal SPA route fallback while that process is alive. There is restart/supervisor recovery work nearby, but no crash-survivor same-port maintenance server or gateway.serveStaticOnCrash-style config surface.

## What This Changes

The issue requests an opt-in Gateway feature that keeps a static WebUI or maintenance page reachable on the Gateway port after the Gateway process crashes.

## Best Possible Solution

Decide the supported resilience model first. If the feature belongs in core, implement it as a Gateway/service-manager capability with typed config, schema/help text, docs, and tests covering port handoff, custom maintenance content, restart recovery, auth/origin behavior, TLS, and safe disclosure.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current-main gap is still high-confidence: the Control UI is served by the Gateway process itself, so a hard process crash leaves no HTTP listener for a fallback page.

## Solution Assessment

Unclear. A simple gateway.serveStaticOnCrash toggle is probably not enough without an owner decision on supervisor/process architecture and security behavior; the safer path is a designed service-manager fallback or explicit docs that supervised restart is the supported answer.
