## Summary

Keep this PR open. Current main still has the MiniMax web-search metadata/onboarding mismatch, but the proposed auto-detection change needs maintainer judgment because a generic MiniMax model key can select MiniMax Search even when the account lacks Token Plan search access.

## What This Changes

The PR updates MiniMax web-search metadata, docs, and a registration test so `MINIMAX_API_KEY` is treated as an accepted web-search credential and MiniMax Search appears in setup.

## Best Possible Solution

Land a revised MiniMax web-search metadata/docs fix after deciding the credential policy: expose MiniMax Search in setup, keep contract/runtime/docs aligned, and either keep automatic detection limited to Token Plan-specific credentials or gate/lower-prioritize `MINIMAX_API_KEY` with tests that lock in the intended provider-selection behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate generic MiniMax keys before auto-detecting search:** `extensions/minimax/src/minimax-web-search-provider.ts:282`
  - body: Adding `MINIMAX_API_KEY` to this provider's `envVars` makes web-search auto-detection treat a generic MiniMax model key as search-capable. Since auto-detect returns the first credentialed provider by `autoDetectOrder`, users without Token Plan search access but with another working search key can be routed to MiniMax and hit runtime failures. Please keep auto-detect on Token Plan-specific keys, add a capability check, or otherwise guard this path.
  - confidence: 0.87
