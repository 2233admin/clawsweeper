## Summary

Keep #58057 open. Current main still lacks `dmPolicy: "dynamic"`, `groupPolicy: "dynamic"`, a pre-allowlist authorization hook, or an unknown-sender routing path. Existing BlueBubbles/iMessage behavior supports static allowlists and DM pairing, but allowlist mode remains static and group authorization remains explicit.

## Best Possible Solution

Keep the issue open for maintainer product/API judgment. A strong implementation path would add a typed channel/plugin authorization seam or BlueBubbles-owned dynamic identity resolver that runs before DM/group allowlist drops, with bounded caching/timeouts, fail-closed semantics, audit logs, tests for denied/allowed/timeout paths, docs for BlueBubbles/iMessage identifiers, and a clear decision on whether group session bindings can satisfy group access.
