## Summary

Keep this issue open. Current main still supports reply threading through assistant output tags, payload metadata, and channel replyToMode settings, but it does not implement an inbound user directive that strips trailing phrases like "reply in thread"/"rit" and forces the outbound response to thread. Related PR #47297 attempted this feature, but it was auto-closed unmerged and its review feedback points to fixable implementation details rather than obsolescence.

## Best Possible Solution

Implement a narrow core auto-reply inbound directive: detect deliberate trailing user phrases, strip them from all prompt/command body fields after finalization, carry a per-turn flag through MsgContext into reply payload building, and set replyToCurrent/replyToId through the existing reply-threading pipeline. Avoid a broad bare "thread" default unless maintainer product review explicitly accepts the false-positive tradeoff.
