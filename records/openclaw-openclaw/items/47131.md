## Summary

Keep this issue open. Current main has outbound reply tags, implicit reply-to-current behavior, and channel reply-threading policy, but it still does not implement the requested inbound trailing user directive that strips phrases such as `reply in thread` or `rit` before the model sees them and forces an explicit reply to the current inbound message. The linked implementation PR is closed and unmerged; its review feedback is actionable rather than evidence that the request is obsolete.

## What This Changes

Add a core inbound user directive that detects deliberate trailing reply-in-thread phrases, removes them from inbound prompt text, and makes the outbound response target the current inbound message when threading is supported.

## Best Possible Solution

Implement the directive in the shared auto-reply inbound pipeline, not in individual channels: strip deliberate trailing trigger text consistently across all inbound body fields, carry a typed per-turn flag on `MsgContext`, and translate that flag through the existing reply payload threading path as an explicit reply-to-current request. Keep channel-specific thread settings such as Feishu `replyInThread` separate, avoid the bare `thread` default unless deliberately approved, and update user-facing docs/changelog when it ships.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: finalize an inbound context with text like `hello rit`; current main leaves the phrase in prompt/command fields and has no context flag or payload parameter that can force reply-to-current from that inbound text.

## Solution Assessment

Yes, with constraints. A narrow shared inbound directive that feeds the existing reply payload threading path is the maintainable fix; channel-specific hacks or a broad bare `thread` default would be riskier.
