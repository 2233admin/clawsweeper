## Summary

Keep open. Current main includes a generic Telegram reply-normalization path, but it still does not implement the PR's core remaining parser-order fix: `parseReplyDirectives()` extracts `MEDIA:` before stripping reply tags, so `[[reply_to:<id>]] MEDIA:...` on the same line is still treated as text instead of media.

## Best Possible Solution

Refresh the PR against the current `extensions/telegram` layout, keep the already-present outbound payload normalization path, and implement the parser-order fix narrowly: strip reply tags before `MEDIA:` extraction only for the extraction path while preserving intentional Markdown/code-block whitespace. Add a regression test for `[[reply_to:<id>]] MEDIA:...` and a Telegram delivery test proving media delivery with the reply target.
