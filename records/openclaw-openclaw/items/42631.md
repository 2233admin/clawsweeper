## Summary

Current main still does not implement the requested job-level model/provider override for cron jobs that use `sessionTarget: "main"` with `payload.kind: "systemEvent"`. The current contract accepts model selection only on `agentTurn` payloads, normalization strips those fields from `systemEvent`, main-session cron execution only enqueues the text event and wakes heartbeat, and the related implementation PR #42708 was closed unmerged.

## Best Possible Solution

Keep this issue open. The best implementation is an additive cron contract for job-level model/provider selection on `main + systemEvent` jobs, validated with the same allowlist rules as isolated cron, then forwarded transiently into the cron-triggered main-session heartbeat/run without mutating unrelated interactive session state. Treat #42708 as prior implementation discussion, not a landed fix.

## Risks / Open Questions

- Closing this now would hide a still-present feature gap for users who need stable main-session system-event cron jobs on a cheaper model without editing internal session metadata.
- A future fix needs to scope the override to the cron-triggered main-session heartbeat/run; persisting it into shared session metadata could leak the cron model into later interactive work.
- This is a cross-surface contract change: protocol schema, normalization, execution plumbing, CLI/UI/docs, and regression tests should move together.
