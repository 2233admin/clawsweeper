## Summary

Current main still does not implement a job-level model/provider override for `sessionTarget: "main"` cron jobs with `payload.kind: "systemEvent"`. The cron protocol, runtime types, normalizer, CLI, UI, docs, and main-session execution path all still scope model selection to `agentTurn`/isolated jobs, while the linked implementation PR #42708 was closed unmerged.

## Best Possible Solution

Keep this issue open. The best path is an additive cron contract for per-job model/provider selection on `main + systemEvent` jobs, validated with the same allowed-model behavior as cron agent turns, then passed transiently into the cron-triggered main-session heartbeat/run without mutating unrelated interactive session state. Use #42708 as prior design context, not as a landed fix.

## Risks / Open Questions

- Closing now would hide a still-current feature gap for users who want cheaper recurring main-session cron runs without mutating internal session metadata.
- A future fix should avoid persisting a cron job's transient model choice into shared main-session state, or it could leak into later interactive work.
- The implementation is cross-surface: protocol schema, normalization, main-session heartbeat plumbing, CLI/UI/docs, and regression tests need to move together.
