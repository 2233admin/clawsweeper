## Summary

Current main still does not provide a job-level model/provider override for cron jobs that run as `sessionTarget: "main"` with `payload.kind: "systemEvent"`. The schema and runtime types still scope `model` to `agentTurn`, normalization strips `model`/`provider` from `systemEvent`, main-session cron execution only enqueues a system event and wakes heartbeat, and the linked implementation PR was closed unmerged.

## Best Possible Solution

Keep this issue open. The best implementation is an additive cron contract for job-level model/provider selection on `main + systemEvent` jobs, validated with the same model allowlist behavior as cron agent turns, then forwarded transiently into the cron-triggered main-session heartbeat/run without mutating unrelated interactive session state. Use #42708 as prior design context, not as a landed fix.

## Risks / Open Questions

- Closing now would hide a still-current feature gap for users who need stable main-session cron jobs to run on a cheaper model without editing internal session metadata.
- A fix should avoid persisting the cron override into shared main-session state, or a recurring job could leak its model choice into later interactive work.
- The remaining work is cross-surface: protocol schema, cron normalization, main-session execution plumbing, CLI/UI/docs, and regression tests need to move together.
