## Summary

Keep open. Current main still has the same product gap described in the report: Feishu tools can route by explicit `accountId`, same-id contextual `agentAccountId`, or Feishu default account, but there is no per-agent target-channel binding for a DingTalk-originated agent whose DingTalk account id differs from its intended Feishu account id.

## What This Changes

The issue requests a per-agent cross-channel account binding, such as `agents.list[].channelBindings.feishu.accountId`, so an agent invoked from one channel can use the correct account when calling tools for another channel.

## Best Possible Solution

Add a generic, documented target-channel account resolver keyed by active `agentId` plus target channel. Feishu should preserve explicit tool `accountId` first, then use the agent's bound Feishu account, then valid same-id contextual routing, then `channels.feishu.defaultAccount` or runtime fallback; update config types/schema/help/docs and add DingTalk-to-Feishu regression coverage.
