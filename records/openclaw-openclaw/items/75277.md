## Summary

Keep open. Current main still makes `memory/dreaming/**` eligible for ordinary QMD recall: the default QMD memory collection indexes the whole `memory/` tree, and the QMD search result path returns those hits as normal `memory` results without generated-artifact classification. Existing docs and tests only establish that dreaming reports are human-review artifacts and excluded from short-term promotion, not from recall.

## What This Changes

The issue asks whether QMD-backed memory recall should index, rank, tag, or exclude dreaming-generated report files under `memory/dreaming/**`.

## Best Possible Solution

Define a clear QMD recall contract for generated dreaming artifacts. Normal memory recall should either exclude `memory/dreaming/**` by default or surface those hits as an explicit generated/dream-report source with ranking and filter controls, while keeping the existing short-term promotion exclusion and updating memory/QMD docs to match.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to use QMD defaults with a markdown file under `memory/dreaming/deep/`: the configured collection pattern includes it, and the search assembly path has no filter or tag for that path. I did not run live QMD because this review was required to keep the checkout read-only.

## Solution Assessment

Unclear. Excluding or demoting generated report paths is likely the right hygiene direction, but the issue is intentionally asking for the intended model, so maintainers should choose exclude-vs-tag-vs-rank semantics before implementation.
