## Summary

Keep open for maintainer review. Current main already has the store-backed `current` alias retry that should satisfy the PR's added stored channel-requester test, while the linked live report also shows the constructed requester key missing from the store; this PR's new fallback still performs only another store lookup and would not resolve that remaining path.

## What This Changes

The branch adds a post-resolution `session_status` fallback that rewrites literal `sessionKey="current"` to the requester key, plus a regression test for a stored channel-style requester session.

## Best Possible Solution

Review #74141 and #72306 together with this PR. If channel `session_status` must work before the requester session is persisted, implement one canonical empty-store-safe path, either by fixing channel session registration or by returning a carefully scoped self-status/synthetic entry that preserves literal `current` precedence and visibility checks; if stored requester sessions are the only intended guarantee, keep the existing main behavior and land only focused coverage if useful.

## Reproduction Assessment

Yes for the code-level path: a channel-style requester with an empty session store and `sessionKey="current"` follows the visible source path to an unresolved store lookup and then throws. The narrower PR test with a stored requester key is expected to pass on current main already.

## Solution Assessment

No. The proposed fallback is placed after an existing store-backed `current` retry and still only calls `resolveSessionEntry`, so it is redundant for stored requesters and does not cover the missing-store failure shown in #74141.
