## Summary

Current main partially improves graceful restarts, but it does not finish the broader session-recovery feature requested here. The drain path now waits for active work before planned restarts, main sessions and subagents have some post-restart recovery paths, and restart sentinels can wake/continue specific config/update/restart flows. Meaningful requested scope remains open: cron runs interrupted by restart are marked failed rather than retried, drain-window message enqueues still fail fast instead of being persisted and replayed, and the proposed readiness gate / force workflow and restart-recovery config surface are absent. A maintainer comment also explicitly scoped PR #71465 as partial and left this issue open for broader recovery/interruption propagation.

## Best Possible Solution

Keep this issue open as the broader restart-recovery tracking item. The best path is to build on the shipped drain, restart-sentinel, main-session, and subagent recovery pieces, then add the missing product surface deliberately: a durable restart manifest or equivalent interruption ledger, explicit cron retry/requeue policy for interrupted runs, drain-window message persistence/replay, and an operator/agent readiness gate with documented force semantics.

## Risks / Open Questions

- A future implementation should avoid duplicating the existing restart sentinel, main-session recovery, and subagent orphan recovery paths; those are already real partial solutions on main.
- Cron interrupt retry/requeue needs careful idempotency and policy design, because current main intentionally records interrupted cron executions as errors instead of replaying them automatically.
- Drain-window replay needs durable idempotency/routing semantics so messages are not lost, duplicated, or delivered to the wrong channel/thread after restart.
