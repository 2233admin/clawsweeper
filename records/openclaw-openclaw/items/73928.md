## Summary

Keep this PR open. Current main still has the explicit `openclaw crestodian` non-TTY path logging an error and returning normally, while the command wrapper only exits non-zero when an error is thrown. The PR is a focused implementation candidate for #73646, but it needs a small test cleanup and maintainer selection against the overlapping open PR #74059 before merge.

## What This Changes

The PR sets `process.exitCode = 1` when `runCrestodian()` rejects non-TTY stdin/stdout and adds two unit tests for the non-TTY stdin and stdout paths.

## Best Possible Solution

Address the test cleanup, then land one canonical narrow fix that makes the explicit Crestodian non-TTY guard exit with code 1 while preserving `--message`, `--json`, and the interactive TUI path.

## Reproduction Assessment

Yes. The high-confidence reproduction path is `openclaw crestodian` through `registerCrestodianCommand()` into `runCrestodian()` with non-TTY stdin or stdout; current main logs the error and returns normally while the wrapper only exits non-zero on thrown errors.

## Solution Assessment

Yes after cleanup. Setting `process.exitCode = 1` at the existing Crestodian guard is a narrow maintainable fix that matches the nearby bare-root behavior; the safer test shape is to restore `process.exitCode` via `afterEach` or `try/finally`.
