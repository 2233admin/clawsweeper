## Summary

Keep this PR open. Current main still returns normally from the explicit Crestodian non-TTY path after printing the error, so the underlying bug is still valid; the PR is a narrow fix, but it needs the small test cleanup already identified and maintainer selection between this branch and the competing open fix in #74059.

## What This Changes

The PR sets `process.exitCode = 1` when `runCrestodian()` rejects non-TTY stdin/stdout and adds regression tests for both non-TTY stream cases.

## Best Possible Solution

Address the test cleanup, then land one canonical narrow fix that makes the explicit Crestodian non-TTY guard exit with code 1 while preserving `--message`, `--json`, and the interactive TUI path.

## Reproduction Assessment

Yes. The high-confidence path is the explicit `openclaw crestodian` command entering `runCrestodian()` with non-TTY stdin or stdout; current main logs the error and returns normally, and the wrapper only exits non-zero on thrown errors.

## Solution Assessment

Yes after cleanup. Setting `process.exitCode = 1` at the existing Crestodian guard is a narrow maintainable fix and matches the nearby bare-root behavior; the safer merge path is to make cleanup unconditional in the tests and pick one canonical PR for #73646.
