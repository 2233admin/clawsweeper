## Summary

Keep this PR open for maintainer review. Current main does not already implement the PR’s intended `skills.entries.<skill>.persist`, `metadata.openclaw.persist`, or `skills.trustedSources` behavior, so it cannot be closed as implemented. The linked issue remains open and already has a Codex review noting that the broader commercial/verified/encrypted skill ask needs maintainer product and security direction. The PR also makes a security-sensitive change by allowing source-class based bypass of skill code-safety scanning, so it needs explicit review rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open and route it through maintainer product/security review with the linked issue. The likely path is to either document and rely on existing `metadata.openclaw.always` for author-controlled eligibility, or design a carefully named config-level operator override with schema/docs/tests. The `trustedSources` scanner bypass should not land as a broad source-class skip without a clearer trust model, warning semantics, and security-owner approval.

## Risks / Open Questions

- Security-sensitive PR diff: `skills.trustedSources` would allow operators to skip installed skill code-safety scanning for whole source classes such as workspace, managed, extra, personal, or project skills. That broad bypass needs explicit security review and documentation before acceptance.
- The proposed `persist` metadata behavior overlaps with existing `metadata.openclaw.always`, while the proposed config-level `persist` field is new. Maintainers should decide whether a new user-facing key is warranted or whether docs around `always` and existing `skills.entries.*.config` are the better path.
- The PR appears narrower than the linked issue’s broader commercial/verified/encrypted skill-wrapper request, so landing it may not actually resolve the product ask without maintainer direction.
- The PR patch’s new audit finding uses `checkId: "skills-code-safety"`, while current audit findings use dotted IDs such as `skills.code_safety`; that inconsistency would need cleanup if the approach proceeds.
