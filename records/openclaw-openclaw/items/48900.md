## Summary

Keep open. Current main still does not implement this PR's intended native audio/video forwarding: there is no `nativeForwarding` config surface, no native media store, audio/video media understanding still produces text transcript/description outputs, and the embedded runner still submits only `ImageContent[]` through the prompt `images` option. The PR remains plausible feature work, but it is not obsolete or shipped, and the PR discussion identifies real unresolved correctness and dependency-contract risks.

## Best Possible Solution

Keep this PR open as useful but unfinished feature work. The best path is to revise it against current main with a typed native audio/video content contract from `@mariozechner/pi-ai`/`pi-coding-agent`, add `tools.media.audio/video.nativeForwarding` to the public config schema/help/generated docs, thread one resolved session/run key through capture and consumption, add cleanup and native-command redirect regression tests, and document provider capability, payload-limit, and fallback behavior for models that reject native audio/video blocks.

## Risks / Open Questions

- Merging the PR as provided would broaden raw audio/video egress to model providers, so the feature needs an explicit opt-in contract, provider capability/fallback behavior, payload limits, and docs/schema alignment.
- The proposed module-level base64 store has an identified cleanup/keying issue in native-command redirect flows that can retain up-to-25 MiB media blocks for the process lifetime.
- The current pinned `@mariozechner/pi-ai` and `@mariozechner/pi-coding-agent` type contracts only expose image content in prompt media, so force-casting audio/video blocks creates deployment-ordering and prompt-shape risk.
- The PR does not include tests or docs/schema/generated config updates for the new public config surface, native-command redirect cleanup, provider rejection behavior, or raw-media forwarding limits.
