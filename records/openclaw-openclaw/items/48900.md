## Summary

Keep open. Current main does not implement this PR's intended native audio/video forwarding: the media config has no `nativeForwarding` key, media understanding still emits text-only transcript/description outputs, and the embedded runner submits only `ImageContent[]` via `activeSession.prompt(..., { images })`. The PR is plausible product work, but it is not obsolete or implemented on main, and the provided review discussion identifies unresolved correctness and dependency-contract risks that should be addressed before landing.

## Best Possible Solution

Keep this PR open as useful but unfinished feature work. The best path is to revise it against current main with a typed native media content contract from `@mariozechner/pi-ai`, add the public config schema/help/generated docs for `tools.media.audio/video.nativeForwarding`, thread a single resolved session/run key through capture and consumption, add cleanup and redirect-flow regression tests, and document provider capability/fallback behavior for models that reject native audio/video blocks.

## Risks / Open Questions

- The proposed feature broadens raw media egress to model providers, so the opt-in contract needs schema, docs, size limits, provider capability checks, and clear failure behavior before merge.
- The PR's module-level base64 store has an identified cleanup/keying bug in redirect flows; that should be fixed with tests before maintainers consider landing it.
- The proposed cast around `@mariozechner/pi-ai` content types creates a deployment-ordering risk unless the dependency contract is updated or guarded at runtime.
- The provided diff does not touch workflows, lockfiles, install scripts, or release automation, so I did not find a supply-chain change of that class in this PR context.
