## Summary

Keep open. The report is a concrete npm global downgrade failure, and current main still routes explicit package downgrades through the same staged npm install plus exact-version verification path that can produce a `global install verify` failure while leaving the live package at the previous version. I did not find current-main source, docs, or tests proving the 2026.4.26 to 2026.4.25 npm downgrade now succeeds.

## What This Changes

The issue reports that `openclaw update --tag 2026.4.25` from a global npm 2026.4.26 install accepts the downgrade prompt but fails `global install verify` and leaves 2026.4.26 installed.

## Best Possible Solution

Fix the npm global downgrade path so an explicit older version installs into the staged prefix, verifies as that exact version, swaps into the live global root, and preserves post-update doctor, plugin sync, completion refresh, and restart behavior. Add focused regression coverage plus live Package Acceptance or Testbox proof for the 2026.4.26 to 2026.4.25 Linux npm global downgrade.

## Reproduction Assessment

Yes. The issue gives concrete Linux npm global steps and output, and current main still contains the staged install plus exact verification path that maps to the reported failure; the remaining gap is live reproduction in a disposable global npm install environment.

## Solution Assessment

Unclear until maintainer installer review. The likely maintainable fix is a narrow change in the npm global staging/swap/verification path with regression and live package proof, not relaxing the exact-version verifier or bypassing downgrade confirmation.
