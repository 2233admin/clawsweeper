## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the PR's intended Feishu SDK lifecycle/onError recovery and endpoint ClientConfig guard, while the PR's new sanitizer leaves the missing-ClientConfig PingInterval failure path unguarded.

## What This Changes

The PR changes Feishu WebSocket client construction to use Lark SDK lifecycle hooks and a sanitizing timeout-aware HTTP instance, updates monitor retry behavior around SDK reconnect exhaustion, and adds Feishu tests plus a changelog entry.

## Best Possible Solution

Land a narrow Feishu-owned fix that uses the Lark SDK lifecycle hooks and HTTP instance seam, avoids global WSClient prototype mutation, and guards missing as well as malformed endpoint ClientConfig values before they reach the SDK, with focused client and monitor tests.

## Reproduction Assessment

Yes. A focused reproduction is an endpoint response with data.URL but no data.ClientConfig: the PR sanitizer returns it unchanged, and the pinned Lark SDK then reads ClientConfig.PingInterval directly.

## Solution Assessment

No as written. The lifecycle/composition direction is maintainable, but the ClientConfig sanitizer needs to synthesize defaults for absent or malformed config instead of returning the failing response unchanged.
