## Summary

Keep open. Current main still hardcodes the Discord gateway startup READY timeout to 15s and runtime READY timeout to 30s, while the Discord config schema, generated metadata, and public docs expose only adjacent gateway metadata timeout settings; the reported restart log remains the active lifecycle path.

## What This Changes

Expose Discord gateway startup READY and runtime READY timeouts as configurable Discord plugin settings or env fallbacks instead of fixed 15s/30s constants for multi-account/operator setups.

## Best Possible Solution

Add Discord plugin-owned startup and runtime READY timeout settings that inherit from `channels.discord` into `channels.discord.accounts.<id>`, optionally use env fallbacks only when config is unset, pass the resolved values into the lifecycle wait/watch paths, and document how they differ from `gatewayInfoTimeoutMs` and event queue listener budgets.

## Reproduction Assessment

Yes for the fixed-timeout behavior. Current lifecycle tests can deterministically reproduce the startup restart by keeping `gateway.isConnected` false past 15s and the runtime force-stop by emitting `Gateway websocket opened` without reaching READY for 30s.

## Solution Assessment

Yes, with a boundary correction. The narrow maintainable solution is a Discord plugin/config surface; adding a core `gateway.discord.*` setting or deriving the lifecycle timeout from startup stagger is less appropriate because the current READY timer starts after the per-account stagger delay.
