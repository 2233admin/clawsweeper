## Summary

Keep open. Current main still uses fixed Discord gateway READY budgets of 15s for startup and 30s for runtime reconnects, and the Discord config schema/docs only expose the adjacent gateway metadata timeout. The existing ClawSweeper comment is consistent with the current code: this remains a narrow Discord plugin configurability gap with deterministic test coverage for the failing timeout paths.

## What This Changes

Request to make Discord gateway startup READY and runtime READY timeout budgets configurable for multi-account staggered startup instead of hardcoded 15s/30s values.

## Best Possible Solution

Add Discord plugin-owned `readyTimeoutMs` and `runtimeReadyTimeoutMs` settings under `channels.discord` with account-level inheritance, optional env fallbacks when config is unset, schema/types/metadata/docs coverage, and lifecycle tests proving both custom budgets are honored. Keep the setting out of core `gateway.*` config unless another channel needs the same generic contract.

## Reproduction Assessment

Yes. The current lifecycle tests can reproduce the startup timeout by keeping `gateway.isConnected` false past 15s and the runtime timeout by emitting `Gateway websocket opened` without READY for 30s.

## Solution Assessment

Yes. The narrow maintainable fix is a Discord plugin/config change following existing account inheritance and timeout metadata patterns; a bundled-output sed patch or core `gateway.discord.*` setting would be the wrong surface for this owner-specific behavior.
