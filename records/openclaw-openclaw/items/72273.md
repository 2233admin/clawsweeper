## Summary

Keep open. Current main and the latest release still hardcode the Discord startup and runtime READY waits, while only the adjacent gateway metadata timeout is configurable; the reported multi-account restart loop matches the existing lifecycle timeout paths.

## What This Changes

Request configurable Discord gateway startup and runtime READY timeouts so multi-account staggered startup does not restart later accounts before they can finish connecting.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under channels.discord for startup and runtime waits, with env fallbacks only when config is unset, schema/types/config metadata/docs coverage, and lifecycle tests proving custom values are honored.

## Reproduction Assessment

Yes. Existing fake-timer lifecycle tests can reproduce the mechanics by keeping gateway.isConnected false past 15s for startup restart, and by emitting 'Gateway websocket opened' without READY for 30s for the runtime force-stop path.

## Solution Assessment

Yes, with a scope correction. The narrow maintainable fix belongs in the Discord plugin/config surface and should thread resolved timeout values into runDiscordGatewayLifecycle; adding a core gateway.discord.* setting or editing bundled output would cross the wrong boundary.

Current Discord plugin code hardcodes startup READY timeout to 15s and runtime READY timeout to 30s in extensions/discord/src/monitor/provider.lifecycle.ts, which can restart staggered multi-account Discord setups before later accounts reach READY. Add fake-timer lifecycle tests for default and custom startup/runtime timeout values plus config/env inheritance where the repo normally tests Discord channel config.
