## Summary

Keep open. Current main still has fixed Discord READY budgets of 15s for startup and 30s for runtime reconnects, the reported restart/error paths remain covered by tests, and neither the strict Discord config schema nor public Discord docs expose a READY timeout override. The existing ClawSweeper comment on this issue reached the same conclusion, and public search did not show an open implementation PR for this timeout request.

## What This Changes

Make Discord gateway startup and runtime READY timeout budgets configurable instead of fixed 15s and 30s values, especially for multi-account Discord setups.

## Best Possible Solution

Add Discord plugin-owned startup and runtime READY timeout settings under `channels.discord` with per-account inheritance, optional env fallbacks when config is unset, schema/types/config metadata/docs coverage, and lifecycle tests that prove both custom values are honored. Keep the setting out of core `gateway.*` config unless multiple plugins need the same contract.

## Reproduction Assessment

Yes. Current lifecycle fake-timer tests already reproduce both paths: keep `gateway.isConnected` false past 15s for the startup restart, or emit `Gateway websocket opened` without READY for 30s for the runtime force-stop.

## Solution Assessment

Yes, with a scope correction. The best fix is a Discord plugin/config change following the existing account-inheritance and metadata-timeout patterns; a core `gateway.discord.*` setting or a sed patch against bundled output would be less maintainable.
