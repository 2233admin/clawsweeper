## Summary

Keep open. Current main still hardcodes Discord gateway READY waits at 15s for startup and 30s for runtime reconnects, while Discord config/docs only expose the adjacent gateway metadata timeout; the reporter's multi-account staggered-start failure matches the current lifecycle code and tests.

## What This Changes

Request configurable Discord gateway startup and runtime READY timeouts so multi-account staggered Discord setups do not restart before later accounts can finish connecting.

## Best Possible Solution

Add Discord plugin-owned, account-inheritable READY timeout settings under `channels.discord` for startup and runtime reconnect waits, with environment fallbacks only when config is unset, schema/types/config metadata/docs coverage, and lifecycle tests proving custom values are honored. Keep this out of core `gateway.*` config unless multiple plugins need the same generic contract.

## Reproduction Assessment

Yes. Current fake-timer lifecycle tests can reproduce the core mechanics by keeping `gateway.isConnected` false past 15s for startup restart, and by emitting `Gateway websocket opened` without READY for 30s for runtime force-stop.

## Solution Assessment

Yes, with a scope correction. The narrow maintainable solution is a Discord plugin/config change using existing per-account inheritance patterns; patching bundled output or adding a core `gateway.discord.*` setting would put owner-specific behavior in the wrong layer.
