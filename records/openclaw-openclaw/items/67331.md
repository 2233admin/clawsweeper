## Summary

Keep this PR open for maintainer review and request changes. Current main still lacks the standardized filesystem permission_denied helper and mutation payload guards, so the work is not obsolete, but the proposed patch has blocking correctness issues in sandbox mount-boundary classification and missing-file read semantics. The open dependent #67332 should be reviewed together with this PR rather than closing either half early.

## What This Changes

This PR adds an agent filesystem permission-denied helper, wraps apply_patch and sandbox filesystem boundary errors with standardized reason metadata, adds write/edit text payload size and NUL-byte guards, and updates related tests.

## Best Possible Solution

Request changes on this PR, then land it only after sandbox mount escapes classify as workspace-boundary denials, missing sandbox reads preserve not-found semantics, boundary-read classification uses the most specific available cause, payload guard limits are explicitly accepted, and focused host/sandbox regression coverage is green.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Map mount escapes to workspace_boundary:** `src/agents/fs-permission-denied.ts:21-24`
  - body: The sandbox bridges now wrap errors whose cause text says `Sandbox path escapes allowed mounts`, but the classifier only checks `escapes sandbox root`, `outside workspace root`, and `outside sandbox root`. Those mount-boundary denials will be emitted as `reason=unknown`, defeating the standardized reason contract this PR adds.
  - confidence: 0.96
- **[P2] Keep missing sandbox reads as not-found errors:** `src/agents/sandbox/fs-bridge-path-safety.ts:80-84`
  - body: `openReadableFile` now wraps every failed boundary open in `E_FS_PERMISSION_DENIED`, including `reason === "path"` misses. Existing behavior and tests expect missing sandbox files to surface as ENOENT/no-such-file, and media/read callers use `bridge.readFile` directly, so nonexistent files would get misleading permission-denied diagnostics.
  - confidence: 0.9
- **[P3] Preserve boundary-read causes when classifying:** `src/agents/apply-patch.ts:382-387`
  - body: `assertBoundaryRead` creates a synthetic `Failed boundary read ... (unsafe path)` cause instead of using `BoundaryFileOpenResult.error` when one exists. The classifier does not recognize that synthetic text, so validation failures that already carry a specific boundary cause can still become `reason=unknown`.
  - confidence: 0.78
