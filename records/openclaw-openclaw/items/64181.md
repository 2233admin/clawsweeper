## Summary

Keep this PR open. Current main still accepts the first embedded-agent payload text as a slug, strips dashes before truncation, and writes any non-null slug into the session-memory filename. The branch is a focused fix and the Greptile trailing-dash test gap was addressed, but it should be amended before merge because it still ignores the runner's explicit `isError` payload signal.

## What This Changes

The PR extracts a testable `slugifyLLMResponse` helper, routes `generateSlugViaLLM` through it, adds response-shape rejection tests, and moves dash stripping after slug truncation.

## Best Possible Solution

Land this PR or an equivalent narrow change after adding an explicit `payload.isError === true` rejection path in `generateSlugViaLLM`, with a regression test that mocks an error payload whose text would otherwise look slug-like. Keep the pure slugification helper and post-truncation dash stripping, then coordinate any broader slug-hardening behavior with #38161.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject error payloads before slugifying:** `src/hooks/llm-slug-generator.ts:133`
  - body: The embedded runner already marks assistant/provider failures with `payload.isError`, but this branch only sends `payload.text` through a length and word-count heuristic. A short error payload such as `quota exceeded` would still produce `quota-exceeded` and be written as the session-memory filename, leaving the reported failure mode partially open. Check the payload error flag before calling `slugifyLLMResponse` and cover that path in `generateSlugViaLLM` tests.
  - confidence: 0.86
