## Summary

Keep this open. Current main still sends non-Windows fs-safe write and copy operations through the Python-backed pinned write helper and still normalizes helper failures to the misleading regular-file error; the existing tests only cover spawn-failure fallback for remove/mkdir. The issue is also already paired with multiple open fixing PRs, so it should remain open until one canonical patch lands or those PRs are closed.

## What This Changes

Make fs-safe writes in slim Docker images without `python3` either keep working or produce a clear missing-Python diagnostic instead of `path is not a regular file under root`.

## Best Possible Solution

Land one canonical fs-safe fix that handles pinned write/copy helper spawn failures while preserving the existing alias, symlink, hardlink, and post-write identity protections; if any Docker runtime path still requires Python, document or diagnose that explicitly.

## Reproduction Assessment

Yes. The issue gives a concrete Docker/Node reproduction, and current main still contains the deterministic source path that spawns `python3` for pinned writes and normalizes spawn failures to the reported regular-file error.

## Solution Assessment

Unclear which open PR is the best final patch. A narrow write/copy spawn-failure fallback appears to solve the user-visible breakage more completely than a message-only diagnostic, but maintainers should choose among the existing candidates.
