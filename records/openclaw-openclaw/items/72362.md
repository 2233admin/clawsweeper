## Summary

Keep open. Current main still routes non-Windows fs-safe write and copy operations through the Python pinned-write helper and generic write-error normalization, and three open implementation PRs already reference this issue with closing intent, so maintainers should review those candidates instead of closing or duplicating the report.

## What This Changes

The issue requests that fs-safe write/copy paths in Docker images without `python3` either work via fallback or report/document the missing interpreter instead of surfacing a misleading regular-file error.

## Best Possible Solution

Land one canonical fix that handles pinned-write helper spawn failures for write and copy paths with regression tests, preserves the existing fs-safe boundary guarantees, and updates Docker docs or diagnostics if `python3` remains an expected runtime dependency.

## Reproduction Assessment

Yes. The issue provides a concrete Docker/Node reproduction, and current main still contains the Python-backed write/copy path plus generic normalization that explains the reported `spawn python3 ENOENT` being hidden behind the regular-file error.

## Solution Assessment

Unclear which open PR is the best final patch. A narrow helper-spawn fallback for write/copy with regression coverage appears more complete than a message-only fix, but maintainers should choose among the existing candidates.
