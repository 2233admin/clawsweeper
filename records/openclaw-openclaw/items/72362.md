## Summary

Keep open. Current `main` still sends non-Windows `writeFileWithinRoot` and `copyFileWithinRoot` through the pinned Python write helper, still normalizes helper failures to the generic `path is not a regular file under root` message, and the Docker runtime/docs do not make `python3` a required dependency. Three open PRs already reference this issue with closing intent, so the maintainer path is to review or replace those PRs rather than close the issue before a fix lands.

## What This Changes

Fix fs-safe write/copy behavior in slim Docker images without `python3`, so session-memory and workspace-only file tools either keep working or surface a clear missing-runtime diagnostic instead of a misleading path error.

## Best Possible Solution

Keep this issue open until one chosen fix lands on `main`. The end state should either remove the `python3` dependency for fs-safe writes/copies or handle missing-helper spawn failures with clear diagnostics and regression coverage for `writeFileWithinRoot` and `copyFileWithinRoot`, plus Docker docs or doctor coverage if `python3` remains a runtime requirement.
