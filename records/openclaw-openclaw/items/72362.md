## Summary

Current main still contains the reported fs-safe pinned-write path: Linux writes and copies call the Python helper and still normalize helper failures to the generic regular-file error. This issue should remain open because three open PRs already use closing intent for the same bug, so maintainer action is to choose and land one canonical fix rather than close or queue another repair.

## What This Changes

The issue asks for fs-safe writes in Python-less Docker images to either avoid the Python helper or surface a clear missing-python diagnostic so session-memory and workspace-only file tools do not silently fail.

## Best Possible Solution

Keep this issue open and land one canonical fix that either removes the Python runtime dependency for fs-safe write/copy paths or falls back safely on helper spawn failures, with regression tests proving genuine boundary failures still fail closed. If python3 remains a supported requirement for other Docker use cases, document the default image behavior and the existing extra apt package build-arg path separately.

## Reproduction Assessment

Yes. The issue gives a direct Docker/Node reproduction, and current main still has the same static path: spawn the pinned Python write helper, then normalize non-safe helper errors to the generic regular-file message.

## Solution Assessment

Unclear which submitted PR is the best final patch without maintainer review. A narrow write/copy fallback matching the existing remove/mkdir fallback pattern appears more complete than diagnostic-only handling, provided tests preserve the fs-safe boundary guarantees.
