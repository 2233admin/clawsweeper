## Summary

Keep open. Current main still routes non-Windows fs-safe write and copy paths through the Python-backed pinned write helper and still maps helper spawn failures to the reported regular-file error, while three open PRs already target this issue and should be reviewed or consolidated before closure.

## What This Changes

The issue reports that Docker/slim installs without `python3` make fs-safe write consumers such as session-memory and workspace-only file tools fail with a misleading regular-file error.

## Best Possible Solution

Keep this issue open and land one canonical fs-safe fix that handles pinned write/copy helper spawn failures with regression coverage while preserving alias, symlink, hardlink, and post-write identity checks; Docker docs or diagnostics should also be clear if images intentionally omit `python3`.

## Reproduction Assessment

Yes. The issue gives direct Docker and Node reproduction steps, and current main still has the deterministic source path that falls back to spawning literal `python3` and maps that failure to the reported regular-file error.

## Solution Assessment

Unclear which open PR is the best final patch. A narrow write/copy spawn-failure fallback that mirrors the existing mkdir/remove fallback appears more complete than a message-only diagnostic, but maintainers should choose one canonical approach.
