## Summary

Keep open. Current main contains a shared code-fence-aware sanitizer and final assistant extraction now uses it, but the user-visible outbound channel boundary is not consistently covered: block replies still sanitize only think/final/tool scaffolding, `sanitizeUserFacingText` does not strip relevant-memory blocks, and the issue is already paired with an open unmerged closing PR that should be reviewed rather than bypassed.

## What This Changes

Strip assistant-reflected `<relevant-memories>` memory scaffolding from outbound messaging-channel deliveries while preserving literal code examples.

## Best Possible Solution

Use the shared code-fence-aware assistant-visible sanitizer at every assistant-origin outbound delivery boundary, including generic user-facing normalization, block reply chunks, message-end safety sends, and preview/partial updates, with focused regression tests for stripped memory scaffolding and preserved fenced-code literals.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass assistant text containing `<relevant-memories>...</relevant-memories>` through `sanitizeUserFacingText` or the `emitBlockChunk` path and observe that current main does not apply the relevant-memory sanitizer there.

## Solution Assessment

Yes for the issue direction, but the linked open PR still needs review against all outbound boundaries. The maintainable solution is to reuse the existing code-fence-aware sanitizer rather than adding a raw regex at one channel-specific endpoint.
