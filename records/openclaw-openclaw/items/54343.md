## Summary

Keep this PR open for maintainer review, but do not merge it as-is. The intended hardening is plausible, yet the current branch has blocking correctness and authorization defects, and the remaining `/stop` scope policy is security-sensitive because the direct gateway abort RPCs still use `operator.write`.

## What This Changes

The PR replaces the `/stop` handler's `rejectUnauthorizedCommand` sender gate with an internal-channel `operator.admin` scope check in `src/auto-reply/reply/commands-session-abort.ts`.

## Best Possible Solution

If maintainers want this hardening, preserve `rejectUnauthorizedCommand(params, "/stop")`, import and add `requireGatewayClientScopeForInternalChannel` immediately afterward for internal-channel `operator.admin` enforcement, and add focused tests for internal and external authorization behavior. If `/stop` should stay aligned with direct gateway abort RPCs, leave the current gate unchanged and document that policy instead.

## Reproduction Assessment

Yes for the PR defects: static inspection of the diff shows the helper is called without being imported, and current source shows that helper bypasses non-internal channels while command dispatch does not globally reject unauthorized senders. The underlying policy claim is only partially reproducible because current gateway abort RPC methods still use `operator.write`.

## Solution Assessment

No. Replacing the existing sender gate is not the best solution; the maintainable implementation is an additive internal scope gate after the existing authorization check, subject to maintainer confirmation of the `/stop` admin policy.
