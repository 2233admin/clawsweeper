## Summary

Keep open. Current main still matches the reported bypass: configured `tools.exec` is converted into an implicit profile allowlist entry for `exec` and `process`, then merged with explicit `alsoAllow`. The focused fix PR #47991 was closed unmerged, and the broader #69190 does not supersede this specific policy bug.

## What This Changes

Ensure restricted agent tool profiles are not widened to `exec` merely because `tools.exec` is configured outside the explicit allowed tool list.

## Best Possible Solution

Revise the tool-policy contract so global `tools.exec` configuration cannot widen restricted agent profiles, keep deny precedence intact, add regression coverage for `messaging` plus explicit `alsoAllow`, document the global-versus-agent semantics, and review the `tools.fs` analogue in the same policy pass.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure global `tools.exec`, set an agent to `tools.profile: "messaging"` with `alsoAllow` excluding `exec`, and `resolveEffectiveToolPolicy()` still produces a `profileAlsoAllow` union containing `exec` and `process`.

## Solution Assessment

No. Current main conflates exec safety configuration with exec access; the safer maintainable fix is to require explicit profile/allow opt-in for runtime access while preserving deny precedence and documenting any intentional per-agent exception.
