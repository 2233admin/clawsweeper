## Summary

Keep open. Current main still has the unresolved native subagent sandbox inheritance gap described in the issue discussion: workspace inheritance is threaded, but the spawned tool context and child agent call do not carry the parent’s resolved Docker env/setup/runtime context, and sandbox resolution still happens independently for the child target.

## What This Changes

The issue reports that sandboxed native subagent runs do not reliably receive configured Docker env variables, current workspace state, or refreshed setupCommand effects after sandbox recreation.

## Best Possible Solution

Define native sessions_spawn sandbox inheritance explicitly: same-agent sandboxed children should observe the effective sandbox Docker env/setup and current intended workspace after recreate, while cross-agent children should not receive parent Docker env/setup unless a reviewed policy allows it. Add regression tests and a small docs clarification for the chosen semantics.

## Reproduction Assessment

Yes for a focused code-level reproduction: configure a sandboxed agent with sandbox.docker.env and setupCommand, spawn a native same-agent subagent, and assert the child runtime sees those values/effects after openclaw sandbox recreate --agent. I did not run a live Docker repro because this review is read-only, but the static path still lacks the propagation point needed for that scenario.

## Solution Assessment

Unclear until maintainers choose the inheritance semantics. A minimal same-agent resolved-context handoff may be enough, but broader container sharing or cross-agent propagation would touch the sandbox/secret boundary and needs explicit review.
