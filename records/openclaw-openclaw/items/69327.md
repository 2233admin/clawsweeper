## Summary

Keep open. Current main still forwards spawned workspace lineage for native `sessions_spawn`, but it does not propagate the parent run's resolved Docker env/setup context into the child sandbox; Docker env/setup support exists generally, so the remaining gap is subagent inheritance/recreate semantics and needs maintainer design review.

## What This Changes

The issue requests that sandboxed subagent runs reliably receive configured `sandbox.docker.env`, current workspace state, and recreated `setupCommand`/bootstrap effects after `openclaw sandbox recreate --agent ...`.

## Best Possible Solution

Keep this open for maintainer design review, then implement a focused fix that defines native `sessions_spawn` sandbox inheritance semantics. A likely direction is same-agent inheritance of the effective resolved Docker env/setup context or an explicitly shared sandbox context, with cross-agent non-inheritance by default, plus regression coverage for env/setup after recreate and workspace freshness.

## Risks / Open Questions

- Copying parent Docker env/setup into children can leak parent or same-agent secrets into cross-agent subagents unless the policy is explicit and covered by negative tests.
- The report bundles Docker env propagation, setupCommand/recreate behavior, and workspace freshness; fixing only env inheritance may leave stale workspace or stale runtime symptoms unresolved.
- `setupCommand` currently runs on container creation, so any fix needs to account for sandbox scope, config hashing, and explicit recreate behavior rather than only adding process-level env forwarding.
