## Summary

Keep this PR open. Current main still resolves `remote.batch.timeoutMinutes` and passes it into provider batch execution, but the shared memory-host SDK HTTP helpers and OpenAI/Gemini/Voyage batch upload/create/status/error-read/download paths still do not accept or forward a remaining HTTP timeout budget. The PR is a live replacement for the stale #49981 idea credited to @yintamaa, but it needs maintainer review because the proposed timeout resolver can disable timeouts when a provided budget expires.

## Best Possible Solution

Keep this PR open, address the timeout resolver/retry-loop review so provided but exhausted budgets fail closed instead of disabling request timeouts, then land this PR or an equivalent narrow replacement that threads one remaining-time budget through memory-host SDK HTTP helpers and OpenAI/Gemini/Voyage batch upload, create, status/error-read, retry delay, and download phases. Preserve the #49981 credit, keep #49937/#49963 as historical direct/inline timeout context, and validate with focused memory-host/provider-batch tests plus the changed gate.
