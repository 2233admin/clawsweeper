## Summary

Keep open. Current main still lacks remote provider-batch HTTP timeout-budget propagation, so this PR is not obsolete, but the proposed patch has a concrete blocker: exhausted timeout budgets can be normalized to an absent timeout and then proceed through shared retry/fetch paths without the configured batch budget.

## What This Changes

This PR adds SDK timeout-budget helpers, docs/changelog text, and provider tests while threading `remote.batch.timeoutMinutes` through memory-host HTTP helpers and OpenAI/Gemini/Voyage remote embedding batch upload, create, polling/error-read, retry-delay, and download calls.

## Best Possible Solution

Land an updated narrow version that keeps the shared remaining-time budget across all provider-batch phases, distinguishes an absent timeout from an exhausted budget, fails fast before retrying or starting another HTTP request when the budget is spent, preserves existing SSRF/proxy behavior, and keeps the #49981 credit with focused SDK/provider regression coverage.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: on current main, `remote.batch.timeoutMinutes` reaches provider `batchEmbed()` but cannot reach `withRemoteHttpResponse()`/`postJson()`/`uploadBatchJsonlFile()`; on the PR branch, a `timeoutMs` function returning `0` reproduces the fail-open resolver path without running the full suite.

## Solution Assessment

No, not as written. Threading one remaining budget through provider batch phases is the right direction, but the resolver/retry behavior should fail closed on expired budgets instead of converting them to `undefined` and falling back to no request timeout.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Fail closed when the batch timeout budget is exhausted:** `packages/memory-host-sdk/src/host/remote-http.ts:20`
  - body: `resolveRemoteHttpTimeoutMs()` returns `undefined` for `0` or negative values. Since `postJsonWithRetry()` uses an undefined result to skip the budget-aware retry loop, and `postJson()`/upload then forward `undefined` into `withRemoteHttpResponse()`, an underflowed remaining-time budget can start another provider HTTP request without the configured batch timeout. Keep “no timeout configured” distinct from “budget exhausted” and cover the `<= 0` case.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete timeout-budget fail-open risk in shared remote memory HTTP plumbing.
