## Summary

Keep open, but the proposed patch is not safe to merge as written. Current main still does not ship a destructive-command catalog or denylist/HIL mode, but the PR’s command-text patterns and usage example do not line up with the production exec allowlist matcher, so an integration following this code could fail open for the security behavior it is meant to enforce.

## What This Changes

This PR adds a new `src/security/destructive-commands.ts` catalog of command-text destructive patterns plus a colocated Vitest file that checks those constants with ad hoc string-prefix matching.

## Best Possible Solution

If maintainers want this capability, implement it as an integrated argv-aware exec approval policy layer with a versioned config shape, production matcher tests, docs, and security review, likely in the same design lane as the open denylist-mode discussion. Otherwise, keep the current allowlist/ask model and do not merge a standalone catalog that implies enforcement without being wired into the approval evaluator.

## Reproduction Assessment

Yes. A focused static reproduction is to evaluate a pattern such as `kubectl delete *` through current `matchAllowlist` with an `ExecutableResolution` for `kubectl`; the matcher compares against the executable path or basename, so the command-text pattern does not match the destructive argv.

## Solution Assessment

No. The PR is not the best way to solve the issue because it adds an unused command-text catalog and points it at a path/basename allowlist API; the safer solution is a dedicated command/argv matcher or a designed denylist policy integrated directly into exec approval evaluation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Match destructive argv with production semantics:** `src/security/destructive-commands.ts:28`
  - body: These entries are full command-prefix strings, but current `matchAllowlist` only compares patterns to the resolved executable path or PATH-invoked executable basename. A future caller following this catalog would not match `kubectl delete pod foo` or `rm -rf ...`, so the HIL gate can fail open; represent the executable and argv with production semantics or add a dedicated command-text matcher before merging.
  - confidence: 0.9
- **[P2] Use the real `matchAllowlist` call shape:** `src/security/destructive-commands.ts:153-155`
  - body: The documented example passes a raw command string as the first argument and the entries array second, but the exported signature is `matchAllowlist(entries, resolution, argv?, platform?)`. Code copied from this example will not exercise the approval matcher correctly and will either fail type checking or produce the wrong result.
  - confidence: 0.95
- **[P2] Test the catalog through the approval matcher:** `src/security/destructive-commands.test.ts:19-31`
  - body: The tests use `startsWith` after splitting/replacing `" *"`, which is not how exec approvals decide allowlist hits. These assertions pass even when every production integration would miss, so they need to exercise the same matcher or a new dedicated destructive-command matcher that the runtime will actually call.
  - confidence: 0.88
