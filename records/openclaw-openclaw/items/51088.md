## Summary

Keep open. Current main supports iMessage acknowledgement behavior through the recommended BlueBubbles channel and has shared inbound message hooks, but the still-documented legacy `channels.imessage` / `imsg` path requested here does not expose channel-owned read-receipt or tapback acknowledgement support. Because that path is explicitly legacy and may be removed, the remaining work needs maintainer product direction rather than conservative cleanup closure.

## What This Changes

Add native sender-visible acknowledgement for inbound legacy iMessage messages, either through read receipts, tapbacks wired to `ackReactionScope`, or a hook-based custom acknowledgement path.

## Best Possible Solution

Make the supported iMessage acknowledgement contract explicit. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement settings, schema/docs, capabilities, inbound behavior, and tests; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and explicitly scope legacy `imsg` out.

## Risks / Open Questions

- Legacy `imsg` is documented as legacy and may be removed, so adding acknowledgement support there could conflict with the intended BlueBubbles migration path.
- The acknowledgement mechanism needs product, UX, and privacy judgment: tapbacks add conversation noise, activation-based read receipts can steal focus and require macOS permissions, and a native non-focus-stealing API is not established in current code.
- The existing generic `message:received` / `message_received` hook surface may be enough for custom scripts, but it does not define channel-owned read-receipt or tapback semantics for legacy iMessage.
