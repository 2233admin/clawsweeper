## Summary

Keep open. Current main has BlueBubbles read receipts/tapbacks and a generic `message_received` hook, but the legacy `channels.imessage`/`imsg` path described here still has no channel-owned acknowledgement config, capability, schema support, or inbound read-receipt/tapback behavior. Because that path is documented as legacy while BlueBubbles is recommended, this needs maintainer product direction rather than automatic closure or repair.

## What This Changes

Request native acknowledgement for inbound legacy iMessage messages, either through read receipts/tapbacks tied to `ackReactionScope` or a hook that lets users acknowledge delivery without polling.

## Best Possible Solution

Make the supported iMessage direction explicit. If legacy `channels.imessage` remains supported, add channel-owned acknowledgement with config/schema/docs and direct/group coverage; if BlueBubbles is the intended answer, document that read receipts and tapback acknowledgements require `channels.bluebubbles` and explicitly scope legacy `imsg` out.

## Risks / Open Questions

- Legacy `imsg` may be removed, so adding acknowledgement behavior there could conflict with the intended BlueBubbles migration path.
- The acknowledgement mechanism needs maintainer UX/security judgment: tapbacks add conversation noise, read-receipt activation can steal focus and require Automation/Accessibility permissions, and no native macOS/imsg read-receipt API is established in current code.
- The existing `message_received` hook may support custom workarounds, but it is fire-and-forget observer plumbing rather than channel-owned acknowledgement semantics.
