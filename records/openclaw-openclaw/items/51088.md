## Summary

Keep open. Current main still lacks native acknowledgment support for the legacy iMessage `imsg` channel: no iMessage reaction/read capability is advertised, the iMessage config schema/type has no ack/read setting, and the inbound monitor dispatches messages without sending a tapback or read receipt. BlueBubbles already covers read receipts and ack reactions, and current generic message hooks can observe inbound messages, but neither closes the legacy iMessage `ackReactionScope` gap described here.

## Best Possible Solution

Keep this issue open for maintainer product decision. If legacy `channels.imessage` remains supported, add channel-owned acknowledgment after access/mention gating and before dispatch, with typed config/schema/docs for tapback/read behavior, `imsg` capability handling, echo/reflection safeguards, and direct/group tests. If maintainers choose BlueBubbles as the only supported iMessage acknowledgment path, document that explicitly on https://docs.openclaw.ai/channels/imessage and close after that decision.

## Risks / Open Questions

- Legacy `channels.imessage` is documented as deprecated and BlueBubbles is recommended for new iMessage setups, so maintainers need to decide whether to add native ack behavior to the legacy path or document BlueBubbles/hooks as the supported answer.
- Read receipts or tapbacks through Messages.app/Apple automation can involve macOS permissions, focus behavior, and `imsg` capability/version constraints; a fix should avoid brittle UI automation unless maintainers explicitly accept that tradeoff.
- Generic inbound message hooks now exist and may support custom local workarounds, but they are not documented as the supported iMessage acknowledgment path and do not implement native channel ack behavior.
