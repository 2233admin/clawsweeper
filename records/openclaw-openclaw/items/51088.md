## Summary

Keep open. Current main still does not add native acknowledgment/read-receipt support to the legacy `channels.imessage` / `imsg` path. BlueBubbles already supports read receipts and tapback ack reactions, and generic `message:received` hooks can support custom workarounds, but the specific legacy iMessage channel gap reported here remains unresolved.

## Best Possible Solution

Keep this open for maintainer follow-up. If legacy `channels.imessage` remains supported, add channel-owned acknowledgment support in `extensions/imessage`: config/schema/type fields for ack/read behavior, capability metadata, a tapback or read-receipt action before dispatch/queue handoff after authorization, and tests for direct and group scope handling. If maintainers want legacy `imsg` frozen, document that BlueBubbles is the supported iMessage acknowledgment path and close only after that explicit product decision.
