## Summary

Keep open. Current main has pre-delivery `message_sending` rewrite/cancel support and a same-model empty-response continuation retry, but it does not expose a hook that can reject generated content and re-enter `agents.defaults.model.fallbacks`; the fallback-chain part of the request remains unresolved.

## What This Changes

Request a post-LLM, pre-delivery message hook that can pass, modify, or reject model output and route rejects into the existing model fallback chain for empty or low-quality responses.

## Best Possible Solution

Keep this open for a generic agent-output quality gate that runs before delivery and before fallback decisions are finalized. The maintainable end state would preserve `message_sending` as delivery transform/cancel, add a bounded typed rejection path with provider/model/finish metadata, and convert approved rejects into a `FailoverReason`/`FailoverError` that can use the existing fallback chain safely.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available from current source and tests: a Pi run with empty assistant text takes the empty-response continuation retry path, while `message_sending` cancellation only suppresses delivery and never invokes fallback model selection.

## Solution Assessment

Unclear/no. A channel-level `message_sending` hook is not the best place to trigger model fallback because it runs in delivery; the safer direction is an agent-loop output gate, or a Pi-compatible extension of the finalization lifecycle, that can reject before delivery side effects and hand a typed reason to failover policy.
