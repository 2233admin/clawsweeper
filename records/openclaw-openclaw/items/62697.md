## Summary

Keep open. Current main still lacks a WhatsApp Newsletter inbound route: `@newsletter` JIDs are not treated as groups, cannot be normalized to an E.164 sender, and are dropped before access control can apply any separate policy. The related open PR is useful adjacent work for outbound newsletter targets, but it explicitly leaves this inbound processing request unresolved.

## What This Changes

Add support for receiving and processing inbound WhatsApp Channels/Newsletter messages from `@newsletter` JIDs with separate access control instead of DM allowlist handling.

## Best Possible Solution

Implement this inside the WhatsApp plugin by detecting `@newsletter` JIDs, adding explicit newsletter policy and allowlist config, and enqueueing authorized newsletter messages with a distinct chat type or metadata while avoiding DM pairing, group mention assumptions, and accidental reply/read-receipt behavior. Keep outbound newsletter target work separate from #73393 unless maintainers choose to combine them deliberately.

## Reproduction Assessment

Yes. A focused static reproduction is clear: a `messages.upsert` item with `remoteJid: "123@newsletter"` is non-group, `resolveJidToE164` returns null, and the monitor drops it before access control or `onMessage`.

## Solution Assessment

Yes, the requested separate newsletter access-control path is the right shape. Reusing `dmPolicy` would be less maintainable and could broaden access because newsletter broadcasts are neither phone-number DMs nor WhatsApp groups.
