## Summary

Keep this PR open. Current main still truncates live `openclaw channels status --probe` allowFrom output to two entries, so the PR is not implemented or obsolete. The PR is narrowly scoped, but it still needs maintainer choice on output/privacy policy, tests, and coordination with the overlapping #51179 summary-path fix.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to consolidate it with #51179 or choose one canonical patch, add a shared allowlist display helper used by both live gateway status and summary/status surfaces, cover 2-entry, 3-entry, exact-limit, and overflow cases in tests, and decide whether status output should show all entries, cap them, or redact them.

## Risks / Open Questions

- The proposed output reveals more `allowFrom` entries, which may be phone numbers or user IDs; maintainers should choose the intended cap/redaction policy before landing.
- The `,+N more` overflow suffix can visually resemble another phone-number-like token; a parenthesized marker would be clearer.
- No current main test covers allowFrom overflow behavior in `formatGatewayChannelsStatusLines()`, and the existing summary-path test still asserts truncation.
- The open #51179 PR overlaps on the config/summary path, so landing only this PR would leave inconsistent allowFrom display behavior.
