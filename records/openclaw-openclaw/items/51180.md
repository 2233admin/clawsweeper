## Summary

Keep open. Current main still truncates `allowFrom` to two entries in the live `channels status --probe` formatter, and the latest release tag has the same behavior. The PR addresses a real diagnostic bug from #51174, but it is not implemented on main and may need consolidation with overlapping PR #51179 plus shared formatting/tests.

## Best Possible Solution

Keep this PR open for maintainer review. The best code direction is a small shared `allowFrom` display helper used by both `formatGatewayChannelsStatusLines()` and `buildChannelSummary()`, with regression tests for 2, 3, exact-limit, and overflow cases, plus an explicit decision about whether full allowlists belong only in sensitive/admin status surfaces. Then merge one canonical PR or consolidate with #51179.

## Risks / Open Questions

- The one-file PR fixes the live `channels status --probe` display but leaves the related channel-summary/config-only truncation path unchanged.
- Expanding displayed `allowFrom` entries can expose more phone numbers or user IDs to status consumers; maintainer review should decide the correct redaction/admin boundary.
- The proposed `,+N more` suffix can look like another phone-number token; a parenthesized or bracketed overflow marker would be clearer.
- Overlapping PR #51179 appears to address the same symptom, so maintainers may want to choose one canonical implementation or consolidate the fixes.
