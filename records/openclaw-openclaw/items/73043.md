## Summary

Keep open. Current main still has no OpenClaw route from simple completion to the Claude CLI backend and no early unsupported-runtime error for the `claude-cli`/`cli://claude` path; the issue's harness matches the current source and upstream pi-ai behavior. The remaining choice is product/runtime design rather than a cleanup close.

## What This Changes

Make Lossless/pi-ai simple completions using the synthetic `claude-cli/claude-sonnet-4-6` model either route through OpenClaw's Claude CLI backend or fail with an explicit unsupported-transport error.

## Best Possible Solution

Define the supported simple-completion contract for CLI runtime models: either expose an OpenClaw-owned helper that can run a non-mutating Claude CLI one-shot completion for context engines, or fail early through the simple-completion/plugin SDK path with a clear unsupported-runtime message and docs guidance for LCM-style plugins.

## Reproduction Assessment

Yes. The issue provides a concrete standalone harness, and static verification shows the reported model is handed to pi-ai's Anthropic HTTP/SSE path with `baseUrl`, not to OpenClaw's Claude CLI backend.

## Solution Assessment

Unclear. Passing `cli://claude` to raw pi-ai is not a maintainable contract; the safer solution is an explicit OpenClaw simple-completion/runtime policy, with either first-class CLI-backend support or a clear unsupported-runtime error.
