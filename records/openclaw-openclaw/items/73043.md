## Summary

Keep open. Current main still does not map simple-completion calls for `claude-cli` or `cli://claude` onto the Anthropic CLI backend, and it also does not fail early with a specific unsupported-runtime error.

## What This Changes

The issue asks OpenClaw's simple-completion/Lossless path to either support `claude-cli`/`cli://claude` Sonnet completions or return a clear unsupported transport error instead of a generic connection error.

## Best Possible Solution

Define the supported simple-completion contract for CLI runtime models: add a first-class OpenClaw helper for non-mutating Claude CLI one-shot completions, or detect CLI runtime/baseUrl inputs and return a clear unsupported-runtime error with docs guidance.

## Reproduction Assessment

Yes. The issue includes a concrete standalone harness, and static verification against current main shows the simple-completion path reaches pi-ai completion without a `claude-cli` backend dispatch or early unsupported-runtime check.

## Solution Assessment

Unclear. Passing `cli://claude` into the raw pi-ai Anthropic path is not the maintainable fix; the safer end state is an explicit OpenClaw simple-completion runtime contract or an early, documented unsupported error.
