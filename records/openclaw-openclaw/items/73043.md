## Summary

Keep open. Current main still has the same source-level gap described in the existing ClawSweeper comment: simple completion prepares a pi-ai model and calls pi-ai completion, while Claude CLI support is registered as a CLI backend/runtime path, not as a `cli://claude` Anthropic HTTP transport. The issueŌĆÖs harness remains a credible reproduction, and there is no merged implementation or documented early unsupported-runtime error yet.

## What This Changes

The issue asks for Lossless/pi-ai simple completion with `claude-cli/claude-sonnet-4-6` and `baseUrl = cli://claude` to either route through OpenClawŌĆÖs Claude CLI transport or fail with a specific unsupported-runtime error.

## Best Possible Solution

Define the supported contract for simple completions against CLI runtime models: either add a first-class OpenClaw helper that runs non-mutating Claude CLI one-shot completions with clear runtime/session policy, or reject CLI-runtime refs early with a specific unsupported-runtime error and docs guidance for Lossless/LCM users.

## Reproduction Assessment

Yes. The issue provides a concrete harness, and static verification of current main plus upstream pi-ai source shows `cli://claude` is treated as an Anthropic SDK base URL rather than dispatched to OpenClawŌĆÖs Claude CLI backend.

## Solution Assessment

Unclear. The reportŌĆÖs two acceptable outcomes both make sense, but the best maintainable fix requires a maintainer decision on whether simple-completion callers may use CLI runtimes or should receive an explicit unsupported-runtime failure.
