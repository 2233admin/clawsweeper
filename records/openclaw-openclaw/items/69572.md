## Summary

Keep open. Current main still implements Feishu typing feedback as Feishu message/comment reactions, the tests and docs still encode that reaction-based contract, and the related open PRs found here continue to work on reaction targeting/customization rather than replacing it with native Feishu/Lark typing status.

## What This Changes

The issue requests replacing Feishu's `Typing` emoji reaction-based feedback with native Feishu/Lark typing-status behavior.

## Best Possible Solution

Keep this open as a bundled Feishu plugin behavior bug. A maintainer should confirm the official Feishu/Lark native typing/status API and permissions, then update the Feishu-owned typing lifecycle in `extensions/feishu/src/typing.ts` and `extensions/feishu/src/reply-dispatcher.ts`, plus `extensions/feishu/src/comment-reaction.ts` if comment typing shares the same setting; docs and tests should distinguish native typing from any reaction fallback.

## Reproduction Assessment

Yes for the current implementation path: with Feishu typing enabled, `createFeishuReplyDispatcher` calls `addTypingIndicator`, which deterministically sends `im.messageReaction.create` with the `Typing` emoji. I did not run a live Feishu client check in this read-only review, but the source-level reproduction matches the reported emoji-reaction behavior.

## Solution Assessment

No current fix is present. Replacing the reaction helper with the narrow Feishu-owned native typing/status lifecycle is the right direction only after confirming the official API and permissions; guessing between adjacent SDK APIs would be riskier than maintainer-confirmed implementation.
