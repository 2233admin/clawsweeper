## Summary

Keep open. Current main still does not expose REST-native session reset/status/compact/model/reasoning endpoints for HTTP `/v1/chat/completions` integrations. The underlying Gateway RPC session methods and schemas exist, but the HTTP router only wires OpenAI-compatible chat/models/embeddings/responses, tools invoke, and adjacent `/sessions/{key}/kill` plus `/sessions/{key}/history` helpers. Related PR #23941 was closed unmerged, and #29985 was explicitly scoped away from the requested REST surface.

## Best Possible Solution

Keep this as the canonical Gateway HTTP API tracker. The best implementation is a focused, authenticated REST surface for session reset/status/compact/model/reasoning that maps onto existing `sessions.resolve`/`sessions.list`, `sessions.reset`, `sessions.compact`, and `sessions.patch` semantics, preserves method-scope authorization, adds route/request/response tests, and updates the public HTTP docs. Do not re-enable slash-command interception on `/v1/chat/completions`.

## Risks / Open Questions

- A REST implementation would expose session-mutating control-plane actions over HTTP, so it needs explicit operator-scope checks and request/response schemas rather than command-text interception.
- The closed #23941 review already flagged design pitfalls around strict `sessions.list` schema validation, path matching, and reset reason validation; those should be addressed before reviving that approach.
- Leaving this open means HTTP-only integrations remain dependent on workarounds such as session-key churn or asking the agent to interpret slash-command-like text.
