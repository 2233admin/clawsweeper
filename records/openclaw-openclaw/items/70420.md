## Summary

Keep open. Current main still separates channel/provider runtime health from persisted session rows, and the issue discussion's diagnosis matches the source. The remaining actionable gap is documentation: the live status/readiness paths exist, but the sessions CLI/tool docs still do not explicitly warn that `openclaw sessions`, `sessions.list`, or `sessions_list` are not provider socket-liveness checks.

## What This Changes

The issue asks OpenClaw to either recreate/refelect Discord session entries after a health-monitor reconnect or clarify that session listing is not a reliable Discord connection-health signal.

## Best Possible Solution

Land a narrow docs clarification that `openclaw sessions`, Gateway `sessions.list`, and the agent `sessions_list` tool enumerate persisted conversation/session-store rows, not provider socket or channel liveness. Keep connectivity guidance pointed at `openclaw channels status --probe`, `openclaw status --deep`, `openclaw health --verbose`, and readiness/channel-status surfaces; avoid synthetic session rows unless maintainers intentionally change the cross-channel runtime contract.

## Reproduction Assessment

Yes. The issue provides logs for the observed reconnect, and current-main source gives a high-confidence reproduction path: after a quiet Discord reconnect, runtime status can be connected while `sessions.list` still returns only persisted session-store rows that already exist.

## Solution Assessment

Yes. A docs-first fix is the narrowest maintainable solution because current code intentionally separates runtime liveness from session persistence; materializing heartbeat/session rows would be a broader cross-channel behavior change.
