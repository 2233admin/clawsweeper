## Summary

Keep open. Current main still matches the reported boundary: health-monitor reconnect restarts the channel runtime, Discord reports runtime/channel status, and `sessions.list`/`sessions_list` still enumerates persisted session-store rows rather than live channel socket state. The docs now point users toward live channel checks, but they still do not explicitly warn that `openclaw sessions`/`sessions_list` is not a provider-liveness signal, which is the false-alarm source reported here.

## Best Possible Solution

Land a narrow guidance fix: document that `openclaw sessions`, Gateway `sessions.list`, and the agent `sessions_list` tool list persisted conversation/session-store entries, not provider socket or channel liveness. Point heartbeat/connectivity checks at `openclaw channels status --probe`, `openclaw status --deep`, `openclaw health --verbose`, and readiness/channel-status surfaces. Do not create synthetic session rows unless maintainers intentionally choose that behavior for all affected channels.
