## Summary

Keep open. Current main and the latest tagged release still resolve an unset QMD embed timeout to 120000ms, pass that value directly to `qmd embed` for vector modes, and log timeout/backoff text without naming `memory.qmd.update.embedTimeoutMs`; the issue is valid and narrow enough for a focused repair.

## What This Changes

The issue requests a local-GGUF-friendly default for `memory.qmd.update.embedTimeoutMs` and clearer timeout diagnostics/docs for QMD vector embedding.

## Best Possible Solution

Implement the fix in QMD-owned surfaces: resolve a local-friendly unset embed timeout for `vsearch`/`query` GGUF embeddings, make timeout warnings name `memory.qmd.update.embedTimeoutMs` and the effective timeout, and add QMD troubleshooting/config docs for the exact timeout symptom and recommended values.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms and passes it to `qmd embed`; a slow or hung embed then logs only the raw timeout/backoff warning.

## Solution Assessment

Yes. The narrow maintainable repair is in QMD config resolution, embed-failure diagnostics, and QMD docs/tests; the separate scheduling and disable semantics tracked in #63626 and #73432 should stay out of this fix.
