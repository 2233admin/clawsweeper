## Summary

Keep open. Current main still resolves an unset QMD embed timeout to 120000ms, passes that value directly to `qmd embed` for vector-backed `vsearch`/`query` modes, and logs only the raw timeout/backoff without naming the config knob. The docs still document generic QMD slowness and search timeouts, but not the reported `qmd embed timed out after 120000ms` recovery path.

## What This Changes

The issue requests a higher or smarter default for `memory.qmd.update.embedTimeoutMs` on local GGUF vector embedding workloads, plus an actionable timeout message and troubleshooting docs.

## Best Possible Solution

Implement a focused QMD-owned fix: make the unset embed timeout local-GGUF-friendly for semantic QMD modes, add a timeout warning that names `memory.qmd.update.embedTimeoutMs` and the effective value, and document the symptom and recommended values in the QMD troubleshooting/config reference pages.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff if the subprocess exceeds that limit.

## Solution Assessment

Yes. The narrow maintainable solution is in QMD config resolution, QMD embed failure diagnostics, and QMD docs; the separate QMD scheduling issues in #63626 and #73432 should remain out of scope.
