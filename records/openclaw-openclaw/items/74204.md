## Summary

Keep open. Current main still resolves QMD embed timeout to 120000ms by default, uses that value for `qmd embed` in vector modes, and logs the raw timeout/backoff without naming the config knob that fixes it. The docs expose the setting only as a generic table row and still lack the requested QMD embed-timeout troubleshooting guidance.

## What This Changes

The issue requests a longer or mode-aware QMD embedding timeout default, an actionable timeout warning naming `memory.qmd.update.embedTimeoutMs`, and QMD troubleshooting documentation for local GGUF embedding timeouts.

## Best Possible Solution

Implement a focused QMD vector-embed fix in the memory plugin path: make the default timeout local-friendly for QMD vector modes or apply an equivalent default-aware policy, surface `memory.qmd.update.embedTimeoutMs` in timeout warnings, and document the timeout symptom and recommended values on the QMD docs/config pages.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and the resolved 120000ms value is passed to `qmd embed`; a hung embed produces the current raw timeout/backoff warning.

## Solution Assessment

Yes, the best fix should stay QMD-owned and narrow. Raising or adapting the QMD embed timeout default plus an explicit warning and docs update solves the reported local-GGUF failure without changing unrelated QMD scheduling issues tracked separately.
