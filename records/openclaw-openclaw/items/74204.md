## Summary

Keep open. Current main and the latest tagged release still resolve an unset QMD embed timeout to 120000ms, pass it directly to `qmd embed` for vector modes, and log the raw timeout/backoff without naming the `memory.qmd.update.embedTimeoutMs` recovery knob. The existing discussion correctly separates this timeout/diagnostic problem from the other QMD scheduling issues.

## What This Changes

The issue asks OpenClaw to make QMD vector embedding timeouts local-GGUF-friendly and make timeout logs/docs point users to `memory.qmd.update.embedTimeoutMs`.

## Best Possible Solution

Make a focused QMD-owned change: use a local-friendly effective timeout for vector embedding when the value is unset, add an actionable timeout warning that names `memory.qmd.update.embedTimeoutMs` and the effective value, and document the symptom and recommended values on the QMD docs/config pages.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure QMD with `searchMode: "query"` or `"vsearch"`, leave `memory.qmd.update.embedTimeoutMs` unset, and current main resolves 120000ms, passes it to `qmd embed`, then logs only the raw timeout/backoff warning if the process exceeds that budget.

## Solution Assessment

Yes. The narrow maintainable solution is in QMD config resolution, QMD embed failure diagnostics, and QMD docs; separate QMD scheduling/disable behavior should remain tracked outside this timeout issue.
