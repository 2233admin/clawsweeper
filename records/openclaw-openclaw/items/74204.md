## Summary

Keep open. Current main still resolves an unset QMD embed timeout to 120000ms, passes that value directly to `qmd embed` in semantic QMD modes, and logs the raw timeout/backoff without naming the `memory.qmd.update.embedTimeoutMs` recovery knob. The docs/config reference expose the knob but still lack a troubleshooting entry for the reported `qmd embed timed out after 120000ms` symptom.

## What This Changes

Request to make QMD semantic embedding timeouts local-GGUF friendly and surface `memory.qmd.update.embedTimeoutMs` in timeout diagnostics and docs.

## Best Possible Solution

Implement a focused QMD-owned fix: use a local-friendly effective default for unset semantic QMD embed runs, preserve explicit user overrides and BM25-only behavior, add an actionable timeout warning that names `memory.qmd.update.embedTimeoutMs`, and document the symptom in the QMD troubleshooting/config pages.

## Reproduction Assessment

Yes. Configure QMD with `searchMode: "query"` or `"vsearch"` and leave `memory.qmd.update.embedTimeoutMs` unset; current main resolves 120000ms, passes it to `qmd embed`, and logs only the raw timeout/backoff if the subprocess exceeds that budget.

## Solution Assessment

Yes. The narrow maintainable solution is in QMD config resolution, QMD embed failure diagnostics, and QMD docs; the separate embed scheduling/disable behavior tracked in #73432 and the closed #63626 should remain out of scope.
