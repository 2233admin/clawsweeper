## Summary

Keep open. Current main still leaves QMD vector/hybrid embedding runs on the 120000ms default and logs the raw timeout/backoff without naming the `memory.qmd.update.embedTimeoutMs` fix. The recent `search`-mode mitigation helps BM25-only setups, but it does not solve the reported `query`/local GGUF timeout path.

## What This Changes

The issue requests a higher or mode-aware QMD embed timeout default, an actionable timeout warning that names `memory.qmd.update.embedTimeoutMs`, and QMD troubleshooting docs for local GGUF embedding timeouts.

## Best Possible Solution

Implement a focused QMD-owned fix: give vector-backed QMD embed runs a local-friendly default timeout, make timeout failures point directly to `memory.qmd.update.embedTimeoutMs`, and add QMD troubleshooting/config docs. Keep the separate scheduling and disable-policy reports tracked separately.

## Reproduction Assessment

Yes. A static current-main check reproduces the reported code path: default config resolves `embedTimeoutMs` to 120000, vector modes pass that value into `qmd embed`, and timeout logging omits the config fix. I did not run a live GGUF embed because this review is read-only.

## Solution Assessment

Yes, the narrowest maintainable solution is in the QMD backend config/manager/docs surface rather than the builtin embedding engine. A mode-aware longer default is safer than changing unrelated QMD scheduling semantics.
