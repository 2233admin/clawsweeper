## Summary

Keep open. Current main still lacks this footer-status plugin API, and the PR is active with concrete demand from the linked ContextClaw use case, but the latest head still has merge-blocking contract and gate issues: the public SDK baseline is not included in the PR file set, status provider ids are validated with a trimmed value but stored unnormalized, and the required changelog entry is missing.

## What This Changes

The PR adds a public `PluginStatusProvider` type and `registerStatusProvider` API, stores providers in `PluginRegistry.statusProviders`, exports the type through `plugin-entry`, updates docs/test fixtures, and adds registry tests for duplicate and blank ids.

## Best Possible Solution

Settle the public footer-status contract, then land a complete supported seam with normalized provider ids, an updated generated SDK API baseline, a changelog entry, docs that match shipped behavior, and either the TUI footer consumer in the same change or a non-closing follow-up for the render path.

Not applicable as a feature PR. The review defects have high-confidence static reproduction paths: inspect the PR file list for missing generated baseline/changelog files, add a padded-then-canonical status-provider registration case, and run `pnpm plugin-sdk:api:check` plus the PR changelog gate after updates.

## Solution Assessment

No, not as currently patched. A narrow plugin API seam is plausible for the linked footer-status request, but the implementation is not ready until the SDK baseline, changelog, normalized-id behavior, and maintainer-owned API/render boundary are resolved.
