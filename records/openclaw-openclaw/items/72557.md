## Summary

Keep this PR open. Current main still has no `PluginStatusProvider`, `registerStatusProvider`, or `statusProviders` surface, and the ContextClaw use case plus linked #64086 make the requested seam concrete. The PR is a plausible narrow SDK/API direction, but this head is not merge-ready because it misses the current SDK test helper, generated API baseline, normalized id storage, and required changelog entry, and the new public footer-status contract still needs maintainer acceptance because the TUI consumer is deliberately left for later.

## What This Changes

The PR adds a public `PluginStatusProvider` and `registerStatusProvider` plugin API, stores registered status providers in `PluginRegistry.statusProviders`, exports the type through the plugin SDK, documents the registration method, and adds registry tests for duplicate and blank provider ids.

## Best Possible Solution

Land this only after maintainers accept the public footer-status API shape, then update the current SDK test helper, normalize stored provider ids, regenerate the plugin SDK API baseline, add the required changelog entry, and keep the TUI render path either in this PR or in a clearly linked follow-up for #64086/#53774.

## Reproduction Assessment

Not applicable as a feature PR. The merge blockers have high-confidence static reproduction paths: typecheck with the current `src/plugin-sdk/plugin-test-api.ts`, run `pnpm plugin-sdk:api:check`, inspect duplicate registration after a padded id, and run the PR merge gate that enforces `CHANGELOG.md` for production SDK changes.

## Solution Assessment
