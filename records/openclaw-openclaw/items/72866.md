## Summary

Keep this PR open. Current main still has the reset-branch asymmetry the PR targets, and the one-line production change is the right narrow fix, but the added regression test still does not enter the bare reset branch because it sets `resetTriggered` instead of the fields that drive `isBareSessionReset`.

## What This Changes

The PR adds `inboundUserContext` to the bare session reset prompt assembly and adds a focused test file for inbound history on reset and non-reset reply paths.

## Best Possible Solution

Keep the production one-line fix, update the test fixture to force the real bare reset branch with `command.softResetTriggered: true` or a true new-session bare command setup, then run the targeted reply tests before maintainer merge review.

## Reproduction Assessment

Yes. A source-level reproduction is a reply turn where `command.softResetTriggered` is true and `sessionCtx.InboundHistory` is the only non-empty user input: current main lets the guard pass through `hasInboundHistoryBody` but builds `effectiveBaseBody` from a reset `baseBodyForPrompt` that omits `inboundUserContext`.

## Solution Assessment

Partly. The production diff is the narrowest maintainable fix, but the proposed test is not the best verification until it drives the actual `isBareSessionReset` condition rather than only setting `resetTriggered`.
