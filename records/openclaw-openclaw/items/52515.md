## Summary

Keep open. The PR addresses a real unresolved doctor false-positive path on current main, but the proposed patch still has blocking cleanup-hint regressions and changes security-sensitive exported SecretRef behavior that needs maintainer/secops review before merge.

## What This Changes

The PR changes doctor gateway cleanup hints to render from detected extra services, wraps doctor execution in an env flag that downgrades unresolved exec SecretRefs to warnings, and updates generated plugin SDK API baselines.

## Best Possible Solution

Revise the patch so cleanup hints are rendered from actual detected service metadata in both doctor and status, user-vs-system service scope is honored, skipped legacy cleanup still leaves actionable manual guidance, and doctor SecretRef warnings use an explicit inspection path while strict exported helpers remain strict for runtime and plugin callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Keep status cleanup hints wired to detected services:** `src/daemon/inspect.ts:28-30`
  - body: `renderGatewayServiceCleanupHints()` now returns `[]` when called without a service list, but `status.print.ts` still calls it with no arguments after reporting `extraServices`. That makes `openclaw status --deep` report extra gateway-like services without any cleanup hints, regressing that troubleshooting path.
  - confidence: 0.93
- **[P2] Preserve manual hints for unremoved legacy services:** `src/commands/doctor-gateway-services.ts:441-442`
  - body: Filtering out every `svc.legacy` before rendering hints drops the only manual cleanup guidance when legacy cleanup is skipped by policy, declined, fails, or is system-scoped. In those paths doctor still reports the legacy service but no longer gives the user a command to remove it.
  - confidence: 0.88
- **[P2] Honor service scope in cleanup commands:** `src/daemon/inspect.ts:38-46`
  - body: The new helper receives `svc.scope` but always emits `launchctl bootout gui/$UID/...` and `systemctl --user ...`. For `doctor --deep` findings from `/Library/LaunchDaemons` or system systemd directories, those commands target the wrong service manager and leave the detected system service installed.
  - confidence: 0.9
