## Summary

Keep open. The linked bug remains valid on current main, but this PR is not merge-ready: the existing review blockers still check out, the SecretRef change crosses a CODEOWNERS-protected security surface, and current main has since moved doctor execution into flow files so the branch needs a real rebase/revision rather than cleanup closure.

## What This Changes

The PR changes doctor gateway cleanup hints to render from detected extra services, wraps doctor execution in a process-wide SecretRef leniency flag, and updates generated plugin SDK API baselines for the new SecretRef helper parameter.

## Best Possible Solution

Revise the fix on current main so doctor and status render cleanup hints from actual detected service metadata, cleanup commands honor user versus system scope, unremoved legacy services retain actionable manual guidance, and exec SecretRefs are handled through explicit read-only inspection/gateway-resolution paths while strict exported helpers stay strict for runtime and plugin callers.

## Reproduction Assessment

Yes. The patch regressions are reproducible by inspection: the PR helper returns no hints without an argument while current status still calls it with none, legacy services are filtered before fallback hints, and system-scoped services carry scope metadata that the PR command renderer ignores.

## Solution Assessment

No. The direction addresses a real bug, but this is not the best implementation because it misses the status call site, drops legacy fallback guidance, emits user-scope commands for system-scope services, and weakens exported SecretRef strictness through process-wide state.

  - confidence: 0.88

  - body: The helper receives `svc.scope`, but the new launchd and systemd commands always target the user manager. For `doctor --deep` findings from LaunchDaemons or system systemd directories, the printed commands target the wrong manager and leave the detected service installed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies or workflows, but it changes exported SecretRef resolution behavior through a process-wide doctor leniency flag.

Concerns:
