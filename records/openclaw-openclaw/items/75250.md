## Summary

Keep open. Current main has partial mitigations for stale service paths and bundled plugin runtime-deps repair, but it still intentionally uses an external macOS Node/global CLI layout and does not make Homebrew Node, the OpenClaw package, launchd service command, plugin requirements, and runtime-deps cache an atomic update unit.

## What This Changes

The issue reports macOS update breakage when Homebrew-managed Node, a global OpenClaw package, launchd service arguments, plugin host-version requirements, and plugin-runtime-deps caches drift after `brew upgrade` or OpenClaw updates.

## Best Possible Solution

Maintainers should choose the macOS Gateway runtime authority first: either document and harden the external Homebrew/npm layout with explicit doctor/status diagnostics and recovery, or move the launchd service to an OpenClaw-managed runtime/package layout. In either path, update/doctor should report and validate CLI Node, service Node, package root, host version, plugin requirements, and runtime-deps root/build key before startup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The issue includes concrete macOS/Homebrew logs that match current code paths, but I did not find a deterministic repo-local reproduction for the full Homebrew ABI churn plus service/package/plugin-cache skew sequence.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A bundled/pinned runtime may solve the broadest class, but that is a maintainer platform decision; a safer first step would be approved diagnostics and runtime-deps/service validation around the chosen policy.
