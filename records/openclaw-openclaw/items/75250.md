## Summary

Keep open. Current main has partial mitigations for package-manager updates, Homebrew Cellar path stability, plugin min-host skew, and bundled runtime-deps cache cleanup, but macOS local mode still intentionally relies on an external global CLI plus Homebrew/system Node rather than an OpenClaw-owned atomic runtime. The open #74630 overlaps the broader runtime/bootstrap regression cluster but does not fully cover this Homebrew native ABI and LaunchAgent/package/runtime drift report.

## What This Changes

The issue asks OpenClaw to keep the macOS service and CLI working across Homebrew/OpenClaw updates by preventing or repairing drift between Node, the global OpenClaw package, plugin SDK files, plugin runtime-deps cache, and the LaunchAgent runtime path.

## Best Possible Solution

Maintainers should first decide the macOS Gateway runtime authority. If OpenClaw continues to support Homebrew/global npm local mode, doctor/status/update should explicitly validate and repair the CLI Node, service Node, package root/version, LaunchAgent command, plugin min-host compatibility, and runtime-deps stage root. If OpenClaw should own the service runtime, the macOS installer/service manager should migrate toward a pinned managed runtime/package layout with clear upgrade semantics.

## Reproduction Assessment

No high-confidence repo-local reproduction is available. The issue includes concrete macOS/Homebrew logs that map to current service/runtime/plugin-cache code paths, but reproducing the full failure requires a macOS Homebrew update sequence or a targeted simulation of Node ABI breakage plus stale service/package/cache state.

## Solution Assessment

Unclear. A bundled or pinned service runtime may solve the broadest failure class, but that is a platform ownership decision; a narrower maintainable first step would be maintainer-approved diagnostics and repair behavior around whichever runtime policy is chosen.
