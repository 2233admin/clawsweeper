## Summary

Keep open. Current main contains several relevant mitigations, but it still intentionally runs macOS local mode through an external Node/global CLI launchd layout, so the Homebrew-managed Node/OpenClaw package/service/plugin-cache stack is not updated atomically and the central reliability concern remains.

## What This Changes

The issue requests a robust macOS update/runtime model that prevents drift between Homebrew Node, the global OpenClaw package, LaunchAgent runtime paths, plugin SDK files, and bundled plugin runtime-deps cache.

## Best Possible Solution

Maintainers should decide the macOS gateway runtime authority first. If external Homebrew/global npm remains supported, doctor/status/update should explicitly validate CLI Node, service Node, OpenClaw package root/version, LaunchAgent command, plugin min-host requirements, and runtime-deps stage key, then repair or prune stale state before startup; if OpenClaw should own the service runtime, the installer/service manager should move toward a pinned managed runtime/package layout.

## Reproduction Assessment

No. The issue includes concrete macOS/Homebrew logs that match current service and plugin-cache code paths, but the full failure requires Homebrew ABI churn and mixed installed service state that I did not reproduce from the repository alone.

## Solution Assessment

Unclear. The existing main-branch mitigations are narrow and appropriate, but the requested atomic runtime fix is a platform ownership decision; a safer first step is maintainer-approved diagnostics around the chosen runtime policy.
