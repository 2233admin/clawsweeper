## Summary

Keep open. Current main has mitigations for some reported symptoms, especially service entrypoint stability and plugin-runtime-deps pruning/repair, but macOS still intentionally runs from an external Node/global CLI layout and does not make Homebrew Node, the OpenClaw package, service launch path, and plugin runtime cache an atomic unit. The open #74630 overlaps the runtime-deps regression family, but it does not fully cover this Homebrew service/runtime drift report.

## What This Changes

The issue asks OpenClaw to survive Homebrew/OpenClaw update cycles without CLI, gateway service, host/plugin version, SDK, native ABI, or plugin runtime cache drift.

## Best Possible Solution

Maintainers should decide the macOS gateway runtime authority first: either keep documenting an external Homebrew/npm-managed runtime with stronger doctor/status diagnostics and recovery, or move the service to an OpenClaw-managed runtime/package layout. In either direction, doctor/update should clearly report CLI Node, service Node, package root, host version, plugin requirements, and runtime-deps root/build key, then validate or prune stale plugin-runtime-deps before startup.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The issue includes concrete macOS/Homebrew logs that match current code paths, but I did not find a deterministic repo-local reproduction for the full brew ABI churn plus service/package/plugin-cache skew sequence.

## Solution Assessment

Is this the best way to solve the issue? Unclear. A bundled/pinned runtime may solve the broadest class, but that is a platform/product decision; a narrower first step would be maintainer-approved diagnostics and cache/service validation around the chosen runtime policy.
