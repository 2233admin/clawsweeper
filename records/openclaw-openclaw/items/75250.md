## Summary

Keep open. Current main partially addresses the plugin runtime cache side of the report, but it does not close the broader macOS/Homebrew failure class: the published CLI still starts through `env node`, the macOS docs still rely on an external global CLI plus external Node, and the reported failure can happen before OpenClaw JavaScript can run a doctor/update repair.

## What This Changes

The issue reports OpenClaw becoming unavailable after macOS Homebrew upgrades because the CLI, launchd gateway service, host package, plugin SDK, and plugin runtime dependency cache can drift across different Homebrew-managed Node/OpenClaw versions.

## Best Possible Solution

Define one supported macOS package/runtime contract and make the installer, updater, launchd service, and plugin runtime-deps cache enforce it consistently: the CLI and service should use compatible runtime/package roots, updates should refresh service metadata and plugin deps atomically, and diagnostics should clearly surface external Homebrew Node ABI breakage when OpenClaw cannot repair it itself.

## Reproduction Assessment

Yes for triage, though not locally in this Linux checkout. The issue includes concrete macOS/Homebrew commands, launchd arguments, versions, dyld output, plugin version-skew logs, and runtime-deps cache paths; current source still shows the CLI depends on `env node` before OpenClaw repair code can run.

## Solution Assessment

Unclear. Current main contains useful partial fixes for service path and plugin runtime-deps drift, but the requested end-to-end resilience needs a maintainer decision on whether macOS remains external-Node based or gets a more controlled launcher/runtime layout.
