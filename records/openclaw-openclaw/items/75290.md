## Summary

Keep open. Current main still follows the reported hot path: embedded attempts build the full OpenClaw coding tool set before the `core-plugin-tools` mark, and plugin tool resolution still calls each registered plugin tool factory during resolution. The report is recent, concrete, and not superseded by the latest release or a current-main source change in this area.

## What This Changes

The issue asks OpenClaw to investigate and reduce repeated `core-plugin-tools` embedded-run startup cost, with possible fixes including context-safe factory memoization, a plugin-tool opt-out, and per-plugin timing marks.

## Best Possible Solution

Keep this open and fix the hot path with a measured, context-safe change: add per-plugin factory timing, provide an explicit embedded-run or agent policy path to skip plugin tool resolution when plugin tools are not needed, and only introduce memoization after separating reusable tool definitions from per-run wrappers or proving a complete cache key.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for triage: the issue includes repeated prep-stage traces and a `disablePluginTools: true` control case, and current source still follows the reported per-resolution factory path. I did not run a live replay in this read-only review.

## Solution Assessment

Is this the best way to solve the issue? No final solution exists on current main. The reporter's directions are plausible, but raw `(pluginId, sandboxKey)` memoization is not the safest first fix; instrumentation plus an explicit plugin-tool skip path should come before any context-sensitive caching.
