## Summary

Keep open. Current main still matches the reported hot path: embedded runs construct the full OpenClaw coding tool set before the `core-plugin-tools` stage mark, and plugin tool resolution still calls every registered factory during each resolution pass. The report is recent, concrete, and not superseded by the latest release or a current-main source change in this surface.

## What This Changes

The issue reports that embedded runs spend about 10-12 seconds in the `core-plugin-tools` prep stage even with a cached plugin registry, and asks whether this is expected, memoizable, configurable, and better instrumented per plugin.

## Best Possible Solution

Keep this open and fix the hot path with a measured, context-safe approach: add per-plugin timing around plugin tool factories, thread an explicit embedded-run or policy-derived plugin-tool opt-out into tool construction, and only memoize factory outputs if the cache key covers all context-sensitive fields or the factory contract is split into reusable definition plus per-run wrapper.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the field report: the issue includes repeated `core-plugin-tools` traces and a control case where `disablePluginTools: true` drops the stage to about 0 ms. This review did not run a live replay, but current source still follows the reported factory-invocation path.

## Solution Assessment

Is this the best way to solve the issue? No current-main solution exists yet. The reporter's direction is plausible, but a raw `(pluginId, sandboxKey)` memo is not enough by itself; the safer path is per-plugin instrumentation plus an explicit plugin-tool skip path, then context-safe memoization if profiling supports it.

## Repair Work Prompt

Fix the embedded-run `core-plugin-tools` latency reported in https://github.com/openclaw/openclaw/issues/75290. Keep the PR narrow. Current main still calls `createOpenClawCodingTools(...)` before the `core-plugin-tools` mark, and `resolvePluginTools` invokes every plugin tool factory on each resolution pass. Add per-plugin factory timing diagnostics for slow plugin tool resolution so logs can identify the heavy plugin. Add a safe plugin-tool skip path for embedded runs that need only core tools, preferably by threading an explicit `disablePluginTools` option through `RunEmbeddedPiAgentParams`/`EmbeddedRunAttemptParams` into `createOpenClawCodingTools`/`createOpenClawTools`, and/or by deriving a skip when effective tool policy explicitly denies or excludes plugin tools. Do not add owner-specific plugin ids in core. Do not globally memoize factory outputs unless the cache key covers config/runtime config, workspace, agent, session key, session id, sandbox, delivery context, sender ownership, and abort/runtime semantics, with regression tests proving isolation. Update public docs only if a user-facing config knob or documented behavior changes. Add a single-line changelog fix entry if runtime behavior changes; diagnostics-only/test-only changes do not need one. Validate focused plugin-tool, tool-policy, embedded-run parameter, and gateway HTTP disablePluginTools tests, then run `pnpm check:changed` in Testbox if promoted.
