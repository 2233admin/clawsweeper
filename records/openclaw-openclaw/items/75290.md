## Summary

Keep open. Current main still matches the reported hot path: embedded runs build the full coding tool set inside the `core-plugin-tools` prep bucket, and plugin tool resolution still invokes each registered factory during that resolution pass. The report is recent, source-backed, and not superseded by a current-main change or the latest release context provided.

## What This Changes

The issue requests investigation and remediation for repeated ~10-12 second `core-plugin-tools` prep latency on embedded runs, likely caused by per-run plugin tool factory resolution despite a cached registry.

## Best Possible Solution

Fix the hot path with measured, context-safe changes: add per-plugin timing around plugin tool factory resolution, thread an explicit embedded-run or policy-derived plugin-tool skip path for agents that do not need plugin tools, and only add memoization after proving the cache key or factory contract preserves per-run isolation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the field report: it includes repeated `core-plugin-tools` traces and a control case where `disablePluginTools: true` drops the stage to about 0 ms. This review did not run a live replay, but current source still follows the reported per-resolution factory path.

## Solution Assessment

Is this the best way to solve the issue? The issue should stay open, but raw `(pluginId, sandboxKey)` memoization is not yet proven safe because the factory context is run-sensitive. The better fix path is instrumentation plus an explicit skip path, followed by context-safe memoization only if profiling supports it.
