## Summary

Keep open. Current main has adjacent tool/provider deferrals after v2026.4.29, but the embedded run path still constructs the full coding tool set before allowlist filtering and `resolvePluginTools` still invokes each registered plugin factory during each resolution pass. The report is recent, trace-backed, and corroborated by a second 2026.4.29 data point.

## What This Changes

Investigate and reduce the repeated embedded-run `core-plugin-tools` prep cost by avoiding unnecessary per-run plugin tool factory work, adding opt-out/diagnostics, and assessing context-safe memoization.

## Best Possible Solution

Keep this open and land a measured hot-path repair: add per-plugin timing around plugin tool factories, skip plugin tool resolution when policy or run configuration excludes plugin tools, expose a narrow agent/run-level plugin-tool opt-out if maintainers want that surface, and only memoize through a context-safe contract split or fully justified cache key.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the reported behavior: the issue includes repeated `core-plugin-tools` traces plus a control where `disablePluginTools: true` drops the stage to about 0 ms, and current source still follows the reported factory-invocation path. The exact current-main latency magnitude remains unverified after adjacent deferral commits.

## Solution Assessment

Is this the best way to solve the issue? No current-main solution exists yet. The reporter's memoization direction is plausible, but a raw `(pluginId, sandboxKey)` cache is too narrow without context-isolation proof; instrumentation plus an explicit skip path is the safer first repair.
