## Summary

Keep open. Current main still only partially addresses the report: repeated-tool loop detection exists but is opt-in by default, and the shared edit parameter validation path still accepts `oldText` and `newText` when they are identical, so the specific no-op edit loop can still reach the upstream edit tool. Related prior cleanup covers unknown-tool and generic loop detection, but it does not remove this edit-specific no-op gap.

## What This Changes

The issue requests pre-validation for no-op edit replacements plus broader repeated tool-failure circuit breaking, error classification, and agent behavior guidance to reduce wasted retries.

## Best Possible Solution

Add an OpenClaw-owned validation guard that rejects edit replacements where `oldText` and `newText` are identical before the upstream edit tool executes, with a clear non-retryable message and focused host/sandbox coverage. Keep the broader circuit-breaker default policy unchanged unless maintainers separately decide to make loop detection default-on.

## Reproduction Assessment

Yes. A focused source-level reproduction is to call the shared edit validation path with `edits: [{ oldText: "same", newText: "same" }]`; current main accepts that payload and the host/sandbox edit factories then delegate to the wrapped base edit tool. I kept this review read-only and did not execute a mutating edit.

## Solution Assessment

No. Current main is not yet the best solution because it relies on opt-in loop detection and upstream edit behavior instead of rejecting the non-retryable no-op edit at OpenClaw's boundary; the narrow validation guard is the maintainable first fix.
