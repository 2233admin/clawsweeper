## Summary

Keep open. Current main still has only partial safeguards: OpenClaw validates edit payload shape but does not reject identical `oldText`/`newText` before delegating to the base edit tool, and the broader tool-loop circuit breaker remains disabled by default.

## What This Changes

The issue requests edit-tool pre-validation for no-op replacements, repeated tool-error circuit breaking, retryability classification, and agent/tool error-handling guidance.

## Best Possible Solution

Add an OpenClaw-owned edit guard that rejects replacements whose `oldText` and `newText` are identical before the base edit tool runs, with a clear non-retryable message and focused tests. Leave default loop-breaker thresholds and always-on policy to a separate maintainer decision.

## Reproduction Assessment

Yes. A focused test can call the OpenClaw edit validation path with `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes OpenClaw validation and reaches the dependency. I did not execute a mutating edit because this review was read-only.

## Solution Assessment

Partly. The requested edit pre-validation is the narrowest maintainable first fix, but enabling a broad default circuit breaker or retry-classification policy should be handled separately after maintainer review.
