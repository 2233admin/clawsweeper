## Summary

Keep open. Current main still only partially addresses the report: the broader loop detector exists as opt-in behavior, but the edit path still lacks an OpenClaw-owned pre-validation guard for identical `oldText` and `newText`, so the reported no-op edit loop remains a valid narrow cleanup target.

## What This Changes

Request to add edit-tool no-op pre-validation for identical replacements, repeated tool-error circuit breaking, retryability classification, and agent error-handling guidance.

## Best Possible Solution

Add a shared edit validation guard that rejects identical replacement text with a clear no-op/non-retryable error before the base edit tool runs, with focused host and sandbox coverage; keep global loop detection opt-in unless maintainers separately choose to change default policy.

## Reproduction Assessment

Yes. A focused unit path can wrap a dummy edit tool with `REQUIRED_PARAM_GROUPS.edit` or use the host/sandbox edit factory and pass `edits: [{ oldText: "same", newText: "same" }]`; current source shows that payload passes OpenClaw validation and reaches the base edit tool. I did not execute a mutating edit during this read-only review.

## Solution Assessment

No, current main is not the best completed solution because it relies on the upstream edit tool's generic no-change rejection instead of rejecting identical params before execution. The narrow maintainable fix is edit-specific pre-validation; changing global circuit-breaker defaults is not needed for that repair.
