## Summary

Keep open. Current main has useful partial safeguards, but the central no-op edit retry gap remains: OpenClaw validates `oldText` and `newText` types, yet does not reject identical replacement text before delegating to the edit tool, and the broader loop detector remains opt-in by default.

## What This Changes

The issue requests edit-tool pre-validation for identical `oldText`/`newText`, clearer non-retryable edit errors, and a broader repeated-tool-error circuit breaker.

## Best Possible Solution

Add a focused OpenClaw edit guard that rejects any replacement where `oldText` and `newText` are identical before the base edit tool runs, with a clear non-retryable error and focused tests; leave default circuit-breaker policy for separate maintainer judgment.

## Reproduction Assessment

Yes. A source-level reproduction is clear: pass `edits: [{ oldText: "same", newText: "same" }]` through the current edit parameter validation path and it satisfies `REQUIRED_PARAM_GROUPS.edit`; I did not execute a mutating edit because this review is read-only.

## Solution Assessment

No, current main is only a partial solution. The narrowest maintainable fix is edit-specific pre-validation in OpenClaw’s edit parameter/recovery path, while the broader default circuit-breaker thresholds need product review.
