## Summary

Keep open. Current main still treats all-model API limit/capacity exhaustion as a visible terminal reply after fallback, not as a durable waiting prompt that is replayed when capacity returns. Existing queues are either in-process followup/command queues or a narrower durable restart-continuation delivery queue, and related open work does not clearly supersede this issue.

## Best Possible Solution

Keep this issue open. The best path is a core durable pending-turn queue for model-capacity/API-limit exhaustion, coordinated with #62540 and the open pending-inbound/followup persistence PRs. It should persist the original prompt, route, status, retry metadata, and idempotency key; expose a waiting state; schedule cooldown-aware replay; and include restart/crash recovery tests that prove no duplicate final replies are emitted.
