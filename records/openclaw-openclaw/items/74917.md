## Summary

Keep open. Current main still matches the reported mismatch: the embedded runner resolves `contextTokens` into `contextTokenBudget`, but the non-context-engine tool-result guard still derives its limits from `params.model.contextWindow`/`maxTokens`. The issue is also paired with two open closing PRs, so it should remain open until one fix is reviewed and merged.

## What This Changes

The issue reports that the embedded runner's non-context-engine tool-result guard enforces thresholds from `model.contextWindow` instead of the resolved runtime `contextTokenBudget` when `contextTokens` is larger.

## Best Possible Solution

Make every runtime context-limit path consume the same resolved budget, preferably by passing `params.contextTokenBudget` into the non-context-engine tool-result guard and adding a focused regression test for `contextTokens > contextWindow`; then land one canonical PR and let it close the issue.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction can configure a model with `contextWindow` lower than the resolved `contextTokenBudget`, run the non-context-engine attempt path, and assert the guard is installed with the lower model value today; no live provider credentials are needed for that proof.

## Solution Assessment

Yes, with a policy caveat. If OpenClaw treats `contextTokens` as the effective runtime budget, using `params.contextTokenBudget` in the guard is the narrowest maintainable fix; if larger-than-native budgets should be invalid, central resolver validation or clamping is safer than preserving divergent enforcement.
