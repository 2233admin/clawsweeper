## Summary

Keep open. Current main still has the reported runtime-budget mismatch, and the issue is already paired with an open closing PR that should be reviewed or replaced rather than pre-closing the issue.

## What This Changes

The issue asks for the non-context-engine tool-result guard to enforce the same resolved runtime context budget used by the embedded runner, instead of deriving its threshold from a smaller `model.contextWindow`.

## Best Possible Solution

Review and land #74934, or an equivalent narrow fix, with focused regression coverage proving the guard receives the resolved `contextTokenBudget` when it differs from `model.contextWindow` while preserving the existing fallback behavior for unresolved callers.

## Reproduction Assessment

Yes. A source-level reproduction can construct an embedded attempt with `contextTokenBudget` larger than `model.contextWindow` and assert the current non-context-engine guard is installed with the smaller `contextWindow` value.

## Solution Assessment

Yes, assuming OpenClaw continues treating `contextTokens` as the effective runtime cap. Passing `params.contextTokenBudget` into the guard before the existing fallbacks is the narrowest maintainable fix; metadata validation would be the alternative only if maintainers want to forbid this configuration.
