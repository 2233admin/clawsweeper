## Summary

Keep open. Current main still routes the affected Gemini CLI OAuth setup to the bundled Google plugin's `google-gemini-cli` CLI backend, OAuth setup still selects that runtime, and CLI turns still spawn the external `gemini` child process. I found no current-main evidence that the reported fixed per-turn startup/auth delay has been fixed, and the related reports cover adjacent but different Gemini CLI OAuth/setup issues rather than this latency regression.

## Best Possible Solution

Keep this issue open for maintainer triage. The best path is to measure standalone `gemini` startup/auth latency versus OpenClaw wrapper overhead on an affected macOS OAuth setup, then either document or doctor a Gemini CLI Keychain remediation, add diagnostics that clearly flag slow external CLI startup/auth, or implement a supported mitigation that avoids repeated per-turn Gemini CLI credential initialization where the Gemini CLI/API contract allows it.
