## Summary

Keep open. Current main still routes Gemini CLI OAuth to the bundled Google plugin's `google-gemini-cli` CLI backend, OAuth setup still selects that runtime, and each turn still invokes the external `gemini` command through the generic CLI runner. I found no current-main fix or canonical merged follow-up for the reported fixed per-turn startup/auth latency; the related reports cover adjacent setup, proxy, multi-account, or older architecture issues rather than this specific latency regression.

## Best Possible Solution

Keep this issue open for maintainer triage. The best path is to measure standalone `gemini --prompt` startup/auth latency versus OpenClaw wrapper overhead on an affected macOS OAuth setup, then either document/doctor a Gemini CLI Keychain remediation, add diagnostics that clearly identify slow external CLI startup/auth, or implement a supported mitigation that avoids repeated per-turn Gemini CLI credential initialization where the Gemini CLI contract allows it.
