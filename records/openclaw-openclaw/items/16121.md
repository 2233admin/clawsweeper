## Summary

Keep this issue open. Current main still does not provide first-class Telegram custom emoji support for outgoing messages, captions, or replies: the shared Markdown IR has no custom-emoji span, Telegram formatting renders the existing safe HTML subset only, and the send/delivery paths still use HTML parse mode without an `entities` or `caption_entities` surface. The related open PR #47562 is about custom emoji reactions, not message/caption rendering, and closed-unmerged PR #13905 remains useful prior art rather than a landed fix.

## What This Changes

Request support for Telegram custom emojis in messages, captions, and replies, including custom emoji identifiers and rendering according to Telegram Bot API formatting semantics.

## Best Possible Solution

Define the representation first, then implement validated Telegram custom emoji entities through the Telegram plugin send, caption, reply, edit, delivery, and streaming paths plus the shared message tool, with focused docs and regression tests that preserve the current HTML fallback behavior.
