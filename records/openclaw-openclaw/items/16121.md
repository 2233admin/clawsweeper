## Summary

Keep this issue open. Current main still lacks first-class Telegram custom emoji support for outgoing messages/captions/tool actions/edits/streaming previews: the shared Markdown IR has no custom emoji span, the Telegram renderer emits only HTML parse-mode markup, and the send/action paths do not accept or forward Telegram `entities`/`caption_entities`. Related #40848 remains open for a narrower entities surface, and PR #13905 was closed unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep this issue open and implement the capability in the bundled Telegram plugin after maintainer API-shape review. The likely direction is a Telegram-owned validated entity/custom-emoji surface, or a safe Markdown/IR custom emoji span, that threads through text, captions, message-tool sends/edits, replies/quotes, preview streaming, docs, and tests while preserving existing HTML/plain-text fallback behavior and coordinating with #40848 and #10944.
