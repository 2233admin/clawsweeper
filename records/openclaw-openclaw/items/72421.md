## Summary

Keep this issue open. Current main still has only partial stale-lane recovery: it can release stale bookkeeping when no active work is visible, but it intentionally preserves active-looking session lanes, heartbeat still defers behind a busy session lane, and the requested `status --lanes` diagnostic surface is absent.

## What This Changes

The issue reports intermittent Discord channel session lanes staying held after a run completes, causing later messages to wait behind a stale lane until manual session cleanup and gateway restart.

## Best Possible Solution

Keep this as a core session-lane reliability item until OpenClaw can distinguish stale queue bookkeeping from real active work, recover or abort stale session lanes without corrupting legitimate turns, cover the Discord cron/heartbeat path in tests, and expose read-only lane state for operators.

## Reproduction Assessment

No, there is not a high-confidence deterministic reproduction for the exact Discord stall. The reporter provided concrete long-run observations, and current source/tests prove a plausible active-lane path that recovery intentionally keeps, but no focused test reproduces the terminal cron/heartbeat race end to end.

## Solution Assessment

No, current main is not the complete fix. The partial queue timeout and stuck-session recovery work is useful, but the maintainable solution needs liveness-aware lane recovery and a diagnostic surface rather than blind auto-release of active-looking work.
