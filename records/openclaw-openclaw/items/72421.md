## Summary

Keep open. Current main has conservative stuck-session diagnostics and recovery, but it still deliberately skips active embedded runs, active reply operations, and active session-lane tasks, which leaves the reported stale active-lane shape unresolved. The requested lane diagnostic surface is also still absent.

## What This Changes

The issue asks OpenClaw to diagnose and recover Discord per-session lanes that remain held after cron, heartbeat, or system-event runs complete.

## Best Possible Solution

Keep this as a specific Discord/session-lane reproduction while maintainers review or replace the active stuck-session recovery work from #73243 or an equivalent focused patch. The end state should include lane task metadata tied to session/run/LLM liveness, safe abort or force-clear semantics, regression coverage for Discord cron/heartbeat session keys, and a read-only lane diagnostic surface.
