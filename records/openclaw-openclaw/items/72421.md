## Summary

Keep this issue open. Current main contains useful partial fixes for stale session-lane recovery and embedded-run lane timeouts, but the reported short stale Discord lane symptom is not fully resolved: diagnostic recovery still keeps a lane when the command queue reports an active lane task, default agent timeouts can keep that task active for up to 48 hours, heartbeat still skips the same session lane while it is busy, and `openclaw status --lanes` is not implemented.

## What This Changes

The issue requests reliable release/recovery for Discord session lanes that remain held after cron, heartbeat, or system-event runs complete, plus lane-state diagnostics for operators.

## Best Possible Solution

Keep this as a core session-lane reliability item. The desired end state is liveness-aware recovery that can distinguish a genuinely active run from stale queue bookkeeping, regression coverage for cron/heartbeat/system-event completion on a Discord session key, and a read-only lane-state diagnostic surface for operators.

## Reproduction Assessment

No, we do not have a high-confidence deterministic reproduction. The reporter provided concrete long-run observations and current source confirms plausible busy-lane behavior, but there is no focused unit or live repro proving the exact stale terminal path.

## Solution Assessment

No, current main is not yet the best complete solution. The partial timeout and stuck-session recovery work is useful groundwork, but the narrow maintainable fix still needs liveness metadata or instrumentation before auto-releasing active-looking session lanes.
