## Summary

Keep open. Current main has useful partial protection for command-lane task timeouts and stuck-session diagnostics, but it still preserves active-looking session-lane tasks, heartbeat still defers when the target session lane is busy, and the requested `status --lanes` diagnostic surface is not implemented.

## What This Changes

The issue requests reliable recovery and operator diagnostics for Discord per-session lanes that remain held after cron, heartbeat, or system-event runs complete.

## Best Possible Solution

Keep this as a core session-lane reliability issue. The maintainable end state is liveness-aware recovery that distinguishes real active LLM/tool/reply work from stale queue bookkeeping, plus focused regression coverage for cron/heartbeat/system-event completion on channel session keys and a read-only lane diagnostic surface for operators.

## Reproduction Assessment

No, we do not have a high-confidence reproduction path yet. The reporter provided concrete long-run observations and source inspection confirms the relevant stale-lane guard paths, but there is no deterministic unit or live repro for the exact missed-release branch.

## Solution Assessment

No, current main is not the complete solution. The existing recovery is deliberately conservative and useful, but it does not yet solve stale active lane bookkeeping or expose the requested lane-state diagnostics.
