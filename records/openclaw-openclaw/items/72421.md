## Summary

Keep this issue open. Current main has partial queue timeout and stuck-session recovery plumbing, but the central reported shape still remains: active-looking session lanes are intentionally kept warning-only, the outer per-session lane is not given the embedded-run task timeout, heartbeat still skips behind a busy target session lane, and there is no `openclaw status --lanes` inspection surface.

## What This Changes

The issue asks for reliable recovery and diagnostics for Discord session lanes that remain held after a run appears complete, especially after cron, heartbeat, or system-event delivery paths.

## Best Possible Solution

Add liveness-aware recovery that can distinguish real active work from stale session-lane bookkeeping, abort or release only when the work is no longer live, cover the Discord cron/heartbeat delivery shape with focused tests, and expose read-only lane snapshots through a supported diagnostic command.

## Reproduction Assessment

No. The reporter provided credible long-run observations and current source/tests show a plausible active-lane state, but there is no high-confidence deterministic reproduction for the exact Discord cron/heartbeat stall on current main.

## Solution Assessment

No. Current main is a partial defensive implementation; the narrower maintainable solution is a maintainer-approved liveness and abort policy plus diagnostics, not unconditional reset of any active session lane.
