## Summary

Keep this issue open. Current main still exposes only payload-level `lightContext` for cron jobs, while prompt-cache/context-pruning behavior is driven from `agents.defaults.contextPruning`; there is no persisted, protocol, CLI, UI, agent-tool, or isolated-runner path for a per-job `disableCache` or `contextPruning` override. The earlier implementation PR #8341 is closed and unmerged, and #24597's direct script/no-LLM path is complementary rather than a replacement for per-job cache control on agent-turn cron jobs.

## Best Possible Solution

Keep this issue open and implement an additive top-level `contextPruning?: AgentContextPruningConfig` cron job override using the existing `mode: "off" | "cache-ttl"` contract. Thread it through cron types, normalization, persistence, gateway schemas, CLI, Control UI, docs, and the agent cron tool, then merge the job override into `cfgWithAgentDefaults` before isolated cron invokes the runner. Add regression coverage proving a job-level `mode: "off"` overrides global/per-agent cache-ttl pruning while leaving #24597-style direct script execution as a separate optimization path.
