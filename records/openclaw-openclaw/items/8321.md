## Summary

Keep this issue open. Current main still lacks per-cron-job cache/context-pruning control: the cron job contract, gateway schema, CLI, Control UI, docs, and isolated runner only expose payload-level `lightContext`, while prompt-cache pruning is still driven from `agents.defaults.contextPruning`. The prior implementation PR #8341 is closed and unmerged, and #24597's direct script/no-LLM path is complementary rather than a replacement for per-job pruning on agent-run cron jobs.

## Best Possible Solution

Keep this issue open. Implement an additive top-level `contextPruning?: AgentContextPruningConfig` cron job override using the existing `mode: "off" | "cache-ttl"` contract; thread it through cron create/update/list types, normalization, persistence, gateway schemas, CLI, Control UI, docs, and agent tool/protocol clients; then merge it into the effective `cfgWithAgentDefaults` before isolated cron invokes the embedded runner. Add a regression proving a job-level `mode: "off"` overrides global or per-agent `agents.defaults.contextPruning.mode: "cache-ttl"`, while leaving #24597-style direct script/no-LLM execution as a separate optimization path.

## Risks / Open Questions

- A partial implementation could add a field to one schema or UI surface without changing the embedded runner, leaving cache-ttl behavior unchanged at execution time.
