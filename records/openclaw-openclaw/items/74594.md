## Summary

Keep open. Current main and the latest release still have skill load-time metadata, local skill status, and `openclaw skills` list/info/check commands, but they do not define a skill capability/permission manifest or the requested `openclaw skills inspect` capability view. The RFC is coherent, current, and security-adjacent, and the related open items are broader enough that maintainers should decide whether to consolidate rather than have this cleanup close it preemptively.

## What This Changes

The issue requests an optional skills-first capability manifest plus an `openclaw skills inspect <skill-id>` command that displays declared tools, file, and exec access without changing runtime behavior.

## Best Possible Solution

Make an explicit maintainer decision on the public skill capability vocabulary and its relationship to #12219 and #10827; if accepted, add a typed schema, parser/status propagation, `openclaw skills inspect`, public docs, and targeted tests with clear inspect-only wording.

## Reproduction Assessment

Not applicable as a feature/RFC rather than a runtime bug. The current-main verification path is source/docs inspection of skill metadata parsing, status rendering, and `openclaw skills` command registration, which shows the requested surface is absent.

## Solution Assessment

Unclear. The narrow inspect-only direction is plausible, but it is not yet clearly the best fix until maintainers choose the manifest location, vocabulary, and wording relative to broader permission-manifest and supply-chain-security work.
