## Summary

Keep open. Current main still has skill metadata for load-time requirements and install hints, plus local `openclaw skills` list/info/check commands, but it does not define an inspectable skill capability manifest or the requested `openclaw skills inspect` command. The RFC is coherent, current, and security-adjacent; related prior art in #12219 and #10827 should guide maintainer product/security review rather than cleanup closure.

## What This Changes

The issue requests an optional skills-first capability manifest and an `openclaw skills inspect` command that shows declared tool, file, and exec expectations without changing runtime enforcement.

## Best Possible Solution

Make an explicit maintainer decision on the public skill capability vocabulary and its relationship to #12219 and #10827. If accepted, add a typed schema/parser, propagate declared capabilities into skill status, add `openclaw skills inspect`, document the inspect-only contract, and cover valid, missing, malformed, unknown-field, and exec-warning cases with targeted tests.

## Reproduction Assessment

Not applicable as a feature/RFC rather than a runtime bug. The repository inspection gives a high-confidence current-state check: the metadata type, parser, CLI command registration, status shape, and docs do not include the requested capability manifest or inspect command.

## Solution Assessment

Unclear. The inspect-only slice is narrow and maintainable, but the permission vocabulary and security messaging are public contract decisions, so the best path is maintainer consolidation/review before implementation rather than an automated fix PR.
