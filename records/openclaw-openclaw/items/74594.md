## Summary

Keep open. Current main still has SKILL.md metadata, load-time requirements, status reporting, and `openclaw skills` list/info/check surfaces, but it does not define a skills capability manifest or an `openclaw skills inspect` command. The related #12219 and #10827 threads are important prior art rather than a clearly canonical replacement for this narrower inspect-only RFC, and the public permission vocabulary needs maintainer product/security alignment before implementation.

## What This Changes

The issue proposes an optional skills-first capability manifest plus an `openclaw skills inspect <skill-id>` command that displays declared tools, file, and exec access without changing runtime behavior.

## Best Possible Solution

Keep this as an open design/RFC item until maintainers decide the skills capability vocabulary and its relationship to #12219/#10827. If accepted, implement a typed schema, parser/status propagation, CLI rendering, public docs, and focused tests with explicit inspect-only wording.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature/RFC. Static inspection gives a high-confidence verification path: current main lacks the requested schema field, parser/status propagation, docs, tests, and `openclaw skills inspect` command.

## Solution Assessment

Unclear. The inspect-only v0 is a plausible narrow first step, but the best maintainable path depends on maintainer agreement about the public permission vocabulary and whether this should stay separate from or fold into the broader open security/manifest threads.
