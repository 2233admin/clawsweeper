## Summary

Keep this PR open, but do not merge it as-is. Current `main` still resolves `loadSessionEntry` through the freshest matching store row rather than an exact requested-key preference, while the submitted shortcut would regress the existing canonical/case-variant freshness contract that current tests cover. A later mainline fix preserves one deleted-main target case, but it does not implement the PR’s broader exact-key behavior.

## What This Changes

The PR changes `src/gateway/session-utils.ts` so `loadSessionEntry` returns a direct requested-key or canonical-key store entry before falling back to freshest-`updatedAt` matching.

## Best Possible Solution

Land a narrowed gateway session fix that prefers an exact requested store key only for explicit non-canonical archived or compound keys, while preserving freshest duplicate selection for canonical logical keys and direct history reads; add regression tests for both the archived-key click case and the existing freshness cases.
