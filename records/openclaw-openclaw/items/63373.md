## Summary

Keep this open. Current main still strips XML-style tool-call scaffolding and WebChat's exact synthetic repair message, but it does not cover the reported uppercase square-bracket `[TOOL_CALL]...[/TOOL_CALL]` / `[TOOL_RESULT]...[/TOOL_RESULT]` Telegram-visible shape, especially with the non-JSON payload shown in the report.

## What This Changes

The issue asks OpenClaw to prevent raw uppercase square-bracket internal transcript markers from being delivered to Telegram users after transcript corruption or repair.

## Best Possible Solution

Keep this issue open and add a narrowly tested shared user-visible sanitization guard for uppercase square-bracket internal transcript blocks such as `[TOOL_CALL]...[/TOOL_CALL]` and `[TOOL_RESULT]...[/TOOL_RESULT]`, including the exact non-JSON Telegram payload shape from the report. The best fix is a shared assistant-visible sanitizer update plus final outbound-delivery defense in depth, with tests that preserve literal code/examples and do not regress existing XML/local-model stripping behavior.
