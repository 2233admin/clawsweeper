## Summary

Keep this PR open. Current main and the latest release still derive the session-memory filename/header date from the command event timestamp, so the requested behavior is not implemented. The PR is useful source work, but it is not merge-ready against current main because its patch no longer applies after the session-memory refactor/local-time fix and its timestamp fallback behavior needs to be reworked before landing.

## What This Changes

The PR modifies the bundled session-memory hook and tests so memory filenames and headers prefer the transcript session creation timestamp while keeping the fallback HHMM slug based on the `/new` event timestamp.

## Best Possible Solution

Implement the creation-date behavior on current main by reading a parseable `{ type: "session", timestamp }` transcript header, formatting that instant through the existing local-time formatter for filename/header date and time, preserving `event.timestamp` for the fallback HHMM slug, and consulting `.reset.*` headers only when the active transcript is actually rotated or empty.
