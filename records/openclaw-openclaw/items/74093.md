## Summary

Keep open. Current main still detects `/new` and `/reset` as reset-triggered turns but does not apply that reset state to the active-run queue policy, and the open linked PR #74144 is the implementation candidate rather than a reason to close before merge.

## What This Changes

Reset-triggered turns such as `/new` and `/reset` should interrupt or clear active queue work instead of being steered or enqueued behind an active run.

## Best Possible Solution

Review and land #74144 or an equivalent core fix that gives effective reset-triggered turns interrupt semantics, suppresses steer/followup handling for that turn, clears queued lane work, aborts the active run, and adds regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`. Keep any remaining Feishu startup or session-index hang as a separate focused report if it still reproduces afterward.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active session run with `messages.queue.mode` set to `steer` or another followup-producing mode, followed by an authorized `/new` or `/reset`; current main sets reset state but still derives the active-run queue action from the configured queue mode.

## Solution Assessment

Yes. Threading reset state through the existing queue-policy and prepared-run cleanup seams is the narrowest maintainable fix for the reported delay; the later Feishu startup symptoms should not expand this queue repair unless they remain after it lands.
