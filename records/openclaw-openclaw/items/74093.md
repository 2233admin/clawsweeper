## Summary

Keep open. Current main still routes reset-triggered turns through the configured active-run queue policy, and an open PR, #74144, uses closing syntax for the exact queue/reset bug, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Reset commands such as `/new` and `/reset` should interrupt active work and start a fresh session immediately instead of being steered or queued behind an active run.

## Best Possible Solution

Review and land #74144 or an equivalent focused core reply-queue fix that treats effective reset-triggered turns with interrupt semantics, suppresses steering/followup for that turn, clears queued session work, aborts the active run, and adds regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`. Keep the later startup/session-index observations separate unless they still reproduce after the queue repair.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with an active run and `messages.queue.mode` set to `steer`, `followup`, `collect`, or `steer-backlog`, an authorized `/new` or `/reset` sets reset state but current main still resolves active-run behavior from the configured queue mode.

## Solution Assessment

Yes, the linked PR direction is the narrowest maintainable fix if review validates the diff: thread reset state into the existing queue-policy seam, suppress active steering/followup for reset turns, and cover the reset queue modes in tests. A broader Feishu startup rewrite is not required for the core queue bug.
