## Summary

Current main still lets reset-triggered model turns use the configured active-run queue mode. In `steer` mode, that path is classified as `enqueue-followup`, so a `/new` or `/reset` turn that continues into model execution can wait behind an active run. Bare `/new` and `/reset` acknowledgements are already special-cased, but that does not cover reset tails or soft reset hook flows.

## Best Possible Solution

Keep this open and fix core reply queue preparation so effective reset-triggered model turns use interrupt semantics for active-run handling and do not steer/followup behind the active run. Preserve Feishu plugin behavior and bare reset acknowledgement behavior, then add focused queue/reset tests.
