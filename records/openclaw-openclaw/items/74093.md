## Summary

Keep open. Current main still computes reset-triggered state but does not apply it to the active-run queue policy, and the open linked PR #74144 is the right implementation path because it uses closing syntax for this issue.

## What This Changes

Reset-triggered `/new` and `/reset` turns should bypass steer/followup queueing, clear queued work for the session, abort the active run, and start the fresh session promptly.

## Best Possible Solution

Review and land or adjust #74144 so effective reset turns suppress steering/followup behavior, clear the session lane, abort the active run with interrupt semantics, and add regression coverage for `steer`, `steer-backlog`, `followup`, and `collect`; split the later startup/health observations into focused follow-ups if they remain reproducible.

## Reproduction Assessment

Yes. The source-level path is high confidence: with an active run and a non-interrupt queue mode, reset state is computed but omitted from both active-run queue policy calls, matching the reporter's delayed Feishu timing logs.

## Solution Assessment

Yes for the core queue symptom. A narrow core reply-queue change that treats effective reset-triggered turns as interrupt-like is the maintainable fix; the additional startup blockers should be validated separately if not covered by that PR.
