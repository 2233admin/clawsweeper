## Summary

Keep this issue open. Current main still computes reset-triggered state but does not feed it into the active-run queue policy, so a reset-triggered turn can still follow the configured `steer`/followup enqueue path. The issue is also already paired with open PR #74144, which references this issue as the intended fix, so the right next step is PR review rather than closing the issue early or queueing a separate fix.

## Best Possible Solution

Keep this issue open while #74144 is reviewed. The best fix is in core reply queue preparation: effective reset-triggered model turns should bypass steer/followup enqueueing, clear/abort active run state with interrupt semantics, preserve bare `/new` and `/reset` acknowledgements, and add focused tests for reset plus `steer`, `steer-backlog`, `followup`, and `collect`; the newer session-index/gateway-hang reproduction should be verified during that review or tracked as a narrower follow-up.
