## Summary

Keep open. Current main still matches the reported queue bug: reset commands are detected, but the active-run queue policy and both call sites still derive behavior from the configured queue mode without a reset override. Open PR #74144 explicitly closes this issue and targets the same queue/reset path, so this should remain open until that PR or an equivalent fix lands.

## What This Changes

Reset-triggered turns such as `/new` and `/reset` should bypass steer/followup queueing, clear pending work, abort the active run, and start the fresh session immediately.

## Best Possible Solution

Land #74144 or an equivalent core fix that makes effective reset-triggered turns use interrupt semantics across the active-run queue policy, steering/followup suppression, lane clearing, active-run abort, and regression tests for `steer`, `steer-backlog`, `followup`, and `collect`. If Feishu startup health issues still reproduce after that, track them as a narrower channel/gateway startup follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction path is an active session run with `messages.queue.mode` set to `steer`, then an authorized `/new` or `/reset`; current main marks the turn as reset-triggered but still resolves active-run queue behavior from `steer`.

## Solution Assessment

Yes. Threading reset state through the existing queue-policy seam and disabling steer/followup behavior for that turn is the narrowest maintainable fix for the reported delay; the Feishu startup blockers should remain separate unless they still reproduce after the queue repair.
