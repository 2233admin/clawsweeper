## Summary

Keep open. Current main still recognizes reset-triggered turns, but the active-run queue policy and both queue-policy call sites still derive behavior only from the configured queue mode, so `/new` or `/reset` can still be enqueued behind active work. The linked fix PR #74144 is open and unmerged, so this issue should wait for that PR review rather than close early.

## What This Changes

The issue reports that reset commands such as `/new` and `/reset` can be routed through normal active-run queue behavior under `messages.queue.mode: steer`, delaying session reset behind existing work.

## Best Possible Solution

Review, land, or revise #74144 so effective reset-triggered turns bypass steer/followup enqueueing, clear queued work for the lane, abort the active run with interrupt semantics, and add focused regression tests. Any remaining Feishu startup or session-index hang after that should be tracked as a narrower follow-up.
