## Summary

Keep this issue open. Current main still starts outbound delivery recovery on gateway startup and replays pending delivery-queue entries with retry/backoff/max-retry gates only; it has no stale-age, TTL, or restart-generation cutoff for old first-attempt entries. PR #54900 is a focused open implementation candidate, but it is not merged on main, so the reported behavior remains actionable.

## What This Changes

The issue requests a stale-message freshness policy for gateway restart recovery so old queued outbound messages are dropped, discarded, or controlled by a configurable max age instead of replayed in a batch.

## Best Possible Solution

Land a narrow core outbound recovery freshness policy that prevents old first-attempt queued replies from being delivered as fresh after restart while preserving intentional retry recovery for genuine transient send failures; include regression tests around startup recovery and update config/docs/changelog only if an operator knob is exposed.
