## Summary

Keep open. Current main and the latest release still have the reported nested default, while multiple open PRs already target this exact bug with closing intent, so the issue should remain open until one implementation lands or maintainers choose a different compatibility scope.

## What This Changes

The issue requests that the interactive `agents add <id>` wizard pre-fill a peer-level workspace path like `workspace-<id>` instead of nesting new agent workspaces under the main `workspace/<id>` directory.

## Best Possible Solution

Review and land one narrow fix that changes the interactive default for brand-new agents to the documented peer-level path, preserves explicit and existing workspaces, keeps runtime resolver compatibility where needed, and adds focused regression tests plus a changelog entry.

## Reproduction Assessment

Yes. The source path is high-confidence: setup writes `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir` as the prompt default, and that resolver appends the non-default agent id under the fallback workspace.

## Solution Assessment

Yes, the narrowest maintainable direction is to change the wizard prefill for new agents rather than silently relocating existing agents through broad runtime resolver changes. The exact PR to land is still a maintainer choice because active fixes take different compatibility approaches.
