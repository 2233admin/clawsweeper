## Summary

Keep open. Current main still seeds the interactive `agents add` workspace prompt from the runtime resolver, the resolver still nests non-default agents under `agents.defaults.workspace`, and the docs still describe the peer-level workspace convention. The issue is already paired with open implementation PRs, so this should stay open until one of those is reviewed and merged or explicitly closed.

## What This Changes

The issue requests changing the interactive `openclaw agents add <id>` workspace prefill from a nested `workspace/<id>` path to the documented peer-level `workspace-<id>` path.

## Best Possible Solution

Land one narrow fix that changes the interactive default for brand-new agents to a peer-level workspace path while preserving explicit workspaces and existing runtime compatibility. Add focused regression coverage for the prompt/default helper and include a user-facing changelog entry when the fix merges.

## Reproduction Assessment

Yes. A config with `agents.defaults.workspace` set and no explicit workspace for a new non-default agent will flow through `agentsAddCommand` into `resolveAgentWorkspaceDir`, which currently returns `workspace/<id>` as the prompt default.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: adjust the wizard default for brand-new agents instead of silently relocating existing agents at runtime. The exact PR to land is still a maintainer choice because active PRs differ in compatibility scope.
