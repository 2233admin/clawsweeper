## Summary

Keep open. Current main still pre-fills the interactive agent workspace prompt from the runtime workspace resolver, and that resolver still nests non-default agents under `agents.defaults.workspace`; the docs continue to describe the peer-level `workspace-<agentId>` convention. The issue is also paired with open implementation PRs, so it should stay open until maintainers choose and merge one fix.

## What This Changes

The issue asks for `openclaw agents add <id>` to pre-fill a peer-level workspace such as `~/.openclaw/workspace-<id>` instead of `~/.openclaw/workspace/<id>`.

## Best Possible Solution

Land one narrow agents fix that changes the interactive default for brand-new agents to the documented peer-level workspace, preserves explicit and existing workspace paths, covers the prompt/default behavior with regression tests, and adds a user-facing changelog entry.

## Reproduction Assessment

Yes. A config with `agents.defaults.workspace` set and no explicit workspace for the new agent flows through `agentsAddCommand` into `resolveAgentWorkspaceDir`, which returns `<main-workspace>/<agentId>` for non-default agents.

## Solution Assessment

Yes. The requested direction is the maintainable fix when scoped to the wizard prefill for new agents; updating docs to bless nested defaults would leave the recursive read/delete footgun in place.
