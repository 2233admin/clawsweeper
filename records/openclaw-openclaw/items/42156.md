## Summary

Keep this issue open. Current main still acknowledges Telegram callback queries only inside the downstream `callback_query` handler, while `sequentialize(getTelegramSequentialKey)` is installed earlier in the middleware chain. The partial approval-lane mitigation only moves parsed exec/plugin approval callbacks to a separate lane; ordinary inline-button callbacks still use the normal chat/topic key and can queue behind an active turn.

## Best Possible Solution

Implement the Telegram plugin fix on current main: add a pre-sequentialize middleware in `extensions/telegram/src/bot-core.ts` that immediately answers any `callbackQuery` and catches/logs API failures, remove or avoid redundant handler-level acknowledgement in `extensions/telegram/src/bot-handlers.runtime.ts`, and add a regression test that runs the middleware chain with a busy same-chat/topic turn plus ordinary callback data to prove `answerCallbackQuery` fires before the queued callback action is released. The closed contributor PRs #42125, #42181, and #42516 are useful references but need a fresh maintainer-ready patch.
