## Summary

Keep open for maintainer or author rework. The proposed one-line change conflicts with current main, the existing DeepSeek unit/live coverage, and DeepSeek's current thinking-mode contract: enabled thinking replay must preserve or backfill `reasoning_content` for tool-call continuations, while only the disabled-thinking path strips it.

## What This Changes

The PR changes the DeepSeek V4 OpenAI-compatible thinking wrapper to strip `reasoning_content` from outgoing messages in the thinking-enabled branch instead of ensuring assistant replay messages carry it.

## Best Possible Solution

Keep the enabled DeepSeek V4 replay path preserving native `reasoning_content` and backfilling missing assistant fields. If maintainers can reproduce a separate current provider rejection, add a targeted cleanup rule with DeepSeek unit and live coverage, provider docs, and a changelog entry without regressing tool-call continuation replay.

## Reproduction Assessment

Yes for the PR-introduced regression: applying this diff would make the existing enabled DeepSeek V4 replay tests fail because they assert `reasoning_content` remains present. The reported provider-side 400 is unclear against current main because no captured failing payload or live reproduction accompanies this PR.

## Solution Assessment

No. Blanket stripping in the enabled-thinking branch is not the best way to solve the reported 400 because it removes the field DeepSeek requires for thinking-mode tool-call replay; the safer direction is a narrow, evidence-backed replay cleanup only if a distinct no-tool rejection is reproduced.
