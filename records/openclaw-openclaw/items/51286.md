## Summary

Keep this PR open. Current main has fixed part of the Telegram native-command path by using agent-scoped slash session keys and recording session metadata, but it has not implemented this PR’s specific sandbox-explain normalization: `openclaw sandbox explain --session telegram:slash:<id>` still returns the raw colon-containing key, so provider inference cannot resolve `telegram` from that input.

## Best Possible Solution

Keep the PR as a focused implementation candidate. Maintainers should decide whether `sandbox explain` should continue accepting legacy/non-agent-scoped slash keys; if yes, rebase the PR, preserve the downstream `channel` and `allowedByConfig` assertions, and run the targeted sandbox-explain test before landing. If the intended product direction is agent-scoped keys only, close this PR only after updating the slash-command docs and the linked user report accordingly.
