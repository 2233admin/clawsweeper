## Summary

Keep this PR open. Current main preserves iMessage text chunking, but it still does not implement the iMessage-owned audioAsVoice payload path: the extension exposes only sendText/sendMedia, the SDK attached-results wrapper does not synthesize sendPayload, and the iMessage runtime/sender has no audioAsVoice option.

## Best Possible Solution

Keep this PR open and ask for a narrowed rebase against extensions/imessage. The best fix is an iMessage-owned sendPayload path, or a generic SDK helper that attached-results channels can explicitly opt into, that threads audioAsVoice through channel.ts -> channel.runtime.ts -> send.ts, suppresses the generic <media:audio> placeholder for voice-intent audio, sends visible captions separately when needed, preserves channels.imessage.textChunkLimit, and adds focused iMessage regression coverage.
