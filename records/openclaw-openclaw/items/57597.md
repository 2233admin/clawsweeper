## Summary

Do not close PR #57597. Current main and the latest release still write ACP spawn labels through the requester command store path, while spawned ACP metadata is keyed to the target agent store. The PR remains a focused, useful bug fix; the provided review-thread concern about a broken test assertion was followed by a PR commit specifically fixing that assertion.

## Best Possible Solution

Keep PR #57597 open and land a cleaned version of the focused fix: resolve the target session store from the spawned ACP `sessionKey`, only mutate the in-memory requester store when it is the same store, and keep the regression test that proves the label is written to the target store rather than the requester store.

## Evidence

- **current-main-label-write:** `persistSpawnedSessionLabel` currently mutates `params.commandParams.sessionStore`, returns when `params.commandParams.storePath` is absent, and persists via `updateSessionStore(params.commandParams.storePath, ...)`. It does not resolve the store from the spawned ACP `sessionKey`, so cross-agent spawns can write the label to the requester store instead of the target agent store.
