## Summary

Keep open. Current main still has the stale sessions-cache behavior, and the linked bug #55705 remains open. The success-path assignment in this PR is the right direction, but the patch is not merge-ready because it also clears last-known-good sessions data on every transient request error and the branch is dirty against current main.

## What This Changes

The PR changes the Control UI sessions controller to assign `sessionsResult = res ?? null` after `sessions.list` and to clear `sessionsResult` in the catch path.

## Best Possible Solution

Revise this PR narrowly on current main: successful `sessions.list` responses should always write `state.sessionsResult = res ?? null`, checkpoint invalidation/refetch should still run only when a real result exists, generic request failures should keep last-known-good data, and the missing-scope clear path should stay intact with focused controller tests.

## Reproduction Assessment

Yes. A focused unit path can seed `state.sessionsResult` with a non-empty result, mock `client.request("sessions.list")` to resolve `undefined`, call `loadSessions`, and observe current main leaving the stale count in place.

## Solution Assessment

No. The success-path `res ?? null` change is the right narrow fix, but clearing `sessionsResult` for every exception is broader than the reported bug and can regress transient reconnect or RPC failure UX.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve cached sessions on transient errors:** `ui/src/ui/controllers/sessions.ts:66`
  - body: `loadSessions` only needs to clear the cache when a successful `sessions.list` response is empty. This added catch-path assignment runs for any RPC or transport exception, so a reconnect hiccup after a successful load replaces the last-known-good sessions snapshot with null/N/A even though sessions may still exist. Keep the existing generic-error behavior and only clear for the missing-scope case.
  - confidence: 0.88
