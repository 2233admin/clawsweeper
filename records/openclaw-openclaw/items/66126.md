## Summary

Keep open. Current main still has the reported compaction-overflow retry shape: after successful compaction, the embedded runner continues the outer loop and reconstructs the same prompt input without durable current-turn identity or inbound-message obligation metadata. Cron still invokes that path with fixed prompt text, and the related open reports show this remains part of an unresolved compaction-resume family rather than cleanup-ready work.

## What This Changes

The issue asks OpenClaw to prevent compaction retries from re-injecting cron/system prompts as fresh user input and to preserve turn/message state across overflow recovery.

## Best Possible Solution

Define a stable compaction-resume contract for embedded runs: preserve current-turn identity and original inbound message metadata across compaction, distinguish resumed attempts from fresh cron/system/user input, avoid replaying deterministic failed tool cascades as new work, and add regression coverage for cron/system-event overflow alongside the related compaction-resume reports.
