## Summary

Keep open. Current main has the partial bounded-list optimization, but sessionId resolution and MCP conversation_get still route through session-list row materialization, and returned rows can still synchronously hydrate transcript-derived usage/title fields; the remaining behavior is real and not fully covered by the open list-optimization PR.

## What This Changes

The issue requests avoiding full session-row materialization and synchronous transcript reads for session list and direct lookup paths by filtering first, indexing summary fields, and using direct session lookup instead of list reuse.

## Best Possible Solution

Keep the shipped early-filtering behavior, then add a metadata-only or indexed lookup path for sessionId/key resolution and MCP conversation_get so they avoid sessions.list row enrichment. Treat persistent write-time summary indexing and broader transcript streaming as a separate design decision, likely coordinated with the related memory/session performance reports.

## Reproduction Assessment

Yes. Static reproduction against current main is high-confidence: sessionId resolution still calls listSessionsFromStore with no limit, and MCP getConversation still calls a 500-row list path whose rows can synchronously hydrate transcript fields.

## Solution Assessment

No. Current main is only a partial solution; the narrow maintainable next step is to remove list-row materialization from direct lookup paths while preserving visibility, deleted-agent, duplicate-session, and spawned-session semantics.
