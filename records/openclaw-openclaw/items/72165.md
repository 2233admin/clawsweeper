## Summary

Keep open. Current main partially fixes the list path by applying search/filter/limit before row construction, but sessionId resolution and MCP conversation_get still reuse list row materialization, and selected rows can still synchronously read transcript data for usage, derived titles, and last-message previews.

## What This Changes

The issue requests a faster session summary and lookup path that avoids full session-row materialization and synchronous transcript reads before search, pagination, and direct session lookup.

## Best Possible Solution

Keep the current early list filtering, then add a metadata-only lookup surface for sessionId/key resolution and MCP conversation_get so those paths can find one session descriptor without constructing list rows or touching transcripts. Treat persistent summary indexing as a separate storage design unless maintainers choose it as the canonical direction.

## Reproduction Assessment

Yes. Static reproduction against current main is high-confidence: sessions.resolve by sessionId still routes through listSessionsFromStore without a limit, and MCP conversation_get still calls listConversations with limit 500 while list rows can synchronously hydrate transcript fields.

## Solution Assessment

Unclear. The current implementation is only a partial fix; the narrowest maintainable next step appears to be direct metadata lookup for resolution and MCP descriptors, while the full SessionSummaryIndex proposal needs maintainer scoping.
