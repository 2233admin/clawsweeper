## Summary

Keep open. Current main has the bounded `sessions.list` optimization, but the broader performance report remains valid because direct `sessionId` resolution still materializes enriched session rows and MCP `getConversation` still scans a 500-row conversation list before selecting one session.

## What This Changes

The issue requests reducing session-list and session-lookup latency by avoiding full session row materialization and synchronous transcript reads before filtering, pagination, or direct lookup.

## Best Possible Solution

Keep the current early filtering for bounded `sessions.list`, then add a metadata-only or indexed direct lookup path for session key/sessionId resolution and MCP `getConversation` so they avoid list-row enrichment. Treat a persisted write-time summary index and broader transcript memory work as a separate design decision coordinated with #57715, #59436, and #69451.

## Reproduction Assessment

Yes. Static reproduction is high confidence: `sessions.resolve` by `sessionId` still calls `listSessionsFromStore` without a limit, and MCP `getConversation` still calls `listConversations` with `limit: 500` before filtering for one session.

## Solution Assessment

No, current main is only a partial solution. The narrow maintainable next step is to remove list-row materialization from direct lookup paths while preserving existing spawned-session visibility, deleted-agent checks, and exact-sessionId behavior; a full persisted `SessionSummaryIndex` needs maintainer product/storage scoping.
