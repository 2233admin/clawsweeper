## Summary

Keep open. Current main has the partial limit-before-row-enrichment optimization, but direct sessionId resolution still reuses full session-list row materialization, MCP conversation_get still fetches up to 500 enriched conversation rows to find one key, and returned rows can still synchronously hydrate transcript-derived usage/title fields.

## What This Changes

The issue asks OpenClaw to avoid full session row materialization and synchronous transcript reads for session list and lookup paths by adding summary, indexing, and direct-lookup behavior.

## Best Possible Solution

Keep the shipped early filtering behavior, then add a metadata-only or indexed lookup path for sessionId/key resolution and MCP conversation_get so they avoid sessions.list row enrichment. Treat persistent write-time summary indexing, broader transcript streaming, and storage layout changes as coordinated follow-up work with the open session-memory reports.

## Reproduction Assessment

Yes. Static reproduction against current main is high-confidence: sessionId resolution still calls listSessionsFromStore without a limit, and MCP getConversation still calls a 500-row list path whose rows can synchronously hydrate transcript fields.

## Solution Assessment

No. Current main is only a partial solution; the narrow maintainable next step is to remove list-row materialization from direct lookup paths while preserving the existing session visibility and deleted-agent semantics.
