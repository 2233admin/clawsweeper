## Summary

Keep open. Current main has partially fixed bounded `sessions.list` calls by filtering, searching, and limiting before row enrichment, but the broader report remains valid: returned rows can still synchronously hydrate transcript-derived fields, `sessionId` resolution still materializes list rows, and MCP `getConversation` still scans a 500-row conversation list.

## What This Changes

Optimize session listing and lookup so filters, limits, and direct session lookup avoid full row materialization and synchronous transcript reads, potentially via a write-time session summary index.

## Best Possible Solution

Keep the current early-filtered `sessions.list` behavior, then add a metadata-only or indexed lookup path for `sessions.resolve` and MCP `getConversation` so they avoid row enrichment for direct lookups. Treat a persisted write-time summary index and transcript streaming/memory work as a separate design track unless maintainers explicitly fold that architecture into this issue.

## Reproduction Assessment

Yes. Static reproduction against current main is high confidence: `sessionId` resolution still calls `listSessionsFromStore` with no limit, MCP `getConversation` still scans a 500-row conversation list, and row enrichment still performs synchronous transcript reads when metadata is missing or previews are requested.

## Solution Assessment

No. Current main is only a partial solution; the narrow maintainable next step is a direct metadata lookup path that preserves existing visibility and deleted-agent semantics, while the full persisted summary-index design needs maintainer scope confirmation.
