## Summary

Keep open. Current main partially fixes the session-list hot path by filtering/searching/limiting before row enrichment, but the full reported problem remains: returned rows still synchronously hydrate transcript-derived usage/title fields, sessionId resolution still materializes `sessions.list` and filters it, and MCP `conversation_get` still reuses `listConversations({ limit: 500 })`. Related #57715/#59436 overlap with the list-query optimization, and #69451 overlaps with full transcript reads, but none covers all remaining surfaces here.

## Best Possible Solution

Keep this open and treat the shipped early limit/search behavior as a partial fix. The next maintainer path should either split or consolidate the remaining work: add a lightweight direct sessionId/key lookup that avoids `buildGatewaySessionRow` for every row while preserving visibility/deleted-agent semantics, change MCP `conversation_get` to avoid `listConversations({ limit: 500 })`, and decide whether transcript-derived usage/title persistence belongs here or in the broader transcript-read/memory work tracked by #69451.
