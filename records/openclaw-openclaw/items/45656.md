## Summary

Keep open. Current main still matches the reported Control UI regression: the composer’s visible bordered area is larger than the actual textarea focus target, and the toolbar controls remain small icon-only buttons with no visible labels.

## What This Changes

The issue requests a Control UI v2 chat composer usability fix for the dead lower click area, wasted space, and tiny tooltip-dependent controls compared with the earlier UI.

## Best Possible Solution

Fix the Control UI composer so all non-control visible text-entry chrome focuses the textarea, align the apparent entry area with the actual input target, and improve practical control hit targets/discoverability while preserving Send, Queue, Stop, attachment, Talk, paste, auto-height, and Shift+Enter behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is to open the Control UI composer and click inside the bordered lower composer area outside the textarea and buttons; current main has no wrapper click handler to focus the textarea from that chrome.

## Solution Assessment

Yes for a narrow repair: composer focus behavior plus hit-target/discoverability fixes are the maintainable path. A full rollback to the v1 button layout should remain a separate product decision.
