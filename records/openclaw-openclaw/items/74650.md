## Summary

Keep open: current main does not provide enough evidence that the ACP/acpx approval loop is fixed, and the code still has a plausible path where ACP connects with admin scope while approval can only be granted by a caller that already has that scope. Because this is a gateway pairing and operator-scope boundary, it needs maintainer/security-aware review rather than conservative closure.

## What This Changes

The issue reports that ACP runtime launches and direct bundled ACPX `openclaw` launches repeatedly request scope-upgrade approval after upgrading to OpenClaw 2026.4.26, and approval in the UI does not let initialization complete.

## Best Possible Solution

Add a focused regression path for ACP bridge and ACPX `openclaw` target startup when an existing paired device is below the requested scope, then either narrow the ACP bridge's initial Gateway scopes to what initialization actually needs or make the approval flow clearly require an admin-scoped approver with actionable UI/CLI feedback. Preserve the existing rule that lower-scoped sessions cannot grant themselves `operator.admin`.

## Reproduction Assessment

Unclear. The static code path matches a plausible scope-upgrade loop, but this read-only review did not run a live ACP/acpx session and the issue lacks logs or paired-device state needed for a high-confidence reproduction.

## Solution Assessment

No, closing is not the best way to solve this. The safer path is a targeted ACP/gateway pairing regression test plus maintainer review of the auth boundary before changing scopes or approval behavior.
