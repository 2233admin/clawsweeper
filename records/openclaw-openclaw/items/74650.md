## Summary

Keep this open. Current main has adjacent stale-pairing fixes for backend calls and native approval clients, but the `openclaw acp` bridge still constructs its GatewayClient without the loopback shared-auth device-identity bypass, so the reported ACP/ACPX approval loop remains plausible and not implemented away.

## What This Changes

The issue requests fixing ACP runtime and direct ACPX OpenClaw launches that repeatedly ask for gateway scope approval after updating to OpenClaw 2026.4.26.

## Best Possible Solution

Extend or extract the existing local loopback shared-auth device-identity bypass so `openclaw acp` uses it when connecting to a loopback gateway with token/password auth, with regression coverage proving ACP startup no longer reuses stale paired device scopes.

## Reproduction Assessment

Unclear. The report does not provide a fully runnable reproduction, but source inspection gives a focused path: seed a lower-scope paired default device identity, start `openclaw acp` against a loopback token/password gateway, and verify the connect path still requests a scope upgrade before hello.

## Solution Assessment

Yes, the narrowest maintainable direction is to apply the same loopback shared-auth exemption pattern to the ACP bridge, or share that helper across local backend clients. A broad change to gateway pairing approval semantics would be riskier and unnecessary.

The observable bug is that ACP runtime/direct ACPX `openclaw` can loop on `scope upgrade pending approval` after a stale lower-scope paired default device identity exists. Add or update a user-facing `CHANGELOG.md` fix entry with `Thanks @vharris817`. Do not change dependencies, release metadata, or general gateway pairing approval semantics.
