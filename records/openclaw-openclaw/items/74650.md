## Summary

Keep open. Current main has an intentional scope-upgrade approval model and recovery docs, but I did not find a merged fix or canonical item that specifically covers this ACP/direct-acpx loop where approval reportedly does not persist; the ACP bridge still connects through the Gateway CLI client path that can request default admin scope before ACP initialize completes.

## What This Changes

The issue reports ACP runtime and direct `acpx openclaw` launches on OpenClaw 2026.4.26 getting stuck in a repeated gateway scope-upgrade approval loop after UI approval.

## Best Possible Solution

Reproduce the report on current main or the next release with a paired device whose approved scopes are narrower than the ACP bridge request, then make the ACP bridge and Gateway approval UX converge: either approving the exact pending request mints/stores a usable token and ACP initialize succeeds, or the UI/CLI clearly reports why the approval was forbidden and which authorized command must be used. Do not auto-widen operator scopes as a shortcut.

## Reproduction Assessment

No high-confidence reproduction yet. The report gives concrete ACP/direct-acpx surfaces and request ids, but the minimal steps and missing logs/config are not enough to prove the approval-persistence failure against current main without a live ACP or targeted gateway-auth test.

## Solution Assessment

Unclear. The current explicit scope-upgrade approval model is the right security direction, but it is not yet shown to solve this ACP-specific initialize loop; the safest path is targeted reproduction before changing auth scope defaults or approval persistence.
