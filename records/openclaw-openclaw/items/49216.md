## Summary

Keep this PR open. Current main has some default archive cleanup because `resetArchiveRetention` defaults to `pruneAfter`, but it has not definitely implemented this PR's intended behavior: deleted-archive cleanup is still gated by archived dirs or reset retention being enabled, and the archive timestamp parser still rejects the legacy `+HH-MM` and compact timestamp variants from the PR.

## Best Possible Solution

Keep the PR open as a focused implementation candidate. The best path is to refresh/review it against current session maintenance, strengthen the regression test around the actual remaining gate, keep the legacy timestamp parsing tightly scoped, and land it if maintainers want deleted archive cleanup to run during enforced maintenance regardless of reset archive retention.

## Risks / Open Questions

- The PR's new integration scenario should be checked carefully: with current defaults, `resetArchiveRetention` already makes the cleanup block run even when no live sessions were pruned. A stronger regression should cover `resetArchiveRetention: false` if the intended contract is deleted-archive cleanup independent of reset-archive cleanup.
- Broader archive timestamp parsing affects deletion eligibility. It should remain scoped to existing `.bak`, `.deleted`, and `.reset` archive suffixes with explicit tests for accepted and rejected filename forms.
