## Summary

Keep this issue open. Current main still has the reported risk shape: `corpus="sessions"` is routed into the builtin memory manager with a sessions source filter, but hybrid keyword search still turns any `searchKeyword` failure into empty keyword results via `.catch(() => [])`. The underlying FTS helper builds a `MATCH` query and only has a limited LIKE fallback for short CJK trigram terms, so the reported English and longer/common CJK misses are not visibly fixed or diagnosed on current main.

## Best Possible Solution

Keep this issue open and fix the builtin `memory-core` sessions search path. The best implementation is a focused change in the FTS keyword layer: surface/log keyword MATCH errors, add a bounded substring fallback when MATCH fails for current-tokenizer terms, preserve `source` and model filters, and add regression coverage for `corpus="sessions"` with English and CJK failing terms plus an explicit thrown-MATCH case. `memory status --deep` should also probe representative FTS query health if that can be done cheaply.
