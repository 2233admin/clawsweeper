## Summary

Keep open. This PR is maintainer-authored and has the protected `maintainer` label, and the author explicitly asked that it remain open for maintainer discussion. Current `main` also does not implement the PR's forced-retirement recovery path: the health monitor still performs a normal stop/start, `ChannelManager.stopChannel` has no options parameter, and timeout handling keeps the hung task registered.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to review the latest head against the lifecycle-overlap concern, confirm the task-scoped runtime cleanup and stale-callback gating are sufficient, run targeted gateway tests plus the normal changed gate if landing, and merge only after maintainers accept forced retirement as the right health-monitor recovery behavior.
