## Summary

Current main still has the reported Telegram model-list callback bug: provider buttons emit raw dotted provider IDs, but the list callback parser rejects dots, so `mdl_list_hf.co_1` does not enter the model-list callback path. The latest release tag shown in this checkout has the same source behavior, and the two directly related fix PRs are closed unmerged, so this should remain open and is a good narrow repair candidate.

## What This Changes

Telegram `/models` provider-list callbacks should support provider IDs containing dots so model lists render as inline keyboard buttons for providers like `hf.co`.

## Best Possible Solution

Keep this issue open and land a narrow Telegram plugin fix: allow dots in the `mdl_list_{provider}_{page}` provider capture, add parser and provider-keyboard regression coverage for `hf.co`, and add a user-facing changelog fix entry with proper credit. Do not change core model catalog behavior, model-selection callback semantics, or generated dist files by hand.

## Repair Work Prompt

Fix Telegram `/models` model-list callbacks for provider IDs containing dots. Current `extensions/telegram/src/model-buttons.ts` builds provider callbacks as `mdl_list_${provider.id}_1`, but `parseModelCallbackData()` only accepts `[a-z0-9_-]+`, so `mdl_list_hf.co_1` is rejected and the model-list button flow falls through. Make the narrow parser change needed for dotted provider IDs, add regression coverage for `parseModelCallbackData('mdl_list_hf.co_1')`, and add a provider-keyboard round-trip expectation for a provider such as `{ id: 'hf.co', count: 2 }`. Do not change core model catalog behavior, model-selection callback semantics (`mdl_sel_*` / compact callbacks), slash add-provider callbacks, Telegram authorization, or generated dist files by hand.
