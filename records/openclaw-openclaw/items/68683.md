## Summary

Kept open: current main has a Perplexity web_search provider and env fallback, but the reported 401 path remains plausible because configured plugin credentials still take precedence over Gateway env and there is no regression or live proof covering stale config, runtime reload, or final auth-source selection.

## What This Changes

Ensure native web_search with provider "perplexity" uses the valid Perplexity credential available to the Gateway and does not return a misleading 401 from credential/runtime resolution.

## Best Possible Solution

Add a maintainer-reviewed repro that captures the redacted credential source, selected endpoint, and reload behavior; then either fix precedence/reload so a valid configured or env Perplexity key is used consistently, or make the intended precedence explicit with diagnostics that identify stale configured credentials without exposing secrets.
