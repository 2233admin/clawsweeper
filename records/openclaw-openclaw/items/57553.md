## Summary

Keep this PR open. The underlying Docker onboarding problem is still present on current main, and the PR is scoped to a real user path, but the current patch duplicates a weaker container detector and needs maintainer review against the related Linuxbrew/apt fallback approaches before merge.

## What This Changes

The PR adds Linux-container detection to skill onboarding, filters brew-only missing skill install options when brew is absent, shows a container-specific note, and adds onboarding/container-context tests.

## Best Possible Solution

Update the PR to reuse or centralize the existing cgroup-aware `isContainerEnvironment()` behavior for onboarding, keep the focused UX tests, and resolve the Docker skill-install policy relationship with the related Linuxbrew/apt fallback PRs before landing.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: Docker setup runs `openclaw onboard`, onboarding lists missing brew-backed skills, and selecting one with `brew` absent triggers the Homebrew guidance path.

## Solution Assessment

No, not as-is. Hiding brew-only options in containers is a narrow and plausible UX fix, but the detector should use the existing broader container signal set instead of adding a partial duplicate, and the competing Linuxbrew/apt proposals need maintainer policy review.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

- **[P2] Reuse the existing container detector:** `src/daemon/container-context.ts:25`
  - body: The new detector only checks OpenClaw/env hints plus `/.dockerenv` and `/run/.containerenv`. Reuse or centralize that detector for this branch.
