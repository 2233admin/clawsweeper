## Summary

Current main still has the reported TUI copy: it tells users to run `openclaw devices list` and approve their request ID, but does not show the concrete `openclaw devices approve <requestId>` command. The CLI, docs, and protocol helpers already support the better guidance, so this remains a narrow actionable TUI copy/test issue.

## Best Possible Solution

Keep this issue open and update `src/tui/tui.ts` so `resolveGatewayDisconnectState` includes `openclaw devices approve <requestId>` after `openclaw devices list`. Prefer using `readConnectPairingRequiredMessage` to display `openclaw devices approve <actual-request-id>` when the close reason contains a sanitized request ID, with `<requestId>` as the fallback. Tighten `src/tui/tui.test.ts` to assert both commands and request-id handling.
