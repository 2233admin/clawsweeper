## Summary

Current main still matches the reported SwiftUI render-loop path: attributed text changes synchronously call updateWindowFrame(), targetFrame() synchronously calls measuredHeight(), and measuredHeight() still mutates observable isOverflowing state that the overlay view reads. The linked menubar PR #60990 is a different macOS surface and does not supersede this VoiceWakeOverlay bug.

## Best Possible Solution

Keep this issue open and fix it in the macOS app owner code. The likely change belongs around apps/macos/Sources/OpenClaw/VoiceWakeOverlayController+Window.swift and apps/macos/Sources/OpenClaw/VoiceWakeOverlayView.swift: avoid synchronous observable-state writes from measuredHeight(), guard or defer/coalesce isOverflowing changes, defer or coalesce attributed-text frame updates, and add focused regression coverage or documented manual macOS verification for the overlay update cycle. Review #60990 separately because it addresses menubar activity state, not this overlay render loop.
