## What This Changes

The issue reports a frequent macOS pinwheel caused by a SwiftUI render loop in VoiceWakeOverlay when attributed-text changes trigger window-frame measurement that mutates observed overflow state.

## Best Possible Solution

Make VoiceWakeOverlay height calculation pure or move guarded overflow-state updates outside SwiftUI view-update handling, coalesce attributed-text frame updates, and add focused regression coverage plus manual macOS verification for repeated overlay open/update/reorder cycles.

## Repair Work Prompt

Fix the macOS VoiceWakeOverlay render loop reported in https://github.com/openclaw/openclaw/issues/43480. Keep the change scoped to the VoiceWakeOverlay owner path. Break the synchronous cycle where `VoiceWakeOverlayView` observes `controller.model.attributed`, calls `updateWindowFrame(animate:)`, `measuredHeight()` runs, and measurement mutates `model.isOverflowing`. Prefer a pure measurement helper that returns height plus overflow, or guard and schedule overflow-state changes outside the SwiftUI view-update cycle; also defer/coalesce attributed-text frame updates so repeated transcript updates cannot trigger multiple updates per frame. Do not broaden this into the unrelated menubar stale-activity work from PR #60990 or a full VoiceSessionCoordinator rewrite. Add focused tests around the new helper/coalescing behavior where feasible, update existing VoiceWakeOverlay tests, and add a single-line `CHANGELOG.md` Unreleased Fixes entry with `Thanks @gdiab`; credit @m13v too if the linked external approach materially informs the patch. Validate with `swift test --package-path apps/macos --filter VoiceWakeOverlay`, `swift build --package-path apps/macos --product OpenClaw`, `pnpm format:swift`, and manual macOS verification of repeated overlay trigger/update/edit/reorder/dismiss cycles with no SwiftUI 'Modifying state during view update' or 'onChange ... tried to update multiple times per frame' log loop.
