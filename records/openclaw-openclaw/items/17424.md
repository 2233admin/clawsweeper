## Summary

Keep open. Current main still does not accept the requested `dimensions` key under default or per-agent `memorySearch`, and the OpenAI inline and batch memory embedding paths still omit an OpenAI-compatible `dimensions` field. There is adjacent support for request-time `dimensions` on the gateway `/v1/embeddings` endpoint and for `outputDimensionality` in other provider paths, but that does not implement this issue’s memory-search config contract.

## Best Possible Solution

Implement a focused memory-search dimension contract: accept the chosen public key for defaults and per-agent overrides, resolve it with the existing memory-search config merge, pass it through OpenAI-compatible inline and batch embedding request bodies, include it in provider/cache/reindex identity, and update docs plus targeted tests. Preserve existing `outputDimensionality` behavior for providers that already use it.

## Risks / Open Questions

- A naive unconditional `dimensions` pass-through could regress OpenAI-compatible providers or models that reject that field; the provider should send it only when configured and appropriate.
- Changing embedding vector length must be included in cache and reindex identity so old vectors generated at a different length are not reused.
- The repo already has `outputDimensionality` for supported provider families, so the public naming or aliasing contract should be deliberate and documented to avoid two confusing knobs.
