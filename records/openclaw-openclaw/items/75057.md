## Summary

Keep open. Current main still deduplicates Feishu inbound receive, debounce, suppressed-record writes, and persistent finalize by account plus bare `message_id`, while the audio `file_key` that would distinguish reused IDs is only parsed later. The issue is also explicitly paired with open PR #75078, so maintainers should review, land, or close that implementation candidate before closing this issue.

## What This Changes

The issue reports that Feishu WebSocket audio messages with the same `message_id` but different audio content are treated as duplicates and the later attachment never reaches the agent.

## Best Possible Solution

Land #75078 or an equivalent Feishu-owned patch that computes one stable media-aware dedupe key for early receive claims, debounce suppression, persistent finalize, and suppressed-record writes, using audio `file_key` while preserving existing behavior for text and true identical retries.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive payloads for the same account with the same `message.message_id`, `message_type: "audio"`, and different JSON `file_key` values collide at the early claim before media parsing. The live Feishu ID-reuse part still depends on reporter logs or a live Feishu confirmation.

## Solution Assessment

Yes. A Feishu-local media-aware dedupe key is the narrow maintainable solution; changing shared core dedupe or adding a broad content-hash fallback would be larger than the reported same-ID/different-audio failure requires.
