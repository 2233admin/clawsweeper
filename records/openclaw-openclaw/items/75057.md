## Summary

Keep open. Current main still deduplicates Feishu inbound messages by `message_id` before it inspects media content, so the stated same-id/different-audio condition would drop the later audio event instead of delivering it to the agent. This is a narrow Feishu plugin bug candidate, not an implemented-on-main cleanup close.

## What This Changes

Feishu audio messages with reused `message_id` but different audio attachments should each be processed independently instead of being collapsed by inbound dedupe.

## Best Possible Solution

Keep the fix Feishu-owned: make the inbound dedupe key media-aware for audio/media events, likely using account, `message_id`, message type, and normalized `file_key` or `image_key`, while preserving message-id-only behavior for text and true duplicate retries. Add regression tests for same `message_id` with different audio `file_key` values and for identical audio retries still deduping.

## Reproduction Assessment

Yes for the OpenClaw side: two Feishu receive events under the same account with identical `message.message_id` and different audio `file_key` values will hit the message-id dedupe path before media resolution. Live Feishu reuse of the id is plausible from the report but not independently proven by logs.

## Solution Assessment

Yes, the best path is a narrow Feishu plugin dedupe-key fix with targeted tests. The shared persistent dedupe helper is behaving as designed; the channel is supplying a key that is too coarse for this Feishu media edge case.
