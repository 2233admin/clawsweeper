## Summary

Keep this PR open. Current main has absorbed part of the internal-scope cleanup, especially gateway-client scope forwarding, command-specific admin gates, and `/exec` persistence blocking for write-scoped internal callers. It has not implemented the PR’s central `chat.send` `CommandAuthorized` change, and it does not include the proposed request-scoped plugin runtime config sink guard. The PR discussion also records an unresolved authorization-policy concern, so this needs maintainer direction rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer direction. The best path is to preserve main's command-specific least-privilege model, decide whether command capability metadata or `requiredScopes` should be the canonical policy layer, and separately decide whether plugin runtime config writes need a generic request-scoped sink guard that covers all current config mutation APIs.

## Risks / Open Questions

- Merging the PR as-is could deny read-only or command-specific slash commands from `operator.write` internal `chat.send` clients before narrower scope policy can run.
- The plugin runtime config write policy remains unresolved; if revived, it should account for current `mutateConfigFile`, `replaceConfigFile`, and deprecated `writeConfigFile` APIs, not only the older write path.
- Because this changes gateway and plugin authorization behavior, it needs maintainer product/security direction and focused regression coverage across Web UI, TUI, mobile, ACP, and plugin-command callers.
