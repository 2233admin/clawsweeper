## Summary

Keep this PR open. Current main still has the stale cached-session writeback path the PR targets: the agent handler builds a session patch from a cached `entry`, includes `modelOverride` and `providerOverride` from that cached entry, then merges that patch over the fresh row re-read under the session-store lock. The related #70575 is a narrower follow-up for delivery fields, not a superseding implementation of this model override fix.

## Best Possible Solution

Keep this PR open and review or land it, or land an equivalent focused fix in `src/gateway/server-methods/agent.ts`: make `nextEntryPatch` contain only current request-derived fields, or build/merge store-owned fields from the fresh row inside `updateSessionStore`. Keep #70575 as the separate delivery-field follow-up and retain regression coverage for stale cached `loadSessionEntry` data racing with a fresh store row containing model overrides.
