## Summary

Keep this PR open. Current main still reads a cached session entry, builds `nextEntryPatch` with store-owned fields such as `modelOverride` and `providerOverride`, then merges that stale patch over the fresh session-store row inside `updateSessionStore`. The PR is still a plausible focused fix for the model-override race it describes, and the related #70575 is a narrower follow-up rather than a superseding implementation.

## Best Possible Solution

Keep this PR open and review or land it, or land an equivalent focused fix in `src/gateway/server-methods/agent.ts`: make `nextEntryPatch` contain only request-derived values, or build/merge store-owned fields from the fresh row inside `updateSessionStore`. Keep #70575 as the separate delivery-field follow-up and retain regression coverage for stale cached `loadSessionEntry` data racing with a fresh store row containing model overrides.
