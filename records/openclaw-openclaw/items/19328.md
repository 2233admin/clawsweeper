## Summary

Keep this PR open. Current main still has the stale cached-session writeback pattern the PR targets: the agent handler reads `entry` before the locked store update, builds a patch containing store-owned fields like `modelOverride` and `providerOverride` from that cached entry, then merges that patch over the freshly re-read store row. The related #70575 tracks a narrower residual delivery-field race, not a superseding implementation of this modelOverride fix.

## Best Possible Solution

Keep this PR open and review or land it, or land an equivalent focused fix in `src/gateway/server-methods/agent.ts`: make `nextEntryPatch` contain only request-derived values, or build/merge store-owned fields from the fresh row inside `updateSessionStore`. Keep #70575 as the separate delivery-field follow-up and retain regression coverage for stale cached `loadSessionEntry` data racing with a fresh store row containing model overrides.
