## Summary

Keep this PR open. Current main still has the stale cached-session writeback shape for `modelOverride` and `providerOverride`, and this branch is a focused candidate that removes those stale override fields from the agent session patch and adds regression coverage.

## What This Changes

The PR changes `src/gateway/server-methods/agent.ts` so the gateway agent session patch no longer carries several cached `entry` fields, and adds regression tests in `src/gateway/server-methods/agent.test.ts` for preserving fresh session-store fields such as `modelOverride`.

## Best Possible Solution

Land this PR or an equivalent focused gateway change that keeps `modelOverride` and `providerOverride` out of stale cached writeback patches, keep the new stale-cache/fresh-store regression coverage, and leave the delivery/channel follow-up to #70575 unless maintainers intentionally broaden the scope.

## Reproduction Assessment

Yes. The source-level reproduction path is high confidence: a stale cached `entry` can place undefined override fields into `nextEntryPatch`, and `mergeSessionEntry` then spreads that patch over the fresh store row. I did not run the tests because this review is explicitly read-only.

## Solution Assessment

Yes for the reported modelOverride/providerOverride failure. Omitting cached override fields from the patch is the narrowest maintainable fix because the fresh store row is already loaded inside `updateSessionStore`; residual delivery-field races have their own open follow-up.
