## Summary

Current main still does not fix the reported Mattermost connected-but-deaf WebSocket behavior. The Mattermost socket wrapper has no ping/pong or ready-state liveness contract, the runtime only polls the bot account `update_at`, and the generic health monitor only performs stale-socket restarts when a channel reports `lastTransportActivityAt`. The related PR #57621 is still an open implementation candidate in the supplied GitHub context, not merged behavior.

## Best Possible Solution

Keep this issue open until PR #57621 or an equivalent Mattermost-owned fix lands. The likely fix belongs in `extensions/mattermost/src/mattermost/monitor-websocket.ts`: add active WebSocket liveness detection such as ping/pong with missed-pong termination, optionally publish provider-proven `lastTransportActivityAt` if the generic health monitor participates, and add focused regression tests for connected-but-deaf WebSocket recovery.
