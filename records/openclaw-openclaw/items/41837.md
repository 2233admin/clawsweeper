## Summary

Keep this issue open. Current main still does not implement Mattermost WebSocket ping/pong, pong-timeout termination, or another provider-proven transport liveness signal for a connected-but-deaf WebSocket. The newer Mattermost watchdog polls the bot account `update_at` and can catch a disable/enable-style failure, but it does not prove inbound WebSocket frames are still flowing. The generic gateway stale-socket policy also requires `lastTransportActivityAt`, and Mattermost does not publish that field today. Open PR #57621 remains an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this open until PR #57621 or an equivalent Mattermost-owned fix lands. The fix should add provider-proven WebSocket liveness, ideally ws ping/pong with missed-pong termination, publish `lastTransportActivityAt` only from real transport evidence if the generic health monitor should participate, and add focused regression tests for connected-but-deaf WebSocket recovery. After merge, maintainers can close this as implemented and consolidate the related duplicate reports.
