## Summary

Keep this issue open. Current main still has Mattermost REST `update_at` polling for one stale-session case, but it still does not implement WebSocket ping/pong, pong-timeout termination, or `lastTransportActivityAt` reporting for Mattermost. The generic channel health policy cannot detect a connected-but-deaf Mattermost socket without provider-proven transport activity, and open PR #57621 remains an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this open until PR #57621 or an equivalent Mattermost-owned fix lands. The fix should add provider-proven WebSocket liveness, ideally ping/pong with missed-pong termination, publish `lastTransportActivityAt` only from real transport evidence if the generic health monitor should participate, and add focused regression tests for connected-but-deaf WebSocket recovery. After merge, maintainers can close this as implemented or consolidate the related duplicates.
