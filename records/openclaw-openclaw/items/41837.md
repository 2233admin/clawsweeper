## Summary

Keep this issue open. Current main still does not detect a Mattermost WebSocket that remains connected but stops delivering inbound frames: the Mattermost monitor has no ping/pong or other provider-proven transport activity signal, and the generic stale-socket policy intentionally requires `lastTransportActivityAt` before it restarts a connected channel. The open PR #57621 remains an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open until PR #57621 or an equivalent Mattermost-owned fix lands. The fix should add a provider-proven liveness signal in `extensions/mattermost/src/mattermost/monitor-websocket.ts`, such as WebSocket ping/pong with missed-pong termination, optionally publish `lastTransportActivityAt` only from real transport evidence, and add focused regression tests for half-open/deaf WebSocket recovery.
