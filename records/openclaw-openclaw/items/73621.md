## Summary

Current main still has the reported failure mode: Control UI follow-up messages are queued only in in-memory `chatQueue`, and both session-switch paths clear that queue and the active run id. The related open reports #45952 and #51549 cover reconnect/refresh durability, but this session-switch trigger is distinct enough that it should remain tracked unless maintainers choose one broader canonical queue-durability issue.

## Best Possible Solution

Keep this open and fix the Control UI queue model. The conservative implementation is to make queued messages session-scoped and preserve the original session's queue when navigating away, then flush it when that session's run becomes idle again. A broader durable fix could share design with #45952 and #51549 by adding localStorage or server-side queue/ACK semantics, but this issue needs at least a focused regression test for switching sessions with a pending queued message.
