## Summary

Keep open. Current main still returns string chat content before any content-block normalization, so JSON-stringified vLLM/OpenAI-compatible content-block arrays remain raw strings. This PR is still relevant, but the latest proposed type-only JSON-array heuristic has a real review-noted risk of dropping unrelated JSON string content.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best fix is a conservative shared normalization helper that parses strings only when they have a recognizable chat content-block array shape, mirrors real-array extraction for text-only and mixed media/text blocks, preserves arbitrary JSON strings, and adds regression tests for vLLM text-only blocks, mixed blocks, invalid JSON fallback, and non-chat typed JSON arrays. After that, review whether gateway prompt/string-body and streaming boundaries should route through the same helper or remain separate.

## Risks / Open Questions

- Closing this PR now would discard a focused implementation candidate while current main still preserves JSON-stringified content-block arrays as raw strings.
- Merging the current PR as-is could drop unrelated JSON string content when a parsed array contains objects with a `type` key but no valid chat text blocks.
- The proposed test plan is still unit-only; the PR body leaves live vLLM `openai-completions` verification unchecked.
- Because `extractTextFromChatContent` is shared across agent, gateway, and reply-text paths, the parse guard should be conservative and preserve arbitrary JSON strings unless the content-block shape is recognizable.
