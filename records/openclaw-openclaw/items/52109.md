## Summary

Keep this PR open. Current main does not implement the PR's intended `every`-schedule refire-gap behavior: `applyJobResult` still applies the `MIN_REFIRE_GAP_MS` lower bound only to cron-expression schedules, while `every` jobs are assigned `naturalNext` directly. The PR also still deserves maintainer review because prior review comments raised cadence/anchor edge cases that are not resolved by current main.

## Best Possible Solution

Do not close this PR as obsolete. Either review and land a narrowly tested scheduler fix for `every` jobs whose computed next run is at or before the just-finished run, or replace it with a maintainer-owned follow-up that covers the same overrun case plus the exact-boundary and preserved-anchor edge cases. Keep the security/supply-chain scope unchanged: scheduler code plus focused cron tests only.
