## Summary

Keep this PR open, but do not merge it as-is. The underlying `every` schedule refire bug still exists on current `main`, while the PR branch would conflict with the newer unresolved cron next-run behavior that now clears unschedulable cron runs instead of inventing a 2-second retry.

## What This Changes

The PR changes cron `applyJobResult` so successful `every` jobs apply a 2-second refire floor when the computed next run is not after `endedAt`, and adds focused regression tests for that scheduling edge case.

## Best Possible Solution

Keep the scheduler fix, but rebase or replace the branch so cron schedules continue using the current unresolved-next-run resolver, malformed `everyMs` stays unscheduled, valid future `every` ticks keep their natural cadence, and only completed `every` runs with a valid computed next run at or before `endedAt` get the minimum refire floor.

## Reproduction Assessment

Yes. A source-level reproduction is an `every` job whose execution reaches or exceeds its interval: current `main` can compute `naturalNext === endedAt` and then persist that as `nextRunAtMs`, making the job immediately due again; the PR body also includes live Docker reproduction details.

## Solution Assessment

No, not as currently proposed. The direction is narrow and useful, but the branch needs to preserve current `main`'s unresolved cron clearing contract, preferably by routing cron through the existing lower-bound resolver and applying the new floor only to valid `every` next-run values.
