## Summary

Keep open. Current main still applies the successful-run minimum refire gap only to cron-expression schedules, while successful `every` jobs persist `naturalNext` directly. The PR is not obsolete or implemented on main, but it needs a current-main update so it does not regress the newer unresolved cron next-run clearing behavior.

## Best Possible Solution

Keep this PR open and update it against current main. The best path is a narrow scheduler change that applies the 2s refire guard to successful `every` completions only when the computed next run would leave the job immediately due, while preserving valid future ticks, forced-run anchor semantics, malformed `everyMs` unscheduling, and current cron unresolved-next-run clearing. Add focused regression tests for overrun, exact-boundary, malformed `everyMs`, and `preserveSchedule` cases.

That leaves `every` schedules outside the persisted refire-gap guard.
