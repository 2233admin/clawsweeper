## Summary

Keep open. The PR may be a valid response-flag semantics tweak, but it should not be merged as the fix for #74837 because the linked report is about explicit runtime model propagation into the child run. Current `main` already attempts to persist model override fields before starting the child agent and the runtime reads stored overrides, while this PR only changes `modelApplied` flag reporting and adds no regression proving the child trajectory uses the requested model.

## What This Changes

The PR changes native sub-agent model planning and spawn acceptance response handling so `modelApplied` is true only when `sessions_spawn` includes a non-empty `model` argument.

## Best Possible Solution

Clarify the public meaning of `modelApplied`, then land a tested fix that proves an explicit `sessions_spawn.model` is persisted and actually used by the child run. If the intended change is only response flag semantics, add focused tests for explicit model, configured sub-agent default, and no-override cases, and do not treat that as resolving the linked runtime propagation bug.

## Reproduction Assessment

Unclear. The linked issue provides concrete release-version steps and trajectory evidence, but this read-only review did not run a live gateway/provider reproduction; source inspection shows a persistence path that should set stored overrides before the child `agent` call, so the remaining proof needs an executable spawn-to-trajectory regression.

## Solution Assessment

No. A flag-only patch is not the best way to solve the linked bug; the safer narrow path is to prove or repair runtime model propagation through the child session and then adjust `modelApplied` semantics only if that API meaning is confirmed.
