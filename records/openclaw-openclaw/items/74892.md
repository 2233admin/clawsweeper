## Summary

Keep open. The branch narrows the `modelApplied` response flag, but the linked report is about explicit `sessions_spawn.model` not taking effect at runtime; the patch does not change or prove the child run model-selection path, so it is not safe to merge as the fix for that bug yet.

## What This Changes

The PR changes native sub-agent model planning and spawn response handling so `modelApplied` is true only when the caller passes a non-empty model override.

## Best Possible Solution

Clarify the API meaning of `modelApplied`, then land a tested fix that proves an explicit `sessions_spawn.model` is persisted and used by the child run. If the desired change is only response flag semantics, add focused tests for explicit model, configured sub-agent default, and no-override cases, and do not close the linked runtime override bug from that flag-only change.

## Reproduction Assessment

Unclear. The linked issue gives concrete reproduction steps and trajectory evidence on `v2026.4.27`, but this read-only review did not run a live gateway/provider reproduction, and current source inspection shows an explicit model should be persisted before the child run.

## Solution Assessment

No. The current PR is not the best way to solve the linked issue because it only changes the accepted response flag; the safer path is to prove or repair explicit model propagation through the child session and runtime trajectory with regression coverage.
