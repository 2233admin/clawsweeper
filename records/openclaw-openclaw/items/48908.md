## Summary

Keep open. Current main still lacks HTTP 421/Misdirected Request handling in both the shared channel retry matcher and Telegram’s strict non-idempotent send retry predicate, so this PR is not obsolete. The proposed code/test path looks functionally correct, with one small docs alignment issue before merge.

## What This Changes

The PR adds HTTP 421/Misdirected Request handling to the shared channel API retry matcher and Telegram strict send retry predicate, with regression tests for both paths.

## Best Possible Solution

Land the narrow Telegram 421 retry fix after updating the retry policy docs, keeping the strict send retry predicate conservative for non-idempotent Telegram sends.

## Reproduction Assessment

Yes. A unit-level reproduction is an error shaped as HTTP 421 or `421 Misdirected Request` through the shared retry runner and Telegram `isSafeToRetrySendError`; current main does not retry that path, while the PR adds explicit coverage.

## Solution Assessment

Yes, with a docs alignment fix. Adding 421 recognition to both the default channel retry matcher and Telegram’s strict safe-send predicate is the narrow maintainable fix; broadly relaxing strict retries would risk duplicate messages.
