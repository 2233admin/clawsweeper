## Summary

Keep open. The report is current and actionable: current main has an admin `channels.start` RPC and internal channel stop/start primitives, but it still lacks standalone operator CLI lifecycle commands and public `channels.stop` / `channels.restart` RPC methods. The related items in the provided context do not supersede this specific recovery gap, and the broader health-monitor retry-budget concern should be split from the narrow CLI/RPC fix.

## What This Changes

The issue requests `openclaw channels start`, `stop`, and `restart` commands so operators can recover a linked but wedged channel account without QR re-pairing or restarting the container.

## Best Possible Solution

Add additive, admin-scoped channel lifecycle controls: a CLI wrapper for the existing `channels.start`, new typed `channels.stop` and `channels.restart` gateway methods backed by the existing channel manager, focused CLI/RPC tests, and docs/runbook updates that describe non-destructive recovery for linked accounts. Leave health-monitor retry-budget backoff or alerting as a separate follow-up.

## Reproduction Assessment

Yes for the requested CLI/API gap: current main can be verified by reading `registerChannelsCli`, the `channels.start` handler, the channel auth login-side call path, and the protocol method/schema lists. The transient DNS/Baileys wedged state was not live-reproduced in this read-only review, but the missing standalone recovery command is independently reproducible from source and docs.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable solution for the operator recovery problem: expose generic channel lifecycle seams through admin CLI/RPC instead of adding WhatsApp-specific recovery logic. The safer implementation boundary is to keep retry-budget policy and health-event changes out of the lifecycle-command PR.
