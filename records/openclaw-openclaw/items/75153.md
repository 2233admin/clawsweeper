## Summary

Keep open. Current main still has the gateway-side start primitive and internal channel manager start/stop hooks, but it does not expose standalone `openclaw channels start`, `stop`, or `restart` commands, and the public gateway method set still lacks `channels.stop` and `channels.restart`. The linked restart-hang report is useful context but does not supersede this operator recovery gap.

## What This Changes

The issue requests standalone CLI channel lifecycle commands so operators can start, stop, or restart an existing configured channel account without QR re-pairing or restarting the container.

## Best Possible Solution

Add additive, admin-scoped generic channel lifecycle controls: `openclaw channels start/stop/restart --channel <name> [--account <id>]`, typed `channels.stop` and `channels.restart` gateway methods backed by the existing channel manager, focused CLI/RPC tests, protocol and CLI docs, and a user-facing changelog entry crediting the reporter. Keep WhatsApp-specific recovery behavior and long-term health-monitor retry policy out of this focused change.

## Reproduction Assessment

Yes for the source-level CLI/RPC gap: current main exposes `channels.start` only through the login side effect and has no standalone lifecycle commands or stop/restart RPCs. The exact transient network failure was not live-reproduced, but it is not needed to verify the missing recovery surface.

## Solution Assessment

Yes. An additive admin-scoped CLI/RPC wrapper around the existing generic channel lifecycle manager is the narrowest maintainable fix; a WhatsApp-specific workaround or a health-monitor policy rewrite would be broader than the requested operator recovery path.
