## Summary

Keep open. Current main still has the operator recovery gap: the gateway exposes `channels.start` and the channel manager has start/stop primitives, but the `openclaw channels` CLI has no start, stop, or restart commands, and there is no public `channels.stop` or `channels.restart` RPC. The related open reload request #72997 overlaps, but this report adds a more concrete stopped-and-still-linked recovery path, so it should remain open and be clustered rather than closed as a duplicate.

## What This Changes

Expose non-destructive channel lifecycle CLI commands for start, stop, and restart so operators can recover a wedged linked channel without QR relink or container restart.

## Best Possible Solution

Add additive, admin-scoped channel lifecycle controls: CLI wrappers for start, stop, and restart, typed gateway protocol/schema entries for stop and restart, method discovery/scope updates, focused gateway and CLI tests, and docs/runbook guidance positioning this as non-destructive recovery for already-linked accounts. Keep automatic backoff or network-recovery policy changes as a separate follow-up.

## Reproduction Assessment

Yes for the actionable CLI/API gap: current main source and docs show `channels.start` exists only as an RPC/login side effect, while CLI and gateway discovery lack standalone start/stop/restart. The exact transient WhatsApp DNS failure was not live-reproduced, but the operator recovery gap is high-confidence from source inspection.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable fix for the immediate operator problem: expose the generic channel lifecycle seams through admin CLI/RPC instead of adding WhatsApp-specific recovery logic or changing the health-monitor retry policy in the same change.
