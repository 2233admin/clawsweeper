## Summary

Keep open. Current main still lacks standalone `openclaw channels start`, `stop`, or `restart` commands and lacks `channels.stop`/`channels.restart` Gateway RPC methods, while the existing `channels.start` RPC is reachable only as a login side effect. The related restart-hang report is separate closed context and does not implement this operator recovery path.

## What This Changes

Request standalone CLI commands to start, stop, and restart channel accounts without forcing QR login/re-pairing or a container restart.

## Best Possible Solution

Add additive, admin-scoped generic channel lifecycle controls: `openclaw channels start/stop/restart --channel <name> [--account <id>]`, typed `channels.stop` and `channels.restart` Gateway methods backed by the existing channel manager, focused tests, docs, and a user-facing changelog entry. Keep longer-term health-monitor backoff/notification policy as separate follow-up work.

## Reproduction Assessment

Yes for the source-backed CLI/RPC gap: current main exposes only `channels.start` and only the login path calls it from the CLI, while no standalone lifecycle commands or stop/restart RPCs exist. The exact network-wedge scenario is not live-reproduced here.

## Solution Assessment

Yes. An additive admin-scoped CLI/RPC wrapper around existing generic start/stop manager hooks is the narrowest maintainable fix for the operator recovery problem; WhatsApp-specific recovery policy changes should not be bundled into this focused repair.
