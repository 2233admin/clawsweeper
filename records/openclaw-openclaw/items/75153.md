## Summary

Keep open. Current main has a working `channels.start` Gateway RPC and internal channel start/stop manager hooks, but the operator CLI still exposes only list/status/capabilities/resolve/logs/add/remove/login/logout, and there is no `channels.stop` or `channels.restart` RPC surface. The report is current, source-backed, has no protected-label or maintainer-author blocker, and is narrow enough for a focused recovery-control PR; the broader auto-recovery-after-restart-budget concern should remain separate.

## What This Changes

Expose channel lifecycle start/stop/restart controls via `openclaw channels` so operators can recover a configured, still-linked account without QR re-pairing or container restart.

## Best Possible Solution

Add additive, admin-scoped generic channel lifecycle controls: a CLI wrapper for the existing `channels.start`, typed `channels.stop` and `channels.restart` Gateway methods backed by the existing channel manager, focused unit tests, and docs/runbook updates that present this as non-destructive recovery for linked accounts. Keep automatic long-term retry/backoff or alerting behavior as a separate health-monitor follow-up.

## Reproduction Assessment

Yes for the CLI/API gap: current main can be checked by reading `src/cli/channels-cli.ts`, `src/cli/channel-auth.ts`, the Gateway method registry, and protocol schemas. The transient DNS/Baileys wedge itself is not live-reproduced here, but the missing recovery command is source-reproducible.

## Solution Assessment

Yes, the requested direction is the narrowest maintainable solution for the operator problem: expose existing generic channel lifecycle seams through admin CLI/RPC rather than adding WhatsApp-specific recovery logic. The broader restart-budget recovery policy should be handled separately.
