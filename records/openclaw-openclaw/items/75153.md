## Summary

Keep open. Current main still lacks standalone `openclaw channels start`, `stop`, and `restart` commands, and the gateway method surface still exposes only `channels.start` plus logout/status, so the operator recovery gap remains valid.

## What This Changes

The issue requests CLI channel lifecycle commands that let operators start, stop, or restart an existing configured channel account without QR re-pairing or restarting the container.

## Best Possible Solution

Add additive, admin-scoped generic channel lifecycle controls: `openclaw channels start/stop/restart --channel <name> [--account <id>]`, typed `channels.stop` and `channels.restart` gateway methods backed by the existing channel manager, focused tests, docs, and a user-facing changelog entry. Keep health-monitor backoff/notification policy as separate follow-up work.

## Reproduction Assessment

Yes for the source-backed CLI/RPC gap: current main exposes only `channels.start`, and the CLI calls it only from the login reconciliation path while no standalone lifecycle commands or stop/restart RPCs exist. The exact network-wedge scenario was not live-reproduced here.

## Solution Assessment

Yes. An additive admin-scoped CLI/RPC wrapper around existing generic start/stop manager hooks is the narrowest maintainable fix for the operator recovery problem; WhatsApp-specific retry policy changes should not be bundled into this focused repair.
