## Summary

Keep open. Current main has a gateway `channels.start` RPC and internal channel start/stop manager hooks, but the operator CLI still exposes only list/status/capabilities/resolve/logs/add/remove/login/logout, and there is no `channels.stop` or `channels.restart` RPC/CLI surface. The request is current, source-backed, and narrow enough for a focused recovery CLI/API PR; the broader automatic backoff-after-restart-budget concern should stay separate.

## What This Changes

Expose `openclaw channels start`, `stop`, and `restart` commands so operators can recover an existing linked channel account without QR relink or container restart.

## Best Possible Solution

Add additive, admin-scoped channel lifecycle controls: a CLI wrapper for the existing `channels.start`, new typed `channels.stop` and `channels.restart` gateway methods backed by the existing channel manager, focused unit tests, and docs/runbook updates that position this as non-destructive recovery for linked accounts. Keep automatic long-term backoff/notification behavior as a separate follow-up.

## Reproduction Assessment

Yes for the CLI/API gap: current main can be checked by reading `src/cli/channels-cli.ts`, `src/cli/channel-auth.ts`, and the gateway method registry/search results. The transient DNS/Baileys wedged state itself was not live-reproduced in this read-only review, but the missing operator recovery command is independently reproducible from source and docs.

## Solution Assessment

Yes, the proposed direction is the narrowest maintainable fix for the operator problem: expose existing generic channel lifecycle seams through admin CLI/RPC rather than adding WhatsApp-specific recovery logic. The broader health-monitor retry-budget behavior should be handled separately so this PR stays focused.
