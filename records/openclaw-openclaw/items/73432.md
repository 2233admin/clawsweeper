## Summary

Keep open. Current main still skips QMD embedding when QMD is in lexical `search` mode, and open PR #73512 directly references this issue with closing syntax and targets the same scheduler guard, so this should wait for that PR and the product contract decision rather than be closed or queued as duplicate repair work.

## What This Changes

The issue requests that QMD automatically run embedding on the configured `memory.qmd.update.embedInterval`, including when `searchMode` is omitted and therefore defaults to lexical `search`.

## Best Possible Solution

Review PR #73512 as the canonical implementation candidate. The desired end state should either preserve lexical `search` as BM25-only with clearer config guidance, or intentionally treat an explicit raw `embedInterval` setting as an embedding opt-in and update the scheduler, backend config state, tests, docs, and schema help together.

## Reproduction Assessment

Yes. A high-confidence static and unit-level reproduction exists on current main: configure QMD with lexical `searchMode: "search"` and `embedInterval: "5m"`, advance the timer, and the existing test expects no `update` or `embed` subprocess calls; the reporter's omitted `searchMode` resolves to that same lexical mode.

## Solution Assessment

Unclear. The linked PR is the right place to resolve this, but the best fix depends on maintainer product intent and likely needs an explicit-config flag rather than treating any positive resolved `embedIntervalMs` as user intent.
