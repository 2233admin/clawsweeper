## Summary

Keep this PR open: the reported Telegram abort bleed is still real on current main, but the PR branch is stale and edits the old `bot.ts` implementation instead of the current `bot-core.ts` fetch wrapper where the shared abort signal is still applied to every Telegram API request.

## What This Changes

The PR attempts to scope Telegram polling restart abort handling to `getUpdates` and updates fetch-abort regression tests so outbound requests such as `sendMessage` are not canceled by the polling restart signal.

## Best Possible Solution

Land a current-main Telegram plugin fix that scopes `opts.fetchAbortSignal` to the long-poll `getupdates` fetch inside `bot-core.ts`, preserves request-level aborts and timeout behavior, and adds regression coverage proving `getupdates` still aborts while `sendmessage` and `sendchataction` survive polling restart cleanup.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: create a Telegram bot with `fetchAbortSignal`, start a non-`getupdates` client fetch such as `sendmessage`, abort the polling session's controller, and current `bot-core.ts` will abort that outbound request through the shared controller signal.

## Solution Assessment

No. The idea is the right narrow fix, but this PR applies it to the old `bot.ts` shape; on current main the maintainable fix belongs in `createTelegramBotCore` and must account for `extractTelegramApiMethod` returning lower-case method names such as `getupdates`.
