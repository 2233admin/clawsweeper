## Summary

Keep this PR open. Current main still threads the polling session fetchAbortSignal into the shared Telegram client fetch wrapper, so aborting the polling-cycle controller can still abort non-getUpdates Telegram API requests. Later liveness work reduces some false stall restarts, but it does not implement the PR's intended getUpdates-only abort boundary.

## Best Possible Solution

Keep the PR open and ask for a rebase or maintainer follow-up. The right fix is a Telegram-plugin change in the current bot-core/polling-session structure that scopes polling-restart aborts to getUpdates, with focused tests proving getUpdates is still aborted during restart while sendMessage/sendChatAction survive.
