## Summary

Keep this PR open. Current main still aborts every Telegram client fetch when the polling session's shared fetch abort signal fires, so the PR's central behavior is not implemented. There are later polling-liveness mitigations that reduce false watchdog restarts during recent non-getUpdates API activity, but they do not scope polling-restart aborts to getUpdates or protect an in-flight send from the shared fetch abort path.

## Best Possible Solution

Keep the PR open and ask for an update against current main. The best path is a Telegram-plugin fix that scopes polling-restart aborts to getUpdates, or separates polling-restart abort from full gateway shutdown semantics, while preserving request timeouts and transport rebuild behavior. Add regression coverage for getUpdates abort plus sendMessage/sendChatAction surviving a polling restart.
