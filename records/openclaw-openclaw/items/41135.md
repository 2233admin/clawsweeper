## Summary

Keep this issue open. Current main still has auth-profile ordering, session-sticky auto-pinning, cooldown/failover rotation, and a narrow provider-owned preferred-profile hook, but it does not implement the requested opt-in provider-profile pool routing layer with named strategies, task/none stickiness, per-profile in-flight/concurrency limits, pool state in status JSON, or `models routing` CLI controls.

## Best Possible Solution

Keep this as an open design/feature issue. The best path is an additive core auth-routing contract: define strict config/schema for pool mode, strategy, stickiness, profile list, weights, and concurrency limits; add runtime in-flight/cooldown-aware profile selection in the embedded runner/provider auth path; expose route state through `models status --json`; then add CLI controls after config and runtime semantics are stable. Provider hooks can participate without owning the generic scheduler.

## Risks / Open Questions

- A correct implementation would span core auth config/schema, embedded runner selection, status JSON, CLI controls, docs, and provider hook participation; treating it as a provider-only tweak could leave batch routing semantics inconsistent.
- The Gemini probe-timeout example may also deserve a narrower bug investigation, but that does not replace the broader missing provider-pool routing contract.
- The existing provider-owned `resolveAuthProfileId` hook is a useful seam, but expanding it into pool routing needs a backwards-compatible design for third-party providers.
