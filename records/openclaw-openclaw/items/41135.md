## Summary

Keep this issue open. Current main still has auth profile ordering, cooldown/disabled failover, provider-owned single-profile preference hooks, and session pinning, but it does not implement the requested opt-in provider-profile pool routing layer with pool config, named strategies, task/none stickiness, in-flight/concurrency state, weights, quota-aware state, pool-state JSON, or models routing CLI controls. The recent prior ClawSweeper comment reached the same keep-open conclusion, and the current source at 28ff82dc still matches that assessment.

## Best Possible Solution

Keep this as an open core feature/design issue. The best path is an additive auth-routing contract with strict config/schema for pool mode, strategies, stickiness, profile lists, weights, and concurrency limits; runtime in-flight/cooldown-aware selection in the embedded runner and fallback path; pool state in models status --json; and CLI controls after the config/runtime semantics are stable. Provider hooks can supply hints, but the generic scheduler and state model should be core-owned.
