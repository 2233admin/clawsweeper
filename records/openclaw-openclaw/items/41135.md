## Summary

Keep this issue open. Current main still implements auth-profile ordering, session-sticky auto-pinning, cooldown/failover rotation, and a narrow provider-owned profile-preference hook, but it does not implement the requested opt-in provider-pool routing layer with routing config, named strategies, per-profile in-flight/concurrency limits, task/none stickiness, pool state in status JSON, or `models routing` CLI controls. The latest issue context also includes a prior Codex/ClawSweeper review keeping it open and a stale-label removal on April 28, 2026.

## Best Possible Solution

Keep this as an open design/feature issue. The best implementation path is an additive core auth-routing contract: define strict config/schema for pool mode, strategy, stickiness, profile list, weights, and concurrency limits; add runtime in-flight/cooldown-aware profile selection in the embedded runner/provider auth path; expose route state through `models status --json`; and add CLI controls after the config and runtime semantics are stable. Provider-specific hooks can then participate without owning the generic pool scheduler.
