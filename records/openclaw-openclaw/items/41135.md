## Summary

Keep open. Current main still has auth-profile order, last-used/cooldown rotation, session stickiness, failure-driven profile failover, provider usage/status snapshots, and a narrow provider-owned profile preference hook, but it does not implement the requested opt-in provider-pool routing layer with named policies, per-profile in-flight/concurrency limits, task/none stickiness, routing config, route state in status JSON, or `models routing` CLI controls. The provided discussion also includes a prior Codex review that kept this open, followed by stale-label removal on April 28, 2026.

## Best Possible Solution

Keep this as an open design/feature issue. The best implementation path is an additive core auth-routing contract: define config/schema for pool mode, strategy, stickiness, profile list, weights, and concurrency limits; add runtime in-flight/cooldown-aware profile selection in the embedded runner/provider auth path; expose route state through `models status --json`; and add CLI controls only after the config and runtime semantics are stable.
