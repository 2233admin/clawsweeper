## Summary

Keep this issue open. Current main has auth-profile ordering, lastUsed/cooldown rotation, session pinning, linear failover, and a provider hook that can prefer one profile from an existing order, but it still lacks the requested opt-in provider-profile pool routing layer: native pool config, named strategies, task/none stickiness, weights, in-flight/concurrency limits, quota-aware state, pool-state JSON, and a `models routing` CLI surface. The prior ClawSweeper review comment reached the same conclusion, and current main at d22a851 remains consistent with that assessment.

## Best Possible Solution

Keep this as an open core feature/design issue. The best path is an additive auth-routing contract with strict config/schema for pool mode, strategies, stickiness, profile lists, weights, and concurrency limits; runtime in-flight/cooldown-aware selection in the embedded runner and fallback path; observable route state through `models status --json`; and CLI controls after the config and runtime semantics are stable. Provider hooks can participate, but the generic scheduler and state model should be core-owned.
