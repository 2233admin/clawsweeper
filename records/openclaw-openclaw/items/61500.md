## Summary

Current main now covers the core remote file I/O problem through the bundled file-transfer plugin: agents can fetch, list, directory-fetch, and write files on paired nodes via dedicated node-host commands and tools, with deny-by-default path policy and gateway-side node.invoke policy enforcement. This did not land as literal system.read/system.write or automatic read/write/edit retargeting, but it addresses the reported need to avoid fragile exec/heredoc workarounds for node-host file reads and writes.

## What This Changes

The issue requests node-protocol file I/O commands so standard read/write/edit tools can operate on a remote node host when exec is routed there.

## Best Possible Solution

Keep the dedicated file-transfer plugin as the current implementation for node-host file reads and writes because it has explicit node commands, per-node path policy, approval hooks, integrity checks, and gateway-side enforcement. If maintainers still want standard edit-tool ergonomics, that should be a narrower follow-up on top of this policy surface rather than reopening the broad system.read/system.write request.

## Reproduction Assessment

Not applicable as a feature request rather than a failing bug report. The current-main source check is high-confidence: remote node file fetch/write paths now exist through file_fetch/file_write, while literal system.read/system.write names were not adopted.

## Solution Assessment

Yes for the reported remote file I/O gap: the current dedicated file-transfer surface is safer and more explicit than a generic system.read/system.write protocol primitive. It does not retarget the built-in read/write/edit tools, so any remaining edit-tool UX should be tracked as a separate, narrower enhancement.
