## Summary

Keep open. Current main still preserves the reported routing chain: allowed elevated sessions without an inline/session/configured elevated default fall back to `on`, omitted `exec` calls inherit that active level, and elevated `auto` routing resolves to `gateway`. The latest release tag still contains the same behavior, and the linked default-off PR was closed unmerged, so this needs maintainer product/security judgment rather than cleanup closure.

## What This Changes

The issue asks for enabled elevated exec to keep sandbox exec as the default unless a call explicitly requests elevation, instead of routing all active elevated sessions to host exec.

## Best Possible Solution

Maintainers should explicitly choose the elevated-mode security contract. If sandbox-default opt-in elevation is desired, update default resolution, directive persistence/status, exec inheritance, prompt wording, docs, and regression tests together; if default-on host execution is intentional, keep the implementation but make the exec docs and security guidance state that tradeoff clearly.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: a sandboxed, allowed elevated session with no explicit elevated default resolves to `on`; an omitted `exec` call inherits that state; and elevated `auto` routing resolves to `gateway`. I did not run a live Discord/Docker session because this review was read-only.

## Solution Assessment

Unclear. The requested default-off or ask-as-opt-in behavior is a plausible safer direction, but it conflicts with current docs and tests that define default-on elevated mode and treat `ask` as equivalent to `on`; maintainer product/security approval should decide the contract before code changes.
