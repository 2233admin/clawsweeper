## Summary

Current main still has the reported behavior: when elevated is allowed and no directive/session/config default is set, the reply path falls back to elevated level `on`; `exec` then treats that as an elevated request and routes elevated `auto` execution to `gateway` unless the target is `node`. The docs now describe `/elevated ask` as an alias for `on` and `agents.defaults.elevatedDefault` as defaulting to `on`, so the reporter's requested sandbox-default, opt-in-per-call elevation behavior remains a real product/security decision rather than something already implemented. The linked fix PR #46908 is closed and unmerged, and I found no open PR closing this issue.

## Best Possible Solution

Keep this open for maintainer decision on the elevated-mode contract. The concrete product path is to decide whether enabled elevated access should default to `off` and whether `ask` should mean sandbox-by-default with only `exec(elevated: true)` escaping to gateway/node. If yes, update the reply default resolution, exec elevated-mode handling, system prompt/directive text, docs, and focused regression tests around sandboxed sessions, `tools.exec.host=auto|sandbox`, and explicit per-call `elevated` values.
