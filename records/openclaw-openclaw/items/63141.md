## Summary

Keep this issue open. Current main still uses a static 100000-token parent fork cap, skips oversized parent inheritance with only a log warning, and starts the child fresh; the related compaction-provider and checkpoint work exists but is not wired into this overflow branch.

## What This Changes

Request auto-compaction, a visible warning, or a model-aware cap when `session.parentForkMaxTokens` is exceeded instead of silently starting a fresh child session.

## Best Possible Solution

Add explicit parent-fork overflow handling in the session owner path: compact the parent before inheritance with bounded failure behavior, or emit a rate-limited visible/operator signal when inheritance is skipped, and update the fork cap logic only if selected-model context metadata is available safely at this point. Lock the chosen behavior with focused session tests and update the public session/config docs.
