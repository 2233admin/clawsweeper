## Summary

Keep open. Current `origin/main` still writes the request-side OpenAI `model` value into chat-completions JSON and SSE responses, and the documented `x-openclaw-model` surface is only a request override, not response attribution. The request is narrow and still valid for the Gateway API.

## What This Changes

Expose the actual backend provider/model used by `/v1/chat/completions` in a response body field or header so third-party clients can display it.

## Best Possible Solution

Add actual-model attribution for Gateway chat completions while preserving request routing: keep OpenAI `model` as the agent target, expose the backend provider/model additively, and document/test the behavior for both normal and streaming responses.

## Repair Work Prompt

Fix the Gateway OpenAI-compatible `/v1/chat/completions` response attribution request. Current `origin/main` echoes the request-side `model` agent target (`openclaw`, `openclaw/<agentId>`) into JSON and SSE chunks and does not set a response header with the actual backend provider/model. Add an additive actual-model attribution path, preferably `X-OpenClaw-Model: <provider>/<model>`, for non-streaming and streaming responses. Preserve the documented agent-first contract: do not change request routing, do not change `/v1/models` to list provider models, and do not make the JSON `model` field a raw provider id unless a maintainer explicitly chooses that. Reuse existing actual model selection data where possible, such as `onModelSelected` or `result.meta.agentMeta.provider/model`, and handle fallback-selected models. Update `src/gateway/openai-http.test.ts` to cover the header for non-streaming and streaming paths, including `x-openclaw-model` override behavior if feasible. Update `docs/gateway/openai-http-api.md` with the response header contract. Add a user-facing `CHANGELOG.md` entry under the active version if this API change is considered user-facing, credited to @k9spy and kept as one single-line bullet.
