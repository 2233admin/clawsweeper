## Summary

Keep this open. Current main only covers the sandbox portion through the existing agent-scoped sandbox config; tag-based provider routing, single-config multi-instance declarations, and gateway-wide path-deny enforcement are still absent from the strict schema and runtime chokepoints, and the remaining work is security/product design rather than conservative cleanup.

## What This Changes

The issue requests config-schema support for global sandbox policy, tag-based routing rules, multi-instance declarations, and gateway node path denylists.

## Best Possible Solution

Keep the existing `agents.defaults.sandbox` / `agents.list[].sandbox` path, then split the remaining work into maintainer-designed tracks: tag-aware provider routing semantics, multi-gateway instance inventory/validation if the profile model is insufficient, and a gateway-wide path-deny contract with explicit runtime coverage and tests.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence check is current-main source and docs inspection: sandbox mode exists under agent config, while `routing.rules`, `instances`, and `gateway.nodes.denyPaths` are not present in the strict config/runtime surfaces.

## Solution Assessment

No, not as one bundled schema addition. The safer maintainable path is to separate the already-supported sandbox concern from the routing, multi-instance, and path-deny designs, then implement each with matching docs, schema metadata, runtime enforcement, and tests.
