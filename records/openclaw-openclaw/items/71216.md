## Summary

Keep this open. Current main only covers part of the request: sandboxing is already modeled under agent config, but tag-based provider routing, single-config multi-instance declarations, and core `gateway.nodes.denyPaths` are still absent from the strict config schema and runtime chokepoints.

## What This Changes

The issue requests first-class config schema support for global sandbox policy, tag-based routing rules, multi-instance declarations, and gateway node path denial.

## Best Possible Solution

Keep the existing `agents.defaults.sandbox.mode` path, then split the remaining work into narrower maintainer-designed tracks: tag provenance and provider-route resolution, single-config multi-instance validation, and a gateway-wide path-deny contract that defines exactly which filesystem and node surfaces must enforce it.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The high-confidence check is current-main schema and docs inspection: the requested missing fields are absent from strict schema surfaces, while `agents.defaults.sandbox.mode` is already present.

## Solution Assessment

No, not as a single bundled change. The maintainable path is to separate the already-supported sandbox concern from the remaining routing, multi-instance, and security-enforcement designs before implementation.

## Risks / Open Questions

- The desired source of `routing.rules.match.tags` is undefined: tags could come from skills, tools, sessions, agents, channels, or plugin metadata, and those have different trust boundaries.
- A core `denyPaths` policy needs an explicit enforcement boundary across local filesystem tools, node-host file-transfer commands, sandbox FS bridges, elevated exec, and plugin-owned tools; the issue does not yet define which paths must be covered.
- The four requested surfaces are not one narrow implementation unit; landing them together risks mixing config schema, provider routing, multi-instance process management, and security policy decisions.
