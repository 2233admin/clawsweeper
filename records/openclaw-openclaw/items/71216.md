## Summary

Keep this open. Current main already has agent-scoped sandbox configuration, but the requested top-level `routing.rules`, single-config `instances`, and core `gateway.nodes.denyPaths` enforcement contract are still absent; the remaining work is broad and security-sensitive enough to need maintainer design.

## What This Changes

The issue requests config schema and gateway enforcement support for global sandbox mode, tag-based provider routing, multi-instance declarations, and gateway-wide node path denial.

## Best Possible Solution

Keep the existing `agents.defaults.sandbox` / per-agent sandbox model, then split the remaining work into maintainer-designed tracks: provider routing based on trusted metadata, multi-instance declaration versus the documented separate-profile isolation model, and a gateway-wide path-deny contract with explicit coverage for node commands, file tools, elevated execution, sandboxes, and plugin-owned operations.

## Reproduction Assessment

Not applicable as a feature/config-schema request. The verification path is source and docs inspection: current main has agent-scoped sandboxing, while `routing.rules`, `instances`, and core `gateway.nodes.denyPaths` are not present in the strict schema or gateway node policy.

## Solution Assessment

No, not as one bundled schema addition. The safer solution is to keep the already-supported sandbox path and design routing, instance validation, and path denial separately so each has a clear enforcement boundary and tests.
