## Summary

Keep open. Current main still exposes the full realtime consult tool path but does not implement the requested low-latency Voice Call memory/session context lookup, and the maintainer discussion plus adjacent PR context confirm the remaining gap is real rather than already shipped or superseded.

## What This Changes

The issue requests a default-off fast memory/session context path for realtime Voice Call conversations, with full agent consult kept as fallback for deeper work.

## Best Possible Solution

Implement a Voice Call-owned, default-off `realtime.fastContext` path: config/schema/manifest/docs, partial-transcript accumulation and normalization, bounded memory/session search through the public memory-host-search seam, explicit snippet or miss injection into the realtime session, reset support for multiple context questions, and full `openclaw_agent_consult` fallback for deeper work.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: enable Voice Call realtime with `realtime.toolPolicy` not `none`; remembered-context questions can only invoke the full consult handler, and non-final transcript chunks are ignored before call-state processing.

## Solution Assessment

Yes. The requested default-off fast-context path is the narrowest maintainable direction because it keeps Voice Call-specific latency behavior inside the plugin, uses an existing public memory search seam, and preserves full consult as fallback instead of changing core realtime semantics.
