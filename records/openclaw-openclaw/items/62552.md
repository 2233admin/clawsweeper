## Summary

Keep open. Current main still has the ACP bridge behavior this PR is trying to fix, so the PR is not obsolete; however it should not merge as-is because the task-maintenance change can mark an active cron job lost based on a stale or terminal stored session row. The existing Codex review comment identifies a real blocker, even though the Greptile summary correctly notes that the ACP portions are targeted and tested.

## What This Changes

The PR changes ACP bridge fallback sessions to `acp-bridge:<uuid>`, adds raw/canonical session-key matching for pending ACP prompts, and updates task maintenance to treat terminal child sessions as missing backing sessions for cron, cli, and subagent tasks.

## Best Possible Solution

Keep the ACP bridge fix direction, but revise the task-maintenance portion so active cron jobs remain governed by cron runtime ownership and durable cron run history. If terminal stored sessions should affect cron, the code should prove it is checking the actual execution session for that run, not the job's configured delivery/context session key.

## Reproduction Assessment

Yes. The ACP gap is source-provable on current main with the PR's added unit-test scenarios, and the blocking cron regression can be reproduced with a stale active cron task whose `sourceId` is active while `childSessionKey` points at a terminal stored session.

## Solution Assessment

No. The ACP fallback-key and raw/canonical matching changes are narrow, but extending terminal child-session gating to active cron tasks is not the safest fix; cron should stay keyed to runtime ownership unless the checked session is the actual execution session.
