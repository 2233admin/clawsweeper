## Summary

Keep open. Current main still lacks MCP serve scope selection, and this PR is the active implementation candidate, but the patch has a blocking auth-scope correctness issue: explicit `--scope` can be overridden by cached device-token scopes in `GatewayClient`, so a least-privilege MCP profile may reconnect with broader scopes than requested.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` flag allowlisted to `operator.read`, `operator.write`, and `operator.approvals`, threads it through the MCP server and bridge, and updates docs, changelog, and focused tests.

## Best Possible Solution

Update this PR so the effective Gateway connect scopes match an explicit MCP `--scope` selection even when a cached device token exists, preserve the current default read/write/approvals behavior when `--scope` is omitted, and add a regression test for the cached-token path before merging.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor explicit MCP scopes with cached tokens:** `src/mcp/channel-bridge.ts:126`
  - body: This only passes the requested scopes into `GatewayClient`, but `GatewayClient` reuses cached device-token scopes before `opts.scopes`. If this device already has a broader stored operator token, `openclaw mcp serve --scope operator.read` can still connect with the stored broader scopes, defeating the least-privilege profile this flag is meant to create. Please make explicit MCP scopes control the effective connect scope set for cached-token auth, or isolate the bridge identity, and add a regression for that path.
  - confidence: 0.87
