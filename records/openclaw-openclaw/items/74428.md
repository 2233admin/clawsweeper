## Summary

Keep open. Current main still lacks MCP serve scope selection, and this PR is the focused replacement for the now-closed broader PR, but the patch needs auth-scope follow-up before merge because explicit MCP scopes can be bypassed by the existing cached device-token scope path.

## What This Changes

This PR adds a repeatable `openclaw mcp serve --scope` flag limited to read, write, and approvals scopes, threads it through the MCP server and bridge, updates docs/changelog, and adds CLI/server tests.

## Best Possible Solution

Update this branch so `--scope` is authoritative across the MCP bridge auth paths it documents, including stored device-token reconnects, while preserving the default read/write/approvals request when the flag is omitted and keeping the explicit allowlist restricted to the intended bridge scopes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor explicit scopes with cached device tokens:** `src/mcp/channel-bridge.ts:126`
  - body: The new flag only changes the `scopes` option passed to `GatewayClient`. When no explicit token/password is supplied and a stored device token exists, `GatewayClient` reuses the cached token's stored scopes instead of `opts.scopes` (`src/gateway/client.ts:655-668`), so `openclaw mcp serve --scope operator.read` can still connect with a previous write/approvals token. Please make explicit MCP scopes force a fresh/narrow request or make the client treat explicit scopes as authoritative for this path.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff is narrow and avoids supply-chain changes, but the new least-privilege scope flag is not enforced when the bridge reuses cached device-token scopes.

Concerns:

- **[medium] Cached token scopes can defeat least-privilege profiles:** `src/mcp/channel-bridge.ts:126`
  - body: The PR exposes explicit MCP bridge scopes, but current gateway client auth reuses stored device-token scopes when reconnecting with a cached token. That can leave a supposedly read-only MCP profile connected with broader previously approved scopes.
  - confidence: 0.84
