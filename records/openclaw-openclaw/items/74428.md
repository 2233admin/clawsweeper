## Summary

Keep open. Current main does not implement `openclaw mcp serve --scope`, and this PR is the active narrow implementation candidate, but it is not merge-ready because the explicit MCP scopes are only forwarded into `GatewayClient` options and can still be superseded by cached device-token scopes.

## What This Changes

This PR adds a repeatable `openclaw mcp serve --scope` option limited to `operator.read`, `operator.write`, and `operator.approvals`, threads selected scopes through the MCP server and bridge, updates MCP docs/changelog, and adds focused CLI/server tests.

## Best Possible Solution

Update this branch or an equivalent narrow implementation so explicit MCP `--scope` selections determine the effective Gateway connect scopes, including cached-token paths, while preserving the existing read/write/approvals default when the flag is omitted and covering the behavior with a regression test.

## Reproduction Assessment

Yes. A focused reproduction is to have a stored operator device token with broader cached scopes, start the MCP bridge with `--scope operator.read` and no shared token/password, and observe that `GatewayClient.resolveConnectScopes()` can send the stored scopes instead of the requested read-only set.

## Solution Assessment

No. Passing `operatorScopes` into the `GatewayClient` constructor is a good first step, but the least-privilege behavior needs either effective-scope enforcement for cached device tokens, an isolated bridge identity, or explicit documentation/tests if cached-token precedence is intentionally retained.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor explicit MCP scopes with cached tokens:** `src/mcp/channel-bridge.ts:126`
  - body: The new `operatorScopes` value only changes the `GatewayClient` constructor scopes. When no token/password is supplied, `GatewayClient` can reuse a stored device token and `resolveConnectScopes()` returns cached token scopes before `opts.scopes`, so `openclaw mcp serve --scope operator.read` may still connect with broader cached operator scopes. Please make explicit MCP scopes control the effective connect scope set, or isolate/rotate the bridge identity, and add a regression for that path.
