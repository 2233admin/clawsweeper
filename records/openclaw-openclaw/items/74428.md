## Summary

Keep open. Current main still lacks explicit `openclaw mcp serve` scope selection, and this unmerged PR appears to implement the narrow requested behavior, but it changes MCP auth-scope semantics and overlaps open #74030, so conservative cleanup should leave it for maintainer review.

## What This Changes

The PR adds a repeatable `openclaw mcp serve --scope` allowlist for `operator.read`, `operator.write`, and `operator.approvals`, threads selected scopes through the MCP server/bridge to `GatewayClient`, documents the flag, adds regression tests, and updates the changelog.

## Best Possible Solution

Land this or an equivalent narrow implementation after maintainer review, preserving the default read/write/approvals behavior when `--scope` is omitted, keeping explicit MCP bridge scopes limited to the intended allowlist, and then resolving #74030 according to the chosen canonical branch.
