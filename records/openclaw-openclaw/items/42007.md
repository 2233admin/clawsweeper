## Summary

Keep open. Current main still does not implement Feishu `mentionTriggers`: the Feishu config schema, reply policy, inbound gate, broadcast path, tests, and docs only support native Feishu @mention gating. This PR remains a useful implementation candidate for a real bundled Feishu plugin gap, but it needs revision before merge.

## Best Possible Solution

Keep this PR open and request a Feishu plugin revision. A mergeable version should validate normalized non-empty trigger strings, resolve trigger lists consistently across top-level/account/wildcard-group/exact-group config, treat a keyword match as an effective mention for single-agent and broadcast dispatch, update `activeAgentId` and `WasMentioned` semantics, and add focused tests plus Feishu docs/config metadata updates.

## Risks / Open Questions

- Merging the PR as-is can silently fail in Feishu broadcast groups because keyword-only messages would pass the new early gate while `activeAgentId` remains null and no active Feishu reply dispatcher is selected.
- The proposed `mentionTriggers: z.array(z.string())` accepts empty strings; `ctx.content.includes("")` is always true and can effectively bypass `requireMention` for every group message if configured.
- The public config field still needs tests, Feishu docs, config metadata, and a defined precedence model across top-level, account, wildcard-group, and exact-group config.
- No supply-chain or CI/security-sensitive file changes were visible in the provided PR file list, but the empty-trigger validation gap is an access-control behavior risk.
