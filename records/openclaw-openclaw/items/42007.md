## Summary

Keep open. Current main still does not implement Feishu `mentionTriggers`, and this PR remains a focused implementation candidate for a real bundled Feishu plugin gap. It should be revised before merge because the proposed diff does not cover broadcast dispatch semantics, per-group precedence, empty-trigger validation, tests, or docs/config metadata.

## Best Possible Solution

Keep this PR open and request a Feishu plugin revision. A mergeable version should add a normalized non-empty `mentionTriggers` contract, define precedence across top-level, account, wildcard-group, and exact-group config, treat a keyword match as an effective mention for single-agent and broadcast dispatch, update `activeAgentId` and `WasMentioned` semantics, and add Feishu tests plus docs/config-reference updates.

## Risks / Open Questions

- Merging the PR as-is can silently fail in Feishu broadcast groups because keyword-only matches would pass the new early gate while no active Feishu reply dispatcher is selected.
- The proposed `z.array(z.string())` plus `ctx.content.includes(t)` accepts empty strings; `includes("")` is always true and can bypass `requireMention` for every group message if configured.
- The user-facing config contract remains incomplete without normalized non-empty trigger validation, top-level/account/wildcard-group/exact-group precedence, focused Feishu tests, and docs/config metadata updates.
- Security/supply-chain pass: the provided PR file list does not touch workflows, dependency sources, lockfiles, install/build/release scripts, package publishing metadata, secrets handling, or downloaded artifacts; the main security concern is the empty-trigger access-control behavior risk.
