## Summary

Keep open. The PR is aimed at a real delivery-reliability gap, but the current patch clears the new durable pending-final-delivery markers before final delivery has actually succeeded in the normal reply path and the heartbeat replay path, so a route/send failure can still lose retry state. The diff is also missing the required user-facing changelog entry; the dedicated security pass found no concrete security or supply-chain concern.

## What This Changes

The PR adds pending-final-delivery persistence for main sessions and subagent records, main-session restart recovery for captured final payloads, heartbeat replay handling, heartbeat deferral narrowing, and a focused restart-recovery unit test.

## Best Possible Solution

Keep the durable pending marker until the actual final dispatch or agent-command delivery succeeds, preserve retry metadata on delivery failure, add focused failure-path tests for first delivery and heartbeat replay, and add a single-line Unreleased changelog entry with appropriate contributor credit.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: the PR clears pending state in runReplyAgent and heartbeat replay before dispatchReplyFromConfig reaches sendFinalPayload, so a simulated route/send failure after resolver return leaves no retry marker.

## Solution Assessment

No. The direction is good, but the patch clears durability state earlier than the existing delivery acknowledgement boundary; the narrower maintainable fix is to clear only after successful delivery and leave retry state intact on failure.
