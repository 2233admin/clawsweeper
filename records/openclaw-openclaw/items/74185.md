## Summary

Keep this PR open. Current main still awaits provider auth discovery before the existing per-provider usage timeout, so the work is not obsolete; however, the proposed timeout-warning detection has a concrete correctness bug and should be revised before merge.

## What This Changes

The PR wraps provider usage auth discovery in `withTimeout`, emits a warning when auth discovery is believed to time out, and adds a unit test for a never-settling `resolveProviderAuths` mock.

## Best Possible Solution

Land a narrow provider-auth timeout only after timeout detection is reliable and the JSON status diagnostics behavior is accepted; the final behavior should let `openclaw status --usage --json` terminate from non-TTY automation without adding false timeout warnings for normal empty-provider cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Track auth timeout without racing an already resolved promise:** `src/infra/provider-usage.load.ts:105`
  - body: When `resolveProviderAuths()` legitimately resolves to `[]`, this race still resolves `true` because `Promise.resolve(true)` wins over the `.then()` continuation. That makes normal no-provider `status --usage --json` runs emit a timeout warning on stderr. Track whether the timeout branch actually fired, or return a sentinel from the timeout helper, before warning.
  - confidence: 0.91
