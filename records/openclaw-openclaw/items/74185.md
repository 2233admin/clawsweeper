## Summary

Keep this PR open. Current main still awaits provider auth resolution before the existing per-provider usage timeout, so the PR is not obsolete or already implemented. The linked bug has a concrete non-TTY `status --usage --json` repro, but the proposed `Promise.race` timeout needs maintainer review because it does not cancel underlying OAuth/plugin/SecretRef work.

## Best Possible Solution

Keep this PR open for focused maintainer review. Validate the linked #74085 repro on current main, then either land this narrow loader timeout with appropriate regression coverage or adjust the fix to make auth resolution cancellable/per-provider-bounded if non-canceling work can still keep the subprocess alive.

## Risks / Open Questions

- The proposed `withTimeout(resolveProviderAuths(...), timeoutMs, [])` shape does not cancel the underlying OAuth/plugin/SecretRef work; if that work leaves an active subprocess, socket, prompt, or timer, the non-TTY CLI process may still fail to exit cleanly.
- Returning `{ providers: [] }` on auth timeout may hide a diagnostic auth failure unless the status JSON surface exposes a degraded/timeout signal or maintainers explicitly accept empty usage as the fallback.
- The added unit test proves a never-settling auth promise no longer blocks the loader call, but a subprocess-level regression test or manual non-TTY repro would better prove the reported CLI hang is actually fixed.
