## Summary

Keep open. The PR targets a valid current-main gap in `status --usage --json`, but the latest diff is not merge-ready because its timeout-warning detection reports a timeout for a normal resolved empty auth list.

## What This Changes

The PR wraps provider usage auth discovery in `withTimeout`, emits a warning when auth timeout is inferred, and adds a unit test for a never-settling auth resolver.

## Best Possible Solution

Land a narrow provider-auth timeout for status usage after timeout detection is explicit and reliable. The final behavior should let automation receive JSON without hanging, preserve a quiet normal empty-provider result, and emit diagnostics only when the timeout branch actually wins.

## Reproduction Assessment

Yes. The original bug has a concrete non-TTY `spawnSync` reproduction in #74085, and current main shows the auth phase is unbounded. The PR regression is reproduced by the focused Node probe showing the new timeout-detection race returns `true` for an already-resolved empty auth list.

## Solution Assessment

No. Wrapping auth discovery is the right boundary, but inferring timeout after `withTimeout()` with a second race is not the best implementation. A safer fix is to have the timeout branch return a sentinel or set an explicit boolean only when that timer branch fires.
