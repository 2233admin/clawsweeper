## Summary

Keep open. The PR targets a real current-main gap in the status usage path, but the current head still has a correctness bug: its timeout-warning detection also fires for normal empty auth resolution, and the user-facing fix is missing the required changelog entry.

## What This Changes

The PR wraps provider usage auth discovery in a timeout, emits a warning when it infers auth resolution timed out, and adds a unit test for a never-settling auth resolver.

## Best Possible Solution

Land a narrow provider-auth timeout for status usage after timeout detection is reliable and normal empty-provider results stay quiet. The final behavior should let `openclaw status --usage --json` terminate from automation without adding spurious stderr, and it should include the required user-facing changelog entry.

## Reproduction Assessment

Yes. The linked report gives a concrete `spawnSync` reproduction for the original non-TTY status command hang, and current main shows the auth phase is unbounded. The PR regression is also high-confidence from the Promise.race ordering on an already-resolved empty auth promise and the existing empty-auth test path.

## Solution Assessment

No. Wrapping auth resolution in a timeout is the right narrow direction, but this implementation infers timeout after the fact with a race that also flags normal empty auth resolution. Use an explicit timeout sentinel or timer-branch boolean, then test both the timeout and resolved-empty paths.
