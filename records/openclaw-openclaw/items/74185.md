## Summary

Keep this PR open. Current main still has the unbounded provider-auth phase that the linked report describes, and the PR is aimed at the right narrow area, but the latest head adds a false timeout-warning path for legitimate empty-auth results that should be fixed before merge.

## What This Changes

The PR races `resolveProviderAuths()` against the provider-usage timeout, returns an empty provider list on timeout, logs a timeout warning, and adds a Vitest case for a never-settling auth resolver.

## Best Possible Solution

Land a narrow provider-auth timeout for status usage after timeout detection is reliable. The final patch should keep normal empty-provider results quiet, expose timeout via a sentinel or timer-branch boolean, and add coverage for both a real auth timeout and a legitimate `[]` auth result.

## Reproduction Assessment

Yes. The linked report gives a concrete `spawnSync` path for the original command hang, current main shows the auth phase is still unbounded, and the PR-specific false-warning bug is reproduced by a focused Node `Promise.race` probe.

## Solution Assessment

No. Wrapping auth resolution is the right narrow direction, but this implementation infers timeout after the fact in a way that also flags normal empty auth resolution; a timeout sentinel or explicit timer-branch flag is the safer fix.

## Review Findings
