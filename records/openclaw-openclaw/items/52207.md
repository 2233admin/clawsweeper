## Summary

Keep open: current main has already shipped the SearXNG portion through a separate merged implementation, but Tavily freshness is still not implemented on main and this PR is not mergeable as-is because it mixes that useful remaining feature with blocking regressions and unrelated model-runtime changes.

## What This Changes

The PR branch adds a bundled SearXNG web-search plugin with docs/config/onboarding wiring, adds Tavily `web_search` freshness-to-days support, updates generated config/audit metadata, and changes Pi embedded model-provider normalization code.

## Best Possible Solution

Keep the shipped SearXNG implementation from #57317. If Tavily freshness is still desired, land it as a narrow Tavily-plugin change against current main that accepts OpenClaw's current freshness vocabulary, maps it deliberately to Tavily request/cache behavior, updates focused Tavily/web-search tests and docs, and leaves provider-model normalization unchanged.

## Reproduction Assessment

Yes. The blocking review issues are reproducible by static inspection: the PR mapper handles `pd`/`pw`/`pm`/`py` while current docs use `day`/`week`/`month`/`year`, the PR accepts a Tavily config path current Tavily resolution does not read, and the PR changes OpenRouter transport normalization outside the web-search scope.

## Solution Assessment

No. This is not the best way to solve the remaining issue because it combines a SearXNG provider already shipped on main, a useful Tavily freshness feature, legacy config-schema additions, and unrelated Pi model-runtime changes; a Tavily-only patch is the safer maintainable path.

  - confidence: 0.82

  - body: Adding `tools.web.search.tavily.apiKey` to the strict runtime schema makes that config validate, but Tavily credential resolution still reads only `plugins.entries.tavily.config.webSearch.apiKey` or `TAVILY_API_KEY`.   - confidence: 0.8
- **[P2] Count keyless SearXNG search in exposure audits:** `src/security/audit-extra.sync.ts:335-339`
  - body: SearXNG is intentionally usable with only a base URL, but this audit heuristic treats web search as configured only when an API key or key env var exists. Active keyless SearXNG search can therefore be underreported in small-model exposure auditing.
  - confidence: 0.78
- **[P2] Do not default SearXNG to blocked localhost:** `extensions/searxng/src/config.ts:5`
  - body: The provider defaults to `http://localhost:8080` while private-network access defaults to false and is enforced by the network guard. A default or onboarding-style SearXNG selection can therefore point at localhost and reject its own request until a separate flag is set.
  - confidence: 0.74
