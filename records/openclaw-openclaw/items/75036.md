## Summary

Keep open. Current main still limits durable cron recovery to active cron task rows, so terminal `lost` cron rows with `backing session missing` remain excluded; this PR targets that remaining gap with a narrow condition and regression coverage, and I found no blocking functional or security issue in the diff.

## What This Changes

The PR updates task-registry maintenance to recover terminal `lost` cron task rows from durable cron run history when the lost error is empty or says `backing session missing`, and adds regression tests for recoverable and non-recoverable lost rows.

## Best Possible Solution

Land a narrow task-registry maintenance fix that reuses the existing durable cron recovery source of truth for terminal false-positive lost rows, while preserving unrelated lost rows, then resolve the linked cron audit issues after the fix merges.

## Reproduction Assessment

Yes. The related reports give a concrete reproduction and live state evidence: task audit shows `lost` with `backing session missing` while `openclaw cron runs` shows the same cron execution finished `ok`; current main source confirms terminal lost rows cannot enter durable cron recovery.

## Solution Assessment

Yes. The PR uses the existing durable cron recovery path and adds a specific guard for the known false-positive lost-error class, which is narrower and more maintainable than pruning or broadly rewriting all lost cron rows.
