## Summary

Keep this PR open. Current main still has the Mattermost edit-in-place preview behavior that the PR is trying to fix, but the proposed patch is not merge-ready because boundary splitting is enqueue-time rather than lifecycle-aware, nested streaming account overrides are not merged correctly, and the public docs/changelog are not aligned with the new config surface.

## What This Changes

The PR adds Mattermost `streaming.mode` and `streaming.toolPreview` config/schema/account resolution, propagates sanitized tool args into tool-start callbacks, wires draft-preview boundary splitting into Mattermost streaming updates, and adds focused tests plus generated config metadata.

## Best Possible Solution

Land a Mattermost-local block preview mode that freezes only materialized preview posts at phase boundaries, preserves nested streaming defaults across per-account overrides, and keeps schema, generated metadata, public docs, and changelog in sync before merge.

## Reproduction Assessment

Yes. Current main can be traced directly: Mattermost phase hooks update the same draft stream and never split the post, and a focused delayed-client unit test can reproduce the PR regression by signaling a boundary before the first create resolves or before pending preview text flushes.

## Solution Assessment

No, not as currently implemented. The proposed `block` mode is the right narrow product direction, but the split operation must wait for or otherwise account for pending/in-flight draft creation, and the public config surface needs merge/docs/changelog fixes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the preview post before splitting:** `extensions/mattermost/src/mattermost/draft-stream.ts:209`
  - body: `signalBoundary()` splits after content is merely enqueued. If the first create is pending, `forceNewMessage()` can clear pending text or be overwritten when the in-flight create later stores `streamPostId`, so the next phase can still edit the old post. Make the boundary wait for the prior update/create to materialize, or cover the race explicitly in the draft stream lifecycle.
  - confidence: 0.89
- **[P2] Deep-merge streaming account overrides:** `extensions/mattermost/src/mattermost/accounts.ts:118`
  - body: Mattermost account merging still deep-merges only `commands`. An account override like `streaming: { toolPreview: "args" }` replaces a channel-wide `streaming: { mode: "block" }`, silently falling back to partial mode. Add `streaming` to the nested merge keys and test mixed root/account overrides.
  - confidence: 0.86
- **[P2] Align Mattermost streaming docs with the schema:** `extensions/mattermost/src/config-schema-core.ts:127`
  - body: This schema now accepts object-form `streaming.mode` values `partial` and `block`, but the checked-in Mattermost and streaming docs still advertise scalar `streaming: "partial"` plus `off` and `progress` modes. Users following those docs will get rejected or stripped config unless the docs are updated or compatibility is added.
  - confidence: 0.83
