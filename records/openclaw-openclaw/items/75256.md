## Summary

Keep this PR open, but do not merge it as-is. Current main still has the Mattermost edit-in-place preview behavior, so the branch addresses a real gap; however the proposed boundary controller can split before the throttled draft update has actually materialized, which means fast phase transitions can still overwrite or lose the preview block it is trying to preserve.

## What This Changes

The PR adds Mattermost `streaming.mode` config/schema/types/account resolution, a draft-preview boundary controller that calls `forceNewMessage()` at assistant/reasoning/tool boundaries, and focused schema/account/draft-stream test coverage.

## Best Possible Solution

Keep the Mattermost-local feature direction, but make boundary splitting synchronize with the draft lifecycle so a block is only split after the prior preview update has flushed or any in-flight create/edit has completed. Then update the Mattermost docs/generated config metadata/changelog and add a delayed-client test for a quick thinking-to-tool or partial-to-tool transition.

## Reproduction Assessment

Yes. A focused test can create a Mattermost draft stream with a delayed `POST /posts`, call `update("Thinking...")`, signal a boundary, and enqueue a tool update before the first request resolves; the current loop can then patch the same post instead of creating a new block.

## Solution Assessment

No, not as currently implemented. The requested `block` mode is a narrow maintainable fix, but the split operation should be async or otherwise lifecycle-aware, and the public config/docs surfaces should be aligned before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Flush the draft before forcing a new post:** `extensions/mattermost/src/mattermost/draft-stream.ts:89`
  - body: `signalBoundary()` calls `forceNewMessage()` as soon as content has been enqueued, but `draftStream.update()` only schedules work through the draft loop. If a tool/assistant boundary arrives while the previous Mattermost create/edit is pending, the in-flight send can restore the same post id and the next update edits that post, so `block` mode still overwrites the prior phase. Make the split wait for the prior update to flush or in-flight work to settle, and cover it with a delayed-client test.
  - confidence: 0.89
- **[P2] Align the public Mattermost streaming config:** `extensions/mattermost/src/config-schema-core.ts:116`
  - body: This adds `streaming` as a public Mattermost config object, but the checked-in docs/generated metadata still do not expose `channels.mattermost.streaming.mode`; the docs currently show scalar `streaming: "partial"` plus modes this schema rejects. Update the docs/generated config metadata or keep a scalar compat path so users and config UI do not get a rejected config.
  - confidence: 0.82
