## Summary

Keep this PR open, but it still needs changes before merge. The underlying Mattermost bug is real on current main, and this PR is pointed at the right adapter-local surface; however, the patch still has concrete correctness gaps around asynchronous preview splitting, nested streaming config overrides, and public docs/changelog alignment.

## What This Changes

The PR adds Mattermost draft-preview streaming and tool-preview config, resolves those settings per account, passes sanitized tool args into tool-start callbacks, wires a Mattermost preview boundary controller into draft updates, and adds focused tests plus generated config metadata.

## Best Possible Solution

Land a Mattermost-local block preview mode that splits only after the previous preview post is safely materialized, keeps nested streaming account overrides predictable, and publishes docs/changelog that match the actual config surface.

## Reproduction Assessment

Yes. Current main can be reproduced by a Mattermost turn with reasoning or tool calls: monitor.ts updates the same draft stream at each phase, and draft-stream.ts edits an existing streamPostId. The PR-specific async race can be reproduced with a delayed createMattermostPost test that signals a boundary before the first create resolves, then observes the next update still editing the first post.

## Solution Assessment

No, not as currently implemented. The PR’s adapter-local block mode is the right narrow direction, but the split operation needs to be lifecycle-aware and the config/docs/changelog contract needs to be aligned before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the preview create before splitting:** `extensions/mattermost/src/mattermost/draft-stream.ts:209`
  - body: `signalBoundary()` calls `forceNewMessage()` after content is only enqueued. If the first Mattermost create is still in flight, that create can later restore `streamPostId`, and the next update edits the same post, so `block` mode can still overwrite the prior phase. Make the split wait for the previous create/edit to flush or add a generation guard, and cover it with a delayed-client test.
  - confidence: 0.89
- **[P2] Deep-merge per-account streaming overrides:** `extensions/mattermost/src/mattermost/accounts.ts:182-183`
  - body: The new resolvers read `merged.streaming`, but Mattermost account merging currently deep-merges only `commands`. An account override like `streaming: { toolPreview: "args" }` replaces a channel-wide `streaming: { mode: "block" }`, so the account silently falls back to `partial`. Add `streaming` to the nested merge keys and test mixed root/account overrides.
  - confidence: 0.86
- **[P2] Align the public Mattermost streaming docs:** `extensions/mattermost/src/config-schema-core.ts:124`
  - body: This adds object-form `channels.mattermost.streaming.mode`, but the checked-in docs still tell users to set scalar `streaming: "partial"` and list `off`/`progress` modes that this schema rejects. Update the Mattermost and streaming docs, or add compatibility for the documented scalar form.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `extensions/mattermost/src/types.ts:108`
  - body: This is a user-facing Mattermost fix/feature, but the PR does not update `CHANGELOG.md`. Add a single-line entry under the active version with contributor attribution so release notes capture the new streaming mode and tool-preview option.
  - confidence: 0.8
