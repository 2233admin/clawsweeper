## Summary

Keep this PR open. The Mattermost-local feature direction addresses a real linked issue, but the current patch still has merge-blocking correctness gaps around draft-preview split timing, nested streaming config resolution, and public config/docs alignment.

## What This Changes

The PR adds Mattermost `streaming.mode` and `streaming.toolPreview` config/schema/account resolution, propagates sanitized tool args to tool-start callbacks, wires a draft-preview boundary controller into Mattermost streaming updates, and adds focused tests plus generated config metadata.

## Best Possible Solution

Land a Mattermost-local block preview mode that freezes only materialized preview posts at phase boundaries, keeps account overrides predictable for nested streaming settings, and aligns schema, generated metadata, docs, and changelog before merge.

## Reproduction Assessment

Yes. A focused delayed-client unit test can enqueue `Thinking...`, signal a boundary before the first `createMattermostPost` resolves, then enqueue a tool status and observe that the next update still edits the first post. A separate account-resolution test can show a root `streaming.mode` being lost when an account overrides only `streaming.toolPreview`.

## Solution Assessment

No, not as currently implemented. The proposed `block` mode is the right narrow direction, but the split operation needs to be lifecycle-aware and the public config surface needs docs/changelog and nested-merge fixes before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Wait for the preview create before splitting:** `extensions/mattermost/src/mattermost/draft-stream.ts:209`
  - body: `signalBoundary()` calls `forceNewMessage()` after content is merely enqueued. If the first Mattermost create is still in flight, it can later store the old `streamPostId`, and the next update edits that same post, so `block` mode still overwrites the prior phase. Make the split wait for the prior update/create to flush or otherwise guard against in-flight creates, and cover it with a delayed-client test.
  - confidence: 0.89
- **[P2] Deep-merge per-account streaming overrides:** `extensions/mattermost/src/mattermost/accounts.ts:182-183`
  - body: The new resolvers read `merged.streaming`, but Mattermost account merging only deep-merges `commands`. An account override like `streaming: { toolPreview: "args" }` replaces a channel-wide `streaming: { mode: "block" }`, so the account silently falls back to `partial`. Add `streaming` to the nested merge keys and test mixed root/account overrides.
  - confidence: 0.86
- **[P2] Align the public Mattermost streaming docs:** `extensions/mattermost/src/config-schema-core.ts:127`
  - body: This adds object-form `channels.mattermost.streaming.mode`, but the checked-in docs still tell users to set scalar `streaming: "partial"` and list `off`/`progress` modes that this schema rejects. Update the Mattermost and streaming docs, or add a compatibility path for the documented scalar form.
  - confidence: 0.83
- **[P3] Add the required changelog entry:** `extensions/mattermost/src/types.ts:108`
  - body: This is a user-facing Mattermost fix/feature, but the PR does not touch `CHANGELOG.md`. Add a single-line entry under the active version with the appropriate contributor attribution so the release notes capture the new streaming mode and tool-preview option.
  - confidence: 0.8
