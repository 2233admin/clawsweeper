## Summary

Current main does not implement this PR’s intended startup behavior. `tools.exec.pathPrepend` is still applied only to exec command environments, while the named binary-detection paths still read `process.env.PATH` directly, so the PR remains a plausible implementation candidate rather than obsolete work.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is either to land a config-aware implementation that reuses the existing path-prepend helper, handles reload removal semantics, adds focused config-load tests, and updates docs/help for the broader contract, or to choose a narrower design where `hasBinary` and media discovery receive an explicit effective PATH without globally mutating `process.env.PATH`.

## Risks / Open Questions

- The PR expands `tools.exec.pathPrepend` from an exec-environment setting into a process-wide PATH mutation; maintainers should decide whether that trust boundary is intended and document it if accepted.
- The proposed append-only mutation can leave removed or reduced `pathPrepend` entries in `process.env.PATH` across in-process config reloads unless the implementation tracks and removes entries it owns.
- Landing this as-is would likely require focused config-load/reload tests and docs/help/schema wording updates so the broader contract is explicit.
