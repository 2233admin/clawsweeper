## Summary

Keep open. Current main still does not implement the intended startup behavior, and this PR should not merge as-is because it changes the documented exec-scoped PATH contract while leaving reload and PATH-key casing defects unresolved.

## What This Changes

The PR adds config-load logic in `src/config/io.ts` to trim and dedupe `tools.exec.pathPrepend` entries and prepend them to `deps.env.PATH`.

## Best Possible Solution

Choose the contract first: either compute an effective PATH for `hasBinary` and media/tool discovery without globally mutating `process.env`, or implement global mutation through the shared helper with owned-entry tracking, removal on reload, Windows/path-key coverage, and docs/help updates.

## Reproduction Assessment

Yes. A source-level reproduction is clear: set `tools.exec.pathPrepend` to a directory absent from `process.env.PATH`, load config, then call `hasBinary`; current main never merges that setting before `hasBinary` scans PATH.

## Solution Assessment

No. The PR addresses the reported symptom, but it is not the best current shape because it expands the setting’s trust boundary and has stale reload semantics; a config-aware effective PATH or a stateful helper-based global mutation would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Remove owned PATH entries when config changes:** `src/config/io.ts:1840-1846`
  - body: This adds a one-way prepend into `deps.env.PATH` on each config load, but it never records or removes entries that came from an earlier `tools.exec.pathPrepend`. If a live config update removes or changes that list, `hasBinary()` and other PATH lookups will still see the old directory until the process restarts, so disabled tools can remain incorrectly eligible.
  - confidence: 0.89
- **[P2] Use the shared PATH prepend helper:** `src/config/io.ts:1839-1844`
  - body: The diff reads and writes `deps.env.PATH` directly instead of using the existing `applyPathPrepend` helper. The current helper and tests handle env objects whose PATH key is `Path` or another casing; this code can miss the existing value, fail dedupe, or create a second PATH key in injected or Windows-style envs.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff expands a documented exec-scope PATH setting into global process PATH resolution, which needs maintainer approval plus tests/docs before merge.

That can be intended, but it widens the binary-hijack surface beyond the current docs and should be accepted deliberately with clear tests and documentation.
