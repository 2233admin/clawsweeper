## Summary

Keep this issue open. Current main still intentionally rejects Telegram group commands when the sender is only present in the DM pairing credentials store, and the related PR #57060 was closed unmerged as superseded while explicitly leaving this issue as the canonical place for maintainer product/security judgment.

## Best Possible Solution

Keep this issue open as the canonical discussion. If maintainers want paired Telegram users to be authorized in groups, implement it deliberately in the Telegram group/native command authorization paths, preferably behind an explicit policy or documented config rule, then update the security/group docs and tests that currently assert pairing-store approvals are DM-only. If maintainers decide the current security boundary is intended, close this with that explicit product decision rather than an automated cleanup close.

## Risks / Open Questions

- Changing this behavior would broaden Telegram group command authorization from DM pairing approvals, so it needs explicit maintainer product/security judgment and coordinated docs/tests updates.
- The closed PR's fallback-only approach would not cover Telegram's normal plugin resolveAllowFrom path, so any implementation should update the Telegram group/native command authorization path directly or add an explicit config-controlled policy.
