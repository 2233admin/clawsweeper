## Summary

Keep open. The report matches current main: the Microsoft Teams startup resolver still treats ID-shaped team keys such as `19:...@thread.skype` as Graph team names, so a documented ID-based config can produce the reported unresolved log even while runtime routing continues to work from the raw configured IDs.

## What This Changes

The issue asks Microsoft Teams startup to stop logging `msteams channels unresolved` for configured team/channel IDs ending in `@thread.skype` when those IDs are otherwise usable.

## Best Possible Solution

Teach the Microsoft Teams plugin to recognize already-configured Teams conversation IDs, including `@thread.skype` and `@thread.tacv2`, as resolved IDs during startup allowlist resolution, then document or test the accepted suffixes so users following the Teams URL-path guidance do not get false unresolved warnings.
