## Summary

Keep open. Current main still lacks the native shared OpenClawChatUI assistant-message copy affordance, but the proposed PR has a concrete privacy/correctness blocker because the copy action uses the raw assistant payload instead of the same visible text transformation used by rendering.

## What This Changes

The PR adds hover state and a native copy button to assistant messages in `ChatMessageViews.swift`, copying text to the platform clipboard with temporary copied-state feedback.

## Best Possible Solution

Keep the PR open and revise the copy feature so the clipboard payload is built from the same `AssistantTextParser` visible-segment path used for rendering, honoring `showsAssistantTrace`, with focused Swift coverage for trace-on and trace-off copy payloads.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Copy only the visible assistant text:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatMessageViews.swift:231`
  - body: `CopyMessageButton` is fed raw `primaryText`, but assistant rendering goes through `AssistantTextParser.segments(..., includeThinking: showsAssistantTrace)`. With traces off, a tagged response can show only the final answer while the copy button places hidden `<think>` content and tags on the clipboard, so the copy payload needs to come from the same visible segments as the renderer.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The diff introduces a privacy-sensitive clipboard mismatch by copying hidden assistant trace payload that may not be visible in the UI.

Concerns:

  - body: The new copy button writes raw `primaryText` to the system clipboard. When `showsAssistantTrace` is false, the UI can hide thinking segments, but the clipboard can still receive those hidden segments and tags.
