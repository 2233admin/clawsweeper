## Summary

Keep open: the PR targets a real Ollama multi-provider routing bug, but the proposed wrapper still forces later requests through the first registered provider's captured base URL. Current main does not already implement the requested per-request routing, so the right outcome is to repair this PR rather than close it or merge it as-is.

## What This Changes

The PR changes the Ollama plugin stream path to recompute chat URL and SSRF policy per request and wraps configured Ollama streams by stamping a resolved base URL onto the runtime model.

## Best Possible Solution

Keep the fix in the Ollama plugin, but have the registered stream function prefer the runtime model's selected provider URL per request, with the captured configured URL only as fallback. Add focused coverage for two providers sharing `api: "ollama"`, and keep alias or manifest recognition work separate unless a maintainer explicitly folds it in.

## Reproduction Assessment

Yes. The PR body provides a concrete cloud-plus-LAN setup, and current source confirms the key source-level path: streams are registered by shared `model.api`, later registrations are dropped, and the current Ollama stream bakes its URL into the first closure.

## Solution Assessment

No. Per-request URL resolution is the right layer, but this implementation overwrites an existing runtime `model.baseUrl` with the first wrapper's `resolvedBaseUrl`; the safer fix is to preserve a non-empty runtime URL and fall back only when it is absent.
