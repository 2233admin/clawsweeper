## Summary

Keep open: the reported multi-provider Ollama misrouting is plausible, but this PR’s wrapper still pins calls to the first registered provider URL by overwriting the runtime model’s `baseUrl` with closure state. The patch needs a small correction and a regression test before it is safe to merge.

## What This Changes

The PR modifies `extensions/ollama/src/stream.ts` so Ollama stream functions can recompute chat URL and SSRF policy from a model `baseUrl`, and wraps configured streams with a resolved base URL.

## Best Possible Solution

Keep the fix in the Ollama plugin, but make the registered stream function prefer the runtime model’s provider URL and headers per request with closure defaults only as fallback. Add a regression test for two providers sharing `api: "ollama"`, and keep any `remote-ollama` alias/manifest work separate or link it to the canonical follow-up.

## Reproduction Assessment

Yes. The PR body gives a concrete cloud-plus-LAN setup, and current source confirms the key failure path: streams are registered by shared `model.api` and later `ollama` registrations are dropped. A focused unit test can reproduce this without live network by invoking the first registered configured stream with a second provider runtime model.

## Solution Assessment

No. Per-request URL resolution is the right layer for the first-write registry bug, but the implementation overwrites the per-request `model.baseUrl` with the first wrapper’s captured `resolvedBaseUrl`. The safer fix is to preserve a non-empty runtime `model.baseUrl` and review runtime headers in the same path.
