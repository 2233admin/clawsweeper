## Summary

Keep this PR open. Current main still derives Slack lifecycle status reactions from the ack-reaction gate, so an explicit `messages.statusReactions.enabled: true` flag alone cannot start the controller. The PR is scoped to the right Slack surface and has no supply-chain-sensitive file changes, but it needs adaptation to current main and a stronger explicit-disable test before merge.

## What This Changes

This PR updates Slack message preparation so `messages.statusReactions.enabled === true` can create the lifecycle status-reaction path without requiring `ackReaction`, and adds prepare tests for explicit enable and disable cases.

## Best Possible Solution

Land an updated version of the Slack prepare change that treats `messages.statusReactions.enabled === true` as an explicit lifecycle-reaction opt-in while preserving the current `sourceRepliesAreToolOnly` guard, preserving `enabled: false` as an explicit disable, and proving both the explicit-enable and plain-ack fallback cases in tests.
