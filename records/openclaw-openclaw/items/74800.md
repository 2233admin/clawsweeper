## Summary

Keep open: current main still has the reported undefined plugin-command result path, and an open implementation PR already targets this exact failure. The issue should stay open until #74989 or an equivalent fix is reviewed and merged.

## What This Changes

The issue requests nullish-result handling for plugin/custom command handlers so Telegram commands do not crash or return the generic error when a handler returns undefined.

## Best Possible Solution

Land #74989 or an equivalent narrow fix that normalizes nullish or non-object plugin command returns at the shared executor boundary, keeps continueAgent semantics intact, adds defensive caller handling, and covers both generic text commands and Telegram native plugin commands with regression tests.

## Reproduction Assessment

Yes. A synthetic plugin command handler that resolves to undefined follows executePluginCommand into handlePluginCommand, where current main reads result.continueAgent; the Telegram native path has the same raw-result exposure.

## Solution Assessment

Yes. Central normalization at executePluginCommand, plus defensive handling at the two consumers, is the narrow maintainable fix because both generic command dispatch and Telegram native command dispatch consume the same plugin command result.
