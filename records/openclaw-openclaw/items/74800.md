## Summary

Keep open. Current main still has the nullish plugin-command result failure path, and this issue is already paired with open PRs that use closing syntax, so the issue should remain open until one canonical fix is reviewed and merged.

## What This Changes

The issue requests that Telegram/plugin command handling tolerate a custom command handler that returns `undefined` instead of throwing while reading reply fields.

## Best Possible Solution

Normalize plugin command handler outputs at the shared `executePluginCommand()` boundary so nullish or non-object runtime returns become an empty no-op reply, preserve `continueAgent` semantics, and add regression coverage for both shared text command dispatch and Telegram native plugin command dispatch.

## Reproduction Assessment

Yes. A synthetic plugin command handler that omits `return` can reproduce the source-level failure path because current main returns that raw handler value and then reads `result.continueAgent`; no live Telegram run was needed for this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is shared executor-boundary normalization, with targeted defensive tests around the shared reply path and Telegram native command path rather than Telegram-only fallback handling.
