## Summary

Keep open. Current main still does not implement durable followup-queue or drain-window message persistence/replay, and the related tracking issue #51620 remains open. This PR is still an implementation candidate, but it needs revision before merge because the submitted approach has unresolved routing/durability review concerns plus an unrelated committed debug log artifact.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The best path is to remove the debug log artifact, split or drop the unrelated `/model` change, and rework persistence so queued/drain-rejected inbound messages replay through the existing followup or dispatch semantics with routing, session wake, untrusted user content, media/command context, queue caps, and summary state covered by focused tests. If another PR becomes the canonical implementation for #51620, close this PR as superseded then.
