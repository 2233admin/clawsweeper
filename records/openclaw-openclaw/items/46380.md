## Summary

Keep this PR open. Current main still does not implement the MiniMax VLM SSRF guard: `minimaxUnderstandImage()` accepts `apiHost`, `MINIMAX_API_HOST`, or `modelBaseUrl`, then sends the MiniMax bearer token and image payload through raw `fetch()`. The PR is still relevant, but its current strict-mode implementation should be revised before landing to preserve current timeout and env-proxy behavior, add audit context, and include focused regression tests.

## Best Possible Solution

Keep the PR open and request a current-main revision: route MiniMax VLM through the repo’s guarded fetch path, preserve timeout handling and env-proxy compatibility, set `auditContext: "minimax-vlm"`, release any dispatcher in a clear `finally`, and add regression tests for private/cloud-metadata `apiHost`, `MINIMAX_API_HOST`, `modelBaseUrl`, and HTTP(S)_PROXY behavior.

## Risks / Open Questions

- Current main can still send the MiniMax bearer token and image payload to a configured private/internal or attacker-controlled `apiHost`, `MINIMAX_API_HOST`, or `modelBaseUrl` through raw `fetch()`.
- The submitted patch’s direct `withStrictGuardedFetchMode()` use may break deployments that rely on `HTTP_PROXY` or `HTTPS_PROXY`, because current main explicitly bootstraps env proxy handling before the MiniMax VLM request.
- The submitted patch lacks focused regression tests for blocking private/cloud-metadata MiniMax VLM hosts and for preserving proxy-compatible MiniMax egress.
- The submitted patch omits `auditContext`, so blocked SSRF audit logs would be less attributable even if the core guard blocks correctly.
