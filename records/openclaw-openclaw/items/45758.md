## Summary

Keep open. Current main still discovers, documents, parses, and writes the root config as JSON/JSON5 only. The discussion's `config.patch` direction helps with comment-safe mutation, but it does not satisfy the requested `openclaw.yaml` / `openclaw.yml` root-config loading behavior. This needs explicit maintainer product judgment because the change belongs in core config discovery/parsing/write semantics, not a plugin.

## Best Possible Solution

Keep this open for explicit maintainer product judgment. If YAML root config support is accepted, implement it in core config discovery/parser selection, include semantics, validation, hot reload, write behavior, tests, and docs with clear JSON5-versus-YAML precedence. If OpenClaw intentionally remains JSON5-only, close manually as a product decision and strengthen the public docs around `openclaw config set`, Control UI config editing, and `config.patch` for comment-safe scripted edits.

## Risks / Open Questions

- YAML support would require explicit precedence rules when both `openclaw.json` and `openclaw.yaml` / `openclaw.yml` exist.
- YAML scalar coercion and parser schema choice could change config meaning, so implementation needs security and compatibility tests for booleans, numbers, dates, aliases, and prototype-pollution-style keys.
- The change is core-owned because plugins load after root config resolution; it cannot be satisfied cleanly as a ClawHub plugin with the current startup model.
- OpenClaw-owned writes, include write-through, hot reload, validation, docs, and CLI/RPC mutation paths would all need aligned behavior.
