## Summary

Current main still does not implement YAML root config support. OpenClaw discovers and documents `openclaw.json`, parses root config and includes with JSON5, and writes JSON. The discussion’s safer mutation direction is real (`openclaw config set --batch-file` and `config.patch`), but it does not satisfy the requested `openclaw.yaml` / `openclaw.yml` loading behavior.

## Best Possible Solution

Keep this open for an explicit maintainer product decision. If YAML support is accepted, implement it in core config discovery, extension-aware parser selection, include semantics, validation, hot reload, write behavior, tests, and docs with clear `openclaw.json` versus `openclaw.yaml` / `openclaw.yml` precedence. If OpenClaw intentionally stays JSON5-only, close manually as a product decision and strengthen docs around `openclaw config set`, Control UI, and `config.patch` for comment-safe scripted edits.

## Risks / Open Questions

- Adding YAML would touch core startup config discovery, parser selection, include parsing, validation, hot reload, write serialization, docs, and tests; it is not plugin-only because plugins load after root config resolution.
- YAML parser semantics could introduce scalar coercion and security/compatibility differences, so any implementation needs explicit precedence, schema, include, write, and scalar-handling tests.
- The existing safer mutation paths reduce comment-preservation pain but do not answer the requested `.yaml` / `.yml` root config-file support.
