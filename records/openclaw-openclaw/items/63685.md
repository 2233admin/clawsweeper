## Summary

Keep open. Current main supports the corrected Gemma 4 model id, but it still accepts the reported `gemma-4-26b` form as a forward-compatible Google model, does not normalize it to the documented id, and would build the same `models/gemma-4-26b` Google API path shown in the 404. The reporter’s follow-up says the value came from terminal setup, so this remains a Google plugin/setup validation or aliasing issue rather than a closeable typo.

## What This Changes

The issue asks OpenClaw to make Google AI Studio Gemma 4 model selection work instead of failing with a 404 for `models/gemma-4-26b`.

## Best Possible Solution

The Google plugin should either stop offering the bad shorthand in setup/model-picker flows, map the exact reported shorthand to the documented Google id if that is an intended convenience alias, or fail before provider dispatch with a clear hint to use `google/gemma-4-26b-a4b-it`; the behavior should be covered by focused Google provider/setup tests.
