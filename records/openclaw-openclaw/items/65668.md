## Summary

Keep open. Current main partially mitigates adjacent cases, but the reported custom-supervisor path still exists for non-container Linux/macOS processes with no recognized supervisor markers, and the doctor/docs guidance still frames `OPENCLAW_NO_RESPAWN=1` as a startup optimization rather than an operational safety setting.

## What This Changes

The issue requests safer gateway SIGUSR1/config restart behavior under custom supervisors and clearer `OPENCLAW_NO_RESPAWN` guidance to prevent orphaned detached gateway processes and `EADDRINUSE` loops.

## Best Possible Solution

Make generic non-update gateway restarts use in-process restart by default for unrecognized/no-supervisor contexts, or require an explicit detached-respawn opt-in, while preserving known supervisor handoff and `update.run` fresh-process replacement. Update doctor/docs/tests/changelog so operators see `OPENCLAW_NO_RESPAWN=1` as a safety workaround for unmanaged or custom-supervised deployments, not merely a startup optimization.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: Linux with no recognized supervisor hints and no container detection returns `mode: "spawned"`, and the run loop exits the tracked parent after spawning the detached child; a PID-tracking supervisor can then restart into the occupied port as reported.

## Solution Assessment

No, current main is not yet the best solution. The container/Windows guards and config reload-plan gating are useful partial fixes, but the narrow maintainable fix is to remove detached generic restarts from unrecognized supervisor contexts or make that behavior explicit opt-in while keeping update restarts separate.
