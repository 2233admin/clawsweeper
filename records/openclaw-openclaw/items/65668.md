## Summary

Current main still preserves the reported unmanaged non-container Linux failure mode for non-update SIGUSR1/config restarts: unknown supervisors are not detected, the generic restart helper detached-spawns, and the run loop exits the original PID. Recent changes mitigate Windows, containers, stale locks, and update.run, but they do not fully solve this issue.

## Best Possible Solution

Keep this open and land a focused core fix: make non-update unmanaged SIGUSR1/config restarts use the in-process restart path by default when no trusted supervisor is detected, preserve `respawnGatewayProcessForUpdate()` for update.run, update doctor/docs wording, and replace the current tests that assert detached unmanaged Linux behavior with regression coverage for the safer default.

## Clownfish Work Prompt

Fix the non-update gateway restart contract for this issue. Current main still lets `restartGatewayProcessWithFreshPid()` detached-spawn and exit on unmanaged non-container Linux when SIGUSR1/config restarts occur under an unknown custom supervisor, orphaning the gateway from PID-tracking supervisors. Implement the narrow fix in core: for ordinary/non-update SIGUSR1/config restarts without a trusted supervisor, prefer in-process restart instead of detached-spawning and exiting the tracked PID. Preserve `respawnGatewayProcessForUpdate()` semantics for `update.run`, because updates still need a fresh OS process after package files change. Do not add brittle generic parent-process-name supervisor detection unless it is strictly needed; the safer boundary is non-update unmanaged restart defaulting to in-process. Update tests that currently assert detached unmanaged Linux behavior, add coverage for non-container Linux with no supervisor hints returning disabled/in-process behavior, keep container/Windows/update coverage intact, and update doctor/docs wording so `OPENCLAW_NO_RESPAWN` is described as supervisor/PID safety rather than only startup overhead. Add a single-line user-facing changelog fix entry under the active version, thanking @mjamiv, and include @iannwu only if the implementation or docs use the launchd corroboration from the discussion. Do not change unrelated restart/update UI behavior or plugin surfaces.
