## Summary

Keep this issue open. Current main still sends the built-in `user` / `existing-session` browser profile through Chrome DevTools MCP over the upstream stdio transport, and the issue discussion already has explicit maintainer triage saying the Termux stdin parent-death failure is separate from the merged Chrome MCP command/args work.

## Best Possible Solution

Keep this issue open and fix the browser Chrome MCP lifecycle path directly. The likely product/code direction is to route Chrome MCP through an OpenClaw-owned stdio/process transport that preserves child stdin correctly on Android/Termux, or add a supported non-stdio attach transport for existing-session profiles, then add a focused regression test that simulates stdin EOF or parent-death during `profile="user"` status/actions and verifies recovery or a useful error.
