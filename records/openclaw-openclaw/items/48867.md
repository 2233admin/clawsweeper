## Summary

Keep this issue open. Current main still routes the built-in `user` existing-session browser profile through Chrome DevTools MCP, and the browser Chrome MCP path still constructs the upstream SDK `StdioClientTransport` directly rather than an OpenClaw-owned process/stdio lifecycle wrapper. The provided maintainer triage also explicitly says the Android/Termux stdin parent-death failure is separate from the merged Chrome MCP command/args work in #71560.

## Best Possible Solution

Keep this issue open and fix the browser Chrome MCP lifecycle path directly. The best direction is an OpenClaw-owned stdio/process transport seam usable by the browser plugin, or a browser-local equivalent that preserves child stdin correctly on Android/Termux and cleans up process trees predictably. Add a focused regression test around Chrome MCP subprocess stdin EOF/parent-death behavior and validate on a real Termux setup when feasible.
