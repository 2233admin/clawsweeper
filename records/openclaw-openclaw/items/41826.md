## Summary

Keep this issue open. Current main still does not implement a custom TUI client display name: the TUI CLI has no `--client-name`, `--display-name`, or `--username` option, `TuiOptions` has no identity field, Gateway-mode TUI forwards only URL/auth options, and `GatewayChatClient` still hard-codes `clientDisplayName: "openclaw-tui"`. The Gateway protocol and presence path already support display names, so this remains a focused TUI wiring gap rather than a protocol request or duplicate of the WebChat/Control UI display-name work.

## Best Possible Solution

Keep this issue open. Add a focused TUI implementation: parse `--client-name` or `--display-name` in `src/cli/tui-cli.ts`, extend `TuiOptions` and TUI relaunch argument building, pass a sanitized value through `runTui` into `GatewayChatClient.connect`, default to `openclaw-tui`, preserve the stable protocol client id, and update TUI docs plus focused CLI/backend tests.
