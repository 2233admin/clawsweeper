## Summary

Keep this open. Current main still does not implement a custom TUI client display name: the CLI has no `--client-name`, `--username`, or `--display-name` option, `TuiOptions` has no identity field, Gateway-mode TUI only forwards URL/auth options, and `GatewayChatClient` still hard-codes `clientDisplayName: "openclaw-tui"`. The Gateway protocol already supports `client.displayName`, so this remains a focused TUI wiring gap rather than a protocol or Control UI/WebChat duplicate.

## Best Possible Solution

Keep this issue open. Add a focused TUI implementation: parse `--client-name` or `--display-name` in `src/cli/tui-cli.ts`, extend `TuiOptions` and TUI relaunch argument building, pass the sanitized value through `runTui` into `GatewayChatClient.connect`, default to `openclaw-tui`, keep the stable protocol client id unchanged, and update TUI docs plus focused CLI/backend tests.

- Implementation should keep `clientName`/client id as `openclaw-tui` and customize only `clientDisplayName`, so Gateway client classification does not regress.
