## Summary

Current main still does not implement custom TUI client display names. The TUI CLI exposes no `--client-name`, `--username`, or `--display-name` option; `TuiOptions` has no client identity field; `runTui` forwards only URL/token/password into `GatewayChatClient.connect`; and `GatewayChatClient` still hard-codes `clientDisplayName: "openclaw-tui"`. The Gateway protocol already supports `client.displayName`, so this is still a focused TUI wiring gap rather than a duplicate of the Control UI/WebChat display-name items in the provided context.

## Best Possible Solution

Keep this issue open. Add a focused TUI implementation: parse `--client-name` or `--display-name` in `src/cli/tui-cli.ts`, extend `TuiOptions` and `launchTuiCli` so relaunches preserve it, pass the sanitized value through `runTui` into `GatewayChatClient.connect`, default to `openclaw-tui`, and cover the CLI/backend wiring plus docs.

## Risks / Open Questions

- Implementation should keep the stable protocol client id as `openclaw-tui` and customize only `clientDisplayName`; changing the id could regress Gateway client classification.
- Free-form display names should be trimmed and sanitized for terminal/log safety before being sent to the Gateway or shown in presence/session UI.
