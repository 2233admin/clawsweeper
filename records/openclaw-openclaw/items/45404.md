## Summary

Current main uses `openclaw` as the canonical managed browser profile and preserves `clawd` only as a legacy config driver alias, but I found no doctor, repair, migration, or docs path that detects or safely archives an unconfigured `~/.openclaw/browser/clawd/user-data` residue directory. GMTekAI's source-pass comment is consistent with the current code, so this remains a valid low-priority cleanup gap.

## Best Possible Solution

Keep this issue open and implement a browser-owned doctor/repair check: scan for `~/.openclaw/browser/clawd/user-data` when the active managed profile is `openclaw`, report it as legacy residue, and offer an explicit interactive archive/trash move rather than deletion. Update the browser/doctor docs to document the canonical managed profile path and the safe cleanup behavior.
