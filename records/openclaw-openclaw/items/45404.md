## Summary

Keep this open. Current main canonicalizes the managed browser profile to `openclaw`, but it still has no browser-owned doctor/repair/docs path that detects and safely archives unconfigured legacy `browser/clawd/user-data` residue.

## What This Changes

Detect legacy `clawd` managed browser profile directories left after the rename migration and offer a safe warning plus archive cleanup path.

## Best Possible Solution

Implement a browser-plugin-owned doctor/repair check that scans for stale unconfigured `browser/clawd/user-data` only when the active managed profile is `openclaw`, reports a clear warning, and offers an explicit safe archive/trash move. Keep the legacy `clawd` driver alias for config compatibility and document the canonical managed profile path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: create an unconfigured `CONFIG_DIR/browser/clawd/user-data` directory beside the canonical `openclaw` profile, then current browser doctor and delete-profile paths will not detect or archive it because they only inspect configured profiles and selected runtime state.

## Solution Assessment

Yes. The issue’s proposed doctor/repair-style warning plus explicit archival flow is the narrowest maintainable solution; startup migration or silent deletion would create unnecessary data-loss and ownership risk.
