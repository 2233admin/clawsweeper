## Summary

Keep open. Current main still performs WhatsApp pairing access-control side effects before proving that a Baileys `messages.upsert` entry contains an actual inbound text/media/contact/location message. Existing safeguards cover tracked outbound echoes, `fromMe` DMs, and stale append history, but not the reported non-content notify/protocol/receipt-style event path.

## Best Possible Solution

Fix this in the WhatsApp plugin by gating pairing access control on extractable inbound user content, or by passing an explicit no-pairing classification for non-content Baileys events before `checkInboundAccessControl` can send. Add regression coverage for `dmPolicy: "pairing"` with protocol/receipt/no-content `messages.upsert` payloads and for a recent outbound-to-JID scenario, asserting that no pairing request is upserted and no pairing reply is sent.
