## Summary

Keep open: current main still appends dreaming promotions to `MEMORY.md` without a durable file-size budget, compaction, or archive layer, and the paired closing PR #74088 remains open and unmerged.

## What This Changes

The issue requests bounding or compacting `MEMORY.md` growth from memory-core dreaming promotions so oversized bootstrap context does not lead to session lock timeouts or Gateway freezes.

## Best Possible Solution

Land PR #74088 or an equivalent memory-core fix that applies a durable pre-write size budget to dreaming promotions, preserves user-authored `MEMORY.md` content, and adds regression coverage for repeated near-limit and over-limit promotion runs.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the unbounded file-growth path: repeated `applyShortTermPromotions` calls with new candidates append to existing `MEMORY.md` without a size guard. The full Gateway freeze remains unclear because the report has no logs or focused lock-timeout reproduction.

## Solution Assessment

Is this the best way to solve the issue? Yes on boundary: the maintainable fix belongs in the bundled memory-core promotion writer, not only in bootstrap truncation. The exact compaction/archive policy should be resolved through review of PR #74088 or an equivalent replacement.
