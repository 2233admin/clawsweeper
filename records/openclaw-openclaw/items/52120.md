## Summary

Keep open. Current main still lacks the Mattermost tool-context/auto-threading hooks needed for the linked bug, but this abandoned PR is not merge-ready because it applies the fallback too late for the current shared outbound runner and does not preserve Mattermost's existing active-thread semantics for `replyToMode: "first"`.

## What This Changes

The PR branch changes the Mattermost message action and tests to derive active thread context and use it as an implicit reply target when no explicit reply target is supplied.

## Best Possible Solution

Adapt the Mattermost-owned fix to the current shared outbound-threading design: add a Mattermost `buildToolContext` that exposes normalized conversation id, active thread root, account-scoped reply mode, and `hasRepliedRef`, and add a Mattermost `resolveAutoThreadId` or equivalent integration so thread roots are resolved before shared reply injection, mirror routing, and plugin dispatch.

## Reproduction Assessment

Yes. The linked bug gives a concrete `replyToMode: "all"` thread-scoped Mattermost message-tool send, and current main source tracing shows the missing `buildToolContext`/`resolveAutoThreadId` path that makes the tool lose the active thread root.

## Solution Assessment

No. The requested direction is correct, but this PR is not the best current-main solution because it handles implicit threading inside `handleAction` after the shared runner has already applied generic reply routing, and it preserves `first` mode inside active Mattermost threads where current behavior keeps follow-ups on the root.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0

Full review comments:

- **[P2] Route Mattermost auto-threading before reply injection:** `extensions/mattermost/src/channel.ts:501-504`
  - body: This fallback runs inside Mattermost `handleAction`, but current main calls `resolveAndApplyOutboundReplyToId` before plugin dispatch. Once `buildToolContext` supplies reply mode and channel context, the shared helper can prefill `params.replyTo` with the current message id, so `readMattermostReplyToId` sees an explicit reply and the PR's `currentThreadTs` fallback never wins. Put Mattermost on `resolveAutoThreadId` or otherwise resolve the thread root before generic reply injection and mirror routing.
  - confidence: 0.78
- **[P2] Promote active Mattermost threads to all mode:** `extensions/mattermost/src/channel.ts:495-496`
  - body: When the active Mattermost session is already inside a thread, preserving `replyToMode="first"` means later tool sends stop inheriting `currentThreadTs` after `hasRepliedRef` is true and fall back to the channel root. Mattermost's existing thread handling keeps follow-ups on the active root, so the effective tool mode should be `all` for an existing thread, not only when the configured mode is `off`.
  - confidence: 0.82
