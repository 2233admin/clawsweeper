## Summary

Keep open. Current main still binds `TaskRecord.task` directly into SQLite, and the proposed patch is a narrow store-boundary hardening with a focused regression test; I found no blocking correctness or security issue in the diff.

## What This Changes

The PR adds serialization for non-string task values before SQLite binding and adds regression coverage that an object task payload is stored as a JSON string.

## Best Possible Solution

Land the narrow SQLite-store normalization and regression test after normal maintainer validation, keeping the read path as a string contract; handle any future public structured-task API separately if maintainers want to support that deliberately.

## Reproduction Assessment

Yes. The SQLite failure is reproducible with an in-memory `node:sqlite` insert that binds an object to `@task`, and the PR adds a focused regression test for the OpenClaw store path.

## Solution Assessment

Yes. Serializing only at the SQLite binding boundary is the smallest maintainable fix for durable `TEXT` storage, preserves existing string values, and avoids changing task display/read semantics.
