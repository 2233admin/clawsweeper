## Summary

Keep open. Current main does not yet support stable cron add ids, and this PR is a relevant continuation of the closed unmerged #40218 work, but the diff introduces a Swift source-compatibility break in the generated public protocol models that should be fixed before merge.

## What This Changes

The PR adds optional stable cron job ids across cron create types, gateway/CLI/tool surfaces, generated Swift protocol models, docs, and tests, plus duplicate-id rejection when adding jobs.

## Best Possible Solution

Keep the additive cron protocol change, but make the generated Swift model source-compatible before merge, either by giving the new optional `id` initializer parameter a `nil` default or by preserving a compatibility overload in the generator/generated outputs. After that, regenerate/check protocol artifacts and run the targeted cron service, protocol, CLI, and Swift/client validation paths.

## Reproduction Assessment

Yes. The compatibility issue is reproducible by source inspection: the PR adds a required `id:` argument to the public `CronAddParams` Swift initializer, while the current generated initializer has no such parameter and the generator emits optional properties without defaults.

## Solution Assessment

No. The TypeScript cron service/gateway direction is a narrow maintainable solution, but the generated Swift public API needs a backward-compatible shape so an optional wire field remains additive for Swift callers.

## Review Findings
