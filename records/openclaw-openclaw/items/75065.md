## Summary

Keep open. Current main still lacks stable cron create IDs and duplicate-id checks, so the feature is not obsolete, but the PR introduces a generated Swift source-compatibility regression that should be fixed before merge.

## What This Changes

The PR adds optional stable cron job IDs across cron create types, cron.add schema/normalization, CLI/tool/docs, generated Swift protocol models, and tests, plus duplicate-id rejection in the cron service.

## Best Possible Solution

Keep the additive cron protocol and service direction, but make generated Swift source-compatible before merge by defaulting optional initializer parameters to `nil` or preserving a compatibility overload, then regenerate protocol artifacts and validate the cron service, protocol, CLI/tool, docs, and Swift client surfaces together.

## Reproduction Assessment

Yes. The current-main gap is reproducible by source inspection of `createJob`, `add`, and `CronAddParamsSchema`; the PR regression is reproducible by comparing the PR's inserted `id: String?` initializer parameter with the generator, which emits optional Swift parameters without defaults.

## Solution Assessment

No. The TypeScript cron service/gateway approach is a narrow maintainable solution, but the PR is not the best merge shape until the optional wire field remains additive for Swift callers.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep Swift initializer source-compatible:** `apps/shared/OpenClawKit/Sources/OpenClawProtocol/GatewayModels.swift:3619`
  - body: Adding `id: String?` to the public `CronAddParams` initializer without a default makes existing Swift callers fail to compile unless they now pass `id: nil`, even though the field is optional and the PR says there is no API break. Please give the generated optional parameter a `nil` default or preserve a compatibility overload, and keep both generated Swift copies in sync.
  - confidence: 0.88
