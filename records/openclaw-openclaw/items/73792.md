## Summary

Keep open. The stale MCP runtime bug is real and current main still has no transport-close eviction, but this PR should not merge as-is because the new handler treats intentional transport closes as unexpected child death and can retire a healthy multi-server runtime during startup cleanup.

## What This Changes

The PR wires MCP transport close/error callbacks into session MCP runtimes, routes closed transports through manager-level session disposal with logging, and adds a SIGKILL-based regression test for stale stdio MCP child sessions.

## Best Possible Solution

Land a guarded version of this lifecycle fix: keep the pre-connect callback installation required by the MCP SDK chaining contract, but mark intentional cleanup before closing a transport and make the close handler no-op when the session was already removed. Add regression coverage for both unexpected child death and multi-server partial startup failure.

## Reproduction Assessment

Yes. The original stale-runtime bug is reproducible by killing a successfully connected stdio MCP child and observing the cached session remain on current main; the PR regression is reproducible with two configured MCP servers where one fails during startup/listing and the other should remain usable.

## Solution Assessment

No, not yet. The proposed direction is the right narrow seam, but the implementation must distinguish unexpected child death from intentional close() calls before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Guard intentional transport closes before eviction:** `src/agents/pi-bundle-mcp-runtime.ts:277-280`
  - body: This handler treats every transport close as an unexpected child death. The pinned MCP Transport contract requires `onclose` to run when `close()` is called, and OpenClaw's stdio transport waits for that child close event; the existing startup-failure catch still calls `disposeSession(session)` before removing the session. In a multi-server config, one failed server cleanup can therefore call `onTransportClosed` and retire the whole runtime, closing healthy sessions or returning a disposed runtime. Mark intentional cleanup before close and no-op when the session was already removed.
  - confidence: 0.88
