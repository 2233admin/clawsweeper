## Summary

Keep open. Current main still lacks transport-close eviction for cached session MCP runtimes, and the PR is a focused fix for a real stale-runtime failure, but the current patch introduces a blocking lifecycle regression by treating intentional startup-cleanup closes as unexpected transport death.

## What This Changes

The PR wires MCP transport `onclose`/`onerror` callbacks into session MCP runtimes, routes close events through manager-level runtime disposal with logging, and adds a SIGKILL regression test for stale stdio MCP child sessions.

## Best Possible Solution

Revise the PR so manager-level eviction only runs for unexpected transport closure. Intentional cleanup after startup/listing failure should mark the server session as removed before closing it, and the regression suite should include a multi-server partial-startup case proving a failed server does not retire healthy sessions.

## Reproduction Assessment

Yes. The reported stale-runtime bug has a clear reproduction path through the PR's SIGKILL probe-server test, and the blocking PR regression has a source-backed reproduction with a multi-server MCP config where one server fails startup/listing while another should remain usable.

## Solution Assessment

No, not yet. Installing the handler before `connectWithTimeout` matches the SDK chaining contract, but the implementation must guard intentional closes, for example by deleting or marking the session before `disposeSession(session)` and no-oping the close handler when the session is already absent.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Guard intentional transport closes before eviction:** `src/agents/pi-bundle-mcp-runtime.ts:277-280`
  - body: This handler also runs when OpenClaw intentionally closes a transport in the startup-failure catch path, because the pinned MCP transport contract says `onclose` fires from `close()` and `OpenClawStdioClientTransport.close()` waits for the child's `close` event. In a multi-server config, one failed server cleanup can therefore call `onTransportClosed`, retire the whole session runtime, and leave the caller with a disposed runtime instead of keeping healthy server sessions alive. Mark intentional cleanup before closing and have the handler no-op when the session is already gone.
  - confidence: 0.9
