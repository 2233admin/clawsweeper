## Summary

Keep this PR open. Current main still has the linked `/compact` cancellation gap, and the PR is a relevant implementation candidate with useful downstream context from the CoClaw plugin report/workaround, but it still needs maintainer review because it lacks focused cancellation regression coverage and bundles an unrelated launchd/profile behavior change.

## Best Possible Solution

Keep this PR open as the active implementation candidate for the linked `/compact` cancellation bug, but revise or review it before merge: register the compacting active-run handle when the queued compaction task actually starts, pass a real abort signal into the compaction runtime, cascade abort to any active reply operation, clean up in `finally`, add targeted cancellation regression tests, and split the unrelated profile/launchd fix into its own PR or issue path.

## Risks / Open Questions

- The PR lacks focused regression coverage proving `/stop`, `chat.abort`, or `abortEmbeddedPiRun(sessionId)` cancels an in-flight manual `/compact`.
- The bundled `OPENCLAW_LAUNCHD_LABEL` change is unrelated to compact cancellation and could change explicit custom-label/service-targeting semantics because current launchd and restart-helper code treat that env var as an override.
- The compact fix introduces a new dependency from the embedded runner compaction path to `auto-reply/reply/reply-run-registry.js`; maintainers should review whether that layering is acceptable or whether a smaller runtime seam is preferable.
