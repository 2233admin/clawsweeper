## Summary

Keep open. Current main still has a plain textarea composer with slash-command-only autocomplete and markdown-only history rendering; the existing `agents.list` data is available to the Control UI but is not wired into @mention autocomplete, mention chips, validation, or known-agent rendering.

## What This Changes

Add Slack-style `@agent-name` autocomplete, styled mention chips, plain-text serialization, known-agent history rendering, and any needed Gateway agent-directory support to the Control UI chat.

## Best Possible Solution

Keep this as a Control UI product/UI task: reuse `agents.list` where possible, add a tested @mention autocomplete flow, serialize selected mentions back to plain `@agent` text for `chat.send`, render only known-agent mentions in history, and add an additive Gateway protocol change only if the existing `AgentSummary` contract is insufficient.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for the current-main absence check: typing `@` in the textarea has no code path to open a mention menu because input handling only calls the slash-command updater and repository search found no mention chip or @mention implementation.

## Solution Assessment

Is this the best way to solve the issue? Unclear for the proposal as written because a separate `agents.directory` endpoint may duplicate the existing `agents.list` contract; the safer maintainable direction is to decide the composer UX first and extend the protocol additively only if the current agent summary data is not enough.
