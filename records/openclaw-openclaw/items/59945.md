## Summary

Keep this issue open. The discussion identified a plausible shared runtime-snapshot bug, and current main still has the same gap: runtime config snapshots are published directly, Slack startup reads credentials from that runtime config, and the redaction restore path only appears on config write RPC handling, not runtime snapshot publication. The exact infinite restart wording is partly outdated because channel restarts are now capped, but the credential/sentinel failure path is still not fixed.

## Best Possible Solution

Fix this in the shared config/runtime layer, not with Slack-specific handling: before publishing a runtime snapshot with a paired source snapshot, restore redacted sentinel placeholders from sourceConfig where valid and reject any remaining __OPENCLAW_REDACTED__ values before live code can consume them. Add the regression test described in the issue discussion using Slack botToken/appToken sentinel leakage, and keep the existing channel restart cap as a separate safety guard.
