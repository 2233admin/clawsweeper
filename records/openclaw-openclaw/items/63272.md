## Summary

Keep open. Current main still lacks the requested `openclaw skills uninstall <slug>` command and still has no skills backend helper that removes both a workspace skill directory and its `.clawhub/lock.json` entry; the related original request and prior dashboard PR are closed without shipping this surface, while the open gateway PR is unmerged and does not add the CLI command.

## What This Changes

Add an `openclaw skills uninstall <slug>` CLI command that safely removes a ClawHub-installed workspace skill and deletes its `.clawhub/lock.json` tracking entry.

## Best Possible Solution

Ship the workspace-scoped CLI inverse of ClawHub skill install/update: safely resolve the target workspace and slug, preview the directory and lockfile entry, remove the workspace skill folder, delete only that slug from `.clawhub/lock.json`, leave sandbox copies and bundled/dashboard behavior out of scope, and update the public CLI/ClawHub docs and focused tests.

## Reproduction Assessment

Yes. Static current-main inspection is enough: after a ClawHub install, `.clawhub/lock.json` records the slug, `updateSkillsFromClawHub` iterates those tracked keys, and there is no CLI/helper that removes the key when a user manually deletes only the skill directory.

## Solution Assessment

Yes, with a sequencing caveat. A CLI command backed by the existing safe install-path and lockfile primitives is the narrow maintainable fix for the reported reappearing-skill behavior, but it should share or reuse #69319's backend removal primitive if that PR lands first.

## Repair Work Prompt

Implement the `openclaw skills uninstall <slug>` feature requested by this issue. First check whether #69319 or an equivalent `uninstallSkillFromClawHub` primitive has landed on current main; if so, reuse it rather than adding a competing backend path.
