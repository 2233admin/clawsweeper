## Summary

Keep open. Current main still leaves the Mattermost plugin send action incomplete: the handler only maps replyToId/replyTo, only reads params.media, and does not forward mediaLocalRoots/mediaReadFile to the Mattermost sender. Shared runner normalization now partly mitigates filePath/path in one path, but explicit threadId and media-root forwarding remain missing in the Mattermost-owned handler, and the related open PRs are partial implementation candidates rather than closure evidence.

## What This Changes

The issue asks Mattermost message-tool sends to honor threadId, local file path aliases, and media local-root access when uploading files into threads.

## Best Possible Solution

Update the Mattermost plugin action adapter so handleAction preserves replyToId precedence, treats explicit threadId as the Mattermost root-post fallback, resolves media aliases consistently, forwards mediaLocalRoots/mediaReadFile to sendMessageMattermost, and adds focused regression tests for local file upload into a thread.
