## Summary

Keep this issue open. Current main still only applies Telegram sendChatAction backoff/suspension to 401 Unauthorized errors, while the reported network/529-style failures remain non-suspending. The shared typing callback also still resets its start guard on every onReplyStart call, so repeated higher-level typing starts can clear the consecutive-failure breaker instead of preserving it across the retry loop. The provided context also shows an open fork PR with closing syntax for this issue, so the issue should stay open while that candidate is reviewed, ported, or superseded.

## Best Possible Solution

Keep the issue open. The best path is to review or port the referenced PR idea into the Telegram plugin: classify non-critical sendChatAction failures such as network/429/529 separately from 401 auth failures, cap retries with backoff or suspension, avoid resetting the typing failure breaker while an existing keepalive cycle is active or tripped, and add a regression test covering repeated higher-level typing starts after non-401 sendChatAction failures.
