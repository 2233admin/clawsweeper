## Summary

Keep open. Current main still appears to have the reported sandbox/skills mismatch: non-rw sandbox setup mirrors eligible skills into the sandbox workspace, but embedded runs can continue rendering a host-built skills snapshot whose prompt wins over sandbox-loaded entries. The issue discussion also has an April 29, 2026 confirmation and workaround, and the adjacent PR heads are not part of current main.

## What This Changes

The issue reports that sandboxed non-main or cron sessions expose managed skill `SKILL.md` locations from `~/.openclaw/skills` instead of sandbox-local mirrored copies, causing sandbox read failures.

## Best Possible Solution

Sandboxed runs with `workspaceAccess: "none"` or `"ro"` should render `<available_skills>` from sandbox-readable copied skill entries, or remap snapshot prompt locations to those copies, while preserving canonical host-resolved skill metadata for runtime config and env handling. Add focused regression coverage for a managed skill used from a non-main sandboxed embedded or cron session.
