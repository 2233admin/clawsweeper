## Summary

Keep open. Current main still has a real gap between the sandbox workspace setup and the prompt used by embedded runs: non-rw sandbox workspaces receive copied skills, but the embedded runner can keep using a host-built skills snapshot whose prompt is returned before sandbox-loaded entries are considered. The same-day discussion confirms the symptom is still active, and the adjacent open PRs in the provided context do not prove the runtime remap contract is fixed.

## What This Changes

Sandboxed non-main sessions should resolve skill `SKILL.md` locations to sandbox-readable copies instead of global `~/.openclaw/skills` paths.

## Best Possible Solution

Sandboxed runs with `workspaceAccess: "none"` or `"ro"` should render `<available_skills>` from sandbox-readable copied skill entries, or remap snapshot prompt locations to those copies, while keeping canonical host-resolved skill metadata available for config, env, and session state. Add regression coverage for a managed global skill used from a non-main sandboxed session.
