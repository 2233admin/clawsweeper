## Summary

Keep this PR open. Current main still lacks the requested audio-level private-network request config and OpenAI-compatible local audio auth handling, but the PR should not merge until the synthetic local-auth path is narrowed because it currently treats RFC1918/.local OpenAI audio endpoints as no-auth local endpoints too.

## What This Changes

The PR adds audio-only `tools.media.audio.request.allowPrivateNetwork` and per-model request support, threads audio request overrides into media provider execution, adjusts OpenAI-compatible audio auth for local endpoints, and updates docs, generated config metadata, tests, and the changelog.

## Best Possible Solution

Revise this PR so audio-only `allowPrivateNetwork` remains accepted and forwarded, image/video stay rejected, and OpenAI-compatible no-auth handling is either restricted to true loopback addresses or explicitly documented and tested for private-network endpoints that must not use provider-default auth.

## Reproduction Assessment

Yes. Current main can be checked statically: schema tests still reject media `allowPrivateNetwork`, runner request sanitization still drops audio-level `allowPrivateNetwork`, and OpenAI-compatible audio still always builds provider Authorization. The remaining PR issue is reproducible with an OpenAI audio model pointed at an RFC1918 base URL and a configured provider API key, because `isLocalBaseUrl` returns true before normal credential resolution.

## Solution Assessment

No, not yet. The PR is in the right owner area and covers the requested config/runtime surfaces, but the local-auth predicate should be tightened or the intended private-endpoint auth semantics made explicit before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Limit synthetic auth to loopback audio URLs:** `src/media-understanding/runner.entries.ts:457-459`
  - body: `isLocalBaseUrl` also returns true for RFC1918 and `.local` hosts, so an OpenAI audio model pointed at a private endpoint like `http://192.168.1.10:8000/v1` will skip normal provider credential resolution and use the `custom-local` marker even when that endpoint expects the configured API key. This regresses the private endpoint path the PR is enabling; use a loopback-only predicate here, or explicitly gate/document the no-auth behavior for non-loopback private hosts.
  - confidence: 0.86
