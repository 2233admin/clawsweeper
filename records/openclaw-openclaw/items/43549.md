## Summary

Keep this issue open. Current main still does not diagnose bad Telegram persisted state in status output, malformed Telegram update-offset JSON is collapsed to a null offset, shared sessions.json read paths still suppress malformed JSON, and there is no Telegram-specific backup/reset/quarantine recovery command. The Keep My Claw skill mentioned in discussion is a useful backup safety net, but it does not replace first-party Telegram status and recovery UX.

## Best Possible Solution

Keep this issue open and implement the fix in the Telegram plugin/status path. Add a read-only inspector for the resolved Telegram update-offset JSON plus relevant shared sessions.json entries, reporting path, size, parse/schema status, mtime, and account. Surface failures as ChannelStatusIssue entries in openclaw channels status / openclaw status, and add a non-destructive Telegram recovery flow that backs up or quarantines only the affected artifact and tells the operator whether to restart the channel or gateway. Use a generic core seam only if multiple channel plugins need the same persisted-state diagnostic API.
