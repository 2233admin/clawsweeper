## Summary

Keep open. Current main still contains source-backed paths for all three reported failure modes: cache-read/cache-write values can feed prompt-pressure snapshots, high-token empty preparations can still commit a structured fallback summary, and reply handling still evaluates preflight compaction before the memory-flush gate.

## What This Changes

The issue requests fixes for compaction token accounting, empty fallback-summary behavior, and memory-flush ordering in prompt-cached Anthropic/Opus long-context sessions.

## Best Possible Solution

Implement a narrow agents/compaction reliability fix that derives destructive compaction pressure from a trustworthy prompt/context envelope, prevents large preparations from committing an empty fallback summary, and guarantees memory flush is completed or recorded before any destructive compaction boundary when memory flush is enabled.

## Reproduction Assessment

Yes. Current main has a high-confidence synthetic reproduction path: feed an inflated cacheRead usage snapshot through derivePromptTokens, pass a high tokensBefore preparation with no entries passing isRealConversationMessage into the safeguard, and inspect runReplyAgent's preflight-before-flush ordering.

## Solution Assessment

Yes in direction, with provider-aware nuance. The maintainable fix is to separate billing/cache telemetry from active context pressure, add a non-destructive high-token fallback path, and coordinate memory flush before compaction rather than blanket-trusting provider usage totals.
