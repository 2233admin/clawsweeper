## Summary

Keep open. Current main has partial context-usage improvements, but the reported destructive paths are still source-backed: cache-read tokens can still contribute to prompt pressure snapshots, high-token empty preparations can still write an empty structured fallback summary, and the agent runner still performs preflight compaction before the memory-flush gate.

## What This Changes

The issue requests fixes to compaction token accounting, empty fallback summary behavior, and memory-flush ordering for prompt-cached Anthropic/Opus 1M sessions.

## Best Possible Solution

Implement a narrow agents/compaction reliability fix that uses a trustworthy prompt/context envelope for compaction pressure, prevents high-token preparations from committing an empty fallback summary, and guarantees memory flush is completed or recorded before any destructive compaction boundary is written when memory flush is enabled.

## Reproduction Assessment

Yes. Current main can be reproduced with focused unit coverage: feed an inflated cacheRead usage snapshot through the prompt-token path, pass a high tokensBefore preparation whose entries fail isRealConversationMessage into the compaction safeguard, and assert runReplyAgent evaluates preflight compaction before memory flush.

## Solution Assessment

No. Current main is not the best solution because it still allows billing/cache telemetry and strict empty-message filtering to drive destructive compaction before memory flush; the safer approach is provider-aware context accounting plus a non-destructive high-token fallback path and explicit flush-before-compact coordination.
