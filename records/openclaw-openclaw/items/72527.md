## Summary

Keep this issue open. Current main has not implemented the requested downgrade recovery: it still emits only a bare future-version warning after successful validation, while the reported downgrade cases include invalid configs that can fail before that warning path; the issue is also paired with open PR #72535, which should be reviewed rather than bypassed by cleanup.

## What This Changes

The issue asks for OpenClaw to detect and clearly repair or guide recovery when a 2026.4.x-written `openclaw.json` is read by an older 2026.3.2 binary and contains newer keys or value formats.

## Best Possible Solution

Keep the issue open while maintainers review PR #72535. The desired end state is a documented downgrade/recovery policy that surfaces actionable guidance from invalid-config paths too, and either implements known safe doctor rewrites for changed value formats or clearly lists manual corrections without overpromising automatic downgrade migration.

## Reproduction Assessment

Yes. The report gives concrete versioned steps, and current source provides a high-confidence control-flow reproduction: invalid downgrade configs can fail validation before the future-version warning runs, while doctor only strips unknown keys and does not generically downgrade changed value formats.

## Solution Assessment

Unclear. A warning-only PR is a useful narrow improvement, but it is not the full fix for the issue as reported unless invalid-config paths also receive the guidance or known downgrade rewrites are added to doctor.
