## Summary

Keep open. The issue is paired with open PR #74060, which uses closing syntax and targets the same Feishu account-scoped credential status/probe gap. Current main has account-aware gateway health plumbing, but the Feishu setup/status resolver still probes only top-level credentials, so this should remain open until the linked PR is reviewed, landed, or closed.

## What This Changes

The issue requests Feishu health/status to recognize multi-account credentials under `channels.feishu.accounts.*` instead of requiring top-level `channels.feishu.appId/appSecret`.

## Best Possible Solution

Review and either land or close PR #74060. The desired end state is that every Feishu status/probe surface resolves the selected Feishu account first and has focused regression coverage for account-scoped credentials.

## Reproduction Assessment

Unclear for the exact live `openclaw status --deep` OFF output without real Feishu credentials. The source-level setup/status gap is clear: account-scoped credentials can make Feishu configured, while `resolveStatusLines` still probes only top-level `channels.feishu` credentials.

## Solution Assessment

Yes. Routing Feishu setup/status probing through the existing account resolver is the narrow plugin-owned fix; no core migration or new API is needed.
