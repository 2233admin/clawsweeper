## Summary

Keep open because this issue is still paired with open PR #74060, which uses closing syntax and targets the remaining Feishu account-credential status/probe gap. Current main’s `status --deep` Health path is account-aware, but the Feishu setup/status surface still probes only top-level credentials, so the maintainer path is to review the linked PR rather than close this issue early.

## What This Changes

The issue asks for Feishu health/status output to treat credentials under `channels.feishu.accounts.*.appId/appSecret` as configured and probeable instead of requiring top-level Feishu credentials.

## Best Possible Solution

Review PR #74060 and either land it as the Feishu-owned setup/status cleanup or close that PR if maintainers decide the current gateway Health behavior already resolves the user-facing symptom. The desired end state is that every Feishu health/setup/status surface uses the selected-account credential resolver with focused account-scoped regression coverage.

## Reproduction Assessment

Unclear for the exact live `openclaw status --deep` Health table on current main because that path now resolves Feishu accounts through gateway health. The narrower source-level gap is clear: account-only credentials make the setup status resolver report configured while its probe still inspects only top-level credentials.

## Solution Assessment

Yes for the remaining setup/status gap: routing the probe through the existing Feishu account resolver is the narrow plugin-owned fix. The issue handling should be linked-PR review, not a separate ClawSweeper repair branch, because PR #74060 already targets this issue with closing syntax.
