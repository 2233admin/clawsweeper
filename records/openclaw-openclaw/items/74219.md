## Summary

Keep open. Current main still lacks the Feishu SDK compatibility facade needed by the reported npm-installed Feishu plugin regression, but this PR is incomplete as submitted because it recreates helper facade files without wiring their historical public subpaths and it overlaps with open competing PR #74223 for the same bug.

## What This Changes

The PR restores Feishu plugin-SDK facade source files, adds the main `feishu` SDK entrypoint/export wiring, and re-exports legacy helpers needed by published Feishu plugin packages.

## Best Possible Solution

Land one narrow, tested compatibility fix that restores the legacy Feishu SDK facade required by #74138, aligns package exports and entrypoint metadata with the intended public scope, documents/deprecates the restored surface, and adds a contract test for the legacy imports before closing the bug and superseded PR.

## Reproduction Assessment

Yes. Current main has no Feishu SDK export, and the published `@openclaw/feishu@2026.3.13` source imports `createScopedPairingAccess` from `openclaw/plugin-sdk/feishu`, so upgrading core to a version without that facade reproduces the reported runtime failure path.

## Solution Assessment

No. Restoring a compatibility facade is the right direction, but this PR is not the best submitted form because it claims to restore helper subpaths while only making `feishu` public and lacks the focused contract coverage needed for this public SDK regression.
