## Summary

Keep this issue open. Current main still supports only static binding match dimensions; `match.content`, regex/prefix content matching, and `stripMatch` are not implemented in the binding type, strict config schema, route resolver, route cache, docs, or representative inbound channel routing calls. The existing maintainer/Codex comment also kept this open, and the related #10969 and #11220 model-routing requests are not canonical replacements for this agent-routing feature.

## Best Possible Solution

Keep this issue open for explicit maintainer product/design work. A good implementation would add a validated content matcher to AgentBindingMatch, update strict Zod validation, generated schema/help/labels/docs, route precedence and cache keys, representative channel/plugin routing calls, and focused tests. If stripMatch stays in scope, define a safe BodyForAgent transformation contract before dispatch. Treat #10969 and #11220 as related model-routing context, and #38138 as a narrower related mention-routing sibling, not canonical replacements.

## Risks / Open Questions

- Content-sensitive routing needs a deterministic precedence decision relative to peer, parent peer, wildcard peer, guild+roles, guild, team, account, and channel tiers.
- Regex or prefix matching in gateway config needs validation and safety limits before accepting operator-provided patterns.
- stripMatch implies a cross-channel message body transformation contract, not only a route resolver field.
- The resolved-route cache is currently keyed only by static route metadata; adding content routing without redesign could misroute later messages.
