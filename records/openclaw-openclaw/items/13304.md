## Summary

Keep open. Current main still supports bindings as static route selectors only: the public binding type and strict validation schema do not include a content matcher, the route resolver has no message-text input or rewritten-body output, and representative channel call sites resolve routes before passing any body text. The related open mention-routing request #38138 is narrower, while #11220 and #10969 are model-routing or middleware requests, so none supersedes this feature.

## What This Changes

This issue requests optional binding content matchers, such as regex or prefix rules, with optional matched-prefix stripping so inbound messages can route to different agents based on message text.

## Best Possible Solution

Define a first-class validated binding content matcher with deterministic precedence, safe text normalization and regex constraints, content-aware or bypassed route caching, representative channel/plugin plumbing for normalized inbound text, documented body-stripping semantics, and regression coverage for static and content-routing combinations.

## Risks / Open Questions

- Precedence is still undecided across static binding tiers, runtime conversation bindings, ACP bindings, focus/thread bindings, and mention gating.
- stripMatch needs an API decision on whether it rewrites only the agent-facing body or also raw body, command text, reply context, grouped history, and channel metadata.
- Regex matching needs normalization and ReDoS guardrails before becoming a public config surface.
- The existing route cache/index strategy is keyed on static metadata and could return stale routes unless the content-routing contract explicitly changes it.
