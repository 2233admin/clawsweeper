## Summary

Keep open. Current main still treats bindings as static route selectors, and the existing mention-pattern support is only activation/gating rather than declarative content-based agent routing. The request is valid, but the safe implementation boundary needs maintainer API decisions across config validation, route precedence, route caching, channel body normalization, regex safety, and optional body rewriting.

## What This Changes

Add an optional message-content matcher to agent bindings so prefixes or regex patterns can route messages in the same channel or peer to different agents, optionally stripping the matched text before delivery.

## Best Possible Solution

Define an additive, validated binding content matcher in the public config contract, specify deterministic precedence with existing route tiers and mention behavior, make the resolver/cache contract content-safe, plumb normalized inbound text through representative channel paths, document `stripMatch`, and add regression coverage for static and content-rule combinations.

## Risks / Open Questions

- Precedence still needs a maintainer decision across static binding tiers, mention gating, parent-thread inheritance, runtime conversation bindings, ACP configured bindings, and default routing.
- Regex or prefix matching needs explicit normalization, safe-regex rules, and length/performance limits before user-controlled message content is evaluated in the route path.
- `stripMatch` needs a public contract for whether it rewrites only the agent-facing body or also command text, raw body, reply context, grouped history, transcripts, and downstream channel/plugin metadata.
- Current route caching is intentionally static; content-sensitive rules need either a content-aware cache key or a documented bypass to avoid routing one message by another message's cached result.
