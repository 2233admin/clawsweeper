## Summary

Current main still does not implement manifest-level install-time configPatch. Provider-auth configPatch exists, but openclaw.plugin.json parsing, install result types, CLI install flows, persistence, tests, and public docs do not expose or apply a manifest-declared patch for service/tool plugins. The issue remains concrete and active, with useful linked use cases from claude-search-proxy and tools.alsoAllow plus a closed, unmerged implementation attempt in #43421.

## Best Possible Solution

Keep this issue open for a core plugin-install/API design. A complete fix should add a typed manifest-level config intent or equivalent install-time plugin config API, parse and sanitize it in the manifest/install path, validate the post-merge config, propagate it through local/link/archive/npm/marketplace/ClawHub/bundled install flows, define array/conflict semantics, show a diff plus skip path, document uninstall guidance, and add regression coverage for unrelated config preservation.

## Risks / Open Questions

- Install-time config mutation is trust-sensitive and needs schema validation, prototype-pollution protection, conflict handling, user-visible diff/skip behavior, and preservation of config includes/unrelated keys before it should ship.
- Array semantics are unresolved: the issue proposes replace semantics, while the tools.alsoAllow use case in the comments likely needs additive/union behavior to avoid clobbering other plugins.
- Related #31251 shows plugin install/uninstall config writes have previously risked unrelated config drift, so any implementation needs regression coverage for preserving unrelated keys and included config files.
