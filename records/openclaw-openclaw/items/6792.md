## Summary

Current main still does not implement manifest-level install-time configPatch. The manifest contract and loader only expose existing metadata such as configContracts and channelConfigs, install results cannot carry a patch, and persistPluginInstall only mutates plugin install state/enablement. The issue should remain open for maintainer design because plugin manifests writing user config is a trust-sensitive public plugin API change; PR #43421 attempted this but is closed and unmerged.

## Best Possible Solution

Keep this issue open for maintainer design of a typed, auditable install-time plugin config API. The best path is to define a manifest-level config intent or equivalent plugin-install contract, sanitize and validate patches before writing, specify merge/conflict/array behavior, show an auditable diff and skip path, document uninstall/manual-revert guidance, and add regression coverage across all plugin install flows while preserving unrelated config.

## Risks / Open Questions

- Manifest-driven config writes are security and trust sensitive: a third-party plugin could alter model, provider, tool, channel, hook, service, or secret-adjacent configuration during install.
- Merge semantics are unresolved for arrays and shared config branches; replacing arrays would clobber entries such as tools.alsoAllow, while union behavior may be field-specific.
- The user-visible contract needs confirmation/skip behavior, validation before write, audit output, backup/recovery guidance, and uninstall/manual-revert semantics before implementation.
- All install entry points would need consistent behavior: local path, linked path, archive, npm, ClawHub, marketplace, bundled, and chat-native plugin install flows.
