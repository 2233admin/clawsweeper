## Summary

Keep open. Current main still has Discord static presence, auto-presence, and the gated `setPresence` message action, but it still does not expose the requested `api.runtime.channel.discord.updatePresence()` helper for code plugins; this is a public plugin-runtime API decision for Discord/plugin-runtime maintainers.

## What This Changes

Expose a Discord plugin runtime helper so plugins can dynamically update bot presence and status without restarting the gateway.

## Best Possible Solution

Keep this open for Discord/plugin-runtime owner review. The best path is either a narrow Discord-owned, documented runtime helper that reuses the existing gateway registry, action gate, validation, account selection, and tests, or an explicit maintainer decision that the existing gated `setPresence` message action is the supported code-plugin path.

## Risks / Open Questions

- A new helper would mutate the live Discord gateway session and should reuse the same operator opt-in, account resolution, connected-gateway checks, validation, and errors as the existing `setPresence` action.
- Auto-presence and static startup presence also write through `gateway.updatePresence(...)`; plugin-driven updates need clear precedence so they are not unexpectedly overwritten.
- The proposed `api.runtime.channel.discord.*` shape touches a public plugin-runtime boundary while current core docs/types describe `api.runtime.channel` as a generic helper surface.
- Implementation should re-check the `@buape/carbon/gateway` `GatewayPlugin.updatePresence` and `UpdatePresenceData` contract before freezing a public SDK helper shape.
