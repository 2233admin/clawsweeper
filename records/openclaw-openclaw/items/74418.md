## Summary

Keep open. Current main still has the resolver gap this PR targets: per-model `params.thinking` is read, but only canonical string levels are accepted before falling through to provider/model defaults, which matters now that DeepSeek V4 advertises a `high` default. The PR is a plausible narrow fix for the linked regression, but it should get normal maintainer review and stronger catalog-backed coverage before merge.

## What This Changes

The PR normalizes per-model `params.thinking` values `false`, `"disabled"`, and `"none"` to `"off"` in `resolveThinkingDefault` and adds three DeepSeek v4-pro model-selection tests for those aliases.

## Best Possible Solution

Merge this PR or an equivalent resolver-level normalization after strengthening the tests to cover the DeepSeek V4 catalog/provider default path, then leave the broader `extra_body` collision to #75040.

## Reproduction Assessment

Yes. The linked regression gives a concrete DeepSeek v4-pro config and current source confirms the path: ignored disabled aliases fall through to the DeepSeek V4 `high` default, while the wrapper only strips replayed `reasoning_content` when the resolved level is disabled.

## Solution Assessment

Yes, the resolver is the narrowest maintainable place to normalize config aliases before provider fallback. The safer adjustment is to add catalog-backed coverage so the tests prove the actual regression path, not only the no-catalog fallback.
