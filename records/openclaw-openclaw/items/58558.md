## Summary

Keep open. Current main still explicitly defines `tool_result_persist` and `before_message_write` as synchronous hooks, warns on Promise returns, and ignores async results, so the reported governance/plugin gap remains real. The AxonFlow plugin use case and SonicBotMan's deployment comment both point to the same missing async pre-persistence policy point rather than a stale or duplicate request.

## What This Changes

The issue requests Promise-return support for the `tool_result_persist` plugin hook, with similar consideration for `before_message_write`, so plugins can asynchronously scan or rewrite tool-result transcript entries before persistence.

## Best Possible Solution

Add a first-class awaited pre-persistence hook path for OpenClaw-owned tool-result transcript writes, preserving priority order, existing final caps/redaction, failure policy, and plugin API compatibility; update hook types, docs, API/baseline checks, and focused regression tests so async handlers can actually affect persisted messages rather than only compile.

## Reproduction Assessment

Yes. The current `src/plugins/hooks.sync-only.test.ts` reproduction registers async handlers and asserts their results are ignored, and source/docs independently show the sync-only contract.

## Solution Assessment

Unclear. Promise support is the right capability direction, but the best implementation needs maintainer design around the synchronous `SessionManager.appendMessage` interception path rather than a narrow type-only patch.
