## Summary

Keep #58558 open. Current main still intentionally treats `tool_result_persist` and `before_message_write` as synchronous-only public plugin hooks, and tests assert that async handlers are warned and ignored. The issue is a concrete plugin API gap for external governance plugins, not a ClawHub-only plugin request; OpenClaw has an async tool-result middleware seam, but current docs and registry enforcement restrict it to bundled plugins.

## Best Possible Solution

Keep the issue open for a core/plugin-API design. The best implementation is an additive async-safe persistence/governance path: allow Promise returns for `tool_result_persist` and likely `before_message_write`, update `runToolResultPersist`/`runBeforeMessageWrite` and all transcript write call sites to await before writing or replaying model context, preserve priority/terminal semantics, add tests for async redaction/blocking/failure policy, and update the public plugin docs. If maintainers prefer not to make transcript append async, define a supported external-plugin trust policy for pre-model tool-result middleware instead.

## Risks / Open Questions

- Changing only the TypeScript signature would be insufficient; the synchronous append/guard path and Codex transcript mirror helpers would still need awaitable handling before persistence.
- A pre-model tool-result governance surface has trust and latency implications similar to the bundled-only middleware, so the maintainer design may need an explicit external-plugin trust policy, timeout/fail-open behavior, or a narrower async hook contract.
