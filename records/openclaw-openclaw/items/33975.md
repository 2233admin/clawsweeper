## Summary

Keep open. Current main covers the model-attribution half through `messages.responsePrefix` template variables, including `{modelFull}` and `{provider}`, but the fallback-approval half is still missing. The model config schema/type expose only `primary`, `fallbacks`, and `timeoutMs`, and `runWithModelFallback` still advances directly to the next candidate without any user approval policy or callback.

## Best Possible Solution

Keep this issue open as the fallback-approval tracker while treating model attribution as satisfied by `messages.responsePrefix` templates. A good implementation would add an opt-in per-agent fallback approval policy to `AgentModelConfig` and `AgentModelSchema`, pause `runWithModelFallback` before non-primary attempts, route approval through an existing approval/notification surface, define timeout proceed/fail semantics, and add docs plus tests for approve, deny, timeout, cancellation, and channel delivery.

## Risks / Open Questions

- Closing this as implemented would lose the still-missing opt-in approval gate before switching from a primary model to a fallback model.
- Existing related visibility work, including response-prefix attribution and usage-footer/display issues, does not replace approve/deny/timeout behavior before fallback execution.
- Fallback approval needs product decisions around approval delivery, approver identity, timeout policy, cancellation, and interaction with existing exec/plugin approval surfaces.
