## Summary

Keep open. Current main covers the model-attribution half through `messages.responsePrefix` template variables, but the fallback-approval half is still missing: the model config schema/type have no approval policy, and `runWithModelFallback` still advances to fallback candidates without an approve/deny/timeout gate.

## Best Possible Solution

Keep this issue open as the fallback-approval tracker while treating model attribution as satisfied by `messages.responsePrefix` templates. A good implementation would add an opt-in per-agent fallback approval policy to `AgentModelConfig` and `AgentModelSchema`, pause `runWithModelFallback` before non-primary attempts, route approval through an existing approval/notification surface, define timeout proceed/fail semantics, and add docs plus tests for approve, deny, timeout, cancellation, and channel delivery.

## Risks / Open Questions

- A fallback approval feature still needs maintainer product decisions around approval delivery, approver identity, timeout policy, cancellation, and interaction with existing exec/plugin approval surfaces.
