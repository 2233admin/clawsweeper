## Summary

Keep open. Current main has partial provider-env scrubbing for the default OpenClaw ACP bridge, but acpx-backed harness launches still do not expose generic auth-contract metadata, per-agent provider env scrubbing, or fail-closed auth preflight for Gemini-style ACP agents. The issue is security-sensitive and needs maintainer design rather than auto-close cleanup.

## What This Changes

The issue requests ACP auth-contract metadata, integration-level provider env scrubbing, fail-closed preflight hooks, and mixed-credential Gemini/ACP documentation.

## Best Possible Solution

Keep this open for a maintainer-owned design that adds a generic ACP/acpx auth contract: per-agent expected auth mode, fallback allowance, provider env denylist/scrub policy, fail-closed preflight, and docs for mixed OAuth/API-key hosts, with Gemini validated in a staged environment.

## Reproduction Assessment

Unclear. The issue includes a credible staged A/B report, and static review confirms the requested OpenClaw hooks are missing, but this review did not reproduce Gemini CLI auth drift live from a runnable fixture.

## Solution Assessment

No. The current partial OpenClaw bridge env scrub is not the best full solution for acpx Gemini harnesses; the maintainable path is a generic ACP auth-contract/preflight seam with Gemini policy layered on top.
