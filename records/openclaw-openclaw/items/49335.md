## Summary

Keep this PR open. Current main still synchronously reads and parses the full transcript in `readSessionMessages()`, so the user-visible freeze problem is not implemented on main. The PR is a plausible focused Gateway fix, but it still needs maintainer review/rework because the discussion includes unresolved correctness concerns around sequence stability, pagination, non-message records, and tail truncation semantics.

## Best Possible Solution

Keep the PR open and require a targeted Gateway implementation that prevents full synchronous transcript reads/parses without breaking active-branch handling, compaction markers, sanitized media-heavy history, or stable session-history sequence numbers. The likely path is a bounded/streaming transcript reader with explicit tests around large JSONL files, oversized lines, non-message records, and pagination cursor stability.

## Risks / Open Questions

- Merging the PR as-is could regress session-history pagination because tail truncation and synthetic placeholders need stable absolute `__openclaw.seq` values for cursor consumers.
- A bounded reader must preserve compaction and non-message semantics; emitting placeholders before knowing the record type can add fake chat turns and hide compaction markers.
- The PR currently shows only a production file change in the provided file list, so it needs regression coverage for large transcripts, oversized UTF-8 lines, compaction records, active-branch transcripts, and session-history cursor behavior before landing.
