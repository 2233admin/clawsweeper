## Summary

Keep this PR open. Current main still synchronously reads and parses the full session transcript before chat.history applies its response limits, so the PR's central freeze fix is not implemented on main. The PR remains a plausible Gateway hot-path fix, but the review discussion identifies real follow-up concerns around stable sequence metadata, pagination cursors, compaction records, and oversized media-heavy entries.

## Best Possible Solution

Keep the PR open and require a targeted Gateway transcript reader that bounds read/parse cost while preserving current active-branch behavior, compaction markers, sanitized media-heavy history, and stable absolute session-history sequence numbers. Add focused tests for large JSONL files, oversized UTF-8 records, active-branch transcripts, compaction records, media-heavy messages, and cursor stability before merging.

## Risks / Open Questions

- Merging this PR as-is could regress session-history pagination or SSE cursor behavior if tail truncation or oversized placeholders renumber or omit __openclaw.seq metadata.
- A raw oversized-line guard before record-type detection could turn compaction or other non-message transcript records into synthetic chat turns instead of preserving existing compaction-divider semantics.
- Media-heavy turns can be much larger on disk than after chat.history sanitization strips inline image data, so a raw-line cap needs tests that preserve useful text/metadata where the sanitized response would fit.
