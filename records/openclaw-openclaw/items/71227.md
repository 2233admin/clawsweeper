## Summary

Keep this open. Current main still performs size-proportional hot session-store updates: the update path re-reads and parses the full `sessions.json`, mutates one entry, stringifies the full store, and writes it back while large `skillsSnapshot` and `systemPromptReport` payloads remain inline. The newer maintenance/cleanup path helps old or over-cap stores, but it does not solve the reported case of a recent store below the default entry cap that is large because snapshots are duplicated.

## What This Changes

The issue reports that large per-agent `sessions.json` files can consume the agent run budget during session metadata updates and cause empty `No reply from agent` responses, and requests a retention or storage redesign.

## Best Possible Solution

Choose a backward-compatible session metadata storage design that makes updating one active session avoid scaling with the full retained store size. A maintainable first step would keep reading existing inline entries while writing repeated `skillsSnapshot` payloads out-of-line or sharding per-session metadata, with regression coverage for large recent stores below the entry cap.

## Reproduction Assessment

Yes. The issue provides concrete production file sizes and gateway logs, and current main confirms the full-file parse/stringify/write path; a focused regression can synthesize many recent entries with repeated snapshots and time a single session update.

## Solution Assessment

Unclear for the implementation choice. Content-addressed `skillsSnapshot` storage is a plausible narrow first step, but sharding or a database-backed store may better address the related memory and latency reports together.
