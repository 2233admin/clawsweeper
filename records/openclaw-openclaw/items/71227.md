## Summary

Current main still has size-proportional session metadata updates: active writes load, parse, stringify, and rewrite the monolithic sessions.json, while skillsSnapshot and systemPromptReport remain inline on SessionEntry. Existing pruning, entry caps, cache work, and disk-budget controls are useful mitigations, but they do not implement snapshot deduplication, per-session sharding, SQLite-backed metadata, or off-critical-path writes. Related #69451 tracks the memory/RSS angle, and open PRs #48425 and #68554 overlap with possible fixes but are not merged and do not supersede this latency/run-budget report.

## What This Changes

The issue asks OpenClaw to prevent large per-agent sessions.json files from consuming the agent run budget during session metadata read/parse/stringify/write updates, especially from repeated inline skillsSnapshot data.

## Best Possible Solution

Keep the issue open until a merged, backward-compatible storage change removes the run-critical size dependence. A conservative path is to stop persisting duplicated skillsSnapshot payloads or store them content-addressed while preserving reads of existing inline entries; the durable path is sharded or SQLite-backed session metadata with large writes moved out of the active run path. Any fix should coordinate with #69451 and the overlapping open PRs #48425 and #68554, then add regression coverage for updating one active session in a large store with repeated snapshot data.
