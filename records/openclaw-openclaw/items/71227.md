## Summary

Keep open. Current main still has size-proportional hot session-store updates, and the newer maintenance controls do not solve large recent stores where duplicated inline snapshots dominate latency.

## What This Changes

Reduce `sessions.json` update latency so large recent stores with duplicated `skillsSnapshot` data do not consume the agent run budget and surface as `No reply from agent`.

## Best Possible Solution

Adopt a backward-compatible session metadata design where updating one active session no longer scales with the full retained store size. The implementation should read legacy inline entries, write repeated snapshot payloads out of line or shard metadata per session, and include regression coverage for large recent stores below the entry cap.

## Reproduction Assessment

Yes. The issue provides production file sizes and logs, and current main confirms the full-file read/parse/stringify/write path; a focused regression can synthesize many recent entries with duplicated snapshots and exercise one session update.

## Solution Assessment

Unclear for the final implementation choice. Content-addressed `skillsSnapshot` storage is a plausible narrow first step, but sharding or a database-backed store may better address this latency report together with the related memory-growth reports.
