## Summary

Keep open for maintainer review. Current main still lacks the Slack hooks required by generic configured ACP binding routing, and this PR follows the existing channel-owned pattern used by other bundled channels. I found no blocking functional or security finding in the supplied diff.

## What This Changes

The PR adds Slack configured ACP binding normalization/matching, Slack inbound configured-binding routing and readiness/drop handling, regression tests for stable configured ACP session keys, and two unrelated extension test lint cleanups.

## Best Possible Solution

Land a focused Slack-owned fix that registers Slack as a configured binding provider, resolves configured ACP routes only when no runtime conversation binding is active, gates dispatch with configured ACP readiness, drops unavailable configured ACP targets, and keeps regression coverage for stable configured ACP session keys.

## Reproduction Assessment

Yes. Code inspection against current main gives a high-confidence reproduction path: configure an ACP binding for Slack and inbound Slack routing has no Slack binding adapter, no configured route fallback, and no readiness gate to honor it.

## Solution Assessment

Yes. The PR’s Slack-local adapter plus routing/readiness changes mirror the existing Discord and Telegram pattern without changing core configured-binding APIs; the unrelated lint cleanup is separable but not a correctness blocker.
