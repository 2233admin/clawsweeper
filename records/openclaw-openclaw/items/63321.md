## Summary

Keep this PR open. Current main still has the strict Tailscale JSON parsing sites the PR targets, but the PR’s shared `return {}` behavior would change whois/auth cache semantics and Funnel diagnostics instead of only preventing crashes.

## Best Possible Solution

Keep the PR open and request a narrower fix: preserve parse failures as errors for whois/auth and as an unknown Funnel status, only tolerating empty/noisy output where that is semantically valid. Add focused regression tests for malformed `tailscale status --json`, malformed `tailscale whois --json` cache TTL behavior, and malformed `tailscale funnel status --json` diagnostics before merge.

## Risks / Open Questions

- Returning `{}` from the shared parser would erase the difference between malformed output and a valid empty object.
- Malformed `tailscale whois --json` output would be cached through the normal success/null TTL instead of the shorter error TTL, potentially prolonging Tailscale Serve auth failures.
- Malformed `tailscale funnel status --json` output would follow the disabled-Funnel setup path even though the real state is unknown.
- The PR lacks focused regression tests for malformed status, whois cache TTL behavior, and Funnel status parse failures.
