## Summary

Keep open. The PR addresses a real Kilocode plugin model-discovery gap, but the proposed fix only reads credentials from process.env, while OpenClaw catalog discovery can resolve Kilocode credentials from auth profiles, config, or an injected catalog env without mutating process.env. That leaves the reported BYOK model-discovery path unfixed for supported credential sources.

## What This Changes

The PR adds Kilocode model-discovery request headers from trimmed KILOCODE_API_KEY and KILOCODE_ORGANIZATION_ID, adds header-focused tests, and updates the raw-fetch allowlist line.

## Best Possible Solution

Thread the resolved Kilocode discovery credential from ProviderCatalogContext into buildKilocodeProviderWithDiscovery and discoverKilocodeModels, preserving env compatibility and static fallback behavior. Add coverage for profile/config-resolved discovery credentials as well as the env-header path, then confirm the Kilo Gateway header contract before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the resolved discovery credential for auth:** `extensions/kilocode/provider-models.ts:131-133`
  - body: The new header reads only process.env, but model discovery runs through ProviderCatalogContext where auth-profile, config, and injected-env credentials are exposed as discoveryApiKey without necessarily mutating process.env. With a Kilocode key stored in an auth profile, this still fetches the catalog unauthenticated and BYOK models can remain hidden. Please pass the resolved discovery key from the catalog context into discoverKilocodeModels and cover that path in tests.
  - confidence: 0.84
