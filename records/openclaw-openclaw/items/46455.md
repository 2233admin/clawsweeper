## Summary

Keep this PR open for normal maintainer review. Current main already shipped the zero-valued poll-default fix and has moved shared rich sends toward `presentation`, but this branch still broadens poll stripping beyond current validation and handles the Discord empty-component skeleton problem in shared core rather than the Discord parser/runtime path; focused related work such as #56831 is a better fit for the remaining Discord issue.

## What This Changes

The branch changes shared message send handling to strip poll params and empty `components` skeletons on `action="send"`, changes poll intent detection to require `pollQuestion`, and updates outbound poll/component tests.

## Best Possible Solution

Keep the shipped zero-default poll behavior unless maintainers explicitly choose broader silent stripping, and address empty Discord component skeletons in `extensions/discord/src/components.parse.ts` or the Discord send runtime with focused tests. Shared rich sends should continue using the generic `presentation` surface instead of adding core cleanup for provider-native `components`.

## Reproduction Assessment

Yes. The poll behavior is reproducible through current send-validation tests that reject non-zero/true poll params on `send`, and the Discord empty-modal path is source-reproducible through `handleDiscordMessageSendAction()` calling `readDiscordComponentSpec()` with `modal.fields: []`.

## Solution Assessment

No. The branch is useful context, but it is not the best current fix because it silently drops explicit non-question poll intent and only sanitizes empty Discord components in the shared runner path rather than in the Discord parser/runtime where native component payloads are consumed.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve non-question poll intent validation:** `src/poll-params.ts:54-57`
  - body: Changing `hasPollCreationParams()` to return true only for non-empty `pollQuestion` means sends with `pollOption`, `pollDurationSeconds: "60"`, `pollPublic: true`, or negative durations are treated as noise and then stripped into a plain send. Current main intentionally preserves those non-zero/true values as poll intent and has send-validation coverage for the corrective error, so keep zero/default tolerance without silently dropping explicit non-question poll fields.
  - confidence: 0.88
- **[P2] Handle empty Discord components in the parser:** `src/infra/outbound/message-action-runner.ts:724`
  - body: `stripEmptyComponents(params)` only runs in the shared message runner before generic send dispatch. The failing Discord path still reads native `ctx.params.components` in `sendMessage` and calls `readDiscordComponentSpec`, so direct Discord actions and legacy Discord component payloads can still throw on an empty modal skeleton. Move the empty-spec tolerance into the Discord parser/runtime so every native component consumer gets the fix.
  - confidence: 0.8
