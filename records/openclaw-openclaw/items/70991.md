## Summary

Keep open. Current main has important backend abort support, but it still does not fully solve the reported Control UI failure after reconnect/reload: the Stop affordance is still tied to client-local run state that reconnect clears, while the connection-drop side still overlaps Gateway keepalive behavior.

## What This Changes

The issue reports that long OpenClaw Control tasks can drop the Gateway connection, hide the Stop button after reload, and leave `/stop` unable to interrupt the still-running model.

## Best Possible Solution

Keep this as the canonical user-facing regression until maintainers land a coordinated fix that combines Gateway connection-health work with a Control UI abort affordance based on server session active-run state, using the existing session-scoped `chat.abort` fallback after reconnect or reload.

## Reproduction Assessment

Unclear overall. The issue gives plausible long-task reproduction steps but no logs; source inspection gives a high-confidence static reproduction for the Stop-button part because reconnect clears `chatRunId` and rendering hides Stop when `chatRunId` is absent.

## Solution Assessment

No. Current main is a partial solution: backend session-scoped abort and queued reconnect aborts are the right primitives, but the UI still needs to surface Stop after local run state is lost and the Gateway keepalive path still needs coordinated handling.
