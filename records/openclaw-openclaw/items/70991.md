## Summary

Keep this issue open. Current main partially supports session-scoped backend aborts and connected `/stop`, but the reported recovery problem is still present: Stop visibility is still tied to volatile browser-local `chatRunId`, reconnect clears that state, the hello snapshot/protocol does not expose active abortable run state, and the Gateway WebSocket path still has no protocol-level ping keepalive. Related #56668 is open and addresses only the keepalive subproblem.

## What This Changes

The issue asks for long-running Gateway/Control UI connections to stay stable and for Stop or `/stop` to remain usable after reload or reconnect while a task is still running.

## Best Possible Solution

Implement an additive Gateway recovery contract that lets Control UI discover abortable active work after reconnect or reload, keep a session-level Stop control available while that work can be aborted, use session-scoped abort when the run id is unknown, and add protocol-level WS ping/pong keepalive with regression coverage and updated Control UI docs.

## Repair Work Prompt

Fix the Control UI/Gateway long-task recovery regression tracked by openclaw/openclaw#70991. Current main has partial session-scoped abort support, but Stop disappears after reload/reconnect because UI state depends on local `chatRunId`, reconnect clears it, the hello snapshot has no active abortable run state, and the Gateway WS handler has no protocol ping keepalive. Implement a narrow maintainer-quality fix: add an additive Gateway/protocol surface or snapshot field that lets Control UI know the current session has abortable active work after reconnect/reload; keep a Stop affordance available for that session even when a specific runId is unknown; call `chat.abort` with `{ sessionKey }` when needed and preserve existing runId-specific abort when known; add server-side WS ping/pong keepalive with cleanup on close, coordinating with or crediting the partial approach in #56668; add regression tests for reload/reconnect active-run recovery, disconnected/local-state behavior, `/stop`/session abort after reload, and WS ping lifecycle. Do not add core owner-specific IDs or broad config migrations. Update `docs/web/control-ui.md` if the public abort/reconnect contract changes, and add a single-line changelog fix entry with `Thanks @alexandre-leng`; credit @joelagnel too if the #56668 keepalive work is incorporated.
