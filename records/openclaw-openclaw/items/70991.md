## Summary

Keep this issue open. Current main has important abort-path improvements, including session-scoped `chat.abort`, `/stop` routing, and queued aborts after reconnect, but the reported user-visible failure is not fully solved: the Control UI still hides the Stop button when `chatRunId` is lost after reconnect/reload, and the connection-drop side still needs coordination with the open keepalive work.

## What This Changes

The issue asks for stable Gateway-to-Control UI connectivity during long tasks and for Stop or `/stop` to remain available after disconnect/reload so active model work can be interrupted.

## Best Possible Solution

Keep the issue as the canonical user-facing regression until maintainers either land the connection-health PR path and add a Control UI session-level abort affordance after reconnect/reload, or split those into linked, narrower follow-ups with clear ownership.
