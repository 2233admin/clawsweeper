## Summary

Keep open. Current main has useful partial abort fixes, including session-scoped backend abort and queued reconnect aborts, but the Control UI still hides Stop after reconnect/reload once its local run id is cleared, and the Gateway WebSocket connection-drop side is still represented by open ping/connection-health PRs rather than a landed main implementation.

## What This Changes

The issue reports long-task Gateway-to-Control UI connection drops and asks for the Stop button and `/stop` fallback to remain able to interrupt the active model run after reconnect or reload.

## Best Possible Solution

Keep this as the canonical user-facing regression until maintainers land one Gateway WebSocket keepalive/connection-health path and update the Control UI to derive abort availability from active session state after reconnect or reload, using session-scoped `chat.abort` when the client no longer has a run id.

## Reproduction Assessment

Unclear for the full bug because no logs were provided and I did not run a live long-task Windows/Kimi scenario. Static inspection gives a high-confidence reproduction for the missing Stop button: start a run, reconnect or reload so `chatRunId` is cleared, and `canAbort` becomes false even if server work is still active.

## Solution Assessment

No, current main is not the complete solution. The backend session-scoped abort and queued reconnect abort are the right primitives, but the visible UI affordance still depends on local run state and the connection stability work is still split across open PRs.
