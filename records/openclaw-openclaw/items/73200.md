## Summary

Keep this PR open. Current main still does not implement the PR's intended behavior: expired or unstamped `lost` task audit findings remain errors, and direct audit inputs still flag any `startedAt < createdAt` ordering as `inconsistent_timestamps` with no tolerance. Existing docs and tests still encode the current policy, so the PR remains a real maintainer policy/review decision rather than obsolete work.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to decide whether terminal `lost` findings should always be warnings, update both task docs if that policy changes, keep or refine the audit-level timestamp-skew regression tests if tolerance is desired beyond registry normalization, and remove the redundant test assertion before merge if maintainers agree.
