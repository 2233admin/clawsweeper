## Summary

Keep this PR open. Current main still has the behavior the PR is trying to change: expired or unstamped `lost` tasks are `error`, and direct audit inputs still flag any `startedAt < createdAt` as `inconsistent_timestamps` with no tolerance. The PR remains a focused behavior change for maintainer review, not obsolete work already implemented on main.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to decide the task-audit policy for terminal `lost` records, update `docs/automation/tasks.md` and `docs/cli/tasks.md` if the severity changes, keep focused regression coverage for any accepted timestamp tolerance, remove the redundant test assertion, then run the targeted task audit tests and changed gate before merge.
