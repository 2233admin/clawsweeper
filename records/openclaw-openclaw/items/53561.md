## Summary

This PR should not be auto-closed as implemented. Current main still sends the Gateway `connect` request through the generic request watchdog, and the PR's proposed `{ timeoutMs: null }` change is not present. Related main-branch handshake/probe fixes reduce nearby false timeouts, but they do not implement this PR’s stated `callGateway` ownership change, and the existing review comment identifies a real availability risk in the PR as written.

## Best Possible Solution

Keep this PR open for maintainer review or author changes. The safe direction is a bounded connect-response deadline that preserves `callGateway`’s overall budget without making raw GatewayClient startup paths unbounded, plus focused regression coverage for both delayed-but-healthy connect responses and stalled connect responses.
