## Summary

Keep this PR open, but do not merge it as-is. The branch is a useful attempt at #73715, including follow-up work for mixed-case model IDs and auth-profile suffixes, but the current patch still misses the reported behavior: the local fallback runs only after model preparation returns an error, while configured providers can prepare and dispatch the typed case-mismatched model ID, and the gateway transport still forwards the typed model half unchanged.

## What This Changes

The PR changes `infer/capability model run` model-ref handling by trimming `--model`, retrying local preparation with a lowercased model-id segment after a strict preparation error, preserving auth profile suffix casing, and adding regression tests for mixed-case CLI model refs.

## Best Possible Solution

Use a shared model-ref resolution or validation step before dispatch for both local and gateway `model run` overrides. It should preserve truly mixed-case canonical model IDs and auth profile suffixes, but resolve or reject case-only mismatches like the Anthropic reproducer before any provider call.

## Reproduction Assessment

Yes. The linked issue gives a concrete CLI reproduction for local and gateway, and source inspection shows why this patch would still miss at least those paths without needing a live provider call.

## Solution Assessment

No. The strict-first retry is too late for configured-provider fallback behavior and is not shared with the gateway override path; the safer fix belongs at the model-ref resolver/validation seam before dispatch.
