## Summary

Keep open. The PR targets a real embedded-runner overflow recovery problem and current main does not contain the proposed outer fallback gate, but the patch still has a blocking correctness issue: the new skip decision can rely on stale prior-turn usage when deciding that tool-result truncation is futile.

## What This Changes

The PR changes embedded runner overflow recovery to estimate tool-result reduction before retrying persisted-session truncation, carries local overflow estimates into compaction, and changes auto-reply reset behavior to reset only on embedded compaction failures.

## Best Possible Solution

Keep the narrow overflow-recovery direction, but rebase onto current main and make the truncation skip depend only on attempt-local evidence, such as observed overflow tokens or attempt.estimatedContextTokens. When usage provenance is uncertain, preserve the existing safer behavior of attempting truncation once instead of skipping it.

## Reproduction Assessment

Yes. The failure can be reproduced in the run overflow harness by seeding lastRunPromptUsage with a large prior-turn value, then returning a current context overflow with a messagesSnapshot whose tool-result reduction would be sufficient; the PR's gate can skip truncation anyway.

## Solution Assessment

No. The intended gate is maintainable, but this implementation is not the best fix because it uses lastRunPromptUsage for a current-attempt decision. A safer fix is to carry attempt-local prompt estimates through EmbeddedRunAttemptResult and only skip truncation when that local estimate or a provider-reported overflow count proves truncation cannot help.

## Review Findings
