## Summary

Keep open: current main still routes plugin runtime subagent runs through direct Gateway `agent` dispatch without the registry metadata that drives subagent completion announce delivery, so the reported gap remains valid. The next step is maintainer API/product judgment because enabling automatic deferred delivery for plugin-created runs could change existing `api.runtime.subagent.run()` semantics.

## What This Changes

Request support or clear documentation for plugin `before_dispatch` hooks that start `api.runtime.subagent.run()` child runs and expect later `subagent_delivery_target` plus final external-channel delivery.

## Best Possible Solution

Define an explicit plugin-runtime deferred completion contract, then either add an opt-in/default registration path that reuses `registerSubagentRun(...)` and `runSubagentAnnounceFlow(...)`, or document `api.runtime.subagent.run()` as manual-only with plugin-owned polling and delivery. The chosen path should include SDK docs and a focused regression test for a `before_dispatch` hook that returns immediately while the child completes later.
