## Summary

Keep open. Current main has adjacent session-lock hardening, including a typed `SessionWriteLockTimeoutError`, but it still does not implement this PR’s central timeout propagation: `runEmbeddedAttempt` calls `acquireSessionWriteLock` without `timeoutMs: params.timeoutMs`. The plugin SDK also still exports `acquireSessionWriteLock` without the matching timeout error/helper, so the Greptile SDK-surface concern remains relevant.

## Best Possible Solution

Keep this PR open for maintainer review or contributor update. The clean path is to rebase it onto current main, decide whether the embedded attempt should pass `params.timeoutMs` as the lock wait or use a narrower/configurable wait, and expose the existing `SessionWriteLockTimeoutError`/code/helper through the supported plugin SDK surface if that public programmatic signal is intended. Add focused tests for embedded-attempt timeout propagation and SDK export if those changes are accepted.
