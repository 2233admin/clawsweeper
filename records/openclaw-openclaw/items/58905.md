## Summary

Keep open. Current main still has the reported Feishu/Lark webhook deadlock: webhook mode requires an encryptKey, but the request handler rejects any request missing Lark signature headers before parsing the body or running URL-verification challenge handling. The related fix PRs #58938 and #59990 are useful prior work, but the provided context shows both were closed unmerged, and their challenge-first behavior is not present in current main.

## Best Possible Solution

Implement the phase-aware Feishu webhook fix in extensions/feishu/src/monitor.transport.ts: read and parse the bounded body, allow only valid url_verification challenge payloads to call Lark.generateChallenge and return the challenge before ordinary signature validation, and keep unsigned non-challenge events rejected. Add regression coverage for unsigned encrypted URL verification and for unsigned non-challenge rejection, using #58938/#59990 as prior art rather than closing this issue.
