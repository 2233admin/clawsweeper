## Summary

Keep this PR open. Current main still leaves the reported gateway/provider-qualified image-capability case and Control UI pending-attachment race unresolved, so the work is not obsolete; the proposed Control UI patch still has correctness issues and the branch is dirty against current main.

## What This Changes

The PR adds a provider-qualified gateway image-support catalog lookup and tests, plus Control UI state intended to block send/Enter while FileReader attachment reads are pending.

## Best Possible Solution

Keep the useful gateway regression, but implement provider-qualified matching in the current shared model-catalog lookup path. Rework the Control UI side around current run controls with live pending-read deltas and latest-state attachment appends, only block user sends that depend on in-flight attachments, preserve command/new-session/compact behavior, and cover concurrent paste/select/drop races in focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Apply pending deltas against live state:** `ui/src/ui/views/chat.ts:601-602`
  - body: `updateAttachmentReadPending` derives the next value from `props.attachmentReadPending`, which is a render-time snapshot. With multiple concurrent reads, increments and `loadend` decrements can be applied to the same stale base, re-enabling send before all attachments finish. Apply deltas in the owning state instead.
  - confidence: 0.93
- **[P2] Let override commands bypass pending attachment reads:** `ui/src/ui/app-chat.ts:276-277`
  - body: This early return runs before `messageOverride` and local command handling, so actions such as new-session or compact can silently no-op while an attachment is loading even though they do not send those attachments. Restrict the guard to normal user sends that depend on pending attachment state.
  - confidence: 0.88
- **[P3] Reflect attachment processing in the send button label:** `ui/src/ui/views/chat.ts:1693`
  - body: When pending reads disable the send button, the title changes to "Processing attachment..." but the accessible name remains "Send message" or "Queue message". Screen-reader users get no reason for the disabled state; include the pending state in `aria-label` too.
  - confidence: 0.78
