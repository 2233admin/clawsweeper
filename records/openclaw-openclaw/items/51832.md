## Summary

Current main still does not cover the reported Slack error cascade. Slack drops messages from the same bot identity but accepts sibling bot messages when `allowBots` is true, and known OAuth refresh failures still become visible gateway auth guidance in non-direct conversations instead of being tagged/suppressed as gateway-generated errors.

## Best Possible Solution

Keep this issue open and implement a narrow Slack/gateway guard: classify OpenClaw-generated gateway failure replies in non-direct Slack channels and prevent sibling OpenClaw bot accounts from treating those messages as new inbound work, while preserving normal `allowBots: true` behavior for non-error bot messages. Add a regression test for two distinct Slack bot identities and an OAuth refresh failure, with optional per-channel repeated-error dedupe as defense in depth.
