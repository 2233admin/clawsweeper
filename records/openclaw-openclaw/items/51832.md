## Summary

Keep open. Current main still has only a narrow non-direct suppression for generic runner failures; known OAuth refresh failures still produce visible gateway auth guidance, and Slack still accepts sibling bot messages when `allowBots` is true. The reported two-bot Slack cascade is not covered by current tests or an implemented guard.

## Best Possible Solution

Keep this issue open and add a dedicated guard for gateway-generated error replies in non-direct, bot-visible Slack channels. The best implementation is to tag or classify OpenClaw gateway error replies so sibling OpenClaw bot accounts ignore them, then add per-channel/per-agent repeated-error dedupe and an auth-failure cooldown as defense in depth. Cover the OAuth refresh failure path with a Slack test using `allowBots: true` and two distinct bot identities.
