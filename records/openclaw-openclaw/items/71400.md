## Summary

Keep this PR open. Current main still lacks adaptive session reset semantics, but the proposed patch should not merge as-is because the help text contradicts the defaulting behavior and the adaptive hook reason path can misreport rollovers after system-event metadata writes.

## What This Changes

The PR adds `session.reset.mode: "adaptive"` across config types, schema/help/docs, session freshness logic, hook reason typing, and focused session tests so daily plus idle reset uses AND semantics.

## Best Possible Solution

Land one coherent public contract for daily-plus-idle AND semantics, either by finishing this focused `adaptive` mode PR or by explicitly superseding it with the broader #59411 design if maintainers prefer that shape.

## Reproduction Assessment

Yes. Current main can be checked by source inspection: `adaptive` is absent from the reset type/schema, and freshness still uses OR semantics; the PR's intended behavior is reproducible with daily-only, idle-only, and both-stale policy cases.

## Solution Assessment

No, not as-is. The high-level focused mode is a maintainable direction, but the patch should first align adaptive hook reasons with lifecycle timestamps and correct the `idleMinutes` wording/defaulting contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Use lifecycle timestamps for adaptive hook reasons:** `src/auto-reply/reply/session.ts:141-143`
  - body: The adaptive branch recomputes daily staleness from `entry.updatedAt`, while the actual reset decision uses `sessionStartedAt` and `lastInteractionAt`. A heartbeat/cron/exec write after the daily boundary can refresh `updatedAt` without keeping the session fresh, so the session rolls over but this path returns `undefined` instead of `"adaptive"`; reuse the same lifecycle stale state or expose it from `SessionFreshness`.
  - confidence: 0.84
- **[P2] Describe adaptive idleMinutes as defaulted:** `src/config/schema.help.ts:1330`
  - body: The new help text says `idleMinutes` is required for adaptive mode, but the resolver defaults it to `DEFAULT_IDLE_MINUTES` when omitted. Users will not get a validation error, so update this text and the generated schema description to say adaptive uses the default idle window unless configured.
  - confidence: 0.91
