## Summary

Keep #39661 open. Current main still implements Discord mention gating as a guild/channel boolean, with `users` and `roles` serving as sender allowlists rather than per-user reply-trigger policy. The preflight path computes mention requirements before/independent of sender-specific allowlist authorization, and the strict config schema does not accept the requested nested per-user policy map.

## Best Possible Solution

Keep this open for bundled Discord plugin work. Add a typed sender-specific mention policy to the Discord guild/channel config contract, validate it in zod and generated metadata, define precedence across guild defaults, guild sender overrides, channel defaults, and channel sender overrides, pass sender identity into mention resolution, and cover two users in the same channel with docs and regression tests.

## Risks / Open Questions

- A fix needs explicit precedence rules so sender authorization allowlists are not conflated with reply-trigger mention policy.
- Allowing per-user `requireMention: false` can widen activation in shared Discord channels if defaults and overrides are ambiguous.
