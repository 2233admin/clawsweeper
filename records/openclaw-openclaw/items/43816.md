## Summary

Keep this PR open. Current main still uses a fixed two-hour cron stale-running cleanup threshold in timer maintenance, while runtime execution timeouts are resolved per job. The PR targets a real remaining reliability gap linked from #30096, but its current head needs changes because the two-hour ceiling can clear locks before explicitly configured long-running jobs have timed out.

## Best Possible Solution

Keep this PR open and request changes: rebase on current main; derive stale-running cleanup from `resolveCronJobTimeoutMs(job) + buffer` for finite normal timeouts; preserve a conservative fallback for `undefined` no-timeout jobs; handle overflow-sized timeout values with an explicit policy that does not shorten valid long configured timeouts; and add regression coverage for short custom timeouts, systemEvent defaults, no-timeout fallback, and explicit long-timeout preservation.
