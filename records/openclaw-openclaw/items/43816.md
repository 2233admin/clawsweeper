## Summary

Keep open. Current main does not implement the PR's intended per-job stale-lock threshold, but the PR head still has a blocking correctness issue: it caps valid finite timeout-based thresholds at two hours, so long configured jobs can have their lock cleared while their original execution is still allowed to run. The earlier no-timeout fallback concern from review was addressed by restoring the conservative two-hour fallback.

## What This Changes

The PR imports cron timeout policy into job maintenance, replaces fixed stuck-lock cleanup with a timeout-plus-buffer threshold capped at two hours, and adds cron regression tests plus a comment update.

## Best Possible Solution

Request changes on this PR: keep timeout-derived cleanup for ordinary finite jobs, preserve the conservative no-timeout fallback, handle only overflow-sized timeout values specially, and add regression coverage for short custom timeouts, default systemEvent and agentTurn thresholds, no-timeout fallback, valid long finite timeouts, and overflow-sized timeoutSeconds.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Preserve locks until explicit long timeouts expire:** `src/cron/service/jobs.ts:393`
  - body: With an explicit timeoutSeconds above two hours but below Node's timer ceiling, runtime still allows the job to run for that configured duration. This clamp clears runningAtMs at the two-hour fallback, so a later maintenance tick can make the same job runnable while the original execution is still active. Handle overflow-sized values separately without shortening valid long timeouts.
