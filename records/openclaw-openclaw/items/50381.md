## Summary

Keep this PR open. Current main still only tails the gateway file log through `openclaw logs` and `logs.tail`; the requested `--source gateway|llm|all` support, cache-trace-backed LLM source, source metadata, hints, and pretty trace rendering are not implemented. The PR is a plausible focused feature, but the existing review comments identify real issues that should be fixed before merge rather than closing the work as obsolete.

## Best Possible Solution

Keep this PR open for maintainer/contributor follow-up. The best path is to rebase it onto current main, fix the review comments, decide whether LLM cache-trace access should remain under `operator.read` or require a narrower/sensitive scope, add docs for `openclaw logs --source llm`, and rerun targeted CLI/gateway tests plus the appropriate changed gate before merge.

## Risks / Open Questions

- Security/privacy review is still needed before merge because cache-trace output can include prompt and message content; exposing it through the read-scoped `logs.tail` surface may require stronger scope, redaction, or explicit documentation.
- The PR’s current server approach reportedly calls `loadConfig()` in the LLM log path, which could regress `openclaw logs` as a diagnostic command when config is invalid.
- The PR’s CLI rendering reportedly does not fully honor `--plain`, so copy/paste and non-styled output contracts need correction.
