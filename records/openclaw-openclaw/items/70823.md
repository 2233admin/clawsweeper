## Summary

Keep this issue open. Current main still models plugin slots as string plugin IDs through the public config type, runtime schema, generated schema, normalization, activation, loader, install/write helpers, tests, and docs. The requested owner/provenance object plus takeover invariant remains a live core config and plugin-contract RFC, not a duplicate, stale report, or ClawHub-only plugin idea.

## What This Changes

Add a backwards-compatible machine-readable ownership object for `plugins.slots.<slot>` with owner/provenance fields while preserving legacy bare-string slot configs and defining takeover rules.

## Best Possible Solution

Keep the RFC open for maintainer design review, then implement a versioned, backwards-compatible slot ownership contract across config types, schema, generated docs/help, normalization, activation, install/update/uninstall, and validation with explicit conflict handling and focused regression tests.

## Risks / Open Questions

- The exact public schema shape is still undecided: `owner` only versus `owner`, `claimed_at`, and `claimed_by_version` has migration, validation, and generated-schema consequences.
- Takeover semantics need maintainer design for default slots, stale owners, missing `plugins.entries`, uninstall/update flows, and whether existing `plugins install --force` should authorize slot replacement.
- Any implementation must preserve legacy bare-string configs, `"none"` disable behavior, `memory-core` and `legacy` defaults, and current activation/allowlist behavior while moving readers to the new canonical shape.
- Provenance fields need a reliable source of truth for package version and claim time across install, link, update, manual config edits, and runtime/plugin registration paths.
