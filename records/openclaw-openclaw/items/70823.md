## Summary

Keep open. Current `main` still treats plugin slots as bare string selectors, with schema, normalization, install selection, activation, and docs all built around plugin-id strings rather than an `{ owner, claimed_at, claimed_by_version }` object. This is a core plugin/config-contract RFC, not a ClawHub-only plugin request, but it needs maintainer design approval before an automated fix should lock in the public shape.

## What This Changes

Request to evolve `plugins.slots.<slot>` from a bare plugin-id string into a backwards-compatible owner/provenance object and later enforce slot overwrite rules.

## Best Possible Solution

Have maintainers decide the minimal slot ownership contract first. If accepted, land staged core work: add a typed `string | { owner, claimed_at?, claimed_by_version? }` read model with tests and docs, then update canonical writes and finally add an explicit overwrite/force invariant once compatibility behavior is settled.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC. The source-level verification is high-confidence: current schema/types/docs only accept string slot selectors, and object-form slot ownership is not normalized or written on current `main`.

## Solution Assessment

Unclear until maintainer design review. The proposed staged rollout is plausible and backwards-compatible, but the best field set, provenance semantics, and overwrite policy should be chosen before a fix PR changes the public config schema.
