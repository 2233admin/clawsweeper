## Summary

Keep open. Current main still models plugin slots as bare string selectors across the public config type, Zod schema, normalization, activation, install selection, tests, and docs; the requested object-form ownership/provenance contract is not implemented. This is a core plugin/config API RFC rather than ClawHub-only work, but it needs maintainer design approval before an automated PR should freeze the public shape or overwrite policy.

## What This Changes

The RFC requests a backwards-compatible `plugins.slots.<slot>` config extension from bare plugin-id strings to an object with machine-readable ownership metadata such as `owner`, `claimed_at`, and `claimed_by_version`.

## Best Possible Solution

Have maintainers decide the minimal slot ownership contract first. If accepted, land staged core work: add a typed read model and normalization for `string | { owner, claimed_at?, claimed_by_version? }` with schema/docs/tests, then update canonical writers, and only then add an explicit overwrite/force invariant once compatibility behavior is settled.

## Reproduction Assessment

Not applicable as a bug reproduction; this is an RFC. The source-level verification is high-confidence: current main still accepts, normalizes, writes, and compares plugin slots as strings rather than object ownership records.

## Solution Assessment

Unclear until maintainer design review. The proposed staged rollout is plausible and backwards-compatible, but the best field set, provenance semantics, and overwrite policy should be chosen before changing the public config schema.
