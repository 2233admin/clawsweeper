## Summary

Keep open. Current main still hardcodes the shared gateway chat-delta throttle at 150 ms, has fixed-window tests/comments around that behavior, and does not include the requested `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver. The linked implementation PR is closed unmerged, and the Telegram throttle issue is adjacent channel-specific work rather than a replacement for this gateway WebSocket path.

## What This Changes

The issue requests a configurable gateway chat-delta broadcast throttle via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the current 150 ms default.

## Best Possible Solution

A maintained solution would centralize gateway chat-delta throttle resolution, preserve the shipped 150 ms default, reject invalid and negative values, explicitly decide whether `0` disables throttling, update fixed-window comments/tests, and document the knob if it becomes a supported operator control.

## Reproduction Assessment

Yes. Source inspection and the existing unit tests provide a high-confidence path: emit two assistant deltas less than 150 ms apart and the second delta is buffered instead of broadcast until a flush path runs.

## Solution Assessment

Unclear pending maintainer product/API review. The proposed resolver is narrow and default-preserving, but choosing a new gateway tuning surface is a core operator-contract decision rather than a purely mechanical repair.
