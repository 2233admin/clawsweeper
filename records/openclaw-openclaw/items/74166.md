## What This Changes

Request to make the Gateway chat-delta WebSocket broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the current 150 ms default.

## Best Possible Solution

If maintainers accept the operator tuning knob, add a small centralized resolver with the 150 ms default, strict invalid/negative fallback behavior, an explicit decision for `0`, focused tests around the resolver and `server-chat` behavior, and docs if this becomes a supported configuration surface.

Yes. Source inspection plus existing unit tests provide a high-confidence path: emit two assistant deltas less than 150 ms apart and the later text is buffered until a final or tool-boundary flush path runs.

Unclear pending maintainer product/API review. The proposed resolver is narrow and default-preserving, but the best public surface for a new gateway tuning control should be decided before implementation proceeds.
