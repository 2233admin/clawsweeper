## Summary

Keep open. Current main still has the gateway chat-delta broadcast throttle hardcoded at 150 ms, no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver exists, and the linked implementation PR is closed unmerged; the remaining decision is whether maintainers want this operator knob as env-only or config-backed.

## What This Changes

Make the gateway chat-delta broadcast throttle configurable via `OPENCLAW_CHAT_DELTA_THROTTLE_MS` while preserving the 150 ms default and honoring `0` as throttle-disabled.

## Best Possible Solution

If maintainers accept the operator surface, add a small gateway-owned throttle resolver with the 150 ms default, strict non-negative parsing, `0` support, updated comments/tests, and docs only if this becomes a supported public knob.

## Reproduction Assessment

Not applicable as a feature request, but the missing capability is high-confidence from source inspection: current main hardcodes `< 150`, tests encode the 150 ms window, and the requested env var/resolver does not exist.

## Solution Assessment

Unclear until maintainer product review. The proposed narrow resolver is technically plausible, but the best supported surface depends on whether maintainers prefer env-only tuning or a config-schema field for Gateway streaming behavior.
