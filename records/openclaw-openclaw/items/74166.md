## Summary

Keep this issue open. Current main still hardcodes the gateway chat-delta broadcast throttle at 150 ms in `emitChatDelta`, has tests that assume that 150 ms window, and has no `OPENCLAW_CHAT_DELTA_THROTTLE_MS` resolver or documentation. The linked implementation PR #74152 is closed unmerged, and related #38066 is Telegram-specific rather than the shared gateway WebSocket broadcast path.

## Best Possible Solution

Keep this issue open for maintainer product/API review. If accepted, implement a focused gateway change that centralizes the default 150 ms throttle in a resolver, honors a validated non-negative `OPENCLAW_CHAT_DELTA_THROTTLE_MS` override, preserves default behavior, updates the throttle comments/tests, and adds docs only if maintainers want the env var treated as a supported operator knob.
