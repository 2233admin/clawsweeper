## Summary

Keep open. Current main still has useful adjacent work for LaunchAgent supervision, service entrypoint drift, extra service scans, and Telegram getUpdates conflict hints, but it does not implement the requested doctor-owned consolidated gateway diagnosis. The mixed output path remains: doctor can print an early `Gateway not running.` from RPC health and later print service/runtime state such as `Runtime: running (pid ...)` from the daemon phase, with no final healthy/degraded/down verdict or JSON diagnosis block.

## Best Possible Solution

Keep this open and implement a doctor-owned gateway diagnosis collector. It should gather service installed/loaded state, LaunchAgent supervision, runtime PID/status, port listener ownership, RPC/read-probe result, service config drift, extra gateway-like services, recent channel-specific signals such as Telegram getUpdates 409 conflicts, and last gateway errors, then print one final healthy/degraded/down verdict with deterministic next steps and an optional JSON diagnosis block. Add regression coverage for mixed RPC failure plus running runtime, entrypoint drift, missing LaunchAgent supervision, port ownership mismatch, extra services, and Telegram duplicate-poller hints.
