## Summary

Keep open. Current main has adjacent service-drift, extra-service, status, and Telegram 409 diagnostics, but `openclaw doctor` still reports gateway RPC health, service/runtime state, service config drift, port state, and channel/log clues as separate checks. The mixed output path remains possible: `Gateway not running.` can be emitted before a later daemon check reports `Runtime: running`, with no final healthy/degraded/down verdict, probable-cause ordering, or doctor-owned JSON diagnosis block.

## Best Possible Solution

Keep this issue open and implement a doctor-owned gateway diagnosis collector. It should collect service installed/loaded/runtime state, RPC health/probe result, port listener ownership, service config drift, extra gateway-like services, recent gateway errors, and recent Telegram `getUpdates` 409 signals, then print one final healthy/degraded/down verdict with probable causes and deterministic next steps. Add a small machine-readable diagnosis block only behind an explicit, documented option or stable output path.
