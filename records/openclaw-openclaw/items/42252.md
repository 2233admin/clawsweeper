## Summary

Keep open. Current main still has useful adjacent improvements for gateway status, service entrypoint drift repair, and Telegram 409 duplicate-poller hints, but `openclaw doctor` does not yet consolidate LaunchAgent/service/runtime/RPC/port/channel signals into one final healthy/degraded/down diagnosis with deterministic next steps. The mixed-message path described in this issue remains possible because gateway RPC health and daemon runtime repair are still separate doctor phases.

## Best Possible Solution

Keep this issue open and implement a doctor-owned gateway diagnosis collector. It should gather LaunchAgent/service installed and loaded state, service runtime PID/status, RPC/read-probe result, port listener ownership, service config drift, extra gateway-like services, recent Telegram `getUpdates` 409 signals, and recent gateway errors, then print one final healthy/degraded/down verdict with ordered next steps and an optional compact machine-readable diagnosis block.
