## Summary

Keep open. Current main has adjacent service, status, and Telegram conflict improvements, but `openclaw doctor` still does not produce a consolidated gateway diagnosis or final healthy/degraded/down verdict for mixed LaunchAgent/runtime/RPC states.

## Best Possible Solution

Keep this issue open and implement a doctor-owned gateway diagnosis collector that gathers service installed/loaded state, LaunchAgent supervision, runtime PID/status, RPC/read-probe result, port listener ownership, service config drift, extra gateway-like services, recent channel-specific signals such as Telegram `getUpdates` 409 conflicts, and last gateway errors. It should print one final healthy/degraded/down verdict with deterministic next steps and, if accepted by maintainers, a compact machine-readable diagnosis block.
