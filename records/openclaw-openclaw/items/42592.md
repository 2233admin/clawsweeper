## Summary

Keep open. Current main still has npm failure diagnostics and a broad retry, but it does not implement the requested structured npm error-code classification, retryable/non-retryable policy, actionable remediation map, or Windows parity. The Docker setup path still masks the optional allowedOrigins read with `|| true`, so unexpected config-read failures remain indistinguishable from an absent value.

## Best Possible Solution

Implement a scoped installer reliability change: add tested npm error-code extraction/classification, retry only known transient npm/network failures, print actionable remediation for terminal codes, make Docker `gateway.controlUi.allowedOrigins` config-get failures explicit without breaking first-run defaults, mirror reasonable behavior in `scripts/install.ps1`, and add focused Unix, Windows, and Docker setup coverage.

## Risks / Open Questions

- A smart retry wrapper should retry only known transient npm/network failures with a small bounded budget, or terminal installer failures could become slower and harder to diagnose.
- Docker config-get cleanup needs to preserve quiet first-run behavior for a genuinely missing optional key while surfacing unexpected CLI/container failures.
- Unix and Windows installers should stay behaviorally aligned so remediation messages and retry policy do not diverge by platform.
