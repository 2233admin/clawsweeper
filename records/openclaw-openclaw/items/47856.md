## Summary

Keep this issue open. Current main has partially addressed the related managed-inbound-media case discussed in the comments, but it still does not implement a configurable image-tool media root allowlist for external paths such as iMessage attachment directories. The boundedagency comments and fork commits are useful context for the workspaceOnly/inbound-media angle, but they are not a merged core fix for the requested config surface.

## Best Possible Solution

Keep open for maintainer/security design. The most likely product direction is an explicit opt-in image/media local root allowlist, with strict absolute path validation and clear interaction with `tools.fs.workspaceOnly`, while preserving the existing managed-inbound-media exemption for gateway-owned files.

## Risks / Open Questions

- Adding configurable image/media local roots changes a local file-read trust boundary and could weaken the meaning of `tools.fs.workspaceOnly` for mixed-trust gateways if not designed carefully.
- Closing now would conflate the current managed-inbound exemption with the still-missing external attachment-directory use case from iMessage.
- A narrow implementation needs schema, docs, tests, and validation for absolute non-root paths, symlinks/realpaths, per-agent precedence, and interaction with dynamic media-source root expansion.
