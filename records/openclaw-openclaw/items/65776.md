## Summary

Keep this issue open. Current main still supports only configured/static remote MCP headers: config and schema accept a scalar header record, the HTTP resolver normalizes that record once, transport construction passes it into SDK RequestInit, and runtime tool calls reuse an already-connected session without a request-scoped header resolver or runtime/session context hook. Related #65590 fixed static header forwarding, and open #62134 covers OAuth/authProvider work, but neither covers arbitrary dynamic tenant/user/correlation headers.

## What This Changes

The issue requests request-scoped/dynamic header support for remote MCP transports so runtime/session context can add auth, routing, tenant, or correlation headers before each outbound MCP request.

## Best Possible Solution

Keep this open for maintainer design. The best path is a typed request-scoped header resolver for remote MCP transports, or a public plugin/runtime API that can safely supply one. It should thread the right session/runtime context into MCP request handling, merge dynamic headers after static config with documented precedence and redaction, cover both streamable-http and SSE, and coordinate with #62134.
