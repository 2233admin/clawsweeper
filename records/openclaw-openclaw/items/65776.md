## Summary

Keep this open. Current main still supports only configured static remote MCP headers, and the related static-header forwarding fix (#65590) plus the open OAuth/authProvider PR (#62134) do not cover arbitrary per-request tenant, user, correlation, or refreshed-token headers.

## What This Changes

Request support for resolving and merging runtime-specific headers before each outbound remote MCP request to support multi-tenant routing, short-lived auth, and per-session metadata.

## Best Possible Solution

Add a typed request-scoped header resolver, or a public plugin/runtime API that can safely provide one, for remote MCP transports. It should pass the right session/runtime context, merge after static config with documented precedence and redaction, cover both `streamable-http` and `sse`, and stay coordinated with OAuth/authProvider support.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Current-main source inspection gives a high-confidence verification path: schema, resolver, transport, and runtime call paths expose no per-request dynamic header entry point.

## Solution Assessment

Unclear pending maintainer design. The hook/headersProvider direction is plausible, but the maintainable fix needs explicit API, context, precedence, redaction, and OAuth coordination decisions before implementation.
