## Summary

Keep this issue open. Current main and the latest release still support only configured/static remote MCP headers: config/schema accept a scalar header record, the resolver normalizes that record once, transport construction passes that record into MCP SDK request init/fetch wiring, and tool calls reuse the connected session without a request-scoped header resolver or session/runtime-context hook. Related #65590 covered static header forwarding, and open #62134 covers OAuth/authProvider work, but neither covers arbitrary dynamic tenant/user/correlation headers.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is a typed request-scoped header resolver for remote MCP transports, or a public plugin/runtime API that can safely supply one. It should thread the right session/runtime context into MCP request handling, merge dynamic headers after static config with documented precedence and redaction, cover both streamable-http and SSE, and coordinate with #62134 so OAuth/authProvider support is not conflated with arbitrary tenant/user/correlation headers.

## Risks / Open Questions

- Dynamic request headers are security-sensitive because they may carry bearer tokens, tenant IDs, org IDs, routing metadata, and correlation IDs; a design needs explicit trust boundaries, precedence, redaction, and audit behavior.
- Open #62134 addresses OAuth/authProvider-style token handling for remote MCP, but this issue asks for arbitrary request-scoped headers; closing or implementing one as the other would leave multi-tenant routing and correlation use cases unresolved.
- A narrow source-only implementation could accidentally cover only streamable-http or only SSE; both remote transports need consistent behavior and tests.
