## Summary

Keep open. Current main still emits `message:received` and `message:sent` internal hooks via fire-and-forget calls without draining hook-produced `event.messages` back to the user, and open PR #41375 remains the maintainer fix path for the reply-capable `message:received` surface.

## Best Possible Solution

Keep this issue open until PR #41375, or an equivalent maintainer-approved fix, lands. The best path is to drain internal hook replies only where OpenClaw has a safe reply target, add regression coverage for hook-produced reply delivery, keep the docs at https://docs.openclaw.ai/automation/hooks aligned with the actual contract, and explicitly document or defer `message:sent` auto-reply behavior.
