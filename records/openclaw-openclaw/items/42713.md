## Summary

Keep this issue open. Current main still does not provide normalized request-origin session metadata on the exhausted model-fallback wrapper path. Per-attempt provider/model data exists, and generic file logging can flatten `session_id` when a log call carries session context, but the fallback runner, summary error, decision log params, and main call sites still do not pass `sessionId`, `sessionKey`, `lane`, or `session_id` into the wrapper/decision logging surface.

## Best Possible Solution

Keep this issue open and implement a narrow observability fix in the model-fallback path: add normalized request-origin metadata such as sessionKey/sessionId or lane to `runWithModelFallback`, carry it through `FallbackSummaryError` and `logModelFallbackDecision`, emit attempted providers/models as structured fields on the exhausted wrapper, wire agent-command, auto-reply, follow-up, and memory-flush session data into the fallback call, and add focused tests for structured decision/error logs plus exhausted multi-provider fallback behavior.
