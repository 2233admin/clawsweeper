## Summary

Keep this PR open. Current main still only accepts skill paths whose realpath stays inside the currently scanned root, so the configured-root symlink behavior in this PR is not implemented. The PR remains useful, but the proposed all-configured-roots allowlist needs revision because bundled skill policy is source-classification based and the review comment identifies a credible `skills.allowBundled` bypass if non-bundled roots can symlink into bundled skills.

## Best Possible Solution

Keep this PR open for maintainer review, but require a revised skills symlink design. The fix should allow explicitly trusted non-bundled configured-root symlink targets while preserving or reapplying the target root's source policy, or excluding bundled roots from non-bundled symlink targets. Add regression coverage for allowed cross-root symlinks, unknown external symlinks still warning/skipping, and the `skills.allowBundled` bypass case before merge.

## Risks / Open Questions

- The submitted approach appears to broaden a security boundary: accepting symlink targets under bundled skills without preserving the target root's bundled source could bypass `skills.allowBundled`.
- The provided PR tests appear to check the older warning text, while current main emits `Skipping escaped skill path outside its configured root`, so the branch likely needs a rebase/test refresh.
- Related open work (#44051, #59688, and #63312) overlaps the same symlink-warning area, so maintainers should choose one consolidated policy rather than land competing partial fixes.
