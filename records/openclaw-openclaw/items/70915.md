## Summary

Keep this PR open. Current main still delegates ZAI image understanding directly to the generic helper, so the intended fix is not already implemented; the PR is a narrow implementation candidate for the linked ZAI vision endpoint bug, and static review did not find a blocking patch bug or security concern.

## What This Changes

The PR changes the ZAI media-understanding provider to rewrite ZAI vision-model requests from coding base URLs to standard base URLs and adds colocated unit tests for URL/model detection and config patching.

## Best Possible Solution

Land a narrow provider-local fix that routes only ZAI vision models on coding-plan base URLs to the corresponding standard multimodal endpoint, keeps text/coding models on their configured coding endpoint, and preserves focused coverage for glm-4.6v, glm-4.5v, glm-5v-turbo, and versioned coding URL conversion.

## Reproduction Assessment

Yes. The linked reports provide a concrete path: configure agents.defaults.imageModel as zai/glm-4.6v with a ZAI coding base URL and call the image tool; current main source shows that path keeps the configured coding base URL through generic image resolution.

## Solution Assessment

Yes. The proposed provider-local wrapper is a narrow maintainable place to handle ZAI-specific endpoint routing, and the current resolver applies the patched providerConfig.baseUrl after discovery; I did not find a blocking correctness issue in the patch from static review.
