## Summary

Current main does not fully implement the requested defensive default in the cost estimator. Runtime config materialization does fill missing model cost fields to zero for normal config loads, and plugin catalog helpers also default manifest costs, but `estimateUsageCost` still multiplies by `cost.cacheRead` and `cost.cacheWrite` directly. The raw agent `models.json` lookup path builds a cost index without coalescing those fields, so an entry that omits either optional schema field can still make the estimated cost non-finite and suppress downstream OpenTelemetry `costUsd` metrics. The linked external PR from factnest365-ops is an implementation candidate and should be reviewed or ported rather than closing this issue early.

## Best Possible Solution

Keep this issue open. The best fix is to port/review the linked external PR or apply an equivalent patch that coalesces missing `cost.cacheRead` and `cost.cacheWrite` to zero at the estimator or cost-normalization boundary, with a focused regression test in `src/utils/usage-format.test.ts` and, if the telemetry contract should include zero values, a diagnostics-otel regression test for emitted cost metrics.

## Risks / Open Questions

- A narrow fix only in runtime config defaults would miss raw `models.json`, cached pricing, test fixtures, or direct callers of `estimateUsageCost`; the estimator itself should be defensive.
- The existing OpenTelemetry service checks `if (evt.costUsd)`, so a valid zero-dollar estimate is intentionally not recorded as a cost metric; confirm whether the desired metric behavior is positive-cost-only or zero-valued reporting before broadening telemetry output.
