## Summary

Keep open. Current main still does not implement the requested Discord voice-as-I/O bridge: voice auto-join only passes a voice channel id, the voice manager still routes the session through that voice channel, `processSegment()` still calls `agentCommandFromIngress` with `deliver: false`, and the typed config/schema/docs do not expose bridge-to-text fields. The maintainer discussion also explicitly leaves this open for a dedicated Discord voice feature; related #53562/#53741 are narrower session-channel routing work, not a full replacement.

## Best Possible Solution

Keep this open for a dedicated bundled Discord plugin feature. The implementation should add typed config/schema/docs for bridge mode and text-channel target selection, route voice STT into the selected text-channel session, post speaker-attributed transcripts, deliver the assistant reply through normal Discord text delivery, and enqueue the same final reply for voice TTS playback. Coordinate with #53562/#53741 if that narrower `sessionChannelId` work becomes the base, but do not close this broader tracker until the full voice-as-I/O behavior is implemented or a maintainer chooses a canonical replacement.
