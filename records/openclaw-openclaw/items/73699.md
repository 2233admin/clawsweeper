## Summary

Keep open. Current main still treats Discord voice as its own voice-channel-routed agent turn with text delivery disabled, and the maintainer discussion explicitly left this as future Discord voice feature work rather than cleanup.

## What This Changes

The issue requests a Discord voice bridge mode where voice transcripts enter the bound text-channel session, replies are posted to text, and the same replies are spoken back through TTS.

## Best Possible Solution

Implement this in the bundled Discord plugin after maintainer design: typed config/schema/docs, explicit bridge target selection, speaker-attributed transcript posting, routing through the selected text-channel session, normal Discord text delivery, and TTS fan-out of the same final reply. Keep realtime audio provider work separate from this bridge feature.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice joins resolve a route from the voice channel, voice turns call agentCommandFromIngress() with deliver:false, and the strict config/schema/docs expose no bridge-to-text fields.

## Solution Assessment

Unclear. The requested direction belongs in the Discord plugin, but the exact config shape, defaults, text target fallback, and permission/privacy model need maintainer design before an automated implementation is the safest path.
