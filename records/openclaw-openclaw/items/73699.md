## Summary

Keep open. Current main still routes Discord voice through the joined voice-channel session, runs voice turns with text delivery disabled, and exposes no bridge-to-text Discord voice config/schema/docs; the maintainer discussion also explicitly left this as dedicated feature work rather than cleanup that can be closed.

## What This Changes

Request Discord voice channels to act as audio input/output for an existing text-channel agent session, including transcript posting, text delivery, and TTS playback of the same replies.

## Best Possible Solution

Implement this in the bundled Discord plugin with typed config/schema/docs, explicit bridge target selection, speaker-attributed transcript posting, routing through the selected text-channel session, normal Discord text delivery, and TTS fan-out of the same final reply; keep realtime audio provider support as a separate follow-up.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice join resolves the route from the voice channel id, voice turns call agentCommandFromIngress() with deliver:false, and the strict Discord voice config/schema/docs expose no bridge fields.

## Solution Assessment

Unclear. The requested direction belongs in the Discord plugin, but the exact bridge config shape, fallback channel behavior, permission model, transcript visibility, and reply fan-out semantics need maintainer design before a narrow automated fix is safe.
