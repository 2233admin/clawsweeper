## Summary

Keep open. Current main still routes Discord voice through the joined voice channel session and disables text delivery for voice turns, while the Discord voice config, schema, UI hints, and docs still expose no bridge-to-text mode. The discussion also includes a maintainer pass explicitly leaving this as dedicated feature work rather than treating related routing fixes as sufficient.

## What This Changes

Request a Discord voice bridge mode that uses a selected text-channel session for voice transcripts and delivers the same assistant reply to both text and TTS playback.

## Best Possible Solution

Implement this as a dedicated Discord plugin feature with typed config/schema/docs, explicit bridge target selection, speaker-attributed transcript posting, routing through the selected text-channel session, normal Discord text delivery, and queued voice playback of the final reply.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: joined voice sessions resolve routes from the voice channel, voice turns call agentCommandFromIngress() with deliver:false, and strict config/schema paths reject the proposed bridge fields.

## Solution Assessment

Unclear. The requested direction belongs in the Discord plugin, but the exact bridge defaults, target-channel fallback, transcript visibility, permission checks, and interaction with realtime audio need maintainer design before a narrow fix is safe.
