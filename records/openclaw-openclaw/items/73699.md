## Summary

Keep open. Current main still routes Discord voice turns through the joined voice-channel route with text delivery disabled, and the Discord voice config/schema/docs still lack bridge-to-text fields; the issue discussion also has a maintainer pass explicitly leaving this as dedicated feature work.

## What This Changes

Request to make Discord voice act as microphone/speaker I/O for a bound text-channel agent session, including transcript posting, shared context, text delivery, and TTS playback.

## Best Possible Solution

Implement this as a dedicated bundled Discord plugin feature: add typed config/schema/docs for bridge mode and target selection, route STT into the selected text-channel session, post speaker-attributed transcripts, deliver the assistant reply through normal Discord text delivery, and enqueue that same final reply for voice playback.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice joins resolve a route from the voice channel id, voice turns call agentCommandFromIngress() with deliver: false, and strict config/schema paths reject the proposed bridge fields.

## Solution Assessment

Unclear. The requested direction belongs in the Discord plugin, but the exact config, target-channel fallback, permission, and transcript visibility semantics need maintainer design; the narrower sessionChannelId proposal is not a complete solution by itself.
