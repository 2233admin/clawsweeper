## Summary

Keep open. Current main still treats Discord voice as its own routed channel session and suppresses text delivery, while the voice config/schema/docs have no bridge-to-text fields; an explicit maintainer comment also leaves this as dedicated feature work.

## What This Changes

The issue asks for Discord voice STT/TTS to act as I/O for an existing text-channel agent session instead of using an isolated voice-channel session.

## Best Possible Solution

Implement this as a Discord plugin feature with typed config/schema/docs, explicit bridge target selection, speaker-attributed transcript posting, text-channel session routing, normal Discord text delivery, and TTS playback of the same final assistant reply.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: voice joins resolve a route from the voice channel id, voice turns call agentCommandFromIngress() with deliver: false, and strict config/schema paths reject the proposed bridge fields.

## Solution Assessment

Unclear. The requested direction belongs in the Discord plugin, but the exact config shape, target-channel fallback, permission checks, and transcript visibility rules need maintainer design; the narrower sessionChannelId proposal is not a complete solution by itself.
