## Summary

Keep open. Current main still lacks the requested Discord voice-as-I/O bridge: voice auto-join and `/vc join` only pass a voice channel id, the voice manager routes the agent session to that voice-channel peer, `processSegment()` still calls `agentCommandFromIngress` with `deliver: false`, and config/schema/docs expose no bridge-to-text or target text-channel fields. The maintainer discussion explicitly says to leave this open for a dedicated Discord voice feature, and related #53741 is a narrower open PR for `sessionChannelId`, not a complete replacement for transcript posting plus text delivery plus TTS fan-out.

## Best Possible Solution

Keep this issue open as the canonical broader Discord voice-as-I/O feature unless maintainers choose a different tracker. The best path is a dedicated Discord plugin change that adds typed config/schema/docs for bridge mode and target channel selection, routes STT into the selected text-channel session, posts speaker-attributed transcripts, delivers the assistant reply through normal Discord text delivery, and also queues the final reply for voice TTS playback. Review #53741 as possible starting material for the narrower session-channel routing piece, but do not close this issue until the full bridge behavior or a maintainer-approved canonical replacement exists.
