## Summary

Keep open. Current main still lacks the requested Discord voice-as-I/O bridge: voice sessions are routed from the voice channel id, voice turns suppress normal text delivery, and the Discord voice config/docs do not expose a bridge-to-text option. The related #53562/#53741 work is overlapping but narrower and remains open; it is not a complete supersession of this issue’s transcript-posting plus text-and-TTS reply behavior.

## Best Possible Solution

Keep this issue open and implement the bridge in the bundled Discord plugin voice path: add typed/schema/docs config for bridge mode and target text channel, resolve voice input against the chosen text-channel session, post speaker-attributed transcripts, deliver the agent reply through normal Discord text delivery, and enqueue the same final reply for TTS playback. Coordinate with #53562/#53741 if maintainers make `sessionChannelId` the base routing mechanism, but do not close this broader issue until transcript posting and dual text/TTS reply fan-out are covered.
