## Summary

Keep open. Current main has restart-sentinel infrastructure and chat/agent restart paths, but direct `openclaw gateway restart` still routes to `runDaemonRestart` without writing a restart sentinel. The PR is still relevant, though the patch as proposed needs rework for timing and daemon-environment correctness before it should land.

## Best Possible Solution

Keep this PR open or replace it with a maintainer patch that writes the CLI restart sentinel after preflight/no-op decisions but before the actual service restart or SIGUSR1 emission, removes it if the restart is rejected or fails, resolves the sentinel path using the daemon service environment when available, and adds focused daemon lifecycle tests for managed and unmanaged restart paths.

## Risks / Open Questions

- The proposed patch writes the sentinel after the restarted gateway has likely already passed its startup sentinel check, so the notification may be delayed until a later unrelated startup.
- The proposed patch calls `writeRestartSentinel` without the daemon service environment, which can write to the CLI user's state dir instead of the managed gateway's state dir for profile/custom-state installs.
- No supply-chain or secret-handling regression is visible in the provided one-file PR diff, but the lifecycle behavior needs tests for success, no-op, failure cleanup, scheduled restart, and daemon-env state-dir resolution.
