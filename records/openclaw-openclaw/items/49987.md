## Summary

Current main still lacks the narrow CLI restart-sentinel behavior for `openclaw gateway restart`, so this PR is not obsolete. The supplied v2 patch is also not clearly safe to land as-is: it writes the sentinel only after `runServiceRestart` returns, which can miss the restarted gateway's startup sentinel check, and it uses the invoking CLI environment rather than the daemon service environment for the sentinel path.

## Best Possible Solution

Keep this PR open or replace it with a maintainer patch that builds the CLI restart sentinel after config/no-op preflight but before service restart, scheduled restart, or unmanaged SIGUSR1 emission; resolves the sentinel path with the daemon service environment when available; removes the sentinel if restart scheduling/emission fails; and adds focused lifecycle tests for managed, unmanaged, no-op, failure cleanup, scheduled, and custom-state-dir paths.

## Risks / Open Questions

- Landing the current patch as-is can write the sentinel after the restarted gateway has already skipped startup sentinel scheduling, leaving the notification for a later unrelated startup.
- The current patch can write `restart-sentinel.json` into the invoking CLI state directory instead of the installed daemon's state directory when `OPENCLAW_STATE_DIR` or `OPENCLAW_CONFIG_PATH` differs.
- Closing this PR would drop a plausible user-visible CLI gap because current main still lacks the requested CLI restart notification behavior.
