## Summary

Current main already addresses the user-visible Feishu leak: Codex/OpenAI-style raw error payloads are parsed and sanitized before external chat delivery, external channels default to a generic failure message, and the shipped changelog/source in v2026.4.26 contains the Codex `server_error`/`request_id` fix. The reporter’s follow-up source split matches the relevant areas, but those paths now route through shared sanitization and failover handling rather than sending raw JSON to Feishu.

## What This Changes

Sanitize Feishu-visible raw provider `server_error` payloads and keep transient retry behavior bounded to safe pre-output failure paths.

## Best Possible Solution

Keep the shared runner/error-formatting implementation as the source of truth so all external chat channels, including Feishu, receive sanitized generic failures by default while raw provider payloads remain available only in logs/diagnostics.
