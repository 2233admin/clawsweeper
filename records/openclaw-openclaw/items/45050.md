## Summary

Keep this open. Current main still sanitizes the raw JSON shape only into provider-shaped user copy, then preserves and forwards that error payload through Feishu after visible output; the requested Feishu-safe generic fallback and post-output suppression are not implemented.

## What This Changes

The issue requests Feishu-safe handling of transient provider `server_error` failures by sanitizing raw payloads, keeping request IDs in logs, and retrying only before user-visible output.

## Best Possible Solution

Add a focused external-chat error presentation policy so transient/internal provider failures become generic user copy in Feishu, raw provider details and request IDs remain diagnostic-only, and final assistant error payloads after already-visible assistant content are suppressed or genericized while preserving diagnostics and the existing pre-output retry behavior.
