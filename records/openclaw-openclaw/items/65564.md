## Summary

Keep open. Current main has the earlier `sessionFile` carryover fix from #65203, but the remaining same-key isolated heartbeat rotation path still looks live: `heartbeat-runner` overwrites the canonical isolated session entry after `resolveCronSession({ forceNew: true })` and only archives stale nested-key entries, not the previous transcript for the normal rotation. An open fix PR, #73809, already references this issue and targets that remaining behavior.

## What This Changes

The issue reports that `heartbeat.isolatedSession: true` rotates `sessionId` while old heartbeat transcript state is reused or left behind, causing growing context/cost and later orphan cleanup or memory-ingestion side effects.

## Best Possible Solution

Land or revise the narrow #73809-style fix in `src/infra/heartbeat-runner.ts`: before overwriting the canonical isolated heartbeat key, capture the previous same-key entry and archive its transcript with `reason: "reset"` through the existing session archival helper, with regression coverage in the isolated heartbeat stability test. Keep the trajectory doctor false-positive tracked separately by #70680.

## Reproduction Assessment

Yes. The issue has versioned reporter steps and current-main source inspection gives a high-confidence static reproduction path: a forced isolated heartbeat rotation has `previousSessionId`, but the heartbeat archive map only includes stale nested keys before the store entry is overwritten.

## Solution Assessment

Yes. The narrow heartbeat-runner archive-on-rotation direction is the best maintainable fix because #65203 already solved stale `sessionFile` carryover, and the remaining bug is heartbeat-specific lifecycle cleanup rather than a generic session-store migration.
