## Summary

Keep open. Current main still has the Telegram native-quote retry gap: delivery builds `reply_parameters` and has a legacy-reply retry path, but the quote-error classifier only recognizes `quote not found`, so `QUOTE_TEXT_INVALID` is still outside the fallback. No open PR appears to be paired with this issue, and the fix boundary is narrow in the Telegram plugin.

## What This Changes

The issue requests that Telegram delivery treat `QUOTE_TEXT_INVALID` and equivalent quote-text-invalid wording like other rejected native quotes and retry without `reply_parameters.quote`.

## Best Possible Solution

Keep native Telegram quote excerpts, but make the Telegram quote-error classifier recognize both the symbolic Bot API code and human-readable quote-text-invalid variants, then pin it with a delivery regression test and a small user-facing changelog fix entry.
