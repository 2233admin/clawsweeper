## Summary

Keep open. Current main supports the earlier triage point that compaction itself does not appear to flip `ttsAuto` to `always`, but that is not enough to close this report: persisted TTS state, config/local prefs, or the always-available `tts` tool can still produce dashboard audio replies, and the reporter followed up that a full reset fixes the recurring symptom.

## Best Possible Solution

Keep the issue open and narrow the investigation. Capture the affected session entry (`ttsAuto`), `~/.openclaw/settings/tts.json`, `messages.tts` config, `/tts status`, and a transcript slice before and after compaction. If dashboard text chats should never synthesize audio unless explicitly requested, add a focused regression around dashboard `chat.send` plus compaction and enforce that policy via a dashboard/session TTS gate or a stricter default prompt/tool contract for the `tts` tool.
