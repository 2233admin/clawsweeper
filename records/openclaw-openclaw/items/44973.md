## Summary

Keep this PR open. Current main still has the same main-session cron duplication path the PR is trying to fix: the cron timer enqueues the raw systemEvent with a plain `cron:<jobId>` context key, heartbeat preflight peeks that pending event, `resolveHeartbeatRunPrompt` wraps cron-tagged pending events in the scheduled-reminder prompt, and the reply path can still drain the same event into `System:` lines. The current tests cover heartbeat noise and ordinary cron wrappers, but not the PR’s exact `sessionTarget="main"`, `wakeMode="now"`, `payload.kind="systemEvent"`, `delivery.mode="none"` dedupe case.

## Best Possible Solution

Review and land this PR, or apply an equivalent main-branch fix that prevents main-session cron `payload.kind="systemEvent"` jobs from being wrapped a second time by heartbeat prompt construction. Keep regression coverage for `sessionTarget="main"`, `wakeMode="now"`, and internal/no-delivery cron behavior.
