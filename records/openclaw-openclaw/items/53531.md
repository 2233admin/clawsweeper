## Summary

Keep open. Current main still resolves compaction to one effective auth profile and returns or throws on that profile's auth/compaction failure, while the normal turn path has ordered same-provider auth candidates, cooldown skipping, and profile advancement. The idempotency concern from the discussion remains a real design constraint because successful compaction also writes hooks, checkpoints, transcript rotation, and session side effects.

## What This Changes

The issue requests same-provider auth-profile rotation and failover for compaction when the selected profile is rate-limited or otherwise unavailable.

## Best Possible Solution

Add shared compaction retry orchestration that snapshots the transcript/checkpoint input once, orders eligible same-provider auth profiles with the existing auth semantics, retries only failover-classified failures, updates cooldown state consistently, and commits one successful summary plus side effects at most once across direct and context-engine compaction paths.
