## Summary

Keep this issue open. Current main still does not give compaction the normal same-provider auth-profile retry loop. The normal agent run path builds ordered profile candidates, skips/probes cooldowns, marks failures, and advances profiles; compaction paths instead pass a single runtime authProfileId or perform one compaction call and surface failure. The discussion’s idempotency requirement is also still a real design constraint, not implemented state.

## Best Possible Solution

Keep this issue open. The best fix is to share the normal same-provider auth-profile ordering, cooldown skip/probe, failure classification, and failure marking with compaction; capture the transcript once; retry eligible same-provider profiles against that snapshot; and commit exactly one successful compaction result. Add focused coverage for manual /compact, legacy context-engine delegation, timeout/overflow recovery, CLI budget compaction, and the idempotency case raised in discussion.

## Risks / Open Questions

- A naive retry loop could double-compact or apply a summary to a transcript that changed after the first failed profile; the fix needs snapshot plus at-most-one commit semantics.
- Manual /compact, queued/context-engine compaction, timeout/overflow recovery, CLI budget compaction, and plugin-owned compaction enter through different paths, so a narrow patch could leave some surfaces single-profile.
