## Summary

Keep open. Current main still has the reported schema-to-runner mismatch, and open PR #73483 uses closing syntax for this issue while #74920 overlaps the same send path, so the issue should remain open until one implementation lands or is explicitly superseded.

## What This Changes

The issue requests preserving the message tool `asVoice` flag through shared send handling so audio reaches downstream delivery as native voice media where supported.

## Best Possible Solution

Land one narrow shared pass-through that maps message-tool and CLI `asVoice` to downstream `audioAsVoice` through the runner, core send service, direct/gateway send contracts, generated protocol clients if needed, docs, and focused regression tests, while leaving plugin-owned transcoding behavior in channel plugins.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: `asVoice: true` is accepted by the shared message tool schema, but the inspected shared runner, core send service, direct send path, and gateway send path have no field that can carry it to delivery.

## Solution Assessment

Yes. The best direction is the narrow shared pass-through, because lower-level outbound delivery and Feishu media handling already honor `audioAsVoice` once it reaches them.
