## Summary

Keep open. The report is recent, detailed, and still matches current main: the bundled claude-cli backend uses live stdio plus bundle MCP, native file/exec tools are intentionally excluded from OpenClaw's loopback MCP tool set, no-output watchdog behavior can produce the reported minute-scale stall shape, and Telegram/external-channel failure paths still fall back to generic user-visible text. I found no current-main source change after the report that resolves the Claude native Read stall or improves the Telegram diagnostic for this path.

## What This Changes

The issue reports a Claude CLI backend Telegram turn that stalls for roughly three minutes during a native Claude Read of an out-of-workspace wrapper path and only returns a generic Telegram failure instead of a clear tool or permission error.

## Best Possible Solution

Keep the issue open until maintainers define the supported workflow, then close the diagnostic gap without loosening boundaries: claude-cli native-tool stalls should fail fast with sanitized, actionable diagnostics; Telegram should surface a bounded timeout/tool-context message where safe; and wrapper execution should be routed or documented through the intended ACP or node-exec path with host approvals preserved.
