## Summary

Keep this issue open. Current main still has the reported shape: the bundled claude-cli backend uses bundle MCP, Claude live stdio turns can fail on the multi-minute no-output watchdog, and external chat failures still collapse to generic Telegram copy when verbose detail is off; I found no source, test, docs, or history evidence that the wrapper-path Read stall is fixed or superseded. The related #72338 report is useful context, but it remains broader and open rather than a canonical replacement for this specific claude-cli plus Telegram failure.

## What This Changes

The issue reports a regression where Telegram routed through claude-cli stalls during a wrapper-path file Read attempt and returns only a generic error instead of running the approved wrapper or failing fast with a clear tool error.

## Best Possible Solution

Keep the issue open and investigate it as a supported-boundary problem: claude-cli bundle-MCP or native-tool stalls should fail fast with useful diagnostics, and the supported Telegram-to-wrapper execution route should be clarified or enforced so approved wrapper execution goes through the intended OpenClaw exec/node-host policy rather than an ambiguous Claude native file Read path.
