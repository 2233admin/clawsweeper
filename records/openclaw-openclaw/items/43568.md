## Summary

Keep open. Current main still supports process-level proxy routing and model-provider request proxy overrides, but it does not expose auth-profile-scoped proxy metadata or a resolved-auth transport path. The prior ClawSweeper comment on this issue correctly identified the remaining gap, and the current main SHA still matches that assessment.

## What This Changes

Requests an optional per-auth-profile proxy setting so provider requests using a selected credential profile can be routed through that profile's dedicated proxy.

## Best Possible Solution

Define a first-class auth-profile transport policy that is secret-aware, validates proxy URLs, documents precedence, preserves redaction and copy semantics, and threads the selected profile's resolved policy into provider request construction without weakening existing process-level and provider-request proxy behavior.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. Static verification on current main is high confidence because the credential types, config schema, resolved-auth type, and runtime-auth override path do not carry profile-scoped proxy policy.

## Solution Assessment

No, adding a raw proxyUrl directly to credential records is not yet the best complete solution. The safer path is a designed profile transport policy that composes with SecretRef/redaction and existing egress precedence before implementation.
