## Summary

Keep open. Current main still has the reported Discord component parsing failure path: any non-empty component object is parsed, `modal.fields: []` still throws, and empty block-only specs can still reach the component builder and fail instead of falling back to plain text. The PR addresses a real gap, but its current diff also has unresolved review concerns around silently swallowing malformed `modal.fields` values and dropping container-only component metadata, so the best path is revision rather than closure.

## Best Possible Solution

Revise this PR or replace it with a narrow Discord plugin patch: treat empty LLM-default component specs as absent so sends fall through to plain text, skip empty block arrays, tolerate empty modal fields only as an empty-default case, keep validation errors for malformed `modal.fields`, preserve container-only specs when fallback content supplies the message text, and add regression tests for the send/action and outbound payload paths.
