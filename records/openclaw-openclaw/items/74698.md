## Summary

Keep open. Current main still has a separate Telegram native-command authorization path that can deny group `/status` when no effective command/group sender authorizer is configured, and the provided discussion shows open PR #74894 as the explicit implementation candidate for this issue.

## What This Changes

The issue reports that Telegram group `/status` slash commands no longer reply in configured group sessions on OpenClaw 2026.4.27 even though normal group messages still route to the agent.

## Best Possible Solution

Land PR #74894 or an equivalent focused Telegram fix so group `/status` replies in the originating group/topic when the group is otherwise allowed, while preserving fail-closed behavior for disabled groups, disallowed chats, and explicitly unauthorized senders.

## Reproduction Assessment

Yes for a code-level reproduction path: with `commands.useAccessGroups` at its default and no effective `commands.allowFrom`, `allowFrom`, `groupAllowFrom`, or owner authorizer, a configured group can pass normal message routing while native `/status` fails the command authorization gate. The exact reporter config is still unknown.

## Solution Assessment

Yes, keeping this issue open for the linked PR review is the right path. Closing now would race an unmerged implementation candidate, and current main does not provide source/test/release proof that the reported behavior is resolved.
