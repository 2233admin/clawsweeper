## Summary

Keep PR #58685 open. Current main still has the raw `channels list` usage-error behavior the PR is trying to improve, so it is not implemented or obsolete. The PR also should not be cleanup-closed as a duplicate: the nearby #59472 report tracks JSON output robustness, while this PR targets human-readable warning UX. However, the existing review comments identify a real gap in the PR approach: current Claude `user:profile` failures normally return as usage snapshot errors and are then formatted as raw `Claude: HTTP 403 ... user:profile` lines, so a catch-only warning path would miss the common case.

## Best Possible Solution

Keep the PR open but require a revision that handles both thrown usage-load failures and provider error snapshots before `formatUsageReportLines` emits raw Claude scope output. The focused implementation likely belongs in the channels list usage presentation layer or a shared usage-warning formatter, with tests for the non-throwing Claude `user:profile` snapshot path, generic provider errors, `--no-usage`, and JSON-mode behavior kept separate from #59472.
