## Summary

Keep open. Current main still does not expose per-conversation identity to provider plugin `createStreamFn` contexts, and open PR #73492 already references this issue with closing syntax as the implementation candidate.

## What This Changes

The issue requests adding optional `sessionId` and `sessionKey` fields to `ProviderCreateStreamFnContext` and forwarding them through provider stream registration paths.

## Best Possible Solution

Land a reviewed version of #73492, or an equivalent narrow core change, that adds optional provider context identity fields, forwards them only from conversation-bearing paths, leaves one-off completion paths unset, and adds focused contract coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main has `sessionId` and `sessionKey` available in runner-side contexts, but `ProviderCreateStreamFnContext` and `registerProviderStreamForModel` cannot deliver them to provider plugins.

## Solution Assessment

Yes. Optional `sessionId` and `sessionKey` fields on the existing provider stream context are the narrowest backward-compatible API fix; provider plugins can use `sessionKey` when present without breaking plugins that ignore it.
