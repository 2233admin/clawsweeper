## Summary

Keep open. Current main still does not expose `sessionId` or `sessionKey` in `ProviderCreateStreamFnContext` or in the `registerProviderStreamForModel` context object, and this issue is already paired with open implementation PR #73492, so it should remain open until that PR is reviewed and merged or rejected.

## What This Changes

The issue requests adding optional per-conversation `sessionId` and `sessionKey` fields to provider plugin `createStreamFn` context so provider plugins can isolate concurrent conversations for the same agent.

## Best Possible Solution

Land a reviewed version of #73492 or an equivalent narrow, backward-compatible provider plugin API change that adds optional `sessionId` and `sessionKey` to the create-stream context, forwards them from conversation-bearing paths, leaves one-off completions unset, and adds focused propagation coverage.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main exposes no `sessionId` or `sessionKey` to provider `createStreamFn`, while the embedded runner and related conversation paths already carry those values nearby.

## Solution Assessment

Yes. Optional fields forwarded through `registerProviderStreamForModel` are the narrowest maintainable fix because existing plugins remain compatible and affected provider plugins no longer need brittle inference from prompts, files, or agent directories.
