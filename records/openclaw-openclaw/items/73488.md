## Summary

Keep open. Current main still does not expose per-conversation identity to provider plugin `createStreamFn`, and open PR #73492 already targets this exact gap with closing syntax, so this issue should stay open until that implementation is reviewed and merged or rejected.

## What This Changes

Request to add optional `sessionId` and `sessionKey` fields to `ProviderCreateStreamFnContext` and plumb them from conversation-bearing provider stream call sites.

## Best Possible Solution

Land a reviewed version of #73492 or an equivalent narrow change that adds optional `sessionId` and `sessionKey` to the provider stream context, forwards them from conversation-bearing paths, leaves one-off completions unset, and adds focused propagation coverage for the provider plugin API.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a provider plugin using `createStreamFn` cannot receive `sessionId` or `sessionKey` because the current context type and registration object omit those fields, while the main runner already has them in scope.

## Solution Assessment

Yes in direction. Optional context fields passed through `registerProviderStreamForModel` are the narrowest backward-compatible fix and avoid forcing provider plugins to infer conversation identity from prompts, filesystem state, or first-message hashes.
