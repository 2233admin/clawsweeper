## Summary

Keep open. Current main still has the user-visible reminder note path with no `messages.unscheduledReminderNote` opt-out, so the PR is addressing a real unresolved behavior, but the branch is not merge-ready because it reads an untyped public config key and leaves generated/help config metadata out of sync. The related open PR #39330 proposes a different UX direction, so maintainers should choose the product/API path before landing one complete fix.

## What This Changes

This PR adds a `messages.unscheduledReminderNote` boolean gate around the unscheduled-reminder note append path, adds the key to the Zod session schema and field labels, and adds focused reminder-guard unit tests.

## Best Possible Solution

Settle the reminder-guard UX, then land one complete implementation: either a fully typed, documented, generated, and tested `messages.unscheduledReminderNote` config option, or the competing delivery-strip/system-annotation path that prevents this internal note from appearing as agent-authored user text.

## Reproduction Assessment

Yes. Current main has a focused source and test path: a reply payload like `I'll remember that` or `I'll remind you tomorrow morning` with `successfulCronAdds: 0` and no same-session active cron reaches `appendUnscheduledReminderNote`, and existing tests assert the appended note behavior.

## Solution Assessment

Unclear. A default-on config opt-out can address the reported pain, but this branch has not completed the public config contract and maintainers should choose it deliberately over #39330's narrower delivery-stripping approach.
