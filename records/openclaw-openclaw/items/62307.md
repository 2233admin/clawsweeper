## Summary

Keep open. Current main still lacks WhatsApp message edit handling, so the feature remains useful, but the proposed implementation is not merge-ready because it misclassifies edits that arrive while agent processing is already in flight and should be rebased onto the current WhatsApp monitor/dedupe code.

## What This Changes

The PR adds WhatsApp `messages.update` handling in `extensions/whatsapp/src/inbound/monitor.ts` to swap edited text while buffered, notify after successful processing, and ignore revocations.

## Best Possible Solution

Rebase the feature onto the current WhatsApp monitor, add a typed `messages.update` listener in the WhatsApp plugin, key edit tracking by the same account/chat/message identity used by inbound dedupe, split buffered versus in-flight versus processed state, and cover the edit timing and revocation cases with focused monitor tests.

## Reproduction Assessment

Yes. A focused unit test can emit a normal WhatsApp upsert, let the debounce flush call `onMessage` with an unresolved promise, then emit `messages.update` for the same ID and observe that the PR still treats it as buffered even though the agent payload is already fixed.

## Solution Assessment

No. Listening to Baileys `messages.update` in the WhatsApp plugin is the right direction, but the bookkeeping should move messages out of the buffered map before `onMessage` starts or use a separate in-flight state so mid-processing edits follow the late-edit path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Separate buffered edits from in-flight deliveries:** `extensions/whatsapp/src/inbound/monitor.ts:167`
  - body: `pendingMessageIds` is only cleared after `await options.onMessage(messageToDeliver)`, so an edit received while the agent is already processing still takes the buffered branch. For batched deliveries the body was already computed, so mutating the pending entry cannot change what the agent sees, and no late-edit notice is sent. Move the entry out of the buffered map before calling `onMessage` or track an explicit in-flight state.
  - confidence: 0.91
