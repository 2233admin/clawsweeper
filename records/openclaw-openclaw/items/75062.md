## Summary

Keep open. Current main still has the reported uninstall data-loss path, and this PR is useful source work, but the proposed guard is not merge-ready because it preserves a nested workspace by skipping the whole state directory, leaving the selected state/config cleanup undone in the default layout.

## What This Changes

The PR adds a workspace preservation option to state cleanup, passes workspace directories from uninstall when the workspace scope is unselected, and adds a regression test for the skipped state-dir dry run path.

## Best Possible Solution

Revise the PR so selecting State + config still removes state/config data that can be safely removed while preserving unselected workspace directories, with a CLI-level regression for the default `~/.openclaw/workspace` layout and clear user messaging before destructive cleanup.

## Reproduction Assessment

Yes. Current main can be reproduced by the default workspace layout under `~/.openclaw/workspace`: selecting state without workspace reaches the unconditional state-dir removal path before workspace handling.

## Solution Assessment

No. The PR prevents workspace deletion, but skipping the entire state directory is not the narrowest maintainable fix because it leaves the selected state/config cleanup behind.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep removing state data around preserved workspaces:** `src/commands/cleanup-utils.ts:116-122`
  - body: When a workspace is inside the state dir, this branch skips the whole state-dir removal. In the default layout that means `--state` no longer removes state/config at all, because the later config/oauth removals are also skipped when they are inside state. Preserve the workspace, but still remove the selected state/config data or block the operation before confirmation.
  - confidence: 0.86
