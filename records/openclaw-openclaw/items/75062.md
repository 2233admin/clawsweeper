## Summary

Keep this PR open, but do not merge it as written. It addresses the paired data-loss bug, and the revised prune-around-workspaces approach is directionally right, but the patch bypasses the existing unsafe-root guard when a preserved workspace is inside an overridden state directory.

## What This Changes

The PR adds workspace-preservation handling to uninstall state cleanup, passes unselected workspace directories into `removeStateAndLinkedPaths`, prunes state contents around preserved workspaces, and adds a filesystem regression test.

## Best Possible Solution

Keep the focused workspace-preservation fix, but make the prune-around-workspaces path reuse the same unsafe-target validation as full state removal before it enumerates or deletes children, then add regression coverage for an unsafe state-dir override and the required changelog entry.

## Reproduction Assessment

Yes. The underlying issue reproduces from source on current main because `openclaw uninstall --state` removes the state directory recursively while the default workspace lives under `~/.openclaw`; the PR regression can be reproduced source-level by setting the state directory to an unsafe ancestor such as `$HOME` and observing that the new pruning path skips the existing refusal check.

## Solution Assessment

No, not as written. Pruning around preserved workspaces is the right narrow approach, but it must preserve the state-root safety guard before deleting child entries.
