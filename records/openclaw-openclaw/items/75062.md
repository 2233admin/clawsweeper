## Summary

Keep this PR open. The underlying data-loss bug is real on current main, and the revised approach targets the right cleanup semantics, but the branch still appears to introduce a typecheck blocker by referring to `fs.Dirent` from the `node:fs/promises` default import.

## What This Changes

The PR adds a workspace-preservation option to uninstall state cleanup, prunes state directory contents around unselected workspaces, passes workspace directories when workspace scope is not selected, and adds a regression test.

## Best Possible Solution

Keep the selective pruning direction, fix the Dirent typing by importing the type from node:fs or avoiding the explicit annotation, then validate that State + config removes config/auth/state data while preserving unchecked workspaces under state.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: the default workspace is inside ~/.openclaw, uninstall treats State + config and Workspace as separate scopes, and state cleanup recursively removes the whole state directory.

## Solution Assessment

No as submitted. The revised algorithm is a maintainable direction, but the branch should not merge until the type issue is fixed and the cleanup-utils/uninstall tests plus changed gate pass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

- **[P2] Import Dirent from node:fs:** `src/commands/cleanup-utils.ts:145`
  - body: This branch annotates `entries` as `fs.Dirent[]`, but `fs` here is the default import from `node:fs/promises`; that module's public types use `Dirent` from `node:fs` in signatures and do not expose it as `fs.Dirent`. Core typecheck will fail before the regression test can run, so import `type { Dirent }` from `node:fs` or infer the readdir result.
