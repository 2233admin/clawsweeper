## Summary

Keep open. The report is new, concrete, and matches a real provider/transport surface: current main still routes `openai-codex/gpt-5.5` through the OpenAI Codex Responses/PI transport, with large model metadata caps, and I did not find a current-main or shipped-release fix for the reported latency regression after v2026.4.26. Existing raw-model-run code and tests show the gateway path should suppress context, tools, and bootstrap, so the remaining delay needs targeted provider/transport timing investigation rather than cleanup closure.

## Best Possible Solution

Keep this issue open and have a maintainer reproduce on current main with request-payload and timing instrumentation for local infer, gateway infer, and agent paths. The investigation should confirm whether `max_output_tokens=128000`, context cap/model metadata, prompt-cache/session setup, PI 0.70.x behavior, or the OpenAI Codex Responses transport changed latency after 2026.4.24, then fix the narrow owner module rather than adding a core migration or broad provider workaround.
