## Summary

Keep this PR open. The reported crash path is real on current main, but the proposed writable guard only handles a non-writable stdin stub; a Linux shell-wrapped missing binary can start with stdin writable and then emit an asynchronous EPIPE after the initial write/end path.

## What This Changes

The PR changes `createChildAdapter` to skip the initial stdin write/end when `child.stdin.writable` is false and adds a unit test for a non-writable stdin stub.

## Best Possible Solution

Update `src/process/supervisor/adapters/child.ts` so the initial stdin write/end path is protected against both already-non-writable stdin and asynchronous stdin `error`/EPIPE after early child exit, while preserving normal writable stdin delivery. Add focused regression coverage in `src/process/supervisor/adapters/child.test.ts` for the non-writable stub and the initially-writable-then-EPIPE shape.

## Reproduction Assessment

Yes. The PR body gives a plausible Google Gemini CLI missing-binary route, and a focused Node probe against the same Linux shell-wrapper shape reproduces `stdin.writable === true` followed by an EPIPE error on current main.

## Solution Assessment

No. The proposed guard is too narrow because it relies on `writable` being false before the write; the maintainable fix should also attach or otherwise handle stdin error events around the initial write/end path.

Please guard the initial input write/end against stdin `error` events as well as the already-non-writable case.
