## Summary

Current main still contains the reported Dashboard/CLI divergence path. The gateway can hydrate update availability from `update-check.json` and return during the throttle interval before live npm resolution, while `openclaw update status` performs a registry-backed check and can report a newer version independently of the gateway cache.

## Best Possible Solution

Keep this open and fix the gateway update-check owner path in `src/infra/update-startup.ts`: align Dashboard availability with the effective registry result used by `openclaw update status` when persisted state is stale or empty, avoid treating unresolved npm channel lookup as a successful full-interval check, schedule a short retry/backoff for empty registry results, and add regression coverage in `src/infra/update-startup.test.ts` for stale-to-current and fresh-empty cache cases.

## Evidence

- **Gateway trusts persisted availability before live lookup:** `runGatewayUpdateCheck` reads `update-check.json`, derives `updateAvailableCache` from `lastAvailableVersion`, then returns inside `resolveCheckIntervalMs` before calling `checkUpdateStatus` or `resolveNpmChannelTag`. If the persisted version is missing or not newer than `VERSION`, the cache becomes null and the Dashboard has no update banner during the interval.

A fresh-but-empty state can therefore suppress the banner until the normal interval expires.
