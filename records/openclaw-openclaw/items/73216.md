## Summary

Keep open. Current main still has the static GitHub Copilot catalog/default-limit behavior, so the PR is not obsolete, and the related #72824/#72825 context supports the dynamic-discovery direction. The PR is not merge-ready as written because the shared fetch path drops agent-scoped auth, caches one account's catalog globally, does not feed cached API models back into dynamic model resolution, and regresses known `xhigh` fallback behavior when discovery fails.

## What This Changes

The PR adds GitHub Copilot `/models` API discovery with mapping, caching, catalog augmentation, dynamic-model warm-up, thinking-profile derivation, and related unit tests plus small plugin-SDK/test cleanups.

## Best Possible Solution

Land a refined GitHub Copilot plugin implementation that fetches authenticated model capabilities through provider-owned hooks, passes agent-scoped auth context consistently, keys or invalidates cached catalogs by credential/baseUrl/agent context, uses fetched capabilities in both catalog and dynamic model resolution, preserves conservative offline fallbacks for known models, and keeps focused tests for auth-profile-only, multi-profile switching, fallback thinking levels, and dynamic resolver cache use.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main has an empty Copilot catalog and static 128K/8K defaults, while PR head drops `agentDir`, shares one global cache, and does not feed the fetched cache into `resolveDynamicModel`. The failures can be locked down with mocked agent-local auth profiles and two-profile catalog tests.

## Solution Assessment

No, not as written. Dynamic Copilot `/models` discovery is the right product direction, but the current patch needs the auth context, cache scope, dynamic-resolution, and fallback-thinking fixes before it is the narrowest maintainable solution.

Full review comments:

- **[P2] Pass agentDir through model discovery:** `extensions/github-copilot/index.ts:54-57`
  - body: `fetchAndCacheModels` resolves the GitHub token without `ctx.agentDir`, even though the catalog and augmentation hooks receive an agent-scoped auth store. An agent with only a local Copilot token profile and no env token can pass the catalog preflight, then the shared fetch reopens the default store and returns no models, leaving `/models list github-copilot` empty.
  - confidence: 0.9
- **[P2] Scope the model cache to credentials:** `extensions/github-copilot/index.ts:44-45`
  - body: The first successful `/models` fetch fills a process-wide map and every later call returns it before checking agentDir, env token, auth profile, config, or Copilot baseUrl. In a multi-agent gateway or after profile switching, model lists and thinking capabilities can come from another Copilot account's plan.
  - confidence: 0.86
- **[P2] Return fetched models from dynamic resolution:** `extensions/github-copilot/index.ts:429`
  - body: `prepareDynamicModel` fetches the Copilot catalog, but the retry path still calls `resolveCopilotForwardCompatModel`, which never reads `cachedModelCapabilities`. If catalog discovery did not already put the model in the registry, first use still resolves to the synthetic 128K/8K fallback instead of the fetched API limits.
  - confidence: 0.84
- **[P2] Keep xhigh fallback for known models:** `extensions/github-copilot/index.ts:463-464`
  - body: When `/models` discovery fails, times out, or has not populated the cache, this fallback now returns only the generic levels without `xhigh`. Current main exposes `xhigh` for known Copilot GPT/Codex model ids, so `/think xhigh` regresses for those models on offline or failure paths.
  - confidence: 0.86
