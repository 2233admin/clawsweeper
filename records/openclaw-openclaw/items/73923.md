## Summary

Keep this PR open. Current main and v2026.4.27 still clear the Control UI Gateway Token on harmless non-identical WebSocket URL edits, while this PR targets that bug with a normalized-scope resolver and focused regression coverage. The patch appears correct, but the live PR is dirty against main and needs maintainer review/rebase rather than cleanup closure.

## What This Changes

The PR adds a Control UI storage helper for resolving Gateway Access tokens during WebSocket URL edits, wires Overview to use it, expands node/browser UI tests, and adds a changelog fix entry.

## Best Possible Solution

Land this PR or an equivalent scoped Control UI fix as the canonical resolution for #41545: preserve tokens for same-normalized endpoint edits, reload or clear scoped tokens when the endpoint changes, keep the regression coverage and changelog credit, then treat the older overlapping PRs as superseded.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction path: enter a Gateway Token in Overview > Gateway Access and edit the WebSocket URL, including adding a trailing slash; the handler writes an empty token for any non-identical trimmed value.

## Solution Assessment

Yes. The PR's normalized-scope helper is narrower and safer than always preserving the token because it avoids carrying a gateway-scoped credential to a different normalized endpoint while preserving harmless same-endpoint edits.
