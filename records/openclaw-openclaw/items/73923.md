## Summary

Keep this PR open. Current main and the latest release still clear the Control UI Gateway Token for any WebSocket URL edit that is not the exact same trimmed string, while this PR adds the missing normalized-scope resolver and targeted regression coverage. The patch is narrow and appears correct, but it is open, dirty, and needs maintainer review/rebase rather than cleanup closure.

## What This Changes

The PR adds a Control UI storage helper for resolving Gateway Access tokens during WebSocket URL edits, wires Overview to it, expands node/browser UI tests, and adds a changelog fix entry.

## Best Possible Solution

Land this PR or an equivalent scoped Control UI fix as the canonical resolution for #41545: preserve tokens for same-normalized endpoint edits, reload or clear scoped tokens when the endpoint changes, keep the regression coverage and changelog credit, then supersede the older duplicate PRs.

## Reproduction Assessment

Yes. Current-main source provides a high-confidence reproduction path: enter a Gateway Token in Overview > Gateway Access and edit the WebSocket URL, including a harmless trailing slash edit; the handler writes an empty token for any non-identical trimmed value.

## Solution Assessment

Yes. The normalized-scope helper is a safer fix than always preserving the token because it avoids carrying a gateway-scoped credential to a different normalized endpoint while still preserving harmless same-endpoint edits.
