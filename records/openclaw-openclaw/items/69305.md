## Summary

Keep open. Current main still omits positional argument specs from zsh leaf-command completions, and this PR is the active canonical fix path for the user-reported wiki ingest completion bug after the duplicate PRs were closed. The patch is scoped and broadly correct, with a low-priority zsh escaping hardening note and a mergeability/rebase check left for maintainers.

## What This Changes

This PR adds zsh positional argument generation for Commander leaf commands, maps filesystem-like positionals to `_files` or `_files -/`, adds completion regression tests, and adds a changelog entry.

## Best Possible Solution

Land this PR or an equivalent maintainer-applied patch after resolving mergeability and deciding whether to share the zsh label escaping fix with #64490. Preserve the contributor credit and keep the implementation in the generic CLI completion generator rather than special-casing `wiki ingest`.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level and deterministic: create a leaf Commander command with `.argument("<path>")`, call `getCompletionScript("zsh", program)`, and observe that current main emits only option specs while the PR tests assert the expected positional `_files` spec.

## Solution Assessment

Yes, mostly. Reading Commander `registeredArguments` only for leaf command functions is the narrow maintainable fix and the zsh optional/variadic forms match upstream docs; the safest follow-through is to resolve the low-priority escaping note and rebase before merge.

## Review Findings

Full review comments:

- **[P3] Escape shell interpolation in zsh positional labels:** `src/cli/completion-cli.ts:225-234`
  - body: `escapeZshLabel` leaves `$`, backticks, and command substitution unescaped even though these specs are emitted inside double-quoted zsh function bodies. That repeats the #64490 class for newly emitted positional labels, so a positional description containing shell interpolation would be evaluated or displayed incorrectly during completion.
