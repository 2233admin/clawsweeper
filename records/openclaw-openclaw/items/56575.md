## Summary

Keep this PR open. Current main still does not provide the requested explicit time-based maintenance API or gateway sweep, but the PR branch has blocking correctness issues confirmed against the current head: the Moonshot cache-TTL test expectations conflict with the mock, reset mode is still coupled to prompt-cache provider eligibility, and expiry can prefer a stale cache-touch timestamp over newer assistant activity.

## What This Changes

The PR adds model-level time-based session maintenance params, session activity/cache timestamps, gateway maintenance sweep logic, defaults, and tests for compacting or resetting idle sessions.

## Best Possible Solution

Finish the feature as a maintainer-approved session lifecycle policy on top of the current cache-pruning, provider-eligibility, session-reset, and config-schema code. The branch should fix the confirmed correctness bugs, add matching docs/schema coverage, and explicitly decide how it relates to #62475 before merge.

  - body: `resolveCacheTtlMs` returns `null` before reading `timeBasedContextCompact` or `sessionCacheTtl` when the provider is not prompt-cache eligible, and the sweep skips `null` TTLs. That makes configured `reset` mode a no-op for non-cache providers even though reset only needs idle timestamps.
  - confidence: 0.91
- **[P2] Use the newest activity timestamp for expiry:** `src/gateway/session-cache-maintenance.ts:53-57`
  - body: `resolveCacheTouchAt` always prefers `lastCacheTouchAt` over `lastAssistantMessageAt`. If maintenance was disabled while assistant turns continued and is later re-enabled, a stale cache-touch timestamp can make a recently active session reset or compact immediately.
  - confidence: 0.84
