## Summary

Current main still does not resolve iMessage inline reply/thread metadata when imsg omits resolved reply_to_* fields. The active legacy iMessage plugin forwards reply context only when reply_to_text is already present, and there is no thread_originator_guid/reply_to_guid modeling or chat.db enrichment before dispatch. Open PR #42683 remains the closest implementation candidate, but it is open, unmerged, and appears to need porting to the active extensions/imessage path.

## Best Possible Solution

Keep this issue open until the bundled legacy iMessage plugin resolves imsg/chat.db reply-origin metadata into the existing inbound ReplyToId, ReplyToBody, and ReplyToSender fields. The likely path is in extensions/imessage/src/monitor/*: enrich missing reply context before inboundDebouncer.enqueue, prefer reliable origin GUID metadata, use read-only local chat.db lookup only when appropriate, fail closed on permission/schema/locking errors, skip unsafe remote assumptions, and add regression tests for GUID-only inline replies. Review, port, or supersede PR #42683 rather than queueing a parallel fix.
