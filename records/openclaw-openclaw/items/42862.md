## Summary

Keep open. Current main still treats a bare image URL typed into the Control UI chat as normal text/link content, while image rendering is wired through explicit attachments, structured image blocks, MEDIA/directive paths, and managed assistant media. The prior ClawSweeper comment reached the same conclusion, and the current 7a23b2d9 main checkout still matches it.

## Best Possible Solution

Keep this open for a deliberate Control UI/Gateway feature: detect eligible bare image URLs in user chat text, ingest or proxy them through same-origin managed media instead of hotlinking, preserve text/link fallback, and add focused coverage for composer/send, `chat.send` history projection, grouped rendering, Markdown sanitization, managed media, and CSP behavior.

## Risks / Open Questions

- A direct regex-to-`<img>` implementation would conflict with the current Control UI CSP and privacy model by allowing arbitrary browser image fetch attempts from chat text.
- A safe implementation likely needs Gateway-managed fetching or media ingestion with SSRF/public-network guards, MIME and size validation, cache/error states, and history/rendering tests.
