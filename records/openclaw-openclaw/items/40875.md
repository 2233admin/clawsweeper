## Summary

Keep this PR open. Current main still lacks heartbeat-specific model fallback guidance, so the docs problem is real, but the submitted patch is stale against the current `ParamField` layout and its strongest cross-provider warning is inaccurate because the runtime can still try the configured primary after a different-provider heartbeat override fails.

## Best Possible Solution

Request a narrow PR revision: add an accurate fallback note inside the current `heartbeat.model` `ParamField`, align it with `docs/concepts/model-failover.md`, state that no override and same-provider overrides follow the applicable fallback chain, state that different-provider overrides skip unrelated configured fallbacks unless the override is already in the configured fallback chain, and mention that the configured primary can still be tried when no explicit fallback override is supplied. Keep #23718 separate as the broader feature request for structured heartbeat fallback configuration.
