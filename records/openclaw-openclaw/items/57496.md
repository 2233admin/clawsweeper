## Summary

Keep open. Current main still lacks the requested Gemini web_search fallback to `models.providers.google.*`, so the PR is not obsolete, but the current PR head still has a blocking credential-order defect in the auto-detection/runtime secret path and overlaps the separate baseUrl work in #61972.

## What This Changes

The PR changes the Google plugin so Gemini web_search can reuse `models.providers.google.apiKey` and `models.providers.google.baseUrl`, adds `webSearch.baseUrl` manifest/docs metadata, and adds Gemini web-search regression tests.

## Best Possible Solution

Land one rebased Google plugin change that keeps the current lazy runtime/shared-file split, adds Gemini web_search API-key and baseUrl fallback, and preserves a single ordered credential contract across direct execution, auto-detection, SecretRef resolution, docs, manifest metadata, and tests. Reconcile the baseUrl pieces with #61972 so the final implementation has one owner.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: on the PR head, configure both `GEMINI_API_KEY` and `models.providers.google.apiKey` with no plugin-scoped key, then the configured-credential lookup returns the provider key and the runtime resolver treats that config value as higher priority than env.

## Solution Assessment

No, not as currently patched. The product direction is reasonable, but the PR needs to preserve `GEMINI_API_KEY` precedence through the runtime auto-detect path and be rebased onto the current main runtime split before it is the narrow maintainable fix.
