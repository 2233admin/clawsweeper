## Summary

Keep open. Current main still lacks the requested post-startup provider-history catch-up for missed inbound messages. The Discord path named in the report wires live MessageCreate events into an in-process handler/queue, with no post-READY fetch after the bot's last outbound message or durable missed-message replay. Slack is similarly live-event based; Telegram polling has provider-specific offset recovery, but that does not satisfy the Discord/all-channel catch-up request. Related restart-loss context (#51620 and #42157) also remains open.

## Best Possible Solution

Keep this issue open and implement a bounded, configurable provider-history catch-up path. The likely design is a channel/plugin SDK contract for providers that can list recent inbound messages, then per-provider implementations starting with Discord: resolve or persist a safe per-conversation anchor such as last outbound bot message, fetch a small time-bounded window after startup/READY, filter bot/self/duplicates through durable keys, and enqueue accepted messages through the existing inbound worker. Add focused Discord tests first, then Slack/other provider coverage where their APIs permit history reads, and document which channel modes support catch-up.
