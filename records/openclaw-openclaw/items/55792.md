## Summary

Keep open. Current main still does not implement the requested broad post-startup catch-up for inbound messages sent while the gateway is offline. BlueBubbles now has a provider-specific startup catchup path, and Telegram long polling has accepted-offset replay protection, but Discord and Slack remain live-event driven and the channel plugin contract has no generic catch-up/list-recent-inbound surface for core to invoke across providers. The related Telegram webhook report #42157 is narrower and remains open, so it is not a canonical replacement for this broader issue.

## Best Possible Solution

Keep this issue open and design an additive channel catch-up surface. A conservative implementation would add a plugin SDK contract for providers that can list recent inbound messages for active conversations, then implement provider-specific startup handlers starting with Discord and Slack: anchor to a safe last-outbound or durable cursor, fetch a small time-bounded window after READY/Socket Mode startup, filter self/bot and duplicate messages, enqueue through the normal inbound path, and document per-channel support. Telegram webhook durability can remain a narrower companion unless the same durable delivery primitives become reusable.

## Risks / Open Questions

- A broad catch-up implementation could duplicate replies or replay stale messages unless it is bounded, configurable, provider-aware, and deduped with durable message keys.
- Telegram webhook loss is an ACK/persistence problem, while Discord and Slack catch-up is provider-history replay; solving one path will not automatically solve the others.
