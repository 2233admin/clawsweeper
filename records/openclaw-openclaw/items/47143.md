## Summary

Keep open. Current main still treats the reported persisted 0/0 memory-flush counter state as already flushed, the latest release has the same helper behavior, and the open adjacent PR changes related counter persistence without resolving this specific ambiguity.

## What This Changes

The issue asks for memoryFlush to run for over-threshold sessions when both compactionCount and memoryFlushCompactionCount are 0 instead of treating that state as already flushed.

## Best Possible Solution

Define a backward-compatible memoryFlush metadata contract that distinguishes never-flushed or legacy rows from a recorded flush at compaction count 0, then update the helper, runner persistence, focused tests, and changelog as needed while keeping broader pre-request context-budget work separate.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: with totalTokens above threshold and both counters at 0, shouldRunMemoryFlush reaches hasAlreadyFlushedForCurrentCompaction, which returns true and skips memoryFlush.

## Solution Assessment

No fix is present on main. The safest solution is not a blind compactionCount === 0 carve-out; it should disambiguate the persisted state contract, likely using memoryFlushAt or an explicit future sentinel, and coordinate with #51421.
