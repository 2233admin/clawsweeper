## Summary

Keep open. Current main still treats the reported 0/0 memory-flush counter state as already flushed, and the latest release snapshot carries the same helper behavior. The bug is concrete, but the fix should be coordinated with the related open memory-flush counter PR because a naive 0/0 special case can conflict with real cycle-0 flush records.

## What This Changes

The issue asks for memoryFlush to trigger for un-compacted sessions with `compactionCount: 0` and `memoryFlushCompactionCount: 0` instead of treating that state as already flushed.

## Best Possible Solution

Define a backward-compatible session-state contract that distinguishes never-flushed or legacy 0/0 rows from recorded cycle-0 flushes, update the helper and runner persistence accordingly, and add focused regression coverage while keeping broader pre-request context-budget work in the related issues.
