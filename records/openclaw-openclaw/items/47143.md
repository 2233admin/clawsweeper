## Clownfish Work Prompt

Fix the memory-flush cycle-0 metadata ambiguity reported in openclaw/openclaw#47143. Current main treats `{ compactionCount: 0, memoryFlushCompactionCount: 0 }` as already flushed, which can suppress the first flush for legacy or externally initialized entries that have no evidence of a prior flush. Do not implement the broader pre-request context-budget guard in this PR. Do not use the naive fix from #47174/#47247 that returns false for every compactionCount=0 state, because review found that it repeats flushes before first compaction. Implement a narrow backward-compatible fix, preferably using existing `memoryFlushAt` or an explicit sentinel, so: no recorded flush at cycle 0 can run once; a recorded cycle-0 flush is still blocked until compaction advances; existing compactionCount > 0 semantics stay unchanged. Update `src/auto-reply/reply/memory-flush.ts` and any necessary entry picks/call sites, plus focused tests in `src/auto-reply/reply/reply-state.test.ts` and `src/auto-reply/reply/agent-runner-memory.test.ts` if integration coverage is needed. Add regression tests for zero/no-`memoryFlushAt` triggers, zero/with-`memoryFlushAt` skips, undefined `memoryFlushCompactionCount` triggers, and nonzero equality/difference behavior. Add a user-facing changelog fix entry if the active changelog requires one, crediting @JameelHao and @k5smdevops; do not add unrelated preflight context changes.
