## Summary

Keep open for maintainer review. The PR targets a real current-main compatibility gap: legacy `embeddedHarness` is still accepted and migrated by doctor, but the shared runtime-policy resolver currently ignores it, while harness selection and startup plugin activation already depend on that resolver.

## What This Changes

The PR updates agent runtime policy resolution to merge legacy `embeddedHarness` runtime/fallback values into missing `agentRuntime` fields and adds focused tests for harness selection, `sessions_spawn` routing, and startup plugin activation.

## Best Possible Solution

Land this PR or an equivalent central resolver compatibility change after maintainer review, keeping explicit `agentRuntime` precedence, fill-missing legacy semantics, and the native Codex versus explicit ACP route split intact.

## Reproduction Assessment

Yes. On current main, a config using only `agents.defaults.embeddedHarness` or `agents.list[].embeddedHarness` reaches schema/doctor compatibility but not the shared resolver used by harness selection and startup activation.

## Solution Assessment

Yes. Centralizing the compatibility merge in `resolveAgentRuntimePolicy` is the narrowest maintainable fix because the relevant consumers already use that resolver and doctor already defines the same fill-missing semantics.
