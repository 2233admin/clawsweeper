## Summary

Keep this PR open for maintainer review. Current main still ignores the deprecated `embeddedHarness` runtime/fallback fields in the shared runtime-policy resolver even though the config surface and doctor migration still accept them; the PR’s central resolver compatibility approach matches the existing migration semantics and I found no concrete blocking defect in the provided latest diff.

## What This Changes

The PR updates agent runtime policy resolution to merge legacy `embeddedHarness` values into missing `agentRuntime` fields and adds focused tests for harness selection, startup plugin activation, and `sessions_spawn` routing.

## Best Possible Solution

Land this PR or an equivalent central resolver compatibility change after maintainer review, preserving explicit `agentRuntime` precedence, fill-missing legacy semantics, plugin disablement behavior, and the documented native Codex versus explicit ACP route split.

## Reproduction Assessment

Yes. A config using only `agents.defaults.embeddedHarness` or `agents.list[].embeddedHarness` is still accepted by current types/schema and doctor migration, but current main’s shared resolver ignores that field before harness selection and startup activation consume policy.

## Solution Assessment

Yes. Centralizing compatibility in `resolveAgentRuntimePolicy` is the narrowest maintainable fix because harness selection, startup plugin activation, agent metadata, and runtime alias helpers already depend on that resolver rather than each carrying separate legacy fallback logic.
