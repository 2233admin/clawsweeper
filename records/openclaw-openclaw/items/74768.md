## Summary

Keep open. Current main still ignores the deprecated embeddedHarness runtime policy in the shared resolver while the type/schema surface still accepts it, and the PR is a focused implementation candidate with no blocking review findings from this sweep.

## What This Changes

The PR updates agent runtime policy resolution to merge legacy embeddedHarness runtime/fallback values into missing agentRuntime fields and adds tests for harness selection, sessions_spawn routing, and startup plugin activation.

## Best Possible Solution

Land this PR or an equivalent central resolver compatibility change after maintainer review, preserving explicit agentRuntime precedence, fill-missing legacy semantics, plugin disablement, and the documented native sub-agent versus explicit ACP route split.

Yes. A config using only agents.defaults.embeddedHarness or agents.list[].embeddedHarness reproduces the gap on current main because the accepted legacy field reaches a shared resolver that only reads agentRuntime.

## Solution Assessment

Yes. Centralizing the compatibility merge in resolveAgentRuntimePolicy is the narrowest maintainable fix because the downstream harness selection, startup activation, metadata, and alias paths already consume that resolver.
