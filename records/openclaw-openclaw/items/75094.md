## Summary

Keep open for normal PR review. Current main still renders assistant media with long-lived token query URLs, while the PR branch implements the scoped-ticket path and the latest commit addresses the earlier stale-ticket cache finding. I did not find a blocking correctness or supply-chain issue, but CI was still incomplete and the linked maintainer-labeled follow-up may need an explicit decision on legacy query-token compatibility.

## What This Changes

The PR mints short-lived HMAC-scoped assistant-media tickets from authenticated metadata checks, lets the Gateway serve media bytes with those tickets, and updates Control UI chat rendering/tests to use Authorization headers for probes and mediaTicket URLs for rendered media.

## Best Possible Solution

Land the scoped-ticket architecture once CI is green and maintainers are satisfied with the legacy query-token compatibility boundary; the shipped end state should keep native browser media rendering while avoiding real long-lived credentials in Control UI-rendered URLs.

## Reproduction Assessment

Yes. On current main, rendering a local assistant attachment with an assistantAttachmentAuthToken builds an assistant-media URL containing token=; on the PR branch, metadata fetches return mediaTicket/mediaTicketExpiresAt and the expiry-refresh test exercises the stale-ticket path.

## Solution Assessment

Yes, with normal maintainer review. Gateway-signed, source-scoped, short-lived HMAC tickets are the narrowest maintainable solution described by #70830, and the latest commit fixes the earlier cache-expiry defect without moving to blob URLs or a broader cookie/session model.
