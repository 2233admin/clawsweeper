## Summary

Keep open. Current main still renders assistant media URLs with `token=`, while this PR implements the requested scoped-ticket direction with focused Gateway/UI changes, tests, and a changelog entry. I did not find a discrete blocking code bug in the diff, but exact-head CI was still in progress and the remaining legacy query-token compatibility boundary should get maintainer review before merge.

## What This Changes

The PR adds short-lived HMAC-scoped assistant media tickets in the Gateway, updates Control UI chat rendering to use Authorization metadata probes and `mediaTicket=` media URLs, adds focused Gateway/UI tests, and records the user-facing fix in the changelog.

## Best Possible Solution

Land this PR after exact-head CI finishes green and maintainers accept or explicitly follow up the temporary legacy query-token compatibility boundary; then the linked follow-up can close through the merge.

## Reproduction Assessment

Yes. Current main reproduces the reported credential-in-URL behavior by rendering local assistant attachments with `assistantAttachmentAuthToken`, which produces `assistant-media?...&token=...`, and the route still accepts that query token.

## Solution Assessment

Yes, with merge-readiness caveats. Short-lived source-scoped HMAC tickets minted after authenticated metadata/local-root checks are the narrowest maintainable way to keep native browser media rendering without embedding long-lived credentials in rendered URLs.
