## Summary

Keep open. Current main still renders assistant-media URLs with long-lived `token=` credentials, while this active PR implements the requested scoped-ticket path and has no discrete code-review finding; the remaining blocker is merge readiness and maintainer validation, not cleanup closure.

## What This Changes

The PR adds short-lived HMAC-scoped assistant media tickets in the Gateway, switches Control UI metadata probes to Authorization headers, renders local assistant media with `mediaTicket=`, adds gateway/UI tests, and updates the changelog.

## Best Possible Solution

Update the branch against current main, keep the scoped-ticket flow plus local media-root checks, decide whether legacy query-token server compatibility is acceptable, then land after targeted gateway/UI tests and the repo-approved changed gate pass.

## Reproduction Assessment

Yes. Current main can be reproduced from source by rendering a local assistant attachment with `assistantAttachmentAuthToken`, which produces an assistant-media URL containing `token=`, and the current gateway route accepts query-token auth for that route.

## Solution Assessment

Yes, with merge-readiness caveats. Source-scoped, short-lived HMAC tickets are the narrowest maintainable way to preserve native browser media rendering without placing long-lived operator credentials in URLs; cookie auth or blob URLs would be broader changes.
