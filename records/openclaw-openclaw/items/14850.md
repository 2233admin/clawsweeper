## Summary

Current main still does not provide a binding-level session key, alias, or non-DM session scope override for Discord guild/channel traffic. Route binding session config is still limited to `dmScope`, and channel peers still resolve to isolated `agent:<agentId>:discord:channel:<channelId>` keys, so the requested cross-channel session unification remains open work.

## Best Possible Solution

Keep this issue open for maintainer product/security design. The best implementation would add a documented explicit opt-in binding session target, such as `bindings[].session.target`, `sessionKey`, or `sessionScope`, preserve default group/channel isolation, validate canonical agent session keys or well-defined aliases, thread the target through route resolution/session-key construction, update config schema/help/docs, and add regression tests proving a Discord channel binding can intentionally share an agent main/alias session while unconfigured channels stay isolated.

## Risks / Open Questions

- A binding-level target that merges guild/channel traffic into a shared agent session can leak channel context into DM history unless it is explicit, validated, documented, and opt-in.
- A fixed binding session target needs route-cache and last-route-policy design so a previously matched binding cannot reuse stale session resolution across different target semantics.
- Discord thread-bound session infrastructure can override effective target session keys for focused/thread workflows, but it is not a documented general `bindings[]` session alias and should not be treated as satisfying this issue.
