## Summary

Keep this PR open. Current main still routes Signal group allowlist checks through a sender-only comparator and does not pass the inbound Signal group id into `resolveSignalAccessState`, so the PR’s intended behavior is not implemented on main. The linked bug remains open, and this PR is still a plausible implementation candidate rather than conservative cleanup material.

## Best Possible Solution

Keep this PR open for Signal maintainer review. The best path is to land or revise a Signal-local fix that compares `groupAllowFrom` against the inbound `groupId` only for the group access path, preserves sender allowlist compatibility where intended, adds regression tests for group-id allowlists and DM non-matching, and adjusts the linked issue/closing wording if the `requireMention` portion remains unresolved.

## Risks / Open Questions

- The current PR diff, as provided, appears to use one sender-or-group callback for both group and DM access resolution; the existing Greptile review notes a semantic leak where a group id in `allowFrom` could affect `dmAccess` computation for a group-message event. It does not look like a practical security bypass because DM access is consumed on the non-group path, but it should be cleaned up or explicitly accepted before landing.
- The PR’s closing reference may be broader than the patch: the linked issue also describes `requireMention` default behavior, while this PR focuses on `groupAllowFrom` group-id matching.
