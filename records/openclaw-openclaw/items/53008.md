## Summary

Keep open. Current main still runs preflight compaction and memoryFlush in the foreground reply operation, neither path passes a dedicated compaction lane, omitted embedded lanes still default to `main`, and same-session work is serialized before the global lane. The closed, unmerged #53154 attempted only part of this, while #73581 is a broader open lane-stall report rather than a replacement for this focused compaction bug.

## What This Changes

The issue asks for session memory compaction and memoryFlush to stop blocking Telegram inbound message processing for 10+ minutes, either by using a separate/background lane, shortening timeout behavior, or adding recovery monitoring.

## Best Possible Solution

Define a compaction-specific scheduling and recovery policy for preflight compaction and memoryFlush: bounded background or dedicated compaction execution that cannot fall back to `CommandLane.Main`, does not starve same-session inbound messages, preserves transcript/session rotation consistency, and includes regression coverage for queued inbound messages during active compaction.
