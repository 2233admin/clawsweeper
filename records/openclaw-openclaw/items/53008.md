## Summary

Current main still does not implement the requested isolation for memoryFlush compaction. The reply path awaits memory maintenance before continuing, the memory-triggered embedded run still omits an explicit lane and therefore falls back to the main global lane, and the linked implementation candidate #53154 was closed unmerged.

## Best Possible Solution

Keep this issue open. The best fix is to route memoryFlush and related pre-compaction work through an explicit background or dedicated compaction lane with bounded timeout/concurrency, while also addressing same-session responsiveness. Add regression coverage proving the memoryFlush run cannot fall back to CommandLane.Main and that inbound user messages are not starved while compaction is pending or stuck.

## Risks / Open Questions

- Default main-lane concurrency is now 4, which reduces process-wide starvation compared with the reporter's totalActive=1 deployment, but configs can still set main concurrency to 1 and same-session messages still wait behind the active memoryFlush run.
- Routing only the global lane to subagent may not fully solve same-session responsiveness because runEmbeddedPiAgent also serializes by session lane; maintainers need to decide whether compaction can yield, bypass, or be preempted within per-session serialization.
