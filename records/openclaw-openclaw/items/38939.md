## Summary

Keep this PR open. Current main still lacks Feishu dispatch-time arbitration across sibling bot accounts, and this PR is not superseded by the related Feishu metadata or merge-forward PRs. The PR should not merge unchanged because review comments and the patch show a hardcoded `main` coordinator and an untested substring text-mention fallback.

## Best Possible Solution

Keep this PR open and request changes or a maintainer-approved replacement. The corrected fix should arbitrate before dispatch using structured mentioned bot open_ids from Feishu text/post events, derive the coordinator from configured Feishu routing rather than a literal account id, avoid substring-only implicit matching, preserve existing account-scoped dedupe for ordinary duplicates, and add regression tests for specialist-only, coordinator-plus-specialist, and multi-specialist group mentions.

## Risks / Open Questions

- Current main still allows sibling Feishu bot accounts to process the same group event when ordinary single-agent dispatch is used, so the duplicate-reply problem remains unresolved.
- The PR should not merge unchanged: a literal `main` coordinator can drop multi-specialist mentions in deployments whose primary Feishu account uses another id such as `default`.
- The PR adds substring-based implicit mention matching without focused tests, so plain text like `@maintenance` or `@mainbot` could unexpectedly trigger `@main` fallback behavior.
- The PR has no obvious supply-chain or CI-permission regression because it does not touch workflows, dependencies, lockfiles, release scripts, or secret-handling files.
