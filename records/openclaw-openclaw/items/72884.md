## Summary

Keep open. Current main still leaves actionable task-maintenance gaps for this report: durable cron recovery is only attempted for active cron rows, so terminal `lost` cron rows with backing-session-missing evidence are not recovered, and stale `cli` tasks owned by persisted subagent session rows can still be treated as live after the run context is gone.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to repair or prune `lost` backing-session-missing task records and stale running tasks, with optional cleanup for benign timestamp inconsistencies.

## Best Possible Solution

Keep this open until terminal lost cron recovery is merged or otherwise implemented, then add a focused fix for stale subagent-owned CLI task liveness; handle timestamp clamping as a separate explicit maintenance policy if maintainers want it.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: model a terminal `lost` cron task with durable run history and observe recovery is skipped, and model an old `runtime: "cli"` task with a persisted `agent:...:subagent:...` child session row but no live run context and observe current main treats the row as backing proof.

## Solution Assessment

No. Current main is not the best solution because it only covers active cron durable recovery and chat-backed CLI rows; the narrow maintainable path is to land or replace the cron PR and separately fix CLI liveness for subagent-owned task rows.
