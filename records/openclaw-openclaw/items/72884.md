## Summary

Keep open. Current main still excludes terminal `lost` cron task rows from durable cron recovery, so `backing session missing` audit errors can remain unresolved, and the related fix PR is open rather than merged.

## What This Changes

The issue asks for `openclaw tasks maintenance --apply` to clean or recover `lost` backing-session-missing task records, stale-running records, and possibly benign timestamp-skew warnings.

## Best Possible Solution

Update and land the narrow terminal-lost cron recovery fix so recoverable `backing session missing` cron rows are finalized from durable cron history and stale lost errors are cleared, then add focused coverage and decide separately whether stale-running cleanup and small timestamp-skew clamping belong in maintenance policy.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an already-terminal cron task with `status: "lost"`, `error: "backing session missing"`, a parseable cron run ID, and matching durable cron run history; current main skips recovery because the durable cron recovery guard only accepts active tasks.

## Solution Assessment

No. Current main is not sufficient for the reported lost-row behavior; the open PR's durable-history approach is the right narrow direction, but it needs to handle stale error cleanup and does not by itself settle the broader stale-running/timestamp scope.
