## Summary

Keep open. Current main still leaves the reported task-maintenance gap: terminal cron rows already marked `lost` cannot enter durable cron recovery, and stale CLI rows backed only by historical subagent session records can still look live. Open PR #75036 targets the cron slice, but it is unmerged and does not cover the newer CLI/subagent evidence or the optional timestamp-clamping policy.

## What This Changes

The issue asks `openclaw tasks maintenance --apply` to reconcile or prune `lost` backing-session-missing records and `stale_running` task records, with an optional cleanup for benign timestamp skew warnings.

## Best Possible Solution

Keep this issue as the canonical tracker until PR #75036, or an equivalent narrow fix, lands for terminal lost cron recovery; then add a focused task-maintenance fix so subagent-owned CLI task rows are not considered live after the active run context is gone. Treat timestamp clamping as a separate policy follow-up.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a terminal `runtime: "cron"`, `status: "lost"`, `error: "backing session missing"` row with matching durable run history, plus a stale `runtime: "cli"` row whose `childSessionKey` is `agent:...:subagent:...`, no live run context, and an old session-store entry; current main leaves both gaps reachable.

## Solution Assessment

No. Current main is only a partial solution: durable cron recovery works for active rows and chat-backed CLI rows, but not terminal lost cron rows or subagent-backed stale CLI rows. The best maintainable path is a narrow cron recovery change plus a separate CLI liveness predicate regression, not blanket pruning of all lost or stale records.
