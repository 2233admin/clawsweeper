## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the retry disablement, and the linked loop regression remains plausible, but the proposed call mutates a persistent Pi settings manager on the trusted file-backed path instead of applying a session-only embedded-run override.

## What This Changes

The PR disables pi-coding-agent SDK auto-retry in `createPreparedEmbeddedPiSettingsManager` and adds a Vitest assertion that the prepared settings manager reports retry disabled.

## Best Possible Solution

Land a narrow embedded-run fix that disables pi-coding-agent agent-level auto-retry only for OpenClaw sessions, keeps the override effective after resource-loader reloads, avoids persistent user/global settings writes, and covers both in-memory and trusted file-backed settings paths in tests.

## Reproduction Assessment

Unclear for the original live tool-call loop because no live Feishu/Qwen run was executed. Yes for the PR regression: configure embedded Pi project settings policy as `trusted` with no plugin settings, and this patch calls a setter that upstream persists to global retry settings.

## Solution Assessment

No, not as written. Disabling the SDK's inner retry is a plausible narrow direction, but it should be a runtime/session-scoped override rather than a persistent `setRetryEnabled(false)` write, and it should be coordinated with the broader #74503 proposal.
