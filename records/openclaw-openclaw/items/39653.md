## Summary

Keep this PR open. Current main still accepts arbitrary non-IP `ws://` hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, and current gateway/onboarding tests still assert that behavior. The PR remains a narrow security hardening candidate; the earlier #37883 attempt was closed unmerged, and the linked #37177 report was closed stale rather than fixed.

## Best Possible Solution

Keep this PR open for maintainer security review. The best path is to land or adapt the narrow validator change so `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` accepts loopback/private IP literals but rejects unresolved hostnames, then update gateway net, client, connection-detail, and onboarding tests to match the chosen contract. If private DNS hostname support is required, track it separately with an explicit allowlist or DNS-resolution design.

## Risks / Open Questions

- Current main still permits plaintext `ws://` to arbitrary hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, so a user-influenced remote URL or DNS target can downgrade credentials and chat traffic away from TLS.
- Landing this PR's fail-closed behavior may break users relying on private DNS, Tailnet DNS, `.local`, or corporate hostnames with the break-glass env; if that support is required, it needs an explicit DNS-resolution or allowlist design rather than the current blanket hostname pass-through.
- The live PR diff only updates `src/gateway/net.test.ts`; current main also has hostname-acceptance expectations in gateway client, connection-detail, and onboarding tests, so the patch likely needs rebasing or expansion before merge.
