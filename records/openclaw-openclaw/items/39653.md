## Summary

Keep this PR open. Current main still accepts arbitrary non-IP `ws://` hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, and gateway/client/connection-detail/onboarding tests still assert that behavior. The PR remains a narrow security hardening change; #37177 was closed stale rather than fixed, and #37883 was closed unmerged.

## Best Possible Solution

Keep this PR open for maintainer security review. The best path is to land or adapt the narrow validator change so `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` accepts loopback/private IP literals but rejects unresolved hostnames, then update gateway net, client, connection-detail, onboarding tests, and any affected docs to match the chosen contract. If private DNS hostname support is required, track that separately with an explicit allowlist or async DNS-resolution design.

## Risks / Open Questions

- Current main still permits plaintext `ws://` to arbitrary hostnames when `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1`, which can expose gateway credentials and chat/conversation data if a user-influenced URL or DNS target is used.
- Landing the fail-closed behavior may break users relying on private DNS, Tailnet DNS, `.local`, or corporate hostnames over plaintext; if that use case is required, it needs an explicit allowlist or DNS-resolution design rather than the current all-hostname fallback.
