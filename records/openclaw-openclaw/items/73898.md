## Summary

Keep open. Current main still lacks a default-off Control UI setting or playback-consent path for assistant TTS audio, and the newer assistant-media audio block path is not normalized into a playable UI attachment.

## What This Changes

Add an opt-in Control app / Control UI setting to auto-play newly generated assistant TTS audio attachments, with graceful loading and blocked-playback handling.

## Best Possible Solution

Implement a focused default-off playback preference in the Control UI, normalize gateway audio content blocks into playable audio attachments, auto-play only new assistant audio after explicit opt-in, show a compact enable/playback-blocked prompt on failure, and document the behavior without changing backend TTS generation or provider/channel semantics.

## Reproduction Assessment

Yes. A code-level reproduction is clear: feed the browser normalizer an assistant message with a gateway `type: "audio"` content block and it will not become an audio attachment; feed it a `MEDIA:...mp3` attachment and it renders a manual `<audio controls>` element with no auto-play path.

## Solution Assessment

Yes. The requested default-off opt-in UI setting is the right direction, with the important addition that the UI should first make embedded assistant audio blocks renderable before attempting auto-play.
