## Summary

Keep open. Current main still treats generated assistant TTS as a normal Control UI audio attachment: backend TTS generation and WebChat media embedding exist, but there is no browser-local auto-play preference, no assistant-attachment `play()` path, and no blocked-autoplay recovery prompt.

## What This Changes

Add an opt-in Control app / Control UI setting that automatically plays newly generated assistant TTS audio attachments when they arrive.

## Best Possible Solution

Implement this as a focused Control UI playback enhancement: store a default-off browser-local assistant audio auto-play preference, expose it in the existing settings surface, auto-play only newly arrived assistant audio attachments after explicit opt-in, show a compact retry/enable prompt when playback is blocked, and leave backend TTS generation, channel delivery, and Talk mode unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: auto-TTS creates assistant audio media, and the Control UI renders that media as a manual `<audio controls preload="metadata">` attachment with no persisted opt-in or playback call.

## Solution Assessment

Yes. The requested explicit browser-local opt-in is the narrowest maintainable solution because browser autoplay policy requires a user gesture; changing backend TTS or channel media semantics would not solve the Control UI playback consent problem.
