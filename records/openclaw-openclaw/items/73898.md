## Summary

Keep open. Current main still generates assistant TTS audio and renders assistant audio attachments in the Control UI, but the UI has no persisted opt-in auto-play setting, no assistant-attachment `play()` path, and no blocked-autoplay recovery prompt. The related duplicate assistant-media report is distinct: it covers duplicate text delivery, not the requested playback preference.

## What This Changes

Add a default-off Control UI setting that auto-plays newly generated assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

Implement a focused Control UI enhancement: add a browser-local, default-off assistant audio auto-play preference, expose it through existing settings patterns, pass it into chat rendering, auto-play only newly arrived assistant audio attachments after opt-in, show a compact enable/retry prompt when `play()` is blocked, and leave backend TTS generation, channel delivery, and Talk mode unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: auto-TTS attaches generated audio to assistant replies, the gateway can persist assistant-media audio content, and the Control UI renderer currently emits only a manual `<audio controls preload="metadata">` element with no persisted playback opt-in or autoplay handling.

## Solution Assessment

Yes. The requested explicit browser-local opt-in is the narrowest maintainable solution; changing backend TTS generation or channel media semantics would not satisfy browser autoplay policy and could cause surprising sound on other surfaces.
