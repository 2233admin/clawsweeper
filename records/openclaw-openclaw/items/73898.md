## Summary

Keep open. Current main still generates assistant TTS media, but the Control UI only exposes generated audio as a manual attachment and has no browser-local opt-in, playback-consent path, or blocked-autoplay prompt for assistant TTS audio.

## What This Changes

Add a default-off Control app or Control UI setting that automatically plays newly generated assistant TTS audio attachments after explicit user opt-in.

## Best Possible Solution

Implement this as a focused Control UI feature: persist a browser-local, default-off assistant audio auto-play preference, expose it in the existing settings UI, auto-play only newly arrived assistant audio attachments after the user enables it, handle play() rejection with a compact prompt, and keep backend TTS generation and channel delivery semantics unchanged.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: auto-TTS attaches audio media to assistant replies, the Control UI normalizes that into an audio attachment, and the renderer outputs a manual audio control with no persisted auto-play setting or playback-consent path.

## Solution Assessment

Yes. The requested browser-local opt-in is the narrowest maintainable direction because browser autoplay policy depends on user gesture; changing backend TTS or global channel semantics would not solve the Control UI playback consent problem.
