## Summary

Keep open. Current main still treats assistant TTS output in the Control UI as a normal manual audio attachment, while the requested default-off browser-local auto-play preference, play/retry path, and blocked-autoplay prompt are not present.

## What This Changes

Add a Control app / Control UI setting that opt-in auto-plays newly generated assistant TTS audio attachments.

## Best Possible Solution

Implement this as a focused Control UI feature: add a default-off browser-local assistant audio auto-play setting, expose it through existing settings patterns, pass it into chat rendering, play only newly arrived assistant audio attachments after opt-in, and show a small enable/retry prompt when the platform blocks playback.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: auto-TTS creates audio-bearing assistant payloads, and the current Control UI renders those audio attachments only as manual controls with no stored auto-play consent or play() path.

## Solution Assessment

Yes. The requested explicit UI-local opt-in is the narrowest maintainable solution; changing backend TTS generation or channel media semantics would not satisfy browser autoplay policy and could surprise users on other surfaces.
