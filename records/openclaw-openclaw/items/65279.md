## Summary

Keep open. Current main has related MiniMax stream-shaping, WSL2 Undici transport, prompt-hook timeout, and memory-recall latency mitigations, but they do not directly prove the reported MiniMax-M2.7 CN/API/OAuth 10+ second pre-lifecycle delay is fixed. The related closed #72365 thread is broader Control UI/gateway WSL2 unresponsiveness, not a canonical resolution for this provider-specific report.

## Best Possible Solution

Keep this issue open until a maintainer can reproduce or disprove it on current main with MiniMax CN API-key and OAuth paths on WSL2. Add timing around `chat.send` receipt, media/link preprocessing, message hooks, session/model/directive resolution, `runEmbeddedAttempt` entry, outbound MiniMax request send, first response bytes, lifecycle start, and first visible assistant text. If reproduced, fix the implicated MiniMax provider wrapper, transport path, prompt-hook/memory path, or pre-run setup and add a focused regression test or diagnostic timing coverage. If current main no longer reproduces with that trace, close it then as implemented or cannot-reproduce with the timing evidence.
