## Summary

Current main still matches the reported sqlite-vec unsafe-reindex failure path. `runSync()` loads/caches vector availability before choosing the unsafe full reindex path; `runUnsafeReindex()` then drops `chunks_vec` through `resetIndex()`, but `syncMemoryFiles()` still prepares `DELETE FROM chunks_vec ...` when `vector.enabled && vector.available` before any later vector-table recreation. The related PR #71611 is open and addresses the safe-path Windows rename/EBUSY problem, not the exact unsafe sqlite-vec table lifecycle crash.

## Best Possible Solution

Keep #70559 open. Add a focused memory-core regression test for unsafe full reindex with vector availability true after `resetIndex()`, then fix the lifecycle by preserving/purging `chunks_vec` during reset or by resetting/recreating vector state before any prepared statement references the vector table. Keep #71611/#64187 linked as related Windows safe-reindex work, but do not treat them as closing this unsafe sqlite-vec crash unless a successor explicitly fixes or removes that path.
