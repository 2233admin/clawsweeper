## Summary

Current main still has the unsafe-reindex/sqlite-vec state mismatch reported here. `sync()` can mark sqlite-vec available, the env-gated unsafe full reindex path calls `resetIndex()`, `resetIndex()` drops `chunks_vec` without clearing `vector.available`, and the next memory/session sync setup prepares `DELETE FROM chunks_vec ...` solely because availability is still true. A direct `node:sqlite` probe confirms preparing that statement against a missing table throws `no such table: chunks_vec`. This should stay open and is a good narrow fix-PR candidate.

## Best Possible Solution

Keep this issue open and fix `memory-core` so unsafe full reindex keeps vector table state consistent: either preserve and clear `chunks_vec` when sqlite-vec is available, or reset/recreate vector state before any sync code prepares statements. Add a focused regression test with vector availability true that fails on a missing `chunks_vec` prepare before the fix. Treat #64187/#71611 as the separate safe-path Windows EBUSY track.
