## Summary

Keep open. The latest PR head is still not mergeable: it introduces build/API breakage in gateway session readers, leaves normal Pi session transcript writes outside the encryption path, and changes at-rest security semantics without a safe key-management design.

## What This Changes

The PR adds session encryption helpers, encrypted `sessions.json` read/write handling, an encrypted wrapper for one assistant transcript append path, and gateway transcript readers that attempt to decrypt encrypted JSONL lines.

## Best Possible Solution

Design transcript-at-rest encryption around a single session persistence boundary that covers normal Pi session writes, compaction, repair, history reads, and metadata storage; pair that with explicit key management/recovery semantics, fail-closed migration behavior, tests, docs, changelog, and maintainer security approval.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists from the PR head: `await` is added inside a non-async function, synchronous readers are changed to return promises while callers remain synchronous, and normal embedded runs still use direct `SessionManager.open` writes on current main.

## Solution Assessment

No. This is not the best way to solve the issue because it patches selected readers and one delivery-mirror writer instead of the central persistence seam, and it introduces key-management behavior that does not satisfy the stated filesystem-compromise threat model.

## Security Review

Status: needs_attention

Summary: The diff introduces at-rest encryption and migration behavior with concrete key-management and plaintext-downgrade concerns that need maintainer security review.

Concerns:

- **[high] Session keys are stored beside the encrypted data:** `src/config/sessions/encryption.ts:45`
  - body: The new wrapper stores both the passphrase file and wrapped master key under the OpenClaw state directory. A same-user filesystem reader who can copy sessions can likely copy these key files too, so this does not provide the claimed defense-in-depth boundary without a different key source such as a user secret or platform keystore.
  - confidence: 0.86
- **[medium] Migration can silently preserve plaintext transcript lines:** `src/config/sessions/migrate-transcripts.ts:35`
  - body: `migrateTranscriptToEncrypted` catches per-line encryption failures, writes the original plaintext line, and still returns success. That can leave sensitive content unencrypted while callers believe migration completed.
  - confidence: 0.88
