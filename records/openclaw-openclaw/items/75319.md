## Summary

Keep open. This is a plausible security-hardening direction, but the current PR head has a confirmed parse blocker, ESM runtime blockers, incomplete transcript-write coverage, and unresolved key-management/threat-model concerns, so it needs maintainer/security-owner design review before it can be merged.

## What This Changes

The branch adds AES-GCM helpers for session transcript and sessions.json encryption, migration utilities, gateway/session reader changes, and a status document for transparent on-disk session encryption.

## Best Possible Solution

Design encryption at the central session persistence boundary with an explicit key source, recovery and rollback semantics, fail-closed migration, full reader/writer coverage, docs, changelog, focused tests, and maintainer security approval before landing.

## Reproduction Assessment

Yes. A high-confidence static and CI-backed reproduction exists: the PR head contains an `await` inside a synchronous function and CI reports the exact parse error, while current main still writes normal embedded transcripts through `SessionManager.open(params.sessionFile)`.

## Solution Assessment

No. The proposed patch is not the best way to solve the issue because it patches selected readers and one secondary append helper instead of the central persistence seam, and it stores local key material alongside the encrypted state it is meant to protect.

Overall confidence: 0.94

Full review comments:

- **[P1] Keep async imports out of the preview reader:** `src/gateway/session-utils.fs.ts:474`
  - body: `readLastMessagePreviewFromOpenTranscript` remains a synchronous function, but this line uses `await import(...)`. The PR head already fails to parse in CI here, so this must be changed to a static import or the full call chain must become async consistently.
  - confidence: 0.98
- **[P2] Avoid CommonJS require in ESM session readers:** `src/gateway/session-utils.fs.ts:107`
  - body: This package is ESM, but the synchronous session reader now calls `require(...)`. Any runtime path that reaches `readSessionMessages` under ESM can fail with `require` undefined; use an ESM import or an existing lazy runtime boundary.
  - confidence: 0.93
- **[P2] Replace require in EncryptedSessionManager.open:** `src/config/sessions/encrypted-session-manager.ts:242`
  - body: `EncryptedSessionManager.open()` is added in ESM-loaded TypeScript but calls `require('@mariozechner/pi-coding-agent')`. Paths that reach this helper can fail under ESM before any encryption behavior runs; import the dependency through ESM instead.
  - confidence: 0.92
- **[P2] Cover the primary transcript writer:** `src/config/sessions/transcript.ts:281`
  - body: This only routes the delivery-mirror append helper through the encrypted wrapper. Normal embedded runs still write through the direct Pi SessionManager path in current main, so ordinary session JSONL writes can remain plaintext despite the claimed transparent encryption.
  - confidence: 0.91
- **[P2] Fail migration instead of writing plaintext lines:** `src/config/sessions/migrate-transcripts.ts:37`
  - body: When a line fails to encrypt, migration appends the original plaintext line and can still return success after rewriting the file. That leaves sensitive content unencrypted while callers believe migration completed.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The diff introduces at-rest encryption and migration behavior with concrete key co-location and plaintext fallback concerns.

Concerns:

- **[high] Session keys are stored beside encrypted state:** `src/config/sessions/encryption.ts:45`
  - body: The PR stores `.session-encryption-key` and `.master-key` under the same OpenClaw state root as the session data. A same-user filesystem reader who can copy transcripts can likely copy these files too, so this does not satisfy the stated filesystem-compromise defense boundary without a different key source such as a user secret or platform keystore.
  - confidence: 0.88
- **[medium] Transcript migration can preserve plaintext lines:** `src/config/sessions/migrate-transcripts.ts:37`
  - body: `migrateTranscriptToEncrypted` catches per-line encryption failures, keeps the original plaintext line, writes the file, and returns success. That can leave sensitive transcript content unencrypted while migration appears complete.
  - confidence: 0.9
