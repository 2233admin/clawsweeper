## Summary

Keep this open. Current main has partial stale-session lane recovery and a happy-path overflow-compaction retry test, but active embedded runs remain warning-only by default and the reported post-compaction retry lock still lacks source-and-test proof that it clears `processing` and accepts follow-up channel work.

## What This Changes

The issue reports that after context-overflow auto-compaction succeeds and retries the prompt, the session remains stuck in `processing` with `queueDepth=0`, blocking later messages until a gateway restart.

## Best Possible Solution

Add a focused regression that drives overflow compaction success followed by retry success, retry error, and a hung/lost retry terminalization case, then fix the runner or recovery path so the active run clears or enters a bounded recoverable state without broadly resetting legitimate long-running work.
