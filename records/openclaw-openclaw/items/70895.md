## Summary

Keep this PR open. Current main still computes webhook agent runs as isolated cron turns and passes `forceNew: true` for every `sessionTarget: "isolated"` job, so the reported session rollover still exists. The PR is also paired with open bug #70894 by the same author and links it with closing syntax, so the maintainer path is to review and land this PR or a narrow equivalent rather than cleanup-close it.

## Best Possible Solution

Review and land this PR or a narrow equivalent on main: compute `forceNew` from true scheduled isolated-cron semantics rather than every isolated agent turn, allow hook/webhook-sourced isolated turns with explicit hook provenance or stable hook session keys to reuse fresh existing sessions, and keep regression coverage proving ordinary isolated cron jobs still force fresh sessions.
