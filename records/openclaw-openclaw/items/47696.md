## Summary

Keep open. Current main still does not implement temporary `/trust`/`/untrust` exec elevation, and the PR is an active implementation attempt rather than obsolete cleanup. However, the proposed patch remains security-sensitive and has blocking correctness issues around time-window enforcement for queued runs plus status visibility, so it needs maintainer/security review and a design update before merge.

## What This Changes

The PR adds `/trust [minutes]` and `/untrust` chat commands, in-memory session-scoped trust-window state, an exec-security override during reply directive resolution, command registry/docs/changelog updates, and unit tests.

## Best Possible Solution

Adopt one trust-window design that treats the window as an execution-time authority, not a parse-time override; surface active trust and remaining time in `/exec` or equivalent status/help output; keep the change scoped to the chosen command surface; and land it with focused tests, docs, and a single credited changelog entry.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting the command registry, handler list, slash-command docs, and exec override resolver; the PR's queued-run defect is reproducible from the diff plus the current queue path that stores and later replays `run.execOverrides`.

## Solution Assessment

No. The feature direction is plausible, but resolving trust into `execOverrides.security` during directive parsing is not the safest way to enforce time-bounded privilege; the trust authority should be revalidated where exec policy is applied or immediately before queued execution.
