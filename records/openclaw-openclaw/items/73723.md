## Summary

Keep open. Current main still has a credible bug path matching the report: ordinary hook readiness and internal hook loading are separate from the Codex native hook relay registry, and a missing relay still returns `native hook relay not found` through `nativeHook.invoke`, which the CLI converts into a fail-closed `Native hook relay unavailable` response for PreToolUse. The related merged PRs #71008 and #71765 explain the intended relay bridge, but they do not make this issue obsolete; the reported restart/health mismatch remains unaddressed in current main.

## Best Possible Solution

Keep this issue open and fix it in the Codex native hook relay path. The likely product/code direction is to distinguish internal hook readiness from active native relay health, add diagnostics for active relay registrations, and either re-register/refresh the relay on Codex thread resume or gateway restart recovery, or fail open/defer to Codex's normal permission path when the relay id is missing. Add a regression test for a stale relay id after restart/resume so `hooks list` being ready cannot mask a broken native relay path.
