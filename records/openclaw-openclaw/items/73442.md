## Summary

Keep open. This is a draft PR from a MEMBER with the protected `maintainer` label, so cleanup should not close it, and current `main` does not already contain the proposed bounded filesystem helpers. The diff also has a concrete boundary bug in the new stat helper that needs maintainer/security review before merge.

## What This Changes

This draft PR adds `statPathWithinRoot`, `readdirWithinRoot`, and `renamePathWithinRoot` to `src/infra/fs-safe.ts` with new `fs-safe` tests for stat, missing paths, directory listing, traversal rejection, and rename behavior.

## Best Possible Solution

Keep the draft open, fix `statPathWithinRoot` to use the existing boundary canonicalization path or equivalent parent-symlink resolution, add a regression test for an in-root symlinked parent pointing outside the root, then rerun the focused `fs-safe` tests and core typecheck before reviewing follow-up consumer migrations.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `root/link/file.txt` where `link` is a symlink to an outside directory, then calling `statPathWithinRoot({ rootDir: root, relativePath: "link/file.txt", followSymlinks: false })`; the code uses `fs.lstat` but records the lexical path as `realPath`. I did not execute it because this ClawSweeper pass is read-only.

## Solution Assessment

No. The current PR direction is useful, but `statPathWithinRoot` should preserve only a final symlink when requested and still canonicalize or reject symlinked parent directories; reusing `resolveBoundaryPath`/`resolvePinnedPathWithinRoot` is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Canonicalize symlink parents when statting paths:** `src/infra/fs-safe.ts:371`
  - body: With `followSymlinks: false`, `fs.lstat` only avoids following the final path component; symlinks in parent directories are still traversed. Since this branch then sets `realPath` to the lexical `resolved` path, `root/link/file.txt` where `link` points outside the root can pass `isPathInside` and return metadata for an out-of-root file. Use the boundary resolver or canonicalize parents while preserving only the final symlink.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff has no workflow, dependency, lockfile, or release-script changes, but it introduces a filesystem-boundary helper with a symlink-parent escape in the `followSymlinks: false` stat path.

Concerns:

- **[medium] Stat helper can misclassify out-of-root paths:** `src/infra/fs-safe.ts:371`
  - body: The new stat helper treats the lexical resolved path as safe when `followSymlinks` is false even though `fs.lstat` still traverses symlinked parent directories. Future callers could rely on this helper as a safe root boundary while receiving metadata for an out-of-root target.
