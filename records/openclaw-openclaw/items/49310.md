## Summary

Keep this PR open. Current main still does not implement the intended TUI busy state for sessions_yield follow-up waits: final chat events settle back to idle/error, the TUI busy-state set lacks an awaiting-follow-up state, and history reload does not restore or clear that state from the sessions_yield context marker. Main does have backend sessions_yield pause/context behavior, so the remaining useful work is the TUI presentation and restore path. Security review found the proposed diff is confined to TUI runtime/tests, with no workflow, dependency, lockfile, package script, release, or secret-handling changes.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The best fix is to add a stable TUI-visible sessions_yield follow-up state, render it as busy, restore it from session history when appropriate, clear it when no longer valid, and cover both live event and history restore paths with focused TUI tests.

## Risks / Open Questions

- The proposed marker-string detection may be brittle if the sessions_yield context text changes; a shared helper/constant or typed yielded/paused signal would be more maintainable.
