## Summary

Keep this draft PR open. Current main still does not implement the intended TUI busy state for `sessions_yield` follow-up waits: the renderer only treats the existing run states as busy, final assistant events settle non-error runs to `idle`, lifecycle handling ignores available `yielded`/`livenessState: "paused"` metadata, and history restore does not preserve or clear an awaiting-follow-up state. The PR remains a focused implementation candidate, with no apparent supply-chain or secrets-handling regression in the proposed file set.

## Best Possible Solution

Keep this draft PR open for maintainer review or revision. The best implementation is a focused TUI change that adds `awaiting follow-up` as a busy activity state, derives it from stable typed/shared `sessions_yield` metadata where available, restores and clears it during history/session changes, and includes targeted live-event plus history-restore tests.
