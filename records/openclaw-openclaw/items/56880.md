## Summary

Keep open. Current main still documents and implements one active normal auto-reply run per session, with busy-session messages queued, collected, steered, or interrupting rather than spawning parallel normal turns. `/btw` is a useful explicit side-question path, but it is tool-less, ephemeral, command-gated, and not the general concurrent per-session handling requested here. The provided GitHub context also shows open PR #57239 uses closing syntax for this issue and remains unmerged, so the issue should stay open while that implementation candidate is reviewed or closed.

## Best Possible Solution

Keep this issue open and use PR #57239, or a replacement PR, as the review target. A good implementation should add a bounded fast-lane or concurrent-turn policy in the auto-reply queue/runner path, preserve the existing safe default queue modes, explicitly guard tool/session-write conflicts, and include focused tests for active runs, streaming state, Discord or external-channel delivery, transcript coherence, and interrupt/cancel behavior. `/btw` should remain documented as the explicit side-question workaround, not the complete fix.

## Risks / Open Questions

- Arbitrary concurrent same-session turns can race session transcript writes, tool execution, routing state, and provider rate limits; the current queue docs call out collision avoidance as the reason for serialization.
- The existing `/btw` path is intentionally tool-less and ephemeral, so treating it as full implementation would leave normal quick follow-up messages queued behind long-running turns.
