## Summary

Keep this issue open. Current main still sends the CLI override as `client.instanceId`, but the gateway’s node identity paths still prefer the paired device id or generic client id, so the custom id is not reliably present in `nodes list`, `nodes describe`, or node invoke targeting. The two linked fix PRs identified the same root cause but both closed unmerged.

## What This Changes

The issue reports that `openclaw node run --node-id <id>` does not make the custom node id usable for node listing, description, or `tools.exec.node` targeting after pairing.

## Best Possible Solution

Implement a narrow gateway-side node id resolver that treats a trimmed non-empty `connect.client.instanceId` as the explicit node id before falling back to `device.id` and then `client.id`, then apply it consistently to live registration, pairing reconciliation, pending-drain identity if appropriate, metadata updates, and node catalog behavior. Add regression coverage for a paired node host using `--node-id my-mac-node` so `nodes list`, `nodes describe --node my-mac-node`, and node invoke targeting all use the custom id without duplicate catalog rows.

## Reproduction Assessment

Yes. The static path is high-confidence: the CLI and node host send the custom id as `client.instanceId`, while current main’s registry and pairing reconciliation ignore that field when choosing the node id. I did not run a live gateway because this review was constrained to read-only inspection.

## Solution Assessment

No, current main is not the best or sufficient solution because it preserves the documented flag but discards it at the gateway identity boundary. The safer fix is a shared, tested node-id resolution helper rather than one-off changes in only `NodeRegistry.register()`.

## Repair Work Prompt

Fix the node id override bug reported in https://github.com/openclaw/openclaw/issues/61569. Current behavior: `openclaw node run --node-id my-mac-node` sends `my-mac-node` as `ConnectParams.client.instanceId`, but gateway registration and pairing reconciliation still key the node by `connect.device?.id ?? connect.client.id`, so `openclaw nodes describe --node my-mac-node` and `tools.exec.node = "my-mac-node"` are not reliable. Treat #61587 and #61719 as useful prior context but do not copy their changes blindly: they mostly changed `NodeRegistry.register()` and may miss pairing/catalog duplication. Implement a small shared resolver for node identity, preferring a trimmed non-empty `connect.client.instanceId`, then `connect.device?.id`, then `connect.client.id`. Apply it consistently where the gateway decides the node id for live registration, node pairing reconciliation, and any node-owned pending-drain/metadata paths that should use the same node id. Preserve device identity for authentication and device pairing; do not weaken device-token/security checks. Add focused regression tests proving a paired node with `instanceId: "my-mac-node"` appears and resolves under that id in `node.list`/`node.describe`, does not produce confusing duplicate catalog rows, and can be used for node invoke routing. Add a user-facing changelog fix entry in the active version with `Thanks @erhhung`. Do not change unrelated gateway protocol fields or broaden node auth behavior.
