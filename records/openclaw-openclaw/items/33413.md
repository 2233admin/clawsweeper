## Summary

Keep this open. Current main still only sets Slack native assistant thread status to the static start/clear values and routes tool-start events to reactions/draft-preview progress, not to `assistant.threads.setStatus`; the direct PR #33420 was closed unmerged, while #42555, #67129, and #73054 are adjacent customization/preview items rather than superseding fixes.

## What This Changes

Update Slack native assistant thread status during tool execution so users see tool-specific progress labels instead of only the static `is typing...` status.

## Best Possible Solution

Implement the narrow Slack-owned wiring on current main: add a Slack-local tool-name-to-human-label helper, call `ctx.setSlackThreadStatus` from Slack dispatch on tool `start` events when an assistant thread target exists and typing status is active, dedupe repeated labels, preserve the existing start/clear behavior, document the native status behavior, and add focused Slack tests for the status call sequence. Keep the configurable status/persona and draft-preview behavior discussions separate.
