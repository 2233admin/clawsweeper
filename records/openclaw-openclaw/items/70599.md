## Summary

Keep open. Current main still does not inspect latest successful tool-result text when resolving empty embedded turns, so the linked bug remains real. PR #70599 targets the right seam, but the current PR discussion identifies an unresolved regression risk: the silent-tool fast path must not override provider-error terminal handling.

## Best Possible Solution

Keep the PR open and ask for a narrow update: preserve the latest successful current-attempt tool-result NO_REPLY path, carry or otherwise derive the current-attempt boundary safely, and gate the silent-tool fast path to clean terminal states so stopReason="error" still surfaces/retries as current code expects. Add tests for the successful tool-result NO_REPLY case, stale prior-turn NO_REPLY, and provider-error-with-NO_REPLY not being treated as quiet success.
