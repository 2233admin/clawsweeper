## Summary

Keep this PR open. Current main still uses order-dependent LAN IPv4 selection for gateway dashboard/status, setup-code pairing, and the bundled device-pair plugin, so the PR is not obsolete. The discussion also shows the author already addressed review feedback for CNI prefixes and default-route preference while preserving bridge/container fallback behavior.

## What This Changes

This PR changes gateway, setup-code, and device-pair LAN IPv4 selection to prefer default-route or physical/default-looking interfaces over Docker/bridge/CNI candidates, with regression tests for those cases.

## Best Possible Solution

Land this PR or an equivalent focused fix after maintainer review: use consistent LAN address priority for gateway dashboard/status, core setup-code pairing, and device-pair; prefer default-route or physical/default-looking LAN interfaces; keep Docker/bridge/veth/CNI/container addresses as fallback candidates; and retain regression coverage for Docker bridge, CNI overlay, default-route, and bridge-default-route cases.
