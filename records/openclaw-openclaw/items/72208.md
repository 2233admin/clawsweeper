## Summary

Keep open. Current main has the non-blocking startup mitigation and documented opt-out for pricing bootstrap, but the Ctrl+C/shutdown side of the report remains actionable because the pricing refresher stop callback does not abort in-flight fetches and a completed refresh can still schedule another normal timer after shutdown has begun.

## What This Changes

The issue reports that OpenClaw 2026.4.24 on Windows npm-global hangs during gateway startup when LiteLLM/OpenRouter pricing fetches time out, leaving the dashboard unreachable and Ctrl+C ineffective.

## Best Possible Solution

Keep the shipped startup mitigation, then make the model-pricing refresh lifecycle shutdown-owned: pass a stop-controlled AbortSignal into OpenRouter/LiteLLM fetches, make refresh completion skip cache/timer scheduling after stop, and unref any background refresh timer so the gateway process can exit cleanly.

## Reproduction Assessment

Yes. The issue gives a concrete released-package repro, and current source supports a focused unit reproduction: start pricing refresh with hanging fetches, call the returned stop callback, and assert the fetch is aborted and no follow-up timer is scheduled.

## Solution Assessment

No. The current implementation is a good partial startup fix, but it is not the best complete solution because it leaves in-flight pricing fetches and post-stop scheduling outside the shutdown lifecycle.
