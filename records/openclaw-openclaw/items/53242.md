## Summary

Keep open. Current main still truncates `chat.history` display text at an 8,000-character default before the larger byte-budget checks, so the reported WebChat/Control UI behavior remains valid. The linked fix PR #53243 is still open and unmerged, so this issue should stay open until that PR or an equivalent gateway fix lands.

## What This Changes

`chat.history` should preserve long assistant text that fits the configured history and per-message byte budgets instead of truncating each text field at the default character cap.

## Best Possible Solution

Update the shared chat display projection default to raise or derive the text cap from the byte budget, preserve the existing config/RPC override semantics, keep UTF-8-heavy truncation byte-aware, add default-path regression coverage for long assistant text, and update config help/docs if the shipped default changes.
