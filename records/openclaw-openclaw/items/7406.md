## Summary

Keep this issue open. Current main captures Telegram forum topic names for inbound/prompt context, but it still does not persist those names into session metadata or expose them through gateway session rows / the Control UI session dropdown. The related implementation PRs are closed unmerged, and duplicate reports were closed back toward this canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical tracker. The best fix belongs in the bundled Telegram/session metadata path: map cached Telegram TopicName into persisted session display metadata, preserve explicit user labels/displayName overrides, keep topic-id/key fallback behavior, expose the result through sessions.list rows, and add regression coverage for topic extraction, metadata persistence, gateway session rows, and Control UI label resolution.

## Risks / Open Questions

- A fix needs to preserve user-set label/displayName precedence and only use Telegram topic names as automatic display metadata or fallback.
- Telegram topic names are learned opportunistically from service-message/cache metadata, so unknown, stale, or renamed topics need stable fallback behavior, probably using the topic id/key.
- The current Control UI can only render what sessions.list exposes, so solving this only in UI parsing would miss CLI/gateway consumers and duplicate formatting policy.
