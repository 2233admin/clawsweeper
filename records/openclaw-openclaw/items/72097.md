## Summary

Keep open. Current main still resolves plugin tools into a concrete tool array for the run/session setup and applies the Pi session allowlist once, while plugin install only refreshes persisted registry metadata and still documents Gateway restart as the runtime boundary. The existing ClawSweeper comment also kept this open, and I did not find a merged implementation or canonical duplicate that closes the active-session tool refresh gap.

## What This Changes

The issue requests that newly installed plugin-provided tools become available to already-active agent sessions without requiring a Gateway restart or a new session.

## Best Possible Solution

Add a generic core lifecycle path that re-resolves plugin-provided tools after successful plugin install or registry refresh and applies updated structured tool definitions to active sessions before the next model call, while preserving the same policy filters and avoiding duplicate plugin runtime loads. Coordinate with #44401 if restart/system-event delivery becomes part of the design, but do not treat that adjacent PR as the plugin-tool refresh fix by itself.

## Reproduction Assessment

Yes. The issue provides a concrete install -> immediate use -> Gateway restart flow, and current-main source inspection confirms install refreshes registry metadata while active session tool definitions are built and applied as a fixed setup snapshot.

## Solution Assessment

Unclear as a patch because this issue has no implementation. The requested direction is sound, but the safest maintainable shape is likely a turn-boundary re-resolution/update path rather than arbitrary mid-stream mutation of Pi session state.
