## Summary

Keep open. Current main still documents plugin runtime/tool changes as requiring a Gateway restart, and source inspection shows plugin tools are resolved into a concrete per-run tool array that is passed into the Pi session setup once. I did not find a merged implementation or canonical duplicate that refreshes active session tool definitions after plugin install; the related open restart-event PR is adjacent but does not solve tool injection.

## What This Changes

Request active agent sessions to refresh or receive newly registered plugin tools after `openclaw plugins install <pkg>` succeeds, without requiring a session or Gateway restart.

## Best Possible Solution

Add a generic core lifecycle path that re-resolves plugin-provided tools after a successful plugin install or registry refresh and applies the updated structured tool definitions at a safe turn boundary, using the same tool-policy filters as normal run setup and avoiding duplicate plugin runtime loads.

## Reproduction Assessment

Yes. The issue gives a concrete install -> immediate use -> Gateway restart comparison, and current-main source inspection confirms active runs are initialized from a fixed tool snapshot rather than a refreshable tool source.

## Solution Assessment

Unclear as a patch because this issue has no implementation. The requested direction is sound, but the safest maintainable shape is likely turn-boundary re-resolution/update rather than arbitrary mid-stream mutation of Pi session state.
