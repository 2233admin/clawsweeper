## Summary

Keep open. Current main still resolves plugin tools into a fixed tool array for a run/session and applies the Pi session tool allowlist once, while plugin install only refreshes registry metadata and still documents Gateway restart as the runtime boundary. The prior ClawSweeper comment also kept this open, and I did not find a merged implementation or canonical duplicate that closes the active-session tool refresh gap.

## What This Changes

Request active agent sessions to refresh plugin-provided tool definitions after successful plugin install so newly installed tools can be used without restarting the Gateway or starting a new session.

## Best Possible Solution

Add a generic core lifecycle path that re-resolves plugin-provided tools after a successful plugin install or registry refresh and applies the updated structured tool definitions to active sessions before the next model call, while preserving the same policy filters and avoiding duplicate plugin runtime loads.

## Reproduction Assessment

Yes. The issue provides a concrete install -> immediate use -> Gateway restart flow, and current-main source inspection confirms install refreshes registry metadata while active session tool definitions are built and applied as a fixed snapshot during run/session setup.

## Solution Assessment

Unclear as a patch because this issue has no implementation. The requested direction is sound, but the safest maintainable shape is likely a turn-boundary re-resolution/update path rather than arbitrary mid-stream mutation of Pi session state.
