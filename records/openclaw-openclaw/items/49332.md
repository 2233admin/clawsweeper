## Summary

Keep this PR open. Current main still lacks a completed chat-run dedupe path for retransmitted terminal Control UI chat events, so the PR is not obsolete or implemented on main. The current PR branch should not merge as-is because it mixes the UI fix with unrelated runtime/provider/Discord/ops changes and review comments identify correctness gaps in the proposed dedupe implementation.

## Best Possible Solution

Keep this PR open for maintainer follow-up, but replace or revise the branch into a minimal UI-only change. The fix should add a typed completed-run dedupe mechanism for terminal chat events, cover own-run `final`/`aborted` and other-run/sub-agent final paths, preserve dedupe state across same-session history reloads, clear it on real session/agent switches, and add focused regression coverage in the Control UI chat controller/gateway event tests. The unrelated Google Vertex, Discord, sandbox, AGENTS, and gaxios changes should be removed or handled in separate reviewed PRs.

## Risks / Open Questions

- Current main still appears vulnerable to duplicate Control UI assistant messages if the same renderable `final` event is delivered again after `chatRunId` has been cleared.
- Merging this PR as-is would mix a narrow UI bug fix with unrelated provider auth, Discord monitor, sandbox policy, gaxios compatibility, and repo-instruction changes.
- The PR's process-wide `globalThis.window` shim is security/supply-chain sensitive because it changes browser-environment detection for every later dependency in the Node process.
- The review comments identify gaps in the PR's dedupe logic for other-run/sub-agent final events and history reload timing, so the UI fix needs revision and focused tests.
