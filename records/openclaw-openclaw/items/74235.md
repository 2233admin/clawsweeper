## Summary

Keep open. Current main still has the Google Chat threading gaps this PR targets, but the latest diff still appears to introduce blocking reply-mode and cross-channel threading regressions, so it needs branch revision and maintainer review before merge.

## What This Changes

The PR carries Google Chat inbound thread resource names through webhook context, reply delivery, routed transport, and message-tool sends, adds Google Chat thread-resource validation, and expands core plus Google Chat regression tests.

## Best Possible Solution

Land a revised PR that keeps shared reply plumbing channel-agnostic, preserves `currentMessageId` as the generic implicit reply target, and performs Google Chat resource-name normalization inside the Google Chat plugin boundary while respecting `replyToMode` across natural delivery, routed delivery, and message-tool sends. After that lands, reconcile #39554, #69422, and #67055 against the shipped fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep implicit replies on current messages by default:** `src/auto-reply/reply/agent-runner-utils.ts:166`
  - body: `resolveAutoThreadingTargets` promotes any `MessageThreadId` to the implicit `replyToId`. Telegram uses `MessageThreadId` for topic ids while its delivery parses `replyToId` as a message id, so forum or DM-topic replies can reply to the topic id instead of the inbound message. Keep this Google Chat-specific or route it through a typed transport/thread hook.
  - confidence: 0.84
- **[P2] Honor off mode before falling back to inbound threads:** `extensions/googlechat/src/monitor-reply-delivery.ts:23`
  - body: When `replyToMode: "off"` strips `payload.replyToId`, this fallback still returns the inbound Google Chat thread. That makes normal replies and media sends land in-thread even though the account asked for unthreaded replies; only use the inbound thread when implicit threading is allowed or replacing an explicit message-resource target.
  - confidence: 0.88
- **[P2] Gate routed thread fallback on reply mode:** `extensions/googlechat/src/channel.adapters.ts:143-144`
  - body: `resolveReplyTransport` promotes a valid `threadId` to `replyToId` even after the shared reply-mode filter has removed the payload target. Routed or follow-up replies for `replyToMode: "off"` accounts can become threaded again, so this fallback needs to check the resolved account reply mode.
  - confidence: 0.8
- **[P2] Respect off mode in message-tool defaults:** `extensions/googlechat/src/actions.ts:132-134`
  - body: The message action now defaults omitted `threadId`/`replyTo` to `toolContext.currentThreadTs`, but the tool context does not carry Google Chat reply mode. A `message` tool send with no explicit target will still post in the inbound thread for off-mode accounts.
  - confidence: 0.78
