## Summary

Keep this PR open. Current main still has the Google Chat thread-preservation gap the PR is trying to fix, but the current branch still needs revision because its new inbound-thread fallbacks can bypass `replyToMode: "off"` after shared reply filtering.

## What This Changes

The PR carries Google Chat thread resource names through webhook context, reply delivery, routed transport, and the message action path, adds thread-resource validation, and expands core plus Google Chat tests.

## Best Possible Solution

Revise the branch so Google Chat uses valid inbound thread resources for normal threaded replies and explicit message-resource tag swaps, while honoring `replyToMode: "off"` for implicit fallbacks in direct delivery, routed/follow-up delivery, and message-tool sends. Add off-mode regression coverage, then land and reconcile #39554, #69422, and #67055.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor off mode before using the inbound thread:** `extensions/googlechat/src/monitor-reply-delivery.ts:38`
  - body: `replyToMode: "off"` removes `payload.replyToId` in the shared reply filter, but this new fallback threads any payload that has a valid `inboundThreadId`. That makes normal Google Chat replies land in-thread even when the account config asks for unthreaded replies; only use the inbound thread when threading is allowed or when replacing an explicit message-resource target.
  - confidence: 0.88
- **[P2] Preserve off mode in routed Google Chat replies:** `extensions/googlechat/src/channel.adapters.ts:143-144`
  - body: `resolveReplyTransport` now promotes a valid Chat `threadId` to `replyToId` without considering account reply mode. Routed and follow-up replies can pass the session `MessageThreadId` after the payload reply target was filtered for `replyToMode: "off"`, so gate the `threadId` fallback on the resolved reply mode while preserving explicit thread-shaped `replyToId` values.
  - confidence: 0.82
- **[P2] Gate message-tool thread defaults on reply mode:** `extensions/googlechat/src/actions.ts:125-134`
  - body: The Google Chat message action now defaults omitted `threadId`/`replyTo` to `toolContext.currentThreadTs`, but the shared message action code suppresses implicit replies when `replyToMode` is off. This bypass sends tool messages into threads for off-mode accounts; consult the tool context reply mode before applying the inbound thread default.
  - confidence: 0.74
