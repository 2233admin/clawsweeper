## Summary

Keep open. Current main still has the Google Chat thread-routing gap this PR targets, but the proposed patch is not merge-ready because it adds shared and Google Chat-specific fallbacks that can bypass `replyToMode: "off"` and can reinterpret non-Google-Chat `MessageThreadId` values as reply message IDs.

## What This Changes

The PR adds Google Chat thread-resource validation and carries inbound thread metadata through webhook context, reply delivery, routed transport, message actions, shared auto-threading helpers, and regression tests.

## Best Possible Solution

Revise the branch so Google Chat normalizes valid thread resources inside the plugin boundary, only falls back to inbound threads when implicit threading is allowed or an explicit message-resource target needs replacement, and keeps existing `currentMessageId` reply semantics for other channels. Add off-mode and non-Google-Chat regression coverage, then land and reconcile #39554, #69422, and #67055.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep generic replies on the current message id:** `src/auto-reply/reply/agent-runner-utils.ts:164-170`
  - body: `resolveAutoThreadingTargets` now promotes any `MessageThreadId` to the implicit `replyToId`. Telegram stores topic ids in `MessageThreadId` while its delivery treats `replyToId` as a message id, so forum-topic replies can stop replying to the inbound message and pass the topic id as `replyToMessageId`. Keep this alternate target channel-specific or add a typed adapter hook.
  - confidence: 0.82
- **[P2] Honor off mode before using the inbound thread:** `extensions/googlechat/src/monitor-reply-delivery.ts:24`
  - body: When `replyToMode: "off"` strips `payload.replyToId`, this fallback still sends `params.inboundThreadId`. That makes normal Google Chat replies, media sends, or fallback sends land in-thread even when the account asked for unthreaded replies; only use the inbound thread when threading is allowed or when replacing an explicit message-resource target.
  - confidence: 0.88
- **[P2] Preserve off mode in routed Google Chat replies:** `extensions/googlechat/src/channel.adapters.ts:143-144`
  - body: `resolveReplyTransport` promotes a valid `threadId` to `replyToId` without checking the resolved reply mode. Routed and follow-up replies whose payload target was filtered out for `replyToMode: "off"` can be threaded again through the session thread id, so gate this fallback on the account reply mode.
  - confidence: 0.78
- **[P2] Gate message-tool thread defaults on reply mode:** `extensions/googlechat/src/actions.ts:125-134`
  - body: The message action now defaults omitted `threadId`/`replyTo` to `toolContext.currentThreadTs`, but the Google Chat tool context does not carry a reply mode and shared off-mode suppression runs before this plugin action. A `message` tool send with no explicit target will still post in the inbound thread for off-mode accounts.
  - confidence: 0.8
