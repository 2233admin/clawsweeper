## Summary

Keep open. Current main still has the diagnostic failure described in the report: embedding provider exceptions propagate to the shared CLI wrapper, and the default wrapper still prints `String(err)`, which drops nested `fetch` causes and `UND_ERR_*` codes. The Voyage path appears to use the shared guarded remote embedding helper without a custom `fetchImpl`, so the remaining fix is narrow error preservation rather than a Voyage-specific fetch rewrite.

## What This Changes

The issue requests that `openclaw infer embedding create --provider voyage` avoid Node 24/global-fetch and runtime-Undici dispatcher mixing and surface the underlying `fetch` cause when provider embedding requests fail.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP helper cause-aware while keeping the existing guarded fetch, runtime Undici, and SSRF policy architecture intact. The desired end state is that transport failures include provider context plus nested cause/code details, and a credentialed Node 24 Voyage smoke confirms whether any separate transport bug remains.

## Reproduction Assessment

Unclear for the live Voyage transport failure because this runner has Node v25.9.0 and no visible Voyage credential. Yes for the diagnostic gap: a provider rejection shaped like `TypeError('fetch failed')` with a nested Undici cause still reaches `runtime.error(String(err))` and loses the nested cause/code.

## Solution Assessment

Yes. A narrow error-preservation change in `runCommandWithRuntime` and/or `postJson`/`fetchRemoteEmbeddingVectors` is the best first fix; changing provider auth, model defaults, Undici versions, or SSRF policy would be premature unless the live smoke still fails after the cause is visible.
