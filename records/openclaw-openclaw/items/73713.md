## Summary

Keep open. Current main still leaves the reported diagnostic failure mode in place: embedding provider errors propagate into the generic CLI wrapper, and that wrapper prints `String(err)`, which loses nested fetch causes such as Undici `UND_ERR_*` details. The Voyage path itself appears to use the shared guarded remote embedding helper without a provider-local `fetchImpl`, so the next fix should be narrow and cause-preserving rather than a Voyage auth or model rewrite.

## What This Changes

The issue asks OpenClaw to audit and fix the Node 24 Voyage embedding CLI path so valid embedding requests succeed and, on fetch failure, surface the underlying fetch cause/code instead of only `TypeError: fetch failed`.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP path cause-aware while preserving the existing guarded fetch and SSRF architecture. The end state should surface nested fetch cause messages and stable codes such as `UND_ERR_INVALID_ARG`, keep Voyage on the shared helper, and include focused regression coverage plus an optional credentialed Node 24 Voyage smoke.

## Reproduction Assessment

Unclear for the live Voyage transport failure because this review environment has Node 24 but no `VOYAGE_API_KEY`. Yes for the diagnostic gap: a provider rejection shaped like `TypeError('fetch failed')` with a nested Undici cause will still be printed through `String(err)` only.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing cause-aware formatter and/or wrap remote embedding transport exceptions with provider context; changing Voyage auth, model selection, dispatcher construction, or SSRF policy would be premature without the underlying cause first being visible.
