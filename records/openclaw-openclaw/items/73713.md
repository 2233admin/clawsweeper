## Summary

Keep open. Current main still matches the reported diagnostic gap: `infer embedding create` lets Voyage embedding fetch failures bubble to the generic CLI wrapper, and that wrapper still prints `String(err)`, which loses nested fetch causes and Undici codes. The Voyage provider itself uses the shared guarded remote embedding path without a Voyage-local custom fetch override, so this remains a narrow CLI/shared remote-HTTP error-preservation bug rather than a closeable core cleanup item.

## What This Changes

The issue asks OpenClaw to audit the Voyage `infer embedding create` path for Node 24 fetch/Undici dispatcher mixing and to preserve the underlying fetch cause when embedding provider requests fail.

## Best Possible Solution

Make the embedding CLI and shared remote embedding HTTP helper cause-aware while preserving the existing SSRF guard and runtime-dispatcher fallback. The desired end state is provider-contextual, redacted error text that includes nested fetch cause messages and stable codes, with focused regression tests for an Undici-shaped `TypeError('fetch failed')`.

## Reproduction Assessment

Partially yes. The live Voyage/Node 24 transport failure still needs a credentialed smoke, but the diagnostic loss is reproducible from source by throwing a `TypeError('fetch failed')` with a nested Undici-style cause through `embedBatch` and observing that the default CLI wrapper prints only `String(err)`.

## Solution Assessment

Yes. The narrow maintainable fix is to use the existing redacted error formatter in the CLI wrapper and/or wrap pre-response remote embedding fetch failures with provider context and preserved cause/code text; changing provider auth, Undici pins, or SSRF policy would be premature without the recovered cause.
