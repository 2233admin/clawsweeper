## Summary

Keep open. Current main still preserves the reported failure mode for diagnostics: `openclaw infer embedding create` lets provider exceptions reach the shared CLI wrapper, and that wrapper still prints `String(err)`, which drops nested fetch causes. The Voyage path does appear to use the shared guarded remote embedding helper without a custom `fetchImpl`, so the remaining actionable work is a narrow cause/code-preserving error path plus a live smoke when credentials are available.

## What This Changes

The issue asks to audit the `openclaw infer embedding create` Voyage provider path on Node 24 and surface nested fetch/Undici causes instead of only `TypeError: fetch failed`.

## Best Possible Solution

Make the embedding CLI and remote embedding HTTP helper cause-aware while preserving the existing guarded fetch and SSRF policy architecture. The fix should surface nested fetch cause messages and stable error codes such as `UND_ERR_INVALID_ARG`, add focused regression tests for `TypeError('fetch failed')` with a nested Undici cause, and then run a credentialed Node 24 Voyage smoke if `VOYAGE_API_KEY` is available.

## Reproduction Assessment

Unclear for the live Voyage transport failure; it needs a Node 24 environment plus a valid credential. Yes for the diagnostic gap: a provider or `postJson` rejection shaped like `TypeError('fetch failed')` with an `UND_ERR_INVALID_ARG` cause will currently be printed through `String(err)` only.

## Solution Assessment

Yes. A narrow error-preservation change in `runCommandWithRuntime` and/or the remote embedding fetch helper is the maintainable first fix; changing provider auth, model selection, dispatcher construction, or SSRF policy would be premature unless the live smoke still fails after the underlying cause is visible.
