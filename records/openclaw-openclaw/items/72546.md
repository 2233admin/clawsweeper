## Summary

Keep open. Current main still has a plausible cron-ok/no-artifacts failure mode: the memory-core managed dreaming path writes reports/events inside per-workspace try blocks, but catches workspace failures, logs them, and still returns a handled processed result. The existing docs and writers confirm artifacts/events are expected, while the REM harness only proves eligible input, not successful mutation.

## What This Changes

The issue reports that on OpenClaw 2026.4.24 the memory-core managed dreaming cron reports `ok` with eligible REM/deep material but writes no new `DREAMS.md`, `memory/dreaming/*`, or `memory.dream.completed` artifacts after April 18, 2026.

## Best Possible Solution

Make the memory-core managed dreaming sweep enforce an observable success invariant: with enabled phases and eligible workspaces, expected report files and `memory.dream.completed` events should be written, and an all-workspace failure before artifact/event writes should surface as a failed or clearly diagnosable cron result instead of an ok-looking processed run.

## Reproduction Assessment

Yes for the reported install path: the issue includes cron history, non-empty rem-harness/deep candidate counts, memory status, and artifact/event inspections. Current-main live reproduction is unclear because this review was intentionally read-only and did not run the mutating cron path.

## Solution Assessment

Not applicable yet because there is no proposed patch. The narrowest maintainable fix is a memory-core regression test around cron-triggered separate-storage artifact writes plus failure propagation when every workspace fails before any expected artifact/event is written.
