## Summary

Keep open. Current main has the intended happy-path artifact writes and the earlier fast no-op cron issue is separately fixed, but the current managed dreaming handler can still catch per-workspace/artifact failures, log them, and return a handled success, which leaves a plausible current path for an `ok` cron run with no new dreaming artifacts or completion events.

## What This Changes

Request that the managed memory dreaming cron write `DREAMS.md`, `memory/dreaming/<phase>/YYYY-MM-DD.md`, and `memory.dream.completed` events when eligible REM/deep material exists instead of reporting success with no visible outputs.

## Best Possible Solution

Make the managed memory-core dreaming cron only report success when the sweep either writes the expected phase artifacts/events or emits an explicit no-work diagnostic consistent with the storage contract. If a workspace sweep or artifact/event write fails, the cron result should surface a non-ok status or actionable failure instead of only logging `failedWorkspaces`.
