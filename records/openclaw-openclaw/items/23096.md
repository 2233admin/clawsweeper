## Summary

Keep this PR open, but do not merge it as-is. Current main still has no Bitwarden/Vaultwarden SecretRef support and the discussion identifies this as the focused candidate, but the current diff introduces a security-sensitive wrapper that resolves the real Bitwarden CLI through PATH after OpenClaw validates only the wrapper script; maintainers also still need to choose the plugin-seam route from #72548 or a deliberately hardened bundled exec-wrapper route coordinated with #29110.

## What This Changes

The PR adds Bitwarden/Vaultwarden SecretRef documentation, a protocol-v1 Bitwarden CLI exec resolver script with TypeScript declarations, and tests for parsing, field extraction, grouping, and child-process resolution.

## Best Possible Solution

Land Bitwarden/Vaultwarden support only after the route is explicit: preferably as a secret-provider plugin once #72548 is available, or as a hardened bundled exec-wrapper/docs integration that validates the secondary Bitwarden binary path, documents trusted installation paths, keeps error redaction tight, and is coordinated with the related Bitwarden Secrets exec work in #29110.

## Reproduction Assessment

Not applicable as a feature PR rather than a bug report. For the blocking review finding, yes: the diff shows execFile("bw") and the docs pass PATH into the wrapper, so putting a different bw earlier in PATH would exercise the unsafe binary-selection path.

## Solution Assessment

No, not as currently proposed. The exec-wrapper approach is plausible, but it must not resolve the Bitwarden CLI through unvalidated PATH, and maintainers still need to decide whether the plugin seam is the narrower long-term home.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Resolve the Bitwarden CLI without PATH search:** `scripts/bw-exec-resolver.mjs:27`
  - body: The configured exec provider validates only this wrapper script, but the wrapper then calls `execFile("bw", ...)` while the docs pass through `PATH`. A writable or overridden PATH can cause OpenClaw to run a different `bw` binary with the Bitwarden session/client env, bypassing the current exec-provider path hardening. Require an absolute and validated Bitwarden binary path, such as a `BW_BIN` override checked against trusted dirs, before spawning it.
  - confidence: 0.86
