## Summary

Keep open. Current main still lacks the PR's global/per-agent sender tool policy surface and channel-scoped sender key support, so the work is not obsolete; however, the proposed patch changes a security-sensitive tool-capability boundary and still needs maintainer review plus a fix for canonical channel/provider matching before merge.

## What This Changes

The PR adds global and per-agent `toolsBySender` config, extends sender keys with `channel:<provider>:<senderId>`, and threads a sender-specific tool policy through core, plugin allowlist, and bundled tool filtering paths.

## Best Possible Solution

Keep this PR open and finish it in the contributor branch if maintainers accept the product direction: canonicalize channel/provider identities before sender-key matching, document the no-sender behavior, add focused regression tests across core/plugin/bundled filtering, and align generated config metadata before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Canonicalize providers before sender key matching:** `src/config/group-policy.ts:248-249`
  - body: The new channel-scoped lookup lowercases `messageProvider` but does not run it through OpenClaw's channel canonicalizer. Registered aliases such as `teams` normalize to `msteams` elsewhere, so a policy like `channel:msteams:<senderId>` can fail to match a run delivered with the alias and fall through to a less restrictive sender or wildcard policy. Canonicalize the provider before building the `channel:<provider>:<senderId>` lookup key.
  - confidence: 0.87

## Security Review

Status: needs_attention

Summary: The diff introduces no dependency/workflow supply-chain changes, but it modifies a security-sensitive tool authorization boundary and leaves channel alias matching inconsistent.

Concerns:

- **[medium] Channel alias mismatch can bypass scoped sender policy:** `src/config/group-policy.ts:248`
  - body: Channel-scoped sender policies are meant to restrict tools by platform and sender, but the proposed matcher uses the raw provider string instead of the canonical channel id. If a runtime supplies an alias, the intended `channel:<canonical>:<senderId>` restriction can miss and fall through to a less restrictive policy.
  - confidence: 0.86
