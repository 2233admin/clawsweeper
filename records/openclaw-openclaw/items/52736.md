## Summary

Keep open. Current main has the attachment groundwork for Control UI chat uploads, paste/drop, previews, removal, and chat.send serialization, but it still does not implement the requested screenshot/scissors composer button or a screen/region capture flow. The existing node screen.snapshot capability is adjacent infrastructure, not a browser Control UI attachment workflow, and the related title-search items are adjacent Control UI requests rather than a canonical duplicate.

## Best Possible Solution

Implement this as a focused Control UI composer enhancement after maintainer product review of the capture path: add a screenshot/scissors button beside the existing paperclip, capture via an approved browser getDisplayMedia flow or native bridge, convert the captured image into the existing ChatAttachment payload store, reuse preview/removal/chat.send.attachments behavior, add targeted UI/controller tests, and update the public Control UI docs.

## Risks / Open Questions

- Screen capture can expose sensitive local pixels, so the intended permission, preview, removal, and send-confirmation behavior should be explicit before implementation.
- Browser getDisplayMedia captures a screen/window/tab stream, while true arbitrary region capture may require a crop UI or native bridge; maintainers should choose the product path before queuing an autonomous fix.
- The implementation should reuse the existing ChatAttachment payload store, preview/removal UI, and chat.send serialization instead of adding a parallel attachment path.
