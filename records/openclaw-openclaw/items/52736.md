## Summary

Keep open. Current main has the Control UI attachment groundwork for uploads, paste/drop, previews, removal, and chat.send serialization, but it still lacks the requested screenshot/scissors button and screen or region capture flow in the chat composer. The existing node-level screen.snapshot command is adjacent infrastructure, not a Control UI composer attachment workflow, and the related items in the provided context are adjacent Control UI/file/button requests rather than a canonical duplicate.

## Best Possible Solution

Implement this as a focused Control UI chat composer enhancement: add a screenshot/scissors button beside the existing paperclip, capture a screen/window/region through an approved browser or native bridge path, convert the captured image into the existing ChatAttachment payload store, reuse preview/removal/chat.send.attachments behavior, add targeted UI/controller tests, and update the public Control UI docs when the workflow exists.

## Risks / Open Questions

- Browser getDisplayMedia, OS-native screenshot bridges, and region selection have different permission and platform behavior; the implementation should choose the intended capture path explicitly.
- Screen capture can expose sensitive pixels, so the preview/remove step and existing attachment validation should remain clear before sending.
- A new screenshot path should reuse the existing ChatAttachment payload store, preview, removal, and chat.send serialization instead of creating a parallel attachment pipeline.
