## Summary

Keep open. Current main still has the ACP runtime-control behavior this PR is trying to change, so the PR is not obsolete; however, the proposed patch also removes caller-visible unsupported-key errors for backends that do advertise a strict key list, which is a correctness regression for the current contract.

## What This Changes

This PR changes ACP manager runtime controls to send persisted runtime config options only when a backend reports `configOptionKeys`, skipping all config options when no keys are advertised.

## Best Possible Solution

Land a tested ACP runtime-control change that prevents strict adapters from receiving unsupported generic config IDs while keeping the capability contract explicit and consistent across pre-turn option application and manual `/acp set` writes. The final shape should include regression tests for undefined `configOptionKeys`, advertised-key subsets, and any intentional warning or error behavior when persisted controls are ignored.

## Reproduction Assessment

Yes. Source inspection and existing tests show current main forwards all persisted config options when a backend supports `session/set_config_option` but omits `configOptionKeys`; the linked bug report adds a concrete Claude ACP reproduction where those generic keys are rejected before output.

## Solution Assessment

No. The PR addresses the reported no-key-list failure path, but it is not the best complete fix because it silently ignores unsupported persisted options for advertised-key backends and contradicts the current capability type documentation without tests or a contract update.

## Risks / Open Questions

- The intended contract for undefined or empty `configOptionKeys` needs maintainer confirmation because current source documentation says that case is permissive, while this PR treats it as skip-all.
- The PR has no regression coverage for either an undefined key list or an advertised key list containing only a subset of persisted runtime options.
