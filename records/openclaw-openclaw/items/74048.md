## Summary

Keep open. Current main still forwards all generated ACP runtime config options when a backend supports `session/set_config_option` but does not advertise `configOptionKeys`, so the linked Claude ACP failure is not obsolete. The PR is directionally relevant, but it needs maintainer review because it changes the advertised-key path from a clear unsupported-control error to silent filtering and has no regression tests.

## Best Possible Solution

Keep this PR open for maintainer review. The best fix is a tested ACP runtime-control change that defines the capability contract explicitly, avoids sending generic unsupported config IDs to strict Claude ACP paths, preserves or intentionally revises the unsupported-key error behavior for advertised key lists, and updates tests/docs/types if undefined `configOptionKeys` is no longer permissive.

## Risks / Open Questions

- The PR silently filters unsupported keys for backends that do advertise `configOptionKeys`, replacing the current `ACP_BACKEND_UNSUPPORTED_CONTROL` error path and potentially hiding ignored `approval_policy` or timeout settings.
- The PR has no regression coverage for missing `configOptionKeys`, advertised subset filtering, or the current unsupported-control contract.
- The proposed skip-all behavior conflicts with the current runtime type comment that treats undefined or empty `configOptionKeys` as permissive.
- Skipping all options when keys are undefined avoids strict Claude failures, but may drop useful `model` or `thinking` controls for adapters that accept generic keys without advertising a list.
