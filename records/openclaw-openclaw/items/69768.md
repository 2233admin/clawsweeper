## Summary

Keep open. Current main still allows JSON Schema conditional keywords to reach Gemini tool schemas, so the reported provider rejection path is not fixed; the remaining action is to review and land the already-open focused fix PR or an equivalent patch.

## What This Changes

The issue reports Gemini Flash cron tool-calling payload failures with HTTP 400 no-body errors and asks whether OpenClaw should sanitize the tool schema path and document safer cron/failure-notification guidance.

## Best Possible Solution

Land #70497 or an equivalent focused patch that adds `if`, `then`, and `else` to the Gemini unsupported-keyword set with regression coverage, while keeping the existing cron failure-destination docs and avoiding broader failover policy changes unless separately reproduced.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a schema containing `if`/`then`/`else` through `cleanSchemaForGemini` or through `normalizeToolParameterSchema` with `modelProvider: "google"`; current main preserves those keys because the cleaner only strips names in its unsupported-keyword set.

## Solution Assessment

Yes. The narrow schema-cleaning fix is the best path because it prevents Gemini from receiving unsupported conditional keywords at the provider boundary; docs or failover-only changes would still leave the invalid tool schema serialization in place.
