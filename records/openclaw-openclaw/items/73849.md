## Summary

Keep this PR open, but it should not merge as-is. The launchd plist change does not solve the reported crash-loop case because `SuccessfulExit=false` still restarts unsuccessful exits, and it also breaks current main's supervised clean-exit relaunch path; the branch also carries unrelated sandbox Docker changes with a concrete supply-chain/runtime regression.

## What This Changes

The PR changes the macOS LaunchAgent plist, launchd audit check, and launchd test from boolean `KeepAlive=true` to `KeepAlive` with `SuccessfulExit=false`, while also modifying sandbox Docker/browser/doctor behavior for Docker daemon and image availability.

## Best Possible Solution

Replace this branch with a focused daemon-only fix for #73673 that preserves explicit `openclaw gateway restart`, update restarts, and supervised clean-exit relaunches while adding a deliberate crash-loop debugging/recovery policy. Keep sandbox Docker daemon/image behavior out of that PR and preserve the documented no-silent-Debian-substitution contract.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: under current main, launchd-supervised restart/update paths return `mode: "supervised"` and exit with status 0 expecting `KeepAlive=true` to relaunch, while `SuccessfulExit=false` would not relaunch those clean exits and would still relaunch non-zero crash exits.

## Solution Assessment

No. The proposed plist primitive is too blunt for the reported bug: it still restarts crashes, breaks clean supervisor relaunches, and mixes in unrelated sandbox Docker behavior; the safer path is a focused launchd lifecycle change with tests for crash-loop handling and intentional restarts.
