## Summary

Keep open. Current main still lacks this archive export/import workflow, but the PR is broad, security-sensitive, overlaps the newer provider-based `migrate` namespace, and still has concrete correctness/security issues that should block merge until maintainer scope is settled.

## What This Changes

The PR branch adds archive-based `openclaw migrate export` and `openclaw migrate import` commands with selective components, secret stripping, path remapping, config merge, CLI registration, import/export infrastructure, and tests.

## Best Possible Solution

Choose the recovery model first. If core stays conservative, evolve `backup` with safer platform targeting, archive conversion, verification, and clear manual restore docs. If OpenClaw owns automated import, integrate it with the existing provider-based migrate namespace and require preview-first UX, strong canonical target checks, rollback/conflict handling, and security regression tests before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Canonicalize targets before destructive writes:** `src/infra/migrate-import.ts:328-333`
  - body: `assertTargetWithinRoot` only compares `path.resolve` strings, so a symlinked existing parent such as `~/.openclaw/agents` can pass validation while the later `fs.rm`/copy operates outside the OpenClaw state tree. Resolve existing target ancestors or reject symlinked target parents before any destructive import write.
  - confidence: 0.89
- **[P2] Do not export default workspace for unknown agents:** `src/commands/migrate-shared.ts:177-180`
  - body: When `--agents` contains an ID that is not in `agents.list`, this branch still adds the default workspace because the requested agent lacks an explicit workspace. That makes `openclaw migrate export --agents typo --include workspace` include unrelated workspace data despite matching no configured agent.
  - confidence: 0.88
- **[P2] Preserve existing migrate provider commands:** `src/cli/program/register.migrate.ts:45-58`
  - body: Current main already uses `openclaw migrate` for plugin-owned provider imports with `list`, `plan`, and `apply`. This registration defines the command as archive export/import only, so after rebase it would conflict with or drop the existing Claude/Hermes migration surface.
  - confidence: 0.82
