## Summary

Current main does not implement this PR's intended OpenClaw-to-OpenClaw `migrate export` / `migrate import` workflow. The existing `openclaw migrate` command is now a plugin-provider import surface for other agent systems, while same-product machine moves are documented as manual state/workspace copy plus `doctor`. The PR also remains product- and security-sensitive: it adds untrusted archive import/restore behavior, and maintainer discussion explicitly raised restore as a large attack surface and suggested a narrower backup/convert direction instead. This should stay open for maintainer product/security review or be replaced by a scoped backup-focused PR, not auto-closed by cleanup.

## Best Possible Solution

Keep this PR open pending maintainer direction. The best path is likely to split the safe export improvements into the existing `backup` surface, consider a platform-targeted `backup convert` command, and only pursue any restore/import behavior after an explicit security design for trusted archives, confirmation, containment, backups, and recovery semantics. The existing linked issues (#45993, #43220, #13616) can remain the product discussion anchors.
