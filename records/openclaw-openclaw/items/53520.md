## Summary

Keep open. Current main has adjacent work, but it does not implement this PR's intended cross-device `openclaw migrate export` / `openclaw migrate import <archive>` workflow. The existing `migrate` command is a provider-based import path for Claude/Hermes and other plugin-owned sources, while `backup` can create and verify local backup archives but has no restore/import command. The PR also remains security-sensitive: review comments and maintainer discussion specifically call out archive import/path traversal/symlink/destructive restore risks and question whether restore belongs under `migrate` at all versus backup-target/convert or user-driven restore.

## Best Possible Solution

Keep this PR open for maintainer product/security review. The best path is for maintainers to choose the intended scope first: either rework this into narrower `backup` enhancements such as platform-targeted export or `backup convert`, or design a carefully gated restore/import flow using existing archive safety helpers, preview/backup/conflict preflight, explicit trust prompts, and focused tests. Current main does not already solve the PR's central export/import request.
