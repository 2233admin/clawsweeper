## Summary

Keep open. Current main mitigates the reporter’s original default-600s case by defaulting the agent runtime timeout to 48 hours, and it now has a plain timeout payload path, but the core misclassification remains: non-compaction run timeouts still have no tool-execution timeout cause and can still enter assistant profile rotation/model fallback as `timeout`.

## Best Possible Solution

Add phase-aware timeout state in the embedded runner, preferably a small `timeoutCause` enum or `timedOutDuringToolExecution` flag derived from the existing item lifecycle active count. When the global run timer fires during active tool execution, return a run/tool timeout or partial-progress result without rotating auth profiles or falling back models. Keep model idle/provider request timeouts on the existing retry/fallback path, and add regression coverage for the failover decision and an embedded attempt with an active tool item at timeout.

## Risks / Open Questions

- The original no-custom-config 600s reproduction is mitigated by the current 48-hour default, so follow-up should reproduce with an explicit shorter run timeout or a long run that actually hits the global runtime timer while a tool item is active.
- Changing the failover path incorrectly could suppress legitimate model idle/provider request timeout retry and fallback behavior, which should remain eligible for same-model retry, auth-profile rotation, or model fallback where appropriate.
- A fix should distinguish model request/idle timeout from run timeout during active tool execution, and should add focused failover-policy plus embedded-runner coverage so this does not regress.
