## Summary

Keep this open. Current main has useful startup, runtime-deps, readiness, and stuck-session mitigations, but the three reported root causes are not proven fixed: the Manifest/clawrouter listener source is not present in this checkout, session-store persistence still holds the store lock through durable atomic writes, and plugin restart cleanup remains limited to registered host hooks rather than arbitrary process-level listeners or sockets.

## What This Changes

The issue reports a gateway restart leak cluster where Manifest port rebinding, signal-handler accumulation, and slow session JSONL persistence can combine to starve WebSocket handshakes while shallow HTTP health checks continue to pass.

## Best Possible Solution

Split the remaining work into maintainer-owned fixes that keep the current mitigations while removing root causes: make the Manifest/clawrouter listener startup idempotent, add or require a generic plugin lifecycle cleanup/dedup surface for process-level resources, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes.

## Reproduction Assessment

Unclear for the full triad: the report provides concrete logs, probes, and independent confirmations, but there is no deterministic repository-local repro. The session-lock/WS-starvation portion has stronger evidence than the Manifest-specific port loop because the listener source is not present in this checkout.

## Solution Assessment

No, current main is not the best complete solution yet. The existing changes reduce several startup and readiness failure modes, but they do not fully address Manifest listener idempotence, arbitrary process-listener cleanup, or durable session-write lock scope.
