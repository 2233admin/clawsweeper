## Summary

Keep open. Current main contains useful mitigations for gateway startup readiness, session-lane recovery, and supervised EADDRINUSE loops, but the reported root causes are not fully removed: durable session-store writes still happen inside the session-store lock, plugin replacement cleanup remains limited to registered host hooks, and the reported Manifest/clawrouter listener bootstrap is not present in the checked-in source. Follow-up comments also confirm the same WebSocket starvation shape on Linux without Manifest, so this is still active gateway lifecycle/performance work.

## What This Changes

The issue reports a three-part gateway restart regression where Manifest port rebinding, process signal listener accumulation, and long session JSONL/session-store lock holds combine to starve local WebSocket handshakes while HTTP health stays responsive.

## Best Possible Solution

Split the remaining work into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, add a generic plugin lifecycle cleanup/dedup contract for process-level resources, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes while preserving the intended durability guarantees.

## Reproduction Assessment

Unclear: the full triad was not reproduced live here, but the issue includes concrete probes for listener counts, same-PID port ownership, session-lock warnings, and WebSocket timeout logs. The follow-up comments provide high-confidence reproduction of the broader non-Manifest WebSocket starvation/session-lock symptom class.

## Solution Assessment

No: current main is not a complete solution because it mitigates symptoms and adds recovery/readiness handling but does not remove all reported root causes. The safer fix is staged subsystem work across Manifest listener ownership, plugin lifecycle cleanup, and session persistence lock scope.
