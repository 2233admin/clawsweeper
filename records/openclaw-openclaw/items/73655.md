## Summary

Keep open. Current main has useful mitigations and diagnostics for Gateway readiness, client handshake timing, stuck-session recovery, and supervised EADDRINUSE restart loops, but the reported root causes are not fully resolved: the Manifest/clawrouter listener implementation is not present in the checked-in source, session-store persistence still runs under the store lock through durable fsync work, and plugin replacement cleanup remains limited to registered host-hook cleanup rather than arbitrary process listeners or sockets.

## What This Changes

The issue requests fixes for a Gateway restart/runtime leak triad: Manifest port rebinding on 2099/2100, process signal-listener accumulation, and slow session JSON/session-store writes starving WebSocket handshakes.

## Best Possible Solution

Split the remaining work into maintainer-owned fixes that preserve the current mitigations: make the Manifest/clawrouter listener startup idempotent, add a generic plugin lifecycle cleanup/dedup surface for process-level resources when plugins restart, and redesign session persistence so slow durable writes cannot block latency-sensitive Gateway handshakes or session lanes.

## Reproduction Assessment

Unclear for the full triad: the original report describes an emergent restart-under-load failure, not a deterministic local test. It does include concrete probes for listener counts, same-PID port ownership, session-lock warnings, and WS timeout logs, and follow-up comments independently reproduce the non-Manifest starvation shape.

## Solution Assessment

No, current main is not the complete best fix because it mitigates readiness, timeout, and stuck-session symptoms while leaving the reported root-cause classes open. The maintainable direction is a staged subsystem fix rather than one broad automated patch.
