## Summary

Keep this open. Current main contains useful mitigations around client readiness, stuck-session recovery, startup tracing, runtime-deps churn, and supervised EADDRINUSE exits, but the reported root-cause triad is not fully resolved: session-store persistence still performs durable atomic writes while the store lock is held, plugin replacement cleanup remains limited to registered host-hook cleanup, and the Manifest/clawrouter listener implementation described in the report is not present in the checked-in source for verification.

## What This Changes

The issue requests fixes for Gateway restart starvation caused by an in-process Manifest/clawrouter EADDRINUSE retry loop, accumulating process signal handlers, and long session persistence lock holds that can block WebSocket handshakes.

## Best Possible Solution

Split the remaining work into maintainer-owned root-cause fixes: make the Manifest/clawrouter listener startup idempotent, provide or require a generic plugin lifecycle cleanup/dedup surface for process-level resources, and redesign session persistence so slow durable writes cannot starve Gateway WebSocket handshakes while preserving the intended crash-durability contract.

## Reproduction Assessment

Unclear. The discussion gives concrete logs and probes for listener counts, same-PID port ownership, session-lock warnings, and WebSocket timeouts, and related comments strengthen the non-Manifest session-lock starvation path, but I did not find or run a deterministic current-main reproduction for the full triad.

## Solution Assessment

No current complete solution is present. The mitigations on main are useful, but the narrow maintainable solution still requires root-cause changes across Manifest listener ownership, plugin lifecycle cleanup, and session persistence lock scope rather than only longer timeouts or restart watchdogs.
