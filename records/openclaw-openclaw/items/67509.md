## Summary

Keep open. Current main and the latest release still lack the requested root-execution guard, and this PR is a focused implementation for the linked DigitalOcean state-corruption bug. The latest patch still needs maintainer handling because it would make current DigitalOcean/VPS docs fail when they tell users to SSH as root and run `openclaw` commands.

## What This Changes

The PR adds a CLI root guard with real/effective UID checks, explicit and container-forwarding bypasses, unit tests, and modern plus legacy entrypoint wiring.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with matching DigitalOcean/1-Click guidance: guard both CLI entrypoints before mutable state/config side effects, check both real and effective UID, preserve intentional container forwarding, and document the supported non-root service-user or wrapper workflow plus any explicit override.

## Reproduction Assessment

Yes. The linked bug has deterministic root SSH reproduction steps, and current main shows root invocations still reach CLI startup with default state under `~/.openclaw` and user-systemd repair paths under the effective user session.

## Solution Assessment

No, not yet as submitted. A core root guard is a narrow plausible fix, but the patch should also reconcile the root-based DigitalOcean setup flow or intentionally scope the guard to the 1-Click/wrapper path maintainers want to support.
