## Summary

Keep open. Current main still lacks a CLI-wide root guard, and this PR remains the active implementation for the linked DigitalOcean root-execution bug, but it needs maintainer policy and docs compatibility review before merge.

## What This Changes

The PR adds a CLI root guard with real/effective UID checks, explicit and container-forwarding bypasses, modern and legacy entrypoint wiring, and unit coverage.

## Best Possible Solution

Land this PR or an equivalent maintainer-approved core fix with matching DigitalOcean and 1-Click guidance: guard both CLI entrypoints before mutable state/config side effects, check real and effective UID, preserve intentional container forwarding, and document the supported non-root service-user or wrapper workflow plus the explicit override.

## Reproduction Assessment

Yes. The linked bug has deterministic root SSH reproduction steps, and current main confirms root invocations can still reach CLI startup with default state under ~/.openclaw and user-systemd repair under the effective HOME.

## Solution Assessment

No, not yet as submitted. A core root guard is a narrow and plausible fix, but the PR should also reconcile documented DigitalOcean root-session commands or explicitly scope the guard to the intended image/wrapper behavior.
