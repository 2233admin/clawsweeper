## Summary

Keep open. Current main still has no root-refusal path for the modern or legacy CLI entrypoints, and the related DigitalOcean bug remains observable from the current startup, state-directory, and documentation surfaces. The PR is focused and its latest discussion addressed the main functional review comments, but root CLI refusal and the exact override/container semantics need maintainer policy review before merge.

## What This Changes

The PR adds a CLI root guard module and tests, then wires it into `src/entry.ts` and `src/index.ts` to block real or effective UID 0 unless an explicit root override or container-forwarding hints are present.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off: guard both CLI entrypoints before mutable state/config side effects, check both real and effective UID, preserve intentional container forwarding, and update DigitalOcean/1-Click guidance to match the chosen root-command behavior.
