## Summary

Keep open. Current main still lacks the requested CLI root guard, and this PR is the active implementation for the linked root-state corruption bug, but the submitted behavior blocks root CLI invocations while current DigitalOcean docs still instruct users to SSH as root and run `openclaw` commands from that shell.

## What This Changes

The PR adds a CLI root guard with real/effective UID checks, explicit/container-forwarding bypasses, entrypoint wiring for modern and legacy CLI paths, and unit coverage.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, keeping early real/effective UID protection and the narrowed container-forwarding bypass, while updating DigitalOcean guidance to use the supported non-root service-user or wrapper workflow.

## Reproduction Assessment

Yes. The linked bug has deterministic reproduction steps on a DigitalOcean root SSH flow, and current main code confirms root invocations still reach CLI startup with mutable state defaulting under the effective home directory.

## Solution Assessment

No, not yet as submitted. A core root guard is a narrow and plausible fix, but the PR should be paired with DigitalOcean install/docs policy updates or a deliberately narrower scope so documented setup commands do not fail unexpectedly.
