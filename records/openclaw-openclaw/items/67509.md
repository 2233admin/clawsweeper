## Summary

Keep open. Current main still does not implement the PR’s root guard, and the linked root/DigitalOcean bug remains a real current behavior gap. The PR is focused, but it should not merge as-is until the root-execution policy is reconciled with the current DigitalOcean/VPS docs that still run OpenClaw commands from a root shell.

## What This Changes

The PR adds a CLI root guard with real/effective UID checks and explicit/container-forwarding bypasses, wires it into the modern and legacy CLI entrypoints, and adds unit coverage.

## Best Possible Solution

Land this PR or an equivalent core fix after maintainer sign-off, with both CLI entrypoints guarded before mutable state/config side effects, real and effective UID coverage retained, intentional container forwarding preserved, and DigitalOcean/1-Click guidance updated to the supported non-root service-user or wrapper workflow.

## Reproduction Assessment

Yes. The linked bug has deterministic root-SSH DigitalOcean reproduction steps, and current main confirms the relevant paths still default mutable state to ~/.openclaw and enter CLI startup without a UID/EUID guard.

## Solution Assessment

No, not yet as submitted. A core guard is a plausible narrow fix, but the PR needs matching install/docs policy or narrower 1-Click wrapper scoping because current public DigitalOcean setup instructions still invoke openclaw from a root shell.
