## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Use flat enums for task tool schemas:** `src/agents/tools/task-tool.ts:31-39`
  - body: `Type.Union(...Type.Literal())` emits `anyOf` in the tool input schema. The local schema helper explicitly avoids that shape because some provider validators reject `anyOf`/`oneOf`, so this new owner-only tool can fail registration until these fields use `stringEnum`/`optionalStringEnum`.
  - confidence: 0.93
- **[P1] Use a flat enum for the team action schema:** `src/agents/tools/team-tool.ts:22`
  - body: This new tool repeats the same provider-incompatible `Type.Union(...Type.Literal())` pattern for `action`. Other action-dispatched core tools use `stringEnum(...)`, which keeps the generated schema as a plain string enum instead of `anyOf`.
  - confidence: 0.92
- **[P2] Avoid emitting the sandbox readiness note twice:** `src/commands/doctor-sandbox.ts:239-252`
  - body: `runSandboxHealth` now calls `noteSandboxCapabilityReadiness()` before `maybeRepairSandboxImages()`, but this non-docker/no-local-runtime branch emits another capability-probe note with overlapping details. A non-docker backend without local sandbox support will show duplicate Sandbox notes in one doctor run.
  - confidence: 0.86
- **[P2] Preserve missing-file errors from sandbox reads:** `src/agents/sandbox/fs-bridge-path-safety.ts:79-83`
  - body: `openBoundaryFile` returns `reason: "path"` for ENOENT/ENOTDIR/ELOOP, but this wrapper converts every failed readable open into `E_FS_PERMISSION_DENIED`. Callers that distinguish an absent file from a denied path can no longer recover correctly; keep path misses as the original path error and only wrap validation/permission failures.
  - confidence: 0.78
