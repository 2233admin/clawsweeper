## Summary

Keep open. Current main still uses the reverse latest-card matcher that this PR replaces, and the PR head now addresses the exact-ID priority regression called out in review, so it remains a relevant unmerged Control UI fix rather than something to close.

## What This Changes

The PR changes `ui/src/ui/chat/tool-cards.ts` so tool results first prefer an exact card id match and otherwise attach to the first unmatched same-name tool card instead of scanning cards from newest to oldest.

## Best Possible Solution

Land this PR or an equivalent narrow patch in `ui/src/ui/chat/tool-cards.ts`, ideally with a focused node test for sequential same-name pairing and exact-id priority so Control UI expanded tool cards consistently show the correct result payload.

## Reproduction Assessment

Yes. A message content array with two same-name `tool_use` blocks followed by two same-name `tool_result` blocks reproduces the pairing problem by source inspection on current main because the matcher scans backward and picks the newest unmatched card first.

## Solution Assessment

Yes. The PR's exact-id-first, first-unmatched-name fallback is the narrow maintainable fix for this renderer helper; the safer enhancement is adding the missing sequential-shape regression test, not changing the product boundary.
