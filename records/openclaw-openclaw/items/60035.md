## Summary

Keep this issue open. Current main has shipped meaningful HTTP forward-proxy support, including `proxy.enabled`, `proxy.proxyUrl`, `OPENCLAW_PROXY_URL`, process HTTP/WebSocket routing, and some env-proxy behavior for provider/web paths, but the issue asks for a broader HTTP/SOCKS plus per-agent/per-tool model. SOCKS remains explicitly unsupported at the top-level proxy schema, `tools.web.search` and `tools.web.fetch` still have no proxy override fields, and managed browser proxying remains an explicit Chrome-flag workflow rather than inheriting gateway/provider proxy env vars.

## Best Possible Solution

Keep this issue open as the canonical follow-up for the remaining proxy gaps. The shipped HTTP forward-proxy implementation should stay, while maintainers decide whether to narrow this issue to top-level SOCKS support plus per-tool/per-agent overrides or split separate follow-ups for SOCKS, web tool proxy overrides, and browser proxy UX that preserves SSRF protections.

## Risks / Open Questions

- Closing now would lose the still-missing SOCKS support requested in the issue; current schema and tests explicitly reject SOCKS URLs for the top-level proxy.
- Closing now would also lose the requested per-tool `web_search`/`web_fetch` proxy override surface, which is not present in the strict config schemas.
- Managed browser proxying has a documented manual path through Chrome flags, but not the automatic/system or tool-level proxy behavior proposed in the issue.
