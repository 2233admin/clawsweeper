## Summary

Keep this PR open. The performance direction is good, but the proposed fast path can return before collecting config-dependent manifest runtime dependencies, so it can skip a required packaged dependency for local memory search.

## What This Changes

The PR adds a pre-scan fast path to `ensureBundledPluginRuntimeDeps` plus a regression test asserting an already-materialized generated package-level manifest avoids reading an unrelated bundled plugin manifest.

## Best Possible Solution

Keep the scan-avoidance optimization, but make the proof cover all dependencies required by the requested plugin under the current config, including manifest-declared runtime dependencies, or fall through to the existing full package-level plan when those config-dependent manifest deps may apply.

## Reproduction Assessment

Yes. A high-confidence fixture follows from existing coverage: packaged `memory-core`, local memory search configured, package deps materialized in the generated manifest, and `node-llama-cpp` missing; the PR's fast path can return before the manifest scan/install that currently adds it.

## Solution Assessment

No. The fast path is the right shape for package.json-only plugins, but as written it is not the safest maintainable solution because it proves a narrower dependency set than the existing requested-plugin plan can require.
