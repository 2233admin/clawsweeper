## Summary

Keep this PR open. Current main still does not implement Discord account-level `agentId` routing: routing reads only persisted `bindings`, the Discord account config type/schema do not accept `accounts.<id>.agentId`, docs still show manual Discord bindings, and the status snapshot surface does not expose an account binding. The PR discussion is useful and active enough to preserve: reviewers found the direction plausible, the author addressed multiple routing/cache/normalization comments, and later comments identify remaining review work around status propagation and tokenless-account synthesis.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto current main, add Discord account-level `agentId` to the accepted config contract, synthesize effective route bindings in the routing layer without writing them back to config, preserve explicit binding precedence, decide and test disabled/tokenless-account behavior for outbound auto-selection, expose the effective binding through the real channel-status surfaces if that remains a product goal, and update docs plus focused routing/status tests.
