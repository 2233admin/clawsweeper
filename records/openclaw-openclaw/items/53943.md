## Summary

Keep this open. Current main still does not accept, forward, or filter Slack `read` requests by `messageId`; it only supports channel history and optional `threadId`, so the exact thread-reply lookup described in this issue remains unresolved.

## What This Changes

Support Slack `message(action: "read")` with `messageId`, including exact lookup of a reply inside a thread when `threadId` is also provided.

## Best Possible Solution

Add exact-message read support in the Slack plugin path: expose `messageId` for read where needed, forward it through the Slack dispatcher and runtime, make `readSlackMessages` return only the requested timestamp for both threaded and channel reads, preserve existing `threadId` behavior, and update focused tests plus user-facing message docs.
