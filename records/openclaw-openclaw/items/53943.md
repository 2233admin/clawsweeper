## Summary

Keep this issue open. Current main still does not implement exact Slack `messageId` reads: the Slack read dispatcher does not forward `messageId`, the runtime only passes `threadId` into `readSlackMessages`, and the helper returns channel history or thread replies without filtering or exact timestamp lookup.

## Best Possible Solution

Keep this issue open and fix it in the Slack plugin read path. Add `messageId` to the Slack `read` dispatcher and runtime options, extend `readSlackMessages` to return or filter the exact Slack timestamp for both `messageId + threadId` and messageId-only channel reads, add focused dispatcher/runtime/helper tests, and cover Slack URL timestamp normalization if that parsing feeds this tool path.

## Risks / Open Questions

- A correct implementation should avoid relying on Slack's default page size; exact timestamp reads may need constrained `oldest`/`latest` with `inclusive` or pagination so older thread replies are not missed.
- The current thread-read helper drops the parent message, so the fix should explicitly define and test behavior when `messageId` equals `threadId`.
- Docs and tool guidance should be updated once the read path actually supports `messageId`, because the generic tool schema currently exposes `messageId` broadly but the Slack read implementation ignores it.
