## Summary

Keep open. Current main still formats drained runtime system events as `System:` text and prepends them to user prompt bodies, and the linked closing PR is open and unmerged, so this should wait for maintainer review/merge or an equivalent fix.

## What This Changes

Runtime background task, deferred maintenance, and queued system-event updates should be delivered through runtime system context instead of polluting user-authored prompt text.

## Best Possible Solution

Land #72761 or an equivalent narrow change that moves drained runtime system events into `extraSystemPrompt` or another system-context channel, keeps them out of `commandBody`, `followupRun.prompt`, and transcript prompt text, and preserves the existing untrusted-event ownership downgrade.

## Reproduction Assessment

Yes. The current tests provide a high-confidence read-only reproduction path: mocked `drainFormattedSystemEvents` output is asserted to appear in `commandBody` and `followupRun.prompt` rather than runtime system context.

## Solution Assessment

Yes for the queued runtime system-event bug. Routing drained events into runtime system context is the narrow maintainable fix; heartbeat prompt semantics should remain a separate maintainer scope decision if they are intended to change.
