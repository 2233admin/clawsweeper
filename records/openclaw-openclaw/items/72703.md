## Summary

Keep open. Current `main` still drains queued runtime system events and prepends them into user prompt bodies, while the linked fix PR is open and unmerged, so this issue should wait for maintainer review or an equivalent merged fix.

## What This Changes

The issue reports that background task updates, deferred maintenance notifications, and heartbeat-like runtime notices are being injected into model input as user-role prompt text instead of system context.

## Best Possible Solution

Route drained runtime-generated system events through runtime system context, keep them out of `commandBody`, queued prompts, and transcript prompts, preserve the current trusted/untrusted ownership downgrade, and align the public `openclaw system event` docs if the user-visible semantics change.

## Reproduction Assessment

Yes. Current `main` has a focused code-level reproduction in `get-reply-run.media-only.test.ts`: mocked drained system-event text is asserted to appear in `commandBody` and `followupRun.prompt` instead of runtime system context.

## Solution Assessment

Yes in direction: moving drained events into `extraSystemPrompt` is the narrow maintainable fix for queued system events. The current linked PR should be narrowed or reviewed carefully because it also includes unrelated plugin CLI memory-slot changes.
