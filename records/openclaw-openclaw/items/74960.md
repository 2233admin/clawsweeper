## Summary

Keep open. Current main still implements image paste and attachment dispatch through the same side-payload store and Gateway attachment parser, but neither source nor tests prove the reported byte-identity invariant; the new reporter comments describe a current, reproducible stale/wrong-image symptom that is distinct from the older paste-support feature closure.

## What This Changes

The issue reports that WebChat paste and related channel image sends deliver stale or incorrect image bytes to the agent instead of the exact original image.

## Best Possible Solution

Add a regression that compares the original clipboard or channel image bytes with the bytes stored in the UI payload store, serialized into `chat.send`, parsed by Gateway, and dispatched to the agent, then fix the smallest failing shared attachment/media boundary. Keep WeChat-specific changes out of core unless the regression proves a missing generic plugin media contract.

## Reproduction Assessment

Yes. The reporter provides concrete Chrome/macOS WebChat paste steps and a second channel symptom, and current main exposes a clear verification path through `handlePaste`, the payload store, `buildApiAttachments`, `parseMessageWithAttachments`, and `dispatchInboundMessage`; this review did not run the live browser repro.

## Solution Assessment

Unclear. The best direction is a narrow byte-identity regression followed by the smallest code fix at the first failing boundary; closing as implemented because paste UI support exists would not solve the reported stale/wrong-image delivery.
