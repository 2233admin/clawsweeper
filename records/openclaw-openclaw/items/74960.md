## Summary

Keep open. Current main still has the same Control UI paste, side-payload-store, chat.send serialization, and Gateway image parsing surfaces, and the newer issue comments describe a byte-identity failure that is not covered by the earlier paste-support closure or by current tests.

## What This Changes

The issue reports that pasted or channel-sent images reach the agent as stale or wrong image bytes instead of the image the user sent.

## Best Possible Solution

Add a regression that compares clipboard/original image bytes against the bytes serialized into chat.send and the bytes parsed for agent dispatch, then fix the smallest failing boundary in the Control UI payload store, send serialization, or Gateway attachment parser. Keep WeChat-specific changes out of core unless the regression proves a generic plugin media contract gap.

## Reproduction Assessment

Yes. The issue gives concrete macOS Chrome WebChat paste steps plus a second WeChat/openclaw-weixin path, and the current code provides a clear instrumentation path through handlePaste, the side payload store, buildApiAttachments, and parseMessageWithAttachments; this review did not execute the live browser/channel repro.

## Solution Assessment

Unclear. The best direction is a narrow byte-identity regression and minimal code fix; closing as implemented based only on paste UI support would not solve the reported stale/wrong-image delivery.
