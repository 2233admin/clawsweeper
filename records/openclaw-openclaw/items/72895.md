## Summary

Keep open. Current main has partial Baileys hardening, but the observable crash report is still actionable: OpenClaw still delegates outbound media upload to Baileys, the pinned Baileys upload helper uses `createReadStream(filePath)` for the encrypted temp file, and current code does not prove every external staged runtime-dependency copy receives the hotfix or that async stream `ENOENT` cannot escape as an uncaught Gateway exception. The related WAHA request is a broader alternative channel idea, not a replacement for fixing this Baileys-backed WhatsApp crash.

## What This Changes

The issue requests hardening WhatsApp outbound media uploads so Baileys encrypted temp-file `ENOENT` failures do not crash the Gateway, while preserving proxy/dispatcher behavior, making upload timeout configurable, and clarifying runtime dependency selection.

## Best Possible Solution

Keep this issue open and land a WhatsApp-owned hardening fix: make the Baileys media-upload hotfix avoid or fully observe `createReadStream` `ENOENT`, apply that hotfix consistently to package-root and external staged runtime dependency roots, plumb a documented media upload timeout if maintainers accept the config surface, and add focused regression coverage that proves upload failures are reported without an uncaught Gateway exception.

## Reproduction Assessment

Yes. The reporter provided a concrete `openclaw message send --channel whatsapp --media /tmp/tiny.png` path plus crash logs, and source inspection confirms that current outbound media sends still reach the Baileys encrypted-temp upload helper; this review did not execute a live WhatsApp send because it is read-only and externally stateful.

## Solution Assessment

No. The current patch is not the best complete solution because waiting for the encrypted write stream to finish addresses one temp-file race but still leaves an async upload read stream and external staged-copy consistency unresolved; a safer fix snapshots the encrypted file or fully handles stream open/read errors and applies the same hotfix to every runtime dependency root used by the plugin loader.
