## Summary

Keep open. Current main still has package-root Baileys hotfix machinery, but selected external bundled runtime-deps roots can be treated as already materialized from dependency specs, installed versions, and entry files without verifying the Baileys `messages-media.js` hotfix content before WhatsApp sends media through Baileys.

## What This Changes

The issue reports that WhatsApp outbound media upload can crash the Gateway when Baileys opens temporary encrypted media files after they have disappeared.

## Best Possible Solution

Keep this issue open until OpenClaw verifies and repairs the Baileys encrypted-media/dispatcher hotfix in every selected external staged runtime-deps root before WhatsApp runtime loading, or upgrades to an upstream Baileys version that removes the temp-file race. The fix should preserve existing package-root hotfix behavior and add regression coverage for stale generated-manifest external roots.

## Reproduction Assessment

Yes. The issue includes concrete CLI reproduction steps and crash logs, and current source confirms outbound WhatsApp media still delegates to Baileys `sock.sendMessage`; I did not perform a live linked-account media send during this read-only review.

## Solution Assessment

No for current main. The package-root postinstall hotfix is useful but incomplete because external staged runtime-deps roots can be accepted as materialized without checking that Baileys `messages-media.js` contains the required hotfix content.
