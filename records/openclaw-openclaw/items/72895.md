## Summary

Keep open. Current main still has only partial Baileys media hotfix coverage: packaged external runtime-deps roots can be accepted as already materialized from dependency specs, package versions, and entry files without verifying that Baileys `messages-media.js` contains the encrypted-media hotfix, so stale unpatched roots can still be reused for WhatsApp outbound media.

## What This Changes

Requests that WhatsApp outbound media sends stop crashing the Gateway on Baileys encrypted temp-file `ENOENT` errors while preserving proxy upload behavior, improving upload timeout handling, and making runtime-deps root selection safer.

## Best Possible Solution

Keep this open until OpenClaw either verifies and repairs the Baileys encrypted-media hotfix in every selected external runtime-deps root before WhatsApp loads, or upgrades to an upstream Baileys version that removes the race. The fix should include a regression test for stale already-materialized external roots and should separately decide whether upload timeout configurability belongs in the WhatsApp socket/options surface.

## Reproduction Assessment

Yes. The issue gives concrete CLI reproduction steps and crash logs, and current source inspection confirms outbound WhatsApp media still delegates to Baileys upload handling; the only missing piece is live account execution, which is outside this read-only sweep.

## Solution Assessment

No. The current postinstall/patch helper is useful but incomplete because external runtime-deps roots can bypass hotfix verification once their package plan looks materialized; the safer solution is to validate or repair the selected runtime root before reuse.
