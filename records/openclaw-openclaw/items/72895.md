## Summary

Keep open. Current main still does not prove that the Baileys media hotfix reaches the external packaged runtime-deps roots that the report names: WhatsApp media sends still delegate to Baileys, the WhatsApp plugin still stages `@whiskeysockets/baileys@7.0.0-rc.9`, and runtime-deps materialization accepts a staged package by generated spec plus installed version/entry files without checking the hotfix content.

## What This Changes

The issue asks OpenClaw to prevent WhatsApp outbound media uploads from crashing the Gateway on Baileys encrypted temp-file ENOENTs, retain the fetch dispatcher bypass behavior, make upload timeout configurable, and handle multiple staged runtime-deps copies consistently.

## Best Possible Solution

Keep this open until OpenClaw applies or verifies the Baileys encrypted-media hotfix for every bundled WhatsApp runtime-deps root that can be loaded, rejects or repairs stale unpatched roots before import, and adds regression coverage for the staged packaged install path. The timeout configurability can be handled in the same hardening pass or a follow-up if maintainers want to split the work.

## Reproduction Assessment

Yes. The issue provides concrete CLI steps with a linked WhatsApp account and ENOENT logs, and current main still has the same high-level path: outbound media buffers are handed to Baileys `sendMessage` through externally staged `@whiskeysockets/baileys`.

## Solution Assessment

No. The current root pnpm patch/postinstall helper is only a partial fix because external runtime-deps installs can still materialize an unpatched Baileys package that passes version/spec checks; the safer solution is to make the packaged runtime-deps path hotfix-aware or move to a dependency artifact that is patched at install source.
