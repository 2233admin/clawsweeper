## Summary

Keep open. Current main has fixed the older sub-agent model-override path and records model fallback steps in trajectory events, but it still lacks the requested closed-loop effective model/provider/thinking fields on subagent task/lifecycle telemetry, and the thinking validation happens before the model fallback result is known.

## What This Changes

Request to record requested-versus-effective model/provider/thinking for sub-agent runs, reconcile fallback-selected runtime model state, and re-clamp thinking when fallback changes model capability.

## Best Possible Solution

Add additive typed requested/effective model/provider/thinking telemetry for subagent runs at the core task/run/lifecycle surfaces, reconcile final fallback-selected model state on completion, and re-derive thinking compatibility against the final model while leaving extension-specific model rewrites to the owning plugin.

## Reproduction Assessment

Yes. A source-level reproduction is clear: an explicit `sessions_spawn.model` is stored as initial child state, fallback can later return a different provider/model, and the task/hook/lifecycle contracts have no effective_* fields for consumers to compare after completion.

## Solution Assessment

Yes, the P1 direction from the report is the narrowest maintainable fix if kept additive and core-owned. The broader retry-budget, consent, and task-router rewrite ideas should stay out of the first repair PR.
