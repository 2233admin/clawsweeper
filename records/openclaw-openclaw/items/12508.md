## Summary

Keep open. Current main still exposes mutable, sequential plugin hooks without hook-chain integrity state, per-handler modification audit records, or a per-hook permission allowlist; the existing maintainer-thread review also kept this for core/plugin-runtime security design. Related #18571 is narrower outbound verification work and does not supersede this broader hook-chain request.

## What This Changes

The issue requests hook-chain integrity checks, audit logging, and per-hook plugin permissions to detect or prevent silent mutation across plugin hook handlers.

## Best Possible Solution

Define host-owned hook provenance and per-handler modification audit semantics in the plugin runtime, decide explicitly how integrity should work for in-process versus isolated plugins, then add a backwards-compatible per-hook permission policy with docs and focused tests.
