## Summary

Keep open. Current main still documents and implements mutable, sequential plugin hooks without hook-chain integrity checks, per-handler modification audit records, or a per-hook permission allowlist. The prior maintainer review in the thread also explicitly kept this open for core/plugin-runtime security design.

## What This Changes

Add hook-chain integrity protection, audit logging, and per-hook permission controls so plugin hook mutations cannot silently tamper with messages, tool calls, or agent context.

## Best Possible Solution

Keep this open for maintainer-owned core/plugin-runtime security design. A good path would define host-owned hook provenance and per-handler modification audit semantics in src/plugins/hooks.ts and call sites, decide whether tamper-evident checksums are meaningful for in-process plugins, and add a backwards-compatible per-hook permission/config or manifest policy with docs and tests. Related #18571 can inform outbound message verification, but it should not replace this broader design.
