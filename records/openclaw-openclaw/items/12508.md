## Summary

Keep open. Current main still documents and implements mutable, sequential in-process plugin hooks without hook-chain integrity checks, per-handler modification audit records, or per-hook manifest permission declarations. A recent maintainer/Codex review in the thread already kept this open, and current source/docs still match that assessment.

## Best Possible Solution

Keep this open for maintainer-owned core/plugin-runtime security design. The likely direction is host-owned hook provenance and per-handler modification audit semantics in src/plugins/hooks.ts and call sites, plus a backward-compatible per-hook permission/config or manifest policy. Related outbound verification work such as #18571 can inform message_sending, but it should not replace the broader hook-chain integrity/audit/permission design requested here.
