## Summary

Keep open. Current main still has the mutable sequential hook pipeline identified in the thread, with no hook-chain integrity state, per-handler modification audit, or per-hook registration allowlist; this remains security-sensitive plugin-runtime design work rather than cleanup closure.

## What This Changes

The issue requests hook-chain integrity verification, mutation audit logging, and per-hook permission controls to detect or constrain plugin message tampering across sequential hooks.

## Best Possible Solution

Define host-owned hook provenance and per-handler mutation audit semantics in the plugin runtime, distinguish in-process native plugins from any isolated-plugin IPC path, then add a backwards-compatible per-hook permission policy with docs and focused runtime/config tests.
