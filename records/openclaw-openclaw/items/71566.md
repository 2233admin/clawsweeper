## Summary

Keep open. Current main still lacks the Gateway directory RPC handlers, discovery entries, and read-scope mappings that this PR proposes, and the PR remains an active, plausible Gateway protocol addition over the existing channel directory adapter contract. It should get maintainer review rather than cleanup closure because the remaining questions are protocol/schema/docs/tests and data-access scope, not obsolescence.

## What This Changes

The PR adds `directory.self`, `directory.peers.list`, and `directory.groups.list` Gateway RPC handlers, registers them in core handler dispatch and method discovery, and classifies them under `operator.read`.

## Best Possible Solution

Land this PR or an equivalent implementation only after the Gateway directory methods are treated as a proper protocol addition: typed params/result validation where appropriate, method discovery and scope tests, handler tests for unsupported/runtime-error paths, public protocol docs or an explicit waiver, and an explicit review of exposing directory data under `operator.read`.
