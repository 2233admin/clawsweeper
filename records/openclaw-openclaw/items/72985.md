## Summary

Keep open. Current main still matches the reported cron isolated-agent failure mode: cron does not pass the embedded-runner incomplete-turn result classifier into its model fallback call, and cron delivery/state persistence can still carry an error result together with delivered=true when an error payload is successfully sent to the channel. The issue is recent, concrete, unprotected, and the related fallback reports track different failure modes rather than superseding this one.

## Best Possible Solution

Keep this issue open and fix the cron isolated-agent path. Wire classifyEmbeddedPiRunResultForModelFallback, or an equivalent cron-owned classifier, into cron's runWithModelFallback call so incomplete terminal embedded results drive fallback/retry/failure policy structurally. Then change cron state semantics so successfully sent error fallback text is not counted as normal cron output delivery, either by forcing lastDelivered=false for error runs or by adding a distinct lastErrorDelivered field, with regression coverage for status=error plus incomplete-turn payload plus successful channel send.
