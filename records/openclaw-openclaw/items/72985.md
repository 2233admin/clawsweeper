## Summary

Keep open. Current main still has the reported cron isolated-agent gap: the cron fallback caller does not use the existing embedded-runner result classifier, and the delivery/persistence path can still record a fatal error payload as delivered when the outbound channel send succeeds.

## What This Changes

The issue asks for cron isolated-agent fallback failures to stop treating incomplete-turn warning text as successful delivered cron content and to expose clearer fallback/error semantics for monitoring.

## Best Possible Solution

Cron should classify embedded-runner incomplete outputs before accepting a model fallback attempt, and failed cron output should use failure-notification semantics or a distinct error-delivery field instead of successful content delivery. A narrow fix should reuse the existing classifier seam, keep core fallback policy generic, and add regression tests for the persisted `status: "error"` plus delivery-state behavior.

## Reproduction Assessment

Yes. A deterministic test can mock `runWithModelFallback` or the embedded runner to return only the incomplete-turn warning as an error payload, let announce delivery succeed, and then assert that the cron result and persisted job state do not report useful content as delivered.

## Solution Assessment

Yes, the best fix is to wire the existing embedded-runner fallback classifier into the cron fallback call and adjust delivery/persistence semantics for fatal cron payloads. A regex-only `lastDelivered` patch would leave the fallback acceptance bug split across layers.
