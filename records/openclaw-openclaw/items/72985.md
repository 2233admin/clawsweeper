## Summary

Keep open. Current main still accepts cron isolated-agent embedded-runner incomplete-turn results without the model-fallback classifier, and the delivery/state pipeline can still persist a failed cron run with successful normal delivery semantics.

## What This Changes

The issue asks cron isolated-agent fallback runs to retry or alert when a fallback provider produces the embedded-runner incomplete-turn warning, and to avoid recording failed cron output as normally delivered.

## Best Possible Solution

Cron should reuse the existing embedded-runner result classifier before accepting model-fallback attempts, then persist delivery state so useful assistant output and delivered error notifications are distinguishable for jobs-state monitoring.

## Reproduction Assessment

Yes. A deterministic reproduction can mock the cron isolated-agent fallback result to return the incomplete-turn `isError` payload, let announce delivery succeed, and observe current main returning/persisting `status: "error"` with `delivered: true`.

## Solution Assessment

Yes. The narrow maintainable fix is to wire the existing embedded-runner fallback classifier into cron and adjust persistence semantics for error payload delivery; a regex-only `lastDelivered` patch would leave fallback acceptance and monitoring semantics split across layers.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/72985. Current main accepts the embedded-runner incomplete-turn warning as a successful cron model-fallback result because `src/cron/isolated-agent/run-executor.ts` does not pass `classifyEmbeddedPiRunResultForModelFallback` into `runWithModelFallback`. That error payload can then be delivered as normal cron output and persisted as `lastRunStatus: "error"` with `lastDelivered: true`.

Expected fix boundary: wire the existing embedded-runner result classifier into the cron isolated-agent fallback path, preferably through the existing runtime boundary if needed; keep fatal/incomplete fallback text from counting as normal delivered assistant output; preserve successful fallback runs with real visible content; if an error notification is intentionally delivered, represent it separately from normal `lastDelivered` semantics. Add deterministic tests for incomplete-turn error payload classification, fallback rejection, and persisted delivery state. Do not change provider default selection, fallback candidate ordering, or unrelated channel delivery behavior.

Changelog: add a user-facing bug-fix entry in the active version if runtime behavior changes, with `Thanks @Allenbluff` attribution.
