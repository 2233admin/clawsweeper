## Summary

Keep open. Current main still has the reported inbound-dispatch hot path, and the paired implementation PR #74118 is open and unmerged with closing syntax for this issue, so the correct maintainer cleanup path is to review, land, or replace that PR rather than close this issue early.

## What This Changes

The issue requests that `ensureRuntimePluginsLoaded` avoid rebuilding the runtime plugin registry on inbound dispatch when a valid active plugin registry is already loaded.

## Best Possible Solution

Keep this issue open until #74118 or an equivalent replacement lands; the shipped end state should make inbound dispatch reuse the already-active runtime plugin registry and include focused regression coverage proving no rebuild when a valid active registry exists.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: inbound dispatch calls `ensureRuntimePluginsLoaded`, that helper always calls `resolveRuntimePluginRegistry` with reduced explicit options, and the resolver falls through to a fresh plugin load when those options do not match the active boot cache key.

## Solution Assessment

Unclear until #74118 is reviewed. The proposed active-registry fast path is the narrowest call-site fix for this issue, while a loader-level subset-compatibility change may be preferable if maintainers want one generic fix for related cache-key misses.
