## Summary

Keep open. Current main and the latest release still contain the reported runtime-plugin hot-path shape, and the issue is paired with open, unmerged PR #74118, so it should stay open until that fix or an equivalent replacement lands.

## What This Changes

The issue asks OpenClaw to avoid rebuilding and re-registering runtime plugins on each inbound dispatch when an active plugin registry is already loaded.

## Best Possible Solution

Land a reviewed fix that makes inbound dispatch reuse an already-active runtime plugin registry, with focused regression coverage proving no rebuild when a valid active registry exists. If maintainers prefer a broader architecture fix, solve the cache-key compatibility gap in the plugin loader and keep this issue as the user-facing tracker until that lands.

## Reproduction Assessment

Yes. The static current-main path is high confidence: inbound dispatch calls `ensureRuntimePluginsLoaded`, the helper always resolves with reduced options, and the resolver can fall through to `loadOpenClawPlugins` when the active cache key differs from the gateway boot key. I did not run a live hosted gateway because this review was read-only.

## Solution Assessment

Unclear until PR #74118 is reviewed. The proposed active-registry fast path is the narrowest call-site fix for this issue, while a loader-level subset-compatibility change may be better if maintainers want one shared solution for related cache-key mismatches.
