## Summary

Keep open. Current main and the latest release still have the reported inbound-dispatch plugin reload path, and the paired implementation PR #74118 remains open and unmerged, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue asks for inbound dispatch to avoid rebuilding runtime plugins through `ensureRuntimePluginsLoaded` when a valid active plugin registry already exists.

## Best Possible Solution

Land a reviewed fix that makes inbound dispatch reuse the already-active runtime plugin registry, with focused regression coverage proving no rebuild when a valid active registry exists; if maintainers prefer a broader design, solve the cache-key compatibility gap in the loader and keep this issue as the user-facing tracker until that lands.

## Reproduction Assessment

Yes. The high-confidence reproduction is the current static path: inbound dispatch calls `ensureRuntimePluginsLoaded`, that helper always resolves reduced options, and the resolver reloads plugins when the active cache key differs from those options.

## Solution Assessment

Unclear until #74118 is reviewed. The proposed active-registry fast path is a plausible narrow call-site fix, while a loader-level compatibility rule may be better if maintainers want one solution for multiple hot paths.
