## Summary

Keep open. Current main still has the reported runtime-plugin hot path: inbound dispatch calls ensureRuntimePluginsLoaded, that helper still builds reduced explicit load options and calls resolveRuntimePluginRegistry unconditionally, and the resolver still falls through to loadOpenClawPlugins when the active registry cache key is not compatible. The paired implementation PR #74118 remains open and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that ensureRuntimePluginsLoaded reuse an already-active runtime plugin registry during inbound dispatch instead of rebuilding plugins on each message.

## Best Possible Solution

Review and land #74118 or an equivalent fix that makes inbound dispatch reuse the already-active runtime plugin registry, with a focused regression test proving no rebuild when a valid active registry exists. If maintainers prefer a broader design, solve the loader cache-key compatibility gap and keep this issue as the user-facing tracker until that lands.

## Reproduction Assessment

Yes. The high-confidence reproduction is the current static path: inbound dispatch calls ensureRuntimePluginsLoaded, the helper always calls resolveRuntimePluginRegistry with reduced explicit options, and the resolver falls through to loadOpenClawPlugins when those options do not match the gateway boot cache key. I did not run a live hosted gateway because this review is read-only.

## Solution Assessment

Unclear until #74118 is reviewed. The proposed active-registry fast path is the narrowest call-site fix for the observed inbound dispatch cost, while a loader-level compatibility change may be better if maintainers want to address the same cache-key mismatch across multiple runtime paths.
