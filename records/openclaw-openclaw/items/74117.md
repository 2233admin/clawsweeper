## Summary

Keep open. Current main still has the reported hot-path shape, and the paired implementation PR #74118 is open and unmerged, so the conservative maintainer action is to review that PR instead of closing this issue early.

## What This Changes

The issue reports that `ensureRuntimePluginsLoaded` rebuilds runtime plugins on every inbound dispatch because dispatch-time load options do not match the broader gateway startup registry cache key.

## Best Possible Solution

Keep this issue open until #74118 is reviewed and either merged with focused regression coverage or replaced by a broader loader compatibility fix that prevents this dispatch-time reload without weakening plugin reconfiguration correctness.

## Reproduction Assessment

Yes for the cache-miss path: current main statically shows inbound dispatch calling the helper, the helper passing a narrower load-options object, and the resolver falling through to `loadOpenClawPlugins` when the active cache key differs. The live latency number was not reproduced in this read-only review.

## Solution Assessment

Unclear until #74118 is reviewed. The proposed `getActivePluginRegistry` fast path is a narrow fit for an ensure helper, while a broader subset-compatible loader cache check may be safer if maintainers want to solve the same strict-key class across multiple call sites.
