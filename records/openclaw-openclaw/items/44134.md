## Summary

Keep open. Current main still constructs the OpenClaw tool catalog inside each embedded attempt and immediately runs provider-owned schema normalization for that attempt; I found provider/plugin resolution caches, but no session-scoped or catalog-level reuse of normalized tool schemas. The issue also has concrete recent discussion tying the same repeated normalization path to Google/Antigravity anti-abuse symptoms and gateway allocation pressure, so it is not a stale or already-implemented cleanup candidate.

## Best Possible Solution

Keep this issue open and implement a conservative normalized-tool-catalog reuse layer around runtimePlan.tools.normalize or normalizeProviderToolSchemas. Key it by provider/model/API context plus stable effective tool catalog identity or content, invalidate for dynamic MCP/LSP/client tool-list changes, and add focused tests proving repeated embedded attempts with unchanged tools do not rerun per-tool schema normalization while changed catalogs still update. Then validate the Google Gemini CLI/Antigravity path separately before making external anti-abuse claims.
