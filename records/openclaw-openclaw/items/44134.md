## Summary

Keep this issue open. Current main still normalizes provider tool schemas inside each embedded attempt, and the provider compatibility helpers still map the full tool catalog without a normalized-catalog cache. Google Gemini CLI now warns about account restriction risk during OAuth setup, but that is only a caution surface; it does not address repeated schema normalization/reloading or the allocation pressure described in the issue comments.

## Best Possible Solution

Keep this issue open and implement a conservative normalized-tool-schema reuse layer in the runtime/tool-schema path. The fix should cache provider-normalized parameters/catalogs for stable tool lists and provider/model/API context, invalidate on actual tool-list or schema changes, and avoid retaining stale per-attempt execution closures. Add focused tests proving repeated attempts with unchanged tools do not rerun per-tool normalization, while changed tool schemas and dynamic bundled tools still update.
