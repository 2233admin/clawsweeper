## Summary

Keep open. Current main still lets the reported Anthropic tool_use/tool_result mismatch miss the replay-invalid classifier and fall through to the generic invalid_request_error formatter, so the raw provider wording can still be sent as user-facing `LLM request rejected: ...`; the issue is also paired with open PR #45393, which uses closing syntax and should be reviewed or replaced before cleanup.

## What This Changes

The issue asks OpenClaw to stop leaking raw Anthropic tool_use/tool_result mismatch errors to chat users, with a friendly formatter guard and repair coverage for orphaned tool calls.

## Best Possible Solution

Review, rebase, land, or replace PR #45393. The desired end state is a mismatch-specific formatter guard before the generic invalid_request_error fallback, provider-scoped transcript repair behavior, and focused regression coverage for the exact Anthropic wording plus orphaned last-message tool-call cases.

## Reproduction Assessment

Yes. A high-confidence unit-level reproduction is to feed the reported Anthropic invalid_request_error JSON into the current formatter path; the current regex classification misses it and the generic fallback emits the raw `LLM request rejected: ...` text.

## Solution Assessment

Yes for the immediate fix direction. A targeted formatter classifier with regression tests is the narrowest maintainable way to stop the user-facing leak; broader retry or transcript mutation changes can be evaluated separately after the root runtime path is clearer.
