## Summary

Keep open. Current main still lacks the requested WhatsApp composing-aware debounce behavior, and the PR is a useful contributor implementation, but it is not merge-ready: the config key is not wired into strict validation, group presence events do not reliably touch the same debounce key, presence subscriptions are repeated per inbound message, and the short-lived JID mapping cache is unbounded.

## What This Changes

The PR adds an opt-in WhatsApp `composingDebounceMs` setting, account/runtime plumbing, Baileys presence subscription handling, and `touch`/`touchByPrefix` helpers on the shared inbound debouncer.

## Best Possible Solution

Land a focused update that makes `composingDebounceMs` a real accepted WhatsApp root/account config field, documents it in the WhatsApp/config reference, covers debouncer touch behavior with tests, keys presence touches exactly like buffered direct and group messages, deduplicates subscriptions, and bounds any short-lived identity cache.

## Reproduction Assessment

Not applicable as a current-main user bug reproduction, but there is a high-confidence source-level reproduction for the PR blockers: parse WhatsApp config containing `composingDebounceMs`, inspect the strict zod schema, and compare the PR's presence prefix with the current group debounce key builder.

## Solution Assessment

No, the PR is not yet the best way to solve the issue. The feature direction is reasonable, but the maintainable solution needs schema/metadata/docs/test coverage plus corrected presence keying and bounded subscription/cache behavior before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Wire the setting into strict config validation:** `src/config/types.whatsapp.ts:85`
  - body: The PR adds `composingDebounceMs` to the WhatsApp config type and reads it at runtime, but it does not add the key to the strict WhatsApp zod schema. User config that sets this option will be rejected as an unknown key before the feature can run.
  - confidence: 0.95
- **[P2] Match group buffers with the sender debounce key:** `extensions/whatsapp/src/inbound/monitor.ts:486-490`
  - body: The presence handler builds an E.164-only prefix, but group debounce buffers prefer the sender identity or participant JID before `senderE164`. In normal group traffic, composing events will not match the buffered message key, so the debounce window is not extended.
  - confidence: 0.9
- **[P2] Deduplicate presence subscriptions per chat:** `extensions/whatsapp/src/inbound/monitor.ts:400-402`
  - body: `sock.presenceSubscribe(remoteJid)` is called for every inbound message while the feature is enabled. Add a per-chat guard and retry-on-failure behavior so active conversations do not create unnecessary WhatsApp presence traffic or rate-limit exposure.
  - confidence: 0.86
- **[P2] Expire the LID mapping cache:** `extensions/whatsapp/src/inbound/monitor.ts:78`
  - body: `jidToE164Cache` records every distinct JID indefinitely, but the mapping only needs to bridge the short interval between an inbound message and its presence event. Add a TTL or max-size cap so long-running sessions do not grow the map without bound.
  - confidence: 0.82
