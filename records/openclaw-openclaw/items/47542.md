## Summary

Keep this PR open. Current main does not implement composing-aware inbound debounce for WhatsApp: there is no `composingDebounceMs` config, no presence subscription/update listener wired into inbound buffering, and the shared debouncer still only exposes `enqueue` and `flushKey`. The proposed PR remains a plausible feature path, but the existing review comments identify real blockers that should be fixed on the PR before merge, especially schema wiring and bounded/deduped presence subscription behavior.

## Best Possible Solution

Keep the PR open and ask for a focused update on the existing branch: add runtime schema/config metadata and tests for `composingDebounceMs`, add covered `touch`/`touchByPrefix` debouncer behavior, wire WhatsApp `presence.update` with key formats that match direct and group debounce buffers, dedupe and bound presence subscriptions/cache entries, update the security-impact text, and validate the WhatsApp plugin tests plus the shared inbound debounce tests.
