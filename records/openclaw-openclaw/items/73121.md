## Summary

Keep this open: current main has strong partial fixes for zombie task reconciliation, but the reported operator problem is not fully closed. Chat-backed CLI tasks and normal async gateway completions now have coverage, while main-session/non-chat CLI rows can still be treated as alive by a persistent session entry, routed JSON listing bypasses reconciliation, and the human task table still prints shortened IDs that exact lookup cannot resolve.

## What This Changes

Issue requests automatic reconciliation of terminated CLI runtime tasks from running to terminal states and clearer task cancellation behavior for copied task identifiers.

## Best Possible Solution

Keep the runtime-aware maintenance foundation, then add a focused task-ledger fix that treats stale CLI rows as backed only by an active run context or an explicit runtime recovery hook, makes JSON and human inspection consistent about reconciliation, and makes list/cancel identifiers copyable or resolvable by unambiguous prefix.

## Reproduction Assessment

Yes for the remaining class at code level: a stale running CLI task with childSessionKey like agent:main:main, no active run context, and an existing session row will pass the current backing-session check after the grace period. The reporter's exact historical rows would still be useful as fixtures, but they are not required to see the gap.

## Solution Assessment

No. Current main is a good partial solution for normal gateway completion, channel-backed CLI rows, cron recovery, and subagent recovery, but it is not the complete fix while non-chat CLI rows and routed JSON listing can still expose stale running records and cancel lookup remains inconsistent with printed IDs.
