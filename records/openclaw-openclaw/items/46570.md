## Summary

Keep open. Current main still has the reported mixed-source builtin hybrid search failure shape: memory and session chunks can be searched together, but strict hybrid hits are returned before relaxed keyword-backed memory hits are considered, so exact memory-file matches can still be crowded out by session results.

## What This Changes

The issue requests that `openclaw memory search` return indexed `MEMORY.md` and `memory/*.md` matches instead of only surfacing session transcript results.

## Best Possible Solution

Fix builtin memory-core search ranking so default mixed memory/session hybrid search preserves relevant memory-file keyword hits. A CLI `--corpus memory|sessions|all` option could be useful as an additive diagnostic control, but the default mixed search should work without requiring that workaround.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, and current source supports a deterministic regression test: enable builtin hybrid search with both memory and sessions, add an exact phrase only in a memory file, add session chunks that produce strict hits, and assert a memory-source hit is still returned.

## Solution Assessment

Yes. A narrow `MemoryIndexManager.search` fix with manager and CLI-equivalent regression coverage is the best path; a CLI-only filter would not fix the default ranking behavior reported here.
