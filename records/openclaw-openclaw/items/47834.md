## Summary

Keep open. Current main still does not provide a policy-safe current browser URL to `before_tool_call` hooks for browser `act`/`type` decisions; the URL is available only from the original params or post-execution browser responses. The reporter's credential-vault workaround is useful and fail-secure, but it confirms the remaining native hook-time API gap.

## What This Changes

The issue requests that browser `before_tool_call` hooks receive the current page URL for `act`/`type` actions so plugins can enforce domain-pinned credential injection.

## Best Possible Solution

Add a narrow, additive browser/plugin contract that gives pre-action browser policy hooks a policy-safe current tab URL for `act`/`type`, or an equivalent supported query seam, with docs and tests proving blocked/private URLs are omitted and supported profiles behave consistently.
