## Summary

Keep this PR open. Current main still wires bundled `session-memory` only to explicit `/new` and `/reset`; automatic daily/idle rollover is handled through session lifecycle hooks and transcript archival, not through bundled internal memory persistence. The PR remains a plausible focused fix, but it needs rebasing and adjustment for the current lifecycle path and the ACP duplicate-write concern.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. Rebase it onto current main, choose the canonical automatic-rollover integration path, then update bundled `session-memory` handler metadata/docs/tests so daily and idle rollovers persist the prior session exactly once while explicit `/new` and `/reset`, including ACP-bound resets, do not double-write.

## Risks / Open Questions

- Functional risk if merged as-is: stale ACP-bound explicit `/new` or `/reset` commands may double-write bundled memory because the PR's automatic-rollover predicate is based on `resetTriggered`, while ACP reset-in-place handling marks `resetHookTriggered` later.
- Integration risk: current main already exposes automatic rollovers through plugin `session_end` and `session_start`; maintainers should choose whether bundled `session-memory` should consume that lifecycle payload or add a new typed internal `session:rollover` event.
- Rebase risk: the PR predates current main's session lifecycle and reset-hook fallback changes, so it needs refreshed coverage for daily rollover, idle rollover, explicit `/new` and `/reset`, and ACP-bound reset/new behavior.
