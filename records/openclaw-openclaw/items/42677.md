## Summary

Keep open. Current main still does not implement the PR's intended bundled `session-memory` behavior for automatic daily/idle rollover: the bundled handler and metadata are command-only, while automatic rollover currently goes through session lifecycle/plugin hooks and transcript archival. The PR remains a plausible focused fix, but it needs rebasing and the ACP duplicate-write review concern still needs a product/code decision.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. Rebase it onto current main, choose the canonical rollover integration path, then update bundled `session-memory` handler metadata/docs/tests so daily and idle rollovers persist the prior session exactly once while explicit `/new` and `/reset`, including ACP-bound resets, do not double-write.

## Risks / Open Questions

- Merging the PR as-is may duplicate bundled memory writes for stale ACP-bound explicit `/new` or `/reset` flows unless the rollover predicate is adjusted around current reset-hook timing.
- Current main already exposes automatic session boundaries through plugin `session_end`/`session_start`; maintainers should choose whether bundled `session-memory` should use that path or add a new typed internal `session:rollover` event.
- The PR predates current main's session lifecycle and reset-hook fallback changes, so it needs a rebase and refreshed tests for daily rollover, idle rollover, explicit `/new` and `/reset`, and ACP-bound reset/new behavior.
