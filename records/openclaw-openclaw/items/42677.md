## Summary

Keep open. Current main still does not implement this PR's intended bundled session-memory behavior for automatic daily/idle session rollover. The bundled handler, metadata, CLI/docs examples, and tests remain command-only for `/new` and `/reset`; automatic stale rollover is handled through session archival and plugin lifecycle hooks, not the bundled internal `session-memory` hook. The PR remains a plausible focused fix, and the author addressed earlier automated review feedback in head commit 01e9906, but it needs a rebase and resolution of the ACP duplicate-write concern before merge.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. Rebase it onto current main, pick the canonical integration path for automatic session boundaries, then update bundled `session-memory` handler metadata/docs/tests so daily and idle rollovers persist the prior session exactly once while explicit `/new` and `/reset`, including ACP-bound resets, do not double-write.

## Risks / Open Questions

- The review comment about stale ACP-bound explicit `/new` or `/reset` remains relevant: current reset handling marks `resetHookTriggered` in the ACP reset branch after session initialization, so a naive `!resetTriggered` rollover predicate can double-run session-memory for that edge case.
- Current main now exposes automatic boundaries through plugin `session_end`/`session_start`; maintainers should choose whether bundled `session-memory` should use that lifecycle path or add the PR's new internal `session:rollover` event.
- The PR predates current main and needs refreshed tests for daily rollover, idle rollover, explicit `/new` and `/reset`, and ACP-bound reset/new behavior before it should land.
