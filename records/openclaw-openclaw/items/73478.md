## Summary

Keep this issue open. Current main still does not prove or implement image information on the live Gateway `chat` WebSocket event for agent-run image replies; media-bearing replies are projected into transcript/history, while the agent lifecycle live delta/final path remains text-buffer based.

## Best Possible Solution

Keep this open and clarify the Gateway WebSocket media contract. If live `chat` events should carry generated images, add an additive media-bearing projection for agent-run final chat events and cover `image_generate`/`mediaUrls` through `chat.send`, asserting both the live final event and `chat.history` expose usable image information without raw base64 leaks. If the intended contract is history-only, document that and make clients reliably reload `chat.history` after media-bearing final events.
