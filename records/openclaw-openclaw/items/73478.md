## Summary

Keep open. Current main still has a Gateway live-chat media gap: agent assistant events can carry generated-image `mediaUrls`, but the agent-event bridge broadcasts live `chat` delta/final payloads from text buffers only; separate non-agent chat/history paths already know how to project media into managed image blocks.

## What This Changes

The issue requests Gateway WebSocket `chat` events for an image-generation run to include generated image/media information instead of only assistant text.

## Best Possible Solution

Add an additive media-aware projection for Gateway agent-run live `chat` final events when assistant event data includes generated-image media, reusing the existing managed outgoing image URL path and avoiding raw base64 in broadcasts. If maintainers intentionally want history-only media delivery, document that WebSocket clients must refresh `chat.history` after final events.

## Reproduction Assessment

Yes. A focused source-level reproduction can use the `server-chat.agent-events` harness: register a run, emit an assistant event with `data.text` plus `data.mediaUrls`, then emit lifecycle end; current live `chat` delta/final payloads only contain text.

## Solution Assessment

Yes. A narrow Gateway agent-event bridge fix is the best path because provider/tool code already preserves media and the Gateway already has managed media projection helpers; changing MiniMax or requiring clients to infer media from history would leave the live event contract inconsistent.
