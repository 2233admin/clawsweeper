## Summary

Keep this PR open, but the proposed patch is not merge-ready. Current main still lacks missing-cwd ACP stale-binding cleanup, and the PR correctly targets the dispatch classifier gap; however, the reset fix only unbinds in the catch path even though the real ACP manager can return recoverable reset init failures as `runtimeNotice` and let `resetAcpSessionInPlace` return success without unbinding.

## What This Changes

The PR extracts ACP stale-session detection, adds missing-runtime-cwd matching, uses it from dispatch and reset-in-place cleanup, shares the stale-unbind reason, and adds dispatch plus lifecycle regression tests.

## Best Possible Solution

Update the PR so stale reset cleanup runs for both thrown `ACP_SESSION_INIT_FAILED` errors and matching recoverable `closeSession.runtimeNotice` results, keep the dispatch missing-cwd classifier narrow, and add a regression that exercises the manager-style resolved-notice path before changed-lane validation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle recoverable close notices before returning success:** `src/acp/persistent-bindings.lifecycle.ts:199-204`
  - body: `closeSession` does not always reject for reset-time `ACP_SESSION_INIT_FAILED` errors: with `discardPersistentState` and `allowBackendUnavailable`, it can resolve with `runtimeNotice`, after which `resetAcpSessionInPlace` returns `{ ok: true }`. Because the new unbind logic only runs in the catch block, the real recoverable reset path can still leave the stale binding attached. Capture the close result and run stale cleanup for matching notices as well as thrown errors.
