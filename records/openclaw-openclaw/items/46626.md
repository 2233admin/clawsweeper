Keep open. Current main does not fully implement the PR’s intended shared own-line `NO_REPLY` suppression, but this PR is not mergeable as-is because the diff has blocking correctness issues and overlaps newer mainline cron/JSON silent-reply work.

## What This Changes

The PR adds an `isSilentTokenOnOwnLine` helper, wires it into reply normalization, directive parsing, cron delivery, subagent follow-up, gateway streaming, and chat history filtering, and adds focused regression tests.

## Best Possible Solution

Consolidate the overlapping `NO_REPLY` work into one maintained patch: use a cached, case-insensitive shared helper for own-line/trailing silent intent, apply it consistently to normalization, directive/outbound planning, cron, subagent follow-up, gateway live projection, and chat history, and preserve media, interactive, direct rewrite, JSON-envelope, and `channelData` semantics with targeted regression coverage.

Yes. Static reproduction is high confidence: current main still has a test expecting `normalizeReplyPayload({ text: "File's there. Not urgent.\n\nNO_REPLY" })` to deliver the preamble, and the parser/gateway paths remain exact-token based for own-line text.

## Solution Assessment

No. The direction is correct, but this PR is not the best mergeable form because it needs case-insensitive cached matching, parser/test alignment, channelData preservation, and a rebase around newer cron and JSON silent-reply behavior.

Full review comments:

- **[P2] Align the inline-token test with parser behavior:** `src/auto-reply/reply/reply-directives.test.ts:33-36`
  - body: The added test expects `parseReplyDirectives("😄 NO_REPLY")` to remove the token, but this patch only clears text for exact or own-line silent replies. Inline text still returns unchanged, so this test fails unless the parser strips inline tokens or the expectation is corrected.
  - confidence: 0.97
- **[P2] Make own-line token matching case-insensitive:** `src/auto-reply/tokens.ts:64`
  - body: Existing silent-token helpers accept mixed-case forms such as `no_reply`, and current cron coverage includes `No_Reply`. The new own-line regex has no `i` flag, so `summary\nNo_Reply` can still leak on paths that rely on this helper.
  - confidence: 0.92
- **[P2] Preserve channelData when parsing own-line silence:** `src/auto-reply/reply/reply-directives.ts:42`
  - body: Marking own-line text as `isSilent` makes outbound planning treat channelData-only payloads as silent because it only excludes media from silence. Group/internal allow-policy paths can drop the entry, and direct rewrite paths can replace the intended empty-text card/button payload.
  - confidence: 0.84
- **[P3] Cache the own-line regex:** `src/auto-reply/tokens.ts:64`
  - body: This helper compiles a new `RegExp` on every call while the patch adds it to streaming and polling paths. Cache it like the neighboring silent-token regex helpers to avoid avoidable per-chunk allocation.
  - confidence: 0.79
