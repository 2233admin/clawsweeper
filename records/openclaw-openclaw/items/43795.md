## Summary

This issue should stay open. Current main still has reachable OpenAI Responses conversion and replay paths that assume assistant `content` is array-shaped, while the report and linked closed PRs describe third-party OpenAI-compatible responses with string, object, or null content. Recent replay work covers legacy string assistant content, but object/null and Responses-converter cases are not fully normalized, and the two repair PRs were closed without merge.

## What This Changes

The issue reports repeated embedded-agent failures from a Feishu session using `gptgod/glm-4-air`, where malformed provider assistant content triggers `500 v.content is not iterable` and retry loops.

## Best Possible Solution

Add a generic malformed assistant-content normalization boundary before OpenAI Responses output and replayed assistant turns are inspected. Preserve valid arrays, convert strings into text blocks where replayable, wrap only safe single block-shaped objects that match the existing content-block contract, and drop or sentinel null/unrecognized values according to the current error-turn policy. Keep the fix provider-agnostic and cover the Responses converter plus replay/session sanitizer paths with focused regression tests.

## Reproduction Assessment

Yes. A malformed Responses message with object-shaped `content`, or a replayed assistant turn with object/null `content`, reaches current-main code paths that call `.some(...)` or `for...of` without proving the value is an array.

## Solution Assessment

Yes. The narrowest maintainable solution is generic core hardening at the Responses conversion and replay normalization boundaries, not provider-specific handling; the closed PRs identified the right bug class but need a current-main replacement aligned with newer replay behavior.
