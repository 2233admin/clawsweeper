## Summary

Keep open. Current main has not implemented native WebChat video uploads, and this PR remains the active implementation candidate for the related feature request, but the latest branch still has a mixed image/video ordering regression introduced by the new media aggregation.

## What This Changes

The branch adds Control UI/WebChat MP4/M4V/MOV/WebM upload acceptance and previews, Gateway video validation and native-or-media-path routing, video model-catalog plumbing, Google native video transport support, OpenAI/Anthropic video drop guards, docs/changelog updates, and regression tests.

## Best Possible Solution

Keep this PR open as the active implementation candidate, fix the mixed native media ordering bug by carrying a single ordered prompt-media list through parsing/routing, then have a maintainer review the video allowlist, MIME hardening, size/retention behavior, catalog contract, provider transport behavior, and Testbox changed-gate proof before merge.

## Reproduction Assessment

Yes. Current main reproduction is straightforward because `ui/src/ui/chat/attachment-support.ts` rejects `clip.mp4`; the PR finding can be reproduced with a focused `chat.send` test that uploads `[video, image]` to a model advertising both `image` and `video` and observes `promptMedia` becoming `[image, video]`.

## Solution Assessment

No, not yet. The PR direction is the right narrow place to solve WebChat/Control UI native video delivery, but the implementation should preserve mixed attachment order before it is the best mergeable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve mixed attachment order:** `src/gateway/server-methods/chat.ts:2135`
  - body: When a turn includes both native images and native videos, this concatenates all images before all videos. `parseMessageWithAttachments` keeps only separate image/video arrays, so a user uploading `clip.mp4` then `diagram.png` to a video+image model will send `diagram.png` first. Prompts that refer to the first or second attachment can be answered against the wrong media; carry one ordered prompt-media list through parsing/routing instead.
  - confidence: 0.84
