## Summary

Keep open. Current main still has concurrency, depth, and active-child controls for subagents, but not the requested lifetime-per-session total spawn cap. The related closed WebSocket retry issue mitigates a downstream failure mode, not this preventive `maxPerSession` feature.

## Best Possible Solution

Keep #57639 open. The best implementation path is a core `agents.defaults.subagents.maxPerSession` or similarly named total-spawns-per-requester-session cap, with optional per-agent override if consistent with existing subagent config. Enforce it before starting both native subagent and ACP `sessions_spawn` work, backed by a registry query that counts all child runs for the requester session rather than only active children. Update config types/schema/generated docs, `docs/tools/subagents.md`, and add focused tests for sequential completed child runs exceeding the cap.

## Risks / Open Questions

- The product decision still needs to define whether archived, deleted, failed, timed-out, and ACP child runs count against the lifetime cap.
- A correct implementation should avoid breaking intentional long-lived orchestrator sessions by providing clear config defaults and per-agent override semantics.
