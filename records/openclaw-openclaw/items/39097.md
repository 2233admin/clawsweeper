## Summary

Current main still lacks the requested operator-configurable browser URL/domain/path denylist or warnlist. The browser config, strict schema, resolved browser policy, navigation guard, route call sites, tests, and docs remain centered on SSRF/private-network policy plus hostname allowlist exceptions, not public-site deny/warn patterns.

## Best Possible Solution

Keep this issue open for maintainer-approved browser URL policy design. The likely path is adding an operator-configurable browser.denylist or browser.blockedUrls field, deciding whether warnlist belongs in the same change, defining exact host/wildcard/path matching semantics, and wiring the result through config types, strict zod/generated schema/help, config resolution, shared navigation-policy options, managed Playwright navigation, Chrome MCP existing-session paths, raw CDP target creation, docs, and focused tests. Any bundled default blocked-domain list should be handled as a separate product/security decision.
