## Summary

Keep open. Current main still lacks an operator-configurable browser URL/domain/path denylist or warnlist; the relevant config, strict schema, resolver, navigation guard, routes, tests, and docs remain centered on SSRF/private-network policy plus hostname allowlist exceptions. The existing ClawSweeper comment reached the same conclusion and I did not find a newer implementation on current main.

## What This Changes

The issue requests an operator-configurable browser URL/domain/path denylist, optionally warnlist, that makes browser navigation fail fast with a clear error for known-blocked sites.

## Best Possible Solution

Keep this issue open for maintainer-approved browser URL policy design. A good implementation would add an operator-configurable browser.denylist or browser.blockedUrls surface, define exact host/wildcard/path and redirect matching semantics, decide whether warnlist belongs in the same change, wire the resolved policy through config types, strict zod/generated schema/help, browser config resolution, managed Playwright/raw CDP/existing-session navigation guards, route responses, docs, and focused tests. Bundled default blocked domains should be handled separately.
