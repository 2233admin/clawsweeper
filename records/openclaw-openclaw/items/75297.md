## Summary

Keep open. The report is fresh, detailed, and overlaps several active gateway performance regressions, but current main only shows partial mitigations rather than proof that the post-2026.4.23 event-loop/session/model slowdown is fixed or fully covered by one canonical tracker.

## What This Changes

The issue asks whether a production regression after OpenClaw 2026.4.23 that saturates the Gateway event loop and makes `sessions.list`/`models.list` very slow should be tracked separately.

## Best Possible Solution

Keep this open as a real regression until maintainers either choose one canonical tracker or bisect the 2026.4.23-to-current range with a large session store, then land focused fixes for the hot path that still saturates the Gateway event loop under Control UI/session/model polling.

## Reproduction Assessment

Yes. The issue gives a high-confidence reproduction shape: run the same large local state on a post-2026.4.23 build, measure `sessions.list`, `models.list`, liveness diagnostics, and CPU, then compare with rollback to 2026.4.23.

## Solution Assessment

Unclear. Existing current-main mitigations around session limiting and model-catalog caching are directionally useful, but the report says v2026.4.29 still degraded, so the best fix needs maintainer triage and probably a narrower bisect before implementation.
