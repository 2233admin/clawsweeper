## Summary

Keep open. Current main still mitigates managed restarts by draining active work and has narrow restart-aborted session recovery, but it does not implement durable partial assistant-generation persistence/resume. Assistant-tail partial output is still explicitly treated as non-resumable, and the regression test asserts that behavior.

## Best Possible Solution

Keep this issue open and narrow it to a maintainer-reviewed design for durable interrupted assistant-generation recovery. The implementation likely needs SessionManager-compatible checkpoint/status writes for in-progress assistant output, startup recovery that distinguishes generating/completed-not-delivered/sent/interrupted states, and channel delivery idempotency. Existing managed restart drain and user/tool/toolResult-tail recovery should remain as mitigations, not be treated as the full fix.

## Risks / Open Questions

- Durable partial-generation persistence must preserve the SessionManager parentId/DAG transcript contract; raw JSONL rewrites could break active branch history or compaction context.
- Provider replay safety is non-trivial: resuming after a partial assistant tail can duplicate visible text, tool calls, reasoning blocks, or provider-specific state unless the recovery model is explicit.
- Channel delivery recovery needs idempotent completed-not-sent versus already-sent handling across preview and final delivery paths for Telegram, Discord, Slack, and other channels.
