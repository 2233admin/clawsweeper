## Summary

Keep open. Current main has meaningful mitigation for managed restarts and explicit continuation after agent-triggered gateway restarts, but it still does not implement durable partial assistant-generation persistence or resume. The main-session recovery path is intentionally narrow and marks assistant-tail interrupted transcripts failed, while streaming assistant text remains subscription/runtime state rather than a persisted generating/complete/sent status model. The related #46860 was closed as a broader proactive-notification/restart-resume item, but it does not supersede this issue's remaining partial-generation recovery gap.

## Best Possible Solution

Keep this issue open and narrow it to the remaining durable recovery gap: persist assistant generation state safely during a turn, record delivery/finalization status, and add startup recovery that can distinguish generating, completed-not-delivered, sent, and interrupted states. The likely implementation should go through SessionManager-compatible transcript/checkpoint APIs plus the existing delivery queue/restart recovery surfaces, with regression coverage for partial assistant tails, completed-before-channel-delivery, crash/forced restart, drain timeout, and duplicate channel output prevention.

## Risks / Open Questions

- A durable fix must preserve the Pi SessionManager parentId/DAG transcript contract; raw JSONL partial writes would risk severing the transcript leaf path.
- Provider replay semantics need careful design so OpenClaw does not resend an already-partial assistant answer as context in a way that triggers duplicate or invalid provider output.
- Delivery recovery must coordinate with preview messages, block replies, final channel delivery, and acknowledgements to avoid duplicate visible output after restart.
