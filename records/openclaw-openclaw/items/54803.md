## Summary

Keep this PR open but do not merge it as-is: the requested `/status` UX improvement is still missing from current main, while the branch is stale against the shared status renderer and bundles unrelated Fly/Docker/workflow/runtime-config changes with blocking correctness and security regressions.

## What This Changes

The PR adds usage-fetch error rendering to the old `/status` command implementation and also adds Fly.io deployment workflow/runtime scripts, Docker image toolchain changes, a `/health` shortcut, plugin SDK outbound-send exports, channel import rewrites, docs, and tests.

## Best Possible Solution

Land the `/status` behavior as a narrow current-main patch in `src/status/status-text.ts` with regression coverage for structured `usageEntry.error` and thrown/timeout usage-summary failures. Split or drop the Fly.io, Docker, workflow, runtime-config, docs-template, and SDK import changes into separate owner-reviewed work.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve existing workspaces without an env override:** `scripts/sync-runtime-config.mjs:120-124`
  - body: This rewrites `agents.defaults.workspace` to `${stateDir}/workspace` on every startup when `OPENCLAW_WORKSPACE_DIR` is unset. Existing deployments with a custom workspace in `openclaw.json` would silently move agent context/session paths after restart; only set this when the env override is explicit or the field is missing.
  - confidence: 0.89
- **[P1] Let the gateway start when Tailscale bootstrap fails:** `scripts/fly-entrypoint.sh:90`
  - body: Because the entrypoint runs with `set -e`, a non-zero `tailscale up` exits before `exec "$@"`. A transient Tailnet or auth-key failure would stop the whole gateway instead of only disabling Tailscale exposure, so this step should retry or degrade non-fatally.
  - confidence: 0.88
- **[P1] Drop privileges before starting OpenClaw:** `Dockerfile:262-263`
  - body: The branch removes the previous `USER node` hardening and the new entrypoint ultimately execs the gateway as root. If root is required for bootstrapping, the image or script should drop to the node user before starting OpenClaw so the runtime app does not run with root privileges.
  - confidence: 0.86
- **[P3] Use the shared probe handler for /health:** `src/gateway/server-http.ts:771-774`
  - body: `/health` is already registered in `GATEWAY_PROBE_STATUS_BY_PATH`, but this early exact-URL handler returns a different response shape and bypasses the shared probe behavior. Move the config-independent handling into the existing probe path or remove the duplicate map entry.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces concrete container and supply-chain risks unrelated to the `/status` UX change.

Concerns:

- **[high] Gateway would run as root in the Docker image:** `Dockerfile:262`
  - body: The PR removes the existing non-root runtime hardening and adds an entrypoint that ultimately execs the gateway as root. That broadens the impact of any runtime compromise and should be fixed by restoring `USER node` or dropping privileges after root-only bootstrap steps.
  - confidence: 0.86
- **[medium] Root-executed installers are not pinned:** `Dockerfile:131`
  - body: The Dockerfile executes remote Bun and Tailscale install scripts without version pins. Each build can execute different upstream code as root, making release images harder to audit and reproduce.
  - confidence: 0.83
- **[medium] Persistent root operator config increases blast radius:** `scripts/fly-entrypoint.sh:38`
  - body: The entrypoint symlinks root-scoped Codex, GitHub CLI, Git, and SSH configuration into the Fly volume. Combined with running the gateway as root, a runtime compromise could persist and reuse operator credentials across restarts.
  - confidence: 0.78
