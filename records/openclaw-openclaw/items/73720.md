## Summary

Keep open. Current main still has the shared resolver and gateway persistence gaps this PR targets: caller-provided group metadata is still considered before server-derived session/spawn context, core tool paths still pass raw group metadata into the shared resolver, and gateway session persistence still writes request or stale stored group metadata without trust validation. The overlapping #72921 is open and unmerged, so it is not a landed superseding fix.

## Best Possible Solution

Keep this PR open and review it as a candidate canonical fix, or reconcile it with #72921 before landing one implementation. The landed fix should validate caller groupId inside resolveGroupToolPolicy for every caller, prefer session/spawn-derived group IDs over caller candidates, clear untrusted group metadata before gateway persistence/runtime forwarding, preserve legitimate first-turn group selectors, and keep resolver, coding-tools, allowlist, and gateway regression tests.
