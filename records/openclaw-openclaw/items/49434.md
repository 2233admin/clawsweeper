## Summary

Keep this PR open. Current main still does not implement the PR's intended cross-account Discord bound-thread suppression: Discord preflight resolves only the current account's runtime/configured binding and then continues into normal route handling, while the thread-binding manager and session-binding service remain scoped by channel/account. The PR remains a focused implementation candidate for the closed linked bug #49373, with the remaining maintainer question centered on avoiding false message drops from stale or runtime-disabled foreign bindings.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for cross-account Discord bound-thread isolation. The best path is a narrow Discord fix that checks for an active foreign thread owner before configured/generic routing, uses the active runtime config to ignore stale/removed/disabled/threadBindings-disabled owners, preserves same-account binding behavior, and lands focused regressions for foreign-owner suppression plus stale/removed/disabled-account cases.

## Risks / Open Questions

- The proposed suppression path is on a security/routing boundary: if it treats stale, removed, disabled, or threadBindings-disabled foreign accounts as active owners, it can drop valid messages for other Discord bots.
- The implementation should use the active effective runtime config consistently with existing thread-binding manager operations, otherwise hot config changes can keep stale foreign ownership decisions alive until expiry.
- No live Discord multi-bot end-to-end run was performed in this read-only review; maintainer review should require focused regression proof or live repro proof before merge.
