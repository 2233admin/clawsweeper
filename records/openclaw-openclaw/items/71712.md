## Summary

Keep this issue open. Current main still exposes general cron management through the owner/admin cron surface, but it does not implement the proposed agent-facing `schedule_task` API with gateway-injected non-forgeable provenance, playbook allowlists, recurring `activeUntil` enforcement, per-agent quotas, or schedule-create audit ledger entries.

## Best Possible Solution

Keep this open as a maintainer-reviewed security/API RFC. The best path is a gateway-mediated scheduling surface or constrained cron mode that injects actor/session provenance server-side, requires registered playbooks with parameter schemas, coordinates with #71711 for recurring expiration, adds per-agent quotas, records schedule-create audit entries, and documents how it differs from unrestricted operator cron.

## Risks / Open Questions

- This is a security-sensitive persistence/control-plane API; a narrow implementation without maintainer design review could increase prompt-injection persistence risk.
- The requested recurring expiration behavior overlaps open #71711, and provenance semantics overlap broader context-provenance work in #54373, so the design should coordinate rather than create parallel fields.
- A plugin-only implementation is unlikely to satisfy the core requirement because the gateway must inject non-forgeable actor/session provenance and enforce scheduler policy before job persistence.
