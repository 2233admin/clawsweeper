## Summary

Keep open: current main still has operator/admin cron APIs and an owner-only full-power cron tool, but not the requested narrowed agent-facing scheduling API with gateway-injected provenance, playbook allowlists, recurring lifetime policy, per-agent quotas, or creation audit contract. The RFC is recent, security-sensitive, and AS76 added concrete multi-agent production context, so this needs maintainer product/security design rather than cleanup closure.

## What This Changes

Request to design and add a narrowed agent-facing scheduling API that creates cron jobs through the gateway with non-forgeable provenance, playbook allowlists, recurring lifetime limits, per-agent quotas, and audit entries.

## Best Possible Solution

Write a short maintainer-approved design/ADR first, then add an additive gateway scheduling method/tool that injects creator identity and timestamp from the authenticated session, accepts only registered playbook IDs plus validated params, enforces quota and recurring lifetime policy, records durable audit entries, and leaves the unrestricted operator cron CLI intact.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source/docs inspection: current main exposes full cron.add and an owner-only cron tool, but not the proposed schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear until maintainers choose the product/security model. The requested direction is narrower and safer than exposing the existing full cron tool to agents, but it should be designed as a core gateway capability rather than attempted as a one-off autonomous patch.
