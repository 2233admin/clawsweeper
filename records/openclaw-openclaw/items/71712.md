## Summary

Keep open: current main still has operator/admin cron creation and an owner-only cron tool, but not the proposed constrained agent-facing scheduling API with gateway-injected provenance, playbook allowlists, quotas, recurring lifetime policy, or schedule-creation audit semantics. The RFC is recent, security-sensitive, and has concrete production-use context in the comments, so it needs maintainer product/security design rather than cleanup closure.

## What This Changes

The issue requests a hardened agent-facing scheduling API that lets agents create or update recurring cron jobs through the gateway with non-forgeable provenance, playbook allowlists, quotas, recurring lifetime policy, and audit logging.

## Best Possible Solution

Write a short core security/design record first, then add an additive gateway scheduling capability that derives creator identity and timestamp from the authenticated session, accepts only registered playbook identifiers plus schema-validated params, enforces per-agent quotas and recurring lifetime/permanent policy, records schedule create/update audit events, and leaves the unrestricted operator cron CLI/admin RPC intact.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main has cron.add and an owner-only cron tool, but no schedule_task/provenance/playbook/quota/lifetime contract.

## Solution Assessment

Unclear until maintainers choose the product and security model. The requested direction is safer than exposing the existing full cron surface to agents, but the fix should be designed as a core gateway/cron/audit capability rather than a one-off autonomous patch.
