## Summary

Keep open. Current main still exposes operator/admin cron creation and the owner-only cron agent tool, but it does not provide the narrower agent-facing scheduling API with gateway-injected provenance, playbook allowlists, recurring lifetime policy, quotas, or creation audit semantics. The issue is recent, security-sensitive, and has concrete production-use context, so it should go to maintainer product/security design rather than cleanup.

## What This Changes

The issue requests an agent-facing scheduling API for constrained cron creation with gateway-injected provenance, playbook allowlists, parameter validation, active lifetime controls, per-agent quotas, and audit records.

## Best Possible Solution

Define a small core design/ADR and follow-up implementation plan for agent-scheduled cron jobs: a separate gateway RPC/tool with gateway-injected creator identity and timestamp, registered playbook IDs with parameter schemas, explicit lifetime/permanent policy, per-agent quotas, task/audit ledger entries, migration behavior for existing jobs, and tests/docs that preserve the unrestricted operator cron CLI.

## Reproduction Assessment

Not applicable as a feature/security RFC rather than a runtime bug. The high-confidence verification path is source and docs inspection: current main has cron.add and an owner-only cron tool, but the requested schedule_task/provenance/playbook/quota/lifetime contract is absent.

## Solution Assessment

Unclear until maintainers choose the product/security model. The proposed separate gateway-enforced API is safer than exposing the full cron tool to agents, but the exact schema and policy should be designed deliberately before implementation.
