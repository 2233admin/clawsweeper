## Summary

Keep this PR open for maintainer rework rather than closing it as implemented. Current main has fixed the generic `openclaw logs` / `logs.tail` path and writer-side daily rollover, but the `openclaw channels logs` command still reads the configured file directly and lacks the channel-specific fallback this PR is trying to add. The PR also bundles unrelated cron warning work that is already represented by open #42245 in the supplied context, and its logging fallback design still has a custom-path/privacy correctness risk.

## Best Possible Solution

Keep this PR open for maintainer follow-up: split the cron warning work back to #42245, then rework the logging change so `openclaw channels logs` reuses the shared safe rolling-log resolver or an equivalent exact-pattern resolver. Add focused tests for missing current-day files, existing current-day precedence, custom/non-rolling `logging.file`, and channel filtering against the resolved fallback file.
