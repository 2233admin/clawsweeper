## Summary

Keep this PR open. Current main still has the channel-specific bug: `openclaw channels logs` reads only the exact configured log file and returns no lines when that file is missing, while the shared rolling-log resolver is used by generic `logs.tail` only. The latest PR branch is a focused repair that reuses the shared resolver, adds regression tests for rolling fallback/current-file precedence/custom non-rolling paths, and has recent maintainer follow-up marking it merge-ready for human review.

## Best Possible Solution

Keep this PR open for maintainer review and landing. The best implementation is the current narrow branch direction: reuse the existing rolling-log resolver in `channelsLogsCommand`, keep fallback restricted to default rolling log filenames, preserve custom non-rolling `logging.file` behavior, and land the focused channel-log regression tests with the changelog credit.
