## Summary

Keep open. The PR is pointed at a real outbound/TTS parity bug and the author provided useful repro context, but the patch is not ready to merge because the new hook can run paid/network TTS work during dry runs and it drops scoped TTS context when calling the synthesis helper.

## What This Changes

The PR adds a tagged-TTS hook to `handleSendAction`, adds a focused outbound regression test, and records the outbound/TTS fix in the changelog.

## Best Possible Solution

Keep the narrow outbound-send TTS integration, but make it side-effect-free for dry runs, pass the same scoped context used by the guard into the TTS runtime helper, add regressions for both cases, and avoid claiming native voice-note delivery until the shared `asVoice`/`audioAsVoice` work in #73483 lands.

## Reproduction Assessment

Yes. A high-confidence code reproduction exists: with the PR applied, a tagged-TTS `openclaw message send --dry-run` would reach `maybeApplyTtsToPayload` before the dry-run send shortcut, and an agent/account-scoped TTS config would pass the guard but be omitted from the helper call.

## Solution Assessment

No, not as currently written. Reusing `maybeApplyTtsToPayload` in `handleSendAction` is the right narrow direction, but the implementation should skip dry-run synthesis and preserve scoped TTS context before it is mergeable.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Skip TTS synthesis during dry runs:** `src/infra/outbound/message-action-runner.ts:591-594`
  - body: The new hook runs before `executeSendAction` reaches the existing dry-run shortcut, and this condition does not check `dryRun`. With tagged TTS configured, `openclaw message send --dry-run ...` can still import the TTS runtime and call the provider/write audio even though the CLI says dry-run prints the payload and skips sending. Please gate this path on `!dryRun` or make the helper dry-run aware, with a regression test.
  - confidence: 0.92
- **[P2] Preserve scoped TTS context for synthesis:** `src/infra/outbound/message-action-runner.ts:596-601`
  - body: `shouldAttemptTtsPayload` is evaluated with `agentId`, `channelId`, and `accountId`, but the actual `maybeApplyTtsToPayload` call drops `agentId` and `accountId`. In configs where TTS is enabled or has its provider only under `agents.list[].tts` or a channel account `tts` block, the guard can pass and then the helper resolves the root config, leaving the tagged text unchanged or using the wrong provider. Pass the same scoped context into the helper and cover an override case.
  - confidence: 0.9
- **[P3] Avoid overclaiming voice-note delivery:** `CHANGELOG.md:326`
  - body: The changelog says message-tool tagged TTS is synthesized into a voice note, but current `executeSendAction`/`sendMessage` do not propagate `asVoice`/`audioAsVoice` on the generic send path; that is still tracked by #73483. Until that lands, this PR can produce an audio attachment, so the release note should not promise native voice-note behavior.
  - confidence: 0.86
