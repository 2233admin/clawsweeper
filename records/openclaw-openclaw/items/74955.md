## Summary

Keep open. Current main still has a real path mismatch: message-tool sends parse and deliver outbound text without the TTS hook, while the auto-reply final path applies the hook before routing. The proposed patch looks directionally correct and I did not identify a blocking review finding from the provided diff, but it still needs normal maintainer review and CI validation.

## What This Changes

The PR adds a tagged-TTS hook to `handleSendAction`, adds a focused outbound regression test, and records the outbound/TTS fix in the changelog.

## Best Possible Solution

Land this hook or an equivalent narrow outbound-send integration after CI passes, and keep shared native voice-message passthrough tracked separately in #73483.

## Reproduction Assessment

Yes. A high-confidence code reproduction exists: configure tagged TTS, send `[[tts:text]]...[[/tts:text]]` through the message-tool/CLI send path, and current main reaches `executeSendAction` without invoking the TTS runtime hook.

## Solution Assessment

Yes. Reusing the existing `maybeApplyTtsToPayload` helper behind `shouldAttemptTtsPayload` inside `handleSendAction` is the narrowest maintainable fix for parity with auto-reply while preserving the explicit-media short circuit.
