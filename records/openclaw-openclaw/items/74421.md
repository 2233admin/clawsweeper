## Summary

Keep open. Current main contains the intended Telegram voice-note preflight path, shared media-understanding dispatch, docs, and mocked regression coverage, and the related #75084 report says Telegram DM transcription worked on v2026.4.27. That still does not fully clear this fresh v2026.4.26 shipped-runtime report, because the reporter says Telegram saved the audio file but no transcription or agent-context transcript was produced, and I did not find live or high-fidelity evidence that reproduces or disproves that exact failure.

## What This Changes

The issue reports that Telegram voice notes on OpenClaw 2026.4.26 are saved as inbound audio but are not transcribed into agent-context text despite `tools.media.audio` being configured for OpenAI Whisper.

## Best Possible Solution

Keep this issue open until a maintainer reproduces current-main or latest-release behavior with a real or high-fidelity Telegram voice update and OpenAI audio auth. If it reproduces, fix the narrow failing layer in Telegram media context, preflight invocation, shared media-understanding, provider/auth resolution, or config validation/docs, then add regression coverage for that exact runtime path.
