## Summary

Keep open. Current main still intentionally defaults group/channel source turns to message-tool-only delivery, so the reported visible final-reply suppression remains true; docs and current-main changelog text now explain more of the all-channel behavior, but the upgrade-warning, migration, and agent-facing config-mutation policy remain unresolved maintainer decisions.

## What This Changes

The issue reports that after upgrading to 4.27, group/channel chats across supported channels no longer receive automatic final replies by default and asks for clearer release guidance, easier config opt-out, reload guidance, and possible migration or warning behavior.

## Best Possible Solution

Keep the group-room safety default only if the upgrade path is explicit: user-facing release notes and docs should name all group/channel chats, the `messages.groupChat.visibleReplies: "automatic"` opt-out should be straightforward for operators to apply, and maintainers should intentionally decide whether existing installs get a warning, doctor check, migration, or no compatibility shim.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with empty config and `ChatType` set to `group` or `channel`, the resolver returns `message_tool_only`, and dispatch tests show final delivery is not queued until `messages.groupChat.visibleReplies: "automatic"` is configured.

## Solution Assessment

Unclear. The current implementation is coherent as a safety default, but it does not fully solve the upgrade surprise; the safest next step is maintainer policy review before changing migration behavior or the protected agent-facing config mutation allowlist.
