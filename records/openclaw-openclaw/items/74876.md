## Summary

Keep open. Current main still defaults empty-config group/channel source turns to message-tool-only delivery, so the reported automatic final-reply suppression remains real; docs now describe the all-channel behavior and the automatic opt-out, but the runtime warning, v2026.4.27 changelog wording, config.patch allowlist, and migration/default policy are not fully resolved. The follow-up PRs already linked from the issue should be reviewed rather than closing this issue early.

## What This Changes

The issue asks OpenClaw to address an upgrade surprise where group/channel final replies became private/message-tool-only by default across channels after 4.27, including warning, docs/changelog, config.patch access, reload guidance, and migration/default policy.

## Best Possible Solution

Land or explicitly close a coordinated maintainer-approved path that keeps the group-room visibility safety posture clear: accurate all-channel release-note wording, operator-accessible config for the automatic opt-out, correct hot-reload guidance, and either a runtime/startup warning or an intentional migration/default decision.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: with empty config and ChatType group or channel, source delivery resolves to message_tool_only and dispatch coverage verifies no automatic final reply is sent; setting messages.groupChat.visibleReplies to automatic restores final delivery.

## Solution Assessment

Unclear. The current implementation is maintainable as a safety default and docs now explain the all-channel behavior, but current main still does not fully solve the upgrade surprise because warning, allowlist, changelog, and migration/default decisions remain unresolved.
