## Summary

Keep open: current main still intentionally blocks ACP spawns from sandboxed sessions, and the request is a real security/architecture feature gap rather than a stale duplicate. The related earlier issue was closed after a reporter-specific workaround, while this issue and comments preserve a broader bridge/sidecar/nested-sandbox design problem that needs maintainer security review.

## What This Changes

Request sandboxed OpenClaw sessions to spawn and control ACP harness sessions without abandoning sandbox isolation, likely through an opt-in bridge, proxy, sidecar, or nested sandbox design.

## Best Possible Solution

Define a maintainer-approved, opt-in ACP bridge design that keeps sandboxed callers confined while exposing a narrow host-side spawn/steer/cancel surface with session ownership checks, agent allowlists, rate limits, audit logging, and clear docs/tests. The implementation likely needs a small core policy/API seam plus acpx or sandbox-backend support rather than a plugin-only workaround.

## Reproduction Assessment

Yes. The current-main reproduction is source-backed and unit-testable: create a sandboxed requester and call sessions_spawn with runtime="acp", which the tool and spawn policy reject before any ACP runtime starts. I did not run tests because this review was read-only, but the current tests assert the same path.

## Solution Assessment

Unclear. The feature goal is valid, but the proposed options choose different trust boundaries; a narrow opt-in bridge or nested sandbox model is more maintainable than simply relaxing the current block.
