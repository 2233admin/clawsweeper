## Summary

Keep open. Current main still has hook ingress branches that return 400/401/404/405/408/413 without a hook warning entry, and the related fix PR was closed unmerged, so the reported diagnosability gap remains actionable.

## What This Changes

The issue requests warning-level gateway journal entries, with branch-specific rejection reasons, for hook requests that return non-2xx responses.

## Best Possible Solution

Add a narrow gateway hook rejection logging helper in `src/gateway/server/hooks-request-handler.ts` and cover the main validation branches with tests. The log should use stable reason codes, path, status, content type, and safe body metadata where already available, while keeping sensitive payload and auth material out of the journal.

## Reproduction Assessment

Yes. Current main visibly reproduces the missing-diagnostic behavior by following the malformed JSON path through `readJsonBody` to `sendJson(..., 400, ...)` with no `logHooks.warn` call.

## Solution Assessment

Yes. The best fix is a focused handler-level diagnostic path plus tests; a plugin/API change would not address server-side journal visibility for rejected ingress requests.

Add warning-level logging for hook ingress requests that return non-2xx from `src/gateway/server/hooks-request-handler.ts`, especially malformed JSON/body read errors, query-token rejection, non-POST rejection, unauthorized/throttled auth, empty path/no route, direct wake/agent validation errors, mapped hook validation errors, invalid channel, disallowed agent, and session-key policy rejects. Keep the change gateway-owned and narrow; do not introduce core/plugin API changes. Add or update focused tests so at least malformed JSON and a mapped `hook mapping requires message` reject prove a warning is emitted, and retain existing response bodies/statuses.
