## Summary

Keep open. Current main still returns hook 4xx/401/404/405/408/413 responses from the gateway hook handler without a warning log, except for auth throttling and mapping exceptions. The linked fix PR was closed unmerged, so the diagnosability gap remains actionable.

## What This Changes

The issue requests warning-level gateway journal entries and more specific rejection reasons for hook requests that return non-2xx validation errors.

## Best Possible Solution

Add a small redacted rejection-logging helper inside the hook request handler, call it before every handler-owned non-2xx hook response, and add focused tests for invalid JSON plus mapped-hook validation failures. If distinct branch reason codes require API shaping, keep that internal to logs unless maintainers explicitly choose to change response strings.

## Reproduction Assessment

Yes. The issue's malformed JSON curl maps directly to the `readJsonBody` failure branch, which returns a 400 JSON response without any `logHooks.warn`; the existing gateway hooks test already exercises the same bad-JSON path.

## Solution Assessment

Yes. A handler-level redacted rejection logger is the narrowest maintainable fix because all of these hook rejection branches are centralized in `createHooksRequestHandler`, while mapped action validation already returns branch-specific errors that can be converted to internal reason codes.

In `src/gateway/server/hooks-request-handler.ts`, add warning-level, redacted logging for every hook request that the handler itself returns as non-2xx, especially query-token 400, method 405, unauthorized 401, empty/missing path 404, body read 400/408/413, wake/agent validation 400, mapping validation 400, channel/agent/session policy 400, and unmapped 404. Prefer one small helper that logs stable internal reason codes, status, path/subpath, content-type, body type/byte count when already known, and a request id if there is an existing safe request-trace source or a minimal generated id is appropriate.
