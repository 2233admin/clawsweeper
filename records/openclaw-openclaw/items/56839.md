## Summary

Keep open. Current main still has a plausible cron failure-notification path where a channel-prefixed Telegram target can be resolved through the WhatsApp outbound adapter, and WhatsApp target normalization still accepts that value as an E.164-like phone number when no allowlist blocks it.

## Best Possible Solution

Keep this issue open. The likely fix belongs at the cron delivery/target-resolution boundary and the WhatsApp owner surface: reject foreign channel prefixes before channel-specific normalization, require `delivery.to` prefixes to match `delivery.channel` or derive the channel explicitly, and add regression tests for cron failure notifications plus WhatsApp target normalization so `telegram:1234567890` cannot become a WhatsApp phone target.
