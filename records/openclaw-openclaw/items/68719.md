## Summary

Keep open. Current main still lacks the requested BlueBubbles-owned pre-dispatch audio transcript enricher: there is no client call for the BlueBubbles transcript endpoint, no `channels.bluebubbles.inboundAudioEnricher` config surface, no audio-UTI transcript rewrite before dispatch, and no scoped `message_transcribed` plugin hook. The existing shared media-understanding path can transcribe staged audio later, but that is not equivalent to using BlueBubbles' on-device transcript endpoint before the agent sees the message.

## What This Changes

The issue requests a BlueBubbles inbound audio enricher that fetches BlueBubbles voice-note transcripts before agent dispatch, rewrites empty or placeholder bodies, emits a scoped hook, and adds opt-out configuration.

## Best Possible Solution

Keep this open until BlueBubbles gets a plugin-owned implementation or maintainers explicitly document that shared `tools.media.audio` is the supported answer. The implementation should reuse the centralized BlueBubbles client auth/SSRF path, detect audio attachments by MIME and UTI before final inbound body construction, gate defaults on server version, avoid logging transcript content, document the new config, and add focused BlueBubbles tests for placeholder voice notes, opt-out behavior, and scoped hook payloads.
