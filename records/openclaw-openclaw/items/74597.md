## Summary

Keep open. The reporter narrowed the original host incident to a manual source-level global install, but current main still lacks the requested guard: update/status collapse a package-root symlink that resolves to a git checkout into `installKind: "git"`, and doctor only compares service entrypoint realpaths. Open PR #74920 appears to address the service-layout diagnostics, but it is still open and maintainer-labeled, so this issue should remain open until that work is reviewed, landed, split, or closed.

## What This Changes

Request diagnostic and guard behavior so npm-global package installs are not silently left as mutable source-checkout symlinks and update/status/doctor can detect mixed package/source layouts.

## Best Possible Solution

Land or split the service-layout diagnostic portion of #74920: record both lexical package roots and realpaths, detect when an npm-global package root points into a git/source checkout unexpectedly, and surface clear status, doctor, update verification, and repair guidance while preserving intentional dev-channel behavior.

## Reproduction Assessment

Yes. The current unit coverage already reproduces the key classification behavior by symlinking a root to a git repo and expecting `checkUpdateStatus` to report `installKind: "git"`; the reported host logs provide the real-world npm-global symlink scenario to validate against.

## Solution Assessment

Unclear for the full open PR until maintainer review, but the requested diagnostic guard is the right narrow direction. Closing this issue now is not the best solution because current main has not landed that guard and the related implementation is still open.
