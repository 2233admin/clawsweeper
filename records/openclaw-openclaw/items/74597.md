## Summary

Keep open. The follow-up comment narrows the original incident from an `openclaw update` direct cause to a manual source-level global install, but the remaining requested guard is still not implemented on current `main`: status/update detection resolves the active package root to a git checkout and treats it as a git install, while doctor only compares service entrypoint realpaths and does not flag an npm-global package root that is a source-checkout symlink.

## What This Changes

The issue requests safeguards so a package-installed gateway cannot silently remain wired through an npm-global `openclaw` symlink into a mutable source checkout, and so status/doctor/update report or repair that mixed install layout.

## Best Possible Solution

Add a small shared install-layout diagnostic that detects when the global package root path is a symlink or realpath into a git checkout while the configured channel/service/plugin layout expects a package install, then surface it in `openclaw status`/`status --json`, `openclaw doctor`, and update pre/post verification with clear repair guidance.
