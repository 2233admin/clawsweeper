## Summary

Keep open. Current main still gives already-built Discord native command objects an early defer, but Discord’s startup command list is still sourced from the runtime plugin command registry. Manifest `runtime-slash` aliases for startup-lazy plugins such as Codex are metadata/activation hints, not deployed handler-backed native command specs, so the specific plugin-registered command lifecycle reported here is not proven fixed.

## What This Changes

The issue asks OpenClaw to prevent Discord plugin slash commands from lazily registering during first interaction and missing Discord’s 3-second acknowledgement deadline.

## Best Possible Solution

Define one Discord-native readiness contract for plugin-owned slash commands: either predeploy manifest-declared runtime slash commands with a lightweight immediate-defer dispatcher that can activate the owning plugin, or only deploy commands whose handlers are already loaded before Discord readiness. Keep the voice `/vc` command path validated separately from generic plugin command dispatch.

## Reproduction Assessment

Unclear: the report has concrete logs and plausible steps, but this review did not run a live Discord bot. A high-confidence reproduction should enable Discord native commands with a startup-lazy runtime alias such as `/codex`, then capture startup command deployment, plugin registration, and received-to-defer timings.

## Solution Assessment

Unclear because there is no proposed patch and current main is not sufficient for startup-lazy plugin-owned native commands. The safer path is an explicit Discord-visible command readiness contract rather than broadly eager-loading every plugin at startup.
