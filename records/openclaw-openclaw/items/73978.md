## Summary

Keep open. Current main has early defer handling for Discord native commands that are already deployed and appends currently registered plugin command specs during Discord startup, but that does not prove the reported startup-lazy/plugin-registered regression is fixed. The issue is recent, concrete, and supported by plausible logs; the remaining gap is the command lifecycle for runtime plugin slash aliases such as Codex plus the separately implemented voice command path.

## What This Changes

The issue reports that Discord slash commands registered by plugins can miss Discord's 3-second interaction acknowledgement deadline, causing Unknown interaction or already-acknowledged errors.

## Best Possible Solution

Define one clear Discord native lifecycle for plugin-owned slash commands: either deploy only commands whose runtime handlers are ready before Discord gateway ready, or add a manifest-backed predeploy contract that immediately acknowledges and then activates the owning plugin. Keep the built-in /vc voice path as a separate check against Carbon subcommand defer/follow-up behavior and related voice-connect failures.

## Reproduction Assessment

Unclear: the reporter provides concrete logs and steps, but this read-only review did not perform a live Discord invocation. A high-confidence reproduction should start Discord with at least one startup-lazy runtime slash alias such as Codex plus a voice command, then measure command deployment timing and received-to-defer latency.

## Solution Assessment

Not applicable as a patch review. The suggested blanket eager registration is directionally plausible, but the maintainable fix should preserve plugin laziness where possible and make Discord-visible command readiness an explicit contract instead of importing every plugin runtime unconditionally.
