## Summary

Keep open. Current main still defers already-created Discord native commands before dispatch and can execute plugin commands that are already in the runtime registry, but Discord startup command deployment still appends only currently registered plugin command specs and does not preload startup-lazy runtime-slash command owners such as Codex. The only relevant post-comment main change touches Codex migration/provider metadata, not the Discord plugin command lifecycle, so the reported plugin-registered command acknowledgment regression remains active.

## What This Changes

The issue requests that Discord slash commands provided by plugins acknowledge within Discord's three-second interaction deadline instead of timing out when plugin command registration happens lazily during first interaction.

## Best Possible Solution

Define a Discord-native lifecycle for plugin-owned runtime-slash commands: either deploy only commands whose handlers are ready before gateway ready, or add a manifest-backed predeploy/activation contract that can acknowledge immediately and then load the owning plugin. Keep the voice /vc path as a separate smoke target for subcommand defer and voice-connect behavior.

## Reproduction Assessment

Unclear: the report has concrete logs, version, plugins, and steps, but this read-only review did not run a live Discord bot. A high-confidence reproduction should run Discord with commands.native enabled and a startup-lazy runtime-slash alias such as /codex, then record received-to-defer timing and command-registration timing.

## Solution Assessment

Unclear because there is no proposed patch. The immediate defer wrapper already exists for deployed native commands, so the safer solution is a readiness/deployment contract for runtime-slash plugin commands rather than blindly eager-loading every plugin at gateway startup.
