## Summary

Keep open. Current main still makes explicit `openclaw memory status --deep` perform an embedding readiness probe and then renders that provider/probe result as a generic `Embeddings: ready|unavailable` line beside persisted index counts. The adjacent doctor/status caching work reduces repeated live probes, but it does not remove the CLI diagnostic ambiguity reported here.

## What This Changes

Separate persisted memory index health from live embedding readiness in `openclaw memory status --deep` so transient probe failures do not look like index or content failure.

## Best Possible Solution

Update the memory diagnostics surface so persisted index counts, vector/FTS storage state, embedding cache state, and live or cached embedding-provider readiness are explicitly separate. A failed probe should read as provider/probe readiness failure while leaving populated index state visible, with CLI regression coverage and docs/changelog follow-through.
