## Summary

Keep open. Current main still makes `openclaw memory status --deep` run a live embedding probe through `probeEmbeddingAvailability()`, and that probe still calls `embedBatchWithRetry(["ping"])`. The CLI does show indexed counts separately and there is now an in-process 30s probe cache, but that does not fully address separate CLI invocations flapping or make the live-probe-vs-persisted-index distinction clear enough to close this issue.

## Best Possible Solution

Keep this open and land a focused memory-core diagnostics fix: make `memory status --deep` explicitly label the embeddings result as a live or cached provider probe, keep persisted index counts/source health separate, add failed-probe-with-populated-index regression coverage, and decide whether any persistent/strict-budget probe caching belongs in this command or in the gateway/doctor path.
