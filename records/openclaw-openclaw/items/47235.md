## Summary

Keep open. Current main still makes `openclaw memory status --deep` run an embedding readiness probe and render that provider/probe result as generic `Embeddings: ready|unavailable` beside persisted index counts. The adjacent cached `doctor.memory.status` work for #71568 helps a related status path, but it does not resolve this CLI diagnostic ambiguity.

## What This Changes

The issue asks for `memory status --deep` to distinguish persisted memory index health from transient live embedding-provider probe availability.

## Best Possible Solution

Update the memory diagnostics surface so persisted index presence/counts, vector storage state, and live or cached embedding-provider readiness are separate, explicitly named states. A failed probe should read like provider/probe availability, not like index corruption, with CLI regression coverage plus docs and changelog updates.
