## Summary

Keep open. Current main still treats top-level `openclaw.json` `env.vars` and env sugar as string-only environment injection, while SecretRef resolution is limited to registered credential/plugin surfaces such as ACPX MCP server env values. The related AWS provider and MCP SecretRef work is useful context, but it does not implement the requested global env SecretRef surface.

## What This Changes

The issue requests SecretRef support in top-level `openclaw.json` `env.vars` so dynamic providers such as exec-backed 1Password can populate runtime environment variables.

## Best Possible Solution

Implement this as a core config/secrets feature: widen `env.vars` to `SecretInput`, resolve those refs through the existing SecretRef runtime pipeline before env application, preserve dangerous-env guardrails, keep resolved plaintext out of persisted state, and update schema, target registry data, generated docs, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`: the schema/type surface rejects it as non-string, and the runtime collector skips non-string values if raw JSON reaches that path.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable solution should extend the existing SecretInput resolver and target registry rather than adding a 1Password-specific or exec-only path.
