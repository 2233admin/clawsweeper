## Summary

Keep open. Current main still treats top-level `openclaw.json` `env.vars` and env sugar as string-only environment injection, while SecretRef resolution is wired for credential/plugin surfaces such as ACPX MCP server env values. The requested exec-backed SecretRef workflow for global env vars remains valid and unimplemented, and the related MCP and AWS-provider work does not cover this broader surface.

## What This Changes

The issue requests SecretRef support in top-level `openclaw.json` `env.vars` so dynamic secret providers such as exec-backed 1Password can populate runtime environment variables.

## Best Possible Solution

Implement this as a core config/secrets feature: widen `env.vars` to `SecretInput`, resolve those refs through the existing runtime SecretRef pipeline before env application, preserve current dangerous-env guardrails, keep resolved plaintext out of persisted config and service snapshots, update schema/docs/generated SecretRef surface data, and add focused tests for env/file/exec refs plus unresolved-ref behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to put a structured `{source, provider, id}` SecretRef under `env.vars`: current schema/type definitions reject it as non-string, and the runtime collector skips non-string values if a raw JSON object reaches that path.

## Solution Assessment

Unclear because there is no proposed patch yet, but the requested direction is plausible. The maintainable path is to extend the existing SecretInput resolver and target registry for `env.vars`, not to add a 1Password-specific or exec-only special case.
