## Summary

Keep open. Current main still treats top-level `env.vars` as a string-only environment injection surface, while the SecretRef runtime, target registry, generated schema, and public docs cover other credential paths and ACPX MCP server env, not this global env block. The request is valid, but it changes a secrets-sensitive runtime/service environment boundary and needs maintainer/security design before implementation.

## What This Changes

The issue requests SecretRef support for top-level `openclaw.json` `env.vars` so runtime environment variables can be resolved from configured secret providers such as 1Password.

## Best Possible Solution

Implement this as a core config/secrets feature: widen the approved top-level `env.vars` surface to `SecretInput`, resolve refs through the existing SecretRef runtime pipeline before env application, keep resolved values runtime-only unless service-env persistence is explicitly approved, preserve blocked-env and unresolved-placeholder protections, and update schema, registry data, generated docs, public docs, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place a structured `{source, provider, id}` SecretRef under `env.vars`: current schema/type validation accepts only strings, and if raw JSON reaches runtime collection the non-string value is skipped.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable solution should extend the generic `SecretInput` resolver and target registry rather than adding a 1Password-specific or exec-only path.
