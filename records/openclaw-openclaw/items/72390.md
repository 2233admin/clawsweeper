## Summary

Keep open. Current main still does not support SecretRef values for top-level `openclaw.json` `env.vars`: the schema and public type remain string-only, runtime env collection ignores structured objects, the SecretRef runtime collectors omit this surface, and the docs only document narrower MCP server env SecretInput support. The related AWS provider PR was closed unmerged and does not supersede this request.

## What This Changes

Add SecretRef/SecretInput support to top-level `openclaw.json` `env.vars` so environment values can be resolved from external providers such as exec-backed 1Password at runtime.

## Best Possible Solution

Implement this as a core config/secrets contract change: widen `env.vars` and top-level env sugar to `SecretInput`, collect `env.vars.*` assignments in the runtime SecretRef resolver, apply only resolved string values through the existing env filters/no-overwrite rules, update the canonical SecretRef surface/docs/schema metadata, and add focused tests for env/file/exec refs plus unresolved-ref and service-env behavior.

## Risks / Open Questions

- This feature would materialize externally resolved secrets into runtime environment variables, so maintainers should confirm that top-level `env.vars` is an acceptable SecretRef credential surface and avoid writing resolved plaintext into config or service metadata.
- The implementation needs careful ordering: resolve SecretRefs before applying env vars while preserving current no-overwrite behavior, dangerous env var filtering, portable key filtering, and unresolved `${VAR}` placeholder guards.
