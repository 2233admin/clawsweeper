## Summary

Keep open. Current main still treats top-level `openclaw.json` `env.vars` and env sugar as string-only runtime environment injection, while SecretRef collection/resolution is registered for other credential surfaces and plugin-declared env paths, not global config env vars.

## What This Changes

Add SecretRef support to the top-level `openclaw.json` `env.vars` surface so externally managed secrets can be resolved into runtime environment variables.

## Best Possible Solution

Implement this as a core config/secrets feature: widen the approved env-var value type to `SecretInput`, collect and resolve refs through the existing SecretRef runtime pipeline before env application, preserve dangerous-env guardrails and non-overwrite precedence, keep resolved plaintext runtime-only, and update schema, generated docs, public docs, registry metadata, and focused tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to place a structured `{ source, provider, id }` SecretRef under `env.vars`: current schema/type validation rejects it as non-string, and the runtime collector skips non-string values if raw JSON reaches that path.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the maintainable solution should extend the generic `SecretInput` resolver and registry rather than adding a 1Password-specific or exec-only path.
