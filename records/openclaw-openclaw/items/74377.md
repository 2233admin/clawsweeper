## Summary

Keep open. Current main still has the same runner/provider boundary described in the report: the embedded runner computes and registers a non-empty tool set, but the Anthropic transport only serializes the tools that arrive in `context.tools`, and I found no merged source or regression proof that Telegram/direct-Anthropic turns preserve that array through the Pi session handoff. The related open #70790 is useful context for strict providers rejecting empty tool payloads, but it does not restore missing Anthropic tool availability.

## What This Changes

Ensure Telegram-channel turns using direct Anthropic preserve OpenClaw's computed tool definitions through the embedded runner and provider stream context.

## Best Possible Solution

Add a focused regression for the Telegram/direct-Anthropic embedded runner path that intercepts the provider stream context and asserts OpenClaw tool definitions reach `context.tools`, then fix the Pi session or stream handoff so a non-empty active tool set cannot be cleared after `effectiveTools` is computed.

## Reproduction Assessment

Yes. The issue provides a high-confidence live reproduction path with channel, provider, config, versions, and debug probes; current main still needs an automated reproduction around the provider stream context.

## Solution Assessment

Yes for the proposed direction, not for the empty-payload workaround. The maintainable fix is to preserve the non-empty tool surface through session state into `context.tools`, while #70790-style stripping is only defensive egress cleanup after tools are already lost.
