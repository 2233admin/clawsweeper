## Summary

Keep open. Current main still builds and allowlists the embedded runner's effective tools, while the Anthropic transport only serializes tools from the provider-facing `context.tools`; I found no current source or regression test proving that a Telegram/direct-Anthropic turn reaches that boundary with a non-empty tools array. The related open #70790 addresses providers rejecting empty tool payloads, but it would not restore missing Anthropic tool availability for this report.

## What This Changes

Telegram/direct Anthropic turns should pass the non-empty OpenClaw tool set computed by the embedded runner through to the provider so exec and other tools are available.

## Best Possible Solution

Add a focused regression that simulates the Telegram/direct-Anthropic embedded runner handoff and captures the context passed to the provider stream, then fix the runner/session boundary so a non-empty `effectiveTools` set remains active in Pi and arrives as non-empty `context.tools`. Do not treat empty-payload stripping as the primary fix for this report.
