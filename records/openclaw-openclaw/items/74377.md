## Summary

Keep open. Current main still has the unresolved Telegram/direct-Anthropic tool handoff gap: the runner computes and registers effective tools, but direct Anthropic can still use the session-managed Pi stream where the provider payload depends on Pi's `context.tools`. I found no source or regression test proving that the reported channel/model path reaches the provider with non-empty tool definitions, and the related open #70790 strips empty tool payloads rather than restoring missing tools.

## What This Changes

The issue reports that Telegram channel runs using direct Anthropic compute 17 tools in the embedded attempt runner but send an empty tools array to the Anthropic provider, preventing real tool calls.

## Best Possible Solution

Add a focused regression that simulates a Telegram/channel embedded run with direct Anthropic and coding tools enabled, asserts the provider-facing stream context receives the expected non-empty tool definitions, then fix the embedded session or stream handoff so non-empty `effectiveTools` cannot be lost before Anthropic sees the request.
