## Summary

Keep open. The report is fresh, concrete, and not covered by current main: OpenClaw still computes and registers a non-empty embedded tool set, but the Anthropic transport only serializes whatever Pi passes in `context.tools`, and there is no regression test proving a Telegram/direct-Anthropic run reaches that transport with non-empty tools.

## What This Changes

Telegram/direct-Anthropic embedded runs should pass the computed coding tools through to the Anthropic provider instead of sending an empty tools array after the runner has built a non-empty tool set.

## Best Possible Solution

Add a focused regression that simulates a Telegram/channel embedded run with direct Anthropic and coding tools enabled, asserts the provider-facing stream context receives the expected non-empty tool definitions, then fix the embedded session or stream handoff so Pi and Anthropic cannot observe an empty tool surface after `effectiveTools` is non-empty.

## Reproduction Assessment

Yes. The issue gives a concrete Telegram/direct-Anthropic path plus debug evidence showing 17 computed tools before the provider boundary and zero tools at `context.tools`; current main has no countervailing regression proof.

## Solution Assessment

No. Current main registers a session allowlist but does not prove or enforce non-empty provider context tools, and the adjacent empty-tools stripping PR solves provider rejection of empty arrays rather than restoring tool availability for this Anthropic path.
