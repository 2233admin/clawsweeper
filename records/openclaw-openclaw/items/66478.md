## Summary

Keep this PR open. Current main still has the reported Feishu reaction bug, and the PR is the right narrow area, but the branch should not merge as-is because the new normalizer uses non-documented Feishu type names for several Unicode inputs and also pulls the Feishu reaction runtime module into the hot channel entrypoint.

## What This Changes

The PR adds Feishu reaction emoji normalization, applies it to add/list reaction API helpers and the channel react action, adds focused reaction tests, and records the fix in the changelog.

## Best Possible Solution

Revise this PR so Feishu reaction normalization uses the official exact emoji_type vocabulary, preserves documented mixed-case type strings, and keeps normalization in a runtime-free helper or entirely inside the lazy reaction helpers before landing with focused regression coverage.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: a `message` tool react call with `emoji: "👍"` is trimmed and forwarded unchanged to Feishu's reaction API fields, while Feishu documents that the API expects an emoji_type value such as `THUMBSUP` and returns 231001 for invalid types.

## Solution Assessment

No, not as currently patched. Normalizing in the Feishu plugin is the right boundary, but the implementation should use documented Feishu type names and avoid statically importing the runtime reaction module into `channel.ts`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use Feishu's documented emoji_type values:** `extensions/feishu/src/reactions.ts:163-188`
  - body: The normalizer maps several Unicode inputs to guessed uppercase names, but Feishu's current emoji reference uses exact mixed-case values for some of them, such as `ThumbsDown`, `Fire`, `CheckMark`, and `CrossMark`. Those inputs will still be sent as invalid reaction types and can keep producing 231001; valid user-supplied mixed-case Feishu values like `Fire` are also uppercased to `FIRE`. Please build the map and known-type pass-through from the documented Feishu vocabulary, or limit the map to verified values.
  - confidence: 0.93
- **[P2] Keep normalization off the hot channel import path:** `extensions/feishu/src/channel.ts:63`
  - body: Importing `normalizeFeishuEmoji` from `./reactions.js` makes `channel.ts` statically load the reaction runtime module; that module imports `client.ts`, which imports the Lark SDK and changes its HTTP interceptor at module evaluation. This defeats the existing `channel.runtime.ts` lazy boundary for reaction API work. Since the PR already normalizes inside `addReactionFeishu` and `listReactionsFeishu`, remove this channel-level import or move only the pure normalizer into a small runtime-free helper.
  - confidence: 0.9
