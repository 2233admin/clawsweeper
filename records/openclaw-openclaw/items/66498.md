## Summary

Keep open. Current main still rejects Gemini `web_search` freshness before building a Gemini request, and the request body still sends an empty Google Search grounding tool with no time range. The broader open #17925 is adjacent, but it is not a canonical replacement for this narrower Gemini freshness gap. Google’s current API reference documents a `GoogleSearch.timeRangeFilter` interval, so the request remains actionable in the Google plugin. ([ai.google.dev](https://ai.google.dev/api/caching?hl=id))

## What This Changes

Request to pass the `web_search` freshness parameter through Gemini Google Search grounding instead of returning an unsupported-filter error.

## Best Possible Solution

Implement the support in the Google plugin: convert supported OpenClaw freshness/date inputs into Gemini Google Search grounding time ranges, keep unrelated provider filters unsupported, include the normalized time range in the Gemini cache key, and update focused tests plus `docs/tools/gemini-search.md` and `docs/tools/web.md`.

## Repair Work Prompt

Fix the Gemini `web_search` freshness gap tracked by this issue. Work in the Google plugin boundary. Verify Google’s current Gemini API contract first, then map supported OpenClaw `freshness` values (`pd`/`pw`/`pm`/`py` and normalized `day`/`week`/`month`/`year`) into a Gemini Google Search grounding time range using the documented interval fields. Handle `date_after`/`date_before` only when they can be mapped faithfully under Gemini’s requirement that both start and end are present; keep `country`/`language` unsupported with clear existing-style errors.
