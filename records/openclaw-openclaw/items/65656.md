## Summary

Keep this open. Current main still matches the reported LINE delivery bug: Markdown tables are extracted into Flex messages, the text reply consumes the reply token, and the generated table/card messages are then sent through the Push API path; the directly related fix PR #65673 is closed and unmerged.

## What This Changes

The issue requests that LINE responses containing both explanatory text and Markdown tables deliver the generated table Flex message in the same reply interaction instead of forcing it through Push API.

## Best Possible Solution

Fix this inside the LINE plugin delivery path by allowing generated table/code Flex messages and media messages to fill unused Reply API slots when text chunks are being replied to, then push only true overflow while preserving quick-reply ordering semantics.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is available by calling deliverLineAutoReply with text that processLineMessage splits into cleaned text plus a table Flex message; current tests already assert the resulting replyMessageLine-for-text plus pushMessagesLine-for-Flex split. Live LINE quota exhaustion is only needed to observe the external 429 symptom.

## Solution Assessment

Yes, the issue's proposed direction is the right maintainable fix, but a simple false-to-true change after sendLineReplyChunks is not sufficient because the helper has already marked the reply token used. The safer fix is to extend sendLineReplyChunks or its caller to include extra Flex/media messages in the first reply batch and return the number consumed.
