## Summary

Keep open. Current main still keeps RTL RLI/PDI isolation private to the TUI formatter, while shared assistant-visible/user-facing sanitizers, gateway live projection, plain-text channel sanitization, and outbound delivery still forward RTL text without a shared bidi-isolation stage. The related #21936 fix proves the desired behavior for TUI only, and #32852 is a Telegram preview-specific prior report rather than a canonical outbound-boundary fix.

## What This Changes

The issue asks OpenClaw to apply Unicode bidi isolation to Hebrew/Arabic text at the shared gateway/outbound reply boundary instead of only in TUI rendering.

## Best Possible Solution

Add a shared RTL isolation helper under src/shared/text, reuse it from the TUI formatter, and apply it at the canonical user-visible outbound boundary before channel chunking/sending; also cover intended gateway live projection behavior if that surface should receive the same treatment. Keep the helper conservative, line-based, and no-op for existing bidi controls.
