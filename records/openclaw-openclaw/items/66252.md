## Summary

Keep open. Current main implements the TTS half, but the STT half remains unresolved: agent entries expose `tts` only, the strict agent schema has no `stt` key, and batch audio transcription still reads the global `tools.media.audio` config without an active-agent overlay. The issue discussion already captures this split, including a maintainer comment that TTS landed while STT needs separate contract/runtime review.

## What This Changes

Add per-agent TTS and STT overrides so different agents can use different speech providers, voices, languages, and transcription settings.

## Best Possible Solution

Keep this open until OpenClaw has a backward-compatible per-agent STT contract in schema/types/docs and the batch transcription runtime resolves an effective audio config for the active agent across preflight and normal media-understanding paths.

## Reproduction Assessment

Yes. A static reproduction is enough: add `agents.list[].stt` as proposed and current strict agent schema has no accepted key, while transcription still reads only global `tools.media.audio` at runtime.

## Solution Assessment

No. The current TTS implementation is the right direction for voices, but it does not solve STT language/provider overrides; the narrow maintainable fix is a separate STT config contract plus shared agent-aware audio config resolution.
