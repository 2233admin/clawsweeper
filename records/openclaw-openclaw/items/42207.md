## Summary

Keep this PR open. Current main still defaults `tools.fs.workspaceOnly` to unrestricted behavior unless explicitly set to true, still documents that default, and has no host filesystem sensitive-path denylist. The PR targets a real hardening gap, but its current diff is not safe to treat as done because review context correctly flags the unguarded host read path and symlink-bypassable denylist design.

## Best Possible Solution

Keep this PR open as an implementation candidate, or replace it with a maintainer-owned hardening patch. The best path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, public security docs, changelog, and regression tests together. If a sensitive-path denylist is kept, enforce it consistently for read/write/edit and make path checks symlink-safe before file access. The same remaining product request is also tracked by #58947.

## Risks / Open Questions

- Merging this PR as supplied could create false assurance because the normal host `read` tool can still read sensitive host paths when `tools.fs.workspaceOnly=false`.
- The proposed denylist design is symlink-bypassable unless existing paths are canonicalized with realpath or an equivalent safe-open strategy before the check and file access.
- Changing `tools.fs.workspaceOnly` from default false to default true is a compatibility and security-policy change that needs aligned config schema/help, docs, changelog, and maintainer review.
