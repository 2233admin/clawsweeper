## Summary

Current main does not implement this PR's requested hardening. `tools.fs.workspaceOnly` still defaults to unrestricted behavior when unset, host read/write/edit paths still allow out-of-workspace access when unrestricted, and there is no general sensitive-path denylist for host filesystem tools. The PR is a plausible security-hardening candidate, but the review context flags real gaps in the proposed diff, so it should stay open for maintainer follow-up rather than be closed as obsolete or implemented.

## Best Possible Solution

Keep this PR open as a security-hardening implementation candidate, or replace it with a maintainer-owned patch. The best path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, public security docs, changelog, and regression tests together. If a sensitive-path denylist remains part of the design, enforce it consistently for read/write/edit and make checks symlink-safe before file access. Coordinate with #58947 for the same remaining default-workspace-only request.

## Risks / Open Questions

- Changing `tools.fs.workspaceOnly` from default false to default true is a compatibility and security-policy change that needs maintainer review and aligned config, schema/help, docs, changelog, and tests.
- Merging this PR as supplied could create false assurance because the host read tool would still be able to read sensitive host paths when `tools.fs.workspaceOnly=false`.
- The proposed denylist needs symlink-safe canonicalization before file access if it is kept as part of the design.
