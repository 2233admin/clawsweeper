## Summary

Keep this PR open. Current main still implements and documents `tools.fs.workspaceOnly` as default-false/unrestricted behavior, has no sensitive-path denylist for host filesystem tools, and routes non-workspace-only host reads through the plain read tool path. The PR targets a real hardening gap, but the supplied diff is not safe to close or treat as superseded because review comments correctly flag that it leaves the main credential-exfiltration read path unguarded and uses symlink-bypassable lexical path checks.

## Best Possible Solution

Keep this PR open as an implementation candidate, or replace it with a maintainer-owned hardening patch. The best path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, public security docs, changelog, and regression tests together. If a sensitive-path denylist is kept, enforce it consistently for read/write/edit and make path checks symlink-safe before file access.

## Risks / Open Questions

- Merging the PR as supplied could create false assurance because the read tool can still read sensitive host paths when an operator explicitly disables workspace-only mode.
- The proposed denylist is lexical and symlink-bypassable unless the implementation canonicalizes existing paths with realpath or an equivalent safe-open strategy before checking prefixes/patterns.
- Changing `tools.fs.workspaceOnly` from default false to default true is a compatibility and security-policy change that needs aligned config schema/help, docs, changelog, and maintainer review.
- A denylist is brittle as the primary secret-protection boundary; if retained, it should be layered behind workspace-only defaults and enforced consistently for read, write, edit, native media/path-loading, and relevant sandbox bridge paths.
