## Summary

Keep open. Current main still returns assistant/user text from the session transcript unchanged and writes that string into the session-memory Markdown file unchanged; the issue is also paired with two open closing PRs, so it should remain open until a fix merges or the PRs are closed/replaced.

## What This Changes

Sanitize session-memory transcript content before persistence so raw chat-template tokens, orphaned role markers, raw tool-call XML, and NO_REPLY artifacts are not re-injected into future sessions.

## Best Possible Solution

Land one narrow session-memory fix that sanitizes at the transcript extraction and/or write boundary, with regression tests for chat-template tokens, raw `<tool_call>` XML, orphaned role markers, and NO_REPLY-only turns. Keep the broader historical-session untrusted wrapping concern coordinated with #68751 rather than mixing unrelated product behavior into this fix.

## Reproduction Assessment

Yes. A synthetic session JSONL containing assistant text such as `<tool_call>...</tool_call><|im_end|>` would flow through `getRecentSessionContent` unchanged and then be written under `## Conversation Summary` by the handler on `/new` or `/reset`.

## Solution Assessment

Yes, sanitizing before session-memory persistence is the narrowest maintainable fix for the reported loop. The current main implementation does not do that yet, so the right next step is to review and land one of the linked PRs after confirming its sanitizer scope.
