## Summary

Current main still does not implement the requested browser upload fallback for non-standard/custom uploaders such as ByteUploader. The browser upload tool, hook route, Playwright helpers, Chrome MCP delegation, schema, CLI, docs, and tests all remain centered on standard file chooser or `<input type="file">` flows, and the prior ClawSweeper review reached the same result. The related open upload issue #38844 covers stale-ref/timeout behavior, not DataTransfer/paste/CDP drag fallback support.

## Best Possible Solution

Keep this issue open and implement a generic browser-plugin upload fallback in `extensions/browser` rather than a ByteDance-specific hardcode. The implementation should keep the current guarded standard upload flow, then add an explicit custom-uploader strategy such as DataTransfer/drop, paste, or CDP drag with target/ref scoping, upload-root path validation, clear success/failure reporting, docs, and tests for a custom uploader whose `files.length` remains zero after the standard path.

## Risks / Open Questions

- A custom uploader fallback would touch a security-sensitive browser/file boundary, so it should preserve the existing uploads-root validation and avoid widening local file exposure.
- The issue names ByteUploader but there is no in-repo ByteUploader reproduction fixture; implementation should add a generic synthetic custom-uploader test rather than hard-coding a ByteDance site.
- A fallback that returns `ok: true` without verifying the target received files could preserve the current false-success symptom.
