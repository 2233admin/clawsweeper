## Summary

Current main still does not implement the requested browser upload fallback for non-standard/custom uploaders such as ByteUploader. The browser upload surface still routes through standard file chooser, direct `<input type=file>` setInputFiles, or Chrome MCP upload_file paths, with no DataTransfer, paste, or CDP drag fallback strategy.

## Best Possible Solution

Keep this open and implement a generic browser-plugin upload fallback, not a ByteDance-specific hardcode. The best path is to preserve the guarded standard upload flow first, then add an explicit custom-uploader strategy such as DataTransfer/drop, paste, or CDP drag with target/ref scoping, upload-root path validation, clear success/failure reporting, docs, and regression tests for a custom uploader where the standard setInputFiles/filechooser path leaves files.length at zero.
