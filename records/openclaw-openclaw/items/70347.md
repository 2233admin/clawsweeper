## Summary

Keep open. Current main has partial cron timeout cleanup, but the outer-timeout winner still returns a plain timeout error after abort/cleanup and does not emit a terminal lifecycle event, so the reported synchronous session-finalization gap remains.

## What This Changes

The issue requests that started isolated cron runs emit a terminal lifecycle error event when the outer timeout fires so `sessions.json` finalizes immediately.

## Best Possible Solution

Add a cron-owned terminal `lifecycle` `error` emission for started timed-out isolated agent runs, using the active run session id/key and timeout metadata, and let the existing gateway lifecycle persistence path update session state.

## Reproduction Assessment

Yes. A focused unit harness can call `onExecutionStarted` with a session id/key, keep `runIsolatedAgentJob` unresolved, advance the cron timeout, and observe that current main records cleanup/error without any lifecycle-finalization assertion.

## Solution Assessment

Yes. Reusing the existing agent-event lifecycle persistence path is the narrow maintainable solution; a direct `sessions.json` write from cron would duplicate gateway session lifecycle logic.

## Repair Work Prompt

Fix issue #70347: when a started isolated cron agent run hits the outer cron timeout, emit a terminal lifecycle error event so gateway session persistence finalizes the active session immediately. Keep the fix in the cron/gateway lifecycle boundary: use the active execution identity captured by `onExecutionStarted` (`sessionId` as run id and `sessionKey` as session key), include timeout metadata such as `error: "cron: job execution timed out"`, `errorKind` or equivalent timeout classifier if supported by the existing event schema, `stopReason: "timeout"`, and `endedAt`, and route through the existing agent-event/session lifecycle path instead of writing `sessions.json` directly. Preserve #41783 behavior: do not start the execution timeout while queued, and do not emit a terminal lifecycle event before execution identity exists. Make terminal emission idempotent so a later-draining embedded run cannot duplicate final chat/session state. Add focused regression coverage in `src/cron/service/timer.regression.test.ts` and/or `src/gateway/server-chat.agent-events.test.ts` proving a timed-out started isolated run emits/persists a terminal lifecycle update with the active session key. Likely files: `src/cron/service/timer.ts`, `src/cron/service/state.ts`, `src/cron/types.ts`, `src/gateway/server-cron.ts`, `src/cron/isolated-agent/run.ts`, `src/cron/service/timer.regression.test.ts`, and possibly `src/gateway/server-chat.agent-events.test.ts`. Add a changelog fix entry for the user-visible reliability behavior with `Thanks @holdenjrussell` unless maintainers decide this is internal-only. Do not change cron lane concurrency, queue-wait timeout semantics, provider fallback behavior, or direct session-store schema beyond what lifecycle persistence already supports.
