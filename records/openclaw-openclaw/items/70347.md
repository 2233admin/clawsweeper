## Summary

Keep open. Current main still has the reported cron outer-timeout gap: the wall-clock timeout aborts and rejects the cron run, but that timeout path does not emit a terminal lifecycle event or otherwise persist the active run session as ended/failed in-band.

## Best Possible Solution

Implement the fix in the cron-owned path: preserve the active run session identity for isolated cron executions and, when the outer timeout fires after session start, publish a terminal lifecycle error/end event with timeout metadata so the existing gateway lifecycle persistence path finalizes sessions.json immediately. Add a focused regression test that forces an isolated cron timeout after session start and asserts the matching session row becomes terminal without waiting for reaper/watchdog repair.

## Evidence

There is no lifecycle event emission or session persistence call in this path.
