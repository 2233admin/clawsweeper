## Summary

Keep open. Current main still has the reported gap: the cron timeout winner aborts and runs cleanup, but it returns a plain cron error without emitting a terminal lifecycle event or carrying the active session identity into the finished cron event, while session-store finalization remains driven by lifecycle events.

## What This Changes

Request that cron outer timeouts emit a terminal lifecycle error with timeout metadata and the active cron session identity so `sessions.json` finalizes immediately.

## Best Possible Solution

Add a cron-owned terminal lifecycle emission for started timed-out isolated runs, routed through the existing agent event/session lifecycle pipeline with the active `sessionKey`/`sessionId`, `stopReason: "timeout"`, and timeout-specific metadata, while preserving the deferred timeout start and forced cleanup behavior from #41783.

## Reproduction Assessment

Yes. A focused unit harness can make `runIsolatedAgentJob` call `onExecutionStarted` with a session id/key and then never settle; after advancing the cron timeout, current main records the cron timeout and cleanup call without any lifecycle finalization assertion.

## Solution Assessment

Yes. Reusing the existing lifecycle persistence path is the narrow maintainable fix; adding a separate direct `sessions.json` write from cron would duplicate gateway lifecycle semantics and increase drift risk.

## Repair Work Prompt

Fix the cron outer-timeout lifecycle gap for https://github.com/openclaw/openclaw/issues/70347. Current main aborts and cleans up a timed-out isolated agent run but returns a plain cron error without emitting a terminal lifecycle event or preserving `sessionId`/`sessionKey` in the timeout result. Implement a narrow, idempotent path that emits or routes a terminal lifecycle error/end event for started timed-out isolated cron runs through the existing gateway agent-event/session lifecycle persistence flow, including the active `sessionKey`/`sessionId` and timeout metadata such as `stopReason: "timeout"` and an appropriate timeout error/errorKind. Preserve #41783 behavior: do not restart the execution timeout before actual isolated execution starts, and keep forced cleanup for runners that ignore abort. Do not add a second bespoke direct `sessions.json` write path if the existing lifecycle persistence path can be reused.
