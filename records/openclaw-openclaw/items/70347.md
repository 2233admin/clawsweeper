## Summary

Current main still has the reported cron timeout lifecycle gap. The cron outer timeout aborts and rejects the run, but it does not preserve the prepared isolated cron session identity or emit a terminal lifecycle event, so the existing gateway lifecycle persistence path cannot finalize the matching session row in-band. The prior discussion around #67785 is useful context, but that PR fixed nested-lane scoping and did not address this timeout finalization path.

## Best Possible Solution

Keep this issue open and implement a narrow cron-owned timeout finalization path. Preserve or surface the prepared isolated cron run session identity once run context is created, and when the outer wall timeout fires after execution start, emit one terminal lifecycle event with timeout metadata through the existing agent event bus so gateway lifecycle persistence finalizes the session row immediately. Add focused regression coverage for a started isolated cron run timing out and the matching session lifecycle being finalized without watchdog repair.
