## Summary

Keep open. Current main partially recognizes `GOOGLE_CLOUD_API_KEY` for direct `google-vertex` auth lookup, but it does not fully implement this PR’s intended change: the key is still absent from the generic provider env-var metadata used for broad secret scrubbing and shell-env discovery, and the user-facing provider docs still describe Vertex auth as gcloud ADC only.

## Best Possible Solution

Keep this PR open and review/rebase it against current main. The best implementation is to preserve the current direct `google-vertex` resolver behavior while adding the missing generic metadata/tests: register `GOOGLE_CLOUD_API_KEY` for `google-vertex` in the manifest or provider env-var surface that feeds `listKnownSecretEnvVarNames()`, `listKnownProviderAuthEnvVarNames()`, and `resolveShellEnvExpectedKeys()`; update the Vertex docs; and make the auth source label distinguish explicit env API-key auth from ADC. Then get the requested secops review before merge.
