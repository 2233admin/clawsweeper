## Summary

Keep this PR open. Current main now has a direct `google-vertex` path that reads `GOOGLE_CLOUD_API_KEY`, but it still does not fully implement the PR’s intended behavior: the key is not in the generic provider env-var metadata that feeds broad secret scrubbing and shell-env discovery, the docs still say Vertex uses ADC only, and the explicit API-key path is still labeled as `gcloud adc` in auth status.

## Best Possible Solution

Keep this PR open and rebase/review it against current main. The best implementation is to preserve the current direct `google-vertex` resolver behavior, add `GOOGLE_CLOUD_API_KEY` to the manifest/setup or provider env-var metadata path that feeds `resolveProviderAuthEnvVarCandidates()`, `listKnownSecretEnvVarNames()`, and `resolveShellEnvExpectedKeys()`, update the Vertex docs, make the auth source label distinguish explicit env API-key auth from ADC, keep focused tests for both auth and scrubbing metadata, and get secops review before merge.
