## Summary

Keep this PR open. Current main still writes timestamped `.clobbered.*` snapshots in both config persistence paths without any per-path cap, while the existing suspicious-read dedupe is keyed by `current.hash:reasons` and therefore does not bound drifting payloads. The PR remains a plausible defensive fix, but the open functional review concern around count-then-write concurrency should be resolved before landing.

## Best Possible Solution

Keep this PR open. The best path is to finish a defensive per-path clobber snapshot cap in both `src/config/io.ts` and `src/config/io.observe-recovery.ts`, address the count/write race with a per-path serialization or atomic slot strategy, preserve audit logging after the cap is reached, and land focused async/sync regression coverage for max file count, bounded warning output, and continued audit records.

## Risks / Open Questions

- The proposed async implementation counts existing files and then writes the new snapshot in separate operations; concurrent suspicious reads for the same path can overshoot 32 unless the cap check and slot allocation are serialized or made atomic.
- Because the cap intentionally skips later forensic snapshots, tests should prove audit records still capture every suspicious observation even after snapshot persistence is suppressed.
- The warning-dedup change should stay covered for both async and sync paths so the fix does not trade unbounded files for unbounded warning logs.
