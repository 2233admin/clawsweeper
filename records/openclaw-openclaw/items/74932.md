## Summary

Keep this PR open. It targets the right heartbeat model-selection path, but the proposed stale-auto-fallback check depends on fallback notice fields that current runtime can clear while leaving the auto override persisted, so the linked heartbeat pinning bug can still reproduce for long-lived sessions without that notice metadata.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice selected-model metadata, threads heartbeat context through reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Persist durable auto-fallback origin metadata, or an equivalent explicit changed-default marker, and use it to clear only stale direct heartbeat auto overrides after the configured default changes while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel model selection, and unchanged-default auto fallback persistence.

## Reproduction Assessment

Yes. Seed a direct session override with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then run heartbeat model selection after changing agents.defaults.model.primary; the PR detector returns false and keeps the old fallback selected.

## Solution Assessment

No. The PR changes the right area, but fallback notice fields are transient status state; the safer solution needs durable origin/default metadata or a deliberately defined compatibility rule for existing missing-metadata entries.
