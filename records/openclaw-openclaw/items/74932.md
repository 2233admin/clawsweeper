## Summary

Keep this PR open. It targets the right heartbeat/model-selection path, but the proposed stale check depends on transient fallback notice fields that current runtime code can clear while leaving the persisted auto override in place, so the linked heartbeat pinning bug can still remain after a successful fallback lifecycle.

## What This Changes

The PR adds heartbeat-only stale auto-fallback override detection and clearing based on fallback notice selected-model metadata, threads heartbeat context through reply model selection, and updates focused tests, model-failover docs, and the changelog.

## Best Possible Solution

Record durable auto-fallback origin metadata, or define an explicit compatibility rule for existing auto entries without that metadata, and use it to clear only stale direct heartbeat auto overrides while preserving explicit heartbeat.model, user and legacy overrides, parent inheritance, channel selection, and unchanged-default fallback persistence.

## Reproduction Assessment

Yes. Seed a direct heartbeat session entry with providerOverride/modelOverride/modelOverrideSource="auto" but no fallbackNoticeSelectedModel, then change agents.defaults.model.primary and run heartbeat model selection without heartbeat.model; the proposed code keeps the old fallback override.

## Solution Assessment

No. The implementation is scoped to the right files, but fallbackNoticeSelectedModel is not durable origin metadata because current runtime clears it when selected and active models converge while the auto override remains persisted.
