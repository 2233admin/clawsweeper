## Summary

Current main still lacks the requested `sessions_send` guard. The tool resolves the target, checks visibility, and starts an internal `agent` run with the canonical `resolvedKey`; it does not refuse or warn for thread-shaped session keys, even though shared parsers and docs recognize thread session keys.

## Best Possible Solution

Keep this open and add a focused guard in `createSessionsSendTool` after canonical target resolution/visibility and before any `agent` call. Use the shared session-thread parser on the canonical target key, return a clear error telling agents to use the parent channel session key, and add tests for raw key, label, and sessionId resolution while preserving thread-bound announce delivery behavior.

## Risks / Open Questions

- The guard should run after canonical session resolution and visibility checks so raw `sessionKey`, label resolution, and sessionId resolution cannot bypass it.
- The guard must be scoped to direct `sessions_send` targets; existing announce, completion, cron, and channel delivery paths intentionally preserve `threadId` and should keep working.
- A regression test should cover thread-shaped raw keys plus resolved label/sessionId targets, not only exact string matching on one Slack example.
