## Summary

Current main still lacks the requested `sessions_send` guard. The tool canonicalizes a raw key, label, or sessionId, checks visibility, then starts an internal `agent` run against the resolved target session without refusing or warning for thread session keys. Shared thread parsing exists and docs define the `:thread:<threadId>` key shape, so this is a valid, narrow follow-up rather than a stale or already-implemented item.

## Best Possible Solution

Keep this issue open and add a focused `sessions_send` guard in `createSessionsSendTool` after canonical target resolution and visibility/access checks, before any history read or nested `agent` call. Use the shared thread-session parser on the resolved canonical key, return a clear `status: "error"` telling callers to use the parent channel session key, and add regression tests for raw `sessionKey`, label-resolved targets, and sessionId-resolved targets while preserving existing thread-bound announce delivery behavior.
