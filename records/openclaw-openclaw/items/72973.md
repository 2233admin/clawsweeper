## Summary

Keep this PR open. Current main and the latest release still lack outbound filename control/bidi stripping at the shared media filename boundary, and the live PR remains an open, focused hardening change that needs normal maintainer review rather than cleanup closure.

## What This Changes

The PR adds a shared outbound filename helper, wires it into Content-Disposition and media basename extraction, and adds unit plus fetch integration coverage for stripping C0/C1 controls, zero-width characters, bidi override/isolate controls, and BOM characters.

## Best Possible Solution

Keep this PR open for maintainer review, decide the intended Unicode strip set, coordinate with the shared filename decoder work in #71517 if needed, and then land or request targeted changes on this branch.

## Reproduction Assessment

Yes. The current-main reproduction is a focused `fetchRemoteMedia` mock with `Content-Disposition: filename*=UTF-8''report%E2%80%AEgpj%E2%80%8B.exe`, which currently flows through the raw `path.basename(decodeURIComponent(...))` path; the PR adds that integration case plus pure helper cases.

## Solution Assessment

Mostly yes. Centralizing the display-label stripping at the shared media extraction boundary is narrower and more maintainable than duplicating the logic in Telegram, Discord, Matrix, and WhatsApp adapters, but the exact LRM/RLM/ALM policy remains a maintainer decision.
