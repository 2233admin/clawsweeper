## Summary

Keep this issue open. Current main still has configurable transient retry only for one-shot cron jobs through global `cron.retry`; recurring failures compute the natural next schedule and then apply a generic backoff floor, so daily/weekly jobs still wait until the next natural slot when that is later than the retry delay. There is no per-job retry policy field in the cron job type, Gateway schema, or CLI add/edit surface, and related PR #69252 was closed unmerged.

## Best Possible Solution

Implement an additive recurring cron execution retry policy: job-level retry config in cron types and Gateway schemas, CLI add/edit flags, durable retry attempt state, transient-error classification, scheduling at `endedAt + configured delay` before the next natural slot while attempts remain, and docs/tests that clearly distinguish one-shot retry, recurring backoff floor, and delayed recurring re-execution.

## Clownfish Work Prompt

Implement configurable delayed execution retry for recurring cron jobs before their next natural schedule. Keep the fix additive and focused.

Observable bug: current `applyJobResult` only uses configurable transient retry for one-shot `schedule.kind === "at"` jobs. Recurring `cron`/`every` failures schedule the later of natural next run and generic backoff, so daily/weekly jobs do not retry at a configured +15m/+30m window.

Expected boundary:
- Add a job-level retry policy to cron job types and Gateway cron schemas, preserving existing jobs and rejecting invalid shapes through existing validators.
- Add CLI support for `openclaw cron add/edit` retry controls such as max attempts, delay/backoff list, retry-on types, and disabling retry. Pick names consistent with existing CLI style; do not remove existing global `cron.retry` one-shot behavior.
- Persist enough retry attempt state so retries survive Gateway restart and reset after success/final failure/natural schedule as appropriate.
- For recurring jobs, when an execution error is transient and attempts remain, schedule `nextRunAtMs` from the failed run end plus the configured delay even when the natural daily/weekly slot is later. After retry budget exhaustion or permanent errors, fall back to existing recurring schedule/backoff/failure-alert behavior.
- Preserve existing one-shot retry semantics and existing safeguards against tight refire loops, schedule compute failures, and restart catch-up behavior.
- Update docs for the public cron job/API/CLI surface and mention changelog only if this becomes user-facing release work.

Related refs: this issue; #24355 for existing one-shot retry; #11641 for existing scheduler hardening/backoff; closed unmerged PR #69252 for prior attempted recurring retry design.
