## Summary

Keep open. The requested manual/scheduled cron parity is valid and current main has not implemented the new flag, but this PR is not mergeable as-is because the new option is not carried through the typed cron service boundary and the runner changes the Gateway process cwd during an awaited cron execution.

## What This Changes

The PR adds an `--as-scheduled` option to `openclaw cron run` and attempts to thread an `asScheduled` boolean through the CLI, gateway cron RPC, cron service execution, and isolated-agent runner.

## Best Possible Solution

Revise the PR to add a typed cron-run options contract from CLI and gateway through `CronService` into the shared execution layer, then implement scheduled-preview semantics through explicit workspace, environment, and context parameters that the agent runner already consumes. Add focused CLI/gateway/service/isolated-run tests, update cron docs, and add the required changelog entry before maintainer review.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: the PR gateway call passes a third argument to `context.cron.enqueueRun`, while current `CronServiceContract` and `CronService.enqueueRun` still expose and forward only `id` and `mode`; the cwd race can also be reproduced by any concurrent Gateway path reading `process.cwd()` during the awaited cron run.

## Solution Assessment

No. The flag direction is plausible, but this implementation is not the best solution because it leaves a typed service boundary unplumbed and uses process-global cwd mutation instead of the existing run-scoped `workspaceDir` execution boundary.

## Security Review

Status: needs_attention

Summary: The diff does not add dependencies or workflow/package changes, but the process-wide cwd mutation introduces a concrete path-confusion risk in the Gateway runtime.

Concerns:

- **[medium] Global cwd mutation can confuse runtime paths:** `src/cron/isolated-agent/run.ts:1050`
  - body: Changing `process.cwd()` during an awaited cron run can make concurrent Gateway, plugin, approval, or tool code that falls back to relative paths resolve inside the cron agent workspace rather than its intended base. That is security-relevant for multi-agent and tool execution boundaries.
