## Summary

Keep open. Current main still has the same warning-only task-registry restore path for malformed SQLite: the production store opens `runs.sqlite`, runs PRAGMAs/schema setup, and SELECTs rows directly, while restore catches failures only to log them. I found no corruption-specific snapshot, rebuild/recover path, or task audit finding, and the related reports in the provided context do not supersede this task-registry database recovery bug.

## What This Changes

Request automated recovery for malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` during task registry restore.

## Best Possible Solution

Task registry should own a corruption-safe restore path: detect malformed SQLite failures during database open/schema/load, preserve the bad database and sidecars, recover what can be recovered or rebuild to a valid empty registry with explicit operator/audit visibility, and document the behavior for task operators.

## Repair Work Prompt

Fix the task registry SQLite corruption restore bug reported in https://github.com/openclaw/openclaw/issues/71689. Current main logs restore failures from malformed `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` but does not recover. Implement this within the task-registry-owned SQLite store/restore path, not as a core startup migration. Detect SQLite malformed/corrupt database failures during open, WAL/PRAGMA/schema setup, migration, and snapshot SELECT. On confirmed corruption, close any handle, snapshot `runs.sqlite` and existing `runs.sqlite-wal` / `runs.sqlite-shm` sidecars to a timestamped operator-visible backup, then attempt a conservative recovery/rebuild path; if full recovery is not possible, create a fresh valid empty registry only after preserving the originals. Surface the event through task audit/operator-facing diagnostics so `openclaw tasks audit` can explain that restore recovery happened or failed. Do not silently delete task state, do not treat permission errors as corruption, and do not introduce a broad startup/load-time core migration. Add focused regression tests that create or simulate a malformed SQLite image and prove gateway/task-registry restore no longer loops on the same malformed DB while preserving the bad files. Update docs/changelog only if behavior becomes user-facing; changelog entry needs reporter credit to @waynegault.
