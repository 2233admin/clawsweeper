## Summary

Keep open. Current main still treats a malformed task registry SQLite image as an ordinary restore exception: startup attempts to load `tasks/runs.sqlite`, catches failures, and logs a warning without classifying corruption, preserving the DB and WAL sidecars, rebuilding a clean registry, or surfacing an audit/operator finding. The related #73517/#74406 work targets maintenance hot-loop lookup cost, not malformed-image restore recovery.

## What This Changes

Handle corrupted `tasks/runs.sqlite` during task registry restore by detecting malformed SQLite, preserving or recovering the bad store, reopening a usable registry, and surfacing operator diagnostics.

## Best Possible Solution

Add task-registry-owned corruption recovery at the SQLite store/restore boundary: detect malformed/not-a-database failures during open, schema setup, statement preparation, and snapshot loading; close any partial handle; preserve the DB plus WAL sidecars; recover when safely possible or reopen an empty registry after preservation; and expose a `restore_failed` or store-corruption diagnostic through task audit/status.

## Reproduction Assessment

Yes. The issue gives a concrete corrupted `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup path with `ERR_SQLITE_ERROR` and `PRAGMA integrity_check` failures, and current source shows that path reaches SQLite restore and only logs the exception. I did not run a live corruption repro because this sweep was read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the task registry SQLite store plus restore/audit surfaces; a generic gateway catch-all or the open maintenance-cache PR would not address malformed image recovery.
