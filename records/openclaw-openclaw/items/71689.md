## Summary

Keep open. Current main still routes task-registry restore through the SQLite store and catches restore failures as a warning only; there is no malformed-image classification, sidecar quarantine, recovery/rebuild path, or task-registry audit signal. The related #73517/#74406 work is adjacent maintenance hot-loop/cache work and does not cover corrupt `runs.sqlite` restore.

## What This Changes

Recover or safely quarantine a malformed `tasks/runs.sqlite` during durable task-registry restore instead of repeatedly warning and leaving the registry unrestored.

## Best Possible Solution

Implement task-registry-owned corruption handling in the SQLite store and restore/audit path: detect corrupt/not-a-database failures during open, schema setup, statement preparation, or load; close any partial handle; quarantine the database plus WAL sidecars; attempt conservative recovery only when safe or reopen a clean empty registry after preservation; and expose the condition through task audit/operator diagnostics.

## Reproduction Assessment

Yes. The issue gives a concrete corrupted `$OPENCLAW_STATE_DIR/tasks/runs.sqlite` startup path with `ERR_SQLITE_ERROR` and `PRAGMA integrity_check` failures, and current source shows that path reaches the SQLite loader and then only logs the exception. I did not run a live corruption repro because this review was read-only.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the task registry SQLite store plus registry restore/audit surfaces; a generic gateway startup catch-all or the #74406 maintenance-cache change would not solve malformed-image restore failure.
