## Summary

Keep open. Current main mitigates the worst reported symptom by making memory search/bootstrap embedding failures non-fatal, but the credential-routing request is still unresolved: auto memory embeddings can still select the OpenAI adapter from OpenAI credentials, and the bundled DeepSeek plugin does not expose a memory embedding provider.

## What This Changes

The issue asks memory-core to avoid selecting unrelated OpenAI embedding credentials for a DeepSeek-backed agent and to prevent embedding failures from blocking message handling.

## Best Possible Solution

Keep the non-fatal memory-search behavior and have a maintainer decide the credential policy: document explicit `agents.defaults.memorySearch.provider`/`remote` configuration, add doctor guidance for mismatched ambient OpenAI credentials, or design a supported DeepSeek/OpenAI-compatible embedding adapter path without coupling memory-core to the active chat provider by default.

## Reproduction Assessment

Yes for the static credential-selection path: with `memorySearch.provider` unset and OpenAI credentials present, current source can auto-select the OpenAI embedding adapter independently of an active DeepSeek chat model. A live 401 requires the reporter's specific invalid OpenAI-compatible key and endpoint, so that part is not reproduced here.

## Solution Assessment

Unclear. The current v2026.4.27-era mitigation is the right narrow fix for preventing agent-message blockage, but it does not solve the requested credential routing, and silently reusing DeepSeek chat credentials for embeddings is not proven to be the best supported design.
