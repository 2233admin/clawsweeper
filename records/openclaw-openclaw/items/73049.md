## Summary

Keep open. Current main and v2026.4.27 mitigate the user-visible blockage by making memory_search embedding failures return an unavailable result instead of throwing through the agent tool path, but the reported credential-routing behavior remains: memory embeddings are selected/configured independently from the active DeepSeek chat provider, and the bundled DeepSeek plugin still does not register a memory embedding adapter.

## What This Changes

The issue requests that memory-core stop selecting OpenAI embedding credentials for a DeepSeek-backed agent and avoid blocking agent message processing when embeddings fail.

## Best Possible Solution

Keep the non-fatal memory_search behavior and have a maintainer choose the credential policy. The maintainable end state is either documented explicit `agents.defaults.memorySearch.provider`/`remote` configuration with doctor guidance for mismatched ambient OpenAI credentials, or a provider-owned embedding adapter path for a supported DeepSeek/OpenAI-compatible embedding endpoint without coupling memory-core to the active chat provider by default.

## Reproduction Assessment

Yes for the static credential-selection path: current source shows `provider: "auto"` can select the OpenAI embedding adapter from OpenAI credentials independently of the active DeepSeek model. Live 401 reproduction is only partial because it requires the reporter's exact invalid key/base-url environment.

## Solution Assessment

Unclear. The v2026.4.27 mitigation is the right narrow fix for preventing agent-message blockage, but routing memory embeddings through DeepSeek chat credentials is not proven to be the best fix because DeepSeek is not currently a registered memory embedding provider and the upstream API reference does not show an embeddings endpoint.
