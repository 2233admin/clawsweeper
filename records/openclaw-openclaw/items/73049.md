## Summary

Keep open. Current main and v2026.4.27 mitigate the worst agent-blocking failure by returning a structured unavailable memory_search result and swallowing search-bootstrap sync errors, but the reported credential-routing behavior remains: memory embeddings still select an embedding adapter independently of the active DeepSeek chat provider, and the bundled DeepSeek plugin does not expose a memory embedding adapter.

## What This Changes

The issue asks memory-core to stop breaking a DeepSeek-backed agent by selecting OpenAI embedding credentials from ambient OpenAI configuration after the v2026.4.25 upgrade.

## Best Possible Solution

Keep the shipped non-fatal memory-search behavior, then make an explicit maintainer decision for embedding credentials: document that memory embeddings require explicit memorySearch provider/remote configuration, add doctor guidance for mismatched ambient OpenAI credentials, or add a supported DeepSeek/OpenAI-compatible memory embedding adapter without coupling memory-core to the active chat model by accident.

## Reproduction Assessment

Yes for the source-level path: with memorySearch.provider left as auto, the memory embedding selector can choose the OpenAI adapter from OpenAI credentials while the active chat model is DeepSeek. Exact 401 reproduction is reporter-environment-specific because it requires the same invalid key/base URL setup.

## Solution Assessment

Unclear. The v2026.4.27 mitigation is the right narrow fix for preventing agent-message blockage, but reusing DeepSeek chat credentials for embeddings is not proven to be the best long-term fix because DeepSeek currently registers no memory embedding provider and the docs present memory embeddings as separately configured.
