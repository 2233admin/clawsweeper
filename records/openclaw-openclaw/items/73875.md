## Summary

Keep open. Current `main` still exposes Claude ACP as a supported ACPX target, but the managed ACP path still turns generic OpenClaw runtime options into `session/set_config_option` calls for `thinking`, `approval_policy`, and `timeout`. The pinned ACPX runtime does not advertise strict config keys, while the pinned Claude ACP adapter only accepts advertised options such as `mode`, `model`, and `effort`, so the reported failure mode remains plausible and source-backed.

## Best Possible Solution

Keep this issue open and fix the ACP runtime-control path generically: derive supported config option IDs from the target ACP session/adapter when available, map OpenClaw `thinking` to adapter-specific keys such as Claude `effort` when advertised, keep run timeout as an OpenClaw/ACPX execution timeout unless the adapter explicitly advertises a timeout config option, skip unsupported generic keys without failing the turn, and add regression coverage with a Claude-like non-Codex adapter that advertises only `mode`, `model`, and `effort`.
