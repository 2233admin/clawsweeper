## Summary

Keep open. Current main still forwards generic ACP runtime config options to non-Codex ACP adapters when no strict advertised key list is available, and an open PR already references this issue with closing syntax, so the issue should wait for that PR or a successor to merge.

## What This Changes

The issue asks OpenClaw-managed ACP sessions to stop sending unsupported generic runtime config options such as `thinking`, `approval_policy`, and `timeout` to Claude ACP unless the adapter advertises or can safely map them.

## Best Possible Solution

Keep the issue open until #74048 or a successor merges. The desired end state is a generic ACP runtime-control path that filters config options against advertised adapter support, maps OpenClaw thinking to adapter-specific keys such as Claude `effort` only when available, treats timeout as an OpenClaw execution timeout unless explicitly advertised by the adapter, and adds a Claude-like regression test.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run a managed ACP session with persisted `thinking`, `permissionProfile`, or `timeoutSeconds` against an ACPX non-Codex adapter that lacks `configOptionKeys`; current main builds those generic keys and forwards them before the turn.

## Solution Assessment

Yes, the requested filtering/mapping direction is the maintainable fix boundary. The current open PR targets the same issue, so the correct maintainer path is to review and land that PR or a successor rather than close this issue early.
