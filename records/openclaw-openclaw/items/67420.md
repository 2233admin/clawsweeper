## Summary

Keep open: current main still lacks per-agent dreaming control, and this unmerged PR is a relevant implementation path for the linked open feature request. The patch still has a concrete fallback bug where an explicit all-agents opt-out can re-enable the default agent workspace.

## What This Changes

This PR adds `agents.list[].dreaming.enabled` to the agent config type, strict Zod schema, generated base schema, memory docs, and memory dreaming workspace resolution, with a unit test for mixed enabled/disabled agents.

## Best Possible Solution

Update this PR or land a successor that keeps the config type, strict schema, generated schema, docs, and resolver aligned; only uses the default-agent fallback when no agents are configured at all; treats explicitly disabled configured agents as a real empty result; and adds tests for mixed agents, all-disabled agents, and config validation. Per-agent schedule support from #67413 should be accepted into the same PR only if maintainers want that scope now.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Skip fallback after explicit dreaming opt-out:** `src/memory-host-sdk/dreaming.ts:603-605`
  - body: When every configured agent has `dreaming.enabled: false`, this guard filters them all out and leaves `agentIds` empty. The existing fallback then pushes `resolveDefaultAgentId(cfg)`, so the managed sweep can still run for the default workspace despite the explicit opt-out, preserving the OOM failure mode for single-disabled or all-disabled configs. Track whether any valid agents were configured before filtering, and only use the fallback when no agents were configured.
  - confidence: 0.92
