## Summary

Keep this PR open. Current main already classifies the broad `An error occurred while processing...` server-error text as a timeout/failover signal, but it does not implement the PR's central user-visible behavior: rewriting the plain-text provider internal error into friendly copy while preserving the support request ID. The PR is not fully obsolete; it likely needs a narrow rebase against the current helper split and the bot review feedback.

## Best Possible Solution

Keep the PR open and ask for a rebase that preserves the already-current timeout classification, adds a narrow plain-text provider internal-error formatter/request-ID extractor in the current helper structure, and covers both `formatAssistantErrorText` and `formatRawAssistantErrorForUi` with focused tests. No core/product decision or ClawHub move is needed; this is still a plausible core error-normalization bug fix.
