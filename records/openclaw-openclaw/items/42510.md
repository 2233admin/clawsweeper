## Summary

Keep open. Current main still does not fully honor Google Chat `replyToMode: "off"`: the setting is accepted and the shared final-reply pipeline can strip `replyToId`, but the Google Chat monitor still creates the typing placeholder inside the inbound thread before that filter runs, and routed outbound text/media still forwards `threadId ?? replyToId` without consulting the resolved Google Chat reply mode. The older fix PRs are closed unmerged, and the related open PR #67055 remains an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open until a Google Chat plugin fix lands on main. The fix should gate implicit Google Chat thread propagation on the resolved `replyToMode`, covering inbound context/tool context, typing placeholders, final text/media delivery, and routed outbound `sendText`/`sendMedia`, while preserving deliberate explicit thread targets according to the channel contract. Add focused regression coverage for off-mode typing, text, media, and routed outbound paths; #67055 or an equivalent maintainer patch is the likely review path.
