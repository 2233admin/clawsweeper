## Summary

Keep this issue open. Current main still accepts Google Chat `replyToMode`, but the Google Chat paths that create or forward thread transport are not consistently gated by `replyToMode: "off"`. The monitor still sends the typing placeholder into the inbound thread, and routed outbound text/media still forwards `threadId ?? replyToId` to the Google Chat API sender. The older fix PRs are closed unmerged, while #67055 remains an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open until a Google Chat plugin fix lands on main. The fix should honor `replyToMode: "off"` across inbound context, typing placeholders, final text/media delivery, and routed outbound `sendText`/`sendMedia`, while preserving deliberate explicit thread targets. Review #67055 or land an equivalent maintainer patch with focused regression coverage for typing, text, media, and routed outbound paths.
