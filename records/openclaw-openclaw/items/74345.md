## Summary

Keep open. This is a recent, concrete gateway reliability report, and current main still contains the same ACP maintenance close paths, warning strings, all-session ACP metadata scan, and session-store lock behavior that match the reported fingerprint. The later subagent orphan-recovery guard is adjacent but does not prove the idle ACP close loop, long lock holds, or event-loop saturation are fixed.

## What This Changes

The issue reports a v2026.4.27 gateway regression where task-registry ACP cleanup repeatedly fails to close sessions, holds session-store locks and the event loop for minutes, and stalls model runs plus Telegram and Discord transports.

## Best Possible Solution

Keep this as a gateway reliability bug until a maintainer reproduces or bisects current main, then make ACP maintenance cleanup bounded and non-blocking: avoid repeatedly attempting the same failing runtime close without backoff/quarantine, preserve active persistent bindings, safely clear recoverable stale metadata, and add focused regression coverage for stale/orphaned ACP sessions plus lock-hold or liveness behavior.

## Reproduction Assessment

Yes for the reported v2026.4.27 regression: the issue gives a concrete bad commit, idle gateway startup steps, five matching log fingerprints, and a rollback control. Current-main reproduction is unclear because this read-only review did not run the live systemd gateway with ACP plugins and channel transports.

## Solution Assessment

Unclear because there is no proposed patch for this issue. The current subagent tombstone fix is adjacent rather than a complete ACP maintenance solution; the narrow maintainable path is a targeted ACP cleanup bound/backoff fix after current-main reproduction or bisect evidence.
