## Summary

Keep open. Current main still contains the exact ACP task-maintenance close paths and warning strings from the report, and I found no current-main live reproduction, test, changelog entry, or related merged fix proving the 480s idle event-loop saturation is resolved. The related reports remain open and partly overlapping, but #74860 explicitly describes a distinct 7s plugin-registry fingerprint, while #75297 is broader and #61895 tracks a different ACP process-orphan leak.

## What This Changes

The issue reports gateway event-loop saturation, ACP session-close maintenance failures, long session-write-lock holds, and transport stalls after upgrading to a 2026.4.27 commit.

## Best Possible Solution

Make ACP task-maintenance cleanup bounded and idempotent: avoid repeated failed close attempts without backoff or quarantine, keep bound persistent sessions safe, make ACP session listing and metadata cleanup avoid long synchronous store scans, and verify with a live idle gateway reproduction that includes stale/orphaned ACP metadata and lock diagnostics.

Yes for the originally reported bad build: the issue gives exact checkout/build/restart steps and a strong log fingerprint. Current-main reproduction is unclear because this read-only review did not run a live gateway, and the source still contains the plausible maintenance and session-store paths.
