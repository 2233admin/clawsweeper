## Summary

Keep open. This is a recent, concrete reliability report with no comments or linked fix PR, and current main still has the exact ACP maintenance, session-lock, event-loop, Telegram, and Discord surfaces described. Several post-9d8de70 commits mitigate adjacent Telegram/session-store/ACP cleanup paths, but I found no source, test, release, or current-main provenance proving the idle ACP close loop and 480s event-loop saturation are fixed.

## What This Changes

The issue reports that 2026.4.27 gateway task-registry maintenance saturates the event loop, repeatedly fails to close ACP sessions, holds session-store locks far beyond their budget, and stalls Telegram/Discord transports after commit 9d8de70.

## Best Possible Solution

Keep this open as a gateway reliability bug. The best next step is a maintainer-led current-main reproduction or bisect around the idle task-registry maintenance path, then a focused fix that makes ACP maintenance cleanup bounded and non-blocking, avoids repeatedly trying the same failing runtime close without backoff/quarantine, preserves active persistent bindings, and adds regression coverage for stale/orphaned ACP sessions plus a targeted liveness/lock-hold assertion.
