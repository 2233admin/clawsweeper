## Summary

Keep open. Current main still contains the reported ACP task-maintenance close path, matching warning strings, synchronous session-store scan, session-write-lock warning source, and no post-report source/test/changelog provenance showing the idle ACP cleanup loop and long event-loop stalls are fixed or bounded.

## What This Changes

The issue reports a 2026.4.27 gateway reliability regression where idle task-registry ACP cleanup repeatedly fails, session-write locks are held for minutes, embedded model calls time out, and Telegram/Discord transports flap.

## Best Possible Solution

Keep this tracked as a gateway reliability bug and fix it in the ACP/task-maintenance owner path: make stale/orphaned ACP cleanup bounded and non-blocking, avoid retrying the same failing close every sweep without backoff or quarantine, preserve actively bound persistent sessions, and add regression coverage for repeated close failure plus lock/event-loop health.

## Reproduction Assessment

Yes for the reported bad build: the issue provides concrete checkout/build/restart steps and a five-signal log fingerprint. For current main, reproduction is not yet high-confidence because this review did not run the live systemd ACP/Telegram/Discord idle scenario.

## Solution Assessment

Unclear because there is no proposed patch and the root cause is not fully isolated. The safest direction is a maintainer-led current-main reproduction or bisect, then a focused task-maintenance/ACP close-path fix rather than transport-specific workarounds.
