## Summary

Close as superseded by #74243. This PR captures the useful under-cap fallback idea, but the branch puts fallback inside `optimizeImageToJpeg`, which would mislabel original non-JPEG bytes and can swallow image input guard failures; #74243 is the open consolidation PR that keeps the same user-facing fix while preserving MIME type, cap behavior, guard failures, diagnostics, and regression coverage.

## What This Changes

The PR changes `optimizeImageToJpeg` so that when all JPEG resize/quality attempts fail it returns the original buffer with `resizeSide` and `quality` set to `0` instead of throwing.

## Best Possible Solution

Do not land this narrower branch. Review and land #74243 or an equivalent caller-level fallback that preserves original MIME, only passes through under-cap non-HEIC images, rethrows image guard errors, keeps optimizer diagnostics, and then closes the canonical bug report plus redundant implementation PRs.

## Reproduction Assessment

Yes. The issue family has live reports in #73424 and duplicates showing `Failed to optimize image`, and the patch risks are reproducible from current code: resize failures include pixel-guard errors, while caller code labels all JPEG optimizer returns as `image/jpeg`.

## Solution Assessment

No. Returning original bytes from `optimizeImageToJpeg` is not the best fix because that helper has lost the original MIME policy and cannot safely distinguish optimizer backend failures from guard failures; the safer solution is a caller-level fallback like #74243.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Preserve image guard failures:** `src/media/web-media.ts:657-662`
  - body: This fallback catches every failed `resizeToJpeg` attempt, including the pixel-limit and unsafe-dimension errors raised before decode. For a byte-small but pixel-oversized image, the branch would return the original buffer under the byte cap and bypass the guard that current tests expect to fail closed.
  - confidence: 0.91
- **[P2] Preserve the original MIME on fallback:** `src/media/web-media.ts:657-662`
  - body: The returned buffer can be the original PNG/WebP bytes, but `optimizeImageWithFallback` still wraps this result as JPEG and `optimizeAndClampImage` emits `image/jpeg`. That can send valid original bytes with the wrong content type and break downstream image parsing.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR can weaken the existing image resource guard by turning guard-triggered resize failures into under-cap original-image passthroughs.

Concerns:

- **[medium] Image guard errors can be bypassed:** `src/media/web-media.ts:657`
  - body: `resizeToJpeg` rejects oversized-pixel images before decode, but the proposed terminal fallback returns the original buffer after all resize attempts fail. If the byte size is under the caller cap, this bypasses the current fail-closed resource guard.
  - confidence: 0.9
