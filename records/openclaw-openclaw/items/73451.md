## Summary

Close as superseded: current main still has the optimizer failure path, but the remaining work is now tracked by the newer canonical PR #74243, which explicitly supersedes this PR and covers the MIME, byte-cap, guard, and diagnostic boundaries this patch misses. This patch is not safe to merge as-is because it returns original bytes from the JPEG helper after all resize failures, including resource-guard failures, and the caller would label non-JPEG fallbacks as JPEG.

## What This Changes

The PR changes `optimizeImageToJpeg()` to return the original buffer with `resizeSide` and `quality` set to `0` when all JPEG resize/quality attempts fail.

## Best Possible Solution

Close this branch as superseded and keep the remaining work centered on #74243 and #73424. The accepted fix should live where MIME, cap, and guard policy are all available, fall back only for sendable under-cap originals, preserve original content type, keep HEIC/pixel/unsafe-dimension guard failures fail-closed, and retain optimizer diagnostics.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path through `loadWebMedia()` into `optimizeImageToJpeg()` when every resize attempt fails, and existing tests still assert the terminal `Failed to optimize image` rejection and the pixel-limit rejection behavior.

## Solution Assessment

No. Returning original bytes from `optimizeImageToJpeg()` is too low in the stack because the caller has already committed to JPEG output; the safer path is the #74243-style caller-level fallback where MIME, byte cap, and guard policy are available.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Keep pixel guard failures fail-closed:** `src/media/web-media.ts:657-662`
  - body: `resizeToJpeg()` enforces the image pixel limit before resize work. Because this patch catches every resize error and then returns the original bytes, a byte-small but pixel-oversized image that currently rejects could pass through if it is under the byte cap. Distinguish backend optimization failures from guard failures and continue throwing the guard errors.
  - confidence: 0.9
- **[P2] Preserve fallback content type:** `src/media/web-media.ts:657-662`
  - body: The caller wraps every `optimizeImageToJpeg()` result as JPEG and returns `image/jpeg`. Returning original PNG/WebP bytes from this helper will mislabel fallback media, which can break downstream providers or channels. Move the fallback to the caller or return explicit format/content-type metadata.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff has no supply-chain changes, but it can bypass the existing image pixel-limit resource guard by treating all resize failures as safe fallback failures.

Concerns:

- **[medium] Image pixel guard can be bypassed:** `src/media/web-media.ts:657`
  - body: `resizeToJpeg()` rejects images over `MAX_IMAGE_INPUT_PIXELS` before resize work. The PR catches that rejection and returns the original buffer, so a small-byte, huge-pixel image could pass media preprocessing if the byte cap accepts it, weakening the existing DoS/resource guard.
  - confidence: 0.89
