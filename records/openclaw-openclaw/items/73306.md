## Summary

Keep open. Current main still passes the raw Active Memory `timeoutMs` into the embedded runner while the wrapper watchdog and hook use the setup-grace-aware budget, and the open closing PR #74480 is still the right path to review rather than closing this issue early.

## What This Changes

The issue reports that Active Memory times out on every run with zero summary output after upgrading to v2026.4.26, causing blocking latency and token/context bloat.

## Best Possible Solution

Land #74480 or an equivalent narrow fix that computes one effective Active Memory recall timeout and passes it consistently through the hook, watchdog, and embedded runner, then validate whether any remaining zero-summary behavior belongs in the broader Active Memory timeout trackers.

## Reproduction Assessment

Yes. The issue has several independent log-based reproductions, and current-main inspection gives a focused synthetic path: enable setup grace, trigger Active Memory, and observe that the embedded runner still receives the raw timeout that drives its abort timer.

## Solution Assessment

Yes for the timeout-budget mismatch. The linked PR direction is the narrowest maintainable fix; broader Active Memory latency or empty-summary behavior should remain separate if it still reproduces after this handoff is corrected.
