## Summary

Keep open. Current main still has both lazy CLI defects described by this PR: nested lazy reparsing starts from the immediate parent and drops parent options, and eager command-group registration still fires dynamic imports without awaiting them. The latest PR head is a focused implementation candidate and appears to have addressed the Greptile fallback-argv review comment, so this PR is not obsolete or implemented on main.

## Best Possible Solution

Keep this PR open as the active implementation candidate. Review the async CLI runtime contract as a plugin-facing API change, ensure any generated SDK API/docs drift is handled, then land this fix after the targeted CLI/browser regression tests and changed gate are green. Current main does not yet implement the intended behavior.

## Evidence

- **current_main_lazy_reparse_still_uses_immediate_parent:** On current main, `reparseProgramFromActionArgs` sets `root` to `actionCommand?.parent ?? program`, reads `rawArgs` from that command, builds fallback argv from only the leaf action command name, and calls `program.parseAsync(...)`. For a nested lazy action called with the immediate `browser` parent, this still loses root argv and parent options such as `--browser-profile`.
