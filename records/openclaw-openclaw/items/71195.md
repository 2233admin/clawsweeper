## Summary

Keep open. Current main and the latest release already include backend/browser realtime Talk and the PCM gateway relay, but native macOS Talk still uses the legacy Apple speech recognition to chat/history polling to TTS/talk.speak path. Maintainer discussion also explicitly narrowed the remaining work to native macOS integration rather than closing the request.

## What This Changes

Add native macOS Talk Mode support for OpenAI Realtime speech-to-speech, reusing existing realtime voice/session plumbing while preserving the current STT/chat/TTS fallback path.

## Best Possible Solution

Keep this as the canonical native macOS Talk Realtime follow-up. The target state should wire Swift mic capture, session lifecycle, assistant audio playback, transcript/clear handling, and cancellation/barge-in to the existing talk.realtime.session contract where possible, while retaining the current STT/chat/TTS path for non-realtime and local providers.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but the behavior gap is high-confidence from source inspection: backend/browser realtime Talk exists, while native macOS Talk still runs local speech recognition, chat.send/history polling, and TTS playback.

## Solution Assessment

Is this the best way to solve the issue? No for the original new-method API sketch as the primary path, because current main already has talk.realtime.session and a gateway relay. The maintainable solution is the narrowed maintainer direction: native macOS integration on top of the existing realtime Talk contract, with new gateway protocol only if the Swift client needs a missing generic seam.
