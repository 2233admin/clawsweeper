## Summary

Keep this issue open. Current main has the shared realtime Talk backend/browser surfaces, including `talk.realtime.session` and a Gateway PCM relay, but native macOS Talk still uses the legacy `SFSpeechRecognizer -> chat.send -> chat.history -> talk.speak` path and has no client wrapper/session implementation for realtime Talk. The issue discussion also has explicit maintainer direction to keep it open and narrow it to macOS client integration.

## Best Possible Solution

Keep this issue open as the canonical native macOS Talk Realtime follow-up. The best path is to reuse the existing `talk.realtime.session` contract and current relay/browser transport shapes, decide explicitly whether native macOS should consume direct WebRTC-SDP sessions or use Gateway relay PCM for OpenAI too, then add a macOS `GatewayConnection` wrapper, a native realtime audio session object, AVAudioEngine capture/playback or WebRTC plumbing, Talk overlay phase/cancellation wiring, and a clean fallback to the existing STT/chat/TTS path. Split any remaining ElevenLabs repeated-403 behavior into a small Mac Talk TTS issue/PR.
