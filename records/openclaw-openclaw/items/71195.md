## Summary

Keep open. Current main still has the native macOS product gap: Gateway/browser realtime Talk exists, including `talk.realtime.session` and relay/broadcast support, but native macOS Talk still uses the legacy Apple speech recognition to chat send/history polling to `talk.speak` playback flow. The issue also has explicit maintainer direction to keep it open and narrow the work to native macOS integration on top of the existing realtime Talk session path.

## What This Changes

Request native macOS Talk Mode support for low-latency OpenAI Realtime speech-to-speech, reusing the existing realtime voice provider path where appropriate while keeping the current STT/chat/TTS path as fallback.

## Best Possible Solution

Keep this as the canonical native macOS Talk Realtime follow-up. The target state should add Swift-side realtime session/audio capture/playback plumbing that consumes the existing `talk.realtime.session` contract where possible, updates overlay phase and cancellation behavior, preserves the current STT/chat/TTS path for fallback providers, and updates Talk/macOS docs once the app behavior changes.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Native macOS Talk still runs local speech recognition and `talk.speak` playback, while Gateway/browser realtime Talk support already exists on current main.

## Solution Assessment

No for the original new-method API sketch as the primary path; current main already has `talk.realtime.session` plus browser and relay transports. The narrower maintainable solution is to reuse that contract for macOS and add only the native client/audio pieces unless maintainers explicitly choose a provider-agnostic backend bridge.
