## Summary

Keep this PR open. Current main still validates cron webhook URLs only as trimmed http(s) URLs before persistence; private/internal hosts are still accepted into cron job configs. Runtime delivery is protected by `fetchWithSsrFGuard`, and current tests explicitly exercise blocking at dispatch time, but that is not the early creation/patch rejection this PR is trying to add.

## Best Possible Solution

Review and land this focused security hardening, or reimplement the same small change on main: call `isBlockedHostnameOrIp(parsed.hostname)` inside `src/cron/webhook-url.ts`, add a dedicated `src/cron/webhook-url.test.ts`, and keep the existing runtime `fetchWithSsrFGuard` protection as defense in depth.
