## Summary

Keep open. Current main still clears requested operator scopes for device-less shared token/password WebSocket operator connections outside a narrow local backend exception, and current regression coverage asserts the resulting missing-scope behavior. The issue also has open implementation candidates #51413 and #57889 with closing syntax, so it should not be closed before maintainers review, merge, or close those PRs.

## Best Possible Solution

Keep this issue open while maintainers review #51413 and #57889 against the gateway auth security model. The best path is either to land a scoped WebSocket fix with regression coverage proving authenticated non-local backend token/password clients retain operator scopes safely, or to explicitly document that remote WebSocket operator scopes require device identity and close the implementation candidates.

## Risks / Open Questions

- Changing this behavior is security-sensitive: preserving self-declared scopes for device-less remote WebSocket clients changes the current trust boundary for shared token/password, Tailscale, or trusted-proxy operator connections.
- The open implementation candidates do not appear identical in scope: #51413 targets token/password shared auth, while #57889 describes Tailscale/trusted-proxy transport mismatch behavior. Maintainers should reconcile the intended policy before landing either.
- Closing as implemented would conflate the HTTP shared-secret scope restoration path with the separate WebSocket connect path that still clears scopes.
