## Summary

Current main still appears to preserve the reported missing-scope behavior for device-less, non-local shared-token WebSocket operator clients. The connect path clears scopes after an allow decision unless a narrow local/backend preservation path applies, regression coverage asserts that valid shared-token operator connects can succeed while remaining scope-limited, and this issue is paired with open closing PRs #51413 and #57889. It should stay open for maintainer review rather than be closed by cleanup.

## Best Possible Solution

Keep this issue open while maintainers review #51413 and #57889 and decide the intended WebSocket trust boundary. If behavior should change, land one scoped gateway auth fix with regression coverage proving authenticated non-local backend token/password clients retain operator scopes safely without weakening device identity, Control UI, or trusted-proxy protections. If current behavior is intentional, document the remote-client device identity requirement clearly and close the implementation PRs.
