## Summary

Keep open. Current main still concatenates all matching assistant text blocks before the user-facing delivery path, and the related #69737 and #74928 reports track distinct raw-error and broader audience-boundary problems rather than superseding this pick-one policy issue.

## What This Changes

The issue asks OpenClaw to add a scoped delivery-layer policy for selecting one visible assistant text block when an assistant message contains multiple matching text content items.

## Best Possible Solution

Add an explicit delivery-scoped selector, or a shared extractor option whose default remains concatenation, so Slack/Telegram-style user delivery emits one chosen visible assistant text block while TUI, gateway previews, and chat-history keep full transcript behavior unless deliberately changed.

## Reproduction Assessment

Yes. A synthetic assistant message with two matching visible `text` blocks deterministically follows the current `parts.join(joinWith)` path into `handleMessageEnd`; live Slack is not required to confirm the source-level behavior.

## Solution Assessment

Unclear until maintainers pick the selector policy. The safest implementation direction appears to be a delivery-only selector or an opt-in extractor option with concatenation preserved as the default for non-delivery callers.
