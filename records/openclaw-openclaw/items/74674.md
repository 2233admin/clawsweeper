## Summary

Keep open. Current main still joins all matching assistant text content blocks before the embedded channel delivery path, while transcript/display-oriented callers still have joined-text expectations; this remains a real delivery policy bug, not cleanup that can close. The split from #69737 and the broader #74928 cross-reference are useful context, but neither supersedes this narrower selector decision.

## What This Changes

The issue requests a delivery-boundary policy that selects one visible assistant text item instead of concatenating multiple assistant text content blocks, while preserving transcript-oriented concatenation elsewhere.

## Best Possible Solution

Add an explicit delivery-boundary selector for public channel replies while leaving shared transcript extraction as concatenation by default. Cover the behavior with focused tests for the shared extractor, embedded visible-text helper, and `handleMessageEnd`, and keep TUI, gateway preview/session dump, and chat-history behavior unchanged unless a maintainer intentionally broadens the policy.

## Reproduction Assessment

Yes. A synthetic assistant message with two matching visible `text` blocks currently flows through `extractAssistantVisibleText` as a newline-joined string and then into `handleMessageEnd` delivery, so the reported concatenation can be reproduced without private Slack data.

## Solution Assessment

Unclear until the selector policy is chosen. A delivery-only selector is the narrowest maintainable direction; a shared extractor option is reasonable only if another caller needs the same pick-one behavior.
