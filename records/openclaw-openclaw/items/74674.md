## Summary

Keep open. Current main still joins all matching assistant text blocks before message-end delivery, and I found no delivery-scoped pick-one selector or regression coverage for the multi-text visible-reply case. The split from #69737 remains valid because that related issue is about raw error text, while #74928 is a broader audience-boundary proposal rather than a canonical replacement for this specific selector policy bug.

## What This Changes

The issue asks for user-facing delivery to select one visible assistant text item instead of concatenating multiple text content blocks, while preserving concatenation for transcript-style surfaces unless explicitly changed.

## Best Possible Solution

Add a delivery-scoped selector, or a shared extractor option whose default remains concatenation, so channel delivery posts one chosen visible assistant text block while TUI, gateway previews, and chat-history output keep full transcript behavior unless deliberately changed. Cover the chosen policy with focused extractor and `handleMessageEnd` regression tests.

## Reproduction Assessment

Yes. A synthetic assistant message with two matching visible `text` content blocks follows a high-confidence static path through `extractAssistantVisibleText` into `handleMessageEnd`, where current main joins the blocks before delivery callbacks can emit them.

## Solution Assessment

Unclear. A delivery-only selector appears to be the narrowest maintainable direction, but maintainers still need to choose the actual selection policy; changing the shared extractor default would not be the best fix because non-delivery callers currently rely on concatenation.
