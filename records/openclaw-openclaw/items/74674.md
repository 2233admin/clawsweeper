## Summary

Keep open. Current main still joins all matching assistant text blocks before the user-facing delivery path, and I found no delivery-scoped pick-one selector or regression coverage for the multi-text case; the split from #69737 remains valid because that related issue tracks raw error delivery instead.

## What This Changes

The issue requests a delivery-layer policy so Slack/Telegram-style user-facing replies choose one visible assistant text block instead of posting concatenated multiple text content items.

## Best Possible Solution

Add an explicit delivery-scoped selector, or a shared extractor option whose default remains concatenation, so user-facing channel delivery posts one chosen visible assistant text block while TUI rendering, gateway previews, and chat-history transcript views keep full-text behavior unless deliberately changed. Cover the behavior with focused extractor and `handleMessageEnd` regression tests.
