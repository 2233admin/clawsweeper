## Summary

Keep open. Current main still joins all eligible assistant text blocks before the embedded delivery path, while transcript-oriented surfaces still depend on joined text behavior, so this remains a valid delivery-policy bug rather than cleanup that can close.

## What This Changes

Request a delivery-boundary policy so assistant messages with multiple visible text blocks send one selected public reply instead of newline-concatenating all blocks.

## Best Possible Solution

Add an explicit delivery-boundary selector that preserves shared transcript extraction as concatenation by default, then cover same-phase multi-text assistant messages in embedded delivery tests and keep transcript/TUI/history behavior unchanged.

## Reproduction Assessment

Yes. A synthetic assistant message with two eligible `text` blocks is enough to trace the current source path from `extractAssistantVisibleText` into `handleMessageEnd` delivery without private Slack data.

## Solution Assessment

Unclear until maintainers choose the exact selector rule. The narrow maintainable direction appears to be a delivery-only selector rather than changing the shared extractor default.
