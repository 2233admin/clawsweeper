## Summary

Keep open. Current main still concatenates matching assistant text blocks in the shared and embedded visible-text extractors, and the channel delivery path still consumes that joined string without a delivery-specific pick-one policy. The related reports remain distinct rather than superseding this narrower delivery selector issue.

## What This Changes

The issue asks for user-facing channel delivery to select one assistant text content item instead of posting a concatenation of multiple visible text blocks.

## Best Possible Solution

Add an explicit delivery-boundary selector that keeps shared transcript extraction as concatenation by default, then cover multi-text assistant messages through the shared helper, embedded helper, and `handleMessageEnd` delivery behavior. If reuse is needed, expose the selector as an opt-in extractor option rather than changing the shared default.

## Reproduction Assessment

Yes. A synthetic assistant message with two matching visible `text` content blocks is enough to follow the current source path from extractor to `rawVisibleText`, where the blocks are newline-joined before delivery.

## Solution Assessment

Unclear until maintainers choose the selector policy. The narrowest maintainable direction appears to be a delivery-only selector; a shared extractor option is reasonable only if another caller needs the same pick-one behavior.
