## Summary

Keep this issue open. Current main still has a real future-config policy gap: ordinary config loading warns and continues, while the hard refusal is limited to service/destructive gateway actions. Later OpenAI-Codex hardening is relevant to the reported symptom area, but it does not resolve the downgrade/version-stamp recovery policy requested here.

## What This Changes

The issue requests a clearer or stricter recovery path when `openclaw.json` has a newer `meta.lastTouchedVersion` than the running binary because that mismatch reportedly caused OpenAI-Codex final replies to disappear in Discord.

## Best Possible Solution

Define one supported downgrade/recovery policy at the config and gateway boundary, document it, and add focused regression or manual proof for a future-stamped config plus Discord/OpenAI-Codex final-message delivery.
