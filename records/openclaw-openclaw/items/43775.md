## Summary

Keep this issue open. Current main still only warns during ordinary config loading when `meta.lastTouchedVersion` is newer, while the hard refusal is scoped to service/process mutations, service-mode gateway startup, and `gateway --force`. I did not find source, docs, or tests proving the ordinary Discord/OpenAI-Codex delivery path is now refused, repaired, downstamped, or regression-covered.

## Best Possible Solution

Keep this issue open until maintainers either extend future-config remediation to ordinary gateway runtime paths that can perform channel delivery, or add focused proof that current service/process guards fully eliminate the Discord/OpenAI-Codex silent final-message failure. The best product path is an actionable startup warning or refusal for unsafe runtime delivery, paired with split-brain install repair guidance and a regression/manual proof for the reported model/channel combination.
