## Summary

Keep open: the swapped-attribution report is valid-looking, but this patch should not merge as-is because it attributes whole session/date-range totals to a mutable session-level origin snapshot and also changes the row-level `channel`/`chatType` API consumed by the usage UI.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider`/`origin.chatType` over delivery `channel`/`chatType`, and adds a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery model: compute channel aggregates from reliable origin granularity, preserve or explicitly expose delivery metadata for row consumers, constrain aggregate keys, and cover swapped webchat/Telegram plus mixed-origin long-lived sessions in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:658`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this line buckets all of those totals by `storeEntry.origin.provider`. Because `mergeOrigin()` overwrites the stored provider on later inbound metadata, a mixed-origin bridged session can have all historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:658-659`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filter options, `channel:`/`chat:` queries, and CSV export, so this removes the delivery-side metadata even though the aggregate bucket is the part that needs origin attribution.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: No workflow, dependency, lockfile, lifecycle, or package-publishing surface is changed, but the data-plane aggregation key change needs attention.

Concerns:

- **[low] Constrain origin provider before aggregation:** `src/gateway/server-methods/usage.ts:658`
  - body: The patch makes `origin.provider` the preferred `byChannelMap` key, while `normalizeMessageChannel()` preserves unknown normalized strings. If origin context can be influenced by inbound metadata, this can spoof usage buckets or create high-cardinality response payloads.
  - confidence: 0.72
