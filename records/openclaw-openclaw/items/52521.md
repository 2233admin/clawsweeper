## Summary

Keep open: current main still has delivery-first channel attribution, and the contributor patch targets a valid-looking swapped-attribution bug, but the diff is not merge-ready because it relabels whole-session totals using a mutable session-level origin, changes row metadata consumed by the usage UI, and uses unconstrained origin-provider values as aggregate keys.

## What This Changes

This PR changes `sessions.usage` to prefer session origin provider/chatType over delivery channel/chatType for usage attribution and adds a regression test for swapped webchat/Telegram sessions.

## Best Possible Solution

Fix the attribution bug with an explicit origin-versus-delivery model: compute channel aggregates from a trustworthy origin dimension at the right granularity, preserve or intentionally expose delivery metadata for row consumers, validate aggregate keys, and cover swapped webchat/Telegram plus mixed-origin long-lived sessions in tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:658`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, but this change buckets all of those totals by `storeEntry.origin.provider`. Since inbound metadata can overwrite the stored origin across turns, a mixed-origin bridged session can have historical usage relabeled as whichever origin was seen last.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:658-659`
  - body: The origin-first `channel`/`chatType` values are also emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those row fields for badges, filters, query terms, and CSV export, so this removes delivery-side metadata even though only aggregate attribution needs origin semantics.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it expands origin metadata into aggregate keys without validation or cardinality bounds.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:658`
  - body: The patch makes `origin.provider` the preferred `byChannelMap` key, while `normalizeMessageChannel()` preserves unknown normalized strings. If inbound origin metadata can be influenced by a caller or plugin, this can spoof channel buckets or create high-cardinality usage responses.
  - confidence: 0.72
