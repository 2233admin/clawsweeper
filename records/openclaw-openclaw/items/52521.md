## Summary

Keep open for maintainer review. Current main still has the reported delivery-channel-first attribution behavior, so this PR is not obsolete, but the PR also changes per-session row semantics and has unresolved review/security questions that make it a normal maintainer review item rather than a conservative cleanup close.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider`/`origin.chatType` over delivery `storeEntry.channel`/`chatType` and adds a regression test for cross-channel webchat/Telegram attribution.

## Best Possible Solution

Preserve the real user-facing fix by defining the usage attribution contract explicitly: origin-based channel totals for the dashboard, stable row metadata for callers, and guarded origin-provider keys. The final patch should include focused regression coverage for cross-channel sessions plus whichever UI/API fields maintainers choose for origin versus delivery metadata.

## Risks / Open Questions

- Changing `sessions[].channel` and `sessions[].chatType` to origin values may remove delivery/group metadata from current UI badges, filters, and CSV exports unless a separate delivery field is added or the change is limited to aggregation.
- Because origin metadata is merged at the session level, the last inbound origin can retroactively bucket all historical usage for a mixed-origin session under one channel.
- `origin.provider` is normalized but not allowlisted before use as an aggregate key, leaving a low-severity spoofing/high-cardinality concern for any path that can influence origin metadata.
- The PR head currently has a failing Windows test shard, so validation needs maintainer inspection before merge.
