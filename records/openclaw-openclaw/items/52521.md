## Summary

Keep this PR open, but do not merge it as-is. Current main still has the delivery-channel attribution behavior the PR is trying to fix, yet the proposed two-line inversion also relabels session rows and buckets whole-session totals by a mutable latest-origin snapshot, so it can misattribute mixed-origin sessions and needs maintainer/API revision.

## What This Changes

The PR changes `sessions.usage` to prefer `storeEntry.origin.provider` and `origin.chatType` over delivery `channel` and `chatType`, and adds a regression test for swapped webchat/Telegram attribution.

## Best Possible Solution

Fix the attribution bug with explicit origin-versus-delivery semantics: keep or rename row metadata deliberately, compute channel aggregates from the correct granularity, constrain aggregate keys, and add tests for both simple swapped-channel sessions and mixed-origin long-lived sessions.

## Reproduction Assessment

Yes. The swapped-channel report is reproducible by store entries where `storeEntry.channel` differs from `storeEntry.origin.provider`; current main chooses delivery metadata, and the PR’s test would fail before the inversion. The mixed-origin regression risk is from code-path inspection rather than an existing executable test.

## Solution Assessment

No. The proposed priority inversion is small, but it is not the best fix because it attributes whole-session totals to a mutable latest-origin value and changes user-visible row fields that existing UI consumers treat as delivery metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Avoid attributing whole sessions to the latest origin:** `src/gateway/server-methods/usage.ts:621`
  - body: `loadSessionCostSummary()` returns totals for the whole session/date range, while `entry.origin.provider` is a mutable session-level snapshot updated by later inbound metadata. In a bridged or mixed-origin session, this moves all historical usage into whichever origin was seen last instead of attributing the usage that actually occurred under each origin.
  - confidence: 0.86
- **[P2] Preserve delivery metadata on session rows:** `src/gateway/server-methods/usage.ts:621-622`
  - body: The same origin-first `channel` and `chatType` values are emitted as `sessions[].channel` and `sessions[].chatType`. The usage UI uses those fields for badges, filter options, `channel:`/`chat:` queries, and CSV export, so this drops delivery-side metadata even though only the aggregate bucket needs origin attribution.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff has no CI, dependency, lockfile, lifecycle, or publishing changes, but it expands origin metadata into aggregate keys without validation or cardinality bounds.

Concerns:

- **[low] Constrain origin provider aggregation keys:** `src/gateway/server-methods/usage.ts:621`
  - body: The patch makes `origin.provider` the preferred `byChannelMap` key, while channel normalization preserves unknown strings. If inbound origin metadata can be influenced by a caller or plugin, usage buckets can be spoofed or made high-cardinality unless the aggregate key is validated or bounded.
  - confidence: 0.72
