## Summary

This PR should stay open. Current main still uses the delivery/session channel before `origin.provider` in `sessions.usage`, so the intended bug fix is not implemented on main. The PR also needs maintainer review or rework because the proposed two-line change affects both aggregate bucketing and per-session row semantics, and it relies on mutable session-level origin metadata for whole-session usage totals.

## Best Possible Solution

Keep the PR open for maintainer review or rework. The best fix is to address the linked swapped webchat/Telegram usage bug with a clear attribution model: either bucket aggregates by a validated origin channel while preserving explicit delivery metadata on rows, or move to per-entry/per-turn origin attribution so mixed-origin sessions are not collapsed under the latest session origin. Add regression coverage for cross-channel swapped sessions, mixed-origin sessions, UI-visible row fields, and invalid/high-cardinality origin providers.

## Risks / Open Questions

- Closing this PR as implemented would be incorrect because current main still has the delivery-first usage bucketing and no regression test for the origin-first scenario.
- Merging the PR as-is may retroactively attribute an entire mixed-origin session to whichever session-level origin was last recorded.
- Merging the PR as-is changes `sessions[].channel` and `sessions[].chatType` semantics used by Usage UI filters, badges, queries, and CSV export.
- Security pass found no final-diff supply-chain or CI surface, but origin-derived aggregation keys should be validated or cardinality-limited if they can be influenced by inbound metadata.
