## Summary

Keep open. Current main still lacks iMessage default/named watcher dedupe, and this PR remains an implementation candidate rather than obsolete. The branch also has unresolved review risks because the dedupe is wired through the global enabled predicate and the PR includes an unrelated outbound-target early return outside the stated iMessage fix.

## What This Changes

The PR adds iMessage watcher-key dedupe and tests for default/named account backends, and also changes outbound target resolution to return early when an implicit target has no channel config or active plugin.

## Best Possible Solution

Keep one canonical iMessage-owned startup dedupe that prevents duplicate monitors for the same effective cliPath/dbPath backend after enabled and configured checks, without changing global account enabled semantics or shared outbound target resolution. Cover duplicate default/named backends, disabled named accounts, placeholder unconfigured accounts, and distinct backends, then close #65141 after the selected PR lands.
