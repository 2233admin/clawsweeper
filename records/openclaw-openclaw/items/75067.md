## Summary

Keep this PR open. The resilience direction is useful and current main still lacks the proposed timeout reduction, disk cache, circuit breaker, and readiness reason, but the PR is not merge-ready because the disk-cache warm path can bypass current configured pricing and the refresh timers do not honor the cache TTL remainder or circuit-breaker cooldown.

## What This Changes

The PR lowers the Gateway pricing fetch timeout, adds disk-backed pricing-cache warming and persistence, adds an in-memory pricing fetch circuit breaker, exposes a startup-pending readiness reason, and updates related Gateway tests.

## Best Possible Solution

Keep the Gateway pricing resilience work, but revise it so disk warming overlays current configured and manifest pricing, retry scheduling uses the actual cache age or breaker cooldown, the required changelog entry is added, and the branch is integrated after or alongside #73486 without reintroducing pricing fetches on the startup-critical path.

## Reproduction Assessment

Yes. The review findings are reproducible from PR-head code inspection: a fresh disk-cache return occurs before configured pricing is collected, and both the warm-cache and open-circuit paths call the fixed 24h scheduler; focused fake-timer/unit tests around fresh cache, changed config cost, and breaker cooldown would exercise them.

## Solution Assessment

No. The proposed architecture is reasonable, but this implementation is not the best fix until it preserves current pricing precedence and schedules refreshes from the cache's remaining TTL or the circuit breaker's cooldown instead of the generic daily refresh helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Apply configured pricing before disk cache warm return:** `src/gateway/model-pricing-cache.ts:1182-1185`
  - body: This branch returns as soon as any fresh disk cache exists, before `collectSeededPricing` reads current `models.providers.*.models[].cost`. If a user changes local pricing while the cache is fresh, stale disk data wins for up to the cache window; merge or overlay current configured and manifest pricing before returning.
  - confidence: 0.91
- **[P2] Retry when the circuit cooldown expires:** `src/gateway/model-pricing-cache.ts:1232`
  - body: The open-circuit path logs the remaining cooldown but schedules via `scheduleRefresh`, which waits the 24h cache TTL. After three failures, a recovered pricing service can be ignored until the daily timer instead of being retried after the 10-minute breaker cooldown.
  - confidence: 0.9
- **[P3] Schedule disk refresh for remaining TTL:** `src/gateway/model-pricing-cache.ts:1184`
  - body: A fresh disk cache always schedules a full 24h delay from process start. A 23-hour-old cache can therefore remain in use for almost another day, and repeated restarts can keep postponing live refreshes; schedule from the loaded cache's remaining TTL instead.
  - confidence: 0.84
- **[P3] Add the required changelog entry:** `src/gateway/model-pricing-cache.ts:82`
  - body: This is a user-facing Gateway `feat`/resilience change, but the PR does not update `CHANGELOG.md`. Add a single-line entry under the active version with appropriate non-bot credit so release notes capture the timeout, cache, breaker, and readiness behavior.
