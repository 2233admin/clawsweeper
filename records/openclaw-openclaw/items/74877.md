## Summary

Keep this PR open. The linked bug is valid, but the current patch is still not mergeable because channel-computed `message_tool_only` modes can bypass the fallback, and the availability check does not match the runtime tool-filtering pipeline.

## What This Changes

The PR adds `messageToolAvailable` plumbing to source reply delivery-mode resolution, computes message-tool availability during auto-reply dispatch, exposes the option through the channel reply SDK helper, adds resolver tests, and includes a Persian locale correction.

## Best Possible Solution

Keep the fallback behavior, but derive message-tool availability from the same effective inventory path used by the runtime, or move the decision to a boundary that already has provider/model, group, sandbox, subagent, and runtime-disable context. The fix should also allow unavailable tools to override channel-computed `message_tool_only` modes while preserving `sendPolicy: deny` and intentional suppression.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Discord or Slack group/channel turn with default visible replies plus a policy stage that removes `message`; the channel passes `message_tool_only` as `sourceReplyDeliveryMode`, and this PR's resolver returns that requested mode before the fallback can apply.

## Solution Assessment

No. Falling back to automatic delivery is the right product direction, but this implementation is not the best fix because it misses real channel-provided modes and checks only part of the runtime tool policy stack.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Let unavailable tools override channel-selected modes:** `src/auto-reply/reply/source-reply-delivery-mode.ts:16-17`
  - body: `resolveSourceReplyDeliveryMode` still returns `params.requested` before the availability fallback. Discord and Slack compute `message_tool_only` first and pass it as `replyOptions.sourceReplyDeliveryMode`, so even when dispatch finds `message` unavailable, automatic delivery remains suppressed. Move the fallback after resolving requested/config-derived modes, or stop passing channel-computed modes as authoritative requests.
  - confidence: 0.94
- **[P2] Check availability against the runtime tool pipeline:** `src/auto-reply/reply/dispatch-from-config.ts:596-607`
  - body: The new guard checks only profile/provider-profile/global/agent policies and calls `resolveEffectiveToolPolicy` without provider/model context. The actual tool inventory also applies provider, agent-provider, group, sandbox, and subagent policies, so those configs can still remove `message` while this guard reports it available.
  - confidence: 0.9
