## Summary

Keep this PR open. The linked bug is valid and the branch moves in the right direction, but the proposed availability check is narrower than the real tool exposure pipeline, so it still leaves reachable configurations where automatic delivery is suppressed while the agent has no `message` tool.

## What This Changes

The PR adds `messageToolAvailable` plumbing to source reply delivery resolution, computes it at the dispatch call site from global and agent tool policy, passes it through the plugin SDK helper, and adds unit coverage for fallback behavior.

## Best Possible Solution

Keep the PR open and update the guard to use the same effective availability model as the agent tool inventory, with dispatch-level regression coverage for at least a profile-denied `message` case such as `tools.profile: "coding"`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a group/channel turn with default visible replies and `tools.profile: "coding"`: the PR's global/agent-only check reports `message` available, while the actual tool pipeline filters it out by profile.

## Solution Assessment

No. The fallback direction is appropriate, but the implementation should derive availability from the full effective tool pipeline or a shared helper rather than only global and agent policy.
