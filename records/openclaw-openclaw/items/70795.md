## Summary

Keep open. Current main still shows a real filesystem-policy mismatch: `tools.fs` is represented as a boolean workspace-only policy, direct `read` wiring is rooted in a single workspace/sandbox root, and broader approved read roots are not part of the config contract. This should not be auto-fixed because widening `read` access touches local state, credentials, and multi-agent privacy boundaries.

## What This Changes

The issue requests that the user-facing `read` tool honor the effective filesystem policy instead of remaining hard-bound to the workspace root when broader read access is intended.

## Best Possible Solution

Define one explicit filesystem access contract covering approved read roots, sensitive-path denial, read-vs-write semantics, sandbox mounts, direct-session tool assembly, and per-agent privacy boundaries; then route `read`, `write`, `edit`, `apply_patch`, media path loading, node tools, and sandbox bridge checks through that shared policy with regression coverage.

## Reproduction Assessment

Yes for the source-level mismatch: current main still has boolean-only `tools.fs.workspaceOnly` policy and single-root `read`/guard wiring. The exact macOS outer tool-host behavior should be live-validated before changing access boundaries.

## Solution Assessment

Unclear as an isolated code fix. Relaxing `read` alone is not the best solution until maintainers define approved roots and sensitive-path behavior; the safer path is a shared filesystem policy contract first.
