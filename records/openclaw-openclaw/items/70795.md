## Summary

Keep open. Current main still has a real filesystem-policy mismatch: the policy model is boolean-only, direct host `read` is assembled around the workspace root, and the reusable guard validates against one root rather than a shared effective policy. Related filesystem hardening and allowed-root PRs are still open, so this is not implemented or safely superseded.

## What This Changes

The issue requests that direct-session `read` honor the effective filesystem policy, including approved non-workspace readonly roots, instead of remaining hard-bound to the workspace root.

## Best Possible Solution

Define one explicit filesystem access contract covering approved roots, read-vs-write semantics, sandbox mounts, sensitive-path denial, per-agent privacy boundaries, and direct-session tool assembly; then route `read`, `write`, `edit`, `apply_patch`, media loading, and node/sandbox bridge guards through that shared policy with regression coverage.

## Reproduction Assessment

Yes for the source-level mismatch: current main still has a boolean-only filesystem policy and direct `read` wiring built around a workspace root. The exact reported macOS outer tool-host failure should be live-validated before changing filesystem access semantics.

## Solution Assessment

Unclear as an isolated fix. The maintainable solution is not simply to relax `read`; it is to make approved roots and sensitive-path handling explicit and shared across all filesystem-capable surfaces.
