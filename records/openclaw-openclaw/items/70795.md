## Summary

Keep open. Current main still models filesystem policy as a single `workspaceOnly` boolean and does not show a unified approved-root policy threaded through the direct-session `read` tool, sandbox bridge, and related filesystem surfaces. The existing coverage proves partial behavior only: unguarded host `read` can access outside the workspace when `workspaceOnly=false`, while workspace guards remain single-root when enabled. Related open work overlaps, but none is merged or canonical enough to close this security-sensitive boundary issue.

## Best Possible Solution

Keep this issue open for maintainer/security triage. The best path is to define the intended filesystem policy model first, then implement it in one shared contract: approved roots, read-vs-write semantics, sandbox mount behavior, sensitive-path handling, and direct-session tool assembly. Coordinate with #43565, #52793, #70573/#70682, and #42207 so the final change does not widen read access while another security fix is trying to tighten it.
