## Summary

Keep open. Current main still does not define an approved non-workspace read-root contract, and the direct `read` assembly plus shared guard code remain centered on a single workspace/sandbox root. Because widening `read` can expose local state and credentials, this needs maintainer filesystem-policy review before a repair PR changes behavior.

## What This Changes

The issue asks OpenClaw to make the direct-session `read` tool honor the effective filesystem policy instead of staying hard-bound to the workspace root.

## Best Possible Solution

Define one explicit filesystem access contract for agent tools: approved read roots, sensitive-path denial, read versus write/edit/apply_patch behavior, sandbox bind semantics, direct-session tool-host enforcement, and per-agent privacy boundaries. Then route `read`, media path loading, sandbox bridge resolution, nodes output guards, and write/edit/apply_patch checks through that shared policy with focused regression tests.

## Reproduction Assessment

Unclear for the exact live macOS direct-chat failure: the issue gives concrete steps and a prior source-level validation comment, but this read-only pass did not run a live direct session. Source inspection does confirm the remaining single-root policy surfaces and lack of approved-root config needed to reproduce the design mismatch.

## Solution Assessment

Unclear as a narrow `read` relaxation. The maintainable solution is not to patch one wrapper in isolation, but to first define the shared filesystem policy and sensitive-path behavior that all tool-host and sandbox layers must consume.
