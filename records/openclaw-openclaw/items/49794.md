## Summary

Keep this PR open. Current main still does not implement the intended whole-tool hard timeout for `memory_search`; the memory search path now lives in `extensions/memory-core`, and it still awaits manager acquisition and search directly. The PR remains relevant to the linked liveness bug, but needs to be ported to the current plugin-owned implementation.

## Best Possible Solution

Keep the PR open as a relevant implementation candidate, but rework it against current main: add a bounded timeout around `extensions/memory-core` `memory_search` execution covering manager acquisition plus search, add timeout-specific unavailable metadata in `extensions/memory-core/src/tools.shared.ts`, and add a fake-timer regression test in `extensions/memory-core/src/tools.test.ts`.

## Risks / Open Questions

- The PR edits the old `src/agents/tools` memory tool path, while current main's `memory_search` implementation is plugin-owned under `extensions/memory-core`; landing it as-is would miss the active code path.
- A fixed 30-second wall-clock timeout may conflict with documented longer embedding and QMD behavior, so the port needs maintainer review of timeout budget and user-facing metadata.
- A `Promise.race` fail-open timeout can let the turn continue while underlying manager/search work continues in the background; the port should account for pending-manager cache and side effects.
