## Summary

Keep open. Current main moved the memory_search implementation into the memory-core plugin, and the PR’s intended tool-boundary hard timeout is not implemented there. The work is still relevant but needs to be ported/rebased onto the current owner files.

## Best Possible Solution

Keep this PR open, but rework it against current main: add a bounded timeout around the current memory-core memory_search execution path, including manager acquisition and search; add timeout-specific unavailable metadata in extensions/memory-core/src/tools.shared.ts; and add a fake-timer regression test in extensions/memory-core/src/tools.test.ts. Then review the timeout budget and lingering-background-work behavior before landing it as the implementation for #49524.

## Risks / Open Questions

- The PR no longer applies directly because the memory tool moved from src/agents/tools into extensions/memory-core; a clean port is needed.
- A fixed 30-second wall-clock timeout may be too aggressive for some slow QMD or embedding setups, so maintainers should confirm the budget or make the boundary configurable before landing.
- Promise.race lets the caller recover, but the underlying manager init/search may continue running; the port should consider pending manager creation/cache side effects after timeout.
