## Summary

Keep open. Current main still exposes `llm_input` and `llm_output` as observation-only void hooks, and both embedded and harness call sites invoke them without awaiting or applying mutation results. Existing prompt hooks can add context or replace system-prompt text, but they do not provide a writable user prompt, history, or assistant-output contract; the related draft #74483 also says it does not implement this request.

## What This Changes

Request adding a writable Plugin SDK hook contract so plugins can modify user prompts, system prompts, history messages, and assistant text around LLM calls for use cases such as local PII sanitization and restoration.

## Best Possible Solution

Keep this open for maintainer Plugin SDK design: add an additive, permissioned, deterministic mutating pre-provider and post-output contract, or a deliberately versioned extension of the current hooks, with SDK types, docs, contract tests, and coverage for embedded runner and harness call sites.

## Reproduction Assessment

Not applicable as a feature request, but the current-main verification path is high confidence: inspect the SDK hook types, hook runner, embedded runner, harness helpers, docs, and related draft context. They consistently show observation-only LLM hooks with no writable result contract.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is valid, but the safest solution is a designed, permissioned SDK contract rather than silently making the current fire-and-forget observation hooks mutable.
