## Summary

Keep this issue open. Current main still exposes `llm_input` and `llm_output` as observation-only Plugin SDK hooks: their handlers return `Promise<void> | void`, the hook runner executes them through the parallel void path, and the embedded/CLI harness call sites do not await or consume a mutation result. Existing prompt hooks can add context or system-prompt text, but they still cannot rewrite the user prompt, history messages, or assistant output.

## Best Possible Solution

Keep this issue open for maintainer Plugin SDK design. The best path is an additive, versioned mutating pre-provider and post-output contract, or a deliberately designed result extension to the existing hooks, with explicit conversation/prompt-injection permissions, deterministic merge behavior, and coverage across the embedded runner, CLI/app-server harness helpers, docs, SDK baseline/export tests, and hook contract tests.
