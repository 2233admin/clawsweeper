## Summary

Keep open. Current main still treats `llm_input` and `llm_output` as observation-only Plugin SDK hooks, and the open related SDK follow-up PR explicitly says it does not implement writable prompt mutation for this issue.

## What This Changes

The issue requests writable Plugin SDK LLM hooks so plugins can rewrite the user prompt, system prompt, history messages, and assistant output around model calls.

## Best Possible Solution

Keep this open for maintainer Plugin SDK design: add an explicit, additive mutating pre-provider and post-output contract, or deliberately extend the existing hooks with versioned result types, permission checks, deterministic merge semantics, docs, SDK baselines, and coverage across embedded, Codex, and CLI harness paths.

## Reproduction Assessment

Not applicable as a runtime bug report. The capability gap is high-confidence to verify by reading current main: the public hook types are void-only and the runner/call sites do not consume mutation results.

## Solution Assessment

Unclear whether mutating the existing observation hooks is the best final shape. A dedicated mutating pre-provider/post-output hook pair may be safer than changing fire-and-forget observation hooks because it can define sequencing, permissions, and merge behavior without surprising existing plugins.
