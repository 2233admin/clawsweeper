## Summary

Keep open. Current main still treats `llm_input` and `llm_output` as observation-only Plugin SDK hooks, and no merged code consumes prompt, history, system-prompt, or assistant-output mutations. The related SDK follow-up PR explicitly does not implement this request, while the matching implementation PRs are still open and unmerged.

## What This Changes

The issue requests mutable Plugin SDK LLM hooks that can rewrite the user prompt, system prompt, history messages, and assistant output for privacy-preserving transformations.

## Best Possible Solution

Define an additive, versioned mutating pre-provider and post-output contract, or a carefully compatible result extension to the existing LLM hooks, with explicit `allowConversationAccess` and prompt-injection policy behavior, deterministic merge/block semantics, and coverage across embedded runner, Codex/CLI harness helpers, SDK docs, API baselines, and hook contract tests.

## Reproduction Assessment

Yes for the feature gap: source inspection on current main shows a plugin can observe `prompt`, `systemPrompt`, `historyMessages`, and `assistantTexts`, but a returned mutation has no supported type and no call site consumes it. I did not run runtime tests because this review is read-only, but the static contract and call-site path are high-confidence.

## Solution Assessment

Unclear. The requested capability is valid, but mutating the existing fire-and-forget hooks in place may not be the safest API shape; maintainers should review the open implementation PRs and settle an additive/versioned contract with explicit security policy semantics.
