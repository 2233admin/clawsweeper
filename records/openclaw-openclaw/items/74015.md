## Summary

Keep open. Current main still exposes `llm_input` and `llm_output` as observation-only Plugin SDK hooks, and the existing prompt hooks can inject context/system text but cannot rewrite the original user prompt, history messages, or assistant output. The related open SDK-seam PR is adjacent but explicitly says it does not implement writable prompt mutation.

## What This Changes

Request to extend existing Plugin SDK `llm_input` and `llm_output` hooks so plugins can return prompt, system-prompt, history, and assistant-output mutations for local sanitization and resensitization.

## Best Possible Solution

Design an additive, versioned Plugin SDK mutation contract for pre-provider input and post-output text, with explicit conversation-access and prompt-injection permissions, deterministic merge/priority rules, timeout/failure behavior, cross-harness parity, SDK API baseline updates, docs, and contract tests.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for the SDK gap: inspecting current types and call sites shows `llm_input`/`llm_output` handlers return `void` and their callers ignore return values.

## Solution Assessment

Is this the best way to solve the issue? Unclear without maintainer SDK design; mutable LLM input/output hooks are a plausible direction, but the safer end state is an additive contract with explicit permission gates rather than silently changing observation-hook semantics.
