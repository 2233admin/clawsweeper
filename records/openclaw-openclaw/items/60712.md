## Summary

Keep open. Current main still resolves a matched route binding whose agentId is not present in agents.list through the default-agent fallback, and Telegram inbound routing uses that resolver before producing the agent/session route. Config validation still accepts binding agent IDs as plain strings without checking that they refer to configured agents. The related items cover adjacent routing/fallback surfaces, but they do not supersede this stale binding-to-default-agent privacy behavior.

## Best Possible Solution

Fix the routing contract in src/routing/resolve-route.ts so an explicit matched binding with an unknown agentId fails closed or returns a diagnostic route state instead of silently falling back to the default agent. Add validation or doctor coverage for bindings[].agentId against agents.list, a core resolveAgentRoute regression test for stale binding IDs, and Telegram inbound coverage for the reported channel path while preserving intentional topic-agent session behavior separately.
