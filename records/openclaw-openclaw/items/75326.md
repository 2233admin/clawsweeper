## Summary

Keep open: this automerge-enabled PR is still an implementation review item, has the protected `maintainer` label, and the proposed helper introduces one blocking behavior regression by treating pre-timeout catalog failures as successful partial responses. The timeout direction is sound and covered by focused tests, but it needs a small semantic fix plus the required changelog entry before merge.

## What This Changes

The PR adds a 750 ms timeout for non-`all` `models.list` catalog discovery, documents the responsive fallback behavior, and adds focused handler tests while keeping `view=all` exact.

## Best Possible Solution

Keep the timeout-based responsiveness design, but race the raw catalog promise so pre-timeout rejections still use the existing `UNAVAILABLE` path, observe any late rejection only after the timeout fallback wins, add a focused rejection regression test, and add one Unreleased changelog bullet under `### Fixes`.

## Reproduction Assessment

Yes. Current main has a source-level reproduction because non-`all` `models.list` awaits `loadGatewayModelCatalog()` before view handling; the PR’s fake-timer tests reproduce the slow-catalog case, and the new regression can be reproduced by making `loadGatewayModelCatalog()` reject before 750 ms.

## Solution Assessment

No as proposed. The timeout fallback is the right narrow fix for slow discovery, but the implementation should not convert immediate catalog failures into empty successful results; preserving the old rejection path is safer and still keeps the slow path responsive.
