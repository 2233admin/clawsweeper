## Summary

Keep this PR open. The linked Feishu table-limit bug is valid and current main still lacks reply-dispatcher card fallback, but the latest PR diff is still too broad: it retries every card-send failure as plain text, can duplicate already delivered card chunks, and can expose block/preview payloads as ordinary Feishu messages. Related PR #73651 is open and outbound-only, so it does not supersede or implement this reply-dispatcher change.

## What This Changes

The PR wraps non-streaming Feishu reply card delivery in a catch block that logs card-send failures, retries through the existing plain-text chunk sender, and adds two reply-dispatcher tests for card failure fallback and successful card delivery.

## Best Possible Solution

Revise the reply-dispatcher fallback to degrade only known Feishu table-limit errors, preserve block/preview suppression, and avoid duplicating already delivered card chunks by falling back only before the first card chunk is sent or by resuming from the failed chunk. Share or align the table-limit classifier with #73651 if that lands first, and add focused regression tests for table-limit fallback, non-table rethrow, block payload handling, and multi-chunk partial delivery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Avoid retrying after delivered card chunks:** `extensions/feishu/src/reply-dispatcher.ts:419-447`
  - body: If sendChunkedTextReply() fails on chunk 2 or later, earlier card chunks have already been sent. The catch then retries the original full text as plain text, duplicating the beginning of the answer and first-chunk mentions. Track whether any card chunk was delivered and only fall back before the first send, or resume from the failed chunk.
  - confidence: 0.88
- **[P2] Limit fallback to final reply payloads:** `extensions/feishu/src/reply-dispatcher.ts:447-450`
  - body: When streaming startup has failed, info.kind === "block" can still reach this catch path. Sending those failed card chunks through sendMessageFeishu can expose block/preview content as normal chat text before the final answer, so the fallback should stay limited to final user-visible replies or explicitly drop block payloads here.
  - confidence: 0.84
- **[P2] Classify table-limit errors before fallback:** `extensions/feishu/src/reply-dispatcher.ts:440-447`
  - body: The catch handles every sendStructuredCardFeishu failure. That hides auth, permission, malformed-card, and rate-limit problems by retrying them as plain text; only the known 230099 / 11310 table-limit rejection should degrade to text, and unrelated failures should still propagate.
  - confidence: 0.8
