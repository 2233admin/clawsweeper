## Summary

Keep this PR open. Current main still sends non-streaming Feishu reply cards through `sendStructuredCardFeishu` with no plain-text fallback when card creation fails, and the linked bug #43690 remains open. The PR contains relevant, scoped work for the inbound reply-dispatcher path, but it is not cleanup-closeable or ready to treat as obsolete because the proposed catch is still broad and can retry the whole answer or expose block-preview payloads as normal Feishu text.

## Best Possible Solution

Keep this PR open and ask for a focused revision: share or align with the table-limit classifier from #70651, fall back only for Feishu table-limit failures such as 230099/11310, avoid resending already delivered chunks, preserve block payload suppression, and add regression coverage for table-limit fallback, non-table-limit rethrow, later-chunk failure, and block payload handling.

## Risks / Open Questions

- The PR fallback catches every structured-card send failure, so auth, permission, malformed-card, rate-limit, or other platform failures could be hidden by a plain-text retry instead of surfacing as delivery errors.
- If a later chunk fails after earlier card chunks were delivered, retrying the entire reply as plain text can duplicate already delivered content and mentions.
- If streaming startup has failed and `info.kind === "block"` reaches static card delivery, the fallback can expose block-preview or reasoning-like content as a normal Feishu message before the final answer.
- Security/supply-chain pass found no workflow, dependency, lockfile, package metadata, download, or secret-handling changes in this PR diff; the notable security-adjacent concern is the broad fallback behavior and possible preview-content exposure.
