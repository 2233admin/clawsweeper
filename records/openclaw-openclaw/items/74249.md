## Summary

Keep this issue open. Current main still has only a global agent-run timeout plus explicit request/cron overrides; there is no `channelTimeoutSeconds` config key, no schema/type/docs support for a per-channel timeout map, and the interactive Gateway/WebChat paths call the shared timeout resolver without a channel-aware resolution step. Cron `payload.timeoutSeconds` already exists, and Telegram has Bot API transport timeouts, but those do not solve the requested Telegram/Discord/Webchat whole-run timeout policy.

## Best Possible Solution

Keep this issue open. The best path is a small design decision followed by implementation of a public, backward-compatible channel-aware agent-run timeout resolver: preserve existing global `agents.defaults.timeoutSeconds` behavior when no channel override is configured, keep cron `payload.timeoutSeconds` as highest priority, add schema/type/docs support for a `channelTimeoutSeconds` map if maintainers approve that shape, and apply it in the interactive Gateway/WebChat/channel request paths using the normalized source/reply channel. Related #43187 and #65517 are useful context for Telegram unresponsiveness, but they do not supersede this config feature.
