## Summary

Keep this issue open. Current main still has global/default or explicit-request agent run timeout resolution, not channel-aware timeout config for Telegram, Discord, WebChat, or cron defaults; the current docs even state Discord queued agent turns do not have a channel-owned timeout. The related Telegram stall and active-memory reports are useful context for the pain, but they do not supersede or implement this requested config surface.

## What This Changes

Request channel-aware agent runtime timeout overrides in `agents.defaults` and per-agent config, while preserving cron job `payload.timeoutSeconds` as the highest-precedence override.

## Best Possible Solution

Add one shared channel-aware timeout resolver used by Gateway agent runs, WebChat/chat.send, and cron fallback paths; preserve existing global behavior when no override is configured, keep cron `payload.timeoutSeconds` first, and update config schemas, generated metadata/help, docs, and focused regression tests together.

## Reproduction Assessment

Yes for the config/runtime gap: current main can be verified by searching for `channelTimeoutSeconds`, inspecting strict config schemas, and following Gateway/WebChat/Cron timeout calls. The soft-throttle provider hang itself remains environment-dependent.

## Solution Assessment

Unclear until maintainers approve the public API shape and default values. The maintainable implementation direction is a shared resolver plus schema/docs coverage, not separate transport-level timeout tweaks in each channel.
