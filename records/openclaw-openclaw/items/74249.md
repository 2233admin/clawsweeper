## Summary

Keep this issue open. Current main still exposes a global agent-run timeout plus explicit request and cron job overrides, but it does not have a channel-aware timeout config key, schema/type surface, docs entry, or runtime resolver for Telegram, Discord, WebChat, or cron defaults.

## What This Changes

The issue requests configurable per-channel agent run timeout overrides for interactive channels and cron/default agent runs while preserving existing global timeout behavior.

## Best Possible Solution

Add one shared channel-aware agent timeout resolver and public config surface, preserving existing `agents.defaults.timeoutSeconds` behavior when unset, keeping cron `payload.timeoutSeconds` highest priority, and covering Gateway/WebChat/channel request paths with schema, generated metadata, docs, and focused regression tests.

## Reproduction Assessment

Yes for the config/runtime gap: current main can be verified by searching for `channelTimeoutSeconds`, inspecting the strict schemas and types, and following the Gateway/WebChat/Cron timeout calls. The live soft-throttle profile hang remains environment-dependent and is not required to prove the missing feature.

## Solution Assessment

Unclear until maintainers approve the public API shape and default values. The maintainable implementation direction is a shared resolver used by all run entry points, rather than transport-specific timeout tweaks.
