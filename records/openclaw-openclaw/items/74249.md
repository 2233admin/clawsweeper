## Summary

Keep this issue open. Current main still lacks a `channelTimeoutSeconds` config surface and a channel-aware agent-run timeout resolver; cron per-job `timeoutSeconds` and Telegram Bot API transport timeouts exist, but they do not solve the requested per-channel whole-run policy for Telegram, Discord, WebChat, and cron.

## What This Changes

Add backward-compatible per-channel agent-run timeout overrides for interactive channels while preserving existing global timeout behavior and cron per-job timeout precedence.

## Best Possible Solution

Keep this open for a small maintainer design decision, then implement a backward-compatible channel-aware agent-run timeout resolver: add schema/types/docs for an approved `channelTimeoutSeconds` shape, preserve current `agents.defaults.timeoutSeconds` behavior when unset, keep cron payload `timeoutSeconds` as highest priority, and apply the resolver in Gateway/WebChat/channel request paths using normalized channel context.
