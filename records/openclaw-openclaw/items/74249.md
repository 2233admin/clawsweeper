## Summary

Keep this issue open. Current main still has no per-channel agent-run timeout configuration; the shared resolver only reads the global agent timeout plus explicit request or cron overrides, and the Gateway/WebChat paths do not feed channel identity into timeout resolution. The related Telegram stall reports are useful motivation, but they address different transport/event-loop failures and do not supersede this config feature.

## What This Changes

The issue requests backward-compatible per-channel agent-run timeout overrides for Telegram, Discord, WebChat, and cron, with config schema validation, runtime resolution, defaults, and documentation.

## Best Possible Solution

Add a small, documented channel-aware agent-run timeout resolver in core config/runtime: keep explicit request and cron `payload.timeoutSeconds` overrides first, preserve existing global `agents.defaults.timeoutSeconds` behavior when no channel override exists, and apply the approved map shape from Gateway/WebChat/channel request paths using normalized source or reply channel.

## Reproduction Assessment

Yes for the repository-side gap: current main can be inspected and has no `channelTimeoutSeconds` schema/type/docs key, while the relevant runtime paths still call the global timeout resolver. The external soft-throttle hang scenario is plausible from the report but was not live-reproduced here.

## Solution Assessment

Unclear pending maintainer API approval. The requested channel-aware timeout direction is likely the narrow maintainable fix, but the public key shape, default values, and channel naming should be confirmed before an automated implementation PR.
