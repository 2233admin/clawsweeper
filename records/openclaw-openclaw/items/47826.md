## Summary

Keep open. Current main still does not expose the requested read-only REST query endpoints for sessions, agents, or combined status. The underlying `agents.list`, `sessions.list`, and `sessions.get` behavior exists on the Gateway WebSocket RPC surface, and the HTTP router/docs currently cover probes, OpenAI-compatible endpoints, `/tools/invoke`, `/sessions/{key}/kill`, `/sessions/{key}/history`, plugin routes, and Control UI. Related PR #47863 proposed this exact surface but was closed unmerged, and adjacent #20934 tracks broader session-management REST work rather than these read-only common queries.

## Best Possible Solution

Keep this open for a maintainer-owned Gateway HTTP API design decision. The best implementation would add a small authenticated read-only REST surface for session list, agent list, combined status, and session detail; reuse existing `listAgentsForGateway`, `listSessionsFromStore`, and `sessions.get` semantics; enforce `operator.read` consistently with Gateway method scopes; add route/auth/schema tests; and document the public endpoint namespace alongside the existing Gateway HTTP APIs. Coordinate with #20934 so read-only query endpoints and any future mutating session-management endpoints share one coherent contract.
