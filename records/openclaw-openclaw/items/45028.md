## Summary

Keep this open. Current main still routes normal Control UI chat sends through the internal WebChat path: the browser sends `chat.send` with `deliver: false`, the Gateway converts that to an internal `webchat` origin before considering stored Telegram delivery context, and the reply dispatcher suppresses external `routeReply` unless `ExplicitDeliverRoute` is set. The discussion comments from dsantoreis and Harimay23 point at the same generic route-preservation problem, while mainguyenanhvu's `dmScope` suggestion may help shared-context setups but does not fix a selected Telegram channel-scoped session whose stored delivery route is bypassed. Related #69326 was closed as a duplicate of this canonical issue, and open PR #46356 attempts a broad webchat mirroring solution but is not merged and mixes this bug with proxy/channel work.

## What This Changes

Control UI replies in Telegram-backed sessions should preserve the underlying Telegram delivery route, or expose an explicit choice between replying only to the operator surface and replying to the external channel session.

## Best Possible Solution

Define a generic operator-surface delivery policy for Control UI/WebChat: either add an explicit external-reply affordance for channel-backed sessions or safely infer it only for clearly selected channel-scoped sessions, while preserving the existing main-session and WebChat leakage guards. The implementation should stay in the Gateway/Control UI routing boundary, with Telegram covered through generic external-channel tests rather than a Telegram-specific workaround.
