## Summary

Keep this PR open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, and the PR’s adapter-side `after_tool_call` result-mutation proposal is not implemented on main or covered by the narrower open #61485 work. The PR changes model-visible prompt/output and tool-result trust boundaries, so the best maintainer path is API/security review or a split rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The likely path is to review or land a focused mutable `llm_input`/`llm_output` implementation such as #61485, then split adapter-side `after_tool_call` result mutation into a separate plugin API design with exactly-once dispatch, validated replacement result shape, documentation, and targeted regression tests.
