## Summary

Keep this PR open. Current `main` still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, and the embedded runner fires `llm_input`/`llm_output` without awaiting or consuming hook results. The overlapping open PR #61485 covers only the `llm_input`/`llm_output` subset, while this PR also proposes adapter-side `after_tool_call` tool-result mutation, which remains an unresolved plugin API and security design surface.

## What This Changes

The PR changes plugin hook contracts and embedded agent call sites so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return modifications for blocking, prompt/system prompt overrides, assistant text replacement, and tool-result redaction.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land or supersede it with one focused, hardened mutable `llm_input`/`llm_output` implementation, likely via #61485 or an equivalent patch, then split `after_tool_call` result mutation into a separate plugin API design that resolves dispatch ownership, result validation, prompt-policy constraints, docs, and regression tests.
