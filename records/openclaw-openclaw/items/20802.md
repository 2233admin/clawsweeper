## Summary

Keep this PR open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, while this PR changes public plugin contracts and adds security-sensitive prompt/system-prompt and tool-result mutation behavior. The discussion already identified a narrower path via #61485 or an equivalent hardened `llm_input`/`llm_output` patch, with adapter-side `after_tool_call` mutation needing separate maintainer API design.

## What This Changes

The PR updates plugin hook result types, hook runners, embedded runner consumers, adapter-side tool execution, and tests so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return modifications instead of only observing events.

## Best Possible Solution

Keep this PR available for maintainer review, use the focused hardened `llm_input`/`llm_output` path as the merge target if it satisfies the middleware use case, and handle pre-model tool-result mutation as a separate plugin API or trusted middleware design with docs and regression coverage.
