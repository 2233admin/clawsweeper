## Summary

Keep this PR open for maintainer review. Current main still leaves the requested mutable `llm_input`/`llm_output` capability unimplemented, but this branch is broader than the focused LLM-hook fix and still introduces blocking correctness/security issues around prompt-policy enforcement, prompt recomputation, output redaction, and duplicate `after_tool_call` dispatch.

## What This Changes

This PR changes plugin hook contracts so `llm_input`, `llm_output`, and adapter-side `after_tool_call` can return modifications, applies those results in embedded agent/tool paths, and updates hook tests.

## Best Possible Solution

Ship a focused, hardened mutable `llm_input`/`llm_output` contract first, with prompt-policy enforcement, prompt/image recomputation, payload cloning, output artifact stripping, docs, and contract tests; handle pre-model tool-result mutation later through a separately designed plugin API or trusted middleware surface.

## Reproduction Assessment

Yes. Static inspection of current main reproduces the missing mutable-hook capability, and static inspection of the PR diff plus existing exact-once tests gives high-confidence reproduction paths for the proposed patch bugs; tests were not run because this review was read-only.

## Solution Assessment

No. The direction is useful, but this branch is broader than the narrowest maintainable fix and currently mixes LLM prompt/output mutation with `after_tool_call` result mutation before prompt-policy and dispatch ownership questions are resolved.

Overall confidence: 0.9

Full review comments:

- **[P1] Apply prompt overrides before building the submission:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: `llmInputResult.prompt` is applied after current main has already derived the prompt submission and prompt-local images. A masking plugin can return a redacted prompt while stale submission/image state from the original prompt is still used or observed after rebase. Move the hook before prompt submission/image resolution or recompute both after applying the result.
  - confidence: 0.9
- **[P1] Enforce prompt-injection policy for llm_input:** `src/plugins/types.ts:1563-1566`
  - body: The PR adds `prompt` and `systemPrompt` mutations to `llm_input`, but `allowPromptInjection=false` only constrains the existing prompt-injection hook list. A plugin with conversation access but prompt injection disabled could still rewrite model instructions through this result type, so add policy coverage or strip those fields before applying the hook result.
  - confidence: 0.9
- **[P2] Keep after_tool_call from firing twice:** `src/agents/pi-tool-definition-adapter.ts:173-184`
  - body: The adapter now calls `runAfterToolCall`, while the subscription handler still invokes the same hook through the new void runner. In embedded runs both paths are active, and current regression coverage explicitly requires one `after_tool_call` per tool call to avoid duplicated plugin side effects.
  - confidence: 0.88
- **[P2] Remove original assistant content when overriding output:** `src/agents/pi-embedded-runner/run/attempt.ts:3177-3179`
  - body: When `llm_output` returns replacement `assistantTexts`, the original `lastAssistant` and message snapshots are still returned downstream. That can leave unredacted assistant content, reasoning, or error fields available after the hook requested replacement output.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces prompt and response mutation surfaces without fully carrying existing prompt-injection and redaction safeguards forward.

Concerns:

- **[high] Mutable llm_input bypasses prompt-injection policy:** `src/plugins/types.ts:1563`
  - body: `llm_input` gains `prompt` and `systemPrompt` result fields, but current policy only treats `agent_turn_prepare`, `before_prompt_build`, `before_agent_start`, and `heartbeat_prompt_contribution` as prompt-injection hooks. Without matching registry constraints, a plugin configured with `allowPromptInjection=false` can still rewrite model instructions through the new hook surface.
  - confidence: 0.9
- **[high] Prompt override can retain stale prompt artifacts:** `src/agents/pi-embedded-runner/run/attempt.ts:2840`
  - body: The PR applies prompt overrides after current main has already computed prompt submission and prompt-local images. For a redaction plugin, stale prompt-derived artifacts can survive unless the implementation moves the hook earlier or recomputes all derived payload state.
  - confidence: 0.9
- **[medium] Output override can leave original assistant text reachable:** `src/agents/pi-embedded-runner/run/attempt.ts:3177`
  - body: The proposed `llm_output` override changes `assistantTexts` but leaves the original `lastAssistant` object and message snapshots intact, so sensitive text that a plugin intended to redact can remain available to downstream handling.
  - confidence: 0.78
