## Summary

Keep open. Current main still matches the reported Slack routing split: ordinary top-level room/channel messages with `replyToMode="all"` compute an auto thread id and reply thread id, but room routing discards that id for the inbound session key, so the parent channel session remains in play. The issue is also paired with open PR #64080, which uses closing syntax and should be reviewed or replaced before this issue closes.

## What This Changes

The issue requests Slack channel/group top-level messages with `replyToMode="all"` use a thread-scoped inbound session key aligned with reply delivery to avoid duplicate parent-session processing and cross-sender context bleed.

## Best Possible Solution

Review and land #64080, or an equivalent maintainer patch, so the Slack room/channel routing layer derives one conversation thread id for both inbound session routing and reply delivery: parent `thread_ts` for real thread replies, message `ts` for ordinary top-level `replyToMode="all"` turns, and no thread id for modes that intentionally keep channel continuity.

## Reproduction Assessment

Yes. A unit-level reproduction is clear from `resolveSlackRoutingContext` with a channel message, `isRoomish: true`, no `thread_ts`, and `replyToMode="all"`: the session key stays on the base channel while `MessageThreadId` resolves to the message timestamp; the issue also has two live reproduction reports.

## Solution Assessment

Yes for the current proposed direction. Unifying the routed thread id with the reply-delivery thread id is the narrowest maintainable fix, as long as the patch keeps `replyToMode=off`, existing true thread replies, and seeded app-mention behavior intact.
