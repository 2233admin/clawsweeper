## Summary

Keep open. Current main still does not expose `OPENCLAW_SESSION_KEY`, and this PR is useful source work, but the proposed patch has correctness issues in env override handling and node-host scoping plus a security/product contract that needs maintainer acceptance.

## What This Changes

The PR injects `defaults.sessionKey` as `OPENCLAW_SESSION_KEY` into the exec environment before host dispatch and adds gateway-focused tests for default injection, absence, and caller override behavior.

## Best Possible Solution

Define one runtime marker contract, then either revise this PR into a narrow gateway/sandbox-scoped resolver that checks explicit `params.env` overrides directly and adds docs/security notes plus sandbox and node-scope coverage, or fold the useful intent into #40423's shared marker resolver.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: set an ambient `OPENCLAW_SESSION_KEY` on the gateway process or sandbox defaults and the proposed merged-env guard preserves that stale value instead of `defaults.sessionKey`; for `host=node`, analysis reads `request.env` while execution forwards only `requestedEnv`.

## Solution Assessment

No. The current patch is close but not the best final shape because it writes the marker before node dispatch and treats inherited env the same as caller overrides; a scoped helper that applies only to intended hosts and distinguishes caller overrides would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Override inherited session markers:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: The guard checks the already-merged environment, so an ambient `OPENCLAW_SESSION_KEY` from the gateway process or sandbox defaults prevents `defaults.sessionKey` from being written. That breaks the promised runtime guarantee by letting a stale session marker reach the child process unless the caller explicitly overrides it.
  - confidence: 0.9
- **[P2] Keep session markers out of node host env:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: This block runs before the `host === "node"` branch, but node execution forwards only `requestedEnv` to remote `system.run` while approval analysis reads the full `request.env`. A node command can be analyzed with `OPENCLAW_SESSION_KEY` present and run remotely without it, so the injection needs to be gated to gateway/sandbox or moved after the node branch.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No dependency, CI, or supply-chain change is present, but the patch intentionally exposes a session identifier through child-process environment variables and needs maintainer acceptance of that runtime contract.

Concerns:

- **[medium] Session key becomes subprocess environment data:** `src/agents/bash-tools.exec.ts:1615`
  - body: The new variable is inherited by affected child processes and descendants, where it can be echoed, logged, captured in diagnostics, or exposed to tools that inspect process environments. That may be the intended integration contract, but it should be explicitly documented and accepted before merge.
  - confidence: 0.88
