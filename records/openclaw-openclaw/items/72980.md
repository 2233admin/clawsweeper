## Summary

Keep this PR open. Current main still does not expose `defaults.sessionKey` as `OPENCLAW_SESSION_KEY` in exec child environments; it only injects `OPENCLAW_SHELL`. The PR is a plausible implementation candidate, but it needs maintainer review of the session-key-in-env contract and should address the existing host=node and sandbox coverage concerns before landing.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land a narrow exec-only implementation after adding an explicit non-node guard, sandbox coverage, docs for the new env marker, and a security note about env exposure, or fold the work into #40423 if maintainers want one shared agent/session marker contract across exec, ACP, and TUI child processes.

## Risks / Open Questions

- Security/product contract: exposing a session key through process env can leak through env dumps, shell tracing, subprocess logs, crash reports, or local process inspection. Maintainers should explicitly approve and document that contract before merge.
- The provided PR diff injects before the `host === "node"` branch. Current node execution forwards `requestedEnv` only, but an explicit non-node guard would reduce future accidental forwarding risk.
- The provided tests are gateway-only. Because sandbox env is composed through `buildSandboxEnv`, sandbox injection and caller-override precedence should be covered before landing.
- This overlaps with open PR #40423. Maintainers should decide whether to land the narrow exec-only contract here or fold the work into the broader runtime marker proposal.
