## Summary

Keep this PR open. Current main does not implement the requested session-key env marker, and the PR is useful, but it is not merge-ready because the injection can preserve a stale inherited session key, it runs before the documented-out-of-scope node-host branch, and the session-key-in-env contract needs maintainer security/product acceptance, especially given overlap with open PR #40423.

## What This Changes

The PR injects `defaults.sessionKey` as `OPENCLAW_SESSION_KEY` into exec child environments and adds gateway tests for default injection, absence, and caller override behavior.

## Best Possible Solution

Define the runtime marker contract once, then either land a narrow exec-only version with explicit gateway/sandbox scope, stale-ambient override handling, docs/security notes, and gateway plus sandbox coverage, or consolidate this work into #40423's shared env-marker approach.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Override inherited session markers:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: The condition checks the already-merged env after inheriting `process.env` or sandbox defaults. If the gateway process already has `OPENCLAW_SESSION_KEY` from another session, `defaults.sessionKey` is not written and the child command sees a stale key even though the PR promises the runtime guarantees the correct value. Preserve `params.env` override semantics by checking the caller override directly.
  - confidence: 0.9
- **[P2] Keep session markers out of node host env:** `src/agents/bash-tools.exec.ts:1615-1617`
  - body: The new block runs before the `host === "node"` branch. Node execution forwards only `requestedEnv` to remote `system.run`, while approval/allowlist analysis reads `params.request.env`, so node commands can be analyzed with `OPENCLAW_SESSION_KEY` present and then run remotely without it. Gate or move the injection to gateway/sandbox paths only.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: No dependency, CI, or supply-chain change was found, but the patch deliberately exposes a session identifier through child-process environment variables and needs maintainer acceptance of that contract.

Concerns:

- **[medium] Session key becomes subprocess environment data:** `src/agents/bash-tools.exec.ts:1615`
  - body: The new env variable makes the agent session key available to every affected child process and its descendants. That may be the desired integration contract, but env values are commonly echoed, inherited, logged, or captured in diagnostics, so this should be explicitly documented and accepted before merge.
  - confidence: 0.88
