## Summary

Keep open. The PR covers synthetic/external abort signals and the HTTP disconnect direction is aligned, but the central embedded run-budget timeout still is not connected to the signal that the fallback layer inspects. The branch also lacks the required changelog entry for a user-facing fallback behavior fix.

## What This Changes

The branch adds terminal abort detection to model fallback, threads optional abort signals through agent, auto-reply, and cron callers, tags HTTP disconnect aborts, and expands model-fallback unit coverage.

## Best Possible Solution

Expose an explicit terminal run-budget marker from the embedded runner to the fallback decision path, keep the external client-disconnect handling where the same signal is available, and add the Unreleased changelog entry before merge.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A focused embedded or cron run with a short `timeoutMs` and configured fallbacks exercises `scheduleAbortTimer`, which aborts the embedded runner's internal controller; the PR's synthetic unit tests do not cover that real internal-timeout path.

## Solution Assessment

Is this the best way to solve the issue? No. Inspecting `opts.abortSignal` is a good fit for HTTP disconnects and cron wrapper aborts, but the embedded run-budget timeout needs a direct embedded-runner/fallback contract rather than relying on the caller signal.
