## Summary

Current main still omits the requester's configured `identity.name` from the `sessions_send` agent-to-agent message, reply, and announce prompt contexts. The related implementation PR #38579 is closed unmerged, #60911 covered adjacent UI display behavior rather than receiver prompt attribution, and the remaining fix is narrow and actionable.

## What This Changes

Include the requester agent's configured identity name in `sessions_send` agent-to-agent prompt contexts so receiving agents can attribute inter-agent messages correctly.

## Best Possible Solution

Keep this issue open and implement the narrow `sessions_send` requester identity prompt fix: derive the requester agent id from the effective canonical requester session key, resolve `agents.list[].identity.name`, pass an optional requester name through message/reply/announce A2A prompt builders, and emit `Agent 1 (requester) name: <name>.` only when a configured name is safely available. Treat #38579 as prior context, not a branch to apply wholesale.
