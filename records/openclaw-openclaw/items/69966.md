## Summary

Keep open. Current main still exposes these bootstrap profile knobs only through `agents.defaults`, rejects the requested `agents.list[]` fields through the strict per-agent schema, resolves runtime context injection and bootstrap budgets from defaults only, and the Control UI preset flow still patches defaults. The related open PR #69988 is useful precedent for per-agent override wiring, but it is scoped to compaction and does not implement this request.

## What This Changes

Request per-agent overrides for `contextInjection`, `bootstrapMaxChars`, and `bootstrapTotalMaxChars` so heterogeneous agents do not all share the global `agents.defaults` bootstrap profile.

## Best Possible Solution

Add a focused core config/runtime/UI/docs feature for direct optional per-agent overrides under `agents.list[]`, with effective resolution by session agent id and fallback to `agents.defaults`. Keep named reusable profile registries out of the first pass unless maintainers explicitly choose that larger product layer.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: place the requested fields under an `agents.list[]` entry and current schema/type surfaces do not accept them, while runtime resolvers ignore `sessionAgentId` and read only `agents.defaults`.

## Solution Assessment

Yes. Direct per-agent overrides with inheritance from `agents.defaults` are the narrowest maintainable solution and match existing per-agent `contextLimits` precedent; reusable named profiles would be a broader follow-up product decision.
