## Summary

Current main at b85edb3f0cf6c2a8bec06537af9211a044266a6c still lacks a user-configurable lane wait diagnostic threshold. `enqueueCommandInLane` still defaults missing `warnAfterMs` to 2,000ms, scheduled cron agent work reaches the queue without an override, manual cron enqueue uses its own fixed 5,000ms value, and the strict diagnostics config/docs expose `stuckSessionWarnMs` but no lane-wait setting. Related implementation PRs #16366 and #14889 are closed unmerged, and #53023 is a broader concurrency/auto-yield follow-up rather than a replacement for this narrow request.

## Best Possible Solution

Keep this issue open and land a focused core patch adding a documented `diagnostics.laneWaitWarnMs` positive-ms setting. Preserve the current 2,000ms default when unset, preserve explicit per-call `warnAfterMs` overrides, apply the configured default to scheduled cron and any other queue enqueues without an override, decide and test whether manual cron should use the same configurable default, update gateway start/reload wiring so config changes take effect according to the existing reload model, and update generated config metadata, public docs, and changelog. Keep #53023 as the broader concurrency/auto-yield follow-up.

Expected behavior:
- Add a positive integer config setting, preferably `diagnostics.laneWaitWarnMs`, defaulting to the current 2,000ms behavior when unset.

- Update config types, zod schema, labels/help/generated schema metadata, public docs, and changelog. This is user-facing; credit the reporter, not Codex/OpenClaw.
- Do not implement #53023 session concurrency, auto-yield, or preemption in this PR.

Likely implementation areas: add a queue default setter/state in `src/process/command-queue.ts`; apply it from gateway lane/config startup and reload code; extend diagnostics config schema/metadata/docs; add regression tests for default behavior, explicit override precedence, scheduled cron lane behavior, and manual cron behavior.
