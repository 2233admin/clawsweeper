## Summary

Keep open. Current main still stores the lane wait warning threshold per queue entry and defaults omitted `warnAfterMs` values to 2,000 ms, while the diagnostics config/schema/docs expose stuck-session tuning but no lane wait warning threshold. The related implementation PR is closed unmerged, and the broader lane concurrency/auto-yield issue does not supersede this narrower logging-control request.

## What This Changes

Request to add a configurable lane wait diagnostic warning threshold so long-running cron or lane work can avoid repeated non-actionable `lane wait exceeded` log warnings.

## Best Possible Solution

Expose a documented positive millisecond `diagnostics.laneWaitWarnMs` field that defaults to the existing 2,000 ms behavior, supplies the default for future queue entries when `opts.warnAfterMs` is omitted, preserves explicit overrides, treats reload as no-restart/default-update behavior, and leaves per-lane/adaptive timeout or session concurrency work to broader follow-up discussions.

## Reproduction Assessment

Yes. A source-level reproduction is clear: enqueue work behind an active lane entry without passing `opts.warnAfterMs`, wait more than 2,000 ms, and current main emits `lane wait exceeded`; adding `diagnostics.laneWaitWarnMs` is not accepted or documented because the schema, types, labels, help, and generated docs lack that field.

## Solution Assessment

Yes. A narrow global diagnostics threshold is the maintainable fix for this issue; per-lane thresholds, auto-yield, or session concurrency would be separate product work and are not required to stop the reported log flood.
