## Summary

Keep open. Current main still defaults omitted lane wait warning thresholds to a hardcoded 2,000 ms value, while diagnostics config, generated metadata, reload planning, and public docs expose stuck-session warning tuning only. The related implementation PR #16366 is closed unmerged, and the broader open #53023 does not supersede this narrower log-noise control request.

## What This Changes

Request to make the lane wait diagnostic warning threshold configurable so long-running cron or agent work does not flood gateway logs.

## Best Possible Solution

Expose a documented positive millisecond diagnostics.laneWaitWarnMs config field that defaults to the current 2,000 ms behavior, feeds the default for future queue entries when opts.warnAfterMs is omitted, preserves explicit per-call overrides, treats config reload as no-restart/default-update behavior, and leaves broader lane concurrency or auto-yield work to #53023.
