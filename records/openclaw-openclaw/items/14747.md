## Summary

Keep open. Current main still emits lane wait diagnostics from a queued-entry threshold and falls back to a hardcoded 2,000 ms default when callers omit `warnAfterMs`; the diagnostics schema/types/docs do not expose a lane wait warning threshold. Related implementation PRs #16366 and #14889 are closed unmerged, and open #53023 tracks broader concurrency/auto-yield behavior rather than this narrower log-noise control.

## What This Changes

Request to add a configurable lane wait diagnostic warning threshold so long-running cron and external-tool jobs can avoid repeated non-actionable `lane wait exceeded` warnings.

## Best Possible Solution

Add a documented `diagnostics.laneWaitWarnMs` positive millisecond config field that defaults to the current 2,000 ms behavior when unset, applies to future queue entries without explicit overrides, updates schema/types/help/generated docs, preserves explicit per-call overrides, covers reload/no-restart behavior, and leaves broader concurrency or auto-yield work to #53023.
