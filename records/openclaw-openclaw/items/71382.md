## Summary

Keep this PR open. Current main still lacks a WhatsApp channel/account `readOnly` config and has ungated outbound WhatsApp paths, while the existing `sendPolicy` path suppresses automatic reply delivery rather than providing a transport-level no-send guarantee. The PR is useful, but the branch still has three concrete correctness gaps that should be fixed before merge.

## What This Changes

The PR adds WhatsApp channel/account `readOnly` configuration, resolves it through account state, threads it into the WhatsApp Web listener, and gates several outbound WhatsApp send paths with focused tests and generated config metadata updates.

## Best Possible Solution

Land a repaired WhatsApp-owned implementation that adds documented channel/account `readOnly` config, resolves it per account, enforces a single hard outbound gate at the WhatsApp transport/listener boundary, keeps inbound processing intact, preserves DM pairing request creation without sending pairing replies, and covers every blocked outbound path with focused tests.

## Reproduction Assessment

Yes. On the PR branch, the remaining issues are reproducible with focused unit checks: empty-body/no-media `sendMessageWhatsApp` under read-only should throw, `sendComposing` under read-only should match the other blocked helpers, and a default pairing DM under read-only should still upsert a pending pairing request while suppressing the WhatsApp reply.

## Solution Assessment

No, not as currently proposed. A transport-level read-only gate is the right direction and is narrower than relying on `sendPolicy`, but this PR should fix the guard ordering, composing contract, and pairing-request persistence before it is the best maintainable solution.
