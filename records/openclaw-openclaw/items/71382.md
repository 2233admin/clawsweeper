## Summary

Keep this PR open. Current main still lacks a true WhatsApp channel/account read-only mode, and the proposed transport-level direction is appropriate, but the branch is not merge-ready because three remaining paths make the hard read-only contract inconsistent or break pairing approval flow.

## What This Changes

The PR adds WhatsApp channel/account `readOnly` configuration, resolves it through account state, threads it into the WhatsApp Web listener, and gates outbound WhatsApp sends, presence, read receipts, and pairing replies with focused tests and generated config metadata.

## Best Possible Solution

Land a repaired WhatsApp-owned implementation that documents `readOnly`, keeps the config/schema/metadata surfaces aligned, enforces one consistent hard outbound gate for all WhatsApp transport sends, preserves inbound processing and DM pairing-request creation while suppressing WhatsApp replies, and adds focused regression coverage plus a user-facing changelog entry.

## Reproduction Assessment

Yes. The remaining PR defects are reproducible with focused checks against PR head: empty-body/no-media `sendMessageWhatsApp` under read-only should throw, inbound `sendComposing` under read-only should match other blocked helpers, and an unknown pairing DM under read-only should still persist a pending pairing request while sending no WhatsApp reply.

## Solution Assessment

No, not as currently proposed. A WhatsApp transport-level gate is the right maintainable direction and covers paths that `session.sendPolicy` does not, but the branch should fix guard ordering, composing error behavior, pairing persistence, docs, and regression tests before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate empty WhatsApp sends before returning:** `extensions/whatsapp/src/send.ts:93`
  - body: With `readOnly` enabled, `sendMessageWhatsApp` still returns `{ messageId: "", toJid }` for an empty body with no media before it reaches the new writable assertion. Move the read-only assertion before this early return so callers get the same blocked-send error for every outbound attempt.
  - confidence: 0.91
- **[P2] Throw for read-only composing attempts:** `extensions/whatsapp/src/inbound/monitor.ts:543-545`
  - body: `sendComposing` silently returns in read-only mode while reply, media, and the send API throw the new read-only error. Callers that use the error signal will treat composing indicators differently from other blocked outbound actions, so align this closure with the rest of the contract.
  - confidence: 0.86
- **[P2] Preserve pairing requests in read-only mode:** `extensions/whatsapp/src/inbound/access-control.ts:136-141`
  - body: When default `dmPolicy: "pairing"` meets `readOnly`, this branch skips the pairing challenge issuer entirely, so no pending pairing request is upserted for operator approval. Suppress only the outbound WhatsApp reply and still create the durable pairing request.
  - confidence: 0.9
