## Summary

Keep open. Current main still matches the reported policy gap: user-invocable skill commands with `command-dispatch: tool` build the raw OpenClaw tool set and apply only owner-only filtering, while normal agent and `/tools` paths use the session-effective tool policy pipeline. Because this is an authorization boundary, it needs maintainer/security follow-up rather than cleanup closure.

## What This Changes

The issue requests that `command-dispatch: tool` skill slash commands honor the same session-effective tool policy as normal agent turns.

## Best Possible Solution

Route this through the tool-policy/security owners and make skill tool dispatch resolve the current session's effective tool surface before executing the requested tool. Add focused regression coverage for a restrictive profile and a leaf subagent session so future skill-command changes cannot bypass profile, group/channel, sandbox, subagent, or owner-only policy.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: define a user-invocable skill with `command-dispatch: tool` and `command-tool: sessions_spawn`, invoke it from a restrictive or leaf-subagent session, and compare the inline handler's raw `createOpenClawTools(...)` lookup with the filtered `createOpenClawCodingTools(...)` surface.

## Solution Assessment

No. The current implementation is not the best way to solve the issue because it constructs a lower-level tool set in a user-invocable path; the safer direction is a shared session-effective tool resolver for deterministic command dispatch.
