## Summary

Keep open. Current main still matches the reported authorization-policy gap: inline skill commands using `command-dispatch: tool` resolve the raw OpenClaw tool set and then apply only owner-only filtering, while normal agent/runtime tool construction applies profile, provider, group/channel, sandbox, and subagent policy. The issue is also paired with an open closing PR, so the right outcome is to review and land or reject that PR rather than close this issue early.

## What This Changes

The issue asks for skill slash commands declared with `command-dispatch: tool` to execute only tools allowed by the current session's effective tool policy.

## Best Possible Solution

Route inline skill tool dispatch through the same session-effective tool resolver used by normal agent turns, preserving session key, target group/thread metadata, sender ownership, model/provider, sandbox, and subagent context. Add regression coverage for at least a restrictive tool profile and a leaf subagent session so this path cannot drift from the documented `/tools` surface again.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: define a user-invocable skill with `command-dispatch: tool` and `command-tool: sessions_spawn`, invoke it from a restrictive or leaf-subagent session, and compare the inline handler's raw `createOpenClawTools(...)` lookup with the filtered `createOpenClawCodingTools(...)` surface.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because it constructs a lower-level tool set in a user-invocable path. The safer solution is to reuse the session-effective policy pipeline; open PR #75125 appears to be the relevant candidate for that direction but still needs review.
