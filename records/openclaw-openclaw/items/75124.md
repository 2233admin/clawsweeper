## Summary

Keep open. Current main still matches the reported authorization gap: inline skill `command-dispatch: tool` slash commands build the raw OpenClaw tool set and apply only owner-only filtering, while normal agent turns and `/tools` use the layered effective tool-policy pipeline. The linked fix PR #75125 is open and unmerged, so this issue should stay open until that PR or an equivalent reviewed fix lands.

## What This Changes

The issue asks for user-invocable skill slash commands with `command-dispatch: tool` to honor the same effective tool policy as normal agent turns, including profile, group/channel, sandbox, and subagent restrictions.

## Best Possible Solution

Route the inline `command-dispatch: tool` branch through the same session-effective tool surface used by normal agent turns and `/tools`, preserving target-session group/subagent/sandbox/provider context and adding focused regression coverage for restrictive profiles and leaf subagent sessions.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: define a user-invocable skill with `command-dispatch: tool` and `command-tool: sessions_spawn`, invoke it from a restrictive or leaf-subagent session, and compare the inline branch's raw `createOpenClawTools(...)` lookup with the filtered `createOpenClawCodingTools(...)` surface.

## Solution Assessment

No for current main. The current implementation is not the best way to solve this because it constructs a lower-level tool set in a user-invocable path; the maintainable direction is to reuse or factor a shared session-effective tool resolver, which #75125 appears intended to do pending review.
