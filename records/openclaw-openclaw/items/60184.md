## Summary

Current main and v2026.4.29 already satisfy the PR's intended behavior: console capture forwards through normal tslog logger methods, and tslog 4.10.2 applies `minLevel` before attached transports run. The proposed guard is behaviorally aligned but redundant, so this PR does not need to remain open.

## What This Changes

The PR wraps console-capture file logger forwarding in `isFileLogLevelEnabled(level)` and adds two mock exports to a delivery-dispatch named-agent test.

## Best Possible Solution

Keep the shipped logger `minLevel` behavior as the source of truth for file-log filtering. If maintainers want extra assurance, a focused console-capture regression test would be useful, but the duplicate runtime guard is not needed.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: current `console.log` capture calls `logger.info`, and tslog drops info entries before attached transports when the logger `minLevel` is `warn`.

## Solution Assessment

No. The PR is functionally safe, but it is not the narrowest maintainable solution because current main already enforces the threshold through tslog; optional coverage would be a better follow-up than duplicate filtering logic.
