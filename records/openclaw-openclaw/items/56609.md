## Summary

Keep open. Current main still uses `.jsonl.deleted.<timestamp>` for session deletion and `.jsonl.reset.<timestamp>` for reset/new rollover, and current maintenance code can physically remove archived transcript artifacts under retention or disk-budget cleanup. The related prior cleanup reports addressed indexing/search visibility for `.reset`/`.deleted` archives; they do not implement this issue's naming and no-hard-delete policy.

## Best Possible Solution

Keep this issue open and handle it as a core session-retention policy change: introduce archive terminology or a dedicated archive directory with backward-compatible readers for existing `.jsonl.deleted.*` and `.jsonl.reset.*` files, then update reset/delete/new archive paths, cleanup/disk-budget semantics, lifecycle hook payload expectations, docs, and tests. The previous related fixes for memory/search indexing should remain as compatibility coverage, not be treated as a resolution of this naming and deletion-policy request.

## Risks / Open Questions

- Renaming the suffix or moving files to `sessions/archive/` has compatibility impact across readers, memory indexing, usage aggregation, lifecycle hooks, docs, tests, and downstream tools that currently recognize `.reset` and `.deleted`.
- The no-hard-delete part conflicts with existing retention and disk-budget cleanup behavior, so this needs an explicit product/data-retention decision rather than a mechanical rename.
