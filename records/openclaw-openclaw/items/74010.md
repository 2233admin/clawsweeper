## Summary

Keep open. Current main still has floor-only compaction gates while the config/Pi settings contract treats `reserveTokens` as the effective headroom above `reserveTokensFloor`; this PR appears to address that valid bug with a narrow helper and focused tests, but it is currently dirty/unmergeable and has failing checks, so it needs maintainer review rather than closure.

## What This Changes

This PR adds an effective Pi compaction reserve resolver, routes preflight compaction and memory-flush thresholds through it, renames the internal gate parameter, adds focused regression tests, and updates the changelog.

## Best Possible Solution

Land this PR or an equivalent narrow core fix after rebasing and clearing relevant checks, preserving the shared effective-reserve resolver and regression coverage so both preflight compaction and memory-flush gates use the same headroom semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a config with `reserveTokens` above `reserveTokensFloor`: current main computes memory-flush and preflight thresholds from the floor only, so a `100000` context with `reserveTokens: 30000` still behaves like the default `20000` floor in these gates.

## Solution Assessment

Yes, with rebase and CI follow-through. The PR is the narrow maintainable fix because it centralizes the effective reserve calculation already used by Pi settings and applies it at the two affected gate call sites with focused tests.
