## Summary

Keep this PR open. Current main still computes both preflight compaction and memory-flush gates from reserveTokensFloor rather than the effective reserveTokens budget, while the config/Pi settings path documents and applies reserveTokensFloor as a floor under reserveTokens. The PR is the live focused implementation candidate for the still-open #66830 and replaces closed, unmerged #69379/#67297 work.

## Best Possible Solution

Keep this PR open and review/land it, or land an equivalent narrow core fix. The implementation should resolve one effective compaction reserve from configured reserveTokens, reserveTokensFloor, and the small-context cap, then use that value consistently in preflight compaction and memory-flush gates with focused regression coverage and changelog credit for the prior contributor attempts.
