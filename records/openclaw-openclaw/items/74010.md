## Summary

Keep this PR open. Current `origin/main` at `e12eb9acdd3c62a7aae64bc1402105a829bc7a11` still computes both preflight compaction and memory-flush gates from `reserveTokensFloor` alone, while the documented and existing Pi settings semantics treat that value as a floor under effective `reserveTokens`. The PR is a focused implementation candidate for #66830 and replaces the earlier closed/unmerged #69379/#67297 work; the supplied diff is source/test/changelog only and does not introduce an obvious security-sensitive surface.

## Best Possible Solution

Keep this PR open and review/land it, or land an equivalent focused core fix. The implementation should compute one effective compaction reserve from configured `agents.defaults.compaction.reserveTokens`, `reserveTokensFloor`, and the small-context cap, then use that reserve consistently in preflight compaction and memory-flush gates with the included regression coverage and changelog credit. Validate with `pnpm check:changed` plus the targeted tests named in the PR body before merge.
