## Summary

Keep this issue open. Current main still treats foreground exec approval as an immediate `approval-pending` tool result with detached follow-up execution after approval resolution, not as a hard pause of the agent turn. The heartbeat comment is still relevant: heartbeat runs use `trigger: "heartbeat"`, while the current deterministic no-route fallback recognizes only `trigger === "cron"`; per-call stricter `security`/`ask` values can also tighten an otherwise permissive policy.

## Best Possible Solution

Keep this issue open for maintainer-scoped design and implementation. The best fix is a core agent/exec change that makes foreground exec approval a hard turn barrier until approval is allowed, denied, or timed out, while preserving explicit background behavior and making heartbeat/automation deterministic. The implementation should reuse the existing approval registration and `exec.approval.waitDecision` primitives where possible, update docs, and add regression coverage that prevents multiple unresolved foreground approval requests from the same turn.

## Risks / Open Questions

- Changing this behavior affects a security-sensitive host command approval boundary, so a maintainer should choose the intended turn-suspension model before implementation.
- The right implementation boundary is not obvious: blocking inside `exec`, suspending the agent loop, or adding a resumable approval state have different transcript, timeout, delivery, and provider-tool-call implications.
- Heartbeat, cron, interactive chat, and explicit background exec runs need separate rules so automation does not hang indefinitely or create approval noise.
- Existing tests currently preserve registration-only waiting and multiple pending approvals, so a fix will intentionally change current behavior and require focused regression updates.
