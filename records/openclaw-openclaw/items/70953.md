## Summary

Keep this open. Current main still treats foreground exec approvals as an immediate `approval-pending` tool result with a later asynchronous waiter/follow-up, so the model can continue the same turn after the approval request rather than being blocked on the operator decision.

## What This Changes

The issue asks for foreground exec approval requests to pause the agent turn until the approval is approved, denied, or times out, while preserving a separate policy for explicit background/headless execution.

## Best Possible Solution

Define one core exec-approval turn-control contract: foreground interactive exec approvals should either await the approve/deny/timeout result before the model can issue more tool calls, or terminate the current turn in a deterministic pending state that cannot trigger more commands until resolution. Keep explicit background/headless automation behavior separate, preserve approval audit binding, and update the exec docs plus regression tests for gateway, node, heartbeat, and denial paths.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: run a foreground gateway or node exec that requires approval from a user or heartbeat turn, and the exec path returns `details.status = "approval-pending"` while a background waiter handles the later decision.

## Solution Assessment

Unclear. Awaiting the final decision matches the requested behavior, but current shipped docs and code intentionally use an async pending/follow-up model, so the safest fix needs maintainer judgment on whether to block the tool call or hard-stop the turn after returning pending.
