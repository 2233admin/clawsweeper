## Summary

Keep this issue open. Current main still does not implement Telegram `message read`: the shared action name exists, but the Telegram plugin does not advertise, map, gate, dispatch, test, or document read/history support. Closed PR #14367 remains useful prior art for an inbound-store design, and closed duplicate #24602 confirms the same user-facing gap, but neither changed current main.

## What This Changes

Add Telegram support for the `message` tool's `read` action so agents can fetch recent Telegram chat or group messages with target and limit-style parameters similar to Discord and Slack.

## Best Possible Solution

Keep this issue open and implement the feature in the Telegram plugin after maintainer review of the no-`getUpdates` design. The likely path is a bounded inbound message store or equivalent runtime-owned cache, then wire `read` through Telegram action discovery, action mapping, runtime aliases/dispatch, config gates, CLI target/topic semantics, docs, and focused Telegram plus message-tool tests.

## Risks / Open Questions

- Telegram bots do not provide a normal per-chat history API, so a correct implementation likely needs a bounded inbound store or runtime-owned cache and must document that only messages observed while OpenClaw is running can be read.
- Reading and retaining Telegram message content, even in memory, has privacy and access-control implications; config gating and docs should be explicit before shipping.
- Prior PR #14367 is useful prior art but was closed unmerged after end-to-end adapter and parameter-alignment concerns, so it should not be treated as a ready fix.
