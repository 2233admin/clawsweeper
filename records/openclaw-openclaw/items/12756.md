## Summary

Keep this issue open. Current main still exposes `read` as a shared message action, but the bundled Telegram plugin does not advertise, map, gate, dispatch, test, or document Telegram message-history reads. The closed inbound-store PR and the closed duplicate are useful context, but neither changed current main.

## What This Changes

Request support for `openclaw message read --channel telegram` so agents can read recent Telegram chat or group messages for monitoring workflows.

## Best Possible Solution

Implement Telegram `message read` inside the Telegram plugin with a bounded runtime-owned inbound message store or equivalent cache, avoid secondary `getUpdates` polling, and wire it through action discovery, action mapping, runtime dispatch, config gating, CLI target/topic semantics, docs, and focused tests.
