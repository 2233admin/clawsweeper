## Summary

Keep this issue open. Current main still does not implement Telegram `message read`: the shared message action name includes `read`, but the Telegram plugin does not advertise, map, gate, dispatch, test, or document a Telegram read action. Closed PR #14367 remains useful prior art, especially around avoiding `getUpdates`, but it was not merged and current main still lacks the end-to-end feature.

## Best Possible Solution

Keep this issue open. The best path is a maintainer-reviewed Telegram plugin implementation: add a bounded inbound message store or another approved no-`getUpdates` mechanism, wire `read` through Telegram action discovery, the action map, runtime aliases/dispatch, config gating, CLI `--target`/`to` and topic/thread semantics, docs, and focused adapter/runtime/contract tests. Treat #14367 as useful prior art, not shipped proof.

## Risks / Open Questions
