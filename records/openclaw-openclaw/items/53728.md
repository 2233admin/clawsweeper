## Summary

Keep this PR open for maintainer review. Current main still lacks the requested startup memory-source report, but the proposed patch has concrete pre-merge correctness issues in the prompt-time Memoria probes and also mixes in unrelated Feishu outbound identifier logging that needs separate owner review.

## What This Changes

The PR adds a memory-source report helper, injects its output into embedded and command system-prompt workspace notes, and updates Feishu reply dispatch logging/tests to include outbound delivery results.

## Best Possible Solution

Land a focused memory visibility design that uses a documented status contract or plugin-owned status seam, bounds all startup probes, reports unknown counts honestly, matches prompt memory documentation, and handles Feishu delivery logging in a separate reviewed change if that logging remains desired.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Bound Memoria probes before prompt construction:** `src/agents/memory-source-report.ts:39-40`
  - body: `buildMemorySourceReport` is awaited while constructing embedded and command system prompts, but `fetchJson` uses bare `fetch` with no timeout. A slow or blackholed configured Memoria URL can stall startup until the platform network timeout instead of degrading quickly to unavailable.
  - confidence: 0.91
- **[P2] Do not report the limited fallback length as the memory count:** `src/agents/memory-source-report.ts:69-72`
  - body: The fallback request uses `/v1/memories?limit=1`, then returns `items.length` as the count. When the stats endpoint is absent, any backend with more than one memory will be reported as having one memory, which makes the injected startup prompt note misleading.
  - confidence: 0.94
