## Summary

Current main still matches the reported bug. The embedded attempt runner records one client tool call, run metadata wraps that one value into a one-element pendingToolCalls array, and both non-streaming and streaming /v1/responses handlers emit only pendingToolCalls[0]. The linked contributor PR #52287 targeted the correct code path, but it is closed and unmerged in the provided GitHub context, so this should stay open and is a good narrow fix-PR candidate.

## Best Possible Solution

Keep this open and land an equivalent of the closed PR #52287: accumulate ordered client tool calls in the embedded attempt result, update the result metadata type and run metadata to carry all calls, emit every pending function_call in both non-streaming and streaming /v1/responses responses, and add regression tests for multi-call order while preserving single-call behavior.
