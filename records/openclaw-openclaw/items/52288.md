## Summary

Keep open. Current `main` still matches the reported bug: the embedded attempt runner records a single client tool call, run metadata turns only that one call into `pendingToolCalls`, and both `/v1/responses` non-streaming and streaming handlers emit only `pendingToolCalls[0]`. The linked contributor PR #52287 directly targets the right code path but is closed and unmerged in the supplied GitHub context, so it is not a reason to close this issue.

## Best Possible Solution

Keep this open and land an equivalent of the closed PR #52287: accumulate ordered client tool calls in the embedded attempt result, update the attempt/result metadata types to carry an array, propagate every call into `pendingToolCalls`, emit all pending calls in both non-streaming and streaming `/v1/responses` responses, and add regression tests that assert multiple returned `function_call` items preserve call order.
