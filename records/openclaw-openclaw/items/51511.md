## Summary

Keep this issue open. Current main still advertises Cursor as an ACP harness route when ACP spawning is available, and I found no regression guard proving that an incidental Cursor URL in a custom Feishu session stays in the current agent instead of being routed to `sessions_spawn` with `runtime: "acp"` and `agentId: "cursor"`. Feishu current-placement binding mitigates the separate-window symptom, but it does not solve unintended ACP session takeover when the model selects that path.

## Best Possible Solution

Keep this open and add a targeted ACP intent-routing guard plus regression coverage. A Feishu/custom-agent message containing a Cursor GitHub link or incidental Cursor text should remain in the current custom-agent session and must not call `sessions_spawn(runtime: "acp", agentId: "cursor")` unless the user explicitly asks to run, use, open, continue, bind, or spawn work in Cursor/ACP. The likely fix belongs in the prompt and ACP router skill wording, with a small deterministic test proving the negative case while preserving explicit Cursor ACP requests.

## Risks / Open Questions

- The remaining behavior is partly model-routing dependent, so a fix should include a deterministic prompt/skill regression test rather than relying only on wording review.
- ACP sessions run external harnesses on the host; an over-broad natural-language trigger can be a security-adjacent UX problem even when the immediate report is framed as session hijacking.
- Narrowing Cursor ACP intent must preserve explicit `/acp`, acpx, and "run this in Cursor" workflows for users who intentionally want ACP harness routing.
