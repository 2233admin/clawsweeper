Keep this open. The report is concrete and has a second user confirming similar latency, while current main only shows adjacent gateway/model-catalog and runtime-dependency mitigations rather than a high-confidence proof that this Feishu/Lark + MiMo regression is fixed.

## What This Changes

The issue reports a regression where Feishu/Lark replies on OpenClaw 2026.4.27 take about 30 seconds instead of about 8 seconds on 2026.4.23, with suspected gateway/provider-discovery or plugin-runtime overhead.

## Best Possible Solution

Keep the issue open until a maintainer can run a current-main or v2026.4.29 Feishu/Lark timing check with the reported provider/plugin mix, capturing per-turn timings around Feishu inbound handling, model catalog/provider discovery, plugin runtime-deps staging, and final reply delivery. If the v2026.4.29 gateway fixes resolve it, close with that release evidence; otherwise add a narrow timing-backed fix in the owner path that actually stalls.
