## Summary

Keep open. Current main has adjacent fixes for model-catalog stale refresh and provider/runtime-deps startup work, but there is no source, test, release, or reporter evidence proving the Feishu 8s-to-30s regression is fixed. The report also has distinct Feishu/channel evidence: a second commenter says delays started with 2026.4.24, and current Feishu reply paths still have a 30s default API timeout plus awaited typing/streaming startup paths that plausibly match the symptom.

## What This Changes

The issue reports a Feishu/Lark response-latency regression after upgrading from 2026.4.23 to 2026.4.27, with downgrading restoring prior ~8s replies.

## Best Possible Solution

Keep the issue open and gather timing proof on current main: compare with `channels.feishu.streaming=false` and `typingIndicator=false`, capture Feishu API timing for typing/card/send calls, and record model-catalog/provider-discovery timings around the same turn. If the 30s delay is the Feishu API timeout path, fix it in the Feishu plugin by bounding or decoupling noncritical typing/streaming-card startup from visible reply delivery; if the delay remains in catalog/runtime-deps work, route it through the Gateway/provider-discovery path instead.

## Reproduction Assessment

Unclear. The report gives a strong manual A/B reproduction across 2026.4.23 and 2026.4.27, but current main has not been live-tested and the available evidence does not isolate which code path accounts for the 30s delay.

## Solution Assessment
