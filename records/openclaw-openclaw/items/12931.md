## Summary

Keep this open. Current main partially reduced token cost for ordinary bare text-channel `/new` and `/reset`, but the requested public policy is still missing: reply-producing reset paths and the gateway agent reset path still build and send a hardcoded reset greeting prompt through an agent run, and the config schema only exposes startup-context preload controls, not greeting suppression or a cheaper greeting model.

## What This Changes

The issue requests config to either use a cheaper model for the bare `/new` and `/reset` greeting turn or suppress that greeting with a static acknowledgement.

## Best Possible Solution

Define one typed reset-greeting policy that covers static chat acknowledgements, reply-producing reset turns, and gateway agent resets, then wire it through config schema, exported types, help text, docs, and regression tests while preserving `/new <message>` and `/reset <message>` behavior.
