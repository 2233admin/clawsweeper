## Summary

Keep open. Current main still enforces the exact HTTPS and `api.firecrawl.dev` host restrictions reported here, the docs and tests still describe/assert that restriction, and an open implementation PR already references this issue with GitHub closing syntax, so the issue should stay open until a PR is reviewed and merged.

## What This Changes

The issue requests support for local, self-hosted Firecrawl endpoints, including HTTP on trusted local/private hosts and non-default HTTPS hosts.

## Best Possible Solution

Land a narrow Firecrawl plugin change that explicitly supports trusted self-hosted endpoints, validates HTTP endpoints as private/local using the existing SSRF SDK seam, preserves strict hosted Firecrawl behavior by default, and updates Firecrawl tests plus the public Firecrawl docs.

## Reproduction Assessment

Yes. The current helper path is deterministic: `resolveEndpoint("http://host.openshell.internal:3002/v1", "/v2/search")` reaches the `https` error, and an HTTPS non-`api.firecrawl.dev` host reaches the host-allowlist error; the existing test suite asserts the same rejection pattern.

## Solution Assessment

No. Current main is not a solution for the reported self-hosted use case; the safer implementation direction is an explicit self-hosted/private-network path rather than removing the Firecrawl endpoint checks unconditionally.
