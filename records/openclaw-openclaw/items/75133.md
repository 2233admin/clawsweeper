## Summary

Keep this PR open. Current main still drops bundle `activation` metadata before bundle registry records are built, and the PR threads the existing manifest activation normalizer through the Codex, Cursor, and Claude bundle path without introducing a new runtime-loading policy. I did not find a discrete blocking correctness or security issue in the static review; the remaining action is maintainer review and CI on the PR branch.

## What This Changes

The PR exports manifest activation normalization, stores normalized activation on Codex, Cursor, and Claude bundle manifests, copies it into bundle registry records, adds Claude bundle registry coverage, and stabilizes the gateway option-collision test environment.

## Best Possible Solution

Land the narrow metadata-preservation fix after maintainer review and CI, keeping activation as manifest/control-plane metadata for bundle records and avoiding special cases in compatibility reporting or runtime startup behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: on current main, a bundle manifest with `activation` is parsed into a `BundlePluginManifest` that has no activation field, then `buildBundleRecord` creates a `PluginManifestRecord` with activation still undefined.

## Solution Assessment

Yes. Reusing the existing manifest activation normalizer and threading the normalized value through bundle parsing and registry assembly is the narrowest maintainable fix for the reported behavior.
