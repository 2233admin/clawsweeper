## Summary

Keep open. Current main still implements cron as Gateway-local scheduling with local files, process-local serialization, local status/CLI surfaces, and observer-only plugin hooks; it has not added distributed leases, leader election, shared coordination backends, HA ownership status, or a cron leader command. The broader horizontal-scaling proposal is still open and does not supersede this cron-specific feature gap.

## What This Changes

Add distributed locking or leader election so cron jobs execute once across multiple Gateway instances in HA deployments.

## Best Possible Solution

Define a supported distributed cron ownership model for HA deployments, then implement it behind a versioned config or plugin/API boundary with backend health, leader or lease visibility, CLI inspection, docs, and tests that simulate separate Gateway processes sharing scheduled jobs.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: two separate Gateway processes sharing the same due cron job can race because the only lock is process-local and timer execution has no external lease before marking jobs running.

## Solution Assessment

Unclear. The issue’s Redis/etcd/DynamoDB approaches are plausible, but the narrowest maintainable fix needs maintainer decisions on ownership semantics, dependency boundaries, failure behavior, health/CLI surfaces, and relationship to #25883.
