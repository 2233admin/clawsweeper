## Summary

Keep this issue open. Current main still runs cron inside each Gateway with local store-path serialization and per-instance state isolation guidance, but it has no distributed cron ownership, leader election, Redis/etcd/DynamoDB lock backend, lock TTL config, coordination health, or `openclaw cron leader`-style status surface. The recent maintainer comment already kept this open and correctly distinguishes #69738 as a per-instance state isolation fix and #25883 as a broader horizontal-scaling discussion, not a replacement for this narrower HA cron coordination request.

## Best Possible Solution

Keep this open for maintainer design. A complete solution should define distributed cron ownership semantics, config/schema/docs, lease TTL and failure behavior, health and CLI visibility, tests that simulate multiple gateway processes/nodes, and its relationship to #25883. If maintainers do not want core-owned backends, the best path is a documented, versioned lock-provider or scheduler-ownership API that plugins/providers can implement.

## Risks / Open Questions

- The capability gap is real, but the implementation needs product and architecture decisions around backend support, lease expiry, fail-open/fail-closed behavior, health/status reporting, and whether coordination belongs in core or behind a versioned provider seam.
- #69738 and current multiple-gateway docs reduce accidental shared-state duplicate fires by recommending isolated state directories, but closing this would lose the remaining HA/shared-scheduler ownership requirement.
- Adding Redis/etcd/DynamoDB or similar dependencies would introduce operational and supply-chain/security review questions, so this should not be auto-promoted as a narrow fix without maintainer design.
