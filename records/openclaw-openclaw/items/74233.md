## Summary

Current main still contains the reported behavior: the embedded runner applies context-engine assembly, then passes the pre-assembly snapshot into the hard pre-prompt overflow check, where the helper chooses the larger unwindowed estimate. Because open closing PR #74255 is still unmerged in the provided context, this issue should remain open until that PR or an equivalent host-side fix lands.

## What This Changes

The issue requests that context-engine assembled messages become the default prompt-authoritative view for overflow precheck, with raw pre-assembly history treated only as an explicit opt-in gate or maintenance/debt signal.

## Best Possible Solution

Land or adapt #74255 so prompt admission checks the assembled prompt by default, preserves overflow protection for the actual submitted prompt, and exposes raw pre-assembly pressure only through an explicit opt-in policy or separate maintenance signal with matching tests and docs.

## Reproduction Assessment

Yes. Current source and the existing helper test reproduce the decision shape: a small assembled message window is paired with a larger unwindowed history, and the precheck still treats the larger estimate as the hard admission gate.

## Solution Assessment

No, current main is not the best solution because it still conflates prompt-admission size with raw-history pressure. The narrow maintainable direction is the generic context-engine contract change proposed in #74255, reviewed against plugin compatibility.
