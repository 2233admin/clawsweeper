## Summary

Keep this issue open. Current main still implements the reported Control UI device-token retry after a shared-token mismatch, the server still advertises that retry, tests and docs codify it as expected behavior, and the UI still lacks a first-class logout/bootstrap state that would satisfy the bundled session-cleanup and login-flash requests.

## What This Changes

The issue asks OpenClaw to invalidate Control UI browser sessions on gateway token rotation, add a logout path that clears persisted gateway/device auth state, and avoid rendering the login form before session restore completes.

## Best Possible Solution

Define the Control UI security contract for shared-token rotation first, then update the client/server auth paths so old shared-token browser sessions cannot silently recover through persisted device tokens unless maintainers intentionally keep that behavior. Add a visible logout action that clears sessionStorage gateway token state, local device auth state, and the active GatewayBrowserClient, plus a restoring/bootstrap UI state before the login form is shown.
