## Summary

Current main still should not close this issue. The central Moonshot/Kimi zero-balance 429 masking path remains: generic HTTP 429 classification wins before explicit billing text can override it, Moonshot has no provider-owned classifier hook, and billing versus rate-limit classification drives different auth-profile recovery behavior. Adjacent improvements exist, including local Control UI /model handling and CJK billing patterns, but they do not solve Moonshot 429 billing classification.

## Best Possible Solution

Keep this issue open. Add a Moonshot-owned classifyFailoverReason hook for 429 payloads whose code or message clearly indicates insufficient balance, credits, top-up, or equivalent Chinese balance wording; add focused regression tests for classification and user-facing copy; verify auth-profile state records billing disabledReason instead of a rate_limit cooldown. Then retest whether background label generation and model-switch escape paths still reproduce after the provider classification is corrected.
