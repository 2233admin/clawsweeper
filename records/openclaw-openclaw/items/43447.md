## Summary

Keep open. Current main still preserves the central failure mode: explicit HTTP 429 wins as `rate_limit` before balance/billing evidence can change the failover reason, mixed rate-limit/billing message text is classified as rate-limit first, and user-facing assistant error copy checks transient rate-limit copy before billing copy. The TeamoRouter workaround mentioned in the discussion can route around provider failures, but it does not fix OpenClaw's Moonshot/Kimi classification path.

## What This Changes

The issue asks OpenClaw to classify Moonshot/Kimi exhausted-balance 429 responses as billing/insufficient funds, avoid retry-loop lockout, and preserve a provider-independent model-switch escape path.

## Best Possible Solution

Add regression coverage for a Moonshot/Kimi HTTP 429 payload that contains explicit exhausted-balance evidence, then make billing evidence override generic 429/rate-limit classification and assistant UI copy while preserving ordinary rate-limit handling. Use the existing generic classifier precedence or the provider-owned classifier hook; keep broader retry, circuit-breaker, and command-routing redesign out of scope unless separately reproduced.
