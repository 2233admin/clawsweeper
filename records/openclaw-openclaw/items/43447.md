## Summary

Keep open. Current main still has the reported precedence bug: a Moonshot/Kimi response that carries HTTP 429 plus exhausted-balance text is classified and formatted as `rate_limit` before billing evidence can win. The TeamoRouter comment is useful external failover context, but it is a workaround outside OpenClaw and does not fix the core classifier path.

## What This Changes

The issue requests clearer Moonshot/Kimi billing error classification and better recovery when a zero-balance API response is surfaced as a rate-limit failure.

## Best Possible Solution

Add focused regression coverage for a Moonshot/Kimi 429 response that contains explicit exhausted-balance evidence, then make high-confidence billing signals override generic 429/rate-limit classification and assistant copy while preserving ordinary 429 rate-limit behavior. Keep broader retry, circuit-breaker, and command-routing changes separate unless tests prove they remain necessary after the masking bug is fixed.

Yes. A source-level reproduction is clear: pass a status 429 error or a raw `429 ... insufficient_balance`/balance payload through the failover classifier or assistant formatter and current main follows the rate-limit branch before billing can win.

## Solution Assessment

Yes. The narrowest maintainable fix is a classifier/UI precedence change with regression tests; provider-specific Moonshot handling may be useful only if a real payload needs a more precise matcher than the shared billing signals already provide.
