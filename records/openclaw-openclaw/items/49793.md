## Summary

Keep this PR open. Current main covers Xiaomi image-capable model metadata for `mimo-v2-omni`, and the generic config-provider fallback can use that for image understanding, but Xiaomi still has no registered media-understanding provider with `describeVideo`. The submitted PR is a relevant implementation candidate for the remaining video feature, but it is stale against the current plugin entrypoint, manifest metadata contract, and provider HTTP helper split.

## Best Possible Solution

Keep this PR open and ask for a current rework. The best path is to add a Xiaomi media-understanding provider in `extensions/xiaomi`, declare it through `contracts.mediaUnderstandingProviders` and `mediaUnderstandingProviderMetadata`, default image and video media understanding to `mimo-v2-omni`, use the shared OpenAI-compatible video helpers plus `openclaw/plugin-sdk/provider-http` request-policy helpers, and add focused tests for Xiaomi registration, manifest metadata, routing/default model selection, and HTTP request behavior.

## Risks / Open Questions

- Security review: the PR adds a new remote HTTP path that sends base64 video bytes and a Xiaomi API key; a current rework should use `openclaw/plugin-sdk/provider-http` request-policy helpers so proxy, dispatcher, private-network, timeout, TLS, and attribution behavior match other providers.
- The PR currently has no focused Xiaomi media-understanding tests for manifest metadata, registration, default image/video model selection, request shape, or transport-policy forwarding.
- The PR’s current provider advertises image support directly; current main already handles Xiaomi image understanding through image-capable model metadata, so the rework should avoid reintroducing text-only default-model failures and explicitly default media understanding to `mimo-v2-omni`.
- The patch targets an older plugin entrypoint shape and must be rebased onto `defineSingleProviderPluginEntry` plus manifest-first capability metadata before it is mergeable.
