## Risks / Open Questions

- Merging the PR as-is would likely fail current-main typechecking because provider HTTP helpers are no longer exported from `openclaw/plugin-sdk/media-understanding`; the current SDK expects those from `openclaw/plugin-sdk/provider-http`.
- Registering Xiaomi for video without manifest `mediaUnderstandingProviderMetadata.defaultModels.video` risks reusing the active/default Xiaomi text model `mimo-v2-flash` for video requests instead of `mimo-v2-omni`.
- The PR’s hand-built video HTTP path sends user video bytes and a Xiaomi bearer token, so it should use `resolveProviderHttpRequestConfig` and forward dispatcher/private-network/proxy policy into `postJsonRequest` like Moonshot does.
- The `reasoning_content` fallback mirrors Moonshot, but for Xiaomi it should be an intentional tested choice so raw reasoning is not unexpectedly surfaced as a user-facing video description.
- The PR currently lacks focused Xiaomi tests for manifest metadata, registration, default model selection, request shape, and provider transport-policy forwarding.
