## Summary

Keep open. Current main still rejects empty Bedrock `aws-sdk` auth in the media-understanding paths before any provider call, and the active closing PR is still open and unmerged.

## What This Changes

The issue asks for the image and audio/video media-understanding auth paths to allow Bedrock `aws-sdk` auth without a static API key, matching the chat path behavior.

## Best Possible Solution

Review, land, or supersede #72092. The desired end state is for the image runtime, MiniMax fallback, and audio/video provider auth path to pass an empty API key only for `aws-sdk` auth, avoid storing empty runtime keys, preserve existing failures for other auth modes, and add regression coverage for Bedrock media calls.

## Reproduction Assessment

Yes. The issue gives concrete steps, and current main shows a source-level reproduction that does not need live AWS: an auth result with `mode: "aws-sdk"` and no `apiKey` reaches `requireApiKey`, which throws immediately.

## Solution Assessment

Yes. Mirroring the chat path's missing-key allowance in the media-understanding auth resolvers is the narrow maintainable fix; weakening `requireApiKey` globally would be broader than necessary.
