## Summary

Keep open. Current main still rejects Bedrock `aws-sdk` media auth before any provider call in the image, MiniMax fallback, and audio/video paths, while the linked implementation PR #72092 is open and unmerged. The workaround/context in the discussion is useful, but it does not replace the core media auth fix.

## What This Changes

The issue requests that media-understanding image/audio/video paths allow Bedrock `aws-sdk` auth without requiring a static API key, matching the chat path behavior.

## Best Possible Solution

Review, rebase if needed, and land or supersede #72092. The desired end state is for media-understanding to pass an empty API key only for `aws-sdk` auth, avoid storing empty runtime keys, preserve missing-key failures for other auth modes, handle audio/video key rotation, and keep focused regression tests for Bedrock media calls.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with `mode: "aws-sdk"` and no `apiKey`, current media auth paths call `requireApiKey`, which throws before any live AWS request is attempted.

## Solution Assessment

Yes. Mirroring the chat path's missing-key allowance in the media auth resolvers and key-rotation path is the narrow maintainable fix; weakening `requireApiKey` globally would be broader and riskier.
