## Summary

Keep open. Current main still models heartbeat as a single per-agent heartbeat config/schedule with an optional one-session override and targeted wake path; it does not implement durable per-session heartbeat entries, per-entry enabled/run state, or first-class entry management. Cron already has persistent session-targeted scheduling and run history, but the code and docs keep that separate from heartbeat.

## Best Possible Solution

Keep this open as a core automation design request. A complete implementation would add durable heartbeat entries scoped to agent/session, explicit skip/queue/parallelism semantics, persisted enabled and last-run state, RPC/CLI/UI inspection and mutation, and docs that clearly separate heartbeat-entry scheduling from session-local gating and cron.

## Risks / Open Questions

- A native design should avoid creating a second durable scheduling truth that conflicts with cron jobs, cron run history, and existing session-target routing.
- Existing targeted wake/session override code may look close, but it is request routing for individual runs, not restart-safe per-session enrollment with operator-visible state.
