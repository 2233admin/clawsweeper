## Summary

Current main does not implement this PR’s intended 80-message Control UI chat hydration/render guard. The PR remains relevant, but it is stale against current main because render slicing now lives in `ui/src/ui/chat/build-chat-items.ts`, while the PR still changes the older view-local render path and its view test would not prove the active code path.

## Best Possible Solution

Keep this PR open and ask for a current-main refresh. The best implementation is to define one shared initial chat history window constant in `ui/src/ui/chat/history-limits.ts` or a nearby chat constants module, use it from `ui/src/ui/controllers/chat.ts` for the `chat.history` request and from `ui/src/ui/chat/build-chat-items.ts` for the render slice/notice, decide whether input-history should intentionally follow that same limit, then update `ui/src/ui/controllers/chat.test.ts` and add direct coverage in `ui/src/ui/chat/build-chat-items.test.ts`. No Gateway protocol, dependency, workflow, or security-sensitive change is needed.
