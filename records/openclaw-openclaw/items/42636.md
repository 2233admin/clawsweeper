## Summary

Current main does not implement this PR's intended 80-message Control UI hydration/render window. The chat controller still requests 200 history messages, the active chat item builder still renders the last 200 messages, and the current controller test still asserts the old limit. The PR remains plausible, but it needs a refresh because current render slicing lives in `ui/src/ui/chat/build-chat-items.ts`, not only the older view-local path shown in the PR diff. The security pass did not find supply-chain or secret-handling concerns in the provided diff.

## Best Possible Solution

Keep this PR open and refresh it against current main. The best implementation is a narrow Control UI change that introduces or reuses one shared initial chat history/render window constant, wires it through `loadChatHistory()` and `buildChatItems()`, explicitly decides whether `input-history.ts` should stay aligned, and adds direct tests for both the controller request limit and builder truncation. No gateway protocol, dependency, CI, or security-sensitive change appears necessary.
