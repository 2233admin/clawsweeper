## Summary

Keep this PR open. Current main still uses a 200-message Control UI chat.history request and a 200-message render window, so the intended 80-message hydration/render guard has not been implemented. The PR is still relevant, but it is stale against current main because active render slicing now lives in `ui/src/ui/chat/build-chat-items.ts`, while this PR still changes the older view-local render path.

## Best Possible Solution

Keep this PR open and ask for a current-main refresh. The best implementation is to define one shared initial chat history window constant in `ui/src/ui/chat/history-limits.ts` or a nearby chat constants module, use it from `ui/src/ui/controllers/chat.ts` for the `chat.history` request and from `ui/src/ui/chat/build-chat-items.ts` for the render slice/notice, keep input-history alignment intentional, update `ui/src/ui/controllers/chat.test.ts`, and add a direct assertion in `ui/src/ui/chat/build-chat-items.test.ts`. No Gateway protocol, server, dependency, workflow, or security-sensitive change is needed.
