## Summary

Current main still has the 200-message Control UI chat hydration and render window, so this PR’s intended behavior is not implemented. The PR remains relevant, but it is stale against current main: rendering now slices history in `ui/src/ui/chat/build-chat-items.ts`, while the provided PR diff still changes the older view-local render path and adds a view test that mocks the active builder.

## Best Possible Solution

Keep this PR open and ask for a current-main refresh. The best path is to define one shared initial chat history window constant near the chat history/render limits, use it from `ui/src/ui/controllers/chat.ts` and `ui/src/ui/chat/build-chat-items.ts`, decide whether input-history should intentionally share the same limit, then update `ui/src/ui/controllers/chat.test.ts` and add direct coverage in `ui/src/ui/chat/build-chat-items.test.ts`. No Gateway protocol or security-sensitive change appears necessary.
