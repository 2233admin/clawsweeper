## Summary

Keep this PR open. Current main still exposes declared node commands on the unapproved reconnect path, current tests still assert that behavior, and the public pairing docs/changelog describe the opposite gate. The PR is also paired with the still-open contributor issue #65168, so it should stay available for maintainer/security review rather than being closed by cleanup.

## What This Changes

The PR changes node reconnect reconciliation to publish an empty command set until node pairing exists, and updates gateway tests so pre-pairing invocation is blocked while legitimate invoke fixtures are explicitly node-paired.

## Best Possible Solution

Land this PR or an equivalent maintainer-approved replacement that fails closed for unapproved node reconnects by exposing no commands until `node.pair.approve` succeeds, updates the allowlist and invoke regression tests, and explicitly decides whether `node.invoke` also needs a direct approved-pairing guard.
