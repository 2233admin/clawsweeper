## Summary

Keep open. Current main still sends ZAI `glm-4.6v` image understanding through the generic OpenAI-compatible streaming image path, and I found no merged ZAI-specific abort, payload, streaming, or endpoint fix. The duplicate report and open adjacent PR are useful context, but they do not prove this abort path is fixed.

## What This Changes

The issue reports that configuring ZAI `glm-4.6v` as the image model makes the bundled image tool abort while direct API calls with the same key and image succeed.

## Best Possible Solution

Keep this as the canonical ZAI `glm-4.6v` abort report until there is a focused regression for the image tool against the reported coding-endpoint configuration, followed by the narrowest ZAI-owned fix in `extensions/zai` or a generic provider-facing image transport seam if the failure proves cross-provider.

## Reproduction Assessment

Yes. Two corroborating reports give a concrete path: configure ZAI with the coding endpoint, set `agents.defaults.imageModel` to `zai/glm-4.6v`, then send an image through Telegram or the image tool; I did not run the live API, but current main still follows that code path.

## Solution Assessment

Unclear. There is no merged fix, and the open route-only PR may address a related 404 class without proving this immediate abort symptom; the safer path is to pin the failing image-tool request before choosing endpoint routing, streaming options, payload transformation, or timeout handling.
