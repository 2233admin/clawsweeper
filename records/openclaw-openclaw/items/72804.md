## Summary

Current main already solves the reported workflow: `browser.extraArgs` is a validated config surface, `--proxy-server=...` is recognized as a Chrome proxy-control flag so OpenClaw does not add `--no-proxy-server`, and the docs explicitly direct users to this config for managed-browser proxying. The proxy-env stripping is intentional safety behavior, not a missing requirement for the explicit Chrome proxy path.

## What This Changes

Request to add a config-based browser proxy option so users can route managed Chrome through a proxy without editing compiled JavaScript after updates.

## Best Possible Solution

Keep the existing generic `browser.extraArgs` path for single managed-browser proxy routing and preserve direct-by-default/env-stripped launches for SSRF safety; broader per-profile or authenticated proxy ergonomics should stay in the separate per-profile proxy/isolation discussion rather than reopening this duplicate request.

## Reproduction Assessment

Not applicable as a feature request, but the requested behavior has a high-confidence verification path: validate a config with `browser.extraArgs: ["--proxy-server=http://127.0.0.1:7890"]` and inspect `buildOpenClawChromeLaunchArgs()` to see the proxy flag included without `--no-proxy-server`. The repo already has regression tests for both checks.

## Solution Assessment

Yes. The current solution is narrower and more maintainable than adding a new `browser.proxy` key for this case because it uses Chrome’s existing proxy flags, keeps defaults unchanged, and documents the explicit opt-in path while leaving proxy-env stripping intact for safety.
