## Summary

Keep this PR open. Current main still does not fire `session_end` from the subagent failed-spawn path or cron isolated-agent completion path, and the linked feature request #10142 remains open. The PR is a plausible implementation candidate, but it needs rework before merge because the proposed failed-subagent payload appears to use the run/idempotency ID as `sessionId`, cron failure/early-return coverage is incomplete, and no regression tests are included.

## Best Possible Solution

Keep this PR open but request changes: emit `session_end` with the stored child session ID, move cron emission to a finalization/finally-style boundary that covers success, abort, and error outcomes if that is the intended contract, update hook docs/types/changelog after semantics are settled, and add focused regression tests. If maintainers decide `cron_changed` is the preferred cron integration surface, document that explicitly and narrow this PR to the remaining session lifecycle gap tracked by #10142.

## Risks / Open Questions

- The PR as proposed appears to send `childRunId` as `sessionId` for failed subagent spawn, which can break plugins that correlate session_start/session_end or look up stored session metadata by session ID.
- The cron insertion shown in the PR covers the post-execution finalize path but not all failure and early-abort returns, so plugins could still miss end signals when cron jobs error out.
- No tests are included in the PR file list; hook emission semantics should have focused coverage for subagent failed spawn, cron success, cron abort, and cron failure paths.
- Adding session_end emissions expands plugin execution on subagent and cron termination paths, so the implementation should preserve guards, failure isolation, and avoid exposing internal delivery or usage details unintentionally.
