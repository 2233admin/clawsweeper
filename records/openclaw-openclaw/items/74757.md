## Summary

Keep open. Current main still has the core mismatch reported in the discussion: Slack actions are implemented as the Slack channel message-action surface, but the Claude/ACPX MCP tool bridge does not expose a callable Slack tool; the requested admin/app-manifest action group is also not present. The reporter's follow-up is useful because it narrows this to a maintainer architecture and security decision rather than a conservative cleanup close.

## What This Changes

The issue requests exposing Slack actions to Claude Code ToolSearch and adding opt-in Slack admin/app-management actions with user/app configuration tokens and action-group gating.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split the work into a Slack-owned callable bridge or harness skill-gating fix plus a separate opt-in admin action group. The admin path should use SecretRef-compatible config, clear Slack scope docs, least-privilege token selection, owner/session policy checks, and targeted tests across Slack action runtime and MCP/tool discovery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main advertises a `slack` skill, registers Slack as a channel plugin, and exposes no Slack tool through the OpenClaw-tools MCP bridge used by ACPX/Claude Code.

## Solution Assessment

Unclear. The proposed direction is plausible, but the best maintainable fix depends on maintainer choice between MCP bridging, skill loading suppression, or both; the high-privilege admin actions should be reviewed as a separate opt-in feature.
