## Summary

Keep open. Current main still matches the report: Slack actions are implemented as a Slack channel message-action surface, the ACPX/OpenClaw-tools MCP bridge available to Claude Code does not expose a Slack callable tool, and the requested Slack workspace/app administration actions are absent. The reporter's follow-up correctly identifies this as an architecture and security decision rather than a conservative cleanup close.

## What This Changes

Request to expose Slack channel actions to Claude Code/ACPX ToolSearch and add opt-in Slack admin/app-manifest actions with user/app configuration token plumbing.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split the work into a Slack-owned bridge or skill-loading fix plus a separate opt-in admin action group. The admin path should be disabled by default, use SecretRef-compatible user/app-configuration token resolution, document required Slack scopes, and add tests for Slack action runtime, config schema, MCP/tool discovery, and ACPX/Claude visibility.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main loads a prompt-visible Slack skill, implements Slack actions as a channel-plugin message-action adapter, and exposes no Slack action through the OpenClaw-tools MCP bridge that Claude Code can call.

## Solution Assessment

Unclear. The proposed direction is plausible, but the narrowest maintainable fix depends on maintainer choice between adding a Slack MCP bridge, suppressing unreachable skill guidance, or both; the high-privilege admin operations should be reviewed as a separate opt-in feature.
