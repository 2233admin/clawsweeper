## Summary

Keep open. Current main still matches the report: Slack is implemented as a channel plugin with message actions and a prompt skill, but the Claude/ACPX OpenClaw MCP bridge does not expose a callable Slack tool, and the requested Slack admin/app-manifest actions are not present. This needs maintainer architecture and security direction before implementation.

## What This Changes

The issue requests exposing Slack actions to Claude Code through OpenClaw tool/MCP discovery and adding opt-in Slack admin actions for channel, user, and app-manifest management.

## Best Possible Solution

Define the Claude/ACPX Slack tool-exposure contract first, then split implementation into a Slack-owned bridge or skill-loading fix and a separate disabled-by-default Slack admin action group with SecretRef-compatible config, least-privilege token resolution, Slack scope docs, and focused Slack plus MCP discovery tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: Slack prompt guidance is present, Slack actions are channel-message adapters, the OpenClaw tools MCP bridge returns only cron, and Claude CLI only allows `mcp__openclaw__*` tools.

## Solution Assessment

Unclear. The proposed direction is plausible, but the narrowest maintainable fix depends on maintainer choice between MCP bridging, skill visibility gating, or both; the admin operations should be reviewed as a separate opt-in security-sensitive feature.
