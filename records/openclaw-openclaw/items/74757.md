## Summary

Keep open. Current main still leaves the Slack tool exposure mismatch unresolved and the requested Slack admin/app-manifest action surface is not implemented; the discussion also correctly narrows this to an ACPX/MCP architecture and security decision rather than conservative cleanup.

## What This Changes

The issue requests exposing Slack actions to Claude Code/ToolSearch and adding opt-in Slack admin operations such as channel creation, invites, member lookup/listing, app manifest create/update, and archiving.

## Best Possible Solution

Split the work into a maintainer-approved Slack/Claude tool-exposure contract and a separate Slack-owned admin action group. The admin path should stay default-off, use SecretRef-compatible token resolution, document required Slack scopes, and add focused Slack action-runtime plus MCP/tool-discovery tests.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: current main has a Slack channel action adapter and prompt skill, while the OpenClaw MCP bridge exposes no Slack tool and Claude is constrained to OpenClaw MCP tool names; no live Claude Code ToolSearch run was needed for this read-only review.

## Solution Assessment

Unclear. The proposed direction is plausible, but the best fix depends on maintainer choice between MCP bridging, skill-loading changes, or both, and the admin operations should be reviewed as a separate opt-in security-sensitive capability.
