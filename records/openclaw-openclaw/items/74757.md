## Summary

Keep open. Current main still has the reported mismatch: the bundled Slack skill tells agents to call a `slack` tool, but Slack is implemented as a channel message-action surface and the ACPX/OpenClaw-tools MCP bridge does not expose a Slack callable tool to Claude Code. The admin/app-manifest actions are also still absent, so this needs maintainer/security direction rather than cleanup closure.

## What This Changes

Request to make Slack channel actions discoverable and callable from Claude Code/ACPX, and to add opt-in Slack admin/app-manifest actions with user/app configuration token support.

## Best Possible Solution

Define the Claude/ACPX tool-exposure contract first, then split the work into a Slack-owned bridge or skill-loading fix plus a separate opt-in admin action group. The admin path should remain Slack-owned, SecretRef-compatible, least-privilege by token type, disabled unless explicitly configured, and covered by Slack action/runtime plus ACPX/MCP discovery tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main advertises a `slack` skill, registers Slack as a channel plugin with channel-message actions, and exposes no Slack tool from the OpenClaw-tools MCP bridge used by ACPX/Claude Code.

## Solution Assessment

Unclear. The proposed direction is plausible, but the narrowest maintainable fix depends on maintainer choice between bridging Slack actions, hiding unreachable skill guidance, or both; the high-privilege admin actions should be reviewed as a separate opt-in feature.
