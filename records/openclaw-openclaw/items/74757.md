## Summary

Keep open. Current main still lacks a Claude Code/ToolSearch-discoverable Slack MCP/tool surface and the requested Slack admin/app-manifest actions; the existing `userToken`/`userTokenReadOnly` support is only partial plumbing. The reporter's follow-up matches the code: this needs maintainer direction across MCP/skill exposure and high-privilege Slack admin behavior.

## What This Changes

The issue requests making Slack channel actions discoverable to Claude Code/ToolSearch and adding opt-in Slack admin/app-manifest actions with user/app-configuration token plumbing.

## Best Possible Solution

Settle the Claude Code exposure contract first, then implement Slack admin as a separate explicitly opt-in action group with SecretRef-compatible `userToken`/app-configuration token semantics, least-privilege scope docs, owner-aware gating, and focused tests for the MCP/tool bridge plus Slack action runtime.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the current MCP resolver does not include Slack, the Claude backend allows only `mcp__openclaw__*`, and the requested Slack admin action names/endpoints are absent from the runtime and docs.

## Solution Assessment

Unclear. The requested direction is plausible, but the best implementation depends on maintainer product/security choice; a small conditional skill-loading fix could reduce confusion, but it would not solve the missing callable Slack/admin surface.
