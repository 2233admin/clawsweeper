## Summary

Keep open. Current main still has the mismatch described in the discussion: Slack is a bundled channel plugin with channel message actions, while the Claude/ACPX MCP surfaces do not expose a callable Slack tool, and the requested Slack admin/app-manifest operations are not implemented. The reporter's follow-up is supported by the source, and the remaining work needs maintainer/security direction rather than conservative cleanup.

## What This Changes

The issue requests a Claude Code-discoverable Slack tool surface plus opt-in Slack admin actions for channel creation/invites/archive, user lookup/listing, and Slack app manifest create/update.

## Best Possible Solution

Define the Claude/ACPX exposure contract first, then split implementation into a callable Slack bridge or skill-loading fix plus a separate Slack-owned admin action group. The admin path should be opt-in, SecretRef-compatible, least-privilege documented, tested through the Slack runtime and MCP/tool bridge, and avoid making high-privilege app manifest operations available by default.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: current main loads Slack as a channel plugin/action adapter while the OpenClaw MCP bridge and Claude allowed-tools configuration do not expose a Slack callable tool, and exact searches show the requested admin endpoints are absent.

## Solution Assessment

Unclear. The requested direction is plausible, but the best fix depends on maintainer choice between an MCP bridge, conditional skill loading, or both; Slack admin actions should be a separate opt-in security-reviewed change rather than bundled into a small cleanup.
