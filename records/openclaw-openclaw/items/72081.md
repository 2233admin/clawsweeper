## Summary

Keep open. Current `main` still preserves dashboard token redaction but does not solve the stranded headless/WSL path when clipboard and browser delivery fail; the open closing PR #72802 remains the active implementation candidate, while #73227 is closed and unmerged.

## What This Changes

The issue reports that `openclaw dashboard` prints a dashboard URL without a usable gateway token path on WSL/headless setups.

## Best Possible Solution

Review and land #72802, or an equivalent narrow dashboard command change, so automatic token delivery stays redacted while failed clipboard/browser delivery prints safe manual recovery guidance pointing to `OPENCLAW_GATEWAY_TOKEN` or `gateway.auth.token` without logging the token value.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is a plaintext `gateway.auth.token` with clipboard copy returning false and either browser open unavailable or `--no-open`; current `main` then logs only the bare URL/fallback hint without a manual token recovery path.

## Solution Assessment

Yes. The best fix is owner-local to `src/commands/dashboard.ts` with a focused regression test: keep tokenized URLs limited to clipboard/browser delivery, keep logs redacted, and add guidance only when automatic delivery fails.
