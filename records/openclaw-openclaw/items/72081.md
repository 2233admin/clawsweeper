## Summary

Keep open. Current main still prints only the clean dashboard URL in the terminal while relying on clipboard/browser delivery for the tokenized URL, so WSL/headless users remain stranded when both delivery channels fail; two open PRs already reference this issue with closing syntax and should be reviewed instead of closing the issue early.

## What This Changes

The issue asks `openclaw dashboard` to provide a usable gateway-token authentication path when the printed URL does not include the token in a WSL/headless run.

## Best Possible Solution

Land one narrow dashboard fix that emits a manual authentication hint only when clipboard and browser delivery both fail, keeps the token value and `#token=` out of logs, and adds regression coverage for the combined-failure and `--no-open` cases.

## Reproduction Assessment

Yes. Configure a literal gateway token, make clipboard copy fail, and run in a headless/SSH/WSL path where browser open is unavailable; source inspection shows current main then logs only the bare URL and SSH/no-open hint.

## Solution Assessment

Yes. The narrowest maintainable direction is the one in the linked PRs: preserve the secure non-tokenized terminal log, but add a recovery hint that names the token source and fragment key when automatic delivery fails.
