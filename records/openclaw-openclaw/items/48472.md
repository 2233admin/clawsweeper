## Summary

Keep this PR open. Current main still does not fully implement the PR's central behavior, but the branch is now dirty against main and overlaps later gateway status, auth, and restart-health work, so it should not be closed or merged as-is.

## What This Changes

The PR changes daemon gateway status and restart-health probing to use a health-only probe, adds explicit-auth handling for daemon URL overrides, adjusts local restart probe timeout budgeting, and extracts explicit gateway auth helpers.

## Best Possible Solution

Rebase and narrow the PR against current main, preserving `--require-rpc` read-probe semantics and #72405's local-vs-remote credential separation while adding only the still-missing health-only daemon probe, strict override-auth boundary, and deadline-bounded restart probe timeouts with focused tests.

## Reproduction Assessment

Yes. Source and test inspection gives a high-confidence path: current main still lacks a health-only `probeGateway` detail mode, still uses `status` for `--require-rpc`, still uses a fixed 3s restart probe timeout, and still accepts a daemon status override path without explicit-auth enforcement.

## Solution Assessment

No, not as currently shaped. The direction is plausible, but the branch is stale and security-sensitive; the maintainable solution is a rebase that reconciles current probe/read semantics and resolves the transport and device-identity review findings before merge.
