## Summary

Keep open. Current main still has separate selection paths for the displayed OAuth profile and provider usage auth, and an open PR explicitly targets this issue with closing syntax, so closing it as stale or implemented would be premature.

## What This Changes

The requested fix is for status and session_status usage/quota lines to use the same active OAuth credential shown in the status auth label when multiple openai-codex profiles are configured.

## Best Possible Solution

Land a narrow generic provider-usage change that passes the session-selected auth profile into usage auth resolution, validates that the preferred profile belongs to the requested provider, preserves existing fallback order when it is absent or mismatched, and adds regression coverage for preferred-profile and mismatched-provider cases.

## Reproduction Assessment

Yes. The issue gives concrete multi-profile openai-codex OAuth steps, and current-main source inspection shows a high-confidence static reproduction path: the status label can prefer sessionEntry.authProfileOverride while usage auth resolves only by provider order.

## Solution Assessment

Yes. The maintainable fix should be a generic provider-usage preferred-profile seam used by status/session_status, not an openai-codex-only special case; open PR #59208 appears to target that boundary and should be reviewed rather than closing this issue early.
