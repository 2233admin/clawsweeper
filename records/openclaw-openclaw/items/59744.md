## Summary

Keep open. Current main still drops Discord attachment filenames during inbound media saving, and there is an open linked fix PR with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Preserve original Discord inbound attachment filenames through media ingest so agents can see or infer user-provided filenames instead of UUID-only saved paths.

## Best Possible Solution

Review and land #59783, or an equivalent narrow Discord plugin change, so `attachment.filename` and sticker candidate names flow into `saveMediaBuffer` with focused Discord media tests. Keep this issue open until the implementation merges.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: create a Discord message attachment with `filename: "RailRoad_TrainStation_photo.png"`; current main uses that value only as a fetch hint and saves the buffer without passing it to `saveMediaBuffer`, so the stored inbound path falls back to UUID-based naming.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable fix because the shared media store already supports `originalFilename`; the Discord media resolver should pass the API-provided filename and add regression coverage rather than changing core media payload contracts first.
