## Summary

Keep open. Current main does not implement a drain manifest, and the PR branch still has merge-blocking lifecycle, schema, and persistence issues: the manifest is deleted on the same successful restart shutdown path that writes it, the registry key is persisted as a session id even though gateway code uses it as a run id, and the recovery file is written directly with default permissions.

## What This Changes

The PR adds a ChatRunRegistry snapshot API, a new drain-manifest read/write/delete module with tests, close-handler wiring for shutdown snapshots, and marks secrets-reloader system events untrusted.

## Best Possible Solution

Revise the PR so shutdown writes a durable owner-only atomic manifest only when active work was snapshotted, preserves that manifest until recovery or an explicit no-active-work cleanup path, uses an accurate runId/session schema, and adds close-handler coverage for empty and active-run shutdown paths.

## Reproduction Assessment

Yes. A focused code review reproduces the main failure: with a non-empty registry, the PR writes the manifest, clears chat state, then deletes the manifest before the next boot can read it.

## Solution Assessment

No. The snapshot API is a reasonable seam, but the current lifecycle, schema naming, and persistence implementation are not the safest maintainable solution for recovery state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve active-run manifests across restart shutdown:** `src/gateway/server-close.ts:471-474`
  - body: After writing the drain manifest, the close handler unconditionally deletes it after any successful close. A normal SIGTERM/restart with in-flight runs therefore removes the only recovery marker before the next boot can read it, so delete only when no active work was snapshotted or after recovery cleanup.
  - confidence: 0.93
- **[P2] Record the registry key as a run id:** `src/gateway/server-drain-manifest.ts:48-52`
  - body: The value written as sessionId comes from the chat-run registry key, but current gateway code registers and looks up this map by run id. A boot consumer that treats this as a persisted session id can resume or route the wrong thing; rename the field to runId or persist an actual session id.
  - confidence: 0.88
- **[P2] Write the manifest atomically with owner-only permissions:** `src/gateway/server-drain-manifest.ts:67`
  - body: draining-sessions.json contains sessionKey and run identifiers, but it is written directly to the final path with default permissions. Use the existing owner-only temp-file, fsync, rename pattern so local readers cannot observe private metadata or partial JSON after interruption.
  - confidence: 0.87
- **[P2] Snapshot process.env before mutating it:** `src/gateway/server-drain-manifest.test.ts:12-25`
  - body: ORIGINAL_ENV stores the live process.env object, so beforeEach mutates the captured value and afterEach can leak OPENCLAW_STATE_DIR or OPENCLAW_TEST_FAST into later tests. Capture a shallow copy up front or use the existing env-stubbing helpers.
  - confidence: 0.84
- **[P3] Return write failures to the close handler:** `src/gateway/server-drain-manifest.ts:71-73`
  - body: writeDrainManifest catches write errors internally, so the caller's try/catch will not record drain-manifest-write and shutdown can report cleanly after failing to snapshot recovery state. Return a status or throw so the close result records the warning.
  - confidence: 0.78
