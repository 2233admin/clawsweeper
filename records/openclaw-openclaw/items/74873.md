## Summary

Keep open. Current main does not have the drain-manifest implementation, and this PR is still useful but not ready to merge because the proposed shutdown path deletes the recovery marker before the next boot, stores a run-registry key under a misleading sessionId field, and writes recovery metadata without the repository's owner-only atomic state-file pattern.

## What This Changes

The PR adds a ChatRunRegistry snapshot API, a new drain-manifest read/write/delete module with tests, close-handler wiring to write/delete the manifest during gateway shutdown, and a small server.impl.ts change that marks secrets-reloader system events untrusted.

## Best Possible Solution

Revise the PR so shutdown snapshots active runs into a durable owner-only atomic manifest that remains available when active work existed, uses either a real persisted session id or an explicit runId field, and leaves cleanup to the no-active-work path or the boot-time recovery consumer. Add close-handler coverage for the active-run restart path and keep the manifest schema aligned with the AI-588 consumer contract.

## Reproduction Assessment

Yes. A focused code-review reproduction is enough: with a non-empty registry, the PR writes the manifest, clears chat state, then unconditionally deletes the manifest before returning from close, so the next boot has no recovery file to read.

## Solution Assessment

No. The registry snapshot seam is a reasonable direction, but this implementation uses the wrong lifecycle boundary, a misleading schema field, and an unsafe persistence pattern; the safer fix is durable owner-only atomic state with cleanup after no-active-work or confirmed recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve manifests for active shutdown recovery:** `src/gateway/server-close.ts:471-474`
  - body: After writing the manifest, the close handler clears chat state and then deletes the manifest on every successful close. A normal SIGTERM/restart with in-flight runs therefore removes the only recovery marker before the next boot can read it; only delete when no active work was snapshotted or after confirmed recovery cleanup.
  - confidence: 0.93
- **[P2] Store the registry key as a run id:** `src/gateway/server-drain-manifest.ts:49`
  - body: The value assigned to sessionId is the chat-run registry key, but current gateway code registers and looks up these entries by run id. A recovery consumer that treats this as a persisted session id can resume or route the wrong thing; rename the field to runId or persist a real session id.
  - confidence: 0.88
- **[P2] Use owner-only atomic writes for the manifest:** `src/gateway/server-drain-manifest.ts:67`
  - body: The manifest contains sessionKey and run identifiers, but it is written directly to the final path with default permissions. Use the existing temp-file plus 0o600 plus rename pattern so local readers cannot observe private metadata or partial JSON after interruption.
  - confidence: 0.86
- **[P2] Snapshot process.env before mutating it:** `src/gateway/server-drain-manifest.test.ts:12-25`
  - body: ORIGINAL_ENV stores the live process.env object, so beforeEach mutates the captured value and afterEach can leak OPENCLAW_STATE_DIR or OPENCLAW_TEST_FAST into later tests. Capture a shallow copy up front or use the existing env-stubbing helpers.
  - confidence: 0.86
- **[P3] Propagate manifest write failures to shutdown warnings:** `src/gateway/server-drain-manifest.ts:71-72`
  - body: writeDrainManifest catches write errors internally, so the caller's try/catch will not record drain-manifest-write in the shutdown result. Return a status or throw from the writer if close() is expected to report a failed recovery snapshot.
  - confidence: 0.78
