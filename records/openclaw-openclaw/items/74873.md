## Summary

Keep open. Current main does not implement drain-manifest session recovery, and this PR is a useful implementation candidate tied to #40712, but the latest head still has merge-blocking schema and persistence issues. The final commit fixed the earlier unconditional-delete and write-warning problems, but the manifest still records a run key as `sessionId`, writes recovery state directly with default permissions, adds Linear-specific metadata in gateway core, and leaves a test environment leak.

## What This Changes

The PR adds `ChatRunRegistry.entries()`, a new drain-manifest read/write/delete module, shutdown wiring to snapshot active chat runs before clearing state, tests for the manifest and close handler, and a small `server.impl.ts` trusted-event hardening change.

## Best Possible Solution

Revise this PR so the manifest uses a generic recovery schema with a real `runId` or actual persisted session ID, writes owner-only atomic JSON using the existing state-file pattern, keeps core free of Linear-specific metadata, and leaves stale-manifest TTL or checkpoint semantics to the boot-time recovery consumer with tests for the active-run and empty-shutdown paths.

## Reproduction Assessment

Yes. A focused code review reproduces the remaining issues: current main registers the chat-run registry by run/client IDs, while the PR serializes that key as `sessionId`, and the PR writes the final recovery file directly with `fs.writeFileSync`.

## Solution Assessment

No. The shutdown snapshot seam is reasonable, but the current schema and persistence mechanics are not the narrowest maintainable solution; the safer path is generic run metadata plus the existing atomic owner-only JSON writer or an equivalent sync temp-file/rename pattern.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Record registry keys as run IDs:** `src/gateway/server-drain-manifest.ts:49`
  - body: `entry.sessionId` is the chat-run registry key, but current callers register this map with `clientRunId`/agent `runId`. A boot recovery consumer that treats this field as a persisted session ID can resume or route the wrong session; rename this manifest field to `runId` or persist the actual session ID separately.
  - confidence: 0.89
- **[P2] Write the manifest atomically with private permissions:** `src/gateway/server-drain-manifest.ts:72`
  - body: `draining-sessions.json` contains session routing metadata but is written directly to the final path with the default file mode. If shutdown is interrupted, readers can see partial JSON, and on permissive umasks the file may be readable beyond the owner; use the existing atomic JSON/state-file pattern or an equivalent `0o600` temp-file rename.
  - confidence: 0.88
- **[P2] Remove Linear-specific metadata from gateway state:** `src/gateway/server-drain-manifest.ts:52`
  - body: The gateway core manifest now parses `linearTicketId` out of arbitrary session keys. Core recovery state should stay tracker-agnostic, and this regex can also misclassify unrelated keys; keep the manifest generic and let external tooling infer work-item metadata outside the gateway contract.
  - confidence: 0.81
- **[P3] Snapshot process.env before mutating it:** `src/gateway/server-drain-manifest.test.ts:13`
  - body: `ORIGINAL_ENV` stores the live `process.env` object, so the test mutations in `beforeEach` also mutate the captured value and `afterEach` can leak `OPENCLAW_STATE_DIR`/`OPENCLAW_TEST_FAST` into later tests. Capture `{ ...process.env }` up front or use `vi.stubEnv`/existing env helpers; the same pattern was also added in the close-handler test.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces persistent local recovery state containing session routing metadata, but the file write is not atomic and does not force owner-only permissions.

Concerns:

- **[low] Manifest file may expose or corrupt routing metadata:** `src/gateway/server-drain-manifest.ts:72`
  - body: `draining-sessions.json` stores `sessionKey`, run IDs, and derived work metadata, but `fs.writeFileSync` writes directly to the final path with default creation mode. Use an owner-only atomic write so local readers cannot observe partial JSON or metadata created under a permissive umask.
  - confidence: 0.88
