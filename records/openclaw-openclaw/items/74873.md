## Summary

Keep open. Current main does not already implement a drain-manifest recovery marker, and this PR is still the active implementation path, but the branch is not mergeable as-is because the manifest is deleted after a successful shutdown and the new state file is written without the existing durable owner-only pattern.

## What This Changes

The PR adds a ChatRunRegistry snapshot API, a new gateway drain-manifest read/write/delete module and tests, close-handler manifest wiring, and a small server.impl.ts change that marks secrets-reloader system events untrusted.

## Best Possible Solution

Revise this PR so shutdown records a durable, owner-only, atomic manifest only when active work is actually snapshotted; keeps that marker until there was no active work or a recovery consumer explicitly clears it; names the persisted identifier according to the real run/session semantics; and adds close-handler coverage for both empty and active-run shutdown paths.

## Reproduction Assessment

Yes. A source review reproduces the main failure path: with a non-empty registry, the PR writes the manifest, clears chat state, completes shutdown, and then deletes the manifest before the next boot can consume it.

## Solution Assessment

No. The snapshot seam is a reasonable direction, but this branch's lifecycle and persistence details are not the safest maintainable implementation; the safer path is an atomic owner-only recovery marker with cleanup tied to no-active-work or confirmed recovery completion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve manifests when active runs were snapshotted:** `src/gateway/server-close.ts:471-474`
  - body: After writing the drain manifest, the close handler deletes it after every successful shutdown. A normal SIGTERM/restart with active runs therefore removes the recovery marker before the next boot can read it, so gate deletion on an empty snapshot or move cleanup to the recovery consumer.
  - confidence: 0.93
- **[P2] Persist the registry key as a run id:** `src/gateway/server-drain-manifest.ts:49-51`
  - body: The value written as sessionId is the chat-run registry key, but current gateway code peeks and shifts that registry by evt.runId. A boot consumer treating this field as a persisted session id can resume or route the wrong thing; rename it to runId or persist an actual session id.
  - confidence: 0.88
- **[P2] Write the manifest atomically with owner-only permissions:** `src/gateway/server-drain-manifest.ts:67`
  - body: draining-sessions.json includes session routing metadata, but writeFileSync writes directly to the final path with process umask permissions and can leave partial JSON if interrupted. Use the existing temp-file, 0o600, fsync/close, rename pattern before making this a recovery marker.
  - confidence: 0.87
- **[P2] Snapshot process.env before mutating it:** `src/gateway/server-drain-manifest.test.ts:12-25`
  - body: ORIGINAL_ENV stores the live process.env object, so beforeEach mutates the captured value and afterEach can leak OPENCLAW_STATE_DIR or OPENCLAW_TEST_FAST into later tests. Capture a shallow copy up front or use the repo's env-stubbing helpers.
  - confidence: 0.84
- **[P3] Return write failures to the close handler:** `src/gateway/server-drain-manifest.ts:71-73`
  - body: writeDrainManifest catches write errors internally, so the caller's try/catch will not record drain-manifest-write and shutdown can report cleanly after failing to snapshot recovery state. Return a status or throw so shutdown warnings reflect the failed snapshot.
  - confidence: 0.78
