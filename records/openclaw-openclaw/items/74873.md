## Summary

Keep open. Current main does not already implement a drain manifest, so this PR is not obsolete, but the current diff still has blocking recovery-schema and persistence problems: it serializes a run-id key as `sessionId`, writes the recovery file directly with default permissions, and bakes Linear-specific metadata into gateway core.

## What This Changes

The PR adds `ChatRunRegistry.entries()`, a new gateway drain-manifest module with read/write/delete helpers, shutdown wiring and tests, and a small system-event trust hardening change in `server.impl.ts`.

## Best Possible Solution

Revise this PR so shutdown snapshots active runs into a generic recovery manifest with a real `runId` and any actual persisted session identifier kept separate, writes the file atomically with owner-only permissions, removes Linear-specific fields from gateway core, and keeps TTL/checkpoint semantics with the boot-time recovery consumer unless a generic contract is needed now.

## Reproduction Assessment

Yes. A focused source review reproduces the blockers: current main registers chat runs by run/client IDs while the PR writes that key as `sessionId`, and the PR writes the recovery file directly with `fs.writeFileSync`.

## Solution Assessment

No. The shutdown snapshot seam is reasonable, but the current schema and persistence mechanics are not the narrowest maintainable solution; a generic run metadata contract plus the existing owner-only temp-file/rename pattern is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Record registry keys as run IDs:** `src/gateway/server-drain-manifest.ts:47`
  - body: `entry.sessionId` is the chat-run registry key, but current callers add and look up this registry by `clientRunId`/agent `runId`. A boot recovery reader that treats this field as a persisted session ID can resume or route the wrong session; rename this manifest field to `runId` or persist the actual session ID separately.
  - confidence: 0.9
- **[P2] Write the manifest atomically with private permissions:** `src/gateway/server-drain-manifest.ts:67`
  - body: `draining-sessions.json` contains session routing metadata, but this writes directly to the final path with the default creation mode. If shutdown is interrupted, readers can see partial JSON, and permissive umasks may expose the file beyond the owner; use the existing temp-file `0o600` plus rename pattern or equivalent.
  - confidence: 0.88
- **[P2] Remove Linear-specific metadata from gateway state:** `src/gateway/server-drain-manifest.ts:12`
  - body: The gateway recovery manifest now includes `linearTicketId` parsed from arbitrary session keys. Gateway core recovery state should stay tracker-agnostic, and this regex can misclassify unrelated keys; keep the manifest generic and let external tooling infer work-item metadata outside the gateway contract.
  - confidence: 0.83
- **[P3] Snapshot process.env before mutating it:** `src/gateway/server-drain-manifest.test.ts:12`
  - body: `ORIGINAL_ENV` stores the live `process.env` object, so the `beforeEach` mutations also affect the captured value and `afterEach` can leak `OPENCLAW_STATE_DIR` or `OPENCLAW_TEST_FAST` into later tests. Capture `{ ...process.env }` up front or use the existing env helpers; the same pattern was also added in the close-handler test.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff introduces persistent local recovery state containing session routing metadata, but the file write is not atomic and does not force owner-only permissions.

Concerns:

- **[low] Manifest file can expose or corrupt routing metadata:** `src/gateway/server-drain-manifest.ts:67`
  - body: `draining-sessions.json` stores session keys, run IDs, and derived work metadata, but `fs.writeFileSync` writes directly to the final path with default permissions. Use an owner-only atomic write so local readers cannot observe partial JSON or metadata created under a permissive umask.
  - confidence: 0.88
