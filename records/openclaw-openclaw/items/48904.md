## Summary

Keep this PR open. Current main has a provider-profile-based thinking architecture, but it still does not implement the PR’s central behavior: allowing a custom `models.providers.*.models[*]` definition to opt into `xhigh` through `compat.supportsXHighThinking`, or an equivalent config/catalog path that exposes `xhigh` for custom OpenAI-compatible model refs. The PR discussion also includes a concrete reproduction from a real custom provider setup, so this is not low-signal cleanup.

## Best Possible Solution

Keep this PR open for maintainer review or contributor refresh. The best path is to adapt the custom-model xhigh opt-in to current main’s `resolveThinkingProfile`/`thinkingLevels` architecture, choose the public config contract, carry the relevant model compat/catalog data into thinking policy resolution, update the strict and generated config schemas plus docs, and add focused regression coverage across reply, gateway session patching, ACP/TUI presentation, agent CLI, cron, and `llm-task`.

## Risks / Open Questions

- The PR predates current main’s provider-owned `resolveThinkingProfile` and gateway `thinkingLevels` architecture, so a refresh should adapt to that model rather than reintroducing older config-aware helper threading unchanged.
- Maintainers need to decide the public API shape: keep the proposed `compat.supportsXHighThinking`, map existing `compat.supportedReasoningEfforts` into thinking profiles, or introduce a broader configured thinking-profile field.
- Any config opt-in must keep binary-provider guards, hints, validators, and runtime payload mapping aligned so a misconfigured custom model does not advertise `xhigh` where validators reject it, or pass validation where the provider cannot handle it.
