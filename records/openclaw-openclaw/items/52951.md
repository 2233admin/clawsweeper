Keep this PR open. Current main still exposes only the binary `tools.fs.workspaceOnly` filesystem policy; it does not implement `tools.fs.roots`, per-root `kind`/`access`, multi-root host guards, or root-scoped outbound media reads. The PR is an active, broad, security-sensitive implementation candidate, so cleanup closure would be wrong; it needs maintainer and security review.

## Best Possible Solution

Keep this PR open. Maintainers should decide whether `tools.fs.roots` is the desired core contract. If yes, review and land a narrowed, security-reviewed implementation that covers config/schema/docs, host read/write/edit/apply_patch enforcement, media/outbound propagation, sandbox ignore behavior, pinned I/O safety, generated baselines, and release notes. If maintainers prefer a different filesystem sandboxing model, close only with an explicit product/security decision and a canonical follow-up.
