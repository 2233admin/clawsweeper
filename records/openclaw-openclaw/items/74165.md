## Summary

Keep open. Current main still lacks the requested env-gated abort-source diagnostic, while the embedded runner still has a single `abortRun()` funnel reached by the named abort paths; the related implementation PR is closed and unmerged, so this remains a maintainer operability/product decision rather than shipped behavior.

## What This Changes

Request to add a default-off `OPENCLAW_LOG_ABORT_SOURCES=1` diagnostic that emits one structured `[abort-source]` log line with abort-source classification and caller frames from embedded run aborts.

## Best Possible Solution

If maintainers accept the surface, add a narrow helper in the embedded runner, call it once at the top of `abortRun()`, classify the existing abort sources without changing behavior, prefer the existing diagnostics flag/event model where appropriate, and document stack-frame sensitivity with focused helper tests.

## Reproduction Assessment

Not applicable as a feature request rather than a bug reproduction. The current-main verification path is high-confidence: the named abort paths exist, but the requested env gate, marker, classifier, stack capture, and tests do not.

## Solution Assessment

Unclear until maintainers accept the core diagnostic surface. Technically, a default-off helper called from `abortRun()` is the narrowest maintainable implementation, but the final design should decide whether stack frames belong in normal logs or a redacted diagnostics event/trajectory path.
