## Summary

Keep open. Current main still lacks the requested env-gated abort-source diagnostic, the centralized embedded-run abort paths are still present, and the related implementation PR is closed unmerged, so this remains a maintainer operability/product decision rather than cleanup-close material.

## What This Changes

Request an opt-in `OPENCLAW_LOG_ABORT_SOURCES` diagnostic that logs structured embedded-run abort source classification and caller stack frames from `abortRun()`.

## Best Possible Solution

If maintainers want this diagnostic, land a narrow default-off helper in the embedded runner that classifies the existing abort sources at the single `abortRun()` entry point, documents stack-frame sensitivity, and adds focused helper coverage plus nearby runner timeout/compaction checks.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. The source verification is high-confidence: current main has the described abort paths and state flags, but no requested env gate, marker, classifier, stack capture, or test file.

## Solution Assessment

Unclear until maintainers accept the logging surface. Technically, a single helper called from `abortRun()` is the narrowest maintainable implementation, but the env-var shape and stack privacy rules should be settled before automation writes it.
