## Summary

Keep open. Current main has useful Google Meet hardening, including the merged #73256 observe-mode and speech-health work, but it still does not satisfy the central request for a reliable headless/listen-first Meet participant with end-to-end audio or transcription proof.

## What This Changes

Request Google Meet plugin support for reliable headless or listen-first agent joins with explicit identity/admission state, realtime audio/transcription health checks, and clear failure guidance.

## Best Possible Solution

Keep this open as Google Meet plugin product/API work. The desired end state is a documented identity and meeting-entry model, a clear official Media API versus Chrome-node participation direction, status that separates browser/lobby/admission/provider/audio/transcript state, preflight checks that fail on missing real audio or transcript movement, and live validation proving admission plus actual Meet audio or transcription.

Unclear. The issue includes concrete live steps and health fields from OpenClaw 2026.4.24, and current main still exposes the same `googlemeet status` and `test-speech` surfaces, but a high-confidence reproduction requires a live Meet/Chrome/audio session that was not run in this read-only review.

## Solution Assessment
