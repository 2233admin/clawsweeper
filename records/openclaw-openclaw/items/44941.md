## Summary

Keep open. Current main has fallback transition tracking, lifecycle events, verbose-mode chat notices, /status fallback visibility, and Control UI indicators, but it still does not provide the requested default-on activation and recovery notices in the affected chat/session when verbose mode is off, nor a dedicated disable/tuning config.

## What This Changes

Requests default-on per-session/per-topic notifications when model fallback activates and when the primary model recovers, plus a config knob to disable the notices.

## Best Possible Solution

Define a product-owned fallback notification policy that sends one-time activation and recovery notices to the affected conversation by default, uses privacy-safe reason formatting, documents recovery and throttling semantics, and exposes a stable config switch to disable or tune the behavior.

## Reproduction Assessment

Yes. Current main has a high-confidence reproduction path in the existing e2e tests: simulate a fallback transition with verbose mode disabled and observe lifecycle events without activation or recovery notice text in the reply payload.

## Solution Assessment

Unclear. The requested default-on behavior may be the right production reliability direction, but the exact solution needs maintainer decisions on wording, redaction, throttling, recovery truth, and config shape before a narrow implementation is safe.
