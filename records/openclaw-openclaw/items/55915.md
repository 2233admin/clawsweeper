## Summary

Keep open. Current main does not implement the requested cloud relay path: mobile connectivity is still documented as LAN, tailnet, or manual host/port; remote QR setup still requires an already reachable gateway URL or Tailscale Serve/Funnel; gateway config/schema exposes `tailscale` and direct/SSH `remote` settings, not a `cloud` auth/API/relay/heartbeat block; and the CLI registry has no `login`, `cloud`, or `invite` root command. I also would not close this as ClawHub in this sweep because the request crosses first-party gateway/mobile transport, OAuth, heartbeat, relay trust, and device-approval behavior, and the same author's dependent invite-code issue #55914 is still open.

## Best Possible Solution

Keep this issue open for maintainer design review. The best path is to decide whether cloud mobile reachability is a first-party Gateway/mobile feature or should be enabled by a new explicit plugin API seam, then implement the chosen path with config/schema/docs, OAuth credential handling, heartbeat/relay lifecycle, mobile UX, and device-approval security tests. Coordinate it with open related issue #55914 so the shared `cloud` config and discovery story stay coherent.
