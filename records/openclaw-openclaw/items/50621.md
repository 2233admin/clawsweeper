## Summary

Current main still matches the reported behavior: main-session `systemEvent` cron jobs use the generic non-agent cron timeout, `wakeMode: "now"` still awaits `runHeartbeatOnce()`, and `systemEvent` payloads still have no `timeoutSeconds` escape hatch. The duplicate report was closed into this issue, while the two targeted fix PRs are closed unmerged, so this remains the canonical open tracker.

## Best Possible Solution

Keep this issue open and make the main-session `systemEvent` cron contract explicit. The likely fix belongs in `src/cron/service/timeout-policy.ts`, `src/cron/service/timer.ts`, and possibly `src/cron/types.ts` / `src/cron/normalize.ts`: for `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and `wakeMode: "now"`, either record success after enqueue-and-wake without waiting for the full heartbeat turn, or expose and document a configurable timeout for this job shape. Add regression coverage beside the timeout-policy and main wake-now tests, and update https://docs.openclaw.ai/automation/cron-jobs if any timeout remains user-visible.

## Risks / Open Questions

- Closing now would hide a current contract gap: main-session `systemEvent` jobs with `wakeMode: "now"` can still wait on heartbeat completion under the generic cron timeout and record false timeout failures for long-running work.
- Only increasing the timeout would reduce failures but would not fully satisfy the stated fire-and-forget expectation unless run-history semantics and docs are updated.
- Changing this path to enqueue-and-wake success affects run-history timing, one-shot deletion timing, and what cron can honestly report as success versus delivered/enqueued.
