## Summary

Current main has fixed the adjacent cross-run false-positive class by scoping isolated cron sessions/history to a run, but it still does not implement this issue's requested warn-to-info downgrade for bounded same-run isolated cron retries. The warning path still emits warn-level logs and structured warning events, and the hook context still lacks sessionTarget, elapsed lifetime, or schedule interval metadata needed to make the proposed decision safely.

## Best Possible Solution

Keep this issue open and land a focused core/cron fix: pass explicit bounded isolated-cron context into loop-warning emission, demote only warning-level loop diagnostics/logs proven to come from isolated cron runs whose lifetime is within their schedule interval, and preserve current critical blocking, thresholds, run-id scoping, and diagnostic audit events.

## Clownfish Work Prompt

Fix the open issue about `[agents/loop-detection]` warning noise for bounded isolated cron sessions. Current main already scopes isolated cron runs by run id/session key, but same-run repeated `exec`/tool calls still emit warn-level loop messages. Implement a focused additive change that passes explicit bounded isolated-cron context into the before_tool_call/loop-warning path and demotes only warning-level loop diagnostics/logs to info when the run is known to be an isolated cron run and its lifetime is within the job schedule interval. Preserve all critical/global circuit breaker blocking, thresholds, non-cron warnings, non-isolated cron warnings, run-id history scoping, and structured audit behavior. Do not infer broad policy from bundled ids or brittle owner-specific strings if a typed context can be passed. Update focused tests for the demotion case and for preserved warning/critical cases; update `docs/tools/loop-detection.md` if user-visible behavior is documented; add a changelog fix entry with `Thanks @cjplanted` if the change is user-facing log behavior.
