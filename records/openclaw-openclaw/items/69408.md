## Summary

Keep open. Current main still emits non-critical tool-loop detections as warn-level logs and diagnostics, and the tool hook still lacks typed isolated-cron boundedness metadata needed to demote only the intended false positives.

## What This Changes

Request to demote loop-detection warnings from warn to info only for bounded isolated cron sessions whose run lifetime is within the cron interval, while preserving warning behavior elsewhere.

## Best Possible Solution

Add explicit typed bounded isolated-cron context from cron execution into tool-loop warning emission, demote only warning-level loop logs/diagnostics for isolated cron runs whose elapsed runtime is within the schedule interval, and preserve critical blocking plus existing run-id scoping.

## Reproduction Assessment

Yes. The issue gives a concrete isolated-cron repeated-exec reproduction, and current source confirms those warning results necessarily flow through log.warn/logToolLoopAction("warning") because the hook has no bounded isolated-cron context.

## Solution Assessment

Yes, with a narrower implementation than the hardcoded fallback suggested in the report. The maintainable fix is to pass typed cron context and computed schedule bounds into the hook path, not infer from session keys or use a fixed 5-minute proxy.
