## Summary

Keep #69408 open. Current main still emits loop-detection warning-level logs without cron/session-target/lifetime context, while cron docs and runtime confirm isolated jobs are fresh per-run sessions. This is a plausible targeted logging-noise bug, not already implemented or cleanly superseded by the related stale-history PR context.

## Best Possible Solution

Keep the issue open. The best fix is to thread a small, generic run context into loop detection or the warning-emission layer, including cron trigger/sessionTarget and enough schedule/run timing metadata to decide when an isolated run is bounded by its interval. Add regression coverage for isolated cron downgrading warning-only detections to info while preserving warning/critical behavior for main, custom, long-running, and critical loop cases, and update the loop-detection/cron docs if the logging policy becomes configurable.
