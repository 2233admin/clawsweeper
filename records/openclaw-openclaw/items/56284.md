## Summary

Keep this issue open. Current main has partial restart-intent and drain machinery, but the Windows Scheduled Task restart path still force-ends the task and terminates gateway listener processes before relaunch, with tests asserting that behavior. The linked implementation candidate #56292 was closed unmerged, so there is no canonical merged fix to close against.

## Best Possible Solution

Keep the issue open and fix the Windows owner path. The likely implementation should make Windows managed restarts use the Gateway-owned draining restart path without immediately killing the listener process, then wait for health. It should also clarify whether CLI-initiated restarts need a restart sentinel/continuation, and add Windows-focused tests that prove active tasks are allowed to drain or time out before process termination.

## Evidence

- **CLI restart path delegates to service manager:** `runDaemonRestart` resolves the service/port and delegates to `runServiceRestart`; it does not call `deferGatewayRestartUntilIdle` or write a restart sentinel before the Windows service-manager restart path.
  - file: [src/cli/daemon-cli/lifecycle.ts:186](https://github.com/openclaw/openclaw/blob/f256eeba431b5f59e6085b7c11c9c0a309e408b7/src/cli/daemon-cli/lifecycle.ts#L186)
  - sha: [f256eeba431b](https://github.com/openclaw/openclaw/commit/f256eeba431b5f59e6085b7c11c9c0a309e408b7)
- **Partial restart-intent mitigation exists:** Generic service restarts write `gateway-restart-intent.json` for the running Gateway PID before calling the service manager, which lets the Gateway treat SIGTERM as a draining restart if the process gets to handle it.
  - file: [src/cli/daemon-cli/lifecycle-core.ts:504](https://github.com/openclaw/openclaw/blob/f256eeba431b5f59e6085b7c11c9c0a309e408b7/src/cli/daemon-cli/lifecycle-core.ts#L504)
  - sha: [f256eeba431b](https://github.com/openclaw/openclaw/commit/f256eeba431b5f59e6085b7c11c9c0a309e408b7)
- **Gateway run loop can drain on restart intent:** The Gateway run loop consumes restart intent on SIGTERM and the restart path waits for active tasks, embedded runs, and runtime dependency installs before closing. This is useful context, but it depends on the process being allowed to handle the signal.
  - file: [src/cli/gateway-cli/run-loop.ts:349](https://github.com/openclaw/openclaw/blob/f256eeba431b5f59e6085b7c11c9c0a309e408b7/src/cli/gateway-cli/run-loop.ts#L349)
  - sha: [f256eeba431b](https://github.com/openclaw/openclaw/commit/f256eeba431b5f59e6085b7c11c9c0a309e408b7)
- **Windows Scheduled Task restart still kills listeners:** `restartScheduledTask` calls `schtasks /End`, then terminates verified gateway listeners and installed startup runtimes before starting the task again. That still matches the core risk reported here rather than a confirmed graceful restart path.
  - file: [src/daemon/schtasks.ts:993](https://github.com/openclaw/openclaw/blob/f256eeba431b5f59e6085b7c11c9c0a309e408b7/src/daemon/schtasks.ts#L993)

- **Tests assert forced listener cleanup on Windows restart:** The Scheduled Task restart test explicitly expects lingering verified gateway listeners to be killed before restart, so current coverage does not prove the requested graceful wait/session-preservation behavior.
