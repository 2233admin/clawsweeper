## Summary

Keep open. The requested warning is still not implemented on current `main`, but this PR is stale against the current config mutation flow and includes an unrelated skills-status change that should not ship in this branch.

## What This Changes

The branch adds a model-catalog warning helper and path-detection test for selected `openclaw config set` model paths, and also changes skill status eligibility to treat parse-error entries as ineligible.

## Best Possible Solution

Implement the warning as a narrow config CLI operation-level check: inspect model-path `ConfigSetOperation` values before persistence, cover single and batch assignments, reuse existing model-ref/profile/catalog helpers, keep the warning non-blocking, and leave skills-status behavior to a separate typed change.

## Reproduction Assessment

Yes for the PR defects. Static inspection shows current `openclaw config set` goes through `runConfigSet` and `runConfigOperations`, while the PR hooks an obsolete inline writer; the `entry.parseError` access is also reproducible from the current `SkillEntry` type.

## Solution Assessment

No. The user-facing warning is a reasonable direction, but this patch is not the best implementation because it misses the current operation-based config flow and mixes in an unrelated skills eligibility change.
