## Summary

Keep this PR open. Current main still does not implement the Prompt Inspector feature: there is no Prompt Inspector client/config/env surface, hook requests still dispatch after ordinary validation, external hook content is wrapped with local regex-based warnings only, and web tool results are not scanned by an ML/API detector. The PR is also security-sensitive because it introduces third-party outbound detection calls, a new API-key env surface, and optional blocking behavior in gateway/agent hot paths, so it needs maintainer product/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The strongest path is to either harden and rebase the core feature with focused tests, docs, config/secret handling, privacy review, timeout/error diagnostics, and final security review, or split out a generic pre-dispatch/external-content inspection capability that a Prompt Inspector plugin can implement without hardcoding one vendor into core.

## Risks / Open Questions

- The PR sends potentially sensitive hook, external-content, or tool-result text to a third-party or custom PMTINSP_BASE_URL endpoint, so privacy, data retention, SSRF/proxy policy, and timeout behavior need maintainer review.
- The PR introduces a new PMTINSP_API_KEY secret/env surface; redaction, env precedence, diagnostics, and fail-open behavior need tests on the final branch.
- The PR can reject webhook messages when PMTINSP_ON_UNSAFE=block, which changes gateway hot-path behavior and should be reviewed against hook idempotency, replay, logging, and operator expectations.
- The current plugin hooks can observe or handle some message/tool flows, but current evidence does not show a generic pre-dispatch HTTP-hook inspection seam that can return the PR's intended built-in 400 response, so closing as ClawHub/plugin-only would be premature.
