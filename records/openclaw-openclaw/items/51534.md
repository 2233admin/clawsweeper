## Summary

Keep open. Current main still supports Discord native reply references via `replyToMode`, and the related merged threading work improves `replyToId` plumbing, but it does not implement the requested opt-in visible `<@originalAuthorId>` injection. The Discord config schema, public docs, outbound send paths, and reply delivery API still lack `channels.discord.autoMentionReply` or equivalent reply-author mention context.

## Best Possible Solution

Keep this issue open. The best fix is a focused Discord plugin change: add a typed/schema/docs opt-in setting such as `channels.discord.autoMentionReply`, carry or resolve the referenced guild message author safely, prepend `<@authorId>` only to the first visible outbound guild reply when absent, and add tests for reply modes, DMs, existing mentions, chunking, streaming previews/finals, media/webhook/component sends, deleted references, and bot/self-message cases. If maintainers prefer this outside the bundled plugin, first add a plugin API hook that exposes enough reply-author context safely.

## Risks / Open Questions

- Visible mention injection changes notification behavior in guilds and could ping the wrong user or spam multi-agent channels if applied outside a narrow opt-in guild-reply path.
- The current delivery path only carries a referenced message ID; resolving the original author may require safe inbound context retention or a REST lookup with careful handling of permissions, deleted messages, rate limits, and webhook/component paths.
- Chunking, streaming previews, media captions, existing explicit mentions, bot-authored messages, and DMs need focused coverage so an implementation does not double-mention, leak mentions into previews, or create bot loops.
