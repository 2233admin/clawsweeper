## Summary

Keep open. Current main still routes image-tool inputs through shared media preprocessing and still propagates optimizer failures without an original-image fallback; the missing `sharp` staging path is only a partial mitigation, and open PR #74243 is already the paired closing implementation for the remaining fallback behavior.

## What This Changes

The issue asks for valid JPEG/PNG images from channel media to reach the configured vision model instead of failing in OpenClaw's shared image preprocessing path with `Failed to optimize image`.

## Best Possible Solution

Review and land #74243 or an equivalent `src/media/web-media.ts` change that only falls back to original bytes for verified, under-cap, sendable images, preserves MIME/content type, keeps HEIC conversion, unsafe-path, pixel-limit, and over-cap failures closed, and preserves optimizer diagnostics. Keep the packaged runtime-deps scan/doctor follow-up in #74392 separate unless maintainers decide it must land first.

## Reproduction Assessment

Yes. The issue and comments provide concrete JPEG/PNG reproductions across LINE, WhatsApp, Telegram, macOS, Linux, and direct `loadWebMedia()` checks, and current source confirms the image tool still depends on shared media loading before the provider call.

## Solution Assessment

Unclear. Current main's runtime-dependency staging is a partial fix for missing `sharp`, but the best complete user-facing fix is still a guarded `loadWebMedia` fallback like #74243 after maintainer review.
