## Summary

Keep open. Current main mitigates the missing-sharp packaged runtime dependency path, but it still does not implement the conservative original-image fallback for optimizer failures; the provided discussion also shows open PR #74243 is explicitly paired as the closing implementation candidate.

## What This Changes

The issue reports that the built-in image/media pipeline fails during JPEG image optimization before the configured vision model is called.

## Best Possible Solution

Review and land #74243 or an equivalent src/media/web-media.ts change that falls back only for valid under-cap originals, preserves the original MIME/content type, keeps HEIC/pixel/over-cap guard failures closed, and retains actionable optimizer diagnostics.

## Reproduction Assessment

Yes. The issue and comments provide concrete LINE, WhatsApp, Telegram, and Discord image failure reports, and current source confirms image-tool media loading still reaches loadWebMedia before the provider call; an optimizer throw remains sufficient to reproduce the pre-provider failure path.

## Solution Assessment

Unclear until #74243 is reviewed. The current staged-runtime-dependency fix is the right root-cause mitigation for missing sharp, but it does not cover remaining optimizer backend failures; the narrower loader-layer fallback appears safer than returning originals directly from optimizeImageToJpeg.
