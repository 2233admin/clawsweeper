## Summary

Keep open. Current main still sends image-tool media through the shared `loadWebMedia` optimization path and still collapses total optimizer failure into `Failed to optimize image`. The issue is also paired with open closing PR #74243, with #74392 covering complementary runtime-dependency staging, so it should remain open until a maintainer lands or supersedes that fix path.

## What This Changes

Fix the image/media preprocessing pipeline so valid in-limit JPEG/PNG media does not fail before VLM/provider calls with `Failed to optimize image`.

## Best Possible Solution

Keep this issue open and review/land #74243 or an equivalent shared `src/media/web-media.ts` fix that preserves MIME/content type, allows safe under-cap originals when optimization itself fails, fails closed for HEIC/pixel/over-cap guard failures, and keeps #74392-style runtime-dependency staging as a complementary packaging fix.
