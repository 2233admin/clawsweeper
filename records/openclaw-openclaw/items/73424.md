## Summary

Keep open. Current main mitigates part of the missing `sharp` runtime-dependency path, but the shared media loader still aborts when image optimization throws, and this issue is paired with open closing PR #74243.

## What This Changes

The issue requests that JPEG/image inputs sent through channels reach the configured vision model instead of failing in the image preprocessing optimizer.

## Best Possible Solution

Land #74243 or an equivalent `src/media/web-media.ts` change that falls back only for valid under-cap original images, preserves original MIME/content type, and keeps HEIC conversion, decoded-pixel limits, unsafe-path, unknown-dimension, and over-cap failures closed.

## Reproduction Assessment

Yes. The issue and comments give concrete JPEG/PNG reproductions, and current source confirms the source-level path: image-tool calls `loadWebMedia`, non-GIF images are optimized, and `optimizeImageToJpeg` still throws when no candidate is produced.

## Solution Assessment

Yes. The guard-aware loader-level fallback proposed by #74243 is the narrow maintainable fix; returning original bytes directly from `optimizeImageToJpeg()` would risk MIME mismatch and weakening guard boundaries.
