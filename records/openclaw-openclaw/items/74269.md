## Summary

Keep open. Current main and v2026.4.29 contain useful WhatsApp delivery diagnostics for the typing-without-send symptom, but the code still needs a live failed-turn log to distinguish WhatsApp provider rejection from an embedded/runtime hang. The related #73496 report is relevant and more detailed, but the two reports are not confirmed to share the same root cause.

## What This Changes

The issue reports that a fresh WhatsApp setup shows composing/typing for an inbound direct message but never sends a WhatsApp reply, while Telegram works on the same setup.

## Best Possible Solution

Keep the report open until maintainers capture a failed WhatsApp turn on v2026.4.29 or current main, then fix the confirmed layer: WhatsApp/Baileys delivery if a payload reaches send without an accepted id, or shared embedded/runtime/session handling if the run never produces a deliverable payload.

## Reproduction Assessment

No. The setup steps are plausible, but a high-confidence reproduction still needs `openclaw logs --follow` around the failed WhatsApp turn and a comparison with the successful Telegram path on the same provider route.

## Solution Assessment

No. The current diagnostic logging is a useful shell check, but it is not the final fix for the user-visible no-send behavior until logs show whether delivery rejects, the provider run hangs, or session state stays stuck.
