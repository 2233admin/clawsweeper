## Summary

Keep open. Current main and the latest release still have the silent revocation close-failure behavior, while the PR head implements the intended warn-and-terminate fallback with focused unit coverage and the earlier unbound logger review concern addressed.

## What This Changes

The PR updates gateway WebSocket revocation paths to warn when `socket.close` fails, fall back to `socket.terminate`, thread revocation loggers through shared-auth call sites, add unit tests, and mark one secrets-reloader system event as explicitly trusted.

## Best Possible Solution

Keep the PR open, finish maintainer/security review and CI, then merge the narrow warn-plus-terminate hardening if the branch remains focused and green.

## Reproduction Assessment

Yes. Current main exposes the issue directly in the three bare catch blocks, and the PR's unit path reproduces it by injecting a throwing `socket.close` into shared-auth and device revocation helpers.

## Solution Assessment

Yes. The latest PR head uses the narrow maintainable fix for this path: keep best-effort revocation loops running, log failed graceful closes, and force-close with `terminate()` as a fallback.
