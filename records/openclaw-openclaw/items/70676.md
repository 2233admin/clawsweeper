## Summary

Keep PR #70676 open. Current main still has the exact silent revocation-close failure pattern the PR targets: shared-auth stale/all disconnect loops and device-removal disconnects call `socket.close()` inside bare `catch {}` blocks with no warning and no `terminate()` fallback. Existing gateway tests cover successful auth-rotation closes, but not a throwing `socket.close()` on these revocation paths. The PR is focused, recent, and the provided review concern about an unbound logger was addressed on the PR branch in `ab83788042055df96c421fa6474417202bb88561`, so it is not obsolete or superseded by current main.

## Best Possible Solution

Keep PR #70676 open for maintainer review and normal validation. The best product/code path is to land a focused gateway hardening change that replaces the three bare revocation `catch {}` blocks with warn-and-terminate fallback handling, keeps revocation loops resilient when one socket fails, and adds the PR's targeted unit coverage for throwing `socket.close()` and throwing `terminate()` edge cases.
