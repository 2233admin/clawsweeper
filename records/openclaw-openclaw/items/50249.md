## Summary

Keep open. Current main does not implement the PR’s central feature: `cli.banner.taglineMode` still only supports `random`, `default`, and `off`, there is no `taglineScriptFile` config key, and banner rendering still only calls `pickTagline`. The PR is a real implementation candidate, but it adds arbitrary JS execution during CLI startup and the review discussion raised concrete startup, suppression, and timeout concerns that need maintainer/product/security judgment before it should land or be closed.

## Best Possible Solution

Keep this PR open for maintainer review. A landable path would rebase onto the current startup policy architecture, make the security model explicit, avoid script execution when banner output is suppressed, cover root/subcommand help, JSON, TTY and non-TTY behavior, and either use a truly abortable execution boundary or narrow the feature to a safer static/custom tagline configuration. If maintainers decide arbitrary JS in CLI banner config is not acceptable, close it with that product/security rationale rather than as implemented.

## Risks / Open Questions

- The proposed feature intentionally executes arbitrary local JS from config during CLI startup with full local user privileges; that needs explicit maintainer security/product acceptance, not automatic cleanup closure.
- Startup reliability is a real concern: async timeouts do not stop synchronous blocking work or hanging top-level module evaluation unless the implementation uses an abortable boundary such as a child process or worker.
- Machine-readable and suppressed-output paths need careful tests so tagline scripts do not run or write output during JSON, piped, hidden-banner, version, or help fast paths where banner behavior is intentionally constrained.
- The PR is not implemented on current main and appears to target older startup plumbing, so closing as implemented or obsolete would discard potentially useful focused work without a canonical replacement.
