## Summary

Keep open. Current main still does not implement this PR's central feature: CLI banner tagline mode remains limited to `random`, `default`, and `off`; there is no accepted `taglineScriptFile` config key; and no script-loading/resolution path exists in the current banner implementation. The related earlier PR #41147 was closed unmerged, so this PR remains the live candidate rather than obsolete. It also needs maintainer product/security review because the proposed diff executes user-configured JavaScript during CLI startup/help rendering.

## Best Possible Solution

Keep this PR open for maintainer review. Decide whether arbitrary JS in CLI banner config is acceptable. If accepted, rebase onto the current `applyCliExecutionStartupPresentation` and root-help/startup-policy surfaces, execute scripts only when banner/help rendering genuinely needs them, use a defensible execution boundary or document a narrower safety model, update schema/generated help/public docs consistently, and add targeted tests for TTY, non-TTY, JSON, version, hidden-banner, root help, and subcommand help behavior. If maintainers do not want code execution at startup, close with that rationale and consider a safer static custom-tagline config instead.

## Risks / Open Questions

- The proposed PR intentionally executes arbitrary local JavaScript configured through OpenClaw config during CLI startup/help rendering, so maintainers should explicitly accept or reject that product/security model before merge.
- A same-event-loop `Promise.race` timeout would not reliably stop synchronous blocking work or slow top-level module evaluation in a tagline script; an abortable worker/child-process boundary or a safer non-code customization model may be needed.
- Suppressed-output paths matter: JSON, version, hidden-banner, non-TTY, root help, and subcommand help flows should not execute user scripts unexpectedly or allow script stdout/stderr to pollute machine-readable output.
