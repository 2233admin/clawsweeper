## Summary

Keep this PR open. Current main has a shared terminal `sanitizeForLog()` helper, but the channel logging helpers still interpolate `channel`, `reason`, `target`, and `String(error)` directly, so the PR's intended channel log-injection hardening is not implemented on main. The PR is still useful, but it should be revised before merge to reuse the existing sanitizer, cover `channel`, and avoid runtime regressions for loosely typed plugin callers.

## Best Possible Solution

Keep the PR open and ask for a narrow revision: use the existing `sanitizeForLog()` from `src/terminal/ansi.ts`, coerce all interpolated log values with `String(...)` before sanitizing, sanitize `channel` as well as `reason`/`target`/`error`, and add focused tests for channel, DEL/ANSI coverage, and non-string runtime inputs. After that, review/merge the PR rather than closing it as implemented.

## Risks / Open Questions

- Merging the PR as-is would leave the `channel` field as a remaining log-forging surface.
- Merging the PR as-is would introduce a second `sanitizeForLog()` with behavior that differs from `src/terminal/ansi.ts`.
- The PR's local sanitizer accepts only `string`, so loosely typed JavaScript plugin callers passing numeric or object `target`/`reason` values could hit a new `value.replace is not a function` runtime error.
