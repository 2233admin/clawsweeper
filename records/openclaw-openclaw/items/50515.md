## Summary

Keep this PR open. Current main still has unsanitized shared channel logging helpers exposed through public plugin SDK subpaths, so the PR is not obsolete or implemented on main. The patch is a useful security hardening candidate, but it needs revision because it leaves `channel` unsanitized, duplicates the existing stronger log sanitizer, and can regress loose JavaScript plugin callers by calling `.replace` on non-string values.

## Best Possible Solution

Keep this PR open and request a narrow revision: reuse `sanitizeForLog` from `src/terminal/ansi.ts`, coerce every dynamic value with `String(...)` before sanitizing, sanitize every interpolated field including `channel`, `reason`, `target`, `action`, and `error`, and add focused channel logging tests for channel injection, DEL/C1/ANSI stripping, and non-string runtime inputs.

## Risks / Open Questions

- Merging as-is would leave `params.channel` as a shared-helper log-forging surface.
- Merging as-is would introduce a second, weaker sanitizer instead of reusing the existing ANSI/C0/C1/DEL log sanitizer.
- Merging as-is could throw for JavaScript plugin callers that currently rely on template-literal stringification for non-string `target` or `reason` values.
