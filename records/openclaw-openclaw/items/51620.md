## Summary

Keep this issue open. Current main still keeps followup queue items in process memory, does not persist or replay pending inbound/followup work on restart, and still rejects drain-window enqueues with `GatewayDrainingError`. The provided discussion also shows open implementation PRs with closing syntax, so the issue should remain open until those candidates are reviewed, merged, or closed.

## Best Possible Solution

Keep this issue open. Maintainers should review and either land or close one of the phase-1 PRs, then decide whether drain-window arrivals and provider downtime catch-up remain in this issue or are split to #55792. The likely implementation path is a typed durable pending-inbound/followup handoff in gateway/auto-reply, with provider-specific ACK or history-catch-up hooks where channel contracts support them.
