## Summary

Keep this issue open. Current main still lacks durable persistence/replay for followup queue items and drain-window inbound work, and the issue is paired with open implementation PRs using closing syntax. Main has in-process restart hardening, but not the requested disk-backed pending-message handoff.

## Best Possible Solution

Keep this issue open. Maintainers should review and either land or close one of the phase-1 PRs (#51623 or #51803), then decide whether drain-window arrivals and provider downtime catch-up remain in this issue or move to #55792. The durable path should be a typed pending followup/inbound handoff with TTL, dedupe/idempotency, routing metadata, and channel-specific ACK or catch-up hooks where supported.
