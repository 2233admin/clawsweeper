## Summary

Current main still does not solve the reported browser-profile config.patch restart path. The direct config.patch restart is no longer unconditional, but browser profile changes still match the browser plugin’s broad restart rule, so hot-mode config.patch can still schedule SIGUSR1 and hybrid mode can still queue a gateway restart.

## Best Possible Solution

Keep this issue open as the canonical tracker. Implement granular browser reload registration so safe profile-scoped changes such as `browser.profiles.<name>.cdpUrl` and likely `browser.defaultProfile` hot-reload or no-op, while preserving restart/reconcile behavior for service-affecting browser settings. Add reload-plan and config.patch regression coverage for hot and hybrid modes, then update the gateway/browser docs if the shipped contract changes.

## Risks / Open Questions

- A fix should stay granular. Making all `browser.*` hot/no-op could incorrectly hot-apply service-affecting or security-sensitive browser settings such as executable paths, SSRF policy, launch mode, or managed-browser invariants.
- Docs currently say `browser` is in the no-restart tools/media category, while current reload registration still treats the browser plugin config tree as restart-required; docs should be aligned with the implementation once fixed.
