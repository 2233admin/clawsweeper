## Summary

Current main still preserves the reported restart path for browser profile config writes. The direct `config.patch` SIGUSR1 path is now reload-plan-driven rather than unconditional, but `browser.profiles.*` still matches the browser plugin's broad `browser` restart metadata, so the core user problem remains valid.

## What This Changes

The issue asks for `config.patch` writes to hot-reloadable browser profile paths such as `browser.profiles.<name>.cdpUrl` to avoid scheduling a SIGUSR1 gateway restart.

## Best Possible Solution

Make browser profile/default-profile reloadability explicit through plugin-owned granular reload metadata or a specificity-aware reload planner, while keeping structural browser settings restart-required. Add regression coverage for `browser.profiles.*` through both the `config.patch` direct-restart path and watcher reload planning.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a `config.patch` that changes `browser.profiles.<name>.cdpUrl` with `gateway.reload.mode` set to `hot`: current main classifies the path under the broad `browser` restart prefix and schedules direct SIGUSR1.

## Solution Assessment

Yes, the narrow fix direction is to refine reload classification rather than remove browser restarts globally. The safer solution is profile-specific no-op/hot handling plus tests, not a hard-coded core exception that bypasses plugin ownership.
