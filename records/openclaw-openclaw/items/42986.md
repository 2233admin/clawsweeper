## Summary

Keep open. Current main still does not wire Telegram native subagent lifecycle hooks, while core requires a registered `subagent_spawning` hook before `sessions_spawn({ mode: "session", thread: true })` can bind a thread. The current Telegram docs/code have adjacent topic and ACP binding support, but native subagent thread binding remains unshipped; PR #42707 is still open and unmerged as the focused implementation candidate.

## Best Possible Solution

Keep this issue open and review/land PR #42707 or an equivalent focused implementation. The change should stay in the Telegram plugin boundary: add `registerFull` hook wiring, implement `subagent_spawning`, `subagent_delivery_target`, and `subagent_ended`, bind native session-mode subagent spawns through Telegram topic/current-conversation state, honor Telegram thread-binding feature flags, clean up bindings on failure/end, resolve completion delivery targets, and update native subagent docs/tests so ACP topic binding and native subagent thread binding stay distinct.

## Risks / Open Questions

- Closing now would conflate Telegram ACP/current-conversation topic binding with native `sessions_spawn` subagent session thread binding; the current code paths remain separate.
- The eventual fix should get routing and security review because it will create or mutate Telegram topic/session bindings, delivery targets, lifecycle cleanup, and completion routing.
- The existing Telegram thread-binding manager should likely be reused to avoid parallel binding state, but it still needs plugin hook wiring for native subagent session spawns.
