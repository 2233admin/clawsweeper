## Summary

Keep open. Current main handles DeepSeek V4 disabled thinking only when OpenClaw resolves the session thinking level to `off`/`none`; the reported `extra_body.thinking.type=disabled` plus boolean `params.thinking=false` path is not treated as disabled before the DeepSeek wrapper patches replay history. Because `extra_body` can overwrite the final payload after the provider wrapper runs, the reported 400 remains a plausible current-main bug.

## What This Changes

DeepSeek V4 Pro conversations configured with disabled thinking should not replay prior `reasoning_content` fields and then fail with the provider's 400 replay-contract error on later turns.

## Best Possible Solution

DeepSeek V4 payload handling should derive one effective disabled-thinking decision before replay repair runs, covering OpenClaw `off`/`none`, DeepSeek `extra_body.thinking.type=disabled`, and the reported boolean config shape, then strip `reasoning_content` from the outgoing history whenever the final DeepSeek payload is disabled while preserving enabled-thinking backfill behavior.

## Repair Work Prompt

Fix the DeepSeek V4 disabled-thinking replay bug from this issue. Current main strips `reasoning_content` only when the resolved OpenClaw `thinkingLevel` is `off`/`none`, but a user can disable DeepSeek thinking via `agents.defaults.models["deepseek/deepseek-v4-pro"].params.extra_body.thinking.type = "disabled"` and currently also reports `params.thinking = false`. In that path the DeepSeek wrapper can add/backfill `reasoning_content`, then the later `extra_body` wrapper overwrites `thinking` to disabled, producing the provider 400 on replay.

Expected boundary: keep this in the DeepSeek/provider-stream payload path, not core startup migration. Teach the DeepSeek V4 wrapper or its caller to compute disabled mode from the effective DeepSeek controls before replay repair runs. Cover at least: OpenClaw `thinkingLevel` `off`/`none`; `extra_body`/`extraBody` with `{ thinking: { type: "disabled" } }`; and the reported boolean `params.thinking: false` if that is accepted as an extra param for this provider. Preserve enabled-thinking behavior that backfills `reasoning_content` on all assistant messages. Be careful with shared helper callers (`extensions/opencode-go`, `extensions/venice`) and do not send native DeepSeek `thinking` controls to Venice.

Add focused tests in `extensions/deepseek/index.test.ts` and, if the shared helper signature changes, `src/plugin-sdk/provider-stream-shared.test.ts`. Add or adjust an extra-params ordering test if needed so the final payload cannot contain disabled `thinking` plus replayed `reasoning_content`. Add a one-line changelog entry under the active release section if code changes are user-facing, crediting `Thanks @simmssun-hashh`. Do not broaden unrelated provider thinking semantics.
