## Summary

Keep open. Current main still leaves the reported disabled-thinking DeepSeek path unresolved: boolean `params.thinking: false` is not normalized to `off`, and `extra_body.thinking.type=disabled` is merged after the DeepSeek wrapper can add enabled-thinking fields and `reasoning_content`; two open unmerged PRs already target those exact gaps with closing syntax.

## What This Changes

Handle DeepSeek v4-pro multi-turn disabled-thinking payloads so intermittently returned `reasoning_content` does not poison replay history and trigger a 400 on the next request.

## Best Possible Solution

Land one narrow fix that derives a single effective disabled-thinking decision before the final DeepSeek V4 OpenAI-compatible payload is sent, covering OpenClaw `off`, boolean `false`, accepted disabled aliases, and `extra_body.thinking.type=disabled`, with a regression test proving the final payload cannot contain conflicting enabled-thinking fields or replayed `reasoning_content` in the disabled path.

## Reproduction Assessment

Yes. The source-level path is clear: configure DeepSeek v4-pro with `params.thinking: false` and `extra_body.thinking.type=disabled`, then inspect the composed OpenAI-compatible payload after provider wrapping and post-plugin extra_body merging.

## Solution Assessment

Unclear which open PR is the best final shape alone. #74418 addresses alias normalization, #74403 addresses the post-`extra_body` collision, and the safest maintainer path is to review or combine them into one consistent payload-normalization fix.
