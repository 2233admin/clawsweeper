## Summary

Keep open. Current main only partially covers the DeepSeek V4 replay family: enabled-thinking plain assistant replay is backfilled, but the reported disabled-thinking path remains plausible because `params.thinking: false` and `"disabled"` are not resolved to `off`, `extra_body.thinking` is still applied after provider wrapping, and two open unmerged PRs still reference this issue with closing syntax.

## What This Changes

The issue asks OpenClaw to prevent DeepSeek v4-pro multi-turn failures when thinking is disabled and prior assistant history contains `reasoning_content`.

## Best Possible Solution

Review #74403 and #74418 together and land one narrow canonical fix that derives the effective DeepSeek thinking mode after per-model params and `extra_body`, accepts common disabled aliases, and normalizes replayed `reasoning_content` according to the final enabled or disabled request contract.

## Reproduction Assessment

Yes for a focused payload-level reproduction: configure DeepSeek V4 with `params.thinking: false` plus `extra_body.thinking.type=disabled`, replay assistant history with `reasoning_content`, and inspect the final OpenAI-compatible payload. Live reproduction is less deterministic because it depends on DeepSeek intermittently returning that field while thinking is disabled.

## Solution Assessment

No, current main is only a partial solution for the broader DeepSeek V4 replay problem. The safest path is a single reviewed fix that handles both thinking-default resolution and post-wrapper `extra_body` collisions, rather than closing this issue while the linked implementation PRs are still open.
