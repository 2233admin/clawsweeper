## Summary

Keep open. Current main still leaves the reported disabled-thinking path unresolved: boolean `params.thinking: false` does not resolve to `off`, the DeepSeek wrapper can still inject enabled-thinking replay fields, and `extra_body` is merged afterward without a final DeepSeek disabled-thinking cleanup. The issue also has two open implementation PRs with closing references, so it should stay open until the fix lands or those PRs are rejected.

## What This Changes

The issue requests that DeepSeek V4 disabled-thinking conversations avoid replaying or injecting `reasoning_content` in a way that makes the next multi-turn request fail with a 400 error.

## Best Possible Solution

Land one narrow fix that computes disabled thinking consistently before provider wrapping and also guarantees the final DeepSeek V4 OpenAI-compatible payload cannot combine `thinking.type=disabled` with replayed `reasoning_content`. The fix should include regression coverage for boolean `false`, string disabled aliases, and `extra_body.thinking.type=disabled` on the final payload.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: configure DeepSeek V4 with per-model `params.thinking: false` and `extra_body.thinking.type=disabled`, then the current wrapper order can produce an enabled-thinking replay mutation before `extra_body` overwrites the top-level thinking field.

## Solution Assessment

Unclear whether either open PR alone is the best complete fix. The safer maintainable solution is a single normalization path plus a final payload guard for `extra_body`, rather than relying only on either alias handling or post-merge stripping.
