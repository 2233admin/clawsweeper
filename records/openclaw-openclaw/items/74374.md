## Summary

Keep open. Current main and v2026.4.29 contain an adjacent DeepSeek V4 plain-assistant replay fix, but the reported disabled-thinking path is still not consistently normalized after `params.thinking: false` and `extra_body.thinking.type=disabled`; two open, unmerged PRs also reference this issue with closing syntax, so it should remain open for maintainer review of those fixes.

## What This Changes

The issue requests DeepSeek V4 payload normalization so disabled-thinking multi-turn conversations do not replay conflicting `reasoning_content` and fail with a 400 provider error.

## Best Possible Solution

Land one narrow, tested normalization path that treats boolean `false` and disabled aliases as `off`, reconciles `extra_body.thinking.type=disabled` after provider wrapping, removes conflicting enabled-thinking fields, and locks the final DeepSeek disabled replay semantics with regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure DeepSeek V4 with per-model `params.thinking: false` and `extra_body.thinking.type=disabled`; current main resolves the model to the provider default, lets the DeepSeek wrapper inject enabled-thinking replay fields, and later overwrites `thinking` to disabled without a final cleanup pass.

## Solution Assessment

Unclear which open PR is the best final shape alone. The maintainable fix should combine alias resolution with final-payload normalization and should settle the disabled assistant `reasoning_content` omit-versus-empty behavior with focused tests or live proof.
