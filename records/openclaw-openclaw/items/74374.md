## Summary

Keep open. Current main contains the plain-assistant DeepSeek V4 backfill fix, but it still does not cover the reporter's exact disabled-thinking configuration: `params.thinking: false` is not resolved to `off`, and `extra_body.thinking.type=disabled` is applied after the DeepSeek wrapper can set enabled thinking and backfill `reasoning_content`. Two open PRs already reference this issue with closing syntax, so the issue should stay open for maintainer review of those fixes.

## What This Changes

The issue reports DeepSeek v4-pro multi-turn 400 failures when disabled thinking is configured through `extra_body` and `params.thinking: false`, leaving the outgoing replay payload inconsistent with DeepSeek's `reasoning_content` contract.

## Best Possible Solution

Land one narrow fix that derives a single effective disabled-thinking decision before the final DeepSeek V4 OpenAI-compatible payload is sent, including OpenClaw `off`, accepted disabled aliases, boolean `false`, and `extra_body.thinking.type=disabled`; then add a regression test proving the final payload cannot contain conflicting enabled-thinking fields or replayed `reasoning_content` in the disabled path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists for the reported config: per-model `params.thinking: false` is ignored, the DeepSeek wrapper can build an enabled-thinking payload, and the later `extra_body` merge can overwrite `thinking` to disabled without re-normalizing the replayed messages.

## Solution Assessment

Unclear which open PR is the best final shape by itself. PR #74418 addresses the boolean/alias resolution gap and PR #74403 addresses the post-`extra_body` collision path; maintainers should choose or combine the narrowest fix that makes the final DeepSeek payload internally consistent.
