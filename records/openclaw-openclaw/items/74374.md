## Summary

Keep open. Current main still does not cover the reported disabled-thinking composition: `params.thinking: false` and `"disabled"` do not resolve to `off`, the DeepSeek V4 wrapper can still prepare an enabled-thinking payload, and `extra_body.thinking.type=disabled` is merged afterward without a final DeepSeek cleanup. The issue also has open closing PRs #74403 and #74418, so it should stay open until maintainers land or reject a fix.

## What This Changes

The issue requests a fix for DeepSeek v4-pro multi-turn requests that fail when disabled thinking via `extra_body` conflicts with replayed `reasoning_content`.

## Best Possible Solution

Land one cohesive DeepSeek V4 fix after reviewing #74403 and #74418 together: normalize common disabled aliases to `off`, guard the final OpenAI-compatible payload after `extra_body` overrides, and add a regression test for the reporter's `params.thinking: false` plus `extra_body.thinking.type=disabled` configuration.

## Reproduction Assessment

Yes. The high-confidence reproduction path is static and matches the report: `params.thinking: false` is ignored by `resolveThinkingDefault`, DeepSeek V4 defaults to enabled thinking, and `extra_body` later overwrites only the top-level `thinking` field.

## Solution Assessment

No, current main is not the best solution because it fixes plain-assistant replay for enabled thinking but not the reported disabled-thinking composition. The maintainable fix is a single normalized disabled-thinking path plus a final payload guard for `extra_body` collisions.
