## Summary

Keep open. Current main still routes Feishu startup, setup/status, and channel status through the SDK-backed `openclaw_bot/ping` probe path described by the report, and the related timeout work is still an open coordination problem rather than a merged fix.

## What This Changes

The issue reports Feishu websocket startup/status failures where `@larksuiteoapi/node-sdk`/axios calls to `open.feishu.cn` time out from the gateway process on Alibaba Cloud ECS with Node.js 24 while native HTTPS succeeds.

## Best Possible Solution

Keep the fix in the Feishu plugin: choose one maintainable transport and timeout policy for startup bot identity, setup/status probes, channel status, and WS connect-token requests; preserve proxy behavior and group mention safety while bot identity is unknown; then add focused unit coverage plus a redacted live verification path for the SDK-vs-direct-request failure mode.

## Reproduction Assessment

Unclear. The issue and duplicate report provide concrete field steps and logs, but a high-confidence repo-local reproduction still needs Feishu credentials and the reported Alibaba Cloud ECS / Node.js 24 behavior or a deterministic fixture that reproduces the SDK transport hang.

## Solution Assessment

Unclear. There is no patch on this issue, and the open startup-latency PR is relevant but not proven to resolve the underlying SDK/axios timeout; the narrowest maintainable solution should be selected by Feishu maintainers after transport verification.
