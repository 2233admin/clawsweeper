## Summary

Keep open. Current main still routes Feishu startup and status through SDK-backed probe paths that match the reported timeout surface, and the overlapping Feishu timeout/latency work is still open rather than shipped as a complete fix.

## What This Changes

The issue reports Feishu websocket startup timing out on the SDK-backed Axios ping to open.feishu.cn from the gateway process on Alibaba Cloud ECS / Node.js 24, with webhook mode as a workaround.

## Best Possible Solution

Keep the fix inside the Feishu plugin: define one consistent transport and timeout policy for startup identity probing, setup/status probing, and SDK HTTP calls, while preserving webhook mode as a documented fallback for deployments where persistent connection transport is unhealthy.

## Reproduction Assessment

Unclear. The issue has concrete field steps and a second matching report, and current main still contains the matching probe/startup paths, but reproducing the exact failure requires Feishu credentials plus the reported Node 24 / Alibaba Cloud ECS behavior or a deterministic SDK transport fixture.

## Solution Assessment

Unclear. There is no patch on this issue; the safest path is maintainer review of #67466 plus a plugin-local transport/probe policy, because a startup cap alone may not fix the SDK Axios timeout from the gateway process.
