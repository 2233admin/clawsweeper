## Summary

Keep open. Current main has adjacent compaction-wait and transcript-size hardening, but I did not find a complete fix for the reported duplicate outbound sends: compaction retries can still replay the prompt after successful message-tool side effects, and the normal message action path does not carry a stable per-turn idempotency guard across that retry.

## What This Changes

The issue asks OpenClaw to recover cleanly from compaction timeouts and prevent compaction retries from repeatedly sending the same outbound channel message.

## Best Possible Solution

Add a retry-stable outbound side-effect guard for message-tool sends so compaction and timeout recovery can retry model work without re-delivering already-successful channel messages, while preserving the existing timeout cleanup and transcript compaction hardening.

## Reproduction Assessment

Yes, at source level for the remaining duplicate-send bug: a successful message-tool send followed by `compaction_end` with `willRetry` clears the sent-message tracking and the retry can execute the same send again. The issue discussion provides real Telegram and Feishu logs for the runtime symptom.

## Solution Assessment

No. Current main’s compaction wait cleanup and timeout recovery are useful partial fixes, but the narrow maintainable solution still needs idempotency or a side-effect ledger that survives compaction retries rather than only suppressing duplicate final assistant payloads.

## Repair Work Prompt

Fix openclaw/openclaw#43661: compaction and timeout recovery retries must not re-send message-tool deliveries that already succeeded earlier in the same logical agent turn. Add a retry-stable side-effect/idempotency guard for message action sends that survives `resetForCompactionRetry` and timeout-triggered compaction retries. Prefer a scoped per-run/per-turn ledger or stable idempotency key derived from run/session/tool action identity plus target/content/media, not a broad global content dedupe. Cover direct core sends, gateway sends, transcript mirroring, and plugin-handled message actions where the current surfaces support it. Do not remove the existing compaction grace, aggregate wait timeout, unsubscribe abort, or successor-transcript duplicate-user-message behavior. Do not block intentional repeated sends in separate turns or to different targets. Add focused Vitest coverage for: successful message-tool send before compaction retry is not delivered twice; failed message-tool send can still retry; timeout-triggered compaction does not retry side-effectful message sends without idempotency; media sends are covered. Add a user-facing changelog fix entry in the active version, crediting @thedanchez and relevant field reporters if the final patch addresses their repros.
