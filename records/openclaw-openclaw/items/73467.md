## Summary

Keep open. Current main has meaningful mitigations for the reported symptoms, including readiness event-loop diagnostics, bounded restart deferral, task-inspection reconciliation, and local log-tail fallback, but the central CPU-hot Gateway event-loop starvation is not proven fixed and the later hot-path tracker #74860 remains open with more specific profile evidence.

## What This Changes

The issue requests Gateway control-plane responsiveness under multi-agent orchestration load, including responsive `/readyz`, resilient operator logs/control traffic, and bounded reload/restart blockers when active run accounting goes stale.

## Best Possible Solution

Keep this as the umbrella Gateway reliability tracker until maintainers either land and validate the narrower #74860 hot-path fix under live orchestration load or split the remaining reload/task-accounting and operator-observability gaps into dedicated canonical issues.

## Reproduction Assessment

Unclear for current main. The original report has strong live operational evidence, but a high-confidence reproduction requires a loaded multi-agent Gateway with Discord/WebUI traffic, memory-style prompt hooks, frequent `/readyz` probes, and CPU/event-loop profiling rather than a simple local unit test.

## Solution Assessment

Not applicable as a patch review. The current mitigations are useful but partial; the maintainable path is to reduce/offload the synchronous hot path, then validate reload/task accounting and control-plane responsiveness under representative load before closing the umbrella.
