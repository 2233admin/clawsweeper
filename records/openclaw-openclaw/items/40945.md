## Summary

Keep open. Current main still implements and documents the opposite behavior as a Control UI remote-fetch hardening policy, and an open PR still references this issue as its target. The remaining decision is not whether the report is reproducible, but whether maintainers want to change the browser remote-media policy.

## What This Changes

The issue requests that Control UI chat markdown render safe remote http/https image URLs inline instead of flattening them to alt text.

## Best Possible Solution

Make an explicit Control UI remote-media policy decision. If remote images are allowed, implement it across the markdown renderer, sanitizer expectations, CSP, docs, tests, and a clear safety model such as opt-in, allowlist, or managed proxy; otherwise keep the current hardening and close the request as a policy choice.

## Security Review

Status: needs_attention

Summary: The requested behavior is security-sensitive because it would let rendered chat content initiate browser requests to arbitrary remote image hosts.

Concerns:

- **[medium] Remote images would trigger browser fetches:** `docs/web/control-ui.md:348`
  - body: Allowing http(s) markdown image URLs inline would let model- or user-supplied chat content cause the operator browser to request third-party hosts, conflicting with the documented Control UI no-remote-fetch policy unless maintainers add an explicit policy control.
  - confidence: 0.93
- **[medium] Current CSP intentionally excludes remote image sources:** `src/gateway/control-ui-csp.ts:47`
  - body: The Control UI CSP currently omits https from img-src. A renderer change alone would either fail at runtime or pressure maintainers to broaden CSP without a reviewed remote-media safety model.
  - confidence: 0.9
