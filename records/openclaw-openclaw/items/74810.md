## Summary

Keep open. Current main still has the reported failure shape: raw model runs and `llm-task` intentionally disable tool creation, but inherited explicit allowlist sources are still evaluated by the empty-callable-tools guard before provider submission. Related open PRs target the same root cause, but no fix has landed on `main`.

## What This Changes

The issue asks for LLM-only `modelRun` and embedded `llm-task` runs to ignore inherited non-empty tool allowlists when tools are intentionally disabled.

## Best Possible Solution

Land a focused fix that treats explicit no-tool call shapes as valid no-tool model calls even when inherited allowlist sources exist, while preserving the empty-allowlist failure for tool-enabled agent runs. The fix should cover generic gateway `modelRun`, `promptMode: "none"`, and plugin-owned `llm-task`, with regression tests around global `tools.profile`/`tools.alsoAllow` inheritance.

## Reproduction Assessment

Yes. The source path gives a high-confidence reproduction: `tools.alsoAllow` produces explicit allowlist sources, `modelRun`/`llm-task` disable and clear tools, and the guard still errors when sources are present and callable tools are empty; the second issue comment also confirms the same failure through `/tools/invoke`.

## Solution Assessment

No landed solution exists yet. The issue’s proposed direction is sound, but the best maintainable fix should be chosen by reviewing the overlapping open PRs and ensuring the final contract covers both modelRun and llm-task without relaxing tool-policy failures outside intentional no-tool runs.
