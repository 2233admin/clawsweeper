## Summary

Keep open. Current main still matches the reported failure shape: raw model runs and `llm-task` intentionally disable tools, but inherited explicit allowlist sources are still collected and the empty-callable-tools guard still turns that into a pre-provider error. Open PRs #73031 and #74139 cover overlapping parts of the fix, but neither is merged into current main.

## What This Changes

The issue requests that intentionally LLM-only `modelRun` and `llm-task` executions ignore inherited tool allowlists instead of failing the empty callable-tools guard.

## Best Possible Solution

Land one focused fix that treats intentional no-tool provider calls as no-tool runs even when inherited allowlist sources exist, with regression coverage for both gateway `modelRun` and `llm-task` under global `tools.profile` or `tools.alsoAllow`; include the user-facing changelog entry when the fix lands.

## Reproduction Assessment

Yes. The issue includes concrete CLI and `/tools/invoke` repros, and current-main source inspection shows the same path: tools are disabled, inherited allowlist sources remain, and the pre-provider guard still errors on an empty callable set.

## Solution Assessment

Yes, the requested direction is the right maintainable fix if scoped to intentional LLM-only runs. The safer implementation should skip or neutralize inherited allowlist checks for those no-tool calls without weakening validation for ordinary tool-enabled runs.
