## Summary

Keep open. Current main still has the reported failure shape: raw/disabled model runs build an empty callable tool set, but the embedded runner still collects inherited explicit allowlist sources and the guard still turns that into the pre-provider “No callable tools remain” error. The related llm-task/Lobster report remains open, and the targeted closing PR was closed unmerged.

## What This Changes

The issue requests that LLM-only gateway `modelRun` and `disableTools` runs ignore inherited tool allowlists instead of failing before the provider when no tools are intentionally exposed.

## Best Possible Solution

Treat explicit no-tool runs as no-tool model calls at the shared embedded-runner/tool-allowlist boundary, with focused regression coverage for both gateway `modelRun` and llm-task under global `tools.profile`/`tools.alsoAllow`, while leaving the empty-allowlist guard strict for tool-enabled runs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: global `tools.alsoAllow` becomes an explicit allowlist source, `modelRun`/`promptMode: "none"` disables tool creation, and the current guard errors before prompt submission when the callable tool list is empty.

## Solution Assessment

Yes, the narrowest maintainable fix is to skip or neutralize the empty explicit allowlist guard only for explicit no-tool runs. Clearing allowlists only at individual call sites would be more fragile unless paired with the shared guard behavior.
