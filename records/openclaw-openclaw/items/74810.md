## Summary

Keep open. Current main still matches the reported failure shape: gateway modelRun/raw model probes disable tool construction, but the embedded attempt still collects inherited explicit allowlist sources and the guard still turns an empty callable set into a pre-provider error when `disableTools` is true.

## What This Changes

LLM-only gateway model runs should ignore inherited tool allowlists instead of failing before provider dispatch.

## Best Possible Solution

Treat explicit no-tool runs (`modelRun`, `promptMode: "none"`, and `disableTools`) as valid no-tool model calls at the embedded-runner precheck boundary, while keeping the empty explicit allowlist guard strict for ordinary tool-enabled attempts. Add focused regression coverage for both gateway modelRun and llm-task with global `tools.profile`/`tools.alsoAllow`.

## Reproduction Assessment

Yes. The deterministic source path reproduces the issue: `tools.alsoAllow` becomes an explicit allowlist, modelRun disables tool creation, the embedded attempt still passes inherited allowlist sources to the guard, and the guard returns a precheck error before provider submission.

## Solution Assessment

Yes, the narrow maintainable fix is to bypass or neutralize the empty-allowlist guard only for explicit no-tool runs. A broad removal of the guard would be unsafe because it would weaken intentional fail-closed tool-policy behavior for normal runs.
