## Summary

Keep open. Current main still matches the reported failure shape: raw model runs intentionally disable tool creation, but inherited explicit allowlist sources are still collected and the empty-allowlist guard still turns that into a pre-provider error. The linked fix PRs #74834 and #74841 were closed unmerged, and related #74019 covers the plugin-owned llm-task/Lobster side rather than replacing this generic modelRun report.

## What This Changes

The issue asks for LLM-only `modelRun`/`disableTools` runs to proceed as no-tool provider calls even when global tool allowlists such as `tools.alsoAllow` are configured.

## Best Possible Solution

Update the embedded runner/tool allowlist guard so intentional no-tool runs (`modelRun`, `promptMode: "none"`, or `disableTools`) bypass inherited explicit allowlist empty-callable failures, while regular tool-enabled runs still fail before provider submission when their requested tools cannot be registered. Add focused regression coverage for both gateway modelRun and llm-task/global `tools.alsoAllow`.

## Reproduction Assessment

Yes. The source path is high-confidence: `modelRun` sets `disableTools`, the embedded attempt builds zero tools, still collects inherited allowlist sources, and the guard/test still expect the `No callable tools remain` error when tools are disabled.

## Solution Assessment

Yes. A narrow guard/embedded-runner change with regression tests is the best route; a global removal of the empty allowlist guard would be unsafe because it would weaken missing-tool detection for normal agent runs.
