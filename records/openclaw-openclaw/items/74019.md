## Summary

Keep open. Current main still matches the reported failure path: `llm-task` starts a JSON-only embedded run with tools disabled while inherited config allowlists are still collected and sent to the empty-tool guard. The issue is also paired with open, unmerged PR #74139, so cleanup should wait for that fix or an equivalent one to land.

## What This Changes

The report asks OpenClaw to let Lobster-invoked `llm-task` JSON-only embedded runs proceed when tools are intentionally disabled even if inherited `tools.alsoAllow` config names `lobster` and `llm-task`.

## Best Possible Solution

Land a narrow guard or embedded-runner contract that treats intentional no-tool runs as valid when only inherited config allowlists are present, while preserving runtime `toolsAllow` fail-closed behavior. Cover the Lobster/`llm-task` repro and the generic model-run shape, and add the appropriate changelog credit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure `tools.alsoAllow`, invoke Lobster -> `llm-task`, and current main passes config plus `disableTools: true`, empties callable tools, and still triggers the empty explicit-allowlist guard.

## Solution Assessment

Unclear until the open PR is reviewed. Ignoring inherited config allowlists for intentional no-tool runs is the right direction, but a broad `disableTools` short-circuit must not bypass runtime `toolsAllow` safety checks.
