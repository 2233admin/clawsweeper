## Summary

Keep open. Current main still matches the reported failure path: `llm-task` starts a no-tools embedded run while passing inherited config, and the empty explicit allowlist guard still turns config-derived allowlist sources into the reported error. An open implementation PR already references this issue for closure, so cleanup should wait for that PR to land, be narrowed, or be replaced.

## What This Changes

The issue asks OpenClaw to let Lobster workflows call the optional `llm-task` JSON-only tool even when inherited `tools.alsoAllow` entries are configured.

## Best Possible Solution

Land a narrow shared embedded-runner fix that ignores inherited config-derived allowlist sources when tools are intentionally disabled for JSON-only or model-only execution, while preserving fail-closed behavior for explicit runtime `toolsAllow`. Add focused regression coverage for `llm-task` and the generic model-run path plus a changelog fix crediting @technicallysecure.

## Reproduction Assessment

Yes. The issue gives a concrete config, Lobster step, and `/tools/invoke` request, and current main statically confirms the same path without needing a live model call.

## Solution Assessment

No for the broad linked-PR shape as currently shown: skipping the guard for all `disableTools` runs would also bypass runtime `toolsAllow` failures. The safer solution is to filter or ignore only inherited config allowlist sources for intentionally no-tool runs.
