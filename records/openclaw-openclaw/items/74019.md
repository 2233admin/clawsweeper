## Summary

Keep open. Current main still follows the reported failing path: `llm-task` starts an embedded run with tools intentionally disabled, the runner builds an empty callable tool set, inherited allowlist sources are still collected, and the guard still errors when no callable tools remain. An open PR already targets this issue with closing syntax, so the issue should stay open until that PR or an equivalent narrowed fix lands.

## What This Changes

Lobster workflows invoking `llm-task` through `/tools/invoke` should not fail when inherited `tools.alsoAllow` entries exist and `llm-task` intentionally disables tools for its JSON-only model call.

## Best Possible Solution

Land #74139 or an equivalent narrowed fix that skips inherited/config empty-allowlist failures for intentional no-tool `llm-task` embedded runs while preserving runtime `toolsAllow` conflicts; then close this issue as implemented.

## Reproduction Assessment

Yes. The issue provides a concrete Lobster plus `/tools/invoke` reproduction, and current main statically confirms the same path without needing a live gateway run in this read-only pass.

## Solution Assessment

Yes, the requested direction is the right narrow product behavior. The safest implementation should distinguish inherited config allowlists from explicit runtime `toolsAllow` so JSON-only no-tool runs proceed without weakening caller-requested tool constraints.
