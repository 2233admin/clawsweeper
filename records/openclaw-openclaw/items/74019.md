## Summary

Keep open. Current main still has `llm-task` start an embedded model run with `disableTools: true` while forwarding the caller config, and the embedded runner still collects config-derived allowlists, including `alsoAllow` promoted into an effective allowlist, before failing when no callable tools remain. This matches the report and is not covered by the older `llm-task` reachability report.

## Best Possible Solution

Keep this open and land a narrow embedded-runner fix: disabled, JSON-only embedded runs should ignore ordinary config-derived tool allowlists when no tools are meant to be callable, while still failing closed for contradictory runtime `toolsAllow` requests and for models/runs that actually requested tools. Add focused regression tests for the guard/attempt path and `llm-task`, plus a user-facing changelog fix entry crediting the reporter.

## Clownfish Work Prompt

Fix the valid bug reported in openclaw/openclaw#74019: Lobster can invoke the optional `llm-task` tool, but `llm-task` then starts a JSON-only embedded model run with `disableTools: true` and the run fails when normal config tool allowlists or `alsoAllow` entries are present. Keep the change narrow. The expected behavior is that config-derived tool allowlists (`tools.allow`, `tools.alsoAllow`, `agents.<id>.tools.allow`, `agents.<id>.tools.alsoAllow`, provider/group/sandbox/subagent config policies as appropriate) do not make a disabled-tools, JSON-only embedded run fail merely because no tools are callable. Preserve fail-closed behavior for an explicit runtime `toolsAllow` supplied to a disabled-tools run, and preserve failures for model/tool runs that actually request tools but resolve to none. Likely implementation paths are `src/agents/pi-embedded-runner/run/attempt.ts` and/or `src/agents/tool-allowlist-guard.ts`; avoid broad changes to plugin enablement, model selection, auth profile handling, or Lobster execution. Add regression tests that cover the `llm-task` case with `tools.alsoAllow: ["lobster", "llm-task"]` and the existing `runtime toolsAllow` + `disableTools` safety case. Update the changelog under the active version Fixes section with a single-line user-facing entry and `Thanks @technicallysecure`.
