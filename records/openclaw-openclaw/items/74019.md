## Summary

Keep open. Current main still matches the reported failure path: `llm-task` forwards the caller config into an embedded JSON-only run with `disableTools: true`, while the embedded runner still collects config-derived allowlists from `tools.alsoAllow` / agent tool policy and feeds them into the empty-tool guard. That can still produce the reported `No callable tools remain` precheck error before prompt submission.

## Best Possible Solution

Land a narrow embedded-runner fix: for disabled JSON-only embedded runs such as `llm-task`, ignore ordinary config-derived allowlist sources when no tools are meant to be callable, while preserving fail-closed behavior for explicit runtime `toolsAllow` and all normal tool-enabled runs. Add focused regression coverage and a changelog fix entry crediting @technicallysecure.
