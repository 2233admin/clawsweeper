Keep open. Current main still follows the reported failing path for `llm-task` with inherited tool allowlists, and the active fix candidate that references this issue with closing syntax is still open and unmerged.

## What This Changes

The issue requests that Lobster workflows invoking `llm-task` allow the embedded JSON-only model run to proceed with tools disabled instead of failing on inherited `tools.alsoAllow` entries.

## Best Possible Solution

Land a focused fix that lets intentional no-tool embedded/model runs ignore inherited config-derived empty allowlist failures while preserving explicit runtime `toolsAllow` conflicts, then close this issue as implemented.

## Reproduction Assessment

Yes. The issue provides concrete config and a Lobster `/tools/invoke` path, and current main statically confirms the same chain from `llm-task` `disableTools: true` through an empty tool set into a non-empty inherited allowlist guard failure.

## Solution Assessment

Yes for the requested direction, but the exact implementation should be scoped. The safest solution is not a blanket allowlist-guard bypass for all disabled-tool runs; it should preserve the runtime `toolsAllow` fail-closed behavior already asserted by tests.
