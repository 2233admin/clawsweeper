## Summary

Keep https://github.com/openclaw/openclaw/issues/74019 open. Current main still has the reported path: `llm-task` disables tools, the embedded runner creates an empty tool set, inherited allowlist sources are still collected, and the empty-allowlist guard still errors instead of treating the run as intentionally no-tool. An open PR already references this issue with closing syntax, so the issue should remain open until that fix or an equivalent lands.

## What This Changes

The issue requests that Lobster workflows invoking the `llm-task` plugin through `/tools/invoke` work when inherited `tools.alsoAllow` includes `lobster` and `llm-task`.

## Best Possible Solution

Land #74139 or an equivalent focused fix that skips inherited/config empty-allowlist failures for intentional no-tool `llm-task` embedded runs while preserving runtime `toolsAllow` conflicts; then close this issue as implemented.

## Reproduction Assessment

Yes. The issue provides a concrete Lobster plus `/tools/invoke` reproduction, and current main statically confirms the same failing path without needing a live gateway run in this read-only pass.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution. The safest implementation should distinguish inherited config allowlists from explicit runtime `toolsAllow` so JSON-only no-tool runs proceed without weakening caller-requested tool constraints.
