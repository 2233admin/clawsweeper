## Summary

Keep open. Current main still records OpenRouter streaming usage through the shared OpenAI-compatible parser and local pricing calculation, with no OpenRouter-owned post-stream generation lookup to reconcile `usage.cost.total` against authoritative billed `total_cost`. The merged tiered-pricing work is adjacent local-estimate support, not a fix for persisted assistant JSONL billing accuracy.

## What This Changes

The issue asks the gateway to stop persisting undercounted OpenRouter streamed `usage.cost.total` for tier-priced models by reconciling completed OpenRouter calls with authoritative generation `total_cost`.

## Best Possible Solution

Add OpenRouter-owned billed-cost reconciliation for completed streaming chat completions: when a verified OpenRouter stream finishes with a response id and credential, fetch `/generation`, validate a finite non-negative `total_cost`, and persist that authoritative billed value while preserving the current estimate on aborts, errors, missing ids, custom endpoints, or lookup failure.

## Risks / Open Questions

- A fix needs to decide whether a successful OpenRouter lookup overwrites `message.usage.cost.total` or adds a separate billed-cost field; overwriting is the narrower correctness path but still affects downstream consumers that currently treat the field as a local estimate.
- OpenRouter generation metadata can be delayed or unavailable after streaming, so the implementation needs a bounded lookup and a non-destructive fallback to the existing streamed estimate.
- The lookup must only run for verified OpenRouter routes with the resolved OpenRouter credential; custom OpenAI-compatible proxy endpoints should not receive OpenRouter-specific follow-up calls or credentials.
