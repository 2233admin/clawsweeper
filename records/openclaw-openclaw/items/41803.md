## Summary

Keep this PR open. Current main still returns raw unrecognized assistant error strings from the final `formatAssistantErrorText` fallback, and that formatter feeds user-facing embedded reply/lifecycle paths. The PR remains a plausible focused fix, but it should be refreshed before merge to address the warning-log security concerns raised in review.

## Best Possible Solution

Keep this PR open and ask for a small refresh against current main: keep the generic user-safe catch-all for truly unrecognized assistant errors, preserve recognized actionable branches before it, redact the full raw text before truncating any operator preview, pass that preview through `sanitizeForLog`, and retain the orphaned `tool_result`/long-string regression tests. After that refresh, this is a reasonable focused bug-fix candidate for the linked raw-error leak reports #11038 and #16948.

## Risks / Open Questions

- Merging the PR as-is could log partial secret fragments because it truncates before applying boundary-sensitive redaction patterns.
- Merging the PR as-is could still allow terminal/log injection or log obfuscation through ANSI/control characters because the new preview only strips CR/LF.
- A broad generic catch-all can hide actionable auth, model-not-found, session-expired, image-dimension, billing, schema, or transport guidance unless explicit branches and regression tests stay ahead of it.
- Leaving current main unchanged continues to expose raw unrecognized assistant/provider error strings on embedded chat reply paths.
