## Summary

Keep open. Current main has overlapping safety primitives, including tool profiles, sandboxing, exec approvals, and the experimental `update_plan` tool, but it still does not provide first-class Ask, Plan, and Modify workflow modes with enforced read-only behavior, plan preview/validation, and explicit transition into execution.

## What This Changes

The issue requests three user-visible workflow modes: Ask for read-only interaction, Plan for previewing intended actions, and Modify for executing changes after explicit confirmation.

## Best Possible Solution

Add a core product/runtime design that reuses existing enforcement surfaces: Ask should enforce a non-mutating tool policy, Plan should produce inspectable intended file/action changes and block mutation until approval, and Modify should execute under existing sandbox, tool policy, and exec-approval rules. The implementation should include UI/session controls plus tests at the runtime/tool-policy boundary proving that mutating tools cannot run in the wrong mode.
