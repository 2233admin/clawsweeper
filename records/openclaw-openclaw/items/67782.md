## Summary

Keep this PR open. Current main still performs delete-then-set for changed non-empty Telegram command menus, while the PR is a narrow contributor implementation that matches the scoped follow-up requested after the earlier superseded skip-delete PR. It needs maintainer review for sequencing with the open retry-after PR and for the stale-menu-on-set-failure tradeoff.

## What This Changes

The PR changes Telegram command-menu sync so non-empty menus call `setMyCommands()` without first calling `deleteMyCommands()`, while empty-menu sync still explicitly deletes commands and tests cover both paths.

## Best Possible Solution

Keep the change scoped to the Telegram plugin: use one `setMyCommands` call for non-empty command-menu updates, keep `deleteMyCommands` for explicit empty-menu clearing, and lock both paths with colocated tests. If maintainers consider stale menu metadata after a failed set unacceptable, add a focused retry or fallback in the same helper without changing broader channel or plugin contracts.
