## Summary

Keep open. Current main still documents and enforces managed-over-workspace hook precedence, so the name collision itself is expected, but the actionable diagnostics gap remains: an enabled hook with empty event metadata is skipped by the runtime loader while status/list/check can still treat it as ready. A prior closed unmerged PR (#30154) is useful source work, but it is not present on main and does not close the current report.

## What This Changes

The issue asks OpenClaw to surface or fix silent workspace hook collisions where same-name managed hooks win and eventless managed metadata leaves the user's workspace hook effectively disabled.

## Best Possible Solution

Keep the managed-over-workspace trust policy, but make the outcome visible and actionable: track ignored same-name workspace hooks in status/list/check output, mark enabled hooks with no metadata events as not ready, expose that reason in text and JSON output, and document rename/remove guidance for colliding managed hooks.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by defining same-name managed and workspace hook entries and observing that the managed entry wins; if the kept managed entry has empty event metadata, the loader skips it while status still computes it as loadable from enablement and requirements only.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable solution; changing workspace override precedence would be a broader trusted-code policy change and should stay out of an automated cleanup fix.
