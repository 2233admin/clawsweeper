## Summary

Keep open. Current main still preserves the managed-over-workspace hook trust boundary, but the reported diagnostics gap remains: an enabled managed hook with empty event metadata can win a same-name workspace collision, be skipped by the loader, and still look ready in hook status output because readiness ignores empty events.

## What This Changes

The issue reports that same-name workspace hooks are ignored in favor of managed hooks, and empty managed hook event metadata turns the intended workspace hook logic into a silent no-op.

## Best Possible Solution

Preserve the current hook trust boundary, but make the failure visible: carry ignored collision diagnostics into status/check/info output, mark enabled hooks with empty event metadata as not ready, provide clear rename/remove guidance in text and JSON output, and add focused tests plus docs/changelog coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to define a managed hook and workspace hook with the same name, leave the managed hook with no metadata events, and observe that current policy keeps the managed entry, the loader skips it, and status readiness still ignores the empty event list.

## Solution Assessment

Yes. A diagnostics-focused repair is the narrowest maintainable fix; allowing workspace hooks to override managed hooks would be a larger trust-policy change and is not needed to make the reported failure actionable.
