## Summary

Keep open. Current main still rejects workspace hooks that collide with managed hook names by design, and the remaining bug is that enabled managed hooks with empty event metadata can still look ready in hook status even though the runtime loader skips them.

## What This Changes

The issue asks OpenClaw to stop silently disabling same-named workspace hooks, or to surface actionable diagnostics when managed hook collisions and empty managed hook event metadata prevent workspace hook logic from running.

## Best Possible Solution

Preserve managed-over-workspace precedence, but make the shadowing actionable: carry collision and empty-event state into hook status, mark enabled hooks with no registered events as not ready, surface that in hooks list/info/check text and JSON, and document that users should rename workspace hooks or repair/remove the managed hook pack.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: configure a managed and workspace hook with the same name, then make the winning managed hook enabled with no metadata events; current main keeps the managed hook, the loader skips it, and status can still report it as loadable.

## Solution Assessment

Yes, the best fix is a diagnostics-focused core change. Allowing workspace hooks to override managed hooks would contradict the current trust policy, docs, and tests, so the narrow maintainable path is to make collisions and no-event hooks visible and not ready.
