## Summary

Keep open. Current main still intentionally prevents workspace hooks from overriding managed hooks, so the collision part is expected policy, but the diagnostics gap remains: an enabled managed hook with no metadata events is skipped by the runtime loader while hook status/list/check can still report it as ready. The same diagnostics behavior is present in the latest release tag, so this is not obsolete.

## What This Changes

The issue reports workspace hook names colliding with managed hook names being ignored at startup, followed by managed hooks with empty event metadata behaving like no-ops.

## Best Possible Solution

Preserve managed-over-workspace precedence, but make the shadowing actionable: include ignored collision information in hook status output, mark enabled hooks with empty event metadata as not ready, expose that reason in text and JSON, and document renaming/removing the colliding managed hook as the user path.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: define same-name managed and workspace hooks, resolve entries, and observe the managed entry wins; if that managed entry has no metadata events, the loader skips it while status still computes it as loadable from enablement and requirements only.

## Solution Assessment

Yes, a diagnostics-focused fix is the best way to solve this issue. Allowing workspace override would be a broader trust-policy change; the narrow maintainable path is to surface collisions and no-event hooks clearly without changing precedence.
