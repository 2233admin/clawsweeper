## Summary

Keep open. Current main still enforces managed/plugin/bundled precedence over same-named workspace hooks and still lets enabled hooks with empty event metadata look ready in status paths, so the diagnostics gap reported here remains actionable.

## What This Changes

The issue asks OpenClaw to avoid silently disabling same-named workspace hooks behind empty-event managed hooks, or at least surface the collision and no-event state through startup/status diagnostics and docs.

## Best Possible Solution

Keep the current no-override policy unless maintainers choose a trust-model change, but make collisions and empty `metadata.openclaw.events` actionable: show ignored same-name workspace hooks and no-event winners in `openclaw hooks list/info/check` including JSON, mark enabled no-event hooks as not ready, and document the rename/remove-managed-hook workaround.
