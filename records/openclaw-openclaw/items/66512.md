## Summary

Keep open. Current main still lacks the requested systemd readiness contract: no `sd_notify`/`NOTIFY_SOCKET` implementation is present, generated systemd units do not use `Type=notify` or `NotifyAccess=main`, and the gateway readiness boundary only updates OpenClaw state and logs readiness.

## What This Changes

The issue requests systemd `sd_notify` support so the gateway emits `READY=1` after it is actually ready and generated Linux units can use `Type=notify`.

## Best Possible Solution

Add a focused Linux gateway/service implementation that sends `READY=1` only after the existing gateway ready boundary, updates generated and documented systemd units to `Type=notify` with `NotifyAccess=main`, and leaves watchdog heartbeat work either explicitly coordinated with #11973 or as a follow-up.

## Repair Work Prompt

Implement systemd readiness notification for the OpenClaw gateway.

Observable problem: Linux systemd units generated by `openclaw gateway install` currently behave like `Type=simple`, so `systemctl restart openclaw-gateway` can return before the gateway has finished sidecar/plugin startup and before `/readyz`/RPC readiness is true. This issue asks for `READY=1` via systemd notify at the existing gateway ready boundary and generated units that use `Type=notify` with `NotifyAccess=main`.

Expected boundary:
- Add a small Linux/systemd notify helper, preferably zero-dependency or using a carefully reviewed optional/lazy dependency, that no-ops when `NOTIFY_SOCKET` is unset.
- Send `READY=1` only after the existing gateway-ready boundary in `startGatewayPostAttachRuntime`, after startup-gated methods are re-enabled and `gateway ready` is logged/ready state is true.
- Update `buildSystemdUnit` and tests so generated systemd units include `Type=notify` and `NotifyAccess=main` in `[Service]`.
- Update user-facing systemd docs/examples that currently show the generated unit shape.
- Do not implement `WATCHDOG=1` heartbeat support in this PR unless a maintainer explicitly asks to include #11973; note the overlap in the PR body.
- Keep core behavior generic and Linux/systemd-specific; do not add service-specific plugin IDs or broad startup migrations.
- Add a changelog entry because this is user-facing daemon behavior. Credit `Thanks @angeliti999`; include #11973 credit only if watchdog scope is intentionally included.

Related refs: this issue, #11973, closed unmerged attempts #11974 and #13014 for background only.

Likely files: `src/daemon/systemd-unit.ts`, `src/daemon/systemd-unit.test.ts`, `src/daemon/systemd.ts` if install/control behavior needs assertions, `src/gateway/server-startup-post-attach.ts`, a new helper/test under `src/infra/` or `src/gateway/`, `src/gateway/server-startup-post-attach.test.ts`, `docs/platforms/linux.md`, `docs/gateway/index.md`, possibly `docs/cli/gateway.md`, and `CHANGELOG.md`.
