## Summary

Current main implements the report's acceptable hard-timeout fix for the restart-deferred deadlock: omitted `gateway.reload.deferralTimeoutMs` now resolves to a 300000 ms bounded wait, and the restart deferral emits the restart on timeout instead of waiting forever behind active task counts. The stricter reject/queue-while-restarting behavior is still not present, but the indefinite deadlock and force-kill symptom are addressed on main.

## What This Changes

Prevent a new inbound message during a deferred restart-required config apply from leaving the main session stuck in processing indefinitely.

## Best Possible Solution

Keep the bounded restart-deferral behavior as the current fix, and track any desired restart-pending inbound UX as a narrower enhancement only if maintainers want reject-or-queue semantics beyond the deadlock safety net.

## Reproduction Assessment

Yes. The issue provides a concrete pre-fix reproduction path, and current-main source/test inspection verifies the indefinite wait no longer holds because omitted restart deferral timeout now resolves to 300000 ms.

## Solution Assessment

Yes. The hard timeout is one of the reasonable fixes named in the report and is narrower than plumbing restart-pending state through every inbound dispatch path; explicit reject/queue behavior can remain a separate UX decision.
