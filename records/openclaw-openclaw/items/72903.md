## Summary

Close as implemented: current main addresses the reported restart-deferred deadlock by making an omitted `gateway.reload.deferralTimeoutMs` resolve to a bounded 300000 ms wait across the config-reload and SIGUSR1 drain paths, with regression coverage and shipped beta provenance.

## What This Changes

The issue requests preventing a restart-required `config.apply` from being deferred indefinitely when a new inbound WebChat message starts a fresh task run during the restart-deferral window.

## Best Possible Solution

Keep the bounded default restart-deferral implementation as the fix for the unbounded deadlock. If maintainers later want a smoother user experience, track immediate inbound reject/queue behavior as a separate enhancement rather than reopening this runtime-deadlock report.

## Reproduction Assessment

Yes. The report gives a concrete active-task plus restart-required config change sequence, and current tests reproduce the core drain blocker by holding one active task while omitting `deferralTimeoutMs` and verifying the restart signal fires at the 300000 ms default.

## Solution Assessment

Yes. The current solution implements one of the reporter's proposed safe fix directions, the hard deferral timeout, in the shared resolver used by both config reload and restart drain paths; adding an inbound-specific `restartPending` gate is no longer required to prevent an indefinite deadlock.
