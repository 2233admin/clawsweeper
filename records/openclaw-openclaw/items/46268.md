## Summary

Keep this PR open. Current main still has the pending-tool-media queue/final-reply merge, but it does not implement this PR's specific fallback from the read tool's original start args when an inline image result has no MEDIA text and no details.path. The linked Feishu issue remains open, and the PR still represents a plausible implementation candidate after a small rebase/fix.

## Best Possible Solution

Keep the PR open as an implementation candidate for the open Feishu read-image media bug. Rebase it onto current main by making the media artifact extractor start-args-aware, falling back to trimmed path/file_path/filePath only when trusted image content exists and no explicit media path is present, then add handler/extractor regression tests and preserve the current trusted-tool filtering.

## Risks / Open Questions

- The PR is based on an older extractor shape; current main uses extractToolResultMediaArtifact plus pendingToolMediaUrls, so it needs rebase/adaptation before merge.
- The proposed fallback should treat empty or whitespace startArgs.path as missing before trying file_path, and should likely include the current filePath alias coverage if maintainers choose that compatibility path.
- Because the change promotes a local path into outbound media, it must remain limited to trusted image tool results and keep the existing local-media trust and outbound allowed-root checks intact.
