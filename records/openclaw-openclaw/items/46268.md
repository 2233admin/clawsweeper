## Summary

Keep this PR open. Current main has the later pending-tool-media queue and final-reply merge, but it still does not implement this PR's specific fallback from the read tool's original start args when an inline image result has no MEDIA text and no details.path. The linked Feishu bug remains open, and this PR is still a plausible implementation candidate after rebasing onto the newer extractor shape.

## Best Possible Solution

Keep the PR open as an implementation candidate for the open Feishu read-image media bug. Rebase it onto current main by making the media artifact extractor start-args-aware, falling back to a trimmed path/file_path/filePath only when trusted image content exists and no explicit media path is present, then add extractor and handler regression tests while preserving the existing trusted-tool filtering and reply-media allowed-root normalization.
