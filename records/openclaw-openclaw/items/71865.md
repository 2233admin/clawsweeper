## Summary

Keep open. Current main still compares canonical 2-space JSON output against the raw existing config byte length, so a PowerShell/BOM/verbose-format config can still trip the destructive size-drop guard before the auth-login config update is committed.

## What This Changes

Ensure `openclaw models auth login` persists provider auth when `openclaw.json` was previously written with PowerShell-style BOM and verbose formatting.

## Best Possible Solution

Update the shared config write safety check to distinguish formatting-only shrink from destructive clobber, preferably by comparing against a canonicalized previous config or a semantic diff while preserving hard rejection for real data loss. Add a regression test using a UTF-8 BOM plus verbose PowerShell-style JSON and verify auth-login messaging only reports success after the config write commits.

## Reproduction Assessment

Yes. The source path is high-confidence: the reported 19,829-byte canonical output is below half of the 59,662-byte raw PowerShell/BOM file, and `models auth login` still flows through the guarded writer.

## Solution Assessment

Yes. A narrow shared config-writer fix is the best direction; an auth-specific bypass would leave the same false positive for other OpenClaw-owned config writes and could weaken clobber protection inconsistently.
