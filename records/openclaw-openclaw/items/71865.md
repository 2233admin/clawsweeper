## Summary

Keep open. Current main still compares canonical 2-space JSON output against the raw previous config byte size, so a semantically equivalent PowerShell/BOM/verbose-formatted config can still trip the blocking size-drop guard before auth-login config updates commit.

## What This Changes

The issue reports that PowerShell-created UTF-8 BOM and verbose-formatted `openclaw.json` files can cause `openclaw models auth login` to report success while its config update is rejected by the size-drop safety guard.

## Best Possible Solution

Update the shared config write safety check to compare against a canonicalized previous config or semantic diff before treating shrink as destructive, then add regression coverage for UTF-8 BOM plus verbose PowerShell-style formatting and ensure auth login does not report success unless the config update commits or clearly surfaces the rejection.

## Reproduction Assessment

Yes. The source path is high-confidence: the reported 59,662 to 19,829 byte rewrite is below the guard’s 50% threshold, and `models auth login` still reaches that guarded writer through `updateConfig` and `replaceConfigFile`.

## Solution Assessment

Yes. A shared config-writer fix is the narrowest maintainable solution; an auth-specific `allowDestructiveWrite` bypass would leave the same false positive for other OpenClaw-owned config writes and weaken protection unevenly.

## Repair Work Prompt

Fix the config write false-positive reported in https://github.com/openclaw/openclaw/issues/71865. Current main serializes the next config as canonical 2-space JSON but compares `nextBytes` against the raw previous `openclaw.json` bytes, so a UTF-8 BOM plus verbose PowerShell-formatted config can shrink by more than 50% without semantic data loss and be rejected as `CONFIG_WRITE_REJECTED` during `openclaw models auth login`. Implement the fix in the shared config writer, not as an auth-only bypass: preserve destructive clobber protection for real data loss, but avoid blocking formatting-only shrink by using a canonicalized previous baseline, semantic diff, or similarly constrained safe check. Add a regression test with a BOM/PowerShell-style verbose formatted config that can be rewritten canonically while preserving content and committing the auth/config update. Also keep or strengthen the existing destructive-write rejection test. If auth login can still report success after a rejected config write, adjust the command/test so success is tied to committed config persistence or the rejection is surfaced. Add a user-facing `CHANGELOG.md` fix entry crediting @balric-seo. Do not set `allowDestructiveWrite` for auth login, do not remove the size-drop guard wholesale, and do not broaden config migrations beyond this writer behavior.
