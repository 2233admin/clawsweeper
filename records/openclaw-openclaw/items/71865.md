## Summary

Keep this issue open. Current main still compares raw existing config bytes against canonical 2-space JSON output, so a PowerShell/BOM/verbose-format config can still trip the size-drop guard even when the semantic config is not being destroyed. The auth login path still reaches that guarded writer, and I found no open PR linked to this issue.

## Best Possible Solution

Implement a narrow config-writer fix that compares the shrink guard against a canonicalized previous config baseline, or otherwise proves that the shrink is formatting-only, while preserving rejection for real semantic drops such as missing `gateway.mode` or large key loss. Add a PowerShell-style UTF-8 BOM plus verbose-indentation regression test and keep auth-login success reporting tied to a committed config update.
