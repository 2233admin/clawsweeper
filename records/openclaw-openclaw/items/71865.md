## Summary

Keep open. Current main still compares canonical 2-space JSON output against the raw previous config byte size, so a semantically equivalent PowerShell/BOM-formatted config can still trip the blocking size-drop guard before auth-login config updates commit.

## What This Changes

The issue requests that `openclaw models auth login` persist auth config updates even when an existing PowerShell/BOM-formatted `openclaw.json` shrinks substantially after canonical JSON rewriting.

## Best Possible Solution

Update the shared config writer so the size-drop safety check compares against a canonicalized previous config or semantic diff before blocking, while retaining hard failures for real destructive changes. Add regression coverage for UTF-8 BOM plus verbose PowerShell-style formatting, and ensure auth login does not report a committed config update unless the write actually commits or the rejection is clearly surfaced.

## Reproduction Assessment

Yes. The static path is high-confidence: a 59,662-byte raw config rewritten to about 19,829 bytes is below the guard’s 50% threshold, and `models auth login` still persists through `updateConfig` and the guarded writer.

## Solution Assessment

Yes. A shared config-writer fix is the narrowest maintainable solution; an auth-only `allowDestructiveWrite` bypass would leave the same false positive in other config mutations and weaken the guard unevenly.
