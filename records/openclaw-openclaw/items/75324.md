## Summary

Keep open. Current main still has a Discord message-tool discovery path that can strictly normalize a raw `channels.discord.token` SecretRef while describing actions, so the reported stack is still plausible despite earlier fixes for send/reply resolution and read-only status accessors.

## What This Changes

Discord action discovery should handle a SecretRef-backed `channels.discord.token` consistently with `secrets audit` and runtime SecretRef resolution.

## Best Possible Solution

Discord action discovery should use a config-only credential-presence/action-gate accessor, or an already scoped resolved runtime snapshot, so it can advertise supported actions without reading token plaintext. Runtime execution should continue to fail closed when the selected Discord token is unresolved.

## Reproduction Assessment

Yes. The issue provides concrete config and the current source path matches the stack: raw SecretRef token -> `describeDiscordMessageTool` -> `resolveDiscordAccount` -> `resolveDiscordToken` -> `normalizeResolvedSecretInputString`. Source inspection is enough to reproduce the failure path without live Discord credentials.

## Solution Assessment

No current solution is in main for this discovery path. The narrow maintainable fix is to split discovery from strict token normalization, similar to the recent read-only accessor pattern, rather than broadening global channel SecretRef resolution.
