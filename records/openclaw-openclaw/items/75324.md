## Summary

Keep open. Current main still has a Discord message-tool discovery path that strictly resolves the bot token while describing available actions, so a raw `channels.discord.token` SecretRef can still hit the reported unresolved SecretRef path even though older send/reply paths have separate scoped resolution.

## What This Changes

Fix Discord runtime message-tool/action discovery so SecretRef-backed `channels.discord.token` behaves consistently with `openclaw secrets audit` and does not fail during runtime Discord operations.

## Best Possible Solution

Discord action discovery should use a config-only credential-presence accessor, or an already resolved runtime snapshot, instead of strict token normalization. Runtime execution should remain strict and continue using the existing scoped SecretRef resolution/fail-closed path.

## Reproduction Assessment

Yes. The issue gives concrete config and current source has the direct path: `discordMessageActions.describeMessageTool` with a raw Discord token SecretRef calls `resolveDiscordAccount` -> `resolveDiscordToken` -> `normalizeResolvedSecretInputString`, matching the reported stack.

## Solution Assessment

No current solution on main. The older send/reply fixes are not enough because Discord `describeMessageTool` still uses strict account resolution; the safer fix is a read-only action-discovery accessor that treats configured SecretInputs as configured without reading token values.
