## Summary

Keep open. Current main still does not implement optional SIE verification in the skill loader: the loader reads and parses SKILL.md, enforces path/size/config eligibility, and builds skill entries without signed-envelope, trusted-issuer, warn/strict, or invalid-signature behavior. The related implementation PR #14893 is closed unmerged, and the adjacent hash-integrity PR #56256 is open and does not supersede this signature-verification design.

## Best Possible Solution

Keep this open for maintainer security design. A good path is a default-off TypeScript loader/config contract for SIE verification, deterministic machine-readable reason codes, warn/strict behavior for unsigned skills, reject behavior for invalid signatures, tests for bundled/managed/workspace/extra/plugin skill roots, and docs explaining how it relates to ClawHub scans, archive hashes, lockfiles, and skill install/update flows.

## Risks / Open Questions

- Security-sensitive loader trust policy needs maintainer design before implementation, especially default behavior, trusted issuer UX, revocation, and failure modes.
- Strict mode could reject existing workspace, managed, bundled, or third-party skills unless defaults and migration behavior are carefully staged.
- A plugin-only implementation appears insufficient because the requested behavior is a load-time gate before skills enter prompt/snapshot surfaces; current plugin skill directories do not provide a generic SIE verification hook.
- Related integrity work such as #56256 may overlap conceptually but does not currently supersede SIE signature verification; maintainers should align signing, hashes, ClawHub scan metadata, install/update, and audit behavior.
