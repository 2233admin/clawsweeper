## Summary

Keep this PR open. Current main at 7a23b2d945cc2d2e5c452e4641213e71de8dcc66 still does not implement the PR's server-side dual-bind behavior: specific tailnet/custom bind hosts still resolve to only that host, while local gateway callers still target loopback. The PR remains a plausible focused fix, but it overlaps with open client-side work in #50751 and changes documented tailnet behavior, so maintainer review is still needed.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either rebase and land the server-side dual-bind fix with specific-IP regression tests plus docs/security expectation updates, or explicitly choose the client-side host-resolution strategy in #50751 and close this PR only after a merged implementation demonstrably covers internal calls, CLI probes, and sessions_spawn for gateway.bind=tailnet and gateway.bind=custom.

## Risks / Open Questions

- Dual-binding tailnet/custom hosts would intentionally change documented behavior: same-host loopback access would begin working where the current docs say it will not.
- The listener-side change can affect gateway locality and auth/origin semantics because loopback and non-loopback requests are treated differently in gateway security code.
- Open #50751 overlaps with a client-side host-resolution strategy; maintainers should choose one coherent listener/target model before closing either implementation candidate.
