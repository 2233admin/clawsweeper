## Summary

Keep this PR open. Current main at 5de284c2e30f915b769fac08dbeab45f57749702 still does not implement the PR’s central server-side dual-bind behavior: specific tailnet/custom bind hosts resolve to only that host, while local gateway callers still default to ws(s)://127.0.0.1:<port>. The PR is focused and potentially useful, but it needs maintainer review because it changes documented tailnet bind behavior and overlaps with the alternate client-side approach in #50751.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either rebase and land the server-side dual-bind fix with specific-IP regression tests plus docs/security expectation updates, or explicitly choose the client-side host-resolution strategy in #50751 and close this PR only after a merged implementation demonstrably covers internal calls, CLI probes, and sessions_spawn for gateway.bind=tailnet and gateway.bind=custom.

## Risks / Open Questions

- Dual-binding tailnet/custom hosts would intentionally change documented behavior: same-host loopback access would begin working where docs currently say it will not.
- The listener change can affect gateway locality and pairing semantics because loopback/direct-local requests are treated differently from non-loopback requests in auth and scope handling.
- Open PR #50751 overlaps with a client-side host-resolution strategy; maintainers should choose one coherent listener/target model before closing either implementation candidate.
