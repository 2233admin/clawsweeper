## Summary

Keep this issue open. Current main has IPv6 support only for loopback alias listening; the reported LAN/container/tailnet dual-stack Gateway bind gap remains present, and the config/runtime contract still prevents IPv6 custom bind hosts.

## Best Possible Solution

Keep this issue open and implement it as a core Gateway networking change. Extend resolveGatewayBindHost/resolveGatewayListenHosts and the config contract so LAN, container auto, tailnet, and custom modes can intentionally support IPv6 or dual-stack listening; add focused regression coverage for ::1, ::, LAN/container behavior, tailnet IPv6 selection, and IPv6 custom hosts; then update gateway configuration/security docs plus status and diagnostic messaging.

## Risks / Open Questions

- Adding IPv6 or dual-stack non-loopback listeners widens Gateway exposure; auth requirements, Control UI allowed origins, trusted-proxy source checks, startup warnings, status output, and docs should be updated together.

- Relying on one wildcard bind behavior without explicit regression coverage may behave differently across host environments; coverage should prove ::1, ::, LAN/container, tailnet, and custom IPv6 cases intended by the product contract.
