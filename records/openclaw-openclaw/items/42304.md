## Summary

Keep this PR open. Current main at 5de284c2e30f915b769fac08dbeab45f57749702 still does not implement Android inline Adaptive Card parsing or rendering: Android chat renders text parts through Markdown and non-text parts as base64 images only. The PR is not merge-ready, but it remains a focused Android implementation candidate, and the same author's paired iOS/macOS Adaptive Cards PR #42350 is still open.

## Best Possible Solution

Keep this PR open for maintainer product review. If native Android rich chat is desired, rework it around the canonical MessagePresentation/client-rendering path with defensive parser/render limits, URL scheme allowlisting, real action routing or disabled controls, template-data support, and focused Android tests. If maintainers decide Adaptive Cards should stay outside core, close the Android and paired mobile/client Adaptive Cards PRs together with an explicit ClawHub/plugin-boundary explanation pointing to the existing openclaw-adaptive-cards plugin.

## Risks / Open Questions

- The PR is not safe to merge as-is without defensive parsing and rendering guards for malformed or model-generated card JSON, including width/rating bounds, recursion depth, multiple card blocks, and template data handling.
- Action routing and URL handling need product/security review: visible controls should either route through a real Android/gateway interaction path or be visibly disabled, and OpenUrl should use a scheme allowlist.
- The marker-based approach bypasses the documented MessagePresentation path, so maintainers should decide whether Android native cards belong in core client UI or in a plugin/ClawHub path before merge.
- Closing this PR now would discard a focused implementation candidate while the paired iOS/macOS PR #42350 remains open.
