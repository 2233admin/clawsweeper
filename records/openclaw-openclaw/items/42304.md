## Summary

Keep this PR open. Current main still does not implement Android inline Adaptive Card parsing or rendering: Android chat renders text through Markdown and non-text parts as base64 attachments only. The provided discussion shows a real Android implementation candidate tied to an existing openclaw-adaptive-cards plugin, but it is not merge-ready and the same author's paired iOS/macOS client PR #42350 is still open.

## Best Possible Solution

Keep this PR open for maintainer product and security review. If Android native rich chat is desired, rework it around the canonical MessagePresentation/client-rendering path with ordered fallback text, multiple-card handling, template data application, defensive render limits, URL scheme allowlisting, real action routing or visibly disabled controls, and focused Android tests. If maintainers decide Adaptive Cards should stay outside core, close the Android and paired #42350 client PRs together with an explicit ClawHub/plugin-boundary explanation that points to the existing openclaw-adaptive-cards plugin and identifies any missing client extension API.

## Risks / Open Questions

- The PR renders model/plugin-generated JSON inside Android chat, so malformed payloads need defensive bounds, recursion, rating, column-width, and multi-card handling before merge.
- OpenUrl actions are a security-sensitive UI path and should use an explicit scheme allowlist or equivalent app policy before shipping.
- The marker-based approach bypasses the documented MessagePresentation path, so maintainers should decide whether Android native rich cards belong in core client UI or need a new client/plugin extension API.
