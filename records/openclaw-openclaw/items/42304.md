## Summary

Keep this PR open. Current main still does not implement Android inline Adaptive Card parsing/rendering, and this PR is a focused Android implementation candidate rather than obsolete cleanup. It is not merge-ready as-is, and the same author's paired iOS/macOS Adaptive Cards PR #42350 is still open, so closing only this PR would split a paired client feature set.

## Best Possible Solution

Keep this PR open for maintainer product review. If native Android rich chat is desired, rework it around the canonical MessagePresentation/client-rendering path with defensive parsing, depth and numeric guards, URL scheme allowlisting, real action routing or visibly disabled actions, template-data handling, and focused Android tests. If maintainers decide Adaptive Cards should stay outside core, close the Android and paired mobile/client Adaptive Cards PRs together with an explicit ClawHub/plugin-boundary explanation that points to the existing `openclaw-adaptive-cards` plugin.

## Risks / Open Questions

- The PR is not safe to merge as-is: model/plugin-supplied card JSON can still crash Android rendering through invalid `Modifier.weight(...)` inputs, negative rating bounds, or deeply recursive layout.
- `Action.Submit` and `Action.Execute` render as clickable controls but only log clicks, so card workflows look interactive without routing back to the agent or gateway.
- `Action.OpenUrl` does not allowlist URI schemes before calling Android's URI handler.
- The implementation creates a marker-based rich-message path that bypasses the documented MessagePresentation contract and would need product/API alignment before becoming core client behavior.
- No Android parser/render regression tests are included for malformed JSON, invalid widths, negative rating bounds, multiple card blocks, template data application, URL schemes, action routing, or depth limits.
- Security/supply-chain pass found no new workflows, dependencies, lockfiles, installers, release scripts, or package metadata in this PR, so the main concerns are runtime handling of untrusted card payloads rather than third-party code execution.
