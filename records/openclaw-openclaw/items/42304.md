## Summary

Keep this PR open. Current main still does not implement Android inline Adaptive Card parsing or Jetpack Compose rendering; Android chat currently renders text through ChatMarkdown and non-text parts as base64 images only. The PR remains a plausible implementation path, but it needs maintainer product and security review before merge or before being redirected to a client-renderer/plugin API path.

## Best Possible Solution

Keep this PR open for maintainer product and security review. If Android-native Adaptive Cards are desired, rework the PR around the current MessagePresentation direction or a documented Android client-renderer extension point, preserve ordered mixed text/card content, handle multiple cards and template data, add defensive render limits and URL/action policy, and add focused Android parser/render tests. If maintainers decide this belongs outside core, define the missing client-renderer API first, then close the client renderer PRs together in favor of the external plugin path.
