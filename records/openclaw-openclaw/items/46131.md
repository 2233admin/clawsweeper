## Summary

Keep open. Current main still has the inline Anthropic API defaulting gap, but this PR should not merge as-is because its branch also changes Telegram non-idempotent send retry classification outside the stated model fix.

## What This Changes

The PR applies provider-level API defaulting to inline provider models with regression tests, and also adds Telegram rate-limit and grammY failed-after send retry handling.

## Best Possible Solution

Land a model-only replacement against current main that normalizes provider policy before inline provider models are built, covers implicit Anthropic API defaulting in `model.inline-provider.test.ts`, and leaves Telegram retry semantics unchanged.

## Reproduction Assessment

Yes. A raw Anthropic inline provider with models and no `api` reaches `buildInlineProviderModels` with `model.api ?? entry?.api` undefined, then `resolveExplicitModelWithRegistry` skips it because `inlineMatch?.api` is false.

## Solution Assessment

No, not as this PR branch. The model-only direction is the right narrow fix, but current main should be patched in `model.inline-provider.ts` using the existing provider-policy normalization surface without the unrelated Telegram retry changes.

## Review Findings

- **[P1] Keep failed-after envelopes out of safe send retries:** `src/telegram/network-errors.ts:214-216`
