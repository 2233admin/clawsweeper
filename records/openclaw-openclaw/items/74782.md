## Summary

Keep open. Current main still treats non-fallback Gateway transport failures in `openclaw logs --follow` as fatal, and open PR #75059 is linked with closing syntax as an implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request that `openclaw logs --follow` retry transient Gateway disconnects with warning/backoff instead of exiting immediately.

## Best Possible Solution

Land or adapt a focused CLI change that retries only transient `logs --follow` transport failures with bounded backoff and warning output, while preserving fatal handling for non-recoverable Gateway errors and the existing local fallback boundary.

## Reproduction Assessment

Yes. The code-level reproduction is a `fetchLogs` / `callGatewayFromCli` transport rejection inside the follow loop; current tests already assert that an explicit Gateway URL close error exits with `process.exit(1)`.

## Solution Assessment

Yes in direction. The maintainable fix belongs around `src/cli/logs-cli.ts` follow polling, with targeted tests for explicit-URL retry behavior and implicit-loopback fallback preservation.
