## Summary

Keep open. Current main still exits from the `logs --follow` polling loop on non-fallback Gateway transport failures, and open PR #75059 is already linked with closing syntax as the implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request `openclaw logs --follow` to retry transient Gateway disconnects with a warning and backoff instead of terminating the log tail session.

## Best Possible Solution

Land or replace the focused `logs --follow` retry/backoff implementation, keeping fatal handling for non-recoverable Gateway errors and preserving the existing local fallback boundary.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to make `callGatewayFromCli("logs.tail")` reject with a Gateway close/timeout error during `logs --follow --url ...`; current main catches that failure and exits with status 1, and the existing explicit-URL test asserts the same fatal behavior.

## Solution Assessment

Yes in direction. The narrow maintainable fix belongs in `src/cli/logs-cli.ts` around the follow polling loop or a small shared one-shot retry helper, while keeping non-recoverable Gateway/auth/config errors fatal.
