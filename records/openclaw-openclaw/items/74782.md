## Summary

Keep open. Current main still exits from the `logs --follow` loop on any `fetchLogs` error that is not absorbed by the narrow local-file fallback, and an open PR specifically references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix merges.

## What This Changes

Request auto-reconnect/backoff for `openclaw logs --follow` after transient Gateway transport disconnects instead of terminating the log tail session.

## Best Possible Solution

Review and either land PR #75059 or replace it with an equivalent narrow change in `src/cli/logs-cli.ts` that retries bounded transient Gateway transport failures during `--follow`, resets retry state after successful polls, keeps non-recoverable errors fatal, and updates focused tests/docs for the new behavior.

## Reproduction Assessment

Yes. The current test suite already has a focused reproduction shape: a mocked explicit `--url` Gateway close error reaches the fatal path and asserts `process.exit(1)`, matching the reported transient-disconnect symptom for non-fallback follow sessions.

## Solution Assessment

Yes, the requested direction is the right narrow product fix, but current main has not implemented it. The maintainable approach is local to the logs CLI polling loop and typed Gateway transport errors, not a broad Gateway client behavior change.
