## Summary

Keep open. Current main still exits the `logs --follow` polling loop on Gateway errors that are not handled by the local-file fallback, and open PR #75059 references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Request `openclaw logs --follow` to reconnect with warning/backoff after transient Gateway transport disconnects instead of exiting the log tail session.

## Best Possible Solution

Review and either land PR #75059 or replace it with an equivalent narrow change in `src/cli/logs-cli.ts` that retries bounded transient Gateway transport failures during `--follow`, resets retry state after successful polls, keeps non-recoverable errors fatal, and updates focused tests/docs for the behavior.

## Reproduction Assessment

Yes. The issue gives concrete runtime steps, and current main has a source/test-level reproduction: an explicit `--url` Gateway close error reaches the fatal path and asserts `process.exit(1)`.

## Solution Assessment

Yes. The narrow maintainable solution is a logs CLI follow-loop retry/backoff path using existing Gateway transport classification, while leaving auth/configuration failures fatal and preserving local fallback behavior.
