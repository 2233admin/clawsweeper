## Summary

Keep open. Current main still lacks the Mattermost read-only account inspector, and the PR is directionally the right owner-local fix, but the latest PR head has a blocking correctness edge where read-only status can report an env token as available even though unchanged runtime resolution still throws on an unresolved configured SecretRef.

## What This Changes

The PR adds a Mattermost read-only account inspector, wires it into the Mattermost config adapter and status snapshot, and adds regression tests for unresolved SecretRef bot tokens.

## Best Possible Solution

Land the Mattermost-owned read-only inspector and status snapshot fix after aligning read-only credential precedence with the unchanged strict runtime resolver, with a regression test for the configured-SecretRef-plus-env-token case. Runtime token fallback should stay strict unless the PR intentionally changes that contract too.

## Reproduction Assessment

Yes. Current main has a concrete reproduction path: configure Mattermost with a botToken SecretRef plus baseUrl and run openclaw status, which reaches strict resolveMattermostAccount without a Mattermost inspectAccount. The PR finding is reproducible by adding MATTERMOST_BOT_TOKEN alongside the unresolved configured SecretRef.

## Solution Assessment

No. Adding a Mattermost inspectAccount is the right narrow approach, but the latest PR should not report env credentials as available when the unchanged runtime resolver still fails on the configured SecretRef first.
