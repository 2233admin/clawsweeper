## Summary

Keep this PR open: it addresses a real Mattermost status bug that still exists on current main, but the latest patch has a correctness issue in the env-token fallback edge. The new read-only inspector can report credentials as available from `MATTERMOST_BOT_TOKEN` while the unchanged runtime resolver would still throw on the configured SecretRef before reaching that env fallback.

## What This Changes

The PR adds a Mattermost read-only account inspector, wires it into the config adapter, preserves `botTokenStatus` in status snapshots, and adds regression coverage for unresolved SecretRef bot tokens.

## Best Possible Solution

Land a Mattermost-owned `inspectAccount` and status snapshot fix after aligning read-only credential precedence with the unchanged runtime resolver, with an added regression test for the SecretRef-plus-env-token edge. Runtime token resolution should stay strict unless the PR intentionally changes that contract too.

## Reproduction Assessment

Yes. The original crash has a concrete config shape and stack trace, and current main still lacks a Mattermost `inspectAccount` while strict resolution dereferences `botToken`; the review finding is reproducible by combining an unresolved config SecretRef with `MATTERMOST_BOT_TOKEN`.

## Solution Assessment

No. Adding a read-only Mattermost inspector is the right narrow fix, but the latest implementation should not report env credentials as available when a configured SecretRef would still make runtime resolution fail.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep SecretRef status ahead of env fallback:** `extensions/mattermost/src/mattermost/accounts.ts:187-198`
  - body: When `botToken` is a configured SecretRef and `MATTERMOST_BOT_TOKEN` is present, this branch reports `botTokenSource: "env"` and `botTokenStatus: "available"`. Runtime `resolveMattermostAccount` still strict-resolves `merged.botToken` before considering env, so the same config can fail startup while status/probe reports credentials OK. Preserve `configured_unavailable` precedence unless runtime fallback semantics change too.
