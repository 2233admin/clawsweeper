## Summary

Keep open. Current main still accepts iOS composer image bytes directly from PhotosPicker/paste, rejects payloads over 5 MB before any transform, and later base64-encodes the stored original bytes for transport. PR #73710 is an open implementation candidate with closing syntax for this issue, so the right cleanup is to leave this issue open until that PR is reviewed, validated, and either merged or closed.

## What This Changes

The issue requests client-side resizing/compression for large iOS chat composer image attachments before transport, including off-MainActor processing, pixel/byte budgets, and multi-image budget handling.

## Best Possible Solution

Land a reviewed implementation that routes iOS composer image attachments through a client-side image processor before storing/sending bytes, reusing or extending `JPEGTranscoder`, preserving the final 5 MB safety gate, and adding focused OpenClawKit tests for size, metadata stripping, invalid data, and non-upscale behavior.
