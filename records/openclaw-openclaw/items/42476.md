## Summary

Keep this PR open. Current `main` still does not implement the PR’s intended hardening: the `/media/:id` handler sets `X-Content-Type-Options: nosniff`, but it does not set the requested `Cache-Control`, `Pragma`, or `Expires` anti-cache headers on temporary media responses. The PR remains a focused security-hardening candidate, but it is stale against the current `createMediaRequestHandler` shape and needs a rebase plus broader response-path assertions before merge.

## Best Possible Solution

Keep this PR open. Port the hardening onto current `src/media/server.ts` by setting `Cache-Control: no-store, private, max-age=0, must-revalidate`, `Pragma: no-cache`, and `Expires: 0` in `createMediaRequestHandler` immediately after a matched `/media/:id` route and before validation/read/error branches. Add or extend tests so success, HEAD, missing, expired, invalid, oversized, and outside-workspace responses assert the anti-cache headers while preserving TTL expiry, cleanup timing, path validation, active-content download behavior, and file-serving semantics.
