## Summary

Keep this PR open. Current main still does not implement the intended temporary media anti-cache hardening: the active media request handler only sets `X-Content-Type-Options: nosniff`, the `src/media` tests do not assert `Cache-Control`, `Pragma`, or `Expires`, and the two-minute `/media/:id` temporary URL model is still current. The PR remains a narrow security-hardening candidate, but it needs to be ported from the older `attachMediaRoutes` shape onto current `createMediaRequestHandler` and broadened across response-path tests.

## Best Possible Solution

Keep this PR open. Port the header setting into `createMediaRequestHandler` immediately after a matched `/media/:id` route and before validation/read/error branches, then expand media server tests to cover success, HEAD, expired, invalid, oversized, missing, and outside-workspace responses while preserving TTL expiry, cleanup timing, path validation, active-content download behavior, and file-serving semantics.
