## Summary

Keep open. Current main still peeks pending system events from the base heartbeat session, switches the model run to the isolated `:heartbeat` session, and then reads active-session pending events from that isolated key without transferring or aliasing the base queue. The related #73297 routing issue was closed as not reproducible and does not close this active-session event-queue contract gap.

## What This Changes

The issue requests carrying or aliasing pending system-event entries from a base heartbeat session into the isolated `:heartbeat` session when `heartbeat.isolatedSession` is enabled.

## Best Possible Solution

Add a narrow isolated-heartbeat event-visibility seam so active-session consumers see the relevant pre-isolation entries exactly once, while preserving the current delivery-target guard and session-scoped queue semantics.

## Reproduction Assessment

Yes. Enqueue a system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and inspect that the model context uses the `:heartbeat` session while active-session event inspection reads only that isolated key.

## Solution Assessment

Yes. A small transfer-or-alias helper around isolated heartbeat session creation is the narrowest maintainable path; a broader rewrite of system-event consumption or dmScope routing would have more blast radius than this issue needs.

## Repair Work Prompt

Fix this issue with a narrow isolated-heartbeat system-event visibility change. Current main peeks `preflight.pendingEventEntries` from the base session, switches the model run to `${base}:heartbeat`, then computes active-session events from `peekSystemEventEntries(runSessionKey)`, leaving base-queued entries invisible to consumers of the active isolated key. Implement a small helper in `src/infra/system-events.ts` or a local seam in `src/infra/heartbeat-runner.ts` that transfers or aliases the inspected base entries to the isolated run exactly once before the model call. Preserve existing `consumeSystemEventEntries(sessionKey, preflight.pendingEventEntries)` semantics, avoid replaying consumed events, and do not reuse base `turnSource` delivery context for isolated wake runs. Do not broaden dmScope routing from #71581. Add regression coverage for a base-session event under `heartbeat.isolatedSession: true` proving active isolated-session inspection sees it, and keep the stale-routing and untrusted-event tests green. Add a changelog entry only if the final behavior is user-visible; otherwise omit it as internal contract hardening.
