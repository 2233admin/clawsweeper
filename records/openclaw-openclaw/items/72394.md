## Summary

Keep this issue open. Current main still peeks pending system events from the base heartbeat session before isolation, creates or refreshes the `:heartbeat` session without copying those event entries, and then reads active-session events from the isolated key separately. Existing tests cover adjacent behavior and currently assert the asymmetry the report describes rather than proving it fixed.

## Best Possible Solution

Keep this issue open and implement a narrow heartbeat/system-events seam: either transfer the inspected base-session event entries into the isolated heartbeat queue before the model run or make active-session event reads explicitly consult the pre-isolation base queue for that run. Add focused regression coverage for cron events, hook/wake exec events, trusted and untrusted flags, delivery context, and exactly-once consumption.

## Risks / Open Questions

- This is currently a contract-hardening issue more than a proven user-visible regression: existing prompt-building paths can inline or consume inspected base-session events, but future consumers of active-session `peekSystemEventEntries(runSessionKey)` would observe an empty queue.
- A fix needs focused exactly-once semantics so transferred or aliased events are not replayed from the base session, do not pin stale delivery routing, and preserve trusted/untrusted handling for cron, exec, and wake-triggered paths.
