## Summary

Keep open. Current main still peeks pending system events on the base heartbeat session during preflight, then switches isolated heartbeat runs to a separate `:heartbeat` session and peeks that active queue separately without copying, moving, or aliasing the base-session entries. The existing tests cover adjacent behavior and still document the asymmetry: base cron events are consumed/inlined from the base queue, while base-session hook/wake exec events are not treated as active isolated-session exec events. The closed Telegram hijack report #73297 is related context but was explicitly closed as not reproducible and distinguished from this queue-boundary issue.

## What This Changes

The issue asks for pending system-event entries queued on the base heartbeat session to be transferred or aliased into the heartbeat-created `:heartbeat` isolated session before the model run.

## Best Possible Solution

Add a narrow heartbeat/system-events seam that makes the events inspected on the base session visible to the active isolated heartbeat run, while preserving trust flags, delivery context, ordering, and exactly-once consumption. Lock it with focused tests for base cron events, exec/hook wake events, untrusted flags, active isolated events, and no replay after success.

## Repair Work Prompt

Implement a narrow fix for openclaw/openclaw#72394: when `heartbeat.isolatedSession: true` causes a heartbeat run to execute on a `:heartbeat` session, pending system-event entries inspected on the base session must also be visible to consumers that read `peekSystemEventEntries(runSessionKey)` for the active isolated session. Keep the change scoped to heartbeat/system-event queue semantics. Preserve event text, ordering, contextKey, deliveryContext, and trusted flags. Preserve exactly-once consumption: successful runs must not replay base entries on later heartbeats, and active isolated-session entries must still be consumed normally. Do not broaden channel routing, `session.dmScope`, heartbeat key normalization, or delivery policy. Reconcile first with any merged or still-open adjacent work in #72814 and #72648; do not duplicate a fix if current main already changed. Likely implementation paths are either a small queue transfer/alias helper in `src/infra/system-events.ts` used by `src/infra/heartbeat-runner.ts`, or a heartbeat-local inspected-event mapping that active-session consumers read before model execution. Add focused tests for base cron events, base exec/hook wake events, active isolated exec events, untrusted flag handling, delivery context preservation, and no replay after successful consumption. Changelog is not expected unless the final patch fixes an observable user-facing regression rather than this future-facing contract gap.
