## Summary

Keep open. Current main still has the isolated-heartbeat/system-event contract gap: preflight inspects the base session queue, isolated execution switches to a `:heartbeat` session key, and active-session event inspection then reads only that isolated key without any queue transfer or alias. The existing ClawSweeper comment and related closed Telegram routing report do not make this implemented, and the overlapping heartbeat-routing PR is still open and unmerged.

## What This Changes

Request to make pending system-event entries queued on a base heartbeat session visible across the `heartbeat.isolatedSession` boundary when the run uses the isolated `:heartbeat` session key.

## Best Possible Solution

Add a narrow heartbeat-boundary transfer or alias for relevant pre-isolation system-event entries so isolated active-session consumers can observe them exactly once, while preserving base-queue consumption, duplicate handling, stale delivery-routing safeguards, and existing prompt inlining behavior.

## Reproduction Assessment

Yes. A focused test can enqueue a trusted or untrusted system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and observe that the model context uses the isolated `:heartbeat` key while active-session event inspection currently sees no copied base entries.

## Solution Assessment

Yes. A scoped transfer-or-alias seam at isolated heartbeat creation is the narrowest maintainable direction; restructuring all event consumption would touch more routing and delivery behavior than this latent contract-hardening issue requires.
