## Summary

Keep open. Current main still peeks pending system-event entries from the base heartbeat session, switches the model run to an isolated `:heartbeat` session, and then reads active-session events from that isolated key without copying, moving, or aliasing the base-session queue. The adjacent tests cover prompt inlining, active-isolated events, and stale routing guards, but they do not prove the requested active-session event contract is fixed.

## What This Changes

The issue requests carrying pending system-event entries from the base heartbeat session into the active isolated `:heartbeat` session when `heartbeat.isolatedSession` is enabled.

## Best Possible Solution

Add a narrow, generic transfer-or-alias seam for isolated heartbeat creation so relevant pre-isolation system-event entries are visible through `peekSystemEventEntries(runSessionKey)` exactly once, while preserving the current stale delivery-target guard and base-queue consumption semantics.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and verify that the active isolated `:heartbeat` key is the model context while `peekSystemEventEntries(runSessionKey)` currently sees no copied base entries.

## Solution Assessment

Yes, the narrowest maintainable solution is a scoped transfer or active-session alias at the isolated heartbeat boundary. A broad rewrite of event consumption would have a larger blast radius than this contract-hardening issue needs.
