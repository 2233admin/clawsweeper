## Summary

Keep open. Current main still has the reported isolated-heartbeat boundary gap: base-session system events are read for prompt construction, but the isolated `:heartbeat` session is created without transferring or aliasing that event queue, and active-session consumers later inspect only the isolated key. No relevant source diff exists between the prior ClawSweeper review SHA and current main.

## What This Changes

The issue requests making pending system event entries queued on a base heartbeat session visible across the isolated `:heartbeat` session boundary.

## Best Possible Solution

Add a narrow, documented heartbeat-boundary transfer or alias for the relevant pre-isolation system-event entries so isolated-session consumers can observe them exactly once, while preserving base-queue ordering, duplicate handling, delivery-context routing, and trust-boundary behavior.

## Reproduction Assessment

Yes. A focused test can enqueue a system event on the base heartbeat session, run `runHeartbeatOnce` with `heartbeat.isolatedSession: true`, and assert that `peekSystemEventEntries` on the isolated `:heartbeat` key remains empty on current main; this review stayed read-only and did not run tests.

## Solution Assessment

Unclear. A scoped transfer-or-alias at isolated heartbeat creation looks narrower than rewriting every consumer, but maintainers should first decide how wake-triggered and untrusted base events are supposed to affect isolated-session inspection.
