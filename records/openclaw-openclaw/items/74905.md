## Summary

Keep open. Current main still sends direct APNs through raw `http2.connect`, so the PR is not obsolete, but the latest head still has blocking security-sensitive CONNECT tunnel issues and the PR has `security`/`maintainer` labels plus a MEMBER author association that require maintainer handling.

## What This Changes

The PR replaces direct APNs HTTP/2 connection creation with an APNs proxy wrapper, adds active managed proxy URL state and CONNECT tunnel code, and updates lint/OpenGrep guardrails, tests, docs, and changelog text.

## Best Possible Solution

Keep the APNs-specific proxy seam, but harden the tunnel before merge: route synchronous CONNECT request failures through the normal cleanup path, bound CONNECT response parsing, and either use the maintained proxy agent path described in the PR or explicitly document and test the custom tunnel contract.

## Reproduction Assessment

Yes. Current main reproduces the underlying bypass by inspecting `sendApnsRequest`, and the remaining PR defects are reproducible by passing malformed percent-encoded proxy credentials or by having a proxy stream an unterminated CONNECT response header.

## Solution Assessment

No, not as-is. The PR direction is narrow and appropriate, but the custom tunnel needs bounded parsing and predictable error cleanup before it is the safest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Route CONNECT write errors through fail():** `src/infra/net/http-connect-tunnel.ts:116`
  - body: `writeConnectRequest()` can throw while decoding proxy credentials, for example with malformed percent encoding. Because `onConnected()` calls it directly from the socket event handler, that exception bypasses `fail()`, skips socket cleanup, and can crash the process instead of rejecting the APNs send.
  - confidence: 0.91
- **[P2] Cap CONNECT response headers before buffering:** `src/infra/net/http-connect-tunnel.ts:121-123`
  - body: `onData()` appends every chunk to `responseBuffer` until it sees `\r\n\r\n`, with no maximum size. A bad proxy can keep sending unterminated header bytes and grow memory until timeout or OOM; enforce a fixed CONNECT header limit and fail the tunnel when exceeded.
  - confidence: 0.88

## Security Review

Status: needs_attention
