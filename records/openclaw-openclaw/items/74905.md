## Summary

Keep open. This is an active draft PR from a MEMBER with the protected `maintainer` label, and the current patch still has a source-backed P2 correctness issue in the APNs proxy-selection path.

## What This Changes

The PR adds a proxy-aware APNs HTTP/2 session helper and HTTP CONNECT tunnel helper, wires direct APNs sends through it, and adds regression tests plus a raw `http2.connect` guard script.

## Best Possible Solution

Keep the PR open for maintainer review, preserve the narrow APNs HTTP/2 connection seam, and change the APNs helper to honor the shared proxy decision semantics before opening a CONNECT tunnel, with a focused regression for matching `NO_PROXY`.

## Reproduction Assessment

Yes. On the PR branch, `HTTPS_PROXY=http://proxy.test:8080` with `NO_PROXY=api.push.apple.com` still reaches the tunnel path because the APNs helper reads the proxy URL directly instead of using the shared proxy decision helper.

## Solution Assessment

No. The helper boundary is the right place to solve direct APNs HTTP/2 proxying, but the patch should first check `shouldUseEnvHttpProxyForUrl(authority, env)` and only tunnel when that returns true.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Honor NO_PROXY before tunneling APNs:** `src/infra/push-apns-http2.ts:37`

Please gate the tunnel with the shared proxy decision helper and add the APNs regression.
