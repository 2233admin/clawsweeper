## Summary

Keep open. Current main still has the direct APNs HTTP/2 proxy gap, and this draft security/maintainer PR is a plausible narrow fix, but current head still ignores the shared NO_PROXY bypass contract before opening a CONNECT tunnel.

## What This Changes

The PR routes direct APNs HTTP/2 sessions through a proxy-aware CONNECT/TLS helper, wires `push-apns.ts` to that helper, and adds tunnel/APNs tests plus raw-HTTP/2 lint/OpenGrep guardrails and a docs note.

## Best Possible Solution

Keep the APNs HTTP/2 wrapper approach, but make tunneling depend on the shared `shouldUseEnvHttpProxyForUrl(authority, env)` decision before resolving and opening the CONNECT tunnel, then add a focused APNs regression for `HTTPS_PROXY` plus matching `NO_PROXY` and let maintainers complete the protected security review.

## Reproduction Assessment

Yes. On current PR head, `HTTPS_PROXY=http://proxy.test:8080` with `NO_PROXY=api.push.apple.com` still reaches the CONNECT helper because `connectApnsHttp2Session` reads the proxy URL directly; current main’s shared proxy test establishes that this combination should return false.

## Solution Assessment

No. The low-level APNs session seam is the right fix boundary, but the proxy-selection step should reuse the existing NO_PROXY-aware helper before opening a tunnel.
