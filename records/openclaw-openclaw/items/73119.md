## Summary

Keep open. Current main still contains the reported raw transcript rewrite mechanism: `sessions_history` is a read-only gateway history call, but the post-turn tool-result recovery path can plan replacements and then branch at the first replacement while re-appending the entire remaining branch suffix, including unchanged messages. The separate WebChat/TTS duplicate issue is closed but does not supersede this transcript-rewrite behavior.

## What This Changes

The issue asks OpenClaw to stop `sessions_history`-triggered large tool-result recovery from duplicating prior assistant and tool-result transcript rows as fresh JSONL branch entries.

## Best Possible Solution

Keep `sessions_history` as a pure bounded read and fix the transcript maintenance path narrowly so tool-result recovery does not create low-value duplicate raw JSONL rows while preserving append-only DAG semantics, branch labels, compaction metadata, and provider overflow recovery. Add regression coverage that counts raw appended entries as well as active-branch history output.

## Reproduction Assessment

Yes. The issue includes concrete transcript and trajectory evidence, a follow-up comment reproduced the mechanism, and current main still has the same replacement-plan to branch-and-reappend path that explains the observed fresh IDs with preserved assistant `responseId`s.

## Solution Assessment

Unclear. The current implementation is not the best solution for this issue because it still replays unchanged suffix entries, but choosing the narrowest maintainable fix needs maintainer judgment around append-only transcript semantics and context-engine maintenance contracts.
