## Summary

Keep open. Current main still has the raw transcript duplication mechanism described in the issue: `sessions_history` itself is a bounded read path, but large history output can become a tool result, trigger tool-result truncation, and then the transcript rewrite helper branches at the first replacement while re-appending the unchanged suffix into append-only JSONL. Active-branch history readers reduce visible duplicates, but they do not remove the raw replay entries reported here.

## What This Changes

The issue asks OpenClaw to stop `sessions_history`-triggered transcript rewrite/replay from duplicating prior assistant and tool-result records in raw session JSONL while preserving correct billing and model context behavior.

## Best Possible Solution

Keep the issue open until #69486, #72511, or an equivalent maintained fix lands and is verified against this `sessions_history`-triggered raw JSONL symptom. The end state should keep `sessions_history` as a pure bounded read while changing transcript rewrite/truncation maintenance so raw replay duplication is avoided or reconciled without breaking append-only branch semantics and sibling ancestry.

## Reproduction Assessment

Yes. The issue gives concrete session JSONL and trajectory evidence, a commenter reproduced and traced the mechanism, and current-main code still contains the truncation-to-rewrite suffix replay path; this review did not run a live repro because the checkout had to remain read-only.

## Solution Assessment

Unclear. Fixing `sessions_history` itself is not the best boundary because the tool only reads; the durable fix belongs in transcript rewrite/truncation cleanup, and maintainers should choose between the existing open repair strategies before another automated patch is queued.
