## Summary

Keep this issue open. Current `main` still only treats direct Slack `<@botUserId>` mentions, `app_mention` events, configured mention regexes, and implicit thread participation as mention sources; it has no Slack user-group token extraction, membership lookup, or config path that would make `<!subteam^S...>` wake an agent. The linked config-only PR was closed unmerged after maintainer feedback favoring Slack API membership resolution, so there is no merged or shipped fix to close against.

## What This Changes

Add Slack user-group mention support so messages containing `<!subteam^S...>` can wake OpenClaw agents whose Slack bot user belongs to that user group.

## Best Possible Solution

Implement this in the Slack plugin, not core: parse `<!subteam^S...>` tokens, resolve each user group through Slack membership lookup when direct bot mention matching fails, treat the message as explicitly mentioned when the current Slack bot user is a member, and document the required scope and expected fallback/error behavior. Keep the closed config-only allowlist as prior art unless maintainers explicitly want it as a fallback.

## Reproduction Assessment

Yes. The issue provides a concrete live Slack reproduction, and the current source trace confirms why it fails: a `<!subteam^S...>` message does not match the `<@...>` direct mention preflight, so `wasMentioned` remains false and `requireMention` room routing skips dispatch.

## Solution Assessment

Unclear for the closed PR, clear for the issue direction. The config-only allowlist was not accepted, but the maintainer discussion and Slack API surface point to a narrower product fit: resolve real Slack user-group membership via `usergroups.users.list` and update Slack docs/manifest guidance.
