## Summary

Keep open. Current main still recognizes Slack explicit mentions only through direct `<@botId>` tokens or configured mention regexes, with no `<!subteam^...>` parsing or Slack user-group membership lookup. The issue discussion also has maintainer feedback closing the config-only PR in favor of an API-backed `usergroups.users.list` path, so this remains valid and bounded.

## What This Changes

Slack user-group mentions should wake eligible Slack agents when channel mention gating is enabled.

## Best Possible Solution

Implement this in the Slack plugin as user-group membership resolution: parse `<!subteam^S...>` tokens, query Slack `usergroups.users.list` with a bounded cache, invalidate on documented subteam events, treat the message as explicit only when the routed bot user is an actual member, and update setup/docs/scope diagnostics for `usergroups:read`.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a room message containing only `<!subteam^S...>` leaves `hasAnyMention=false`, `explicitlyMentioned=false`, and `wasMentioned=false`, so a Slack channel with `requireMention=true` reaches the `no-mention` skip path.

## Solution Assessment

Yes for the maintainer-preferred direction, not for the closed config-only PR. The API-backed path is the narrower maintainable fix because it follows actual Slack user-group membership and avoids duplicating group membership in OpenClaw config.
