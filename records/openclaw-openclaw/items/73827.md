## Summary

Keep open. Current main still treats only direct Slack `<@botUserId>` tokens, configured mention regexes, and existing implicit thread cases as mention sources; `<!subteam^S...>` user-group tokens are absent from runtime, config, docs, setup scopes, and tests. The linked same-author PR is closed unmerged, and maintainer discussion points to an API-backed membership check rather than the config-only allowlist proposed there.

## What This Changes

The issue asks the Slack plugin to treat Slack user-group mention tokens as explicit mentions for agents that are actual members of the mentioned user group.

## Best Possible Solution

Implement the fix in the Slack plugin’s mention-gating path: parse Slack subteam mention tokens, resolve each mentioned user group through Slack’s user-group membership API with bounded caching, mark the message explicit only when the current bot user is returned, and update Slack setup/docs/tests for the new scope and behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with only `<!subteam^S...>` in `message.text`, current main sets the direct Slack mention checks false, so a Slack room with `requireMention=true` reaches the `no-mention` skip path instead of dispatching.

## Solution Assessment

Yes, the maintainer-directed API-backed membership check is the best current path. The closed config-only allowlist PR is useful context but is not the best fix because it can wake bots that are not actual Slack user-group members and creates duplicate configuration outside Slack.
