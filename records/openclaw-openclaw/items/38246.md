## Summary

Current main still lacks the requested WhatsApp safeguard. Visible auto-replies are still delivered through inbound `reply`/`sendMedia` closures bound to the sender's original chat, and the current config/schema/docs only cover last-route pinning, allowlists, and explicit outbound sends, not a policy that blocks or forces explicit targets for non-owner DM auto-replies.

## Best Possible Solution

Keep this open and have maintainers choose an explicit WhatsApp plugin routing policy for allowlisted non-owner DMs. The implementation likely belongs in the visible WhatsApp auto-reply path before `deliverWebReply` sends payloads, with typed config/schema/docs and regression tests for owner DMs, allowlisted non-owner DMs, and explicit `message` tool sends that should remain intentional.

## Risks / Open Questions

- The report describes a confidentiality leak class in WhatsApp DM sessions, so the fix should be treated as security-sensitive routing policy, not a cosmetic option.
