## Summary

Keep open. Current main still has adjacent config-clobber and service-env hardening, but the specific cross-HOME update safeguard remains missing: update/service discovery starts from the updater process environment, systemd unit lookup is HOME-keyed, and forced service refresh still drops prior HOME and OPENCLAW control paths instead of preserving, migrating, warning, or blocking when the installed service root differs.

## What This Changes

The issue asks OpenClaw update/service refresh to preserve, migrate, or warn before abandoning an existing gateway config and state root when the service HOME changes during an update.

## Best Possible Solution

Add a Linux update/service-refresh guard that discovers the previously installed gateway service root before rewriting it, compares HOME, OPENCLAW_HOME, OPENCLAW_STATE_DIR, and OPENCLAW_CONFIG_PATH against the updater environment, then preserves the prior root when safe or emits a clear migration/blocking warning. Cover the dedicated-user service refreshed from a different HOME case with update, daemon install, and systemd regression tests; keep the GHCR version-tag request as separate rollback/pinning work.
