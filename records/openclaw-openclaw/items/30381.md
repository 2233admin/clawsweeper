## Summary

Keep this issue open. Current main still validates the OpenAI request-body `model` through `resolveOpenAiCompatModelOverride` and returns 400 for non-OpenClaw model ids before chat-completions dispatch, even when `x-openclaw-agent-id` has already selected the agent. The latest release tree has the same behavior, and the issue comments describe an active production compatibility need.

## Best Possible Solution

Implement a narrow Gateway compatibility fix: when `x-openclaw-agent-id` or its compatibility alias resolves the agent for an OpenAI-compatible chat-completions request, do not reject a generic OpenAI request-body `model`; route to the selected agent and continue using the agent's configured model unless `x-openclaw-model` is explicitly supplied and allowed. Add regression coverage, make an intentional `/v1/responses` choice, and update the public OpenAI HTTP docs.
