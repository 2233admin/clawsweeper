## Summary

Keep this PR open, but do not merge it as-is. The reported invalid-input timing issue is still visible on current main, and the added null/undefined regression cases are useful, but the patch reintroduces SHA-256 hashing of candidate credentials in shared auth code that current main just moved away from for CodeQL remediation.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs to empty strings, compare SHA-256 digests with `timingSafeEqual` plus bitwise type guards, and add null/undefined-versus-empty regression cases.

## Best Possible Solution

Keep the current padded timing-safe byte comparison as the equality primitive, add a reviewed invalid-input path that performs comparable dummy work before returning false, and retain the new null/undefined-versus-empty regression cases so normalized-empty false positives cannot slip in.

## Reproduction Assessment

Yes. Current main visibly returns before buffer allocation and `timingSafeEqual` for non-string inputs, while the PR diff visibly normalizes invalid inputs and executes digest comparison; no timing benchmark was run because this was a read-only review.

## Solution Assessment

No. The PR's goal is valid and the edge-case tests are useful, but replacing the CodeQL-remediated padded byte comparison with SHA-256 digest comparison plus a suppression is not the safest maintainable fix.
