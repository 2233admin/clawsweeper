## Summary

Keep this PR open. Current main still has the invalid-input fast-fail that the PR is trying to harden, but the latest PR head reintroduces SHA-256 hashing of candidate credentials, which current main just removed for CodeQL remediation and which GitHub Advanced Security now flags on the PR.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs to empty strings, compare SHA-256 digests with `timingSafeEqual` plus bitwise type guards, and add null/undefined-versus-empty regression cases.

## Best Possible Solution

Keep the current padded timing-safe byte comparison as the equality primitive, add a reviewed invalid-input path that performs comparable dummy work before returning false, and retain the new null/undefined-versus-empty regression cases so normalized-empty false positives cannot slip in.

## Reproduction Assessment

Yes. Current main visibly returns before buffer allocation and `timingSafeEqual` for non-string inputs, while the PR head visibly hashes candidate credentials and has a fresh CodeQL review comment on that line; no timing benchmark was run because this was a read-only review.

## Solution Assessment

No. The PR's goal is valid, and the added edge-case tests are useful, but replacing the CodeQL-remediated padded byte comparison with SHA-256 digest comparison is not the safest maintainable fix.
