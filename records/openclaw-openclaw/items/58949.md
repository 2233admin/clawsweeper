## Summary

Keep open. The reported invalid-input fast path still exists on current main, so the PR is not obsolete, but the current branch is not mergeable: it calls an unimported `createHash` and it reintroduces SHA-256 digest comparison that current main just removed for CodeQL remediation.

## What This Changes

The PR changes `safeEqualSecret` to normalize null/undefined inputs to empty strings, compare SHA-256 digests with `timingSafeEqual`, require both original inputs to be strings, and add null/empty regression cases.

## Best Possible Solution

Keep the current padded timing-safe byte comparison as the equality primitive, add a reviewed invalid-input path that performs comparable dummy work before returning false, and retain the new null/undefined-versus-empty regression cases so normalized-empty false positives cannot slip in.

## Reproduction Assessment

Yes. Current main visibly returns before buffer allocation and `timingSafeEqual` for non-string inputs, and the PR head visibly calls an unimported `createHash`; no timing benchmark or local build was run because this was a read-only review.

## Solution Assessment

No. Normalizing invalid inputs is directionally relevant, but replacing the current padded byte comparison with digest comparison conflicts with the CodeQL-remediated implementation and the branch currently does not compile because `createHash` is unresolved.
