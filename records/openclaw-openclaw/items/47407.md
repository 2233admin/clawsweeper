## Summary

Keep open. Current main still lacks an edit-time warning for disallowed cron model overrides, so the PR is not obsolete, but the proposed warning still reads local CLI config and does not match the gateway/runtime model-resolution path closely enough to merge safely.

## What This Changes

The PR adds a non-blocking warning to `cron edit --model` by loading config/catalog in the CLI and checking the parsed model against the allowed model set before sending the cron update.

## Best Possible Solution

Put the warning or validation in the gateway/runtime context, or expose a gateway-side validation RPC, and have it reuse `resolveCronModelSelection`/`resolveAllowedModelRef` semantics with focused coverage for disallowed refs, aliases, shorthand refs, configured defaults, auth-profile suffixes, and remote-gateway usage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Validate against the target gateway config:** `src/cli/cron-cli/register.cron-edit.ts:218-219`
  - body: `cron edit` can target a remote gateway via `--url` and `--token`, but this warning reads `loadConfig()` in the local CLI process. When local and remote `agents.defaults.models` differ, the command can warn for a model that will run or stay quiet for one the gateway will reject, so the diagnostic does not reliably predict execution behavior.
  - confidence: 0.93
- **[P2] Reuse runtime model resolution before warning:** `src/cli/cron-cli/register.cron-edit.ts:220-227`
  - body: The warning builds the allowlist with hard-coded defaults and compares a `parseModelRef(model, DEFAULT_PROVIDER)` key, while cron execution resolves configured defaults, aliases, bare refs, and auth-profile suffixes through `resolveAllowedModelRef`. Valid cron model inputs can therefore get a false warning, and the check can disagree with the runtime error path it is trying to preview.
  - confidence: 0.94
