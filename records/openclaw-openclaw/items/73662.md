## Summary

Keep open. The PR still does not fix the default `openclaw status` path reported in #73525: the default text command keeps live channel status disabled unless `--all` or `--deep` is set, and the new merge only runs when a live `channels.status` payload exists. The added overview-level test does not exercise that command path.

## What This Changes

The PR adds a gateway-status merge helper for the Channels table, wires it into status overview collection, adds helper and overview tests, exports the helper through the status runtime module, and adds a changelog entry.

## Best Possible Solution

Revise the PR so plain `openclaw status` gets populated channel rows while preserving the fast-status contract, then add a command-level `scanStatus({ json: false })` regression test for the #73525 scenario. The maintainable path is either a bounded gateway enrichment only when read-only rows are missing, or a cheap setup-discovery fallback that does not reload heavyweight channel runtimes.

## Reproduction Assessment

Yes. #73525 gives concrete default-command steps, and source inspection at PR head shows `scanStatus({ json: false })` still disables the live payload required by the new merge helper.

## Solution Assessment

No. The helper is a reasonable augmentation for callers that already fetch `channels.status`, but the current wiring does not solve the reported plain `openclaw status` behavior; the safer fix must cover that command path directly without undoing the fast-status work.
