## Summary

Keep open. The PR is a useful implementation candidate for the linked Feishu hot-loop report, and current main does not already provide a Feishu doctor repair path, but the latest head has a blocking data-safety bug: `doctor --fix` can delete/archive healthy Feishu direct-session entries when the only finding is corrupt Feishu local JSON state.

## What This Changes

The PR adds a Feishu doctor adapter, wires it into the Feishu channel registration, and adds tests for detecting and repairing corrupt Feishu local state plus Feishu-scoped direct session entries.

## Best Possible Solution

Revise the PR so repair targets are derived from findings: corrupt Feishu JSON should rebuild only the Feishu state directory, while session transcript findings should remove/archive only the specific session keys and artifacts referenced by those findings. Add a regression for corrupt local JSON plus a healthy Feishu session, then rerun the focused Feishu doctor/session tests and changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Limit repairs to sessions with transcript findings:** `extensions/feishu/src/doctor.ts:587`
  - body: `entriesByStore` is built from every inspected Feishu session entry, not from the entries that produced session findings. When the only finding is corrupt Feishu JSON under the local state directory, `doctor --fix` still deletes all healthy Feishu direct-session keys and archives their transcript artifacts. Build the repair set from `missing-session-transcript`, `invalid-session-transcript`, and `blank-user-message-run` findings instead.
  - confidence: 0.95
