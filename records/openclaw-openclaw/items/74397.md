## Summary

Keep open. Current main still lacks a Feishu doctor repair path, but the PR head has a concrete blocking data-safety bug: repair mode derives removals from every inspected Feishu direct-session entry, so an unrelated corrupt local JSON finding can remove healthy Feishu conversation state.

## What This Changes

The PR adds a Feishu doctor adapter, wires it into the Feishu channel plugin, and adds tests for detecting and repairing corrupt Feishu local state plus Feishu-scoped session entries.

## Best Possible Solution

Revise the PR so corrupt-state-json findings rebuild only the Feishu state directory, while session removals and artifact archiving are derived only from missing-session-transcript, invalid-session-transcript, and blank-user-message-run findings. Add a regression for corrupt local JSON plus a healthy Feishu direct session, then rerun focused Feishu doctor/session tests and the changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P2] Derive session repairs from findings:** `extensions/feishu/src/doctor.ts:587`
  - body: `entriesByStore` is populated from every inspected Feishu session entry, but repair runs for any Feishu finding. When the only finding is corrupt local JSON, `doctor --fix` will still delete healthy Feishu direct-session keys and archive their artifacts. Build the repair set from the session-specific findings instead.
  - confidence: 0.95
