## Summary

Keep open: current main still has the misleading fallback that this PR targets, but the proposed branch has a correctness bug. Its new missing-provider check only looks for generic provider auth/config, so configured media model entries, CLI entries, or image-model auto resolution can be misreported as “No provider configured” after an attempted run returns empty output.

## What This Changes

The PR adds CLI fallback logic and tests so `infer image describe` and `infer audio transcribe` emit capability-specific missing-provider messages when media understanding returns no text and no configured provider is detected.

## Best Possible Solution

Keep the PR open and change the CLI decision to use the media-understanding resolver’s actual entry/attempt state, or expose that decision through the runtime result, so the missing-provider message is emitted only when no entry was resolved or attempted. Preserve the existing empty-output errors whenever a configured provider, CLI entry, shared media model, image default, or auto-detected local entry actually ran.

## Reproduction Assessment

Yes. The linked issue gives a deterministic current-main no-provider reproduction, and the PR-specific regression can be reproduced with a `tools.media.audio.models` or `tools.media.image.models` CLI entry that returns empty output while no generic provider auth/config exists.

## Solution Assessment

No. The UX direction is right, but this PR’s helper duplicates only part of the media resolver instead of using the same resolved-entry or attempt-state signal as the runner, so it is not the narrowest maintainable fix yet.
