## Summary

Keep open: the requested CLI UX fix is valid and current main still has the old path-focused fallback, but this PR's helper only checks generic provider auth/config and misses first-class media-understanding entries. That can mislabel configured CLI, shared media, per-capability media, or image-model attempts as "no provider configured" when they return empty output.

## What This Changes

The PR adds CLI fallback logic and tests so `infer image describe` and `infer audio transcribe` emit missing-provider configuration errors separately from configured-provider empty-output errors.

## Best Possible Solution

Keep the PR open and update the CLI fallback decision to use the media-understanding runner's resolved entries, attempted-entry metadata, or an equivalent shared resolver signal. The missing-configuration message should appear only when nothing was resolved or attempted, while configured provider/CLI/model attempts that produce empty text should keep the existing empty-result error.

## Reproduction Assessment

Yes. The linked issue gives deterministic no-provider steps, and current main source still throws the old path-focused error when media understanding returns no text. The PR-specific regression is reproducible by configuring `tools.media.audio.models`, `tools.media.image.models`, shared `tools.media.models`, or `agents.defaults.imageModel` with an entry that is attempted but returns empty output while no generic provider auth is present.

## Solution Assessment

No. The UX direction is right, but the helper duplicates only part of media-understanding resolution rather than using the same entry/attempt state that actually drove the run. A narrower maintainable fix would base the CLI message on resolver output or runtime decision metadata.
