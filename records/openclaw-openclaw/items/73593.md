## Summary

Keep open. Current main still has the misleading `infer audio transcribe` / `infer image describe` empty-result errors, and this PR targets the right user-facing bug, but the proposed fallback check is too narrow and can mislabel configured media attempts as missing-provider configuration. The earlier Greptile comment treated the branch as safe, while the existing ClawSweeper comment flagged the same P2; this pass confirms the P2 against current main and the PR diff.

## What This Changes

The PR adds CLI missing-provider error handling for image description and audio transcription plus regression tests for missing-provider versus configured empty-output cases.

## Best Possible Solution

Return or expose the media-understanding decision/attempt state to the CLI, then emit the missing-provider configuration message only when no entries were resolved or attempted. Keep the empty-output error when any configured provider, configured CLI entry, shared media model, default image model, or auto-detected local CLI was actually tried, and add the required changelog entry for the user-facing fix.

## Reproduction Assessment

Yes. The current-main bug is source-confirmed by the CLI throwing path-focused empty-result errors whenever media understanding returns no text, and the PR regression follows from a configured CLI/media model returning empty output while the new helper still reports no configured provider.

## Solution Assessment

No. The UX direction is right, but the proposed helper duplicates only generic provider-auth detection instead of using the runner's resolved-entry or attempt-state signal, so it is not the narrowest maintainable fix.
