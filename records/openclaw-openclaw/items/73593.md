## Summary

Keep open: the reported current-main behavior is real, but the proposed fix has a correctness hole because the new configured-provider check ignores configured media model and CLI entry paths, so it can emit a false missing-provider error after a configured media-understanding entry returns empty output.

## What This Changes

The PR updates the capability CLI and tests so `infer image describe` and `infer audio transcribe` distinguish missing media provider configuration from configured-provider empty-output errors.

## Best Possible Solution

Keep the PR open and change the fallback decision to align with the media-understanding resolver: only emit the missing-configuration error when no configured or auto-resolved entry was available, and preserve the empty-result error whenever a configured provider or CLI entry was actually attempted and produced no text.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Account for configured media entries before throwing:** `src/cli/capability-cli.ts:493-502`
  - body: `hasConfiguredMediaUnderstandingProvider` only looks for registry providers with generic auth/config. The media resolver also treats `tools.media.<capability>.models`, shared `tools.media.models`, and CLI entries as configured media-understanding paths; when one of those entries runs and returns empty text, this helper returns false and the CLI now reports “No ... provider configured” instead of preserving the intended empty-output error. Please include configured media entries or the runner attempt state before taking the missing-provider branch.
