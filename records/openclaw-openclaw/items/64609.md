## Summary

Keep open. Current main still has the reported Telegram config-resolution split: multi-account account merging drops channel-level `groups`, while prompt/topic/skills resolution only reads that merged snapshot; the mention policy still falls back to channel-level groups. The only linked implementation PR, #64650, is closed and unmerged, and no open replacement was found in the targeted search.

## What This Changes

The issue requests consistent Telegram multi-account handling so channel-level group/topic `systemPrompt`, `skills`, and topic config are not silently ignored when `requireMention` from the same block applies.

## Best Possible Solution

Add a Telegram-owned prompt-context fallback that keeps routing, access control, `enabled`, `allowFrom`, and `groupPolicy` based on account-scoped resolved config, while letting group/topic `systemPrompt` and `skills` fall back from `accounts.<id>.groups` to channel-level `groups` when no account-level group override exists. Preserve the current #30673 account merge tests and add a regression test for two accounts plus channel-level group/topic prompts and skills.

## Repair Work Prompt

Fix the Telegram multi-account group/topic prompt fallback reported in this issue. Current main preserves #30673 by making `mergeTelegramAccountConfig` drop channel-level `groups` when multiple Telegram accounts exist, but `resolveTelegramGroupConfig` in `extensions/telegram/src/bot-core.ts` then reads only the merged account snapshot, so channel-level group/topic `systemPrompt` and `skills` are missing while `requireMention` still falls back through `extensions/telegram/src/group-policy.ts`.

Implement a narrow Telegram-owned prompt/skills fallback: account-scoped group/topic config should win; when an account has no group override, channel-level `channels.telegram.groups` may feed prompt-context fields (`systemPrompt`, `skills`, topic prompt/skills) for inbound message context and native command context. Do not make `resolveTelegramAccount()` or `mergeTelegramAccountConfig()` inherit channel-level groups in multi-account mode. Do not broaden fallback for routing, delivery, `enabled`, `allowFrom`, `groupPolicy`, or other access-control fields.

Use #64650 as source context if helpful, but adapt to current main filenames (`bot-core.ts` now owns `resolveTelegramGroupConfig`). Add a regression test in `extensions/telegram/src/bot.create-telegram-bot.test.ts` with two configured Telegram accounts plus channel-level group/topic `systemPrompt` and `skills`, asserting `GroupSystemPrompt` and `skillFilter` reach the dispatch path. Keep the existing #30673 tests in `extensions/telegram/src/accounts.test.ts` passing. Add a single-line changelog entry under the active version because this is a user-visible Telegram bug fix, crediting `Thanks @briqt` and crediting #64650's contributor if source work is reused. Avoid unrelated docs or behavior changes unless maintainers intentionally choose a doctor/docs warning instead of support.
