## Summary

Keep this issue open. Current main still has the reported failure class: Feishu wiki `space_id` is schema-declared as a string, but the plugin casts runtime params and forwards `p.space_id` directly to the Lark API, while the likely GLM/Z.AI path uses the shared OpenAI-compatible tool-call machinery rather than the Ollama-only unsafe-integer parser. The discussion correctly identifies classic JavaScript precision loss when a 19-digit ID is emitted as a bare JSON number.

## What This Changes

The issue requests that Feishu/Lark knowledge-base `space_id` values remain exact when used by Feishu document/wiki tools instead of being rounded or rewritten during tool calls.

## Best Possible Solution

Keep the issue open and add a focused Feishu/Lark guard that rejects or recovers unsafe numeric `space_id`/`target_space_id` inputs before any Lark API call, plus stronger schema and skill wording that tells models to pass these IDs as quoted strings. If maintainers want full cross-provider protection, lift the existing Ollama unsafe-integer parser into a shared tool-argument parser with targeted OpenAI-compatible/Z.AI tests.
