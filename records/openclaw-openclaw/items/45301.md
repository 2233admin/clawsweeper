## Summary

Keep this issue open. Current main still has the reported Feishu/Lark wiki ID precision-loss path: wiki space IDs are documented and schema-declared as strings, but runtime tool parameters are cast and forwarded without rejecting numeric unsafe integer values that have already been rounded by JSON parsing.

## What This Changes

The issue reports that Feishu/Lark wiki space identifiers returned by the plugin are later rounded or modified when passed into document/wiki tools.

## Best Possible Solution

Feishu wiki tools should treat wiki space identifiers as opaque strings end to end: accept quoted string IDs unchanged, reject number-typed `space_id` and `target_space_id` values before any Lark API call with a clear precision-loss message, strengthen schema/skill wording so models quote long IDs, and add focused regression coverage.
