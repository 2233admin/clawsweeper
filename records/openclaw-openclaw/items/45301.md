## Summary

Keep open. Current main still has the Feishu/Lark wiki ID precision-loss path: the tool schema says wiki space IDs are strings, but runtime params are cast and forwarded to the Lark client without rejecting already-rounded number values. The discussion correctly identified the JavaScript unsafe-integer failure mode, and the latest release/current-main comparison does not show a fix in the affected wiki runtime, schema, or skill files.

## What This Changes

The issue reports that user-provided Feishu/Lark wiki knowledge-base `space_id` values are changed before document or wiki-node tool calls reach Lark.

## Best Possible Solution

Feishu wiki tools should treat wiki space identifiers as opaque strings end to end: keep quoted strings exact, reject number-typed `space_id` and `target_space_id` with a clear precision-loss message, strengthen schema/skill guidance so models quote long IDs, and add focused Feishu plugin regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `JSON.parse` rounds the reported bare numeric IDs, and current main forwards those rounded values from Feishu wiki tool params into Lark request fields without a type guard.

## Solution Assessment

Yes. The narrowest maintainable fix is Feishu-owned validation plus schema/skill wording; a global big-integer parser would have broader blast radius and still cannot recover digits after a provider or client has already rounded a bare number.
