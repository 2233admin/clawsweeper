## Summary

Keep this PR open for maintainer review. Current main still does not surface Feishu WebSocket/webhook lifecycle state into gateway health snapshots, so the work is not implemented on main. This PR has useful Feishu monitor work, but it is incomplete against the normal gateway path because `channel.ts` still does not pass `ctx.setStatus` into `monitorFeishuProvider`; it is also stale against the newer Feishu WebSocket reconnect/heartbeat code on main. Related PR #39489 covers the gateway sink and event/activity side more directly but is also open and currently not mergeable, so the best path is consolidation or a rebase rather than auto-closing this PR.

## Best Possible Solution

Keep the PR open and ask for maintainer consolidation with #39489 or a focused rebase. The best implementation should wire Feishu channel startup to a gateway status sink, propagate that sink through `monitorFeishuProvider`, `monitorSingleAccount`, and both transports, report WebSocket/webhook connected/disconnected/reconnect/error state plus inbound event/activity timestamps, preserve the current #72411 WebSocket heartbeat/reconnect behavior, and include targeted Feishu monitor/channel tests plus a user-facing changelog entry if this lands.
