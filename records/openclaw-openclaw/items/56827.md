## Summary

Keep open. Current main has adjacent session maintenance, but the reported failure mode is still plausible: doctor still loads the whole `sessions.json`, the session-store path still clones large store objects, and neither doctor nor `openclaw sessions cleanup` has a stale `sessions.json.*.tmp` sweep. The issue comment's manual `find ... -delete` workaround is useful context, but it is not a shipped fix.

## Best Possible Solution

Keep the issue open. A good maintainer fix would add stale `sessions.json.*.tmp` detection and repair to doctor or `openclaw sessions cleanup`, avoid large full-store clones in doctor/session maintenance paths, and add regression coverage for an oversized `sessions.json` plus stale temp files. The SQLite-backed metadata store can remain a broader storage-design follow-up.
