## Summary

Keep open. Current main still exposes the embedding helper surface through the memory-core-named SDK subpath only; no generic embeddings subpath exists, and the linked implementation PR was closed unmerged. This is an OpenClaw SDK boundary question, not ClawHub-only plugin work, so it needs maintainer/API-owner direction before implementation.

## What This Changes

Request a stable `openclaw/plugin-sdk/embeddings` SDK subpath for memory embedding provider factories, registry access, types, and batch helpers independent of the `memory-core` plugin lifecycle while preserving the existing memory-core-named path.

## Best Possible Solution

Add an owner-approved, additive `openclaw/plugin-sdk/embeddings` facade that re-exports the agreed embedding contracts/helpers, keep `plugin-sdk/memory-core-host-engine-embeddings` backward compatible, document the preferred path, update package exports/entrypoint metadata/API baseline, and cover the contract with focused SDK/provider tests plus a changelog entry.

## Reproduction Assessment

Yes. Static inspection of current main is enough: `src/plugin-sdk/embeddings.ts`, the package export, SDK entrypoint metadata, and docs row for `openclaw/plugin-sdk/embeddings` are all absent while the memory-core-named facade remains present.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive subpath appears maintainable and consistent with the SDK boundary rules, but the exact public contract and deprecation scope should be approved before a fix PR is queued.
