## Summary

Keep open. Current main still exposes the embedding helper surface only through the memory-core-named SDK path, while the requested generic `openclaw/plugin-sdk/embeddings` style public contract is absent from source, package exports, docs, and the SDK entrypoint registry. This is a real SDK boundary decision rather than stale cleanup, and the related implementation PR was closed unmerged rather than superseded.

## What This Changes

Expose a stable public SDK subpath for embedding provider APIs so third-party memory-slot plugins can use the existing embedding stack without importing memory-core-owned paths.

## Best Possible Solution

Add an owner-approved, backward-compatible SDK subpath such as `openclaw/plugin-sdk/embeddings` that re-exports only the intended embedding helpers, types, and registry access; keep the existing memory-core-named path working, mark migration guidance clearly in docs, and update package exports, SDK entrypoint metadata, API baseline/export checks, and focused contract tests together.

## Reproduction Assessment

Yes. This is reproducible by static inspection on current main: there is no `src/plugin-sdk/embeddings.ts`, package export, entrypoint registry item, or docs entry for a generic embeddings SDK subpath, while the old memory-core-named facade exists.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive SDK subpath appears consistent with the plugin boundary rules, but maintainers should approve the stable path name and exported contract before an automated fix PR.
