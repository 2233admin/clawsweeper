## Summary

Keep open. Current main still lacks a generic stable `openclaw/plugin-sdk/embeddings` subpath, while the public export map, SDK entrypoint registry, docs, and bundled plugin imports continue to expose the embedding helpers through the memory-core-named facade. The related implementation PR was closed unmerged, so it does not supersede this issue.

## What This Changes

Request a stable public plugin SDK subpath for embedding provider APIs so third-party memory-slot plugins can use OpenClaw embedding infrastructure without importing memory-core-owned SDK paths.

## Best Possible Solution

Add an owner-approved, additive SDK subpath such as `openclaw/plugin-sdk/embeddings`, keep `plugin-sdk/memory-core-host-engine-embeddings` backward compatible, and update the package exports, SDK entrypoint metadata, docs, API/export checks, and focused contract coverage together.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the gap: there is no generic `plugin-sdk/embeddings` source file, package export, entrypoint registry entry, or SDK docs row, while the memory-core-named facade remains present.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive subpath appears consistent with the plugin boundary rules, but the stable name and exported contract should be approved before implementation is queued.
