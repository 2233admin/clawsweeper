## Summary

Keep open. Current main has a public memory-core-named embedding facade and cold provider-resolution support, but it still does not provide the requested plugin-neutral `openclaw/plugin-sdk/embeddings` contract or deprecate the memory-core-named import; the related implementation PR was closed unmerged, so this remains a valid SDK/API-shape follow-up.

## What This Changes

Expose memory embedding provider helpers through a stable plugin-neutral SDK subpath so third-party memory-slot plugins do not depend on memory-core-named imports or internals.

## Best Possible Solution

Add an owner-approved, backwards-compatible SDK facade such as `openclaw/plugin-sdk/embeddings` that re-exports the existing memory-host embedding contracts and provider registry helpers, keep the memory-core-named path working as compatibility, and update package exports, SDK entrypoint metadata, docs, API/export checks, and focused contract coverage together.

## Reproduction Assessment

Yes. Static inspection on current main is enough: `openclaw/plugin-sdk/embeddings` has no source file, package export, SDK entrypoint registry item, or docs entry, so that import path would not resolve.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive alias appears maintainable, but #67242 was closed unmerged as too broad, and the public contract shape should be approved before a repair PR changes the SDK surface.
