## Summary

Keep open. Current main still exposes the embedding helpers only through the memory-core-named SDK subpath, with no generic `openclaw/plugin-sdk/embeddings` export or docs entry. The request is a valid public SDK contract decision, and the related implementation PR was closed unmerged rather than superseded by main.

## What This Changes

Expose memory embedding provider contracts, registry access, and helper functions through a stable plugin SDK subpath that does not depend on the memory-core plugin being active.

## Best Possible Solution

Add an owner-approved, backwards-compatible SDK subpath such as `openclaw/plugin-sdk/embeddings` that re-exports the existing memory embedding contracts/helpers, keep `memory-core-host-engine-embeddings` working, and update package exports, SDK entrypoint metadata, docs, API baseline/export checks, and focused contract tests together.

## Reproduction Assessment

Yes. The missing public surface is reproducible by static inspection on current main: there is no `src/plugin-sdk/embeddings.ts`, package export, entrypoint registry item, or docs entry for `openclaw/plugin-sdk/embeddings`.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive SDK subpath appears consistent with the plugin boundary rules, but maintainers should decide the stable name and exported contract before an automated fix PR.
