## Summary

Keep open. Current main still exposes the embedding helper surface only through the memory-core-named SDK subpath, with no generic `openclaw/plugin-sdk/embeddings` export or docs entry; the related implementation PR was closed unmerged, so this remains an API-owner contract decision rather than conservative cleanup.

## What This Changes

Request a stable documented plugin SDK subpath for memory embedding helpers and provider registry access that third-party memory-slot plugins can use without depending on the memory-core-named facade.

## Best Possible Solution

Add an owner-approved, backwards-compatible SDK subpath such as `openclaw/plugin-sdk/embeddings`, keep `memory-core-host-engine-embeddings` working, and update package exports, SDK entrypoint metadata, docs, API baseline/export checks, and focused contract tests together.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the missing public surface: there is no `src/plugin-sdk/embeddings.ts`, package export, SDK entrypoint registry item, or docs entry for `openclaw/plugin-sdk/embeddings`.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive subpath appears consistent with the SDK boundary rules, but maintainers should approve the stable name and exported contract before an automated fix PR.
