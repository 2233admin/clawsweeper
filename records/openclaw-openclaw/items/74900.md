## Summary

Keep open. Current main still lacks a generic `openclaw/plugin-sdk/embeddings` subpath, while the documented package export and SDK registry continue to expose the helper surface only under the memory-core-named path. The linked implementation PR was closed unmerged, and this is a real public SDK contract decision that needs maintainer/API-owner shape approval before repair automation.

## What This Changes

Expose embedding provider helpers through a stable, documented plugin SDK subpath that does not depend on memory-core ownership, while preserving the existing memory-core-named path for compatibility.

## Best Possible Solution

Add an owner-approved, backwards-compatible SDK subpath such as `openclaw/plugin-sdk/embeddings` that re-exports the existing embedding contracts/helpers, keep `memory-core-host-engine-embeddings` working, and update package exports, SDK entrypoint metadata, docs, API baselines, and focused contract tests in one narrow change.

## Reproduction Assessment

Yes. Static inspection against current main reproduces the missing public surface: there is no generic embeddings SDK subpath, and the only documented/exported helper path remains `memory-core-host-engine-embeddings`.

## Solution Assessment

Unclear pending SDK owner input. A narrow additive SDK subpath appears maintainable and consistent with plugin boundary rules, but the stable name and exported contract should be approved before automation opens a replacement PR.
