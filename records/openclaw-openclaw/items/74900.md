## Summary

Keep open. Current main still exposes the embedding helper/registry surface only through the memory-core-named SDK subpath, while the requested generic `openclaw/plugin-sdk/embeddings` public contract is absent from package exports, SDK entrypoint metadata, docs, and source files. The related implementation PR was closed unmerged, so this remains an SDK/API-owner decision rather than cleanup that can be closed.

## What This Changes

Request to expose memory embedding provider helpers and registry access through a stable, documented `openclaw/plugin-sdk/embeddings` subpath that works independently of memory-core being the selected memory plugin.

## Best Possible Solution

Add an owner-approved, backwards-compatible SDK subpath such as `openclaw/plugin-sdk/embeddings`, keep `openclaw/plugin-sdk/memory-core-host-engine-embeddings` working, and update package exports, SDK entrypoint metadata, docs, API baselines, and focused contract tests together.

## Reproduction Assessment

Yes. Static inspection on current main reproduces the missing public surface: there is no generic `plugin-sdk/embeddings` source file, package export, SDK entrypoint, or docs entry.

## Solution Assessment

Unclear pending SDK-owner input. A narrow additive SDK subpath appears maintainable and consistent with the plugin boundary rules, but the exact stable contract should be approved before an automated repair PR.
