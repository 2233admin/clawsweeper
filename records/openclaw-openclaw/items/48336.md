## Summary

Keep this PR open. Current main still does not implement the requested general per-IP gateway HTTP request limiter; it only has failed-auth rate limiting and a separate control-plane write throttle. The PR is a plausible implementation candidate, but it is not mergeable as-is because the submitted integration keys by the raw socket address before trusted-proxy client-IP resolution, which defeats the stated reverse-proxy use case. The overlapping public-network hardening work in #44884 and the linked security issue #20373 are still open, so they do not prove this work has shipped.

## Best Possible Solution

Keep the PR open but do not merge it as submitted. The best path is to revise or explicitly consolidate it with #44884: add a public `gateway.requestRateLimit` config/schema/docs surface if this is meant to be operator-configurable, instantiate and dispose the limiter in gateway startup, enforce it after `resolveRequestClientIp(req, trustedProxies, allowRealIpFallback)`, and add focused tests for direct clients, trusted reverse proxies, loopback proxy sources, and 429 `Retry-After` behavior. If maintainers choose #44884 as the canonical implementation, close this PR only after that path is landed or after an explicit maintainer consolidation decision.

## Risks / Open Questions

- Merging the PR as-is would leave common reverse-proxy deployments unprotected or create cross-user 429s because it keys limits by the TCP peer address instead of the resolved client IP.

- Closing this PR as implemented would be inaccurate because current main has no general request-rate-limit module, config surface, docs, or HTTP enforcement path.
