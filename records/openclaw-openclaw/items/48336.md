## Summary

Keep this PR open. Current main still does not ship a general per-IP Gateway HTTP request limiter; it has failed-auth throttling and a separate control-plane write limiter only. The PR remains a plausible implementation candidate, but the submitted integration is incomplete for the stated public/reverse-proxy protection goal and the review comments correctly identify a proxy-IP bypass risk.

## Best Possible Solution

Keep this PR open for maintainer follow-up or explicit consolidation with #44884/#20373. The best implementation path is to add an operator-facing gateway.requestRateLimit config/schema/docs surface if maintainers want this feature, instantiate and dispose the limiter during gateway startup, enforce it after resolveRequestClientIp(req, trustedProxies, allowRealIpFallback), and add regression coverage for direct clients, trusted reverse proxies, loopback proxy sources, forwarded headers, 429 Retry-After behavior, and memory-bounding/pruning behavior.

## Risks / Open Questions

- Merging the PR as submitted would leave same-host reverse-proxy deployments effectively unprotected because the limiter keys by the proxy socket address before trusted-proxy client-IP resolution, and loopback is exempt by default.
- For non-loopback reverse proxies, the submitted integration can make unrelated clients share one limiter bucket and cause cross-user 429s.
- The PR adds an optional limiter parameter but, based on the provided changed-file list, does not add operator-facing config/schema/docs or startup instantiation/disposal, so the feature would not protect a normal gateway unless another caller wires it separately.
- The overlapping public-network hardening PR #44884 and linked security issue #20373 are still open in the provided context, so they do not prove this PR has been superseded by a landed implementation.
