## Summary

Keep open: current main still treats WhatsApp QR-login 408 failures as terminal, and the PR carries useful source work, but the proposed retry path is not merge-ready because it bypasses the existing socket-restart callback contract and can repeatedly churn sockets under persistent 408s. Related shipped WhatsApp work covers QR rotation, socket timing, and post-408 listener recovery, but does not supersede this QR-login retry change.

## What This Changes

The PR adds a WhatsApp QR-login 408 timeout branch that closes and recreates the socket in `waitForWebLogin`, plus a timed-out log branch in the WhatsApp session socket handler.

## Best Possible Solution

Implement 408/timedOut handling as a bounded restart path in the shared WhatsApp login waiter, preserving `authDir`, `isLegacyAuthDir`, socket timing, QR refresh callbacks, and active socket replacement state, with regression coverage for retry success, retry exhaustion, refreshed QR propagation, and unchanged 401/515 behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Preserve QR callbacks on 408 retries:** `extensions/whatsapp/src/login-qr.ts:287-289`
  - body: The 408 retry creates the replacement socket with only `authDir`. `createWaSocket` captures `onQr` when the socket is constructed, so a fresh QR emitted after the timeout will not update the active login state and users can keep seeing an expired QR. Reuse the shared restart path or pass the same QR/timing/replacement callbacks.
  - confidence: 0.91
- **[P2] Bound repeated 408 socket retries:** `extensions/whatsapp/src/login-qr.ts:283-298`
  - body: A persistent 408 re-enters this branch each time the replacement waiter fails until the outer wait deadline, repeatedly closing and opening sockets without a per-login cap. Track a bounded retry count, or handle 408 in the shared waiter with an explicit maximum, so network faults do not churn sockets and logs for the full wait window.
  - confidence: 0.86
