## Summary

Keep this issue open. Current main reduced the practical pain by raising the default run timeout to 48 hours, adding a model-stream idle watchdog, allowing CLI timeout overrides of 0, and emitting an explicit timeout payload instead of going silent. It still does not implement the requested activity-aware run timeout or split idle/max run timeout contract: the embedded runner keeps a single run-level wall-clock abort timer, and the public agent defaults schema still exposes only positive `timeoutSeconds`.

## Best Possible Solution

Keep this issue open. The best implementation is to add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, retain a separate absolute max runtime cap, expose the new config contract in schema/help/docs, and add focused tests proving active multi-tool runs are not aborted solely because the original wall-clock window elapsed.
