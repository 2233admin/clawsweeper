## Summary

Keep this issue open. Current main has useful mitigations, including a 48-hour default runtime, model-stream idle watchdogs, zero timeout support for some overrides, and explicit timeout error payloads, but the embedded run abort timer is still a single run-level wall-clock timer and is not refreshed by assistant or tool activity.

## Best Possible Solution

Add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute maximum runtime cap, expose both fields in config schema/help/docs, and add focused tests proving active multi-tool runs are not aborted solely because the original wall-clock window elapsed.

## Risks / Open Questions

- Operators who configure shorter agents.defaults.timeoutSeconds can still have active multi-tool runs aborted by elapsed wall-clock time rather than inactivity.
- A future fix should retain a separate absolute max runtime cap so slow continuous activity cannot keep a run alive indefinitely.
- Schema, generated config help, and docs need to change with the implementation so users can distinguish idle timeout from maximum runtime.
