## Summary

Keep this open. Current main has mitigations for the original 600s failure mode, including a 48-hour default agent timeout, explicit timeout error text, model-stream idle watchdogs, and compaction timeout recovery, but the run-level timer is still a wall-clock abort timer and there is no separate run idle/max timeout config refreshed by tool or assistant activity.

## Best Possible Solution

Add a run-level idle timeout refreshed by assistant stream activity and tool lifecycle completion, keep a separate absolute maximum runtime cap, expose both through config schema/help/docs with backward-compatible semantics for agents.defaults.timeoutSeconds, and add focused tests proving active multi-tool runs can exceed the idle window while truly idle or over-cap runs still abort.
