## Summary

Keep open. Current main still sends managed deep-dreaming promotions straight through the existing `MEMORY.md` apply path, while the separate dreaming storage/report surface remains only a phase-report destination. The linked storage-mode and raw-snippet reports are adjacent context, but they do not supersede the requested final-promotion review queue, and no open PR currently references this issue or a `promotionWriteMode`-style fix.

## What This Changes

Request an additive memory-core configuration or workflow that stages deep dreaming promotion output in a review queue before, instead of or in addition to, appending to `MEMORY.md`.

## Best Possible Solution

Keep the issue open until maintainers choose an additive final-promotion destination or approval contract, then implement it in memory-core with the existing direct `MEMORY.md` behavior preserved as an explicit mode and staged review artifacts available for approval or distillation before canonical long-term memory writes.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior is source-backed: managed deep dreaming calls `applyShortTermPromotions`, and that helper appends selected candidates to `MEMORY.md` without a review destination.

## Solution Assessment

Unclear until maintainers choose the workflow. The narrow maintainable direction is an explicit final-promotion destination or approval contract rather than expanding `dreaming.storage.mode` implicitly from phase/report storage to canonical memory writes.
