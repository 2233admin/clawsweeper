## Summary

Keep open. Current main still ranks deep Dreaming candidates and sends accepted promotions directly through the MEMORY.md append path, while the existing storage setting only controls phase/report artifacts and there is no promotion review/off/both destination contract.

## What This Changes

The issue requests a configurable deep-Dreaming promotion destination so promoted candidates can be staged for review before, instead of or in addition to, automatic MEMORY.md writes.

## Best Possible Solution

Keep this as an open memory-core design item, then add an additive promotion destination contract that preserves the current MEMORY.md default, can stage structured review artifacts, and updates schema, resolver, CLI/docs/tests, and any Dreams UI affordance consistently.

## Reproduction Assessment

Not applicable as a feature/design request. The current behavior is high-confidence from source and test coverage: managed deep Dreaming calls applyShortTermPromotions, and that helper writes accepted candidates to MEMORY.md.

## Solution Assessment

Unclear until maintainers choose the public workflow. The narrow maintainable path is a new final-promotion destination/approval contract, not overloading dreaming.storage.mode, because phase report storage and canonical memory approval are separate concerns.
