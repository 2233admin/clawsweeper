## Summary

Keep open. Current main still does not turn newly observed or migrated Telegram groups into directory entries or allowed reply targets, and the reported live migration log maps to a current no-op path when the old group ID was not already configured. The remaining fix is actionable, but it changes Telegram group discovery and authorization semantics, so it needs maintainer policy before implementation.

## What This Changes

The issue asks OpenClaw to discover and ingest Telegram group chats so a Telegram bot that handles direct messages also responds in allowed groups after privacy/admin setup.

## Best Possible Solution

Define the intended behavior for newly observed and migrated Telegram groups, then implement it in the Telegram plugin with explicit authorization boundaries. A maintainable end state would expose enough diagnostics or read-only observed-group information for users to configure the new `-100...` ID, while preserving explicit group and sender authorization before replies or tool-capable turns run.

## Reproduction Assessment

Yes. The issue provides live steps and logs, and those logs map directly to current main's migration handler plus the config-backed directory and group-policy gates.

## Solution Assessment

Unclear. Auto-discovering groups and enabling replies from observation alone is not the safest solution; the narrower maintainable direction is a policy-backed observed/migrated group flow with explicit authorization before response handling.
