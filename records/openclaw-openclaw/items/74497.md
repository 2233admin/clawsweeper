## Summary

Keep open. Current main still has Telegram group and supergroup ingress, but the reporter's migration log maps to the existing no-op path for an unconfigured old group ID, and Telegram group directory listing remains config-backed rather than live-discovered. The closed related report #74694 is folded into this same tracker, and newer comments add more live confirmations rather than evidence that current main fixed it.

## What This Changes

Telegram group chats should be discovered and ingested by the bot so group messages can trigger replies and appear in `openclaw directory groups list --channel telegram` while direct messages continue to work.

## Best Possible Solution

Define the intended Telegram behavior for newly observed and migrated groups, then implement that policy without weakening group or sender authorization. A maintainable endpoint would either keep directories config-backed with clearer diagnostics and doctor guidance for the new `-100...` ID, or add a read-only observed-groups surface that does not grant reply authorization by itself.

## Reproduction Assessment

Yes. The issue provides concrete live setup steps and logs, multiple users report the group-only failure, and the reported `No config found for old group ID` message maps directly to the current migration handler branch.

## Solution Assessment

Unclear. The requested outcome is valid, but auto-discovering and enabling Telegram groups is not obviously the safest fix; the narrow solution needs maintainer direction on observed-group listing, migrated-ID diagnostics, and reply authorization boundaries.
