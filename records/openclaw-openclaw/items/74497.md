## Summary

Keep open. Current main has Telegram group/supergroup ingress and exact configured-ID migration, but the reporter's fresh-group migration evidence still maps to the current no-op path when the old group ID is not already configured, and Telegram group directory listing remains config-backed rather than live-discovered. The closed related report #74694 appears to have been folded back into this issue, so this is the canonical open tracker.

## What This Changes

This issue requests that Telegram group and supergroup messages, including migrated new-group IDs, be discovered/listed and ingested/responded to when the bot is configured for groups.

## Best Possible Solution

Define Telegram's intended behavior for newly observed and migrated groups, then implement it without weakening group allowlists: either keep directories config-backed and add clear diagnostics/doctor guidance for the new `-100...` ID, or add a read-only observed-groups surface that does not grant reply authorization by itself. Runtime coverage should include migrated unconfigured groups, wildcard group config, configured old-ID migration, group ingress, directory output, and sender/mention policy boundaries.

## Reproduction Assessment

Yes. The issue provides concrete live steps and logs, and the reported `No config found for old group ID` message maps directly to the current migration handler branch; this review used source-level proof rather than running a live Telegram bot.

## Solution Assessment

Unclear. The requested end state is reasonable, but blindly auto-discovering and enabling Telegram groups is not the safest fix; the maintainable path is a policy-backed migration/discovery behavior that preserves explicit group and sender authorization.
