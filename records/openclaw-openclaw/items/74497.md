## Summary

Keep open. Current main still follows the reported Telegram group-only failure path: migration events are logged, but a newly observed old group ID with no existing config is not persisted or made discoverable, and Telegram group directories remain config-backed. The fix needs an explicit maintainer policy decision because observed group discovery and reply authorization affect which chats can trigger the bot.

## What This Changes

The issue asks OpenClaw to discover and ingest Telegram group chats, including freshly created groups that migrate to supergroup IDs, while direct messages already work.

## Best Possible Solution

Define Telegram’s intended behavior for newly observed and migrated groups, then implement it inside the Telegram plugin without granting reply authority from observation alone. A maintainable end state would either expose the new supergroup ID as a diagnostic/read-only directory result with explicit config guidance, or persist an observed/disabled entry that still requires explicit group and sender authorization before replies are enabled.

## Reproduction Assessment

Yes. The issue gives live setup steps and logs, and those logs map directly to the current `migrate_to_chat_id` handler plus the helper branch that does nothing when the old group ID was never configured.

## Solution Assessment

Unclear. Automatically discovering and enabling Telegram groups is not clearly the safest solution; the narrower maintainable path needs maintainer policy on observed groups, migration diagnostics, and reply authorization.
