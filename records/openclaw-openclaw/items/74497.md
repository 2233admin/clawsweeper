## Summary

Keep open. Current main still matches the reported Telegram group failure path: migration events are logged but do not create or authorize a new group entry when the old ID was never configured, Telegram group directory output remains config-backed, and the discussion includes multiple live confirmations that DMs work while groups fail.

## What This Changes

The issue asks OpenClaw to discover, ingest, and respond to Telegram group chats, including newly migrated supergroup IDs, while preserving working direct-message behavior.

## Best Possible Solution

Define the intended Telegram semantics for newly observed and migrated groups, then implement that in the Telegram plugin without granting reply authority from observation alone. A maintainable end state would either keep directories config-backed with clearer diagnostics for the new supergroup ID, or add a read-only observed-groups/doctor surface plus explicit config steps before replies are enabled.

## Reproduction Assessment

Yes. The issue gives concrete live setup steps and the migration/no-config log, and that log maps directly to the current migration handler and helper behavior on main; no Telegram token was present locally for an additional live run.

## Solution Assessment

Unclear. Automatically discovering and enabling Telegram groups from observed traffic is not clearly the safest fix; the narrow maintainable solution needs an explicit maintainer policy for observed/migrated groups, directory visibility, and sender authorization.
