## Summary

Keep open. Current main still has Telegram group/supergroup ingress and configured-ID migration, but the reporter's fresh-group migration log maps to the current no-op path for an unconfigured old group ID, and Telegram group directory listing remains config-backed rather than live-discovered. The closed related report #74694 appears to have been folded back into this issue, so this remains the canonical open tracker.

## What This Changes

The issue requests that Telegram group and supergroup chats, including migrated new-group IDs, be discovered/listed and ingested/responded to when the bot is configured for groups.

## Best Possible Solution

Define Telegram's intended behavior for newly observed and migrated groups, then implement it without weakening group and sender authorization. Either keep directories config-backed with clearer diagnostics/doctor guidance for the new `-100...` ID, or add a read-only observed-groups surface that does not grant reply authorization by itself, with focused coverage for migration, wildcard config, directory output, ingress, and policy boundaries.

## Reproduction Assessment

Yes. The issue provides concrete live steps and logs, and the reported `No config found for old group ID` message maps directly to the current migration handler branch; this review used source-level proof rather than running a live Telegram bot.

## Solution Assessment

Unclear. The requested end state is reasonable, but auto-discovering and enabling Telegram groups is not obviously the safest fix; the narrow maintainable solution needs an explicit policy-backed behavior for observed and migrated groups.
