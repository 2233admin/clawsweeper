## Summary

Keep open. Current main has Telegram group/supergroup ingress and a migration handler, but the reporter's fresh migration log matches the existing no-op path for unconfigured old group IDs, and Telegram directory groups are still config-backed rather than live-discovered.

## What This Changes

The issue asks for Telegram group chats, including migrated supergroups, to be discovered by `openclaw directory groups list --channel telegram` and ingested/replied to when direct messages already work.

## Best Possible Solution

Define Telegram's intended post-migration behavior explicitly: either keep directory groups config-backed and improve status/docs/doctor guidance for newly observed `-100...` IDs, or add a read-only observed-groups surface that never grants access by itself. Any runtime change should add targeted migration/directory/group-access coverage and preserve allowlist boundaries.
