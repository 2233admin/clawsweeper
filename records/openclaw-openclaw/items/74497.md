## Summary

Keep open. Current main has Telegram group ingress and explicit old-ID-to-new-ID migration support, but it still does not implement live Telegram group discovery, and the reporter's new migration log points to the existing path that only migrates already-configured old group IDs. The response failure still needs maintainer review with redacted config and verbose skip logs before changing discovery or config-write behavior.

## What This Changes

The issue requests that Telegram group chats be discovered by `openclaw directory groups list --channel telegram` and that Telegram group messages be ingested/responded to after bot setup and group-to-supergroup migration.

## Best Possible Solution

Define the Telegram product boundary first: either keep group IDs explicit and improve migration/status/doctor guidance for discovering and adding the new supergroup id, or add a read-only seen-groups/live-directory surface that does not silently authorize groups. Then reproduce the post-migration normal-message path on current main with redacted config and verbose logs before changing ingress or writeback policy.
