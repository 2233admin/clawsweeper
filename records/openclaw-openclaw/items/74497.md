## Summary

Keep open. Current main still has the Telegram group ingress path, but the reporter's migrated fresh-group log maps to a no-op migration branch for unconfigured old IDs, the group directory remains config-backed, and multiple commenters confirm group-only failures after normal Telegram setup steps. This is actionable, but the fix needs maintainer policy because group discovery and migrated-group handling affect authorization.

## What This Changes

The issue requests that Telegram group chats be discovered and ingested/responded to like direct Telegram messages, including freshly migrated supergroup IDs.

## Best Possible Solution

Define the intended Telegram behavior for newly observed and migrated groups, then implement that policy in the Telegram plugin. A maintainable end state would either keep directories config-backed with clearer diagnostics for the new `-100...` ID, or add a read-only observed-groups surface plus an explicit authorization step before replies are allowed.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes: the issue provides concrete live steps and logs, and the reported migration/no-config messages map directly to the current migration handler and helper behavior on main.

## Solution Assessment

Is this the best way to solve the issue? Unclear: treating observed group traffic as discovered or authorized automatically is not obviously safe, so the best fix needs an explicit maintainer policy for observed/migrated groups before code changes.
