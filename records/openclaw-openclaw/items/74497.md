## Summary

Keep open. Current main still has Telegram group/supergroup ingress, but the reporter's fresh migration log matches the existing no-op path for unconfigured old group IDs, Telegram group directories are config-backed, and wildcard/unmentioned group setups can still be blocked by sender-authorization policy unless explicitly opened. This needs maintainer policy review rather than conservative cleanup.

## What This Changes

The issue requests that Telegram group chats be discovered, ingested, and answered by the OpenClaw bot, with `openclaw directory groups list --channel telegram` showing the group when direct messages already work.

## Best Possible Solution

Define Telegram's intended post-migration and discovery behavior explicitly: either keep directory groups config-backed and improve status/docs/doctor guidance for newly observed `-100...` IDs, or add a read-only observed-groups surface that never grants access by itself. Any runtime change should preserve explicit group allowlist and sender-authorization boundaries.
