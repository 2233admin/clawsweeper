## Summary

Keep open. Current main still does not strip U+0000/NUL from iMessage echo-cache text keys: it only adds a later adjacent cleanup for leading FEFF/FFFD/FFFE/FFFF markers, and JavaScript trim still leaves NUL-prefixed echoes unmatched. The PR remains a useful narrow fix, but it is now stale/unmergeable against current main and should be rebased or replaced with an equivalent patch that composes with the merged leading-marker behavior.

## What This Changes

The PR updates iMessage sent-message echo-cache text normalization to remove NUL bytes before CRLF/trim normalization and adds regression coverage for single and repeated leading NUL-prefixed echoed text.

## Best Possible Solution

Rebase this PR onto current main and compose it with the existing leading attributedBody marker cleanup by adding U+0000 handling for the reported leading NUL prefix, plus regression coverage for one and multiple leading NUL bytes. Preserve the current leading-only/interior-marker tests unless maintainers intentionally choose broader NUL normalization.
