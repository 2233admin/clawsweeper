## Summary

Keep this issue open. Current main still passes arbitrary Google Chat thread strings into `sendGoogleChatMessage`, and several outbound paths can still supply raw message/session/action IDs instead of a `spaces/{space}/threads/{thread}` resource. Related PR #67055 is useful context for the `replyToMode: "off"` part of the problem, but it is still open and does not cover API-boundary validation, actions, attached-results sends, or `[[reply_to_current]]` message-resource leakage.

## Best Possible Solution

Keep this issue open until the Google Chat plugin has a shared thread-routing helper used by monitor delivery, streamed/block replies, attached-results text/media sends, and message actions. The helper should preserve valid `spaces/{space}/threads/{thread}` resources, prefer inbound `message.thread.name` over `message.name` when `[[reply_to_current]]` resolves to a message resource, omit invalid raw IDs before constructing the Chat API request, honor `replyToMode: "off"`, and add focused regression tests for direct replies, streamed/block replies, media, attached results, and action sends.
