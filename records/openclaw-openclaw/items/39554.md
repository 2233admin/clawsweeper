## Summary

Keep this issue open. Current main still sends any truthy Google Chat thread string directly as `body.thread.name`, and multiple outbound paths can still feed raw `threadId`, `replyToId`, `replyTo`, or `[[reply_to_current]]`-derived message ids into that API boundary. Related PR #67055 is open and useful, but it targets only part of the `replyToMode: "off"` propagation problem and does not replace the broader thread-resource validation/normalization work reported here.

## Best Possible Solution

Keep this issue open until the Google Chat plugin owns a shared thread-routing helper used by monitor delivery, streamed/block replies, attached-results sends, media sends, and message actions. That helper should preserve valid `spaces/{space}/threads/{thread}` resources, prefer the inbound thread resource over `message.name` when `[[reply_to_current]]` resolves to a message resource, omit invalid raw ids before constructing the Chat API request, honor `replyToMode: "off"`, and add regression tests for direct replies, streamed/block replies, media, attached results, and action sends.
