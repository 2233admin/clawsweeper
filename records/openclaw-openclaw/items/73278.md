## Summary

Keep open. Current main still contains the broad Codex runtime and bundled Discord skill guidance that can steer an agent toward the message tool, while Discord autoThread final-response routing already targets the created thread. Three open, unmerged PRs currently reference this issue, and the eventual fix should also respect the newer message_tool_only guild/channel visibility policy.

## What This Changes

Prevent Discord autoThread channel mentions from producing both a parent-channel message.send reply and a normal final response in the created thread.

## Best Possible Solution

Review and consolidate the open fix PRs into one policy-aware change: allow final assistant delivery when OpenClaw will automatically deliver to the current conversation or created thread, but preserve message-tool guidance for message_tool_only guild/channel turns and explicit sideband messaging.

## Reproduction Assessment

Yes. The issue gives concrete Discord autoThread mention steps, and current main still has the prompt inputs plus the final-delivery thread routing needed for the duplicate path when automatic visible delivery is active.

## Solution Assessment

Yes, with a caveat. Prompt/skill clarification is the narrowest likely fix area, but it should be conditional on sourceReplyDeliveryMode rather than unconditionally banning message.send for normal Discord channel replies.
