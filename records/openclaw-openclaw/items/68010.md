## Summary

Keep this PR open. Current main still has the eager compatibility export the PR is trying to fix, but the branch needs maintainer review because the regression test does not actually prove post-import laziness and the provider env-var hardening is security-sensitive and incomplete.

## What This Changes

The PR aliases `PROVIDER_ENV_API_KEY_CANDIDATES` to the lazy provider auth env-var proxy, adds a compat lazy-loading test, and adds provider env-var candidate validation/copying hardening in `src/secrets/provider-env-vars.ts`.

## Best Possible Solution

Keep the PR open as the implementation candidate for the OpenRouter lazy-loading bug, then rebase it onto current main, keep the compat export delegated to the lazy proxy, fix the lazy regression test so it asserts after module evaluation and before property access, and either split or explicitly approve the provider env-var security hardening with targeted coverage.

## Security Review

Status: needs_attention

Summary: The diff does not touch CI, dependencies, or publishing, but it changes provider env-secret resolution and leaves concrete hardening gaps that need maintainer review.

Concerns:

- **[medium] Close the descriptor mutation bypass:** `src/secrets/provider-env-vars.ts:203`
  - body: PR head returns defensive copies from the proxy `get` trap, but `getOwnPropertyDescriptor` still exposes the cached array as `value`. Callers can mutate that descriptor value and poison the shared env-var candidate cache after validation.
  - confidence: 0.88
- **[medium] Decide workspace env-var trust policy explicitly:** `src/secrets/provider-env-vars.ts:48`
  - body: `shouldUsePluginProviderEnvVars` still includes workspace plugin `providerAuthEnvVars` by default unless callers pass `includeUntrustedWorkspacePlugins: false`. Because `resolveEnvApiKey` reads those env names from `process.env`, the regex and small denylist are not enough to settle the exfiltration policy raised in review.
  - confidence: 0.78
