## Summary

Keep this PR open. Current main still unconditionally serializes Discord inbound jobs through the bundled Discord plugin's per-session `KeyedAsyncQueue`, so the PR's intended steer/steer-backlog bypass has not landed. The PR remains useful, but it targets the old `src/discord` layout and needs rebase/adaptation to `extensions/discord`, including the session-override concern already raised in review.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rebase to the current bundled Discord plugin. The best fix is a queue-mode-aware Discord inbound worker path that bypasses the outer per-session `KeyedAsyncQueue` for `steer` and `steer-backlog`, preserves serialization for non-steer modes, honors session-level queue overrides where possible, stays within the plugin SDK boundary, and adds focused regression tests for bypass and non-bypass behavior.

## Risks / Open Questions

- The PR cannot be merged mechanically because it targets the old `src/discord` layout; current main uses `extensions/discord`.
- A config-only bypass can violate per-session `/queue` overrides because `sessionEntry.queueMode` has higher precedence than channel/global config.
- A rebased fix must not import core auto-reply internals from the Discord plugin; if early queue-mode resolution is needed, use or add an SDK-safe seam.
- Bypassing the outer worker for steer modes changes same-session concurrency, so regression coverage should include steer, steer-backlog, non-steer serialization, lifecycle/deactivation, replay dedupe, and timeout behavior.
