## Summary

Keep this PR open. Current main still sends every Discord inbound job through the bundled Discord plugin's per-session KeyedAsyncQueue before processDiscordMessage can reach the inner queue-mode logic, so the intended steer and steer-backlog bypass has not landed. The PR remains useful, but it targets the removed src/discord layout and needs a rebase to extensions/discord plus an SDK-safe way to honor session queue overrides.

## Best Possible Solution

Keep this PR open for maintainer review and ask for a rebase onto extensions/discord. The best implementation is a queue-mode-aware Discord inbound worker that bypasses the outer per-session KeyedAsyncQueue for steer and steer-backlog, preserves serialization for collect/followup/interrupt where appropriate, honors session-level queue overrides, and uses an existing or new plugin-SDK-safe queue settings helper instead of importing core auto-reply internals from the bundled plugin.

## Risks / Open Questions

- The PR cannot be merged mechanically because it targets the old src/discord layout rather than extensions/discord.
- A config-only bypass can be behaviorally wrong when a per-session /queue override requests a different mode.
- Bypassing the outer worker changes same-session concurrency, so the rebased fix needs focused coverage for steer, steer-backlog, non-steer serialization, lifecycle/deactivation, replay dedupe, and timeout behavior.
- Security review: the provided diff only changes Discord runtime/test files and does not touch workflows, lockfiles, package publishing, dependency sources, or scripts; the main review concern is behavioral concurrency plus avoiding a core-internal import when rebasing to the plugin layout.
