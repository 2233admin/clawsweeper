## Summary

Keep open. Current main still emits a macOS LaunchAgent with unconditional `KeepAlive=true` and a one-second throttle, tests assert that policy, and an open linked PR references this issue with closing syntax, so this issue should stay open until that PR is reviewed, revised, merged, or closed.

## What This Changes

Request a macOS launchd Gateway supervision path that avoids rapid crash loops after startup failures so users can inspect and debug the root cause.

## Best Possible Solution

Keep the issue open and settle the macOS launchd policy deliberately: either revise and land the linked PR if it truly covers startup-crash debugging, or replace it with a targeted backoff/debug/stop-after-threshold design in the LaunchAgent/runtime supervision path, with service-audit guidance, docs updates, and focused launchd plist tests while preserving explicit `openclaw gateway restart` behavior.

## Reproduction Assessment

Yes. The policy-level reproduction is high confidence from current source and tests: a generated LaunchAgent has unconditional `KeepAlive=true` with `ThrottleInterval=1`, and the test suite asserts no `SuccessfulExit` condition; I did not run a live macOS launchd crash loop in this read-only review.

## Solution Assessment

No for current main, and unclear for the open linked PR. `SuccessfulExit=false` is worth reviewing, but launchd documentation says it still restarts non-zero exits, so a startup-crash report likely needs crash-loop/backoff or debug-stop semantics rather than only clean-exit handling.
