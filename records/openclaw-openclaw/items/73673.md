## Summary

Keep open. Current main still generates a macOS LaunchAgent with unconditional `KeepAlive=true` and a one-second throttle, while tests, service audit, and docs continue to treat crash relaunch as expected behavior. The issue also has an open closing PR, so this should remain the tracker until that PR is reviewed, narrowed, replaced, or merged.

## What This Changes

The issue requests a way to avoid rapid macOS launchd restart loops after gateway startup failures so users can inspect and debug the failure.

## Best Possible Solution

Add a targeted macOS launchd early-failure strategy that preserves reliable explicit `openclaw gateway restart` behavior while giving users a debuggable stop/backoff path for repeated startup failures, with matching updates to plist generation, service audit, tests, and macOS Gateway docs.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: install or generate the LaunchAgent, trigger an invalid gateway startup configuration, and launchd will see a nonzero process exit while the generated plist still has `KeepAlive=true` with a one-second throttle.

## Solution Assessment

No. Current main has not solved the issue, and changing only to `KeepAlive` with `SuccessfulExit=false` would still restart nonzero startup crashes; the safer solution needs an explicit repeated-failure/debug policy around the launchd lifecycle.
