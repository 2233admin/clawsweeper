## Summary

Keep open. Current main still emits a macOS LaunchAgent with unconditional `KeepAlive=true` and a one-second throttle, while tests, service audit, and docs all preserve crash relaunch as the expected launchd behavior. The issue is also paired with open PR #73849 using closing syntax, so the issue should remain open until that PR is reviewed, narrowed, or replaced.

## What This Changes

The issue asks OpenClaw to avoid rapid macOS launchd Gateway restart loops after startup failures so users can inspect and debug the underlying failure.

## Best Possible Solution

Keep this issue open and settle a targeted macOS launchd crash-loop/debug strategy. The maintainable end state should preserve reliable explicit `gateway restart` behavior while adding backoff, stop-after-threshold, or a debug/diagnostic mode for repeated early startup failures, with matching updates in launchd plist generation, restart runtime flow, service audit, tests, and macOS Gateway docs.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: install/generate the LaunchAgent and trigger a startup/config failure; current code emits `KeepAlive=true` with a one-second throttle, and startup failures exit nonzero, so launchd relaunches rapidly. I did not perform a live macOS launchd run in this Linux checkout.

## Solution Assessment

No. Current main has not solved the issue, and the linked PR is not clearly the best complete fix because `SuccessfulExit=false` still restarts nonzero startup crashes and conflicts with clean-exit supervised restart behavior unless the restart path is redesigned too.
