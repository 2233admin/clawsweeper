## What This Changes

The issue requests that Terminal Hatch local TUI show an active loader and avoid appearing stuck or ultra slow during onboarding post-connect initialization.

## Best Possible Solution

Land PR #75120 if review confirms the TUI state transitions remain correct, or merge an equivalent narrow fix that marks post-connect `refreshAgents()` and `loadHistory()` work as busy. Any separate performance work on those initialization calls should be tracked independently.

Yes for the UI-state bug: `openclaw onboard` -> Terminal Hatch launches local embedded TUI, and current `client.onConnected` renders `local ready` while awaiting `refreshAgents()` and `loadHistory()` before `sendMessage()` can start the normal loader. The exact duration is unclear without reporter-machine logs or profiling.

## Solution Assessment
