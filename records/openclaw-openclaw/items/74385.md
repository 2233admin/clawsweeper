## Summary

Keep open. Current main still shows the reported Terminal Hatch idle-looking gap: the onboarding path launches local embedded TUI, and TUI connection setup awaits agent refresh and history load before the first auto-message while the busy loader only covers sending, waiting, streaming, and running. PR #75120 is open, unmerged, and explicitly targets this issue, so the right cleanup path is normal PR review rather than closing this issue now.

## What This Changes

Terminal Hatch should show a responsive loader or busy status while the local embedded TUI completes post-connect initialization instead of appearing idle or stuck.

## Best Possible Solution

Land PR #75120 if review confirms its busy-state transition is correct; otherwise replace it with an equally narrow TUI change that marks post-connect refresh/history work as busy without clobbering real sending, streaming, reconnect, or error states.

## Reproduction Assessment

Yes for the UI-state bug: current source shows Terminal Hatch enters local embedded TUI, then waits on refreshAgents() and loadHistory() while status can remain local ready/idle. The exact one-minute slowdown is unclear without the reporter's machine, logs, or profile data.

## Solution Assessment

Yes, keeping this issue open behind PR #75120 is the best current path. The proposed fix boundary is narrow and focused on reusing the existing TUI busy loader during initialization, while any deeper performance work can be handled separately after profiling.
