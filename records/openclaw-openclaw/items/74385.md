## Summary

Keep open: current main still shows the Terminal Hatch TUI as `local ready | idle` while post-connect initialization runs, and PR #75120 is open, unmerged, and explicitly targets this issue with closing syntax. The issue should stay open until that PR is reviewed and merged or replaced.

## What This Changes

The issue reports that `openclaw onboard` Terminal Hatch feels stuck or very slow because the TUI does not show a loader while startup work runs before the first message.

## Best Possible Solution

Review and land PR #75120 if its initialization busy-state approach is sound; otherwise replace it with the same narrow TUI fix that marks post-connect refresh/history work as busy without clobbering real send, wait, stream, or run states.

## Reproduction Assessment

Yes for the UI-state bug: the source path for `openclaw onboard` Terminal Hatch sets `local ready` before awaiting `refreshAgents()` and `loadHistory()`, and the loader only starts later in `sendMessage`. The exact reported duration still needs machine-specific logs or profiling.

## Solution Assessment

Yes, the linked direction is the narrowest maintainable solution: reuse the existing TUI busy-status renderer for the post-connect initialization phase and cover it with focused TUI tests. Closing this issue before the linked PR merges would be premature.
