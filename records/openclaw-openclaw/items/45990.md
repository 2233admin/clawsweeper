## Summary

Keep this PR open. Current main still emits raw `llm_input` once before the embedded prompt and raw `llm_output` once after the attempt with aggregate output/usage; assistant stream `message_start`/`message_end` handlers do not emit those raw hooks per round. The related open issue #64268 tracks the same product need, but it is not a merged implementation, and this PR remains a plausible implementation candidate with unresolved public hook-contract questions.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The best path is to align it with #64268, define the public hook contract explicitly, add a stable per-call identifier such as `roundIndex` or `callId`, normalize per-call usage, preserve existing final aggregate semantics or document a migration, keep raw prompt/response access behind `allowConversationAccess`, and add regression coverage for multi-round success plus failures before streaming starts.

## Risks / Open Questions

- The PR changes observable `llm_output` payload semantics from final aggregate data to per-round data; plugins that compute cost or transcripts from `usage` and `assistantTexts` could break without a migration path or explicit new fields.
- Moving `llm_input` emission to `message_start` can miss failures that happen before the first stream event, reducing diagnostics for auth, rate-limit, or network failures before streaming begins.
- A revised raw per-round implementation must preserve the existing `allowConversationAccess` boundary for non-bundled plugins and should avoid broadening raw prompt/response access accidentally.
- The product/API shape is still unsettled: maintainers need to decide whether to change `llm_input`/`llm_output`, add a new per-call hook such as `llm_call`, or emit both per-call and final aggregate events.
