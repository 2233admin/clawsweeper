## Summary

Keep this PR open. Current main still emits raw `llm_input` before the embedded prompt and raw `llm_output` after the attempt with aggregate usage/output, while assistant message stream handlers only record usage and do not emit those hooks per round. The related open issue #64268 tracks the same product need, but it is not a merged implementation; this PR remains a plausible implementation candidate with unresolved hook-contract risks from the review discussion.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The best path is to align it with #64268, define the public contract explicitly, add `roundIndex` or `callId`, normalize per-call usage, preserve the existing final aggregate semantics or provide a documented migration, keep raw prompt/response access behind `allowConversationAccess`, and add regression tests for multi-round success plus prompt failures before streaming starts.

## Risks / Open Questions

- The PR's proposed `llm_output` behavior changes observable plugin payload semantics from final aggregate data to per-round data; cost and transcript plugins could break without a migration path or explicit new fields.
- Emitting raw conversation hooks from stream events could miss failures before the first stream event unless input emission is handled before prompt execution or otherwise covered.
- Any revised implementation must preserve the existing conversation-access gate for raw `llm_input`/`llm_output` data and avoid broadening access for non-bundled plugins.
- The product/API shape is still unsettled: maintainers need to decide whether this should be per-call `llm_input`/`llm_output`, a new `llm_call` hook, or dual aggregate plus per-call events.
