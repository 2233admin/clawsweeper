## What This Changes

The PR adds an acpx process reaper for OpenClaw-generated Codex ACP wrappers and plugin-runtime-deps processes, wires it into runtime cancel/close and service startup, hardens generated ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land a scoped acpx-plugin fix after maintainer review and green required checks, keep the ownership checks limited to OpenClaw-generated wrappers and staged plugin-runtime-deps commands, and make unsupported startup process listing quiet or explicitly skipped.

## Reproduction Assessment

Yes. Current main reproduces the cleanup gap by inspection: cancel/close do not use stored pid metadata, startup has no stale orphan reaper, and docs still prescribe manual stale process cleanup. I did not run live process tests because this review is read-only.

## Solution Assessment

Mostly yes. The acpx plugin is the right owner boundary, and the pinned acpx 0.6.1 SessionRecord contract supports the agentCommand and pid fields used by the patch; the safer refinement is to guard or quiet the Unix process-listing startup reaper on unsupported hosts.

## Review Findings

Overall correctness: patch is correct

Overall confidence: 0.84

Full review comments:

- **[P3] Skip startup reaping where Unix ps is unavailable:** `extensions/acpx/src/service.ts:220-222`
  - body: The startup path now calls the stale-process reaper unconditionally, and the default lister shells out to `ps -axo`. Guard this path or make unsupported process listing a quiet no-op.
