## Summary

Keep open. Current main still lacks acpx recorded process-tree cleanup, and this PR is a plausible plugin-boundary fix, but the latest head still has concrete service/runtime cleanup defects that can leave the target orphaned processes behind or create noisy startup warnings.

## What This Changes

The PR adds an acpx process reaper, wires startup/cancel/close cleanup for OpenClaw-owned Codex ACP wrapper and plugin-runtime-deps process trees, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the cleanup inside the acpx plugin boundary, make startup reaping match the generated wrapper root from the service state directory, preserve owned recorded-root cleanup when process listing is unavailable, make unsupported startup scans quietly no-op, and cover those cases with focused tests before normal CI review.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure acpx with a custom plugin stateDir, start the service so Codex wrappers are generated under ctx.stateDir/acpx, then leave a stale PPID=1 generated wrapper there; the PR head filters startup reaping against pluginConfig.stateDir/acpx and misses it.

## Solution Assessment

No, not as-is. The overall direction and ownership boundary are maintainable, but the current patch should use the actual generated wrapper root and keep a recorded-root kill fallback before it reliably solves the orphan cleanup problem.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the generated wrapper state dir for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: Startup writes Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this passes `pluginConfig.stateDir` into the reaper. With a custom acpx `stateDir`, stale generated wrappers under the service state dir are skipped; pass `ctx.stateDir` or the explicit wrapper root instead.
  - confidence: 0.94
- **[P3] Preserve recorded-root cleanup when listing fails:** `extensions/acpx/src/runtime.ts:510-512`
  - body: When process listing throws, this path stores an empty process snapshot. `cleanupOpenClawOwnedAcpxProcessTree` can validate `rootCommand`, but it then walks an empty tree and sends no signal to the recorded PID; add a direct owned-root fallback for cancel/close cleanup.
  - confidence: 0.86
- **[P3] Quiet unsupported startup process scans:** `extensions/acpx/src/service.ts:220-231`
  - body: Startup now always invokes the Unix `ps` scanner and logs thrown failures as warnings. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.8
