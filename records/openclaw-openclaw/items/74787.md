## Summary

Keep open. Current main still lacks the acpx process-tree cleanup this PR targets, and the PR remains useful, but the latest head still has a blocking startup reaper path mismatch plus two smaller cleanup robustness issues before it should merge.

## What This Changes

The PR adds an acpx process reaper, wires startup and cancel/close cleanup for OpenClaw-owned Codex ACP wrapper and plugin-runtime-deps process trees, hardens generated ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the cleanup inside the acpx plugin boundary, make startup reaping use the actual generated wrapper root, preserve a direct recorded-root kill fallback when process listing is unavailable, and make unsupported process scans a quiet no-op with focused regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure acpx with a custom plugin stateDir, start the service so wrappers are generated under ctx.stateDir/acpx, then leave a stale PPID=1 generated wrapper there; the PR filters startup reaping against pluginConfig.stateDir/acpx and misses it. A second source-level path is process-list failure during cancel/close, which leaves an empty snapshot and sends no signal to the recorded PID.

## Solution Assessment

No, not as-is. The overall acpx-owned cleanup direction is maintainable, but the current patch should use the generated wrapper root and keep a recorded-root fallback before it reliably solves the orphan cleanup problem.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Use the generated wrapper root for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: Startup generates Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this filters stale-wrapper ownership with `pluginConfig.stateDir`. With a custom acpx `stateDir`, stale generated wrappers under the service state directory are skipped; pass `ctx.stateDir` or an explicit wrapper root instead.
  - confidence: 0.94
- **[P3] Keep a recorded-root kill fallback when ps fails:** `extensions/acpx/src/runtime.ts:510-512`
  - body: When process listing throws, this stores an empty snapshot. `cleanupOpenClawOwnedAcpxProcessTree` can validate the recorded `rootCommand`, but it walks an empty tree and sends no signal to the recorded PID, so cancel/close cleanup is lost on hosts where listing is unavailable.
  - confidence: 0.86
- **[P3] Avoid warning on unsupported startup scans:** `extensions/acpx/src/service.ts:221-222`
  - body: Startup now always invokes the Unix `ps` scanner and logs failures as warnings. On Windows or hosts without that scanner, every acpx startup can warn for a best-effort cleanup path; gate unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.8
