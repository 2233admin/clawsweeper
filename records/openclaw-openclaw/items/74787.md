## Summary

Keep open. The PR is a plausible acpx-owned fix for orphaned Codex ACP process trees, but the latest head still has a blocking startup ownership bug: it filters generated wrapper processes with the configurable acpx session-store directory instead of the service state directory where those wrappers are generated. Current main also lacks the cleanup behavior, so the work is not obsolete.

## What This Changes

The PR adds acpx process cleanup for OpenClaw-generated Codex/Claude ACP wrappers and plugin-runtime-deps processes on startup, cancel, and close, hardens generated wrappers, and updates acpx tests, docs, and the changelog.

## Best Possible Solution

Keep the fix inside the acpx plugin boundary, pass the generated wrapper root derived from ctx.stateDir into startup reaping, make unsupported process enumeration a quiet no-op, and then rerun the focused acpx tests plus the changed gate before maintainer review.

## Reproduction Assessment

Yes. Current main reproduces the cleanup gap by source inspection, and the PR-specific startup bug reproduces with a custom acpx stateDir because wrappers are generated under ctx.stateDir/acpx while the PR filters startup reaping with pluginConfig.stateDir.

## Solution Assessment

No, not as-is. The acpx plugin is the right owner boundary and recorded pid cleanup is a reasonable approach, but the startup ownership root and unsupported-host behavior need to be corrected before this is the narrowest maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the generated wrapper root for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: The wrapper is generated under ctx.stateDir/acpx, but startup reaping filters wrapper ownership with pluginConfig.stateDir. With a custom acpx session-store stateDir, stale OpenClaw-generated wrappers no longer match the configured ownership path and remain orphaned. Pass ctx.stateDir or the generated wrapper root to the reaper instead.
  - confidence: 0.9
- **[P3] Skip unsupported startup process scans quietly:** `extensions/acpx/src/service.ts:220-232`
  - body: Startup calls the stale-process reaper unconditionally, and the default lister shells out to ps -axo. On Windows or minimal hosts without ps, every acpx service start logs a warning even though PPID=1 orphan scanning is Unix-specific. Guard this path or make unsupported process listing return an empty result without warning.
  - confidence: 0.82
