## Summary

Keep open. Current main still lacks acpx process-tree cleanup, and this is an active contributor implementation PR with a plausible, scoped fix; the remaining action is maintainer review, final CI, and a small non-blocking portability follow-up rather than conservative closure.

## What This Changes

The PR adds an acpx process reaper for OpenClaw-generated Codex ACP wrappers and plugin-runtime-deps processes, wires cleanup into cancel/close and startup, hardens generated ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Land a scoped acpx-plugin fix after maintainer review and green required checks, keep ownership checks limited to OpenClaw-generated wrappers and staged plugin-runtime-deps commands, and make unsupported process-listing hosts quiet or explicitly skipped.

## Reproduction Assessment

Yes. Current main reproduces the cleanup gap by inspection: acpx cancel/close do not use stored pid metadata, startup has no stale orphan reaper, and the docs still prescribe manual stale process cleanup. I did not run live process tests because this review is read-only.

## Solution Assessment

Mostly yes. The acpx plugin is the right owner boundary, and the pinned acpx 0.6.1 SessionRecord contract provides the agentCommand and pid fields used by the patch; the safer refinement is to guard or quiet the Unix process-listing startup reaper on unsupported hosts.

## Review Findings

Overall correctness: patch is correct

- **[P3] Skip the Unix startup reaper on unsupported hosts:** `extensions/acpx/src/service.ts:219-221`

Guard this path or make unsupported process listing a quiet no-op.
