## Summary

Keep open. The PR targets a real current-main acpx orphan-process cleanup gap, but the latest head still has a blocking startup ownership-path mismatch: OpenClaw generates wrappers under the service state directory while the new startup reaper filters with the configurable acpx session-store directory.

## What This Changes

The PR adds acpx process-reaping code, wires recorded pid cleanup into runtime cancel/close, reaps stale OpenClaw-owned acpx/Codex ACP orphans on service startup, hardens generated Codex/Claude ACP wrappers, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the fix in the acpx plugin boundary, pass the generated wrapper state root to startup ownership checks, keep cancel/close cleanup scoped to pid records whose command is OpenClaw-owned, make unsupported process enumeration a quiet no-op, and rerun the focused acpx tests plus the changed gate before maintainer review.

## Reproduction Assessment

Yes. Current main reproduces the cleanup gap by source inspection: runtime cancel/close do not read pid metadata, service startup has no stale orphan reaper, and the docs still describe manual stale process cleanup; the PR-specific startup bug reproduces with a custom acpx stateDir because wrappers are generated under ctx.stateDir/acpx while the PR filters with pluginConfig.stateDir.

## Solution Assessment

No, not as-is. The acpx plugin is the right owner boundary and the acpx 0.6.1 session record contract supports pid-based cleanup, but startup reaping should use the generated wrapper root and unsupported Unix process scans should not warn on every startup.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Use the generated wrapper state dir for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: The wrapper is generated under `ctx.stateDir/acpx`, but startup reaping filters wrapper ownership with `pluginConfig.stateDir`. With a custom acpx session-store `stateDir`, stale OpenClaw-generated wrappers no longer match `${stateDir}/acpx/codex-acp-wrapper.mjs` and remain orphaned. Pass the service state dir/generated wrapper root instead of the session-store directory here.
  - confidence: 0.9
- **[P3] Skip unsupported startup process scans quietly:** `extensions/acpx/src/service.ts:221-231`
  - body: Startup calls the stale-process reaper unconditionally, and the default lister shells out to `ps -axo`. On Windows or minimal hosts without `ps`, every acpx service start logs a warning even though PPID=1 orphan scanning is Unix-specific. Guard this path or make unsupported process listing return an empty result without warning.
  - confidence: 0.82
