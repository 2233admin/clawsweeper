## Summary

Keep open. Current main still lacks acpx process-tree cleanup, so the PR addresses a real orphan-process problem, but the proposed patch has a blocking startup state-dir mismatch plus smaller cleanup-platform gaps that should be fixed before merge.

## What This Changes

The PR adds acpx-owned stale process reaping for generated Codex/Claude ACP wrappers and plugin-runtime-deps process trees, cleans recorded harness trees on cancel/close, hardens generated wrapper signal handling, and updates tests, docs, and the changelog.

## Best Possible Solution

Keep the cleanup inside the acpx plugin boundary, pass the generated wrapper root or ctx.stateDir into startup reaping, add a direct recorded-root kill fallback when process listing is unavailable, make unsupported startup scans quiet, and keep the broader parent-reset child ACP cleanup tracked separately unless this PR explicitly expands to cover it.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure acpx with a custom plugin stateDir, start the service so wrappers are generated under ctx.stateDir/acpx, then leave a stale PPID=1 generated wrapper there; the PR head filters startup reaping against pluginConfig.stateDir/acpx and skips it.

## Solution Assessment

No, not as-is. The acpx-owned cleanup direction is maintainable, but the current diff should use the actual generated wrapper state root and preserve a recorded-PID fallback before it reliably solves the orphan cleanup problem.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the generated wrapper state dir for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: Startup writes Codex/Claude ACP wrappers under `ctx.stateDir/acpx`, but this call filters orphan wrappers against `pluginConfig.stateDir`. If acpx config uses a custom stateDir, stale generated wrappers under `ctx.stateDir/acpx` are skipped on startup; pass `ctx.stateDir` or the wrapper root into the reaper.
  - confidence: 0.93
- **[P3] Add a direct-root fallback when process listing fails:** `extensions/acpx/src/runtime.ts:531`
  - body: `resolveCleanupContextFromRecord` turns process-list failures into an empty list, and this line passes that empty snapshot to cleanup. The cleanup can verify `rootCommand`, but `collectProcessTree([])` returns no root, so cancel/close sends no signal despite having an owned recorded PID.
  - confidence: 0.84
- **[P3] Skip unsupported startup process scans quietly:** `extensions/acpx/src/service.ts:221-222`
  - body: Startup always invokes the Unix `ps` scanner through the reaper and logs a warning on failure. On Windows or hosts without that scan, every acpx startup can emit an avoidable best-effort cleanup warning; guard unsupported platforms or make unsupported listing return an empty result.
  - confidence: 0.79
