## Summary

Keep open. The latest PR head still has a blocking startup reaping mismatch: generated ACP wrappers are created under the plugin service state directory, but the new startup reaper filters against the acpx session-store state directory, so common/custom layouts can leave the target orphaned wrappers untouched.

## What This Changes

The PR adds acpx-owned stale process reaping, generated Codex/Claude ACP wrapper signal handling, cancel/close process-tree cleanup, docs, tests, and a changelog entry.

## Best Possible Solution

Keep the cleanup inside the acpx plugin boundary, but pass the generated wrapper root used by prepareAcpxCodexAuthConfig into startup reaping, add a direct recorded-root fallback when process listing is unavailable, quiet unsupported startup scans, and cover the split state-dir layout with focused tests.

## Reproduction Assessment

Yes. A source-level repro is clear: start the acpx service with ctx.stateDir different from pluginConfig.stateDir, leave a stale wrapper under ctx.stateDir/acpx, and the PR head's startup reaper filters against pluginConfig.stateDir/acpx instead.

## Solution Assessment

No, not as-is. The overall owner boundary and cleanup direction are right, but the startup root mismatch should be fixed before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the generated wrapper root for startup reaping:** `extensions/acpx/src/service.ts:222`
  - body: The reaper filters wrapper commands under the stateDir passed here, but service startup generates Codex/Claude ACP wrappers under `ctx.stateDir/acpx`. When `pluginConfig.stateDir` differs from the plugin service state dir, stale generated wrappers are skipped on startup; pass the wrapper root or `ctx.stateDir` instead.
  - confidence: 0.93
- **[P3] Kill the recorded root when listing fails:** `extensions/acpx/src/runtime.ts:512`
  - body: When process listing throws, this path stores an empty process list and passes it as authoritative. Cleanup can verify the recorded root command, but collects no root process from the empty list, so cancel/close sends no signal despite having the pid; synthesize the root or add a direct-root fallback.
  - confidence: 0.84
- **[P3] Skip unsupported startup process scans quietly:** `extensions/acpx/src/service.ts:221-222`
  - body: Startup invokes the default Unix `ps -axo` scanner and then warns on failure. On hosts where that scan is unsupported, every acpx service start can log a best-effort cleanup warning; guard the startup scan or make unsupported listing return an empty result.
  - confidence: 0.79
