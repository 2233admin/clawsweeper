## Summary

Keep open. Current main still lacks cross-Gateway session-tool routing, and this PR is the active implementation for the linked feature gap, but the branch has blocking protocol, auth, and security issues before it can merge.

## What This Changes

The branch adds `gateway.peers` config/schema/help, a peer resolver, cross-Gateway `gateway`/`gatewayUrl`/`gatewayToken` handling for `sessions_send` and `sessions_spawn`, and peer-resolution tests.

## Best Possible Solution

Land a maintainer-approved Gateway federation design that keeps raw URL overrides behind existing guardrails, defines trusted config-backed peer credentials and TLS/caller authorization, sends valid Gateway protocol payloads, preserves local session semantics, updates docs, and adds end-to-end remote send/spawn coverage.

## Reproduction Assessment

Yes. Static inspection is enough for the blocking PR issues: current main lacks the requested session-tool fields, while the PR head sends remote `agent` params without required `idempotencyKey` and routes explicit peer URLs around the existing agent-tool URL guard.

## Solution Assessment

No. The PR addresses a real feature gap, but this is not yet the best solution because it widens raw remote Gateway routing and remote execution semantics without a complete trusted-peer/auth policy or valid protocol payloads.
