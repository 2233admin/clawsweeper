## Summary

Keep open. Current main still does not implement outbound-message secret redaction for send or poll delivery, and the PR remains security-sensitive with unresolved review findings. The current send/poll path still forwards raw action params to channel plugin handlers and raw fallback content to core delivery; related #72733 is open but not merged or present on main.

## What This Changes

The PR adds dynamic logging redaction APIs and config/env secret discovery, initializes redaction from the CLI preaction hook, and attempts to redact outbound send and poll text before Telegram, Discord, Slack, and other channel delivery, with docs and tests.

## Best Possible Solution

Keep this PR open for maintainer/security rework or explicit closure in favor of a better-designed guardrail. A viable implementation should sanitize send and poll params before both plugin dispatch and core fallback delivery, cover Telegram/Slack/Discord-style plugin sends and polls with regression tests, preserve existing logging redaction semantics, and avoid blind config/env scanning unless backed by reviewed secret metadata. Coordinate with #72733, but do not treat it as superseding until a guardrail actually lands.

## Risks / Open Questions

- Merging the PR as-is could create false confidence because plugin-handled Telegram, Discord, and Slack sends/polls would still receive raw message and poll params.
- The proposed config/env scan is security-sensitive: broad key matching such as /auth/i can redact benign values, while omitted branches can still miss secret-bearing config.
- The proposed dynamic replacement path needs security review because replacement metacharacters in preserved prefixes/suffixes can reinsert matched secret text if replacement is not done with a function replacer or escaped string.
- The PR modifies core redaction tests and, per review context, drops existing coverage for mode switching, custom patterns, bounded-regex safety, and short-token behavior.
