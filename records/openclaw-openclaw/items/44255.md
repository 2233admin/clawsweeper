## Summary

Keep open. Current main still does not redact sensitive values before outbound channel delivery, and this PR is not merge-ready as written. The current outbound send/poll path forwards raw tool params into plugin handlers, while the PR diff redacts fallback values without sanitizing the plugin-dispatch params that Telegram, Slack, Discord-style plugins consume. The PR also mixes security-sensitive dynamic config/env secret scanning into logging redaction and has unresolved review findings around false positives, mode handling, replacement safety, and lost existing redaction coverage.

## Best Possible Solution

Keep this PR open for maintainer rework or explicit closure in favor of a better-designed security guardrail. A viable fix should sanitize send and poll params before both gateway/plugin dispatch and core fallback delivery, cover Telegram/Slack/Discord-style plugin handlers with regression tests, preserve existing logging redaction behavior and tests, and use a bounded secret-source design tied to config schema or SecretRef metadata rather than broad blind config/env scanning. Coordinate with #72733 if maintainers prefer one broader security-guardrail direction, but neither current main nor the open broader PR makes this PR implemented today.

## Risks / Open Questions

- Merging this PR as-is could tell users outbound Telegram/Discord/Slack-style messages are protected while plugin-handled sends and polls still receive raw secret-bearing params.
- The dynamic config/env scanning approach is security-sensitive: broad key matching such as `/auth/i` can redact benign content, omitted config branches can miss real secrets, and replacement-string handling can leak values containing `$` metacharacters.
- The PR removes existing `redact.test.ts` coverage for current bounded-regex, mode, custom-pattern, short-token, and PEM behavior, increasing regression risk in existing log/UI redaction surfaces.
- A correct design needs explicit maintainer policy for whether outbound messages should always be redacted, be configurable, or preserve user-intended secret delivery for some workflows.
