## Summary

Keep open. Current main still does not implement this PR's intended behavior of redacting sensitive values before outbound Telegram/Discord/Slack-style message or poll delivery. The PR is also not merge-ready as supplied: the review discussion identifies plugin dispatch bypasses and unsafe dynamic redaction edge cases that need rework before a security hardening change could land.

## Best Possible Solution

Rework this as a focused security-hardening PR. Sanitize outbound send and poll params before both local and gateway plugin dispatch, then pass the same sanitized content to core fallback delivery and transcript mirroring. Add regression tests for plugin-handled send and poll fields, preserve the existing logging redaction coverage, and base dynamic secret discovery on existing config schema/SecretRef metadata rather than broad blind env/config scans.

## Risks / Open Questions

- Merging the PR as-is could give users a false sense that outbound channel messages are protected while plugin-handled sends and polls still receive raw secret-bearing params.
- The proposed dynamic config/env scan is security-sensitive: broad key matching can redact legitimate user content, omitted config branches can miss real secrets, and unsafe string replacement can reinsert sensitive values.
- A correct implementation needs to preserve the existing logging redaction behavior and tests while adding focused outbound-delivery regression coverage.
