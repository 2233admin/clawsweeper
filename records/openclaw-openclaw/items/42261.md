## Summary

Current main still matches the reported bug. `streamReasoning` is only enabled when `reasoningMode === "stream"`, reasoning is visible, and `params.onReasoningStream` exists; `emitReasoningStream()` also returns before `emitAgentEvent({ stream: "thinking" })` when that callback is absent. WebChat `chat.send` does not provide `onReasoningStream`, and internal webchat typing policy resolves to no typing signal, so callback-less WebSocket clients remain blocked upstream before gateway delivery.

## Best Possible Solution

Keep this issue open. Fix `src/agents/pi-embedded-subscribe.ts` so live reasoning state is based on stream mode and existing visibility gates rather than callback presence, emit `stream: "thinking"` before optionally invoking `params.onReasoningStream`, add a callback-absent regression test proving WS/agent-event clients receive thinking events, and then update `/reasoning stream` docs/ack text plus Control UI handling if visible WebChat rendering is intended.
