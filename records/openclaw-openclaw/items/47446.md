## Summary

Keep open. The PR is aimed at a real remaining Discord proxy gap: current main already bootstraps gateway-wide undici env-proxy handling, but Discord gateway WebSocket proxy fallback is still explicit-config only. The patch should not merge as-is because it crosses the plugin/core package boundary, ignores NO_PROXY for the new Discord env fallback, and places the undici keepAlive policy where undici 8.1.0 does not apply it to proxied tunnel reuse.

## What This Changes

The PR bootstraps undici env-proxy handling in gateway startup, adds ambient HTTP(S) proxy fallback for Discord gateway proxy agents, changes Discord/undici proxy agent construction to pass keepAlive false, updates focused proxy tests, and includes small gateway command-path refactors.

## Best Possible Solution

Revise or replace the PR narrowly: keep the useful Discord env-proxy fallback, implement it through plugin-safe SDK exports, gate Discord REST and gateway hosts with NO_PROXY-aware logic, apply the undici proxy transport/pooling policy through supported options with focused tests, and drop unrelated gateway command-path refactors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Stop importing core internals from the Discord plugin:** `extensions/discord/src/monitor/gateway-plugin.ts:7`
  - body: This production import reaches from the Discord plugin into src/infra through a relative path. Bundled plugins are packaged and loaded through their own package files and public SDK subpaths, so this can fail in distributed installs and bypasses the documented plugin boundary. Use openclaw/plugin-sdk/fetch-runtime or a local plugin artifact instead.
  - confidence: 0.94
- **[P1] Honor NO_PROXY before using Discord env proxy fallback:** `extensions/discord/src/monitor/gateway-plugin.ts:271`
  - body: resolveEnvHttpProxyUrl("https") only selects a proxy URL; it does not check NO_PROXY for discord.com or gateway.discord.gg. With HTTPS_PROXY set and Discord excluded, the new fallback still forces Discord traffic through the proxy the operator asked to bypass.
  - confidence: 0.9
- **[P2] Put EnvHttpProxyAgent proxy policy on supported options:** `src/infra/net/undici-global-dispatcher.ts:126`
  - body: Passing keepAlive under connect does not configure the proxied HTTPS path for EnvHttpProxyAgent: undici forwards those options into ProxyAgent, whose proxy tunnel connector is built from proxyTls/requestTls and whose Agent connect function is replaced. The claimed ECONNRESET mitigation can remain ineffective for env-proxied fetch traffic.
  - confidence: 0.84
- **[P2] Use supported ProxyAgent options for Discord proxy agents:** `extensions/discord/src/monitor/rest-fetch.ts:18`
  - body: The explicit Discord ProxyAgent construction also puts keepAlive under connect, but undici's ProxyAgent does not use that option for the tunnel Agent it installs. Explicit Discord proxy REST and gateway metadata calls can still reuse stale proxy tunnels, so the fix is not actually enforced here.
  - confidence: 0.82
