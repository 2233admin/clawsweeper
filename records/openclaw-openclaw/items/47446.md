## Summary

Keep open. Current main has generic gateway/CLI EnvHttpProxyAgent bootstrap, but it does not make this PR obsolete: Discord gateway WebSocket creation still only uses an explicit/debug `channels.discord.proxy`, Discord gateway metadata fetches go through the strict SSRF guarded path without trusted env-proxy mode, and the ECONNRESET keep-alive change is not implemented on main. The submitted PR also should not merge as-is because the provided diff crosses the plugin boundary, misses NO_PROXY semantics for the derived Discord env proxy, likely applies undici proxy tunnel options on the wrong option layer, and includes unrelated gateway node-command authorization edits.

## Best Possible Solution

Keep this PR open as a useful candidate, but require a narrower revision or replacement patch. The best path is to keep the existing generic gateway env-proxy bootstrap, add Discord gateway WebSocket and metadata env-proxy behavior through a plugin-safe SDK/local helper that honors NO_PROXY for Discord hosts, apply undici proxy transport policy at the correct `proxyTls`/`requestTls` layer with focused tests, and drop the unrelated gateway node-command authorization edits.

## Risks / Open Questions

- Security/package-boundary risk: as submitted, the Discord plugin imports a core `src/**` file by relative path, which violates the bundled-plugin boundary and can break packaged plugin loading paths.
- Proxy correctness risk: deriving a Discord proxy from `HTTPS_PROXY`/`HTTP_PROXY` with `resolveEnvHttpProxyUrl('https')` alone ignores `NO_PROXY` exclusions for `discord.com` and `gateway.discord.gg`.
- Reliability risk: the PR's `connect: { keepAlive: false }` placement is unlikely to affect undici proxy tunnel reuse; current repo and upstream evidence point to `proxyTls` for proxy connection settings and `requestTls` for HTTPS target tunnel settings.
- Scope/security review risk: the submitted diff also changes gateway node-command authorization files unrelated to Discord proxy behavior; that command-execution path needs separate justification or removal.
