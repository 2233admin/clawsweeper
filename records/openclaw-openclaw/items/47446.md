## Summary

Keep open. Current main covers part of the PR by bootstrapping the Gateway's global undici env-proxy dispatcher, but it still does not implement the PR's central Discord WebSocket fallback from HTTP_PROXY/HTTPS_PROXY when channels.discord.proxy is unset. The PR also should not merge as-is because the submitted diff crosses the plugin boundary, does not honor NO_PROXY for the derived Discord proxy, appears to put undici CONNECT-tunnel keep-alive policy on the wrong option layer, and includes unrelated gateway node-command authorization edits.

## Best Possible Solution

Keep this PR open as a useful candidate, but require a narrower revision or replacement patch. The best path is to keep the existing Gateway env-proxy bootstrap on main, add Discord WebSocket env-proxy fallback through a plugin-safe SDK/local seam that respects NO_PROXY for Discord hosts, apply the undici proxy transport policy at the tested proxyTls/requestTls layer with focused tests, and drop the unrelated gateway node-command authorization edits.

## Risks / Open Questions

- Security/supply-chain review: the submitted PR introduces a Discord plugin production import from core src internals, which violates the plugin packaging boundary and can break bundled or external plugin loading paths.
- Proxy correctness review: deriving a Discord proxy directly from HTTPS_PROXY/HTTP_PROXY must honor NO_PROXY for discord.com and gateway.discord.gg, otherwise operators can be routed through a proxy they explicitly bypassed.
- Reliability review: setting keepAlive: false under undici connect options is likely ineffective for proxied HTTPS CONNECT tunnel reuse; current repo tests point to proxyTls/requestTls for that layer.
- Scope review: the PR also changes gateway node command authorization files unrelated to Discord proxy behavior; those security-sensitive command-path edits need separate justification or removal.
