## Summary

Keep open. Current main still has the provider-id normalizer, but the Usage/Control UI path continues to aggregate, filter, suggest, export, and display raw provider strings, so supported aliases such as `z-ai`/`z.ai` and `aws-bedrock` can still appear or split from their canonical providers. The reported `bailian` example is not supported by current source or docs, so the fix should not add that alias without maintainer approval.

## What This Changes

The issue requests that Control UI provider displays use canonical normalized provider ids instead of raw configured aliases such as `z-ai` or `aws-bedrock`.

## Best Possible Solution

Land a focused Usage-surface fix that reuses the existing provider-id canonicalization contract for display labels, filters, suggestions, query matching, CSV export, and aggregate grouping while preserving raw stored session/config fields and leaving new alias contracts to maintainer approval.
