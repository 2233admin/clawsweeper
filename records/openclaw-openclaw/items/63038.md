## Summary

Keep this PR open, but do not merge it as-is. Current main still has the WhatsApp shutdown gap, and the PR touches the right shared debouncer and WhatsApp close path, but its one-shot `flushAll()` can still return before late enqueues or active keyed flush work have quiesced, so the socket can still close while reply/media sends are pending.

## What This Changes

The PR adds `flushAll()` to the shared inbound debouncer, awaits it from the WhatsApp Web inbox close path before socket teardown, and updates several channel/test debouncer mocks for the new return shape.

## Best Possible Solution

Revise this PR around a real debouncer quiescence/drain operation: repeatedly flush current buffers, wait for active keyed chains, and only return once the debouncer is stable. Then call that API from the WhatsApp close path after detaching inbound listeners and before `closeInboundMonitorSocket(sock)`, with focused regression coverage for send-before-close, late enqueue during shutdown, and in-flight keyed work with no buffer entry.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Drain late and active debounce work before closing:** `src/auto-reply/inbound-debounce.ts:237`
  - body: `flushAll()` snapshots `buffers.keys()` once and awaits only those `flushKey()` calls. During WhatsApp shutdown, an already-running `handleMessagesUpsert` can enqueue after that snapshot, and timer-backed flushes can have active `keyChains` after their buffer was deleted. In both cases this returns to `close()` too early, so the socket can still be closed while the debounced reply or media send is pending.
  - confidence: 0.9
