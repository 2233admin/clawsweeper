## Summary

Keep open. Current main still has the reported exec-approval identity mismatch, and the provided GitHub context shows open PR #73543 already uses closing syntax for this issue, so the right path is maintainer review of that PR or an equivalent fix rather than cleanup closure.

## What This Changes

The issue requests a canonical executable identity for exec approval checks so allowlist, trust, persistence, and execution do not diverge across symlinked binaries.

## Best Possible Solution

Land #73543 or an equivalent focused fix that makes allowlist and trust decisions use the same canonical executable identity as approval-backed execution, with regression coverage for real-path allowlist matching through symlinks and symlink target drift.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a resolution with `resolvedPath` set to `/opt/homebrew/bin/rg` and `resolvedRealPath` set to `/opt/homebrew/Cellar/ripgrep/14.1.1/bin/rg`: current `matchAllowlist()` checks only the symlink path while approval-backed execution later pins the realpath.

## Solution Assessment

Unclear until the open PR is reviewed. Matching `resolvedRealPath` in the allowlist matcher is the narrow primary fix, but maintainers should decide whether the same canonical identity must also cover persistence, safe-bin trust, skill auto-allow, and audit fields before closing this issue.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because current main can approve one executable path identity while executing another.

Concerns:

- **[medium] Approval identity can drift across symlink target changes:** `src/infra/exec-command-resolution.ts:376`
  - body: Path-shaped allowlist matching checks `resolvedPath`, but approval-backed execution uses `resolvedRealPath` when available. An approval tied to a symlink path can therefore continue matching after the symlink target changes while a different real binary is executed.
