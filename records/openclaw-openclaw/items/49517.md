## Summary

Keep this issue open. Current main still treats `/new` as reset-style rollover in the auto-reply and Gateway paths, Control UI still warns that `/new` resets the current chat, and tests/docs still encode reset-alias behavior. The TUI has a local unique-session-key behavior, but that does not resolve the cross-surface contract requested here. The related PR #49595 is closed unmerged, and #73450 covered a different Telegram empty-request failure mode.

## Best Possible Solution

Keep this issue open for maintainer product/implementation follow-up. Define the cross-surface `/new` contract first, then implement `/new` as a non-destructive create/switch flow, likely using or extending `sessions.create` and session selection/history UI, while preserving `/reset` as the destructive reset path. Update Gateway/chat handling, Control UI, ACP/thread binding behavior if applicable, docs, and regression tests together.
