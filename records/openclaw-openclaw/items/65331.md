## Summary

Keep open. Current main still routes plugin text transforms through the indirect active-runtime helper that probes `./runtime.js`/`./runtime.ts`, while the direct active registry accessor remains available and the CLI/provider prompt paths still consume this resolver. The PR is useful source work for the real bug tracked in #64921, but it is stale against the current split into `active-runtime-registry` and needs regression coverage before it should land.

## What This Changes

The PR changes plugin text-transform resolution to prefer `getActivePluginRegistry()` directly, falling back to the existing runtime loader.

## Best Possible Solution

Revise the fix on current main so runtime text-transform resolution reads the active plugin registry through a direct or well-documented generic seam, add a focused regression test for main-process plugin text transforms, and use that to resolve #64921.
