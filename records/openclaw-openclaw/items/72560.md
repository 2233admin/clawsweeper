## Summary

Keep open. Current main still only has adjacent mechanisms: active-run queue policy, explicit `/btw` side-question handling, and plugin hooks that can intercept a reply but do not receive the prior assistant question text or outbound timestamp needed for this proposal. This needs maintainer product/API direction rather than conservative cleanup closure.

## What This Changes

The issue proposes an optional pre-turn context-staleness check that compares delayed inbound user messages against the last assistant question before treating them as answers, or documents the standalone helper for users.

## Best Possible Solution

Define a config-gated turn-admission design that can see the last assistant question text, outbound timestamp, inbound text/timestamp, session and channel metadata, then return allow, clarify, ignore, or side-note handling. It should cover web chat plus at least one external channel and document how it interacts with `/btw` and active-run queue modes.

## Reproduction Assessment

Not applicable as a feature/API proposal rather than a concrete shipped regression. The high-confidence verification path here is source and docs inspection, which shows no automatic classifier on current main; a future fix should add a deterministic delayed-message scenario.

## Solution Assessment

Unclear. The standalone module is useful prototype evidence, but directly wiring those exact heuristics as middleware is not clearly the best maintainable solution until maintainers decide whether this belongs in core turn admission or behind a public plugin API seam.
