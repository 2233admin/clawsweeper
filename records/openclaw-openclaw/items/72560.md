## Summary

Keep open. Current main has adjacent active-run queueing, steering, explicit `/btw` side questions, and plugin hook surfaces, but I found no automatic pre-turn classifier that compares a new inbound message with the last assistant question before treating it as the next answer. The existing hooks also do not clearly expose the full last-outbound-question context needed to move this wholly out to a plugin today.

## What This Changes

The issue proposes an optional pre-turn context-staleness check so a slow-running assistant does not treat a later unrelated user message as the answer to its previous question.

## Best Possible Solution

Define a small, config-gated turn-admission design: either a core pre-turn classifier or a plugin hook that receives last assistant question text, outbound timestamp, inbound timestamp, inbound text, route/session metadata, and can allow, clarify, or treat the message as side context. Cover web chat and at least one external channel path, and document interaction with `/btw`, `steer`, `collect`, and follow-up queues.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a feature/API proposal. The high-confidence verification path is source inspection of active-run queueing, `/btw`, and plugin hook contracts, which shows no current automatic context-staleness classifier.

## Solution Assessment

Unclear. The standalone Node module may be a useful prototype, but directly wiring heuristics into middleware is not clearly the best maintainable solution until maintainers choose the product boundary and API shape.
