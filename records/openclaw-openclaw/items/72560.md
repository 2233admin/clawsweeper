## Summary

Keep open. The prior ClawSweeper comment matches current-main source inspection: OpenClaw has active-run queueing, explicit `/btw` side questions, and adjacent plugin hooks, but no automatic pre-turn check that compares a delayed inbound message to the last assistant question before treating it as the next normal turn. This is recent, concrete, and likely needs maintainer product/API judgment rather than cleanup closure.

## What This Changes

The issue proposes an optional pre-turn middleware that checks whether a delayed inbound message actually answers the last outbound assistant question before normal turn processing.

## Best Possible Solution

Define a small, config-gated turn-admission design: either a core pre-turn classifier or a versioned plugin hook that receives the last assistant question, outbound timestamp, inbound timestamp/text, and route/session metadata, then can allow, clarify, or treat the message as a side note. Cover web chat plus at least one external channel with focused tests, and document how it interacts with `/btw`, active-run queue modes, and plugin permissions.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a feature/design proposal. The high-confidence verification path is source inspection of active-run queueing, `/btw`, and plugin hook contracts, which shows no current automatic context-staleness classifier.

## Solution Assessment

Unclear. The standalone module may be a useful prototype, but directly wiring it as middleware is not clearly the best maintainable solution until maintainers decide whether the contract belongs in core turn admission or a plugin API seam.
