## Summary

Keep open. Current main has adjacent protections for busy-run queueing, steering, and explicit `/btw` side questions, but it still does not implement the requested automatic pre-turn classifier that compares a delayed inbound message against the last assistant question before treating it as an answer. The current plugin hooks also look close but incomplete for a plugin-only close because they do not expose the last outbound assistant question plus timing as a typed contract.

## What This Changes

The issue proposes an optional pre-turn context-staleness middleware that uses timestamp and intent heuristics before treating a delayed inbound user message as an answer to the assistant's last question.

## Best Possible Solution

Define a small, config-gated turn-admission design: either a core classifier or an additive plugin hook that receives the last assistant question, outbound timestamp, inbound timestamp/text, and route/session metadata, then can allow, clarify, or treat the message as a side note. Cover WebChat plus at least one external channel path, and document how it interacts with `/btw`, queue modes, and steering.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not as a normal bug reproduction because this is a feature/design proposal; the high-confidence verification path is source inspection of the current reply pipeline, busy-run queueing, BTW behavior, and plugin hook contracts, which shows no automatic stale-answer classifier today.

## Solution Assessment

Is this the best way to solve the issue? Unclear. The standalone module is useful prototype evidence, but wiring it directly as middleware is not clearly the narrowest maintainable solution until maintainers decide whether this belongs in core turn admission or a versioned plugin API seam.
