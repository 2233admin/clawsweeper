## Summary

Keep PR #39726 open. Current main still only enables empty-success background exec notifications when `notifyOnExitEmptySuccess` is explicitly true, so reply-capable chat sessions still get the reported “started, then silence” behavior for `exit 0` commands with no output. The latest PR context shows the author addressed the earlier review concerns by preserving explicit opt-out and avoiding generic provider labels such as `openai`.

## Best Possible Solution

Keep PR #39726 open or rework it onto current main. The best path is the narrowed behavior already represented by the latest PR head: default empty-success notifications only for real reply-capable chat channels when the config value is unset, preserve explicit `false`, keep generic provider labels like `openai` silent, and land the regression coverage with any accepted changelog or docs update.
