## Summary

Keep this PR open. Current main still treats quiet successful background exec notifications as explicit opt-in only, so the PR's intended chat-context default is not implemented on main. The latest PR diff is focused and also addresses earlier review concerns around explicit false opt-out and generic provider labels.

## Best Possible Solution

Review and either land or intentionally decline this PR based on the chat-surface notification policy. If accepted, keep the current narrow shape: default empty-success notifications only when a real gateway message channel and delivery target are present, preserve explicit notifyOnExitEmptySuccess: false, keep generic provider labels silent, and retain the regression tests plus changelog credit.
