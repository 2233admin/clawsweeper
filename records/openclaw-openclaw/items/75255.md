## Summary

Keep open. Current main has a bundled Xiaomi provider, but it only wires auth, static model catalog, usage metadata, and TTS; it does not expose Xiaomi-native `web_search` config or append Xiaomi's native tool to chat-completion payloads. The reporter's follow-up gives concrete live API behavior, so this is an actionable provider-plugin feature request rather than stale cleanup.

## What This Changes

Add Xiaomi MiMo provider support for its native `web_search` chat-completions tool, including opt-in config and optional request parameters.

## Best Possible Solution

Implement this inside the Xiaomi provider plugin: add plugin-scoped `webSearch` config, validate supported models and official endpoint/key behavior, append the native `web_search` tool without disturbing normal function tools, preserve or expose returned citations if the transport needs help, and document the opt-in behavior and cost implications.

## Reproduction Assessment

Not applicable as a traditional bug reproduction, but yes for the absence check: current main has no Xiaomi webSearch schema, runtime wrapper, provider search registration, or docs for native `web_search`. The reporter supplied a concrete live API request/response example for expected behavior.

## Solution Assessment

Yes. A Xiaomi-owned plugin change using the existing manifest config and provider stream-wrapper seam is the narrowest maintainable direction; a broader core native-tool abstraction should wait until more providers need the same contract or Xiaomi proves the existing seam insufficient.
