## Summary

Keep open. Current main has a bundled Xiaomi provider, but it only wires model catalog/auth usage and TTS; it has no Xiaomi `webSearch` config and no payload wrapper that appends the provider-native `web_search` tool. This should not be closed as ClawHub because Xiaomi is already a bundled provider plugin and the existing provider hook surface appears suitable for a narrow provider-owned fix.

## What This Changes

Add Xiaomi MiMo support for opt-in provider-native `web_search` tools in chat-completions requests, including optional force/search-limit/location settings.

## Best Possible Solution

Add a provider-owned Xiaomi implementation that declares plugin-scoped web-search config, appends the native `web_search` tool only for Xiaomi OpenAI-compatible chat-completions models when enabled, preserves existing function tools, documents Xiaomi console/plugin activation and billing/cost behavior, and avoids hardcoding Xiaomi policy into unrelated core paths unless a generic provider-native-search seam is genuinely required.

## Reproduction Assessment

Not applicable as a bug reproduction, but there is a high-confidence absence check: current main has no Xiaomi `webSearch` config schema and no Xiaomi payload wrapper or native-tool injection path. A live end-to-end reproduction is blocked only by missing local Xiaomi/MiMo credentials.

## Solution Assessment

Yes, the best direction is a narrow bundled Xiaomi provider-plugin change using the existing provider `wrapStreamFn` payload-patch hook and manifest config schema. A broader core change should be reserved only if tests prove the current provider hook cannot safely express native-search activation and tool-policy gating.
