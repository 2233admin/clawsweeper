## Summary

Keep open. Current main still has identity-based Discord bound-thread webhook suppression and message-id-based inbound dedupe, but I did not find a current-main path that canonicalizes a PluralKit original/proxy pair with different Discord message IDs and missing or nonmatching webhook metadata before dispatch. The paired implementation PR #52007 is closed unmerged, while the issue discussion includes a live Discord/PluralKit repro and agreement that the existing webhook suppression is only the first line of defense.

## What This Changes

Fix Discord bound-thread handling so PluralKit original/proxy message pairs are processed as one inbound turn without duplicate context pressure or lingering typing indicators.

## Best Possible Solution

Add Discord-owned provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before they reach dispatch. Prefer an explicit Discord or PluralKit signal when available; if none is reliable, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and ensure suppressed duplicates do not start or extend typing indicators.
