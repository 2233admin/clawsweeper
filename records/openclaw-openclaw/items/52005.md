## Summary

Keep open. Current main still has identity-based Discord bound-thread webhook suppression plus shared MessageSid dedupe, but I did not find a current-main path that collapses a PluralKit original/proxy pair with different Discord message ids and missing or nonmatching webhook metadata before dispatch. The related implementation PR #52007 is closed unmerged, and the issue discussion includes a live repro report plus useful clarification that the existing webhook suppression is only the first line of defense.

## What This Changes

The issue asks Discord bound-thread handling to treat PluralKit original/proxy message pairs as one inbound turn, with one run/reply and no lingering typing indicator.

## Best Possible Solution

Add Discord provider/preflight canonicalization for bound-thread PluralKit original/proxy pairs before dispatch. Prefer an explicit Discord or PluralKit signal; if unavailable, use a short-lived fallback scoped to Discord bound-thread agent-session traffic and ensure suppressed duplicates do not start or extend typing indicators.

## Repair Work Prompt

Fix the Discord bound-thread PluralKit duplicate-turn bug tracked by https://github.com/openclaw/openclaw/issues/52005. The observable failure is that a PluralKit original Discord message plus webhook-proxied copy in an already-bound Discord thread can both pressure the inbound pipeline when they have different message ids and missing or nonmatching webhook metadata, causing duplicate context/run wobble and lingering typing. Work primarily in the Discord plugin preflight/process path. Prefer an explicit Discord or PluralKit canonical signal before dispatch; if none is available, add a short-lived fallback scoped only to Discord bound-thread agent-session traffic, keyed narrowly enough to include account/thread/session and normalized content/sender context, with a small TTL. Ensure suppressed duplicates do not start or extend Discord typing indicators. Do not broaden generic shared content dedupe across all channels, do not change PluralKit config defaults or Discord allowBots behavior, and do not move owner-specific behavior into core unless a small generic seam is clearly needed. Use related PR https://github.com/openclaw/openclaw/pull/52007 as context, not as code to apply blindly. Add focused regression tests for matching/nonmatching webhook metadata, original/proxy different message ids, and typing suppression. Add a single-line changelog Fixes entry with Thanks @acgh213 if production behavior changes.
