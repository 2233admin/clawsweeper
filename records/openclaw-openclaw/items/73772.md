## Summary

Keep open. Current main still does not implement the proposed scoped `memory.*` Gateway RPC family, but this PR is not ready to merge because the branch mixes several stacked memory changes with unrelated regressions to existing Gateway/plugin/session/node protocol surfaces and leaves retention issues in the new long-lived memory RPC state.

## What This Changes

The PR adds scoped memory Gateway RPC handlers, protocol schemas, generated Swift models, and memory-core/host SDK scoping changes for memory status, source listing, debug search, indexing jobs, and source opening.

## Best Possible Solution

Keep the PR open for normal maintainer review, but require a narrowed additive gateway/protocol patch: preserve current plugin/session/node protocol contracts, keep memory-host package code within its package boundary, add bounded retention for job/source-ref state, and re-run protocol generation plus focused gateway/protocol/memory-core validation after #73769-#73771 are resolved.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the review findings: compare the PR diff to current main and the removed validators/handlers/types are still imported, documented, generated, or tested by current code. For the requested feature itself, reproduction is not a bug flow; current main absence is reproducible by inspecting method discovery, method scopes, handler registration, and protocol schemas.

## Solution Assessment

Is this the best way to solve the issue? No. The desired scoped memory RPCs may be a reasonable additive direction, but this branch is not the narrowest maintainable solution because it removes unrelated shipped Gateway/plugin/session/node protocol surfaces and leaves unbounded long-lived state in new RPC handlers.

  - confidence: 0.93

  - body: The diff removes `sessions.pluginPatch` from method discovery and scope lists while the handler, schema, docs, and plugin session-extension contract remain on main. That hides/breaks the public plugin session state seam; memory RPC discovery should be additive.
  - confidence: 0.9

  - body: The package implementation currently imports local `./error-utils.js`, but this change points it at `../../../../src/infra/errors.js`, which violates the memory-host package boundary the repo reports. The same hunk also uses `.length` on a `Set`, so the package copy should stay local and use `.size` for the FTS rebuild check.
  - confidence: 0.88
- **[P2] Bound memory index job retention:** `src/gateway/server-methods/memory.ts:52-53`
  - body: Every `memory.index.run` stores a job in the module-level `indexJobs` map and neither `memory.index.run` nor `memory.index.jobs` prunes or caps it. A long-lived gateway can leak memory as clients trigger indexing; add TTL/cap cleanup that preserves running jobs.
  - confidence: 0.91
- **[P2] Prune source refs when issuing debug results:** `src/gateway/server-methods/memory.ts:377-410`
  - body: `memory.search.debug` can mint up to 50 `sourceRef` tokens per call, but stale refs are pruned only when `memory.source.open` runs. A debug-only workflow can grow `sourceRefs` indefinitely; prune before issuing refs or from the shared requester path.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The scope model is directionally cautious, but the new long-lived RPC state introduces concrete availability risks that should be fixed before merge.

Concerns:

- **[medium] Unbounded index job map:** `src/gateway/server-methods/memory.ts:52`
  - body: `memory.index.run` appends every job to a module-level map with no TTL or maximum size, allowing repeated authorized calls to grow gateway memory until restart.
  - confidence: 0.91
- **[low] Debug source refs accumulate without open calls:** `src/gateway/server-methods/memory.ts:377`
  - body: `memory.search.debug` issues server-side source refs but pruning only occurs in `memory.source.open`, so read-scoped debug-only clients can accumulate expired refs in a long-lived process.
  - confidence: 0.88
