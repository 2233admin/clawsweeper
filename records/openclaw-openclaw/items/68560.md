## Summary

Keep open. Current main still does not strip inline image data at the tool-result persistence boundary; it only caps text/details on write and prunes or sanitizes image blocks later for replay/context. The PR is relevant, but it needs normal maintainer review because its added test duplicates the private helper instead of exercising production wiring, and immediate write-time stripping is a transcript storage contract change.

## What This Changes

The PR adds a `guardSessionManager` tool-result persistence transform that clears `type: "image"` block `data` fields after plugin hooks and adds a new test file for that stripping behavior.

## Best Possible Solution

Land this only after the production persistence path is tested through `guardSessionManager` or an exported pure helper, and after maintainers confirm the desired placeholder semantics for image-only and `MEDIA:`-less tool results. The storage boundary should strip raw image bytes without weakening plugin hook ordering, redaction, synthetic tool-result repair, or replay sanitization.
