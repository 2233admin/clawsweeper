## Summary

Keep open. Current main still matches the reported custom-session cron transcript gap: the cron run uses the persistent destination session key and can deliver to the channel, but the direct delivery call does not pass the mirror metadata required for transcript persistence. PR #74913 and a linked fork PR already target the exact bug, so the issue should remain open until one fix lands.

## What This Changes

Cron jobs targeting `session:<id>` visibly deliver assistant responses to the configured channel but do not persist those responses into the destination session transcript.

## Best Possible Solution

Review and land PR #74913, or an equivalent focused fix, so successful direct cron deliveries mirror their visible assistant text into the resolved destination session transcript through the existing outbound mirror contract with regression coverage for `session:<id>` jobs.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: follow the reported custom-session direct cron delivery path, then observe that `dispatchCronDelivery` omits `mirror` while transcript append in `deliverOutboundPayloads` requires it.

## Solution Assessment

Yes. Keeping this issue open while reviewing the existing focused PR is the narrow maintainable path; the fix should reuse the outbound mirror contract rather than adding a separate cron-specific transcript writer.
