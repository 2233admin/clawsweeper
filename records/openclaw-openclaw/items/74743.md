## Summary

Keep open. Current main still matches the reported cron custom-session transcript gap: custom/current cron jobs can run against a persistent session and deliver directly to a channel, but the direct delivery call does not provide outbound mirroring, and the transcript append path only runs when a mirror object is present. PR #74913 and the linked fork PR already target this exact bug, so the issue should remain open until one of those fixes lands.

## What This Changes

The issue reports that cron jobs created with `--session session:<id>` deliver replies to the destination channel but do not write the delivered assistant text into that session's transcript, leaving later turns unaware of the cron response.

## Best Possible Solution

Review and land PR #74913, or an equivalent focused fix, so successful direct cron deliveries mirror their visible assistant text into the resolved destination session transcript with an idempotency key while preserving existing isolated-cron duplicate-delivery and awareness behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: create a custom-session announced cron job and force direct delivery, then observe that the cron delivery path calls `deliverOutboundPayloads` without `mirror` while outbound transcript persistence requires `params.mirror`. A live Discord run would add end-to-end proof but is not necessary to identify the code gap.

## Solution Assessment

Yes. Keeping this open and reviewing the existing focused PR is the narrow maintainable path; direct cron delivery should supply the same transcript mirror metadata that outbound delivery already understands, rather than creating a separate persistence path.
