## Summary

Keep open. Current main still matches the reported cron transcript gap: direct cron delivery sends to the resolved channel with a session context, but it does not pass outbound `mirror` metadata, while transcript persistence only happens when `params.mirror` is present. The linked open PR #74913 and the linked fork PR already target the same bug with closing syntax, so this should stay open until a fix lands rather than spawning duplicate repair work.

## What This Changes

Custom-session announced cron jobs should persist the delivered assistant response into the destination `session:<id>` transcript so later turns can see what the agent sent.

## Best Possible Solution

Review and land PR #74913, or an equivalent focused fix, so successful direct cron deliveries mirror their visible assistant text into the resolved destination session transcript using the existing outbound mirror contract and an idempotency key.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: create a custom-session announced cron job and force direct delivery, then the source path shows the message is delivered without `mirror` while transcript append requires `params.mirror`. The reporter's live steps match that code path.

## Solution Assessment

Yes. The best fix direction is to reuse `deliverOutboundPayloads` mirror metadata for direct cron delivery and cover `session:<id>` with a regression test, rather than adding a separate cron-only transcript writer.
