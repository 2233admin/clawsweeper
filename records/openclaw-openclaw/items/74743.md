## Summary

Keep open. Current main still has the reported custom-session cron transcript gap: custom/current cron jobs can run in a persistent session and directly announce to a channel, but the direct delivery path does not pass a transcript mirror, while the outbound layer only appends assistant text when `mirror` is present. An open fork PR already targets this exact issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that cron jobs created with `--session session:<id>` deliver responses to the configured channel but do not persist those assistant responses into the destination session transcript.

## Best Possible Solution

Land a focused cron delivery fix that mirrors successful custom/current-session direct cron sends into the targeted persistent session transcript, preserves isolated-cron duplicate-delivery and awareness behavior, and clarifies or bridges the `session:<id>` session-key versus stored `sessionId` contract in docs or code.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: create a custom-session announce cron job, force direct delivery, and observe that `dispatchCronDelivery` calls `deliverOutboundPayloads` without `mirror`, while the transcript append path only runs with `params.mirror`. A live Discord run would add end-to-end proof but is not required to see the current code gap.

## Solution Assessment

Unclear until the open PR is reviewed. Passing a mirror object from direct cron delivery is the narrow, maintainable direction, but the final fix should cover payload/media projection and idempotency and decide whether the CLI should accept stored session IDs or require internal session keys.
