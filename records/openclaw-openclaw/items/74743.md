## Summary

Keep open. Current main has custom-session cron execution and channel announce delivery, but the delivery-awareness path is intentionally limited to isolated cron jobs while session-bound cron delivery is explicitly tested to skip the queued awareness event. That leaves the reported channel-memory gap plausible and not already covered by a shipped fix.

## What This Changes

Cron jobs created with `--session session:<id>` should persist channel-delivered cron replies into the destination session history so later turns in that channel can remember them.

## Best Possible Solution

Make custom/current cron channel sends durable in the same persistent session the user targets, including direct fallback delivery and message-tool delivery, and clarify whether `session:<id>` accepts session-store keys only or can resolve stored `sessionId` values.

## Repair Work Prompt

Fix the custom-session cron persistence bug reported in https://github.com/openclaw/openclaw/issues/74743. Observable behavior: a cron job created with `openclaw cron add --session "session:<target>" --message ... --wake now --channel discord --to "channel:<id>"` can deliver text to the channel, but a later turn in that destination session does not know what the cron job just sent. Keep the fix in the cron/session delivery boundary; do not change provider defaults, channel plugin internals, release metadata, or unrelated cron output-target features.

Expected direction: add a focused regression test showing successful custom-session/channel cron delivery leaves durable context in the targeted persistent session. Cover the direct fallback path and, if practical, the message-tool path. Decide explicitly whether `session:<id>` should resolve a stored `sessionId` from `sessions.json` or remain a session-store key only; if it remains key-only, update CLI/docs/help so users do not copy the wrong field. Preserve the existing duplicate-delivery and best-effort safeguards for isolated jobs.
