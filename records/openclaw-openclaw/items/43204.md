## Summary

Current main still does not implement this PR's intended TTL-expiry feedback path. The typing TTL timer still only logs and cleans up, `GetReplyOptions` has no TTL-expiry callback, `getReplyFromConfig` cannot wire one, and `dispatchReplyFromConfig` does not inject a default visible still-working notice. The PR remains potentially useful, but it should not be merged as-is because its head is stale and includes unrelated macOS/onboarding and Tlon package-resolution changes alongside the auto-reply fix.

## Best Possible Solution

Keep this PR open as reference, but do not merge it as-is. The maintainable path is a fresh or heavily rebased auto-reply-only change against current main that adds the chosen TTL-expiry feedback contract, contains both synchronous throws and async rejections from custom handlers, preserves routed-reply ordering, adds focused tests for TTL expiry, suppressed typing, and routed replies, and drops unrelated macOS/Tlon/package-resolution changes. Coordinate with #40760 before choosing TTL-expiry notices versus longer timeout-aware typing budgets or both.

## Risks / Open Questions

- Merging this PR as-is could add or reintroduce unrelated macOS/onboarding changes outside the stated auto-reply TTL feedback fix, increasing regression surface.
- The PR's Tlon package-resolution change is supply-chain-sensitive and unrelated to the typing TTL behavior; current main has moved past that dependency shape.
- The PR discussion includes a remaining handler-safety concern: `Promise.resolve(ttlHandler()).catch(...)` does not catch synchronous throws because the handler is evaluated before the promise chain exists.
- Related open PR #40760 addresses a neighboring timeout/typing visibility problem, so maintainers should choose the product direction between longer timeout-aware typing budgets, TTL-expiry notices, or both before landing either approach.
