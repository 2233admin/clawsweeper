## Summary

Current main still does not implement the requested `/agent` read-path mapping for `workspaceAccess: "ro"` sandbox sessions when the workspace-root guard is active. The guard only remaps `containerWorkdir` paths, the sandboxed read-tool wiring only passes `sandbox.containerWorkdir`, current coverage still expects `/agent/secret.txt` to be rejected under `tools.fs.workspaceOnly`, and the mount/docs surfaces still identify `/agent` as the read-only agent workspace mount.

## Best Possible Solution

Keep this issue open and land a focused sandbox/file-tool fix: make the read workspace-root guard validate through the sandbox mount table, or pass an explicit `/agent` mapping only when the active sandbox context has `workspaceAccess: "ro"` and that mount exists. Add regression coverage proving `/agent/...` reads succeed while write/edit/apply_patch remain blocked, and leave `/tmp` reads for a separate policy decision.
