## Summary

Keep open. Current main has a useful failure-class-aware fallback foundation, including 401/429 classification, auth/profile cooldown state, same-provider auth skipping once all profiles are unavailable, and models-status unusable-profile reporting. It still does not implement the requested hard-auth provider quarantine policy, invalid-request fail-fast behavior, or provider quarantine/health in normal status; closed #61328 only covers part of the remaining scope.

## What This Changes

Request provider fallback to classify failures, quarantine auth-broken providers, handle rate limits and network failures differently, fail fast on invalid requests, and expose provider health in status output.

## Best Possible Solution

Define a documented provider/profile health model that separates hard auth, transient refresh, rate-limit, network, billing, quota, schema, and invalid-request failures; apply it consistently in fallback selection; decide invalid-request fail-fast semantics; and expose a redacted provider health snapshot in normal status while keeping deeper probe details under models status.
