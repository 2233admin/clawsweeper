## Summary

Keep this issue open. Current main has a real failure-class-aware fallback foundation: 401/403 classify as auth, auth-cooldowned providers are skipped across same-provider model candidates, rate-limit/overloaded/timeout cooldowns can still be probed or walked differently, and `models status --json` exposes unusable auth profiles. The remaining request is more specific: the reported `OAuth token refresh failed` case is treated as generic `auth` with short profile cooldowns, not a 30-minute provider quarantine, and the documented `openclaw status`/`/status` surface still does not show provider health/quarantine state as requested.

## Best Possible Solution

Keep this issue open as a focused follow-up to the completed fallback foundation. The best path is a maintainer decision on hard-auth classification policy first, then a narrow implementation that distinguishes permanent/re-auth-required OAuth failures from transient refresh timeout/contention, applies an appropriate provider/profile quarantine window, and exposes provider health in the intended status surface without leaking secrets. If maintainers decide the current short cooldown plus `models status --json` behavior is the intended policy, the issue should be closed only after that is documented explicitly.
