## Summary

Keep open. Current main still rejects a sessions_send payload containing both sessionKey and label before target resolution, which is the reported failure mode, and the latest release still contains the same guard. There is also an open implementation path referenced from the discussion, so this issue should remain open until maintainers land one canonical fix or deliberately choose a different precedence policy.

## What This Changes

The issue requests that sessions_send stop failing when client or UI argument assembly redundantly supplies label alongside an intended sessionKey target.

## Best Possible Solution

Land one canonical sessions_send fix in src/agents/tools/sessions-send-tool.ts that treats an explicit sessionKey/sessionId target as authoritative when redundant label or agentId fields are present, preserves label-only resolution and all visibility/A2A policy checks, updates the tool description to guide callers, and adds focused regression tests in src/agents/tools/sessions.test.ts for redundant and conflicting selector cases.
