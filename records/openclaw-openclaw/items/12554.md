## Summary

Current main has useful scoped exec/process building blocks, but it still lacks the requested session-wide operator surface to list all running exec processes for an OpenClaw session and terminate them as a unit. The existing session abort path only delegates to chat abort, the HTTP kill route is subagent-specific, and the CLI/docs expose no `sessions exec-list`, `sessions kill`, or `abort --kill-exec` equivalent.

## Best Possible Solution

Keep this issue open for maintainer design of an additive Gateway/RPC/CLI operator surface: resolve the canonical session key, list managed exec/process runs for that session scope, terminate matching runs through supervisor cancellation plus process-tree fallback, return per-run status, document the contract, and cover same-session kill, unrelated-session isolation, background exec, HTTP/RPC/CLI validation, node-host/sandbox behavior, and subagent cancellation interactions.

## Risks / Open Questions

- A session-wide exec list/kill API must preserve session isolation and authorization; one operator must not be able to inspect or terminate unrelated sessions' exec processes.
- The implementation should not make normal abort/stop kill intentional background exec by default; the kill behavior should be an explicit operator action or explicit abort flag.
- Node-host and sandbox execution may need separate handling because session-scoped process records may live outside the Gateway process for remote `system.run` paths.
- The existing POST /sessions/:sessionKey/kill route already means subagent kill; any new surface should avoid ambiguous semantics or explicitly version/extend that route.
