## Summary

Keep open. Current main still sends Feishu text/card, image/file, and initial streaming-card messages through single Feishu SDK create/reply calls, while non-zero message API responses are converted directly into exceptions instead of being classified for bounded retry/backoff. The related #14206 remains a broader pacing/rate-limiting enhancement and does not supersede this narrower delivery-retry bug.

## What This Changes

Add bounded retry/backoff handling for Feishu message-send API rate-limit and flow-control failures.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry helper around outbound message create/reply delivery, covering text/post/card sends, image/file sends after upload, and initial streaming-card sends while preserving withdrawn-reply fallback behavior and using explicit retryable Feishu/API codes and timing hints.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by mocking a Feishu SDK thrown error or non-zero API response for a rate-limit code in the current send helpers; current main performs one create/reply call and throws without another attempt.

## Solution Assessment

Yes. A narrow Feishu plugin retry wrapper is the best direction; a global channel retry policy would be broader and riskier for non-idempotent sends.
