## Summary

Keep open. Current main still has one-shot Feishu outbound message delivery for text/post messages, cards, image/file sends, and initial streaming-card sends; retryable rate-limit or flow-control failures are still thrown rather than retried. The related pacing request #14206 is complementary but not a substitute, and #64448 is an important duplicate-delivery/idempotency caution for any retry implementation.

## What This Changes

Add Feishu-plugin retry handling for outbound message sends that fail with rate-limit or flow-control API errors such as 2200, 11232, or 429.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry/backoff helper around outbound message create/reply delivery. It should classify retryable Feishu response bodies and thrown SDK/Axios errors, preserve withdrawn-reply fallback and thread-reply safety behavior, cover text/post, card, media, and initial streaming-card sends, and use provider-supported idempotency/deduping where available instead of a broad core retry policy.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by injecting a retryable Feishu SDK response or thrown error into the current send helpers; current main performs one create/reply call and rejects without a second attempt.

## Solution Assessment

Yes. A narrow Feishu-owned retry wrapper is the best direction because message sends can be non-idempotent, so a broad shared retry layer would risk duplicate deliveries or behavior changes in unrelated plugins.
