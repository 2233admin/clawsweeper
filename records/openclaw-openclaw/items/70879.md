## Summary

Keep open. Current main still sends Feishu text, card, image, file, and streaming-card messages through direct Feishu SDK create/reply calls, with non-zero message API responses converted to immediate exceptions rather than a bounded retry path. Feishu backoff handling exists for typing reactions and Feishu docs describe 429/99991400/11232 rate-limit behavior, but the outbound message delivery path does not classify or retry those failures.

## Best Possible Solution

Keep this open and fix it inside the bundled Feishu plugin. Add a Feishu-owned bounded retry wrapper around outbound message create/reply delivery for text, card, image, file, and the initial streaming-card send; classify explicit Feishu rate-limit/flow-control responses, honor x-ogw-ratelimit-reset or retry-after hints, consider Feishu message uuid where retry safety requires dedupe, and add focused tests for retry/no-retry behavior without changing global channel retry policy.

## Clownfish Work Prompt

Fix openclaw/openclaw#70879 in the bundled Feishu plugin. Current main sends Feishu text/card/image/file messages and the initial streaming card through one-shot client.im.message.create/reply calls, then throws on non-zero API responses. Add a Feishu-owned bounded retry wrapper for outbound message delivery only. Cover sendMessageFeishu, sendCardFeishu, sendImageFeishu, sendFileFeishu, and the initial streaming-card send path if it uses im.message.create/reply. Consider the Feishu message uuid field for safe dedupe if retrying cases where the provider may accept duplicate create/reply calls. Do not add broad regex retries for ambiguous ECONNRESET/timeout-after-send failures that could duplicate non-idempotent sends. Reuse existing plugin-sdk retry-runtime helpers if suitable, but keep the fix inside extensions/feishu unless a small existing public helper already fits. Add focused Vitest coverage for retryable non-zero responses, thrown 429/response.data.code shapes, retry-after/reset delay handling, no retry on permanent errors, and media create/reply behavior. If user-facing retry config is added, update extensions/feishu config schema/tests, docs/channels/feishu.md, docs/concepts/retry.md, and a one-line CHANGELOG.md Fixes entry with Thanks @AxelHu.
