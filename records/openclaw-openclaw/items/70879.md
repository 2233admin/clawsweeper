## Summary

Keep open. Current main still has one-shot Feishu outbound message create/reply calls for text, card, media, and initial streaming-card delivery, while non-zero Feishu message API responses are converted directly into exceptions. The related pacing request #14206 is useful context, but it tracks proactive rate limiting rather than this narrower retry-on-rate-limit delivery failure.

## What This Changes

Add bounded retry/backoff in the Feishu plugin for outbound message sends that fail with Feishu rate-limit or flow-control API errors.

## Best Possible Solution

Add a Feishu-plugin-owned bounded retry helper for outbound message create/reply delivery, covering text/post, cards, image/file sends after upload, and initial streaming-card sends while preserving withdrawn-reply fallback behavior and deliberately classifying Feishu-specific retryable responses and thrown HTTP/SDK errors.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by injecting a Feishu SDK response or thrown error for a rate-limit/flow-control code into the current send helpers; current code performs one create/reply call and throws without a second attempt.

## Solution Assessment

Yes. A narrow Feishu plugin retry wrapper is the best direction because it can respect Feishu-specific response codes, timing, and reply fallback semantics; a global channel retry policy would be broader and riskier for non-idempotent sends.
