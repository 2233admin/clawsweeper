## Summary

Keep open. Current main and the latest release still have the reported QQ Bot SecretRef gap: a literal `secretref:/...` value is accepted as a plain non-empty string, can flow into the QQ token request, and QQ Bot still lacks the channel secret-contract artifact used by runtime SecretRef materialization. The related open PR #74772 is the right focused implementation path, but it has not landed.

## What This Changes

The issue asks OpenClaw to correctly resolve or reject QQ Bot `clientSecret` values configured as `secretref:/...` instead of passing that marker to the QQ OpenAPI after upgrade.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds the channel secret contract, runtime assignment/materialization coverage, generated SecretRef credential-surface docs, and explicit policy for literal `secretref:/...` values so the raw marker cannot reach QQ as `clientSecret`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or an account-level `clientSecret` as `secretref:/QQBOT_...`; current main accepts it as a string, returns it as available, and can post it to the QQ token endpoint.

## Solution Assessment

No implemented solution is present on main. The narrow maintainable direction is the open QQ Bot plugin plus shared SecretRef contract/validation fix represented by PR #74772, with the literal-marker policy settled before merge.
