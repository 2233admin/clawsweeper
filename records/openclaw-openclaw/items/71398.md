## Summary

Keep open. Current main still has the reported QQ Bot SecretRef gap: a literal `secretref:/...` value is accepted as a normal non-empty string, QQ Bot can pass that value into the QQ access-token request, and QQ Bot still lacks the channel secret contract/registry coverage used by the shared SecretRef tooling. The prior maintainer-posted Codex review also kept this open, and I did not find a later current-main fix or superseding PR in the provided context or local history.

## What This Changes

The issue reports that QQ Bot account client secrets configured as `secretref:/...` are not resolved after upgrading to v2026.4.23, causing QQ OpenAPI token requests to receive the literal marker and fail authentication.

## Best Possible Solution

Add a QQ Bot-owned SecretRef contract and runtime assignment coverage for `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`, align the generated/public SecretRef credential surface and QQ Bot docs, and add regression tests for structured refs plus the literal marker case once the legacy-marker policy is decided.
