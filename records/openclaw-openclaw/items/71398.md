## Summary

Keep open. Current main still leaves the reported QQ Bot SecretRef path unsafe to close: a literal `secretref:/...` value is accepted as a configured non-empty string, there is no `secretref:/` parser or QQ Bot channel secret-contract artifact on main, and the related focused fix PR #74772 is still the active implementation lane rather than a merged fix.

## What This Changes

The issue reports that QQ Bot `clientSecret` values configured as `secretref:/...` are passed through as raw strings after upgrading to v2026.4.23, causing QQ token requests to fail with `invalid appid or secret`.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds the channel secret contract, runtime materialization coverage, generated SecretRef surface docs, and a clear migration or validation policy so raw SecretRef marker strings cannot reach the QQ token endpoint as `clientSecret`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to set `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` to `secretref:/QQBOT_...`; current main treats it as a non-empty string and can post that exact value in the QQ token request.

## Solution Assessment

Yes. Keeping this open and finishing the focused QQ Bot SecretRef PR is the best current path; starting a parallel ClawSweeper repair would duplicate the active implementation lane.
