## Summary

Keep open. Current main still has the reported QQ Bot SecretRef gap for the literal `secretref:/...` form: it is accepted as a normal non-empty string, can flow into the QQ token request, and QQ Bot still lacks the channel secret-contract coverage needed for normal runtime materialization; open PR #74772 is the right follow-up path, not a reason to close before merge.

## What This Changes

The issue reports QQ Bot failing after upgrade when `channels.qqbot.clientSecret` or an account-level `clientSecret` is configured as `secretref:/QQBOT_...` instead of a resolved secret value.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract coverage, runtime materialization, credential-surface docs, and an explicit policy for raw SecretRef-looking marker strings so they cannot be posted to QQ as credentials.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure QQ Bot `clientSecret` as `secretref:/QQBOT_...`; current main accepts it as a non-empty plaintext string and can pass that value into the QQ access-token request.

## Solution Assessment

Yes. Keeping this issue open is the right current action because the narrow maintainable fix belongs in the QQ Bot plugin plus shared SecretRef policy coverage, and the already-open PR should be reviewed or iterated rather than bypassed with a separate cleanup close.
