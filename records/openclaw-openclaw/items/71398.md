## Summary

Keep open. Current main still treats a literal `secretref:/...` QQ Bot `clientSecret` as a normal non-empty string, QQ Bot can pass that value to the token endpoint, and the related PR #74772 is open but not merged.

## What This Changes

Restore QQ Bot `clientSecret` SecretRef handling, including the reported `secretref:/...` configuration, so raw SecretRef-looking markers are not sent to QQ.

## Best Possible Solution

Land PR #74772 or an equivalent QQ Bot-owned fix that adds top-level and account-level `clientSecret` secret-contract targets, runtime materialization, docs/credential-surface updates, and an explicit legacy-marker policy so raw SecretRef-looking strings cannot be sent to QQ.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `channels.qqbot.clientSecret` or `channels.qqbot.accounts.<id>.clientSecret` as `secretref:/QQBOT_...`; current main treats that non-empty string as plaintext and can post it as the QQ `clientSecret`.

## Solution Assessment

Yes. Keeping this open is the right path because the fix belongs in the QQ Bot plugin plus the shared SecretRef validation/credential-surface policy; the open policy choice is whether the legacy marker should migrate/resolve or fail fast.
