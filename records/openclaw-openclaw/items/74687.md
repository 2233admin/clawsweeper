## Summary

Keep open. Current main still has the unguarded synthetic restart-recovery replay path, and this PR is a narrow implementation with focused coverage; I did not find a definite blocking correctness or security issue. The remaining question is maintainer acceptance of the behavior boundary that any persisted channel/origin/last-route metadata should force manual recovery.

## What This Changes

The PR adds channel-backed-session detection to main-session restart recovery, fails those restart-aborted sessions for manual recovery, and adds a Vitest regression for a Discord-backed session.

## Best Possible Solution

Land a narrow restart-recovery fix that prevents automatic replay for externally delivered/channel-backed main sessions, preserves automatic recovery for local non-channel sessions, and keeps focused agents-core regression coverage for both paths.
