## Summary

Keep open. Current main still lacks the proposed per-field metadata length limits, and this PR is not obsolete. The security hardening is directionally useful, but the PR should be revised before merge because mapped hook agent dispatches can still carry the same oversized metadata fields outside `normalizeAgentPayload()`.

## Best Possible Solution

Keep this PR open and revise it into a shared hook-agent metadata validator used by both direct `/hooks/agent` normalization and mapped `/hooks/<name>` agent dispatch. Add targeted tests for direct and mapped over-limit rejection, exact-boundary acceptance including `agentId` and `sessionKey`, and unchanged behavior for the existing overall body-size limit.

## Risks / Open Questions

- Landing the PR as written would leave mapped `/hooks/<name>` agent actions able to pass oversized `name`, `sessionKey`, `to`, `model`, and `thinking` values into replay-key construction, dispatch, logging, storage, and delivery paths.
- The PR’s exact-boundary test, per the existing review comments, should include `agentId` and `sessionKey` so all proposed capped fields are covered at the boundary.
