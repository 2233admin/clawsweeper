## Summary

Keep open. Current main still lacks the requested operator-triggered single-channel reload command or Gateway RPC; the existing code only exposes status/start/logout plus automatic config, secrets, and health-monitor restart paths. Related open reload items are adjacent and do not supersede the manual reload surface.

## What This Changes

Request an operator command or Gateway action that reloads one named channel provider without restarting the Gateway or interrupting unrelated channels and sessions.

## Best Possible Solution

Add an additive admin-scoped `channels.reload` Gateway method and `openclaw channels reload --channel <provider> [--account <id>]` CLI wrapper that reuses existing channel stop/start lifecycle primitives, touches only the requested channel/account, documents the protocol and CLI surface, and adds focused tests plus a user-facing changelog entry.

## Reproduction Assessment

Not applicable as a bug reproduction because this is a feature request. The current-main gap is high-confidence from static source and docs inspection: no CLI command, Gateway method, protocol schema, validator, or admin scope exists for manual channel reload.

## Solution Assessment

Yes, the maintainable direction is an additive `channels.reload` method in the existing plural `channels.*` lifecycle surface, scoped as an admin operation and backed by the existing stop/start channel manager. An ad hoc singular `channel.reload` action would be less consistent with current Gateway naming.
