## Summary

Keep open. Current main has automatic per-channel restart paths for config hot reload, secrets reload, and health monitoring, but it still does not expose the requested operator-triggered channel reload CLI command or Gateway RPC method; the existing `channels.start` path is idempotent and is not a forced reconnect for an already-running account.

## What This Changes

Add a targeted channel reload command or Gateway method that reconnects one provider/account without restarting the gateway process or interrupting unrelated channels and sessions.

## Best Possible Solution

Add an admin-scoped `channels.reload` Gateway method plus a matching `openclaw channels reload --channel <provider> [--account <id>]` CLI wrapper that reuses the existing channel lifecycle, preserves unrelated channels and sessions, returns structured status, and updates protocol schemas, docs, and focused tests.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The current-main gap is high-confidence from source and docs inspection: the CLI, method list, admin scopes, protocol schemas, and channel docs do not expose manual channel reload.

## Solution Assessment

Unclear pending maintainer API direction. A targeted reload surface is likely the right shape, but it should align with existing account-scoped lifecycle semantics and Gateway admin authorization rather than adding an ad hoc `channel.reload` action.
