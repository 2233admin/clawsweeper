## Summary

Keep this issue open. Current main has adjacent automatic channel restart behavior for config/secret reloads and health-monitor recovery, but it still does not expose the requested manual `openclaw channels reload <provider>` command or a `channels.reload`/`channel.reload` Gateway RPC. The existing `channels.start` path starts a stopped account but does not stop and reconnect an already-running channel, so it is not equivalent to the requested operator-triggered reload.

## Best Possible Solution

Add an admin-scoped `channels.reload` Gateway method plus an `openclaw channels reload --channel <provider> [--account <id>]` CLI wrapper. It should reuse the existing channel manager stop/start lifecycle, apply the same active-work deferral semantics where appropriate, return structured channel/account status, and update protocol schemas/scopes, CLI docs, Gateway docs, and focused tests. If maintainers do not want a manual reload surface, document the supported alternatives instead.

## Risks / Open Questions

- A manual reload implementation should define account-scoped behavior, admin authorization, structured status, and whether it reuses the active-work deferral policy from config hot reload; otherwise it could interrupt active channel work unexpectedly.
- Maintainers may decide the automatic config hot reload, `secrets.reload`, health-monitor recovery, and relink/login flows are sufficient, but that would need an explicit product/docs decision rather than a cleanup close.
