## Summary

Current main still lacks the requested `MEDIA:/path#doc` parsing/stripping and WhatsApp document-forcing behavior. The existing `forceDocument` / `asDocument` path remains Telegram-only in code and docs, while WhatsApp direct sends and auto-replies still classify image MIME as image and route it to the image payload. Related PR #42044 is closed unmerged prior art, not shipped behavior.

## Best Possible Solution

Keep this issue open. The best fix is to add a shared outbound media delivery hint, either by parsing a reserved `#doc` suffix before media resolution or by carrying structured `asDocument` metadata, then wire it through WhatsApp direct sends and auto-reply delivery so image MIME can be preserved while Baileys receives a document payload. Add parser/loader tests, WhatsApp send-path tests, auto-reply tests, and WhatsApp docs; use #42044 as prior art, not as a close reason.

## Risks / Open Questions

- A future implementation needs explicit semantics for real URL fragments, local filenames containing `#`, filename derivation, and safe local path resolution.
- WhatsApp has at least two outbound paths here: direct sends and auto-reply delivery. Updating only one would leave inconsistent behavior.
- The existing `forceDocument` / `asDocument` wording is currently Telegram-only, so broadening it should update the CLI/tool docs and tests together.
