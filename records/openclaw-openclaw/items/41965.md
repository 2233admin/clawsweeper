## Summary

Keep open. Current main still does not implement a supported `MEDIA:/path#doc`, WhatsApp `forceDocument`/`asDocument`, or equivalent outbound media hint that sends image MIME as a WhatsApp document attachment. The direct implementation candidate, #42044, was closed unmerged, and the current code still routes image MIME through image/photo payloads in both normal outbound and auto-reply paths.

## Best Possible Solution

Keep this issue open. The best fix is to add a shared outbound media delivery hint, either by parsing a reserved `#doc` suffix before media resolution or by carrying structured `asDocument` metadata, then wire it through WhatsApp normal send and auto-reply delivery so image MIME can be preserved while Baileys receives a document payload. Add parser/loader tests, WhatsApp send-path tests, auto-reply tests, and WhatsApp docs. #42044 remains useful prior art but should not close the issue because it did not merge.

## Risks / Open Questions

- A future `#doc` suffix needs precise semantics for real URL fragments, local filenames containing `#`, filename derivation, and local-read safety checks.
- WhatsApp has both normal outbound and auto-reply media paths; implementing only one would leave inconsistent user-visible behavior.
- The existing `forceDocument` and `asDocument` names are parsed generically but documented as Telegram-only, so a WhatsApp fix should update the contract, docs, and tests together.
