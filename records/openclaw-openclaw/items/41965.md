## Summary

Keep open. Current main still does not provide a supported `MEDIA:/path#doc`, `forceDocument`, `asDocument`, or equivalent hint that makes WhatsApp deliver image media as document attachments. The only direct implementation candidate in the provided discussion, #42044, was closed unmerged.

## Best Possible Solution

Keep this issue open. The best fix is to add a shared outbound media delivery hint, either by parsing a reserved `#doc` suffix before media resolution or by carrying structured `asDocument` metadata, then wire it through normal WhatsApp send and auto-reply delivery so image MIME can still be preserved while Baileys receives a document payload. Add parser/loader tests, WhatsApp image-as-document tests for both send paths, and WhatsApp docs. #42044 is useful prior art but should not close the issue because it did not merge.

## Risks / Open Questions

- A future fix needs a precise reserved suffix or structured metadata contract so real URL fragments, local filenames containing `#`, filename derivation, and local-read safety checks remain well-defined.
- WhatsApp has both normal outbound and auto-reply media paths; implementing only one would leave the user-visible behavior inconsistent.
- The existing generic-looking `forceDocument` and `asDocument` names are easy to mistake for a WhatsApp solution, but current main only wires that behavior through Telegram-facing paths.
