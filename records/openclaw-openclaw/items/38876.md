## Summary

Keep open. Current main still has string-only `extractToolText` handling for embedded Control UI tool-result blocks, so array-shaped `text` or `content` payloads like the linked `content[0].text` case still fall through to `No output`. The PR remains the active implementation path, but its current branch needs the security review concern addressed before merge because it joins all discovered fragments without a count or size bound.

## Best Possible Solution

Keep this PR open as the canonical implementation candidate. Amend or review the branch to bound collected tool-output fragments by count and total characters, preserve the array-shaped `text` and `content` regression tests and changelog credit, then land after maintainer/security review and normal changed-surface validation. If another change lands first, close this PR only after current main contains bounded array extraction with regression coverage.
