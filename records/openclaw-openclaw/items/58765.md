## Summary

Keep open. Current main wires `memorySearch.outputDimensionality` through memory-search config and provider requests, but the built-in local GGUF provider still ignores it and returns the full normalized vector from `node-llama-cpp`; the local provider cache key also omits dimensionality, so changing the setting would not force the intended reindex/cache separation for local embeddings.

## What This Changes

The issue requests `memorySearch.outputDimensionality` support for local GGUF embedding models, including truncation before storage and reindexing when the configured dimension changes, with query-instruction support mentioned as a bonus.

## Best Possible Solution

Implement local GGUF dimensionality truncation in the local embedding provider, include the configured dimension in the local provider runtime/cache identity so dimension changes trigger a full reindex through the existing provider-key path, add focused unit tests for query and batch vectors, and document the local support. Treat configurable query instructions as a follow-up API decision unless a maintainer explicitly expands this issue’s scope.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to instantiate `createLocalEmbeddingProvider` with a mocked `node-llama-cpp` vector and `outputDimensionality: 2`; current code ignores the option and returns the full normalized vector. I did not run tests because this review was read-only, but the existing local provider test harness already supports this mock-based check.

## Solution Assessment

Yes, for the main requested fix. The narrow maintainable solution is local-provider truncation plus cache/reindex identity and tests; adding a new `queryInstruction` config at the same time would be a broader config/API change and is better handled separately unless maintainers choose otherwise.
