## Summary

Keep open. Current main still does not provide the proposed `gateway watchdog` command or the PR's `startup-error.txt` file, and the PR branch remains an open implementation with blocking packaging, CLI validation, spawn-error, and token-default defects plus product overlap with the broader rescue-watchdog work.

## What This Changes

The PR adds startup-error file persistence for fatal CLI/Gateway startup failures and registers a `openclaw gateway watchdog` subcommand that spawns a Bash script to poll Gateway health, restart the Gateway, log events, and send desktop notifications.

## Best Possible Solution

Land a rebased and reviewed implementation only after the watchdog boundary is clear. The final shape should either split startup-diagnostic persistence from a packaged first-class watchdog command, or be superseded by the broader rescue-watchdog service; any public watchdog command should use current fatal-hook/stability diagnostics, ship its runtime asset, validate CLI input before spawning, honor documented auth defaults, document the user-facing command, and include focused CLI/infra/script tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Ship the watchdog script with the packaged CLI:** `src/cli/gateway-cli/run.ts:573`
  - body: The command resolves `scripts/openclaw-watchdog.sh` at runtime, but the npm package `files` list does not include that script. Installed package users would get a `gateway watchdog` command that points at a missing file unless the script is moved into a shipped runtime asset or explicitly packaged.
  - confidence: 0.94
- **[P1] Read the watchdog token from the environment default:** `scripts/openclaw-watchdog.sh:22`
  - body: The CLI advertises `OPENCLAW_GATEWAY_TOKEN` as the default and forwards it in the child environment, but the script initializes `TOKEN` to an empty string and only updates it from `--token`. Token-protected gateways that rely on env-based auth will be treated as unhealthy and can enter restart flows incorrectly.
  - confidence: 0.86
- **[P2] Clear startup errors after startup actually succeeds:** `src/cli/gateway-cli/run.ts:463-466`
  - body: The cleanup runs only after `await startLoop()` returns, but the Gateway loop is designed to stay running until shutdown or exit. On normal successful startup this block never executes, leaving stale `startup-error.txt` content visible after the Gateway has recovered.
  - confidence: 0.88
- **[P2] Reject invalid watchdog numeric options before spawning:** `src/cli/gateway-cli/run.ts:585-590`
  - body: Both numeric parsers can return `NaN` for inputs like `--interval foo` or `--max-attempts foo`, and that value is forwarded into shell arithmetic. The command should fail fast with a Commander validation error instead of starting a script that aborts with an opaque shell error.
  - confidence: 0.9
- **[P2] Handle child process spawn errors with an error listener:** `src/cli/gateway-cli/run.ts:615-623`
  - body: Wrapping `spawn("bash", ...)` in try/catch does not handle normal spawn failures such as `ENOENT`; Node emits those on the returned `ChildProcess` as an `error` event. Without an `error` listener, missing Bash can become an uncaught exception instead of a controlled CLI error.
  - confidence: 0.86
