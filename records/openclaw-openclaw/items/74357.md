## Summary

Keep open. Current main still applies the private-network-allowing SSRF policy through the shared trusted web-tools/web-search path used by bundled providers and SDK exports, and the linked implementation PR remains open rather than merged. Because this is SSRF-sensitive and plugin-facing, closure should wait for maintainer/security review and a landed fix.

## What This Changes

The issue asks to stop applying a private-network-allowing SSRF policy to all bundled web-search providers and instead make private-network access an explicit self-hosted provider opt-in.

## Best Possible Solution

Land or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted/private-network opt-in for SearXNG-style providers, additive SDK exports for third-party self-hosted plugins, and focused tests proving public providers block private redirects while SearXNG remains usable.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists on current main: `withTrustedWebToolsEndpoint` applies `dangerouslyAllowPrivateNetwork`, the shared web-search helpers and bundled providers route through it, and the SSRF guard skips private-network checks for that policy flag.

## Solution Assessment

Yes. Splitting strict public-provider behavior from an explicit self-hosted/private-network opt-in is the narrowest maintainable direction; the open PR appears aimed at that boundary but still needs security review and merge/supersession.
