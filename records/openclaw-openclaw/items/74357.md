## Summary

Keep open. Current main still applies the private-network-allowing SSRF policy through the shared trusted web-tools endpoint used by web-search helpers and plugin SDK exports, and the linked implementation path, PR #74360, remains open/unmerged. Because this is SSRF-sensitive and public SDK-facing, it should stay open for maintainer/security review rather than conservative cleanup.

## What This Changes

The issue asks to split web-tools SSRF policy so public web-search providers use a strict default while self-hosted SearXNG-style providers explicitly opt into private-network access.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted/private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while self-hosted SearXNG remains usable.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `withTrustedWebToolsEndpoint` always passes a private-network-allowing policy, and the shared web-search helpers used by bundled providers call that endpoint.

## Solution Assessment

Yes for the requested direction, but the exact patch still needs review. A strict default plus explicit self-hosted opt-in is the narrowest maintainable shape because it preserves SearXNG while removing broad private-network access from public provider endpoints.
