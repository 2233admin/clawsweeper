## Summary

Keep open. Current main still routes the shared trusted web-search/web-tools helpers through an SSRF policy that allows private-network targets, and the linked closing PR remains open rather than merged. Because the report is security-sensitive and touches plugin-facing helpers, it should stay open for maintainer/security review.

## What This Changes

The issue asks OpenClaw to make web-search SSRF policy strict by default and reserve private-network access for explicitly self-hosted providers such as SearXNG.

## Best Possible Solution

Review, land, or supersede PR #74360 with a strict-by-default trusted web-tools/web-search policy, an explicit self-hosted/private-network opt-in for SearXNG-style providers, additive SDK exports, and tests proving public providers block private redirects while self-hosted SearXNG remains usable.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `withTrustedWebToolsEndpoint` still passes `dangerouslyAllowPrivateNetwork: true`, the shared web-search helpers still call it, and the current unit test asserts that policy.

## Solution Assessment

Yes, the issue’s proposed split is the narrowest maintainable direction: keep cloud/public providers on strict SSRF defaults and add an explicit self-hosted endpoint for SearXNG-style private-network providers. The remaining work is to review the open PR’s exact SDK/test coverage rather than close this issue early.
