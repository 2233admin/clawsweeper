## Summary

Current main still has the reported model-case gap. The local `infer model run` path resolves provider casing, but the Anthropic forward-compat dynamic resolver matches `claude-opus-4-7` case-insensitively and then preserves the user-supplied model casing in the runtime model sent onward. That means the issue is not implemented, not a duplicate of the already-closed empty-prompt report, and still represents actionable OpenClaw CLI/provider-resolution work.

## Best Possible Solution

Keep this issue open. The best fix is to add provider/catalog-aware validation in the `infer model run` simple-completion selection path: either canonicalize a case-only match to the catalog/provider-owned canonical id before dispatch, or reject it before provider dispatch with `Unknown model: <input>` and ideally a case-only suggestion. Add regression coverage for mixed-case Anthropic refs and preserve existing behavior for providers with intentionally case-sensitive model ids.
