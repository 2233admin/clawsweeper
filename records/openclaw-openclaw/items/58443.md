## Summary

Keep open. Current main contains several partial duplicate-delivery fixes for inbound dedupe, Telegram offset replay, and Slack same-event races, but the subagent announce path still matches the remaining failure described in the discussion: announce idempotency is derived from the mutable child run id while orphan recovery can remap the same logical child session to a fresh run id.

## What This Changes

The issue reports duplicate gateway injection or delivery of the same inbound message or stale assistant completion across Telegram, Slack, and subagent announcement recovery paths.

## Best Possible Solution

Keep the issue open and add a narrow subagent announce lineage fix: store a stable logical announce/completion id on each subagent run, preserve it across orphan recovery and replaceSubagentRunAfterSteer remaps, and use that identity for direct and queued announce idempotency while leaving the existing Telegram, Slack, and core inbound guards intact.

## Reproduction Assessment

Yes. The discussion gives timestamped field reproductions, and current main has a static reproduction path for the subagent case: remap an orphaned child session to a new run id, then observe that the announce id changes because it is derived from childRunId.

## Solution Assessment

No. Current main is a partial solution: it addresses several narrower replay paths, but using mutable childRunId for subagent announce idempotency is not the best remaining fix because orphan recovery deliberately changes run ids.
