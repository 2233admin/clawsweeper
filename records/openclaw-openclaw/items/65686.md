## Summary

Keep open. Current main has channel-owned final-text collapse for Telegram and Discord, but it still lacks the requested operator-configurable full/last/none isolated-cron outbound payload policy; config/schema/docs omit it, resolver/dispatch code still preserve all payloads by default, and the linked implementation PR is closed unmerged.

## What This Changes

Add a configurable isolated-cron outbound delivery payload policy that defaults to full delivery and can instead send only the last resolved payload or suppress runner channel delivery.

## Best Possible Solution

Add a typed isolated-cron runner outbound payload policy with default `full`, `last` applied after existing payload resolution, and `none` that suppresses runner outbound delivery without synthetic-text fallback. Keep channel plugin final-text preferences compatible, update config schema/help/generated docs and cron docs, and add focused resolver/dispatch tests.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: multiple text payloads without a channel final-text preference still resolve and dispatch as multiple outbound payloads, and current tests assert that behavior for non-final-preference channels.

## Solution Assessment

Yes, the narrow maintainable path is a typed cron payload-mapping policy in the existing resolver/dispatch flow. Existing `delivery.mode=none` is not a complete substitute because it controls runner fallback delivery broadly and preserves message-tool availability rather than selecting `full`/`last`/`none` after resolved payload selection.
