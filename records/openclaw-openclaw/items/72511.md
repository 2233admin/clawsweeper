## Summary

Keep this PR open for maintainer review. Current main still replays transcript-rewrite suffix entries without an idempotency guard, and the linked overflow-recovery bug remains open, but the proposed patch introduces a broader role+content message identity that can delete legitimate repeated turns during any transcript rewrite.

## What This Changes

This PR adds SHA-256 based suffix replay deduplication to `rewriteTranscriptEntriesInSessionManager`, remaps skipped duplicate ids, and adds regression coverage for duplicate messages, compactions, custom entries, and file-level replay.

## Best Possible Solution

Revise the PR so replay cleanup targets the polluted overflow-recovery shapes without globally collapsing same-role, same-content messages. A good end state would preserve legitimate repeated prompts, assistant messages, and tool results, add regression coverage for those repeats, and keep the broader duplicate-transcript tracking under #69208 separate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve legitimate repeated messages during replay:** `src/agents/pi-embedded-runner/transcript-rewrite.ts:41`
  - body: The new message identity is only `role + sha(content)` and is remembered across the whole replayed suffix. Because transcript rewrites are also used outside the exact polluted overflow case, a normal repeated prompt, assistant response, or tool result after the rewritten entry can be skipped and remapped away. Please scope message dedupe to the pollution signal or include enough per-turn identity/windowing to preserve legitimate repeated turns.
