## Summary

Keep open. Current main still has the reported Feishu startup shape: startup bot-identity probing can wait on 30-second Feishu SDK/API timeouts before transport starts, and the focused fix path is the still-open #74187 rather than a merged implementation.

## What This Changes

The issue requests that Windows Gateway startup avoid waiting 30 seconds on Feishu/Lark API token or probe calls and bring the Feishu transport up promptly.

## Best Possible Solution

Land or revise #74187 so Feishu startup uses bounded startup-safe bot-identity and SDK token/WebSocket waits, starts account transport promptly when identity lookup is slow, recovers identity in the background, and adds focused regression coverage for the Windows startup timeout path.
