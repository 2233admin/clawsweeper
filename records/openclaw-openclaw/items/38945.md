## Summary

Keep open. The reported Unicode tokenizer problem is still valid on current main, but this PR is stale against the active implementation: it changes old `src/memory` files while user-facing memory search now runs through `extensions/memory-core`, where MMR, FTS, and manager-search tokenization still need the Unicode/NFC port.

## What This Changes

The PR updates the old `src/memory` MMR and FTS tokenizer helpers and tests to NFC-normalize input and tokenize Unicode letters, combining marks, numbers, and underscores.

## Best Possible Solution

The active `memory-core` tokenizers should NFC-normalize consistently and support Unicode letters, marks, numbers, and underscores across MMR, FTS MATCH query building, trigram planning, and fallback matching, with focused regressions proving accented Latin, combining-mark scripts, NFC/NFD equivalence, and existing CJK behavior.
