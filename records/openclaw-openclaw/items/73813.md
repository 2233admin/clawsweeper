## Summary

Keep open. Current main still finalizes the isolated cron run from the first yielded parent result: the embedded runner exposes yielded metadata, but cron does not propagate or interpret it, and the timer clears running state and emits `finished` as soon as the core promise resolves. Descendant-aware delivery can wait in some text announce paths, but the cron outcome/duration lifecycle remains incomplete, especially for `delivery.mode: "none"` and non-text delivery paths.

## What This Changes

The issue asks for isolated cron jobs that spawn a subagent and call `sessions_yield` to avoid recording a terminal `ok` status and short duration before the child and resumed parent finish.

## Best Possible Solution

Define yielded isolated cron runs as nonterminal, or as an explicit pending/yielded lifecycle state, and emit the cron `finished` event with final `durationMs` only after the resumed parent orchestration reaches a real terminal result. If maintainers choose the additive path first, the remaining duration/finalization gap should stay tracked.

## Reproduction Assessment

Yes. The reported path is reproducible from current code inspection: an isolated cron agent turn can return a yielded embedded result, cron maps that to `ok`, and `executeJob` finalizes duration immediately; a live repro was not run because this review was read-only.

## Solution Assessment

Unclear. The complete fix is likely the deferred-finished lifecycle, while adding `yielded?: boolean` to `CronRunOutcome` is safer and smaller but does not fully satisfy the reported status/duration semantics.
