## Summary

Keep open. Current main has the embedded-runner yield metadata, but cron run bookkeeping still treats the yielded parent turn as an ordinary terminal `ok` result and emits the `finished` event from that first core result. The newer descendant-followup code improves announce delivery, but it is delivery-scoped and does not establish a cron lifecycle contract that keeps the run active until the resumed parent reaches a real terminal outcome.

## What This Changes

The issue requests accurate cron status and duration bookkeeping for isolated cron `agentTurn` jobs that spawn a subagent and call `sessions_yield`.

## Best Possible Solution

Model yielded isolated cron turns as nonterminal cron work until the resumed parent and descendant run graph reaches a true terminal outcome, then emit `finished` and calculate duration from the full orchestration. If maintainers want an incremental step first, add explicit yielded/pending metadata across `CronRunOutcome`, `CronEvent`, run logs, and task reconciliation while keeping the full deferred-lifecycle fix tracked.

## Reproduction Assessment

Yes. The issue provides a concrete isolated cron `agentTurn` with `sessions_spawn`, `sessions_yield`, and a long-running child, and current source inspection shows that yielded metadata is produced by the embedded runner but still collapses to an ordinary cron `ok` outcome.

## Solution Assessment

Unclear until maintainers choose the cron lifecycle contract. Deferred finalization best satisfies the reported status and duration expectation; an additive `yielded` marker is narrower and compatible, but would not by itself make `durationMs` accurate.
