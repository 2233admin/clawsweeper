## Summary

Keep open. Current main has the embedded-runner yielded metadata, but the isolated cron outcome contract still treats the first yielded parent run as a terminal `ok` result and finalizes cron state/duration from that promise. The delivery layer has descendant-aware suppression/waiting for some announce paths, but cron bookkeeping still lacks a yielded/nonterminal outcome or a deferred-finished lifecycle for the full resumed parent orchestration.

## Best Possible Solution

Keep this issue open and have maintainers choose the cron lifecycle direction. The strongest product fix is to make yielded isolated cron runs nonterminal: preserve a running or explicit yielded/pending state while descendant subagents and the resumed parent are still responsible for completion, then emit `finished` and compute `durationMs` only when the resumed parent reaches a true terminal outcome. If maintainers prefer an incremental step, add yielded metadata to the cron outcome contract first, but track the remaining duration/finalization gap explicitly.

## Risks / Open Questions

- A defer-finished fix touches cron terminal bookkeeping used by `deleteAfterRun`, retry/backoff counters, task ledger finalization, `cron_changed` hooks, failure alerts, and run-log history.
- An additive `yielded: true` outcome would improve observability but would not fully satisfy the reported `finished`/`durationMs` semantics unless consumers understand that the duration is partial.
- Current descendant-aware delivery logic may hide the bug for some text announce runs, so regression coverage should include `delivery.mode: "none"`, webhook/structured delivery, and normal announce delivery.
