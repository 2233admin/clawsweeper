## Summary

Keep open. Current main still has the cron-side gap described in the discussion: the embedded runner can mark a yielded turn as paused, but isolated cron finalization still classifies the parent turn as ordinary `ok`, emits `finished`, and writes the pre-yield duration to the run log before descendant completion is represented in cron bookkeeping.

## What This Changes

The issue requests accurate isolated cron run status and duration when an agent turn uses `sessions_yield` to wait for spawned subagents.

## Best Possible Solution

Cron run bookkeeping should model yielded isolated agent turns as nonterminal until the resumed parent and descendants reach a true terminal outcome, so `finished`, `status`, task state, and `durationMs` describe the whole orchestration. If maintainers want a smaller intermediate step, it should be explicit yielded/pending metadata that does not pretend the duration is final.

## Reproduction Assessment

Yes. The issue gives a concrete isolated cron `agentTurn` with `sessions_spawn`, `sessions_yield`, and a long child run; source inspection on current main shows that yielded runner metadata still flows into cron finalization as ordinary `ok`.

## Solution Assessment

Unclear until maintainers choose the cron contract. Deferred finalization best matches the reported expected behavior and metrics, while an additive yielded flag is narrower but would not by itself make `durationMs` accurate.
