## Summary

Keep this issue open. Current main has related startup/import deferral work, but it still does not implement the requested lazy channel connect lifecycle: there is no consumed `channels.lazyConnect` config, gateway sidecars still start channels during startup, the channel manager still invokes each plugin account's `startAccount()`, and the public channel gateway contract has no register/connect split.

## What This Changes

Request adding a default-on `channels.lazyConnect` gateway config so channel plugins register at startup but defer transport connection until first use, with opt-out eager startup and registered-vs-connected status semantics.

## Best Possible Solution

Define an additive lazy channel lifecycle before implementation: a config or successor to `channels.lazyConnect`, plugin capability metadata for channels that can safely defer transport startup, eager fallback for channels that must listen immediately, first-use start behavior where feasible, clear status/readiness semantics, and regression coverage for multi-account bundled and external channel plugins.
