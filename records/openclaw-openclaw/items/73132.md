## Summary

Keep open. Current main still lets the macOS LaunchAgent stop path report a degraded stop after launchctl/bootout without verifying that the gateway listener or underlying Node process is gone; the safer PID/port cleanup exists only in Windows and unmanaged paths. The report is recent, concrete, and narrow enough for a focused fix PR.

## What This Changes

The issue requests that macOS `openclaw gateway stop` fully terminate lingering gateway Node processes and fail if the gateway port remains bound.

## Best Possible Solution

Extend the macOS LaunchAgent stop path to resolve the managed gateway port, wait for the listener to disappear after `launchctl stop` or bootout, terminate only verified OpenClaw gateway listener PIDs with graceful-then-force cleanup, and return a non-zero error if the port remains occupied.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source shows the same failure class: degraded LaunchAgent stop can return without PID or port verification. I did not run live launchd reproduction in this read-only review.

## Solution Assessment

Yes, the requested SIGTERM/SIGKILL plus port-free verification is the right class of fix, but it should be implemented through existing verified gateway-process and port-inspection helpers rather than raw `pgrep` matching.
