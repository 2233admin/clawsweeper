## Summary

Keep open. Current main still has the reported macOS managed-stop gap: the LaunchAgent stop path can return after launchd state checks without proving the OpenClaw gateway listener exited or that the configured port was released.

## What This Changes

The issue requests that macOS `openclaw gateway stop` verify and terminate lingering OpenClaw gateway processes after LaunchAgent stop or bootout so the gateway port is free before reporting success.

## Best Possible Solution

Add a macOS managed-stop postcondition for the Gateway service path: resolve the configured service port, wait for the verified OpenClaw listener to exit after `launchctl stop` or bootout, escalate only verified lingering gateway PIDs, and return a non-zero failure if the port remains occupied or cannot be safely attributed.

## Reproduction Assessment

Yes. The issue provides concrete macOS steps, and current source confirms the loaded LaunchAgent stop path can complete without listener or port verification; a macOS LaunchAgent smoke is still the final acceptance check.

## Solution Assessment

Yes. A narrow lifecycle postcondition is the best maintainable fix; it should reuse the existing verified gateway-process and port-inspection helpers instead of redesigning launchd management or killing arbitrary `node` processes.
