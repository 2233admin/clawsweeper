## Summary

Keep open. Current main still allows the loaded macOS LaunchAgent stop path to report success after launchd stop/bootout state only, without verifying that the gateway listener is gone or clearing a lingering verified gateway process. The issue is recent, concrete, and narrow enough for a focused fix PR.

## What This Changes

The issue asks for macOS `openclaw gateway stop` to terminate or fail when the underlying gateway process/listener survives launchd stop/bootout.

## Best Possible Solution

Extend the macOS managed stop path so it resolves the LaunchAgent gateway port, waits for the listener to disappear after launchctl stop/bootout, terminates verified lingering gateway processes with graceful-then-force cleanup, and returns non-zero if the port remains occupied while preserving existing LaunchAgent disable/start/restart recovery semantics.

## Reproduction Assessment

Yes. The issue gives concrete macOS steps, and current source confirms a matching source-level path: degraded bootout can return without any gateway PID or port verification; a final fix still needs macOS launchd validation.

## Solution Assessment

No. Current main is not the best solution yet; the narrow maintainable fix is to add the same verified process cleanup and port-release gate already present in the Windows stop path to the macOS LaunchAgent stop flow.
