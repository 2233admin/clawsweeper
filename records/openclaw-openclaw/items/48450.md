## Summary

Current main still has the Discord `thread-create` partial-success bug. The plugin creates the thread, then awaits a separate initial-message POST for non-forum threads; if that second call rejects, the created thread id is lost and the tool adapter returns a generic error, leaving agents likely to retry a non-idempotent create.

## Best Possible Solution

Keep this issue open and fix the Discord plugin `thread-create` path. Preserve the created thread id when the non-forum follow-up message post fails, return a structured partial/error result that lets the agent continue in that known thread, and add regression coverage for first POST success plus second POST rejection.

## Clownfish Work Prompt

Fix the Discord `message` tool `action=thread-create` partial-success bug reported in openclaw/openclaw#48450. Current behavior: for non-forum threads, `createThreadDiscord` first creates the thread and then sends the initial message to `Routes.channelMessages(thread.id)`. If the second POST fails, the helper throws, the runtime never returns the created thread id, and the PI tool adapter reports a generic `{ status: "error" }`, causing agents to retry and create duplicate ghost threads.

Expected boundary: keep the fix in the Discord plugin. Preserve the created thread id when the follow-up initial-message send fails and return a structured result that lets the agent continue in the known thread instead of retrying creation. Prefer a small typed result/partial-success shape over core retry changes. Do not add bundled-owner special cases to core, do not broaden Discord permissions, and do not implement a broad retry/idempotency framework for this PR.

Likely implementation paths: `extensions/discord/src/send.messages.ts`, `extensions/discord/src/actions/runtime.messaging.ts`, `extensions/discord/src/send.types.ts` if a typed result changes, and regression tests in `extensions/discord/src/send.creates-thread.test.ts` plus `extensions/discord/src/actions/runtime.test.ts` if the runtime result shape changes. `extensions/discord/src/actions/handle-action.ts` may not need code changes but should be checked.

Regression coverage: add a test where the first `rest.post(Routes.threads(...))` resolves with `{ id: "t1" }` and the second `rest.post(Routes.channelMessages("t1"), ...)` rejects; assert the created thread id is preserved in the returned tool/runtime result and the failure detail is structured enough for the agent to avoid recreating the thread. Keep the existing successful non-forum and message-attached thread tests green.

Changelog: this is user-facing Discord behavior, so add a single-line active-version fix entry if the project expects changelog entries for bundled plugin bug fixes, with `Thanks @dahifi` attribution and no wrapping.
