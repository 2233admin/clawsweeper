## Summary

Keep this PR open. Current main does not implement the opt-in per-agent LLM request header, and the closed related feature reports do not appear to correspond to a shipped implementation. The PR is still a plausible implementation candidate, but it needs maintainer review and at least the normalized-provider lookup issue from the existing review comment addressed before landing.

## Best Possible Solution

Keep the PR open and ask for a focused update: resolve provider config through the existing normalized-provider helper pattern, keep the option explicitly opt-in, add regression coverage for alias/case-normalized provider IDs plus enabled/disabled/missing-agent cases, and document the privacy/trusted-proxy implication. If maintainers prefer this outside core, first extend the provider/plugin hook context to include `agentId` so a plugin can implement dynamic per-agent request headers cleanly.

## Risks / Open Questions

- The submitted implementation should not land as-is without fixing provider config lookup through normalized provider IDs or aliases; otherwise the new option will be unreliable for providers whose runtime ID differs from the config key.
- Even though the header is opt-in, `X-OpenClaw-Agent` exposes local agent identifiers to the configured provider/proxy. The help/docs should frame it as a trusted-proxy/local-operator feature and avoid implying it is safe for every upstream.
- Because the related feature reports #50117 and #58413 are closed while current main still lacks this behavior, closing this PR now would likely drop the only in-repo implementation candidate instead of preserving a clear review path.
