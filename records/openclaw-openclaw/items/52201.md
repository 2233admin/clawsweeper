## Summary

Keep open. Current main still has agent-scoped workspace selection for normal route bindings: `bindings[].workspace` is not typed, the strict route-binding schema would reject it, resolved routes carry no workspace payload, and reply/runtime setup still chooses the workspace from the matched agent. The related implementation PRs are closed unmerged, and the adjacent ACP `cwd` plus spawned-run workspace overrides do not satisfy normal per-conversation route binding isolation.

## What This Changes

Request to add an optional normal route-binding `workspace` override so one agent can keep separate workspace, memory, and context per matched conversation.

## Best Possible Solution

Keep this as an explicit maintainer product review item. If accepted, implement a typed and documented `bindings[].workspace` field for normal route bindings, validate/trim it, thread it through resolved routes and inbound contexts, apply it consistently to runtime workspace consumers, persist it only where interaction continuity requires it, and test precedence plus stale-override clearing.

## Reproduction Assessment

Not applicable as a runtime bug reproduction; this is a feature request. The absence is high-confidence from current-main source inspection: the schema rejects the proposed key and the route/reply paths have nowhere to carry or apply it.

## Solution Assessment

Unclear until maintainer approval. The proposed `bindings[].workspace -> agent.workspace -> default` shape directly matches the requested behavior, but the safest implementation needs explicit decisions on persistence, reroute clearing, and which workspace-scoped subsystems must honor the override.
