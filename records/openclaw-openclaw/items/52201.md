## Summary

Keep this issue open. Current main still has agent-scoped workspace resolution, strict route bindings without a `workspace` field, and route results that carry only agent/session routing data. The related implementation PRs in the provided context are closed unmerged, and current docs still describe separate agents/bindings, or ACP-specific `cwd`, as the available isolation mechanisms.

## Best Possible Solution

Keep this issue open for maintainer product review and a fresh focused PR. If accepted, add a typed and documented `bindings[].workspace` field for route bindings, trim/validate it in schema and generated docs, carry the matched override through route resolution and inbound contexts, apply it consistently to bootstrap/context/memory/tool/plugin workspace selection, persist it only for outbound interactions that need continuity, clear it on reroute/rebind paths, and cover precedence plus stale-override behavior with focused route/channel tests.

## Risks / Open Questions

- Binding-level workspace selection is a security/product boundary change: stale overrides after reroute/rebind could put tools, memory, or bootstrap reads in the wrong directory.
- ACP `bindings[].acp.cwd` and spawned-run workspace inheritance are adjacent but not equivalent; a fix should avoid treating them as satisfying normal route-binding workspace isolation.
- A complete implementation likely crosses config schema/docs, route resolution, inbound channel contexts, reply runtime, plugin loading, memory/bootstrap, and outbound interaction persistence, so narrow tests should cover precedence and rebind clearing.
