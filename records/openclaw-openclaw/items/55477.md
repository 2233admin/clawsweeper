## Summary

Keep this PR open but do not merge it as-is. The requested LiteLLM observability is useful, and the prior Greptile review correctly noted smaller coverage/guard issues, but the proposed wrapper writes only to `options.metadata`; current LiteLLM LLM calls use OpenAI-completions transport, whose request builder does not serialize `options.metadata` into the outbound payload. The wrapper also applies to every provider stream function, which is broader than the LiteLLM-scoped problem and exposes stable channel/session identifiers without an opt-in boundary.

## What This Changes

The PR adds a `runEmbeddedAttempt` stream function wrapper that stamps `session_key`, `message_channel`, and `context_limit` into `options.metadata` for outbound model calls.

## Best Possible Solution

Implement a narrow, tested LiteLLM/OpenAI-completions metadata path that serializes only approved fields when the configured provider opts in, preserves existing caller metadata, and documents the privacy semantics for session identifiers.
