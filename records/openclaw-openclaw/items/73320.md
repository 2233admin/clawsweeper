## Summary

Keep open. Current main still embeds task-completion results without a size cap, so the PR is addressing a real gap, but the patch is not ready because it caps only the rendered prompt text while the raw `internalEvents` payload can still be sent through the gateway request and fallback path, and it also weakens a provider catalog helper that current main no longer needs weakened.

## What This Changes

The PR adds a 12,000-character head/tail cap for task-completion internal-event prompt rendering, adds formatter coverage, and adjusts provider catalog test expectations/helpers for Codex mini.

## Best Possible Solution

Land a focused version that bounds task-completion result text before it enters the delivery/gateway `internalEvents` payload, keeps the prompt-formatting cap as defense in depth, preserves head/tail context with an explicit truncation marker, and leaves provider catalog tests exact against current main's expected fixture.

## Reproduction Assessment

Yes. On current main, a large `task_completion.result` flows through both runtime-context and plain prompt formatters unchanged; the remaining PR gap is visible on the delivery path where the raw `internalEvents` object is still passed into the JSON-serialized gateway request.

## Solution Assessment

No, not as currently patched. The formatter boundary is useful but too late for the internal event payload itself; cap the value before constructing or delivering the event, then keep formatter tests to prove both prompt renderers stay bounded.
