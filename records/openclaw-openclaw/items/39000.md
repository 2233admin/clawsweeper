## Summary

Keep this issue open. Current main still supports plugin registration of gateway methods but does not expose a plugin-side `api.callGatewayMethod()` or `api.runtime.gateway.dispatchMethod()` caller, and the related implementation candidate, PR #46485, is still open/unmerged in the supplied GitHub context.

## What This Changes

Request to add a plugin-side gateway-method caller so one plugin can invoke gateway methods registered by another plugin.

## Best Possible Solution

Keep the issue open and resolve it through Plugin SDK/runtime design review. The best path is to review, revise, or replace PR #46485 with a documented and tested request/response seam, likely on `api.runtime.gateway.dispatchMethod` or a narrower typed service/capability API, with explicit caller identity and scope behavior rather than exposing the whole gateway RPC bus casually.

## Risks / Open Questions

- A generic plugin-side gateway dispatcher could broaden what plugins can invoke through gateway RPC, so the API needs explicit caller identity, scope, and authorization semantics before being exposed as a stable SDK surface.
- Current main has a private dispatcher that already handles synthetic clients and request scope for subagent/node helpers; exposing it directly without tests could bypass the narrower helper contracts plugin authors use today.
