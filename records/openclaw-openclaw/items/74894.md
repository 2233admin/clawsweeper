## Summary

Keep this PR open, but do not merge it as written. Current main still has the reported Telegram group `/status` no-reply path for explicitly configured groups with no effective sender allowlist, but the PR broadens command authorization for every empty effective group allowlist, including per-group `groupPolicy: "open"` cases that current tests and docs keep unauthorized for random senders.

## What This Changes

The PR changes Telegram inbound body command gating so text control commands skip the access-group gate when the effective Telegram command allowlist has no entries.

## Best Possible Solution

Land a narrower Telegram fix that lets text control commands through for explicitly configured group sessions that already passed group-policy authorization, while preserving sender allowlists, `commands.allowFrom`, owner-only command checks, and the current denial of unauthorized commands in open groups. Add focused regression tests for explicit groups, wildcard groups, open groups, and configured sender allowlists.

## Reproduction Assessment

Yes. The current-main static path is high-confidence: an explicit group entry can satisfy group policy with no effective sender entries, then `resolveControlCommandGate` has `useAccessGroups: true` and no configured authorizer, causing `/status` to return null at the early Telegram body gate. I did not run tests because this review is read-only.

## Solution Assessment

No. The proposed switch is too broad because empty effective allowlists also occur in per-group open configurations where current docs and tests keep text commands unauthorized. A safer fix should pass an explicit group-policy authorization fact into the gate, or add a dedicated authorizer for explicit configured group entries without weakening open-group sender command auth.
