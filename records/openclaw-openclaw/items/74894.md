## Summary

Keep open, but do not merge as written. The PR targets a credible Telegram regression, but the patch widens text-command authorization for empty group allowlists beyond the reported explicit-group case and conflicts with current docs and tests that keep inline shortcuts restricted to authorized senders.

## What This Changes

The PR changes Telegram inbound body command gating so access-group command enforcement is skipped when the effective command allowlist has no entries.

## Best Possible Solution

Keep the targeted Telegram fix, but narrow it to the explicit configured-group case that already passed group-policy authorization. Preserve sender allowlist enforcement, `commands.allowFrom`, owner-only command checks, and the current per-group-open unauthorized-sender guard, then add regression coverage for explicit groups, wildcard groups, open groups, and configured sender allowlists.

## Reproduction Assessment

Yes. The current-main static path reproduces the reported drop for an explicitly configured group with no effective sender allowlist: group policy can allow the group, then `resolveControlCommandGate` has no configured authorizer and `resolveTelegramInboundBody` returns null for `/status`. The PR’s regression risk is also covered by the existing per-group-open `/status` test.

## Solution Assessment

No. The patch is narrower than a broad refactor, but using `allowForCommands.hasEntries` as the switch is too broad because empty allowlists also occur in open-group configurations where current docs and tests keep text commands unauthorized. A safer fix should pass an explicit group-policy authorization fact into the command gate or add a separate authorizer for dedicated `groups` entries only.
