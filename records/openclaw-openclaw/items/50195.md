## Summary

Keep this issue open. Current main still sends WhatsApp text as a plain Baileys `{ text }` payload, exposes no `linkPreview` field through the WhatsApp outbound/send API, has no `channels.whatsapp.generateHighQualityLinkPreview` config key, and the Gateway `send` RPC remains a single-message method rather than a batch-send API. The related discussion is useful: #43397 was closed unmerged, and #6722 does not cover the pre-built-preview or batch-send request that remains here.

## Best Possible Solution

Keep this issue open, but split implementation into focused reviewable work: first add typed WhatsApp pre-built `linkPreview` pass-through through the outbound payload/Gateway send surfaces with tests and docs; separately decide whether automatic `generateHighQualityLinkPreview` should be opt-in and how to document SSRF risk; then design a generic Gateway batch-send contract with per-message idempotency and partial-result semantics before implementing it.

## Risks / Open Questions

- Enabling automatic high-quality link previews can introduce SSRF/network-fetch behavior, so any config option needs explicit opt-in semantics and clear docs.
- Pre-built link preview payloads should be typed and treated as untrusted display metadata; avoid letting arbitrary preview text/images enter agent context or bypass existing outbound sanitization expectations.
- A batch send RPC needs product/API decisions for per-item idempotency, partial failures, ordering, rate limiting/backoff, transcript mirroring, and result shape before it is safe to implement broadly.
