## Summary

Keep open. Current main still lacks the requested WhatsApp pre-built `linkPreview` pass-through, lacks a WhatsApp socket/config opt-in for automatic high-quality previews, and exposes only the single-message Gateway `send` RPC. The related prior work is useful context, but it does not resolve current behavior: #18414 was merged but the field is absent on current main, #43397 is closed unmerged, and #6722 was closed as superseded by this broader tracker.

## What This Changes

The issue requests WhatsApp support for caller-provided Baileys link previews, an opt-in `generateHighQualityLinkPreview` config, and a batch Gateway send RPC for multi-message workflows.

## Best Possible Solution

Split the request into staged work: first add a typed WhatsApp-owned pre-built `linkPreview` pass-through through the outbound/Gateway send path with tests and docs, then separately design automatic preview fetching and batch Gateway sending with explicit security and protocol behavior.

## Risks / Open Questions

- Automatic high-quality preview fetching can cause the Gateway to fetch arbitrary outbound-message URLs, so the opt-in needs explicit SSRF, proxy, timeout, and trust semantics before implementation.
- `send.batch` is a Gateway protocol expansion with idempotency, ordering, partial-failure, transcript mirroring, and rate-limit semantics; it should not be folded into the narrow WhatsApp pass-through without maintainer design.
