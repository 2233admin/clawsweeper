## Summary

Current main still does not fully implement the requested behavior. Provider/run-level cron failures are sanitized now, but fatal isolated cron outcomes are still converted into `isError: true` delivery payloads and passed through normal announce delivery instead of being suppressed and left to run history plus explicit failure-notification paths. The related implementation PR #46236 was closed unmerged, so there is no shipped fix on main.

## Best Possible Solution

Keep this issue open. Add a guard at the isolated cron finalize or dispatch boundary so `hasFatalErrorPayload`/error-status runs skip normal `delivery.mode: "announce"` delivery, while still recording `lastRunStatus: "error"`, preserving backoff/error counters/run history, and allowing only explicit failure notification paths to send sanitized failure messages. Add regression coverage for provider/run-level errors with announce delivery and `bestEffort: false` asserting no outbound announce send occurs.

## Risks / Open Questions

- The exact raw JSON leak may be partly mitigated by current sanitization into `cron isolated run failed: ...`, but the issue's core requirement remains unmet because normal announce delivery can still post failed-run output.
- A fix needs to distinguish normal announce delivery from explicit `failureAlert`/`failureDestination` notifications so operators still receive configured sanitized failure alerts.
