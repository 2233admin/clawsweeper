## Summary

Keep open. Current main still converts fatal isolated-cron/provider failures into `isError: true` delivery payloads and then routes those payloads through normal announce delivery; the related fix PR #46236 is closed and unmerged.

## Best Possible Solution

Keep this issue open and fix the cron-owned failure path: fatal/error-status isolated cron runs should record `lastRunStatus: "error"` and trigger configured failure notifications, but should not perform normal `delivery.mode: "announce"` fallback delivery regardless of `delivery.bestEffort`. Prefer a fatal-run/status guard over a raw-text regex classifier.

## Clownfish Work Prompt

Fix the cron announce failure leak tracked by openclaw/openclaw#42243. Current main synthesizes fatal isolated-cron/provider failures into `deliveryPayloads` with `isError: true` and then sends them through normal `delivery.mode: "announce"` delivery. Add a guard so fatal/error-status isolated cron runs do not perform normal announce fallback delivery, regardless of `delivery.bestEffort`, while still recording the run as `status: "error"`/`lastRunStatus: "error"` and preserving configured failure notifications through the separate alert path. Prefer passing or deriving a fatal-run signal (`hasFatalErrorPayload` or equivalent) over a raw-text regex classifier; do not implement only `isLikelyRawErrorOutput`. Preserve successful announce delivery, message-tool duplicate suppression, NO_REPLY/silent-token behavior, subagent final-output handling, and direct structured/media delivery for non-fatal successful runs. Add focused regression coverage for provider/run-level failure with announce delivery and `bestEffort: false` asserting no outbound announce send, plus coverage that configured failure alerts still send. Changelog expected because this is user-facing; credit `Thanks @sercada`. Do not change provider retry policy, channel adapters, or `bestEffort` defaults unless tests prove it is required.
