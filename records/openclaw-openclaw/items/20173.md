## Summary

Keep open. Current main still only wires Discord inbound text through MessageCreateListener, does not register or dispatch Carbon MessageUpdateListener, has no edited-message config/docs surface, and the current replay key would treat a same-message edit as the same account/channel/message delivery unless the implementation becomes edit-aware.

Keep this issue open as the canonical Discord edit-to-reprocess tracker. Implement the feature in the bundled Discord plugin with an opt-in config/type/schema/docs surface, Carbon MessageUpdateListener registration, content-change filtering, edit-aware replay semantics, and focused Discord monitor tests. Carry forward #48894's added-mention acceptance criteria, treat #38505 as overlapping, and leave delete-to-cancel from #53654 out unless maintainers explicitly include it.

Implement Discord edited-message reprocessing for the bundled Discord plugin.

Observable bug/feature: Discord user edits currently do not trigger a new OpenClaw turn. Current main only registers `MessageCreateListener`; Carbon 0.16.0 exposes `MessageUpdateListener`; replay dedupe is keyed by `accountId:channelId:messageId` and needs edit-aware semantics.

Expected boundary:
