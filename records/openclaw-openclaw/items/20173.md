## Summary

Keep open. Current main still wires Discord inbound text through Carbon MessageCreateListener only; it does not register MessageUpdateListener, expose handleEdits/editDebounceMs config or docs, or make inbound replay semantics edit-aware. The existing maintainer-side comment remains accurate against current main 20e21173715ef488850023a90f72585a01cc379f.

## What This Changes

Enable the bundled Discord plugin to detect edited user messages via MESSAGE_UPDATE and optionally reprocess them with debounce and backward-compatible config.

## Best Possible Solution

Keep this issue open as the canonical Discord edit-to-reprocess tracker. Implement the feature in the bundled Discord plugin with an opt-in Discord config/type/schema/docs surface, Carbon MessageUpdateListener registration, content-change filtering, per-message debounce, edit-aware replay semantics, and focused Discord monitor tests. Carry forward #48894's add-mention-on-edit acceptance criteria, treat #38505 as overlapping, and keep #53654's delete-to-cancel behavior separate unless maintainers explicitly choose to include it.

Add an opt-in Discord config surface, likely channels.discord.handleEdits?: boolean defaulting false and channels.discord.editDebounceMs?: number with a conservative default when enabled, plus generated config metadata/docs if required by the repo. - The current replay guard intentionally suppresses duplicate same-id deliveries, so edit support must add explicit edit-aware semantics without reintroducing duplicate create-message processing.
- Mention-gated guild behavior needs focused tests so edits that add a bot mention are accepted while ordinary unmentioned edits remain ignored when requireMention is true.
