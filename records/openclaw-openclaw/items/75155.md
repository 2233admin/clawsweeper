## Summary

Keep open. Current main still has the native macOS WebChat gap: Talk Mode sends the finalized speech prompt through `chat.send`, the gateway emits transcript updates, and browser Control UI handles `session.message`, but the native SwiftUI chat transport/view model still has no `session.message` event path or per-session message subscription to reveal externally submitted user turns promptly.

## What This Changes

The issue asks for macOS Talk Mode speech transcripts to render as user messages in the WebChat/Control UI thread, matching typed chat input.

## Best Possible Solution

Add native macOS/shared chat handling for external transcript updates: subscribe the active native WebChat session to `session.message`, map that gateway event through the Swift transport, refresh or merge active-session history without disrupting active runs, and make Talk Mode display the spoken transcript rather than the model-facing Talk wrapper. Keep browser Control UI behavior intact and add focused Swift plus gateway/UI regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: open native macOS WebChat on the active/main session, submit a Talk Mode transcript, and the Talk path uses direct `chat.send` while the native transport drops the `session.message` update that would reveal the external user turn. I did not perform a live macOS app run in this read-only review.

## Solution Assessment

Yes. The narrow maintainable fix is native event/subscription handling plus transcript display cleanup; adding an opt-in `talk.echoTranscript` setting would mask a consistency bug instead of making voice input behave like typed input.
