## Summary

Keep open. Current main still has an actionable native macOS WebChat gap: Talk Mode sends through `chat.send` outside the typed-input path, while the native SwiftUI transport neither subscribes to `session.message` transcript updates nor maps that event into a chat history refresh. The browser Control UI already has the needed event handling, so this is a narrow native follow-up rather than a core scope question.

## What This Changes

Render macOS Talk Mode spoken transcripts as visible user messages in the WebChat/Control UI thread, matching typed input behavior.

## Best Possible Solution

Add native SwiftUI WebChat handling for external transcript updates: subscribe to the active session's transcript message stream, map `session.message` into the shared chat transport/view model, refresh or merge active-session history when Talk Mode submits a turn, and make the visible user card show the spoken transcript rather than Talk's model-facing wrapper.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to open native macOS WebChat on the main session and submit a Talk Mode transcript: Talk uses direct `chat.send`, while native WebChat has no `session.message` subscription or event mapping to reveal the user turn. I did not run the macOS app live in this read-only review.

## Solution Assessment

Yes, the best fix is native event/subscription handling plus transcript display sanitation, not an opt-in `talk.echoTranscript` setting. Browser Control UI behavior should stay as-is except for regression coverage if shared contracts move.
