## Summary

Keep open. Current main has gateway/browser-side support for user transcript update broadcasts, but the macOS SwiftUI WebChat path does not subscribe to or map those `session.message` events, while Talk Mode submits through `GatewayConnection.chatSend` outside the chat view model's normal optimistic typed-input path. The related Talk UI work covers the visible Talk surface, not this narrower transcript-rendering gap.

## What This Changes

Talk Mode voice transcripts should render as user messages in the WebChat/Control UI chat history alongside assistant replies.

## Best Possible Solution

Add a native macOS SwiftUI WebChat path for external session transcript updates: subscribe to the relevant gateway session-message stream, map `session.message` into the shared chat transport, and update or refresh the active thread when Talk Mode submits a user turn. Keep the existing gateway/browser behavior and consider showing only the spoken transcript, not Talk's model-facing prompt wrapper, if history currently exposes that wrapper.

## Reproduction Assessment

Yes, there is a high-confidence source-level reproduction path: open native macOS WebChat on the main session, submit a Talk Mode transcript, and the user turn is emitted as `session.message` by the gateway but is not subscribed to or mapped by the native transport. I did not perform a live macOS runtime reproduction in this read-only review.

## Solution Assessment

Yes, the requested direction is the right product behavior, but the narrow maintainable fix should be event/subscription handling in the native WebChat/Talk path rather than a new opt-in `talk.echoTranscript` setting. Browser Control UI behavior should be preserved and covered with regression tests.
