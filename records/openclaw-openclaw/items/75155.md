## Summary

Keep open. Current main still has the native macOS WebChat gap: Talk Mode sends the transcript through `chat.send`, the gateway emits `session.message` updates, and browser Control UI reacts to them, but the native SwiftUI WebChat transport/view model still has no `session.message` path to refresh or merge the externally submitted user turn.

## What This Changes

The issue requests that macOS Talk Mode spoken user transcripts render in the WebChat/Control UI chat history like typed user messages.

## Best Possible Solution

Add native macOS SwiftUI WebChat handling for external transcript updates: map active-session `session.message` events through the chat transport, refresh or merge history without disrupting active runs, and make Talk Mode's visible user row show the spoken transcript rather than the model-facing Talk wrapper. Keep the browser Control UI behavior intact and add focused native/shared chat regression coverage.

## Reproduction Assessment

Yes. A source-level reproduction is clear: open native macOS WebChat on the active/main session, submit a Talk Mode transcript, and the Talk path uses direct `chat.send` while the native transport drops the gateway `session.message` update that would reveal the user turn. I did not run a live macOS app in this read-only review.

## Solution Assessment

Yes. The narrow maintainable solution is native event/subscription handling plus Talk transcript display sanitation; adding an opt-in `talk.echoTranscript` setting would hide a consistency bug instead of making voice input behave like typed input.
