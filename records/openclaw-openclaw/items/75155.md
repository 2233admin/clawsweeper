## Summary

Keep open. Current main still has a native macOS WebChat transcript-update gap: Talk Mode sends the user turn through `chat.send`, the gateway emits `session.message`, and the browser Control UI reloads history for that event, but the SwiftUI WebChat transport/view model does not represent or handle that event path.

## What This Changes

Talk Mode voice transcripts should render as visible user messages in the WebChat / Control UI chat history, like typed input does.

## Best Possible Solution

Add native macOS WebChat support for external session transcript updates: subscribe to the relevant session-message stream, map `session.message` into the shared chat transport, refresh or reconcile the active thread, and ensure the visible user message is the spoken transcript rather than the Talk prompt wrapper.

## Reproduction Assessment

Yes. The source-level reproduction path is clear: submit a Talk Mode transcript on macOS, which calls native `GatewayConnection.chatSend`; the gateway emits `session.message`, but native SwiftUI WebChat currently drops that event and does not refresh the active thread. I did not run a live macOS Talk session in this read-only review.

## Solution Assessment

Yes, the requested behavior is the right fix direction, but the narrow maintainable solution should be native WebChat event/subscription handling plus display projection for the spoken transcript, not a new opt-in `talk.echoTranscript` setting.
