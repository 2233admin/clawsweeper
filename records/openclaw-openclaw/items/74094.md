## Summary

Keep open. Current main has task status snapshots, progress/terminal summaries, CLI tables, /tasks, /status, session_status, plugin task views, and notification formatting, but it still does not have the requested shared compressed operational-summary contract with current state, last successful checkpoint, current blocker, and recommended next action. The request is concrete and appears suitable for one focused implementation PR.

## Best Possible Solution

Implement a shared deterministic task operational-summary builder in the task subsystem, then reuse it across CLI list/show, /tasks, /status/session_status task lines, task-domain views, and task notification rendering. Keep raw task fields intact, add tests for active, blocked, failed, timed-out, cancelled, lost, and succeeded tasks, and update user-facing docs/changelog because this changes operator-visible output.

## Clownfish Work Prompt

Implement the open issue request to add compressed operational summaries for background tasks. Add a shared deterministic task operational-summary builder in the task subsystem, likely in src/tasks/task-status.ts or a new src/tasks/task-operational-summary.ts. The builder should return a structured additive summary with at least current state, last good checkpoint when derivable, current blocker when present, recommended next action when known, and a concise rendered text form like `blocked · awaiting approval · last good step: patch applied · next: approve exec to push branch`.

Use existing TaskRecord fields first: status, runtime, taskKind/sourceId, deliveryStatus, notifyPolicy, progressSummary, terminalSummary, terminalOutcome, error, timestamps, and TaskEventRecord history where available. Do not remove or rename existing task JSON fields. Keep the summary deterministic, sanitized through existing task status sanitizers, and conservative about `next` guidance. Avoid leaking internal runtime context or raw approval IDs/commands.

Have the relevant surfaces consume the shared builder instead of hand-rolling separate phrasing where feasible: CLI list/show in src/commands/tasks.ts, /tasks in src/auto-reply/reply/commands-tasks.ts, /status in src/status/status-text.ts, session_status in src/agents/tools/session-status-tool.ts, task notifications in src/tasks/task-executor-policy.ts, and task-domain/plugin views in src/tasks/task-domain-views.ts / src/plugins/runtime/task-domain-types.ts if an additive JSON field is useful. Update docs/automation/tasks.md and docs/cli/tasks.md. Add a user-facing changelog entry under the active version with credit to @neofdezf-cloud.

Keep the PR narrow: do not change task lifecycle semantics, task status names, retention, reconciliation, delivery policy, or plugin ownership boundaries. Add colocated tests that prove the same builder output is reused for active, queued, blocked, failed, timed-out, cancelled, lost, and succeeded tasks, including sanitization and truncation.
