## Summary

Keep this issue open. Current main still models the relevant phone-number and phone-identifier channel fields as plain string/string-array config, and the SecretRef credential surface is still limited to declared credential paths such as voice-call Twilio authToken, not phone numbers. The issue also has recent maintainer validation and active contributor follow-up, so it is live work rather than cleanup-close material.

## Best Possible Solution

Keep this issue open. The best implementation is a backward-compatible, channel/plugin-owned change that keeps plain strings valid while adding SecretInput/SecretRef support for phone-number and peer identifier fields such as WhatsApp allowFrom/defaultTo/groupAllowFrom, Signal account/defaultTo/allowFrom/groupAllowFrom, and voice-call fromNumber/toNumber/allowFrom. It should update schemas, exported types, generated metadata/docs, secret target discovery, runtime resolution before routing/outbound/auth checks, and redaction/status output. Any plaintext-to-SecretRef conversion should be an explicit doctor or secrets workflow, not an implicit startup rewrite.
