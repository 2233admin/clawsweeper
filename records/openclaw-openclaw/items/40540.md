## Summary

Keep open. Current main still has the lock-sensitive live package-root rename in the package-manager updater, and the Gateway path still schedules restart only after the update runner succeeds, so the reported Windows EBUSY self-update failure is not resolved. #44614 remains the open implementation candidate for the detached Windows updater design.

## What This Changes

The issue asks for `openclaw update` to avoid Windows npm-global EBUSY rename failures when updating the active OpenClaw package on Windows.

## Best Possible Solution

Keep this as the canonical Windows npm-global self-update EBUSY report until #44614 or an equivalent design lands. The desired end state is a Windows-safe update path that releases the active CLI/Gateway process from the live package tree before npm replaces it, with regression coverage for EBUSY/EPERM activation and a real Windows update smoke path.
