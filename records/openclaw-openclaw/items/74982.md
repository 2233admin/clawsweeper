## Summary

Keep open. Current main still carries the observable Telegram gap: the shared outbound layer propagates `audioAsVoice`, and the low-level Telegram sender only switches to `sendVoice` when it receives `asVoice`, but both current Telegram outbound adapter paths drop that mapping. The PR is a focused owner-local fix for the missing option propagation, with no blocking review finding from this read-only pass.

## What This Changes

The PR wires `audioAsVoice` through Telegram legacy outbound `sendMedia`/`sendPayload` and the registered plugin `sendPayload` handler as `asVoice`, and adds targeted outbound-adapter regression tests.

## Best Possible Solution

Review and land this focused Telegram plugin fix, ideally with CI proof for the listed Telegram tests and, if maintainers want tighter coverage, one small registered-plugin payload test for `channel.ts`. The fix should stay in the Telegram plugin adapter rather than adding core special cases.

## Reproduction Assessment

Yes. On current main, `deliverOutboundPayloads` forwards `audioAsVoice`, while Telegram adapter code drops it before `sendMessageTelegram`; the low-level sender only uses `sendVoice` when `asVoice` is true. The related #42060 TTS + `[[audio_as_voice]]` flow is a concrete end-to-end reproduction path, though I did not run live Telegram in this read-only review.

## Solution Assessment

Yes. Mapping the existing channel context flag to Telegram's existing `asVoice` option in the owner plugin is the narrowest maintainable fix. A registered-handler regression test would improve confidence, but the implementation direction is correct.
