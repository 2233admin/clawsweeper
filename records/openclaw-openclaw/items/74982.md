## Summary

Keep open. Current main still drops Telegram `audioAsVoice` before it reaches the low-level sender in both the legacy outbound adapter and the registered plugin `sendPayload` path, while this PR wires the existing outbound hint to Telegram's existing `asVoice` option and appears functionally correct.

## What This Changes

The PR maps Telegram outbound `audioAsVoice` to `asVoice` in `extensions/telegram/src/outbound-adapter.ts` and `extensions/telegram/src/channel.ts`, and adds legacy outbound adapter regression tests.

## Best Possible Solution

Review and land this Telegram plugin fix after targeted validation so the existing outbound voice hint reaches Telegram's existing `asVoice` option inside the plugin boundary, without adding core Telegram special cases.

## Reproduction Assessment

Yes. A voice-marked payload reaches shared outbound delivery with `audioAsVoice`, but current main drops that hint in both Telegram outbound adapters before `send.ts`, where voice delivery only happens when `opts.asVoice === true`.

## Solution Assessment

Yes. The PR takes the narrow maintainable path: it maps an existing channel outbound flag to an existing Telegram sender option in the owning Telegram plugin paths; adding registered-path regression coverage would improve proof but the implementation direction is sound.
