## Summary

Keep open for normal maintainer review. Current main still drops Telegram `audioAsVoice` before the low-level sender, while this PR targets the missing Telegram-owned adapter paths and appears to map the existing flag to the existing `asVoice` transport option without introducing a blocking bug.

## What This Changes

The branch threads `audioAsVoice` through Telegram legacy `sendMedia`/`sendPayload` and registered-plugin `sendPayload` as `asVoice`, with outbound-adapter regression tests.

## Best Possible Solution

Review and land this Telegram plugin fix after targeted validation. The behavior belongs in the Telegram plugin adapters by mapping the existing `audioAsVoice` context flag to the existing Telegram `asVoice` option, with no core special case needed.

## Reproduction Assessment

Yes. The current-main path is reproducible by a TTS/media payload with `[[audio_as_voice]]`: shared delivery forwards `audioAsVoice`, Telegram adapters drop it, and `send.ts` only calls `sendVoice` when `asVoice` is true.

## Solution Assessment

Yes. The PR is the narrowest maintainable fix for the observed behavior because it preserves an existing flag through owner-local Telegram send options; a small registered-handler test would improve proof but is not required to understand the fix direction.
