## Summary

Keep this PR open. Current main still starts the optional model-pricing refresh during initial Gateway runtime setup, before the post-attach and sidecar-ready path, and the PR’s code change is a narrow fit for that lifecycle bug. I did not find a functional or security regression in the diff, but the user-facing gateway fix is missing the required `CHANGELOG.md` entry before it is merge-ready.

## What This Changes

The PR moves Gateway model-pricing refresh from initial runtime startup to scheduled-service activation after post-attach and sidecar readiness, with updated lifecycle tests and shutdown handling.

## Best Possible Solution

Add the missing Unreleased changelog fix entry, then land or adapt this PR after maintainer review and required checks. Keep the scope focused on deferring optional pricing refresh until Gateway sidecars/channels are ready; leave timeout reduction, disk cache, and circuit-breaker behavior to the related resilience work in #75067 and the broader tracker #73323.

## Reproduction Assessment

Yes. The narrow startup-order issue is reproducible from current-main source: pricing refresh is started by initial runtime services before post-attach and sidecar readiness, while deferred sidecars can complete after `startGatewayPostAttachRuntime()` returns. The broader Windows/macOS timeout degradation is documented in #73323 and #74128, but this PR only needs the lifecycle-order proof.

## Solution Assessment

Yes for the narrow lifecycle bug. Moving the optional pricing refresh into the existing scheduled-services activation path and gating it on sidecar readiness is smaller and more maintainable than disabling pricing or changing fetch timeouts in this PR; broader network resilience belongs in a separate follow-up.
