## Summary

Keep open. Current main still has the reported media capability-provider path: image/video/music tool setup eagerly lists providers, `resolvePluginCapabilityProviders()` builds a capability-specific compat config, and then calls `resolveRuntimePluginRegistry({ config, activate: false })` without `onlyPluginIds`. The existing cache only memoizes manifest-derived plugin ids, not provider results or full snapshot registry loads, so this is not covered by older provider-discovery or Feishu startup fixes.

## Best Possible Solution

Keep this issue open and fix the media capability provider path directly. The likely best implementation is a stable per-config/env/workspace capability provider result cache, or a narrower snapshot load that passes the manifest-derived `onlyPluginIds` into `resolveRuntimePluginRegistry`, plus regression coverage that constructs image, video, and music tools or calls the three list-provider helpers in sequence and proves they no longer trigger three redundant full unscoped registry loads. Coordinate with #73728 and #73730, but do not treat either as a duplicate of this specific resolver bug.
