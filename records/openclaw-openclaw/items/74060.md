## Summary

Keep this PR open. Current main still has the Feishu setup-wizard status probe reading only top-level credentials, while the PR targets that remaining account-resolution gap and I did not find a blocking correctness or security issue in the diff.

## What This Changes

This PR updates Feishu setup-wizard status probing to resolve the selected/default Feishu account before probing and adds a regression test for defaultAccount credentials under accounts.*.

## Best Possible Solution

Land this PR or an equivalent Feishu-plugin-owned change that uses the existing account resolver for setup/status probing, keeps focused regression coverage, and then close the linked health issue only after maintainers are satisfied the setup-status gap explains the remaining user-visible symptom.

## Reproduction Assessment

Yes for the setup-status defect: a multi-account Feishu config with credentials only under defaultAccount/accounts makes current main report configured but skip the setup probe because it inspects only top-level credentials. The exact live status --deep table from #72930 was not reproduced without real Feishu credentials.

## Solution Assessment

Yes. Reusing resolveFeishuAccount in the Feishu setup/status surface is the narrowest maintainable fix because it stays in the owning plugin, uses the existing account contract, and does not add Feishu-specific core behavior.
