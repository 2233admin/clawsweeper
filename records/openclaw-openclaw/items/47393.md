## Summary

Keep open. Current main still does not implement this PR’s intended Discord cross-account route-owner behavior or the application-id fallback for failed bot identity lookup. The PR remains a plausible bug-fix candidate, but it needs refresh against the current plugin boundary and should address the existing DM and inactive-account review concerns before merge.

## Best Possible Solution

Keep this PR open as a real Discord bug-fix candidate. The best path is to refresh it onto current main, implement Discord route-owner resolution inside `extensions/discord` using plugin SDK routing exports, compare only active/configured Discord accounts, preserve `botUserId` with an application-id fallback when `fetchUser("@me")` fails, pass `botUserName` only if name-based mention bypass remains needed, skip or redesign owner blocking for DMs, and add focused regression tests for cross-account channel ownership, explicit alternate-bot mentions, inactive account bindings, and identity-fetch failure.

## Risks / Open Questions

- Merging this PR as-is would likely violate the current bundled-plugin boundary because the provided production diff reaches into core `src/**` internals instead of using `openclaw/plugin-sdk/*` or extension-local exports.
- The proposed owner guard needs a DM-specific fix or design decision; the existing review comments describe a plausible silent-drop path for direct messages in multi-account configurations.
- The proposed owner resolver should restrict candidates to active/configured Discord accounts so stale bindings cannot make a live bot drop channel traffic for an account that is not running.
