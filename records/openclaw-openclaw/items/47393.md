## Summary

Keep this PR open. Current main still lacks the proposed cross-account Discord owner guard and still fails Discord startup when bot identity lookup fails, but the patch should not merge as-is because the new owner logic can silently drop DMs, can choose unavailable accounts as owners, violates the bundled plugin boundary, and is stale against the current routing matchedBy contract.

## What This Changes

This PR adds Discord cross-account route-owner resolution, explicit bot mention/name bypass handling, `botUserName` plumbing, an application-id bot identity fallback, and focused Discord monitor/provider tests.

## Best Possible Solution

Refresh or replace the PR on current main with a Discord-owned implementation that uses plugin SDK/local extension APIs, applies owner suppression only where another active Discord account can actually receive the message, preserves explicit mention bypass and application-id identity fallback, and adds focused regression tests for owner precedence, explicit alternate-bot mentions, unavailable accounts, identity-fetch failure, and any chosen reply-target behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Exclude direct messages from owner blocking:** `extensions/discord/src/monitor/message-handler.preflight.ts:437`
  - body: The new owner guard also runs for DMs, but `bypassOwnerBecauseMentioned` is always false for DMs. If another account has a more specific direct-peer binding, this bot drops the DM even though that other bot will not receive the DM event, leaving the user with no responder.
  - confidence: 0.91
- **[P2] Ignore unavailable accounts when resolving the owner:** `extensions/discord/src/monitor/route-owner.ts:36-39`
  - body: `listDiscordCandidateAccountIds` includes account IDs that exist only in bindings. A stale or disabled Discord account can therefore win ownership and cause active accounts to suppress channel messages for an owner that is not running.
  - confidence: 0.88
- **[P2] Use plugin SDK exports from Discord production code:** `extensions/discord/src/monitor/route-owner.ts:1-7`
  - body: The new production module imports config and routing internals via `../../../../src/**`. Bundled Discord code is expected to use `openclaw/plugin-sdk/*` or local extension barrels, so this bypasses the plugin boundary current main enforces.
  - confidence: 0.86
- **[P2] Add the wildcard route priority:** `extensions/discord/src/monitor/route-owner.ts:11-18`
  - body: Current main's `ResolvedAgentRoute["matchedBy"]` includes `binding.peer.wildcard`, but the new exhaustive `ROUTE_PRIORITY` table omits it. After refreshing this PR, the module will fail the type contract or treat wildcard owner matches with an undefined priority.
  - confidence: 0.84
