## Summary

Keep open. Current main still rejects plugin tools whose runtime object lacks a `parameters` record, and the reported external memory-plugin failure has not been implemented away. The TypeBox-specific theory remains unproven because the validator accepts any non-array object and the referenced TypeBox constructors return object schemas, so maintainers need either the actual failing runtime value or an explicit plugin API/diagnostics decision.

## What This Changes

The issue asks OpenClaw to stop dropping TypeBox-backed plugin memory tools as malformed, improve the malformed-tool diagnostic, and consider supporting `inputSchema` as an alias for `parameters`.

## Best Possible Solution

Keep this open until maintainers can capture the failing plugin’s actual runtime tool object or decide on a targeted core enhancement. The maintainable end state is either a plugin-side fix that returns the documented `parameters` object, or an OpenClaw change that improves malformed-tool diagnostics and, if desired, documents and tests any new schema alias behavior.

## Reproduction Assessment

Unclear. The issue provides concrete external steps and logs, but the failing plugin source/runtime value is not present here; current-main source inspection shows a normal TypeBox `Type.Object()` should pass the validator.

## Solution Assessment

Unclear. Better diagnostics in `describeMalformedPluginTool` looks like a narrow safe improvement, but accepting `inputSchema` or adding TypeBox-specific bypasses is not clearly the best fix until the actual malformed runtime value is known.
