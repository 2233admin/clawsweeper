## Summary

Keep open. Current main intentionally skips malformed plugin tools that do not expose a `parameters` object, and the documented plugin contract uses `parameters: Type.Object(...)`; however, the reported external memory-plugin failure is not implemented away on main. The TypeBox-specific root-cause theory is not proven because @sinclair/typebox 0.34.49 returns a normal object for `Type.Object()`, so maintainers need either the actual runtime value from the failing plugin or an explicit decision to broaden diagnostics/API compatibility.

## What This Changes

The issue requests that OpenClaw stop dropping memory plugin tools registered with TypeBox schemas, improve malformed-tool diagnostics, and possibly accept MCP-style `inputSchema` as a plugin tool schema alias.

## Best Possible Solution

Keep the issue open until maintainers can capture the actual `tool.parameters` value from the failing plugin or decide on a targeted core enhancement. The maintainable end state is either a plugin-side fix that returns the documented `parameters` object, or a deliberate OpenClaw change that improves runtime diagnostics and documents any new alias behavior with focused contract tests.

## Reproduction Assessment

Unclear. The issue provides concrete external steps, but the failing plugin source/runtime value is not present here, and source inspection shows a real @sinclair/typebox `Type.Object()` should pass the current `isRecord` validator.

## Solution Assessment

Unclear. Better malformed-tool diagnostics would be a narrow safe improvement, but accepting `inputSchema` as a native plugin alias changes the public plugin contract and needs maintainer judgment because current docs/tests intentionally require `parameters`.
