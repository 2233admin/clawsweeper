## Summary

Keep this issue open. Current main still routes GitHub Copilot Gemini catalog rows and forward-compatible Gemini refs to `openai-responses`, matching the reported unsupported `/responses` failure; the manual `openai-completions` header symptom appears partly addressed in the shared OpenAI transport, but the default Gemini route remains wrong and the docs/tests still preserve that behavior.

## What This Changes

The issue requests a GitHub Copilot provider fix so Gemini/chat-completions-only Copilot models use the completions endpoint with required Copilot IDE headers instead of falling back after Responses API failures.

## Best Possible Solution

Land a focused GitHub Copilot provider fix: route known Gemini and forward-compatible Gemini Copilot model IDs to `openai-completions`, keep or add regression proof that Copilot IDE headers are present on that completions path, update the provider docs and changelog, and leave fallback-cost warning UX for a separate issue.

## Reproduction Assessment

Yes. The code path deterministically resolves bundled and synthetic GitHub Copilot Gemini IDs to `openai-responses`, and the reporter provided live curl evidence that Copilot rejects Gemini on `/responses` while accepting it on `/chat/completions`.

## Solution Assessment

Yes. A provider-local routing fix with tests is the narrowest maintainable solution; adding a broad core fallback warning should not block fixing the broken Copilot Gemini transport.
