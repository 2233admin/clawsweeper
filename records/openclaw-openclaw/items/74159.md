## Summary

Keep open. Current main still routes GitHub Copilot Gemini IDs through `openai-responses`, both for bundled catalog rows and dynamic forward-compatible IDs, matching the reporter's unsupported Responses endpoint failure. The shared OpenAI transport now appears to inject Copilot IDE headers for `github-copilot`, so the remaining central bug is provider-owned Gemini transport selection plus regression coverage and docs/changelog cleanup.

## What This Changes

The issue asks for GitHub Copilot Gemini models to use the chat-completions transport with required Copilot IDE headers instead of failing on Responses and silently falling back to paid providers.

## Best Possible Solution

Land a focused GitHub Copilot provider fix that routes known and forward-compatible Gemini model IDs to `openai-completions`, preserves Copilot IDE headers on that transport with regression tests, updates the provider docs and changelog, and leaves the optional repeated-fallback cost warning as a separate follow-up.

## Reproduction Assessment

Yes. The source-level path is high confidence: current main resolves bundled and dynamic GitHub Copilot Gemini IDs to `openai-responses`, and the reporter supplied direct Copilot API verification that Gemini rejects that endpoint.

## Solution Assessment

Yes. The narrow maintainable repair is provider-owned Gemini transport selection plus header regression coverage; changing generic OpenAI transport policy or bundling the broader fallback-cost warning into this fix would be wider than needed.
