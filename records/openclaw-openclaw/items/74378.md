## Summary

Keep open. Current main still lacks a clear fix for the Windows short-lived CLI process-lifetime regression: `status` is routed as a fast read-only command, but the shared CLI startup path can still install process-wide proxy/Undici state before routing and the finalizer does not restore or close that env-installed dispatcher. The prior issue comment is consistent with the current source evidence, and the remaining uncertainty is the exact active handle on Windows rather than whether this is actionable.

## What This Changes

The issue reports that OpenClaw 2026.4.26 on Windows leaves short-lived npm CLI invocations such as `openclaw.mjs status` and positional `openclaw.mjs version` alive as `node.exe` processes after the command should have completed.

## Best Possible Solution

Add a focused CLI lifecycle fix so short-lived non-daemon commands clean up process-wide proxy/Undici/debug-capture resources and exit naturally on Windows, while preserving long-running gateway/agent behavior and existing proxy routing semantics.

## Repair Work Prompt

Fix the Windows npm CLI lifecycle regression reported in https://github.com/openclaw/openclaw/issues/74378. Short-lived non-daemon commands such as `openclaw status` must exit naturally after completion and must not leave `node.exe ... openclaw.mjs <command>` alive. Start by identifying which active handle remains on Windows when proxy env/debug capture/Undici setup is present. Keep the fix scoped to CLI lifecycle cleanup for short-lived commands; do not change managed proxy routing policy, plugin load policy, gateway/agent long-running behavior, or provider/proxy security semantics. Prefer closing/destroying/restoring resources installed by the CLI startup path over forcing `process.exit` after successful routed commands. Likely implementation files are `src/cli/run-main.ts` and `src/infra/net/undici-global-dispatcher.ts`, with regression coverage in `src/cli/run-main.exit.test.ts`, `src/infra/net/undici-global-dispatcher.test.ts`, and possibly `src/cli/program/routes.test.ts`. Add a user-facing changelog fix entry for the active version with `Thanks @2ndNuts`. Validate with targeted tests and formatter, then use the changed gate via the repo's normal Testbox path if available. Include a Windows npm smoke result if the worker has access: run `node <npm-global-openclaw>/openclaw.mjs status` and `node <npm-global-openclaw>/openclaw.mjs --version`; both should exit within a few seconds and leave no matching `node.exe ... openclaw.mjs <command>` process.
