## Summary

Keep open. The report is current and has an open, unmerged linked fix PR with closing syntax, so the issue should not be closed before that PR is reviewed, revised, landed, or closed. Current main has a related CLI text-hang fix for plain `status`, but I did not find a merged fix that directly proves the Windows npm `node.exe ... openclaw.mjs status` lingering-process symptom is resolved.

## What This Changes

The issue asks OpenClaw to stop short-lived Windows npm CLI commands, especially `status`, from leaving `node.exe ... openclaw.mjs` processes alive after completion.

## Best Possible Solution

Review PR #74425 against the current CLI lifecycle and either land a scoped fix or replace it with narrower resource cleanup that makes Windows npm `status` and `--version` exit promptly while preserving complete stdout/stderr output and long-running gateway/provider behavior.

## Reproduction Assessment

Yes. The reporter provided a concrete Windows npm reproduction for `node <npm-global-openclaw>/openclaw.mjs status`, including observed lingering `node.exe` processes after 3 and 13 seconds; this review did not execute the Windows smoke locally.

## Solution Assessment

Unclear. A fix is still warranted, but the current linked PR's hard `process.exit()` direction may be too blunt compared with releasing the leaking process-wide resource or adding a narrower termination boundary with stream-drain proof.
