## Summary

Keep open. The report is current and reproducible enough, current main still returns from the CLI entrypoint without a normal-success shutdown path for routed `status`, and open PR #74425 is the linked implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue reports that short-lived Windows npm CLI invocations such as `status` and `--version` leave separate `node.exe ... openclaw.mjs` processes alive after command completion.

## Best Possible Solution

Review and land PR #74425 or a narrower replacement that closes the resource keeping short-lived Windows CLI commands alive, with Windows smoke coverage for `status` and `--version` and no behavior change to long-running gateway/node processes.

## Reproduction Assessment

Yes. The reporter gave a concrete Windows 11 npm reproduction with `status` and `--version`, timing observations, memory growth, and separation from the gateway process; I did not rerun it on Windows during this read-only review.

## Solution Assessment

Unclear. Keeping this issue open is the right state while PR #74425 is open, but the implementation should confirm whether explicit process exit is the narrowest maintainable fix or whether targeted teardown of the lingering resource is safer.
