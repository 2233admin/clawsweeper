## Summary

Keep this PR open. Current main still does not implement the PR’s intended observability hardening, and the submitted patch is not ready to close as obsolete: it is stale against the current allow-from helper, uses `info` for TLS permission failures, and the proposed `logWarn(..., { error })` calls can throw in formerly non-throwing fallback/cleanup paths.

## Best Possible Solution

Keep this PR open for author or maintainer revision. A safe version should rebase onto current main, move malformed allow-from JSON logging into `src/pairing/allow-from-store-file.ts`, log cleanup failures without passing metadata objects as `RuntimeEnv`, use warning severity for TLS chmod failures, preserve non-throwing fallback behavior, and add focused regression tests for these error paths.

## Risks / Open Questions

- The proposed `logWarn(..., { error })` calls can throw `TypeError` in cleanup/fallback paths because `{ error }` is not a `RuntimeEnv`, masking original I/O errors or making malformed allow-from files fatal.
- The TLS chmod failure would be logged at `info` in the PR, weaker than the warning severity requested by review for possible private-key/cert permission degradation.
- The pairing-store hunk is stale against current main because sync allow-from parsing now lives in `src/pairing/allow-from-store-file.ts`.
- No focused regression tests currently cover TLS chmod failure logging, file handle close-failure logging, or malformed allow-from JSON logging behavior.
