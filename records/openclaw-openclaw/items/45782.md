## Summary

Keep this PR open. Current main still does not implement the intended observability hardening, and the submitted patch is not ready to close as obsolete because it is stale against the current allow-from helper, logs TLS permission failures at info level, and passes `{ error }` to `logWarn` where the logger expects a `RuntimeEnv`. The diff does not add third-party code, dependencies, workflow changes, package metadata, or obvious supply-chain expansion, but it does introduce a functional/security risk in formerly non-throwing cleanup and fallback paths.

## Best Possible Solution

Keep this PR open for author or maintainer revision. A safe version should rebase onto current main, move malformed allow-from JSON logging into `src/pairing/allow-from-store-file.ts`, log cleanup failures without passing metadata objects as `RuntimeEnv`, use warning severity for TLS chmod failures, preserve non-throwing fallback behavior, and add focused regression tests for these error paths.

## Risks / Open Questions

- The submitted patch can make file-handle cleanup and malformed allow-from JSON fallback throw `TypeError` because `{ error }` is not a `RuntimeEnv`.
- TLS key/cert permission restriction failures would be logged at `info`, below the severity requested in review for a possible private-key/cert permission degradation.
- The allow-from logging hunk targets `pairing-store.ts`, while current main's sync parsing/fallback behavior lives in `src/pairing/allow-from-store-file.ts`.
- No focused regression tests currently prove TLS chmod failure logging, close-failure logging, or malformed allow-from JSON logging behavior.
