## Summary

Keep open. Current main still reports fast log growth as both truncation and reset, while this PR contains the narrow code/test fix for the `logs.tail` and `openclaw logs --follow` behavior; the remaining blocker is the required changelog entry plus normal maintainer merge review.

## What This Changes

The PR removes the fast-growth `reset = true` assignment in `readLogSlice` and adds log-tail unit coverage for fast growth versus real file shrink/rotation.

## Best Possible Solution

Land the semantic split so `reset` is reserved for cursor-past-EOF/file shrink cases and `truncated` covers byte-budget skips during fast growth, with the new regression tests and a user-facing changelog entry kept in the branch.

## Reproduction Assessment

Yes. The focused reproduction is to call `readConfiguredLogTail`, append more than `maxBytes` while keeping the prior cursor valid, then call it again; current main sets both `truncated` and `reset`, which the CLI renders as both truncation and rotation notices.

## Solution Assessment

Yes for the code path. The PR uses the narrow maintainable fix by changing only the incorrect flag assignment and adding tests for both affected branches; before merge it should also add the required changelog entry.
