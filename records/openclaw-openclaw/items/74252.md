## Summary

Keep open. Current `main` still reports fast log growth as a reset, while this PR makes the narrow semantic fix and adds focused regression coverage; the only actionable gap I found is the required user-facing changelog entry.

## What This Changes

The PR changes log tailing so fast growth beyond `--max-bytes` reports `truncated` without `reset`, keeps real file shrink/rotation reporting `reset`, and adds two regression tests in `src/logging/log-tail.test.ts`.

## Best Possible Solution

Land the semantic split and regression tests after adding a single Unreleased/Fixes changelog entry, keeping the broader `logs --follow` disconnect topic separate from this focused false-rotation notice fix.

## Reproduction Assessment

Yes. Against current `main`, read the log once, append more than `maxBytes` while the previous cursor is still within the file, then read with that cursor; `src/logging/log-tail.ts:92` sets `reset: true`, which the CLI renders as a rotation notice.

## Solution Assessment

Yes for the code path. Removing `reset` only from the fast-growth branch is the narrowest maintainable fix because real file shrink still sets `reset`, skipped bytes still set `truncated`, and the payload shape stays unchanged.
