## Summary

Keep open. The false rotation notice is a real current-main bug, but the proposed semantic split is incomplete because one internal Control UI consumer still uses `reset` to distinguish a skipped log range from a contiguous append.

## What This Changes

The PR narrows `readLogSlice` so fast log growth reports `truncated` without `reset`, and adds unit coverage for fast growth versus file shrink/rotation.

## Best Possible Solution

Keep `reset` reserved for true cursor invalidation while preserving skipped-range handling: update internal consumers such as the Control UI to replace or visibly separate entries when `truncated` means bytes were skipped, and add regression coverage for both CLI notices and Control UI append behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve log stream gap handling in the Control UI:** `src/logging/log-tail.ts:95-100`
  - body: Removing `reset` from the fast-growth payload fixes the CLI rotation notice, but `ui/src/ui/controllers/logs.ts` currently uses `payload.reset` to decide whether to replace `logsEntries` instead of appending. After this change, a `truncated: true, reset: false` response will append the latest max-bytes tail after old entries even though bytes were skipped, making the UI present a non-contiguous log as one stream. Please update that consumer or add a separate gap signal before changing the payload semantics.
