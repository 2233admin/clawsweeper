## Summary

Keep this PR open. Current main still does not implement the PR's intended graceful cron drain: hot reload still stops the old cron service synchronously, `CronService`/`CronServiceContract` have no `stopGraceful()` API, and cron state has no stopping flag or tracked in-flight timer/manual-run promises. Main has adjacent force-reload-before-persist protections, but those are not the same product behavior as quiescing the old service before replacement.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to refresh it against current main, preserve timer and manual-run drain coverage, add a bounded drain or explicit fallback for no-timeout/stuck runs, update the cron service contract and gateway reload tests, and then use the graceful stop path in `src/gateway/server-reload-handlers.ts` if maintainers want the stronger guarantee. If maintainers decide the current force-reload merge behavior is the intended product fix instead, close the PR only after documenting that decision against the linked cron state-overwrite report.

## Risks / Open Questions

- Functional availability risk before merge: the PR's `stopGraceful` approach waits for tracked run promises without a deadline, while current timeout policy maps `timeoutSeconds <= 0` to no timeout, so a stuck or intentionally unbounded cron run could block `cron.*` hot reload indefinitely.
- Product decision risk: current main has related force-reload stale-write mitigations, so maintainers should decide whether the stronger quiesce-before-replacement guarantee is desired before merging or closing the PR.
