## Summary

Keep open. Current main has partial Google Vertex support, but it is not equivalent to this PR's dedicated provider/service-account ADC work, and the latest PR head still has two concrete merge-blocking auth-path issues: ADC refresh bypasses the injected fetch seam, and the new ADC token cache is not keyed by credential identity.

## What This Changes

The PR registers a dedicated `google-vertex` provider in the Google plugin, adds Vertex region/catalog/ADC helpers, supports authorized_user and service_account token exchange, changes Vertex request routing, and updates Google transport tests.

## Best Possible Solution

Land or replace this with a maintainer-approved Google plugin implementation that gives `google-vertex` exact provider ownership, keeps selection-time and runtime ADC semantics aligned, supports authorized_user and service_account credentials, preserves injected fetch/proxy behavior, keys token caching by credential identity, and includes focused routing/auth tests plus user-facing setup docs.

## Reproduction Assessment

Yes for the PR findings. The latest branch source shows `resolveGoogleVertexAdcToken()` is called without the stream's injected fetch option, and `resolveGoogleVertexAdcToken()` returns an unexpired cached token before reading current ADC/env state.

## Solution Assessment

No, not as-is. The dedicated-provider direction is likely the maintainable boundary, but this patch should preserve fetch injection and credential-keyed token caching before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass the stream fetch into ADC refresh:** `extensions/google/transport-stream.ts:281`
  - body: `resolveGoogleVertexAdcToken()` always uses global `fetch`, while current main threads the stream option's fetch implementation into Vertex ADC refresh. Proxy-aware callers and tests that inject a custom fetch will have model requests use the guarded path but token refresh bypass it.
  - confidence: 0.88
- **[P2] Key the ADC token cache by credential source:** `extensions/google/vertex-adc.ts:151-152`
  - body: The cache returns any unexpired token before reading the current ADC path, refresh token, service account identity, or env. In a long-running gateway, switching `GOOGLE_APPLICATION_CREDENTIALS` can reuse a token minted for the previous account until expiry.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Security review needs attention because the PR adds ADC token minting and has a concrete credential-isolation issue in the token cache.

Concerns:

- **[medium] Unkeyed ADC token cache can reuse prior credentials:** `extensions/google/vertex-adc.ts:151`
  - body: The new cache returns an unexpired token without checking the current credential path, refresh token, service account identity, or project env. A gateway that rotates or switches Google credentials can send Vertex requests using a previous account's token.
  - confidence: 0.84
