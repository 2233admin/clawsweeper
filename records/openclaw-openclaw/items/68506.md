## Summary

Keep this PR open. Current main still lacks the structured `Cause:` / `Fix:` recovery output for `status --all`, while the PR targets the right command surface but needs rebase and review fixes before it is mergeable.

## What This Changes

The branch adds recovery-guidance helpers and tests to `status --all` diagnosis output for invalid config, gateway log and health failures, unreachable or query-failed gateways, bind failures, and OAuth re-auth hints.

## Best Possible Solution

Land a rebased version of this status diagnosis improvement that keeps the structured recovery hints, reuses the shared OAuth refresh-failure helper, avoids duplicate unreachable guidance, bounds HTTP status-code matching, and adds regression tests for those edge cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Reuse the sanitized OAuth refresh helper:** `src/commands/status-all/diagnosis.ts:64-67`
  - body: The provider parsed from health error text is interpolated directly into a pasteable login command. Current main already has `classifyOAuthRefreshFailure` and `buildOAuthRefreshFailureLoginCommand` to sanitize provider IDs before formatting commands; use those helpers here so malformed health text cannot shape the suggested command.
  - confidence: 0.86
- **[P2] Match HTTP auth codes with boundaries:** `src/commands/status-all/diagnosis.ts:123-128`
  - body: `lower.includes("401")` and `lower.includes("403")` can match unrelated values such as ports or durations before the transport-failure checks run, causing `status --all` to suggest gateway auth fixes for connectivity failures. Match bounded HTTP status tokens or prioritize transport errors first.
  - confidence: 0.84
- **[P2] Deduplicate unreachable recovery guidance:** `src/commands/status-all/diagnosis.ts:393-397`
  - body: When the health error is `gateway unreachable` and `gatewayReachable` is false, the channel-skip block emits the unreachable cause/fix and the health classifier emits effectively the same guidance again under `Gateway health:`. Suppress one path or make the health section aware that the skipped-channel diagnosis already covered it.
  - confidence: 0.82
