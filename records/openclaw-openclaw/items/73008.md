## Summary

Keep open. Current main still lacks the DeepSeek provider thinking profile, and this open PR is the active, narrow implementation for the linked DeepSeek V4 `/think xhigh|max` validation bug, so the right next step is maintainer review and current-head validation rather than cleanup closure.

## What This Changes

The PR adds a DeepSeek V4 provider thinking profile exposing `off|minimal|low|medium|high|xhigh|max`, shares the V4 model matcher, adds regression coverage, and updates DeepSeek/thinking docs plus the changelog.

## Best Possible Solution

Land the provider-owned DeepSeek V4 thinking profile with the shared local matcher, keep the positive V4 and negative non-V4 regression coverage, publish the docs/changelog updates, then close the linked native DeepSeek issues while treating #74788 as a separate OpenRouter provider follow-up.

## Reproduction Assessment

Yes. The failure is reproducible by static trace on current main: `/think max` or `/think xhigh` reaches directive validation, DeepSeek has no provider profile, the base profile omits those levels, and the directive is rejected before the DeepSeek V4 stream wrapper can map the payload.

## Solution Assessment

Yes. The proposed path is the narrowest maintainable fix because it keeps DeepSeek-specific level exposure in the DeepSeek plugin and relies on the existing shared transport wrapper for request shaping instead of changing core thinking fallback behavior.
