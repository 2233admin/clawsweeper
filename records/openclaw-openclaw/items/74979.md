## Summary

Keep open. The bug is valid and current main still has the unsafe prompt-wrapped-after-creation shape, but this PR is still a partial fix because Pi can do async preflight after the new guard and before `Agent.prompt()` creates an active run; aborts in that window can still leave an uncancelled prompt chain.

## What This Changes

The PR adds an abort-state guard before embedded Pi prompt submission, aborts and clears the active Pi agent during abort/cleanup, and adds regression coverage for abort-before-prompt zombie loops.

## Best Possible Solution

Make the actual Pi prompt-start boundary abort-aware: check the run abort signal immediately before Pi calls `Agent.prompt()` and fail without starting `activeRun`, then keep focused coverage for pre-aborted, abort-before-activeRun, and abort-after-activeRun cases.

## Reproduction Assessment

Yes. #74859 gives a deterministic pre-aborted-signal reproduction, and the remaining PR gap can be reproduced by delaying `AgentSession.prompt()` preflight after this PR's guard and aborting before `Agent.prompt()` creates `activeRun`.

## Solution Assessment

No. The PR is a useful partial fix for the already-aborted and abort-after-activeRun paths, but `agent.abort()` is still a no-op before Pi creates `activeRun`, so the narrow maintainable fix needs to move cancellation to the prompt-start boundary.
