## Summary

Keep open. The reported `sessions_spawn` model override bug is valid and this PR targets the right surface, but the current branch is not mergeable: it mixes in broken Gateway HTTP RPC wiring and the model-forwarding change still lacks authorization proof for the native subagent Gateway call path.

## What This Changes

The PR forwards the resolved subagent model as `provider`/`model` params on the Gateway `agent` call, while also leaving stray Gateway HTTP RPC stage wiring and a whitespace-only server plugin change.

## Best Possible Solution

Keep the focused `sessions_spawn` fix, remove the unrelated HTTP RPC/whitespace changes, route the explicit subagent model through an authorized native subagent path or persisted override field that `agentCommandFromIngress` actually consumes, and add a seam test that proves the requested model reaches the runtime without granting owner-only behavior accidentally.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Authorize the forwarded model override:** `src/agents/subagent-spawn.ts:1083-1090`
  - body: This spread sends `provider`/`model` on the Gateway `agent` call, but native subagent calls use the least-privilege write path while the Gateway rejects provider/model overrides unless the caller is admin/owner or has `internal.allowModelOverride`. Because `resolvedModel` always falls back to a model, this can make ordinary native subagent spawns fail with an unauthorized model override instead of starting.
  - confidence: 0.86
- **[P1] Remove the undefined HTTP RPC handler call:** `src/gateway/server-http.ts:571`
  - body: The added stage calls `handleHttpRpcEndpoint`, but the final PR diff does not leave any import or `src/gateway/http-rpc-handler.ts` module behind. A build/typecheck will fail before the model-forwarding fix can ship, so drop this unrelated line or add the handler in a separate reviewed PR.
  - confidence: 0.98
- **[P1] Register HTTP RPC as a separate stage:** `src/gateway/server-http.ts:570-571`
  - body: The new `http-rpc` `name`/`run` pair is inserted inside the existing `gateway-probes` object, creating duplicate `name` and `run` keys rather than a separate request stage. If this route belongs at all, it needs to be its own object in a separate Gateway HTTP RPC change.
  - confidence: 0.95
