## Summary

Keep open. The report is recent, detailed, and an open implementation PR, #73627, explicitly uses closing syntax for this issue. Current main still has the cron embedded-run call passing the run session key without an explicit sandboxSessionKey, while the embedded attempt resolves sandbox policy from sandboxSessionKey || sessionKey || sessionId before calling resolveSandboxContext. The docs contract says sandbox mode off means no sandboxing, and the Docker image-inspect error path still exists if the backend is reached.

## Best Possible Solution

Keep this issue open and review #73627 or an equivalent maintainer patch. The fix should pass the correct sandbox/runtime-policy session key for isolated cron, heartbeat, and sub-agent paths, keep Docker/backend initialization behind the sandboxed=true short-circuit, and add regression coverage proving sandbox.mode off never touches Docker for those automated sessions.
