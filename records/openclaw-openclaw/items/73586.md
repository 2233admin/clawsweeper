## Summary

Keep open. This is a fresh, detailed regression report with an open implementation candidate, #73627, that explicitly uses closing syntax for this issue, plus an alternative open fix attempt in #73671. Current main has not incorporated the main call-site change proposed by #73627: isolated cron still calls `runEmbeddedPiAgent` with the run-scoped `sessionKey` and no explicit `sandboxSessionKey`, while the embedded runner already has a separate sandbox-policy key parameter and falls back to `sessionKey` when it is absent. The generic mode-off guard exists, but this issue should not be closed before maintainers choose and land a fix with caller-specific regression coverage proving Docker is not touched for automated sessions when sandboxing is off.

## Best Possible Solution

Keep this issue open and review #73627, #73671, or a maintainer-authored replacement. The fix should ensure cron, heartbeat, and sub-agent session startup resolve sandbox policy from the intended session/runtime-policy key, make `sandbox.mode: "off"` unable to initialize Docker/backend code, and add focused regression coverage that fails if Docker image inspection is reached while sandboxing is off.
