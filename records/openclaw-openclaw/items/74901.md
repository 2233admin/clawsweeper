## Summary

Keep open. The PR addresses a real TTS-to-Telegram voice delivery regression, but the proposed patch broadens local-file media trust from the current registered/internal tool provenance model to a result-controlled `trustedLocalMedia` flag, matching the existing ClawSweeper security finding in the PR discussion.

## What This Changes

This PR changes `filterToolResultMediaUrls` so `details.media.trustedLocalMedia === true` can bypass the exact `builtinToolNames` local-media gate.

## Best Possible Solution

Keep the intended TTS delivery fix, but constrain local-path authorization to an internally trusted exact `tts` source with non-external provenance, and add positive coverage for the missing-`tts` registration case plus negative coverage for MCP, unregistered, and case-variant tool results.

## Reproduction Assessment

Yes. The linked TTS report gives concrete user steps, and the PR regression has a focused source-level reproduction: an MCP or unregistered tool result can set `details.media.trustedLocalMedia: true` for a local path that current main rejects.

## Solution Assessment

No. Respecting TTS-owned `trustedLocalMedia` is a reasonable direction, but this implementation is broader than the current trust model because the result object itself can bypass external-result and exact-name checks.

## Review Findings
