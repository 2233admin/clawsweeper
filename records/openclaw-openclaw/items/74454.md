## Summary

Keep open. The workspace dotenv blocklist part is useful, but the latest PR head no longer implements the stated `reg.exe` path hardening: it preserves `SystemRoot`/`WINDIR`-derived registry executable candidates and adds a test expecting a readable non-C system root to win. That makes the patch incomplete for the security behavior described in the PR body and needs maintainer/security review before merge.

## What This Changes

The PR adds `SYSTEMROOT` and `WINDIR` to the workspace dotenv blocklist and updates Windows install-root registry-probing tests/call sites while preserving env-derived `reg.exe` candidate behavior on the latest head.

## Best Possible Solution

Land a maintainer-approved security fix that blocks workspace `.env` from setting Windows system-root variables and makes registry probing execute only a trusted, non-workspace-controlled `reg.exe` path, with tests that assert env-derived registry executable paths are not selected.
