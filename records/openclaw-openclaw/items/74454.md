## Summary

Keep open. Current main still has the reported Windows registry executable trust-boundary behavior, and the latest PR head only partially fixes it: workspace `.env` blocks `SystemRoot`/`WINDIR`, but the branch preserves env-derived `reg.exe` selection instead of enforcing the fixed executable path described in the PR body.

## What This Changes

The PR adds `SYSTEMROOT` and `WINDIR` to the workspace dotenv blocklist, extends dotenv tests, changes the registry query call site, and adds a test that prefers an env-derived non-`C:` `reg.exe` candidate.

## Best Possible Solution

Land a maintainer-approved security hardening that blocks workspace `.env` from setting Windows system-root variables and makes registry probing execute only a trusted registry client path, with tests that assert env-derived `reg.exe` paths are not selected.
