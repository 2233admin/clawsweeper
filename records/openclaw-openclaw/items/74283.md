## Summary

Keep open. Current main still allows the reported leak shape: unmatched approval follow-up text is parsed as raw `other`, passed through the generic sanitizer, and sent by the session-resume failure fallback when a deliverable channel route exists. The issue discussion and follow-up maintainer comment also identify this as distinct from #72184's recovery-queue path.

## What This Changes

Prevent the `/approve` post-approval follow-up fallback from sending raw internal handler/source-like text to Telegram when automatic session resume fails.

## Best Possible Solution

Add a fail-closed boundary for direct exec approval follow-ups: only verified structured exec completion or denial payloads should be eligible for direct channel fallback, malformed/source-like payloads should become a generic unavailable status or be dropped, and the producer path should be traced so handler-source-like text cannot populate `resultText`. Cover the session-resume failure path with regression tests using the reported JavaScript-like fragments.
