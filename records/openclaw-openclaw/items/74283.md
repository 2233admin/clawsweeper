## Summary

Keep open. Current main still has the reported approval-followup failure shape: unrecognized `resultText` is parsed as `kind: "other"`, passed through the generic user-facing sanitizer, and sent directly to the originating channel when the follow-up `agent` session resume fails. The discussion correctly separates this from #72184, which covered a different restart delivery-recovery path.

## What This Changes

Prevent the `/approve` post-approval follow-up recovery path from sending raw internal handler/source-like `resultText` directly to Telegram when session resume fails.

## Best Possible Solution

Add a fail-closed boundary in the exec approval direct fallback so only verified structured exec completion or denial payloads are eligible for direct channel delivery; malformed/source-like payloads should be dropped or replaced with generic unavailable copy. Then trace and fix the producer so handler-source-like state cannot become `resultText`, with regression tests for the reported snippets.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, deliverable Telegram route, rejected `callGatewayTool("agent", ...)`, and JavaScript-like `resultText`; current main parses it as `kind: "other"` and sends sanitized raw text through direct fallback.

## Solution Assessment

Partly. The suggested Layer 1 guardrail is the narrow immediate containment, but the safer durable fix is to validate structured follow-up payloads at this boundary and separately repair the producer that can create malformed `resultText`.
