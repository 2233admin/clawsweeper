## Summary

Keep open. Current `main` still has the approval-followup fallback shape described in the report: malformed or source-like `resultText` is parsed as unstructured text, sanitized generically, then eligible for direct channel delivery after session resume fails. The issue is narrow and current comments correctly distinguish it from the closed restart-recovery path, but it is an information-leak boundary and should get maintainer review before repair automation.

## What This Changes

The issue reports that approval follow-up fallback delivery can send raw internal handler-source-like text to Telegram after `/approve` session resume fails.

## Best Possible Solution

Add a fail-closed direct-delivery boundary for exec approval follow-ups so only verified structured exec completion or denial payloads can be posted directly; malformed/source-like payloads should be dropped or replaced with a generic unavailable message. Then trace the producer so internal handler state cannot become `resultText`, with regression coverage for the reported JavaScript-like snippets.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, a deliverable Telegram route, a rejected `callGatewayTool("agent", ...)`, and a JavaScript-like `resultText`; current `main` parses it as `kind: "other"` and sends the sanitized raw text through direct fallback.

## Solution Assessment

Unclear for the full two-layer proposal. The Layer 1 fail-closed guard is the narrowest immediate containment, but the best complete fix also needs producer tracing so handler-source-like state cannot enter `resultText` in the first place.
