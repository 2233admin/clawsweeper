## Summary

Keep open. Current main still allows the reported approval-completion fallback path to deliver unrecognized `resultText` directly after session resume failure, and the existing sanitizer does not fail closed on JavaScript/source-like payloads. The discussion correctly distinguishes this from #72184, and #68648 is only an adjacent session-wait improvement, not a containment fix for this leak boundary.

## What This Changes

The issue reports that exec approval follow-up recovery can send raw internal handler/source-like text to Telegram when session resume fails and direct fallback delivers unstructured `resultText`.

## Best Possible Solution

Add a fail-closed boundary in the exec approval direct fallback so only recognized structured completion/denial payloads, or otherwise verified safe user-facing text, can be sent directly after session resume failure. Then trace the producer path that can put handler-state/source-like text into resultText and add regression coverage for the reported JavaScript-like snippets.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call sendExecApprovalFollowup with a session key, deliverable Telegram route, a rejected callGatewayTool("agent", ...), and JavaScript-like unstructured resultText; current main parses it as kind "other" and sends the sanitized raw text through direct fallback. I did not run tests or live Telegram because this review was explicitly read-only.

## Solution Assessment

Unclear for the full two-layer proposal. The Layer 1 fail-closed delivery guard is the narrow immediate containment, while the complete fix should also repair the upstream producer so internal handler state cannot become resultText.
