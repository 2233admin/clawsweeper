## Summary

Keep open. Current main still permits the reported approval-followup leak shape: malformed or source-like approval `resultText` is parsed as unstructured text, passed through the generic sanitizer, and sent directly after session resume fails. The discussion correctly distinguishes this from the already-closed restart delivery-recovery issue.

## What This Changes

The issue reports that `/approve` post-approval fallback delivery can send raw internal handler-source-like text to Telegram after session resume fails.

## Best Possible Solution

Add a fail-closed boundary for exec approval follow-up direct fallback so only verified structured exec completion or denial payloads can be posted directly; malformed/source-like payloads should be dropped or replaced with generic unavailable copy. Then trace the producer so internal handler state cannot become `resultText`, with regression coverage for the reported JavaScript-like snippets.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to call `sendExecApprovalFollowup` with a session key, a deliverable Telegram route, a rejected `callGatewayTool("agent", ...)`, and JavaScript-like `resultText`; current main parses it as `kind: "other"` and sends the sanitized raw text through direct fallback.

## Solution Assessment

Unclear for the full two-layer proposal. The immediate fail-closed guard is the narrowest containment, but the complete fix also needs producer tracing so handler-source-like state cannot enter `resultText` in the first place.
