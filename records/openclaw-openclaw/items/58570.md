## Summary

Keep open. Current main still drops Slack messages denied by explicit channel policy through verbose logging only, and an open related PR already references this issue with closing syntax, so the issue should remain open until that PR is reviewed and merged or replaced.

## What This Changes

Request a structured warning when a resolved channel route denies a message because the channel entry is explicitly disabled by `allow: false` or its current `enabled: false` equivalent.

## Best Possible Solution

Land or replace the linked implementation with a narrow change that emits a warning-level, machine-readable diagnostic for explicit denied channel routes while preserving fail-closed routing semantics; if multiple channel plugins share the same gap, use the existing channel logging SDK helper or extend it rather than adding divergent ad hoc messages.

## Reproduction Assessment

Yes. A Slack room message whose channel resolves to a disabled entry (`enabled: false`, the current equivalent of legacy `allow: false`) follows the source path through `ctx.isChannelAllowed` and is dropped with only `logVerbose`; the existing tests already exercise that denial branch.

## Solution Assessment

Unclear. Raising the denied Slack drop from verbose to warning is the narrowest immediate fix for the observed incident, but the best final patch should also include the structured reason and avoid noisy repeated logs, and maintainers should decide whether this needs a generic channel helper.
