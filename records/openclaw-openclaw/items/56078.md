## Summary

Keep open. I rechecked current main at f256eeba431b5f59e6085b7c11c9c0a309e408b7 and the core no-channel cron path is still not fixed by default: Control UI isolated cron jobs still default to announce delivery, backend-created isolated agent-turn jobs still default missing delivery to announce, and unresolved announce targets still become delivery-target errors unless best-effort delivery is explicitly enabled. The related PRs called out in the issue discussion are useful context, but the provided GitHub context marks them closed and unmerged, and current main still matches the earlier ClawSweeper keep-open finding.

## Best Possible Solution

Keep this issue open and fix the cron product path directly. The conservative fix is to make local/webchat-only isolated cron jobs that are intended to run internally use delivery.mode="none" by default when no deliverable channel exists, or to downgrade unresolved implicit announce delivery non-fatally without masking explicit delivery-target failures. Add regression coverage for a Control UI-created isolated cron job in a no-channel config and update the public cron docs for the default/workaround.
