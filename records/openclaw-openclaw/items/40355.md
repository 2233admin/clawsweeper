## Summary

Keep open. Current main still lacks a generic doctor/config check for default and per-agent model refs, and this PR remains a useful implementation candidate for linked #39811, but the current head has blocking correctness issues around catalog source semantics and alias resolution plus a low-severity terminal-output hygiene issue.

## What This Changes

The PR adds a config-validation helper, a doctor health contribution, and tests to warn when configured default or per-agent primary model references are missing from a loaded model catalog.

## Best Possible Solution

Land a focused doctor or shared validation path that resolves model refs with the same alias/provider semantics as runtime and model-picker code, validates against provider/runtime/static catalog evidence without treating user-declared configured rows as proof, keeps normal doctor inspection read-only where feasible, sanitizes terminal output, and leaves broader fallback notification work to #39818.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate against catalog rows that do not come from config:** `src/flows/doctor-health-contributions.ts:408`
  - body: The new doctor check passes `loadModelCatalog({ config: ctx.cfg })`, but that catalog appends `models.providers.*.models` from the same config. In linked #39811, the bogus model is first added under `models.providers.openai.models`, so this check would treat it as present and emit no warning. Use a catalog/source that distinguishes provider/runtime/static rows from user-declared config rows, or carry origin metadata into the validation.
  - confidence: 0.88
- **[P2] Resolve aliases before checking configured primaries:** `src/config/validation.ts:1344`
  - body: `collectModelConfigWarnings` calls `resolveModelRefFromString` without `cfg` or an alias index, so a valid configured alias such as `fast` is parsed as `anthropic/fast` and warned as missing. Runtime and model-picker paths build `agents.defaults.models` aliases before resolving, so doctor should use the same resolution semantics.
  - confidence: 0.9
- **[P3] Sanitize model warning text before printing it:** `src/flows/doctor-health-contributions.ts:411`
  - body: The warning message includes model identifiers derived from local config and is passed straight to `note()`. A model id containing ANSI or control characters can spoof doctor output; route the path and message through the existing terminal sanitizers before display.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch does not add dependencies, workflows, or package-resolution changes, but it does print config-derived model warning text without terminal sanitization.

Concerns:

- **[low] Unsanitized doctor warning output:** `src/flows/doctor-health-contributions.ts:411`
  - body: Model ids and warning paths from local config are interpolated into `note()` output without `sanitizeTerminalText`, so control characters in config can spoof or obscure doctor output. This is local-config scoped but should be fixed before merge.
