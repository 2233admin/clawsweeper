## Summary

Keep this PR open. Current main still has the IRC proactive-send behavior the PR targets: without an explicit ready client, `sendMessageIrc` opens a transient IRC connection, sends `PRIVMSG`, and quits. The PR is not implemented on main, and the same-author #47234 stack is still open rather than a shipped superseding fix. The PR remains plausible but needs maintainer review for process-boundary and JOIN-readiness concerns before it is the right final implementation.

## Best Possible Solution

Keep this PR open for maintainer review. The strongest implementation is a focused IRC send path that reuses a verified joined monitor/gateway-owned connection when available, preserves a clear no-monitor fallback, avoids process-local-only assumptions for standalone proactive sends, and adds targeted tests for reuse, fallback, identity-safe cleanup, and JOIN confirmation timing. Coordinate with #47234 so the transport fix can land separately from the groupScope routing proposal if maintainers want the smaller bug fix first.

## Risks / Open Questions

- A process-local client registry would not fix proactive sends launched from a separate CLI process unless those sends are routed through the gateway or another shared monitor-owned execution path.
- Publishing the monitor client as active before server-confirmed JOIN membership can still race startup or reconnect on networks that require membership before channel PRIVMSG delivery.
- Current main tests do not cover monitor-client reuse, transient fallback, identity-safe deregistration, or JOIN/readiness timing for proactive IRC sends.
- The same-author #47234 branch includes this work plus routing changes; maintainers should avoid merging the IRC transport fix accidentally bundled with unrelated group-scope behavior.
