## Summary

Current main still has the IRC proactive-send behavior this PR targets: `sendMessageIrc` only reuses an explicitly supplied client, otherwise it opens a transient IRC connection that is not joined to configured channels. The PR is narrow and relevant, but it still needs maintainer review around process-local client state, JOIN readiness, and regression coverage before merge.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused IRC delivery fix that reuses a verified joined monitor or gateway-owned connection when available, preserves the no-monitor fallback deliberately, handles direct-delivery process boundaries explicitly, and adds targeted regression tests for monitor reuse, identity-safe cleanup, fallback behavior, and startup JOIN timing. Coordinate with #47234 so the small IRC delivery fix can land separately from the broader group-scope routing proposal if maintainers want the delivery bug fixed first.
