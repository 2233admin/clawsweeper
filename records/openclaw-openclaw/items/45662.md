## Summary

Keep this PR open. Current main still does not implement the IRC active monitor-client reuse described by the PR: proactive IRC sends without an explicit client still create a transient connection, send PRIVMSG, and quit. The same-author #47234 is an open stacked PR that includes this work rather than a merged replacement, and the supplied review context raises unresolved design concerns that should be addressed before landing.

## Best Possible Solution

Keep this PR open for maintainer review, but revise it before landing. The best fix is to make IRC proactive sends use the gateway/monitor-owned IRC connection for proactive surfaces that have a running gateway, preserve a clear fallback for no-monitor cases, and avoid channel sends until the persistent client has actually joined or otherwise has a verified delivery path. Add targeted IRC tests for active-client reuse, fallback behavior, identity-checked removal, and startup join timing, and coordinate with the open #47234 stack rather than treating it as already shipped.

## Risks / Open Questions

- The PR's process-local registry would not help proactive sends executed from a separate CLI process unless those sends are routed through the gateway or another shared monitor-owned connection path.
- Publishing the monitor client after `connectIrcClient` resolves can still race IRC channel JOIN confirmation because readiness resolves after sending JOIN commands, not after membership is confirmed.
- The PR currently lacks focused regression tests for persistent-client reuse, transient fallback, identity-checked deregistration, and join/readiness timing.
