## Summary

Keep this PR open. Current main still does not implement the PR’s intended IRC monitor-client reuse: proactive sends without an explicit client still create a transient IRC connection, send PRIVMSG, and quit. The same-author #47234 stack is also still open and includes this work rather than superseding it as shipped, and the remaining review concerns around process-local lookup and JOIN readiness are still relevant.

## Best Possible Solution

Keep this PR open for maintainer review and coordinate it with the open #47234 stack. The strongest fix is a monitor/gateway-owned IRC send path that reuses a verified joined connection when available, preserves a clear no-monitor fallback, avoids process-local-only assumptions for CLI proactive sends, and adds focused IRC tests for reuse, fallback, identity-safe deregistration, and JOIN/readiness timing.

## Risks / Open Questions

- The proposed in-memory client registry would not help proactive IRC sends executed in a separate CLI process unless those sends route through a shared gateway/monitor-owned connection path.
- Publishing a monitor client as soon as `connectIrcClient` resolves can still race channel JOIN confirmation on servers that require membership before channel PRIVMSG delivery.
- Current tests do not cover active monitor-client reuse, transient fallback, identity-checked deregistration, or startup JOIN timing for proactive IRC sends.
