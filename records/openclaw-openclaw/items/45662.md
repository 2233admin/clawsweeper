## Summary

Keep this PR open. Current main still does not implement the proposed IRC monitor-client reuse for proactive sends, and the same-author #47234 stack is open and unmerged rather than a shipped supersession. The PR is small and security-sensitive scope looks contained, but maintainers still need to resolve correctness questions around process boundaries, JOIN readiness, and regression coverage.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is a focused IRC transport fix that reuses a verified joined monitor or gateway-owned connection when available, preserves the transient no-monitor fallback intentionally, handles cross-process proactive sends explicitly, and adds targeted tests for reuse, fallback, cleanup identity, and JOIN readiness. Coordinate with #47234 so the smaller delivery fix can land separately from the broader groupScope routing proposal if maintainers want the bug fix first.

## Risks / Open Questions

- A process-local active-client registry would not fix proactive IRC sends that execute from a separate CLI process unless those sends are routed through the gateway or another shared monitor-owned process.
- Publishing the monitor client as active immediately after connect can still race channel membership because readiness resolves after JOIN commands are sent, not after confirmed membership.
- The PR still needs targeted regression tests for monitor-client reuse, fallback when no monitor is running, identity-safe cleanup, and JOIN readiness before it is safe to land.
