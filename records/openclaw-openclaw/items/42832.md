## Summary

Keep this PR open. Current main still has the two Nostr DM delivery failures that make the PR useful: Nostr gateway startup resolves immediately and the DM subscription still passes an array-wrapped filter even though the pinned `nostr-tools` 2.23.3 API expects a single `Filter`. The PR should not merge as-is because the lifecycle code now lives in `extensions/nostr/src/gateway.ts`, and the proposed `sendMedia` fallback conflicts with current no-media Nostr metadata/docs and has a raw-URL/path disclosure risk.

## Best Possible Solution

Keep this PR open as a useful but stale implementation candidate. The maintainer path is a refreshed Nostr-owned patch on current main: make `startNostrGatewayAccount()` stay pending until `ctx.abortSignal` with idempotent cleanup, pass the single `Filter` object required by pinned `nostr-tools` 2.23.3 in the DM subscription path, add focused lifecycle and subscription-shape regression tests, and leave media unsupported unless a separate public-URL-only media design updates the channel contract. Open #53858 should remain the canonical user-facing restart-loop report until that corrected fix lands.

## Risks / Open Questions

- Merging this PR as-is is likely stale against current main because the Nostr lifecycle implementation now lives in `extensions/nostr/src/gateway.ts`, while the PR's lifecycle patch targets the older inline `extensions/nostr/src/channel.ts` path.
- The proposed `sendMedia` fallback would mark Nostr as media-capable even though current metadata and docs say media attachments are unsupported.
- Security review: the PR's `sendMedia` fallback formats the raw `mediaUrl` directly into DM text. Since runtime send paths pass `mediaUrl` through when an adapter implements `sendMedia`, this could disclose local filesystem paths or send unusable non-public attachment references to Nostr recipients.
- Closing this PR now would discard useful root-cause work while current main still has the Nostr restart-loop and `subscribeMany()` filter-shape problems.
