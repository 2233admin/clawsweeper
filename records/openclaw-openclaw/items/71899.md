## Summary

Keep open. Current main still has the reported `agents add` nested workspace default, and this PR remains a relevant implementation candidate, but the patch is not ready to merge because its managed-workspace guard still misses a reachable profile-transition setup case and there are overlapping open PRs for the same bug that need a canonical maintainer choice.

## What This Changes

The PR changes agent workspace resolution to avoid treating setup-created main workspaces as shared bases for added agents, adds resolver regression tests, documents peer-level workspaces, and adds a changelog entry for #71889.

## Best Possible Solution

Land one canonical fix for #71889 that makes brand-new `agents add` wizard defaults peer-level, preserves explicit workspaces and intentional shared-base behavior, adds direct wizard initial-value coverage plus resolver/helper edge coverage, and then close the duplicate implementation PRs after the chosen fix merges.

## Reproduction Assessment

Yes. The source path is high-confidence: setup persists `agents.defaults.workspace`, the wizard uses `resolveAgentWorkspaceDir(cfg, agentId)` for the prompt default, and the current resolver joins that fallback with the non-default id.

## Solution Assessment

No. This patch improves the default setup case, but changing the runtime resolver is broader than the reported wizard prefill and the new guard still misses a profiled setup workspace when the later `agents add` run uses a different profile environment.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Recognize profiled setup workspaces before nesting:** `src/agents/agent-scope-config.ts:72-73`
  - body: The guard only treats `<stateDir>/workspace` and the current profile's `resolveDefaultAgentWorkspaceDir(env)` as setup-managed. If setup ran with `OPENCLAW_PROFILE=dev`, it persists `<stateDir>/workspace-dev`; running `agents add work` later with the default profile leaves that fallback unrecognized, so this branch still returns `<stateDir>/workspace-dev/work`. That is the same nested-under-main layout this PR is meant to prevent for profile transitions.
  - confidence: 0.84
