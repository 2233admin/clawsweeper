## Summary

Keep open. Current main still extracts and authorizes the trusted-proxy user, but the WebSocket client record does not preserve that server-authenticated identity and WebChat chat.send builds sender fields from client-declared Control UI metadata. The issue is actionable, but the exact public identity contract crosses a trusted-proxy security boundary and should be chosen by maintainers before an automated repair.

## What This Changes

Request to propagate the trusted-proxy X-Forwarded-User identity from WebChat gateway authentication into the agent message context, likely through sender identity fields and optionally session metadata.

## Best Possible Solution

Persist the accepted trusted-proxy user on server-owned WebSocket client state after handshake, derive WebChat message identity from that stored value, and add regression coverage proving client-declared WebChat metadata cannot spoof proxy identity. If maintainers choose a distinct authenticated-user field, document and version that context contract alongside the existing sender-field behavior.

## Reproduction Assessment

Yes. The issue gives concrete proxy/config steps, and current-main source inspection reproduces the gap: trusted-proxy auth returns a user, the WebSocket client does not store it, and chat.send uses generic Control UI client metadata for sender fields.

## Solution Assessment

Unclear until maintainer review chooses the public identity contract. The implementation boundary should be server-side propagation from authResult.user; reusing SenderId/SenderName may be right, but a distinct authenticated-user field is a plausible safer contract.
