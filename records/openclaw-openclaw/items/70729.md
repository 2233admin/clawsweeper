## Summary

Current main still does not surface the trusted-proxy `X-Forwarded-User` identity into WebChat agent message context. Trusted-proxy auth extracts and authorizes the user, but the WebSocket client state does not carry that user forward, and `chat.send` still fills `MsgContext` sender fields from generic Control UI client metadata.

## Best Possible Solution

Keep this open. The best path is a gateway-owned design decision on the agent-visible identity contract, then a narrow implementation that persists the accepted trusted-proxy user on the WebSocket connection and uses that stored server-authenticated identity for WebChat `chat.send` sender context, with negative tests proving client metadata cannot spoof it.
