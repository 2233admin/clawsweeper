## Summary

Keep open. Current main still does not implement Linux desktop notification delivery for the existing core `system.notify` node surface: the Linux gateway policy allows the command, but the headless node host does not advertise or handle it, docs and CLI still describe notification delivery as macOS/node-app-only, and plugins cannot register the reserved `system.notify` command to fill the exact gap. The related closed #24616 covered the error experience, not Linux notification delivery.

## What This Changes

Add Linux desktop notification support for OpenClaw node notifications, preferably through `notify-send`/libnotify with graceful fallback on headless Linux sessions.

## Best Possible Solution

Implement a focused core Linux headless node-host `system.notify` path with desktop-session/backend detection, structured unavailable or permission-style failures, updated CLI/docs/changelog, and tests for advertisement plus invocation behavior while leaving macOS, iOS, and Android notification behavior unchanged.

## Repair Work Prompt

Implement Linux desktop notification support for the core headless node-host `system.notify` command. Add a focused node-host implementation that detects Linux desktop availability (`DISPLAY` or `WAYLAND_DISPLAY`) and an available backend such as `notify-send`; return structured unavailable/failure errors when no desktop session or backend exists. Spawn backends without a shell and pass title/body as argv fields only. Advertise `system.notify` from `src/node-host/runner.ts` only when the node host can support the command, or document/test the chosen unavailable-at-invocation behavior if maintainers prefer always advertising on Linux desktops. Update `openclaw nodes notify` wording, nodes/node docs, and add a single-line changelog entry with `Thanks @holistic-medical-supply-base` (and credit @origamium only if the related error UX from #24616 is touched).
