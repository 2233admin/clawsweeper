## Summary

Keep open. Current main still lacks Linux desktop notification delivery for the reserved core `system.notify` node command: Linux is allowlisted at the Gateway policy layer, but the headless node host neither advertises nor dispatches `system.notify`, and CLI/docs still describe notification support as macOS-app or non-headless behavior. The related closed #24616 addressed the misleading unsupported-command error path, not Linux notification delivery.

## What This Changes

The issue requests Linux desktop notification support for OpenClaw node notifications, primarily via `notify-send`/libnotify or D-Bus with graceful headless fallback.

## Best Possible Solution

Implement a focused core Linux headless node-host `system.notify` path with desktop-session detection, safe backend invocation, structured unavailable results for missing display/backend, docs and CLI updates, and tests proving advertisement plus invocation behavior while leaving macOS, iOS, and Android notification behavior unchanged.

## Repair Work Prompt

Fix Linux desktop notification support for the core `system.notify` node command requested by https://github.com/openclaw/openclaw/issues/14079. Implement this in the headless node-host path, not as a plugin override: advertise `system.notify` from `openclaw node run` on Linux-capable headless hosts, dispatch it in the node-host invoke path, and use a safe Linux backend such as `notify-send` with `child_process.spawn`/argv only and `shell: false`. Gracefully return a structured unavailable-style result when there is no desktop session (`DISPLAY`/`WAYLAND_DISPLAY` absent) or no supported backend. Keep macOS app node behavior, Android/iOS node behavior, `system.run`, exec approvals, and plugin node-host registration semantics unchanged. Do not add a broad D-Bus or `dunstify` implementation unless it stays small and tested. Update CLI text and public docs so Linux headless support and headless fallback behavior are accurate. Add a user-facing changelog fix entry crediting the reporter, e.g. `Fixes #14079. Thanks @holistic-medical-supply-base.` Also preserve the improved unsupported-command error distinction from related #24616.
