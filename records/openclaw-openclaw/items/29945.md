## Summary

Current main still has the reported behavior: unpaired DM pairing creates a branded `OpenClaw: access not configured.` response, sends it when a new pairing request is created, and has no `unpairedResponse` config surface. The linked implementation PR #39065 remains open and unmerged, so this issue should stay open for maintainer review rather than be closed by cleanup.

## Best Possible Solution

Keep this issue open and review, land, or deliberately supersede #39065. The accepted fix should add a documented `unpairedResponse` policy, wire `silent`, `code-only`, and `branded` through shared pairing helpers and channel/plugin call sites, update config schema/metadata/docs, add regression tests for the modes, decide the default explicitly, and audit adjacent pairing logs/error paths for raw sender identifiers.
