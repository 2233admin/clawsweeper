## Summary

Keep this issue open. Current main still emits branded pairing replies for newly created unpaired DM requests, the requested `unpairedResponse` config surface is absent, docs/tests still describe or assert pairing-code replies, and open PR #39065 uses closing syntax for this exact work, so the issue should stay open until that PR lands or is explicitly superseded.

## What This Changes

The issue requests configurable unpaired DM response behavior, especially a silent mode/default, so unknown senders do not receive a branded OpenClaw pairing reply that fingerprints the platform.

## Best Possible Solution

Keep this issue open and review, land, or explicitly supersede PR #39065. The accepted fix should add a documented `unpairedResponse` policy, wire `silent`, `code-only`, and `branded` through shared pairing helpers and all DM pairing call sites, update config schema/metadata/docs, add regression coverage for all modes, choose the default deliberately, and audit nearby pairing logs/error paths for raw sender identifiers.
