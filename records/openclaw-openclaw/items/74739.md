## Summary

Keep open. Current main still performs package-update restart health/version verification in the parent update process after the updated install is used for the restart itself, while this PR moves that verification into the freshly installed CLI and adds targeted coverage. I did not find a discrete blocking review finding in the proposed diff; this is active implementation work that needs maintainer review and normal gate validation rather than cleanup closure.

## What This Changes

The PR changes the update CLI so package-mode restart verification is delegated to a hidden updated-install `openclaw update` mode, preserves gateway auth environment for that child process, and updates restart-verification tests.

## Best Possible Solution

Land this implementation or an equivalent maintainer-approved variant once CI/Testbox validation confirms package updates verify the restarted gateway from the updated install while retaining stale-PID recovery, expected-version checks, and the non-package fallback.

## Reproduction Assessment

Yes. The source-level reproduction is clear: current main's package-update test still expects parent-process `probeGateway` verification after updated-install restart, and the PR adds focused tests showing the updated-install verifier path instead.

## Solution Assessment

Yes. Delegating only restart health/version verification to the updated install is a narrow maintainable fix that matches the existing fresh-process update pattern without broadening core update behavior.
