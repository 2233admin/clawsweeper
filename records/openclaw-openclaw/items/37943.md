## Summary

Keep open. Current main still matches the reported failure shape: the macOS node service launches `openclaw node run`, that command is not in the invalid-config allowlist and does not bypass the config guard, and the node runner still reads strict runtime config before connecting. Gateway startup has last-known-good/degraded recovery that the node path does not share, so this needs a maintainer policy decision rather than cleanup closure.

## What This Changes

Request that the macOS headless node service accept or better diagnose an `openclaw.json` that is valid JSON and works for the Gateway instead of failing with `Config invalid` and gateway connection errors.

## Best Possible Solution

Keep this open for maintainer policy. Decide the intended `openclaw node run` startup contract, then either add a narrow node-specific best-effort or last-known-good config path for the TLS/auth/plugin-registry data it needs, or keep node startup strict but emit node-specific diagnostics and repair guidance. Add regression coverage for invalid effective config on node startup and update `docs/cli/node.md` if behavior changes.
