## Summary

Keep open. Current main still has the reported failure shape: the macOS node service launches `openclaw node run`, the invalid-config guard still does not allow `node`, and `runNodeHost()` still reads strict runtime config before connecting. Gateway startup has a separate best-effort/last-known-good recovery path that node startup does not share.

## Best Possible Solution

Keep this open and decide the intended `openclaw node run` startup contract. The likely product choice is either to add a narrow node-specific best-effort or last-known-good config path for TLS/auth/plugin registry needs, or to keep node startup strict but emit node-specific diagnostics and repair guidance. Any chosen fix should add regression coverage for invalid effective config on node startup and update `docs/cli/node.md` / the public Node docs if behavior changes.
