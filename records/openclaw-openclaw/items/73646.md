## Summary

Keep open. Current main still has the explicit Crestodian no-TTY branch logging the error and returning normally, so the wrapper treats the command as successful. The provided discussion also shows two open, unmerged PRs with closing references for the same bug, so the issue should remain open until one is reviewed and landed.

## What This Changes

Make `openclaw crestodian` exit non-zero when invoked without `--message` in a non-TTY context instead of logging the interactive-TTY error and returning success.

## Best Possible Solution

Review and land one focused fix that makes the explicit `runCrestodian()` no-TTY guard fail with exit code 1, adds regression coverage for stdin-not-TTY and stdout-not-TTY, and preserves `--message` plus `--json` headless behavior.

## Reproduction Assessment

Yes. The current-main source gives a high-confidence reproduction path: invoke `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and it logs the error then returns normally. The reporter also provided deterministic shell reproductions against v2026.4.27.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the existing Crestodian no-TTY guard fail through the CLI's normal non-zero path, with unit coverage around that branch; a broader shared TTY helper can be handled separately from this issue and #73562.
