## Summary

Keep open. Current main still lets the explicit Crestodian no-TTY branch log the error and return normally, while two open closing PRs already target the same defect, so this should wait for maintainer review and one merged fix.

## What This Changes

Make `openclaw crestodian` return non-zero when invoked without an interactive TTY and no `--message` or `--json` shortcut is used.

## Best Possible Solution

Land one focused Crestodian fix that makes the explicit no-TTY guard fail through the CLI's normal non-zero path and adds regression coverage for stdin-not-TTY and stdout-not-TTY, while preserving `--message` and `--json` headless behavior.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: call `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and current main logs the TTY error then returns normally.

## Solution Assessment

Yes. The narrowest maintainable fix is to change the existing Crestodian guard to set a non-zero exit or throw through `runCommandWithRuntime()`, with targeted tests; a broader shared prompt/TTY helper can be handled separately with #73562 if maintainers choose that cleanup.
