## Summary

Keep this issue open. Current main still matches the reported bug: the explicit `openclaw crestodian` path logs the missing-TTY error and returns normally, so the command wrapper has no thrown error to convert into exit 1. The related bare-root `openclaw` path already sets a non-zero exit code, and `models auth login` throws on the same no-TTY class, so this is a real current inconsistency rather than a duplicate or implemented cleanup item.

## Best Possible Solution

Keep the issue open and fix the explicit Crestodian command path on main. The narrow fix is for `runCrestodian()` to throw or set a non-zero exit when no interactive TTY is available and no `--message` or `--json` path is active, then add a regression test for `openclaw crestodian < /dev/null` semantics. A broader follow-up can normalize related TTY/prompt handling with #73562, but this issue should not close before the explicit Crestodian path is fixed.
