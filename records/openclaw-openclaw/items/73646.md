## Summary

Keep open. Current main still has the Crestodian no-TTY branch logging the error and returning normally, so the explicit `openclaw crestodian` subcommand can still complete successfully in non-TTY contexts. The issue is also already paired with two open, unmerged closing PRs, so maintainers should review or choose one of those implementations rather than close the issue now.

## What This Changes

The issue requests that explicit `openclaw crestodian` fail non-zero when invoked without an interactive TTY and without `--message`, matching other TTY-required CLI commands.

## Best Possible Solution

Review and land one focused fix that makes the explicit `runCrestodian()` no-TTY guard fail with exit code 1, adds regression coverage for stdin-not-TTY and stdout-not-TTY, and preserves `--message` plus `--json` headless behavior.

## Reproduction Assessment

Yes. A high-confidence reproduction is `openclaw crestodian < /dev/null` on current main; source inspection shows the no-TTY path logs the error and returns normally through a wrapper that only exits non-zero on thrown errors.

## Solution Assessment

Yes. The narrowest maintainable solution is to make the existing Crestodian no-TTY guard set the CLI's non-zero failure path and add unit coverage around that branch; a broader shared TTY helper for related prompt issues can be handled separately.
