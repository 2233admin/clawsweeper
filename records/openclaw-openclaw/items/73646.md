## Summary

Keep this issue open. Current main still has the reported Crestodian no-TTY behavior: `runCrestodian()` logs the interactive-TTY error and returns normally, while the explicit `openclaw crestodian` command only exits non-zero when `runCommandWithRuntime()` catches an error. The linked implementation PRs are closed unmerged, so there is no merged or shipped fix to close against.

## Best Possible Solution

Keep this issue open and land a narrow CLI fix: make the interactive no-TTY branch in `runCrestodian()` propagate failure through the existing runtime wrapper, add focused regression coverage for the no-TTY path, and preserve `--message` and `--json` behavior for headless automation. The related `plugins uninstall` EOF issue can remain separate unless a shared prompt/TTY helper is deliberately introduced without broadening this fix.
