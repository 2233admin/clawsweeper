## Summary

Current main still has the reported behavior: the explicit Crestodian no-TTY branch logs the error and returns normally, while the CLI wrapper only exits non-zero when an error is thrown. The linked implementation PRs were closed unmerged, so there is no shipped or merged fix to close against.

## Best Possible Solution

Keep this issue open. Reopen, rebase, or replace the closed fix PR approach: make the explicit `runCrestodian()` no-TTY branch throw or otherwise propagate exit 1 through `runCommandWithRuntime`, add focused regression coverage for `openclaw crestodian < /dev/null`, and keep `--message` and `--json` working for headless automation.
