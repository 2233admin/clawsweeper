## Summary

Keep this issue open. Current main still matches the reported bug: explicit `openclaw crestodian` detects the missing interactive TTY, logs the error, and returns normally without setting a non-zero exit status. The bare `openclaw` Crestodian fast path already sets `process.exitCode = 1`, and sibling interactive commands throw on no-TTY, so the explicit Crestodian subcommand remains the outlier rather than being implemented or superseded.

## Best Possible Solution

Keep the issue open and fix the explicit Crestodian command path on main. The narrow fix is for `runCrestodian()` or `register.crestodian.ts` to throw or set exit code 1 when no interactive TTY is available and no non-interactive `--message`/`--json` path is being used, then add a regression test that mocks non-TTY stdin/stdout and asserts the error message plus non-zero exit. A broader follow-up can unify TTY-required command handling with #73562, but this issue should not close before the explicit `openclaw crestodian < /dev/null` path is fixed.
