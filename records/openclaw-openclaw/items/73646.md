## Summary

Keep open. Current main still has the explicit Crestodian no-TTY branch logging the error and returning normally, and two open unmerged PRs already target this exact bug, so the issue should remain open until maintainers choose and land a fix.

## What This Changes

The issue requests that `openclaw crestodian` return a non-zero exit code when invoked without an interactive TTY and without `--message`.

## Best Possible Solution

Review and land one focused fix that makes the explicit `runCrestodian()` no-TTY guard fail with exit code 1, adds regression coverage for stdin-not-TTY and stdout-not-TTY, and preserves the working `--message` and `--json` headless paths.

## Reproduction Assessment

Yes. The current-main source gives a high-confidence reproduction path: invoke `runCrestodian()` without `message` or `json` while stdin or stdout is not a TTY, and it logs the error then returns normally; the reporter also provided deterministic shell reproductions against v2026.4.27.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the existing Crestodian no-TTY guard fail through the CLI's normal non-zero path with unit coverage around that branch; a broader shared TTY helper can be handled separately from this issue and #73562.
