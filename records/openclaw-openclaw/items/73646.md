## Summary

Keep open. Current main still logs the Crestodian non-TTY error and returns normally from the explicit subcommand path, so the observable zero-exit bug remains; the issue is already paired with two open closing PRs, so it should stay open until maintainers land or reject one of them.

## What This Changes

Make `openclaw crestodian` fail with a non-zero exit code when invoked without an interactive TTY and without a `--message` request.

## Best Possible Solution

Land one focused implementation that makes the existing `runCrestodian()` no-TTY guard fail with exit code 1, adds regression coverage for stdin and stdout not being TTY, and preserves `--message` and `--json` headless behavior; broader TTY prompt normalization can be handled separately with #73562.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: invoke the explicit `crestodian` subcommand without `message` or `json` while stdin or stdout is not a TTY, and `runCrestodian()` logs the error then returns normally through a wrapper that only exits non-zero on thrown errors.

## Solution Assessment

Yes. The narrowest maintainable fix is to make the existing Crestodian no-TTY guard set a non-zero exit through the CLI's normal failure path and cover that with unit tests; a shared helper is useful later but not required to resolve this issue.
