## Summary

Current main still appears vulnerable to the reported profile override bug on macOS gateway lifecycle commands. The root `--profile` parser sets `OPENCLAW_PROFILE`, but it does not clear an inherited `OPENCLAW_LAUNCHD_LABEL`; launchd service resolution still prefers `OPENCLAW_LAUNCHD_LABEL` over `OPENCLAW_PROFILE`, and restart/stop/status paths pass `process.env` through to the service implementation. The closed PR #47594 attempted a related fix but was not merged, and the open related issue #65643 tracks the same stale-label root cause for status; this issue should stay open because the restart case remains the more destructive surface.

## Best Possible Solution

Keep this issue open and fix the explicit root `--profile` path so it wins over inherited gateway service context. The likely implementation is to have `applyCliProfileEnv` clear stale `OPENCLAW_LAUNCHD_LABEL` when a root `--profile` is explicitly parsed, then add focused tests proving profile-targeted status/restart/stop resolve the profile-derived LaunchAgent label while preserving intentional custom label behavior when no explicit profile is supplied. Coordinate with #65643 as the same root-cause cluster.

## Clownfish Work Prompt

Observable bug: `openclaw --profile rescue gateway restart` can still target the current session's LaunchAgent when `OPENCLAW_LAUNCHD_LABEL` is inherited, because `applyCliProfileEnv` sets `OPENCLAW_PROFILE` but leaves the stale label and launchd resolution prefers the label.
