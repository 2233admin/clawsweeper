## What This Changes

The PR adds a disabled web search/fetch fast path in secrets runtime that skips provider discovery when the surface is `enabled: false` without credential-bearing config, plus focused regression tests.

## Best Possible Solution

Land one canonical fix that makes explicitly disabled web surfaces avoid unnecessary provider discovery while preserving inactive SecretRef warnings for credential-bearing config; track any remaining enabled-tool manifest-scan stall separately if it is still observable.

## Reproduction Assessment

Yes. #74896 gives a concrete 2026.4.26 Linux/Docker startup trace and minimal config, and current main still has the code path that enters web-fetch provider discovery whenever the fetch object is present. I did not run the live Docker reproduction in this read-only review.

## Solution Assessment

Yes for the reported escape hatch. Short-circuiting before `resolveRuntimeWebProviderSurface` is the narrowest maintainable fix for `enabled: false`, and the second commit keeps discovery for credential-bearing values so inactive SecretRef warnings are not silently lost.
