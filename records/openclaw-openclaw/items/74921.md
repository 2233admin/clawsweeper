## Summary

Keep open for maintainer review. Current main still resolves the web-fetch provider surface whenever a `tools.web.fetch` object exists, while this PR adds a narrow disabled-surface fast path and regression coverage; I did not find a blocking correctness issue in the updated diff.

## What This Changes

The PR updates `src/secrets/runtime-web-tools.ts` and its tests so explicitly disabled web search/fetch surfaces without plugin-scoped config or credential-bearing values skip provider discovery.

## Best Possible Solution

Land one canonical fix that makes explicitly disabled web search/fetch surfaces avoid unnecessary provider discovery while preserving inactive SecretRef warnings for credential-bearing config; track any remaining enabled-tool manifest-scan stall separately if it is still reproducible.

## Reproduction Assessment

Yes. #74896 provides a concrete Linux/Docker startup trace and minimal config, and current main still has the code path that enters fetch provider discovery whenever the fetch object is present. I did not run the live Docker reproduction in this read-only review.

## Solution Assessment

Yes for the reported escape hatch. The updated PR avoids discovery before `resolveRuntimeWebProviderSurface` only for disabled, credential-free surfaces, while retaining discovery for plugin-scoped or credential-bearing config where inactive SecretRef warnings depend on provider metadata.
