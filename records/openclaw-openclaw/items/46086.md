## Summary

Current main still does not support `memorySearch.multimodal` with the bundled `local` embedding provider. The implementation validates multimodal memory against provider adapter capability, the local adapter is text-only, Gemini is the only adapter declaring multimodal support, and the docs/merged multimodal PR describe the shipped scope as Gemini-only. This should stay open for maintainer product/provider judgment rather than be auto-closed.

## Best Possible Solution

Keep this open for maintainer judgment. The clean product path is either to explicitly confirm/document that bundled local multimodal memory is unsupported, or add a real local multimodal embedding provider/adapter only when the dependency can accept structured non-text inputs, with tests for provider capability validation, indexing behavior, and docs updates.

## Risks / Open Questions

- Restoring `local` multimodal behavior is not a mechanical bug fix unless the local embedding runtime can embed non-text inputs; falling back to filename/label-only embeddings would not provide true cross-modal search.
- A fix must preserve the local-vs-remote privacy boundary. Multimodal indexing can upload binary files to a remote provider, so any provider fallback or automatic Gemini routing needs explicit config and docs.
- The report does not include the exact config, error output, or a minimal reproduction, and the stated version transition predates the merged Gemini-only multimodal PR, so maintainer clarification would help before implementation.
