## Summary

Keep open. Current main and the latest release still enforce a Gemini-only multimodal memory contract, while the bundled local provider remains text-only. The report is not stale enough for cleanup closure and the remaining path needs maintainer product/provider judgment rather than automatic closure.

## What This Changes

The issue reports a regression where `memorySearch.multimodal` now requires a Gemini-capable embedding provider instead of working with `provider: "local"` after upgrading from 2026.2.12 to 2026.2.13.

## Best Possible Solution

Keep one explicit memory embedding contract: either local embeddings stay text-only with config/docs/doctor guidance that rejects multimodal settings clearly, or local multimodal support is added through an adapter that declares `supportsMultimodalEmbeddings` and implements structured `embedBatchInputs` with dependency-backed tests and docs.

## Reproduction Assessment

Yes. A config with `memorySearch.provider: "local"` and active `memorySearch.multimodal.modalities` follows `resolveMemorySearchConfig` and throws because the local adapter has no multimodal capability declaration; the existing unsupported-provider test covers the same fail-closed path.

## Solution Assessment

Unclear. The current Gemini-only implementation is internally consistent and documented, but it does not satisfy the reporter’s local-provider expectation; maintainers need to choose unsupported-local guidance or real local multimodal embedding support.
