## Summary

Keep this PR open. Current main still suppresses direct-WebChat conversation identifiers but still emits `Sender (untrusted metadata)` whenever sender fields are present, so the PR’s intended behavior is not implemented on main.

## Best Possible Solution

Keep this PR open and review or land a focused `buildInboundUserContextPrefix` change that suppresses the sender block only when the resolved inbound channel is direct WebChat. Include this PR’s regression test or an equivalent `Provider`/`Surface` WebChat case with `openclaw-tui` / `gateway-client`, and keep the existing external-direct and group sender metadata behavior covered.
