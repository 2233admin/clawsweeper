## Summary

Keep this PR open. Current main still suppresses direct-webchat conversation identifiers but still emits a `Sender (untrusted metadata)` block whenever the TUI/webchat sender fields contain transport client identity, so the PR's intended behavior is not implemented on main.

## Best Possible Solution

Keep this PR open and review or land a focused `buildInboundUserContextPrefix` change that suppresses `Sender (untrusted metadata)` only for direct webchat/TUI sessions, while preserving existing sender metadata for external direct and non-direct chats. Include the PR's regression test or an equivalent direct-webchat transport-sender case, then run the targeted inbound-meta test and the appropriate changed gate before merge.
