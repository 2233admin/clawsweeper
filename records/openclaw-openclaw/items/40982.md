## Summary

Keep this issue open. Current main still has the reported 180s fresh-session minimum and 180s resumed-session maximum, still resolves CLI no-output watchdogs to a positive numeric timeout, and still passes that timeout into process supervision. The provided discussion also shows #41022 is still an open PR with closing syntax for this issue, so the maintainer path is to review, revise, land, or close that PR rather than close this issue early.

## What This Changes

Raise or remove the hard 3-minute CLI no-output watchdog cap for fresh and resumed CLI sessions, ideally allowing operators to tune or disable the cap while preserving the global timeout.

## Best Possible Solution

Review and land #41022 if the maintainer agrees with the constants-only scope, or replace it with a broader fix that updates shared defaults, resolver semantics, config schema/generated schema, docs, and regression tests while explicitly deciding whether 0 or null disables the no-output watchdog. Keep #71803 as the narrower follow-up if Monitor expected-idle behavior needs a separate state-aware fix.
