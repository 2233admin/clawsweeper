## Summary

Current main still has the reported 180-second resumed CLI no-output cap and no 0/null disable path, and the latest shipped release still contains the same defaults. This should stay open because open PR #41022 directly references this issue with closing syntax and is the active implementation candidate.

## What This Changes

The issue asks to raise or remove the hard 3-minute CLI no-output watchdog cap for long-running fresh and resumed CLI backend requests, ideally with a configurable or disableable cap.

## Best Possible Solution

Land or revise the linked implementation so shared CLI watchdog defaults no longer terminate legitimate long-running CLI turns at 180 seconds, define the supported disable/configuration semantics, preserve global timeout enforcement, and add regression coverage for fresh and resumed backend resolution.
