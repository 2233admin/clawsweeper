## Summary

Keep this issue open. Current main still has the reported 180s fresh minimum and resumed-session maximum, still resolves CLI no-output watchdogs to a positive numeric timeout, and still passes that timeout into CLI process supervision. The issue also has open implementation PR #41022 linked with closing syntax, so the right maintainer path is to review and land, revise, or close that PR rather than close this issue early.

## What This Changes

Raise or remove the hardcoded 3-minute CLI no-output watchdog cap for fresh/resumed CLI requests, with optional operator control to tune or disable the cap.

## Best Possible Solution

Keep this issue open and review #41022, or a maintainer-owned replacement, against current main. The fix should update the shared CLI watchdog defaults, resolver semantics, config schema/generated schema, and regression tests; explicitly decide whether 0 or null disables the no-output watchdog; keep global timeout enforcement intact; and treat #71803 as a related Monitor-aware follow-up if the generic cap change does not solve expected idle states.
