## Summary

Keep open. Current main still has the reported MCP channel bridge pending-map retention pattern, and the linked same-author closing PR remains open and unmerged, so this issue should stay open until that fix or an equivalent replacement lands.

## What This Changes

The issue asks to bound `OpenClawChannelBridge` pending Claude permission and approval maps with TTL/sweeping, close cleanup, and a cap so long-running `openclaw mcp serve` processes do not retain stale entries.

## Best Possible Solution

Land or replace #71648 with an equivalent MCP bridge fix that clears both pending maps on close, prevents post-close writes, expires stale Claude permission and approval entries, and either resolves the cap requirement in this issue or explicitly tracks it as a narrower follow-up without changing the public MCP protocol surface.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: create unique Claude permission requests without matching replies, or approval requested events without matching resolved frames, and the corresponding maps retain entries; `close()` also leaves both maps untouched.

## Solution Assessment

Yes for keeping this issue open. The narrow maintainable fix belongs in `src/mcp/channel-bridge.ts` with focused MCP bridge tests, and the linked open PR is the current implementation candidate rather than a reason to close early.
