## Summary

Keep open. Current main still has the gateway stop-timeout stale-store behavior, and this PR is relevant but incomplete because clearing the task map without guarding the old task lifecycle callbacks can let a timed-out task later overwrite replacement runtime state; the same-author linked bug #70024 is also still open.

## What This Changes

The PR changes `stopChannel` timeout handling to delete `store.aborts` and `store.tasks`, set `running: false`, record `lastStopAt`, and update the existing gateway channel restart test to expect a successful restart after timeout.

## Best Possible Solution

Revise the gateway channel manager so task-owned runtime writes, error writes, final stop writes, and restart bookkeeping are guarded by the current task identity, while still allowing stop-timeout recovery and a fresh restart. Add a regression where a timed-out task eventually settles after a replacement start and cannot clobber the replacement snapshot.
