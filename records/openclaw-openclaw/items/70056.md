## Summary

Keep open. Current main still has the stop-timeout stale-store bug this PR targets, and this PR is paired with open same-author bug #70024. The PR is relevant but incomplete as-is because clearing the task map on timeout allows a replacement task while old task lifecycle callbacks can still mutate runtime state.

## Best Possible Solution

Keep this PR open and ask for a revision that preserves the stop-timeout recovery goal while guarding task-owned status writes, error writes, final stop writes, and restart bookkeeping by current task identity. Add a regression test where a timed-out task eventually settles after a replacement start and does not clobber the replacement runtime state. Leave #70024 open until a revised implementation merges.
