## Summary

Keep open. Current main still has the ACPX/ACP SDK validation boundary that can reject the reported nullable `usage_update.used` payload before ACPX's lenient runtime projection can handle it, and I did not find a merged or open narrow fix for this specific failure.

## What This Changes

The issue asks OpenClaw to accept or safely ignore ACP `session/update` `usage_update` notifications when fields such as `used` are `null` instead of rejecting the whole notification.

## Best Possible Solution

Align the ACPX runtime boundary with the ACP SDK contract: either normalize or drop nullable `usage_update` fields before SDK validation, or consume an upstream ACPX/SDK release that does so, then add regression coverage for a `session/update` payload with `used: null` on the bundled ACPX runtime path.
