## Summary

Keep this issue open. Current main still lacks an exact `openclaw cron list --agent <id>` filter, and the Gateway/agent-tool `cron.list` path still returns the global cron list rather than auto-scoping to the requesting agent. The existing substring `query` support includes `job.agentId`, but that is not equivalent to multi-agent isolation or the requested default-agent handling for jobs with no explicit `agentId`.

## Best Possible Solution

Implement this in core: add an exact `agentId` filter to the cron list protocol/types and `CronListPageOptions`, wire `openclaw cron list --agent <id>` through the CLI, filter in `src/cron/service/ops.ts`, and have the agent `cron` tool default list calls to the requesting session's agent unless an explicit global/admin scope is requested. Add regression tests for exact matching, unset-agent/default-agent visibility, CLI forwarding, protocol validation, and agent-tool auto-scoping, then update the cron CLI docs.
