## Summary

Keep this PR open. Current main does not implement the intended `--expect-final` post-hook assistant payload behavior: payload construction still feeds `attempt.assistantTexts` and `attempt.lastAssistant` directly into `buildEmbeddedRunPayloads`, while `before_message_write` can rewrite or block persisted messages before the snapshot is taken. The PR is a plausible focused fix, but its open review comment about falling back to a prior assistant when the current user write is blocked should be resolved before merge.

## Best Possible Solution

Keep the PR open and ask for a small revision or maintainer review: derive the final payload from the post-hook current-turn assistant without reusing stale raw `assistantTexts`, and cover replaced assistant, blocked assistant, and blocked-user-with-history cases in focused runner/helper tests. After that, the PR is likely the right narrow owner-module fix for the pi embedded runner payload path.
