## Summary

Keep this issue open. Current main already covers much of the reported compaction/tool_result pairing root cause, but the remaining user-visible Anthropic validation error path is still actionable: JSON invalid_request_error payloads can still format as `LLM request rejected: <provider message>`, and `unexpected tool_use_id` is not classified as replay-invalid or rewritten into a session recovery message.

## Best Possible Solution

Keep this issue open and make a narrow fix in the provider error/replay path: classify Anthropic `unexpected tool_use_id` / orphaned `tool_result` validation failures as session-history replay errors, return a friendly recovery message instead of `LLM request rejected: ...`, and, only where the existing runner can prove replay is safe, re-sanitize history and retry once. Preserve the existing pairing repair and add focused regression coverage around formatter output and channel payload delivery.
