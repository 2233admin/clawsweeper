## Summary

Keep this issue open. Current main has strong tool_use/tool_result pairing repair in replay, limiting, and compaction paths, so the broad compaction root cause from 2026.3.13 looks mostly covered. The reported user-visible failure is still actionable: JSON-wrapped Anthropic invalid_request_error payloads can still format as `LLM request rejected: <provider message>`, with no specific classifier, retry, or regression coverage for `unexpected tool_use_id`.

## Best Possible Solution

Keep this issue open. Preserve the existing pairing repair, then add a narrow Anthropic `unexpected tool_use_id` / orphaned `tool_result` classifier in the provider error path. When the attempt is safe to replay, re-sanitize replay history and retry once; otherwise return a friendly session-history recovery message. Add focused coverage for `formatAssistantErrorText` and payload delivery, plus embedded runner coverage if retry behavior is added.
