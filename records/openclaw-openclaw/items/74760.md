## Summary

Keep open. Current main still matches the reported group-chat no-visible-reply signature after the v2026.4.27 private-by-default group/channel reply change, and the remaining question is whether explicit group mentions should use the documented config restore path or get a new cross-channel visible-reply exception. The Mac vs Windows question appears answerable from code: the relevant resolver is driven by chat context/config/tool availability, not an OS branch.

## What This Changes

The issue reports that after updating to v2026.4.27, mentioning a bot in a POPO group chat dispatches to the agent but produces no visible reply while P2P still works.

## Best Possible Solution

The desired end state is a clear maintainer policy for explicit group mentions: either keep the shipped private-by-default behavior with support guidance to set messages.groupChat.visibleReplies to "automatic", or add a narrow cross-channel explicit-mention rule in the source-reply delivery resolver with focused tests and docs.

## Reproduction Assessment

Yes for the shared core behavior: current regression coverage reproduces a group/channel turn resolving to message_tool_only with queuedFinal=false and no sendFinalReply. Live POPO reproduction is unclear because that channel is external to this repo.

## Solution Assessment

Unclear. The documented config restore path is the narrowest solution if private-by-default group replies are intended; if explicit mentions should always reply visibly, the safer fix is a narrow cross-channel policy change rather than a POPO-specific core special case.
