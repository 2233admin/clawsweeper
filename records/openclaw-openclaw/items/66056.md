## Summary

Keep open. Current main still hardcodes user-facing runtime failure text in English, `sanitizeUserFacingText` has no locale-aware API or message catalog, and the `messages` config/schema surface has no locale field. Control UI localization remains browser-local UI text, while plugin hooks can rewrite outbound strings but do not expose structured error keys or locale context, so closing as implemented or plugin-only would be premature.

## What This Changes

The issue requests locale-aware user-facing runtime error messages with English fallback and a configurable locale surface.

## Best Possible Solution

Add a small stable user-facing error-key layer with English fallback, document a locale configuration or resolution policy, wire it through sanitizer and reply-runner failure paths, and cover fallback plus at least one non-English locale in focused tests. If maintainers prefer plugin-owned localization, first expose structured error keys and locale context so plugins translate stable identifiers instead of matching English literals.
