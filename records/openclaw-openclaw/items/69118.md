## Summary

Keep open. Current main still includes first-turn-only `groupIntro` in the static extra system prompt text used for CLI session reuse hashing, while `resolveCliSessionReuse` still resets on any prompt-hash mismatch; the related merged wake-up fix #74171 explicitly did not address this group-intro drift.

## What This Changes

The issue requests a Claude CLI session-continuity fix so group-channel follow-up turns do not reset when `groupIntro` appears on turn 1 but not later turns.

## Best Possible Solution

Keep the actual first-turn/re-intro group instructions in the system prompt, but remove turn-scoped `groupIntro` from the stable CLI prompt identity used for session reuse. Add a regression test that builds two consecutive group-channel turns and proves the CLI binding is reused without `reason=system-prompt`.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a first group turn includes `groupIntro` in `extraSystemPromptStatic`, a follow-up group turn omits it, and the resolver treats the resulting hash difference as a system-prompt reset.

## Solution Assessment

Yes, a narrow fix at the stable prompt identity boundary is the best first solution. Dropping all prompt/MCP reuse keys would be a broader policy change, while excluding first-turn-only group intro preserves current invalidation behavior for genuinely durable prompt changes.
