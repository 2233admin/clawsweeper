## Summary

Keep this issue open. Current main still does not forward iMessage tapback reactions to the agent: the imsg RPC watcher only handles ordinary `message` notifications, subscribes only with attachment options, parses only `raw.message` payloads, and the inbound dispatch/context path is text-message oriented. The issue is also paired with same-author open PR #39322, with related filtering work in #60677, so maintainer review should converge those rather than closing this issue.

## Best Possible Solution

Keep this issue open as the canonical legacy iMessage tapback-to-agent request. Review, rebase, or supersede #39322 with an implementation that subscribes to reaction notifications, parses reaction target/action metadata, routes tapbacks through a deliberate agent-facing event or accepted summary format, defines access-control and dedupe behavior, and updates config, docs, and focused iMessage tests. Coordinate with #60677 so tapback filtering and opt-in forwarding do not conflict.
