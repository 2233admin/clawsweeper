## Summary

Current main still lacks the safety guarantees requested here. Startup catch-up still selects overdue cron work from stored due-time or missed-slot state, deletion and cron stop do not own a service-level active-run abort registry, and gateway shutdown stops schedulers/channels rather than providing an emergency outbound halt. The linked maintenance-window PR remains open, and the merged gateway startup race fix is adjacent rather than a fix for this behavior.

## What This Changes

The issue requests that cron-scheduled WhatsApp deliveries respect configured daytime windows after reconnect/restart and that stop/delete commands immediately halt active outbound sends.

## Best Possible Solution

Keep this open and fix it in cron/gateway core with explicit missed-run policy, active execution cancellation, and operator emergency controls. The end state should prevent reconnect/startup replay from violating contact windows, make cron remove/disable and gateway stop abort matching active runs where possible, expose a clear emergency outbound halt, and add regression tests for an America/Guatemala out-of-hours reconnect plus active delivery cancellation.
