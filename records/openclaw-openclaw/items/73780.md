## Summary

Keep open. Current main still leaves the Telegram-visible part unresolved: provider/agent rate-limit payloads marked `isError` are delivered as Telegram replies, while Telegram `errorPolicy` is only applied after Telegram delivery failures, not before successfully sending provider-error text.

## What This Changes

The issue asks OpenClaw to stop repeatedly sending API rate-limit error replies through Telegram when an upstream provider is rate-limiting or blocking requests.

## Best Possible Solution

Add a Telegram pre-send policy for provider/agent `isError` payloads: classify rate-limit and overload text before `deliverReplies`, apply account/group/topic `errorPolicy` with per account/chat/thread cooldown, preserve normal successful replies, and align Telegram docs/tests with the actual config contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to dispatch repeated Telegram final `ReplyPayload` objects with `isError: true` and rate-limit text, or use the existing runner path that injects `⚠️ API rate limit reached`; current main will still pass those payloads to `deliverReplies`.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram delivery boundary because provider fallback/cooldown already addresses upstream retry behavior, while the remaining user-visible bug is repeated Telegram delivery of successfully sent error replies.
