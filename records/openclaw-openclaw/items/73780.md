## Summary

Keep open. The reporter's Z.ai account block is external, but current main still has an OpenClaw-visible Telegram behavior: provider rate-limit error payloads marked `isError` are delivered as normal Telegram replies, while Telegram `errorPolicy` is only applied to Telegram delivery failures after a send attempt fails.

## What This Changes

The issue reports repeated provider API rate-limit errors being sent into Telegram and asks OpenClaw to avoid spamming the chat during those failures.

## Best Possible Solution

Add Telegram pre-send handling for provider/agent error payloads: classify rate-limit and overload `ReplyPayload.isError` replies before Telegram delivery, apply the existing account/group/topic `errorPolicy` and per account/chat/thread cooldown, keep successful replies unchanged, and update docs/tests to match the actual config contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to deliver repeated Telegram final payloads with `isError: true` and rate-limit text; current main forwards them through `sendPayload` and `deliverReplies` rather than suppressing or coalescing them before Telegram delivery.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram agent-delivery boundary, because provider fallback/cooldown work is already separate and the remaining observable bug is repeated Telegram delivery of successful provider error replies.
