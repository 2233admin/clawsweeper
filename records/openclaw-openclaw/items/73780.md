## Summary

Keep open. Current main still sends Telegram `isError` provider/agent error payloads as normal outbound replies, while Telegram `errorPolicy` is only applied in the delivery-failure callback after Telegram rejects a send. The reporter's Z.ai high-concurrency block is external, but the OpenClaw-visible repeated Telegram error-message behavior remains a narrow valid bug.

## What This Changes

The issue reports repeated Telegram-visible API rate-limit/provider error messages and asks for OpenClaw to stop spamming the chat during that failure mode.

## Best Possible Solution

Add Telegram pre-send handling for provider/agent error payloads: classify `isError` rate-limit/overload replies before delivery, apply the existing account/group/topic/thread `errorPolicy` and `errorCooldownMs` scope, keep successful replies unchanged, and update docs/tests to match the actual `always`/`once`/`silent` contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to deliver repeated Telegram `ReplyPayload` values with `isError: true` and rate-limit text under `errorPolicy: "once"` or `"silent"`; current main still forwards them to `deliverReplies` instead of suppressing/coalescing them before Telegram delivery.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram agent-delivery boundary, because provider fallback/cooldown already handles upstream retry storms while the remaining OpenClaw bug is repeated Telegram delivery of successful error replies.
