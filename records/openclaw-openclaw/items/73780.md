## Summary

Keep open. Current main still leaves the Telegram-visible part unresolved: provider/agent error payloads marked `isError` are still sent as normal Telegram replies, while Telegram `errorPolicy` suppression only runs after Telegram delivery failures rather than before successfully posting provider rate-limit text.

## What This Changes

The issue requests that repeated Telegram API rate-limit errors be reduced or suppressed instead of repeatedly posting rate-limit messages to the chat.

## Best Possible Solution

Add a Telegram pre-send policy for provider or agent error replies: classify rate-limit/overload `isError` payloads before delivery, apply the configured account/group/topic policy with a per account/chat/thread cooldown, preserve ordinary successful replies, and update focused Telegram tests and docs to match the real config contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to dispatch repeated Telegram `ReplyPayload` values with `isError: true` and rate-limit text, or exercise the mid-turn provider 429 runner path; current main still reaches Telegram delivery instead of coalescing those payloads.

## Solution Assessment

Yes. The narrowest maintainable solution is at the Telegram delivery boundary, because provider cooldown/fallback logic mitigates upstream retry storms while the remaining user-visible defect is repeated Telegram posting of successful error replies.
