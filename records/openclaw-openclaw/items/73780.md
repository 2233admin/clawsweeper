## Summary

Keep open. Current main still treats provider rate-limit failures as visible Telegram error replies; the Telegram error policy only governs Telegram delivery failures, not successfully delivered provider-error payloads.

## Best Possible Solution

Handle this in the Telegram/agent delivery boundary: classify provider rate-limit error payloads before sendPayload, coalesce repeated identical errors per account/chat/thread using the configured cooldown, keep a first visible error with useful retry/cooldown text, and align Telegram errorPolicy docs/schema/runtime values so operators can tune the behavior predictably.
