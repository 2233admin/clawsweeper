## Summary

Keep open. The upstream Z.ai account block described by the reporter is external, but current main still has an OpenClaw Telegram behavior to fix: provider rate-limit/overload payloads marked `isError` are still sent as normal Telegram replies, while the existing Telegram `errorPolicy` only gates errors after Telegram delivery itself fails.

## What This Changes

The issue requests reducing repeated Telegram-visible API rate-limit errors when an upstream model provider returns rate-limit or high-concurrency failures.

## Best Possible Solution

Add Telegram pre-send handling for provider/agent rate-limit and overload error payloads: classify those `isError` replies before `deliverReplies`, apply the existing account/group/topic `errorPolicy` plus account/chat/thread cooldown, keep successful replies and normal tool errors unchanged, and align docs/tests with the actual config contract.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to deliver repeated Telegram final `ReplyPayload` values with `isError: true` and rate-limit text; current main forwards them through `sendPayload` and `deliverReplies` instead of suppressing or coalescing them before Telegram delivery.

## Solution Assessment

Yes. The narrowest maintainable fix is at the Telegram agent-delivery boundary, because provider fallback/cooldown behavior is already separate and the remaining observable bug is repeated Telegram delivery of successful provider error replies.
