## Summary

Keep this issue open. Related provider-side backoff work exists on current main, but the remaining Telegram-visible problem is not fully solved: provider rate-limit errors still become visible Telegram reply payloads, and the Telegram error policy path is only applied after delivery failures, not before sending successful error replies.

## Best Possible Solution

Keep this open and handle it in the Telegram/agent delivery boundary: classify provider rate-limit error payloads, coalesce or rate-limit repeated identical error replies per account/chat/thread, optionally queue or defer retries based on provider cooldown/Retry-After data, and align the Telegram errorPolicy docs/schema/runtime so operators can tune visible error spam predictably.
