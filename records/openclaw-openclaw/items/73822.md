## Summary

Keep open. Current main still lacks the requested SecretRef support for these phone-number and sender-identity fields, and this PR is the active implementation for #56502, but the proposed patch still has a security-sensitive open-policy guard regression that should block merge until fixed on the existing PR branch.

## What This Changes

The PR adds SecretRef-capable phone-number and handle config fields, secret target collection, UI sensitivity hints, docs, changelog, generated metadata, and tests across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin, plus an unrelated Telegram test-helper cleanup.

## Best Possible Solution

Keep the SecretRef support for phone-number and handle fields, but preserve the existing visible wildcard contract for dmPolicy="open": SecretRefs can count as configured allowlist entries, while only a literal resolved "*" should opt a channel into public-open DM behavior.

## Reproduction Assessment

Yes. On the PR head, a focused schema/helper test using dmPolicy="open" with allowFrom containing only an object SecretRef would be accepted by requireOpenAllowFrom even though current main requires a literal "*".

## Solution Assessment

No. The feature direction is useful, but the proposed open-policy refinement is too broad; the safer narrow fix is to allow SecretRefs for allowlist-style phone fields while requiring a visible wildcard for open policy.
