## Summary

Keep open. The PR addresses a real remaining SecretRef gap for phone-number and handle config fields, and the earlier WhatsApp test-helper concern is addressed on the latest head, but the current patch still lets any SecretRef satisfy the explicit public-open DM wildcard guard. That is a security-sensitive behavior change from current main and should block merge until revised.

## What This Changes

The PR adds SecretRef support for phone-number and sender-identity fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin, with schema, secret-contract, manifest, UI hint, docs, changelog, and test updates.

## Best Possible Solution

Keep the SecretRef feature on this PR branch, but preserve the visible public-open contract: SecretRefs may count as configured entries for allowlist policy, while dmPolicy="open" should still require a literal normalized "*" in reviewed config. Update the new SecretRef allowFrom test to reject SecretRef-only open policy and then rerun the focused config/channel secret tests before the changed gate.

## Reproduction Assessment

Yes. On latest PR head 62d18ae17386ef007d62a656a4b3a3604f621ede, source inspection shows requireOpenAllowFrom returns when any SecretRef is present, and the new test expects a SecretRef-only open policy to pass.

## Solution Assessment

No. The feature direction is useful and mostly well-scoped, but accepting a hidden SecretRef as the public-open wildcard is not the safest implementation; the narrow fix is to keep SecretRefs valid for allowlists while requiring a visible literal wildcard for open policy.
