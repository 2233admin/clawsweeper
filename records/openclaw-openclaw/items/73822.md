## Summary

Keep open. Current main still does not support SecretRef-backed phone-number fields, and this PR is the active implementation for the linked feature request, but the latest head still weakens the explicit wildcard guard for open DM policy by accepting any object SecretRef as the opt-in.

## What This Changes

The branch adds SecretRef-capable phone-number and handle config fields, secret target collection, UI sensitivity hints, docs, changelog, generated metadata, and tests across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin.

## Best Possible Solution

Keep the SecretRef support for allowlist-style phone-number and handle fields, but preserve the explicit visible wildcard contract for dmPolicy="open" before landing the implementation for #56502.

## Reproduction Assessment

Yes. On the PR head, requireOpenAllowFrom accepts allowFrom containing only an object SecretRef for dmPolicy="open"; the focused reproduction is a schema/helper test using `{ dmPolicy: "open", allowFrom: [envRef("PHONE")] }`, which should remain invalid unless a literal or resolved `*` is present.

## Solution Assessment

No. The PR is directionally the right feature work, but accepting any SecretRef as the open-policy opt-in is not the narrowest safe solution; allow SecretRefs for phone allowlists while requiring a literal wildcard for open policy.
