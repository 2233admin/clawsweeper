## Summary

Keep open. The feature still fills a real gap on current main, but this PR should not merge as-is because it lets any SecretRef satisfy the explicit `dmPolicy="open"` wildcard opt-in, moving public-open access control out of visible reviewed config.

## What This Changes

The PR adds SecretRef support for phone-number and handle config fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin, including schemas, runtime secret collection, manifest metadata, docs, UI hints, generated metadata, changelog, and tests.

## Best Possible Solution

Keep the SecretRef feature direction, but preserve the access-control contract: SecretRefs may count as configured allowlist entries for `allowlist` policy, while `dmPolicy="open"` should require a visible literal `"*"` in config.

## Reproduction Assessment

Yes. On PR head `b6de50391d48664d9a974185120b1251353ebcd0`, the diff makes `requireOpenAllowFrom` accept any SecretRef and adds a unit test for SecretRef-only open policy.

## Solution Assessment

No. The broad feature is useful, but this implementation weakens the visible open-policy guard; the safer narrow fix is to keep SecretRefs valid for phone-number allowlists while requiring a literal wildcard for public-open DM policy.
