## Summary

Keep open: current main still lacks the requested SecretRef support for these phone-number and sender fields, and this PR is the active implementation for #56502, but the latest diff still weakens the shared explicit-open DM guard by treating any SecretRef as the required wildcard.

## What This Changes

The PR adds SecretRef support for phone-number and sender identity config fields across WhatsApp, Signal, iMessage, BlueBubbles, and the voice-call plugin, including schemas, runtime secret collection, UI hints, docs, generated metadata, changelog, and tests.

## Best Possible Solution

Keep the SecretRef support for phone-number and handle fields, but preserve the explicit-open safety contract: SecretRefs can count as configured entries for allowlist policy, while dmPolicy="open" should still require a literal "*" in the effective config or another equally explicit non-secret opt-in. Add a negative test for SecretRef-only open policy before running the targeted schema/secret-contract tests and changed gate.

## Reproduction Assessment

Yes. The diff-level reproduction is to call requireOpenAllowFrom with policy="open" and allowFrom containing only an env SecretRef object; the PR's added test expects zero issues, while current main and docs require a literal "*" for open policy.

## Solution Assessment

No. The overall direction is useful and compatible, but accepting any SecretRef in the shared open-policy refinement is not the narrowest maintainable solution; require a visible wildcard for open and keep the SecretRef escape hatch only for non-empty allowlist validation.
