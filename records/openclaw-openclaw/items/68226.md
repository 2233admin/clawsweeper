## Summary

Keep open. Current main still builds the shared status footer from only the session key and last-updated/no-activity text, while the session start timestamp is already stored or recoverable. The adjacent timing items do not supersede this narrower footer request.

## What This Changes

Add elapsed session duration or a requester-approved absolute session start timestamp to the shared status footer used by `/status` and `session_status`.

## Best Possible Solution

Implement the start visibility in the shared status renderer, using the durable `sessionStartedAt` value first and the existing lifecycle fallback when needed, so both `/status` and `session_status` show the same footer segment before the existing updated/no-activity text.

Implement the session start visibility requested in this issue. In the shared status renderer, add a footer segment before the existing `updated ...` / `no activity` text that shows either `started <absolute timestamp>` using the requester-approved format or a compact `duration <elapsed>` if maintainers prefer duration. Resolve the timestamp from `sessionEntry.sessionStartedAt` first and fall back through `resolveSessionLifecycleTimestamps()` so legacy transcript headers work. Keep the change shared so both chat `/status` and the `session_status` tool benefit. Stay scoped: do not solve broader response-time observability from #57442 or machine-readable current-time output from #51683 beyond any tiny shared helper needed for this footer.
