## Summary

Keep open. Current main still drops the requested Feishu card-action fields before dispatch, and this PR's visible diff handles them only in the handler after the live parser has already rebuilt `action` with `value` and `tag` only. The linked same-author issue remains open and overlapping Feishu PRs show this needs consolidation rather than closure.

## What This Changes

The PR extends Feishu card-action handling and handler tests so `form_value`, `input_value`, `name`, `option`, and `options` can be forwarded as structured synthetic message content when those fields are present.

## Best Possible Solution

Consolidate the Feishu fix so raw callback parsing, the `FeishuCardActionEvent` type, decoder/fallback behavior, and handler dispatch all preserve `form_value`, `input_value`, `name`, `option`, and `options`, with parser-to-handler tests that also lock in existing button text/command behavior.
