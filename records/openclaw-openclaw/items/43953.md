## Summary

Keep open. The PR addresses a real Feishu card-action data-loss problem, but the current patch is incomplete because live `card.action.trigger` callbacks are normalized before `handleFeishuCardAction`; current main still rebuilds `action` with only `value` and `tag`, so the new handler branch would not see real `form_value`, `input_value`, `option`, or `options` fields.

## What This Changes

The PR adds optional Feishu card-action fields and direct handler tests, serializing form/input/select callback fields into a structured JSON synthetic text message when those fields are present.

## Best Possible Solution

Implement one Feishu-owned card-action callback contract end to end: preserve `form_value`, `input_value`, `name`, `option`, and `options` in the raw parser, expose them through the decoder/event surfaces, serialize them in one backward-compatible format, and add parser-to-handler regression tests for real `card.action.trigger` payloads plus existing plain button commands.

## Reproduction Assessment

Yes. A current-main source check reproduces the issue path: a raw callback with `action.form_value` reaches `parseFeishuCardActionEventPayload`, which rebuilds `action` with only `value` and `tag`, so downstream handler logic cannot see the field.

## Solution Assessment

No. The PR's direct-handler branch is not the best complete fix because live callbacks are normalized before the handler; the safer fix is to preserve the fields at the parser/decoder boundary and then choose one handler serialization contract.
