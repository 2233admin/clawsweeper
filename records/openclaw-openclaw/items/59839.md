## Summary

Keep this issue open because current main still binds the visible Control UI/WebChat New session action to the reset-backed `/new` flow, which replaces the active session entry and archives the prior transcript instead of creating a separate selectable chat. The recent issue discussion also confirms the user-visible data-loss impression remains, and the focused fix is not merged yet.

## What This Changes

The issue asks for Control UI/WebChat New session to start a fresh chat while keeping previous chats visible and switchable from the UI.

## Best Possible Solution

Land #73725 or an equivalent focused change that makes the visible WebChat New session/New Chat affordance call `sessions.create`, switch to the returned dashboard session key, refresh session options, and keep `/new` or `/reset` as explicit reset behavior. Reset archives should either remain clearly reset-only or get a first-class recovery/listing path if users are expected to reopen them.
