## Summary

Keep this issue open. Current main still preserves media only after the extractor has found a media path; it does not implement the reported fallback from trusted read-image content to the original tool-call args when `details.path` is absent. Two open PRs still track that exact fix direction, including one from the issue author.

## Best Possible Solution

Keep this issue open and review the linked implementation PRs. The fix belongs in the embedded tool-result media extraction path: pass stored `tool_execution_start` args into extraction, fall back to `path` / `file_path` / `filePath` only when trusted image content exists and no explicit media path is present, then preserve the resolved media through the existing pending-tool-media merge and Feishu delivery path. Add focused extractor and handler regression tests before closing after the canonical PR lands.
