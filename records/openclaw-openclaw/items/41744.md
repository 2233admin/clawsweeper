## Summary

Keep this issue open. Current main still lacks the reported fallback from trusted read-image tool output to the original tool-call args when `details.path` is absent, and the provided GitHub context shows open PRs with closing syntax that are implementation candidates for the same bug shape.

## Best Possible Solution

Keep this issue open and review the linked implementation PRs. The fix belongs in the embedded tool-result media extraction path: pass stored `tool_execution_start` args into extraction, fall back to `path` / `file_path` / `filePath` only when trusted image content exists and no explicit media path is present, then preserve the resolved media through the existing pending-tool-media merge and Feishu delivery path with focused regression tests.
