## Summary

Keep open. Current main still has the hardcoded Synology Chat webhook-layer 120s race that can reject the agent delivery path before a longer configured `agents.defaults.timeoutSeconds` budget completes.

## Best Possible Solution

Keep the issue open and fix the Synology Chat plugin path: remove the hardcoded outer 120s race or derive it from the same configured agent runtime budget, passing config/runtime timeout into the webhook handler as needed. Add focused Synology Chat webhook coverage with `agents.defaults.timeoutSeconds` above 120 seconds so this channel-level cap cannot regress.

## Risks / Open Questions

- Closing this now would leave Synology Chat inbound replies able to fail at 120 seconds even when the operator configured a longer agent runtime timeout.
- A fix should avoid introducing an unbounded webhook worker leak; either remove the duplicate outer race in favor of the existing agent runtime timeout or derive any channel-level watchdog from the same configured budget.
