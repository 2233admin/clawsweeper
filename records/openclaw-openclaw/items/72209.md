## Summary

Keep open. Current main still lacks the passive checkpoint and prompt-recovery layer, while the PR remains a useful but not merge-ready implementation candidate because the new recovery-state path has concrete redaction and hot-path I/O issues.

## What This Changes

The PR adds agent session-recovery prompt guidance, local Recovery Bundle/Task Ledger persistence, and passive checkpoint writes from update_plan, sessions_yield, and sessions_spawn.

## Best Possible Solution

Land a rebased, maintainer-reviewed passive recovery layer that writes minimal recovery metadata asynchronously with secure local permissions, redacts secrets without corrupting normal summaries, treats recovered context only as evidence, requires fresh user confirmation and approvals, and has focused tests for prompt composition, persisted state, and all three checkpointing tools.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Tighten recovery redaction before persisting summaries:** `src/agents/session-recovery-state.ts:115`
  - body: The new redaction rule matches bare `token`, `secret`, and `authorization` labels followed by prose, so normal recovery summaries like `authorization: waiting for review` are silently damaged. The same line also misses common standalone hyphenated `sk-...` keys, so the new ledger/bundle can both lose context and persist a key it meant to redact.
  - confidence: 0.89
- **[P2] Move checkpoint writes off the tool hot path:** `src/agents/session-recovery-state.ts:205-206`
  - body: `appendTaskLedgerEvent` uses synchronous mkdir/append calls, and each checkpoint also synchronously rewrites the recovery bundle. With recovery enabled from update_plan, sessions_yield, and sessions_spawn, a slow state directory blocks the primary tool call even though this path is described as best-effort.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds a new local secret-handling persistence surface, and the redaction patterns do not cover common standalone hyphenated secret prefixes before writing recovery text.

Concerns:

- **[medium] Incomplete recovery secret redaction:** `src/agents/session-recovery-state.ts:113`
  - body: `SECRET_PATTERNS` appears intended to catch `sk`/`pk`/`rk` style keys, but it only matches underscore-prefixed forms. A standalone `sk-...` value in a plan or yield summary would be written to the new recovery ledger/bundle; local 0600 files reduce exposure, but this is still a new persisted secret surface.
  - confidence: 0.76
