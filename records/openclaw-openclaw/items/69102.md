## Summary

Keep open. Current main still lacks a Google/Gemini service-tier parameter path: the bundled Google transport does not type or emit `service_tier`, the extra-param wrapper only forwards existing generic params plus `cachedContent`, and the Google provider docs/tests only cover cache reuse. The issue is concrete, recent, unprotected, and narrow enough for a focused provider fix.

## What This Changes

Request support for configuring Gemini Flex service tier so Google model requests can send `service_tier: "flex"`.

## Best Possible Solution

Add a small bundled Google provider implementation that accepts `serviceTier` and `service_tier` from model/agent extra params, forwards `flex` as top-level `service_tier` on direct Gemini REST payloads, documents the config, and leaves retry/timeout policy as a separate decision.

## Reproduction Assessment

Yes. A static reproduction is enough: configure `agents.defaults.models["google/<model>"].params.serviceTier = "flex"` and inspect the generated direct Gemini payload path; current main has no transport option, extra-param forwarding, request field, docs, or test that can produce `service_tier`.

## Solution Assessment

Yes. The best fix is the narrow provider-owned path through `extensions/google` plus the existing extra-param wrapper; a core scheduler-specific change or generic provider API expansion would be broader than the reported need.
