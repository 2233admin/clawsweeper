## Summary

Current main still does not implement usage-aware load balancing for multi-account auth profiles. Auth selection remains explicit/configured/stored order plus credential-type, last-used, and reactive cooldown sorting; Anthropic usage windows are fetched for status/reporting but are not used as a request-time routing score or proactive circuit breaker. Related open issues overlap the design rather than superseding it.

## What This Changes

Add opt-in usage-aware routing for multi-account auth profiles so requests can prefer profiles with more quota headroom instead of relying only on static order and reactive cooldowns.

## Best Possible Solution

Keep this open and consolidate it with #41135, #24107, and #45771 into an opt-in generic auth-profile routing contract. The design should consume provider-owned usage snapshots, preserve current session-sticky/default behavior, support proactive thresholds and fallback for profiles without usage data, expose routing decisions/status, and update config schema, docs, and tests before enabling Anthropic OAuth usage-aware balancing.
