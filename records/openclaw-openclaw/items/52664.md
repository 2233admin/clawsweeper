## Summary

Keep open. Current main still does not expose `rawBody` on the relevant plugin hook events, and the PR addresses a real plugin ergonomics gap already motivated by the `RawBody` precedent in #51502. The proposed patch is not merge-ready because it derives and forwards `rawBody` for every prepared reply run, which contradicts the PR's own non-channel contract and would expose heartbeat/cron/system-event prompt text as raw user input.

## What This Changes

The PR adds optional `rawBody` to `before_prompt_build`, legacy `before_agent_start`, and `agent_end` hook events, threading it through reply, embedded-runner, deferred follow-up, collect queue, and generated SDK baseline surfaces.

## Best Possible Solution

Land a rebased version that exposes `rawBody` only for user-originated channel runs across immediate, deferred follow-up, and collect paths, keeps CLI/heartbeat/cron/exec-event runs undefined, documents the hook field for plugin authors, and adds focused tests for both channel raw text and non-channel omission.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate rawBody to user-originated channel runs:** `src/auto-reply/reply/get-reply-run.ts:403-404`
  - body: The new hook contract says `rawBody` is undefined for CLI, heartbeat, cron, and other non-channel sources, but these lines derive it from `CommandBody`/`RawBody`/`Body` for every prepared reply and later forward it into the hook pipeline. Existing heartbeat coverage populates all three fields with the scheduler prompt, so plugins would receive system-event text as raw user input unless this value is gated on a real user/channel-originated run.
  - confidence: 0.86
