## Summary

Keep this issue open. Current main has fixed the narrower descendant-subagent announce case, but the report and follow-up comments also describe background exec/process work outliving the subagent’s own agent run. Current task docs and lifecycle code still make agent-run completion authoritative for the subagent task, with registered background exec handled as a separate process notification path rather than as a blocker for subagent completion.

## Best Possible Solution

Keep this open and clarify the intended product contract: whether subagent completion should remain pending for registered background exec/process sessions owned by the child session, or whether those must remain separate task/process notifications. If maintainers choose the stronger invariant, add a generic liveness seam from the process registry/session async-task layer into subagent status and announce deferral, with timeout/cancel behavior and docs/tests. If not, document the boundary and make `/tasks`/`session_status` surface active child-session process work clearly.
