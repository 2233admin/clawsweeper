## Summary

Keep open. Current main has partial hardening around systemd restart limits, doctor/status SecretRef handling, browser SingletonLock cleanup, and multiple-gateway/state-dir guidance, but the central reported startup failure remains: Telegram still inherits the shared retry schema where `jitter` is number-only, so `channels.telegram.retry.jitter: true` is still not accepted by the config validator. The focused fix candidate #52137 is closed and unmerged in the provided context.

## Best Possible Solution

Keep this issue open or split it. The concrete maintainer path is to either accept boolean retry jitter values by preprocessing `true`/`false` in the shared retry schema with focused config regression tests, or explicitly keep jitter numeric-only and add doctor/config repair guidance that flags or converts the boolean before gateway startup.
