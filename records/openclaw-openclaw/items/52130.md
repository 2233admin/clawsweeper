## Summary

Keep open. Current main has partial hardening for service restart limits, SecretRef/status/doctor diagnostics, multiple state dirs, and browser SingletonLock cleanup, but the central startup failure remains: Telegram still uses the shared retry schema where `jitter` is number-only, so `channels.telegram.retry.jitter: true` is still invalid. The focused fix PR #52137 is closed and unmerged in the provided context.

## Best Possible Solution

Keep this issue open and land a narrow fix for the remaining startup crash path: either preprocess boolean retry jitter values (`true` to the default jitter factor, `false` to `0`) in the shared retry schema with focused config regression tests, or deliberately keep jitter numeric-only and add doctor/config repair guidance that detects or converts the boolean before gateway startup. The already-hardened service, SecretRef, state-dir, and browser-lock work can stay as supporting context or be split into narrower follow-ups if maintainers want separate closure criteria.
