## Summary

Keep this issue open. Current main has partial backend mitigation for channel status fan-out, but the Control UI still contains the main UI-side churn paths described in the report: the Channels tab can wait on an 8s status request, generic browser RPCs cannot be aborted or marked stale, node polling starts globally on connect, and session change events still trigger session reloads immediately.

## What This Changes

The issue asks OpenClaw to keep the Control UI responsive during long-lived dashboard sessions even when channel status probes, reconnect churn, polling, or session reloads are slow.

## Best Possible Solution

Land one reconciled Control UI responsiveness fix that scopes polling to active views, debounces or coalesces session reloads, ignores or aborts stale long-running tab requests, keeps slow channel status probes from blocking broader navigation, and adds regression coverage for slow status responses plus session-event churn.

## Reproduction Assessment

Unclear for a fully deterministic end-to-end reproduction: it needs a long-lived dashboard and a slow or reconnecting provider. The contributing source paths are high-confidence on current main, including the 8s channel status wait, global node polling, non-abortable browser RPCs, and immediate sessions.changed reload behavior.

## Solution Assessment

No, current main is not yet the best solution. The backend channel-status concurrency work helps, but the UI still lacks the narrower stale-request, active-view polling, and session-reload controls needed to resolve the reported progressive stuck behavior.
