## Summary

Keep this issue open. Current main has some backend mitigation for `channels.status` fan-in via bounded concurrency, but it still has the reported 8s Control UI `channels.status` request, the original 5s/2s/3s Control UI polling intervals, and no clear request-abort or stale-response guard for channel refreshes. Open PRs #45930 and #69454 are still relevant implementation candidates for the remaining Control UI churn rather than proof that this report is fixed.

## Best Possible Solution

Keep this open and reconcile the remaining work with #45930 and #69454. The best fix is a focused Control UI churn reduction that scopes polling to active views, avoids eager node/device/session refreshes, debounces repeated session reloads, keeps logs tail reliability intact, and adds regression coverage for slow `channels.status` and stale in-flight UI refreshes before closing this report.
