## Summary

Current main still leaves the reported Feishu tenant_access_token/proxy failure unresolved. The Feishu SDK HTTP client path delegates to the Lark SDK default axios instance with timeout injection only; proxy handling on main is limited to the Feishu WebSocket client, and there is no Feishu HTTP proxy policy, config/docs surface, or HTTP_PROXY/NO_PROXY regression coverage for tenant-token/API calls.

## Best Possible Solution

Keep this issue open. The best fix is a maintainer-approved Feishu HTTP proxy policy in the Feishu plugin's Lark.Client httpInstance path, preserving timeout/User-Agent behavior and clearly defining whether ambient env proxying, NO_PROXY bypasses, and any explicit channel/account proxy setting are supported. Add focused regression tests for HTTP_PROXY/HTTPS_PROXY with NO_PROXY on SDK HTTP calls, and document the supported workaround or new setting in the Feishu channel docs.
