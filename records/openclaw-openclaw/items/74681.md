## Summary

Keep open: current main still arms the cron timeout for non-main `agentTurn` jobs from `onExecutionStarted`, so preparation and runner-lane delay before that callback are not bounded by the configured `timeoutSeconds`. The report is concrete, but the requested hard wall-time behavior conflicts with the recently documented/tested #41783 execution-lane semantics, so this needs maintainer contract judgment before an automated fix PR.

## What This Changes

The issue requests `timeoutSeconds` for isolated cron `agentTurn` jobs to be enforced as a hard wall-clock deadline set at job dispatch rather than being deferred until the runner execution-start callback fires.

## Best Possible Solution

Define cron timeout semantics explicitly and then update code, docs, and regression tests together: either split queue/preparation and execution budgets, or make `timeoutSeconds` a total deadline from cron dispatch while adding a separate queue-wait policy so #41783 does not regress silently.
