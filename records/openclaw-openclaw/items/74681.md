## Summary

Keep open. Current main still defers the isolated non-main `agentTurn` timeout until the runner reports execution start, and the existing regression test intentionally protects the #41783 lane-wait behavior, so the requested dispatch/start-time hard deadline is not implemented and needs maintainer contract judgment.

## What This Changes

Request hard wall-time enforcement of cron isolated `agentTurn` `timeoutSeconds` from job start, independent of when the runner callback fires.

## Best Possible Solution

Define the cron timeout contract explicitly, then update code, tests, docs, and changelog together. The most maintainable direction is likely separate total-deadline and execution-budget semantics, or a clearly named queue/preparation budget, so long pre-execution delays are bounded without regressing #41783.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` waits longer than `timeoutSeconds` before calling `onExecutionStarted`; the existing #41783 regression test demonstrates that the abort signal remains un-aborted during that wait.

## Solution Assessment

No, not as a direct automated change that starts every isolated `agentTurn` timeout at dispatch. That would address this report narrowly but contradict the currently tested #41783 semantics, so a safer fix needs explicit maintainer choice on split or renamed budgets.
