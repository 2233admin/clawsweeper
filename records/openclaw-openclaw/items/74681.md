## Summary

Keep open. Current main still defers isolated non-main `agentTurn` cron timeout arming until the runner reports execution start, and the existing regression coverage intentionally protects the conflicting #41783 lane-wait behavior, so the requested hard dispatch-time deadline is not implemented.

## What This Changes

The issue requests that cron `agentTurn` `timeoutSeconds` be enforced as a fixed wall-clock deadline from job start, independent of isolated runner callbacks or async work.

## Best Possible Solution

Define the cron timeout contract explicitly, then update code, tests, and docs together. A maintainable direction is likely separate total-deadline and execution-budget semantics, or a clearly named queue/preparation budget, so #41783 does not regress while long pre-execution delays remain bounded.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` waits longer than `timeoutSeconds` before calling `onExecutionStarted`; the existing #41783 regression test already demonstrates the abort signal remains un-aborted during that wait.

## Solution Assessment

No, not as a direct automated change to start every isolated `agentTurn` timeout at dispatch. That would be narrow for this report but would contradict the currently tested #41783 lane-wait semantics, so the best fix needs an explicit maintainer decision on split or renamed budgets.
