## Summary

Keep open: current main still matches the reported callback-deferred timeout behavior. Isolated cron agent-turn jobs arm the timeout from `onExecutionStarted`, and existing regression coverage explicitly asserts that the configured timeout is not spent before that callback fires.

## What This Changes

The issue requests hard wall-time enforcement for isolated cron `agentTurn` `timeoutSeconds`, independent of callback timing or async runner state.

## Best Possible Solution

Cron should establish an absolute deadline when the run is reserved or marked active, abort the active CLI/embedded run when that deadline arrives even if `onExecutionStarted` is late, keep bounded cleanup once execution metadata is available, and update the #41783 regression/docs to make the chosen queue-wait semantics explicit.

## Repair Work Prompt

Fix isolated cron `agentTurn` timeout enforcement for https://github.com/openclaw/openclaw/issues/74681. Current main arms the timeout from `onExecutionStarted`, so late runner callbacks can make `timeoutSeconds` overshoot. Implement a hard absolute deadline for the cron run that is established when the job is reserved/marked active or otherwise dispatched, not when the callback fires. Preserve `timeoutSeconds <= 0` as no-timeout. Keep timeout cleanup for active isolated runs once execution metadata is known. Reconcile the open #41783 lane-wait regression deliberately: either split queue-wait vs active-execution behavior with explicit tests/error semantics, or update the existing test to reflect the new wall-time contract with a clear comment. Do not change unrelated provider/model timeout behavior, runner lanes, or delivery policy. Update docs if the cron timeout semantics change, and add a changelog fix entry if user-facing behavior changes, crediting the relevant reporter(s).
