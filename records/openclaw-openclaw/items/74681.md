## Summary

Keep open. Current main still deliberately arms non-main `agentTurn` cron timeouts from `onExecutionStarted`, so the reported pre-callback overshoot remains possible, but making `timeoutSeconds` a job-start wall-clock deadline would conflict with the closed #41783 lane-wait semantics and the current regression test that protects them.

## What This Changes

The issue requests that isolated cron `agentTurn` `timeoutSeconds` be enforced as a hard wall-clock deadline from job start instead of from the runner execution-start callback.

## Best Possible Solution

Define the cron timeout contract explicitly, then update code, docs, and regression tests together. The maintainable shape is likely separate queue/preparation and execution budgets, or a new total deadline distinct from `timeoutSeconds`, so operators can bound zombie runs without silently charging lane wait against useful execution time.

## Reproduction Assessment

Yes for the central timeout-boundary problem: a fake isolated runner that withholds `onExecutionStarted` while timers advance shows the cron abort does not fire until that callback. The stale-config data points would need production state/log snapshots to reproduce separately.

## Solution Assessment

Unclear. A hard job-start timer would address the reported overshoot, but it is not clearly the best fix because it conflicts with the shipped #41783 execution-lane semantics; a split budget or renamed total-deadline contract is safer.
