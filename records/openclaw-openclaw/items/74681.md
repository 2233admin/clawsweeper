## Summary

Keep open. Current main still arms the non-main isolated `agentTurn` cron timeout only after the runner reports execution start, while this issue asks for a hard dispatch-time wall-clock deadline. The remaining work conflicts with the shipped #41783 lane-wait semantics, so it needs maintainer contract judgment rather than cleanup closure.

## What This Changes

The issue requests cron isolated `agentTurn` `timeoutSeconds` enforcement as a fixed wall-clock deadline from job start, independent of runner callbacks or async work in flight.

## Best Possible Solution

Define the cron timeout contract explicitly and update implementation, tests, and docs together. The maintainable end state is likely separate total-deadline and execution-budget semantics, or a clearly named queue/preparation budget, so long pre-execution delays are bounded without reintroducing #41783 false timeouts.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an isolated `agentTurn` fake-timer path where `runIsolatedAgentJob` waits longer than `timeoutSeconds` before calling `onExecutionStarted`; the existing #41783 regression test already demonstrates that the abort signal remains un-aborted during that wait.

## Solution Assessment

No, not as a direct automated change to start every isolated `agentTurn` timeout at dispatch. That would match this report narrowly but contradict the currently tested #41783 lane-wait semantics, so the safer solution is an explicit split or renamed budget contract.
