## Summary

Keep open: current main still implements the callback-deferred timeout behavior described here. The cron timeout for non-main `agentTurn` jobs is armed only from `onExecutionStarted`, and an existing regression test explicitly asserts that the configured timeout is not spent while waiting for the runner lane.

## What This Changes

Enforce isolated cron `agentTurn` `timeoutSeconds` as a hard wall-clock deadline from job start rather than from the embedded runner callback.

## Best Possible Solution

Cron should establish one deadline when the run is reserved or marked started, abort the active embedded/CLI run when that deadline arrives, preserve timeout cleanup once execution metadata becomes available, and keep `timeoutSeconds <= 0` as the documented no-timeout escape hatch.
