## Summary

Keep open. Current main still arms isolated cron agentTurn timeouts from the execution-start callback rather than from cron dispatch/job start, and existing regression coverage intentionally protects that behavior for the recently shipped #41783 lane-wait fix; changing it now needs a maintainer contract decision rather than cleanup closure.

## What This Changes

The issue requests that cron isolated agentTurn `timeoutSeconds` be enforced as a hard wall-time deadline from job start, independent of runner callbacks or async work in flight.

## Best Possible Solution

Define cron timeout semantics explicitly, then update code, docs, and regression tests together: either add a total wall-time deadline plus a separate lane/preparation budget, or document `timeoutSeconds` as execution-lane-only and add bounded pre-run/queue controls so this failure mode has a clear policy.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists in current main: the #41783 fake-timer regression advances past `timeoutSeconds` before `onExecutionStarted` and verifies the abort signal remains unset until the callback fires.

## Solution Assessment

Unclear. A hard job-start timer would address this report, but by itself it would reverse the shipped #41783 lane-wait behavior; the safer solution is a maintainer-approved split or clarified timeout contract.
