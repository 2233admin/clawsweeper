## Summary

Keep open. Current main still defers non-main `agentTurn` cron timeout arming until `onExecutionStarted`, so the reported wall-time overshoot remains plausible; however, changing it directly would conflict with the recently closed #41783 execution-lane timeout semantics that current tests and changelog now encode.

## What This Changes

The issue requests enforcing isolated cron `agentTurn` `timeoutSeconds` as a hard wall-time deadline from job start or dispatch rather than from the later execution-start callback.

## Best Possible Solution

Define cron timeout semantics explicitly, then update code, docs, and regression tests together: either make `timeoutSeconds` a total dispatch deadline with a separate queue-wait policy, or keep execution-lane timeout semantics and add a distinct bounded queue/preparation budget so late callbacks cannot create silent overshoot.

## Reproduction Assessment

Yes for the core behavior: a focused fake-timer check can delay `onExecutionStarted` past `timeoutSeconds` and observe that the abort signal remains un-aborted until the callback fires. The current #41783 regression test already encodes that opposite-of-requested behavior.

## Solution Assessment

Unclear. A hard wall-time timer would address the reported overshoot, but applying it directly would regress the recently shipped #41783 behavior; the safer solution is a maintainer-approved split between queue/preparation and execution budgets.
