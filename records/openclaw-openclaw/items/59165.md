## Summary

Keep open. Current main has static SecretRef support and model-provider auth hooks, but it does not ship the requested cross-domain CredentialProvider/CredentialRef plugin contract or consumer runtime API. This is not a ClawHub-only cleanup item because the useful external providers discussed in the issue depend on a missing core/plugin-sdk seam.

## Best Possible Solution

Keep the RFC open for maintainer product/security design. If accepted, implement it as an additive plugin SDK capability: typed CredentialProvider/CredentialRef contracts, registry, runtime consumer helper, trust/scoping policy, interactive setup support, docs, and contract tests, while preserving SecretRef as the static env/file/exec mechanism. AWS Secrets Manager or the referenced agent-identity approach can become reference providers after the core seam exists.

## Risks / Open Questions

- Credential-provider plugins would be a cross-domain security boundary; implementation needs explicit trust, scoping, redaction, and lifecycle policy before accepting third-party providers.
- The RFC overlaps SecretRef, AuthProfileStore, ProviderAuthMethod, skills, tools, and channel setup, so the best path is a narrow additive contract with compatibility tests rather than piecemeal provider-specific code.
