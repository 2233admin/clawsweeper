## Summary

Current main still does not implement the requested `assistantPrefix` contract. The issue describes a real automation reliability gap, but cron `agentTurn` payload validation, cron TypeScript types, agent RPC params, agent defaults validation, CLI flags, and cron execution all lack a field or provider-prefill path. Existing `messages.responsePrefix` is only a delivery-time prefix and does not force the model/provider response to start with the fixed text.

## Best Possible Solution

Keep this open for maintainer API/design review. If accepted, implement it as an additive core protocol/runtime feature: add a validated optional prefix field to cron `agentTurn` payloads, agent/default config, and `agent` RPC/CLI surfaces; carry it through cron and normal agent execution into a typed provider payload hook; support at least Anthropic and one OpenAI-compatible route with explicit thinking-mode behavior; define no-op/debug semantics for unsupported providers; and document/test streaming plus fallback-delivery behavior.

## Risks / Open Questions

- Provider-native assistant prefill semantics differ by provider and can conflict with thinking modes; current Anthropic thinking support strips assistant prefill in some paths.
- Treating `messages.responsePrefix` as a substitute would only prefix delivered text after the model has already produced content, so it would not prevent the reported streamed narration/preamble failure mode.
- The requested feature crosses public protocol schemas, config validation, CLI flags, cron execution, streaming/fallback behavior, and provider wrappers, so it needs explicit maintainer API design before implementation.
