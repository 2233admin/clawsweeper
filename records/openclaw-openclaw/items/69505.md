## Summary

Keep open. Current main still lacks an `assistantPrefix` or equivalent provider-prefill contract across cron payloads, direct agent RPC, agent defaults, CLI, docs, and the cron execution path; the existing `messages.responsePrefix` is an outbound delivery prefix applied after model output exists, so it does not solve the reported reliability gap.

## What This Changes

Request to add an optional assistant reply prefix/pre-fill field to cron `agentTurn` payloads, agent defaults/direct agent runs, CLI/docs, and provider adapters so generated replies can be forced to begin with fixed text.

## Best Possible Solution

If maintainers accept the direction, add a backwards-compatible typed field that flows from cron payloads, direct agent runs, and agent defaults into a provider-facing continuation/pre-fill hook; document unsupported-provider no-op behavior, support at least Anthropic and one OpenAI-compatible path, and cover streaming/fallback delivery so fixed-template automations cannot leak preamble before the required first text.

## Reproduction Assessment

Yes for the missing contract: a cron `agentTurn` payload containing `assistantPrefix` has no schema/type/CLI/runtime support on current main, and the strict cron schema would reject the unknown property. The model-specific production narration incident itself was not live-reproduced in this read-only review.

## Solution Assessment

Unclear until maintainer API review. The requested provider-prefill direction is plausible and `messages.responsePrefix` is not equivalent, but the exact contract should be designed centrally because continuation semantics vary by provider and runtime.
