## Summary

Keep open. Current main still leaves the reported cron-through-shell failure mode possible: the reply guard suppresses the visible note only when `successfulCronAdds` is nonzero or an existing same-session cron is found, while the counter is still updated only for the native `cron` tool path. The direct fix attempt in #53024 is closed unmerged, and related open PRs address broader or adjacent reminder-note behavior rather than this exact successful `openclaw cron add --at` telemetry gap.

## What This Changes

The issue reports that assistant-created one-time reminders scheduled via `openclaw cron add --at` can still receive the contradictory unscheduled-reminder note.

## Best Possible Solution

Implement a narrow scheduling-signal fix so successful current-turn reminders are recognized consistently across the native cron tool and the assistant shell/exec CLI path, with regression tests for successful, failed, and unrelated shell commands plus the final reply behavior. Keep broader note-stripping, opt-out config, and plugin-SDK scheduling signals as separate maintainer-reviewed decisions.

## Repair Work Prompt

Fix the false unscheduled-reminder note for reminders successfully created through an assistant shell command such as `openclaw cron add --at ...`. Current main only increments `successfulCronAdds` for the native `cron` tool add path in `src/agents/pi-embedded-subscribe.handlers.tools.ts`, so an `exec` or `bash` command can create a real Gateway cron job while the reply guard still sees zero current-turn cron adds and appends the contradictory note. Use PR #53024 as source context, but implement a conservative current-main patch: count successful exec/bash cron-add commands only when the command completed successfully, avoid unrelated shell commands and failed cron adds, and preserve the native cron-tool behavior. Add focused tests in `src/agents/pi-embedded-subscribe.handlers.tools.test.ts` for successful exec/bash cron add, failed exec/bash cron add, and unrelated exec commands; add or adjust reply-level coverage in `src/auto-reply/reply/agent-runner.misc.runreplyagent.test.ts` only if needed to prove the final visible reply stays clean. Do not introduce a broad delivery sanitizer, a new public config key, a Telegram-only workaround, or plugin SDK scheduling APIs in this repair. Be aware of related PRs #39330, #47663, and #71170, but keep this PR narrowly on shell/exec cron-add telemetry. Add a user-facing changelog fix entry if required by the repo convention, crediting the original reporter and useful prior contributor rather than any bot.
