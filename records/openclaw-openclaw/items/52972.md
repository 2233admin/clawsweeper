## Summary

Current main still has the reported failure mode. The reminder note is still appended when a reply contains a reminder commitment and `successfulCronAdds` is zero, while current code only increments that counter for the native `cron` tool path. A successful `openclaw cron add --at ...` run through exec/bash can still create a Gateway cron job via the CLI without updating the run-local counter. The direct fix PR #53024 is closed and unmerged, so this remains actionable.

## Best Possible Solution

Keep this open. Implement #53024 or an equivalent current-main fix that treats successful `openclaw cron add --at ...` executions through exec/bash as successful reminder scheduling for the current turn, or ensure assistant-created reminders consistently use the native cron tool. Add regression coverage for successful exec/bash cron-add suppression, failed exec/bash cron-add still warning, and Telegram delivery not receiving the false note after successful scheduling.

## Risks / Open Questions

- The same-session cron-store fallback can mask some reproductions when an enabled matching job already exists; regression coverage should isolate a fresh session or empty cron store.
- A broad suppression based only on command text would be fragile and could hide unsupported reminder promises; the fix should be tied to successful scheduling telemetry or route reminders through the native cron tool.
- Related external-scheduler opt-out work such as #71170 does not directly fix this CLI cron-add path because this report is about OpenClaw’s own `openclaw cron add` invoked through exec/bash.
