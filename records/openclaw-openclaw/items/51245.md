## Summary

Keep open. Current main still does not handle the exact raw `telegram:slash:<id>` sandbox-explain session shape from the report: colon-containing input is returned unchanged, provider inference only parses agent-scoped session keys, and elevated diagnostics require a resolved channel before `allowedByConfig` can become true. The Telegram native slash runtime path now carries stronger metadata, but open same-purpose PRs with closing syntax remain implementation candidates, so this issue should not be closed before maintainers reconcile them.

## Best Possible Solution

Keep this issue open. Maintainers should reconcile #51286 and #51584 against current main, then land or write a focused fix that makes `sandbox explain --session telegram:slash:<id>` resolve `telegram`, preserves current agent-scoped native slash behavior, and adds regression coverage for both the CLI diagnostic and the Telegram runtime elevated authorization path.
