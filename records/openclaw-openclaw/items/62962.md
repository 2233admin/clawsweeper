## Summary

Keep open. Current main still lacks the requested schema-driven tool security classification contract; it has adjacent controls such as owner-only filtering, allow/deny policy, plugin before_tool_call approvals, plugin tool display metadata, and provider-level parallel_tool_calls knobs, but no public per-tool metadata surface for the six requested properties or consumers that use them.

## What This Changes

Add optional tool security classification metadata to tool definitions and use it for approval prompts, parallel scheduling, interrupt behavior, open-world policy, classifier input, and conformance checks.

## Best Possible Solution

Define a backwards-compatible public tool-security metadata contract, align its concurrency semantics with the upstream AgentTool executionMode contract, preserve it through core tools, plugin registration, wrappers, adapters, inventories, docs, and conformance snapshots, then wire consumers behind conservative fallbacks that keep current approval, policy, sandbox, and redaction controls in force.

## Risks / Open Questions

- This is a public core/plugin SDK contract change, so defaults and optionality need explicit compatibility design for third-party plugins and existing bundled tools.
- toAutoClassifierInput could leak secrets, SecretRef-derived values, media references, URLs, or sensitive tool arguments unless the contract includes redaction and bounding rules before any classifier prompt.
- Concurrency metadata must not become an implicit bypass around owner-only gating, allow/deny policy, approval hooks, sandboxing, provider transport limits, or host exec approvals.
- The pinned upstream AgentTool already has executionMode for sequential/parallel behavior, so maintainers should decide whether isConcurrencySafe wraps, aliases, or replaces that existing dependency contract.
