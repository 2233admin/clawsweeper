## Summary

Keep this PR open for maintainer review. Current main still lets the run-side typing TTL expire during active keepalive loops, the branch is narrow with no concrete blocking bug found, and the broader related presence items are open or unmerged rather than superseding this change.

## What This Changes

The PR adds a `refreshTypingTtl()` call to the shared typing keepalive tick and adds a typing persistence regression test for active runs that continue beyond the initial TTL.

## Best Possible Solution

Land this narrow fix after maintainer review and CI, or deliberately fold the same TTL renewal and regression test into #65919. The final behavior should preserve run-complete, dispatch-idle, sealed-controller cleanup, and the existing provider-facing keepalive cadence.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to start a typing controller with a 6s keepalive interval and 10s TTL, advance fake timers to the first keepalive tick, then advance past 10s total; on current main cleanup fires because the keepalive tick did not refresh the TTL. The PR's added test encodes this path.

## Solution Assessment

Yes. The proposed fix is the narrowest maintainable change for this specific bug: renew the run-side TTL when an allowed keepalive tick successfully sends typing, without increasing the tick rate or bypassing the `runComplete` start guard. Broader active-run presence leasing can remain in #65919.
