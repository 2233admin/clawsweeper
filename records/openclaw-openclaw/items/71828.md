## Summary

Keep open. Current main still has the run-side typing TTL gap this PR targets, while the PR is a narrow two-file change with a direct regression test and no concrete correctness or security issue found; the remaining blocker is maintainer review plus current CI failures on the PR head.

## What This Changes

The PR refreshes the shared typing controller TTL on each allowed keepalive tick and adds a Vitest regression for an active long run that continues beyond the original TTL window.

## Best Possible Solution

Land this narrow PR after maintainer review and green or explained CI, or deliberately fold the same one-line TTL renewal and regression test into the broader #65919 presence work. The final behavior should keep run-side typing TTL alive only while the existing keepalive loop is allowed to send typing and should preserve run-complete, dispatch-idle, and sealed-controller cleanup protections.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to create a typing controller with a 6s keepalive interval and 10s TTL, start the loop, advance fake timers through the first keepalive tick and then past 10s total; on current main the cleanup path can fire because the keepalive tick does not refresh the TTL.

## Solution Assessment

Yes. The proposed patch is the narrowest maintainable fix for this specific bug because it renews the TTL inside the existing start guard after a keepalive tick sends typing, without increasing cadence, bypassing run-complete blocking, or changing channel-facing typing behavior.
