## Summary

Keep open. Current main includes meaningful Active Memory timeout mitigations, but the reported MiniMax-M2.7 failure is still not fully resolved: timeouts without partial assistant text still inject no memory, and the documented/configured fallback remains model-selection only rather than runtime failover after a slow model times out.

## What This Changes

The issue requests that Active Memory work reliably with MiniMax-M2.7, either by completing within the timeout budget or by falling back when the selected recall model is too slow.

## Best Possible Solution

Keep this open until maintainers choose and implement an explicit Active Memory recovery policy for slow recall models: for example runtime retry to a configured fallback model, local memory-search summary recovery, or an asynchronous/fail-open path with docs and regression tests that make the no-summary timeout behavior intentional.

## Reproduction Assessment

Yes. Multiple users provided matching MiniMax-M2.7 logs showing Active Memory timeout with `summaryChars=0`, and current source still has a no-partial timeout path that returns no prompt context; I did not perform a live MiniMax call in this read-only pass.

## Solution Assessment

No. Current main is not the complete solution for this report: it adds setup grace, hard-deadline behavior, partial-summary recovery, and circuit-breaker mitigation, but it still does not complete recall or switch to another/local fallback when MiniMax times out without producing text.
