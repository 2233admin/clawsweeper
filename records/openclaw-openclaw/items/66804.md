## Summary

Keep open. Current main has meaningful Active Memory timeout mitigations, including hard-deadline handling, partial transcript recovery, and a consecutive-timeout circuit breaker, but it still does not implement fallback-after-timeout for the selected recall model. A MiniMax-M2.7 recall timeout can still end with `summary: null`, and the prompt hook injects no memory context.

## What This Changes

Active Memory should complete or fail open quickly with MiniMax-M2.7, ideally by falling back to a configured backup or local recall path when the selected recall model times out.

## Best Possible Solution

Keep the report open or explicitly consolidate it into a canonical Active Memory timeout/fallback issue. The end state should make Active Memory fail open within the foreground budget and either retry a configured recall fallback within that budget or provide a documented local recall fallback that does not depend on the slow selected model.
