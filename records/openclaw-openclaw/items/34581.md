Current main still does not implement this PR's intended Microsoft Teams webhook behavior: the Teams monitor still uses strict `express.json`, forwards non-413 parse errors, and has no raw-body JSON escape repair or authenticated HTTP 200 fallback for unrecoverable Bot Framework payloads. The PR remains a focused implementation candidate, with prior review comments already addressed by the author, but it changes a security-sensitive webhook parsing boundary and should stay open for Teams maintainer/security review rather than be closed by cleanup.

## Best Possible Solution

Keep this PR open as the implementation candidate. Rebase it onto current main, review the Teams webhook security boundary, preserve auth-before-body-parsing and the existing size limit, repair only truly bare invalid JSON escapes, return HTTP 200 only for intended authenticated Bot Framework parse failures, and add integrated middleware coverage before landing.
