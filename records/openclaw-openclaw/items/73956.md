## Summary

Keep open. Current main still has the stale assistant-media fallback path that can append a text-only `:assistant-media` transcript entry, and open PRs #73962 and #74502 already target the same bug with closing syntax, so this issue should stay open until one fix lands.

## What This Changes

The issue reports duplicate Control UI/WebChat assistant replies when a stale or non-playable assistant-media/TTS payload appends a second gateway-injected transcript entry containing the final answer text.

## Best Possible Solution

Land one narrow gateway fix that skips agent-run assistant-media transcript appends unless resolved media content exists, while preserving resolved auto-TTS audio supplements and non-agent command media fallback behavior.

## Reproduction Assessment

Yes. The issue gives a concrete stale-media/TTS sequence, and current source shows the static reproduction path: a media-bearing agent payload with no resolved media can still fall back to text and append a `:assistant-media` transcript message.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix: require actual persisted media content before writing the supplemental assistant-media transcript entry, with a regression test for stale local audio.
