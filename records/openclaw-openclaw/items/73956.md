## Summary

Keep open. Current main and the latest release still contain the agent-run assistant-media append path that can persist a text fallback with a `:assistant-media` idempotency key when no playable media is resolved, and two unmerged PRs already target the same issue with closing syntax.

## What This Changes

The issue reports duplicate Control UI/WebChat assistant replies caused by the gateway `assistant-media` transcript path replaying text after TTS/media has no playable content.

## Best Possible Solution

Land one canonical narrow gateway fix that writes supplemental agent-run assistant-media transcript entries only when resolved non-text media exists, add regression coverage for stale/non-playable media, and close the duplicate implementation PR.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an agent-run final payload with visible text plus a stale or non-playable `mediaUrl`: the media builder returns no media message, but the append path can still persist text with an `:assistant-media` idempotency key.

## Solution Assessment

Yes. Guarding the agent-run supplemental append on resolved audio/image content is the narrowest maintainable fix, while preserving the command/non-agent TTS path that has no separate model transcript.
