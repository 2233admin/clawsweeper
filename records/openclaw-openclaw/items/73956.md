## Summary

Keep open. Current main still has the gateway path that can append a supplemental `:assistant-media` assistant transcript after falling back to visible text when no embeddable media is resolved, and two open PRs already target this issue with closing syntax.

## What This Changes

The issue reports duplicate Control UI/WebChat assistant replies caused by the assistant-media/TTS delivery path persisting a second text-only `gateway-injected` transcript message after the normal final answer.

## Best Possible Solution

Review and land #74502 or an equivalent narrow gateway fix that appends the supplemental assistant-media transcript only when resolved non-text media content exists, while preserving command/non-agent media fallback behavior; then close or supersede #73962 if it is no longer needed.

## Reproduction Assessment

Yes. The issue gives a concrete sequence, and source inspection provides a high-confidence seam reproduction: an agent-run final payload with visible text plus stale/non-embeddable media can pass the raw media check and fall back to text for the supplemental assistant-media append.

## Solution Assessment

Yes. The open PR direction is the narrowest maintainable fix because it keeps the normal model transcript responsible for text and limits the supplemental assistant-media message to resolved media content; the issue should stay open until one closing PR merges.
