## Summary

Keep this issue open. Current main supports the working oMLX route through the OpenAI-compatible memory embedding provider, but the requested oMLX-specific discoverability docs are still missing, and the linked open PR #74761 is the active closing implementation path.

## What This Changes

Request to document, and optionally first-class support, oMLX as an Apple Silicon OpenAI-compatible memorySearch embedding provider.

## Best Possible Solution

Review and either land PR #74761 or an equivalent docs-only change that adds the oMLX recipe, warns against configuring it as Ollama, and tells users to reindex after changing embedding vector space; treat a native adapter as a separate follow-up only if maintainers want provider-specific defaults or diagnostics.

## Reproduction Assessment

Yes for the docs gap: current main can be checked with an `oMLX`/`omlx`/`MLX` search across the relevant docs and source files. This is a feature/docs request rather than a runtime bug, and the issue includes a verified working runtime configuration.

## Solution Assessment

Yes, the docs-only PR path is the narrowest maintainable solution for the current user-facing problem because the OpenAI-compatible runtime path already exists. A first-class `omlx` adapter would be additional UX/product scope, not required to resolve the immediate discoverability issue.
