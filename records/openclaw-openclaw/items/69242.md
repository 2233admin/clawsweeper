## Summary

Keep open. Current main still preserves signal and failure classification inside the supervisor/runtime, but foreground exec results and process poll details do not consistently expose that structured termination cause to callers. The merged Linux OOM-score work explicitly left this as separate SIGKILL observability work, and the open macOS report shows the same generic diagnostic gap remains cross-platform.

## What This Changes

The issue asks OpenClaw to avoid or clearly diagnose intermittent Linux exec SIGKILLs during broad find/grep discovery commands when host OOM evidence is absent.

## Best Possible Solution

Add a narrow, generic exec/process termination diagnostics path that surfaces the existing supervisor/runtime metadata in foreground exec results and background process list, poll, and log details. Keep resource limits, recursive-scan guardrails, and OOM victim-selection policy as separate product work unless maintainers explicitly scope them in.

## Reproduction Assessment

Partially yes: the exact intermittent broad-search SIGKILL is not deterministic from the report, but the remaining actionable diagnostics gap is source-verifiable on current main by following a signal or timeout through RunExit, ExecProcessOutcome, and the public tool details. A fix PR should add unit tests that force signal and timeout exits rather than depend on host memory pressure.

## Solution Assessment

Yes: the best next step is an additive observability fix that reuses existing supervisor/runtime fields. It is narrower and safer than changing Linux resource policy or adding broad find/grep guardrails in the same patch.
