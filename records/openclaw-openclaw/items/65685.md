## Summary

Keep this issue open. Current main still enqueues the followup, only drains immediately when the active snapshot is stale, and then unconditionally cleans up the typing controller before returning, so the reported live-active queued followup liveness bug remains present.

## What This Changes

The issue requests that queued followups behind an active run keep a visible typing or liveness signal instead of immediately clearing typing.

## Best Possible Solution

Update the active `enqueue-followup` path to compute post-enqueue liveness once: drain immediately when the active snapshot is stale, but when a followup is truly queued behind a live run, keep or refresh typing and defer cleanup until the queued turn runs or is otherwise resolved.

Fix the queued-followup typing liveness bug described in openclaw/openclaw#65685. In `src/auto-reply/reply/agent-runner.ts`, update the `activeRunQueueAction === "enqueue-followup"` branch so it computes post-enqueue liveness once after enqueue. If the active snapshot is stale, call `finalizeWithFollowup(...)` to drain immediately. If the followup is truly queued behind a still-active run, keep or refresh the typing signal, for example through the existing `typingSignals.signalToolStart()`, and do not call `typing.cleanup()` before returning. Preserve cleanup behavior for heartbeat/drop/silent/completed paths and avoid changing queue policy semantics or Discord-specific code unless tests prove it is necessary. Add focused coverage in `src/auto-reply/reply/agent-runner.runreplyagent.e2e.test.ts` for both stale-active immediate drain and still-active queued liveness, including an assertion that cleanup is not called while `isRunActive()` remains true and that typing is refreshed/started.
