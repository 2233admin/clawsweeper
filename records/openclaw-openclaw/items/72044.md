## Summary

Keep this issue open. Current main has adjacent DeepSeek V4 replay fixes, docs, and short replay coverage, but the reported failure is a longer DeepSeek-origin multi-turn tool-call/subagent chain where a later assistant tool-call message can lack its own thinking block; current wrapper code still backfills missing `reasoning_content` with an empty string instead of recovering prior real reasoning content.

## What This Changes

The issue requests a complete fix for DeepSeek V4 Pro high-thinking multi-turn tool-call and `sessions_spawn` flows that fail with a 400 `reasoning_content` replay error after several tool turns.

## Best Possible Solution

Add a focused regression for a DeepSeek-origin chain with multiple assistant tool-call continuations, preferably including a `sessions_spawn`-shaped subagent flow, then update the DeepSeek/OpenAI-compatible replay wrapper to preserve or recover real prior `reasoning_content` only where DeepSeek requires it while keeping the existing provider-switch compatibility behavior intentional.

## Reproduction Assessment

Yes. The issue gives a concrete DeepSeek V4 high-thinking `sessions_spawn`/multi-tool path, and three independent comments report the same 400 `reasoning_content` failure; the missing piece is a current-main automated or live reproduction of the longer chain.

## Solution Assessment

No, closing as already fixed is not the best path. The current implementation and tests cover short replay and blank fallback cases, but the safer maintainable solution is to add the long-chain regression first and only then adjust the DeepSeek replay wrapper where the regression proves real reasoning is lost.
