## Summary

Keep open: current main still hard-fails Telegram deleteMessage errors, and the latest PR head is not merge-ready. The current head still lets benign delete failures hit the shared ERROR logger before the outer catch handles them, its new test table still expects `MESSAGE_DELETE_FORBIDDEN` to soft-fail even though the production classifier no longer does, and CI lint annotations show the branch is red on the new boolean comparisons.

## What This Changes

The PR changes Telegram `deleteMessage` handling to return soft-fail warnings for selected Bot API 400s, surfaces `deleted: false` warnings from the Telegram action runtime, adds regression tests, and updates the changelog.

## Best Possible Solution

Revise this PR in place so one benign-delete predicate is reused for both `requestWithDiag(..., { shouldLog })` and the catch, only true already-gone/stale delete states fail soft, forbidden/permission failures stay loud, the tests match that classifier, and the branch clears targeted Telegram tests plus lint.

## Review Findings
