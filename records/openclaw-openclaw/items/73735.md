## Summary

Keep this PR open. Current main still has the reported Telegram deleteMessage hard-fail behavior, but the latest PR head still has two blocking issues: benign deletes are caught after the shared ERROR logger has already run, and `MESSAGE_DELETE_FORBIDDEN` is still classified as a benign no-op despite Telegram's permission-sensitive delete contract. Later comments say those points were fixed, but GitHub raw/API content for head `a243b68231f7746939af1d2f43c6196925dbb937` still shows the old classifier and call shape.

## What This Changes

The PR changes Telegram `deleteMessage` handling to soft-fail selected Bot API 400s, surfaces `deleted: false` warnings from the Telegram action runtime, adds `send.ts` regression coverage, and updates `CHANGELOG.md`.

## Best Possible Solution

Revise this PR so only true idempotent no-op delete states are fail-soft and no-log. Use one local predicate for the already-gone cases, pass it into `requestWithDiag(..., { shouldLog })`, reuse it in the catch, keep permission/auth failures such as `MESSAGE_DELETE_FORBIDDEN` throwing and logging normally, and add focused tests for both logger suppression and forbidden-error propagation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Suppress benign deletes before the API logger runs:** `extensions/telegram/src/send.ts:1094`
  - body: `requestWithDiag` wraps this call in `withTelegramApiErrorLogging`, which logs `telegram deleteMessage failed: ...` before rethrowing. Catching outside it changes the return value, but it still emits the ERROR-level log that the linked bug is trying to stop. Pass a benign-delete `shouldLog` predicate into `requestWithDiag` and reuse the same predicate in the catch.
  - confidence: 0.96
- **[P2] Do not soft-fail forbidden delete errors:** `extensions/telegram/src/send.ts:1106`
  - body: `MESSAGE_DELETE_FORBIDDEN` can mean the bot lacks delete rights or is operating in the wrong context, not that the message is already gone. Because the action runtime converts every soft result into `{ ok: true, deleted: false }`, this turns permission/configuration regressions into successful no-ops. Keep forbidden errors out of the benign classifier and test that they still throw.
  - confidence: 0.9
