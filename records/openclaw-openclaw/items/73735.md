## Summary

Keep this PR open: current main still has the reported Telegram deleteMessage hard-fail behavior, but the latest PR head has two merge blockers. It catches benign delete errors only after the shared API logger has already emitted the ERROR-level log the fix is meant to avoid, and its new test table still expects MESSAGE_DELETE_FORBIDDEN to fail soft even though the PR code and changelog now say it should propagate.

## What This Changes

The PR changes Telegram deleteMessage handling to return fail-soft warnings for selected Bot API 400s, surfaces deleted:false warnings from the Telegram action runtime, adds regression coverage, and updates the changelog.

## Best Possible Solution

Revise this PR in place so one benign-delete predicate is reused for both requestWithDiag(..., { shouldLog }) and the catch, only already-gone or stale-message delete states fail soft, permission failures keep throwing, and the tests prove both logger suppression and forbidden-error propagation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P2] Suppress benign deletes before the API logger runs:** `extensions/telegram/src/send.ts:1094`
  - body: `requestWithDiag` wraps this call in `withTelegramApiErrorLogging`, which logs `telegram deleteMessage failed: ...` before rethrowing unless a per-call `shouldLog` predicate suppresses it. Catching outside the call changes the return value, but it still emits the ERROR-level log that the linked bug is trying to stop; pass the same benign-delete predicate into `requestWithDiag(..., { shouldLog })` and reuse it in the catch.
  - confidence: 0.97
- **[P2] Move forbidden deletes out of the benign test table:** `extensions/telegram/src/send.test.ts:2076`
  - body: The latest code no longer matches `MESSAGE_DELETE_FORBIDDEN`, and the changelog says permission failures propagate, but this new test row still expects that error to return `{ ok: false }`. As written, the test suite will fail on this row; move it to a non-benign rejection case instead.
  - confidence: 0.98
