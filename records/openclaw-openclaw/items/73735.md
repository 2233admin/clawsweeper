## Summary

Keep this PR open. Current main still throws/logs benign Telegram deleteMessage failures, and this PR is a useful implementation candidate, but the proposed catch runs after the shared Telegram API error logger has already emitted the ERROR-level log that the linked bug is trying to remove.

## What This Changes

The PR changes Telegram deleteMessage handling to return a warning for selected Bot API 400 errors, surfaces that warning from the Telegram action runtime, adds send.ts regression coverage, and adds an Unreleased changelog entry.

## Best Possible Solution

Revise this PR so deleteMessage classifies benign Telegram delete errors before the shared API error logger runs, preferably with one local predicate reused for `requestWithDiag(..., { shouldLog })` and the catch. Keep real failures propagating, surface `{ deleted: false, warning }` from the action runtime, and add focused tests that prove benign cases do not emit the error logger while non-benign failures still throw.
