## Summary

Keep this PR open. The requested Telegram behavior is still absent from current main, and the PR remains a focused fix candidate, but the latest head is not merge-ready because it still logs benign delete failures before the catch handles them, has a fail-soft test row that no longer matches the production classifier, and currently fails lint on the new boolean comparisons.

## What This Changes

The PR changes Telegram `deleteMessage` handling to return soft-fail warnings for selected 400 errors, updates the Telegram action runtime to surface `deleted: false` warnings, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Revise this PR in place so one benign-delete predicate is reused for both `requestWithDiag(..., { shouldLog })` and the catch, only already-gone or stale-message delete states fail soft, permission failures continue to throw and log normally, and the Telegram tests plus lint pass on the final head.
