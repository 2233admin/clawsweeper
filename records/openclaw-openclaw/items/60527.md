## Summary

Keep this PR open. Current main has fixed the Telegram/private-proxy symptom by clearing the target hostname allowlist only when `allowPrivateProxy` is true, but it still does not implement this PR's broader public explicit-proxy behavior. The PR diff is focused and its security-sensitive behavior should be reviewed or deliberately rejected by maintainers, not closed as implemented.

## Best Possible Solution

Keep this PR open for maintainer review. Either adapt/land the proxy-specific policy change with tests for both private and public explicit proxies while preserving private-network blocking without `allowPrivateProxy`, or explicitly document and test that `allowPrivateProxy` is the required trust bit for every proxy hostname outside the target `hostnameAllowlist`.

## Risks / Open Questions

- The PR intentionally broadens explicit-proxy validation semantics for public proxies by no longer applying a target `hostnameAllowlist` to the proxy host; that is plausible for operator-configured proxies but should be accepted deliberately because this is SSRF policy behavior.
- Current main lacks the PR's public-proxy regression test, so closing this as implemented would leave that behavior undecided and uncovered.
- If maintainers intend `allowPrivateProxy` to mean any proxy host outside the target allowlist is trusted, that contract needs to be documented and covered with a public-proxy negative test.
