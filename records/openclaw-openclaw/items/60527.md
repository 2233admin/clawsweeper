## Summary

Keep this PR open. Current main fixes the Telegram/local private-proxy symptom by clearing `hostnameAllowlist` only when `allowPrivateProxy` is true, but it still preserves the caller's target allowlist for default/public explicit proxy validation, which is part of this PR's stated behavior. Because that remaining behavior changes SSRF/proxy trust semantics, it should be reviewed or explicitly rejected by maintainers rather than closed as implemented.

## Best Possible Solution

Keep this PR open for maintainer review. Either adapt and land the proxy-specific policy change with tests for private proxy, public proxy, and private-proxy-without-`allowPrivateProxy` invariant preservation, or deliberately keep `allowPrivateProxy` as the required trust bit for proxy hostnames outside the target `hostnameAllowlist` and add a test/docs note for that policy.
