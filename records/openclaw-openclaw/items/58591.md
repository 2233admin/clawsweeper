## Summary

Close as implemented on main. The Feishu dispatcher still does not directly thread `mediaLocalRoots`, but the observable reply-attachment failure is already solved earlier in the production reply path: current main normalizes local reply media with agent-scoped access, stages it into managed media storage, and then hands Feishu a default-allowed managed path. That shipped in `v2026.4.15`, so this narrow PR is now redundant.

## What This Changes

The PR imports `getAgentScopedMediaLocalRoots` into the Feishu reply dispatcher, forwards the derived roots to `sendMediaFeishu` for reply attachments, and adds a unit test for that forwarding.

## Best Possible Solution

Keep the shipped shared reply-media staging design as the canonical fix. If a future report identifies a Feishu path that bypasses `dispatchReplyFromConfig`, track that as a fresh narrow bug instead of merging duplicate channel-local root plumbing for the already-covered production reply flow.

## Reproduction Assessment

Yes. The old repro was a Feishu agent reply carrying a local media path under the agent workspace; current main can be traced through `dispatchReplyFromConfig` into `createReplyMediaPathNormalizer`, and tests cover staging absolute workspace media before channel delivery.

## Solution Assessment

Yes for the current-main solution, no for merging this PR now. Centralizing local-file validation and staging before channel dispatch is the narrower maintained design; the PR's direct Feishu dispatcher plumbing is reasonable but redundant for production replies.
