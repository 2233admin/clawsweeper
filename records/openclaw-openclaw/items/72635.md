## Summary

Keep this PR open for maintainer handling, not cleanup-close. Current main still has the missing-python write failure reported in #72362, but this patch puts the fallback inside the raw pinned-write helper and would use a path-based writer when the Linux pinned helper cannot spawn, which weakens the fs-safe boundary; the same remaining work is also being pursued by open PR #73791 with a narrower fs-safe-level fallback.

## What This Changes

The PR adds spawn-error detection to `runPinnedWriteHelper`, waits one event-loop turn before writing to the Python helper, falls back to the existing local writer on ENOENT/EACCES/ENOEXEC, and adds helper-level buffer/stream fallback tests.

## Best Possible Solution

Fix #72362 without weakening fs-safe write boundaries: either handle spawn failures at the `fs-safe.ts` write/copy call sites using the already validated legacy writer/copy path, or add a truly pinned non-Python implementation. The merged solution should cover buffer writes, copy/stream writes, non-spawn helper errors, and symlink race cases, and then close or supersede the duplicate PR.

## Reproduction Assessment

Yes. The issue has a high-confidence Docker repro from #72362 using a slim Node image without `python3`, and current main source shows the write helper falls back to spawning bare `python3` and then normalizes that failure to `invalid-path`. A small Node 24 check also confirmed ENOENT spawn errors are emitted before `setImmediate`, matching the PR's intended early-spawn path.

## Solution Assessment

No. The intended missing-Python fallback is useful, but putting it inside `runPinnedWriteHelper` is not the best fix because it bypasses the safer fs-safe-level legacy writer and makes stream/timing behavior fragile; a call-site fallback like #73791 or a clear fail-fast error is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep the fallback out of the raw write helper:** `src/infra/fs-pinned-write-helper.ts:216`
  - body: This branch makes python-less Linux writes use `runPinnedWriteFallback`, whose implementation follows lexical paths with `fs.mkdir`/`writeFile`/`rename` instead of the pinned dir_fd/O_NOFOLLOW walk. A workspace symlink parent can be retargeted after `fs-safe` resolves the target, so the write can clobber outside the root before post-write verification throws. Handle spawn fallback at the `fs-safe` write/copy layer or fail with a clear error.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete fs boundary regression risk by falling back from the pinned Python writer to a path-based writer on Linux when the helper cannot spawn.

Concerns:

- **[high] Path-based fallback weakens workspace-only writes:** `src/infra/fs-pinned-write-helper.ts:216`
  - body: The new spawn-error fallback calls `runPinnedWriteFallback`, which constructs paths with `path.join` and writes/renames through the live filesystem namespace. In a python-less container this can follow a symlink-parent race outside the workspace; later verification may detect the mismatch, but only after the external write has happened.
  - confidence: 0.86
