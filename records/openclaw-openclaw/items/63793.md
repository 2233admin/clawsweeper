## Summary

Keep this PR open. Current main still does not implement the PR's central ACP lifecycle fixes: `/acp spawn` still reports success after `initializeSession` without a direct post-init runtime status probe, and `AcpSessionManager` still only treats `acpx exited ...`/persistent no-session cases as recoverable stale-handle signals. The PR is a focused implementation candidate, but it still needs ACP/Gateway maintainer review and should address the review concern about resolved `dead`/`no-session` status probes before landing.

## Best Possible Solution

Keep this PR open for ACP/Gateway lifecycle review. The best path is a narrow fix on this PR: add a direct post-init runtime status/liveness probe before `/acp spawn` reports success, clean up failed post-spawn sessions, treat thrown gateway disconnects and resolved `dead`/`no-session` statuses as unhealthy, and add an explicit stale gateway-disconnect close-code allowlist for close, passive cancel, active-turn cancel, and early-turn retry while preserving negative coverage for non-stale errors like `1008`. The related #63656 command-routing fix is separate and does not supersede this lifecycle work.
