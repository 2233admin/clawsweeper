## Summary

Keep open. Current main still lacks this PR's central ACP lifecycle behavior: `/acp spawn` reports success immediately after `initializeSession` without a direct post-init runtime status probe, and `AcpSessionManager` still only treats `acpx exited ...` as the stale-handle recovery signal for close/cancel/early-turn retry. The related `/acp` command-routing fix is present on main, but it is separate and does not supersede this work.

## Best Possible Solution

Keep this PR open for ACP/Gateway lifecycle review. The best fix is narrow: add a direct post-init runtime status/liveness probe before `/acp spawn` reports success, clean up failed post-spawn sessions, treat thrown gateway disconnects and resolved `dead`/`no-session` statuses as unhealthy, and add an explicit stale gateway-disconnect close-code allowlist for close, passive cancel, active-turn cancel, and early-turn retry while preserving negative coverage for non-stale errors like `1008`.
