## Summary

Keep this issue open. The reported config intentionally allowed any Telegram DM (`dmPolicy: "open"` with `allowFrom: ["*"]`), and current docs/security audit already warn that public or multi-user DMs should use isolated `session.dmScope`. However the report is fresh, concrete, Telegram-specific, and alleges cross-user/session presentation plus a malicious command chain; there is no merged canonical fix or exact duplicate that makes it safe to close during cleanup.

## Best Possible Solution

Keep this open for maintainer/security follow-up. Ask the reporter for sanitized Telegram update payloads, gateway logs around 2026-04-28 23:00-23:15 GMT+7, `session.dmScope`, full redacted channel/tool policy config, session key shown in logs/UI, and whether Telegram Business/chat bridge features were enabled. If the behavior is confirmed as expected public-DM plus shared-main-session behavior, close later with guidance to use `dmPolicy: "allowlist"`, `session.dmScope: "per-channel-peer"`, strict tool policy/sandboxing, and `openclaw security audit`; if logs show cross-sender routing under isolated DM scope, fix it in the Telegram plugin routing/session path.
