## Summary

Keep open. Current main still has no literal `wangtao` path, but the actionable bug from the discussion remains: default workspace paths are resolved to host-absolute paths and then persisted by onboarding/setup/configure flows, so copied or synced config can carry another machine's home directory.

## Best Possible Solution

Land a focused fix like the closed #56810 attempt: when onboarding/setup/configure/migration flows persist the canonical default workspace, store `~/.openclaw/workspace` or the profile-aware portable default instead of the resolved host-absolute path, while continuing to resolve paths absolutely at runtime and preserving explicit custom absolute workspace values.

## Clownfish Work Prompt

Fix the workspace config portability bug reported in openclaw/openclaw#51429 and previously attempted in #56810. Current main has no literal hardcoded username, but onboarding/setup/configure flows can persist the canonical default workspace as a host-absolute path. Implement a small helper that detects the canonical default workspace for the current home/profile and returns a portable persisted value (`~/.openclaw/workspace`, and the matching profile-aware default when `OPENCLAW_PROFILE` is non-default). Apply it only when serializing default workspace choices in local onboarding, non-interactive onboarding, migration import, plain setup, and configure wizard. Do not change runtime absolute workspace resolution, do not rewrite explicit custom absolute workspace values, and do not introduce startup/load-time migrations. Add focused regression tests for classic onboarding, non-interactive onboarding, setup, configure wizard, onboard-config/helper behavior, and profile-aware defaults. Add one user-facing changelog Fixes bullet with proper thanks attribution to the reporter and prior PR author if the fix uses that context.
