## Summary

Keep this PR open. The direction matches the concrete bug in #74665, but the patch only normalizes the historical `sanitizeSessionHistory` path and leaves the live OpenAI Responses stream wrapper unchanged, so same-run tool continuations can still bypass the new normalization before pi-ai splits the ids.

## What This Changes

The PR adds an OpenAI Responses-specific tool-call id normalizer, exports it, wires it into `sanitizeSessionHistory`, and adds regression coverage for overlong call ids and malformed item ids.

## Best Possible Solution

Apply the same Responses id normalization to both historical replay and the live stream wrapper in `src/agents/pi-embedded-runner/run/attempt.ts`, preserving canonical `call_*|fc_*` same-model reasoning pairs while deterministically rewriting malformed or overlong ids before pi-ai payload conversion. Add coverage for the live same-run continuation path and update transcript hygiene docs for the new Responses-specific exception.

## Reproduction Assessment

Yes. #74665 includes a deterministic payload-capture repro, and dependency inspection of `@mariozechner/pi-ai@0.70.6` confirms same-model Responses ids are split directly into `function_call` and `function_call_output` fields.

## Solution Assessment

No. Normalizing before pi-ai splits ids is the right fix direction, but the current PR applies it only to `sanitizeSessionHistory`; the existing live wrapper path exists specifically because same-run tool continuations bypass that history pass.
