Current main already detects future-written config during the config snapshot read path that Doctor runs before health contributions, and the shipped newer-config guard also blocks risky gateway/service mutations with documented remediation. ## What This Changes

The PR branch adds a `doctor:version-skew` health contribution, a helper that warns on newer `meta.lastTouchedVersion`, and unit coverage for matching, newer, older, missing, and non-string values.

## Best Possible Solution

Keep the shipped shared config-version warning and future-action guard as the canonical solution; future wording refinements should improve that shared path rather than adding a second Doctor-only check.

Yes. A high-confidence source reproduction is to set `meta.lastTouchedVersion` newer than the running `VERSION` and run `openclaw doctor`; Doctor preflight reads the config snapshot, which invokes the current future-config warning path before health contributions.

## Solution Assessment

Yes. The current shared implementation is the better long-term solution because it covers Doctor plus other config-read and gateway/service mutation paths, while the PR’s Doctor-only health contribution would duplicate only one surface.
