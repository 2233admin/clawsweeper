## Summary

Keep open: this PR is a useful ACPX-focused contribution, but it should not merge as-is because the gateway change removes the current explicit-approval guard for scope upgrades and the new ACPX health fallback hides health-check exceptions without diagnostics. The existing Greptile review comments are consistent with the current main code and pairing docs.

## What This Changes

The PR wraps the ACPX startup backend health check in a defensive callback and removes the gateway WS connect guard that forces scope-upgrade device pairings to be non-silent.

## Best Possible Solution

Keep the ACPX runtime fix focused on the v0.6.1 probe/handshake failure, preserve explicit approval for scope upgrades, and add observable logging plus regression coverage around any health-check fallback.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Restore explicit approval for scope upgrades:** `src/gateway/server/ws-connection/message-handler.ts:1002-1007`
  - body: Removing the `scope-upgrade` override makes `silent` follow `allowSilentLocalPairing`, and that helper returns true for local scope-upgrade requests. A locally connected client asking for broader scopes could therefore be auto-approved, contradicting current main and the pairing docs. Keep the explicit false branch or narrow the helper before merging.
  - confidence: 0.93
- **[P2] Log health-check exceptions before treating backend as healthy:** `extensions/acpx/src/service.ts:241-243`
  - body: The new catch converts any `isHealthy()` exception into `true` without logging. The registry and availability helpers currently treat thrown health checks as unavailable, so this masks backend probe failures and can send ACP work to a broken runtime with no diagnostic trail.
  - confidence: 0.82
