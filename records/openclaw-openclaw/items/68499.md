## Summary

Keep this PR open. Current main still has only the generic task failure bucket on the task registry, tasks, status, and flows surfaces, so the branch is not obsolete; however, the proposed patch has unresolved correctness and table-formatting issues that should be fixed before merge.

## What This Changes

The PR adds recent and historical task failure counts to task summaries and updates tasks, status, flows, and targeted tests to show the split instead of one generic failure bucket.

## Best Possible Solution

Revise the PR so all recent/historical task pressure counts use the same filtered or unfiltered record set as the rows and JSON they accompany, fix the widened flows list header, make the new computed summary fields required unless there is a documented compatibility reason, and add regression coverage for cleanup-expired task rows before rerunning the targeted task/status/flow tests and normal changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Keep summary failures on the same record set:** `src/tasks/task-registry.summary.ts:51`
  - body: `summarizeTaskRecords` now derives `failures` from the cleanup-aware status snapshot while `total` and `byStatus` still count every record in `taskList`. A retained failed task whose `cleanupAfter` has passed will still appear in `byStatus.failed` but disappear from `failures`, producing inconsistent CLI/JSON aggregates.
  - confidence: 0.9
- **[P2] Align task pressure with the listed task rows:** `src/commands/tasks.ts:129`
  - body: `tasks list` still renders the full `tasks` array, but the new pressure line uses `buildTaskStatusSnapshot`, which filters records whose `cleanupAfter` has expired. Before maintenance pruning, the table can show failed rows while the pressure line reports no recent or historical failures.
  - confidence: 0.88
- **[P2] Align flow failure summary with linked tasks:** `src/commands/flows.ts:244`
  - body: `flows show` prints the new failure split from `taskSummary`, but the linked-task section still renders every record from `listTasksForFlowId`. Cleanup-expired failed tasks can therefore appear below a headline that says there are no failures, which is misleading for triage.
  - confidence: 0.86
- **[P2] Pad the flows list header to the widened task column:** `src/commands/flows.ts:105`
  - body: The row data now pads the task summary to `TASKS_PAD`, but the header still pads `Tasks` to the old width. That shifts the `Goal` column relative to the rows whenever the list is rendered.
  - confidence: 0.84
