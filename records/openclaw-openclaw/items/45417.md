## Summary

Keep this PR open. Current main does not implement the PR's intended live-tail inbound user-message behavior for TUI --session mode: the chat protocol still has no user-message state, the TUI does not handle that state, and the TUI is not subscribed to session.message transcript events. The PR also still needs maintainer review because the proposed chat.send hook is narrower than the linked Telegram/external-channel workflow and the author acknowledged that channel ingress paths need separate coverage.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The strongest fix is to cover the shared inbound/transcript path used by Telegram and other channels: either have TUI --session subscribe to sessions.messages.subscribe and render session.message user turns with dedupe/echo protection, or add a versioned user-message chat event emitted from the shared dispatch pipeline with schema, TUI handling, privacy/session scoping, and regression tests. Coordinate the result with linked #45388 and split unrelated TDZ/auth changes if they are still present in the branch.
