## Summary

Keep open. Current main at f256eeba431b5f59e6085b7c11c9c0a309e408b7 still does not implement the PR’s intended immediate live-tail of inbound user messages for TUI --session mode: there is no user-message chat protocol state, the TUI only consumes chat/chat.side_result/agent events, and it does not subscribe to existing session.message transcript events. The PR addresses a real #45388 workflow, but it still needs maintainer review or revision for the shared external-channel ingress path and protocol/privacy scoping.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The strongest path is either to have TUI --session call sessions.messages.subscribe for the active session and render session.message user turns with current-session filtering and echo/dedupe protection, or to emit a versioned user-message event from the shared dispatch/transcript pipeline used by Telegram and other channels. Update the gateway protocol schema/docs/tests, add TUI regression coverage for inbound user turns, and split unrelated TDZ/model-selection changes.
