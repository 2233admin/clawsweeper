## Summary

Keep this PR open. The success-path fallback rollback addresses a real current-main gap, but the selected-vs-runtime persistence change loses the actual fallback runtime identity on status/cost surfaces, so the patch needs one more focused fix before merge.

## What This Changes

The PR rolls back automatic fallback model overrides after successful CLI and embedded fallback attempts, threads selected-vs-runtime model persistence through reply/follow-up accounting, updates focused tests, and adds an Unreleased changelog entry.

## Best Possible Solution

Land a narrow fix that rolls back successful temporary fallback overrides while preserving separate selected-routing and actual-runtime identities for status, fallback notices, context windows, and cost accounting, with regression coverage for a selected primary that succeeds through a different fallback model.

## Reproduction Assessment

Yes. The current-main sticky behavior is reproducible from source: fallback candidates are persisted before attempts and only rolled back on classified failures/errors, while successful paths return first. The PR regression is also source-reproducible by a selected model that falls back to a different priced/runtime model and then renders `/status`.

## Solution Assessment

No. The success rollback is the right boundary, but persisting the selected model into `modelProvider`/`model` is not the best complete solution because existing status and cost code treats those fields as the active runtime identity.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve the actual runtime model for status cost:** `src/auto-reply/reply/session-usage.ts:137-138`

Keep a runtime model identity available for status/cost while storing the selected routing separately.
