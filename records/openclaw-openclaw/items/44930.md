## Summary

Keep open. The report is still valid as an issue because #73993 is an open PR that explicitly fixes this issue, and current main still lacks the PR's fetched-update durability barrier plus separate completed watermark. The existing main code has accepted-offset tracking, but it does not yet persist the highest fetched update before returning `getUpdates` results to grammY or restart from a durable completed watermark.

## What This Changes

Telegram polling can silently lose direct-chat messages when Telegram-side update offset acknowledgement gets ahead of OpenClaw completion for in-flight updates.

## Best Possible Solution

Review and land a narrow Telegram polling fix that durably records fetched accepted offsets before grammY can advance polling, separately persists the completed watermark, and emits recovery logging when accepted and completed offsets diverge; close this issue only after that implementation merges or an equivalent replacement lands.
