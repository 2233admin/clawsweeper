## Summary

Keep open. Current main still has Telegram polling state that persists a single accepted update watermark while completion, pending, and failed update state remain process-local, so it does not prove recovery for an update accepted by grammY before handler completion and then lost across a restart. The linked PR #73993 is open, unmerged, and explicitly scoped as the implementation candidate for this issue, so the issue should stay open until that PR lands or is deliberately replaced.

## What This Changes

The issue reports that Telegram direct-chat updates can be silently skipped when polling offset advancement races ahead of OpenClaw message processing, especially while another message or cron job is active.

## Best Possible Solution

Land or deliberately replace the narrow Telegram polling durability fix so fetched, accepted, in-flight, and completed update state cannot let Telegram-side offset confirmation outrun OpenClaw handler completion, while preserving existing duplicate guards and adding a regression for a two-update batch interrupted before the second update completes.
