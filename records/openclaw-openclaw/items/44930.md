## Summary

Keep open: current main still persists a single accepted Telegram update watermark and restores it as the skip cutoff, so an accepted-but-unfinished polling update can still be skipped after restart. The issue is also paired with open, unmerged PR #73993 using closing language, so the issue should remain open until that implementation or an equivalent fix lands.

## What This Changes

The issue asks the Telegram polling path to stop silently losing direct-chat messages when concurrent or long-running processing lets update offsets advance past in-flight messages.

## Best Possible Solution

Land PR #73993 or an equivalent Telegram plugin-local fix that durably separates accepted/fetched and completed offsets, restarts from the completed-safe watermark, logs recovery for accepted-but-unfinished ranges, and covers a two-update interrupted batch without weakening existing duplicate/replay guards.

## Reproduction Assessment

Yes for the restart durability race: current source and tests show an accepted update can be persisted, restored as the skip cutoff in a new process, and rejected before completion. The reported no-crash variant is still unclear without runtime logs or a narrower failing scenario.

## Solution Assessment

Yes, the best direction is a narrow Telegram plugin polling durability fix, not a core channel change. Closing the issue before the linked implementation merges would be premature.
