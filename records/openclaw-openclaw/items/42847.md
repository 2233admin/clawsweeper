## Summary

Keep this PR open. Current main still does not implement the intended Slack behavior: read operations in the active Slack plugin remain user-token-first even when `userTokenReadOnly` is false, while this PR edits the obsolete pre-plugin Slack action files. The work is plausible but needs maintainer product/security review and a rebase/port to the current plugin path.

## Best Possible Solution

Keep this PR open for maintainer review and rebase. If OpenClaw wants `userTokenReadOnly: false` to make Slack reads prefer the bot token, port the selector change to `extensions/slack/src/action-runtime.ts`, consider matching directory-read behavior if intended, add regression tests for the false/read path plus bot-denied fallback semantics, and update `docs/channels/slack.md`. If maintainers prefer the current user-token-first read model, close manually with the security/product rationale and document the workaround from #42752.

## Risks / Open Questions

- The PR cannot land as-is because it targets deleted pre-plugin files rather than `extensions/slack/src/action-runtime.ts` and related active tests/docs.
- Bot-token-first reads could expose broader Slack data for tool-supplied channel/thread/file IDs unless maintainers add an authorization boundary, explicit opt-in, or clear product policy.
- A bot-token-first selector without retry can regress workspaces where both tokens exist but only the user token can read a private channel or file.
