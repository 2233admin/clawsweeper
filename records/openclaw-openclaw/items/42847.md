## Summary

Current main still does not implement this PR's intended Slack read-token behavior. The active Slack plugin continues to prefer the configured user token for reads even when `userTokenReadOnly` is false, the tests and docs preserve that model, and this PR targets pre-plugin Slack action files that are no longer tracked. The PR remains plausible Slack plugin work, but it needs maintainer review because the requested bot-token-first read behavior has real product and security tradeoffs already raised in discussion.

## Best Possible Solution

Keep this PR open for maintainer review and rebase. If OpenClaw wants `userTokenReadOnly: false` to make the bot token primary for Slack reads, move the change into `extensions/slack/src/action-runtime.ts`, add regression coverage for the false/read path plus bot-denied fallback behavior, and update the Slack token-model docs. If maintainers prefer the current least-privilege model, close manually with that product/security rationale and document the workaround from #42752 or introduce a separate explicit config for bot-token read preference.

## Risks / Open Questions

- The PR cannot land as-is because it modifies pre-plugin Slack action files that are absent from current main.
- Changing reads to prefer the bot token when `userTokenReadOnly` is false could broaden read access for tool-supplied channel, thread, or file IDs unless maintainers also define an authorization, allowlist, or explicit opt-in policy.
- A bot-token-first read selector without retry can regress workspaces where both tokens are configured but only the user token can read a specific private channel or resource.
