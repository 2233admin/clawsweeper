## Summary

Current main still does not implement this PR's intended Slack read-token behavior. The active Slack plugin keeps read operations user-token-first, tests and docs preserve that model, and this PR targets obsolete pre-plugin Slack action paths that are not tracked on main. The requested bot-token-first read behavior remains plausible but needs maintainer review because the PR discussion raises real security and fallback tradeoffs.

## Best Possible Solution

Keep this PR open for maintainer review and rebase. If OpenClaw wants `userTokenReadOnly: false` to make Slack action reads bot-token-first, port the change to `extensions/slack/src/action-runtime.ts`, add regression coverage for the false/read path and bot-denied fallback behavior, and update the Slack token-model docs. If maintainers prefer the current least-privilege model, close manually with that product/security rationale and document the workaround from #42752 or add a separate explicit config for bot-token read preference.

## Risks / Open Questions

- The PR cannot land as-is because it edits pre-plugin Slack action files that are absent from current main.
- Changing reads to prefer the bot token when `userTokenReadOnly` is false could broaden access for tool-supplied channel, thread, or file IDs unless maintainers define an authorization check, allowlist, or explicit opt-in policy.
- A bot-token-first selector without retry can regress workspaces where both tokens are configured but only the user token can read a specific private channel or resource.
