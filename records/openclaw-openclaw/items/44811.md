## Summary

Keep this PR open. Current main still does not implement the intended stale runtime-snapshot writeback fix: the exported config write wrapper projects runtime-derived writes onto the stored runtime source snapshot before the inner writer reads the current disk file, so disk-only keys such as `session.dmScope` can still be turned into deletions. The linked #15146 report has a concrete user-facing security-config symptom, and this PR remains a focused implementation candidate rather than obsolete.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. The best path is to update or review it so runtime writeback uses the latest existing on-disk source/resolved snapshot as the projection base before computing the runtime patch, handles missing and invalid-readable disk snapshots deliberately, avoids the redundant read if practical, and lands with the `session.dmScope` stale-snapshot regression test plus the normal changed gate.

## Risks / Open Questions

- Before merge, maintainers should resolve the existing review concern that a valid-only latest-snapshot guard can still fall back to a stale runtime source for invalid-but-readable disk configs.
- Before merge, maintainers should either address or explicitly accept the double-read/TOCTOU concern in the write path by threading the already-read snapshot into the inner writer or otherwise preserving the invariant.
- Because the affected path writes user config, including security-relevant `session.dmScope`, the fix should land with targeted regression coverage for stale runtime snapshots plus normal changed-gate validation.
