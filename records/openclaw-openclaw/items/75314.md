## Summary

Keep open. Current main still sends WebChat new/reset actions through `chat.send`, the reset command handler intentionally returns only the synthetic acknowledgement without continuing to an agent run, and WebChat can persist that acknowledgement as an assistant-shaped `gateway-injected` transcript row. This is not a duplicate of #75235 because this report isolates the WebChat display/transcript contract, while #75235 centers on corrupted Anthropic replay recovery plus a related reset-ready signal.

## What This Changes

The issue asks WebChat fresh-session/reset acknowledgements to stop appearing as the only assistant-looking startup response, or to be clearly distinguished from real agent greetings.

## Best Possible Solution

Preserve the documented no-model behavior for bare chat `/new` and `/reset`, but make WebChat and persisted history distinguish reset acknowledgements from real assistant turns. The clean end state is a visible session-control/system event, plus tests proving `gateway-injected` reset acknowledgements no longer appear as the first assistant-looking conversational bubble.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: trigger Control UI/WebChat `/new`, observe it is sent through `chat.send`, the reset handler returns only the acknowledgement with `shouldContinue: false`, and the current transcript projection can store/broadcast it as an assistant-shaped `gateway-injected` row.

## Solution Assessment

Yes for the narrow repair direction, but not for changing default reset semantics. The best maintainable fix is to classify/render the acknowledgement separately from assistant model output while keeping the existing no-model reset policy unless maintainers explicitly choose a product change.
