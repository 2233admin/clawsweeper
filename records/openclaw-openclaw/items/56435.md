## Summary

Keep this draft PR open. Current main has later Feishu duplicate-final suppression for streaming cards, but it still does not implement the PR’s central in-place interactive-card update flow: no `update` interaction kind, no pending card-update registry, no card-update dispatcher context, and no Feishu card-update docs/API export. The PR remains useful but needs revision and review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open and continue review/revision against current main. The best implementation path is to add the missing `account` wiring, pass `cardUpdate` through both single-agent and broadcast dispatch, preserve Feishu size/chunking behavior for final updates, and split or clearly justify the `signal-core` plugin-SDK entrypoint change. Close only after a corrected version merges or a newer canonical PR fully ships in-place Feishu interactive-card updates.

## Risks / Open Questions

- Security/supply-chain review: the PR changes `scripts/lib/plugin-sdk-entrypoints.json` and Signal runtime exports in addition to Feishu behavior. That may be a valid build fix, but it is a public plugin-SDK surface change and should be reviewed intentionally rather than merged as incidental drift.
- Security review: the proposed pending update registry uses short `Math.random()`-based update IDs and resolves update context from synthetic message text via regex. Before landing, maintainers should confirm update IDs cannot be spoofed across chats/accounts and cannot patch the wrong Feishu card.
- Functional risk: the proposed final card patch path bypasses the existing Feishu chunking path for long final text, matching the unresolved review concern that large outputs could leave the original card stuck in processing.
