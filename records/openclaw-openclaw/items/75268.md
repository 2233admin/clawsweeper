## Summary

Keep this open. Current `main` still has the stream-parser path described in the existing ClawSweeper comment: Mistral structured thinking/text chunks can reach the OpenAI-compatible completions parser as non-string `delta.content`, then be concatenated into assistant text and later read by memory ingestion.

## What This Changes

The issue reports that Mistral thinking responses are rendered and persisted with leading `[object Object]` text in assistant replies and memory.

## Best Possible Solution

Normalize structured Mistral content chunks at the OpenAI-compatible completions stream boundary: route `thinking` chunks into hidden thinking blocks, append only `text` chunks to visible assistant text, and prevent `[object Object]` from entering stored session text or memory-ingested text.

## Reproduction Assessment

Yes. The issue has concrete versioned steps, and static current-main inspection gives a high-confidence mocked reproduction path: feed the completions parser a Mistral-style `delta.content` array with thinking/text objects and assert no `[object Object]` appears in output content.

## Solution Assessment

Yes. The narrowest maintainable fix is in the stream parser/provider-normalization boundary; filtering memory ingestion alone would leave corrupted live replies and persisted session transcripts.
