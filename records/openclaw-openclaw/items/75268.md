## Summary

Keep this open. Current main still routes Mistral OpenAI-compatible streaming content through a string-only parser path, so documented structured Mistral thinking/text chunks can still become visible assistant text and then persisted memory text.

## What This Changes

The issue asks OpenClaw to stop showing and storing `[object Object]` when Mistral thinking output is enabled.

## Best Possible Solution

Normalize structured Mistral content at the OpenAI-compatible completions stream boundary: treat `thinking` chunks as hidden thinking blocks, append only `text` chunk text to visible assistant output, and prevent structured objects from entering persisted session text or memory-ingested text.

## Reproduction Assessment

Yes. The issue has concrete current steps, and static current-main inspection gives a high-confidence mocked reproduction path: feed the completions parser a Mistral-style `delta.content` array with `thinking` and `text` objects and assert no `[object Object]` appears in assistant text.

## Solution Assessment

Yes. The narrowest maintainable fix is provider response normalization in the stream parser path; filtering memory ingestion alone would leave corrupted live replies and persisted transcripts.
