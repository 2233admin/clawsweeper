## Summary

Current main still has the provider parsing gap: Mistral routes through the OpenAI-compatible completions parser, that parser appends truthy `delta.content` as text without a runtime string check, and Mistral native reasoning can emit structured `content` chunks rather than plain strings. ([docs.mistral.ai](https://docs.mistral.ai/studio-api/conversations/reasoning/native))

## What This Changes

Prevent Mistral thinking/native reasoning chunks from appearing as `[object Object]` in agent replies, persisted sessions, and memory ingestion.

## Best Possible Solution

Normalize structured Mistral content at the OpenAI-compatible completions stream boundary: convert `thinking` chunks into hidden thinking blocks, append only `text` chunks to visible assistant text, and add regression coverage so corrupted text never reaches session storage or memory extraction.

## Reproduction Assessment

Yes. The issue has concrete steps, and a high-confidence mocked reproduction is to feed `processOpenAICompletionsStream` a Mistral-style `delta.content` array with `thinking` and `text` objects; current code will treat that non-string value as text and coerce it during concatenation.

## Solution Assessment

Yes. The narrowest maintainable fix is in the stream parser/provider-normalization boundary; filtering only memory ingestion would leave live replies and persisted session transcripts corrupted.
