## Summary

This should stay open: the current OpenAI-compatible streaming path still assumes `choice.delta.content` is already a string, while the linked Mistral native-reasoning documentation and the issue report point to structured content chunks that can be coerced into `[object Object]` before they are persisted and later indexed by memory.

## What This Changes

The issue reports that Mistral thinking responses render and persist leading `[object Object]` text in agent replies, session files, and memory.

## Best Possible Solution

Normalize Mistral structured content chunks at the OpenAI-compatible completions stream boundary, preserving final text as visible output and keeping thinking blocks out of user-visible text, then add a regression that proves `[object Object]` cannot enter stored assistant text or memory-ingested text.

## Reproduction Assessment

Yes. The issue has clear current-version steps, and static inspection shows a direct coercion path: a non-string Mistral `content` chunk reaches text concatenation without normalization.

## Solution Assessment

Yes. A narrow stream-parser regression fix is the maintainable path; fixing memory ingestion alone would leave corrupted session text and channel replies in place.

## Repair Work Prompt

Fix the Mistral thinking `[object Object]` leak reported in https://github.com/openclaw/openclaw/issues/75268, related to prior symptom report https://github.com/openclaw/openclaw/issues/70806. Start in `src/agents/openai-transport-stream.ts`: the OpenAI-compatible completions stream must not treat non-string `choice.delta.content` as visible text. Add a parser/normalizer for Mistral-style structured `content` chunks shaped like `{ type: "thinking", thinking: [{ type: "text", text: ... }] }` and `{ type: "text", text: ... }`, preserving visible final text and representing thinking as OpenClaw thinking blocks or dropping it from user-visible text according to existing provider policy. Keep the fix shape-based or narrowly provider-compatible; do not add downstream memory-only filtering and do not stringify unknown objects into visible text. Add regression coverage in `src/agents/openai-transport-stream.test.ts` proving structured Mistral content does not produce `[object Object]` and that final text is stored cleanly.
