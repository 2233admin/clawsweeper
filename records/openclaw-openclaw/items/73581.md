## Summary

Keep open. Current main partially addresses the report: memory-core managed dreaming cron startup retry is present, and gateway diagnostics can release stale lanes when no active work remains. The main requested fix is still missing because active embedded runs, reply operations, and active session-lane tasks remain warning-only by default, with no configurable active stuck-session abort threshold in config/docs/schema. The related open PRs #73243 and #39117 are implementation candidates, not proof that the issue is fixed.

## Best Possible Solution

Keep this issue open. Treat the memory-core dreaming cron startup race as fixed on current main, then review/land a focused active stuck-session recovery implementation such as #73243 or #39117, or build the same behavior in core with a configurable active-run abort threshold, safe mapping from diagnostics to active run/reply/lane state, abort or force-clear after a defensible timeout, a synthetic user-visible terminal error/final state, preserved queued-message semantics, and regression coverage for hung model, tool, delivery, WebSocket-disconnect, and gateway-restart paths.
