## Summary

Keep open. Current main partially improves the report by retrying memory-core startup cron reconciliation and adding a Codex app-server silent-turn timeout, but the broader requested stuck-session recovery is still incomplete: diagnostic heartbeat recovery does not enable active aborts, active embedded/reply/lane work remains warning-only by default, and the public config still exposes only a warning threshold.

## What This Changes

The issue requests lane-level timeout/recovery for stuck agent processing sessions and retry/defer reconciliation for memory-core dreaming cron registration after Gateway restart.

## Best Possible Solution

Keep the issue open until one canonical core recovery path lands: keep the memory-core cron retry, then add a documented configurable active stuck-session abort threshold that safely maps diagnostics to active embedded/reply/lane work, aborts or force-clears only after that threshold, preserves queued-message semantics, and emits a clear terminal user-visible error with regression coverage for hung model, tool, delivery, WebSocket-disconnect, and restart paths.

## Reproduction Assessment

Yes for the remaining code-level issue: current source and tests provide a focused reproduction that diagnostic recovery does not abort or release active embedded/reply/lane work by default. The full live trigger remains intermittent, but the issue body and corroborating comments provide enough logs to treat this as a valid reliability bug.

## Solution Assessment

No for current main as a complete solution. The memory-core race and Codex app-server silent-turn case are improved, but the requested general active-lane timeout/recovery still needs one reviewed canonical implementation, likely from #39117, #73243, or an equivalent narrow core patch.
