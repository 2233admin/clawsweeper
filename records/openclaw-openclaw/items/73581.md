## Summary

Keep this issue open. Current main partially fixes the memory-core dreaming cron race, but the broader agent-lane recovery request is still not implemented: stuck-session diagnostics emit warnings/events, while lane reset or active-run abort remains tied to explicit interrupt/abort/restart flows rather than the diagnostic watchdog.

## Best Possible Solution

Keep the issue open for the agent processing lane recovery work. The likely core fix belongs around diagnostics plus session/run ownership: when a session exceeds a configured recovery threshold, resolve sessionKey to the active embedded run, abort or mark the run failed, clear queued lane/followup work safely, and surface a synthetic error/final state so later messages can proceed. The memory-core dreaming cron half can be treated as already fixed on main by e84ebeaf/#73493 and closed separately once shipped if maintainers split the report.
