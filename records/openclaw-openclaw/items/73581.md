## Summary

Keep open. Current main partially addresses the report: memory-core managed dreaming cron startup reconciliation now retries, and diagnostics can release stale session-lane bookkeeping when no active work is present. The central processing-lane timeout/recovery request remains unresolved because the watchdog still calls recovery without active-abort enablement, while source, tests, and docs keep active embedded runs, active reply work, and active lane tasks warning-only by default.

## Best Possible Solution

Keep this issue open for the remaining core recovery work. Treat the memory-core cron startup race as fixed on current main, then review/land or close implementation candidates such as #73243 and #39117, or build an equivalent core fix with a configurable active-run recovery threshold, safe mapping from stuck diagnostics to active run/reply/lane state, abort/force-clear only after a defensible timeout, synthetic terminal error/final state, preserved queued-message semantics, and regression coverage for hung model, tool, delivery, and WebSocket-disconnect paths.
