## Summary

Keep this issue open. Current main fixes the memory-core dreaming cron startup race, but the broader agent processing lane recovery request is still not implemented: stuck-session diagnostics warn and emit telemetry, while active-run abort/reset remains tied to explicit interrupt/reset flows rather than automatic watchdog recovery. A later commenter also reported the same problem, so this is not stale cleanup material.

## Best Possible Solution

Keep this issue open for the core agent processing lane recovery work. The best fix is a configurable watchdog recovery path that resolves a stuck `sessionKey` to the active run, aborts or marks that run failed, clears or preserves queued work according to queue policy, and surfaces a synthetic terminal error/final state so later messages can proceed. Treat the memory-core dreaming cron race as fixed on main by e84ebeaf/#73493 and split or note it separately when maintainers want to close that portion.

## Risks / Open Questions

- The report combines an already-fixed memory-core plugin race with still-open core lane recovery work, so maintainers may want to split the fixed half into a separate closeable follow-up.
- Automatic stuck-lane recovery needs careful ownership checks: it must map `sessionKey` to the active embedded run, abort only the wedged run, clear queued/follow-up state safely, and emit a clear terminal error without corrupting valid long-running turns.
