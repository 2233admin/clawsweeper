## Summary

Keep open. Current main partially addresses the report: memory-core managed dreaming cron reconciliation now retries after startup, and gateway diagnostics can release stale session-lane bookkeeping. The central processing-lane timeout/recovery request remains unresolved because the watchdog still calls recovery without active-abort enablement, while source, tests, and docs explicitly keep active embedded runs, active reply work, and active lane tasks warning-only by default.

## Best Possible Solution

Keep this issue open for the remaining core recovery work. Treat the memory-core cron startup race as fixed, then either land a reviewed implementation candidate such as #73243 or #39117, or build the same behavior directly in core with a configurable active-run threshold, safe mapping from stuck diagnostics to active run/reply/lane state, abort or force-clear only after a defensible timeout, synthetic terminal error/final state, preserved queued-message semantics, and regression coverage for hung model, tool, delivery, and WebSocket-disconnect paths.
