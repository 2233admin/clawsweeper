## Summary

Keep open. Current main partially improves the report, especially by adding memory-core startup cron retry and conservative stale-lane release, but it still does not implement the requested active stuck-session timeout/recovery: diagnostics warns and can clear stale bookkeeping, while active embedded runs, reply work, and active lane tasks remain observe-only by default.

## What This Changes

The issue requests automatic recovery for stuck agent processing lanes and retry/readiness handling for memory-core managed dreaming cron registration after gateway restart.

## Best Possible Solution

Keep the memory-core cron retry on main, then land one canonical core recovery path that adds a documented configurable active stuck-session abort threshold, safely maps diagnostics to embedded/reply/lane work, aborts or force-clears only after that threshold, emits structured diagnostics plus a clear user-visible terminal error, and covers hung model, tool, delivery, WebSocket-disconnect, and restart paths.

## Reproduction Assessment

Yes for the remaining code-level issue: current source and tests show diagnostics does not abort or release active embedded/reply/lane work by default. The full live trigger is intermittent, but the issue’s logs and corroborating comments provide enough production evidence to treat it as a valid reliability bug.

## Solution Assessment

No, current main is not the complete solution because it only implements conservative stale-lane cleanup and the memory-core cron retry. The safer path is to review, revise, or land one canonical active-recovery PR such as #73243 or the relevant portion of #39117, with config/docs/schema and regression coverage.
