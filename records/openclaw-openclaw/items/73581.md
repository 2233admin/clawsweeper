## Summary

Keep open. Current main partially addresses the report by adding memory-core managed dreaming cron retry/reconciliation, but the main processing-lane timeout/recovery request is still not implemented: diagnostics still calls recovery without active-abort enablement, active embedded/reply/lane work remains warning-only by default, tests encode that conservative behavior, and public docs describe the same limitation. The related open PRs #73243 and #39117 are implementation candidates rather than proof that this issue is fixed.

## Best Possible Solution

Keep this issue open. Treat the memory-core dreaming cron startup race as fixed on current main, then review/land a focused active stuck-session recovery implementation such as #73243 or #39117, or build the same behavior in core with a configurable active-run threshold, safe mapping from diagnostics to active run/reply/lane state, abort or force-clear after a defensible timeout, a synthetic terminal user-visible error/final state, preserved queued-message semantics, and regression coverage for hung model, tool, delivery, WebSocket-disconnect, and gateway-restart paths.
