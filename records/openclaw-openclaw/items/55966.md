## Summary

Keep this issue open. Current main fixes the downstream zero-byte transcription crash with a minimum-audio-size guard, and the Telegram offset store is now atomic, but the inbound media buffer store still writes directly to the final media path. That leaves the reported zero-byte/partial artifact class unclosed, even though part of the user-facing failure path is better now. Related ClawSweeper context covers broader Telegram media fetch/proxy/polling fixes, but not this storage atomicity gap.

## Best Possible Solution

Keep the issue open and finish it with a small shared media-store fix: make `saveMediaBuffer` write to a same-directory temp file, fsync/close when feasible, rename into place only after the full write succeeds, and remove the temp file on failure. Add a focused ENOSPC/partial-write regression test in `src/media/store.test.ts`, keep the existing zero-byte transcription guard, and consider mapping Telegram media-store `ENOSPC` failures to a clearer operator-facing log/warning.
