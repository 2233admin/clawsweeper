## Summary

Keep open. Current main still enforces allow/deny at whole-tool granularity, and the discussion already treats this as the canonical tracker for action-aware message and gateway authorization. The remaining work is a core safety contract design, not cleanup-closeable stale work.

## What This Changes

Request action-level tool policy so operators can deny mutating actions such as `message:send` while keeping read/search/react actions available through the same multiplexed tool.

## Best Possible Solution

Define a maintainer-approved, typed action-aware tool-policy contract with documented grammar, wildcard behavior, precedence, and per-call enforcement for multiplexed tools, then cover `message` and `gateway` read/write splits in source, schema, docs, and regression tests.
