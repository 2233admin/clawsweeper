## Summary

Keep open. Current main still implements only whole-tool allow/deny: config and docs expose tool-name lists, the matcher and pipeline filter by `tool.name`, and `message`/`gateway` still read their `action` argument only after the tool has passed policy admission. Maintainer and prior ClawSweeper comments already treat this as the canonical tracker for the related gateway-specific reports, so this remains valid core authorization design work.

## What This Changes

The issue requests config-level action-aware tool policy so operators can deny actions such as `message:send` while preserving read/search/react actions and approved workflow sends.

## Best Possible Solution

Keep this as the canonical core tracker and define a maintainer-approved, typed action-aware tool-policy contract with documented grammar, wildcard behavior, precedence, and regression coverage for `message` and `gateway` read/write splits before changing enforcement.
