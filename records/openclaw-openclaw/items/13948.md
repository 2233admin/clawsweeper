## Summary

Current main still does not implement action-level tool allow/deny. The config type and strict runtime schema are whole-tool based, the matcher evaluates only a tool name, and the `message` and `gateway` tools still dispatch multiplexed `action` parameters after policy filtering. Related #46684 and #54260 were closed back toward this issue, so this should remain the canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The best path is a maintainer-approved, typed action-aware tool policy contract with documented grammar and wildcard behavior, clear precedence across existing policy scopes, enforcement before multiplexed actions execute, and regression tests for cases like denying `message:send` while allowing `message:read`, and denying `gateway:config.patch` while allowing `gateway:config.schema.lookup`.
