## Summary

Current main still only supports whole-tool allow/deny policy. The typed config and strict runtime schema expose `allow`, `alsoAllow`, `deny`, and `profile`; the matcher and pipeline filter available tools by tool name before execution args exist; and `message` plus `gateway` remain multiplexed tools that dispatch on an `action` parameter after policy filtering. Related #46684 and #54260 were already closed back to this as the canonical action-level tool-policy tracker, so this should stay open for maintainer design.

## What This Changes

The issue requests config-level action-aware tool policy, such as denying `message:send` or `gateway:config.patch` while preserving other actions on the same multiplexed tool.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The best path is a maintainer-approved typed action-aware tool-policy contract with documented grammar, wildcard behavior, and precedence, enforced before multiplexed actions execute, with regression coverage for denying `message:send` while allowing read/search-style message actions and denying high-risk `gateway` actions while preserving targeted read-only actions.
