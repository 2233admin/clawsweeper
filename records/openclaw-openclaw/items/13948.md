## Summary

Current main still only supports whole-tool allow/deny policy, so the requested `message:send` or `gateway:config.patch` granularity remains valid. The config types/schema expose only `allow`, `alsoAllow`, `deny`, and `profile`; the shared matcher filters by tool name; and both `message` and `gateway` are still single multiplexed tools that dispatch on an `action` argument after policy admission. Related #46684 and #54260 are already closed back to this as the canonical tracker, so this should stay open for maintainer design.

## What This Changes

The issue requests config-level action-aware tool policy so operators can deny specific actions such as `message:send` while preserving other actions on the same tool.

## Best Possible Solution

Keep this issue open as the canonical core tracker. The desired end state is a maintainer-approved, typed, documented action-aware tool-policy contract with clear grammar, wildcard behavior, and precedence, enforced before multiplexed tool actions execute and covered for `message` and `gateway` read/write splits.
