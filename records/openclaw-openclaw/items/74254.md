## Summary

Keep this PR open. Current main and the latest release still verify only the first Docker apt key fingerprint before dearmoring the full downloaded key file, while this PR is a narrow hardening patch with focused structure coverage and no blocking review finding found in the diff.

## What This Changes

The PR adds a Dockerfile guard that requires exactly one primary `pub` key in the downloaded Docker apt GPG file before fingerprint comparison and `gpg --dearmor`, plus a Dockerfile test that locks in that ordering.

## Best Possible Solution

Merge a narrow Docker apt signing-key hardening patch that rejects multi-primary-key key files before adding Docker's apt keyring, keep the focused Dockerfile ordering test, and let the linked bug close through the merged fix.

## Reproduction Assessment

Yes. A source-level reproduction is clear on current main: a constructed Docker key file with a valid first fingerprint plus an extra primary key would only have the first `fpr` compared before the whole file is dearmored; no live Docker build was run in this read-only pass.

## Solution Assessment

Yes, with maintainer sign-off. Requiring one primary `pub` key before fingerprint comparison and dearmor is a narrow maintainable fix for the reported trust gap; the closed unmerged related PR is not a current-main solution.
