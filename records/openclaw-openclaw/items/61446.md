## Summary

Keep open. Current main still lacks the hoisted sibling `@google/gemini-cli-core` credential lookup this PR adds, and the existing npm shim/bundle coverage does not prove the reported npm v7+ global layout works when the bundle fallback lacks parseable OAuth credentials. The diff is narrow and matches the bundled Google plugin ownership boundary.

## What This Changes

The PR adds sibling `gemini-cli-core` OAuth file search paths to Gemini CLI credential extraction and adds a focused mocked-filesystem regression test for npm global hoisted layouts.

## Best Possible Solution

Land this narrow Google plugin fix, or an equivalent maintainer-owned patch, so credential discovery checks the hoisted sibling `gemini-cli-core` paths while preserving the existing nested, bundle, and bounded tree fallbacks.

## Reproduction Assessment

Yes. The PR body gives a concrete npm v7+ global install layout, and current main can be checked by source inspection: with `geminiCliDir` at `.../node_modules/@google/gemini-cli`, the known-path and tree fallbacks never inspect sibling `.../node_modules/@google/gemini-cli-core` unless the bundle fallback happens to contain parseable credentials.

## Solution Assessment

Yes. The proposed change is the narrowest maintainable fix for this layout: add deterministic sibling paths inside the Google plugin credential extractor and lock it with a mocked filesystem unit test, without changing PATH discovery, auth flow behavior, or broader recursive search policy.
