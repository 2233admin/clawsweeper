## Summary

Keep this issue open. Current main still does not expose Click-to-WhatsApp ad click metadata: the WhatsApp inbound monitor extracts text, location, contact, reply, media, mentions, and sender/session fields, but it never extracts Baileys `ContextInfo.externalAdReply.ctwaClid` or source fields into the inbound message, agent context, plugin hook context, or session metadata. Baileys 7.0.0-rc.9 does expose the relevant `ExternalAdReplyInfo` fields, so this is a missing WhatsApp plugin implementation rather than an unsupported dependency. No exact ctwa/ctwa_clid PR was found in the live GitHub search, and the reporter recently bumped the issue, so stale cleanup is not appropriate.

## Best Possible Solution

Implement this in the WhatsApp plugin: extract `contextInfo.externalAdReply`/CTWA fields from the first inbound Baileys message, normalize a minimal structured payload containing `ctwaClid`, `sourceUrl`, `sourceType`, `sourceId`, and headline/title-like fields, pass it through `WebInboundMessage` into `buildWhatsAppInboundContext` as untrusted structured metadata, and add focused tests for extraction, prompt/context visibility, and any intended session metadata persistence.

## Risks / Open Questions

- The implementation should treat CTWA ad metadata as untrusted and privacy-sensitive; `ctwaClid` is an ad attribution identifier and should not become trusted command text or broadly exposed plugin-hook data without matching WhatsApp privacy rules.
- A good fix likely needs both an agent-facing context path and a session-level retrieval/storage decision; current generic session metadata only stores origin/group fields, so blindly adding fields to `MsgContext` may not satisfy the session metadata part.
