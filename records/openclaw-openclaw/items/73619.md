## Summary

Keep this issue open. Current main still resolves omitted message-tool account selection through the inherited/default account before the calling agent's binding, and open PR #74052 is the active unmerged fix path with closing syntax.

## What This Changes

Default omitted `message` tool `accountId` values to the calling agent's configured channel account binding before falling back to inherited/default account context.

## Best Possible Solution

Review and land PR #74052, or an equivalent narrow change, so shared message-action account precedence becomes explicit `accountId`, then calling-agent channel binding, then inherited/default account fallback.

## Reproduction Assessment

Yes. A focused `runMessageAction` regression can set `defaultAccountId: "parent-account"`, configure a different binding for `agentId`, omit `params.accountId`, and observe current main choose the default before checking the binding.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable fix because the bug is in shared message account precedence, not Slack-specific delivery; the implementation should stay in the wrapper/runner path with a combined precedence test.
