## Summary

Keep this issue open because current main still gives an inherited/default message account precedence over the calling agent's channel binding, and PR #74052 is the active unmerged fix with closing syntax. The issue is valid, but policy says the issue should stay open until that linked implementation merges or is closed.

## What This Changes

The issue requests that omitted `message` tool `accountId` values resolve to the calling agent's configured channel account, while preserving explicit `accountId` overrides.

## Best Possible Solution

Land PR #74052 or an equivalent narrow shared outbound fix that keeps explicit `accountId` authoritative, prefers the calling agent's channel/account binding when the tool omits `accountId`, and falls back to inherited/default account only when no binding exists.

## Reproduction Assessment

Yes. A focused `runMessageAction` regression can configure both `defaultAccountId` and a different binding for `agentId`, omit `params.accountId`, and observe current main choose the default before checking the binding.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable fix because the bug is shared message-account precedence in the wrapper/runner path, not Slack delivery itself.
