## Summary

Keep this issue open. Current main has useful account-binding fallback paths for some subagent and cron routing, but the `message` tool still turns an omitted `accountId` into the inherited `agentAccountId` before dispatch. Because the outbound runner only falls back to the calling agent's channel binding when no explicit or default account is present, the exact Slack attribution footgun described here can still exist on paths where the runtime inherited the parent/test account.

## Best Possible Solution

Add a regression test for an omitted Slack `message` tool `accountId` where the runtime/default account is the parent or authoring account but the calling agent has its own `channels.slack` binding; the send should use the calling agent's bound Slack account while explicit overrides keep working. The likely implementation belongs around `createMessageTool`/`runMessageAction` account selection: resolve the effective calling agent and channel binding before accepting inherited `deliveryContext.accountId` as the default, with matching coverage for subagent and cron/routine entry points.

## Risks / Open Questions

- Some current subagent and cron paths already resolve bound accounts correctly, so the fix should be narrow: preserve explicit `accountId`, preserve intentional same-session behavior where product wants it, and only change the omitted-account precedence that can leak an inherited parent/test account.
