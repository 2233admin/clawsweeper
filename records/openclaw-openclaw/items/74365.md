## Summary

Keep this issue open. Current main has a shared CLI progress helper plus progress indicators on several slow paths, but the request is broader than the covered commands and there are still command surfaces that perform network/install work without the shared progress wrapper. Because the report is fresh and does not name specific commands or timings, this needs maintainer UX scoping rather than conservative cleanup closure.

## What This Changes

The issue asks for loading/progress states across slow OpenClaw CLI commands so users can tell whether commands are progressing or stuck.

## Best Possible Solution

Keep the issue as a scoped CLI UX tracking item: identify the slow command families users actually hit, apply the existing progress helper consistently at those boundaries, and update targeted tests/docs so the documented long-running-command behavior matches the implementation.
