## Clownfish Work Prompt

Fix the Feishu group activation regression reported in this issue. Current behavior: `/activation mention` persists `sessionEntry.groupActivation = "mention"`, but Feishu's hard group gate still resolves `requireMention` only from `channels.feishu` config and continues dispatching non-@ group messages when config/groupPolicy allows them.

Expected boundary: keep the fix in the Feishu plugin unless a very small, backwards-compatible SDK helper is clearly needed. Resolve effective activation for Feishu group messages as session override over existing config (`groupActivation: "mention"` => require mention, `"always"` => no mention required). Preserve existing Feishu groupPolicy/open-vs-allowlist defaults, top-level/per-group `requireMention`, group admission, broadcast behavior, topic/thread routing, pending-history behavior, and command authorization. Do not add config migrations or core bundled-channel special cases.

Likely implementation path: after resolving the group session and route/session key early enough for the gate, read the current session entry safely using the existing session store path conventions or a narrow helper, then feed the effective activation into the Feishu mention gate. Add regression tests showing an open Feishu group with config/default no mention required drops an unmentioned message after the session has `groupActivation: "mention"`, and still dispatches after `groupActivation: "always"`. Keep `/activation` control-command behavior working.

Docs/changelog: this is user-facing Feishu behavior; add a single-line active-version changelog fix entry with proper Thanks attribution, and update `docs/channels/feishu.md` only if the supported `/activation` behavior needs to be documented or clarified.
