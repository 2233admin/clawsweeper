## Summary

Keep open. Current main still lacks the target-derived outbound tool context needed by forwarded exec-approval notices, and this PR is useful source work, but the PR head is not mergeable as-is because it has compile-time import gaps and incomplete runtime propagation through real outbound delivery and queued replay.

## What This Changes

The PR adds target-derived `toolContext` to forwarded exec-approval deliveries, extends outbound channel context typing, and adds regression assertions for forwarded request and resolved notices.

## Best Possible Solution

Land an adapted current-main fix that makes target-derived channel identity part of the typed outbound delivery contract, passes it into real channel adapter contexts, persists and replays it through the delivery queue, and covers immediate plus recovered forwarded approval notices.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: current main resolves the forwarded approval target but sends no `toolContext`, while message-action channel selection depends on `toolContext.currentChannelProvider` before throwing the multi-channel channel-required error.

## Solution Assessment

No. The direction is right, but this PR is not the best mergeable fix until it is adapted to current main, imports the new type correctly, forwards `toolContext` into handler creation, and preserves it through queued delivery replay.
