## Summary

Keep this PR open. Current main still keeps local Gateway CLI calls pointed at loopback while tailnet/custom Gateway binds can listen only on non-loopback addresses, and the silent close formatter still has no auth-specific hint. The PR is not merge-ready as written because review found mode-specific auth-hint problems, missing tests, and a security-policy interaction with plaintext non-loopback WebSocket URLs, but the underlying work is still real and not fully superseded on main.

## Best Possible Solution

Keep the PR open as a plausible implementation candidate, but ask for an update rather than closing it. Maintainers should choose whether #40247's server-side loopback-plus-bind approach is the canonical fix for tailnet/custom local calls; if so, land or close that path first and keep this PR focused on a tested, mode-aware auth diagnostic. Any revised client-side host-resolution approach should live in `src/gateway/connection-details.ts`, preserve the existing `ws://` security policy, update `urlSource`/diagnostics, and add focused tests in `src/gateway/call.test.ts` or the connection-details test surface.

## Risks / Open Questions

- The PR is not merge-ready without resolving the plaintext non-loopback WebSocket security-policy interaction, especially for default `gateway.tls.enabled=false` tailnet/custom setups.
- The auth hint must be mode-aware: token, password, and trusted-proxy need different guidance, and trusted-proxy should not be told to pass a token.
- The code has moved host selection into `src/gateway/connection-details.ts` on main, so the PR would need rebasing onto the current split and targeted tests before review can complete.
