## Summary

Keep this issue open. Current main still defaults to an automatic daily 4 AM session reset, does not expose a supported `session.reset.mode: "off"`, and the lazy rollover path still creates a new session when the next real message arrives after the boundary. Main has a Control UI confirmation for manual `/new`/`/reset` button sends and compaction-checkpoint restore APIs, but those do not address automatic reset warning/cancel/postpone or daily reset archive restore.

## Best Possible Solution

Keep open for maintainer/product follow-up in the session reset contract and user surfaces: add a supported manual-only/off policy or equivalent, expose the active reset policy in setup/status/config UI, add a channel-supported pre-rollover warning or confirmation where feasible, and define a restore or branch path for daily reset archives.

## Risks / Open Questions

- Automatic reset also bounds context growth and provider cost, so any off/manual-only mode should be explicit and visible rather than silently disabling safety behavior.
- A daily-reset restore/rebind feature needs clear retention, archive-discovery, and disk-budget semantics because reset transcripts are archived and later maintenance can prune them.
