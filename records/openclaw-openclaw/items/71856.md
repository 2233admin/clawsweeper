## Summary

Keep open. Current main still does not implement this PR's intended dynamic TUI startup conversation summary: the TUI connection path loads history and renders, while session preview data is only used in existing session picker flows. The PR remains a plausible implementation for the linked feature request, but it still needs maintainer review around reconnect duplication, focused tests, and the product/privacy call for showing prior-session text automatically at startup.

## Best Possible Solution

Keep this PR open for maintainer review. If the feature direction is accepted, rebase it onto current main, use the existing `sessions.list` derived-title/last-message API from the TUI startup path, add a first-connect-only guard, make the automatic prior-session preview policy explicit, add focused tests around success/empty/failure/local/reconnect cases, and update the public TUI docs if the behavior ships.

## Risks / Open Questions

- The supplied latest patch still appears to append the startup summary on every non-local `onConnected` callback, so reconnects could duplicate the summary unless guarded with `!reconnected` or equivalent first-connect state.
- Automatically displaying a previous session title or last-message preview at terminal startup may expose prior conversation text in a visible terminal; maintainers should decide whether that is acceptable by default, gated, or opt-in.
- The supplied diff does not show focused tests for success, empty results, API failure swallowing, local-mode skip, or reconnect no-duplication behavior.
- The implementation should be checked for session relevance, such as current-agent/current-session filtering, before merge so startup does not surface an unrelated recent session.
