## Summary

Keep open. Current main still leaves `openclaw tasks maintenance` scoped to task and Task Flow maintenance, while the requested conservative session-registry sweep is not wired into that command; PR #74023 is the open implementation candidate and uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` also prune stale cron run entries from `sessions.json` while preserving entries for jobs still running according to `jobs-state.json`.

## Best Possible Solution

Land PR #74023 or an equivalent focused change that adds preview/apply session-registry maintenance to `tasksMaintenanceCommand`, protects active cron jobs using durable job state, updates JSON/text output and docs, and covers the behavior in targeted command tests.

## Reproduction Assessment

Yes. A stale `agent:<id>:cron:<jobId>:run:<runId>` entry in `sessions.json` is outside the current `tasksMaintenanceCommand` apply path, because that command only invokes task-registry and Task Flow registry maintenance.

## Solution Assessment

Yes. The requested command-level sweep with `jobs-state.json` protection is the narrowest maintainable direction; current main does not implement it, so reviewing and landing the already-open focused PR is the best path.
