## Summary

Keep open. Current main still limits `openclaw tasks maintenance` to task-registry and Task Flow maintenance, while the requested session-registry sweep is represented by open, unmerged PR #74023 with closing syntax.

## What This Changes

The issue requests that `openclaw tasks maintenance --apply` also prune stale cron session-registry entries older than seven days while preserving sessions for jobs still marked running in durable job state.

## Best Possible Solution

Land PR #74023 or an equivalent focused implementation that adds a `tasks maintenance` session-registry sweep, derives protected cron run sessions from durable cron job state, reports preview/apply counts in text and JSON output, and documents the new command behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: current main’s command code never loads or updates `sessions.json`, and its JSON output has no session-registry maintenance field, so a stale cron run session cannot be pruned by `openclaw tasks maintenance --apply`.

## Solution Assessment

Yes for the requested direction, pending PR review. The narrow maintainable solution is to extend `tasks maintenance` at the command/task-maintenance boundary with a conservative protected-key guard, rather than relying on the cron timer’s existing age-only reaper.
