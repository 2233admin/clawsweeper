## Summary

Keep this PR open, but do not merge it as-is. Current main still has the FreeBSD pinned-helper identity mismatch path, while the proposed fallback route introduces a data-integrity regression for newly created targets by extending legacy pre-create semantics to FreeBSD; the PR discussion already acknowledges this tradeoff needs maintainer judgment.

## What This Changes

The PR changes `src/infra/fs-safe.ts` so `writeFileWithinRoot` and `copyFileWithinRoot` route FreeBSD through the existing legacy fallback path instead of the Python pinned-write helper.

## Best Possible Solution

Land a tested FreeBSD-compatible filesystem write/copy fix that avoids the Python-versus-Node device identity mismatch while preserving atomic new-target materialization, hardlink rejection, symlink and root-boundary checks, and post-write verification.

## Reproduction Assessment

Yes for triage: the reporter provided FreeBSD 15/ZFS jail steps and logs, and current main still sends FreeBSD through the pinned helper with strict non-Windows device comparison. I did not have a live FreeBSD jail in this read-only review.

## Solution Assessment

No, not as-is. Reusing the legacy fallback is narrow, but it changes new-target crash semantics on FreeBSD; a safer fix would normalize the pinned-helper identity flow or add a FreeBSD-safe path that preserves all-or-nothing target materialization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write; when the destination is missing, that creates the final inode before rename. If the process is killed in that window, FreeBSD can retain a zero-byte target where the pinned helper would not materialize the final path until replacement.
  - confidence: 0.84
