## Summary

Keep this PR open, but do not merge it as-is: current main still leaves FreeBSD on the pinned Python write path and strict identity comparison, while the proposed fallback route extends the legacy new-target zero-byte crash window to FreeBSD.

## What This Changes

The PR adds FreeBSD to the platform checks that send `writeFileWithinRoot` and `copyFileWithinRoot` through the existing legacy filesystem fallback in `src/infra/fs-safe.ts`.

## Best Possible Solution

Land a FreeBSD-compatible filesystem write/copy fix that avoids the Python-vs-Node device identity mismatch while preserving the desired atomic new-target behavior, symlink and hardlink defenses, root-boundary checks, and post-write verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write; when the destination is missing, that creates the final inode before rename. A crash in that window can leave a zero-byte target where the pinned helper would not materialize the final path until replace, so please normalize the pinned-helper identity flow or use a FreeBSD path that preserves new-target atomicity.
  - confidence: 0.82
