## Summary

Keep this PR open for maintainer review. Current main still sends FreeBSD writes and copies through the pinned Python helper, so the reported FreeBSD/ZFS identity mismatch is not implemented on main; however, the proposed fallback route should not merge as-is because it extends legacy new-target crash semantics to FreeBSD on a hardened filesystem path.

## What This Changes

The PR adds `process.platform === "freebsd"` to the Windows-only legacy fallback guards in `writeFileWithinRoot` and `copyFileWithinRoot`.

## Best Possible Solution

Land a tested FreeBSD-compatible write/copy fix that avoids the Python-vs-Node device identity mismatch while preserving the intended pinned-helper safety properties: atomic new-target materialization, hardlink rejection, symlink/root-boundary checks, and post-write verification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve all-or-nothing writes for new FreeBSD targets:** `src/infra/fs-safe.ts:551`
  - body: Routing FreeBSD into the legacy path calls `openWritableFileWithinRoot` before the temp write. For a missing destination, that creates the final inode before rename, so an interrupted write can leave a zero-byte target where the pinned helper would not materialize the final path until replacement. Please normalize the pinned-helper identity flow or use a FreeBSD path that keeps new-target atomicity.
  - confidence: 0.84
