## Summary

Keep open. Current main still allows stored provider credentials in `auth-profiles.json` to become runtime candidates when no matching `auth.profiles` declaration exists, and the Pi auth controller consumes those candidates without an undeclared-profile warning. Current Claude CLI auth aliasing narrows part of the Anthropic reproduction when `anthropic:claude-cli` remains declared, but it does not resolve the broader removed-declaration credential reuse reported here.

## What This Changes

The issue requests that runtime auth ignore or visibly warn on credentials left in per-agent `auth-profiles.json` after their `openclaw.json` `auth.profiles` declarations are removed, so Pi fallback cannot silently use retired provider credentials.

## Best Possible Solution

Make credential declaration authority explicit for runtime selection: either ignore stored profiles that are no longer declared for normal runtime use, or select them only with a high-visibility warning and a clear `openclaw doctor`/`models status` drift report plus non-destructive cleanup guidance. Avoid destructive startup pruning until maintainers choose the migration and recovery semantics.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide a valid stored provider credential, remove the matching `auth.profiles` declaration so no declared profile remains for that provider, and observe `resolveAuthProfileOrder` returning the stored profile that Pi then injects through the embedded auth controller.

## Solution Assessment

No. The current silent stored-profile fallback is not the best way to solve this billing/security-sensitive case; a narrow declaration allowlist or explicit warning plus doctor/status drift detection is safer than hidden runtime use or unconditional boot-time credential deletion.
