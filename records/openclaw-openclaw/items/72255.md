## Summary

Keep open. Current main still lets stored auth-profile state outrank config declarations in the shared auth-order resolver, and the Pi runtime consumes that resolver before injecting credentials into runtime auth. Doctor/auth health still summarizes stored profiles without detecting stored-but-undeclared profile drift. Related #40363 and #43945 are adjacent, but do not supersede this removed-declaration/Pi credential-selection case.

## Best Possible Solution

Keep this open and define declaration authority explicitly: runtime auth selection should ignore or loudly warn on stored profile IDs that are no longer declared when config declarations exist for that auth provider, while preserving intentional `auth.order` and safe profile-id drift repair. Add doctor/models-status detection for stored-but-undeclared profiles with a non-destructive quarantine or repair path, plus focused tests for removed Anthropic setup-token credentials, stored order, and `claude-cli` alias handling.
