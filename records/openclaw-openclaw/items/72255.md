## Summary

Keep open. Current main still permits a valid stored provider credential to be selected when there is no matching `auth.profiles` declaration, and the Pi runtime path can inject that resolved credential into runtime auth. The related stale-key and silent-fallback reports are useful context but do not supersede this removed-declaration billing path.

## What This Changes

The issue requests that runtime auth selection stop silently using credentials left in per-agent `auth-profiles.json` after their declarations are removed from `openclaw.json`, or at least warn and diagnose that drift.

## Best Possible Solution

Adopt a declaration-aware credential authority model for runtime selection: store-only provider credentials should either be ignored unless explicitly ordered or surfaced with a high-visibility warning, and `doctor` or `models status` should report stored-but-undeclared profiles with non-destructive remediation guidance.

## Reproduction Assessment

Yes. A source-level reproduction is to provide config with no matching `auth.profiles` declaration for `anthropic`, keep a valid `anthropic:*` credential in the auth store, and observe `resolveAuthProfileOrder` returning that stored profile before the Pi auth controller resolves and injects it. No live run was performed because this review was required to be read-only.

## Solution Assessment

No for the current behavior. Silent reuse is not the narrowest maintainable answer; a resolver allowlist or loud selection-time warning plus diagnostic drift reporting is safer than hidden billing through a removed declaration, while automatic secret deletion needs maintainer policy.
