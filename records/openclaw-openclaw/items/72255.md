## Summary

Keep open. Current main still allows stored per-agent auth-profile credentials to remain eligible when their `auth.profiles` declaration is absent, and the Pi embedded path consumes that resolved order without a stored-but-undeclared warning or doctor check.

## What This Changes

The issue asks OpenClaw to prevent or warn on Pi runtime selection of per-agent stored credentials whose `openclaw.json` auth-profile declarations were removed, with proactive doctor/status detection for that drift.

## Best Possible Solution

Make declaration authority explicit in resolver and Pi candidate selection: stored credentials whose profile IDs are no longer declared should be ignored or surfaced with a high-visibility warning, and `openclaw doctor`/models status should report stored-but-undeclared profiles with a safe quarantine or repair flow.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide config with no declared `anthropic:claude-set-up-token`, keep that provider credential in the auth store, and observe `resolveAuthProfileOrder` feeding it into the Pi `profileCandidates` path.

## Solution Assessment

No, the current behavior is not the best way to solve the issue because it treats the per-agent store as selectable runtime state after the user removes the declaration they edit and reason about. A resolver-level allowlist plus doctor/status detection is safer than destructive startup pruning.
