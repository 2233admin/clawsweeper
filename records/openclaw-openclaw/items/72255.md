## Summary

Keep open. Current main still has source-level paths where stored auth-profile credentials can remain selectable without a current `auth.profiles` declaration, and the embedded Pi runner passes those candidates into runtime auth injection. Existing doctor/auth health checks summarize stored profile health but do not flag stored-but-undeclared drift. Current Claude CLI aliasing narrows one exact subcase, but it does not remove the underlying orphan-credential selection risk.

## What This Changes

The issue requests that credentials removed from `openclaw.json` `auth.profiles` stop being silently usable by the Pi runtime, or at least produce visible runtime and doctor/status warnings.

## Best Possible Solution

Make declaration authority explicit in the auth resolver and Pi candidate path: ignore or loudly warn on stored profile IDs that are no longer declared when config declarations exist for that provider, add doctor/status detection for stored-but-undeclared profiles, and preserve legacy drift recovery through a deliberate migration or repair flow.

## Reproduction Assessment

Yes. A focused fixture can remove a provider profile from `auth.profiles`, leave its stored credential and/or stored auth order in the per-agent store, then verify `resolveAuthProfileOrder` feeds it to the Pi auth controller. The Claude CLI alias subcase should be validated separately against a real current-main store layout.

## Solution Assessment

No, the current implementation is not the best way to solve the issue because it lets stored runtime state remain authoritative after the user removes the declaration they reason about. A narrow resolver-level allowlist plus doctor/status warning is safer than destructive startup pruning.
