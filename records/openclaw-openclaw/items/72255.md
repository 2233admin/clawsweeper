## Summary

Current main still leaves the reported credential-authority gap open: stored per-agent auth profiles can remain runtime candidates when there is no active matching declaration, and the Pi embedded path consumes those candidates without a visible undeclared-profile guard.

## What This Changes

The issue asks OpenClaw to stop or loudly surface Pi fallback use of credentials left in `agents/<id>/agent/auth-profiles.json` after their `auth.profiles` declarations are removed from `openclaw.json`.

## Best Possible Solution

Make declaration authority explicit at credential selection time: stored profiles whose IDs are no longer declared should either be ignored for normal runtime selection or produce a high-visibility warning, while doctor/status tooling should report stored-but-undeclared profiles with a recovery or quarantine path that does not destroy intentional legacy recovery state.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to provide a config with no active matching `auth.profiles` declaration for a provider, leave a valid credential for that provider in the store, and observe `resolveAuthProfileOrder` producing that stored profile, which Pi then turns into a runtime auth candidate.

## Solution Assessment

No. The current behavior is not the best solution because hidden per-agent store state can outrank what the user removed from visible config; selection-time allowlisting or loud drift reporting is safer than silent reuse or destructive startup pruning.
