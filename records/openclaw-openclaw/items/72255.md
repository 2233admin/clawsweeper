## Summary

Keep open. Current main still lacks a declaration-authority guard for runtime auth-profile selection, so valid credentials left in the per-agent store can remain selectable outside the user-visible `auth.profiles` declarations; #40363 and #43945 are adjacent but do not cover this removed-declaration/Pi candidate path.

## What This Changes

The issue asks OpenClaw to prevent or loudly report Pi runtime use of per-agent auth-profile credentials after their `openclaw.json` `auth.profiles` declarations have been removed.

## Best Possible Solution

Make declaration authority explicit in the auth resolver and Pi candidate path: stored credentials whose profile ids are no longer declared should be ignored or loudly warned about at selection time, while `openclaw doctor`/`models status` should report stored-but-undeclared profiles and offer a safe quarantine or repair flow that preserves intentional legacy drift recovery.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure no matching declared profile for a provider, leave a valid credential for that provider in the auth store, and observe `resolveAuthProfileOrder` returning the stored profile that Pi then passes into the embedded auth controller.

## Solution Assessment

No. The current behavior is not the best solution because it lets hidden per-agent store state outrank what users removed from `openclaw.json`; a narrow resolver allowlist or selection-time warning plus doctor/status drift detection is safer than silent reuse or destructive startup pruning.
