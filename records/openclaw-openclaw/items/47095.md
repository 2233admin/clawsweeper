## Summary

Keep open. Current main still sends Discord voice STT turns directly through `agentCommandFromIngress` without the Discord text-path inbound context that computes and forwards `GroupSystemPrompt` from `channels.discord.guilds.<guild>.channels.<channel>.systemPrompt`. The related Slack issue was closed as implemented for Slack/text routing and explicitly left this Discord voice gap separate.

## Best Possible Solution

Fix this in the Discord plugin voice path by resolving the voice session channel config, reusing the existing Discord group-system-prompt helper or inbound-context builder, and passing the resulting prompt into the voice agent run as `extraSystemPrompt` or through the normal Discord inbound/reply pipeline. Add a focused Discord voice manager regression test that configures `channels.discord.guilds.<guild>.channels.<voiceChannel>.systemPrompt` and asserts the agent run receives it.

Observable bug: when `channels.discord.guilds.<guild>.channels.<voiceChannel>.systemPrompt` is configured, Discord text messages get that prompt via `GroupSystemPrompt`, but Discord voice STT turns in `extensions/discord/src/voice/manager.ts` call `agentCommandFromIngress` directly without any `extraSystemPrompt`/`GroupSystemPrompt`.

Expected fix boundary: stay in the Discord plugin owner module unless a tiny generic seam is already established and clearly needed.
