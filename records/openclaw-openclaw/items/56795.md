## Summary

Keep this PR open. Current main has a narrower Feishu interactive-card text parser for fetched quoted/replied/thread messages, but it does not implement this PR's central inbound-event behavior: fetching richer content for the current inbound interactive message, exposing raw card JSON in Feishu message context, or appending that raw card JSON to the agent context. The related merged maintainer work explicitly left the broader inbound interactive-card parsing work open.

## Best Possible Solution

Keep this PR open as an implementation candidate for the still-open inbound Feishu interactive-card parsing work in #41609. The best path is to rebase it on the current main parser, preserve `RawBody` compatibility outside interactive cards, add a typed/documented Feishu message-read request for raw card content, and keep the inbound tests that prove extracted text plus raw card JSON reach the agent context.

## Risks / Open Questions

- Security review pass: the PR adds an extra Feishu message fetch for inbound interactive events and forwards raw card JSON into agent context. That is a plausible implementation path, but maintainers should review payload size/bounds, trust labeling, and prompt-injection handling before landing.
- The PR discussion includes an unresolved backward-compatibility concern that `RawBody` could change from legacy plain text to raw JSON for non-interactive Feishu messages if `rawContent` is populated too broadly. Current main still sets `RawBody` to `agentFacingContent`, preserving the plain-text behavior.
- The PR uses an SDK type workaround for requesting richer card content. Before landing, the implementation should align with the current Feishu SDK/API parameter shape used by main and include a focused test proving the request parameter is actually sent.
