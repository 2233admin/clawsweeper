## Summary

Keep open. Current main still treats `mcp.servers` as a global registry, loads the full merged server map for embedded Pi and CLI bundle MCP paths, and only applies per-agent tool policy after MCP tools are materialized. The older per-agent MCP PRs found during review are closed unmerged, and the adjacent open MCP env-var issue does not cover pre-spawn server scoping.

## What This Changes

Add per-agent MCP server scoping so configured MCP servers can be limited to matching agents before process startup or remote connection.

## Best Possible Solution

Add an explicit, documented, backward-compatible MCP server scope that is resolved against the active agent before any stdio process is spawned or remote MCP connection is opened. Omitted scope should preserve today’s all-agents behavior, and the same filtered effective server map should feed embedded Pi runtime creation, CLI bundle MCP generation, cache fingerprints, reload cleanup, schema/help, docs, and targeted tests.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Current main builds a full effective `mcpServers` map, connects/lists tools for every configured server, and only then applies per-agent tool policy.

## Solution Assessment

Unclear until maintainers approve the public config shape. The right implementation direction is pre-spawn filtering of the effective MCP server map; relying on `tools.deny` or filtering after tool materialization is not sufficient for the reported process and least-privilege problem.
