## Summary

Keep open. Current main still has a gateway-wide MCP server registry and materializes every configured server before per-agent tool policy runs, so the reported memory and least-privilege gap remains valid.

## What This Changes

The issue requests an optional per-agent scope on `mcp.servers.*` so configured MCP servers are only started or connected for matching agents while preserving current all-agents behavior when omitted.

## Best Possible Solution

Add a backward-compatible MCP server scope resolved against the active agent before any server is spawned, connected, injected into CLI backend config, cataloged, or fingerprinted. Omitted scope should preserve the current all-agents default, and the change should update config types/schema/help/docs plus focused tests for embedded Pi, CLI bundle MCP, config reload/cache behavior, and interaction with existing tool policy.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. A config with multiple agents and any `mcp.servers` entry will feed the same merged server map into each session MCP runtime before per-agent tool policy is applied.

## Solution Assessment

Unclear until maintainers approve the public config shape. The narrow maintainable direction is pre-spawn/pre-connect filtering of the effective MCP server map; `tools.deny` or post-materialization filtering is not sufficient for the reported memory and least-privilege boundary.
