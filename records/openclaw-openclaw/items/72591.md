## Summary

Keep open. Current main still has no typed or documented per-agent MCP server scope, and the runtime/CLI paths build from the full merged `mcp.servers` map before agent tool policy can hide MCP tools. The issue is valid, but the config contract and MCP process/security boundary need maintainer approval before an automated repair should choose semantics.

## What This Changes

The issue requests an optional per-MCP-server agent scope so multi-agent gateways only start selected MCP servers for matching agents.

## Best Possible Solution

Add a backward-compatible, documented MCP server scope that filters server definitions before any stdio process is spawned, HTTP/SSE connection is opened, or downstream CLI MCP config is generated. Omitted scope should preserve the current all-agents behavior, and matching should reuse existing normalized agent/glob utilities with focused tests and docs.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. Current main merges the full `mcp.servers` map, connects/list-tools every loaded server, and applies per-agent tool policy only afterward.

## Solution Assessment

Yes in direction, with maintainer approval needed for the public contract. Pre-spawn/pre-connect filtering is the narrowest maintainable fix for both process/RAM usage and the least-privilege concern raised in the comments.
