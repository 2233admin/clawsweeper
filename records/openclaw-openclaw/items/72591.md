## Summary

Keep open. Current main still lacks a typed or documented per-agent scope for `mcp.servers`, and both embedded Pi and CLI bundle MCP paths still feed the full merged server map into runtime startup/config injection before per-agent tool policy can hide tools. The resource-use report and the personal-data guardrail comment remain valid.

## What This Changes

Request optional per-server agent scoping for `mcp.servers.*` so only matching agents spawn or connect to selected MCP servers while omitted scope preserves the current all-agent default.

## Best Possible Solution

Add a backward-compatible MCP server scope that filters the effective server map before stdio spawn, HTTP/SSE connection, tool cataloging, cache fingerprinting, and CLI backend config generation. Omitted scope should preserve today's all-agent behavior, and docs/tests should make the process boundary distinct from tool visibility policy.

## Reproduction Assessment

Not applicable as a feature request, but the current behavior gap is high-confidence from source inspection. The runtime builds a full effective `mcpServers` map, iterates every entry during catalog creation, and only applies per-agent tool policy after MCP tools are materialized.

## Solution Assessment

Yes in direction, pending maintainer approval of the public config syntax. Pre-spawn filtering is the narrowest maintainable solution for the reported RAM/process issue and the least-privilege guardrail; filtering only registered tools would not solve either concern.
