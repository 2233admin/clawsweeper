## Summary

Keep this issue open. Current main still does not expose the requested private-network opt-in for `web_fetch`; it only supports the narrower RFC2544 fake-IP proxy allowance. The same-author implementation candidate #39630 is still open and uses closing syntax for this issue, so maintainer review should resolve that PR or choose a replacement design before this issue is closed.

## Best Possible Solution

Keep this issue open while maintainers review #39630 or pick a replacement design. The strongest core path is likely a default-off `tools.web.fetch.ssrfPolicy` option with explicit dangerous naming and/or narrower hostname controls, wired through the strict schema, exported config types, generated help/labels, runtime policy forwarding, cache isolation, docs, and focused SSRF regression tests.

## Risks / Open Questions

- Any eventual private-network opt-in expands SSRF reach when enabled; it needs a strict default-off posture, explicit dangerous naming or narrow hostname controls, cache-key isolation, docs, and regression tests for direct hosts, DNS resolution, redirects, and fallback/provider paths.
- The shipped RFC2544-only support helps fake-IP proxy users but does not solve the localhost/private-LAN workflows described by the reporter and commenters.
