## Summary

Keep open. Current main still short-circuits capability provider resolution when `plugins.enabled` is false before reading the active runtime registry, and the PR's reorder matches the TTS fallback path while preserving the no-compat-loading guard. I found no blocking review findings, so this should proceed through normal maintainer review and CI rather than be closed or replaced.

## What This Changes

The PR moves the global plugins-disabled guard after active runtime registry lookups in capability provider resolution and updates the focused tests to expect active providers while skipping compat and manifest loading.

## Best Possible Solution

Land the focused resolver change after normal maintainer review and CI completion, keeping the active-runtime-registry lookup available even when plugin discovery and bundled runtime dependency repair remain disabled.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with `cfg.plugins.enabled === false`, current main returns before `resolveRuntimePluginRegistry()`, so `listSpeechProviders(cfg)` can be empty and `resolveTtsProviderOrder` falls back to only the configured primary provider.

## Solution Assessment

Yes. The PR is the narrow maintainable fix because it restores access to already-loaded providers while still preventing compat, manifest, and dependency-repair work when plugins are globally disabled.
