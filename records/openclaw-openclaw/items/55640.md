## Summary

Keep this issue open. Current main still matches the reported retry window: messaging tool text/target state is captured as pending at tool start, but it is only committed after a successful tool result, while both block-reply and final-payload suppression read only committed sent state. A failed send followed by an in-flight retry can still let the assistant text reach delivery before the retry result commits. Related #25592 is the broader channel text-leak tracker, but this issue has a focused failed-send/retry duplicate-delivery repro that remains useful.

## Best Possible Solution

Keep this issue open and fix the embedded messaging-tool lifecycle. Track same-turn pending/retried message sends separately from committed deliveries, then let both block-reply and final-payload suppression consider a pending same-target, same-text retry while still falling back to normal replies when no successful or in-flight retry exists. Add a focused regression test for failed first message send, retry start, message_end before retry result, and retry success.

## Risks / Open Questions

- A naive pending-send suppression set could hide the only user-visible reply if the retry also fails or never reaches a successful tool result.
- The fix must preserve cross-target behavior so a message tool send to another channel does not suppress the current conversation's final reply.
- The broader unphased text-leak work in #25592 may overlap the delivery boundary, so the final implementation should coordinate with that tracker rather than adding a Telegram-only special case.
