## Summary

Keep this issue open. Current main still has the failed-send/retry gap: messaging tool sends are recorded as pending at tool start, but failed results delete that pending state and only successful sends become suppression state. Both message_end block replies and final payload shaping still consult committed successful sends, not an in-flight retry, so the reported duplicate-delivery window remains actionable.

## Best Possible Solution

Keep this issue open and fix the embedded messaging-tool lifecycle. Track same-turn in-flight or retried message sends separately from committed deliveries, let both message_end block-reply suppression and final-payload suppression consider a matching in-flight retry, and preserve fallback delivery if no send succeeds. Add a focused regression test covering failed first send, retry start, message_end before retry result, and retry success.
