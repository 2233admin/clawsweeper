## Summary

Keep open. Current main supports isolated managed browser profiles and Chrome MCP existing-session attach profiles, but it does not provide the requested explicit one-shot import/clone flow from an existing user browser profile into a managed OpenClaw profile. The request is recent, concrete, and belongs in the bundled browser plugin/profile lifecycle rather than cleanup or ClawHub closure.

## Best Possible Solution

Implement this as a guarded browser plugin profile-lifecycle feature: add an explicit CLI/API import command, resolve a source existing-session/userDataDir profile, stop or reset the destination managed profile, copy allowed browser profile data into the managed OpenClaw user-data dir while excluding transient locks/runtime files, leave the source untouched, optionally relaunch the destination, and document the one-time import semantics.

## Risks / Open Questions

- A correct implementation needs explicit consent and clear docs because it copies sensitive personal browser state into a managed automation profile.
- Copying Chromium profile data can race a live browser and can preserve transient lock/runtime files unless the flow stops or verifies the destination and filters files such as Singleton* artifacts.
- This likely needs browser-plugin tests around source resolution, destination reset/stop behavior, excluded files, source untouched guarantees, and docs for one-time import versus live sync.
