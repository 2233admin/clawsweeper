## Summary

Keep this PR open for maintainer review. Current main has overlapping manifest/capability fallback for cold memory embedding provider lookup, but it still lacks this PR's direct memory CLI bootstrap and CLI-specific regression coverage, and the PR is part of an active stacked memory series rather than obsolete cleanup.

## What This Changes

The PR re-exports memory embedding provider registry helpers through the memory CLI host barrel, registers built-in memory embedding providers before memory manager creation, and adds CLI regression tests for empty and pre-populated provider registries.

## Best Possible Solution

Keep the PR open and make an explicit maintainer call on the architecture: either accept the direct memory CLI bootstrap as a supported compatibility path, or keep manifest-first capability fallback as the contract and add only the targeted CLI regression coverage needed to lock that behavior down.

## Reproduction Assessment

Yes, for the proposed regression path at unit-test level: a cold provider list before `memory status` should register the built-in `local` adapter before `getMemorySearchManager` is called. I did not run the test because this was a read-only review, but the patch and current tests show the intended reproduction shape.

## Solution Assessment

Unclear. The patch is narrow and follows the adjacent capability CLI bootstrap pattern, but current main already has a generic manifest/capability fallback for cold memory provider lookup, so maintainers should decide whether explicit CLI registration or fallback-only behavior is the preferred long-term contract.
