## Summary

Keep this PR open. The current-main schema/runtime mismatch is real, but this patch is too broad because it lowers the registered agent-facing schema for every provider and would under-advertise providers that intentionally support higher result counts.

## What This Changes

The PR changes the generic `web_search` count schema maximum from `20` to `MAX_SEARCH_COUNT` and adds a regression test asserting the generic tool schema matches that shared constant.

## Best Possible Solution

Make the agent-facing `web_search` count contract provider-aware through lightweight provider metadata, or choose a generic fallback that does not over-promise common providers while preserving configured higher-limit provider contracts. Add regression coverage for common 10-result providers and at least one higher-limit provider.

## Reproduction Assessment

Yes. Static inspection reproduces the mismatch on current main and the PR regression path: compare `createWebSearchTool()` schema, shared count normalization, and Tavily/Exa provider schemas and runtimes; no live API calls are needed.

## Solution Assessment

No. Replacing the generic maximum with `MAX_SEARCH_COUNT` is not the best fix because it resolves the common-provider mismatch by under-advertising Tavily and Exa, whose current contracts intentionally allow more than 10 results.
