## Summary

Keep open. Current main has DeepSeek V4 reasoning-content handling for the native DeepSeek provider and known bundled proxy providers, but it still lacks a generic per-model compatibility path for arbitrary OpenAI-compatible proxy providers with prefixed or differently named DeepSeek V4 model IDs.

## What This Changes

Allow a per-model compatibility setting so DeepSeek V4 `reasoning_content` replay and stripping can apply to DeepSeek models served through non-DeepSeek providers.

## Best Possible Solution

Implement a generic model-compat path, preferably reusing `models.providers.<provider>.models[].compat.thinkingFormat: "deepseek"`, that applies the shared DeepSeek V4 wrapper for OpenAI-compatible models including slash-prefixed DeepSeek V4 IDs and documents the custom-provider configuration.

## Reproduction Assessment

Yes. A focused source-level reproduction is to configure a custom `openai-completions` model with `compat.thinkingFormat: "deepseek"` and a prefixed DeepSeek V4 id, then inspect the outgoing payload and see that the generic wrapper chain never installs the shared DeepSeek repair.

## Solution Assessment

Yes, the generic per-model compatibility direction is the narrow maintainable fix. Reusing the existing `compat.thinkingFormat` field is safer than adding an overlapping `compat.reasoningFormat` field unless maintainers intentionally want a new public key.
