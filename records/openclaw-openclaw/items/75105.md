## Summary

Keep open. Current main already has the model-level `compat.thinkingFormat` surface and a shared DeepSeek V4 replay/thinking wrapper, but custom/OpenAI-compatible providers still do not get a generic path that installs that wrapper from model compat, while the native DeepSeek path remains provider and exact-id gated.

## What This Changes

Request to allow model-level DeepSeek `reasoning_content` compatibility for custom or proxy OpenAI-compatible providers so proxied DeepSeek V4 model IDs can use the same replay/thinking repair as native DeepSeek.

## Best Possible Solution

Add a generic model-compat-driven DeepSeek V4 path that reuses `compat.thinkingFormat: "deepseek"`, handles namespaced DeepSeek V4 model IDs for custom OpenAI-compatible providers, preserves provider-local exceptions such as Venice, and documents the custom provider configuration without hard-coding Requesty or another proxy.

## Reproduction Assessment

Yes. Configure a custom `openai-completions` provider with a model such as `deepseek/deepseek-v4-pro` and `compat.thinkingFormat: "deepseek"`; source inspection shows rendering reads the compat value, but the generic stream wrapper chain never installs the DeepSeek V4 request wrapper and the native DeepSeek wrapper only matches provider `deepseek` with exact IDs.

## Solution Assessment

Yes, the generic per-model compat direction is the narrow maintainable fix, but current main already has `compat.thinkingFormat`, so reusing that key is safer than adding a second overlapping `compat.reasoningFormat` key unless maintainers explicitly rename the public contract.
