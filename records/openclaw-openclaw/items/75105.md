## Summary

Keep open. Current main has DeepSeek V4 `reasoning_content` handling for the native DeepSeek provider and a couple of bundled proxy providers, but it still does not expose a generic per-model path for arbitrary OpenAI-compatible providers with prefixed or differently named DeepSeek model IDs. The request is a real core config/runtime compatibility gap rather than something already implemented or covered by the disabled-thinking follow-up.

## What This Changes

Allow a per-model compatibility setting so DeepSeek `reasoning_content` replay and stripping can apply to DeepSeek V4 models served through non-DeepSeek providers with different model IDs.

## Best Possible Solution

Implement this as a generic model-compat path rather than a provider allowlist: prefer reusing `models.providers.<provider>.models[].compat.thinkingFormat: "deepseek"` as the switch, apply the shared DeepSeek V4 wrapper for OpenAI-compatible models including slash-prefixed DeepSeek V4 IDs, and document the custom-provider configuration.

## Reproduction Assessment

Yes. A focused source-level reproduction is to configure a custom `openai-completions` model with `compat.thinkingFormat: "deepseek"` and a prefixed DeepSeek V4 ID, then verify the outgoing payload is not patched because the current DeepSeek wrapper is provider/exact-id gated and the generic wrapper chain never installs the shared DeepSeek repair.

## Solution Assessment

Yes, the generic per-model compatibility direction is the maintainable fix. Reusing the existing `compat.thinkingFormat` field is safer than adding an overlapping `compat.reasoningFormat` field unless maintainers explicitly want a new public key.
