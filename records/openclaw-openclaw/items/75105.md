## Summary

Keep open. Current main still has DeepSeek V4 reasoning_content handling only through native or known bundled provider hooks, not through a generic per-model compatibility path for arbitrary OpenAI-compatible proxy providers. The existing ClawSweeper keep-open comment remains consistent with the current tree.

## What This Changes

Allow a per-model compatibility setting so DeepSeek reasoning_content replay and stripping can apply to DeepSeek V4 models served through non-DeepSeek providers with different model IDs.

## Best Possible Solution

Implement this as a generic model-compat path, preferably reusing models.providers.<provider>.models[].compat.thinkingFormat: "deepseek", then apply the shared DeepSeek V4 wrapper for OpenAI-compatible models including slash-prefixed DeepSeek V4 IDs and document the custom-provider configuration.

## Reproduction Assessment

Yes. A focused source-level reproduction is to configure a custom openai-completions model with compat.thinkingFormat: "deepseek" and a prefixed DeepSeek V4 id, then inspect the outgoing payload and see that the generic wrapper chain never installs the shared DeepSeek repair.

## Solution Assessment

Yes, the generic per-model compatibility direction is the narrow maintainable fix. Reusing the existing compat.thinkingFormat field is safer than adding an overlapping compat.reasoningFormat field unless maintainers intentionally want a new public key.
