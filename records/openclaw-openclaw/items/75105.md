## Summary

Keep open. Current main has DeepSeek V4 reasoning_content repair for the native DeepSeek provider and a few bundled proxy providers, but arbitrary custom OpenAI-compatible providers still cannot opt into that repair per model: the generic post-plugin wrapper chain does not consult model compat, and the native DeepSeek guard remains provider/exact-id gated.

## What This Changes

Request to add per-model DeepSeek reasoning_content compatibility so proxy or custom OpenAI-compatible providers can use DeepSeek V4 models with prefixed or non-native model IDs.

## Best Possible Solution

Implement a generic model-compat path for OpenAI-compatible custom providers: when a resolved model declares DeepSeek thinking format and its model ID matches the DeepSeek V4 family, apply the existing shared DeepSeek V4 wrapper, then document and test the custom-provider configuration.

## Reproduction Assessment

Yes. A focused source-level reproduction is to configure a custom openai-completions model such as requesty/deepseek-v4-pro or deepseek/deepseek-v4-pro with compat.thinkingFormat set to deepseek, then verify current main never installs the DeepSeek V4 payload repair wrapper for that provider/model.

## Solution Assessment

Yes, the per-model compatibility direction is the maintainable fix. Reusing the existing compat.thinkingFormat surface is safer than adding an overlapping compat.reasoningFormat key unless maintainers explicitly want the new public name.
