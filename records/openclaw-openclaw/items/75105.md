## Summary

Keep open. Current main has a model-level `compat.thinkingFormat` surface and a shared DeepSeek V4 payload repair wrapper, but arbitrary custom/OpenAI-compatible providers still do not get a generic path that installs that wrapper from model compat, and the native DeepSeek path remains provider/exact-id gated.

## What This Changes

The issue requests per-model configuration to enable DeepSeek `reasoning_content` compatibility for DeepSeek V4 models served through third-party or custom OpenAI-compatible providers.

## Best Possible Solution

Add a generic model-compat-driven DeepSeek V4 path that reuses `compat.thinkingFormat: "deepseek"`, matches DeepSeek V4 IDs including provider-prefixed forms such as `deepseek/deepseek-v4-pro`, installs the shared replay wrapper for custom OpenAI-compatible providers, and documents the custom provider configuration without hard-coding Requesty or another proxy provider.

## Reproduction Assessment

Yes. Configure a custom `openai-completions` provider with a model id such as `deepseek/deepseek-v4-pro` and `compat.thinkingFormat: "deepseek"`; source inspection shows current main uses the compat value for rendering but does not install the DeepSeek V4 request wrapper in the generic stream chain.

## Solution Assessment

Partially. The model-level compat direction is the right boundary, but the narrowest maintainable implementation should reuse the existing `compat.thinkingFormat` key unless maintainers explicitly want a new public alias such as `reasoningFormat`.
