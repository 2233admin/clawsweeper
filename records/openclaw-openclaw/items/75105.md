## Summary

Keep open. Current main still has the same source-backed compatibility gap: model config can mark a custom OpenAI-compatible model with DeepSeek thinking format, but the generic stream wrapper chain does not use that compat flag to install the DeepSeek V4 replay wrapper for proxied or provider-prefixed model ids.

## What This Changes

The issue requests per-model configuration so DeepSeek V4 `reasoning_content` handling works for third-party or custom OpenAI-compatible providers with differently named model ids.

## Best Possible Solution

Add a generic model-compat-driven DeepSeek V4 path that reuses `compat.thinkingFormat: "deepseek"`, matches DeepSeek V4 Pro/Flash including provider-prefixed ids such as `deepseek/deepseek-v4-pro`, installs the shared wrapper for custom OpenAI-compatible providers, and documents the custom provider configuration without hard-coding Requesty, OpenRouter, Wisgate, or another proxy provider.

## Reproduction Assessment

Yes. Configure a custom `openai-completions` provider with a model id such as `deepseek/deepseek-v4-pro` and `compat.thinkingFormat: "deepseek"`; current source uses that compat value for rendering but never installs the DeepSeek V4 request wrapper in the generic stream chain.

## Solution Assessment

Partially. A per-model compatibility switch is the right boundary, but the narrowest maintainable solution should reuse the existing `compat.thinkingFormat` key unless maintainers explicitly want a public alias named `reasoningFormat`.
