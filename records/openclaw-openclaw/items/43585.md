## Summary

Keep open. Current main still lacks deterministic `$include` NUL-byte rejection and raw/resolved include-path length validation, so this PR is not obsolete or implemented on main. The PR remains a focused security-hardening candidate, but it should be revised before merge to address the byte-length and boundary-test review feedback.

## Best Possible Solution

Keep this PR open as the implementation candidate. Revise it to reject NUL bytes before path resolution, enforce raw and normalized/resolved include-path limits with UTF-8 byte length, make the fewer-than-4096-byte boundary explicit in errors and docs, add regression tests for NUL, raw limit, resolved limit, near-limit acceptance, and multibyte byte-limit cases, then run the targeted config include test and changed gate before maintainer merge.

## Risks / Open Questions

- Current main still lets malformed `$include` paths reach later Node/fs behavior instead of failing early with a stable include-path validation error.
- Merging the PR as-is would enforce a filesystem-style limit with UTF-16 code units rather than UTF-8 bytes, so multibyte include paths could exceed the intended byte limit without deterministic validation.
- The PR's current tests reject 4096/4097-length strings but do not prove the accepted boundary just below the limit or multibyte byte-limit behavior.
