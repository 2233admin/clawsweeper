## Summary

Keep this PR open. Current main still lacks equivalent plugin logging best-practice guidance, and the PR is a useful docs-only contribution, but it should be revised before merge because it documents `api.logger` and required logger methods as optional even though the SDK contract only makes `debug` optional. The exact `openclaw config schema` example also needs careful wording because current startup policy bypasses the config guard for that command.

## Best Possible Solution

Revise this PR and then review/merge it: add concise logging guidance to the plugin developer docs, warn against stdout-writing `console.log` in plugin registration/runtime code, and align examples with the current contract: `api.logger.info(...)`, `api.logger.warn(...)`, `api.logger.error(...)`, and `api.logger.debug?.(...)`. Keep the `openclaw config schema` mention accurate against current manifest/config-guard behavior. A docs-only change like this likely does not need a changelog entry unless maintainers want one for plugin author documentation updates.
