## Summary

Keep this issue open. Current main still has no `costCurrency` or equivalent pricing-currency contract, and the existing cost model remains USD-specific from config/schema through formatting, docs, API/session field names, CLI/chat output, Control UI, macOS, and telemetry.

## Best Possible Solution

Keep this issue open for a core pricing-currency design. Maintainers should decide whether OpenClaw stores explicit USD-only estimates, per-provider/per-model currency metadata, global display currency, conversion, or split mixed-currency summaries, then update config schema/types/generated help, pricing lookup and aggregation, transcript/session/API fields, CLI/chat/status output, Control UI/macOS displays, telemetry/docs, and tests.

## Risks / Open Questions

- A display-only currency symbol could mislead users when a session aggregates providers or models priced in different currencies unless mixed-currency behavior is defined first.
- Persisted/API/metrics names such as `estimatedCostUsd`, `costUsd`, `formatUsd`, and `openclaw_model_cost_usd_total` mean this needs compatibility planning beyond a narrow formatter option.
