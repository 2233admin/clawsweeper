## Summary

Keep this issue open. Current main still has no `costCurrency` or equivalent currency metadata/display contract; model pricing remains USD-oriented in config, runtime cost estimation, persisted/session fields, docs, tests, chat/CLI/Control UI, and macOS displays.

## Best Possible Solution

Keep this issue open for a real core pricing-currency contract. Maintainers should first choose the boundary, such as global, per-provider, per-model, or explicit USD storage plus display conversion, then update config schema/types/generated docs, runtime pricing lookup, transcript/session/API fields, CLI/chat/status output, Control UI/macOS displays, tests, and mixed-currency aggregation behavior.

## Risks / Open Questions

- A display-only symbol option could mislead users when sessions aggregate providers priced in different currencies unless the contract defines split, conversion, suppression, or explicit labeling behavior.
- Existing persisted/API names such as `estimatedCostUsd`, `costUsd`, and `formatUsd` mean this needs compatibility planning rather than a narrow formatter rename.
