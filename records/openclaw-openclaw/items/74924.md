## Summary

Keep open for maintainer review. The PR targets the right startup regression from #74896, but the current patch skips the web-fetch resolver too broadly and removes the existing inactive SecretRef diagnostic for plugin-scoped `webFetch` credentials when top-level fetch is disabled; it also overlaps with the still-open #74921 implementation.

## What This Changes

The PR changes secrets runtime fast-path detection, runtime web-fetch provider resolution, and tests so `tools.web.fetch.enabled: false` skips web-fetch provider discovery, including when plugin-scoped `webFetch` config exists.

## Best Possible Solution

Land one selected fix that makes credential-free `tools.web.fetch.enabled: false` skip web-fetch provider discovery during gateway startup, while preserving inactive SecretRef diagnostics for plugin-scoped or credential-bearing disabled fetch config; then close #74896 and close or supersede the overlapping implementation PR.

## Reproduction Assessment

Yes. #74896 provides a concrete disabled-fetch config and Linux/Docker startup trace, and current main source verifies that this config still enters secrets runtime web-fetch provider discovery before the gateway reaches later startup markers. The patch-specific regression is also visible by comparing current main's inactive warning test with this PR's deleted assertion.

## Solution Assessment

No. The fast-path direction is appropriate, but this patch's resolver-level guard is too broad because it treats top-level `tools.web.fetch.enabled: false` as a reason to skip plugin-scoped `webFetch` config diagnostics; a narrower credential-aware guard, such as the approach in #74921, is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve inactive fetch SecretRef warnings:** `src/secrets/runtime-web-tools.ts:683-686`
  - body: When `tools.web.fetch.enabled` is false but plugin-scoped `webFetch.apiKey` is still configured as a SecretRef, current main still emits `SECRETS_REF_IGNORED_INACTIVE_SURFACE` without resolving the secret. This guard skips the whole fetch branch for the `hasPluginWebFetchConfig` case and the tests remove that warning assertion, so users lose the diagnostic for an ignored credential. Keep the startup fast path for credential-free disabled fetch, but preserve the credential-bearing plugin-config warning path.
  - confidence: 0.86
