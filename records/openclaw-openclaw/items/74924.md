Current main still routes an explicitly disabled `tools.web.fetch` object through the secrets slow path and web-fetch provider discovery, while this PR is a focused implementation candidate for the linked gateway-startup regression; I did not find a blocking correctness or supply-chain issue in the proposed patch.

## What This Changes

The PR changes secrets runtime fast-path detection, web-fetch runtime provider resolution, and regression tests so `tools.web.fetch.enabled: false` skips web-fetch provider discovery.

## Best Possible Solution

Land one narrow fix that makes `tools.web.fetch.enabled: false` a reliable gateway-startup escape hatch before provider discovery, then close the linked bug and the duplicate implementation PR after the chosen patch merges.

Yes. #74896 gives a concrete Linux/Docker startup trace and config workaround, and current main source shows the disabled fetch key still forces the secrets slow path into provider discovery before the gateway reaches the bind step.

## Solution Assessment
