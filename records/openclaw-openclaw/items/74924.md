## Summary

Keep this PR open, but do not merge it as-is. The reported startup regression is valid on current main, and the fast-path direction is useful, but this branch also skips the existing inactive SecretRef diagnostic path for plugin-scoped web-fetch credentials when top-level fetch is disabled; overlapping PR #74921 appears to preserve that contract more carefully.

## What This Changes

The PR changes secrets runtime fast-path detection, runtime web-fetch provider resolution, and regression tests so `tools.web.fetch.enabled: false` skips web-fetch provider discovery.

## Best Possible Solution

Land one selected fix that makes credential-free `tools.web.fetch.enabled: false` skip web-fetch provider discovery during gateway startup while preserving inactive SecretRef diagnostics for credential-bearing plugin-scoped fetch config; then close #74896 and close or supersede the overlapping implementation PR.

## Reproduction Assessment

Yes. #74896 gives a concrete Linux/Docker startup trace and disabled-fetch config, and current main source confirms that shape still bypasses the secrets fast path and awaits web-fetch provider discovery before later gateway startup markers.

## Solution Assessment

No for this exact patch as currently shaped. The fast-path change is the right direction, but the resolver-level guard should be narrower so plugin-scoped or credential-bearing disabled fetch config can still emit inactive SecretRef warnings without resolving the secret.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve inactive fetch SecretRef warnings:** `src/secrets/runtime-web-tools.ts:683-686`

This guard skips the entire fetch branch for that case, and the test removes the warning assertion, so users lose the diagnostic for an ignored SecretRef.
