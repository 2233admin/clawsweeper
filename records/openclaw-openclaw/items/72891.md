## Summary

Keep open. Current main still implements Discord voice channels as WAV capture, STT, text agent turn, and TTS playback; Gemini Live/realtime provider support exists in adjacent Voice Call, Google Meet, and shared realtime-voice surfaces, but no Discord voice realtime schema, runtime bridge, or docs path has landed. The related #73699 discussion is an open design dependency for Discord session/transcript behavior, not a duplicate or shipped fix.

## What This Changes

The issue requests an additive Discord voice mode that can use Gemini Live or another native realtime audio provider instead of the current STT-to-text-agent-to-TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge, keeps channels.discord.voice.model as the text-pipeline LLM override, and first settles how Discord sessions, transcripts, delivery, permissions, and realtime tool/consult calls interact with #73699 before updating schema, runtime, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Static current-main verification is high confidence: the strict Discord voice schema rejects a realtime branch, and the runtime/docs show WAV capture to STT to text agent turn to TTS rather than a native realtime provider session.

## Solution Assessment

Unclear. A separate Discord voice realtime config path is plausible and matches the request, but the best maintainable fix needs maintainer direction for voice-as-I/O, transcript/delivery behavior, routing, permissions, and tool policy before implementation.
