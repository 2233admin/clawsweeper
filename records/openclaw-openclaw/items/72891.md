## Summary

Keep open. Current main still routes Discord voice through the existing STT/transcript -> text agent command -> TTS playback path, while realtime/native audio provider support is only present in adjacent voice bridge surfaces and not wired into the Discord voice configuration or manager. The related open voice-as-I/O issue #73699 is adjacent but does not supersede this request because it addresses text-session bridging rather than selecting a Gemini Live-style realtime audio backend for Discord voice.

## What This Changes

Request a Discord voice configuration path that can select a realtime/native audio provider such as Gemini Live instead of only choosing the text response model for the current STT -> LLM -> TTS pipeline.

## Best Possible Solution

Add a Discord-owned realtime voice mode that reuses the shared realtime voice bridge/provider contract, keeps `channels.discord.voice.model` as the text-pipeline response-model override, and explicitly defines transcript, permissions, session routing, and tool/context behavior for Discord before exposing Gemini Live as a selectable backend.

## Reproduction Assessment

Not applicable as a feature request. The current-main check is source-based: Discord voice config and runtime still expose only the text-mediated voice path, while realtime provider code exists outside the Discord voice path.

## Solution Assessment

Unclear. The requested direction is reasonable, but the best implementation should be decided alongside #73699 so the realtime audio backend does not bypass Discord text-session context, permissions, transcripts, or agent-tool behavior.
