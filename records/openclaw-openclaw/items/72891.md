## Summary

Keep open. Current main still implements Discord voice as capture-to-WAV, batch STT, a text agent turn, and TTS playback; the shared realtime voice provider path and Google Gemini Live provider exist elsewhere, but Discord voice has no realtime config branch, bridge session, transcript policy, or docs path. The related #73699 discussion is an open design dependency rather than a duplicate or shipped fix. Reviewed item: https://github.com/openclaw/openclaw/issues/72891

## What This Changes

Add a Discord voice configuration and runtime path that can select Gemini Live or another realtime audio provider instead of only the current STT-to-text-agent-to-TTS pipeline.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge without overloading channels.discord.voice.model, then specify how Discord voice sessions, transcripts, delivery, permissions, and consult/tool calls interact with #73699 before updating schema, types, runtime, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. Static current-main verification is high confidence: the strict schema rejects a Discord realtime branch, and the runtime/docs show Discord voice still uses STT, a text agent turn, and TTS instead of a native realtime provider session.

## Solution Assessment

Unclear. A separate channels.discord.voice.realtime or equivalent config path is plausible, but the best solution needs maintainer direction for voice-as-I/O, transcript behavior, routing, permissions, and tool policy before implementation.
