## Summary

Keep open. Current main still lacks a Discord voice realtime-provider mode: Discord voice remains the STT -> text agent turn -> TTS path the issue describes, while Gemini Live/realtime audio exists in adjacent Voice Call, Google Meet, and Talk surfaces. This should not be closed as ClawHub or duplicate because it changes the bundled Discord plugin's runtime/config semantics and is tied to the still-open #73699 architecture question.

## What This Changes

Request support for selecting an end-to-end realtime audio provider such as Gemini Live for Discord voice channels instead of only the existing STT -> text agent -> TTS pipeline.

## Best Possible Solution

Define an additive Discord plugin realtime mode that reuses the shared realtime voice provider bridge, preserves `channels.discord.voice.model` as the text-pipeline LLM override, and documents session binding, transcript delivery, permissions, and tool-consult behavior alongside #73699 before code lands.

## Reproduction Assessment

Not applicable as a feature request. Current-main verification is high confidence: schema, types, runtime, docs, and tests all show Discord voice still uses separate STT, text agent routing, and TTS rather than a realtime provider session.

## Solution Assessment

Unclear. A separate `channels.discord.voice.realtime` or equivalent mode is plausible, but the narrow maintainable fix depends on maintainer decisions for #73699 and native-audio transcript/tool semantics.
