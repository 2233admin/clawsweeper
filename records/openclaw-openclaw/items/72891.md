## Summary

Keep open. Current main still implements Discord voice as STT into a text agent turn followed by TTS playback, while Gemini Live/realtime provider support is only wired into adjacent Voice Call, Google Meet, Talk, and VoiceClaw surfaces. The open #73699 architecture discussion affects the same Discord voice session and transcript model, so this is valid feature work that still needs maintainer direction rather than cleanup closure.

## What This Changes

Add a Discord voice configuration and runtime path that can select a realtime audio provider such as Google Gemini Live instead of only selecting the text-response model used after STT.

## Best Possible Solution

Define an additive Discord voice realtime mode that reuses the shared realtime voice provider bridge without overloading channels.discord.voice.model, then specify how Discord voice sessions, transcripts, delivery, permissions, and consult/tool calls interact with #73699 before implementing schema, runtime, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. Current-main static verification is high confidence: schema, runtime, docs, tests, and targeted search all show Discord voice still uses STT, a text agent turn, and TTS rather than a native realtime provider session.

## Solution Assessment

Unclear. A separate channels.discord.voice.realtime or equivalent config path is plausible and maintainable, but the best shape depends on maintainer direction for #73699 and Discord-specific transcript, routing, permission, and tool policy.
