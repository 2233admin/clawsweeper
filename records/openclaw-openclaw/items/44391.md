## Summary

Keep this PR open. Current main still has the followup delivery bug the PR describes: the followup payload helper applies text/media messaging-tool dedupe before it checks whether the tool send targeted the originating conversation. The PR is not obsolete, but it is broader than the title because it also carries model retry, delivery serialization, and Mattermost dependency changes with unresolved behavioral and supply-chain review risks.

## Best Possible Solution

Keep this PR open, but ask for a narrowed version: put the same-target dedupe guard in `resolveFollowupDeliveryPayloads` or the followup payload path, preserve legacy dedupe when target metadata is unavailable, add followup text and media cross-target regression tests, and split model retry, delivery serialization, and Mattermost dependency changes into separate PRs.
