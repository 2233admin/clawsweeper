## Summary

Keep open. Current main still lacks the requested staged, provider-generic model migration workflow, and the issue discussion includes explicit maintainer guidance to keep it open for product follow-up. Existing pieces cover immediate default-model setting, manual fallback management, OpenRouter-specific scan/probe, and runtime failover, but not candidate staging, verification-before-promotion, old-primary preservation, or rollback.

## What This Changes

The issue requests a staged model migration workflow that can test a candidate model as fallback before promoting it to primary, preserve the old primary for rollback, and provide CLI helpers for verification and rollback.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration flow that can stage any provider/model candidate, verify it before promotion, preserve the old primary in the fallback chain, store rollback state, and document/test the resulting config transitions.
