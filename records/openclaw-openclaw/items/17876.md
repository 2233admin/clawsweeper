## Summary

Keep open. Current main still lacks the staged, provider-generic model migration workflow, and prior discussion includes explicit maintainer keep-open guidance. The existing surface supports immediate default-model changes, manual fallback editing, OpenRouter-specific scan/probe application, and runtime failover, but not candidate staging, verification-before-promotion, old-primary preservation, or rollback state.

## What This Changes

Request a staged `openclaw models` migration workflow that tests a new model as fallback, promotes it after verification, preserves the old primary, and supports rollback.

## Best Possible Solution

Add a maintainer-designed `openclaw models` migration flow that stages any provider/model candidate, verifies it against the intended agent surfaces before promotion, preserves the old primary in the fallback chain, stores rollback state, and documents/tests the resulting config transitions.

## Reproduction Assessment

Not applicable as a feature request rather than a bug report. Source and docs inspection provide a high-confidence current-main check: the requested test/migrate/promote/rollback workflow is not present.

## Solution Assessment

Unclear until maintainers define the product contract. The safest direction is a generic migration workflow built around existing set, fallback, status/probe, scan, and runtime failover primitives, coordinated with #6599 rather than a small rollback-only addition to `models set`.
