## Summary

Current main still does not expose the requested universal pre-filter raw inbound plugin hook. The typed hook catalog and public plugin API only allow known hook names, unknown typed hooks are ignored, existing message hooks run downstream of routing/agent handling, and WhatsApp still drops denied inbound messages before downstream hook emission. The related PR #39240 is useful prior art, but it is closed and unmerged.

## Best Possible Solution

Keep this open for maintainer API design. A good path would add an observe-only, privacy-aware pre-filter inbound contract to the plugin/channel SDK, wire participating channels before allowlist/sender/mention checks only behind explicit opt-in, document the privacy model, and add contract plus channel-ordering tests. If maintainers decide ambient raw observation is out of scope, document that policy and point users to the narrower current message_received alternatives.

## Risks / Open Questions

- A pre-filter raw inbound observer would expose message content, phone numbers, sender identifiers, group ids, and metadata before current access control and mention-gating protections; any implementation needs explicit operator opt-in, privacy docs, and security review.
- The API shape is unresolved: this could be a universal typed hook, a channel-scoped observe sink, or a narrower per-channel contract. Treating current message_received opt-in as equivalent would miss the issue's pre-filter requirement.
- A channel-specific implementation should avoid creating a private bundled-plugin backdoor; the plugin/channel SDK contract needs to be explicit if third-party plugins are expected to use it.
