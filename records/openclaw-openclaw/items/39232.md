## Summary

Current main still lacks the requested universal pre-filter raw inbound plugin hook. The existing typed hook catalog exposes `inbound_claim`, `message_received`, and `before_dispatch`, but no `on_raw_inbound`, `onRawInbound`, or `before_channel_filter`; unknown typed hooks are rejected. WhatsApp now has an explicit `message_received` privacy opt-in, but it fires after group allowlist/mention gating, so it does not satisfy the passive pre-filter observation use case. The related PR #39240 remains closed and unmerged, so it is useful prior art rather than a superseding implementation.

## Best Possible Solution

Keep this open for maintainer API and privacy design. If accepted, add an observe-only, explicitly opt-in pre-filter inbound contract to the plugin/channel SDK, wire participating channels before allowlist/sender/mention checks, document the privacy/security model, and add contract plus channel-ordering tests. If maintainers decide ambient raw observation is out of scope, document that policy and point users to the narrower `message_received` alternatives.

## Risks / Open Questions

- A pre-filter observer would expose message content, phone numbers, sender identifiers, group ids, and metadata before current allowlist and mention-gating protections; it needs explicit operator opt-in, privacy docs, and security review.
- A channel-specific shortcut would risk creating a private bundled-plugin bypass. If this is accepted, the contract should be explicit in the plugin/channel SDK and usable by third-party plugins.
- The current WhatsApp `message_received` opt-in is easy to confuse with the requested behavior, but it fires after group gating and does not observe denied or unmentioned raw inbound messages.
- The related implementation attempt is closed and unmerged, so maintainers still need to decide the API shape before implementation.
