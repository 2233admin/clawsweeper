## Summary

Keep this PR open: current main still lacks a persistent `openclaw tunnel up/down/status` command, but the proposed patch is not merge-ready because it is stale against the current descriptor-based CLI registration and has concrete PID/process-management and startup-readiness bugs.

## What This Changes

The PR adds a new `openclaw tunnel up/down/status` CLI, docs page/nav entry, and detached SSH process plus PID-file state management for remote gateway port forwards.

## Best Possible Solution

Refresh the feature onto the current descriptor-based CLI registration, reuse or extend the existing SSH tunnel helper for target parsing/readiness, persist enough ownership metadata to manage only OpenClaw-started SSH processes, and add focused lifecycle tests plus public docs for the persistent tunnel command.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Verify PID ownership before signaling stored PIDs:** `src/cli/tunnel-cli.ts:265`
  - body: `tunnel down` sends SIGTERM/SIGKILL to whatever process currently has the stored numeric PID. If the SSH process exits and the OS reuses that PID, this can terminate an unrelated user process and `status` will also report the stale PID as active. Store and verify process ownership metadata before signaling.
  - confidence: 0.94
- **[P1] Require the requested port to be free before success:** `src/cli/tunnel-cli.ts:203-205`
  - body: The startup loop treats any bound local port as proof that the new SSH tunnel is ready. If another process already owns the port, SSH can still be briefly alive before `ExitOnForwardFailure` exits, so the command can report success and write stale state. Check port availability before spawning or race listener readiness against child exit using the existing helper semantics.
  - confidence: 0.92
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:34`
  - body: `writeTunnelState` writes to `STATE_DIR/tunnel.pid.json` without ensuring the directory exists. On a fresh profile or new `OPENCLAW_STATE_DIR`, this can throw after the detached SSH process starts, leaving a tunnel running that `down` and `status` cannot manage.
  - confidence: 0.95
- **[P3] Propagate unexpected port-probe errors:** `src/cli/tunnel-cli.ts:55`
  - body: `isPortBound` converts every non-`EADDRINUSE` listen error into `false`, which makes OS errors indistinguishable from a free port and can produce a generic timeout instead of the real failure. Surface unexpected errors or reuse the existing connect-based readiness helper.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds detached SSH process management and currently signals stored PIDs without proving they still belong to the OpenClaw-started tunnel.

Concerns:

- **[medium] Unverified PID signaling can kill unrelated processes:** `src/cli/tunnel-cli.ts:265`
  - body: The PID file stores only a numeric PID, and `tunnel down` signals that PID without verifying it still identifies the SSH tunnel started by OpenClaw. PID reuse can make this terminate an unrelated same-user process, so the persistent state needs ownership verification before SIGTERM/SIGKILL.
  - confidence: 0.93
