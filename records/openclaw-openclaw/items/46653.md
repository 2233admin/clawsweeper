## Summary

Keep open. Current main does not implement the persistent managed tunnel command, and the PR still needs a branch refresh plus process-control hardening before it is safe to merge.

## What This Changes

The PR adds a top-level `openclaw tunnel up/down/status` CLI, stores tunnel PID state, registers the command, and adds a CLI reference page plus docs navigation entry.

## Best Possible Solution

Refresh the feature onto current main, register `tunnel` through the descriptor catalog and current lazy registration helpers, reuse or extend the existing SSH tunnel readiness helper, persist enough process identity to stop only OpenClaw-started tunnels, and add focused CLI lifecycle tests plus updated remote-access docs.

## Reproduction Assessment

Yes. Current-main absence is reproducible by inspecting the command descriptor catalog, docs navigation, and remote access docs; the PR defects are reproducible from the missing `STATE_DIR` creation, bound-port readiness loop, and stale PID kill path in the added file.

## Solution Assessment

No. A managed tunnel CLI is a reasonable direction, but this patch is not the safest implementation until it is rebased onto the current CLI architecture and hardened around readiness and PID ownership.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P2] Register tunnel in the current command catalog:** `src/cli/program/register.subclis.ts:168-176`
  - body: Current main discovers root commands from `subcli-descriptors.ts` and loads them through `register.subclis-core.ts`. Adding `tunnel` only to this older wrapper path will miss the maintained help/completion/lazy registration flow after the branch is refreshed.
  - confidence: 0.87
- **[P1] Verify PID ownership before killing stored tunnel PIDs:** `src/cli/tunnel-cli.ts:291`
  - body: `tunnel down` signals whatever process currently has the stored numeric PID. If the SSH tunnel exits and the OS reuses that PID, this can terminate an unrelated same-user process; persist and verify ownership metadata before SIGTERM/SIGKILL.
  - confidence: 0.94
- **[P1] Reject occupied ports before reporting tunnel success:** `src/cli/tunnel-cli.ts:222-224`
  - body: The readiness loop treats any listener on the requested port as proof that the new SSH child succeeded. If another process already owns the port, SSH can still be alive briefly before `ExitOnForwardFailure` exits, causing a false success and stale state.
  - confidence: 0.92
- **[P2] Create the state directory before writing tunnel state:** `src/cli/tunnel-cli.ts:39`
  - body: `writeTunnelState` writes under `STATE_DIR` without ensuring the directory exists. On a fresh profile or new state dir, `tunnel up` can start SSH and then throw `ENOENT`, leaving an unmanaged detached tunnel.
  - confidence: 0.95
- **[P3] Surface unexpected port-probe errors:** `src/cli/tunnel-cli.ts:62-63`
  - body: `isPortBound` collapses every listen error other than `EADDRINUSE` into `false`, so permission or OS errors become a generic startup timeout. Propagate unexpected errors or reuse the existing connect-based readiness helper.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff adds detached SSH process management and currently stops stored PIDs without verifying they still belong to an OpenClaw-started tunnel.

Concerns:

- **[medium] Unverified PID signaling can stop unrelated processes:** `src/cli/tunnel-cli.ts:291`
  - body: The PID file stores only a numeric PID, and `tunnel down` sends SIGTERM/SIGKILL through `killPid` without checking command identity, process start time, or another durable ownership marker. PID reuse can make this terminate an unrelated same-user process.
  - confidence: 0.93
