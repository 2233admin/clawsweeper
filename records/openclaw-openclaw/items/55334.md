## Summary

Keep this open. Current main has real bounded-growth mitigation for session stores, cron run sessions, and disk budgets, but the central reported problem is still present: retained session rows still persist full `skillsSnapshot` and `systemPromptReport` payloads inline, and the latest discussion reports the same retained-row bloat on v2026.4.26.

## What This Changes

The issue asks OpenClaw to stop `sessions.json` from growing into an OOM/lock-latency source by pruning ephemeral cron/subagent rows, deduplicating inline skill/system-prompt snapshots, and cleaning orphan transcript/archive artifacts.

## Best Possible Solution

Keep this as the broad session-store bloat tracker and design a backward-compatible static-payload storage path: retain the current pruning/capping/reaper mitigations, move skill snapshots and system-prompt reports out of per-session rows by hash/reference or recomputation, compact or migrate legacy rows, and add coverage proving many cron/subagent/Telegram sessions no longer duplicate large static payloads while old sessions still load.
