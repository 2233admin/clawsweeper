## Summary

Keep open. Current main still has the reported Active Memory classifier gap: Telegram DM topic sessions can be routed as `agent:main:main:thread:{chatId}:{topicId}`, but `resolveChatType` still checks the unnormalized key and only treats an unthreaded three-part main key as `direct`, so the hook skips before recall or transcript persistence.

## What This Changes

Active Memory should treat Telegram DM topic session keys like `agent:main:main:thread:{chatId}:{topicId}` as direct-message sessions instead of silently skipping them.

## Best Possible Solution

Fix this inside the Active Memory plugin by normalizing the generic `:thread:` suffix before chat-type classification, then add a focused regression test for `agent:main:main:thread:{chatId}:{topicId}` with Telegram provider/channel context. Leave Telegram session-key generation and core routing semantics unchanged.

## Reproduction Assessment

Yes. Static current-main reproduction is high confidence: a Telegram DM topic test proves the key shape, and the Active Memory classifier can be traced to `undefined` for that same shape before the hook returns early.

## Solution Assessment

Yes for the suffix-stripping classifier fix; it is narrower and safer than treating all unknown provider/channel session keys as direct. The broad fallback suggested in the issue could accidentally enable Active Memory on future non-direct key shapes.
