## Summary

Keep this issue open. Current main still implements and documents the existing `workspaceAccess` contract: `rw` selects the agent workspace for `/workspace`, non-`rw` marks managed mounts read-only, `ro` split layouts can expose the agent workspace at `/agent`, and both local and remote filesystem bridges reject writes unless `workspaceAccess === "rw"`. The requested change is a broader sandbox/security semantics decision, not covered by the narrower writable-`none` work in #37634 or #46300.

## What This Changes

Refactor `agents.defaults.sandbox.workspaceAccess` semantics so sandbox mode always uses the session sandbox at `/workspace`, never exposes the main workspace implicitly, and separates mount target selection from read/write permissions.

## Best Possible Solution

Keep this open for explicit sandbox/security maintainer review. If maintainers accept the new semantics, implement it as a coordinated contract change across sandbox layout resolution, Docker/browser/SSH/OpenShell mount builders, FS bridge write gates, embedded sandbox metadata, memory/bootstrap behavior, tests, config/status surfaces, and public sandbox docs; treat #37634 and #46300 as narrower writable-`none` work, not replacements for this broader decision.
