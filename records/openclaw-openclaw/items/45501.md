## Summary

Keep open. Current main still lacks a `session.resetPrompt` or equivalent override, and both bare `/new` and `/reset` paths still feed the hardcoded startup text into the next model turn. The discussion includes concrete Ollama/gemma4 impact tied to the exact wording, and the related greeting-model/suppression and reset-archive ideas are adjacent but do not supersede this narrower config request.

## What This Changes

Add a global and per-agent `session.resetPrompt` config option that replaces the bare `/new`/`/reset` startup prompt while preserving the current default when unset.

## Best Possible Solution

Keep this as the canonical narrow config feature: add a full replacement prompt string with fallback to the existing default, support root and per-agent precedence, wire it into both gateway and auto-reply bare reset/new prompt construction, and update schema/help/generated config docs/tests/changelog without expanding into reset backup or greeting-model policy.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction path: bare `/new` or `/reset` without a trailing message still flows through `buildBareSessionResetPrompt()`, while the strict root and per-agent schemas reject the requested config shape.

## Solution Assessment

Yes, if maintainers accept the public config key. A full replacement string with current behavior as fallback is the narrowest maintainable fix for the reported wording problem; reset backups and configurable greeting model/suppression should remain separate work.
