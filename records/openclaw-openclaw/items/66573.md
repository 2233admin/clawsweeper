## Summary

Keep open. Current `main` still has the reported mismatch: `sessions_history` accepts and forwards oversized limits, while `chat.history` rejects values above 1000 before the handler cap can apply. A prior comment said this was fixed, but the current source still contradicts that, and open PR #66650 remains the focused implementation candidate.

## What This Changes

The issue reports that `sessions_history` can call `chat.history` with `limit: 2000`, exceeding the Gateway maximum of 1000 and silently losing session context after `INVALID_REQUEST`.

## Best Possible Solution

Land PR #66650 or an equivalent narrow fix that caps `sessions_history.limit` at 1000 before calling `chat.history`, exposes the same maximum in the tool schema, and adds regression coverage for oversized caller-supplied limits.
