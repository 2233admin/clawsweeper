Current main still treats Linux env-configured version-manager roots as part of the expected minimal service PATH, so the PR's central PATH hardening is not implemented; the branch is useful but stale and dirty against current main, with a later security hardening in the same code path to preserve during rebase.

## What This Changes

This PR changes Linux daemon minimal service PATH assembly and tests to exclude nvm, fnm, Volta, asdf, and Bun roots while keeping stable user-bin, npm, pnpm, and Nix paths; macOS version-manager behavior remains in scope.

## Best Possible Solution

Rebase the PR onto current main and adapt the same Linux policy change inside the current service-env helpers, preserving workspace PATH injection protections and existing-dir handling; add/update focused service-env and service-audit assertions for Linux env-configured version-manager roots.

## Solution Assessment

Yes, with current-main adaptation. Removing Linux version-manager roots at the minimal-PATH source is narrower than special-casing audit output, but the branch cannot be applied as-is until it is rebased over the newer security and path-normalization helpers.
