## Summary

Keep this PR open for normal maintainer review. Current main still routes unknown macOS Talk providers to local system speech, while this PR narrowly changes that path to use Gateway `talk.speak` for non-local providers and adds focused regression coverage; I found no blocking functional or security issue in the diff.

## What This Changes

The PR adds a macOS Talk playback plan that sends unsupported local Talk providers through Gateway `talk.speak`, forwards configured voice IDs for remote providers, and updates the Talk runtime speech-plan test for OpenAI/custom providers.

## Best Possible Solution

Land this PR or an equivalent narrow macOS runtime change that makes remote/custom Talk providers use Gateway `talk.speak` as the primary playback path, keeps explicit `system` and `mlx` local behavior unchanged, and leaves provider-selection UI as a separate product decision if still desired.

## Reproduction Assessment

Yes. A code-level reproduction is high confidence: on current main, `TalkModeRuntime.playbackPlan(provider: "openai", apiKey: nil, voiceId: "onyx")` reaches the default branch and returns `.systemVoiceOnly`, so no `talk.speak` call is attempted for that provider.

## Solution Assessment

Yes for the routing bug. The PR uses the existing macOS `playGatewayTalkSpeak` helper and gateway RPC contract instead of adding a new API, while preserving the direct ElevenLabs, MLX, and system-voice paths.
