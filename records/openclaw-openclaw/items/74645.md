## Summary

Keep open. This is a valid, narrow implementation PR for an unfixed macOS Talk routing bug: current main still sends unknown/non-local Talk providers to system speech, while the PR routes them through the existing Gateway `talk.speak` path and has no blocking review findings.

## What This Changes

The PR changes macOS Talk playback planning so non-local providers use Gateway `talk.speak`, carries the configured voice for remote providers, and updates the focused Swift playback-plan test.

## Best Possible Solution

Land this PR or an equivalent narrow macOS runtime change that makes remote/custom Talk providers use Gateway `talk.speak` as the primary playback path, while keeping explicit `system`, `mlx`, and direct ElevenLabs paths unchanged and tracking provider-selection UI separately if maintainers still want it.

## Solution Assessment

Yes. The PR uses the existing macOS `playGatewayTalkSpeak` helper and existing Gateway RPC contract instead of adding a new API, while preserving local-provider behavior and focused test coverage.
