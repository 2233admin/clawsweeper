## Summary

Keep this open. Current main has active runtime-registry reuse for some plugin-tool paths, but the provider-auth lookup still rebuilds manifest-backed candidate and evidence maps when repeated tool creation calls `resolveEnvApiKey` without injected maps, matching the reported CPU-profile shape.

## What This Changes

The issue reports a 2026.4.27 gateway performance regression where repeated tool creation synchronously rebuilds plugin registry and provider-auth env-var metadata, causing multi-second event-loop stalls.

## Best Possible Solution

Thread explicit, current-flow provider auth candidate/evidence maps through tool construction and model auth checks, preferably derived from the active gateway plugin metadata snapshot or manifest registry, while preserving immediate freshness when config, env, workspace plugin trust, or installed plugin inventory changes.

## Reproduction Assessment

Yes. The issue has concrete production reproduction steps and CPU-profile fingerprints against 2026.4.27, and current-main source still shows the same repeated lookup shape; this review did not run a live gateway stress replay.

## Solution Assessment

Unclear because there is no proposed patch. The likely best fix is narrow request- or active-runtime-scoped reuse of provider auth maps and active registry metadata, not a broad hidden plugin metadata cache or wall-clock TTL.
