## Summary

Keep open. Current main has some active plugin-registry reuse for gateway-bindable plugin tools, but the provider auth/env-var path still performs manifest-backed registry loading with `preferPersisted: false` on parameterized lookups used by tool construction, and there is no current-main test or release proof that the reported 7-second Gateway stalls are gone. The follow-up comments add useful regression evidence around new `providerRequest` manifest metadata and a second macOS install showing similar stalls.

## What This Changes

The issue requests eliminating Gateway event-loop stalls after `v2026.4.27` by avoiding repeated synchronous plugin registry rebuilds and provider-auth env-var resolution during `createOpenClawTools` / `createOpenClawCodingTools` passes.

## Best Possible Solution

Implement a narrow hot-path repair that reuses an active/request-scoped plugin metadata snapshot during tool creation and memoizes provider auth candidate/evidence maps for stable config/workspace/env inputs, while preserving plugin metadata freshness and workspace trust rules. Add regression coverage for repeated `createOpenClawTools` / `createOpenClawCodingTools` calls so media/provider readiness checks do not re-read manifests or rebuild the derived plugin registry each pass.

## Reproduction Assessment

Yes. The issue provides a high-confidence production reproduction path and CPU-profile fingerprint, and current code inspection shows the same tool-construction and provider-env lookup surfaces still exist; I did not run a live Gateway replay in this read-only review.

## Solution Assessment

Unclear because there is no proposed patch yet. The best direction is a scoped cache/snapshot handoff for the tool-creation pass and provider auth lookup maps, not a broad persistent plugin metadata cache that would violate freshness contracts.
