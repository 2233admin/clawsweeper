## Summary

Keep this issue open. Current main has some active-registry reuse around plugin tools, but the reported hot surfaces still exist: tool construction still resolves plugin tools, provider env-var resolution can still rebuild manifest-backed provider maps with `preferPersisted: false`, and there is no current-main fix or regression proof that the 7-second gateway stalls are gone.

## What This Changes

The issue reports gateway event-loop stalls after `2026.4.27`, attributed to repeated synchronous plugin-registry and provider-auth env-var resolution during tool creation.

## Best Possible Solution

Keep the issue open and fix the hot path by making tool creation reuse the already-active gateway plugin registry or an explicit request/startup snapshot, and by memoizing provider auth env-var maps for stable config/workspace/env inputs without violating plugin metadata freshness rules.

## Reproduction Assessment

Yes, we have a high-confidence reported reproduction path and profile fingerprint: install the affected release, enable multiple plugins/providers, drive tool-heavy Telegram traffic, and inspect gateway event-loop logs plus `.cpuprofile` flame graphs. I did not run a live gateway replay in this read-only review.

## Solution Assessment

Unclear as a code fix because there is no proposed patch yet. The best direction is a narrow hot-path repair with regression tests for repeated tool creation, not a broad loader rewrite or unconditional metadata cache.

Keep the change narrow: repeated `createOpenClawTools` / `createOpenClawCodingTools` calls during gateway agent runs should not synchronously rebuild the installed plugin registry or re-resolve manifest-backed provider auth env-var maps when config/workspace/env inputs are stable. Prefer passing/reusing an active gateway plugin registry, explicit snapshot, `PluginLookUpTable`, or request/startup-owned manifest data. Do not add a broad persistent metadata cache that violates `src/plugins/AGENTS.md`; preserve plugin disablement, workspace plugin trust, SecretRef semantics, and current extension/plugin API contracts. Changelog only if the fix changes user-visible release behavior; otherwise keep it test/internal.
