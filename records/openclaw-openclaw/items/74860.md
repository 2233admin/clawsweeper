## Summary

Keep this issue open. Current main has partial active-registry reuse for gateway-bound plugin tools, but the reported hot surfaces are still plausible: tool creation still enters plugin-tool resolution, provider auth env-var lookup still loads manifest-backed provider data with persisted registry reads disabled, and the tests I found do not prove repeated tool creation avoids manifest/registry rebuilds under the reported configuration.

## What This Changes

The issue reports a 2026.4.27 gateway performance regression where repeated tool creation synchronously reloads plugin registry metadata and provider auth env-var data, causing multi-second event-loop stalls and gateway restarts.

## Best Possible Solution

Keep the issue open and fix the hot path by reusing the active gateway plugin registry or an explicit startup/request snapshot during tool creation, and by memoizing provider auth env-var maps for stable config/workspace/env inputs without weakening plugin metadata freshness or workspace-plugin trust rules.

## Reproduction Assessment

Yes. The issue gives a high-confidence production reproduction path and profile fingerprint: install 2026.4.27, enable multiple plugins/providers, drive tool-heavy Telegram traffic, then inspect gateway event-loop logs and cpuprofiles for the named frames; I did not run a live gateway replay in this read-only pass.

## Solution Assessment

Unclear as a code fix because there is no proposed patch. The best direction is a narrow hot-path repair with regression tests for repeated tool creation and provider-env lookup reuse, not a broad plugin loader rewrite.
