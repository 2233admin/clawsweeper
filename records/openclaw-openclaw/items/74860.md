## Summary

Keep this issue open. Current main has partial mitigation for plugin-tool registry reuse and v2026.4.29 documents runtime-deps mirror memoization, but the source still has a repeated provider-auth metadata path: `resolveEnvApiKey` derives candidate and evidence maps when callers do not inject them, and those helpers still load manifest-backed provider metadata with persisted reads disabled. The production CPU-profile report and added macOS data point are concrete enough to treat this as a valid remaining performance regression rather than stale cleanup.

## What This Changes

The issue asks OpenClaw to stop gateway event-loop stalls caused by repeated tool-construction passes rebuilding plugin registry metadata and provider-auth env-var metadata.

## Best Possible Solution

Fix the hot path with scoped reuse: tool construction should reuse the active gateway/plugin registry when the load context is compatible, and provider-auth candidate/evidence maps should be computed once per stable config/workspace/environment snapshot and threaded into repeated auth/model/tool checks without adding a stale global plugin metadata cache.

## Reproduction Assessment

Yes. The issue provides a high-confidence production reproduction path and CPU-profile fingerprint for 2026.4.27, plus an independent macOS data point; this review confirmed current-main source paths that can still repeat the implicated work, but did not run a live gateway replay.

## Solution Assessment

Unclear because there is no proposed patch. The reporter's memoization/instrumentation direction is appropriate, but the maintainable solution should be narrow, request- or snapshot-scoped, and consistent with the plugin registry freshness rules.
