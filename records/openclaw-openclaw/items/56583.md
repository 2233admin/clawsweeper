## Summary

Keep this PR open. Current main has a newer, different overload retry mechanism: a fixed opt-in `auth.cooldowns.overloadedBackoffMs` delay defaulting to `0` ms plus a same-provider rotation cap. It does not implement this PR’s central requested behavior: an embedded Pi exponential overload-failover policy with a 30s default ceiling and `agents.defaults.embeddedPi.overloadBackoffMaxMs` config. The PR also needs maintainer/author cleanup because its branch mixes the agent backoff change with live-test HOME/profile/auth staging work that drew unresolved review concerns.

## Best Possible Solution

Rework the PR against current main if maintainers still want a longer default or exponential overload policy. The likely path is to adapt the change to the shipped `auth.cooldowns` surface, decide explicitly whether the default should remain `0` or become nonzero, update schema/docs/tests, and split out or drop the unrelated live-test HOME/profile/auth staging changes. If the `2026.3.31` no-delay design is intentional, a maintainer can close this PR manually as superseded by that product direction.

## Risks / Open Questions

- Changing the default overload delay from the currently documented `0 ms` to 30s would be a user-visible product behavior change: less churn under sustained overload, but slower fallback/recovery when another provider or profile could answer quickly.
- Adding `agents.defaults.embeddedPi.overloadBackoffMaxMs` now would create a second public config surface beside the shipped `auth.cooldowns.overloadedBackoffMs` and `auth.cooldowns.overloadedProfileRotations` knobs unless maintainers intentionally migrate/alias the setting.
- The PR branch mixes overload policy changes with live-test HOME/profile/auth staging changes. That test-harness work touches credential/config handling and should be split or reviewed independently before any landing decision.
