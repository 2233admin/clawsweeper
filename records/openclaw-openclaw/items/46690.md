## Summary

Keep open. Current main has documented canonical skill roots and some path-escape hardening, but it still does not detect arbitrary skill-like directories such as `~/my-skill/SKILL.md` that sit outside those roots. The `openclaw skills list/check` path only renders status for skills discovered from the active workspace/config roots, and `package_skill.py` still accepts any provided skill folder without warning that it is outside an OpenClaw-loaded location.

## Best Possible Solution

Keep this open for maintainer product judgment, then implement a narrow diagnostic if approved: likely an explicit `openclaw skills check` or doctor/status warning that compares shallow, user-approved candidate locations against the documented skill roots and reports `SKILL.md` directories that will not be loaded. Avoid broad gateway-startup home scans unless maintainers explicitly want that behavior, and update docs/tests around the final canonical-root definition.
