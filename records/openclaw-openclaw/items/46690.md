## Summary

Keep open. Current main still documents and enumerates only known skill roots, and `openclaw skills check`, status construction, and the security audit do not discover arbitrary orphan `SKILL.md` directories outside those roots. The related open PR #63312 is adjacent because it concerns symlinked skills inside recognized `.agents/skills` roots, not orphan skill directories outside every load root.

## What This Changes

The issue asks OpenClaw to warn about skill-like directories created outside canonical skill roots and to steer scaffolding/packaging workflows toward workspace skills paths.

## Best Possible Solution

Add a bounded diagnostic that compares candidate `SKILL.md` directories against the documented load roots and reports folders that will not be loaded. The likely maintainable destination is `openclaw skills check`, doctor, or security audit after maintainers decide whether scanning is opt-in, configured, or limited to safe candidate roots; docs and targeted tests should make the canonical-root behavior explicit.

## Reproduction Assessment

Yes. A valid `$HOME/my-new-skill/SKILL.md` outside the documented roots will not be returned by `loadWorkspaceSkillEntries`, and `openclaw skills check` renders only the resulting discovered status report.

## Solution Assessment

Unclear. The requested broad home-directory or startup scan would catch the example, but a bounded explicit diagnostic is safer unless maintainers approve implicit filesystem scanning.
