## Summary

Keep open. Current main still has the TUI-specific TLS trust gap described in the report: the TUI resolves the local WSS gateway URL and auth, but the resolved connection has no TLS fingerprint field and the constructed GatewayClient never receives one. Status/probe already has the scoped local TLS runtime fingerprint path, and the directly related TUI fix PR #22453 was closed unmerged, so there is no implemented fix to close against.

## What This Changes

Request that `openclaw tui` load and pass the configured local gateway TLS certificate fingerprint to GatewayClient for local WSS gateways, matching the working status/probe behavior.

## Best Possible Solution

Implement the TUI trust handoff in src/tui/gateway-chat.ts: add an optional tlsFingerprint to the resolved connection, load the local gateway TLS runtime only for configured local wss:// targets with gateway.tls.enabled === true and no URL override or configured remote URL, pass it into GatewayClient, and add focused tests in src/tui/gateway-chat.test.ts for propagation, unavailable-runtime fallback, URL override, and remote cases. Prior PR #22453 is useful source work but needs adaptation to current main.
