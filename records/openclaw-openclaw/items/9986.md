## Summary

Keep this issue open. Current main still treats context overflow as active-model overflow recovery: compact, optionally truncate oversized tool results, retry the same provider/model, and then return a blocked `context_overflow` result. The model fallback path still explicitly excludes context overflow, and the related closed issues cover adjacent overload/output/graceful-overflow/default-chain behavior rather than the requested smaller-primary to larger-context fallback routing.

## Best Possible Solution

Keep this issue open until maintainers either explicitly decide and document that context overflow remains compaction-only, or implement a guarded fallback path that runs only after compaction and oversized-tool-result truncation are exhausted, filters fallback candidates by sufficient context headroom, preserves replay safety, logs a model fallback event, and adds regression coverage for a smaller primary model falling back to a larger-context configured model.

## Risks / Open Questions

- Automatically switching models after context overflow can replay a prompt after partial side effects unless it is gated by replay/liveness metadata and side-effect detection.
- A naive implementation could select a fallback with a smaller or unknown context window and make the failure worse; candidate filtering needs sufficient context-headroom checks.
- The current docs and config surface expose fallbacks but do not clearly document that context overflow is compaction-only today, so operators can reasonably expect the fallback chain to help here.
