## Summary

Keep open. Current main has mitigations around queue modes, stop/abort signaling, and a 120s cap for `process.poll`, but the reported core behavior remains: `process.poll` receives the run abort signal and then ignores it while sleeping, so stop/steer/new-message paths still depend on the pending tool call returning. The latest issue discussion includes a fresh production confirmation from 2026-04-29, and the KeepMyClaw comment is useful recovery context rather than a native OpenClaw fix.

## Best Possible Solution

Keep this issue open and implement native abort-aware long-poll behavior. The narrow first step is to make `process.poll` observe the combined tool/run `AbortSignal` while waiting, using an abortable sleep/helper and returning or throwing a clear abort result. Then add regression coverage for a long `process.poll(timeout=120000)` interrupted by `chat.abort`/Stop and by channel-style inbound messages, including the `queueEmbeddedPiMessage` non-streaming path and deterministic queued follow-up ordering.

## Risks / Open Questions

- Abort-aware `process.poll` needs careful transcript and lifecycle handling so an aborted tool call records a deterministic abort result instead of leaving a half-written run state.
- Changing inbound behavior while a tool is pending can affect all channel surfaces, so the policy should be explicit for default queueing, configured interrupt mode, `sessions.steer`, `sessions.abort`, and follow-up ordering.
- The embedded `not_streaming` path is easy to overlook because existing stop/abort tests prove the signal is raised, but not that every long-running tool observes it.
