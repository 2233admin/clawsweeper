## Summary

Close as superseded: current main fixes the concrete `process.poll` cancellation freeze with an abort-aware, 30-second-capped wait, while the broader framework-level tool interruption and steer semantics are already tracked by the open design issue #65223.

## What This Changes

The issue requests native interruptibility for long `process.poll` waits so inbound user messages or Stop do not leave the session unresponsive.

## Best Possible Solution

Close this issue after linking #65223 for the broader framework-level interruption work, and keep the focused main-branch `process.poll` cap plus abort-aware wait as the product behavior for the concrete reported freeze.

## Reproduction Assessment

Yes. The high-confidence reproduction is to start a background process, call `process.poll(timeout=120000)`, then abort the run while the poll loop is sleeping; current main has focused unit coverage for the abort path and the long-timeout clamp.

## Solution Assessment

Yes for this cleanup. The narrow `process.poll` fix is the maintainable owner-local change, while all-tools framework racing and steer-aware tool yielding belong in the open design tracker #65223 rather than keeping this older satellite issue open.
