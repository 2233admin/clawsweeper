## Summary

Keep open. Current main still leaves `process.poll` abort-unaware: the tool executor receives `_signal` but the poll branch waits in a plain `setTimeout` loop until the process exits or the 120s capped deadline expires. The run-level abort signal is already passed into tools, and queue/stop paths still depend on the active tool returning. Open PR #73892 appears to be the focused implementation candidate, but it is not merged, so this issue should stay open for review/landing rather than be closed.

## Best Possible Solution

Keep this issue open and review/land #73892 or an equivalent focused fix. The best implementation is to make `process.poll` observe the combined tool/run `AbortSignal` while waiting, return promptly with a clear running/aborted result, preserve the background child process unless an explicit kill is requested, and add regression coverage for an aborted long poll plus the relevant Stop/inbound-message queue behavior.
