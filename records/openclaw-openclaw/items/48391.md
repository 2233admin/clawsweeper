## Summary

Keep this PR open. Current main still lacks the PR's intended cron read-model fields and richer `cron.run` acknowledgment, and the PR is not superseded by a shipped implementation. The discussion is useful: bot review flagged the optimistic queued-vs-running acknowledgment, and the author acknowledged that tradeoff while addressing the writable-schema concern in the second commit.

## Best Possible Solution

Keep this PR open for normal maintainer review. The best path is to land a reviewed variant that adds separate cron read-state/read-job types, projects both `list()` and `listPage()` through that read model, keeps derived fields out of writable cron update state, updates gateway/client/docs/tests, and resolves the queued-versus-persisted semantics of the `cron.run` in-progress acknowledgment.

## Risks / Open Questions

- The PR changes Gateway protocol/read response shape, so schema, docs, UI, Swift/client types, and tests need to stay additive and synchronized before merge.
- The proposed immediate `cron.run` acknowledgment reports `currentStatus: "running"` before queued work necessarily persists `runningAtMs`; maintainers should settle, document, or rename that semantic.
- Derived read-only fields must remain excluded from `CronJobPatchSchema` so clients cannot persist bogus running/completed status.
