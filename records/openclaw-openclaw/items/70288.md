## Summary

Keep open. Current main still returns `queuedFinal: false` for the plugin-owned `inbound_claim` handled path, and existing tests still assert that behavior. The linked bug/PR discussion is concrete and maps to the Telegram silent-reply fallback path, so this is not cleanup-ready. The PR may need maintainer review because the public Telegram dispatch path also finalizes `queuedFinal` based on `counts.final > 0`, which could require an integration-level adjustment or test beyond the one-line change.

## Best Possible Solution

Keep the PR open and ask for a small maintainer review/update: preserve an explicit async plugin-claim response signal through the provider dispatch boundary, update the direct dispatch tests, and add an integration test around `dispatchReplyWithBufferedBlockDispatcher` or Telegram silent-reply fallback so a handled plugin-owned `inbound_claim` does not trigger the fallback while declined/error/no-handler paths still do.
