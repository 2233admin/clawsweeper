## Summary

Keep open. Current main still has the reported plugin-bound handled path returning `queuedFinal: false`, and Telegram still uses `!queuedFinal` plus no recorded delivery to send the silent-reply fallback. The PR is directionally relevant, but it likely needs more work than the current one-line source change because the real Telegram path finalizes dispatch results by requiring a nonzero final count, which would turn an async plugin claim with zero immediate final replies back into `queuedFinal: false`.

## Best Possible Solution

Keep this PR open, but revise it before merge. The best fix is to preserve the distinction between a plugin that has claimed and will answer asynchronously versus a run that produced no visible response, through the full `dispatchReplyWithBufferedBlockDispatcher` path used by Telegram. Update the plugin-bound handled branch, the dispatch finalization/result contract as needed, direct dispatch tests, wrapper-level tests, and Telegram fallback coverage while keeping declined, error, missing-plugin, and no-handler paths on the fallback behavior.
