## Summary

Keep open: the hardening goal for #74234 is valid, but this PR is not mergeable as written. The Dockerfile guard counts `fpr` records, while Docker's current Debian apt key has one primary key, one subkey, and two fingerprint records, so opt-in Docker CLI image builds would fail; the branch also deletes many unrelated Unreleased changelog entries.

## What This Changes

The branch updates `Dockerfile` to reject Docker apt key files unless exactly one `fpr` record is found before fingerprint comparison, and rewrites the top of `CHANGELOG.md` to add a Docker/security fix entry while deleting many existing Unreleased entries.

## Best Possible Solution

Use a narrow Dockerfile hardening that counts exactly one primary `pub` key before fingerprint comparison and before `gpg --dearmor`, preserves Docker's legitimate subkey-bearing key, restores unrelated changelog entries, and keeps or adds focused Dockerfile test coverage. #74254 is already shaped closer to that end state if maintainers prefer one canonical implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Count primary keys instead of fingerprints:** `Dockerfile:248-253`
  - body: `gpg --with-colons` emits an `fpr` row for the primary key and for each subkey. Docker's current Debian apt key has one `pub`, one `sub`, and two `fpr` rows, so this guard rejects the official key whenever `OPENCLAW_INSTALL_DOCKER_CLI` is set. Count `pub` records or armor blocks before comparing the primary fingerprint.
  - confidence: 0.97
- **[P2] Restore the unrelated changelog deletions:** `CHANGELOG.md:9-35`
  - body: This patch removes many existing Unreleased bullets that are unrelated to the Docker key fix. Merging it would drop release notes for current main work; the branch should be rebased and add only the Docker/security entry.
  - confidence: 0.93
