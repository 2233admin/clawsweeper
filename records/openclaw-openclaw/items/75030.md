## Summary

Keep open. Current main still has the OpenAI Codex OAuth compatibility gap described by the linked issue, while this PR is an active, narrow implementation candidate with no blocking review finding from the read-only diff/source pass.

## What This Changes

The PR branch adds OpenAI Codex OAuth alias normalization during persisted auth-profile parsing, adds a metadata-only mode-mismatch hint to missing-auth errors, and adds regression tests for both behaviors.

## Best Possible Solution

Land this narrow auth compatibility PR, or an equivalent patch, after maintainer review and green required checks; keep the diagnostic limited to profile/provider/mode metadata and let the linked #47964 close through the PR after merge.

## Reproduction Assessment

Yes. A high-confidence source-traced reproduction is an openai-codex OAuth profile containing only alias fields such as access_token, refreshToken, and expires_at; current main leaves those unnormalized and PI conversion rejects them. A second reproduction is a valid stored OAuth profile paired with auth.profiles.<id>.mode = "api_key", which current ordering filters before the generic missing-key error.

## Solution Assessment

Yes. Normalizing known legacy aliases at persisted-profile parsing and appending a redacted mode-mismatch hint at the final missing-auth error is the narrowest maintainable solution I found, and the latest PR head also handles provider auth aliases for the diagnostic path.
