## Summary

Keep open. Current main still lacks both the masked-profile diagnostic and the latest branch's legacy OpenAI Codex OAuth alias normalization, while the PR diff is a focused auth-layer implementation with regression coverage and no concrete blocking review finding found.

## What This Changes

This PR normalizes legacy OpenAI Codex OAuth field aliases on auth-profile load and adds a metadata-only missing-auth hint when stale config mode masks a stored OAuth profile, with regression tests.

## Best Possible Solution

Land a small auth-layer compatibility fix after maintainer review and green checks, keeping canonical credential fields authoritative, accepted legacy aliases narrow, and all diagnostics limited to provider/profile/mode metadata.

## Reproduction Assessment

Yes. On current main, a stored openai-codex OAuth profile paired with auth.profiles.<id>.mode set to api_key is filtered as mode_mismatch and then falls through to the generic missing-API-key error; an alias-only OAuth profile also remains unusable because the loader does not populate canonical access/refresh/expires fields.

## Solution Assessment

Yes, with maintainer policy confirmation. The latest diff targets the two observed subcases in the auth loader and final error path without changing credential precedence or exposing token values; the safer alternative would be to split alias normalization from the diagnostic if maintainers want separate review tracks.
