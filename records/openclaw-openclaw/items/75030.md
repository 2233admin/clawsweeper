## Summary

Keep open. Current main still has the two auth-profile compatibility gaps described in #47964, and this PR is a relevant implementation candidate that addresses them with narrow loader normalization, a metadata-only diagnostic, and targeted regression coverage. The patch is broadly correct, with one low-priority diagnostic consistency issue around provider auth aliases for maintainer review.

## What This Changes

The PR normalizes legacy OpenAI Codex OAuth alias fields on auth-profile load and adds a metadata-only missing-auth hint plus tests for stale `mode=api_key` config masking a stored OAuth profile.

## Best Possible Solution

Land the narrow auth compatibility fix once maintainer review and CI are satisfied, keeping diagnostics secret-free and either addressing or explicitly accepting the low-priority provider-auth-alias diagnostic gap before closing #47964.

## Reproduction Assessment

Yes. On current main, a stored `openai-codex:default` OAuth profile paired with `auth.profiles.openai-codex:default.mode = "api_key"` is filtered as `mode_mismatch` and then falls through to a generic missing-API-key error; an alias-only OAuth profile also remains unusable because loader and PI conversion only honor canonical OAuth fields.

## Solution Assessment

Yes for the central direction. Normalizing a small set of legacy OpenAI Codex OAuth aliases on load and appending a metadata-only final diagnostic is the narrow maintainable fix, though the diagnostic should use the same provider auth alias resolver as eligibility for full consistency.
