## Summary

Keep open. Current main intentionally removed the direct node shell wrapper and routes node shell execution through `exec host=node`, so this is still a live feature/product request rather than implemented cleanup. Restoring direct CLI remote shell execution would need maintainer product and security judgment.

## What This Changes

The issue requests restoring or replacing the removed `openclaw nodes run` CLI workflow so shell commands can be executed directly and batch-style on paired nodes.

## Best Possible Solution

If this product direction is accepted, add a first-class one-shot CLI path that reuses the existing `exec host=node` resolver and approval pipeline, with explicit node selection, predictable timeout/output behavior, and tests. Avoid restoring the old duplicated raw `nodes.run` wrapper.

## Reproduction Assessment

Yes. Current main has a focused test proving `nodes run` is not registered, and source/docs show `nodes invoke --command system.run` is blocked, so the reported direct CLI workflow is absent on the reviewed surface.

## Solution Assessment

No, not if the proposed solution means restoring the old raw wrapper. The safer maintainable direction would be a new CLI surface layered on the existing `exec host=node` implementation so shell execution keeps the same approval and binding guarantees.
