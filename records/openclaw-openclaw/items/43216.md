## Summary

Keep this issue open. Current main has duplicate heartbeat suppression back in place, but the plugin-channel heartbeat fallback gap remains: `resolveHeartbeatDeliveryTarget()` can recover a stored plugin-style session route and still drop it to `channel: "none", reason: "no-target"` when strict outbound plugin resolution cannot recognize the channel from an empty/unbootstrapped registry. Related PR #46203 was closed unmerged, so it does not supersede the remaining work.

## Best Possible Solution

Keep this issue open and narrow it to the remaining plugin-channel heartbeat fallback. The likely fix belongs in `src/infra/outbound/targets.ts` or the outbound channel-resolution seam: when `target: "last"` recovers a stored plugin route from session metadata, preserve or safely bootstrap that route when outbound normalization fails only because the plugin registry is empty/unbootstrapped. Leave the restored duplicate suppression intact and add unmocked regression coverage for `resolveHeartbeatDeliveryTarget()` with an empty active plugin registry.
