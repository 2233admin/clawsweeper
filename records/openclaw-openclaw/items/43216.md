## Summary

Keep this issue open. Current main has restored duplicate heartbeat suppression, but the plugin-channel fallback gap remains: `target: "last"` can recover a stored plugin-style route and then drop it to `channel: "none", reason: "no-target"` when the strict outbound plugin resolver cannot see the channel in an empty or unbootstrapped registry.

## Best Possible Solution

Keep this issue open and narrow the fix to the remaining plugin-channel heartbeat fallback. The likely code change belongs in `src/infra/outbound/targets.ts` or the outbound channel-resolution seam: when `target: "last"` recovers a stored plugin route from session metadata, preserve or safely bootstrap that route if outbound normalization fails only because the plugin registry is empty or unbootstrapped. Leave the restored duplicate suppression intact and add unmocked regression coverage for `resolveHeartbeatDeliveryTarget()` with an empty active plugin registry.
