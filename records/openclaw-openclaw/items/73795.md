## Summary

Keep open. Current main still has the reported cross-channel gap for WhatsApp and LINE: their strict config surfaces do not expose `streaming.preview.toolProgress`, and their inbound reply dispatch does not pass the shared suppression option that prevents generic tool/progress messages from reaching chat recipients. Slack, Telegram, and Matrix have adjacent support, but that does not close the missing config/runtime contract for the affected non-preview channels.

## What This Changes

The issue asks OpenClaw to preserve and honor `streaming.preview.toolProgress: false` for channels that do not already declare `streaming.mode`, so internal tool progress is not shown to end users.

## Best Possible Solution

Define a documented, schema-backed delivery contract for suppressing generic tool/progress messages independently of preview streaming. Expose it for WhatsApp and LINE, wire those runtimes to `suppressDefaultToolProgressMessages` or a documented successor, preserve existing Slack/Telegram/Matrix preview semantics, and add config/runtime/docs regression coverage.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: WhatsApp and LINE do not accept the nested `streaming.preview.toolProgress` config and do not pass the shared suppression option, while the dispatcher only suppresses generic tool/progress messages when that option is explicitly true.

## Solution Assessment

No current solution on main is sufficient. Preserving the nested config alone would not fix delivery; the runtime must also map the effective setting into the shared suppression path or a clearer channel delivery contract.
