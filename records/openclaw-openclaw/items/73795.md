## Summary

Keep open. Current main still lacks a supported WhatsApp/LINE `streaming.preview.toolProgress: false` config path, and those channel dispatch paths do not pass the suppression flag that the shared auto-reply runtime requires to hide default tool/progress messages. The related Slack PR is narrower and does not resolve the cross-channel config/runtime gap.

## What This Changes

The issue requests preserving and honoring `streaming.preview.toolProgress: false` for channels without an explicit `streaming.mode`, so end users do not receive internal tool/progress narration.

## Best Possible Solution

Add a documented, schema-backed suppression contract for channels that do not have preview streaming, then wire WhatsApp and LINE dispatch to suppress generic tool/progress delivery when the effective config disables tool progress. Keep Slack’s default behavior work separate or reconcile it explicitly, and add regression coverage for config preservation plus outbound progress suppression.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: WhatsApp and LINE strict schemas do not support `streaming.preview.toolProgress`, and their dispatch paths do not pass the suppression flag consumed by the shared progress-delivery code.

## Solution Assessment

No current fix is present on main. Preserving the config key alone is not the best solution because the runtime also needs to translate the effective setting into default tool/progress suppression for non-preview channels.
