## Summary

Keep open. Current main and the latest tagged release still lack a supported config and runtime path for the reported setting on WhatsApp and LINE: their strict schemas/generated metadata omit nested `streaming.preview.toolProgress`, and their inbound reply paths do not pass the shared dispatcher suppression flag. Because the report is privacy/security-adjacent and existing preview-channel semantics for `toolProgress: false` are inconsistent, this needs maintainer/API judgment rather than cleanup closure.

## What This Changes

The issue requests that `streaming.preview.toolProgress: false` be preserved and honored for channels that do not already declare `streaming.mode`, so internal tool/progress narration is not exposed to end users.

## Best Possible Solution

Define a documented, schema-backed progress-suppression contract for channels without preview streaming, preserve it through strict config schemas/generated metadata, and wire affected channel runtimes such as WhatsApp and LINE to suppress generic tool/progress delivery when the effective setting disables it. The final contract should reconcile preview-channel behavior so `toolProgress: false` has one documented meaning across channels.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: WhatsApp and LINE schemas/metadata reject or omit the nested setting, and their dispatch paths do not pass the suppression flag consumed by the shared reply dispatcher. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No current fix is present on main. Preserving the config key alone would not solve the observable leak; the runtime also needs to map the effective setting to `suppressDefaultToolProgressMessages` or a new shared delivery policy with clear cross-channel semantics.
