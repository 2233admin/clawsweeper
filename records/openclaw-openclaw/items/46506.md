## Summary

Keep this PR open. Current main still removes only the cron job record and emits the removed event; it does not clean the deleted job's session-store entries or unlink its per-job run log. The PR remains a plausible implementation candidate for #46369, but it needs maintainer review around data-deletion precision, in-flight run/log races, and unrelated churn.

## Best Possible Solution

Keep this PR open as an implementation candidate for #46369, or replace it with a maintainer-owned fix that cleans the removed job's canonical and run cron session entries from the correct agent store, unlinks the per-job run log through the hardened run-log path helper, handles or documents in-flight run/log-write races, adds focused regression tests, and removes unrelated changes.

## Risks / Open Questions

- Data-deletion precision matters: a landed fix should target the removed job's actual agent-scoped sessions.json and match cron session keys without deleting unrelated session rows.
- In-flight isolated runs can re-persist the canonical cron session or append a run log after one-time cleanup; maintainers should choose coordination, re-cleaning, or explicit retention behavior.
- The PR includes unrelated file churn in src/agents/model-compat.test.ts and src/gateway/server/ws-connection/message-handler.ts, which should be split or dropped before merge.
- Security-sensitive surface is local deletion of session metadata and run-log artifacts. No supply-chain expansion was observed, but the cleanup needs focused tests and careful path/key constraints.
