## Summary

Keep this PR open. Current main still removes only the cron job record and emits the removal event; it does not clean the deleted job's canonical cron session entry or unlink its per-job run log. The linked bug remains open, and this PR is still a plausible implementation candidate, though it needs maintainer review for data-deletion precision, in-flight run/log races, and unrelated file churn.

## Best Possible Solution

Keep this PR open as an implementation candidate for #46369, or replace it with a maintainer-owned fix that cleans the removed job's canonical cron session entry and per-job run log from the correct agent store, handles or documents in-flight run/log-write races, adds focused regression tests, and removes unrelated file churn.

## Risks / Open Questions

- Landing this PR as-is could still be nondeterministic for active cron runs: a finishing run can re-persist the canonical cron session after immediate cleanup, and a pending run-log append can recreate the deleted log file after unlink.
- The PR touches unrelated files outside the stated cron cleanup surface, including an agents test helper deletion and a gateway formatting change, which should be split or dropped before merge.
- Because this is a data-deletion path, maintainers should verify exact agent-scoped session-store targeting and exact cron-job key matching before landing.
