## Summary

Keep this PR open. Current main still does not add baseline security headers to the standalone `startCanvasHost` HTTP server, while the Gateway-hosted canvas path has its own shared header handling. The PR remains a focused hardening candidate, but it needs maintainer review on the exact header set, especially `X-Frame-Options: DENY`.

## Best Possible Solution

Keep the PR open as a focused security-hardening candidate. The best path is to rebase it onto current main, align the standalone canvas header policy with current Gateway policy, add 200/404 coverage for `startCanvasHost`, and get maintainer confirmation before adding `X-Frame-Options: DENY` to a canvas/A2UI surface.

## Risks / Open Questions

- `X-Frame-Options: DENY` may conflict with canvas/A2UI embedding expectations; current shared Gateway header policy intentionally avoids frame restrictions for that surface.
- The best header set may be the shared baseline helper, a canvas-specific subset, or a new helper rather than copying Control UI policy into canvas.
- Current main appears to use `createCanvasHostHandler` for the Gateway-hosted canvas surface; `startCanvasHost` has no non-test call sites found in `src`, so maintainers should decide whether to retain and harden the standalone path or remove it separately.
