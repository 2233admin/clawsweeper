## Summary

Keep open. Current main still ignores `threadId` in the Mattermost direct message-tool send path, and this PR is a focused fix that aligns that path with existing Mattermost outbound threading semantics. Before merge it still needs the required changelog entry and a stale fallback comment update.

## What This Changes

The PR updates Mattermost message-tool send handling to use `replyToId`, then `threadId`, then legacy `replyTo` when choosing the Mattermost reply root, with regression tests for trimmed `threadId` and `replyToId` precedence.

## Best Possible Solution

Land this focused Mattermost plugin fix after adding the release-note entry and updating the fallback comment, while leaving #67784 open for the remaining media-path and local-root message-tool work.

## Reproduction Assessment

Yes. Call `mattermostPlugin.actions.handleAction` with `action: "send"`, a Mattermost channel target, and `threadId` but no `replyToId`; current main forwards no `replyToId` to `sendMessageMattermost`, so no Mattermost `rootId` is set.

## Solution Assessment

Yes, with small maintenance fixes. The `replyToId ?? threadId ?? replyTo` order is the narrow maintainable behavior change because it matches the shared thread-aware precedence and the existing Mattermost outbound helpers while preserving explicit `replyToId` precedence.
