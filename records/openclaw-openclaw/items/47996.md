## Summary

Keep open. Current main does not already implement the PR's stale-expiry or direct-dedupe request, and the branch is not safe to merge as-is because it conflicts with current active-claim, queued-metadata, and per-call outbound delivery semantics.

## What This Changes

The PR branch adds 10-minute stale queued-outbound expiry during recovery, a 1.5s direct reply/mirror outbound dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI formatting changes.

## Best Possible Solution

Prepare a narrow outbound-only replacement that adds age-based recovery expiry after claiming and re-reading each pending entry, keeps current queued metadata and reconnect-drain protections intact, and treats direct-reply dedupe as a separate design only if every per-call delivery boundary is preserved.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Scope direct dedupe by per-call delivery context:** `src/infra/outbound/deliver.ts:375-397`
  - body: The dedupe key includes target/payload/mirror fields but not `session`, `gatewayClientScopes`, `abortSignal`, or callback identity. A cache or inflight hit can return another request's result without running `deliverOutboundPayloadsCore`, skipping session-scoped media roots, authorization-sensitive adapter behavior, hooks, and callback delivery for a distinct send.
  - confidence: 0.9
- **[P1] Bypass dedupe for queued recovery replays:** `src/infra/outbound/deliver.ts:386-418`
  - body: Recovery replays call `deliverOutboundPayloads` with `skipQueue: true` for each stored entry. Since the key only records `skipQueue` and not the queue entry id, two legitimate queued reply-shaped entries with the same target and payload can coalesce inside the 1.5s window, letting the second entry be acked without a real send.
  - confidence: 0.9
- **[P2] Preserve active recovery claims before expiring entries:** `src/infra/outbound/delivery-queue-recovery.ts:176-199`
  - body: The recovery loop acts on the startup snapshot directly. Current main re-reads each queue file under an in-process claim before replaying or failing it; without that guard, startup recovery can race reconnect/live drains and move or ack an entry after another path has taken ownership.
  - confidence: 0.87
- **[P2] Replay stored session and reply-mode metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65-81`
  - body: Current main stores and replays `replyToMode` and `session` for queued deliveries. This branch's recovery params omit both, so recovered messages can route replies, hooks, and media-local-root scoping differently from the original send.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR changes security-sensitive outbound delivery coalescing and recovery behavior without preserving all caller/session authorization boundaries.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:375`
  - body: The direct outbound cache key omits `gatewayClientScopes` and `session`, while current main passes both into delivery for adapter behavior, media scoping, and hooks. A hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent queued messages:** `src/infra/outbound/deliver.ts:416`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path with only `skipQueue` in the key. Duplicate queued records can be treated as delivered without a second adapter send, silently dropping pending outbound content.
  - confidence: 0.9
- **[medium] Startup expiry lacks the current ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries from the initial startup snapshot without the current main claim and re-read guard, so it can race another drain that already owns or acked the same queue file.
  - confidence: 0.84
