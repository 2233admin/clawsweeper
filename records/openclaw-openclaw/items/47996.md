## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the requested stale recovery expiry and direct reply dedupe, so the work is not obsolete; however the PR head is stale against current outbound recovery semantics and introduces blocking delivery, replay, and security concerns plus unrelated UI churn.

## What This Changes

The PR branch adds 10-minute queued outbound recovery expiry, a 1.5-second direct reply/mirror dedupe cache, targeted outbound tests, one channel setup type loosening, and broad Control UI render formatting changes.

## Best Possible Solution

Prepare a narrowed outbound-only revision or replacement: add age-based recovery expiry only after claiming and re-reading the current queue entry, preserve all queued replay metadata and reconnect-drain safeguards, and split direct-reply dedupe into a separate design unless it can preserve per-call delivery semantics and queue identity.

## Reproduction Assessment

Yes. The regressions have source-level reproduction paths: replay two identical queued reply-shaped entries with skipQueue during recovery to hit the dedupe cache, and let startup recovery act on a stale queue snapshot after another drain has claimed or acked the same entry.

## Solution Assessment

No. The stale-expiry idea is plausible, but this PR combines it with a security-sensitive direct-send cache and unrelated UI churn; a smaller recovery-only change is the safer maintainable path.

  - body: The dedupe key omits session, gatewayClientScopes, abortSignal, and per-call callbacks even though delivery uses them for media roots, adapter scopes, cancellation, hooks, and callback delivery.   - confidence: 0.9

  - body: The stale-expiry branch moves entries from the recovery snapshot directly. Current main claims each queue id and re-reads the current file before acting, which prevents startup recovery from racing reconnect or live delivery drains.
  - confidence: 0.86

  - body: Current main's recovery replay passes stored replyToMode and session through to delivery. This branch's buildRecoveryDeliverParams omits both, so recovered messages can route, hook, or scope media differently from the original send.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces outbound coalescing and recovery movement that do not preserve all caller/session and queue ownership boundaries.

Concerns:

- **[high] Dedupe crosses caller scope boundaries:** `src/infra/outbound/deliver.ts:375`
  - body: The direct outbound cache key omits gatewayClientScopes and session while delivery passes both into adapters, media access scoping, and hooks. A cache or inflight hit can reuse a result produced under another caller or session context.
  - confidence: 0.88
- **[medium] Recovery replay can acknowledge unsent messages:** `src/infra/outbound/deliver.ts:416`
  - body: Queued recovery calls are eligible for the new inflight/recent-result dedupe path without queue entry identity in the key, so duplicate stored records can be treated as delivered without a second adapter send.
  - confidence: 0.9
- **[medium] Startup expiry lacks ownership guard:** `src/infra/outbound/delivery-queue-recovery.ts:193`
  - body: The stale-expiry path moves entries based on the initial recovery snapshot rather than the claimed and re-read queue entry, so it can race another drain that has already owned or acked the same queue file.
  - confidence: 0.84
- **[medium] Recovery replay drops session metadata:** `src/infra/outbound/delivery-queue-recovery.ts:65`
  - body: The branch's recovery params omit stored session and reply mode metadata, which can change media access policy, hook attribution, and reply routing during replay.
  - confidence: 0.82
