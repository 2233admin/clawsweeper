## Summary

Keep open. Current main still performs two independent task reconciliation passes in one `openclaw status` summary, and open PR #73650 is an unmerged implementation linked with closing syntax, so the issue should remain open for review, landing, or closure of that PR.

## What This Changes

Request to remove redundant task reconciliation in `openclaw status` by reusing one reconciled task snapshot or cache for the registry and audit summaries.

## Best Possible Solution

Review and land PR #73650 if its per-status shared-snapshot approach is correct, or apply an equivalent narrow change that reconciles tasks once inside `getStatusSummary()` while preserving the zero-argument helper behavior for other call sites.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: current `getStatusSummary()` calls both task summary helpers, and each helper currently invokes `reconcileInspectableTasks()` independently; no live 10K-record timing run was performed in this read-only review.

## Solution Assessment

No for the issue’s proposed module-level process cache as the first fix. A per-status-call shared reconciled snapshot is narrower, avoids cache invalidation and shared mutable-reference hazards, and matches the already linked implementation direction.
