## Summary

Keep open. Current main still matches the reported performance bug: `getStatusSummary()` builds task registry and task audit summaries separately, and both helpers independently call `reconcileInspectableTasks()`, which still calls the clone/sort-heavy `listTaskRecords()` path. The prior ClawSweeper comment also reached the same conclusion, and the current checkout has not added a shared status snapshot, reconciliation cache, or invalidation wiring.

## Best Possible Solution

Keep this issue open and fix the status/task maintenance path. The most conservative implementation is to compute `reconcileInspectableTasks()` once inside `getStatusSummary()` and derive both `tasks` and `taskAudit` from that snapshot, with a focused regression test proving one status summary invocation does not invoke the expensive task-listing/reconciliation path twice. If maintainers prefer reusable caching, add documented invalidation around task registry observer events and `configureTaskRegistryMaintenance()`.

## Risks / Open Questions

- A module-level cache could expose mutable cached task objects unless callers receive immutable/cloned snapshots or the cache is scoped carefully.
- Any cross-call cache needs explicit invalidation on restored/upserted/deleted task events and maintenance configuration changes; a per-status-call shared snapshot is likely the narrower first fix.
