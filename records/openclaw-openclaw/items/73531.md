## Summary

Keep open. Current main still follows the duplicated status task-reconciliation path, and the provided GitHub context shows PR #73650 is an open, unmerged implementation candidate linked with closing syntax, so this issue should stay open for maintainer review or landing of that PR.

## What This Changes

The issue asks OpenClaw to avoid reconciling the full inspectable task list twice during `openclaw status`, either with an in-process cache or an equivalent shared snapshot.

## Best Possible Solution

Review and land PR #73650 if its per-status-call shared snapshot approach is correct, or apply an equivalent narrow change that reconciles tasks once inside `getStatusSummary()` and derives both registry and audit summaries from that snapshot while preserving zero-argument helper behavior for other call sites.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: `getStatusSummary()` calls both task summary helpers, each helper currently invokes `reconcileInspectableTasks()`, and that reaches the clone/sort `listTaskRecords()` path; timing tests were not run because this pass is read-only.

## Solution Assessment

No for the issue's module-level cache as the first choice. A per-status-call shared snapshot is the narrower maintainable fix because it removes the duplicated work without cache invalidation or shared mutable-reference hazards.
