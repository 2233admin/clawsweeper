## Summary

Keep open. Current main still matches the reported performance bug: `getStatusSummary()` calls the task registry summary and audit helpers back to back, and both helpers independently call `reconcileInspectableTasks()`, which in turn calls `listTaskRecords()` and pays the clone/sort cost each time. The latest release source has the same call shape, so this is not already implemented on main or superseded by the related status-performance issues.

## Best Possible Solution

Keep this issue open and fix the status/task maintenance path. The narrowest maintainer-friendly direction is to compute the reconciled task list once for the status summary path and reuse it for both registry summary and audit summary, or add a documented in-process reconciliation cache with explicit invalidation through task registry observer events and `configureTaskRegistryMaintenance()` changes. Add a focused regression test that proves a single `getStatusSummary()` invocation does not call the expensive task-listing/reconciliation path twice.

## Risks / Open Questions

- A module-level cache must avoid exposing mutable cached task objects to callers that might mutate them; a per-status-call shared reconciled snapshot may be safer if the cache contract is unclear.
- Cache invalidation would need to cover task registry restore/upsert/delete events plus maintenance configuration changes, because reconciliation can project cron-recovered or lost task state without mutating the registry.
