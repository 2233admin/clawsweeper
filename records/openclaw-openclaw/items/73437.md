## Summary

Keep open. Current main still only re-runs context-engine assembly inside the Pi tool loop for active context engines with `ownsCompaction === true`; non-owning engines still get the generic tool-result guard plus the one pre-prompt `assemble` call. The public plugin hook requested by the related closed PR, `transform_context`, is still absent from the hook types/docs.

## Best Possible Solution

Keep this issue open for a focused plugin/API design. The most conservative path is to add a documented per-LLM-call context transform seam, or explicitly broaden the context-engine loop assembly contract beyond `ownsCompaction === true`, then add tests for owning and non-owning engines across multi-tool-call loops and update the Plugin SDK/context-engine docs.

## Risks / Open Questions

- Changing `assemble` to fire for every model call could be a compatibility change for existing context engines; a new additive `transform_context` hook may be safer but needs maintainer API judgment.
- The current implementation relies on Pi's private `agent.transformContext` boundary, so any public API should be backed by dependency-contract verification and regression tests around multi-tool-call loops.
