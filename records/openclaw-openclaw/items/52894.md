## Summary

Keep open. Current main still leaves the reported behavior unresolved: skill onboarding prints the Homebrew installer command, while the backend skill installer and gateway/UI path still return a missing-brew failure rather than bootstrapping Homebrew. The prior ClawSweeper comment also correctly tied this to the still-open skill-install UX work in #57555, and the remaining question is maintainer product/security policy for executing Homebrew's external installer from OpenClaw flows.

## What This Changes

The issue asks OpenClaw to automatically install Homebrew when a brew-backed skill install or onboarding flow detects that `brew` is missing.

## Best Possible Solution

Keep the issue open for maintainer product/security direction. If maintainers approve Homebrew bootstrapping, add an explicit opt-in macOS flow shared by skill onboarding and gateway-backed installs, re-check `brew` after installation, preserve dangerous-skill scanning and clear user consent, update UI/CLI/docs, and add focused tests; if not approved, improve the manual Homebrew requirement in prompts, UI, and docs.
