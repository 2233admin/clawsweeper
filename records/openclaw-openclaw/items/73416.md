## Summary

Keep open. Current main still gates Google thinking payload sanitization on `google-generative-ai` only, while the Google provider and Gemini CLI paths share the same stream wrapper family; the PR matches the linked Vertex/CLI bypass described in #38327 and should proceed through maintainer review and CI rather than being auto-closed or replaced.

## What This Changes

The PR changes the shared Google thinking payload wrapper guard to cover `google-generative-ai`, `google-vertex`, and `google-gemini-cli`, and adds wrapper regression tests for those Google API shapes plus non-Google passthrough.

## Best Possible Solution

Land this guard widening, or an equivalent main-branch patch, after normal maintainer review and current CI/Testbox confirmation; keep #38327 open until a fix merges, and handle the gaxios/native-fetch concern as a separate dependency-patch decision.

## Reproduction Assessment

Yes. The focused reproduction is a wrapper payload with `thinkingConfig: { thinkingBudget: -1 }` and `model.api` set to `google-vertex` or `google-gemini-cli`; current main leaves it intact because the guard checks only `google-generative-ai`, while the PR tests assert it is stripped.

## Solution Assessment

Yes. Widening the existing Google-family guard is the narrowest maintainable fix for this bypass, preserves non-Google passthrough, and avoids mixing in the separate gaxios dependency patch or broader thinking-policy PRs.
