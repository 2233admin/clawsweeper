## Summary

Keep open. Current main still applies the `coding` profile before per-agent `tools.allow`, so the reported browser/Gmail allowlist remains an empty intersection under this config. Current main has better fail-closed behavior and docs point to `alsoAllow`, but the core product question is still unresolved.

## What This Changes

The issue requests that browser and Gmail plugin tools remain available for an agent whose `agents.list[].tools.allow` names those tools even when top-level `tools.profile` is set to `coding`.

## Best Possible Solution

Decide the intended contract for `tools.profile` plus per-agent tool overrides, then either make additive per-agent tool exceptions participate in profile-stage filtering or add a targeted diagnostic that explains the empty profile/agent allowlist intersection and points to `agents.list[].tools.alsoAllow` for this setup.

## Reproduction Assessment

Yes. The issue provides concrete config and commands, and current source plus browser-profile tests show the same high-confidence path: `coding` filters out browser/plugin tools before `agents.main.tools.allow` is applied.

## Solution Assessment

Unclear. The current `alsoAllow` workaround is narrow and documented, but whether `agents.list[].tools.allow` should become additive under a profile is a maintainers' product/security contract decision.
