## Summary

Keep this PR open: current main still has the Slack interactive-reply wake gap, but the PR branch is not ready for conservative cleanup or automerge because its wake path still diverges from the normal Slack routing/threading semantics and overlaps with broader open Slack interaction work.

## What This Changes

The PR adds a Slack `wakeSlackReplySession` path for interactive reply button clicks and extends Slack interaction tests for dispatch, threading, dedupe, and reply delivery behavior.

## Best Possible Solution

Land one canonical Slack interaction wake implementation that preserves the existing enqueue/update UX, proactively starts exactly one authorized agent turn for reply buttons, and reuses the normal Slack inbound routing, chat-type reply-mode resolution, session-key derivation, dedupe, and reply delivery planning semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use the Slack reply planner for wake deliveries:** `extensions/slack/src/monitor/events/interactions.block-actions.ts:732-733`
  - body: `replyThreadTs` is computed once and reused for every delivery callback. For top-level clicks with `replyToMode: "first"` or `"batched"`, normal Slack dispatch consumes a delivery plan per send, so this path can incorrectly thread every interaction-triggered reply instead of following the configured first/batched behavior.
  - confidence: 0.86
- **[P2] Resolve reply mode by chat type:** `extensions/slack/src/monitor/events/interactions.block-actions.ts:690-693`
  - body: The wake path reads `ctx.replyToMode` directly, but normal Slack inbound handling first applies `replyToModeByChatType` and legacy `dm.replyToMode`. With a direct-message override such as default `all` plus direct `off`, a button click would derive a thread session/reply while the next normal DM stays on the base session.
  - confidence: 0.84
