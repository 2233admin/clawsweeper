## Summary

Current main still has the configured media-provider auth failure, while this active PR addresses the prior auth-precedence feedback and I found no blocking correctness or security issue in the final patch; it needs maintainer review and CI/Testbox validation rather than cleanup closure.

## What This Changes

The PR adds a media-provider synthetic-auth hook, uses it as a resolver-first fallback for audio/video execution, documents local no-auth media providers, and adds regression tests for no-auth execution, credential precedence, explicit profile failures, and fail-closed defaults.

## Best Possible Solution

Review and land a resolver-first, provider-owned synthetic auth hook for configured media execution, with the added docs and regression coverage; keep any broader auto-selection behavior change as a separate, explicit follow-up if maintainers want it.

## Reproduction Assessment

Yes. Current main has a high-confidence code-path reproduction: a configured plugin-only audio provider with no model auth reaches resolveProviderExecutionAuth, then requireApiKey, and fails before transcribeAudio runs; the linked report supplies matching minimal steps.

## Solution Assessment

Yes. The proposed direction is the narrow maintainable fix for the reported execution path because it preserves the existing resolver first, keeps explicit profile errors hard-failing, and requires provider-owned opt-in instead of weakening core auth globally.
