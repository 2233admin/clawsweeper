## Summary

Keep open. Current main still rejects plugin-only local/no-auth media providers before their audio/video implementations run, while the latest PR head appears to address the linked bug and the prior auth-precedence review finding without a remaining blocking issue.

## What This Changes

The PR adds a `resolveSyntheticAuth` media-provider hook, threads provider instances into audio/video media auth resolution, documents the local no-auth pattern, and adds regression coverage for synthetic auth, credential precedence, explicit profile failures, and fail-closed defaults.

## Best Possible Solution

Merge a resolver-first, explicit synthetic-auth hook with the added regression coverage and docs so local media providers can opt into no-auth execution while unmarked or remote providers continue to fail closed.

## Reproduction Assessment

Yes. Current main can be reproduced with a plugin-only `mediaUnderstandingProviders` audio provider that has no model auth configured; `resolveProviderExecutionAuth(...)` calls the model-auth resolver and `requireApiKey(...)` before `transcribeAudio(...)` can run.

## Solution Assessment

Yes. The latest PR approach is the narrow maintainable fix: keep existing literal/env/profile resolver precedence, preserve explicit profile failures, and use provider-owned synthetic auth only as the missing-key fallback.
