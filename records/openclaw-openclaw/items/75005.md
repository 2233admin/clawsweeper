## Summary

Keep open. Current main still routes audio/video media providers through the model auth gate before provider execution, and this active PR is a narrow implementation with docs, regression coverage, addressed review feedback, and no blocking finding from this pass.

## What This Changes

The PR adds a media-provider `resolveSyntheticAuth` hook, threads the provider into audio/video execution auth, uses the hook as a missing-key fallback after normal auth resolution, and documents/tests local no-auth media providers.

## Best Possible Solution

Review and land the explicit, provider-owned synthetic-auth hook if maintainers accept the SDK contract, then let the linked media no-auth issue close through the merged PR; keep broader auto-selection or discovery semantics as separate follow-up work.

## Reproduction Assessment

Yes. Current main has a high-confidence code-path reproduction: a plugin-only audio provider with no configured credentials reaches `resolveProviderExecutionAuth`, then `resolveApiKeyForProvider`/`requireApiKey`, before `transcribeAudio` can run; the linked issue and a second commenter report the same failure shape.

## Solution Assessment

Yes. This is the narrow maintainable direction because it preserves normal literal/env/profile/config auth precedence, keeps explicit profile errors hard-failing, and requires provider-owned opt-in instead of weakening the core media auth gate globally.
