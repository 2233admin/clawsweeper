## Summary

Keep this issue open. Current main has foreground Telegram HTTP(S) env-proxy support, but it does not fully satisfy the report: Telegram still treats ALL_PROXY-only environments as direct transport, installed gateway services intentionally persist only OPENCLAW_PROXY_URL rather than ambient HTTP_PROXY/HTTPS_PROXY/ALL_PROXY, and the exact HTTP_PROXY failure needs a process-env repro to distinguish a Telegram fetch bug from service environment policy.

## Best Possible Solution

Keep this open, reproduce v2026.4.26/current main with the actual gateway process environment logged or inspected, then decide the intended contract: if foreground HTTP_PROXY/HTTPS_PROXY is bypassed despite being present, fix Telegram transport; if this is an installed-service setup, point users to durable OPENCLAW_PROXY_URL/config or make an explicit service-env policy change; and if ALL_PROXY is intended for Telegram, wire Telegram to the shared ALL_PROXY-aware EnvHttpProxyAgent options with focused regression tests and docs.
