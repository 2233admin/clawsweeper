## Summary

Keep this open. Current main still does not fully satisfy the report: Telegram foreground HTTP_PROXY/HTTPS_PROXY routing appears implemented through EnvHttpProxyAgent, but ALL_PROXY-only Telegram routing is explicitly treated as direct transport, and installed gateway services intentionally do not persist ambient HTTP_PROXY/HTTPS_PROXY/ALL_PROXY values. The exact HTTP_PROXY symptom needs a process-env/live repro, but there is a narrow Telegram proxy gap worth fixing or confirming by maintainer policy.

## Best Possible Solution

Keep the issue open, reproduce with v2026.4.26/current main while logging the gateway process proxy env, then land a narrow Telegram transport fix if ALL_PROXY is intended: make Telegram's resolver-scoped EnvHttpProxyAgent use the shared ALL_PROXY-aware EnvHttpProxyAgent options, add uppercase HTTP_PROXY and ALL_PROXY regression tests, document supported env vars versus installed-service OPENCLAW_PROXY_URL, and leave the service-env policy unchanged unless maintainers choose otherwise.

## Clownfish Work Prompt

Fix the Telegram proxy-env gap reported in this issue without changing installed service env policy. Current main uses hasEnvHttpProxyConfigured("https") in extensions/telegram/src/fetch.ts and constructs EnvHttpProxyAgent without explicit httpProxy/httpsProxy options, while extensions/telegram/src/fetch.test.ts explicitly treats ALL_PROXY-only as direct. Update the Telegram resolver-scoped transport so it continues to honor HTTP_PROXY/HTTPS_PROXY and also honors ALL_PROXY/all_proxy by using the existing shared EnvHttpProxyAgent option semantics. Add regression tests for uppercase HTTP_PROXY fallback to HTTPS Telegram requests and ALL_PROXY-only Telegram requests using EnvHttpProxyAgent.
