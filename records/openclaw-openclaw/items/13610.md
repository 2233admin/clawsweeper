## Summary

Keep open. Current main has the shipped SecretRef baseline for env/file/exec providers, but it still does not implement native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, or Google Cloud Secret Manager providers. The related plugin-provider seam is still an open PR candidate, so this remains a valid umbrella for maintainer follow-up.

## Best Possible Solution

Keep this issue open as the native secrets-manager provider umbrella. The maintainer path is to review and either land a generic secret-provider plugin SDK seam like #72548, then move AWS/Vault/Azure/GCP implementations to plugins/ClawHub, or explicitly choose first-class native providers and update config types, zod and wire schemas, resolver dispatch, CLI configure/audit/apply flows, docs, and tests together.

## Risks / Open Questions

- This is security-sensitive provider architecture, not a narrow bug fix; native cloud/Vault integrations would introduce dependency, credential-loading, caching, audit, and failure-mode review concerns.
- Closing now would lose the umbrella for AWS/Vault/Azure/GCP native provider work, because current main only supports env/file/exec and #72548 is still open.
- Exec wrappers are a practical workaround, but they do not provide provider-specific schemas, SDK dispatch, CLI UX, cache/audit semantics, or native tests for the requested managers.
