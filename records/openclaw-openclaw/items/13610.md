## Summary

Keep this issue open. Current main has the shipped SecretRef baseline for env/file/exec providers, but it still lacks native AWS Secrets Manager, HashiCorp Vault, Azure Key Vault, and Google Cloud Secret Manager sources, and the related plugin-provider seam is still only an open PR candidate.

## Best Possible Solution

Keep this issue open as the native secrets-manager provider umbrella. The maintainer path is to review and either land a generic secret-provider plugin SDK seam like #72548, then move AWS/Vault/Azure/GCP implementations to plugins/ClawHub, or explicitly add first-class native providers across config types, zod and wire schemas, resolver dispatch, CLI configure/audit/apply flows, docs, and tests.

## Risks / Open Questions

- Exec providers can wrap Vault, 1Password, cloud CLIs, or custom resolvers, but that does not provide native provider source types, provider-specific schema, resolver dispatch, CLI UX, caching/audit semantics, or tests for AWS/Vault/Azure/GCP.
- Closing this umbrella now would hide remaining first-class vendor-provider work that the issue discussion and #72548 still track as unresolved.
- Native secret-manager integrations are security-sensitive and should route through @openclaw/secops plus a product decision on bundled core providers versus a public secret-provider plugin/ClawHub path.
