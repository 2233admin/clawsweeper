## Summary

Keep open. Current main still retries Telegram `getFile()` only; after that succeeds, the Bot API file download still awaits one `fetchRemoteMedia()` call, and the existing unit test explicitly asserts that download failures are not caught or retried. The related Telegram proxy/SSRF DNS work has shipped, but it addresses proxy propagation rather than bounded backoff for transient download-stage failures, and the linked implementation PR is closed unmerged.

## What This Changes

The issue asks OpenClaw to retry Telegram inbound media file downloads when a configured proxy or network path fails transiently after `getFile()` succeeds.

## Best Possible Solution

Add a bounded Telegram plugin retry around only the remote media download stage, retrying transient `fetch_failed` download errors while preserving the existing transport, `dispatcherAttempts`, explicit-proxy DNS behavior, SSRF policy, local Bot API file-path handling, and permanent-failure behavior.

## Reproduction Assessment

Yes. The current unit test mocks a successful `getFile()` followed by a failed `fetchRemoteMedia()` and asserts that `resolveMedia()` rejects without retrying, directly reproducing the missing download-stage retry path described here.

## Solution Assessment

Yes. A small `retryAsync()` wrapper around the `fetchRemoteMedia()` download step is the narrowest maintainable fix; automatic direct fallback is not the best solution because it could bypass an operator-configured proxy and SSRF policy boundary.
