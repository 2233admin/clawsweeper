## Summary

Keep open. Current main still retries Telegram `getFile()` only; the actual Bot API file download calls `fetchRemoteMedia()` once, and the existing regression test explicitly asserts download errors are not caught/retried. The related proxy-propagation bug family was fixed and shipped, but this issue’s requested transient download retry remains open. Prior PR #46013 targeted the right files but was closed unmerged, and a current open-PR search for this issue returned no candidates.

## Best Possible Solution

Implement a narrow Telegram plugin fix: wrap only the remote `fetchRemoteMedia()` download inside `downloadAndSaveTelegramFile()` with bounded `retryAsync()` backoff, retry transient `fetch_failed` download failures, keep permanent errors non-retryable, preserve the current transport/proxy/SSRF arguments exactly, and replace the current no-retry test with focused retry/non-retry coverage. Do not add direct fallback.

Implement a narrow fix in `extensions/telegram/src/bot/delivery.resolve-media.ts`: wrap the remote `fetchRemoteMedia()` call with `retryAsync()` using bounded exponential backoff, ideally 3 attempts with roughly 500ms-2000ms delays and jitter. Confirm `getFile()` is not re-run unnecessarily when only the download attempt is retried, and ensure `saveMediaBuffer` is called only after a successful fetch.

Use the closed unmerged PR #46013 only as background; rebase the approach onto current main instead of resurrecting old code wholesale. ## Risks / Open Questions

- Retrying must stay limited to transient download failures such as `MediaFetchError` with code `fetch_failed`; retrying `http_error`, `max_bytes`, SSRF/policy failures, or absolute local path validation would waste time or weaken safety expectations.
- Do not add automatic direct/no-proxy fallback. The issue discussion correctly notes that bypassing an operator-configured proxy can violate network policy; the safer fix is retrying the configured/proxied download path.
- Preserve existing explicit-proxy DNS and SSRF behavior from the 2026.4.14 fix while adding retry; this path is security-sensitive because it downloads remote media through the SSRF guard.
