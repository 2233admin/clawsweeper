## Summary

Keep open. Current main has ordered reactive model fallback and provider usage reporting, but it still does not have a proactive token/quota/cost threshold policy that changes model selection before a provider fails, nor the requested reverse failback or threshold-switch notification behavior.

## What This Changes

Requested automatic switching between configured LLM providers based on token or cost thresholds, with ordered fallback, optional reverse failback, and notifications.

## Best Possible Solution

Add a documented core usage-aware model-routing policy that evaluates provider usage snapshots and local token/cost state before candidate selection, preserves existing reactive error/cooldown fallback, defines reset and reverse-failback semantics, and exposes clear user notifications/status for threshold-triggered switches.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source and docs inspection: current main has configured fallbacks and usage snapshots, but no threshold policy feeding model routing.

## Solution Assessment

Unclear until maintainers define the product policy. The capability likely belongs in core fallback/model-selection policy rather than a standalone plugin because it changes normal provider routing, but thresholds, stale-data behavior, failback rules, and notification surfaces need design first.
