## Summary

Keep open. Current main supports configured reactive model fallback and separate provider usage reporting, but it still does not route models proactively from token/quota/cost thresholds, reverse fail back after reset, or emit user-visible threshold-switch notifications. Related #45771 overlaps on usage pace awareness, but it remains open and does not fully cover ordered token-based model failover.

## What This Changes

The issue requests configurable token or cost thresholds that automatically switch from a primary model/provider to ordered fallbacks, optionally switch back after reset, and notify the user when failover happens.

## Best Possible Solution

Add a documented core usage-aware model routing policy that evaluates provider usage snapshots and local token/cost state before candidate selection, preserves the existing error/cooldown fallback behavior, defines reset and reverse-failover semantics, and coordinates with #45771 rather than treating this as a standalone plugin request.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is reproducible by source/docs inspection: configured fallbacks and usage snapshots exist, but no token/cost threshold policy feeds model candidate selection.

## Solution Assessment

Unclear until maintainer design is done. The capability likely belongs in core model selection/fallback because it changes how configured providers are selected for normal runs, but the exact threshold, stale-data, reverse-failover, and notification rules need design first.
