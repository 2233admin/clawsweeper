## Summary

Keep open. Current main has ordered reactive model fallback, auth-profile cooldown handling, and provider usage reporting, but the requested proactive token/cost threshold routing, reverse failover after reset, and user-visible threshold-switch notifications are still missing.

## What This Changes

The issue requests configurable token or cost thresholds that automatically switch from a primary model/provider to ordered fallbacks and optionally switch back when usage resets.

## Best Possible Solution

Add a documented core usage-aware model routing policy that evaluates provider usage snapshots and local token/cost state before candidate selection, preserves existing runWithModelFallback error/cooldown behavior, defines reset and reverse-failover semantics, emits user-visible lifecycle notifications, and coordinates with #45771 so pace throttling and model switching use one coherent model.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is high-confidence from source/docs inspection: configured fallbacks and provider usage snapshots exist, but no threshold policy feeds into candidate selection.

## Solution Assessment

Unclear until maintainer design is done. The capability likely belongs near core model selection/fallback rather than a standalone plugin, but thresholds, stale usage data, reverse switching, and notification UX need product decisions first.
