## Summary

Keep #71770 open. Current main implements ordered fallbacks, auth-profile cooldowns, rate-limit rotation caps, provider usage/quota reporting, and fallback lifecycle notices, but it does not implement proactive token/quota/cost-threshold routing, ordered threshold failover before a call, or reverse failover when usage windows reset.

## Best Possible Solution

Keep the issue open for maintainer product/design work in the model selection and fallback path. A robust implementation would add a documented config surface for token, quota, or cost thresholds; evaluate provider usage snapshots and local cost/session state before choosing each candidate; integrate with `runWithModelFallback` without weakening existing cooldown/error fallback; emit lifecycle and user-visible notifications for threshold switches; and define reverse-failover semantics when quota windows reset. If maintainers want this policy to be plugin-owned, core should first expose a deliberate usage-aware routing API rather than relying on the current prompt-only `before_model_resolve` hook.

## Risks / Open Questions

- Provider usage data is uneven across providers, so threshold semantics need explicit behavior for providers with no quota API, multiple windows, or only post-response token usage.
- Automatic switching can increase costs or surprise users unless fallback order, threshold units, notifications, and reverse-failover behavior are explicit and auditable.
- A plugin-level pre-resolve override can cover simple custom routing, but current hooks do not provide enough usage-aware routing state to treat the core issue as already satisfied outside the repo.
