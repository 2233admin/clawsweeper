## Summary

Keep open. Current main still has reactive model fallback and separate provider usage reporting, but it does not implement proactive token/quota/cost threshold routing, reverse failback semantics, or user-facing notifications when a threshold causes a model/provider switch. The overlapping pace-aware rate limiting tracker remains open and does not fully subsume this cross-provider failover request.

## What This Changes

The issue requests configurable token or cost thresholds that automatically switch between ordered LLM provider/model fallbacks, optionally switch back after reset, and notify users when routing changes.

## Best Possible Solution

Add a documented core usage-aware model routing policy that evaluates provider usage snapshots and local token/cost state before candidate selection, preserves existing error/cooldown fallback behavior, defines reset and reverse-failover semantics, and coordinates with #45771 rather than treating this as a standalone plugin request.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is reproducible by source and docs inspection: configured fallbacks and usage snapshots exist, but no token/cost threshold policy feeds model candidate selection.

## Solution Assessment

Unclear until maintainer design is done. The capability likely belongs in core model selection/fallback because it changes normal provider routing, but threshold, stale-data, reverse-failover, and notification rules need product design first.
