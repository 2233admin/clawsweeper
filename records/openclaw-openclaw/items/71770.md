## Summary

Keep open. Current main has ordered reactive model fallback and provider usage reporting, but it still does not proactively choose fallback models from token/quota/cost thresholds, reverse fail back after reset, or emit threshold-switch notifications. The related pace-aware rate limiting item overlaps on usage awareness but remains open and does not fully cover ordered token-based model routing.

## What This Changes

Request to add automatic model/provider failover based on configurable token, quota, or cost thresholds, including ordered fallbacks, optional reverse failover, and user notifications.

## Best Possible Solution

Add a documented core usage-aware model routing policy that evaluates provider usage snapshots and local token/cost state before candidate selection, preserves existing error/cooldown fallback behavior, defines reset and reverse-failover semantics, and coordinates with #45771 for shared pace/usage concepts.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The gap is verifiable by source and docs inspection: configured fallbacks and usage snapshots exist, but no token/cost threshold policy feeds model candidate selection.

## Solution Assessment

Unclear until maintainer design is done. The capability likely belongs in core model selection because it changes normal provider/model routing, but the exact threshold, stale-data, reverse-failover, and notification rules need product direction first.
