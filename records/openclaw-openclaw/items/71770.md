## Summary

Keep open. Current main has ordered reactive model fallback for configured candidate chains, auth/profile cooldowns, and failover-worthy provider errors, but it does not implement proactive token/quota/cost threshold routing, reverse failover after reset, or user-visible threshold-switch notifications. Related #45771 overlaps on usage pace awareness, but it is open and does not fully cover this ordered model failover request.

## What This Changes

The issue requests automatic switching among configured LLM providers or models based on configurable token or cost thresholds, with ordered fallback, optional reverse failover, and user notifications.

## Best Possible Solution

Add a documented core usage-aware model routing policy that evaluates provider usage snapshots and local session token/cost state before candidate selection, preserves the existing runWithModelFallback error/cooldown behavior, emits user-visible lifecycle notifications for automatic threshold switches, defines reset/reverse-failover semantics, and coordinates the usage-policy model with #45771.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The feature gap is high-confidence from source and docs inspection: configured fallbacks and usage snapshots exist, but no token/cost threshold policy feeds into model candidate selection.

## Solution Assessment

Unclear until maintainer design is done. The capability likely belongs in core model selection/fallback rather than a standalone plugin, but thresholds, stale usage data, reverse failover, and notification UX need product decisions before a narrow fix PR is safe.
