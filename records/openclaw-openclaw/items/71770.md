## Summary

Keep open. Current main still has reactive model failover based on configured fallback chains, auth/profile cooldowns, and failover-worthy provider errors. It does not implement proactive token/quota/cost threshold routing, reverse failover after quota reset, or user-visible threshold-switch notifications. Related #45771 overlaps on pace/usage awareness but is not a complete duplicate of this ordered model failover request.

## Best Possible Solution

Keep this issue open for maintainer product/design work in the model selection and fallback path. A good implementation would add a documented usage-aware routing policy, evaluate provider usage snapshots and local session cost/token state before candidate selection, preserve existing runWithModelFallback cooldown/error behavior, emit user-visible lifecycle notifications for threshold switches, define reverse-failover/reset semantics, and coordinate with #45771 so pace-aware throttling and threshold-based model switching share one policy model.
