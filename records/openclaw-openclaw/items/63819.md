## Summary

Keep open. Current main still has the timer-only pending terminal lifecycle-error path described by the reporter's workaround, so a later non-error event can still clear the deferred terminal cleanup before session lifecycle state is finalized and persisted.

## What This Changes

The issue reports that Telegram sessions remain persisted as `running` after local llama.cpp request timeouts in v2026.4.9, despite the earlier `phaseBeforeAbort` cleanup fix.

## Best Possible Solution

Add a focused gateway regression test for a deferred lifecycle error followed by a later non-error event with no lifecycle end, then update the pending terminal lifecycle-error handling so the retained terminal event is finalized when it is cleared outside an intentional retry/fallback replacement.

## Repair Work Prompt

Create a narrow fix for openclaw/openclaw#63819. Current main stores pendingTerminalLifecycleErrors as Map<string, NodeJS.Timeout>; a later non-lifecycle/error event clears that timer and loses the deferred lifecycle error before persistGatewaySessionLifecycleEvent runs, which can leave the session status persisted as running. Add a regression in src/gateway/server-chat.agent-events.test.ts for lifecycle start, then lifecycle error scheduling pending cleanup, then a subsequent non-error agent event arriving without a lifecycle end; require terminal lifecycle persistence/cleanup rather than leaving the session running. Update src/gateway/server-chat.ts to retain the deferred event/options with the timer and finalize it when clearing is not an intentional retry/fallback replacement. Preserve the retry/fallback behavior from #60043/#64051 and the chat.send lifecycle-error behavior from #69747. Do not add Telegram, llama.cpp, provider-id, or bundled plugin special cases. Add a single-line changelog Fixes entry with Thanks @irvinheard and @mjamiv if the fix is user-facing.
