## Summary

Keep open. Current main still resolves the default or explicit heartbeat target `none` to `channel: "none"` with `reason: "target-none"`, and the heartbeat runner forwards that reason to the persisted/last-heartbeat event. The docs confirm `target: "none"` is intentional no-delivery behavior, and related PR #46870 proposed the rename but is closed unmerged.

## Best Possible Solution

Keep this issue open and land a narrow heartbeat DX fix. The cleanest implementation is to make the no-delivery-by-configuration path explicit, either by renaming this skipped reason to `delivery-disabled` or by adding target metadata / CLI display text that says delivery is disabled because `target=none`. The fix should not change heartbeat scheduling, delivery routing, or the default `target: "none"` behavior.
