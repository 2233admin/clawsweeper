## Summary

Keep open. Current main still emits `reason: "target-none"` when heartbeat delivery target is unset or explicitly `none`, and that reason is surfaced through the last-heartbeat event path. The related rename PR #46870 was closed unmerged, and the current tree has no `delivery-disabled` implementation.

## Best Possible Solution

Keep this issue open and make a narrow heartbeat DX fix in `src/infra/outbound/targets.ts` plus the heartbeat target tests. Either rename the skipped reason for `target: "none"` to a configuration-oriented value such as `delivery-disabled`, or preserve the existing reason while adding explicit metadata or a CLI hint so users can distinguish intentional disabled delivery from route resolution failure. Update heartbeat troubleshooting/docs if the public reason string changes.
