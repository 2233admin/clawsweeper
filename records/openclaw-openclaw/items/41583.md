## Summary

Keep this issue open. Current main still injects trusted inbound metadata and user-role untrusted metadata for channel messages without a config gate. Telegram and Discord have access, history, and supplemental-context visibility controls, but no option to disable or minimize the per-message metadata blocks. The related open PR #41299 does not supersede this request; it is scoped to adding a separator/stripper and explicitly avoids API or config changes.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment. A good path is a documented shared reply-runtime option, defaulting metadata injection to enabled, with narrow per-channel/account or trusted-DM scoping, Telegram and Discord tests, and docs that explain the provenance tradeoff.

## Risks / Open Questions

- A broad metadata-off switch could weaken prompt provenance, sender verification, routing clarity, and copied-metadata stripping. Any implementation should be opt-in, default metadata on, and scoped to trusted direct-chat cases.
- The implementation would need tests across shared prompt construction plus Telegram/Discord trusted DM paths, because current tests encode the existing metadata-on behavior.
