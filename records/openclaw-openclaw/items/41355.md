## Summary

Keep this issue open. Current main still lets the Discord adapter attach ThreadStarterBody without a previous-session guard, and the reply prompt path can still prepend that value as `[Thread starter - for context]` on follow-up turns. The same-author PR #50522 remains the implementation candidate with closing syntax, so maintainers should review, land, or supersede that PR before closing this issue.

## Best Possible Solution

Keep this issue open as the tracker. Review and land #50522 or an equivalent Discord adapter change that only supplies ThreadStarterBody for the first thread session turn, and add focused Discord regression coverage mirroring the Slack existing-session assertion. After that lands on main, close this as implemented by the landed commit.
