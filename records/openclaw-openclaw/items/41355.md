## Summary

Keep open. Current main still has the Discord source path described by the report: it resolves thread starter text for thread messages and passes `ThreadStarterBody` into the inbound context without guarding on the existing session timestamp. The reply prompt path can still prepend `[Thread starter - for context]` on follow-up turns when that field is present, and the linked same-author PR #50522 remains the implementation candidate.

## Best Possible Solution

Keep this issue open as the tracker. Review and land #50522 or an equivalent Discord adapter change that only supplies `ThreadStarterBody` on the first thread-session turn, then add focused Discord regression coverage mirroring the existing Slack assertion for existing thread sessions.
