## Summary

Keep this issue open. Current main still lets the Discord adapter pass `ThreadStarterBody` into inbound context without a first-turn guard, the shared reply path can still prepend the thread-starter marker on follow-up turns if that field is present, and the same-author implementation PR #50522 remains open and unmerged with closing syntax for this issue.

## Best Possible Solution

Keep this issue open as the canonical tracker. Review and land PR #50522 or an equivalent Discord adapter change that only supplies `ThreadStarterBody` on the first turn of the routed thread session, with focused Discord regression coverage. If maintainers want defense in depth, also tighten the shared follow-up `threadContextNote` fallback so accidental channel-supplied `ThreadStarterBody` cannot re-enter existing-session prompts.
