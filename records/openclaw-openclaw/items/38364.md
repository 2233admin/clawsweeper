## Summary

Current main still does not implement the central reply-metadata request. The no-output/NO_REPLY side has current suppression behavior and tests, but `chat.send`, `sessions.send`, Gateway `MsgContext` construction, and the Control UI send request still have no structured reply-to or parent-message path into agent sessions.

## What This Changes

Expose structured reply-to metadata from the Control UI/chat send flow to agent sessions, and avoid misleading no-output completion notices when no meaningful assistant reply exists.

## Best Possible Solution

Keep this issue open and implement a focused additive change: add validated reply metadata to `chat.send` and likely `sessions.send`, capture a selected parent/reply target in the Control UI composer, map that metadata in Gateway `chat.send` into the existing `MsgContext.ReplyTo*` fields with bounded sanitized quoted content, update protocol/docs/tests, and preserve current silent-output filtering.
