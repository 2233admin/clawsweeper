## Summary

Keep open. Current main has adjacent runtime handling for reasoning deltas, reasoning-only terminal classification, and vLLM/Nemotron visible-content request shaping, but it still does not implement the requested benchmark/reporting schema, extraction-source counters, separate visible-content versus reasoning-aware scores, or first-class profile/routing policy. The issue discussion also confirms this is intentionally broader than the already-closed scorer bug in #74017.

## What This Changes

The issue requests reasoning-aware output ingestion, visible-content compatibility reporting, and profile/routing controls for native reasoning models whose useful answer may be in `message.reasoning` while `message.content` is null or incomplete.

## Best Possible Solution

Keep this open for maintainer design. The best end state is an opt-in reporting-first slice that records content/reasoning presence and extraction source, computes visible-content, reasoning-aware, and strict-content score breakdowns, and defaults away from raw reasoning retention until redaction and artifact-retention rules are explicit; provider/profile routing can follow once that measurement contract is stable.

## Reproduction Assessment

Not applicable as a normal bug reproduction; this is a feature/reporting request. Current-main verification is high-confidence through exact symbol search plus source/docs/test inspection, while reproducing the reported benchmark percentages requires the reporter’s local fixtures or redacted equivalents.

## Solution Assessment

Unclear whether the exact proposed profile field names are the final contract, but keeping this open is the right current handling. A narrow automated fix would be premature because the remaining work crosses benchmark schema, provider/profile policy, runtime routing, and reasoning-data privacy.
