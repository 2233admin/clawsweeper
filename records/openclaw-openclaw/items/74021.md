## Summary

Keep this issue open. Current main has partial runtime handling for native reasoning streams, reasoning-only retry recovery, and vLLM/Nemotron visible-content request shaping, but it still lacks the requested benchmark/reporting schema, separate reasoning-aware versus visible-content scores, extraction-source counters, and first-class model/profile policy for opt-in reasoning-field final-answer extraction.

## What This Changes

The issue requests opt-in support and reporting for native reasoning-model outputs where useful answers may appear in reasoning fields rather than visible content.

## Best Possible Solution

Keep the issue open for maintainer design. The end state should first add a measurement/reporting layer that records content/reasoning presence, extraction source, visible-content compatibility, strict-output compatibility, and reasoning-aware quality under explicit opt-in, then add any runtime/provider-profile policy once the reporting semantics and privacy rules are settled.

## Reproduction Assessment

Not applicable as a feature request. Current-main static checks reproduce the absence of the requested schema and reporting fields, but the reporter's exact benchmark numbers cannot be reproduced from this checkout because the cited artifacts are local and not checked in.

## Solution Assessment

Unclear for implementation because there is no proposed patch here. The safest direction is the reporter's clarified split: reporting-only instrumentation with redacted fixtures first, then runtime/profile policy after maintainer review of reasoning-field visibility and retention.
