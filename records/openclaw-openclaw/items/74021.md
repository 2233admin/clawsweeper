## Summary

Keep open. Current main has adjacent runtime support for reasoning-field ingestion, reasoning-only retry/fallback, and vLLM/Nemotron visible-content request shaping, but it still lacks the requested benchmark/report schema, extraction-source counters, separate reasoning-aware versus visible-content scores, and first-class profile policy knobs.

## What This Changes

Request support for native reasoning-model outputs by tracking content versus reasoning-field extraction, enforcing visible final answers when configured, and reporting separate reasoning-aware, visible-content, and strict-content benchmark scores.

## Best Possible Solution

Keep this as a maintainer design item. The best path is an opt-in reporting-first slice that records content/reasoning presence and extraction source without raw reasoning by default, adds visible-content versus reasoning-aware versus strict-content score breakdowns, and only then extends provider/profile routing once privacy and compatibility semantics are documented.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature/reporting request. There is a high-confidence current-main verification path for the missing surfaces through exact field search and code/docs inspection, but the benchmark percentages require reporter-provided or regenerated artifacts.

## Solution Assessment

Yes, keeping this open for design is the best current handling. A narrow automated fix would be unsafe because the remaining work crosses report schema, model/profile contracts, runtime routing policy, and reasoning-data retention rules.
