## Summary

Keep open. Current main still has in-memory fallback attempt details, subsystem telemetry, session-store fallback notice state, and lifecycle/user-visible fallback notices, but it does not persist the requested ordered session JSONL custom event for the failed primary attempt before the `model-snapshot` entry. The prior Codex review comment remains accurate against current main.

## Best Possible Solution

Keep this open and add a narrow, sanitized transcript audit event at the fallback/reply-runner boundary. It should be written before the fallback model snapshot/retry and include selected primary provider/model, active fallback provider/model, failure reason/class, status/code when known, duration if available, and bounded redacted or hashed error detail. Add focused tests for JSONL ordering and redaction while preserving the existing subsystem logs, lifecycle events, status state, and user-visible fallback notices.

## Risks / Open Questions

- Persisting provider failure details into long-lived transcript JSONL needs strict redaction: no raw secrets, prompt/request payloads, stack traces, unbounded provider text, or raw request IDs.
- The implementation should guarantee deterministic ordering so the fallback-trigger audit entry lands before the fallback `model-snapshot` and retry, including around session-store updates and transcript rotation/compaction.
