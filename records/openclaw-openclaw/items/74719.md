## Summary

Keep open. The PR targets a real runtime-hardening gap in reply payload normalization, but it is not merge-ready: `chat.send` already schema-validates `message` as a string, the new blank-message sanitizer rejection would break supported attachment-only sends, and the reply normalizer hunk is written against older code instead of current main.

## What This Changes

The PR adds defensive guards around `.trim()` calls in reply normalization, chat send sanitization, chat abort parsing, and TUI stream assembly.

## Best Possible Solution

Land a rebased, narrower patch that fixes runtime narrowing in the current `normalizeReplyPayload` flow while preserving the existing `chat.send` attachment-only contract; redundant TUI/chat-abort optional chaining can be omitted unless a focused test proves a reachable crash.

## Reproduction Assessment

Yes. The attachment regression is reproducible from current gateway coverage that accepts `message: ""` with an attachment, and the remaining original crash has a focused code path in `normalizeReplyPayload` when media/channelData keeps a payload alive while `text` is a truthy non-string runtime value.

## Solution Assessment

No, not as submitted. The maintainable fix is to narrow reply payload text on the current main normalizer path and avoid moving blank-message rejection into `sanitizeChatSendMessageInput`, where it changes `chat.send` semantics.
