## Summary

Keep this PR open but do not merge it as written. The patch documents `bash -s --` as a stdin-isolation fix, but Bash documentation and a focused pipe smoke show child prompts still inherit and can consume the piped script stream; the linked installer bug needs a validated installer-side or consistently documented caller-side fix.

## What This Changes

This PR changes the Getting Started macOS/Linux installer snippet to use `curl ... | bash -s --` and adds a note saying those flags prevent stdin consumption when piping the installer.

## Best Possible Solution

Validate the #73814 failure against the hosted installer, then harden `scripts/install.sh` so interactive prompts and subprocesses use terminal-backed stdin or explicit noninteractive paths. After that, update all public install commands consistently and add focused installer stdin regression coverage.

## Reproduction Assessment

Yes for the PR's claimed fix: the Bash pipe smoke shows `bash -s --` does not prevent child stdin consumption. The full WSL2 installer hang from #73814 still needs live installer validation before selecting the final product fix.

## Solution Assessment

No. `bash -s --` is appropriate for passing arguments to a piped script, but it is not the narrowest maintainable fix for interactive stdin consumption; installer tty handling or a validated caller-side pattern should be chosen and applied consistently.
