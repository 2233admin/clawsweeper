## Summary

Keep this PR open. The linked installer problem is real enough to keep investigating, but the proposed documentation change is not correct: `bash -s --` does not isolate child-process stdin, and the PR would add a misleading guarantee while only changing one public install surface.

## What This Changes

This PR changes the Getting Started macOS/Linux install snippet to use `curl ... | bash -s --` and adds a note claiming those flags prevent stdin consumption when piping the installer.

## Best Possible Solution

Validate the installer failure against the hosted script, then harden `scripts/install.sh` so interactive prompts and subprocesses that need user input use terminal-backed stdin or explicit noninteractive paths. After that, update every public install command consistently and add focused installer stdin regression coverage.

## Reproduction Assessment

Yes for the PR's claimed fix: a focused Bash pipe smoke demonstrates that child stdin consumption is unchanged by `bash -s --`. The complete WSL2 installer hang from #73814 still needs live installer validation.

## Solution Assessment

No. Changing one quickstart command to `bash -s --` is not the best fix because it does not change child stdin inheritance and leaves other install docs inconsistent; installer hardening or a validated caller-side pattern is safer.
