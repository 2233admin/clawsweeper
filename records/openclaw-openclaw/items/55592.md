## Summary

Keep open. The reported bug is real on current main, but this PR is not merge-ready because its destructive truncate path maps UI sequence numbers against raw JSONL order while current history sequence numbers come from the active SessionManager branch, which can delete the wrong transcript range in branched sessions.

## What This Changes

This PR adds a `sessions.truncate` Gateway RPC with protocol and Swift model wiring, operator scope registration, server-side transcript truncation, and Control UI delete handling that calls the backend instead of only hiding messages locally.

## Best Possible Solution

Keep the PR open and require the truncate implementation to resolve the requested UI sequence through the active SessionManager branch entry or id before choosing the raw file cut point. The mergeable shape should include focused regression coverage for linear transcripts, branched transcripts, compaction entries, stale/no-op seq handling, active-run cleanup, webchat rejection, rollback, file-mode preservation, and Control UI rollback behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a tree transcript with an abandoned branch before the active branch: current `readSessionMessages()` assigns UI seq values from `getBranch()`, while the PR handler counts raw JSONL entries.

## Solution Assessment

No. The overall direction is right, but the proposed implementation is not the safest fix until destructive truncation uses the same active-branch semantics as chat history before rewriting the transcript.
