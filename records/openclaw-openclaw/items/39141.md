## Summary

Keep open. Current main still lacks the prerequisite `sessions.activity.get/list` API and activity state-change WebSocket contract from #39127, and the adjacent diagnostics/stuck-session recovery paths do not provide the requested opt-in watchdog with configurable alert classes, dedupe/cooldowns, operator delivery, or optional tokenized summaries.

## What This Changes

Add an opt-in gateway-side session activity watchdog that consumes the future sessions.activity API and emits configurable alerts for stale or blocked sessions.

## Best Possible Solution

Keep this open and sequence it after #39127 or an equivalent per-session activity/attention API with WebSocket state-change events. Then implement an opt-in gateway watchdog that consumes that contract, emits no-token structured alerts with configurable thresholds and dedupe/cooldowns, exposes safe UI/log/webhook/plugin delivery seams, gates optional tokenized summaries behind explicit config and rate limits, and adds focused protocol/runtime/docs tests. Treat #39305, #73327, and the open adjacent #53922 work as inputs, not replacements for the broader watchdog.
