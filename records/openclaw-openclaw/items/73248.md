## Summary

Keep open. Current main still presents skills through a flat `<available_skills>` catalog and budget/compact mitigations; it does not yet provide hierarchical categories, tag-based selection, or lazy skill discovery. Related #68734 is an open implementation candidate, while #52861 and #74100 overlap on token pressure without superseding this specific design request. https://github.com/openclaw/openclaw/issues/73248

## What This Changes

Request to reduce large skills-prompt overhead by replacing or augmenting flat all-skills XML discovery with hierarchical categories, lazy loading, or tag/group based skill selection.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A maintainable end state should add an additive, documented narrowing or discovery contract while preserving deterministic prompt ordering, existing allowlists, prompt budgets, plugin-bundled skills, and on-demand `SKILL.md` loading.

## Reproduction Assessment

Yes for the structural behavior: current source, docs, and tests show skills are still rendered as a flat `<available_skills>` catalog. No for the exact token measurements because no 80-plus-skill reproduction fixture or captured prompt was provided.

## Solution Assessment

Unclear at the design level. The issue identifies a real limitation, but maintainers still need to choose the narrowest compatible path among tag/category config, hook-based narrowing, lazy retrieval, or a skill graph.
