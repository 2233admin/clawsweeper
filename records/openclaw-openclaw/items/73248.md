## Summary

Keep open. Current main still exposes skills through a flat `<available_skills>` prompt block with budgeted compact/truncation controls and explicit allowlists, but it does not implement hierarchical categories, tag-based discovery, or lazy per-turn skill selection. Open #68734 is a relevant implementation candidate, while #52861 and #74100 overlap on prompt-token overhead without superseding this specific design request.

## What This Changes

Request to reduce large skills prompt overhead by replacing flat all-skills prompt injection with hierarchical, tagged, or lazy skill discovery.

## Best Possible Solution

Keep this as a core skills prompt/discovery design tracker. A maintainable end state should add an additive, documented narrowing or discovery contract while preserving deterministic skill ordering, explicit allowlists, prompt budgets, plugin-bundled skill eligibility, and on-demand SKILL.md loading.

## Reproduction Assessment

Yes for the structural behavior: source and docs show current main still builds a flat `<available_skills>` catalog and asks the model to scan descriptions. No for the exact reported token counts because no runnable 80-plus-skill fixture or prompt capture was provided.

## Solution Assessment

Unclear. The issue identifies a real scaling limitation, but the best solution still needs maintainer design choice between tags/categories, hook-based narrowing, lazy retrieval, or a skill graph; the safer direction is an additive contract with focused tests and docs.
