## Summary

Keep open. Current main still routes `systemctl --user` through inherited process environment, still classifies `Failed to connect to bus: Permission denied` as a user-bus-unavailable condition, and the provided GitHub context shows an open unmerged implementation candidate with closing syntax, so this issue should not be closed before that PR or an equivalent fix lands.

## What This Changes

The issue asks for `openclaw gateway status` and `openclaw gateway install` to recognize a working Ubuntu user-scope systemd service instead of reporting `systemctl --user` as unavailable.

## Best Possible Solution

Review and land #73888 or an equivalent narrow fix in `src/daemon/systemd.ts` that derives missing Linux user-scope DBus/XDG spawn environment for `systemctl --user`, preserves existing env and sudo target-user semantics, and adds regression coverage for status and install before closing the issue.

Yes. The issue has detailed Ubuntu steps and logs, and current main has a code-level reproduction path: mocked `systemctl --user` calls that return `Failed to connect to bus: Permission denied` still classify as user-bus-unavailable and drive the reported status/install behavior.
