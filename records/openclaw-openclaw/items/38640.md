## Risks / Open Questions

- The PR still has design risk around retrying non-idempotent prompt calls: a retry after partial provider processing can duplicate token spend or produce unexpected model-side effects.
- The PR's `signal` option does not line up with current `retryAsync`, which has no `AbortSignal` support and sleeps via non-abortable `sleep(delay)`. Review comments already flagged possible post-cancel prompt traffic.
- The requested behavior overlaps with open configurable retry work in #49376/#61443, so landing this narrower hard-coded wrapper could conflict with the chosen provider-level configuration model.
