## Summary

Keep open. Current main still lacks outbound Slack `reply_broadcast` support, and the pinned Slack SDK models the requested API parameter, so the PR is a useful implementation candidate. The branch is not merge-ready because it is stale against the current Slack message-tool discovery surface and still has concrete upload/media and docs/schema correctness issues.

## What This Changes

The PR branch adds Slack `replyBroadcast` plumbing across docs, action runtime, action adapter, message dispatch, message-tool schema, send payload construction, and focused tests for threaded Slack sends.

## Best Possible Solution

Keep this PR open as the narrow outbound implementation candidate. The desired end state is Slack-owned `replyBroadcast` support exposed through the current message-tool schema contribution API, mapped from `threadId`/`replyTo` to Slack `thread_ts`, emitted as `reply_broadcast: true` only for explicit threaded `chat.postMessage` sends, documented with current tool parameters, and either implemented or explicitly rejected for upload/media actions while #63904 continues tracking broader mirroring policy.
