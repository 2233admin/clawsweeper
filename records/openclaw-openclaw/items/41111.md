## Summary

Keep this issue open. Current main still registers and documents only `openclaw backup create` and `openclaw backup verify`, with no native `openclaw backup restore <archive>` command. The keepmyclaw ClawHub plugin linked in the discussion is useful prior art and a workaround, and open PR #59280 remains an implementation candidate, but the requested first-party backup restore workflow is not implemented on main.

## Best Possible Solution

Keep this issue open until OpenClaw ships a safe native `openclaw backup restore <archive>` workflow or maintainers consolidate it into a canonical backup/restore tracker. The practical next step is to review and either land or close PR #59280, with implementation in the backup CLI surface plus docs and focused tests for parsing, validation, dry-run, conflict handling, force/staging, and rollback behavior.

## Risks / Open Questions

- Closing now would mark a missing native, state-mutating backup lifecycle command as resolved while users still have no first-party `openclaw backup restore <archive>` on main.
- Treating keepmyclaw as the resolution would leave the requested core CLI workflow unimplemented, even though it is useful prior art and a workaround.
- A restore command needs maintainer-approved safety semantics for archive validation, conflict handling, dry-run/force behavior, staging, permissions, and rollback before it should ship.
