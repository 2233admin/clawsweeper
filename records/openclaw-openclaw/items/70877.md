## Summary

Keep this PR open. Current main still leaves generic tool-loop detection disabled by default in the detector and production hook path, and the PR contains a plausible focused bug fix backed by a concrete `web_fetch` retry-loop report. However, the current PR patch is incomplete: it changes the production resolver and one generated schema description, but it does not update the detector default, source schema/help/docs, or UI-visible boolean default, and its added direct no-config detector test conflicts with current implementation. The related reports in the provided context are overlapping open attempts rather than a merged or designated superseding fix.

## Best Possible Solution

Keep this PR open for maintainer review or consolidation with a designated loop-detection PR. The best fix is a minimal core change that enables generic loop detection by default in the authoritative runtime path, preserves explicit opt-out and global-to-agent inheritance semantics, updates source zod/schema help/docs plus generated schema baselines, and adds focused tests for no-config default-on, explicit false, global false with partial agent overrides, production hook behavior, and UI/schema default consistency.

## Risks / Open Questions

- Merging the current PR as shown is likely to fail the added direct no-config `detectToolCallLoop()` test because the detector itself still defaults omitted config to disabled.
- A runtime-only default risks schema/docs/UI drift: source docs and help text still say default false, and the config form falls back to `false` when a boolean schema default is absent.
- Default-on loop detection can create false positives for legitimate repeated calls, so explicit `enabled: false`, global false plus partial agent overrides, and production hook behavior need focused coverage.
