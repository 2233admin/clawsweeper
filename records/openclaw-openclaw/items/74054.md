## Summary

Keep open. Current main bounds a single Active Memory recall with a watchdog and generic runner retry caps, so the literal infinite-loop wording looks overstated. The remaining user problem is still valid: Active Memory has no plugin-owned retry budget or circuit breaker, timeout results are not cached, and each eligible prompt can start another recall against the same slow model.

## Best Possible Solution

Add an Active Memory-owned timeout circuit breaker in extensions/active-memory: track consecutive timeout results per agent/session/provider/model, skip recall for a configurable cooldown after the threshold, persist a clear status/debug line when recall is skipped, and document the new config. Keep generic provider retry/fallback behavior out of scope.
