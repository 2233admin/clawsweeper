## Summary

Keep open. Current main still loses structured MCP content fields on the HTTP tools/call path, and this PR is a focused implementation candidate for #68037, but the patch needs revision because it treats any object with a string type as a valid MCP content block instead of validating the SDK-required shape before passthrough.

## What This Changes

This PR broadens the MCP HTTP content normalizer to return text, image, audio, resource, and resource_link blocks, passes through typed objects, and adds helper-level tests for content variants and fallback cases.

## Best Possible Solution

Revise this PR, or a narrow replacement, so the normalizer preserves only SDK-valid MCP text, image, audio, embedded resource, and resource_link blocks, converts strings, unknown types, and malformed blocks to valid text fallback output, and adds an HTTP tools/call regression that proves image data and mimeType survive through the gateway.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Validate content blocks before passthrough:** `src/gateway/mcp-http.handlers.ts:25-30`
  - body: The new guard returns any object with a string `type` as `McpContentBlock`. A malformed text block such as `{ type: "text" }` now bypasses the old text fallback and is returned without the required `text` field; similarly, `resource_link` without required `name` and image/audio blocks without media fields are treated as valid. Since `tools/call` serializes this result directly, clients can still receive invalid MCP content. Validate the known MCP shapes before returning the original block, and fall back to text otherwise.
  - confidence: 0.93
