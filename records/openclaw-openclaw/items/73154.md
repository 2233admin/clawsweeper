## Summary

Keep open. Current main has in-memory queue preservation across session switches but still lacks localStorage-backed queue persistence, so the PR is not obsolete; however, the proposed persistence is not merge-ready because restored queued attachments lose their payloads and the branch predates the current session-switch helper.

## What This Changes

The PR adds a localStorage-backed chat queue persistence helper and wires queue save/restore calls into Control UI chat enqueue, drain, reconnect, and session-switch paths.

## Best Possible Solution

Implement queue persistence on top of the current app-render.helpers session-switch flow, add focused tests for refresh, reconnect, and session switching, and handle queued attachments explicitly through an appropriate payload storage/recovery path or clear user-visible unsupported-state behavior.

## Reproduction Assessment

Yes. Queue a message with an attachment while chat is busy, persist the queue, refresh the browser, then let the restored queue drain; current queue entries contain only attachment metadata while the send path requires a dataUrl, so the attachment is filtered out.

## Solution Assessment

No. Persisting the existing metadata-only chatQueue directly to localStorage is not sufficient; the durable layer needs to preserve or intentionally reject attachment payloads and be rebased onto current main's chatQueueBySession helper.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve attachment payloads when restoring the queue:** `ui/src/ui/chat-queue-persistence.ts:23-24`
  - body: Queued attachments are already stored as metadata-only objects, with their dataUrl kept in the in-memory payload map. Serializing the queue directly means a browser refresh restores attachments that buildApiAttachments drops, so text+file queued messages send without the file and file-only queued messages can drain as empty sends. Persist/recover the payloads through an appropriate storage path or mark those queued items as not restorable.
  - confidence: 0.91
