## Summary

Keep this PR open for maintainer review. Current main still lacks split tool-turn model routing, but the proposed patch is not ready to merge because it only swaps the `Model` argument after a tool result exists while the surrounding tool enablement, provider transport, auth profile, transcript policy, wrappers, diagnostics, context budgeting, and fallback behavior remain primary-model-bound.

## What This Changes

The PR adds a `model.tool` config field, resolver/schema support, a Pi embedded runner stream wrapper for tool-continuation calls, and unit tests for the new config and wrapper behavior.

## Best Possible Solution

Keep the PR open while maintainers decide whether this belongs in the core `agents.defaults.model` contract or behind a new plugin/API seam. If it stays core, the implementation should make the effective per-call model drive the whole model-call pipeline; if it moves to plugin design, #13225 is the canonical routing-seam discussion to continue.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Route tool turns through the matching transport:** `src/agents/pi-embedded-runner/run/attempt.ts:1287-1293`
  - body: The wrapper swaps only the `Model` argument while reusing the stream function that was already resolved for the primary provider. In a split-provider setup such as Ollama primary plus Anthropic tool model, tool-continuation calls can still go through primary-bound transport/auth/wrapper behavior instead of the configured tool provider.
  - confidence: 0.92
- **[P1] Enable the tool model before the first tool call:** `src/agents/pi-embedded-runner/run/attempt.ts:1280-1283`
  - body: This routing only activates after the last message is a tool result, but the first assistant turn still uses the primary model and `toolsEnabled` is still based on `params.model`. If the primary local model cannot use tools, no tool result is ever produced and the configured tool model is never exercised.
  - confidence: 0.9
- **[P2] Update the generated config contract:** `src/config/zod-schema.agent-model.ts:8`
  - body: Adding `tool` to the strict agent model schema changes the public config surface, but the PR does not update the generated config baseline/help/docs. The config docs check is expected to report drift and users would not see the new supported key in the generated contract.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff introduces a concrete provider/auth boundary risk because tool-model turns can reuse primary-provider transport and credentials.

Concerns:

- **[medium] Tool turns can cross provider/auth boundaries:** `src/agents/pi-embedded-runner/run/attempt.ts:1287`
  - body: The added wrapper changes the model object after the stream function, provider-specific wrappers, and API-key resolution have already been built for the primary provider. In cross-provider configs, model input intended for the configured tool provider can be sent through primary-provider transport/auth behavior or fail under the wrong credential boundary.
  - confidence: 0.88
