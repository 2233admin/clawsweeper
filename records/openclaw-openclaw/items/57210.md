## Summary

Keep open. Current main has useful pieces: a gateway `sessions.create` RPC, a Control UI session selector, session admin/rename/delete UI, and per-session history loading. But the central requested Control UI workflow is still missing: the visible plus/New session button is wired to send `/new`, which resets the current session rather than creating and switching to a separate dashboard chat session. The existing `sessions.create` backend is not used by the Control UI chat button.

## Best Possible Solution

Keep the issue open and implement the visible Control UI workflow on top of the existing `sessions.create` and `sessions.list` RPCs: the plus/New session action should create a fresh dashboard session, switch `state.sessionKey` to the returned key, sync the URL/settings, load empty history, refresh the session list, and add focused UI/controller tests. If session sidebar, import/export, and retention policy are too broad for one issue, split those into narrower follow-ups after the core create-and-switch behavior lands.
