## Summary

Keep open. Current main still lacks the requested configurable output-level filter surface, and the existing sanitizer/delivery paths only remove structured reasoning, known tags, tool-call scaffolding, and special tokens. The related open PR #60551 targets the same plain-text reasoning-preamble leak class, so this should stay with maintainer review rather than be closed or queued as a competing repair.

## What This Changes

The issue requests an opt-in per-agent output filter that strips configured regex or pattern matches from assistant messages before channel delivery.

## Best Possible Solution

Keep this as the tracker until maintainers either accept a version of #60551, add a documented opt-in config or plugin/API hook for outbound text filtering, or explicitly document that supported cleanup will remain heuristic/shared rather than user-defined regex filtering.

## Reproduction Assessment

Yes for the requested surface gap: current main has no outputFilters/outputFilter key in types, strict schema, docs, or delivery sanitization, and existing sanitizer tests preserve normal prose lead-ins. I did not live-run a multi-agent Discord scenario because static config and sanitizer checks are enough to show the requested feature is absent.

## Solution Assessment

Unclear. A configurable regex filter is plausible, but it creates a new user-facing config contract and false-positive risk; #60551’s narrower heuristic sanitizer is also plausible, so maintainers should choose the boundary before implementation.
