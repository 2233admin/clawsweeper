## Summary

Keep open. Current main still lacks the requested opt-in output-level regex/pattern filter; existing behavior covers separate reasoning payloads and known fixed-format scaffolding, not user-configurable stripping of untagged meta-planning text. Related PR #60551 and tracker #41494 show this is still active work rather than an obsolete request.

## What This Changes

Add an optional per-agent/global output-level pattern filter that strips configured leaked reasoning or meta-planning text from assistant messages before channel delivery.

## Best Possible Solution

Keep this issue open and converge on a conservative shared outbound filtering design: schema/docs for global and per-agent scope, deterministic ordering, code-block pass-through, safe pattern handling, and tests across shared sanitization, channel delivery, TTS, and plugin-hook interactions. If maintainers prefer a plugin approach, first document the hook pattern and confirm the hook metadata supports per-agent policy.

## Risks / Open Questions

- A configurable regex delivery filter can false-positive normal prose, quoted examples, code blocks, and multilingual replies unless it is opt-in, scoped, and well tested.
- Regex-based filtering may need safety constraints or bounded execution to avoid ReDoS-style slow patterns on long assistant outputs.
- There is overlap with open PR #60551 and tracker #41494, so maintainers should consolidate rather than queue a competing sanitizer path.
