## Summary

Keep this PR open. It targets a real Agents/Control UI label gap from #67988, but the patch is not merge-ready because the new catalog label path can collapse distinct catalog entries with the same display name into indistinguishable dropdown labels; the existing chat model display helper already preserves provider/id disambiguation for that case.

## What This Changes

The PR updates Agents/Control UI model option building to prefer gateway model catalog labels for configured and raw catalog model options, and adds tests for Anthropic Claude display names plus custom alias rendering.

## Best Possible Solution

Use the shared catalog display lookup behavior, or an equivalent helper, for Agents/Control UI model options so configured entries get catalog alias/name display metadata with duplicate-name disambiguation and raw fallback. Add focused tests for duplicate catalog display names, configured Anthropic refs from #67988, and intentional custom aliases.

## Reproduction Assessment

Yes. The linked bug reproduces from current main by configuring `agents.defaults.models` with `opus`/`sonnet` aliases and opening the Agents primary-model dropdown; source inspection shows `resolveConfiguredModels` labels those entries before catalog rows can replace them. The PR's added tests encode the happy-path reproduction, though I did not run them in this read-only review.

## Solution Assessment

No. The direction is right, but the current implementation should reuse the existing disambiguating chat model display lookup instead of introducing a separate helper that drops provider/id context for duplicate catalog names.
