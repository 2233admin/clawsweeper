## Summary

Keep #70634 open. Current main still uses a lane-aware FIFO command queue with `push` on enqueue and `shift` on dequeue, and the queue entry/options contract has no priority, origin, or provenance field. OpenClaw does have `InputProvenance` values that distinguish `external_user` from `inter_session`, and `sessions_send` marks inter-agent turns as `inter_session`, but that provenance is passed into the agent run rather than used by the session/global lane scheduler. The issue discussion's two-tier priority-queue workaround is therefore still relevant, and the related `sessions_send`/A2A reports do not supersede this specific human-message starvation request.

## Best Possible Solution

Keep the issue open for a core scheduler fix. The likely implementation is to extend the command lane enqueue contract with a small priority/origin hint derived from existing `InputProvenance` at channel/gateway boundaries, drain external-user work ahead of inter-session/background work while preserving FIFO within each tier, and add regression tests where queued `sessions_send` A2A turns do not prevent a later human-originated message from running promptly on the target session lane. Update queue docs once the behavior is defined.
