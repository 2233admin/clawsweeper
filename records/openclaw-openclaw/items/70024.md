## Summary

Keep open: current main still matches the reported gateway channel lifecycle bug, and PR #70056 is an open, unmerged closing PR for the same fix path. The issue should remain open until that PR or an equivalent focused fix lands.

## What This Changes

The issue asks for `stopChannel` timeout handling to mark the account stopped, clear stale task/abort store entries, record stop state, and allow later start or health-monitor recovery.

## Best Possible Solution

Review and land PR #70056, or an equivalent narrow gateway fix, so the timeout path clears only the stopped lifecycle's stale task/abort entries, records stopped runtime state, and updates the regression test to prove a restart can occur after a stop timeout.

## Reproduction Assessment

Yes. The current test fixture in `src/gateway/server-channels.test.ts` creates a start task that never settles, advances the 5 second stop timeout, then demonstrates the later start is skipped and the runtime remains `running: true`; source inspection confirms the same path on current main.

## Solution Assessment

Yes, the suggested fix direction is the narrowest maintainable solution if implemented race-safely: the timeout branch should stop advertising a running account and should clean up the stale task/abort entries that block restart.
