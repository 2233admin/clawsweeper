## Summary

Keep #70024 open. Current main still has the reported stop-timeout behavior: the timeout branch leaves the runtime as running, does not clear the task/abort maps on that path, and start still refuses to start while a task entry exists. The linked PR #70056 appears to be the focused fix but is still open, so the issue should remain as the canonical bug until a fix lands or maintainers explicitly choose the current behavior.

## Best Possible Solution

Keep the issue open and review PR #70056 or an equivalent gateway fix. The implementation should make the timeout state truthful enough for recovery while preserving the no-duplicate-transport invariant, then update server-channels coverage to assert the intended restart behavior after a timed-out stop.
