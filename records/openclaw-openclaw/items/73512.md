## Summary

Keep this PR open. It targets a real QMD scheduling complaint, but the proposed guard is not safe against current main because `embedIntervalMs > 0` can come from the resolved default, and the added test conflicts with the existing lexical/BM25-only contract in tests and docs.

## What This Changes

The PR changes QMD embed execution and timer scheduling so a positive resolved `embedIntervalMs` bypasses the lexical search-mode vector guard, and adds a regression test for `searchMode: "search"` with `embedInterval: "5m"`.

## Best Possible Solution

Settle the QMD lexical-mode contract first. If explicit `embedInterval` should opt lexical setups into embedding, preserve an explicit-config flag during backend config resolution, use that flag in both embed guards, and update the conflicting tests plus memory docs/schema help; otherwise keep the documented BM25-only behavior and close the paired bug/PR with clearer guidance to use `vsearch` or `query` for vector maintenance.

## Reproduction Assessment

Yes. Current main has a focused unit-level path: configure QMD with lexical `searchMode: "search"`, `interval: "0s"`, and `embedInterval: "5m"`, advance fake timers, and the existing test expects no `update` or `embed` calls; the PR adds the opposite expectation for the same effective setup.

## Solution Assessment

No. The PR's intended direction may be valid if maintainers want explicit opt-in embedding, but this implementation uses resolved `embedIntervalMs > 0` as intent even when the value came from the default and does not update the existing lexical-mode contract.
