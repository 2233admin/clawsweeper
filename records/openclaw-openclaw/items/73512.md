## Summary

Keep this PR open for maintainer review. The proposed direction may be valid for the paired bug, but current main intentionally treats QMD `searchMode: "search"` as BM25-only/no-embed, and the patch bypasses that contract using a resolved interval value that cannot distinguish explicit config from the default.

## What This Changes

The PR changes QMD embed execution and periodic embed timer scheduling so a positive resolved `embedIntervalMs` can bypass the lexical search-mode vector guard, and adds a regression test for `searchMode: "search"` with `embedInterval: "5m"`.

## Best Possible Solution

Settle the QMD lexical-mode contract first. If explicit `embedInterval` should opt users into embedding, preserve an explicit raw-config flag during backend config resolution, use that flag in both embed guards, update the conflicting tests and docs, and add the required user-facing changelog entry; otherwise keep lexical mode BM25-only and close or redirect the paired bug/PR with that behavior documented.

## Reproduction Assessment

Yes. The focused reproduction is unit-level: configure QMD with `searchMode: "search"`, `interval: "0s"`, and `embedInterval: "5m"`; current main expects no QMD update/embed calls, while the PR adds the opposite expectation for the same effective setup.

## Solution Assessment

No. The PR is too broad for the stated explicit-config behavior because it uses `embedIntervalMs > 0` after defaults are resolved; a safer implementation needs to carry explicitness from raw config and then update tests/docs around that contract.
