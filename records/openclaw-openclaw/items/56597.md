## Summary

Keep open. Current main still persists `/exec` directive fields into session entries and later-turn/run wiring still prefers session-stored exec values over configured exec defaults. Tests and docs currently describe this as persistent per-session behavior, so this needs an explicit product/security decision rather than cleanup closure. The related #59273 stale-config report is adjacent but does not supersede this persisted-directive precedence.

## Best Possible Solution

Decide the intended precedence for `/exec` session overrides versus updated `tools.exec` policy. If global policy should win for Discord/chat sessions, change the session persistence or resolution order for `execHost`, `execSecurity`, `execAsk`, and `execNode`, add regression coverage for a resumed Discord session after config changes, and update the `/exec` docs to describe the new reset/invalidation behavior. If persistence is intentional, add clearer operator-facing reset or hard-policy guidance instead of closing this as fixed.
