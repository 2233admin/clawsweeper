## Summary

Keep open. Current main still evaluates browser `act` input by wrapping `fn` as a parenthesized expression in both ref-based and page-level Playwright paths, so statement bodies beginning with `var`, `let`, or `const` still fail before execution. The public browser tool surface still exposes `act` with `kind: "evaluate"` and an `fn` string, with no statement mode or replacement `browser.exec`; the related closed #55598 request does not supersede this issue.

## Best Possible Solution

Keep this issue open and implement statement support in the browser plugin evaluate path, likely around `evaluateViaPlaywright`, by detecting expression versus statement bodies or adding a compatible explicit mode. Preserve current function-expression/ref behavior and return values, keep timeout/abort/navigation guards intact, and add focused tests for page and ref evaluate with `var`, `let`, `const`, async statement bodies, and existing expression/function inputs.

## Risks / Open Questions

- Auto-wrapping all evaluate strings could change existing expression and function-expression return semantics unless expression-vs-statement handling is deliberate.
- The evaluate path executes arbitrary page JavaScript, so a fix must preserve `browser.evaluateEnabled`, timeout clamping, abort disconnect behavior, and navigation/SSRF safety checks.
- Existing-session / Chrome MCP evaluate uses the separate `evaluate_script` adapter, so managed Playwright and existing-session behavior should be verified separately.
