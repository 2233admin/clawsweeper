## Summary

Keep open. Current main has implemented the hard context-window safety path for smaller-model switches: stale context budgets are cleared on model override, the target model's effective context budget is resolved before a run, and pre-prompt overflow can compact or truncate before submission. But this issue asks for a user-visible `/model` switch check and rate-limit-safe warning before the next request. The current `/model` handlers still acknowledge the switch without comparing current session usage against the target model, and OpenAI Codex gpt-5.4's current effective cap is 272k tokens, so the reported ~200k-token case would not necessarily trigger the existing overflow precheck even though it can still be risky for TPM/rate-limit pressure.

## Best Possible Solution

Keep this open and narrow the implementation to a shared model-switch context-risk warning. The best path is to resolve the target model's effective `contextTokens`/`maxTokens`, compare it with fresh session prompt usage when available, and append a clear warning to `/model` acknowledgements across WebChat/TUI and messaging channels when the current context is near or above a conservative target threshold. The warning should recommend `/compact` or `/new`; auto-compaction before switching should be a separate explicit product decision. Coordinate with #58957 for broader silent model-switch recovery gaps and with the already-closed smaller-window enforcement issues so the remaining work stays focused on pre-switch UX and rate-limit-safe policy.
