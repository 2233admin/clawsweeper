## Summary

Current main no longer sends Synology Chat inbound replies through the channel-prefixed `To` value. Production webhook registration dispatches through `dispatchSynologyChatInboundTurn`, which chooses the numeric `chatUserId` or webhook `user_id` as `sendUserId` and passes that value to `sendMessage`. The prior exact parser-hardening PR #50109 was closed unmerged, but the shipped inbound reply path now avoids the reported failure mode.

## Best Possible Solution

Close this issue as implemented by the shipped Synology Chat inbound reply flow. Keep the current numeric `sendUserId` delivery path, and only reopen or file a narrower follow-up if maintainers want low-level `sendMessage`/`parseNumericUserId` to accept prefixed strings even when callers bypass normal target normalization.
