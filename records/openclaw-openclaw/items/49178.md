## Summary

Keep open. Current main still lacks a reusable, platform-agnostic `@openclaw/gateway-client` package or equivalent public SDK; the Node and browser clients remain separate internal implementations, and the existing plugin gateway-runtime facade is not the requested Node/browser/RN client package. The Brani5 external client linked in discussion is useful prior art, but it does not satisfy an OpenClaw-supported public API surface.

## Best Possible Solution

Keep this open for maintainer API/product design. The best path is an official gateway client package or equivalent public SDK surface that shares the generated protocol contract, separates transport/storage/signing adapters, documents Node/browser/RN support boundaries, incorporates the current protocol docs and Brani5 reference implementation as input, and then migrates or parity-tests the CLI and Control UI clients against the shared core.
