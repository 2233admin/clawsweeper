## Summary

Keep open. Current main does not implement `openclaw update review`, so this PR is not obsolete; the proposed CLI/docs/tests are a useful implementation candidate. The latest patch addresses several earlier review concerns, but it still has a concrete git-install downgrade misclassification and lacks the required changelog entry for a user-facing feature.

## What This Changes

The PR adds an `openclaw update review` subcommand with typed JSON output, risk heuristics based on npm/GitHub release data, focused tests, and CLI documentation.

## Best Possible Solution

Keep this PR as the implementation candidate, repair the update-mode parity bug, add the required changelog entry, and then land it only if maintainers want a local CLI risk-review command while leaving broader release-stability metadata tracked separately.

## Reproduction Assessment

Yes for the blocking review finding: a unit-level mock with `installKind: "git"`, local `VERSION` greater than the npm target, and no package-mode switch exercises the PR's unconditional `isDowngrade` branch. Current main’s `updateCommand` scopes downgrade detection to non-git updates, so the mismatch is source-backed.

## Solution Assessment

No, not as currently patched. The feature direction is plausible, but the implementation should mirror `openclaw update` install-mode semantics by scoping downgrade guidance to package/switch-to-package paths and handling git freshness separately.
