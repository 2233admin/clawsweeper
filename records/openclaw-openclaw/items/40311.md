## Summary

Keep open. The requested Brave Goggles feature is still valid and is not implemented on current main, but this PR now targets the old core web-search implementation instead of the active Brave provider-owned schema and runtime, so it should not merge as-is.

## What This Changes

The PR branch adds a Brave-only optional `goggles` parameter to `web_search`, forwards it to Brave web and llm-context requests, separates cached results by Goggles value, and updates tests, docs, and changelog.

## Best Possible Solution

Land a rebased provider-owned port in `extensions/brave` that exposes per-query `goggles`, forwards it to both Brave endpoints, preserves exact Goggles values in cache separation, and updates the current provider docs, overview docs, focused tests, and credited changelog.

## Reproduction Assessment

Yes. A static current-main trace shows `createWebSearchTool` delegates to provider definitions, `resolveWebSearchDefinition` calls the provider’s `createTool`, and the active Brave provider has no `goggles` schema, URL forwarding, or cache-key handling.

## Solution Assessment

No. The capability is a good fit, but the PR’s current approach is no longer the narrow maintainable solution because it changes stale core files instead of the active Brave plugin-owned contract and runtime.
