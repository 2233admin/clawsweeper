## Summary

Keep this PR open. Current main still rejects self-hosted Firecrawl endpoints, so the PR is not obsolete, but the proposed endpoint relaxation is security-sensitive, overlaps #63877, #59666, and #63941, and has a concrete review finding around allowSelfHosted scoping.

## Best Possible Solution

Keep this PR open for maintainer review, but do not land it as-is. The best path is one canonical Firecrawl self-hosting fix for #63877 that preserves strict defaults, scopes any explicit self-hosted opt-in per operation if the flag remains, allows custom public hosts only over HTTPS, allows cleartext HTTP only for private or loopback targets after `assertHttpUrlTargetsPrivateNetwork`, routes private self-hosted endpoints through the trusted web-tools guard, keeps the official hosted endpoint on the strict path, updates manifest/docs/tests, and supersedes #59666 and #63941 as appropriate.

## Risks / Open Questions

- The PR's `allowSelfHosted` resolver leaks across `webSearch` and `webFetch`: enabling it for one operation relaxes endpoint validation for the other.
- The PR permits arbitrary HTTP endpoints when `allowSelfHosted` is true, without private/loopback validation, so the Firecrawl bearer token could be sent over cleartext to an unintended host.
- There are multiple open overlapping attempts (#59666 and #63941) plus the canonical bug #63877, so landing one narrow PR without reconciling the endpoint policy could leave conflicting behavior and docs.
