## Summary

Keep this PR open for maintainer review. Current main still rejects self-hosted Firecrawl endpoints, so the PR is not obsolete, but the proposed endpoint relaxation is security-sensitive, overlaps #63877, #59666, and #63941, and has a concrete review finding around allowSelfHosted scoping.

## Best Possible Solution

Keep this PR open, but do not land it as-is. The best path is one canonical Firecrawl self-hosting fix for #63877 that preserves strict defaults, scopes any explicit self-hosted opt-in per operation if the flag remains, permits custom public hosts only over HTTPS, permits cleartext HTTP only for private or loopback targets, routes trusted private endpoints through the appropriate web-tools guard, updates Firecrawl/web_fetch docs, and supersedes overlapping PRs #59666 and #63941 as appropriate.

## Risks / Open Questions

- The PR's allowSelfHosted resolver leaks across webSearch and webFetch config: enabling self-hosted mode for one operation also relaxes endpoint validation for the other.
- The PR allows arbitrary HTTP endpoints when allowSelfHosted is true, without private-network validation, which can send the Firecrawl bearer token over cleartext to a public host if misconfigured.
- The PR relaxes endpoint validation but does not switch private/local Firecrawl calls to withTrustedWebToolsEndpoint, so local self-hosted endpoints may still be blocked by the strict SSRF guard.
- The PR adds a new public config flag without updating Firecrawl or web_fetch docs and without focused tests for per-operation scoping or guarded-fetch routing.
