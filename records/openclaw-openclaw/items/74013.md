## Summary

Keep this PR open, but do not land it as-is. Current main still blocks self-hosted Firecrawl endpoints, so the PR is not obsolete; however, the proposed endpoint relaxation widens a credential-bearing network path without enough scoping or HTTP target validation, and it overlaps the canonical bug plus two other open Firecrawl self-hosting PRs.

## What This Changes

The PR adds Firecrawl `allowSelfHosted` config under `webSearch` and `webFetch`, passes it into endpoint resolution for search and scrape requests, and updates endpoint tests to allow custom self-hosted URLs when the flag is enabled.

## Best Possible Solution

Land one canonical Firecrawl self-hosting fix for #63877 that keeps official-host defaults strict, scopes any explicit opt-in per operation if that flag remains, allows custom public hosts only over HTTPS, allows HTTP only after private or loopback validation, routes private self-hosted endpoints through the trusted web-tools guard, and updates manifest, docs, and tests while superseding the overlapping PRs.

## Reproduction Assessment

Yes. The PR diff itself reproduces both review findings: `resolveFirecrawlAllowSelfHosted` ORs the search and fetch flags, and `resolveEndpoint` accepts any `http:` base URL when that global flag is true.

## Solution Assessment

No, not as-is. The requested direction is valid, but the safer solution should follow the existing SearXNG-style split: HTTPS for public custom hosts, `assertHttpUrlTargetsPrivateNetwork` for HTTP, and operation-scoped configuration semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Scope self-hosted opt-in to each Firecrawl operation:** `extensions/firecrawl/src/config.ts:202-206`
  - body: `resolveFirecrawlAllowSelfHosted` returns true when either the search or fetch config enables the flag. That means an operator enabling self-hosted mode for scrape can also relax search endpoint validation, and vice versa. Thread the relevant operation through the resolver instead of OR-ing both config branches.
  - confidence: 0.9
- **[P1] Restrict cleartext Firecrawl endpoints to private targets:** `extensions/firecrawl/src/firecrawl-client.ts:74`
  - body: With `allowSelfHosted` enabled, any `http:` Firecrawl `baseUrl` is accepted and the request still sends the Firecrawl bearer token. Keep public custom hosts on HTTPS and require private/loopback validation before allowing HTTP, matching the SearXNG pattern.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The patch broadens a credential-bearing Firecrawl endpoint path and needs stricter scoping plus private-network validation before merge.

Concerns:

- **[high] Self-hosted flag leaks across operations:** `extensions/firecrawl/src/config.ts:202`
  - body: A single `allowSelfHosted` boolean is resolved from both `webSearch` and `webFetch`, so enabling it for one operation silently relaxes endpoint validation for the other.
  - confidence: 0.9
- **[high] Cleartext HTTP can target arbitrary hosts:** `extensions/firecrawl/src/firecrawl-client.ts:74`
  - body: When the flag is true, the endpoint resolver accepts any `http:` host and `postFirecrawlJson` sends the API key as a bearer token, creating avoidable credential exposure for public or mistyped hosts.
  - confidence: 0.86
