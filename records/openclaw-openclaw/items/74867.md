## Summary

Keep open. Current main still lacks the requested SDK-facing environment discovery RPCs, while this PR is an active implementation for linked maintainer issue #74708. The patch is relevant and appears aligned with the additive/read-only Gateway protocol boundary, so this should proceed through normal maintainer API review and CI rather than conservative cleanup closure.

## What This Changes

The PR adds read-only `environments.list` and `environments.status` Gateway RPCs with SDK call-through types, read-scope classification, protocol schemas/generated Swift models, docs, changelog, and focused tests while leaving create/delete unsupported.

## Best Possible Solution

Land a narrow additive protocol change for read-only environment discovery if maintainers accept the API shape: typed Gateway schemas and handlers, SDK call-through, least-privilege read scoping, generated native models, docs, and tests that keep create/delete plus per-run environment overrides unsupported until provisioning semantics are defined.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: `oc.environments.list()` and `oc.environments.status(...)` throw explicit unsupported errors in the SDK, and the Gateway registry/handler map lacks matching `environments.*` methods.

## Solution Assessment

Yes, with maintainer review. The PR's approach is the narrowest maintainable path I found for the linked request: additive read-only RPCs backed by existing node catalog discovery, SDK types/call-through, scope classification, docs, generated models, and tests without enabling provisioning or run-time environment selection.
