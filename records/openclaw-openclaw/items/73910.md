## Summary

Keep open. Current main has fixed the unsupported Codex ACP timeout handoff, but the isolated CODEX_HOME auth problem remains and is already paired with open, unmerged PR #73943, which should be reviewed rather than closing this issue early.

## What This Changes

The issue asks OpenClaw-managed Codex ACP to either reuse or bridge valid Codex authentication for its isolated CODEX_HOME and stop forwarding timeout config options that Codex ACP rejects.

## Best Possible Solution

Keep the timeout guard already on main, then land or replace #73943 with a narrow Codex ACP auth solution that exposes only the required auth material or documents a supported isolated-home login flow, keeps generated config isolation, and adds regression coverage plus ACP docs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: current main still generates an isolated codex-home with only config.toml and no auth bridge, while the timeout rejection path is now covered by a Codex-specific guard and test.

## Solution Assessment

Unclear until the auth PR receives maintainer security review. The timeout guard is the right narrow fix for the config rejection, but the auth half should be limited to Codex ACP auth material and must not import broader host config, hooks, or runtime policy.
