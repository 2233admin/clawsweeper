## Summary

Keep open. Current main fixes the unsupported Codex ACP timeout handoff, but the isolated CODEX_HOME auth gap remains: the generated Codex ACP wrapper still points Codex at an OpenClaw-managed codex-home that only gets generated config, and the current regression test still asserts source Codex auth is not bridged. The open #73943 PR references this issue with closing syntax, so this issue should wait for that PR or a successor to merge.

## What This Changes

The issue asks OpenClaw-managed Codex ACP to reuse, bridge, or document Codex authentication for its isolated CODEX_HOME and to stop sending unsupported timeout config options to Codex ACP.

## Best Possible Solution

Review and land #73943 or a successor that bridges only the required Codex auth.json into the isolated Codex ACP home, or documents a supported isolated-home login flow, while keeping OpenClaw-generated config isolated and retaining the current Codex timeout guard with regression tests and public ACP docs.

## Reproduction Assessment

Yes. The issue includes a clear normal-CODEX_HOME versus isolated-CODEX_HOME reproduction, and current main statically reproduces the broken setup because the wrapper sets CODEX_HOME to the generated codex-home while prepareIsolatedCodexHome writes only config.toml.

## Solution Assessment

Yes, the narrow auth.json bridge or documented isolated-home login flow is the maintainable direction. Dropping the isolated CODEX_HOME entirely would be broader and could re-import user Codex config hooks that the current wrapper intentionally isolates.
