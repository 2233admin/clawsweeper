## Summary

Keep open. Current main appears to have fixed the unsupported Codex ACP timeout handoff by ignoring `timeout` and `timeout_seconds` for Codex ACP, but the isolated `CODEX_HOME` auth problem remains: the generated wrapper still forces a private `codex-home`, writes only a generated config, and current tests explicitly assert that source Codex auth is not copied or bridged. The related #70511 cleanup was for the native Codex app-server path, while this report concerns the explicit ACP/acpx Codex adapter route.

## Best Possible Solution

Keep this issue open and treat it as a real ACPX/Codex compatibility bug. Preserve the current main timeout guard, then decide the Codex ACP auth design explicitly: either add a safe, documented opt-in bridge/reuse path for existing Codex authentication into the isolated `CODEX_HOME`, or document and automate a setup flow for authenticating that isolated home without copying unsafe Codex config hooks.
