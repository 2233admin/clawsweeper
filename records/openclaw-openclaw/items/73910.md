## Summary

Keep open. Current main has the Codex ACP timeout guard, but the isolated Codex home still does not bridge or document authentication, and an open linked PR (#73943) uses closing syntax for the remaining auth fix, so this issue should stay open until that PR or a successor lands.

## What This Changes

The issue reports that OpenClaw-managed Codex ACP sessions fail because the ACPX wrapper uses an isolated Codex home without authentication and also sends timeout config that Codex ACP rejects.

## Best Possible Solution

Keep the issue open, keep the timeout guard already on main, and land or revise #73943 as a narrow Codex ACP auth fix that exposes only the needed Codex auth material or documents a supported isolated-home login flow while preserving config isolation and adding regression coverage plus ACP docs.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: prepareAcpxCodexAuthConfig with a source CODEX_HOME containing auth.json still generates an isolated codex-home containing only config.toml, and current tests assert no auth bridge; the timeout half no longer reproduces on main because runtime ignores Codex timeout config writes.

## Solution Assessment

Unclear until #73943 or a successor receives maintainer review. A narrow auth.json bridge or explicit isolated-home login flow is the maintainable direction, but it must avoid importing source config hooks or broader Codex state.
