## Summary

Keep open. Current main still has the shared health/status mismatch class for Discord. Fresh health snapshots can inherit runtime fields such as `running` and `connected`, but plain `openclaw health --json` is still a non-probe, cache-eligible RPC path, and health still builds the Discord summary from a runtime projection instead of the richer per-account status snapshot that carries resolved credential metadata. That leaves the reported `tokenSource: "none"` vs `tokenSource: "config"` mismatch actionable, and related health/status reports such as #59287 remain open as the broader fix cluster.

## Best Possible Solution

Keep this open and fix the shared health snapshot path, preferably in the same canonical PR as the related health/status mismatch cluster tracked by #59287. Health should reuse `buildChannelAccountSnapshot` or an equivalent safe helper so Discord receives runtime state plus plugin-owned credential-source metadata before `buildChannelSummary` runs, while preserving non-sensitive output filtering. Add a Discord regression test with a configured token and running/connected runtime snapshot expecting `running: true`, `connected: true`, and `tokenSource: "config"`; also cover the cache/startup behavior if `health --json` is intended to be monitoring-authoritative.
