## Summary

Keep open. Current main has the v2026.4.26 runtime-state improvement, so fresh gateway health snapshots can now inherit Discord runtime fields such as `running`/`connected`. The reported mismatch is still not fully fixed: plain `openclaw health --json` remains a non-probe, cache-eligible path, and the health summary still bypasses the per-account status snapshot builder that adds Discord credential-source fields. That means Discord health can still report `tokenSource: "none"` while `channels status --probe --json` reports the resolved config token source.

## Best Possible Solution

Keep this issue open. The best fix is to make the shared health snapshot path reuse `buildChannelAccountSnapshot` or otherwise merge plugin-owned credential-source fields from the resolved account before calling `buildChannelSummary`, while preserving the v2026.4.26 runtime-state projection. Add a Discord regression test with a configured token and running/connected runtime snapshot expecting health JSON to preserve both `running: true` and `tokenSource: "config"`; separately clarify or adjust cache semantics if `health --json` is expected to be authoritative for liveness.
