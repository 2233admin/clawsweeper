## Summary

Keep open. Current main still has the Discord health/status mismatch class: default `openclaw health --json` is non-probe and cache-eligible, and the health snapshot path still feeds `buildChannelSummary` from runtime-projected fields rather than the full account snapshot path used by `channels.status`. That leaves the reported `tokenSource: "none"` vs `tokenSource: "config"` mismatch actionable despite the partial runtime-backed health work shipped in v2026.4.26.

## What This Changes

`openclaw health --json` should report Discord channel runtime and credential-source state consistently with `openclaw channels status --probe --json`.

## Best Possible Solution

Keep this open and land one shared health/status snapshot fix. Health should reuse `buildChannelAccountSnapshot` or an equivalent safe helper so Discord receives runtime state plus plugin-owned credential-source metadata before `buildChannelSummary` runs, while preserving non-sensitive output filtering. Add a Discord regression test for a configured running/connected account expecting `running: true`, `connected: true`, and `tokenSource: "config"`, and coordinate with the related health/status mismatch cluster tracked by #59287, #46494, #46527, and #49758.
