## Summary

Keep #39999 open. Current `main` still reports global package update verification or post-update doctor failures without attempting an automatic reinstall of the known previous version. The PR remains a plausible core update-safety fix, but it needs to be rebased onto the current shared package update helper and should address the existing review concern that rollback must only run after the final package install attempt succeeds.

## Best Possible Solution

Keep the PR open and revise it against the current shared package update helper. The best implementation is to pass the known `beforeVersion` into package update handling, attempt `packageName@beforeVersion` rollback only after the final package-manager install step succeeds but install verification or post-update doctor fails, rerun doctor when the restored entrypoint exists, preserve the overall error result, and add focused helper/CLI tests for doctor failure, verification failure, rollback doctor behavior, and failed-install-with-no-rollback.

## Risks / Open Questions

- The submitted patch targets an older inline CLI flow; current `main` centralizes global package update behavior in `src/infra/package-update-steps.ts`, so a direct transplant would miss the shared helper and likely the gateway update path.
- Rollback must be guarded on a successful final install attempt, including the npm omit-optional fallback path, so network, registry, or version-resolution failures do not trigger a redundant reinstall of an unchanged prior version.
