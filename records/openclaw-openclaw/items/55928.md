## Summary

Keep this PR open. Current main does not implement the PR’s intended HTTP history guard: `handleSessionHistoryHttpRequest` still resolves the requested key, loads the session store, and serves history without checking subagent, cron, or ACP internal namespaces. The PR discussion also shows the contributor addressed the bare `cron:*` bypass and intentionally reverted the WebSocket-side block because internal subagent control still needs `chat.history`; that makes the remaining HTTP-only patch plausible rather than obsolete.

## Best Possible Solution

Review and land this focused HTTP guard, or replace it with an equivalent main-branch patch that rejects raw and canonicalized internal session keys before any HTTP history store read while leaving WebSocket `chat.history` available for the internal runtime flows that still depend on it.
