## Summary

Keep this PR open for maintainer review. Current main still has the HTTP history exposure, and the latest PR shape is a focused, correct access-control tightening for the HTTP surface; it should be reviewed/landed rather than closed or replaced by an autonomous repair job.

## What This Changes

The PR adds an internal gateway session-key predicate, rejects HTTP session-history requests for raw or canonical subagent, cron, and ACP keys, and adds regression coverage while preserving WebSocket `chat.history` for runtime flows.

## Best Possible Solution

Land this focused HTTP-only guard, or an equivalent main-branch patch, with a central internal-session predicate that covers raw and canonical subagent, cron, and ACP keys while preserving WebSocket history until there is a separate caller-trust model for runtime/internal flows.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed an internal entry such as `agent:main:cron:daily`, request `/sessions/cron%3Adaily/history` with `operator.read`, and observe current main canonicalize and read the transcript instead of returning 403.

## Solution Assessment

Yes. The current PR is the narrowest maintainable fix I found: it blocks the external HTTP read surface before store I/O, checks both raw and canonical keys, and documents that WebSocket `chat.history` remains available for existing internal runtime flows.
