## Summary

Keep open. Current main still emits read-based Skills guidance, normalizes available tool names without passing them into the skills section, and has tests/docs that continue to assert and describe the read path. Plugin tools can be registered by name and runtime tool names reach prompt construction, so the requested Skill-tool preference remains a real, narrow gap. The adjacent #63940 PR is still open and only changes selection wording; it keeps the read-based loading path.

## Best Possible Solution

Keep this issue open. The best path is a maintainer-approved prompt contract in `src/agents/system-prompt.ts`: pass normalized available-tool information and caller casing into `buildSkillsSection`, prefer invoking a registered Skill tool only under the chosen contract, preserve the existing read fallback otherwise, add focused coverage in `src/agents/system-prompt.test.ts`, and update `docs/concepts/system-prompt.md` / public docs to describe both loading paths.
