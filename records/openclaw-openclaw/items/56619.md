## Summary

Current main has a generic `openclaw pairing approve --notify` path, but WhatsApp does not implement a pairing approval notifier, so the requested WhatsApp sender confirmation remains missing. The related #56621 is adjacent pairing management UX, not a canonical replacement for this behavior.

## Best Possible Solution

Keep this open and implement the WhatsApp-owned pairing notification path in `extensions/whatsapp`: add a `notifyApproval`/text pairing adapter that sends `PAIRING_APPROVED_MESSAGE` via the active WhatsApp listener, pass `accountId` through the generic notify path, and add focused CLI/channel tests plus WhatsApp docs for `pairing approve whatsapp <CODE> --notify`. A separate decision can decide whether notification should become the default or stay opt-in.

## Risks / Open Questions

- The current CLI docs can make users expect `--notify` to work for every pairing channel, but WhatsApp currently no-ops because its adapter has no notifier.
- The notification path does not forward `accountId` from `pairing approve --account`, so a WhatsApp implementation should also preserve account routing for multi-account setups.
