## Summary

Keep this open. Current main still constructs Feishu WebSocket clients with an ambient proxy-derived agent whenever HTTP(S) proxy env is configured, and the regression tests still assert that old default. The issue is also paired with open, unmerged PR #65802 using closing syntax for the exact Feishu-plugin-local fix, so the right maintainer path is to review that PR instead of closing this issue early.

## What This Changes

The issue asks for Feishu WebSocket connections to bypass ambient HTTP(S)/SOCKS proxy environment variables by default, with any proxy use gated behind an explicit opt-in.

## Best Possible Solution

Review and land PR #65802, or an equivalent Feishu-plugin-local change, so Feishu WebSocket does not use ambient proxy env by default. If maintainers want to support proxied Feishu WS deployments, keep the opt-in explicit, test it in extensions/feishu/src/client.test.ts, and document the supported environment variable if it ships.

## Reproduction Assessment

Yes, for the actual reported environment with HTTP_PROXY or HTTPS_PROXY set: current source and tests show Feishu creates and passes a proxy agent into WSClient. The abbreviated ALL_PROXY-only step in the body is not by itself a proven current-main reproduction because hasEnvHttpProxyConfigured ignores ALL_PROXY-only envs, but the reporter's live environment included HTTP_PROXY and HTTPS_PROXY too.

## Solution Assessment

Yes. The best fix boundary is the Feishu plugin's WebSocket client construction, because this is channel-specific proxy policy; the shared ambient proxy helper can remain available for callers that intentionally opt into inherited process proxy settings.
