## Summary

Keep open. Current main clarifies that `non-main` is session-scoped, but it still cannot express the reported policy as a single default mode: dynamically created peer agents inherit the default sandbox mode and their own `agent:<id>:main` sessions compare equal to their per-agent main session, so those primary peer conversations are unsandboxed under `non-main`. This is a valid security-sensitive sandbox API/product decision, not an implemented, duplicate, or stale report.

## What This Changes

The issue asks for a way to make only the main agent bypass sandboxing while dynamic or peer agents remain sandboxed, or for clearer naming/docs around the existing session-scoped `non-main` behavior.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the reported policy explicitly, likely an additive `main-agent-only` mode or equivalent trusted-default-agent selector. The implementation should update config types/schema, runtime status, `sandbox explain`, docs, and focused tests for dynamic peer-agent main sessions while preserving existing `non-main` behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: set `agents.defaults.sandbox.mode` to `non-main`, create or model a dynamic Feishu agent without `agents.list[].sandbox`, then resolve `agent:feishu-ou_x:main`; current runtime resolves that same per-agent main key and returns `sandboxed=false`.

## Solution Assessment

Unclear until maintainers choose the contract. Documentation now reduces surprise, but it is not the best complete solution because the requested main-agent-only policy remains verbose and easy to misconfigure for dynamic agents; an additive mode with tests is the safer likely direction.
