## Summary

Keep open. Current main still implements `non-main` as a per-session comparison against each agent's own main session key, the schema still only accepts `off`, `non-main`, and `all`, and dynamic Feishu agents inherit defaults without adding sandbox overrides. The docs now warn about session-scoped behavior, but they do not provide the requested single-mode way to express “only the default main agent may bypass sandbox.”

## What This Changes

The issue asks for explicit or clearer sandbox mode semantics so only the primary main agent can bypass sandbox while dynamic peer agents' main sessions remain sandboxed without per-agent overrides.

## Best Possible Solution

Add a backward-compatible, schema-backed way to express the policy explicitly, likely an additive `main-agent-only` mode or equivalent default-agent-aware selector, then update `sandbox explain`, config docs, and regression tests for dynamic peer agent main sessions.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to set `agents.defaults.sandbox.mode` to `non-main` and evaluate `resolveSandboxRuntimeStatus` for `agent:feishu-ou_x:main`; the resolver derives that same peer agent's main session key and returns `sandboxed=false`.

## Solution Assessment

Unclear until maintainer/security review chooses the public contract. The narrowest maintainable direction appears to be an additive mode plus docs/tests, because docs-only still leaves the requested safe default verbose and easy to misconfigure.
