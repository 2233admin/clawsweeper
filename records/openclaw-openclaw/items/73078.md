## Summary

Keep open. Current main still implements `non-main` as a per-session comparison against the resolved main session for the session's own agent, and the schema still has no single `main-agent-only` or `non-main-session` mode. The docs now warn about the session-scoped meaning, but the requested main-agent-only sandbox policy remains unresolved and security-sensitive.

## What This Changes

The issue requests clearer or new sandbox mode semantics so only the primary `main` agent bypasses sandboxing while dynamic or peer agents remain sandboxed without verbose per-agent overrides.

## Best Possible Solution

Add a backward-compatible explicit policy for this use case, likely an additive `main-agent-only` mode or equivalent default-agent-aware selector, then update schema/types, `sandbox explain`, docs, and tests for dynamic peer-agent main sessions while preserving current `non-main` compatibility.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure `agents.defaults.sandbox.mode: "non-main"` and evaluate `resolveSandboxRuntimeStatus` for `agent:feishu-ou_x:main`; the resolver derives `feishu-ou_x` as the agent id, resolves the same per-agent main session key, and returns `sandboxed=false`.

## Solution Assessment

Unclear until maintainer/security review chooses the product contract. The narrowest maintainable direction appears to be an additive schema-backed mode plus docs/tests, because documentation alone leaves the main-agent-only policy verbose and easy to misconfigure.
