## Summary

Keep open. Current main still does not implement the proposed `sessions_broadcast` tool: the shipped session tool surface is `sessions_list`, `sessions_history`, `sessions_send`, `sessions_spawn`, `sessions_yield`, `subagents`, and `session_status`, and the docs still describe only single-target cross-session send. The author did address the first automated comments in PR head commit `36715207`, but the final diff still needs maintainer/security review before it could merge: it uses an unsupported `inputProvenance.kind`, does not update the same policy surfaces that treat `sessions_send`/`sessions_spawn` as sensitive, and imports a dependency name not declared by current package metadata.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is to rebase onto current main, decide whether a core broadcast primitive is the right product shape, and if so implement it as a policy-aware sibling to `sessions_send`: valid provenance semantics or an intentional protocol update, explicit subagent and Gateway HTTP policy handling, mutation classification, catalog/docs updates, and regression tests for visibility, delivery failures, denied surfaces, and scope filtering.
