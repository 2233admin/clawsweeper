## Summary

Keep this PR open. Current main still does not implement `sessions_broadcast`, and the PR remains a plausible implementation candidate rather than obsolete work. The author did address the first automated review points in head commit `36715207b7775a4ef71fe54608bb1ac0e3039d48`, but the final diff still needs maintainer/product and security review before merge because it adds a cross-session injection surface with unresolved provenance, policy, and dependency issues.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is to rebase onto current main, decide whether a core broadcast primitive is the right product shape, and if so implement it as a policy-aware sibling to `sessions_send`: valid provenance semantics or an explicit protocol update, subagent and Gateway HTTP policy handling, mutation classification, catalog/docs updates, and regression tests for visibility, delivery failures, denied surfaces, and scope filtering. Related #53228 is adjacent admin/UI notification work, not a full substitute for this agent session broadcast proposal.
