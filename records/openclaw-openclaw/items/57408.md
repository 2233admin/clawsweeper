## Summary

Keep #57408 open. Current main still blocks project-local `.env` from setting `OPENCLAW_CONFIG_PATH`, and the shipped latest tag has the same behavior. The linked PR discussion is useful evidence, but the fix is not present on main and the issue remains the canonical bug report.

## Best Possible Solution

Keep the issue open. The best fix is a maintainer-reviewed dotenv policy update: either allow a narrowly-scoped, tested `OPENCLAW_CONFIG_PATH` workspace `.env` override, probably absolute-path-only, while preserving the security block for dangerous runtime keys, or document that project-local `.env` cannot set this variable and point users to shell/global `.env` configuration instead.

## Risks / Open Questions

- Changing this area has a security tradeoff: allowing workspace `.env` to redirect config/state paths can affect trust boundaries. A narrow absolute-path-only allowance for `OPENCLAW_CONFIG_PATH`, with `OPENCLAW_STATE_DIR` still blocked or separately justified, appears safer than broadly unblocking `OPENCLAW_` workspace variables.
- There is a docs/product mismatch if maintainers intentionally want workspace `.env` to block all OpenClaw runtime controls; in that case the issue should stay open until docs and CLI messaging explicitly say project-local `.env` cannot set `OPENCLAW_CONFIG_PATH`.
