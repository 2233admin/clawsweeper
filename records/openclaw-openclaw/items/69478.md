## Summary

Keep open. Current main still only suppresses immediately consecutive duplicate system-event text, while this report is about retry/failover approval fanout where the same command can reappear after interleaved events and under fresh approval/run IDs. The reporter's #69482 cross-reference is a related but distinct approvals-schema bug, and open PR #73040 still uses closing syntax for this issue, so the right maintainer path is to review and adjust that implementation rather than close this issue now.

## What This Changes

Requests deduplicating or cancelling exec approval system events across heartbeat failover retries so identical commands do not generate repeated chat approval prompts under fresh IDs.

## Best Possible Solution

Keep this issue open until a fix lands that coalesces approval retry intent, not just adjacent queue text. The likely durable fix is to carry a stable command/session-level dedupe key from the exec approval or failover layer into system events, or cancel pending approval events before replacement runs enqueue new ones, with regression tests for same command under different approval/run IDs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to enqueue the same approval-facing event, interleave any other event, then enqueue the same command again under a fresh ID/context; current `enqueueSystemEvent` only compares `lastText`, so it accepts the later duplicate.

## Solution Assessment

No for current main. The requested direction is valid, but the best maintainable solution should use stable intent-level dedupe or cancellation because the problematic retry path changes ID-bearing text and context keys.
