## Summary

Keep open. Current main and the latest release still only suppress consecutive duplicate system-event text, while this report is about retry/failover fanout where the same user-visible approval can be requeued after interleaved events or under fresh approval IDs. The issue is also paired with open PR #73040, which explicitly uses closing syntax and targets this exact system-event dedupe gap, so the best maintainer path is to review/land or close that PR rather than closing the issue now.

## Best Possible Solution

Keep this issue open and use #73040, or a maintainer-approved equivalent, as the implementation path. The fix should add queue-wide dedupe in src/infra/system-events.ts for identical text within the same stable context, preserve lastContextKey semantics when skipping or when no context is supplied, and include regression tests for non-consecutive duplicates, same text under different contexts, skipped-duplicate context preservation, and omitted-context no-clobber behavior. If maintainers decide approval IDs themselves need stable coalescing, track that as a narrower follow-up after the system-event fix lands.

## Risks / Open Questions

- Current main can still enqueue identical system-event text again after any interleaved event changes lastText.

- A partial fix needs to preserve context-change semantics: skipped duplicates should not mutate lastContextKey, and context-less events should not clobber a prior stable context.
