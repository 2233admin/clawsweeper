## Summary

Keep open. Current main still only suppresses immediately consecutive duplicate system-event text, while this report covers retry/failover fanout where identical approval-facing events can recur after interleaved events and under fresh approval IDs. The latest release still has the same behavior, and open PR #73040 uses closing syntax for this issue, so the maintainer path is to review/land or close that PR rather than close this issue now.

## What This Changes

Requests deduplicating exec approval system events across heartbeat failover retries by stable run/context keys so identical commands do not fan out as repeated chat approval prompts.

## Best Possible Solution

Keep this issue open and use #73040, or a maintainer-approved equivalent, as the implementation path. The fix should add queue-wide dedupe in src/infra/system-events.ts for identical text within the same stable context, preserve lastContextKey semantics when skipping duplicates or when no context is supplied, and include regression tests for non-consecutive duplicates, same text under different contexts, skipped-duplicate context preservation, and omitted-context no-clobber behavior. If maintainers decide approval IDs themselves need stable coalescing, track that as a narrower follow-up after the system-event fix lands.
