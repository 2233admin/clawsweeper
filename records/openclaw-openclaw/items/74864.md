## Summary

Keep open. Current main still treats `abortedLastRun: true` as sufficient to resume a subagent session, and the post-release stuck-session recovery path does not persist a wedged marker or bound restart-to-restart recovery attempts for an accepted-but-rewedged run.

## What This Changes

The issue asks OpenClaw to stop subagent orphan recovery from repeatedly resuming a wedged subagent after gateway restarts and to provide safer cleanup or maintenance handling.

## Best Possible Solution

Add a persisted subagent recovery health gate: record repeated stuck/recovery-loop signals on the child session or run record, skip automatic orphan recovery after a small bounded threshold, mark the associated task/run lost or failed with a clear reason, and expose the condition through tasks maintenance or doctor without broadening the age-bound behavior tracked separately in #69778.

## Reproduction Assessment

Yes at the source level: seed an active subagent run whose child session entry has `abortedLastRun: true`, then `recoverOrphanedSubagentSessions` will attempt a synthetic resume. The full live death spiral still needs a wedged model turn or harness that re-orphans the same child shortly after accepted recovery.

## Solution Assessment

No current-main solution is not sufficient because it only bounds failed `callGateway` attempts inside one recovery scheduler and does not persist a wedged/recovered-again signal across gateway restarts. The narrow maintainable fix is a persisted attempt/tombstone guard tied to subagent orphan recovery and task maintenance.
