## Summary

Keep open. Current main still lacks both requested pieces: a configurable auto-TTS text-delivery policy for voice-note channels and speech-first guidance in the auto-TTS system prompt. The explicit TTS-tool duplicate-text fix and the older spoken-rewrite request do not cover this narrower auto-TTS behavior.

## What This Changes

Add an auto-TTS voice-only/caption delivery mode and strengthen the auto-TTS system prompt so generated replies are written for spoken output.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with default current behavior, apply suppression/caption behavior only for successful auto-TTS voice-note delivery while preserving text fallback on synthesis or native-voice send failure, and update the auto-TTS prompt hint with concise speech-first writing guidance without implementing the broader spoken-rewrite pipeline from #22028.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: enable auto-TTS for a final reply with successful synthesis on a voice-capable channel, and `maybeApplyTtsToPayload()` returns visible text plus voice media while config, docs, and commands expose no suppression setting.

## Solution Assessment

Yes, with scope control. A generic auto-TTS delivery policy plus a stronger prompt hint is the narrow maintainable solution; channel-specific hacks or a full spoken-rewrite pass would be broader than this issue requires.
