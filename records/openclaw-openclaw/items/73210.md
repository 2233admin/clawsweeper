## Summary

Keep open. Current main still lacks both requested pieces: a configurable auto-TTS text-delivery policy and prompt guidance that tells the model to write for spoken delivery. The related spoken-rewrite issue is closed as stale, and the duplicate-text fix only covers the explicit TTS tool path, so neither supersedes this request.

## What This Changes

Add an auto-TTS voice-only/text-delivery mode and strengthen the auto-TTS system prompt so generated replies are more natural when spoken aloud.

## Best Possible Solution

Add a backward-compatible `messages.tts` delivery policy with default current behavior, apply suppression only after successful auto-TTS audio generation and voice-capable delivery, preserve text fallback on synthesis or native-voice failure, and update the auto-TTS prompt hint with concise speech-first writing guidance without implementing the larger stale spoken-rewrite pipeline.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with auto-TTS enabled and synthesis succeeding, `maybeApplyTtsToPayload()` returns the original visible text plus audio metadata, while config and docs expose no text-suppression option.

## Solution Assessment

Yes, with scope control. A generic TTS delivery policy plus a stronger prompt hint is the narrow maintainable direction; channel-specific hacks or a full spoken-rewrite pass would be broader than this issue requires.
