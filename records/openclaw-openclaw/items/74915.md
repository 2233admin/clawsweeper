## Summary

Keep open. Current main still resolves the `web_fetch` fallback provider from bundled plugin candidates, even when a runtime/external provider was selected, which matches the reported failure mode and conflicts with the documented plugin contract for web fetch providers.

## What This Changes

The issue requests that `web_fetch` resolve explicitly configured external/non-bundled plugins that declare `contracts.webFetchProviders` and register a provider with `api.registerWebFetchProvider(...)`.

## Best Possible Solution

Update web-fetch provider resolution so an explicitly configured or runtime-selected provider can resolve across active/installed non-bundled plugin origins that declare `webFetchProviders`, with focused regression coverage for Readability-disabled external provider fallback and no bundled plugin id hard-coding.

## Reproduction Assessment

Yes. The issue body's reproduction maps directly to current source: external providers are filtered by bundled origin before provider id matching, and the Readability-disabled branch throws when provider resolution returns null.

## Solution Assessment

Yes. The narrow maintainable fix is to honor explicit/runtime web-fetch provider selection through the existing plugin provider runtime seams, while retaining bundled-only behavior only where it is truly required for compatibility or setup fast paths.

## Repair Work Prompt

Fix `web_fetch` so explicitly configured external/non-bundled web fetch providers work. The observable bug is that a plugin loaded from `plugins.load.paths` can declare `contracts.webFetchProviders` and call `api.registerWebFetchProvider`, but `tools.web.fetch.provider` is resolved only against bundled providers; with `tools.web.fetch.readability: false`, the tool fails with `Readability disabled and no fetch provider is available`. Keep the fix inside the generic web-fetch/plugin-provider seams; do not add owner-specific plugin ids or document external providers as unsupported. Likely direction: make `resolveWebFetchDefinition` honor `preferRuntimeProviders` similarly to web_search, using runtime/active provider resolution for explicitly selected non-bundled providers while preserving bundled Firecrawl auto-detect and setup/public-artifact fast paths. Add regression coverage for an external/runtime web-fetch provider selected by `tools.web.fetch.provider` and a Readability-disabled fallback path. Update or remove tests that currently assert bundled-only behavior when runtime providers are preferred. Add a user-facing changelog fix entry crediting `@mingmingtsao`; docs probably only need changes if behavior wording is clarified. Validate with targeted web-fetch/provider tests and the changed gate in Testbox before handoff.
