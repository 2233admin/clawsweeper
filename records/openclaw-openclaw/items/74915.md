## Summary

Keep this open. Current main still resolves web_fetch fallback candidates through the bundled-origin plugin path before explicit or runtime-selected provider matching, and the provided GitHub context shows #74962 as an open, unmerged PR with closing syntax for this issue.

## What This Changes

External plugins that declare and register webFetchProviders should be usable by web_fetch when explicitly configured or selected at runtime.

## Best Possible Solution

Review and land #74962 or an equivalent narrow fix that lets non-sandboxed, explicitly configured or runtime-selected web_fetch providers resolve across active non-bundled plugin origins while preserving the sandboxed bundled-provider restriction and existing SSRF and redirect protections.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction on current main: external provider candidates are filtered out by the bundled-origin resolver, then the Readability-disabled branch throws the reported no-provider error when no bundled fallback is available.

## Solution Assessment

Yes. The narrow maintainable direction is to adjust web_fetch provider resolution and add regression coverage for explicit and runtime-selected non-bundled providers, rather than documenting the already-exposed plugin contract as bundled-only.
