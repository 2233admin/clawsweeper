## Summary

Keep open. Current main still resolves `web_fetch` fallback providers through bundled-origin candidates even when runtime providers are preferred, and the issue is already paired with an open fix PR that references it with closing syntax.

## What This Changes

Allow `web_fetch` to resolve explicitly configured or runtime-selected `webFetchProviders` from external/non-bundled plugins instead of only bundled plugins.

## Best Possible Solution

Review and land the linked fix PR, or replace it with an equivalent narrow change that makes non-sandboxed `web_fetch` honor active external provider registrations while keeping bundled-only behavior where it is required for sandbox or setup safety.

## Reproduction Assessment

Yes. The issue body's reproduction maps directly to current source: external providers are filtered out by bundled-origin resolution before provider matching, and the Readability-disabled branch throws when no fallback provider is found.

## Solution Assessment

Yes for the proposed direction. The narrow maintainable solution is to use the existing runtime web-provider seam for non-sandboxed fetch provider selection and add regression coverage for external providers, rather than hard-coding external plugin ids in core.
