## Summary

Keep open. Current main still filters `web_fetch` provider resolution through bundled plugin candidates, which matches the reported failure for external `webFetchProviders`; an open linked implementation PR already references this issue with closing syntax, so this issue should remain open until that PR lands or is closed.

## What This Changes

External plugin `webFetchProviders` should be resolvable by `web_fetch` when explicitly configured or runtime-selected, instead of being limited to bundled provider candidates.

## Best Possible Solution

Review and land PR #74962 or an equivalent narrow fix that resolves non-sandboxed explicit/runtime web-fetch providers across active installed plugin origins, with focused tests for Readability-disabled external provider fallback and retained sandbox restrictions.

## Reproduction Assessment

Yes. The issue’s reproduction maps directly to current source: external candidates are filtered by bundled origin before provider matching, and the Readability-disabled branch throws when no provider fallback resolves.

## Solution Assessment

Yes. The narrow maintainable solution is to use the existing runtime/provider resolution seams for non-sandboxed explicit or runtime-selected web-fetch providers, while keeping bundled-only behavior only where the sandbox or setup path requires it.
