## Summary

Keep this open. Current main still filters web_fetch provider candidates to bundled plugin origins before explicit or runtime provider matching, and the provided GitHub context shows an open focused PR (#74962) with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

Allow web_fetch to resolve explicitly configured or runtime-selected external webFetchProviders instead of limiting provider lookup to bundled plugins.

## Best Possible Solution

Review and land #74962 or an equivalent narrow fix that lets non-sandboxed, explicitly configured or runtime-selected web_fetch providers resolve across active non-bundled plugin origins while keeping sandboxed web_fetch restricted to bundled providers and preserving existing guarded-fetch protections.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: external candidates are filtered out by the bundled-origin resolver, then the Readability-disabled path throws the reported no-provider error when provider fallback resolution returns null.

## Solution Assessment

Yes. The narrow maintainable solution is to adjust web_fetch provider resolution and add regression coverage for explicit/runtime non-bundled providers, rather than documenting the public plugin contract as bundled-only.
