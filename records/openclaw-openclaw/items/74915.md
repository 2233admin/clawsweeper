## Summary

Keep this issue open. Current main still resolves the `web_fetch` fallback provider from bundled-origin provider candidates, which matches the reported external-provider failure, and the issue is already paired with open PR #74962 using closing syntax.

## What This Changes

The issue reports that external plugins declaring `contracts.webFetchProviders` and registering `api.registerWebFetchProvider(...)` are ignored by `web_fetch` when an external provider is selected and Readability is disabled.

## Best Possible Solution

Review and either land or close PR #74962. The desired end state is that non-sandboxed, explicitly configured or runtime-selected web-fetch providers resolve across active installed plugin origins that declare `webFetchProviders`, while sandboxed behavior and fetch safety checks remain deliberate.

## Reproduction Assessment

Yes. The issue body's configuration maps directly to current source: external providers are filtered out by bundled-origin lookup, and with `readability: false` the fallback path throws the reported no-provider error.

## Solution Assessment

Yes. The narrow maintainable solution is to honor explicit/runtime web-fetch provider selection through the existing plugin provider seams, with tests updated for non-sandboxed external providers and unchanged sandbox policy.
