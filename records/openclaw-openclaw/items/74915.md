## Summary

Keep open. Current main still filters web_fetch provider resolution through bundled-origin candidates before matching an explicit or runtime-selected provider, and the linked open PR #74962 is already the active implementation candidate for this issue.

## What This Changes

The issue requests that web_fetch resolve explicitly configured or runtime-selected web fetch providers from active non-bundled plugins that declare contracts.webFetchProviders and register via api.registerWebFetchProvider.

## Best Possible Solution

Land #74962 or an equivalent narrow resolver/test change that lets non-sandboxed web_fetch use explicitly configured or runtime-selected providers from active non-bundled plugin origins while preserving sandboxed provider restrictions.

## Reproduction Assessment

Yes. The issue has a high-confidence source-level reproduction on current main: configure an external provider with Readability disabled, and the current bundled-origin candidate filter removes that provider before the no-provider error path runs.

## Solution Assessment

Yes. The maintainable fix is to adjust web_fetch provider candidate resolution with focused regression coverage, rather than documenting webFetchProviders as bundled-only after exposing the manifest contract and registration API to plugins.
