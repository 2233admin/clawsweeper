## Summary

Keep this PR open. Current main still does not implement the reconnect-grace behavior: after startup connect grace, a running channel with `connected: false` is still evaluated as unhealthy immediately and the health monitor can restart it. Main now has related Discord hardening, typed `lastDisconnect` data, stale-socket improvements, and restart cooldown/hour caps, but those do not replace the PR’s central delayed-intervention behavior.

## Best Possible Solution

Keep this PR open as a still-useful implementation candidate. The best path is to rebase it onto current main, add a generic reconnect-grace window to `ChannelHealthPolicy`/`ChannelHealthTimingPolicy`, consume the already-typed Discord `lastDisconnect.at` data with a clear fallback policy, update both the monitor and readiness call sites, preserve current stale-socket and restart-cap behavior, and add focused policy/monitor/readiness tests for recent versus expired disconnects.
