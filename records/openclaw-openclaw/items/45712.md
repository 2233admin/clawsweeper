## Summary

Keep this PR open. Current main still does not implement the PR’s central reconnect-grace behavior: a running channel with `connected: false` is still marked unhealthy immediately after startup connect grace, and the health monitor still restarts it. Related landed Discord hardening is narrower restart-throttling/lifecycle cleanup, not a replacement for reconnect grace.

## Best Possible Solution

Keep this PR open as a still-useful implementation candidate. The best path is to rebase it onto current main, add reconnect-grace handling to the generic channel health policy using the already-typed Discord `lastDisconnect` data, preserve current stale-socket and restart-throttling behavior, update readiness intentionally, and add focused policy/monitor tests proving recent disconnects do not trigger provider teardown while older disconnects still recover.
