## Summary

Keep open. Current main still constructs the pre-compaction memory flush with an empty `transcriptPrompt`, converts it into a runtime-only submission whose provider-facing prompt is `""`, and explicitly bypasses the blank-prompt skip guard for runtime-only turns. The related #75313 report is adjacent blank-message provider strictness, but it is a different session-file repair path rather than a duplicate of this memory-flush handoff.

## What This Changes

The issue reports that pre-compaction memory flush sends an empty user message to Anthropic during auto-compaction, causing a provider 400 and session reset.

## Best Possible Solution

Keep the memory-flush instructions hidden in runtime/system context, but make the submitted provider-facing turn non-empty and valid for strict providers, then add regression coverage that follows the memory-flush handoff through the embedded attempt path and, if needed, the Anthropic request-shape boundary.

## Reproduction Assessment

Yes. Current main has a high-confidence source reproduction path: memory flush passes `transcriptPrompt: ""`, runtime-context resolution returns `prompt: ""` with `runtimeOnly: true`, the skip guard exempts runtime-only turns, and the attempt runner submits that blank prompt.

## Solution Assessment

No for the current implementation. The narrow maintainable fix is to preserve silent runtime-only memory flush behavior while replacing the provider-facing blank user prompt with a non-empty housekeeping/sentinel prompt or equivalent provider-safe submission shape.
