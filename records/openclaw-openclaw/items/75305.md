## Summary

Keep open. Current main still routes the pre-compaction memory flush through a runtime-only path that turns the real flush prompt into `prompt: ""`, then submits that empty prompt to the embedded Pi session. The earlier blank-prompt guard covers visible user submissions but explicitly exempts runtime-only turns, so it does not address this memory-flush failure mode.

## What This Changes

The issue reports that automatic pre-compaction memory flushing can send an empty user message to Anthropic, causing a 400 error and an unexpected Telegram session reset.

## Best Possible Solution

Fix the shared runtime-only embedded-runner submission path so memory flushes can carry hidden/system runtime context without creating an empty user turn for providers, while preserving the visible blank-prompt guard and the silent pre-compaction memory behavior.

## Reproduction Assessment

Yes. A memory-triggered embedded Pi run with a non-empty flush plan prompt and `transcriptPrompt: ""` deterministically reaches the runtime-only branch that calls `activeSession.prompt("")`; this can be reproduced in a focused unit or integration test without live Telegram.

## Solution Assessment

Yes for a narrow repair direction: the maintainable fix belongs at the shared runtime-only prompt boundary, not as a Telegram-only or Anthropic-only workaround. The change should make the provider-facing turn safe while keeping runtime context hidden from the user transcript.
