## Summary

Keep open. Current main still has the reported compaction-retry delivery race: the runner drains block replies before waiting for the retry, but it does not drain again after the retry wait unblocks; lifecycle resolution still happens before the async block/channel flush completes, and current tests preserve that ordering. The closed source PR #47338 was not merged, and the open final-payload PR #62157 is adjacent rather than a fix for this retry ordering problem.

## What This Changes

The issue reports that auto-compaction rollover can record a retry-generated assistant reply in the transcript without delivering it to Discord, Telegram, or another messaging channel.

## Best Possible Solution

Add a retry-specific ordering guarantee so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, with regression coverage for pending compaction retry plus an async `onBlockReplyFlush`.
