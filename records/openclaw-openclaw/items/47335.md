## Summary

Keep open. Current main still exposes the reported compaction-retry delivery race: the runner drains the block reply pipeline before the retry wait, lifecycle resolution can unblock that wait before the async channel/block flush finishes, and current tests still lock in that ordering.

## What This Changes

The issue asks OpenClaw to ensure replies generated during compaction session rollover retries are delivered to messaging channels, likely by draining the block reply pipeline after retry completion.

## Best Possible Solution

Add a retry-specific ordering guarantee so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, while preserving the broader resolve-before-slow-channel-flush behavior from #59308.

## Repair Work Prompt

Fix the compaction-retry reply delivery race reported in https://github.com/openclaw/openclaw/issues/47335. Current main flushes `onBlockReplyFlush` before `waitForCompactionRetryWithAggregateTimeout()` in `src/agents/pi-embedded-runner/run/attempt.ts`, but retry-generated block replies can still be in flight because `handleAgentEnd()` resolves compaction retry before awaiting the async block/channel flush. Add a retry-specific ordering guarantee so the attempt cannot finish while retry-generated block replies are still draining. Keep the fix narrow: do not broadly make slow channel delivery block ordinary compaction completion, and preserve the intent of #59308. Use #47338 as source context, but account for current main's #59308 ordering and the adjacent #62157 final-payload work. Add or update focused regression coverage in the embedded lifecycle/runner payload tests. Add a user-facing changelog fix entry with `Thanks @frankhli843` if runtime behavior changes. Do not change channel-specific transports or compaction timeout semantics beyond the needed delivery barrier.
