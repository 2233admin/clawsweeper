## Summary

Keep open. Current main still has the relevant compaction-retry ordering gap: the runner drains block replies before the retry wait, lifecycle resolution can unblock that wait before the async reply/channel flush finishes, and the existing tests still preserve that ordering. The issue is source-verifiable and still actionable rather than stale or implemented.

## What This Changes

Ensure session rollover after compaction retry cannot finish before retry-generated replies have been delivered to messaging channels.

## Best Possible Solution

Add a retry-specific delivery barrier so compaction retry completion cannot let the attempt finish while retry-generated block replies are still draining, while preserving the general resolve-before-slow-channel-flush behavior for non-retry compaction paths.

## Repair Work Prompt

Fix openclaw/openclaw#47335. Current main can let a compaction retry resolve the runner wait before retry-generated block replies have finished flushing to the channel. Implement a retry-specific delivery ordering guarantee so the attempt cannot finish before retry reply blocks drain. Preserve #59308's broader behavior: ordinary slow channel flushes must not cause the compaction aggregate wait to time out again. Do not change channel-specific transport code, compaction policy, plugin APIs, or unrelated payload dedupe behavior. Consider either an idempotent post-retry flush in `runEmbeddedAttempt` only when a retry actually occurred, or a small lifecycle/subscription barrier that represents retry reply flush completion. Add regression coverage that simulates compaction retry resolution while `onBlockReplyFlush` is still pending and proves the retry path waits for delivery, plus coverage that the normal slow-channel ordering from #59308 still holds. If a user-facing changelog entry is required, add one single-line fix entry with appropriate thanks to @frankhli843 and @Ryce, following the current changelog section style.
