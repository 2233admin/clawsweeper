## Summary

Keep this PR open. Current main still lacks stale-running cron session recovery, so the change is not obsolete, but the proposed patch is not merge-ready because it only repairs `:run:` alias rows, skips recovery when retention pruning is disabled, and needs maintainer confirmation on the terminal status for interrupted work.

## What This Changes

The PR adds a two-hour stale-running recovery pass to `sweepCronRunSessions`, returns an additive `orphansRecovered` count, and adds a unit test for marking old cron `:run:` rows as done.

## Best Possible Solution

Land a rebased cron session recovery patch that runs independently of retention pruning, repairs the stable base cron session row plus any legacy `:run:` aliases, uses the maintainer-approved terminal status for interrupted work, and adds focused regression coverage for retention-disabled, base-key, stale-versus-active, and short-retention cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Recover stable cron session rows too:** `src/cron/session-reaper.ts:108`
  - body: This loop only reaches `isCronRunSessionKey` entries, so the recovery update can flip a hidden `:run:` alias while leaving the stable `agent:<agent>:cron:<job>` row stuck as `running`. Current isolated cron persistence writes lifecycle state to that base key, and session listing hides `:run:` aliases, so the visible row can remain stale after the sweep.
  - confidence: 0.91
- **[P2] Run recovery when retention is disabled:** `src/cron/session-reaper.ts:83-85`
  - body: When `cron.sessionRetention` is `false`, this early return skips the orphan scan entirely. That setting disables pruning, but stale crashed sessions still need status recovery, so users who keep full cron history would never get the new behavior.
  - confidence: 0.94
