## Summary

Keep open. Current main still intentionally keeps browser WebChat-originated chat.send replies on the internal WebChat surface unless explicit external delivery is requested, and the Control UI/WebChat send path still sends deliver: false. The requested WeChat delivery behavior conflicts with the current WebChat route-safety policy, so this needs maintainer product/security judgment rather than conservative cleanup closure.

## Best Possible Solution

Keep this open for maintainer judgment. The best path is to decide whether WebChat should remain local-only for safety, or whether OpenClaw should add an explicit opt-in external delivery affordance for selected channel-scoped sessions. If the opt-in path is chosen, update the WebChat UI request path, chat.send route resolution, dispatcher tests, session-route tests, and WebChat/channel-routing docs together while preserving the #39175 leakage safeguards.
