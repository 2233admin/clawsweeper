## Summary

Keep open. Current main still matches the reported macOS Talk routing gap: macOS reads Gateway Talk config but routes non-local/non-ElevenLabs providers to system voice instead of using `talk.speak`, while an open unmerged PR already references this issue with closing syntax and should be reviewed rather than spawning duplicate cleanup.

## What This Changes

The issue requests that macOS Talk mode use Gateway `talk.speak` for configured TTS providers, respect `talk.providers`, and expose provider selection instead of always falling back to local macOS speech.

## Best Possible Solution

Land a narrow macOS runtime fix that sends every configured remote Talk provider through Gateway `talk.speak`, keeps explicit `system` and `mlx` local paths, preserves any deliberate ElevenLabs optimized path only with a gateway fallback, and adds Swift coverage for an `openai` or custom provider. Provider picker UI can be handled as a follow-up if maintainers want first-class settings beyond the schema-driven config form.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: set `talk.provider` to a configured provider such as `openai`, fetch `talk.config`, then observe current macOS `playbackPlan` returns `.systemVoiceOnly` for that provider and never calls `playGatewayTalkSpeak`.

## Solution Assessment

Yes for the requested direction, but not yet implemented on main. Routing unsupported/non-local macOS Talk providers through existing Gateway `talk.speak` is the narrowest maintainable fix because the Gateway method, docs, and Android path already establish that contract.
