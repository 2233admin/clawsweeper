## Summary

Keep this PR open. It addresses a real TUI shutdown bug and the exact-head checks are green, but it is not merge-ready because the new exit cleanup can still leave a pending status timeout able to recreate the loader interval after `tui.stop()`.

## What This Changes

The PR adds POSIX `SIGHUP` handling for TUI shutdown, an exit-only bounded drain/stop helper, status UI teardown before exit cleanup, regression tests, and a changelog entry for the terminal-hangup bug.

## Best Possible Solution

Land this PR after a narrow follow-up ensures shutdown is final for all status UI paths: pending status timeouts and late connection/status callbacks should be cleared or ignored after exit is requested, while preserving the new SIGHUP handling, bounded drain, and unchanged normal Ctrl+C behavior.

## Reproduction Assessment

Yes. The linked report gives concrete macOS terminal-close and gateway-restart steps, and current `main` confirms the missing SIGHUP handler plus unbounded exit drain; the remaining PR gap can be reproduced with a focused unit test that shuts down while a busy status TTL is pending.

## Solution Assessment

No, not yet. The PR's SIGHUP and bounded drain approach is the right narrow direction, but the exit cleanup must also prevent delayed status rendering from restarting loader timers after `tui.stop()`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Prevent status timeouts from restarting the loader:** `src/tui/tui.ts:967-976`
  - body: `stopStatusUi()` stops the current loader and timers, but it leaves `statusTimeout` active. If a terminal closes while the TUI is busy and a 4s/5s status TTL is pending, that callback can run after `tui.stop()`, call `renderStatus()`, and create a fresh `Loader` interval. That preserves the orphaned spinner loop this PR is trying to bound, so clear/null the timeout and/or guard status rendering once exit is requested.
  - confidence: 0.88
