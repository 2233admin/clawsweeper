## Summary

Current main still does not implement custom HTTP headers for CLI Gateway WebSocket connections. The request remains valid for remote gateways behind authenticating reverse proxies, and the related implementation PR #49646 was closed unmerged. This should stay open for a core Gateway CLI/client/config fix, with maintainer review on safe header policy and redaction.

## Best Possible Solution

Keep this open and implement the feature in core Gateway CLI/client/config after a maintainer sets the safe header policy. The implementation should support repeatable CLI headers plus env/config sources with explicit CLI > env > config precedence, validate header names and reserved names, treat all configured header values as sensitive, thread the map through CallGatewayBaseOptions and GatewayClientOptions into ws ClientOptions.headers, audit all Gateway-backed CLI wrappers including devices/nodes/shared gateway RPC, and update docs/config help/tests.

## Risks / Open Questions

- Header values may contain service-token secrets, so implementation must redact values anywhere config snapshots, diagnostics, logs, errors, or status output can surface them.
- Allowing arbitrary WebSocket upgrade headers needs a conservative policy for reserved or dangerous names such as Host, Connection, Upgrade, Sec-WebSocket-*, Origin, Authorization/Cookie handling, and duplicate casing.
- Only adding headers to openclaw gateway call would miss the reported openclaw devices list workflow and other Gateway-backed CLI wrappers with bespoke option helpers.
- The closed PR #49646 is useful prior art but had review feedback about dead parameters and comma-delimited env parsing, so it should not be copied without tightening the design.
