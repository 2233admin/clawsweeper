## Summary

Keep open. Current main still applies recurring cron error backoff as a lower bound against the natural next schedule, without using transient-error classification to retry long-period recurring jobs sooner. The issue also has an open same-author implementation PR, #46423, with `Closes` syntax, so the right cleanup path is to review/refresh that PR rather than close this issue early.

## Best Possible Solution

Keep this issue open and use #46423 as the implementation candidate. The desired fix belongs in `src/cron/service/timer.ts`: classify recurring `cron`/`every` errors for built-in transient cases, schedule long-period transient failures on the short backoff path, preserve permanent-error natural schedule behavior, and add regression coverage for daily/hourly `cron` and `every` jobs plus restart/backoff interactions. Before landing, maintainers should settle the `cron.retry` contract for recurring jobs and remove unrelated PR drift.
