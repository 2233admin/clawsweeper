## Summary

Keep open. Current main still contains the reported memoryFlush counter synchronization path: a flush-triggered compaction can persist the post-increment compactionCount as memoryFlushCompactionCount, while the flush gate skips when those counters are equal. The latest release tag v2026.4.26 also contains the same path, and PR #51421 is still the canonical open fix candidate with closing syntax for this issue.

## Best Possible Solution

Keep this issue open. Review and either land or explicitly reject PR #51421, or apply an equivalent patch that stops a flush-triggered compaction from writing the post-increment compactionCount into memoryFlushCompactionCount. Update the synchronized-counter test and add a regression test proving consecutive eligible auto-compaction cycles run memoryFlush each time.
