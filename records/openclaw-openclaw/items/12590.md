## Summary

Keep open. Current main still persists the post-increment compaction count as the memory-flush marker while the flush gate skips equal counters, and open PR #51421 already targets this exact bug with closing syntax.

## What This Changes

The issue asks for `memoryFlush` to run on every eligible auto-compaction cycle instead of being skipped by synchronized `compactionCount` and `memoryFlushCompactionCount` values.

## Best Possible Solution

Land #51421 or an equivalent focused patch that preserves the pre-increment flush marker after flush-triggered compaction, updates the synchronized-counter test expectation, adds consecutive eligible auto-compaction coverage, and preserves the documented once-per-cycle behavior.

## Reproduction Assessment

Yes. The static current-main path is high-confidence: after a flush-triggered compaction, the runner persists the post-increment count, and the next eligible cycle sees equal counters and skips the flush.

## Solution Assessment

Yes. The narrow maintainable fix is to stop recording the post-increment `nextCount` as the flush marker, with tests for consecutive eligible auto-compaction cycles and a cross-check against #47143.
