## Summary

Keep open. Current main has partial model override support for memory-core dreaming, but the requested direct `dreaming.execution.model` shape and configured fallback-chain support are still missing from the strict manifest schema, resolver/runtime contract, docs, and tests.

## What This Changes

Request to add a memory-core dreaming `execution` configuration surface with model override and fallback-chain support for cheaper background Dream Diary runs.

## Best Possible Solution

Keep this open until maintainers choose the public dreaming model-selection API. If accepted, align the manifest schema, SDK resolver/types, Dream Diary attempt order, subagent trust/allowlist handling, docs, and focused tests; otherwise document `dreaming.model` plus session-default retry as the canonical supported behavior.

## Reproduction Assessment

Yes. Static inspection of the current strict manifest and resolver is enough to reproduce the remaining gap: `dreaming.execution.model` and configured fallback lists are not accepted or consumed, while only `dreaming.model`, `execution.defaults.model`, and per-phase `execution.model` are represented.

## Solution Assessment

No. The current partial implementation helps users pick one cheaper Dream Diary model, but it does not implement configured fallback chains or the direct `execution.model` shape requested here; the safest next step is a maintainer API decision before code changes.
