## Summary

Current main only partially satisfies this request. It supports `dreaming.model`, `dreaming.execution.defaults.model`, per-phase `execution.model`, and retrying once with the session default when the configured Dream Diary model is unavailable. It still does not support the requested direct `dreaming.execution.model` shape or user-configured fallback model chains.

## Best Possible Solution

Keep this issue open for a maintainer/API decision on the remaining fallback-chain behavior. A good resolution would either explicitly bless the current `dreaming.model` plus session-default retry as sufficient, or add a backward-compatible configured fallback surface such as `dreaming.execution.defaults.fallbacks` or a documented direct `dreaming.execution.model` alias, then update the memory-core manifest, resolver types, docs, generated config/help references, Dream Diary narrative wiring, and focused tests while preserving plugin subagent trust gates.
