## Summary

Keep open. Current main still starts legacy iMessage by spawning the configured `imsg rpc` child from the Gateway/Node runtime, and the user-facing docs still require Full Disk Access for the process context running OpenClaw/`imsg`; no stable signed helper, XPC bridge, daemon, or app-owned TCC identity exists for this path. Related PR #72898 is useful diagnostics work for permission-denied output, and related issue #51299 tracks a broader LaunchAgent/TCC variant, but neither implements the requested Node-upgrade-stable process identity.

## What This Changes

The issue requests that legacy `imsg` access `chat.db` under a stable `imsg`-owned process identity so Homebrew Node upgrades do not require re-granting Full Disk Access to the parent Node process.

## Best Possible Solution

The durable end state should either put legacy `imsg` database access behind a stable signed OpenClaw- or `imsg`-owned process identity with clear TCC guidance, or explicitly retire the legacy `imsg` path in favor of BlueBubbles with migration docs and actionable diagnostics for users who hit this macOS permission class.

## Reproduction Assessment

No. A high-confidence live reproduction requires macOS TCC state, Full Disk Access grants, a real `~/Library/Messages/chat.db`, and a Homebrew Node resolved-path upgrade; static inspection only verifies that current main still lacks the requested stable process boundary.

## Solution Assessment

Unclear. The requested direction is valid, but the best fix depends on maintainer decisions around macOS TCC, signing, helper or XPC architecture, and whether legacy `imsg` should continue to be supported beside the recommended BlueBubbles path.
