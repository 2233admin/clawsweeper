## Summary

Keep this issue open. Current main still runs legacy iMessage through gateway/plugin-owned `imsg rpc` child processes, and the docs still say Full Disk Access is required for the OpenClaw/`imsg` process context. I did not find a stable signed helper, launchd/XPC bridge, or imsg-owned daemon boundary that would make the parent Node process irrelevant for macOS TCC.

## Best Possible Solution

Keep this open for bundled iMessage plugin product/security handling. If legacy `imsg` remains supported, move Messages database access behind a stable macOS process identity such as a signed OpenClaw-owned helper, imsg-owned daemon, launchd/XPC bridge, or another explicit helper boundary. If maintainers prefer not to invest in legacy `imsg`, document and execute a retirement path toward BlueBubbles instead. The likely touched surfaces are `extensions/imessage/src/client.ts`, probe/send/monitor paths, setup copy, public iMessage docs, and macOS TCC reproduction coverage.
