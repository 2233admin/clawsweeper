## Summary

Keep open. Current main still lacks a group/channel-to-main session-scope option, and this PR remains incomplete: it wires the new option only through inbound route resolution, leaves outbound/proactive session derivation and Telegram topic overrides on per-channel keys, misses user-facing privacy/docs/schema metadata, and bundles the separate IRC proactive-send work from #45662.

## What This Changes

This PR adds a top-level route-binding `groupScope` option for matched group/channel peers to use an agent's main session and also adds an IRC active-client registry for proactive sends.

## Best Possible Solution

Refresh this as a first-class route-binding session-scope change, ideally after splitting or landing the IRC proactive-send fix separately. The accepted implementation should choose the canonical config shape, update strict types/Zod/generated schema/help/docs, carry the scope through inbound routing, outbound session derivation, Telegram topic overrides, and focused tests, and document the privacy consequence of merging a selected group/channel into main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Propagate groupScope into outbound session derivation:** `src/routing/resolve-route.ts:673`
  - body: The new option is only threaded through inbound `resolveAgentRoute()`. Outbound paths such as `gateway send` and tool-driven sends still use `resolveOutboundSessionRoute()` -> `buildOutboundBaseSessionKey()`, which only sees global `dmScope`, so a merged group/channel can fork back into a per-channel session on proactive sends.
  - confidence: 0.89
- **[P2] Preserve groupScope in Telegram topic overrides:** `src/routing/resolve-route.ts:673`
  - body: Telegram forum topics with `topicAgentId` recompute `route.sessionKey` using `buildAgentSessionKey()` and only `dmScope`. A topic that matches `groupScope: "main"` will be overwritten back to a per-topic group session, so that supported channel path will not honor the new option.
  - confidence: 0.84
- **[P3] Fall back when the active IRC send fails:** `extensions/irc/src/send.ts:72-74`
  - body: When an active monitor client is found, the new IRC path calls `active.sendPrivmsg()` and returns. If the monitor connection drops after `isReady()` but before the write, proactive sends fail instead of trying the transient fallback preserved for missing active clients.
  - confidence: 0.72
