## Summary

Keep this PR open. Current main still has no per-binding `groupScope` or equivalent group/channel-to-main-session routing path, while docs and tests continue to describe group/channel sessions as isolated. The PR remains potentially useful, but it needs maintainer review and likely a refresh/split before landing because outbound routing, Telegram topic overrides, docs, tests, and the bundled IRC active-client work are not fully resolved.

## Best Possible Solution

Keep this PR open for maintainer review or author refresh. The best path is to split/rebase out the IRC active-client work into #45662, then implement a first-class per-binding group/channel session scope across the config type, strict schema, generated config metadata/help, inbound route resolution, outbound session derivation, Telegram topic overrides, focused routing/outbound tests, and a docs/security note for the main-session privacy boundary.

## Risks / Open Questions

- Merging the groupScope part as-is could route inbound group/channel traffic to main while outbound/proactive sends still derive per-channel session keys when no explicit sessionKey is provided.
- Telegram topic agent overrides rebuild route session keys with `buildAgentSessionKey()` and only pass `dmScope`, so a group-scope feature would need explicit handling there.
- The feature intentionally collapses a group/channel privacy and non-main sandbox boundary into the main session for matched bindings; docs, config help, and tests should make that consequence explicit.
- The PR bundles unrelated IRC active-client work that is already tracked by open PR #45662, increasing review scope for an otherwise routing/config change.
- The IRC active-client diff noted in review has a robustness gap if the active client drops between readiness check and send; that should be addressed in the IRC-specific PR or split out before landing here.
