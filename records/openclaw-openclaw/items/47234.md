## Summary

Keep this PR open. Current main still does not implement per-binding `groupScope` or an equivalent group/channel-to-main-session routing path. Main has per-binding `session.dmScope`, but that only affects direct-message peers; group/channel peers still resolve to non-main session keys, and docs/tests still describe that behavior.

## Best Possible Solution

Keep this PR open for maintainer review or author refresh. The best path is to split or rebase away the IRC active-client work into #45662, then implement a first-class per-binding group/channel session scope across the config type, strict schema, generated config metadata/help, inbound route resolution, outbound session derivation, and channel-specific overrides such as Telegram topics. Add focused tests for `groupScope: "main"`, absent/default per-channel behavior, explicit per-channel behavior, DM unaffected behavior, outbound/proactive sends, and a docs/security note explaining the main-session privacy boundary.

## Risks / Open Questions

- Merging as-is would leave outbound/proactive sends and mirrored history able to derive per-channel session keys while inbound group traffic may use main, splitting one configured conversation across sessions.
- Telegram topic agent overrides rebuild route session keys and would need explicit group-scope handling before this feature works reliably across Telegram forum topics.
- The PR bundles unrelated IRC active-client work from open PR #45662, increasing review scope and making conservative landing harder.
- The feature intentionally collapses a group/channel privacy and sandbox boundary into the main session for matched bindings; docs, config help, and tests should state that consequence clearly.
