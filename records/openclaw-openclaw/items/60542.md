## Summary

Keep this issue open. Current main has several adjacent fixes for the same failure family, including terminal lifecycle patch persistence, hidden lifecycle event persistence, restart-interrupted running-session recovery, and role-ordering reset recovery, but I did not find the specific stale terminal main-session row versus newer transcript detection described here. The closed related PR #60642 proposed exactly that targeted check and was not merged, so the remaining report is still actionable rather than safely closable as implemented.

## What This Changes

The issue requests that persisted main-session rows stay consistent with their transcript so stale terminal registry entries cannot block new prompts across Telegram and Control UI.

## Best Possible Solution

Add a focused session-store recovery path that detects a main-session terminal registry row that is older than its transcript/lifecycle tail and safely rolls forward to an accept-new-input state, with a regression test that preserves existing terminal lifecycle persistence and restart recovery behavior.

Add a focused regression for a main direct-chat session where `sessions.json` has `agent:main:main` in a terminal state (`done`, `failed`, `killed`, or `timeout`) with an older `updatedAt`/`endedAt`, while the referenced transcript has newer lifecycle evidence such as a prompt timeout/error tail or a newer transcript mtime. The expected behavior is that the next accepted prompt does not keep reusing the stale terminal row in a way that wedges input; it should roll forward to a fresh usable session or otherwise reconcile lifecycle fields safely. Do not add broad startup migrations or plugin-specific ids. Include regression coverage in `src/auto-reply/reply/session.test.ts` or the smallest gateway/session test that proves the stale terminal main-session row is recovered. No changelog entry is required unless the final behavior is user-visible beyond the bug fix. Validate with targeted tests and formatter only.
