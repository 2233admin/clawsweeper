## Summary

Keep open. Current main still has a credible path where a compaction/retry lifecycle can surface a normal lifecycle `end`, `agent.wait` maps that to `status: ok`, and the subagent registry immediately freezes whatever child assistant text is visible at that moment. The newer compaction-retry waiting code helps the embedded runner wait internally, but I did not find source or test coverage that suppresses the gateway wait/completion event until the resumed/final turn has actually ended.

## What This Changes

The issue asks for subagent completion handling to stay pending across internal session compaction so the parent receives the real final handoff instead of a mid-task `frozenResultText` snapshot.

## Best Possible Solution

Make the lifecycle contract compaction-aware: when a subagent turn is compacting or retrying after compaction, keep `agent.wait` and subagent completion pending, then freeze and announce the child result exactly once after the resumed/final turn reaches a true terminal state. Add regression coverage that emits compaction start/end-with-retry plus an intermediate lifecycle end before the final assistant handoff.

## Reproduction Assessment

Yes. The issue provides detailed 3-of-3 workspace reproductions, and current source gives a high-confidence synthetic path: compaction marks the run paused, a lifecycle `end` can still be emitted, `agent.wait` treats that as `ok`, and the registry freezes current child history before the later final handoff exists.

## Solution Assessment

Yes, the requested direction is the right fix boundary. The narrowest maintainable solution is to suppress or transfer terminal wait/completion across compaction resume, rather than only lengthening the output-freeze polling window.
