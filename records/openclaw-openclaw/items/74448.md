## Summary

Keep open. The report is concrete, current, and source-backed: current main still treats a non-yielding `agent.wait` terminal result as subagent completion, then freezes the latest child assistant text immediately. I did not find code or tests that transfer the wait across compaction/resume or suppress the premature completion described here.

## What This Changes

The issue requests that subagent completion only fire after the child task actually ends, not when mid-task session compaction/resume causes an intermediate wait result.

## Best Possible Solution

Add a compaction-aware lifecycle contract so subagent completion remains pending across internal compaction/resume and freezes the final child handoff exactly once after the resumed turn actually reaches a terminal state.

## Repair Work Prompt

Fix the subagent completion/compaction bug reported in openclaw/openclaw#74448. Observable bug: a child subagent that compacts mid-task can make the parent receive a completed-successfully internal completion event with a short mid-task `frozenResultText`, while the resumed child continues and later writes the real handoff. Keep the fix scoped to subagent wait/completion semantics and compaction/resume handling. Preferred direction: make compaction an internal lifecycle state for `agent.wait`/subagent completion, either by keeping the wait attached to the resumed run id or by suppressing/deferring the first terminal-looking completion until the resumed child turn is actually terminal. Do not broaden channel delivery behavior, plugin APIs, unrelated retry policy, or provider-specific compaction code unless a generic seam is required. Add focused regression coverage proving that a compaction/resume sequence does not freeze a pre-compaction assistant fragment and that the final resumed assistant handoff is delivered once. Preserve existing `sessions_yield`, timeout, restart/orphan recovery, and nested subagent behavior. If the fix changes user-visible subagent behavior, add one single-line changelog Fixes entry in the active version with `Thanks @mlopez14`.
