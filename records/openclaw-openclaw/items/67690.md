## Summary

Keep open. Current main has cache-TTL pruning, documented pruning knobs, context usage reporting, and a Control UI context-pressure notice, but it still lacks the central request: pruning preview/history, pruning metadata or lifecycle events, pruning notifications, and keyword/tag-style retention controls.

## What This Changes

Request better visibility and control for context pruning, including pruning previews/logs, clearer UI usage and notifications, configurable retention strategy, and model/version-specific documentation.

## Best Possible Solution

Define a maintainer-approved pruning observability/control design: have the pruning path produce safe metadata, expose an additive pruning lifecycle event coordinated with #8606, surface a concise Control UI/chat notice when pruning actually happens, document existing and new knobs on the public pruning docs, and keep overflow-message specifics aligned with #9409.
