## Summary

Keep open. Current main still lacks the requested standalone Telegram processing-indicator message config and lifecycle; the existing implementation provides typing cues, ack reactions, status reactions, streaming edits, and fallback error replies, but not a tracked placeholder message that is deleted or edited on completion. The related open waiting-tips PR and closed direct attempts show this is still an unresolved Telegram UX/config decision rather than a cleanup-close item.

## What This Changes

The issue requests an opt-in Telegram processing indicator that posts a visible placeholder message while work is running, deletes it on successful reply, and edits it on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after eligible inbound acceptance, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high confidence: strict schema inspection shows the proposed config is not accepted, and Telegram inbound/dispatch inspection shows no stored placeholder message lifecycle.

## Solution Assessment

Unclear whether the narrow placeholder design or the broader waiting-tips design is the best product solution. The maintainable path is for maintainers to choose the UX/config direction first, then implement it in the Telegram plugin and shared config surfaces only as needed.
