## Summary

Current main still does not implement the requested Telegram standalone processing-indicator message lifecycle. The code has adjacent feedback mechanisms: early typing cues, ack/status reactions, native-command progress placeholders, and send/edit/delete Telegram primitives, but there is no accepted processingIndicator config, no placeholder message ID carried through normal inbound dispatch, and no delete/edit cleanup tied to final replies. Related implementation attempts #43021 and #43720 are closed unmerged, while #40421 is open and adjacent rather than a current-main fix.

## Best Possible Solution

Keep this issue open as the canonical bundled Telegram feature request. The best implementation path is an opt-in Telegram plugin change that adds typed config/schema/help/docs, sends a standalone placeholder after an eligible inbound message is accepted, carries its message ID through the normal dispatch context, deletes it after successful final delivery, and edits or replaces it on dispatch failure. Maintainers should decide how this relates to #40421 before queueing another automated implementation PR.
