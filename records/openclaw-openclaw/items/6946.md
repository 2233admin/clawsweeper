## Summary

Keep open. Current main still lacks the requested Telegram processing-placeholder lifecycle and does not accept either proposed config surface; existing typing cues, ack/status reactions, draft streaming, native-command progress placeholders, and generic error replies do not cover a standalone inbound-turn placeholder that is deleted or edited on completion. The issue also has an open overlapping waiting-tips PR and prior direct implementation attempts that closed unmerged, so this remains a maintainer product/config decision rather than a cleanup-close item.

## What This Changes

The issue requests an opt-in Telegram processing indicator that sends a visible placeholder message while an inbound request is running, deletes it after a successful reply, and edits or replaces it on failure.

## Best Possible Solution

Choose the intended bundled Telegram feedback UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after an eligible inbound message is accepted, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.
