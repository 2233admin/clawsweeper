## Summary

Keep open. Current main still lacks the requested standalone Telegram processing-placeholder message lifecycle; existing typing, ack-reaction, status-reaction, and fallback error paths do not send a visible message that is later deleted or edited. The overlapping waiting-tips PR is open and unmerged, while the direct processing-indicator attempts were closed unmerged, so this needs maintainer product/config direction rather than cleanup closure.

## What This Changes

Add an opt-in Telegram processing indicator that sends a visible placeholder while a request runs, deletes it after a successful reply, and edits or replaces it on failure.

## Best Possible Solution

Choose the intended bundled Telegram UX, then land an opt-in Telegram plugin change with typed config/schema/help/docs, dispatch wiring that sends and tracks a standalone placeholder after an eligible inbound message is accepted, deletion after successful final delivery, and edit-or-replace behavior on failure with focused regression coverage.
