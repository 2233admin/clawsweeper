## Summary

Keep this issue open. Current main still does not inject or expose the requested sibling-agent roster to the model: the system prompt only receives runtime metadata and tool names, `agents_list` remains scoped to `sessions_spawn` allowlists, and agent-to-agent reachability is enforced separately rather than returned as requester-specific roster data.

## Best Possible Solution

Keep this open and implement a focused agent-facing roster surface. The conservative path is to extend `agents_list` or add a dedicated discoverable tool returning peer `id`, `identity.name`, `identity.theme`, and explicit `agentToAgentAllowed` computed for the requester, filtered by current session/tool/sandbox policy. Update prompt guidance to direct agents to that live roster, then add payload-shape, policy-filtering, and docs coverage.

## Risks / Open Questions

- A roster surface must avoid leaking agent metadata beyond what the requester can actually use; `subagents.allowAgents`, `tools.sessions.visibility`, sandbox clamping, and `tools.agentToAgent` are separate gates today.
- Injecting a static full roster into every system prompt would add prompt-cache churn and can become stale during long sessions; a live discoverable tool plus prompt guidance may be safer.
- Extending `agents_list` should preserve the existing `sessions_spawn` discovery contract or add fields in a backwards-compatible way.
