## Summary

Keep open. Current main still resolves, validates, and passes plugin config only from `plugins.entries.<id>.config`; the issue discussion adds evidence that this is broader than one observability plugin and needs a core plugin-config contract decision rather than ClawHub-only work.

## What This Changes

Request an env-var override and/or writable-overlay path for `plugins.entries.<id>.config` so policy-locked sandboxes can change plugin config without rebuilding images.

## Best Possible Solution

Add an additive, opt-in plugin-config resolution seam that merges baked `plugins.entries.<id>.config` with a maintainer-approved writable overlay and/or declared env mappings, validates the final merged value against the plugin manifest schema, exposes only that final value through `api.pluginConfig`, and documents precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: source inspection on current main shows plugin config is resolved, validated, and exposed only from `plugins.entries.<id>.config`. The external NemoClaw write failure is supplied by the report and comments rather than live-reproduced here.

## Solution Assessment

No current solution is in place. The issue's preferred direction is reasonable, but the best maintainable fix needs a core contract for overlay path, merge depth, env mapping shape, precedence, and validation timing rather than asking plugins to independently read arbitrary `process.env`.
