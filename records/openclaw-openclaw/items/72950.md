## Summary

Keep open. Current main still validates and passes only `plugins.entries.<id>.config` to plugins, with no host-side writable overlay or explicit plugin-declared env mapping before manifest validation. The existing global `${VAR}` substitution can help pre-baked configs, but it does not provide the runtime overlay path the issue and follow-up comments are asking for in policy-locked plugin deployments.

## What This Changes

Requests a core plugin-config resolution path so `plugins.entries.<id>.config` can be overridden from declared environment variables and/or a writable overlay in policy-locked sandboxes without rebuilding images.

## Best Possible Solution

Add an additive plugin-config resolution layer in core: start from baked `plugins.entries.<id>.config`, merge an explicitly approved writable overlay and/or plugin-declared env mappings, validate the final object against the plugin manifest schema, expose only the validated result through `api.pluginConfig`, and document precedence plus policy-locked sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side: current-main source inspection shows `entry.config` is the only value resolved, validated, and passed into plugin APIs. The NemoClaw EACCES path is supplied by the issue discussion and would need that sandbox to live-run, but it is not required to confirm the missing OpenClaw seam.

## Solution Assessment

Yes, the requested host-side overlay/env-resolution direction is the maintainable shape, provided it is explicit and validated before plugin activation. Letting plugins independently read arbitrary `process.env` would bypass the unified config and schema contract this issue is trying to preserve.
