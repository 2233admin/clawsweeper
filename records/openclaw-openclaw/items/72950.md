## Summary

Keep open. Current main has general environment loading and some plugin-specific env fallbacks, including diagnostics-otel, but the generic plugin config path still resolves `api.pluginConfig` directly from `plugins.entries.<id>.config`. I did not find a generic env-var mapping (`configEnv`/`fromEnv`) or writable overlay source for arbitrary plugin config, and the issue discussion includes another affected external plugin (`openclaw-weixin`), so this is still a real core/plugin-contract design gap rather than cleanup-close material.

## Best Possible Solution

Keep this issue open and handle it as a plugin config contract/design task. The likely fix is an additive, opt-in core surface such as manifest-declared or config-declared env mappings for selected plugin config keys, or a documented writable overlay path with clear precedence, schema validation, SecretRef behavior, reload/restart semantics, and tests around `api.pluginConfig` resolution. If maintainers choose not to add a core surface, the fallback should be explicit docs for policy-locked environments plus a plugin-author pattern for env-only runtime configuration.

## Risks / Open Questions

- Docs/generated metadata mention `plugins.entries.<id>.env`, but the strict source schema/type inspected here do not accept it. Maintainers should decide whether that is stale documentation/metadata or an incomplete feature.
- A generic writable overlay or env override needs explicit opt-in and precedence rules so policy-locked deployments can reconfigure safely without letting writable data silently override protected static config.
