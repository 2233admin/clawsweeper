## Summary

Current main still lacks a generic plugin-config env override or writable overlay. Plugin config is resolved from `plugins.entries.<id>.config`, validated from that value, and passed to `api.pluginConfig`; existing env substitution only works for pre-authored `${VAR}` strings and does not override a baked literal value. Related context also points here as the canonical thread after #72944 was closed in favor of this issue, with another plugin maintainer reporting the same pain for openclaw-weixin.

## Best Possible Solution

Keep this issue open as a plugin config contract/design task. The strongest path is an additive opt-in mechanism, such as manifest- or config-declared env mappings for selected `plugins.entries.<id>.config` keys, or a documented writable plugin-config overlay with clear precedence, schema validation, SecretRef handling, restart or reload semantics, and focused tests around `api.pluginConfig` resolution. If maintainers reject a core surface, document the policy-locked sandbox constraint and the supported plugin-author pattern explicitly.

## Risks / Open Questions

- Closing now would leave policy-locked sandbox operators without a documented way to override a baked literal plugin config value without rebuilding the image.
- `${VAR}` substitution and plugin-specific env fallbacks are useful adjacent mechanisms, but treating them as a full fix would miss precedence, schema validation, SecretRef handling, and writable-overlay semantics.
- A future fix needs careful precedence and security design so environment or overlay values do not silently override sensitive config for plugins that did not opt in.
