## Summary

Keep open. Current main still has no generic env-var override or writable overlay path for plugin config: the schema only accepts `enabled`, `hooks`, `subagent`, and `config`; validation reads `entry.config`; and plugin APIs receive that validated value directly. The discussion also shows this is broader than one observability plugin, including the openclaw-weixin comment and #72944 having already been closed in favor of this canonical issue.

## What This Changes

The issue requests a runtime override path for `plugins.entries.<id>.config` so policy-locked sandboxes can adjust plugin config without rebuilding images.

## Best Possible Solution

Add an additive plugin-config resolution seam that merges baked `plugins.entries.<id>.config` with a clearly scoped writable overlay and/or plugin-declared env mappings, validates the final merged value against the plugin manifest schema, exposes only that final value through `api.pluginConfig`, and documents precedence plus sandbox behavior.

## Reproduction Assessment

Yes for the OpenClaw side. A focused current-main check is to configure a literal `plugins.entries.demo.config` value and a matching env var: the schema, resolver, loader, and API builder still use `entry.config` only, while the issue body supplies the external NemoClaw read-only file symptom.

## Solution Assessment

Unclear because there is no patch yet. The suggested direction is probably the right maintainable shape, but maintainers should first choose the overlay/env contract and merge semantics rather than letting plugins independently read arbitrary env vars outside the unified config path.
