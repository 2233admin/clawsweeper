## Summary

Keep open. Current main still resolves plugin runtime config from `plugins.entries.<id>.config` only, validates and passes that value to `api.pluginConfig`, and has no generic env-var override mapping or writable plugin-config overlay. Existing `${VAR}` substitution only fills authored string placeholders during config load; it does not override an already baked literal config in a read-only sandbox. The related #72944 was closed in favor of this issue, and the openclaw-weixin comment supports treating this as a broader plugin-config contract gap.

## Best Possible Solution

Keep this open as a core plugin configuration contract/design task. The likely direction is an additive opt-in mechanism: merge baked `plugins.entries.<id>.config` with a documented writable plugin-config overlay and/or explicitly declared env mappings, then validate the final merged object against the plugin schema before exposing it as `api.pluginConfig`. The design should define precedence, path location, SecretRef support, reload/restart behavior, diagnostics for read-only config files, and public docs. If maintainers reject an overlay or env override surface, document the build-time-only constraint for policy-locked sandboxes and the supported plugin-author pattern.
