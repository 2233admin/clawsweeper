## Summary

Keep open. Current main still resolves plugin config from `plugins.entries.<id>.config` and passes that validated object to plugins; the adjacent `${VAR}` substitution and `$include` features do not provide the requested plugin-declared env override or writable sibling overlay. The thread also shows broader impact beyond the original observability/NemoClaw case, including the openclaw-weixin comment, and #72944 is already closed in favor of this issue.

## What This Changes

Allow plugin configuration under `plugins.entries.<id>.config` to be reconfigured in policy-locked sandboxes through declared env overrides and/or a writable overlay instead of only a baked `openclaw.json`.

## Best Possible Solution

Add an additive plugin-config resolution seam that merges baked config with an explicitly supported writable overlay and/or plugin-declared env mappings, validates the final merged value against the plugin manifest schema, exposes only that final value through `api.pluginConfig`, and documents precedence plus sandbox operational behavior.

## Reproduction Assessment

Yes for the OpenClaw side: current main can be checked by setting a literal `plugins.entries.demo.config` value and a matching env var; the resolver, schema, and loader still use `entry.config` only. The external NemoClaw read-only filesystem symptom is supplied in the issue body but was not live-tested here.

## Solution Assessment

Unclear until maintainers choose the contract shape. The narrow maintainable direction is a central, additive, opt-in config resolution contract; telling plugins to read arbitrary `process.env` independently would weaken the unified config contract the report is trying to preserve.
