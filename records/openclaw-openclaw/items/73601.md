## Summary

Keep this issue open. Current main still matches the reported Google Live browser Talk failure mode: the Google Live WebSocket transport parses every inbound frame with `JSON.parse(String(data))`, silently drops parse failures, and only marks Talk live when a JSON `setupComplete` message is seen. That leaves the reporter's observed binary audio frames with no playback path and can leave the composer stuck at `Connecting Talk...`. Related #73460/#73466 cover a narrower queued-audio interruption bug in the same file, and #73427 covers a similar error-surfacing anti-pattern for OpenAI, but neither supersedes this Google Live connection/playback/wedge report.

## Best Possible Solution

Keep this issue open and fix it in the Google Live browser Talk path. The likely implementation should teach `ui/src/ui/chat/realtime-talk-google-live.ts` to handle the actual browser WebSocket data shapes (`string`, `Blob`, `ArrayBuffer`) without dropping binary audio, transition to `listening` on a reliable readiness/audio signal, surface provider parse/protocol errors, and add focused UI transport tests for binary audio, status transition, playback, stop cleanup, and the post-failure `chat.send` wedge reproduction. Coordinate with #73460/#73466 for the scheduled-source cleanup, but do not treat that narrower interruption PR as covering this report.
