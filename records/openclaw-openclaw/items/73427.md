## Summary

Keep open. Current main still matches the reported OpenAI browser Realtime Talk failure: the Control UI CSP omits the OpenAI Realtime origin while the browser posts SDP there, the OpenAI browser client-secret payload only configures output audio, and the WebRTC data-channel handler still drops OpenAI error and lifecycle events. The closed Google Live report is adjacent transport work, not a duplicate of this OpenAI WebRTC path.

## What This Changes

Request to fix Control UI OpenAI Realtime Talk so CSP permits the WebRTC call, browser sessions include needed input audio/VAD settings, and the UI surfaces OpenAI realtime error and lifecycle events.

## Best Possible Solution

Land a focused OpenAI browser Talk fix in the existing bundled path: add a tight Control UI CSP allowance for the OpenAI Realtime WebRTC origin, mint browser sessions with the required input audio/VAD/transcription settings, map OpenAI error and basic speech/response lifecycle events into Talk status, and cover the behavior with targeted tests plus a changelog entry.

## Reproduction Assessment

Yes. The issue gives concrete 2026.4.26 steps, and current main statically reproduces the two core blockers by combining a direct browser SDP POST to OpenAI with a CSP that omits that origin and a data-channel handler whose default branch drops OpenAI error/lifecycle events.

## Solution Assessment

Yes. The narrow maintainable fix is in the existing OpenAI browser Talk path across CSP, provider session creation, and WebRTC event handling; a broad configurable CSP surface would be a separate product/security decision.
