## Summary

Keep this issue open. Current main still matches the reported failure modes: Control UI CSP does not allow the OpenAI Realtime HTTPS endpoint, the OpenAI WebRTC browser client posts SDP directly to that endpoint, the data-channel handler drops server error/lifecycle events, and the browser session minting payload only configures output voice rather than explicit input audio/VAD/transcription settings.

## Best Possible Solution

Keep this issue open and fix it in the bundled Control UI/OpenAI realtime path: either allow the exact OpenAI Realtime HTTPS origin in Control UI CSP or route SDP through the Gateway, add explicit OpenAI browser-session input audio/VAD/transcription config where appropriate, surface data-channel `error` and basic speech/response lifecycle events in the UI, and add focused CSP/provider/UI regression coverage plus the existing realtime Talk live smoke when maintainers can run it.
