## Summary

Keep open. Current main still matches the reported OpenAI browser Talk failure modes: the Control UI CSP omits the OpenAI HTTPS origin, the WebRTC transport still posts SDP directly to OpenAI from the browser, OpenAI data-channel diagnostic events still fall through, and the browser-session payload still lacks the requested input audio/VAD configuration. The closed Google Live report is related only as a similar UI-status pattern; it does not supersede this OpenAI WebRTC path.

## What This Changes

The issue asks to fix Control UI OpenAI Realtime Talk by allowing the required WebRTC SDP connection, configuring browser speech audio/VAD behavior, and surfacing realtime errors and lifecycle status in the UI.

## Best Possible Solution

Ship a focused OpenAI browser Talk fix: either narrowly allow the required OpenAI realtime HTTPS origin or proxy the SDP exchange through the Gateway, keep standard provider API keys server-side, add explicit browser-session audio/VAD configuration where needed, surface OpenAI error and speech/response lifecycle events through the existing Talk status UI, and cover the behavior with targeted CSP, WebRTC, UI, and OpenAI provider tests.

## Reproduction Assessment

Yes. The issue provides concrete repro steps, and current main gives a high-confidence static reproduction: the browser fetch target is outside the CSP connect-src directive, and diagnostic data-channel events are ignored after connection setup.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix if scoped to the OpenAI browser Talk path; a broader configurable CSP feature should remain separate, and a Gateway SDP proxy is the safer alternative if maintainers do not want direct browser connect-src allowance.
