## Summary

Keep open. Current main still lacks the reported provider/model-specific tagged tool-call normalization: Qwen/ModelStudio only patches request thinking fields, OpenAI-compatible streaming only promotes structured `delta.tool_calls`, and the existing Kimi marker parser does not cover marker sections carried in thinking/reasoning blocks.

## What This Changes

The issue requests a fix for subagents announcing completion when affected Qwen/ModelStudio/Kimi/MiniMax routes emit tagged tool-call markup inside thinking or reasoning text instead of structured tool-call blocks.

## Best Possible Solution

Add provider-owned Qwen/ModelStudio response normalization, or a narrowly gated shared OpenAI-compatible helper reused by Qwen and Kimi, that extracts only validated standalone tagged tool-call sections from response text/thinking/reasoning content, emits structured `toolCall` blocks, and changes `stopReason` to `toolUse` only after successful extraction.

## Repair Work Prompt

Fix the provider/model-specific tagged tool-call handling behind this issue: Qwen/ModelStudio-compatible routes can emit `<|tool_calls_section_begin|>...<|tool_call_begin|>...` sections inside response text, reasoning, or thinking content instead of structured `delta.tool_calls`, which leaves `stopReason` as `stop` and causes subagent completion handling to treat the run as done without executing the intended tools.

Expected boundary:
- Prefer a Qwen/ModelStudio-owned stream normalization in `extensions/qwen/stream.ts` and tests in `extensions/qwen/stream.test.ts`, or a narrowly gated shared OpenAI-compatible helper if both Qwen and Kimi should reuse it.
- Reuse or generalize the existing Kimi tagged parser pattern from `extensions/kimi-coding/stream.ts` only with conservative validation.
- Extract only standalone, complete tagged tool-call sections with valid object JSON arguments; do not parse arbitrary prose, partial markers, markdown examples, or mixed visible text as executable tools.
- Convert validated markers to structured `toolCall` blocks and change `stopReason` to `toolUse` only after at least one valid structured tool call is emitted.
- Cover reasoning/thinking carrier fields used by the OpenAI-compatible transport (`reasoning_content`, `reasoning`, `reasoning_text`, and `reasoning_details` where applicable) without breaking ordinary reasoning display.
- Do not edit `dist/**`, do not add bundled provider IDs to core when a plugin wrapper or generic capability seam is enough, and do not broaden tool execution semantics outside the affected provider family.
- Add a user-facing changelog fix entry with reporter credit if maintainers consider this shipped behavior user-facing.

Related refs: https://github.com/openclaw/openclaw/issues/53810, https://github.com/openclaw/openclaw/pull/61801, https://github.com/openclaw/openclaw/pull/61481
