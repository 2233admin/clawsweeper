## Summary

Keep open. Current main has compaction identifier-preservation prompts and quality checks, but the reporter supplied credible `raw_params` logs showing corrupted file paths after long-session compaction, and the current safeguard only audits a bounded recent identifier slice rather than deterministically carrying every important path through compaction.

## What This Changes

The issue reports that after context compaction, full file paths can be replayed into tool calls with inserted spaces, causing `ENOENT` reads on corrupted paths.

## Best Possible Solution

Harden compaction so path-like literals needed for future tool calls are preserved outside normal prose summarization: extract relevant file paths from the full summarization input and file-operation metadata, carry them into the exact-identifiers section or suffix in a deterministic form, and add regression coverage proving slash-separated absolute paths survive safeguard compaction unchanged.

## Reproduction Assessment

No, not with high confidence from the current checkout alone. The issue includes credible `raw_params` logs, but it does not include the transcript/checkpoint or a deterministic MiniMax long-context scenario that can be replayed locally.

## Solution Assessment

Unclear for a complete fix, because the current strict identifier prompts are a mitigation but not a guarantee. The narrow maintainable direction is to make file-path preservation deterministic in the compaction safeguard instead of relying only on the summarizing model to restate paths exactly.
