## Summary

Keep open. Current main still does not implement this PR's intended Nextcloud Talk rich-object attachment parsing: webhook content is passed through as raw text, inbound messages have no media URL field, and the inbound path still drops empty-text messages before media handling. The upstream Nextcloud Talk bot docs confirm `object.content` is a JSON-encoded dictionary with `message` and `parameters`, so the user problem is real, but the PR needs security and MIME handling revisions before merge. ([nextcloud-talk.readthedocs.io](https://nextcloud-talk.readthedocs.io/en/latest/bots/))

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but require a revision before merge: add a Nextcloud Talk rich-object parser in the plugin, resolve display text, forward `MediaUrls` plus MIME hints, allow media-only inbound messages, constrain extracted URLs to trusted Nextcloud attachment origins with SSRF-safe behavior, and add regression tests for file shares, malformed JSON fallback, extensionless image links, private/loopback rejection, no-placeholder rich objects, and mention placeholders covered by #66700.

## Risks / Open Questions

- Merging the current PR as-is could turn user-controlled rich-object JSON into remote media fetches; the fix should validate URL origin/host against the trusted Nextcloud backend or configured base URL and preserve existing SSRF guard behavior.
- The PR's described URL-only extraction may still leave extensionless Nextcloud share links classified as unknown unless MIME types are carried through as `MediaTypes`.
- Open #66700 overlaps rich-object placeholder handling for Nextcloud Talk mentions; parser work should be coordinated so attachment text and mention gating do not diverge.
