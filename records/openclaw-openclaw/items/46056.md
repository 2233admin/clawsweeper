## Summary

Keep open. Current `main` still has the reported allowlist gap for pathless POSIX shell builtins such as `cd`, and the exec approvals docs still do not explain that ordinary path or bare-name allowlist entries cannot satisfy those builtin segments.

## What This Changes

The issue requests that shell builtins such as `cd` stop unexpectedly triggering exec approval allowlist misses, or at minimum that the behavior be documented.

## Best Possible Solution

Add a narrow documentation clarification to the exec approvals allowlist section explaining that pathless POSIX shell builtins cannot satisfy ordinary path or bare-name allowlist entries, with a safer workaround such as using tool-native directory flags. Treat any runtime builtin auto-allow behavior as a separate opt-in policy design with focused tests for cwd and environment mutation semantics.

## Reproduction Assessment

Yes. The high-confidence reproduction is source-level on current `main`: `cd <dir> && git status` is split into shell-chain parts, the `cd` segment has no resolved executable path, `matchAllowlist` returns `null`, and the chain remains an allowlist miss.

## Solution Assessment

Yes for a docs-only first fix. It is the narrowest maintainable response to the documented current behavior; auto-allowing builtins by default is not the best runtime fix without a separate security design.
