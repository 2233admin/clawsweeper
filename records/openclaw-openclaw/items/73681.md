## Summary

Keep open. Current main still has a Google Chat inbound reaction gap: the webhook path can parse non-MESSAGE event types, but the Google Chat monitor returns before any route, system event, plugin hook, or agent pipeline sees them; Google Chat config/docs only cover outbound reaction actions. The linked #9764 and #17840 are related but do not supersede this ingress issue: #9764 is about outbound/user-OAuth capabilities, and #17840 is broader reaction-triggered turn policy.

## What This Changes

The issue requests inbound Google Chat reaction webhook events to be routed instead of dropped, with a normalized reaction event/hook and Google Chat docs.

## Best Possible Solution

Add Google Chat-owned inbound reaction support for the official reaction event payloads, normalize them through the existing channel/plugin-visible reaction notification pattern with clear gating, and document the required Google Chat subscription/auth setup. Keep outbound reaction OAuth work in #9764 and generic reaction-triggered turns in #17840 separate.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: a non-MESSAGE Google Chat event reaches processGoogleChatEvent and returns at extensions/googlechat/src/monitor.ts:59 before routing; official SpaceEvent-style reaction payloads also do not fit the current top-level space/message parser.

## Solution Assessment

Unclear. Routing inbound reactions is the right direction, but the requested Google Chat-specific onReaction hook is not proven to be the narrowest maintainable API; the safer starting point is the existing channel reaction notification/system event pattern unless maintainers choose a new hook contract.
