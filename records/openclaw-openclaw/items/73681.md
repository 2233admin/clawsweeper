## Summary

Keep open. Current main still has a source-level drop for authenticated non-MESSAGE Google Chat webhook payloads, but the provider contract check shows Google Chat reaction notifications are documented through Google Workspace Events/Pub/Sub event types rather than the reported Chat interaction webhook strings, so this needs maintainer direction before a safe implementation path is chosen.

## What This Changes

The issue asks OpenClaw to route inbound Google Chat reaction events to a normalized/plugin-visible path instead of accepting and silently dropping them.

## Best Possible Solution

Add Google Chat-owned inbound reaction support on the Google-supported ingress surface: verify and parse the actual reaction payload, apply the existing sender/space access policy, resolve the affected session, emit a durable system or plugin-visible reaction event consistent with other channels, and document inbound semantics separately from outbound OAuth work and opt-in reaction-triggered turns.

## Reproduction Assessment

Yes for the source-level drop: an authenticated non-MESSAGE payload with a valid space can be accepted by the webhook parser and then ignored by processGoogleChatEvent. Provider-real reproduction is unclear because Google’s documented reaction delivery uses Workspace Events/Pub/Sub event names rather than the reported REACTION_ADDED/REACTION_REMOVED webhook strings.

## Solution Assessment

Unclear. The requested Google Chat reaction handling is a valid gap, but the best fix should first align with Google’s actual Workspace Events/Pub/Sub contract instead of only branching on REACTION_ADDED and REACTION_REMOVED in the current interaction-webhook handler.
