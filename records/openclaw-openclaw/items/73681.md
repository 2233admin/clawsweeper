## Summary

Keep open. Current main still does not implement inbound Google Chat reaction handling: authenticated non-message webhook payloads can reach the Google Chat event processor, but the processor returns immediately unless the event type is `MESSAGE`, and there is no Google Chat reaction normalizer, system-event emission, or plugin-facing reaction hook documented or typed today.

## What This Changes

The issue requests inbound Google Chat reaction webhook support for add/remove reaction events, including normalized routing and plugin/docs surfaces.

## Best Possible Solution

Keep the issue open and design a Google Chat-owned inbound reaction path that accepts the correct Google event source/payload, applies the same access and routing boundaries as messages, emits a normalized system event or documented hook, and updates Google Chat docs. The broader opt-in behavior for reactions starting agent turns should remain coordinated with #17840 rather than hidden in this channel fix.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: any non-`MESSAGE` Google Chat event that reaches `processGoogleChatEvent` returns at the event-type guard, so a reaction event cannot reach downstream routing or plugin-visible handling on current main.

## Solution Assessment

Unclear. The suggested direction of adding reaction handling is right, but the exact event names and transport need maintainer/API design because official Google reaction events are Workspace Events API subscription events, not ordinary Chat interaction `EventType` values.
