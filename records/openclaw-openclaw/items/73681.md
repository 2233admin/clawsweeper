## Summary

Keep open. Current `main` still drops Google Chat non-`MESSAGE` events in the channel processor, and the existing related items do not supersede this: #9764 is about outbound/user-OAuth capabilities, while #17840 is about whether reaction events should start agent turns after they are already ingested. Official Google docs also suggest the implementation may need Workspace Events/Pub/Sub reaction event handling, not only a `REACTION_ADDED`/`REACTION_REMOVED` branch in the current Chat interaction webhook. citeturn1view3turn0search0

## What This Changes

Add inbound Google Chat reaction handling so reaction add/remove events are normalized and exposed to the runtime or plugins instead of being ignored.

## Best Possible Solution

Add Google Chat inbound reaction support in the Google Chat plugin boundary after confirming the Google-supported delivery path. The durable shape should parse the official reaction event payloads, apply the same account and space/user access policy as message ingress, enqueue normalized passive system events like Slack and Discord do, and document any Workspace Events/Pub/Sub setup; opt-in reaction-triggered agent turns can remain with #17840.

## Reproduction Assessment

Yes for the current code path: a non-`MESSAGE` Google Chat event that reaches `processGoogleChatEvent` returns at the event-type guard before any runtime-visible handling. The live provider path still needs verification because Google reaction events appear to arrive through Workspace Events subscription infrastructure rather than the current interaction webhook.

## Solution Assessment

Unclear. The requested direction is valid, but a direct `REACTION_ADDED`/`REACTION_REMOVED` branch plus a Google-specific `onReaction` hook may not be the best boundary; the safer repo-aligned design is official Workspace Events payload support plus normalized system events, with turn-trigger policy handled generically.
