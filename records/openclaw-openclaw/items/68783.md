## Summary

Keep this PR open, but it should not merge as-is. The linked cron delivery regression remains plausible on current main because timer/startup executions still have no delivery snapshot or narrower immutable-field reconciliation, but the PR restores the entire pre-run delivery object and would overwrite legitimate delivery writebacks that current main intentionally preserves after force-reloading the cron store.

## What This Changes

The PR snapshots `CronJob.delivery` before manual, timer, and startup catch-up cron executions, restores that snapshot after execution/reload, adds delivery-channel preservation tests, and changes an existing Telegram writeback regression to expect the original target.

## Best Possible Solution

Revise the fix so execution cannot mutate user-selected delivery channel/mode back to `last` while preserving legitimate delivery writebacks from the isolated run. A narrower field-wise reconcile or execution-only clone should be covered for manual, timer-driven, and startup catch-up paths, with tests for both channel preservation and Telegram target normalization/writeback.
