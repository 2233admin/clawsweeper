## Summary

Keep open. Current `main` still passes raw `logging.file` values through the logger filesystem boundary, and the latest shipped `v2026.4.29` tag shows the same behavior; the only direct fix PR, #73614, is closed unmerged.

## What This Changes

Restore home-prefix expansion for `logging.file` so gateway startup does not crash-loop when config uses a `~/...` log path.

## Best Possible Solution

Land a focused logger-boundary fix that expands leading `~` with the existing home helper before rolling-path detection and directory creation, with regression coverage for normal and rolling log filenames; keep the broader path-like config audit as separate scoped work.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: a `logging.file` value like `~/.openclaw/logs/gateway.log` flows from `readLoggingConfig()` into `resolveSettings()`, then through `resolveActiveLogFile()` unchanged before `fs.mkdirSync(path.dirname(activeFile))`. I did not run a live gateway because this review is read-only and live logging would write files.

## Solution Assessment

Yes. Expanding leading `~` at the logger filesystem boundary is the narrowest maintainable fix for the startup crash; rewriting config loading or auditing every path-like setting is useful follow-up work but not required for this failure.
