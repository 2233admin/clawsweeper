## Summary

Keep open. Current main still lets a raw `~/`-prefixed `logging.file` reach logger directory creation, and the only directly linked fix PR was closed unmerged, so the reported gateway startup crash remains actionable.

## What This Changes

Restore support for `~/`-prefixed `logging.file` values so gateway startup does not try to create literal `~/.openclaw/...` directories.

## Best Possible Solution

Land a narrow logger-boundary fix that expands a leading `~` before rolling-path detection and `fs.mkdirSync`, with regression coverage for both stable and rolling `logging.file` values read from active config. Treat broader config-derived path coverage as a follow-up unless the fix can reuse existing helpers without changing unrelated startup behavior.

## Reproduction Assessment

Yes. The source path is high-confidence: a raw `logging.file` string from config flows into `resolveActiveLogFile`, non-rolling values are returned unchanged, and `buildLogger` then calls `fs.mkdirSync` on `path.dirname` of the literal tilde path.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing home-expansion helper at the logger filesystem boundary and add focused regression tests; a broad startup-time config rewrite would carry more risk than fixing the path before the logger touches `node:fs`.
