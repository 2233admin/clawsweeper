## Summary

Keep open. Current `main` and the latest stable tag still leave `logging.file` unexpanded before logger directory creation, and the linked fix PR is closed unmerged, so the startup crash remains actionable.

## What This Changes

The issue reports a gateway startup crash loop when `openclaw.json` sets `logging.file` to a path beginning with `~`.

## Best Possible Solution

Land a focused logger-boundary fix that expands a leading `~` before rolling-path detection and directory creation, with regression coverage for both stable and rolling log filenames. Keep the wider config-derived path audit as follow-up unless the logger fix naturally reuses existing helpers without changing unrelated startup behavior.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with a configured non-rolling `~/.openclaw/logs/gateway.log`, current `main` reads the value raw, returns it unchanged from `resolveActiveLogFile`, and sends the literal tilde path to `fs.mkdirSync`.

## Solution Assessment

Yes. The best direction is the narrow helper-based approach from the closed PR, but a replacement should compute expected test home paths through the same production home resolver to avoid the `OPENCLAW_HOME=~/...` mismatch noted in review.
