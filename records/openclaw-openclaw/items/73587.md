## Summary

Keep open. Current main and the latest shipped v2026.4.29 tag still pass raw `logging.file` values from config into logger directory creation without expanding a leading `~`, and the related fix PR #73614 is closed unmerged.

## What This Changes

The issue asks OpenClaw to expand leading `~` in `logging.file` before logger filesystem operations so gateway startup does not crash-loop.

## Best Possible Solution

Land a focused logger-boundary fix that expands leading home prefixes before rolling-path detection and directory creation, adds regression coverage for both stable and rolling log filenames, and treats any wider config-path audit as a separate follow-up unless maintainers explicitly expand scope.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: set `logging.file` to `~/.openclaw/logs/gateway.log` with file logging enabled, and current main passes the literal path through `resolveActiveLogFile()` into `fs.mkdirSync(path.dirname(activeFile))`.

## Solution Assessment

Yes. The narrow maintainable solution is to reuse the existing `expandHomePrefix()` helper at the logger filesystem boundary; rewriting config loading or auditing every path-like key is not required to stop the startup crash.
