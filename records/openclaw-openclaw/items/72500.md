## Summary

Keep open. Current main still conflates disconnected, first-connect, and transient reconnect states: `renderApp` immediately chooses the full credential gate whenever `connected` is false, and the gateway close path clears `connected` before scheduling recoverable reconnects. Open PR #72522 is already linked with closing syntax and remains the active implementation path, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue reports that Control UI WebChat briefly shows the full credentials/login gate on transient tab re-focus or reconnect because prior successful sessions and unauthenticated disconnected sessions share the same render state.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that keeps chat mounted after a prior successful session during recoverable transport reconnects, while restoring the credential gate for first connect and explicit auth, pairing, origin, or device-identity failures.

## Reproduction Assessment

Yes. The deterministic source path is enough: after a successful session, any current-client close runs `onClose`, clears `connected`, and `renderApp` immediately selects the credential-gate branch before recoverable reconnect completes.

## Solution Assessment

Yes. A sticky prior-successful-session or explicit reconnecting state is the maintainable direction; a timing-only debounce would hide the flash but still conflate transient transport loss with first-connect and auth-failure states.
