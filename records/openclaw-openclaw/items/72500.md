## Summary

Keep open. Current main still has the reported two-state Control UI behavior: any disconnected state renders the credentials gate, while reconnect start and active WebSocket close both immediately clear `connected`. PR #72522 is an open, unmerged closing PR for the same bug, so the issue should remain open until that PR is reviewed and merged, closed, or replaced.

## What This Changes

The issue asks for Control UI/WebChat to preserve an authenticated chat view across transient WebSocket reconnects instead of flashing the credentials gate on tab refocus.

## Best Possible Solution

Review and land PR #72522 or an equivalent narrow Control UI fix that adds a prior-successful-session/reconnecting state, keeps the chat UI mounted across transient transport closes, and restores the credentials gate for explicit auth, device identity, origin, or pairing failures with focused tests.

## Reproduction Assessment

Yes. The deterministic code path is enough to reproduce: after a successful session, any active WebSocket close calls the host `onClose`, clears `connected`, and `renderApp` immediately renders the credentials gate for that state.

## Solution Assessment

Yes, the linked sticky prior-successful-session direction is the narrowest maintainable fix if review confirms the auth and pairing reset cases. A timing-only debounce would reduce the flash but would not model the missing authenticated-reconnecting state as directly.
