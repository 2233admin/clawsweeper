## Summary

Keep open. Current main still routes every disconnected Control UI state to the credentials gate, and the linked unmerged PR #72522 is the active implementation path rather than a reason to close before merge.

## What This Changes

The issue asks the Control UI/WebChat to avoid flashing the credentials gate during transient reconnects after a previously successful authenticated session.

## Best Possible Solution

Land PR #72522 or an equivalent narrow Control UI fix that tracks a prior successful session, keeps chat mounted during recoverable transport reconnects, restores the credentials gate only when user action is required, and covers those transitions with focused tests.

## Reproduction Assessment

Yes. The deterministic reproduction path is visible in current main: after a successful connection, a recoverable WebSocket close sets `connected=false`, and `renderApp` immediately selects the credentials-gate branch before reconnect completes.

## Solution Assessment

Yes. A sticky prior-successful-session or explicit reconnecting state is the narrow maintainable solution; a timing-only debounce would hide the flash but still conflate transient transport disconnects with first-connect and auth-failure states.
