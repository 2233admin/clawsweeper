## Summary

Keep open. Current main still has the reported Control UI state transition: any disconnected state renders the credentials gate, and the gateway sets `connected = false` immediately on connect start and active WebSocket close. The issue also has open closing PR #72522, so it should stay open until that PR is reviewed and merged, replaced, or closed.

## What This Changes

The issue asks for Control UI WebChat to preserve the authenticated chat UI, or show a reconnecting state, during transient WebSocket reconnects instead of flashing the full credentials gate.

## Best Possible Solution

Review and land PR #72522, or an equivalent narrow fix, that models a prior successful session/reconnecting state in the Control UI, keeps the chat surface mounted across transient transport closes, and returns to the credentials gate only for first connect or explicit auth, identity, or pairing failures.

## Reproduction Assessment

Yes. The deterministic path is visible in current source: a recoverable WebSocket close calls the gateway `onClose`, `connectGateway` sets `connected = false`, and `renderApp` immediately renders the credentials gate for that state.

## Solution Assessment

Yes, with review. The open PR's sticky prior-successful-session direction is a better fit than a timing-only debounce because it models the missing UI state directly, provided it resets on explicit auth and pairing failures.
