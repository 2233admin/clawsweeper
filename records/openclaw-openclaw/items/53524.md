## Summary

Keep open. The requested Discord post-reset startup marker is not implemented on current main, and the PR addresses a real bundled Discord/session surface, but the proposed patch still has blocking one-shot semantics and active-session routing regressions before it can merge.

## What This Changes

The PR adds a persisted `postRotationStartupUntilMs` session field and uses it to wrap Discord channel/thread post-reset queued owner messages with a startup-recovery steer prompt.

## Best Possible Solution

A durable fix should add one persisted Discord reset-startup marker, apply it only to the first qualifying owner turn, clear it after both normal and embedded-queue steer consumption paths, preserve current active-session-id resolution, and include focused Discord channel and thread regression coverage.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction: after a reset window is set, a non-owner Discord run can start and clear the marker before the owner's first turn, and a successful embedded steer enqueue can return before clearing the marker at all.

## Solution Assessment

No. The direction is useful, but this PR is not the best final fix until it makes the marker one-shot on actual steer consumption and preserves the active session id used by current main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Clear the window only after owner steer consumption:** `src/auto-reply/reply/agent-runner.ts:854`
  - body: `startedRun` is set for any agent turn, but the startup steer only applies to owner Discord messages. If a non-owner channel message starts a run before the owner's first post-reset turn, this `finally` block clears `postRotationStartupUntilMs` and the owner turn never receives the recovery prompt.
  - confidence: 0.88
- **[P2] Clear the window before embedded steer returns:** `src/auto-reply/reply/agent-runner.ts:276-277`
  - body: When `queueEmbeddedPiMessage` succeeds, this early return exits before the later clear path runs. The marker remains active, so later owner messages inside the 30 second window can be wrapped again as the first post-reset message.
  - confidence: 0.86
- **[P2] Preserve the active session id for embedded steers:** `src/auto-reply/reply/agent-runner.ts:276`
  - body: Current main resolves the active session id through `replyRunRegistry.resolveSessionId(sessionKey)` before queueing an embedded steer. This PR queues with `followupRun.run.sessionId`, which can target a stale pre-reset id after session rotation.
  - confidence: 0.74
