## Summary

Keep this PR open, but do not merge it as-is. Current main still has the active-but-not-streaming steering gap, while the proposed forward-compat Pi seam appears to misclassify existing steering callers that omit the new option as interrupt-style steering once the paired Pi API lands.

## What This Changes

The PR changes embedded Pi reply steering to attempt delivery while a run is active but not streaming, adds per-message inject/interrupt steering behavior, and expands related run-registry and auto-reply tests.

## Best Possible Solution

Land this only after the OpenClaw side treats omitted/default active-run steering as normal inject behavior, reserves interrupt for stop-like messages, covers the non-auto-reply steering callers, and validates the final behavior against the paired Pi API without broadening the dependency or queue-mode scope.

## Reproduction Assessment

Yes. The current main gap is source-reproducible by following a steer-mode run with isActive true and isStreaming false through runReplyAgent and queueEmbeddedPiMessage; the introduced PR defect is source-reproducible by following existing subagent announce steering callers that do not pass toolCallSteeringBehavior.

## Solution Assessment

No, not yet. Steering active non-streaming Pi runs is the right direction for the reported behavior, but a session-wide interrupt default is not the narrowest maintainable implementation unless every normal steering path explicitly defaults to inject.
