## Summary

Keep this PR open but do not merge it as-is. Current main still gates Pi steering on streaming state, so the PR targets a real active-run steering gap, but its session-wide interrupt default would make existing normal steering callers inherit interrupt behavior unless every omitted/default path is handled as inject.

## What This Changes

The PR changes embedded Pi reply steering to target active non-streaming runs, adds per-message inject/interrupt steering behavior, removes the embedded run registry streaming guard, and expands related auto-reply and run-registry tests.

## Best Possible Solution

Land the active non-streaming steering fix only after omitted/default Pi steering behavior remains normal inject, stop-like text is the only interrupt path, subagent wakeup/completion callers are covered, and the forward-compatible Pi seam is validated against the final dependency API.

## Reproduction Assessment

Yes. Current main is source-reproducible by following a steer-mode reply with isActive true and isStreaming false through runReplyAgent and queueEmbeddedPiMessage; the PR defect is source-reproducible by following subagent announce steering callers that do not pass toolCallSteeringBehavior.

## Solution Assessment

No, not yet. Steering active non-streaming Pi runs is the right direction, but a session-wide interrupt default is not the narrowest maintainable implementation unless omitted/default steering paths explicitly resolve to inject.
