## Summary

Keep this PR open, but do not merge it as-is. Current main still has the active non-streaming steering gap this PR targets, but the patch introduces a blocking default-behavior regression: normal steering callers that omit the new option would inherit interrupt-style Pi tool steering once the paired Pi API exists.

## What This Changes

The PR changes embedded Pi reply steering so active non-streaming runs can receive injected or interrupt steering messages, adds a typed forward-compatible Pi steering option, and expands Pi runner and auto-reply tests.

## Best Possible Solution

Keep the active non-streaming steering behavior, but make the OpenClaw-side contract default omitted Pi tool-call steering behavior to normal inject, reserve interrupt for explicit stop-like steering, cover non-auto-reply callers, and validate against the final published Pi API without changing dependency versions in this PR.

## Reproduction Assessment

Yes. The current-main gap is source-reproducible by following a steer-mode reply with isActive true and isStreaming false through runReplyAgent and queueEmbeddedPiMessage; the PR defect is source-reproducible by tracing existing subagent steering callers that omit toolCallSteeringBehavior.

## Solution Assessment

No, not yet. Steering active non-streaming Pi runs is the right direction, but a session-wide interrupt default is not the narrowest maintainable solution unless ordinary omitted/default steering is inject and interrupt remains explicit.
