## Summary

Keep open. Current main still has no supported `openclaw mcp serve --scope` or mode selector, and the MCP bridge still constructs its gateway client with the fixed read/write/approvals scope set. An open unmerged PR, #74030, explicitly uses closing syntax for this issue, so the maintainer action is to review/land or close that PR rather than close this issue now.

## Best Possible Solution

Keep this issue open and review #74030 as the current focused implementation candidate. The best product/code path is a backward-compatible repeatable `openclaw mcp serve --scope <operator.scope>` surface that threads `OperatorScope[]` through `OpenClawMcpServeOptions`, `OpenClawChannelBridge`, and `GatewayClient`, preserves the current default when omitted, validates with `isOperatorScope`, updates `docs/cli/mcp.md`, and adds targeted CLI/server tests. Treat tool-list filtering as a separate follow-up unless maintainers decide it is required for the first UX.

- Pairing/device-token UX needs review so requested scopes, cached device-token scopes, and previously approved scopes remain understandable when a bridge requests a narrower or different set.
- If `tools/list` remains static, documentation or invocation-time errors should make clear that authorization is enforced on tool calls rather than by tool registration visibility.
