## Summary

Keep open. Current main still lacks a supported way for `openclaw mcp serve` to request narrower operator scopes: the CLI exposes no scope/mode option, the serve options do not carry requested scopes, and the MCP channel bridge still constructs its Gateway client with the fixed read/write/approvals scope set. The related #49183 remains broader per-session tool-policy work and does not supersede this narrower MCP bridge pairing-scope request.

## Best Possible Solution

Keep this issue open and implement a small core MCP bridge scope-selection surface. A conservative path is to preserve the current default `[operator.read, operator.write, operator.approvals]`, add a validated repeatable `openclaw mcp serve --scope <operator.read|operator.write|operator.approvals>` and/or explicit named modes, thread the selected scopes through `OpenClawMcpServeOptions` into `OpenClawChannelBridge`, and add CLI/MCP bridge tests plus docs at `docs/cli/mcp.md`. Treat `tools/list` filtering as a follow-up unless maintainers choose to make it part of the UX.
