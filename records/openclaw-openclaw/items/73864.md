## Summary

Keep open. Current main still has no configurable `openclaw mcp serve` operator-scope request path: the CLI exposes no `--scope` option, the MCP serve options cannot carry scopes, and the bridge still constructs `GatewayClient` with the fixed `[operator.read, operator.write, operator.approvals]` set. The provided GitHub context also shows open unmerged PR #74030 uses closing syntax for this issue, so the right cleanup path is to review/land or close that PR first, not close this issue now.

## Best Possible Solution

Keep this issue open and review #74030 as the current focused implementation candidate. The best code path is validated repeatable `openclaw mcp serve --scope <operator.scope>` plumbing through `OpenClawMcpServeOptions`, `OpenClawChannelBridge`, and `GatewayClient`, preserving the current default when omitted, updating the public MCP docs, and adding CLI/server regression tests; treat tool-list filtering as a follow-up unless maintainers require it for the first UX.
