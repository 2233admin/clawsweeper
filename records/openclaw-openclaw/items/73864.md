## Summary

Keep open. Current main still lacks configurable `openclaw mcp serve` operator scopes, and the provided discussion shows the request is already paired with open replacement PR #74428 after #74030 was closed unmerged; the maintainer path is to review or land the linked PR, not close this issue or queue duplicate work.

## What This Changes

Add a supported way for `openclaw mcp serve` to request narrower operator scope sets for least-privilege MCP bridge identities.

## Best Possible Solution

Land #74428 or an equivalent narrow implementation that validates repeatable MCP bridge scopes, threads an `OperatorScope[]` through CLI, server, bridge, and `GatewayClient`, preserves the default read/write/approvals request when omitted, documents the contract, and covers explicit/default/cached-token behavior with tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists on current main: `mcp serve` has no scope option or serve option field, and the bridge hardcodes read/write/approvals when constructing `GatewayClient`.

## Solution Assessment

Yes. A validated repeatable scope option limited to MCP bridge-relevant operator scopes is the narrow maintainable direction; #74428 is the active implementation vehicle, while static `tools/list` filtering can remain a documented or follow-up behavior if invocation-time gateway auth is canonical.
