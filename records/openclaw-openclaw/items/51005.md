## Summary

Keep open. Current main still rejects Brave `llm-context` freshness/date filters even though Brave documents `freshness` support, so the user problem remains valid; the PR should be rebased and adjusted because it targets stale pre-runtime-split surfaces and leaves one-sided date-range behavior unresolved.

## What This Changes

The PR removes Brave `llm-context` rejection guards for existing freshness/date parameters, converts date filters into Brave freshness ranges, updates docs, and adds regression coverage.

## Best Possible Solution

Port the fix into `extensions/brave/src/brave-web-search-provider.runtime.ts`, keep `ui_lang` rejected for `llm-context`, pass supported `freshness` and deliberate date-range values to Brave, then update the Brave/web docs and focused extension regression tests.
