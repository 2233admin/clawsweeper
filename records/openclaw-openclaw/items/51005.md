## Summary

Keep open. The reported Brave LLM Context filter bug is still real on current main, and the PR has useful source work, but the branch is not merge-ready because it targets the older monolithic Brave provider/test layout while current main executes Brave search through the lazy runtime. The branch also needs a maintainer decision on one-sided date-range semantics before it is the best fix.

## What This Changes

The PR unblocks Brave `llm-context` `freshness`, `date_after`, and `date_before`, converts date filters into Brave `freshness` ranges, adjusts the cache key, updates web-search docs, and changes tests to assert the LLM Context endpoint receives those parameters.

## Best Possible Solution

Rebase the useful Brave-specific change onto current main, move the unblock and date-range forwarding into `extensions/brave/src/brave-web-search-provider.runtime.ts`, keep `ui_lang` rejected for `llm-context`, make one-sided date semantics explicit, and update both `docs/tools/web.md` and `docs/tools/brave-search.md` with focused Brave plugin tests.

## Reproduction Assessment

Yes. On current main, Brave `mode: "llm-context"` plus `freshness`, `date_after`, or `date_before` deterministically returns local unsupported errors in `executeBraveSearch` before any Brave network request.

## Solution Assessment

No. The direction is correct, but this PR is not the best mergeable solution as written because it edits stale integration points and leaves one-sided date-range behavior unresolved; the safer fix belongs in the active Brave runtime with matching docs and tests.

  - body: When only `date_after` is set, this builds `freshness=<date_after>to<today>`. A future start date produces a reversed range, so the newly enabled path would send an invalid Brave request instead of returning a deterministic local validation error.
  - confidence: 0.83
- **[P2] Preserve open-ended date_before semantics:** `extensions/brave/src/brave-web-search-provider.ts:237-238`
  - body: `date_before` is documented as results before the given date, but this branch rewrites it to `1970-01-01to<date_before>`, silently excluding older sources. Historical searches in `llm-context` mode would be scoped more narrowly than requested.
  - confidence: 0.78
