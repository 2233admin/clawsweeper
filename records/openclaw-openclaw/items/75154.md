## Summary

Keep open. Current main still excludes Responses-family transports from the malformed tool-call argument repair gate, and this issue is asking for maintainer/provider-contract guidance before changing that policy. The source shows HTTP/SSE Responses can emit the internal `toolcall_delta` shape the wrapper watches, but current tests intentionally keep `openai-responses` off and the OpenAI WebSocket path is different, so an automatic close or gate-only repair would be premature.

## What This Changes

The issue asks whether `openai-codex-responses`, and possibly `openai-responses` and `azure-openai-responses`, should opt into the existing malformed streamed tool-call argument repair gate.

## Best Possible Solution

Decide the Responses-family policy explicitly. If maintainers confirm these transports can emit recoverable malformed argument fragments, add only the supported transport IDs to the gate with focused tests for the exact model APIs and separately cover or document the WebSocket final-response behavior.

## Reproduction Assessment

Unclear. The current exclusion is reproducible from source and tests, but the issue explicitly says no malformed Codex arguments have been observed, so there is not yet a high-confidence reproduction of a user-facing failure.

## Solution Assessment

Unclear. A narrow gate expansion may be correct for HTTP/SSE Responses streams, but the WebSocket Responses path is different enough that maintainer/provider-contract review should come before an automated gate-only patch.
