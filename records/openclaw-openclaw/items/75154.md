## Summary

Keep open. Current main still excludes `openai-codex-responses` and other Responses-family transports from the malformed tool-call argument repair gate, and the issue is a fresh provider-contract question rather than an obsolete or already-implemented request.

## What This Changes

The issue asks whether malformed tool-call argument repair should be enabled for `openai-codex-responses`, `openai-responses`, and `azure-openai-responses`.

## Best Possible Solution

Maintainers should decide the Responses-family repair policy explicitly. If Codex/Responses can emit recoverable malformed argument deltas, add only the confirmed transport IDs to the gate with focused tests and either cover or document the WebSocket final-response path separately.

## Reproduction Assessment

Unclear. The current exclusion is reproducible statically from the gate and tests, but the issue explicitly says no malformed Codex arguments have been observed, so there is no high-confidence runtime reproduction of a broken tool call.

## Solution Assessment

Unclear. A narrow gate expansion may be correct for HTTP/SSE Responses because that path emits internal `toolcall_delta` events, but WebSocket Responses uses a different final-message path, so maintainer/provider-contract review is needed before choosing the implementation.
