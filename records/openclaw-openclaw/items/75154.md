## Summary

Keep open. Current main still excludes Responses-family transports from the malformed tool-call argument repair gate, and this new report is asking for maintainer/provider-contract guidance before a PR rather than reporting an observed failure. The code suggests HTTP Responses streams do produce internal `toolcall_delta` events, but the WebSocket Responses path is different, so an automatic close or blind gate expansion would be premature.

## What This Changes

The issue asks whether malformed tool-call argument repair should also be enabled for `openai-codex-responses` and possibly other Responses-family transports.

## Best Possible Solution

Decide the Responses-family policy explicitly. If maintainers confirm Codex/Responses can emit recoverable malformed argument deltas, add the exact supported transport IDs to the gate with focused tests for `openai-responses`, `openai-codex-responses`, and `azure-openai-responses`, and cover or document the WebSocket final-response path separately.

## Reproduction Assessment

Unclear. We can reproduce the current exclusion statically from the gate and tests, but the issue explicitly says no malformed Codex arguments have been observed, so there is not yet a high-confidence runtime reproduction of the user-facing failure.

## Solution Assessment

Unclear. A narrow gate expansion may be right for HTTP/SSE Responses because that path emits `toolcall_delta`, but the WebSocket path does not use the same internal delta stream for tool arguments, so the best fix needs maintainer/provider-contract review before implementation.
