## Summary

Keep open. Current main still excludes `openai-codex-responses`, `openai-responses`, and `azure-openai-responses` from the malformed tool-call argument repair gate, and the provided GitHub context shows open PR #75281 already targets this issue with closing syntax, so cleanup should not close the issue before that PR is reviewed, landed, or closed.

## What This Changes

Clarify whether malformed tool-call argument repair should be enabled for `openai-codex-responses` and possibly other Responses-family transports.

## Best Possible Solution

Make the Responses-family policy explicit through the linked fix path: enable repair only for transports whose stream actually emits compatible `toolcall_delta` events, keep or separately handle WebSocket-capable paths, and add focused positive and negative gate tests for the chosen matrix.
