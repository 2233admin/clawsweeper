## Summary

Keep open. Current main still excludes Responses-family transports from the malformed tool-call argument repair gate, and this report is asking for maintainer/provider-contract guidance rather than reporting an observed runtime failure. The source shows the HTTP/SSE Responses transport emits internal `toolcall_delta` events that the wrapper can watch, but the WebSocket Responses path completes through a different final-response parser, so a gate-only change needs maintainer review before implementation.

## What This Changes

The issue asks whether malformed tool-call argument repair should be enabled for `openai-codex-responses` and possibly other Responses-family transports.

## Best Possible Solution

Decide the Responses-family policy explicitly. If maintainers confirm these transports can emit recoverable malformed argument deltas, add the exact supported model API IDs to the gate with focused tests for `openai-responses`, `openai-codex-responses`, and `azure-openai-responses`, and either cover or document the separate WebSocket final-response path.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence reproduction of the current exclusion, but the issue explicitly says no malformed Codex arguments have been observed, so there is no high-confidence runtime reproduction of a user-visible failure yet.

## Solution Assessment

Unclear. A narrow gate expansion may be the right solution for HTTP/SSE Responses because that path emits `toolcall_delta`, but the WebSocket path does not use the same repair surface, so maintainer/provider-contract review is the safer next step.
