## Summary

Keep open. Current main still excludes `openai-codex-responses`, `openai-responses`, and `azure-openai-responses` from the malformed tool-call argument repair gate, and PR #75281 is already the open implementation candidate with closing syntax, so this issue should not be closed before that PR is reviewed, merged, or abandoned.

## What This Changes

The issue asks whether malformed tool-call argument repair should be enabled for `openai-codex-responses` and possibly other Responses-family transports.

## Best Possible Solution

Resolve the transport policy through the linked implementation path: enable the wrapper only for Responses-family IDs with proven compatible `toolcall_delta` partial-JSON streams, keep WebSocket-capable exclusions explicit unless tested, and land focused positive/negative gate tests plus a changelog entry if behavior changes.

Yes for the current exclusion: current main statically shows the gate returns true only for Kimi `anthropic-messages` and `openai-completions`. No for a live user-facing malformed Codex failure: the report says malformed args have not been observed yet.

## Solution Assessment

Unclear until PR #75281 is reviewed. A narrow transport-specific expansion may be correct, but maintainers should verify the Codex, Azure, and generic OpenAI Responses paths before closing this issue.
