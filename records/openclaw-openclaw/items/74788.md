## Summary

Keep open. Current main still lacks an OpenRouter-owned thinking profile or legacy xhigh hook, so OpenRouter DeepSeek V4 models fall back to the base `off|minimal|low|medium|high` profile and explicit `/think xhigh` can still be rejected before the OpenRouter transport wrapper runs. The linked open PR #74839 is the active implementation candidate and should be reviewed rather than closing this issue early.

## What This Changes

Allow OpenRouter DeepSeek V4 Pro and Flash models to accept `/think xhigh` by advertising provider-owned thinking support instead of falling back to the base thinking levels.

## Best Possible Solution

Land or revise the OpenRouter-local fix so `deepseek/deepseek-v4-pro` and `deepseek/deepseek-v4-flash` expose the correct thinking levels through `resolveThinkingProfile`, with focused tests for positive V4 cases and negative non-V4/non-reasoning cases. Keep the change inside the OpenRouter plugin boundary; no core resolver change appears necessary.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: choose `openrouter/deepseek/deepseek-v4-pro` or Flash, send `/think xhigh`, and the directive validation path will consult the base OpenRouter profile and reject `xhigh` because current main has no provider profile or compat effort metadata for those models.

## Solution Assessment

Yes. The narrow maintainable solution is an OpenRouter-owned thinking profile or equivalent compatibility hook, because the shared resolver already consumes provider profiles and the OpenRouter stream wrapper already knows how to forward accepted `xhigh` requests.
