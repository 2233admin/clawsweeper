## Summary

Keep open. Current main still does not implement the requested opt-in coalescing/final-only delivery for tool-interleaved assistant text. Final payload assembly preserves multiple assistant text blocks as separate reply payloads, generic dispatch sends each payload separately, and the block-streamed path still flushes before tool execution. The closed related PR #16275 remains useful prior art but was not merged.

## Best Possible Solution

Keep this issue open as the focused core reply-pipeline feature. The best implementation is an opt-in coalescing or final-only delivery mode that covers both final embedded-run assistantTexts and block-streamed text around sequential and parallel tool execution, likely using hold/defer semantics similar to #16275, plus config schema/docs and regression tests for text -> tool -> text -> tool -> text turns.
