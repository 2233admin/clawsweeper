## Summary

Keep open. Current main still does not implement opt-in coalescing of tool-interleaved assistant text into one outbound message; the final payload path preserves multiple assistant text blocks as multiple reply items, and the streamed block path still flushes before tool execution.

## Best Possible Solution

Keep this issue open as the focused core reply-pipeline feature. The best implementation is an opt-in coalescing mode that merges assistant text blocks from one agent turn into one outbound text reply, covers both final embedded-run payload assembly and block-streamed text around sequential/parallel tool execution, preserves non-text delivery metadata, updates config schema/docs, and adds regression tests for text -> tool -> text -> tool -> text turns.

## Clownfish Work Prompt

Implement an opt-in core reply-pipeline feature for this issue: when enabled, assistant text blocks from one agent turn that are interleaved with tool calls should be delivered as one outbound text message, separated by blank lines, instead of one outbound message per text block. Keep current behavior as the default. Use prior art from #16275 for hold/resume around tool execution, but make the final embedded-run payload path work too. Expected boundary: add a documented config option under agent defaults, wire schema/help/docs, merge only compatible visible text payloads, and preserve media, reply target, reasoning, compaction notice, silent reply, TTS/audio, and channel-specific metadata boundaries. Do not introduce channel-specific Telegram-only policy and do not suppress final/media/error delivery. Add regression coverage for text -> tool -> text -> tool -> text with block streaming off and on, including sequential and nested/parallel tool execution where practical. Changelog only if maintainers consider the new config user-facing enough for the active release notes.
