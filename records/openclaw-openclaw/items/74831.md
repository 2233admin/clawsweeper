## Summary

Keep open. The PR targets a real sandbox-registry contention problem, but the current branch removes same-entry mutation ordering and hides unexpected delete failures, so it should not be merged as-is.

## What This Changes

The PR shards sandbox container and browser registries into per-entry JSON files, adds single-entry registry lookup for Docker ensure, migrates legacy monolithic files, and updates sandbox registry/config-hash tests.

## Best Possible Solution

Keep the sharded-registry direction only after the branch preserves same-entry mutation ordering with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol, and only suppresses expected missing-file removal cases while restoring focused race and delete-error regression coverage.

## Reproduction Assessment

Yes for the introduced regression. Current main has focused tests that gate a registry write, race a removal, and assert the removed entry stays deleted; the PR removes that protection and its read-then-write per-entry path can write after delete.

## Solution Assessment

No. Per-entry sharding is a plausible performance direction, but this implementation is not the best maintainable fix until it preserves same-entry ordering and propagates unexpected I/O failures.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:219`
  - body: Current main serializes registry mutations and tests that a delayed update cannot resurrect an entry removed after that update started. This write can read the old entry and later recreate the per-entry file after removeRegistryEntry deletes it, leaving stale container or browser state after recreate/prune. Add per-entry locking, tombstones, compare-and-swap, or equivalent ordering and restore the race tests.
  - confidence: 0.9
- **[P3] Propagate unexpected registry removal errors:** `src/agents/sandbox/registry.ts:112-116`
  - body: fs.rm(..., { force: true }) already ignores missing files, but this catch also hides EACCES, ENOTDIR, and other I/O failures. Sandbox recreate/prune can report success while the registry file remains, so only suppress ENOENT or a deliberately handled concurrent-remove case.
  - confidence: 0.82
