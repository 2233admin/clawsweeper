## Summary

Keep open. The latest PR head still drops the same-entry mutation ordering that current main implements and tests, so an in-flight update can recreate a registry entry after recreate/prune removes it; the shared remove helper also hides unexpected filesystem failures.

## What This Changes

The PR shards sandbox container and browser registries from monolithic JSON files into per-entry JSON files, adds `readRegistryEntry` for Docker ensure, migrates legacy registries, and rewrites related sandbox registry tests.

## Best Possible Solution

Keep the sharded-storage direction only if it preserves same-runtime-name update/remove ordering with a per-entry lock, tombstone, compare-and-swap, or equivalent protocol; unexpected filesystem errors should propagate, legacy migration should remain covered, and focused container/browser race tests should be restored.

## Reproduction Assessment

Yes for the PR regression. Current main already has focused tests that delay an update, race removal, and assert the entry remains deleted; the same scenario against the PR head’s uncoordinated per-entry write/remove path can resurrect stale registry state.

## Solution Assessment

No. Sharding disjoint entries is a plausible performance direction, but dropping all coordination for mutations of the same runtime name is not the narrowest maintainable fix; per-entry coordination keeps the performance goal without weakening recreate/prune semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve same-entry remove/update ordering:** `src/agents/sandbox/registry.ts:223`
  - body: Current main serializes registry mutations and tests that a remove starting after an in-flight update leaves the entry deleted. Here `updateRegistry` can read the entry and later finish `writeJsonAtomic` after `removeRegistryEntry` deletes the file, so recreate/prune can leave stale runtime state for the same container name. Keep per-entry coordination or equivalent tombstone/compare semantics for same-name mutations.
  - confidence: 0.9
- **[P3] Propagate registry removal failures:** `src/agents/sandbox/registry.ts:112-116`
  - body: `fs.rm(..., { force: true })` already ignores missing files, but this catch also hides `EACCES`, `ENOTDIR`, and other I/O failures. Sandbox recreate/prune can report success after the runtime is removed while the registry file remains, so later list/prune sees stale state. Only ignore expected missing or concurrent-remove cases.
  - confidence: 0.82
