## Summary

Keep open. Current main still accepts an announce cron delivery for `feishu`/`lark` without `delivery.to` at the protocol/gateway/service/CLI creation boundaries, while Feishu outbound delivery still requires a concrete target at send time. The issue also has open PRs with closing syntax, so the right maintainer path is to review and land or close one canonical fix PR rather than closing the issue now.

## Best Possible Solution

Keep this issue open and review one canonical focused PR. The best fix is create/update validation at the cron service/gateway boundary for Feishu/Lark announce delivery when neither explicit `delivery.to` nor a deterministic resolved/default target exists, with optional CLI/UI fail-fast feedback and regression coverage for missing target rejection plus explicit target acceptance.
