## Summary

Keep open. Current main still has useful abort, queue cleanup, and process-cancel primitives, but it does not provide the requested unified per-session run invalidation, broad stale-output fencing, pre-tool currentness gate, or default new-message takeover; the same-author draft PR #70363 remains open and explicitly partial.

## What This Changes

Request stronger run-scoped interruption semantics so `/stop` or a new message invalidates the active run, fences stale output, cancels subprocesses, skips pending tools, and lets the new message take over.

## Best Possible Solution

Keep this as the canonical core interruptibility tracker. Review or split #70363, then land focused core changes for a session/run generation primitive, stale-output fences across reply, typing, followup, tool-progress, and event emission paths, a pre-tool abort/currentness gate in the Pi tool path, new-message takeover that invalidates and cancels the active run, and regression tests proving stale runs cannot emit visible effects or launch further tools.
