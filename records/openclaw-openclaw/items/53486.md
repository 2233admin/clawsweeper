## Summary

Keep this issue open. Current main still routes a valid Feishu native card JSON string in the plain `message` parameter as text: the shared runner only parses structured `presentation`, `delivery`, and `interactive` fields, and the Feishu action handler only sends cards when normalized `presentation` is present. The merged native-card payload work supports explicit `channelData.feishu.card` payloads, but it does not cover the compatibility path reported here, and the issue discussion includes a fresh 2026-04-29 confirmation on OpenClaw 2026.4.25.

## What This Changes

The issue asks for Feishu `message(action=send)` to detect valid interactive-card JSON supplied as the plain `message` string and send it as a native interactive card instead of a post/text message.

## Best Possible Solution

Add Feishu-local compatibility detection for valid native interactive-card JSON strings in `send` and `thread-reply`, sanitize through the existing native-card sanitizer, route valid cards to `sendCardFeishu`, and leave malformed card JSON or ordinary JSON on the text path. Keep the shared message schema provider-agnostic unless a small generic plugin seam becomes necessary.
