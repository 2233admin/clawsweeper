## Summary

Keep open. Current main partially covers the bare gateway `/new` path by injecting a non-empty reset/startup prompt, but the embedded runner still has provider-dispatch paths that can submit an empty prompt with no substantive replayable messages. The remaining gaps match the report's active-memory/runtime-context race and stale blank-message concern, so this is not safe to close as implemented.

## Best Possible Solution

Add a conservative embedded-runner dispatch-boundary guard shared by Anthropic/OpenAI-style submissions: treat a request as empty unless it has a non-blank prompt, images, or provider-replayable messages with non-empty content after runtime-context/custom-message stripping. For bare `/new`/`/reset` startup cases, reuse the existing session-reset prompt behavior; otherwise skip or fail cleanly with a terminal diagnostic instead of leaving the session processing. Cover blank/stale history and runtime-only context with targeted tests.

## Clownfish Work Prompt

Fix this issue. Current main already injects a non-empty bare reset prompt for the gateway `/new` path, but the embedded runner can still submit `activeSession.prompt("")` for runtime-only context, and `hasPromptSubmissionContent` treats any history object as content even when provider-replayable content is blank. Add a conservative guard at the embedded runner/provider dispatch boundary so no Anthropic/OpenAI-style provider request is made with an empty prompt, no images, and no substantive replayable messages. Treat runtime-context custom messages and blank text blocks as empty for this guard. Preserve the existing transcript-hygiene behavior: do not persist hidden runtime context as user content, and do not edit generated `dist` files. For real bare `/new`/`/reset` startup, keep using the existing reset/startup prompt wording from `src/auto-reply/reply/session-reset-prompt.ts`; for other empty submissions, skip or fail cleanly with a terminal diagnostic so sessions do not remain processing. Add regression tests for empty prompt plus empty messages, blank/stale user-message history, and runtime-only context with no provider-replayable history. Add a user-facing changelog fix entry with `Thanks @giangthb` if the code behavior changes.
