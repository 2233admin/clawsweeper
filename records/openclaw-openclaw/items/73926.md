## Summary

Keep open. Current main fixes the bare `/new` visible-prompt case and skips blank non-runtime user prompts, but the active-memory/runtime-only path can still submit `prompt: ""` with empty Anthropic-compatible `messages`, so the reported provider-boundary failure is only partially addressed.

## What This Changes

The issue asks OpenClaw to prevent `/new` or `/reset` active-memory races from dispatching empty prompt/messages payloads to Anthropic-compatible and OpenAI Responses providers.

## Best Possible Solution

Define and implement a provider-safe runtime-only contract: hidden active-memory/runtime context should stay out of the visible transcript, while Anthropic-compatible, MiniMax, and OpenAI-family transports never receive an empty model-visible request. The fix likely belongs at the embedded-runner/provider-adapter boundary with focused tests for active-memory flush, bare reset, and empty Anthropic messages.

## Reproduction Assessment

Yes at source level. A non-empty runtime prompt with `transcriptPrompt: ""` becomes `runtimeOnly` with `prompt: ""`, bypasses the blank-prompt skip, and Anthropic transport currently converts empty context history into `messages: []`.

## Solution Assessment

No, current main is not the full solution. #74168 is the right narrow fix for blank visible user prompts and bare reset markers, but it does not settle runtime-only provider input for active-memory-style hidden context.
