## Summary

Keep open. Current main prevents blank visible user prompts and fixes the bare `/new` transcript-marker case, but the runtime-only path still intentionally permits an empty visible prompt to reach the embedded runner, and the linked Anthropic transport PR is open and scoped separately from the embedded-runner contract.

## What This Changes

The issue requests preventing `/new` or `/reset` with active memory from dispatching empty prompt/messages payloads to providers and leaving sessions stuck in processing.

## Best Possible Solution

Define one provider-safe runtime-only contract at the embedded-runner or provider-adapter boundary: hidden runtime context can stay out of the visible transcript, but every provider call must still have acceptable model-visible input or be deliberately skipped with structured diagnostics.

## Reproduction Assessment

Yes at source level. A non-empty effective runtime prompt with `transcriptPrompt: ""` resolves to `prompt: ""` and `runtimeOnly: true`; the active-memory flush path supplies exactly that shape, and the runner submits it instead of skipping it.

## Solution Assessment

No, current main is not the complete solution. The merged guard is the right narrow fix for blank visible user prompts and bare reset markers, but it intentionally leaves runtime-only empty visible submissions unresolved.
