## Summary

Keep open. Current main fixes bare reset transcript markers and non-runtime blank visible prompt submissions, but the active-memory/runtime-only path can still submit an empty visible prompt to the embedded session, and the pinned Anthropic-compatible provider code drops that blank user turn into an empty `messages` payload.

## What This Changes

The issue reports that `/new` or `/reset` with active-memory enabled can race into an empty embedded-run prompt/messages payload, causing Anthropic/OpenAI-compatible provider errors and stuck processing sessions.

## Best Possible Solution

Define a provider-safe runtime-only embedded-run contract so hidden runtime context stays out of the visible transcript while every provider call still has accepted model-visible input or intentionally does not call the provider. The implementation should cover active-memory flushes, runtime events, and heartbeat-like turns with focused runner/helper tests.

## Reproduction Assessment

Yes, at source level. A memory flush with non-empty `prompt` and blank `transcriptPrompt` resolves to `prompt: ""`, bypasses the blank-prompt guard because it is `runtimeOnly`, and is submitted to a dependency path that filters blank Anthropic user content into an empty message list.

## Solution Assessment

No, the suggested local patch is not clearly the best upstream fix because injecting a visible greeting at the runner boundary may leak runtime-only work into transcripts. A safer maintainable fix should first settle the runtime-only provider contract, then implement that contract at the embedded-runner or provider-adapter boundary.
