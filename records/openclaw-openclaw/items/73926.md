## Summary

Keep open. Current main and v2026.4.29 cover the bare `/new` marker and blank visible-prompt subcases, but the runtime-only path still deliberately converts non-empty hidden context plus blank `transcriptPrompt` into `prompt: ""` and submits it to the embedded session. That leaves the provider-visible empty-input contract unresolved rather than fully implemented.

## What This Changes

The issue asks OpenClaw to prevent `/new` or `/reset` races with active-memory/runtime context from dispatching an empty prompt and empty message payload to Anthropic-style providers.

## Best Possible Solution

Define a provider-safe runtime-only contract at the embedded-runner or provider-adapter boundary so hidden runtime context can remain outside visible transcript turns while provider calls never contain no model-visible input. The solution should explicitly cover memory flush, active-memory prompt context, heartbeat/system events, and Anthropic-compatible providers.

## Reproduction Assessment

Yes at source level. A non-empty `effectivePrompt` with blank `transcriptPrompt` resolves to `prompt: ""` and `runtimeOnly: true`, the skip helper returns null for runtime-only submissions, and the runner still submits that empty prompt.

## Solution Assessment

No, the local patch direction of injecting a visible greeting everywhere is not clearly the best fix because it can change hidden runtime-context and transcript semantics. A narrower maintainer-designed runtime-only provider contract or adapter-level hidden-input mechanism is safer.
