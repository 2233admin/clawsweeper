## Summary

Keep open. Current main still hardcodes short-term promotion output to `MEMORY.md`, exposes no `promotionTarget` or equivalent memory-core config/CLI option, and the docs still describe both manual and managed deep promotion as writing only to `MEMORY.md`. The adjacent open memory-core PRs address contamination, size budgeting, and recall gates, but not this configurable target-file feature.

## What This Changes

The issue requests a backward-compatible memory-core setting that lets short-term promotions write to a configurable Markdown target file, including date-token paths, while keeping promoted content searchable.

## Best Possible Solution

Implement the target selection inside the bundled memory-core promotion path, defaulting to `MEMORY.md`, with a safe workspace-relative Markdown resolver, deterministic date-token expansion, config schema/docs updates, both manual CLI and managed dreaming plumbing, and indexing/search coverage for the resolved target file.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `applyShortTermPromotions()` always writes to `path.join(workspaceDir, "MEMORY.md")`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review is read-only.

## Solution Assessment

Yes. The requested memory-core-owned config option is the narrowest maintainable direction because it preserves the current default while routing all promotion writers through one safe resolver and search/indexing contract. Requiring users to maintain `memorySearch.extraPaths` alone would not solve the write target behavior.

Add a backward-compatible setting under `plugins.entries.memory-core.config.dreaming.phases.deep` or another clearly documented memory-core-owned config location, defaulting to `MEMORY.md`.
