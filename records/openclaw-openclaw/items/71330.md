## Summary

Keep open. Current main still hardcodes short-term promotion output to `MEMORY.md`; there is no `promotionTarget` or equivalent config in the memory-core manifest/runtime config, no CLI or managed dreaming target override, and docs/tests still describe promotion as appending into `MEMORY.md`. The related memory/wiki issues are broader lifecycle or vault-structure proposals and do not supersede this specific target-file feature.

## Best Possible Solution

Implement this in the bundled memory-core plugin as a backward-compatible promotion target setting defaulting to `MEMORY.md`. Resolve a safe workspace-relative Markdown target with documented date-token support, thread it through manual `memory promote --apply` and managed deep dreaming, create missing parent directories/files with the existing long-term-memory header behavior, and ensure the resolved target is indexed/searchable by builtin and QMD memory search. Update manifest schema, config resolution, focused promotion/CLI/dreaming/indexing tests, public memory docs, and the user-facing changelog.
