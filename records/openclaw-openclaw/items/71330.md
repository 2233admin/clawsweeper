## Summary

Keep #71330 open. Current main still hardcodes short-term promotion writes to `MEMORY.md`; there is no `promotionTarget` or equivalent memory-core config surface, and docs/tests still describe and assert `MEMORY.md` as the only deep-promotion target. The related issues cited in the report are broader memory-wiki/adaptive-memory work and do not supersede this specific bundled memory-core request.

## Best Possible Solution

Implement this in the bundled `memory-core` plugin as an additive, backward-compatible config option with default `MEMORY.md`. The target path should be workspace-relative, reject absolute/path-escaping values, optionally support a small documented date-token set, create parent directories and a sensible header when missing, update `applyShortTermPromotions`, CLI/dreaming reports, event payloads, docs, and tests, and make search/indexing behavior explicit for targets outside the current `MEMORY.md` plus `memory/**/*.md` defaults.
