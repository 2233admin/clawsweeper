## Summary

Keep open. Current main still hardcodes short-term/deep memory promotions to `MEMORY.md`, exposes no `promotionTarget` or equivalent config/CLI field, and documents manual plus managed promotion as `MEMORY.md`-only. The related open memory-core PRs address contamination and size-budget symptoms, but neither implements the requested configurable target-file behavior.

## What This Changes

Request a backward-compatible `memory-core` setting that lets short-term promotion writes target a configurable or date-templated Markdown file while keeping promoted content searchable.

## Best Possible Solution

Implement the target resolver inside bundled `memory-core`, shared by manual `memory promote --apply` and managed deep dreaming. The end state should default to `MEMORY.md`, allow deterministic date-token expansion, create missing Markdown targets with the expected header behavior, reject unsafe paths, and automatically include the resolved target in memory indexing/search.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `applyShortTermPromotions()` always sets the destination to `workspaceDir/MEMORY.md`, and both manual CLI and managed dreaming call it without a target override. I did not run a live write because this review is read-only.

## Solution Assessment

Yes. A memory-core-owned config path is the narrowest maintainable solution because the promotion writer owns the destination; adding `memorySearch.extraPaths` alone would not change where promotions are written.
