## Summary

Keep open. Current main still lacks the terminal one-shot startup guard, so persisted pending one-shot ACP metadata with stable ACPX ids can still increment `checked` and produce the reported gateway warning; closed unmerged PR #72023 and adjacent PR #46537 do not resolve the historical metadata path on main.

## What This Changes

The issue asks startup ACP identity reconciliation to skip or quietly handle terminal one-shot or closed ACP session records whose identity remains pending so gateway startup does not warn as if ACP routing is broken.

## Best Possible Solution

Add a narrow guard inside `AcpSessionManager.reconcilePendingSessionIdentities` before `checked` increments so terminal one-shot ACP records are skipped without touching normal persistent pending-identity reconciliation. Add focused regression coverage for pending one-shot records with stable ACPX ids, and only add historical closed-state compatibility if current stored data can still contain that raw shape.

## Reproduction Assessment

Yes. A persisted ACP record with `acp.mode: "oneshot"`, `identity.state: "pending"`, and `identity.acpxSessionId` satisfies the current pending-plus-stable predicate and reaches the warning-producing `checked` path; this can be reproduced with a manager unit fixture without live ACPX.

## Solution Assessment

Yes. The issue's startup skip direction is the narrowest maintainable fix for historical terminal one-shot metadata; resolving future ensure identities or post-turn cleanup helps new sessions but does not prevent old pending one-shot records from being retried on every gateway restart.
