## Summary

Keep open. Current main has a configurable config reload debounce, but it still defaults to 300 ms, only watches the active config path in the gateway reloader, drops the watcher event path before scheduling, and does not provide plugin-runtime hot reload or broader scoped reload for non-config files. The related open items cover adjacent pieces, but none supersedes the combined debounce, attribution, and reload-scope request here.

## What This Changes

The issue requests coalescing rapid watcher events into fewer reloads, scoped reload behavior for non-config watched files, and log attribution for the file that triggered a reload.

## Best Possible Solution

Keep the issue open and scope it into a maintainable reload plan: add explicit trigger-source aggregation/logging and a documented batching policy for gateway config reloads, then route plugin-runtime and account-scoped reload granularity through the existing canonical hot-reload/channel-reload discussions unless maintainers intentionally fold them into this item.

## Reproduction Assessment

Yes for the config-reload debounce portion: source inspection shows events more than the current 300 ms debounce apart will schedule separate reload runs, matching the report’s multi-save log pattern. The non-config scoped reload portion is less directly reproducible from this issue because the gateway reloader currently watches only the active config path and plugin runtime hot reload is documented as unsupported.

## Solution Assessment

No current implementation is not the best final solution for the reported operator problem. The narrowest maintainable path is likely to preserve the existing config reload planner, add source-aware coalescing/logging, and split broader plugin/channel scoped reload work into the dedicated hot-reload surfaces already tracked elsewhere.
