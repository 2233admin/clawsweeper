## Summary

Keep open. Current main still does not satisfy the reported operator problem: config reload has only a short default debounce, watcher events lose their source path, plugin config still routes to restart-required behavior, and channel reload planning still restarts whole channel ids. The adjacent open issues cover related plugin/channel/include-file pieces but do not supersede this issue's combined debounce, attribution, and reload-scope request.

## What This Changes

The issue requests coalescing rapid watched-file changes into fewer gateway reloads, logging the triggering file, and narrowing reload scope for non-config or subsystem-specific changes.

## Best Possible Solution

Keep this open and scope it into a maintainable reload plan: add source-aware coalescing and trigger logging for gateway config reloads, choose an intentional default/config policy for batching, and handle plugin runtime plus channel/account reload granularity through the existing related design threads rather than burying those decisions in a debounce-only patch.

## Reproduction Assessment

Yes for the config-reload portion: source inspection shows watcher events more than the current 300 ms default debounce apart can schedule separate reload runs, and the trigger file is discarded before logging. The non-config scoped-reload portion is only partially reproducible from this report because current main watches the active config path for config reload, while plugin runtime hot reload is documented as unsupported.

## Solution Assessment

No, current main is not the best final solution for the reported issue. The safer path is a narrow source-aware config reload coalescing/logging change plus maintainer-scoped follow-up for plugin and channel reload granularity.
