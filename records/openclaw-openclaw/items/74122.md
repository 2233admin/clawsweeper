## Summary

Keep open. Current main has a configurable config-reload debounce, but the default is still 300 ms, watcher callbacks still discard the file event path, and the broader scoped plugin/channel reload behavior is only partially addressed by adjacent open items rather than implemented here.

## What This Changes

The issue requests coalescing rapid watched-file changes into fewer gateway reloads, source-aware reload logging, and narrower reload scope for non-config watched files.

## Best Possible Solution

Split the remaining work into maintainable pieces: add source-aware config reload coalescing and trigger logging in the gateway reloader, then handle plugin runtime hot reload and targeted channel reload granularity through the adjacent open items rather than bundling those product decisions into a debounce-only patch.

## Reproduction Assessment

Yes for the config-reload portion: source inspection shows watcher events outside the current 300 ms debounce can schedule separate reload evaluations, and the event path is not retained. The broader non-config scoped reload portion is only partially reproducible from this code path because the gateway config reloader watches the active config path, while other mentioned files belong to separate watcher surfaces.

## Solution Assessment

No. Current main is not the best final solution because it provides only short config debounce and diff-path logging; the safer direction is source-aware coalescing/logging plus explicit scoping decisions for plugin and channel reload behavior.
