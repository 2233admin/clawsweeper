## Summary

Current main still has the core gap described by this issue. Gateway clients already send structured connect metadata that distinguishes Control UI/browser and macOS, and the WebSocket server stores it, but the agent runtime contract still only renders channel/capability information in the Runtime line. The chat path maps client id/displayName only into sender metadata, which is explicitly untrusted prompt context rather than a stable runtime field agents should branch on.

## Best Possible Solution

Keep this issue open and implement a narrow trusted Gateway client-context path in core: sanitize selected fields from client.connect.client, carry them through chat.send and queued/follow-up run params into RuntimeInfoInput, render deterministic Runtime fields such as client=openclaw-macos, client_mode=ui, and client_platform=macos/web, and add prompt/Gateway chat tests plus WebChat/Control UI docs for the privacy-safe contract.

## Risks / Open Questions

- A fix should expose only sanitized, low-entropy fields such as client id, mode, platform, and maybe device family; raw userAgent, instanceId, and modelIdentifier need an explicit privacy decision before prompt exposure.
- Runtime-line changes alter prompt bytes and prompt-cache behavior, so field order and omission rules should be deterministic and covered by prompt tests.
- The current untrusted sender metadata may be useful for debugging, but it should not become the documented branching contract for agents.
