## Summary

Keep open. Current main does not implement this PR's intended Feishu workspace artifact SDK exports: there is no `openclaw/plugin-sdk/feishu` package export, no `feishu` SDK entrypoint, and `plugin-sdk/temp-path` only re-exports existing temp-download helpers. The PR also should not be merged unchanged against current main because the SDK boundary has moved toward generic subpaths and explicit plugin-owned classification, while the PR adds a channel-branded subpath and has security-review concerns around lexical workspace containment.

## Best Possible Solution

Keep this PR open for maintainer review, but do not merge the branch unchanged. If the product need remains, replace it with a fresh plugin-neutral workspace artifact helper on a generic SDK subpath such as `plugin-sdk/temp-path` or `plugin-sdk/file-access-runtime`, use safe root-scoped write/copy behavior, keep the existing subpath guardrails intact, and update SDK docs/entrypoint metadata/tests together.

## Risks / Open Questions

- The PR as written adds a channel-branded `openclaw/plugin-sdk/feishu` public subpath, which conflicts with the current documented SDK direction unless maintainers intentionally classify and document it as a public plugin-owned surface.
- The PR's workspace containment check is lexical and does not appear to handle symlink escape safely; a revised implementation should build on `fs-safe`/pinned write semantics or otherwise canonicalize the boundary before writes.
- The PR removed the `discord` subpath exclusion guard in its test diff, matching the existing bot-review concern that it weakens unrelated SDK surface protection.
- The PR's filename/path sanitizer needs another pass for Windows-invalid characters and for making `workspacePath` nonoptional if the helper contract guarantees it.
