## Risks / Open Questions

- The proposed command is destructive: it mutates session stores and removes transcript files, so the unresolved correctness comments should be addressed before merge.
- The PR implementation routes deletion through `updateSessionStore`; current `saveSessionStoreUnlocked` can run maintenance during writes, which could make real execution remove unrelated sessions while dry-run/output only reports predicate matches.
- The PR removes transcript files directly and reports paths as removed even when `fs.promises.rm` fails, which can mislead users about cleanup success.
- The diff does not add supply-chain or CI execution risk, but local filesystem deletion is a safety-sensitive behavior that needs careful review and tests.
