## Summary

Keep open. Current main still does not implement xAI as a `/voiceclaw/realtime` provider: the gateway type/factory path is Gemini-only, the gateway docs document Gemini Live, and the xAI docs in this repo explicitly mark realtime voice as not exposed. The issue is also paired with open draft PR #73032 using closing language, so cleanup should wait for that PR to be reviewed, merged, or closed.

## What This Changes

Request xAI Realtime Voice Agent support for `/voiceclaw/realtime`, preferably through a shared OpenAI-Realtime-compatible adapter that could later serve OpenAI Realtime too.

## Best Possible Solution

Review the paired draft PR as the canonical implementation path. If accepted, the end state should add provider-aware `/voiceclaw/realtime` dispatch, xAI adapter coverage, operator docs for `XAI_API_KEY`, and tests for audio passthrough, transcripts, tools, reconnect/rotation, and secret redaction while preserving existing Gemini behavior.

## Reproduction Assessment

Not applicable as a feature request, but the missing capability is reproducible by static inspection. Current main accepts only `openai` or `gemini` in the session config type, forces the runtime provider to Gemini, requires `GEMINI_API_KEY`, and documents xAI realtime voice as not exposed.

## Solution Assessment

Unclear. The shared adapter direction is technically plausible because xAI documents OpenAI Realtime compatibility and event-name differences, but the repo has both the core VoiceClaw gateway adapter and the plugin realtime-voice provider surface, so the best surface and scope need maintainer review through #73032. citeturn0search0
