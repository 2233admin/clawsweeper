## Summary

Current main still does not show activity from the first session-matching system-injected run event. The TUI adopts runs from matching chat events, but agent lifecycle/tool events are still rejected unless their run id is already active, known from a prior chat event, or finalized. Gateway-side payloads already carry session-scoped signals, and related PR #66400 remains the open implementation candidate rather than proof that main is fixed.

## Best Possible Solution

Keep this issue open and review/land PR #66400 or implement the equivalent narrow TUI fix: add `sessionKey` to TUI agent events, adopt session-matching lifecycle/tool events before the known-run guard, set `activeChatRunId` and running status on lifecycle start, and add regression coverage for externally started runs before the first chat delta without leaking cross-session events or disturbing /btw behavior.
