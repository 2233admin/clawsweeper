## Summary

Current main still does not adopt a system-injected run from its first session-matching agent lifecycle event. The TUI only records a run after a matching chat event or an already-known active/finalized run, so lifecycle start events for externally started work can still be dropped before they set the status to running. Related PR #66400 remains the relevant unmerged implementation candidate rather than proof that main is fixed.

## Best Possible Solution

Keep this issue open and review/land PR #66400 or implement the equivalent targeted TUI fix: include `sessionKey` on TUI agent events, adopt session-matching lifecycle/tool events before the known-run guard, set `activeChatRunId` and running status on lifecycle start, and add regression coverage for externally started runs before the first chat delta while preserving cross-session and /btw behavior.
