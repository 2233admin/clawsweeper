## Summary

Keep open: current main does not implement MCP trace-context propagation, and this active implementation PR still has concrete correctness and telemetry trust findings that should be revised before merge.

## What This Changes

The PR adds an opt-in `diagnostics.otel.mcp` config surface, MCP JSON-RPC diagnostic events, bundled MCP `traceparent` injection, OTEL MCP span export, generated config schema/help updates, and targeted tests.

## Best Possible Solution

Keep this PR as the implementation candidate, but require revisions before merge: either implement full `traceparent`/`tracestate` behavior or narrow the public text to traceparent-only, classify JSON-RPC `error` responses as MCP errors, make trusted-event handling symmetric, document the MCP canary and its trust model in the OpenTelemetry docs, and validate the revised path with targeted tests plus the changed gate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Propagate tracestate with the canary:** `src/gateway/mcp-http.trace.ts:64-65`
  - body: The PR body and config/help text promise `traceparent`/`tracestate` propagation via `params._meta`, but inbound extraction reads only `_meta.traceparent` and the bundled outbound request only writes `traceparent`. With the canary enabled, vendor state and sampling hints in `tracestate` are silently dropped, so either implement `tracestate` handling or narrow the public surface to traceparent-only.
  - confidence: 0.88
- **[P2] Classify JSON-RPC error responses as MCP errors:** `src/gateway/mcp-http.trace.ts:141-142`
  - body: `completeMcpTraceScope` only treats `response.result.isError === true` as an error. JSON-RPC error objects returned by `jsonRpcError` for unsupported methods go through the completed path, so failed MCP requests export `mcp.request.completed` instead of `mcp.request.error`.
  - confidence: 0.9
- **[P2] Require trusted metadata for completed and error spans:** `extensions/diagnostics-otel/src/service.ts:2100-2118`
  - body: `recordMcpRequestStarted` rejects untrusted diagnostic events, but the completed/error handlers only check `tracesEnabled`. If an untrusted MCP diagnostic event is emitted, the OTEL plugin can still create a span even though no trusted start span was accepted; mirror the trusted guard across these handlers.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds no dependency, CI, or secret-handling changes, but the new telemetry trust boundary needs attention before merge.

Concerns:

- **[medium] Remote MCP trace context becomes exported parent context:** `src/gateway/mcp-http.trace.ts:64`
  - body: When enabled, the canary parses caller-supplied `params._meta.traceparent` and uses it as parent context for exported spans. That is the intended feature, but it allows trace-correlation spoofing into an operator collector unless the opt-in risk and trust model are documented and reviewed.
  - confidence: 0.78
- **[low] Untrusted MCP events can create OTEL spans:** `extensions/diagnostics-otel/src/service.ts:2100`
  - body: The completed/error span handlers lack the `metadata.trusted` guard used by the started handler, so an untrusted MCP diagnostic event could produce exported MCP spans if such an event path is introduced.
  - confidence: 0.82
