## Summary

Keep open. Current main has generic trusted diagnostic traceparent support for provider/model-call paths, but it does not implement this PR’s opt-in `diagnostics.otel.mcp` canary for MCP JSON-RPC `_meta.traceparent` propagation or MCP request spans. This PR is a draft, unmerged implementation candidate and should be reviewed on its merits rather than auto-closed.

## Best Possible Solution

Review this draft as the implementation candidate for MCP trace-context propagation. The likely landing path is to keep the feature opt-in under `diagnostics.otel.mcp`, resolve the `traceparent` versus `tracestate` scope mismatch, verify the trust/privacy behavior with targeted tests plus an OTLP collector smoke when feasible, add user-facing OpenTelemetry docs if accepted, and then merge or request changes. Do not close as implemented until this behavior lands on main.

## Risks / Open Questions

- Security/privacy-sensitive behavior needs maintainer review before merge because the PR accepts caller-provided MCP `params._meta.traceparent` as remote parent context and emits request/session/tool metadata into diagnostics. The diff appears opt-in/default-off, but the trust boundary should be confirmed.
- The PR body and config help mention `tracestate`, but the patch search shows implementation and tests for `traceparent` only. Maintainers should either add `tracestate` behavior or narrow the wording.
- The author notes that local Jaeger/OTLP trace-chain E2E was not run; this canary should get collector-chain proof or an explicit maintainer decision that targeted unit coverage is sufficient.
- The PR adds new public config keys but current `docs/gateway/opentelemetry.md` does not document `diagnostics.otel.mcp`; docs should be added if the canary is accepted.
