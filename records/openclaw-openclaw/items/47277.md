## Summary

Keep this PR open. Current main still uses shared memory discovery, shared memory read access, and agent-wide memory search managers; it does not implement the user-specific memory isolation described by the PR. The PR remains a plausible but security-sensitive implementation candidate, with unresolved review concerns around identity derivation, path scoping, QMD parity, config schema/docs, and manager lifecycle.

## Best Possible Solution

Keep the PR open for maintainer review and rework. The best path is a revised memory-core and memory-host-sdk implementation that defines a safe user isolation key, threads trusted requester identity through memory_search and memory_get, scopes builtin and QMD indexing/read paths consistently, updates config schema and public docs, handles shared-root compatibility deliberately, and adds regression tests for unsafe paths, multi-channel IDs, disabled isolation, QMD fallback, flush behavior, and manager lifecycle.

## Risks / Open Questions

- This is a privacy boundary change: partial isolation could leak memory_search or memory_get results across users if builtin indexing, QMD indexing/fallback, read-file checks, and flush targets are not scoped consistently.
- The PR derives filesystem paths from session or sender identity, so it needs safe encoding and regression tests for separators, dot segments, case-insensitive filesystems, colon-containing IDs, missing sender IDs, and channel/account scoping.
- Default-on isolation changes config and behavior contracts; schema, docs, compatibility, migration, and disabled-isolation behavior need maintainer review before merge.
- Per-user manager cache keys can create unbounded long-lived SQLite connections, watchers, or sync loops unless lifecycle/eviction is designed deliberately.
