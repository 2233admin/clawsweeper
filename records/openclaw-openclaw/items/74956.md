## Summary

Keep open. Current main still has the generic internal `message:received` hook context gap: the dispatch path has already resolved `sessionAgentId` and `workspaceDir`, but the emitted internal hook context and docs omit both fields. The reported WAL artifact is not present in the source tree, but the core hook-contract problem is source-backed and narrow.

## What This Changes

The issue requests that `message:received` hook events expose enough per-agent routing context, such as `workspaceDir` and/or `agentId`, so hooks can write to the correct agent workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal `message:received` hook contract, populate them from the already-resolved dispatch route, and update docs/types/tests without adding WAL- or active-memory-specific policy to core.

## Reproduction Assessment

Yes for the root cause: current-main code inspection shows the dispatch path resolves the agent id and workspace before emitting `message:received`, but the emitted context omits them. The downstream WAL write symptom remains plausible but not directly reproducible from this checkout because the named dist hook artifact is absent.

## Solution Assessment

Yes, the narrow maintainable fix is an additive generic hook API change. Reconstructing the WAL hook or adding active-memory-specific behavior in core would be a less clean boundary.
