## Summary

Keep open. Current main still exposes the internal `message:received` hook context without `agentId` or `workspaceDir`, while the shared inbound dispatch path has already resolved both values before emitting the hook. The exact reported `wal-protocol` dist artifact is not present in this source checkout, but the generic hook-contract gap is source-backed and narrow enough for a focused fix.

## What This Changes

The issue requests adding per-agent routing data such as `workspaceDir` and/or `agentId` to `message:received` hook context so hooks can write state into the correct agent workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the generic internal `message:received` hook context, populate them from the already-resolved dispatch route, update docs/types/tests, and keep WAL- or active-memory-specific policy out of core.

Yes for the root cause: current-main code inspection shows the inbound dispatch path resolves `sessionAgentId` and `workspaceDir` before emitting `message:received`, but the emitted context omits both. ## Solution Assessment

Yes. The narrow maintainable fix is an additive hook-contract extension on the existing internal `message:received` event, with tests proving the resolved agent/workspace fields are emitted; reconstructing WAL behavior in core would be the wrong boundary.
