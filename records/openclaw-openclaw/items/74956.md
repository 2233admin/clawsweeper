## Summary

Keep open. Current main still exposes the internal `message:received` hook without `agentId` or `workspaceDir`, even though the inbound dispatch path has already resolved both values before emitting the event. The named WAL dist artifact is not present in the source tree, but the underlying hook-context gap is real, generic, and narrow enough for a focused fix.

## What This Changes

The issue requests that `message:received` hook context expose per-agent routing data such as `workspaceDir` or `agentId` so WAL-style hooks can write to the correct agent workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal `message:received` hook contract, populate them from the already-resolved dispatch routing state, update docs and tests, and keep the change generic rather than adding WAL- or active-memory-specific behavior to core.

## Reproduction Assessment

Yes for the root cause: current-main source inspection shows `dispatchReplyFromConfig` resolves `sessionAgentId` and `workspaceDir` before emitting `message:received`, while the emitted context omits both. The downstream WAL write symptom is plausible but not directly reproducible from this checkout because the named dist artifact is absent.

## Solution Assessment

Yes. The narrow maintainable fix is an additive hook-contract update plus emitter/test/docs coverage; reconstructing WAL behavior or adding active-memory-specific policy in the shared dispatcher would be broader than needed.
