## Summary

Keep open. Current main still exposes `message:received` internal hook context without `agentId` or `workspaceDir`, even though the shared inbound dispatch path has already resolved both values before emitting the hook. The exact `wal-protocol` artifact named in the report is not present in the source checkout, but the reported hook context gap is source-backed and affects non-main agent hook authors.

## What This Changes

The issue requests per-agent routing fields on the internal `message:received` hook context so hooks can write state to the correct agent workspace.

## Best Possible Solution

Add additive `agentId` and `workspaceDir` fields to the internal `message:received` context across emitters that have enough routing data, update docs/types/tests, and keep the fix generic rather than adding active-memory or WAL-specific policy to core.

## Reproduction Assessment

Yes for the root cause: inspect or exercise the current inbound dispatch hook path and the `message:received` event context lacks `agentId` and `workspaceDir` while `dispatchReplyFromConfig` has already resolved them. The exact WAL file-write symptom is not directly reproducible from this source checkout because the named `dist` hook artifact is absent.

## Solution Assessment

Yes, enriching the internal hook event context is the narrow maintainable fix because it preserves the existing hook model and gives hook authors the same per-agent routing data command hooks already receive. The safer boundary is to add only the needed per-agent fields, not a hook-specific migration or active-memory special case.
