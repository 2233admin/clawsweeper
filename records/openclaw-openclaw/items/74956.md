## Summary

Keep open. Current main still omits `agentId` and `workspaceDir` from the internal `message:received` hook context even though the shared dispatcher has already resolved both values before emitting that hook; the reported WAL artifact is absent from this source tree, but the generic hook-contract gap is current and source-backed.

## What This Changes

The issue requests that internal `message:received` hook events expose per-agent routing context, such as `workspaceDir` and/or `agentId`, so per-agent state hooks can write to the correct workspace.

## Best Possible Solution

Expose optional `agentId` and `workspaceDir` on the internal `message:received` context, populate them from already-resolved routing/session data at each emitter, update the hooks docs and focused tests, and keep WAL-specific policy outside core.

## Reproduction Assessment

Yes for the root cause: current-main source and docs show `message:received` lacks `agentId`/`workspaceDir` while the shared dispatcher has already resolved both. The exact WAL file-write symptom is not directly reproducible from this checkout because the reported hook artifact is absent.

## Solution Assessment

Yes, an additive hook-contract enrichment is the narrowest maintainable fix. Reconstructing WAL state placement in core would cross the wrong boundary; core should provide generic per-agent context to hooks.
