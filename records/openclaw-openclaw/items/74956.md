## Summary

Keep open. Current main still omits `agentId` and `workspaceDir` from the internal `message:received` context even though the shared dispatch path has already resolved both before emitting the hook; the prior ClawSweeper comment remains accurate against `8093ae60292f4d9009b3d48f234b39b5cbc538e5`.

## What This Changes

The issue reports that `message:received` hooks cannot identify the routed agent workspace, causing per-agent WAL/session-state data to be written to the main workspace instead of the active agent workspace.

## Best Possible Solution

Add optional `agentId` and `workspaceDir` to the internal `message:received` hook context, populate them from already-resolved session or route data at each emitter, document the additive fields, and keep WAL-specific policy outside core.

## Reproduction Assessment

Yes for the root cause: current source and docs show `message:received` lacks `agentId` and `workspaceDir` while `dispatchReplyFromConfig` already has both values. The exact WAL artifact symptom is plausible but not directly reproducible from this checkout because the reported dist hook file is absent.

## Solution Assessment

Yes, an additive hook-contract enrichment is the narrowest maintainable fix. Rebuilding WAL behavior in core would cross the wrong boundary; core should expose generic routed-agent context to hooks.
