## Summary

Keep open. Current main still has the reported internal hook-contract gap: `message:received` is emitted after the shared dispatcher has resolved the session agent and workspace, but the context type, mapper, docs, and focused tests still omit `agentId` and `workspaceDir`. The reported WAL artifact is not present in this source checkout, so the downstream write symptom cannot be audited directly, but the generic per-agent hook context bug is source-backed and narrow.

## What This Changes

The issue asks for `message:received` hook context to include agent/workspace routing so per-agent hook state is not written to the default workspace.

## Best Possible Solution

Expose optional `agentId` and `workspaceDir` on the internal `message:received` context, populate them from already-resolved routing/session data at each internal emitter, document the additive contract, and keep WAL/active-memory-specific write policy outside core.

## Reproduction Assessment

Yes for the root cause: current-main source and docs show `message:received` lacks `agentId` and `workspaceDir` while `dispatchReplyFromConfig` has already resolved both before emitting the hook. The exact WAL artifact symptom is plausible but not directly reproducible from this checkout because the named dist hook artifact is absent.

## Solution Assessment

Yes. The narrow maintainable fix is an additive hook-context enrichment with optional routing fields; moving WAL or active-memory policy into core would cross the owner boundary and is not needed to unblock hook authors.
