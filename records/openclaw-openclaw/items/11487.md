## Summary

Current main still does not implement optional human-readable metadata on group/channel config entries. The relevant strict Zod schemas, TypeScript group config types, generated channel config metadata, and public docs remain behavior-only, so `name`, `label`, or `description` would still be rejected for representative group entries. The issue is valid, but it needs maintainer product/API judgment on the canonical field name before an automated fix PR is a good lane.

## Best Possible Solution

Keep this issue open. The best path is a deliberate core config-contract change: choose one optional string metadata field for group/channel entries, add it consistently to relevant TypeScript types and strict Zod schemas, regenerate bundled channel config metadata, preserve the field through config load/write while keeping runtime behavior unchanged, update public group/channel docs, and add validation tests proving the field is accepted but ignored functionally.
