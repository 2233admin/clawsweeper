## Summary

Keep this issue open. Current main still lacks the requested hard token flush threshold, priority-aware compaction strategy, and embedding-cache TTL config keys, while adjacent fixes only cover narrower flush-gating behavior and do not resolve the full durability proposal.

## What This Changes

The issue requests three memory durability configuration additions: a forced hard memory-flush threshold, a priority-preserving compaction strategy, and a TTL for memory-search embedding cache entries.

## Best Possible Solution

Split or design this as a memory durability plan: land the existing narrow flush-threshold repair if accepted, then decide explicit semantics for forced token flush, priority-preserving compaction annotations, and cache eviction/freshness before adding public config keys and docs.

## Reproduction Assessment

Yes for the missing config keys: the current strict schemas still reject the three proposed paths. The reported large-scale memory-loss behavior was not live-reproduced, and the cache-staleness claim needs a focused reproduction because current embeddings are content-hash keyed.

## Solution Assessment

Unclear. The requested direction is plausible, but priority-preserving compaction and forced flush semantics need product design, and TTL may not be the narrowest freshness fix for a content-hash cache.
