## Summary

Keep this PR open. Current main still applies the under-10-character TTS guard to explicit `[[tts:text]]hello[[/tts:text]]` content, while the latest PR diff is a narrow runtime/test fix and the earlier empty-markdown review edge has been addressed.

## What This Changes

The PR scopes speech-core short-text skips to untagged TTS text, adds a post-markdown empty-text guard, and adds regression tests for explicit short tagged text, untagged short text, and markdown-only tagged text.

## Best Possible Solution

Land this PR or an equivalent narrow patch so non-empty explicit `[[tts:text]]...[[/tts:text]]` content bypasses the untagged short-reply noise guard, while untagged short replies and markdown-only explicit tags still skip synthesis. After merge, linked #73758 can close through the normal closing reference.

## Reproduction Assessment

Yes. The current-main code path is enough to reproduce: parse `[[tts:text]]hello[[/tts:text]]`, get `ttsText === "hello"`, then return at the under-10-character guard before `textToSpeech`; the PR adds focused regression coverage for that path.

## Solution Assessment

Yes. This is the narrowest maintainable fix for the reported guard bug: it distinguishes explicit tagged speech text from untagged auto-TTS text and keeps the post-markdown empty guard. A shared contract regression could be added later, but the extension-level runtime tests are sufficient for this small patch.
