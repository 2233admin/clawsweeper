## Summary

Keep open. Current main still matches the reported failure mode: wake requests that arrive while the heartbeat wake handler is running are retained and re-scheduled after the active run, and targeted/session wake requests still bypass the interval not-due guard before calling the heartbeat runner. The prior issue discussion and ClawSweeper comment are consistent with the current source.

## What This Changes

Prevent automatic external heartbeat wake events from causing additional heartbeat runs inside the configured interval when they arrive during a scheduled heartbeat run.

## Best Possible Solution

Implement a targeted heartbeat scheduler/wake-path guard so automatic external wakes queued during an already-running interval heartbeat do not cause a second heartbeat inside the configured interval, while preserving explicitly requested immediate wake behavior. Add regression coverage for an interval run in progress, a queued exec/session wake, no second in-window model call, and the next scheduled interval still firing normally.

## Repair Work Prompt

Fix the duplicate heartbeat run bug reported in openclaw/openclaw#64016. On current main, an interval heartbeat can be running when an automatic external wake such as `exec-event` is queued; after the interval run completes, `heartbeat-wake.ts` drains the queued wake and `heartbeat-runner.ts` targeted/non-interval paths bypass the interval not-due guard, causing a second model call/checkpoint within the configured interval.

Expected boundary: add a narrow scheduler/wake-path guard so automatic external wakes queued during or immediately after an interval heartbeat do not trigger another heartbeat for the same agent/session inside the current interval window. Preserve explicitly documented immediate wake behavior for manual/system `now`, cron `--wake now`/`cron:*`, and other intentionally immediate user actions. Do not broadly disable targeted wake routing, delivery target overrides, or per-agent heartbeat scheduling.

Before editing, check whether open PR #71582 has landed or changed these files; if it has, adapt to the landed code instead of duplicating it. Likely files: `src/infra/heartbeat-runner.ts`, `src/infra/heartbeat-wake.ts`, `src/infra/heartbeat-runner.scheduler.test.ts`, possibly `src/infra/heartbeat-wake.test.ts`, `src/gateway/server-node-events.ts`, and `CHANGELOG.md`.

Add focused Vitest coverage that simulates: heartbeat interval is due; `runOnce` for the interval stays in progress; an `exec-event`/session wake is requested while it is in progress; after the interval run resolves, no second `runOnce` occurs before the configured interval expires; the next scheduled interval still fires. Also keep existing tests for session-scoped exec wake routing and distinct targeted wake coalescing green. Add a user-facing changelog fix entry in the active version with `Thanks @foreverxdord`.
