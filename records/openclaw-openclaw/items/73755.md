## Risks / Open Questions

- The PR depends on Tailscale `funnel status --json` output shape and host-level Serve/Funnel mutation behavior. The parser now has useful unit coverage, but maintainers may still want a live tailnet smoke before landing because a unit test cannot prove Tailscale daemon behavior.
- The linked regression asks about both preserving existing Funnel routes and token-auth Funnel parity. This PR intentionally solves the preserve-existing-Funnel route path and leaves the password-only Funnel policy untouched, so any remaining token-auth Funnel product/security decision should stay tracked separately.
- The PR adds generated config help for `gateway.tailscale.preserveFunnel` but does not update `docs/gateway/tailscale.md`; maintainers should decide whether the public Tailscale gateway docs need an explicit example before merge.
