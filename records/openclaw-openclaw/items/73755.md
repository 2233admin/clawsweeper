## Summary

Keep this PR open. Current main still unconditionally reapplies Tailscale Serve and lacks the new `preserveFunnel` config surface, while the PR is an active, narrow implementation for the linked regression; the remaining work is maintainer review plus rebase/conflict resolution because GitHub reports the branch as dirty and far behind current main.

## What This Changes

The PR adds optional `gateway.tailscale.preserveFunnel` config, wires it into gateway Tailscale startup, checks `tailscale funnel status --json` in serve mode, skips reapplying Serve when an existing Funnel route covers the gateway port, and adds generated config help, tests, and a changelog entry.

## Best Possible Solution

Land a rebased version of this PR after maintainer review, preserving the opt-in behavior and password-only OpenClaw-managed Funnel policy while avoiding broader auth/runtime-config changes.

## Reproduction Assessment

Yes. The high-confidence reproduction is `gateway.tailscale.mode="serve"`, an externally configured Funnel route for the gateway port, then a gateway restart; current main statically shows the restart path calls `tailscale serve --bg --yes <port>` without an existing-Funnel precheck.

## Solution Assessment

Yes. The opt-in `preserveFunnel` path is the narrowest maintainable fix for token-auth users who manage Funnel externally, and it avoids changing the intentionally stricter `mode="funnel"` auth policy; the safer alternative is not a replacement PR, but maintainer rebase/review of this active branch.
