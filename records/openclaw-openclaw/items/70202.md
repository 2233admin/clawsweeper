## Summary

Keep this issue open. Current main supports model allowlists, fallback chains, `/model` switching, and configure-time multi-model selection, but first-run onboarding still routes through the single-default-model picker and only applies the selected primary model. The remaining gap is real and needs maintainer UX/product direction before changing onboarding scope.

## What This Changes

Request first-run onboarding support for selecting multiple usable LLM models while preserving one default model, so users can optimize cost, latency, and capability.

## Best Possible Solution

Add a first-run onboarding step, after provider auth/default selection, that preserves the chosen primary model and lets the user select additional provider-scoped models for the allowlist/fallback chain using the existing model-picker helpers. Keep automatic task-complexity routing as a separate follow-up unless maintainers explicitly expand the scope.

## Reproduction Assessment

Not applicable as a runtime bug report, but there is a high-confidence source-level behavior check: onboarding calls promptDefaultModel/applyPrimaryModel, while configure uses promptModelAllowlist plus fallback/allowlist writers.

## Solution Assessment

Unclear until maintainer product direction is set. The narrowest maintainable implementation appears to be reusing the existing allowlist/fallback helpers in src/wizard/setup.ts with focused onboarding tests and docs, not designing automatic routing in this issue.
