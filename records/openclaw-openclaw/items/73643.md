## Summary

Keep open. Current main still replays or defers missed cron work after Gateway restart and has no public global, per-job, or freshness-window policy to skip stale catch-up runs; related #42039 overlaps but remains narrower than this request.

## What This Changes

The issue requests a global, per-cron, or freshness-window option to disable stale missed cron job catch-up after Gateway restart.

## Best Possible Solution

Keep the default replay/defer behavior for compatibility, then add an explicit cron-owned missed-run policy that can skip stale catch-up safely. A complete implementation should update config, protocol or CLI surfaces if per-job policy is chosen, docs, generated schema/help metadata, changelog, and restart catch-up tests while coordinating with #42039.

## Reproduction Assessment

Yes. Source and existing tests give a high-confidence reproduction: persist a recurring cron job with a past nextRunAtMs or a missed previous cron slot, start CronService, and current main enqueues/replays it or reschedules it instead of skipping stale runs.

## Solution Assessment

Unclear. The issue identifies a valid user problem, but the best API is not settled; a backward-compatible explicit skip/freshness policy is safer than changing startup defaults or treating exposed stagger/limit knobs alone as a complete fix.
