## Summary

Keep open. Current main still replays or defers missed cron jobs on Gateway startup and does not expose a global, per-job, or freshness-window policy to skip stale catch-up runs. Related #42039 overlaps on exposing internal stagger/limit knobs, but it does not fully settle the requested skip/window semantics.

## What This Changes

The issue requests a global or per-cron option to skip, run, stagger, or time-limit missed cron job catch-up after Gateway restart.

## Best Possible Solution

Keep default replay behavior for compatibility, then add an explicit cron-owned missed-run policy that can skip stale catch-up safely. The design should coordinate with #42039 and cover config/schema, protocol, CLI, docs, and restart catch-up tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists from current source and tests: persist a recurring cron job whose latest slot was missed or whose nextRunAtMs is overdue, start CronService, and current main enqueues/replays or defers it instead of skipping stale runs.

## Solution Assessment

Unclear. The report identifies the right product problem, but the best maintainable fix needs maintainer API direction before choosing global config, per-job policy, or freshness-window semantics.
