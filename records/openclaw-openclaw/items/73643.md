## Summary

Keep open. Current main still has restart catch-up as replay-or-defer behavior: Gateway startup calls `runMissedJobs`, the planner selects overdue jobs, and deferred jobs are rescheduled rather than skipped; no global, per-job, or freshness-window policy is exposed in config, protocol, CLI, or docs.

## What This Changes

Request an option to disable or constrain stale missed cron catch-up after Gateway restart, either globally, per cron job, or by a freshness window.

## Best Possible Solution

Keep the current replay behavior as the compatibility default, then add an explicit cron-owned missed-run policy that can skip stale catch-up safely. The end state should align config types/schema, gateway protocol, CLI flags, docs, and restart catch-up tests, and should coordinate with #42039 so exposed stagger/limit knobs do not masquerade as a stale-run skip policy.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists from current source and tests: persist a recurring cron job with a past `nextRunAtMs`, start `CronService`, and current main enqueues/replays it immediately or reschedules deferred overflow rather than skipping stale work.

## Solution Assessment

Unclear. The issue identifies a valid user problem, but the best maintainable fix needs product/API direction; exposing `maxMissedJobsPerRestart=0` alone is not enough because current code treats overflow as deferred future runs, not skipped missed runs.
