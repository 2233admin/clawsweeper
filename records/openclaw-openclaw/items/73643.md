## Summary

Current main still does not provide a user-facing way to disable or time-bound cron missed-job catch-up after gateway restart. The request is actionable core cron scheduler/config work: startup still calls `runMissedJobs`, tests assert overdue jobs replay on restart, and the public config/schema/docs expose cron retention, run-log, concurrency, retry, and delivery settings but no missed-run policy.

## Best Possible Solution

Keep this issue open and implement an explicit missed-run policy in the cron owner surface. A conservative path is a backward-compatible global default under `cron` plus optional per-job override, wired through `CronConfig`, strict zod/schema help, gateway protocol/CLI/edit surfaces, docs, and `runMissedJobs` planning. Tests should cover default replay, global skip, per-job skip/run override, and optional max-age/window behavior for market-hours style jobs.

## Risks / Open Questions

- Restart catch-up was added and hardened for real recovery cases, so an implementation should preserve the current default and cover skip/window behavior with targeted restart tests.
- The existing `maxMissedJobsPerRestart` and `missedJobStaggerMs` fields are internal/test dependency knobs, not public config or per-job policy, so they do not satisfy the user request.
