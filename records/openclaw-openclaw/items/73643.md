## Summary

Keep open. Current main still lacks a global, per-job, or freshness-window policy to skip stale missed cron runs after Gateway restart; it only replays immediately or defers replay. Related #42039 overlaps but does not fully cover the requested skip/window semantics.

## What This Changes

The issue requests a global, per-cron, or freshness-window option to disable missed cron job catch-up after Gateway restart.

## Best Possible Solution

Keep default replay behavior for compatibility, then add an explicit cron-owned missed-run policy that can skip stale catch-up safely. The implementation should update config, protocol, CLI, docs, and restart catch-up tests, and should coordinate with #42039 rather than treating exposed stagger/limit knobs as a full substitute.

## Reproduction Assessment

Yes. A high-confidence reproduction exists from current tests and source: persist a recurring cron job with a past nextRunAtMs or a missed previous cron slot, start CronService, and current main enqueues/replays it or reschedules it instead of skipping stale runs.

## Solution Assessment

Unclear. The issue identifies the right user problem, but the best public API is not settled; a backward-compatible explicit skip/freshness policy is safer than changing startup defaults or relying on maxMissedJobsPerRestart=0, which currently defers rather than skips.
