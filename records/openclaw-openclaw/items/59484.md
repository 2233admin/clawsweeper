## Summary

Keep this PR open, but not merge-ready. The underlying behavior is real on current main, but the patch rethrows every `--probe` gateway failure, which also bypasses the existing redacted config-only fallback for gateway transport failures.

## What This Changes

The PR adds an early rethrow for `channels status --probe` `callGateway` failures and a focused command-flow regression test for a probe failure.

## Best Possible Solution

Keep the PR open and revise the catch path so probe-mode only bypasses config fallback for errors that occur after the gateway request is accepted or answered, while gateway transport errors continue through the existing redacted config-only fallback. Add regression coverage for both a true probe failure and a `--probe` unreachable-gateway/redaction fallback case.

## Reproduction Assessment

Yes. A focused unit reproduction is available by mocking `callGateway` to reject under `probe: true`: a generic after-gateway probe error should be rethrown, while a `GatewayTransportError` or URL-bearing timeout should remain redacted and use config-only fallback. I did not run tests because this review was explicitly read-only.

## Solution Assessment

No. Rethrowing all probe-mode failures is broader than the stated fix and skips existing error redaction; a narrower error-classification guard, likely based on gateway transport errors versus gateway request/probe errors, is the safer maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve redaction before rethrowing probe errors:** `src/commands/channels/status.ts:309-311`
  - body: This early rethrow bypasses `formatChannelsStatusError`, so `openclaw channels status --probe` will hand raw gateway timeout/close errors to the CLI wrapper, which prints `String(err)`. Those errors can include URL credentials or token query parameters from gateway connection details; keep transport errors in the sanitized path or rethrow only a redacted error.
  - confidence: 0.89
- **[P2] Keep unreachable gateways on the config fallback path:** `src/commands/channels/status.ts:309-311`
  - body: The guard fires for every probe-mode `callGateway` rejection, including a gateway that is not reachable. That removes the documented config-only fallback for `channels status --probe` even though the PR only intends to preserve failures after the gateway path is reached.
  - confidence: 0.86
