## Summary

Keep open. Current main still exposes memory as separate backend, QMD, memory-search, plugin-slot, and compaction-flush controls, and it does not implement the strict unified `memory.type` contract requested in the discussion. Related memory CRUD work in #70049 and draft PR #70540 is useful context, but that only covers a daily Markdown block and `memory_add` slice, not the requested mode semantics, disabled-memory behavior, QMD-only behavior, fallback policy, or workspace-scoped SQLite storage.

## What This Changes

The issue asks for a strict unified memory management contract so built-in SQLite, QMD Markdown daily memory, disabled memory, mixed behavior, and workspace-scoped storage behave consistently across installs.

## Best Possible Solution

Define one coherent memory-mode contract that covers durable writes, recall backend selection, QMD fallback policy, disabled memory, workspace and agent storage scope, migration for existing Markdown and SQLite artifacts, schema/docs, and compatibility for current `memory.backend` users.

## Reproduction Assessment

No high-confidence end-user runtime reproduction is available because the report does not include the three users' configs, logs, or exact memory plugin state. The missing contract itself is reproducible by static inspection: current main has no `memory.type` and still splits backend, QMD, memory search, plugin-slot, and compaction-flush controls.

## Solution Assessment

No, the current scattered configuration is not the requested solution. A small alias for `memory.type` would also be unsafe because the real fix needs maintainer decisions about public semantics, migration, fallback behavior, storage scope, and docs.
