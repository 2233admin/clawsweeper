## Summary

Keep open. Current main still exposes separate controls for builtin SQLite recall, QMD recall, compaction-time Markdown writes, QMD fallback, and memory search enablement; it does not provide the strict memory-mode contract requested in the issue. The linked memory_add work is relevant, but it is an open draft first slice and does not settle builtin-only, qmd-only, mixed, disabled-memory, or workspace-scoped SQLite semantics.

## What This Changes

The issue requests a strict, consistent memory management contract so users do not diverge between builtin SQLite memory, QMD daily Markdown memory, mixed behavior, or no memory.

## Best Possible Solution

Treat this as a focused memory product/design task. The end state should define one coherent memory-mode contract or documented successor covering durable writes, recall backend, QMD fallback, disabled memory, workspace and agent storage scope, migration for existing Markdown and SQLite artifacts, and clear operator guidance in config docs.
