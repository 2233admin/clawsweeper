## Summary

Keep open. Current main still does not contain a targeted fix for the reported MiniMax-M2.5 plus Feishu loop: `sessions_list` can still return an empty list without explaining the caller's current session, loop detection remains opt-in, and the Feishu/message send path still lacks a narrow per-turn duplicate-send guard for repeated identical outbound sends. The KeepMyClaw comment is useful recovery context after a loop, but it does not prevent the duplicate sends in OpenClaw itself.

## Best Possible Solution

Keep this issue open and fix it with a narrow regression-driven change: make current-session state unambiguous when `sessions_list` returns no rows, steer self-session checks toward `session_status`, and add a per-turn/run duplicate outbound-message guard for identical `message(action="send")` calls to the same Feishu target. Add tests for the empty `sessions_list` plus repeated identical send sequence, and avoid broad default changes to tool-loop detection.
