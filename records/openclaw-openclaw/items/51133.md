## Summary

Keep open. Current main still does not contain a hard fix for the reported Feishu/MiniMax loop: `sessions_list` can still return an empty filtered list without explaining the current session, tool-loop protection remains opt-in, and Feishu direct outbound sends still do not have a per-turn duplicate-send guard for repeated model `message(action=send)` calls.

## Best Possible Solution

Keep this issue open. Make current-session state unambiguous to the model, preferably by steering self-session checks to `session_status` or by making `sessions_list` explicitly include/annotate the caller's current-session state when the list is empty. Add a regression test for a Feishu/MiniMax-style sequence where `sessions_list` returns zero and the model attempts repeated identical `message` sends, plus a narrow duplicate outbound-message guard for identical sends to the same target within one turn.
