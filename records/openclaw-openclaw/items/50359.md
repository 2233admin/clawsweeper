## Summary

Current main has newer structured media extraction and a pending-tool-media queue, but it still does not implement this PR’s specific fallback from embedded `read` image results to the original tool start args. In fact, current tests assert that image content with no `MEDIA:` token and no `details.path` produces no media path, while the pinned upstream `@mariozechner/pi-coding-agent` read tool returns image content without `details.path`. The PR also has an open paired issue by the same author, so it should stay open for maintainer review rather than be cleaned up.

## Best Possible Solution

Keep this PR open for maintainer review. The best fix is a narrow equivalent of the PR: either thread `tool_execution_start` args into media extraction for trusted `read` results, or make `createOpenClawReadTool` attach structured `details.media.mediaUrl`/`details.path` to upstream image results before the subscription handler extracts media. Keep regression coverage for `path`, `file_path`, `filePath`, and final outbound media delivery, and resolve the paired #41744 issue only after the fix lands.
