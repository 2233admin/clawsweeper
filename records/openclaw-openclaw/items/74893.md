## Summary

Keep this PR open for normal maintainer review. Current main still lacks the proposed cron diagnostic, and the supplied diff is a narrow runtime warning plus one test fixture adjustment with no blocking correctness or security finding from this read-only review.

## What This Changes

The PR adds a WARN log during isolated cron `agentTurn` preparation when the payload has neither `lightContext: true` nor a non-empty `toolsAllow`, and updates one model-override test fixture to opt into light context.

## Best Possible Solution

Land a narrow diagnostic if maintainers want this behavior, preferably with a focused warning assertion; creation-time validation, timeout-default changes, or more prominent docs can remain separate product follow-up for the linked bug.

## Reproduction Assessment

Yes, at code-path level: current main forwards `lightContext` and `toolsAllow` into the embedded runner, and `toolsAllow` changes prompt/skills loading. A deterministic live timeout reproduction is less certain because the exact cold-start duration depends on hardware and workspace size.

## Solution Assessment

Yes for a diagnostic fix: a runtime warning is a narrow, low-risk way to explain the silent timeout mode without changing scheduling or execution semantics. It is not a full prevention mechanism, so maintainers may still want docs or create/update validation later.
