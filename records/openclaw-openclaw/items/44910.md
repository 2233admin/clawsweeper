## Summary

Keep open because current main still does not classify a top-level FastAPI/Codex-style `detail` string as an API error payload, so the reported short JSON can still pass through as user-facing text. The issue is also paired with open, unmerged PR #45216, which uses closing syntax for the narrow parser/test fix.

## What This Changes

The issue requests that Codex/OpenAI provider errors from unconfigured paths not leak raw JSON into user-facing Telegram chat.

## Best Possible Solution

Land or replace the narrow parser and regression-test fix from #45216, then verify whether unconfigured Codex/provider probes can still fire and add credential-scoped suppression if that trigger remains.
