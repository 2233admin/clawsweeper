## Summary

Keep open. Current main still lacks the core gateway protections this PR is trying to add, but the PR is not merge-ready: it is currently unmergeable, overlaps the open Control UI fix in #72004, and its inbound fallback does not cover the reported unstable-id retry path because any non-empty MessageSid still takes the primary key path.

## What This Changes

This PR adds a content-hash fallback to inbound dedupe when MessageSid is absent, threads a chat.send user transcript idempotencyKey, and reorders Control UI final/aborted stream clearing with a regression test.

## Best Possible Solution

Keep the PR open as useful source work, but require a rebase onto current main, preserve the current route-key helper contract, make the inbound dedupe logic cover unstable retry IDs rather than only missing IDs, add focused gateway/user-transcript tests, and coordinate or split the overlapping Control UI fix with #72004.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Handle unstable message IDs before the fast path:** `src/auto-reply/reply/inbound-dedupe.ts:76-78`
  - body: The PR says some retry paths generate a new idempotencyKey per attempt, but this fast path returns a key containing any non-empty MessageSid. chat.send sets MessageSid from clientRunId, so a retry with a fresh id still bypasses the new content hash and can duplicate the exact regression this PR is meant to fix. Gate the fast path on a provider-stable message id, or add a separate stable retry/content key before returning.
  - confidence: 0.88
