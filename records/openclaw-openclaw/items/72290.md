## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the reported duplicate-delivery protection, and this PR is an active contributor fix for open regression #72176, but the branch is unmergeable against current main and the proposed dedupe/transcript changes do not fully cover the reported regenerated-id path.

## What This Changes

The PR adds a content-hash fallback to inbound dedupe, threads `clientRunId` into `chat.send` user transcript updates as an idempotency key, and reorders Control UI chat stream clearing with a regression test.

## Best Possible Solution

Rebase the gateway fix onto current main, preserve the current route-key helper contract, add retry-safe dedupe for missing or unstable message IDs without suppressing legitimate repeated messages, and either split the Control UI ordering change to #72004 or explicitly choose one PR as the canonical UI fix.

## Reproduction Assessment

Yes. The code-level reproduction is to process the same body/timestamp/peer twice with different `MessageSid` values; current main and the PR's `if (messageId)` fast path produce distinct dedupe keys, matching the #72176 regenerated-id report.

## Solution Assessment

No, not as-is. The direction is plausible, but the PR needs a current-main rebase, a dedupe design that handles regenerated ids, and transcript idempotency wired through a path that actually performs an append-time duplicate check.
