## Summary

Keep open. Current main still keeps the TUI streaming watchdog as a hardcoded/internal timeout, still emits the normal watchdog warning without a per-run warned set, and has no `/watchdog` command or persisted TUI setting. Related #69081 is now addressed for proof-of-life/reconnect behavior, and #69026 is an open narrower PR, but neither implements this combined request.

## What This Changes

Request add a TUI `/watchdog` command to show, disable, set, and persist the streaming watchdog timeout while suppressing duplicate watchdog warnings for the same run.

## Best Possible Solution

Implement a TUI-local watchdog preference with status/off/duration parsing and completion, persist it under the existing OpenClaw state-dir pattern, pass the active value into the event handlers, and add a per-run warning guard that resets when a run finalizes, terminates, or the session changes. Keep global schema/protocol configuration under #68596 and preserve the merged #69081 proof-of-life/reconnect behavior.

## Reproduction Assessment

Yes. A focused fake-timer path can send a chat delta, advance past the watchdog window, send another delta for the same run after `activeChatRunId` was cleared, and advance again to observe another `streaming watchdog` system message; `/watchdog` absence is directly reproducible from the command catalog and handler fallthrough.

## Solution Assessment

Yes for the TUI-local portion. A per-run warned set plus local `/watchdog` command is narrower than changing global config, while a complete silent suppression or fixed 120s default alone would not satisfy the requested runtime control and persistence.
