## Summary

Keep open. Current main still has source-only inter-session provenance plus pairwise A2A ping-pong limiting; it does not carry a visited-agent chain or reject A->B->C->A sessions_send cycles. The related PR #37945 proposed the right shape but is closed unmerged, and #51832 tracks a different Slack gateway error-cascade path.

## Best Possible Solution

Keep this issue open and implement a focused core sessions_send fix: add an optional chain/visited-agent field to InputProvenance and the gateway schema, thread it through createSessionsSendTool, runAgentStep, and A2A follow-up runs, reject or warn when the target agent/session is already in the chain, and add regression tests for triangular/polygonal cascades without changing the existing pairwise maxPingPongTurns semantics.

## Clownfish Work Prompt

Fix graph-aware sessions_send cycle detection for this issue. Current main only has source-only InputProvenance and pairwise maxPingPongTurns, so A->B->C->A cascades can bypass the guard. Implement an additive optional chain/visited-agent field on InputProvenance and the gateway schema; thread it through createSessionsSendTool, runAgentStep, and any A2A follow-up run path that creates inter-session provenance; before delivery, parse requester/target agent ids from canonical agent session keys and reject the send with a clear error when the target already appears in the chain. Preserve existing direct A<->B maxPingPongTurns semantics and do not change unrelated Slack/gateway error cascade behavior from #51832. Avoid false positives for legacy/non-agent-prefixed session keys. Add focused tests for A->B->C->A rejection, normal A->B->C non-cycle delivery, legacy key behavior, and existing pairwise ping-pong behavior. Update docs only if the public sessions_send behavior description changes, and add a user-facing changelog fix entry crediting the reporter and relevant prior contribution, not Codex.
