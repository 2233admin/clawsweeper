Preserve structured ACP backend error classifications so clients can distinguish transient or operational failures from normal `end_turn` completion.

## Repair Work Prompt

Fix OpenClaw issue #44294 with a narrow ACP bridge change. Current main receives structured gateway chat `errorKind` values but `AcpGatewayAgent` only maps `refusal` to `stopReason: "refusal"`; all other classes become plain `{ stopReason: "end_turn" }`. Keep ACP stop reasons compliant with `@agentclientprotocol/sdk` 0.20.0: do not add non-standard stop reasons such as `timeout` or `rate_limit`. Instead, preserve non-refusal backend classifications in `PromptResponse._meta` using a stable OpenClaw-owned metadata key so ACP clients can distinguish `timeout`, `rate_limit`, `context_length`, and `unknown` from normal completion. Update `src/acp/translator.ts`, focused tests in `src/acp/translator.error-kind.test.ts` and related stop-reason tests, and document the metadata behavior in `docs/cli/acp.md` if it becomes a supported client contract. Add a user-facing changelog entry with reporter credit if behavior changes. Do not change gateway protocol values unless the ACP projection requires a generic additive seam.
