## Summary

Keep open. Current main still preserves `errorKind` only as far as Gateway chat events; `AcpGatewayAgent` still resolves ACP error turns as `refusal` only for `errorKind: "refusal"` and `end_turn` for every other structured backend error. The targeted tests still assert that `timeout` and `unknown` resolve to plain `{ stopReason: "end_turn" }`, so ACP clients cannot distinguish transient/backend failures from normal completion yet.

## Best Possible Solution

Keep this issue open and implement a narrow ACP bridge fix: keep ACP-compliant `stopReason` values, map `refusal` to `refusal`, and preserve non-refusal backend error classification through `PromptResponse._meta` so clients can distinguish `timeout`, `rate_limit`, `context_length`, and `unknown` from normal `end_turn`. Add targeted ACP translator tests and update the ACP bridge docs/changelog if the client-visible response changes.

## Clownfish Work Prompt

Fix the ACP bridge so structured Gateway chat error kinds are preserved for ACP clients instead of disappearing behind plain `stopReason: "end_turn"`.

Observable bug: current `AcpGatewayAgent.handleChatEvent` receives chat error payloads with `errorKind` values such as `timeout`, `rate_limit`, `context_length`, and `unknown`, but `finishPrompt` resolves only `{ stopReason: "end_turn" }` for all non-refusal errors. ACP clients cannot distinguish backend failures from normal completion.

Expected boundary: keep ACP SDK-compatible stop reasons. Do not emit unsupported stopReason strings like `timeout` or `rate_limit`. Preserve the structured classification through `PromptResponse._meta`, preferably under an OpenClaw-owned namespaced key that includes at least the backend `errorKind` and, if already available, a sanitized `errorMessage`. Keep `errorKind: "refusal"` mapped to `stopReason: "refusal"` and include consistent metadata if useful.

Likely files: `src/acp/translator.ts`, `src/acp/translator.error-kind.test.ts`, possibly `src/acp/translator.stop-reason.test.ts`, `docs/cli/acp.md`, and `CHANGELOG.md` if the client-visible behavior changes. Avoid broad gateway protocol changes unless a test proves the current chat payload is insufficient.

Related refs: this issue, #55919 as a sibling provider-classification bug, merged #41187 as the predecessor `error -> end_turn` behavior, and commit `c2e2b87f28f0fae2fa3b7395c66077be31ec74f7` as the current partial errorKind plumbing.

Validation: add/update targeted tests for `refusal`, `timeout`, `rate_limit`, `context_length`, and `unknown`; verify final responses include ACP-compliant `stopReason` plus structured metadata for non-refusal errors. Run the focused ACP translator tests and the gateway chat error-kind test. Use the repo’s formatter and changed gate policy before handoff.

Changelog: add a single-line user-facing fix entry with reporter attribution if the response metadata is externally visible. Do not change unrelated ACP runtime, acpx harness, provider classification, or gateway startup behavior.

## Risks / Open Questions

- ACP SDK 0.20.0 does not allow `timeout`, `rate_limit`, `context_length`, or `server_error` as `stopReason`, so a fix should preserve those through `_meta` or an agreed protocol extension rather than emitting invalid stop reasons.
- Changing non-refusal backend errors to rejected ACP promises could be a compatibility change for clients that expect `session/prompt` to resolve; a narrow metadata-preserving response is the lower-risk path unless maintainers choose otherwise.
- Error metadata should be namespaced, for example under an OpenClaw-owned `_meta` key, so ACP clients do not mistake it for a standardized protocol field.
