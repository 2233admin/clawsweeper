## Summary

Keep open. Current main has structured gateway chat `errorKind` plumbing, but the ACP translator still exposes only `refusal` distinctly and collapses `timeout`, `rate_limit`, `context_length`, and `unknown` errors to `stopReason: "end_turn"`. Targeted tests currently assert that collapse, so ACP clients still cannot distinguish transient/backend failures from normal completion.

## Best Possible Solution

Keep this issue open. The maintainer path is to thread `ChatEvent.errorKind` through `AcpGatewayAgent.finishPrompt` into a client-visible `PromptResponse` field, probably `_meta` under the current ACP SDK contract, while preserving the existing `refusal` stopReason mapping. Add targeted translator tests for `timeout`, `rate_limit`, `context_length`, and unknown/server failures so ACP clients can distinguish backend failure classes from normal `end_turn` completion.
