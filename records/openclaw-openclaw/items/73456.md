## Summary

Keep open. Current main still lacks Azure Speech realtime transcription, and this PR is a useful plugin-owned implementation that appears to have fixed the earlier Greptile lazy-connect audio-drop report, but it is not merge-ready because the Azure SDK session can leak if closed during an in-flight connect and the PR still omits required generated config baseline and changelog artifacts.

## What This Changes

The PR adds Azure Speech as a bundled Voice Call realtime transcription provider with provider/session code, manifest metadata, docs, unit/live tests, a Microsoft Speech SDK runtime dependency, and small incidental formatting changes.

## Best Possible Solution

Keep the Azure Speech implementation in the Azure Speech plugin, fix the async teardown race, commit the generated config baseline and a user-facing changelog entry, then merge after maintainer dependency review and targeted Azure Speech, Voice Call, extension type/lint, config docs/schema, and package/runtime checks pass.

## Reproduction Assessment

Yes for the review blockers: Voice Call starts STT connect asynchronously and closes sessions on stream stop, while the PR session does not guard against close() having already run after async Azure SDK startup. The manifest and file list also directly reproduce the missing config baseline and changelog artifacts.

## Solution Assessment

No, the patch is not the best mergeable form as-is. The plugin-owned provider design is the right boundary, but the maintainable solution needs the teardown guard, regenerated baseline, and changelog entry before normal maintainer review of the Microsoft SDK dependency and live-provider behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Abort connect when the session is already closing:** `extensions/azure-speech/realtime-transcription-session.ts:245-246`
  - body: Voice Call starts `sttSession.connect()` without awaiting it and calls `close()` when the media WebSocket stops. If the caller closes while `deps.loadSdk()` or Azure startup is still pending, `teardown()` can run before `recognizer`/`pushStream` exist; this code later marks the session connected and leaves the newly-created recognizer open. Check `closing` after async startup, close any created SDK resources, clear pending audio, and add a close-during-connect regression test.
  - confidence: 0.84

The baseline generator loads bundled plugin manifests from `extensions/`, so `pnpm config:docs:check` should report drift until the generated hash is committed.
  - confidence: 0.91
