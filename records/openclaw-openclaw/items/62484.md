## Summary

Keep open. Current `main` still has the synchronous CLI help path described in the report, and the issue is already paired with an open PR that explicitly targets the bug, so cleanup should not close the issue before that PR is reviewed and resolved.

## What This Changes

The issue reports that `openclaw tui -h` hangs at 100% CPU because synchronous Commander help rendering reads banner config through the heavy config loader.

## Best Possible Solution

Keep the issue open while maintainers review PR #62485 or land an equivalent fix that keeps synchronous CLI help rendering free of heavy config/plugin discovery work, with regression coverage for the real help-to-banner path.

## Reproduction Assessment

Yes. The report gives a concrete command, and current `main` still connects `openclaw tui -h` help generation to `formatCliBannerLine()` and then to `createConfigIO().loadConfig()` through the banner config reader.

## Solution Assessment

Unclear for the linked PR until it receives normal patch review. The right solution belongs in the CLI banner/config boundary, not in the TUI runtime itself, and should preserve configured tagline behavior outside the synchronous help path.
