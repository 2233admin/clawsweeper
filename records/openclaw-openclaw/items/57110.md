## Summary

Keep open. Current main documents the Tailscale Serve trust boundary and offers `gateway.auth.allowTailscale: false` to require token/password instead of Tailscale identity, but it does not implement the requested secondary-auth mode where verified Tailscale identity and a shared token/password are both required. Related #50630/#50631 track the narrower `auth.mode="none"` exposure gap, not this dual-auth enhancement.

## Best Possible Solution

Keep this open for maintainer product/security design. If OpenClaw wants the requested defense-in-depth mode, add a first-class gateway auth policy that requires verified Tailscale Serve identity and token/password for WS Control UI, update the config schema/help/docs and macOS/CLI surfaces, add startup or audit warning coverage for tokenless Serve, and add regression tests proving both factors are required. Keep #50630/#50631 focused on the separate `auth.mode="none"` Serve exposure bug.

## Risks / Open Questions

- Adding dual auth is security-sensitive: maintainers should define whether identity, scopes, device identity, and token/password failures are all required or whether token/password remains an alternative fallback.
- The existing documented workaround, `gateway.auth.allowTailscale: false`, may be enough for some deployments but loses Tailscale user identity; closing this now would drop that product decision.
