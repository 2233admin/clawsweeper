## Summary

Keep open. Current main still lacks the requested root/env log-file override, and this PR is a useful but not merge-ready implementation because route-first commands can bypass the new flag and daemon status can report the CLI process log path instead of the inspected service path.

## What This Changes

The PR adds a root `--log-file <path>` option, an `OPENCLAW_LOG_FILE` resolver, logger/log-path precedence changes, daemon status coverage, CLI parser tests, and docs for one-off log-file overrides.

## Best Possible Solution

Revise the PR so root log-file parsing and env promotion happen before route-first execution, the log-file env resolver accepts an explicit env with `process.env` as the default, daemon status passes the merged service env, and the focused CLI/logger/status/docs coverage remains aligned with that precedence model.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Promote --log-file before routed commands:** `src/cli/program/preaction.ts:96-99`
  - body: The new env var is set only from Commander `preAction`, but `runMain` returns from `tryRouteCli()` before Commander parses. Routed paths like `status`, `gateway status`, and `config get/unset` will still ignore `openclaw --log-file /tmp/x ...`, so the advertised global override is unreliable.
  - confidence: 0.94
- **[P2] Thread service env into log path resolution:** `src/logging/log-file-path.ts:40`
  - body: `resolveConfiguredLogFilePath()` now reads `OPENCLAW_LOG_FILE` from the current process env, but daemon status is reporting the gateway service and already computes `mergedDaemonEnv`. When the CLI env and service env differ, `status.logFile` can show the wrong file sink.
  - confidence: 0.9
