## Summary

Keep this PR open. Current main still does not implement account-scoped channel config form resolution, and the PR remains a plausible fix candidate for related #51335, but the current proposed diff has unresolved functional blockers around schema fallback and patch-path alignment.

## Best Possible Solution

Keep this PR open as a focused fix candidate, but require a revised implementation: pass the preferred/default account id into channel config rendering, resolve account-scoped values when present, fall back both schema and value handling to the root channel config for root-backed channels, align the patch path with the object actually displayed, trim account ids consistently, update extras/status consumers if they should reflect account-scoped fields, and add regression tests for account-scoped plus root-fallback cases.

## Risks / Open Questions

- Merging the current PR as written can make root-schema single-account channels such as Microsoft Teams render “Channel config schema unavailable” when a default account id is present.
- The proposed value fallback can display root channel config while form edits patch channels.<id>.accounts.<accountId>, so users may write to a different object than the one shown.
- resolveChannelExtras remains root-only on current main and in the PR discussion, so account-scoped fields may still be omitted from status metadata.
- The PR body reports that the targeted Vitest run was still hanging in the contributor checkout, so regression-test proof is incomplete.
