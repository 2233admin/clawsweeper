## Summary

Keep open. Current main still does not provide opt-in session wall-clock duration caps or cumulative per-session token caps; the existing implementation covers daily/idle reset freshness, per-run timeouts, context prechecks, thread-binding max age, and session-store disk maintenance, but none of those enforce the requested runaway-session cost guard.

## What This Changes

Request optional `session.maxDurationMinutes` and `session.maxTokensPerSession` config keys that gracefully stop, compact, or reset runaway long-lived sessions based on wall-clock age or cumulative token use.

## Best Possible Solution

Implement this as a core session-budget feature: add optional schema/types/help/docs, persist durable per-session lifecycle and cumulative usage-budget state, enforce before LLM calls and tool execution plus lifecycle checks, emit deterministic operator/user feedback, and add focused regression coverage across chat, cron, heartbeat, provider-owned CLI/ACP sessions, and compaction/reset paths.
