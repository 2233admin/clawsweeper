## Summary

Keep open. Current main still lacks the interactive custom-provider metadata prompt, and the PR is useful, but the proposed patch has two merge-blocking correctness bugs around provider-prefixed model IDs and clearing model-level API overrides.

## What This Changes

The PR adds a model metadata prompt helper and tests, then wires custom API-key setup to optionally edit display name, API adapter, reasoning, input modalities, context window, max output tokens, and token pricing.

## Best Possible Solution

Rebase onto current main, keep the metadata step limited to custom provider setup, reuse the shared model-ref parsing/key helpers for native IDs containing slashes, make Auto remove model-level API overrides back to provider inheritance, account for the current contextWindow/contextTokens split, and add focused tests for those cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Resolve provider-prefixed model ids before lookup:** `src/commands/model-metadata.ts:77-80`
  - body: findPrimaryModelEntry splits the primary key and searches for only the suffix. Current modelKey preserves native IDs that already start with the provider prefix, so a custom provider entry such as provider openrouter with model ID openrouter/foo persists primary as openrouter/foo; this code searches for foo and silently skips the metadata prompt.
  - confidence: 0.87
- **[P2] Clear api when Auto is selected:** `src/commands/model-metadata.ts:131-132`
  - body: Selecting Auto (inherit from provider) leaves result.api unset, and the later merge spreads ...entry.model before ...updated, so an existing model-level api survives. The wizard offers provider inheritance but cannot actually clear a previous override.
  - confidence: 0.86
