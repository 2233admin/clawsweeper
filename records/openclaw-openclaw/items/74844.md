## Summary

Close as implemented on main. The reported 2026.4.26 symptom matches the earlier bundle-MCP profile filtering bug, and current main now merges configured `mcp.servers`, lists their MCP tool catalogs, materializes them as `bundle-mcp` agent tools, preserves them in `coding`/`messaging` profiles, and appends the filtered tools into the embedded Pi run’s effective tool list before provider requests are built. The fix commit is included in the shipped `v2026.4.27` release.

## What This Changes

Expose configured Gmail/Outlook MCP server tools from `mcp.servers` to the embedded agent so they appear in the model request `tools[]` array.

## Best Possible Solution

Keep the generic bundle-MCP implementation in core: configured MCP servers should continue to be materialized through the shared session MCP runtime and controlled by normal tool-profile policy, with service-specific Gmail/Outlook behavior remaining outside core.

## Reproduction Assessment

Yes. The issue provides a high-confidence reproduction and captured provider request body for 2026.4.26; current-main verification is source, docs, and regression-test evidence rather than a live Gmail/Outlook credentialed run.

## Solution Assessment

Yes. The current fix is the narrow maintainable path: it preserves the existing MCP runtime and policy model by tagging MCP tools as `bundle-mcp`, including that key in the intended profiles, and retaining the explicit `tools.deny` opt-out.
