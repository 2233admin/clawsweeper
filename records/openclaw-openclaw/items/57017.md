## Summary

Keep open. Current main still reads cron run-log JSONL files with full `fs.readFile` calls before parsing, filtering, sorting, and paginating. The all-jobs path does the same for every `*.jsonl` file in the runs directory. There is also an open unmerged implementation candidate, #57088, whose PR body targets this exact OOM issue and uses GitHub closing syntax for this issue, so this should stay open until maintainers merge or close that PR.

## Best Possible Solution

Keep this open and treat #57088 as the active implementation candidate unless maintainers decide to replace it. The best fix is a race-safe bounded read for both `readCronRunLogEntriesPage` and `readCronRunLogEntriesPageAll`, preferably tail-reading a configured byte window without mutating the log. If read-time pruning is used instead, it should be serialized through `writesByPath`, respect configured `cron.runLog` limits, and avoid introducing another unbounded oversized-file read. Add regression coverage for oversized per-job and all-jobs reads, prune failure resilience, configured limits, and append/read race safety.
