## Summary

Keep this issue open. Current main still full-reads cron run-log JSONL files before pagination in both the per-job and all-jobs read paths, while pruning remains write-path-only. There is also an open unmerged implementation candidate, #57088, whose PR body says it would close this issue, so the issue should stay open until maintainers review and merge or close that PR.

## Best Possible Solution

Keep this issue open and treat #57088 as the active implementation candidate unless maintainers decide to replace it. The best fix is a race-safe bounded read for both `readCronRunLogEntriesPage` and `readCronRunLogEntriesPageAll`, preferably tail-reading a configured byte window without mutating the log; if read-time pruning is used, it should be serialized through `writesByPath`, use configured `cron.runLog` limits, and avoid an unbounded prune read. Add regression coverage for oversized per-job and all-jobs reads, prune failure resilience, configured limits, and append/read race safety.

## Risks / Open Questions

- Leaving the current read path unchanged means an oversized per-job run log can still be loaded fully into memory by `cron.runs`.
- The all-jobs history path can read multiple oversized JSONL files concurrently, so its worst-case memory pressure is higher than the single-job path.
- Any fix that prunes during reads must avoid racing with `appendCronRunLog`; the issue discussion and #57088 review comments already identified serialization through `writesByPath` or a non-mutating bounded tail read as safer directions.
