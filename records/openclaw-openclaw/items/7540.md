## Summary

Keep this issue open. Current main still does not subscribe to Baileys WhatsApp call events, does not expose call activity through the WhatsApp plugin hook/config/session surface, and the pinned Baileys version does provide a typed `call` event. Maintainer context also explicitly scoped this as valid WhatsApp-plugin-owned work rather than a PSTN voice-call or ClawHub item.

## Best Possible Solution

Keep this issue open and implement the feature inside `extensions/whatsapp`: subscribe to Baileys `call`, normalize and sanitize `WACallEvent` payloads, apply the same account, DM/group, self-chat, allowlist, and privacy boundaries as inbound messages, add an opt-in config/plugin setting if maintainers want it, expose the resulting activity through a documented session-event/plugin-hook/tool surface, and cover it with focused tests plus WhatsApp docs updates.
