## Summary

Keep open. Current main has Feishu duplicate-final mitigations, but it still does not prove the reported failover coordination bug is fixed: Feishu still closes and resets the active streaming card on delivery error, generic final dispatch still iterates every non-reasoning final payload, and current Feishu tests intentionally preserve distinct non-streaming finals. The issue is also already paired with open candidate PR #59771, so the best path is to review/land that PR or an equivalent fix rather than close this issue now.

## Best Possible Solution

Keep this issue open until #59771 or an equivalent fix lands. The fix should prevent a failed primary attempt from surfacing a user-visible final when a fallback attempt later succeeds, keep recovered Feishu streaming replies on one visible card where appropriate, and add regression coverage for the reported non-streaming failover path or explicitly split that remaining path into a narrower follow-up.
