## Summary

Keep this issue open. Current main still bounds and sanitizes WebChat history, and it has a separate paginated REST/SSE session-history endpoint, but the affected Control UI path still calls the WebSocket `chat.history` latest-window request with no cursor/hasMore contract, still materializes the transcript snapshot before trimming, and still renders a capped Lit `repeat()` list rather than paginated or virtualized history.

## Best Possible Solution

Keep this issue open and implement a current-main-native WebChat history window. The safest path is to add cursor/hasMore semantics to the WebSocket `chat.history` path used by Control UI, or explicitly route Control UI through the existing authenticated `/sessions/:key/history` endpoint, then add load-earlier behavior with scroll/session/streaming tests. Follow with virtualization or chunked rendering if realistic long-transcript proof still shows main-thread jank.

Use the existing WebSocket Control UI path as the primary integration: extend `chat.history` to support a backward cursor based on transcript seq (for example `cursor`/`nextCursor` plus `hasMore`) and reuse or align with `src/gateway/session-history-state.ts` pagination semantics where practical.
