## Summary

Keep open. Current main has an internal stable per-agent heartbeat phase and deterministic cron-expression staggering, but the requested public scheduling contract is still missing: no prime cadence mode, no configurable heartbeat phase/jitter fields, no resolved phase/status exposure, and cron `every` schedules remain fixed intervals with CLI/protocol normalization still treating staggering as cron-expression-only.

## What This Changes

Request to add prime-number cadence options and deterministic per-agent phase jitter for heartbeat and cron `every` schedules to reduce synchronized load spikes.

## Best Possible Solution

Keep this open as a scheduler/API design item. Preserve the existing internal stable heartbeat phase, then decide additive public controls for heartbeat phase/jitter, optional prime cadence, and cron `every` jitter/prime semantics. If accepted, implement the contract through config normalization/types/help, Gateway protocol/schema, CLI, status or Control UI exposure, docs, and focused heartbeat/cron scheduler tests, with backward-compatible defaults unless maintainers explicitly choose a default change.

## Risks / Open Questions

- Changing default heartbeat or cron `every` timing could break users who rely on exact fixed intervals, so any behavior change needs explicit additive semantics or a carefully reviewed migration/default strategy.
- The deterministic seed should not expose raw gateway, agent, channel, or account identifiers in logs/status output if public phase observability is added.
- Cron expression staggering, missed-job startup staggering, heartbeat internal phasing, and future `every` jitter are related but distinct behaviors; conflating them could create surprising delivery timing or duplicate-run regressions.
