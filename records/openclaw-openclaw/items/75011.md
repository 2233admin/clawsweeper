## Summary

Keep open. Current main does not contain the requested troubleshooting section, and the PR remains useful, but the current head still needs one docs correction: it tells operators to retry after approval without documenting the reconnect/restart needed for newly approved node commands to become visible on an already-connected node. The earlier placeholder regression called out by the bot review appears resolved in the current PR diff.

## What This Changes

This PR updates `docs/gateway/pairing.md` with Gateway-owned pairing troubleshooting for dropped commands, missing pending requests, and stale tokens after re-pair, plus quote and apostrophe punctuation changes.

## Best Possible Solution

Merge a docs-only update after adjusting the troubleshooting guidance to match current node pairing behavior: keep the required CLI placeholders, explain the node-pairing/device-pairing distinction, and tell operators to reconnect or restart the node host after approving command upgrades before retrying commands.

## Reproduction Assessment

Yes. Reading the PR diff reproduces the docs issue, and the current source provides a focused verification path: approve-node-pairing persists state, while live command visibility is fixed at node connect time until reconnect.

## Solution Assessment

No, not quite as proposed. Adding troubleshooting near the node command gating docs is the right direction, but the maintainable fix is to document the reconnect/restart step instead of implying that retrying the command alone will refresh live command visibility.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Document the reconnect before retrying approved commands:** `docs/gateway/pairing.md:209-211`
  - body: The new troubleshooting text tells operators to re-trigger a command after approval, but current gateway code snapshots a node's effective commands when the node connects. For command-upgrade pairing, approval updates the pairing store only; an already-connected node still reports the old live command list until it reconnects, so the recovery step should say to reconnect or restart the node host before retrying.
