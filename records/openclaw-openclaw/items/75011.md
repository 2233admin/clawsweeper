## Summary

Keep open. The PR is a useful docs-only direction, but the proposed troubleshooting text still does not match current gateway behavior: current main keeps allowlisted declared commands visible before node pairing exists, command upgrades on already paired nodes remain filtered until reconnect, and approval/rejection only changes the pairing store rather than refreshing an active session.

## What This Changes

This PR adds a Troubleshooting section to docs/gateway/pairing.md for dropped-command, missing pending-request, and stale-token recovery scenarios, with minor quote normalization.

## Best Possible Solution

Land a docs-only correction that reconciles docs/gateway/pairing.md with current gateway behavior: distinguish first-time node pairing from paired-node command upgrades, document reconnect/restart before retrying newly approved command surfaces, and describe stale/lost token recovery as remove/re-pair or create a new request before approval.

Yes. The docs gap and stale command-gating framing are visible in docs/gateway/pairing.md, and the patch defects are reproducible by tracing current connect-time reconciliation, node-list command snapshots, approval/rejection store updates, and existing node-pairing tests.

## Solution Assessment

No, not as currently proposed. Adding troubleshooting near the node-pairing docs is the right place, but the wording should be corrected before merge so it does not imply approval refreshes an active node session or that rejecting a request issues a fresh token.
