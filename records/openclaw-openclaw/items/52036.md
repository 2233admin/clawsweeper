## Risks / Open Questions

- Merging this PR as-is would make guarded appendMessage asynchronous at runtime while the pinned dependency and current OpenClaw code expose a synchronous string-returning contract.
- Moving pending tracking before hook/write acceptance can synthesize tool results for assistant messages that were blocked or failed to persist.
- Current main’s idle-before-flush mitigation may cover some linked synthetic-result symptoms, but it does not prove the PR’s broader async append-adapter race is solved.
