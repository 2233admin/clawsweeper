## Summary

Keep this PR open for maintainer judgment, but do not merge it as-is. Current main does not implement the PR's append-message mutex, while the submitted approach conflicts with the upstream synchronous SessionManager.appendMessage contract and has credible review-identified regressions around return type and pending-state ordering.

## Best Possible Solution

Do not merge this PR as-is. If maintainers still see a reproducible appendMessage interleaving after the current wait-for-idle cleanup, the fix should preserve the synchronous appendMessage contract or deliberately update that contract and all returned-message-id call sites, and it should add a regression test that fails on current main through a real async boundary or explicit async append adapter. If the idle-before-flush path fully covers the observed synthetic-result failures, close this PR manually as obsolete after that product decision.

## Risks / Open Questions

- Merging the PR as submitted would make guarded appendMessage return a Promise at runtime while the upstream and OpenClaw contract is synchronous string-returning appendMessage.
- Moving pending tracking before write acceptance can synthesize tool results for assistant tool calls that were never persisted if hooks block the message or originalAppend fails.
- Current main's idle-before-flush fix may already cover the real linked user failures, but it does not prove the PR's direct concurrent appendMessage race or make the mutex implementation safe.
