## Summary

Keep this PR open for maintainer handling, but do not merge it as submitted. Current main has a separate idle-before-flush mitigation for the linked synthetic-result failures, while this branch introduces blocking contract and transcript-pairing regressions in the guarded append path.

## What This Changes

The PR adds a promise-chain mutex around `installSessionToolResultGuard`, moves pending tool-call tracking before append persistence, and adds concurrent append regression tests.

## Best Possible Solution

Keep the current idle-before-flush mitigation unless linked failures still reproduce, and require any follow-up fix to preserve the synchronous append contract, avoid pending ids for blocked or failed writes, and prove the real async boundary with targeted regression coverage.

## Reproduction Assessment

Unclear. The historical synthetic-result reports are credible and current main has idle-before-flush coverage, but this PR's exact concurrent `appendMessage` race is not high-confidence against the pinned dependency because `SessionManager.appendMessage` is synchronous.

## Solution Assessment

No. The submitted mutex is not the best fix because it silently changes the guarded append runtime contract and records pending tool calls before the transcript write is accepted; a narrower contract-preserving rework would be safer.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0

Full review comments:

- **[P1] Preserve synchronous appendMessage behavior:** `src/agents/session-tool-result-guard.ts:310-312`
  - body: `appendMutex.run(...)` always returns a Promise and defers the append body, but the cast keeps TypeScript believing `appendMessage` still returns the synchronous entry id. Current `SessionManager.appendMessage` is declared and implemented as `string`, and guarded callers can read state, emit ids, or flush immediately after append, so this patch lets that code run before the append has happened.
  - confidence: 0.93
- **[P1] Track pending calls only after accepted writes:** `src/agents/session-tool-result-guard.ts:282`
  - body: Recording pending tool-call ids before the write hook and `originalAppend` means a blocked assistant message or thrown append can leave ids pending even though no tool-call turn was persisted. The next non-tool message can then synthesize tool results for calls that never existed in the transcript.
  - confidence: 0.91
