## Summary

Keep open. Current main still cannot express an independent memorySearch fallback remote/model: fallback remains a provider string, the resolver emits one shared remote block, and both fallback execution paths reuse that shared configuration.

## What This Changes

The issue requests separate memorySearch fallback provider configuration so a primary embedding provider and fallback embedding provider can use independent model and remote endpoint settings.

## Best Possible Solution

Add a backward-compatible fallback object form to the memory search config contract and thread fallback-specific provider, model, remote, headers, input-type, dimensionality, and local options through validation, resolved config, provider creation, docs/help/generated schema, and focused regression tests while keeping the existing string form intact.
