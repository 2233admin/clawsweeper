## Summary

Keep open. Current main still has the reported configuration limitation: memory search accepts only a string fallback provider and one shared remote block, so a Gemini primary plus OpenAI-compatible fallback cannot use independent endpoints. The closed broad follow-up #55590 and superseded PR #44878 both point back to this narrower issue as the right active tracking item.

## What This Changes

Request support for separate memorySearch fallback provider, model, and remote settings so primary and fallback embedding providers can use independent endpoints.

## Best Possible Solution

Add a backward-compatible fallback object form for memorySearch that preserves the existing string shorthand while allowing fallback provider, model, remote, headers, input type, dimensions, and local settings to resolve independently through schema, public types, runtime provider creation, docs/generated config help, and focused regression tests.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: the requested fallback object is rejected by the strict schema, and the current string fallback path reuses the single resolved remote block that Gemini reads before provider defaults.

## Solution Assessment

Yes. A fallback object is the narrowest maintainable direction because it keeps existing configs compatible while avoiding a growing set of separate top-level keys such as fallbackRemote and fallbackModel.
