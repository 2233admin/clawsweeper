## Summary

Keep this PR open for maintainer review, but do not merge it as-is. Current main still lacks per-job cron directory storage, so the intended feature is not implemented; the proposed patch is also partial and has blocking correctness issues already noted in review comments, plus a security-sensitive filename sanitization problem.

## What This Changes

The PR adds helper functions and tests for an opt-in `cron/jobs/` directory where each cron job is stored as a separate JSON file, including filename-derived IDs, disabled filename prefixes, and a migration helper from `jobs.json`.

## Best Possible Solution

Rework the feature on current main as a real cron storage backend: keep `jobs.json` backward compatibility, make directory activation explicit, use stable sanitized id-keyed filenames or another collision-safe scheme, preserve the split runtime-state sidecar contract, wire load/add/update/remove/migration/status paths, and add docs/doctor coverage for overlap and migration behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Wire directory files into cron load and persistence:** `src/cron/store.ts:196`
  - body: The added directory store is never used by the live service. `loadCronStoreDir` and `hasCronJobsDir` are exported, but `ensureLoaded` still reads `state.deps.storePath` via `loadCronStore`, and add/update/remove still persist the monolithic store, so creating `cron/jobs/` would not change gateway behavior.
  - confidence: 0.97
- **[P1] Key job files by immutable IDs:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` builds filenames from `job.name`, but cron ids are UUIDs and names are mutable/nonunique. Duplicate names can overwrite during migration or save, and a rename writes a second file for the same job id instead of replacing the original.
  - confidence: 0.96
- **[P1] Delete persisted files by the job id inside the file:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested id with the filename-derived id, while files written by this patch are name-derived. For a normal job such as id `brew-1`, name `Daily Brew`, delete returns false and leaves the file behind.
  - confidence: 0.96
- **[P2] Keep filename fallbacks inside the jobs directory:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, `slugifyCronJobName` returns raw input. A name made only of dot or slash characters can then reach `path.join(jobsDir, ...)` as a relative segment and escape `cron/jobs/`; use a safe fallback and validate the resolved path.
  - confidence: 0.9
- **[P2] Preserve the split runtime-state contract:** `src/cron/store.ts:248`
  - body: Current main stores mutable runtime fields in `jobs-state.json`, but this patch serializes the whole `CronJob` into each directory file. If directory storage is wired in, pending slots and run state would become part of user-editable config files again unless the per-job format also separates config from runtime state.
  - confidence: 0.85

## Security Review

Status: needs_attention

Summary: The diff adds new filesystem write paths and contains a concrete path traversal risk in filename generation; no third-party dependency or CI supply-chain change was found.

Concerns:

- **[medium] Sanitize directory job filenames:** `src/cron/store.ts:151`
  - body: `slugifyCronJobName` returns the raw name/id when no alphanumeric slug remains, and `saveCronJobFile` joins that value into a path. A crafted local cron job name made of relative path characters can escape `cron/jobs/`; the fix should use a guaranteed-safe fallback and check the resolved path is under the jobs directory.
  - confidence: 0.9
