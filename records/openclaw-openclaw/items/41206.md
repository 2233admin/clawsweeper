## Summary

Keep this PR open. Current main still only supports the single `jobs.json` cron store, so the feature is not obsolete, but the proposed patch is not merge-ready because the new directory-store helpers are not wired into live cron persistence and have correctness and filename-safety issues.

## What This Changes

The PR adds exported helpers and tests for an optional `cron/jobs/` one-file-per-job cron store, including filename-derived IDs, disabled filename prefixes, per-job save/remove helpers, and migration from `jobs.json`.

## Best Possible Solution

Rework the feature as a first-class cron storage backend on current main: keep `jobs.json` compatibility, make directory activation and precedence explicit, use stable collision-safe ID-keyed filenames, preserve the `jobs-state.json` runtime sidecar split, wire service/CLI/gateway/doctor/docs paths, and cover overlap and migration behavior with focused tests.

## Reproduction Assessment

Yes. Static reproduction is high-confidence: a job with id `brew-1` and name `Daily Brew` would be saved as `daily-brew.json`, while removal by `brew-1` compares against the filename-derived `daily-brew` and leaves the file behind.

## Solution Assessment

No. The requested direction is valid, but this PR is not the best current implementation because it adds unused helpers, keys files by mutable display names, writes runtime state into per-job config files, and leaves the live storage contract undefined.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Wire the directory store into live persistence:** `src/cron/store.ts:196`
  - body: `loadCronStoreDir` is exported here, but the cron service, CLI/gateway, and doctor paths still use the existing single-file `loadCronStore`/`saveCronStore` flow. Creating `cron/jobs/` would not change runtime behavior, so the advertised opt-in store is unreachable.
  - confidence: 0.97
- **[P1] Use immutable IDs for job filenames:** `src/cron/store.ts:246-247`
  - body: `saveCronJobFile` derives the path from `job.name`, but names are mutable and nonunique while cron job IDs are stable. Duplicate names can overwrite each other, and a rename writes a second file for the same job ID that reloads as a duplicate.
  - confidence: 0.96
- **[P1] Delete files by persisted job IDs:** `src/cron/store.ts:271-272`
  - body: `removeCronJobFile` compares the requested ID with the filename-derived ID, but files written by this patch are name-derived. A normal job whose stable ID differs from its display name will not be removed.
  - confidence: 0.96
- **[P2] Keep fallback filenames inside jobs:** `src/cron/store.ts:151`
  - body: When slugification produces an empty string, this helper returns the raw input. Values containing relative path segments can then reach `path.join(jobsDir, ...)`, so the fallback must be safe and the resolved file path must remain under `jobs/`.
  - confidence: 0.9
- **[P2] Preserve the runtime state sidecar:** `src/cron/store.ts:248`
  - body: Current main strips mutable runtime fields from `jobs.json` and stores them in `jobs-state.json`, but this writer serializes the whole `CronJob` into each per-job file. If wired in, runtime execution state becomes user-editable config again.
  - confidence: 0.85
