## Summary

Keep open. Current main still drains only the active session event queue, so the reported delivery gap is not implemented there. The PR is useful source work, but its current patch is not merge-ready because it drains the configured default main key rather than the active agent's main key and broadens fallback draining to every non-main session without maintainer-approved scope or coverage.

## What This Changes

The PR changes formatted system-event draining to merge the configured main-session event queue into non-main session drains and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side fix onto current main using the active session's agent id for the fallback main key, then either restrict the fallback to approved direct-message scopes or document and test the broader non-main-session behavior explicitly.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: `resolveMainSessionKey(params.cfg)` returns the configured default agent's main key, not the active session's agent. For a session like `agent:ops:telegram:direct:...`, this can drain `agent:main:main` while still missing `agent:ops:main`, so agent-specific events can cross agent boundaries or remain stuck. Derive the fallback from `params.sessionKey` instead.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The `mainSessionKey !== params.sessionKey` check also matches group, channel, and thread sessions. That lets an unrelated non-DM turn consume and clear a main-session wake event, injecting that context outside the intended conversation. Restrict this fallback to approved DM scopes or add explicit maintainer-approved coverage for the broader behavior.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The patch can route untrusted or agent-specific system-event context into the wrong agent/session until the fallback key and scope are constrained.

Concerns:

- **[medium] Cross-agent system-event drain:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: Using the config-default main key means a non-default agent turn can drain another agent's main-session queue. System events can include untrusted hook text, so this crosses agent isolation boundaries and can inject context into the wrong prompt.
  - confidence: 0.86
- **[low] Over-broad non-main session drain:** `src/auto-reply/reply/session-system-events.ts:83`
  - body: The fallback applies to every non-main session, including groups, channels, and threads. That can move main-session wake context into unrelated conversations and clear it before the intended session sees it.
  - confidence: 0.74
