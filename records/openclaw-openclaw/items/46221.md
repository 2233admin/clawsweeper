## Summary

Keep this PR open. Current main still drains formatted system events only from the active session key, while the documented system-event and wake paths still enqueue those events onto the canonical main session key. The PR is a plausible fix for linked issue #42919, but its current patch still needs maintainer rework around active-agent scoping and intended non-main session breadth before it should land.

## Best Possible Solution

Keep the PR open for focused rework. Implement the receiver-side merge in `src/auto-reply/reply/session-system-events.ts`, but derive the fallback key from the active session's agent id using `resolveAgentIdFromSessionKey(params.sessionKey)` plus `resolveAgentMainSessionKey(...)`. Add regression tests for per-channel-peer delivery, intended broader non-main scopes, and multi-agent isolation; then document the intended scope in code comments/tests before landing.

## Risks / Open Questions

- Merging the PR as currently described could let a non-default active agent session drain the default agent's main system-event queue because the fallback key is config-default based rather than derived from `params.sessionKey`.
- The intended breadth still needs maintainer agreement: the author clarified that all non-main sessions should receive main-key events, while review comments flagged group/channel sessions consuming unrelated main-session events.
