## Summary

Keep open. Current main still has the reported delivery gap, and this contributor PR is a useful direction, but the proposed fallback drain is not merge-ready because it can drain the default agent's main queue from a different active agent session and applies to every non-main session shape without maintainer-approved scope semantics.

## What This Changes

The PR changes formatted system-event draining to merge the canonical main-session queue into non-main session drains, sorts merged events by timestamp, and adds a per-channel-peer Telegram DM regression test.

## Best Possible Solution

Port the receiver-side merge onto current main using the active session's agent id for the fallback main key, then either restrict fallback delivery to approved direct-message isolation scopes or explicitly document and test broader non-main-session semantics, including multi-agent isolation and group/channel/thread behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Scope fallback drain to the active agent:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: The new code calls resolveMainSessionKey(params.cfg), which resolves the configured default or first agent, not the agent encoded in params.sessionKey. In a non-default agent session such as agent:ops:telegram:direct:..., this can drain agent:main:main while still leaving agent:ops:main stuck. Derive the fallback from the active session's agent id before merging queues.
  - confidence: 0.91
- **[P2] Constrain the main-key merge scope:** `src/auto-reply/reply/session-system-events.ts:83-87`
  - body: The mainSessionKey !== params.sessionKey guard matches every non-main session, including group, channel, thread, and custom sessions. That lets an unrelated conversation consume and clear a main-session wake event, so restrict the fallback to approved DM isolation scopes or add explicit maintainer-approved coverage for the broader behavior.
  - confidence: 0.78

## Security Review

Status: needs_attention

Summary: The diff can route untrusted system-event context across agent or conversation boundaries until the fallback key and delivery scope are constrained.

Concerns:

- **[medium] Cross-agent system-event drain:** `src/auto-reply/reply/session-system-events.ts:82`
  - body: Using resolveMainSessionKey(params.cfg) for the fallback means a non-default agent turn can drain the default agent's main queue. System events can include untrusted hook text, so this can inject context into the wrong agent/session prompt.
  - confidence: 0.86
- **[low] Over-broad non-main session drain:** `src/auto-reply/reply/session-system-events.ts:83`
  - body: The fallback applies to every non-main session shape, including group, channel, thread, and custom sessions. That can move main-session wake context into unrelated conversations and clear it before the intended session sees it.
  - confidence: 0.74
