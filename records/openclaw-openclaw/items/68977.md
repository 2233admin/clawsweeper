## Summary

Keep open. Current main still exposes `enqueueSystemEvent` as an ephemeral in-memory session queue with only consecutive duplicate text suppression and a boolean result; adjacent durable idempotency primitives exist, but they do not implement scoped idempotency on this public system-event API.

## What This Changes

The issue requests durable `(sessionKey, idempotencyKey)` scoped deduplication and result reporting for the plugin-facing `enqueueSystemEvent` API.

## Best Possible Solution

Design an additive plugin SDK/core contract for durable session-scoped idempotent event enqueueing, including cleanup after processing and tests for cross-session keys, duplicate results, restart persistence, and payload conflicts; alternatively, document and expose an equivalent durable resume primitive that still wakes the target session.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence verification path for the missing capability: current `enqueueSystemEvent` lacks an `idempotencyKey`, durable store, and `EnqueueResult` return contract.

## Solution Assessment

Unclear until maintainer API design. Scoped idempotency is plausible, but the safest solution must reconcile the intentionally ephemeral system-event contract with existing durable next-turn injection semantics rather than silently changing behavior.
