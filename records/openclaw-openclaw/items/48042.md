## Summary

Current main still appears to reproduce the reported config drift. With top-level `browser.cdpUrl` plus `defaultProfile: "user"` and no explicit `profiles.user`, the browser resolver still preserves `user` as the default, synthesizes `user` as an `existing-session` Chrome MCP profile, and does not inherit the top-level CDP URL into that profile. The latest release tag `v2026.4.26` retains the same resolver behavior, and the focused fix PR #48068 is closed unmerged.

## Best Possible Solution

Keep this issue open and land a narrow browser plugin fix. In `extensions/browser/src/browser/config.ts`, when top-level `browser.cdpUrl` is set and `defaultProfile: "user"` is configured without an explicit `profiles.user`, avoid silently selecting the synthesized Chrome MCP `user` profile; either skip synthesizing `user` for that case or rewrite the default to the CDP-backed `openclaw` profile. Add a regression test for the exact reported config and update `docs/tools/browser.md` if maintainers choose a different supported configuration shape.
