## Summary

Keep open. The regression is real on current `main`, and this PR targets the right resolver path, but the proposed no-context suppression check still relies on an implicit unconditional-only behavior that the manifest suppression contract does not guarantee for provider-API-only conditions.

## What This Changes

The branch adds an inline-model suppression check in `resolveExplicitModelWithRegistry`, a regression test for configured `openai-codex/gpt-5.4-mini` rows with `api`, and a changelog entry.

## Best Possible Solution

Keep the inline OpenAI Codex rejection, but implement it through an explicit unconditional-suppression path or corrected condition semantics, then cover both the stale OpenAI inline row and a provider-API-only conditional suppression case.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Use an explicit unconditional suppression path:** `src/agents/pi-embedded-runner/model.ts:596-599`
  - body: The new no-context call assumes `shouldSuppressBuiltInModel({ provider, id })` only matches unconditional suppressions. The matcher skips `providerConfigApiIn` when `config` is omitted, while the public manifest contract allows that condition independently, so an API-only conditional suppression from a plugin could now reject valid configured inline rows. Please add an explicit unconditional-suppression check or make missing config fail `providerConfigApiIn`, with coverage for that case.
  - confidence: 0.86
