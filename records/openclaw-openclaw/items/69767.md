## Summary

Keep open. Current main still has the reported completion-delivery race: when no child completion text, child findings, or fallback reply exists, the announce flow promotes `(no output)` into the requester-facing completion event, and the focused regression coverage still asserts that behavior. The related hardening PR remains open and unmerged, so this is not implemented on main.

## What This Changes

The issue asks OpenClaw to defer async/subagent completion announcements until a usable user-facing result payload, structured payload, or sanitized fallback exists.

## Best Possible Solution

Land PR #68464 or a narrower successor that persists a user-safe completion payload or retryable delivery state, defers completion delivery until real child output or a sanitized fallback exists, and changes the current placeholder regression expectation into a fail-closed test.

## Reproduction Assessment

Yes. Current main has a focused test path where child history contains only user text, `expectsCompletionMessage` is true, and the announce still succeeds with `(no output)`; the issue discussion also includes production symptoms from Telegram and Control UI.

## Solution Assessment

No. The current implementation is not the best solution because it can promote a placeholder as the completion result; the narrow maintainable fix is to defer/retry or persist a user-safe payload before delivery.
