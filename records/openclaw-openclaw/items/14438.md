## Summary

Keep open. Current main still treats plugin runtime changes as restart-required, exposes no plugin reload or cache-clear CLI/RPC path, and the maintainer discussion explicitly left this for plugin-loader and Gateway lifecycle design rather than cleanup closure.

## What This Changes

Add a supported plugin development hot-reload or cache-clear workflow so TypeScript plugin changes can invalidate Jiti/module state and refresh plugin registrations without restarting the Gateway/container.

## Best Possible Solution

Define a supported plugin development reload path in the plugin loader and Gateway lifecycle model: invalidate Jiti/module/runtime caches, safely replace active plugin registrations, dispose stale lifecycle resources, expose an additive CLI/RPC UX such as `openclaw plugins reload <id>` or `openclaw plugins clear-cache`, and document the container/cache/security behavior. If live runtime reload remains unsupported, keep restart-only behavior documented with accurate cache guidance.
