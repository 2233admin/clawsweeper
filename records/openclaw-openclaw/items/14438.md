## Summary

Current main still leaves the requested developer workflow unresolved: native plugin runtime changes remain documented as restart-only, the `openclaw plugins` CLI exposes no reload or clear-cache command, and Gateway reload machinery handles config/channel/secrets paths rather than supported Jiti/module invalidation plus live plugin re-registration. The existing maintainer comment also explicitly kept this open for design follow-up.

## Best Possible Solution

Keep this open as core plugin loader and Gateway lifecycle developer-UX work. The best path is a maintainer-approved design for `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache`, backed by a Gateway method that invalidates loader/Jiti/module state, safely disposes plugin-owned runtime registrations/resources, re-registers tools/hooks/providers/services, and documents Docker/cache-location behavior. If native plugin runtime hot reload remains intentionally unsupported, the CLI and plugin docs should clearly document the supported restart/cache workflow instead.

## Risks / Open Questions

- A cache-clear-only command could remove Jiti files while leaving stale live Gateway registrations, hooks, tools, providers, services, and plugin-owned resources in memory.

- Docs are not perfectly aligned: `docs/concepts/context-engine.md` mentions `dispose()` being called during plugin reload, while the plugin docs say native plugin runtime hot reload is unsupported. A maintainer design should reconcile that wording.
