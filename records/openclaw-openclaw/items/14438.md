## Summary

Keep open. Current main still documents native plugin runtime changes as restart-only, the plugin CLI has no reload or cache-clear command, and plugin config changes still plan a Gateway restart rather than a live plugin swap. The newer config debounce support is adjacent watcher behavior, not the requested plugin-code reload workflow, and a maintainer comment already explicitly kept this open for plugin loader and Gateway lifecycle design.

## What This Changes

The issue requests a supported plugin development reload or cache-clear workflow so TypeScript plugin edits can take effect without restarting the OpenClaw process/container.

## Best Possible Solution

Design a supported plugin development reload workflow backed by a Gateway/API path that invalidates loader, Jiti, and module state and safely swaps plugin registrations; expose it through `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache`, and update docs. If live runtime reload remains out of scope, keep restart-only behavior documented with precise container/cache guidance.
