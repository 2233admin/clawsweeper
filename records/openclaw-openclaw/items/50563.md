## Summary

Keep open. Current main still has the reported WhatsApp ack-before-process shape: accepted inbound messages are marked read before enrichment, inbound dedupe claim/commit, debounced `onMessage` delivery, or any durable inbound replay boundary. Ryce's KeepMyClaw comment is useful complementary crash-state recovery context, but it does not replace a bundled WhatsApp plugin ordering or persistence fix.

## What This Changes

The issue requests WAL, persist-before-ack, or delayed-ack behavior so a WhatsApp gateway crash cannot silently lose an inbound message after OpenClaw has acknowledged it.

## Best Possible Solution

Fix the bundled WhatsApp plugin so accepted inbound messages are either durably recorded for replay before read receipt or only marked read after successful `onMessage` delivery and inbound dedupe finalization, with docs and focused tests covering the new ordering.
