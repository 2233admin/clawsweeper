## What This Changes

The issue asks OpenClaw to prevent WhatsApp gateway message loss when a crash occurs after a read receipt is sent but before inbound processing or durable storage completes.

## Best Possible Solution

Fix the bundled WhatsApp plugin so accepted inbound messages are either durably recorded for replay before `readMessages` or only marked read after successful `onMessage` delivery and inbound dedupe finalization, with focused tests and WhatsApp docs updated for the new acknowledgment boundary.

## Repair Work Prompt

Fix the WhatsApp ack-before-process reliability bug tracked by https://github.com/openclaw/openclaw/issues/50563. In the bundled WhatsApp plugin, do not send `sock.readMessages` for accepted inbound messages before they are durably recorded for replay or have successfully reached `onMessage` and inbound dedupe finalization. Prefer a narrow plugin-local delayed-read-receipt change over a broad core WAL unless a small durable helper is already available and clearly fits. Preserve self-chat and blocked-sender privacy behavior, keep append/offline-history behavior intentional, and avoid changing unrelated Baileys socket timing, auth, reconnect, or provider surfaces. Update focused WhatsApp monitor/dedupe tests so crashes or handler failures before successful delivery do not mark messages read, successful delivery does mark them read when enabled, disabled read receipts still skip, and stale append behavior is explicitly covered. Update `docs/channels/whatsapp.md` and add a changelog entry if behavior changes user-visible read receipt semantics. Run the targeted tests and formatting checks listed in `workValidation`; before handoff, run `pnpm check:changed` in Testbox if code or docs changed.
