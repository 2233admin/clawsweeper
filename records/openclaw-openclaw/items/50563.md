## Summary

Keep open. Current main still sends WhatsApp read receipts for accepted inbound messages before enrichment, dedupe claim, enqueue, and agent processing. The WhatsApp inbound dedupe is memory-only in this path, while the write-ahead queue I found is outbound/session-delivery only. Ryce’s KeepMyClaw comment is useful complementary recovery context, but snapshots do not replace a core persist-before-ack/read-receipt path for accepted WhatsApp inbound turns.

## Best Possible Solution

Keep this issue open and design a WhatsApp inbound durability boundary: persist accepted inbound messages before sending read receipts or delay read receipts until processing has reached a durable/replayable point. The implementation should cover notify and append behavior, debounce, dedupe release/commit, media metadata, ack reactions, docs, and targeted WhatsApp monitor tests. Keep KeepMyClaw-style state snapshots as complementary recovery context, not the core fix for read-receipt ordering.
