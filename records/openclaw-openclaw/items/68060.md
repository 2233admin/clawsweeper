## Summary

Keep open. Current main still has only the existing heuristic memory-lancedb auto-capture path, so this feature PR is not obsolete; however the proposed patch has blocking integration and LLM-safety issues that need maintainer-directed rework before merge.

## What This Changes

This PR adds an opt-in `smartExtraction` config path, a new LLM-backed smart extractor, parser tests, and an auto-capture branch for atomic memory extraction in the bundled `memory-lancedb` plugin.

## Best Possible Solution

Keep the PR open and rework it in the memory-lancedb plugin: add the matching manifest, UI hint, and docs surface; gate candidate text with the same capture safety contract before any LLM call; escape and frame stored memories as untrusted prompt data; make extractor failures observable or fall back safely; and add focused config, hook, parser, and security tests for the opt-in path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P1] Sync smartExtraction with the manifest schema:** `extensions/memory-lancedb/config.ts:139`
  - body: The runtime parser now accepts `smartExtraction`, but the plugin manifest remains strict and does not declare that property. Because the loader validates user config against the manifest before plugin runtime code receives it, enabling the new option will fail config validation and skip the plugin instead of activating smart extraction.
  - confidence: 0.93
- **[P1] Filter texts before invoking smart extraction:** `extensions/memory-lancedb/index.ts:633-638`
  - body: The smart-extraction branch passes every collected user text to `extractMemories` before applying `shouldCapture` or `captureMaxChars`. That bypasses the existing prompt-injection and length gates, so messages the current path rejects can be sent to the extraction model and stored when this option is enabled.
  - confidence: 0.94
- **[P1] Escape existing memories before prompting the extractor:** `extensions/memory-lancedb/smart-extractor.ts:109-110`
  - body: `existingMemories` are previously captured user-controlled strings, but this prompt appends them verbatim as instructions-adjacent text. A stored memory containing instruction-like content can steer extraction or deduplication, while the existing recall path already escapes and labels memories as untrusted.
  - confidence: 0.9
- **[P2] Report extraction failures instead of empty success:** `extensions/memory-lancedb/smart-extractor.ts:202-203`
  - body: Catching every chat-completion error and returning an empty result makes auth, model, endpoint, and transient provider failures look like a normal no-op. In production this can silently disable smart auto-capture with no warning or fallback for the operator.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The PR introduces a new LLM network path for memory capture that currently bypasses existing capture safety gates and reuses stored memory text as untrusted prompt input without escaping.

Concerns:

- **[high] Capture filters bypassed before external LLM call:** `extensions/memory-lancedb/index.ts:633`
  - body: When `smartExtraction.enabled` is true, the PR sends collected user text to the extractor without `shouldCapture` or `captureMaxChars`, so prompt-injection-like or oversized content rejected by current main can leave the process and be persisted as memory.
  - confidence: 0.94
- **[high] Stored memories are injected verbatim into extractor prompt:** `extensions/memory-lancedb/smart-extractor.ts:110`
  - body: Existing memories can be created from prior user input or the memory_store tool. Appending them directly to the extractor prompt creates a prompt-injection path through durable memory.
  - confidence: 0.9

## Risks / Open Questions

- Merging as-is would likely reject user `smartExtraction` config because the strict plugin manifest schema does not include the new property.
- Merging as-is would bypass existing `shouldCapture` and `captureMaxChars` gates before a new external LLM call, expanding the amount and type of user content sent to the extractor.
- Stored memories are user-controlled data; inserting them verbatim into the extractor prompt creates a prompt-injection route that current recall code already avoids by escaping and labeling memory text as untrusted.
- Extractor provider failures currently become indistinguishable from “nothing to remember,” making smart auto-capture outages hard to diagnose.
- The PR branch is behind current main’s memory-lancedb config/runtime shape and needs a careful rebase so provider auth, recall limits, and cursor behavior are preserved.
