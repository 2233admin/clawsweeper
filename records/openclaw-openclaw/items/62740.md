## Summary

Keep open. The underlying Zalo bug is real on current main, but the latest PR diff still introduces a concrete security/reliability issue by buffering the full non-2xx response body and then using remote body text as the thrown error message that can reach logs.

## What This Changes

The branch modifies `extensions/zalo/src/api.ts` so `ZaloApiError` can carry an HTTP status and `callZaloApi` checks `response.ok`, preserves structured Zalo JSON errors when possible, truncates non-JSON bodies, and separates HTTP 408 from app-level polling timeout 408.

## Best Possible Solution

Land a revised Zalo API client guard that caps and sanitizes HTTP error bodies before logging or surfacing them, keeps structured Zalo JSON errors intact, distinguishes HTTP status from app-level error codes, and adds focused `api.test.ts` coverage for non-JSON HTTP errors, non-2xx structured JSON errors, and HTTP 408 behavior.

## Reproduction Assessment

Yes. A focused injected `ZaloFetch` returning `new Response('<html>bad gateway</html>', { status: 502 })` is enough to show current main reaching `response.json()` and throwing a parse error instead of `ZaloApiError`; a large text body on the PR branch would exercise the unbounded `response.text()` path.

## Solution Assessment

No. The requested direction is right, and the second commit addresses structured errors and HTTP 408 disambiguation, but the implementation should not buffer an arbitrary error body or put remote body text directly into the standard error message/log path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Cap the HTTP error body before buffering:** `extensions/zalo/src/api.ts:126`
  - body: `response.text()` reads the entire non-2xx response before the code truncates it to 200 characters. A large HTML/proxy error page can still be fully buffered and stall or bloat the gateway, so read a bounded stream/snippet or skip the body based on size before constructing the error.
  - confidence: 0.86
- **[P2] Keep remote body text out of logged error messages:** `extensions/zalo/src/api.ts:143-145`
  - body: For non-JSON HTTP errors, the truncated remote body becomes `ZaloApiError.message`. `formatZaloError()` logs `error.stack`/`error.message` in polling and startup paths, so upstream body text and control characters can enter logs; use a stable local message and store only a sanitized snippet separately if needed.
  - confidence: 0.78

## Security Review

Status: needs_attention
