## Summary

Current main still does not restore Control UI abortability after a refresh during an active stream. The prior ClawSweeper comment identified the missing hydration path, and the current 5de284c main tree still clears `chatRunId` on hello/reconnect while neither hello nor `chat.history` returns an active run snapshot for the UI to restore before rendering the controls.

## Best Possible Solution

Keep this issue open. Add an authoritative active-run hydration path for Control UI on page load/reconnect, likely via hello/bootstrap or `chat.history`, returning the active run id plus buffered stream state for the visible session. The UI should set `chatRunId`, `chatStream`, and `chatStreamStartedAt` before rendering the toolbar, and tests should cover refresh during active streaming plus terminal-event cleanup.

## Risks / Open Questions

- Closing now would leave refreshed Control UI tabs showing a non-abortable toolbar during an active run, inviting users to start a new session or send as if no stream is active.
- A minimal UI-only fix that adopts any later delta run id should still account for session matching, ownership/authorization, and terminal event ordering so it does not make the wrong run abortable.
