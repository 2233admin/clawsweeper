## Summary

Current main still does not restore the Control UI active chat-run identity after refresh/reconnect during streaming. The toolbar remains keyed to `chatRunId`, reconnect clears that state, `chat.history` and hello snapshots do not hydrate an active run, and incoming deltas update stream text without adopting the run id.

## Best Possible Solution

Keep this issue open. Add an authoritative active Control UI chat-run hydration path on page load/reconnect, likely through hello/bootstrap or `chat.history`, returning the active run id plus buffered stream state for the visible session. The UI should set `chatRunId`, `chatStream`, and `chatStreamStartedAt` before rendering controls, and tests should cover refresh during an active stream plus terminal-event cleanup.

## Risks / Open Questions

- A minimal UI-only fix that adopts any matching-session delta run id could make the wrong run abortable if it does not account for session, ownership/authorization, and terminal event ordering.
- Until hydration exists, refreshed Control UI tabs can show a stale toolbar during an active stream, encouraging users to start a new session or send as if no run is active.
