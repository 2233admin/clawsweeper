## Summary

Keep open. Current main still does not implement the PR's intended broadcast `runInboundClaim` call for regular, non-plugin-bound messages; the code only invokes the targeted plugin-bound variant, and the dispatch tests still assert the old no-broadcast behavior.

## Best Possible Solution

Keep this PR open as useful but incomplete implementation work. The best path is to refresh it against current main, add broadcast `runInboundClaim` for regular non-plugin-bound messages, preserve plugin-owned binding and fast-abort precedence, call `markIdle` on claimed early return, and replace the old no-broadcast assertion with targeted claimed/unclaimed regression tests.

## Risks / Open Questions

- Functional merge risk remains: inserting broadcast claim before the existing fast-abort path could let a blanket `inbound_claim` handler swallow documented `stop`/`abort` messages unless abort precedence is preserved and tested.
- Functional merge risk remains: broadcasting after plugin-bound missing-plugin/no-handler fallback could let an unrelated plugin claim a turn that should fall back to OpenClaw.
- Test risk remains: current main still has a no-broadcast assertion and lacks focused regular-message claimed/unclaimed dispatch-path tests.
- Security and supply-chain risk appears low from the provided diff because it changes one TypeScript runtime dispatch file and does not touch dependencies, CI, release scripts, secret handling, or package metadata.
