## Summary

Keep this PR open. Current main still does not implement WhatsApp inbound reaction forwarding: the WhatsApp monitor has no `messages.reaction` subscription, `WebInboundMessage` has no reaction metadata fields, and source/docs/tests do not show a shipped inbound reaction path. The PR remains a plausible focused implementation candidate, with overlap to #48570 and the open user-facing report #57288.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. Rebase it onto current main, reconcile it with #48570 and #57288, then land a WhatsApp-owned `messages.reaction` path with targeted tests and explicit access-control, pairing-suppression, dedupe, status-JID filtering, and listener-cleanup behavior.

## Risks / Open Questions

- The PR has no regression tests in its changed files, while the behavior needs coverage for DM/group/status filtering, reaction removal/re-apply dedupe, access control, pairing suppression, self reactions, raw-JID rejection, and listener cleanup.
- The PR overlaps broader open work in #48570 and the open user-facing bug #57288, so maintainers should choose one implementation path or extract the focused reaction pieces to avoid duplicate handling.
- Prior review context identifies at least one remaining correctness edge: removing then re-applying the same reaction within the dedupe TTL can be silently dropped unless the dedupe cache is cleared or keyed differently.
