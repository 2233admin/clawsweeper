## Summary

Keep this PR open, but do not merge it as-is. Current main still does not forward WhatsApp inbound reaction events, while the proposed handler conflicts with the Baileys reaction event contract and has a removal/re-reaction dedupe bug; there is also overlapping open work in #48570 that a maintainer should choose between.

## What This Changes

The PR adds WhatsApp `messages.reaction` handling, forwards non-empty reactions as `WebInboundMessage` objects with reaction metadata, moves one inbound activity metric call, and adds reaction fields to the inbound message type.

## Best Possible Solution

Land one WhatsApp-owned inbound reaction implementation that follows the Baileys event contract, routes accepted reactions through the existing inbound access-control and dedupe patterns, handles removal/re-reaction behavior explicitly, cleans up listeners through the current lifecycle helper, and adds focused tests plus docs/changelog coverage if the agent-visible contract changes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Read the reactor from `reaction.key`:** `extensions/whatsapp/src/inbound/monitor.ts:453-455`
  - body: Baileys emits `messages.reaction` as `{ key, reaction }`, where the top-level `key` is the reacted-message key and the reaction event key is on `reaction.key`; there is no top-level `userJid`. Because this handler reads `userJid` and never reads `reaction.key`, group reactions without `userJid` hit the `group && !reactorJid` guard and are never forwarded.
  - confidence: 0.89
- **[P2] Clear reaction dedupe entries on removals:** `extensions/whatsapp/src/inbound/monitor.ts:442-443`
  - body: Reaction removals have a falsey `reaction.text`, so this early `continue` runs before the dedupe key is computed or cleared. The previous reaction remains cached for the 20-minute window, and re-applying the same emoji to the same message can be silently dropped.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The diff does not add supply-chain surfaces, but it introduces agent input from WhatsApp reactions with a concrete actor-identity/access-control concern.

Concerns:

- **[medium] Reaction actor identity can be misclassified:** `extensions/whatsapp/src/inbound/monitor.ts:453`
  - body: The handler derives the reactor from a top-level `userJid` that Baileys does not emit for `messages.reaction`; it should use the reaction event key. Misclassification affects whether a reaction is treated as self/outbound, allowed, blocked, or attributed to a group sender.
  - confidence: 0.86
