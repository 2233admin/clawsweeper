## Summary

Keep open. Current main still classifies orphan transcripts by absence from the active `sessions.json` map, so valid older primary transcripts remain eligible for the interactive `.deleted.<timestamp>` archive path; the recent archived-discovery and interactive-confirmation mitigations reduce damage but do not fix the classifier.

## What This Changes

The issue requests safer `doctor --fix` handling so historical session JSONL transcripts are not classified and archived as orphans, with follow-up improvements for prompt wording, recovery, corrupt-store handling, and CI coverage for doctor recommendations.

## Best Possible Solution

Define a real orphan transcript in the doctor state-integrity path instead of using absence from the active session-key map. A focused first change should preserve valid historical primary transcripts, keep sidecar/archive exclusions, and add regression coverage; prompt wording, restore tooling, corrupt-store warnings, and CI hardening should be split into scoped follow-ups.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: create a sessions directory with one active `sessions.json` entry and an older valid `<uuid>.jsonl` absent from that store, then `noteStateIntegrity` will classify the older primary transcript as orphan because it is missing from `referencedTranscriptPaths`.

## Solution Assessment

No. The current interactive-confirmation mitigation is safer than silent repair, but it does not solve the definition bug; the narrow maintainable fix is to change the classifier and tests first, then handle UX and CI hardening separately.
