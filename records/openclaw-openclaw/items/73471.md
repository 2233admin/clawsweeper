## Summary

Keep open. Current main still treats any primary transcript absent from the active session-store map as an orphan candidate; the shipped mitigation only prevents non-interactive or auto-approved repair from renaming those files, so the classifier, prompt wording, restore path, corrupt-store behavior, and CI-hardening asks are not fully resolved.

## What This Changes

The issue reports that `openclaw doctor --fix` can misclassify historical primary session transcripts as orphan files and asks for safer orphan classification plus broader hardening of doctor repair recommendations.

## Best Possible Solution

Define a real orphan transcript in the doctor state-integrity path instead of using absence from the active session-key map. The first maintainable change should preserve valid historical primary transcripts and add focused regression coverage; prompt wording, restore support, corrupt-store diagnostics, and doctor recommendation CI hardening can then be split into scoped follow-ups.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: create a sessions directory with one active `sessions.json` entry and an older valid `<uuid>.jsonl`; current `noteStateIntegrity` will reference only the active store entry and offer to archive the older transcript.

## Solution Assessment

Unclear for the full issue. The current v2026.4.27 mitigation is a useful safety improvement, but the narrow best first fix is changing the orphan definition and tests rather than relying on interactive confirmation wording.
