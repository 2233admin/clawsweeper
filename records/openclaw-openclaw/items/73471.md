## Summary

Keep open. Current main still treats any primary transcript absent from the active session-store map as an orphan candidate, and the prompt still describes the repair as safe archival even though this can rename valid historical chat transcripts; recent changes are useful mitigations, not the classifier fix requested here.

## What This Changes

The issue asks OpenClaw to stop `openclaw doctor --fix` from classifying valid historical session transcripts as orphan files and to harden doctor repair recommendations for user-content safety.

## Best Possible Solution

Define a real orphan transcript in the doctor state-integrity owner path instead of using absence from the active session-key map. A focused first fix should preserve valid historical primary transcripts, keep sidecar exclusions, add regression coverage with an older real transcript, and then split prompt wording, restore support, corrupt-store warnings, and doctor recommendation CI hardening into maintainer-scoped follow-ups as needed.

## Reproduction Assessment

Yes. A focused source-level reproduction is clear: create a sessions directory with one active `sessions.json` entry plus an older valid `<sessionId>.jsonl`, then `noteStateIntegrity` will classify the older file as orphan because it is primary JSONL and absent from `referencedTranscriptPaths`.

## Solution Assessment

Unclear for the full issue, because the requested end state mixes classifier correctness, user-facing prompt semantics, restore tooling, store-load diagnostics, and CI policy. The best first solution is the narrow classifier/test fix, with the broader doctor hardening split after maintainer scoping.
