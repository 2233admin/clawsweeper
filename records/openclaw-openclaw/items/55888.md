## Summary

Keep this issue open. Current main supports the reported tuning knobs and documents the relevant `agents.defaults.compaction` surface, but it still keeps the conservative default reserve floor and 4k memory-flush buffer that produce the reported low threshold on 32k local Ollama models. The remaining product question is whether OpenClaw should add an opt-in high-memory local-model profile, clearer docs, or hardware-aware compaction defaults.

## Best Possible Solution

Keep the issue open as a concrete enhancement. The best maintainer path is to add a documented high-memory local Ollama compaction profile, or an explicit opt-in setting/preset that lowers `reserveTokensFloor` and explains the 4k `memoryFlush.softThresholdTokens` buffer. If product wants automation, implement hardware-aware compaction policy behind a conservative opt-in and add tests around the threshold calculation and config precedence.

## Risks / Open Questions

- Changing the global default reserve floor could reduce headroom for tool-heavy or smaller-context sessions, so the safer path is likely an opt-in local-model/high-memory profile or clearer docs rather than a blanket default change.
- The issue's performance numbers are hardware- and model-specific; maintainers would need live validation or a documented support boundary before promising M4 Pro or 64GB behavior.
