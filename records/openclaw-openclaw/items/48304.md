## Summary

Keep this issue open. Current main has related safeguards, but it still lacks a configurable per-action, per-sender authorization gate for `message.send` or `message.poll`. The trusted requester identity is propagated through parts of the message execution path, yet public `before_tool_call` and `message_sending` hooks do not expose enough trusted requester identity for an external plugin to implement the requested policy reliably. The related guardrail PRs #55053 and #64868 were closed without merging, so they do not supersede this issue.

## Best Possible Solution

Keep this open for a core/API-backed authorization design. A narrow first step would add a `tools.message` policy checked before `send` and `poll`, using server-trusted requester identity and returning a configured denial reason. A broader follow-up should expose a trusted pre-tool policy surface with requester identity, action classification, agent/session/channel/account context, and block or owner-confirmation decisions before tool params execute. Update config types, strict zod schema, generated config docs/help, plugin hook docs, and targeted message-action tests.
