## Summary

Keep open. Current main has auth cooldown and cooldown-probe throttling, but the reported `model-fallback/decision` log spam is not coalesced or throttled: every fallback decision still goes through a direct warning log, and existing tests cover probe throttling rather than repeated decision-log suppression.

## Best Possible Solution

Keep this issue open and implement bounded coalescing or throttling around `logModelFallbackDecision`, keyed by the stable decision/provider/model/reason/error fingerprint and scoped enough to preserve first-occurrence diagnostics. Add focused tests that repeated auth-cooldown fallback runs emit the first warning plus a summarized/suppressed follow-up rather than one warning per loop.
