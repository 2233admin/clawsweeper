## Summary

Keep open. Current main has a useful partial mitigation for fast status output, but the detailed memory-status path can still initialize the active memory plugin locally and probe vector availability; additionally, open PR #62643 uses closing syntax for this issue and should be reviewed or closed before this issue is resolved.

## What This Changes

The issue asks for `openclaw status` to avoid loading the active memory plugin in the CLI process and to avoid reporting memory/vector state from a separate local runtime.

## Best Possible Solution

Resolve the open linked PR first: either land the passive `purpose: "status"` contract if that is the intended product boundary, or replace it with a narrow runtime change that prevents status from reporting local false vector state and adds focused status/memory tests. After that PR is merged or intentionally closed, revisit this issue for closure.

## Reproduction Assessment

Yes for the code path: current source shows the detailed status memory snapshot can still call the active memory manager with `purpose: "status"`, probe vector availability, and initialize the configured memory runtime locally. I did not live-run the reporter's macOS plugin setup in this read-only review.

## Solution Assessment

Unclear as a complete solution. Current main's fast-path skip is a good partial mitigation, but the remaining behavior needs maintainer choice between live gateway-backed status, a documented passive plugin contract, or suppressing local vector probes in status.
