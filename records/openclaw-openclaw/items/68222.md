## Summary

Keep open. Current main still exposes the Kimi Coding provider path, still rewrites Kimi tagged tool-call text into executable tool calls, and still treats `sessions_yield` as an intentional paused/end-turn state. The recent adjacent subagent fix for #73413 preserves legitimate yielded subagent runs, but I did not find source, test, docs, release, or discussion evidence that unwanted Kimi-generated `sessions_yield` calls are ignored or auto-resumed.

## What This Changes

The issue reports that Kimi Code can emit `sessions_yield` during long-running conversations, ending the turn and requiring manual input to continue.

## Best Possible Solution

Keep the issue open and reproduce current main with a captured Kimi Coding transcript around the first `sessions_yield`. If Kimi calls `sessions_yield` without an outstanding child result to wait for, add a guard or recovery path in the Kimi stream wrapper or embedded runner so the root task continues without user nudging, with focused coverage for the Kimi tagged-tool path and the normal subagent-yield path.
