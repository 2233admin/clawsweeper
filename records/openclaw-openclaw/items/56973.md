## Summary

Keep open. Current main still has the reported exported `normalizeChannelId` wrappers with different return types, and current call sites still alias both wrapper names to recover the more precise canonical names. The request is a valid internal cleanup, but it should be handled carefully because the plugin-side wrapper is also re-exported through public plugin SDK subpaths.

## Best Possible Solution

Keep this issue open and handle it as a compatibility-aware refactor: switch internal repo code to `normalizeChatChannelId` and `normalizeAnyChannelId` where those names express the return type, update the plugin barrel exports and tests, and only remove or rename SDK-facing `normalizeChannelId` after a deprecation/compatibility plan for plugin authors.

## Risks / Open Questions

- A direct removal of the plugin-side wrapper may break plugin SDK consumers because `src/plugin-sdk/channel-runtime.ts` and `src/plugin-sdk/channel-targets.ts` currently re-export `normalizeChannelId`, and the SDK subpath contract test names that export.
- The refactor has wider call-site fanout than the issue's narrow example; current main has many imports from `src/channels/plugins/index.ts` and `src/channels/plugins/registry.ts` that still use the ambiguous wrapper name.
