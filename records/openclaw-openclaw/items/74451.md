## Summary

Keep open. Current main has useful partial coverage for this regression, but it is not enough to confidently close: the OpenAI plugin now suppresses the built-in/discovered `openai-codex/gpt-5.4-mini` row, while the model resolver can still resolve a configured inline model row with an `api` before applying that suppression. The issue explicitly calls out a stale configured OpenAI Codex model list, so the reported path is still plausible.

## What This Changes

Reject or safely recover from switching a ChatGPT-backed `openai-codex` session to unsupported `openai-codex/gpt-5.4-mini`, especially from Discord live model-switch flows.

## Best Possible Solution

The OpenAI Codex route should have one provider-owned capability gate that rejects or skips unsupported Codex OAuth models across catalog rows, configured provider model rows, live model switches, and fallback candidates before any assistant turn reaches the vendor transport. Stale overrides should either be cleared/reverted to the prior supported model or fall through to a supported configured fallback with a clear diagnostic.
