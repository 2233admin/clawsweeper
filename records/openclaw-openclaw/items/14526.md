## Summary

Current main still does not implement the proposed opt-in safe-update workflow. OpenClaw has adjacent primitives: manual backup create/verify, update status/run paths, auto-update timing policy, restart/doctor health checks, Control UI update triggering, and manual rollback docs. The missing center remains automatic pre-update backup, retention policy, channel or real-agent smoke verification, and automatic rollback/state restore. The Bortlesboat auto-update kit linked in the discussion is useful prior art, and #65011/#17215 are related open follow-ups, but they do not supersede this issue.

## Best Possible Solution

Keep this open until maintainers either design and implement an opt-in core safe-update mode or explicitly decide the workflow should live outside core. A core solution should build on existing update, backup, doctor, health, channel status, and Control UI primitives; add automatic pre-update backup creation, retention controls, post-update gateway/status/channel/real-agent smoke checks, and controlled downgrade plus state restore; and coordinate scope with #65011, #17215, and the external openclaw-auto-update-kit prior art.

## Risks / Open Questions

- A core implementation would touch backups containing credentials and must preserve permissions while avoiding accidental archive leakage.

- Gateway health alone can miss lane/session failures like the #65035 scenario cited in the discussion, so health criteria need maintainer product design.
- The scope overlaps #65011 and #17215; maintainers should choose the API/UI boundary before implementation.
