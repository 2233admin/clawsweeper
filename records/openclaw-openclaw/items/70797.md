## Summary

Keep open. Current main still has the reported leak class: bare OpenAI-style `tool_calls` JSON is detected only for diagnostics, while the shared visible-text sanitizer and Telegram/direct partial-preview path can still pass it as normal assistant text. The two related fix PRs are closed unmerged and no open canonical fix was found in the provided context or issue search.

## What This Changes

Prevent raw `tool_calls` JSON payloads from appearing in Telegram/direct user-visible replies during background exec/process polling flows.

## Best Possible Solution

Add a shared, streaming-aware guard for bare OpenAI-style `tool_calls` JSON at the assistant-visible boundary, reuse it for final text and partial preview delivery, and preserve ordinary JSON plus fenced/inline code examples.

## Reproduction Assessment

Yes for the leak class: the reported bare JSON shape can be traced through current source because the sanitizer returns it unchanged and the partial-preview path can send resulting text to `onPartialReply`. The exact live Telegram/tinyrouter timing remains not fully minimized.

## Solution Assessment

Yes: the narrow maintainable solution is to suppress internal tool-call payload text before user-visible delivery, not to add provider-specific workarounds or try to execute downgraded assistant text as a tool call.
