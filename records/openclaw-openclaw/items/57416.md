## Summary

Keep open and do not merge as-is. The PR addresses a real linked bug and the discussion refined the original `OPENCLAW_STATE_DIR` redirect problem, but current main has since shipped and documented a fail-closed policy that blocks all `OPENCLAW_*` keys from untrusted workspace `.env` files. Allowing a workspace `.env` to set `OPENCLAW_CONFIG_PATH` would weaken that runtime-control trust boundary.

## What This Changes

The PR removes `OPENCLAW_CONFIG_PATH` from the workspace dotenv blocklist, precomputes the global dotenv path before workspace loading, and updates dotenv tests to allow project-local config while keeping `OPENCLAW_STATE_DIR` blocked.

## Best Possible Solution

Keep the shipped fail-closed workspace dotenv behavior unless maintainers explicitly approve a trusted project-local config mechanism. If project-local config selection is still desired, implement it as a documented opt-in with tests proving workspace files cannot redirect state, credentials, plugin roots, runtime `.env` loading, or other control-plane paths.

## Reproduction Assessment

Yes. Current-main source and tests give a high-confidence reproduction path: put `OPENCLAW_CONFIG_PATH=...` in a workspace `.env`, load it through `loadWorkspaceDotEnvFile`, `loadDotEnv`, or `loadCliDotEnv`, and the key remains unset. I did not run tests because this review is read-only.

## Solution Assessment

No. This PR is not the best way to solve the linked bug because it adds a runtime-control exception to a later shipped fail-closed dotenv policy. A safer solution needs explicit maintainer/security direction, likely shell/global-env documentation or a trusted opt-in mechanism.
