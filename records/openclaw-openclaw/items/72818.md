## Summary

Keep open: the PR addresses a real configured-channel startup gap, but the current diff still bypasses plugin activation policy and can start disabled or denied bundled channel plugins. The fallback path also does not ensure runtime/log support for channels that were absent from the loaded registry.

## What This Changes

The PR adds a configured-channel fallback to gateway channel startup, records context trigger data in diagnostic stability output, adds QA-channel thread coverage, and includes small test/CLI cleanup adjustments.

## Best Possible Solution

Revise the PR so fallback startup candidates come from the policy-approved configured-channel owner set, preferably the existing startup activation plan or `resolveConfiguredChannelPluginIds`-style policy helper, and ensure any fallback-started channel has the same logger/runtime support as registry-started channels. Add regression coverage for global plugin disablement, denylist, per-plugin disablement, disabled channel config, and one allowed configured bundled channel.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a config with a bundled channel under `channels` plus `plugins.enabled=false`, a deny entry, or `plugins.entries.<id>.enabled=false`: the PR’s raw channel-key loop can still resolve the bundled plugin through `getChannelPlugin()`. I did not run tests because this review is required to keep the checkout read-only.

## Solution Assessment

No. Starting configured channels absent from the loaded registry is the right product direction, but raw config-key enumeration is not the narrow maintainable fix because it bypasses current plugin policy and supporting startup state.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Gate fallback startup through plugin policy:** `src/gateway/server-channels.ts:692-696`
  - body: This loop starts every raw configured channel id via `getChannelPlugin()`, whose bundled fallback is not policy-aware. When the active registry omitted a plugin because `plugins.enabled=false`, a denylist, or `plugins.entries.<id>.enabled=false`, this can still resolve bundled code and start it despite explicit disablement. Derive fallback candidates from the existing policy-approved configured-channel owner set before direct lookup.
  - confidence: 0.9
- **[P2] Seed runtime state for fallback-started channels:** `src/gateway/server-channels.ts:693-695`
  - body: These fallback plugins are specifically absent from `listChannelPlugins()`, but the Gateway builds `channelLogs` and `channelRuntimeEnvs` from that registry before the manager runs. Starting such a plugin here can hand `undefined` for `ctx.runtime`/`ctx.log`, so the absent-registry path can fail or run without required startup context. Load the approved plugin into the registry first or provide lazy logger/runtime entries for fallback channels.
  - confidence: 0.8
