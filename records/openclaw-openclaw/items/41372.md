## Summary

Keep this issue open. Current main has meaningful partial fixes for config validation/recovery, Discord timeout tuning, cron staggering/concurrency, and detached update restarts, but the umbrella report still has concrete unresolved cron documentation and operator ergonomics gaps. In particular, the public cron guide still says plain `openclaw cron list` lists all jobs while the CLI/service default is enabled-only unless `--all`/`includeDisabled` is set, and `cron add` remains create-only rather than an idempotent setup/upsert path.

## Best Possible Solution

Keep this issue open for maintainer triage. Fix the cron docs so `openclaw cron list` is documented as enabled-only by default with `openclaw cron list --all` for disabled jobs, document the `jobs`/`entries` JSON wrapper shapes where appropriate, and decide whether cron needs an idempotent create/upsert command for setup scripts. Then split any still-current Discord receive, isolated web-search, onboarding, update, and small-VPS resource observations into focused issues with fresh reproduction data.
