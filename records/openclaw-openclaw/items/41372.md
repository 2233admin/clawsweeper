## Summary

Keep this issue open. Current main has improved parts of the field report, especially config schema/doctor recovery and cron tool guidance, but the issue is not cleanly closeable: cron docs still say plain `openclaw cron list` lists all jobs while CLI/service behavior defaults to enabled-only unless `--all`/`includeDisabled` is used; cron JSON wrapper shapes remain underdocumented for scripting; and `cron add` is still create-only with no idempotent setup/upsert path. The newer #68015 overlaps the operator field-report theme but tracks different remaining work, so it is not a canonical replacement for this issue.

## Best Possible Solution

Keep this issue open for maintainer triage. First fix the cron documentation so plain `openclaw cron list` is documented as enabled-only with `openclaw cron list --all` for disabled jobs, document the `jobs` and `entries` JSON wrapper shapes for cron scripting, and decide whether cron needs an idempotent setup/upsert command or a documented safe setup pattern. Then split or link narrower follow-ups for any still-current Discord receive, isolated web-search, onboarding-token, update-process, and small-VPS resource findings.
