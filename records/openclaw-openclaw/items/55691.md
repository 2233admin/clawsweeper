## Summary

Keep open. Current main already has run-cancellation primitives (`/stop`, `chat.abort`, `sessions.steer`, and queue `interrupt`/`steer` modes), but I did not find the requested platform message recall/edit lifecycle path that correlates an edited/deleted inbound message to the active run and cancels or reruns it. The plugin hook surface models `message_received`, `message_sending`, and `message_sent`, while inspected Telegram, Discord, and Feishu handlers do not consume edit/delete/recall events as cancellation triggers. Related #52993 is useful Feishu recall/delete action prior art, but it is not a merged broad cancel-on-recall implementation for channel runs.

## Best Possible Solution

Keep this open as a core/channel lifecycle feature. The best path is to add a typed inbound message lifecycle event contract for edit/delete/recall in the plugin SDK, correlate platform message IDs to session/run state, route authorized recall/delete events to the existing `chat.abort` or session abort path, and add channel-specific implementations and contract tests for Telegram, Discord, Feishu, and any other bundled channel that exposes reliable events. Treat edit as cancel-and-rerun first unless maintainers explicitly want live transcript mutation.

## Risks / Open Questions

- Implementing this needs strict sender/origin correlation so another participant or unrelated platform delete event cannot abort the wrong run.
- Message edit support should define transcript semantics carefully: cancel-and-rerun is safer initially than mutating already-consumed prompt history mid-run.
