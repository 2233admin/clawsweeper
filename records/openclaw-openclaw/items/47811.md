## Summary

Keep this issue open. Current main supports named profile homes, isolated multi-gateway docs, and partial config-drift diagnostics, but it still does not provide the requested authoritative multi-profile target report or a clear custom-profile casing policy for macOS service targeting.

## Best Possible Solution

Keep this open and narrow follow-up to core CLI/gateway behavior: define the supported custom-profile casing policy, fail loudly or migrate safely when casing is ambiguous, make status/restart/agent surfaces report the resolved profile, config path, state directory, service label/process boundary, port/probe target, and session store path, and add macOS LaunchAgent tests for multiple named homes and case-mismatched profile names.
