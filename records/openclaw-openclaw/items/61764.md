## Summary

Keep open. Current main still has multiple source-backed instances of the reported channel dashboard behavior, including inline raw snapshot stringification, eager per-channel config schema rendering, per-render channel sorting/lookup work, and relative timestamp formatting without a shared UI tick. The report is concrete and recent, but it is too broad for automatic closure or a single conservative cleanup PR because it mixes quick perf fixes with a product-level UX redesign.

## What This Changes

The issue requests a performance and UX overhaul of the Control UI channel dashboard, including deferring raw JSON/schema rendering, shared timestamp updates, memoized lookups/sorts/schema analysis, search/filter/status-oriented layout, and better first-run/accessibility states.

## Best Possible Solution

Split the work into a narrow performance PR first, then a separate UX design pass. The first PR should hide/defer the raw snapshot, avoid eager schema analysis/rendering for collapsed channel config, hoist channel metadata/account/recent-activity derivations per render, and add focused tests around those render behaviors; the end-user dashboard redesign should follow with maintainer-approved status grouping, search/filter, first-run states, and docs links.

## Reproduction Assessment

Yes for the source-level behavior: rendering the Channels tab with any non-null snapshot still serializes the full snapshot and walks config schema/form nodes during render. No live profile proves the exact <16ms acceptance target yet, but the cited render paths are high-confidence reproduction anchors.

## Solution Assessment

No, not as one monolithic issue. The requested direction is valid, but the best maintainable path is to split quick render-path fixes from the broader dashboard UX redesign and validate each with focused UI tests or profiling.
