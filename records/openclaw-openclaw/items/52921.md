## Summary

Keep open. Current main still does not implement the intended missing-session recovery, and the linked bug remains valid, but this PR should not merge as-is because it introduces blocking correctness and security issues and mixes unrelated cron scheduler changes into the session fix.

## What This Changes

The PR adds structured session-miss error codes, session auto-bootstrap, a new `sessions.start` gateway method and `sessions_start` agent tool, `sessions_send` message tracking/read receipts, generated Apple/tool-display updates, and cron scheduler fallback/force-run changes.

## Best Possible Solution

Land a focused session recovery fix for the linked bug that defines the missing-session contract, preserves current visibility/A2A/sandbox policy, updates protocol schemas/generated clients/tool catalogs/docs/tests together, and moves cron scheduler changes into a separate review path.

## Reproduction Assessment

Yes. The report has concrete steps, and current main statically reproduces the core failure path: sending to a configured agent session with no stored `sessionId` reaches `handleSessionSend` and returns `INVALID_REQUEST`/`session not found` instead of bootstrapping or returning a structured session-miss code.

## Solution Assessment

No. The PR targets the right regression, but this is not the best mergeable solution because it changes gateway protocol, agent tool policy, generated clients, session bootstrap behavior, read-receipt flow, and cron scheduling while leaving blocking correctness and security gaps.

Full review comments:

- **[P1] Use the repo's TypeBox package:** `src/agents/tools/sessions-start-tool.ts:1`
  - body: The new tool imports `@sinclair/typebox`, but this repo depends on `typebox` and the PR does not add `@sinclair/typebox` to package metadata. Loading this file will fail module resolution.
  - confidence: 0.94
- **[P1] Don't redirect missing named sessions to main:** `src/gateway/server-methods/sessions.ts:408-410`
  - body: When `sessions.send` misses an agent-scoped key, this branch bootstraps the agent's main session and continues with that canonical key. A missing key like `agent:ops:incident-123` would be delivered into `agent:ops:main`, merging unrelated conversations instead of returning `SESSION_NOT_FOUND`.
  - confidence: 0.91
- **[P1] Allocate a fresh id before bootstrapping dead sessions:** `src/gateway/sessions-bootstrap.ts:119-122`
  - body: `applySessionsPatchToStore()` preserves an existing store entry. If the dead main entry exists without `sessionId`, this patch does not allocate a new id before transcript creation, so bootstrap can fail and roll back the saved entry instead of recovering the session.
  - confidence: 0.88
- **[P2] Enforce visibility policy before sessions_start:** `src/agents/tools/sessions-start-tool.ts:39-45`
  - body: `sessions_start` forwards any `agentId` directly to the gateway and returns session metadata without the visibility, sandbox, and A2A checks used by `sessions_send`. Restricted agents can probe or start sessions outside their allowed surface.
  - confidence: 0.92
- **[P2] Classify sessions.start under write scope:** `src/gateway/server-methods-list.ts:70`
  - body: Adding `sessions.start` to the public method list without adding it to `method-scopes.ts` leaves it unclassified. Non-admin clients with normal `operator.write` scopes will hit the default `operator.admin` requirement when calling the new API.
  - confidence: 0.9
- **[P2] Register sessions_start in profile catalogs:** `src/agents/openclaw-tools.ts:263-265`
  - body: `createOpenClawTools` exposes the new tool, but profile filtering is driven by `CORE_TOOL_DEFINITIONS`. Without a catalog entry, normal coding or messaging profiles can filter out the recovery tool before execution.
  - confidence: 0.86
- **[P1] Keep the cron fallback near the next occurrence:** `src/cron/schedule.ts:126-128`
  - body: Changing the fallback anchor to `now + 48h` can skip the next daily occurrence whenever this recovery path is used. Existing regression coverage documents this 48-hour jump failure mode, so the fallback should stay anchored near the next day.
  - confidence: 0.87
- **[P1] Catch schedule failures in forced cron preflight:** `src/cron/service/ops.ts:542-543`
  - body: Calling `computeJobNextRunAtMs()` directly during force preflight can throw for malformed schedules before the manual run is enqueued. The maintenance path catches those failures and records schedule errors, so forced runs should use the same recoverable behavior.
  - confidence: 0.84
- **[P2] Restore prior entries on bootstrap rollback:** `src/gateway/sessions-bootstrap.ts:138-140`
  - body: When transcript creation fails, rollback unconditionally deletes the target store entry. In force or replacement flows, a transient filesystem error can erase a previously valid session record instead of preserving it.
  - confidence: 0.82

## Security Review
