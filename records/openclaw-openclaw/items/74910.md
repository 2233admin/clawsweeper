## Summary

Keep open. Current main still matches the report: the doctor migration detects `agents.defaults.llm`, but the fix path deletes the block without carrying or echoing `idleTimeoutSeconds`, and the active model idle-timeout resolver only honors provider request timeout, explicit run timeout, `agents.defaults.timeoutSeconds`, or the 120s default.

## What This Changes

The issue asks for `openclaw doctor --fix` to preserve or clearly echo the legacy `agents.defaults.llm.idleTimeoutSeconds` value and for runtime startup to warn when that legacy config remains present.

## Best Possible Solution

Update the doctor migration so `idleTimeoutSeconds` is not silently lost, preferably by echoing the exact legacy value and detected candidate providers in the fix output, with an optional single-provider auto-apply only when unambiguous. Add a one-shot runtime/config-load warning for source configs that still contain `agents.defaults.llm`, and cover both paths with focused tests plus a changelog entry.

## Reproduction Assessment

Yes. A high-confidence source reproduction is the existing migration test shape with `agents.defaults.llm.idleTimeoutSeconds`, which currently deletes the block, combined with the idle resolver path that falls back to 120s when `models.providers.<id>.timeoutSeconds` is absent.

## Solution Assessment

Yes, with a conservative boundary. Preserving the value in doctor output and warning at runtime is the narrowest maintainable fix; broad automatic provider rewrites should be limited to clearly single-provider configs to avoid changing fallback behavior unexpectedly.
