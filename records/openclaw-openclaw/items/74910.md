## Summary

Keep open. Current main still matches the report: the doctor migration detects `agents.defaults.llm` but deletes it without preserving or echoing `idleTimeoutSeconds`, and the runtime idle-timeout resolver has no legacy path before falling back to the 120s default. The provided timeline also shows open PR #74940 with closing syntax for this issue, so the issue should remain open until that implementation lands or is closed.

## What This Changes

Make `doctor --fix` preserve or clearly echo legacy `agents.defaults.llm.idleTimeoutSeconds`, and add runtime diagnostics when the legacy block is still present.

## Best Possible Solution

Review and land #74940, or an equivalent narrow fix, so doctor preserves the user's legacy timeout intent in its repair output and runtime config loading emits a one-shot warning for the retired path. The fix should keep automatic provider rewrites conservative unless there is an unambiguous single-provider target.

## Reproduction Assessment

Yes. A high-confidence source reproduction is the existing migration test shape with `agents.defaults.llm.idleTimeoutSeconds`, which currently expects deletion without value preservation, plus the resolver path that returns the 120s default when no provider timeout is configured.

## Solution Assessment

Yes. Preserving or echoing the exact legacy value in doctor output and adding a runtime warning is the narrowest maintainable fix; broad automatic edits to provider config should be limited to unambiguous cases.
