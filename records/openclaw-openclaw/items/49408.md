## Summary

Current main still does not implement the reported workflow of symlinking an external skill directory under `~/.openclaw/skills`. The loader now sees symlinked child directories, but it then realpath-checks each candidate against the configured skill root, so root-escaping symlinks are deliberately skipped. The remaining decision is a maintainer/security policy choice: support external managed-skill symlinks with an explicit trust model, or document that `skills.load.extraDirs`/ClawHub installs are the supported path.

## Best Possible Solution

Keep this issue open for maintainer policy judgment. If OpenClaw should support symlinked managed skills that point outside `~/.openclaw/skills`, add an explicit opt-in trust model and update the loader, security audit, docs, and regression tests together. If the containment policy is intentional, document that root-escaping symlinks under managed skill roots are unsupported and direct users to `skills.load.extraDirs` or ClawHub installs.
