## Summary

Keep open. Current main has terminal chat-event handling and manual browser Web Push subscribe/test support, but it does not implement an opt-in completion sound or automatic turn-complete notification in the Control UI. The related implemented `agent_end` plugin hook is useful context, but it does not satisfy the requested browser-local WebChat sound/setting behavior.

## What This Changes

Request an optional Control UI/WebChat notification sound when an agent turn finishes, with settings support and possible custom sound configuration.

## Best Possible Solution

Add an off-by-default, browser-local Control UI setting that plays a short built-in completion chime when the active chat run reaches a successful final state. Wire it through the existing terminal chat event path and settings UI, document the behavior, and defer arbitrary remote custom sound URLs unless a safe validation/privacy policy is agreed.

## Reproduction Assessment

Yes. Code inspection of current main is enough: final chat events are handled and clear the active run, while settings and notification surfaces have no completion-sound option or playback path.

## Solution Assessment

Yes, the narrowest maintainable path is a Control UI-owned browser setting tied to terminal chat events. Reusing the existing plugin `agent_end` hook alone would not solve the requested WebChat background-tab audio cue.

## Repair Work Prompt

Implement an opt-in Control UI/WebChat completion sound for https://github.com/openclaw/openclaw/issues/69186. Scope it to the browser Control UI: add a browser-local `UiSettings` preference, default off, exposed in the settings/notifications UI, and play a short built-in/generated chime when the active chat run reaches a successful `final` terminal event. Do not play for unrelated side-result terminal events, another run's final event, aborted runs, or errors unless the design explicitly separates those states. Respect browser autoplay limits: only enable after an explicit user action, catch playback failures without surfacing noisy errors, and do not block chat handling. Prefer Web Audio API or a packaged local asset; do not add arbitrary remote custom sound URL support in this first pass. Update English UI strings/source of truth as needed, keep generated locale handling consistent with `ui/AGENTS.md`, add focused unit/browser tests around settings persistence and terminal event triggering, update `docs/web/control-ui.md`, and add a user-facing changelog entry with reporter credit if the repo's active changelog policy applies. Avoid changing plugin hook semantics, gateway push delivery, or unrelated notification behavior.
