## Summary

Keep #69678 open. Current main still hard-codes cron run summaries to 2000 UTF-16 code units before they become the `summary` stored in run-log entries, and the existing `cron.runLog` config only controls file pruning by bytes and retained lines. The requested configurable or larger per-entry summary limit remains a real code/docs gap.

## Best Possible Solution

Implement a cron-owned fix: add `cron.runLog.maxSummaryChars` to `CronConfig`, zod schema, help/generated schema, and docs; plumb it into summary selection or log-entry creation so isolated cron outputs longer than 2000 chars can be preserved up to the configured limit while existing `maxBytes`/`keepLines` still bound disk usage. Add focused regression coverage for a >2000-character agent output and for invalid/default config handling.
