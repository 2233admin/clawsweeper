## Summary

Current main still hard-codes isolated cron run summaries to 2000 UTF-16 code units before storing them as cron run-log `summary` values. The `cron.runLog` config group still only controls file pruning (`maxBytes` and `keepLines`), so the requested configurable or larger per-entry summary limit remains unimplemented.

## Best Possible Solution

Add a cron-owned `cron.runLog.maxSummaryChars` config field with a conservative default, validation, help/generated schema, and docs. Plumb the resolved value through isolated cron summary selection and delivery/subagent resummary paths so long agent outputs can be retained up to the configured cap while `maxBytes` and `keepLines` still bound file growth. Add focused tests for default behavior, configured summaries above 2000 characters, invalid config handling, configured truncation with ellipsis, and UTF-16-safe boundaries.

## Clownfish Work Prompt

Fix this issue by making isolated cron run-log summary truncation configurable under `cron.runLog`.

Current behavior: isolated cron summaries are capped by `pickSummaryFromOutput` at 2000 UTF-16 code units before being stored in `cron/runs/<jobId>.jsonl`, while `cron.runLog.maxBytes` and `cron.runLog.keepLines` only bound file pruning. Long agent outputs delivered to Slack/Feishu/etc. can therefore be complete while `openclaw cron runs --id <jobId>` shows only the first part.

Expected behavior: add `cron.runLog.maxSummaryChars` with a conservative default chosen by maintainers. Trim empty text as today, preserve UTF-16-safe truncation, append an ellipsis only when the configured per-summary cap is exceeded, and keep `maxBytes` / `keepLines` as the disk-growth bounds. A configured value above 2000 should allow run-log summaries above 2000 characters.

Implementation boundary: update `src/config/types.cron.ts`, `src/config/zod-schema.ts`, config help/labels/generated schema, and cron/configuration docs. Resolve the summary cap from `params.cfg.cron?.runLog` in the isolated cron path and pass it through `resolveCronPayloadOutcome`, `pickSummaryFromOutput`, `pickSummaryFromPayloads`, fatal delivery summaries, and delivery-dispatch descendant final-output resummary calls. Include `run-executor` interim outcome calls if the helper signature requires it. Keep the change cron-owned; do not change unrelated channel delivery, provider behavior, active-memory `maxSummaryChars`, or run-log pruning semantics.

Tests: add or update focused Vitest coverage for default truncation behavior, configured preservation above 2000 characters, configured truncation with ellipsis, invalid/default config handling, and UTF-16-safe boundary behavior. Update docs and add a single-line user-facing changelog fix entry under the active version with `Thanks @wanhuahua0108`.
