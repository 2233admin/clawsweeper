## Summary

Keep open. Current main still truncates isolated cron run summaries at 2000 UTF-16 code units before the Gateway writes the JSONL run-log entry, and the existing `cron.runLog` config only controls file pruning by bytes and retained lines. The report is concrete, reproducible from source, not paired with an open fixing PR, and narrow enough for a focused repair.

## What This Changes

The issue requests a configurable or larger cron run-log summary length cap so long assistant outputs are not silently truncated around 2000 characters.

## Best Possible Solution

Add a bounded `cron.runLog.maxSummaryChars` config option under the existing run-log group, use it everywhere cron derives persisted `summary` text, keep `outputText` and delivery payloads unchanged, and document that file growth remains bounded by `maxBytes` / `keepLines`.

## Reproduction Assessment

Yes. A source-level reproduction is enough here: any isolated cron payload or final assistant text longer than 2000 UTF-16 code units is reduced by `pickSummaryFromOutput` to 2000 characters plus an ellipsis before the Gateway persists the run-log entry.

## Solution Assessment

Yes. Making the cap configurable in the existing `cron.runLog` config group is the narrowest maintainable fix because it addresses audit/readability without changing delivery routing, session storage, or the existing file-pruning controls.
