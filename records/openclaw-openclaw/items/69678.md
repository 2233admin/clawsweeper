## Summary

Keep this issue open. Current main still hard-caps isolated cron run summaries at 2000 characters before the run-log entry is written, while the existing `cron.runLog` config and docs only expose file pruning controls (`maxBytes` and `keepLines`), not a per-summary length cap.

## What This Changes

Request to make cron run-log summary truncation configurable, or raise the current hard-coded 2000-character cap, so long agent outputs remain auditable in `openclaw cron runs`.

## Best Possible Solution

Add a validated and documented `cron.runLog.maxSummaryChars` option, or an equivalent clearly named field, with a larger bounded default; pass the resolved value into cron summary extraction so per-entry summaries remain useful while `maxBytes` and `keepLines` continue to bound total log growth.

## Reproduction Assessment

Yes. The current-main source path is deterministic: any isolated cron output longer than 2000 cleaned UTF-16 characters is truncated by `pickSummaryFromOutput` before `evt.summary` is appended to the per-job JSONL run log.

## Solution Assessment

Yes. A config-backed summary cap under the existing `cron.runLog` group is the narrowest maintainable fix; only raising the constant would reduce pain but would not give operators control over storage versus audit detail.
