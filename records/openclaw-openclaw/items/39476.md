## Summary

Current main still allows the reported A2A reverse-delivery shape: sessions_send exposes the requester session in A2A prompts, peer A2A still runs requester/target follow-up and announce turns as normal agent runs, and the gateway agent request schema has no per-run tool deny/allow field that could suppress sessions_send only inside those nested turns.

## Best Possible Solution

Keep this issue open and fix the core A2A sessions_send lifecycle. Add a framework-level delivery intent or per-run tool policy that prevents target/reply/announce A2A turns from calling sessions_send back to the requester, narrow requester session-key prompt exposure where possible, preserve the parent-owned ACP child skip, and add regression coverage for peer A2A duplicate visible delivery. Treat #44309 as related dispatch-only semantics unless it lands the same hard reverse-send guard.
