## Summary

Keep open. Current main still permits the reported A2A reverse-delivery shape: the target run receives the requester session in prompt context, nested A2A reply/announce turns run as normal agent turns, and the agent RPC schema has no per-run tool deny/allow field to suppress sessions_send only for those nested turns. Related #44309 tracks broader dispatch-only semantics, but it does not supersede this hard reverse-send guard.

## Best Possible Solution

Keep this issue open and fix the core A2A sessions_send lifecycle. Add a framework-level guard that prevents target/reply/announce A2A turns from calling sessions_send back to the requester, reduce requester session-key exposure where possible, preserve the parent-owned ACP child skip, and add regression coverage for duplicate visible delivery. Treat #44309 as related dispatch-only semantics unless it also lands the hard reverse-send guard.
