## Summary

Keep this issue open. Current main still matches the reported agent-facing host exec path: non-sandbox exec builds a PowerShell argv via `getShellConfig()` and then spawns that wrapper with `windowsHide: true`; the merged lower-level `runExec`/`runCommandWithTimeout` hide fix from #59466 does not remove the PowerShell wrapper described here.

## What This Changes

The issue reports that Windows host exec commands are still launched through PowerShell, causing a visible console flash despite `windowsHide`, and asks for a direct or better-hidden spawn path for commands that do not need shell features.

## Best Possible Solution

Keep the issue open until maintainers choose and ship a Windows host-exec launcher strategy that avoids a visible PowerShell console for simple direct executable invocations, preserves PowerShell fallback for shell features and Windows utility output capture, keeps approval/security analysis consistent, and adds unit plus native Windows smoke coverage.

## Reproduction Assessment

Yes for the triggering execution shape: current main still builds `[PowerShell, -NoProfile, -NonInteractive, -Command, execCommand]` for normal host exec and then spawns it with `windowsHide: true`. The actual visible flash still needs native Windows desktop/service verification, but the code path in the report is high-confidence.

## Solution Assessment

No, current main is not the best solution for this report because it only hides the spawned wrapper and still launches PowerShell for every normal host exec. A safer fix needs an explicit Windows strategy for direct simple commands versus shell-feature commands, rather than assuming `windowsHide` on PowerShell is sufficient.
