## Summary

Current main already fixes the reported gateway-scoped stdio MCP leak with generic lifecycle handling: stdio MCP transports close through process-tree termination, reset/delete cleanup retires the session MCP runtime, and an idle TTL sweep disposes unreleased session runtimes. The fix is shipped in v2026.4.25 and remains present in the latest release line.

## What This Changes

Clean up session-scoped MCP runtimes and stdio child processes such as `vibe-trading-mcp` when Gateway sessions end so processes do not accumulate.

## Best Possible Solution

Keep the shipped generic MCP lifecycle cleanup. If accumulation still reproduces on v2026.4.27 or newer, it should be reopened as a fresh report with updated process listings, timestamps, `mcp.sessionIdleTtlMs`, server config, and whether leases remain active.

## Reproduction Assessment

Yes. The reproduction path is to configure a stdio MCP server, invoke it through a Gateway-backed session, end/reset/delete or idle the session, and inspect whether the child process remains; current tests cover the generic reset/delete, idle sweep, and process-tree close behavior without exercising `vibe-trading-mcp` specifically.

## Solution Assessment

Yes. The current implementation fixes the lifecycle at the generic OpenClaw MCP runtime and transport layers, which is narrower and more maintainable than adding a server-specific `vibe-trading` cleanup path.
