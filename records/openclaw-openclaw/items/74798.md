## Summary

Current main and shipped releases cover the reported bundled-MCP gateway leak: session reset/delete now retires the session MCP runtime, stdio transports tear down child process trees, and session-scoped runtimes have an idle reaping backstop. The follow-up comment correctly points this v2026.4.9 report at newer releases.

## What This Changes

The issue requests gateway session-end cleanup for stdio MCP runtimes so vibe-trading-mcp child processes do not remain under openclaw-gateway after use.

## Best Possible Solution

Keep the shipped lifecycle model: explicit retirement for reset/delete and one-shot run paths, transport-level process-tree cleanup for stdio MCP servers, and the configurable idle TTL as a backstop for long-lived gateway sessions.

## Reproduction Assessment

Yes for the reported class: run a stdio MCP server through a gateway session, end/reset/delete or let the session idle, then inspect remaining child PIDs. I did not run a live Feishu/vibe-trading case in this read-only review, but the report gives a concrete v2026.4.9 symptom and current source/tests cover the cleanup path.

## Solution Assessment

Yes. The current solution is the narrow maintainable fix because it cleans up at the session/runtime boundary and at the stdio transport boundary, without adding provider-specific vibe-trading behavior to gateway core.
