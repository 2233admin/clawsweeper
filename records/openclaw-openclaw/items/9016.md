## Summary

Keep this issue open. Current main has adjacent local estimated-cost reporting, transcript-backed aggregation, `/usage cost`, and session `estimatedCostUsd`, but it still does not expose OpenRouter's provider-reported per-response `usage.cost` or `cost_details.upstream_inference_cost` to agent runtime/session metadata. The useful ClawRouter workaround and closed #9696 estimate work do not close this exact runtime metadata gap.

## Best Possible Solution

Keep this issue open. The best implementation is to capture OpenRouter raw usage at the OpenAI-compatible response/stream boundary, preserve provider-reported cost separately from OpenClaw's estimated USD cost with explicit source and units, carry `usage.cost`, `cost_details.upstream_inference_cost`, and cache write/read token details through transcript/session usage and `EmbeddedPiAgentMeta`, then expose per-turn and cumulative values through `session_status` or a dedicated usage/cost tool. Telegram and other channels can then render exact OpenRouter costs when available and fall back to current estimates otherwise.

## Risks / Open Questions

- Closing now would conflate local estimated-cost support with exact OpenRouter provider billing data and could make users think `usage.cost` is available to agents when it is not.
- The ClawRouter link in the discussion is useful prior art for cost visibility, but it is an external routing workaround and does not prove OpenClaw core exposes OpenRouter's returned `usage.cost`.
- A future fix needs an explicit contract for provider-reported cost source, units, and fallback behavior so current estimated USD fields are not silently reinterpreted as exact charges.
- Both streaming final usage chunks and any non-streaming OpenAI-compatible response path should be covered when implementing this, because OpenRouter exposes usage at the response boundary.
