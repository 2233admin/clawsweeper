## Summary

Keep open. Current main still does not implement `after_agent_complete` or the PR's provider-neutral post-run/pre-delivery suppress/reinject API. The newer `before_agent_finalize` hook partially overlaps by allowing one more model pass before finalization, but it does not expose silent suppression, injected-context reruns, timing/channel context, or a runner-level wrapper around `runAgentTurnWithFallback`. The PR is stale and has unresolved design/security concerns, but it is not safely auto-closeable as implemented-on-main or superseded.

## Best Possible Solution

Keep this PR open for maintainer design. The best path is to consolidate with the current `before_agent_finalize` direction where possible, then decide whether speedtrap-style stale-response handling needs an extension of that hook, a new provider-neutral post-run/pre-delivery hook with explicit suppress/context semantics, or a smaller replacement PR with tests for non-streaming, block streaming, transcript state, hook policy gating, and conversation/channel context.

## Risks / Open Questions

- Merging this PR as-is would add an overlapping lifecycle API now that current main has `before_agent_finalize`; it needs a rebase/redesign against the current hook model.
- The proposed hook can mutate prompt flow and observe raw response content, so policy gating should explicitly cover both prompt injection and conversation-content access.
- The PR's acknowledged streaming and transcript-state limitations could violate the stated pre-delivery suppress/reinject contract without additional runtime design.
