## Summary

Keep open. Current main still does not implement the PR's intended `before_tools_resolve` plugin hook or an equivalent list-level pre-LLM tool filtering surface. Main has static tool allow/deny policy and per-tool-call hooks, but no plugin hook that receives the assembled tool list before it is sent to the model. The PR is still relevant but security-sensitive, so the best path is maintainer product/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. If the feature is accepted, rework it against current `src/plugins/hook-types.ts`/`src/plugins/hooks.ts`, evaluate the hook once over the final model-visible tool set in both normal runs and compaction, include core, client, bundled MCP, and LSP tools, normalize tool names, fail closed for policy-hook failures, preserve deny-union and allow-intersection semantics, and add focused docs plus regression tests for multi-hook intersections and bypass cases. If maintainers decide this policy boundary is not wanted in core, close it explicitly as a product decision rather than as already implemented.
