## Summary

Keep open. Current main still cleans up the ACP runtime for only the session being reset; it does not enumerate ACP child sessions owned by the parent, while spawned ACP sessions are persisted with ownership metadata that would allow that lookup. The existing docs and tests still match the reported gap, and the related open PR covers a narrower dormant-oneshot cache path rather than this parent-reset cleanup.

## What This Changes

The issue requests reliable cleanup for ACP mode=run child processes, including parent reset cleanup of child ACP session keys and fallback handling for orphaned ACP adapter processes.

## Best Possible Solution

Implement parent-reset ACP child cleanup in the gateway reset path using existing session ownership metadata, add focused regression coverage, then live-verify whether acpx close/cancel fully terminates adapter descendants before adding any PID or process-group fallback.

When sessions.reset or /new resets a parent session, enumerate ACP child sessions whose entry.spawnedBy or entry.parentSessionKey matches the canonical reset parent and close/cancel those ACP runtimes before or alongside the parent mutation. Start in src/gateway/session-reset-service.ts; prefer the existing listAcpSessionEntries seam from src/acp/runtime/session-meta.ts and the existing closeAcpRuntimeForSession cancel+close path.
