## Summary

This PR addresses a real ACP identity bug that is still present on current main and in the latest release, but the proposed patch should not merge as-is because it promotes ensure-sourced identities into the same precedence class as status/event identities and can persist stale ACP IDs.

## What This Changes

This PR marks ACP identities created from `ensureSession()` as resolved when stable backend or agent session IDs are already present, adds focused `createIdentityFromEnsure()` tests, and updates a direct ACP spawn fixture expectation.

## Best Possible Solution

Land a corrected ACP identity change that resolves stable-ID ensure results for new or unresolved identities while preserving status/event authority over already resolved identities, with focused coverage for stable ensure, record-only ensure, stale ensure versus fresh status precedence, startup skip behavior, and status-refresh failure behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Prevent ensure identities from overwriting resolved IDs:** `src/acp/runtime/session-identity.ts:164-166`
  - body: Marking ensure-derived identities as `resolved` makes `mergeSessionIdentity()` treat them as authoritative when merged into an already resolved identity. A later `ensureRuntimeHandle()` call can replace fresher status/event `acpxSessionId` or `agentSessionId` values with stale `ensureSession()` values, and the `failOnStatusError: false` path can return before status repairs them.
  - confidence: 0.91
