## Summary

This PR should stay open because current main still has the ACP ensure-identity gap, but the proposed patch is not merge-ready: promoting every ensure-derived identity with a stable ID to resolved can let stale ensureSession identifiers override already-resolved status/event identifiers in the existing merge path.

## What This Changes

The PR marks ensure-derived ACP identities as resolved when the ensured handle already has backend or agent session IDs, adds focused createIdentityFromEnsure tests, and updates a direct ACP spawn fixture expectation.

## Best Possible Solution

Resolve stable ensure results for new or still-unresolved identities, while preserving status/event authority over already-resolved identifiers; add focused coverage for stable ensure, record-only ensure, stale ensure versus fresh status/event, startup skip behavior, and status-refresh failure behavior.

## Reproduction Assessment

Yes. The regression path is reproducible from source inspection: ensureRuntimeHandle merges ensure metadata into an existing identity, mergeSessionIdentity accepts resolved incoming values, and the failOnStatusError:false path can return before status repairs stale identifiers.

## Solution Assessment

No. The PR fixes the reported pending state too broadly; the safer solution is a source-aware merge or equivalent precedence rule that lets ensure resolve new identities without replacing already-resolved status/event IDs.
