## Summary

Keep this PR open. Current main does not expose `authProfileOverride` from model-resolution hooks, so the feature is not implemented on main; however, the proposed branch is not ready to merge because its legacy auto auth-profile compatibility relies on a new compaction-count runner parameter that is not threaded through the normal queued reply path.

## What This Changes

The PR extends `before_model_resolve` and legacy `before_agent_start` hooks to return `authProfileOverride`, merges it by hook priority, feeds it into embedded-runner auth-profile selection and live-switch comparison, and updates related plugin documentation/tests.

## Best Possible Solution

Keep the API direction, but update the branch so legacy auth-profile source metadata is normalized or forwarded through every runner entry path that can carry a session auth profile. Add a regression for a migrated chat session with `authProfileOverride` plus `authProfileOverrideCompactionCount` and no source, proving a trusted `before_model_resolve` hook can still prefer a different automatic profile while explicit user-pinned profiles remain locked.

## Reproduction Assessment

Yes. A static reproduction path is clear: create a migrated session entry with `authProfileOverride` and `authProfileOverrideCompactionCount` but no `authProfileOverrideSource`, enter the normal queued reply path, and the PR helper receives no compaction count, classifies the profile as user-pinned, and suppresses hook auth-profile rotation.

## Solution Assessment

No. The proposed hook/API shape is a reasonable narrow solution, but the implementation is incomplete until the legacy auto marker is preserved across the normal reply runner path and any other `runEmbeddedPiAgent` callers that forward session auth-profile ids.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Thread legacy auto auth markers through queued runs:** `src/agents/pi-embedded-runner/run/setup.ts:116-120`
  - body: `resolvePreferredRunAuthProfile` depends on `requestedAuthProfileCompactionCount` to distinguish migrated automatic auth-profile selections from user-pinned profiles. The normal queued reply path currently forwards only `authProfileId` and `authProfileIdSource`, so a legacy entry with `authProfileOverrideCompactionCount` but no source reaches this helper without the marker and is treated as user-pinned. That blocks `before_model_resolve` from applying `authProfileOverride` in live chat for migrated auto profiles; carry the marker through the followup/run-base params or normalize the source before calling the runner.
