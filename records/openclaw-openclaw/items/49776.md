## Summary

Keep open. Current main still does not prove this Control UI freeze is fixed: the WebSocket Control UI path still makes one fixed `chat.history` request, the protocol has no pagination cursor, and the Gateway still reads/parses the transcript before applying response caps. Live related context also shows #49335 remains open for the full-transcript read freeze vector, while #47979 and #48514 remain open for adjacent Control UI/browser and accumulated-history freeze paths.

## Best Possible Solution

Keep this issue open. Ask the reporter for a current-main reproduction with DevTools console errors, failed Network/WS frames, browser/OpenClaw versions, session file sizes, and whether resetting sessions still restores the UI. If session-size recovery reproduces, review and land #49335 or an equivalent bounded transcript reader. If the browser locks before any `chat.history` request, route it through the Chrome/Lit Control UI path tracked by #47979. If accumulated history is the root cause, wire the existing paginated session-history API or an additive WebSocket cursor contract into Control UI with load-older/virtualized rendering, coordinating with the existing open UI history PRs.
