## Summary

Keep open. Current main still has the single preauth WebSocket handshake timer and does not implement this PR's two-phase connect-auth watchdog, so the contribution is not obsolete. The PR should not merge as-is because the new watchdog is resettable by repeated valid preauth `connect` frames while async auth is still pending, which preserves the unauthenticated resource-retention problem the PR is trying to bound.

## What This Changes

The PR adds a Gateway WebSocket connect-auth watchdog after a valid `connect` frame, moves preauth timer clearing earlier, updates gateway timeout docs and changelog, and adds focused gateway timeout tests.

## Best Possible Solution

Repair this PR branch in place: make the connect-auth watchdog one-shot or set an explicit connecting state before the first auth await, reject or close duplicate preauth `connect` frames deterministically, keep timer cleanup on close and successful registration, and add regression coverage for repeated valid connect frames during stalled auth. Preserve the #49751 contributor credit and keep docs/changelog aligned with the final behavior.

## Reproduction Assessment

Yes. The source-level reproduction is to send a valid `connect` frame that stalls in `resolveConnectAuthState` or `resolveConnectAuthDecision`, then send another valid preauth `connect` before the watchdog fires; because `getClient()` is still null and the PR re-arms the watchdog, the deadline is extended.

## Solution Assessment

No. The two-phase timer direction is the right narrow design for the reported slow-auth race, but this implementation is not the best fix because the auth-phase deadline can be reset; the safer approach is a one-shot watchdog plus a duplicate-connect state transition before any auth await.
