## Summary

Keep open. The PR targets a real Gateway WebSocket auth-timeout gap that current main still has, but the proposed connect-auth watchdog can be reset by repeated valid preauth `connect` frames during async auth, which undercuts the security property the PR is adding. The existing Aisle, Greptile, and ClawSweeper comments all point at the same blocker, so this needs repair before merge rather than cleanup closure.

## What This Changes

The PR adds a two-phase Gateway WebSocket handshake timeout, clearing the preauth timer after a valid `connect` frame, arming a connect-auth watchdog, updating Gateway protocol docs/changelog, and adding focused Gateway tests.

## Best Possible Solution

Repair the existing PR branch with a one-shot connect-auth watchdog or an explicit preauth `connecting` state set before the first auth await, deterministically reject or close duplicate preauth connect frames, keep cleanup on close and successful registration, and add regression coverage for repeated valid connect frames while auth is stalled.

## Reproduction Assessment

Yes. Send one valid `connect` frame that stalls in `resolveConnectAuthState` or `resolveConnectAuthDecision`, then send another valid preauth `connect` before the watchdog fires; because no in-flight state is set and no client is registered yet, the second frame can re-enter the connect path and rearm the watchdog.

## Solution Assessment

No. The two-phase timer direction is the right narrow fix, but this implementation is not the best solution while the auth-phase deadline is resettable; a one-shot watchdog or explicit connecting state before any auth await is safer.
