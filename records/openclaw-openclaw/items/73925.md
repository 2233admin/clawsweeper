## Summary

Keep open. The PR targets a real current-main gap in Gateway WebSocket handshakes, but the proposed branch leaves the new connect-auth watchdog resettable by repeated valid preauth `connect` frames, which is a blocking security/correctness issue before merge.

## What This Changes

The PR adds a Gateway WebSocket connect-auth watchdog, moves preauth timer clearing to valid `connect` handling, adds closed-socket guards, and updates gateway tests, docs, and changelog coverage.

## Best Possible Solution

Repair this PR branch in place by making connect auth a one-shot state transition before any auth await, rejecting or closing duplicate preauth `connect` frames deterministically, clearing both timers on close/register paths, and adding regression coverage for both stalled auth and repeated valid connect frames.

## Reproduction Assessment

Yes. Send a valid `connect` frame that stalls in `resolveConnectAuthState` or `resolveConnectAuthDecision`, then send another valid preauth `connect` before the watchdog fires; because no client or connecting state exists yet, the second frame re-enters the connect path and re-arms the watchdog.

## Solution Assessment

No. The two-phase timer direction is the narrow maintainable approach, but this implementation is not the best fix until the auth-phase deadline cannot be reset by duplicate preauth frames.

## Review Findings
