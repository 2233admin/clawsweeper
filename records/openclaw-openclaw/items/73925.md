## Summary

Keep this PR open. Current main still has the single preauth WebSocket handshake timer and clears it only after async connect auth and registration prep, so the PR's intended two-phase preauth/connect-auth watchdog is not implemented on main. The PR branch is relevant and carries forward #49751, but it still has an unresolved security finding: repeated valid preauth connect frames can re-arm the new connect-auth watchdog and defeat the bounded-auth guarantee.

## Best Possible Solution

Keep this PR open and fix the branch in place: make the connect-auth watchdog one-shot or add an explicit connecting state before the first auth await, reject or close duplicate preauth connect frames deterministically, clear both timers on all close/register paths, and add regression coverage for stalled auth plus repeated valid connect frames. Preserve @sashakhar1's #49751 credit and rerun the focused gateway Vitest command plus `pnpm check:changed` before merge.
