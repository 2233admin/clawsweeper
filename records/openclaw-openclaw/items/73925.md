## Summary

Keep this PR open. Current main still lacks the valid-connect/auth-phase split, so the work remains relevant, but the proposed branch has a blocking preauth DoS regression: duplicate valid connect frames can re-arm the new connect-auth watchdog while auth is still pending.

## What This Changes

The PR adds a Gateway WebSocket connect-auth watchdog, moves preauth timer clearing to valid connect handling, updates gateway timeout docs and changelog, and adds gateway tests for stalled-auth timeout behavior.

## Best Possible Solution

Repair this PR branch in place: make the connect-auth watchdog one-shot or set an explicit connecting state before the first auth await, reject or close duplicate preauth connect frames deterministically, keep both timers cleared on close/register paths, and add regression coverage for stalled auth plus repeated valid connect frames. Preserve the original #49751 contribution credit and keep the docs/changelog aligned with the final behavior.

## Reproduction Assessment

Yes: the reproduction path is to send a valid `connect` frame that stalls in `resolveConnectAuthState` or `resolveConnectAuthDecision`, then send another valid preauth `connect` before the watchdog fires. Because `getClient()` remains null and no in-flight state is set, the second frame re-enters the connect path and re-arms the watchdog.

## Solution Assessment

No: the two-phase timer direction is the right narrow design, but the current implementation is not the best way because the auth-phase deadline is resettable. A safer fix is a one-shot watchdog plus an explicit duplicate-connect state transition before any auth await.
