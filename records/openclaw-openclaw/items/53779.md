## Summary

Keep open. The linked bug is real and still relevant, but this PR does not fix the reported PI runtime failure path: it only drops providers that fail OpenClaw's Zod schema, while the missing-apiKey case passes that schema and fails later when PI loads models.json. The diff also introduces raw config-derived terminal warning output.

## What This Changes

The PR adds a pre-validation pass to src/config/validation.ts that drops Zod-invalid models.providers entries and logs a warning, plus a new test file for valid, mixed-invalid, warning, and non-provider-invalid cases.

## Best Possible Solution

Implement provider isolation at the models.json generation or PI runtime discovery boundary with PI-equivalent per-provider semantic validation. Valid providers should preserve baseUrl, auth, headers, models, overrides, and metadata; invalid providers should be skipped with sanitized diagnostics and regression coverage for mixed providers, all-invalid providers, missing-apiKey custom providers, invalid API enums, and generated implicit-provider cases.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure one valid custom provider with a baseUrl and one non-built-in provider with models but no apiKey; OpenClaw's schema accepts it, then PI validateConfig rejects models.json and returns empty custom overrides.

## Solution Assessment

No. The proposed fix catches only OpenClaw Zod provider-schema errors before models.json generation, while the reported failure happens later in PI semantic validation; warning output also needs the existing terminal-sanitization path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Recover from PI semantic provider failures:** `src/config/validation.ts:251`
  - body: This pre-pass only calls ModelProviderSchema.safeParse, where apiKey is optional, and it runs before generated or merged providers are loaded by PI. A non-built-in provider with custom models but no apiKey can still pass here, then PI rejects the whole models.json and drops unrelated valid overrides.
  - confidence: 0.9
- **[P2] Sanitize provider warning text:** `src/config/validation.ts:264`
  - body: The warning interpolates the provider key and validation reason directly from user-editable config into console.warn. Existing config diagnostics sanitize path and message first, so crafted provider ids or issue text can inject newlines or terminal control characters into logs.
  - confidence: 0.88
