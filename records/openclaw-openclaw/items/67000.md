## Summary

Keep this issue open. Current main still does not implement warm standby or persistent embedded-agent session reuse for active-memory: uncached lookups create a fresh active-memory sub-agent session id and session file, then call the embedded runner, which still performs per-call queueing, workspace/plugin/model/auth setup, resource loading, and AgentSession creation. Related #50963, #73501, and #73808 are adjacent and open, but none clearly supersedes this active-memory-specific request.

## Best Possible Solution

Keep this issue open as active-memory embedded-agent warm reuse work. A maintainer-quality fix likely belongs either in the active-memory plugin as a bounded per-parent-session memory sub-agent reuse pool, or in a generic embedded-runner/session API if multiple plugin-owned embedded agents need the same warm-standby semantics. It should include TTL/invalidation rules, isolation/security boundaries, and tests proving repeated active-memory turns reuse initialized state or skip the expensive setup path without cross-session leakage.

## Risks / Open Questions

- A warm reuse implementation must define invalidation for config, model, auth profile, tool allowlist, memory backend, prompt, workspace, and plugin runtime changes.
- Session reuse must preserve per-parent-session isolation and avoid leaking recalled memory, tool state, auth state, or transcript state across conversations or agents.
- The adjacent provider/tool-factory regressions in #73501 and #73808 may reduce first-message latency, but they would not by themselves satisfy persistent active-memory embedded-session reuse.
