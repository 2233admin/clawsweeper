## Summary

Keep this PR open. Current main fixes the narrow empty/NO_REPLY interim-ack path, but it still does not implement the PR’s broader intended behavior: recovering arbitrary successful raw-stdout toolResult text as final isolated cron output when assistant-visible text is empty. The submitted patch remains directionally useful, but needs a rewrite because its retry suppression is prompt-keyword based and its transcript-to-output promotion needs current-attempt gating.

## Best Possible Solution

Keep this PR open, but require a rebase/rewrite on current main. The best fix is structural current-attempt toolResult recovery in the embedded runner or cron outcome flow: recover only verified successful current-run toolResult text, use it when assistant output is empty, suppress interim retry only when recovered output or an exact silent token proves completion, and add focused tests for raw stdout success, acknowledgement-without-tool-result retry, stale/older or missing timestamps, errored tool results, malformed transcript entries, and exact NO_REPLY.

## Risks / Open Questions

- Current main can still complete without surfacing arbitrary raw stdout when a successful toolResult exists but assistant payload/final visible text is empty.
- The submitted patch can suppress the only interim-ack retry for a raw-stdout phrasing even when no successful tool result was captured.
- Transcript-to-output promotion can expose stale, unrelated, missing-timestamp, or cross-run output unless tied to verified current-attempt state.
- The PR test writes a fixed /tmp transcript with a timestamp captured before runStartedAt, making the test timing-sensitive and cleanup dependent on success.
