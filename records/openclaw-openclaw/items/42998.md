## Summary

Keep this PR open. Current main fixed the narrow empty/NO_REPLY interim-ack rerun path, but it still does not implement the PR's broader intended behavior: recovering arbitrary successful raw-stdout toolResult text as the final isolated cron output. The submitted patch remains directionally relevant, but needs a rewrite because its retry suppression is prompt-keyword based and its transcript scan can surface stale or unrelated tool output.

## Best Possible Solution

Keep this PR open but require a rebase/rewrite on current main. The best implementation is structural current-attempt tool-result recovery in the embedded-runner/cron outcome flow: recover only a verified successful current-run toolResult, use that recovered text in cron output resolution, suppress interim retry only when recovered output or an exact silent token proves completion, and add focused tests for raw stdout success, exact NO_REPLY, acknowledgement-without-tool-result retry, stale/older transcript entries, malformed timestamps, and errored tool results.

## Risks / Open Questions

- Functional: current main can still finish without surfacing arbitrary raw stdout when a successful tool result exists but assistant payload/final visible text is empty.
- Functional regression risk in the submitted patch: prompt-keyword-based retry suppression can skip the only interim-ack retry even when no successful current-run tool result has been captured.
- Security/correctness risk in the submitted patch: reading session transcript text and promoting a toolResult to final cron output can expose stale, malformed, errored, or cross-run tool output unless the recovery is tied to verified current-attempt state.
- Security review pass: the supplied diff does not add workflows, dependencies, lockfile changes, lifecycle hooks, package publishing metadata, permissions, or third-party code execution; the security-sensitive change is the transcript-to-output data path.
