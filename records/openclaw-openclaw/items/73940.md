## Summary

Keep this PR open, but do not merge it as-is. Current main still lacks the Feishu post-start WSClient retry-exhaustion recovery this PR targets, while the PR introduces a concrete backoff/reset problem and overlaps with an open maintainer-labeled recovery PR.

## What This Changes

This PR updates Feishu client construction, WebSocket monitor transport, cleanup tests, and the changelog to supervise SDK retry exhaustion or stalled reconnect state and recreate WSClient instances with abortable backoff while preserving bot identity.

## Best Possible Solution

Keep the recovery work inside the Feishu plugin transport/client boundary, but choose one canonical maintainer-reviewed implementation. The final fix should observe only terminal SDK retry exhaustion or a carefully proven stalled reconnect state, preserve bot identity across replacement cycles, keep abort cleanup deterministic, and avoid resetting backoff until the connection has been healthy for a meaningful stable period.

## Reproduction Assessment

Yes. Current main has a high-confidence code-level reproduction path: after wsClient.start() returns, monitorWebSocket waits only for abort, while the SDK reports retry exhaustion through lifecycle state/callbacks rather than rejecting that awaited start path. citeturn1view0

## Solution Assessment

No, not as-is. The overall direction is appropriate, but resetting supervisor backoff after any prior connection is not the safest fix; a stable-connection reset threshold or monotonic backoff across rapid cycles would be safer, and #73739 is the maintainer-owned competing path to evaluate.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Keep backoff across rapid reconnect failures:** `extensions/feishu/src/monitor.transport.ts:319-323`
  - body: After any `onReady`/`onReconnected`, `lifecycle.hasConnected()` stays true, so every supervisor-triggered recycle resets `attempt` to 0 before incrementing. A network that briefly connects and then exhausts again will recreate clients every 1s indefinitely, causing client churn and log amplification instead of exponential backoff. Reset only after a stable healthy period, or keep backoff increasing across rapid cycles.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a low-severity availability concern by allowing repeated post-connect failures to stay at the minimum reconnect delay.
