## Summary

Keep open. Current main still does not implement the pending-inbound store or active-turn restart recovery, and the latest PR head is not merge-ready: earlier review items were addressed, but the branch still lacks a production capture path and has prompt-boundary and local-state hardening blockers.

## What This Changes

The PR adds a file-backed pending-inbound and active-turn store, startup replay/recovery hooks, embedded run lifecycle persistence, JSON-file helper changes, and related tests.

## Best Possible Solution

Finish the restart-resilience design end to end: capture inbound messages at the real channel or auto-reply boundary before drain rejection, replay external content with trusted: false, harden the pending store with bounded reads and safe backup behavior, and either align with or explicitly supersede the SQLite plan in #65650.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Persist drain-time inbound messages before replaying them:** `src/process/command-queue.ts:230-231`
  - body: The PR adds `isGatewayDraining()` but does not use it, or `writePendingInbound`, from any production inbound handler. When `enqueueCommandInLane` rejects work during restart drain, the inbound message is still not serialized, so startup replay only handles manually or test-created store entries.
  - confidence: 0.91
- **[P2] Mark replayed inbound previews untrusted:** `src/gateway/server-startup-post-attach.ts:428-431`
  - body: `eventText` includes stored sender/text fields from an external inbound message, but this call omits `trusted: false`. Since system events default to trusted, the next prompt is formatted as `System:` and owner-downgrade logic does not run for replayed user-controlled content.
  - confidence: 0.88
- **[P2] Bound and harden corrupt pending-store reads:** `src/infra/pending-inbound-store.ts:333-352`
  - body: `readPendingInboundFile` reads the entire store before validation and writes invalid raw content to a predictable `.bak` path. A large or symlinked state file can cause memory/disk amplification or redirect the backup write; use a hard byte cap and exclusive/no-follow bounded backup creation, or avoid raw backups.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: The PR introduces security-sensitive local state-file and prompt-boundary behavior that still needs hardening before merge; it does not change workflows, dependencies, lockfiles, or package publishing metadata.

Concerns:

- **[medium] Trusted replay of user-controlled inbound preview:** `src/gateway/server-startup-post-attach.ts:428`
  - body: Pending replay formats sender and text preview from external inbound data and enqueues it without `trusted: false`, so prompt formatting treats it as trusted `System:` content.
  - confidence: 0.88
- **[high] Predictable corrupt-store backup can follow symlinks:** `src/infra/pending-inbound-store.ts:352`
  - body: The parse-failure path writes raw content to `${filePath}.bak` with `fs.writeFile`; if the state directory is writable or misconfigured, a preexisting symlink can redirect that overwrite.
  - confidence: 0.79
- **[medium] Oversized pending store can exhaust memory or disk:** `src/infra/pending-inbound-store.ts:333`
  - body: The pending store is read entirely before parsing, and corrupt content is backed up in full, so a large local state file can force high memory use and disk amplification during startup or recovery.
  - confidence: 0.83
