## Summary

Keep this PR open. Current main still lacks the Discord cross-instance ownership registry that would solve the reported wrong-instance denial, but the proposed branch is currently dirty against main and still has concrete access-routing bugs that should be fixed before maintainer consideration.

## What This Changes

The PR adds a Discord shared instance-claim registry, provider refresh wiring, and message/native-command gates so sibling OpenClaw instances sharing one bot token can skip channels owned by another instance.

## Best Possible Solution

Keep or replace the Discord-plugin-owned fix under extensions/discord/src/monitor with a rebased implementation that keys claims by instance and bot, uses the same resolved channel/parent/name/wildcard semantics as the existing allowlist helpers, excludes disabled channels, serializes or isolates claims writes, and covers message preflight, autocomplete, and native command dispatch with focused regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Use the resolved Discord message channel for claims:** `extensions/discord/src/monitor/message-handler.preflight.ts:781`
  - body: `preflightDiscordMessage` already resolves `messageChannelId` for payloads where `message.channelId` is missing or stale, but the new ownership gate looks up `message.channelId`. In those cases the resolver returns `no-entry`, so the wrong instance can still fall through to the local allowlist denial this patch is meant to avoid. Pass the resolved `messageChannelId` instead.
  - confidence: 0.88
- **[P2] Scope native claim lookups to the active bot:** `extensions/discord/src/monitor/native-command.ts:466-472`
  - body: The provider writes claims under a bot key, but both native command claim lookups omit `botId`, so `resolveDiscordClaimOwnership` scans every bot bucket in the shared file. On hosts with multiple Discord accounts or bots, a different bot's claim can make this bot return `claimed-by-other` and silently drop valid autocomplete or slash-command traffic. Pass the interaction application/client bot id in both native call sites.
  - confidence: 0.9
- **[P2] Do not claim disabled channel overrides:** `extensions/discord/src/monitor/instance-claims.ts:218-221`
  - body: `buildClaimEntries` records every numeric channel key as owned, regardless of whether that channel override is disabled. Because the new claim gates run before local allowlist checks, sibling instances will skip a channel that the claiming instance later drops, effectively blackholing that channel. Filter claims through the same effective channel access semantics used by preflight/native command authorization.
  - confidence: 0.86
- **[P2] Serialize shared claims file updates:** `extensions/discord/src/monitor/instance-claims.ts:238-247`
  - body: `refreshDiscordClaims` loads, mutates, and rewrites the whole shared claims file without an inter-process lock, compare-and-swap, or per-instance file layout. Concurrent refreshes can overwrite another instance's fresh claims, creating `no-entry` windows where both instances may handle or deny the same channel. Add serialization or avoid multi-writer updates.
  - confidence: 0.84
- **[P2] Include name-based channel allowlist keys in claims:** `extensions/discord/src/monitor/instance-claims.ts:215-216`
  - body: Discord channel configs can match by channel name or slug, but the new claim builder discards every nonnumeric channel key. Named allowlist entries therefore produce no ownership claim, so sibling instances continue down their own local allowlist path and can reproduce the wrong-instance denial for those channels.
  - confidence: 0.87
- **[P2] Preserve wildcard ownership when channel overrides exist:** `extensions/discord/src/monitor/instance-claims.ts:204-206`
  - body: A guild channel map with `"*"` plus specific overrides still allows unlisted channels through the wildcard, but `claimsEntireGuild` is only true when `"*"` is the sole key. Those allowed wildcard channels are not claimed, so peer instances will not skip them and may still emit the local denial. Represent wildcard ownership together with explicit disabled/overridden exceptions.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: No dependency or workflow supply-chain changes were found, but the patch changes Discord access-routing decisions and still has concrete scoping and claims-integrity issues.

Concerns:

- **[medium] Native claims are not scoped to the active bot:** `extensions/discord/src/monitor/native-command.ts:466`
  - body: Native command ownership checks omit `botId`, allowing claims from another Discord bot/account in the same shared file to suppress this bot's valid interactions. This is an access-routing integrity issue for multi-account deployments.
  - confidence: 0.9
- **[medium] Shared claims writes can overwrite peer claims:** `extensions/discord/src/monitor/instance-claims.ts:238`
  - body: The read/modify/write claims file update has no cross-process serialization, so concurrent instances can drop each other's ownership data and temporarily lose the routing guarantee that prevents wrong-instance handling.
  - confidence: 0.84
