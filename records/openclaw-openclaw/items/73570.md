## Summary

Keep this PR open. Current main still only runs the outbound `message_sending` plugin hook and does not bridge it into the internal hook registry, while `message_sent` already has that internal-hook bridge. The PR targets a live gap tracked by #73542, but the proposed patch needs correction before merge because the provided review comments identify stale content and cancelled-send notification problems. Security pass did not find supply-chain-sensitive changes; the diff is limited to outbound delivery code.

## Best Possible Solution

Keep this PR open and amend it rather than closing it as obsolete. The best path is a tested internal `message:sending` bridge in outbound delivery that fires only for non-cancelled sends, uses the effective post-plugin hook content, documents the internal event key, and adds regression coverage for normal, rewritten, and cancelled `message_sending` cases before landing the fix for #73542.

## Risks / Open Questions

- The PR as written can fire the new internal notification for messages cancelled by `message_sending` plugin hooks.
- The PR as written can send stale pre-rewrite content to internal hook subscribers when a plugin hook modifies outbound content.
- The final implementation should define and document the intended internal event key; internal hooks currently register exact event strings such as `message:sent`, while the linked report uses the plugin-style `message_sending` spelling.
