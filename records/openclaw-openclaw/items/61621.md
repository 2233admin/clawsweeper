## Summary

Keep this issue open. Current main still has a Feishu-specific path that builds agent-facing quoted/replied message text with the reported `[message_id: ...]` line and `ou_...:` sender prefix, while the Feishu outbound dispatcher can send model-produced text without stripping that legacy inline form. The right cleanup is narrow and belongs in the bundled Feishu plugin plus targeted sanitization/tests.

## What This Changes

The issue requests that Feishu quoted/replied messages stop exposing OpenClaw internal metadata such as `message_id`, `sender_id`, and raw `ou_...` IDs in user-visible message content, or provide a config switch to hide them.

## Best Possible Solution

Move Feishu’s message and sender identifiers out of inline `BodyForAgent` text and rely on the existing structured inbound metadata fields for agent/tool context, then add a regression test that quoted/replied Feishu turns preserve `MessageSid`/`ReplyToBody` while no user-facing reply text contains `[message_id: ...]` or raw `ou_...:` labels.

## Reproduction Assessment

Yes. A focused unit reproduction can use the current Feishu handler or `buildFeishuAgentBody` with a quoted message: current tests already show `BodyForAgent` containing `[message_id: ...]\nou-...: ...`; a live Feishu UI check should still be part of final validation.

## Solution Assessment

No, current main is not the best end state because it keeps volatile Feishu identifiers in inline body text even though `finalizeInboundContext` already receives structured `MessageSid`, `SenderId`, `ReplyToId`, and `ReplyToBody`. The safer fix is Feishu-local cleanup plus targeted sanitizer/test coverage, not a new broad core migration.
