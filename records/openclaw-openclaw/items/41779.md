## Summary

Keep this issue open. Current main still advertises inline attachment fields on `message.send`, but the shared send path does not hydrate `buffer`/`filename` for `send`, and the Telegram runtime still forwards only URL/path-style media to `sendMessageTelegram`. The related open PRs #42674 and #63578 remain implementation candidates, not proof that the bug is fixed on main.

## Best Possible Solution

Keep this issue open until main either supports `buffer`/`filename`/`mimeType` attachments for `message.send` through the shared outbound path and Telegram plugin, or rejects those fields explicitly for `send` with a clear error. The best maintainer path is to review, update, and land or replace #42674/#63578 with focused regression tests for direct buffer sends and Telegram dispatch.
