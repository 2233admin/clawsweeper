## Summary

Keep open. Current main still routes gateway-backed agent tools through the WebSocket client, and this PR addresses a real self-contention surface, but the proposed local-dispatch shortcut bypasses current auth, scope, timeout, and serialization contracts and should not merge as-is.

## What This Changes

The PR adds a local gateway dispatch registry, registers it from gateway startup, and makes `callGatewayTool` prefer in-process dispatch for embedded calls except explicit `gatewayUrl` and `expectFinal` cases, with tests.

## Best Possible Solution

Keep the in-process dispatch direction, but implement it as an explicit gateway request adapter that preserves WebSocket-equivalent behavior: resolved token intent, least-privilege or explicit scopes, default-deny unknown methods, timeout handling, `expectFinal` support or deliberate fallback, and JSON-frame-equivalent params normalization.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Preserve gateway scopes before local dispatch:** `src/agents/tools/gateway.ts:151-158`
  - body: The local branch can return before `callGatewayTool` computes least-privilege scopes or honors explicit `extra.scopes`. `dispatchGatewayMethod` then runs under the ambient runtime client or a synthetic `operator.write` client, so embedded admin cron calls can fail and dynamically scoped callers can execute under a different authorization contract than the WebSocket path.
  - confidence: 0.9
- **[P1] Honor token and timeout options locally:** `src/agents/tools/gateway.ts:151-158`
  - body: Calls with `gatewayToken` or `timeoutMs` but no explicit URL still take the local path, so they never pass through `resolveGatewayOptions` or the WebSocket client. That silently ignores explicit auth intent and removes bounded timeout behavior used by existing node invoke and approval paths.
  - confidence: 0.88
- **[P2] Normalize params before dispatching locally:** `src/agents/tools/gateway.ts:155-157`
  - body: The local path forwards the live `params` object directly, while the WebSocket path validates a request frame and sends it through `JSON.stringify`. Optional properties set to `undefined` can therefore reach gateway validators locally even though the same call over WebSocket would omit them.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The diff adds a security-sensitive in-process gateway dispatch path without preserving the existing auth, scope, token, and timeout contract.

Concerns:

- **[high] Local dispatch bypasses auth and scope resolution:** `src/agents/tools/gateway.ts:151`
  - body: `callGatewayTool` can return from local dispatch before resolving explicit gateway tokens, least-privilege method scopes, or explicit scope overrides, so embedded calls may execute under ambient or synthetic gateway identity rather than the caller's intended authorization contract.
  - confidence: 0.88
- **[medium] Timeout bypass can hang privileged operations:** `src/agents/tools/gateway.ts:151`
  - body: Because `timeoutMs` is not applied on the local path, a stuck in-process handler can block an active agent turn instead of failing under the bounded timeout behavior used by the WebSocket request path.
  - confidence: 0.84
