## Summary

Keep open. Current main still has the reported empty-success notification gap, but this PR is not mergeable as-is because it is stale against the current `exec-event` heartbeat contract and leaves generated/public config documentation inconsistent with the proposed default change.

## What This Changes

The PR makes empty-output successful background exec completion notifications opt-out instead of opt-in, aligns runtime storage of that default, updates tests and config help/types, and adds a 50 ms exec-exit wake coalesce value.

## Best Possible Solution

If maintainers accept the default behavior change, rebase this work onto current main, keep completion wakes classified as `exec-event` with immediate coalescing, make the default consistently opt-out in both tool creation and runtime storage, regenerate schema/help artifacts, update public exec/background-process docs, and then adjust coding-agent guidance to rely on parent-owned completion handling where appropriate.

## Reproduction Assessment

Yes. Current main can be checked statically and through the existing no-op notification test path: omitted `notifyOnExitEmptySuccess` flows through `=== true`, and the default no-op case expects no completion event.

## Solution Assessment

No, not as currently patched. The default-on direction may be the right maintainer-owned fix, but the branch should preserve the current heartbeat routing contract and update generated schema plus public docs before merge.
