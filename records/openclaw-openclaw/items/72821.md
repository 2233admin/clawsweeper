## Summary

Keep open. Current main and the latest release do not contain the affected emotion-tag parser or `/emotions` surface, while the ambiguity is real in the still-open emotion-mode PR stack and needs a maintainer semantics decision before that stack lands.

## What This Changes

The issue asks OpenClaw to resolve a line-start bracket-pair ambiguity where proposed emotion-tag stripping can treat Markdown reference-link syntax such as `[fast][1]` as adjacent emotion tags.

## Best Possible Solution

Keep this as a review tracker for the emotion-mode stack. Before merging #72229, define the line-start rule explicitly and add focused tests for numeric references, definition-backed named references, named references without definitions, and adjacent allowlisted emotion pairs.

## Reproduction Assessment

No high-confidence reproduction exists on current main because the affected parser and `/emotions` command are absent. There is a high-confidence PR-head reproduction by source inspection: at line start, `hasNonWhitespaceBeforeOnLine` returns false, so `[fast][1]` is not protected as a Markdown reference pair.

## Solution Assessment

Unclear until maintainers choose the intended semantics. The issue correctly rejects a label-only heuristic; the maintainable fix is either definition-aware Markdown reference detection or an explicit product rule that line-start allowlisted pairs remain emotion tags when no reference definition exists.
