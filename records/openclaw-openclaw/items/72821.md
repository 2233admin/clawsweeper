## Summary

Keep open. The ambiguity is real in the proposed emotion-mode PR stack, but current main still has no emotion-tag parser, no `emotionMode` state, and no `/emotions` command surface, so this is not eligible for cleanup closure or a current-main repair.

## What This Changes

The issue asks for a product and parser decision for line-start `[label1][label2]` text so markdown reference links like `[fast][1]` are not misclassified as adjacent emotion tags by the proposed emotion-tag stripper.

## Best Possible Solution

Keep this as a PR-stack semantics tracker. If the emotion-mode work proceeds, define whether line-start allowlisted bracket pairs prefer emotion tags or markdown reference links, then update #72229 with either document-context reference-definition detection or an explicit rule plus focused tests for numeric references, named references with and without definitions, and adjacent emotion pairs.

## Reproduction Assessment

No, not against current main: the affected file and `/emotions` surface are absent. The branch-level reproduction is high-confidence by source inspection on #72229, where line-start reference-link detection is gated by preceding non-whitespace.

## Solution Assessment

Unclear until maintainers choose the product rule. The issue correctly rejects a label-only heuristic; the safer implementation direction is document-context reference-definition detection, or an explicit rule that line-start emotion pairs remain emotion tags when no definition exists.
