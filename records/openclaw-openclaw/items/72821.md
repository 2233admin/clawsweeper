## Summary

Keep open. Current main still does not contain the emotion-tag sanitizer or `/emotions` surface, while the concrete ambiguity lives on the still-open #72229/#72266 PR stack by the reporter and needs maintainer review of the intended markdown-vs-emotion semantics before it can be resolved.

## What This Changes

The issue asks maintainers to resolve line-start `[label1][label2]` parsing in the proposed emotion-tag sanitizer so markdown reference links such as `[fast][1]` are not stripped as adjacent emotion tags.

## Best Possible Solution

Keep this issue open as a focused review tracker for the emotion-mode PR stack. If the stack proceeds, resolve the ambiguity in the branch by either using document-level reference-definition context or explicitly choosing the line-start emotion-pair rule, then add tests for numeric references, named references, definition-backed references, and adjacent emotion pairs before merge.

## Reproduction Assessment

No high-confidence reproduction path exists on current main because the affected `emotion-tags` implementation and `/emotions` command are absent. The branch-level reproduction is concrete on #72229: `[fast][1]` at line start reaches the `hasNonWhitespaceBeforeOnLine` guard while existing tests preserve leading-text reference pairs and strip adjacent emotion tags.

## Solution Assessment

Unclear. The issue correctly rejects a label-content-only heuristic; the maintainable fix is either document-context-aware markdown reference detection or an explicit maintainer product decision that line-start emotion-word pairs win when no reference definition exists.
