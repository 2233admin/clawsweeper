## Summary

Keep open. Current main still lacks the requested unified ingress authorization contract, and representative channel/event paths continue to authorize through lower-level shared helpers or channel-local gates. The existing discussion already treats this as a security/design RFC, so cleanup closure would be premature.

## What This Changes

This issue requests a cross-channel `authorizeIngress()` contract that centralizes DM/group allowlist and event-kind authorization while leaving channel-specific identity/context adaptation to plugins.

## Best Possible Solution

Keep the RFC open until maintainers and secops approve an additive ingress authorization contract. The desired end state is one shared policy decision point with channel-specific identity/context adapters, representative migrations across message/reaction/pin/interaction/callback/slash/voice/media ingress, and contract tests that lock the shared behavior without bypassing plugin ownership boundaries.
