## Summary

Keep this issue open. Current main still has a real contract mismatch: the agent prompt and one Telegram doc example advertise legacy `buttons=[[{text,callback_data,style?}]]` / top-level `buttons`, while the current message tool schema exposes structured `presentation` blocks and Telegram runtime derives inline keyboards from presentation/interactive payloads, not raw final-answer JSON text.

## Best Possible Solution

Fix this as a contract-alignment bug. Update the agent prompt and Telegram docs to steer models to `message(action="send")` with `presentation.blocks[{type:"buttons"}]`, keep the existing Telegram presentation renderer, and add regression coverage for the prompt/tool schema plus final delivery behavior. Only add legacy `[[...callback_data...]]` parsing or top-level `buttons` compatibility if maintainers explicitly want that behavior and it is validated tightly.
