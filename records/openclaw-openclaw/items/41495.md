## Summary

Keep this issue open. Current main still has the reported Telegram inline-button contract mismatch: the agent prompt and a Telegram docs example still advertise legacy/top-level `buttons` with `callback_data`, while the message tool schema exposes `presentation.blocks[{type:"buttons"}]` and the Telegram runtime derives inline keyboards from normalized presentation/interactive payloads. The prior ClawSweeper comment already found this mismatch, and the same evidence is still present on current main at `7a23b2d945cc2d2e5c452e4641213e71de8dcc66`.

## Best Possible Solution

Fix this as a contract-alignment bug. Update the agent prompt and Telegram docs to tell models to call `message(action="send")` with `presentation.blocks[{type:"buttons", buttons:[{label,value,style?}]}]`, keep Telegram's existing presentation renderer, and add regression coverage for the prompt/schema plus Telegram final delivery behavior. Only add legacy raw `[[...callback_data...]]` or top-level `buttons` compatibility if maintainers explicitly want that behavior and it is tightly validated.
