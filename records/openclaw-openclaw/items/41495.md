## Summary

Keep this issue open. Current main still advertises legacy Telegram `buttons=[[{text,callback_data,style?}]]` guidance in the agent system prompt and locks that wording in a prompt test, while the shared `message` tool schema exposes structured `presentation.blocks[].buttons` and the Telegram runtime derives inline keyboards from `presentation` or `interactive` payloads. Raw assistant text containing JSON-like button rows is still treated as message text, so the reported Gemini failure is not fixed on main.

## Best Possible Solution

Keep this issue open and fix the contract mismatch: update the agent system prompt and prompt regression test to instruct models to call `message(action="send")` with `presentation.blocks[{type:"buttons", buttons:[{label,value,style?}]}]`, align Telegram docs with the public message-presentation contract, and keep Telegram runtime behavior centered on structured presentation/interactive payloads. Avoid adding a freeform raw-text button parser unless maintainers explicitly choose that compatibility path and validate it separately.
