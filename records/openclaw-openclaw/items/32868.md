## Summary

Keep open. Current main still has the same-channel block-streaming await gap: `sendBlockReply` is boolean-only, the dispatcher schedules actual channel delivery on an internal `sendChain`, and `dispatch-from-config` still fire-and-forgets the same-channel block path while awaiting only the cross-channel route. This remains distinct from open PR #65541, which covers disabled block streaming rather than enabled `text_end` same-channel delivery ordering.

## What This Changes

The issue requests that same-channel `text_end` block-streaming replies be delivered to the channel before the next tool execution proceeds.

## Best Possible Solution

Keep this issue open as the canonical core bug. The best fix is a focused reply-dispatcher/core change that makes accepted same-channel block delivery awaitable, waits for that delivery before `blockStreamingBreak: "text_end"` lets tool execution proceed, preserves dropped-payload and cross-channel semantics, updates ACP/dispatcher contracts and mocks, and adds a regression test proving the actual dispatcher `deliver` promise settles before tool-start processing continues.

## Repair Work Prompt

Fix the same-channel block-streaming delivery ordering bug from this issue. Current main's `ReplyDispatcher.sendBlockReply` is boolean-only and returns after enqueue; `dispatch-from-config` awaits cross-channel `sendPayloadAsync` but fire-and-forgets same-channel dispatcher delivery. Implement a narrow core fix that makes accepted same-channel block delivery awaitable, preferably by returning a per-enqueued delivery promise or equivalent while preserving `false` for dropped/silent payloads. Await that delivery before `onBlockReply` resolves so `blockStreamingBreak: "text_end"` flushes complete before `handleToolExecutionStart` continues. Preserve the already-awaited cross-channel path, avoid channel-specific parsing/workarounds, and do not change disabled block-streaming behavior covered by #65541. Update `dispatch-acp-delivery`, `ReplyDispatcher` types, mocks, and tests consistently. Add regression coverage proving the actual dispatcher `options.deliver` promise settles before the next tool-start processing, not only an async `onBlockReply` mock. Add a changelog entry if the PR changes user-visible delivery ordering, with proper contributor attribution.
