## Summary

Keep this issue open. Current main at b85edb3f0cf6 still has the same same-channel block-streaming delivery gap: the block pipeline awaits the `onBlockReply` callback, but that callback only enqueues same-channel dispatcher delivery and returns before the channel `deliver` promise settles. The issue discussion has independent WhatsApp, Telegram, Discord, and WeCom confirmations, and the direct fix PRs linked from the timeline were closed without merging.

## Best Possible Solution

Keep this issue open as the canonical core bug. The best fix is a focused core change that makes same-channel block delivery awaitable, awaits it before tool execution proceeds for `blockStreamingBreak: "text_end"`, preserves the existing awaited cross-channel path, updates dispatcher/ACP/plugin SDK contracts and mocks consistently, and adds a regression test proving the actual channel `deliver` promise completes before the next tool starts.

## Clownfish Work Prompt

Fix the same-channel block streaming delivery ordering bug tracked by this issue. Current main still lets `dispatch-from-config` resolve `onBlockReply` after `dispatcher.sendBlockReply(...)` merely enqueues delivery, so `blockStreamingBreak: "text_end"` can start the next tool before the block reply reaches the channel. Implement a focused core fix: make same-channel block reply delivery awaitable, await it from the generic same-channel `onBlockReply` path, preserve cross-channel `sendPayloadAsync` behavior, update dispatcher/ACP/plugin SDK contract types and mocks consistently, and add regression coverage proving the actual dispatcher `deliver` promise completes before `tool_execution_start` continues. Do not broaden this into message-tool batching, preview streaming, channel-specific media parsing, or disabled-block-streaming product changes. Check adjacent open #65541 before editing because it touches nearby block delivery behavior. Add a changelog entry because this is user-visible channel delivery ordering.
