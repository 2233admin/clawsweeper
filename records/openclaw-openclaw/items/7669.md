## Summary

Keep this issue open. Current main still does not provide a first-class npm `dev` dist-tag publish path: the public npm release workflow only allows `beta` and `latest`, real publishes require release tags, the publish helper and release-plan types only model stable/beta, tests cover stable/beta only, and the live npm registry currently exposes only `beta` and `latest` for `openclaw`.

## Best Possible Solution

Keep this issue open until maintainers either implement a first-class npm `dev` snapshot channel for moving `main` builds, including workflow trigger/input support, snapshot versioning, publish-helper and release-plan support, tests, post-publish verification, and docs; or explicitly retire npm `dev` snapshots and update release-channel docs, updater behavior, installer/FAQ text, and release docs so `dev` is documented as git-only or otherwise unsupported for npm.

## Risks / Open Questions

- Implementing npm `dev` snapshots touches security-sensitive release infrastructure: dist-tag mutation currently lives in a private workflow because it needs `NPM_TOKEN`, while public npm publish is OIDC-only.
- A moving-main npm `dev` tag needs explicit versioning and overwrite policy; npm cannot republish the same package version, so snapshots need a durable version scheme and cleanup/promotion story.
- Closing now would leave docs and package-mode update behavior pointing at an npm `dev` tag that the live registry does not expose.
