## Summary

Current main does not implement the requested Feishu pushback when a Web Control UI follow-up takes over a Feishu-started main-session conversation. The Control UI sends normal chat turns with `deliver: false`, and the gateway explicitly treats non-delivery WebChat turns as internal. Tests also assert that UI clients on main sessions do not inherit an external delivery route, even when delivery is enabled. The related supplied matches are other channel-specific routing fixes, not a canonical duplicate for this Feishu/Web Control UI cross-surface case.

## Best Possible Solution

Keep this issue open. The best path is a maintainer product decision on whether Web Control UI should be able to continue and deliver into the original Feishu conversation; if yes, implement an explicit, permissioned route inheritance or delivery mode in `chat.send`/Control UI that preserves existing cross-channel safety, with regression coverage for Feishu main-session and Feishu channel-scoped follow-ups during active runs.
