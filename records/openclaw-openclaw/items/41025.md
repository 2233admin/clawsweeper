## Summary

Keep this PR open. Current main still flattens remote markdown images in the Control UI markdown renderer, and the served Control UI CSP/docs still describe and enforce a no-remote-fetch image policy. The PR is a focused implementation candidate for open issue #40945, but it is incomplete as-is because it changes only the renderer/tests while leaving CSP, docs, and the remote-media privacy policy unresolved. A commenter also explicitly asked that the feature not be closed, and #64607 remains the broader inline-media product discussion.

## Best Possible Solution

Keep this PR open as an implementation candidate for #40945, but require a maintainer-approved design before merge. A complete solution should align the markdown renderer, Control UI CSP, docs, and tests around an explicit remote-media policy, likely HTTPS-only at minimum and preferably an allowlist, opt-in setting, managed-media proxy, or documented privacy tradeoff. If maintainers choose to preserve the current no-remote-fetch policy, close the implementation attempts only after recording that product/security decision on #40945 or the broader inline-media issue #64607.

## Risks / Open Questions

- Security/privacy review remains unresolved: rendering arbitrary remote `http://` or `https://` images lets chat/model content trigger third-party browser requests that can disclose IP address, user agent, and read timing.
- The PR allows plain `http://` images; if CSP is widened, HTTPS Control UI deployments may hit mixed-content and integrity/privacy problems.
- A renderer-only merge would likely still not display remote markdown images in the normal served Control UI because current Gateway CSP blocks remote `img-src`.
- No CI workflow, dependency, lockfile, release, or package-publishing changes are visible in the supplied PR diff; the security concern is the new browser network-fetch surface, not supply-chain execution.
