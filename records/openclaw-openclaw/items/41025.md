## Summary

Keep this PR open. Current main still does not implement the PR’s intended behavior: Control UI markdown image rendering only emits inline images for `data:image/...` sources, while `http://` and `https://` markdown images are flattened to alt text. The PR is still a focused candidate for open #40945, but it needs maintainer product/security direction because the current CSP, docs, tests, and adjacent export-html behavior intentionally preserve a no-remote-image-fetch posture.

## Best Possible Solution

Keep this PR open as an implementation candidate for #40945. A complete maintainer-approved solution should make an explicit remote-media policy decision and align the markdown renderer, served CSP, docs, tests, and adjacent export/share surfaces around that policy, likely HTTPS-only at minimum and preferably an allowlist, opt-in setting, managed-media proxy, `referrerpolicy="no-referrer"`, and the existing `markdown-inline-image` styling. If maintainers choose to preserve the current no-remote-fetch policy, record that on #40945 or #64607 before closing focused implementation attempts.

## Risks / Open Questions

- Merging the PR as-is would let chat/model output trigger arbitrary browser fetches for `http://` and `https://` image URLs if CSP is relaxed, exposing viewer IP address, user agent, and read timing to third-party hosts.
- The PR allows `http://` as well as `https://`, which creates mixed-content and downgrade concerns for HTTPS Control UI deployments.
- The PR patch shown in the provided context changes only the renderer/tests, so it does not align the served CSP, docs, changeloged policy, export-html posture, or privacy mitigations such as allowlisting, proxying, or `referrerpolicy="no-referrer"`.
- The PR’s emitted remote `<img>` markup in the provided patch omits the current `markdown-inline-image` class, so it would not use the established Control UI image styling unless rebased/adjusted.
