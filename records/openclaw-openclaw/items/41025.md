## Summary

Keep this PR open. Current main still flattens remote HTTP/HTTPS markdown images in Control UI chat, while the PR is a plausible implementation candidate for #40945. It needs maintainer product/security review because the renderer-only change is now stale against the current markdown-it renderer and would conflict with the current CSP/docs no-remote-fetch policy.

## Best Possible Solution

Keep this PR open as a focused implementation candidate for #40945, but require maintainer product/security direction before merge. The final path should either intentionally support remote images by updating the current markdown-it renderer, CSP, tests, and docs with mitigations such as HTTPS-only, referrerpolicy="no-referrer", lazy loading, allowlisting, or proxying; or document that remote image fetches remain intentionally blocked and close implementation attempts in favor of #40945 or the broader media discussion in #64607.

## Risks / Open Questions

- Merging the renderer-only patch would not fully render remote images under the current served CSP, which still omits https: and http: from img-src.
- If CSP is relaxed, arbitrary chat/model content could trigger third-party image requests from the operator browser, exposing IP address, user agent, and read timing to the image host.
- The PR allows http:// as well as https://, which adds mixed-content and downgrade concerns for HTTPS Control UI deployments.
- The patch is stale against current main's markdown-it renderer and would need to be adapted from htmlEscapeRenderer.image to md.renderer.rules.image.
- Current docs and tests explicitly enforce a no-remote-fetch posture, so any accepted implementation needs aligned renderer tests, CSP tests, docs, and product/security sign-off.
