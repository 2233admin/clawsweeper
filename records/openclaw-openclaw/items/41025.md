## Summary

Keep this PR open. Current main still intentionally flattens remote markdown images to alt text, the regression test and Control UI CSP still enforce that no-remote-fetch behavior, and the docs describe remote http(s) image URLs as rejected. The PR is a focused implementation candidate for open issue #40945, but it needs maintainer product/security direction before merge because a renderer-only change conflicts with current CSP/docs and introduces arbitrary browser remote-image fetches.

## Best Possible Solution

Keep this PR open as an implementation candidate for #40945, but require maintainer-approved design before merge. A complete solution should align the markdown renderer, Control UI CSP, docs, and tests around an explicit remote-media policy, likely HTTPS-only at minimum and preferably an allowlist, opt-in setting, managed-media proxy, and/or referrerpolicy="no-referrer". If maintainers choose to preserve the current no-remote-fetch policy, record that product/security decision on #40945 or #64607 before closing implementation attempts.

## Risks / Open Questions

- Merging this PR as-is would conflict with the current served Control UI CSP and docs; remote images may still be blocked in normal served Control UI unless the CSP and policy are changed too.
- Allowing arbitrary remote image URLs in chat/model output creates privacy risk: third-party hosts can observe viewer IP address, user agent, and read timing. The PR also allows http:// URLs, which adds mixed-content and downgrade concerns on HTTPS deployments.
- Closing this PR now would discard a focused implementation candidate while open issue #40945 still tracks the exact requested behavior.
- The PR is older than current main and would likely need rebase/design updates around the current markdown-inline-image class, CSP tests, and documented policy.
