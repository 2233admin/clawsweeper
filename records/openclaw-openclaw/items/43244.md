## Summary

Keep this PR open. Current main still validates `baseHash` against an early gateway snapshot, computes the replacement config from that same snapshot, then commits a precomputed payload without carrying the caller's `baseHash` into the lower-level optimistic writer. The PR remains relevant, but it needs a current-main rebase and the review gaps addressed before it should land.

## Best Possible Solution

Keep this PR open as a still-relevant implementation candidate. Rebase or reimplement it on current main so `config.set`, `config.patch`, and `config.apply` serialize the full read/base-hash-check/validate/write sequence, or pass the caller `baseHash` into a fresh lower-level mutation that recomputes from current state before writing. Preserve SecretRef and shared-auth behavior, map queue rejection errors to RPC failures, and add stale concurrent-write tests for all three methods.

## Risks / Open Questions

- The submitted patch was written against older handler code and must be rebased over current SecretRef preflight plus shared-auth generation/disconnect behavior.
- The queued-write approach needs explicit handling for command-lane rejection errors so gateway clients receive an RPC failure instead of waiting indefinitely during restart/drain windows.
- Race coverage should include `config.set` as well as `config.patch` and `config.apply`.
- The broader macOS Talk Mode report #64973 may still require separate handling for direct non-gateway config writers; this PR is scoped to gateway RPC handlers.
