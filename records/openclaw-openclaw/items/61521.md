## Summary

Keep this PR open for maintainer review. Current main still lacks the proposed post-assembly channel setup validation hooks, but the branch is not merge-ready because the deferred wizard path can return the wrong account id, can reject valid existing-credential update flows by revalidating sparse draft input, and hardcodes VK-specific CLI fields into shared core surfaces that are now plugin-metadata driven.

## What This Changes

The PR adds deferred candidate-config validation hooks for channel setup, threads VK-oriented community credential fields through `openclaw channels add`, and adds a focused async validation wizard test.

## Best Possible Solution

Add a generic, SDK-safe candidate-config validation seam in the current channel setup files, with candidate config and resolved account id passed consistently through wizard and CLI flows. VK-specific command fields should be supplied by the channel/plugin metadata rather than hardcoded in core, and raw field validation should remain per-step while complete/async validation operates on the assembled candidate config.

## Reproduction Assessment

Yes. The PR defects are reproducible by inspecting the branch source: the deferred wizard path computes `draftAccountId` and returns the original `accountId`, and the new candidate validator calls raw `validateInput` with sparse draft input even though existing credential keep flows do not populate that draft.

## Solution Assessment

No. The proposed direction is useful, but this patch is not the best implementation as written; a current-main rebase should return the resolved account id, validate the assembled candidate without reusing sparse raw-input credential checks, and keep provider-specific CLI options plugin-owned.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Return the resolved deferred account id:** `src/channels/plugins/setup-wizard.ts:970`
  - body: When `deferApplyUntilValidated` is enabled, this path computes `draftAccountId` from the assembled input and validates the candidate config with it, but still returns the original `accountId`. Any later account recording, binding prompt, or config-change hook can then refer to a different account than the one the candidate config was applied under.
  - confidence: 0.88
- **[P2] Avoid revalidating sparse draft input:** `src/channels/plugins/setup-wizard.ts:331-335`
  - body: `draftInput` only contains fields explicitly changed during prompts. If a user keeps an existing credential, the candidate config can still be valid while `validateInput` sees no credential and rejects it, especially for adapters that require credentials in raw setup input. Keep raw validation at input steps and use the complete/async hooks for candidate-config validation here.
  - confidence: 0.84
- **[P2] Keep VK CLI fields out of the shared command:** `src/cli/channels-cli.ts:168-169`
  - body: Adding `--community-id` and `--community-access-token` directly to the core `channels add` command makes a VK-specific setup surface part of shared core. Current main already gathers channel add options from bundled plugin metadata, so the VK plugin should contribute these flags through that path instead of hardcoding provider fields here.
  - confidence: 0.78
