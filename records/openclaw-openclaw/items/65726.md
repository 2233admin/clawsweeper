## Summary

Keep open. Current main still does not provide an ACP binding/runtime `notifyPolicy` carrier, so configured ACP-bound conversations still fall back to the task registry default terminal notification. The bound-conversation routing paths retarget inbound messages to the ACP session, but I did not find a thread-ownership guard that suppresses duplicate outbound replies from a separate session targeting the same channel thread.

## What This Changes

Request to add ACP binding/runtime `notifyPolicy` config and a bound-thread dedup guard so ACP-managed conversations do not show task-completion notices or duplicate replies.

## Best Possible Solution

Add `notifyPolicy` to the typed ACP runtime and binding config contract, strict schema, docs, binding spec/metadata, materialization, and ACP task creation so configured bound sessions can choose `silent`. Separately define a channel/account/conversation/thread ownership claim for active ACP bindings and apply it only to automated duplicate delivery paths, with an escape hatch for deliberate operator sends.
