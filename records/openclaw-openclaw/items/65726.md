## Summary

Current main still lacks both requested pieces: ACP binding/runtime config cannot carry `notifyPolicy`, and the existing binding/dedupe paths do not implement a policy guard for suppressing duplicate replies from a separate session targeting an already ACP-bound thread.

## What This Changes

Request to add configurable ACP binding/runtime task notification policy plus a bound-thread guard that detects active ACP sessions and prevents duplicate Slack replies.

## Best Possible Solution

Extend the typed ACP binding/runtime config contract with a task notify policy, carry it through schema/help/docs, binding normalization/materialization, session metadata where needed, and ACP task creation before records are written. Then add an explicit active-binding ownership/claim mechanism for channel/account/conversation/thread delivery so duplicate automated replies can be suppressed without blocking deliberate human or operator sends.
