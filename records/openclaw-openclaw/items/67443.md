## Summary

Keep open. The PR targets a real status/session gap that current main has not made obsolete, but the proposed patch currently introduces two blocking regressions in OpenAI-compatible model alias handling and default session context-token reporting. The existing Codex and Greptile review comments are consistent with the current code and PR-head source inspection.

## What This Changes

The PR shares status/session context-token and model-resolution helpers, allows `openclaw/<provider>/<model>` compat model overrides, derives `session_status` group activation from channel config, and adds focused regression tests.

## Best Possible Solution

Revise the branch so the shared resolver keeps the old context-token precedence, `openclaw/<defaultAgentId>` remains valid without an explicit `agents.list`, and the group activation fallback stays config-aware with targeted coverage for all three behaviors.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve default agent aliases without agents.list:** `src/gateway/http-utils.ts:223-230`
  - body: This now rejects `openclaw/main` unless `main` appears in `cfg.agents.list`. When no agent list is configured, OpenClaw still resolves `main` as the default agent and `/v1/models` advertises `openclaw/main`, so compat requests using that listed model id and `/v1/models/openclaw/main` become invalid.
  - confidence: 0.93
- **[P2] Pass the default context cap into the shared resolver:** `src/gateway/session-utils.ts:948-953`
  - body: `getSessionDefaults` used to honor `cfg.agents.defaults.contextTokens` and fall back to `DEFAULT_CONTEXT_TOKENS`. The new call passes neither value, so users with a configured cap can see the wrong TUI default and users without discovery/config data can get `null` instead of the 200k fallback.
  - confidence: 0.94
