## Summary

Keep open. Current main has useful task-ledger cleanup and configurable session-store maintenance, but it does not fully resolve the reported memory shape: a 471-entry, 26-day session store remains below the default age/count cleanup thresholds, disk-budget eviction is opt-in, `skillsSnapshot` is still persisted inline per session entry, and under-cap stores are still parsed eagerly at load time.

## What This Changes

The issue requests bounding Gateway memory by pruning or archiving accumulated session/task records, reducing duplicated `skillsSnapshot` data, detecting zombie tasks, and avoiding eager loading of historical session metadata.

## Best Possible Solution

Keep the issue open for a maintainer-owned storage design that preserves the current task sweeper, then decides how to address the remaining session-store memory shape: compatible skill-snapshot interning or lazy lookup, stricter default retention or disk-budget behavior, and load-time handling that avoids keeping large historical stores resident unnecessarily.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores `skillsSnapshot` inline per session and the reported 471-entry, 26-day store is below the default cleanup thresholds. No live heap reproduction was attempted in this read-only sweep.

## Solution Assessment

Unclear. Current main's task cleanup and optional session disk budget are necessary partial fixes, but they are not the complete solution because duplicated snapshots and under-threshold session stores remain; the narrowest maintainable next step needs a storage-policy decision first.
