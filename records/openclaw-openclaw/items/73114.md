## Summary

Keep open. Current main has partial guardrails for task cleanup and session-store maintenance, but it still does not fully address the reported memory shape: the cited 471-entry, 26-day session store remains below the default age/count thresholds, and each session entry still persists the full skills snapshot inline.

## What This Changes

The issue asks OpenClaw to reduce Gateway memory growth by bounding accumulated session/task records, deduplicating per-session skills snapshots, detecting zombie tasks, and avoiding eager historical session loading.

## Best Possible Solution

Keep the task/session sweepers that already landed, then choose a compatible storage policy for the remaining memory bloat: likely stricter detached-session retention or disk-budget defaults plus skill-snapshot interning or lazy metadata loading that preserves transcript, trajectory, and skill-runtime behavior.

## Reproduction Assessment

Yes for the remaining static gap: current main still stores skillsSnapshot inline per session, and the default 30-day/500-entry maintenance thresholds would not prune the reported 471-entry, 26-day store. No live heap reproduction was run in this read-only sweep.

## Solution Assessment

Unclear. The current task TTL and zombie reconciliation are useful partial fixes, but they are not enough to close the broader issue while session snapshot duplication and under-threshold eager loading remain.
