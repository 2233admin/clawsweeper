## Summary

Keep open. Current main still awaits automatic TTS synthesis inline before reply delivery on final, tool, block, and streamed-final fallback paths, and Telegram ordinary updates remain serialized by chat/topic. The related #42539 is narrower Telegram text-plus-voice UX work, while #48179 is broader voice pipeline performance; neither supersedes this specific non-blocking auto-TTS delivery request.

## What This Changes

Request non-blocking automatic TTS delivery so generated audio does not delay visible text replies or later inbound messages.

## Best Possible Solution

Add a shared automatic TTS delivery policy that keeps the current synchronous behavior as the compatibility default while supporting a text-first or background mode with bounded concurrency and channel-safe follow-up media delivery. Align the Telegram-specific split-text voice UX with #42539 and the broader voice pipeline goals in #48179.

## Reproduction Assessment

Yes. A focused reproduction is to enable automatic TTS with a slow OpenAI-compatible provider on Telegram, trigger a multi-chunk reply, and observe that current dispatch awaits maybeApplyTtsToReplyPayload before reply routing while ordinary same-topic Telegram messages remain serialized.

## Solution Assessment

Unclear until maintainer design. The requested direction is valid, but the maintainable fix should likely be a shared TTS delivery policy and background delivery abstraction rather than a Telegram-only workaround or unbounded parallel TTS calls.
