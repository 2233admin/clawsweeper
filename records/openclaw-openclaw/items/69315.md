## Summary

Keep open. Current main still applies auto-TTS synchronously in the reply delivery path: final/tool/block payloads await TTS generation before normalized delivery, speech-core awaits provider synthesis, and Telegram ordinary inbound updates are serialized by chat/topic while the handler is still running. I found no `messages.tts.delivery`/async/background mode or canonical duplicate that covers the non-blocking queue problem.

## Best Possible Solution

Keep the issue open and implement this in the shared speech-core/reply-dispatch boundary, likely with a backwards-compatible `messages.tts.delivery` policy defaulting to current sync behavior. The async path should deliver visible text immediately, queue/bound TTS synthesis with concurrency and cancellation semantics, then deliver or edit channel-safe voice/audio follow-ups with regression coverage for `dispatch-from-config`, ACP delivery, and Telegram per-topic serialization.

## Risks / Open Questions

- Async TTS needs explicit idempotency, cleanup, and late-delivery failure handling so background audio does not duplicate, leak files, or attach to the wrong turn.
- Telegram edit-versus-follow-up semantics, ACP and non-ACP dispatch paths, and block/tool/final payload ordering need to stay aligned if TTS leaves the main reply path.
