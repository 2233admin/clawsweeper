## Summary

Keep open. Current main still awaits auto-TTS synthesis before routing final, tool, block, and streamed-final audio payloads, and Telegram still serializes ordinary updates by chat/topic, so slow TTS can still delay later same-topic inbound messages. The related #42539 is open but narrower: it asks for separate Telegram text and voice delivery, not a full non-blocking/background TTS pipeline.

## Best Possible Solution

Keep this issue open for maintainer design. The likely direction is a shared messages.tts delivery policy plus a background delivery path that sends visible text immediately, generates TTS with bounded concurrency, and posts or attaches the resulting voice/audio through existing channel delivery abstractions without blocking the active inbound turn. Coordinate with #42539 so Telegram text-plus-voice UX and async/background delivery do not diverge.
