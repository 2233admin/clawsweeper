## Summary

Keep this issue open. Current main still uses file-backed `node:sqlite` `DatabaseSync` for builtin memory and task registries, exposes only the `sqlite` memory-store driver, and has no better-sqlite3/env-driver/runtime-lock-probe fallback for the reported ARM64 container `SQLITE_BUSY` path. Recent QMD-to-builtin fallback work is narrower memory-search resilience and still falls back to the builtin SQLite manager, not to an alternate storage engine.

## Best Possible Solution

Keep this open as core storage resilience work. The best path is a shared SQLite storage seam or runtime file-lock probe used by builtin memory, QMD read paths where applicable, task registry, and Task Flow registry, with either an automatic fallback or explicit supported driver setting, regression coverage for first-statement `SQLITE_BUSY`, and matching config/docs.
