## Summary

Keep open. The reported authorization gap is real and the PR moves in the right direction, but the current branch is not ready to merge because the new inline coding-tools call does not preserve all runtime and policy context needed for plugin tools and group-scoped tool policy.

## What This Changes

The PR switches inline skill `command-dispatch: tool` execution from the OpenClaw tool factory plus owner-only filtering to `createOpenClawCodingTools`, passes config/sender/model/message context, and adds regression coverage for non-owner messaging-profile dispatch.

## Best Possible Solution

Revise this branch or a narrow replacement so inline skill tool dispatch uses `createOpenClawCodingTools` with the same complete context as normal coding runs and the old inline dispatch path: gateway subagent binding, message route, target/group fallback, sender identity fields, model/provider, session policy, and regression coverage for restricted non-owner and group-policy cases.

## Reproduction Assessment

Yes. The source-level reproduction is clear: current main uses `createOpenClawTools` plus only owner-only filtering for inline tool dispatch, while the PR's intended test path exercises the missing profile filtering; the remaining review findings can be reproduced by asserting the arguments passed into `createOpenClawCodingTools`.

## Solution Assessment

No. The direction is the right maintainable seam, but this implementation is incomplete because it drops existing gateway binding and incomplete group/sender policy context; the safer fix is to thread the full existing runtime context through the central coding-tool policy factory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve gateway subagent binding for plugin tools:** `src/auto-reply/reply/get-reply-inline-actions.ts:198`
  - body: The old inline dispatch path opted plugin tools into gateway subagent binding with `allowGatewaySubagentBinding: true`. This new `createOpenClawCodingTools` call does not pass that option, and `pi-tools` only forwards it when present, so plugin-backed `command-dispatch: tool` commands that use the gateway subagent or node runtime will now fail as unavailable outside the Gateway. Pass the opt-in through this call.
  - confidence: 0.88
- **[P2] Pass the full group policy context:** `src/auto-reply/reply/get-reply-inline-actions.ts:208-215`
  - body: This call is now responsible for enforcing group tool policy, but it only passes `sessionEntry?.groupId` and `senderId`. Group policy resolution also uses the ctx-derived group fallback plus sender name, username, and E.164 values, which normal tool inventory and agent runs already pass. In group sessions keyed by those fields, inline dispatch can miss a deny/allow rule and expose a tool that should be filtered. Thread the full group and sender context into `createOpenClawCodingTools`.
  - confidence: 0.84
