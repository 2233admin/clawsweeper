## Summary

Keep open. Current main still has the reported Gemini-behind-proxy gap, but the branch is stale against the moved schema-normalization code and has a concrete correctness issue: the new modelId-triggered Gemini path would skip existing modelCompat unsupported-keyword stripping for custom providers.

## What This Changes

This PR adds modelId-based Gemini detection to tool parameter schema normalization and tests custom OpenAI-compatible proxy cases.

## Best Possible Solution

Land a rebased version in `src/agents/pi-tools-parameter-schema.ts` that detects Gemini from normalized modelId as well as provider, composes Gemini cleaning with `modelCompat` keyword stripping, and extends tests for proxy Gemini plus compat-declared unsupported keywords.

## Reproduction Assessment

Yes. A focused static reproduction is clear: call `normalizeToolParameterSchema()` with a custom proxy provider and a Gemini modelId; current main accepts `modelId` but only checks `modelProvider`, while the PR tests encode the missing path.

## Solution Assessment

No. The direction is right, but the proposed implementation is not the best landing shape because it targets an older file and would bypass existing compat keyword stripping; the safer fix is to compose both cleaners in the current implementation file.
