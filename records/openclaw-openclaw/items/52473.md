## Summary

Keep open. Current main still resolves explicit skill slash commands, but normal SKILL.md-backed skills are converted into a regular model prompt instead of loading and injecting the selected SKILL.md content. The docs and system prompt still describe model-mediated on-demand reading, so the deterministic activation path requested here is not implemented.

## Best Possible Solution

Keep this issue open and implement a deterministic core activation path for explicit skill invocation. After `/skill <name>` or a direct user-invocable skill command resolves to a normal SKILL.md-backed skill, the gateway should load that selected skill through the existing skills subsystem and inject its SKILL.md content into the agent run context before the user args. Preserve `command-dispatch: tool`, keep authorization and skill visibility gates intact, and add focused regression coverage for direct aliases, `/skill <name>`, disabled model invocation, transcript/provider compatibility, and docs.

## Clownfish Work Prompt

Implement deterministic SKILL.md loading for explicit skill slash commands. Observable bug: when a user invokes a normal user-invocable skill via `/skill <name> [args]` or a direct skill command, OpenClaw currently rewrites the request to `Use the "<skill>" skill for this request.` and relies on the model to voluntarily read SKILL.md. Expected behavior: for explicit skill invocation only, after the command resolves and authorization/skill visibility gates pass, load the selected normal SKILL.md-backed skill content through the existing skills subsystem and inject it into the agent run context before the user args so the model receives the full skill instructions deterministically. Preserve existing `command-dispatch: tool` direct execution, preserve Claude-bundle promptTemplate behavior unless the code shows it must share the same path, preserve per-agent skill filters, `user-invocable`, `disable-model-invocation`, sandbox skill mirroring, and owner/authorization checks. Do not change natural-language skill matching. Keep core extension-agnostic and avoid bundled skill IDs. Add focused regression tests for direct skill aliases, `/skill <name>`, skills hidden from available_skills via `disable-model-invocation`, tool-dispatch unaffected behavior, transcript/provider-compatible context shape, and docs updates. Changelog: add a user-facing fix entry if behavior changes ship, with reporter attribution per repo rules. Suggested validation: targeted Vitest for skill command resolution/inline actions/system prompt or runner context, targeted docs/format checks, then `pnpm check:changed` in Testbox before handoff if code/docs change.
