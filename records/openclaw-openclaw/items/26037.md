## Summary

Keep this issue open. Current main has bundled Qwen/Bailian Coding Plan setup and a Qwen-owned `enable_thinking` stream wrapper, but the requested reasoning path for the named Coding Plan models is still incomplete: `qwen3.5-plus`, `glm-5`, and `kimi-k2.5` remain non-reasoning catalog entries, the wrapper only patches `model.reasoning` models, and the public config schema still does not accept the `qwen` thinking format workaround mentioned in the prior review context.

## Best Possible Solution

Implement a focused Qwen/Bailian Coding Plan provider fix: verify which Coding Plan models support DashScope `enable_thinking`, mark only those catalog entries as reasoning-capable, ensure OpenClaw thinking levels map to top-level `enable_thinking` without unsupported budget fields, decide whether the public compat schema should accept a Qwen thinking format for custom Bailian providers, update Qwen docs, and add regression tests for catalog metadata, payload construction, and config validation.
