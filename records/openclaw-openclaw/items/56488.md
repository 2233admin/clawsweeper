## Summary

Keep open. Current `main` still has the normal cron timer and manual due-run scans calling the runnable-job collector without the completed-one-shot guard, and the issue is already paired with an open implementation PR that references it with closing syntax.

## What This Changes

The issue reports that `kind: "at"` cron jobs with `deleteAfterRun: true` can repeatedly retrigger after their scheduled time instead of being removed after their first successful run.

## Best Possible Solution

Land #56509 or an equivalent focused cron scheduler patch that applies the existing completed-one-shot guard to the normal timer and manual due-run scans, with regression coverage for an enabled past-due `at` job whose terminal state is already persisted.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist an enabled `kind: "at"` job with `deleteAfterRun: true`, terminal `lastStatus`, and a past `nextRunAtMs`, then exercise `onTimer` or `runDueJobs`; current `main` selects it because both paths omit the guard.

## Solution Assessment

Yes, the narrowest maintainable fix is to reuse the existing `skipAtIfAlreadyRan` guard at the two missing call sites and add focused regression tests. A broader rewrite of one-shot deletion or retry policy is unnecessary for this issue.
