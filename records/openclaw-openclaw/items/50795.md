## Summary

Keep open: current main still has the reported live embedded compaction bug. The subscribe compaction handler still clears every assistant usage snapshot on non-retry compaction end, while the replay sanitizer already has boundary-aware logic that preserves fresh post-compaction usage; the closed fix PR was not merged and the newer status token-estimate fallback does not correct this helper.

## What This Changes

The issue reports that post-compaction assistant usage snapshots are zeroed, causing the TUI context token counter to show 0 after compaction.

## Best Possible Solution

Patch the embedded subscribe compaction handler so it identifies the latest compaction summary and zeroes only assistant usage snapshots before that boundary, preserving fresh post-compaction usage. Keep provider usage normalization, UI display logic, replay sanitization, and session-store token fallback behavior unchanged unless a small shared helper can be extracted cleanly.

## Reproduction Assessment

Yes. A focused reproduction can construct a session with assistant usage before and after a compactionSummary, trigger a non-retry compaction_end, and observe current main zeroing both assistant usage snapshots through clearStaleAssistantUsageOnSessionMessages.

## Solution Assessment

Yes. Boundary-gating the existing helper is the narrowest maintainable fix because it aligns the live subscribe path with the already-tested replay sanitizer; the compactionTokensAfter status fallback improves a related display path but does not preserve transcript usage.
