## Summary

Keep this PR open. The reported memory-lancedb contamination problem is real on current main, and this PR is a useful focused replacement for the closed #67475 attempt, but the current patch still misses current inbound metadata shapes and whitespace-tolerant sentinel matching, so it should not merge as-is.

## What This Changes

The PR adds memory-lancedb envelope-sludge detection and sanitization, filters auto-capture and auto-recall paths, strips recalled media annotations before prompt formatting, and adds focused unit tests.

## Best Possible Solution

Revise this PR in the memory-lancedb plugin so the detector and sanitizer mirror the current inbound metadata contract: accept leading whitespace before sentinel headers, handle Location and arbitrary structured untrusted metadata blocks, keep the fix extension-local, add regression tests for those cases, then rebase onto current main and rerun the focused memory-lancedb tests plus the changed gate.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection gives a focused reproduction: current main stores and recalls raw memory text, while the PR regexes miss indented metadata sentinels and current Location/structured untrusted metadata blocks.

## Solution Assessment

Is this the best way to solve the issue? No, not as-is. The extension-local sanitizer and recall filter are the right boundary, but the implementation must cover the full current metadata contract before it is the narrowest maintainable fix.

## Review Findings

Overall correctness: not a patch

Overall confidence: 0

Full review comments:

- **[P2] Allow indented metadata sentinels:** `extensions/memory-lancedb/index.ts:533`
  - body: The new detector and plaintext sanitizer match sentinels with `^${escapedSentinel}`, but the canonical inbound metadata stripper treats a sentinel line as metadata after trimming whitespace. An indented `Sender (untrusted metadata):` block can therefore bypass the new guard and still be stored or recalled if the remaining text contains a memory trigger.
  - confidence: 0.9
- **[P2] Reject all current inbound metadata labels:** `extensions/memory-lancedb/index.ts:498-505`
  - body: The sentinel list omits current `Location (untrusted metadata):` and arbitrary structured labels such as `<label> (untrusted metadata):`. Those blocks use payload keys like `latitude`, `longitude`, `source`, `type`, and `payload`, so the JSON fallback will not catch them and location or structured metadata can still persist in memory.
  - confidence: 0.86

## Security Review

Status: not_applicable

Summary: No workflow, dependency, lockfile, package publishing, generated, vendor, or install-script changes were found, but the patch leaves concrete metadata-retention bypasses in a privacy-sensitive memory filter.

Concerns:

- **[medium] Indented sentinels bypass sludge filtering:** `extensions/memory-lancedb/index.ts:533`
  - body: The patch's sentinel regexes require the sentinel at column zero, while core stripping treats trimmed sentinel lines as metadata. Leading whitespace can therefore preserve transport metadata in long-term memory and later recall.
  - confidence: 0.88
- **[medium] Location metadata can remain in long-term memory:** `extensions/memory-lancedb/index.ts:498`
  - body: Current main emits `Location (untrusted metadata):` blocks with location payload fields, but the PR does not detect or strip that label or its JSON keys. A location block followed by a memory trigger can still be persisted and recalled.
  - confidence: 0.84
