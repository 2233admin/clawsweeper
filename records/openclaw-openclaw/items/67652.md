## Summary

Keep this PR open. Current main still accepts only `mediaUrls` and `mediaUrl` in the shared reply-payload SDK path, so the PR's central `media` alias behavior is not implemented. The PR is focused and relevant, but it still needs maintainer review/rebase and the empty-string `mediaUrl` fallback concern resolved.

## Best Possible Solution

Keep the PR open for maintainer review or rebase. If OpenClaw wants this compatibility behavior, land a refreshed patch in `src/plugin-sdk/reply-payload.ts` that consistently accepts `media` in normalization, media resolution, content detection, sendable-part extraction, and chunked media sending, while treating blank aliases and blank `mediaUrl` values as missing. Add focused regression tests in `src/plugin-sdk/reply-payload.test.ts` for string and array aliases, blank alias entries, empty `mediaUrl` fallback, and delivery wrapping through `createNormalizedOutboundDeliverer`.
