## Summary

Keep open. Current main still builds the announce wake from a typed child task-completion event plus a rewrite instruction, and I found no requester-conversation context field in the event type, gateway schema, delivery calls, docs, or regression tests. The related routing fixes and the overlapping open prompt-context PR are useful context, but neither has landed a current-main fix for this issue.

## What This Changes

The issue requests bounded parent/requester conversation context for subagent and cron announce-mode completions so the generated user-facing update stays relevant to the active session.

## Best Possible Solution

Add a bounded, sanitized requester-context handoff for announce replies and align completion-wake prompt synthesis with normal requester-session context, while keeping child result content untrusted and preserving existing delivery routing, idempotency, and nested-subagent behavior.

## Reproduction Assessment

Yes. Static source inspection gives a high-confidence reproduction path: the announce trigger is built from child completion metadata/result plus rewrite instruction, and queued/steered/direct delivery paths forward it without a requester-context payload. The Telegram stale-session report still needs a focused session-key fixture before it can be called the same root cause.

## Solution Assessment

Yes, the requested direction is the right narrow fix, but the safest implementation is not raw last-N transcript injection. A bounded sanitized summary or stable requester-context reference, backed by schema/docs/tests, is the more maintainable solution.
