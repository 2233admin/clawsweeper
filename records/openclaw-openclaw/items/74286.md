## Summary

Keep open. Current main still builds subagent/cron announce replies from the child completion event plus a rewrite instruction, and I found no requester-conversation context field in the internal event type, gateway schema, delivery calls, docs, or regression tests. The related delivery-routing fixes improve where completions are sent, but they do not solve the prompt-context failure reported here.

## What This Changes

The issue asks OpenClaw to ground subagent and cron announce-mode completion replies in bounded parent/requester conversation context so the generated user-facing update is relevant to the active session.

## Best Possible Solution

Add an explicit, bounded requester-context handoff for announce replies, preferably a sanitized summary or stable context reference resolved at delivery time, while keeping child output untrusted and preserving the existing direct-delivery/session-queue routing behavior. The docs and regression tests should define the context boundary for subagent, cron, active steered, and dormant direct-agent delivery paths.

## Reproduction Assessment

Yes. There is a high-confidence static reproduction on current main: the announce trigger is built only from child completion metadata/result plus rewrite instruction, and both active and dormant delivery paths forward that trigger without requester-context payload. The Telegram stale-session addendum needs its own focused fixture before treating it as the same bug.

## Solution Assessment

Yes, the requested direction is valid, but the safest implementation is not raw last-N transcript injection. A narrow bounded/sanitized requester-context handoff with protocol/schema/tests is more maintainable and avoids leaking or overloading the announce turn.
