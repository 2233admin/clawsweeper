## Summary

Keep open. Current main still constructs announce replies from the child completion event plus rewrite instruction, and I did not find requester conversation context in the internal event schema, trigger construction, delivery forwarding, docs, or regression expectations. The Telegram stale-session addendum is useful related evidence, but it should get its own focused fixture before being treated as the same root cause.

## What This Changes

The issue reports that announce-mode subagent or cron completions are injected into the requester session without enough parent conversation context, causing irrelevant parent replies.

## Best Possible Solution

Add an explicit bounded requester-context handoff for announce replies so the completion rewrite can be grounded in the active requester conversation, while preserving child output as untrusted data and keeping existing route/idempotency behavior intact. The contract, docs, and regression tests should define the context boundary for subagent, cron, active steered, and dormant direct-agent paths.

## Reproduction Assessment

Yes. The current source provides a high-confidence static reproduction path: build an announce flow and observe that the trigger is derived from child task_completion fields plus rewrite instruction, then forwarded unchanged by active and direct delivery paths. I did not run live tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes, a narrow announce-handoff fix is the best direction. The safer version is not raw last-N transcript injection; it should pass a bounded sanitized requester summary or existing requester-session context at delivery time, with tests proving the reply turn has that grounding.
