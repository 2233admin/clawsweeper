## Summary

Keep open. Current main still builds subagent/cron completion announces from the child result plus a rewrite instruction, and the active-session steer path, dormant direct-agent path, documented announce context, and regression tests do not show a bounded requester-conversation context handoff. The added Telegram stale-session comment is useful adjacent routing evidence, but it does not make the original prompt-context bug obsolete.

## What This Changes

The issue asks for subagent and cron announce completion handoffs to include enough requester-session context that the parent can reply in the current conversation instead of treating the child result as standalone.

## Best Possible Solution

Add a bounded requester-context handoff for subagent and cron completion announces, covering active steered requesters and dormant direct-agent calls, while keeping child results untrusted and avoiding raw/unbounded transcript copying. The fix should preserve existing delivery routing behavior unless a targeted stale-session regression test proves a routing correction is also needed.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run the existing announce flow/test harness and inspect the generated triggerMessage/internalEvents: current main produces only the child completion event plus rewrite instruction, with no requester-context field or summary.

## Solution Assessment

Unclear as written: injecting the last N raw parent messages would solve context loss but is not the safest implementation. The best path is a bounded, sanitized requester-context summary or session-history-aware handoff with explicit tests for active, dormant, nested, and cron announce paths.
