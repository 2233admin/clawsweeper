## Summary

Keep open. Current main still launches bundled LSP servers with a direct `spawn(config.command, ...)`, and the linked fix PR #75343 is open, unmerged, and already reviewed with concrete blockers, so this issue should remain open until that PR or a narrow replacement lands.

## What This Changes

Fix Windows LSP plugin startup by resolving npm `.cmd` shims for bundled LSP server commands before spawning them.

## Best Possible Solution

Update, review, and land PR #75343 or a narrow replacement so LSP spawning uses the shared Windows resolver with the merged child environment, adds focused LSP runtime coverage and the required changelog entry, then let the issue close through the merge.

## Reproduction Assessment

Yes. The report gives a concrete Windows 11 setup with `typescript-language-server`, and current main statically matches the failure path because `.lsp.json` stdio servers are launched through direct `spawn(config.command, ...)` without Windows `.cmd` resolution.

## Solution Assessment

Partly. `resolveWindowsSpawnProgram()` is the right shared seam, but the active PR should pass the merged LSP launch environment into both resolution and spawn, plus add focused coverage and release-note coverage, before it is the best fix.
