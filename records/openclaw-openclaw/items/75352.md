## Summary

Keep open. Current main still launches bundle LSP servers by passing the bare configured command directly to `child_process.spawn`, and the linked fix PR is open and unmerged with closing syntax for this issue, so closing before merge would be premature.

## What This Changes

The issue requests that Windows LSP server launches use the existing Windows spawn resolver for npm `.cmd` shims instead of calling `child_process.spawn` directly on the bare command.

## Best Possible Solution

Review and land PR #75343, or an equivalent narrow patch, so `spawnLspServerProcess()` materializes LSP commands through the existing Windows spawn helper while preserving direct non-Windows behavior and avoiding shell fallback except where intentionally allowed.

## Reproduction Assessment

Yes. A high-confidence reproduction path is Windows 11 with a globally installed npm LSP CLI such as `typescript-language-server`, an enabled bundle LSP config using that command, and current main’s direct `spawn(config.command, ...)` path; source inspection and Node’s Windows `.cmd` process contract support the report. ([nodejs.org](https://nodejs.org/api/child_process.html))

## Solution Assessment

Yes. Reusing `resolveWindowsSpawnProgram()`/`materializeWindowsSpawnProgram()` in the LSP spawn path is the narrowest maintainable direction because the helper already owns PATH/PATHEXT and wrapper policy; the issue should stay open until the linked PR or equivalent fix is merged.
