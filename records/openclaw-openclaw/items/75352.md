## Summary

Keep open. Current main still launches bundled LSP servers with a direct `child_process.spawn(config.command, ...)` path, so the reported Windows `.cmd` shim failure remains plausible and the linked fixing PR is open and unmerged.

## What This Changes

Request that bundled LSP server startup on Windows resolve npm `.cmd` shims through the shared Windows spawn resolver instead of directly spawning the configured command.

## Best Possible Solution

Use the existing Windows spawn resolver in the LSP runtime with the merged child environment, add focused coverage for a Windows `.cmd` LSP shim, include the required user-facing changelog entry, and let the linked PR or a narrow replacement close the issue after merge.

## Reproduction Assessment

Yes. The report gives a concrete Windows 11 setup with `typescript-language-server`, and current main statically follows the failing path by spawning the bare configured LSP command without Windows PATH/PATHEXT wrapper resolution.

## Solution Assessment

Yes. Reusing the shared Windows resolver at the LSP spawn boundary is the narrow maintainable fix; it should be wired with the exact environment passed to `spawn` rather than switching broadly to `shell: true`.
