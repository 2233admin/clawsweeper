## Summary

Keep this issue open. Current main still has the reported stop-timeout path: the timeout branch marks the account running and returns before clearing the stale task/abort entries, while the start path still no-ops when the stale task remains. An open closing PR, #73648, already targets this bug, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that a channel stop timeout leaves stale task/abort entries and a misleading running runtime state, causing health-monitor restarts to silently fail.

## Best Possible Solution

Land PR #73648 or an equivalent gateway fix that clears stale `store.aborts` and `store.tasks` entries on stop timeout, records the timed-out runtime as stopped, prevents old timed-out task callbacks from overwriting a replacement task, and updates the gateway regression tests to prove a fresh task can start after timeout.
