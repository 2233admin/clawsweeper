## Summary

Keep open. Current `main` still has the reported stop-timeout path: `stopChannel` returns before clearing stale task/abort bookkeeping, marks the account as running, and `startChannelInternal` still no-ops when the stale task remains. The latest release tag I inspected has the same behavior, and open PR #72912 is the right implementation candidate to review rather than queueing a separate repair job.

## What This Changes

The issue reports that `stopChannel` timeout leaves stale channel task bookkeeping and a `running: true` runtime, causing health-monitor restarts for Telegram and Slack accounts to silently no-op.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that retires wedged health-monitor restarts, clears stale task/abort bookkeeping for the retired lifecycle, marks the account stopped or disconnected, gates late callbacks from retired tasks, and updates the gateway regression test so a fresh task is registered after timeout.

## Reproduction Assessment

Yes. Current `main` has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`: after a stop timeout and a second start attempt, the test expects the replacement start not to run and the runtime to remain `running: true`.

## Solution Assessment

Unclear until PR #72912 is reviewed, but the narrow map-deletion patch alone is likely not the full safest fix. The more maintainable direction is forced lifecycle retirement plus callback gating so a timed-out old task cannot corrupt the replacement lifecycle.
