## Summary

Keep open. Current main still has the reported stop-timeout path: the timed-out task remains registered, the runtime is marked running, the next start silently no-ops, and an open maintainer-labeled PR already targets the same gateway lifecycle bug.

## What This Changes

The issue asks OpenClaw to recover channel health-monitor restarts when `stopChannel` times out by clearing stale lifecycle bookkeeping and allowing a fresh channel task to start.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that force-retires wedged health-monitor restarts, clears stale task and abort bookkeeping, marks the runtime stopped or disconnected, gates late callbacks from retired tasks, and replaces the current regression test with one proving a fresh task is registered after the timeout.

## Reproduction Assessment

Yes. Current main has a unit-level reproduction in `src/gateway/server-channels.test.ts`: after a stop timeout and a second start attempt, the test expects the replacement start not to run and the runtime to stay `running: true`.

## Solution Assessment

Unclear until PR #72912 is reviewed, but the best fix direction is likely a forced-retirement path with lifecycle-gated callbacks rather than only deleting the maps, because late callbacks from the timed-out task could otherwise overwrite the replacement lifecycle.
