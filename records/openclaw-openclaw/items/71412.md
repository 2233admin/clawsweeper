## Summary

Keep open. Current main still has the reported gateway lifecycle bug: the stop-timeout path returns before clearing stale task bookkeeping, marks the account running, and the health monitor then calls the start path that no-ops on the retained task. The latest release still contains the same behavior, while open PR #72912 is the implementation candidate rather than a reason to close this issue before merge.

## What This Changes

The issue asks for channel stop-timeout recovery so health-monitor restarts can replace a timed-out channel task instead of leaving the account apparently running with no active lifecycle.

## Best Possible Solution

Land #72912 or an equivalent gateway lifecycle fix that can forcibly retire a wedged health-monitor restart, clear stale task and abort bookkeeping, mark the retired runtime stopped or disconnected, gate late callbacks from retired tasks, and replace the current regression test with one proving a fresh task starts after timeout.

## Reproduction Assessment

Yes. Current main has a unit-level reproduction in `src/gateway/server-channels.test.ts`: a task that ignores abort times out after 5 seconds, the second start attempt is blocked, and the runtime remains `running: true`.

## Solution Assessment

Yes for the repair direction, but it is not on main yet. A forced-retirement lifecycle path like #72912 is safer than only deleting map entries because it also handles late callbacks from the retired task.
