## Summary

Keep open. Current main still has the reported channel stop-timeout failure mode: the stale task remains registered, the runtime is marked running, and the health monitor can proceed into a start path that silently returns. The issue is also paired with open closing PR #73648, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

Request a gateway channel lifecycle fix so stop-timeout cleanup releases stale task and abort entries, records the account as stopped, and lets health-monitor restarts register a fresh channel task.

## Best Possible Solution

Land PR #73648 or an equivalent gateway fix that clears stale `store.aborts` and `store.tasks` entries on stop timeout, records the timed-out runtime as stopped, prevents old timed-out task callbacks from overwriting replacement task state, and updates the gateway regression tests to prove a fresh task can start after timeout.
