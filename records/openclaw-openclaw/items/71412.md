## Summary

Current main still contains the reported gateway lifecycle bug: a stop timeout leaves the stale task registered, marks the account running, and lets the health-monitor restart flow fall into a start no-op. This should stay open while the already-linked maintainer-labeled PR #72912 or an equivalent fix is reviewed and landed.

## What This Changes

The issue requests clearing stale channel task/abort bookkeeping and reporting a stopped runtime after `stopChannel` times out so health-monitor restarts can create a fresh channel lifecycle.

## Best Possible Solution

Land PR #72912 or an equivalent gateway lifecycle fix that force-retires wedged health-monitor restarts, clears stale `store.aborts` and `store.tasks` entries, marks the retired runtime stopped or disconnected, lifecycle-gates callbacks from retired tasks, and replaces the current regression with proof that a fresh task starts after timeout.

## Reproduction Assessment

Yes. Current main has a deterministic unit-level reproduction in `src/gateway/server-channels.test.ts`: after a stop timeout and a second start attempt, the test expects no replacement task and a still-running runtime with the timeout error.

## Solution Assessment

Unclear until PR #72912 is reviewed. The minimal map deletion in the issue addresses the start no-op, but the safer maintainable fix should also prevent late callbacks from the retired task from overwriting the replacement lifecycle.
