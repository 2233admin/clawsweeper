## Summary

Keep open. Current main still has the reported blanket same-target suppression path: a media-only message tool send can commit a target/media URL, `shouldSuppressMessagingToolReplies()` only checks the target match, and `buildReplyPayloads()`/followup delivery still return `[]` when that target suppression is true. The maintainer comment on 2026-04-25 matches the current source state and explicitly says the incomplete-turn fallback work does not fix this layer.

## Best Possible Solution

Keep #59743 open and land a focused core fix in the reply payload filtering layer. The most conservative direction is to remove the final blanket `[]` shortcut for same-target sends and rely on content/media-level dedupe, or add committed-send metadata that distinguishes text from media-only sends before suppressing. Add regression coverage for same-target media-only delivery plus a distinct final text reply in `agent-runner-payloads` and the followup delivery path, while retaining tests for duplicate same-target text, cross-target preservation, and account mismatch.
