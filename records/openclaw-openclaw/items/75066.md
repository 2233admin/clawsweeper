## Summary

Keep open. Current main still has the reported session freshness problem, and this PR targets the right narrow seam by preserving activity timestamps for metadata-only compaction and memory-flush writes. The production approach matches the existing route-update precedent; I found one low-severity test portability issue that should be fixed before normal maintainer review and CI finish.

## What This Changes

The PR adds a `preserveActivity` option to session-store entry updates, uses it for compaction and memory-flush metadata writes, stops compaction count bookkeeping from refreshing `updatedAt`, and adds regression tests for legacy daily/idle reset behavior.

## Best Possible Solution

Land a narrow version of this fix after switching the new test fixture to the platform temp directory and validating the touched session, compaction, and memory-flush paths. The core reset policy should remain unchanged; metadata-only writers should opt into the existing preserve-activity behavior.

## Reproduction Assessment

Yes. The linked bug report gives concrete daily and idle legacy-row scenarios, and current main source confirms the necessary path: compaction/flush metadata can advance `updatedAt`, while reset freshness falls back to `updatedAt` when explicit lifecycle timestamps are missing.

## Solution Assessment

Mostly yes. Preserving activity only for metadata-only compaction/flush writes is the narrowest maintainable fix and follows the existing `updateLastRoute` precedent; the safer adjustment is to make the new regression test use `os.tmpdir()` before merge.
