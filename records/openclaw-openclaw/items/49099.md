## Summary

Keep open. Current main still normalizes inbound WhatsApp video media to a fixed `<media:video>` placeholder, does not inspect `videoMessage.gifPlayback`, and does not extract `contextInfo.externalAdReply` metadata. The closed companion PR #49103 did not land, while the pinned Baileys 7.0.0-rc.9 dependency exposes the requested GIF and external-ad-reply fields.

## What This Changes

The issue requests that inbound WhatsApp GIF picker media be distinguished from regular video and expose provider metadata such as title, body, and source URL to the agent.

## Best Possible Solution

Implement the fix inside the WhatsApp plugin: classify media-only `videoMessage.gifPlayback` as GIF, extract relevant `externalAdReply` title/body/source URL from inbound media, and forward that data through explicit untrusted structured media context with focused tests and user-facing docs/changelog if the prompt surface changes.

## Repair Work Prompt

Implement a focused fix for inbound WhatsApp GIF/media metadata. Current main returns `<media:video>` for `videoMessage` even when `gifPlayback` is true and ignores `contextInfo.externalAdReply`. Keep the fix inside the WhatsApp plugin unless a generic already-existing seam is needed. Add a typed helper around the unwrapped inbound image/video message that extracts `externalAdReply.title`, `externalAdReply.body`, and `externalAdReply.sourceUrl` when present. Classify media-only `videoMessage.gifPlayback === true` as a GIF placeholder, but do not put provider-fetched strings into inline pseudo-XML attributes or trusted body text. Instead, carry the metadata as explicit `untrustedStructuredContext` media entries through the monitor/dispatch path so `src/auto-reply/reply/inbound-meta.ts` renders it as untrusted JSON. Update focused tests for extractor behavior, monitor/enqueue or dispatch forwarding, and prompt rendering. Update `docs/channels/whatsapp.md` and `CHANGELOG.md` if the prompt-visible behavior changes, with normal OpenClaw attribution rules. Do not broaden core behavior, do not add media processing or ffmpeg dependency work, and do not edit dependency patches or lockfiles.
