## Summary

Keep open. Current main still maps inbound WhatsApp video messages to plain `<media:video>` and has no `gifPlayback` or `externalAdReply` extraction. The companion PR #49103 implemented the requested shape but was closed unmerged by the contributor for follow-up, so it is evidence of a plausible fix path rather than a resolved item.

## Best Possible Solution

Keep this issue open and implement the feature in the WhatsApp plugin with the existing untrusted metadata pattern: distinguish `videoMessage.gifPlayback` as GIF media, surface `externalAdReply.title/sourceUrl/body` with clear WhatsApp media provenance, sanitize/truncate values through the untrusted structured-context path, and add focused inbound tests plus docs updates.

## Risks / Open Questions

- The issue is valid, but the suggested inline pseudo-XML attributes would put provider-derived `externalAdReply` title/body/source metadata directly into the message body. Current WhatsApp structured-object handling prefers explicit untrusted metadata blocks for similar data.
- Related security context exists around platform-fetched preview metadata reaching the model as ordinary user input, so the implementation should preserve provenance and avoid treating GIF metadata as trusted visible sender text.
