## Summary

Keep open. Current main still hardcodes the MiniMax coding-plan usage request to `api.minimaxi.com`, and the MiniMax plugin passes no configured baseUrl into the usage fetcher. An open closing PR, #65059, specifically targets this issue, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that MiniMax usage polling derive the coding-plan remains endpoint from the configured MiniMax provider baseUrl instead of always using the CN host.

## Best Possible Solution

Land a focused MiniMax usage fix that derives the remains endpoint host from the configured MiniMax baseUrl, strips chat path suffixes such as `/anthropic` or `/v1`, preserves a deliberate CN fallback, and adds regression tests for both global and CN request URLs.

## Reproduction Assessment

Yes. The high-confidence static reproduction path is `openclaw status --usage` -> `loadProviderUsageSummary` -> MiniMax plugin `fetchUsageSnapshot` -> `fetchMinimaxUsage`, where the current helper always constructs the CN URL.

## Solution Assessment

Yes, the narrow maintainable solution is to keep this in the MiniMax plugin/shared usage helper using the existing `ProviderFetchUsageSnapshotContext.config`; no new core plugin API appears necessary.
