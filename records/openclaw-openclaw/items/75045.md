## Summary

Keep open. Current main still builds the plugin loader cache key without bundled OpenClaw package version or package-root identity, while bundled runtime-deps roots are versioned by package version and path hash; the open, unmerged PR #75071 is already the implementation candidate for this issue.

## What This Changes

The issue asks OpenClaw to stop browser and bundled plugin loads from reusing stale `plugin-runtime-deps` roots after a 2026.4.26 to 2026.4.27 package upgrade, even after doctor repair and gateway restart.

## Best Possible Solution

Review and land PR #75071, or an equivalent generic loader/runtime-deps fix, so bundled plugin cache identity changes when the packaged OpenClaw version or package-root identity changes. Keep the repair at the shared plugin loader/runtime-deps seam rather than adding browser-specific recovery policy.

## Reproduction Assessment

Yes. The issue gives a concrete path: run 2026.4.26 with the browser plugin enabled, upgrade to 2026.4.27, run doctor repair, restart the systemd gateway, then run `openclaw browser status` and observe an import from the old runtime-deps root.

## Solution Assessment

Likely yes. Adding packaged OpenClaw version/package-root identity to the loader/runtime-deps cache key is narrower and more maintainable than a browser-specific workaround, and current main still lacks that cache dimension.
