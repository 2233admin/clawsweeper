## Summary

Keep this open. The report is fresh and concrete, current `main` still does not visibly include package-version/package-root identity in the plugin loader cache key, and open PR #75071 is already the closing implementation candidate.

## What This Changes

The issue requests a fix for browser bundled plugin runtime-deps continuing to import from a stale `2026.4.26` staged root after upgrading to `2026.4.27` and running doctor repair.

## Best Possible Solution

Land PR #75071 after normal review, or an equivalent generic loader/runtime-deps fix, so bundled plugin registry/runtime-deps cache identity changes when the packaged OpenClaw version or package root identity changes. Keep the repair at the generic plugin loader seam, with a regression test for same-path package upgrades, rather than adding browser-specific recovery policy.

## Reproduction Assessment

Yes. The issue provides a high-confidence path: browser enabled on 2026.4.26, upgrade in place to 2026.4.27, run `openclaw doctor --repair`, restart the systemd gateway, then run `openclaw browser status` and observe an import from the old staged runtime-deps root.

## Solution Assessment

Likely yes for the linked PR direction, subject to maintainer review. Including package-version/package-root identity in the loader cache is narrower and more maintainable than a browser-only workaround, and current `main` still lacks that visible cache-key input.
