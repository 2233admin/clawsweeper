## Summary

Current main still has static browser SSRF behavior: fail-closed defaults, broad private-network opt-in, and hostname exceptions. It does not implement the requested confirmation mode, CIDR/port allowlist/blocklist policy, non-overridable cloud metadata blocking under broad opt-in, timeout/rate-limit/audit behavior, or temporary `openclaw ssrf` CLI controls. The design comment by mdlmarkham adds useful security requirements, especially combining allowlists with confirmation and hard-blocking cloud metadata endpoints.

## Best Possible Solution

Keep this issue open as the canonical browser private-network confirmation/security design request. Maintainers should first decide the policy contract and precedence rules: confirm vs allow vs deny, CIDR/port allowlist and blocklist semantics, non-overridable cloud metadata blocks, timeout/rate-limit/audit behavior, redirect and DNS-rebinding handling, browser proxy/CDP reachability behavior, and whether `openclaw ssrf` temporary controls are an accepted CLI surface. After that, implement the contract across shared SSRF policy, browser config/schema, browser navigation/control paths, UI approval plumbing, public docs, and focused security tests.
