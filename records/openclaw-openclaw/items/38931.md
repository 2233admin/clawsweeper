## Summary

Keep open. Current main still exposes only static browser SSRF allow/block behavior plus hostname exceptions, while the issue asks for a security-sensitive middle policy with per-request confirmation, CIDR/port allow/block rules, hard metadata blocking, audit/rate-limit/timeout behavior, and optional CLI temporary controls.

## What This Changes

Request a browser SSRF private-network access policy that can ask for user confirmation, support allow/block lists, and optionally expose temporary CLI controls instead of only broad allow or deny.

## Best Possible Solution

Keep this open as the canonical browser private-network confirmation/security design request. Maintainers should first settle policy precedence for deny/allow/confirm, CIDR and port allow/block lists, non-overridable cloud metadata blocks, timeout/rate-limit/audit behavior, redirect and DNS-rebinding handling, browser proxy/CDP behavior, and whether an openclaw ssrf temporary CLI surface is accepted, then implement it across the shared SSRF policy, browser config/schema, browser navigation/control paths, UI approval flow, docs, and focused security tests.
