## Summary

Keep open. Current main still resolves shared thinking hooks from the active plugin registry, while the OpenAI plugin is enabled by default but not startup-active, so the reported OpenAI Codex `xhigh` visibility gap is still present. The supplied context also shows open PR #75295 explicitly linked as the fix path, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks gateway/session thinking validation and metadata to accept `xhigh` for `openai-codex/gpt-5.5` by resolving the OpenAI Codex provider thinking profile even when the OpenAI plugin is enabled but not startup-active.

## Best Possible Solution

Land #75295 or an equivalent generic provider-owned thinking-profile lookup that can resolve enabled provider runtime hooks before startup activation, then close this issue after the fix is merged and validated across gateway rows, session patching, agent command validation, and directive handling.

## Reproduction Assessment

Yes. The discussion gives concrete 2026.4.27 gateway and local agent commands that reject `--thinking xhigh`, and current main source still shows the same active-registry-only provider hook path that would omit OpenAI Codex `xhigh` when the OpenAI plugin is not startup-active.

## Solution Assessment

Yes. The requested direction and linked PR shape are the maintainable solution because they keep model-specific `xhigh` policy in the provider layer and make shared validation resolve provider-owned hooks generically instead of hardcoding OpenAI Codex rules in core.
