## Summary

Keep open. Current main still has the reported provider-profile visibility gap, and the supplied GitHub context shows open PR #75295 is already the implementation candidate linked to close this issue, so this issue should stay open until that PR lands or is closed.

## What This Changes

Ensure gateway/session thinking validation for `openai-codex/gpt-5.5` can see the provider-owned thinking profile and accept `xhigh` even when the OpenAI plugin is enabled but not startup-active.

## Best Possible Solution

Review and land PR #75295 or an equivalent generic provider-runtime fallback that resolves provider-owned thinking hooks for enabled startup-lazy providers, then keep the regression coverage on session defaults/rows, `sessions.patch`, and agent command validation without hardcoding OpenAI Codex rules in shared core.

## Reproduction Assessment

Yes. The issue discussion gives concrete 2026.4.27 commands and config that hard-reject `--thinking xhigh`, and current-main source still shows the same failure path when the OpenAI provider hook is absent from the active registry.

## Solution Assessment

Yes for the requested direction: resolving provider-owned thinking policy through a generic runtime lookup is the narrow maintainable fix. The issue should remain open until the linked PR or an equivalent change is merged and validated.
