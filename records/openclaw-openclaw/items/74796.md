## Summary

Keep open. Current main still has the reported provider-profile visibility gap: OpenAI Codex declares xhigh for gpt-5.5, but shared thinking validation resolves provider hooks only from the active runtime registry, and the OpenAI plugin is enabled but not startup-active. The issue also includes a second live 2026.4.27 reproduction showing the same explicit rejection before the request reaches Codex.

## What This Changes

The issue requests that gateway and agent-session thinking validation advertise and accept xhigh for openai-codex/gpt-5.5 instead of downgrading or rejecting it when the provider plugin is not startup-active.

## Best Possible Solution

Add a generic provider-owned thinking-profile lookup that works for configured/default model validation before startup activation, then use it consistently across gateway session rows, `sessions.patch`, agent command validation, directive validation, and plugin runtime policy helpers without hardcoding OpenAI Codex model rules in core.

## Reproduction Assessment

Yes. The issue gives concrete live commands and config that reject `--thinking xhigh`, and current-main source shows the same failure path when the OpenAI provider hook is absent from the active registry and no catalog compat row supplies xhigh for `openai-codex/gpt-5.5`.

## Solution Assessment

Yes. The maintainable fix is the requested generic provider-profile plumbing or an equivalent lightweight provider policy artifact; startup-activating the whole OpenAI plugin or adding Codex-specific xhigh checks in shared core would be a weaker boundary fit.
