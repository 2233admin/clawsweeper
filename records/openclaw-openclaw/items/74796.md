## Summary

Keep open. Current main still has the reported provider-profile visibility gap: OpenAI Codex declares `xhigh` for `gpt-5.5`, but the shared thinking resolver used by gateway/session validation only sees active runtime-registry providers or catalog compat metadata. The OpenAI plugin is enabled by default but explicitly not startup-active, and the gateway paths still call the shared resolver before a non-startup OpenAI provider profile is guaranteed to be present.

## What This Changes

Gateway thinking validation and session metadata should expose and accept `xhigh` for `openai-codex/gpt-5.5` when the OpenAI plugin is enabled but not startup-active.

## Best Possible Solution

Add a generic, plugin-owned provider-profile resolution path that can answer thinking-policy questions for configured or owner-resolved providers before startup activation. Reuse it for gateway session rows/defaults, `sessions.patch`, agent/subagent validation, and runtime policy helpers, keeping OpenAI-specific model policy inside the OpenAI plugin or a lightweight public artifact rather than hardcoding Codex model lists in core.

## Reproduction Assessment

Yes. The issue gives concrete OpenClaw 2026.4.27 config and command paths, a second live reproduction confirms local/gateway/subagent rejection, and current main source still shows the active-registry-only lookup that explains the failure.

## Solution Assessment

Yes, the suggested direction is the right boundary: expose provider-owned thinking profiles through a generic lightweight/provider-runtime lookup that works before startup activation. Eagerly loading OpenAI at gateway startup or duplicating Codex `xhigh` rules in core would be a less maintainable fix.
