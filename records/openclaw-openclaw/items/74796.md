## Summary

Keep open. Current main still has the reported provider-profile visibility gap: the OpenAI Codex provider declares `xhigh` for `gpt-5.5`, but the shared thinking hooks used by gateway/session validation still resolve provider thinking policy only from the active plugin registry; the bundled OpenAI plugin is enabled by default but explicitly not startup-active.

## What This Changes

The issue requests that gateway sessions accept and advertise `xhigh` for `openai-codex/gpt-5.5` instead of rejecting it or remapping it to `high` when the OpenAI provider plugin has not been startup-activated.

## Best Possible Solution

Add a generic provider-owned thinking-profile lookup that works for configured/default model validation before startup activation, then share it across gateway session rows, `sessions.patch`, agent command validation, and plugin runtime policy helpers without hardcoding OpenAI Codex model rules in core.

## Reproduction Assessment

Yes. The issue and follow-up provide concrete 2026.4.27 configs and commands that reject `--thinking xhigh`; current-main source still shows why that happens when the OpenAI provider hook is not present in the active registry.

## Solution Assessment

Yes. The maintainable direction is a generic provider-profile/runtime lookup or lightweight catalog contribution for thinking policy; startup-activating the full OpenAI plugin or adding Codex-specific xhigh checks in core would be narrower for this symptom but weaker as a plugin contract.
