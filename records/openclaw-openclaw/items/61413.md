## Summary

Keep this PR open, but do not merge it as-is. The transparent-proxy problem is not implemented on current main, but the proposed fetch-guard change drops target SSRF policy validation in trusted proxy mode, and the diagnostic helper can throw while formatting an already-caught error.

## What This Changes

The PR changes guarded fetch so trusted env-proxy mode uses the default fetch path when no proxy env vars are set, and it adds nested error-cause details to tool execution errors with tests.

## Best Possible Solution

Keep the useful diagnostic and transparent-proxy goal, but rework the fetch path so target policy validation always runs before transport selection. If default fetch is allowed for operator-controlled transparent-proxy endpoints, cover that with explicit tests for allowlists, private hosts, NO_PROXY/ALL_PROXY behavior, and no-env mode, and reuse a guarded error formatter for cause-chain diagnostics.

## Reproduction Assessment

Yes for the blocking regression: under this PR diff, a TRUSTED_ENV_PROXY request with a restrictive hostnameAllowlist can skip target policy validation before fetch. For the original transparent-proxy failure, reproduction is not live here, but current main has a focused test proving no-env trusted mode still uses a pinned dispatcher.

## Solution Assessment

No. The requested behavior may be valid, but the current implementation is not the best fix because it couples transparent-proxy support to removal of SSRF policy checks; the safer path is validate policy first, then choose a default-fetch transport only for the narrowly trusted no-env case.
