## Summary

Keep this PR open. Current main still drops Feishu card-action `form_value`, so the intended behavior is not implemented yet. The PR is a relevant Feishu bug-fix candidate, but it appears incomplete because the live `card.action.trigger` parser still reconstructs `action` with only `value` and `tag`, so real callback `form_value` data would not reach the new decoder/handler path. It should be reviewed or consolidated with the broader open Feishu callback-field work in #42754, #42908, #43202, and #43953.

## Best Possible Solution

Keep this PR open for maintainer review, but revise or consolidate it with #42908/#43953. The best implementation should preserve `form_value`, `input_value`, `option`, and `options` in `parseFeishuCardActionEventPayload`, carry the fields through the typed decoder/handler path, add parser-level and handler-level regression tests using a real `card.action.trigger` payload shape, and define an explicit structured or escaped encoding policy before synthetic command dispatch.

## Risks / Open Questions

- The PR does not update `parseFeishuCardActionEventPayload`, so real Feishu `card.action.trigger` callbacks may still lose `action.form_value` before reaching the new code.
- Appending raw `key=value` pairs to command text is ambiguous for whitespace, arrays/objects, and command-parser-sensitive characters; the encoding policy needs maintainer review.
- Overlapping open PRs #42908 and #43953 cover broader Feishu callback fields, so landing multiple partial fixes could create inconsistent payload shapes.
