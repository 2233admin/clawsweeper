## Summary

Keep open. Current main still has the reported WebChat gap: `emitChatFinal` builds and broadcasts the final chat payload directly without consulting the plugin hook runner, while the shared outbound and Telegram reply paths do run `message_sending`. The request is narrow enough for a channel-local parity fix, and no open fixing PR was found.

## What This Changes

Request WebChat/WebSocket assistant final delivery to invoke the `message_sending` plugin hook so plugins can rewrite or cancel outbound content before the UI receives it.

## Best Possible Solution

WebChat final delivery should run the plugin `message_sending` hook before constructing visible assistant content, pass stable context such as `channelId: "webchat"`, `sessionKey`, and `runId`, apply rewritten text to both WebSocket and node-session chat payloads, and isolate hook failures so final delivery still completes. Cancellation should suppress the message content but preserve a terminal final event.

## Reproduction Assessment

Yes. A focused source check or unit test around `createAgentEventHandler` can emit assistant text plus lifecycle end and observe that the final WebChat payload is broadcast without any `runMessageSending` call; the reference outbound paths show the expected hook call.

## Solution Assessment

Yes for the narrow parity bug, with one caveat: implement cancellation as content suppression plus terminal final state rather than dropping the completion frame. Full pre-send enforcement for streamed deltas should stay with the broader outbound-policy issue.
