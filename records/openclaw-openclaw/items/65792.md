## Summary

Current main still lacks the requested `message_sending` hook on the WebChat final WebSocket delivery path. `emitChatFinal` resolves and flushes buffered text, then directly broadcasts the final `chat` payload with no hook runner call, while shared outbound and dispatcher paths already support `message_sending` rewrite/cancel semantics. No open PR appears to cover this exact issue, and the related umbrella issues do not make this focused report obsolete.

## Best Possible Solution

Keep this issue open and implement a focused Gateway WebChat fix: run plugin `message_sending` for final assistant WebSocket delivery, pass `channelId: "webchat"` plus `sessionKey` and run correlation, apply rewritten content, honor `cancel: true`, and isolate hook failures so WebChat delivery still completes. Add focused tests for rewrite, cancel, thrown-hook fallback, no-hook behavior, and the existing delta-before-final behavior.

- `createAgentEventHandler` is currently synchronous and `runMessageSending` is async, so a fix needs to preserve event ordering and avoid clearing run context before the hook has the session/run fields it needs.
- The current final path flushes any pending delta before sending the final payload. A final-only hook can rewrite or cancel the final event, but it cannot take back already-streamed deltas; tests should make that behavior explicit.
- There are adjacent WebChat/TUI hook gaps in #70928 and broader lifecycle coverage work in #50126/#56349. A narrow fix should not claim universal outbound hook coverage unless those paths are also addressed.
