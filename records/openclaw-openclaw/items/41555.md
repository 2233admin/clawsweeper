## Summary

Keep this issue open. Current main still treats bare `/new`, `/reset`, and gateway `sessions.reset` as conversation resets that preserve explicit user model selections plus thinking/reasoning session overrides. That means the reported recovery expectation, a bare `/new` returning a stuck Opus/thinking session to a non-thinking/default model state, is still not implemented. Recent fixes clear auto-fallback pins, but they intentionally preserve user-selected model state.

## Best Possible Solution

Keep this open for an explicit stuck-run recovery design in core session/run controls. The most conservative path is a separate recovery/reset option that aborts active work and can clear selected model, thinking, and reasoning overrides back to defaults, while preserving current `/new` semantics unless maintainers intentionally change them. If current behavior is intended, document the recovery path clearly, for example `/think off`, `/new <model>`, or switching away from the affected model before resetting.
