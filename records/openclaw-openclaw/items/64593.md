## Summary

Keep open. The report is recent, concrete, and still matches current main: the managed browser launcher stores and stops only the direct Chromium ChildProcess, while the issue describes long-running PID-1 containers accumulating defunct Chromium and chrome_crashpad descendants after normal browser use. The adjacent startup-failure issue and open gateway-orphan PR do not cover this normal-exit browser lifecycle problem.

## What This Changes

The issue requests that Chromium and chrome_crashpad child processes spawned by the browser control service be reaped after normal exits so long-running container gateways do not exhaust PIDs.

## Best Possible Solution

Fix this in the bundled browser plugin/runtime lifecycle so managed Chromium is container-safe across normal exits, explicit stops, restarts, and startup cleanup. The end state should include a supervised helper/init/subreaper or other safe process-tree strategy that cannot signal the Gateway, focused unit coverage, a Docker/browser regression that repeatedly starts and exits managed Chromium while checking for defunct chromium/chrome_crashpad descendants, and docs for any supported container requirement or workaround.

## Repair Work Prompt

Fix the managed browser zombie-process leak described in https://github.com/openclaw/openclaw/issues/64593. The observable bug is that long-running Linux/Docker gateways running as PID 1 can accumulate defunct Chromium and chrome_crashpad descendants after normal browser use until child_process.spawn starts failing with EAGAIN. Keep the fix scoped to the bundled browser plugin/runtime lifecycle unless a small generic process-runtime seam is unavoidable. Start from `extensions/browser/src/browser/chrome.ts`, `extensions/browser/src/browser/server-context.availability.ts`, and `extensions/browser/src/browser/server-lifecycle.ts`. Do not use a naive Unix process-group kill that can signal the Gateway; account for the warning in `src/process/kill-tree.ts` about shared process groups. Prefer a supervised helper/init/subreaper or another safe tree/reaping strategy that handles normal exits, explicit `openclaw browser stop`, restart/retry cleanup, and process teardown without broad core policy changes. Add focused unit tests plus a Docker/browser smoke or equivalent regression that repeatedly starts/exits managed Chromium and asserts no defunct `chromium` or `chrome_crashpad` descendants remain under the OpenClaw process. Update browser/Linux container docs if the fix requires or recommends a supported init/subreaper/container option. Add a user-facing changelog fix entry crediting `Thanks @theandyroberts`; do not change unrelated browser behavior, provider surfaces, or global process-kill semantics. Related context: #8611 is startup failure only, #38247 was a closed unmerged process-group attempt, and #47776 is adjacent gateway-orphan cleanup but not this normal-exit browser leak.
