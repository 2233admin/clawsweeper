## Summary

Keep this PR open, but do not merge the current diff as-is. The change addresses a real shared-auth cooldown problem for request/schema failures, but the blanket null policy for ambiguous reasons would bypass the existing OpenAI Codex WHAM health probe before provider-specific cooldown logic can run.

## What This Changes

The PR narrows the pi embedded runner auth-profile cooldown policy to record only auth-like health signals and adds helper tests for non-auth and positive failure reasons.

## Best Possible Solution

Keep the narrower auth-health direction, but make the policy provider/profile aware or preserve a separate probe path so Copilot `format` and other request-shape failures do not cooldown shared profiles while OpenAI Codex ambiguous failures still reach WHAM-based cooldown handling. Add regression coverage for both sides.

## Reproduction Assessment

Yes. The source path is high-confidence: non-null policy reasons flow to `markAuthProfileFailure`, null reasons return early, and existing tests prove `unknown` is an intentional Codex WHAM cooldown input. The Copilot cascade itself is supported by the PR's log excerpt and matching current auth-state flow, not by a live run in this read-only review.

## Solution Assessment

No. The proposed generic narrowing is maintainable in principle, but this exact patch drops ambiguous reasons before provider-specific health logic can classify real Codex profile exhaustion; a provider-aware policy or probe-preserving path is safer.
