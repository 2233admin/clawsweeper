## Summary

Keep this PR open, but do not merge it as-is. The report targets a real shared auth-profile cooldown problem, but the patch also drops ambiguous OpenAI Codex failures before the provider-owned WHAM cooldown probe can run, which would regress current Codex profile health handling.

## What This Changes

The PR narrows the embedded runner auth-profile failure policy so only selected auth-health failover reasons mark shared profiles failed, and adds focused unit coverage for rejected non-auth reasons and accepted auth reasons.

## Best Possible Solution

Keep the narrower auth-health policy direction, but preserve provider-owned health probes. A maintainable fix should either make the policy provider-aware or pass ambiguous reasons through to `markAuthProfileFailure` where provider-specific code can decide whether to cooldown, with regression coverage for both the GitHub Copilot format case and Codex WHAM ambiguous failures.

## Reproduction Assessment

Yes. The source path is high-confidence: current main sends non-null policy reasons into `markAuthProfileFailure`, null returns skip that call, and existing tests prove `unknown` Codex failures are expected to produce WHAM-backed cooldown handling.

## Solution Assessment

No. The generic narrowing is the right direction for Copilot format failures, but returning null for all ambiguous reasons is not the narrowest safe fix because it bypasses established provider-specific Codex cooldown logic.
