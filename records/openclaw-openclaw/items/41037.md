## Summary

Keep this PR open. Current main and v2026.4.27 still reject auth profiles that omit `provider`, so the linked user problem remains valid, but the proposed normalization needs revision because it can silently convert malformed explicit provider values into valid inferred strings instead of letting schema validation reject them.

## What This Changes

The branch adds pre-schema auth-profile normalization that infers a missing `provider` from `<provider>:<name>` profile keys and adds config schema regression tests for inferred and explicit-provider cases.

## Best Possible Solution

Implement the inference as a narrow config-normalization step after deprecated-key stripping and before legacy/schema validation, with a guard that only fills missing or blank string providers; preserve other explicit values for Zod and cover that with a regression test.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve invalid provider values for schema validation:** `src/config/validation.ts:247-248`
  - body: The backfill treats every non-string `profileValue.provider` as absent, then writes an inferred provider from the profile id. A config such as `{ provider: 123, mode: "oauth" }` would pass even though current schema rejects non-string providers; only fill when `provider` is missing or a blank string.
