## Summary

Keep this PR open. Current main does not implement the Brazilian WhatsApp ninth-digit equivalence, and the PR still targets a real linked user problem from #20187. It is not safe to close as implemented or obsolete because current main still uses exact normalized-number equality in the relevant allowlist, command authorization, same-phone, heartbeat, reaction, and session-routing paths. The PR also has unresolved later review comments about preserving the original chat target for reactions and outbound session routing after canonical allowlist matching, so the best path is to revise/rebase rather than close.

## Best Possible Solution

Keep the PR open and ask for a rebase onto current main that ports the BR-variant equivalence into `extensions/whatsapp`. The revised fix should use equivalence for authorization and allowlist checks while preserving the original explicit chat target for reactions, quote lookup, and outbound session routing. Add focused regression tests in the current plugin paths for normalization, outbound allowlist matching, inbound access/command auth, self-chat, heartbeat recipients, reactions, and per-peer session routing.

## Risks / Open Questions

- The PR changes auth-sensitive allowlist equivalence. The implementation should keep the match narrowly bounded to WhatsApp direct BR mobile variants and avoid broadening groups, LID/JID handling, or unrelated countries by accident.
- The canonical-allowlist rewrite in the PR context can affect code-execution-adjacent channel actions, especially reactions and session routing. Authorization equivalence should be separated from the actual chat target used for message keys and per-peer session identity.
- Current main has moved this surface into the bundled WhatsApp plugin under `extensions/whatsapp`, so the PR needs a rebase/port from the older `src/whatsapp` shape before it can be reviewed or merged cleanly.
