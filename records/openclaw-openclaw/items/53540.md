## Summary

Keep open. Current main has configurable provider/request timeout plumbing, but the default OpenRouter to Anthropic path still resolves through the OpenAI-compatible transport, native Anthropic fine-grained tool-argument streaming is not applied there, and no current regression proves that a silent stream while large tool-call arguments are generated is handled.

## What This Changes

The issue requests that embedded LLM calls avoid or clearly handle `Network connection lost` failures when a provider withholds large tool-call parameters long enough to exceed the current request or idle timeout window.

## Best Possible Solution

Keep this as the canonical tracker for OpenRouter/Anthropic large tool-parameter generation failures. The durable fix should add a synthetic OpenAI-compatible streaming regression that withholds tool-call argument chunks, then implement the chosen provider/runtime policy: supported provider-side tool-argument streaming, a safe timeout policy for silent tool-argument generation, or a tool-loop strategy that avoids one huge `Write` payload.
