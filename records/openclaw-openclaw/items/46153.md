## Summary

Keep open. Current main still generates the macOS Gateway LaunchAgent without `StandardInPath=/dev/null`, and the install plus restart-repair flows both use that same plist renderer. The broader #58414 was closed as superseded by this focused issue, and no open PR was found carrying a closing fix, so this remains a narrow actionable daemon repair.

## What This Changes

The issue requests that macOS Gateway LaunchAgent startup stop hanging silently under launchd by making the generated service definition safe for launchd-spawned processes without interactive stdin.

## Best Possible Solution

Add `StandardInPath` pointing to `/dev/null` in the central macOS Gateway LaunchAgent plist generation, or an exactly equivalent central stdin-closing wrapper, so fresh installs and restart-repair rewrites both produce a daemon-safe plist. Cover both generated-plist and restart-rewrite behavior with focused tests, add a user-facing changelog fix entry, and run a macOS LaunchAgent smoke when available.

Yes. The issue provides concrete `launchctl` reproduction steps, and the current source still matches the implicated condition by generating LaunchAgent plists without `StandardInPath`; live launchd execution was not run in this read-only pass.
