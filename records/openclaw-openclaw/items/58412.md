## Summary

Keep PR #58412 open. Current main does not implement the malformed KimiCodingPlan/Kimi Coding message ordering workaround, and I did not find an equivalent provider-side repair. The PR is also not ready to close as implemented or superseded because its own review context contains unresolved correctness concerns around asynchronous synthetic message_end handling and stale-end correlation.

## Best Possible Solution

Keep the PR open and ask for a reworked fix. The best path is a focused regression test for the Kimi malformed sequence from #58358 plus an implementation that preserves buffered text, awaits or serializes synthetic closing with the existing event chain, and correlates delayed stale `message_end` events by message identity or another reliable boundary rather than visible text alone. If this is solved provider-side, `extensions/kimi-coding/stream.ts` is the likely boundary; if solved generically, the change belongs in `src/agents/pi-embedded-subscribe.handlers.messages.ts` with state added to `EmbeddedPiSubscribeState`.
