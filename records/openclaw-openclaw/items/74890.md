## Summary

Keep this issue open. Current main has strong recovery and the browser Control UI path now uses gateway-side base-hash validation, but the macOS direct/local fallback writer can still commit a root config that drops `gateway.mode` and only records the anomaly after the write.

## What This Changes

The issue requests that Dashboard/Control UI/macOS app config writes preserve required root envelope fields such as `meta` and `gateway.mode` instead of producing partial config snapshots that trigger last-good restore churn.

## Best Possible Solution

The macOS app should never replace an existing valid `openclaw.json` with a root missing required envelope fields; direct local fallback should merge against the freshest source config and validate or reject before rename, while keeping the existing gateway RPC `config.set`/`config.patch` semantics for browser Control UI writes.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to start with an existing config containing `gateway.mode: local`, then call `OpenClawConfigFile.saveDict` with a partial root whose `gateway` object lacks `mode`; current main writes the payload and only logs `gateway-mode-removed` afterward.

## Solution Assessment

Yes, a narrow macOS writer fix is the right direction. The safer alternative to caller-by-caller patches is a shared guard in `OpenClawConfigFile.saveDict` or `ConfigStore` local fallback that mirrors the TypeScript writer's pre-commit destructive-write rejection.
