## Summary

Current main still does not recover an active Control UI/WebChat run across a WebSocket reconnect: the hello path clears UI run/stream state, the Gateway snapshot has no active-run recovery field, and open PR #41067 remains the relevant implementation candidate rather than a merged fix.

## What This Changes

Control UI/WebChat should avoid terminating or losing active AI sessions across WebSocket reconnects and should preserve or recover in-flight session state.

## Best Possible Solution

Keep this issue open as the canonical Control UI/WebChat active-run reconnect recovery report. Review and refresh PR #41067 or land an equivalent focused fix that adds an additive active chat-run recovery contract in the Gateway hello/snapshot or a dedicated reconnect RPC, preserves/reconciles chatRunId and busy/stream state across reconnect, avoids clearing active state until the run is known terminal, and adds regression coverage for WebSocket close/reconnect during an active chat.send.
