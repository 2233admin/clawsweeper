## Summary

Keep open. This PR targets the right Active Memory startup and timeout problem, but the new memory-only fast path bypasses normal embedded-runner provider setup used by inherited/current-session models, so it needs a branch update before merge. https://github.com/openclaw/openclaw/pull/75200

## What This Changes

This PR adds a memory-only embedded runner path for Active Memory, threads runtime `toolsAllow` into core and plugin tool construction, and removes the prior 30s Active Memory setup grace from the blocking deadline.

## Best Possible Solution

Keep the early tool-construction filtering and tighter Active Memory deadline, but share or reuse the normal embedded-runner provider setup so memory-only runs still apply configured extra params, provider stream wrappers, prompt-cache setup, and tool-call compatibility.

## Reproduction Assessment

Yes. The linked bug has direct-chat Active Memory steps and timeout logs, and current main shows Active Memory sends a narrow `toolsAllow` while still using setup grace. The new PR finding is reproducible by comparing the memory-only early return with the normal provider setup path in source.

## Solution Assessment

No. The direction is narrow and maintainable, but the current PR shape should not replace the normal provider setup with a smaller memory-only stream path; the safer fix is to preserve that setup while avoiding unrelated tool materialization.
