## Summary

Keep open. Current main does not implement this PR's intended Active Memory startup change, and the PR still has a correctness bug: the new memory-only early return bypasses embedded-run provider setup that current main applies to normal embedded attempts.

## What This Changes

The PR adds an Active Memory memory-only embedded-run mode, threads runtime `toolsAllow` into core/plugin tool construction, and changes Active Memory timeout handling/tests to use the configured recall timeout plus a small transcript readback grace.

## Best Possible Solution

Keep the early tool-construction filtering and tighter Active Memory blocking deadline, but route memory-only Active Memory runs through shared embedded-run provider setup so configured extra params, provider stream wrappers, transcript repair, and tool-call compatibility remain consistent with normal embedded attempts.

## Reproduction Assessment

Yes. The linked bug has direct-chat Active Memory steps and timeout logs, and the PR-specific regression is reproducible by source inspection: the new memory-only early return runs before the normal provider setup path while Active Memory can inherit the current session model.

## Solution Assessment

No. The direction is narrow, but the current implementation should not fork a reduced embedded runner path that skips provider setup; the safer fix is to share or factor the normal setup and only avoid constructing tools that cannot survive `toolsAllow`.
