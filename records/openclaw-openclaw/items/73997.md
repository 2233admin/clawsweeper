## Summary

Keep open. Current main still does not implement the requested WhatsApp read-only DM observe policy: the config contract rejects the new value, blocked DMs are dropped before any observation store, and admitted DMs still enter the normal session and agent-turn path. The existing sendPolicy and core observeOnly primitives are adjacent but do not satisfy the issue’s mechanical no-session/no-turn requirement.

## What This Changes

Add a WhatsApp `dmPolicy: "observe"` mode that records incoming DMs in a read-only/queryable store without creating agent sessions, firing agent turns, or exposing a reply path.

## Best Possible Solution

Define the privacy and product contract first, then implement a WhatsApp-owned pre-routing observe policy: accept the config value, append bounded observation records before session resolution, skip hooks/route/session/agent dispatch for observed DMs, and expose a guarded read tool with retention, redaction, account, sender, and time-range controls.

## Reproduction Assessment

Yes for source-level verification, not as a live WhatsApp repro. Current main rejects `dmPolicy: "observe"`, drops blocked DMs before persistence, and sends admitted DMs through `runInboundReplyTurn`.

## Solution Assessment

Unclear until maintainers approve the privacy model. A WhatsApp pre-routing observe branch is the narrowest direction that matches the issue; using `session.sendPolicy` or the generic `observeOnly` turn admission would still create/record session context or process inbound messages.
