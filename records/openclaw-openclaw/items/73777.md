## Summary

Keep open. Current main has code intended to enqueue and wake main-session cron system events, but the reported manual `cron.run` symptom is not fixed or obsolete: `deliveryStatus: "not-requested"` is still the expected run-history value for main/sessionTarget `systemEvent` jobs without fallback delivery, and existing Gateway coverage does not prove that the CLI/Control UI manual-run path creates a user-visible agent turn for `wakeMode: "now"`.

## Best Possible Solution

Keep the issue open. Add a targeted regression test for manual `cron.run` through the Gateway/Control UI path using `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and default or explicit `wakeMode: "now"`; assert the event reaches the intended session, the heartbeat runner uses the cron-event prompt, the event is consumed, and a user-visible agent turn/output occurs. If `deliveryStatus: "not-requested"` is intended, add a separate wake/heartbeat execution status or clearer UI/docs so run history does not look like a lost cron event.

## Risks / Open Questions

- `deliveryStatus` is overloaded: in current code it means fallback delivery status, not proof that a main-session heartbeat turn executed. Treating `not-requested` as either definitely broken or definitely fine would be misleading without an agent-turn assertion.
- Existing tests leave a real product gap: they prove service-level wake calls and heartbeat prompt handling separately, but not the full manual CLI/Control UI path from `cron.run` to a visible main-session agent turn.
