## Summary

Current main still does not give enough evidence to close this regression report. The manual CLI/Control UI path queues `cron.run`, main-session `systemEvent` jobs enqueue and wake heartbeat, and heartbeat can build a cron-event prompt, but the run-history field reported by the issue still records `deliveryStatus: "not-requested"` for main jobs without fallback delivery. Existing coverage locks those pieces separately and even asserts the reporter-visible `not-requested` status; it does not prove the manual Gateway/UI path creates a visible main-session agent turn/output.

## Best Possible Solution

Keep this issue open. Add a focused regression test for manual `cron.run` through the Gateway/Control UI path with `sessionTarget: "main"`, `payload.kind: "systemEvent"`, and `wakeMode: "now"`; assert the event reaches the intended main session, heartbeat receives the cron-event prompt, the event is consumed, and a visible main-session agent turn/output occurs. If `deliveryStatus: "not-requested"` is intentionally only fallback-delivery status, add a distinct wake/heartbeat execution status or clearer UI/docs so operators can distinguish a successful main-session wake from a lost cron event.
