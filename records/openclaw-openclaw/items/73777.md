## Summary

Keep open. Current main explains why the reporter still sees `deliveryStatus: "not-requested"`, but that field is fallback-delivery state rather than proof that a manual main-session cron system event was consumed by the heartbeat turn; the remaining gap is an end-to-end regression for the manual Gateway/Control UI run path.

## What This Changes

The issue reports that manually running a `sessionTarget: "main"` cron job with a `systemEvent` payload records `deliveryStatus: "not-requested"` and does not wake the main agent session.

## Best Possible Solution

Add an end-to-end regression around manual `cron.run` for a main-session `systemEvent` job, then either fix the narrow handoff bug it exposes or add a distinct execution/wake status so operators can tell successful main-session consumption apart from missing fallback delivery.

Unclear for current main. The issue gives concrete v2026.4.26 steps, and source inspection explains the `not-requested` result, but the checkout still lacks a high-confidence current-main reproduction proving that the manual run fails to create the intended heartbeat/agent turn.

## Solution Assessment

Unclear. The current implementation is directionally right because it enqueues the system event and queues a targeted heartbeat wake, but the best next fix is to close the end-to-end proof gap before changing status semantics or cron delivery policy.
