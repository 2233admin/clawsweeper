## Summary

Keep open. The PR adds a real missing file-transfer plugin and gateway policy seam that current main does not have, but it still has a blocking `dir.fetch` authorization time-of-check/time-of-use gap: preflight entries are authorized, then a fresh archive is fetched and only the root canonical path is rechecked before the archive can be returned and unpacked.

## What This Changes

The branch adds a bundled file-transfer plugin with file and directory node commands/tools, a generic plugin-owned `node.invoke` policy seam in the gateway and SDK, node-tool redirects, media-store subdir hardening, docs, changelog, labeler, lockfile, and tests.

## Best Possible Solution

Keep the generic plugin-owned gateway policy seam, but bind `dir.fetch` success to the exact authorized contents: either fetch an archive tied to the preflight manifest or re-list/recheck the actual archive entries against `evaluateFilePolicy` before any success path returns or unpacks bytes.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Recheck dir.fetch archive entries after the actual fetch:** `extensions/file-transfer/src/shared/node-invoke-policy.ts:694`
  - body: The policy approves the `entries` returned by the preflight call, but the later `ctx.invokeNode({ params: forwardedParams })` performs a fresh `dir.fetch` and only post-checks the root canonical path. If the directory changes after preflight, or a node returns different archive contents, entries outside `allowReadPaths` can still be returned and unpacked without `evaluateFilePolicy` seeing them.
