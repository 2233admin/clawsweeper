## Summary

Keep open. Current main does not contain the file-transfer plugin or the plugin node-invoke policy surface, so this PR is not obsolete; it is the active replacement carrying forward #74134 with attribution. The branch is not merge-ready because `dir.fetch` authorizes a preflight directory snapshot, then performs a second fetch whose archive entries are not rechecked against the path policy before tool-side unpacking.

## What This Changes

The PR adds a bundled file-transfer plugin, file and directory node-host commands/tools, a plugin-owned node-invoke policy seam, media-store subdir hardening, and related docs, labeler, changelog, SDK registry, gateway, and tests.

## Best Possible Solution

Keep the generic gateway policy seam, but make `dir.fetch` authorize the exact archive contents that are ultimately returned, either by binding the archive to the preflight manifest or by re-listing and rechecking actual archive entries before any success path can expose them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Recheck dir.fetch archive entries after the actual fetch:** `extensions/file-transfer/src/shared/node-invoke-policy.ts:694`
  - body: The policy approves the `entries` returned by a preflight call, but this later call performs a fresh `dir.fetch` and the postflight path check only covers the root canonical path. If the directory changes after preflight, or a node returns different archive contents, entries not covered by `allowReadPaths` can still be unpacked by the tool without `evaluateFilePolicy` ever seeing them. Please bind the archive to the authorized manifest or recheck the actual archive entries against the policy before returning success.
  - confidence: 0.87
