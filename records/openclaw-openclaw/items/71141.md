## Summary

Keep open: current main still lacks the reported Telegram reaction-directive behavior, and this draft PR is the active paired implementation for the linked issue, but the branch has a blocking type problem before it can merge.

## What This Changes

The PR parses `[[react:...]]` and `[[react_to_current:...]]` into Telegram channel data, permits reaction-only Telegram replies, and calls native Telegram reaction delivery before optional text/media sends.

## Best Possible Solution

The best end state is to keep the shared parser change, preserve current-main markdown image extraction, carry reaction intent as structured channel data, and apply native Telegram reactions only when a valid reply target exists, with parser, payload, delivery, and adapter tests covering reaction-only and text/media-plus-reaction replies.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: `[[react_to_current:...]]` is not parsed into reaction metadata, and reaction-only Telegram replies reach the no-text/no-media rejection path instead of `reactMessageTelegram`.

## Solution Assessment

No, not as currently submitted. The implementation direction is right, but the branch needs a rebase and a type fix for `gatewayClientScopes` before it is the narrow maintainable solution.
