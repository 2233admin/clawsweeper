## Summary

Keep open. Current main still matches the reported Signal SSE timeout path: the Signal client keeps a 10s default deadline for event streams, the monitor reconnect loop calls it without a long-lived-stream override, and the v2026.4.27 Signal files are unchanged on this surface. The issue is also paired with open fixing PR #74744, with #73086 as an overlapping alternative, so it should remain open until maintainers choose and land a fix.

## What This Changes

The Signal channel should consume signal-cli `/api/v1/events` receive events without aborting the long-lived stream after 10 seconds, then reflect inbound activity in channel status.

## Best Possible Solution

Review and land one narrow Signal-plugin fix, preferably one of the existing open PRs, that disables the short deadline only for the long-lived receive stream while preserving bounded RPC/probe/send deadlines, reconnect cleanup, parsing limits, and access-control behavior. The fix should include coverage for a delayed or idle SSE receive stream and status activity after an inbound event.

## Reproduction Assessment

Yes for the core timeout path: a local `/api/v1/events` SSE endpoint that accepts the connection but delays response headers or the first receive event beyond 10 seconds exercises the same default `streamSignalEvents` path and rejects with the reported error. Live `signal-cli` was not run here, and `lastInboundAt` should be validated with the chosen fix.

## Solution Assessment

No, current main is not the best solution because the long-lived monitor stream still inherits the bounded 10s client deadline. The maintainable solution is a Signal-local stream timeout override or disabled deadline for the receive monitor only, with RPC/check/send deadlines left bounded; existing PRs #74744 and #73086 are the right review candidates.
