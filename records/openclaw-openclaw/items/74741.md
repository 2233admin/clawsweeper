## What This Changes

The issue requests that the Signal channel consume the signal-cli `/api/v1/events` SSE stream without aborting idle streams after 10 seconds, and update inbound activity when messages arrive.

## Best Possible Solution

Review and land one narrow Signal-plugin fix that disables the short deadline only for the long-lived receive stream while preserving bounded RPC/probe/send deadlines, abort cleanup, reconnect behavior, parsing limits, and access-control behavior. Add coverage for an idle SSE receive stream and for inbound activity/status after a received event.

## Reproduction Assessment

Yes. A local `/api/v1/events` SSE endpoint that accepts the connection but remains idle past the current default deadline exercises the same monitor path and produces the reported timeout; the provided live logs show the same failure against signal-cli.

## Solution Assessment

No, current main is not the best solution because the monitor stream still inherits the bounded 10s deadline. The maintainable fix is Signal-local: disable or override that deadline only for the long-lived receive monitor while keeping RPC/check/send deadlines bounded.
