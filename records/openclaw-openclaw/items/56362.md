## Summary

Keep open. The PR is aimed at a real current-main gap in reset-hook emission for custom reset triggers, but the proposed gateway transcript path change introduces blocking correctness and path-safety regressions that should be fixed before merge.

## What This Changes

This PR changes custom reset-trigger fallback hook emission, gateway reset transcript path rotation, and adds Discord bang-prefix reset-trigger coverage.

## Best Possible Solution

Land a revised branch that emits reset hooks for every matched custom resetTrigger, rotates generated/default transcript files to a fresh sessionId, preserves only validated explicit custom transcript files, and covers bang triggers, in-dir custom filenames, and invalid outside sessionFile fallback.

## Reproduction Assessment

Yes. Current main can be traced from resetTriggers matching in initSessionState to the slash-only fallback in get-reply.ts, and the PR regression can be checked against the existing custom-owned-child transcript reset test shape.

## Solution Assessment

No. The hook-emission change is the right narrow direction, but the session reset path should validate preserved transcript paths and distinguish explicit custom filenames from stale generated filenames instead of using directory location alone.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Preserve explicit in-dir sessionFile paths:** `src/gateway/session-reset-service.ts:336-340`
  - body: The custom path check only treats a transcript as custom when its directory is outside `defaultSessionsDir`. Existing reset coverage seeds `custom-owned-child-transcript.jsonl` in the store directory and expects reset to keep that exact path; this branch would rewrite it to `<nextSessionId>.jsonl`, splitting pinned transcript storage.
  - confidence: 0.91
- **[P1] Validate custom transcript directories before writing:** `src/gateway/session-reset-service.ts:339`
  - body: When `currentEntry.sessionFile` is outside `defaultSessionsDir`, the branch uses its directory directly and later creates/writes `next.sessionFile` there. Existing path helpers fall back for sessionFile values outside known agent session dirs, so this bypass can write under arbitrary stale or corrupt persisted paths.
  - confidence: 0.86

## Security Review

Existing resolver behavior falls back for outside unknown paths, so this should be validated through the existing path helpers before writing.
