## Summary

Keep open: current main still exposes isolated heartbeat session rows as normal Control UI chat choices and sends WebChat input to the selected key, so the reported misrouting remains a plausible current-main bug rather than a stale or duplicate report.

## What This Changes

The issue reports that Control UI WebChat can route user messages into an isolated `:heartbeat` session when pruning removes the base main session row.

## Best Possible Solution

Add an explicit synthetic/background session contract for isolated heartbeat rows, preferably marker-based instead of suffix-based, then make Control UI hide or render those rows read-only and normalize default chat selection back to the resolved main session when the active key is a heartbeat sibling. The base main session should remain selectable or be recreated when pruned, and user-originated `chat.send` should not default to a heartbeat row.

## Reproduction Assessment

Yes. The issue provides concrete configuration and steps, and current code inspection shows maintenance can prune an unpreserved base row while heartbeat persists a marked `:heartbeat` row that the session picker can show and `chat.send` can target.

## Solution Assessment

Yes for keeping this open and attempting a narrow fix. The safest implementation is to expose/use the existing `heartbeatIsolatedBaseSessionKey` marker across the session-list/UI boundary rather than relying only on a `:heartbeat` suffix, because real user sessions can legitimately contain that suffix.
