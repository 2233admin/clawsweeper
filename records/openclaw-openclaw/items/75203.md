## Summary

Keep open. Current main still exposes the reported route: the stored heartbeat-isolated marker is not projected through the gateway/UI session-row contract, the session picker still adds ordinary heartbeat-suffixed rows, and WebChat still sends to the selected key directly. The issue is also paired with open closing PR #75356, so it should remain open until that PR lands or is replaced.

## What This Changes

The issue requests that Control UI WebChat avoid listing or sending user input into synthetic isolated heartbeat sessions after the primary session row is pruned.

## Best Possible Solution

Land or replace the narrow marker-backed fix: project heartbeatIsolatedBaseSessionKey or an equivalent synthetic-session flag through the gateway session-list contract, hide or normalize heartbeat-isolated rows in the Control UI picker, and guard chat.send so stale heartbeat selections route to the base session while preserving real user sessions that merely end in :heartbeat.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by presenting sessions.list with only agent:main:main:heartbeat after the base row is pruned: current main adds that row to the picker and sends WebChat input with sessionKey: state.sessionKey.

## Solution Assessment

Yes, the active PR direction is the narrow maintainable fix if review confirms the details: use the stored heartbeat-isolated marker rather than suffix-only matching, normalize stale WebChat sends back to the base session, and add focused UI/gateway regression coverage.
