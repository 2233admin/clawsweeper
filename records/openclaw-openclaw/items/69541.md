## Summary

Keep open. Current main still lacks the requested declarative plugin-injected context tag contract: plugin manifests do not declare injected context tags, prompt hook results only carry raw injection strings, and downstream consumers still rely on local hardcoded sanitizer knowledge. This is a public plugin manifest/SDK plus privacy/sanitization design issue, so it needs maintainer API review rather than autoclose, ClawHub routing, or a narrow automated repair PR.

## What This Changes

The issue requests an additive manifest/SDK registry for plugin-injected context tags so downstream display, compaction, export, and memory ingestion paths can strip declared plugin-owned context blocks from one source of truth.

## Best Possible Solution

Keep this open for maintainer API review. The likely end state is an additive, documented manifest/SDK contract for plugin-owned injected context blocks, a normalized aggregated registry/getter built from plugin manifests, and staged adoption by display/history, assistant-visible text, compaction, session export, and memory/dreaming ingestion with tests for user-authored lookalikes and fenced-code preservation.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a design RFC, but there is high-confidence current-main verification: source search and file inspection show no manifest field/getter and multiple consumers still hardcode tag knowledge.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainer API review, but the manifest-backed registry direction is the narrowest maintainable product shape visible from current code because producers already have manifest contracts and consumers need shared static metadata before plugin runtime loads.
