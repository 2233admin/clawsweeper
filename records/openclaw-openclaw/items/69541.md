## Summary

Keep this issue open. Current main still lacks the requested declarative plugin-injected context tag registry: plugin manifests only declare existing static capability ownership lists, prompt hooks still return raw context strings, and downstream consumers still strip known prompt-injected blocks through local hardcoded rules. This is an additive public plugin manifest/SDK and sanitizer contract with privacy implications, so it needs maintainer API review rather than closure or an automated one-shot fix.

## What This Changes

Add a plugin manifest/SDK contract for declaring prompt-injected context tags and expose an aggregated registry that downstream display, compaction, session export, and memory/dreaming ingestion paths can use for stripping or sanitization.

## Best Possible Solution

Keep this issue open for maintainer API review. The best path is an additive manifest/SDK contract for plugin-owned injected context blocks, an aggregated registry exposed through a stable core/plugin-facing getter, and staged adoption by display/history, assistant-visible text, compaction, session export, and memory/dreaming ingestion paths. Coordinate with #54373, but do not treat it as a duplicate because provenance-for-model-reasoning and downstream strip/sanitize contracts solve different parts of the same boundary problem.
