## Summary

Keep this PR open. Current main and v2026.4.26 still lack a per-channel/account/recipient outbound delivery serializer, so the branch is not obsolete, but the proposed implementation still needs maintainer/contributor rework before merge because it serializes after queue I/O, derives the key from raw fields, omits thread/cancellation/hung-send handling, and mixes in unrelated Mattermost dependency churn.

## What This Changes

This PR adds an in-memory outbound `DeliverySerializer`, wraps `deliverOutboundPayloadsCore` with it, adds serializer unit tests, and also adds an unrelated Mattermost `@sinclair/typebox` runtime dependency plus lockfile entries.

## Best Possible Solution

Keep this PR open for maintainer/contributor rework. The best path is to rebase on current main, remove or split the Mattermost dependency change, acquire same-target ordering before queue I/O can reorder calls, derive an unambiguous normalized key from channel/account/recipient/thread identity, preserve abort behavior while waiting, bound or recover from hung sends, and add focused regression coverage for queue/recovery interaction plus channel target normalization.

## Risks / Open Questions

- The PR currently serializes after write-ahead queue I/O, so two same-target calls can still invert before they reach the mutex.
- The proposed key uses raw `channel:accountId:to` fields, which can bypass serialization for equivalent normalized targets or over-serialize distinct threaded destinations.
- A per-destination in-memory queue can create head-of-line blocking if a channel send hangs; current Discord webhook sends use bare `fetch()` without a visible timeout in the inspected path.
- The branch adds an unrelated Mattermost runtime dependency and lockfile change, creating unnecessary supply-chain/package-review surface for an outbound-ordering fix.
