## Summary

Keep this PR open. Current main and v2026.4.26 still do not have a destination-level outbound delivery serializer, so the intended ordering fix is not obsolete; however, the proposed branch needs maintainer/contributor rework before merge because review evidence shows queue-ordering, target-normalization, cancellation, hung-send, and unrelated dependency-change concerns.

## What This Changes

This PR adds an in-memory outbound DeliverySerializer, wraps core outbound delivery with it, adds serializer unit tests, and also changes the Mattermost plugin package and lockfile to add @sinclair/typebox.

## Best Possible Solution

Implement the ordering fix as a focused outbound change: acquire same-destination ordering before queue writes can reorder live sends, derive an unambiguous normalized key from channel/account/recipient/thread identity through existing channel/plugin seams, preserve abort semantics while waiting, add a bounded or recoverable path for hung sends, and split out unrelated package dependency changes.

## Risks / Open Questions

- The proposed serializer is acquired after write-ahead queue I/O, so two same-target calls can still invert before they reach the mutex.
- The proposed key uses raw channel/account/recipient strings, which can bypass serialization for equivalent normalized targets or serialize unrelated threaded destinations together.
- A per-destination in-memory queue can create head-of-line blocking if a channel send hangs; the inspected Discord webhook path has no visible timeout at the bare fetch boundary called from outbound delivery.
- The branch mixes in an unrelated Mattermost runtime dependency and lockfile change, creating unnecessary supply-chain and package-review scope for an outbound-ordering fix.
