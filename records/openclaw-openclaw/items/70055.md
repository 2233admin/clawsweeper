## Summary

Keep #70055 open. Current main no longer matches the exact old implementation described in the issue: external CLI credentials are now resolved as runtime-only overlays and the previous user-facing `info` sync message is not present. However, the requested first-class disable surface is still missing: external CLI auth discovery remains hardwired into auth-profile overlay resolution, `auth` config has no `externalCliSync`/disable field, and the only `syncExternalCli` options currently found are not wired into the runtime overlay path that reads Codex/Claude/MiniMax CLI credentials.

## Best Possible Solution

Keep the issue open and implement a small core-owned opt-out because the behavior is bundled auth-profile resolution, not an optional third-party plugin request. Add a typed `auth.externalCliSync` config surface or an environment override, thread it into `overlayExternalAuthProfiles`/`resolveExternalAuthProfileMap`, remove or correctly wire the stale `syncExternalCli` option if appropriate, and add focused tests proving Codex/Claude/MiniMax external profiles are skipped while default-on behavior remains unchanged. Update the auth/config docs when the public config key is added.

## Risks / Open Questions

- Current main appears to have reduced the original log-noise symptom by using debug-level runtime overlay messages rather than the old `info` sync line, so maintainers should scope the remaining work to the still-missing disable control rather than re-litigating the old logging path.
- Without an opt-out, users who intentionally keep upstream CLI auth on the same host still cannot prevent OpenClaw from reading those CLI credential sources and surfacing runtime-only profiles.
