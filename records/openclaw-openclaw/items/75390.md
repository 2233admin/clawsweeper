## Summary

Keep this issue open. Current main still defines both `visibleReplies` config fields as strict string enums, invalid snapshots still make gateway hot reload skip, and the issue is already paired with an open PR that uses closing syntax, so it should remain open until that PR or an equivalent fix merges.

## What This Changes

Accept or otherwise give actionable handling for boolean `messages.visibleReplies` and `messages.groupChat.visibleReplies` values so they do not invalidate config and leave channels empty without clear guidance.

## Best Possible Solution

Review and land PR #75392 or an equivalent schema-layer normalization that accepts `true`/`false` at both `visibleReplies` sites while preserving the downstream string union, with focused config-schema tests and any needed user-facing changelog/docs note. After the fix merges, this issue can close as implemented.

## Reproduction Assessment

Yes. A config containing `messages.groupChat.visibleReplies: true` or `messages.visibleReplies: true` is outside the current Zod enum, and current reload code skips invalid snapshots; the reporter also supplied a live v2026.4.29 reproduction affecting multiple channels.

## Solution Assessment

Yes, the linked schema-level coercion direction appears to be the narrowest maintainable fix because all config load, reload, doctor, and config mutation paths share those schemas while downstream code can keep using `"automatic" | "message_tool"`. A separate status-message improvement could help diagnostics, but it would not by itself prevent the outage.
