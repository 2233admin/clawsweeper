## Summary

Keep this issue open. Current main still synthesizes visible media placeholder text for blank media-only Signal and iMessage sends, audio MIME types still map to the `audio` placeholder kind, and the strict `messages` config/schema/docs surface still has no `suppressMediaPlaceholders` or granular `mediaPlaceholders` option. The related implementation PRs are closed unmerged, so there is no implemented-on-main or superseding fix to close against.

## Best Possible Solution

Keep this open as the canonical product/config request. A focused fix should add a documented default-off `messages.suppressMediaPlaceholders` option, update typed config, strict zod schema, generated schema/help/labels/docs, preserve current placeholder behavior by default, and wire the option through Signal and iMessage media-only send paths with focused regression tests for audio and image/document media sends.

## Clownfish Work Prompt

Implement the canonical fix for https://github.com/openclaw/openclaw/issues/15840. Add a documented, default-off `messages.suppressMediaPlaceholders?: boolean` option that suppresses visible `<media:...>` fallback text for media-only outbound sends while preserving current behavior when unset/false. Scope the first PR to Signal and iMessage, because current main has explicit placeholder synthesis in `extensions/signal/src/send.ts` and `extensions/imessage/src/send.ts` and prior closed PRs targeted those paths. Update `MessagesConfig`, the strict messages zod schema, schema help/labels/generated schema metadata as required by the repo, and user-facing docs under the messages/TTS/channel config docs. Add focused tests for blank media-only sends with audio and image/document media: default behavior still emits placeholders; `messages.suppressMediaPlaceholders: true` avoids visible placeholder text while still sending the attachment. If Signal requires a non-empty message body, use a channel-appropriate non-visible fallback only under the opt-in setting and cover it in tests. Do not change inbound media placeholders, prompt/history media context, transport auth, unrelated TTS behavior, or provider/channel defaults. Include a user-facing changelog entry with proper Thanks attribution.
