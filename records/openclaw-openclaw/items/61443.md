## Summary

Keep open. Current main does not already implement provider-level retry, and the latest PR head is still an active, merge-conflicted implementation with blocking correctness issues around unsupported transports and timeout preservation.

## What This Changes

The branch adds opt-in `models.providers.<name>.retry` configuration, provider retry policy/registry code, abort-aware retry plumbing, OpenAI-compatible transport wrapping, generated config schema/help updates, focused retry tests, and several test-fixture type cleanups.

## Best Possible Solution

Land one canonical provider-retry implementation after rebase. The finished design should either wire retry through every advertised model transport or explicitly scope/validate it to OpenAI-compatible providers, preserve current timeout request options, keep generated config surfaces aligned, and include focused regression coverage for config reloads, retry classification, aborts, Retry-After handling, and supported transports.

## Reproduction Assessment

Not applicable as a user-facing bug reproduction; this is a feature PR review. The blocking issues are statically reproducible from the PR head: generic config accepts `retry` while non-OpenAI transports do not consume it, and the PR helper body omits main's timeout option.

## Solution Assessment

No, not yet. The approach is plausible, but the current branch is not the best merge shape until it rebases cleanly, preserves OpenAI SDK timeout behavior, and resolves the generic-provider versus OpenAI-compatible-only retry scope.
