## Summary

Keep open. The cron parser guard is useful because current main still has the nullish `.trim()` crash path, but this PR should not merge as-is because it also adds an unrelated package-update early return that conflicts with the maintained same-version refresh contract and existing tests/docs.

## What This Changes

The PR widens `parseAbsoluteTimeMs` to accept nullish input with parser tests, and adds a package-update early exit when the current and target versions match.

## Best Possible Solution

Split the branch before merge: keep the cron nullish parser guard, add parser and CLI malformed-`at` regression coverage, and remove the package-update early return. Any same-version update optimization should be designed separately against the current refresh, plugin sync, completion, restart, and doctor contract.

## Reproduction Assessment

Yes. A malformed persisted cron job with `schedule: { kind: "at" }` and no `at` can flow through `printCronList` to `formatSchedule`, then into `parseAbsoluteTimeMs(undefined)`, which still calls `input.trim()` on current main.

## Solution Assessment

No. The cron guard is the right narrow direction, but this PR is not the best complete fix because it bundles an unrelated update-policy change that contradicts current docs and tests; the safer solution is a cron-only patch plus targeted CLI regression coverage.
