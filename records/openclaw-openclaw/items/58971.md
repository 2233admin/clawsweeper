## Summary

Keep open. Current main does not implement this opt-in, and the PR remains coherent and focused, but it needs maintainer/security policy review because it deliberately widens cron webhook SSRF behavior and overlaps open #49120's stricter reject-at-save direction.

## What This Changes

The PR adds a `cron.webhookAllowPrivateNetwork` config/docs/schema flag, forwards `{ allowPrivateNetwork: true }` into cron webhook delivery and failure webhook paths, and adds gateway cron tests for default and opt-in policy forwarding.

## Best Possible Solution

Choose one cron webhook network policy and implement it consistently: either land a tightly documented operator-only opt-in with final naming, warnings, schema/docs/tests, and all webhook paths covered, or supersede this with the stricter reject-at-save behavior from #49120.
