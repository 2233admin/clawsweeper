## Summary

Keep open: current main still uses a count-only Control UI history cap, so this PR is not obsolete, but the proposed patch is not merge-ready because it adds an unbounded recursive raw-content walk in the chat render path.

## What This Changes

The PR branch changes the Control UI chat item builder to cap visible history by message count and estimated render characters, adds coverage for tool-result and hidden-tool cases, and records the fix in the changelog.

## Best Possible Solution

Keep the client-side history character budget, but make the estimator iterative or explicitly bounded by remaining budget plus depth/node limits, then keep the focused tests for count caps, direct and nested tool-result content, hidden tool messages, and raw-walk limits.

## Reproduction Assessment

Yes. Current main has a clear source-level reproduction path: fewer than 200 chat-history messages with large renderable tool/text payloads still pass the count-only window, and the PR tests encode deterministic budget scenarios.

## Solution Assessment

No. The direction is appropriate, but the current implementation should bound raw-content traversal before merge so the render-budget guard cannot itself become a synchronous UI stall or stack-overflow path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Bound the raw content traversal:** `ui/src/ui/chat/build-chat-items.ts:156-169`
  - body: `estimateRawContentChars` recursively walks arrays and nested `content` fields without any remaining-budget, depth, or node cap. A deeply nested tool-result payload can now spend unbounded synchronous time or hit the call-stack limit in `buildChatItems` before the render budget protects the UI, which reintroduces the availability problem this PR is meant to reduce.
