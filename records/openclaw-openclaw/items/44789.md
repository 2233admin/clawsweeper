## Summary

Keep open. Current main still does not satisfy the reported LiteLLM onboarding regression: the interactive path asks for only the LiteLLM API key, keeps the proxy URL as an existing config value or `http://localhost:4000`, and exposes a one-model static catalog rather than discovering the configured proxy models.

## Best Possible Solution

Fix this in the bundled LiteLLM provider plugin. Add provider-owned interactive setup that prompts for the LiteLLM proxy base URL with the existing localhost default, collects/stores the API key, discovers selectable models from the configured OpenAI-compatible `/models` endpoint when reachable, falls back to manual model entry when discovery fails, and keeps non-interactive setup and existing config compatibility intact. Update LiteLLM-focused tests, docs, and the changelog.

## Clownfish Work Prompt

Fix openclaw/openclaw#44789. Current main still lets interactive LiteLLM onboarding collect only the API key, defaults the proxy URL to `http://localhost:4000` unless config already has `models.providers.litellm.baseUrl`, and exposes a one-model static catalog. Implement the fix in the bundled LiteLLM plugin, not generic core.

Expected boundary:
- Add provider-owned interactive LiteLLM setup that prompts for the LiteLLM base URL with default `http://localhost:4000`, collects/stores the LiteLLM API key, discovers models from the configured OpenAI-compatible `/models` endpoint when reachable, and lets the user choose a default model.
- Preserve a manual fallback model entry when discovery fails or returns no models.
- Keep existing non-interactive `--auth-choice litellm-api-key --litellm-api-key ... --custom-base-url ...` behavior compatible. Do not break existing configs that already set `models.providers.litellm.baseUrl` or `models.providers.litellm.models`.
- Prefer existing provider setup/discovery helpers such as `openclaw/plugin-sdk/provider-setup` where they fit; avoid core special cases for `litellm`.
- Update `extensions/litellm/openclaw.plugin.json`, LiteLLM docs, and add a user-facing changelog entry crediting `Thanks @Hylance` and, if appropriate, `Thanks @dsantoreis`.

Tests should cover interactive URL prompt/config persistence, discovery success, discovery failure/manual fallback, and non-interactive compatibility. Keep changes scoped to LiteLLM unless a tiny reusable helper extension is clearly needed.
