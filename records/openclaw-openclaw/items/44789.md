## Summary

Keep this issue open. Current main still does not satisfy the reported LiteLLM interactive onboarding regression: the bundled LiteLLM provider defaults to localhost, prompts only for the API key in interactive setup, and exposes a static one-model catalog rather than prompting for the proxy URL and discovering models from the configured proxy.

## Best Possible Solution

Keep this issue open and fix it in the bundled LiteLLM provider plugin. The best implementation is a provider-owned setup flow that prompts for the LiteLLM base URL with the current localhost default, stores that URL with the API key profile, discovers available models from the configured OpenAI-compatible `/v1/models` endpoint when reachable, offers a manual fallback when discovery fails, updates the default model from the chosen model, and adds focused LiteLLM onboarding tests plus docs for interactive and non-interactive parity.
