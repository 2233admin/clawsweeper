## Summary

Keep this issue open. Current main still does not implement Telegram inline-query UX: the Telegram plugin has no inline_query or chosen_inline_result runtime handler, no answerInlineQuery/InlineQueryResult response path, and no config/docs/test surface for enabling or constraining inline mode. Closed PR #51963 is useful reference material, but it was not merged and had security/correctness review concerns.

## What This Changes

The issue requests Telegram inline-query support so users can type @botname query from any Telegram chat, route the query through OpenClaw, and return shareable inline results with an explicit Telegram config surface.

## Best Possible Solution

Keep this as the canonical Telegram inline-query tracker. The desired end state is a reviewed implementation inside the bundled Telegram plugin with explicit config/schema/docs, conservative authorization and session semantics, bounded request handling, InlineQueryResultArticle responses, optional chosen-result handling where needed, and focused Telegram plugin tests; #51963 should be mined only as reference material.

## Risks / Open Questions

- Inline queries can originate from arbitrary Telegram contexts, so the feature needs explicit sender authorization, privacy/session scoping, rate limiting or debounce behavior, and bounded execution deadlines before routing through the agent flow.
- The current inlineQueries status flag can mislead operators because it reports Telegram bot account capability, not OpenClaw support for answering inline queries.
