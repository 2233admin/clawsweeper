## Summary

Keep open. The underlying subagent result-loss bug still exists on current main, but this PR should not merge as-is because it misses the current run-manager completion path, cannot extract normal result envelopes, and can still return a result-less lifecycle snapshot before the dedupe payload arrives.

## What This Changes

The PR branch threads `agent.wait` result payloads through gateway dedupe responses, uses that text as a subagent completion fallback when chat history is late, and adds gateway plus subagent regression tests.

## Best Possible Solution

Land a current-main implementation that defines `result` in the `agent.wait` contract end to end, extracts reply text from structured `{ payloads, meta }` result envelopes, and fills `frozenResultText` only as a fallback before completion delivery without adding provider-specific core behavior.

## Reproduction Assessment

Yes. The source path is reproducible by inspection: current main stores `payload.result`, then omits it from the dedupe snapshot, `agent.wait` response, `waitForAgentRun`, and the active subagent run-manager completion path.

## Solution Assessment

No. The PR is directionally useful, but it is not the best current fix because it targets an obsolete registry wait path, does not parse normal result envelopes, and leaves a live wait race that can still return without a result.

  - confidence: 0.85
