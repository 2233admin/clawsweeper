## Summary

Keep this issue open. Current main still rejects mixed DNS results if any resolved address is blocked, before IPv4-first pinning can choose a valid public address. The related implementation PRs in the provided context are closed and unmerged, and the shipped RFC2544 opt-in only covers the separate 198.18.0.0/15 fake-IP case.

## Best Possible Solution

Keep the issue open and implement a narrow SSRF DNS-result fix in `src/infra/net/ssrf.ts`: classify results, filter out blocked answers, throw only when no allowed answer remains, and build pinned lookups/dispatcher candidates only from the filtered allowed set. Add focused regression coverage in `src/infra/net/ssrf.pinning.test.ts` and, if useful, `src/agents/tools/web-fetch.ssrf.test.ts` for public IPv4 plus `::`, public IPv4 plus Teredo/reserved IPv6, all-blocked DNS still failing, IPv4-first ordering after filtering, and pinned lookup results excluding blocked addresses.

## Risks / Open Questions

- This is SSRF-sensitive code: the eventual fix should filter blocked DNS answers before constructing pinned lookups or dispatcher override candidates, while still failing when every resolved answer is blocked.
- The RFC2544 opt-in should not be treated as a complete fix because it only changes classification for 198.18.0.0/15 and does not address special-use IPv6 mixed with public IPv4.
