## Summary

Current main still matches the reported mixed-DNS failure mode. `web_fetch` uses the strict guarded fetch path by default; DNS results are rejected if any resolved address is blocked before IPv4-first pinning runs; the pinned address pool is still built from the original unfiltered DNS result set. The related implementation PRs in the provided context are closed and unmerged, and the shipped RFC2544 opt-in addresses a separate fake-IP IPv4 case rather than mixed public IPv4 plus blocked IPv6 answers.

## Best Possible Solution

Keep this issue open. Implement a narrow fix in `src/infra/net/ssrf.ts` that classifies DNS results, filters blocked answers, throws only when no allowed answer remains, and builds pinned lookups/dispatcher override candidates only from the filtered allowed set. Add focused tests in `src/infra/net/ssrf.pinning.test.ts` and, if useful, `src/agents/tools/web-fetch.ssrf.test.ts` for public IPv4 plus `::`, public IPv4 plus Teredo/reserved IPv6, all-blocked DNS still failing, IPv4-first ordering after filtering, and pinned lookup results excluding blocked addresses.

## Risks / Open Questions

- This is SSRF-sensitive code: the fix should allow mixed DNS only by filtering blocked answers out before constructing pinned lookups or dispatcher overrides, while still failing when every resolved answer is blocked.
- The existing RFC2544 opt-in should not be treated as a complete fix because it only changes classification for `198.18.0.0/15` under explicit configuration.
