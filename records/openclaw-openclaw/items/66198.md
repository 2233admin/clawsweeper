## Summary

Keep this PR open. The hardening goal is relevant, and current main still leaves Signal inbound text raw on the model-facing path, but the submitted patch is not merge-ready because it wraps only legacy envelope strings while `BodyForAgent` remains raw and preferred by reply assembly. The existing Greptile and Codex discussion correctly points to follow-up fixes around history metadata and deterministic wrappers.

## What This Changes

The PR imports `wrapExternalContent` and wraps Signal's legacy formatted inbound envelope body and pending group-history body in `extensions/signal/src/monitor/event-handler.ts`.

## Best Possible Solution

Rework the Signal hardening around the actual prompt contract: wrap or attach untrusted user text on the `BodyForAgent`/structured `UntrustedContext` path, preserve raw `CommandBody` and `BodyForCommands` for command parsing, keep generated metadata outside untrusted blocks, make pending-history wrappers stable, and add focused Signal coverage for DMs, groups, debounced messages, quote-only context, and pending group history.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Wrap the model-facing Signal body:** `extensions/signal/src/monitor/event-handler.ts:159-162`
  - body: Wrapping only the legacy `Body` passed to `formatInboundEnvelope` does not protect Signal replies, because this handler still finalizes the context with raw `BodyForAgent` and `RawBody`. Core prompt assembly prefers `BodyForAgent`, so inbound Signal messages still reach the model unwrapped.
  - confidence: 0.95
- **[P2] Avoid regenerating history wrappers during render:** `extensions/signal/src/monitor/event-handler.ts:179-183`
  - body: `wrapExternalContent` creates a fresh random marker ID on every call. Calling it inside `formatEntry` makes unchanged pending group history byte-different across turns, undermining deterministic prompt caching; wrap at ingest or use stable marker IDs instead.
  - confidence: 0.91
- **[P3] Keep generated history IDs outside the untrusted body:** `extensions/signal/src/monitor/event-handler.ts:180`
  - body: The proposed template includes the application-generated `[id:...]` suffix inside the untrusted wrapper with sender text. That metadata should be appended outside the external-content block so trusted routing metadata is not presented as user-controlled content.
  - confidence: 0.86
