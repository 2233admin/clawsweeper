## Summary

Keep open. Current main still stages the backup manifest under ambient `os.tmpdir()` while macOS service environments set `TMPDIR` under the OpenClaw state directory, and the backup plan recursively includes that state directory. The provided GitHub context also shows PR #75031 is open and linked as the active fix candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Report that `openclaw backup create` can archive two root `manifest.json` entries when macOS service `TMPDIR` points under `~/.openclaw/tmp`.

## Best Possible Solution

Land PR #75031 or an equivalent focused change so backup staging uses `resolvePreferredOpenClawTmpDir()` or another trusted temp root outside every included backup source, with a regression test proving exactly one root `manifest.json` when service `TMPDIR` is under the state directory.

## Reproduction Assessment

Yes. The static reproduction path is high-confidence: macOS service env sets `TMPDIR` inside state, backup creation writes `manifest.json` under `os.tmpdir()`, and tar archives both that explicit manifest and the recursive state source.

## Solution Assessment

Yes in direction. Replacing ambient `os.tmpdir()` with the existing trusted OpenClaw temp resolver, plus focused manifest-count regression coverage, is the narrow maintainable fix.
