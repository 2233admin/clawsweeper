## Summary

Keep open. Current main still stages `openclaw backup create` metadata under `os.tmpdir()`, while macOS service setup intentionally sets `TMPDIR` to the OpenClaw state temp directory; because the archive packs the explicit manifest path plus the state source tree, the staging manifest can be swept into the payload and remapped as a second root manifest. The reported `backup verify` impact is likely outdated for current main because the verifier now rejects duplicate root manifest entries, but the underlying temp-in-payload bug remains valid and narrow.

## What This Changes

The issue reports that `openclaw backup create` can include two root `manifest.json` entries when macOS LaunchAgent `TMPDIR` points inside the backed-up OpenClaw state directory.

## Best Possible Solution

Keep this open and land a focused backup fix that always stages the manifest outside any included source tree, preserving atomic archive publication and adding a regression for `TMPDIR` inside the state directory with exactly one root manifest and successful verification.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists: macOS service env resolves `TMPDIR` to the state `tmp` directory, `createBackupArchive()` stages `manifest.json` under `os.tmpdir()`, and the same state tree is later passed to `tar.c`.

## Solution Assessment

Yes, with a guard. The narrow maintainable fix is to use `resolvePreferredOpenClawTmpDir()` or an equivalent backup staging resolver, then assert the chosen staging root is outside `plan.included` before writing the manifest; broader backup exclude UX is not needed for this bug.
