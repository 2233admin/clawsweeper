## Summary

Keep this issue open. Current main has atomic archive publication and normal exception cleanup for `openclaw backup create`, but the reported timeout/process-termination failure mode is still not handled: the backup path still writes a large adjacent `.tmp` file while `tar.c(...)` is active, and I found no backup-owned SIGTERM/SIGINT/AbortSignal cleanup for that known temp path. Related #41830 covers ordinary partial-archive cleanup, and #42273 covers broader large-backup/stall policy; neither supersedes this timeout-leaked-temp-file report.

## Best Possible Solution

Keep this issue open and implement backup-owned graceful cleanup for timeout/termination: track the exact per-run temp archive path while `tar.c(...)` is active, register scoped SIGINT/SIGTERM or abort cleanup during that window, remove handlers in `finally`, preserve atomic publish/no-overwrite behavior, and add regression coverage for the signal/abort path. Leave broad large-backup exclusion/progress work to #42273 and live-file race handling to the existing backup PR threads.
