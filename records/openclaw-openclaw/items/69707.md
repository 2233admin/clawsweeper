## Summary

Keep this PR open. It targets a valid memory-lancedb compatibility bug that is still present on current main, but the current patch still needs one correctness fix before merge because the new validation accepts fractional dimensions even though the PR now performs local vector slicing.

## What This Changes

This PR changes memory-lancedb's OpenAI-compatible embedding path, config parser, manifest schema, and tests so configured dimensions are omitted from upstream embedding requests, returned vectors are truncated locally, and non-positive dimensions are rejected.

## Best Possible Solution

Land this after tightening embedding.dimensions to a positive finite integer in runtime config parsing, manifest JSON schema, and regression coverage, while keeping the direct /embeddings transport, local truncation, and short-vector error behavior.

## Reproduction Assessment

Yes. Current main has a source-level reproduction because configured dimensions are added to the outbound /embeddings body, and the current test locks that in; the PR body also provides a concrete vLLM-compatible provider reproduction. The new review finding is reproducible with a fractional dimensions value such as 2.5, which passes the new guard and is then floored by JS slicing/length handling.

## Solution Assessment

No, not quite yet. Omitting upstream dimensions and truncating normalized vectors locally is the right narrow direction for the reported provider compatibility bug, but the validation should reject fractional dimensions rather than accepting a value that will be silently rounded down by runtime operations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Reject fractional embedding dimensions:** `extensions/memory-lancedb/config.ts:183-186`
  - body: The new guard still accepts values like `dimensions: 2.5`. Since this PR stops relying on the upstream provider to validate dimensions and then uses `slice(0, this.dimensions)`, fractional values are silently floored during local truncation instead of failing as invalid config. Please require positive finite integers here and mirror that in the manifest schema/tests.
  - confidence: 0.9
