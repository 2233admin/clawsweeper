## Summary

Keep open. Current main and the shipped v2026.4.27 release still return raw plugin command handler results and then dereference them, so the reported crash is real; this PR is a narrow, plausible fix with focused regression coverage. It overlaps with #74838, so maintainers should choose one canonical implementation rather than close this as obsolete.

## What This Changes

This PR normalizes nullish plugin command handler results to an empty reply payload, adds defensive nullish guards in text and Telegram native command consumers, and adds regression tests for undefined handler results.

## Best Possible Solution

Land a single canonical fix that normalizes nullish handler returns at the plugin execution boundary while keeping the public SDK type strict, with regression coverage for text and Telegram/native command consumers, then reconcile the overlapping PR.

## Reproduction Assessment

Yes. A plugin command handler with no explicit return reproduces the source-level path: current main returns undefined from `executePluginCommand`, then text and Telegram command consumers treat it as a reply payload.

## Solution Assessment

Yes. Normalizing at the executor boundary is the narrowest maintainable fix because all plugin command surfaces consume that helper; the added call-site guards are redundant after the central fix but acceptable defense for mocks or alternate wrappers.
