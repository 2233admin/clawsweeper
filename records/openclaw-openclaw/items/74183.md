## Summary

Keep this open. Current main still lacks per-server bundle-MCP startup cooldown/backoff for rate-limited streamable HTTP or SSE initialization failures, and the overlapping retry PR was closed unmerged with maintainer/security concerns rather than landing a replacement.

## What This Changes

Request per-server exponential backoff, jitter, and attempt limits for bundle-MCP server initialization when remote MCP servers return rate-limit errors.

## Best Possible Solution

Add a small shared bundle-MCP startup cooldown layer in `src/agents/pi-bundle-mcp-runtime.ts`: classify transient rate-limit initialization failures for streamable HTTP/SSE servers, track attempts per server across session runtimes with capped jittered backoff and an hourly attempt cap, skip only cooling servers while preserving other MCP tools, and reset state after successful startup.

## Reproduction Assessment

Yes. A high-confidence reproduction can use a fake streamable-HTTP MCP server that returns 429 or `temporarily_unavailable` during initialize/listTools, then start two separate bundle-MCP sessions; current source shows each session will call `getCatalog()` and attempt startup without shared cooldown.

## Solution Assessment

Yes, the narrow runtime-level cooldown is the best direction. Placing it in materialization would be too session-local, while reviving the broader #66542 reconnect patch would bring unrelated parallel startup/reconnect/security changes that maintainers already rejected.

## Repair Work Prompt

Fix bundle-MCP startup retry storms for rate-limited remote MCP servers. Implement a narrow per-server startup cooldown/backoff layer in `src/agents/pi-bundle-mcp-runtime.ts`, not a broad reconnect rewrite. The observable bug is that streamable HTTP/SSE MCP initialization failures such as HTTP 429 or JSON bodies containing `temporarily_unavailable` / rate-limit text are retried by each new session with no shared cooldown. Expected boundary: classify transient rate-limit startup failures, record attempts per server across session runtimes, apply capped jittered exponential backoff with an hourly attempt cap, skip only servers currently cooling down while keeping other MCP servers available, reset state on successful initialization, and leave permanent auth failures such as `invalid_token` visible rather than cooling them indefinitely. Do not revive PR #66542 wholesale; do not add parallel startup, broad callTool reconnect classification, new third-party dependencies, SDK version changes, or unrelated config schema changes unless a maintainer asks. Coordinate with adjacent streamable HTTP reconnect PR #71545 if it remains open. Add focused tests in `src/agents/pi-bundle-mcp-runtime.test.ts`; update docs/changelog only if the behavior is user-facing, following repo attribution rules and never crediting @codex. Related refs: https://github.com/openclaw/openclaw/issues/74183, https://github.com/openclaw/openclaw/issues/68527, https://github.com/openclaw/openclaw/pull/66542, https://github.com/openclaw/openclaw/pull/71545.
