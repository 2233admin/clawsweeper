## Summary

Keep this issue open. Current main still starts configured bundled MCP servers during catalog discovery and, on startup failure, only logs, disposes the failed session, and continues. There is no per-server cooldown/backoff state, and the related LLM provider cooldown work does not cover this MCP server lifecycle path.

## Best Possible Solution

Add a small shared bundle-MCP startup cooldown layer in the MCP runtime path: classify transient rate-limit initialization failures for streamable HTTP/SSE servers, record per-server attempts across sessions with jittered capped backoff and an hourly attempt cap, skip only the cooling server while keeping other MCP servers available, and reset state on successful initialization. Cover it with targeted runtime tests and document/changelog the user-facing stability behavior if implemented.

## Clownfish Work Prompt

Implement per-server startup backoff for bundled MCP runtime initialization. Current issue: streamable HTTP MCP server startup failures such as HTTP 429 or `temporarily_unavailable` are retried by later sessions without shared cooldown, which can amplify rate limits. Scope the fix to bundle MCP lifecycle code, likely `src/agents/pi-bundle-mcp-runtime.ts` plus tests. Add a shared cooldown tracker keyed by MCP server identity and config fingerprint/server definition so cooldown survives different sessions but resets when config changes. Before `connectWithTimeout()`, skip a cooling server and log the cooldown. On transient rate-limit startup failures, record an attempt and compute jittered capped backoff, with behavior close to the request: initial delay around 5s, cap around 5 minutes, and a max-attempts-per-hour guard. Reset failure state on successful initialization. Keep other configured MCP servers materializing normally. Do not change MCP SDK versions, plugin IDs, or unrelated provider/LLM cooldown behavior. Do not blanket-cooldown permanent auth failures such as invalid_token; classify only clear transient rate-limit/temporary-unavailable shapes, honoring Retry-After if safely extractable. Add unit tests with injected time/jitter or helper functions covering: rate-limit failure records cooldown, later session skips during cooldown, cooldown expires and retries, success resets state, non-rate-limit failures do not enter rate-limit cooldown, and config fingerprint changes do not inherit stale cooldown. Add a user-facing changelog Fixes entry with appropriate reporter attribution per repo rules if the behavior change is shipped.
