## Summary

Keep open. Current main still lacks the requested explicit chat slash command for manually executing Task Flow or Lobster workflows. The repo has status/inspection surfaces (`/tasks`, `openclaw tasks flow list|show|cancel`) and an optional Lobster agent tool, but no `/lobster`, `/taskflow`, `/flow run`, or equivalent command that directly executes workflows from a channel. The plugin command API appears capable of hosting this, but channel-triggered workflow execution is security-sensitive and needs maintainer design.

## Best Possible Solution

Keep this issue open for maintainer design. The likely best path is a Lobster-owned plugin command using `api.registerCommand()` and the existing Lobster/Task Flow runtime, with owner or explicitly authorized execution, structured JSON parsing, safe path/cwd handling, clear approval/resume behavior, user-safe errors, docs, and focused tests. If maintainers decide this should not be bundled, it can move to a ClawHub plugin only if the current plugin API is sufficient for the desired execution and security model.

## Risks / Open Questions

- A chat slash command that runs Lobster or Task Flow workflows can trigger local in-process workflow execution from a channel, so owner enforcement, allowlists, audit output, and approval/resume semantics need explicit design.
- Free-form JSON arguments, workflow paths, pipeline strings, cwd values, approval ids, and resume tokens could become command-injection, path-boundary, or confused-deputy bugs if routed directly into the runner.
- Native command providers have different argument limits and menu semantics, so a JSON-bearing slash command may need provider-aware syntax or a text-only fallback.
