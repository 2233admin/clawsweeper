## Summary

Current main does not implement the requested explicit chat slash command for manually running Task Flow or Lobster workflows. The existing surfaces are `/tasks` for status, `openclaw tasks flow list/show/cancel` for CLI inspection and cancellation, and the optional `lobster` agent tool. The plugin command API appears capable of hosting this kind of feature, but chat-triggered workflow execution is security-sensitive enough that maintainer product/security review should precede an automated fix PR.

## Best Possible Solution

Keep this issue open for maintainer design. The likely best implementation is an owner-gated Lobster-owned plugin command, or a ClawHub plugin if maintainers do not want it bundled, using `api.registerCommand()` and the existing Task Flow/Lobster runtimes. Define the exact syntax, JSON argument handling, run/resume semantics, docs, and command authorization before adding code; avoid a core built-in unless a missing generic plugin API is identified.

## Risks / Open Questions

- A chat slash command that runs Lobster workflows may execute local shell-backed pipelines inside the Gateway process, so authorization, owner routing, audit output, and approval/resume behavior need explicit design.
- JSON argument parsing and workflow path/pipeline handling could reintroduce command-injection or path-boundary issues if implemented casually.
- Native command providers have different argument and payload limits; a text-only command, native command, or plugin-specific syntax choice needs product review.
