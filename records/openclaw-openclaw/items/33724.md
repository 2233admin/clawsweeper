## Summary

Current main still uses a hard-coded adjacent five-file `.bak` ring for OpenClaw config writes, with no typed config/schema/UI surface for backup directory, retention count, or trigger policy. The related implementation PRs were closed unmerged, so this remains a real product/config decision rather than cleanup-close material.

## Best Possible Solution

Keep this open for maintainer product decision. If accepted, implement a typed core backup policy with conservative defaults for directory, retention count, and trigger behavior; thread it through backup rotation, config writes, include mutations, recovery/audit paths, schema/help/UI/docs, and regression tests. If maintainers prefer the cron/archive workaround instead, close only after an explicit maintainer rejection and document the recommended replacement path.

## Risks / Open Questions

- Config backups may contain credentials or other sensitive config data, so configurable paths/retention must preserve 0o600 hardening, path validation, and safe cleanup behavior.
- Changing trigger behavior can weaken last-known-good recovery if users disable or delay backups without clear guardrails.
- A full implementation would touch config schema/types, generated help/schema artifacts, Control UI form surfaces, recovery/audit code, docs, and tests, so it needs maintainer product review rather than a drive-by cleanup close.
