## Summary

Keep open. Current main still does not classify the reported top-level string-code JSON shape as a billing error or as a structured API error payload, so `formatAssistantErrorText` can still fall through to returning the raw JSON. The supplied related item #48915 is a separate snake_case tool-call repair issue, not a duplicate of this billing/error-formatting path.

## Best Possible Solution

Add structured support for top-level string-code API error envelopes in the shared assistant error parser, then teach the billing classifier to recognize structured billing codes such as `insufficient_balance`, `insufficient_credits`, `payment_required`, and `quota_exceeded` without broad prose false positives. Cover `formatAssistantErrorText`, `sanitizeUserFacingText` in error context, failover classification, and raw formatter behavior with targeted regressions. Treat URL surfacing as optional and conservative: preserve a clear billing/manage URL only when it is a string field from a recognized billing payload.

## Clownfish Work Prompt

Fix the assistant billing-error formatter for top-level string-code JSON error envelopes reported in https://github.com/openclaw/openclaw/issues/74079. Current main can return raw JSON for payloads like `{"error":"insufficient_balance","message":"Insufficient MBT balance. Top up or upgrade your subscription to continue.","upgradeUrl":"/settings/billing"}`.

Expected boundary:
- Update the shared assistant error parsing path so `parseApiErrorPayload` / `parseApiErrorInfo` accepts provider envelopes where top-level `error` is a non-empty string code and top-level `message` is a non-empty string. Treat the string `error` as the error type/code and the top-level `message` as the user-facing provider message.
- Update billing classification conservatively so structured codes such as `insufficient_balance`, `insufficient_credits`, `payment_required`, and `quota_exceeded` classify as billing/rate limit only in appropriate error payload context. Avoid broad prose false positives; preserve existing safeguards for normal assistant text discussing billing.
- Ensure `formatAssistantErrorText` returns the existing friendly billing copy for the reported payload, including provider/model context when supplied.
- Ensure `sanitizeUserFacingText(..., { errorContext: true })`, `formatRawAssistantErrorForUi`, and failover classification behave consistently for the new payload shape.
- Treat URL surfacing (`upgradeUrl`, `billing_url`, `manage_url`) as optional; if implemented, only include safe string URLs from recognized billing payloads and do not broaden unrelated formatter behavior.
- Add focused regression tests. Do not change unrelated provider defaults, plugin boundaries, or broad fallback policy.
- Add a single-line user-facing changelog fix entry under the active version `### Fixes` with `Thanks @jarvis-ai-gregmoser`.
