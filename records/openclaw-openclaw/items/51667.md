## Summary

Keep open. Current main still routes inbound audio through the media-understanding/STT transcript path and only forwards native prompt media as images. The related open PR #48900 is a useful implementation candidate, but it has not landed in current main and the requested opt-in raw audio forwarding for WhatsApp/Telegram voice notes remains missing.

## Best Possible Solution

Keep this issue open and review or evolve PR #48900 into a shared implementation: add an opt-in native-audio config, audio-capable model/provider detection, typed raw audio content-part forwarding through the embedded runner and provider adapter layer, secure size-bounded staging/cleanup, STT fallback, and transcript/hash/duration/interpretation metadata so WhatsApp and Telegram voice notes can reach omni-modal models without losing auditability.

## Risks / Open Questions

- Closing now would strand a real, unimplemented user-facing feature: WhatsApp/Telegram voice notes still reach the primary model as STT text or placeholders, not raw audio content parts.
- Model catalog metadata can now represent `audio` input, which may make native audio look supported even though the embedded run pipeline lacks audio prompt blocks.
- A correct implementation needs bounded payload handling, secure staging/cleanup, provider content-block compatibility, STT fallback, and audit/continuity metadata such as hash, duration, transcript, or model interpretation.
