## Summary

Keep open. Current main still lacks opt-in native-audio forwarding for WhatsApp/Telegram voice notes: audio media understanding still produces transcripts, channel preflight paths rewrite audio placeholders to transcript text, and the embedded runner prompt path only forwards images. Related PR #48900 remains open/unmerged, so it is an implementation candidate to review rather than a reason to close this issue.

## Best Possible Solution

Keep this issue open and review or evolve PR #48900 into a shared implementation: add an opt-in native-audio config, audio-capable model/provider detection, typed raw audio content-part forwarding through the embedded runner and provider adapter layer, size-bounded staging/cleanup, STT fallback, and transcript/hash/duration/interpretation metadata so WhatsApp and Telegram voice notes can reach omni-modal models without losing auditability.
