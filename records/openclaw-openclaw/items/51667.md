## Summary

Keep open. Current main still treats inbound audio as transcription output and the embedded prompt path only forwards native image blocks, while the directly related implementation PR #48900 remains open and unmerged.

## What This Changes

Add an opt-in native-audio path so WhatsApp/Telegram voice notes can be sent as raw audio media parts to audio-capable omni-modal models instead of always going through STT transcription.

## Best Possible Solution

Keep this issue open and review or evolve PR #48900 into a shared, opt-in implementation that adds audio-capable model detection, typed raw audio content-part forwarding through the embedded runner/provider layer, STT fallback, and minimal hash/duration/transcript-or-interpretation metadata for continuity and auditability.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists: a WhatsApp or Telegram voice note enters the preflight `transcribeFirstAudio` path, the shared audio runner returns `audio.transcription`, and the embedded runner has no audio prompt option to pass downstream.

## Solution Assessment

Unclear for the open PR, but the requested direction is plausible only as a shared opt-in pipeline with provider compatibility checks and bounded raw-media handling. Current main is not the best solution for this issue because it still forces audio through transcription before the primary model sees it.
