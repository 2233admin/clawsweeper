## Summary

Keep open. Current main still classifies `channels.telegram.*` edits as a Telegram channel hot reload, and that hot reload is implemented as `stopChannel` followed by `startChannel`; the Telegram bot runtime then creates a fresh `groupHistories` map on start, so the channel-specific state-loss report still matches current behavior.

## What This Changes

The issue asks for Telegram config hot reloads to avoid destroying in-memory Telegram conversation context, especially group history state, when `openclaw.json` changes.

## Best Possible Solution

Keep the issue open until Telegram either preserves plugin-owned runtime state across config-driven channel restarts or hot-reloads safe Telegram config keys in place. The fix should stay plugin-owned unless a generic channel lifecycle seam is clearly needed by multiple channels, and it should include focused tests proving group history survives the reload path.

## Reproduction Assessment

Yes. A deterministic source-level reproduction exists: a `channels.telegram.historyLimit` change matches the Telegram reload prefix, produces `restartChannels: {telegram}`, runs the stop/start hot-reload path, and starts a Telegram bot with a new empty `groupHistories` map.

## Solution Assessment

No for the current behavior. Full channel stop/start is a reasonable fallback for token or transport changes, but it is too coarse for runtime-tunable settings when Telegram stores conversation context in process memory; a plugin-owned retained-state or narrower in-place reload path would be safer.
