## Summary

Keep this open. The report is concrete, current main still shows the invariant gaps in the named reset and compaction paths, and an open same-author PR explicitly closes this issue, so the issue should stay open until that PR is reviewed, merged, replaced, or closed by a maintainer.

## What This Changes

The issue asks for compaction and reset hook paths to preserve normalized `messageProvider` context and for successful compaction to refresh token/cost accounting even when the result is zero tokens.

## Best Possible Solution

Keep the issue open while maintainers review PR #69270 or replace it with an equivalent narrow fix. The desired end state is a shared provider-context resolver for compaction/reset hooks plus accounting logic that treats finite zero-token compaction results as fresh state, with regression coverage for channel-scoped keys, non-channel keys, casing, and zero-token outcomes.

## Reproduction Assessment

Yes. The source inspection gives a high-confidence static reproduction path: reset hooks and subscribe-time compaction hooks omit provider context, and one compaction accounting helper ignores `tokensAfter: 0`. The issue body and related #69286/#69287 reports also provide concrete user-level reproduction steps and observed safeguard-loop symptoms.

## Solution Assessment

Unclear for the exact open PR until its diff and CI are reviewed, but the coordinated direction is the maintainable one. A shared resolver plus zero-aware accounting is safer than patching each emitter independently.
