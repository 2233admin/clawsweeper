## Summary

Keep open. Current main still has the route-relative Control UI asset behavior reported here: the Vite build defaults to `./`, the header logo and assistant fallback helpers return relative paths when `basePath` is empty, and the Gateway 404s route-local static asset misses. An open PR already uses closing syntax for this issue, but it is unmerged and the latest discussion expands the bug from the header icon to deep-route static asset loading.

## What This Changes

The issue reports a Control UI regression after the 2026.3.24 update where the header logo, favicon, and related static assets can appear missing when served by the Gateway.

## Best Possible Solution

Keep this issue open until PR #56357 or an equivalent replacement lands. The end state should make default root-served Control UI assets absolute, preserve configured `gateway.controlUi.basePath`, fix empty-basePath logo/avatar fallbacks, avoid treating unknown routes as phantom base paths, and add focused regression coverage for hard-refresh deep routes.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: with default `base: "./"`, a hard refresh under a nested route resolves static assets relative to that route, and the Gateway returns 404 for route-local static asset misses.

## Solution Assessment

No, current main is not the best solution because it still preserves the failing route-relative defaults. The likely fix should cover the build-time Vite base default plus the runtime logo/avatar and base-path inference edges, not only the header image.
