## Summary

Keep open. Current main still rejects prefixless Anthropic setup tokens and enforces the legacy 80-character minimum, while this PR's proposed guard would also regress validation by accepting too-short legacy-prefixed tokens.

## What This Changes

This PR relaxes Anthropic setup-token validation to accept prefixless tokens at a 40-character minimum while preserving legacy-prefixed token support.

## Best Possible Solution

Confirm the accepted Anthropic setup-token formats, then update the shared validator in `src/plugins/provider-auth-token.ts` with a format-agnostic but unconditional minimum-length guard, plus focused unit coverage for valid and too-short legacy and prefixless inputs.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Reject too-short legacy setup tokens:** `src/commands/auth-token.ts:33-38`
  - body: The new guard only checks length when the token lacks `sk-ant-oat01-`, so truncated legacy values such as `sk-ant-oat01-x` would be accepted and stored instead of failing at paste/setup time. Keep relaxed prefix handling separate from an unconditional minimum-length check.
  - confidence: 0.94

## Security Review

Status: needs_attention

Summary: The patch weakens validation on an auth-token input path by letting malformed legacy-prefixed tokens pass.

Concerns:

- **[low] Malformed legacy tokens bypass validation:** `src/commands/auth-token.ts:33`
  - body: Because length validation only applies to prefixless tokens, a too-short token with the legacy prefix would pass upfront validation and be stored as a credential, making auth failures harder to diagnose and weakening the existing guardrail.
