## Summary

Keep open: current main still uses the strict shared Anthropic setup-token validator, while this PR is stale and would weaken the existing auth guard by letting too-short legacy-prefixed tokens pass. Prefixless setup-token support may still be worth considering, but it needs maintainer confirmation and a reworked patch across the active validator and Anthropic runtime path.

## What This Changes

The PR lowers the Anthropic setup-token minimum length in `src/commands/auth-token.ts` from 80 to 40 and accepts tokens without the `sk-ant-oat01-` prefix.

## Best Possible Solution

Confirm the accepted Anthropic setup-token formats first. If prefixless setup tokens are valid, update `src/plugins/provider-auth-token.ts` with a format-aware validator that keeps an unconditional minimum-length guard, adjust the Anthropic runtime token classifier as needed, keep `src/commands/auth-token.ts` as a compatibility re-export, and add focused command/provider/runtime tests.

## Reproduction Assessment

Yes. The PR diff is enough to reproduce the regression by inspection: `sk-ant-oat01-x` starts with the legacy prefix, so the new combined guard skips the length check and returns success where current main rejects it as too short.

## Solution Assessment

No. The proposed patch changes a stale implementation location, relaxes validation without preserving the legacy length guard, and does not address downstream Anthropic runtime handling for prefixless setup-token auth.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.92

Full review comments:

- **[P2] Reject too-short prefixed setup tokens:** `src/commands/auth-token.ts:33-38`
  - body: The new guard only checks length when the token lacks `sk-ant-oat01-`, so truncated legacy values such as `sk-ant-oat01-x` would be accepted and stored instead of failing at paste/setup time. Keep prefix relaxation separate from an unconditional minimum-length check.
  - confidence: 0.94
