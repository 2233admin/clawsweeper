## Summary

Keep open. Current main still has a credible stale HTTP-route registry path for workspace/deferred plugin routes, and open PR #73877 is already linked as the implementation candidate; the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that workspace plugins can register HTTP routes and show them in inspection, but gateway dispatch still returns 404 instead of invoking the registered handler.

## Best Possible Solution

Review, land, or replace PR #73877 with a narrow gateway/plugin-runtime fix that lets the dispatcher resolve the registry owning workspace/deferred HTTP routes, with regression coverage for the reported curl path and no broad plugin loader redesign.

## Reproduction Assessment

Yes. The issue provides concrete Docker workspace-plugin and curl steps, and current-main source inspection shows the same stale-registry failure mode when a later plugin registry owns HTTP routes after the gateway pinned the startup HTTP-route surface.

## Solution Assessment

Yes. The maintainable direction is a narrow route-registry fix in the gateway/plugin runtime surface with regression coverage, while preserving the channel registry protection; documentation or plugin workarounds would not address the dispatcher bug.
