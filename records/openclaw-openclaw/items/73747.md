## Summary

Keep open. Current main still contains the pinned plugin HTTP-route registry path that can leave dynamically loaded workspace plugin routes invisible to the dispatcher, and an open unmerged PR (#73877) already targets this issue with closing syntax, so the maintainer action is to review or replace that PR rather than close the issue early.

## What This Changes

The issue reports that workspace plugin `api.registerHttpRoute(...)` registrations appear successful, but gateway HTTP dispatch returns 404 and never invokes the handler.

## Best Possible Solution

Keep this issue open until maintainers review and land #73877 or an equivalent narrow fix that makes dynamically loaded workspace plugin HTTP routes visible to the gateway dispatcher while preserving intentional startup/channel registry behavior.

## Reproduction Assessment

Yes. The issue gives a concrete Docker/workspace-plugin curl reproduction, and current-main source inspection shows a matching failure path where runtime state pins the HTTP-route surface before deferred plugin reloads and pinned surfaces ignore later registry updates.

## Solution Assessment

Yes at the product boundary: the narrow maintainable solution belongs in the gateway/plugin runtime registry path so dispatch resolves the registry that contains workspace plugin routes. The current open PR is the right next object for maintainer review, but this issue should not close before that fix lands or is replaced.
