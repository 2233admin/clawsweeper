## Summary

Keep open. Current main has the general plugin HTTP route dispatcher, route-registry pinning, and unit coverage for stale registry cases, but this is a fresh Docker/workspace-plugin report against v2026.4.26 where `plugins inspect` sees a route while real HTTP dispatch still returns 404. I did not find a current-main fix after v2026.4.26 that specifically proves that reported startup/runtime path is resolved.

## Best Possible Solution

Reproduce this against current main in Docker with a minimal trusted workspace plugin under the default agent workspace that registers `/plugins/webhooks/gmail-push`; verify both `plugins inspect` route count and an actual HTTP POST handler invocation. If it reproduces, fix the Gateway startup/runtime route-registry wiring so workspace `api.registerHttpRoute(...)` registrations are visible to the dispatcher, then add an integration regression for that exact path. If it does not reproduce, document the operational distinction between inspect-time registration and live Gateway runtime state, including the restart/config checks for container deployments.
