## Summary

Keep open. Current main still pins the HTTP-route registry at gateway runtime-state creation, so a later workspace/deferred plugin registry can contain the registered route while the dispatcher continues resolving against the stale startup route registry. The linked open PR #73877 already targets this exact bug with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Workspace plugin HTTP routes registered with `api.registerHttpRoute(...)` should dispatch through the gateway instead of returning 404 after successful plugin load and route registration.

## Best Possible Solution

Keep this open until maintainers review and land #73877 or an equivalent narrow fix that makes the plugin HTTP dispatcher resolve the registry containing workspace/deferred HTTP routes, with regression coverage for that dispatch path and no regression to channel registry protection.

## Reproduction Assessment

Yes. The issue supplies a concrete workspace-plugin Docker/curl path, and current-main source inspection shows a matching stale-registry path when routes are registered into a later plugin registry after the startup HTTP-route registry has been pinned.

## Solution Assessment

Yes. The best fix direction is in the gateway/plugin runtime registry surface, not a docs workaround, because the documented `api.registerHttpRoute(...)` contract should work for loaded workspace plugins.
