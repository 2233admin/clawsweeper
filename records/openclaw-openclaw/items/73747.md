## Summary

Keep open. Current main still has a source-backed failure path where the gateway pins the plugin HTTP-route registry before deferred workspace plugin reloads, and pinned surfaces ignore later active-registry updates. An open unmerged PR (#73877) explicitly targets this issue with closing syntax, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that workspace plugin `api.registerHttpRoute(...)` endpoints dispatch to their handlers instead of returning 404 after successful registration.

## Best Possible Solution

Review and land #73877 or a narrow equivalent fix that makes dynamically loaded workspace plugin HTTP routes visible to the gateway dispatcher while preserving intentional channel-registry pinning and existing dynamic lifecycle route behavior.

## Reproduction Assessment

Yes. The issue gives a high-confidence Docker/workspace-plugin curl reproduction, and current-main source inspection shows the same state path: a pinned HTTP-route surface can ignore later active registry updates before dispatch.

## Solution Assessment

Yes for the issue direction. The narrow maintainable fix belongs in the gateway/plugin runtime registry path so route dispatch resolves the registry containing workspace plugin routes; the open PR is the right object for maintainer review before closing this issue.
