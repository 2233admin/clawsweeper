## Summary

Keep open. Current main still contains the startup HTTP-route registry pinning that can leave the dispatcher resolving an empty or stale registry after later workspace-plugin registration, and the issue is already paired with open PR #73877. Because that PR is open and explicitly targets this bug, this issue should stay open until the fix is reviewed and merged.

## Best Possible Solution

Keep this issue open and review/validate PR #73877 or an equivalent targeted fix. The likely code direction is to stop pinning the HTTP route registry at Gateway startup while preserving channel registry pinning, then add an integration regression that loads or simulates a dynamically registered workspace plugin route and verifies an actual POST dispatches to the handler instead of falling through to 404. Close only after the paired PR merges or another fix lands.
