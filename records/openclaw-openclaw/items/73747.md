## What This Changes

The issue reports that workspace plugin `api.registerHttpRoute(...)` routes appear registered but gateway HTTP dispatch returns 404 without invoking the handler.

## Best Possible Solution

Keep this issue open until PR #73877 or an equivalent narrow fix lands, with regression coverage that exercises the production failure path: workspace plugin routes registered after runtime state creation must be visible to HTTP dispatch without weakening channel registry pinning or route auth behavior.

## Reproduction Assessment

Yes. The issue provides a concrete Docker/workspace-plugin curl reproduction, and current-main source inspection shows the same failure shape: startup route pinning, pinned-surface sync bailout, deferred plugin reload, and dispatch through the pinned route registry.

## Solution Assessment

Unclear for the exact PR implementation until maintainer review resolves the test-coverage concern, but the product direction is right: fix the gateway/plugin route-registry path so dynamically loaded workspace plugin `api.registerHttpRoute(...)` routes are visible to dispatch.
