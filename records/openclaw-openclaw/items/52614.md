## Summary

Keep open. Current main still lacks any initial-replay or preexisting-message suppression in embedded subscribe, so the PR addresses a plausible narrow problem, but the proposed patch is not merge-ready against current main because its new eligibility helper would drop the existing transcript-only assistant filter for `delivery-mirror` and `gateway-injected` messages.

## What This Changes

The PR snapshots preexisting embedded session messages, adds an initial replay flag to suppress replayed embedded subscribe events, and adds a regression test that only emits fresh assistant replies after attaching to a replaying session.

## Best Possible Solution

Land or replace a rebased narrow embedded subscribe fix that suppresses historical replay/preexisting assistant outward delivery, preserves current `delivery-mirror`/`gateway-injected` and commentary filters, and keeps focused regression coverage for both historical replay and transcript-only assistant suppression.

## Reproduction Assessment

Yes for the PR’s modeled failure: a session whose `subscribe` synchronously replays historical `agent_start`, assistant `message_start`/`message_end`, and `agent_end` events into current main will treat the historical assistant final like a fresh outward reply because no replay/preexisting guard exists. Production impact still depends on confirming the integration that emits those replay events, since the pinned `AgentSession.subscribe` implementation does not replay by itself.

## Solution Assessment

No, not as-is against current main. The replay/preexisting guard is a narrow direction, but the helper must preserve `isTranscriptOnlyOpenClawAssistantMessage` and the maintainer should confirm the replay contract before merge.
