## Summary

Keep this open. Current main still misses the reported Feishu username case: live user lookup only checks `open_id` and a direct lowercased display-name substring, so `jaydenli` does not match `Jayden Li`, and the shared resolver then returns the reported `Unknown target` error. The earlier direct fix in #62510 was closed unmerged by queue policy and is not present on current main.

## What This Changes

The issue requests Feishu outbound message delivery to handle a human username such as `jaydenli`, or at least return clearer target-format guidance instead of failing with `Unknown target`.

## Best Possible Solution

Implement the fix inside the Feishu plugin directory lookup: normalize user-facing query fields consistently, match compact usernames against display names with spaces plus safe email/nickname fields, preserve exact ID handling, and add focused tests for successful, missing, and ambiguous matches.

## Reproduction Assessment

Yes. A source-level reproduction is clear: a Feishu send target of `jaydenli` reaches directory lookup, but current matching does not make `jaydenli` equal or contained in `jayden li`, so the resolver falls through to the `Unknown target` error path.

## Solution Assessment

Yes, the narrowest maintainable fix is Feishu-owned directory matching and tests, not a core resolver special case. The prior #62510 direction is useful, but a replacement should omit unrelated dependency/script changes and avoid overly broad reverse-substring matching.
