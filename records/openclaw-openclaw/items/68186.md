## Summary

Keep this PR open. Current main still has the reported macOS screen.snapshot bugs, and this PR is the active implementation candidate for the linked open bug, but the latest head still has a concrete correctness issue: the raw-byte precheck can reject near-limit snapshots before the more precise projected-frame guard gets a chance to measure whether the serialized response actually fits.

## What This Changes

The PR hardens macOS screen.snapshot handling by rejecting malformed params, sanitizing and classifying snapshot errors, adding raw/projected payload-size bounds with nodeId plumbing through the bridge request, adding focused Swift runtime tests, and updating the changelog.

## Best Possible Solution

Keep the narrow hardening approach, but adjust or remove the raw precheck so only snapshots that cannot fit the real serialized node.invoke.result frame are rejected; then validate the macOS runtime tests and shared OpenClawKit bridge/session tests before maintainer review and merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.8

Full review comments:

- **[P2] Avoid rejecting near-limit snapshots before projection:** `apps/macos/Sources/OpenClaw/NodeMode/MacNodeRuntime.swift:25-26`
  - body: The raw precheck subtracts the 1 KiB envelope reserve before applying base64 math, so payloads above 19,660,032 bytes return `UNAVAILABLE` before `projectedOuterFrameBytes(...)` can measure the actual frame. A low-escape payload with short ids can be slightly above this raw cutoff and still fit under the 25 MiB WebSocket limit, so this introduces false oversized-snapshot failures. Make the precheck only reject raw sizes that are impossible to fit, or rely on the projected frame check for the near-limit case.
  - confidence: 0.82
