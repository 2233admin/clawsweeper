## Summary

Keep open. Current main partially addresses the reader-side mismatch by loading date-prefixed slugged memory files, but it still writes per-session `YYYY-MM-DD-slug.md` files and still calls the LLM slug generator by default on `/new` and `/reset`. This PR remains a relevant implementation candidate, but it is not safe to merge as submitted because the posted review comments identify a real concurrent header race, a timeout parsing bug, and a removed workspace-context regression test.

## Best Possible Solution

Keep this PR open as useful but require a refreshed patch against current main: preserve the existing slugged-file reader compatibility, decide whether the writer should truly switch to canonical `memory/YYYY-MM-DD.md`, make `/new` and `/reset` avoid LLM work by default with matching docs/config coverage, implement daily appends without the header TOCTOU race, and retain the workspaceDir/display-session-key regression test. After that, run targeted session-memory and fs-safe tests plus the changed gate appropriate for hook/infra changes.

## Risks / Open Questions

- The PR as submitted can corrupt the intended daily Markdown shape under concurrent first writes because header detection and content append happen in separate opens.
- The PR's timeout fallback treats `0` as absent; maintainers should define whether zero is invalid or meaningful and use explicit positive finite/nullish handling.
- The current diff removes workspaceDir/display-session-key regression coverage that still protects a real current code path.
- Changing `appendFileWithinRoot` to return `SafeWritableOpenResult` expands a security-sensitive helper contract; even without supply-chain red flags, the final design should avoid leaking unnecessary file-handle state and keep symlink/race tests intact.
