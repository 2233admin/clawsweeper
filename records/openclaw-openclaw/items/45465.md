## Summary

Keep this PR open. Current main still does not implement cron lifecycle hooks: there is no `cron.hooks` config surface, no per-job `hooks` field, no Gateway protocol schema support, and scheduled, catch-up, and manual cron paths still call core execution directly. The PR remains distinct unmerged work, but it also introduces a security-sensitive dynamic script execution surface with unresolved hardening questions, so the right outcome is maintainer review or revision rather than cleanup closure.

## Best Possible Solution

Keep the PR open as an implementation candidate. Maintainers should first decide whether cron lifecycle hooks belong as a core cron API or as a new plugin/API extension; if the feature proceeds, the PR should harden per-job path/type validation, settle the hook import caching policy, and update the cron config schema, Gateway protocol, docs, and tests together.

## Risks / Open Questions

- The PR adds config/job-driven dynamic script execution inside Gateway cron paths, which is a security-sensitive operator capability and needs explicit maintainer product/security review before merge.
