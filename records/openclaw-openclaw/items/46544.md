## Summary

Current main still does not give `openclaw message send --media` an agent-scoped CLI selection path. The message CLI exposes `--account` but no `--agent`, `messageCommand` always passes the configured default agent into `runMessageAction`, and the agent-scoped media-root helper only adds the workspace for the agent id it is given. That leaves the reported multi-account/custom-workspace case unresolved, especially under workspace-only or restricted filesystem policy.

## Best Possible Solution

Keep this issue open and decide the product/security shape explicitly. The narrowest likely fix is to add an `--agent` selector to `openclaw message`, thread it through `messageCommand` into `runMessageAction`, and optionally infer the agent from a unique channel/account binding when `--account` is provided. Regression coverage should prove that a non-default agent workspace is included in `mediaLocalRoots` without adding unrelated workspaces.
