## Summary

Keep open. Current main still parses loaded `.env` files with `dotenv.parse` and assigns accepted values directly to `process.env`, so aliases such as `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remain literal. The related implementation PR #38254 is closed unmerged, but its review comments are useful prior art for expansion order, escaped-dollar handling, identifier matching, and quiet/error behavior.

## What This Changes

Expand references such as `${VAR}` and `$VAR` in dotenv-loaded `.env` values while preserving shell/env precedence.

## Best Possible Solution

Keep this issue open and implement dotenv-level expansion in `src/infra/dotenv.ts` after key normalization and workspace/global filtering but before assignment to `process.env`. Support `${VAR}` and `$VAR` using process env plus earlier accepted dotenv entries, preserve existing precedence/conflict behavior, define missing/cyclic/escaped-dollar semantics, and decide whether `src/config/state-dir-dotenv.ts` should share the same semantics for service env collection. Add focused regression coverage and update `docs/help/environment.md` plus `CHANGELOG.md` if the behavior change ships.
