## Summary

Keep open. Current main still loads dotenv values literally: `readDotEnvFile` uses `dotenv.parse` and the runtime loaders assign parsed values directly to `process.env`, so `${VAR}` and `$VAR` aliases in `.env` files are not expanded. The separate config substitution path only handles `${VAR}` inside config values and explicitly leaves bare `$VAR` unchanged. Closed PR #38254 is useful prior art, but it was closed unmerged and the requested behavior is still missing on main.

## Best Possible Solution

Keep the issue open and implement dotenv-level expansion in `src/infra/dotenv.ts` before assigning accepted entries to `process.env`. Support both `${VAR}` and `$VAR` using process env plus earlier loaded dotenv entries, preserve the current precedence and workspace blocklist behavior, and add focused regression tests in `src/infra/dotenv.test.ts`. Reuse closed PR #38254 as prior art, but rework it against the current loader and security constraints.

## Risks / Open Questions

- A fix must preserve the current precedence order: process env first, then workspace `.env`, then global/state-dir `.env`, then `gateway.env` fallback conflict handling.
- Workspace `.env` expansion must not bypass the existing blocklists for runtime-control, credential, endpoint, and trust-root variables.
- Expansion semantics still need explicit tests for missing references, chained references, same-file ordering, cross-file fallback ordering, escapes, and cycles.
