## Summary

Keep this issue open. Current main still parses dotenv files with `dotenv.parse` and assigns accepted values directly to `process.env`, so `.env` aliases like `OPENAI_TRANSCRIPTION_API_KEY=${OPENAI_API_KEY}` remain literal. The related PR #38254 attempted the right area but is closed unmerged, and its discussion is useful prior art for the future fix.

## Best Possible Solution

Keep this issue open and implement dotenv-level expansion in `src/infra/dotenv.ts` after key normalization and workspace/global filtering but before assignment to `process.env`. Support `${VAR}` and `$VAR` using process env plus earlier accepted dotenv entries, preserve existing precedence/conflict behavior, define missing/cyclic/escaped-dollar semantics, and decide whether the state-dir service env reader in `src/config/state-dir-dotenv.ts` should share the same semantics. Add focused regression coverage and update `docs/help/environment.md` plus `CHANGELOG.md` if the behavior change ships.
