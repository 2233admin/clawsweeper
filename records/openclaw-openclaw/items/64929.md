## Summary

Keep open. The report is recent, concrete, and corroborated, and current main still leaves local managed loopback browser profiles on the raw CDP/json tab-management path with repeated tab discovery before route actions. Persistent Playwright connection caching exists for page-level work after a target is selected, but current source and tests do not show that the reported local managed Brave/OpenClaw per-action overhead has been fixed.

## Best Possible Solution

Keep this issue open as a focused browser plugin performance bug. Add lightweight timing instrumentation or a loopback benchmark for local managed browser actions, then reduce repeated ensureTabAvailable/listTabs/target-resolution overhead by safely reusing a persistent CDP/Playwright connection or cached target metadata for local managed profiles while preserving existing security and profile-mode contracts.
