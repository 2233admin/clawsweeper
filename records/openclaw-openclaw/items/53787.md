## Summary

Keep this PR open. Current main still has the Feishu receive-id fallback behavior, and the PR is a plausible Feishu-owned fix for the linked regression, but the proposed patch currently leaks a nullable receiveIdType through resolveFeishuSendTarget and should not merge as-is.

## What This Changes

The PR changes Feishu receive-id type resolution to return null for invalid target formats and adds guards in send-target resolution plus streaming-card startup before Feishu/Lark API calls.

## Best Possible Solution

Land a Feishu-owned fix that rejects truly invalid send targets before API calls, preserves valid bare direct IDs, exposes a non-null FeishuIdType from the guarded send-target helper, handles nullable resolver callers explicitly, and adds focused regression coverage for invalid targets plus existing group/open_id/user_id routes.

## Reproduction Assessment

Yes for the PR defect: the diff makes resolveReceiveIdType nullable while resolveFeishuSendTarget still exposes ReturnType<typeof resolveReceiveIdType>, and downstream direct send/media paths require a non-null ID type. The linked runtime bug has reporter steps in #53376, but this read-only review did not run live Feishu API validation.

## Solution Assessment

No. The PR is aimed at the right Feishu plugin boundary, but the helper should narrow the guarded result to FeishuIdType or NonNullable<ReturnType<...>> before callers consume it.
