## Summary

Keep open. Current main still has the Feishu receive-id fallback that the PR is trying to fix, and the PR remains a relevant implementation candidate for still-open #53376, but the current patch leaks the new nullable resolver type into send paths that require a non-null Feishu ID type, so it should not merge as-is.

## What This Changes

The PR changes Feishu receive-id type resolution to return null for invalid targets and adds guards in send-target resolution plus streaming reply startup before calling the Feishu/Lark API.

## Best Possible Solution

Land a Feishu-owned fix that rejects invalid send targets before API calls, preserves valid bare direct IDs, exposes a non-null `FeishuIdType` from the guarded send-target helper, handles every nullable resolver caller, and adds focused regression coverage for invalid targets plus existing group/open_id/user_id routes.

## Review Findings
