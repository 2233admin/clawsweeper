## Summary

Keep open. Current main still does not expose reply-to metadata to the `inbound_claim` hook, and this PR is a narrow additive propagation of existing optional context fields through the existing hook metadata object. I found no blocking correctness or security issue; the remaining action is normal maintainer review, with a small regression test as the main landing improvement.

## What This Changes

This PR adds optional `replyToId` and `replyToBody` fields to `CanonicalInboundMessageHookContext`, maps them from `FinalizedMsgContext`, and emits them in `inbound_claim` event metadata.

## Best Possible Solution

Land the additive hook mapper change after maintainer review, preferably with a small `src/hooks/message-hook-mappers.test.ts` assertion that reply-to fields flow from `FinalizedMsgContext` into `inbound_claim` metadata.
