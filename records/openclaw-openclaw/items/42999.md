Current main still has the static 16k/32k context-window guard behavior requested for replacement, and the linked implementation candidate #72958 is still open and unmerged with explicit closing syntax for this issue. Repository policy says to keep the issue open while that PR is reviewed rather than closing ahead of the implementation.

## Best Possible Solution

Keep this issue open while #72958 is reviewed. The concrete path is to land a focused core change that derives guard thresholds from the effective context window with safe floors, stops passing fixed 16k/32k overrides from the embedded runner, preserves `agents.defaults.contextTokens` cap semantics, and updates the guard tests plus public local-model docs. If #72958 is merged, close this as implemented by the merge; if it is rejected, leave this issue open for a revised implementation or an explicit maintainer product decision.
