## Summary

Keep this PR open. Current main still does not implement the PR's central startup deadline: `runGatewayLoop` directly awaits `params.start({ startupStartedAt })`, and there is no `STARTUP_TIMEOUT_MS`, startup-timeout error, or matching regression test in the checked-out source. The PR is a focused CLI/runtime change rather than a supply-chain-sensitive diff, but it still needs review work because the latest bot review raised a remaining P1 around late cleanup close failures after timeout.

## Best Possible Solution

Keep the PR open for normal review. The best product direction is a robust startup deadline for the gateway loop or the specific startup network call, with regression coverage for first-start supervisor exit and late-start cleanup. Before landing, resolve the remaining P1 cleanup concern, ideally by adding cancellation via an `AbortSignal` or by making late-resolution cleanup observable and safe rather than silently swallowing a failed close.
