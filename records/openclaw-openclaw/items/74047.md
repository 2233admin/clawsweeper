## Summary

Keep open. Current main has useful guardrails for gateway service entrypoint drift, but it does not resolve the reported split user/system install plus newer-service `/v1/chat/completions` timeout. The related open PRs cover adjacent diagnostics and legacy unit resolution, but no merged current-main change proves the reliability issue or supported migration policy is fixed.

## What This Changes

The issue asks OpenClaw to clarify or fix split user/system gateway installs after upgrade, safe service-path migration, and GPT-5.5 `/v1/chat/completions` reliability under the live gateway service.

## Best Possible Solution

The desired end state is a clear Linux gateway upgrade path where status/doctor exposes the active service unit, scope, package root, real entrypoint, and version; update either refreshes the right supervisor safely or gives exact system-scope instructions; and a service-env `/v1/chat/completions` GPT-5.5 smoke either succeeds or fails with actionable diagnostics instead of a timeout.

## Reproduction Assessment

Unclear: the issue gives concrete machine state and rollback proof, but not the gateway logs, exact HTTP request, service environment, or minimal config needed to reproduce the newer-install timeout. Current main can be inspected for the service-path drift behavior, but not for the reported timeout without live service evidence.

## Solution Assessment

Unclear: current main's conservative doctor/update behavior is not a complete solution, and unconditional auto-tracking of a user install from a running systemd service is risky. A safer fix is diagnostics plus health-checked migration or explicit system-scope restart guidance, followed by endpoint/runtime work only if logs place the timeout there.
