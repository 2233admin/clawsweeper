## Summary

Keep open. Current main has useful partial guardrails for stale gateway service entrypoints, package-update restarts, duplicate global installs, and OpenAI Codex OAuth transport, but it still does not settle the supported split user/system install policy or prove that repointing the managed Gateway to the newer install no longer causes /v1/chat/completions timeouts.

## What This Changes

The issue asks OpenClaw to clarify and harden upgrades where the CLI/user install and active Gateway service install path diverge, especially when promoting openai-codex/gpt-5.5 through /v1/chat/completions.

## Best Possible Solution

Keep this issue as the canonical tracker until OpenClaw documents the supported split user/system install migration path, surfaces active CLI versus service entrypoint/package root/version/scope in status or doctor output, and has live or fixture-backed proof that a managed Linux Gateway can move to the newer install and reliably serve /v1/chat/completions with the intended Codex model.

## Reproduction Assessment

No. The issue has credible host-state and rollback evidence, but this read-only review found no current-main live service layout, Gateway logs, or test fixture that reproduces the newer-install timeout with high confidence.

## Solution Assessment

Unclear. The nearby diagnostics, updater, doctor, and Codex transport fixes are the right surfaces, but the narrowest maintainable fix depends on whether the remaining failure is split-install policy, systemd unit scope/ownership, or a Gateway runtime regression.
