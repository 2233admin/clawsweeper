## Summary

Keep open. Current main has partial guardrails for service entrypoint drift, package-update service refresh, duplicate-install warnings, and the OpenAI Codex OAuth transport, but it does not fully resolve the split user/system install policy or prove that repointing a managed Linux gateway to the newer install no longer causes `/v1/chat/completions` timeouts.

## What This Changes

The issue asks OpenClaw to clarify or fix split user/system install upgrades so the CLI and managed gateway service agree on the active install, warn or migrate stale service paths, and reliably serve `openai-codex/gpt-5.5` through `/v1/chat/completions` after promotion.

## Best Possible Solution

Keep this issue as the canonical tracker until OpenClaw has a documented split user/system install migration path, status/doctor output that shows the active CLI and service entrypoint/package root/version/scope, and a live or fixture-backed proof that a managed Linux gateway can move to the newer install and reliably serve `/v1/chat/completions` with the intended Codex model.

## Reproduction Assessment

No. The issue has credible host-state and rollback evidence, but there is no current-main fixture, log bundle, or live service layout in this review that reproduces the newer-install timeout with high confidence.

## Solution Assessment

Unclear. The current doctor/update/status and Codex transport changes are the right nearby surfaces, but the narrowest maintainable fix depends on whether the remaining failure is install-root policy, systemd scope/ownership, stale legacy unit resolution, or a gateway runtime regression.
