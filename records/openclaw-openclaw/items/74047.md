## Summary

Keep open. Current main has meaningful adjacent guardrails for service-layout visibility, system-scoped restart guidance, package update survival, and OpenAI Codex OAuth transport, but it still does not prove or document a safe split user/system install migration for the reported managed Gateway path, nor does it reproduce the newer-install `/v1/chat/completions` timeout.

## What This Changes

The issue asks OpenClaw to clarify or fix split user/system install upgrades where the Gateway service remains pinned to an older install path and newer Gateway entrypoints can make `/v1/chat/completions` unreliable.

## Best Possible Solution

Keep this issue as the canonical tracker until OpenClaw has a documented split user/system install migration policy, status/doctor output that makes the CLI install and live service install easy to reconcile, operator-safe handling for system-scoped units, and live or fixture-backed proof that a managed Linux Gateway can move to the newer install and reliably serve `/v1/chat/completions` with the intended Codex model.

## Reproduction Assessment

No. The report has credible host-state and rollback evidence, but current main has no high-confidence fixture or live run that reproduces the split user/system service repoint plus `/v1/chat/completions` timeout; the upgrade-survivor lane starts the updated package directly and checks Gateway status only.

## Solution Assessment

Unclear. The merged diagnostics, systemd restart guidance, package survivor lane, and Codex OAuth transport fixes are the right nearby surfaces, but the narrowest maintainable fix depends on whether the remaining failure is install-root policy, systemd ownership/environment, or provider transport under the Gateway HTTP endpoint.
