## Summary

Keep open. Current main still documents plaintext credential storage as supported and writes auth/config JSON with restrictive file modes but no transparent encryption layer; SecretRefs, audit, and one-way scrubbing are useful opt-in mitigations, not an implementation of encrypted credentials at rest. A related open PR (#39059) may partially address auth-store sealing, but it is not merged and does not close this issue.

## What This Changes

The issue requests encrypted-at-rest storage for sensitive OpenClaw credentials, using a keychain or master password with backward-compatible migration from existing plaintext storage.

## Best Possible Solution

Define and ship a security-owned credential-storage policy end to end: either add transparent encrypted-at-rest storage for OpenClaw-owned credential stores with migration, recovery, rotation, docs, and tests, or explicitly narrow the supported posture to SecretRef-backed external secret managers and update the threat model accordingly.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a runtime bug, but yes for current behavior: docs, types, and persistence paths on current main show plaintext credentials remain supported and persisted as JSON when SecretRefs are not used.

## Solution Assessment

Is this the best way to solve the issue? Unclear until maintainers choose the credential-storage contract. The request points at the right risk, but the implementation should be security-designed rather than a narrow bot patch because key management and compatibility are central.
