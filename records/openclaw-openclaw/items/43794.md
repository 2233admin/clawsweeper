## Summary

Keep open. Current main has meaningful SecretRef, permission, and audit mitigations, but it still does not implement transparent encryption at rest for config/auth-profile credentials using a system keychain or master password; plaintext storage remains documented and supported. Because this is security-sensitive credential storage and key-management design, it needs maintainer/security-owner review instead of automatic closure or a safe autonomous repair PR.

## What This Changes

Add encryption-at-rest for stored tokens and API keys in OpenClaw config/auth-profile storage, using system keychain or master-password decryption while preserving backward compatibility.

## Best Possible Solution

Define and ship a security-owned credential-storage policy that either adds real encrypted-at-rest storage for persisted OpenClaw credentials or explicitly narrows the product commitment to SecretRef-backed external secret managers, with migration, audit, docs, and token-rotation semantics aligned.
