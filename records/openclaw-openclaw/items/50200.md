## Summary

Keep this PR open. Current main still does not implement the intended inbound Slack files.info fallback for DM file attachments: resolver code only uses event-provided private URLs, and the message/thread preparation paths do not pass a Slack WebClient into media resolution. The related action-only download path does use files.info, but that does not cover automatic inbound message attachment handling. Related PR #62927 addresses a different Slack attachment-drop path and logging, so it does not supersede this work.

## Best Possible Solution

Keep this PR open for maintainer review or a corrected successor. The best path is to thread the Slack WebClient through direct message, forwarded attachment, and thread-starter media resolution; call files.info when event file URLs are missing; consider retrying with fresh metadata after stale URL failures; preserve the current URL/auth/SSRF protections; and add regression tests for the affected inbound paths.

## Risks / Open Questions

- The PR as described fixes the missing-URL case but may still leave stale or expired event URLs unresolved unless the inbound resolver retries through files.info after a failed download, as noted in review discussion.
- Any implementation should keep fresh Slack file URLs behind assertSlackFileUrl, authenticated same-origin redirect handling, and the existing Slack media SSRF allowlist.
- files.info failure behavior should be observable enough for debugging; the existing review comments about verbose logging and failure-path assertions are reasonable follow-up polish.
