## Summary

Keep this issue open. Current main shows the normal transcript path is UTF-8-oriented and QQBot now decodes websocket payloads as UTF-8, but I did not find a specific fix, regression test, release note, or live Windows/QQBot proof that resolves the reported GBK/UTF-8 transcript corruption. The related GBK work in the supplied context is about Windows process output decoding, not QQBot transcript JSONL persistence.

## Best Possible Solution

Keep open and add a narrow regression/hardening fix: exercise a QQBot Chinese message through gateway decode, event dispatch, inbound context, transcript append, and memory/session indexing, then assert the JSONL bytes decode as valid UTF-8 and preserve the original text. If the failure reproduces before persistence, fix it in the QQBot gateway/content pipeline; if it reproduces at persistence, wrap or update SessionManager usage so transcript appends/writes pass explicit UTF-8 and add coverage for Windows-safe behavior.
