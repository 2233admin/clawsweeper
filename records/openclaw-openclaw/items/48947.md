## Summary

Keep this issue open. Current main has shipped adjacent fixes for important subcases, including #72587 null/missing empty tool arguments and #34574 enabled loop-detection outcome comparison, plus live Control UI tool-event plumbing. Those do not fully resolve this issue's default web UI report: tool-loop detection is still disabled by default, there is no documented `tools.maxToolCalls` knob, and the broader complaints about runaway tool calls, stale UI refresh, silent terminal states, interruptions, and accidental config edits remain a plausible product/runtime bug rather than a closeable duplicate.

## Best Possible Solution

Keep the issue open and route it to the agent runtime plus Control UI/Gateway owners. The best fix is likely a default-safe runaway guard or hard per-turn tool-call circuit breaker with visible progress/failure summaries, plus an end-to-end web UI regression that proves live tool events, terminal states, abort/steer, and final history refresh work during repeated tool calls. The docs should also state the real supported knobs instead of implying an unavailable `tools.maxToolCalls`, and maintainers should decide whether agent-initiated config writes need additional safeguards or clearer approval UX.
