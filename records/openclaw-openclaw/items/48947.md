## Summary

Keep this issue open. Current main has several adjacent fixes for tool-loop detection, null empty tool arguments, unknown-tool loops, and live Control UI tool-event plumbing, but it still does not fully address the reported default web UI behavior: loop detection remains disabled by default, there is no documented `tools.maxToolCalls` knob, and the broader complaints about runaway tool-only turns, stale UI refresh, silent terminal states, interruptions, and accidental config edits remain plausible product/runtime work.

## Best Possible Solution

Keep this open and route it to agent runtime plus Control UI/Gateway maintainers. The best path is a default-safe runaway guard or hard per-turn tool-call circuit breaker with visible progress/failure summaries, an end-to-end web UI regression covering live tool events, terminal states, abort/steer/interruption, and final history refresh during repeated tool calls, plus docs that state the real supported knobs and maintainer judgment on safeguards for agent-initiated config writes.
