## Summary

Keep #59405 open. Current main has several fixes adjacent to the report, including Telegram direct-session policy-key handling, Telegram tool-result delivery plumbing, and OpenAI Responses tool-result replay tests, but I did not find a high-confidence implementation or regression test that proves the exact reported Telegram + OpenAI/Codex Responses + exec/grep output-to-agent-context failure is fixed. The thread follow-up reports a workaround/current-release success after switching to `openai-codex/gpt-5.4` and using `/model`, which is useful evidence, but it does not prove the original `openai/gpt-5.1-codex` Telegram parity bug is resolved on main.

## Best Possible Solution

Keep the issue open until there is targeted proof. The best maintainer path is to add or run a focused regression around a Telegram direct-session context using an OpenAI Responses or OpenAI Codex Responses model, with a stubbed exec/grep tool result, and assert that the follow-up model request receives the actual tool output as a `function_call_output` or equivalent provider input. If the remaining problem is only stale model/session selection, document the `/model` requirement and close with that evidence after confirming the original repro no longer fails on a current supported model.
