## Summary

Current main still does not implement opt-in custom safe headers for fetch-guard cross-origin redirects. The redirect header allowlist is fixed, GuardedFetchOptions has no additional safe-header field, provider/media redirect paths still use the fixed helper, and regression tests still expect x-api-key to be stripped. The related #57588 is useful security context, but it tracks bearer-auth fallback defaults rather than this redirect allowlist feature.

## Best Possible Solution

Keep this issue open. If maintainers accept the feature, implement a narrow fetch-guard extension that preserves the current strict default, validates custom header names, blocks credential-like names such as authorization, cookie, proxy-authorization, token, secret, and api-key unless an explicitly reviewed scope says otherwise, and exposes the option only through the intended per-call/config or SDK surface. Add tests for allowed custom headers, denied credential headers, same-origin redirects, cross-origin redirects, media downloads, and provider request config/schema behavior.

## Risks / Open Questions

- A broad additionalSafeHeaders setting could leak API keys, bearer-like credentials, cookies, or tenant secrets across attacker-controlled cross-origin redirects unless it validates header names and stays fail-closed by default.
- fetchWithSsrFGuard is exported through plugin/runtime surfaces and is used by provider, media, plugin marketplace, memory-host, and tool paths, so the design should be narrow and covered by regression tests.
