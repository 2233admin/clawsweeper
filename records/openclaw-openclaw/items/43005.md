## Summary

Current main still does not implement the requested opt-in custom safe-header behavior for fetch-guard cross-origin redirects. The allowlist is fixed, `x-api-key` is still expected to be stripped, and there is no per-call, SDK, or config/docs surface for `additionalSafeHeaders` or equivalent. The linked #57588 remains related security context, not a duplicate or superseding tracker.

## Best Possible Solution

Keep this issue open. If maintainers accept it, implement a narrow fetch-guard extension that preserves the current strict default, validates caller-provided header names, denies credential headers such as authorization, cookie, proxy-authorization and credential-like custom names unless deliberately scoped otherwise, and exposes the option only through the intended per-call/config or SDK surface. Add regression tests for allowed custom headers, denied credential headers, same-origin redirects, cross-origin redirects, media downloads if affected, and schema/docs updates if a user config surface is added.

## Risks / Open Questions

- Allowing custom redirect-safe headers can leak API keys or bearer-like credentials across attacker-controlled cross-origin redirects unless the design validates names, stays fail-closed by default, and blocks credential-like headers or scopes them tightly.
- fetchWithSsrFGuard is a shared core and plugin-facing runtime surface used by providers, plugins, marketplace, media, cron, tools, and memory-host paths, so a broad config-only knob could create cross-surface security regressions.
