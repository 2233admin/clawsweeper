## Summary

Current main still has no opt-in custom safe-header mechanism for fetch-guard cross-origin redirects. The helper now lives in `src/infra/net/redirect-headers.ts`, but the allowlist is still fixed, `GuardedFetchOptions` has no `additionalSafeHeaders`-style option, provider/media paths still use the fixed behavior, and tests still assert `x-api-key` is stripped on cross-origin redirects. The related security issue #57588 is adjacent bearer-auth fallback work, not a duplicate for redirect header allowlisting.

## Best Possible Solution

Keep this issue open. If maintainers accept the feature, add a narrow fail-closed fetch-guard extension that validates header names, preserves the current strict default, blocks credential-like names by default, exposes the option only through the intended per-call/config or SDK surface, and adds tests for allowed custom headers, denied credential headers, same-origin redirects, cross-origin redirects, media downloads, and provider request config/schema behavior.

## Risks / Open Questions

- Allowing arbitrary custom headers across cross-origin redirects is security-sensitive; `x-api-key`, token, secret, cookie, and authorization-like names can leak credentials to attacker-controlled redirect targets if the design is too broad.
- `fetchWithSsrFGuard` is exported through plugin/runtime surfaces and used by provider, media, plugin, memory-host, and extension paths, so any option needs narrow defaults and regression coverage across those call sites.
