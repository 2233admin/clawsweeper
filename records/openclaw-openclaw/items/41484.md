## Summary

Keep this issue open. Current main has partial mitigations for cron and heartbeat trust boundaries, but it still does not implement the requested per-job cron elevated flag or heartbeat provenance/explicit elevated marker. Elevated exec availability remains provider/sender allowlist based, while cron job storage, Gateway schemas, Control UI types, docs, and heartbeat runtime have no per-job or per-file trust metadata.

## Best Possible Solution

Keep this issue open for maintainer design. The best implementation is an additive trust model: add a cron job elevated/trust field to cron types, normalization, persistence, Gateway schemas, CLI/Control UI surfaces, docs, and tests; gate setting or changing that field on owner/elevated-approved senders; and add heartbeat provenance or an explicit heartbeat elevated marker so HEARTBEAT.md-driven runs do not rely on broad cron-event or heartbeat wildcard allowlists.
