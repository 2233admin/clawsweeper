## Summary

Keep open. Current main still has cron and heartbeat hardening, but it does not implement the requested persisted per-job elevated/trust flag, owner-approved mutation rule, or heartbeat provenance/explicit elevated metadata.

## What This Changes

Request per-job elevated exec scoping for cron and heartbeat so trusted scheduled work can run elevated without granting wildcard elevated access to internal cron-event or heartbeat sources.

## Best Possible Solution

Keep this open for maintainer/secops design. The best path is an additive trust model that persists cron job elevated/provenance metadata, gates setting or changing elevated=true on an approved owner/elevated sender rather than the internal cron-event wildcard, updates cron normalization/storage/protocol/CLI/UI/agent tool/docs/tests, and adds heartbeat provenance or an explicit heartbeat elevated marker so HEARTBEAT.md-driven runs do not rely on broad internal event allowlists.
