## Summary

Keep this issue open. Current main has useful mitigations around owner-only cron tooling, isolated cron session cleanup, and untrusted heartbeat events, but it still does not implement the requested per-job cron elevated flag or heartbeat provenance/explicit elevated marker. Cron jobs, gateway schemas, CLI/UI/docs, and heartbeat runtime still rely on provider-class elevated allowlists such as `cron-event` and `heartbeat`, not per-job/per-file trust metadata.

## Best Possible Solution

Keep this open and implement an additive trust model: add a cron job elevated/trust field to cron types, gateway schemas/validators, normalization/persistence, cron tool, CLI/UI surfaces, docs, and tests; restrict setting or changing it to owner/elevated-approved senders; and add heartbeat provenance or an explicit heartbeat elevated marker so `HEARTBEAT.md`-driven runs do not inherit broad internal-event elevated access by wildcard alone.
