## Summary

Keep this PR open. Current main has related string-content compatibility for OpenAI-compatible completions backends, but it does not implement the PR’s null-to-empty-string normalization for assistant tool-call messages; the upstream `pi-ai` conversion path can still emit `content: null` when empty assistant text is filtered and tool calls remain. The linked bug report and follow-up comments provide concrete provider failure evidence and a reporter-confirmed local fix, while the PR now needs maintainer rebase/review because its branch is dirty against current main.

## What This Changes

The PR adds an OpenAI stream wrapper and tests that normalize assistant messages with `tool_calls` from `content: null` to `content: ""` for `openai-completions` payloads, then wires that wrapper into the embedded runner extra-params flow.

## Best Possible Solution

Land a rebased, narrow OpenAI-completions payload normalizer that changes only assistant messages with non-empty `tool_calls` and nullish content to `""`, preserves Responses API and non-tool-call behavior, and carries targeted regression coverage in the current wrapper composition.
