## Summary

Keep open. Current main still forwards the configured compaction reserve into Pi summarization without clamping it to the selected model output cap, and the pinned Pi implementation still derives `maxTokens` as 80% of that reserve. With `reserveTokensFloor: 300000`, the built-in path can still request 240000 output tokens, while the Anthropic transport preserves positive requested `maxTokens` values as `max_tokens`. Open PR #54392 remains the same-author implementation candidate and uses closing syntax for this issue, so this should stay open until that PR or an equivalent patch lands.

## What This Changes

Clamp compaction summarization output budgeting so Anthropic 1M-context compaction cannot request `max_tokens` above the selected model's output limit.

## Best Possible Solution

Land PR #54392 or an equivalent maintainer-approved patch that clamps built-in LLM compaction reserve/output budgeting against the resolved summarization model output cap before Pi `generateSummary` or `completeSimple` can request `max_tokens`, while preserving provider/plugin compaction behavior and Anthropic header handling. Add regression coverage for an Anthropic Claude 4.6 1M-context setup with `reserveTokensFloor: 300000` proving the request cannot exceed 128000 output tokens.
