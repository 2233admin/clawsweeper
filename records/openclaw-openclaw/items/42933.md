## Summary

Keep open. Current main at db40ec404a9178294dba3121758ff59459983a11 still does not implement the PR's intended configurable session-level circuit breaker. The repo has adjacent mitigations, including auth-profile cooldowns, cron error backoff/failure alerts, context-overflow recovery text/reset paths, and heartbeat duplicate suppression, but it lacks the requested `agents.defaults.circuitBreaker`/per-agent config, persisted `cb*` session state, and heartbeat/cron/reply breaker integration. The PR remains a plausible implementation candidate, but the review discussion and current diff show correctness and alert-routing/security follow-up work before merge.

## Best Possible Solution

Keep this PR open and request a focused revision before merge. The best path is a core-owned session-level breaker with typed `agents.defaults.circuitBreaker` and per-agent schema, persisted per-session breaker state, structured model-failure metadata from interactive reply, heartbeat, and cron execution, correct isolated-heartbeat session ownership, `/new`-equivalent reset semantics, validated/sanitized alert delivery, and targeted tests across heartbeat, cron, and interactive reply paths. If maintainers want less scope, split the generic config/state and structured failure signal from optional alert/reset actions.

## Risks / Open Questions

- Closing this PR now would drop an active implementation candidate while current main still lacks the requested session-level breaker capability.
- Merging as-is could still mis-handle heartbeat error classification, especially because the PR relies on formatted warning-text prefixes for some non-throwing model failures rather than a structured reply error signal.
- The PR's reset action mutates session identity/state directly; review should confirm it matches `/new` semantics, clears heartbeat dedupe state where needed, and targets isolated heartbeat sessions rather than the user's main session when `heartbeat.isolatedSession` is enabled.
- The alert action is security-sensitive: it can send session keys and last-error details to configured external channels. Before merge, maintainers should validate/sanitize alert routing and ensure model error text cannot leak secrets or route to an unintended target.
