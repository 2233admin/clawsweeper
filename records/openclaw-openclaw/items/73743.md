## Summary

Keep open. The report is current, detailed, and not implemented on current main: the named `recipes`/`kitchen` plugin source is not in this checkout, but core still exposes the relevant subprocess and plugin-CLI behavior, including timeout results that can carry `termination: "timeout"` while leaving `code` null and deprecated runtime config warnings that go through normal warning logging. That leaves both the startup/fan-out regression and the cache-poisoning hardening unresolved.

## What This Changes

The issue reports slow OpenClaw CLI subprocess startup on macOS that compounds under many launchd workflow ticks and causes ClawKitchen to cache partial timed-out `recipes list` output as success.

## Best Possible Solution

Keep this open and treat it as a core/plugin-startup investigation with a smaller hardening follow-up: profile `openclaw recipes list` and `recipes workflows worker-tick` under the reported macOS launchd setup, reduce plugin CLI initialization for single-plugin read-only commands, and make subprocess timeout results unambiguously failed so partial stdout cannot be cached as success.

## Reproduction Assessment

Yes for a maintainer with the matching macOS setup: the issue includes concrete versions, launchd shape, commands, timings, process/RSS observations, and the downstream cache symptom. It was not live-reproduced in this Linux read-only checkout, and the external `recipes`/`kitchen` sources are not available here.

## Solution Assessment

No patch is proposed here, and current main is not the best completed solution. The safest direction is to split the work into measurable startup profiling/optimization for the plugin command path plus a focused subprocess timeout and stdout-warning hardening change.
