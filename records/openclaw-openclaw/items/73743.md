## Summary

Keep open for maintainer follow-up. The report is detailed and current main does not contain the recipes or ClawKitchen worker/cache implementation needed to prove the central startup/fan-out problem is fixed; the in-repo core timeout wrapper now avoids treating timeout kills as success, but that only addresses one downstream failure mode.

## What This Changes

Fix a reported 2026.4.26 regression where slow `openclaw recipes` subprocess startup and launchd fan-out make workflow ticks run for minutes and poison ClawKitchen's cached `recipes list` output.

## Best Possible Solution

Profile current main with the external recipes and Kitchen plugins enabled, then fix the narrow owner module: make the recipes CLI/worker path command-scoped and lightweight, add jitter or a gateway-backed queue claim path if worker fan-out is expected, and ensure Kitchen refuses to cache timed-out or non-JSON subprocess output. Add a core plugin API seam only if that profiling shows existing `activation.onCommands` and `runtime.system` contracts are insufficient.

## Reproduction Assessment

Yes, but it is environment-bound. The issue gives a high-confidence reproduction path on macOS with OpenClaw 2026.4.26, 19+ launchd `worker-tick` plists, and the external recipes/Kitchen plugins; the current checkout alone cannot reproduce it because those plugin implementations are absent.

## Solution Assessment

Unclear. The suggested directions are plausible, and current core already handles subprocess timeouts more safely, but the best fix depends on profiling whether startup cost is from core plugin loading, external recipes/Kitchen runtime initialization, or launchd scheduling.
