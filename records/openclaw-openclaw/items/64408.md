## Summary

Keep this open. Current main still has the reported warning path, and it still only logs the tool call id, argument type, and optional preview, which is not enough to correlate the malformed embedded `read` call back to a run, session, agent, trigger, provider, or channel. The adjacent canonical `path` validation is useful but does not resolve the requested diagnostics/root-cause work.

## What This Changes

The issue asks OpenClaw to identify, diagnose, and prevent embedded/background `read` tool-start events that omit the required `path` parameter.

## Best Possible Solution

Add structured correlation to the missing-path warning using existing embedded run identity, then use those logs to identify the owner that constructs the malformed `read` call and either supply canonical `path` or reject the call before it reaches the tool-start handler.
