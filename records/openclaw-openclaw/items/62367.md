## Summary

Keep open. Current main still does not make `openai-codex/gpt-5.4-nano` a first-class Codex OAuth route: the direct OpenAI provider has nano catalog/runtime handling, but the OpenAI Codex provider and manifest only carry GPT-5.4, pro, mini, and 5.5-pro. A recent main commit restored only the Codex mini route, which is useful adjacent evidence but does not resolve the nano report.

## What This Changes

The issue reports that `openai-codex/gpt-5.4-nano` may normalize to another Codex model or fall through to another provider instead of executing as nano or failing explicitly.

## Best Possible Solution

Make the Codex OAuth provider treat `openai-codex/gpt-5.4-nano` consistently with the intended product contract: either add first-class nano catalog, dynamic resolution, runtime preference, and tests analogous to mini, or add a targeted unsupported-model suppression/error so it cannot silently normalize or fall through to another provider.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: request resolution for `openai-codex/gpt-5.4-nano` reaches the Codex provider, which has no nano dynamic-model or manifest row on current main, while direct OpenAI nano support exists.

## Solution Assessment

Yes, a narrow provider-owned fix is the best path. The change should stay in the OpenAI plugin/provider metadata and focused model-resolution tests rather than adding a core fallback special case.
