## Summary

Keep this issue open. Current main has related mitigations and diagnostics, but the central gateway high-CPU/stalled-reply problem is still actionable: the outbound pending-spawn path can still synchronously read the persisted subagent registry during normal runtime, `channels status` still depends on a live Gateway WebSocket RPC that times out when the event loop is saturated, and the newest issue comments add current cross-platform reproductions rather than removing the need for a fix.

## Best Possible Solution

Keep this issue open as the canonical Gateway high-CPU/reply-stall investigation. The likely next fix is to remove synchronous disk reads from the outbound pending-spawn hot path by using an in-memory or cached subagent-run snapshot with explicit invalidation or bounded refresh, plus a regression/performance test proving an empty `subagents/runs.json` is not reread per outbound payload plan. In parallel, investigate the CLOSE-WAIT/self-loopback timeout data and the narrower plugin-loader report #73532, but do not close this issue until the broad Gateway stall behavior is fixed or cleanly split into resolved canonical follow-ups.
