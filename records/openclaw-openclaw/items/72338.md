## Summary

Keep open. Current main has useful mitigations for plugin manifest/runtime-deps caching, handshake timeout tuning, and stuck-session recovery, but it does not yet prove the reported gateway CPU/event-loop starvation is fixed. The hot path identified in the discussion is still present: Telegram delivery can call outbound payload planning with a session key, outbound planning still resolves pending spawned children, and that registered predicate still reaches a production `subagents/runs.json` disk read.

## Best Possible Solution

Keep this issue open as the canonical gateway CPU/event-loop starvation report. The best next step is to remove or bound synchronous persisted subagent-registry reads from normal outbound reply planning, keep cross-process visibility through an explicit cached snapshot or invalidation path, and add regression/performance coverage proving Telegram/outbound planning does not reread `subagents/runs.json` per reply. Treat #73532 and #73581 as linked narrower work, not replacements for this canonical report.
