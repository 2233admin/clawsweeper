## Summary

Keep open. Current main has useful partial mitigations for stuck sessions, plugin manifest/runtime-deps caching, handshake timeouts, and liveness diagnostics, but the concrete outbound hot path from the discussion is still present: Telegram reply delivery can call outbound payload planning with a session key, the planner still queries pending spawned children, and the registered predicate still reaches a production `subagents/runs.json` read through `loadJsonFile`/`fs.readFileSync`. The issue is recent, corroborated across several platforms and Node versions, and related narrower reports remain active rather than superseding this canonical gateway CPU/event-loop starvation report.

## Best Possible Solution

Keep this issue open as the canonical gateway CPU/event-loop starvation report. The next best code step is to remove synchronous persisted subagent-registry reads from normal outbound reply planning by using an in-memory or bounded cached snapshot with explicit invalidation/refresh for cross-process visibility, plus a regression test proving repeated outbound planning does not reread `subagents/runs.json`. Keep #73532 and #73581 linked for the narrower plugin-loader and stuck-lane work, and close this only after the main stall path is fixed and shipped.

Current main path: Telegram delivery calls `createOutboundPayloadPlan` with a session key (`extensions/telegram/src/bot/delivery.replies.ts`), outbound planning calls `resolvePendingSpawnedChildren` (`src/infra/outbound/payloads.ts`), the registered query calls `countPendingDescendantRuns` (`src/agents/subagent-registry.ts`), and production `getSubagentRunsSnapshotForRead` rereads persisted `subagents/runs.json` through `loadSubagentRegistryFromDisk` / `loadJsonFile` (`src/agents/subagent-registry-state.ts`, `src/agents/subagent-registry.store.ts`, `src/infra/json-file.ts`).
