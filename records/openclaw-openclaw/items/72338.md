## Summary

Keep open. #72338 is a fresh, detailed bug against v2026.4.24, and current main does not yet implement the requested gateway-level recovery or login isolation. The code has partial channel-level health and Telegram polling safeguards, but `channels status` still just times out and falls back when the Gateway is unresponsive, the health monitor is an in-process timer based on channel snapshots, WhatsApp CLI login remains an interactive wait path, and the post-release history does not show a relevant fix. Related reports do not supersede it: #71529 was a narrower EHOSTUNREACH crash fix, while #45759 and #66360 remain open for separate Telegram/CPU surfaces.

## Best Possible Solution

Keep #72338 open and investigate it as a current gateway reliability bug. The best fix path is to reproduce or collect diagnostics on v2026.4.24/current main, add process-level event-loop lag or responsiveness watchdog evidence for loopback RPC stalls, isolate or guard interactive channel login launched from agent/gateway execution paths, and improve stability dumps with active run/channel/task context so future unhandled-rejection bundles are actionable. Any fix should include targeted regression coverage around `channels.status` responsiveness and channel/login runtime isolation.
