## Summary

Keep open. Current main still has the same actionable Gateway/runtime performance path identified in the discussion: outbound payload planning can synchronously query pending spawned children, that query reaches the subagent registry, and production registry reads still load subagents/runs.json from disk with synchronous file I/O. The Apr 28 comments add concrete cross-architecture reproductions and CLOSE-WAIT growth, so this is not stale and is not safely superseded by the older broad Gateway CPU reports.

## Best Possible Solution

Keep this issue open as the current canonical Gateway/runtime performance bug. Remove synchronous disk reads from the outbound pending-spawn hot path by using an in-memory or cached subagent-run snapshot with explicit invalidation or bounded refresh, while preserving the cross-process visibility that needs persisted state. Add a regression/performance test proving outbound payload planning does not reread an empty subagents/runs.json per plan. Separately investigate the CLOSE-WAIT/self-loopback report and consider an in-process Gateway dispatch path or bounded internal RPC pool with guaranteed client close on timeout.
