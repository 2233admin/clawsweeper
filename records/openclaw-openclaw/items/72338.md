Keep open. The report is recent, multi-host, and high-impact; current main has several plausible mitigations on the reported hot paths, including runtime-deps fast paths, subagent registry read caching, runtime mirror reuse, and event-loop-aware clients, but the central Gateway CPU/starvation symptom is not yet proven fixed across affected packaged installs or shipped in a release that includes the latest runtime-deps change.

The issue asks OpenClaw to prevent Gateway high-CPU/event-loop starvation that stalls Telegram replies and causes local loopback status probes to time out.

Keep this issue open as the broad Gateway CPU/event-loop starvation tracker until maintainers can retest main/dev or the next package on affected hosts and tie successful idle CPU, loopback probe, and Telegram/WebChat turn behavior to a specific shipped release or explicit canonical follow-up.
