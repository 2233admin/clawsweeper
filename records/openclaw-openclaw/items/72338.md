## Summary

Keep open. Current main includes partial mitigations and diagnostics for this Gateway high-CPU/stall class, but the central report is still actionable: outbound reply planning can still synchronously query pending subagent descendants, and the production snapshot path still rereads `subagents/runs.json` with `fs.readFileSync`. `channels status --probe` also still depends on a live Gateway WebSocket RPC and only falls back to config-only output after that RPC times out. The newer related plugin-loader report #73532 is useful, but it does not supersede this issue because this issue still covers the `runs.json` polling and local RPC timeout/stall pattern.

## Best Possible Solution

Keep this issue open as the canonical Gateway/runtime stall thread. The best fix is to make the pending-spawn outbound predicate truly cheap by using an in-memory or cached subagent-run snapshot with explicit invalidation or bounded refresh, while preserving cross-process visibility only where required. Add a regression test proving outbound payload planning does not reread `subagents/runs.json` per plan. Separately keep #73532 linked for the plugin-loader hot loop and investigate the CLOSE-WAIT/local RPC timeout path with a bounded internal dispatch or guaranteed client close-on-timeout.
