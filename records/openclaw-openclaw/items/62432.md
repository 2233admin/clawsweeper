## Summary

Keep open. Current main has partial protection for repeated running `exec` outcomes when `tools.loopDetection` is enabled, but the default runtime still does not prevent an identical in-flight `exec` from launching another background process, and the related closed issues address adjacent wait/completion-routing bugs rather than this process fan-out pattern.

## What This Changes

The issue reports that Xiaomi/MiMo-backed sessions can relaunch identical long-running `exec` commands after a non-terminal running result instead of polling the existing process.

## Best Possible Solution

Add a narrow exec-layer guard that detects an identical command/workdir/scope still running from the same agent session and returns or coalesces to the existing pollable session instead of spawning another process, while preserving the broader loop-detection work as a separate maintainer decision.

## Reproduction Assessment

Yes for the core runtime gap: current source and tests show default sessions do not run loop detection, and the registry accepts each new background exec under a fresh session id. The exact Xiaomi/MiMo provider behavior still needs a live model trace to prove the model-side trigger on current main.

## Solution Assessment

Yes, a scoped in-flight `exec` dedupe or explicit existing-session response is the narrowest maintainable fix for this issue; globally enabling all loop detection or changing Xiaomi routing is broader product policy and should remain separate unless maintainers choose that path.
