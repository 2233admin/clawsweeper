## Summary

Current main still has adjacent cron hardening but not the requested fail-closed delivery policy. Cron now handles structured errors, denial markers, run-level failures, stale acknowledgement replies, exact/trailing NO_REPLY suppression, delivery-target checks, and mutating tool-error warnings, but it does not suppress fabricated final assistant text, marker phrases like "placeholder"/"mock data"/"simulate", missing required-tool evidence, or unsupported success claims.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is a narrow unattended cron/session delivery policy, likely opt-in first, that suppresses or cleanly fails delivery when required resources or tool calls failed, final output contains self-referential draft/fabrication markers, or success claims are unsupported by turn/run evidence. Add focused regression tests around resolveCronPayloadOutcome, dispatchCronDelivery, and the embedded-run payload/error path while preserving existing NO_REPLY, stale-ack, denial-token, run-level-error, mutating-tool-warning, and --no-deliver behavior.
