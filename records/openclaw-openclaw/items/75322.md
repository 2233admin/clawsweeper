## Summary

Current main still has the reported mismatch: the internal async exec completion prompt asks for `HEARTBEAT_OK` without showing `<final>` wrapping, while the enforced-final runtime can turn an unwrapped `NO_REPLY` response into zero payloads. This is narrow, reproducible from source, and should stay open for a focused fix.

## What This Changes

Request that internal async exec completion prompts use the runtime silent-reply sentinel with required `<final>...</final>` wrapping so delivery-disabled completions do not surface incomplete-turn errors.

## Best Possible Solution

Update the exec completion prompt and regression coverage so delivery-disabled async exec completions produce a recognized silent terminal reply, preferably by using the shared silent token constant and an exact `<final>NO_REPLY</final>` example for internal-only completions while preserving current visible-delivery behavior.

## Reproduction Assessment

Yes. A focused mocked run can reproduce the path by returning bare `NO_REPLY` from a final-tag-enforced heartbeat exec completion with `deliverToUser=false`; current source shows that the prompt invites the mismatch and final-tag enforcement strips unwrapped text to zero payloads.

## Solution Assessment

Yes, a narrow prompt-and-test fix is the best first repair. If tests show the runtime still sees stripped raw silent tokens, add a small guard for exact raw `NO_REPLY` in the incomplete-turn path rather than broadening final-tag enforcement.
