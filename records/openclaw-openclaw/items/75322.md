## Summary

Current main still has the reported prompt/runtime mismatch, so this should stay open and is a good narrow repair candidate. The exec-completion prompt still tells delivery-disabled runs to answer with HEARTBEAT_OK, while the embedded runner can still retry and surface incomplete-turn errors for zero-payload assistant turns when the run is not independently marked as silent-allowed.

## What This Changes

The issue asks OpenClaw to align delivery-disabled async exec completion prompts with the runtime silent-reply contract so suppressed completions do not surface incomplete-turn noise.

## Best Possible Solution

Keep the issue open and make a focused fix in the heartbeat exec-completion prompt path, backed by tests that cover delivery-disabled and empty-output exec events producing a runtime-accepted silent final reply without broad changes to heartbeat acknowledgements or generic empty-response fallback policy.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `buildExecEventPrompt(..., { deliverToUser: false })` currently asks for HEARTBEAT_OK, and a zero-payload assistant turn that is not silent-allowed follows the existing empty-response retry and incomplete-turn path.

## Solution Assessment

Yes, the suggested direction is likely the narrowest maintainable fix: align the delivery-disabled exec prompt with the existing `NO_REPLY` silent sentinel and explicit final-answer wrapping. A broader runtime exception for all delivery-disabled heartbeat turns would be riskier unless tests show prompt alignment is insufficient.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75322. Current main has `buildExecEventPrompt` telling `deliverToUser=false` and empty-output async exec completion turns to reply `HEARTBEAT_OK only`, which can race the embedded runtime's silent-reply/empty-payload detection and surface incomplete-turn noise. Make a narrow fix in the exec completion prompt path so suppressed exec completions instruct the model to produce the runtime silent sentinel with explicit final-answer wrapping, e.g. exactly `<final>NO_REPLY</final>` and nothing else. Prefer importing/reusing `SILENT_REPLY_TOKEN` rather than hard-coding a second token string where practical. Update `src/infra/heartbeat-events-filter.test.ts` so delivery-disabled and empty/metadata-only exec completion prompts no longer expect `HEARTBEAT_OK` and do expect the wrapped silent reply. Add a focused regression around the runner only if needed to prove zero-payload suppressed exec completions do not surface incomplete-turn errors. Do not change ordinary heartbeat `HEARTBEAT_OK` ack semantics, cron prompt behavior, generic empty-response retry policy, or unrelated silent-reply policy. Add a single-line `CHANGELOG.md` fix entry under Unreleased/Fixes, crediting `Thanks @mbrya859-byte`.
