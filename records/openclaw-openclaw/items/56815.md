## Summary

Keep open. Current main still does not solve the reported large memory reindex failure mode: `TypeError("fetch failed")` and common transport codes are not retryable in the memory embedding retry classifier, async sync warnings still flatten errors with `String(err)`, and failed atomic reindex still removes the temp database. Related #44167 is an open implementation candidate for the transport retry/split subset, #56879 was closed unmerged, and #26772 tracks the checkpoint/resume portion, so this issue should remain open until those pieces land or are deliberately split by a maintainer.

## Best Possible Solution

Keep this issue open. Review and land #44167 or an equivalent memory-core change that classifies narrow transient embedding transport failures and preserves result order during split fallback, then address the issue-specific diagnostics and progress-loss pieces: redacted stack/cause-aware memory sync warnings and a safe checkpoint/resume or temp-index preservation design that retains atomic swap safety. Coordinate the resume design with #26772 if maintainers want that broader work tracked there, and add focused tests for transport-shaped `TypeError("fetch failed")` plus the chosen failed-reindex recovery behavior.
