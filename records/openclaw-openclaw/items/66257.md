## Summary

Keep open. Current main still uses Gateway-first `devices approve` behavior and lacks the PR's explicit token/password fallback guard, so the intended change is not implemented; however the draft PR changes a device approval path and needs maintainer judgment on Gateway side effects and explicit-auth semantics before merge.

## What This Changes

The PR changes the devices CLI to try implicit local loopback pairing approval before Gateway approval, skip local pairing fallback when explicit token/password auth is supplied, and update CLI tests for direct local approval plus unknown-request fallback.

## Best Possible Solution

Keep the PR open and route it through maintainer review: adapt it to current main, preserve structured pairing-required parsing, keep local-first approval limited to implicit local loopback, fall back to Gateway approval when the local request is unknown, document accepted token/password behavior, and either mirror or deliberately waive Gateway approval side effects.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve device approval broadcasts:** `src/cli/devices-cli.ts:172-174`
  - body: By returning after `approveLocalPairingRequest` on implicit local loopback, the CLI skips the Gateway `device.pair.approve` path for approvals the Gateway could otherwise handle. That Gateway method logs the approval and broadcasts `device.pair.resolved`, and the Control UI reloads devices from that event, so subscribed clients can stay stale after a local-first approval. Add an equivalent notification path or keep Gateway in the successful approval flow.
  - confidence: 0.84
