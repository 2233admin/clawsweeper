## Summary

Keep open. The linked duplicate-reply bug is plausible and current main still contains the broad message-tool guidance, but this PR is not safe as written because it conflicts with current Discord guild/channel visibility policy: default final replies are private/message_tool_only, while the PR tells Discord agents not to use `message.send` or `message.thread-reply` for normal current-conversation replies.

## What This Changes

The PR updates Codex app-server developer instructions, the bundled Discord skill, and a Codex unit test so current-conversation Discord replies use final assistant delivery rather than the message tool.

## Best Possible Solution

Revise the guidance so it is conditional on the active source-delivery policy: use final assistant responses when OpenClaw will deliver that final to the current conversation or thread, but keep message-tool guidance for Discord guild/channel turns where visible replies are intentionally message_tool_only. Add focused tests for both automatic auto-thread delivery and default private guild/channel delivery.

## Reproduction Assessment

Yes. The linked bug has concrete 2026.4.26 steps and current main still contains the prompt and skill wording that can steer Codex toward `message.send`; on current main the exact visible duplicate path likely requires automatic visible replies, because the default guild/channel mode now suppresses automatic final delivery.

## Solution Assessment

No. The direction is useful, but the patch is too broad because it tells Discord agents to avoid the message tool even in current-main modes where normal final replies are intentionally private and the message tool is the visible reply path.
