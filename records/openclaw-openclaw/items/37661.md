Keep open. ## What This Changes

Requested a guard against LLM streaming/final assistant output loops where the same phrase repeats many times and duplicate user-channel messages are delivered.

## Best Possible Solution

Implement a conservative assistant-output repetition guard in the streaming/final reply path and a shared per-turn outbound duplicate/message cap before channel delivery. Keep #34574 scoped to tool-call loops and treat #39536 as heartbeat-specific context, not a full replacement for this generic guard.

## Repair Work Prompt

Implement a conservative fix for assistant text repetition loops described by this issue. The observable bug is pure assistant output repetition: a model appends the same phrase/sentence fragment many times during streaming/final output and OpenClaw forwards it as one unreadable message or multiple duplicate channel messages. Keep #34574 scoped to tool-call loops and use #39536 only as related heartbeat-specific context.

Expected boundary: add a small reusable assistant-output repetition guard for normalized appended text, likely near `src/agents/pi-embedded-subscribe.handlers.messages.ts` and its state/types, and apply it before partial/block reply emission and final assistant text delivery. Add a shared per-turn outbound duplicate/message cap in or around `src/auto-reply/reply/block-reply-pipeline.ts` so channel delivery cannot send many repeated assistant blocks in one turn. Preserve existing exact provider-resend dedupe, block streaming semantics, reasoning/media/TTS directive behavior, and tool-loop detection behavior. Do not add model/provider-specific logic.

Tests: add focused Vitest coverage for repeated delta appends of the same phrase exceeding the guard threshold, repeated logical block replies in one turn, and existing safe cases: repeated `text_end` resend remains deduped, normal repeated words in legitimate prose are not blocked too early, and final payload suppression still works. Update docs/changelog if a user-facing config or behavior changes; a changelog fix entry should credit @himiko776-v-6 and @ahaoostar if added.
