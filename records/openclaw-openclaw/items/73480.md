## Summary

Keep open. Current main still preserves the Matrix recovery-key, backup-usability, and full identity-trust split, still has coverage for the `recoveryKeyAccepted: false` plus `backupUsable: true` diagnostic state, and still lacks the `/keys/upload` OTK-collision or MAS cross-signing mitigations from the active related PRs.

## What This Changes

The issue reports that Matrix recovery with a recovery key fails on OpenClaw 2026.4.26, with `verify device --recovery-key-stdin` leaving the device untrusted and `matrix encryption setup` failing on a `/keys/upload` one-time-key collision.

## Best Possible Solution

Keep this issue open as the Matrix E2EE recovery umbrella until maintainers review and land or replace the focused OTK-collision and MAS/cross-signing fixes, then verify the reported `verify device --recovery-key-stdin` and `matrix encryption setup` sequence against an affected or disposable Matrix account.

Unclear. The issue includes concrete commands and current main has unit coverage for the accepted-no/backup-usable diagnostic state, but a high-confidence live reproduction requires an affected Matrix account/homeserver with the same recovery and OTK-collision state.
