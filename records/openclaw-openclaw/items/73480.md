## Summary

Keep open: current main still preserves the Matrix recovery-key/device-trust split and lacks the MAS UIA reset handling and `/keys/upload` one-time-key collision mitigation described by the active related work, so this regression is not implemented or superseded yet.

## What This Changes

The issue reports a Matrix E2EE recovery regression where `matrix verify device --recovery-key-stdin` and `matrix encryption setup` fail to recover a device, including an accepted-no/backup-usable diagnostic and a `/keys/upload` one-time-key collision.

## Best Possible Solution

Matrix maintainers should land or reject focused fixes for MAS cross-signing reset/UIA handling and `/keys/upload` OTK collision handling, then verify the reported recovery-key and encryption setup commands on an affected or disposable homeserver and keep the CLI/docs/tests aligned with the final trust-state semantics.
