## Summary

Keep this issue open. It is a fresh Matrix regression report with concrete command output from v2026.4.26, and current main still has the same recovery-key/device-trust behavior for the reported surface. The existing ClawSweeper comment is still consistent with the current checkout: docs explain Matrix recovery-key backup usability and full cross-signing trust as separate states, but current source/tests do not prove the reported `Recovery key accepted: no` with `Backup usable: yes` plus `keys/upload` setup failure is resolved.

## Best Possible Solution

Keep the issue open for Matrix maintainer follow-up. Reproduce the Windows/Docker `openclaw matrix verify device --recovery-key-stdin` and `openclaw matrix encryption setup` sequence on current main, then fix the recovery-key acceptance logic, the CLI guidance for `recoveryKeyAccepted: false` with `backupUsable: true`, or the setup bootstrap handling for Matrix `keys/upload` one-time-key conflicts. If the current behavior is intended for an invalid attempted key plus previously cached backup material, the CLI should say that directly and point to the next repair step instead of looping back to the same command.
