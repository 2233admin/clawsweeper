## Summary

Keep open. This is a fresh Matrix regression report with concrete CLI output, and current main still contains the same recovery-key/device-trust behavior. The prior ClawSweeper comment kept it open, and a current-main pass confirms there is no relevant post-v2026.4.26 implementation fix for the reported recovery path.

## Best Possible Solution

Keep this issue open for Matrix maintainer follow-up. Reproduce the `matrix verify device --recovery-key-stdin` and `matrix encryption setup` sequence on current main, then fix the recovery-key acceptance logic, the CLI guidance for `recoveryKeyAccepted: false` with `backupUsable: true`, or the bootstrap handling/reporting for Matrix `keys/upload` one-time-key conflicts. If the behavior is intended, make the CLI and docs distinguish invalid/mismatched recovery keys from already-usable cached backup material and point to the next repair step.
