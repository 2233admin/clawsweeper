## Summary

Keep this issue open. Current main still pushes the final residual chunk unchanged, and open draft PR #72491 still explicitly targets the bug with closing syntax, so the issue should stay open until that PR or an equivalent fix is merged or closed.

## What This Changes

The issue reports that `chunkTextByBreakResolver` trims intermediate chunks but leaves trailing whitespace on the final residual chunk.

## Best Possible Solution

Review and land PR #72491, or an equivalent narrow fix, so the shared resolver trims and skips only over-limit final residual chunks, adds regression coverage for the reported cases, and keeps the Matrix runtime copy plus contract guardrail aligned.

## Reproduction Assessment

Yes. Current main still has a deterministic source-level reproduction: once `remaining` becomes `! ` or `b ` after the split loop, `src/shared/text-chunking.ts:31` pushes it without `trimEnd()`.

## Solution Assessment

Yes. The narrow maintainable fix is to apply `trimEnd()` only to the final residual chunk, skip it if empty, and cover the reported counterexamples while preserving existing under-limit behavior and Matrix copy compatibility.
