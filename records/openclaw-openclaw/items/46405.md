## Summary

Keep open: current main supersedes the gateway restart half, but it still strips MiniMax XML tool calls instead of promoting them to structured toolUse blocks. The PR remains useful source work, but its MiniMax parser has blocking XML parsing and entity-decoding bugs, so it should not be closed as implemented or merged as-is.

## What This Changes

The PR branch adds MiniMax M2.5 XML invocation parsing/promotion into assistant toolUse blocks and modifies gateway config.patch restart handling, prompt/tool wording, and regression tests.

## Best Possible Solution

Rebase or replace this PR with a focused MiniMax-only change that promotes XML invocations into the existing toolUse execution path using robust XML/attribute parsing and entity decoding, while leaving gateway config.patch behavior to the current main implementation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Parse XML attributes independent of order:** `src/agents/pi-embedded-utils.ts:35`
  - body: `parseMinimaxToolCallXml` only matches `<invoke>` when `name` is the first attribute, so valid XML like `<invoke id="x" name="Bash">...` is skipped. Because the later sanitizer strips the XML, the intended tool call is silently lost; the parser should read attributes in any order for both invoke and parameter tags.
  - confidence: 0.9
- **[P2] Decode XML entities before passing tool inputs:** `src/agents/pi-embedded-utils.ts:48-51`
  - body: Parameter text is copied directly from XML captures, so valid encoded values such as `&amp;`, `&lt;`, or `&quot;` reach tools unchanged. That changes commands and file contents from what the model encoded in the XML, so decode standard XML entities before building the `toolUse.input`.
  - confidence: 0.88
