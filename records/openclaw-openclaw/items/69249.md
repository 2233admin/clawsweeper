## Summary

Keep #69249 open. Current main has useful restart hardening, but the specific Discord-visible recovery gap is still not fully solved: managed restart-sentinel flows can post a routed notice, while direct/startup main-session recovery still resumes internally with `deliver: false` and does not post an apology/recovery/resume message into the original Discord channel.

## Best Possible Solution

Keep the issue open and implement a generic channel-visible restart-abort recovery path. The likely fix belongs in startup/main-session recovery: use persisted `deliveryContext`/last route metadata for restart-aborted main sessions, enqueue an idempotent recovery notice or routed continuation to the original Discord target, and add regression coverage for an interrupted `agent:main:discord:*` turn with no visible assistant follow-up. Coordinate with #38836 so the solution is generic across channel plugins rather than Discord-only.
