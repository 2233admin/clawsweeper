## Summary

Keep open. Current main still has the reported channel-visible recovery gap: Discord preserves enough routing metadata to know where the interrupted turn came from, but the abort path exits without posting a recovery notice, main-session restart recovery resumes internally with `deliver: false`, and the restart-sentinel notice path only runs when a sentinel file already exists. Related open reports show this should be handled generically rather than as a Discord-only patch.

## What This Changes

The issue asks OpenClaw to automatically resume or post a same-channel recovery notice when a gateway restart interrupts an in-flight Discord-backed turn.

## Best Possible Solution

Implement a generic post-restart channel-visible interruption recovery path: after startup identifies restart-aborted or recently abandoned channel-originated main sessions, resolve the persisted delivery context or last route, enqueue one safe same-channel recovery notice, and only run the existing internal resume path when the transcript tail is explicitly resumable. Keep the implementation channel-agnostic and reuse the restart-sentinel/outbound delivery machinery where practical.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: Discord stores route metadata, the aborted dispatch path returns silently, main-session restart recovery uses `deliver: false`, and sentinel notices require an existing sentinel file. No live Discord restart was run because this review was read-only.

## Solution Assessment

Yes, the best direction is a generic channel-visible recovery path rather than a Discord-only branch. That addresses this report and the related Feishu report while preserving the existing tail-safety checks for actual auto-resume.
