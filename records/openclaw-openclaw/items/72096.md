## Summary

Keep open. Current main still builds the light Dream Diary narrative from the fixed light lookback window and appends successful narrative text as a new timestamped diary entry; the existing diary dedupe intentionally keeps identical bodies when their timestamps differ. Open PR #70332 directly targets body-level diary dedupe, so the useful next step is maintainer review of that fix shape rather than closing or queuing a competing repair.

## What This Changes

The issue asks for light-phase Dream Diary work summaries to avoid repeating identical commit-list content across repeated overnight dreaming cycles, either by emitting only new inputs since the previous cycle or by deduping repeated diary output.

## Best Possible Solution

Land or rework a targeted `memory-core` change that prevents repeated automatic light Dream Diary material from being appended across repeated sweeps with unchanged input, preferably with an explicit input/body fingerprint or phase emission state plus regression coverage for repeated light sweeps and genuinely new snippets.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed daily or short-term recall material, run the light dreaming sweep twice within the same lookback window with a deterministic narrative/fallback, and observe that current main can append the same body at different timestamps.

## Solution Assessment

Unclear. The proposed `lastEmittedAt` cutoff is plausible for light input, while PR #70332's body-level dedupe may cover more diary sources; maintainers should choose the narrowest policy that suppresses repeated automatic output without hiding legitimately new narrative entries.
