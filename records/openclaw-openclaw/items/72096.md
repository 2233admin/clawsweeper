## Summary

Keep open. Current main still builds light Dream Diary content from the fixed light lookback window and appends successful narrative text as a new timestamped entry; there is no per-phase cutoff, input fingerprint, or body-level duplicate check. Open PR #70332 directly targets duplicate diary bodies, so maintainer review should choose that policy or a phase-state alternative before this can close.

## What This Changes

The issue asks memory-core light Dream Diary output to stop repeating identical work-summary blocks across repeated overnight cycles, either by emitting only new inputs or by deduping repeated output.

## Best Possible Solution

Land or rework a targeted `memory-core` change that prevents repeated automatic light Dream Diary material from being appended when repeated sweeps see unchanged input, with regression coverage for repeated light sweeps and for legitimately new snippets that should still append.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed daily or short-term recall material, run the light dreaming sweep twice inside the same lookback window with a deterministic narrative result, and observe that current main can append the same body at different timestamps.

## Solution Assessment

Unclear. The issue's `lastEmittedAt` proposal is plausible for light input, while PR #70332's body-level dedupe may cover more diary sources; maintainers should choose the narrowest policy that suppresses repeated automatic output without hiding valid recurring diary entries.
