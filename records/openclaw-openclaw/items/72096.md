## Summary

Keep open. Current main still selects light Dream Diary material from the configured lookback window and appends each successful narrative as a new timestamped diary entry; the existing dedupe path intentionally keeps the same body when timestamps differ. Open PR #70332 directly overlaps the duplicate diary output symptom, so the right next step is to review or rework that PR rather than close this report.

## What This Changes

The issue asks for light-phase Dream Diary work summaries to be incremental or duplicate-suppressed so identical content is not appended across scheduled dreaming cycles.

## Best Possible Solution

Land or rework a targeted memory-core change that prevents repeated automatic light Dream Diary material from being appended when the light input has not changed, with regression coverage for repeated light sweeps and for genuinely new snippets that should still produce a diary entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed stable daily or short-term recall material, run two light dreaming sweeps inside the same lookback window with deterministic narrative output, and observe that current main can append the same body at different timestamps; no test was run because this review is read-only.

## Solution Assessment

Unclear. The proposed lastEmittedAt cutoff is plausible for the light input path, while PR #70332's body-level dedupe may cover more diary sources; the maintainable fix should suppress unchanged automatic output without discarding distinct human-readable diary prose.
