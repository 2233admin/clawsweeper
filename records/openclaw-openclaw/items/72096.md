## Summary

Keep open. Current main still builds light Dream Diary input from the fixed configured lookback window and appends successful narrative text under a new timestamp without checking for a repeated body or a repeated phase input fingerprint. The existing diary dedupe intentionally keeps identical bodies when their timestamps differ, and open PR #70332 appears to target this remaining gap, so this should stay open for maintainer review rather than be closed or queued as duplicate repair work.

## What This Changes

The issue reports that light-phase Dream Diary work summaries repeat identical content across successive dreaming cycles because the work-summary input is not incremental or deduped across emitted entries.

## Best Possible Solution

The desired end state is a memory-core change that suppresses repeated automatic Dream Diary material across repeated light sweeps while preserving genuinely new material. A phase input/body fingerprint stored in dreaming state or checked during append, with regression coverage for repeated light sweeps and new-snippet appends, would address the observable bug without broad core changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to seed a daily/short-term recall entry, trigger light dreaming multiple times inside the same lookback window with deterministic narrative output, and observe that current main can append the same body under separate timestamps.

## Solution Assessment

No for current main. It intentionally preserves same-body entries at different timestamps; PR #70332 is a plausible narrow implementation candidate, but maintainers should decide whether body-level dedupe or a per-phase input/cutoff fingerprint is the safer policy before landing it.
