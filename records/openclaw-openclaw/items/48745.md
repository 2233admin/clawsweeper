## Summary

Keep this PR open. Current main still sends WhatsApp document fallbacks from the active plugin send path as bare `file`, and the current test suite still asserts that behavior. The PR remains useful, but it targets the old `src/web` layout and needs to be rebased/ported into `extensions/whatsapp` with the plugin-safe MIME helper.

## Best Possible Solution

Keep this PR open and ask for a rebase/port to the current WhatsApp plugin. The best implementation is a small shared WhatsApp document filename resolver near `extensions/whatsapp/src/outbound-media-contract.ts`, using `extensionForMime()` from `openclaw/plugin-sdk/media-mime`, applied to both `createWebSendApi` and auto-reply normalization, with tests for `file.pdf`, a dynamic case such as `file.csv`, and the chosen behavior for explicit extensionless filenames.
