## Summary

Keep this PR open. The current WhatsApp plugin still sends document fallback filenames as bare `file` and tests still assert that behavior, so the user-visible bug is not implemented on main. The PR is useful but targets the old `src/web` layout and needs to be ported to `extensions/whatsapp` with the plugin SDK MIME helper.

## Best Possible Solution

Keep this PR open and ask for a rebase/port to the current WhatsApp plugin. The best implementation is a small shared WhatsApp document filename resolver near `extensions/whatsapp/src/outbound-media-contract.ts`, using `extensionForMime()` from `openclaw/plugin-sdk/media-mime`, applied to both `createWebSendApi` and auto-reply normalization, with focused tests for `file.pdf`, a dynamic case such as `file.csv`, and the chosen behavior for explicit extensionless filenames.
