## Summary

Current main still does not implement the requested Feishu send-time cross-bot message broadcast. The bundled Feishu plugin supports multiple accounts and an inbound broadcast-to-agents path, but outbound Feishu sends and replies go directly to the Feishu send APIs without synthesizing peer-bot inbound events. The prior ClawSweeper keep-open review remains accurate against current main.

## Best Possible Solution

Keep this issue open. Implement a Feishu-owned opt-in send-time synthetic inbound fan-out after successful outbound group sends, covering reply-dispatcher replies, message-tool/direct sends, card/text/media paths as appropriate, origin metadata, account/group scoping, dedupe, loop limits, tests, and docs. Consider a generic channel/plugin seam only after Feishu plus another channel prove the same contract is needed.

## Risks / Open Questions

- A send-time synthetic bot-to-bot path can create reply loops and token/cost amplification unless it is opt-in, scoped by account/group, deduped, and guarded by consecutive-bot-turn limits reset by human messages.
- Existing Feishu inbound broadcast semantics intentionally give observer agents no-op reply dispatchers; an implementation should avoid accidentally making observer agents send Feishu replies or mutate sessions via commands.
- The same product need may exist in Telegram or other channels, but current evidence points to a bundled Feishu plugin gap first; a generic channel/plugin seam should be added only if multiple channel owners need the same API.
