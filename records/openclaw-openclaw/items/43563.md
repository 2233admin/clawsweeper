## Summary

Keep open. Current main still only supports Feishu broadcast while handling platform-delivered inbound group events; outbound Feishu text, cards, media, and final replies still send directly without a post-send synthetic peer-bot event, and there is no Feishu cross-bot opt-in config or loop-limit surface. The related inbound broadcast PR and broader Feishu lost-message issue are adjacent, not replacements for this canonical send-time cross-bot request.

## What This Changes

Request an opt-in Feishu cross-bot broadcast mechanism so messages sent by one Feishu bot in a shared group are synthesized as inbound events for other registered bots in that group.

## Best Possible Solution

Keep this issue as the canonical maintainer design item, then add a Feishu-owned opt-in send-time fan-out only if maintainers approve the bundled behavior: after successful outbound group sends, emit clearly marked synthetic inbound events for peer accounts in the same group, with config schema/help/docs, loop guards, dedupe, and focused tests; extract a shared seam only after another channel proves the same contract is needed.
