## Summary

Keep open. Current main still has the reported Control UI ordering race: the dropdown persists the selected model through an async `sessions.patch`, but the composer/send path and send controls have no shared pending model-switch state to await or disable on, and `chat.send` cannot carry a direct model override.

## What This Changes

The issue requests that Control UI model selection be persisted or awaited before the next chat send so an immediate send uses the newly selected model.

## Best Possible Solution

Add a Control UI model-switch barrier: track a per-session pending `sessions.patch`, disable the model picker and send/queue controls while it is pending, await it before dispatching chat sends, roll back the optimistic picker state and suppress the send if the patch fails, and keep the Gateway `chat.send` protocol unchanged unless maintainers choose a broader API design.

## Repair Work Prompt

Fix the Control UI model-selection race described by issue 54240. Implement a narrow UI-side barrier so a model selected from the chat header dropdown is persisted through `sessions.patch` before the next chat send can start. Track a per-session pending model-patch operation/status in shared Control UI state, have the model picker set/clear it around `switchChatModel`, disable the model picker and send/queue controls while the active session has a pending model switch, and make `handleSendChat` await the pending operation before normal `chat.send` dispatch. If the model patch fails, roll back the optimistic picker state as today, surface the error, and do not send the message that was waiting on that failed switch. Keep the Gateway `chat.send` protocol unchanged and do not add a one-turn model override field. Consider whether `/btw`, queued follow-ups, and steer sends should share the same barrier; document the chosen behavior in tests. Add regression coverage that leaves `sessions.patch` unresolved, immediately attempts a send, verifies `chat.send` is not called until the patch resolves, and verifies patch failure suppresses send and restores the picker state. This is UI runtime behavior, so add a changelog fix entry crediting @betamod.
