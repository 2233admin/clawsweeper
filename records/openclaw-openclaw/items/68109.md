## Summary

Keep open. Current main still routes direct Linux systemd gateway restarts through a synchronous `systemctl --user restart` call and then reports completion, with no detached or scheduled handoff that would let the invoking CLI survive when it is inside the managed service process tree. The linked fix PR is closed unmerged and its patch was not a safe canonical fix.

## What This Changes

The issue asks for successful `openclaw gateway restart` invocations on Linux/systemd to exit with code 0 instead of a null child-process exit result.

## Best Possible Solution

Add a narrow Linux/systemd restart handoff for direct gateway service restarts when the caller is at risk of being killed by the managed service restart, using the existing scheduled-restart result shape where appropriate while keeping normal synchronous failure reporting for external invocations.

## Repair Work Prompt

The observable bug is that a successful restart invoked from the managed systemd user-service context can leave the caller with child `exit=null` instead of a successful `exit=0`. Implement a narrow systemd restart handoff or equivalent exit-safe path for direct gateway service restarts when the caller is at risk of being killed by `systemctl --user restart`, using the existing `{ outcome: "scheduled" }` restart result shape where it fits. Preserve synchronous error reporting for external invocations and preserve non-zero behavior for real `systemctl` failures.
