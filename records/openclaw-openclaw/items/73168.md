## Summary

Keep this PR open, but it is not merge-ready. Current main still lacks the linked Android reverse-proxy transport-auth feature, while the latest PR head still has source-level credential-scoping bugs in the canvas WebView paths and a security-sensitive TLS bypass that needs maintainer review.

## What This Changes

The branch adds Android manual/onboarding inputs and SecurePrefs persistence for bearer/basic reverse-proxy credentials plus a TLS-verification bypass, threads those credentials into gateway WebSocket and canvas loading paths, and adds focused Android tests.

## Best Possible Solution

Revise the Android implementation so proxy Authorization is persisted but only applied to the gateway WebSocket upgrade and strictly same-origin canvas HTTP(S) requests, with scheme/host/port checked by a shared production helper and covered by tests. Remove the TLS bypass from this PR, or split it behind an explicitly approved, warning-heavy development-only design before landing the feature for #51872.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure bearer/basic proxy auth, trigger canvas.navigate with an external HTTPS URL, and observe that the PR's wv.loadUrl(currentUrl, headers) path would attach Authorization; a second source-level path is an HTTPS gateway canvas page loading http://same-host:same-port, which passes the authority-only interceptor check.

## Solution Assessment

No. The feature direction matches the linked reverse-proxy request and the latest head appears to fix the earlier persistence and duplicate-header issues, but this is not the safest implementation until credential headers are scoped to the exact trusted origin and the TLS bypass is removed or separately approved.

- **[P1] Scope canvas load headers to the gateway origin:** `apps/android/app/src/main/java/ai/openclaw/app/node/CanvasController.kt:166`

## Security Review

Summary: Security review needs attention because the diff can leak reverse-proxy credentials from Android canvas navigation and adds a persistent full TLS validation bypass.

Concerns:

- **[high] Top-level canvas navigation can leak proxy credentials:** `apps/android/app/src/main/java/ai/openclaw/app/node/CanvasController.kt:166`
  - body: The PR applies bearer/basic `Authorization` headers to every WebView `loadUrl` canvas navigation, while canvas navigation accepts arbitrary URLs; this can disclose reverse-proxy credentials to a non-gateway origin.
