## Summary

Keep this PR open. Current main still lacks Android transport-level bearer/basic proxy auth, so the contribution is relevant, but the branch is not merge-ready because the new proxy credentials are not persisted from either Android connection flow, canvas request interception can leak the Authorization header to off-origin resources, and the PR adds a persisted TLS verification bypass that needs explicit maintainer/security review.

## What This Changes

The PR adds Android manual/onboarding fields for bearer token and HTTP basic reverse-proxy auth, forwards those credentials on gateway WebSocket and canvas requests, stores related preferences, and adds a Disable TLS Verification option with TLS fingerprint mismatch handling.

## Best Possible Solution

Keep the Android reverse-proxy auth work in this PR only after it persists credentials consistently, scopes canvas Authorization headers to the intended gateway origin, and either removes or deliberately hardens the TLS verification bypass with visible runtime safeguards; then decide separately whether #51872 needs follow-ups for iOS, setup-code/QR payloads, custom headers/cookies, and OIDC.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: enter bearer/basic proxy credentials in either Android connection flow, connect once, then restart or auto-reconnect; NodeRuntime reloads these values from SecurePrefs, but the screens never save them, so the proxy auth fields resolve null.

## Solution Assessment

No. The high-level Android direction is useful, but this implementation is not the narrowest safe solution until credential persistence is fixed, WebView auth forwarding is origin-scoped, CI is green, and the TLS bypass is explicitly accepted or redesigned.

Overall confidence: 0.9

Full review comments:

- **[P1] Persist proxy credentials before connecting:** `apps/android/app/src/main/java/ai/openclaw/app/ui/ConnectTabScreen.kt:315-317`
  - body: The Connect tab passes bearer/basic values into the immediate `connect()` call but never saves them with the new SecurePrefs setters. After app restart or any reconnect path that calls `resolveGatewayConnectAuth()`, the reverse-proxy credentials are gone and the connection fails until the user re-enters them.
  - confidence: 0.91
- **[P1] Persist onboarding proxy credentials before connecting:** `apps/android/app/src/main/java/ai/openclaw/app/ui/OnboardingFlow.kt:937-939`
  - body: The onboarding path has the same persistence gap: it trims the bearer/basic fields and passes them to `connect()`, but only persists the gateway password. Completing onboarding behind a proxy will work only for the current attempt, then lose the proxy credentials on the next launch/reconnect.
  - confidence: 0.9
- **[P1] Scope canvas auth headers to the gateway origin:** `apps/android/app/src/main/java/ai/openclaw/app/ui/CanvasScreen.kt:113-120`
  - body: `shouldInterceptRequest` runs for every HTTP(S) resource the WebView loads and this code adds the proxy Authorization header without checking the request origin. A canvas page that loads an off-origin image, script, or fetch would send the user's bearer/basic proxy credential to that host; restrict this to the gateway origin or another explicit allowlist.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The PR introduces security-sensitive transport credential handling and a TLS verification bypass, and the current diff has a concrete credential-leak path in WebView interception.
