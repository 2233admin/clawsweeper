## Summary

Keep open. Current main still drops blank user-role session entries during session-file repair, and the existing unit test explicitly expects that destructive behavior. The related reports are adjacent, but none is a merged fix for this specific persisted-session repair path.

## What This Changes

The issue asks OpenClaw to preserve or rewrite blank user-role session messages during session-file repair so strict OpenAI-compatible chat templates still receive at least one user message.

## Best Possible Solution

Change session-file repair so blank-only user entries are rewritten to a non-empty neutral placeholder instead of removed, preserve the existing media-block cleanup behavior, update transcript hygiene docs if needed, and add regression coverage for a session that would otherwise repair to assistant-only history.

## Reproduction Assessment

Yes. A focused fixture with a session header, one blank user entry, and assistant entries exercises the current `repairSessionFileIfNeeded` path and shows the only user entry being removed; the live Qwen/mlx-vlm provider confirms the downstream strict-template failure but is not required to reproduce the source bug.

## Solution Assessment

Yes. The narrowest maintainable fix is in `src/agents/session-file-repair.ts` with matching tests; a provider-specific OpenAI transport guard should only be added if a focused test shows file repair is insufficient.
