## Summary

Keep open. Current main still drops blank-only persisted user messages during session-file repair, and the focused unit test explicitly expects that destructive behavior; the related reports are adjacent replay/diagnostic failures rather than a merged fix for this persisted-session repair path.

## What This Changes

The issue asks for session-file repair to rewrite blank-only user-role messages to a neutral non-empty placeholder instead of dropping them, so strict OpenAI-compatible providers do not see assistant-only replay history.

## Best Possible Solution

Update session-file repair so blank-only persisted user entries are rewritten to a neutral non-empty placeholder while preserving existing media-block cleanup, then add regression coverage for a repaired session that would otherwise become assistant-only history.

## Reproduction Assessment

Yes. A focused JSONL fixture with a session header, one blank user entry, and assistant entries exercises `repairSessionFileIfNeeded`; current source and tests show that the only user entry is removed and logged as dropped.

## Solution Assessment

Yes for the persisted-session bug. The narrowest maintainable fix is in `src/agents/session-file-repair.ts` with focused tests; a provider-specific transport guard should only be added if a targeted regression proves file repair and replay hygiene are insufficient.
