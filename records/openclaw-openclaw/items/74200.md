## Summary

Keep this PR open. Current main still wipes the sandbox target skills directory before recopying eligible skills, so the PR is not obsolete, but the proposed incremental sync is not merge-ready because it can retain deleted files and preserve stale or no-longer-eligible skill directories in sandbox workspaces.

## What This Changes

The PR changes `syncSkillsToWorkspace` from wholesale target directory deletion to incremental per-skill syncing with mtime-based skip logic and adds regression tests for preservation and recopy behavior.

## Best Possible Solution

Replace the wholesale wipe with an authoritative owned-sync strategy: build a successful eligible manifest, preserve user-owned target directories only if that is an intentional product contract, prune previously synced owned skills that are absent from the effective manifest, and use full-tree copy semantics or a content manifest rather than a shallow newest-mtime heuristic.

## Reproduction Assessment

Yes. A code-level reproduction is clear: sync a skill once, then remove a source file or remove the skill from the effective allowlist and sync again; the proposed branch can leave the old file or directory in the sandbox target. I did not execute a filesystem test because this review was required to stay read-only.

## Solution Assessment

No. The PR addresses a real data-loss class, but preserving all non-manifest target directories and skipping by newest mtime is not the narrowest maintainable fix for sandbox skill staging; an owned manifest or marker-based sync would be safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Prune stale skills when the allowed manifest changes:** `src/agents/skills/workspace.ts:968-976`
  - body: Replacing the target wipe with `mkdir` and never removing non-manifest directories means a sandbox that previously synced a skill can keep it after agent allowlists, config gates, plugin eligibility, or remote eligibility remove that skill. Current tests and docs treat the target as the effective eligible set, so the sync needs an owned-manifest cleanup path instead of preserving all prior target dirs.
  - confidence: 0.87
- **[P2] Do not skip copies using newest mtime alone:** `src/agents/skills/workspace.ts:1017-1019`
  - body: The `destMtime >= srcMtime` shortcut misses source deletions: after copying files A and B, deleting newer file B lowers the source newest mtime while the destination still has B, so the next sync skips and B remains indefinitely. Use full-tree content/manifest comparison or recopy the owned skill directory when the tree shape changes.
  - confidence: 0.92
