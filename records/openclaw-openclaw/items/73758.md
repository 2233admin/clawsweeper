## Summary

Keep open. Current main still has the reported short tagged-TTS failure path: hidden `[[tts:text]]hello[[/tts:text]]` is captured and stripped, then the TTS runtime returns before synthesis because the spoken text is under 10 characters. The open closing PR #73911 already targets the narrow fix, so this issue should stay open until that PR lands or is replaced.

## What This Changes

The issue requests restoring tagged TTS synthesis for claude-cli final replies when `[[audio_as_voice]]` and hidden `[[tts:text]]...[[/tts:text]]` content produce a voice-only Telegram payload.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that lets non-empty explicit `directives.ttsText` bypass the generic short-text suppression, while preserving the short untagged auto-TTS guard and adding a regression test for `[[tts:text]]hello[[/tts:text]]`.

## Reproduction Assessment

Yes. A final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a source-visible path where hidden text is captured, visible text is stripped, the runtime returns before synthesis because the speech text is under 10 characters, and Telegram skips the empty voice-only payload.

## Solution Assessment

Yes. The linked PR direction is the narrowest maintainable solution because it changes only explicit tagged speech semantics and does not require changing provider credentials, directive parsing, Telegram delivery policy, or untagged auto-TTS noise suppression.
