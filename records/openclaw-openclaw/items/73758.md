## Summary

Keep open. Current main has generic TTS dispatch for final replies, but the exact reported short hidden `[[tts:text]]hello[[/tts:text]]` + `[[audio_as_voice]]` shape can still become an `audioAsVoice` payload with no text or media, which Telegram then skips. I found no current-main regression coverage for the claude-cli final-reply path that proves this case is fixed.

## Best Possible Solution

Keep the issue open and fix the TTS fallback semantics in the final-reply path. A good fix would add a regression test for a claude-cli-style final payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`, then either synthesize tagged short hidden text or preserve a visible fallback and clear `audioAsVoice` so Telegram never receives an empty voice-only payload.
