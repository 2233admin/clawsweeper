## Summary

Keep open. Current main still short-circuits explicitly tagged short TTS text before synthesis, which leaves the hidden text stripped and can produce the reported empty voice-only Telegram payload. An open PR, #73911, already targets this exact fix with closing syntax, so the issue should remain open until that PR or an equivalent change lands.

## What This Changes

The issue requests that explicit `[[tts:text]]...[[/tts:text]]` plus `[[audio_as_voice]]` replies synthesize and deliver voice messages on the claude-cli/Telegram path instead of being silently skipped.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as user opt-in to synthesis even below the generic short-text threshold, while preserving short-text suppression for untagged automatic TTS and adding regression coverage for a short hidden tagged block.

## Reproduction Assessment

Yes. A final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a source-visible path where hidden text is captured and stripped, the TTS runtime returns before synthesis because the speech text is under 10 characters, and Telegram then skips an empty voice-only payload.

## Solution Assessment

Yes, the current open PR direction is the narrowest maintainable fix: adjust the TTS runtime semantics for explicitly tagged hidden speech text and add focused coverage, without changing provider setup, Telegram delivery policy, or untagged auto-TTS noise suppression.
