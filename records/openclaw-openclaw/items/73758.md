## Summary

Keep open. Current main still has the reported source-visible failure for short explicit tagged TTS text, and the issue is already paired with open closing PR #73911, so it should remain open until that PR lands or is replaced.

## What This Changes

The issue reports that short tagged TTS directives such as `[[tts:text]]hello[[/tts:text]]` are parsed but not synthesized, leaving Telegram with an empty voice-only reply to skip.

## Best Possible Solution

Land #73911 or an equivalent narrow fix that treats non-empty explicit `directives.ttsText` as an opt-in to synthesis even below the generic short-text threshold, while preserving untagged short auto-TTS suppression and adding the missing short tagged regression case.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. A final payload containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows current main through directive capture and visible-text stripping, then returns before synthesis because the spoken text is below the 10-character guard.

## Solution Assessment

Is this the best way to solve the issue? Yes. The linked PR direction is the narrowest maintainable fix because it changes only explicit tagged-speech short-text semantics and leaves provider config, directive parsing, Telegram delivery policy, and untagged short-text suppression intact.
