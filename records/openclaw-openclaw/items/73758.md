## Summary

Keep this issue open. Current main has added a generic final-reply TTS hook, so the original hypothesis is partly improved, but the reported short explicit tagged payload still returns before synthesis: `[[tts:text]]hello[[/tts:text]]` is stripped from visible text, the hidden TTS text is under the 10-character guard, `audioAsVoice` can remain on the payload, and Telegram still skips an empty voice-only reply.

## Best Possible Solution

Keep this issue open. Add a regression test for a final claude-cli-style payload like `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]`, then update TTS fallback semantics so explicit hidden tagged text is synthesized even when short, or so skipped synthesis clears `audioAsVoice` and preserves a visible fallback before Telegram delivery. Preserve the intentional short-text guard for untagged or automatic inbound TTS unless maintainers explicitly change that product behavior.
