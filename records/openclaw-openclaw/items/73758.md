## Summary

Keep open. Current main still strips the hidden tagged TTS text and then skips synthesis for text under the 10-character threshold, which can leave Telegram with an empty voice-only payload. The open, unmerged PR #73911 already targets this exact behavior with closing syntax, so the issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for tagged TTS directives, especially short hidden `[[tts:text]]` blocks combined with voice delivery, to synthesize and deliver audio on the claude-cli final-reply path instead of disappearing.

## Best Possible Solution

Land #73911 or an equivalent narrow change that treats non-empty explicit `directives.ttsText` as user opt-in to synthesis even below the generic short-text threshold, while preserving short untagged auto-TTS suppression and adding regression coverage for short hidden tagged text.

## Reproduction Assessment

Yes. A final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` follows a source-visible path where hidden text is captured and stripped, the TTS runtime returns before synthesis because the speech text is under 10 characters, and Telegram skips the resulting empty voice-only payload.

## Solution Assessment

Yes. The narrowest maintainable fix is to bypass the short-text guard only for explicit non-empty tagged speech text, leaving provider setup, Telegram delivery policy, and untagged short auto-TTS behavior unchanged.
