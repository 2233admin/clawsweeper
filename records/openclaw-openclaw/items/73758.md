## Summary

Keep open. Current main still strips the hidden TTS text and returns before synthesis when the explicit `ttsText` is under the 10-character guard, which can leave Telegram with an empty `audioAsVoice` payload. The provided context also shows #73911 is open, unmerged, and uses closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Fix tagged TTS voice delivery so explicit `[[tts:text]]...[[/tts:text]]` blocks, including short text, synthesize and deliver voice notes on claude-cli/Telegram instead of being stripped and skipped.

## Best Possible Solution

Review and land #73911 or an equivalent narrow change that treats non-empty explicit `directives.ttsText` as user opt-in to synthesis below the generic short-text threshold, with regression coverage for a short hidden tagged block and no changes to provider setup or Telegram skip policy.

## Reproduction Assessment

Yes. A final reply containing `[[audio_as_voice]] [[tts:text]]hello[[/tts:text]]` in tagged TTS mode follows a source-visible path where hidden text is captured and stripped, synthesis returns early because the speech text is under 10 characters, and Telegram skips the empty voice-only payload.

## Solution Assessment

Yes. The narrow PR direction is the best maintainable fix: bypass the short-text guard only for explicit non-empty tagged speech text, while keeping the noise-control guard for untagged automatic TTS replies.
