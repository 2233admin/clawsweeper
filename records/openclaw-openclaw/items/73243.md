## Summary

Keep open. Current main still has warning-only stuck-session handling without `diagnostics.stuckSessionAbortMs`, and this PR is a relevant implementation candidate, but the PR head still has blocking correctness gaps around false recovery results and abort telemetry ordering; it also misses required public docs/changelog updates for the new config and default behavior.

## What This Changes

This PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-driven stuck-session abort/recovery, `session.aborted` diagnostics and OTel recording, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a corrected version of this PR that aborts or releases stuck sessions only after the configured threshold, keeps diagnostics in `processing` and retries when recovery reports no actual release, emits `session.aborted` only after confirming the stale state is still current and recovery succeeded, and updates the public config docs plus changelog while keeping #71127 open until merge.

## Reproduction Assessment

Yes. Current main can be checked by starting the diagnostics heartbeat with a session in `processing` past `diagnostics.stuckSessionWarnMs`: it emits `session.stuck` and calls the conservative recovery path, but there is no abort threshold or public `stuckSessionAbortMs` config. The PR adds fake-timer coverage for the main abort path, but it still needs coverage for recovery returning false and for suppressing abort telemetry after natural recovery.

## Solution Assessment

No, not as-is. The implementation direction is narrow and relevant, but the PR should avoid marking a session idle when no run/lane was recovered, move abort telemetry behind the latest-state guard, and update docs/changelog before it is the maintainable fix.
