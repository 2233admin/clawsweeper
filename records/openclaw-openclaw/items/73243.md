## Summary

Keep open. The PR targets a real gateway recovery gap that is not implemented on current main, but the proposed abort-completion path can mark sessions idle and emit abort telemetry even when recovery skipped or the stale state is no longer current. The linked bug remains open and this should stay in normal maintainer review rather than be auto-closed.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-triggered stuck-session abort/recovery, `session.aborted` diagnostics and OTel recording, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a revised version that aborts confirmed stale embedded and reply runs after a maintainer-approved threshold, only emits `session.aborted` and idles diagnostic state after the latest-state guard confirms the stale processing state is still current and recovery actually happened, and updates docs/changelog/config surfaces before the linked bug is closed.

## Reproduction Assessment

Yes. A focused fake-timer heartbeat test with an injected `recoverStuckSession` that resolves `false` or resolves after a natural idle/new-processing transition exercises the two introduced failure modes without needing live gateway traffic.

## Solution Assessment

No as-is. The overall direction is narrow and relevant, but the completion branch needs stronger success/state guards and the recovery runtime needs to abort reply-run work on the stuck-timeout path before this is the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard abort completion before logging or idling:** `src/logging/diagnostic.ts:763-776`
  - body: `recoverStuckSession` can resolve after the stale state has changed, or resolve `false` when recovery skipped active reply/lane work. This branch emits `session.aborted` before the guard and then idles the session regardless of `false`, which can hide live work and stop future abort attempts. Check the latest state and a confirmed recovery before logging/idling, and clear the retry flag otherwise.
  - confidence: 0.91
- **[P2] Abort reply runs on the stuck-timeout path:** `src/logging/diagnostic-stuck-session-recovery.runtime.ts:74-80`
  - body: When there is active reply work but no embedded handle, `resolveActiveEmbeddedRunSessionId` can identify that reply run, yet this block returns `false` before `abortAndDrainEmbeddedPiRun` can call the reply-run abort path. That leaves one of the PR's target stuck-run kinds un-aborted while the heartbeat may still mark it recovered.
  - confidence: 0.86
- **[P3] Align docs with the default abort threshold:** `src/logging/diagnostic.ts:324-335`
  - body: The resolver falls back to a 15 minute abort threshold when config is absent, but the public agent-loop docs still say active embedded runs, reply operations, and lane tasks remain warning-only by default. Update the docs/changelog for the new default behavior or make the abort threshold opt-in.
  - confidence: 0.82
