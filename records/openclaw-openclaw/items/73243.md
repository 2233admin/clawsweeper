## Summary

Keep this PR open. Current main still only warns on stuck processing sessions and does not implement the requested diagnostic recovery path, while the linked #71127 remains open with production and deterministic reproduction context. The PR is a relevant implementation candidate, but its current head still has unresolved async recovery hazards noted in review: a recovery failure can block retries, and a late `.then` can clobber a session that naturally recovered and started new work.

## Best Possible Solution

Keep the PR open as the implementation candidate for #71127. Update the recovery path to capture and guard the abort timestamp, clear or retry after recovery errors, avoid clobbering sessions that have naturally transitioned, and add focused tests for successful recovery, recovery failure retry, and natural-recovery race behavior. Then align the config/docs/telemetry surfaces for `diagnostics.stuckSessionAbortMs` and run the changed gateway/diagnostics checks before merge.

## Risks / Open Questions

- Merging the current PR head could preserve the original indefinite-block failure mode if `recoverStuckSession` throws, because retries are skipped while `stuckAbortRequestedAt` remains set.
- The async `.then` path can overwrite diagnostic state after natural recovery and a new run, which could silently mark live work idle and drop queue depth.
- Automatic diagnostic aborts need conservative threshold semantics and regression coverage so legitimate long-running model/tool turns are not canceled unexpectedly.
