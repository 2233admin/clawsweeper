## Summary

Keep this PR open, but the current patch should not merge as-is. Current main still lacks the configurable active stuck-session abort behavior this PR targets, but the PR can mark diagnostics idle and emit abort telemetry even when recovery did not actually happen or when the session has already naturally moved on.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, `session.aborted` diagnostics and OpenTelemetry recording, heartbeat-driven stuck-session abort/recovery wiring, config reload/schema updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a focused stuck-session abort fix that builds on current main's recovery runtime, only emits `session.aborted` and marks diagnostics idle after a guarded successful recovery, documents the new default behavior, and validates the diagnostics/recovery/config surfaces before closing #71127.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Only mark idle after successful recovery:** `src/logging/diagnostic.ts:772-776`
  - body: `recoverStuckSession` can resolve `false` when recovery skipped active reply work, active lane work, duplicate in-flight recovery, or failed to release anything. This handler still emits `session.aborted` and clears the diagnostic state to `idle`, which can hide a still-stuck session from later heartbeat recovery attempts. Gate the idle transition on `recovered === true` or explicitly handle retryable false results.
  - confidence: 0.9
- **[P2] Move abort telemetry behind the state guard:** `src/logging/diagnostic.ts:763`
  - body: The promise handler emits `session.aborted` before checking that the captured stuck state is still current. In the natural-recovery-then-new-work race, the guard returns without changing state, but telemetry has already recorded an abort for healthy new work. Move `logSessionAbortRequested` after the latest-state guard.
  - confidence: 0.86
