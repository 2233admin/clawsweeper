## Summary

Keep open. Current main still only warns and conservatively releases stale lanes; this PR is the active implementation candidate, but the PR head has blocking correctness issues in abort completion and reply-run recovery, plus default active-abort policy/docs need maintainer review before merge.

## What This Changes

The PR adds `diagnostics.stuckSessionAbortMs`, heartbeat-triggered stuck-session abort/recovery, `session.aborted` diagnostics and OTel recording, config schema/reload updates, and focused diagnostics/config tests.

## Best Possible Solution

Land a revised version that aborts confirmed stale embedded and reply runs after a maintainer-approved threshold, emits abort telemetry only after confirming stale state and actual recovery, and updates public config/docs/changelog surfaces before the linked bug closes.

## Reproduction Assessment

Yes. A focused fake-timer heartbeat test with an injected `recoverStuckSession` can reproduce current-main warning-only behavior and the PR’s late-recovery/false-recovery failure modes without live gateway traffic.

## Solution Assessment

No as-is. The implementation direction is relevant, but the branch should guard abort completion on both latest state and confirmed recovery, cover reply-run aborts, and align docs/changelog before it is the safest fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Guard abort completion before logging or idling:** `src/logging/diagnostic.ts:763-776`
  - body: `recoverStuckSession` can resolve after the stale state has changed, or resolve `false` when recovery skipped active work. This branch emits `session.aborted` before the guard and then idles the session regardless of recovery success, which can misreport healthy work and stop later recovery attempts. Move logging/idling behind the latest-state and recovered checks, and clear the retry flag otherwise.
  - confidence: 0.91
- **[P2] Abort reply runs on the stuck-timeout path:** `src/logging/diagnostic-stuck-session-recovery.runtime.ts:74-80`
  - body: When active reply work exists without an embedded handle, `resolveActiveEmbeddedRunSessionId` identifies the reply run but this block returns `false` before `abortEmbeddedPiRun` can abort it by session id. That leaves one of the PR’s stated stuck-run kinds un-aborted while the heartbeat can still mark diagnostic state recovered.
  - confidence: 0.86
- **[P3] Document the default abort threshold:** `src/logging/diagnostic.ts:324-335`
  - body: The resolver defaults absent config to a 15 minute abort threshold, but the public agent-loop docs still say active embedded runs, reply operations, and lane tasks remain warning-only by default, and the changelog has no user-facing entry. Update the docs/changelog for the new behavior or make the abort threshold opt-in.
  - confidence: 0.82
