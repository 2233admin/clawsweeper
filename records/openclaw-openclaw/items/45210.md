## Summary

Keep open. Current main still has global session maintenance plus transient active-key preservation; it does not provide durable per-session/topic pins, command-managed pin state, or documented retention rules that exempt selected sessions from both age pruning and count capping.

## What This Changes

The issue requests a durable pin/protect mechanism so selected session keys or Telegram topics are excluded from `session.maintenance.pruneAfter` and `maxEntries` cleanup.

## Best Possible Solution

Define one maintainer-approved retention-protection contract that can exempt selected session keys, topics, or matchers from stale pruning and count capping, then align config types, strict/generated schemas, cleanup call sites, CLI/docs, and regression tests with that contract.
