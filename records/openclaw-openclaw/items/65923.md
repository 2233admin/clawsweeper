## Summary

Current main still matches the report: omitted `session.reset` resolves to an implicit `daily` reset at 4 AM, the auto-reply session path still evaluates that policy and rolls stale sessions, and tests intentionally lock that behavior. The issue should stay open because changing unset semantics versus making the implicit default highly visible is a product/compatibility decision, and the broader #61238 discussion overlaps but does not supersede this narrower unset-default report.

## Best Possible Solution

Keep this open as a focused product/UX bug. Maintainers should decide the contract for omitted `session.reset`: either add a supported manual-only/no-reset mode and update types, schema, runtime, tests, docs, and config surfaces, or deliberately keep the daily default and make it visible in config help, FAQ, setup/status output, and startup/operator warnings. Coordinate with #61238, but do not treat that broader issue as closing this narrower unset-default report.
