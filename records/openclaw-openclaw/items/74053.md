## Summary

Keep this PR open: current main still builds heartbeat runtime context with `ctx.To` from `sender`, so the branch's requested context hardening is not implemented on main. The runtime change is narrow and directionally correct, but the added regression test does not actually force the `heartbeat` sender-sentinel case and should be tightened before merge.

## What This Changes

The PR changes heartbeat runtime context so `ctx.To` comes from the resolved delivery target, adds a target-none regression test for preserving session `lastTo`, and adds an Unreleased changelog entry.

## Best Possible Solution

Land the narrow `ctx.To = delivery.to` hardening after strengthening the regression coverage to exercise a true no-candidate sender sentinel and assert the exact expected recipient shape. Keep the existing session-level preservation guard because it protects shared-session routing even if future automated contexts carry surprising values.

## Reproduction Assessment

Unclear for the historical session-store corruption on current main: source inspection verifies the `ctx.To = sender` condition, but current `initSessionState` preserves `lastTo` for heartbeat/cron/exec providers. The PR's added test does not reproduce the sentinel path because its seeded `lastTo` prevents `sender` from resolving to `heartbeat`.

## Solution Assessment

Yes, deriving `ctx.To` from `delivery.to` is the narrowest runtime hardening for this context leak while preserving `From` as the synthetic self-event identity. The safer merge path is to pair that code change with a regression test that actually drives `sender` to the sentinel and expects `ctx.To` to be undefined.
