## Summary

Keep open. Current main still lacks the requested dreaming stagger configuration, and this PR is the active implementation candidate, but the latest branch still has a real config-removal defect: after an operator removes `dreaming.staggerMs`, the managed cron job can keep the old explicit stagger instead of returning to unset/default behavior.

## What This Changes

The PR adds a `dreaming.staggerMs` manifest/schema field, parses it in the memory host SDK, and passes/logs it on the memory-core managed dreaming cron schedule.

## Best Possible Solution

Expose `dreaming.staggerMs` as a documented memory-core setting whose create, update, explicit `0`, unset, and removal semantics are all predictable, with focused tests covering config propagation and clearing a previously configured stagger.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear stale stagger when config is removed:** `extensions/memory-core/src/dreaming.ts:294`
  - body: When an operator removes `dreaming.staggerMs`, the desired schedule omits the field and this guard skips the update even if the stored managed cron job still has the old explicit stagger. Since `applyJobPatch` preserves an existing cron `staggerMs` when a patch omits it, the job keeps using stale jitter instead of returning to the unset/default behavior described by the PR.
