## Summary

Keep open. Current main still has the Memory Wiki bridge and status paths reading public artifacts through module-local memory capability state, and the runtime mirror boundary implicated by the report still exists. The issue also has a same-author draft PR already open for the narrow fix, so it should remain open while that PR is reviewed rather than being closed by cleanup.

## What This Changes

The issue reports that Memory Wiki bridge/status return zero memory-core artifacts in packaged installs because canonical dist and managed plugin-runtime-deps mirror code observe separate memory-state singletons.

## Best Possible Solution

Land a reviewed fix that makes the memory capability registry canonical across duplicate runtime module instances, while preserving the existing plugin SDK API and adding regression coverage for mirrored runtime loading plus Memory Wiki bridge/status artifact counts.

## Reproduction Assessment

Yes. The issue provides concrete packaged Gateway steps, instrumentation showing two `memory-state` module instances, and a second same-version confirmation; current main still has the same module-local read/write paths.

## Solution Assessment

Unclear. The issue’s suggested directions are plausible, but the best maintainable solution should be validated in the paired draft PR: a shared or host-owned memory capability seam with mirror regression tests, not patching generated runtime-deps output.
