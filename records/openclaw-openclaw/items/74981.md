## Summary

Keep open. Current main still has Memory Wiki bridge/status artifact reads going through module-local memory capability state, while bundled plugin loading can remap plugin code into a staged runtime mirror. The provided second reproduction and the open same-author draft PR #75064 make this an active, source-supported bug rather than an item to close as already fixed by the earlier CLI-to-Gateway routing work.

## What This Changes

The issue reports that Memory Wiki bridge/status artifact reads return zero on packaged 2026.4.27 because memory-core registers publicArtifacts in one memory-state module instance while the mirrored plugin-runtime-deps path reads a separate empty memory-state singleton.

## Best Possible Solution

Keep this open until a reviewed fix lands that makes the memory capability registry canonical across duplicate runtime module instances, with regression coverage for Memory Wiki bridge import and status under mirrored bundled-runtime loading.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps and instrumentation, another user confirmed the same version/setup, and current main still has the module-local producer/consumer state paths needed for the split-singleton failure.

## Solution Assessment

Unclear until #75064 is reviewed. The right end state is a narrow shared or host-owned memory capability seam with mirrored-runtime tests, not patching generated plugin-runtime-deps files or relying on the earlier CLI routing fix alone.
