## Summary

Keep open. Current main still has Memory Wiki bridge/status artifact reads depending on the module-local memory capability state, and the reported plugin-runtime-deps mirror boundary can still create a separate empty module instance. The same author has draft PR #75064 open for this exact split-state problem, so the issue should stay open while that paired implementation is reviewed and validated.

## What This Changes

The issue requests that Memory Wiki bridge/status artifact reads see the active memory-core publicArtifacts capability across canonical and plugin-runtime-deps mirrored runtime module instances.

## Best Possible Solution

Land or replace #75064 with a narrow shared memory capability registry that works across duplicated runtime module instances, with regression coverage for Memory Wiki bridge import and status under mirrored runtime loading. Avoid patching generated plugin-runtime-deps output directly.

## Reproduction Assessment

Yes. The issue has concrete packaged Gateway steps, direct instrumentation of the two module instances, an independent confirmation comment, and current main still contains the module-local state and bridge/status read paths needed for the failure.

## Solution Assessment

Unclear until #75064 is reviewed. The maintainable direction is a shared or host-owned memory capability seam plus mirror regression tests; the current main implementation is not sufficient, and generated runtime mirror patching would be fragile.
