## Summary

Keep open. Current main still has Memory Wiki bridge import and status reading public artifacts through the `openclaw/plugin-sdk/memory-host-core` module singleton, while Memory Core registers `publicArtifacts` into `src/plugins/memory-state.ts`; the packaged runtime-deps mirror still copies root dist chunks and builds a mirrored SDK alias, so the reported split singleton remains source-supported and distinct from the earlier CLI-to-Gateway routing fix.

## What This Changes

The issue requests fixing Memory Wiki bridge/status artifact reads so packaged `plugin-runtime-deps` Gateway code sees `memory-core.publicArtifacts` instead of an empty mirrored `memory-state` singleton.

## Best Possible Solution

Route Memory Wiki bridge artifact reads through a canonical host-provided memory capability accessor, or otherwise guarantee that mirrored bundled plugin code resolves the same memory capability state as the Gateway host, then add packaged mirror regression coverage proving `wiki.bridge.import` sees artifacts registered by `memory-core`. Keep the broader dependency-retention work in #74777 separate unless the same targeted change naturally fixes both surfaces.

## Reproduction Assessment

Yes. The issue gives concrete packaged Gateway steps on v2026.4.27, another user confirmed the same result, and current main still has the exact module-instance-sensitive read, registration, and mirror paths that explain the zero-artifact outcome.

## Solution Assessment

Unclear which implementation is best, but the current direct SDK singleton read from `memory-wiki` is not robust across packaged runtime mirrors. A host-injected accessor or narrowly shared memory-capability seam is safer than patching generated mirror files or globally externalizing unrelated core chunks.
