## Summary

Keep open. Current main still lacks the requested `<notts>` visual-only TTS behavior in the active speech-core runtime, but this PR should not merge as-is because it targets the old `src/tts` surface and mixes the TTS feature with unrelated gateway authorization changes that create a concrete node-role scope bypass risk.

## What This Changes

The PR adds `<notts>` TTS preprocessing, tests, docs, and system-prompt hints, while also changing gateway node authorization, macOS speech recognition hints, media transcription conversion, MIME handling, and `audioAsVoice` propagation.

## Best Possible Solution

Split the work: implement `<notts>` support in `extensions/speech-core/src/tts.ts` with focused runtime/contract tests and `docs/tools/tts.md` updates, leave side changes already present on main out of the branch, and handle node-role UI access as a separate security-reviewed gateway design.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Rebase preprocessing onto speech-core:** `src/tts/tts.ts:42`
  - body: Current main routes TTS through the `speech-core` runtime via the plugin-sdk facade, while this branch wires preprocessing into the old `src/tts` implementation. After rebase, the new `<notts>` preprocessing would miss the runtime path where auto-TTS actually runs.
  - confidence: 0.9
- **[P1] Keep node role from bypassing operator scopes:** `src/gateway/role-policy.ts:21-23`
  - body: Adding UI/read/write methods to the node-allowed set makes them callable by node-role clients, and gateway authorization returns before operator-scope checks for role `node`. That grants paired node clients scope-free access to methods such as `chat.send` and `config.get`.
  - confidence: 0.88
- **[P2] Handle notts in the shell filter:** `scripts/tts-filter.sh:22-44`
  - body: The added `tts-filter.sh` only strips or unwraps `<tts>` tags. A `<notts>...</notts>` block would show raw tags in text output and be spoken in voice output, contradicting the visual-only contract.
  - confidence: 0.92
- **[P2] Respect disabled tag processing in tagged mode:** `src/tts/tts.ts:869-874`
  - body: In `auto: "tagged"`, the new tag check can trigger TTS from literal `<tts>` markup even when tag processing is disabled. Gate tag-based auto-TTS on the resolved `processTtsTags` setting so disabling tag processing also disables the tag trigger.
  - confidence: 0.84
