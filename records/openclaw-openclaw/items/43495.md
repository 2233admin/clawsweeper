## Summary

Keep this PR open for maintainer handling. The feature request is reasonable, but the branch is stale against the current speech-core TTS runtime, has confirmed `<notts>` behavior gaps, and mixes in an unrelated gateway role authorization expansion that should block merge without security review.

## What This Changes

The PR adds `<notts>` visual-only TTS preprocessing, TTS prompt/docs/tests, a shell TTS filter, and unrelated gateway role, macOS speech, media conversion, MIME, and tool-result audio delivery changes.

## Best Possible Solution

Implement `<notts>` as a narrow current-main change in `extensions/speech-core`, with focused runtime/docs/tests for visible/spoken splitting, tagged auto-mode behavior, and disabled tag processing; split gateway authorization, macOS speech, media conversion, MIME, and tool-result audio work into separately reviewed PRs.

## Reproduction Assessment

Yes. The PR diff itself reproduces the defects: `scripts/tts-filter.sh` has no `<notts>` regex handling, the tagged-mode trigger only checks `<tts>`, and current main has no `<notts>` support in the active speech-core runtime.

## Solution Assessment

No. The requested feature direction is maintainable, but this PR is not the best solution because it is stale against current main, incomplete in companion paths, and includes an unrelated security-sensitive gateway authorization expansion.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Remove the node-role operator bypass:** `src/gateway/role-policy.ts:20-22`
  - body: This change lets node-role clients call operator UI RPCs such as `config.get`, `chat.send`, `chat.history`, and `sessions.preview`. Current role policy keeps node methods separate from operator-scoped RPCs, so this broad authorization expansion is security-sensitive and unrelated to the TTS tag feature.
  - confidence: 0.9
- **[P2] Handle `<notts>` in the shell filter:** `scripts/tts-filter.sh:24-40`
  - body: The added shell filter mirrors the TTS preprocessing path but only handles `<tts>`. A message containing `<notts>visual</notts>` would print raw tags in the text output and speak the visual-only content in the voice output, contradicting the new tag semantics.
  - confidence: 0.94
- **[P2] Gate tagged mode on all enabled TTS tags:** `src/tts/tts.ts:869-870`
  - body: Tagged auto-TTS now fires only for `<tts>`, so a reply that uses only `<notts>` plus normal spoken summary text still skips audio. The same check also ignores `processTtsTags: false`, where literal tags should not force tagged-mode audio.
  - confidence: 0.88
