## Summary

Keep open. Current main appears to have partial MCP lifecycle cleanup that reduces the duplicate-process leak described in the companion report, but it does not yet implement the broader safeguards requested here: MCP config preflight, explicit per-server backoff/circuit-breaker policy, process/RSS status accounting, or default systemd memory/task guardrails.

## What This Changes

The issue requests MCP startup backoff, per-server circuit breaking, MCP config preflight validation, process/resource accounting, and deployment guardrails to prevent misconfigured MCP servers from exhausting VM resources.

## Best Possible Solution

Treat this as a maintainer-owned reliability hardening item: define a generic MCP launch policy keyed by server/config fingerprint, add bounded start retry or unhealthy-state semantics, add an explicit operator-facing MCP preflight/check path, expose useful MCP child/resource status, and decide whether systemd guardrails belong in generated units, doctor repair, docs, or a staged combination.
