## Summary

Keep this issue open. Current main has useful adjacent work for PDF-specific tools, document-extract plugin support, OpenResponses input_file parsing, and gateway attachment offload/MIME preservation, but the core read tool path still does not preprocess binary document files like .docx/.xlsx/.pptx/PDF into extracted text or a binary-file placeholder before model context.

## Best Possible Solution

Keep this issue open and implement a scoped read-tool document preprocessing path: detect non-text binary files before UTF-8 decoding, use existing MIME detection and document-extractor plugin contracts for supported formats, return a concise binary-file placeholder for unsupported formats, and add regression tests proving .docx/.xlsx/PDF bytes no longer enter model context as decoded binary garbage.

## Risks / Open Questions

- A fix should be careful not to turn every read call into heavy document parsing. The likely maintainable path is a bounded read-tool preprocessing layer that reuses the existing MIME/document-extractor plugin contract, starts with PDF support, and returns an explicit binary-file marker for unsupported formats.
- The requested docx/xlsx/pptx extraction is still missing on current main; adding it likely belongs in plugin-owned document extractors plus a generic core seam, not Python-library-specific logic hard-coded into core.
