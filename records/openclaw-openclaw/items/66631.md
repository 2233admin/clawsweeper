## Summary

Keep this open. Current main still preserves Feishu `thread_id` for topic session routing and thread-history lookup, but it does not resolve a `thread_id`-only topic event to the root `om_*` message before choosing outbound reply targets; the normal reply path and ACP-init error path still fall back to the triggering message. The open cross-referenced PR #73972 is explicitly scoped to P2P direct-message threads and says this topic-group issue is out of scope.

## What This Changes

The issue requests that Feishu topic-group replies stay inside the existing topic when inbound events include `thread_id` but no `root_id`, instead of anchoring replies to the triggering message and creating new topics.

## Best Possible Solution

Implement the fix inside the Feishu plugin by resolving topic-group `thread_id` containers to their root `om_*` message before any ACP failure notice or normal/broadcast reply dispatcher is created, while preserving #32980 normal-group behavior and keeping `omt_*` IDs out of reply-target fields.
