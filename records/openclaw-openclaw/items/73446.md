## Summary

Keep open. The live docs still reproduce the reported path loss, current main has no durable locale path-preservation hook, and the two prior standalone-script PRs were closed because that approach is not proven to deploy through this repo's Mintlify publish path or stay synchronized with generated locales.

## What This Changes

The issue requests that the Mintlify docs language switcher preserve the current page path, such as navigating from `/install/docker` to `/zh-CN/install/docker` instead of the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose the boundary: either get Mintlify to preserve page paths upstream, or wire a repo-owned workaround through a supported docs publish/config mechanism that derives locale mappings from `GENERATED_LOCALES` and is verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue's live-docs steps still reproduce in a headless Chromium check: selecting Simplified Chinese from `/install/docker` lands on `/zh-CN` even though `/zh-CN/install/docker` exists.

## Solution Assessment

Unclear. The user-facing behavior should be fixed, but the previously attempted standalone docs-root script is not the best solution because deployment is unproven and its locale mapping would drift from the publish sync source of truth.
