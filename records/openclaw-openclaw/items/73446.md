## Summary

Keep open. Current main still has generated locale navigation and the docs publish pipeline, but no verified path-preserving handler or supported Mintlify custom-script wiring; live docs also still do not serve the standalone root JS assets that the closed fix PRs depended on.

## What This Changes

The issue asks the docs language dropdown to preserve the current page path when switching locales instead of navigating to the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers either get Mintlify to preserve page paths upstream or wire a repo-owned workaround through a supported publish/config mechanism that derives locale mappings from GENERATED_LOCALES and is proven in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue gives concrete live-docs steps, prior report #34021 corroborates the same behavior, and current source/live asset checks show no deployed path-preserving handler; the final acceptance check should still be an interactive switch from /install/docker to /zh-CN/install/docker.

## Solution Assessment

Unclear. The previously attempted standalone docs-root JavaScript file is not the best fix because the assets are not served and hardcoded mappings would drift; a verified Mintlify hook, publish-pipeline integration, or upstream Mintlify fix is the safer direction.
