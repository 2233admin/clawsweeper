## Summary

Keep open. Current main still lacks a durable path-preserving docs locale switcher, live docs still do not serve the proposed standalone docs-root script path, and the two attempted standalone-script PRs were closed because the deployment hook and locale source-of-truth integration were not proven. The issue is a valid canonical tracker for a maintainer/docs-platform decision rather than conservative cleanup.

## What This Changes

The issue asks for the Mintlify docs language switcher to preserve the current page path when switching locales instead of navigating to the target locale homepage.

## Best Possible Solution

Keep this as the canonical tracker until maintainers choose the boundary: either get Mintlify to preserve page paths upstream, or wire a repo-owned workaround through a supported docs publish/config mechanism that derives locale mappings from GENERATED_LOCALES and is verified in a Mintlify preview or live deployment.

## Reproduction Assessment

Yes. The issue gives concrete live-docs steps, #34021 reports the same user-visible behavior, and live checks confirm both source and target pages exist while no path-preserving asset is deployed; final fix validation should still use the browser dropdown from /install/docker to /zh-CN/install/docker.

## Solution Assessment

No for the previously attempted standalone root JavaScript file. The maintainable solution is a verified Mintlify-supported hook or publish-pipeline integration, or an upstream Mintlify platform fix, with locale mappings derived from the publish sync source of truth.
