## Summary

Keep open for maintainer handling, but the PR should not merge as-is. Current main has explicit contracts and tests that conflict with two of the changes, and the cron orphan-recovery heuristic can mark legitimate long-running jobs as done before the runtime has actually ended them.

## What This Changes

The PR changes WebChat final-event reload handling, gateway session-entry resolution, and cron session reaper behavior, with added cron reaper coverage.

## Best Possible Solution

Split and rework the changes against current main: preserve WebChat’s final-event reload fallback, keep the freshest-duplicate session-store contract unless a narrower exact archived-key path is designed, and base cron orphan recovery on runtime/durable run ownership rather than a fixed updatedAt threshold.

## Reproduction Assessment

Yes. Current unit tests reproduce the WebChat no-message final and freshest duplicate-session contracts, and the cron risk is directly supported by the 48-hour default timeout plus lifecycle code that does not heartbeat updatedAt during a long run.

## Solution Assessment

No. The proposed patch is not the best way to solve the reported bugs because it mixes three policy changes and regresses established current-main behavior; a safer fix would be split, rebased, and tied to runtime-owned cron reconciliation evidence.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve no-message final history reloads:** `ui/src/ui/chat-event-reload.ts:8-10`
  - body: Returning `false` when `payload.message` is absent disables the fallback reload path for final events with no renderable assistant payload. Current main has a focused test expecting no-message finals to reload history, and app-gateway reconciliation uses that reload to resolve deferred session.message updates, so this can leave the active chat stale.
  - confidence: 0.89
- **[P2] Keep freshest duplicate session rows:** `src/gateway/session-utils.ts:399-402`
  - body: This exact-key preference returns a stale canonical row before checking fresher duplicate/legacy rows. Current session-utils tests assert that `loadSessionEntry` selects the freshest duplicate row for a logical key, so this regresses existing session metadata resolution when stale and fresh variants coexist.
  - confidence: 0.91
- **[P1] Verify cron ownership before marking runs done:** `src/cron/session-reaper.ts:104-109`
  - body: The reaper marks any `running` cron session older than two hours as done based only on `updatedAt`. Agent runs can legitimately run far longer, and lifecycle persistence only proves start/end transitions, so an active cron job can be finalized and later pruned while still running.
  - confidence: 0.88
