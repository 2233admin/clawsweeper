## Summary

Keep this PR open. Current main still does not implement offline APNs chat-reply alerts, and this PR is an active but incomplete implementation candidate; the diff also has blocking payload-kind and main-session matching problems before lifecycle wiring should land.

## What This Changes

The PR adds a gateway `chat-apns-notify` helper and colocated Vitest tests for sending APNs alerts when an offline iOS requester has a main-session chat reply complete.

## Best Possible Solution

Land a complete gateway+iOS implementation or a clearly scoped replacement: add a dedicated chat-reply APNs payload kind and iOS route, decide whether notification bodies should be generic or opt-in/redacted, wire both agent-event and direct `chat.send` completion paths with requester device/connection tracking, use the canonical main-session helpers, and cover direct, relay, skip, and cleanup cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Use a chat-specific APNs payload kind:** `src/gateway/chat-apns-notify.ts:73-84`
  - body: `maybeSendChatReplyApnsAlert` sends chat replies through `sendApnsAlert`, but the current alert sender hardcodes `openclaw.kind` to `push.test`. Once this helper is wired, chat notifications will be classified as diagnostic push-test payloads instead of a chat-reply route, so this needs a dedicated payload builder/send function with a chat-specific kind.
  - confidence: 0.92
- **[P2] Compare against canonical main session keys:** `src/gateway/chat-apns-notify.ts:35-39`
  - body: The main-session guard compares `params.sessionKey` to `params.mainSessionKey` or raw `cfg.session.mainKey`, but current Gateway main sessions are canonicalized to values like `global` or `agent:<id>:<mainKey>`. Valid main-session replies would return early here, so the helper should use the existing canonical main-session helpers before deciding to skip.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff adds a privacy-sensitive APNs alert path that copies assistant reply text into system notifications, without a documented opt-in/redaction decision.

Concerns:

- **[medium] Avoid exposing chat replies in APNs alerts by default:** `src/gateway/chat-apns-notify.ts:53`
  - body: `normalizeChatReplyNotificationBody(params.replyText)` turns the assistant reply into the APNs alert body. APNs alerts can appear on the lock screen and pass through Apple or relay infrastructure, so this should use a generic body or an explicit opt-in/redaction policy before being wired to real chat completions.
  - confidence: 0.86
