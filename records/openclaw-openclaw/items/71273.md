## Summary

Keep open. Current main still exposes the Kimi Coding tool-use path and I found no targeted current-main fix or regression for repeated identical Kimi tool calls after a tool-result turn; the original stop-reason theory is not sufficient because current source already leaves plain final assistant text as `stop` unless tagged tool-call markup was parsed.

## What This Changes

The issue reports that Kimi Code via the Kimi provider can repeatedly execute identical tool calls after receiving tool results instead of producing a final assistant answer.

## Best Possible Solution

Capture a current Kimi Coding transcript or minimal replay fixture for the first assistant response after tool results, then fix the owning Kimi stream or replay path with a focused regression that proves identical calls are not reissued or replayed indefinitely.

## Reproduction Assessment

Unclear: the issue gives concrete steps and repeated-log evidence, but I could not live-run Kimi Coding from this checkout. A high-confidence reproduction needs the post-tool-result assistant payload showing whether Kimi emits fresh tagged calls or OpenClaw reuses stale calls.

## Solution Assessment

No: the proposed stop-reason-only change is not the best proven fix because current source already leaves text-only final answers unchanged and only rewrites after tagged-call parsing succeeds. The safer path is transcript-first diagnosis followed by a targeted Kimi provider regression.
