## Summary

Keep open. Current main still has the active Kimi Coding provider and no targeted merged fix for the repeated identical tool-call loop; the reported symptom is plausible, but the issue's proposed stop-reason root cause is contradicted by current wrapper behavior and existing tests.

## What This Changes

The issue reports that Kimi Code via the bundled Kimi provider repeats identical tool calls after tool results instead of producing a final assistant answer.

## Best Possible Solution

Capture a current Kimi Coding trajectory or replay fixture for the turn immediately after tool results, then make the narrowest fix in the Kimi provider stream/replay path or a generic loop guard while preserving valid tagged tool-call execution and adding focused regression coverage.

## Reproduction Assessment

Unclear: the issue has concrete manual steps and log evidence, but this review did not have a raw transcript or replay fixture that proves the post-tool-result assistant output shape on current main.

## Solution Assessment

No: the proposed stop-reason guard is not the best proven fix because current code already leaves plain final assistant text unchanged, and role-gating can block valid Kimi tagged calls when role metadata is absent. The safer path is transcript-first diagnosis followed by a targeted provider or runner regression.
