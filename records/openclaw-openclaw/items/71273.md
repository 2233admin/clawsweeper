## Summary

Keep open. Current main still has the reported Kimi Coding tool-use path and no targeted regression or merged fix for repeated identical tool calls after a tool-result turn; the report is recent and detailed, but the proposed stop-reason root cause is contradicted by current source because plain final assistant text already stays as `stop`.

## What This Changes

The issue reports that the Kimi Code provider can repeatedly execute identical `read`/`exec` tool calls after tool results instead of ending with a final assistant answer.

## Best Possible Solution

Keep the issue open until a current Kimi Coding transcript or replay fixture proves whether Kimi is re-emitting identical tagged calls, OpenClaw is replaying stale calls, or another stream/replay normalization step is responsible; then fix the owning Kimi provider path with a focused regression while leaving generic loop detection as an opt-in guardrail unless maintainers decide to change defaults.

## Reproduction Assessment

Unclear: the issue gives concrete steps and repeated-log evidence, but I did not have credentials to live-run Kimi Coding against current main. A high-confidence reproduction needs the first assistant response after tool results, showing whether it contains fresh tagged tool-call markup or whether OpenClaw reused previous calls.

## Solution Assessment

Unclear: the closed proposed stop-reason change is not the best proven fix because current source already leaves plain text-only final answers unchanged and only rewrites after tagged tool-call parsing succeeds. The safer path is transcript-first diagnosis followed by a targeted Kimi stream or replay regression.
