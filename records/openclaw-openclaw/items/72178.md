## Summary

Keep open. Current main still lacks cursor-based `chat.history` pagination and web UI load-older behavior, so the PR is not obsolete. The latest PR head is still not merge-ready because it computes `before`/`start`/`end` but continues projecting the full raw history, which means older-page requests can return the latest page again and be prepended as duplicates.

## What This Changes

The PR adds `before`-based `chat.history` pagination with `cursor`/`hasMore`, web UI load-older state/rendering/scroll compensation, partial `NO_REPLY` stream suppression, protocol docs, and focused tests.

## Best Possible Solution

Keep the PR open and require the gateway handler to select the raw page before projection, or otherwise carry stable raw indices through projection/sanitization/capping. Add handler-level tests for `before` with normal history, sanitized `NO_REPLY`/commentary drops, and byte-capped leading trims before considering merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Apply the cursor slice before projecting history:** `src/gateway/server-methods/chat.ts:1525-1530`
  - body: When `before` is provided, the handler computes `end` and `start` but still calls `projectRecentChatDisplayMessages(rawMessages, ...)` on the full transcript. That helper limits from the tail, so loading older pages can return the latest page again and the UI will prepend duplicates while the cursor fails to move to older history. Select the raw `[start, end)` window, or carry raw indices through projection, before normalizing and capping.
  - confidence: 0.94
