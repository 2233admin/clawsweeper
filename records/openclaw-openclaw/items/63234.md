## Summary

Keep open. Current main still has the generic plugin-config validation paths that emit the reported missing-required-field error whenever a runtime or metadata plugin load sees missing `entry.config`, and the codebase still lacks either UnraidClaw source or a synthetic required-config startup/catalog regression proving valid config is preserved across the implicated phases. The recent post-review plugin change is runtime-dependency staging work, not a fix for this valid-config startup-spam path.

## What This Changes

The issue asks OpenClaw to stop repeatedly logging invalid-config errors during gateway startup when a plugin has valid required config under `plugins.entries.<id>.config` and later registers tools successfully.

## Best Possible Solution

Add a narrow generic regression that keeps resolved required plugin config attached through gateway startup, active registry reuse, `tools.catalog`, and post-ready plugin tool resolution, and fails if a valid config is later observed as `{}` or if repeated invalid-config logs are emitted.

## Repair Work Prompt

Fix the valid required-plugin-config startup spam reported in openclaw/openclaw#63234. Add a synthetic native tool plugin fixture with a manifest config schema requiring `serverUrl` and `apiKey`, configure it under `plugins.entries.<id>.config`, and exercise gateway startup plus post-ready plugin tool/catalog resolution so the plugin is never reloaded with empty config and repeated `[plugins] <id> invalid config` logs are not emitted.
