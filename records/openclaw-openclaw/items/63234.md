## Summary

Keep open. Current main has adjacent generic improvements for plugin-local invalid config degradation and gateway/plugin registry reuse, but the exact reported valid-config startup spam is not proven fixed. The UnraidClaw plugin is not present in the source tree, loader paths can still emit `[plugins] <id> invalid config` on repeated loads that see a missing `entry.config`, and current tests do not cover a synthetic required-config tool plugin through startup plus post-ready tool/catalog paths.

## Best Possible Solution

Keep this open and land a narrow generic regression/fix. Add a synthetic tool plugin whose config schema requires `serverUrl` and `apiKey`, configure it under `plugins.entries.<id>.config`, simulate gateway startup plus the post-ready tool inventory/catalog path that can reload plugins, and assert the plugin is not reloaded with empty config or repeatedly logged as invalid. If current main already passes, land the focused regression coverage and document that the related generic fixes cover the path.
