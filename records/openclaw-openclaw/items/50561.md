## Summary

Keep open. Current main still does not auto-apply the full doctor repair sequence on gateway start; startup only performs narrow recovery/degraded-startup handling and plugin auto-enable persistence, while `doctor --fix` remains an explicit repair path. The follow-up comments also add credible safety concerns: an automatic `doctor --fix` startup path reportedly changed intentional sandbox and Telegram settings silently, so this needs maintainer product/safety judgment rather than conservative cleanup closure.

## Best Possible Solution

Keep this open for maintainer product/safety design. The safest direction is a narrow, explicit startup repair contract: classify which doctor mutations are safe for unattended startup, exclude security/channel availability changes unless protected/opted-in, log every changed path with backup/recovery details, and keep broader repair behind manual `openclaw doctor --fix` or a clearly named opt-in setting.
