## Summary

Keep open. Current main still does not auto-apply the general doctor repair sequence during ordinary gateway startup; startup uses targeted recovery/degraded loading plus a few explicit startup writes, while broad repair stays behind explicit doctor repair/yes modes. The issue discussion also contains credible counterexamples where running doctor --fix before startup silently changed intentional sandbox and Telegram settings, so this needs maintainer policy before implementation.

## What This Changes

Request auto-applying safe, non-destructive doctor fixes on gateway startup or adding an opt-in setting, with logs showing what was applied.

## Best Possible Solution

Define a narrow startup repair contract first: only explicitly classified, idempotent, low-risk mutations should run unattended; sandbox, auth, allowlist, exec, channel availability, and plugin disablement changes should remain manual or require a clear opt-in. Any startup repair should write backups and log every changed config path, with docs and tests covering protected/intentional config behavior.
