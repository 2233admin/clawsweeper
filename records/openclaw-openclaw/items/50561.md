## Summary

Keep open. Current main still does not auto-apply general safe doctor fixes on ordinary gateway startup: startup has targeted recovery paths and manual `openclaw doctor --fix` hints, while the broader repair sequence only runs from explicit repair/yes modes. The discussion includes credible reports that a startup `doctor --fix` workaround silently changed intentional sandbox and Telegram settings, and the related open PR is scoped to failure-path diagnostics rather than this broader startup auto-repair request.

## What This Changes

The issue requests automatic safe/non-destructive doctor repairs during gateway startup, or an opt-in `doctor.autoFix` setting, with logs showing what was applied.

## Best Possible Solution

Define a narrow startup repair contract first: classify which doctor mutations are safe for unattended gateway startup, exclude security/sandbox/channel-availability/plugin-disablement changes unless explicitly opted in, log every changed config path with backup/recovery details, and keep broader repair behind manual `openclaw doctor --fix` or a clearly named opt-in setting.
