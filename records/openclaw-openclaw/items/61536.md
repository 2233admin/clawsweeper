## Summary

Keep open. Current main still matches the reported native Discord voice output-contract gap: `/vc` transcribes speech, enters the agent as Discord ingress, and only plays audio when the agent returns text, while the `tts` tool remains available because the tool policy only hides it for the `voice` provider.

## What This Changes

The issue asks OpenClaw to make native Discord `/vc` generate and play a spoken reply after successful voice capture/STT instead of allowing a `tts` tool call followed by `NO_REPLY` to produce silence.

## Best Possible Solution

Add a native Discord VC spoken-output contract that preserves Discord session/routing metadata while running voice-channel turns under a voice-safe tool/output policy, so the agent returns spoken text for `DiscordVoiceManager` to synthesize and play instead of satisfying the turn with `tts` plus `NO_REPLY`.

## Repair Work Prompt

Fix the native Discord `/vc` spoken-output contract for the open Discord voice silence bug. Current `extensions/discord/src/voice/manager.ts` sends transcribed speech through `agentCommandFromIngress` as `messageChannel: "discord"`, `deliver: false`, then only synthesizes playback from returned `payload.text`. Because `src/agents/pi-tools.message-provider-policy.ts` hides `tts` only for `messageProvider: "voice"`, the agent can call `tts` and finish with `NO_REPLY`, leaving Discord VC with no reply text to synthesize. Implement a voice-specific output/tool policy for Discord VC: preserve Discord session/routing metadata, but make VC turns use a voice-safe provider/context and/or an explicit spoken-output system prompt similar to `extensions/voice-call/src/response-generator.ts`. Do not special-case final `NO_REPLY` as a successful spoken reply, do not revive the old `discord-voice` plugin, and do not change DAVE receive/decrypt recovery behavior. Add focused regression coverage proving VC segments use the voice-safe policy/hide `tts` and that tool-only `tts` plus `NO_REPLY`/empty final output cannot silently satisfy playback. Keep normal Discord text ingress behavior unchanged. Add a user-facing changelog fix entry under the active version with `Thanks @aounakram` and `Thanks @jasonluo-tw`.
