## Summary

Keep this issue open. Current main has automatic Discord typing infrastructure for inbound replies and heartbeat targets, but it still does not expose the requested agent-callable `typing` or `typing-indicator` action through the shared `message` tool. The related discussion points to lifecycle bugs and cross-provider design concerns, not to a merged implementation or canonical duplicate.

## Best Possible Solution

Keep the issue open until maintainers either add a shared `typing` or `typing-indicator` message-tool action, or explicitly decide automatic typing is sufficient. A concrete implementation should add the shared action name, target-mode/schema support, Discord `describeMessageTool` advertising, a `handleDiscordMessageAction` route to `sendTypingDiscord`, lifecycle semantics, Discord regression coverage, and public docs. If cross-provider parity is desired, design the action so Telegram, Slack, Discord, and similar plugins can opt in through the channel plugin contract.

## Risks / Open Questions

- Manual typing needs clear one-shot, refresh, and stop semantics so it does not recreate the stuck-indicator lifecycle bugs cited in the discussion.
- Maintainers may decide automatic inbound-reply and heartbeat typing is the intended product boundary, but that would be a product decision rather than evidence that the requested action already exists.
