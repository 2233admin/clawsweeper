## Summary

Keep this issue open. Current main has automatic Discord typing for inbound reply and heartbeat paths, but it still does not expose an agent-callable `typing` or `typing-indicator` action through the shared `message` tool. The shared action registry, message tool schema, CLI validation, Discord discovery, and Discord action dispatcher all still omit that surface.

## Best Possible Solution

Keep the issue open until maintainers either add a shared `typing`/`typing-indicator` message-tool action or explicitly decide that automatic inbound and heartbeat typing are the intended product boundary. A concrete implementation should update the shared action registry, message-tool schema, CLI normalization, Discord discovery, Discord dispatcher, lifecycle semantics, regression tests, and public docs.

## Risks / Open Questions

- Manual typing control needs a bounded one-shot/refresh contract; Discord typing is a temporary TTL cue, not a durable start/stop state.
- A generic cross-channel typing action would touch shared SDK/action enums, target-mode mapping, CLI routing, plugin discovery, Discord dispatch, docs, and regression coverage.
- The related discussion cites prior stuck-indicator bugs, so lifecycle cleanup and timeout behavior should be specified before exposing manual control.
