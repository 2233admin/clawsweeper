## Summary

Keep this PR open. Current main does not implement the PR's intended Docker-facing proxy examples: `docker-compose.yml` still lacks `HTTP_PROXY`, `HTTPS_PROXY`, or `NO_PROXY` examples, and `docs/install/docker.md` has no Docker proxy configuration section. The related current proxy docs cover OpenClaw's `proxy.enabled` / `OPENCLAW_PROXY_URL` path, but that is not the same Docker Compose guidance this PR proposes. The PR is docs/config-only, but it needs rebase and review against the current proxy semantics before it could land.

## Best Possible Solution

Keep the PR open for contributor or maintainer revision. The best path is a current-main Docker docs update that either documents the supported `proxy.enabled` / `OPENCLAW_PROXY_URL` approach for container deployments or adds accurate `HTTP_PROXY` / `HTTPS_PROXY` Compose examples with NO_PROXY values that match OpenClaw's actual runtime behavior, plus a brief security note linking the public network proxy docs.

## Risks / Open Questions

- If merged as-is, the PR's CIDR-style `NO_PROXY` sample may mislead users because current OpenClaw/undici-mirrored matching only covers host/domain/wildcard/port patterns, not private-network CIDR bypasses.

- The branch predates current Docker docs/compose changes such as the plugin runtime dependency volume and already-present `host.docker.internal` mapping, so it needs a rebase and a smaller current-main patch.
