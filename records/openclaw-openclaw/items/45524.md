## Summary

Keep open. Current main still lacks the requested startup warning for explicit `agents.defaults.sandbox.mode="off"`, while this PR's head fixed the original sandbox false-positive but is stale against the current core dangerous-flag helper and still has a concrete trusted-proxy warning bug.

## What This Changes

The PR adds startup dangerous-config warnings for explicit `agents.defaults.sandbox.mode="off"`, `gateway.auth.mode="none"`, and trusted-proxy `allowUsers=[]` in `src/security/dangerous-config-flags.ts`.

## Best Possible Solution

Rebase onto current main, add the explicit sandbox-off startup warning in the current core dangerous-flag helper with tests proving absent sandbox config stays quiet and explicit `mode="off"` warns, then handle the auth warnings separately or normalize them to the same contracts as runtime and `openclaw security audit`.

## Reproduction Assessment

Yes. A config with `gateway.auth.mode="trusted-proxy"`, a valid `trustedProxy.userHeader`, and omitted `trustedProxy.allowUsers` exercises the PR bug because runtime treats omitted allowUsers as allow-all while the PR warning only detects an explicit empty array.

## Solution Assessment

No. The explicit sandbox-off equality check is the right narrow direction, but this PR is not the best current-main fix until it is rebased and the trusted-proxy warning is either removed from scope or made consistent with `allowUsers ?? []` semantics.
