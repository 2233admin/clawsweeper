## Summary

Keep open. Current main has Discord native approval delivery and a newer manual fallback notice path, but it still does not prove the reported native Discord approval card/buttons are delivered for the multi-account embedded Discord setup; the code still has an account-match gate that can skip Xavier delivery for an approval sourced from another Discord account.

## What This Changes

The issue requests restoring native Discord exec approval card/button delivery when an approval is created and manual `/approve` works.

## Best Possible Solution

Keep this open and clarify the intended Discord account-selection contract, then add a focused regression that proves a Discord-originated exec approval either posts the native card/buttons to the configured approval surface or emits the documented same-chat fallback when delivery genuinely fails.

## Reproduction Assessment

Yes for the likely static code path: a Discord approval whose `turnSourceAccountId` does not match the enabled Xavier native runtime can be skipped by `shouldHandleDiscordApprovalRequest`, while manual `/approve` can still resolve through command authorization. A live Discord reproduction was not run in this read-only review.

## Solution Assessment

No. Current main has useful fallback plumbing, but fallback text is not the same as delivering the native Discord approval card/buttons requested by the report, and the multi-account routing semantics remain unresolved.
