## Summary

Keep open. This is a fresh, specific Discord native approval delivery regression against v2026.4.26, and current main still does not prove that a Discord approval card/button is delivered when the approval itself exists and manual `/approve` works. The current code has the intended native card path, but it can also keep the gateway approval pending because an approval client or turn-source route exists while the native Discord delivery path produces no pending message entry.

## Best Possible Solution

Keep the issue open and add a focused regression around Discord embedded exec approvals: simulate a Discord-originated approval with configured approvers, assert that the native runtime attempts and records a Discord card/button delivery, and make failure/no-entry cases visible to the tool result or fallback path. If cross-account approval routing is intended, extend the Discord native approval account-selection logic rather than relying only on the originating account match.

## Evidence

- **Issue context describes a concrete current regression:** The supplied issue body reports OpenClaw 2026.4.26 with Discord embedded channel: an approval id is created, manual `/approve ...` in Discord succeeds, but no native Discord approval card/buttons appear. The issue also links #71804, #46708, and #39978 as related but not a canonical open fix for this exact report.
- **Discord native runtime is implemented but failed sends return no entry:** The Discord native approval runtime builds a component payload with action buttons, sends it via Discord REST, and returns a pending entry only when Discord returns a message id.
