## Summary

Keep open. Current main still gates Discord native approval delivery to the approval request's source Discord account, so the reported multi-account setup can still produce a valid approval ID that manual `/approve` accepts while the configured approver account receives no native card. Changing that behavior is approval-routing and disclosure policy work, not a conservative cleanup close.

## What This Changes

Discord exec approvals create valid approval IDs and accept manual `/approve`, but native Discord approval cards/buttons are not delivered in a multi-account Discord configuration.

## Best Possible Solution

Define the Discord multi-account approval routing contract first. If cross-account approval surfaces are supported, route native cards to the configured approver account only with explicit authorization and focused regression tests; otherwise document origin-account-only native delivery and make the manual fallback deterministic for this configuration.

## Reproduction Assessment

Yes. At source level, configure native Discord approvals only on one account and create an approval request whose `turnSourceAccountId` is another account; the current gate predicts no native card for the configured approval account while manual `/approve` remains possible for an authorized approver.

## Solution Assessment

Unclear. Broadening native delivery is not automatically the best fix because approval cards include command details and actionable buttons; the safer path is maintainer-approved routing semantics plus tests, or explicit documentation of origin-account-only behavior.
