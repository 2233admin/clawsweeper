## Summary

Keep open. Current main still does not prove native Discord approval card/button delivery for the reported multi-account shape: the native Discord path remains gated to the request’s source account, while manual `/approve` can still succeed through the current account’s approver authorization. The recent #74472 fix helps loopback Gateway approval clients but does not change Discord account routing, so this needs an explicit maintainer decision before broadening approval delivery across accounts.

## What This Changes

The issue requests restoring native Discord exec approval cards/buttons when approvals are enabled on one Discord account and an approval can otherwise be manually resolved with `/approve`.

## Best Possible Solution

Define the Discord multi-account approval contract first. If cross-account approval surfaces are supported, implement explicit configured-approver-account delivery with tests and docs; if not, document origin-account-only native delivery and make the same-chat/manual fallback behavior clear for this configuration.

## Reproduction Assessment

Yes, at source level: configure native Discord approvals only for xavier, create an approval request with turnSourceAccountId set to another Discord account, and current main’s shouldHandle/target resolver path returns false/null. I did not perform a live Discord run.

## Solution Assessment

Unclear. Broadly removing the account gate would be risky because approval cards include command text; the safer path is a maintainer-approved routing contract plus narrow tests for the chosen behavior.
