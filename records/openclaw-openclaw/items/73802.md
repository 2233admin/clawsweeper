## Summary

Keep open. Current main adds a useful fallback for failed native delivery, but it still does not prove the requested Discord native approval card/button delivery for the reported multi-account setup; the Discord native runtime still gates handling to the approval request's source account, so an enabled approval account can skip requests originating from another Discord account.

## What This Changes

Discord exec approvals should deliver the native Discord approval card/buttons when an approval is created, instead of requiring the operator to type `/approve` manually.

## Best Possible Solution

Keep the issue open until maintainers decide and document the Discord multi-account approval contract. If cross-account approval surfaces are supported, add a focused regression proving that a request from a non-enabled Discord account reaches the configured approver account with a native card/buttons or an explicit same-chat fallback; if not, document origin-account-only behavior and make the fallback text explicit.

## Reproduction Assessment

Yes, at source level: configure Discord approvals only on `xavier`, originate an exec approval from another Discord account/session, and the current account gate predicts no `xavier` native card attempt while manual `/approve` can still resolve through the approver path. I did not run a live Discord transport test.

## Solution Assessment

Unclear. The current fallback mitigation is useful for silent delivery failures, but it is not the full native-card solution requested here unless maintainers confirm that origin-account-only delivery is the intended behavior.
