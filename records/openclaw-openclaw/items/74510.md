## Summary

Keep this open. Current main still has the reported history/UI failure mode for persisted assistant messages that have ordinary assistant text plus `stopReason: "toolUse"` but no top-level `phase` or phased `textSignature`; the projection only suppresses explicit commentary and the Control UI renders surviving unphased assistant text through the shared extractor.

## What This Changes

The issue asks OpenClaw to prevent intermediate assistant commentary/progress text from rendering as same-level Control UI replies alongside the final answer for one user turn.

## Best Possible Solution

Preserve `commentary` and `final_answer` metadata through provider conversion, transcript persistence, `chat.history`, and Control UI extraction, then add a constrained legacy fallback for the reported phaseless intermediate-commentary shape so the final answer remains the primary visible reply.

## Reproduction Assessment

Yes. A source-level fixture with a user message, an unphased assistant text message using `stopReason: "toolUse"`, and a later final assistant message follows the current path: phase resolution is undefined, history projection keeps the first assistant entry, and Control UI extraction can render it.

## Solution Assessment

Yes. The maintainable direction is metadata preservation plus a narrow legacy fallback; a frontend-only repaint or a broad stop-reason-only suppression would either miss persisted metadata loss or hide valid assistant text.
