## Summary

Current main still leaves the reported legacy transcript shape visible: explicit phased commentary is handled, but an assistant message with plain text, `stopReason: "toolUse"`, and no `phase` or `textSignature` can pass through `chat.history` and render as a normal Control UI assistant bubble before the final answer. The related #74928 report is broader and still open, so it does not supersede this narrower UX bug.

## What This Changes

Control UI should avoid showing intermediate assistant commentary/progress text as same-level replies beside the final answer for one user turn.

## Best Possible Solution

Preserve `commentary` and `final_answer` metadata through provider output, Pi transcript persistence, `chat.history`, and Control UI rendering, then add a narrow compatibility fallback for legacy unphased intermediate assistant rows so the final answer remains the primary visible reply.

## Reproduction Assessment

Yes. A source-level reproduction is a persisted assistant row with text content, `stopReason: "toolUse"`, and no `phase`/`textSignature`, followed by a final assistant answer; current projection does not drop the first row and the UI extractor can render it.

## Solution Assessment

Yes, but the fix should start at metadata preservation and use any `stopReason` fallback only as legacy compatibility. A UI-only blanket suppression would be riskier because unphased legacy assistant text is still an intentional compatibility path.
