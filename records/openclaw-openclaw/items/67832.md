## Summary

Keep open. Current main still has the underlying `sessions_spawn` agent tool and the direct HTTP `/tools/invoke` endpoint, but it does not provide the reporter’s narrowed user-facing CLI path for direct session spawning or generic tool invocation. The remaining work is real, but the CLI shape and default policy are security-sensitive enough to need maintainer design/review before an automated repair job should be queued.

## What This Changes

The issue requests a direct session-spawn API/CLI for external automation, later narrowing the remaining ask to a CLI wrapper because the HTTP `/tools/invoke` endpoint already exists.

## Best Possible Solution

Keep this as the focused tracker for a conservative CLI design that reuses the existing Gateway `/tools/invoke` contract, documents the opt-in policy for dangerous tools, and adds targeted CLI/docs/tests once maintainers choose the command shape.
