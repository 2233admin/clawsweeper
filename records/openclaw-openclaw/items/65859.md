## Summary

Keep open. Current main has partial WebChat audio support, including `audio/*` in the picker and `<audio controls>` for already-normalized audio attachments, but concrete gaps remain: filename-based upload filtering still rejects `.webm` and `.mp4` audio containers, shared audio filename classification omits `.webm`, and Gateway-produced `{type:"audio", source:{...}}` blocks are not normalized into playable UI attachments.

## What This Changes

Add WebChat file-picker support for common audio uploads and render received audio/TTS audio attachments inline with HTML audio controls.

## Best Possible Solution

Keep this issue open and make one focused WebChat audio follow-up: accept browser-reported audio WebM/MP4-style audio without allowing video uploads, add `.webm` where trusted audio filename classification is needed, normalize Gateway base64 audio content blocks into playable attachment items, preserve the existing `attachment.kind === "audio"` renderer and #67572 non-image `chat.send` routing, and add targeted regression coverage.

## Repair Work Prompt

Fix WebChat audio upload/playback gaps for openclaw/openclaw#65859. Current main partially supports audio: the picker advertises `audio/*`, normalized `attachment.kind === "audio"` renders with `<audio controls>`, and #67572 handles generic Gateway non-image `chat.send` offload. Remaining bugs: `isSupportedChatAttachmentFile` rejects `.webm`, `.mp4`, and `.mpeg` by filename even when the File MIME is audio; shared `isAudioFileName` omits `.webm`; and Gateway-produced `{ type: "audio", source: { type: "base64", media_type, data } }` assistant content blocks are not normalized into playable UI attachments. Implement a focused fix in the WebChat/media boundary: accept audio WebM/MP4-style files when the browser MIME/trusted context is audio while still rejecting video WebM/MP4/MOV uploads; add `.webm` only where it is safe as audio or require MIME when ambiguous; convert Gateway base64 audio content blocks to `type: "attachment"` items with `kind: "audio"`, a data URL, label, and MIME so the existing renderer plays them; preserve existing MEDIA URL audio parsing, voice-note behavior, and non-image `chat.send` routing. Add or extend focused tests for upload filtering, message normalization of raw Gateway audio blocks, renderer audio controls, shared media MIME classification, and Gateway WebChat media output. Changelog only if maintainers consider the user-visible WebChat behavior change release-noteworthy; do not change unrelated channel/provider behavior.
