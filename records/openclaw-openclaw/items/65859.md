## Summary

Keep open. Current main and v2026.4.27 now cover the main WebChat audio playback/upload path, including `audio/*` in the picker and an inline HTML audio player for audio attachments, but the requested `audio/webm` upload case is still blocked because `.webm` filenames are rejected as video before the MIME type can allow audio.

## What This Changes

Add audio file upload support to WebChat and render received audio attachments inline with playback controls.

## Best Possible Solution

Keep the shipped inline audio player and broad `audio/*` picker support, then add a narrow WebChat attachment classifier/test fix so `audio/webm` uploads are accepted without enabling video uploads.

## Reproduction Assessment

Yes. A source-level reproduction is clear: `handleFileSelect` and `handleDrop` call `isSupportedChatAttachmentFile`, and `File { name: "clip.webm", type: "audio/webm" }` reaches the `.webm` filename rejection path despite being audio.

## Solution Assessment

Yes. The narrow maintainable fix is in the WebChat attachment support helper: allow `file.type.startsWith("audio/")` before extension-based video rejection, with focused tests that audio WebM is accepted and video WebM is still rejected.

## Repair Work Prompt

Fix the remaining WebChat audio upload gap from https://github.com/openclaw/openclaw/issues/65859. Current main already includes `audio/*` in `CHAT_ATTACHMENT_ACCEPT` and renders assistant audio attachments with `<audio controls>`, but `ui/src/ui/chat/attachment-support.ts` still rejects `.webm` filenames before respecting `file.type`, so `audio/webm` uploads are silently skipped. Update the UI attachment classifier to accept files whose MIME starts with `audio/` while preserving rejection for `video/*` and known video-only extensions. Add focused tests, preferably in `ui/src/ui/views/chat.test.ts` or a small attachment-support test, proving `audio/webm`/`clip.webm` is accepted and `video/webm` remains rejected. Do not change backend non-image offload semantics, do not enable video uploads, and avoid broad UI refactors. Changelog is optional unless maintainers want one for this user-visible enhancement follow-up.
