## Summary

Keep this issue open. Current main still sends sender metadata into the model-facing prompt path for both internal TUI/webchat-style clients and Telegram, and the existing stripping code is documented/display hygiene rather than prompt removal. The related open PR #47216 is a useful partial TUI/webchat candidate, but it does not cover the broader Telegram/privacy behavior reported here.

## Best Possible Solution

Keep this issue open and land a scoped prompt-context fix: preserve trusted internal sender/requester identity for routing, authorization, approvals, and delivery, but stop exposing transport-only internal client identity and privacy-sensitive channel sender identifiers in the model-facing user prompt unless a channel explicitly opts into a minimized label. Use #47216 as a partial TUI/webchat implementation reference and add regression coverage for Telegram/direct-channel sender privacy as well.
