## Summary

Keep open: current main still contains the reported sticky IPv4 fallback path, the merged watchdog work addresses an adjacent wedged-runner class rather than proving this host-specific transport loop is fixed, and maintainer comments already say local successful probes do not invalidate the affected macOS/India report.

## What This Changes

The issue reports that Telegram long-polling fails to consume Bot API updates in the gateway on a reachable macOS/India host while the sticky IPv4 fallback path repeatedly logs transport failures.

## Best Possible Solution

Keep the issue open until the affected host is retested on current main or a current release containing #71502 with `OPENCLAW_DIAG=telegram` through a full stall/rebuild cycle. If an in-gateway `resolveTelegramFetch` probe fails, the fix belongs in Telegram transport fallback diagnostics and dispatcher lifecycle; if the probe succeeds while polling wedges, continue in polling-session runner ownership and restart behavior.
