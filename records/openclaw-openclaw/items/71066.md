## Summary

Keep open. Current main contains meaningful Telegram polling hardening, but the maintainer discussion explicitly kept this network/region-specific sticky IPv4 fallback report open for affected-host diagnostics, and the current source still has the same fallback path rather than proof that it now succeeds on the reported macOS/India host.

## What This Changes

The issue reports that Telegram long polling never consumes `getUpdates` in the gateway despite a valid token and reachable Bot API, repeatedly logging the sticky IPv4 fallback warning while updates accumulate on Telegram.

## Best Possible Solution

Keep this open until an affected-host current-main retest captures `OPENCLAW_DIAG=telegram` through a full stall/rebuild cycle. If the in-gateway Telegram fetch wrapper fails for `getMe` or `getUpdates`, fix the transport fallback/dispatcher diagnostics in `extensions/telegram/src/fetch.ts`; if the wrapper succeeds while gateway polling still wedges, focus on `extensions/telegram/src/polling-session.ts` and `extensions/telegram/src/monitor.ts` using #71502 and de1ac12f as the baseline.

## Reproduction Assessment

No high-confidence current-main reproduction is available. The original steps and logs are concrete for 2026.4.22, but the missing proof is a current-main retest on the affected macOS/India host with gateway diagnostics and an in-gateway `resolveTelegramFetch` probe.

## Solution Assessment

Unclear. Current main has useful recovery and visibility changes, but none is proven to be the narrow fix for this exact network/region-specific sticky IPv4 fallback loop, so the next solution should be evidence-led rather than another guessed dispatcher or watchdog change.
