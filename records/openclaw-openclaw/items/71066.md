## Summary

Keep open. Current main has meaningful Telegram polling hardening, and v2026.4.29 also ships a timeout clamp that could explain a short-cadence reconnect class, but the maintainer discussion explicitly treats those changes as retest baselines rather than proof that this India/macOS sticky IPv4 fallback loop is fixed on the affected host.

## What This Changes

The issue reports that Telegram long-polling never consumes `getUpdates` in the gateway even though the Bot API and standalone undici probes work from the same host.

## Best Possible Solution

Keep the issue open and isolate the failing layer on current main. The fix should stay in the Telegram plugin, likely in `extensions/telegram/src/fetch.ts` or `extensions/telegram/src/polling-session.ts`, after diagnostics separate dispatcher fallback behavior from polling runner ownership and LaunchAgent environment effects.

## Reproduction Assessment

No high-confidence current-main reproduction is available. The original 2026.4.22 steps and logs are concrete, but the missing proof is an affected-host retest on current main plus an OpenClaw `resolveTelegramFetch` or gateway-side `getMe`/`getUpdates` probe from the same LaunchAgent environment.

## Solution Assessment

Unclear. The existing transport rebuild, watchdog, status, and timeout-floor changes are reasonable baselines, but diagnostics should come before another code change because the remaining symptom could be transport fallback, polling lifecycle ownership, configuration, or launch-environment specific.
