## Summary

Keep open. Current main still has the reported cron message serialization path, so this PR is not obsolete, but the proposed patch adds a prompt-content logging regression and should not merge as written.

## What This Changes

This PR changes `src/cron/service/timer.ts` to unwrap object-shaped cron `agentTurn.payload.message` values from `text`, `message`, or `content`, then JSON-stringify and debug-log unrecognized shapes before calling the isolated cron runner.

## Best Possible Solution

Land a tested cron fix that preserves the public `message: string` contract: repair or normalize malformed stored payloads before execution where appropriate, and keep any defensive extraction at the isolated execution boundary narrow and free of prompt-content logging.

## Reproduction Assessment

Yes. A malformed persisted or runtime `agentTurn` payload with an object-valued `message` reproduces the observable failure because current main forwards that value unchanged and the isolated runner interpolates it into the prompt; #54579 also gives a manual cron add/run scenario, though the object-shape source still needs confirmation.

## Solution Assessment

No, not as written. Defensive extraction is a reasonable part of the fix, but logging serialized cron prompt content and omitting regression coverage make this PR unsafe to merge without revision; logging only shape metadata or keys would be the safer fallback.
