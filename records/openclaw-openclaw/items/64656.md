## Summary

Keep this PR open. Current main has overlapping failover/reset and bounded overflow recovery behavior, but it still does not contain the PR's proposed dedicated failover state or LCM conversation identity split, and the branch has unresolved correctness findings that need maintainer review before merge.

## What This Changes

The PR moves auto-fallback persistence into `SessionEntry.failoverState`, adds `memoryConversationKey` for `/new` versus `/reset` context-engine identity, and extends embedded-runner overflow diagnostics and recovery limits.

## Best Possible Solution

Rebase and narrow the branch into a maintainer-reviewed design: choose whether OpenClaw should keep the current source-field failover model or migrate to `failoverState`, implement the LCM identity split consistently while preserving legacy `/reset` continuity, and keep only overflow diagnostics or breaker behavior that improves current bounded recovery.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use one context-engine key for every lifecycle call:** `src/agents/pi-embedded-runner/run.ts:876`
  - body: Switching overflow compaction to `memoryConversationKey` while bootstrap, assembly, finalization, ingest, and maintenance can still use `sessionKey` makes the context engine operate on two conversations. Once the new key exists, recovery can compact a different thread than the prompt being assembled, so the active overflow may not be reduced.
  - confidence: 0.9
- **[P2] Preserve legacy identity on first reset:** `src/gateway/session-reset-service.ts:604-606`
  - body: For existing sessions without `memoryConversationKey`, `/reset` assigns a new UUID instead of continuing to use the old `sessionKey` identity. That changes reset semantics for legacy sessions and can detach their existing LCM conversation even though the new comment says reset should preserve continuity.
  - confidence: 0.88
