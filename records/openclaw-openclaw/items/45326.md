## Summary

Keep open. Current main still clears the TUI editor before any active-run/busy acceptance check and the normal send path still sends chat while `state.activeChatRunId` may already be set. The issue remains a valid narrow TUI behavior bug; the adjacent merged TUI reconciliation work in #59800 did not add the active-generation submit guard, and the direct prior implementation PR #45586 was closed unmerged.

## What This Changes

The issue requests that TUI input typed or pasted during active model generation either interrupt explicitly or be blocked/preserved so stale text cannot be swallowed and replayed into a later turn.

## Best Possible Solution

Add a TUI-local guard for normal chat submissions before `editor.setText("")`, preserve the typed draft when blocked, and surface a short busy/abort hint. Keep slash commands and bang-lines usable during active runs, and avoid changing gateway `chat.send` concurrency or queue semantics unless maintainers make a separate product decision.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit normal text while `state.activeChatRunId` is non-null: current production wiring does not expose a busy guard, the submit handler clears the editor first, and `sendMessage` does not reject the active-run state.

## Solution Assessment

Yes. The narrowest maintainable fix is at the TUI submit boundary before the editor is cleared; a guard only inside `sendMessage` would still lose the draft, and a gateway-wide concurrency ban would be broader than the reported TUI UX bug.
