## Summary

Keep open. Current main still clears TUI editor input before any busy-state check, wires the submit handler without an active-run guard, and `sendMessage` sends normal chat input without rejecting `state.activeChatRunId`. The thread's diagnosis remains accurate: current main has adjacent optimistic-message/history reconciliation work, but it does not implement the requested active-generation submit block or draft preservation.

## Best Possible Solution

Implement a TUI-side normal-message submit guard while `state.activeChatRunId` or an optimistic send is pending, preserve the editor draft when blocked, surface a short status/system hint such as pressing Esc or `/abort` to cancel, and add focused regression tests. Keep explicit control paths like Esc and `/abort` available; do not change Gateway `chat.send` concurrency semantics unless a separate product decision says concurrent sends should be forbidden globally.
