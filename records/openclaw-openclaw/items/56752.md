## Summary

Keep open. Current main still does not provide native desktop multi-window chat sessions: the macOS WebChat manager stores a single normal window controller, and opening a different session closes that existing window before creating another. The Gateway chat path is already non-blocking and session-scoped, so the remaining gap is primarily in the macOS/WebChat UI surface rather than a reason to close this issue.

## Best Possible Solution

Keep the issue open and implement the feature in the macOS/WebChat surface: replace the singleton normal chat window slot with a collection of `WebChatSwiftUIWindowController` instances keyed by window/session identity, add an explicit “new/open chat window” path that does not close existing windows, keep each window’s `OpenClawChatViewModel` subscribed to its own session events, and add macOS tests covering two simultaneously open chat windows and independent active runs. Reuse the existing non-blocking `chat.send`, `chat.history`, `chat.subscribe`, and `chat.abort` contracts rather than adding core owner-specific behavior unless a concrete Gateway API gap appears.
