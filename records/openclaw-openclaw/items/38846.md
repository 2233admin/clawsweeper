## Summary

Keep this PR open. Current main does not implement the proposed exported Windows argument validator, null/control-character checks, or PowerShell-pattern guard, but the PR as written is not safe to merge because it rejects normal Windows paths, slash switches, equals-style flags, and risks leaking full argument values in errors.

## What This Changes

This PR modifies `src/process/exec.ts` to broaden Windows cmd.exe argument rejection, add null/control-character and PowerShell-pattern validation, and export a Windows command-argument validator.

## Best Possible Solution

Keep this PR open for maintainer/security review or replace it with a narrower Windows command-hardening PR. The fix should stay scoped to real cmd.exe wrapper hazards, preserve path separators, scoped package specs, slash switches, and equals flags, avoid applying PowerShell-specific policy outside PowerShell execution, redact argument values in errors, and add focused Windows regression tests.

## Risks / Open Questions

- Merging the PR as written would likely break Windows `.cmd`/`.bat` execution because backslash is rejected even though resolved Windows command paths commonly contain it.
- Rejecting `/` and `=` would break valid Windows switches, scoped package names, install specs, and existing npm flags used by OpenClaw update/install paths.
- The PR’s validation errors include full argument values, which can expose secrets or tokens in exception messages and logs.
- Applying PowerShell cmdlet-pattern policy inside the cmd.exe wrapper path may block unrelated legitimate arguments and mixes shell-specific policies.
