## Summary

Keep this issue open. The discussion narrowed the regression to malformed persisted cron rows, and current `main` still has both backend and Control UI paths that assume every row has a valid `schedule.kind` and `payload.kind`; the KeepMyClaw state-snapshot workaround mentioned in the thread can help diagnose divergence, but it does not fix OpenClaw's own dashboard crash path.

## What This Changes

The issue reports that persisted cron jobs can run and appear in the CLI while the Control UI dashboard renders empty or crashes when cron rows are missing required schedule or payload data.

## Best Possible Solution

Harden persisted cron loading so rows missing required `schedule` or a valid payload are inferred when safe, otherwise quarantined or surfaced with clear diagnostics, and make the Control UI cron dashboard/edit paths defensive so one bad row cannot blank the whole page.

## Reproduction Assessment

Yes. The issue discussion gives a high-confidence malformed `jobs.json` reproduction path, and current `main` has direct `schedule.kind`/`payload.kind` dereferences on the dashboard render path that match the reported browser exception.

## Solution Assessment

Yes. The narrow maintainable fix is to validate or quarantine malformed persisted rows at the cron store boundary, then add defensive Control UI rendering as a second line of protection; changing generated `dist` files or only optional-chaining one UI call site would not solve the underlying state contract.

## Repair Work Prompt

Fix the Control UI cron dashboard crash tracked by https://github.com/openclaw/openclaw/issues/51871. The observable bug is that malformed persisted `~/.openclaw/cron/jobs.json` rows, especially rows missing `schedule` or with missing/incomplete `payload.kind`, can make cron rows fail to load/render and blank the Control UI dashboard with `Cannot read properties of undefined (reading 'kind')`. Keep the fix in source files, not generated `dist` assets. Start at `src/cron/service/store.ts` and `src/cron/normalize.ts`: infer legacy schedule/payload shapes when there is enough data, and quarantine/skip or mark unrecoverable rows with clear diagnostics rather than exposing invalid `CronJob` objects to callers. Then harden `ui/src/ui/controllers/cron.ts`, `ui/src/ui/views/cron.ts`, and `ui/src/ui/app-render.ts` so one malformed row cannot break filtering, suggestions, row rendering, or edit/clone form state. Add regression tests for store load/list behavior and Control UI rendering/controller helpers with malformed rows. Do not silently delete user jobs, do not edit generated locale bundles unless English UI copy changes, and add a user-facing changelog fix entry with reporter credit if behavior changes are user-visible.
