## Summary

Keep this issue open. The discussion narrowed the report to malformed persisted cron rows, especially missing `schedule` or incomplete/missing `payload`, and current `main` still lets those rows reach Control UI code paths that dereference `schedule.kind` and `payload.kind` directly. Recent CLI/startup hardening covers some malformed-row cases, but it does not close the dashboard crash class reported here.

## Best Possible Solution

Keep open. Harden persisted cron loading in `src/cron/service/store.ts` and `src/cron/normalize.ts` so rows missing required `schedule` or a valid `payload.kind` are inferred when safe or quarantined with clear diagnostics. Then make the Control UI defensive in `ui/src/ui/controllers/cron.ts`, `ui/src/ui/views/cron.ts`, and `ui/src/ui/app-render.ts`, and add regression coverage for `cron.list`/store load plus Control UI rendering with malformed `jobs.json` rows.
