## Summary

Keep open. Current main still has display-layer filtering for Control UI/WebChat heartbeat noise, but the reported underlying path is not fully fixed: internal exec/heartbeat prompts can still be generated as normal heartbeat turns, are not tagged with internal-system provenance at origin, and related open PRs are still covering different parts of the contamination path.

## What This Changes

The issue requests that Control UI consume async completion and heartbeat-only system events internally instead of rendering or persisting their raw orchestration text in visible chat.

## Best Possible Solution

Keep the issue open until maintainers choose and land a coordinated fix that tags internal heartbeat, cron, and exec turns at origin, applies history/SSE/Control UI suppression from that provenance, suppresses no-op system-event replies, and removes redundant subagent exec heartbeat wakes without hiding meaningful user-facing async completion summaries.

## Reproduction Assessment

Unclear: there is no minimal deterministic live repro, but the issue comments provide transcript-store evidence and current source shows the internal exec prompt and heartbeat wake path. A focused repro would trigger a metadata-only async exec completion in Control UI and inspect both chat.history and the raw session transcript.

## Solution Assessment

No: the current display filtering is a mitigation, not the best complete solution for the reported transcript/session contamination. The maintainable fix should be source-level internal provenance plus no-op delivery suppression, coordinated with the overlapping open PRs.
