## Summary

Keep open. Current main contains useful chat-history and Control UI display filters for heartbeat prompts and `HEARTBEAT_OK`, but the broader transcript contamination report is still valid: internal exec/heartbeat prompts are still generated as ordinary heartbeat-run input, `internal_system` provenance is not wired into the heartbeat path, and overlapping open PRs are still the likely fix path.

## What This Changes

Async completion and heartbeat-only system events should be consumed internally instead of being stored or rendered as normal Control UI/WebChat conversation messages.

## Best Possible Solution

Land a coordinated fix that tags heartbeat, cron, and exec-event turns with source-level internal provenance, suppresses no-op heartbeat/system replies before delivery, and keeps gateway history/SSE/Control UI filtering as defense-in-depth rather than the primary correctness boundary.

## Reproduction Assessment

Unclear: there is not a minimal deterministic repro, but the issue comments provide transcript-store evidence and current source/tests expose the relevant paths. A focused repro would enqueue a metadata-only async exec/system event for a visible WebChat session and verify both transcript persistence and `chat.history`/SSE output.

## Solution Assessment

No: the current display filtering is a mitigation, not the best complete fix for the reported source contamination. The safer direction is provenance at origin plus no-op delivery suppression and subagent wake guards, with #68262, #66749, and #73785 reviewed together.
