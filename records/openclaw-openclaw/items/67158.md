## Summary

Keep open. Current main still does not implement a provider-boundary fix for the ChatGPT/Codex-account unsupported-model rejection; it recognizes and synthesizes Codex model rows, but the exact backend entitlement error is only treated as a known live-test blocker, not as account-specific model-unavailable status, fallback, or diagnostics. The issue discussion also has a maintainer-facing comment leaving this as a distinct OpenAI Codex provider-boundary track and a later 2026-04-27 report that similar runtime rejection still happens on OpenClaw 2026.4.24 with gpt-5.5.

## What This Changes

The issue requests that OpenAI Codex models other than the working gpt-5.4 route stop hard-failing under ChatGPT/Codex OAuth, or be handled by OpenClaw instead of surfacing the upstream unsupported-model rejection.

## Best Possible Solution

The OpenAI Codex provider should treat the exact backend unsupported-ChatGPT-account model error as an account-specific model-unavailable/entitlement condition, let configured fallback and status diagnostics respond clearly, and document the limitation without hiding models from accounts that may still be entitled to run them.
