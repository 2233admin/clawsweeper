## Summary

Keep this open. Current main has partial mitigation around QMD update/embed scheduling, but it still lacks the requested supported Apple Silicon local-embedding path: built-in GGUF embeddings cannot select CPU/GPU/Metal backend, local batches still run concurrently against one node-llama-cpp context, and public docs do not give Metal-safe or CPU-only guidance. The keepmyclaw backup workaround linked in the discussion is useful recovery context, but it does not close the OpenClaw reliability/configuration gap.

## What This Changes

The issue asks for a supported Apple Silicon local-embedding path that avoids ggml-metal gateway crashes, including CPU/GPU backend guidance, safer batch behavior, and non-fatal failure handling.

## Best Possible Solution

Keep this as the canonical Apple Silicon local-embedding stability tracker until OpenClaw has a documented macOS support matrix for built-in local and QMD embeddings, a validated node-llama-cpp CPU/GPU/backend config path, bounded or serialized local GGUF embedding calls, and a gateway-safe failure boundary or fallback for native embedding crashes.

## Reproduction Assessment

Yes. The issue and follow-up comments provide a high-confidence reproduction shape using Apple Silicon, EmbeddingGemma GGUF, node-llama-cpp/ggml-metal, gateway restart/shutdown or QMD semantic embed/vsearch cleanup, and SIGABRT/non-zero exit behavior; current-main inspection confirms the missing controls are still present.

## Solution Assessment

Unclear. The existing QMD safeguards and PR #60347 address useful slices, but the best maintainable fix needs maintainer design across config schema, docs, node-llama-cpp backend selection, local batching semantics, and crash isolation rather than closing on the current partial state.
