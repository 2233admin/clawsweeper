## Summary

Keep this issue open. Current main still leaves the reported Apple Silicon local-embedding stability and guidance problem unresolved: the built-in local provider still runs node-llama-cpp in-process, still batches local embeddings with Promise.all, and exposes/docs only modelPath, modelCacheDir, and contextSize for local GGUF embeddings. The April 2026 comments provide fresh reproductions, so stale/cannot-reproduce closure is not appropriate.

## Best Possible Solution

Keep this issue open as the canonical Apple Silicon local-embedding stability and guidance tracker. The best path is to land or replace the narrow GPU/backend configuration work, document supported macOS CPU/GPU modes for local embeddings, serialize or otherwise constrain local GGUF embedding batches, and decide whether native local embedding failures need worker/process isolation or another gateway-safe containment boundary.

## Risks / Open Questions

- A native ggml/Metal abort can still terminate the gateway process because the built-in local provider has no process boundary around node-llama-cpp.
- The current docs do not give users a supported Apple Silicon CPU-only or GPU-backed local embedding path, so users are still relying on ad hoc LaunchAgent environment changes and upstream behavior.
- A narrower GPU/backend config PR such as #60347 may address part of the request, but closing this issue before a merged fix would lose tracking for docs, batch serialization, and crash containment.
