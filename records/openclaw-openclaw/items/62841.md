## Summary

Keep this PR open, but do not merge it as-is. The branch is useful source work for the linked `/activate` request, but the final regex regresses the existing `/activation` spelling, and the alias is not registered with command detection, which group-channel mention bypass relies on.

## What This Changes

The PR adds `/activate` as an alias for `/activation` in auto-reply group activation command parsing and adds unit coverage for the alias forms.

## Best Possible Solution

Revise the PR so both `/activation` and `/activate` work end to end: preserve the existing spelling in the parser, add `/activate` to the activation command text aliases so command detection and group mention bypass recognize it, and cover both parser and `hasControlCommand()` behavior in focused tests.

## Reproduction Assessment

Yes. A direct regex probe against the PR-head pattern reproduces the regression: `/activate mention` matches, while `/activation mention` does not.

## Solution Assessment

No. A parser-only regex change is not the best fix; the safer narrow path is to accept both spellings in parsing and register `/activate` as a real text alias so dispatcher/detection paths agree.

## Review Findings

- **[P2] Preserve `/activation` when adding the alias:** `src/auto-reply/group-activation.ts:31`
  - body: The proposed pattern `activate(?:ion)?` matches `/activate` and `/activateion`, but not the existing `/activation` command because `activation` drops the `e` before `ion`. This regresses the documented command and the existing parser test for `/activation mention` would fail; use a pattern that accepts both spellings, such as `activat(?:e|ion)`.
  - confidence: 0.97

  - body: The new tests only call `parseActivationCommand()`, but group channels use `hasControlCommand()` from the command registry to bypass mention gating and debounce command messages. Since the activation registry entry still only exposes `/activation`, `/activate mention` can still be skipped before this parser runs; add `/activate` to the activation text aliases and cover detection.
  - confidence: 0.86
