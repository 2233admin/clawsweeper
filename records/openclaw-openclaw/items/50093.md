## Summary

Keep open. Current main has a useful narrow WhatsApp reconnect mitigation for Baileys `append` upserts, but it does not implement the requested monitored-chat cursor/backfill after reconnection. The issue discussion's proposed per-JID last-seen cursor and gap warning are still relevant: current code only consumes events Baileys already emits, processes recent append messages within a 60-second grace window, and skips older append history.

## Best Possible Solution

Keep this issue open and treat the current append handling as a partial mitigation. The best path is a WhatsApp-plugin design for bounded reconnect catch-up: persist last-seen message state per account/chat, validate the Baileys/WhatsApp Web history mechanism for fetching a gap, dedupe by message ID before enqueueing, limit the replay window, and log a clear gap warning when reliable provider backfill is unavailable. Add focused tests beside the append-upsert and reconnect suites, and update WhatsApp docs/config only if a user-facing setting is introduced.

## Risks / Open Questions

- Baileys exposes append/history events and an on-demand history helper, but the upstream helper shape is not a clear after-cursor API; a fix needs provider validation before promising reliable gap recovery.
- Broadening append processing beyond the current 60-second grace window could replay stale history and cause unexpected replies unless paired with durable message-id dedupe and clear product rules.
- A useful interim gap-detected warning may be easier than full recovery, but that is a product decision because it changes operator expectations without actually restoring missed messages.
