## Summary

Keep open. Current main still lacks the requested generic RPM/RPD quota contract, persisted minute/day quota counters, and pre-request quota-aware skip that writes exact cooldownUntil windows. The adjacent OpenAI Codex WHAM probe and shorter reactive cooldowns do not solve this provider-generic fallback behavior, and related #38775 remains broader rather than a clear superseding canonical issue.

## What This Changes

Request to add per-provider rpm/rpd quota configuration so model fallback can preemptively skip quota-exhausted providers and set precise cooldownUntil windows instead of relying on generic reactive cooldowns.

## Best Possible Solution

Keep this open as a core auth-profile/model-fallback enhancement. Maintainers should first agree on the generic provider quota contract, then implement persisted minute/day quota state, exact reset-time calculation, pre-request candidate skipping that writes cooldownUntil, config schema/help/docs updates, and focused tests. Coordinate with #38775, but only close as duplicate if maintainers explicitly make that broader issue the canonical umbrella.

## Risks / Open Questions

- Adding rpm/rpd is a public config/API contract change that needs schema, help, docs, generated config artifacts, and provider-plugin boundary agreement.
- Quota counters and cooldown writes would need to use the auth-profile store locking/update path to avoid races across concurrent runs.
- The existing OpenAI Codex WHAM probe is provider-specific and post-failure; treating it as the generic solution would leave other low-quota providers unresolved.
