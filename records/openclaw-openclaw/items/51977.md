## Summary

Keep this issue open. Current main has narrowed the cron amplification path by persisting isolated cron state only under the stable cron session key, but the central retained-prompt problem remains: `SessionSkillSnapshot.prompt` is still part of persisted session state, normal reply and CLI paths still write it, reset and follow-up paths still carry it, and the session store still serializes and caches the whole entry graph.

## Best Possible Solution

Keep this issue open and fix the root persistence model: make durable session skill snapshots metadata-only, store skill list/filter/version/hash rather than rendered prompt text, rebuild or share the rendered prompt lazily by hash/version/filter, strip full snapshots from reset and queued follow-up payloads, and add regression coverage proving `sessions.json`, reset, follow-up, CLI, normal reply, and cron paths do not retain large rendered skill prompts. Maintainers may also choose to consolidate this with #55334 explicitly, but that should be a deliberate duplicate/supersession action.
