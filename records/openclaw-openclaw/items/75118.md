## Summary

Keep open. Current main still lacks the requested per-server remote MCP caller-context opt-in, so this PR is not obsolete; however, the PR head has a blocking header-preservation bug on the new merge helper and is currently dirty against main on a security-sensitive caller/session forwarding boundary.

## What This Changes

The PR adds an opt-in `mcp.servers.<name>.injectCallerContext` setting for bundled CLI remote MCP URL servers, injects OpenClaw caller/session headers for trusted opted-in servers, updates placeholder env handling, and adds config schema, changelog, generated baseline, and unit tests.

## Best Possible Solution

Keep the per-server owner/runtime opt-in design, fix the header merge so every existing header entry is preserved while only missing caller headers are added case-insensitively, then rebase and run the targeted bundle MCP tests plus config docs and changed-gate validation before maintainer merge review.

## Reproduction Assessment

Yes. The feature gap is reproducible on current main by configuring a bundled CLI remote MCP server and observing there is no `injectCallerContext` field, schema entry, or merge helper; the PR bug is reproducible in the new helper with an opted-in URL server that has an existing numeric or boolean header.

## Solution Assessment

No, not as written. The per-server owner opt-in is the narrow maintainable direction, but the implementation must preserve the full existing header object and rebase cleanly before it satisfies the stated non-destructive contract.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve non-string headers during caller injection:** `src/agents/cli-runner/bundle-mcp-caller-context.ts:53-54`
  - body: `McpServerConfig.headers` and the zod schema allow `string | number | boolean`, but this helper normalizes existing headers through `normalizeStringRecord` and then assigns that filtered copy back to `server.headers`. An opted-in remote server with an existing numeric or boolean header loses it when caller context is enabled, which breaks the promised "existing headers win" behavior. Build the lowercase key set from all existing header names and merge caller headers into a copy of the original header object.
  - confidence: 0.88
