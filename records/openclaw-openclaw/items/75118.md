## Summary

Keep open for maintainer review. Current main does not implement per-server caller-context injection for remote bundled MCP servers, and this PR is the active implementation for the paired open feature request, but the proposed header merge has a concrete bug that can defeat the stated “operator headers always win” contract.

## What This Changes

This PR adds an opt-in `mcp.servers.<name>.injectCallerContext` config flag, caller-context header merging for bundled CLI MCP configs, placeholder environment population, schema/changelog updates, and unit coverage across Claude, Codex, and Gemini bundle MCP paths.

## Best Possible Solution

Keep the per-server, default-off API direction, but make the merge truly non-destructive by preserving the original header map and detecting existing caller-header names case-insensitively before adding defaults. Add tests for alternate header casing and accepted non-string header values, then run the targeted bundle MCP and config schema checks before merge.

## Reproduction Assessment

Yes. A source-level reproduction is clear from the patch: configure a remote MCP server with `injectCallerContext: true` and an existing `X-Session-Key` or non-string `x-session-key` header, then the new merge path does not treat that operator header as present before adding the lowercase caller placeholder.

## Solution Assessment

No, not as currently patched. The per-server opt-in is the right narrow shape, but the implementation must preserve user headers under HTTP’s case-insensitive header semantics before this is the best maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve existing headers before injecting caller context:** `src/agents/cli-runner/bundle-mcp-caller-context.ts:40-42`
  - body: The merge rebuilds `headers` through `normalizeStringRecord` and then checks exact key names. A valid config with `X-Session-Key` or a non-string `x-session-key` will not be treated as present, so the generated config can add `x-session-key` anyway and violate the PR's “user headers always win” contract. Preserve the original header object and compare header names case-insensitively before adding defaults.
  - confidence: 0.88
