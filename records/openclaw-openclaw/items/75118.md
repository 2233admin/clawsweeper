## Summary

Keep open. Current main does not implement this feature, and the PR is the open implementation paired with the still-open feature issue #75116. The branch is useful, but it adds a caller/session header forwarding path to remote MCP URLs and needs maintainer resolution of the bundle-plugin trust boundary before merge.

## What This Changes

The PR adds a per-server `mcp.servers.<name>.injectCallerContext` option for bundled MCP configs, injects OpenClaw caller headers into opted-in remote MCP servers, populates placeholder env when needed, and updates config schema, docs baseline, changelog, and bundle-MCP tests.

## Best Possible Solution

Keep this PR open and have a maintainer decide the trust model: either restrict caller-context forwarding to explicit owner-managed config, or intentionally bless bundle-provided opt-in with matching docs and tests before merging.

## Reproduction Assessment

Yes. A static reproduction is enough: a bundle-provided remote MCP server can set `injectCallerContext: true`, and the PR applies caller-context injection after bundle defaults and owner config are merged into the same map. Current main also has no first-class `injectCallerContext` key, so the feature itself is not already implemented.

## Solution Assessment

Unclear. The per-server, default-off option is a narrow shape for owner-managed remote MCP config, but applying it to the whole merged bundle layer is not clearly the safest solution unless plugin-bundle opt-in is an intentional trusted-plugin capability.
