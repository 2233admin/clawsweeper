## Summary

Keep open. Current main still does not implement the systemd service-scope behavior this PR is trying to address. The active `openclaw gateway restart` lifecycle path remains user-scope systemd only, and the older restart helper still tries `systemctl --user restart` before `systemctl restart` without detecting or reporting the active scope. The PR also is not merge-ready as-is because it bundles an unrelated Docker mount validation change and review comments identify a wrong-scope selection risk.

## Best Possible Solution

Keep this PR open for maintainer action, but require a focused revision or replacement patch against current main. The best fix belongs in the actual gateway lifecycle surfaces, especially `src/daemon/systemd.ts`, with any needed `src/infra/restart.ts` update kept separate. It should prefer the running unit over enabled-only state, report the selected scope, cover restart/stop/status behavior for system-scope units, and drop the unrelated Docker validation change. #44772 is useful prior art, but it is closed and unmerged, so it is not a shipped implementation.

## Risks / Open Questions

- The PR should not be merged as-is: the Docker setup validation removal is unrelated to systemd scope detection and weakens mount/YAML safety review surface.
- The proposed detection priority can choose an enabled-but-inactive unit before an active-only unit, which could restart or start the wrong service scope.
- Current main has system-level service discovery in status/doctor paths, so a final fix should explicitly define when OpenClaw may operate on externally owned system units instead of blindly falling back.
