## Summary

Keep open. Current main still lacks any cron postcondition contract or scheduler-owned verifier that can turn a clean isolated agent session into a deterministic failure/retry/escalation path, and existing maintainer-side review context already routes the issue to follow-up rather than cleanup closure.

## What This Changes

The issue requests an optional cron `postconditions` field that runs a deterministic post-run shell check for isolated agent jobs, records verifier state, injects retry context, and escalates after configured failures.

## Best Possible Solution

Add a reviewed, additive cron post-run verification design: typed config/state/protocol support, bounded deterministic verifier execution after isolated agent completion and before state application, durable run-log/UI/CLI/docs projection, retry context injection, and integration with existing retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported fail-capable cron post-run hook that can request scheduler-owned retry or escalation.
