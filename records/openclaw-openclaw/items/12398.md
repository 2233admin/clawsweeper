## Summary

Keep open. Current main still has no cron postcondition contract or equivalent scheduler-owned verifier, and the recent steipete/ClawSweeper comments already routed this to maintainer follow-up instead of cleanup closure. The request is valid but broad and security-sensitive because the proposed verifier would execute configured host commands after agent completion.

## What This Changes

Add an optional cron-job `postconditions` contract that runs a deterministic post-run check after isolated agent-turn cron jobs and can retry or escalate when the check fails.

## Best Possible Solution

Add a reviewed additive cron post-run verification design across typed job config/state, gateway protocol, runtime execution, run logs, CLI/UI/docs, retry context injection, and failure-alert delivery. If maintainers prefer plugin ownership, first expose a supported post-run hook that can return a scheduler-owned failure, retry, or escalation decision.

## Reproduction Assessment

Not applicable as a normal runtime bug reproduction; this is a missing feature. The absence is high-confidence from current-main searches plus inspection of cron types, closed gateway schemas, detached execution, state application, docs, UI/CLI surfaces, and plugin hooks.

## Solution Assessment

Unclear. The proposed deterministic shell-check shape addresses the reported verification gap, but the safest maintainable path needs maintainer design around core contract versus plugin seam and host command-execution boundaries before implementation.
