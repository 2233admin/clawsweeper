## Summary

Keep open. Current main still computes Telegram DM session keys from the active routing config, so changing `session.dmScope` from `per-channel-peer` back to `main` still sends the same DM to `agent:<agentId>:main` and can bypass the fresher `/new` session created under the isolated key. I found inverse cleanup for stale main-session delivery metadata, but not a migration, rebind, fresher-session preference, or warning for isolated-to-main routing changes.

## Best Possible Solution

Keep this issue open and fix the generic session/routing behavior rather than closing as already implemented. The strongest path is to persist reset intent or conversation-to-session binding metadata when `/new` rotates a session, then on later dmScope/routing changes either rebind the fresh session to the new canonical key, migrate the fresh session, or emit a clear diagnostic/warning. Add regression coverage for `per-channel-peer` to `main` after `/new`, while preserving the existing main-to-isolated cleanup behavior.
