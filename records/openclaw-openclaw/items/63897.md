## Summary

Keep this issue open. Current main still has the reported cleanup safety hole: `sessions cleanup --fix-missing` checks one `resolveSessionFilePath()` result, which can prefer a stale persisted `sessionFile`, while normal transcript readers scan fallback candidates including `<sessionId>.jsonl`. A closed unmerged PR targeted this issue, and an open related PR targets the same cleanup fallback for #50248, but no equivalent fix has landed on main.

## Best Possible Solution

Keep this issue open until #69954 or an equivalent patch lands. The fix should make `pruneMissingTranscriptEntries()` use the same candidate semantics as normal transcript readers, keep explicit custom transcript paths safe, repair stale generated `sessionFile` metadata when the canonical transcript exists, and add focused cleanup tests for stale generated path plus valid `<sessionId>.jsonl`, valid custom path, and both-missing cases.
