## Summary

Keep open. Current main still accepts and documents `dmHistoryLimit` for iMessage, but the iMessage runtime only tracks and injects supplemental history for group turns; the reporter’s `attributedBody` observation is useful implementation evidence, but the confirmed current-main bug is a broader DM-history contract gap rather than an identifiable direct `chat.db` `text`-column query path.

## What This Changes

Fix iMessage direct-message history so `dmHistoryLimit` provides decoded conversation context instead of blank or missing outbound history entries.

## Best Possible Solution

Maintainers should make the iMessage contract explicit: either implement a decoded DM-history adapter inside the iMessage plugin using `imsg` output that preserves outbound message text, or remove iMessage from the documented supported `dmHistoryLimit` list while keeping backward-compatible config acceptance if needed.

## Reproduction Assessment

Yes for the current-main contract gap: a direct iMessage payload has `is_group: false`, so `historyKey` is undefined and `InboundHistory` is never built. The exact blank outbound `chat.db` symptom is not live-reproduced in this review, but source inspection confirms direct-message history is missing.

## Solution Assessment

Unclear. The reporter’s suggested decoded `imsg` history path is the right runtime direction if iMessage DM history remains a supported contract, but a docs/schema-honesty path is also viable for a legacy integration and should be chosen by maintainers.
