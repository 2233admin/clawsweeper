## Summary

Keep open. Current main still has an iMessage DM-history contract mismatch: config and docs accept/advertise `dmHistoryLimit`, but the iMessage monitor only builds supplemental history for group messages. The reporter's `attributedBody` evidence remains useful for any runtime implementation, but the current source path does not appear to implement direct-message history loading at all, and the related docs-only PR was closed unmerged.

## What This Changes

iMessage `dmHistoryLimit` should inject direct-message history with decoded outbound content instead of producing blank or missing context.

## Best Possible Solution

Maintainers should make the iMessage contract explicit: either implement a decoded iMessage direct-message history adapter in the iMessage plugin, preferably through `imsg` output that preserves outbound message text, or update docs/schema metadata so iMessage is no longer presented as supporting `dmHistoryLimit` while keeping compatibility for existing configs if needed.

## Reproduction Assessment

Yes for the current-main contract gap: direct iMessage dispatches get no `historyKey`, and `buildIMessageInboundContext` only sets `InboundHistory` for groups. The reporter's exact `attributedBody` database behavior remains unverified here, but the source path confirms direct-message history is currently absent.

## Solution Assessment

Unclear. Using decoded `imsg` history is the right implementation direction if OpenClaw wants to keep iMessage DM history support, but the docs-honest alternative also addresses the advertised contract; maintainers should choose that product boundary before automation changes behavior.
