## Summary

Keep open. Current main still has an iMessage DM-history contract mismatch: config/schema/docs expose `dmHistoryLimit` for iMessage and the generic session-history limiter recognizes `imessage`, but the iMessage plugin runtime only wires pending group history. Direct messages get no `historyKey` and no decoded `imsg` history adapter, so the reporter's `attributedBody` detail remains useful implementation guidance rather than proof of a fixed code path. The related docs-only PR #73775 is closed unmerged.

## What This Changes

The issue requests fixing iMessage direct-message history so `dmHistoryLimit` injects decoded outbound message text instead of blank history entries from macOS Messages storage.

## Best Possible Solution

Maintainers should make the iMessage DM-history contract explicit: either add a decoded direct-message history adapter in the iMessage plugin, with coverage for outbound text, or remove iMessage from documented DM-history support while preserving backward-compatible config parsing if needed.

## Reproduction Assessment

Yes for the current-main contract gap: source inspection shows direct iMessage decisions have no `historyKey`, and `InboundHistory` is only populated for group decisions. The reporter's database-specific `attributedBody` evidence remains unverified here but points to the adapter behavior needed if runtime support is implemented.

## Solution Assessment

Unclear. Using decoded `imsg` RPC/history output is the right runtime direction if OpenClaw intends to support legacy iMessage DM history, but a docs-honest de-advertising path is also plausible and was already proposed in #73775, so an automated repair should not choose the contract alone.
