## Summary

Keep open. Current main still accepts and documents iMessage `dmHistoryLimit`, and the generic session-history limiter recognizes `imessage`, but the iMessage runtime only computes group history and only populates `InboundHistory` for group decisions. The related docs-only PR #73775 was closed unmerged, and a maintainer comment already reframed this as an unresolved iMessage contract gap rather than a proven current `chat.db` direct-read bug.

## What This Changes

The issue requests fixing iMessage direct-message history so `dmHistoryLimit` supplies decoded outbound message text instead of blank/null history entries.

## Best Possible Solution

Make the iMessage contract explicit: either add a decoded direct-message history adapter in the iMessage plugin using `imsg` history/RPC output with regression coverage for outbound text, or remove iMessage from documented DM-history support while keeping the accepted config field backward compatible.

## Reproduction Assessment

Yes for the current-main contract gap: configure `channels.imessage.dmHistoryLimit` and the config/docs/generic limiter recognize it, but iMessage direct-message dispatch has no `historyKey` and `InboundHistory` is group-only. The reporter's exact `attributedBody` database evidence remains useful implementation guidance but was not live-reproduced here.

## Solution Assessment

Unclear. Using decoded `imsg` history/RPC output is the right runtime direction if OpenClaw intends to support legacy iMessage DM history, but the docs-honest removal path was also proposed in #73775 and closed unmerged, so maintainers should choose the intended contract first.
