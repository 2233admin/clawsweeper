## Summary

Keep open. Current main still has an iMessage DM-history contract mismatch: configuration and docs advertise `dmHistoryLimit` for iMessage, but the iMessage runtime only builds supplemental history for group messages and has no decoded direct-message history adapter. The reporter's `attributedBody` evidence remains useful for the eventual runtime path, but the source-backed root cause on current main is broader than a direct `message.text` database read.

## What This Changes

The issue requests that iMessage direct-message history honor `dmHistoryLimit` without injecting blank outbound messages when macOS stores outbound content in `attributedBody`.

## Best Possible Solution

Maintainers should choose the iMessage contract explicitly: either add a decoded direct-message history adapter in the iMessage plugin using `imsg` output that preserves outbound text, or stop advertising iMessage as supporting `dmHistoryLimit` while preserving backward-compatible config acceptance where needed.

## Reproduction Assessment

Yes for the current-main contract gap: a direct iMessage dispatch has no `historyKey`, and `buildIMessageInboundContext` only sets `InboundHistory` for groups. The exact blank outbound `attributedBody` database behavior still needs a macOS live check with `imsg` 0.5.x.

## Solution Assessment

Unclear. Using decoded `imsg` history is the right runtime direction if iMessage DM history remains supported, but a docs-honest de-advertising path is also viable and needs maintainer product judgment for this legacy channel.
