## Summary

Current main still leaves the core reported case open. It can recover stale active cron task rows from exact durable cron run history, but that recovery is guarded by `isActiveTask`, so an already-terminal `status: "lost"` cron row can still be reported by task audit instead of being reconciled from the matching finished cron run.

## Best Possible Solution

Keep this issue open and add a narrow maintenance/audit fix for cron rows already stored as `lost`: when a lost cron task has exact run identity and durable cron run logs or job state prove that same run finished, recover the task to the matching terminal status. Rows without exact identity should remain on the existing cleanupAfter/prune path rather than being guessed by job id alone.

## Clownfish Work Prompt

Fix the remaining cron/task audit mismatch for this issue. Current main recovers stale active cron task rows from durable cron run logs/job state, but `resolveDurableCronTaskRecovery()` returns early for terminal rows, so a task already stored as `status: "lost"` can still be reported by `openclaw tasks audit` even when the exact cron run history says the same execution finished. Implement a narrow maintenance/audit repair in `src/tasks/task-registry.maintenance.ts`: for cron tasks already stored as `lost`, only when the task has exact parseable run identity (`cron:<jobId>:<startedAt>` matching `sourceId`) and durable run logs or cron job state prove that same run reached a terminal result, recover it to `succeeded`, `failed`, or `timed_out` using the existing mapping and cleanupAfter behavior. Do not guess by job id alone, do not broadly downgrade unrelated terminal states, and preserve offline CLI audit/runtime-authority semantics for active cron rows. Add regression coverage in `src/tasks/task-registry.maintenance.issue-60299.test.ts` for an already-lost cron task with matching `finished` + `ok` run history becoming `succeeded` and no longer producing a lost audit finding after maintenance; also cover the no-exact-run-id case staying on cleanup/retention. Update docs only if the documented behavior changes, and add a single-line changelog fix entry with `Thanks @andyylin`. Related refs: openclaw/openclaw#63196, openclaw/openclaw#60299, and the partial cron/tasks recovery noted for #71963.
