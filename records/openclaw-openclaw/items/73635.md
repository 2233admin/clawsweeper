Keep open for maintainer review. Current main still misses the embedded-wrapper case the PR targets, and the updated PR head appears to address the earlier mojibake regression without introducing a definite blocking correctness or security issue.

## What This Changes

The PR adds shared extraction for status-prefixed or embedded HTML HTTP errors, routes 401/403 HTML pages to auth-specific copy/classification, and adds formatter, sanitizer, runtime-classification, and truncation regression tests.

## Best Possible Solution

Review the updated PR head, rerun the focused formatter/sanitizer/runtime tests, and land it if those pass; helper deduplication can be folded in only if maintainers want the extra cleanup in this focused fix.

## Reproduction Assessment

Yes. On current main, the focused path is an error string like Error: fetch failed: 403 <!DOCTYPE html><html>...</html> passed through classifyProviderRuntimeFailureKind, formatAssistantErrorText, or sanitizeUserFacingText with errorContext; source inspection shows that status inference and sanitizer ordering miss that embedded status.

## Solution Assessment

Yes, with a minor maintainability caveat. A shared HTML-status extractor plus earlier auth-copy precedence is the narrowest maintainable fix for both direct and embedded HTML auth pages; the duplicated predicate is a cleanup preference rather than a correctness blocker.
