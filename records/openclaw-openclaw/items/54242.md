## Clownfish Work Prompt

Fix the backup hardlink-target safety bug reported here: `openclaw backup create --verify` can pass even when the generated tar contains a hardlink target with `..`, which macOS `bsdtar` rejects during broad extraction. Current main remaps only `entry.path` in `src/infra/backup-create.ts`; `tar` 7.5.13 writes hardlink `entry.linkpath` separately, and `src/commands/backup-verify.ts` only validates entry paths.

Expected fix boundary: make backup archives restore-safe for hardlinked payload files. Prefer a generic archive-safe hardlink treatment: either remap hardlink `linkpath` to the already-remapped archive path for the first file and validate it stays under the archive root/payload, or dereference hardlinks into regular file entries if tar's API makes reliable linkpath rewriting brittle. Extend `openclaw backup verify` to reject unsafe hardlink link targets containing traversal/absolute/backslash forms. Do not implement the broader exclude-pattern feature, service-aware backup redesign, or a restore command in this PR; keep #40786 and #67990 separate.

Tests: add a real archive fixture with two hardlinked files in an included state/workspace path where the pre-fix tar link target would include `..`; inspect `tar.t` entries including `linkpath` and assert no unsafe hardlink target remains. Add a crafted archive/entry test proving `backup verify` rejects an unsafe hardlink target even when all entry paths are safe. Preserve existing duplicate-path and plugin-node_modules behavior.

Docs/changelog: update `docs/cli/backup.md` only if verify semantics or restore expectations are user-visible. Add a single-line changelog fix entry under the active version with credit to @amzzzzzzz. Do not change dependency versions or lockfiles unless a maintainer explicitly approves.
