## Summary

Keep open. Current main still remaps tar entry names and verifies only tar entry paths, so the reported hardlink `linkpath` target with `..` can still pass `backup verify` while failing broad extraction.

## What This Changes

The issue asks for `openclaw backup create` and `backup verify` to handle hardlink targets containing `..` so backup archives remain restore-safe with common tar extractors.

## Best Possible Solution

Backup creation should emit a restore-safe representation for hardlinked files after OpenClaw's archive path remapping, and backup verification should reject unsafe tar link targets before reporting success, with focused regression coverage for a hardlinked payload whose generated link target would otherwise contain `..`.
