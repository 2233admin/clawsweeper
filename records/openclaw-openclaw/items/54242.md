## Summary

Keep open. Current main still remaps tar entry paths during backup creation but does not rewrite or avoid tar hardlink link targets, and backup verification still validates only entry paths. The related symlink duplicate-path and plugin dependency exclusions reduce some backup archive problems, but they do not cover this hardlink `linkpath` gap.

## What This Changes

The issue requests that `openclaw backup create` and `openclaw backup verify` stop producing or accepting backup archives whose hardlink targets contain traversal segments that broad extraction tools reject.

## Best Possible Solution

Keep the issue open until backup creation emits restore-safe hardlink handling and backup verification rejects unsafe tar link targets, with regression tests for a hardlinked payload whose generated link target would otherwise contain `..`. Keep configurable excludes in #40786 and the broader backup redesign in #67990 as separate tracks.

## Repair Work Prompt

Fix the backup hardlink target safety gap from this issue. Current `openclaw backup create` remaps tar entry paths in `src/infra/backup-create.ts` but leaves hardlink `entry.linkpath` untouched, and `openclaw backup verify` in `src/commands/backup-verify.ts` validates only `entry.path`. Make backup creation emit restore-safe hardlink handling by either rewriting hardlink targets to the remapped archive layout or avoiding hardlink entries in favor of normal file entries, using the `tar` 7.5.13 API/types instead of ad hoc assumptions. Extend verification to inspect `ReadEntry.linkpath` for link entries and reject absolute, backslash, `.`/`..`, or archive-root-escaping link targets. Add focused regression tests for a hardlinked payload whose tar link target would otherwise contain `..`, plus verify tests for unsafe link targets. Keep configurable excludes and broader service-aware backup redesign out of scope. Update docs/changelog only if the fix changes user-visible backup behavior; do not change package versions or dependency overrides.
