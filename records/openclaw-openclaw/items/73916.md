## Summary

Current main has useful Gateway liveness, readiness, diagnostics, config-recovery, and lock-loop hardening, but it does not prove this old-state macOS hang is fixed. The report is credible because the same install worked with a clean isolated state and the reporter preserved the broken state, while source still has several startup phases that can await sidecar/channel work before the normal ready transition completes.

## What This Changes

The issue asks OpenClaw to diagnose and fix a Gateway startup regression where an Intel macOS install with existing `~/.openclaw` state binds `127.0.0.1:18789` but HTTP/WebSocket requests never receive a response.

## Best Possible Solution

Keep the issue open and use the saved broken state or a sanitized/minimized fixture to identify the exact stuck startup phase. The desired end state is that Gateway health/readiness remain responsive, diagnostics name the blocking phase, and any stuck sidecar/channel/state path is bounded or isolated with focused regression coverage.

## Reproduction Assessment

Unclear: the issue has a strong reproduction shape on the reporter's Intel macOS machine using preserved old state versus clean temporary state, but the checkout does not contain that state or a minimized fixture. A high-confidence reproduction requires retesting the saved `~/.openclaw.broken-*` state on current main or extracting a safe reduced config/state trigger.

## Solution Assessment

No: closing on the existing readiness/diagnostics/lock hardening is not the best solution because those changes do not prove the old-state post-bind starvation is gone. The safer path is a diagnostics-driven fix in the owner path that actually blocks startup or the event loop.
