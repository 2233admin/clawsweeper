## Summary

Keep open. Current main still lacks manifest-level conversation-access metadata or a loader/registry merge path, enforcement remains host-config-only, and open PR #72690 is already the implementation candidate for maintainer review.

## What This Changes

Issue requests an additive native plugin manifest field for declaring conversation-content hook access while keeping the existing host config flag compatible.

## Best Possible Solution

Review and land PR #72690 or an equivalent narrow patch that adds manifest request metadata, preserves it through manifest registry/status surfaces, keeps explicit host approval for effective access, and updates hooks/config/manifest docs plus loader and manifest-registry tests.

## Reproduction Assessment

Yes. The code-level reproduction is a non-bundled plugin registering `llm_input`, `llm_output`, `before_agent_finalize`, or `agent_end`: current tests show those hooks are rejected without host config and accepted only with `plugins.entries.<id>.hooks.allowConversationAccess=true`, while manifest parsing ignores `permissions.conversationAccess`.

## Solution Assessment

Unclear for the exact RFC sketch. A plugin-owned manifest value should not grant raw conversation access by itself; the safer maintainable solution is manifest request metadata plus explicit operator approval and documented precedence.

## Security Review

Status: needs_attention

Summary: The issue is security-sensitive because it changes how non-bundled plugins declare access to raw conversation-content hooks.

Concerns:

- **[medium] Preserve operator approval for transcript access:** `src/plugins/registry.ts:1956`
  - body: If `permissions.conversationAccess` becomes an effective grant from a plugin-owned manifest, an untrusted plugin could self-authorize raw conversation hook access. Any implementation should treat the manifest as request/disclosure metadata or define explicit host approval and deny precedence.
