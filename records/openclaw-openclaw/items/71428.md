## Summary

Keep open. Current main still enforces conversation-content typed hooks from per-plugin host config only; the native manifest contract does not parse, preserve, or apply a `permissions.conversationAccess` field. The issue is also paired with open PR #72690 using closing syntax, so this should not be closed before that PR is reviewed and merged or superseded.

## What This Changes

Request to add a manifest-declared `permissions.conversationAccess` field that can replace per-plugin host config schema patches for allowing conversation-content typed hooks.

## Best Possible Solution

Define an additive native manifest permission contract with explicit approval and precedence semantics, then land #72690 or a replacement that updates manifest parsing, registry records, hook policy plumbing, docs, and tests for manifest-only, config-only, both-present, and default-rejection behavior.
