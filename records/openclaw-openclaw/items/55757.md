## Summary

Keep this PR open. Current main still computes and passes the CLI no-output watchdog into normal child spawns even when the resolved prompt is sent through stdin that the child adapter closes immediately. The PR is still a plausible implementation candidate, but it needs a current-main rebase, raw-stdin-presence scoping, live-session preservation, clearer timeout text, and regression coverage before it should land.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase onto current main, disable the CLI no-output watchdog only for normal child spawns where stdin is actually provided and closed, use raw stdin presence or explicit input mode rather than payload truthiness, keep Claude live-session watchdog behavior deliberate, avoid misleading 0s timeout text, and add focused regression tests for stdin-fed child spawns plus overall-timeout preservation.

## Risks / Open Questions

- Landing the PR as-is would still leave an explicit empty stdin payload watched, because the guard is based on stdinPayload truthiness instead of raw stdin presence or explicit input mode.
- A broad rebase could accidentally disable Claude live-session no-output protection, even though that path uses stdin mode with a pipe-open live process and separate turn timers.
- The implementation still needs regression coverage for stdin-fed normal child spawns, including empty-string stdin and proof that the overall timeout still terminates silent runaway processes.
- Security review did not find a supply-chain or permissions concern in the supplied one-file diff; the remaining risk is functional timeout scoping.
