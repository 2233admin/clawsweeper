## Summary

The requested direct video upload workflow is still missing on current main. Adjacent non-image attachment and media-understanding work exists, but Control UI still blocks videos and Gateway routes non-image uploads as agent-readable media paths rather than native video chat content for provider LLM APIs.

## Best Possible Solution

Keep this open for maintainer product/design follow-up. The best path is a first-class video chat attachment design: allow and preview MP4/WebM/MOV in Control UI, define a video-capable chat attachment/content contract, persist transcript metadata, route by provider/model video capability with size/offload policy, preserve media-understanding fallback where appropriate, and cover UI, Gateway parsing, agent dispatch, provider/native media behavior, and docs with focused tests.

## Risks / Open Questions

- A narrow change that only removes the UI video filter would allow video files to be staged as generic media without delivering provider-native video content to the active chat model.
- Video uploads need explicit size/offload, media-store, sandbox, WebSocket/base64 memory, provider capability, cost, and transcript/history design before implementation.
- Existing media-understanding and video-generation surfaces are adjacent and useful, but they could be mistaken for the requested direct video chat attachment workflow.
