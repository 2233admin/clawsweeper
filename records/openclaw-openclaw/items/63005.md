## Summary

Keep this issue open. Current main still has a concrete contract mismatch: inbound explicit raw session keys are preserved as raw keys, while gateway/session-tool lookup and migration canonicalize the same bare key under the default agent and can prune the raw alias that future inbound traffic still uses. The narrower Open WebUI UUID discussion does not cover arbitrary structured keys such as `conversation:pair:...`.

## What This Changes

The issue requests stable continuity for external raw session keys after `sessions_send` or nested-agent activity, without splitting one logical conversation into multiple session records.

## Best Possible Solution

Define one durable store-key contract for explicit external session keys across inbound initialization, gateway resolution, combined-store listing, and mutating session-tool paths. Either preserve raw external keys as canonical when they already have active rows, or keep durable aliases that future inbound traffic can resolve without losing the original `sessionId` and transcript.
