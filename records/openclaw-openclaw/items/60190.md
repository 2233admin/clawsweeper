## Summary

Keep open. Current main still has inbound debouncing, but the shared policy and common channel integrations intentionally keep it text-only; QQ Bot only merges already queued group bursts and does not provide a configurable quiet-window aggregation path for image-plus-text turns, especially DMs.

## Best Possible Solution

Implement a media-aware inbound aggregation path on top of the existing createInboundDebouncer/messages.inbound surface, with opt-in per-channel configuration. It should batch by channel, conversation, and sender; preserve combined text plus media paths/attachments; use the latest message for reply/thread ids; define command bypass or flush behavior explicitly; and add focused tests/docs for QQ Bot plus at least one shared channel integration.

## Risks / Open Questions

- A broad default aggregation delay could add unwanted latency or delay control commands; this should be opt-in or carefully scoped by channel/conversation.
- Existing narrow coalescing behavior in BlueBubbles and Telegram should be preserved while adding any shared media-aware path.
