## Summary

Keep open. Current main still does not implement the PR's central feature: spoken Discord voice phrases that persistently escalate or de-escalate the session model. Main only has static `channels.discord.voice.model` and the normal Discord `/model` picker. The gateway HTTP-route registry part bundled into this PR is already superseded by #47902, but the voice feature remains unmerged and needs a rebase/narrowing pass.

## Best Possible Solution

Keep this PR open only as a feature candidate. The best maintainer path is a fresh, narrow Discord plugin change on current main: drop the obsolete gateway-registry edits, implement explicit spoken model commands in `extensions/discord/src/voice/manager.ts`, consume matched command-only phrases instead of sending them to the agent, persist first-turn overrides correctly, avoid hardcoded policy-sensitive model defaults where configured aliases/allowlists can be used, and add focused voice manager tests plus Discord voice docs if maintainers want the feature.

## Risks / Open Questions

- The PR targets the old `src/discord/voice/manager.ts` path while current Discord voice code lives under `extensions/discord/src/voice/manager.ts`, so it needs a non-trivial rebase.
- The PR bundles a gateway registry fix that main already solved via #47902 with an unmerged voice feature; rebasing as-is could revive obsolete registry changes.
- Security/privacy review: ordinary transcribed speech would be able to persistently switch a session to hardcoded cloud models, so any implementation should use explicit low-false-positive commands, consume command-only phrases, and honor configured/allowlisted model policy.
