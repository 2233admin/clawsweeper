## Summary

Keep open. Current main still does not implement the PR's central feature: spoken Discord voice phrases that persistently escalate or de-escalate the session model. Main supports a static `channels.discord.voice.model` override and the normal Discord `/model` picker, while the unrelated gateway HTTP-route registry fix bundled into this PR is already solved on main by the pinned HTTP-route registry design shipped via #47902.

## Best Possible Solution

Keep this PR open only as a feature candidate. The best maintainer path is a fresh, narrow Discord plugin change on current main: drop the obsolete gateway-registry edits, implement explicit low-false-positive spoken model commands in `extensions/discord/src/voice/manager.ts`, consume matched command-only phrases instead of sending them to the agent, ensure first-turn persistence works when the session entry does not exist yet, honor configured/allowed model references instead of hardcoded policy-sensitive defaults where possible, and add focused voice manager tests plus Discord voice docs if maintainers want the feature.

## Risks / Open Questions

- The PR branch targets the old `src/discord/voice/manager.ts` path while current Discord voice code lives under `extensions/discord/src/voice/manager.ts`, so it needs a non-trivial rebase.
- The PR bundles an already-solved gateway registry fix with the still-missing voice feature; rebasing as-is could reintroduce obsolete registry design instead of using the current pinned-route surface.
- Security/privacy pass: the diff does not add CI, dependency, lockfile, release, or external code-execution changes, but it would let ordinary transcribed speech persistently switch a session to hardcoded cloud models. That needs explicit, low-false-positive commands plus configured/allowed model policy checks before merge.
- Existing review findings remain material: false positives, command text being sent to the agent, and first-turn persistence should be fixed before this feature lands.
