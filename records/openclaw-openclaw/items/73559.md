## Summary

Keep open. Current main still has a credible OpenClaw-owned auth propagation gap for embedded `openai-codex/gpt-5.5` runs: the OAuth token is resolved by the run layer, but the boundary-aware Codex Responses stream path can return the transport without injecting that token into `options.apiKey`. The concrete issue logs match the failure mode of the OpenAI Responses transport being constructed with an empty API key.

## Best Possible Solution

Add a focused regression test for an embedded Control UI-style run using `openai-codex/gpt-5.5` that proves the resolved OAuth profile token reaches the `openai-codex-responses` request, then fix `resolveEmbeddedAgentStreamFn` or the shared boundary-aware transport wrapper so boundary-aware Codex Responses streams receive the same resolved `apiKey` and signal handling as provider-owned streams. Keep adjacent OAuth-scope issues in view, but do not close this issue as a duplicate until a canonical fix covers this exact missing-bearer path.
