## Summary

Current main still has the ordering described in the report: both prompt-side and assistant-side rate-limit failover await auth-profile failure marking before calling advanceAuthProfile. That marking path still takes the auth profile store file lock, so the linked open PR #57283 remains relevant rather than obsolete.

## Best Possible Solution

Keep the issue open and resolve it through a focused runner change: rotate the auth profile before any potentially blocking cooldown-store update in both prompt and assistant failover paths, make the cooldown marking bounded/non-blocking with error logging, and add regression coverage that a delayed markAuthProfileFailure cannot delay advanceAuthProfile on rate_limit failover.
