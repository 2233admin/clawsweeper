## Summary

Keep this PR open. Current main covers the exec default-host PATH problem and the cron deadlock shape through newer designs, but the browser part is still not implemented in the current `extensions/browser` plugin layout: status has no layered diagnostics payload, CLI status does not merge Control UI/auth/origin checks, and non-rate-limit browser HTTP errors still surface upstream body text directly.

## Best Possible Solution

Keep this PR open, but require a split or rebase. Drop the exec and cron changes already covered on current main, then port the still-missing browser layered status diagnostics and bounded HTML upstream-error summarization into `extensions/browser`. Address both existing review comments, keep config-derived diagnostics redacted and timeout-bounded, and verify with focused browser client/server/CLI tests plus a security review for reflected error text and config exposure.

## Risks / Open Questions

- Merging the PR as-is would target stale root browser/CLI paths instead of the current `extensions/browser` plugin layout.
- The browser HTML error summarization touches untrusted upstream response text; the existing review comment about fallback length and tag-stripping robustness should be resolved before any port lands.
- The PR's browser status path adds a config snapshot call; a rebased implementation should keep the snapshot redacted and bounded by a short status timeout to avoid CLI hangs or secret exposure regressions.
- The exec and cron hunks touch command execution and scheduling surfaces that are now covered differently on main, so carrying those older changes forward could reintroduce semantic drift.
