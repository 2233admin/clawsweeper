## Summary

Keep this PR open. Current main already covers the exec PATH problem and the cron self-deadlock shape through newer designs, and it has WSL2 remote-CDP troubleshooting docs, but the layered browser status diagnostics and bounded HTML upstream-error summarization from the PR are still not implemented in the current browser plugin layout.

## Best Possible Solution

Keep this PR open, but require a split or rebase. Drop the exec and cron changes already covered on main, then port the still-missing layered browser status diagnostics and bounded HTML upstream-error summarization into `extensions/browser`. Address the existing review comments, keep config-derived diagnostics redacted and timeout-bounded, and verify with focused browser client/server/CLI tests plus a security pass on error reflection and config exposure.

## Risks / Open Questions

- Merging as-is would be stale against the current `extensions/browser` plugin layout and could reintroduce old `src/browser`/root CLI assumptions.
- The PR's browser HTML summarization has an unresolved robustness/security concern: untrusted proxy HTML fallback text should be capped and sanitized before it reaches CLI output, logs, or agent-visible errors.
- The PR's browser status path adds a `config.get` diagnostic call; the unresolved review comment about using a short timeout should be addressed to avoid status hangs, and the implementation must only use redacted config snapshots.
- The exec and cron parts touch command execution and scheduling surfaces but are already solved differently on main, so a rebase should drop those hunks instead of mixing old semantics with current behavior.
