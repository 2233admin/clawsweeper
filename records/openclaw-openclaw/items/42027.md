## Summary

Keep this PR open. Current main covers the exec default-host PATH problem and the cron self-deadlock shape through newer designs, and it has WSL2 remote-CDP troubleshooting docs, but the PR's layered browser status diagnostics and bounded HTML upstream-error summarization are still not implemented in the current browser plugin layout.

## Best Possible Solution

Keep this PR open, but require a split or rebase. Drop the exec and cron changes already covered on main, then port the still-missing layered browser status diagnostics and bounded HTML upstream-error summarization into `extensions/browser`. Address the existing review comments, keep config-derived diagnostics redacted and timeout-bounded, and verify with focused browser client/server/CLI tests plus a security pass on error reflection and config exposure.

## Risks / Open Questions

- Merging as-is would be stale against the current `extensions/browser` plugin layout and could reintroduce old root `src/browser` / `src/cli` assumptions.
- The PR's HTML fallback summarization has an unresolved robustness/security issue: untrusted proxy HTML fallback text should be capped and sanitized before reaching CLI output, logs, or agent-visible errors.
- The PR's browser status path adds a `config.get` diagnostic call without the short browser-status timeout; a rebased version should keep config-derived diagnostics redacted and timeout-bounded.
- The exec and cron hunks touch command execution and scheduling surfaces but are already solved differently on main, so a rebase should drop those pieces instead of mixing old semantics with current behavior.
