## Summary

Keep open. Current main still drops the auth profile on auto-reply cross-provider fallback and still invalidates CLI session reuse on stored/current auth profile mismatches, so the reported Gemini CLI fallback reset is not resolved.

## What This Changes

The issue reports that `google-gemini-cli` fallback runs lose CLI session context because provider-scoped auth profile identity is unstable across fallback and later CLI session reuse.

## Best Possible Solution

Fix the auto-reply CLI fallback auth path so a `google-gemini-cli` fallback run resolves and stores a stable provider-appropriate auth profile identity for CLI session binding and reuse, while retaining the cross-provider auth isolation from #1156. Add focused regression coverage for fallback-created bindings and stale bindings where Gemini CLI auth profile identity is missing on one side.
