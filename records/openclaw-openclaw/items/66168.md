## Summary

Keep open. Current main does not implement this PR's MCP Apps gateway surface: there are no `mcp.tools.*` or `mcp.resources.*` WebSocket methods, no MCP resource registry/read/list support, and no `_meta.ui` tool metadata in the current MCP loopback schema. The PR is still a plausible core gateway/protocol change, but it needs maintainer review rather than cleanup closure because the discussion contains unresolved security/product questions around account context, resource scoping, and `callerRole` semantics.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to either revise this PR or split it into focused protocol/resource/schema changes that preserve the current loopback policy inputs, define session/resource scoping explicitly, document `callerRole` as non-security metadata if that is intended, and add matching gateway protocol docs plus targeted tests. Close only after a merged implementation lands or a maintainer decides the MCP Apps host/resource surface belongs behind a different core/plugin API design.
