## Summary

Keep #39780 open. Current main still has the reported Control UI gap: the Overview insecure-HTTP recommendation tells users to set `gateway.controlUi.allowInsecureAuth: true` manually and links docs, while the existing base-hash guarded config diff/apply machinery is limited to Config/Quick Settings flows and is not wired into Overview recommendation callouts.

## Best Possible Solution

Keep #39780 open until Control UI recommendation callouts can stage narrow base-hash guarded config changes, show before/after values, offer accept/dismiss controls, apply through the existing config write path, and confirm the saved value. The work likely belongs in a shared recommendation/apply component reused by the insecure HTTP Overview path and future config-hardening suggestions, with focused UI/controller tests and public Control UI docs updated.

## Risks / Open Questions

- `gateway.controlUi.allowInsecureAuth` is security-sensitive; any implementation should keep HTTPS/Tailscale Serve as the preferred path and require explicit confirmation before writing the compatibility toggle.
- The related open issues #38249 and #38420 are useful design context, but they are broader than this focused Control UI recommendation-apply requirement and do not supersede it.
