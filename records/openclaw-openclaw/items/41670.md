## Summary

Keep this PR open. Current main still does not send `encoding_format` on the relevant outbound memory embedding request bodies, and the submitted change is not a complete current-main fix because it edits the `src/memory-host-sdk` mirror path while the active plugin SDK export goes through `packages/memory-host-sdk`. The OpenAI direct and batch embedding paths also build request bodies separately and still omit the parameter. Related #43910 is open, but not merged or shipped.

## Best Possible Solution

Keep this PR open and ask for a current-main rework: add `encoding_format: "float"` where the actual OpenAI-compatible memory embedding request bodies are built, including the active package helper if generic remote providers are intentionally covered and the OpenAI direct and batch bodies. Add focused regression tests for a strict OpenAI-compatible proxy. Alternatively, let #43910 absorb the behavior if maintainers prefer a configurable field, then close this PR after that replacement merges.

## Risks / Open Questions

- Landing the PR exactly as filed may not affect the active plugin SDK helper because the exported implementation is sourced from `packages/memory-host-sdk/src/host/embeddings-remote-provider.ts`.
- Adding `encoding_format` only to the generic helper would still miss the OpenAI direct and batch embedding paths that build request bodies separately.
- Hard-coding `encoding_format` broadly in the shared helper could affect DeepInfra, Mistral, and LM Studio adapters that also use the helper; provider compatibility should be verified or the fix should be scoped to OpenAI-compatible owner paths.
