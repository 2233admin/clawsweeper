## Summary

Keep this PR open. Current main still has the reported channel stop-timeout zombie-task failure, and this PR is a plausible targeted gateway fix, but the latest diff still has a replacement-race status write that should be guarded before merge.

## What This Changes

This PR changes gateway channel task lifecycle handling to clear timed-out task/abort entries, ignore stale task lifecycle status writes, update regression tests, and add a changelog entry.

## Best Possible Solution

Land this PR or an equivalent gateway fix after guarding replacement races and clearing CI: stale timeout task state should be released, live runtime state should only be written by the current task/stop owner, and regression coverage should prove both restart and stale-callback behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Guard the timeout status write against replacement races:** `src/gateway/server-channels.ts:689-694`
  - body: The timeout branch identity-checks the task and abort maps before deleting, but then writes `running: false` unconditionally. If two `stopChannel` calls overlap on the same hung task and a fresh `startChannel` registers after the first timeout returns, the later timeout can still overwrite the fresh task's runtime with the stale timeout error. Skip the status write when `store.tasks` now points at a different task.
  - confidence: 0.82
