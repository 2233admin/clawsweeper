## Summary

Keep this PR open. The overall RuntimePlan/Harness V2 direction still matches the active RFC thread, but the extracted model/auth planner drops current-main auth-profile discovery and default plugin-harness forwarding behavior, so the branch is not ready to merge as-is.

## What This Changes

The PR extracts RuntimePlan embedded-runner helpers, adds neutral embedded-runner barrels and import guardrails, introduces owner-scoped Harness V2 factory registration through the plugin SDK, updates Codex/docs/API baselines, and adds related regression tests.

## Best Possible Solution

Restore current-main auth parity inside the extracted model/auth helper, add focused tests for external CLI discovery and default plugin-harness profile forwarding, add the required changelog entry for the public plugin API, then continue maintainer review of whether the consolidated stack should land whole or be split.

## Reproduction Assessment

Yes for the PR regressions: the source comparison shows current main passes scoped external CLI discovery and forwards plugin-harness auth profiles, while the PR helper omits both. A focused unit reproduction can mock ensureAuthProfileStore and a Codex-style plugin harness with no explicit authProfileId.

## Solution Assessment

No, not yet. The refactor/API direction is maintainable, but the extracted auth helper must preserve existing discovery/forwarding semantics before this is the best way to land the cleanup.
