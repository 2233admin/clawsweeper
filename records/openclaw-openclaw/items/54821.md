## Summary

Keep open. Current main still lacks the requested opt-in thinking-event capability, so the PR is not obsolete, but the branch is not merge-ready because it can corrupt delta-only reasoning streams and can leave pre-registered thinking recipients alive after early dispatch failure.

## What This Changes

The PR adds a `thinking-events` Gateway client capability, routes thinking agent events only to opted-in WebSocket recipients, decouples reasoning streaming from the callback path, and updates related Gateway/agent tests plus generated SDK baseline line numbers.

## Best Possible Solution

Land a corrected, rebased version of the cap-gated thinking-event design: preserve current-main thinking visibility guards, use a deterministic delta/full-snapshot contract for reasoning updates, clean up recipient registries on every early failure and lifecycle path, and document/test the new Gateway capability before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Preserve delta-only reasoning chunks:** `src/agents/pi-embedded-subscribe.ts:648-651`
  - body: This branch also receives native `thinking_delta` fallback chunks where `text` is only the new token. After an initial `"Checking"`, a later `" files"` chunk has no meaningful prefix/suffix overlap, so this branch marks it as a replacement and emits truncated `text`/`delta` instead of accumulating `"Checking files"`. The fix needs an explicit full-snapshot vs delta signal, or a fallback that appends known delta-only events.
  - confidence: 0.88
- **[P2] Clean up pre-registered recipients on dispatch failure:** `src/gateway/server-methods/chat.ts:1557-1558`
  - body: Recipients are now registered before `dispatchInboundMessage`, but if dispatch rejects before any lifecycle event is emitted, the catch/finally path never marks the tool/thinking recipient registries final. Those run-to-connection entries can stay alive for the registry TTL and misroute events if a client run id is reused, so the failure path needs to remove or finalize the pre-registered run.
  - confidence: 0.84
- **[P2] Keep the thinking-level visibility guard:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: Current main only enables reasoning display/streaming when `params.thinkingLevel !== "off"`, but this branch enables `streamReasoning` solely from `reasoningMode === "stream"`. After rebase, a session with thinking disabled could still emit reasoning tokens to opted-in Gateway clients, so preserve the existing `canShowReasoning` guard for both included and streamed reasoning.
  - confidence: 0.81

## Security Review

Status: needs_attention

Summary: The intended capability gate improves privacy, but the PR currently has concrete reasoning-token exposure risks around stale subscribers and the dropped thinking-level guard.

Concerns:

- **[medium] Stale thinking subscribers after early dispatch failure:** `src/gateway/server-methods/chat.ts:1558`
  - body: `chat.send` pre-registers thinking recipients before dispatch, but if dispatch fails before lifecycle `end` or `error`, `thinkingEventRecipients.markFinal` is never called. The stale mapping can outlive the failed run and receive reasoning events for a reused run id.
  - confidence: 0.82
- **[medium] Reasoning can stream when thinking is disabled:** `src/agents/pi-embedded-subscribe.ts:77`
  - body: The PR head drops current-main `params.thinkingLevel !== "off"` gating when enabling reasoning streaming. That can expose reasoning tokens through the new Gateway capability even when the session's thinking level is explicitly disabled.
  - confidence: 0.78
