## Summary

Keep open for maintainer review. Current main still has the reported gap: automatic daily/idle rollover creates a new session without emitting the internal `command:reset` hook that the bundled `session-memory` handler listens to, and this PR is a narrow implementation with focused coverage. It should not be cleanup-closed, though maintainers should decide how to handle the broader linked issue scope and overlapping open PR #50933.

## What This Changes

The PR emits synthetic `command:reset` internal hook events with `auto:daily` or `auto:idle` sources during automatic session rollover, adds rollover/fresh-session tests, and documents the opt-in session-memory behavior.

## Best Possible Solution

Land one canonical narrow fix that emits the existing internal reset hook for automatic daily and idle rollover, preserves workspace scoping and reason markers, and keeps docs/changelog aligned. If the broader hook config schema problem in #11960 remains valid, keep that tracked separately or adjust the closing relationship before merge.

## Reproduction Assessment

Yes. On current main, a stale daily or idle session can be reproduced by seeding a session store entry that evaluates stale, sending the next inbound message through `initSessionState`, and observing that the session rolls over while no internal `command:reset` hook is emitted for `session-memory`.

## Solution Assessment

Yes for the narrow bundled session-memory bug: reusing the existing `command:reset` hook avoids a new public hook contract and keeps explicit reset and fresh-session paths guarded. The only caveat is issue-scope management for #11960 and the overlapping #50933 branch.
