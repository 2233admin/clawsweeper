## Summary

Keep open. Current main has alias resolution in the Gateway `sessions.patch` path for bare aliases, but the Control UI/TUI selection state can still turn a server-reported display alias plus provider into a provider-qualified raw alias such as `lmstudio-moe/MoE`, and the Gateway then treats that as a direct provider/model ref rather than resolving `MoE` to the configured canonical model ID.

## What This Changes

The issue asks for TUI/Control UI mid-session model switching to resolve configured display aliases to canonical provider/model IDs before persisting session overrides.

## Best Possible Solution

Resolve configured model aliases at the shared model-selection boundary before persistence, including provider-qualified display aliases such as `provider/Alias`, and add focused UI/Gateway tests proving the picker and `/model` paths persist the canonical provider/model ID.

## Reproduction Assessment

Yes. The code path is reproducible by a session row with `modelProvider: "lmstudio-moe"`, `model: "MoE"`, and a catalog/config entry whose canonical ID is `qwen3.6-35b-a3b` with alias `MoE`; current UI helpers produce `lmstudio-moe/MoE`, which the Gateway parses as a direct model ref.

## Solution Assessment

Unclear for the reporter's suggested UI-only fix. The safer narrow fix is to resolve aliases in the shared Gateway/model-selection layer and keep the UI picker using canonical option values, with UI tests only to prevent regressions.
