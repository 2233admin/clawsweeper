## Summary

Keep this PR open, but not merge-ready as written. Current main still lacks the requested Bedrock per-model context-window override surface, and this PR is a plausible implementation candidate for the open Bedrock context-window report, but the diff has concrete correctness gaps in stripped-ID regex matching and config schema alignment that need maintainer review alongside the competing family-fallback approach.

## What This Changes

The PR adds an Amazon Bedrock `modelContextWindowOverrides` discovery config field, regex fallback matching and cache-key handling, discovery tests, plugin manifest/docs updates, and a shared config type update.

## Best Possible Solution

Keep the PR open and either revise it so runtime lookup, cache keys, manifest schema, strict config schema, docs, and tests all agree on the regex override contract, or choose the narrower built-in Bedrock Claude family/table fallback if maintainers prefer not to add a regex config surface.

## Reproduction Assessment

Yes. The PR's own added stripped-ID inference-profile test provides a high-confidence reproduction path: with `inferenceProfileId: "us.example.unknown-text-v1:0"` and pattern `^example\.`, the new fallback tests only the raw ID, so the expected override is not applied.

## Solution Assessment

No, not as currently written. Regex overrides can solve the reported user problem, but the patch needs to test normalized candidates consistently and align the shared strict schema; maintainers should also decide whether #73382's built-in family fallback is the better policy.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Test regex overrides against stripped IDs:** `extensions/amazon-bedrock/discovery.ts:175`
  - body: The override fallback only runs `override.pattern.test(modelId)` after the known-table loop. For system-defined inference profiles where `baseModelId` is unavailable, `resolveKnownContextWindow` is called with a region-prefixed ID such as `us.example.unknown-text-v1:0`, so a documented/tested pattern like `^example\.` never matches the stripped candidate. This makes the new stripped-ID test fail and leaves those profile overrides ineffective.
  - confidence: 0.94
- **[P2] Keep the strict discovery schema in sync:** `src/config/types.models.ts:140`
  - body: Adding `modelContextWindowOverrides` to the shared `BedrockDiscoveryConfig` type makes the legacy `models.bedrockDiscovery` compatibility path appear to accept the key, but `BedrockDiscoverySchema` remains strict and still rejects it. Add the same field to the strict schema, or keep the new type scoped to the plugin config path so typed configs do not fail validation before discovery can use them.
  - confidence: 0.78
