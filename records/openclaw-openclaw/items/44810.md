## Summary

Current main still matches the reported failure mode: parent-owned ACP/Codex runs can surface the same generic 60s “may be waiting for interactive input” stall, and the parent relay still cannot distinguish prompt-submitted/no-first-output from a true interactive wait or upstream network/proxy failure. The linked implementation PR #44949 was closed unmerged, and the shipped ACPX adapter pinning work for #73202 is useful adjacent robustness but does not implement the requested classification/diagnostics.

## Best Possible Solution

Keep this issue open and land a focused ACP diagnostics fix: preserve/bridge prompt-submitted and runtime activity signals into parent-owned ACP run events, teach the parent relay to classify no response after prompt separately from interactive wait, surface terminal runtime/network errors to the parent session, and add redacted proxy-env/log-path diagnostics plus targeted tests. PR #44949 is the closest prior candidate but should be revived only as a smaller current-main patch.
