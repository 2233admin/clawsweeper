## Summary

Keep open. Current main still has a plausible cold-start regression in the native `/subagents spawn` path: the slash command reaches `spawnSubagentDirect`, which resolves the active context engine before the spawn path initializes the built-in `legacy` engine. The adjacent subagent cleanup path already performs that initialization before resolving the engine, so this looks like a narrow missed initialization call rather than an obsolete report.

## What This Changes

The issue reports that `/subagents spawn` fails on OpenClaw 2026.04.26 because the default legacy context engine is not initialized.

## Best Possible Solution

Keep this open and fix the native subagent spawn boundary so built-in context engines are initialized before resolving the active engine, with a regression test for a cold default-legacy `/subagents spawn` or `spawnSubagentDirect` path.

## Reproduction Assessment

Yes, at source level: a native `/subagents spawn <agentId> <task>` reaches `spawnSubagentDirect`, then `prepareContextEngineSubagentSpawn`, then `resolveContextEngine` before the spawn path initializes the built-in legacy engine. A focused regression test should reset or isolate the context-engine registry and prove default legacy spawn succeeds.

## Solution Assessment

Yes, the narrowest maintainable fix appears to be wiring `ensureContextEnginesInitialized()` into the spawn-side context-engine resolution path, mirroring the subagent registry cleanup path. A broader registry auto-initialization change would have more global side effects and is not needed to address this report.
