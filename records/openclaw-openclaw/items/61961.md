## Summary

Keep open. Current main does not implement a `readonly` tool profile, and this PR is useful but incomplete: the proposed runtime/catalog additions leave public config types, generated config schema, and Gateway catalog profile schema out of sync, so typed clients or schema-driven config/UI surfaces can reject `readonly` even though the runtime catalog starts advertising it.

## What This Changes

The PR adds a `readonly` tool profile across the core tool catalog, runtime config parsing, gateway tool catalog typing, tests, and tools documentation.

## Best Possible Solution

Land a complete additive `readonly` profile implementation that keeps the runtime tool catalog, exported config types, generated config schema/baseline, Gateway protocol schema and generated clients, Control UI profile options, focused tests, docs, and changelog aligned while preserving existing profile behavior and excluding mutating tools.

## Reproduction Assessment

Yes. Static verification is high-confidence here: current main lacks `readonly`, and the provided PR diff shows newly advertised/runtime `readonly` support without the matching generated config schema, public config type, and full Gateway catalog profile contract updates.

## Solution Assessment

No, not as submitted. The direction is narrow and maintainable, but the best fix must update all public contract and generated surfaces in the same change so runtime, config, Gateway, and UI clients cannot drift.
