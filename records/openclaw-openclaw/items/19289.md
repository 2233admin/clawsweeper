## Summary

Keep this issue open. Current main still does not expose a browser backend/driver registration interface for third-party MCP/custom adapters. The shipped surfaces support replacing the whole bundled browser plugin, managed/CDP profiles, and the built-in Chrome DevTools MCP existing-session driver, but profile config, schema validation, runtime profile resolution, profile creation, SDK browser profile types, and public plugin capability docs remain closed to built-in driver modes.

## Best Possible Solution

Keep this open for maintainer design. The best path is an additive browser backend/adapter extension point in core and the Plugin SDK: define a typed backend contract, allow plugins to register driver ids, broaden browser profile config/schema/help/client/CLI/control routes to accept registered ids, route runtime dispatch through that registry, document the contract in browser and plugin SDK docs, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale. Treat #52190 as backend-specific prior art and #55840 as related extension-relay demand, not as canonical replacements for this API gap.

## Risks / Open Questions

- Closing this now would lose tracking for a real plugin/API gap: third-party code can replace the entire browser tool, but cannot register a profile-selectable backend id for the bundled browser runtime.
- A browser backend adapter seam is security-sensitive because it would cross native plugin execution, browser control, CDP/Chrome MCP, Gateway auth, existing-session access, and SSRF/private-network policy boundaries.
- The likely fix is broad public API work across config, schema/help generation, browser runtime dispatch, Plugin SDK exports, CLI/control routes, docs, and contract tests, so it is not a narrow cleanup PR.
