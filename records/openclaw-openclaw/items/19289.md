## Summary

Keep this issue open. Current main still does not provide a browser backend or driver registration interface for third-party MCP/custom adapters: config/schema/client/runtime surfaces use a closed driver set, profile creation rejects unknown drivers, and the Plugin SDK exposes browser config facades but no browser-backend registration capability. The existing maintainer-side Codex comment and related #52190/#55840 context align with this as a real core/plugin API gap, not a duplicate or ClawHub-only item.

## What This Changes

The issue requests a pluggable browser backend interface so third-party MCP/custom browser drivers can be registered without modifying OpenClaw core.

## Best Possible Solution

Keep this open for maintainer design. The best path is an additive, versioned browser backend/adapter extension point in core and the Plugin SDK: define a typed backend contract, allow plugins to register driver ids, broaden browser profile config/schema/help/client/CLI/control routes to accept registered ids, route runtime dispatch through that registry, document the contract in browser and plugin SDK docs, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale. Treat #52190 and #55840 as related demand signals, not replacements.
