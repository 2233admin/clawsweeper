## Summary

Keep open. Current main still has a Telegram document path where a failed `getFile` call returns no media object, the inbound body builder falls back to `<media:document>`, and the agent receives no `MediaPaths`. The related PR #55970 is open and directly addresses the ambiguity by proposing an explicit unavailable reason, but that work has not landed on main.

## Best Possible Solution

Keep this issue open and use #55970 as the current implementation candidate, or replace it with a narrower Telegram fix that surfaces an agent-visible unavailable reason, documents the public Bot API 20 MB limit versus `mediaMaxMb`, and points larger-file users to the existing `apiRoot` plus `trustedLocalFileRoots` self-hosted Bot API path where appropriate.

## Risks / Open Questions

- Closing now would lose a valid current behavior gap: main still produces placeholder-only delivery in this path.
- A fix should not promise that public Bot API downloads can exceed Telegram's 20 MB `getFile` limit; larger-file support needs documented self-hosted Bot API configuration or a different transport.
- The related PR addresses the ambiguity but not necessarily the full expected behavior of obtaining a real attachment for oversized public Bot API files.
