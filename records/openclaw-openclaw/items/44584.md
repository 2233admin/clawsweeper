## Summary

Keep open. Current main still does not implement the combined fix described in this issue: Discord thread starter/history metadata is still present on follow-up prompt paths, and the `openclaw agent --channel discord --to ...` path still derives the session from `--to` before applying channel/account delivery context. The metadata half overlaps with open #41355, but the CLI delivery-context session lookup remains distinct.

## Best Possible Solution

Keep this issue open or split it deliberately. Fix the CLI resolver so `openclaw agent --channel discord --to <thread-id>` can find an existing session by normalized delivery context (`channel`, `to`, `accountId`, and Discord `channel:<id>` forms), add targeted regression coverage for that resolver path, and either resolve the remaining Discord thread metadata behavior here or explicitly track it under #41355.
