## Summary

Keep open. Current main satisfies the Slack channel-history and thread-history portions through the existing `read`/`readMessages` path with user-token-preferred read routing, but Slack message search is still not implemented or advertised on main. Open PR #65691 is the active implementation candidate for the remaining `search.messages` work, so this issue should wait on review of that PR rather than be closed or duplicated.

## What This Changes

The issue requests read-only Slack message retrieval actions for history, search, and thread reads that work with user-token read-only workflows without enabling Slack write capabilities.

## Best Possible Solution

Review and either land or close #65691. The desired end state is a Slack-owned generic `search` message action backed by Slack `search.messages`, advertised only for usable user-token read credentials, wired through the existing message action contract and CLI, documented on the public Slack channel page, and covered by focused discovery, dispatch, runtime, formatting, and least-privilege tests. The current `read` path can remain the solution for channel and thread history.
