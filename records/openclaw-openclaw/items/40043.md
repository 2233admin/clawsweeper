## Summary

Keep #40043 open. Current main still does not fully implement the PR: skill discovery remains immediate-child based with only a narrow `dir/skills/*` unwrap, and gateway startup logging still has no provider-auth diagnostic. The linked Anthropic-default symptom is partially reduced by current defaults, and #69663 is a focused open PR for recursive skill scanning, but this combined PR is neither fully implemented nor fully superseded. It also should not merge as-is because the submitted auth helper and recursive scanner have real correctness risks.

## Best Possible Solution

Keep #40043 open for maintainer rework or contributor split. The recursive skill discovery part should align with #69663 or a similar focused PR and include bounded traversal, containment and symlink-loop protection, watcher/docs alignment, candidate budgeting, and regression tests for managed/workspace/extra skill roots. The startup auth warning should be split into a separate best-effort diagnostic that uses the same runtime auth contract as `hasAvailableAuthForProvider` through a read-only/non-throwing path, or be explicitly dropped if maintainers decide the current Anthropic auth-signal defaulting fully resolves #39903.

## Risks / Open Questions

- Merging #40043 as written could put write-capable auth-store I/O on the gateway startup path and turn a diagnostic warning into a startup failure.
- Merging the auth warning as written could emit false warnings for valid auth sources such as custom provider keys, synthetic local auth, Bedrock/AWS auth, or broader runtime profile resolution.
- Merging the recursive scanner as written could traverse large non-skill trees without preserving the existing `maxCandidatesPerRoot` scan budget.
- Closing #40043 before #69663 or equivalent work lands would leave at least part of the nested skill-layout request unresolved unless maintainers explicitly decide the narrow `dir/skills/*` support is sufficient.
