## Summary

Keep open. Current main still has only partial Discord/gateway recovery: READY gating, transport-activity status, and generic health-monitor restarts are present, but the operator-visible post-READY Discord login log is still not emitted when startup begins in the pending state, and there is still no Discord/gateway-specific recovery or docs path for repeated `certificate has expired` TLS failures. The additional Linux report keeps this from being only a macOS CA-rotation cleanup item.

## Best Possible Solution

Keep this open as an active Discord gateway reliability and observability issue. The best path is to land a narrow READY observability fix that logs the bot identity when `runDiscordGatewayLifecycle()` actually reaches READY, then make a maintainer-reviewed decision for repeated `certificate has expired` TLS failures: channel-only restart, supervised gateway process restart, or documented/operator-exposed watchdog hook. Coordinate with related Discord reconnect work such as #71546 and the closed #38596/#50046 history, but do not close this issue until both the operator-visible READY signal and cert-error recovery policy are addressed or split into canonical follow-ups.
