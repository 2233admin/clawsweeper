## Summary

Keep this PR open. Current main still has the false-positive path the PR targets: splitMediaFromOutput only protects fenced code spans, then trimStart() lets 4-space or tab-indented MEDIA lines enter extraction. The PR is open, unmerged, and remains an implementation candidate rather than cleanup material; the closed linked issue does not make the unmerged PR obsolete.

## Best Possible Solution

Keep the PR open for maintainer review or a corrected merge. The best path is to land the indented-code protection in src/media/parse.ts, preserve code indentation during cleanedText normalization, add the media and fence regression tests, and explicitly align streaming-directive MEDIA case handling with the parser. Because current main does not implement the intended fix, closing this PR would drop useful active work.
