## Summary

Keep open. Current main still has the reported indented-code MEDIA extraction bug, and this PR is the active contributor fix path, but the latest patch still makes MEDIA directive case handling inconsistent with current block-reply and streaming behavior.

## What This Changes

The PR changes media and fence parsing/tests to skip MEDIA extraction from indented code blocks, preserve retained code indentation during cleanup, reject fenced-code closers with trailing text, and remove case-insensitive MEDIA matching.

## Best Possible Solution

Land a focused version that keeps the indented-code and indentation-preservation protections, then either preserves lowercase MEDIA support or updates splitMediaFromOutput, block-reply auto detection, streaming buffering, docs, and tests consistently in the same PR.

## Reproduction Assessment

Yes. Current main has a source-level reproduction: splitMediaFromOutput checks only fenced spans before trimStart().toUpperCase().startsWith("MEDIA:"), so an output containing a blank line followed by four spaces and MEDIA can be stripped as an attachment directive.

## Solution Assessment

No. The proposed fix addresses the central false-positive bug, but it is not the best mergeable solution until MEDIA directive case semantics are aligned across parser, block-reply, streaming, and tests.
