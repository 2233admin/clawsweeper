## Summary

Keep open. Current main still has the reported indented-code MEDIA false positive, and this PR is a useful active implementation, but the latest head still changes lowercase MEDIA compatibility inconsistently and has a code-indentation edge case after audio tag stripping.

## What This Changes

The PR updates media and Markdown fence parsing plus tests so indented-code MEDIA lines are preserved, retained code indentation is protected during cleanup, invalid fence closers with trailing text stay open, and MEDIA token matching becomes case-sensitive.

## Best Possible Solution

Land a focused version that keeps the indented-code protection and code-aware cleanup, preserves existing lowercase MEDIA behavior unless every directive surface is intentionally migrated together, reuses code-aware leading normalization after audio tag stripping, and adds the required Unreleased changelog entry.

## Reproduction Assessment

Yes. The original bug is reproducible from current main by passing splitMediaFromOutput text with a blank line followed by a 4-space or tab-indented MEDIA line; the PR regressions are source-backed by existing lowercase block-reply coverage and unchanged case-insensitive streaming buffering.

## Solution Assessment

No, not as submitted. The indented-code heuristic is the right boundary, but the branch mixes in an inconsistent case-sensitivity change and misses one code-aware cleanup path; the safer repair is to preserve compatibility and keep the cleanup logic shared.
