## Summary

Keep this PR open. Current main still has the MEDIA false-positive path this PR targets: `splitMediaFromOutput` only protects fenced code, trims leading indentation before directive detection, and has no indented-code regression coverage. The linked issue #69311 is closed, but this PR remains the active implementation candidate rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open for maintainer review or a corrected maintainer-apply. The best path is to add indented-code protection in `src/media/parse.ts`, preserve retained code indentation during media cleanup, add focused media and fence regression tests, and settle MEDIA case-sensitivity across `splitMediaFromOutput`, streaming directive buffering, and reply delivery tests before merging.

## Risks / Open Questions

- The PR changes shared `parseFenceSpans` behavior used by markdown code spans, auto-reply chunking, canvas rendering, and the embedded block chunker, so the fence trailing-text correction needs maintainer review across those consumers.
- MEDIA case semantics need explicit alignment before merge: current main still has lowercase/insensitive behavior in reply delivery and streaming buffering, while the PR removes the `i` flag from `MEDIA_TOKEN_RE`.
- The proposed indented-code heuristic is intentionally approximate and may need tests or documented tradeoffs for list and blockquote container contexts.
