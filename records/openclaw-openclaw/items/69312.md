## Summary

Keep this PR open. The linked parser bug is still present on current main, and the branch is a useful active implementation, but the latest head still changes MEDIA case handling without aligning block-reply and streaming directive surfaces, and it still has an audio-tag cleanup path that can strip tab indentation from retained code.

## What This Changes

The PR updates media and markdown fence parsing plus tests to skip MEDIA extraction from indented code blocks, preserve retained code indentation, reject fence closers with trailing text, and make MEDIA token matching case-sensitive.

## Best Possible Solution

Land a focused version that keeps the indented-code protection, but first aligns MEDIA case semantics across splitMediaFromOutput, block-reply normalization/tests, streaming buffering, and any user-facing guidance, then reuse the retained-code-line state when normalizing text after audio tags. If the fence-closing tightening remains, validate the shared fence-parser consumers along with the new parser tests.

  - body: Removing the `i` flag makes `splitMediaFromOutput` ignore lowercase `media:`, but block-reply parsing still enters directive parsing for `/media:/i`, existing coverage expects lowercase directives to produce `mediaUrls`, and the streaming accumulator at this head still buffers lowercase/mixed-case MEDIA prefixes. With `[[audio_as_voice]]` followed by a tab-indented code line, the tab can still be stripped, turning retained code such as `\tMEDIA:/tmp/code.png` into top-level-looking text.
