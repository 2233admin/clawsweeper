## Summary

Keep this PR open. Current main still has the false-positive path this PR targets: `splitMediaFromOutput` protects fenced code spans only, then trims leading whitespace before checking for `MEDIA:`, so 4-space or tab-indented code-block lines can still be extracted as media. The PR is open, active, and unmerged, so it remains an implementation candidate rather than cleanup material.

## Best Possible Solution

Keep the PR open for maintainer review or a corrected maintainer-apply. The likely product path is to add indented-code protection in `src/media/parse.ts`, preserve code indentation during media cleanup, add focused media and fence regression tests, and decide the MEDIA case-sensitivity contract consistently across `splitMediaFromOutput`, streaming directive buffering, and existing reply-delivery tests.
