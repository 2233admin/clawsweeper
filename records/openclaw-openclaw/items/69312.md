## Summary

Keep open. The underlying indented-code MEDIA extraction bug is real on current main, and the main fix direction is useful, but the current PR is not merge-ready because it changes lowercase MEDIA directive semantics while block-reply parsing and streaming buffering still treat lowercase media directives as supported.

## What This Changes

The PR changes media and fence parsing to skip MEDIA extraction from indented code blocks, preserve retained code indentation during cleanup, validate fenced-code closers with trailing text, remove case-insensitive MEDIA token matching, and add media/fence regression tests.

## Best Possible Solution

Land a focused version that keeps the indented-code and cleanup protections, then makes MEDIA directive case handling consistent across splitMediaFromOutput, block-reply auto-detection, streaming directive buffering, docs, and tests. If lowercase support is intentionally removed, update every caller and regression test in the same PR.

## Reproduction Assessment

Yes. Current main has a direct source-level reproduction for the reported bug via splitMediaFromOutput on an indented `MEDIA:` line, and the PR regression is covered by the existing lowercase block-reply test plus current streaming-buffer code.

## Solution Assessment

No. The patch is close, but it is not the best mergeable solution until the case-sensitivity change is either reverted or propagated consistently through the reply and streaming directive surfaces.
