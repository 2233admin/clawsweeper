## Summary

Keep this PR open. Current main still has the reported indented-code false-positive path, so the contribution is relevant, but the latest PR head is not ready to merge because it changes lowercase MEDIA semantics without updating block-reply/streaming behavior and still has an indentation preservation gap after audio-tag cleanup.

## What This Changes

The PR changes media and fence parsing to skip `MEDIA:` lines in indented code blocks, preserve retained code indentation, tighten fence closing validation, make MEDIA token matching case-sensitive, and add media/fence regression tests.

## Best Possible Solution

Land a focused version that keeps the indented-code protection and code-indentation preservation, aligns MEDIA case semantics across `splitMediaFromOutput`, block-reply parsing, streaming buffering, docs, and tests, and validates all shared fence-parser consumers if the fence change remains in scope.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Keep MEDIA case semantics aligned:** `src/media/parse.ts:17`
  - body: Removing the case-insensitive flag makes `splitMediaFromOutput` ignore lowercase `media:`, but block-reply parsing still enters directive parsing for `/media:/i` and current coverage expects lowercase directives to become media. The PR also leaves streaming buffering case-insensitive, so `media: ./report.pdf` will remain visible text or be buffered inconsistently instead of producing `mediaUrls`.
  - confidence: 0.9
- **[P2] Preserve tab code indentation after audio tags:** `src/media/parse.ts:738`
  - body: The new audio-tag cleanup path does not reuse `firstLineIsCode` or `keptLineIsCode`, so `[[audio_as_voice]]` followed by a tab-indented code line can lose its leading tab during post-processing. That turns retained code like `\tMEDIA:/tmp/code.png` into a top-level-looking `MEDIA:` line, breaking the indentation preservation this patch is adding.
  - confidence: 0.84
