## Summary

Keep this PR open. Current main still does not implement a provider-usage TTL cache or the proposed `usage.cacheTtlMs` config, and the active status/usage paths still call `loadProviderUsageSummary` directly. The PR remains a plausible implementation candidate for the problem described in #45332, but it is stale against the current plugin-owned provider-usage loader shape and needs rebase plus cache-boundary tests before merge.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but require a rebase or replacement patch against current main. The best fix should add the TTL cache in or near `src/infra/provider-usage.load.ts`, preserve injected config/env/workspace/auth contracts, wire any accepted `usage.cacheTtlMs` config through types/schema/help/labels/generated config metadata, and add regression tests for cache hit, TTL expiry, disabled cache, provider-list boundaries, agent/workspace/config boundaries, explicit-auth bypass, and non-caching of transient failures.

## Risks / Open Questions

- Merging the PR as-is would likely conflict with or regress the current plugin-owned provider-usage loader shape because current main passes injected `config`, `env`, `workspaceDir`, and credential-source controls through the usage path.
- A provider-usage cache can leak or return incorrect quota data unless its key includes the right agent, workspace, provider-list, config/env, and explicit-auth boundaries, and unless transient failures such as 429/Timeout are not cached.
