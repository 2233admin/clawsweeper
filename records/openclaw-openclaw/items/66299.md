## Summary

Keep this open. Current main still returns the explicit-token failure before attempting the ACP bound-thread resolver, while the issue discussion and docs agree the thread binding should be tried before surfacing the bad-token diagnostic.

## What This Changes

Make ACP session-target resolution fall through from a failed explicit token lookup to the thread-bound session lookup before returning the bad-token error.

## Best Possible Solution

Land a focused resolver and regression-test change: successful explicit token resolution stays first, unresolved tokens try resolveBoundAcpThreadSessionKey next, the existing bad-token diagnostic is preserved when no bound thread resolves, and the user-facing fix is recorded under Unreleased fixes.
