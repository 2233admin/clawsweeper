## Summary

Keep open. Current main still violates the documented plugin SDK contract: `api.resolvePath` is wired directly to `resolveUserPath(input)` instead of being rooted at the plugin record's `rootDir`, and registered service start handlers can still use that captured API after registration. The report is current, narrow, and suitable for a focused fix PR.

## What This Changes

The issue requests that `api.resolvePath` for externally installed extension plugins return a plugin-root-relative path during service startup instead of `undefined` or a process-CWD path.

## Best Possible Solution

Bind native plugin APIs to a root-aware resolver at the registry/loader boundary: `.` and other relative paths should resolve under `record.rootDir` when available, while absolute paths and home-relative paths should keep existing user-path behavior. Add focused regression coverage for service-start use of a captured API and keep the public SDK docs aligned.
