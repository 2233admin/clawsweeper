## Summary

Keep this issue open. Current main still lacks first-class provider Batch API routing for delayed/background LLM agent turns; existing batch code is limited to memory embedding indexing and media-style provider jobs, while cron, sessions_send, sessions_spawn, heartbeat, and provider plugin contracts still route through immediate stream-style turns. Related #49326, #56126, and #70606 are closed context, and I found no open PR tied to this issue.

## Best Possible Solution

Keep this as the canonical product/API tracker. The best path is an additive provider/runtime batch-turn contract, followed by explicit opt-in fields for cron agentTurn jobs, heartbeat/background runs, sessions_send, and sessions_spawn/subagents, plus provider plugin implementations, docs, config/schema updates, and focused regression coverage.

## Risks / Open Questions

- Batch-related names in current code can mislead triage because implemented batch support is for memory embeddings/media-style jobs, not delayed LLM agent turns.
- A cron-only batch flag would bypass the provider/plugin boundary; the missing design is a generic lifecycle covering submit, poll, cancel, timeout/failure handling, usage accounting, and transcript/result reconciliation.
