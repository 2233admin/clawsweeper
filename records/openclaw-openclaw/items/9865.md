## Summary

Keep this issue open. Current main still lacks first-class Batch API routing for delayed/background LLM agent turns: provider runtime hooks are StreamFn-oriented, cron/session/subagent/heartbeat surfaces have no batch opt-in, and existing batch support is scoped to memory embeddings or media-style work rather than agent turns. The related closed reports #49326, #56126, and #70606 are narrower context for this canonical tracker, not shipped fixes.

## What This Changes

The issue requests Batch API support for non-real-time background LLM work such as cron agent turns, spawned background agents, and heartbeat or maintenance tasks.

## Best Possible Solution

Keep this as the canonical product/API tracker. The best path is an additive provider/runtime batch-turn contract first, then opt-in protocol/config fields for cron agentTurn jobs, sessions_send, sessions_spawn/subagents, and heartbeat/background runs, followed by OpenAI/Anthropic provider plugin implementations, usage/accounting/fallback semantics, docs, and focused protocol/provider/runtime tests.
