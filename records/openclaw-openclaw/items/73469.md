## Summary

Keep open. Current main still sends Feishu streaming-card text after the initial interactive card reference message, so agent-authored mention tags can render in the card without going through a notification-bearing message send. The adjacent open syntax-normalization issue #60103 does not cover this Card Kit notification timing problem.

## What This Changes

Issue requests Feishu streaming-card replies to trigger push notifications when an agent-authored `<at user_id="...">...</at>` mention appears in final streamed card text.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy for notification semantics before coding. The end state should handle agent-authored raw `<at user_id>` tags, existing `mentionTargets`, text/card syntax conversion, topic replies, duplicate-final suppression, and a documented fallback when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: with Feishu streaming enabled and card rendering selected by `renderMode: "card"` or markdown-triggered `auto`, the initial send is only a card reference and the mention text lands later through Card Kit update calls.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` is unlikely to be sufficient because `close()` updates Card Kit content/settings rather than sending a new notification-bearing Feishu message; maintainers should choose between fallback, companion notification, or another Feishu-supported pattern.
