## Summary

Current main still matches the reported Feishu streaming-card notification gap. Streaming card replies are created first as a placeholder Card Kit card reference, while the final mention-bearing text is applied later through Card Kit element updates and close. The dispatcher does not parse agent-authored `<at user_id="...">...</at>` tags into a notification-capable send path, and the existing related issue #60103 only covers mention syntax normalization, not the first-notification timing problem.

## Best Possible Solution

Keep this open for a Feishu plugin-owned fix. First confirm whether Feishu can notify on Card Kit updates; if it cannot, detect mention-bearing outbound replies before starting a streaming card and route them through a notification-capable path, such as an initial mention-carrying send or a controlled non-streaming final fallback. Add focused tests for `renderMode: "card"`, auto-mode card replies, agent-authored `<at user_id="...">...</at>` tags, existing inbound `mentionTargets`, thread replies, and duplicate-final suppression.
