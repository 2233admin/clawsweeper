## Summary

Keep open. Current main still sends Feishu streaming cards as an initial Card Kit card reference and applies final text, including any mention-looking content, later through Card Kit updates, so the code path does not solve the reported push-notification boundary. The related open #60103 issue is adjacent syntax-normalization work, but it does not cover the Card Kit notification timing problem.

## What This Changes

Feishu streaming-card replies should send user push notifications when agent-authored response text contains Feishu `<at user_id="...">...</at>` mentions.

## Best Possible Solution

Keep this as a Feishu plugin-owned bug and choose a tested outbound strategy that preserves notification semantics for agent-authored mentions. The end state should cover raw `<at user_id>` parsing, existing `mentionTargets`, card/text syntax conversion, topic replies, duplicate-final suppression, and a documented fallback when Feishu cannot notify from Card Kit updates.

## Reproduction Assessment

Yes. Configure Feishu with streaming enabled and `renderMode: "card"` or markdown-triggered `auto`, have the agent return a `<at user_id="...">...</at>` tag, and compare the current code path: the first Feishu message is only a card reference while the mention text is applied later by Card Kit updates.

## Solution Assessment

Unclear. Adding a mentions parameter to `close()` does not look sufficient because `close()` only updates Card Kit content/settings; the maintainable fix likely needs a Feishu-owned decision about when to stream, fall back, or send an additional notification-bearing message.
