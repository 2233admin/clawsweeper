## Summary

Keep open. Current main still has the reported gap: the gateway restart tool accepts a continuation, but the already-in-flight coalescing branch returns successful `coalesced: true` before the continuation-writing hook can run. The linked open PR should be reviewed as the likely fix path, but this is not implemented on main yet.

## What This Changes

The issue asks OpenClaw to preserve or explicitly report failed delivery of a gateway restart `continuationMessage` when the restart request coalesces with an already pending or in-flight restart.

## Best Possible Solution

Review and land PR #73812 if it covers this exact coalesced in-flight continuation loss; otherwise make a narrow restart scheduler/tool fix so coalesced success either durably queues the accepted continuation exactly once or returns an explicit not-queued/error result.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: get the scheduler into an unconsumed emitted restart state, then call the gateway restart tool with `continuationMessage`; current main returns coalesced success before the sentinel write hook can persist the continuation.

## Solution Assessment

Yes. The narrow maintainable solution is to preserve restart coalescing while making the accepted continuation contract truthful; a broad checkpoint or full task-resume redesign is not required for this specific accepted-parameter loss.
