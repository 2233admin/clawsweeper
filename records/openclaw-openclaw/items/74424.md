## Summary

Current main still contains the reported failure path, and the related open implementation candidate has not landed. The gateway tool accepts a continuation, but the in-flight restart coalescing branch can return success before the hook that writes the restart sentinel runs.

## What This Changes

The issue requests that `gateway` restart calls with `continuationMessage` either durably queue the continuation across coalesced restarts or explicitly report that no continuation was queued.

## Best Possible Solution

Keep this issue open until PR #73812 or an equivalent narrow fix lands. The fixed behavior should either persist or merge the accepted continuation for already-in-flight coalesced restarts, or return an explicit not-queued result, with regression coverage for the unconsumed SIGUSR1 branch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a SIGUSR1 restart signal unconsumed, then invoke the gateway restart tool with `continuationMessage`; current main returns from the in-flight coalescing branch before the gateway tool's sentinel-writing hook can run.

## Solution Assessment

No. Current main is not the best solution because it can report a successful coalesced restart while losing an accepted continuation; the narrow maintainable fix is to make continuation queuing explicit and tested in that coalesced path.
