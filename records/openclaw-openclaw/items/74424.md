## Summary

Keep open: current main still has the reported source-level failure path, and the linked implementation candidate has not landed. The gateway restart tool queues the continuation only from a restart emit hook, but the already-emitted/unconsumed coalescing branch returns `ok` before registering or running that hook.

## What This Changes

The issue requests reliable delivery or explicit non-queue reporting for `gateway` restart `continuationMessage` values when the restart request coalesces with an already in-flight restart.

## Best Possible Solution

Keep this issue open until PR #73812 or an equivalent narrow fix lands. The fix should either durably merge/persist the continuation for already-in-flight coalesced restarts or report that no continuation was queued, with regression coverage for the unconsumed SIGUSR1 branch.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to leave a SIGUSR1 restart emitted but not marked handled, then call the gateway restart tool with `continuationMessage`; current main returns from the coalesced branch before the tool's sentinel-writing hook can run.

## Solution Assessment

No. Current main is not the best solution because it can acknowledge a coalesced restart while dropping the accepted continuation; the narrow maintainable path is to make continuation queuing explicit in the scheduler/tool boundary and test that branch.
