## Summary

Keep open. Current main still has the reported gap: the agent-facing gateway restart tool only persists its continuation through restart emit hooks, while the already-in-flight coalescing path returns a successful coalesced response before those hooks are registered or run. The broader continuation support from #50137 is present, but it does not cover this narrower in-flight coalescing contract.

## What This Changes

Fix the gateway restart tool so a restart request with `continuationMessage` does not silently lose that continuation when it coalesces with an already in-flight gateway restart.

## Best Possible Solution

Preserve restart coalescing, but make the accepted continuation contract truthful: either durably attach/merge the explicit `continuationMessage` into the already-emitted restart handoff exactly once, or return a clear non-queued/error shape so the agent can choose a recovery path.

## Repair Work Prompt

Fix openclaw/openclaw#74424. Current main accepts `gateway` tool `action: "restart"` with `continuationMessage`, but if `scheduleGatewaySigusr1Restart()` coalesces with an already in-flight restart (`hasUnconsumedRestartSignal()`), the function returns success before the tool's `emitHooks.beforeEmit` writes the restart sentinel, so the continuation is silently lost. Keep restart coalescing semantics, but make this contract truthful: preferably durably queue/merge the explicit continuation exactly once for the already-emitted restart, or otherwise return a clear `continuationQueued: false`/error shape so the agent can recover. Do not implement the broader checkpoint/auto-resume RFC, do not change plugin/channel ownership boundaries, and do not write restart sentinels from agents directly. Add focused regression coverage for the in-flight coalesced path and preserve existing scheduled-coalescing hook behavior. Likely changelog: user-facing bug fix under the active release Fixes section with `Thanks @songshikang0111` if the repo's changelog policy requires it.
