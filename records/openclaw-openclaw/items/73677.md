## Summary

Keep this issue open. Current main has Feishu inbound sticker handling only as generic media placeholders/downloads, and rich-text `emotion` rendering only preserves fields already present in post payloads. The plain text message path returns only `parsed.text`, so there is no current implementation that preserves or maps Feishu bracket-style custom stickers such as `[OK]`, `[完成]`, or `[调皮]` when they are not already present in the text field. The provided related items mostly cover Feishu reaction API emoji normalization or Telegram custom emoji entities, not this inbound Feishu custom sticker context gap.

## Best Possible Solution

Keep this open and fix it in the bundled Feishu plugin. Capture the actual Feishu event/API payload for mixed text plus custom stickers, then extend the Feishu inbound parser to preserve a stable text representation or map known sticker labels to Unicode where safe. Add focused tests for plain text with sticker metadata, standalone sticker placeholders, rich-text `emotion` elements, and mixed Chinese bracket labels so the agent prompt retains the user's intent.
