## Summary

Keep this open. Current main still lacks a Feishu/Lark inbound path that preserves custom sticker labels: text parsing keeps only `text`, standalone stickers become a generic media placeholder, and the existing emotion renderer is limited to rich-text posts.

## What This Changes

The issue asks for Feishu/Lark inbound custom stickers such as [OK], [完成], and [调皮] to be preserved for agents as Unicode emoji or stable text labels.

## Best Possible Solution

Fix this in `extensions/feishu` by preserving sticker labels or stable text fallbacks from inbound payloads, using Unicode mappings only for verified sticker names. Add focused parser and inbound tests for mixed text plus sticker metadata, standalone `sticker` messages, rich-text `emotion` elements, Chinese bracket labels, and unknown-label fallback behavior.

## Reproduction Assessment

Unclear for the exact live mixed text/sticker event because the issue does not include raw event JSON. Source-level reproduction is clear: current `main` returns only `parsed.text` for text messages and `<media:sticker>` for standalone sticker messages.

## Solution Assessment

Yes, a Feishu plugin parser fix is the right maintainable direction and should not require a core channel API change. The safest path is label/text preservation first, then verified Unicode mapping only where the Feishu payload fields make that reliable.
