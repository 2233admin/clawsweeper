## Summary

Current main still has the reported regression: TUI /new creates a unique tui-<uuid> session via local setSession(...) only, while the hook-visible command:new path is still sessions.reset. An open implementation PR (#49945) explicitly tracks this issue and is unmerged, so this should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Review and land PR #49945, or apply the equivalent on main: keep the unique tui-<uuid> session key for TUI /new, but route that key through client.resetSession(uniqueKey, "new") or the equivalent sessions.reset path so command:new hooks fire. Add or update the TUI command-handler regression test to assert /new uses the reset path with reason new while /reset still resets the current session with reason reset.
