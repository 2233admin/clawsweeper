## Summary

Keep open. Current main has the merged observability work, but the reported behavior is still possible on the configured/default TTS paths: automatic replies and `/tts audio` still use the resolved provider as the first provider in a fallback chain, so an ElevenLabs primary failure can still end with another configured provider such as OpenAI. The issue discussion also already records maintainer context that #57953 and #57954 were partial diagnostics fixes and intentionally left provider/playback stickiness open.

## What This Changes

The issue requests that a configured ElevenLabs TTS provider produce and play ElevenLabs audio instead of silently falling back to an OpenAI voice.

## Best Possible Solution

Choose and implement an explicit TTS provider fallback policy: either make configured or locally selected providers fail closed by default, or add a documented config/preference switch for strict provider pinning. The selected behavior should cover automatic replies, `/tts audio`, status diagnostics, and regression tests for an ElevenLabs primary with OpenAI configured as a fallback.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction exists by configuring a primary TTS provider and another configured fallback, then making the primary fail; current tests and source show `synthesizeSpeech` can return the fallback provider with `fallbackFrom` set. The exact reporter environment was not live-reproduced.

## Solution Assessment

No. The current merged diagnostics work is useful but not the complete fix for the reported behavior; the safest next step is a maintainer-approved fallback policy change rather than closing this as implemented.
