## Summary

Keep open. Current main has the merged TTS fallback observability work from #57953 and #57954, plus newer explicit-selection paths that can disable fallback for gateway/capability requests, but the reported configured/default TTS paths still allow ElevenLabs to fall through to another configured provider such as OpenAI after a primary-provider failure.

## Best Possible Solution

Keep this issue open for a focused TTS fallback policy fix. The likely path is to decide whether an explicitly configured `messages.tts.provider` or `/tts provider <id>` should fail closed by default, or add a documented config/preference switch to disable fallback for pinned providers; then thread that through automatic TTS and `/tts audio`, update `/tts status`/docs, and add regression tests covering ElevenLabs primary with OpenAI configured as a fallback.
