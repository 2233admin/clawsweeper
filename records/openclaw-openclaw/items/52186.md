## Summary

Keep open. Current main has the diagnostic/fallback-observability work from #57953 and #57954, but configured/default TTS still treats the selected provider as the primary in an automatic fallback chain. The remaining provider-stickiness/fail-closed behavior called out in the issue discussion is not implemented for automatic TTS or `/tts audio`.

## Best Possible Solution

Keep this issue open for maintainer policy judgment on TTS fallback semantics. The likely implementation is either fail-closed behavior for explicitly configured `/tts provider` or `messages.tts.provider`, or a documented config/preference switch that disables fallback for pinned providers, threaded through automatic TTS, `/tts audio`, agent TTS tools, status output, docs, and regression tests.
