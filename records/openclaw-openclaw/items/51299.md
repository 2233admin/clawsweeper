## Summary

Keep this issue open. Current main still documents legacy iMessage/imsg as needing Full Disk Access and Automation in the same process context, but it does not provide a proven Sequoia/Tahoe-safe approval path for a bare LaunchAgent. The iMessage runtime still spawns `imsg rpc` directly and the probe still reports generic failures for the reported `permissionDenied` and AppleEvents `-1743` class of errors. BlueBubbles remains the recommended iMessage path for new setups, but it does not resolve the legacy LaunchAgent/imsg TCC question.

## Best Possible Solution

Keep this issue open and resolve it with explicit maintainer guidance in the legacy iMessage docs and/or channel diagnostics. A good fix would document a tested Sequoia/Tahoe-safe path for imsg under launchd, including whether `gateway install --wrapper` with an app/run-as helper is supported, or clearly state that bare LaunchAgent imsg is not reliable under modern macOS TCC and point users to BlueBubbles or a GUI-user remote Mac setup. Adding probe/status text for `permissionDenied`, AppleEvents `-1743`, Full Disk Access, Automation, and LaunchAgent context would make the failure actionable.
