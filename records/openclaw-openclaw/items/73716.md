## Summary

Keep open. Current main still rejects `chat` events before checking the active `runId`, while the gateway canonicalizes raw `chat.send` session keys such as `main`; the provided context also shows open PR #73721 uses closing syntax for the same bug, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Fix Control UI live chat so a send made with a raw session key still displays gateway chat events emitted under the canonical session key for the same in-flight run.

## Best Possible Solution

Review and land PR #73721, or an equivalent small Control UI change, so same-`runId` live `chat` events are accepted across raw/canonical session-key aliases while unrelated session and run events remain filtered.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `handleChatEvent` with `state.sessionKey = "main"`, `state.chatRunId = "run-1"`, and a payload using `sessionKey = "agent:main:main"` plus `runId = "run-1"`; current main returns `null` at the strict session-key check.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in Control UI event acceptance or session-key normalization, not provider/model handling; the linked active-run fallback direction is plausible if it continues to reject events where neither session key nor active run id matches.
