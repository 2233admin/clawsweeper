## Summary

Keep this issue open. Current main still has the reported raw-versus-canonical session-key mismatch: the Control UI sends `chat.send` with its active `state.sessionKey`, `handleChatEvent` rejects events whose `payload.sessionKey` differs before checking the matching active `runId`, and the gateway canonicalizes raw `main` to the main session key before broadcasting chat finals. Open PR #73721 directly targets this issue but is still open and unmerged.

## What This Changes

Fix the Control UI so live chat responses are displayed when a send starts from raw alias `main` but gateway chat events arrive under canonical session key `agent:main:main`.

## Best Possible Solution

Land PR #73721 or an equivalent narrow Control UI fix that either normalizes the active session key to the gateway canonical key or accepts live `chat` events for the active client run when `runId` matches, while preserving the guard against unrelated sub-agent or fleet events and adding regression coverage for raw `main` versus canonical `agent:main:main`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `handleChatEvent` with state `{ sessionKey: "main", chatRunId: "run-1" }` and payload `{ sessionKey: "agent:main:main", runId: "run-1", state: "delta" }`; current main returns `null` at the session-key check before considering the matching run id.

## Solution Assessment

Yes. The narrowest maintainable solution is in the Control UI event handler or session-key normalization layer; changing provider/model behavior or gateway persistence would not address the UI-side live event rejection as directly.
