## Summary

Keep open. Current main still sends the Control UI's active raw session key, canonicalizes it in the gateway, and then drops returning live `chat` events before the active `runId` can be considered. The linked implementation PR #73721 remains separate from `main` and unmerged, so this issue should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue requests that Control UI show live chat responses when `chat.send` uses a raw session alias such as `main` but gateway `chat` events for the same run use a canonical key such as `agent:main:main`.

## Best Possible Solution

Review and land PR #73721, or an equivalent small Control UI fix, so same-`runId` live `chat` events are accepted across raw/canonical session-key aliases while unrelated session and run events remain filtered.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `handleChatEvent` with `state.sessionKey = "main"`, `state.chatRunId = "run-1"`, and a `chat` payload using `sessionKey = "agent:main:main"` plus `runId = "run-1"`; current main returns `null` at the session-key check.

## Solution Assessment

Yes. The narrowest maintainable solution belongs in Control UI event acceptance or session-key normalization, not provider/model handling; the active-run fallback direction is plausible if it continues to reject events where neither the session key nor the active run id matches.
