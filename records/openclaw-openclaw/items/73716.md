## Summary

Keep this issue open. Current main still sends `chat.send` with the UI's active raw session key, canonicalizes that key in the gateway, and rejects the returning live `chat` event in `handleChatEvent` before matching the active `runId`; open PR #73721 is the implementation candidate and is not merged yet.

## What This Changes

Fix Control UI live chat handling so responses for an in-flight run are not dropped when the UI used a raw session key like `main` and the gateway emits the canonical key like `agent:main:main`.

## Best Possible Solution

The desired end state is a small Control UI fix that either normalizes the active session to the gateway canonical key or accepts live `chat` events for the active client run when `runId` matches, while preserving the unrelated-run/sub-agent guard and adding regression coverage for raw `main` versus canonical `agent:main:main`.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `handleChatEvent` with state `{ sessionKey: "main", chatRunId: "run-1" }` and a payload `{ sessionKey: "agent:main:main", runId: "run-1", state: "delta" }`; current main returns `null` at the session-key check before considering the matching run id.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the web Control UI event handler or session-key normalization layer; PR #73721's active-run fallback is a plausible implementation if it keeps dropping events where neither the session key nor active run id matches.
