## Summary

Keep this issue open. Current main still has the raw/canonical session-key mismatch in the Control UI live chat path, and open PR #73721 is the unmerged implementation candidate rather than a reason to close the issue early.

## What This Changes

The issue requests that Control UI live chat responses display without refresh when `chat.send` is sent with a raw key such as `main` but gateway chat events arrive with the canonical key such as `agent:main:main`.

## Best Possible Solution

The desired end state is a small Control UI fix that either normalizes the active session to the gateway canonical key or accepts live `chat` events for the active client run when `runId` matches, with regression coverage for raw `main` versus canonical `agent:main:main` and preserved rejection of unrelated sessions/runs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `handleChatEvent` with state `{ sessionKey: "main", chatRunId: "run-1" }` and payload `{ sessionKey: "agent:main:main", runId: "run-1", state: "delta" }`; current main returns `null` at the session-key check before considering the matching run id.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Control UI event handler or session-key normalization layer; PR #73721's active-run fallback is a plausible implementation path if it keeps dropping events where neither the session key nor active run id matches.
