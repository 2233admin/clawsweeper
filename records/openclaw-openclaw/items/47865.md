## Summary

Keep open. Current main still has the reported split: `delivery.mode="none"` records `not-requested` for cron delivery bookkeeping, but main-session `systemEvent` jobs are still enqueued into the target session and wake heartbeat with `target: "last"`, so an agent response can still go through the normal user-visible heartbeat path. The exec notification path is separate and still enqueues exec completion/failure system events by default.

## Best Possible Solution

Keep this issue open. Maintainers should define the policy for `sessionTarget: "main" + payload.kind: "systemEvent" + delivery.mode: "none"`: either make `delivery.mode="none"` thread an internal-only/no-user-delivery flag through cron enqueueing and heartbeat wake/run, or update validation/docs/UI to state that main-session system events can still produce user-visible heartbeat replies. Review #58893 as the existing draft implementation candidate, and coordinate with #73900 for the inherited `heartbeat.to` routing/privacy case. Add regression coverage for explicit main cron `delivery.mode="none"`, inherited heartbeat destination clearing or preservation, and exec `notifyOnExit`/SIGTERM notification behavior.
