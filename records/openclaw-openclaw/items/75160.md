## Summary

Keep open. The production fallback already exists on current main, but this PR is a narrow test-only regression guard that is not yet present on main and appears appropriate for normal review rather than cleanup closure.

## What This Changes

The PR updates `src/gateway/tools-invoke-http.test.ts` to add mocked memory tools, track tool-factory contexts, and assert `/tools/invoke` falls back to plugin-enabled resolution for `memory_search` while ordinary known core tools stay plugin-disabled.

## Best Possible Solution

Keep the runtime fallback as implemented on main and merge this focused regression coverage after normal CI/review. Broader real-plugin integration coverage can be a separate follow-up if maintainers want end-to-end memory slot proof.

## Reproduction Assessment

Yes. The PR creates a deterministic unit reproduction by making `memory_search` a known core id that is absent when plugin tools are disabled, then verifying `/tools/invoke` retries with plugin tools enabled and succeeds.

## Solution Assessment

Yes. Because production behavior is already present, a small targeted regression test is the narrowest maintainable change; an integration test would add confidence but is not required to prove this control-flow branch.
