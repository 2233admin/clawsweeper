## Summary

Keep open. Current main still has the unsafe OpenAI-completions stream path, and this PR targets the right parser layer, but the submitted patch drops the nested Mistral thinking-array shape and is missing the required changelog entry before it is merge-ready.

## What This Changes

The PR adds a structured `delta.content` normalizer for OpenAI-compatible completions streams, reuses it for reasoning-field fallback payloads, and adds unit tests around string, object, and typed-block array content.

## Best Possible Solution

Land one parser-side OpenAI-completions fix that normalizes structured `delta.content` arrays, preserves provider block order, flattens nested Mistral thinking sub-block arrays, falls through when content arrays contain no supported blocks so same-chunk reasoning/tool calls still process, and includes focused regression coverage plus the required changelog credit.

## Reproduction Assessment

Yes. The linked issue gives concrete Mistral-with-thinking steps, and current main has a focused mocked reproduction path by feeding `processOpenAICompletionsStream` a `delta.content` typed-block array and observing the direct append path can stringify it into `[object Object]`.

## Solution Assessment

Mostly yes, but not as submitted. The OpenAI-compatible stream parser is the narrow maintainable layer, but the patch should flatten nested thinking arrays and add the release-note entry before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle nested thinking blocks before dropping them:** `src/agents/openai-transport-stream.ts:1602-1603`
  - body: The normalizer only reads `thinking` when it is a string. Mistral reasoning streams can carry `type: "thinking"` blocks whose `thinking` value is an array of `{ type: "text", text }` parts, so this drops real reasoning content instead of routing it to a thinking delta. Flatten those nested text parts before returning the thinking part.
  - confidence: 0.84
