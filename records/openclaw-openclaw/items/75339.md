## Summary

This PR should close as superseded: the remaining user-visible Mistral typed-block `delta.content` bug is already tracked by the broader open parser PR #68418, while the related `reasoning_content` array path is separately tracked by #67203. Current `main` still has the bug, and this patch is useful context, but its submitted implementation misses the nested thinking-array shape documented by Mistral and lacks the required changelog entry.

## What This Changes

The PR adds a structured `delta.content` normalizer to the OpenAI completions stream, reuses it for reasoning-field fallback normalization, and adds unit coverage for string, object, array, and unrecognized content shapes.

## Best Possible Solution

Use #68418, or a consolidated replacement built from it, as the canonical OpenAI-completions parser fix: normalize structured `delta.content` arrays including nested Mistral thinking blocks, preserve provider block order, keep unsupported content from swallowing same-chunk reasoning/tool calls, and include focused tests plus the required changelog credit. Keep #67203 or equivalent coverage for the separate `reasoning_content` array path if it is not folded into the same patch.

## Reproduction Assessment

Yes. The linked report gives concrete Mistral-with-thinking steps, current `main` still passes `choice.delta.content` directly into a string append path, and the Mistral docs show the nested typed-block shape that reproduces the corruption in a mocked stream chunk.

## Solution Assessment

No. The parser layer is the right place, but this specific patch is not the best merge target because it drops nested thinking arrays, misses the changelog requirement, and overlaps with broader open parser work that already covers the central `delta.content` failure mode.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle nested thinking arrays before dropping them:** `src/agents/openai-transport-stream.ts:1603-1605`
  - body: The normalizer only reads `thinking` when it is a string. Mistral's documented native-reasoning output uses `type: "thinking"` blocks whose `thinking` value can be an array of `{type:"text", text}` parts, so this patch would silently lose the reasoning content for that real stream shape. Flatten those nested text parts before returning the thinking delta.
  - confidence: 0.86
- **[P2] Add the required changelog entry:** `src/agents/openai-transport-stream.ts:1457`
  - body: This is a user-facing contributor fix, but the branch only changes the parser and tests. OpenClaw policy requires an Unreleased `### Fixes` changelog bullet with contributor credit before merge.
  - confidence: 0.95
