## Summary

Keep open. Current main does not implement plugin trust tiers, and the PR still targets a real core/docs/CLI surface, but the proposed patch has a correctness issue in security-facing metadata: omitted-format plugins would be recorded as `openclaw` while the new trust resolver labels them `content`. The PR also needs maintainer product/security review before making this taxonomy public.

## What This Changes

The PR adds plugin trust-tier types and resolution, stores the tier on plugin records, prints it in plugin CLI output, and adds trust-tier documentation and tests.

## Best Possible Solution

Land a rebased version only if maintainers want this public taxonomy, with trust tiers computed from the same normalized effective plugin format used in PluginRecord, docs that separate current capabilities from reserved future behavior, and tests covering native, bundle, missing-format fallback, loader record paths, and CLI list/info output.

## Reproduction Assessment

Yes for the blocking PR bug. A manifest without an explicit format flows into createPluginRecord, which normalizes the record format to openclaw, while the PR's raw resolveTrustTier call would classify the same record as content.

## Solution Assessment

No, the proposed patch is not the best way to solve the issue as written. The safer narrow fix is to resolve trust from the effective normalized format at both loader call sites and align the undefined-format docs/tests with that contract before landing security-facing metadata.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Resolve trust tiers from the normalized plugin format:** `src/plugins/loader.ts:849-853`
