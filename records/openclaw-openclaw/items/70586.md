## Summary

Keep this PR open for revision. Current main still misses app-activity tracking for accepted Slack message-event branches that return before the message handler, but the submitted placement is too broad: normal handled messages already track in `createSlackMessageHandler`, and the stale-socket rationale is obsolete because health uses transport activity, not `lastEventAt`.

## Best Possible Solution

Request a narrower revision: thread `trackEvent` into `registerSlackMessageEvents`, but invoke it only for accepted message-event branches that return before `handleSlackMessage`, such as authorized subtype system events and DM/MPIM `app_mention` skips. Keep `createSlackMessageHandler` as the single tracking point for normal handled messages/app mentions, and add tests proving handled events track once, accepted early returns track once, and mismatched, unauthorized, self-generated, duplicate, or invalid events do not track. Reframe the change as app-activity timestamp completeness, not stale socket liveness.
