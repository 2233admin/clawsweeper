## Summary

Keep this PR open for revision. The underlying Slack message-event activity gap still exists for accepted early-return paths, but the submitted placement fires before existing message-handler acceptance filters, causing duplicate status updates for normal handled messages and app mentions.

## What This Changes

The PR threads `trackEvent` into `registerSlackMessageEvents`, invokes it from Slack `message` and `app_mention` handlers after the mismatch guard, and adds Slack message-event tracking tests.

## Best Possible Solution

Revise the PR so `trackEvent` is passed into `registerSlackMessageEvents` but invoked only for accepted branches that return before `createSlackMessageHandler`, such as authorized subtype system events and DM/MPIM `app_mention` skips. Leave normal handled messages/app mentions to the handler-level callback, add tests proving each path tracks exactly once, and frame the change as app-activity timestamp completeness rather than socket liveness.
