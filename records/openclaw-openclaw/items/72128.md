## Summary

Keep open. The PR is a reasonable implementation candidate for #72074, and the duplicate-token review comment has already been addressed with `replaceAll`, but the current branch still computes the fallback label before native and ACP spawn paths resolve the actual target agent and it does not update the checked-in config schema artifacts required for a new public config key.

## What This Changes

The PR adds optional `labelTemplate` config under `agents.defaults.subagents` and `agents.list[].subagents`, then uses it as the `sessions_spawn` label fallback when no explicit label is supplied.

## Best Possible Solution

Keep the PR open and update it into a target-aware, documented config feature: generate labels only after the native/ACP target agent is known, define whether per-agent templates are requester- or target-scoped, update generated config artifacts, and add focused tests plus docs/changelog coverage.

## Reproduction Assessment

Yes. A deterministic reproduction is to configure `agents.defaults.subagents.labelTemplate` as `{agentId}-{taskPreview}` and call `sessions_spawn` without `agentId`; the PR substitutes an empty `{agentId}` even though native spawn later falls back to the requester agent and ACP may fall back to `acp.defaultAgent`.

## Solution Assessment

No. The feature direction is sound, but computing the fallback label in the wrapper before target resolution is not the narrowest maintainable solution; the label fallback should run beside the existing target/requester resolution or use a shared helper fed with resolved ids.
