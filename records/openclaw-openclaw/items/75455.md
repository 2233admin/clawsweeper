## Summary

Keep open. Current main still routes BlueBubbles DM fallback through `senderId` when `chatIdentifier` is present, while the downstream resolver already supports `chat_identifier` targets; the PR looks directionally correct but needs the required changelog entry before merge.

## What This Changes

This PR changes BlueBubbles inbound DM chat resolution to prefer a provided trimmed `chatIdentifier` over `senderId`, and updates the focused resolver unit tests.

## Best Possible Solution

Merge the narrow resolver change after adding the required Unreleased Fixes changelog entry and running the focused BlueBubbles resolver test, while preserving the existing group behavior that refuses sender-handle fallback when group chat identifiers are missing.

## Reproduction Assessment

Yes. Current main has a focused helper-level reproduction: `buildBlueBubblesInboundChatResolveTarget` returns a sender handle for `isGroup: false` even when `chatIdentifier` is populated, and the existing test asserts that old behavior.

## Solution Assessment

Yes, with one process fix. Prefering a provided DM `chatIdentifier` before falling back to `senderId` is the narrowest code change, and downstream `chat_identifier` resolution is already supported; the PR still needs the changelog entry required for user-facing fixes.
