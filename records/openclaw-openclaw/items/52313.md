## Summary

Keep this open. Current main still respawns the gateway by reusing `process.execArgv` plus `process.argv.slice(1)` for both normal SIGUSR1 restarts and `update.run`; the update health fallback improves recovery, but it does not rewrite pnpm versioned realpaths to a stable package wrapper. The related implementation PR #52319 was closed unmerged, so the requested stable-entrypoint behavior remains unimplemented.

## Best Possible Solution

Keep this issue open and implement a narrow respawn planner in `src/infra/process-respawn.ts`: detect pnpm versioned OpenClaw realpaths, resolve a stable package-root or symlinked `openclaw.mjs` entrypoint, preserve dev/source entrypoints, and use the planner from both `restartGatewayProcessWithFreshPid()` and `respawnGatewayProcessForUpdate()`. Add focused regression tests for package installs, pnpm realpaths, source/dev runs, Bun/Node behavior, and update fallback behavior; reuse the stable service symlink logic from `src/daemon/program-args.ts` where it fits, and treat #52319 as prior art rather than a landed fix.

## Risks / Open Questions

- Closing now would leave the pnpm versioned-realpath respawn failure untracked for normal SIGUSR1 restarts, where the parent exits after spawning and there is no health probe before exit.
- The `update.run` health fallback reduces dead-gateway risk, but it can fall back to the old in-process module graph instead of proving that a stable updated wrapper was launched.
- A future fix touches process spawning across Node, Bun, source checkouts, package installs, `execArgv`, and `NODE_OPTIONS`; it should stay narrow and avoid reintroducing the broad unrelated surface from the closed XL PR.
