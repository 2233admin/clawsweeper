## Summary

Keep open. The latest force-push moved from removing the alias to preferring exact provider-map keys, which is safer for Kimi compatibility, but it does not cover the runtime paths that still re-normalize an already-resolved `kimi-coding` provider back to `kimi`.

## What This Changes

The PR adds case-insensitive exact provider-key lookup before alias fallback in `findNormalizedProviderValue` and `findNormalizedProviderKey`, with model-selection tests for those helper semantics.

## Best Possible Solution

Revise the PR so exact configured provider IDs such as `kimi-coding` survive default resolution, runtime execution, allowed-model checks, fallback candidate construction, and status display, while retaining `kimi` and `kimi-code` compatibility for the bundled Kimi provider. Add an end-to-end regression test for the linked configuration, not only helper-level lookup tests.

## Reproduction Assessment

Yes. The linked report gives a concrete `models.providers.kimi-coding` plus `agents.defaults.model.primary: kimi-coding/k2p5` configuration, and code inspection shows current runtime paths can still re-normalize that exact provider after selection.

## Solution Assessment

No. Exact-first lookup in the provider-map helpers is useful, but it is not the narrow complete fix for the reported behavior because the agent runtime and fallback paths still pass exact refs back through the global alias normalizer.
