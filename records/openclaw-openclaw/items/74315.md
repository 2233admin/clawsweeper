## Summary

Keep open. Current main still aliases `kimi-coding` to `kimi`, so the intended fix is not already implemented, but this PR is incomplete because the one-line alias removal would break existing Kimi plugin compatibility for legacy explicit provider overrides and leaves provider-scoped tool policy alias semantics inconsistent.

## What This Changes

The PR removes `kimi-coding` from `normalizeProviderId`'s Kimi alias branch and updates three model-selection expectations to preserve `kimi-coding` as the provider.

## Best Possible Solution

Land a revised patch that preserves `kimi-coding` for catalog and selected-model refs, adds a Kimi-plugin-local legacy lookup for existing `models.providers["kimi-coding"]` overrides, and updates the shared alias-policy fixture/tests so provider-scoped tool policy behavior is intentional.

## Review Findings

- **[P2] Preserve legacy Kimi explicit-provider lookup:** `src/agents/provider-id.ts:17`
  - body: Removing `kimi-coding` from the global normalizer makes the Kimi plugin stop matching existing `models.providers["kimi-coding"]` overrides when it looks up config for provider `kimi`. Add a plugin-local compatibility lookup, or those users lose their configured baseUrl/header overrides despite the existing Kimi tests covering that behavior.
  - confidence: 0.9

With this line changed, configs such as `tools.byProvider.kimi.deny` no longer apply to `kimi-coding` runs while the policy tests still encode the opposite contract.
  - confidence: 0.9
