## Summary

Keep open. Current main still has no SEA packaging surface, and this PR is a live implementation candidate rather than obsolete cleanup. The force-pushed branch addresses some older review concerns by removing the committed generated config and adding SHA256 verification, but it still needs maintainer follow-up because the Linux-target build can package host-platform node_modules and the downloaded production Node binary provenance needs an explicit security decision.

## What This Changes

The PR adds `build:sea` scripts and docs for a Node SEA sidecar package, adds Node-binary download/checksum tooling, skips CLI respawn under SEA/Bun, and adds a bundled runtime dependency mirror test.

## Best Possible Solution

Keep this PR open for maintainer review. If SEA packaging belongs in core, revise the builder so the sidecar dependency tree is produced for the target platform, strengthen Node binary provenance with a signed or pinned verification path, wire the new docs into the install navigation, and add focused artifact/container smoke coverage before considering merge.

## Reproduction Assessment

Yes for the main blocking concern: on any non-Linux host, the code path for `pnpm build:sea:linux-x64` or `pnpm build:sea:linux-arm64` still stages the checkout's host `node_modules` into `dist-sea`. I did not run the build because this review is read-only, but the staging behavior is explicit in `scripts/build-sea.mjs`.

## Solution Assessment

No, not yet. The sidecar-trampoline approach is plausible, and the force-pushed branch fixed several older review issues, but a container-targeted package needs target-platform dependencies and stronger binary provenance before this is the narrowest maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Build target-platform node_modules for Linux SEA packages:** `scripts/build-sea.mjs:170-184`
  - body: The Linux SEA targets download a Linux Node binary, but the sidecar still symlinks or copies the checkout's host `node_modules`. Cross-building from macOS will package macOS/native artifacts for dependencies such as `@lydell/node-pty`, `sqlite-vec`, or `sharp`, so the resulting `dist-sea/openclaw` can fail when run in a Linux container. Build the sidecar inside the target container or install/copy target-platform production deps before writing `dist-sea/node_modules`.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff introduces a production binary download path that now verifies SHA256, but still trusts an unsigned checksum fetched from the same origin.

Concerns:

- **[medium] Verify the downloaded Node binary with signed provenance:** `scripts/fetch-node-for-sea.mjs:138`
  - body: `fetch-node-for-sea.mjs` downloads `SHASUMS256.txt` and the Node archive from the same release origin, then trusts that checksum to create the executable base. That catches corruption, but a compromised origin/CDN/TLS path could replace both files; for a production container binary, prefer verifying Node's signed checksum file or pinning expected digests in a trusted release/build input.
  - confidence: 0.74
