## Summary

Keep open. Current main still has media-drop paths that need a visibility fix, but the latest PR head introduces a blocking silent-turn regression: a `NO_REPLY` payload whose media normalization fails now survives as a dropped-media warning even though the existing focused test expects no outbound payload.

## What This Changes

The PR adds a `DroppedMediaItem` pipeline for failed `MEDIA:` normalization through reply parsing, routing, outbound delivery, JSON projection, transcript notices, warning logs, and focused tests.

## Best Possible Solution

Keep the dropped-media visibility direction, but preserve the existing silent-reply contract by dropping or clearing `droppedMedia` for parsed-silent payloads whose media normalization fails before the renderable-payload filter runs. Then rerun the focused reply media, route, outbound payload, and test typecheck lanes before normal maintainer review.

## Reproduction Assessment

Yes. The existing `agent-runner-payloads.test.ts` case for `NO_REPLY\nMEDIA: ./missing.png` plus a throwing `normalizeMediaPaths` is a high-confidence reproduction path; the PR-head control flow now leaves `droppedMedia` on that payload and makes it renderable.

## Solution Assessment

No. The PR's general pipeline is a maintainable way to surface real dropped media, but the current implementation should special-case parsed-silent payloads so attachment diagnostics do not turn an explicit no-reply into a user-visible message.
