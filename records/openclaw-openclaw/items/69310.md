## Summary

Keep open. Current main still lacks the requested dropped-media pipeline for mixed media failures, so the PR remains useful, but the latest branch has a concrete user-visible regression: all-failed media keeps the legacy generic warning and also emits the new detailed dropped-media notice. The prior stale `reason` field issue appears fixed, and the security pass found no supply-chain concern.

## What This Changes

The PR threads dropped-media metadata through reply normalization, routing, outbound projections, delivery, transcript mirroring, warning logs, and focused tests so failed attachments produce user-visible notices.

## Best Possible Solution

Keep the dropped-media pipeline direction, remove the legacy generic warning when droppedMedia is populated for all-failed media, add or adjust an all-media-failed end-to-end test, and require the focused reply/outbound tests plus the changed gate to pass before merge.

## Reproduction Assessment

Yes. Current main can be checked from source with a reply containing one valid media URL and one invalid `MEDIA:` source: the invalid item is caught and skipped without a dropped-media notice; the PR finding is reproduced by an all-invalid media payload, where the normalizer keeps `⚠️ Media failed.` and delivery appends the detailed dropped-media notice.

## Solution Assessment

No, not as-is. The broad approach is the right maintainable direction, but the all-failed branch should not preserve the old generic warning once the new detailed dropped-media notice is emitted.
