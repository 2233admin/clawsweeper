## Summary

Keep open. Current main still drops some failed MEDIA attachments without a user-visible per-item notice, so the PR targets a real gap, but the latest branch can show duplicate warnings when every media item fails normalization.

## What This Changes

The PR adds a DroppedMediaItem pipeline through reply media normalization, routing, outbound payload projection, delivery notices, transcript mirroring, logs, and focused tests.

## Best Possible Solution

Keep the PR active, remove the legacy all-fail generic media warning when droppedMedia will be rendered, add a focused all-media-failed test that expects one detailed notice, then run the targeted reply/outbound tests, test typecheck, and changed gate before merge.

## Reproduction Assessment

Yes. On current main, a mixed payload with one bad MEDIA source and one valid source follows the catch-and-continue path without a user-visible notice for the failed item; on the PR branch, an all-media-failed payload follows reply-media-paths line 240 and deliver.ts line 1164, producing both the old generic warning and the new detailed notice.

## Solution Assessment

No, not quite yet. The droppedMedia pipeline is the right narrow direction, but the current implementation should not preserve the legacy generic warning in the same payload that will render a detailed dropped-media notice.
