## Summary

Keep this issue open. Current main still does not implement ACP session skill-context injection: `sessions_spawn({ runtime: "acp" })` has no `skills` selector/default/auto option, ACP-specific spawn params do not carry skill context, and the ACP runtime turn contract still forwards only text/attachments/control fields. Existing OpenClaw skill snapshot machinery and the bundled `acp-router` skill help the parent OpenClaw agent decide when to spawn ACP, but they do not inject selected SKILL.md content into the spawned ACP harness session.

## Best Possible Solution

Implement this as an additive ACP/session-spawn feature in core: add a typed `skills` selector or documented inherited default to `sessions_spawn`, resolve it through `buildWorkspaceSkillSnapshot` using the target agent’s allowlist and prompt limits, add a structured ACP prompt-context field or carefully bounded task-prefix fallback, and cover explicit lists, defaults/auto behavior, truncation, isolation, and ACP host-security behavior in ACP-specific tests and docs.

## Risks / Open Questions

- A naive implementation that prefixes raw SKILL.md content into ACP task text could leak local workflow details or sensitive instructions to external harnesses; the feature should be explicit, allowlisted, bounded, and auditable.
- ACP harnesses run outside OpenClaw’s normal tool/runtime boundary, so injected skills must remain instructions-only and must not imply access to OpenClaw tools, skill environment variables, provider credentials, or host permissions.
- This should not be closed as ClawHub-only: ClawHub can publish skills, but the requested behavior needs a core ACP/session-spawn API seam to choose and inject skill context for spawned ACP sessions.
