## Summary

Keep open. Current main does not implement per-session runtime envelopes, and this PR is a useful implementation path, but the diff introduces a security-sensitive authorization boundary with a concrete command-allowlist bypass that should block merge until fixed and reviewed.

## What This Changes

The PR adds a Gateway session `envelope` schema/store field for `sessions.create` and `sessions.patch`, enforces it in the before-tool-call path, and updates protocol docs plus focused tests.

## Best Possible Solution

Merge only after the command envelope uses complete shell-aware validation, with regression coverage for `pnpm test & curl https://example.com`; the deferred network policy should remain out of this field until runtime sandbox or proxy enforcement can provide a real boundary.

## Reproduction Assessment

Yes. A focused source check and JS probe show `bashCommandAllowlist: ["pnpm test"]` would allow `pnpm test & curl https://example.com`, and current main executes Bash commands through a shell where `&` is meaningful.

## Solution Assessment

No. The overall Gateway/session-envelope direction is plausible, but the current prefix-plus-regex filtering is not the best way to protect an authorization boundary unless it rejects all shell control operators or is replaced with structural shell-command analysis.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Reject single ampersand shell chaining:** `src/agents/session-runtime-envelope.ts:35`
  - body: The plain allowlist path checks the command suffix against `SHELL_CONTROL_PATTERN`, but that pattern omits a lone `&`. A command like `pnpm test & curl https://example.com` starts with the allowed prefix, has the required whitespace boundary, and passes this check, so Bash can run the extra command through the shell. Include `&` in the rejected operators or switch to structural shell parsing, and add this as a regression case.
  - confidence: 0.91
