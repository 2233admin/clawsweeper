## What This Changes

The PR adds optional `envelope` data to Gateway session create/patch APIs, persists it on session entries, enforces tool/path/command constraints before tool execution, and documents/tests the new protocol surface.

## Best Possible Solution

Review and, if tests and maintainer/security review pass, land the additive session-envelope implementation as the core Gateway surface while keeping network sandbox/proxy enforcement and any external authorization callback as explicit follow-up work tracked outside this PR.

## Reproduction Assessment

## Solution Assessment

Yes, with maintainer review still required: enforcing a persisted, declarative session envelope in the existing before-tool-call path is the narrowest maintainable core seam for tool/path/command constraints. Deferring network constraints is appropriate because hook-level URL scanning is not a reliable network boundary.
