## Summary

Keep open. Current main still lacks the requested per-session envelope surface, and this PR is the active implementation for the related open feature request, but the proposed patch introduces security-sensitive authorization bypasses that should block merge until fixed.

## What This Changes

The PR adds an optional session runtime envelope to Gateway `sessions.create` and `sessions.patch`, persists it on `SessionEntry`, evaluates tool/path/command/network constraints before tool execution, and updates docs plus session/envelope tests.

## Best Possible Solution

Keep the PR open and require the envelope to be enforced on the final executable tool params, fail closed when a stored envelope cannot be loaded, and either implement network constraints through enforceable runtime controls or narrow/document the field so it is not presented as a security boundary. After those changes, review it together with the related #74632 request and security expectations.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Recheck the envelope after hook parameter rewrites:** `src/agents/pi-tools.before-tool-call.ts:414-418`
  - body: The new guard evaluates the original `params` before trusted policies and ordinary `before_tool_call` hooks run. Those hooks can return replacement params, and the existing hook path later executes the merged params, so a hook can turn an allowed call into a denied path, URL, or command after the envelope has already passed. Run the envelope check on the final params as well so the persisted session policy remains the most restrictive result.
  - confidence: 0.92
- **[P1] Fail closed when the session envelope cannot be read:** `src/agents/session-runtime-envelope.ts:165-166`
  - body: `readSessionRuntimeEnvelope` catches every config/store error and returns `undefined`; `evaluateSessionRuntimeEnvelope` treats that as allowed. For a session that already has an envelope, any store read or path-resolution failure disables the policy and lets the tool execute unrestricted, so lookup failures need to block or surface an authorization error.
  - confidence: 0.89
- **[P1] Do not treat URL scanning as network enforcement:** `src/agents/session-runtime-envelope.ts:209-223`
  - body: The network branch only inspects literal `http(s)://` strings. `networkAccess: "none"` still allows Bash commands such as `curl example.com` or `ssh host`, and `localhost`/`allowlist` allow network-like tools with query-only params because no URL is found. Block network-capable tools unless enforcement is available, or wire this field to sandbox/proxy controls before exposing it as a runtime envelope.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff adds a new authorization boundary, but the implementation has concrete fail-open and bypass paths.

Concerns:

- **[high] Hook rewrites can bypass the envelope:** `src/agents/pi-tools.before-tool-call.ts:414`
  - body: Envelope evaluation happens before trusted and normal hook parameter rewrites, so the final executed parameters can violate the persisted session policy.
  - confidence: 0.92
- **[high] Network policy is bypassable:** `src/agents/session-runtime-envelope.ts:209`
  - body: The network policy only scans literal HTTP(S) URLs and does not reliably constrain Bash or query-style network tools, so clients could believe `networkAccess` is enforced when it is not.
  - confidence: 0.86
- **[medium] Stored policy lookup fails open:** `src/agents/session-runtime-envelope.ts:165`
  - body: Any exception while loading the session envelope returns `undefined`, which is treated as no policy and allows the tool call.
  - confidence: 0.89
