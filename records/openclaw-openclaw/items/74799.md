## Summary

Keep open. This is an active implementation PR for a still-missing Gateway/tool-authorization feature on current main, and the remaining action is maintainer/security review plus validation rather than conservative cleanup or an automated replacement.

## What This Changes

The PR adds an optional per-session `envelope` to `sessions.create` and `sessions.patch`, persists it on `SessionEntry`, enforces tool/path/command constraints in the before-tool-call path, and updates docs/tests.

## Best Possible Solution

Review and, if validation passes, land the additive session-envelope implementation as the core Gateway session surface while tracking network sandbox/proxy enforcement and any external authorization callback as separate follow-up work.

## Reproduction Assessment

Yes. Current main has no `envelope` field in the create/patch schemas and no before-tool-call envelope lookup, so the feature gap can be verified directly from the schema and enforcement path.

## Solution Assessment

Yes, with maintainer review still required. Persisting a declarative session-owned envelope and enforcing it in the existing before-tool-call path is the narrowest maintainable core seam for tool/path/command constraints; deferring network policy is appropriate because hook-level command scanning is not a reliable network boundary.
