## Summary

Keep this issue open. Current main still has adjacent agent event streams, session.tool/session.message broadcasts, session lifecycle notifications, background task records, Task Flow state, and trajectory capture, but none of those exposes the requested durable session-level process.* / step.* Gateway contract with stable step identity and related tool/message/artifact ids. The live issue has no protected labels and no open PR found by the unauthenticated GitHub search that references this item, so there is no paired implementation candidate to route through instead.

## Best Possible Solution

Keep this issue open and implement an additive Gateway protocol surface for durable session process events: likely a typed session.process or process/step event family plus a per-session/per-run replayable ledger. It should derive from existing lifecycle, item, plan, tool, transcript, task, and artifact signals where possible; provide stable step ids plus sessionKey, runId, sequence, timestamp, type, optional step title, and related ids; document the contract; and add tests for live streaming, reconnect, and reload recovery.

## Risks / Open Questions

- This is a public Gateway protocol addition, so the eventual implementation should be additive, typed, documented, and covered across schema, server broadcasts, clients, and reconnect/reload behavior.
- Existing live agent/session/tool broadcasts are close enough to reuse as inputs, but treating them as the durable ledger would miss the core requirement to make process state replayable and independent from assistant markdown.
- Task Flow is adjacent durable infrastructure for background workflows, but normal interactive chat sessions do not create task records, so it should not be considered a complete substitute without a session process bridge.
