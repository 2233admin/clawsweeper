## Summary

Keep this open. Current main still treats `sessions_send` and `sessions_spawn` as local-gateway session tools, the gateway config has only a single remote-client target rather than peer routing, and this issue is paired with an open same-author implementation PR using closing syntax.

## What This Changes

The issue requests native cross-gateway routing for `sessions_send` and `sessions_spawn` so agents on different gateway instances can message or delegate work without SSH scripts.

## Best Possible Solution

Keep this as the canonical native cross-gateway session messaging tracker. Maintainers should review, rework, land, or deliberately replace #43656 with an additive design that covers config schema/help, session-tool schemas, gateway auth, remote transcript/history ownership, visibility policy, tests, and public docs.
