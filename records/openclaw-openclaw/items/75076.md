## Summary

Keep open. Current main does not contain the proposed Control UI provenance/checksum/readback-health work, and this active PR is the reviewable implementation, but the latest head still has a blocking correctness issue: the new generic channel health summary treats unknown connection state as disconnected.

## What This Changes

The PR adds Control UI auth/CSP/media hardening, bootstrap build provenance, checksum artifacts in CI/release workflows, Slack/channel readback health status fields and warnings, related docs/scripts, and targeted tests.

## Best Possible Solution

Keep this PR as the reviewable implementation, but revise the generic channel health summary so it only warns on explicit disconnected states or on channel/account types that declare connection-state support. Preserve the Slack-specific mode-aware warning and the Control UI hardening work, then rerun the targeted Control UI, Slack, Gateway, script, and changed-gate validation before merge.

## Reproduction Assessment

Yes. Construct a ChannelsStatusSnapshot with a configured and running channel whose channel/account status omits connected, such as a passive/probed status summary; resolveChannelHealthSummary emits a no-active-connection/readback warning even though the state is unknown, not disconnected.

## Solution Assessment

No. The hardening direction is reasonable, but this PR is not the best mergeable solution until unknown connection state is distinguished from explicit disconnection; the safer fix is to gate the generic warning on connected === false or a declared connection-state capability/mode.
