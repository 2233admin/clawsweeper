## Summary

Keep this PR open for author revision. Current main does not already contain the requested Control UI hardening/provenance work, but the latest PR head still introduces a false channel-health warning by treating missing connection state as a disconnected channel.

## What This Changes

The PR hardens Control UI bootstrap, media, and CSP handling; adds build provenance and checksum artifacts; extends Slack/channel health status; updates docs, workflows, and scripts; and adds targeted tests.

## Best Possible Solution

Keep the PR as the reviewable implementation, but revise the generic channel health summary so it only warns on explicit disconnected states or on channel/account types known to publish connection state. Preserve the Slack-specific mode-aware warning and the Control UI hardening changes, then rerun the targeted Control UI, Slack, Gateway, script, and changed-gate validation before merge.

## Reproduction Assessment

Yes. Construct a ChannelsStatusSnapshot with a configured and running channel whose channel/account status omits connected, such as a passive/probed channel summary or Slack HTTP-mode account, and resolveChannelHealthSummary will emit the no-active-connection/readback warning.

## Solution Assessment

No. The hardening direction is reasonable, but the current PR head is not the best mergeable solution until unknown connection state is distinguished from an explicit disconnected state; the safer fix is to gate the warning on connected === false or a declared connection-state capability/mode.
