## Summary

Keep open. Current main still has a credible raw `codex-cli/gpt-5.4` helper failure path: the session-memory slug helper forwards the resolved provider/model unchanged, and the embedded runner still model-resolves that supplied provider before running. The newer provider/runtime split and doctor migration explain the intended legacy-config repair path, but they do not prove that an already-loaded raw helper call is normalized or rejected before hitting `Unknown model`.

## What This Changes

The issue requests that `codex-cli/gpt-5.4` helper and embedded runs resolve and authenticate consistently with the working `openai-codex/gpt-5.4` path.

## Best Possible Solution

Keep this issue open until maintainers choose and prove one explicit path: either normalize legacy `codex-cli/*` helper runs before embedded model resolution with session-memory slug-generator regression coverage, or make doctor/config validation/docs clearly enforce that raw `codex-cli/*` primary refs are legacy and unsupported for embedded PI helpers.
