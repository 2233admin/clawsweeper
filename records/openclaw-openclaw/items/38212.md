## Summary

Current main still does not prove the reported raw `codex-cli/gpt-5.4` session-memory/helper path is fixed. The helper still forwards the resolved provider/model into the embedded runner unchanged, model resolution still happens before auth alias selection, and `codex-cli` remains modeled as a CLI backend/auth compatibility alias rather than an `openai-codex` model-resolution provider alias. Recent docs and doctor migration point toward canonical provider refs plus runtime config, but that is a product-boundary decision, not proof that the original helper path is fixed.

## Best Possible Solution

Keep this issue open until maintainers choose and prove one product path. Either normalize legacy `codex-cli/*` helper/embedded runs to the intended `openai-codex/*` Codex OAuth route before model resolution and add a focused session-memory slug regression, or explicitly enforce/document that raw legacy `codex-cli/*` primary refs are unsupported for PI helper hooks and must be migrated to canonical model refs plus runtime configuration, with regression coverage proving loaded config cannot reach the helper as raw `codex-cli/gpt-5.4`.
