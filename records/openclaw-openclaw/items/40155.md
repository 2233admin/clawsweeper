## Summary

Keep this issue open. Current main still has a credible Telegram/WebChat divergence path: Telegram can pass per-DM, per-group, or per-topic skill filters into the shared reply path, including an intentional empty list that removes all skills, while WebChat chat.send has no equivalent skill-filter field. The generic vLLM tool_choice:auto fix is present, but current vLLM docs still describe model/server cases where skills/tools do not run or tool calls arrive empty/text-only, so the reported vLLM+himalaya mismatch is not proven fixed.

## Best Possible Solution

Keep this issue open. Reproduce on current main with the same agent/session, vLLM gpt-oss-20b, himalaya, and Telegram route; capture Telegram DM/group/topic config, effective replyOptions.skillFilter, built skillsSnapshot, runtime tool list, provider payload/tool_choice, and structured-versus-text tool-call events for both WebChat chat.send and Telegram. If Telegram unintentionally filters out himalaya, fix the Telegram filter inheritance or add diagnostics and a Telegram dispatch regression test. If the effective tool set matches but vLLM emits empty or text-only tool calls, fix or document that provider/model path more directly.
