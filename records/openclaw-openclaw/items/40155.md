## Summary

Keep this issue open. Current `main` still has a credible Telegram/WebChat divergence path around effective skill availability: Telegram can resolve per-DM/group/topic `skills` config, including an explicit empty list, and forward it as `replyOptions.skillFilter`, while WebChat `chat.send` has no comparable route skill-filter field. Provider-side vLLM fixes exist for some OpenAI-compatible tool-call behavior, but the docs still describe empty/text-only vLLM tool-call cases, so the reported Telegram plus vLLM plus himalaya mismatch is not proven fixed.

## Best Possible Solution

Keep the issue open. Reproduce on current `main` with the same agent/session, vLLM gpt-oss-20b, himalaya, and Telegram route; capture Telegram DM/group/topic config, effective `replyOptions.skillFilter`, built `skillsSnapshot`, provider payload/tool_choice, and structured-versus-text tool-call events for both WebChat `chat.send` and Telegram. If Telegram unintentionally filters out himalaya, fix the Telegram filter inheritance or add diagnostics and a Telegram dispatch regression test. If the effective tool set matches but vLLM emits empty or text-only tool calls, fix or document that provider/model path directly.
