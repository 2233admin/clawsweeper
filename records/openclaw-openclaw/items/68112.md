## Summary

Keep open: current main still has the startup-before-first-timer failure mode, and this PR remains relevant candidate work. The branch needs maintainer review and a rebase because current main has newer cron startup deferral and maintenance-recompute behavior that the PR must preserve.

## What This Changes

The PR wraps cron startup missed-job catch-up in error handling, adds recovery for stale running markers and interrupted one-shot state, introduces runtime zombie-marker cleanup, and adds start/zombie resilience tests.

## Best Possible Solution

Rebase this PR onto current main and land an equivalent startup recovery fix that guarantees cron reaches `armTimer(state)` after startup catch-up or final startup persistence failures while preserving deferred missed agent jobs, interrupted one-shot handling, and maintenance-only next-run recomputation.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Preserve deferred startup agent catch-up:** `src/cron/service/ops.ts:138-140`
  - body: Current main passes `deferAgentTurnJobs: true` when `start()` calls `runMissedJobs`, so overdue isolated agent-turn jobs are rescheduled after gateway startup. The guarded call added here only forwards `skipJobIds`; after rebase this would run those agent jobs synchronously during startup again and can block channel/provider startup. Include `deferAgentTurnJobs: true` in the guarded call.
  - confidence: 0.82
- **[P2] Keep startup maintenance recompute semantics:** `src/cron/service/ops.ts:197`
  - body: Current main's final startup block uses `recomputeNextRunsForMaintenance(..., { recomputeExpired: true })` so due jobs are not advanced without execution. This branch changes the touched final arm path back to `recomputeNextRuns(state)`; after a caught catch-up failure that can move past-due jobs forward and skip a run while the timer survives. Preserve the maintenance recompute call when rebasing this guard.
  - confidence: 0.78
