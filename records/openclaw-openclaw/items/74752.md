## Summary

Keep open. Current main still exposes a credible TTS-to-Telegram voice delivery gap: trusted local TTS media can be filtered before queueing, and Telegram outbound delivery does not forward the voice-note flag. Open PR #74901 explicitly targets this report, while #74920 overlaps the adjacent media delivery path, so the issue should stay open until maintainers land or consolidate the fix.

## What This Changes

The issue reports that the TTS tool returns `(spoken)` with Microsoft TTS on Telegram but sends a text message instead of a Telegram voice message, while direct Telegram media sending with `asVoice: true` works.

## Best Possible Solution

Keep this issue open and land a coordinated narrow fix that treats structured `trustedLocalMedia` TTS results as safe for local delivery, propagates `audioAsVoice` through Telegram `sendMedia`/`sendPayload` as `asVoice`, and adds regression coverage for TTS-originated Telegram voice media.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a TTS result with `(spoken)`, `/tmp/...ogg`, `trustedLocalMedia: true`, and `audioAsVoice: true` through `handleToolExecutionEnd` when `builtinToolNames` is present but lacks raw `tts`; current main can filter the local media before queueing, and the Telegram adapter also lacks the voice-flag mapping.

## Solution Assessment

Yes, a coordinated narrow fix is the best way to solve it. PR #74901 covers the local-media trust gate, but current main also shows a Telegram `audioAsVoice` forwarding gap, so maintainers should consolidate with #74920 or an equivalent small patch before closing.
