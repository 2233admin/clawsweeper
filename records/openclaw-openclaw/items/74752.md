## Summary

Keep open. Current `main` still shows a credible TTS-to-Telegram voice delivery gap: trusted local TTS media can be filtered before it is queued, and Telegram's outbound adapter still does not map `audioAsVoice` into the low-level `asVoice` option. Open PR #74901 uses closing syntax for this issue and PR #74920 overlaps the same media-delivery path, so the issue should remain open until maintainers land, consolidate, or replace that work.

## What This Changes

The issue asks for the TTS tool's generated audio to be delivered automatically as a Telegram voice message instead of returning `(spoken)` while sending text.

## Best Possible Solution

Land or replace the active fix work with one coordinated patch that lets trusted TTS local media survive tool-result filtering without weakening alias or MCP safeguards, maps `audioAsVoice` to Telegram `asVoice` through both `sendMedia` and `sendPayload`, and adds regression coverage for a TTS-originated Telegram voice payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a TTS result with local `/tmp/...ogg` media, `trustedLocalMedia: true`, and `audioAsVoice: true` through the tool-result handler while `builtinToolNames` is present but lacks raw `tts`; current main can filter the media away before queueing, and the Telegram adapter also drops the voice flag before low-level send.

## Solution Assessment

Unclear whether PR #74901 alone is the best complete fix because it covers the trusted-local-media gate while current main also shows a Telegram `asVoice` forwarding gap. The safest maintainable direction is a narrow coordinated fix across both boundaries with tests.
