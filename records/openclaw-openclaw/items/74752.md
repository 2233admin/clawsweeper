## Summary

Keep open. Current main still has a concrete TTS-to-Telegram delivery gap, and open PR #74901 explicitly targets this issue; overlapping PRs #74663 and #74920 touch adjacent Telegram voice forwarding, so the right cleanup is maintainer review/landing rather than closing the issue now.

## What This Changes

The issue reports that the TTS tool returns `(spoken)` but Telegram receives a text message instead of the generated voice message when TTS auto-delivery is enabled.

## Best Possible Solution

Land a coordinated narrow fix that honors trusted TTS local media through the tool-result filter and carries `audioAsVoice` through the Telegram registered payload/media paths into `sendMessageTelegram` as `asVoice`, with regression coverage for a TTS-originated Telegram voice payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a TTS tool result with a local `/tmp/...ogg` media path, `trustedLocalMedia: true`, and `audioAsVoice: true` through the tool-result handler when the exact registered tool set does not include raw `tts`, or through the Telegram payload path; current main can drop the local media before queueing or drop the voice flag before `sendVoice`.

## Solution Assessment

Unclear whether open PR #74901 is sufficient by itself because it addresses the trusted-local-media gate while current main also shows Telegram `audioAsVoice` forwarding gaps. The maintainable solution is a coordinated update covering both boundaries, then close this issue after the PR merges.
