## Summary

Keep open. Current main still has two credible gaps for the reported TTS-to-Telegram path: trusted local TTS media can be filtered before queueing, and Telegram outbound paths still do not map `audioAsVoice` to the low-level `asVoice` send option. Open PR #74901 uses closing syntax for this issue, while related media-delivery PR work only partially covers the path, so this should remain open until maintainers land or replace the coordinated fix.

## What This Changes

The issue requests that audio generated by the TTS tool be delivered automatically to Telegram as a voice message instead of returning `(spoken)` while sending text.

## Best Possible Solution

Land or replace the active fix work with one coordinated patch that preserves trusted TTS local media without weakening alias/MCP local-file safeguards, maps `audioAsVoice` to Telegram `asVoice` in the Telegram-owned outbound paths, and adds regression coverage for a TTS-originated Telegram voice payload.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to pass a TTS result with local `/tmp/...ogg` media, `trustedLocalMedia: true`, and `audioAsVoice: true` through tool-result handling when `builtinToolNames` is present but lacks raw `tts`; current main can filter the media away before queueing, and Telegram outbound code still drops the voice flag before `sendVoice`.

## Solution Assessment

Not yet. PR #74901 addresses the local-media filter direction but is not a complete solution by itself; the maintainable fix needs both the safe trust-boundary change and the Telegram adapter/channel `audioAsVoice` to `asVoice` handoff.
