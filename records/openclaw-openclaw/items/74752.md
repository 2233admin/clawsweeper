## Summary

Keep open. Current main still has a concrete Telegram delivery gap: the TTS tool and shared outbound pipeline preserve `audioAsVoice`, and Telegram's low-level sender can turn that into `sendVoice`, but the registered Telegram payload path does not forward the flag as `asVoice`. The existing ClawSweeper comment and overlapping PR #74663 are relevant context, but that PR does not fully cover the current registered plugin path described here.

## What This Changes

The issue requests that Telegram TTS tool output be delivered automatically as a Telegram voice message instead of returning `(spoken)` while sending text or a regular media message.

## Best Possible Solution

Forward `audioAsVoice` from TTS-generated `ReplyPayload` data through the registered Telegram `sendPayload` and media send path into `sendMessageTelegram` as `asVoice`, while preserving the existing direct message-tool behavior. Add a regression that starts from a Telegram-bound TTS/tool media payload and proves the final Telegram send uses `sendVoice` for voice-compatible audio.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to deliver a Telegram `ReplyPayload` with an `.ogg` or `.mp3` `mediaUrl` and `audioAsVoice: true` through the registered Telegram `sendPayload` path; current main reaches `sendMessageTelegram` without `asVoice`.

## Solution Assessment

No, relying on the current main behavior or open PR #74663 is not the best complete fix because the registered Telegram payload helper still drops the flag. The narrow maintainable fix is to carry `audioAsVoice` at the Telegram plugin boundary and cover both direct media and structured payload sends with tests.
