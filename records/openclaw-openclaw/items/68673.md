## Summary

Keep this PR open. Current main does not contain the proposed update_plan-specific loop-detection normalization, and the PR targets a real excessive-planning-loop problem, but the latest branch still has an unresolved behavior tradeoff: it collapses all plan step text into status/step-presence shape, which can make materially different plans look like identical no-progress churn.

## What This Changes

The PR adds update_plan-specific argument and outcome normalization in src/agents/tool-loop-detection.ts and regression tests for wording churn, status-progress, and outcome-progress cases in src/agents/tool-loop-detection.test.ts.

## Best Possible Solution

Keep the PR alive, but require a maintainer decision or branch update that distinguishes meaningful plan evolution from cosmetic update_plan wording churn before merge. The desired end state is targeted loop protection for repeated planning-only churn without blocking legitimate plan rewrites, with coverage for args and echoed result details.
