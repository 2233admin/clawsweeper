## Summary

Keep open. Current main still selects `.ogg` and `.opus` for multimodal memory audio, preserves the detected MIME and original bytes, and forwards them unchanged to Gemini `inlineData`; there is still no Gemini-specific audio format validation, Opus codec handling, transcode path, or non-fatal skip for invalid-format provider errors.

## Best Possible Solution

Keep this issue open and fix the Gemini multimodal memory path. The safest near-term product fix is provider-aware audio handling: unsupported Gemini audio codec/MIME combinations should not be sent to Gemini and should not abort the whole index; docs and tests should describe the supported raw formats. A fuller follow-up can transcode OGG/Opus to a Gemini-supported format through existing trusted ffmpeg plumbing if maintainers want true embed-success behavior for those files.

## Clownfish Work Prompt

Fix the Gemini multimodal memory OGG/Opus regression from this issue. Scope the PR narrowly: provider-aware Gemini audio handling should stop unsupported `.ogg`/`.opus` or unsupported audio MIME/codec inputs from being sent raw to Gemini, and memory indexing must continue with a clear warning instead of aborting the whole index. Do not add a new ffmpeg dependency or broad transcode pipeline in this PR. Update `docs/reference/memory-config.md` so Gemini multimodal memory does not promise raw OGG/Opus embedding support unless the implementation truly supports it. Add regression coverage for the unsupported audio path and for the existing successful `.wav` path. Keep the generic multimodal extension list provider-agnostic only if the provider-specific skip/warning is explicit and tested. Do not change unrelated providers, batch API behavior for text-only embeddings, or memory backend defaults. Changelog is expected because this is user-facing bug behavior.

## Risks / Open Questions

- A docs-only change would stop overpromising `.ogg`/`.opus`, but would not satisfy users who expect existing audio files to be embedded successfully.
- An ffmpeg-based transcode fix would touch local binary execution, resource limits, latency, cache identity, and failure behavior; that needs careful scoping even though OpenClaw already has trusted ffmpeg helpers for other media paths.
- A MIME-only OGG check can be wrong because Google's docs distinguish OGG Vorbis from Opus, so a robust implementation may need codec probing rather than treating all `audio/ogg` equally.
