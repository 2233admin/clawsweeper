## Summary

Keep open. Current main still selects `.ogg` and `.opus` as multimodal memory audio, preserves the detected MIME and raw bytes, and forwards them unchanged to Gemini `inlineData`. The issue discussion's workaround and dsantoreis's confirmation remain consistent with the current source; no provider-aware audio filter, codec check, or transcode path has landed.

## Best Possible Solution

Keep this issue open and fix the memory/Gemini embedding path. The best maintainer path is provider-aware audio handling: either transcode OGG/Opus to a Gemini-supported format before building `inlineData`, or skip unsupported codec/MIME combinations with a clear warning that does not abort the whole index. Update `docs/reference/memory-config.md` and add regression tests for `.ogg`/`.opus` so advertised formats, provider limits, and indexing behavior stay aligned.

## Risks / Open Questions

- A docs-only change would stop overpromising `.ogg`/`.opus`, but would not satisfy the issue's expected behavior that existing audio files embed successfully.
- A transcode fix needs careful handling of optional `ffmpeg`, cache identity, retries, cost, and latency for larger audio files.
- A provider-aware skip/filter fix should distinguish OGG Vorbis from OGG/Opus where possible; filtering only by `audio/ogg` may reject some files Gemini can accept.
