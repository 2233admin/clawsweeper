## Summary

Keep this issue open. Current main has partial runtime required-parameter checks for some built-in file tools and improved retry guidance, but it does not implement the requested generic, configurable pre-call validation layer or the proposed `tools.validation.enabled` / `autoPrompt` behavior. The related context is tool-specific and does not supersede this broader request.

## Best Possible Solution

Keep the issue open and design a core tool-dispatch validation path, likely after `before_tool_call` parameter rewriting and before the actual `tool.execute` call. It should use the normalized tool schema where possible, preserve plugin hooks and plugin-owned tools, add focused regression tests for `read({})` and `exec({})`, and make an explicit maintainer decision on whether `autoPrompt` should be a config option or handled through normal tool-error retry guidance.

## Risks / Open Questions

- A generic validator could break plugin tools with intentionally permissive or provider-normalized schemas if it validates too early or too strictly.
- Validation ordering needs product judgment: running before `before_tool_call` hooks could block plugin-owned parameter repair, while running after hooks may not satisfy every interpretation of pre-call validation.
- The proposed auto-prompt behavior can add hidden model turns and API calls, so it needs clear transcript, UX, and cost semantics before implementation.
