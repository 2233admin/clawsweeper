## Summary

Keep open. Current main still has partial internal exec/process diagnostics, but foreground exec results and process polling do not expose enough structured termination metadata to distinguish timeout, manual cancellation, supervisor SIGKILL, or raw OS signal; the related reports overlap but do not supersede this macOS user-facing diagnostics request.

## What This Changes

The issue requests better user-facing diagnostics when macOS exec commands are intermittently SIGKILLed, especially during broad scans over large OpenClaw state or log directories.

## Best Possible Solution

Add a focused diagnostics improvement that carries termination metadata through foreground exec and background process results: exitSignal, failureKind, exitReason or terminationReason, timedOut/noOutputTimedOut, timeout limit, run/session id, useful output tail, and a log or diagnostics correlation point. Keep signal escalation policy and drain-timeout changes in the broader supervisor thread unless maintainers decide to fold them together.

## Reproduction Assessment

Unclear. The intermittent macOS SIGKILL itself does not have a deterministic current-main reproduction, but the missing user-facing metadata is verifiable by inspecting and unit-testing the existing simulated SIGKILL, timeout, and process-poll paths.

## Solution Assessment

Yes for a first repair: enriching the result contract and tests is the narrowest maintainable fix for the reported diagnostic gap. Recursive-scan guardrails and SIGTERM-to-SIGKILL escalation should stay separate unless maintainers explicitly expand the scope.
