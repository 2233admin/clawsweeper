## Summary

Keep open. Current main still has the macOS SIGKILL diagnostics gap: the exec runtime and supervisor retain signal, timeout, and failure-kind metadata, but foreground exec results and process polling do not expose enough of that metadata to distinguish timeout, manual cancellation, supervisor kill, or OS signal causes. The related open PR and issues are adjacent, not a merged fix for this user-facing surface.

## What This Changes

The issue requests richer diagnostics when macOS exec commands are terminated with SIGKILL, including timeout/cancellation/source metadata, captured output context, and log/run correlation.

## Best Possible Solution

Add an additive diagnostics contract improvement that propagates existing supervisor/runtime metadata through foreground exec and background process results: exitSignal, failureKind, exitReason or terminationReason, timedOut, noOutputTimedOut, relevant timeout limits, and bounded output tail context. Keep graceful SIGTERM/SIGKILL policy changes in #66399 and broad-scan guardrails as separate follow-ups unless maintainers intentionally combine scope.

## Reproduction Assessment

Unclear: the intermittent macOS SIGKILL incident is not deterministic from the report. The missing user-facing metadata is source-reproducible on current main by tracing simulated signal, timeout, no-output-timeout, cancellation, foreground-result, and process.poll paths.

## Solution Assessment

Yes for the first repair step: enriching exec/process result metadata is the narrowest maintainable fix for the observable diagnostics gap. Broader signal escalation, OS memory diagnosis, or broad-scan guardrails should remain separate unless maintainers deliberately expand the work.
