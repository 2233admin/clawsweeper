## Summary

Keep open. Current main still has the same user-facing diagnostics gap: exec runtime and supervisor paths track some signal, timeout, and failure-kind data internally, but foreground exec results and background process polling do not expose enough structured termination metadata to distinguish timeout, manual cancellation, supervisor SIGKILL, or OS SIGKILL. The related open reports overlap but do not supersede this macOS-focused diagnostics request.

## What This Changes

The issue requests richer diagnostics for intermittent macOS exec SIGKILL failures, including termination metadata, preserved output tail, timeout context, and log/run correlation for broad state/log scans.

## Best Possible Solution

Add a focused exec diagnostics improvement that carries structured termination metadata through both foreground exec and background process results: exitSignal, failureKind, exitReason or terminationReason, timedOut/noOutputTimedOut, timeout limit, session/run id, useful output tail, and a diagnostics/log correlation point where the existing architecture supports it.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Unclear: the issue describes a credible recurring macOS pattern, but not a deterministic command/environment that reliably SIGKILLs on current main. The missing user-facing metadata is still reproducible by inspecting and unit-testing the simulated SIGKILL and timeout paths.

## Solution Assessment

Is this the best way to solve the issue? Yes for the first step: enrich the exec/process result contract and tests without changing process-kill semantics. Broader SIGTERM/SIGKILL escalation belongs in #66399 or in a maintainer-approved combined PR.
