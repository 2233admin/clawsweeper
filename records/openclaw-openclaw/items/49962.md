## Summary

Keep open. Current main still has the exact behavior this PR targets: explicit `groups` entries enable allowlist mode even when `groupPolicy` is explicitly `"open"`, so an unlisted group can still be rejected. The PR is a focused implementation candidate, not obsolete or superseded.

## Best Possible Solution

Keep this PR open and review/land this narrow fix or an equivalent patch: change `src/config/group-policy.ts` so `hasGroups` only implies allowlist mode when `groupPolicy` is not explicitly `"open"`, keep the backwards-compatible implicit allowlist behavior for unset policy, add the channel-level and account-level regression tests proposed by the PR, and clarify the groups docs around explicit-open overrides.

## Risks / Open Questions

- The PR changes access-control-adjacent behavior by broadening admission for configs that explicitly set `groupPolicy: "open"` and also define explicit `groups`; that appears intentional and documented, but maintainers should review whether any users relied on the older contradictory implicit allowlist behavior.
- The docs currently also say configured `groups` keys act as a group allowlist; the best landed version should clarify that this is for allowlist/implicit-allowlist mode, while explicit `groupPolicy: "open"` treats exact group entries as overrides.
