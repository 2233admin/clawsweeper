## Summary

Keep open. Current main still has the reported GitHub Copilot Claude Opus 1m capability gap, and the same reporter already has open PR #72825 with a focused fix while #72829 overlaps part of the same provider surface.

## What This Changes

The issue asks OpenClaw to treat GitHub Copilot Claude Opus 4.6/4.7 1m variants as 1M-context, 64K-output models and to allow `xhigh` thinking for `claude-opus-4.7-1m-internal`.

## Best Possible Solution

Land one provider-owned GitHub Copilot capability fix, likely #72825 or a reconciled canonical equivalent, that keeps dynamic resolution, static/default catalog metadata, session/status context lookup, max-output metadata, and thinking profiles consistent for the Claude Opus 1m ids; leave live `/models` discovery as a separate product decision.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: configure or select `github-copilot/claude-opus-4.7-1m-internal`, and the current Copilot fallback/default metadata still resolves to 128k/8192 while the provider thinking profile does not advertise `xhigh` for that id.

## Solution Assessment

Yes for the narrow direction. A provider-local capability table or equivalent metadata resolver shared by dynamic model resolution and default/static catalog generation, plus the targeted `xhigh` profile entry, is the smallest maintainable fix; broader live `/models` discovery can follow separately.
