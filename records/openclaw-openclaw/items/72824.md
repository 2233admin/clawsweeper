## Summary

Keep open. Current main still matches the reported GitHub Copilot behavior: unknown Claude Opus 1m model ids fall through to the synthetic 128k/8192-token defaults, and the Copilot xhigh allowlist does not include the Claude 4.7 1m internal id. The same author has an open paired PR, #72825, so the maintainer path is to review that PR or an equivalent fix rather than close this issue first.

## What This Changes

GitHub Copilot's Claude Opus 1m variants should use their advertised 1,000,000-token context and 64,000-token output limits, and `claude-opus-4.7-1m-internal` should allow `xhigh` thinking.

## Best Possible Solution

Keep this issue open until #72825 or an equivalent provider-local fix lands. The end state should source these Copilot model capabilities from a single GitHub Copilot-owned path used by both dynamic resolution and default model seeding, with regression coverage for the 1m context/output values and the `xhigh` thinking profile.

## Reproduction Assessment

Yes. On current main, selecting `github-copilot/claude-opus-4.7-1m-internal` as an unregistered Copilot model follows the synthetic dynamic resolver path that assigns 128k/8192 limits, and the Copilot thinking profile cannot expose `xhigh` because that id is not in `COPILOT_XHIGH_MODEL_IDS`.

## Solution Assessment

Yes for the immediate bug. A small GitHub Copilot-owned capability table used by both model-definition paths plus an `xhigh` allowlist update is the narrow maintainable fix; live discovery from Copilot `/models` would be a better future generalization, but it is not required to resolve the reported regression.
