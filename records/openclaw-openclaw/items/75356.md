## Summary

Keep open. The PR targets a valid Control UI/Gateway routing bug from #75203, but the latest head still only rewrites outbound `chat.send` while leaving the active UI session state on the stale heartbeat key, so base-session chat events can be dropped; it also omits the required changelog entry for a user-facing fix.

## What This Changes

This PR projects isolated-heartbeat base-session metadata through Gateway/UI session rows, hides marker-backed heartbeat rows from the Control UI session picker, retargets stale heartbeat `chat.send` requests to the base session, and adds focused regression tests.

## Best Possible Solution

Land a corrected version that keeps the marker-backed heartbeat contract, hides isolated heartbeat rows from normal chat selection, and normalizes the Control UI's active chat target consistently for send, stream/final event handling, abort, history reload, and session overrides. Add a single-line Unreleased Fixes changelog entry and focused tests for stale heartbeat selection completing on the resolved base session.

## Reproduction Assessment

Yes. Seed `sessionsResult` with only `agent:main:main:heartbeat` carrying `heartbeatIsolatedBaseSessionKey: "agent:main:main"`, send a WebChat message, then deliver a `chat` event for `agent:main:main`; source inspection shows the PR sends to the base session while `handleChatEvent` still filters against the stale heartbeat `state.sessionKey`.

## Solution Assessment

No. The marker-backed routing direction is right, but the current patch is incomplete because it rewrites only outbound send and not the rest of the active chat state; resolving the active session once and using that same key for send, event acceptance, abort, history, and session controls is the safer maintainable fix.
