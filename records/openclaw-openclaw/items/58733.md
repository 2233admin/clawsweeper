## Summary

Keep open. Current main still routes bundled Grok web_search through the shared 30 second web-search timeout, and the related timeout fix PR is still open/unmerged rather than shipped.

## What This Changes

The issue reports that bundled Grok web_search on the xAI Responses path inherits the shared 30 second timeout and asks for a provider-specific default long enough for slower Grok web searches.

## Best Possible Solution

Land or refresh a narrow xAI-owned fix that keeps the shared web_search default unchanged for other providers, gives Grok a longer fallback only when the user has not configured a timeout, and adds focused xAI web-search regression coverage. If maintainers prefer an explicit operator knob, align the xAI plugin schema and docs with that setting.

## Reproduction Assessment

Yes. The reporter gave a direct `requestXaiWebSearch(...)` reproduction with a working xAI key and timings at the 30 second boundary, and current source confirms the 30 second value still reaches the xAI Responses request.

## Solution Assessment

Yes, the narrowest maintainable direction is provider-owned: change the Grok default or expose a Grok-specific timeout without raising the shared default for all web_search providers. The open #43609 direction appears aligned, but it still needs maintainer review/rebase or replacement before this issue can close.
