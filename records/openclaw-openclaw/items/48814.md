## Summary

Keep this open. Current main still documents and implements queued follow-up turns after the active run finishes, while completed final payloads are preserved for delivery; the only delivery-time suppression I found is a narrower Telegram abort fence, not the requested general same-session pending-queue policy.

## What This Changes

The issue requests a pre-send check that suppresses or defers a completed channel reply when newer same-session inbound messages are already pending.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending follow-ups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define streaming preview cleanup, side-effect disclosure, routed delivery, plugin hook order, and whether content-aware classification is a later layer.
