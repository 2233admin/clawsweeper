## Summary

Keep open. Current main still queues newer inbound messages for a follow-up turn, but I did not find a same-session pre-send check that suppresses a completed stale final reply before channel delivery. The issue discussion is concrete and current enough to keep: comments also call out useful edge cases around tool side effects, content-aware routing, and late async/subagent results.

## Best Possible Solution

Keep this open for maintainer design. The likely product/code path is an opt-in core queue/delivery policy that checks same-session pending followups before visible final delivery, decides whether to suppress/defer the final payload, and preserves enough suppressed reply/tool context for the next collected turn. The design should explicitly cover block streaming, routed follow-up delivery, tool side effects, and whether content-aware classification belongs in a later follow-up.

## Risks / Open Questions

- A naive final-reply drop could hide the fact that tool calls or other side effects already happened during the superseded turn.
- Block streaming or tool-result delivery may already have emitted visible content before a final pre-send check runs, so the design needs clear scope for final-only versus all reply kinds.
- Changing default delivery behavior across all channels could surprise existing users; this likely needs an explicit, conservative policy before implementation.
