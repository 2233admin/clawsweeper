## Summary

Keep this open. Current main still queues newer same-session messages into a later follow-up/collect turn while preserving the active run's completed final payload for normal delivery; the only stale-final suppression I found is Telegram's explicit authorized-abort fence, not a generic pending-message pre-send policy.

## What This Changes

Request adding a pre-send same-session queue check that suppresses or defers a completed reply when newer inbound messages are already queued.

## Best Possible Solution

Add a tested, opt-in core queue/delivery policy that checks same-session pending follow-ups immediately before eligible visible final delivery, suppresses or defers only stale final payloads, and carries enough suppressed reply/tool context into the next collected turn. The design should explicitly define behavior for streaming previews, side-effectful tools, routed follow-up delivery, and whether content-aware classification is a later layer.
