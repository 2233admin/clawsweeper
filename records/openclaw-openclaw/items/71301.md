## Summary

Keep open. Current main has useful partial support through packaged docs, local docs path discovery, prompt guidance, and `config.schema.lookup`, but the central request remains missing: there is no native local-first docs search/get/API retrieval surface for agents, and the CLI docs search still uses the live Mintlify MCP endpoint.

## What This Changes

The issue requests a version-matched bundled docs knowledge layer plus first-class local-first docs retrieval for agents so onboarding, config, and API answers match the installed OpenClaw version.

## Best Possible Solution

Define a release-packaged docs manifest/search index, expose local-first docs search/get lookup through a maintained agent-facing runtime surface, keep live Mintlify search as an explicit fallback, and add tests/docs that distinguish packaged docs discovery from native retrieval.

## Reproduction Assessment

Not applicable as a runtime reproduction: this is a feature/product request. The missing capability is high-confidence by source inspection of the package allowlist, docs path resolver, system prompt, gateway tool/action inventory, gateway RPC list, CLI docs implementation, and docs/generated contents.

## Solution Assessment

Unclear: the requested direction is plausible and directly addresses onboarding/docs drift, but the exact interface still needs maintainer choice. A local-first manifest/index plus native retrieval API is safer than relying only on prompt guidance, generic file reads, or the live docs MCP endpoint.
