## Summary

Keep open. Current main still has only partial support: packaged/local docs are surfaced to the agent prompt and config.schema.lookup covers field-level config docs, but there is no native local-first docs search/get/API/changed-since retrieval surface for agents, and the CLI docs command still calls the live Mintlify MCP search endpoint.

## What This Changes

Request to ship version-matched bundled OpenClaw docs with each release and expose them through first-class local-first agent docs retrieval tools.

## Best Possible Solution

Define a release-built docs manifest/search index that ships with OpenClaw, expose local-first docs search/get lookup through a maintained agent-facing Gateway/tool surface, keep live Mintlify search as an explicit fallback, and add tests/docs proving the installed-version path is preferred.

## Reproduction Assessment

Not applicable as a runtime reproduction because this is a feature/product request. The missing capability is high-confidence by source inspection of the docs package allowlist, docs path resolver, system prompt, Gateway tool actions, Gateway RPC inventory, CLI docs command, and docs references.

## Solution Assessment

Unclear. The requested direction is plausible and well aligned with onboarding and docs-drift problems, but the exact package/index format, agent API, and fallback semantics need maintainer design before a narrow implementation PR is safe.
