## Summary

Keep this PR open. Current main still does not implement `agent_to_agent_turn` or an equivalent A2A turn-level plugin hook, and the PR is focused core plugin API work with a concrete channel-plugin use case from the linked DingTalk @sub-agent discussion. It needs maintainer design/security review and a rebase onto the current hook-contract layout rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer design review. The best path is to rebase it onto current main, implement a first-class typed A2A turn hook in `src/plugins/hook-types.ts` and `src/plugins/hooks.ts`, emit it from `src/agents/tools/sessions-send-tool.a2a.ts` for the initial reply and intermediate turns, document it at `https://docs.openclaw.ai/plugins/hooks`, and add focused tests for payload shape, routing context, ordering, timeouts, and conversation-access policy.

- Security/API review is still needed: the proposed hook exposes raw A2A reply text and routing metadata to plugins, so maintainers should decide whether it belongs in `CONVERSATION_HOOK_NAMES` and requires `plugins.entries.<id>.hooks.allowConversationAccess=true` like other conversation-content hooks.
- The PR adds no dependency, workflow, lockfile, release-script, or package-publishing changes in the provided file list, so the main supply-chain concern is the new plugin data-access surface, not third-party code execution.
- The PR should add focused tests and docs before landing: hook registration, turn 0 emission, flow ID stability, delivery target/thread fields, ordering and timeout behavior, final announce chronology, and conversation-access gating.
