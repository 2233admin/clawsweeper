## Summary

Keep open. Current main appears to handle workspace-local avatars through an authenticated same-origin route, but the remote `http(s)` case from the report remains a real contract mismatch: CLI/config still accept remote avatar values while Control UI intentionally refuses to render them under the current CSP/privacy posture.

## What This Changes

The issue requests that avatars configured with `openclaw agents set-identity --avatar` display in Control UI for both remote URLs and local workspace paths.

## Best Possible Solution

Resolve the avatar contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, content-type, size, redirect, cache, and failure-feedback rules, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config schema/help, docs, validation, and UI diagnostics while preserving the current local avatar route.
