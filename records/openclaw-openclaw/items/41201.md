## Summary

Keep open: current main covers the local workspace-avatar path, but the reported remote `http(s)` avatar case remains unresolved. The code, tests, and docs intentionally block remote image rendering for CSP/privacy reasons while CLI/config surfaces still accept remote avatar values.

## What This Changes

The issue requests that avatars configured via `openclaw agents set-identity --avatar`, including remote URLs and local workspace paths, render in Control UI instead of appearing broken.

## Best Possible Solution

Resolve the contract deliberately: either support remote avatars through an authenticated same-origin proxy/cache with explicit CSP, privacy, MIME, size, caching, diagnostics, and network controls, or make remote avatars clearly unsupported for Control UI by tightening CLI help, config schema/docs, validation, and UI feedback while preserving the current local avatar route.
