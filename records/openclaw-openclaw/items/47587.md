## Summary

Keep open. Current main partially improves the report: Telegram media groups now carry all downloaded images through `MediaPaths`, and image descriptions no longer suppress the described image from the prompt media header. The remaining reported behavior is still real enough to track: shared media understanding selects only the first matching image by default, so an album with multiple images will not produce N image descriptions unless `tools.media.image.attachments.mode = "all"` is configured, and Telegram still sets `MediaUrls` equal to local paths, which makes the media header render duplicate `path | path` text.

## Best Possible Solution

Keep this open and make an explicit maintainer decision on expected default behavior for multi-image inbound media. The most complete fix would preserve all Telegram `MediaPaths`, avoid rendering duplicate `path | path` when URL equals path, and add a focused regression test showing a Telegram album with a configured image model can produce descriptions for every selected image, either by changing the default for Telegram albums or by documenting/configuring `tools.media.image.attachments.mode = "all"` with a safe cap.

## Risks / Open Questions

- Changing the default image-understanding selection from first image to all images can multiply provider calls, latency, and cost for multi-image messages across channels, not just Telegram.
- A Telegram-only fix should not regress the existing generic media-understanding config contract or the current behavior where all image paths remain available to native vision models and downstream tools.
- The duplicate `path | path` media-note output is likely a narrow formatter/context cleanup, but it is separate from the all-images-transcribed behavior and should not be treated as the whole fix.
