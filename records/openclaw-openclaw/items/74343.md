## Summary

Keep open. Current main still lacks an account-aware Slack native-command naming or ownership path for duplicate slash command names across multiple Slack Apps, and the related per-account native prefix request remains open rather than merged or shipped. The report is recent, concrete, and current source/docs/tests do not prove the multi-app duplicate-command case is supported or intentionally unsupported.

## What This Changes

The issue requests correct Slack native slash-command routing when multiple Slack Apps in one workspace register identical commands and each app is expected to target a different agent/channel.

## Best Possible Solution

Keep this open until maintainers choose and implement the Slack multi-app policy: either document duplicate native command names across Slack Apps as unsupported, or add account-aware native command naming/mapping plus dispatch and argument-menu tests proving each app routes to its intended channel-bound agent.
