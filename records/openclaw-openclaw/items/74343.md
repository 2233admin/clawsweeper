## Summary

Keep open. Current main still lacks an implemented or documented support boundary for duplicate Slack native slash command names across multiple Slack Apps; the Slack handler resolves the command using the app/account that receives Slack's payload, while native command naming and lookup remain provider-scoped rather than account-scoped.

## What This Changes

The issue asks OpenClaw to route Slack native slash commands correctly when multiple Slack Apps in one workspace register the same command names for different agents.

## Best Possible Solution

Maintainers should define the supported Slack boundary. If multi-app native slash commands are intended, add an account-level native command naming or prefix mapping through config schema/types, generated config metadata, Slack native spec generation, dispatch lookup, and argument-menu callback handling; otherwise document that identical Slack native command names across Apps are unsupported and recommend unique per-app command names.

## Reproduction Assessment

Yes. The issue gives concrete two-app Slack reproduction steps, and current main has a source-level reproduction path: Slack can deliver a duplicate command to the winning app, while OpenClaw routes from that receiving account and has no account-specific native-name map.

## Solution Assessment

Unclear. There is no patch here, and the narrowest maintainable solution depends on product judgment between documenting duplicate identical Slack command names as unsupported or implementing per-account native command names as tracked by #38302.
