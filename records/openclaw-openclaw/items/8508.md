## Summary

Keep open. Current main still exposes ack reactions as a single configured value plus lifecycle status-reaction states; it does not have an `ackReactions` array, selector/strategy contract, random/contextual chooser, or pre-ack model hook. The prior maintainer/Codex comment on this issue reached the same conclusion, and the current checkout at `28ff82dcdae7f7fb11491deefe48bd82b68eb0bb` still matches it.

## Best Possible Solution

Keep this issue open for maintainer product/API design. Preserve the existing `ackReaction` string contract, then add a backward-compatible `messages.ackReactions` array or explicit selector/strategy contract only after maintainers choose deterministic, random, rule-based, or model-driven behavior. The implementation should update exported config types, zod schemas, help/labels/generated metadata, public docs, any SDK channel-feedback seam needed by third-party plugins, and focused channel tests for Slack, Discord, Telegram, Matrix, WhatsApp, and BlueBubbles behavior/fallbacks.
