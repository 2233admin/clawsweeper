## Summary

Keep open. Current main covers the reporter's concrete `multimedia.nt.qq.com.cn` class through the default `*.qq.com.cn` QQ Bot media allowlist, but the requested `channels.qqbot.mediaAllowHosts` setting still is not exposed or threaded into QQ Bot media downloads. The remaining work is valid and security-sensitive because inbound attachment downloads and outbound fallback downloads still use a fixed SSRF policy.

## What This Changes

Add a `channels.qqbot.mediaAllowHosts` configuration option that merges user-provided QQ Bot media host entries with the default SSRF hostname allowlist.

## Best Possible Solution

Implement the option in the bundled QQ Bot plugin with a conservative default policy, normalized host-pattern merge semantics, focused tests, and docs. The merged policy should apply to both inbound attachment downloads and outbound URL fallback downloads while preserving HTTPS-only handling, redirect safety, and private-network protections.
