## Summary

Keep open. Current main still lacks the proposed plugin/context-engine LLM completion surface, so the PR is not obsolete; however, the branch should not merge as-is because the runtime implementation reloads config ambiently, defaults model/auth resolution to the config default agent instead of the active session agent, and needs public SDK baseline/docs follow-through.

## What This Changes

The PR adds `api.runtime.llm.complete()`, public `PluginLlmComplete*` SDK types, a simple-completion-backed runtime LLM facade, and injects that helper into context-engine compaction, after-turn, and maintenance runtime contexts.

## Best Possible Solution

Ship an additive LLM runtime seam only after it is bound to the active agent/session or an explicit allowed agent selector, uses the active runtime config snapshot, documents the plugin-facing contract, updates the SDK API baseline, and has tests covering default and non-default agent model/auth resolution. If maintainers want a broader API shape, keep #45865 as the canonical design thread.

## Security Review

Status: needs_attention

Summary: The diff does not change CI, dependencies, or package scripts, but it adds a new model-inference capability that can use configured model credentials and spend tokens from plugins/context engines.

Concerns:

- **[medium] Review plugin LLM credential and cost controls:** `src/plugins/runtime/types-core.ts:160`
  - body: Adding `PluginRuntimeCore.llm.complete` lets loaded plugins and context engines invoke model completions through OpenClaw credentials. The patch does not add an explicit capability gate, audit/cost attribution, or active-session binding, so maintainers should settle those controls before exposing this surface.
  - confidence: 0.84
- **[medium] Bind completion auth to the active agent:** `src/plugins/runtime/runtime-llm.runtime.ts:19`
  - body: The implementation defaults omitted `agentId` to the config default agent, which can route completions from non-default sessions through the wrong model/auth profile. That is both a correctness and credential-scope concern for multi-agent installs.
