## Summary

Current main still appears to leave the reported bug open. The media-understanding provider resolver still returns any active non-speech provider list before compat fallback, so a partial active registry can still omit configured Deepgram audio STT while Discord voice uses that shared transcription path.

## What This Changes

The issue requests stable Discord voice follow-up transcription when Deepgram is configured through `tools.media.audio.models` and available through bundled compat provider loading.

## Best Possible Solution

Extend capability provider resolution so requested `tools.media.audio` media-understanding providers are merged into a partial active registry by provider id, while still avoiding unrequested bundled provider fanout. Add a regression test for active OpenAI/MiniMax media providers plus configured Deepgram audio STT.

## Repair Work Prompt

Fix the Discord voice Deepgram follow-up transcription bug reported in https://github.com/openclaw/openclaw/issues/65687. Keep the change focused in `src/plugins/capability-provider-runtime.ts` and tests. The observable bug is that `resolvePluginCapabilityProviders({ key: "mediaUnderstandingProviders", cfg })` returns a partial active media registry before compat loading, so configured `tools.media.audio.models: [{ provider: "deepgram", model: "nova-3" }]` can fail with `Media provider not available: deepgram` after Discord voice captures a valid follow-up WAV. Add a generic requested media-understanding provider collection/merge path for `tools.media.audio` that merges only requested missing providers from compat loading by provider id; do not pull unrelated bundled media providers, do not special-case Discord or Deepgram beyond test fixtures, and do not change Discord voice capture/TTS behavior or Deepgram auth semantics. Add a regression test where active `mediaUnderstandingProviders` contains OpenAI/MiniMax, bundled compat loading can provide Deepgram, config requests Deepgram audio STT, and the resolved provider ids include the active providers plus Deepgram without unrelated providers. Changelog is likely not needed if this is treated as internal bugfix/test coverage only; add one only if maintainers require user-facing release notes.
