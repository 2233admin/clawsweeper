## Summary

Keep open. The voice-call half remains useful because current main still passes an empty `remoteAddress` through to a limiter that treats empty keys as fail-open. The Google Chat half needs revision before merge: current main already supplies a non-empty default key in the shared resolved webhook pipeline, and the PR's raw `req.url` key can still fragment buckets for normalized path variants.

## What This Changes

The PR adds a fallback non-empty in-flight key for voice-call webhooks when `remoteAddress` is missing and passes an explicit Google Chat webhook in-flight key derived from the request path without the query string.

## Best Possible Solution

Keep the voice-call fallback-key fix, change Google Chat to derive its key from the resolved normalized webhook path, and add focused tests for the missing-remote voice-call case plus Google Chat query and trailing-slash key stability.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Normalize the Google Chat path before keying limits:** `extensions/googlechat/src/monitor-webhook.ts:194`
  - body: This builds the limiter key from raw `req.url` before `withResolvedWebhookRequestPipeline` resolves and normalizes the target path. The gateway and target resolver can treat variants such as `/googlechat` and `/googlechat/` as the same webhook, but this key gives them separate buckets, leaving a limiter-fragmentation bypass. Build the key from the resolved `path` callback or normalize `new URL(...).pathname` first.
