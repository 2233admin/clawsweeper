## Summary

Keep this PR open. The hook direction addresses a real extension boundary, but the proposed merge point lets plugin-returned env vars overwrite the already-sanitized host exec environment, bypassing the current PATH and loader-variable policy for gateway/node execution.

## What This Changes

The PR adds a typed `resolve_exec_env` plugin hook, a hook-runner method, exec-tool wiring that merges returned env vars, and focused hook tests.

## Best Possible Solution

Revise this into a documented additive exec-environment seam that preserves the existing host env reserved-key policy, or gates reserved-key overrides behind an explicit trusted-plugin contract. The final design should carry the needed channel/sender context, cover gateway/node/sandbox behavior, and update plugin docs plus changelog before merge.

## Reproduction Assessment

Yes for the blocking review finding: register a `resolve_exec_env` handler returning `PATH` or `LD_PRELOAD` for `host=gateway`/`node`; in the PR diff, the hook output is assigned after `sanitizeHostExecEnvWithDiagnostics`, so the blocked value would reach the final execution env. I did not run tests because this review was required to stay read-only.

## Solution Assessment

No, not as-is. The hook is a plausible direction, but the merge point should reuse the host env sanitizer for hook output or introduce explicit trusted-plugin semantics before exposing this as a public SDK seam.
