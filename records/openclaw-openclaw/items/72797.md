## Summary

Keep this PR open. Current main still does not expose `resolve_exec_env`, does not return a `runResolveExecEnv` hook runner method, and builds the exec environment without a plugin injection point. The PR is a recent, focused SDK/runtime seam rather than an obsolete or ClawHub-only integration; the remaining maintainer decision is the security contract for plugin-provided exec environment keys.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to decide whether channel-specific exec environment injection should be a dedicated typed hook or an extension of existing tool hooks; if the dedicated hook is accepted, land it with documented host/sandbox/node semantics, SDK docs, merge-order tests, error-isolation coverage, and an explicit trusted-plugin/reserved-env-key policy.

## Risks / Open Questions

- The proposed hook touches exec environment construction, a code-execution path. Maintainers should explicitly decide whether trusted plugins may override reserved keys such as `PATH`, `LD_*`, `DYLD_*`, `HOME`, proxy, TLS, or credential-routing vars, and whether that differs for gateway, node, and sandbox hosts.
- If accepted, this becomes a public plugin hook and needs matching docs/API export expectations plus tests for merge order, error isolation, host-specific behavior, and reserved-key policy.
- The PR discussion already fixed the context alias and error-isolation test, while intentionally leaving env clobbering as a trusted-plugin design choice; that remaining tradeoff is review material, not a cleanup-close reason.
