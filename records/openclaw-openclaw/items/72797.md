## Summary

Keep this PR open. Current main at f6d23ab5c2d8e7cbf50522b6c8137241524b6646 still does not define or run a resolve_exec_env plugin hook, so the proposed SDK seam is not obsolete. The PR remains security-sensitive because its patch would merge plugin-provided environment keys into exec after the current host/sandbox env construction and host override checks.

## Best Possible Solution

Keep this PR open for maintainer review. If maintainers accept the seam, land it as a typed plugin SDK hook with explicit host/node/sandbox semantics, channel/account/thread context, docs/API coverage, focused tests, and a clear reserved-key or trusted-plugin policy. If broad env injection is too risky, redirect the work to a narrower exec/tool extension design instead of closing it as obsolete.

## Risks / Open Questions

- Merging the PR as-is could let plugin hook results clobber PATH, HOME, loader variables, or other env keys after the current host env override sanitization path has already run.
- A new public plugin hook needs stable host/gateway/node/sandbox semantics, merge ordering, error behavior, docs, and SDK export coverage before it becomes a third-party contract.
- The proposed context only includes messageProvider/channelId in the patch; the motivating channel use cases may also need accountId/thread/source-target semantics, which current exec defaults carry but PluginHookAgentContext does not fully expose.
- Closing this PR as implemented would drop a still-missing plugin SDK seam; current main has no equivalent hook.
