## Summary

Keep open. Current main still only registers direct Moonshot under `moonshot`, there is no `moonshotai` direct-provider alias in the normalizer or Moonshot manifest, and the supplied GitHub context shows open PR #74946 still targets this issue with closing syntax.

## What This Changes

Accept OpenRouter-style `moonshotai/<model>` direct Moonshot model refs without breaking existing `moonshot/<model>` configs.

## Best Possible Solution

Land or replace #74946 with a narrow backwards-compatible alias that accepts `moonshotai/<model>` for direct Moonshot catalog and runtime resolution while preserving canonical `moonshot` provider/auth/plugin ids and adding regression coverage for both catalog planning and model selection.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: `parseModelRef("moonshotai/kimi-k2.6", ...)` keeps provider `moonshotai`, while current main only has Moonshot catalog/auth/provider entries under `moonshot`, which leads to the reported unknown-model path.

## Solution Assessment

Yes for a compatibility alias, no for the issue body's suggested hard rename. The narrow maintainable solution is to use the existing manifest catalog alias seam or equivalent provider-id normalization while keeping `moonshot` canonical.
