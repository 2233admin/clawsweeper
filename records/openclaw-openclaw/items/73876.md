## Summary

Current main still rejects the reported direct `moonshotai/kimi-k2.6` form. Moonshot is registered, onboarded, and documented as `moonshot`, OpenRouter examples use `openrouter/moonshotai/...`, and provider normalization has no `moonshotai` to `moonshot` alias. The linked fix PR #73934 is closed unmerged, so the issue still needs a focused compatibility fix.

## Best Possible Solution

Keep this issue open and implement a backward-compatible provider alias: accept `moonshotai/<model>` and probably `moonshot-ai/<model>` as direct Moonshot by normalizing those provider ids to canonical `moonshot`, while preserving the existing `moonshot` plugin id, auth profiles, config keys, and docs as canonical. Add regression coverage and a short docs/changelog note; #73934 is useful prior art but is closed unmerged.
