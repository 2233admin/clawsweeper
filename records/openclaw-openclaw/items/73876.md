## Summary

Current main still has the reported Moonshot provider slug mismatch. Direct Moonshot is registered, discovered, onboarded, and documented as `moonshot`, while `moonshotai` is only used as a nested routed catalog slug for OpenRouter/proxy providers. `moonshotai/kimi-k2.6` still parses as provider `moonshotai` and has no current alias to the direct Moonshot provider.

## Best Possible Solution

Keep this issue open and implement a backward-compatible direct-provider alias so `moonshotai/<model>` resolves to direct Moonshot without changing the canonical `moonshot` plugin id, auth profile ids, or existing `moonshot/<model>` configs. Prefer plugin-owned alias/catalog surfaces, add regression tests for parsing/catalog/allowed-model/fallback behavior, and document the accepted alias if it becomes user-facing.
