## Summary

Current main still has the reported mismatch: direct Moonshot is registered and documented as `moonshot`, while `moonshotai` appears as an OpenRouter/proxy catalog slug. Model refs split on the first slash and provider normalization has no `moonshotai` -> `moonshot` alias, so `moonshotai/kimi-k2.6` remains a real current-main bug rather than something implemented or superseded.

## Best Possible Solution

Keep this issue open and implement a backward-compatible alias or canonicalization path that accepts `moonshotai/<model>` as direct Moonshot while preserving existing `moonshot/<model>` configs, auth profiles, plugin id, and docs. Add regression coverage for model parsing/resolution, allowlist/fallback behavior, onboarding/docs output, and a clearer unknown-model hint if maintainers intentionally reject the alias.
