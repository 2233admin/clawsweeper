## Summary

Keep open. Current main still registers direct Moonshot only as `moonshot`, has no `moonshotai` direct-model alias, and open PR #74946 is still the unmerged fix candidate with closing syntax for this issue.

## What This Changes

The issue requests that direct Moonshot model refs copied from OpenRouter, such as `moonshotai/kimi-k2.6`, resolve through the direct Moonshot provider while preserving existing `moonshot/...` configs.

## Best Possible Solution

Keep `moonshot` as the canonical provider id and land or revise a narrow compatibility alias so `moonshotai/<model>` resolves through the direct Moonshot catalog/runtime/auth path, with regression coverage for catalog planning and actual model selection.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: configure `moonshotai/kimi-k2.6`; current parsing treats `moonshotai` as the provider, provider normalization leaves it unchanged, and the Moonshot manifest only owns `moonshot`.

## Solution Assessment

Yes for the compatibility-alias direction, no for the issue's suggested hard rename. The existing manifest alias seam is the narrowest maintainable path because it preserves existing `moonshot/...` configs while accepting the OpenRouter-style slug.
