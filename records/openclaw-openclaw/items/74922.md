## Summary

Keep open: current main still suppresses Discord ack/status reactions for guild/channel source turns when source reply delivery is `message_tool_only`, which matches the reported `ackReactionScope: "all"` guild/channel shape. The confirmed TLS/channel-resolution failures are useful context, but current source shows the startup channel-resolution fallback preserves configured guild entries, while the #73133 change is on inbound reaction notifications rather than automatic ack reactions.

## What This Changes

The issue requests restoring automatic Discord `ackReaction` in guild/channel messages with `ackReactionScope: "all"` after upgrading to v2026.4.27, especially when channel resolution logs TLS/fetch failures.

## Best Possible Solution

Define the Discord/group visibility contract explicitly. If configured ack reactions are allowed in tool-only rooms, change the ack/status gate narrowly and update Discord plus shared group visibility tests/docs; if not, keep the privacy behavior and add clear docs/logging that ack/status reactions are suppressed unless `messages.groupChat.visibleReplies: "automatic"` is configured.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists in `extensions/discord/src/monitor/message-handler.process.test.ts`: a guild route with `ackReactionScope: "all"` and `ackReaction: "👀"` currently expects no Discord reaction calls because source delivery is `message_tool_only`.

## Solution Assessment

Unclear. Exempting `ackReactionScope: "all"` from the tool-only gate would address this report, but it may conflict with the v2026.4.27 privacy contract that deliberately suppressed automatic status-reaction output in group/channel rooms.
