## Summary

Keep open: current main still suppresses Discord ack/status reactions for guild/channel source turns that resolve to `message_tool_only`, and the reported `ackReactionScope: "all"` shape matches an existing test that expects no Discord reaction call. The channel-resolution failure is useful live context, but the source code falls back to configured guild entries; the unresolved part is the product contract between explicit ack reactions and the v2026.4.27 privacy behavior for always-on rooms.

## What This Changes

The issue requests that automatic Discord `ackReaction` fire in guild channels with `ackReactionScope: "all"` after upgrading to v2026.4.27, even when Discord channel resolution logs a fetch failure.

## Best Possible Solution

Define the Discord guild/channel contract explicitly. If configured ack reactions are allowed in tool-only rooms, narrow the Discord ack/status gate and update the matching tests and docs; if they are intentionally suppressed, document the caveat in Discord ack-reaction docs and add clearer logging that points operators to `messages.groupChat.visibleReplies: "automatic"` when they want visible automatic reactions.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: the Discord process test configures a guild/channel route with `ackReactionScope: "all"` and `ackReaction: "👀"`, then expects no reaction because the source reply mode is `message_tool_only`.

## Solution Assessment

Unclear. Removing the `!sourceRepliesAreToolOnly` gate would satisfy this report, but it could conflict with the shipped #73046 privacy contract that suppresses automatic status-reaction output in always-on rooms; the safer path is a maintainer decision followed by a narrow code/docs/test update.
