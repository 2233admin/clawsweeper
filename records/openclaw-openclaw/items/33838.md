## Summary

Keep this issue open. Current main still models node command declarations as command-name arrays across the WebSocket connect schema, node pairing, runtime registry, node discovery, generated/mobile clients, and docs. `node.invoke` validates the envelope and command allowlist, then forwards unknown params/results without command-specific input/output schema validation. The adjacent #8287 is related, but it tracks node-registered agent tools rather than this narrower gateway command-schema protocol gap.

## Best Possible Solution

Keep open for maintainer-designed additive gateway protocol work. Preserve `commands: string[]` compatibility, add optional per-command schema metadata, carry it through connect, node pairing request/persistence, registry/session state, `node.list`/`node.describe`, generated Swift/Kotlin clients, node-host/plugin runtime APIs, docs, and focused tests. Define explicit validation semantics for `node.invoke` params and results.
