## Summary

Keep open. Current main still documents and implements ACP/acpx as a Gateway-host external harness runtime, while remote node-host support is scoped to `exec host=node` and `system.run`; there is still no ACP runtime node selector, Discord binding host selector, or acpx remote process-placement contract. The prior ClawSweeper comment also kept this open, and the stale label was removed afterward, so this remains a product/docs follow-up rather than conservative cleanup.

## What This Changes

The issue asks OpenClaw to clarify or fix the Discord ACP flow for a split topology where the Gateway runs on a Raspberry Pi and Claude Code/Codex harness execution should run on a remote Mac node.

## Best Possible Solution

Keep this open for maintainer product/docs judgment. The short-term end state should explicitly document that bundled Discord `/acp spawn` and acpx harnesses run on the Gateway host today, with practical workarounds such as running the Gateway/acpx on the Mac or using an operator-managed wrapper. If first-class support is desired, it should add a typed remote ACP runtime node selector and process-execution contract across config schema, ACP runtime SDK, acpx integration, node-host/RPC handling, Discord docs, and targeted tests.
