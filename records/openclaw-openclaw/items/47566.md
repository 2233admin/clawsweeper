## Summary

Keep this PR open: current main still lacks a plugin scaffold command, so the contribution is not obsolete, but the branch is not merge-ready because its new validator rejects scoped plugin ids that current plugin install/update paths already support.

## What This Changes

The PR adds an `openclaw plugins init [directory]` CLI subcommand with a scaffold generator and unit tests for manifest, entrypoint, package, validation, and overwrite behavior.

## Best Possible Solution

Land a rebased version that reuses the existing plugin-id validator from `src/plugins/install-paths.ts`, adds scoped-id regression coverage for `@scope/name`, and updates the generated template plus plugin CLI/authoring docs so new scaffolds follow current entrypoint and activation guidance.

## Reproduction Assessment

Yes. The issue is reproducible by static inspection: the PR rejects any `/` in scaffold ids, while current main accepts and tests `@scope/name` through `resolvePluginInstallDir`.

## Solution Assessment

No. The feature direction is useful, but the maintainable solution is to share the existing plugin install validator rather than introduce a stricter CLI-only validator that diverges from install/update behavior.
