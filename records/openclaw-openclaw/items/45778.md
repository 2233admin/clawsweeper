## Summary

Keep open. Current main still lacks both requested platform paths: cron payloads are still limited to `systemEvent` and `agentTurn`, detached cron still dispatches through the isolated agent runner and model-selection flow, and skill env injection still only applies configured `skills.entries.<key>.env` / `apiKey` values rather than foreground chat reply-route variables. The SignalRadar ClawHub use case and the 2026-04-29 reporter update keep the issue current, but the directExec and reply-route env options both need maintainer product/security direction before a safe fix PR is appropriate.

## What This Changes

Request to let skills run zero-LLM background monitoring while still delivering chat messages, either through an opt-in directExec cron payload or foreground reply-route environment variables.

## Best Possible Solution

Keep this open for maintainer product and security review. The likely path is either a scoped foreground reply-route injection contract for skills, including channel/account/thread semantics and user authorization, or an opt-in directExec cron payload restricted to approved installed-skill commands and wired through existing cron delivery, run history, timeout, and failure-notification behavior without invoking a model. Whichever path is chosen should update the cron tool/CLI/UI/docs and add focused cron plus skill-env tests.

## Risks / Open Questions

- A directExec cron payload would execute configured commands as the Gateway process user unless it is scoped by explicit operator opt-in, installed-skill command provenance, timeouts, audit/run logs, and failure-delivery policy.
- Foreground reply-route env vars are privacy and permission sensitive because a skill could persist channel/account/thread targets and later send messages outside the original foreground context.
- The issue intentionally offers two viable platform designs, so implementing one without maintainer direction could create the wrong public contract or bypass later plugin/skill API design.
