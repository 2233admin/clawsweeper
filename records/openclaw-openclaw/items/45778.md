## Summary

Keep open. Current main still only supports cron payloads that enqueue a system event or run an agent turn, so the requested zero-LLM direct execution path is not implemented. Current skill env injection is limited to configured skill env/apiKey values and does not expose the foreground chat reply route as OPENCLAW_REPLY_CHANNEL / OPENCLAW_REPLY_TARGET. The SignalRadar ClawHub skill context and dsantoreis's comments point to a real missing platform seam, but the directExec option needs explicit maintainer security/product review before implementation because it would run commands as the Gateway user and deliver output to chat.

## Best Possible Solution

Keep this issue open for maintainer design review. The best product path is a narrow core/platform seam: either foreground reply-route injection with clear channel/account/thread scoping, or an opt-in directExec cron payload that runs only approved installed-skill commands and reuses existing cron delivery/run-log/failure semantics without invoking the model. Any implementation should update cron types, normalization/validation, agent-facing cron tool schema, docs, and focused cron + skill env tests.

## Risks / Open Questions

- A directExec cron mode would execute configured commands as the Gateway process user unless it has an explicit operator opt-in, command/source restriction, audit trail, timeout, and delivery policy.
- Injecting reply-route env vars into skill processes is privacy and permission sensitive because skills could persist channel/account/thread targets and later send messages outside the original foreground context.
- The issue presents two different solution paths, and choosing the wrong one could either under-solve the zero-LLM cron use case or add a broad execution surface without a settled trust model.
