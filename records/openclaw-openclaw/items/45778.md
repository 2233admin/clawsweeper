## Summary

Keep open. Current main still lacks both requested platform paths: cron has no zero-LLM direct execution payload, isolated cron execution still requires an agent turn, and skill environment injection does not expose a foreground reply route. The reporter reconfirmed the SignalRadar workaround recently, and the remaining decision is product/security scoping rather than stale cleanup.

## What This Changes

Request zero-LLM background skill notifications by either adding a directExec cron payload with announce delivery or injecting foreground reply-route environment variables for later crontab sends.

## Best Possible Solution

Expose one explicit zero-LLM notification contract that preserves channel permissions: either scoped foreground reply-route injection for user-authorized skill runs, or an opt-in directExec cron payload restricted to approved installed-skill commands and wired through existing cron delivery, logging, timeout, and failure-notification semantics.

## Reproduction Assessment

Yes. Static current-main inspection is enough: directExec is absent from the cron type/schema/CLI/UI surfaces and detached cron execution rejects non-agentTurn payloads, while repo search finds no OPENCLAW_REPLY_* route injection path for skills.

## Solution Assessment

Unclear until maintainers choose the contract. Option A is the more general scheduler fix but needs host-exec security controls; Option B is smaller but still needs consent, account/thread scoping, persistence, and revocation semantics.
