## Summary

Keep open. Current main still has a plausible static path for the reported phantom `sessions.list` row: the gateway loads every session-store entry, filters by key shape/agent/include flags, and maps each remaining entry into a `GatewaySessionRow` without requiring a real `sessionId` or finite `updatedAt`. The related PR #57419 directly targeted this issue, but it was closed unmerged, and I found no equivalent filter or regression test on main.

## Best Possible Solution

Keep the issue open and land a narrow gateway fix: filter `sessions.list` rows to real session entries before `buildGatewaySessionRow`, at minimum excluding `agent:<id>:sessions` store-level placeholders and entries without a usable `sessionId`/activity timestamp. Add a focused regression test around `listSessionsFromStore` with the reported `agent:sean:sessions` fixture, and consider a doctor/fix cleanup if persisted placeholder keys can remain from older versions.
