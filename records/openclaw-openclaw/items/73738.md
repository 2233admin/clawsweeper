## Summary

Keep this PR open. Current main still lacks the requested sandbox workspace lifecycle, and this PR remains the active implementation candidate for #14804, but the proposed patch is not ready because the default Docker runtime path still reuses stable per-scope container names instead of the new run-scoped lifecycle key.

## What This Changes

The PR adds a `workspaceLifecycle` sandbox config option, schema/docs metadata, run-scoped sandbox workspace resolution, embedded cleanup hooks, unregistered runtime cleanup fallbacks, and targeted tests for ephemeral sandbox workspaces.

## Best Possible Solution

Land the feature after the run-scoped lifecycle key is threaded through the default Docker and browser runtime naming paths, cleanup remains idempotent and path-safe, docs/schema/config stay aligned, and targeted sandbox/config tests plus the appropriate changed gate or Testbox proof pass.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Thread ephemeral scope keys into Docker runtimes:** `src/agents/sandbox/context.ts:293-299`
  - body: When `workspaceLifecycle` is ephemeral, this code derives a run-scoped `scopeKey` and per-run `workspaceDir`, but the default Docker backend still recomputes the old stable scope from `sessionKey`. Consecutive ephemeral runs can therefore reuse the same hot container and its previous workspace mount, leaking files across runs. Thread `params.scopeKey` through the Docker backend/container naming path and add Docker-backed coverage.
  - confidence: 0.91
- **[P2] Use run-scoped names for ephemeral browser containers:** `src/agents/sandbox/browser.ts:467`
  - body: The extracted browser-name helper still collapses every `scope: "shared"` browser to the same `shared` container name. With an ephemeral run-scoped `scopeKey`, that makes shared-scope browser cleanup/reuse cross run boundaries. Honor the provided run-scoped key when lifecycle is ephemeral and cover `scope: "shared"` with browser enabled.
  - confidence: 0.82

## Security Review

Status: needs_attention

Summary: The patch improves cleanup intent but currently leaves a concrete sandbox isolation gap in the default Docker/runtime naming path.

Concerns:

- **[high] Ephemeral Docker runs can reuse old workspace mounts:** `src/agents/sandbox/context.ts:293`
  - body: The new lifecycle relies on a run-scoped `scopeKey`, but the default Docker backend ignores that key and derives the container name from the stable session/agent/shared scope. That can make an isolated run execute in a container still mounted to a previous run workspace, undermining the requested disposable workspace security property.
  - confidence: 0.9
- **[medium] Shared browser runtime names are not disposable per run:** `src/agents/sandbox/browser.ts:467`
  - body: For shared scope, browser container naming still collapses to `shared` even when the sandbox context has a run-scoped ephemeral key, so browser state and cleanup can cross isolated runs.
  - confidence: 0.8
