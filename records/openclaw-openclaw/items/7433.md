## Summary

Keep this issue open. Current main still lacks the requested Baileys `getMessage` retry callback, socket-level `cachedGroupMetadata`, proactive group metadata/participant event maintenance, and `lid-mapping.update` persistence. The existing WhatsApp plugin is the right owner, and open PR #73732 is related implementation work for the group inbound subset, but it is not merged and does not complete the broader retry/cache request.

## Best Possible Solution

Keep this issue open. The best path is to finish the WhatsApp plugin reliability work: review or supersede #73732 for the group inbound subset, add a bounded account-scoped message retry store wired through Baileys `getMessage`, expose a shared group metadata cache through `cachedGroupMetadata`, seed and update it from connect hydration plus group/participant events, and decide whether current `signalRepository.lidMapping` plus auth-dir reverse files are enough or whether `lid-mapping.update` needs explicit persistence. Add focused WhatsApp plugin tests before closing.
