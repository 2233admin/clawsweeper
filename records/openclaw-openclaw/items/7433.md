## Repair Work Prompt

Fix openclaw/openclaw#7433 as a focused WhatsApp plugin repair. Current main has only the partial reconnect-time group metadata fallback from #73732; it still lacks Baileys `getMessage`, socket-level `cachedGroupMetadata`, proactive `groups.upsert`/`groups.update`/`group-participants.update` handling, and `lid-mapping.update` handling. Keep the work inside `extensions/whatsapp` unless a tiny generic seam is truly unavoidable. Add typed options to the socket/session path so Baileys receives `getMessage` and `cachedGroupMetadata` at `makeWASocket` construction time, and thread the existing group metadata cache through the connection/inbound boundaries without moving WhatsApp-specific behavior into core. Update group metadata from Baileys group and participant events, and implement `lid-mapping.update` using the actual Baileys v7.0.0-rc.9 payload shape `{ lid, pn }`, not the object-entry shape shown in the issue example. Add regression tests for the socket options and the group/LID event handlers, preserving existing reconnect fallback behavior. Changelog: add a user-facing WhatsApp/Web fix entry with reporter credit to @octopuslabs-fl if the patch changes runtime behavior.
