## Summary

Keep this issue open. Current main still lacks the requested WhatsApp/Baileys reliability hooks: no socket-level `getMessage`, no `cachedGroupMetadata`, no proactive group metadata or participant event maintenance, and no `lid-mapping.update` persistence path. The related reconnect/listener work is adjacent but has not landed this broader retry/group-cache implementation.

## Best Possible Solution

Keep this issue open. Implement the missing reliability hooks in the bundled `@openclaw/whatsapp` plugin: add a bounded/account-scoped message retry store and wire it through Baileys `getMessage`; expose a shared group metadata cache through `cachedGroupMetadata`; seed and update it from connect hydration plus group/participant events; and decide whether current `signalRepository.lidMapping` plus auth-dir reverse files are sufficient or whether `lid-mapping.update` needs persistence. Review or supersede open PR #73732 for the group inbound subset, then add focused WhatsApp plugin tests before closing.
