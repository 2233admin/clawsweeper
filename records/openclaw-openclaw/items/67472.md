## Summary

Keep open. Current main still uses the existing `tar.c` archive path and has no archive-time ENOENT/ENOTDIR skip behavior, while this PR remains tied to open bug #67417 and overlaps with open draft PR #72251. The PR is useful, but it needs maintainer review or revision around `tar.Pack` error handling, manifest/verify semantics, and test fidelity before it should merge or be superseded.

## Best Possible Solution

Keep this PR open for maintainer review or revision, or replace it with a superseding fix that closes #67417. The best implementation should add archive-time ENOENT/ENOTDIR resilience in `src/infra/backup-create.ts`, keep manifest/result/verify semantics consistent for omitted payloads, add a focused regression test for a session transcript disappearing during packing, and coordinate explicitly with #72251's volatile-file filtering work.

## Risks / Open Questions

- Current main still appears vulnerable to the reported archive-time deletion race because it passes planned source paths directly to `tar.c`.
- The PR's proposed `Pack` error listener may not be sufficient: node-tar `Pack` emits filesystem errors but does not appear to remove the failed job or close the stream when a listener merely returns, so the skip path needs a real or faithful regression test.
- The manifest is written before archive-time skipped paths are known, so the archive summary, embedded manifest, and `backup verify` semantics can diverge unless the implementation explicitly handles omitted payloads.
- The overlapping #72251 work addresses related live-backup races and should be coordinated so one fix does not obscure EOF, volatile-file, or ENOENT behavior from the other.
- Backup archives contain local state and credentials; any graceful skip behavior should be explicit in CLI/JSON output and should not silently publish an unverifiable or misleading archive.
