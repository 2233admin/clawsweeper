## Summary

Keep open. Current main still uses the existing `tar.c` backup archive path and does not implement archive-time ENOENT/ENOTDIR skipping, while this PR is tied to open bug #67417 and overlaps with open PR #72251. The branch is useful, but needs maintainer review or revision around node-tar Pack lifecycle semantics, manifest/verify consistency, and regression-test fidelity before merge or supersession.

## What This Changes

This PR replaces the backup archive creation path in `src/infra/backup-create.ts` with a `tar.Pack`-based helper that skips not-found paths during packing and adds a `Pack` mock to backup test support.

## Best Possible Solution

Keep this PR open for maintainer review or revision, or replace it with a superseding fix that closes #67417. The best implementation should add archive-time ENOENT/ENOTDIR resilience in `src/infra/backup-create.ts`, keep manifest/result/verify semantics consistent for omitted payloads, add a focused regression test for a session transcript disappearing during packing, and coordinate explicitly with #72251's volatile-file filtering work.

## Risks / Open Questions

- The proposed `tar.Pack` error listener may not be enough to continue archiving: tar 7.5.13 emits errors from queued stat/open/read work without marking the job ignored or done, so swallowing ENOENT needs real proof that packing completes rather than hanging or leaving a corrupt temp archive.
- The branch records archive-time skipped paths after writing `manifest.json`; current verify semantics require manifest-declared payloads to exist, so omitted payloads can create inconsistent backup artifacts.
- The change overlaps with #72251's volatile-file filter/retry work; merging either PR without coordination could duplicate or conflict in the backup archive path.
- Security review found no unexplained workflow, dependency, lockfile, package script, secret, permission, or downloaded-artifact changes. The remaining risk is local backup correctness and consistency.
