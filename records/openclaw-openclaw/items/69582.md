## Summary

Keep open. Current main has optional loop detection and some edit/read parameter guards, but it does not yet provide a default fix for the reported GLM `{}` tool-argument contamination: `process(action={})` can still fall into the same failed-tool-result path, and loop detection is disabled unless configured.

## What This Changes

Request to prevent GLM/local tool calls from injecting or appending `{}` into tool parameters and causing repeated invalid process/edit invocations.

## Best Possible Solution

Add schema-aware runtime validation at the tool invocation boundary and a narrow `process` action validation path that returns clear, non-misleading retry guidance for wrong parameter types. If a GLM/Z.AI streamed-arguments fixture proves a provider-specific parse artifact, repair only that artifact rather than globally trimming `{}` from string values.

## Reproduction Assessment

Unclear for the end-to-end GLM injection because the issue has no raw model/tool transcript. Source inspection does give a high-confidence focused check for the reported `process(action={})` symptom: repeated calls with `{ action: {} }` under default config return the same failed process result while loop detection is off.

## Solution Assessment

No, the proposed blanket parameter cleaning is not the safest fix because `{}` can be valid user content. A narrower schema-aware validation/repair path, plus targeted tests for `process(action={})` and trailing-brace edit parameters, is the maintainable direction.
