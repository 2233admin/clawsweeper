## Summary

Keep open. Current main has generic loop-detection infrastructure and some tool-call hardening, but the reported malformed-parameter path is not clearly fixed: the process tool still casts raw args without a runtime type/enum guard for `action`, edit parameter validation accepts non-empty strings such as `path.md{}` or `value{}`, and loop detection is disabled by default. The report is plausible, but the safe fix needs maintainer direction because broad trailing-`{}` cleanup could corrupt legitimate user input.

## What This Changes

Request schema-aware handling for local GLM tool parameters that arrive as trailing `{} ` string contamination or empty-object fields, so invalid tool calls fail clearly instead of looping.

## Best Possible Solution

Keep this open for maintainer triage and implement schema-aware runtime validation for tool parameters, at minimum making malformed `process.action` values fail with a clear terminal error. If repeated invalid registered-tool calls should be interrupted by default, add a narrow guard for identical validation failures rather than silently rewriting arbitrary strings.

## Reproduction Assessment

No. Source inspection gives a focused code-path check for malformed `process` arguments reaching normal tool logic, but there is no high-confidence end-to-end reproduction of the local/glm-5 parameter injection loop because the report does not include a transcript, provider config, or runnable steps.

## Solution Assessment

Unclear. The proposed framework-level cleanup is risky because `{}` may be legitimate string content; the safer direction is explicit schema-aware validation and, if needed, a repeated-invalid-tool-call breaker with tests.
