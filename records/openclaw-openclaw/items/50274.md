## Summary

Keep this issue open. Current main has generic model fallback chaining and an OpenRouter model scan command, but it does not complete the requested follow-up: no latency/quality/cost fallback-order policy or benchmark report exists for the named active models, and current config validation still accepts `models.providers.*.models[*].input` values such as `audio` without a model-specific compatibility guard. The issue is concrete and only recently became stale, so stale/insufficient-info closure is not appropriate.

## Best Possible Solution

Keep this open for maintainer triage. The best path is to split the work into a policy-backed benchmark/update for the active fallback chain, plus a narrowly designed validation guardrail that catches impossible `models.providers.*.models[*].input` declarations without breaking legitimate audio/video-capable model catalogs. Related fallback diagnostics work such as #6599 can inform validation, but it does not supersede this issue.
