## Summary

Keep open. Current main still uses configured fallback list order at runtime and only offers a narrower OpenRouter scan helper, while config validation accepts model input enum values without checking those modalities against actual model capabilities. The remaining work needs benchmark-backed fallback policy and a trusted capability-source design, so stale closure would drop real follow-up work.

## What This Changes

The issue requests benchmark-ranked fallback ordering for the named active model chain and a config guardrail that fails fast on unsupported `models.providers.*.models[].input` modalities.

## Best Possible Solution

Define a documented fallback policy artifact for the active chain using repeatable benchmark criteria, then add capability-aware validation for configured model inputs using trusted provider or catalog metadata while preserving valid multimodal custom-provider catalogs.
