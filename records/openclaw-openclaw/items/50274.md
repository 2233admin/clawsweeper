## Summary

Keep open. Current main still uses the configured fallback order and has only enum-name validation for model input modalities; it does not yet provide benchmark-backed latency/quality/cost fallback policy for the named chain or a capability-aware guardrail for unsupported `models.providers.*.models[].input` entries.

## What This Changes

The issue requests benchmark-backed fallback ordering plus a capability-aware guardrail for `models.providers.*.models[].input` values.

## Best Possible Solution

Define a repeatable benchmark artifact for the active fallback chain, document the selected fallback-ordering policy in model failover/model CLI docs, then add capability-aware validation for configured model inputs using trusted provider or catalog metadata while preserving valid custom multimodal catalogs.
