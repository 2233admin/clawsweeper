## Summary

Keep this issue open. Current main still has the reported block-streaming paragraph-boundary gap: the chunker emits text before paragraph/newline separators and advances past them, text_end can flush that chunk as a separate outbound delivery, and the shared block reply payload still carries only plain text/media/reply fields with no boundary metadata for clients reconstructing one visible reply from multiple deliveries.

## Best Possible Solution

Keep this issue open and fix the shared block-streaming pipeline. Preserve boundary semantics from EmbeddedBlockChunker through block reply coalescing and delivery, either by attaching consumed separators to emitted chunks while normalizing only dedupe comparisons, or by carrying explicit boundary metadata such as paragraph/newline/hard split. Add a regression test that streams Markdown like "# Title\n\nFirst paragraph." across at least two flushed text_end block deliveries and asserts reconstructed visible text remains structurally equivalent.

## Risks / Open Questions

- A fix needs to preserve separators across flush boundaries without double-inserting joiners when chunks are coalesced into the same outbound payload.
- Final-payload suppression currently normalizes whitespace across streamed fragments, so regression coverage should include duplicate suppression as well as visible reconstruction.
