## Summary

Keep this issue open. Current main still has the reported block-streaming paragraph-boundary gap: the chunker emits text before a paragraph separator and advances past the separator, text_end streaming can flush that chunk as its own outbound delivery, and the downstream block reply payload has no boundary metadata for clients reconstructing one visible reply from multiple deliveries.

## Best Possible Solution

Keep this open and fix the shared block-streaming pipeline. Preserve boundary semantics from EmbeddedBlockChunker through delivery, either by attaching consumed separators to emitted chunks while normalizing only dedupe comparisons, or by carrying explicit boundary metadata such as paragraph/newline/hard split. Add a regression test that streams Markdown like "# Title\n\nFirst paragraph." across at least two flushed text_end block deliveries and asserts reconstructed visible text remains structurally equivalent.

## Risks / Open Questions

- A naive fix that appends separators to chunks could double-insert separators when chunks are coalesced into one outbound payload.
- Final-payload suppression currently compares streamed fragments after stripping whitespace, so regression coverage should assert visible reconstruction rather than only dedup behavior.
