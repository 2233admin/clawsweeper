## Summary

Keep this issue open. Current main still consumes paragraph/newline separators in the shared block-streaming chunker, existing tests still assert separator-stripping output, and downstream block-reply delivery only carries plain text after a flush boundary, so edit-in-place clients cannot reliably reconstruct Markdown paragraph breaks across separate streamed deliveries.

## Best Possible Solution

Fix the shared block-streaming pipeline so paragraph/newline/hard-split boundary semantics survive from EmbeddedBlockChunker through block-reply delivery. The narrowest path is to retain or carry the consumed separator while normalizing dedupe/coalescing, then add a regression that reconstructs visible text from multiple flushed block deliveries and preserves Markdown such as # Title\n\nFirst paragraph.

## Clownfish Work Prompt

Fix the block-streaming paragraph-boundary loss tracked by https://github.com/openclaw/openclaw/issues/42106. Current main consumes paragraph/newline separators in `EmbeddedBlockChunker` before `onBlockReply` receives plain text, so clients that reconstruct a single visible reply by concatenating separate streamed deliveries can lose Markdown structure. Keep the fix in the shared block-streaming pipeline; do not change channel defaults, introduce token-delta channel streaming, or add bundled channel-specific workarounds. Prefer retaining/carrying boundary semantics from `src/agents/pi-embedded-block-chunker.ts` through block reply delivery while normalizing dedupe/coalescing so final-payload suppression still works. Add regression coverage that reconstructs visible text from multiple flushed block deliveries and preserves `# Title\n\nFirst paragraph.` across flush boundaries. Update existing tests that currently assert separator stripping. Add a single-line user-facing changelog fix entry crediting `Thanks @wbxl2000`.
