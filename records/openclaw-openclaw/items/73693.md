## Summary

Keep open. Current main still has the ACP timeout/config-control failure this PR targets, and the branch is a focused implementation candidate, but it needs maintainer review before merge because the diff introduces raw config-value logging and a changelog version claim that does not match the package files or upstream acpx state.

## What This Changes

This PR stops mirroring ACP timeoutSeconds into backend config, routes timeout slash commands through local runtime options, adds JSON-RPC error-detail extraction, fail-soft skips clearly unsupported ACP controls, forwards acpx handle-aware capabilities, and updates ACP tests/docs/changelog.

## Best Possible Solution

Land this PR or a cleaned-up successor after redacting rejected config values from verbose logs, correcting the acpx 0.6.2 release-note wording, running the targeted ACP/acpx tests and type/lint/build checks, and completing a real ACP smoke against claude-agent-acp v0.31+. Keep the acpx 0.6.2 dependency bump as a separate follow-up unless the upstream release is available before merge.

## Reproduction Assessment

Yes. Current main can be checked without live credentials by following the timeoutSeconds path: runtime options emit a timeout config pair, applyManagerRuntimeControls sends it through setConfigOption, and /acp timeout also writes key "timeout" directly; related issue #74225 provides the live sessions_spawn failure trace against claude-agent-acp v0.31+.

## Solution Assessment

No, not as-is. The main product direction is the right narrow fix, but the branch should redact unsupported-control log values and fix the changelog/dependency wording before merge; the acpx 0.6.2 advertised-key path should remain coordinated with openclaw/acpx#282.
