## Summary

Keep this issue open. Current main has partial TUI status handling and a streaming watchdog, but it still renders coarse internal states (`idle`, `waiting`, `streaming`, `running`) rather than the requested operator-facing `Idle`, `Generating`, `Working`, `Waiting`, `Reconnecting`, and `Disconnected` model. The specific `sessions_yield` case remains especially relevant: current runtime behavior deliberately leaves the parent session idle after yield, so the TUI still lacks a durable way to show that the parent is passively waiting on a subagent.

## Best Possible Solution

Keep the issue open. The best fix is to add a small explicit TUI status-state model: map token deltas to `Generating`, lifecycle/tool work to `Working`, connected no-run state to `Idle`, reconnect/backoff to `Reconnecting`, and unreachable/no active session to `Disconnected`. For `sessions_yield`, propagate durable paused/yielded plus child-run activity metadata to the TUI so the parent can show `Waiting` until the subagent completion arrives. Cover it with focused tests in `src/tui/tui-event-handlers.test.ts` and sessions-yield/subagent status tests, then update `docs/web/tui.md`.
