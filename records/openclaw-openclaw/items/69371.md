## Summary

Keep open. Current main has shipped adjacent ARM64/startup, cron retry, and outbound delivery resilience work, but this issue is a recent concrete ARM64 stability umbrella and several central requested capabilities are still missing or only partially covered.

## Best Possible Solution

Keep the issue open and split or scope it into concrete core tasks: add a bounded/lightweight sessions/status inspection path for large ARM64 session stores, decide on configurable context high-water compaction triggers, reuse the existing delivery queue for explicit `message send --background/--retry` behavior if desired, and clarify whether `sessions_send` should expose a final receipt beyond accepted/ok/pending metadata. Treat already-shipped handshake, startup, cron retry, and Pi docs/doctor hints as partial evidence, not closure.
