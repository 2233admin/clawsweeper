## Summary

Keep open. Current main still tracks model fallback state, lifecycle events, and verbose/status visibility, but it does not proactively notify the active user channel during provider retry/failover for non-verbose users and the requested notification config is absent.

## Best Possible Solution

Keep this issue open and implement the feature in the core reply-runner/model-fallback path: add backwards-compatible notification config, use existing fallback signals to send sanitized delayed retry and successful fallback notices through active-channel delivery, and add tests proving non-verbose users receive notices only when enabled without duplicating current verbose/status observability.
