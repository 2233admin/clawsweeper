## Summary

Keep open. Current main still has the hardcoded finite Slack Socket Mode reconnect cap, the latest release tag has the same cap, and the issue is paired with open implementation PR #73162 plus maintainer discussion about an upstream Slack SDK reconnect fix path.

## What This Changes

The issue reports that a previously working Slack connection silently stops responding while OpenClaw remains usable through the web interface.

## Best Possible Solution

Slack Socket Mode should retry indefinitely for recoverable disconnects, keep channel health state accurate while reconnecting or failed, preserve fast failure for non-recoverable auth errors, and then reassess transient reconnect-window delivery loss after the upstream SDK update path is validated.

## Reproduction Assessment

Yes. The discussion provides current-main logs and reproduction notes, and the current source has a deterministic path where repeated disconnects or start failures increment `reconnectAttempts` until the hardcoded cap throws.

## Solution Assessment

Unclear whether the open one-line uncap is the complete fix. It is the narrowest fix for permanent reconnect exhaustion, but maintainers should also account for the upstream Slack SDK pong-stall fix and the transient slash/message loss reported during reconnect windows.
