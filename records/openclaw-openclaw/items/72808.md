## Summary

Keep open. Current main still has a hardcoded finite Slack Socket Mode reconnect cap and still throws after the cap is reached; the config/docs only expose ping/pong timeout tuning, not reconnect attempts. The discussion also includes current-main reproduction, live logs, an upstream Slack SDK pong/reconnect path, and an open unmerged PR with closing syntax, so this should stay open until that PR or a broader Slack transport fix lands.

## What This Changes

Keep Slack Socket Mode responsive after long-lived disconnect or pong-timeout instability instead of silently dropping Slack replies while other OpenClaw surfaces continue working.

## Best Possible Solution

Land a Slack-owned transport fix that removes or configures the finite Socket Mode reconnect cap while preserving fail-fast behavior for non-recoverable auth errors, add regression coverage for repeated disconnect/start-failure and status behavior, and update the Slack SDK once the upstream pong/reconnect fix is available and validated.

## Reproduction Assessment

Yes. The issue discussion provides current-main reproduction and live logs, and the code has a deterministic path: repeated Slack Socket Mode disconnect or start-failure events eventually hit the hardcoded `maxAttempts: 12` guard.

## Solution Assessment

Unclear. Removing the cap is a narrow and likely useful fix for permanent reconnect exhaustion, but the best complete solution should also account for upstream Slack SDK pong stalls and transient delivery failures during reconnect windows.
