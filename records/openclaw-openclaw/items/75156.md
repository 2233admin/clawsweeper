## Summary

Keep this open. Current main still has the reported mixed text-plus-media block-stream dedupe path, and open PR #75180 references this issue with closing syntax, so the issue should stay open until that PR is reviewed and merged or rejected.

## What This Changes

The issue reports that a Telegram direct-chat assistant reply containing visible text plus a MEDIA attachment can deliver the text while dropping the attachment on the first send.

## Best Possible Solution

Review PR #75180 as the active implementation candidate and land its narrow media-only sent-media accounting fix with regression coverage, or close it with an equivalent replacement that makes the block/final reply media boundary explicit.

## Reproduction Assessment

Yes. The issue has live Telegram DM steps, and current source gives a high-confidence unit reproduction path: a mixed text-plus-media block payload can populate getSentMediaUrls(), after which final payload construction can strip the same media while preserving text.

## Solution Assessment

Yes. The best fix belongs in auto-reply block sent-media accounting with focused regression tests; changing Telegram delivery would miss the shared block/final payload boundary and Telegram already has captioned media delivery support.
