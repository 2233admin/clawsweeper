## Summary

Keep open. Current main covers the array `input`, `stream: true`, and prompt-cache stripping parts of the report, but it still lacks stable opt-in controls for generic `openai-responses` fallback `instructions` and for omitting `max_output_tokens` on stricter Responses-compatible relays.

## What This Changes

The issue asks for a stable `openai-responses` compatibility layer for third-party Responses-compatible relays, including non-empty instructions fallback, forced array input/streaming, and optional omission of unsupported fields like `max_output_tokens`.

## Best Possible Solution

Add explicit per-model Responses proxy compatibility fields across config types/schema, catalog normalization, payload policy, transport payload construction, focused tests, generated config docs, provider/local-model docs, and changelog, while preserving current native OpenAI/Codex/Azure defaults.

## Reproduction Assessment

No, not for the relay-specific HTTP 400 because it depends on a stricter third-party Responses-compatible endpoint. Static inspection does give high confidence that current main cannot configure the remaining fallback `instructions` or omit-`max_output_tokens` behavior.

## Solution Assessment

Yes, an opt-in compatibility layer is the narrowest maintainable direction. Applying the changes globally to all `openai-responses` requests would risk native OpenAI/Codex/Azure behavior, so this should be model-config and policy driven.
