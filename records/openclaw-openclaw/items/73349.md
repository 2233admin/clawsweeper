## Summary

Keep this PR open, but do not merge it as-is. The underlying feature is still valid and current main does not expose these trusted identifiers in exec child environments, but the PR's node-host path puts them into normal requested env overrides that shell-wrapper sanitization drops before the child process starts.

## What This Changes

The PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec runtime environments, forwards them through the gateway/node exec paths, and updates docs plus regression tests.

## Best Possible Solution

Revise the PR so `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` are injected as trusted runtime metadata after node-host shell-wrapper env filtering in both the TypeScript node host and macOS runtime, while preserving local gateway/sandbox/PTY behavior, always-set empty-string docs, and spoof-prevention semantics.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: a `host=node` exec command is shell-wrapped, forwarded as `system.run`, then `sanitizeSystemRunEnvOverrides({ shellWrapper: true })` drops non-allowlisted env overrides before the final process spawn.

## Solution Assessment

No. The PR's local exec injection is on the right path, but the node-host solution is not sufficient because it treats trusted identifiers as ordinary requested env overrides instead of carrying them past the shell-wrapper sanitizer.

## Review Findings

Overall correctness: patch is incorrect
