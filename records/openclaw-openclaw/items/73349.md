## Summary

Keep this PR open. It targets a real current-main gap and is the active replacement for the earlier unmerged session-key PR, but the proposed node-host implementation still does not deliver the new variables to the final child process because shell-wrapper env sanitization drops the added `OPENCLAW_*` keys.

## What This Changes

The PR injects `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` into exec child-process environments across local exec paths and node-host forwarding, with docs and regression tests.

## Best Possible Solution

Revise the PR so `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` are injected as trusted runtime metadata after node-host shell-wrapper env filtering in both the TypeScript node host and macOS runtime, then keep the always-set empty-string docs and add tests proving the final child process sees the values while user-provided env cannot spoof them.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve trusted env vars through node-host sanitization:** `src/agents/bash-tools.exec.ts:1688-1692`
  - body: This adds `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` to `requestedEnv` for `host=node`, but node exec commands are shell-wrapped and both node-host implementations filter shell-wrapper env overrides before spawning. These keys are not in the allowed override set, so node-host children still cannot read them despite the docs and tests claiming node coverage. Inject them after sanitizer handling or explicitly carry them as trusted runtime env in both node-host paths.
  - confidence: 0.92
