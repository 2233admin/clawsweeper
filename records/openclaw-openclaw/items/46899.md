## Summary

Current main still leaves the reported cron delivery mismatch unresolved for object-shaped persisted `delivery.channel` values. String-shaped Telegram delivery is supported, but `{ id: "telegram" }` is still rejected by the string-only parser/normalizer, can be stripped during store load, and can then fall back to `last`, which is the same class of Feishu/Telegram misrouting described in the issue. The closed #46907 PR is a useful prior fix attempt but was not merged.

## Best Possible Solution

Keep this issue open and land a narrow cron delivery compatibility fix. Add one shared extractor for channel ids that accepts strings and `{ id: string }`, use it in `resolveCronDeliveryPlan()` and persisted delivery parsing/normalization, canonicalize the result back to the existing lowercase string form, and add focused tests for direct resolver behavior plus persisted store-load normalization. Preserve existing configured-channel validation and do not broaden the public `CronDelivery.channel` type beyond the canonical string shape.

## Risks / Open Questions

- Object-shaped persisted `delivery.channel` values can still degrade to `last`, allowing a Telegram target id to be resolved through a previous Feishu route.
- Persisted store-load normalization can remove the explicit channel before execution, so fixing only the runtime resolver would leave reload/restart behavior vulnerable.
- The fix should stay scoped to known channel-id object compatibility such as `{ id: string }`; accepting arbitrary objects broadly could weaken provider validation.
- There is no current regression coverage for object-shaped `delivery.channel` in resolver behavior or persisted store-load normalization.
