## Summary

Keep open. Current main still has the reported data-loss path: state/config uninstall recursively removes the state directory, while the default workspace normally lives inside that same `.openclaw` tree. Two open linked PRs already target this issue, so maintainer cleanup should review and consolidate those rather than close the issue or dispatch another repair lane.

## What This Changes

The issue reports that deleting state/config during uninstall can remove workspaces stored under `.openclaw` even when workspace deletion is not selected.

## Best Possible Solution

Make `openclaw uninstall` honor state and workspace deletion choices independently: when state deletion would cover a configured workspace and workspace deletion is not selected, preserve that workspace with selective cleanup or stop with a clear warning. Review and land one canonical fix from #75061 or #75062 with regression coverage and docs for nested workspaces under the state directory.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but current-main source gives a high-confidence reproduction path: default workspaces are under `.openclaw`, and state-only uninstall recursively removes that parent directory.

## Solution Assessment

Yes. A narrow uninstall cleanup guard with regression coverage is the best direction; the remaining maintainer choice is which of the two linked PRs should become the canonical fix.
