## Summary

Keep open. Current main still has the reported data-loss path: state/config-only uninstall removes the whole state directory recursively, while the default workspace is normally nested under the same `.openclaw` tree. The existing ClawSweeper comment on this issue is consistent with the current source review, and no open PR in the current search results references this issue as a fix.

## What This Changes

The issue reports that `openclaw uninstall` can delete a workspace under `.openclaw` when state/config removal is selected but workspace removal is not selected.

## Best Possible Solution

Make `openclaw uninstall` honor state/config and workspace choices independently: when state deletion would cover a configured workspace and workspace deletion is not selected, preserve that workspace through selective state cleanup, or stop before deletion with a clear warning. Update dry-run output, CLI/install docs, and regression tests so the UI contract and filesystem behavior agree.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but the source path is high-confidence: the default workspace is under `.openclaw`, and state-only uninstall removes the parent state directory recursively.

## Solution Assessment

Yes. A narrow CLI cleanup fix is the best path; preserving configured nested workspaces during state-only uninstall is safer than changing workspace defaults or treating this as a broad migration problem, with an explicit abort/warning as a fallback if selective deletion cannot be made robust.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75052. Current main deletes `~/.openclaw/workspace` during state/config-only `openclaw uninstall` because `uninstallCommand` calls `removeStateAndLinkedPaths`, which recursively removes the whole state directory, before considering the independent workspace scope. Implement the smallest safe fix in the uninstall/cleanup helpers so state-only uninstall preserves configured workspace directories inside the state directory when `--workspace` is not selected. Prefer selective state cleanup that removes config/state files while excluding the configured workspace directories; if that cannot be made robust, abort before destructive deletion with a clear warning. Preserve `reset --scope full` and `uninstall --workspace`/`--all` semantics when workspace deletion is explicitly selected. Reuse or mirror the path-overlap/realpath safety approach from `src/commands/agents.commands.delete.ts` where appropriate. Update dry-run output, the preservation tip, `docs/cli/uninstall.md`, and `docs/install/uninstall.md` if behavior/docs change. Add regression coverage for a workspace nested under the state dir during state-only uninstall, plus a check that explicit workspace/full deletion still removes workspaces. Do not change unrelated installer, backup, or agents-delete behavior; changelog only if the repo expects a user-facing bugfix entry for this lane.
