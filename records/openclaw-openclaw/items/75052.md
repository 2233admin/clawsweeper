## Summary

Keep open. Current main still has the reported data-loss path: state/config cleanup recursively removes the state directory, while default workspaces are under that same `.openclaw` tree, and two open closing PRs already target the fix.

## What This Changes

The issue requests that uninstall preserve workspaces stored inside `.openclaw` when state/config deletion is selected but workspace deletion is not selected.

## Best Possible Solution

Land one canonical fix from #75061, #75062, or an equivalent maintainer patch that makes state/config cleanup preserve configured workspaces nested inside the state directory unless the workspace scope is selected, with focused regression coverage and uninstall docs for the nested-workspace case.

## Reproduction Assessment

Yes. A destructive live uninstall was not run in this read-only review, but the current source gives a high-confidence reproduction path: default workspaces live under `.openclaw`, and state-only uninstall recursively removes that parent directory before any workspace-specific branch runs.

## Solution Assessment

Yes. A narrow uninstall cleanup guard with regression tests is the best way to solve this; moving default workspaces or opening another repair branch would be less direct while #75061 and #75062 are already open.
