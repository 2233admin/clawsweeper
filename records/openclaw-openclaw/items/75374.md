## Summary

Current main still appears to have the reported macOS LaunchAgent service-env gap: durable `.env` values are converted into managed-key metadata and removed before the launchd env file is generated, while launchd does not re-read the state-dir `.env` the way systemd does. This should stay open as a narrow daemon/runtime bug.

## What This Changes

The issue reports that macOS LaunchAgent service-env regeneration records managed env key names but omits the actual exported values, leaving env-ref gateway auth tokens and other managed secrets missing at runtime.

## Best Possible Solution

Add a focused macOS launchd service-env fix so durable state-dir `.env` and config-managed values listed in `OPENCLAW_SERVICE_MANAGED_ENV_KEYS` are exported into the owner-only LaunchAgent env file, while keeping systemd behavior unchanged and preserving the no-plaintext-in-plist contract.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a state-dir `.env` key becomes durable env, `writeManagedServiceEnvKeysToEnvironment` removes the actual key while leaving metadata, and launchd writes only the remaining passed entries into the sourced env file.

## Solution Assessment

Yes. The narrowest maintainable solution is in the launchd service-env handoff/regeneration path, not gateway auth resolution; it should mirror systemd's durable env-file behavior or otherwise rehydrate managed durable values only into the owner-only launchd env file.
