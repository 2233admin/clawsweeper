## Summary

Keep open. Current main still lacks the PR's followup-before-dispatch-idle barrier, and the related #65761 thread is closed unmerged rather than a shipped replacement. The PR is also not merge-ready as-is: unresolved review comments call out turn-ordering, multiple-followup tracking, and abort/stop regressions in the proposed session-wide wait. Security pass found no workflow, dependency, script, lockfile, secret, or package-publishing changes in the PR file list.

## Best Possible Solution

Keep the PR open as a potentially useful but incomplete implementation candidate. The best path is to rework it into a turn-scoped dispatch barrier: bind the wait to the specific dispatcher/final-reply delivery that produced the queued followup, re-track followup outbound sends until they drain, make `/stop` cancel followups waiting before embedded run startup, and add regression tests for newer-turn isolation, multiple followups, and abort during the drain window. If a later canonical PR lands that complete fix, close this PR as superseded after that merge.

## Risks / Open Questions

- Merging the PR as-is could preserve or worsen transcript ordering because its session-wide idle wait can wait behind newer turns rather than the specific dispatcher/turn that produced the queued followup.
- The PR needs a design that re-tracks followup delivery until its own outbound sends drain; otherwise multiple queued followups can still race after the original dispatcher unregisters.
- The proposed pre-run wait needs abort/stop integration so a followup parked before `runEmbeddedPiAgent` starts cannot reply after a user stop request.
- No supply-chain or secret-handling regression was found from the changed file list, but the behavioral surface is high-risk async reply ordering.
