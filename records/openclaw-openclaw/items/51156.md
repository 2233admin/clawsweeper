## Summary

Keep this PR open. Current main still has the existing final-tag filtering machinery, but it does not implement the PR's intended user-facing `agents.defaults.enforceFinalTag` option or propagate that config value through interactive and cron agent runs. The PR remains a focused implementation candidate for the external-channel reasoning leakage reported in #25592, and the discussion includes useful independent validation from a local backport.

## Best Possible Solution

Review and land this PR, or an equivalent patch, that adds `agents.defaults.enforceFinalTag` to the exported config type and strict schema, documents it on the public agents configuration page, propagates it through interactive reply and cron runs, enables the system prompt tag hint from the flag, and keeps focused tests around provider-gated defaults plus explicit config opt-in. Keep the linked leakage issue open until the implementation merges.

## Risks / Open Questions

- This is security-adjacent delivery behavior: external messaging channels can expose unintended model text, so review should require regression coverage for both interactive replies and cron delivery on non-tag providers such as Anthropic/OpenAI.
- The opt-in can suppress replies from models that fail to follow the `<final>` contract, so docs and tests should make the default-off behavior and failure mode explicit.
