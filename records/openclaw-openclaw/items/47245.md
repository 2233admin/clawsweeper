## Summary

Keep this PR open. Current main still has the shared command palette hover/active background, so the PR targets a real remaining UI issue, but the proposed CSS lets the hover selector override the active selector when a row is both hovered and keyboard-active.

## What This Changes

The PR changes `ui/src/styles/components.css` to replace the shared command palette hover/active background rule with separate theme-aware `color-mix` backgrounds.

## Best Possible Solution

Land a one-file CSS update that keeps hover and keyboard-active command palette rows visually distinct across dark and light themes, with an explicit active-hover rule or matched specificity so selected-state styling remains intentional without changing command palette logic.

## Reproduction Assessment

Yes. Current main reproduces the original styling issue from source because hover and active share one background rule; the proposed patch's remaining bug reproduces when a command palette item is both `:hover` and `.cmd-palette__item--active`.

## Solution Assessment

No, not as written. The narrow CSS-only approach is the right boundary, but the selectors need an active-hover rule or matched specificity so hover does not accidentally hide the selected-state styling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Make active styling win over hover:** `ui/src/styles/components.css:3536-3549`
  - body: The added hover selectors are more specific than the active selectors, so a row with both `:hover` and `.cmd-palette__item--active` renders the hover color instead of the selected color. That can make keyboard selection disappear in the overlap state during mixed pointer/keyboard navigation; add an active-hover rule or match specificity.
  - confidence: 0.87
