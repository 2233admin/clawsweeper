## Summary

Keep open. Current main exposes service/channel health and periodic tick events, but it does not implement the requested connection-level `health` fields (`connected`, `rttMs`, `lastHeartbeatAt`) or a separate `gateway.ping` RPC. The adjacent related reports in the provided context are channel-specific liveness/state issues and do not supersede this generic gateway transport-health request.

## Best Possible Solution

Keep the issue open and implement this as a small additive gateway protocol/API change: either add a `connection`/`transport` object to the `health` response or add a lightweight `gateway.ping` RPC. The implementation should compute caller-specific `connected`, recent RTT, and last transport heartbeat/tick time without conflating it with the existing agent `last-heartbeat` RPC, then update `HealthSummary` types, protocol docs, CLI/Control UI consumers, and focused gateway client/server tests.
