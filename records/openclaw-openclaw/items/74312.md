## Summary

Keep open. Current main still has the reported auth-epoch sensitivity for Claude CLI credentials that parse as token-shaped credentials, and the issue is now paired with an open, unmerged closing PR that should be reviewed or landed before the issue closes.

## What This Changes

The issue requests stabilizing Claude CLI session auth epochs so OAuth token rotation or transient access-token-only keychain reads do not reset active sessions mid-conversation.

## Best Possible Solution

Review and land PR #74493, or an equivalent narrow patch, so Claude token-shaped credentials and auth-profile token rotation use stable identity fields, real identity changes still invalidate sessions, the auth epoch version is bumped for migration safety, and focused tests cover the partial-read/token-rotation path.

## Reproduction Assessment

Yes. A deterministic source-level reproduction is to resolve an epoch for a Claude token credential, change only the token, and observe the current test expects the epoch to change; with an existing same-version binding, `resolveCliSessionReuse()` then returns `auth-epoch`.

## Solution Assessment

Yes, the linked PR direction appears to be the narrowest maintainable fix: remove rotating token material from the auth-epoch hash while preserving identity fields and bumping the auth epoch version. The issue should remain open until that PR is reviewed and merged or deliberately replaced.
