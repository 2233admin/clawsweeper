## Summary

Keep open. Current main still hashes Claude token-shaped credentials and token auth-profile material into the CLI auth epoch, so a token rotation or transient access-token-only Claude credential can still invalidate reusable sessions with `reason=auth-epoch`. The linked open PR #74493 is the right implementation path to review before this issue can close.

## What This Changes

The issue requests stable Claude CLI auth-epoch hashing that ignores rotating token material so OAuth token refreshes do not reset active reusable CLI sessions.

## Best Possible Solution

Review and land #74493, or an equivalent narrow change, so Claude token-shaped credentials and token auth-profile epochs are identity-based, the epoch version is bumped for migration, and regression tests cover token rotation plus real identity changes.

## Reproduction Assessment

Yes. Current main has a source-level reproduction path: a Claude CLI credential parsed as `type: "token"` changes the epoch when `credential.token` changes, and a same-version epoch mismatch resets reusable sessions with `reason=auth-epoch`.

## Solution Assessment

Yes. Identity-only hashing at the epoch layer is the narrowest maintainable fix for the observed token-rotation reset; parser hardening can remain a separate defense-in-depth follow-up.
