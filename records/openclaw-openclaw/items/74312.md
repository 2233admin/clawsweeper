## Summary

Keep open. Current main still has the reported auth-epoch sensitivity for Claude CLI credentials that parse as `type: "token"`: the Claude credential encoder hashes the access token, auth-profile token credentials also include token material in the epoch, and the Claude parser falls back to token mode when `refreshToken` is absent. The macOS keychain race itself was not live-reproduced here, but the current source and tests confirm the code path can invalidate a reusable CLI session with `reason=auth-epoch` when a token-shaped Claude credential changes.

## What This Changes

Prevent Claude CLI OAuth token rotation or partial keychain reads from changing the CLI auth epoch and resetting live sessions mid-conversation.

## Best Possible Solution

Keep this open and implement a narrow fix in the Claude CLI auth-epoch path: make token rotation or transient access-only keychain reads stable for OAuth-owned Claude CLI sessions while still invalidating sessions on real identity/account changes where identity evidence exists. Add targeted tests for partial Claude keychain credentials and auth-profile/token epoch behavior, and include a user-facing changelog fix credited to the reporter if behavior changes ship.
