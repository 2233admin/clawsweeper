## Summary

Keep open. Current main still uses the older `maskApiKey` behavior that exposes prefix and suffix characters, and the linked cleanup is not implemented or superseded by a merged PR. The PR is focused and has no supply-chain-sensitive files, but it needs revision before merge because its proposed `> 4` branch exposes more of 5-6 character secrets than the current behavior.

## Best Possible Solution

Keep the PR open, but revise it before merge. The best product/code direction is to make `maskApiKey` use a stricter credential display policy across all call sites, preferably no suffix disclosure and either constant masking or a very small prefix only with special handling for short values. Move the misplaced `" short "` assertion into the longer-key test block, add regression assertions for short values and suffix non-disclosure, then run the targeted util/auth overview tests before maintainer review.
