## Summary

Keep open: current main still has the suffix-revealing helper, but this PR is not ready because its catch-all first-four policy exposes more material for 5- and 6-character credentials than current main.

## What This Changes

The PR changes `maskApiKey` to reveal only the first character for keys up to 4 characters and the first four characters for longer keys, then updates unit tests to assert suffix nondisclosure.

## Best Possible Solution

Revise the shared `maskApiKey` policy so auth displays never expose suffixes and never reveal a larger fraction of short credentials than current main, then cover empty, <=4, 5-6, medium, long, whitespace-trimmed, and suffix-nondisclosure cases plus the affected auth overview/status surfaces.

## Reproduction Assessment

Yes. The source-level reproduction is direct: current main returns suffix-bearing values for long keys, while the PR diff would make `maskApiKey("abcde")` return `abcd...`, exposing four of five characters.

## Solution Assessment

No. The PR is directionally useful for long keys, but the catch-all first-four branch is not the safest fix because it increases short-secret exposure; a constant mask or separate stricter short-key branch is safer.
