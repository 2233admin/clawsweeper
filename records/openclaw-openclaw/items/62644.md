## Summary

Keep open. Current main still enforces the OC-07 `tree` default for `sessions_send`, still returns the terse cross-agent visibility error, and has no obvious doctor/update migration warning path for existing isolated cron workflows; docs and changelog mention the setting, but they do not fully address the reported breakage UX.

## What This Changes

The issue requests migration warnings, clearer errors, validation/doctor coverage, and migration docs for isolated cron jobs whose `sessions_send` cross-agent delivery broke after the `tools.sessions.visibility` default changed to `tree`.

## Best Possible Solution

Keep the security default, but add maintainer-approved upgrade UX around it: clearer `sessions_send` forbidden text with safe context, docs that call out isolated cron cross-agent delivery migration, and a doctor/update warning or audit that flags likely affected cron setups without auto-relaxing permissions.

## Reproduction Assessment

Yes. A source-level reproduction is high confidence: with default config, a requester like `agent:kk:cron:<job>` targeting `agent:main:main` is cross-agent, and the shared session visibility guard returns the exact forbidden send error unless `tools.sessions.visibility` is `all`.

## Solution Assessment

Unclear. Improving the warning/error/docs is the right direction, but automatic migration detection and the wording around `visibility=all` need maintainer security judgment; changing the default back would not be the safest fix.
