## Summary

Keep this PR open. Current main still lacks the PR's static bundled-channel fallback in `sandbox explain`, and the related Telegram report is not fully solved by this PR alone because documented bare `telegram:slash:<id>` input still bypasses agent-scoped parsing. The open related PR #51286 is complementary, not a complete supersession.

## Best Possible Solution

Keep this PR open and review it together with #51286 and the linked Telegram bug #51245. The best fix is a narrow `src/commands/sandbox-explain.ts` update that handles both static bundled-channel normalization and non-agent-scoped session-key normalization, applies the fallback to stored channel metadata as well as session-key inference, and adds regression tests for agent-scoped and documented bare Telegram slash sessions.

## Risks / Open Questions

- Landing this PR alone likely fixes agent-scoped keys like `agent:main:telegram:slash:<id>` when the CLI channel registry is empty, but not the documented bare `telegram:slash:<id>` form.
- `resolveActiveChannel` still normalizes stored `lastChannel`/`channel` metadata through `normalizeAnyChannelId` only, so stored `telegram` metadata can still fall through when the CLI plugin registry is empty.
- Because this affects elevated host exec availability, maintainers should require focused regression tests before merge.
