## Summary

Keep this issue open. Current main still exposes `session_status` as a rendered `/status`-style tool response with minimal structured details, while raw token/cost/session aggregates remain on separate gateway `sessions.usage` RPCs and UI usage types. Related #39400 is adjacent but narrower and does not supersede the broader agent-facing structured token usage API request.

## Best Possible Solution

Keep this open until OpenClaw adds a documented structured agent-facing usage surface, preferably under `session_status.details.usage` or an equivalent supported tool. The implementation should reuse existing transcript/session usage loaders, include session totals, last-turn usage where available, cost, provider quota/window availability/source metadata, and preserve current visibility and sandbox checks. Related #39400 can be handled as a narrower stability/availability facet of the same contract.

## Clownfish Work Prompt

Implement a documented structured agent-facing usage surface for the existing `session_status` tool. Scope the fix to adding stable raw usage data under `session_status` result details, preferably `details.usage`, while preserving the existing rendered status text. Reuse existing session/transcript usage loaders and gateway usage helpers rather than parsing status text. Include session totals, last-turn usage when available, cost fields when available, provider quota/window data with explicit `available`/`source`/`reason` metadata when unavailable, and keep all existing session visibility, sandbox, and cross-agent access checks before reading transcript or cost data. Do not broaden gateway method scopes, do not add owner-specific behavior, and do not expose usage for sessions the caller cannot already inspect. Update docs at `docs/concepts/session-tool.md` and `docs/concepts/usage-tracking.md`; add a user-facing changelog entry with `Thanks @saphid`. Cover the new contract and guard behavior with targeted tests.
