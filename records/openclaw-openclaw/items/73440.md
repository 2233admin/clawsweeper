## Summary

Keep open. Current main still has the large `ownerAllowFrom` hot path, and this PR is the active fix candidate for #50289, but the branch is not merge-ready because it is stale against a newer owner-prefix auth fix on main and still carries a no-op test reset helper.

## What This Changes

This PR adds a WeakMap-backed cache around owner allowlist normalization in command authorization and adds colocated memoization regression tests.

## Best Possible Solution

Land a current-main-compatible memoization fix for owner allowlist normalization, preserve the existing fail-closed provider-prefix semantics, remove or implement the test reset helper, and close #50289 through the linked PR once targeted command-auth tests and the changed gate are green.

## Reproduction Assessment

Yes. Current main still walks each `ownerAllowFrom` entry in `resolveOwnerAllowFromList`, and the authorization state builder calls that resolver for both config and context owner lists. A large `commands.ownerAllowFrom` config like the 9k-entry report in #50289 exercises the repeated O(n) path.

## Solution Assessment

Not yet. The memoization approach is a narrow maintainable fix for repeated normalization, but this branch should first incorporate current main's owner-prefix guard and clean up the misleading reset helper; Set-based membership or config-parse improvements can remain separate follow-up work.
