## Summary

Keep open. The PR targets a real current-main performance path and the prior empty-array cache miss appears fixed on the current head, but the memoization key is still too narrow for the channel plugin contract because `formatAllowFrom` receives `cfg`; caching only by raw array plus provider/account/plugin can reuse stale formatted owner allowlists across effective config changes. The exported reset helper is also a no-op and should not be left as a misleading test utility.

## What This Changes

The PR adds WeakMap memoization and colocated regression tests for `resolveOwnerAllowFromList` so large `commands.ownerAllowFrom` arrays are filtered once per raw array/key instead of on every command authorization pass.

## Best Possible Solution

Keep the memoization direction, but key or scope the cache by the effective config identity as well as raw allowlist/provider/account/plugin, then remove or implement the reset test helper. After that, validate the new memo tests alongside existing command-auth owner/default coverage and decide separately whether #50289 needs follow-up work for config parsing or membership lookup costs.

## Reproduction Assessment

Yes. The original performance path is reproducible by code inspection and the linked report's 9000-entry `commands.ownerAllowFrom` setup: current main walks the list twice per authorization pass. The PR regression can be reproduced with a channel plugin whose `formatAllowFrom` output depends on `cfg` while the same raw owner array reference is reused across effective config objects.

## Solution Assessment

No. WeakMap memoization per raw list is a reasonable narrow optimization, but it is not yet the best fix because the cache key omits `cfg` even though the plugin formatter contract includes it; the reset helper also needs to be real or removed.
