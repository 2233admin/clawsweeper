## Summary

Keep open. Current main still rejects mixed-case valid `--resources` values because `parse_resources()` strips tokens and validates the original spelling against lowercase `ALLOWED_RESOURCES`; the linked fix PRs are closed and unmerged, so the reported parser bug remains actionable.

## What This Changes

The issue asks for `skills/skill-creator/scripts/init_skill.py --resources` to accept valid resource names case-insensitively while preserving validation and order-preserving deduplication.

## Best Possible Solution

Normalize each non-empty resource token by stripping and lowercasing before validation and first-seen deduplication, return canonical lowercase resource names, and add focused `parse_resources()` regression tests for mixed-case valid input, case-insensitive duplicates, empty-token handling, and invalid-resource rejection.

## Reproduction Assessment

Yes. Importing `parse_resources()` with bytecode disabled reproduces current-main behavior: lowercase resource values succeed, while `Scripts,References,ASSETS` and `scripts,Scripts` exit with unknown-resource errors.

## Solution Assessment

Yes. A narrow parser normalization plus focused Python unittest coverage is the smallest maintainable fix; no core architecture, plugin API, or product decision is needed.
