## Summary

Keep open. Current main has generic empty-turn liveness handling, but the specific custom `anthropic-messages` post-tool path still does not produce a successful user-visible reply or sanitized success fallback when the final assistant turn is an empty nonzero-output `stop`.

## What This Changes

The issue requests that successful Anthropic-compatible tool-result runs produce a user-visible assistant reply or a safe fallback instead of ending with an empty payload and `No reply from agent`.

## Best Possible Solution

Add targeted embedded-runner recovery for successful `anthropic-messages` tool-result follow-ups: replay-safe empty final turns should get one visible-answer continuation, while replay-unsafe successful tool executions should return a sanitized user-visible completion acknowledgement without exposing raw tool output or retrying side effects.

## Reproduction Assessment

Yes for the remaining behavioral gap. A focused unit test can mock a successful tool result followed by a custom `anthropic-messages` assistant `stop` with `content: []` and nonzero usage; current main then reaches the generic error fallback rather than a successful reply or safe completion fallback.

## Solution Assessment

No. The current generic incomplete-turn fallback is useful liveness protection, but it is not the best solution because it treats a successful post-tool completion as an error and does not retry or synthesize a safe success response for the reported custom Anthropic-compatible path.
