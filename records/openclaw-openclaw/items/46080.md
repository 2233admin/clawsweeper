## Summary

The report still looks actionable on current main. Current code has generic empty-turn safety that should avoid a completely empty CLI payload in some cases, but I did not find a targeted fix for an `anthropic-messages` tool-result follow-up where the provider returns `stop` with `content: []` after a successful tool call. The existing fallback path surfaces an error/warning rather than preserving, retrying, or safely synthesizing a successful final assistant reply from the tool result.

## Best Possible Solution

Keep this issue open and add a targeted embedded-runner fix for `anthropic-messages` tool-result follow-ups: replay-safe successful tool calls that end in `stop` with empty assistant content should get one visible-answer continuation or another sanitized, user-visible success fallback; mutating/side-effectful tool runs should not be blindly replayed or dump raw tool output, but should still avoid a silent `No reply from agent` result. Add regression coverage for both read-like and mutating tool-result cases.
