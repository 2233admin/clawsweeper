## Summary

Keep this issue open. Current main still enables macOS node TLS trust-on-first-use for a fresh wss:// host:port, the TLS delegate accepts and stores the presented certificate before SecTrustEvaluateWithError, and the macOS remote config path still does not read gateway.remote.tlsFingerprint. The issue is also paired with open PR #50643, so it should remain open until that fix or a replacement lands.

## Best Possible Solution

Remove the stale label, review and land PR #50643 or a maintainer replacement, and make the macOS node wss:// path fail closed unless normal SecTrustEvaluateWithError succeeds or an explicit pinned fingerprint is configured/confirmed. The fix should also add Swift coverage for first-use wss:// behavior and for gateway.remote.tlsFingerprint being honored by the macOS direct remote path.
