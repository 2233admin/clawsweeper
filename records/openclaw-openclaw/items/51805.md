## Summary

Keep open. Current main still keeps direct-message identity sharing separate from group/channel session routing, while the issue discussion asks for a distinct opt-in model that merges only one linked user's group turns and DM turns without leaking other group participants. #19929 overlaps on shared group/channel buckets, but it does not cover this issue's per-identity DM/group privacy-filtering requirement.

## Best Possible Solution

Keep this open as a core session-routing design item. The best path is an additive, opt-in generic contract that lets an authorized sender's group/channel turns and direct turns resolve to a shared canonical session, filters out other group participants, preserves reply delivery to the originating chat, and updates config schema, docs, Telegram/Discord/Slack routing, and focused routing tests. Coordinate with #19929 so shared-session grouping uses one coherent contract instead of channel-specific special cases.

## Risks / Open Questions

- A naive shared key could leak unrelated group participants' messages into a private DM context, which the issue discussion explicitly calls out.
- Group/channel sessions are documented as non-main and may have different sandbox/tool posture from DMs; merging them changes a safety boundary.
- Reply delivery and last-route metadata must still send responses back to the originating group or DM even when several origins share one canonical session.
- The design overlaps with #19929, so implementing this separately could create two incompatible shared-session contracts.
