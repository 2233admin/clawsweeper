## Summary

Keep open. Current main still treats DM identity linking as direct-message-only and continues to route group/channel peers to isolated session keys; the related open shared-session trackers overlap, but they do not cover this issue's per-identity group-to-DM filtering and privacy requirements.

## What This Changes

Request an opt-in session routing mode where a linked user's DMs and group/chat messages can share one conversation context while excluding other group participants.

## Best Possible Solution

Keep this as a core session-routing design item. The desired end state is an additive, opt-in generic contract that maps authorized sender identities from group/channel and direct turns to a shared canonical session, filters out other participants' content, preserves reply delivery to the originating chat, and updates config schema, docs, and focused routing tests across the affected channel surfaces.

## Reproduction Assessment

Yes for the current behavior: source inspection and existing routing tests show DMs and group/channel peers resolve to distinct keys, and identityLinks only affects direct-message scopes. This is a feature gap rather than an intermittent runtime bug.

## Solution Assessment

Unclear. The proposed config examples are directionally useful, but the maintainable solution needs an explicit privacy-filtered identity merge model plus runtime-policy and reply-routing safeguards coordinated with the existing shared-session trackers.

## Security Review

Status: needs_attention

Summary: The requested feature is privacy-sensitive because merging group and DM context could expose group participant content in private sessions if designed too broadly.

Concerns:

## Risks / Open Questions

- A naive merge could leak unrelated group participants' messages into a private DM transcript, which the issue discussion explicitly calls out.
- Group/channel sessions are documented as non-main and may have different sandbox/tool posture from DMs, so sharing transcript context must not collapse runtime safety boundaries accidentally.
- Multiple origins sharing one canonical transcript need deterministic serialization and route provenance so replies still go back to the triggering group or DM.
- The design overlaps with #19929 and #7524, so maintainers should align on one additive contract rather than channel-specific special cases.
