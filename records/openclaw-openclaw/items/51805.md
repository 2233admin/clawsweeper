## Summary

Keep open. Current main still keeps DM identity sharing separate from group/channel session routing, and the issue discussion asks for a distinct privacy-filtered per-user group-to-DM merge model. The broader open #19929 overlaps on shared channel sessions, but it does not cover this issue's per-identity DM/group filtering requirement.

## Best Possible Solution

Keep this open as a core session-routing design item. The best path is an additive, opt-in generic model that lets an authorized sender's group/channel turns and direct turns resolve to a shared canonical session, filters out other group participants, preserves reply delivery to the originating chat, and updates config schema, docs, Telegram/Discord/Slack routing, and focused routing tests. Coordinate with #19929 so shared-session grouping uses one coherent contract instead of channel-specific special cases.

## Risks / Open Questions

- A naive shared session key could leak unrelated group participants' messages into a private DM context, which the issue discussion explicitly calls out.
- Group/channel sessions are intentionally non-main in current docs and can have different sandbox/tool posture from DMs, so merging them changes a safety boundary.
- Reply delivery and last-route behavior must still send responses back to the originating group or DM even when multiple origins share one canonical session.
- The design overlaps with #19929 but is not fully covered by it because this issue needs per-identity group-to-DM filtering.
