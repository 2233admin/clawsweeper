## Summary

Keep open with changes requested. The reported data-loss path exists on current main because session-store load failures can become `{}` and the dreaming scrubber then treats every old dreaming transcript as unreferenced, but this PR’s unconditional empty-store guard is too broad and disables valid orphan cleanup for agents whose `sessions.json` is legitimately empty.

## What This Changes

The PR adds a warning and early `continue` in `scrubDreamingNarrativeArtifacts` when the loaded session store has no keys.

## Best Possible Solution

Keep the data-loss protection, but make it failure-aware: skip orphan archival only when the store read/parse was not trustworthy, while allowing a valid empty store to continue through the existing stale dreaming transcript cleanup path.

## Reproduction Assessment

Yes. Current main can be reproduced with an agent sessions directory containing an invalid `sessions.json` and old dreaming `.jsonl` files; the introduced regression can be checked with a valid `{}` store plus an old unreferenced dreaming transcript, which this PR would now skip.

## Solution Assessment

No. The direction is right for corrupt store reads, but the implementation is not the narrowest maintainable fix because it conflates failed loads with legitimate empty stores.
