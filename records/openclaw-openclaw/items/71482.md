## Summary

Keep open. The reported data-loss path is credible, but this PR is not safe to merge as-is because the empty-store guard also disables legitimate orphan transcript cleanup for a valid empty session store.

## What This Changes

The PR adds an early warning and `continue` in memory-core dreaming cleanup when `loadSessionStore()` returns an empty object.

## Best Possible Solution

Make the scrubber failure-aware: skip or warn only when the store read/parse is not trustworthy, while preserving stale dreaming transcript cleanup for a valid empty store, and add focused regression tests for both corrupt and valid-empty store states.

## Reproduction Assessment

Yes. The original current-main bug can be reproduced with an invalid `sessions.json` plus old dreaming `.jsonl` files, and the PR regression can be reproduced with a valid `{}` store plus an old unreferenced dreaming transcript that should still be archived.

## Solution Assessment

No. The direction is right for corrupt store reads, but the implementation is not the narrowest maintainable fix because an empty object is also a legitimate session-store state.
