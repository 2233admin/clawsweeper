## Summary

Keep open. Current main still preserves non-off API thinking defaults for Claude 4.6 and generic reasoning-capable catalog models, cron jobs without a payload-level thinking override still fall through to the default resolver, and Microsoft Teams rendering still lacks the reasoning-payload suppression guard used by Mattermost.

## What This Changes

The issue requests cost-safe opt-in defaults for Claude/API thinking, a global reasoning visibility default, startup or doctor warnings for enabled thinking, and separation of API thinking from rendered reasoning output.

## Best Possible Solution

Define one explicit policy: API-level thinking is off unless explicitly configured or required by the provider/model, cron jobs inherit that same resolved global/per-agent policy, user-facing channel renderers suppress reasoning payloads unless the surface explicitly opted in, and billing-affecting default changes are called out in docs and release notes.

## Reproduction Assessment

Yes. The source-level reproduction is to resolve defaults for a reasoning-capable catalog entry or Claude 4.6 model with no explicit thinking setting, and to pass a `Reasoning:` or `isReasoning` payload through the Teams render path.

## Solution Assessment

No. Current main is a partial mitigation because global reasoning visibility config exists and Opus 4.7 defaults off, but the cost-driving API thinking defaults, cron fallback behavior, and Teams rendering guard are not fully resolved.
