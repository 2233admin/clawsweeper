## Summary

Keep this PR open for maintainer/author revision. Current main still has the forced-harness support-check bug, and the selector part of the PR is the right direction, but the branch should not merge as-is because it broadens the Codex harness default provider set in a way that makes `auto` mode claim `openai` and `openai-codex` models despite the documented conservative runtime boundary.

## What This Changes

The PR validates forced and session-pinned plugin harness selection with `supports()`, widens the Codex harness default provider ids, updates harness/status tests, adds Codex harness support tests, and adds a changelog entry.

## Best Possible Solution

Land a narrowed fix that calls `supports({ provider, modelId, requestedRuntime })` before selecting forced or session-pinned plugin harnesses, routes unsupported candidates to PI only when policy allows, and preserves the Codex auto-mode opt-in boundary by making `openai` / `openai-codex` support conditional on explicit Codex runtime or by updating the docs/tests through an intentional product decision.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: current main returns a forced registered harness before `supports()`, and the PR’s changed default provider set would make the existing auto-mode `supports()` probe select Codex for `openai` / `openai-codex` providers.

## Solution Assessment

No, not as currently written. The forced-selection `supports()` check is the narrow maintainable fix, but the Codex provider widening needs to be gated by `requestedRuntime: "codex"` or paired with an explicit docs/test/product update for broader auto-mode behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep Codex provider support out of auto mode:** `extensions/codex/harness.ts:8`
  - body: Adding `openai` and `openai-codex` to the default provider set makes `supports()` return true even when the selector is probing with `requestedRuntime: "auto"`. Since auto-mode selection calls this predicate directly, configs that should stay on PI, especially `openai-codex/*` subscription-auth routes, can be silently moved to the native Codex app-server runtime. Gate the widened provider ids on explicit `requestedRuntime: "codex"` or update the docs/tests for an intentional contract change.
  - confidence: 0.91
