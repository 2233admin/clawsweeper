## Summary

Keep this issue open. Current main has partially addressed the dangerous silent fallback path by treating explicit per-agent primary models as strict unless fallbacks are configured, and remote custom providers without apiKey now fail with a missing-key error. The remaining Cloudflare AI Gateway multi-provider credential problem is still real: credentials are provider-scoped, onboarding only creates the bundled provider profile, and the docs do not explain the same-gateway custom-provider workaround.

## Best Possible Solution

Keep this open for a generic provider-auth policy: document the current per-provider apiKey or SecretRef requirement for Cloudflare AI Gateway custom providers, add an explicit shared credential mechanism if maintainers want same-gateway reuse, and add regression coverage for a second remote Cloudflare-backed provider without apiKey so strict per-agent selections fail loudly while intentional fallback chains still work.

## Risks / Open Questions

- Implicitly sharing credentials by matching baseUrl could leak one provider's credential to an unrelated custom provider; any fix should use explicit SecretRef reuse or a named/scoped gateway credential.
- Changing all 401/403 behavior globally could break intentional configured fallbacks, auth-profile rotation, and provider cooldown handling; strict failure should stay tied to explicit user/agent selection unless maintainers choose otherwise.
