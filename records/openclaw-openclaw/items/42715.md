## Summary

Keep this issue open. Current main still has provider-scoped credentials for custom/gateway-backed model providers and auth failures remain eligible for configured model fallback, so the reported Cloudflare AI Gateway multi-provider behavior is not fully fixed despite improved auth resolution/logging behavior since the original report.

## Best Possible Solution

Keep this issue open. The best fix is a generic model-provider auth policy rather than Cloudflare-only special casing: allow explicit SecretRef reuse or a scoped shared gateway credential mechanism for gateway-backed custom providers, and make missing or unauthorized credentials for a per-agent selected remote provider fail loudly unless that agent/run explicitly opted into fallback. Add regression coverage for a second Cloudflare AI Gateway-backed custom provider without apiKey and update the Cloudflare AI Gateway plus model failover docs.

## Risks / Open Questions

- Changing this behavior touches shared model failover semantics; it needs to preserve intentional fallback chains, auth profile rotation, cron/default fallback policy, and the already strict explicit user-selection path.
- Gateway credential sharing should be explicit and scoped, for example SecretRef reuse or a named gateway credential, because implicit base URL matching could leak one provider's credential to an unrelated custom provider.
- Docs and regression tests need to move together because current documentation still states auth failures can trigger fallback for configured model chains.
