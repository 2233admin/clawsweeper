## Summary

Keep open. Current main partially mitigates the silent fallback path by making per-agent primary models strict unless `fallbacks` is explicit, but the Cloudflare multi-provider credential gap remains: additional remote gateway-backed providers still need their own provider-scoped `apiKey` or SecretRef, and the docs do not yet explain that for same-gateway custom providers.

## Best Possible Solution

Keep this issue open for a generic, explicit model-provider auth policy: document the current per-provider SecretRef workaround for Cloudflare AI Gateway custom providers, add a safe shared credential mechanism if maintainers want gateway-level reuse, and add regression coverage for a second remote Cloudflare AI Gateway-backed provider without `apiKey` so per-agent strict selections fail loudly while intentional fallback chains remain intact.

## Risks / Open Questions

- Implicit credential sharing based only on matching `baseUrl` could leak one provider's credential to an unrelated custom provider; any fix should use explicit SecretRef reuse or a named/scoped gateway credential.
- Changing auth-failure fallback globally could break intentional fallback chains, auth-profile rotation, cron/default model fallback policy, and auto fallback overrides; the strict behavior should stay tied to explicit agent/user selection unless maintainers decide otherwise.
