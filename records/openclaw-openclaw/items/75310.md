## Summary

Keep open. The linked runtime-deps repair bug is valid, but the proposed detector still relies on a materialization check that can pass after an interrupted install has already written the generated manifest and package sentinels, so the patch can still skip the reinstall it is meant to force.

## What This Changes

The PR adds an incomplete bundled runtime-deps install detector, wires it into gateway startup and `openclaw doctor --repair`, and adds regression tests plus a changelog entry for the linked runtime-deps staging bug.

## Best Possible Solution

Use a completion signal written only after successful bundled runtime-deps staging, or strengthen materialization so it verifies the actual loadable package surface for packages without `main`; then gate the gateway and doctor early returns on that reliable signal and cover the manifest-present, sentinel-present, contents-incomplete case.

## Reproduction Assessment

Yes. A focused source-level reproduction can create an install root with `node_modules`, the generated `openclaw-runtime-deps-install` manifest, and `node_modules/<dep>/package.json` for a package with no `main`, while omitting real package contents; the current helper chain can still treat that plan as materialized.

## Solution Assessment

No. The affected gateway and doctor early returns are the right boundary, but `isRuntimeDepsPlanMaterialized` is not a reliable incomplete-install detector because the generated manifest is written before install completion and no-main packages can pass with only the sentinel file.

## Review Findings

Overall correctness: patch is incorrect

- **[P2] Use a post-success completion marker:** `src/plugins/bundled-runtime-deps.ts:222-224`

Gate the early returns on a marker written only after successful staging, or strengthen package verification before using this check.

Fix PR #75310 / issue #75309 by making incomplete bundled runtime-deps staging detection reliable. Do not rely on `isRuntimeDepsPlanMaterialized` alone as a completion signal, because the generated `openclaw-runtime-deps-install` manifest is written before install completion and packages without `main` can pass with only `node_modules/<pkg>/package.json`. Add either a post-success marker written only after `finalizeBundledRuntimeDepsInstall` succeeds, or strengthen materialization so packages without `main` still require a real loadable package surface. Wire the reliable detector into both gateway startup and `openclaw doctor --repair` early-return paths. Add regressions for generated manifest present, package sentinel present, package contents incomplete, and ensure the existing missing-manifest case still repairs. Keep the fix within bundled runtime-deps staging/repair; do not change package-manager selection, dependency versions, install script policy, or plugin activation semantics.
