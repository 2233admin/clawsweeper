## Summary

Keep this issue open. Current main still has no source-backed fix for the reported Slack DM 160-character inbound truncation: the Slack inbound path ultimately builds the agent-facing body from `message.text` plus attachment/media fallbacks, while top-level Slack `blocks` are not represented in the current message event type or parsed in the content resolver. The only explicit 160-character slice I found is a system-event/log preview, not the body passed to the agent, so it explains a possible confusing symptom but does not resolve the reporter's claim that Slack DM `text` is already truncated before OpenClaw builds `BodyForAgent`. The older related #28344 was closed stale and is not a canonical fix for this human-to-bot DM report.

## Best Possible Solution

Keep this issue open and fix it in the Slack plugin inbound content path. The likely direction is to capture a representative Slack DM payload over 160 characters, add a regression test asserting `BodyForAgent` and `RawBody` preserve the full tail, then parse the correct Slack block/rich-text source only when `message.text` is missing or appears to be a preview. The existing 160-character system-event preview should stay compact but must remain separate from agent input.

## Evidence

- **Issue context:** The report says human-to-bot Slack DMs are cut at exactly 160 characters, while Slack channel messages and Web UI messages are not affected; it also points to `text` versus `blocks` parsing as the suspected root cause. The only related linked issue, #28344, covered bot-to-bot channel truncation and was closed stale, so it does not supersede this issue.
- **Inbound event type lacks top-level blocks:** `SlackMessageEvent` currently includes `text`, `files`, and `attachments`, but no top-level `blocks` field. That means a normal Slack DM payload with full content only in top-level blocks would not be represented by the internal type used downstream.
