Keep open. Current main still lacks the response-completion sound settings and still uses a stable service-worker fallback notification tag, so the PR is not obsolete; however, the latest patch still has two narrow merge blockers: it can play a cue for a stream-only silent `NO_REPLY` final, and it lacks the required user-facing changelog entry. The PR adds browser-local response-completion sound controls to the Control UI Notifications surface, adjusts web-push notification tag/renotify behavior, restyles push action buttons, persists the new UI settings, and adds focused UI tests.

## Best Possible Solution

Land the Control UI notification-controls work after carrying forward the existing no-visible-reply semantics into the cue trigger, adding the required Unreleased changelog entry, and reconciling any overlap with the narrower Notifications-tab PR if it lands first.

## Reproduction Assessment

Yes. ## Solution Assessment

No, not yet. The PR direction is maintainable, but the cue trigger should receive a visible-output/silent-completion result from chat handling or inspect the pre-final stream before it is cleared; the user-facing feature also needs a changelog entry before merge.
