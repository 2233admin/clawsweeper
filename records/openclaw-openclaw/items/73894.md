## Summary

Keep open. Current main still lacks the response-completion cue work, and the PR remains useful, but the latest patch still has a concrete P2 correctness issue: the new cue call can fire for a stream-only `NO_REPLY` completion that the chat UI intentionally hides. The earlier Greptile concerns appear addressed in the current PR head, and the remaining overlap with #74715 is a normal PR-reconciliation item rather than a close reason.

## What This Changes

The PR adds Control UI response-completion sound settings and playback, exposes Notifications controls for response cues and web push, changes service-worker notification tag/renotify defaults, restyles push action buttons, and adds UI tests for cue and Notifications-tab behavior.

## Best Possible Solution

Keep the PR open, fix the silent stream-only completion cue edge by carrying visible-output/silent-stream state through the final-event path, reconcile the narrower Notifications-tab overlap if #74715 lands first, then run the PR's targeted UI checks before merge.

## Reproduction Assessment

Yes. With completion sound enabled, start an active run whose stream text is `NO_REPLY`, then emit a final chat event without `message`; current chat handling returns `final` with no visible assistant message, while the PR calls the cue with `message: undefined`.

## Solution Assessment

No. The general PR direction is reasonable, but the cue should be gated on whether the final event produced visible assistant output, or should inspect the pre-clear stream state before signaling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Skip cueing silent stream-only completions:** `ui/src/ui/app-gateway.ts:559-563`

This new call then passes `undefined` to `signalResponseCompletion`, so the PR plays a completion sound for a response the UI intentionally hides; carry visible-output/silent-stream state through before signaling.
