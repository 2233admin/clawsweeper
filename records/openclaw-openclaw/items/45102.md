## Summary

Keep open. Current main has relevant hardening around audio payload limits, media-task concurrency, session maintenance, opt-in transcript-byte compaction, and liveness diagnostics, but none of that definitively fixes the reported Windows + FunASR slowdown/restart loop. The cross-referenced #64767 is useful related evidence for gateway hangs, but it describes a different, more specific session-JSONL root cause, so this issue should not be closed as a duplicate without current-main diagnostics tying the symptoms together.

## What This Changes

Reports that the Gateway becomes slow and unresponsive after several local FunASR voice-recognition requests on Windows, causing `/v1/responses` timeouts until `openclaw gateway restart` is run.

## Best Possible Solution

Keep the issue open until maintainers can collect a current-main diagnostics export, Gateway stability output, session/transcript sizes, memory growth, and the exact FunASR/audio configuration. If that evidence matches #64767's bloated-transcript failure mode, consolidate the work there; otherwise, add a targeted fix around the specific audio/Gateway bottleneck with a regression test or operational diagnostic.
