## Summary

Keep open. Current main still starts the voice-call initial greeting before the OpenAI realtime STT connection is ready, and an open PR with closing syntax is already the active implementation candidate.

## What This Changes

Twilio inbound voice-call media streams using OpenAI realtime transcription should avoid timing out so caller speech is transcribed and routed to the voice-call agent path.

## Best Possible Solution

Review and either land PR #75257 or replace it with an equivalent narrow fix that establishes STT readiness before the initial greeting path can consume runtime resources, with focused regression coverage around media-stream startup ordering and STT failure handling.

## Reproduction Assessment

Yes for triage: the issue includes concrete live steps, redacted config, exact logs, persisted call state, and a direct OpenAI WebSocket comparison. Current main still shows the matching startup order and timeout path, though I did not run a live carrier call locally.

## Solution Assessment

Unclear until PR #75257 is reviewed. Its stated direction, awaiting STT readiness before firing onConnect and the initial greeting, matches the current source-level race and appears to be the narrowest maintainable fix boundary.
