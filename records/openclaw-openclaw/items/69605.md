## Summary

Keep this issue open. Current main still matches the reported Control UI behavior: code-block copy buttons are rendered with `data-code`, the delegated click handler copies that dataset value, assistant message-level copy actions are still absolutely positioned without the `has-copy` spacing class on normal bubbles, and there is no focused regression coverage for rendered code-block clipboard text or action overlap. Two open, unmerged PRs already reference this issue with closing syntax, so the issue should stay open until maintainers review and land or close the canonical fix.

## What This Changes

The issue reports that the Control UI WebChat code-block Copy button copies the whole assistant markdown message instead of only the fenced code block content.

## Best Possible Solution

Keep the issue open and review or consolidate the linked Control UI PRs into one canonical fix that copies the rendered code-block text only, prevents parent/message-level copy actions from winning the click target, and adds focused regression tests for multiple fenced blocks, special characters, inner button targets, propagation, and the assistant-bubble spacing contract.

## Reproduction Assessment

Yes. The issue includes deterministic Control UI steps, and current main source confirms the relevant path: assistant bubbles can render message-level copy actions without the existing spacing class, while code-block copy handling still depends on the button dataset rather than the rendered code element.

## Solution Assessment

Yes for the intended product direction, but no single current-main implementation solves it yet. The narrow maintainable solution belongs in the Control UI render/click handling and tests, not in gateway, model, or provider code.
