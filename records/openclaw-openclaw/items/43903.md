## Summary

Keep this open. Current main still uses one shared gateway token/password path, while the newer browser-local Control UI identity only helps display attribution and does not create per-token credentials, scopes, trusted auth identity, or token-derived session isolation; the team-deployment comment makes the gap concrete rather than stale.

## What This Changes

The issue requests multiple `gateway.auth` tokens with per-token labels/scopes so shared Control UI deployments can authenticate and isolate users without switching to trusted-proxy mode.

## Best Possible Solution

Define a backwards-compatible core gateway auth contract for multiple token entries, resolve the matched entry into an authenticated identity with explicit scope semantics, propagate that identity where Control UI/session metadata needs it, preserve existing single-token/password/device/trusted-proxy/Tailscale behavior, and document/test the security model.
