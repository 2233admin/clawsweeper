## Summary

Keep open. Current main still supports token-mode gateway auth as a single shared secret and does not resolve token entries into server-authenticated labels, scopes, or principals; the browser-local Control UI identity work helps local attribution but does not provide multi-user session isolation. Related open work covers adjacent RBAC, author-attribution, and gateway-auth-scope design, but none is a canonical implementation of this request.

## What This Changes

Support multiple gateway tokens mapped to per-user identity and scope metadata so shared Control UI access can isolate users and sessions.

## Best Possible Solution

Define an additive, backwards-compatible gateway auth contract for multiple shared-token entries, resolve the matched entry into a server-authenticated principal with explicit scope semantics, preserve current single-token behavior, and propagate identity only through documented Control UI/session/author surfaces after the trust model is agreed.

## Reproduction Assessment

Not applicable as a feature request. Current-main source verification is high-confidence: config types/schema, resolved auth, token authorization, and the WebSocket auth schema all lack a multi-token map or token-principal metadata.

## Solution Assessment

Unclear. The proposed tokens map is plausible, but the best solution needs maintainer/security design and should coordinate with open RBAC, shared-auth scopes, service-identity, and author-attribution discussions before code changes.
