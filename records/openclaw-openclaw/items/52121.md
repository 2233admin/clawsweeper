## Summary

Keep this PR open. Current main still does not implement equivalent gateway startup diagnostics for conflicting exec/sandbox/elevated settings. The existing docs, security audit checks, runtime exec errors, and hard no-auth bind refusal partially cover adjacent cases, but they do not replace the PR’s central non-throwing startup warning behavior. The PR discussion is also still relevant: the author addressed some early review feedback, but later review comments identify correctness and security/operational risks that need maintainer follow-up before merge.

## Best Possible Solution

Keep this PR open for maintainer review or a refreshed implementation. The best path is a non-throwing startup diagnostic that runs after effective runtime config is known, reuses existing security-audit semantics where possible, evaluates per-agent exec/elevated/sandbox combinations, treats the already-enforced no-auth bind case consistently, corrects the `exec.security` guidance, and adds regression coverage for per-agent overrides, CLI/runtime bind overrides, SSH sandbox config, and the original channel-session debugging path.

## Risks / Open Questions

- Merging the PR as-is could miss conflicts when effective exec/elevated/sandbox settings come from `agents.list[]` overrides rather than global/default config.
- The warning-only detector can become a startup blocker if it calls sandbox resolution paths that throw on unrelated SSH sandbox normalization errors.
- Logging a critical bind/auth warning from raw config before runtime CLI/auth overrides are applied can produce false positives for safe loopback runs.
- The PR’s `exec.security=full` hardening guidance conflicts with current docs and audit semantics, where `full` is a broad-trust host exec posture.
