## Summary

Keep this open. Current main still has only provider-side interruption handling: the Google Meet command-pair and paired-node realtime paths continue to forward shared captured audio to the realtime provider without a local side-channel human barge-in detector, playback-state loopback suppression, or transport-facing bridge interruption hook. The linked draft PR #73834 is the active implementation candidate rather than a reason to close the issue before review/merge.

## What This Changes

Request Google Meet realtime voice support for human barge-in during assistant playback, including optional side-channel microphone detection, local loopback suppression, provider interruption, and output-buffer clearing.

## Best Possible Solution

Keep this tracked until #73834 or a reshaped replacement lands an additive realtime voice interruption hook plus Google Meet-owned optional side-channel human speech detection, loopback suppression while assistant output is active, local output-buffer clearing, config metadata/docs, and focused command-pair coverage.

## Reproduction Assessment

No high-confidence live reproduction was run. The source-level path is clear: current main forwards shared command-pair input while assistant output can be active and lacks the requested side-channel barge-in/suppression path.

## Solution Assessment

Unclear until the draft PR is reviewed. The proposed additive provider hook plus optional Google Meet side-channel command fits the extension boundary, but threshold defaults, external bridge compatibility, live behavior, and local command security need maintainer review.
