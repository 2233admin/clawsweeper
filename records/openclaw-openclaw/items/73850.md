## Summary

Keep this issue open. Current main still lacks the requested Google Meet transport-level human barge-in path: command-pair audio is forwarded to the realtime bridge during playback, the shared realtime voice session surface has no transport-callable interruption hook, and Google Meet config/docs do not expose an optional side-channel microphone command or threshold controls. The linked draft PR #73834 remains the open implementation candidate rather than a reason to close this issue before merge.

## What This Changes

Issue asks for Google Meet realtime voice to stop assistant playback quickly when a human speaks, avoid treating assistant loopback as user speech, and preserve existing Chrome command-pair and external bridge behavior.

## Best Possible Solution

Keep this issue tracked until #73834 or a reshaped PR lands an additive realtime voice interruption hook plus Google Meet-owned optional side-channel human speech detection, loopback suppression during assistant playback, local output-buffer clearing, config/docs/tests, and representative command-pair validation.

## Reproduction Assessment

No; I do not have a high-confidence live reproduction path because no Google Meet session with shared BlackHole routing was run. Static inspection does show the current command-pair paths forward shared input during playback and lack the requested side-channel barge-in mechanism.

## Solution Assessment

Unclear until the draft PR is reviewed. The proposed additive provider hook and optional Google Meet side-channel command fit the plugin boundary, but thresholds, command trust, live audio behavior, and provider semantics need maintainer and security review.
