## Summary

Keep open. Current main still lacks durable normal WebChat send handling across a WebSocket reconnect window: disconnected submits return before queueing, in-flight `chat.send` RPCs are rejected on close, and reconnect only resumes the queue for seq-gap recovery even though the gateway already has an ACK/idempotency contract that can support retry.

## What This Changes

The issue requests client-side WebChat outbound queuing, gateway ACK handling, reconnect retry for unACKed messages, and failed-delivery UI feedback.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the message before transport, preserves a stable idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and shows bounded failed-delivery feedback without broadening into full refresh/history persistence.
