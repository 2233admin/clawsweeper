## Summary

Current main still appears to have the reported WebChat reconnect data-loss path: normal sends return before queueing when disconnected, failed `chat.send` RPCs are converted into UI errors instead of retained for replay, and reconnect only resumes queued follow-ups for seq-gap recovery. The server already exposes ACK and idempotency behavior, so this is a narrow Control UI reliability fix rather than a stale or already-implemented issue.

## What This Changes

The issue requests reliable Control UI/WebChat message delivery across WebSocket reconnects by adding client-side outbound queuing, ACK-aware retry, and failure feedback for normal chat sends.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that captures the message before transport, preserves a stable idempotency key through ACK, retries transient closed/not-connected failures after reconnect hello, and shows bounded failed-delivery feedback while relying on the existing gateway ACK/dedupe contract unless it proves insufficient.

## Repair Work Prompt

Fix the WebChat/Control UI reconnect-window data-loss bug tracked by https://github.com/openclaw/openclaw/issues/45952. Current main drops normal outbound sends when the socket is disconnected or when `chat.send` is rejected during close/reconnect. Implement a focused Control UI outbound-send queue for normal `chat.send`: capture the message and attachments before transport, assign and preserve a stable idempotency key until the Gateway ACKs, retry transient closed/not-connected failures after reconnect hello, and surface bounded failed-delivery feedback after retry exhaustion. Use the existing Gateway `chat.send` ACK/dedupe contract; do not change the gateway protocol unless code inspection proves the current contract is insufficient. Be careful that ACK-lost retries reuse the original idempotency key to avoid duplicate agent turns. Keep the fix in Control UI/WebChat code and tests, not channel/provider code. Add or update focused tests for disconnected submit, in-flight RPC rejection during close, reconnect replay, idempotency-key reuse, and attachment handling.
