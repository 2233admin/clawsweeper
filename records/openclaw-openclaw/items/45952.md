## Summary

Keep this open. Current main still has the reported reconnect-window message-loss path: normal sends are not captured before the transport check, pending WebSocket RPCs are rejected on close, and normal reconnect does not replay an unACKed `chat.send`, while the Gateway already exposes ACK/idempotency behavior that a focused Control UI retry queue can use.

## What This Changes

The issue requests reliable WebChat delivery by queueing outbound Control UI messages across WebSocket reconnects, retrying until server ACK, and showing failed-delivery feedback.

## Best Possible Solution

Add a focused Control UI outbound-send queue for normal `chat.send` that records the request before transport, preserves a stable idempotency key until ACK or terminal reconciliation, retries transient closed/not-connected failures after reconnect hello, and shows bounded failed-delivery state without changing the Gateway protocol unless the existing ACK contract proves insufficient.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to submit while `host.connected` is false, or close the browser WebSocket while `chat.send` is awaiting ACK; current main either returns before queueing or rejects the pending RPC and does not replay it after reconnect.

## Solution Assessment

Yes. The narrow maintainable fix is in the Control UI send/reconnect lifecycle, using the existing Gateway ACK/idempotency contract rather than redesigning the protocol first.
