## Summary

Current main still lacks a durable WebChat outbound queue/ACK retry path for messages submitted during a WebSocket reconnect. The existing Control UI queue is for messages deferred while a chat run is already active, and the gateway client rejects or flushes in-flight RPCs when the socket closes. Related work exists in open PR #73154 for #51549, but that PR is not merged and does not explicitly close this issue's reconnect-window send case.

## Best Possible Solution

Keep this issue open and coordinate it with #51549, #73621, and PR #73154. The best fix is a focused Control UI outbound send queue that treats the existing `chat.send` response as the server ACK, preserves the same `idempotencyKey` until ACK, replays transient `gateway closed` / `gateway not connected` sends after reconnect, drains after `onHello`, and shows a bounded failure state if delivery cannot be confirmed. Avoid a new protocol surface unless the current RPC ACK is insufficient.
