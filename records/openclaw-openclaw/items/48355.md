## Summary

Keep open. Current main only implements part of the intended surface: it has an internal agent event stream with `thinking` events and a host-owned plugin event subscription API, but it does not implement the PR's requested `thinking_start` / `thinking_end` plugin hooks, `api.onAgentEvent(...)` registration API, or `agent_end` token/tool metadata.

## Best Possible Solution

Keep the PR open for maintainer review or author follow-up. The best path is to rebase onto current main, decide whether the existing `registerAgentEventSubscription(...)` host seam should be the public channel-plugin path, then add only the missing contract pieces: typed thinking lifecycle hooks if still desired, `agent_end` token/tool metadata, docs/API baseline updates, and focused tests that resolve the review comments. If the remaining product goal is only channel presence liveness, route that narrower work through #65919 instead.

## Risks / Open Questions

- The PR touches a security-sensitive data surface: agent event subscriptions can include assistant text, thinking deltas, and tool activity. Any merge needs a clear session/run scoping model and tests for hidden/non-Control-UI channel runs.
- Current main already has `registerAgentEventSubscription(...)` and runtime `events.onAgentEvent(...)`; adding a separate `api.onAgentEvent(...)` shape could create overlapping plugin contracts unless reconciled.
- The proposed thinking lifecycle hooks need edge-case tests for native events, tagged reasoning, duplicate end events, message-end fallback, and code-span handling before they become public SDK API.
