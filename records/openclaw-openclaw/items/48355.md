## Summary

Keep open. Current main partially overlaps through the internal agent event stream and host-owned plugin event subscriptions, but it does not implement this PR's proposed `thinking_start`/`thinking_end` plugin hooks, top-level `api.onAgentEvent(...)` channel-plugin API, or `agent_end` token/tool metadata. The PR also changes a security-sensitive event surface that still needs maintainer review.

## Best Possible Solution

Keep this PR open for maintainer review or author follow-up. The best path is to rebase onto current main, reconcile the proposed streaming API with `registerAgentEventSubscription(...)` and `api.runtime.events.onAgentEvent(...)`, decide whether typed `thinking_start`/`thinking_end` hooks are desired, add `tokenUsage`/`toolCallCount` to `agent_end` only with aligned SDK docs and baselines, and include tests for event scoping, listener cleanup, and native/tagged reasoning edge cases.

## Risks / Open Questions

- The proposed API can expose assistant text, thinking deltas, and tool activity to plugins; maintainer review should require explicit session/run scoping and cleanup semantics before merge.
- The current main already has `registerAgentEventSubscription(...)` and trusted `api.runtime.events.onAgentEvent(...)`, so adding another public event API could create overlapping SDK contracts unless maintainers choose one shape.
- Thinking lifecycle hooks need focused tests for native events, tagged reasoning, duplicate/orphaned end events, message_end fallback, and code-span filtering before this can safely land.
