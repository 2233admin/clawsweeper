## Summary

Keep #38568 open. Current main still exposes context usage through `/status` and `session_status`, but the system prompt Runtime line is built from static runtime metadata only and has no per-turn context-window usage input or renderer field.

## Best Possible Solution

Implement this in core prompt assembly. Add a typed context-usage input to the runtime/system prompt params, populate it in embedded and CLI prompt wiring from the current prompt token snapshot and effective context token budget, render a compact Runtime field such as `context=49%` or `context=98k/200k (49%)`, and add focused tests for `buildRuntimeLine` plus runner/helper wiring while keeping `/status` and `session_status` behavior unchanged.

## Risks / Open Questions

- A correct implementation needs a fresh pre-request prompt/context snapshot; using stale after-turn provider usage could produce misleading percentages.
- The denominator should be the effective runtime context cap, including configured `contextTokens`, not only native model `contextWindow`.
- Because the field is volatile, tests should preserve prompt-cache behavior; Runtime is already below the cache boundary, but the new value should stay there.
