## Summary

Keep this issue open. Current main still renders the system prompt Runtime line from static runtime metadata only; context-window usage is available through status/session_status, but it is not injected into the prompt Runtime section as requested. The previous ClawSweeper review reached the same conclusion, and the checked-out main still matches that state.

## Best Possible Solution

Implement this in core prompt assembly. Add a small context usage object to RuntimeInfoInput or adjacent system prompt params, populate it from the effective context token budget and current prompt/context token estimate before buildEmbeddedSystemPrompt/buildAgentSystemPrompt runs, render a compact Runtime field such as context=49% or context=98k/200k (49%), and update focused tests/docs/changelog. Coordinate with the open Runtime-line identifier proposals #40044 and #66312 so the Runtime line changes land coherently.

## Clownfish Work Prompt

Implement context-window usage injection into the system prompt Runtime line for this issue. Current main exposes context usage through /status and session_status, but RuntimeInfoInput/buildRuntimeLine and the embedded/CLI prompt wiring do not pass or render it.

Expected fix boundary:
- Add a small typed context usage value to RuntimeInfoInput or adjacent system prompt params, e.g. used tokens + budget and/or percent.
- Render a compact Runtime field such as `context=49%` or `context=98k/200k (49%)` in buildRuntimeLine when reliable data is available; omit it when unavailable rather than guessing.
- Populate embedded runs from the effective runtime context budget (`ctxInfo.tokens` / contextTokens cap) and a fresh pre-request prompt/context token estimate. Do not use stale last-turn provider usage if it can lag.
- Define CLI behavior explicitly: either populate from a real available snapshot or omit the field; do not invent a percentage.
- Keep the field below the prompt cache boundary and add tests proving stable-prefix behavior is not regressed.
- Leave status/session_status behavior intact except for reusing a formatter if that keeps output consistent.
- Coordinate with Runtime-line identifier work in #40044 and #66312 so fields do not conflict.
- Update docs/concepts/system-prompt.md and add a single-line CHANGELOG.md entry under the active Changes section with appropriate Thanks @blortski attribution.

Do not add plugin-specific behavior, startup migrations, broad prompt rewrites, dependency changes, or status output churn unrelated to this feature.

## Risks / Open Questions

- The implementation should use a fresh pre-request prompt/context snapshot; last-turn provider usage can lag by one turn.
- The denominator should be the effective runtime context token budget, including contextTokens/config caps, not only the model-native contextWindow.
- The new field is intentionally volatile, so tests should prove it stays below the prompt cache boundary and does not churn the stable prefix.
- CLI-backed runs may not have the same live token snapshot as embedded runs; the fix needs an explicit omit or fallback behavior.
