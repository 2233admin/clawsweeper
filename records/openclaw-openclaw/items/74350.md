## Summary

Keep open. Current main still classifies the reported Anonymous/Guest/Interactive/Local/Network Windows SIDs and names as `group`, not `world`; that makes `worldWritable` false for those ACLs, and at least the state-dir audit path treats world-writable as critical while group-writable is only a warning.

## What This Changes

The issue requests treating additional broad or unauthenticated Windows well-known SIDs and principal names as world-equivalent in Windows ACL auditing.

## Best Possible Solution

Keep this issue open for maintainer security triage, then make a narrow classifier/test change in `src/security/windows-acl.ts` and `src/security/windows-acl.test.ts` after confirming which well-known SIDs should be treated as world-equivalent. Add at least one audit-level regression for a state-dir ACL using one of the affected SIDs so the critical-vs-warn behavior is covered.

## Risks / Open Questions

- This is security-sensitive: the current classifier appears to downgrade some broad Windows principals from world to group, which can reduce audit severity on surfaces that distinguish the two.
- The proposed list mixes unauthenticated principals with broad-but-contextual logon groups such as Interactive, Local, and Network; maintainers should confirm the exact world-equivalent policy before landing.
- Some consumers already treat group and world together, so the fix should include targeted regression coverage for the world-only or severity-sensitive audit paths rather than assume a uniform bypass.
