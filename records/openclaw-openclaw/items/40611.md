## Summary

Keep open. Current main still requeues `requests-in-flight` heartbeat skips through the shared 1 second wake retry, tests still assert the 1 second cooldown, and the heartbeat schema/types/docs still do not expose a separate busy-session retry delay. The latest release has the same core behavior, and the direct fix candidate in #40629 is closed unmerged.

## What This Changes

Add a heartbeat busy-session retry/backoff so `requests-in-flight` skips do not retry every 1 second and immediately block active Telegram conversations after the current turn finishes.

## Best Possible Solution

Keep this open and implement a narrow heartbeat busy-session retry policy: preserve the no-drift schedule behavior from #39182/#29762, add a separate configurable delay/backoff for `requests-in-flight` skips under default and per-agent heartbeat config, update schema/types/help/generated docs/public docs, and add wake/scheduler regression tests. Leave broad session-lock scope to #13744 and activity-based heartbeat semantics to #14051.

Fix #40611 by adding a separate heartbeat busy-session retry delay for `requests-in-flight` skips. Current main uses `DEFAULT_RETRY_MS = 1_000` for retryable busy skips; do not reintroduce the full-interval schedule drift fixed by #39182/#29762. Add a configurable default and per-agent heartbeat field, for example `agents.defaults.heartbeat.retryDelayMs` and `agents.list[].heartbeat.retryDelayMs`, defaulting around 60 seconds, and use it only for `HEARTBEAT_SKIP_REQUESTS_IN_FLIGHT`. Keep thrown-error retry behavior and cron/nested-lane busy retry behavior unchanged unless a maintainer explicitly chooses a broader policy. Wire the value through `startHeartbeatRunner.updateConfig` into heartbeat wake state, update strict schema/types/help/labels/generated config docs, heartbeat config-honor inventory, and `docs/gateway/heartbeat.md`. Add regression tests for the default delay and custom config. Use #40629 as prior art but adapt to current main, where retryable busy reasons now include `cron-in-progress` and `lanes-busy` and `skipWhenBusy` exists. Do not change Telegram/Feishu plugin code, broad session write-lock scope (#13744), or activity-based heartbeat semantics (#14051).
