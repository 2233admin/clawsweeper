## Summary

Keep open. Current main still retries heartbeat `requests-in-flight` skips through the shared 1 second wake retry, and there is still no heartbeat-specific busy-session retry/backoff config in schema, types, or docs. The latest release tag inspected has the same behavior, and the direct fix candidate #40629 is closed unmerged.

## Best Possible Solution

Implement a narrow heartbeat busy-session retry policy: preserve the no-drift behavior, add a separate configurable delay/backoff for `requests-in-flight` skips under `agents.defaults.heartbeat` and per-agent heartbeat config, wire it into the wake/runner path without changing generic error retry, and update schema/types/docs plus wake/scheduler regression tests. Keep #13744 for broader session-lock scope and #14051 for activity-based heartbeat semantics.

## Clownfish Work Prompt

Fix the heartbeat busy-session retry regression described by this issue. Current main retries `requests-in-flight` heartbeat skips through the shared 1 second wake retry, which can make a heartbeat start immediately after an active Telegram conversation releases the session lock. Add a separate heartbeat busy-session retry delay/backoff, defaulting conservatively around 60s unless maintainers choose another value, while preserving the no-drift behavior from #39182 and not changing generic handler-error retries. Expose the setting under `agents.defaults.heartbeat` and per-agent `agents.list[].heartbeat`, update the strict zod schema, TypeScript config types, public docs, and any generated config/help artifacts required by the repo. Keep #13744 and #14051 out of scope except for comments/tests that make the boundary clear. Do not change Telegram channel behavior or the session write-lock implementation. Add focused regression tests proving `requests-in-flight` skips use the new busy retry delay/config and do not advance `nextDueMs` by the full heartbeat interval. Changelog: add a user-facing fix entry only if repo policy expects one for this behavior change, credited to the reporter/contributor, not Codex.

## Risks / Open Questions

- A longer retry must not advance `nextDueMs` or otherwise reintroduce the drift fixed for #29762.
- Changing the shared `DEFAULT_RETRY_MS` would affect generic wake-handler error retries; the fix should separate busy-session retry/backoff from error retry.
- The default busy-session delay is product-sensitive: too short can keep blocking active Telegram conversations, too long can delay legitimate heartbeat/task notices.
