## Summary

Keep open. Current main still lacks a typed delivery intent or prompt/runtime carve-out for explicit channel-delivery requests. `sessions_send` still accepts only the existing free-text fields, wraps requests as generic `inter_session` input, and routes through A2A reply/announce handling where `REPLY_SKIP`, `ANNOUNCE_SKIP`, and `NO_REPLY` remain valid suppression outcomes. Related #43848 overlaps the A2A suppression area, but it is open and Telegram-topic-specific, so it is not a canonical duplicate for this WhatsApp explicit-delivery issue.

## Best Possible Solution

Keep this issue open and decide the core sessions_send/A2A contract: preferably add a backward-compatible typed delivery-intent signal, or a narrowly documented runtime/prompt guard, so explicit requests to emit a normal channel message outrank generic REPLY_SKIP, ANNOUNCE_SKIP, and NO_REPLY guidance. Add a focused WhatsApp group regression for authorized always/allowed activation, explicit delivery wording, and forbidden skip-token outcomes while preserving coordination-chatter suppression.
