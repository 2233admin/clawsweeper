## Summary

Current main still leaves inline `models.providers.<id>.apiKey` credentials outside durable auth-profile cooldown tracking: inline config auth resolves without a `profileId`, billing failure marking returns when no profile id/profile exists, and the later fallback skip path only runs when stored auth profiles exist. The related fix PR #39836 is closed and unmerged in the provided context, so the issue still needs a focused fix.

## Best Possible Solution

Keep this issue open and implement a stable, non-secret cooldown identity for inline `models.providers.<id>.apiKey` credentials. The fix should let billing/auth failures persist disabled state for that inline identity, make later turns skip the disabled inline provider and continue to configured fallbacks, preserve existing auth precedence, and add regression coverage for billing 402 on an inline-key provider.

## Clownfish Work Prompt

Fix the inline API-key billing cooldown gap for this issue. Current main resolves `models.providers.<provider>.apiKey` credentials without a durable `profileId`, so billing/auth failures cannot persist `disabledUntil` and later fallback runs cannot skip the dead inline provider. Implement a stable non-secret cooldown identity for inline API-key providers, either by returning a synthetic profile id such as `inline:<provider>` from inline config auth resolution and teaching auth-profile usage to create/use a minimal no-secret profile entry, or by adding an equivalent provider-level cooldown key. Preserve existing auth precedence: explicit `auth: "api-key"` config still wins over ambient profiles, stored real profiles still work, local synthetic markers should not be treated as failed remote credentials, and no plaintext inline API key may be written to `auth-profiles.json`. Add regression coverage proving a billing 402 on an inline configured provider records disabled state, persists it, and a subsequent model fallback run skips that provider and uses a configured fallback. Use #39836 only as prior context; do not copy unsafe assumptions blindly. Do not touch dependencies, lockfiles, release metadata, or unrelated provider behavior. Changelog only if the maintainer-facing behavior is user-visible enough under the active release notes policy.
