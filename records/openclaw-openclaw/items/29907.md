## Summary

Keep open. Current main still only supports cron payloads for `systemEvent` and `agentTurn`; there is no first-class `exec` or `lobster` cron payload, no `condition`/`onCondition` handoff contract, and detached cron execution still enters the isolated agent runner. Lobster exists as an optional tool, but not as a scheduler executor. #18160 remains the narrower direct-exec feature request, while this issue tracks the broader deterministic pipeline plus conditional agent-turn design.

## What This Changes

Request a new cron execution kind for direct shell or Lobster pipeline runs that evaluates deterministic output conditions and only starts an `agentTurn` when the condition passes.

## Best Possible Solution

Keep this open for maintainer design. The best path is either an additive cron executor contract for `exec` and Lobster payloads with bounded condition evaluation and conditional `agentTurn` handoff, or an explicit plugin/ClawHub plan that first adds the missing executor API seam. Coordinate with #18160 for the narrower direct-exec subset, but keep this issue tracking the broader deterministic pipeline plus conditional-agent workflow.

## Risks / Open Questions

- A native scheduled `exec` or Lobster path would broaden unattended local command/workflow execution, so the design needs explicit approval policy, sandboxing, allowlists, timeouts, output caps, run-log redaction, templating, and delivery semantics.
- The request overlaps #18160 but is not a duplicate; closing it into direct exec alone would lose the Lobster pipeline and conditional `agentTurn` handoff requirements.
- The current plugin cron surface can observe and manage gateway-owned cron jobs, but it is not enough to implement a safe external cron executor without either a core cron payload change or a new plugin executor API seam.
