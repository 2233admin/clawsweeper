## Summary

Keep this PR open. The requested timeout-validation bug still exists on current main, but the proposed branch is stale against the lazy gateway-rpc runtime split and would also change the current channels probe default, so it should be rebased and corrected rather than closed as implemented or merged as-is.

## What This Changes

The PR adds stricter CLI timeout parsing helpers, wires them into gateway, gateway-rpc, devices, and channels status timeout paths, and adds focused regression tests.

## Best Possible Solution

Land one rebased timeout-validation fix that rejects explicit invalid timeout values at the active gateway-facing CLI boundaries, applies validation in current runtime modules, preserves existing defaults including gateway-rpc 30s and channels status probe 30s, and keeps focused tests for both Commander parsing and direct-call helpers.

## Reproduction Assessment

Yes. A current-main static reproduction path is callGatewayFromCli with an invalid timeout, which reaches gateway-rpc.runtime.ts and forwards Number(...) output; channelsStatusCommand({ probe: true }) without timeout also provides a clear default-preservation check.

## Solution Assessment

No. The validation direction is sound, but this branch must be rebased onto the lazy runtime split and must preserve the current channels status probe fallback before it is the narrowest maintainable fix.
