## Summary

Keep open. The reported timeout-validation bug is real and the PR contains useful source and tests, but current main has since split the gateway-rpc implementation into a lazy runtime module, so the patch misses the active gateway-rpc call path and also changes the channels status probe fallback from 30s to 10s.

## What This Changes

The PR adds strict timeout parsing helpers, wires them into gateway, gateway-rpc, devices, and channels status CLI timeout paths, and adds focused regression tests for invalid timeout rejection and valid/default timeout passthrough.

## Best Possible Solution

Land one rebased timeout-validation fix that validates explicit invalid timeout values at the gateway-facing CLI boundary, updates the active gateway-rpc runtime path, preserves existing defaults including gateway-rpc 30s and channels status probe 30s, adds focused direct-call and Commander coverage, and explicitly supersedes or coordinates the overlapping timeout PRs.
