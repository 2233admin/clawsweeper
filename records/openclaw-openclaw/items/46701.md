## Summary

Keep open. Current `main` still drops ordinary Telegram group messages from senders rejected by the group allowlist path without sending the requested onboarding reply, and the visible authorization replies that do exist are separate DM-pairing or native-command flows. The request is valid, but it needs maintainer product/privacy direction before implementation.

## What This Changes

The issue requests a Telegram group onboarding reply when a sender is rejected by `groupAllowFrom` / `groupPolicy` allowlist checks.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is topic-aware, deduplicated or rate-limited, uses the existing Telegram send/error helpers, handles sender ID disclosure deliberately, documents the privacy tradeoff, and tests ordinary group messages separately from DM pairing and native command authorization.

## Security Review

Status: needs_attention

Summary: No PR patch is under review, but the requested behavior itself needs privacy and spam-control review before implementation.

Concerns:

- **[medium] Avoid public Telegram user ID disclosure by default:**
  - body: The requested reply would post the sender's numeric Telegram user ID into the group; that may be acceptable when explicitly enabled, but it should not be hard-coded as an unconditional public default.
  - confidence: 0.87
- **[medium] Rate-limit unauthorized group notices:**
  - body: Replying to every rejected group sender can create group spam or make the bot easier to probe, so any implementation should include opt-in and dedupe or rate limiting.
  - confidence: 0.84
