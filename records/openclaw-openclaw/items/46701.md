## Summary

Keep open. Current main still silently drops ordinary Telegram group messages from senders rejected by groupPolicy/groupAllowFrom, and the existing visible authorization replies are for native command or DM-pairing paths rather than this group-message path. The request is valid, but the public-reply default, Telegram user ID disclosure, admin-contact wording, and rate-limit/dedupe behavior need maintainer product and privacy direction before implementation.

## What This Changes

Request that Telegram reply to group senders rejected by groupAllowFrom/groupPolicy with admin-contact instructions and the sender's Telegram user ID.

## Best Possible Solution

Add a Telegram-owned, explicit opt-in unauthorized-group-sender notice that is topic-aware, uses existing Telegram send/error helpers, avoids public user-ID disclosure unless deliberately configured, includes dedupe or rate limiting, documents the privacy tradeoff, and tests ordinary group messages separately from DM pairing and native command authorization.

## Security Review

Status: needs_attention

Summary: No PR patch is under review, but the requested behavior itself needs privacy and spam-control review before implementation.

Concerns:

- **[medium] Avoid public Telegram user ID disclosure by default:**
  - body: The requested sample reply would post the sender's numeric Telegram user ID into the group; that may be useful when explicitly enabled, but it should not be hard-coded as an unconditional public default.
  - confidence: 0.88
- **[medium] Rate-limit unauthorized group notices:**
  - body: Sending a bot reply for every rejected group message can spam the group and make the bot easier to probe, so an implementation should be opt-in and deduped or rate-limited.
  - confidence: 0.85
