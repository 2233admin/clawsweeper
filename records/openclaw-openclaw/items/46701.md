## Summary

Keep open. Current main still silently drops ordinary Telegram group messages from senders blocked by `groupAllowFrom`/`groupPolicy`; the existing visible “not authorized” reply only covers native command handling. The adjacent open #38346 is related to collecting Telegram user IDs from group traffic, but it does not fully track this issue’s requested admin-contact auto-reply UX.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment. A good implementation would likely add an explicit Telegram group unauthorized-reply option, default it off or rate-limit it, include safe thread-aware delivery, reuse existing Telegram formatting/error-policy helpers, document the setting, and test ordinary group messages separately from native command auth replies.
