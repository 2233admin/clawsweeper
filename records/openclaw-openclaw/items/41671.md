## Summary

Keep open. Current main still recognizes explicit Telegram network config, but the Telegram transport still arms IPv4-only fallback based on dispatcher mode rather than the config source, and inbound media still passes those fallback attempts into `fetchRemoteMedia`. PR #42059 remains open and unmerged as a possible implementation path.

## Best Possible Solution

Keep this issue open. Review and land #42059 if its Telegram network patch is still acceptable, or apply an equivalent narrow fix in `extensions/telegram/src/fetch.ts` so sticky IPv4 fallback is not armed when `channels.telegram.network.autoSelectFamily` or `channels.telegram.network.dnsResultOrder` was explicitly configured. Add regression coverage in `extensions/telegram/src/fetch.test.ts` for `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"`, and keep media-path coverage proving `fetchRemoteMedia` no longer receives forced IPv4 fallback attempts for that explicit-config case.
