## Summary

Current main still has the reported Telegram network behavior. Explicit `channels.telegram.network` config is recognized, but the runtime sticky fallback is still armed from dispatcher mode and still builds IPv4-only fallback attempts that inbound media downloads consume. Related PR #42059 is still open and unmerged, so this issue should remain open for maintainer follow-up rather than be closed.

## Best Possible Solution

Keep this issue open. Review #42059 only as a possible source of intent, but the best current fix is likely a narrow Telegram plugin patch in `extensions/telegram/src/fetch.ts` that does not arm sticky IPv4 fallback when `channels.telegram.network.autoSelectFamily` or `channels.telegram.network.dnsResultOrder` came from explicit config. Add regression coverage in `extensions/telegram/src/fetch.test.ts` and media-path coverage proving `fetchRemoteMedia` no longer receives IPv4-only fallback attempts for the explicit IPv6-friendly config case.
