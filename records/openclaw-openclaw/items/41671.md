## Summary

Current main still allows Telegram direct transports to promote to an IPv4-only sticky fallback after retryable fetch errors, even when the initial dispatcher honored explicit IPv6-friendly network config. The issue also has an open implementation candidate, #42059, so this should stay open until that fix or an equivalent narrow patch lands.

## Best Possible Solution

Keep this issue open. Review and land #42059 if it is still the right patch, or apply an equivalent narrow fix in `extensions/telegram/src/fetch.ts` so sticky IPv4 fallback is not armed when Telegram network family/order settings were explicitly configured for the account. Add regression coverage for API fetch and inbound media download with `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"`.
