## Summary

Current main still appears to have the reported Telegram network bug. The initial Telegram transport honors explicit `channels.telegram.network` settings, but direct transports still arm a sticky IPv4-only fallback after retryable fetch errors without checking whether the user explicitly configured IPv6-friendly behavior. Telegram media downloads consume those dispatcher attempts, so the media path can still be pushed onto IPv4. The related open PR #42059 is still an implementation candidate, not evidence that this issue is fixed on main.

## Best Possible Solution

Keep this issue open. Review and land #42059 if it is still the right patch, or apply an equivalent narrow fix in `extensions/telegram/src/fetch.ts` so sticky IPv4 fallback is not armed when Telegram network family/order settings were explicitly configured. Add regression coverage for API fetch and inbound media download with `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"`.
