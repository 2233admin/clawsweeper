## Summary

Keep open. Current main and v2026.4.29 still await provider usage auth before JSON output can be written, and the open closing PR #74185 is unmerged, so this should remain open for that fix or an equivalent narrow change.

## What This Changes

The issue reports that `openclaw status --usage --json` regressed in OpenClaw 2026.4.26 by hanging or failing from non-TTY subprocess automation, especially with OpenAI Codex OAuth usage enabled.

## Best Possible Solution

Land #74185 or an equivalent focused fix that bounds `resolveProviderAuths` in the usage summary path, keeps `status --usage --json` parseable from non-interactive subprocesses, and adds a regression test where usage auth never settles.

## Reproduction Assessment

Yes. The issue includes a concrete `spawnSync('openclaw', ['status','--usage','--json'])` reproduction, and source inspection shows a current-main path where JSON output waits on unbounded provider auth resolution.

## Solution Assessment

Yes for triage. Keeping this issue open while #74185 or an equivalent narrow fix is reviewed is the right path; the timeout needs to wrap auth resolution itself, not only later usage fetches.
