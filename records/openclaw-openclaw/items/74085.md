## Summary

Keep open. The report is a fresh, concrete regression for `openclaw status --usage --json` from a non-TTY subprocess, and current main does not contain a clear fix for the status/usage JSON path. The command is routed and intended to support `--usage` with `--json`, but JSON output still waits for the provider usage path before writing stdout, and the timeout coverage starts after provider auth/plugin resolution rather than covering the whole machine-readable command path.

## Best Possible Solution

Keep this issue open and fix the machine-readable status path. `openclaw status --usage --json` should have a top-level non-interactive deadline/catch around provider usage collection, including auth/profile/plugin loading, and should still emit parseable JSON with provider-level error details instead of hanging or exiting with empty stdout. Add a non-TTY subprocess regression for the reporter's command and a unit test where usage auth/provider resolution never settles.
