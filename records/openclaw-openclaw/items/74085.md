## Summary

Keep open. Current main still awaits provider usage auth resolution before any status JSON is written, and the existing timeout only wraps later usage snapshot fetches. The linked open PR #74185 is the right implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue asks for `openclaw status --usage --json` to reliably emit machine-readable JSON from non-TTY subprocesses, including the OpenAI Codex OAuth usage path.

## Best Possible Solution

Bound provider usage auth resolution before status JSON waits on usage, emit parseable degraded JSON on auth timeout instead of empty output or a hang, and add a focused regression test where auth resolution never settles.

## Reproduction Assessment

Yes. The issue provides a concrete non-TTY `spawnSync` reproduction, and the current source path gives a high-confidence unit reproduction by making `resolveProviderAuths` never settle while `status --usage --json` waits for usage before writing JSON.

## Solution Assessment

Yes. Wrapping provider auth resolution at `loadProviderUsageSummary` with the existing timeout pattern is the narrowest maintainable fix because fetches are already bounded later and the hang occurs before that boundary.
