## Summary

Keep open. Current main still waits for provider usage auth resolution before writing `status --usage --json` output, and the only exact implementation candidate is the open, unmerged PR #74185, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that `openclaw status --usage --json` can hang or fail without stdout/stderr when invoked from a non-TTY subprocess after upgrading to 2026.4.26.

## Best Possible Solution

Keep the issue open and review/land PR #74185 or an equivalent narrow patch that bounds provider usage auth resolution, returns parseable degraded JSON instead of empty output or a hang, and adds a regression test where usage auth resolution never settles.

## Reproduction Assessment

Yes. The reporter supplied a concrete non-TTY `spawnSync` reproduction, and current main has a deterministic unit seam: a never-settling `resolveProviderAuths` blocks `loadProviderUsageSummary` before JSON output can be written.

## Solution Assessment

Yes, the narrow maintainable fix is to place the timeout/degraded fallback around auth resolution in `loadProviderUsageSummary` and cover that path with a never-settling-auth test. PR #74185 appears to target that boundary but still needs normal review/merge before this issue can close.
