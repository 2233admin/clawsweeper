## Summary

Current main still has the reported memory-wiki lint behavior: wikilink extraction runs regexes over raw page content, and broken-link validation indexes only extensionless paths and basenames. The discussion adds credible live-vault confirmation, and related #64696 overlaps on bridge/link noise but does not supersede this root-cause extractor and title-resolution bug.

## What This Changes

The issue requests that memory-wiki lint ignore literal `[[...]]` syntax in non-link Markdown contexts and resolve wikilinks against frontmatter titles and normalized page identities, not only slugged paths.

## Best Possible Solution

Fix memory-wiki lint at the plugin boundary: make `extractWikiLinks` Markdown-aware before storing `linkTargets`, then replace direct path-only broken-link matching with a deterministic normalized target index that includes page paths, basenames, frontmatter titles, ids, slugified titles, and aliases.

## Reproduction Assessment

Yes. A minimal vault page containing a fenced or inline literal `[[reply_to_current]]` plus a page linking `[[Example 配置清单]]` to a slugged file whose frontmatter title is `Example 配置清单` follows the current extractor and resolver code deterministically.

## Solution Assessment

Yes. The best fix is a generic Markdown-aware extractor plus title-aware normalized resolver; a tag-specific allowlist would cover only one bridge symptom and would not fix natural-title links or future literal wikilink examples.
