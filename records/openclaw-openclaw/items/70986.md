## Summary

Keep open: current main still has both reported memory-wiki lint behaviors, and the related open PR only addresses the resolver side. The extractor remains raw-regex based, while the lint resolver still compares link targets only to extensionless paths and basenames.

## What This Changes

The issue requests that memory-wiki lint ignore literal wikilink syntax in Markdown/code/transcripts and resolve wikilinks by frontmatter titles as well as path/basename forms.

## Best Possible Solution

Keep this as the canonical combined memory-wiki lint bug until both halves are fixed: a normalized resolver that accepts page titles and slug variants, plus a Markdown-aware extractor that ignores non-prose link examples, with focused regressions in the memory-wiki test files.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence reproduction path: a page containing fenced or inline literal `[[reply_to_current]]` still flows through the raw extractor, and a title-only link such as `[[Example 配置清单]]` misses the path-only lint resolver when the on-disk file uses a slugified basename.

## Solution Assessment

Yes. The root-cause extractor plus resolver fix is the narrow maintainable direction; tag-specific sanitization would only hide one bridge symptom, while the existing resolver PR covers only part of this issue.
