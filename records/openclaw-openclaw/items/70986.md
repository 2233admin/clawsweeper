## Summary

Keep open: current main still has both reported memory-wiki lint behaviors. The extractor still scans raw Markdown for `[[...]]`, and the lint resolver still accepts only extensionless paths and basenames, so #64696 only overlaps with the reply-tag symptom and does not supersede the frontmatter-title resolver bug.

## What This Changes

The issue requests a Markdown-aware memory-wiki wikilink extractor and a lint resolver that accepts frontmatter titles, aliases, ids, and normalized slug forms instead of only path-derived targets.

## Best Possible Solution

Fix this inside the bundled memory-wiki plugin: make link extraction Markdown-aware before `linkTargets` are recorded, then validate links against one deterministic normalized target index containing paths, basenames, titles, ids, aliases, and slugified title/alias forms, with focused regression tests and a short docs note for accepted link forms.

## Reproduction Assessment

Yes. Current main deterministically reproduces this with a vault page containing fenced or inline literal `[[reply_to_current]]`, plus a slugged source page whose frontmatter title is linked as `[[Example 配置清单]]`; both flow through raw extraction and path-only validation.

## Solution Assessment

Yes. A generic Markdown-aware extractor plus normalized title-aware resolver is the narrow maintainable fix; a tag-specific allowlist would only hide one bridge symptom and would not resolve natural-title wikilinks.
