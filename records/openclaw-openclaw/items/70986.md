## Summary

Keep this issue open. Current main still has both reported memory-wiki lint defects: wikilink extraction scans raw page text with regexes, including fenced/inline literal examples, and broken-link validation only accepts extensionless relative paths plus basenames, not frontmatter titles or normalized title/id keys.

## What This Changes

Fix memory-wiki lint so wikilink extraction is Markdown-aware and broken-link resolution accepts natural frontmatter-title targets instead of only slugified paths or basenames.

## Best Possible Solution

Keep this open and fix it in extensions/memory-wiki by making link extraction ignore Markdown non-link contexts, then replacing lint's direct path/basename check with a normalized target resolver that accepts page path, basename, title, slugified title, id, and future aliases where available. Add focused regression coverage for literal fenced/inline [[...]] text, bridged reply tags, and Unicode or special-character frontmatter-title links.

Normalize consistently with existing memory-wiki helpers, including slash normalization, optional `.md` stripping, whitespace/case handling, and Unicode-safe title behavior.
