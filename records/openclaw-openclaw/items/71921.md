## Summary

Keep open. Current main still preserves stale explicit Bedrock model rows without `input`, so affected Claude entries can remain text-only even after discovery would know they are image-capable; an open unmerged PR already claims the issue with a focused implementation path.

## What This Changes

Ensure Amazon Bedrock Claude Opus/Sonnet 4.x models carry image input capability so image attachments are not rejected as unsupported.

## Best Possible Solution

Land PR #72404 or a provider-owned replacement that refreshes or safely fills missing Bedrock input capability metadata after discovery succeeds, while preserving intentional explicit text-only overrides and covering foundation, regional, and global Claude ids.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an explicit/static `models.providers.amazon-bedrock.models` entry for a Claude 4.x Bedrock id with no `input`: current merge code can preserve it unchanged, and the image gate then treats it as not supporting images.

## Solution Assessment

No for the original registry-only framing. The durable fix should handle stale explicit/static catalog metadata in the merge or provider refresh path, not only mark current discovered registry entries.
