## Summary

Keep open. Current main still preserves the stale Bedrock explicit/static catalog path: discovery can produce `input: ["text", "image"]`, but explicit-only Bedrock rows without `input` are kept text-only downstream. The issue is also paired with open PR #72404 using closing syntax, so it should remain open until that PR or a replacement lands.

## What This Changes

The issue asks OpenClaw to mark Bedrock-hosted Claude Opus/Sonnet 4.x models as image-capable so image attachments are not rejected as unsupported.

## Best Possible Solution

Land PR #72404 or a provider-owned replacement that refreshes or inherits missing Bedrock `input` capability metadata for stale explicit/static rows after discovery succeeds, while preserving intentional explicit text-only overrides and covering foundation, regional, and global Claude ids.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a non-empty `models.providers.amazon-bedrock.models` list containing a Claude 4.x row with no `input`: current merge code preserves that row, model resolution defaults it to `["text"]`, and image support checks return false.

## Solution Assessment

Unclear. The original registry-only request is not the full durable fix because the comment thread identifies stale explicit catalog metadata; PR #72404 is a plausible narrow fix, but maintainers should decide whether it also needs a discovery refresh or dependency-staging change.
