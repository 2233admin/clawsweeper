## Summary

Keep open. Current main still accepts Telegram DM-level `tools` and `toolsBySender` config, and Telegram DM prompt overrides use the resolved `direct` entry, but mounted-tool filtering still goes through group/channel policy resolution that rejects `direct` session keys before policy lookup. The related PR #53530 remains open and unmerged, so this should stay tracked for maintainer review.

## What This Changes

Ensure Telegram DM `direct.<peer>.tools` and `toolsBySender` either enforce mounted tool policy per DM peer or are removed/clarified as unsupported config.

## Best Possible Solution

Route Telegram `direct[peer] ?? direct["*"]` policy through a shared direct-session mounted-tool resolver with account scoping, explicit-over-wildcard precedence, `toolsBySender`, thread/parent fallback, and fail-closed filtering. If maintainers decide DM tool policy is not supported, remove or deprecate the schema/type surface and document the supported alternative.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a Telegram DM session key such as `agent:main:telegram:direct:<peer>` is rejected by the group/channel-only session parser before mounted-tool policy lookup, while the same `direct` config is accepted and used for prompt overrides.

## Solution Assessment

Yes for the current triage path. Keeping this issue open while maintainers review, land, split, or replace PR #53530 is safer than closing it or queueing a parallel automated repair for an authorization-sensitive policy path.
