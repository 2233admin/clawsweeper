## Summary

Keep open. Current main still accepts and types Telegram `direct.<peer>.tools` / `toolsBySender`, and Telegram DM prompt handling consumes the same direct config, but the mounted-tool path still goes through group-policy resolution and rejects `direct` session keys before any direct policy can be found. Related PR #53530 is useful implementation context, but it is open and unmerged in the provided context, so it is not proof that current main fixes this issue.

## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` policy is intended to enforce mounted tools and, if so, to fix the runtime mismatch or clarify the config/docs surface.

## Best Possible Solution

Route Telegram's `direct[peer] ?? direct["*"]` config surface through a shared direct-session tool-policy resolver, preserving account scoping, explicit-over-wildcard precedence, `toolsBySender`, DM topic/thread fallback, and normal mounted-tool filtering. Add focused regression coverage for Telegram DM direct tools and update docs only if maintainers decide the accepted config surface should be narrowed instead.
