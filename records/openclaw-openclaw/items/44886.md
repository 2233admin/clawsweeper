## Summary

Keep open. Current main still accepts Telegram `direct.<peer>.tools`/`toolsBySender` and uses the same direct config for DM prompts, but mounted tool filtering still resolves only group/channel policy contexts. The related open PR #53530 is useful evidence that the shared direct-session resolver gap was being worked, but it is unmerged and does not by itself adapt Telegram's legacy `direct` surface.

## What This Changes

The issue asks whether Telegram per-DM `direct.<peer>.tools` is meant to enforce actual tool permissions and, if so, to fix that enforcement or clarify the config surface.

## Best Possible Solution

Adapt Telegram's legacy `channels.telegram.accounts.<account>.direct[peer] ?? direct["*"]` surface onto the shared direct-session tool-policy resolver, including `tools`, `toolsBySender`, explicit-over-wildcard precedence, account scoping, topic/thread fallback, normal DM runs, and native-command target sessions. Add focused Telegram regression tests and document the behavior if it remains supported.
