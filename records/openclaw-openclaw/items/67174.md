## Summary

Keep this PR open. Current main still does not implement Microsoft Teams `graphTenantId` / `MSTEAMS_GRAPH_TENANT_ID` support: Teams credentials, config schema/types, docs/plugin env metadata, monitor setup, and thread Graph token calls are all still tenantId-only. The PR remains the active implementation candidate, but it needs maintainer review because it bundles the graph-tenant change with broader webhook parsing, thread-cache/mention behavior, and runtime logging changes plus unresolved review/security concerns.

## Best Possible Solution

Keep the PR open as the active implementation candidate. Maintainers should ask for a rebase and scope split: land a narrow `graphTenantId` slice first, covering Teams config type/schema/manifest/docs, env fallback, secret and federated credential propagation, Graph token-provider selection in monitor and CLI Graph helpers, and focused tests. The broader webhook repair, thread-cache fallback, mention-only behavior, and runtime logging changes should either be separated or explicitly reviewed with the existing security and behavior comments resolved.

## Risks / Open Questions

- The PR is broader than its title: it mixes cross-tenant Graph token support with Teams webhook JSON repair, mention-only thread behavior, in-memory thread cache fallback, SDK/export adjustments, generated config metadata, changelog, and runtime logging changes.
- The proposed in-memory thread fallback touches authorization-sensitive context: review comments identify a risk that cached fallback messages could bypass group allowlist filtering when Graph is unavailable.
- The webhook parser change moves from `express.json()` to raw-body parsing plus repair logic on an internet-facing Teams endpoint, so it needs careful maintainer review and focused tests before merge.
- Current main still lacks the requested behavior, so closing this PR now would drop the only direct implementation candidate described in the provided context.
