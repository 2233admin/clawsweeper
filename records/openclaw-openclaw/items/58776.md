## Summary

Keep open. Current main has timeout-grace and orphan-recovery handling for aborted sub-agent runs, but it still does not expose the requested first-progress or typed startup-failed lifecycle signal; the existing public mappings continue to treat plain aborted lifecycle ends as timeouts.

## What This Changes

Add additive sub-agent lifecycle signals so parent runs can distinguish first progress, startup aborts, startup timeouts, empty startup output, and authentication failures from slow starts.

## Best Possible Solution

Add a generic lifecycle-progress seam for native embedded/subagent runs: emit one first-progress signal after the first qualifying visible assistant text or tool start, emit a typed startup-failed signal when the run terminates before first progress, and update Gateway wait handling, the subagent registry, SDK types/normalization, docs, and focused regression tests while preserving existing start/end/error compatibility.

## Reproduction Assessment

Yes. A focused unit reproduction can register a subagent run, emit lifecycle `{ phase: "end", aborted: true }` before any assistant/tool progress, and observe the registry/Gateway/SDK timeout path; the unavailable live corpus is not required to verify the current-main mapping gap.

## Solution Assessment

Yes. The requested additive lifecycle events are the narrowest maintainable direction if implemented generically in the embedded lifecycle and consumer normalization paths; avoid plugin-specific IDs, startup migrations, or policy changes in core.
