## Summary

Keep this issue open. Current main still has coarse task and Task Flow statuses, operational audit codes, raw error/summary text, and a plugin runtime DTO that mirrors those fields, but it does not provide a shared machine-readable blocker/failure taxonomy; related #64413 is narrower delivery-failure work and does not supersede the broader task taxonomy request.

## What This Changes

The issue requests a shared task blocker/failure taxonomy so task status, audits, summaries, and user-facing output can use normalized machine-readable classes.

## Best Possible Solution

Define an additive, version-safe task blocker/failure classification model in the task owner area, map known task, flow, ACP, cron, delivery, and cancellation conditions through shared helpers, expose optional fields through persisted records and runtime DTOs, then update audit, status, CLI, and docs surfaces to use the normalized classes while preserving existing status/error fields.

## Reproduction Assessment

Not applicable as a feature request rather than a bug. The high-confidence verification path is static inspection of current main: task models, Task Flow models, audits, DTOs, CLI, tests, and docs all show coarse statuses and raw prose without a shared taxonomy.

## Solution Assessment

Unclear because there is no proposed patch to evaluate. The requested direction is reasonable, but the best implementation needs a maintainer-approved additive taxonomy and compatibility plan rather than a narrow string-only mapping.
