## Summary

Keep this issue open. Current main loads local skills for runtime/status surfaces, but `skills.search`, the CLI search command, and the Control UI discovery flow remain ClawHub-only; there is still no `skills.register` API, `openclaw skills register <path>` command, or local-result contract for registering/searching local skill directories.

## Best Possible Solution

Keep this issue open. The best implementation is a core-owned skill discovery change: either extend `skills.search`/`openclaw skills search` to return source-aware local and ClawHub results backed by existing local skill loaders, or add an explicit `skills.register` / `openclaw skills register <path>` flow. It should update the gateway protocol, CLI, Control UI, docs, and tests while keeping ClawHub install/update semantics distinct from local skill paths.

## Risks / Open Questions

- A combined local-plus-ClawHub search needs a source discriminator so local skill names/paths are not treated as ClawHub slugs for install/update/detail flows.
- A local register or auto-index flow should preserve current skill precedence, agent allowlists, realpath confinement, symlink/path trust boundaries, and installer safety scanning semantics.
