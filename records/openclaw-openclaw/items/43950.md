## Summary

Current main still does not implement local skill registration into the skill_search index. The existing local skill loader supports prompt/runtime/status/list/info/check flows, but the gateway, CLI, and Control UI search paths remain ClawHub-only and there is no register command/API or source-aware local search result contract.

## Best Possible Solution

Keep this issue open. The best implementation is a core-owned skill discovery change that either extends skills.search/openclaw skills search with source-aware local or combined results backed by the existing local skill loaders, or adds an explicit skills.register / openclaw skills register <path> flow. It should update the gateway protocol, CLI, Control UI, docs, and tests while keeping current ClawHub install/update semantics distinct from local skill paths.

## Risks / Open Questions

- A combined local-plus-ClawHub search needs a source discriminator so local skill identifiers are not treated as ClawHub slugs for install/update flows.
- A register/local-index flow would need to preserve local skill precedence, agent allowlists, eligibility filtering, symlink/path confinement, and local path trust boundaries.
