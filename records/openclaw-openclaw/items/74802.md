## Summary

Keep this PR open. It addresses a real current-main packaging bug for local memory embeddings, but the patch still only detects the literal `memorySearch.provider: "local"` case and misses supported local-selection paths such as auto-local model selection and custom provider ids whose `models.providers.<id>.api` resolves to `local`.

## What This Changes

The PR adds a config-gated `node-llama-cpp@3.18.1` runtime dependency for `memory-core`, updates the local embedding setup hint to use `openclaw doctor --fix`, and adds bundled runtime-deps tests.

## Best Possible Solution

Keep the managed runtime-deps approach and keep `node-llama-cpp` out of root package dependencies, but make the selector follow the same memory provider resolution contract as runtime code: literal `local`, auto-local when a configured local model file is eligible, configured provider ids whose `api` resolves to `local`, and local fallback if supported. Add focused runtime-deps tests for each path before merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists by configuring a supported local selection path that is not the literal `provider: "local"` path, then scanning or ensuring bundled runtime deps; this PR still omits `node-llama-cpp` for those cases because the detector only checks the raw provider string.

## Solution Assessment

No, not yet. The dependency should be installed through the managed bundled runtime-deps plan, but the proposed detector should align with the runtime memory provider resolver instead of duplicating only one string comparison.
