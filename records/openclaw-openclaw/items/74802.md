## Summary

Keep this PR open. The underlying bug is real and the managed runtime-deps boundary is the right place to fix it, but the proposed detector only covers the literal `local` provider string and still misses supported paths that can select the local adapter.

## What This Changes

The PR adds config-gated `node-llama-cpp@3.18.1` to bundled `memory-core` runtime dependency staging for literal local memory search, updates the local setup hint, and adds runtime-deps tests.

## Best Possible Solution

Keep `node-llama-cpp` out of root package dependencies, but update the runtime-deps selector to match the existing memory provider resolution contract: literal `local`, auto mode where a configured local model path makes local eligible, and configured provider ids whose `api` resolves to the local adapter. Add focused tests for each path before merge.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure local memory search, then scan or ensure bundled runtime deps for `memory-core`; current main imports `node-llama-cpp` by bare specifier but only retains declared/mirrored deps, and this PR still misses non-literal local selections.

## Solution Assessment

No. The managed runtime-deps approach is the narrow maintainable solution, but the proposed implementation should align with the resolver instead of checking only `memorySearch.provider === "local"`.
