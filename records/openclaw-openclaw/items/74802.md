## Summary

Keep this PR open for normal maintainer review. Current main still has the reported literal-local gap, and the PR is a targeted managed runtime-deps fix, but the detector is narrower than the documented/runtime provider resolution paths for local embeddings.

## What This Changes

The PR adds config-gated `node-llama-cpp@3.18.1` to bundled `memory-core` runtime dependency selection when local memory search is configured, updates the missing-runtime hint, and adds runtime-deps tests.

## Best Possible Solution

Keep the managed runtime-deps approach and keep `node-llama-cpp` out of root package dependencies, but make the selector match the memory provider resolver's supported local paths before merge. The final patch should cover literal local, auto-local where the configured local model path makes local eligible, and configured provider ids that resolve to the local adapter, with focused runtime-deps tests for each path.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure local memory search, then scan or ensure bundled runtime deps for `memory-core`; current main imports `node-llama-cpp` by bare specifier but only retains declared memory-core deps and mirrored root deps, so the managed runtime can omit `node-llama-cpp`.

## Solution Assessment

No, not yet. The managed runtime-deps boundary is the right solution, but the proposed detector should align with the existing memory provider resolution contract instead of matching only the literal `local` provider string.
