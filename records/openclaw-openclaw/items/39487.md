## Summary

Keep open. Current main still does not implement this PR's intended Feishu liveness/status plumbing: the Feishu channel start path only records the webhook port, the monitor/account/transport types do not carry a status callback, inbound handlers cannot publish `lastEventAt`, and transport lifecycle paths cannot publish connected/disconnected or transport activity. The PR head also has unresolved functional review concerns around premature `connected: true` and early-abort handling, while related implementation candidates #39489 and #41940 remain open and unmerged.

## Best Possible Solution

Keep this PR open for maintainer follow-up. The best path is to choose a canonical implementation among this PR, #39489, and #41940; thread a typed status callback from Feishu `startAccount` through `monitorFeishuProvider`, `monitorSingleAccount`, inbound handlers, and transport lifecycle; report `connected` only after WebSocket start or webhook listen succeeds; report disconnected/error state on abort, startup failure, runtime failure, and early-abort paths; update `lastEventAt` for app-level inbound events; publish `lastTransportActivityAt` only for real transport activity if Feishu should participate in stale-socket health checks; and add focused lifecycle, inbound-event, and early-abort regression tests.
