## Summary

Keep open. Current main still deliberately treats Feishu broadcast-only `@all`/`@_all` as not mentioning the bot, and both tests and docs encode that behavior. The report asks maintainers to reconsider that policy and move suppression to prompt/config context; that is not implemented on main, and the related open Feishu issue is broader rather than a narrower canonical replacement.

## What This Changes

Request to stop dropping Feishu broadcast-only `@all`/`@_all` mentions at the mention-gating layer and instead expose or suppress them via agent prompt or configuration policy.

## Best Possible Solution

Represent Feishu broadcast mentions as a distinct channel fact that can be included in agent context without automatically satisfying bot-mention gates, then let maintainers choose prompt/config semantics and update the Feishu tests and docs in the same change.

## Reproduction Assessment

Yes. A focused unit fixture with a group message containing only `@_all` or Feishu mention metadata `{ key: "@_all", id.open_id: "all" }` reproduces `mentionedBot=false`, and the group handler then drops it when `requireMention` is true.

## Solution Assessment

Unclear. The requested revert is not obviously the best fix because it may reopen the broadcast false-positive that PR #72658 addressed; a safer design is to preserve broadcast mention context separately from the boolean bot-mention gate.
