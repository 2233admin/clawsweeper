## Summary

Keep open. Current main intentionally implements the Feishu broadcast-mention behavior the reporter disputes: broadcast-only `@all`/`@_all` does not satisfy the bot-mention gate and can be dropped before dispatch when `requireMention` is true. The report is specific and reproducible from source, but it is a channel policy/product question rather than obsolete cleanup; a direct revert could reintroduce the closed noisy-reply behavior from #37706.

## What This Changes

The issue asks OpenClaw to stop suppressing Feishu broadcast-only `@all`/`@_all` mentions at the channel mention gate and instead expose them to the agent prompt or make suppression configurable.

## Best Possible Solution

Represent Feishu broadcast mentions as a distinct channel fact that can be included in dispatched agent context, while keeping a separate explicit policy for whether broadcast-only messages satisfy mention gating or are suppressed by prompt/configuration.

## Reproduction Assessment

Yes. A focused source-level fixture with a group message containing only `@_all` text, mention metadata `{ key: "@_all", id.open_id: "all" }`, or post `user_id: "all"` produces `mentionedBot=false`; with `requireMention=true`, the group handler returns before agent dispatch.

## Solution Assessment

Unclear. The requested revert is not demonstrably the safest fix because it can restore the #37706 false-positive; a narrower maintainable solution would separate broadcast context visibility from the boolean bot-mention gate.
