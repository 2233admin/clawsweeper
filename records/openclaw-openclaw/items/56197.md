## Summary

Keep open. Current main has generic embedded-run auth retry support only when a provider supplies runtime auth state, but the ZAI provider appears API-key based and does not register a runtime auth refresh hook. The embedded error formatting path also still preserves a generic `HTTP 401: "Invalid token"` message for final error payloads, so the reported Feishu user-visible failure is not proved fixed or superseded.

## Best Possible Solution

Keep this issue open and fix it at the provider/embedded-runner boundary: add provider-owned ZAI/AutoGLM runtime auth preparation and refresh if the proxy token is refreshable, then add a focused regression test that a 401 invalid-token response retries before replying. Independently, update embedded assistant error formatting so a generic auth 401 is converted to a user-safe auth failure instead of sending the raw HTTP text through Feishu.
