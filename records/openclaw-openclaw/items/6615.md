## Summary

Keep this issue open. Current main still has no native exec-approvals denylist mode, denylist array, `ask: "on-match"`, or deny-pattern evaluation path; source, protocol schema, Control UI, tests, and docs all remain allowlist-focused. The SafeClaw and CRE links plus the closed implementation PRs are useful prior art, but they do not replace a merged native policy change, and the latest maintainer comment explicitly kept this open for follow-up.

## What This Changes

Add native exec-approvals denylist or deny-pattern support so operators can allow most host exec commands while prompting or blocking specific dangerous command and argument patterns.

## Best Possible Solution

Define an additive, version-safe native exec approvals policy that evaluates deny patterns before broad allow decisions, supports command and argument matching consistently across gateway and node exec paths, exposes configuration in CLI and Control UI, documents the security model, and either folds #41140 into this design or makes #41140 the canonical narrower tracker.
