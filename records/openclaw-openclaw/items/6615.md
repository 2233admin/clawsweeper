## Summary

Keep this issue open. Current main still has no native exec-approvals denylist mode, denylist array, `ask: "on-match"`, or deny-pattern evaluation in the approval policy, protocol schema, CLI/UI, tests, or docs. The CRE plugin and SafeClaw links in the discussion are useful prior art/workarounds, and #41140 is a related open tracker, but the closed implementation PRs did not merge and do not make current main complete.

## What This Changes

Request native exec-approvals denylist support that allows broad host exec while prompting or blocking matching dangerous command and argument patterns.

## Best Possible Solution

Keep this open for maintainer-owned native deny-pattern design. The best path is an additive exec-approvals policy model that defines denylist matching semantics before allowlist/full behavior, supports command/argument matching consistently across gateway and node exec paths, updates the protocol schema and validators, exposes CLI and Control UI editing, documents the security model, and decides whether #41140 should become the canonical narrower tracker.
