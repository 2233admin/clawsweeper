## Summary

Keep this PR open. Current main still accepts credentialed cron webhook URLs because `normalizeHttpWebhookUrl` only trims, parses, and checks `http(s)`, so the intended hardening is not implemented. The PR is a plausible focused fix, but its submitted diff needs revision because the existing invalid-webhook path can log raw `delivery.to`, legacy `notify`/`cron.webhook` migration is still verbatim, and there is no regression coverage for URL userinfo cases.

## Best Possible Solution

Keep the PR open as a valid hardening candidate, but require a revised patch that rejects URL userinfo at the shared cron webhook normalization boundary, redacts invalid stored webhook targets before logging, handles or warns on legacy `cron.webhook` doctor migration, and adds focused regression tests for primary delivery plus failure destinations.

## Risks / Open Questions

- As submitted, rejecting credentialed URLs can route existing stored jobs into an invalid-webhook warning that logs raw `delivery.to`, which may expose the same embedded secret the PR intends to protect.
- Legacy `notify: true` jobs using global `cron.webhook` need an explicit migration/warning path; current doctor migration copies that URL verbatim into per-job delivery.
- Without tests for `https://user:pass@host`, username-only, password/userinfo edge cases, and failure destinations, this security-sensitive validation can regress silently.
