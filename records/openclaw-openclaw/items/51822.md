## Summary

Keep this PR open. Current main still accepts cron webhook URLs with embedded userinfo, so the hardening is not implemented or obsolete. The PR is a focused security candidate, but the submitted one-file guard is not yet safe to merge because it can make stored credential-bearing webhook URLs invalid and then log them raw, silently disable legacy notify+cron.webhook deliveries, and let doctor migrate credential-bearing legacy webhooks without guidance.

## Best Possible Solution

Keep this PR open as a valid but incomplete cron webhook hardening candidate. The best implementation is to reject URL userinfo in the shared cron webhook normalizer, redact every invalid/skipped webhook log path with the existing sensitive-URL helpers, add explicit warning or migration behavior for legacy `notify: true` plus `cron.webhook`, prevent doctor from migrating credential-bearing legacy webhooks without guidance, and add targeted regression tests across service validation, failure destinations, the agent cron tool, runtime notifications, and doctor migration. Keep #55365 separate because it covers the browser tool URL surface, not cron webhooks.

## Risks / Open Questions

- Merging the PR as-is can cause existing stored credential-bearing webhook URLs to become invalid and then be emitted raw in the skipped-delivery warning.
- Legacy `notify: true` jobs that depend on `cron.webhook` with URL userinfo can stop delivering without an invalid-target warning.
- `openclaw doctor --fix` can still migrate a credential-bearing legacy `cron.webhook` value into `delivery.to`, producing an invalid migrated job without user guidance.
- Without regression tests for userinfo rejection, redaction, runtime stored-job behavior, failure destinations, the agent cron tool, and doctor migration, this security behavior can regress quietly.
