## Summary

Keep this issue open. Current main still excludes direct Amazon Nova Bedrock model IDs from cache-retention eligibility and forces them through the Bedrock no-cache wrapper, while AWS now documents Nova prompt caching support for `system` and `messages`; upstream pi-ai also still treats Nova as automatic-only rather than injecting explicit Nova cache points. ([raw.githubusercontent.com](https://raw.githubusercontent.com/badlogic/pi-mono/main/packages/ai/src/providers/amazon-bedrock.ts))

## What This Changes

Request native Amazon Nova prompt caching support for Bedrock by allowing valid cache retention and using Nova-compatible cache metadata.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin: recognize AWS-supported direct, regional, and global Nova model IDs; preserve `cacheRetention: "none"`; treat Nova `short` as 5-minute explicit checkpoints; handle `long` conservatively unless AWS supports extended TTL for Nova; inject cache points only in `system` and `messages`; and update tests, docs, and changelog together.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: direct Nova IDs are rejected by `resolveCacheRetention`, and the Bedrock provider wrapper then forwards them with `cacheRetention: "none"`; existing tests assert both behaviors.

## Solution Assessment

Yes. A focused Amazon Bedrock plugin repair is the narrowest maintainable path because the remaining gap is provider-family semantics and AWS cache-point placement, not a generic core cache contract change.

## Repair Work Prompt

Fix the current Amazon Bedrock Nova prompt-caching gap tracked by https://github.com/openclaw/openclaw/issues/29707. Current main excludes direct `amazon.nova-*` model IDs from `resolveCacheRetention` and then routes them through `createBedrockNoCacheWrapper`, so user-configured Nova cache settings cannot produce explicit Bedrock cache checkpoints. Implement this in the bundled `amazon-bedrock` plugin/provider-owned cache helpers, not as a broad core policy change unless a small generic seam is clearly required.

Expected boundary:
- Recognize AWS-supported Nova text model IDs, including direct, region-prefixed, and global refs such as `amazon.nova-micro-v1:0`, `amazon.nova-lite-v1:0`, `amazon.nova-pro-v1:0`, `amazon.nova-premier-v1:0`, and `amazon.nova-2-lite-v1:0`.
- Preserve `cacheRetention: "none"` as an opt-out.
- For Nova, inject Bedrock Converse `cachePoint` checkpoints only in locations AWS supports for Nova: `system` and `messages`.
- Do not send Anthropic `cache_control` metadata to Nova.
- Do not blindly map `cacheRetention: "long"` to `ttl: "1h"` for Nova; AWS docs currently list Nova TTL as 5 minutes, so either downgrade long to Nova-compatible short semantics or explicitly validate a current AWS contract before adding 1-hour TTL.
- Keep unsupported non-Anthropic Bedrock models on the no-cache path so #20815 does not regress.
- Keep app inference profile Claude behavior from #69953 intact.

Likely files: `extensions/amazon-bedrock/register.sync.runtime.ts`, `extensions/amazon-bedrock/index.test.ts`, `src/agents/pi-embedded-runner/prompt-cache-retention.ts`, `src/agents/pi-embedded-runner/anthropic-family-cache-semantics.ts`, `src/agents/pi-embedded-runner/bedrock-stream-wrappers.ts`, `src/agents/pi-embedded-runner/extra-params.cache-retention-default.test.ts`, `src/config/config.pruning-defaults.test.ts`, `docs/reference/prompt-caching.md`, `docs/providers/bedrock.md`, and `CHANGELOG.md`.

Update docs to describe Nova support and TTL limits. Add a single-line changelog entry if behavior changes, crediting the reporter as `Thanks @tribecar-shurn`. Do not change unrelated provider caching behavior or dependency pins.

Validation: run the targeted tests and formatter checks listed in the ClawSweeper work validation, then `pnpm check:changed` in Testbox before handoff if code/docs/changelog changed.
