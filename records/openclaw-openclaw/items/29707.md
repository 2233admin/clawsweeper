## Summary

Keep this issue open. Current main still treats direct Amazon Nova Bedrock model IDs as non-cache-eligible, forces them through the no-cache Bedrock wrapper, and documents that non-Anthropic Bedrock models are intentionally forced to `cacheRetention: "none"`; AWS Bedrock now documents Nova prompt-caching support with `system` and `messages` checkpoints, so the requested bundled-provider repair remains valid. citeturn0search0

## What This Changes

Allow Amazon Nova Bedrock models to use native prompt caching by recognizing cache-eligible Nova IDs, forwarding `cacheRetention`, and using Nova-compatible Bedrock `cachePoint` payloads.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin: recognize supported direct, region-prefixed, and global Nova model IDs; preserve explicit valid cache retention; inject only Bedrock/Nova-compatible `cachePoint` checkpoints on supported fields; keep unsupported non-Anthropic Bedrock models on the no-cache path; and update tests, public docs, and changelog together.

Current main excludes direct Nova IDs from `resolveCacheRetention`, wraps direct Nova streams with `createBedrockNoCacheWrapper`, and documents/tests that non-Anthropic Bedrock models are forced to `cacheRetention: "none"`. Implement only the supported Nova path: recognize AWS-supported Nova direct, region-prefixed, and global model IDs; preserve explicit valid `cacheRetention`; inject only Bedrock/Nova-compatible `cachePoint` checkpoints for supported `system` and `messages` fields; avoid blindly adding `ttl: "1h"` for Nova unless the current AWS contract proves it is supported; keep unsupported non-Anthropic Bedrock models on the no-cache path. Update focused unit tests, prompt-caching and Bedrock provider docs, and CHANGELOG.md with user-facing attribution, likely `Thanks @tribecar-shurn` and `Thanks @ProFire` if the #20815 crash path is addressed.
