## Summary

Keep this issue open. Current main still excludes direct Amazon Nova Bedrock model IDs from cache-retention eligibility and forces them through the Bedrock no-cache wrapper, while the current AWS Bedrock prompt-caching contract documents Nova prompt caching with cache checkpoints for system/messages. This is a valid narrow Amazon Bedrock provider repair, not cleanup to close.

## What This Changes

Request native Amazon Nova prompt-caching support in the bundled Amazon Bedrock provider by allowing supported Nova model IDs to use Bedrock-compatible cache settings instead of the current forced no-cache path.

## Best Possible Solution

Add provider-owned Nova prompt-cache support in the bundled Amazon Bedrock plugin: recognize supported direct, region-prefixed, global, and relevant profile Nova IDs; preserve valid explicit cacheRetention settings; add Bedrock/Nova-compatible cachePoint checkpoints only where AWS accepts them; keep unsupported non-Anthropic Bedrock models on the no-cache path; and update tests, public docs, and changelog together.

## Repair Work Prompt

Fix native Amazon Nova prompt-caching support for the bundled Amazon Bedrock provider. Current main still treats direct Nova model IDs such as `amazon.nova-micro-v1:0`, `amazon.nova-lite-v1:0`, `amazon.nova-pro-v1:0`, `amazon.nova-premier-v1:0`, and `amazon.nova-2-lite-v1:0` as non-cache-eligible and forces `cacheRetention: "none"`. Implement provider-owned Nova support without changing unrelated providers: add a dedicated helper for supported Nova family IDs, including direct IDs and region/global-prefixed Bedrock IDs; preserve explicit `cacheRetention` where valid; add Bedrock Converse `cachePoint` checkpoints for Nova only in supported `system` and `messages` positions; do not send Anthropic `cache_control` metadata to Nova; keep unsupported non-Anthropic Bedrock models on the no-cache path. Verify AWS Bedrock Nova TTL semantics before mapping `cacheRetention: "long"`; do not add `ttl: "1h"` for Nova unless current AWS docs or SDK behavior explicitly supports it. Update focused tests in the Amazon Bedrock plugin and cache-retention helpers, update `docs/reference/prompt-caching.md` and `docs/providers/bedrock.md` as needed, and add a user-facing changelog fix with reporter/contributor attribution. Do not broaden core with bundled-provider IDs unless a generic seam is strictly needed.
