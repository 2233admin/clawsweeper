## Summary

Keep this issue open. Current main still treats direct Amazon Nova Bedrock model IDs as non-Anthropic: explicit cacheRetention is dropped before streaming, the Amazon Bedrock plugin wraps direct Nova streams with cacheRetention: "none", tests assert that behavior, and docs still say non-Anthropic Bedrock models are forced off prompt caching. AWS Bedrock docs now list Amazon Nova prompt caching support on system/messages, so this remains a real bundled amazon-bedrock provider gap rather than cleanup.

## Best Possible Solution

Implement a focused bundled amazon-bedrock plugin/runtime fix: recognize supported Amazon Nova direct and region-prefixed model IDs as cache-eligible, preserve explicit cacheRetention for Nova, clamp or map Nova retention so unsupported 1-hour ttl is not sent, inject Bedrock/Nova-compatible cachePoint blocks on system and the last user message, keep other non-Anthropic Bedrock models forced to no-cache, and update tests, prompt-caching docs, and changelog.

## Clownfish Work Prompt

Implement a focused fix for native Amazon Nova prompt caching on Bedrock. Current main drops explicit cacheRetention for direct Nova IDs and wraps direct Nova streams with cacheRetention: "none". Add a Nova-family eligibility path for supported direct and region-prefixed Bedrock model IDs such as amazon.nova-micro-v1:0, amazon.nova-lite-v1:0, amazon.nova-pro-v1:0, amazon.nova-premier-v1:0, amazon.nova-2-lite-v1:0, and region/system prefixes like us.amazon.nova-* or global.amazon.nova-* if those are valid in existing Bedrock model-ref handling. Preserve the existing no-cache wrapper for unsupported non-Anthropic/non-Nova Bedrock models such as Mistral.

Expected implementation boundary: extensions/amazon-bedrock/register.sync.runtime.ts should recognize Nova as cache-eligible and inject Bedrock Converse cachePoint blocks on system and the last user message using the existing payload-patch pattern where appropriate. src/agents/pi-embedded-runner/prompt-cache-retention.ts and/or anthropic-family-cache-semantics.ts should preserve explicit cacheRetention for Nova Bedrock models without treating Nova as Anthropic. Do not send ttl: "1h" to Nova unless the AWS docs/source currently prove Nova supports 1-hour TTL; if cacheRetention: "long" is configured for Nova, either clamp to the default 5-minute Bedrock cache point or document/test the chosen conservative behavior. Do not move provider-specific Nova policy into generic core except for a small generic seam if truly needed.

Update tests in extensions/amazon-bedrock/index.test.ts and src/agents/pi-embedded-runner/extra-params.cache-retention-default.test.ts to cover direct Nova no longer forced to none, explicit retention preserved, cachePoint injection on system/messages, long-retention TTL safety, region-prefixed Nova IDs, and non-Nova non-Anthropic models still forced to no-cache. Update docs/reference/prompt-caching.md and add a user-facing changelog entry with reporter/contributor credit. Do not change unrelated provider caching behavior, dependency versions, or bundled provider discovery semantics.

## Risks / Open Questions

- Nova's Bedrock prompt-caching contract supports the default 5-minute cache, while current makeCachePoint maps cacheRetention: "long" to ttl: "1h"; a fix must not send unsupported Nova 1-hour TTL unless AWS adds that support.
- A blanket removal of createBedrockNoCacheWrapper would likely regress unsupported non-Anthropic Bedrock models such as Mistral and revive #20815-style extraneous cachePoint failures.
- Direct Nova support should use an explicit Nova allowlist/family check and Bedrock-compatible cachePoint injection on system/messages, while preserving existing Claude and application-inference-profile behavior.
