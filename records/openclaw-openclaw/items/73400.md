## Summary

Keep this PR open, but do not merge this revision as-is. Current main still has the silent-reply classification bug, and the PR is aimed at the right area, but the latest patch still treats a generic parsed thread suffix as internal provenance and leaves the group/channel ordering inconsistent with the stated invariant.

## What This Changes

The PR imports `parseThreadSessionSuffix` into silent-reply classification, treats parsed `:thread:` session suffixes as internal before direct/dm handling, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a refined narrow fix that keeps trusted generated internal thread sessions silent without letting arbitrary caller-shaped direct session keys opt out of direct silent-reply rewrites, with explicit coverage for direct-thread, cron/internal-thread, group/channel-thread, and malformed suffix cases.

## Reproduction Assessment

Yes. On current main, `classifySilentReplyConversationType({ sessionKey: "agent:main:telegram:direct:123:thread:99" })` follows the direct branch, and `createOutboundPayloadPlan` rewrites direct `NO_REPLY` payloads under default settings.

## Solution Assessment

No. The PR is close in scope, but using the shared parser result as the trust signal is not the narrowest maintainable fix; the safer path is a trusted-thread predicate or explicit metadata plus a clear decision for group/channel thread semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Validate thread provenance before returning internal:** `src/shared/silent-reply-policy.ts:81-83`
  - body: The new branch treats any non-empty `parseThreadSessionSuffix()` result as internal. That parser accepts any non-empty final `:thread:` tail, and `sessions.create` preserves explicit caller-shaped keys, so a direct key ending in `:thread:x` can skip the direct silent-reply rewrite defaults. Gate this on generated/internal thread metadata or a stricter trusted shape before returning `internal`.
  - confidence: 0.86
- **[P3] Classify trusted threads before group policy:** `src/shared/silent-reply-policy.ts:72-73`
  - body: The PR says threaded sessions are internal regardless of parent surface, but the group/channel branch still returns before the new thread check. If group and internal silent-reply settings diverge, a `...:group:...:thread:...` key will still take group behavior, so either move the trusted-thread predicate above group/channel or narrow the stated behavior and add coverage.
  - confidence: 0.72
