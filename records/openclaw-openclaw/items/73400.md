## Summary

Keep open. The PR addresses a real current-main bug, but the current patch is not ready to merge because the new `:thread:` predicate uses a generic suffix parser as internal provenance and can make caller-shaped or user-facing direct threaded sessions bypass the direct silent-reply rewrite path.

## What This Changes

The PR imports `parseThreadSessionSuffix`, classifies parsed `:thread:` session keys as internal before the direct/dm branch, adds direct-thread and cron-thread regression tests, and adds a changelog entry.

## Best Possible Solution

Land a refined narrow fix that marks only trusted generated internal thread sessions, such as cron, ACP, subagent, or explicit internal conversation metadata, as internal; direct user-facing threaded sessions should keep direct policy unless a trusted internal producer is proven, with tests for internal direct threads, user-facing direct threads, group/channel threads with divergent policy, and malformed suffixes.

## Reproduction Assessment

Yes. Source inspection is enough to reproduce the current-main bug: `classifySilentReplyConversationType({ sessionKey: "agent:main:telegram:direct:123:thread:99" })` follows the direct branch, and `createOutboundPayloadPlan` rewrites a direct `NO_REPLY` by default; tests were not run because this review is read-only.

## Solution Assessment

No. The PR is close in scope, but using a generic parsed `:thread:` suffix as proof of internal provenance is not the narrowest maintainable fix; a safer approach should gate on trusted generated thread shapes or explicit conversation metadata.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Validate thread provenance before returning internal:** `src/shared/silent-reply-policy.ts:81-83`
  - body: The new branch returns `internal` whenever `parseThreadSessionSuffix()` yields a `threadId`, but that parser accepts any final `:thread:` tail and explicit/session-route keys can preserve that shape. A caller-shaped or user-facing direct thread key can therefore bypass the direct silent-reply rewrite path; gate this on trusted internal producers or explicit metadata before returning `internal`.
  - confidence: 0.88
- **[P3] Resolve group-thread policy ordering:** `src/shared/silent-reply-policy.ts:81-83`
  - body: The thread check is inserted after the `:group:`/`:channel:` return, so channel-thread keys still classify as `group` even though the PR says threaded sessions are internal regardless of parent surface. If `group` and `internal` policies diverge, this keeps using group behavior; move the trusted-thread predicate above group/channel or narrow and test the intended semantics.
  - confidence: 0.74
