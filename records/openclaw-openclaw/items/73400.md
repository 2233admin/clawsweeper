## Summary

Keep this PR open, but do not merge this revision as-is. The underlying current-main bug is real, but the patch still uses a generic `:thread:` suffix parse as proof of internal provenance and leaves group/channel-thread ordering inconsistent with the PR's stated invariant.

## What This Changes

The PR imports `parseThreadSessionSuffix`, classifies parsed thread-suffixed silent-reply session keys as internal before direct/dm classification, adds direct/cron thread regression tests, and adds a changelog entry.

## Best Possible Solution

Land a refined narrow fix that keeps trusted generated internal thread sessions silent without allowing arbitrary caller-shaped direct session keys to opt out of direct silent-reply rewrites, with explicit coverage for direct-thread, cron/internal-thread, group/channel-thread, and malformed suffix cases.

## Reproduction Assessment

Yes. Source inspection on current main shows `agent:main:telegram:direct:123:thread:99` reaches the direct branch, and the outbound planner rewrites direct `NO_REPLY` payloads under default settings; tests were not run because this review was read-only.

## Solution Assessment

No. The direction is right, but the current patch treats the shared parser result as a trust signal and does not fully implement the stated `:thread:` ordering invariant; a trusted-thread predicate or explicit metadata is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Gate thread classification on trusted session provenance:** `src/shared/silent-reply-policy.ts:82-83`
  - body: The new branch treats any non-empty `parseThreadSessionSuffix()` result as internal. That parser uses the last `:thread:` marker and accepts any remaining tail, while `sessions.create` can preserve caller-shaped keys, so a direct key ending in `:thread:x` can bypass the direct `NO_REPLY` rewrite defaults. Restrict this to generated/internal thread shapes or explicit metadata before returning `internal`.
  - confidence: 0.86
- **[P3] Classify trusted threads before group policy:** `src/shared/silent-reply-policy.ts:72-73`
  - body: The PR says threaded sessions should be internal regardless of parent surface, but the group/channel branch still returns before the new thread check. If group and internal silent-reply settings diverge, a `...:group:...:thread:...` key will still take group behavior, so move the trusted-thread predicate above group/channel or narrow the stated behavior and add coverage.
  - confidence: 0.72
