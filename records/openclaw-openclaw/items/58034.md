## Summary

Keep this PR open. Current main contains shared env-proxy machinery, but it still does not implement the PR's central user-facing change: a default-off `tools.web.fetch.useTrustedEnvProxy` config field wired into built-in `web_fetch`. #59007 merged the narrower DNS-before-proxy ordering fix, but current main still lacks the explicit web_fetch opt-in, schema/docs surface, and trusted-env-proxy hostname-policy hardening proposed here.

## Best Possible Solution

Keep this PR open as the canonical implementation candidate for proxy-only built-in `web_fetch` support. The best path is maintainer review of the latest branch, preserving strict default behavior while landing the default-off `tools.web.fetch.useTrustedEnvProxy` config field from schema/docs through `createWebFetchTool` into `fetchWithWebToolsNetworkGuard`, with hostname-policy checks before trusted proxy dispatch and regression tests for default strict mode, matching proxy env vars, no matching proxy, NO_PROXY fallback, and dangerous hostnames.

## Risks / Open Questions

- This changes a security-sensitive `web_fetch` network boundary: when enabled, post-DNS IP pinning is delegated to an operator-controlled HTTP(S) proxy, so maintainers should review the trust model, option naming, docs, and regression coverage before merge.
- Provider fallback behavior, especially Firecrawl web_fetch fallback API calls, is intentionally left as a scope boundary in the PR discussion and should be decided explicitly as a follow-up or documented non-goal.
