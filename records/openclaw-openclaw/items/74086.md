## Summary

Keep open. The report is current, concrete, and still matches current main behavior: Telegram polling still waits for `deleteWebhook` cleanup before starting, recoverable cleanup failures still retry instead of checking whether the webhook is already absent, and native command sync still calls `deleteMyCommands` then `setMyCommands`. Current main has adjacent mitigations for Windows plugin loading and `apiRoot` 404 diagnosis, but those do not prove the reported Windows/npm 2026.4.24+ Telegram regression is fixed.

## Best Possible Solution

Keep this issue open as the broader Windows Telegram regression tracker. The best path is to review or land #47384, or an equivalent Telegram-owned fallback that verifies an already-empty webhook before blocking polling on recoverable `deleteWebhook` failures, then run a Windows/npm package smoke against the reporter's 2026.4.23 vs 2026.4.24+ scenario, including command sync and `bot-native-commands.runtime` loading. If diagnostics show a full `/bot<TOKEN>` `apiRoot`, route that through `openclaw doctor --fix`, but do not treat it as the only root cause without reproducing the release regression.
