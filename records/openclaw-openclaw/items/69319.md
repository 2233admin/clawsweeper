## Summary

Current main still does not register or implement `skills.uninstall`, so this PR is not obsolete. The PR is a plausible implementation candidate, but it should stay open for rework because the proposed backend can delete an existing skill directory without first proving ClawHub provenance, and the handler diff shown in the PR is stale against current `context.getRuntimeConfig()` Gateway handler patterns.

## Best Possible Solution

Keep this PR open for contributor or maintainer rework. The narrow target should be a `skills.uninstall` Gateway RPC that only removes ClawHub-proven skill folders or lockfile-tracked entries, treats invalid slug/path errors as `INVALID_REQUEST`, uses the current Gateway runtime-config pattern, and keeps schema exports, generated clients, method scopes, docs, and focused backend/Gateway tests aligned. #63272 should continue tracking the separate CLI uninstall design.

## Risks / Open Questions
