## Summary

Keep PR #69319 open. Current main still does not expose a `skills.uninstall` Gateway RPC or a ClawHub skill-uninstall backend path, and the related older PR #64197 is closed unmerged. The PR addresses a real remaining gateway lifecycle gap from #69299, but the current PR discussion also identifies correctness issues that should be fixed before merge.

## Best Possible Solution

Keep #69319 open and ask for a tightened implementation: add the `skills.uninstall` schema, validator, handler, method registration, scope registration, docs, generated clients, and tests, but gate deletion on ClawHub provenance, classify invalid slugs as request errors, and clarify lockfile-only cleanup results. Then run the targeted agent/gateway tests and the changed gate before merge.

## Risks / Open Questions

- The PR as described may delete a manually managed or gateway-installer skill folder if its slug collides with an untracked ClawHub slug; deletion should be gated on ClawHub provenance from origin metadata or lockfile membership.
- Malformed slug errors should be classified as `INVALID_REQUEST`, not `UNAVAILABLE`, so clients do not retry bad input as a server outage.
- Lockfile-only cleanup semantics should be made explicit so operators can distinguish a pure no-op from removal of stale tracked metadata.
