## Summary

Keep open. Current main still sends Telegram edit payloads to `editMessageText` without a length guard, so the PR addresses a valid Telegram-plugin failure path, but PR head is not merge-ready because the truncation footer is derived from raw source length while truncation is triggered by rendered HTML length; the existing Greptile and ClawSweeper review comments point to the same concrete gap and the diff still contains it.

## What This Changes

The PR adds a Telegram edit payload truncation helper, wires it into `editMessageTelegram` for HTML and plain fallback edits, and adds focused tests for long edit payloads.

## Best Possible Solution

A corrected Telegram-plugin edit guard should cap both HTML and plain fallback edit payloads under Telegram's edit-text limit, preserve HTML/entity boundaries where possible, always append one visible marker whenever content is actually cut, and add a regression for rendered HTML exceeding the cap while the raw source is still within the soft budget.

## Reproduction Assessment

Yes. Current main can be exercised through `editMessageTelegram` with content over Telegram's edit-text limit, including the native-command progress-result path; the PR-specific regression is reproducible by calling `truncateTelegramEditPayload` with `rawText.length <= 4000` and `htmlText.length > 4096`.

## Solution Assessment

No, not as submitted. The fix belongs in the Telegram plugin edit path, but the helper should base its footer reservation and omitted-count logic on the payload/visible content actually being cut so every real truncation has a user-visible marker.
