## Summary

Keep open. The PR targets a real Telegram edit-path gap on current main, but the latest diff still has a correctness bug in the truncation helper: it decides to truncate from rendered HTML length while deriving the footer count from raw text length, so HTML/entity expansion can cut content with no visible truncation marker.

## What This Changes

The PR adds a Telegram edit-payload truncation helper, applies it to HTML and plain `editMessageText` calls, logs truncation, and adds Telegram send tests for long edit payloads.

## Best Possible Solution

A corrected Telegram-plugin edit guard should cap both HTML and plain fallback edit payloads below Telegram's `editMessageText` limit, base the truncation decision and omitted-count marker on the text length Telegram counts after entity parsing, preserve HTML/entity boundaries where possible, and add a regression test for rendered HTML expansion where raw/plain text is at or below the soft limit.

## Reproduction Assessment

Yes. Current main can reproduce the original bug by calling `editMessageTelegram` with an over-4096-character edit payload; the PR's remaining bug is reproducible by exercising `truncateTelegramEditPayload` with raw/plain text at or below 4000 characters and rendered HTML above 4096 characters.

## Solution Assessment

No, not as submitted. The Telegram plugin edit path is the right boundary, but the helper needs to use Telegram-counted/plain length for truncation and guarantee a visible marker whenever content is actually cut.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P2] Base truncation on Telegram-counted text length:** `extensions/telegram/src/send.ts:215-218`
  - body: The helper truncates whenever `htmlText.length > 4096`, but Telegram's limit is after entities parsing and `omittedChars` is derived only from `rawText.length - 4000`. For escaped/entity-expanded input, raw text can be within the limit while rendered HTML exceeds it; this code then cuts content and can append no footer at all. Base the guard and omitted-count calculation on the Telegram-counted/plain length, or otherwise ensure every actual cut gets a marker.
  - confidence: 0.93
