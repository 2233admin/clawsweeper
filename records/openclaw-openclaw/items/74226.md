## Summary

Keep open. The PR targets a real Telegram edit-message limit bug in current main, but the submitted helper still has a correctness gap: it can truncate rendered HTML that exceeds Telegram's limit while reporting zero omitted characters and appending no visible truncation marker.

## What This Changes

The PR adds a Telegram edit-payload truncation helper, wires it into `editMessageTelegram` for HTML and plain fallback edits, and adds focused tests for long edit payloads and helper behavior.

## Best Possible Solution

A corrected Telegram plugin edit guard should cap both HTML and plain fallback payloads below Telegram's edit limit, preserve HTML/entity boundaries where possible, compute the truncation footer from the actual rendered cut, and add a regression case where raw text is under the soft limit but rendered HTML exceeds 4096 characters.

## Reproduction Assessment

Yes. Current main can be checked by calling `editMessageTelegram` with rendered text over 4096 characters; the PR-specific bug is reproducible by calling `truncateTelegramEditPayload` with `rawText.length <= 4000` and `htmlText.length > 4096`.

## Solution Assessment

No, not as submitted. The fix belongs in `extensions/telegram/src/send.ts`, but the helper should base its omitted count and footer reservation on the actual rendered/truncated payload rather than only the raw input length.
