## Summary

Keep open. Current main still has an unguarded Telegram edit path, and this PR is a useful narrow direction, but the latest head still has a blocking truncation-notice edge case when rendered HTML exceeds the limit while the raw source does not.

## What This Changes

The PR adds a Telegram edit payload truncation helper, wires it into `editMessageTelegram` for HTML and plain fallback edits, and adds focused send tests for long edit payloads.

## Best Possible Solution

A corrected Telegram-plugin edit guard should cap both HTML and plain fallback edit payloads below Telegram's `editMessageText` limit, preserve entity boundaries where possible, always append a visible marker when content is actually cut, and leave the separately bounded streaming-preview path alone.

## Reproduction Assessment

Yes. Current main can be exercised through `editMessageTelegram` with a payload over the Telegram edit limit or through the native-command progress-result path; the PR's remaining bug is reproducible by calling `truncateTelegramEditPayload` with `rawText.length <= 4000` and `htmlText.length > 4096`.

## Solution Assessment

No, not as submitted. The fix belongs in the Telegram plugin edit path, but the submitted helper should base its omitted-count/footer reservation on actual rendered or truncated length so every real truncation gets a user-visible marker.

## Review Findings

Overall correctness: patch is incorrect
