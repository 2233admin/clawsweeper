## Summary

Keep this issue open. Current main still waits for Telegram media resolution and inbound context construction before the existing early typing cue, and that context path can await audio preflight transcription for voice/audio messages. The closed duplicate was explicitly closed in favor of this issue, while the related Telegram optimization PR remains open and unmerged rather than a shipped fix.

## What This Changes

The issue requests that Telegram voice messages trigger a typing indicator immediately on receipt, before media download, STT transcription, and agent-loop start delays.

## Best Possible Solution

Implement a Telegram-plugin-only receipt-time typing cue for eligible inbound voice/audio messages after cheap access/thread checks but before media download and transcribeFirstAudio, reusing existing sendChatAction backoff/thread helpers and keeping the existing model-loop typing as a later refresh path.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: send an authorized Telegram DM voice note with audio preflight enabled and current main reaches resolveMedia and transcribeFirstAudio before the existing sendTyping call in createTelegramMessageProcessor.

## Solution Assessment

Yes, with a narrower boundary than the issue's unconditional wording. The maintainable fix is a best-effort Telegram typing action after cheap eligibility checks, not a core migration or a broad channel API change until multiple plugins need the same pre-STT contract.
