## Summary

Keep open. Current main has classic onboarding, section-based reconfiguration, plugin field prompts, doctor repair paths, and Crestodian typed setup/repair, but it still does not implement the requested assistant that interviews the user about goals, probes host capabilities, recommends best-practice settings, and revisits those recommendations as features ship. The existing bot comment and later maintainer mention support leaving this for product scoping rather than cleanup-close.

## What This Changes

The issue requests a built-in guided configuration assistant that asks users about goals, probes host resources such as CPU, memory, and GPU, recommends OpenClaw settings, and can be rerun after new features ship.

## Best Possible Solution

Define a scoped next-generation setup/configuration UX that extends existing onboarding and Crestodian boundaries: collect user goals, expose explainable host/resource discovery only with consent, let plugins contribute typed recommendation metadata where appropriate, and keep every persistent config write approved, audited, reversible, schema-backed, and documented.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Not applicable as a feature request, but current-main source and docs inspection provides a high-confidence gap check: the existing onboarding, configure, plugin prompts, doctor repairs, and Crestodian paths do not provide the requested goal-driven host-capability recommender.

## Solution Assessment

Is this the best way to solve the issue? No, not as a broad blind config-writing automation. The maintainable path is a maintainer-scoped design that extends typed setup/config surfaces and plugin metadata before adding recommendation logic or host probes.
