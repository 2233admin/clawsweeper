## Summary

Keep open. Current main still treats `ANNOUNCE_SKIP` as an exact trimmed reply, so a multi-line sub-agent output ending with the token remains visible announce content. The issue is also paired with open PR #74136, which uses closing syntax for this behavior, so it should remain open until that PR lands or is replaced.

## What This Changes

The issue requests that sub-agent announce suppression accept `ANNOUNCE_SKIP` when it appears as a standalone final line after other output, not only when it is the entire reply.

## Best Possible Solution

Review and land PR #74136, or an equivalent narrow patch, so announce suppression accepts the canonical exact token plus a standalone final-line `ANNOUNCE_SKIP` while preserving `REPLY_SKIP` and `NO_REPLY` semantics. Add focused helper coverage, an announce-flow regression test, and align the prompt/docs contract if final-line suppression becomes intended behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `isAnnounceSkip("summary\nANNOUNCE_SKIP")`, which returns false on current main because the helper compares the whole trimmed string to the token.

## Solution Assessment

Yes. The narrowest maintainable solution is to change only `isAnnounceSkip` to recognize a standalone final-line token, keep reply-skip semantics strict, and cover the helper plus announce-flow path; PR #74136 is already the active candidate for that approach.
