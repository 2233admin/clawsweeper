## Summary

Keep open. Current main still treats `ANNOUNCE_SKIP` as an exact whole-message token, so the reported multi-line final-line case remains valid; open PR #74136 already targets this issue with closing syntax, so the issue should stay open until that PR lands or is replaced.

## What This Changes

Allow sub-agent announce suppression when multi-line assistant output ends with `ANNOUNCE_SKIP` as a standalone final line.

## Best Possible Solution

Review and land PR #74136, or an equivalent narrow patch, so announce suppression accepts the existing exact token plus a standalone final-line `ANNOUNCE_SKIP` while preserving `REPLY_SKIP` and `NO_REPLY` semantics. Add focused helper coverage, one announce-flow regression test, and align docs or prompt wording if the tolerant form becomes the public contract.

## Reproduction Assessment

Yes. A source-level reproduction is `isAnnounceSkip("summary\nANNOUNCE_SKIP")`, which still returns false from the current helper; the announce call sites then treat that text as visible reply content.

## Solution Assessment

Yes. The narrow maintainable fix is to extend only announce-skip recognition to a standalone final line while preserving existing exact-token and silent-reply behavior; the active PR is already the implementation candidate for that path.
