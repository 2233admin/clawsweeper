## Summary

Keep open. Current main still blocks the LINE webhook HTTP 200 on event processing in the reusable Node handler, the Express-style middleware, and the current gateway shared-path route; the tests also preserve the blocking/error-before-ACK behavior. The related fix PR was closed unmerged, so there is no implementation on main to close against.

## What This Changes

The issue requests that LINE webhooks acknowledge verified requests with HTTP 200 immediately, then process events asynchronously and deliver slow replies via push/fallback handling instead of holding the webhook response open.

## Best Possible Solution

Implement the fix inside the LINE plugin: after raw-body read, signature validation, target resolution, and JSON parsing succeed, send the 200 response immediately and run event processing fire-and-forget with runtime error reporting. Preserve pre-auth body limits, signature-integrity behavior, account disambiguation, and existing push/reply fallback semantics, then update tests to prove ACK-before-processing for both direct and shared-path handlers.
