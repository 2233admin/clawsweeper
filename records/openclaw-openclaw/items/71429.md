## Summary

Keep this open. Current main has related reload deferral, durable generic outbound recovery, and a best-effort Telegram fallback when delivery fails while the process stays alive, but ordinary Telegram same-surface final replies still go through an in-process dispatcher and direct `deliverReplies` Bot API delivery without a durable write-ahead entry before send. The maintainer discussion already called out this exact gap, and the current main code still matches that boundary.

## What This Changes

The issue requests durable replay or a visible hard failure for in-flight Telegram outbound replies when gateway hot reload and `sendChatAction` or Bot API network failures occur mid-turn.

## Best Possible Solution

Add a focused Telegram regression for same-surface auto-replies interrupted by reload or delivery failure, then either journal normal Telegram final replies before Bot API delivery and replay them safely, or fail the turn visibly in Telegram and UI with retry guidance.

## Reproduction Assessment

Yes at the focused integration-test level: simulate a Telegram inbound turn where `Provider` and `Surface` are both `telegram`, make final `deliverReplies` fail during a reload window, and verify that current main has neither a durable replay entry nor a guaranteed visible failure if the process does not survive to send the fallback. A live Telegram hot-reload run was not performed here.

## Solution Assessment

No, current main is not the best solution yet. The requested direction is sound, but the maintainable fix should reuse existing durable delivery primitives or add a small plugin-owned durable seam while preserving Telegram duplicate-send and preview/final semantics.
