## Summary

Keep this open. Current main still has related reload deferral and fallback-on-delivery-error behavior, but normal Telegram same-surface replies are still delivered through an in-process dispatcher and direct Telegram delivery path, not a durable replay journal; persistent Bot API/network failure can still leave the generated response visible internally but absent from Telegram.

## What This Changes

Ensure Telegram replies generated during hot reload and sendChatAction or Bot API network failures are either durably replayed or visibly failed instead of silently disappearing.

## Best Possible Solution

Add a focused Telegram regression for same-surface auto-replies during reload/network failure, then either journal normal Telegram final replies before Bot API delivery and replay them safely, or fail the turn visibly in Telegram and UI with retry guidance.

## Reproduction Assessment

Yes. A focused unit/integration reproduction can simulate a Telegram inbound turn with Provider and Surface both set to telegram, make deliverReplies or sendChatAction fail during a reload window, and assert that no durable replay entry or guaranteed visible failure exists; a live Telegram run was not performed in this read-only review.

## Solution Assessment

No current-main behavior is not the best solution yet. The narrow maintainable fix is to reuse the existing durable outbound/session delivery primitives for normal Telegram same-surface final replies, or explicitly fail the turn with a retryable user-visible status if safe replay is not possible.
