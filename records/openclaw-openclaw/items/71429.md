## Summary

Keep open. Current main still has related reload deferral and generic durable delivery queues, but ordinary same-surface Telegram auto-replies still pass through an in-process dispatcher and direct Bot API send path without a proven durable journal or visible hard-failure contract for the reported hot-reload/sendChatAction failure case.

## What This Changes

The issue asks OpenClaw to prevent silent Telegram reply loss during gateway hot reload when sendChatAction or network delivery fails, either by durable resend or by surfacing a clear retryable failure.

## Best Possible Solution

Normal Telegram auto-replies should have a tested delivery contract during reload: either persist final Telegram payloads before Bot API delivery and recover them after restart, or convert unrecoverable delivery failure into a visible Telegram/UI failure with retry guidance.
