## Summary

Keep open. Current main has related restart deferral and fallback notification attempts, but normal Telegram same-surface final replies still flow through the in-memory dispatcher to direct Telegram Bot API delivery rather than a durable queue, and the maintainer discussion already identified that exact gap as not closed.

## What This Changes

The issue requests reliable handling for Telegram replies during hot reload/network failure so in-flight outbound responses are either durably replayed or visibly failed instead of silently lost.

## Best Possible Solution

Normal same-surface Telegram final replies should be journaled through existing durable delivery primitives before Bot API delivery, or the turn should transition to an explicit retryable failure state that is visible in Telegram, UI, and agent/session state.

## Reproduction Assessment

Yes. A high-confidence regression can simulate a Telegram inbound turn where Provider and Surface are both telegram, force sendChatAction or direct deliverReplies failure during a reload window, and assert that current main has no durable replay for the final same-surface reply.

## Solution Assessment

Yes. Reusing the existing outbound/session delivery primitives or introducing an explicit visible abort is the narrow maintainable direction; a Telegram-only ad hoc journal would be weaker unless shared queue semantics cannot preserve preview/final ordering safely.
