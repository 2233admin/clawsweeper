## Summary

Keep this open. Current main still has restart deferral and durable outbound/restart-continuation queues, but ordinary Telegram same-surface auto-replies still use the in-process dispatcher and direct Telegram Bot API delivery path, so the reported silent-loss window remains plausible and is not covered by the existing hardening.

## What This Changes

The issue requests durable replay or explicit visible failure for Telegram in-flight outbound replies when hot reload coincides with sendChatAction or Bot API network failures.

## Best Possible Solution

Normal Telegram same-surface final replies should either be journaled before Bot API delivery and replayed safely after reload, or the turn should fail with a retryable user-visible status in Telegram and UI when safe replay is not possible.

## Reproduction Assessment

Yes. A high-confidence unit/integration reproduction can simulate a Telegram inbound turn with Provider and Surface both set to telegram, force deliverReplies or sendChatAction-related delivery to fail during a reload window, and assert that no durable replay entry or guaranteed visible failure exists.

## Solution Assessment

No. Current-main behavior is not the best solution yet; the narrow maintainable path is to reuse the existing outbound/session delivery primitives for normal Telegram same-surface final replies or explicitly fail the turn visibly when replay cannot be made idempotent.
