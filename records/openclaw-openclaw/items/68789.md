## Summary

Keep open. Current main still leaves the reported direct Slack WebClient failure path unresolved: Slack write calls use a no-retry WebClient, the send path propagates request errors from `conversations.open` and `chat.postMessage`, and the existing Socket Mode and global unhandled-rejection work only addresses adjacent stale-socket or crash-loop behavior. The report is recent, concrete, and distinct from #24581 because the gateway remains running while fresh Slack sends continue to fail.

## What This Changes

The issue requests a Slack-plugin fix or diagnostics for intermittent Slack WebClient `EAI_AGAIN slack.com` failures when host DNS and standalone Node HTTPS checks are healthy.

## Best Possible Solution

Keep this as a Slack-plugin reliability bug. Add operation-specific diagnostics for `conversations.open`, `chat.postMessage`, upload setup/completion, and Socket Mode reconnect failures, then add conservative transient-DNS recovery only where duplicate delivery is prevented or the Slack operation is safe to retry. Regression coverage should inject Slack-like `slack_webapi_request_error` wrappers with nested `EAI_AGAIN` and assert both the diagnostic label and the retry/no-retry behavior.
