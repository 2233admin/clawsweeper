## Summary

Keep open. Current main still emits the narrower compaction event payload, and this PR is useful but not merge-ready because it targets stale `auto_compaction_*` event names, omits the promised `hasResult` field from emitted end events, and raises routine lifecycle logs to `warn`.

## What This Changes

The PR branch adds compaction start/end event metadata, structured compaction lifecycle logging, dispatcher forwarding for start details, and a focused compaction payload test file.

## Best Possible Solution

Rework the change on current main so the dispatcher passes `compaction_start` into the handler, start events expose a safe `reason`, end events expose `completed`, `hasResult`, `willRetry`, and sanitized `errorMessage`, and compaction count, retry, reconciliation, hooks, liveness, UI behavior, and fallback policy remain unchanged except for the richer observability contract.

Yes. The high-confidence reproduction path is source-level: inspect or exercise current main's `compaction_start` / `compaction_end` handler path and observe that it still emits the narrow payload; no runtime test was run because this was a read-only review.

## Solution Assessment

No. The requested direction is narrow and useful, but this exact patch is not the best solution because it needs to be rebased onto the current event names, emit `hasResult`, and keep normal lifecycle logs below `warn`.
