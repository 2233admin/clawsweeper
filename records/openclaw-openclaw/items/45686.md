## Summary

Current main still only partially hardens compaction failure handling. It has static compaction model selection, timeout/overflow retry caps, timeout cleanup, and tool-result truncation, but it does not implement the requested repeated-failure fallback circuit breaker, forced lossy transcript pruning, live failing-compaction status, or full channel transport decoupling; related PR #74470 remains open and covers only the fallback slice.

## What This Changes

Request adds a compaction reliability circuit breaker with fallback-model retry, forced lossy truncation after repeated failures, Telegram transport isolation, and gateway status visibility.

## Best Possible Solution

Define a core compaction failure policy that tracks per-session compaction failures, uses an approved fallback route/model for implicit compaction, applies documented lossy recovery after a threshold, exposes live/failing compaction state through session or gateway status, and keeps channel transports resilient without embedding Telegram-specific policy in core.
