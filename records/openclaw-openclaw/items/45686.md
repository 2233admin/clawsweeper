## Summary

Keep this open. Current main has meaningful partial mitigations for compaction stalls, including timeout-recovery compaction, bounded overflow compaction attempts, static compaction-only model override, and narrower Telegram polling network handling, but it does not implement the requested repeated-failure compaction circuit breaker with automatic fallback model, lossy force-truncate/reset policy, transport-isolated background compaction, or gateway-visible compaction failure health.

## What This Changes

Request a compaction circuit breaker that falls back to a faster compaction model, force-truncates after repeated compaction failures, keeps Telegram polling from being taken down by compaction, and exposes stuck compaction health in gateway status.

## Best Possible Solution

Define and implement a durable compaction-failure policy in the embedded/context-engine compaction path: track consecutive compaction failures per session, retry with an explicit configured fallback or compaction model policy, then perform a documented lossy recovery that preserves the newest usable transcript state and records health metadata for status/diagnostics. Telegram should observe that state without treating compaction failures as polling network failures.
