## Summary

Keep open. Current main has useful partial mitigations for compaction failures, but it does not implement the requested automatic compaction circuit breaker: no repeated-failure fallback model, no lossy force-truncate after failed compactions, no compaction failure health in gateway status, and inbound run queuing is still blocked while a run is compacting. The discussion is also active, with a contributor offering a circuit-breaker component for maintainer review.

## Best Possible Solution

Keep this issue open and route it to a focused compaction reliability design/PR. The likely implementation belongs in the embedded runner and compaction safeguard path: track per-session compaction failure state, retry with a configured or automatically selected fallback model/provider after repeated primary failures, apply a bounded lossy recovery strategy after the cap, keep inbound channel messages queued or acknowledged while compaction is running, and expose active/failure state through session or gateway status. Existing `agents.defaults.compaction.model`, provider registration, timeout retry caps, and compaction wait timeout should be reused rather than replaced.
