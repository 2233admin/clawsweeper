## Summary

Keep open. Current main still replays pending outbound delivery queue entries by retry/backoff/max-retry policy only, and the strict messages config still has no delivery TTL or expiry-action surface. The production reproduction and prior maintainer review remain valid; the related SQLite/outbox attempts are closed and unmerged, and there is no open PR currently referencing this issue.

## What This Changes

Add an optional TTL/expiry policy for outbound delivery queue messages so stale pending entries are not replayed after gateway restart or reconnect drains.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: accept optional strict messages.delivery config, expire stale pending entries before replay in both startup recovery and reconnect drains, preserve default no-TTL compatibility, log expiry counts, and cover recovery, reconnect, storage, policy, and config acceptance with focused tests.

Keep the fix narrow: do not revive the closed SQLite/message_outbox rewrites, do not change channel adapters, and do not alter unrelated queue semantics. Add focused tests for startup recovery expiry, reconnect-drain expiry, storage behavior, policy/action handling, and strict config acceptance/rejection.
