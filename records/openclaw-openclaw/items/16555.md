## Summary

Keep open. Current main still has the file-backed outbound delivery queue, startup recovery and reconnect drain only enforce retry/backoff/permanent-error policy, and there is still no `messages.delivery.maxAgeMs` or `expireAction` config surface. The production repro and the latest maintainer review remain applicable; the related SQLite/outbox TTL PRs are closed and unmerged rather than canonical fixes.

## What This Changes

Add configurable TTL/expiry handling for outbound delivery queue messages so stale persisted entries are skipped, deleted, or quarantined instead of replayed after gateway restart or reconnect.

## Best Possible Solution

Implement a narrow core outbound delivery-queue expiry policy: optional `messages.delivery.maxAgeMs` plus a closed `expireAction`, applied after claim/re-read in both `recoverPendingDeliveries` and `drainPendingDeliveries`. Preserve default no-TTL behavior, log expired counts, prefer a failed/quarantine path when configured, and update config types/schema/help/docs plus focused recovery, reconnect-drain, storage/policy, and config tests.
