## Summary

Keep open. Current main still lacks configurable TTL/expiry for the file-backed outbound delivery queue: startup recovery and reconnect drains replay pending entries using max-retry/backoff policy only, and the strict `messages` config has no `delivery.maxAgeMs` or `expireAction` surface. The production repro and maintainer follow-up in the discussion still apply; related SQLite/outbox alternatives in the provided context are closed and unmerged rather than canonical fixes.

## What This Changes

The issue requests configurable TTL/expiry for persisted outbound delivery queue messages so stale entries are not replayed on gateway restart.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue: add optional strict `messages.delivery` config, expire stale pending entries before replay in both startup recovery and reconnect drains, keep default no-TTL behavior, log expiry counts, and cover recovery, reconnect, storage, policy, and config acceptance with focused tests.
