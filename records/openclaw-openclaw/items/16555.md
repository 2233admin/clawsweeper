## Summary

Keep open. Current main still lacks outbound delivery-queue TTL/expiry: startup recovery and reconnect draining load pending queue files and enforce retry/backoff/max-retry policy, but do not check queued age, and the strict messages config has no `messages.delivery.maxAgeMs` or `expireAction` surface. The production repro and prior maintainer direction remain applicable; the related SQLite/outbox TTL PRs are closed and unmerged rather than canonical fixes.

## What This Changes

Add an optional outbound delivery-queue age limit and expire action so stale queued messages are skipped, deleted, or quarantined instead of replayed on gateway restart or reconnect.

## Best Possible Solution

Implement a narrow core expiry policy for the existing file-backed outbound delivery queue, preserving the default no-TTL behavior while letting operators opt into a bounded stale-message policy with clear logs and tests for startup recovery, reconnect drain, storage, policy, and config acceptance.

## Repair Work Prompt

Implement optional outbound delivery-queue expiry for stale queued messages. Add a backward-compatible `messages.delivery.maxAgeMs?: number` and closed `expireAction` config surface, preserving default no-TTL behavior when unset. Apply the age check after claiming and re-reading a queue entry in both `recoverPendingDeliveries` and `drainPendingDeliveries`, before max-retry/backoff/delivery. Support a safe quarantine/move-to-failed behavior for expired entries and log expired counts; include skip/delete only if the chosen enum and docs make those semantics explicit. Update config types, strict zod schema, schema help/labels, generated config artifacts if required, and focused docs if there is a public config surface. Add tests for startup recovery expiry, reconnect-drain expiry after re-read, storage/quarantine behavior, policy helper behavior, and config acceptance/rejection. Do not replace the queue with SQLite/outbox and do not change channel adapters beyond what the expiry policy requires.
