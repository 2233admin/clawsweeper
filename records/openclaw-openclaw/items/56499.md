## Summary

Keep this PR open. Current main still fails fast on startup secret-provider resolution without retry/backoff, and the PR's secondary WhatsApp append/pairing guard is also not present in the current inbound path. The work is plausible but needs a rebase onto the moved startup-secret activator plus focused regression tests before maintainers can land it.

## Best Possible Solution

Rebase the secret-provider retry onto `src/gateway/server-startup-config.ts` around `prepareRuntimeSecretsSnapshot`, add focused tests for startup-only retry count/delays/final error and no reload/restart-check retry, then either split the WhatsApp append/pairing change into its own PR or keep it with explicit tests proving append DMs cannot issue pairing challenges before history filtering. After that, review and land the updated PR to resolve the linked secret-provider crash-loop issue.
