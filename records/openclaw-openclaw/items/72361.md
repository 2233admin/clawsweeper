## Summary

Keep this PR open, but do not merge it as-is. Current main has the reported `/allowlist --store` bypass, and the PR adds the right kind of guard for the same-channel case; however the new guard authorizes against the target `--channel`/`--account` scope instead of the message source scope, leaving a cross-channel/account variant of the same `configWrites=false` bypass.

## What This Changes

The PR adds a `configWrites` authorization check before store-only `/allowlist add|remove --store` mutations and adds focused allow/block tests for channel-level and account-level policy behavior.

## Best Possible Solution

Keep the PR focused, but change the store-write authorization to pass the actual command source channel/account as the origin and the selected store channel/account as the explicit target, then add regression coverage for `--channel` and `--account` cases where source and target policies differ.

## Reproduction Assessment

Yes. The linked issue provides a high-confidence same-channel reproduction, and the code path on current main confirms store-only edits reach `updatePairingStoreAllowlist` without a `configWrites` check. A second focused reproduction should cover source channel/account disabled with `--channel` or `--account` targeting a write-enabled store.

## Solution Assessment

No. Adding a store-only `resolveConfigWriteDeniedText` guard is the right direction, but using the target scope as the origin is not the best fix because it leaves a bypass when the command source differs from the target store.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use the source channel for store write authorization:** `src/auto-reply/reply/commands-allowlist.ts:566-567`
  - body: The new store-only guard passes the target `channelId`/`accountId` as the authorization origin. With `/allowlist add dm --store --channel discord ...` sent from a channel whose `configWrites` is false, the origin check evaluates Discord instead of the source channel, so the write-disabled source can still mutate a write-enabled target store. Pass the command source channel/account as the origin and keep the selected store scope as the target, with a regression test for the cross-channel/account case.
  - confidence: 0.86
