## Summary

Keep this PR open. It targets a valid access-control bug, but the proposed store guard still authorizes with the selected target channel/account after `--channel` resolution, so a write-disabled source channel can still initiate a cross-channel pairing-store mutation; the PR also lacks the required user-facing changelog entry.

## What This Changes

The PR adds a `configWrites` authorization guard before `/allowlist --store` pairing-store mutations and adds focused allowlist command tests for blocked and allowed store-targeted edits.

## Best Possible Solution

Use distinct source and target scopes for `/allowlist` writes: derive origin from the actual command source (`params.command.channelId` or `params.command.channel`) and source account context, derive target from the selected allowlist channel/account, enforce that for both config and store mutations, add a cross-channel regression test, and add the active-version changelog entry.

## Reproduction Assessment

Yes. Source inspection gives a high-confidence path: send `/allowlist add dm --store --channel discord 789` from a Telegram command source with `channels.telegram.configWrites=false` while Discord writes are enabled; the proposed guard sees Discord as the origin.

## Solution Assessment

No. The PR uses the right policy helper, but it wires the store-only branch with one `channelId` for both origin and target; the safer fix is to pass separate origin and target scopes consistently.
