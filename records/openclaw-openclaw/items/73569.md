## Summary

Keep open. Current main still collapses the no-provider media-understanding path into the generic empty transcript/description errors, and multiple open PRs already reference this issue with closing syntax, so the maintainer action is to choose and land a canonical fix rather than close it now.

## What This Changes

The issue requests clearer `infer audio transcribe` and `infer image describe` errors when no audio or image-understanding provider is configured, instead of path-blaming empty-result messages.

## Best Possible Solution

Land one canonical fix that preserves the real provider-ran-but-empty errors while carrying the no-provider/no-attempt media-understanding state through to an actionable missing-configuration error for image describe, describe-many, and audio transcribe, with focused CLI/runtime regression tests.

## Reproduction Assessment

Yes. Source inspection of current main gives a high-confidence reproduction path: no media entries resolve, `runCapability` returns a skipped/no-output decision, `runMediaUnderstandingFile` returns `text: undefined`, and the CLI emits the generic path-based error.

## Solution Assessment

Yes. The narrow maintainable solution is to separate missing-provider/no-attempts from provider-empty-output at the media runtime or CLI boundary; closing this issue is not appropriate until one linked fix merges.
