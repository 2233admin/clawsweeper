## Summary

Keep open. Current main still collapses the no-provider media-understanding skip path into the same undefined text result used for empty provider output, and the CLI still reports the generic path-blaming audio/image errors. Three open PRs already reference this issue with closing syntax, so the maintainer action is to choose and land a canonical fix rather than close the issue or queue another repair.

## What This Changes

The issue requests clearer missing-provider configuration errors for `infer audio transcribe`, `infer image describe`, and related image describe flows while preserving the existing empty-result errors when a provider actually ran.

## Best Possible Solution

Land one canonical fix that preserves provider-ran-but-empty errors, carries or detects the no-provider/no-attempts media-understanding state, and reports an actionable configuration error for image description, image describe-many, and audio transcription with focused CLI/runtime regression coverage.

## Reproduction Assessment

Yes. The source path on current main is enough to reproduce with high confidence: no resolved media entries produce a skipped decision with no outputs, the runtime returns `text: undefined`, and the CLI converts that into the reported generic error.

## Solution Assessment

Yes. The narrow maintainable solution is to separate missing-provider/no-attempts from empty provider output at the media runtime or CLI boundary; closing the issue is not the right solution until one linked fix PR merges.
