## Summary

Keep open. Current main still conflates the no-provider/no-attempt media-understanding path with generic empty-output errors, and three linked fix PRs remain open, so this should stay open until one canonical fix lands.

## What This Changes

The issue requests clearer missing-provider configuration errors for `infer audio transcribe` and `infer image describe` when no audio or image-understanding provider is configured.

## Best Possible Solution

Land one canonical fix that preserves the existing empty-result errors after a provider attempt, but carries missing-provider/no-attempt state far enough for `infer audio transcribe`, `infer image describe`, and `infer image describe-many` to emit actionable configuration errors with focused regression tests and a user-facing changelog entry.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source inspection on current main shows the deterministic path: no resolved media provider returns a skipped/no-output decision, the runtime converts that to `text: undefined`, and the CLI throws the generic file-path error.

## Solution Assessment

Is this the best way to solve the issue? Yes: the narrow maintainable fix is to distinguish missing-provider/no-attempt from provider-ran-but-empty-output at the media runtime or CLI boundary. The remaining maintainer choice is which linked PR should become canonical.
