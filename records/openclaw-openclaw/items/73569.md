## Summary

Keep open. Current main still matches the report: the no-configured-provider path in media understanding returns a skipped/no-output result, and the CLI wrappers turn that into the generic path-blaming "No description/transcript returned" errors. The existing ClawSweeper comment is consistent with a fresh source pass; the issue remains a concrete OpenClaw CLI/media-understanding bug, not a duplicate or external setup request.

## Best Possible Solution

Keep this issue open and fix the CLI/media-understanding boundary. The implementation should preserve the existing generic errors for the real "provider ran but returned empty text" case, but surface a separate actionable error when no media-understanding provider/model can be resolved. For image, point users at `agents.defaults.imageModel.primary`, `tools.media.image.models`, or shared `tools.media.models`; for audio, point at `tools.media.audio.models`, shared `tools.media.models`, local audio CLIs, or provider auth/env setup. Add focused CLI/runtime tests for both `infer image describe`/`describe-many` and `infer audio transcribe` under an all-unconfigured-provider branch.
