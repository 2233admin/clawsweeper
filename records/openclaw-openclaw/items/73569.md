## Summary

Keep open. Current main still collapses the no-configured-provider media-understanding path into the same generic file-path error, and three open PRs already reference this issue as their fix target, so this should stay open until one canonical fix merges.

## What This Changes

The issue asks for `infer audio transcribe` and `infer image describe` to report missing audio/image-understanding provider configuration instead of `No transcript/description returned for ... <path>` when no provider is configured.

## Best Possible Solution

Land one canonical fix that preserves missing-provider/no-attempt state across the media-understanding runtime or otherwise exposes it to the infer CLI, then emit actionable audio and image-understanding configuration errors while keeping the existing empty-result errors for configured providers that return no text.

## Reproduction Assessment

Yes. Current-main source gives a high-confidence reproduction path: no resolved media provider returns a skipped decision with no outputs, the runtime converts that to `text: undefined`, and the CLI maps that to the generic path-focused error.

## Solution Assessment

Yes. The requested direction is the narrow maintainable solution: distinguish missing provider/no attempts from provider-ran-but-empty-output at the runtime or CLI boundary, with focused tests for audio transcribe, image describe, and describe-many.
