## Summary

Current main still does not implement this PR’s intended Gateway final-message fix. The delta path suppresses `NO_REPLY` lead fragments while still buffering raw/projected text, and finalization explicitly reprojects with lead-fragment suppression disabled, so lifecycle-ending fragments such as `NO_` can still be emitted as final chat payloads. The PR remains useful, but the latest branch needs maintainer review because its suppression provenance is keyed by `clientRunId`, which comes from caller-provided `idempotencyKey`.

## Best Possible Solution

Keep this PR open as the canonical narrow Gateway fix candidate. Revise or confirm suppression provenance so it is scoped to a server-owned run/session identity, keep regression coverage for lifecycle-ending `NO_` or `NO_RE`, keep intentional uppercase `NO` preservation coverage, then run the targeted Gateway test and normal changed gate before merge.
