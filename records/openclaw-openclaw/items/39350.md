## Summary

Keep this PR open. Current main still lacks the run-provenance/final-message guard needed for lifecycle-ending `NO_REPLY` fragments: a suppressed `NO_`/`NO_RE` fragment can remain in the Gateway buffer and then be emitted by the final chat event. The live PR head remains the relevant fix candidate, but maintainers should review the `clientRunId`-scoped suppression state before merge.

## Best Possible Solution

Keep this PR open as the canonical Gateway fix candidate. Adjust or justify the suppression provenance so it cannot collide across sessions, preserve intentional uppercase `NO` replies, keep lifecycle-end `NO_`/`NO_RE` regression coverage, then run the normal Gateway changed gate before merge.
