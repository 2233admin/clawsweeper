## Summary

Keep this PR open. Current main still buffers suppressed NO_REPLY lead fragments and then finalizes with lead-fragment suppression disabled, so lifecycle-ending fragments such as `NO_` can still become final Control UI chat messages. The PR remains a useful narrow fix candidate for the linked WebChat leak, but its latest branch needs maintainer review for the suppression-provenance state keyed by `clientRunId`.

## Best Possible Solution

Keep this PR open as the canonical narrow fix candidate for the final-message leak. Update or confirm the branch so lead-fragment provenance is scoped to an appropriate server-owned run/session identity or is otherwise consistent with the Gateway idempotency contract, keep regression coverage for lifecycle-ending `NO_` and intentional uppercase `NO`, then run the targeted Gateway test and changed gate before merge. Leave #40165 as the broader follow-up for suffix and mixed-content handling.
