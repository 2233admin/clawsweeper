## Summary

Keep open. Current main still lacks Signal container REST/WebSocket support, so this is not obsolete or implemented-on-main; however, the PR head has merge-blocking issues around raw inbound Signal payload logging and missing config schema validation for the new apiMode setting.

## What This Changes

The PR adds a Signal native/container adapter, a bbernhard REST/WebSocket client, Signal caller rewiring, channels.signal.apiMode docs/types/tests, a changelog fragment, and a direct ws runtime dependency for the Signal plugin.

## Best Possible Solution

Ship the Signal plugin-local transport adapter after removing or redacting raw payload logs, adding apiMode to the strict config schema/generated config/help surfaces, preserving native JSON-RPC/SSE behavior, and providing focused native/container tests plus live bbernhard container proof or a maintainer waiver.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Remove raw Signal payload logging:** `extensions/signal/src/client-container.ts:202`
  - body: streamContainerEvents logs the first 200 characters of each inbound WebSocket frame before parsing it. Those frames can contain private Signal message text plus sender, group, and attachment metadata, so normal container receive would write sensitive content to gateway logs; remove this log or replace it with redacted metadata behind explicit debug output.
  - confidence: 0.93
- **[P2] Validate the apiMode setting:** `src/config/types.signal.ts:68`
  - body: apiMode is added to the public SignalConfig type and docs, but the strict SignalConfigSchema still extends SignalAccountSchemaBase without accepting apiMode. A user setting channels.signal.apiMode: "container" can fail config validation before the adapter sees it, so the schema and generated config/help surfaces need to be updated with the new enum.
  - confidence: 0.91
