## Summary

Keep this issue open. Current main still performs the SQLite-backed `hasIndexedContent()` check from the memory search hot path, and an open contributor PR already references this issue with closing syntax, so the right maintainer path is to review that PR rather than auto-close or create duplicate repair work.

## What This Changes

The issue requests caching `hasIndexedContent()` in the memory manager so repeated `search()` calls do not query SQLite every time.

## Best Possible Solution

Land a memory-core-owned cache for the indexed-content preflight once the linked PR, or a maintainer-approved equivalent, proves invalidation on chunk inserts, deletes, full resets, and DB handle swaps without changing core behavior.

## Reproduction Assessment

Yes. Source inspection on current main gives a high-confidence reproduction path: call `MemoryIndexManager.search()` repeatedly and instrument `db.prepare` or SQLite tracing to observe the `chunks` and optional FTS probes from `hasIndexedContent()`.

## Solution Assessment

Unclear until the linked PR is reviewed, but the best fix shape is narrow: keep the cache inside memory-core and invalidate it at every index mutation/reindex boundary rather than adding a core-level workaround.
