## Summary

Keep open for maintainer review. Current main still lacks the unversioned `python` shim, and this PR's current diff is narrow, addresses the earlier review comments, and has no blocking correctness finding from this read-only pass.

## What This Changes

The PR adds a conditional `python`-to-resolved-`python3` symlink to the gateway and sandbox Dockerfiles when `python3` is present and no `python` command is already on PATH.

## Best Possible Solution

Merge or request small scope adjustments for this narrow alias if maintainers want `python` to work wherever shipped or operator-built images already contain `python3`; keep the separate default gateway `python3` installation decision in #75041.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current sandbox Dockerfiles install `python3` and current main has no unversioned `python` shim, so a built sandbox image can compare `python3 --version` with failing `python --version`; the gateway case reproduces when built with `OPENCLAW_DOCKER_APT_PACKAGES=python3`.

## Solution Assessment

Yes for the narrow alias problem. Resolving the target with `command -v python3` and skipping when any `python` command already exists is the smallest maintainable Dockerfile change, while installing `python3` by default belongs in the separate gateway-image discussion.
