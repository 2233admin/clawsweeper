## Summary

Keep open. Current main does not contain the shim, and the PR's v2 diff looks like a narrow, safe Docker improvement that should receive normal maintainer review rather than cleanup closure; the related #75041 report should stay separate because it is about installing `python3`, not aliasing `python` when `python3` already exists.

## What This Changes

Adds a conditional Dockerfile layer that creates `/usr/local/bin/python` pointing at the resolved `python3` binary in the gateway and sandbox images when `python3` exists and no `python` command is already on PATH.

## Best Possible Solution

Land the symlink-only change if Docker maintainers agree OpenClaw images should expose `python` whenever Python is already baked in, and keep any default gateway `python3` installation or documentation decision in the separate #75041 track.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to build one of the current sandbox images, or the gateway image with `OPENCLAW_DOCKER_APT_PACKAGES=python3`, then run `command -v python3 && command -v python`; current main installs or can install `python3` but has no unversioned shim.

## Solution Assessment

Yes for the symlink-only scope. The proposed `command -v` guard and resolved symlink target are the narrowest maintainable way to preserve operator-provided `python` while adding the alias when `python3` is already present; installing `python3` by default is a separate product/image-size decision.
