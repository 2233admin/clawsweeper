## Summary

Keep open. This protected maintainer-labeled MEMBER PR is still the active artifact-RPC implementation candidate for #74706, but the latest PR head still has a blocking run/task provenance issue: scoped queries filter on transcript metadata that current transcript reads do not provide for normal messages.

## What This Changes

The PR adds transcript-derived Gateway `artifacts.list/get/download` RPCs, read-scope registration, protocol and Swift models, SDK `oc.artifacts` helpers/types/docs, handler/SDK tests, changelog coverage, and task-scope lookup hardening.

## Best Possible Solution

Land a maintainer-approved revision that keeps the transcript-derived artifact projection, `operator.read` gating, unsupported unsafe/local downloads, SDK/docs/protocol synchronization, and generated client models, while also making advertised run/task scopes work through persisted/readable transcript provenance or by narrowing the public scope contract until that provenance exists.

## Reproduction Assessment

Yes. The PR-head reproduction is to read ordinary transcript messages through current `readSessionMessages()` semantics, then call `collectArtifactsFromMessages()` with a `runId` or `taskId`; messages without explicit run/task metadata are filtered out even though the same artifacts appear by `sessionKey`.

## Solution Assessment

No, not as-is. The direction is maintainable, but the run/task filter must be backed by real transcript provenance or the advertised run/task query surface should be deferred to avoid returning empty results for normal transcript artifacts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve run/task provenance before filtering artifacts:** `src/gateway/server-methods/artifacts.ts:226-230`
  - body: The handler resolves a `runId` or `taskId` to a session, then drops every message whose transcript metadata does not carry the same id. Current `readSessionMessages()` attaches only `id` and `seq` for normal transcript entries, and this PR does not update the writers/readers to persist run/task provenance, so real artifacts can appear for `sessionKey` but disappear for the advertised scoped queries. Please persist/read explicit provenance before applying these filters, or limit the API to session-scoped queries until that exists.
  - confidence: 0.88
