## Summary

Keep open. Current main still has the Discord Components v2 transcript gap described by the linked bug, but this patch should not merge as-is because it changes shared captioned-media transcript behavior and leaves adapter-provided transcript metadata without a clear size or trust boundary.

## What This Changes

The PR adds Discord component transcript summarization, returns that summary as outbound result metadata, changes shared delivery mirroring to prefer enriched adapter text, and adds Discord/outbound regression tests.

## Best Possible Solution

Land a narrow enriched-transcript implementation that mirrors Discord component text and interactive labels through one idempotent write path, gates text-plus-media merging to explicit enriched adapter summaries, preserves ordinary captioned-media mirror semantics, and bounds or labels UI-derived text before it becomes assistant session memory.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new text+media branch in resolveMirroredTranscriptText runs for every caller, not only Discord component results. Current main's delivery mirror coverage expects a captioned media send to persist the filename (report.pdf) rather than caption plus filename, so this would change ordinary media transcripts across channels. Keep the old media-only behavior unless the caller explicitly passes enriched adapter transcript text.
  - confidence: 0.92
- **[P2] Deduplicate base text in mixed enriched batches:** `src/infra/outbound/deliver.ts:813`
  - body: In mixed result batches, prepending params.mirror.text to adapter transcript parts can duplicate content when enriched adapter transcript text already includes the same base message. Merge or dedupe the base text before appending adapter summaries so one mirrored assistant entry does not repeat itself.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff promotes generic adapter metadata into persisted assistant transcript text without a clear size or trust boundary.

Concerns:

- **[medium] Bound adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: deliverOutboundPayloadsCore would join meta.transcriptText from delivery results and persist it as assistant transcript text without a character, line, or result cap. Because channel adapters/plugins can populate result metadata, this should be bounded before it can inflate session files or downstream context.
  - confidence: 0.78
- **[medium] Treat UI-derived transcript text as lower-trust context:** `src/infra/outbound/deliver.ts:816`
  - body: The PR stores component labels/options and adapter-provided transcript metadata as an assistant message. That may be acceptable for context recovery, but it should be explicitly labeled or constrained so UI-derived text is not silently promoted to high-authority assistant instructions.
  - confidence: 0.72
