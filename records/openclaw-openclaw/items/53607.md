## Summary

Keep open: the PR targets a real Discord Components v2 transcript gap that still exists on current main, but the proposed patch should not merge as-is because it broadens shared media transcript behavior and leaves the adapter transcript metadata trust/size boundary unresolved.

## What This Changes

The PR adds Discord component transcript summarization, returns the summary as outbound result metadata, teaches shared delivery mirroring to prefer that metadata, adjusts media transcript resolution, and adds Discord/outbound regression tests.

## Best Possible Solution

Land a refreshed implementation that mirrors Discord component text and interactive labels through one idempotent outbound transcript path, keeps ordinary media filename mirroring unchanged, and bounds or labels UI-derived enriched transcript text before it becomes assistant session context.

## Reproduction Assessment

Yes. The source path on current main is enough to reproduce the underlying gap: Discord component sends register interaction entries but return only message/channel ids, while the shared mirror path ignores delivery result metadata. The PR regression is also reproducible from existing captioned-media mirror coverage that expects only the filename.

## Solution Assessment

No. The overall direction is right, but this patch is not the best mergeable form because it changes shared media transcript semantics and promotes adapter metadata into assistant memory without an explicit enriched-text boundary, size cap, or trust labeling.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new text+media branch in `resolveMirroredTranscriptText` runs for every caller, not only Discord component results. Existing delivery coverage expects a captioned media send to mirror `report.pdf`, but this change would persist `caption\nreport.pdf` across ordinary channels and add transcript noise. Keep the old media-only behavior unless the caller explicitly passes enriched adapter transcript text.
  - confidence: 0.92
- **[P2] Deduplicate base text in mixed enriched batches:** `src/infra/outbound/deliver.ts:815`
  - body: For mixed result batches, prepending `params.mirror.text` to `adapterParts` can repeat the same base message when the enriched component transcript already includes it, such as `Pick a color` plus `Pick a color\n[Red]`. Merge or dedupe the base text before appending adapter summaries so one mirrored assistant entry does not repeat itself.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The diff promotes channel adapter transcript metadata into persisted assistant transcript text without a clear size or trust boundary.

Concerns:

- **[medium] Bound adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: `deliverOutboundPayloadsCore` would join `meta.transcriptText` from delivery results and persist it as assistant transcript text without a character, line, or result cap. Channel adapters can populate this metadata, so it should be bounded before it can inflate session files or downstream model context.
  - confidence: 0.78
- **[medium] Treat UI-derived transcript text as lower-trust context:** `src/infra/outbound/deliver.ts:817`
  - body: The PR stores component labels/options and adapter-provided transcript metadata as an assistant message. That may be acceptable for context recovery, but UI-derived text should be explicitly labeled or constrained so it is not silently promoted to high-authority assistant instructions.
  - confidence: 0.72
