## Summary

Keep open: current main still lacks enriched transcript mirroring for Discord component payloads, so the PR addresses a real bug, but the proposed patch should not merge as-is because it changes shared captioned-media transcript behavior and leaves the adapter transcript metadata trust/size boundary unresolved.

## What This Changes

The PR adds Discord component transcript summarization, returns that summary as `meta.transcriptText` from the Discord outbound adapter, updates shared outbound mirroring to prefer adapter transcript text, and adds Discord/outbound regression tests.

## Best Possible Solution

Land a narrow enriched-transcript implementation that mirrors Discord component text and interactive labels through one idempotent write path, preserves ordinary captioned-media mirror semantics, and bounds or labels UI-derived transcript text before storing it as assistant session history.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Gate media transcript merging to enriched sends:** `src/config/sessions/transcript-mirror.ts:44`
  - body: The new text+media branch in `resolveMirroredTranscriptText` runs for every caller, not only Discord component results. Current main has coverage expecting a captioned media send to persist `report.pdf` rather than `caption\nreport.pdf`; this patch would change ordinary captioned media transcripts across channels and duplicate assistant text. Keep the old media-first behavior unless the caller explicitly passes enriched adapter transcript text.
  - confidence: 0.92

## Security Review

Status: needs_attention

Summary: The diff introduces a new adapter-result metadata-to-assistant-transcript path that needs bounded, labeled handling before merge.

Concerns:

- **[medium] Bound adapter transcript metadata before persistence:** `src/infra/outbound/deliver.ts:808`
  - body: `deliverOutboundPayloadsCore` would join `meta.transcriptText` from delivery results and persist it as assistant transcript text without a size or line cap. Because channel adapters/plugins can populate this metadata, the path should bound the stored text before it can inflate session files or downstream context.
  - confidence: 0.78
- **[medium] Treat UI-derived transcript text as lower-trust context:** `src/infra/outbound/deliver.ts:816`
  - body: The PR stores component labels/options and adapter-provided transcript metadata as an assistant message. That may be acceptable for context recovery, but it should be explicitly labeled or constrained so UI-derived text is not silently promoted to high-authority assistant instructions.
  - confidence: 0.72
