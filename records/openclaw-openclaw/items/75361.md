## Summary

Keep open. This is a live maintainer-labeled PR and current main does not already contain the stronger published-baseline Package Acceptance controls; the latest force-push appears to fix the prior regular survivor fixture regression, but the new baseline package input is still too permissive before `npm install`.

## What This Changes

The PR adds published-baseline controls to Package Acceptance and Docker E2E, splits regular versus published upgrade-survivor harness paths, seeds baseline config through JSON-backed CLI recipes, emits summary diagnostics, updates docs, and extends workflow/planner tests.

## Best Possible Solution

Keep the PR open, validate `published_upgrade_survivor_baseline` with the same strict OpenClaw package-spec rules used by package candidate resolution before any npm install, add focused coverage for rejecting alias/file-like specs, then rerun the targeted workflow/script tests plus both survivor Docker lanes.

## Reproduction Assessment

Yes. The review path is source-level: a dispatch value such as `openclaw@npm:<package>` matches the PR's `openclaw@*` branch and reaches `npm install` before any installed-version check can reject it.

## Solution Assessment

No. The published-baseline harness direction and the regular fixture split are maintainable, but the implementation should first constrain baseline specs to `openclaw@latest`, `openclaw@beta`, or exact OpenClaw release versions before installing packages.
