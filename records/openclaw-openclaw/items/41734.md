## Summary

Keep open. Current main does not contain the proposed `src/verify` module, and the PR is still useful feature work, but it is not mergeable as-is because the policy surfaces it parses have drifted: the tool catalog is now 32 tools, owner-only policy is approval-class based, and the subagent deny-list fixtures no longer match current runtime behavior. Prior automated comments correctly flagged parser soundness concerns; the latest sweep confirms those are still merge-blocking for security-verification tooling.

## What This Changes

The PR adds a new `src/verify` parser module with README, AST parsers, shared types, a Vitest parser suite, and a standalone runner for extracting tool catalog, alias, owner-only, subagent-deny, and policy-pipeline metadata for future SMT-LIB2 verification.

## Best Possible Solution

If maintainers want this verification lane in core, rebase the branch, update extraction for the current 32-tool catalog, approval-class owner-only map, and current subagent policy, make unsupported AST forms fail closed, and align the README with parser-only scope unless the generator and Z3 lane are added in the same reviewed change.

## Security Review

Status: needs_attention

Summary: The diff is inert at runtime, but it adds security-verification tooling whose stale policy target and fail-open extraction can produce misleading future proof inputs.

Concerns:

- **[medium] Verification parser can silently omit policy data:** `src/verify/parse-policies.ts:92`
  - body: Policy and catalog array helpers filter unsupported AST entries instead of failing on every unparsed element, which can weaken the extracted model while runtime still enforces the omitted policy entry.
