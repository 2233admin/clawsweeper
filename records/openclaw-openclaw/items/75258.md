## Summary

Keep this PR open. The patch moves reasoning suppression ahead of the channel-capability boundary, so it would drop opted-in Telegram reasoning while current main already has generic final/block guards for channels without a reasoning lane.

## What This Changes

The PR filters `isReasoning` payloads during final reply assembly and streaming normalization, then adds regression coverage for mixed reasoning and visible reply payloads.

## Best Possible Solution

Revise the hardening so generic/no-reasoning-lane dispatch suppresses marked reasoning, including tool-result delivery if needed, while preserving explicit Telegram `/reasoning on` and `/reasoning stream` lanes. Add paired coverage for WhatsApp suppression and Telegram opted-in delivery.

## Reproduction Assessment

Yes. The privacy boundary is reproducible with mixed `ReplyPayload[]` and reasoning-marked block/tool callbacks, and the PR regression is reproducible with a Telegram session at `reasoningLevel: "on"` receiving an `isReasoning` `Reasoning:` block or final payload.

## Solution Assessment

No. Filtering all marked reasoning inside final assembly and the shared streaming normalizer is earlier than the maintainable channel boundary; a capability-aware or generic-dispatch suppression point is safer.

## Review Findings
