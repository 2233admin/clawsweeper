## Summary

Keep this PR open for normal maintainer review. Current main already has downstream generic-dispatch guards for `isReasoning` payloads, but the PR still adds useful earlier defense-in-depth at final payload assembly and streaming normalization for the privacy-sensitive leak described in #74886; I did not find a blocking correctness or supply-chain issue in the provided diff.

## What This Changes

The PR filters `ReplyPayload.isReasoning` out of final reply payload assembly, skips flagged reasoning payloads in the shared streaming text normalizer, and adds regression coverage for mixed reasoning plus visible replies.

## Best Possible Solution

Keep the narrow delivery-boundary hardening, ensure it preserves the explicit `/reasoning` visibility behavior, and prefer reusing the existing shared reasoning-suppression helper so future semantics stay centralized.

## Reproduction Assessment

Yes for the narrow boundary this PR targets: a mixed `ReplyPayload[]` with `{ isReasoning: true }` plus a visible answer exercises `buildReplyPayloads()`, and current main lacks a final-assembly reasoning filter there. The broader WhatsApp instability in #74886 is not fully reproduced by this PR alone.

## Solution Assessment

Yes, this is a maintainable defense-in-depth layer for the reported leak path because it filters marked internal payloads before generic final and streaming delivery paths. A small cleanup to reuse `shouldSuppressReasoningPayload()` and optional streaming-callback regression coverage would reduce drift but does not make the current patch incorrect.
