## Summary

Keep open for normal PR handling: current main does not yet wire WhatsApp into the reasoning-stream callback path, and the latest branch is a narrow fix with no blocking correctness or security findings from static review. Existing main already suppresses marked final/block reasoning at the generic and WhatsApp delivery layers, while this PR covers the remaining stream callback path without removing explicit reasoning lanes for channels that support them.

## What This Changes

The PR adds a no-op WhatsApp `onReasoningStream` reply option, moves one skipped-turn reply-part calculation below the silent-turn guard, and adds focused tests that preserve reasoning-marked payloads for channel-specific lanes.

## Best Possible Solution

Land the narrow channel-boundary hardening that consumes WhatsApp reasoning streams as a no-op, keep generic marked-payload suppression in `dispatch-from-config`, preserve Telegram's explicit reasoning lane, and leave the broader WhatsApp session/fallback investigation to #74886.

## Reproduction Assessment

Yes. The static reproduction path is high confidence: current main only enables embedded stream reasoning when `onReasoningStream` exists, while WhatsApp does not provide that callback; the branch adds that callback and existing tests cover generic final/block suppression.

## Solution Assessment

Yes. For this PR's scope, the no-op callback at the WhatsApp boundary is the narrowest maintainable fix because it avoids core channel IDs or global reasoning drops and does not interfere with channels that intentionally expose reasoning.
