## Summary

Current main still does not implement the requested filesystem sandboxing config. The shipped surface is the narrower `tools.fs.workspaceOnly` boolean; `tools.fileAccess`, `mode`, `allowedPaths`, and `denyPaths` are not accepted by the strict schema, and arbitrary `exec` is still configured through exec policy rather than a filesystem allow/deny layer. The related open PR #60981 is useful implementation context, but it is unmerged and explicitly leaves arbitrary shell execution out of scope.

## Best Possible Solution

Keep this issue open until OpenClaw has a versioned filesystem access policy surface, likely under `tools.fs` or a successor config contract, with deny-wins `allowedPaths`/`denyPaths` or equivalent path rules enforced across read/write/edit/apply_patch/local media and an explicit exec strategy. Review #60981 as the current partial implementation candidate; if exec remains out of scope, split that gap into a canonical follow-up before closing this issue.
