## Summary

Keep #69066 open. Current main still has no first-class internal service identity axis: gateway auth modes/config/schema/protocol only cover existing user/shared auth paths, internal gateway clients still bootstrap with token/password, and trusted-proxy loopback remains fail-closed. The issue is also recent and has an active affected-use-case comment for Telegram native approvals, so it is not stale cleanup or a ClawHub/plugin item.

## Best Possible Solution

Keep the RFC open for maintainer gateway/security judgment. The best implementation path is an additive core gateway design: add typed gateway.service config/schema/docs, persist and rotate a service token, inject it only into gateway-owned children, extend the WS auth protocol/client bootstrap to carry service credentials, authorize service identity before user auth, migrate internal callers such as subagents, cron, browser tool, exec approvals, and Telegram native approvals, and cover trusted-proxy loopback internal callers with focused regression tests. If maintainers prefer narrower tracking, create a canonical core issue/PR for that service-identity work and supersede this RFC there.
