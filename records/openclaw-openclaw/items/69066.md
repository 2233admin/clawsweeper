## Summary

Keep open. Current main still lacks a separate internal service-identity axis: Gateway auth methods and config only model existing user/shared-auth modes, trusted-proxy still owns loopback handling with only a password fallback for local-direct clients, and GatewayClient still sends token/password/device credentials rather than a service credential. The issue is also a fresh auth RFC with active trusted-proxy deployment confirmations, so it needs maintainer security/design review rather than cleanup closure.

## What This Changes

The issue requests a first-class Gateway service identity mechanism separate from user auth so internal/local Gateway clients can authenticate without depending on trusted-proxy, password, or device-pairing fallbacks.

## Best Possible Solution

Keep the RFC open, agree on the service-identity trust model and phasing, then implement approved pieces as small additive PRs covering config/schema, auth resolution order, GatewayClient credential injection, role/scope limits, token storage/rotation/recovery, docs, doctor/migration behavior, and focused gateway auth regression tests.
