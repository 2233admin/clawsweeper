## Summary

Keep open. Current main still has no separate gateway service-identity path: internal/local clients continue to use the existing token/password/device credential flow, and trusted-proxy local-direct behavior remains governed by loopback rejection plus password fallback. The active user confirmations and nickytonline’s maintainer feedback make this a live auth-design RFC, not conservative cleanup.

## What This Changes

The RFC asks OpenClaw to separate gateway-internal service authentication from user-facing gateway auth modes, especially for local/internal callers in trusted-proxy deployments.

## Best Possible Solution

Keep the RFC open for maintainer auth/security design review, agree on the local service-auth trust model and Phase 0 sequencing, then implement approved service identity as small additive PRs covering config schema, SecretRef-backed token storage, GatewayClient injection, auth result/principal handling, docs, doctor/migration behavior, and targeted regression tests.
