## Summary

Keep this issue open. Current main at ef58307f843eed1a7832e9c23c519591f1b7cdbd has tactical trusted-proxy loopback/password fallback behavior, but it does not implement the RFC's separate internal service identity axis, service-token config/storage/rotation, `OPENCLAW_SERVICE_TOKEN` injection, `method: "service"`, or backend service principal scopes. The discussion also remains active, with a concrete Telegram native approvals reproduction in the issue comments and several related open reports/PRs in the provided timeline.

## Best Possible Solution

Keep this issue open as the canonical RFC/design thread. The best implementation path is an additive gateway service-identity design: new typed config/schema/docs for a generated service token and rotation command, explicit gateway-child env injection, `GatewayClient` support for service auth, a scoped service/backend role in auth and connect policy, migration of known internal callers, and regression tests around trusted-proxy, Telegram native approvals, subagents, cron, browser tooling, and exec approvals. Treat current password fallback and `allowLoopback` behavior as Phase 0 relief, not the final architecture.
