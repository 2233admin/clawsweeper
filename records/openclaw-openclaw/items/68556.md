## Summary

Keep this PR open. Current main has not implemented the PR's intended behavior: explicit custom QMD collection names are still preserved only for paths outside the current workspace, while workspace-local explicit names remain agent-scoped. The PR is a plausible behavior change, but it needs a migration/compatibility path and a maintainer product decision on multi-agent isolation before it can be treated as cleanup-obsolete or merge-ready.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to either preserve the current scoped behavior, or update the PR with a backward-compatible scoped-to-unscoped migration/rebind path, corrected test descriptions, and explicit coverage for multi-agent same-name custom paths before considering merge.

## Risks / Open Questions

- Merging the PR as-is would rename existing workspace-local explicit custom collections from `<name>-<agent>` to `<name>` without a proven scoped-to-unscoped migration path for older or metadata-poor QMD collection lists.
- Using the same explicit collection name across multiple agents with different workspace-relative paths could collapse separate agent collections into the same QMD collection name, which needs an explicit product decision or guardrail.
