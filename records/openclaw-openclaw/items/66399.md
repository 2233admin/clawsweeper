Current main still routes supervisor cancellation for exec and CLI agent runs through an immediate SIGKILL, while related comments and duplicate reports show the remaining user impact is real. ## What This Changes

The issue requests SIGTERM-first process-supervisor cancellation for exec timeouts/replacements, followed by bounded SIGKILL escalation and drain-timeout coverage.

## Best Possible Solution

Implement a focused core supervisor change that sends SIGTERM on cancellation, waits a bounded unref'ed grace period, escalates to SIGKILL if the run has not settled, and keeps the existing child/PTY wait fallback and process-tree safety contracts intact.

## Repair Work Prompt

Implement a focused fix for the process supervisor cancellation policy. Current main still calls `adapter.kill("SIGKILL")` immediately in `src/process/supervisor/supervisor.ts` when manual cancellation, scoped replacement, overall timeout, or no-output timeout fires. Change supervisor cancellation to send `SIGTERM` first, start an unref'ed bounded grace timer, and escalate to `SIGKILL` only if the run has not settled. Clear the escalation timer in settle/error/dispose paths. Preserve existing child adapter `killProcessTree(pid, { detached: childIsDetached })` behavior and PTY behavior; do not reintroduce process-group kills for non-detached children. Do not broaden this into agent-side SIGTERM handling; #71710 is separate. Update tests so no-output timeout, overall timeout, manual cancel/scoped replacement assert SIGTERM first and SIGKILL after the grace period, plus a case where exit after SIGTERM prevents SIGKILL. Verify the existing child/PTY SIGKILL wait fallbacks still work. Add a single-line changelog fix entry if the repository expects one for this user-facing behavior, crediting the reporter/context with `Thanks @kagura-agent` and any maintainer-approved additional reporter credit.

## Risks / Open Questions

- Windows signal semantics need platform-aware handling because Node does not provide POSIX-style SIGTERM behavior there.
- The fix must preserve the `detached:false` process-tree safety from #71662 so fallback-spawned or service-managed children do not group-kill the gateway process group.
- Agent-side SIGTERM responsiveness remains separate in #71710; supervisor escalation still needs a hard SIGKILL guarantee when the child ignores SIGTERM.
- Explicit pipe-closing before cancellation may have more blast radius than SIGTERM-first escalation and should be added only with focused tests.
