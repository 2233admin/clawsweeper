## Summary

Keep open. Current main still cancels supervisor-managed exec and CLI-agent runs through an immediate `adapter.kill("SIGKILL")`, and the supervisor tests still encode that behavior. The adapter-level wait fallback and process-tree safety work are partly present, but the requested supervisor-level SIGTERM-first grace period with hard-kill escalation remains missing.

## What This Changes

The issue requests changing process-supervisor cancellation for exec/CLI-managed runs from immediate SIGKILL to graceful SIGTERM first, bounded SIGKILL escalation, and reliable drain timeout behavior.

## Best Possible Solution

Implement a focused core supervisor change that sends SIGTERM on cancellation, waits a bounded unref'ed grace period, escalates to SIGKILL if the managed run has not settled, and keeps the existing child/PTY wait fallback plus kill-tree process-group safety contracts intact.

## Risks / Open Questions

- Windows signal semantics and `taskkill` behavior differ from POSIX, so the grace-period implementation needs platform-aware expectations rather than assuming `SIGTERM` is catchable everywhere.
- The related #71710 report suggests agent processes may ignore SIGTERM in some cron/provider paths; this issue still needs hard SIGKILL escalation and may not fully solve agent-side signal handling by itself.
- Any change must preserve the recent non-detached child safety guard so a group kill cannot signal the gateway's own process group.
