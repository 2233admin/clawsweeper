## Summary

Keep open. Current main does not already implement the requested composer affordance, and the PR remains useful, but latest head is not mergeable as-is because it still mirrors `pickerItems` observation on two sibling `PhotosPicker` views, which can duplicate selected attachments.

## What This Changes

The PR changes the shared SwiftUI chat composer to render selected attachments as thumbnail tiles with remove controls and a trailing add-more picker tile.

## Best Possible Solution

Keep this PR as the canonical UI implementation candidate, but update the branch so picker selection is observed exactly once from a stable parent view, with an empty-reset guard, then rerun Swift build plus real-device focused composer flows. Keep #73710 separate because it owns image processing and privacy, not this composer UI affordance.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: after at least one attachment, both toolbar and add-more pickers can be mounted with observers on the same `pickerItems`, so one selection can schedule two loader tasks before `pickerItems` is cleared; the author also reported this duplicate-load behavior during device testing.

## Solution Assessment

No. The feature direction is appropriate, but latest head is not the best implementation because per-picker observers duplicate shared state handling; a single parent-level `.onChange(of: pickerItems)` with an empty-array guard is the narrower maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Move picker observation to one stable owner:** `apps/shared/OpenClawKit/Sources/OpenClawChatUI/ChatComposer.swift:254-255`
  - body: The new add-more picker adds a second `.onChange` on the same `pickerItems` state while the toolbar picker still has its own observer. Once attachments exist and both picker views are mounted, a new selection can start two `loadPhotosPickerItems` tasks; because the loader clears `pickerItems` only after awaiting item loads, both tasks append the same selected images. Observe `pickerItems` once from an always-present parent and ignore the empty reset instead of mirroring the handler per picker.
