## Summary

Keep this PR open as a plausible focused iOS chat-composer improvement, but the latest head still has a blocking duplicate-processing bug and is missing the required changelog entry. The supplied discussion is useful: Greptile found the original focused-state silent-drop path, the author later acknowledged the dual-observer duplicate issue during device testing, and current head `fc916584d936236153a13d7abf33534869808acd` still contains the dual-observer implementation and stale idempotency comment.

## What This Changes

The PR changes `ChatComposer.swift` to replace the existing attachment capsule strip with thumbnail tiles, per-tile remove buttons, and a trailing add-more PhotosPicker/file-picker tile.

## Best Possible Solution

Keep this PR as the canonical UI candidate, but change the implementation to observe `pickerItems` exactly once from an always-present composer-level owner, ignore the empty reset, remove the stale dual-observer comment, add the required active changelog entry, and then validate toolbar pick, focused add-more pick, multi-select, remove, and send on a real iPhone.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on PR head `fc916584d936`: when the toolbar picker and add-more picker are both mounted, both observe the same `pickerItems` change while the loader clears only after awaited processing, so one picker selection can append duplicate attachments.

## Solution Assessment

No. The product direction is appropriate, but the current implementation is not the best fix; a single stable observer with an empty-selection guard is narrower and matches the bug root cause without concurrent duplicate processing.
