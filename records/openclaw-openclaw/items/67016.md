## Summary

Keep open. Current main still exposes only partial Dreams visibility: the gateway payload, UI controller, and view cover enabled state, aggregate counts, phase next-run timing, and diary content, but they do not expose the requested last-run timestamp, run outcome or partial-failure state, processed trace count, or diary updated/workspace status. The two existing ClawSweeper comments match the current code review, and no direct open PR was found that implements this specific request.

## What This Changes

Expose Dreaming/Dreams UI run metadata including last run time, processed item counts, diary update/workspace status, run result, and next scheduled run.

## Best Possible Solution

Add a memory-core-owned Dreams run summary with conservative multi-workspace semantics, expose it through the doctor Dreams status surface or a sibling status method, normalize it in the Control UI controller, render it in the Dreams tab, and update focused backend/UI/memory-core tests plus the Dreaming docs and changelog.
