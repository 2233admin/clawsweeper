## Summary

Keep this issue open. Current main has reduced the cold-start trigger by making gateway-start QMD refresh opt-in, but it still does not coordinate QMD update/search access across the long-lived gateway process and a transient CLI process. The update queue is process-local, `openclaw memory search` only waits for the same manager instance's pending update, and regression tests still treat `SQLITE_BUSY` during QMD result lookup as an error/fallback path rather than preventing the contention.

## What This Changes

The issue requests that QMD-backed `openclaw memory search` coordinate with gateway/background QMD memory work so ordinary runtime searches do not intermittently surface SQLite lock contention.

## Best Possible Solution

Add bounded per-agent cross-process coordination around QMD update/search/read access so gateway background QMD work and foreground CLI searches share a lock or wait/skip policy, while preserving lazy gateway startup, transient CLI managers, builtin fallback for hard QMD failures, and regression coverage for the gateway-process plus CLI-process race.
