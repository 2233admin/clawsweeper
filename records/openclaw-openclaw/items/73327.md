## Summary

Keep open. Current main still exposes only the global `diagnostics.stuckSessionWarnMs` warning threshold and applies it uniformly to every processing session, while cron jobs can configure execution `timeoutSeconds` but cannot declare a per-job stuck-warning threshold; the reported false-positive path remains actionable.

## What This Changes

The issue requests an additive per-cron-job stuck-session warning threshold so slow scheduled jobs can raise the warning window without changing the global default for other sessions.

## Best Possible Solution

Add a backward-compatible cron job threshold field, validate and persist it through cron normalization, protocol, tool, CLI, UI, and docs surfaces, capture the effective value on diagnostic session state when a cron run starts, and have the heartbeat use that per-session value with the current global/default threshold as fallback.

## Reproduction Assessment

Yes. A code-level reproduction is clear: simulate a cron-backed processing session that runs past the default 120 seconds while still healthy; current main compares it only against the one global threshold and emits `session.stuck`. The reporter also supplied production durations and successful `status=ok` run outcomes for the slow jobs.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it preserves the aggressive default for interactive sessions and fast jobs while letting known slow cron jobs opt into a higher SLA without downstream log filtering.
