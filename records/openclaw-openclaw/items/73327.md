## Summary

Keep open. Current main still applies one global `diagnostics.stuckSessionWarnMs` to every processing diagnostic session, while cron job types, Gateway protocol schemas, agent tool/CLI/UI surfaces, and docs do not expose a per-job warning threshold. The related stuck-session recovery and watchdog issues are adjacent but do not supersede this narrower false-positive control for legitimately slow cron jobs.

## What This Changes

Allow cron jobs to define a per-job stuck-session warning threshold so legitimately slow scheduled agent turns do not emit false-positive 120-second stuck-session warnings.

## Best Possible Solution

Add a bounded additive per-cron-job stuck-warning threshold, preserve the current global/default behavior when absent, capture the effective threshold into diagnostic session state at cron run start, and surface it through cron schema/tool/CLI/UI/status/docs without changing cron execution timeout or recovery semantics.

## Reproduction Assessment

Yes. A focused reproduction is to start the diagnostic heartbeat with default config, mark a cron-backed session as `processing`, and advance beyond 120 seconds; current main compares that session to the single global `diagnostics.stuckSessionWarnMs` threshold. The reporter's successful 130-210 second cron runs match that warning path.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it keeps the aggressive default for interactive sessions while allowing known slow cron jobs to declare their expected warning window. Raising the global threshold or reusing `timeoutSeconds` would mask unrelated sessions or change abort behavior.
