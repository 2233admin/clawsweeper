## Summary

Keep open. Current main has a configurable global stuck-session warning threshold, but cron jobs still cannot declare a per-job threshold, and the diagnostic heartbeat still applies one resolved warning value to every processing session. The reporter's slow-successful-cron false-positive path remains valid and is a narrow core cron/diagnostics fix rather than a duplicate of the broader recovery/watchdog issues.

## What This Changes

Allow cron jobs to declare an optional per-job stuck-session warning threshold so legitimately slow jobs do not emit warnings at the global/default threshold.

## Best Possible Solution

Add a backwards-compatible cron job threshold field, validate it through the cron schema/normalization/tool/CLI surfaces, capture the effective value on the diagnostic session when a cron run starts, and keep the existing global/default threshold as the fallback for all other sessions.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is clear: put a cron-backed diagnostic session in `processing` longer than the resolved global threshold and the heartbeat emits `session.stuck` because no per-job threshold is stored or consulted. The reporter also provided production durations where long cron runs finished with `status=ok` after warning logs.

## Solution Assessment

Yes. A per-job warning threshold is the narrowest maintainable fix because it preserves the aggressive default for interactive sessions and fast jobs while letting known slow cron jobs declare their expected SLA without downstream log filtering.
