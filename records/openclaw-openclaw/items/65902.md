## Summary

Keep this issue open. Current main still permits the reported mismatch: `openclaw status --json` can show persisted WhatsApp direct session rows from the session store, while agent-side `sessions_list` and `sessions_send` are still constrained by the default `tools.sessions.visibility="tree"` policy and can hide same-agent, non-current WhatsApp DM sessions that were not spawned by the requester.

## Best Possible Solution

Keep this issue open. Reproduce on current main with a linked WhatsApp account, `session.dmScope=per-channel-peer`, successful cron delivery, and agent-side `sessions_list`/`sessions_send` from the same automation context. If maintainers want on-demand automation to target existing same-agent WhatsApp DMs by default, implement a generic deliverable-session discovery or target-resolution path that respects `tools.sessions.visibility` and sandbox clamps, add WhatsApp direct coverage, and document the required visibility setting at the session-tools/config layer.
