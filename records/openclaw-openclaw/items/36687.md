Keep this open. Current main has shipped routing guardrails for the WebChat/TUI stale external-route leakage symptom, but it still treats an omitted `session.dmScope` as `main` and I did not find a current migration, doctor, or update-path fix that proves or repairs the specific 2026.3.2 `dmScope` reset claim. The maintainer comments in the issue already split those two concerns and left the reset-path question open for more evidence.

## Best Possible Solution

Keep this open for maintainer verification of the 2026.3.2 update/reset path. If update, doctor, onboarding, or config normalization can drop or ignore an existing isolated `session.dmScope`, add a targeted repair/migration plus regression coverage that preserves the field. If omitted `dmScope` intentionally remains `main` and no reset-path reproduction appears, close only after explicit maintainer product judgment while keeping the shipped stale-route guardrails.
