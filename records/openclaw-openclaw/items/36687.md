## Summary

Keep this issue open. Current main has shipped guardrails for the WebChat/TUI stale external-route leakage symptom, but it still intentionally treats an omitted `session.dmScope` as `main`; the narrower 2026.3.2 update/reset claim remains unverified rather than implemented or disproven.

## Best Possible Solution

Keep this open for maintainer verification of the 2026.3.2 update/reset path. If update/doctor/onboarding can drop or ignore an existing isolated `session.dmScope`, add a targeted migration or doctor repair plus regression coverage around preserving that field. If omitted `dmScope` intentionally remains `main` and no reset-path reproduction appears, close only after explicit maintainer product judgment while retaining the shipped stale-route guardrails.
