## Summary

Keep open. Current main only implements part of the surrounding session-memory work: it writes host-local slugged memory files and startup context now reads recent slugged artifacts, but main does not implement this PR's opt-in LLM synthesis mode or canonical daily-file append. The PR head still contains distinct unmerged behavior, so this remains a live implementation candidate for maintainer review rather than conservative cleanup.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase it on current main, account for the already-shipped slugged startup-memory preload and timezone fix, and decide whether core should own opt-in transcript synthesis. If yes, land the synthesis path and canonical append with dedupe-aware startup behavior, focused tests, docs, and explicit privacy/latency acceptance. If not, split the synthesis direction into the broader open memory-preservation discussion (#40418) before closing this PR.

## Risks / Open Questions

- Merging this PR would add an opt-in LLM pass over recent session transcript content during `/new` and `/reset`; maintainers should explicitly accept the privacy, latency, and model-cost tradeoff even though it follows an existing embedded-agent pattern.
- Current main already loads both canonical daily files and slugged same-day memory artifacts during startup. If the PR writes the same entry to both paths without startup-side dedupe, users may see duplicate memory context on the next session.
