## Summary

Keep open. Current main still has the unguarded OpenAI Completions tool payload path, and the PR’s transport guard is useful for #74664, but the branch also introduces an unrelated persistent Pi retry-setting change and one weak regression fixture, so it needs revision before merge.

## What This Changes

The PR adds a `supportsModelTools(model)` guard around OpenAI Completions `tools`/`tool_choice` emission and tool-history fallback tests, and also changes prepared embedded Pi settings to disable retry globally.

## Best Possible Solution

Keep the OpenAI Completions guard focused on #74664, remove or split the #73781 retry behavior unless it is implemented as an intentional runtime-only override, and update the fallback test to use real `toolCall` or `toolResult` history before normal validation.

## Reproduction Assessment

Yes. On current main, a direct call to `buildOpenAICompletionsParams()` with `compat.supportsTools: false` and non-empty `context.tools` still produces a `tools` payload, and the fallback path can be checked with real assistant `toolCall` or `toolResult` history.

## Solution Assessment

No. The transport guard is the right narrow fix for #74664, but persisting `setRetryEnabled(false)` during run preparation is not the best way to solve the separate replay-loop report; that needs either a runtime-only override or a separate reviewed change.
