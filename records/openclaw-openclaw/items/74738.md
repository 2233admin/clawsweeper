## Summary

Keep open. The requested OpenAI Completions `supportsTools: false` guard is still missing from current `main`, and this PR is a useful implementation direction, but the branch also changes Pi retry settings in a way that can persistently rewrite user agent settings under the trusted project-settings path. That needs revision before merge.

## What This Changes

The PR adds a `supportsModelTools(model)` guard around OpenAI Completions `tools`/`tool_choice` payload emission, adds regression tests for disabled tool support, and also disables Pi agent-level retry in prepared embedded Pi settings.

## Best Possible Solution

Keep the OpenAI Completions guard focused on #74664, fix the regression test to use real `toolCall`/`toolResult` history, and split or rework the #73781 retry behavior so any retry override is runtime-only, survives `resourceLoader.reload()` intentionally, and does not persist user Pi settings without explicit maintainer approval.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Avoid persisting Pi retry settings from run preparation:** `src/agents/pi-project-settings.ts:57`
  - body: `setRetryEnabled(false)` is not a transient override in `@mariozechner/pi-coding-agent`; it updates global settings and calls `save()`. When `createEmbeddedPiSettingsManager()` returns the file-backed manager for trusted project settings with no bundle settings, preparing an OpenClaw run can permanently write `retry.enabled: false` into the agent settings. Please make this a non-persistent runtime override or split the retry fix into a dedicated, reviewed change.
  - confidence: 0.9
- **[P3] Use real tool-history shapes in the fallback test:** `src/agents/openai-transport-stream.test.ts:2411-2420`
  - body: This fixture uses `type: "tool-call"` and `role: "tool"`, but `hasToolHistory()` only detects assistant `toolCall` blocks or `toolResult` messages. As written, the test would pass even if the fallback branch were not exercised; use the existing `toolCall`/`toolResult` message shape so it proves `tools: []` is suppressed.
  - confidence: 0.92
