## Summary

Keep open. Current main still lacks custom cron job ID support on the add path, and open PR #66023 is the active implementation candidate linked to this issue, so the issue should remain open until that PR is reviewed and merged or closed.

## What This Changes

The issue requests an optional `openclaw cron add --id <id>` flag so new cron jobs can use stable human-readable IDs while preserving the default UUID behavior.

## Best Possible Solution

Land a narrow cron feature PR that threads an optional id through the CLI, normalization, protocol schema, and service create path, validates path-safe unique IDs with run-log safety in mind, and adds focused CLI/protocol/service/docs coverage.

## Reproduction Assessment

Yes. A source-level check is enough: current main has no `--id` option in `cron add`, the gateway add schema rejects unknown `id`, and the service always calls `crypto.randomUUID()` when creating a job.

## Solution Assessment

Yes, the requested direction is maintainable if implemented end-to-end with conservative ID validation and collision checks. Because PR #66023 already carries that implementation, the best path is to review and land or close that PR rather than queue a separate repair lane now.
