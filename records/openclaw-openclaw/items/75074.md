## Summary

Keep open. Current main still has no opt-in Responses config for built-in tool-call capture, and both JSON and SSE `/v1/responses` paths only surface assistant output plus caller-provided client-tool `function_call` items. The issue is paired with open closing PR #75075, while parallel PR #75107 is closed unmerged, so the right next step is maintainer review of the open PR rather than issue closure or a duplicate repair job.

## What This Changes

Add an opt-in Gateway Responses flag that surfaces built-in tool-call start events as `function_call` output items in `/v1/responses` JSON and SSE output.

## Best Possible Solution

Review and land PR #75075, or replace it with an equivalent narrow default-off implementation that updates the Gateway Responses config contract, schema/generated baseline, JSON and SSE output, focused gateway tests, public docs, and changelog while preserving the existing client-tool delegate path.

## Reproduction Assessment

Yes. Static inspection gives a high-confidence reproduction path: run `/v1/responses` with an agent turn that invokes built-in tools, and current main has built-in `tool` start events available while the Responses output paths only include assistant messages and caller-provided `pendingToolCalls`.

## Solution Assessment

Unclear until the linked PR is reviewed. The default-off `function_call` audit-item approach is a plausible narrow solution, but maintainers should confirm security/redaction behavior, output ordering, and client-tool dedupe semantics before merge.
