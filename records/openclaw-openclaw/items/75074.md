## Summary

Keep open. Current main still does not expose built-in agent tool calls through `/v1/responses` JSON or SSE output, and this issue is already paired with open closing PR #75075, so the right cleanup action is maintainer review of that PR rather than closing this issue or queueing a duplicate repair.

## What This Changes

Add an opt-in Gateway Responses config flag that surfaces built-in agent tool starts as `function_call` output items in `/v1/responses` JSON and SSE responses.

## Best Possible Solution

Review and land PR #75075, or replace it with an equivalent narrow implementation that updates the config type/schema/generated baseline, JSON and SSE response output, gateway tests, public docs, and changelog while keeping the behavior opt-in and preserving the existing client-tool delegate path.

## Reproduction Assessment

Yes. A high-confidence static reproduction is to run `/v1/responses` for a turn that invokes built-in tools: current main emits built-in tool start events on the agent event bus, but the HTTP response output is built only from assistant payloads and `pendingToolCalls`.

## Solution Assessment

Unclear until the linked PR is reviewed. The requested default-off `function_call` audit item approach is a plausible narrow fix, but maintainers should verify security/redaction semantics, event ordering, and client-tool conflict behavior before merge.
