## Summary

Keep open. Current main still lacks the requested opt-in flag and still only returns assistant messages plus caller-provided client-tool calls, while built-in tool events remain available only on the agent event stream. The issue is also paired with open closing PR #75075, so the right cleanup outcome is maintainer review of that PR rather than closing or queuing duplicate repair work.

## What This Changes

Request an opt-in Gateway Responses flag that surfaces built-in agent tool calls as OpenAI-style `function_call` output items in `/v1/responses` JSON and SSE results.

## Best Possible Solution

Review and land #75075, or an equivalent narrow default-off implementation, covering config types/schema/generated docs, JSON and SSE output, client-tool dedupe, focused gateway tests, docs, and changelog. Tool results, compact digests, and built-in delegate mode should remain separate follow-up design work.

## Reproduction Assessment

Yes. A static reproduction path is high confidence: run `/v1/responses` with an agent turn that invokes built-in tools, and current main will return only assistant output unless `pendingToolCalls` contains caller-provided tools; meanwhile the embedded runner emits ignored `stream: "tool"` start events.

## Solution Assessment

Yes, the requested default-off `function_call` audit-item shape is a plausible narrow solution aligned with the Responses surface. The issue should stay open until the paired PR or equivalent code is merged and reviewed for security, ordering, and dedupe behavior.
