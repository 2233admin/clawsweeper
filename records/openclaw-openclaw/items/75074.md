## Summary

Keep open. Current main still omits built-in agent tool calls from `/v1/responses` output even though run-scoped built-in tool start events exist, and the issue is paired with open closing PR #75075, so the next step is maintainer review of that implementation rather than closure or a separate repair job.

## What This Changes

The issue requests a default-off Gateway Responses config flag that surfaces built-in agent tool calls as OpenResponses `function_call` output items in JSON and SSE responses.

## Best Possible Solution

Review and land PR #75075, or replace it with an equivalent narrow implementation that updates the Gateway config schema, generated config baseline, JSON and SSE Responses behavior, focused gateway tests, public docs, and changelog while keeping the feature opt-in and preserving the existing client-tool delegation path.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: run `/v1/responses` with an agent turn that invokes built-in tools, and the response output is built from assistant text plus pending client-tool calls only while built-in `tool` events remain outside the Responses output.

## Solution Assessment

Unclear until PR #75075 is reviewed. The default-off audit flag using existing sanitized tool-start events is a plausible narrow solution, but maintainer review should settle the HTTP exposure contract, ordering, and client-tool dedupe semantics.
