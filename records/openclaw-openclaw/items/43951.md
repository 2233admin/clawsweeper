Keep this PR open for maintainer review. The underlying bug is still present on current main, but the PR is not merge-ready because it is stale against the current hook-handler split, regresses the current announce message-tool policy, and introduces hook prompt/account-selection security concerns.

## What This Changes

The PR adds `accountId` parsing and forwarding for `/hooks/agent`, routes hook dispatch through cron normalization, adjusts cron message-tool policy, adds regression tests, and adds a changelog entry.

## Best Possible Solution

Land a rebased gateway fix that preserves and trims `accountId` through `HookAgentPayload`, hook replay scope, and `CronJob.delivery`, keeps sanitized hook metadata in prompt context, preserves the current announce message-tool behavior, documents hook account selection, and adds the required changelog credit.

## Reproduction Assessment

Yes. Current-main inspection gives a high-confidence reproduction path: send `/hooks/agent` with `accountId`, and the current normalizer plus dispatch builder cannot carry that field into `job.delivery`.

## Solution Assessment

No. Forwarding `accountId` is the right direction, but this PR is not the best merge-ready form because it targets stale hook-handler code, regresses current cron delivery policy, and misses security/docs/changelog follow-through.
