## Summary

Keep this PR open. Current main still does not preserve `accountId` through `/hooks/agent`, while the PR is paired with open #43866 from the same author and remains an implementation candidate. The change also needs maintainer review before landing because the cron tool-policy code on main has moved to `deliveryMode`/`forceMessageTool`, and forwarding caller-supplied account routing is security-sensitive.

## Best Possible Solution

Keep the PR open. Best path is to rebase onto current main, implement the narrow hook-owned fix by normalizing/trimming `accountId`, propagating it into the cron delivery object, including it in replay-key scope, surfacing dispatch normalization failures observably if the normalizer is used, adding focused hook normalization and end-to-end dispatch regression tests, updating the public webhook docs, and adapting or dropping stale cron tool-policy changes before maintainer review.
