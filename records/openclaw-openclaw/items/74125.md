## Summary

Keep open. Current main does not have evidence of a landed fix for the reported wake/system-event cold-start path. The code still builds CLI resume identity from the live heartbeat/cron-event context, while system-event turns can lack the original Discord chat metadata; if that changes the static prompt hash, `--resume` is skipped. The fallback `openClawHistoryPrompt` is also only available from compacted CLI transcript history, so ordinary un-compacted sessions can still arrive cold. The observability ask is valid too: the current `cli exec` info log omits `useResume`, trigger source, and session reuse status.

## Best Possible Solution

Keep this issue open and land a narrow runtime fix: wake/heartbeat/cron system-event turns targeting a persisted Discord Claude CLI session should preserve provider-owned CLI continuity when auth and MCP identity still match. Derive the stable CLI reuse prompt identity from persisted session route/chat metadata for system-event turns, or otherwise avoid treating missing system-event `ChatType` as a prompt-hash change. Add redacted resume diagnostics and a regression test that simulates a post-restart wake as the first turn after a Discord Claude CLI conversation.

## Clownfish Work Prompt

Fix #74125. Current main can deliver a first post-restart wake/system-event turn for a Discord Claude CLI session with heartbeat/cron-event context instead of the original Discord chat context. That can change `extraSystemPromptStatic`/`extraSystemPromptHash`, make `resolveCliSessionReuse` withhold the stored Claude CLI binding, skip `--resume`, and then fall back to an empty raw prompt when there is no compaction-derived `openClawHistoryPrompt`.

Expected fix boundary: preserve provider-owned CLI session continuity for wake/heartbeat/cron system-event turns targeting an existing session when auth profile/auth epoch/MCP identity still match. Prefer deriving the stable CLI reuse prompt identity from persisted session metadata/delivery context for system-event turns, or otherwise avoid treating missing system-event `ChatType` as a prompt-hash change. Do not weaken auth-profile, auth-epoch, MCP, real static prompt, or explicit reset invalidation. Do not add bundled plugin/channel ids to core outside existing generic contracts.

Also add safe observability: extend the `cli exec` info path with redacted diagnostics such as trigger source, `useResume`, session-id presence or short fingerprint, and invalidation reason. Do not log full argv, raw prompts, secrets, or unredacted tokens.

Add focused regression coverage for a persisted Discord channel/session entry with a `claude-cli` binding, then a post-restart cron/heartbeat/system-event wake as the first turn. The test should prove the stored CLI binding is still supplied and resume is selected, or that a non-cold history fallback is built if resume is intentionally invalidated. Keep normal user-message resume behavior covered. Add a user-facing changelog fix entry thanking @aounakram.
