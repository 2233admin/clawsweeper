## Summary

Keep open. Current main still has the fixed tool-schema and prompt overhead described in the report, and the discussion treats this as the canonical multi-step tracker rather than a completed or superseded item.

## What This Changes

The issue requests reducing per-session tool schema token overhead through channel-aware schema pruning, lazy or slim tool loading, shorter descriptions, and possible mega-tool splitting.

## Best Possible Solution

Keep this as the canonical performance tracker and sequence incremental core changes: remove or make optional duplicated Tooling text, avoid emitting definitions when no effective tools remain, prune high-footprint message fields by active channel and capability, shorten long descriptions where behavior stays covered, and design a compatible slim or lazy schema mode for plugins and declarative tools.
