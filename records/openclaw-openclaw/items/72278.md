## Summary

Keep open. Current main still has only the low-level SecretRef batching path: env/file/exec sources, exact SecretRef dedupe, grouping by source/provider, and no generic external-vault provider model or pass-scoped backend/entity memoization. A related pluggable-source PR exists, but it is open and does not make this implemented on main.

## What This Changes

The issue asks OpenClaw to add a first-class generic SecretRef external-vault integration model that batches refs per backend and memoizes backend/container/item/payload lookups during startup resolution.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported shared external-backend pattern, whether via a pluggable provider-source seam or an improved exec resolver contract, with backend identity in SecretRef ids, in-memory pass-scoped memoization for repeated backend lookups, updated configure/docs guidance, and regression coverage proving duplicate backend calls are avoided without disk caching or plaintext logging.
