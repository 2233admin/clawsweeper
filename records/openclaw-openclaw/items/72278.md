## Summary

Keep open. Current main still has only the generic env/file/exec SecretRef model, batches only at exact-ref plus source/provider granularity, and still documents 1Password/Vault examples in the one-exec-provider-per-secret shape this report asks OpenClaw to replace or steer away from. The related external-provider PR/planning threads are useful context, but they are closed/unmerged or narrower symptoms and do not supersede the remaining core product/API gap.

## What This Changes

The issue requests a first-class, backend-aware SecretRef integration model that batches and memoizes external vault lookups generically across providers such as 1Password, Vault, AWS, GCP, and Azure.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is one supported external-backend pattern where users configure one provider per backend/account/context, put backend secret paths in SecretRef ids, resolve active refs in one startup pass, memoize repeated backend entity lookups in memory only, and update configure/apply/docs/tests so the batchable pattern is the default guidance.

## Reproduction Assessment

Yes for the missing supported capability: current-main source and docs show only env/file/exec providers, source/provider batching, and per-secret exec examples. A live 1Password/Vault reproduction is not needed to prove that the requested first-class external-vault model is absent.

## Solution Assessment

Unclear as an implementation choice because this issue is asking for product/API design, not proposing a patch. The generic direction is sound, but maintainers still need to choose whether the best contract is an extended exec resolver, first-party adapter interface, plugin-backed provider API, or a combination of those.
