## Summary

Keep open. Current main still has only the generic env/file/exec SecretRef provider model, batches at exact-ref plus source/provider granularity, and still documents 1Password and Vault examples in the one-exec-provider-per-secret shape that this issue asks OpenClaw to replace or steer away from.

## What This Changes

The issue requests a first-class external SecretRef provider model that batches and memoizes external vault lookups across backend contexts instead of treating each secret as its own exec command.

## Best Possible Solution

Keep this as a core SecretRef product/API design item: define a supported external-backend pattern where one provider represents a backend/account/context, backend secret paths live in ids, one startup pass batches refs per backend, repeated container/item/entity lookups are memoized in memory only, and configure/apply/docs/tests steer users to that pattern by default.

## Reproduction Assessment

Yes for the missing supported capability: current source and docs show env/file/exec providers, exact-ref/provider grouping, and per-secret exec examples. The original 1Password Connect traffic numbers are not live-reproduced here, but the unbatchable documented shape is visible on current main.

## Solution Assessment

Unclear as an implementation choice because this is a product/API/security design issue, not a patch. The generic direction is sound; a narrow 1Password workaround or plugin-only answer would not fix the documented core configuration shape, but maintainers still need to choose the contract.
