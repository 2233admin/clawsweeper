## Summary

Keep open. Current main still exposes only env/file/exec SecretRef providers, batches at exact SecretRef plus source/provider alias granularity, and still documents 1Password/Vault examples in the one-exec-provider-per-secret shape this issue asks OpenClaw to replace or steer away from. The related external-provider PR/planning threads are closed/unmerged or narrower symptoms, so they do not supersede the remaining core SecretRef product/API gap.

## What This Changes

The issue requests a first-class SecretRef external-vault integration model with backend-level batching and pass-scoped memoization across all vault-style backends.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported external-backend pattern where one configured provider represents a backend/account/context, SecretRef ids carry backend secret paths, active refs resolve in one startup pass, repeated backend entity/payload lookups are memoized in memory only, and configure/apply/docs/tests steer users toward that batchable pattern by default.

## Reproduction Assessment

Yes for the missing supported capability: current-main source and docs show only env/file/exec providers, source/provider batching, and per-secret exec examples. A live 1Password/Vault reproduction is not needed to prove that the requested first-class external-vault model is absent.

## Solution Assessment

Unclear as an implementation choice because this issue is a product/API request, not a patch. The generic direction is sound, but maintainers still need to choose the narrowest maintainable contract and security boundary before code changes begin.
