## Summary

Keep open. Current main still matches the reported product/API gap: SecretRef providers are env/file/exec only, batching stops at the configured provider alias, and public external-vault examples still model 1Password and Vault as one exec command per secret with `id: "value"`. The related reports show useful symptoms, but none is a merged or canonical fix for this generic external-backend batching/memoization request.

## What This Changes

The issue requests a first-class SecretRef integration model that batches and pass-scope memoizes external secret backend lookups across vault providers rather than relying on one exec provider command per secret.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is a supported external-backend pattern where one provider represents a backend/account/context, SecretRef ids carry backend secret paths, startup resolution batches active refs per backend, repeated container/item/payload work is memoized in memory for that pass only, and docs/configure/apply/tests guide users away from per-secret exec commands.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: the schema and types only support env/file/exec providers, the resolver groups by exact provider alias, and the public 1Password/Vault examples still use per-secret exec providers with `id: "value"`.

## Solution Assessment

Unclear as a concrete implementation. The requested generic direction is maintainable, but maintainers still need to choose the durable contract across SecretRef config shape, exec compatibility, plugin/provider boundaries, configure/apply UX, docs, and secret-handling guarantees.
