## Summary

Keep open. Current main still matches the reported product/API gap: SecretRefs are limited to env/file/exec providers, batching deduplicates exact refs only within one configured source/provider alias, and the public external-vault docs still show one exec provider command per external secret with `id: "value"`. The linked 1Password reports are useful evidence, but they do not supersede this broader generic SecretRef design item.

## What This Changes

The issue requests a first-class external SecretRef provider model that batches active refs per backend/account context and memoizes repeated vault/entity/payload lookups during one resolution pass.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The desired end state is one supported external-backend pattern where users configure one provider per backend/account/context, put backend secret paths in SecretRef ids, resolve active refs in one startup pass, memoize repeated backend entity lookups in memory only, and update configure/apply/docs/tests so the batchable pattern is the default guidance.

## Reproduction Assessment

Yes. Current-main source and docs provide a high-confidence static reproduction: only env/file/exec providers are supported, resolver grouping is exact-ref plus source/provider scoped, and the public 1Password/Vault examples still use one exec command per secret with `id: "value"`.

## Solution Assessment

Unclear as a concrete implementation. The requested generic direction is sound, but maintainers still need to choose the durable contract across config schema, exec compatibility, plugin/provider boundaries, configure/apply UX, docs, and security guarantees.
