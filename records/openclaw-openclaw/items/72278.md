## Summary

Keep open. Current main still exposes only the generic env/file/exec SecretRef provider model, batches exact refs by source/provider, and documents/configures external vault examples in the per-secret exec-command shape this issue asks to replace or steer away from. The related closed/unmerged external-secret-provider work does not supersede the remaining core product/API gap.

## What This Changes

Request a first-class SecretRef pattern for external vault backends that batches and memoizes multiple refs across one backend/account/provider context instead of modeling each external secret as its own exec command.

## Best Possible Solution

Keep this as a core SecretRef product/API design item. The end state should be one supported external-backend pattern where users configure one provider per backend/account/context, put backend secret paths in SecretRef ids, resolve active refs in one startup pass, memoize repeated backend entity/payload lookups in memory only, and update configure/apply/docs/tests so the batchable pattern is the default guidance.

## Reproduction Assessment

Yes for the missing supported capability: current-main source and docs show only env/file/exec providers, exact-ref/provider grouping, configure file/exec choices, and per-secret exec examples. A live external-provider traffic run is not necessary to prove that the first-class external-vault model requested here is not implemented.

## Solution Assessment

Unclear as an implementation because this issue is a product/API design request, not a patch. The generic direction is sound, but maintainers still need to choose the durable contract; a narrow 1Password workaround or plugin-only answer would not address the documented config shape.
