## Summary

Current main still leaves the reported post-compaction context-switch behavior actionable. The default context engine is still `legacy`, its assembly path returns prior messages unchanged, and the embedded runner submits the fresh prompt beside the assembled history without the explicit `<next_user_message>` priority wrapper that exists only for CLI reseeding.

## Best Possible Solution

Keep this open and add a focused embedded/default-session regression: build a transcript with an old-topic compaction summary, append a fresh unrelated user prompt after the compaction boundary, and assert the assembled prompt or recorded trajectory treats the latest prompt as the active task while preserving the summary as historical context. If that fails, fix the generic embedded/default context assembly path and document the intended post-compaction semantics.
