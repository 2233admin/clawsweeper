## Summary

Keep open. Current main has useful compaction mitigations, including summary quality checks, recent-turn preservation, post-compaction AGENTS.md refresh, and CLI reseed handling. I still do not see source or regression coverage proving the specific reported behavior is fixed: a fresh unrelated user question after an old-topic compaction boundary should dominate, or the stale compaction summary should be explicitly demoted.

## Best Possible Solution

Keep this issue open. Add a focused regression around embedded/default prompt or context assembly: create a transcript with an old-topic compaction summary, append a fresh unrelated user message after the boundary, and assert the assembled prompt or trajectory makes the latest user request the active task while treating the summary as historical background. If that assertion fails, fix it in the generic prompt/context assembly layer or legacy context engine behavior, not with extension-specific topic heuristics.
