## Summary

Keep this issue open. Current main has several compaction mitigations, post-compaction refresh guidance, and a CLI reseed wrapper, but I do not see source or regression coverage proving that normal embedded sessions demote an old compaction summary when a fresh unrelated user question arrives after compaction.

## Best Possible Solution

Keep this issue open. Add a focused regression in the embedded/default context assembly path: build a transcript with an old-topic compaction summary, append a fresh unrelated user message after the compaction boundary, and assert the assembled prompt/trajectory makes the latest user request the active task while treating the summary as historical background. If it fails, fix the generic prompt/context assembly path rather than adding extension-specific keyword heuristics.
