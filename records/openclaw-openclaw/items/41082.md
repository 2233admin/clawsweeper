## Summary

Keep this issue open. Current main has useful compaction mitigations, including CLI reseeding and post-compaction AGENTS.md refresh, but I did not find a generic embedded/default-session guarantee that a fresh unrelated user request after compaction is prioritized over the old compaction summary.

## Best Possible Solution

Add a focused embedded/default-session regression that builds a transcript with an old-topic compaction summary, appends a fresh unrelated user message after the compaction boundary, and asserts the assembled prompt or trajectory treats the latest user request as the active task while keeping the summary as historical background. If it fails, fix the generic prompt/context assembly path rather than adding channel- or keyword-specific heuristics.
