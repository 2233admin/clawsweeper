## Summary

Keep this issue open. Current main still handles Discord reaction add/remove events as routed system events only; neither the plugin hook catalog/runner nor the internal hook docs/types/tests expose a reaction hook surface. Related #52658 and #60805 were closed toward this canonical hook/API gap, while #17840 separately tracks opt-in reaction-triggered agent turns.

## What This Changes

Add Discord reaction add/remove event support to the Hooks/plugin automation surface so automations can respond to user reactions.

## Best Possible Solution

Keep this issue open as the canonical Discord reaction hook/API tracker. The best implementation is an additive, documented reaction event surface that fires after existing Discord reaction authorization/filtering and route resolution, carries structured reaction metadata, updates plugin and internal hook docs/types/tests, and leaves reaction-triggered agent turns to #17840.

## Repair Work Prompt

Implement additive Discord reaction hook support for openclaw/openclaw#38714. Current Discord reaction add/remove handling in extensions/discord/src/monitor/listeners.ts authorizes, resolves channel/thread config, resolves the agent route, then enqueues a system event; add a hook/API delivery path after that existing authorization/filtering/routing point. Prefer structured reaction events such as plugin hooks `reaction_added` and `reaction_removed` plus matching internal hook events like `reaction:added` and `reaction:removed`, with typed metadata for action, emoji label/name/id, reactor id/name/tag, target message id, target author when available, channel/account/guild/thread/parent identifiers, sessionKey, and provider metadata. Update plugin hook types/runner/tests, internal hook types/docs/tests, Discord docs, and Discord reaction tests. Add a user-facing changelog entry only if behavior changes are implemented; credit @windhood-jza and other non-bot related reporters if appropriate, never @codex or @openclaw.
