## Summary

Keep #38714 open. Current main still receives Discord reaction add/remove events and queues them as session-scoped system events, but it does not expose reaction add/remove through internal Hooks, typed plugin hooks, or the plugin `message_received` bridge. Related #11747 remains a separate open reaction-wake/auto-response tracker; closed #52658 and #60805 were consolidated into #38714 rather than shipping a replacement.

## Best Possible Solution

Keep #38714 open until maintainers define and implement a Discord reaction event contract. The likely fix belongs in the Discord reaction listener plus internal/plugin hook surfaces: either add typed reaction add/remove hooks, or bridge authorized reactions into structured `message_received` and internal `message:received` events with emoji, action, reactor, target message, route/session, account/guild/channel/thread metadata, and removal state. Add Discord listener tests, plugin/internal hook tests, and public docs; keep #11747 focused on whether reactions should wake or auto-dispatch an agent turn.
