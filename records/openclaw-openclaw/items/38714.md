## Summary

Keep this issue open. Current checked-out main still treats Discord reaction add/remove as passive session-scoped system events, while internal hooks and plugin hooks expose message lifecycle hooks but no reaction event surface. Related duplicate reports were already closed toward this canonical hook/API gap, and the separate wake/auto-response behavior remains tracked by #17840.

## Best Possible Solution

Keep this issue open as the canonical Discord reaction hook/API tracker. The best path is an additive, documented reaction event surface that fires after existing Discord reaction authorization/filtering and route resolution, carries structured reaction metadata, and has focused Discord, plugin-hook, internal-hook, and docs coverage. Keep opt-in wake/auto-response behavior in #17840 unless maintainers deliberately combine the designs.
