The PR adds a shared suspicious-reply detector, wires a new `suspicious` skip reason into reply normalization, and adds normalizer/dispatcher tests for malformed internal orchestration text.

## Best Possible Solution

Land a narrow, rebased shared guard in the current reply normalization path that strips only structured internal route-marker/tool-JSON leakage, preserves media/channel data and `transformReplyPayload` behavior, honors custom silent tokens, and keeps focused regression coverage for both suppression and explanatory prose.

Yes. Source inspection gives a high-confidence focused path: current main sends final replies through `createReplyDispatcher` into `normalizeReplyPayload`, and that path has no suspicious route-marker/tool-JSON detector beyond existing silent, heartbeat, and generic sanitizer handling.

## Solution Assessment

No, not as currently written. The shared-normalization concept is the right direction, but the patch must be rebased onto the current `hasContent`/`hasReplyPayloadContent` shape before it can preserve current payload semantics.
