## Summary

Keep open. Current main still resolves isolated cron embedded-agent primaries from sub-agent model config, but the fallback resolver and embedded failover observation only consult payload, agent `model`, or default `model` fallback policy, so `subagents.model.fallbacks` can be skipped as reported.

## What This Changes

The issue reports that isolated cron embedded-agent runs time out on `kimi/kimi-code` without trying configured sub-agent fallback models.

## Best Possible Solution

Keep this open and add generic sub-agent-aware fallback-policy plumbing so the same `subagents.model` object that supplies the isolated cron primary also supplies its fallback chain, while preserving explicit payload fallback overrides and explicit empty fallback arrays.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or a per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and the current source path selects the sub-agent primary while fallback resolution and `fallbackConfigured` do not read those sub-agent fallbacks.

## Solution Assessment

Yes. The narrow maintainable fix is to make the fallback resolver follow the same sub-agent config source as model selection, rather than adding Kimi-specific timeout handling or changing provider policy.
