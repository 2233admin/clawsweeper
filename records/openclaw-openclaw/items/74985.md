## Summary

Keep open. Current main still has the reported sub-agent fallback-policy gap: isolated cron can select a primary from `subagents.model`, but the fallback override and embedded `fallbackConfigured` paths still resolve only the normal agent/default `model` fallback chain.

## What This Changes

The issue reports that isolated embedded agent runs select a sub-agent primary model but do not use the configured sub-agent fallback chain on timeout.

## Best Possible Solution

Add a generic sub-agent-aware fallback resolution path so the model config object that supplies the selected sub-agent primary also supplies its fallback chain when payload fallbacks are absent, while preserving payload fallback precedence, explicit `fallbacks: []` strictness, and existing timeout/failover classification.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and current main has no code path that reads those sub-agent fallbacks for the cron fallback override or embedded `fallbackConfigured` calculation.

## Solution Assessment

Yes. The narrow maintainable fix is fallback-policy plumbing for the sub-agent model selection source; a Kimi-specific workaround or timeout-classification change would not address the generic contract gap.
