## Summary

Keep open. Current main still has the reported gap: isolated cron can select a primary from `subagents.model`, but the fallback override and embedded `fallbackConfigured` paths only inspect payload fallbacks plus normal agent/default `model` fallbacks, so `subagents.model.fallbacks` can be skipped. The existing issue comment correctly identifies this as a narrow current-main bug, while #72984 is adjacent primary-selection work and #44353 is a broader provider-error fallback report rather than a complete duplicate.

## What This Changes

Reported bug asks isolated embedded cron agent runs to fail over from `kimi/kimi-code` to the fallback chain configured under `subagents.model` when the primary times out.

## Best Possible Solution

Add a sub-agent-aware fallback resolution path so the model object that wins isolated cron sub-agent primary selection also supplies its fallback chain when payload `fallbacks` is absent, while preserving payload precedence, explicit `fallbacks: []`, strict user selections, and existing timeout/error classification.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and inspect the current code path: primary selection reads that object, but fallback resolution never reads its `fallbacks`.

## Solution Assessment

Yes. The best fix is a narrow generic fallback-policy change shared by isolated cron and embedded failover observation, not a Kimi-specific workaround or a provider timeout-classification change.
