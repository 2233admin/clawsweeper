## Summary

Keep open. Current `main` still selects isolated cron embedded-agent primaries from `subagents.model` while fallback resolution and embedded failover observability continue to read only payload, normal per-agent `model`, or `agents.defaults.model` fallback policy, so `subagents.model.fallbacks` can be skipped exactly as reported.

## What This Changes

The issue reports that isolated embedded-agent cron runs select `kimi/kimi-code` from sub-agent model configuration but do not use the configured `subagents.model.fallbacks` fallback chain after a timeout.

## Best Possible Solution

Add generic sub-agent-aware fallback-policy plumbing so the same `subagents.model` source that provides the isolated cron primary also provides its fallback chain when payload fallbacks are absent, while preserving explicit `fallbacks: []`, payload fallback precedence, and non-sub-agent/user-session strict model semantics.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or `agents.list[].subagents.model` as an object with `primary: "kimi/kimi-code"` and non-empty `fallbacks`, run an isolated cron `agentTurn` without payload fallbacks, and the current source path selects the sub-agent primary but does not pass those sub-agent fallbacks into the fallback runner or `fallbackConfigured` observation.

## Solution Assessment

Yes. The best fix is a narrow shared resolver for the selected sub-agent model policy, not a Kimi-specific timeout workaround or a change to provider timeout classification.
