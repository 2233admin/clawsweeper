## Summary

Keep open. Current main at a102f4dede6af52523473ac29575c7e442493d15 still selects isolated cron embedded-agent primaries from `subagents.model`, but the fallback policy and embedded failover observation only read payload/default/per-agent `model` fallback settings, so `subagents.model.fallbacks` can still be skipped as reported.

## What This Changes

The issue requests that isolated cron embedded-agent runs use configured `subagents.model.fallbacks` when the primary Kimi model times out, instead of logging no fallback and failing silently.

## Best Possible Solution

Add generic sub-agent-aware fallback plumbing so the same `subagents.model` object that supplies the isolated cron primary also supplies its fallback chain when payload fallbacks are absent, while preserving explicit empty fallback arrays, user strict selections, timeout classification, and existing default-model fallback behavior.

## Reproduction Assessment

Yes. Configure `agents.defaults.subagents.model` or per-agent `subagents.model` as `{ primary: "kimi/kimi-code", fallbacks: [...] }`, run an isolated cron `agentTurn` without payload fallbacks, and the current source path selects the sub-agent primary while fallback resolution and `fallbackConfigured` do not read those sub-agent fallbacks.

## Solution Assessment

Yes. The narrow maintainable fix is a shared sub-agent-aware fallback resolver used by isolated cron execution and embedded failover observation, not a Kimi-specific timeout workaround or provider-policy change.
