## Summary

Keep this issue open. Current main still allows API/plugin-created cron jobs to become enabled without an origin-aware consent path, still lets plugin gateway-start hooks mutate cron jobs, still has bundled managed cron reconciliation that creates or re-enables a recurring job, and still resolves isolated cron jobs without an explicit model back to agent/default model selection. I did not find the requested cron diff/dry-run, expensive inherited-model guard, or cron-specific quota/cost circuit breaker.

## Best Possible Solution

Keep this open and implement a generic cron safety contract in core: add persisted/protocol origin and owner metadata for cron jobs; require explicit activation or a consent/diff flow for newly registered recurring external/plugin jobs; surface cadence, resolved model, inherited model source, and estimated call volume in doctor/update dry-runs; warn or block high-cost inherited defaults unless a job explicitly selects a model; and add a configurable usage/quota circuit breaker that can pause non-critical recurring cron jobs. Cover the public API, CLI, plugin hook surface, bundled managed-cron reconciliation, docs, and migration/backcompat behavior with focused tests.
