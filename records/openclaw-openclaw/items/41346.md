## Summary

Keep this issue open. The discussion has concrete reproduction data for v2026.3.7/v2026.3.8, and current main still lacks the requested cron safety contract: API-created cron jobs can default to enabled, plugin startup hooks can mutate cron, bundled memory-core reconciliation creates and re-enables an enabled recurring cron job, and isolated cron jobs without an explicit model still fall back to agent/default model selection. I did not find a cron-specific consent/diff flow, inherited expensive-model guard, doctor dry-run estimate, or quota/cost circuit breaker.

## Best Possible Solution

Keep this issue open and implement a generic cron safety contract in core: persist origin/owner metadata for cron jobs, require explicit activation or a consent/diff flow for newly registered recurring external/plugin jobs, show cadence/resolved model/inherited model source/estimated call volume in doctor or update dry-runs, warn or block high-cost inherited defaults unless a job explicitly selects a model, and add a configurable usage/quota circuit breaker that can pause non-critical recurring cron jobs. Cover the public cron API, plugin hook surface, bundled managed cron reconciliation, CLI/docs, and migration/backcompat behavior with focused tests.
