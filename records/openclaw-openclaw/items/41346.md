## Summary

Keep open. Current main still enables cron jobs by default when callers omit `enabled`, still allows agent/tool and Gateway cron registration without an origin-aware consent plan, still lets isolated cron jobs fall through to agent/default model selection when `payload.model` is absent, and still lacks the requested dry-run cost estimate or quota circuit breaker.

## Best Possible Solution

Design and implement an origin-aware cron safety contract in core: persist owner/origin metadata, require explicit activation or an interactive/non-interactive consent plan for newly registered external/plugin/agent-tool recurring jobs, show cadence/resolved model/model source/estimated daily call volume in doctor/update dry-runs, warn or block expensive inherited defaults unless a recurring job explicitly sets `payload.model`, and add a configurable quota/usage circuit breaker that can pause non-critical recurring cron jobs. Cover Gateway `cron.add`, CLI add/edit, the agent cron tool, plugin `getCron`, doctor/store migration, docs, and backcompat tests.

## Risks / Open Questions

- A blanket default-disabled change could break existing one-shot reminders, user-created CLI jobs, and managed recurring jobs that intentionally rely on enabled-by-default behavior.
- The fix needs origin/owner metadata and migration/backcompat rules so user-created jobs, agent-created jobs, plugin-managed jobs, and externally synced jobs can be treated differently.
- A quota circuit breaker needs provider-window semantics and a clear definition of critical versus pausable cron work to avoid silently stopping expected automation.
- Cost estimates can be approximate because pricing config, OAuth quota windows, provider usage surfaces, fallbacks, and tool-loop retries vary by provider.
