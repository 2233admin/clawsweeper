## Summary

Keep this open. Current main still allows cron jobs created through the public/API path to become enabled by default, still exposes cron mutation to plugin gateway-start hooks, still has at least one bundled plugin reconciler that creates or re-enables managed recurring cron, and isolated cron model selection still falls back to the agent/default model when a job has no explicit model. I did not find an origin-aware consent/diff flow, cost guard for inherited expensive models, or quota/usage circuit breaker for non-critical recurring cron jobs.

## Best Possible Solution

Keep this open and implement a generic cron safety contract in core: add cron job origin/owner metadata to the persisted/protocol model, require explicit activation or a consent/diff flow for newly registered recurring external/plugin jobs, surface cadence and resolved/inherited model in doctor/update dry-runs, warn or block high-cost inherited defaults unless the job has an explicit model, and add a configurable usage/quota circuit breaker that can pause non-critical recurring cron jobs. Cover the API, CLI, plugin hook surface, bundled managed-cron reconciliation, docs, and migration/backcompat behavior with focused tests.
