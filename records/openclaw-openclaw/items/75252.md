## Summary

Keep open. Current main still routes Mattermost draft-preview lifecycle transitions into the same editable post and does not call the existing Mattermost `forceNewMessage()` seam, so the reported overwrite/data-loss behavior remains plausible and source-backed. The related open streaming-mode work should be considered, but it is not evidence that this issue is fixed on main.

## What This Changes

Mattermost draft preview should preserve prior visible phase content by supporting a block-style preview mode and splitting draft posts at reasoning, assistant-message, and tool boundaries.

## Best Possible Solution

Keep the fix Mattermost-local: resolve the canonical preview streaming mode for the Mattermost account, honor `off`, preserve existing `partial` behavior by default, and in `block` mode call `draftStream.forceNewMessage()` at assistant, reasoning, tool-start, and tool-end boundaries before later updates can overwrite prior phase posts. Align schema/types/generated metadata/docs and add focused Mattermost monitor/draft-stream/account tests.

## Reproduction Assessment

Yes. The issue gives a live DM/tool-call path, and source inspection gives a high-confidence code reproduction: every transition writes through `draftStream.update(...)`, and the draft stream edits the existing `streamPostId` until `forceNewMessage()` or cleanup clears it.

## Solution Assessment

Yes, the requested direction is the narrow maintainable fix: match the existing Discord preview-boundary semantics inside the Mattermost plugin rather than changing shared core dispatch. The implementation should be coordinated with #73252 because that PR appears to cover related streaming-mode config plumbing.
