## Summary

Keep open. Current `main` still has the Gateway-side abort seam for agent RPC runs, but the one-shot `openclaw agent` CLI path still does not bridge OS `SIGTERM`/`SIGINT` from cron/coreutils `timeout` into the Gateway request, accepted run abort, or embedded/local run cancellation; the public docs also still lack the reported `timeout -k` mitigation for external cron wrappers.

## What This Changes

The issue requests bounded SIGTERM handling for cron-spawned `openclaw agent` processes, or at minimum documentation telling cron users to use SIGKILL escalation such as `timeout -k`.

## Best Possible Solution

Implement a focused one-shot CLI shutdown path: create a SIGTERM/SIGINT-owned `AbortController` for `openclaw agent`, pass it through `callGateway` and `GatewayClient.request`, use the known run id/session key to abort accepted Gateway runs, propagate cancellation into `--local` and embedded fallback, preserve session-lock cleanup, enforce a short grace window, and document `timeout -k` as compatibility guidance for external cron users and older releases.

## Reproduction Assessment

Yes. The issue gives concrete Linux cron/coreutils `timeout` steps, and current source supports a high-confidence unit reproduction by holding the Gateway request or embedded run pending, delivering SIGTERM, and observing that no CLI-owned abort or bounded exit path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort machinery is the narrowest maintainable root fix; documenting `timeout -k` is useful as an interim operator workaround but should not be the only product fix.

## Repair Work Prompt

Fix the cron SIGTERM leak reported in https://github.com/openclaw/openclaw/issues/71710. Implement a narrow shutdown path for one-shot `openclaw agent` runs: install SIGTERM/SIGINT handling in the CLI path, create a caller AbortController, propagate the signal through `callGateway` and `GatewayClient.request`, and when an agent RPC has a known idempotency/run id and session key, abort the accepted Gateway run through the existing `chat.abort`/`sessions.abort` machinery or an equivalent internal helper. Also propagate the same cancellation into `--local` and embedded fallback `agentCommand` runs, preserve existing session-lock cleanup behavior, and enforce a bounded grace window so cooperative work can drain but the process does not hang indefinitely. Add operator docs for external cron wrappers, including `timeout -k <delay> <duration> openclaw agent ...` as mitigation for older versions and non-cooperative supervisors. Do not broaden this into the exec/process-supervisor work tracked by #66399, do not change provider defaults, and do not add bundled plugin-specific behavior. Add focused tests for SIGTERM/SIGINT wiring, Gateway request abort cleanup, accepted-run abort dispatch, local embedded abort propagation, and docs coverage.
