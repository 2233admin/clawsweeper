## Summary

Keep open. Current main still has the Gateway-side abort seam, but the `openclaw agent` CLI path does not convert SIGTERM/SIGINT into a caller abort, does not pass a cancellation signal through `callGateway` or `GatewayClient.request`, and the public docs still lack the reported `timeout -k` mitigation for external cron wrappers.

## What This Changes

The issue reports that cron-wrapped `openclaw agent` processes do not exit on SIGTERM from coreutils `timeout`, causing long-lived process chains and host memory/swap exhaustion.

## Best Possible Solution

Implement a focused one-shot CLI shutdown path: create a SIGTERM/SIGINT-owned AbortController for `openclaw agent`, pass cancellation through `callGateway` and `GatewayClient.request`, use the accepted run id/session to abort the active Gateway agent run through the existing abort machinery, propagate cancellation into `--local` and embedded fallback, enforce a bounded graceful-exit deadline, and document `timeout -k` for external cron wrappers and older releases.

## Reproduction Assessment

Yes. The issue gives a concrete Linux cron/coreutils `timeout 600 openclaw agent ...` reproduction, and current source supports a focused unit reproduction by holding the Gateway or embedded agent request pending, delivering SIGTERM, and observing that no CLI-owned abort/bounded exit path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort seam is the narrowest maintainable root fix; adding `timeout -k` documentation is useful compatibility guidance but should not be the only fix.
