## Summary

Keep open. Current main still has the same product gap: the one-shot `openclaw agent` CLI waits on a Gateway request without a SIGTERM/SIGINT-owned cancellation path, `callGateway` and `GatewayClient.request` do not accept a caller abort signal, and the docs still do not describe the external `timeout -k` mitigation. The adjacent Gateway agent abort machinery exists, so this is a narrow repair rather than a duplicate of the broader process-supervisor reports.

## What This Changes

The issue asks for `openclaw agent` runs launched by external cron/coreutils `timeout` to handle SIGTERM with bounded cleanup and exit, or at minimum document `timeout -k` guidance for cron wrappers.

## Best Possible Solution

Implement a focused one-shot CLI shutdown path: create a process-signal-owned AbortController for `openclaw agent`, propagate it through Gateway and local embedded execution, abort the accepted Gateway run through the existing run-id/session abort machinery, preserve lock/runtime cleanup, enforce a small graceful-exit deadline, and document `timeout -k` as operator guidance for cron wrappers and older releases.

## Reproduction Assessment

Yes. The issue gives concrete Linux cron/coreutils reproduction steps, and current source supports a focused unit reproduction by holding the Gateway request or embedded run pending, delivering SIGTERM, and observing that no CLI-owned abort or bounded exit path is wired.

## Solution Assessment

Yes. Bridging process signals into the existing Gateway agent abort path is the narrowest maintainable root fix; documenting `timeout -k` is useful compatibility guidance but should not be the only fix.
