## Summary

Keep this issue open. Current main explains the reported 8 visible input tokens in the normal native subagent path because the full task is intentionally moved from the first user turn into `extraSystemPrompt`, and v2026.4.26 shipped that dedupe change for #72019. However, current main still has a plausible task-loss path for agents with `systemPromptOverride`: embedded, compaction, and CLI prompt assembly choose the override before calling the builder that receives `extraSystemPrompt`, so the delegated subagent task can still be dropped. The narrower open tracker #73624 covers that concrete remaining failure path, but this issue does not include enough config/trajectory detail to prove it is exactly the same case, so a conservative maintainer cleanup should leave it open for consolidation or repro.

## Best Possible Solution

Keep this issue open while maintainers confirm whether the affected agent had `systemPromptOverride` configured. If yes, consolidate with #73624 and fix prompt composition so `extraSystemPrompt` is appended to overrides in embedded, compaction, and CLI runner paths. If no, ask for a redacted trajectory/config excerpt showing the spawned runtime, context mode, `lightContext`, first child request prompt assembly, and token accounting so the broader delivery failure can be reproduced.
