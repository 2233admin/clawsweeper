## Summary

Keep open. Current main still lacks the proposed per-agent `env.scope` surface, so this PR is not obsolete, but the branch is not ready to merge: the diff has blocking environment-resolution bugs and is stale against current main's agent-scope and exec-tool structure.

## What This Changes

This PR adds per-agent `env` configuration, schema validation, an agent environment resolver, and exec-tool wiring intended to restrict inherited environment variables for collaborator agents.

## Best Possible Solution

Keep the PR open, but require a revised implementation on current main that adds the env config to the current agent-scope config seam, resolves agent vars whenever an agent env block exists, preserves empty string values, uses the current runtime env collector and host env sanitization rules, wires through the lazy exec path, and documents/tests the new contract.

## Reproduction Assessment

Yes. A source-level reproduction is clear: in the PR diff, an agent with `env.vars` and omitted/default `scope` never calls `resolveAgentEnvVars()`, and current main has no `env.scope` field while exec still inherits `process.env`.

## Solution Assessment

No. The feature direction is plausible, but this patch is not the best current solution because it skips default-scope vars, drops empty strings, uses a deprecated env collector, duplicates agent lookup, and needs to be rebased onto current main's split helper and lazy exec paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Pass agent vars for the default scope:** `src/agents/pi-tools.ts:470-472`
  - body: `scope` defaults to `all`, but this block only resolves `agentEnvVars` when `scope` is present and not `all`. An agent configured with `env.vars` and no explicit scope falls back to raw `process.env`, so the new per-agent vars/overrides are silently ignored.
  - confidence: 0.9
- **[P2] Preserve empty agent env values:** `src/agents/agent-scope.ts:412`
  - body: The `if (value)` guard drops valid empty-string values from `env.vars`. That breaks the stated highest-priority override contract and can leave an allowed inherited value in place instead of applying an intentional blank override.
  - confidence: 0.88
