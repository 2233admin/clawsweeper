## Summary

Keep this issue open. Current main has substantial partial hardening around structured SecretRefs, active runtime snapshots, command-scoped config resolution, host exec env sanitization, transcript/log redaction, and plugin hooks, but it still does not implement the native masked-secrets boundary requested here: no `{{secret:NAME}}` substitution for arbitrary agent-executed commands, no `secrets.mask` or `maskedSecrets` config contract, no exact configured secret mask list for arbitrary exec output, and no general policy preventing an agent from reading otherwise-readable env files or inherited environment values. A maintainer-associated comment in the provided thread also explicitly kept this open, and the linked external workarounds/plugins are useful mitigations rather than core closure evidence.

## Best Possible Solution

Keep this issue open until OpenClaw ships or explicitly supersedes a native masked-secret or credential-provider design for agent-executed commands and process output. A strong solution likely needs secops-owned design around dispatch-time credential injection, raw values kept out of agent-readable files/env/context, exact output redaction for configured secrets, and clear plugin/SDK hooks for external credential providers. Keep acknowledging `jmkritt/openclaw-secrets-hardening` and `openclaw-credential-vault` as current mitigations.
