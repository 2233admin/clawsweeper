## Summary

Keep this PR open. Current main still has the no-op process-supervisor orphan reconciliation contract, the process tool does not reconcile before serving requests, the linked same-author issue #65983 is still open, and the PR has unresolved review feedback for a real false-cancellation risk around `sessionKey`-owned runs.

## Best Possible Solution

Keep this PR open and request changes: make reconciliation ownership-aware across both generated process session ids and `sessionKey`, narrow the reconciliation reason type to forced-cancellation reasons, add regression coverage proving `sessionKey`-owned runs are preserved, and keep #65983 open for durable restart/cold-start orphan reaping unless that broader work is also implemented.

## Risks / Open Questions

- Merging the current PR as-is could cancel legitimate background execs whose supervisor record is owned by `sessionKey` while the process registry is keyed by generated session slug.
- The proposed `reason?: TerminationReason` type would allow natural-exit reasons like `exit`, `signal`, or `spawn-error` to be recorded for forced orphan cancellation.
- The PR intentionally handles only in-process/session-loss cleanup, so durable restart orphan reaping should remain tracked by #65983 or a narrower follow-up if a partial fix lands.
