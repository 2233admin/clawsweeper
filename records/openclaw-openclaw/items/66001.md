## Summary

Keep this PR open. Current main still exposes `reconcileOrphans()` as a deliberate no-op, the process tool does not reconcile before serving requests, and the PR's proposed slug-only ownership check has a real false-cancellation risk for runs owned by `sessionKey`.

## Best Possible Solution

Keep this PR open and request changes: make reconciliation ownership-aware across both generated process session ids and `sessionKey`, restrict reconciliation reasons to forced-cancellation reasons, add regression coverage proving `sessionKey`-owned runs are preserved, and keep #65983 open for durable restart/cold-start orphan reaping unless that broader work is also implemented.

## Risks / Open Questions

- The PR as written can misclassify active `sessionKey`-owned exec runs as orphaned because the proposed `hasSession` lookup only checks registry slug ids.
- The proposed `reason?: TerminationReason` type allows natural-exit reasons such as `exit`, `signal`, and `spawn-error` for forced orphan cancellation, which would make lifecycle records misleading.
- This PR intentionally addresses only in-process/session-loss cleanup; durable restart/cold-start orphan reaping from #65983 would still need separate design or follow-up work.
