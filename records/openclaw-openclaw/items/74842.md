## Summary

Keep open. Current main still intentionally defaults Discord guild/channel source replies to message-tool-only delivery, and the docs/changelog explain the `automatic` opt-in, but I did not find a Discord doctor warning or migration for upgraded configs that suddenly stop showing normal final replies. The remaining work is a narrow upgrade-diagnostics fix, not a duplicate of the shipped private-by-default behavior.

## What This Changes

Requests that Discord guild/channel replies either remain visible after upgrade or that OpenClaw migrate/warn when `messages.groupChat.visibleReplies` defaults suppress normal final replies.

## Best Possible Solution

Add an owner-local Discord doctor diagnostic, and optionally a `--fix` hint if maintainers want it, that detects configured Discord guild/channel usage with no explicit `messages.groupChat.visibleReplies` choice and explains that normal final replies are private unless the agent calls `message(action=send)` or the user sets `automatic`. Keep the shipped private-by-default runtime policy unless a maintainer explicitly chooses a broader product change.

## Reproduction Assessment

Yes. The source path and current tests give a high-confidence reproduction: a Discord guild message is treated as `ChatType: "channel"`, the default reply mode becomes `message_tool_only`, and visible final replies resume when `messages.groupChat.visibleReplies` is `automatic`. I did not run a live Discord gateway in this read-only review.

## Solution Assessment

No for reverting the default globally; that would conflict with the shipped #73046 behavior. The best narrow fix is a Discord doctor warning or repair hint for upgrade confusion, with targeted tests in the Discord doctor surface.
