## Summary

Keep open. Current main still does not implement `--profile-id` for `openclaw models auth login`, and the linked feature request #40402 remains open. This PR is a plausible implementation candidate, but it should not be auto-closed as implemented: the current source only supports `--profile-id` on `paste-token`, and the PR's generic post-login profile remap has a real provider-side-effect risk called out in review and confirmed against the GitHub Copilot provider code.

## Best Possible Solution

Keep this PR open for maintainer review rather than closing it as implemented. The best path is to revise it against current main: add the login flag, pass the requested target through a typed provider auth context/option, make OpenAI Codex honor it when building its OAuth result, avoid generic post-login result remapping, update docs and focused tests, and link the work to the open canonical feature request #40402.

## Risks / Open Questions

- Merging the PR as written could duplicate or misprioritize credentials for providers whose auth method writes credentials/config before returning, as GitHub Copilot currently does.
- A generic post-result profile-id rewrite does not give the provider a chance to choose the correct account/profile target during login; the safer implementation is an explicit provider-facing option consumed by providers that support named login profiles.
