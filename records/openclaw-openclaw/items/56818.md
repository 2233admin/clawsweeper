## Summary

Keep open. Current main still has a plausible path for the reported Feishu failure: the card-action parser prefers top-level `open_message_id` over `context.open_message_id`, so a callback payload with top-level `card-action-c-*` plus a real context `om_*` id can still route streaming through Feishu `message.reply` with the temporary callback id.

## Best Possible Solution

Fix this in the Feishu plugin: prefer or preserve the real `context.open_message_id` when the top-level `open_message_id` is a temporary `card-action-c-*` callback id, fall back to plain send/create mode when only a callback id is available, and add a lifecycle regression test with top-level `card-action-c-*` plus context `om_*`.

## Evidence

- **Issue-reported payload/error shape:** The issue body and first comment report that Feishu card action callbacks can carry a temporary top-level `open_message_id` like `card-action-c-xxx`, while the original message id is available in `context.open_message_id`; Feishu rejects the temporary id when used for streaming replies.
  - command: `provided GitHub context for issue #56818`

If both fields are present, the top-level callback id wins and is written back into both `open_message_id` and `context.open_message_id`, losing the original context id.
