## Summary

Keep this PR open. Current main still lacks Telegram `/new` and `/reset` control-lane routing, while the PR is a focused source/test change for a real Telegram sequentializer gap; the remaining question is maintainer review of state-changing reset commands bypassing an active lane.

## What This Changes

The PR refactors Telegram control-command resolution, routes registered `/new` and `/reset` session commands to the Telegram `:control` sequentializer lane, and adds sequential-key unit cases for those commands.

## Best Possible Solution

Land a scoped Telegram fix, if maintainers agree with the policy, that routes only registered `/new` and `/reset` lifecycle commands to the control lane while keeping `/export-session` and other mutating commands serialized; add key-resolution coverage and an active-run Telegram middleware regression showing resets bypass a busy lane without stale reply/finalization behavior.
