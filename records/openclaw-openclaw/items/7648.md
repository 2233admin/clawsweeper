## Summary

Current main still does not implement the requested Control UI/WebChat auto-scroll preference or a Scroll to Top action. Browser Control UI now respects manual scroll-up after the initial bottom pin, but send/initial-load/bottom-follow behavior remains hard-coded around the bottom, persisted UI settings and gateway config expose no scroll mode, and native WebChat remains bottom-oriented. The related #44130 is TUI-specific and does not supersede this browser/native WebChat request.

## Best Possible Solution

Keep this issue open. A conservative implementation should add an explicit chat scroll preference/action for Control UI and native WebChat, persist it in browser/native UI settings, and only add documented gateway-level config if maintainers want operator policy. Thread the mode through send, streaming, refresh, initial-load, manual-scroll, and bottom/new-message indicator behavior, with focused tests for disabled auto-scroll and/or a visible Scroll to Top action. Keep TUI-specific work tracked separately in #44130.
