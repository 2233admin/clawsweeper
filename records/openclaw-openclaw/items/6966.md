## Summary

Keep this issue open. Current main has strong adjacent handling for rate-limit failover, auth-profile rotation, model-scoped cooldowns, cooldown probing, and ordered configured fallbacks, but it still does not implement the requested dynamic rate-limit-driven model routing policy: no model strategy config, no per-model health store, no health-aware or round-robin selector, and no automatic downgrade plus upgrade-back behavior. The aashari fork linked in the discussion is useful prior art, not shipped upstream behavior.

## Best Possible Solution

Keep this open for maintainer product/design follow-up. A good upstream path would define a core model-routing policy such as failover versus health-aware/round-robin/downgrade routing, decide provider/model/profile health state and recovery semantics, expose observable downgrade and upgrade-back events, extend config schema and docs, and add focused regression tests for rate-limit scenarios. The aashari fork linked in the discussion should be treated as useful prior art rather than shipped OpenClaw behavior.
