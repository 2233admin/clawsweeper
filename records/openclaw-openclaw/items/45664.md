## Summary

Keep this PR open. Current main still does not implement the central BlueBubbles route-first behavior: `monitorBlueBubblesProvider` awaits `fetchBlueBubblesServerInfo` before registering the webhook target, so the startup/restart 404 window described by the PR still exists. The PR is not ready as-is because it bundles unrelated Slack/json-file changes plus a committed log artifact, and the Slack diff has a concrete unbounded body-read security concern from review comments.

## Best Possible Solution

Keep this PR open for a focused rebase: move `registerBlueBubblesWebhookTarget` before `fetchBlueBubblesServerInfo` in `extensions/bluebubbles/src/monitor.ts`, preserve server-info caching/logging, and add a regression test proving the route is registered while server-info is still pending. Split or separately decide the reconnect-attempt cap, and drop the Slack/json-file changes plus the committed log artifact unless they are reworked in their own reviewed PRs.
