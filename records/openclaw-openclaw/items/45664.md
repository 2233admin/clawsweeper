## Summary

Keep this PR open. Current main still does not implement the PR’s central BlueBubbles route-first change: `monitorBlueBubblesProvider` awaits the server-info fetch before registering the webhook route. The BlueBubbles change is directionally valid, but the PR is not ready as-is because it bundles unrelated Slack/json-files changes plus a committed log file, and review comments identify a concrete Slack webhook body-read security regression.

## Best Possible Solution

Keep the PR open and ask for a focused rebase against current main: move `registerBlueBubblesWebhookTarget` before `fetchBlueBubblesServerInfo`, preserve server-info caching/logging, and add a regression test proving the route is registered while server-info is still pending. Separately decide or split the reconnect-attempt cap. Drop the committed log file and split or safely rework the Slack/json-files changes, using bounded request-body reads if Slack url_verification handling remains in scope.

## Risks / Open Questions

- Merging the PR as-is could introduce a Slack HTTP webhook memory or slow-client DoS regression because the PR prereads the full request body before enforcing existing size and timeout limits.
- The PR bundles unrelated Slack url_verification handling, json-file retry logic, and a committed log artifact into a BlueBubbles startup-race fix, which makes review and rollback riskier.
- A focused update should account for current main’s later BlueBubbles catchup work without treating catchup as a substitute for registering the webhook route before the server-info fetch.
