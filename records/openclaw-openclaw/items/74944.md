## Summary

Keep open. The report is concrete and matches a plausible macOS launchd/Homebrew failure mode, while current main only shows partial mitigations: gateway startup/service PATH bootstrapping and no restart after a failed `update.run`. I did not find the reported `launchctl submit` updater script generator in this source tree or a regression test proving npm shims can resolve Homebrew Node from a submitted launchd job.

## What This Changes

The issue asks OpenClaw to make the macOS launchd self-update path resolve Homebrew-installed Node when running npm from a minimal launchd environment.

## Best Possible Solution

Add a small, deterministic update-command environment helper used by global package updates and any launchd update handoff script so npm shims can resolve the active Node/Homebrew Node path from a minimal launchd environment, with regression coverage for macOS/Homebrew PATH behavior and failed-update restart suppression.

## Reproduction Assessment

Yes, for the reported environment: a macOS Homebrew global npm install running an update through the launchd-submitted updater should reproduce by inspecting the update log for `env: node: No such file or directory` and npm exit 127. I did not live-run macOS launchd from this Linux checkout.

## Solution Assessment

Unclear. The suggested `export PATH=...` is directionally right, but the maintainable fix should centralize update PATH construction so it covers both current `update.run` global installs and any launchd handoff script without weakening service PATH hardening.
