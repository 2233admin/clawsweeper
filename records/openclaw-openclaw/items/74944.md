## Summary

Keep open. Current main still has a narrow updater reliability gap: the global npm update path can invoke an absolute Homebrew npm shim, but the install subprocess environment is not constructed with the active Node/Homebrew runtime path as a first-class updater contract. The report is concrete, unprotected, and already contains enough environment and log detail to make this a good focused repair candidate.

## What This Changes

The issue requests fixing macOS launchd-triggered self-updates so Homebrew npm shims can resolve `node` instead of failing with `env: node: No such file or directory` and npm exit 127.

## Best Possible Solution

Centralize update subprocess PATH construction so Gateway `update.run`, CLI package updates, and any launchd handoff run npm with a deterministic env that includes the active runtime dir, safe system dirs, the owning npm prefix bin, and Homebrew Node/bin locations, without adding project-local bins or weakening service PATH hardening.

## Reproduction Assessment

Yes. The report gives a high-confidence reproduction path: a Homebrew global npm install running a launchd-triggered update with PATH `/usr/bin:/bin:/usr/sbin:/sbin` should make npm's `env node` shim fail with exit 127; current source shows the relevant child env is not explicitly repaired in the global update helper.

## Solution Assessment

No. The suggested hard-coded shell `export PATH=...` would address the observed machine, but the maintainable fix is a shared updater env helper with tests for minimal launchd PATH, active runtime dir, owning npm prefix bin, and Homebrew npm shims.
