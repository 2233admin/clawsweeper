## Summary

Keep open. Current main still routes gateway and CLI global npm updates through an environment helper that does not add the active Node/Homebrew runtime path needed by npm shims under launchd's minimal PATH, so the reported macOS Homebrew failure remains actionable and narrow.

## What This Changes

The issue reports that macOS launchd-submitted OpenClaw self-updates fail for Homebrew global npm installs because npm's `env node` shim cannot find Homebrew Node under launchd's minimal PATH.

## Best Possible Solution

Centralize update subprocess PATH construction for global package updates so gateway `update.run`, CLI package updates, and launchd handoff use an env that includes the active runtime dir, safe system dirs, the owning npm prefix bin, and Homebrew Node/bin locations without adding project-local bins or weakening service PATH hardening.

## Reproduction Assessment

Yes. A Homebrew global npm install running `update.run` from launchd with PATH `/usr/bin:/bin:/usr/sbin:/sbin` should reproduce the npm shim `env node` failure; the report gives the exact paths and log signature, and current source still leaves that PATH unresolved.

## Solution Assessment

No for the literal hard-coded shell `export PATH=...` suggestion; it is a useful workaround but not the best repository fix. The maintainable fix is a shared update-env helper with regression coverage for minimal launchd PATH and Homebrew npm shims.

Fix openclaw/openclaw issue #74944. The observable bug is that macOS launchd-submitted gateway `update.run` / global npm self-update can call an absolute Homebrew npm shim while the subprocess PATH is still launchd-minimal, causing `env node` to fail with npm exit 127.
