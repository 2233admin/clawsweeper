## Summary

Keep open. Current main still has the nested exec-approval metadata bug this PR targets, but the branch is not merge-ready because it broadens approval-wrapper parsing while that parser is still applied to generic command output, and the latest PR diff still carries unrelated generated plugin SDK hash drift. The existing ClawSweeper and Aisle comments correctly identified the same security-sensitive parsing issue.

## What This Changes

The PR replaces exec approval denied/finished regex parsing with a balanced-parentheses metadata parser, adds nested-metadata parser and followup tests, and currently changes the generated plugin SDK API hash file.

## Best Possible Solution

Land a narrowed patch that keeps the balanced metadata parser and nested denied/finished regression tests, removes unrelated generated baseline churn, and constrains approval-result parsing to trusted exec lifecycle data or structured exec status before accepting broader wrapper syntax.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction: the denied/finished regexes stop at the first closing parenthesis, while the gateway emits `approval-timeout (allowlist-miss)` and followup routing relies on the denied parser branch.

## Solution Assessment

No, not as currently proposed. The parser direction addresses the reported bug, but the merge-ready fix should remove the generated baseline drift and close the spoofable generic-output parsing path first.
