## Summary

Keep this PR open. Current main does not implement Teams-specific live voice, transcript fallback, a Windows/.NET Teams media worker, Teams voice config, or the proposed plugin SDK surface. The PR is also draft and has an explicit maintainer review saying the direction is useful but not mergeable in this large shape, so the best maintainer path is review/splitting rather than auto-closing.

## Best Possible Solution

Keep the draft open and ask for a staged replacement plan: first land documented capability tiers, config schema, and no-op capability negotiation in the MS Teams plugin; then separately review the generic SDK/API contract if needed; then add the .NET worker and live audio pipeline only after `dotnet build`, mock gRPC integration, Microsoft permission/RSC validation, compliance gating tests, and a real Teams/Azure worker proof are available.

## Risks / Open Questions

- Security review pass: the PR proposes a new Windows/.NET media worker, gRPC bridge, Teams call media handling, app secret inputs, and wildcard .NET package references. Those are security- and supply-chain-sensitive changes that need explicit review, pinned dependency policy, and build/live proof before merge.
- The PR adds a plugin-specific public SDK seam in `src/plugin-sdk/msteams.ts`; current SDK boundary guidance prefers narrow generic seams or plugin-local API exports unless there is a justified reusable contract.
- The compliance behavior is high-risk because the PR handles live meeting audio and recording status. The maintainer comment already notes that audio processing must be proven blocked until `updateRecordingStatus` compliance is active.
