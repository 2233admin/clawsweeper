## Summary

Keep open. Current main still uses the numbered `.bak.1` through `.bak.4` config backup ring, and this PR is still a live implementation candidate paired with the same-author feature issue #39923. The remaining work needs maintainer consolidation with the parallel open PR #39982, especially around legacy backup migration and timestamp collision behavior.

## Best Possible Solution

Keep this PR open for maintainer review alongside #39923 and #39982. The best path is one consolidated datetime-backup implementation in `src/config/backup-rotation.ts` with collision-safe destinations, bounded retention, owner-only permission hardening, and a migration policy that preserves or safely converts legacy `.bak.N` recovery points until equivalent datetime retention exists; update `src/config/config.backup-rotation.test.ts` and add docs or changelog only if the behavior ships.

## Risks / Open Questions

- The PR design, as described in the body and review comments, can delete legacy `.bak.N` rollback points on the first config write after upgrade before the datetime backup set is populated.
- The timestamp-only destination still has a same-millisecond collision concern that can lose one historical backup slot depending on platform rename semantics.
- There is a parallel open implementation candidate in #39982, so maintainers should consolidate behavior, migration policy, and tests before landing either PR.
