## Summary

Keep this PR open. Current main and the latest release still use the numbered `.bak` / `.bak.1` through `.bak.4` config-backup ring, while this PR is an unmerged implementation candidate for the still-open datetime-backup request. The PR diff is scoped to config backup code and tests, but review comments identify unresolved migration and collision behavior that should be consolidated with the parallel open PR #39982 before landing either version.

Keep this PR open for maintainer review alongside #39923 and #39982. ## Risks / Open Questions

- Closing this PR now would be premature because current main and `v2026.4.26` still lack the requested datetime suffix behavior.
- The PR's proposed cleanup deletes legacy `.bak.N` files during migration before an equivalent datetime set is populated, reducing rollback depth on first upgrade.
- Timestamp-only destinations can collide during same-millisecond writes; the review thread documents this as a known edge case that can silently lose one historical backup on POSIX.
- Parallel open PR #39982 targets the same behavior with different collision handling, so maintainers should consolidate the design before landing either implementation.
- Config backups can contain credentials and tokens, so the final implementation must preserve owner-only permission hardening and avoid deleting useful recovery points during migration.
