## Summary

Keep open. The PR targets the right `memory status --deep` bug from #74544, but it should not merge as-is because the top-level budget race can leave the inner 600s embedding-batch timeout alive, verbose non-TTY progress can still be silent, and the changelog entry misses required contributor attribution.

## What This Changes

The PR changes the memory-core CLI to race the deep embedding probe against an 8-second budget, render a distinct timeout state, expose `timedOut` on embedding probe result types, add a probe-budget test, and update memory CLI docs plus the changelog.

## Best Possible Solution

Move the diagnostic budget into a path that does not start or leave behind the normal 600s batch timeout, or make the manager/provider probe cancellable with a short status-specific budget. Keep non-TTY progress visible for both verbose and non-verbose status runs, then add the required changelog credit and focused tests that cover the real manager probe timer behavior.

## Reproduction Assessment

Yes. #74544 provides user-level steps, and source inspection gives a high-confidence focused reproduction path: a cache-miss `memory status --deep` calls the live embedding batch probe, which uses the local 600s batch timeout before status output is rendered.

## Solution Assessment

No. The PR's top-level `Promise.race` is not the best fix because it does not cancel or avoid the inner `embedBatchWithRetry` timeout; a manager-level short-budget or cancellable probe is safer and matches the command-runtime requirement.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Avoid leaving the inner batch timeout alive:** `extensions/memory-core/src/cli.runtime.ts:122`
  - body: This races the public `manager.probeEmbeddingAvailability()` call, but on a cache miss that method has already entered `embedBatchWithRetry(["ping"])`, whose `withTimeout` creates the normal local 600s timer. Because the PR does not cancel or avoid that inner probe, the CLI can report the 8s timeout while the ref-counted 600s timer still keeps the process open, so the command runtime is not actually bounded.
  - confidence: 0.92
- **[P2] Keep verbose non-TTY progress visible:** `extensions/memory-core/src/cli.runtime.ts:732`
  - body: When `--verbose` is set this chooses `fallback: "line"`, but `createCliProgress` only renders `line` on TTY streams and returns a no-op for non-TTY unless the fallback is `log`. As a result, `openclaw memory status --deep --verbose | cat` still suppresses the `Probing ...` labels during the diagnostic wait, which leaves part of the reported non-TTY problem unfixed.
  - confidence: 0.9
- **[P2] Add the required changelog credit:** `CHANGELOG.md:9`
  - body: The repo changelog policy requires every added user-facing changelog entry to include at least one `Thanks @author` attribution. This new fix entry only has `Fixes #74544`, so it should credit the contributor before merge.
  - confidence: 0.97
