## Summary

Keep open. Current main still preserves a raw `x-openclaw-session-key` as the execution session key while the selected model agent is not passed as an explicit execution override, so a non-`agent:<id>:` session key can still route through the configured default agent. The related open PR #69060 overlaps the header contract but intentionally preserves the existing full-override semantics, so it does not supersede this report.

## What This Changes

The issue requests that `/v1/chat/completions` honor `model: "openclaw/<agentId>"` routing even when the request also sends `x-openclaw-session-key`.

## Best Possible Solution

Define a deterministic HTTP session contract, then implement it consistently: the selected agent from `model` or `x-openclaw-agent-id` should be pinned into execution unless the caller supplies a valid canonical `agent:<id>:` session key that intentionally targets another agent, and docs plus regression coverage should cover both `/v1/chat/completions` and `/v1/responses`.

## Reproduction Assessment

Yes. Static proof on current main shows a raw header is forwarded as the session key, no explicit agent override is forwarded, and non-agent session keys resolve to the configured default agent; a focused HTTP test can reproduce by combining `model: "openclaw/beta"` with `x-openclaw-session-key: my-session-key`.

## Solution Assessment

Unclear until the API contract is decided. The reporter's desired behavior is reasonable, but the current docs and #69060's design preserve `x-openclaw-session-key` as a full override, so the safest path is maintainer-approved precedence followed by code, tests, and docs together.
