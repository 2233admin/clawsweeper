## Summary

Keep open. Current main still has the reported routing hazard: the raw `x-openclaw-session-key` header is preserved before agent-scoped key generation, while `/v1/chat/completions` does not pass the model-selected agent as an execution override, so downstream routing can fall back to the default agent for non-`agent:<id>:` session keys. The docs and related open PR #69060 preserve full-override semantics for that header, so this needs maintainer API-contract judgment rather than cleanup closure.

## What This Changes

The issue requests that `/v1/chat/completions` respect `model: "openclaw/<agentId>"` routing even when callers provide `x-openclaw-session-key` as the session identifier.

## Best Possible Solution

Define the public HTTP session-routing contract, then align `/v1/chat/completions` and `/v1/responses` implementation, docs, and regression tests around that contract. The end state should make raw session hints, canonical `agent:<id>:` session keys, `model`, `x-openclaw-agent-id`, and response/session reuse behavior unambiguous.

## Reproduction Assessment

Yes. The static current-main path is high-confidence: a request with `model: "openclaw/cathy"` and raw `x-openclaw-session-key: my-session-key` keeps `my-session-key`, passes no `agentId` override, and downstream agent selection derives from the non-agent session key as the default agent.

## Solution Assessment

Unclear. The reporter’s requested precedence may be right for API ergonomics, but it conflicts with current docs that the header fully controls session routing; the safer path is a maintainer-owned contract decision before an automated code change.
