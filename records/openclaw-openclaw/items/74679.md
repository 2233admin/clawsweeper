## Summary

Keep open. Current main still has the reported routing hazard: `/v1/chat/completions` resolves `model: "openclaw/<agentId>"` to an agent, but a raw `x-openclaw-session-key` is returned unchanged and the selected agent id is not forwarded as an execution override. The public docs also describe that header as full session-routing control, and related open PR #69060 preserves that precedence rather than resolving this report.

## What This Changes

The issue requests that `x-openclaw-session-key` on `/v1/chat/completions` provide a session key without overriding the agent selected by `model: "openclaw/<agentId>"`.

## Best Possible Solution

Define and document the public precedence contract for full-routing session keys versus agent-scoped session hints, then make Chat Completions and OpenResponses consistent with regression tests for raw headers, canonical `agent:<id>:` headers, `x-openclaw-agent-id`, model-selected agents, and previous-response/session reuse.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: send `model: "openclaw/beta"` with raw `x-openclaw-session-key: my-session-key`; current main forwards that raw key without an explicit `agentId`, and the execution path falls back to the default agent for non-`agent:<id>:` keys.

## Solution Assessment

Unclear. The requested behavior may be the right operator experience, but it conflicts with the documented full-override meaning of `x-openclaw-session-key`; the safer maintainable path starts with a maintainer API-contract decision, possibly using a separate agent-scoped hint header like #69060.
