## Summary

Keep open. Current main still has the reported routing hazard: a raw `x-openclaw-session-key` header is preserved as the execution session key, and the selected `model: "openclaw/<agentId>"` agent is not pinned into the downstream agent command, so non-`agent:<id>:` session keys route through the default agent. The remaining decision is the public HTTP session contract because docs currently describe `x-openclaw-session-key` as a full routing override and related PR #69060 intentionally preserves that behavior.

## What This Changes

The issue requests that `/v1/chat/completions` honor `model: "openclaw/<agentId>"` agent routing even when callers provide `x-openclaw-session-key`.

## Best Possible Solution

Define one canonical public contract for raw versus agent-scoped session keys across `/v1/chat/completions` and `/v1/responses`, then update code, docs, and regression tests together. If raw `x-openclaw-session-key` remains a full override, callers need a documented agent-scoped alternative such as #69060; if `model` should dominate, the Gateway should canonicalize raw keys for the selected agent or pass an explicit agent override into execution.

## Reproduction Assessment

Yes. The source-level path is high-confidence: send `model: "openclaw/cathy"` with raw `x-openclaw-session-key: my-session-key`, and current main forwards `my-session-key` without an agent override, causing agent execution to derive the default agent from a non-canonical key.

## Solution Assessment

Unclear. The issue's requested precedence is a plausible user fix, but it conflicts with the documented full-override semantics for `x-openclaw-session-key`; the narrow maintainable fix depends on maintainer API-contract judgment first.
