## Summary

Keep open. Current main still has the reported routing hazard: the Chat Completions request resolves the model-selected agent, but a raw `x-openclaw-session-key` is forwarded unchanged and no explicit `agentId` is sent to execution, so downstream agent selection falls back to the default agent for non-`agent:<id>:` keys. The docs and related open PR #69060 also treat this header as a full session-routing override, so this needs maintainer API-contract judgment rather than cleanup closure.

## What This Changes

The issue requests that `/v1/chat/completions` respect `model: "openclaw/<agentId>"` agent routing when `x-openclaw-session-key` is supplied as the session identifier.

## Best Possible Solution

Define the public precedence contract for full-routing session keys versus agent-scoped stable session hints, then make Chat Completions and OpenResponses consistent with regression tests for raw headers, canonical `agent:<id>:` headers, model-selected agents, `x-openclaw-agent-id`, and previous-response/session reuse.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a Chat Completions call with `model: "openclaw/beta"` plus raw `x-openclaw-session-key: my-session-key`; current main forwards the raw key without `agentId`, and execution resolves that non-agent key to the default agent.

## Solution Assessment

Unclear. The requested behavior may be the right operator experience, but it conflicts with the current documented meaning of `x-openclaw-session-key` and with #69060's stated preservation of that precedence, so a maintainer should decide the API contract before implementation.
