## Summary

Keep open. Current main still has the reported routing hazard, and the behavior conflicts with the documented public meaning of `x-openclaw-session-key` as full session-routing control, so this needs maintainer API-contract judgment rather than cleanup closure.

## What This Changes

The issue requests that `x-openclaw-session-key` on `/v1/chat/completions` carry a session key without overriding the agent selected by `model: "openclaw/<agentId>"`.

## Best Possible Solution

Define the public precedence contract for full-routing session keys versus agent-scoped session hints, then make Chat Completions and OpenResponses consistent with regression tests for raw headers, canonical `agent:<id>:` headers, model-selected agents, `x-openclaw-agent-id`, and previous-response/session reuse.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: send `model: "openclaw/beta"` with raw `x-openclaw-session-key: my-session-key`; current main forwards the raw key without an explicit agent override, so execution falls back to the default agent for a non-`agent:<id>:` key.

## Solution Assessment

Unclear. The requested behavior may be the right operator experience, but it changes a documented HTTP contract; a safer maintainable path starts with a maintainer decision on whether to change `x-openclaw-session-key` or steer callers to a separate agent-scoped hint such as #69060.
