## Summary

Keep open. Current main still preserves a raw `x-openclaw-session-key` as a full session-routing override, so a `/v1/chat/completions` request can resolve `model: "openclaw/<agentId>"` and still execute under the default agent when the supplied session key is not `agent:<id>:...` shaped. The remaining blocker is a public HTTP API contract choice, because docs currently describe both model-based agent routing and full session-key override behavior, while related open PR #69060 proposes a separate agent-scoped session header that intentionally preserves the existing override semantics.

## What This Changes

The issue asks for `/v1/chat/completions` to honor `model: "openclaw/<agentId>"` agent routing when callers also send a raw `x-openclaw-session-key` header.

## Best Possible Solution

Choose and document the HTTP session contract, then implement it consistently across `/v1/chat/completions` and `/v1/responses`: either scope raw session hints under the selected agent while preserving canonical `agent:<id>:...` keys, or land a separate scoped-session header and make the existing full-override semantics explicit. Add focused regression coverage for raw header plus model, canonical header plus model, `x-openclaw-agent-id`, and response-continuation parity.
