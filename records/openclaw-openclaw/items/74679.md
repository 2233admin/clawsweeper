## Summary

Keep open. Current main still has the reported routing hazard, and the remaining question is the public HTTP session contract rather than cleanup closure: a raw session-key header is preserved as the routing key while the selected model agent is not pinned into execution.

## What This Changes

The issue asks `/v1/chat/completions` to respect `model: "openclaw/<agentId>"` routing when callers also send a raw `x-openclaw-session-key` header.

## Best Possible Solution

Define the public contract for raw versus canonical session headers, then implement and document it consistently across `/v1/chat/completions` and `/v1/responses`, with regression coverage for raw header plus `model`, canonical `agent:<id>:...` header, `x-openclaw-agent-id`, and previous-response/session reuse behavior.
