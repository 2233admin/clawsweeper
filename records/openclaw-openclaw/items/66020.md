## Summary

Keep open. Current main still rejects the reported `<qqmedia>` path shape when the existing local file is outside OpenClaw-managed media roots, and the requested auto-mirroring changes a deliberate QQ Bot outbound local-file security boundary rather than an already-implemented bug fix.

## What This Changes

QQ Bot should copy existing local `<qqmedia>` files from outside QQ Bot media storage into `~/.openclaw/media/qqbot/` before sending them.

## Best Possible Solution

Define a security-preserving QQ Bot staging path for outbound media: either keep requiring agents/tools to create files under OpenClaw media storage, or add a constrained mirror helper limited to explicitly trusted OpenClaw-owned output roots with canonicalization, symlink/traversal protections, size checks, and regression coverage.

## Reproduction Assessment

Yes. The issue body gives concrete steps, and current-main source inspection shows an existing workspace file outside the media roots still reaches `resolveOutboundMediaPath` and returns the reported storage-boundary error.

## Solution Assessment

No for unrestricted auto-mirroring. The maintainable path is a maintainer-approved, narrowly scoped staging rule or helper that preserves the existing local-file security boundary instead of copying arbitrary absolute paths.
