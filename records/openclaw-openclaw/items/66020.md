## Summary

Keep open. Current main still rejects existing local `<qqmedia>` paths outside the OpenClaw/QQ Bot media roots instead of mirroring or staging them, and the discussion correctly flags arbitrary-file disclosure risk if this is fixed too broadly.

## Best Possible Solution

Keep this open and design a QQ Bot-owned constrained staging path: accept only files proven to be inside agent-scoped workspace/media allowlists, realpath-check and size/type-check them, copy them into managed QQ Bot outbound media storage, and preserve rejection for outside-root and symlink-escape paths. The fix should route `<qqmedia>` through that path and update QQ Bot tests plus qqbot-media skill guidance.
