## Summary

Keep open. Current main still intentionally rejects existing local `<qqmedia>` files outside the OpenClaw media root, and the discussion correctly flags that automatic mirroring could weaken a local-file exfiltration boundary.

## What This Changes

Allow QQ Bot outbound `<qqmedia>` replies that reference existing local files outside `~/.openclaw/media/qqbot/` to mirror those files into QQ Bot media storage before sending.

## Best Possible Solution

Preserve the existing media-root upload boundary unless maintainers approve a narrower design. If mirroring is accepted, implement it as an explicit, audited, allowlisted path such as configured workspace-to-media staging with realpath containment, symlink-safe open/copy, size limits, collision-safe filenames, tests, and updated QQ Bot docs/skill guidance.

## Reproduction Assessment

Yes. The source path is high-confidence: `<qqmedia>` is parsed into an auto media send, and an existing absolute file outside `~/.openclaw/media` fails `resolveQQBotPayloadLocalFilePath`, producing the reported storage-root error.

## Solution Assessment

No for the blanket suggested fix. Auto-copying arbitrary existing local files is not the narrowest maintainable solution without an explicit security policy; an opt-in or allowlisted staging mechanism would be safer.
