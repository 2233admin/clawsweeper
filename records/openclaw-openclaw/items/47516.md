## Summary

Keep open. Current main still registers Discord reaction add/remove listeners unconditionally in `registerDiscordMonitorListeners`; newer handler code only short-circuits `reactionMode: "off"` before `fetchChannel`, which mitigates part of the slow-listener path but does not implement the startup registration guard requested here. The linked PR #47588 was closed unmerged, and #73133 addressed a different fetch-before-mode issue.

## Best Possible Solution

Keep this issue open and implement a narrow Discord plugin fix in `registerDiscordMonitorListeners`: skip registering `DiscordReactionListener` and `DiscordReactionRemoveListener` only when no configured path can emit reaction notifications, with focused startup tests for skip/register cases and without moving Discord-specific behavior into core.
