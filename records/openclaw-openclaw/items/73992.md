## Summary

Keep open. This remains a fresh, concrete `openclaw nodes list` loopback timeout report against 2026.4.25. Current main has adjacent gateway-probe and handshake improvements, but the `nodes list` command still performs gateway RPCs with the same 10s default timeout surface, and I did not find source, test, release, or related-PR proof that the reported intermittent `node.pair.list`/`node.list` timeout plus follow-up probe wedge is fixed.

## Best Possible Solution

Keep this issue open and investigate on current main or the next tagged build with targeted gateway logs, `/readyz` event-loop diagnostics, RPC timing around `node.pair.list` and `node.list`, and an immediate `gateway probe --json` after a failure. If connect succeeds but read diagnostics fail, route toward gateway responsiveness/probe classification; if `node.pair.list` itself stalls, focus on pairing-state I/O or gateway request scheduling.
