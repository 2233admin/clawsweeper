## Summary

Keep open. Current main still has the same `openclaw nodes list` RPC shape that was reported: unfiltered `nodes list` starts with `node.pair.list`, then best-effort `node.list`, both through the shared 10s Gateway RPC timeout. Adjacent probe/readiness diagnostics exist, but I found no source, test, release, or related-item proof that the 2026.4.25 intermittent loopback timeout plus immediate `gateway probe` connect-timeout window is fixed.

## Best Possible Solution

Keep this issue open and investigate on current main or the next tagged build with Gateway logs, `/readyz` event-loop diagnostics, RPC timing around `node.pair.list` and `node.list`, and an immediate `gateway probe --json` after any `nodes list` timeout. If the probe connects but read diagnostics fail, route toward probe/status classification; if connect times out or `node.pair.list` stalls, fix the confirmed Gateway responsiveness or pairing-state path with focused regression coverage.
