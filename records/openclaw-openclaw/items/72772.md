## Summary

Keep open. Current main still stretches the shared Control UI shell, Config form/action bar, and data table primitives, while existing capped surfaces are partial and the related chat-width items are narrower rather than superseding this app-wide layout request.

## What This Changes

The issue requests ultrawide-aware Control UI layouts that cap readable content, use side panels or multi-column patterns where useful, and avoid stretching forms and tables across very wide viewports.

## Best Possible Solution

Define a Control UI-wide responsive layout policy: cap readable form/detail content, intentionally preserve or redesign data-dense tables, use side panels where a view has real secondary content, and decide whether a full-width or density preference should coexist with the default layout.

## Reproduction Assessment

Yes. A high-confidence source-backed path exists: inspect the Control UI Config and Sessions/table surfaces at a viewport wider than 2560px, where current `.content`, `.config-layout`, and `.data-table` rules still span the available content column.

## Solution Assessment

Unclear. The proposed direction is reasonable, but the narrowest maintainable solution is likely a shared layout token/pattern plus view-specific form/table decisions, not a blanket CSS cap applied everywhere.
