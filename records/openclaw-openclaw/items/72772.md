## Summary

Keep open. Current main still has full-width Control UI layout primitives for the main content shell, config action bar, and sessions data table, while only some newer surfaces such as Quick Settings already have a capped wide layout. This is a valid broad UI enhancement, not a conservative close candidate.

## What This Changes

Request to make the Control UI ultrawide-aware by capping or centering primary content and using side panels or table-specific layouts instead of stretching forms and tables across the full viewport.

## Best Possible Solution

Define a Control UI wide-screen layout policy and apply it consistently: cap and center primary settings/form content where reading width matters, keep action buttons in the same visual column as their forms, and handle tables with deliberate max-width, horizontal scrolling, filters, or detail/preview side panels instead of one global stretch rule.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to open the Control UI settings or sessions views on a viewport wider than 2560px and inspect the config action bar and data table; current source shows those containers remain full-width while Quick Settings alone has a 1520px cap.

## Solution Assessment

Unclear. The requested direction is likely right, but the best implementation should be a reviewed Control UI layout policy across forms and tables, not a one-line global max-width that could make dense operational screens worse.
