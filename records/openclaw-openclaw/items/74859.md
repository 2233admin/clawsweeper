## Summary

Keep open. Current main still has the reported abort race in the embedded Pi runner, and an open PR already references this issue with closing syntax, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue reports that a pre-aborted embedded runner prompt submission can start a floating Pi Agent loop that keeps making LLM calls after the OpenClaw attempt has exited.

## Best Possible Solution

Land PR #74979 or an equivalent narrow runtime fix that checks the run abort state before every activeSession.prompt(...) call, adds focused abort/queue cleanup for escaped Pi Agent work, and includes a deterministic pre-aborted AbortSignal regression test. Treat the proposed max-LLM-call cap as a separate hardening discussion unless maintainers ask to bundle it.

## Reproduction Assessment

Yes. The issue includes a deterministic pre-aborted AbortSignal test shape, and current-main source inspection confirms the same call-order hazard remains; I did not execute the test because this review was read-only.

## Solution Assessment

Yes, the narrow pre-prompt guard plus cleanup backstop is the best immediate fix because it prevents the floating prompt promise at the source while preserving normal non-aborted runs. The hard-call cap is useful defense-in-depth, but it is not required to fix the lifecycle race.
