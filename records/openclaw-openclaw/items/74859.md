## Summary

Keep open. Current main still has the reported pre-aborted-signal prompt race, and the issue is now paired with open PR #74979 using closing syntax, so the right path is to review and land or close that implementation PR rather than auto-close the issue early.

## What This Changes

Prevent pre-aborted embedded-run abort signals from starting a floating Pi Agent prompt loop, especially when interrupt queue mode aborts a run before prompt submission.

## Best Possible Solution

Review PR #74979 or an equivalent narrow fix that adds a pre-prompt abort guard before every activeSession.prompt(...) call, adds aborted-run cleanup for escaped Pi Agent work, and lands a focused regression test for the pre-aborted AbortSignal path. The configurable max-LLM-call cap is better handled as follow-up hardening unless maintainers explicitly want it in the same change.

## Reproduction Assessment

Yes. The issue provides a deterministic pre-aborted AbortSignal test shape, and current main still has the matching call-order hazard; this read-only review did not run the test.

## Solution Assessment

Yes, the proposed guard-before-prompt plus aborted-run cleanup is the narrowest maintainable fix for the lifecycle bug. The hard per-run LLM-call cap would be useful defense in depth, but it is not required to fix the root race and should not block the targeted PR.
