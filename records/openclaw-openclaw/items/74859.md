## Summary

Keep open. Current main still has the reported abort race at the embedded Pi prompt boundary, and the issue is paired with open PR #74979 using closing syntax, so it should remain open until that PR is revised, merged, or replaced.

## What This Changes

The issue requests preventing pre-aborted embedded runner abort signals from starting an unowned Pi Agent prompt loop that can keep making LLM calls after attempt cleanup.

## Best Possible Solution

Revise and land #74979, or an equivalent narrow replacement, so prompt startup observes the run abort signal before every Pi Agent prompt-start boundary, escaped active runs are cancelled and queues cleared during abort cleanup, and regression coverage locks pre-aborted, pre-activeRun, and normal prompt paths.

## Reproduction Assessment

Yes. The issue provides a deterministic pre-aborted `AbortSignal` test shape, and current main still has the same source-level call-order hazard; this read-only review did not execute the test.

## Solution Assessment

Yes for the narrow abort-boundary repair: pre-prompt/pre-activeRun cancellation and cleanup backstops are the maintainable fix. The proposed configurable max-LLM-call cap is useful defense-in-depth, but should remain a separate policy hardening unless maintainers want broader work now.
