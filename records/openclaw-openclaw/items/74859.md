## Summary

Keep open. Current main still has the reported abort race: pre-aborted caller signals mark the run aborted before prompt submission, but each prompt path still creates `activeSession.prompt(...)` before `abortable(...)` can reject, so the Pi agent chain can escape the attempt lifecycle. The pinned Pi 0.70.6 source and OpenClaw tool-abort wrapper support the reported loop mechanism, making this a valid narrow runtime bug rather than cleanup noise.

## What This Changes

Fix pre-aborted embedded prompt attempts so aborting a run cannot leave a Pi Agent loop making background LLM calls after the OpenClaw attempt exits.

## Best Possible Solution

Guard before creating any `activeSession.prompt(...)` promise when the run signal is already aborted, then make abort cleanup explicitly terminate any active Pi Agent work and clear steering/follow-up queues. Add a focused regression test for the pre-aborted signal path; keep a configurable per-run LLM-call cap as separate hardening unless maintainers ask to include it.

## Reproduction Assessment

Yes. The issue provides a deterministic pre-aborted `AbortSignal` test shape, and current main still has the exact call-order hazard; I did not run the test because this review is read-only.

## Solution Assessment

No. The current implementation is not the best way to solve this lifecycle issue because abortability is applied after prompt promise creation. The narrow maintainable fix is a pre-prompt abort guard plus cleanup that actively aborts/drains the Pi Agent; a hard-call cap is useful backup protection but not the minimal fix.

## Repair Work Prompt

Fix the pre-aborted embedded Pi prompt zombie-loop bug reported in https://github.com/openclaw/openclaw/issues/74859. Keep the change narrow: in `src/agents/pi-embedded-runner/run/attempt.ts`, ensure a pre-aborted `params.abortSignal`/`runAbortController.signal` is checked before any `activeSession.prompt(...)` promise is created, including runtime-only, image, and normal prompt paths. Add a focused regression test using the existing embedded attempt harness or a cheaper helper seam that proves a pre-aborted signal does not call/start `activeSession.prompt()` and does not leave background LLM/tool iterations after `runEmbeddedAttempt` returns. Preserve the existing interrupt/steering queue behavior. Add a user-facing changelog fix entry crediting the issue reporter if project policy requires it for runtime fixes.
