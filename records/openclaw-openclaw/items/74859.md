## Summary

Keep open. Current main still has the reported abort race in the embedded Pi runner, and the issue is now paired with open PR #74979 using closing syntax, so it should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests fixing an embedded Pi runner abort race so pre-aborted or interrupted runs cannot start a floating Agent prompt loop that keeps making LLM calls after the attempt exits.

## Best Possible Solution

Land a revised #74979 or equivalent narrow fix that prevents prompt creation when the run signal is already aborted, makes the actual Pi prompt-start boundary abort-aware for preflight races, and adds focused regression coverage for pre-aborted, abort-before-active-run, and abort-after-active-run cases. Treat a configurable per-run LLM-call cap as follow-up hardening unless maintainers explicitly want it bundled.

## Reproduction Assessment

Yes. The issue includes a deterministic pre-aborted `AbortSignal` test shape, and current source plus the pinned Pi 0.70.6 prompt/abort contract confirm the same call-order hazard remains on main; I did not execute tests under the read-only review constraint.

## Solution Assessment

No. Current main is not the right lifecycle handling because it checks abort state only after creating the prompt promise; the maintainable fix is to guard before prompt creation and ensure aborts during Pi prompt preflight cannot later start an uncancelled Agent loop.
