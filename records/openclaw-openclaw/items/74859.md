## Summary

Keep open. Current main still has the reported abort race: a pre-aborted external signal trips `runAbortController`, but the prompt call sites still evaluate `activeSession.prompt(...)` before `abortable(...)` rejects, and cleanup does not wait for or clear a started Pi Agent run. The pinned Pi 0.70.6 runtime behavior supports the reported zombie-loop mechanism, so this is a valid narrow runtime bug rather than stale cleanup.

## What This Changes

The issue requests preventing pre-aborted or rapidly interrupted embedded agent attempts from starting or escaping a Pi Agent prompt loop that continues making LLM calls after the attempt exits.

## Best Possible Solution

Add a pre-prompt abort guard before every `activeSession.prompt(...)` call so pre-aborted runs never start the Pi prompt chain, and add focused cleanup that terminates any active Pi Agent work and clears queues if a run is aborted. Cover the pre-aborted signal path with a regression test; treat a configurable max-LLM-call cap as a follow-up hardening decision unless maintainers explicitly want it in the same PR.

## Reproduction Assessment

Yes. The issue provides a deterministic pre-aborted `AbortSignal` test shape, and current source inspection confirms the same call-order hazard remains on main; this review did not execute the test under the read-only constraint.

## Solution Assessment

No, the current implementation is not the best way to solve the lifecycle issue because it checks abort state only after the prompt promise has already been created. The narrow maintainable fix is to guard before prompt creation and ensure cleanup awaits or otherwise forces active Pi Agent termination; the hard-call-cap layer is useful defense-in-depth but not required for the minimal bug fix.
