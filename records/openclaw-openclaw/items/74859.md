## Summary

Keep open. Current main still has the reported abort race at the embedded Pi prompt boundary, and this issue is paired with open PR #74979 using closing syntax, so the issue should remain open until that PR is reviewed, merged, or replaced.

## What This Changes

The issue requests preventing pre-aborted embedded runner abort signals from starting an unowned Pi Agent prompt loop that can keep making LLM calls after attempt cleanup.

## Best Possible Solution

Land #74979 or an equivalent narrow PR that makes prompt startup abort-aware before every `activeSession.prompt(...)` call, cancels and clears escaped active runs during abort cleanup, and adds focused regression coverage for pre-aborted and normal prompt paths.

## Reproduction Assessment

Yes. The issue provides a deterministic pre-aborted `AbortSignal` test shape, and current main still has the same call-order hazard; this read-only pass verified the source path but did not execute tests.

## Solution Assessment

Yes. The narrowest maintainable fix is a pre-prompt abort guard plus cleanup cancellation coverage; a configurable max-LLM-call cap is useful defense-in-depth but should be a separate policy change unless maintainers want broader hardening now.
