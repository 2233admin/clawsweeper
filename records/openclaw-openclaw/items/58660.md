## Summary

Keep open. Current main has shipped WeChat plugin docs, manifest-id based install behavior, and doctor cleanup for stale plugin references, but it does not implement the requested explicit openclaw-weixin to openclawwechat replacement mapping or install-time cleanup. Because this is recent and crosses external plugin naming plus core install semantics, it needs maintainer judgment rather than cleanup closure.

## What This Changes

Request to prevent the plugin installer from confusing openclawwechat installs with old openclaw-weixin state by adding cleanup, ID mapping, and migration guidance.

## Best Possible Solution

Keep the external WeChat plugin on one canonical manifest id where possible, document the supported install/upgrade path, and only add core support if maintainers want a generic, explicit replacement metadata contract that routes through safe uninstall/doctor cleanup instead of name-similarity heuristics.

Unclear: the issue has logs, versions, and a plausible residual-state scenario, but not enough current config/install-index detail to reproduce confidently on current main. Source inspection confirms existing update/force and doctor cleanup paths, but not the requested install-time replacement behavior.

## Solution Assessment

Unclear/no: automatic cleanup of same-or-similar plugin names is probably not the safest generic fix. A safer direction is external-plugin canonical manifest metadata plus docs/CLI migration, or a designed opt-in core replacement field with conservative deletion rules.
