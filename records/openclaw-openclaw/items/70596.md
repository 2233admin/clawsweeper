## Summary

Keep this PR open. Current main still skips builtin memory startup work, so the intended explicit-local embedding prewarm is not implemented. The PR targets a real cold-start gap, but its current approach still needs an update because probing the cached default builtin manager can preserve a failed provider initialization promise and make later real searches reuse the failure.

## Best Possible Solution

Keep this PR open and request an update. The best fix is a narrow explicit-local builtin prewarm that avoids poisoning the cached default manager on transient failure, either by prewarming through a transient/non-default purpose or by clearing rejected provider initialization promises so later real searches can retry. Keep the no-`auto` and no-remote-provider boundary, add a regression test for startup prewarm failure followed by later retry, and separate the provider test-helper change unless it is required.
