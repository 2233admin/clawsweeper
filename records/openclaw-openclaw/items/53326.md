## Summary

Keep this PR open. The requested compatibility fix is still not implemented on current main or in the latest release, and #44783 was closed as superseded by this PR, but the current PR head has blocking issues around legacy dot-path probes, root filename collision, and incomplete legacy-state promotion.

## What This Changes

The PR modifies `src/agents/workspace.ts` to write workspace setup state at `<workspace>/workspace-state.json`, read legacy `.openclaw/workspace-state.json` as a fallback, and stop creating the `.openclaw` state directory.

## Best Possible Solution

Revise this PR around the same compatibility goal: keep setup state out of dot-prefixed workspace entries, use a collision-resistant non-dot filename or guard against foreign files, make legacy dot-path probes best-effort, promote successfully loaded legacy state to the new location, and add focused workspace tests for first run, legacy migration, collision handling, and dot-path I/O failures.

## Reproduction Assessment

Yes. The blocking issues are reproducible from the PR head: fallback reads call the legacy dot path through a helper that rethrows non-`ENOENT` errors, foreign root JSON can be treated as empty setup state before a later overwrite, and fallback-loaded state does not mark the new path dirty.

## Solution Assessment

No. The direction is reasonable, but this patch is not the safest maintainable fix until it handles legacy probe failures, prevents root filename clobbering, and completes migration from the legacy state path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Make legacy state probes best-effort:** `src/agents/workspace.ts:269`
  - body: `readWorkspaceSetupStateForDir` now probes `.openclaw/workspace-state.json` whenever the new root file has no markers, but `readWorkspaceSetupState()` rethrows non-`ENOENT` errors. On dot-restricted filesystems, setup and status checks can still fail before the new root state is written; catch and ignore legacy-probe I/O errors in the fallback path.
  - confidence: 0.91
- **[P2] Avoid overwriting foreign workspace-state files:** `src/agents/workspace.ts:207`
  - body: Writing OpenClaw state to `<workspace>/workspace-state.json` can clobber an existing workspace file with the same generic name. If that file parses without OpenClaw markers, setup treats it as empty state and later overwrites it, so use a safer non-dot name or detect foreign files before writing.
  - confidence: 0.88
- **[P2] Promote fallback state to the new location:** `src/agents/workspace.ts:401`
  - body: When `ensureAgentWorkspace` loads state from the legacy path, `stateDirty` remains false, so the later write never creates the new root state file. Existing workspaces keep depending on the dot-prefixed directory and can lose completion markers if that directory is unavailable; persist fallback-loaded state to the new path.
  - confidence: 0.86
