## Summary

Current main still does not implement the requested per-channel or broader tool-output visibility policy. `messages.suppressToolErrors` exists, but mutating failures such as `message`, `write`, and related send/edit actions still produce user-facing `isError` warning payloads unless a run-level `suppressToolErrorWarnings` override is used or the assistant explicitly acknowledges the failed action. Related PR #51091 and related issues #39406 and #73268 remain open rather than shipped behavior.

## Best Possible Solution

Keep this open for maintainer design and implementation. The best path is a documented tool-output visibility policy with a safe default, a global setting and any needed per-channel override, applied before chat-surface delivery while leaving tool failures available to the agent. Coordinate with #39406, #51091, and #73268, and add focused coverage for mutating `message`, `write`, and `exec` failures plus at least Telegram streaming delivery behavior.

## Risks / Open Questions

- The existing documented `messages.suppressToolErrors` wording reads broader than the tested mutating-tool exception, so users can reasonably expect suppression that current main does not provide.
- Hiding unresolved mutating failures is safety-sensitive: a failed `write`, `message`, or `exec` action can otherwise look successful if only the assistant's final text is shown.
- A per-channel policy needs cross-surface coverage so Telegram, Discord, Slack, Matrix, and other chat deliveries filter user-facing errors consistently while preserving tool errors in the agent context.
- PR #51091 is an active related implementation candidate, but it is open and does not by itself settle the broader per-channel/product policy requested here.
