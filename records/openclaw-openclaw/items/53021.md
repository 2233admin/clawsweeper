## Summary

Keep open. Current main still has raw `lan` bind flowing into several Control UI display URL surfaces, but this PR is not merge-ready because the branch is stale against the active status renderer and bundles unrelated user-facing error-message and bundled-skill changes.

## What This Changes

The PR normalizes some Control UI display URL call sites from `gateway.bind: "lan"` to loopback, changes embedded-agent transient/transport error text, and adds a bundled Drop Box document-intake skill.

## Best Possible Solution

Land a narrow current-main fix that centralizes display-only LAN-to-loopback normalization for browser Control UI URLs and applies it to dashboard, configure completion, status overview/status-all, daemon status, and setup finalization while preserving probe/runtime bind, custom bind, tailnet, basePath, and TLS semantics. Split the embedded-agent error-copy work and Drop Box skill into separate reviews.

## Reproduction Assessment

Yes. Code inspection gives a high-confidence reproduction path: configure `gateway.bind: "lan"`, then any display caller that passes raw bind into `resolveControlUiLinks` can render a LAN URL while the dashboard command demonstrates the intended loopback display behavior.

## Solution Assessment

No. The LAN-to-loopback display direction is right, but this PR is not the best way to solve it because it updates stale status code and bundles unrelated security/privacy-sensitive changes; a shared display-mode helper or resolver option used by all current display call sites is safer.
