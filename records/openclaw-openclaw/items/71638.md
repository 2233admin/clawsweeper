## Summary

Keep this issue open. Current main still has the hook metadata escape hatch and Telegram already normalizes visible reply/forward context for agent prompts, but the received-hook mappers do not carry Telegram forward, via-bot, or reply provenance into MessageReceivedHookEvent metadata, and the silent-ingest Telegram path builds the same limited received-hook context.

## What This Changes

The issue requests an additive Telegram hook metadata path that exposes inbound forward, inline-via-bot, and reply provenance on MessageReceivedHookEvent.context.metadata.

## Best Possible Solution

Implement an additive Telegram-owned provenance metadata path for received hooks: derive metadata.forward, metadata.viaBot, and metadata.replyTo from the inbound Telegram message after existing visibility checks, reuse it for normal dispatch and silent-ingest emits, cap reply text, and cover the mappers plus Telegram emit paths with focused tests.
