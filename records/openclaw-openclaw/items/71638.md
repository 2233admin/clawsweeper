## Summary

Keep this issue open. Current main still drops the requested Telegram provenance from `message:received` hook metadata: the shared mappers only emit generic sender/routing metadata, Telegram already derives reply and forward context for agent prompts, and there is still no `via_bot`/`viaBot` source path.

## What This Changes

Request to add Telegram forwarded, via-bot, and reply-to provenance to `MessageReceivedHookEvent.context.metadata`.

## Best Possible Solution

Add a bounded Telegram-owned provenance payload under existing hook metadata, thread it through the canonical inbound hook context, and map it consistently to both plugin and internal `message:received` events.

## Reproduction Assessment

Yes. A focused check can inject a Telegram message containing `forward_origin`, `via_bot`, and `reply_to_message`, then observe that current main's emitted `message:received` hook metadata lacks `forward`, `viaBot`, and `replyTo`.

## Solution Assessment

Yes. The narrowest maintainable solution is additive metadata plumbing inside the Telegram plugin plus the shared canonical hook mapper, reusing existing reply/forward helpers and adding via-bot extraction without hard-coding Telegram fields into unrelated channels.
