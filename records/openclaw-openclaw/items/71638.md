## Summary

Keep this issue open. Current main still has the reported Telegram received-hook metadata gap: the SDK and plugin hook surfaces support provider-specific metadata, and Telegram already derives visible reply/forward context, but the shared received-message hook mappers only emit generic sender/routing metadata and there is no via-bot plumbing.

## What This Changes

Request to surface Telegram forward, via-bot, and reply provenance on inbound Telegram message-received hook metadata without changing existing hook fields.

## Best Possible Solution

Add a Telegram-owned additive provenance payload under received-hook metadata, sourcing visible forward/reply data from the existing finalized Telegram context where possible, adding via_bot extraction from the raw message, bounding reply text, and covering both normal dispatch and mention-skip ingest paths with focused tests and docs/changelog updates.

## Reproduction Assessment

Yes. A focused test can construct a Telegram inbound message with forward_origin, via_bot, and reply_to_message, then assert that current main's message_received plugin/internal hook payloads omit metadata.forward, metadata.viaBot, and metadata.replyTo even though Telegram already stores visible ReplyTo*/Forwarded* fields in the finalized context.

## Solution Assessment

Yes. The additive metadata approach is the narrowest maintainable fix because plugin docs already reserve metadata for channel-specific extras, and it avoids adding Telegram-specific top-level SDK fields to the cross-channel hook contract.
