## Summary

Keep open. Current main still does not prove that the MiniMax tool-result path emits visible assistant text before the WebSocket chat final, and the reporter's follow-up OpenClaw gateway log keeps the likely failure inside the OpenClaw streaming/gateway boundary rather than only an external Java/Netty relay.

## What This Changes

The issue reports MiniMax WebSocket chat runs that skip assistant delta/text output and jump from tool/lifecycle events to a final chat state without message content.

## Best Possible Solution

Reproduce the MiniMax tool-result sequence on current main, capture raw provider/pi-agent-core and gateway WebSocket events, then fix the narrow layer that loses visible assistant text and add fixture-backed regression coverage for tool result followed by assistant text before chat final.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? No. The issue includes useful logs, but it lacks a replayable current-main MiniMax setup and raw provider event capture needed to reproduce the failing sequence with high confidence.

## Solution Assessment

Is this the best way to solve the issue? Unclear until the current-main capture identifies whether MiniMax transport, embedded subscription filtering, or gateway projection drops the text. The safest path is targeted reproduction first, not a broad final-event workaround.
