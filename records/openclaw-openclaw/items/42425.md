## Summary

Current main still does not implement this PR's intended behavior. Gateway startup loads internal hooks once from `defaultWorkspaceDir`, the hook loader discovers a single `<workspace>/hooks` directory per call, and `listAgentWorkspaceDirs` is not wired into the startup hook path. The PR remains a useful fix candidate, but it needs a rebase against the current `server-startup-post-attach.ts` path and alignment with current workspace-hook opt-in/no-override policy.

## Best Possible Solution

Keep this PR open as a fix candidate, but ask for a rebase or replacement patch against current main. The best implementation should wire multi-workspace hook loading into `src/gateway/server-startup-post-attach.ts`, enumerate workspaces with `listAgentWorkspaceDirs`, load shared hooks once, load workspace-local hooks once per configured workspace, preserve current explicit opt-in and no-override semantics, scope local handlers to the matching event/session/agent workspace, isolate bad workspace failures, and add focused Gateway startup regression tests.

## Risks / Open Questions

- The PR is stale against current main because the runtime startup integration point is now `src/gateway/server-startup-post-attach.ts`; `src/gateway/server-startup.ts` is only a re-export wrapper.
- Loading hooks from every configured workspace at Gateway startup broadens trusted local code execution. A refreshed implementation needs to preserve explicit workspace-hook opt-in, boundary checks, per-workspace scope guards, and per-workspace failure isolation.
- Current main's hook policy and docs say workspace hooks cannot override bundled, managed, or plugin-provided hooks. The PR's old review discussion around shared/local override suppression needs to be reconciled with that current contract.
- Current main lacks regression coverage proving that an explicitly enabled hook in a non-default agent workspace is discovered, loaded, and scoped during Gateway startup.
