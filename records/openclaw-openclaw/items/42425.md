## Summary

Keep this PR open. Current main still does not implement the PR’s central behavior: Gateway startup loads internal workspace hooks from only the default workspace, while this PR proposes loading workspace-local hooks for every configured agent workspace with scoping and failure isolation. The linked issue #42072 is closed, but the current source still shows the same default-workspace-only startup path.

## Best Possible Solution

Keep this PR open as a useful stale fix candidate. The best path is a rebase or replacement patch against current main that integrates multi-workspace hook loading in `src/gateway/server-startup-post-attach.ts`, uses `listAgentWorkspaceDirs`, preserves explicit workspace-hook opt-in and the existing no-override policy, scopes workspace-local handlers to the matching event/session/agent workspace, isolates per-workspace failures, and adds focused regression tests for non-default workspace loading, startup events, precedence, and security boundaries.

## Risks / Open Questions

- The PR is stale against current main because the startup integration point moved to `src/gateway/server-startup-post-attach.ts`; the patch would need a rebase or replacement implementation.
- Expanding startup hook loading from one workspace to all configured agent workspaces increases trusted local code execution surface, so review should require explicit hook opt-in, workspace scoping, boundary checks, failure isolation, and no-override semantics.
- Current main lacks regression coverage proving that an explicitly enabled hook in a non-default agent workspace is loaded and isolated during Gateway startup.
