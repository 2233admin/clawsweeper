## Summary

Keep open. Current main partially handles the Telegram side by downloading document attachments and forwarding MediaPath/MediaTypes into the shared media pipeline, and shared extraction covers text-like files plus PDFs. The requested Office-style document extraction is still missing: default file MIME allowlists omit DOCX/XLSX/PPTX, application/vnd Office MIME types are treated as binary before extraction, and the bundled document-extract plugin only registers a PDF extractor.

## Best Possible Solution

Keep this issue open and solve it in the shared document extraction path. The likely direction is to add selected Office document MIME support through the existing document-extractor contract and bundled document-extract plugin, allow those MIME types through only when a bounded extractor is available, preserve binary safety checks, and cover Telegram attachments plus shared extraction with focused tests and docs.

## Risks / Open Questions

- A Telegram-only patch would leave the same gap for chat.send/OpenResponses and other shared media-entry paths; the fix should be shared, not channel-specific.
- Office/DOCX/XLSX/PPTX extraction parses untrusted ZIP/XML or legacy binary document content, so dependency choice, size/page/entry limits, and prompt-injection wrapping need maintainer review.
- The current application/vnd.* binary guard is a safety boundary with test coverage; loosening it globally without a narrow extractor allowlist could reintroduce prompt input inflation.
