## Summary

Keep open. Current main still propagates `stopReason:"length"` as metadata on the normal terminal path, but native run-mode subagent waiting and announce logic still convert terminal lifecycle `end` into a successful child outcome and there is no configurable truncation fail/continue policy. The issue also has fresh production and local-model reproductions plus related #63210/#63689 context, so it is actionable rather than stale or already implemented.

## What This Changes

The issue requests configurable handling for `stopReason:"length"` in native `mode:"run"` subagent sessions so truncation can hard-fail or continue instead of silently completing.

## Best Possible Solution

Add a narrow, opt-in native-subagent truncation policy first: when a run-mode child reaches terminal `stopReason:"length"`, configured fail mode should surface an error/partial signal to the parent instead of `{status:"ok"}`. Keep automatic continuation out of the first patch until replay safety, side effects, and continuation prompts are explicitly designed.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to emit a lifecycle `end` with `stopReason:"length"` for a child run and observe `agent.wait` plus `applySubagentWaitOutcome` still produce `status:"ok"`; the issue also provides production and local-model repros.

## Solution Assessment

Yes for the first-stage fail policy, not for automatic continuation yet. Hard-failing on configured `length` truncation is the narrowest maintainable fix because it removes silent success without replaying potentially side-effecting work.
