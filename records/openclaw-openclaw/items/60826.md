## Summary

Current main still persists raw CLI argv into config-audit records. The write audit helper, async/sync observe paths, and observe-recovery helper all copy `process.argv.slice(0, 8)` into records, and the append helpers serialize records directly without redaction. The two related fix PRs, #60871 and #70468, are closed unmerged, so they are useful references but do not resolve the remaining credential-exposure path.

## Best Possible Solution

Keep this issue open. The best fix is to centralize config-audit process-info sanitization, or omit argv entirely, before any write, observe, or recovery audit record is appended. Reuse the existing redaction infrastructure where it fits, extend it for OpenClaw-specific CLI flags if needed, and add focused regression tests for write audit records plus async/sync observe and recovery audit records. The closed PRs #60871 and #70468 are useful implementation references but are not merged.

## Risks / Open Questions

- Secret-bearing CLI arguments in the first eight argv entries can be persisted to `~/.openclaw/logs/config-audit.jsonl` during config write, observe, and observe-recovery audit paths.
- The existing redaction infrastructure covers several token/key patterns, but config-audit serialization bypasses it today.
- A fix should cover both `--flag value` and `--flag=value` forms, plus OpenClaw-specific token flags such as gateway, bot, app, access, and custom API key variants.
