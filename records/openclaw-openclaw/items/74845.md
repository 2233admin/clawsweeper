## Summary

Keep open. Current main still implements `node.invoke` as a single request with one final `node.invoke.result`, while existing node-originated events are separate gateway/system notifications rather than progress updates delivered to the caller waiting on the invoke. The related timeout PR would only make some long calls wait longer; it does not provide the requested caller-visible progress/intermediate-update primitive.

## What This Changes

The issue requests an additive node-host command path that can keep an invoke alive and deliver progress or intermediate updates before final completion.

## Best Possible Solution

Design an additive, versioned node command progress contract that preserves existing one-shot invokes while allowing opt-in progress events for node-host and plugin-owned commands, with documented authorization, lifecycle, timeout/cancellation, cleanup, delivery, and compatibility tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence static contract inspection of node invoke, protocol schemas, plugin node-host handlers, gateway dispatch, tests, and docs, all of which show final-result behavior rather than a progress-capable invoke path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction for the stated platform need; increasing invoke timeouts or reusing generic `node.event` system notifications would not satisfy caller-visible progress for a pending invoke.
