## Summary

Keep open. Current main still models node-host invocation as one request with one final result, while existing exec events are separate terminal system events rather than a caller-visible progress stream for a pending node.invoke call. The request is valid but needs maintainer-owned protocol, SDK, lifecycle, and security design before implementation.

## What This Changes

Request an opt-in streaming/progress-capable node-host command path alongside the existing one-shot node.invoke flow so callers can receive intermediate updates before the final result.

## Best Possible Solution

Design and document an additive, versioned node command progress contract that keeps existing one-shot invokes compatible while allowing opt-in progress events for node-host and plugin-owned commands, with tests across gateway protocol, node host dispatch, plugin SDK/runtime.nodes.invoke, caller delivery, timeout, cancellation, and security policy paths.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence static inspection: node.invoke, protocol schemas, node-host dispatch, plugin node-host command types, tests, and docs all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction for the stated platform problem; polling or integration-specific events would not satisfy the requested reusable caller-visible progress contract.
