## Summary

Keep open. Current main still exposes node-host invocation as a one-request/one-final-result contract, while node-originated progress-like exec events are routed as system events rather than as caller-visible updates for a pending invoke; the adjacent progress/session issues do not supersede this broader node-host API request.

## What This Changes

Request support for long-running node-host commands that can send progress or intermediate updates before a final result.

## Best Possible Solution

Design an additive, versioned node command progress contract that preserves existing one-shot invokes while allowing opt-in progress events for node-host and plugin-owned commands, with documented authorization, lifecycle, timeout, cancellation, cleanup, and caller delivery semantics across gateway protocol, node host, plugin SDK/runtime, CLI/runtime helpers, docs, and focused tests.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence static contract inspection: node invoke, plugin node-host handlers, plugin runtime `nodes.invoke`, protocol docs, and exec event handling all show final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive appears to be the maintainable direction, while simply raising timeouts or relying on integration-specific `node.event` messages would not satisfy the requested reusable caller-visible progress contract.
