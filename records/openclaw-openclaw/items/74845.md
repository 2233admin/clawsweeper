## Summary

Keep open. Current main still treats `node.invoke` as a one-request/one-final-response contract, while existing node-originated events and channel preview streaming are separate mechanisms that do not deliver progress to the caller waiting on a pending node-host command.

## What This Changes

Add an opt-in long-running or streaming node-host command path that can emit progress or intermediate updates before returning a final result.

## Best Possible Solution

Add a versioned, opt-in node-invoke progress contract that preserves existing one-shot behavior while letting node-host and plugin-owned commands emit caller-visible progress before the final result, with protocol schemas, SDK/docs, lifecycle/security rules, and focused tests updated together.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is high-confidence static inspection of `node.invoke`, protocol schemas, node-host dispatch, plugin runtime types, tests, and docs showing final-result behavior rather than a progress-capable command path.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol/SDK primitive is the maintainable direction for the stated platform need; increasing timeouts or reusing general `node.event`/channel preview notifications would not satisfy caller-visible progress for a pending invoke.
