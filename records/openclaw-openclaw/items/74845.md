## Summary

Keep open. Current main still exposes node-host command invocation as a one-request/one-final-result contract: the gateway stores one pending invoke, plugin node-host handlers return a final string, and node-originated exec events are terminal system events rather than progress updates for the pending caller. The request appears valid but needs maintainer-level API, lifecycle, and security design before implementation.

## What This Changes

Add an opt-in long-running or streaming node-host command path that can send progress or intermediate updates before a final result.

## Best Possible Solution

Design an additive, versioned node command progress contract that preserves existing one-shot invokes while allowing opt-in progress events for node-host and plugin-owned commands, with documented authorization, lifecycle, timeout, cancellation, cleanup, buffering, and caller delivery semantics across gateway protocol, node host, plugin SDK, runtime helpers, CLI, docs, and tests.

## Reproduction Assessment

Not applicable as a feature request. Static verification against current main is high-confidence: node.invoke has one pending final result and plugin node-host handlers return a final string, with no caller-visible progress channel tied to the invoke id.

## Solution Assessment

Unclear pending maintainer design. A shared additive protocol and SDK primitive is the maintainable direction for the platform problem; increasing timeouts or adding integration-specific events would not satisfy the reusable progress contract requested here.
