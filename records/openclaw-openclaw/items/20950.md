## Summary

Keep this issue open. Current main still only supports a single aggregate cron definition store plus a derived runtime-state sidecar; there is no public `storeDir`/`jobsDir` config, no directory enumeration of per-job JSON files, and no per-job write/delete path. Open PR #41206 remains the existing implementation candidate, while #53581 only shipped the complementary `jobs-state.json` split and #70531 was already closed as a duplicate into this request.

## What This Changes

Request support for a directory-backed cron job store where each job can live in its own JSON file for NixOS/GitOps-style declarative management, alongside or instead of the existing `jobs.json` store.

## Best Possible Solution

Keep this issue open as the canonical product request and review/land or replace #41206 with an additive implementation that preserves existing `cron.store`/`jobs.json` compatibility, adds a documented directory-store option, deterministically loads per-job JSON definitions, keeps runtime state in the current `jobs-state.json` sidecar model, and updates config schema/help/docs/doctor plus focused cron store and gateway tests.
