## Summary

Keep open. Current main still models cron definitions as one aggregate `cron.store`/`jobs.json` file with a derived `jobs-state.json` sidecar, and the issue discussion already identifies open PR #41206 as the active implementation candidate, so this should remain the tracker for maintainer review rather than be closed or queued as a duplicate repair.

## What This Changes

The issue requests an opt-in cron jobs directory where each JSON file defines one cron job for declarative NixOS/GitOps management.

## Best Possible Solution

Keep this as the canonical tracker and ship an additive, documented directory-store mode that remains compatible with `cron.store`/`jobs.json`, deterministically loads per-job JSON definitions, keeps runtime state out of declarative files, and updates schema/help, doctor behavior, docs, and focused cron store/Gateway tests.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is high-confidence verifiable by source and docs inspection: config/schema, Gateway startup, store load/save, and user docs still use one aggregate definition file rather than a per-job directory.

## Solution Assessment

Yes for triage. Leaving this issue open while maintainers review, land, rework, or replace #41206 is the narrowest maintainable path; a separate ClawSweeper repair job would duplicate an already-open implementation candidate.
