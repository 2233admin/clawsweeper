## Summary

Keep open. Current main still has Discord guild/channel access built around `requireMention`, `users`, and `roles`; it does not implement the proposed `bypass`/`mention`/`deny` lists, permission shortcuts, or explicit channel inheritance semantics. The updated proposal also intentionally overlaps open #73387, so this should stay open for maintainer design review rather than conservative cleanup.

## What This Changes

Request to replace Discord guild/channel `requireMention` and flat sender allowlists with `bypass`, `mention`, and `deny` access lists supporting users, roles, permission shortcuts, and per-channel inheritance/overrides.

## Best Possible Solution

Keep this as an open Discord plugin design task: add a maintainer-approved access resolver and config schema, preserve old `requireMention`/`users` behavior during a documented deprecation window, coordinate the `groupPolicy`-aware channel inheritance work with #73387, and cover runtime, schema, doctor/audit, and docs together.

## Reproduction Assessment

Yes for the current-main gap. A source-level check shows the schema and preflight path still use boolean `requireMention` plus `users`/`roles`, and the existing unit test demonstrates that an unmatched configured channel is denied.

## Solution Assessment

Unclear. The proposal is coherent, but because it changes Discord access control and migration defaults, the best implementation should be split or approved by maintainers before an automated fix attempts it.
