## Summary

Keep open. Current main still exposes Feishu `read`, but the handler only supports single-message lookup by `messageId`; the requested paginated chat-history path is not implemented and is not fully covered by the broader pagination issue.

## What This Changes

The issue requests paginated Feishu chat-history reads through the message read action rather than only single-message lookup by messageId.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that preserves existing `messageId` lookup, uses `client.im.message.list` with `container_id_type: "chat"`, returns multiple parsed messages plus pagination metadata, and documents the supported `limit`/`pageToken`/time filter parameters.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: call Feishu `read` with a chat target and no `messageId`, and the handler throws before any list call; call it with `messageId`, and it uses the single-message `get` path.

## Solution Assessment

Yes. A narrow Feishu plugin change is the best direction because the shared action runner already forwards arbitrary params, and the missing behavior is the Feishu handler/runtime mapping to the SDK list endpoint.
