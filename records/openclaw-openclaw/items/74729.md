## Summary

Keep open. Current main still exposes Feishu `read`, but the handler only accepts a `messageId` and dispatches a single-message lookup; the requested paginated chat-history read path is still missing and is narrow enough for a Feishu plugin fix.

## What This Changes

Request Feishu `message read` support for paginated chat history by chat target with limit, page token, and before/after filters instead of only single-message lookup by `messageId`.

## Best Possible Solution

Add the paginated history path inside the Feishu plugin: preserve `messageId` lookup, add chat-target list support through the Feishu message-list API, reuse existing message parsing, return messages plus pagination metadata, and document/test the new Feishu read behavior.

## Reproduction Assessment

Yes. A static current-main reproduction is high confidence: call the Feishu action handler with `action: "read"` and chat-history params but no `messageId`; the code throws `Feishu read requires messageId` before any list call can occur.

## Solution Assessment

Yes. The narrowest maintainable fix is Feishu-owned because the shared action name and generic params already exist; core does not need a new channel abstraction for this specific missing API mapping.
