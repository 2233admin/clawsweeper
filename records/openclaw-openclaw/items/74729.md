## Summary

Keep open. Current main still exposes Feishu `read`, but the handler only accepts `messageId` and routes to a single-message lookup; there is no chat-history list path for `chatId`/`limit`/`pageToken` or time filters.

## What This Changes

The issue requests paginated Feishu message-read support so callers can fetch chat history by chatId with limits, page tokens, and before/after filters instead of only reading one message by messageId.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that preserves the existing `messageId` lookup and returns a messages array plus pagination metadata when the caller supplies a chat target and list parameters.

## Reproduction Assessment

Yes. On current main, a Feishu `read` action with `chatId` and `limit` but no `messageId` reaches the handler and throws `Feishu read requires messageId.`, while the only successful path calls `getMessageFeishu` for one message.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin: add a chat-history list helper and dispatch branch, update Feishu tests/docs, and avoid changing core shared message contracts unless a small alias or schema exposure is required.
