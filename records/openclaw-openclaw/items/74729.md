## Summary

Keep open. Current main still exposes Feishu `read`, but the handler only accepts `messageId` and calls the single-message Feishu API; chat-history pagination with `chatId`/target, `limit`, time filters, and `pageToken` is still missing.

## What This Changes

The issue requests paginated Feishu `message read` support for chat history instead of only single-message lookup by `messageId`.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path while preserving the existing `messageId` lookup, then cover both modes in Feishu tests and document the supported CLI/tool parameters.

## Reproduction Assessment

Yes. A static reproduction is enough: call Feishu `read` with chat/target and pagination fields but no `messageId`, and current main throws `Feishu read requires messageId`; with `messageId`, it only calls the single-message helper.

## Solution Assessment

Yes. The narrow maintainable fix belongs in the Feishu plugin by adding a chat-list helper around Feishu `im.message.list` and wiring it into `read`, without changing core unless a shared parameter contract gap appears.
