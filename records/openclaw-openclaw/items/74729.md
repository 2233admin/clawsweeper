## Summary

Keep open. Current main still advertises Feishu `read`, but the handler only accepts `messageId` and calls the single-message lookup path; `chatId`/`limit`/`pageToken`/time-filter history reads cannot reach a Feishu chat-history list call. The related #71452 is broader shared pagination work and does not supersede this Feishu-specific API gap.

## What This Changes

Feishu `message read` should support chatId-based paginated history retrieval with limit, page token, and time filters instead of only single-message lookup by messageId.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path using `client.im.message.list` with `container_id_type: "chat"`, while preserving the existing `messageId` lookup. Update Feishu action aliases, docs, tests, and changelog; keep #71452 as the broader cross-channel pagination follow-up.

## Reproduction Assessment

Yes. Static current-main reproduction is high-confidence: `action=read` with only `chatId`, `limit`, `before`, `after`, or `pageToken` cannot reach a list path and fails before dispatching any Feishu history request.

## Solution Assessment

Yes. A narrow Feishu plugin change is the maintainable fix because the shared runner already passes action params through to channel plugins; core should only need alias/normalization coverage if `chatId` must satisfy generic read target validation.
