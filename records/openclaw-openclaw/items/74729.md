## Summary

Keep open. Current main still exposes Feishu `read`, but the handler only supports a single `messageId` lookup and never routes chatId/target plus pagination parameters to Feishu's message-list API.

## What This Changes

The issue requests paginated Feishu chat-history reads through the message read action using chatId/target, limit, before/after, and page-token style parameters.

## Best Possible Solution

Add a Feishu-owned paginated chat-history read path that preserves the existing `messageId` lookup, uses `client.im.message.list` for chat containers, returns messages plus continuation metadata, and documents the supported parameters.

## Reproduction Assessment

Yes. A static current-main reproduction is high-confidence: call Feishu `read` with a chat target and no `messageId`, and the handler throws before any list call; call it with `messageId`, and it uses the single-message `get` path.

## Solution Assessment

Yes. The narrow maintainable solution belongs in `extensions/feishu`, with a small helper and tests/docs, because core already passes generic fetch parameters and the missing behavior is Feishu-specific dispatch.
