## Summary

Keep open. Current main has partial fixes for this class, including Gateway-authoritative cron reconciliation, durable cron terminal recovery, and bounded restart deferral, but the core operator recovery path is still incomplete: cron task cancellation still reaches the unsupported runtime branch, offline `tasks maintenance --apply` intentionally cannot mark Gateway-owned cron tasks lost from an empty local active-job set, and TaskFlow cancellation still blocks while child tasks remain active.

## What This Changes

The issue requests recovery for stuck `running` task records by making `openclaw tasks cancel`, `openclaw tasks maintenance --apply`, gateway restart, cron deletion cleanup, and TaskFlow cancellation clear or reconcile zombie cron/task records.

## Best Possible Solution

Define a safe Gateway-backed operator recovery path: cron-owned task cancellation should either delegate to the live Gateway/cron runtime or require an explicit force mode, maintenance should reconcile only when it has authoritative liveness evidence or durable terminal history, TaskFlow cancellation should have a deliberate escape hatch for unrecoverable children, and restart/update drains should keep bounded behavior without relying on stale task records to clear themselves.

## Reproduction Assessment

Yes. A source-backed reproduction exists without mutating the checkout: a `runtime: "cron"` active task reaches the unsupported branch in `cancelTaskById`, and the regression test for `cronRuntimeAuthoritative: false` shows offline maintenance leaves stale cron tasks `running`.

## Solution Assessment

No. Current main is a good partial fix for Gateway-owned cron reconciliation and restart deferral, but it is not the complete solution because ordinary cron cancellation and a deliberate force/Gateway-backed operator cleanup path remain unresolved.
