## Summary

Keep this issue open. Current main only partially addresses the stale-task recovery class: CLI task cancellation and some stale cron maintenance cases have landed, but cron task cancellation still returns unsupported, offline `openclaw tasks maintenance --apply` intentionally does not treat its empty cron runtime as authoritative, TaskFlow cancellation can still stop on active children that cannot be cancelled, and restart/reload drain remains indefinite unless `gateway.reload.deferralTimeoutMs` is configured. The April 28 discussion also reports a v2026.4.26 restart/update failure mode, so the report is not obsolete.

## Best Possible Solution

Keep this issue open and scope the next maintainer fix to the remaining operator recovery paths: add a cron/task-flow stale-running force-cancel or recovery path, provide an explicit Gateway-authoritative or force mode for stale cron rows in `tasks maintenance --apply`, and make restart/update preflight stale tasks, exclude proven-stale records from drain counts, or enforce a safe bounded timeout by default. Treat #60299 and #73867 as related partial/narrow follow-ups, not canonical closure for this broader issue.
