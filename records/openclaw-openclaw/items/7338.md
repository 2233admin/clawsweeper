## Summary

Keep this issue open. Current main still does not implement a shared, opt-in, domain-scoped OpenClaw agent attestation contract for external HTTP requests, web_fetch, skills, or arbitrary tool/plugin fetches. The closed provider-header work in #11137 is adjacent model-provider traffic governance, not a substitute for agent-origin attestation.

## Best Possible Solution

Keep this issue open as the canonical product/API design tracker. A complete solution should define whether attestation belongs in core or behind a new plugin/API seam, add opt-in domain-scoped config, specify static versus cryptographic header semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests for allowed-domain injection plus non-leakage to other domains.

## Risks / Open Questions

- A cryptographic attestation feature needs explicit trust, signing-key, audience, replay-expiry, verifier, and rotation design before implementation.
- Opt-in and domain allowlisting are important privacy requirements; a broad default header could leak stable agent identity across unrelated services.
- A service-specific plugin can sign requests it owns, but that does not solve the requested shared attestation contract across built-in web_fetch, skills, and arbitrary tool HTTP clients.
