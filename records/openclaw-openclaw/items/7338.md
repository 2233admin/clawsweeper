## Summary

Keep this open. Current main still lacks opt-in, domain-scoped OpenClaw agent attestation for arbitrary outbound HTTP requests; the existing attribution code is limited to verified model-provider transports. The issue discussion already has maintainer and ClawSweeper keep-open comments, and the related header reports are narrower provider/transcript work rather than a canonical replacement.

## What This Changes

Add optional static or signed OpenClaw agent attestation headers to outbound external requests, gated by user or skill configuration and domain allowlists.

## Best Possible Solution

Define the identity and trust model first, then add an opt-in, domain-scoped config/API surface with privacy-preserving defaults, clear static-versus-cryptographic semantics, docs, and tests proving allowlisted injection plus non-leakage.

## Security Review

Status: needs_attention

Summary: This issue is security-sensitive because implementing it would assert OpenClaw agent identity to third-party services and could leak stable identifiers if scoped incorrectly.

Concerns:

- **[medium] Define the attestation trust model:**
  - body: Before implementation, maintainers need to decide who signs attestations, how services obtain verification keys, how replay is prevented, and what issuer/audience semantics mean for OpenClaw agents.
  - confidence: 0.91
- **[medium] Prevent cross-domain identity leakage:**
  - body: Attestation must be opt-in and allowlisted so agent IDs, versions, skill context, or signed tokens are not attached to arbitrary external requests, fallback providers, or redirects.
  - confidence: 0.9
