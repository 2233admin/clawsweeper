## Summary

Keep this issue open. Current main still does not implement opt-in, domain-scoped OpenClaw agent attestation for external HTTP requests, and the existing provider/header plumbing is adjacent provider transport behavior rather than a general agent-origin attestation contract. The related sender-identity request #11137 was closed as a duplicate of this issue, so this remains the canonical tracker.

## Best Possible Solution

Keep this open as the canonical product/security API design tracker. A complete solution should decide the trust model first, then add an opt-in domain-scoped config/API surface, define static versus cryptographic semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests proving allowed-domain injection plus non-leakage to other destinations.

## Risks / Open Questions

- A naive default header implementation could leak stable agent, workspace, user, or skill identity to unrelated domains; opt-in behavior and domain allowlisting are central privacy requirements.
- Cryptographic attestation needs an explicit trust model: issuer identity, key management, public-key distribution, audience scoping, expiry, replay protection, and rotation.
- Existing provider attribution/request-header plumbing is easy to mistake for this feature, but it is scoped to model-provider traffic and does not verify OpenClaw-origin requests to arbitrary external services.
