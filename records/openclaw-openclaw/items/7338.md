## Summary

Keep open. Current main still lacks a shared, opt-in, domain-scoped OpenClaw agent attestation contract for external HTTP requests. The built-in web_fetch path emits only ordinary fetch headers, config/schema/docs expose no attestation fields, existing attribution headers are limited to model-provider routes, and the prior steipete/Codex issue comment explicitly kept this open while distinguishing adjacent provider-header work in #11137.

## Best Possible Solution

Keep this open as the canonical product/API design tracker. A complete path should decide whether attestation belongs in core or behind a new plugin/API seam, define opt-in domain-scoped config, specify static versus cryptographic semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests for allowed-domain injection plus non-leakage to other destinations.

## Risks / Open Questions

- A naive default header implementation could leak stable agent identity across unrelated services; opt-in and domain allowlisting are central privacy requirements.
- Cryptographic attestation needs explicit issuer, key management, audience, expiry, replay, verifier, and rotation design before implementation.
- Adjacent provider-header work can be mistaken for a fix, but it only covers model-provider request attribution and does not prove agent origin for general external HTTP requests.
