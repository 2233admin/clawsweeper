## Summary

Keep open. Current main still lacks a general, opt-in, domain-scoped agent attestation mechanism for arbitrary external HTTP requests, and the existing provider-attribution work is limited to provider transports rather than agent-origin proof. The prior maintainer and ClawSweeper comments also keep this as unresolved product/security design work.

## What This Changes

Add optional, user-configurable OpenClaw agent attestation headers or signed tokens to outgoing external HTTP requests, optionally scoped by skill and allowed domains.

## Best Possible Solution

Define the identity and trust model first, then implement a privacy-preserving opt-in config/API surface with strict domain allowlisting, clear static-versus-cryptographic semantics, redirect/non-leakage protections, docs, and targeted tests.

## Reproduction Assessment

Not applicable as a runtime bug reproduction. There is a high-confidence current-main verification path: inspect web_fetch headers, fetch config/schema, skill metadata, docs, and provider-attribution tests to confirm the requested surface is absent.

## Solution Assessment

Unclear until maintainers settle the design. Static headers alone are not the best solution for agent-only services; the maintainable path needs scoped consent, signing/replay semantics, redirect protections, and non-leakage tests.

## Security Review

Status: needs_attention

Summary: This feature is security-sensitive because it would assert OpenClaw agent identity to third-party services and could leak stable identifiers if scoped incorrectly.

Concerns:

- **[medium] Define the attestation trust model:**
  - body: Before implementation, maintainers need to decide who signs attestations, how third-party services obtain verification keys, how replay is prevented, and what issuer/audience claims mean for OpenClaw agents.
  - confidence: 0.93
- **[medium] Prevent cross-domain identity leakage:** `src/agents/tools/web-fetch.ts:425`
  - body: Any attestation must be opt-in and allowlisted so agent ids, versions, skill context, or signed tokens are not attached to arbitrary external requests, redirects, custom proxies, or fallback providers.
  - confidence: 0.91
