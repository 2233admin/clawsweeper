## Summary

Keep open. Current main still lacks the requested per-skill readonly exec scope and bounded readonly command allowlist; the closest existing path is agent-level `autoAllowSkills`, which is broader and explicitly treated as widened host-exec trust. This needs maintainer/security design rather than cleanup closure.

## What This Changes

The issue requests a per-skill `readonly` exec-approval scope and bounded readonly command allowlist for healthcheck/canary skills.

## Best Possible Solution

Define an additive, security-reviewed approval contract that binds readonly auto-approval to trusted skill identity and explicit argv patterns, keeps outbound and mutating operations gated, updates Gateway/node/macOS/UI/CLI/docs together, and adds regression coverage for allowed readonly commands plus denied side-effect commands.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing/types and exec approval schema/runtime behavior; replaying the exact WSL2 canary loop would require the reporter's workspace skill/session.

## Solution Assessment

Unclear. The requested direction is plausible, but `metadata.openclaw.execScope` should not be accepted as the final shape until maintainers settle the trust boundary; existing `autoAllowSkills` is broader and not a safe substitute.
