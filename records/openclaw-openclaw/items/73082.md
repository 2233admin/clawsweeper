## Summary

Keep open. Current main has per-agent exec approval policy, durable exact-command trust, path/argument allowlist entries, and the broad `autoAllowSkills` convenience path, but it does not implement the requested per-skill `execScope: readonly` contract or bounded readonly command allowlist. Because the request changes host exec approval semantics, it needs maintainer/security design rather than cleanup closure.

## What This Changes

The issue requests a per-skill readonly execution scope and bounded readonly command allowlist for healthcheck/canary skills so selected introspection commands can avoid repeated exec approvals.

## Best Possible Solution

Design an additive approval contract that binds auto-approval to explicit per-skill or per-agent argv patterns, treats skill metadata as advisory unless the skill source is trusted, keeps outbound and mutating operations fail-closed, and updates the Gateway/node/macOS/UI/docs/test surfaces together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing/types and exec approval schema/evaluation; replaying the reporter's exact WSL2 canary loop would require their workspace skill and session state.

## Solution Assessment

Unclear. A bounded readonly scope is plausible, but the proposed `metadata.openclaw.execScope` shape needs maintainer/security review because a skill-declared approval bypass could weaken host exec prompts; a config-owned trust contract may be safer than trusting arbitrary skill frontmatter.
