## Summary

Keep open. Current main still lacks the requested per-skill readonly execution scope and bounded readonly command allowlist; the existing `autoAllowSkills` path is broader skill-bin trust, so this needs maintainer/security design rather than cleanup closure.

## What This Changes

Add SKILL.md metadata for a readonly exec scope plus a per-skill readonly command allowlist for healthcheck/canary skills while keeping side-effecting commands approval-gated.

## Best Possible Solution

Define an additive approval contract that binds explicit per-skill argv patterns to a trusted skill identity, keeps mutating and outbound operations gated, updates Gateway/node/macOS/UI/docs surfaces together, and adds regression coverage for both allowed healthcheck commands and denied side effects.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing/types and exec approval schema/allowlist behavior; replaying the reporter's full WSL2 canary loop would require their exact workspace skill/session.

## Solution Assessment

Unclear. A per-skill readonly allowlist is directionally plausible, but the exact `execScope: readonly` metadata shape should be settled by maintainer/security review because it changes the host exec approval trust model.
