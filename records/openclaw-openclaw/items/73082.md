## Summary

Keep open. Current main still lacks the requested skill-scoped readonly exec approval contract, and the request intentionally changes host exec approval bypass semantics, so it needs maintainer/security design rather than cleanup closure.

## What This Changes

Request a per-skill readonly exec scope and bounded command allowlist in SKILL.md metadata so healthcheck/canary skills can auto-approve selected readonly OpenClaw commands while keeping mutating and outbound actions approval-gated.

## Best Possible Solution

Keep this open until maintainers define an additive approval contract that connects trusted skill identity, per-skill argv patterns, host/node protocol state, UI/docs, and explicit deny precedence for outbound or mutating operations, then implement it with regression coverage.

## Reproduction Assessment

Not applicable as a feature request. The missing capability is statically reproducible on current main by inspecting the skill metadata parser/types and exec approvals schema/allowlist behavior; replaying the exact WSL2 canary run would require the reporter’s workspace skill/session.

## Solution Assessment

Unclear. A per-skill readonly allowlist is a plausible direction, but the exact execScope shape should be settled by maintainers/security because it would create a new approval-bypass path for host exec.

## Security Review

Status: needs_attention

Summary: The requested feature is security-sensitive because it would intentionally add a skill-scoped path for selected host exec commands to run without per-invocation approval.

Concerns:

- **[medium] Design readonly skill exec as a trust boundary:**
