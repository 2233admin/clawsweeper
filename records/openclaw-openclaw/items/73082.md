## Summary

Keep open. Current main still lacks a per-skill readonly exec scope or bounded readonly command allowlist, and the requested behavior changes host exec approval semantics enough that it needs maintainer/security design instead of conservative cleanup closure.

## What This Changes

Add a skill-declared readonly exec scope and bounded readonly command allowlist so healthcheck/canary skills can run approved introspection commands without repeated per-invocation exec approval prompts.

## Best Possible Solution

Define an additive approval contract that binds readonly auto-approval to trusted skill identity plus explicit argv/pattern rules, keeps outbound delivery, filesystem writes, service restarts, cron mutations, and other side effects fail-closed, and updates protocol/config/UI/docs/tests together.

## Reproduction Assessment

Not applicable as a feature request. The missing surface is statically reproducible on current main by inspecting skill metadata parsing, exec approval schema, docs, and node-host allowlist behavior; replaying the exact WSL2 canary loop is not needed to confirm the requested API is absent.

## Solution Assessment

Unclear. The requested direction is plausible, but the exact execScope contract should be settled by maintainers/security because current autoAllowSkills is intentionally broader and documented as implicit trust rather than per-skill readonly approval.
