## Summary

Keep this issue open. Current main still uses the BlueBubbles API/server password as the inbound webhook credential, and the same-author closing PR #70757 is open and unmerged, so the issue should remain open until that PR or an equivalent maintainer-approved fix lands.

## What This Changes

The issue requests separating the BlueBubbles inbound webhook secret from the BlueBubbles server/API password so public webhook setup no longer exposes a reusable route credential.

## Best Possible Solution

Land #70757 or an equivalent maintainer-approved security fix that adds a dedicated BlueBubbles `webhookSecret`, authenticates inbound webhooks only against that credential, keeps the API password reserved for BlueBubbles server access, updates setup/docs/SecretRef metadata, and ships regression coverage for top-level and per-account credential resolution.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Source and test evidence on current main provide a high-confidence path: configure BlueBubbles with `channels.bluebubbles.password`, send a POST to `/bluebubbles-webhook?password=<password>` with a caller-controlled `handle.address`, and the handler authenticates against that password before the normal allowlist flow consumes the supplied sender identity.

## Solution Assessment

Is this the best way to solve the issue? Yes. A distinct `webhookSecret` for inbound webhook authentication is the narrow maintainable fix; header-only transport hardening would reduce URL exposure but would not separate the reusable server/API credential from the public webhook credential.
