## Summary

Keep open. Current main and the latest release still authenticate BlueBubbles webhooks by comparing query/header tokens to the BlueBubbles server/API password, the public docs still show password-bearing webhook URLs, and the same-author closing PR #70757 is open and unmerged.

## What This Changes

The issue requests a dedicated BlueBubbles inbound webhook secret so the public webhook credential is separate from the BlueBubbles server/API password, with matching setup, docs, config, and test updates.

## Best Possible Solution

Land #70757 or an equivalent maintainer-approved security fix that adds a dedicated BlueBubbles `webhookSecret`, authenticates inbound webhooks only against that credential, keeps the API password reserved for BlueBubbles server access, updates setup/docs/SecretRef metadata, and ships regression coverage for top-level and per-account credential resolution.

## Reproduction Assessment

Yes. Configure BlueBubbles with `channels.bluebubbles.password`, send a POST to `/bluebubbles-webhook?password=<password>` with a caller-controlled `handle.address`, and current main authenticates against that password before allowlist and command-gating logic consumes the supplied sender identity.

## Solution Assessment

Yes. A distinct BlueBubbles `webhookSecret` is the narrow maintainable fix; header-only hardening would reduce URL leakage but would not separate the reusable server/API credential from the public webhook credential.
