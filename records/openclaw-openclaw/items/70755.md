## Summary

Keep this issue open. Current main still authenticates BlueBubbles inbound webhooks with the BlueBubbles API/server password, docs still publish that credential in the webhook URL, and the same-author closing PR #70757 remains open and unmerged.

## What This Changes

The issue requests a dedicated BlueBubbles inbound webhook secret so public webhook setup no longer reuses or exposes the BlueBubbles server/API password.

## Best Possible Solution

Land #70757 or an equivalent maintainer-approved security fix that adds a dedicated BlueBubbles `webhookSecret`, authenticates inbound webhooks only against that credential, keeps the API password reserved for BlueBubbles server access, updates setup/docs/SecretRef metadata, and ships regression coverage for top-level and per-account credential resolution.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Configure BlueBubbles with `channels.bluebubbles.password`, send a POST to `/bluebubbles-webhook?password=<password>` with a caller-controlled `handle.address`, and current main authenticates against that password before the allowlist flow consumes the supplied sender identity.

## Solution Assessment

Is this the best way to solve the issue? Yes. A distinct `webhookSecret` for inbound webhook authentication is the narrow maintainable fix; header-only transport hardening would reduce URL exposure but would not separate the reusable server/API credential from the public webhook credential.
