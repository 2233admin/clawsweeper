## Summary

Keep open. Current main has an unmanaged stop fallback, but the process verifier still rejects the reported standalone `openclaw-gateway` argv shape before the opted-in gateway-binary check can pass, so `gateway stop` can still fall through to the service-disabled message.

## What This Changes

The issue asks for `openclaw gateway stop` to stop foreground-launched gateway processes in no-service-manager/container environments instead of only reporting that the service is disabled.

## Best Possible Solution

Update gateway process verification so an opted-in standalone `openclaw-gateway` executable is accepted before requiring a separate `gateway` argv token, preserve the existing verified-process safety boundary, and add regression coverage for unmanaged stop against the reported foreground child-binary shape. If listener discovery is touched, the no-`lsof` Linux/container path should reuse or mirror the existing `ss` fallback instead of silently returning no PIDs.

## Reproduction Assessment

Yes. The reporter supplied a concrete foreground-gateway repro, and current source reproduces the verifier gap: `['openclaw-gateway']` fails the standalone `gateway` token guard before the gateway-binary allowance is evaluated.

## Solution Assessment

Yes. Keeping this open and routing a narrow fix is the best current path; the suggested reorder is the smallest maintainable fix, with a follow-up check for the missing-`lsof` container path if the implementation touches listener discovery.
