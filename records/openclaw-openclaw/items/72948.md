## Summary

Keep open. Current main still has the reported foreground-stop failure path, and the related open PR #72224 appears to contain the exact verifier and listener-discovery fix but is not merged and is currently dirty.

## What This Changes

The issue asks for `openclaw gateway stop` to stop a foreground-launched unmanaged gateway instead of returning `Gateway service disabled` while leaving the listener process alive.

## Best Possible Solution

Rebase and review #72224, or land an equivalent narrow daemon/infra PR that accepts bare `openclaw-gateway` only under the existing gateway-binary verification gate, adds `/proc` or `ss` listener discovery when `lsof` is missing, preserves strict gateway-only signaling before SIGTERM, and includes regression coverage for the foreground-stop path.

## Reproduction Assessment

Yes. The issue includes concrete foreground-gateway reproduction steps and logs, and current source shows the reported single-token `openclaw-gateway` argv plus missing-`lsof` path can produce an empty verified PID list followed by the service-disabled fallback.

## Solution Assessment

Yes. A narrow verifier and listener-discovery fix is the best maintainable solution; changing only the message would leave container and sandbox users manually killing verified foreground gateways.
