## Summary

Keep open. Current main still has the foreground gateway stop gap: unmanaged stop exists, but the verifier rejects a single-token `openclaw-gateway` listener before the opt-in binary check can match, and the Unix listener discovery path still returns no candidates when `lsof` is unavailable. Open PR #72224 contains the same source-level fix discussed in the issue, but it is still open rather than merged.

## What This Changes

`openclaw gateway stop` should terminate a verified foreground-launched gateway process instead of reporting only that the service-manager registration is disabled.

## Best Possible Solution

Land #72224 after review/rebase, or land a narrower equivalent that accepts verified single-token `openclaw-gateway` processes under the existing `allowGatewayBinary` gate, adds Linux listener discovery that works without `lsof`, and preserves strict gateway-only verification before SIGTERM.

## Reproduction Assessment

Yes. The reporter supplied concrete foreground-gateway reproduction steps, and current source inspection shows the same path can still produce an empty verified PID list followed by the service-disabled fallback.

## Solution Assessment

Yes. A narrow verifier plus listener-discovery fix is the best maintainable direction because it makes `gateway stop` useful in containers while preserving process verification; a message-only change would leave users manually killing a verified gateway.
