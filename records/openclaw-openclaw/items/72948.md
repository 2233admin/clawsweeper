## Summary

Keep open. Current main still has the unmanaged stop fallback, but the gateway argv verifier still rejects the reported standalone `openclaw-gateway` listener shape, so `openclaw gateway stop` can still fall through to the service-disabled message without sending SIGTERM.

## What This Changes

The issue asks `openclaw gateway stop` to stop foreground-launched gateway processes in service-manager-free environments instead of reporting only that the service is disabled.

## Best Possible Solution

Accept an opted-in standalone `openclaw-gateway` executable, both bare basename and full path, before requiring a separate `gateway` token; keep all other argv shapes rejected and add focused tests for unmanaged stop with the foreground child-binary process shape.

## Reproduction Assessment

Yes. The reporter supplied concrete foreground-gateway steps, and current source inspection shows the verifier rejects the exact single-token `openclaw-gateway` process shape needed for the fallback to signal it.

## Solution Assessment

Yes. A narrow verifier fix plus regression coverage is the maintainable path; a docs-only or message-only change would still leave container and sandbox users manually killing foreground gateways.

## Repair Work Prompt

Fix the foreground gateway stop bug tracked by https://github.com/openclaw/openclaw/issues/72948 and related closed sibling #72942. On current main, `openclaw gateway stop` can miss a foreground child process whose argv is a single `openclaw-gateway` token because `isGatewayArgv()` requires a separate `gateway` token before the `allowGatewayBinary` check, and the binary check only matches `/openclaw-gateway` paths. Update the verifier so `opts.allowGatewayBinary === true` accepts a verified standalone `openclaw-gateway` executable by bare basename and full path before the standalone `gateway` token guard. Preserve strict rejection of unrelated processes and preserve the default non-opt-in behavior. Add focused tests in `src/infra/gateway-process-argv.test.ts` for `['openclaw-gateway']`, `['/usr/local/bin/openclaw-gateway']`, and Windows `.exe` variants with and without the opt-in flag; add or adjust lifecycle/process tests only if needed to prove unmanaged stop signals verified listeners. Do not add pid-file semantics, do not kill arbitrary port listeners, and do not change service-manager behavior beyond this verifier/fallback path. Because this is user-facing CLI behavior, add a single-line `CHANGELOG.md` fix entry with `Thanks @mayank6136`. Validate with the targeted tests first, then run the changed gate in Testbox before handoff.
