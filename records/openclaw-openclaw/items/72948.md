## Summary

Keep open. Current main still contains the reported no-op path for foreground gateway stops: unmanaged stop exists, but current listener discovery and argv verification can return no verified PID for the `openclaw-gateway` child process shape described by the reporter. The linked #72224 appears to contain the relevant fix direction, but it is open rather than merged.

## What This Changes

Request that `openclaw gateway stop` stop directly or foreground-launched gateway processes when no service manager is loaded, instead of only reporting `Gateway service disabled.`

## Best Possible Solution

Land #72224 after maintainer review and rebase, or land a narrower equivalent that accepts verified single-token `openclaw-gateway` processes under the existing opt-in gate and adds Linux listener discovery that works when `lsof` is unavailable, with regression coverage for foreground `gateway stop`.

## Reproduction Assessment

Yes. The reporter supplied concrete foreground-gateway steps, and current source inspection shows the same path: no verified listener PID leads to the service-disabled fallback instead of SIGTERM.

## Solution Assessment

Yes. A narrow verified-process fallback is the best solution because it preserves gateway-only process verification before signaling; a message-only change would leave foreground gateway users manually killing a valid OpenClaw process.
