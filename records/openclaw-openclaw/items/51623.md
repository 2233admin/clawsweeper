## Summary

Keep this PR open. Current main still does not implement durable followup-queue persistence/replay across gateway restarts, the paired tracker #51620 is still open, and the related #51803 PR is also open rather than a merged replacement. The PR remains a useful implementation candidate, but its replay approach needs maintainer review before merge, especially around trusted system-event injection of user-originated message text.

## Best Possible Solution

Keep this PR open for maintainer review or replacement. The best path is a durable pending-inbound/followup recovery design that persists typed followup snapshots before shutdown, handles drain-window rejection separately, consumes persisted data idempotently on startup, replays through normal untrusted inbound semantics or trusted:false system events, and includes focused tests for routing, trust, malformed state, duplicate replay, and restart timing. If #51803 or another branch becomes the selected implementation, close this PR only after that replacement is merged or maintainers explicitly choose the canonical path.
