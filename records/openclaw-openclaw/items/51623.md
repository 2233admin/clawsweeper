## Summary

Keep open. Current main still does not persist or replay FOLLOWUP_QUEUES across a real gateway restart, and the paired upstream issue #51620 remains open. This PR is still an implementation candidate, but it needs maintainer review because the proposed diff has unresolved functional and security review concerns.

## Best Possible Solution

Keep the PR open for maintainer review or replacement. The best path is a durable pending-inbound/followup recovery design tied to the existing session-delivery or inbound routing contracts, with idempotent snapshot consumption, no hard short TTL for valid queued user work, replay text treated as untrusted content, and explicit coverage for the open #51620 phases.

## Risks / Open Questions

- Merging this PR unchanged could elevate replayed user message text into trusted System: prompt lines unless the replay path marks recovered content untrusted and formats/quotes it accordingly.
- The proposed five-minute staleness guard would still lose queued messages after slower manual downtime or repeated failed restarts, which conflicts with the stated restart-survival goal.
- The in-process restart path preserves some command-lane state; a followup persistence design needs to avoid double-processing by clearing or idempotently consuming queue snapshots after a successful durable handoff.
