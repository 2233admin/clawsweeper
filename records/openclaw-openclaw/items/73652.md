## Summary

Keep open. Current main still intentionally binds the Gateway HTTP/WebSocket port before post-attach sidecars finish and before the `gateway ready` log. It exposes `/readyz` for startup readiness and gates a small set of RPC methods, but the WebSocket upgrade/connect path itself does not return an explicit not-ready signal while `startup-sidecars` is pending. The newer v2026.4.26 report therefore is not safely closable as implemented by the older startup/readiness fixes.

## Best Possible Solution

Keep this issue open and treat it as a remaining Gateway startup/readiness bug. The best fix should be in the Gateway startup/WS boundary: either delay accepting WebSocket upgrades until the readiness contract is satisfied, or return a prompt, retryable not-ready response during `startup-sidecars` and teach Control UI/TUI/CLI clients to retry it cleanly. Add a regression test that opens a client while sidecars are deliberately held pending and asserts no long handshake timeout.
