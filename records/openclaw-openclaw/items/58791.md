## Summary

Keep the PR open. Current main still does not expose `registerStreamFnWrapper` or a generic `streamFnWrappers` registry, and the existing alternatives are narrower: provider plugins can use `createStreamFn`/`wrapStreamFn`, while non-provider plugins only have pre-turn model selection and observation hooks. The PR is not auto-closeable as ClawHub because it is specifically proposing a missing core Plugin SDK/API seam, but its current design needs maintainer/security review before merge.

## Best Possible Solution

Keep the PR open for maintainer API review. The best path is likely a narrower, explicitly ordered per-call model/stream hook that runs before provider stream selection when model routing is intended, validates plugin outputs, preserves core/provider safety wrappers as non-bypassable, and ships with Plugin SDK docs plus tests for registration mode, composition, invalid return values, and multi-call tool-loop behavior.

## Risks / Open Questions

- The proposed broad `(streamFn) => streamFn` API may be too powerful for untrusted/non-provider plugins unless core preserves non-bypassable safety, auth, telemetry, and timeout wrappers.
- Applying wrappers after provider stream resolution may not satisfy cross-provider or cross-API per-call model routing, which is a central PR motivation.
- A public SDK addition needs contract tests, docs, and capability/security policy, not just registry plumbing.
