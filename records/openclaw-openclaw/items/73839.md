## Summary

Keep open. Current main still has the Pi auto-compaction guard gap, and this PR is pointed at a real bug, but the latest diff is still incomplete because documented provider-backed safeguard compaction can resolve to safeguard while the new guard only receives the raw `compaction.mode` value.

## What This Changes

The PR adds a `compactionMode` parameter to the Pi auto-compaction guard, passes the raw configured mode from embedded and CLI compaction paths, reapplies the guard after resource reload, and adds focused `pi-settings` tests.

## Best Possible Solution

Land a narrow fix that disables Pi's internal auto-compaction whenever OpenClaw's resolved compaction lifecycle is safeguard-owned, including provider-forced safeguard mode, while preserving the existing context-engine `ownsCompaction` behavior and the post-reload reapply step.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists with `agents.defaults.compaction.provider` set and `compaction.mode` omitted or explicitly `"default"`: current runtime enables the safeguard extension, but this PR's guard still sees a non-safeguard raw mode and leaves Pi auto-compaction enabled.

## Solution Assessment

No. The direction is right for explicit `mode: "safeguard"`, but the maintainable fix should share or mirror the existing provider-aware resolved safeguard predicate for both embedded and CLI call sites.
