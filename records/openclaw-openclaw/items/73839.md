## Summary

Keep open: this PR addresses a real current-main compaction gap, and later commits appear to have fixed the earlier type-shape review comments, but the patch still passes only the raw configured mode into the guard. That misses the documented provider-backed safeguard path, so the same Pi/OpenClaw compaction conflict can remain for supported configs.

## What This Changes

The PR extends the Pi auto-compaction guard to accept `AgentCompactionMode`, passes `agents.defaults.compaction.mode` through embedded and CLI compaction paths, reapplies the guard after embedded resource reload, and adds focused `pi-settings` unit tests.

## Best Possible Solution

Land a narrow repair that computes one provider-aware “OpenClaw owns compaction” decision for explicit safeguard mode, provider-forced safeguard mode, and context-engine `ownsCompaction`, then use that decision consistently before and after embedded resource reload. Add focused tests for explicit safeguard and provider-backed safeguard, plus the required active-version changelog entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `agents.defaults.compaction.provider` set with raw `compaction.mode` unset or `"default"`: current docs and `resolveCompactionMode()` enable safeguard, while the PR guard would still see only the raw mode and leave Pi auto-compaction enabled.

## Solution Assessment

No. The PR is close for explicit `mode: "safeguard"`, but the best fix should share the provider-aware resolver or pass a resolved ownership boolean rather than threading raw config mode values through call sites.
