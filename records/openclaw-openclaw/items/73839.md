## Summary

Keep this PR open. Current main still has the reported Pi/safeguard compaction conflict, and the PR is the focused implementation candidate for the linked bug, but the latest diff should align the guard with the same provider-aware safeguard predicate used by the runtime before merge.

## What This Changes

The PR threads `AgentCompactionMode` into the Pi auto-compaction guard, disables Pi auto-compaction for `safeguard` mode, passes the mode from embedded and CLI compaction paths, reapplies the guard after resource reload, and adds targeted `pi-settings` tests.

## Best Possible Solution

Land a narrow fix that disables Pi's internal auto-compaction whenever OpenClaw's resolved compaction lifecycle is safeguard-owned, including provider-backed safeguard activation, while preserving the existing `ownsCompaction` behavior and reapplying the guard after resource reload. Add coverage for explicit safeguard, explicit default, context-engine ownership, and provider-forced safeguard mode.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: on current main, configure safeguard compaction and no owning context engine; the guard cannot see that mode, while the Pi SDK dependency keeps internal threshold compaction enabled by default and the linked bug provides repeated `already_compacted_recently` logs.

## Solution Assessment

Not yet. The PR takes the right narrow direction, but the changed call passes only raw `agents.defaults.compaction.mode`; the runtime's actual safeguard activation is provider-aware, so the guard should share or mirror that resolved predicate before this is the best fix.
