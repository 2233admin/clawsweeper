## Summary

Keep this open. Current main still does not implement the requested per-event SystemEvent audience/classification primitive; exec, cron, and heartbeat handling still relies on event text plus delivery/visibility routing, and the related audience implementation remains an open partial PR rather than merged main behavior.

## What This Changes

Requests a canonical backend event-level audience/classification primitive for system-event-shaped exec, cron, and heartbeat events so internal events do not leak to user-facing transcripts or channel delivery.

## Best Possible Solution

Keep this as the canonical design tracker until maintainers either land an accepted SystemEvent audience model across the event shape, producers, heartbeat relay, and consumer filters, or explicitly supersede the remaining work with narrower canonical issues for the tactical heartbeat/exec paths.

## Reproduction Assessment

Yes for the implementation gap. A high-confidence source-level reproduction is to enqueue exec- or cron-shaped SystemEvent entries and inspect the current heartbeat/consumer paths: main derives visibility from delivery flags and text predicates because there is no event audience field to honor.

## Solution Assessment

Unclear pending maintainer design. The proposed audience primitive is a plausible maintainable direction, but current main only has tactical text-based filtering/sanitization and the open #72201 scope deliberately does not cover all delivery-routing cases raised here.
