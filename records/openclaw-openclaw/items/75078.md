## Summary

Keeping this PR open for normal maintainer review. Current main still deduplicates Feishu inbound/final processing by the raw `message_id`, while this PR applies a focused Feishu-owned media-aware key using parsed media identifiers and adds a regression test for the linked audio reuse bug.

## What This Changes

The PR changes Feishu inbound debounce, processing-claim, suppression, and final dedupe keys to include parsed media keys for media events, and adds a regression test for two audio events sharing a `message_id` but using different `file_key` values.

## Best Possible Solution

Land a Feishu-owned media-aware dedupe fix after maintainer review, preserving raw `message_id` behavior for non-media events and true retries while allowing distinct media attachments with reused Feishu ids to reach the agent.

## Reproduction Assessment

Yes. Current main has a high-confidence trace reproduction: two Feishu audio events for the same account with the same `message_id` and different `file_key` values collide in both the early processing claim and final persistent dedupe paths.

## Solution Assessment

Yes. The PR is a narrow maintainable solution because it stays inside the Feishu plugin and uses media identifiers already parsed from Feishu payloads; consolidating the duplicated helper into one local module would be cleaner but is not required before merge.
