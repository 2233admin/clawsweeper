## Summary

Keep open. Current main still has a global `subagent` lane and a reject-only per-session active child cap; it does not provide the requested parent-keyed queued spawn throttle, and the related open issues cover per-target and named-lane concurrency rather than this exact behavior.

## What This Changes

The issue requests a configurable per-parent concurrency cap for subagent spawns, with excess child spawns queued instead of all contending on the gateway/WebSocket at once.

## Best Possible Solution

Add a documented requester/parent-keyed spawn throttle that sits alongside the global subagent lane cap. Queued native and ACP child runs should be visible in task/subagent state, preserve fairness for unrelated parents, and define stop, restart, cleanup, and completion behavior explicitly.

## Reproduction Assessment

Yes for the missing product behavior: source, docs, and tests on current main show global lane concurrency plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not reproduced live in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but the exact scheduler semantics and lifecycle integration are broad enough that a maintainer design pass should come before an automated fix PR.
