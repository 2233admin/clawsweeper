## Summary

Keep open. Current main still has a global `subagent` execution lane plus a per-session active-child reject guard, but it does not implement the requested per-parent queued spawn concurrency throttle with a default around 2-3. The related open issues #8061 and #10467 are adjacent design work, but they track per-target-agent and multi-lane concurrency rather than this exact per-parent queueing behavior.

## What This Changes

The issue requests a configurable per-parent subagent spawn concurrency cap where excess spawns queue instead of saturating the global lane or being rejected.

## Best Possible Solution

Add a documented per-requester spawn concurrency throttle that is distinct from the global `subagent` lane and the reject-only `maxChildrenPerAgent` cap. Queued native and ACP child runs should remain visible in task/subagent state, start when that parent has capacity, preserve unrelated-parent fairness on the global lane, and define restart/cancel behavior explicitly.

## Reproduction Assessment

Yes for the missing product behavior: current source and tests show `maxConcurrent` is global, not parent-keyed, and `maxChildrenPerAgent` rejects instead of queueing. The reported large-fleet Slack timeout symptom was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle is the right product shape for the report, but the exact defaults, fairness rules, task-state representation, and restart/cancel semantics need maintainer design before this is a safe narrow implementation.
