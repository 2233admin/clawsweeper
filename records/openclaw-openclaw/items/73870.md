## Summary

Keep open. Current main still has a global subagent lane cap plus a per-session active-child reject guard, but it does not implement the requested queued per-parent spawn throttle; related open items cover adjacent per-agent and multi-lane concurrency questions rather than replacing this report.

## What This Changes

The issue requests a configurable per-parent queued concurrency cap for subagent spawns so one parent cannot flood the gateway and cause event-loop/channel timeouts.

## Best Possible Solution

Design and implement a parent-scoped spawn scheduler that limits concurrently running children per requester while queuing excess accepted spawns, applies consistently to native and ACP runtime paths, composes fairly with the global `subagent` lane, and updates config schema, docs, and regression tests to clarify how it differs from `maxChildrenPerAgent`.

## Reproduction Assessment

Yes for the missing product behavior: current source and `src/agents/subagent-spawn.depth-limits.test.ts` show `maxConcurrent` is not a per-parent spawn gate and `maxChildrenPerAgent` rejects instead of queues. The reported Slack CPU/pong-timeout symptom itself still needs a live stress reproduction.

## Solution Assessment

No, the current global lane plus reject-only child cap is not the best solution for this issue because it does not provide per-parent fairness or queued excess work. The safer direction is a deliberate parent-scoped queue with tests across native and ACP spawns.
