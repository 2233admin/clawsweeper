## Summary

Keep open. Current main still provides one global `subagent` execution lane and a reject-only active-child guard per requester session; it does not add the requested queued per-parent spawn throttle. The related open concurrency issues cover per-target caps and named lanes, but neither fully subsumes parent-keyed queued fanout control.

## What This Changes

The issue requests a configurable per-parent concurrency cap for subagent spawns, with excess child spawns queued to reduce gateway event-loop and WebSocket contention during large fanout.

## Best Possible Solution

Add a documented requester/parent-keyed queued spawn throttle that is separate from the global subagent lane and the active-child reject cap. The design should cover native and ACP child runs, task/subagent state visibility, fairness between unrelated parents, and stop/restart/cleanup/completion semantics.

## Reproduction Assessment

Yes for the missing product behavior: current source, docs, and tests show global lane concurrency plus reject-only per-session child limits, not queued per-parent throttling. The large-fleet Slack timeout symptom itself was not live-reproduced in this read-only pass.

## Solution Assessment

Unclear. A queued per-parent throttle matches the report, but scheduler defaults, fairness, state representation, and lifecycle semantics are broad enough to need maintainer design before a narrow repair is safe.
