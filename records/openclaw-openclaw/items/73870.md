## Summary

Keep open. Current main still has a global subagent lane cap and a per-session active-child reject guard, but it does not implement the requested per-parent queued spawn throttle with a default around 2-3. A current regression test explicitly preserves that `agents.defaults.subagents.maxConcurrent` is not used as a per-parent spawn gate.

## Best Possible Solution

Keep this issue open for a maintainer design pass. The best fix is a real per-parent queued subagent spawn throttle, likely near `src/agents/subagent-spawn.ts`, `src/agents/acp-spawn.ts`, subagent registry/task tracking, and the command lane boundary: add a clear config key/default, queue excess children from one requester while allowing unrelated parents to use the global lane fairly, preserve `sessions_spawn`'s non-blocking accepted response, and cover native plus ACP behavior in tests and docs.

## Risks / Open Questions

- The existing global `subagent` lane cap can reduce total work but still allows one parent to occupy all subagent lane slots, so it does not solve per-parent fairness.
- `maxChildrenPerAgent` is a reject-only active-child ceiling with default 5, not the requested queued per-parent concurrency default around 2-3.
- The reported Slack pong timeout may be caused by spawn start pressure, completion announce delivery over the shared WebSocket, or both; the design should verify the hot path before choosing the throttle point.
- A queued per-parent throttle must compose with the non-blocking `sessions_spawn` accepted result contract, native and ACP spawn paths, child run registry state, cancellation, and the existing global lane without introducing starvation or deadlocks.
