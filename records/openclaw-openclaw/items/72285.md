## Summary

Keep this PR open. Current main still treats `totalTokensFresh: false` as unknown in the shared status formatter and does not use compaction checkpoint `tokensAfter`, so the proposed fix addresses a real remaining status bug, but the branch should be adjusted for one checkpoint-order edge case before merge.

## What This Changes

The PR updates the shared status message builder to fall back to compaction checkpoint `tokensAfter` when session totals are stale and adds a regression test for the post-compaction Context line.

## Best Possible Solution

Land a focused formatter fix that uses a valid `tokensAfter` from the actual latest compaction checkpoint when no fresh total is available, leaves stale totals unknown when the latest checkpoint has no post-compaction count, and keeps regression coverage tied to #72140.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Use only the newest checkpoint's tokensAfter:** `src/status/status-message.ts:264-274`
  - body: `tokensAfter` is optional, but this helper filters first and then chooses the newest checkpoint among the remaining entries. If a later compaction checkpoint lacks `tokensAfter`, status would reuse an older checkpoint's post-compaction count as if it were current. Pick the latest checkpoint first, then use its `tokensAfter` only when it is a valid number.
  - confidence: 0.84
- **[P3] Remove the unreachable checkpoint fallback branch:** `src/status/status-message.ts:656-657`
  - body: After the new fallback expression, `totalTokens` is only `undefined` when `totalTokensFresh === false`, which the preceding `if` already handles; otherwise the fallback total is always numeric. Dropping this branch keeps the stale-total logic easier to audit without changing behavior.
  - confidence: 0.91
