## Summary

Keep this PR open. Current main still has a real status-reporting gap for stale totals with checkpoint data, but the proposed helper is not quite correct because it filters out checkpoints without `tokensAfter` before selecting the latest checkpoint, which can report an older post-compaction count as current.

## What This Changes

The PR updates shared status message rendering to fall back from stale session totals to compaction checkpoint `tokensAfter` values and adds a regression test in `src/auto-reply/status.test.ts`.

## Best Possible Solution

Land a focused formatter change that selects the actual latest compaction checkpoint first, uses its `tokensAfter` only when it is a valid non-negative number, leaves stale totals unknown when the latest checkpoint lacks a post-compaction count, and adds coverage for both the normal fallback and the newer-checkpoint-without-token case.

## Reproduction Assessment

Yes. The supplied test shape reproduces the intended bug by calling `buildStatusMessage()` with `totalTokensFresh: false`, a stale `totalTokens`, and a checkpoint with `tokensAfter`; the PR should show the checkpoint value. A second high-confidence counterexample is two checkpoints where the newer checkpoint lacks `tokensAfter`.

## Solution Assessment

No. The requested direction is right, but this implementation is not the best fix because it finds the newest checkpoint only after filtering out checkpoints without `tokensAfter`; the safer solution is to choose the newest checkpoint first and then validate that checkpoint's token value.

## Review Findings

Full review comments:

- **[P2] Select the newest checkpoint before reading tokensAfter:** `src/status/status-message.ts:264-274`
  - body: `tokensAfter` is optional, but this helper filters out checkpoints without it before choosing the newest checkpoint. If a later compaction checkpoint has no token count, status will reuse an older checkpoint's post-compaction number as if it were current. Pick the latest checkpoint first, then use its `tokensAfter` only when it is a valid number.
  - confidence: 0.86
- **[P3] Remove the unreachable checkpoint fallback branch:** `src/status/status-message.ts:656-657`
  - body: After the new fallback expression, `totalTokens` is only `undefined` when `entry.totalTokensFresh === false`, which the preceding branch already handles. Dropping this `else if` keeps the stale-total fallback easier to audit without changing behavior.
  - confidence: 0.91
