## Summary

Keep this issue open. Current main still has a credible Telegram DM echo gap: DM voice notes are preflight-transcribed so the agent can receive text, but that same preflight path marks the audio as already transcribed and the shared echo logic only runs for new audio transcription outputs from the later media-understanding pass. This looks narrower than the broader open Telegram DM delivery reports and is actionable as a focused media/Telegram handoff fix.

## What This Changes

The issue requests that Telegram direct-chat voice-message transcripts be echoed back when `tools.media.audio.echoTranscript` is enabled, matching the behavior observed in Telegram forum groups.

## Best Possible Solution

Make preflight transcription participate in the same echo contract as normal media understanding: carry the preflight transcript as structured context or emit through a small shared media-runtime seam, then send the configured echo exactly once before agent processing without re-running STT or changing Telegram DM/forum routing.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a Telegram private voice message with no text and `tools.media.audio.echoTranscript: true`: Telegram preflight transcribes it, marks the media index as already transcribed, and the later shared media pass skips the audio so `sendTranscriptEcho` is never called.

## Solution Assessment

Yes, the best fix is to repair the preflight-to-echo handoff rather than treating this primarily as a Telegram send routing failure. The existing Telegram outbound route can deliver text to `OriginatingTo`; the missing piece is that preflight transcripts do not enter the echo branch.
