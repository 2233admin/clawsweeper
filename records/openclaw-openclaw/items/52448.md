## Summary

Keep open. Current main does not implement this PR’s intended sandbox authority-link hydration, and the PR is not clearly superseded by the related context provided. The work is a focused sandbox behavior change, but it still needs maintainer product/security review because the proposed approach changes what `workspaceAccess: "none"` and `"ro"` can expose inside `/workspace`.

## Best Possible Solution

Keep this PR open for maintainer review only if OpenClaw wants first-class sandbox authority-link hydration. The best implementation would preserve the current safety contract: explicit allowlist, resolved-target validation rooted in the seed workspace, recursive copy that rejects symlinks and hardlinks, clear freshness semantics for reused sandbox workspaces, Windows-safe tests, and docs updating the public sandbox/agent-workspace contract. If maintainers decide that symlinked authority docs conflict with the current seed model, close this manually with that product/security rationale rather than as automated cleanup.

## Risks / Open Questions

- The proposed change is security-sensitive: copying allowed symlink names without validating the resolved target against the seed workspace can expose arbitrary host files inside `/workspace` for `workspaceAccess: "none"` or `"ro"`.
- Hardlinks inside copied authority trees need the same rejection semantics as `openBoundaryFile`; otherwise alias-escape protections can be bypassed by copying linked content into the sandbox.
- The PR’s symlink-heavy tests need platform guards or equivalent Windows-safe coverage, matching existing sandbox symlink tests that use `it.runIf(process.platform !== "win32")`.
- Agent-scoped sandbox workspaces are reused, so the design should define whether authority copies are refreshed, replaced, or intentionally stale across sessions.
