## Summary

Keep open: current main still only seeds fixed bootstrap files, so this PR is not obsolete, but the latest PR head still expands the sandbox filesystem boundary without enough target validation or hardlink protection. The prior ClawSweeper security review remains substantively unresolved against the current patch.

## What This Changes

The PR adds sandbox workspace hydration for allowlisted top-level authority symlinks such as README.md, CLAUDE.md, docs, and voro-docs, plus unit coverage for materialization, allowlisting, nested symlink skipping, and non-symlink exclusion.

## Best Possible Solution

Define a documented authority-link model before merging: explicit supported names, resolved-target validation against approved roots and blocked roots, symlink and hardlink rejection throughout copied trees, deliberate refresh behavior for reused sandboxes, Windows-safe tests, and docs that explain exactly which authority files can be hydrated.

## Reproduction Assessment

Yes. The review findings can be reproduced from the PR head by pointing an allowlisted README.md or docs symlink at an outside path, or by placing a same-filesystem hardlink inside an allowed authority tree; the PR copy path does not use the current boundary-file root and hardlink checks.

## Solution Assessment

No. Copying resolved authority targets into the sandbox without a documented root policy or hardlink rejection is not the safest maintainable solution for this sandbox boundary; the current seed contract should remain restrictive unless maintainers explicitly expand it.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority targets before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted name and then copies that resolved path without checking it against an approved authority root or blocked-root policy. A workspace can point `README.md` or `docs` at a credential directory or sibling checkout and have that content materialized into a `workspaceAccess: "none"`/`"ro"` sandbox.
  - confidence: 0.93
- **[P1] Reject hardlinked authority files:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `lstat()` plus `copyFile()` treats hardlinked files as ordinary files. Current bootstrap seeding rejects hardlink aliases through `openBoundaryFile`, but this path would copy a same-filesystem hardlink inside an allowed authority tree into the sandbox as readable content.
  - confidence: 0.9
- **[P2] Refresh authority copies for reused sandboxes:** `src/agents/sandbox/workspace.ts:40-42`
  - body: `copyAuthorityTreeIfMissing` returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later changes to `README.md` or `docs` symlink targets, or edits left in the copied sandbox files, will not be refreshed for later sessions.
  - confidence: 0.86
- **[P2] Guard symlink tests on Windows:** `src/agents/sandbox.workspace.test.ts:25-34`
  - body: The new tests create file and directory symlinks unconditionally. Existing sandbox symlink/hardlink tests are skipped on `win32` because Windows runners may lack symlink privileges, so this added file can fail the Windows lane before reaching the sandbox assertions.
  - confidence: 0.88
