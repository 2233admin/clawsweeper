## Summary

Keep open: current main still does not implement authority-link hydration, but this PR changes the sandbox filesystem boundary and the latest PR head still has unresolved blocking trust-model issues around resolved-root validation and hardlink alias handling. The prior ClawSweeper review comments match the current diff and remain relevant.

## What This Changes

The PR adds allowlisted top-level symlink authority hydration in `ensureSandboxWorkspace` plus sandbox workspace tests for materialization, allowlisting, nested symlink skipping, and non-symlink exclusion.

## Best Possible Solution

Define a documented authority-link model before merging: supported names, resolved-target validation against the intended root policy, symlink and hardlink rejection throughout copied trees, deliberate refresh behavior for reused sandboxes, Windows-safe tests, and docs that explain exactly which authority files can be hydrated.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Validate authority links before copying:** `src/agents/sandbox/workspace.ts:86-91`
  - body: `fs.realpath(src)` resolves an allowlisted workspace name and then copies that target without checking it is inside the seed workspace or another explicitly trusted root. A workspace can point `README.md` or `docs` at a credential directory or sibling checkout and materialize that content into a non-rw sandbox.
  - confidence: 0.93
- **[P1] Reject hardlinks while copying authority trees:** `src/agents/sandbox/workspace.ts:56-57`
  - body: `lstat()` plus `copyFile()` treats hardlinked files as ordinary files. Current bootstrap seeding rejects hardlink aliases through `openBoundaryFile`, but this path would copy a same-filesystem hardlink inside an allowed authority tree into the sandbox as readable content.
  - confidence: 0.9
- **[P2] Refresh existing authority copies:** `src/agents/sandbox/workspace.ts:40-42`
  - body: `copyAuthorityTreeIfMissing` returns as soon as the destination exists. Agent-scoped sandboxes are reused, so later source changes or sandbox-side edits can leave future sessions reading stale authority content instead of current workspace docs.
  - confidence: 0.86
- **[P2] Guard symlink tests on Windows:** `src/agents/sandbox.workspace.test.ts:25-34`
  - body: The added tests create file and directory symlinks unconditionally. Existing sandbox alias tests are skipped on `win32` because Windows runners may lack symlink privileges, so these cases can fail the Windows lane before reaching the intended assertions.
  - confidence: 0.88
