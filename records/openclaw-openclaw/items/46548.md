## Summary

Current main only partially addresses the report. Retry-noise is mostly handled by clearing a stored mutating tool failure when the same action later succeeds, but the normal non-verbose chat warning for visible mutating tool failures still omits `lastToolError.error`; existing tests explicitly assert that file/write/edit failure details are absent. The related PR #46592 was closed unmerged and, by its own body, did not implement retry-success confirmation.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment. A good path would define a redacted or allowlisted failure-reason policy for safe built-in tool failures, preserve raw details for verbose/full mode, and decide whether retry success should remain silent suppression or emit a confirmation only when a failure warning was already delivered.

## Risks / Open Questions

- Surfacing raw `lastToolError.error` in non-verbose chat can leak local paths, command output, channel targets, provider responses, or secrets into Discord/Telegram-style monitoring surfaces unless the detail is redacted or allowlisted.
- Existing policy and tests intentionally omit error details for default mutating tool warnings, so changing this globally may alter privacy and chat-noise expectations.
- Explicit retry-success confirmation could create additional channel noise unless it is tied to a previously delivered warning rather than every internally retried tool failure.
