## Summary

Keep this PR open. Current main already covers the chat model picker half by building provider-qualified option values, but it does not implement the PR's central top-level `llm` shorthand config key. The proposed config work also needs maintainer/config-owner review because it adds a credential-bearing public config alias and the existing review comments identify correctness gaps around provider/model merging, fallback preservation, and provider validation.

## Best Possible Solution

Keep this PR open for maintainer/product judgment or request a focused revision. If OpenClaw wants a root `llm` shorthand, implement it in the config owner area with explicit type/schema/generated-schema/help/docs support, SecretInput and redaction coverage, provider/API/ref validation, non-destructive provider/model merging, fallback-preserving primary assignment, and regression tests. Rebase away the chat picker hunk because current main already has that behavior.

## Risks / Open Questions

- Closing this PR would drop the still-unimplemented root `llm` shorthand request even though only the UI picker part is redundant on current main.
- Merging the PR as-is could create invalid or dangling primary model refs when `provider_id` collides with an existing provider whose model list does not include `llm.model`.
- The proposed `api_key` root alias is credential-bearing; it needs generated schema/help/redaction coverage and SecretInput behavior reviewed before becoming a public config surface.
- The PR's free-form `llm.provider` and minimally validated `provider_id` could accept values that downstream provider API/ref parsing does not support.
