## Summary

Keep this issue open. Current main has addressed or documented adjacent pieces, including DeepSeek provider onboarding, additive config patch behavior, and unmanaged gateway restart fallback, but the central model-switch UX is still not resolved as one coherent product path: picker visibility still follows `agents.defaults.models` when present, doctor does not appear to warn about a hidden newly configured provider/default combination, and existing sessions continue honoring stored runtime/override model state until explicitly patched.

## What This Changes

The issue reports five UX problems encountered while switching the default model from Qwen to DeepSeek, covering provider addition, dual model config, allowlist behavior, gateway restart, and active-session model refresh.

## Best Possible Solution

Split this into maintainer-owned decisions: keep the bundled DeepSeek onboarding path, add high-signal model/doctor/status warnings when a configured provider or default is hidden by `agents.defaults.models`, and design a first-class `openclaw models add/set` flow that updates provider config and allowlist together without making active sessions silently inherit new defaults.

## Reproduction Assessment

Partially. The report gives concrete symptoms and environment details, and current source/tests verify the remaining allowlist and session-state behavior, but the exact local config and patch payload needed to replay all five symptoms end to end are missing.

## Solution Assessment

Unclear. Automatically expanding allowlists or rewriting active sessions on config reload may override intentional user scoping, so warnings plus an explicit model setup/switch command look safer than a silent core policy change.
