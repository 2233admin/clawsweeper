## Summary

Keep this issue open. Current main has bounded safe-send retry for clearly pre-connect Telegram send failures and a shared outbound crash-recovery queue, but the interactive Telegram final-reply path still sends directly through `deliverReplies`, rethrows delivery failures, and does not persist pending or ambiguous final chunks for runtime drain after polling recovery.

## What This Changes

The issue requests durable runtime Telegram outbound delivery recovery for polling-stall windows, including safe/ambiguous/permanent failure classification and stateful retry handling for already-computed replies.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct send, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and hooks eligible recovery into polling restart as well as startup where the shared queue contract fits.
