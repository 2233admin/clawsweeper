## Summary

Current main has related reliability pieces, but it does not fully cover the reported Telegram inbound final-reply loss path. The shared `deliverOutboundPayloads` path has a write-ahead queue and startup recovery, and the standalone Telegram send helper has conservative safe-send retry logic. However, Telegram inbound reply delivery still routes final payloads through `bot-message-dispatch` -> `deliverReplies` -> `sendTelegramText` -> direct `bot.api.sendMessage`; that wrapper only handles quote/thread/format fallbacks and rethrows other send failures. Polling stall recovery rebuilds the polling transport, but I did not find a runtime worker or post-restart drain for these direct final reply chunks. The issue should stay open for maintainer follow-up.

## Best Possible Solution

Keep this open and design a Telegram reply-delivery recovery path that covers direct inbound final replies, not only CLI/tool/shared outbound sends. A good fix would persist or otherwise safely recover already-computed final reply chunks before direct `bot.api.sendMessage`, retry only pre-connect/rate-limit safe failures with bounded backoff, preserve ambiguous failures without blind duplication, and trigger a drain after Telegram polling recovery as well as startup. Tests should cover `deliverReplies`/lane final reply failures, pre-connect safe retry, ambiguous no-retry preservation, and polling-restart drain behavior.

## Risks / Open Questions

- Non-idempotent Telegram sends cannot be blindly replayed for ambiguous errors such as timeouts or connection resets without duplicate-message risk; the fix needs explicit safe/ambiguous/permanent classification.
- The existing generic outbound queue does not cover Telegram direct inbound replies. Wiring it in naively could duplicate message hooks, reply-thread consumption, preview finalization, or media behavior.
- Related #53895 is closed, but the current direct auto-reply path still shows the remaining runtime recovery gap; this issue should not be closed as a duplicate unless a merged fix covers that path.
