## Summary

Keep this open. Current main still has the durable shared outbound queue and bounded Telegram safe-send retry, but interactive Telegram final replies are still delivered through the direct `deliverReplies` path and failed sends are rethrown without creating pending, retryable, or ambiguous outbox state. The related closed queue and safe-retry items cover adjacent or narrower fixes, not the continuous runtime recovery requested here.

## What This Changes

The issue requests runtime Telegram outbound delivery recovery for polling-stall windows, including safe retry classification and durable pending/ambiguous state for failed final replies.

## Best Possible Solution

Add a maintainer-reviewed Telegram reply-delivery recovery design that records already-computed final/block reply chunks before or during direct delivery, retries only failures classified as safe, preserves or surfaces ambiguous failures without blind duplication, and triggers eligible recovery during polling restart as well as startup where the shared queue contract fits.
