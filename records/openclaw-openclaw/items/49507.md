## Summary

Keep this open. Current main still does not implement the requested cron `agentId` to `accountId` inference, and the remaining behavior can still select a session-derived `lastAccountId` before any agent-bound account lookup. The related direct fix PRs are closed unmerged, and the related Slack identity issue addressed presentation identity rather than this outbound account-routing gap.

## Best Possible Solution

Keep the issue open for a focused maintainer-reviewed cron delivery identity fix. The likely direction is to keep explicit `delivery.accountId` highest priority, decide when cron-owned explicit delivery should prefer the job agent's bound outbound account over stale session state, and only add `agentId === accountId` as a documented/validated compatibility fallback if that is an intended contract. Add regression coverage for Feishu/Telegram-style multi-account cron jobs, explicit `--account`, binding-derived account IDs, stale `lastAccountId`, and session-bound routes that intentionally preserve live account context.
