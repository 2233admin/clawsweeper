## Summary

Keep open: current main still enables the safeguard extension independently of context-engine ownership, but the Pi auto-compaction guard only disables Pi auto-compaction for context engines with `ownsCompaction === true`. Open PR #73839 is the active, unmerged implementation candidate that explicitly targets this report, so this issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests disabling or coordinating Pi SDK auto-compaction when OpenClaw safeguard compaction mode is active so preflight safeguard compaction does not get stuck behind repeated `already_compacted_recently` failures.

## Best Possible Solution

Land #73839 or an equivalent narrow fix so the central Pi auto-compaction guard treats configured safeguard mode as OpenClaw-owned compaction, threads that policy through embedded and CLI compaction paths, and reapplies it after resource-loader settings reloads while preserving default/off and context-engine-owned behavior.

## Reproduction Assessment

Yes. Configure `agents.defaults.compaction.mode = "safeguard"` without an owning context engine; current main enables the safeguard extension but `shouldDisablePiAutoCompaction` still returns false for the Pi guard, leaving Pi SDK auto-compaction active.

## Solution Assessment

Yes. The narrow maintainable direction is to centralize the ownership decision in `pi-settings` and pass the resolved compaction mode through the existing embedded and CLI guard call sites, rather than adding Telegram-specific recovery or post-failure retries.
