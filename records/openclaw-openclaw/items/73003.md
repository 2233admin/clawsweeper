## Summary

Keep open. Current main still disables Pi auto-compaction only when a context engine declares `ownsCompaction`, while safeguard compaction is enabled through the embedded extension path from config mode/provider; the open unmerged PR #73839 specifically references and attempts the narrow fix, so this issue should remain open until that PR or an equivalent patch lands.

## What This Changes

The issue reports that `compaction.mode: "safeguard"` should disable Pi SDK auto-compaction so OpenClaw-owned safeguard compaction is not blocked by repeated `already_compacted_recently` failures.

## Best Possible Solution

Review and land #73839 or an equivalent narrow patch that makes the Pi auto-compaction guard mode-aware, disables Pi auto-compaction when OpenClaw safeguard owns the lifecycle, threads the mode through embedded and CLI compaction paths, and adds focused regression tests for safeguard, default/off, context-engine-owned, and unsupported-settings-manager cases.

## Reproduction Assessment

Yes. The issue provides concrete config, model, channel, and repeated `already_compacted_recently` outcomes, and current-main source shows the guard cannot fire for safeguard mode unless a context engine owns compaction.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because it keeps the policy in the Pi settings guard and its existing run/CLI call sites instead of adding a startup migration or broader session rewrite behavior.
