## Summary

Keep open. Current main still matches the reported behavior: light dreaming sorts by `lastRecalledAt` before `recallCount`, daily/session ingestion records `daily` signals that refresh `lastRecalledAt`, deep ranking counts daily/grounded signals in the `minRecallCount` gate, and rehydration still uses the narrow `preferredSpan + 3` / `8` window. The reporter also provided concrete environment and store observations, so this is actionable rather than stale cleanup.

## Best Possible Solution

Keep this issue open and fix it in `memory-core` with targeted behavior tests: rank light sleep candidates so real recall strength is visible before fresh daily-only entries, decide whether deep `minRecallCount` should count organic recalls only or use a renamed generic signal threshold, recalibrate score averaging by signal type, and broaden or otherwise make robust the rehydration relocation window for shifted multi-line candidates. Update the current daily-only promotion test and memory dreaming docs to match the chosen contract.

## Risks / Open Questions

- Changing `minRecallCount` to mean organic recalls only would conflict with the current test and docs that allow repeated daily/grounded evidence to satisfy deep promotion gates.
- The best fix may need separate thresholds or weights for organic recall, daily ingestion, grounded backfill, and phase reinforcement so grounded backfill is not accidentally regressed while addressing daily-only noise.
