## Summary

Keep open. Current main still contains the reported Memory Dreaming behavior, and open PR #74051 is already linked with closing syntax as the active implementation candidate, so this issue should remain open until that PR or a focused replacement lands.

## What This Changes

The issue reports that Memory Dreaming lets daily/session ingestion recency and combined signal counts hide real recall data, and that promotion rehydration searches too narrow a span for shifted snippets.

## Best Possible Solution

Review and land #74051, or a focused replacement, so real recalls drive Light ordering and default recall-gate eligibility, daily/session signals can still inform scoring without satisfying recall gates alone, rehydration handles shifted snippets robustly, and tests/docs describe the corrected contract.

## Reproduction Assessment

Yes. Source inspection on current main reproduces the mechanism: daily/session ingestion refreshes lastRecalledAt, Light sorts by that timestamp before recallCount, and deep promotion gates still use combined signalCount rather than real recallCount.

## Solution Assessment

No for the report's initial sort-only patch. The best fix is the broader memory-core/docs/test change represented by #74051 or an equivalent focused replacement that separates recall-gate semantics from daily/session scoring and hardens rehydration.
