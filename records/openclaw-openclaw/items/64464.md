## Summary

Keep open. Current main still does not implement the requested visible WSL2 Chrome launch behavior: it falls back to headless only when headed mode was not explicitly selected, and it returns a no-display error when `headless: false` is explicit. The related DISPLAY fallback PRs are closed/unmerged for the target repository, and the maintainer comment correctly notes that the landed Linux fallback does not synthesize a WSL display.

## What This Changes

The issue asks OpenClaw to set `DISPLAY=:0` when spawning non-headless Chrome from WSL2 so a visible managed browser launch can initialize.

## Best Possible Solution

Add a narrowly scoped WSL2 managed-browser display fallback that only applies to local OpenClaw-launched, headed Chrome when both `DISPLAY` and `WAYLAND_DISPLAY` are unset, while preserving existing display variables and leaving generic Linux no-display hosts on the current headless/error path. Cover the preflight and real spawn-env path with tests, and update the browser troubleshooting docs if the user-facing behavior changes.

## Reproduction Assessment

Yes. The issue body gives WSL2 steps and current main has a deterministic reproduction path: Linux local managed profile with explicit `headless: false` and no `DISPLAY`/`WAYLAND_DISPLAY` reaches `getManagedBrowserMissingDisplayError` before spawning Chrome.

## Solution Assessment

No for the literal unconditional suggestion, but yes for a narrower variant. The maintainable fix should not always set `DISPLAY=:0`; it should be WSL2-specific, preserve user-provided display settings, and include tests around both config resolution and the actual spawn environment.
