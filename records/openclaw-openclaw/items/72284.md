## Summary

Keep open. Current main still has credible paths where REM phase artifacts are written but no Dream Diary entry is appended, and the product contract remains mixed between documented human-readable DREAMS.md output and best-effort narrative generation.

## What This Changes

The issue requests that successful memory-core REM dreaming create or update DREAMS.md so the Control UI Dreams tab can show diary content.

## Best Possible Solution

Make the Dream Diary contract explicit and consistent. If REM reports are meant to guarantee visible UI diary output, memory-core should append a deterministic fallback entry whenever usable REM material exists but narrative generation cannot produce text; otherwise docs and the Control UI empty state should clearly say phase reports can exist without diary entries.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: current main can write the REM phase report before diary generation, and tests assert DREAMS.md remains absent on no-snippet and timeout narrative paths.

## Solution Assessment

Unclear. A deterministic fallback diary write is the narrowest product fix if DREAMS.md is guaranteed output, but current tests and docs also preserve best-effort behavior, so maintainers should choose that contract before implementation.
