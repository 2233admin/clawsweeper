## Summary

Keep this open. Current main still treats the REM phase report and the Dream Diary as separate outputs: REM can write `memory/dreaming/rem/YYYY-MM-DD.md` while `DREAMS.md` remains absent when there is no subagent, no narrative text, a timeout/error, or no usable snippets. The related open PRs are adjacent but do not implement the requested `DREAMS.md` guarantee.

## What This Changes

The issue requests that successful memory-core REM dreaming runs create or update `DREAMS.md` so the Control UI Dreams tab can show diary content.

## Best Possible Solution

Maintainers should make the Dream Diary contract explicit. If successful REM output must always be visible in the Control UI, memory-core should write a deterministic fallback diary entry when the narrative subagent is absent, times out, or produces no text; otherwise the docs and Dreams empty state should explain the current best-effort behavior.

## Reproduction Assessment

Yes. A high-confidence source-level path exists: current main can write the REM phase report first, then skip `DREAMS.md` when no diary narrative is appended, and tests already assert missing `DREAMS.md` for no-snippet and timeout paths.

## Solution Assessment

Unclear. A narrow fallback diary write would solve the reported UI gap, but it should wait for maintainer confirmation because the docs currently mix a `DREAMS.md` expectation with best-effort narrative semantics.
