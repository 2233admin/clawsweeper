## Summary

Keep open. Current main partially improves the Dream Diary path with a request-scoped fallback, but the reported symptom is still a valid product-contract gap: REM phase reports can be written while `DREAMS.md` remains absent, and the Control UI still reads only the diary file.

## What This Changes

The issue requests that a successful REM dreaming run create or update root `DREAMS.md` so the Control UI Dreams diary is populated.

## Best Possible Solution

Make the Dream Diary contract explicit. If a visible diary is guaranteed whenever REM produces usable material, memory-core should append a deterministic fallback entry for no-subagent, timeout, no-text, and request-scoped cases; otherwise docs and the Dreams UI should clearly state that phase reports may exist without diary entries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: `runRemDreaming` can write a REM report before diary generation, and current tests show `DREAMS.md` remains absent for no-snippet and timeout narrative paths.

## Solution Assessment

Unclear. A narrow fallback diary write would solve the reported UI gap if `DREAMS.md` is intended as a guarantee, but current docs and tests preserve best-effort behavior, so maintainer contract choice should come first.
