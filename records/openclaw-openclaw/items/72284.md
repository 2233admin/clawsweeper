## Summary

Keep open. Current main still treats Dream Diary generation as optional and best-effort, so a REM phase report and phase signals can exist while `DREAMS.md` is still absent; the Control UI still reads only the diary file for diary content.

## What This Changes

The issue asks memory-core to create or update `DREAMS.md` after a successful REM phase so the Control UI Dreams tab is populated.

## Best Possible Solution

Define the Dream Diary contract explicitly. If visible diary output is guaranteed after REM has usable material, memory-core should append a deterministic fallback diary entry for timeout, empty narrative, and unavailable narrative paths; otherwise docs and the Dreams UI should surface report-only runs clearly instead of presenting an empty diary as a broken state.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: invoke the REM narrative path with usable snippets but a subagent `waitForRun` timeout, and current tests assert `DREAMS.md` remains absent.

## Solution Assessment

Unclear. A narrow fallback write would solve the reported empty Dreams tab, but current implementation and tests intentionally preserve best-effort no-write behavior, so maintainers should choose the contract before an automated repair changes it.
