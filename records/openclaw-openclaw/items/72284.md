## Summary

Keep this open. Current main still treats REM phase reports and `DREAMS.md` diary entries as separate outputs, with diary creation gated behind best-effort narrative generation; the remaining question is the intended product contract, not a cleanup-safe close.

## What This Changes

The issue requests that a successful memory-core REM phase create or update `DREAMS.md` so the Control UI Dreams diary is populated when REM output exists.

## Best Possible Solution

Maintainers should choose the Dream Diary contract explicitly. If REM output with usable material must always appear in the Control UI, memory-core should write a deterministic fallback diary entry when the narrative subagent is absent, times out, or returns no text; otherwise the docs and Dreams empty state should say that phase reports can exist without a diary entry.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: current code writes the REM report before the diary attempt, and tests show `DREAMS.md` remains absent for no-snippet and timeout narrative paths; the reporter's exact installed environment was not live-tested.

## Solution Assessment

Unclear. Guaranteeing diary creation may be the right user-facing fix, but current docs also describe the diary append as best-effort, so this should not be closed or auto-fixed until maintainers confirm the intended contract.
