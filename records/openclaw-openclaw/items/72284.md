## Summary

Keep open. Current main still allows a REM phase report to be written without a `DREAMS.md` diary entry, and the Control UI only renders diary content from `DREAMS.md` or `dreams.md`; the docs and tests still leave the guarantee-vs-best-effort contract unresolved.

## What This Changes

The issue requests that a successful memory-core REM dreaming run create or update `DREAMS.md` so the Control UI Dreams tab shows Dream Diary content.

## Best Possible Solution

Maintainers should make the Dream Diary contract explicit. If phase artifacts are meant to guarantee visible UI output, memory-core should append a deterministic diary fallback whenever REM has usable report material but narrative generation is unavailable, times out, or produces no text; otherwise docs and the Control UI empty state should clearly explain that phase reports can exist without diary entries.

## Reproduction Assessment

Yes. There is a high-confidence source-level reproduction path: current code can write `memory/dreaming/rem/YYYY-MM-DD.md` before any diary append attempt, and tests assert `DREAMS.md` stays absent on no-snippet and timeout narrative paths.

## Solution Assessment

Unclear. A narrow fallback diary write may be the right fix if `DREAMS.md` is a product guarantee, but current tests and docs also support a best-effort interpretation, so this needs maintainer contract choice before an automated repair PR.
