## Summary

Keep this issue open. Current main still has the reported behavior: cron task records are explicitly created with `notifyPolicy: "silent"`, while subagent task records omit `notifyPolicy` and therefore resolve to the task registry default of `done_only`. The broader #64081 is related, but it does not fully cover the cron parent-inheritance option requested here, so this is not a high-confidence duplicate close.

## What This Changes

The issue requests a configurable default notification policy for subagents, including a cron-friendly parent-inheritance option so silent cron jobs do not emit subagent completion notifications.

## Best Possible Solution

Add a backwards-compatible subagent notification policy resolver that preserves current done_only behavior when unset, supports silent and done_only explicitly, and treats inherit_from_parent as silent for canonical cron parent sessions. Keep schema/types/docs/tests aligned, or intentionally fold this into the broader task-default design in #64081 if maintainers choose that direction.

## Reproduction Assessment

Yes. A static source reproduction is high-confidence: a cron run creates a silent cron task, then a default sessions_spawn subagent creates a pending subagent task without notifyPolicy, which resolves to done_only in the task registry.

## Solution Assessment

Unclear. The requested subagent-level config is plausible and narrow, but maintainers should choose whether this belongs as agents.defaults.subagents.notifyPolicy or as part of the broader task notify policy design tracked in #64081.
