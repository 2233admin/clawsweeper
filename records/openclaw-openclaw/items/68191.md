## Summary

Keep open. Current main has partial durable cron recovery, but Gateway-owned lost reconciliation still falls back to the process-local active-job set, and current startup catch-up/manual-run paths still do not mark jobs active while they execute.

## What This Changes

The issue asks OpenClaw to avoid marking cron task records `lost` based only on transient `activeJobIds` state after a gateway restart, preferably by using durable cron run state or by skipping cron lost-marking without durable failure evidence.

## Best Possible Solution

Keep the issue open until cron task reconciliation either has durable live/terminal cron state for all cron execution paths, or the active-job symmetry gap is closed and terminal stale lost-row cleanup is deliberately handled through #63196.
