## Summary

Keep open. Current main has partial durable cron recovery, but the cron lost-task path still depends on transient in-memory active-job state once the Gateway is authoritative, and related fixes for active-job symmetry plus terminal lost-row recovery are still open rather than shipped.

## What This Changes

The issue asks task-registry maintenance to stop falsely marking cron task rows as `lost` solely because a restart cleared transient `activeJobIds`, using durable cron state or another proof-based reconciliation path instead.

## Best Possible Solution

Keep this open until cron reconciliation consistently uses durable live or terminal cron state, all cron execution paths maintain the active-job ownership contract while running, and existing false-positive terminal `lost` rows can be repaired or explained with useful diagnostics.

## Reproduction Assessment

Yes. The current source and regression test harness still provide a high-confidence reproduction: a stale cron task with no active-job membership is reconciled to `lost`, and the manual run path on main still does not mark the job active while executing.

## Solution Assessment

Unclear for the issue's two suggested alternatives. Durable cron-state reconciliation is the better long-term direction; unconditionally skipping cron lost-marking would remove the false positives but also discard real lost-task detection, so the safer path is coordinated review of #71040 and #75036.
