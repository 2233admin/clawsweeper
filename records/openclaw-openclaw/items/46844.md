## Summary

Keep open. Current main still has only intra-turn Talk silence handling and manual/explicit Talk shutdown; the macOS wake-to-Talk path starts Talk Mode, then resumes listening after each assistant reply without a configurable between-turn idle timeout. The related heartbeat idle issue is adjacent but does not cover this macOS Talk Mode lifecycle.

## Best Possible Solution

Keep this issue open and implement a narrow Talk Mode idle timeout for the macOS wake-to-Talk flow, backed by the gateway Talk config contract, Swift parsing/runtime tests, and user-facing docs. The fix should preserve `silenceTimeoutMs` as the intra-turn pause window and keep the 120s Voice Wake capture hard stop separate.

## Clownfish Work Prompt

Fix the Talk Mode idle timeout requested in openclaw/openclaw#46844. Add a configurable between-turn idle timeout for macOS Voice Wake -> Talk Mode so that, after wake-triggered Talk Mode has no user speech or assistant reply activity for the configured interval, Talk Mode turns off and Voice Wake re-arms. Keep `talk.silenceTimeoutMs` as the intra-turn transcript finalization pause; do not shorten or repurpose the 120s Voice Wake capture hard stop; do not change Push-to-Talk behavior. Prefer the repo's ms-based config naming (`talk.idleTimeoutMs`) unless maintainers have an established reason to use seconds. In `apps/macos/Sources/OpenClaw/TalkModeRuntime.swift`, track an idle timer that resets on user speech/finalized transcript and after assistant playback/resume; when it expires while Talk is enabled, unpaused, listening, and has no pending transcript, call through the existing app state/controller path to disable Talk Mode so overlay teardown and Voice Wake refresh remain normal. Add focused tests for config parsing/schema and a small testable idle-timeout helper or runtime seam. Add a changelog entry because this is user-facing config/behavior, crediting Thanks @reidmcgill.
