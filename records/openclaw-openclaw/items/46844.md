## Summary

Keep open. Current main still lacks the requested between-turn Talk Mode idle timeout: Voice Wake can activate Talk directly, but Talk only uses the short silence window to finalize non-empty transcripts and then resumes listening after replies or failures. The related heartbeat/session-cleanup reports cover broader lifecycle behavior, not this Voice Wake-triggered Talk runtime gap.

## What This Changes

Add a configurable Talk Mode idle timeout that auto-deactivates a Voice Wake-triggered Talk session after a period with no user interaction.

## Best Possible Solution

Add an additive Talk lifecycle setting that is validated in config, included in the gateway Talk protocol/config payload, documented on the Talk docs page, and honored by macOS Talk so idle wake-triggered sessions return to wake listening while silenceTimeoutMs remains scoped to transcript-finalization pauses.

## Reproduction Assessment

Yes. A high-confidence source-level path is to enable Voice Wake with Trigger Talk Mode, wake into Talk, complete a turn, then remain silent; current code ignores empty between-turn silence and resumes recognition after replies or errors.

## Solution Assessment

Unclear as proposed. The direction is narrow and maintainable, but a maintainer should define the public Talk lifecycle contract before implementation chooses naming, defaults, limits, disabled behavior, and reset points.
