## Summary

Keep this open. Current main still has the reported Telegram regression: non-off `replyToMode` adds ordinary inbound text to the native quote candidate map, and the answer-preview stream guard still treats any populated candidate map as incompatible with partial streaming.

## What This Changes

The issue asks for Telegram `streaming.mode: "partial"` to continue working with non-off `replyToMode` for plain replies, while still disabling preview streaming for actual selected Telegram quote replies.

## Best Possible Solution

Land a narrow Telegram dispatcher fix that scopes the answer-preview disable guard to genuine selected quote metadata, preserves final-delivery native quote candidates, and adds regression coverage for `replyToMode: "first"` plus partial streaming on a plain message.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: `replyToMode: "first"`, `streamMode`/`streaming.mode: "partial"`, and a plain non-empty Telegram message currently populate `replyQuoteByMessageId`, set `hasNativeQuoteReply`, and prevent answer draft stream creation.

## Solution Assessment

Yes. The best fix is the narrow dispatcher guard/test change, not broader per-conversation configuration work; the draft-stream path already supports plain `reply_to_message_id` threading.
