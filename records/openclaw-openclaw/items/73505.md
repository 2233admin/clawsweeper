## Summary

Keep open. Current main still contains the reported Telegram regression: plain inbound Telegram text can populate the native quote candidate map, and the dispatcher still treats any populated candidate map as a reason to disable answer draft preview streaming. The closed #73534 remains useful source work, while open #39905 is broader per-conversation configuration work and is not a landed fix for this specific bug.

## What This Changes

The issue asks for Telegram `streaming.mode: "partial"` preview edits to remain active for plain replies when `replyToMode` is set to `first` or another non-off value.

## Best Possible Solution

Land a narrow Telegram plugin fix that separates the preview-streaming incompatibility check from the final-delivery quote candidate map: only selected Telegram quote metadata should suppress answer preview streaming, while plain reply threading continues to create and edit the draft preview.

## Reproduction Assessment

Yes. The current-main source path is high confidence: with `replyToMode: "first"`, `streamMode: "partial"`, and a plain non-empty Telegram message, the dispatcher can add a current-message quote candidate and then disable `createTelegramDraftStream` because the candidate map is non-empty.

## Solution Assessment

Yes. The issue’s suggested direction is the narrowest maintainable fix: base the preview-stream skip on genuine selected quote state such as `replyQuoteText` plus a resolvable `replyQuoteMessageId`, not on the candidate map used by final delivery. Open #39905 is adjacent but broader and should not be required to fix this regression.
