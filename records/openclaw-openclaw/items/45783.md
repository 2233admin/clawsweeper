## Summary

Keep this PR open. Current main has log/transcript redaction and static plugin/provider text-transform surfaces, but it still does not implement the PR's central feature: a native LLM-boundary privacy detector/replacer with encrypted mappings, bidirectional restore, a `privacy` config section, custom JSON5 rules, and coverage across embedded runner, BTW, TTS, cache traces, and logs. The PR is also security-sensitive and needs maintainer architecture/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review or replacement by a smaller architecture-approved implementation. The best path is to define a core LLM-boundary privacy/filtering contract with config/schema/docs, explicit fail-open versus fail-closed behavior, mapping persistence semantics, and tests covering embedded runner, cache traces, BTW, TTS, logs, and provider stream/result shapes. If maintainers decide this belongs outside core, first add a plugin API that can safely modify and restore LLM-boundary traffic across providers rather than relying only on static text transforms or provider-specific wrappers.
