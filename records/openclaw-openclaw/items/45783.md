## Summary

Keep this PR open. Current main still does not implement the central feature: a native LLM-boundary privacy detector/replacer with encrypted mappings, bidirectional restore, `privacy` config, custom JSON5 rules, and coverage across provider calls. Existing main has log/transcript redaction and static plugin text transforms, but those do not solve outbound provider privacy filtering. The PR is also security-sensitive and still needs maintainer architecture/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer architecture and security review, or replace it with a smaller maintainer-approved implementation. The best path is to define a core LLM-boundary privacy contract with config/schema/docs, explicit fail-open versus fail-closed behavior, mapping persistence semantics, custom-rule validation, and tests for embedded runner, cache traces, BTW, TTS, logs, provider stream/result shapes, and OpenAI-style text blocks. If maintainers decide this belongs outside core, first expose a plugin API that supports dynamic per-session detection/replacement/restoration rather than only static text transforms.
