## Summary

Keep open. Current main still does not implement the requested `--profile-id` flag, nor an equivalent operator-chosen OAuth profile alias, for `openclaw models auth login`. The current code only accepts `--profile-id` on `models auth paste-token`; OAuth login still persists provider-returned profile IDs derived from provider identity data. The related open PR #49315 remains the best implementation candidate to review or supersede.

## Best Possible Solution

Keep this issue open as the tracker for explicit OAuth profile aliases. Review and land #49315 or equivalent work that adds `--profile-id <id>` to `models auth login`, carries it through login options and provider auth persistence, validates the requested ID belongs to the selected provider, handles collisions and ambiguous multi-profile results safely, and updates CLI docs plus parser/persistence tests.
