## Summary

Keep open. Current main still has a provider-neutral TTS system prompt, and the speech-provider contract has no provider-owned prompt hint hook. The existing model-provider system-prompt contribution seam targets the active LLM provider/model, not the active speech provider, so it does not solve this TTS-specific discoverability gap.

## What This Changes

Add provider-specific, model-aware TTS prompt hints so the active speech provider can teach agents which expressive controls are valid at runtime.

## Best Possible Solution

Add a backward-compatible optional prompt-hint hook to SpeechProviderPlugin, have speech-core append only the selected speech provider's short hint when TTS is enabled, let the Google provider own Gemini square-bracket cue guidance, and cover active-provider-only plus policy-gated behavior in tests and docs.

## Repair Work Prompt

Implement provider-specific TTS prompt hints for openclaw/openclaw#67539. Add a backward-compatible optional hook to SpeechProviderPlugin for speech-provider-owned runtime prompt guidance. Have speech-core resolve the active speech provider and append only that provider's short hint from buildTtsSystemPromptHint when TTS is enabled. The hook context should include enough resolved state for provider-specific decisions, including providerConfig, resolved model/voice where available, target, and modelOverrides; do not hard-code Gemini guidance in speech-core. Add the Google/Gemini provider hint for square-bracket expressive cues inside spoken TTS text, and make sure hints do not advertise inactive providers or off-policy controls. Update SDK exports/types/docs/contract coverage as needed. Add a changelog entry if maintainers treat this as user-facing runtime behavior; credit Thanks @barronlroth. Do not change unrelated provider behavior, dependency versions, lockfiles, or broad prompt architecture.
