## Summary

Keep open. Current main still has a credible observability gap for isolated cron runs that call `sessions_yield`: the cron service finalizes the cron run from the first agent-run result, while `sessions_yield` intentionally ends that embedded run as an idle/paused turn and the parent can be resumed later by subagent completion. The existing subagent-aware cron code mainly protects delivery text from stale interim output; it does not clearly make the cron job/run log remain running until the resumed parent session finishes.

## Best Possible Solution

Add an explicit cron orchestration lifecycle for yielded isolated runs: when `runCronIsolatedAgentTurn` returns `meta.yielded` with active descendant/follow-up work, keep the cron task/run record in a non-terminal running or paused state until the parent session’s resumed turn produces a final result, or record a separate total orchestration duration that includes the resumed parent completion. Cover it with an isolated cron regression test for `sessions_spawn` plus `sessions_yield` where the child sleeps/completes and the parent writes a final post-yield result; assert `lastStatus`, `lastDurationMs`, run log timing, and task status reflect the full workflow.
