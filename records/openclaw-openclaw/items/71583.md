## Summary

Keep open. Current main still matches the reported QQ Bot failure mode: token acquisition posts to `bots.qq.com` without a plugin-defined timeout or probe-specific diagnostic, the gateway connection waits for that token before opening the QQ WebSocket, and the QQ Bot docs/status surface still does not isolate token-endpoint reachability. The issue also includes concrete A/B uplink and core-health evidence, so this is actionable rather than stale support noise.

## What This Changes

The issue requests OpenClaw-specific diagnostics or workarounds for QQ Bot access-token fetches to `bots.qq.com` timing out while the core gateway remains healthy.

## Best Possible Solution

Keep the repair in the bundled QQ Bot plugin: bound token acquisition with an explicit timeout, return structured timeout/network metadata, expose a QQ token-endpoint account probe through the existing status adapter seam, and document `bots.qq.com` troubleshooting on the public QQ Bot docs.

## Reproduction Assessment

Yes. A high-confidence reproduction path is to make `https://bots.qq.com/app/getAppAccessToken` hang or time out and start a configured QQ Bot account; current `connect()` awaits token acquisition before opening the QQ gateway WebSocket while core gateway health is independent.

## Solution Assessment

Yes. A narrow QQ Bot plugin reliability/docs fix is the best route; core changes should be limited to a generic seam only if the existing `probeAccount` and fetch-helper patterns prove insufficient.
