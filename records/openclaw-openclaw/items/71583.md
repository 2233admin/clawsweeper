## Summary

Keep open. Current main still leaves the QQBot token stage under-instrumented: token acquisition is a bare fetch, gateway startup waits there before WebSocket connection, and channel probing has no QQBot token-reachability check, so the reported egress timeout is not surfaced well enough even if the external network failure is outside OpenClaw.

## What This Changes

The issue asks for OpenClaw-specific diagnostics or a workaround when QQBot access-token fetches to bots.qq.com time out while the core gateway and dashboard remain healthy.

## Best Possible Solution

Keep the repair inside the QQBot plugin: add a bounded token-fetch timeout, surface token-stage failures through runtime status/probe output, and document `bots.qq.com:443` egress checks without adding QQ-specific policy to core gateway code.

## Reproduction Assessment

Yes for the OpenClaw failure mode: current source shows token startup depends on an un-aborted fetch before WebSocket connection, and a mocked hanging fetch would reproduce the pending token stage. The exact external network timeout is environment-specific and did not reproduce from this runner.

## Solution Assessment

Yes. A QQBot-owned timeout, status probe, and troubleshooting update is the narrowest maintainable path; a core gateway special case for Tencent egress would be the wrong boundary.
