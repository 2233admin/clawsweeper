## Summary

Keep open. Current main still does not implement the central OpenResponses `input_file` request for Excel or generic binary uploads: Excel MIME types are not in the default file allowlist, unsupported MIME types are rejected before materialization, and accepted files are decoded into prompt context rather than persisted as durable workspace artifacts with metadata or stable tool-facing receipts. Recent WebChat/chat.send work partially supports non-image attachments, including `.xlsx`, but that is a separate attachment path and does not satisfy the gateway `input_file` contract requested here.

## Best Possible Solution

Keep this issue open as the canonical OpenResponses/Gateway file-ingest request. The best implementation is a documented `input_file` contract that accepts explicitly allowlisted Excel and generic binary MIME types, fails closed with clear errors, preserves existing URL-fetch and size controls, materializes accepted binaries into managed workspace/media storage, records metadata such as filename, MIME, sha256, and size, and exposes stable receipts or paths to tools and skills. The existing WebChat non-image attachment path should either remain chat-specific or be generalized through a deliberate shared media-ingest seam.
