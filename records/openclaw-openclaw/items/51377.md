## Summary

Keep open. Current main still does not implement the central Gateway/OpenResponses `input_file` request for Excel or generic binary uploads: the default file MIME allowlist excludes `.xls`/`.xlsx`, unsupported MIME types still reject before any durable materialization, and accepted files are decoded into ephemeral prompt context rather than saved as workspace artifacts with metadata or tool-facing receipts. The merged chat/WebChat attachment work is useful and adjacent, but it is a separate `chat.send` path, not the `input_file` contract requested here.

## Best Possible Solution

Keep this issue open as the canonical Gateway/OpenResponses file-ingest request. The best path is a documented `input_file` contract that explicitly allowlists Excel and selected binary MIME types, fails closed with clear errors, preserves existing URL-fetch and size controls, materializes accepted binaries into managed workspace or media storage, records filename/MIME/sha256/size metadata, and exposes stable receipts or paths to tools and skills. The existing `chat.send` non-image attachment path should either remain chat-specific or be generalized through a deliberate shared media-ingest seam after the API contract is settled.
