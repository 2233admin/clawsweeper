## Summary

Keep open. Current main still gates alias lookup behind `!model.includes("/")`, while isolated cron `payload.model` flows through that resolver via `resolveAllowedModelRef`. A slash-form configured alias such as `xiaomi/mimo-v2-pro-mit` will still be parsed as provider `xiaomi` and model `mimo-v2-pro-mit` instead of resolving to the configured canonical ref.

## Best Possible Solution

Keep this issue open and fix `resolveModelRefFromString()` so it checks the configured alias index before provider/model parsing for all inputs, then add focused regression coverage for both direct resolver calls and `resolveAllowedModelRef()` with a slash-form alias used by cron `payload.model`.
