## Summary

Keep this PR open. Current main still does not implement the proposed `directReply`/`direct_reply` tool-result contract, does not expose an `abortRun` callback to embedded tool handlers, and does not have direct-reply helpers in `src/agents/tools/common.ts`. The PR remains a live implementation candidate, but it needs maintainer product/security review because it lets successful tool output bypass final model synthesis and send text/media directly to users.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants this capability, land it as a generic documented tool-result contract with a typed direct-reply marker, explicit trust policy, typed abort reason, media filtering through the existing block-reply pipeline, and an embedded-run integration test proving the direct reply is delivered once while the follow-up LLM inference is skipped. If maintainers prefer ACP/subagent/external-harness delivery instead, close only after that product decision is explicit.

## Risks / Open Questions

- Security/trust boundary: the proposed marker would allow any successful tool result that can set `directReply: true` to bypass the final model synthesis path and send text/media directly to the user. Maintainers should decide whether that is allowed for all tools, only trusted tools, or only through an explicit documented capability.
- Media handling needs maintainer review: the PR appears to reuse the existing tool-result media filtering path, which is good, but direct block delivery should still be checked against trusted-local-media and external/MCP media restrictions before merge.
- Contract durability: current main aliases `OpenClawAgentToolResult` to the upstream Pi tool-result type and validates middleware results by `content`/`details` shape only, so a stable direct-reply marker should be typed and documented rather than treated as an incidental extra property.
- PR hygiene: the provided PR file context still shows a changelog conflict marker and a forbidden `Thanks @codex` attribution, so the branch should be cleaned before merge consideration.
- Security review pass: the provided diff does not add CI workflows, third-party dependencies, lockfile changes, install/build/release scripts, or package publishing metadata, so the main security concern is the runtime trust boundary introduced by the new direct-delivery behavior.
