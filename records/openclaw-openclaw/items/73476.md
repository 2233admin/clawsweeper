## Summary

Keep this PR open. The feature direction is useful and current main does not already implement it, but the latest diff still has blocking delivery-policy issues: direct replies can bypass silent/tool-only visible-reply rules, and media direct replies can be emitted once by the new branch and again by the existing pending-tool-media flush. The branch also mixes unrelated workflow and test maintenance into the feature PR, so it needs author fixes plus maintainer product/security review before merge.

## What This Changes

The branch adds directReply tool-result helpers and embedded-runner wiring to emit successful tool output directly and treat direct_reply aborts as successful, while also updating changelog, a parity-gate workflow env, and several unrelated tests.

## Best Possible Solution

Land a narrow, typed direct-delivery contract only after it preserves existing silent/tool-only visible-reply behavior, avoids double delivery through the pending media queue, defines which tool/plugin surfaces may use direct replies, documents the helper/API surface, and splits unrelated CI/test maintenance out of the feature branch.

## Reproduction Assessment

Yes. A focused code-level reproduction is to return a successful `{ directReply: true }` tool result with text/media under a run where `silentExpected` is true and `onBlockReply` is set; the PR branch emits visibly while current message/block paths would suppress it. A media-only directReply also exercises the duplicate path because existing tool handling queues the media before the new branch emits it directly.

## Solution Assessment

No. The overall direct-reply concept may be the right core seam, but this implementation is not the safest merge path until it threads visibility policy into `ToolHandlerContext`, chooses one media-delivery path, and removes unrelated workflow/test churn.
