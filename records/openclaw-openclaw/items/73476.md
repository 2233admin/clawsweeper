## Summary

Current main still does not implement this PR’s direct tool-result reply behavior: there is no directReply/direct_reply contract, no helper constructors, no subscription abort seam, and the embedded tool-end handler still emits normal tool output and returns after after_tool_call hooks. The PR is a live implementation candidate; the author’s scenario makes the need concrete, but the remaining decision is maintainer product/security review of whether trusted successful tool output may bypass final LLM synthesis.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants this capability, land it as a generic typed tool-result contract with an explicit trust policy, safe media filtering through the existing block-reply path, a typed abort reason, focused embedded-run tests proving direct delivery without a follow-up model pass, and docs/changelog cleanup. If maintainers prefer a different product direction, close the PR only after that decision is explicit.
