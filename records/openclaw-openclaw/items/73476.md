## Summary

Keep open. The contributor’s external-agent use case is coherent, but the current PR still routes direct replies through the existing block-reply callback and then aborts the run, even though that callback is not a guaranteed user-delivery path. In default non-streaming text-only turns, and in suppressed source-delivery modes, this can produce a successful run with no visible reply.

## What This Changes

The PR adds an optional `directReply` tool-result flag and helper constructors, wires an abort callback through embedded agent subscription, treats `direct_reply` aborts as successful terminal runs, adds tests/changelog coverage, and includes QA/parity/test stabilization changes.

## Best Possible Solution

Land a narrow direct-reply contract only after delivery policy is explicit: direct replies should use a delivery path that returns allowed/sent/suppressed status, abort only after allowed successful delivery, fall back to normal inference or message-tool guidance when delivery is suppressed, and update the public SDK baseline/docs if the helper is meant for plugin authors.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a default text-only `directReplyResult("done")` turn with block streaming disabled: current main’s block-reply handler does not send text-only block replies in that mode, while the PR aborts the run before final delivery can carry the text.

## Solution Assessment

No. The direct-reply seam may be the right direction for external-agent tools, but this implementation aborts based on an attempted block emit rather than policy-aware delivery success; a safer solution should thread delivery status or route through final/message-tool delivery semantics.

## Review Findings

Full review comments:

- **[P1] Abort only after direct reply delivery succeeds:** `src/agents/pi-embedded-subscribe.handlers.tools.ts:1200-1205`
  - body: `emitBlockReply` is not a guaranteed send path: with block streaming disabled, text-only block replies are intentionally left for final delivery, and source-delivery suppression can drop them entirely. Since this branch aborts immediately afterward, a normal text-only direct reply can complete with no visible user message. Thread delivery status into this path or use a direct/final delivery callback that can report suppression before aborting.
  - confidence: 0.9

  - body: `agent-runtime` publicly re-exports `../agents/tools/common.js`, so adding exported direct-reply helpers changes the plugin SDK surface. Either keep these helpers internal or regenerate and commit the plugin SDK API baseline hash so `pnpm plugin-sdk:api:check` does not report drift.
  - confidence: 0.82
