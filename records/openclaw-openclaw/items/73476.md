## Summary

Current main still does not implement this PR’s intended direct tool-result reply behavior, and the open PR remains a focused implementation candidate rather than cleanup material. The existing embedded Pi path emits normal tool output/media and then continues through the agent loop; there is no `directReply`/`direct_reply` contract, no helper constructors, and no subscription seam that lets a tool-completion handler emit a final block reply and abort the active run. The supplied discussion also supports keeping it open: Greptile’s P2 review comments appear addressed by later PR commits, leaving a real maintainer product/security decision about whether tool output may bypass final LLM synthesis.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants the capability, land it as a generic typed tool-result contract with an explicit trust policy, safe media filtering through the existing block-reply path, focused embedded-run tests proving one direct user reply and no follow-up model pass, and docs/changelog polish. If maintainers reject the product direction, close it only with that explicit decision.

## Risks / Open Questions

- Security/product review is still needed because `directReply: true` would let successful tool output bypass final LLM synthesis and deliver text/media directly to the user.
- The contract should be explicit and durable across tool-result serialization and middleware boundaries, with typed abort reasons rather than an incidental marker only understood by one handler.
- Media delivery must continue using the existing trusted local-media and remote-media filtering rules, with regression tests proving direct delivery does not widen local file exposure.
- The PR diff does not show dependency, CI, workflow, install, release, or package-resolution changes; the observable security-sensitive surface is the runtime trust boundary created by direct replies.
