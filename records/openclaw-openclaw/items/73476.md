## Summary

Current main still does not implement this PR's intended direct tool-result reply behavior. The author clarified the target use case: a simplified skill calls a separate sandbox/API that returns final high-value content, and OpenClaw should not summarize that content again. That is a plausible product need, but the PR changes core agent-loop delivery and abort semantics, so it should stay open for explicit maintainer product/security review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer product/security review. If OpenClaw wants this capability, land it as a generic typed tool-result delivery contract, preferably aligned with upstream AgentToolResult.terminate semantics where possible, with an explicit trust policy, preserved sanitized text/media filtering, a typed abort reason, integration tests proving direct delivery without a follow-up model pass, and docs/changelog coverage. Split or justify the unrelated gateway/CI assertion changes before merge.

## Risks / Open Questions

- Directly delivering external tool output bypasses the normal final model synthesis path; maintainers need an explicit trust/redaction policy for which tools may set the flag and how that interacts with plugin hooks.
- The PR aborts the active embedded run from a tool_execution_end subscriber; in parallel or multi-tool batches this needs review against upstream terminate semantics and active tool cancellation behavior.
- The PR includes unrelated gateway/CI test assertion edits alongside the agent-loop feature, which should be split out or justified before merge.
- The contract should be typed and documented; a top-level directReply marker that is not part of upstream AgentToolResult or a documented OpenClaw wrapper is easy for plugin authors to misuse.
