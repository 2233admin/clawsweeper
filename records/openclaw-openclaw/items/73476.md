## Summary

Keep this PR open. The checked-out main at dc541662f81503aecc08c193bf9ddf3208a95abb still does not implement a directReply/direct_reply tool-result contract, does not forward abortRun into embedded subscription handlers, and does not expose emitBlockReply/abortRun to ToolHandlerContext. The PR remains the live implementation candidate, but it needs maintainer product/security review because it lets tool output bypass the final model synthesis path and deliver text/media directly to users.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants this capability, land it as a generic documented tool-result contract with a typed direct-reply marker, explicit trust policy, typed abort reason, media filtering through the existing block-reply pipeline, and an embedded-run integration test proving the direct reply is delivered once while the follow-up LLM inference is skipped. If maintainers prefer ACP/subagent/external-harness delivery instead, close only after that product decision is explicit.

## Risks / Open Questions

- Security/trust boundary: any successful tool able to set the proposed directReply marker could bypass final LLM synthesis and send text/media directly to the user, so maintainers should decide whether this is allowed for all tools, only trusted tools, or a documented capability with policy checks.

- Contract durability: directReply is not part of the current OpenClawAgentToolResult/AgentToolResult contract, so the final design should define a stable typed marker and behavior through middleware/serialization paths.
