## Summary

Keep this PR open. Current main does not implement the requested direct tool-result reply path, but the latest PR branch still has a blocking delivery-policy bug: it can abort the agent turn after a direct-reply callback that current delivery wrappers intentionally suppress, leaving no visible response.

## What This Changes

The PR adds a `directReply` tool-result flag, helper constructors, embedded subscription and runner abort wiring, tests, a changelog entry, QA mock/test stabilizers, and a parity-gate workflow Node 24 runtime env setting.

## Best Possible Solution

Land a narrow direct-reply contract only after it is policy-aware: direct replies should be delivered only when visible delivery is allowed or otherwise fall back to normal assistant output, the no-callback path should preserve returned text/media, plugin-facing typing/docs should be explicit if this is public API, and unrelated CI/QA changes should be split out.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a successful tool result with `{ directReply: true, content: [{ type: "text", text: "done" }] }` during a `message_tool_only` or silent/source turn, or any run without `onBlockReply`; the PR branch can abort after a callback that current main either suppresses or does not have.

## Solution Assessment

No. The direct-reply seam is plausible, but this implementation is not the best merge path until delivery success or fallback is threaded through the abort decision and the unrelated workflow/test churn is separated or explicitly approved.
