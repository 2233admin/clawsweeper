## Summary

Keep this PR open. Current main does not implement the requested agent tool-result direct-reply contract, and the PR is a plausible core implementation candidate, but the latest diff still needs maintainer and author work because it aborts after a direct-reply attempt without proving delivery is allowed/sent and it bundles unrelated workflow and test maintenance into the feature branch.

## What This Changes

The PR adds `directReply` tool-result helpers plus embedded runner/subscription wiring to emit tool output directly and abort the model turn, while also changing the changelog, parity-gate workflow env, qa-lab mock behavior, and several unrelated test assertions.

## Best Possible Solution

Land a narrow typed direct-reply contract only after it honors source reply visibility and silent-turn policy, aborts only when direct delivery is allowed or known to have happened, documents the helper/API surface, and splits unrelated CI/test stabilization out of the feature branch.

## Reproduction Assessment

Yes. A focused code-level reproduction is a successful tool result with `{ directReply: true, content: [{ type: "text", text: "done" }] }` in a run whose source delivery mode is `message_tool_only` or whose run is `silentExpected`; current delivery wrappers suppress that block reply while the PR branch still aborts further inference.

## Solution Assessment

No. The direct-reply concept may be the right core seam for external-agent tools, but this implementation is not the safest merge path until it threads visibility/delivery policy into the tool handler and removes unrelated workflow/test churn.
