## Summary

Current main still does not implement this PR’s intended direct tool-result reply behavior. The embedded Pi tool-end path emits normal tool output and after_tool_call events, then lets the agent loop continue; there is no directReply/direct_reply tool-result contract, no direct-reply helpers, and no abortRun seam forwarded into the tool-completion handler. The PR remains the active implementation candidate, with the main remaining question being maintainer product/security review of whether successful tool output may bypass final LLM synthesis.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants the capability, land it as a generic, typed tool-result contract with explicit trust policy, typed abort reason, media filtering through the existing safe block-reply path, focused embedded-run tests proving direct delivery without a follow-up model pass, and docs/changelog cleanup. If maintainers prefer a different product direction, close only after that decision is explicit.

## Risks / Open Questions

- Direct replies would let successful tool output bypass final LLM synthesis and deliver text/media directly to the user; maintainers should decide whether this is allowed for all tools, trusted tools only, or an explicit declared capability.
- The directReply marker should be a durable typed contract across Pi/OpenClaw tool-result shapes and middleware or serialization boundaries, not an incidental property understood by one handler.
- Media delivery must keep using existing filterToolResultMediaUrls/local-media trust checks so external plugin or MCP tool results cannot leak local paths.
- Abort-after-direct-reply needs lifecycle coverage proving one visible user reply, no follow-up model inference, and no lost cleanup/hook behavior.
