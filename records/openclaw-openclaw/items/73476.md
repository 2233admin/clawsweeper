## Summary

Current main still does not implement this PR's intended direct tool-result reply behavior. The embedded Pi tool-end path emits normal tool result output and runs after_tool_call hooks, but there is no directReply/direct_reply contract, no direct-reply helper constructors, no abortRun subscription seam, and ToolHandlerContext cannot emit a block reply or abort the run after a successful tool result. The PR remains a live implementation candidate; the remaining decision is product/security review of whether successful tool output may bypass final LLM synthesis.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants this capability, land it as a generic typed tool-result contract with an explicit trust policy, safe media filtering through the existing block-reply path, a typed abort reason, focused embedded-run tests proving direct delivery without a follow-up model pass, and docs/changelog cleanup. If maintainers prefer a different product direction, close the PR only after that decision is explicit.
