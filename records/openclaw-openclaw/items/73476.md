## Summary

Keep this PR open. Current main still has no agent/tool `directReply` contract, and the contributor’s external-agent use case is coherent, but the latest PR diff still has a policy bug: it aborts the model turn after requesting a block reply without knowing whether that reply is allowed to be delivered. The author fixed the earlier empty-content and helper typing feedback, so the remaining blocker is focused on delivery policy and maintainer trust-model review.

## What This Changes

The PR adds a `directReply` tool-result flag and helper constructors, wires an `abortRun` callback through the embedded subscription/runner path, emits direct text/media block replies from successful tool results, adds tests and a changelog entry, and includes unrelated qa-lab/test/workflow stabilization changes.

## Best Possible Solution

Land a narrow, typed direct-reply contract only after it respects source-reply visibility and silent-turn policy, aborts only when direct delivery is allowed or confirmed, adds focused regression coverage for suppressed-delivery cases, documents the plugin-facing helper/API if this is public surface, and splits unrelated CI/qa-lab stabilization into separate owner-reviewed changes.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a successful tool result like `{ directReply: true, content: [{ type: "text", text: "done" }] }` during a group/channel source turn using `message_tool_only` or a silent turn; current delivery wrappers can drop the block reply while the PR branch still aborts further inference.

## Solution Assessment

No. The direct-reply seam may be the right direction for external-agent tools, but this implementation is not the safest merge path until it threads delivery policy or delivery success into the abort decision and separates unrelated workflow/test changes.
