## Summary

Keep open. Current main still resolves a mixed MCP server config with both `command` and non-empty `url` as stdio before evaluating `transport: "streamable-http"`, so the reported timeout remains plausible; the linked open fix PR #72515 is the active implementation path and should be reviewed before this issue closes.

## What This Changes

The issue reports that an MCP server configured with `command`, `url`, and `transport: "streamable-http"` starts and listens on its HTTP port, but OpenClaw times out during MCP connection setup.

## Best Possible Solution

The desired end state is an explicit MCP transport contract: reject ambiguous mixed stdio/HTTP server definitions with a clear diagnostic and tests if managed HTTP child processes are unsupported, or intentionally add a managed streamable-http process mode with documented lifecycle behavior.

## Reproduction Assessment

Yes. A config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` follows the stdio branch on current main; if the spawned process only serves HTTP and never speaks MCP over stdout, the runtime can hit the existing 30-second connection timeout.

## Solution Assessment

Unclear until maintainers choose the intended product boundary. The linked rejection approach is the narrowest maintainable fix for the documented contract, while supporting spawned HTTP servers would be a larger feature.
