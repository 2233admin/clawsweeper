## Summary

Keep open. Current main still prefers the stdio resolver for a mixed MCP server config with both `command` and `url`, which makes the reported 30-second timeout path plausible; an open implementation PR, #72515, already references this issue with closing syntax and should be reviewed before this issue is closed.

## What This Changes

The issue asks OpenClaw to handle an MCP `streamable-http` server configured with both a spawned command and HTTP URL without timing out during connection startup.

## Best Possible Solution

The desired end state is an explicit MCP transport contract: either reject ambiguous mixed stdio/HTTP definitions early with a clear diagnostic and tests, or add a deliberate managed streamable-http subprocess mode with readiness, connection timeout, and cleanup behavior covered in code and docs.

## Reproduction Assessment

Yes. A config with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` follows the stdio branch on current main; if the spawned process only serves HTTP and never speaks JSON-RPC over stdout, the existing 30-second connection timeout can be reached.

## Solution Assessment

Unclear until maintainers decide the intended product boundary. Early rejection, as proposed in #72515, is the narrowest maintainable fix if managed HTTP child processes are unsupported; adding process-managed HTTP transport would be a larger feature.
