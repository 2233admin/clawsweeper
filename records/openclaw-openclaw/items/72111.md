## Summary

Keep open: current main still resolves the reported mixed `command` + `url` streamable-http config as stdio before HTTP handling, so the timeout path remains plausible; #72515 is an open unmerged closing PR and should be reviewed before this issue is closed.

## What This Changes

The issue reports that an MCP server entry with a spawned command, `transport: "streamable-http"`, and a URL starts listening but OpenClaw times out while connecting.

## Best Possible Solution

Land or revise #72515 or an equivalent fix so mixed `command` + non-empty `url` MCP entries are rejected early with a clear diagnostic and regression tests, unless maintainers explicitly decide to add managed HTTP child-process support as a separate feature.

## Reproduction Assessment

Yes. A config object with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` follows the stdio branch on current main; if the spawned process only serves HTTP and never speaks JSON-RPC over stdout, the existing 30-second `client.connect()` timeout is the expected failure path.

## Solution Assessment

Unclear until maintainers choose the intended boundary. Given the current docs already say mixed `command` and `url` entries are rejected, early rejection with tests is the narrowest maintainable fix; managed streamable-http spawning would be larger feature work.
