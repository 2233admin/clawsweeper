## Summary

Keep open. Current main still resolves a non-empty `command` plus `url` MCP server as stdio before it reaches the streamable-http branch, so the reported timeout path remains plausible; open PR #72515 already carries the implementation candidate and closing linkage.

## What This Changes

The issue asks OpenClaw to handle a spawned MCP server configured with `transport: "streamable-http"` and `url` without falling through to a 30-second stdio connection timeout.

## Best Possible Solution

Review and either land PR #72515 as the narrow resolver/docs-aligned fix, or replace it with an explicit managed streamable-http child-process design that covers lifecycle, readiness, timeout behavior, and regression tests.

## Reproduction Assessment

Yes. A config object with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` still resolves to stdio on current main; an HTTP-only server process would then fail to complete JSON-RPC over stdout and can hit the existing 30-second connect timeout.

## Solution Assessment

Unclear until maintainers choose the boundary. Early rejection of mixed configs is the narrowest maintainable fix under the current docs; managed HTTP child-process support would be a larger feature.
