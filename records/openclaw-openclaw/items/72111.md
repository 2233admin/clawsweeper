## Summary

Keep open. Current main still resolves a mixed MCP server config with both `command` and `url` as stdio before the streamable-http branch can run, so the reported timeout remains plausible; open PR #72515 is already linked with closing syntax and should be reviewed before this issue is closed.

## What This Changes

The issue reports that an MCP server configured with `command`, `url`, and `transport: "streamable-http"` starts listening over HTTP but OpenClaw times out while connecting.

## Best Possible Solution

The desired end state is an explicit MCP transport contract: reject ambiguous mixed stdio/HTTP server definitions with a clear diagnostic and tests, or deliberately add managed streamable-http child-process support as a separate feature with lifecycle coverage.

## Reproduction Assessment

Yes. A config object with non-empty `command`, non-empty `url`, and `transport: "streamable-http"` follows the stdio branch on current main; if the spawned process only serves HTTP and never speaks JSON-RPC over stdout, the runtime can hit the existing 30-second connection timeout.

## Solution Assessment

Unclear until maintainers choose the intended boundary. Early rejection, as proposed in PR #72515, is the narrowest maintainable fix if managed HTTP child processes are unsupported; full managed HTTP spawning would be larger feature work.
