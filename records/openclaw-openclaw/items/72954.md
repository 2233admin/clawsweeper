## Summary

Keep open. Current main still contains the reported chain: isolated cron announce awareness is mirrored into the main session as `trusted: false`, heartbeat turns convert pending untrusted events into `ForceSenderIsOwnerFalse`, command auth hard-overrides owner identity, and owner-only tools are filtered from the next run. The bug is valid, but the fix changes trust and owner-only authorization semantics and overlaps the open #72201 audience/internal-context proposal.

## What This Changes

The issue reports that isolated cron `announce` delivery queues untrusted awareness into the owner DM session, causing the next verified owner DM to lose owner-only tools such as `cron`, `gateway`, and `nodes`.

## Best Possible Solution

Add a narrow provenance- or audience-aware path for isolated cron announce awareness so output from an owner-created cron job can inform the next verified owner turn without exposing owner-only tools to unrelated untrusted event content. The #72201 internal-audience design is a plausible maintainable endpoint if maintainers agree with that contract.

## Reproduction Assessment

Yes. The issue includes a concrete live trace from `openclaw@2026.4.23`, and current main has a source-level reproduction path through untrusted cron awareness queueing, heartbeat owner downgrade, command-auth override, and owner-only tool filtering.

## Solution Assessment

Unclear. Trusting owner-created cron output is plausible for this specific report, but the safer maintainable fix is likely narrower than changing general owner resolution, such as the event-audience/internal-context approach in #72201 or an equivalent provenance check scoped to cron awareness.
