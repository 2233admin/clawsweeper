## Summary

Keep open. Current main still contains the reported cron-awareness and owner-downgrade chain, and the remaining fix is security-sensitive because it must preserve owner-only tool downgrades for genuinely untrusted system events.

## What This Changes

The issue asks OpenClaw to stop isolated cron `announce` delivery awareness from making the next verified owner DM lose owner-only tools such as `cron`, `gateway`, and `nodes`.

## Best Possible Solution

Add a narrow provenance- or audience-aware path for isolated cron announce awareness so output from an owner-created cron job can inform the next verified owner turn without stripping owner-only tools, while externally influenced hook, webhook, node, and exec system events continue to force non-owner tool policy.

## Reproduction Assessment

Yes. The issue includes a concrete live reproduction on `openclaw@2026.4.23`, and current main has a source-level reproduction path through cron awareness queueing, heartbeat untrusted-event detection, owner downgrade, and final owner-only tool filtering.

## Solution Assessment

Unclear. The report's cron-authorship fix is plausible, but the safer maintainable path is narrower than changing general owner resolution: distinguish owner-created cron awareness from other untrusted event sources, likely through provenance or the event-audience design in #72201.
