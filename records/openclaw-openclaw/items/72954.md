## Summary

Keep open. Current main still contains the cron awareness trust flag, heartbeat owner downgrade, command-auth hard override, and owner-only tool filtering path described in the report; the related merged PRs explain how the interaction arose but do not supersede or fix it.

## What This Changes

Owner-created isolated cron `agentTurn` jobs with `announce` delivery should not cause the next verified owner DM to lose owner-only tools.

## Best Possible Solution

Add a targeted cron-awareness trust model that lets owner-created isolated cron announce results avoid downgrading the next verified owner turn while preserving the existing downgrade for external hooks, exec/node events, and other untrusted prompt text. The implementation should include regression coverage at cron delivery, heartbeat context construction, command authorization, and final tool inventory filtering.

## Reproduction Assessment

Yes. The current-main source path is deterministic: an isolated cron announce queues an untrusted awareness event, heartbeat converts it to `ForceSenderIsOwnerFalse`, command auth hard-overrides owner status, and owner-only tool policy removes `cron`, `gateway`, and `nodes`. This can be reproduced with focused seam tests without live Discord or Telegram.

## Solution Assessment

Unclear. The report's first suggested fix is narrow and plausible, but `trusted: false` for cron awareness was deliberately added as security hardening, so the best fix needs maintainer trust-model review before changing cron awareness or owner-downgrade semantics.
