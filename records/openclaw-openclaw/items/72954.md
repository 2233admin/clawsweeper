## Summary

Keep this open. Current main still contains the reported cron-awareness-to-owner-downgrade path, and existing tests lock in the untrusted event behavior that triggers it; the remaining fix changes trust and owner-only tool semantics, so it needs maintainer security/product review before an automated repair changes the policy.

## What This Changes

The issue asks OpenClaw to prevent cron-fired isolated announce delivery from stripping owner-only tools on the next verified owner DM when the queued awareness event came from an owner-created cron job.

## Best Possible Solution

Add a provenance-aware trust or authorization policy for isolated cron announce awareness so owner-created cron output does not suppress owner-only tools on the next verified owner turn, while untrusted hook/webhook/exec protections continue to downgrade owner access. Cover both the direct reply drain path and heartbeat pending-event path with focused regressions.

## Reproduction Assessment

Yes. Current main has a deterministic reproduction path without live Discord: queue an isolated cron announce awareness event with `trusted: false`, process the next owner turn through the direct reply or heartbeat path, and observe `senderIsOwner=false` followed by owner-only tool filtering.

## Solution Assessment

Unclear. The report's first suggested fix is narrow, but prior merged hardening deliberately made these awareness events untrusted, so the safest solution is a targeted owner-created-cron exemption or separate trust signal rather than broadly ignoring `ForceSenderIsOwnerFalse` or marking all cron awareness trusted.

- Both the direct inbound reply path and heartbeat path can turn untrusted queued events into non-owner tool inventories, so a fix that covers only one path may leave the reported behavior partially unresolved.
