## Summary

Keep open. Current main covers exact-session resume through lower-level `sessionId` and `sessions.resolve` primitives, and it has parent-key based session creation/forking internals, but it still does not expose the requested public SDK API for `forkSession(id)` or `continue: true`. The issue is recent, has supportive product signal, and is not a stale/duplicate/implemented cleanup candidate.

## What This Changes

Request first-class session APIs comparable to open-agent-sdk: fork an existing session by ID, resume by session ID, and continue the most recent session without knowing its ID.

## Best Possible Solution

Define an OpenClaw-native App SDK session contract that reuses the existing `sessionId`, `sessionKey`, and `parentSessionKey` primitives without leaking internal keys: add a safe fork helper, keep explicit resume supported, and add a most-recent-session helper only after selection and authorization rules are settled and documented.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The source/doc check is high-confidence: current main has partial lower-level primitives but no public `forkSession(id)`, `sessions.fork(...)`, or `continue: true` SDK surface.

## Solution Assessment

Unclear whether copying the open-agent-sdk shape directly is the best OpenClaw API. The safer direction is a small SDK design pass that maps the requested workflows onto OpenClaw’s existing session model, then implements the narrow public helpers with Gateway tests.
