## Summary

Keep this issue open. Current main wires `channels.telegram.apiRoot` through the Telegram client, but the reported migration failure still has real remaining work: persisted polling offsets are scoped only by account and bot token, while the polling cleanup path still treats Telegram 400 `Logged out` from `deleteWebhook` as a fatal non-recoverable setup error.

## What This Changes

The issue asks Telegram polling to support self-hosted Bot API migrations by dispatching local-server updates, handling endpoint-specific update offsets, and avoiding crash loops during Bot API `logOut` cooldown.

## Best Possible Solution

Implement the fix inside the Telegram plugin: persist a normalized `apiRoot` identity with update-offset state, treat mismatched or legacy endpoint records as stale during polling startup, handle the local Bot API `logOut` cooldown response narrowly, and document the migration path for self-hosted Bot API users.
