## Summary

Keep this PR open. Current main still lacks the proposed cache-only Chrome MCP health probe, and the passive browser status/profile/tabs paths can still reach Chrome MCP session creation when the in-memory cache is empty. The PR remains a focused implementation candidate rather than obsolete cleanup.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate. The best fix is a cache-only Chrome MCP health probe wired into `GET /`, profile listing, and `GET /tabs`, with regression tests proving those passive paths do not call the session factory or spawn `chrome-devtools-mcp` when the in-memory cache is empty, while explicit start/attach paths continue to work.

## Risks / Open Questions

- The security-sensitive surface is the subprocess boundary. The provided diff appears to reduce unintended process spawning and does not include unexplained supply-chain or CI changes, but maintainers should still verify explicit `POST /start` attach behavior remains unchanged.
