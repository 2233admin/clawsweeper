## Summary

Current main still reports Claude CLI workspace/project-dir health against the default agent once any Claude CLI signal exists, even though canonical Claude CLI setup can be per-agent via `agentRuntime.id`. The issue is valid and narrow enough for a focused fix PR.

## Best Possible Solution

Keep this issue open and fix the Claude CLI doctor check to be runtime-policy aware: detect agents whose effective runtime is `claude-cli`, skip the section when no agent uses Claude CLI and there are no relevant legacy/default signals, and report workspace/project-dir health for the relevant agent(s) instead of always using the default agent. Add focused tests for a default Codex/OpenAI agent plus a non-default Claude CLI agent, and preserve existing legacy defaults-model behavior.

## Clownfish Work Prompt

Fix `openclaw doctor` so the Claude CLI section is scoped to agents that actually use the Claude CLI runtime instead of always reporting the default agent workspace. Current main computes the Claude CLI workspace with `resolveAgentWorkspaceDir(cfg, resolveDefaultAgentId(cfg))` in `src/commands/doctor-claude-cli.ts`; update it to detect effective Claude CLI usage from `agents.defaults.agentRuntime`, `agents.list[].agentRuntime`, legacy `claude-cli/*` model refs, defaults-level `cliBackends.claude-cli`, auth/profile signals, and `OPENCLAW_AGENT_RUNTIME` where appropriate. Prefer a combined `Claude CLI` note that lists each relevant agent id with its workspace and Claude project dir; skip the workspace/project-dir subsection when only global auth/binary signals exist and no configured agent uses Claude CLI. Preserve existing legacy defaults-model behavior and headless/no-prompt credential handling. Add focused tests in `src/commands/doctor-claude-cli.test.ts` for a default Codex/OpenAI agent plus a non-default `agentRuntime.id: "claude-cli"` agent, no-Claude-agent skip behavior, and legacy defaults `claude-cli/*` compatibility. Do not add a broad `--agent` CLI option in this PR unless it is strictly necessary. Changelog likely not needed unless maintainers want user-facing doctor fixes recorded.
