## Summary

Keep open. Current `main` supports normal WhatsApp route bindings through `resolveAgentRoute` and documents sender routing as `match.peer`, so the reporter's `accountId`-as-phone example is mis-scoped. The remaining configured ACP binding gap is still real: WhatsApp does not register the configured-binding provider hooks or apply the configured/runtime binding route step that comparable channels use, and an open PR (#56866) is attempting related work. Because this affects an agent privilege-routing boundary, it should stay open for maintainer/security review.

## What This Changes

The issue requests that WhatsApp `bindings` route inbound messages to the configured restricted agent, or warn when the requested binding shape is unsupported instead of falling through to the default agent.

## Best Possible Solution

Make the WhatsApp binding contract explicit: keep documented `match.peer` routing for normal route bindings, add WhatsApp configured ACP binding provider and inbound configured/runtime binding resolution if that support is intended, and add validation or clear diagnostics for unsupported or mis-scoped WhatsApp binding rules so operators do not mistake a fallback route for an enforced restricted-agent boundary.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence reproduction path for the missing WhatsApp configured-binding provider and missing inbound configured-binding route step, but the issue's concrete route example uses `accountId` where current docs require `match.peer` for sender routing.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` addresses the ACP binding gap, but the safer maintainer solution also needs docs or validation around `accountId` versus sender `peer` because the reported security assumption came from that ambiguity.
