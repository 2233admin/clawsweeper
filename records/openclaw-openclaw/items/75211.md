## Summary

Keep open. Current main still lacks a WhatsApp configured-binding provider and the WhatsApp inbound path does not invoke configured conversation binding route resolution; however, the reporter's concrete `accountId` example appears to use a sender phone number where current route docs require `match.peer` for sender routing. Because the report is about an agent privilege boundary, this needs maintainer/security judgment rather than cleanup closure.

## What This Changes

The issue asks for WhatsApp to honor configured route bindings, or at least warn when WhatsApp bindings are unsupported or mis-scoped.

## Best Possible Solution

Define the intended WhatsApp binding contract, then either wire WhatsApp into the configured-binding provider and inbound binding route path with clear direct/group normalization, or add validation/docs warnings that prevent operators from treating unsupported or mis-scoped WhatsApp bindings as an agent privilege boundary.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence reproduction for the missing WhatsApp configured-binding provider and missing configured-binding route call, but the exact example uses `accountId` where current docs and router code use `match.peer` for sender routing.

## Solution Assessment

Unclear. Adding only the two provider hooks may solve the configured-binding gap, but it would not address the reporter's `accountId` sender-routing confusion; the safer solution likely combines explicit WhatsApp binding support with validation or docs around `accountId` versus `peer`.
