## Summary

Keep open. Current main appears to support documented WhatsApp sender routing through `match.peer`, so the issue's `match.accountId` example is not the normal route-binding shape for a sender phone number. However, the report is security-sensitive and still exposes a real contract gap/confusion: WhatsApp has no configured-binding provider, and unsupported or mis-scoped bindings can leave operators thinking a restricted-agent boundary is enforced when routing falls back to the default agent.

## What This Changes

The issue requests that WhatsApp bindings either route inbound WhatsApp conversations to the configured agent or warn clearly when a binding shape is unsupported, to avoid unexpected fallback to a more privileged default agent.

## Best Possible Solution

Define the WhatsApp binding contract explicitly: keep documented sender routing on `match.peer`, add validation or diagnostics for WhatsApp `accountId` values that look like sender phone numbers, and decide whether WhatsApp should also implement configured ACP/conversation bindings. Cover the chosen contract with routing/plugin tests and update the WhatsApp and multi-agent docs if behavior or warnings change.

## Reproduction Assessment

Partial. Static inspection gives high confidence that documented `match.peer` WhatsApp route bindings should work and that the reported `match.accountId` sender-number shape will not match a sender; the missing WhatsApp configured-binding provider is also reproducible by source search and compiler behavior, but no live WhatsApp run was performed.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` / `matchInboundConversation` would not fix the reporter's `accountId` sender-routing confusion; the safer solution is a maintainer-chosen contract that combines explicit WhatsApp support or warnings with docs/config validation around `accountId` versus `peer`.
