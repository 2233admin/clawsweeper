## Summary

Keep open. Current main supports normal WhatsApp route bindings through the generic router, and the issue's example appears to use `accountId` where current docs say sender routing should use `match.peer`. However, WhatsApp still lacks the channel-specific configured-binding provider and configured-binding route integration that other channel plugins expose, so this security-relevant routing report should not be closed as stale or implemented.

## What This Changes

The issue requests WhatsApp inbound routing to honor configured bindings for restricted-agent routing, or to warn when a WhatsApp binding cannot be enforced.

## Best Possible Solution

Decide the intended WhatsApp contract, then either add a WhatsApp binding adapter that normalizes direct E.164 chats and group JIDs and wires inbound through configured-binding routing, or add validation/docs warnings so operators cannot mistake `accountId` for sender-based routing. The security-sensitive end state is that unsupported or mis-scoped bindings are not silently trusted as an agent privilege boundary.

## Reproduction Assessment

Unclear. Static source gives a high-confidence reproduction for the missing WhatsApp configured-binding provider, but the exact route-binding example uses `accountId` where current docs say sender routing requires `match.peer`, so the reporter's exact fallback needs a corrected config or live replay to isolate.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` / `matchInboundConversation` would address the configured-binding gap, but it would not make the issue's `accountId` example a per-sender route; the maintainable fix likely needs either full WhatsApp configured-binding support or explicit config validation/docs around `accountId` versus `peer`.
