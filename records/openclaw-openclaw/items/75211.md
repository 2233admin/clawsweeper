## Summary

Keep open. Current main supports documented WhatsApp sender routing through `match.peer`, so the report's `accountId` example is mis-scoped, but WhatsApp still lacks the configured-binding provider and inbound configured-binding route integration that comparable channels use. Because the remaining gap touches an agent privilege-routing boundary, this needs maintainer/security follow-up rather than cleanup closure.

## What This Changes

The issue asks OpenClaw to make WhatsApp bindings route inbound messages to the configured restricted agent, or warn when a WhatsApp binding shape is unsupported instead of falling back silently.

## Best Possible Solution

Define the WhatsApp binding contract explicitly. Keep documented sender routing on `match.peer`, add validation or diagnostics for unsupported/mis-scoped WhatsApp bindings, and if WhatsApp should support configured ACP/conversation bindings, add the provider hooks plus inbound configured/runtime binding resolution with regression coverage.

## Reproduction Assessment

Unclear. Static inspection gives a high-confidence path for the missing WhatsApp configured-binding provider and missing inbound configured-binding route step, but the exact reported route example uses `accountId` where current docs and router behavior require `match.peer` for sender routing.

## Solution Assessment

Unclear. Adding only `compileConfiguredBinding` and `matchInboundConversation` would address part of the configured-binding gap, but the safer solution also needs validation or docs around `accountId` versus sender `peer` so operators do not mistake a nonmatching rule for an enforced privilege boundary.
