## Summary

Keep this issue open. Current main has generic Control UI reconnect handling and bounded cached-device-token recovery, but there is still no targeted source/test/release proof that the reported Control UI browser-control permission loss after Gateway restart or token rotation is fixed. The related PR #58022 was closed unmerged, and current browser control still requires `operator.admin`, which conflicts with the report's expectation that `operator.read` plus `operator.write` is sufficient.

## Best Possible Solution

Keep this issue as the canonical tracker. The best fix is to reproduce the Control UI `chrome-relay` browser command path after Gateway restart and device-token rotation, then add targeted recovery and regression coverage: distinguish normal code-1000 closes from stale-token or scope-upgrade failures, surface actionable approval/token guidance, and decide explicitly whether browser control should remain `operator.admin` or gain a narrower plugin/API permission model.
