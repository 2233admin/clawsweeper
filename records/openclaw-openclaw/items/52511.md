## Summary

Keep this issue open. Current main has generic Control UI reconnect handling and some token-drift recovery, but there is still no targeted source/test/release proof that the reported Control UI browser-control permission loss after Gateway restart or token rotation is fixed. The related PR #58022 was closed unmerged, and current browser control remains `operator.admin` scoped, which conflicts with the report's `operator.read` plus `operator.write` expectation.

## Best Possible Solution

Keep this issue as the canonical tracker. A maintainer should reproduce the Control UI `chrome-relay` browser command path after Gateway restart and device-token rotation, decide whether browser control should stay `operator.admin` or gain a narrower permission model, then add targeted recovery/UX guidance plus regression coverage for the chosen behavior.
