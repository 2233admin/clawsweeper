## Summary

Keep open. Current main still has the Slack capability failure path: scope discovery only tries `auth.scopes` and `apps.permissions.info`, does not parse `auth.test` metadata or Web API scope headers, and failed results still render as `Bot scopes: ...`. The reproduction comment and closed unmerged PR #44692 make this a valid narrow Slack plugin bug rather than cleanup closure.

## What This Changes

The issue requests that Slack capability checks succeed for manifest-created `xoxb-...` bot tokens instead of reporting `unknown_method` from the legacy Slack scope methods.

## Best Possible Solution

Implement the fix inside the Slack plugin: discover scopes for modern granular bot tokens via `auth.test` response metadata or accessible Web API scope headers, keep the legacy scope methods compatible, and add focused coverage for scope extraction plus `channels capabilities` diagnostics.

Yes. The issue has concrete CLI steps, martingarramon reproduced it on current upstream/main, and source inspection at current main still shows the exact two-method failure path. ## Solution Assessment

Yes. A Slack-plugin fix is the narrow maintainable solution because capability diagnostics and Slack probing are already plugin-owned; no core channel API change is needed. The smaller repair is to add `auth.test` after the legacy methods, while maintainers can choose to make it primary if modern bot tokens should be the fast path.
