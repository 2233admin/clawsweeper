## Summary

Keep this issue open. Current main still does not provide native per-peer outbound-only WhatsApp messaging: inbound DM admission is governed by `dmPolicy` plus `allowFrom`, outbound target authorization also uses `allowFrom`, `direct` peer config only supports prompt overrides, and bindings route admitted messages to agents rather than suppressing inbound processing.

## Best Possible Solution

Keep the issue open and implement this in the bundled WhatsApp plugin surface: add a direction-specific per-direct-peer inbound policy, enforce it immediately after WhatsApp DM admission/normalization and before enqueue/session/agent/reply work, keep outbound target authorization working for allowed notification targets, then update the WhatsApp schema, generated config docs/help, public WhatsApp docs, and focused tests for an allowlisted peer whose inbound messages are ignored while outbound sends still succeed.

## Risks / Open Questions

- This is messaging access-control behavior, so any implementation should preserve secure defaults for `dmPolicy`, pairing-store approvals, same-phone/self-chat handling, wildcard `allowFrom`, read-receipt behavior, and outbound echo protection.
- A generic binding-level `inbound` flag would affect all channels; a narrower WhatsApp owner-surface under the bundled WhatsApp plugin may be safer unless maintainers intentionally want a cross-channel routing contract.
