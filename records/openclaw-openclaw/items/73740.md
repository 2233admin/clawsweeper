## Summary

Keep open. Current main does not provide native per-peer outbound-only WhatsApp messaging: WhatsApp DM access is still governed by account/root `dmPolicy` plus `allowFrom`, outbound target authorization also uses `allowFrom`, bindings only choose an agent for already-admitted inbound messages, and the WhatsApp `direct` peer map only supports prompt customization.

## Best Possible Solution

Keep this issue open and implement it in the bundled WhatsApp plugin surface: add a direction-specific per-direct-peer inbound policy, enforce it before session creation/enqueue/reply, keep outbound send authorization working for allowed targets, and update the WhatsApp schema, docs, and focused tests for an allowlisted peer whose inbound messages are ignored while outbound sends still succeed.

## Risks / Open Questions

- The implementation touches messaging access control, so it should preserve current secure defaults, `dmPolicy="open"` wildcard validation, pairing behavior, same-phone handling, and outbound echo protections.
- A generic binding-level `inbound` field would affect all channels; a narrower WhatsApp account/direct-peer policy may be safer unless maintainers want a cross-channel routing contract.
