## Summary

Keep this issue open. Current main still does not provide native WhatsApp per-peer outbound-only control: direct-peer config only carries prompt metadata, the strict schema has no direction fields, inbound DM admission is still governed by dmPolicy/allowFrom/pairing, accepted messages proceed into read/enqueue/session/reply handling, and outbound authorization still reuses allowFrom rather than a separate outbound-only permission surface.

## Best Possible Solution

Keep this issue open and implement the feature in the bundled WhatsApp plugin/config surface: add a direction-specific per-direct-peer inbound policy, enforce it immediately after WhatsApp DM admission/normalization and before read receipts, enqueueing, session metadata, hooks, agent execution, or fallback replies, keep outbound sends authorized for notification-only peers, and update WhatsApp types, strict schema, generated config metadata/help, public docs, and focused tests for an outbound-allowed peer whose inbound messages are ignored.

## Risks / Open Questions

- A fix changes messaging access control and must preserve secure defaults for dmPolicy, pairing approvals, same-phone/self-chat handling, fromMe echo suppression, wildcard allowFrom, groupPolicy, and groupAllowFrom.
- The inbound-disabled decision should happen before read receipts, media downloads, session metadata, hooks, agent execution, and fallback replies; enforcing it later would still leak side effects for notification-only contacts.
- A generic binding-level inbound flag would affect multiple channel contracts. A WhatsApp-owned per-direct-peer policy is lower blast radius unless maintainers explicitly want a shared channel API.
