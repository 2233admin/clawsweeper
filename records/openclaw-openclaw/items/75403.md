## What This Changes

The PR updates `src/channels/typing.ts` to track pending typing starts and send a duplicate stop after a successful start resolves post-cleanup, with regression coverage in `src/channels/typing.test.ts`.

## Best Possible Solution

Land the focused shared typing-callback fix once CI is green, keeping the pending-start cleanup serialization in core with the new regression coverage rather than adding a Feishu-only workaround.

## Reproduction Assessment

Yes. A high-confidence reproduction exists by stubbing `start` to resolve after `onCleanup`: current main sends the only stop before Feishu has assigned `typingState`, and the delayed start can leave a persistent indicator with no later removal.

## Solution Assessment

Yes. The PR preserves the non-blocking typing start behavior while making cleanup pending-aware in the shared helper, which is the narrow maintainable fix for both initial starts and keepalive starts.
