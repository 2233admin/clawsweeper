## Summary

Keep this issue open. Current main has meaningful built-in config write validation, SecretRef preflight, base-hash guarded writes, and an agent-facing config mutation allowlist, but it still does not expose the requested synchronous plugin/API-level config pre-apply veto that a validator such as config-guard could rely on before config mutations are persisted.

## Best Possible Solution

Keep this issue open and route it through the config validation/mutation RFC direction. The best product/code path is a documented, synchronous, rejectable config-write interception point in the shared config mutation path, invoked after pending config computation plus built-in validation/preflight and before persistence. It should cover config.set, config.patch, config.apply, CLI/channel/plugin runtime config writers, and include current config, pending config, action/source, raw patch or full payload where available, baseHash, actor/client metadata, changed paths, and typed rejection results with tests proving rejected writes leave disk and active runtime config unchanged.

## Risks / Open Questions

- The requested hook is security-sensitive: fail-open/fail-closed behavior, hook timeout handling, recursion avoidance, SecretRef preflight ordering, active runtime snapshot consistency, and actor metadata all need explicit design before implementation.
- A partial RPC-only hook would miss many existing config writers that call replaceConfigFile or mutateConfigFile directly, including plugin runtime config mutation helpers and channel/setup flows.
- before_tool_call can mitigate agent tool calls but cannot reliably cover non-tool config writers or provide the pending config context requested by this issue.
