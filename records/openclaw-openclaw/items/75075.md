## Summary

Keep this PR open for maintainer review: current main still lacks the requested built-in tool-call exposure, the branch implements the missing opt-in path with focused docs/schema/tests, and I did not find a concrete correctness or security blocker in the latest head.

## What This Changes

The PR adds a default-off `gateway.http.endpoints.responses.exposeBuiltInToolCalls` config flag and wires Gateway Responses JSON/SSE output to include audit-only built-in tool-start events as `function_call` items, with docs, changelog, generated config schema, and e2e coverage.

## Best Possible Solution

Land this PR, or an equivalent maintainer-approved version, if maintainers accept the public opt-in audit contract: keep it default-off, use the existing sanitized run-scoped tool-start events, preserve client-tool delegation semantics, and keep tool results, compact summaries, and built-in delegate mode as separate follow-ups.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by running `/v1/responses` against current main with an agent turn that emits a run-scoped built-in `stream: "tool"`, `phase: "start"` event; current main has no JSON or SSE branch that forwards that event into `output`, and the PR's mocked e2e tests model the same path.

## Solution Assessment

Mostly yes. The proposed patch is narrow and additive, consumes the existing sanitized event stream instead of adding a second runtime channel, and keeps behavior default-off; the remaining question is maintainer approval of the public flag name and full-trace output semantics.
