## Summary

Keep this PR open for maintainer review: current main does not expose built-in tool calls on `/v1/responses`, and the linked feature issue is still open. The proposed default-off audit flag is narrowly scoped and uses existing sanitized tool-start events, with no blocking correctness or security finding found in the read-only review.

## What This Changes

The PR adds a default-off `gateway.http.endpoints.responses.exposeBuiltInToolCalls` flag, captures non-client agent tool-start events, surfaces them as `/v1/responses` JSON/SSE `function_call` output items, and updates config schema, docs, tests, generated hash, and changelog.

## Best Possible Solution

Land a maintainer-approved version that keeps the feature opt-in, preserves existing client-tool delegation semantics, documents the audit-only contract, and has CI proof for JSON and SSE ordering plus config-schema drift.

## Reproduction Assessment

Not applicable as a feature PR rather than a current crash or regression. For current-main verification, yes: code inspection shows `/v1/responses` only emits assistant items and caller-provided client `function_call` items, while built-in tool start events already exist on the agent event bus.

## Solution Assessment

Mostly yes: using a default-off Gateway config flag over existing sanitized `stream: "tool"` start events is a narrow, backward-compatible way to provide audit visibility. The safer pre-merge cleanup is to align non-stream JSON output ordering with the PR's documented assistant-to-audit ordering and assert it in tests.
