## Summary

Keep open for maintainer review: current main still lacks any built-in tool-call audit output on `/v1/responses`, and this PR is an active, coherent implementation paired with an open feature issue rather than obsolete work.

## What This Changes

The PR adds a default-off `gateway.http.endpoints.responses.exposeBuiltInToolCalls` flag, Gateway JSON/SSE capture of sanitized built-in tool-start events, config schema/docs/changelog updates, and focused OpenResponses tests.

## Best Possible Solution

Land a maintainer-approved version that keeps the behavior opt-in, uses the existing sanitized run-scoped tool-start events, preserves client-tool delegation semantics, and ships with schema, docs, changelog, and JSON/SSE ordering coverage.

## Reproduction Assessment

Yes. A focused reproduction is to emit a run-scoped `stream: "tool"`, `phase: "start"` event during a `/v1/responses` turn; current main has no code path that forwards it into `output`, while the PR tests model that path for JSON and SSE.

## Solution Assessment

Mostly yes. The patch is narrow and additive, consumes the existing sanitized event stream instead of adding a new runtime channel, and leaves the default response shape unchanged; the remaining question is maintainer approval of the public flag and output semantics.
