## Summary

Keep open. Current main has materially better Telegram token diagnostics, reload behavior, polling recovery, status warnings, and Windows/proxy transport handling than the reported v2026.3.13 build, but this issue still mixes Telegram auth/config visibility, cross-surface delivery, and sub-agent loop symptoms without a current Windows retest proving the whole user problem is fixed. The newer #73323 remains open for gateway-wide Windows degradation, while closed #74086 covers only a Telegram startup/network subset.

## What This Changes

The issue asks OpenClaw to make Telegram on Windows reliably apply config changes, avoid sync or response loops, and route messages between webchat and Telegram without 401 or polling failures.

## Best Possible Solution

Keep this as a consolidation point until a current Windows retest narrows the remaining behavior. If the reproduced symptom is gateway-wide runtime degradation, consolidate into #73323; if it only matches the closed #74086 startup subset, verify the release containing the current fixes; otherwise split focused follow-ups for Telegram auth/config visibility and sub-agent loop/session cleanup with logs, config paths, status output, and session identifiers.
