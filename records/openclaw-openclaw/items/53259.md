## Summary

Keep open. The PR targets a real documented Discord elevated-authorization bug, but current main has not implemented the fallback and the PR branch still needs maintainer/author revision before merge: it is dirty against main, hard-codes Discord config fallback in core despite an existing channel-plugin fallback seam, and its current diff regresses the human `sandbox explain` diagnostics output.

## What This Changes

The PR adds a Discord elevated `allowFrom` fallback resolver, uses it from runtime elevated authorization and `sandbox explain` with account context, and adds focused regression tests.

## Best Possible Solution

Land a focused rebased fix that implements Discord fallback through `ChannelElevatedAdapter.allowFromFallback` on the Discord plugin or an equivalent extension-owned adapter, shares the generic fallback resolution between runtime auth and `sandbox explain` with account context, preserves existing human and JSON diagnostics, and tests channel-level fallback, account-level fallback, and explicit empty override behavior.

## Reproduction Assessment

Yes. The issue is reproducible at the config-resolution level: current docs promise the Discord fallback, current runtime only calls the plugin fallback hook, the Discord plugin does not provide that adapter, and `sandbox explain` only reads `tools.elevated.allowFrom[channel]`.

## Solution Assessment

No. The intended behavior is correct, but the PR is not the best final shape because it puts Discord-specific fallback policy in core and removes existing human diagnostics; the safer solution is an extension-owned fallback adapter plus preserved `sandbox explain` output.
