## Summary

Keep this issue open. Current main still routes the macOS Skills settings install action through the gateway-backed `skills.install` dependency-installer path, still defaults to preferring Homebrew when `brew` exists, and still formats failed installer commands generically without a CLT/macOS 26 classifier, remediation, or alternate-installer retry after a selected brew install fails. The prior ClawSweeper keep-open review remains consistent with current main because the relevant files have not changed since that review SHA.

## Best Possible Solution

Fix the gateway-backed skill dependency installer path: detect Homebrew/Xcode CLT unsupported-macOS errors, return a targeted remediation message, avoid choosing brew when the host brew toolchain is known unusable, retry a viable uv/node/go/download alternative when the selected skill metadata provides one, and add focused tests for the failure classifier plus installer-selection/fallback behavior.
