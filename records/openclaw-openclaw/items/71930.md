## Summary

Keep open. Current main still filters Mattermost WebSocket post handling to `posted` events only, so `post_edited` cannot reach the existing post, mention, and wake path; the linked open PR is the active implementation candidate and the issue should remain open until that work lands or is closed.

## What This Changes

Mattermost should process `post_edited` WebSocket events so @mentions added by editing an existing message trigger the normal mention/wake flow.

## Best Possible Solution

Review and land PR #72513, or an equivalent Mattermost-only patch, so `post_edited` is treated as a post-bearing WebSocket event, the updated message body flows through the existing access and mention gates, and edited-post replay behavior is deliberate.

## Reproduction Assessment

Yes. A `post_edited` WebSocket payload on current main follows a deterministic early-return path in `monitor-websocket.ts` before `onPosted`, and the issue provides concrete Mattermost steps confirmed on April 25, 2026.

## Solution Assessment

Yes. The narrow maintainable solution belongs in the Mattermost plugin WebSocket/parser and replay-guard path; no core channel API change is evident from the current source, scoped plugin boundary rules, or Mattermost docs surface.
