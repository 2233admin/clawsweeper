## Summary

Keep this issue open. Current main still documents the WSL2 + Windows Chrome CDP path as IPv4-only and the browser diagnostics still probe only the configured CDP endpoint, so the reported Chrome IPv6 loopback / v4tov4 portproxy failure remains actionable. The closed duplicate #68666 corroborates the same failure mode and should be treated as supporting context for this canonical issue.

## Best Possible Solution

Keep this issue open and make a narrow browser-plugin docs/diagnostics update. The WSL2 + Windows CDP guide should add Windows-side checks for 127.0.0.1 and [::1], netstat binding inspection, and v4tov6 portproxy guidance when Chrome listens only on IPv6 loopback. Browser doctor/status should add a targeted remote-CDP hint that distinguishes configured IPv4 endpoint failure from a likely Windows IPv6 loopback listener or portproxy mismatch while keeping the transport and SSRF policy generic.
