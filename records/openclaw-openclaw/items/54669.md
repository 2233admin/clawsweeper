## Summary

Keep this issue open. Current main still lacks the requested WSL2 + Windows Chrome CDP IPv6 documentation and still reports generic browser CDP reachability diagnostics against only the configured profile endpoint. The related #68666 report corroborates the same failure mode but was closed as a duplicate of this issue, so it is not a separate canonical tracker.

## Best Possible Solution

Keep this issue open and make a narrow browser-plugin docs/diagnostics update. The WSL2 guide should add Windows-side checks for both `127.0.0.1:9222` and `[::1]:9222`, `netstat -ano | findstr :9222`, and `netsh interface portproxy add v4tov6 ... connectaddress=::1` when Chrome listens only on IPv6 loopback. The browser doctor/status path should add a remote-CDP hint that distinguishes configured IPv4 endpoint failure from a likely Windows IPv6 loopback listener or portproxy mismatch, while leaving the core CDP transport generic.
