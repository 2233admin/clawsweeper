## Summary

Keep this issue open. Current main still has partial safeguards, including static hook timeouts, fail-open/fail-closed hook policy, session-lock watchdog cleanup, diagnostics, and subsystem-specific circuit breakers, but it does not implement the requested framework-level plugin circuit breaker with rolling plugin budgets, lock attribution, persisted open/half-open/forced-open state, operator reset/status surfaces, or per-plugin fail modes. The newer user comment and related open #55253 reinforce that this remains real gateway/plugin-runtime reliability work rather than conservative cleanup.

## Best Possible Solution

Keep this open as plugin/gateway runtime stability work. The likely path is an additive core plugin-runtime breaker around hook and plugin-owned runtime execution: record plugin id, hook name, session id, timing, timeout, error, and lock-overrun telemetry; maintain rolling failure budgets; persist open/half-open/forced-open state; expose status, metrics, and operator reset controls; support per-plugin fail modes; and add tests proving one failing plugin degrades without blocking unrelated plugins. Coordinate with #55253 if maintainers want one canonical tracker.
