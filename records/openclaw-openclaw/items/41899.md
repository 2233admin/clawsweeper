## Summary

Keep this issue open. Current main has useful partial safeguards, including hook error catching, one default modifying-hook timeout, invalid plugin config isolation, and subsystem-specific circuit breakers, but it still does not implement the requested plugin-scoped breaker with rolling budgets, lock attribution, persisted open/half-open/forced-open state, status visibility, or operator reset controls. Related #55253 overlaps but remains open and narrower.

## Best Possible Solution

Keep this open as core plugin/gateway stability work. The best implementation is an additive plugin-runtime breaker around hook and plugin-owned runtime execution, with structured plugin id plus hook/session telemetry, lock-overrun attribution, rolling failure and timeout budgets, persisted breaker state, status/diagnostics output, operator reset and forced-open controls, and tests proving one failing plugin degrades without blocking unrelated plugins. Coordinate with #55253 if maintainers want one canonical tracker.

## Risks / Open Questions

- Closing this now would lose a concrete core stability request that remains broader than the existing partial safeguards and the open related tracker #55253.
- A naive implementation could accidentally skip fail-closed security hooks such as before_tool_call; the breaker design needs per-hook/per-plugin fail mode policy so security hooks do not silently degrade open.
