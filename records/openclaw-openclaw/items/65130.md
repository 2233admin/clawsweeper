## Summary

Current main and the shipped stable release already resolve the reported npm global update EEXIST failure by staging npm installs in a temporary prefix, verifying the staged package, then swapping the package tree and npm bin shim into the real global prefix. The earlier force-based PR was closed as obsolete, and the later fork PR is the same superseded approach rather than remaining core work.

## What This Changes

The issue requests that Docker/container package-manager updates recover from npm global install EEXIST failures when `/usr/local/bin/openclaw` already exists.

## Best Possible Solution

Keep the shipped staged install/swap implementation as the maintained fix, and treat force-based follow-up PRs as superseded unless a new report shows a distinct failing container layout against `v2026.4.26` or newer.

## Reproduction Assessment

Yes. The reported failure maps to npm installing into the live global prefix and colliding with an existing `/usr/local/bin/openclaw`; current source and regression tests show npm now installs into a staged prefix before replacing the live package and shim.

## Solution Assessment

Yes. The current staged install, verify, swap, and rollback flow is safer than adding broad npm `--force` because it avoids direct live-prefix overwrite during the package-manager step and keeps verification before activation.
