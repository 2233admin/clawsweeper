## Summary

Keep open. The branch addresses a real ACP completion-delivery pain point, but current main has moved parent-owned ACP completion toward the task/subagent announce path, while this PR adds a separate direct outbound route that can corrupt streamed text and bypass duplicate-suppression and completion-policy boundaries.

## What This Changes

The PR adds best-effort `routeReply` final/error delivery from the ACP parent stream relay for non-threaded parent delivery contexts, plus tests for direct chats, threaded suppression, lookup failure, and error delivery.

## Best Possible Solution

Keep a single ACP completion owner. Prefer the current task/subagent completion announce path for final user-visible output; if maintainers explicitly want `streamTo: "parent"` to deliver final raw text, add it through that owner with sanitization, idempotency, whitespace-preserving aggregation, and duplicate terminal-notification suppression.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the PR defects: a PR-branch relay receiving assistant chunks like `hello`, ` `, `world` would drop the standalone whitespace before direct delivery, and a non-threaded ACP run with a pending task terminal update can receive both the new direct reply and the existing terminal task notification.

## Solution Assessment

Is this the best way to solve the issue? No. The direct `routeReply` side path is not the narrowest maintainable fix against current main’s parent-owned ACP completion model; extending or explicitly revising the task-completion announce path is safer.
