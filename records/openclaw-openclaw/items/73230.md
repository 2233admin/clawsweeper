## Summary

Keep open. Current main still treats cron `delivery.accountId` as a static explicit account and gives it precedence over session and binding-derived account IDs, so a rotating channel account can leave scheduled delivery pinned to a stale identity. The report is recent, source-reproducible, and the fix belongs in generic cron delivery resolution rather than a WeChat-specific core branch.

## What This Changes

Cron delivery should dynamically resolve or recover stale `delivery.accountId` values for channel plugins whose account IDs can rotate after re-login or gateway restart.

## Best Possible Solution

Implement channel-agnostic delivery-time account resolution: preserve an explicit account when it still maps to an eligible account for the resolved channel, recover only when the stored account is stale and exactly one eligible configured account exists, and otherwise fail with a clear actionable error. Cover the resolver and cron delivery surfaces with focused tests and document the behavior in cron docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure an isolated cron announce delivery with a stale `delivery.accountId`; current `resolveDeliveryTarget` returns that stale value because explicit account IDs have highest precedence before outbound dispatch.

## Solution Assessment

Yes, if implemented conservatively in the generic cron delivery path. Delivery-time stale-account recovery with strict ambiguity handling is narrower and safer than startup mutation, silent job rewrites, or a WeChat-specific branch.
