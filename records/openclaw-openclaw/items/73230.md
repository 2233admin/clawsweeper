## Summary

Keep open. Current main still treats cron delivery account IDs as static explicit identities, so rotating channel account IDs can leave scheduled announces pinned to stale accounts; the related spoof-guard work is important safety context but does not implement stale-account recovery.

## What This Changes

Cron delivery should dynamically resolve or safely recover stale channel account IDs for channels whose login/session lifecycle can rotate the active account ID.

## Best Possible Solution

Implement channel-agnostic delivery-time recovery: keep an explicit account ID when it still resolves for the target channel, recover only when the stored value is stale and exactly one eligible configured or active account exists for that same channel, and otherwise fail with an actionable error telling the user how to update the cron delivery account.

## Reproduction Assessment

Yes. A source-level reproduction is clear: create an isolated announce cron job with a stale delivery.accountId; current resolveCronDeliveryPlan and resolveDeliveryTarget preserve that stale explicit value, and sendCronAnnouncePayloadStrict forwards it to outbound delivery.

## Solution Assessment

Yes, for the preferred direction. Delivery-time recovery with strict single-account ambiguity handling is narrower and safer than startup mutation, and it can stay generic instead of adding a WeChat-specific core branch.
