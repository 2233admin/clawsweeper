## Summary

Keep this issue open. Current main still preserves and prioritizes a static `delivery.accountId` for cron delivery, and the existing fallback only fills an account from session history or agent/channel bindings when no account id is present. I did not find an implemented `auto`/`active` account sentinel, a stale-account fallback, or startup warning/repair for cron jobs that reference rotated channel account ids. The WeChat plugin itself is external, but the reported failure sits on a generic cron delivery contract, so this is not a clean ClawHub-only close.

## Best Possible Solution

Keep this issue open and implement a generic cron delivery account-resolution path. The likely core direction is an additive, channel-agnostic contract such as `delivery.accountId: "active"` or omitted-account fallback that resolves at delivery time only when the channel has exactly one configured/active account, plus clear errors or doctor/fix warnings when a stored account id no longer resolves. If plugin state is needed to know the active account, add a typed SDK/plugin capability instead of hardcoding `openclaw-weixin`; the external plugin should also prefer stable account ids when possible.
