## Summary

Keep this PR open for maintainer judgment. Current main still does not ship the proposed devcontainer, benchmark workflow, or release SBOM workflow, so it is not implemented or superseded. The PR is actionable repo CI/release work, but the latest diff still has unresolved workflow and security review issues, and it bundles three separate surfaces that would be safer as focused maintainer-reviewed changes.

## Best Possible Solution

Keep the PR open for maintainer revision or explicit CI/release-policy decline. The best path is to split devcontainer, benchmark, and SBOM into focused PRs; reuse the maintained setup-node-env action; wire benchmark CI through the existing startup benchmark and budget scripts; avoid or checksum-verify privileged third-party tool installs; generate any release SBOM from the shipped npm tarball or production dependency set; and correct the unrelated closing reference.

## Risks / Open Questions

- Benchmark workflow installs a downloaded .deb with sudo without integrity verification on PR/push CI.
- SBOM workflow grants release `contents: write` and uses third-party actions to attach a release asset, so trigger scope and action choice need explicit maintainer review.
- SBOM generation scans the full checkout after a full dependency install instead of the shipped npm tarball or production dependency set.
- The PR duplicates existing setup and benchmark harnesses instead of using `.github/actions/setup-node-env/action.yml` and `test:startup:bench`/`test:startup:bench:check`.
- The PR bundles devcontainer, benchmark CI, and release SBOM policy into one external-author infra change, which makes review and rollback harder.
