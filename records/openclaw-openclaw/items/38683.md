## Summary

Keep this PR open. Current main still does not implement startup/pre-validation Telegram `allowFrom` auto-heal for `dmPolicy="open"`; it keeps explicit wildcard validation plus an explicit `openclaw doctor --fix` repair path. The PR is small, but it mutates Telegram access control by adding `"*"`, so it needs maintainer security/product review against the merged #74112 open-DM policy semantics.

## What This Changes

This PR adds Telegram config normalization in `src/config/zod-schema.providers-core.ts` to insert `"*"` into top-level or account `allowFrom` when the effective `dmPolicy` is `open`, and updates a config validation test to expect the auto-heal behavior.

## Best Possible Solution

Keep this PR open for maintainer security/product judgment. If startup recovery is still wanted, rework it on current main around #74112 semantics: mutate only the intended Telegram top-level or account surface, never widen sibling inherited allowlists, avoid module-level warning state, cover missing/empty/non-wildcard/inherited account cases, and update docs/changelog. If maintainers prefer the current explicit contract, close this PR intentionally and keep `openclaw doctor --fix` plus clear validation guidance as the supported repair path.

## Risks / Open Questions

- Auto-inserting `"*"` during startup can turn a restrictive-looking Telegram config into public DM access for a tool-capable bot, so this is security-sensitive access-control behavior.
- The PR branch predates current main’s merged #74112 open-DM allowlist semantics and needs review/rebase against the current effective-policy model.
- The proposed module-level warning singleton can make warning behavior process-global and order-dependent if tests or validation flows assert warning delivery.
- Current main already has an explicit `openclaw doctor --fix` repair path, so replacing cold-start failure with implicit startup mutation is a maintainer product/security decision rather than a narrow bug fix.
