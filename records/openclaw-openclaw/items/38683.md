## Summary

Keep this PR open. Current main still validates Telegram `dmPolicy="open"` strictly: top-level and per-account configs must already include `"*"` in the effective `allowFrom`, and normal validation fails before runtime materialization. Main has a doctor-side repair for this class of invalid config, but not the requested startup/pre-validation auto-heal. The PR also changes Telegram access-control posture by auto-inserting a public wildcard, so it needs maintainer security/product judgment, especially with #73756 still open.

## Best Possible Solution

Keep this PR open for maintainer security/product judgment. If maintainers want startup recovery, implement a narrow Telegram pre-validation normalization that only mutates the intended top-level channel or account, never widens sibling inherited allowlists, emits a testable warning without module-order coupling, and adds missing/empty/non-wildcard/inherited/sibling-account tests plus docs and changelog. If maintainers prefer the current explicit contract, close this PR intentionally after documenting `openclaw doctor --fix` as the supported repair path and reconciling #73756.
