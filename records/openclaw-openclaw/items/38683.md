## Summary

Keep open. Current main still enforces the explicit Telegram wildcard contract at config validation time, while the existing recovery path is `openclaw doctor --fix` rather than startup/schema auto-heal; newer open-DM allowlist hardening around #73756/#74112 also makes automatic wildcard insertion a security/product decision.

## What This Changes

This PR adds Telegram config normalization in `src/config/zod-schema.providers-core.ts` to insert `"*"` into top-level or account `allowFrom` when `dmPolicy` is `open`, and updates a config validation test to expect the auto-healed result.

## Best Possible Solution

Maintainers should decide whether startup auto-heal is still desired after the newer fail-closed Telegram policy. If yes, rework this on current main so it mutates only the intended Telegram top-level or account surface, preserves restrictive parent allowlists, avoids process-global warning state, and updates docs/changelog/tests; if not, close the PR intentionally and keep explicit validation plus `openclaw doctor --fix` as the supported recovery path.
