## Summary

Keep open: current `main` still has the reported timeout path. After `sessions_send` starts the target run and `agent.wait` times out, the handler returns `{ status: "timeout" }` before launching the existing background A2A announce flow, while the fire-and-forget and successful-wait paths do launch that flow.

## What This Changes

The issue requests that `sessions_send` still start the agent-to-agent announce flow asynchronously when the caller wait times out.

## Best Possible Solution

Update the `sessions_send` timeout branch to invoke the existing A2A flow with the target `runId` before returning timeout to the caller, add focused regression coverage for that branch, and keep broader announce-timeout policy separate.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make `waitForAgentRunAndReadUpdatedAssistantReply` return `timeout` after `startAgentRun`; current main returns at `src/agents/tools/sessions-send-tool.ts:379` before any `startA2AFlow` call.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing `startA2AFlow(undefined, runId)` and `waitRunId` helper path in the timeout branch while preserving the owned-ACP-child skip guard.
