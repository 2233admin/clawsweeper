## Summary

Keep this PR open. Current main still does not implement the requested Feishu `replyToMode` config surface or runtime policy; the remaining work is Feishu-owned and still useful, but the PR needs a current rebase plus complete handling across streaming cards, chunked text/cards, media sends, metadata, docs, and tests. Security review did not find supply-chain or secret-permission expansion in the provided diff.

## Best Possible Solution

Keep this PR open and ask for a Feishu-owned rebase onto current main. The best implementation should add `replyToMode` to the strict Feishu top-level/account/group schemas, plugin manifest, generated config metadata, and public Feishu docs; resolve group/account/top-level precedence; apply one reply-target policy across streaming start, chunked text/card sends, media sends, and typing behavior; and add focused tests for `off`, `first`, and `all`. Reusing the existing generic reply-threading helpers or adding a Feishu threading adapter is reasonable, but the Feishu dispatcher still needs to consume the policy instead of sending a fixed reply target.
