## Summary

Current main still has the routing gap described in the report: the guarded OpenAI Responses transport exists and handles terminal 429s, but embedded runs only select it when the session stream is undefined or exactly `streamSimple`; a custom session stream remains `session-custom` and can be wrapped by Copilot without gaining the guarded fetch behavior.

## What This Changes

The issue asks OpenClaw to make Copilot `openai-responses` 429 quota failures fail fast on embedded `session-custom` streams instead of timing out through the idle watchdog.

## Best Possible Solution

Keep this issue open and fix the invariant at the embedded/provider stream boundary: every `openai-responses` Copilot run should use the guarded Responses fetch or an equivalent provider-owned transport before the idle watchdog, while preserving Copilot's header/payload wrapper and avoiding a core hard-code for one provider where a generic Responses seam is enough.

## Reproduction Assessment

Yes. The issue supplies a live v2026.4.29 reproduction with immediate upstream HTTP 429 and `session-custom` timeout telemetry, and current main still has the exact source condition that lets a custom session stream bypass the boundary-aware Responses transport.

## Solution Assessment

Yes, the requested invariant is the right direction. The safest implementation is a narrow generic change in embedded stream resolution or provider stream registration so Responses transports get guarded fetch behavior even when the session starts with Pi's default custom closure; avoid broad retry-policy changes or Copilot-only core branching unless tests prove a generic seam is insufficient.
