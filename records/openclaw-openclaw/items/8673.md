## Summary

Keep open. Current main still lacks a generic bounded retry for transient OAuth refresh failures: the shared refresh path awaits the provider/plugin refresh once, the manager adds locking/timeouts/adoption fallbacks but no transient retry loop, and the only retry-like path is still specific to refresh_token_reused recovery. The two linked retry PRs are closed and unmerged in the provided GitHub context.

## Best Possible Solution

Implement a bounded, tested retry policy around the generic OAuth refresh attempt, likely near the oauth-manager adapter.refreshCredential call so both provider-plugin refresh and the pi-ai fallback are covered. Reuse src/infra/retry.ts, add a conservative transient-error classifier, explicitly avoid retrying terminal OAuth errors such as invalid_grant, revoked/expired credentials, and refresh_token_reused, and preserve the current lock, adoption, main-store fallback, and reused-token recovery behavior.

## Risks / Open Questions

- Retrying the wrong OAuth errors could spend or rotate refresh tokens, hide invalid_grant/revoked-token failures, or delay required re-authentication.
- Adding retry inside the refresh lock can increase lock hold time; the retry budget must remain consistent with OAUTH_REFRESH_CALL_TIMEOUT_MS and OAUTH_REFRESH_LOCK_OPTIONS.
- A fix scoped only to getOAuthApiKey would miss provider-plugin refreshOAuth failures now routed through refreshProviderOAuthCredentialWithPlugin.
