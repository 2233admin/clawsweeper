## Summary

Keep open. Current main still has the plausible Windows host-exec failure path described in the report: Windows exec selects one PowerShell wrapper, non-PTY exec sends that argv to the supervisor, and the shared spawn fallback still retries only EBADF by default rather than spawn-time EPERM. The related Microsoft tracker is open inventory, not a merged fix or canonical duplicate that resolves this issue.

## What This Changes

The issue requests a fix for Windows 11 agent exec commands failing immediately with `spawn EPERM` through Feishu or the web UI while the `openclaw` CLI itself still runs.

## Best Possible Solution

Keep this open for a focused Windows exec investigation: capture the selected shell path, cwd, original spawn error code/syscall, and command context, then either add a narrow validated PowerShell fallback or improve diagnostics with mocked Windows coverage and one native Windows smoke proof.

## Reproduction Assessment

No. The report gives plausible Windows 11 steps and current main still matches an unhandled `spawn EPERM` path, but this review did not reproduce it on Windows and the issue has no logs or selected-shell evidence.

## Solution Assessment

Unclear; there is no proposed patch. A blanket EPERM retry is not the best default for host exec, so the safer path is targeted diagnostics plus a constrained fallback only after reproducing the failing Windows shell selection.
