## Summary

Keep open. Current main still has the plausible Windows host-exec failure path: Windows exec selects a PowerShell wrapper, non-PTY host exec passes that argv to the supervisor, and the shared spawn fallback retries only EBADF by default, not EPERM. The related Microsoft tracker is open inventory, not a merged fix.

## What This Changes

The issue requests fixing Windows 11 host exec so commands launched through chat or the web UI no longer fail immediately with `spawn EPERM`.

## Best Possible Solution

Keep this open and investigate on native Windows: log or capture the selected shell path, cwd, command text, service/user context, and original spawn error fields, then add a narrow validated fallback or clearer diagnostic with mocked Windows coverage and a Windows smoke check.

## Reproduction Assessment

No. The report gives plausible Windows 11 steps, and current main still has a code path where spawn-time EPERM would surface immediately, but there is no native Windows reproduction log or selected-shell evidence in the issue data.

## Solution Assessment

Unclear; there is no proposed patch. A blanket EPERM retry is not the best first move for host exec, so the safer path is targeted Windows reproduction plus a constrained fallback or diagnostic once the failing shell/permission case is known.
