## Summary

Keep this PR open. Current main still has the Overview WebSocket URL input behavior the PR targets, so the intended UI fix has not shipped. The PR is not ready to land as-is because its UI change preserves a gateway token while changing the endpoint, and it also includes an unrelated memory-lancedb retry hunk that current main already handles more completely.

## Best Possible Solution

Keep this PR open only as an implementation candidate and require rework before landing: split or drop the memory-lancedb hunk, preserve URL-editing ergonomics without carrying credentials across endpoint scopes, and add focused regression coverage. A good product path is a draft URL or confirmation flow that clears or reloads the token only when the normalized selected gateway actually changes, coordinated with the linked bug #41545 and other open implementation candidates.

## Risks / Open Questions

- Security: preserving settings.token while changing settings.gatewayUrl can send a token intended for one gateway to a different WebSocket endpoint on the next Connect action.
- Storage regression: because settings persistence writes the current token under the current gateway URL scope, preserving the token during input can store it under intermediate or newly typed endpoint scopes.
- Scope risk: this PR bundles an unrelated memory-lancedb retry change, and current main already has fuller retry handling and coverage for that area.
