## Summary

Keep this PR open. Current main still has the reported Overview behavior, so the PR is not obsolete or implemented. However, the proposed UI hunk should not land as-is because it preserves a gateway token while the endpoint is edited, which conflicts with the documented/tested per-gateway token boundary; the PR also bundles an unrelated memory-lancedb retry hunk that current main already handles more completely.

## Best Possible Solution

Keep this PR open as a rework candidate for #41545. The best path is to split or drop the memory-lancedb hunk, then implement a Control UI fix that avoids clearing the token on harmless in-progress URL edits while still clearing, reloading, or explicitly confirming credentials when the normalized selected gateway actually changes. Add focused regression coverage for editing ergonomics and per-gateway token isolation, and coordinate with #41546 and #42001.

## Risks / Open Questions

- Landing the UI hunk as-is can send a token intended for one gateway to a different edited WebSocket endpoint on Connect.
- Landing the UI hunk as-is can persist the old token under newly typed gateway URL scopes, weakening the documented per-gateway token boundary.
- The PR bundles an unrelated memory-lancedb retry change that current main already covers more completely, making the branch dirty for a focused UI fix.
- There are overlapping open implementation candidates for #41545 (#41546 and #42001), so maintainers should choose or request one canonical security-aware fix rather than merge duplicate partial fixes.
