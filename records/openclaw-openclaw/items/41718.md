## Summary

Keep this PR open. Current main still has the Overview WebSocket URL input behavior the PR targets, so the intended UI bug fix has not shipped. The PR is not ready to land as-is because the UI hunk keeps a token while changing the endpoint, and the bundled memory-lancedb retry hunk is unrelated and already covered more completely on current main.

## Best Possible Solution

Keep this PR open only as an implementation candidate and require rework before landing: split or drop the memory-lancedb hunk, preserve URL-editing ergonomics without carrying credentials across endpoint scopes, and add focused regression coverage. A good path is a draft URL or confirmation flow that clears or reloads the token only when the normalized selected gateway actually changes, coordinated with the linked bug #41545 and the other open implementation candidates.

## Risks / Open Questions

- Security: the PR's UI hunk can preserve a token from one gateway and send it to a different WebSocket endpoint on the next Connect action.
- Storage regression: preserving settings.token while typing a new gatewayUrl can persist the old token under intermediate or newly typed gateway scopes.
- Scope risk: the PR bundles an unrelated memory-lancedb retry change, while current main already handles failed MemoryDB initialization retries with coverage.
