## Summary

Keep this PR open for maintainer follow-up. Current main still has the Overview WebSocket URL edit behavior the PR targets, so the intended UI fix has not shipped. The PR is not ready to land as-is because its UI hunk preserves credentials across endpoint edits, and its memory-lancedb hunk is unrelated to the stated UI bug and is already covered more completely on current main.

## Best Possible Solution

Keep this PR open only as an implementation candidate and require rework before landing: split or drop the memory-lancedb hunk, fix the URL-editing ergonomics without carrying credentials across gateway scopes, and add focused regression coverage. A good direction is a draft URL or confirmation flow that avoids clearing the token on harmless in-progress edits but clears or reloads the token when the normalized selected gateway actually changes, coordinated with #41545 and the other open implementation candidates.

## Risks / Open Questions

- Security: the PR's UI hunk can preserve a token from one gateway and send it to a different WebSocket endpoint on the next Connect action.
- Storage regression: preserving settings.token while gatewayUrl changes can persist the old token under newly typed gateway URL scopes.
- Scope risk: the PR bundles an unrelated memory-lancedb retry change, while current main already has a broader retry reset with tests.
