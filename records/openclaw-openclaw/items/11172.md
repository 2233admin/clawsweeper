## Summary

Keep open. Current main still does not inject `OPENCLAW_AGENT_ID`, `CLAWDBOT_AGENT_ID`, or `OPENCLAW_SESSION_KEY` into exec child-process environments for subagent sessions. The current implementation derives agent/session metadata for descriptions, routing, approvals, and subagent session keys, but the environment path only merges inherited/request/sandbox env and adds `OPENCLAW_SHELL=exec`. The related implementation PR #11497 is closed unmerged, #11477 was closed as a duplicate of this issue, and open PR #53262 concerns `sessions_spawn.agentId` schema behavior rather than exposing identity to spawned commands.

## Best Possible Solution

Keep this issue open until OpenClaw ships or explicitly rejects a documented core exec identity contract. The likely implementation belongs in shared exec environment construction after agent/session context is resolved: inject `OPENCLAW_AGENT_ID` and `OPENCLAW_SESSION_KEY` consistently, decide whether `CLAWDBOT_AGENT_ID` compatibility is supported, define override/spoofing semantics, and add focused tests plus docs for gateway, node, sandbox, PTY, and subagent-spawned sessions.
