## Summary

Keep open: current main still lacks outbound view-once support for WhatsApp and Signal, so the PR is not obsolete. The patch should not merge as-is because it introduces concrete privacy/correctness gaps around Signal media-kind support, preflight media fetching, and WhatsApp auto-reply text handling.

## What This Changes

This PR adds a `viewOnce` send option across the CLI, agent tool schema, gateway send protocol, outbound delivery and queue recovery, WhatsApp Baileys sending/session handling, Signal JSON-RPC sending, and auto-reply forwarding paths.

## Best Possible Solution

Land a rebased, maintainer-approved implementation that treats `viewOnce` as an additive media-only contract, validates unsupported channels and media kinds before any send starts, avoids consuming remote media during validation unless the loaded bytes are reused, preserves queue/recovery semantics, and documents the CLI/tool/gateway behavior for WhatsApp and Signal.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Reject Signal view-once videos before RPC send:** `extensions/signal/src/send.ts:179-180`
  - body: `signal-cli` documents view-once as supported only for messages with an image attachment, but this branch treats `attachmentKind === "video"` as supported and sends `"view-once": true`. A Signal video send can therefore reach the daemon as an unsupported view-once request instead of failing predictably.
  - confidence: 0.91
- **[P2] Avoid preflight media downloads for view-once validation:** `src/infra/outbound/deliver.ts:698-701`
  - body: This validation loads every media URL, but the WhatsApp and Signal senders load the same URL again for the actual send. One-shot or short-lived URLs can be consumed by validation, so an otherwise valid view-once send fails before delivery.
  - confidence: 0.86
- **[P2] Reject view-once replies that still send permanent text:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:151`
  - body: Forwarding `replyResult.viewOnce` only on media payloads leaves the existing text-only and remaining-text branches untouched. A view-once auto-reply with no media, or a long caption split into later chunks, will still send permanent WhatsApp text despite the disappearing-media flag.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the diff introduces privacy-sensitive sending behavior with validation gaps that can misrepresent view-once delivery.

Concerns:

- **[medium] Permanent text can bypass the view-once expectation:** `extensions/whatsapp/src/auto-reply/deliver-reply.ts:151`
  - body: WhatsApp auto-replies only attach `viewOnce` to media payloads; text-only replies and split caption text still use permanent `msg.reply` sends, which can leak content a caller expected to be disappearing.
  - confidence: 0.79
- **[low] Signal view-once support exceeds the dependency contract:** `extensions/signal/src/send.ts:179`
  - body: The PR advertises and sends Signal view-once for video attachments even though the signal-cli command contract documents view-once support for image attachments, creating an unreliable privacy surface.
  - confidence: 0.86
