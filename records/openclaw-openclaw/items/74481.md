## Summary

Keep open. Current main still has static/configured catalog plumbing and provider-local discovery precedents, but it does not implement configured OpenAI-compatible `/v1/models` refresh for the Control UI picker. The same author has an open implementation slice in #74488, but that PR explicitly does not wire discovery into catalog resolution, so this issue should remain open for maintainer API/product direction.

## What This Changes

Request adding an opt-in dynamic provider model catalog refresh that discovers configured OpenAI-compatible `/v1/models`, caches and merges or replaces it, and updates the Control UI model picker.

## Best Possible Solution

Settle the provider-catalog contract first, preferably by reusing or deliberately extending the existing `catalog` and model-cache architecture, then implement configured OpenAI-compatible discovery with endpoint/auth-scoped caching, clear merge/replace behavior, Gateway/Control UI refresh semantics, tests, docs, and a security pass around network fetches.

## Reproduction Assessment

Not applicable as a traditional crash reproduction, but there is a high-confidence verification path by inspection: configure `models.providers.openai.baseUrl` and an API key without explicit models, then the Control UI still calls `models.list` and current main has no code path that fetches that provider's `/models` endpoint.

## Solution Assessment

Unclear until maintainer review. The proposed direction is plausible and matches existing provider-local discovery patterns, but #74488 is only a discovery primitive and the complete solution needs an agreed provider hook, cache model, refresh trigger, and security boundary.
