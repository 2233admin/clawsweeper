## Summary

Keep open. Current main still lacks configured OpenAI-compatible `/v1/models` catalog refresh for the Control UI picker, and the same-author paired PR #74488 remains the active implementation discussion for the discovery primitive and API seam.

## What This Changes

Request an opt-in dynamic provider catalog refresh path that fetches configured OpenAI-compatible `/v1/models` endpoints, maps returned model metadata, caches by credential/base URL scope, and refreshes the Control UI model picker.

## Best Possible Solution

Land OpenAI-compatible discovery only after the provider-catalog contract is chosen, preferably by reusing the existing provider-owned catalog/refresh-cache model where possible, with credential/baseUrl-scoped caching, deterministic merge or replace semantics, Gateway and Control UI refresh behavior, focused tests, docs, and changelog coverage.

## Reproduction Assessment

Not applicable as a feature/API request. The absence is high-confidence from current-main source inspection: OpenAI remains static/forward-compatible only, while the configured model-list path has no OpenAI `/v1/models` refresh trigger.

## Solution Assessment

Unclear until maintainers choose the seam. The requested behavior is valid, but adding a brand-new hook may not be the best maintainable solution if the existing provider catalog discovery contract and explicit refresh-cache path can be extended safely.
