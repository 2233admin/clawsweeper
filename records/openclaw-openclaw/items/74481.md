## Summary

Keep open. Current main still has static OpenAI catalog rows and generic provider-catalog plumbing, but it does not implement configured OpenAI-compatible `/v1/models` discovery for the Control UI picker. The newly cross-referenced #74488 is a useful first-slice PR, but its own body says it is not wired into catalog resolution yet, so this issue remains the right design/API discussion point.

## What This Changes

Add an opt-in dynamic model-catalog refresh from configured provider `/v1/models` so the OpenAI-compatible Control UI picker reflects proxy/upstream model availability and metadata.

## Best Possible Solution

Settle the provider catalog contract first, then implement OpenAI-compatible discovery in the OpenAI plugin or approved shared seam with endpoint/auth-scoped caching, fail-soft fallback to static rows, explicit merge/replace semantics, Gateway/Control UI refresh behavior, tests, and docs.
