## Summary

Keep open. Current main still accepts bearer auth for HTTP Control UI reads, but it does not bridge a reverse-proxy-injected shared secret into the browser WebSocket connect path; shared-secret WebSocket auth is still sourced from `connect.params.auth`. The existing maintainer-side comments remain consistent with current code, and this is a Gateway/Control UI auth trust-boundary feature rather than conservative cleanup.

## What This Changes

The issue requests Control UI authentication bootstrap from reverse-proxy-provided `Authorization` or `X-OpenClaw-Token` headers instead of token-in-URL or manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap model for Control UI: either validate `Authorization`/`X-OpenClaw-Token` on the WebSocket upgrade after existing origin/proxy/rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session credential for WebSocket auth. Keep fragment/manual token flows and existing Tailscale/trusted-proxy identity modes backward compatible, with docs and focused Gateway/UI tests.
