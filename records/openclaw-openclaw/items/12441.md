## Summary

Keep open. Current main still has the reported split: Control UI HTTP reads can use bearer auth, but shared-secret WebSocket auth still comes from the browser connect frame or existing identity-header modes, so proxy-injected Authorization/X-OpenClaw-Token is not a complete Control UI bootstrap path. Because the requested change affects Gateway admin authentication semantics, it needs maintainer/secops design before implementation.

## What This Changes

Allow the Control UI to bootstrap Gateway shared-secret authentication from reverse-proxy-injected Authorization or X-OpenClaw-Token headers instead of requiring URL/manual token entry.

## Best Possible Solution

Define one explicit reverse-proxy shared-secret bootstrap contract for the Gateway admin Control UI: either validate Authorization/X-OpenClaw-Token during the WebSocket upgrade after trusted-proxy, origin, and rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser credential for the WebSocket. Preserve URL-fragment/manual token flows and existing Tailscale/trusted-proxy identity modes, then document and test the chosen model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure shared-secret gateway auth, put the Control UI behind a proxy that injects Authorization for the page/config request, leave UI token/password state empty, and observe that the WebSocket connect path still authenticates only connect.params.auth-derived credentials.

## Solution Assessment

Unclear as an implementation plan. The requested direction is valid, but the best fix requires maintainer/secops selection of the header trust boundary and credential-bridging model before code changes are safe.
