## Summary

Current main still does not implement the requested reverse-proxy shared-secret bootstrap for Control UI WebSocket auth. HTTP Control UI reads can use bearer auth, and trusted-proxy identity auth is a separate supported mode, but a proxy-injected Authorization or X-OpenClaw-Token shared secret is not bridged into the browser WebSocket connect flow.

## What This Changes

Request that Control UI accept a gateway token from reverse-proxy-supplied Authorization or X-OpenClaw-Token headers instead of requiring URL token bootstrap or manual UI entry.

## Best Possible Solution

Define one documented reverse-proxy shared-secret bootstrap model: either validate Authorization/X-OpenClaw-Token during the WebSocket upgrade after explicit origin, trusted-proxy, and rate-limit checks, or validate the header on a same-origin bootstrap endpoint and issue a short-lived browser credential for the WebSocket. Preserve URL-fragment/manual token flows plus existing Tailscale and trusted-proxy identity modes, then add focused Gateway/UI tests and docs.
