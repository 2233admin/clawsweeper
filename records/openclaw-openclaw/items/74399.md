## Summary

Keep open. Current main still starts the classic `openclaw onboard` wizard without an upfront ETA or long-running setup expectation, and the main CLI onboarding docs still do not warn that provider auth, channel setup, daemon install, skills, plugins, and network-bound work may take several minutes or longer. This is narrow, current, and suitable for a focused UX/docs repair.

## What This Changes

Request to show an upfront duration or long-running setup expectation during `openclaw onboard`.

## Best Possible Solution

Add a concise early note to the classic CLI onboarding flow before the main prompts begin, explaining that setup can take several minutes and may take longer for provider auth, channels, daemon install, skills/plugins, optional dependencies, and network conditions; mirror the expectation in the public CLI onboarding docs without changing runtime/provider behavior.

## Reproduction Assessment

Yes. Source review gives a high-confidence path: `openclaw onboard` enters `setupWizardCommand` -> `runInteractiveSetup` -> `runSetupWizard`, and the first visible wizard actions contain no ETA or long-running setup note before prompts begin.

## Solution Assessment

Yes. A small UX/docs change is the narrowest maintainable solution; conservative variable-duration wording is safer than a fixed 40-minute promise unless maintainers have timing data.

Fix the onboarding expectation gap for https://github.com/openclaw/openclaw/issues/74399. Add a concise early notice to the classic `openclaw onboard` CLI flow before the main prompts begin, explaining that setup can take several minutes and may take longer depending on provider auth, channel setup, daemon install, skills/plugins, optional dependencies, and network conditions. Do not hard-code a 40-minute promise unless maintainers explicitly provide timing data. Mirror the same expectation in the user-facing CLI onboarding docs, preferably `docs/start/wizard.md` and any CLI/reference page that describes `openclaw onboard`.
