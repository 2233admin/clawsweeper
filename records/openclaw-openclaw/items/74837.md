## Summary

Current main addresses the reported release behavior: the spawn path persists the requested child model as override fields before launch, and the current agent-command path validates stored overrides against the full model catalog before using them as the effective provider/model. The reported Venice override appears consistent with a `v2026.4.27` allowlist/catalog-scope edge, but the current main implementation no longer uses that scoped catalog path.

## What This Changes

The issue reports that `sessions_spawn(model="venice/...")` returns `modelApplied: true` while the child run still uses the orchestrator default model.

## Best Possible Solution

Keep the current main behavior, which fixes the runtime selection boundary generically for provider-specific subagent model overrides, and let it ship in the next release after `v2026.4.27`. No Venice-specific special case is needed.

## Reproduction Assessment

Yes. A high-confidence static reproduction path exists by comparing `v2026.4.27` to current main: the release validates stored child overrides through a scoped catalog path, while current main validates against the full catalog and then uses the stored provider/model for the child run.

## Solution Assessment

Yes. The current main approach fixes the generic stored-override selection path and catalog validation path, which is narrower and safer than special-casing Venice or changing the `modelApplied` response semantics alone.
