## Summary

Keep open. Current main still has phase-aware assistant text handling, but no explicit public/internal assistant audience contract in the shared signature parser, visible-text extraction, OpenResponses schema, replay conversion, UI projection, or message-end delivery boundary. The reporter's sanitized package-artifact evidence matches the current source-level gap, and the related leak/phase reports are adjacent open issues rather than a canonical replacement for this broader contract work.

## What This Changes

The issue requests an explicit public/internal assistant audience contract so internal assistant text is not selected for user-visible delivery while public assistant text remains eligible under delivery policy.

## Best Possible Solution

Define one assistant audience contract in the shared message/signature layer, carry it through provider stream parsing, replay conversion, transcript projection, OpenResponses surfaces, UI history extraction, and channel delivery, then enforce public-only selection at user-visible boundaries while preserving an explicit legacy phase-only compatibility path.

## Reproduction Assessment

Yes. A source-level synthetic assistant message with textSignature JSON containing v: 1, an id, phase: "final_answer", and audience: "internal" reproduces the gap because current parsing ignores audience and visible extraction selects the final_answer text.

## Solution Assessment

Unclear as an immediately safe automated patch, but the requested direction is sound. The narrowest maintainable solution is a shared contract plus end-to-end propagation, not another late delivery scrubber that only covers one channel path.
