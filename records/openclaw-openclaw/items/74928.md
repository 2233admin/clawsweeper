## Summary

Keep open. Current main and the latest shipped v2026.4.29 tag still have a phase-only assistant text contract and no public/internal assistant audience gate at shared visible-text extraction, OpenResponses schema, or message-end delivery, matching the reporter's sanitized package-artifact evidence.

## What This Changes

The issue requests an explicit assistant audience contract so public assistant text can be delivered while internal or invalid-audience assistant text and exact silent sentinels fail closed at visible delivery boundaries.

## Best Possible Solution

Define one versioned assistant audience contract (`public` and `internal`) in the shared message-content layer, propagate it through provider stream parsing, replay, transcript/history projection, OpenResponses, UI extraction, and channel delivery, then enforce public-only visible selection while keeping a deliberate legacy phase-only compatibility path.

## Reproduction Assessment

Yes. A synthetic assistant text block with `textSignature` JSON containing `v: 1`, an id, `phase: "final_answer"`, and `audience: "internal"` follows the current parser/extractor path because audience is ignored and final-answer text is selected as visible.

## Solution Assessment

Yes, the requested shared contract direction is the maintainable fix. A delivery-only sanitizer would leave replay, history, OpenResponses, and UI extraction with inconsistent semantics.
