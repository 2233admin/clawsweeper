## Summary

Keep open. Current main supports normal Feishu interactive button/card payloads, and recent native-delivery fallback copy helps some failed native approval routes, but Feishu document/comment targets still explicitly degrade interactive payloads to plain text without explaining why buttons are unavailable or where the user should finish approval. The Feishu docs also do not document this approval fallback difference.

## Best Possible Solution

Keep this open and land a narrow Feishu fallback UX fix: when approval-style interactive buttons degrade to text on Feishu document/comment surfaces, preserve the exact `/approve ...` command path and add explicit copy explaining that interactive buttons are unavailable in the current Feishu surface, whether same-chat text approval is valid, and when Web UI or terminal UI is the right fallback. Update focused Feishu/approval docs and regression tests without changing approval authorization semantics.
