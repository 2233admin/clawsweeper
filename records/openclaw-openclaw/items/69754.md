## Summary

Close as implemented on main. Current main now routes shared approval button payloads through Feishu native interactive cards instead of leaving normal Feishu approval prompts as ambiguous raw text, and the generic approval fallback paths/documentation explain when `/approve` text, Web UI, or terminal approval is the remaining path. The Feishu card fix shipped in v2026.4.27 via #72667.

## What This Changes

The issue requests clearer Feishu approval fallback guidance when interactive approval buttons are unavailable or a flow degrades to manual approval text.

## Best Possible Solution

Keep the shipped shared approval-payload plus Feishu native-card implementation as the baseline. Any future complaints should be narrowed to a specific Feishu surface that still cannot render cards, with that surface's fallback copy adjusted without adding Feishu-specific approval policy to core.

## Reproduction Assessment

Yes. The source-level path is clear: build a shared approval pending payload with interactive buttons, deliver it through Feishu sendPayload, and it becomes a native Feishu card; existing tests cover both the shared approval payload and Feishu card conversion.

## Solution Assessment

Yes. The current solution is the maintainable path because approval actions stay in the shared renderer while Feishu only owns native card presentation and documented text fallback behavior.
