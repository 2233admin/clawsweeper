## Summary

Keep #69754 open. Current main still does not give Feishu-specific approval fallback guidance when built-in approval prompts degrade to text: Feishu only exposes approval auth, not native/render approval behavior, and its outbound adapter only delegates plain text/media delivery. Core forwarded approvals do attach generic interactive buttons, but Feishu has no outbound sendPayload/renderPresentation hook to consume them, so the user can still see text fallback with raw /approve commands rather than clear Feishu-specific explanation.

## Best Possible Solution

Keep the issue open and fix it in the Feishu bundled plugin plus shared approval rendering path. The likely best path is to add Feishu approvalCapability.render and/or nativeRuntime support that turns built-in approval payloads into Feishu structured interactive cards with ocf1 callbacks; when buttons cannot be used in a specific context, the fallback text should explicitly say Feishu buttons are unavailable here, whether /approve can be sent in-channel, and which Web UI or terminal path remains valid. Update docs/channels/feishu.md to distinguish message-tool presentation cards from built-in approval UX.
