## Summary

Keep open. Current main still appears to route a plugin-bound `inbound_claim` handled result back as `queuedFinal: false`, Telegram still uses that false final signal plus no delivery to send the silent-reply fallback, and the issue is paired with open unmerged PR #70288, so it should not be closed before that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to prevent Telegram's silent-reply fallback from firing when a plugin claims an inbound message with `{ handled: true }` and responds asynchronously.

## Best Possible Solution

Land PR #70288 after updating it, or land an equivalent maintainer patch, so a successful plugin-owned async inbound claim produces a durable response-committed signal through dispatch finalization to Telegram while preserving missing/no-handler/declined/error fallback behavior. Add focused regression coverage for the direct dispatch result, the wrapper finalization path, and Telegram's silent fallback condition.

## Reproduction Assessment

Yes. A source-level reproduction path is clear: a plugin-owned binding whose `inbound_claim` returns `{ handled: true }` yields `queuedFinal: false` with no immediate delivery, satisfying Telegram's silent fallback condition before the plugin's asynchronous response arrives.

## Solution Assessment

No, current main is not the best solved state because it still preserves the false final signal. The proposed fix direction is plausible, but it should include the wrapper/finalization contract so a zero-count async plugin claim is not rewritten back into a fallback-triggering state.
