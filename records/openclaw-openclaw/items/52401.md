## Summary

Keep this PR open. Current main still lacks an `agent_error` hook or equivalent lifecycle-error replacement seam, but the proposed core plugin API change still has merge-blocking contract issues around conversation-access policy and per-handler hook timeouts.

## What This Changes

The PR adds an `agent_error` plugin hook, invokes it from embedded Pi agent lifecycle error handling with a 2s outer timeout, adds lifecycle tests, and adds a changelog entry.

## Best Possible Solution

Keep the PR active as a core plugin API proposal, but require the new hook to follow the existing conversation-access and timeout contracts, document the processed/redacted error semantics, add loader and runner contract coverage, and fix changelog attribution before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P2] Gate agent_error behind conversation access:** `src/plugins/hook-types.ts:68`
  - body: `agent_error` observes lifecycle/provider error text but is added as an ordinary public hook. Because it is not in `CONVERSATION_HOOK_NAMES`, non-bundled plugins can register it without `plugins.entries.<id>.hooks.allowConversationAccess=true`; add it to the guarded list and cover the blocked/allowed registration paths before shipping this hook.
  - confidence: 0.91
- **[P2] Apply per-handler timeoutMs in agent_error:** `src/plugins/hooks.ts:690-695`
  - body: This loop awaits each `hook.handler` directly, so `api.on("agent_error", handler, { timeoutMs })` is ignored. A slow first handler can consume the entire outer lifecycle timeout and starve later replacement handlers, which violates the documented per-hook timeout contract.
  - confidence: 0.87

  - body: The added user-facing changelog bullet has no `Thanks @...` credit. Repo changelog rules require each new entry to include at least one credited GitHub username, so this needs attribution before merge.
