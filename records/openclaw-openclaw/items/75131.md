## Summary

Keep open. Current main still leaves the core bug cluster unresolved: permanent Telegram size/auth errors are not classified as permanent, queue enqueue still creates a new random queue id for each enqueue rather than a stable logical-send identity, and Telegram message actions can still use the direct default send path that does not pre-split long text.

## What This Changes

The issue requests fixing Telegram delivery queue retry idempotency, permanent failure classification for unrecoverable Telegram 400/403 errors, and long-message splitting at Telegram's 4096-character limit.

## Best Possible Solution

Land one focused reliability fix: add a stable logical-send identity at the queue boundary, classify unrecoverable Telegram 400/403 delivery errors before retrying, and route Telegram action/default long text through the same chunk-aware path used by normal outbound Telegram replies.

## Reproduction Assessment

Yes. Current main has a high-confidence source-level reproduction path: the permanent matcher returns false for the reported Telegram size/403 strings, enqueueDelivery always generates a new UUID, and Telegram action sends call the non-chunked default send branch for long text.

## Solution Assessment

Yes. The issue's requested direction is the narrow maintainable fix if implemented through existing seams: queue storage/recovery for retry identity and classification, and Telegram outbound/action send paths for chunk planning rather than adding a separate truncation-only workaround.
