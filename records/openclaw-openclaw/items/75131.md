## Summary

Keep open. Current main still leaves the core reported Telegram delivery defects unresolved: permanent error classification misses the reported Telegram size/auth variants, delivery queue enqueue still creates a fresh UUID per enqueue without a stable logical-send key, and Telegram action/default send paths can bypass the existing chunk-aware outbound adapter.

## What This Changes

The issue requests a Telegram delivery fix that makes send retries queue-idempotent, treats unretryable Telegram 400/403 delivery errors as permanent failures, and pre-splits outbound Telegram text at the platform limit.

## Best Possible Solution

Land one focused fix that adds a stable logical-send identity at the delivery queue boundary, classifies known unretryable Telegram delivery errors as permanent before retry backoff, and routes Telegram action/default long text through the existing chunk-aware Telegram outbound behavior instead of adding a parallel truncation path.

## Reproduction Assessment

Yes. A high-confidence reproduction can be written as focused unit tests: `isPermanentDeliveryError('400: Bad Request: message is too long')` currently misses the permanent matcher by inspection, repeated enqueue calls still produce separate UUIDs, and a Telegram action/default send above 4096 characters can be mocked to show one unsplit send.

## Solution Assessment

Yes. The requested direction is the narrow maintainable solution if implemented through the existing delivery queue storage/recovery and Telegram outbound adapter seams; a safer alternative to truncation is to reuse the existing chunk planner so user-visible content is preserved.
