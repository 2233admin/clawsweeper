## Summary

Keep open. Current main has some delivery-queue safeguards and Telegram chunking for the normal outbound adapter, but it still misses the core reported gaps: the permanent-error classifier does not cover the named Telegram oversize/403 failures, queue entries are still created with a fresh UUID per enqueue, and the Telegram message action can still bypass the chunk-aware outbound adapter path.

## What This Changes

The issue requests idempotent delivery-queue retries, permanent failure handling for Telegram oversize and other non-transient 400/403 errors, and pre-splitting Telegram send-message content at the Bot API text limit.

## Best Possible Solution

Land one focused fix that gives delivery-queue entries a stable logical-send identity or upsert path for retries, expands permanent Telegram error classification for non-transient oversize/auth/routing failures, and routes Telegram message-action sends through the same chunk-aware delivery behavior used by normal outbound Telegram replies.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the permanent-error and long-message action paths: a recovery test with `400: Bad Request: message is too long` should remain pending today, and a Telegram `sendMessage` action over 4096 characters reaches the default direct send branch without chunk planning. The duplicate-UUID path is supported by the incident evidence and queue storage code, but still needs a regression test for the initiating retry path.

## Solution Assessment

Is this the best way to solve the issue? Yes. The narrow maintainable fix is to keep queue idempotency in the queue/retry boundary, extend the existing permanent-error classifier, and reuse Telegram's existing chunking machinery instead of adding a separate truncation-only action path.
