## Summary

Keep open. Current main still leaves the reported bug cluster materially unresolved: queue enqueue uses a fresh storage UUID per live send attempt, recovery still treats the reported Telegram size/403 failures as retryable, and Telegram message actions still call the direct default send path that builds one non-html text chunk for long content.

## What This Changes

The issue requests idempotent delivery-queue retries, permanent-failure classification for Telegram long/auth/routing errors, and Telegram send-message pre-splitting at the 4096-character Bot API limit.

## Best Possible Solution

Land one focused fix that carries a stable logical-send identity into queue storage, extends permanent delivery-error classification for Telegram size/auth/routing failures, and makes Telegram message actions use the same chunk-aware text delivery behavior as normal outbound replies.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the recovery and long-message paths: a recovery regression with `400: Bad Request: message is too long` should remain pending today, and a Telegram `sendMessage` action above 4096 characters follows the direct default send path. The duplicate-UUID incident is strongly supported by the report and source shape, but the exact higher-level retry trigger still needs a focused regression test.

## Solution Assessment

Is this the best way to solve the issue? Yes, if kept narrow: fix queue idempotency at the enqueue/retry boundary, reuse the existing permanent-error policy, and route Telegram action sends through existing chunk planning rather than adding an unrelated truncation path.
