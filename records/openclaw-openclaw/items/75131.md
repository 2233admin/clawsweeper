## Summary

Current main still leaves the reported Telegram delivery reliability cluster unresolved: queue entries are created with a fresh UUID for each enqueue, the permanent-error matcher does not cover the reported Telegram length/deactivation/bot-target failures, and Telegram action sends can still use the default non-chunked send path.

## What This Changes

Request to make Telegram outbound delivery retries idempotent, permanently fail unrecoverable Telegram 400/403 errors, and pre-split long Telegram send-message payloads at the platform limit.

## Best Possible Solution

Land one focused reliability fix through existing seams: preserve a stable queue identity for a logical delivery attempt, classify the specific unrecoverable Telegram size/auth/target errors as permanent before retry, and route Telegram action/default long text through the same chunk-aware send path used by normal outbound Telegram replies.

## Reproduction Assessment

Yes. Current main provides a high-confidence source-level reproduction path: repeated deliverOutboundPayloads calls enqueue fresh UUIDs, the permanent matcher returns false for the reported Telegram errors, and Telegram sendMessage actions do not opt into the chunked html send path.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix if implemented in the delivery queue/recovery policy and Telegram outbound/action send seams; a truncation-only workaround would leave retry duplication and stale poisoned entries unresolved.
