## Summary

Keep open. Current main still leaves meaningful parts of the reported Telegram delivery failure cluster unresolved: recovery does not classify the reported size/auth 400/403 failures as permanent, message-tool Telegram sends bypass the normal chunk-aware outbound adapter, and queue enqueue still creates a fresh UUID for every enqueue without a stable logical-send key.

## What This Changes

The issue requests idempotent delivery-queue retries, permanent-failure handling for Telegram non-transient 400/403 send errors, and pre-splitting Telegram send-message content at the 4096-character limit.

## Best Possible Solution

Land one focused reliability fix that gives delivery queue entries a stable logical-send identity across retries, expands permanent Telegram failure classification for non-transient 400/403 responses, and routes Telegram message-tool text through the same chunk-aware send path used by normal outbound replies.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes for the permanent-failure and long-message paths: a queued Telegram delivery failing with 400 message is too long will miss the current permanent classifier, and a Telegram message action with over-4096-character text follows the direct send path. The duplicate UUID path is strongly evidenced by the incident but needs a new regression test to pin the exact retry trigger.

## Solution Assessment

Is this the best way to solve the issue? Yes. The maintainable direction is to fix queue idempotency at the enqueue/retry boundary, extend the existing permanent-error classifier, and reuse Telegram's existing chunking seam instead of adding a separate truncation-only path.
