## Summary

Keep open. Current main still builds OpenAI-compatible Chat Completions payloads from the upstream converter without an outgoing strict message-key policy; existing compatibility hooks only flatten pure text content arrays and adjust unrelated request fields. The KeepMy Claw workaround mentioned in the discussion is useful external routing context, but it does not resolve the in-repo provider compatibility gap.

## What This Changes

The issue requests sanitizing outgoing `openai-completions` message objects so strict OpenAI-compatible providers do not reject extra keys such as tool-call and provider-specific fields.

## Best Possible Solution

Add an explicit provider/model-level OpenAI-compatible Chat Completions serialization policy that runs after message conversion and before dispatch in both direct and embedded payload paths, preserves stored transcripts, and documents how it interacts with `supportsTools`, `requiresStringContent`, and historical tool-call replay.
