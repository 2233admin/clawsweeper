## Summary

Keep open. Current main still routes OpenAI-compatible Chat Completions payloads through the upstream converter and existing compatibility wrappers without a provider/model policy to strip message object keys for strict role/content-only backends. The KeepMy Claw gateway mentioned in the discussion is useful external routing context, but it does not fix OpenClaw's direct custom-provider path.

## What This Changes

For `openai-completions` providers, outgoing Chat Completions messages should be sanitized with a provider/model policy that removes unsupported extra message keys before dispatch.

## Best Possible Solution

Add an explicit OpenAI-compatible Chat Completions message serialization policy at the provider/model compat layer, apply it after message conversion and before dispatch in both direct and embedded payload paths, preserve stored transcripts, and document how it interacts with `supportsTools` and `requiresStringContent`.

## Repair Work Prompt

Fix the strict OpenAI-compatible provider message-key bug tracked by https://github.com/openclaw/openclaw/issues/50374, with related context in https://github.com/openclaw/openclaw/issues/45706. Add a provider/model-level compat policy for `openai-completions` Chat Completions message serialization that can strip outbound message objects down to an explicit allowed key set for strict providers. Apply it after `convertMessages` and before dispatch in the direct OpenAI completions transport, and also in the embedded stream payload wrapper path so direct and embedded runs behave consistently. Preserve stored transcripts and existing default behavior for tool-capable OpenAI-compatible providers; avoid a blanket strip-all default that would break providers accepting `tool_calls`/`tool_call_id`. Document the new compat/config field and its interaction with `supportsTools` and `requiresStringContent`. Add tests covering assistant tool-call history and tool-result history against a strict role/content-only policy, plus config schema acceptance/rejection. Do not change dependency versions, provider auth, unrelated replay repair policy, or broad transcript storage behavior. Add a changelog fix entry with reporter credit if repo policy requires it for this user-facing provider compatibility fix.
