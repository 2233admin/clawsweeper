## Summary

Current main still lacks a strict outgoing message-key sanitizer for OpenAI-compatible Chat Completions providers. The code has related mitigations for string-only content, tool-call id hygiene, turn ordering, and tool-schema pressure, but nothing strips `messages[]` objects down to a provider-accepted key set for Infomaniak-style endpoints. The KeepMy Claw workaround mentioned in the discussion is useful routing advice, not an OpenClaw implementation.

## Best Possible Solution

Keep this issue open and implement a narrow, documented strict OpenAI-compatible message serialization policy. Prefer an opt-in model/provider compat control or provider-owned hook that sanitizes outgoing `openai-completions` `messages` after conversion and before dispatch in both the OpenClaw transport and embedded wrapper paths, with tests for prior tool-call/tool-result history and docs explaining how it interacts with `requiresStringContent` and `supportsTools`.
