## Summary

This issue should remain open. Current main still treats Signal `editMessage` envelopes as ordinary inbound messages: the edit envelope type does not model `targetSentTimestamp`, the handler flattens `editMessage.dataMessage` into the normal message path, queued message identity still comes from the new envelope timestamp, and there is no Signal edit-aware transcript replacement, dedupe, marker, or reply-suppression behavior. Related PR #9701 is closed and unmerged, so it is useful history but not shipped behavior.

## Best Possible Solution

Keep this issue open and implement the behavior in the bundled Signal plugin. The fix should type `editMessage.targetSentTimestamp`, route edit envelopes through an explicit Signal edit path, preserve an `[edited]` marker or structured edit metadata in context, use the target timestamp for dedupe or transcript replacement, suppress automatic replies for non-mentioned group edits, and add focused Signal tests for edit envelopes, message identity, context history, and mention-gated behavior. PR #9701 is useful prior art but should not be treated as shipped.
