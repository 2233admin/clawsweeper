## Summary

Keep open. The performance goal is valid, but this patch is not ready for current main: the active session parser and sync path now live under the memory-host SDK and memory-core plugin, current main still full-parses selected session files, and the PR diff has concrete correctness and retention blockers.

## What This Changes

The PR adds a previous-entry append-only fast path to session JSONL parsing, caches parsed session entries during memory sync, and adds an append-only regression test under the old `src/memory` implementation.

## Best Possible Solution

Rebase or replace the optimization against `packages/memory-host-sdk/src/host/session-files.ts` and `extensions/memory-core/src/memory/manager-sync-ops.ts`, using byte-level prefix validation, one-read safe fallback to the existing full parser, bounded cache lifecycle, and regression coverage for ASCII append, non-ASCII append, rewritten-prefix fallback, line maps, timestamps, redaction, and transcript classification.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Verify the cached prefix before append reuse:** `src/memory/session-files.ts:125-130`
  - body: The fast path only checks that the file is at least as large as the cached entry before reusing `previous.content`. If an earlier JSONL record is rewritten or compacted and the file remains the same size or grows, the returned entry can keep stale historical content; validate the cached prefix byte-for-byte or fall back to full parsing.
  - confidence: 0.91
- **[P1] Slice append data by byte offset:** `src/memory/session-files.ts:127-129`
  - body: `previous.size` comes from `fs.stat` in bytes, but the patch uses it as a JavaScript string index for `raw.slice(...)`. Non-ASCII transcripts can put the append boundary at the wrong character, dropping or duplicating parsed messages and producing an incorrect `lineMap`.
  - confidence: 0.94
- **[P1] Bound the session entry cache:** `src/memory/manager-sync-ops.ts:149`
  - body: The new `sessionEntryCache` stores full `SessionFileEntry` objects, including transcript `content`, and has no eviction or cleanup path when sessions are pruned or indexes reset. A long-lived gateway can retain transcript strings indefinitely and grow heap usage over time.
  - confidence: 0.88

## Security Review
