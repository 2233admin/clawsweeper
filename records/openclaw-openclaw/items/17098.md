## Summary

Keep open. Current main still lacks xCloud coverage, but the proposed official guide is not ready as-is because it contains a provider-credential error and documents a hosted Control UI path without enough auth, origin, and device-pairing detail for current Gateway behavior.

## What This Changes

The PR adds a new xCloud install guide plus xCloud redirects, docs navigation, and a VPS provider-card entry.

## Best Possible Solution

Publish an xCloud guide only after a maintainer or xCloud/provider owner verifies the managed deployment against current OpenClaw Gateway and Control UI contracts, then revise the page to document the exact auth mode, origin allowlist, device-pairing or trusted-proxy path, remote-access posture, Telegram allowlist behavior, and provider credential handling.

## Reproduction Assessment

Not applicable as a docs feature PR rather than a runtime bug. The high-confidence verification path is source/docs inspection: current main lacks xCloud docs, while the PR head adds the page and exposes the provider wording and hosted-Control-UI gaps.

## Solution Assessment

No, the PR is not the best way to solve the docs gap as-is. The docs surface is reasonable, but the content needs provider wording fixes and explicit, verified hosted Control UI auth/origin/device-pairing details before it becomes official documentation.
