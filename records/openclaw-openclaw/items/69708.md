## Summary

Keep open. Current main still rejects the reported `aws-sdk` auth-profile marker at store load, while the Bedrock docs and resolver treat `aws-sdk` as a valid runtime auth mode; the related fix PR was closed unmerged, so the issue remains actionable.

## What This Changes

The issue asks OpenClaw to stop treating Bedrock `auth-profiles.json` entries with `type: "aws-sdk"` as invalid so EC2 IMDS credentials continue through the AWS SDK default chain.

## Best Possible Solution

Add a narrow auth-store compatibility path for SDK-managed marker types such as `aws-sdk`, leaving Bedrock request signing on the existing provider-level AWS SDK runtime path and covering both clean marker handling and mixed real-invalid-entry warnings with regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is an `auth-profiles.json` profile entry whose normalized `type` is `aws-sdk`; current main falls through the accepted-type set and records `invalid_type`, matching the reported warning and dropped profile.

## Solution Assessment

Yes. The narrowest maintainable direction is to treat `aws-sdk` as a non-secret SDK-managed marker during auth-store load, rather than broadening `AuthProfileCredential` into a real credential variant and touching every downstream credential switch.
