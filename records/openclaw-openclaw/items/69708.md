## Summary

Keep open. Current main still rejects stored auth-profile credentials whose `type` is `aws-sdk`, so the reported `invalid_type` drop is still present even though the provider-level Bedrock `auth: "aws-sdk"` path is implemented. The linked fix PR was closed unmerged, and there is no current regression coverage for SDK-managed auth-profile markers.

## What This Changes

The issue asks OpenClaw to accept or safely ignore `type: "aws-sdk"` entries in `auth-profiles.json` for Amazon Bedrock so EC2 IMDS/default AWS SDK credentials are not dropped as invalid.

## Best Possible Solution

Add a narrow, documented auth-profile store behavior for SDK-managed marker types: `aws-sdk` should not be reported as a genuine invalid credential, real invalid entries should still warn, and Bedrock runtime auth should continue to flow through provider config and the AWS SDK default chain. The fix should include regression coverage for a canonical `auth-profiles.json` entry containing `amazon-bedrock:default` with `type: "aws-sdk"`.

## Reproduction Assessment

Yes. A canonical auth-profile entry with `type: "aws-sdk"` reaches `parseCredentialEntry`, misses the three allowed credential types, and is rejected as `invalid_type`; the load path then drops it and logs the warning described in the report.

## Solution Assessment

Unclear for the exact implementation shape. The narrow skip-marker approach from PR #69753 looks plausible because the marker has no secret material, but maintainers should confirm whether the durable contract should skip it, migrate it, or represent it as a typed non-secret marker.
