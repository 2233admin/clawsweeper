## Summary

Current main still has the Unix `lsof -Fpc` command-name filter that this PR is trying to replace, and `pollPortOnce` still parses the same filtered lsof output. The PR remains a relevant implementation candidate for the paired macOS launchd/stale-gateway bug, but it should stay open for normal review because the PR head still has review comments about unbounded poll-path `ps` timeouts.

## Best Possible Solution

Keep this PR open as the active implementation candidate for the paired macOS stale-gateway cleanup bug. The best next step is to update the PR so Unix discovery parses listening PIDs, verifies gateway identity via argv with bounded `ps` timeouts in both initial scan and polling paths, keeps the self/ancestor exclusion invariant, avoids unnecessary lsof command-field dependence, and preserves focused regression coverage for macOS `cnode` output, unrelated listeners, ancestor exclusion, and poll timeout behavior.

## Risks / Open Questions

- Merging the PR as currently reviewed could let a slow or hung `ps` verifier call inside `pollPortOnce` consume the full 2000 ms port-free budget unless both verifier calls use `POLL_SPAWN_TIMEOUT_MS`.
- Because this code decides which processes may be signaled during stale gateway cleanup, the Unix argv verifier should stay tightly bounded to real OpenClaw gateway argv patterns and avoid false positives.
- Until this PR or an equivalent fix lands, current main still appears able to miss macOS gateway PIDs when `lsof` reports `cnode` while argv/process title identifies the process as `openclaw-gateway`.
