## Summary

Keep this PR open. Current main still does not implement the PR's central behavior: Feishu sender-name resolution receives only the historical `senderOpenId` value, so events carrying both `sender_id.user_id` and `sender_id.open_id` still look up the open_id path first. The PR remains a focused bundled Feishu plugin fix candidate, with review history already identifying cache/fallback details that should be checked during rebase.

## Best Possible Solution

Keep this PR open and review it as a focused Feishu bundled-plugin fix. The best path is to rebase it onto current main, add `senderUserId` plumbing to the Feishu message context, update `extensions/feishu/src/bot-sender-name.ts` to try lookup candidates in `user_id`, `open_id`, then fallback order, preserve best-effort failure handling, and keep regression tests for priority, fallback, cache ordering, and cache-poisoning behavior.
