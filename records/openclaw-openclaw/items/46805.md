## Summary

Keep open. The PR addresses a real Feishu sender-attribution gap on current main, but the branch is not merge-correct as-is because current main resolves names through the split `bot-sender-name.ts` helper while the PR changes the older in-file resolver shape in `bot.ts`.

## What This Changes

The PR carries `senderUserId` through Feishu message context, changes sender display-name lookup to prefer `user_id` before `open_id` with fallback/cache handling, and adds Feishu regression tests.

## Best Possible Solution

Port the ordered lookup behavior onto the current split resolver: add an optional `senderUserId` to the Feishu message context, pass both IDs from parsing into `handleFeishuMessage`, implement per-candidate cache and API lookup in `bot-sender-name.ts`, and keep focused regression tests for priority, fallback, cache, and permission-error behavior.

## Reproduction Assessment

Yes. A static current-main reproduction is enough: use an inbound Feishu event with both `sender_id.user_id` and `sender_id.open_id`; the parser sees both, but the handler passes only `ctx.senderOpenId` into the single-ID resolver, so `user_id` cannot be attempted first.

## Solution Assessment

No. The direction is sound, but this branch is no longer the best landing shape because current main moved sender-name resolution into `bot-sender-name.ts`; the maintainable fix is to adapt the behavior to that resolver boundary.
