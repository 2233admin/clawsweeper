## Summary

Keep open. Current main still resolves Feishu sender names through a single `senderOpenId` lookup even when events carry `sender_id.user_id`, so the user problem remains. The PR is useful, but it needs rebase/adaptation because main has moved sender-name resolution into `extensions/feishu/src/bot-sender-name.ts` while the PR patch still targets the old in-file resolver in `bot.ts`.

## What This Changes

The PR updates the Feishu plugin to carry `senderUserId`, try user_id before open_id/fallback IDs for sender-name lookup, and add Feishu sender-name priority/cache tests.

## Best Possible Solution

Adapt the PR onto current main by adding `senderUserId` to the Feishu message context, moving the ordered candidate lookup into `extensions/feishu/src/bot-sender-name.ts`, and keeping the cache priority, fallback-after-error, client-construction, cross-ID cache, and cache-poisoning regression coverage focused in Feishu plugin tests. No core/plugin SDK change appears necessary.
