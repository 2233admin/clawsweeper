## Summary

Keep this PR open. Current main still does not implement the PR’s central behavior: Feishu sender-name resolution still receives and resolves only one lookup id, which is `senderOpenId` when both `sender_id.open_id` and `sender_id.user_id` are present. The PR remains a plausible bundled Feishu plugin fix candidate, with review history already identifying cache and fallback details that should be preserved during rebase.

## Best Possible Solution

Keep this PR open and review it as a focused Feishu bundled-plugin fix. The best path is to rebase onto current main, add `senderUserId` plumbing to `FeishuMessageContext`, update `extensions/feishu/src/bot-sender-name.ts` to try lookup candidates in `user_id`, `open_id`, then historical fallback order, and keep regression tests for priority, fallback, cache ordering, and cache-poisoning behavior.
