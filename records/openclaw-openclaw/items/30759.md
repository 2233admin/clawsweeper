## Summary

Keep open. Current main now supports selecting and transporting non-video WebChat/Control UI files, including PDF/TXT/DOC/DOCX-shaped uploads, but the requested document text extraction is still incomplete: default extraction supports text-like files and PDF only, the bundled document-extract plugin advertises only PDF, and the media-understanding file-block path skips OOXML/vendor Office MIME types before extraction. Related #57612 is the clearest DOCX extraction candidate, while legacy .doc still needs explicit maintainer policy/security review.

## What This Changes

Add WebChat support for selecting PDF, Word, and TXT documents, extracting their text, and sending that content to the assistant.

## Best Possible Solution

Keep this open and narrow the remaining work to document extraction policy: review or adapt #57612 for native DOCX extraction, wire DOCX through the WebChat/media-understanding file-block path if maintainers want automatic extraction from uploads, explicitly reject or defer legacy .doc unless a safe extractor is approved, and align tests/docs across Control UI uploads, Gateway staging, input-file extraction, media-understanding, and the document-extract plugin contract.

## Risks / Open Questions

- Legacy .doc extraction is security-sensitive because binary Word files should not be coerced through text heuristics or an unvetted parser.
- DOCX extraction should be wired through the existing document-extractor/plugin or media-understanding contract rather than bypassing plugin boundaries in core.
- Current upload transport makes Office files reachable as managed media, so an incomplete extraction path can look like support while the assistant still does not receive usable document text.
