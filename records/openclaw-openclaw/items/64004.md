## Summary

Keep open. Current main still exposes the same unresolved Control UI latency surface: Sessions navigation refreshes through `sessions.list`, Overview still overlaps several secondary refreshes, and the slow endpoints named in the report remain active request paths without a landed click-to-render instrumentation or latency fix.

## What This Changes

The issue requests that Control UI Sessions navigation update promptly on Windows when `sessions.list` returns quickly, instead of appearing stalled while other Control UI RPCs are slow.

## Best Possible Solution

Keep this as a concrete Control UI performance bug until a landed change measures the full click-to-render path and prevents slow secondary panels from blocking or obscuring visible navigation feedback. The end state should include per-action timing attribution plus lazy, cancellable, stale-guarded, or paged refreshes around the slow `sessions.usage`, `usage.cost`, `node.list`, and `cron.runs` surfaces.
