## Summary

Keep open. This PR is a useful start for the linked Moonshot alias bug, but it only adds catalog-planning metadata; current runtime provider normalization still leaves `moonshotai` as an unmatched provider, so the reported direct `moonshotai/kimi-k2.6` path can still fail with `Unknown model`.

## What This Changes

The PR adds a `moonshotai -> moonshot` `modelCatalog.aliases` entry to the Moonshot plugin manifest and a manifest-planner test for provider-filtered alias catalog rows.

## Best Possible Solution

Amend this PR, or replace it narrowly, so `moonshotai/<model>` and likely `moonshot-ai/<model>` canonicalize to the existing `moonshot` provider on the shared runtime provider-id path. Keep `moonshot` as the canonical plugin/provider/auth/config id, preserve OpenRouter nested refs, retain the catalog alias only if maintainers want alias-filtered model listings, and add focused runtime plus catalog regression coverage and a changelog entry.

## Reproduction Assessment

Yes. The linked issue gives concrete steps, and the current source confirms the path: `moonshotai/kimi-k2.6` parses to provider `moonshotai`, `normalizeProviderId` does not map it to `moonshot`, and runtime resolution continues under that unmatched provider id.

## Solution Assessment

No. The PR solves only the manifest catalog-planner slice; the narrow maintainable fix is runtime provider-id alias normalization with tests that exercise direct model-ref parsing or `resolveModelWithRegistry`.
