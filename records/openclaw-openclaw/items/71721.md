## Summary

Keep open. Current main has shipped Bonjour-specific handling for the reported ciao `uv_interface_addresses` failure, but gateway LAN and tailnet helpers still propagate throwing `os.networkInterfaces()` calls, and PR #71950 remains the open closing-syntax implementation candidate for the remaining gateway-critical paths.

## What This Changes

The issue reports that `os.networkInterfaces()` throws `uv_interface_addresses returned Unknown system error 1` in a Nemoclaw/OpenShell sandbox, crashing the gateway and disconnecting the TUI.

## Best Possible Solution

Keep the shipped Bonjour hardening, then review and either land PR #71950 or an equivalent narrow fix so startup-critical, best-effort LAN and tailnet discovery paths use `safeNetworkInterfaces()` while preserving the raw reader for callers that intentionally need error propagation.

## Reproduction Assessment

Yes. A high-confidence unit reproduction exists by mocking `os.networkInterfaces()` to throw and calling `pickPrimaryLanIPv4()` or `listTailnetAddresses()`; current tests still assert those helpers throw, while Bonjour tests cover the exact ciao SystemError classification.

## Solution Assessment

Yes. The best direction is the narrow linked PR approach plus the existing Bonjour-specific mitigation: harden best-effort gateway/tailnet discovery callers without changing the global `readNetworkInterfaces()` contract for all consumers.
