## Summary

Keep open. Current main still turns each accepted Android `notifications.changed` node event into an immediate heartbeat wake, and the only linked implementation candidate is still open and unmerged, so the issue is not implemented or superseded yet.

## What This Changes

The issue asks for Android companion notification events to be queued as heartbeat context instead of triggering one agent run per notification event.

## Best Possible Solution

Land a clean, narrow gateway fix that keeps Android notification forwarding as passive heartbeat context by default, preserves session scoping and untrusted-system-event handling, and adds focused tests proving notification events do not trigger one run per event. If the linked fork PR cannot be cleaned up, close it and replace it with an equivalent small PR.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to pass a valid `notifications.changed` payload with `change` and `key` through `handleNodeEvent`; current main enqueues the system event and calls `requestHeartbeatNow`, and existing tests assert that behavior.

## Solution Assessment

Unclear for the linked PR as submitted. The described direction is likely the narrow maintainable fix, but the actual fork PR appears broad and dirty, so the best path is a clean focused PR or a carefully reduced version of that change.
