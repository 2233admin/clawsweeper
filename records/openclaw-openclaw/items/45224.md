## Summary

Keep open. Current main still has the reported crash class: OpenClaw has a plugin-owned unhandled-rejection seam, but the browser plugin only consumes the Playwright dialog-race shape, while a `CRSession._onMessage` / `CRConnection._onMessage` assertion remains an unclassified generic rejection that reaches the Gateway fatal fallback.

## What This Changes

The issue requests that Playwright CDP session assertion failures from browser control be isolated so they do not crash the Gateway process or unrelated agent sessions.

## Best Possible Solution

Add a browser-owned stability boundary for Playwright CDP session assertions: classify direct and wrapped `Assertion error` rejections with `CRSession._onMessage` / `CRConnection._onMessage` stack markers, log them as browser/CDP degradation, evict or reset the affected Playwright connection when possible, and keep the core fatal fallback for unknown unhandled rejections.

## Reproduction Assessment

Yes for a focused unit-style reproduction: synthesize an unhandled rejection with `Assertion error` and CRSession/CRConnection stack markers while the browser runtime handler is registered, then verify current main leaves it unhandled. A live Chrome trigger is less deterministic because it depends on receiving an unexpected CDP message.

## Solution Assessment

Yes. The best near-term fix is a narrow browser-plugin classifier and recovery path using the existing runtime unhandled-rejection seam; a blanket global `Assertion error` suppression or broad Playwright catch-all would risk masking unrelated dependency failures.
