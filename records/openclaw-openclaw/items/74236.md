## Summary

Keep this PR open. Current main still has the targeted hot path: `loadModelCatalog` calls `shouldSuppressBuiltInModel` inside the model registry loop, and that path reloads/plans manifest model suppressions per lookup. The PR is a plausible focused performance fix, but the proposed cached resolver needs review/repair because the diff appears to bypass the existing provider alias canonicalization before manifest suppression lookup.

Keep the PR open and repair/review the cached resolver approach: compute manifest suppressions once per `loadModelCatalog` call, preserve the existing `shouldSuppressBuiltInModel` semantics including provider alias normalization and conditional baseUrl/config matching, add focused regression tests for both caching and alias behavior, then run the targeted agent/plugin tests plus build or changed-gate validation before merge.
