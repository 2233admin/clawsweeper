## Summary

Keep open. Current main still has a pre-bind Gateway startup path where a present `tools.web.fetch` object disables the secrets fast path and reaches web-fetch provider discovery before the HTTP listener binds; there are also two open PRs that explicitly claim to close this issue, so the issue should remain open until maintainers land, replace, or close those fixes.

## What This Changes

The issue reports a regression where Gateway startup hangs before binding its port when `tools.web.fetch` is present in config, including the expected `enabled:false` escape hatch.

## Best Possible Solution

Keep the issue open and review a canonical fix that prevents optional or disabled web-fetch configuration from blocking Gateway bind. The final patch should either skip credential-free disabled surfaces entirely or defer nonessential provider discovery until after the listener is live, while preserving fail-fast behavior for active provider SecretRefs.

## Reproduction Assessment

Yes. The issue gives a concrete config, command, trace boundary, version range, and removal workaround, and source inspection shows the matching pre-bind path through runtime web-tool discovery.

## Solution Assessment

Unclear. The open PR direction is plausible for the `enabled:false` escape hatch, but the full issue also reports `enabled:true`; the best maintainable solution should be validated against both shapes before this issue closes.
