## Summary

Keep open. The issue is fresh, concrete, and current main still has a plausible pre-bind startup path where `tools.web.fetch` forces runtime web-tool provider resolution during `config.auth`; I found no code, test, changelog entry, or post-release commit proving that this specific `tools.web.fetch` startup hang is fixed.

## What This Changes

The issue reports that Gateway startup should still bind its local port when a `tools.web.fetch` config object is present, including when `tools.web.fetch.enabled=false`.

## Best Possible Solution

Keep this open and fix the startup contract so optional/local `tools.web.fetch` configuration cannot block Gateway binding. Runtime web-fetch provider discovery should be skipped, deferred, or tightly scoped during startup unless an explicitly active provider credential must be resolved, with regression coverage for both `enabled:false` and local-only fetch settings.

## Reproduction Assessment

Yes. The issue gives concrete version, config, command, trace boundary, and removal workaround, and the current source path explains why a present `tools.web.fetch` object can affect pre-bind startup.

## Solution Assessment

No current solution is present on main. The narrow maintainable fix is in startup/runtime web-tool secret resolution, not a broad gateway bind rewrite or a plugin-system migration.
