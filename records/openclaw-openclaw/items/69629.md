## Summary

Keep open. Current main still protects Discord slash-command execution after an interaction is submitted, but it does not implement per-guild-channel command picker visibility for same-guild, multi-bot deployments; the older related items cover execution routing and command authorization, not Discord UI visibility.

## What This Changes

Request per-channel Discord slash-command visibility so multi-bot same-guild deployments show only the commands for the bot configured for the current channel.

## Best Possible Solution

Keep this as a Discord plugin feature request until maintainers choose the supported end state: either implement account/guild/channel application-command permission reconciliation with explicit operator setup, or document the supported workaround path such as per-account native command disablement, text commands, or server isolation.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path exists: configure multiple Discord accounts in one guild with separate allowed channels and native commands enabled; current main still deploys each account's command catalog independently of those channel lists and only rejects after the interaction is submitted.

## Solution Assessment

Unclear. Discord channel command permissions match the requested UX, but automating them is not a narrow local filter because the permission endpoint requires a sufficiently privileged Bearer token rather than the bot token, so maintainer product/API judgment should come before a fix PR. citeturn1view2
