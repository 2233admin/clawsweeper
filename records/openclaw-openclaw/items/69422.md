## Summary

Keep #69422 open. Current main still does not carry Google Chat thread identity as a first-class outbound/block/queued routing field. The lower-level Google Chat sender can thread a reply when `payload.replyToId` is already a valid `spaces/.../threads/...` value, but the inbound-to-block and queued-followup paths still do not reliably preserve that value across the reported streamed block and concurrency cases.

## Best Possible Solution

Keep the issue open and fix it in the Google Chat-owned path with a narrow generic seam where needed: preserve `message.thread.name`/`threadKey`/space through inbound context, streamed block payloads, coalescer keys/conflicts, and `FollowupRun` routing; ensure Google Chat delivery uses the valid thread resource for every chunk/final/media send; add `googlechat` to the queue by-channel schema; and add regression tests for typing-placeholder update, chunk 2+, coalesced chunks, and queued concurrent messages in the same thread.
