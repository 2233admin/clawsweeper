## Summary

Keep this issue open. Current main still lacks a first-class Google Chat thread route through outbound reply payloads, streamed block coalescing, and queued follow-up delivery. The lower Google Chat sender can thread correctly only when `payload.replyToId` is already a valid `spaces/.../threads/...` resource; upstream paths can still fall back to `spaces/.../messages/...` or omit thread metadata entirely.

## Best Possible Solution

Keep this issue open as the canonical Google Chat streamed-block and queued-followup thread-routing bug. The best fix is a consolidated Google Chat-owned patch with a narrow generic seam only where needed: preserve inbound `message.thread.name` and any needed space/thread-key metadata through outbound reply payloads, block pipeline keys/conflict checks, and follow-up queue routing; ensure every Google Chat text chunk, media send, edit/fallback, and queued drain uses a valid `spaces/.../threads/...` value instead of `spaces/.../messages/...`; add `googlechat` to `messages.queue.byChannel`; and add focused regression tests for typing-placeholder update, chunks 2+, coalesced chunks, and concurrent queued messages in the same thread.
