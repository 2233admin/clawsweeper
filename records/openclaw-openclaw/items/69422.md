## Summary

Keep this issue open. Current main still does not preserve a Google Chat thread resource as `MessageThreadId`, still routes queued follow-ups from `ctx.MessageThreadId`, and still has Google Chat delivery/coalescing paths that depend on `replyToId` being the right thread-shaped value. The linked PR #74235 is an active fix candidate, but it is open and not present on main, so this should not be closed yet.

## What This Changes

The issue asks OpenClaw to preserve Google Chat thread/reply routing metadata across streamed block replies, continuation chunks, and queued follow-up drains, with a per-channel queue-mode mitigation for Google Chat.

## Best Possible Solution

Land a consolidated Google Chat-owned fix that carries the inbound `message.thread.name` as a valid thread resource through turn context, queued follow-up routing, natural/block reply delivery, message-tool sends, and Google Chat API validation; then decide whether the `messages.queue.byChannel.googlechat` mitigation should be included in the same patch or tracked separately.

## Reproduction Assessment

Yes. The issue gives concrete DM/space repro steps for streamed chunks and concurrent queued messages, and current source inspection shows the missing `MessageThreadId` and replyTo-only delivery paths that explain those symptoms.

## Solution Assessment

Unclear until PR #74235 is reviewed against this full issue. Its channel-boundary `MessageThreadId` approach looks like the narrow maintainable direction, but maintainers should confirm queued follow-up, coalesced chunk, and `messages.queue.byChannel.googlechat` coverage before treating the issue as fixed.
