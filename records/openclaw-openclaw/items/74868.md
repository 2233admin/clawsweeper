## Summary

Keep this issue open. Current main still resolves group/channel source replies to tool-only delivery from config alone, suppresses automatic delivery, and has no visible fallback when the `message` tool is removed by `disableMessageTool` or tool policy.

## What This Changes

Request fallback to automatic source delivery when `messages.groupChat.visibleReplies` selects `message_tool` but the effective agent tool inventory does not include `message`.

## Best Possible Solution

Make source reply visibility policy aware of the effective `message` tool availability at the dispatch/run boundary. If the configured mode is `message_tool_only` but `message` is unavailable, use automatic source delivery and adjust prompt guidance/tests accordingly while preserving native-command and send-policy behavior.

## Reproduction Assessment

Yes. The static reproduction is high confidence: set a group/channel turn with default visible replies and omit `message` via `disableMessageTool` or a restrictive tool allowlist; current main still suppresses automatic delivery and has no availability guard.

## Solution Assessment

Yes, the requested fallback is the narrow maintainable direction, but it should be implemented from the actual effective tool inventory or the same policy inputs that build it, not by config inspection alone.

## Repair Work Prompt

Fix the group/channel visible reply fallback for https://github.com/openclaw/openclaw/issues/74868. Current behavior: `messages.groupChat.visibleReplies` defaults to `message_tool`, `resolveSourceReplyDeliveryMode()` returns `message_tool_only`, and `dispatchReplyFromConfig()` suppresses automatic delivery even when the effective agent tool list does not include `message` because of `disableMessageTool`, tool allowlists, or policy. Implement a narrow availability-aware guard so source delivery falls back to `automatic` when `message_tool_only` was selected but the `message` tool is unavailable. Prefer using the actual effective tool inventory or the same final policy inputs used to build it; do not bypass `sendPolicy: deny`, native command handling, explicit requested delivery modes where callers intentionally force behavior, or plugin/channel delivery contracts. Also update prompt guidance so it does not tell the agent to call `message(action=send)` when `message` is unavailable, and add regression tests covering default group/channel tool-only behavior with and without the message tool. No changelog entry is required unless maintainers want one for the user-facing bugfix.
