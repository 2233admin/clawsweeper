## Summary

Current main still has the reported stale writeback class. The gateway agent handler reads a cached `entry`, builds `nextEntryPatch` with cached fallbacks for `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `channel`, then merges that prebuilt patch over the fresh row loaded inside `updateSessionStore`. Related PR #19328 is still open and unmerged, and its current diff still leaves these residual delivery/channel writebacks in place.

## Best Possible Solution

Keep this issue open. The best fix is to keep cached `entry` fallback values out of `nextEntryPatch`, derive any fallback delivery/channel values from the fresh row inside the `updateSessionStore` mutator when needed, and add regression coverage for `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `channel` using the stale cached-entry/fresh store pattern. Because #19328 is active in the same code path, maintainers should choose whether to fold this into that PR or land a focused follow-up.
