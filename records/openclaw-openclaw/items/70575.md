## Summary

Keep open. Current main still constructs a session writeback patch from the cached `entry` before the locked store update, and that patch still carries cached or undefined `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, `modelOverride`, `providerOverride`, and `channel` values over the fresh row. The related PR #19328 is open and unmerged, so the remaining race is not implemented on main or shipped.

## What This Changes

Remove stale cached `entry` fallbacks from the agent handler session writeback for delivery/channel fields and add stale-cache versus fresh-store regression coverage.

## Best Possible Solution

Keep the agent-handler writeback from carrying stale cached fallbacks into the locked store merge: derive any needed delivery/channel fallbacks from the fresh row inside the `updateSessionStore` mutator, or conditionally include only newly computed request/effective values, with focused regression tests for fresh store preservation.

## Reproduction Assessment

Yes. The source-level reproduction is clear: load a stale cached session entry, let `sessions.patch` update the fresh store row, then let the agent handler merge its prebuilt patch over that row inside `updateSessionStore`.

## Solution Assessment

Yes technically. The requested direction is the narrowest maintainable fix, but the implementation should be coordinated with open PR #19328 because it edits the same session writeback path.
