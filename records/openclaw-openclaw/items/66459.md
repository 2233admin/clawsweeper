## Summary

Keep open. Current main still contains the shared same-target message-tool suppression path that can drop a distinct final text reply after a same-origin file/media send. The Telegram reasoning/commentary leak class appears covered separately, but source, tests, and maintainer discussion all point to this remaining reply-payload bug as still valid.

## What This Changes

The issue requests reliable Telegram delivery of user-visible final text when an assistant turn includes reasoning/text content or follows a same-target message-tool file/media send.

## Best Possible Solution

Narrow same-target message-tool suppression to exact duplicate text/caption/media delivery, preserve distinct final text after file/media sends in both buildReplyPayloads and follow-up delivery, and replace the existing blanket-suppression tests with regression coverage for the file/media-followed-by-text case.

## Reproduction Assessment

Yes. The current-main unit-level path is high confidence: buildReplyPayloads and resolveFollowupDeliveryPayloads still return no payload for same-target message-tool sends, and existing tests assert suppression even when the final text differs from the recorded message-tool text.

## Solution Assessment

Yes, keeping this open is the right handling. The narrow maintainable fix is shared auto-reply dedupe, not another Telegram-only streaming workaround or an implemented-on-main close.
