## Summary

Keep open. Current main still uses a module-local sandbox backend registry, and this PR is a narrow core/API fix for third-party sandbox backend plugins that can be loaded through a separate jiti/native module graph. I found no blocking correctness finding and no concrete security or supply-chain concern.

## What This Changes

This PR changes `src/agents/sandbox/backend.ts` to store sandbox backend registrations in a `resolveGlobalSingleton(Symbol.for(...))` map instead of a module-local `Map`.

## Best Possible Solution

Land the minimal core registry fix after maintainer review, preferably with a focused regression test that registers a sandbox backend through the plugin/jiti path and resolves it through the runtime backend lookup. Optional sandbox implementations should continue to live as plugins.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is to load a sandbox backend plugin through the jiti plugin path, have it call `registerSandboxBackend("x", ...)`, then resolve an agent sandbox with backend `x` through the runtime backend module on current main; the module-local map can miss the registration and throw `Sandbox backend "x" is not registered`.

## Solution Assessment

Yes. A process-global registry keyed by `Symbol.for` is the narrowest maintainable fix for this split-module registry problem and follows the existing singleton helper pattern; adding a targeted regression test would improve proof without changing the direction.
