## Summary

Keep this PR open for normal maintainer review. Current main still lacks the proposed fast path, the diff targets the source-backed cache-miss path described in #74117, and I did not find a concrete blocking correctness or security issue in the proposed patch.

## What This Changes

The PR adds an active-registry fast path to `ensureRuntimePluginsLoaded` and updates the runtime plugin unit test to assert that the shared registry resolver is not called when a registry is already active.

## Best Possible Solution

Review and, if CI agrees, land a narrow fix that reuses the already-active gateway runtime registry for inbound dispatch while leaving broader cache-key subset compatibility to a separate loader change.

## Reproduction Assessment

Yes. The current-main source path reproduces the cache-miss shape: gateway boot populates an active registry with richer load options, while inbound dispatch calls `ensureRuntimePluginsLoaded` with a smaller option set that fails the strict cache-key comparison; #74117 supplies production stack/timing logs for the same path.

## Solution Assessment

Yes for the stated gateway-dispatch regression. The PR’s helper-level fast path is the narrowest fix for avoiding redundant dispatch-time plugin reloads; a broader `getCompatibleActivePluginRegistry` subset-compatibility change would touch more shared loader behavior and is better handled separately.
