## Summary

Keep this PR open. The Codex tool-stream bridge addresses a real current-main gap, but the current patch still emits unsanitized Codex tool args and full-output tool payloads into the normalized agent event stream; that is a concrete data-exposure regression compared with the existing Pi tool-stream path.

## What This Changes

The PR adds Codex app-server projector support for normalized `stream: "tool"` start, update, and result events plus regression tests for command events, turn-completed result deduplication, and non-full-output redaction of bulky payloads.

## Best Possible Solution

Keep the Codex tool-stream bridge, but make the Codex projector sanitize start args, streamed partial results, and final full-output results through an SDK-safe shared tool payload sanitizer before emitting agent events; add regression coverage with secret-bearing command, MCP, dynamic, image, and full-output payloads, and include the required user-facing changelog entry.

## Reproduction Assessment

Yes. Instantiate `CodexAppServerEventProjector` with `onAgentEvent`, send a `commandExecution` or MCP/dynamic item containing a secret-looking arg/output under full output, and inspect the emitted `stream: "tool"` event; the PR head forwards those raw fields before any sanitizer.

## Solution Assessment

No. The direction is the right narrow fix for missing Codex live tool cards, but this implementation is not the best safe version until it preserves the same producer-side redaction guarantee as the existing Pi tool stream and restores the required changelog coverage.
