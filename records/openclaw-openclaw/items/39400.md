## Summary

Current main still does not provide the structured, cron-safe `session_status` usage contract requested here. Provider quota windows remain optional human-readable status text, and timeout/error/no-auth states can still be omitted instead of returned as explicit structured availability metadata. Related #8635 is adjacent broader usage API work, but it remains open and does not supersede this narrower provider quota-window stability request.

## Best Possible Solution

Keep this open and implement an additive `session_status.details.usage` contract. Preserve the existing human-readable status card, but return stable provider/source, availability or unavailable reason, plan when known, normalized quota windows such as 5h/day/week with usedPercent/percentLeft/resetAt, and reset explanation only when upstream data supports it. Treat #8635 as related broader token/session budgeting work, not as a replacement unless it explicitly covers provider quota-window availability and reset metadata.
