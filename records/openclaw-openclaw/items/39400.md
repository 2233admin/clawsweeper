## Summary

Keep this issue open. Current main still exposes provider quota windows to `session_status` as optional human-readable status text and does not return a structured `details.usage` payload, explicit availability state, source metadata, or reset explanation. Related #8635 is broader token-usage API work, and #45342/#45332 address Anthropic polling/cache behavior rather than this cron-safe `session_status` contract.

## What This Changes

Request stable, structured `session_status` usage data for provider quota windows, including 5h/day percentages, reset times, source, unavailable state, and reset metadata instead of intermittently omitted text lines.

## Best Possible Solution

Implement an additive `session_status.details.usage` contract while preserving the current status text. The payload should expose provider/source, availability or unavailable reason, plan when known, normalized quota windows such as 5h/day/week with usedPercent or percentLeft and resetAt, and reset explanation only when upstream data supports it; keep #8635 focused on broader per-session token/cost accounting but align field names where the concepts overlap.
