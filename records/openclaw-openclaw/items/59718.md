## Summary

Keep this issue open. Current main still has the exact gap described: custom Docker binds are parsed and resolved with per-bind writability, but sandbox filesystem mutations still require global `workspaceAccess === "rw"`, so `workspaceAccess: "none"` rejects writes even when the resolved target is on a `:rw` bind. The related open reports cover adjacent sandbox bind/source and workspace permission problems, but they do not fully supersede this least-privilege writable-bind authorization request.

## Best Possible Solution

Implement mount-aware write authorization in the sandbox filesystem bridge: allow `write`, `edit`, `apply_patch`, mkdir/remove/rename only when the resolved target is on a writable mount, keep `workspaceAccess: "none"` workspace and `:ro` binds read-only, add regression coverage for `:rw` and `:ro` custom binds under `workspaceAccess: "none"`, and coordinate separately with #57271 for external bind-source usability.
