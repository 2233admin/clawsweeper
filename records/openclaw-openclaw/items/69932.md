## Summary

Keep PR #69932 open. Current main has not incorporated the BlueBubbles monitor-level private-API guards or disabled-path monitor test. The low-level chat helper already skips requests when the cached status is explicitly false, but monitor-processing still calls typing start/restart/stop unconditionally when the status is unknown, and the PR is a focused bundled BlueBubbles behavior change with active author follow-up rather than an obsolete cleanup target.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land this PR or an equivalent BlueBubbles-owned fix after deciding the unknown-status policy: either accept the PR's conservative `privateApiEnabled` gate plus disabled-path test, or adjust the implementation to skip only explicit `false` while preserving typing attempts when the probe cache is `null`.

## Risks / Open Questions

- The PR as written intentionally treats unknown private-API status the same as disabled, which can suppress a typing bubble during a cold/expired/failed probe on servers where Private API is actually enabled. Maintainers should choose between that trade-off, an explicit false-only guard, or a lazy probe before merge.
- The exact steady-state `private_api: false` spam described in the PR is partly mitigated in current main by the low-level `chat.ts` early return when the false status is cached, so the remaining value is mostly monitor-level consistency and unknown-status behavior.
