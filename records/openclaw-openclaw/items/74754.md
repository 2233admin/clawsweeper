## Summary

Keep this issue open. Current main still has a credible gateway package auto-update failure path, and the linked open PR #74870 is an implementation candidate that should be reviewed, landed, revised, or closed before this issue is resolved.

## What This Changes

The issue asks for Homebrew-installed gateways with `update.auto.enabled` to actually apply a detected stable update after the configured rollout delay instead of only recording it in update state.

## Best Possible Solution

Resolve the Homebrew package auto-update contract explicitly: either add a safe Homebrew-aware out-of-process apply flow, or document and log that Homebrew-managed installs must be upgraded manually while preventing the doomed npm-style auto-update attempt. The linked PR #74870 is the current concrete path to review for that outcome.

## Reproduction Assessment

Yes. The high-confidence code-level reproduction is a managed gateway service with package auto-update enabled: the scheduled child update inherits gateway service markers and `openclaw update` refuses package replacement before invoking the package manager.

## Solution Assessment

Unclear. Skipping Homebrew auto-update with a clear `brew upgrade openclaw` hint is a narrow and plausible fix, but it changes the expected behavior from auto-install to explicit unsupported/manual upgrade, so maintainers should decide whether #74870 is the right product contract.
