## Summary

Keep open. Current main still has a credible Homebrew/package auto-update failure path, and the linked PR remains open with closing syntax for this issue, so the safe maintainer path is to review, amend, land, or replace that PR rather than close this issue early.

## What This Changes

The issue reports that a Homebrew-installed gateway detects a newer stable version but does not auto-install it after the configured rollout delay.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or detect Homebrew-managed installs and skip with a clear `brew upgrade openclaw` hint plus aligned tests, docs, and changelog coverage.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the auto-update child inherits gateway service markers, then `openclaw update` refuses package replacement from inside that service process.

## Solution Assessment

Unclear. The linked skip-and-hint PR is a plausible narrow fix if maintainers choose not to auto-upgrade Homebrew installs, but it does not implement the originally expected auto-install behavior and still needs PR review follow-through.
