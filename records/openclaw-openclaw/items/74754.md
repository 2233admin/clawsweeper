## Summary

Keep this issue open. Current main still has a credible Homebrew/package auto-update failure path, and open PR #74870 already references this issue with closing syntax, so the right cleanup outcome is maintainer review of that PR or a replacement rather than closing the issue.

## What This Changes

The issue requests that Homebrew-managed OpenClaw gateway auto-update install a detected newer stable version after the configured delay instead of remaining on the old version.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or detect Homebrew-managed installs and skip with a clear `brew upgrade openclaw` hint plus aligned docs and tests. Since PR #74870 is already open, review, land, or replace that PR as the canonical fix path.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due, for example by setting `stableJitterHours: 0` or waiting past the computed `applyAfter`; the child update command inherits gateway service markers and the CLI refuses package replacement inside that service process.

## Solution Assessment

Unclear. PR #74870 is a narrow and plausible fix if maintainers choose skip-and-hint for Homebrew, but it does not implement the originally expected auto-install behavior, so the product contract needs maintainer review.
