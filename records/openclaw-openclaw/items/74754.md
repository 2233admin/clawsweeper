## Summary

Keep this issue open. Current main still has a credible package auto-update failure path for managed gateway installs, and open PR #74870 is already linked with closing syntax, so this should be resolved by reviewing or replacing that implementation rather than auto-closing the issue.

## What This Changes

Investigate a Homebrew-installed gateway auto-update path where a newer release is detected but not installed after the configured stable delay.

## Best Possible Solution

Resolve the Homebrew package auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or skip Homebrew-managed installs with a clear `brew upgrade openclaw` hint and aligned docs so users are not told auto-update will apply when it cannot.

## Reproduction Assessment

Yes. The high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the spawned child inherits gateway service markers, and `openclaw update` refuses package replacement before invoking the package manager.

## Solution Assessment

Unclear. The linked PR #74870 is a narrow plausible fix if maintainers choose manual Homebrew upgrades as the supported contract, but it changes the expected behavior from auto-install to skip-and-hint, so maintainer product review is still needed.
