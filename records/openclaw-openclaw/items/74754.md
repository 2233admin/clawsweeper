## Summary

Keep this issue open. Current main still has the credible Homebrew/package auto-update failure path, and open PR #74870 already references this issue with closing syntax, so cleanup should wait for that PR to merge, be replaced, or be explicitly rejected.

## What This Changes

The issue reports that a Homebrew-installed gateway detects a newer OpenClaw version but never auto-installs it after the configured stable delay.

## Best Possible Solution

Resolve the Homebrew auto-update contract explicitly: either implement a safe Homebrew-aware apply flow outside the live gateway process, or land the skip-and-hint approach from #74870 with aligned docs and tests so users are told to run `brew upgrade openclaw`.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is a package-managed gateway service with auto-update due: the auto-update child inherits gateway service markers, then `openclaw update` refuses package replacement inside that service process.

## Solution Assessment

Unclear. PR #74870 is a plausible narrow fix if maintainers choose skip-and-hint for Homebrew installs, but it does not implement the originally expected auto-install behavior.
