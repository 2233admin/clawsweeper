## Summary

Keep open. Current main still uses the kickstart-based macOS LaunchAgent restart path before any LaunchAgent bootout/PID-wait/start sequence, tests explicitly preserve that healthy-path behavior, and the adjacent merged recovery work only handles installed-but-unloaded supervision states rather than the requested ordering change.

## What This Changes

Request changing macOS `restartLaunchAgent` to stop the LaunchAgent, wait for the previous process to exit, then bootstrap/kickstart the service with coverage for detached handoff behavior.

## Best Possible Solution

Make an explicit maintainer decision on the macOS LaunchAgent restart contract. If accepted, implement the stop/wait/start ordering in the launchd owner module while preserving managed restart UX, detached self-restart, unloaded-service recovery, disabled-state recovery, and stale listener cleanup.

## Reproduction Assessment

Unclear for a high-confidence live race reproduction because the failure depends on macOS launchd timing. The acceptance gap is reproducible by inspection on current main: restart and handoff paths still use `kickstart -k` before any explicit old-service-exit-before-start sequence.

## Solution Assessment

Unclear pending maintainer judgment. The proposed two-phase sequence directly targets the race, but the maintainable route needs launchd owner approval plus unit and live macOS validation because it changes a deliberately kickstart-based restart contract.
