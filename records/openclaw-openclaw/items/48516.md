## Summary

Keep open. Current main still matches the reported dual-source routing gap: Android notification forwarding is opt-in, but enabled blocklist mode still forwards native-channel app notifications unless manually blocked; the gateway still accepts `notifications.changed` packages without server-side native-provider filtering; and the message tool's cross-context guard only applies when a trusted current channel target exists.

## Best Possible Solution

Keep this issue open and implement a layered maintainer-approved fix: Android should default to a narrow allowlist or built-in deny defaults for native OpenClaw channel packages; the gateway should reject or ignore denied `notifications.changed` package names before enqueueing/waking heartbeat; and notification-originated runs should be denied or route-locked for outbound message sends unless a trusted originating channel/peer/session tuple exists. Add Android policy tests, gateway notification filter tests, and outbound route-lock coverage.
