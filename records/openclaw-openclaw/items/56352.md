## Summary

Keep this issue open. Current main has improved related failover behavior, including cross-provider fallback after overloaded/profile failures, and it avoids treating an already streamed block reply as an empty-result fallback failure. But the central requested behavior is still not implemented: a session lane remains occupied until the active `runEmbeddedPiAgent` task resolves, and follow-up inbound messages for an active session are enqueued for later rather than processed immediately after a Telegram block/partial reply is emitted.

## Best Possible Solution

Keep this issue open and implement a focused core fix around session-lane lifecycle for delivered replies. The likely path is to add a safe post-delivery/final-reply boundary that allows the next inbound message to proceed only after transcript and delivery state needed for session consistency is finalized, plus a regression test covering Telegram block streaming with fallback success and delayed cleanup. Do not close it as a duplicate of the broader closed lane/failover reports unless a narrower canonical open issue or PR is created for this exact lifecycle problem.

## Risks / Open Questions

- Changing this naively could let two same-session runs mutate the same transcript, tool state, or reply pipeline concurrently. The fix needs a clearly defined safe release boundary, not a generic unlock on the first partial chunk.
- The existing related fixes for provider fallback and block-reply result classification may mask some reproductions, so a regression test should model the exact Telegram/block-streaming failover case with a delivered reply followed by delayed cleanup.
