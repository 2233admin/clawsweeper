## Summary

Keep open. Current main still lacks the requested backup SDK subpath, and the PR is focused, but it should not merge as-is because it promotes backup plan resolution into a public SDK contract without documenting the local path inventory and trust boundary. The earlier build-entry concern appears superseded by current generated tsdown wiring.

## What This Changes

The PR adds an `openclaw/plugin-sdk/backup` subpath that re-exports backup archive helpers, backup plan resolution/types, and backup create option/result types, then registers the subpath in package exports and SDK entrypoint metadata.

## Best Possible Solution

If maintainers want backup planning to become supported plugin API, land a rebased narrow SDK subpath with docs that define intended consumers, absolute-path exposure, trust assumptions, aligned SDK metadata/API checks, and validation. Otherwise keep lower-level backup planning private and continue exposing only the existing archive creation helper.
