## Summary

Keep open. Current main still has the reported audit false positive, and the PR is useful but not ready because its latest auth-store guard still misses dual-field profiles where an explicit `keyRef`/`tokenRef` coexists with an inline `$VAR` or `${VAR}` value.

## What This Changes

The PR adds `$VAR` env SecretRef shorthand parsing, changes auth-profile audit classification for inline env refs, and adds parser/audit regression tests.

## Best Possible Solution

Update this PR so auth audit classifies the inline credential value itself before the plaintext fallback, regardless of an explicit sibling ref, while keeping real leftover plaintext under `keyRef`/`tokenRef` reportable and coordinating any config reporting changes with #64344.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Classify inline auth env refs before plaintext fallback:** `src/secrets/audit.ts:303-305`
  - body: `resolveSecretInputRef` returns `inlineRef: null` when `keyRef`/`tokenRef` is present, so this guard only skips plaintext for env refs without an explicit sibling ref. A profile with `keyRef` plus `key: "$OPENAI_API_KEY"` or `"${OPENAI_API_KEY}"` still reaches the plaintext fallback and reports `PLAINTEXT_FOUND`; check the inline value itself before treating it as plaintext while preserving the explicit-ref-plus-real-plaintext case.
