## Summary

Keep open. Current main still lacks the stale first-attempt recovery guard, and this PR remains a focused implementation candidate, but the current diff misses the recovery-budget deferral path because that path changes retry state without an actual delivery attempt.

## What This Changes

The PR adds a `maxEntryAgeMs` stale-entry guard, `skippedStale` recovery accounting, and recovery tests for startup replay of never-attempted pending delivery entries.

## Best Possible Solution

Rebase the PR over current main and make stale protection depend on true attempted-delivery state, or preserve never-attempted status across recovery-budget deferrals. Add a regression test where an old never-attempted entry is budget-deferred once and then skipped as stale on the next startup, while preserving recent crash recovery for fresh entries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve stale filtering after budget deferrals:** `src/infra/outbound/delivery-queue-recovery.ts:216-217`
  - body: The new gate only treats `retryCount === 0` entries with no `lastAttemptAt` as stale candidates, but `deferRemainingEntriesForBudget` calls `failDelivery`, which increments `retryCount` and sets `lastAttemptAt` without a send attempt. In a large backlog, an old never-attempted entry can be budget-deferred once and then bypass the stale guard on the next startup, so stale replay can still happen.
  - confidence: 0.9
