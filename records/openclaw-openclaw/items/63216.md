## Summary

Keep open. Current main has useful mitigations for small context windows and aggregate tool-result truncation, but the remaining issue is still present: the hard-reset fallback can still recommend reserveTokensFloor even when that is already high, overflow recovery still retries compaction up to three times, and documented/current post-compaction paths still re-inject bootstrap/post-compaction context on retry-like flows.

## Best Possible Solution

Keep this issue open and have a maintainer scope the remaining work: make context-overflow reset guidance aware of effective reserve floor, contextWindow/contextTokens, and already-high reserve settings; then decide whether post-compaction/retry paths should suppress, shrink, or dedupe bootstrap/context reinjection. Keep #66126 as the narrower prompt-reinjection loop and #55366 as the hard-reset state-loss UX follow-up unless maintainers intentionally merge those scopes.
