## Summary

Keep open. Current main partially addresses the original timeout complaint with a 120-second default, but this PR still proposes behavior main does not implement, and the patch removes the first A2A wait cap without adding a product-level maximum or making the rest of the A2A waits consistent.

## What This Changes

The PR raises the subagent announce timeout default and matching test from 60 seconds to 180 seconds, and changes the first `sessions_send` A2A wait to use the configured announce timeout instead of a 60-second cap.

## Best Possible Solution

Define one A2A timeout policy on current main: keep or deliberately change the 120-second default, add an explicit maximum or shared clamp for `sessions_send.timeoutSeconds`, and make the initial wait, nested reply/announce waits, and timeout-triggered background announce behavior consistent with focused regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Bound the A2A wait timeout before removing the cap:** `src/agents/tools/sessions-send-tool.a2a.ts:34`
  - body: This line lets `sessions_send.timeoutSeconds` drive `agent.wait` with no product-level maximum. The schema only lower-bounds the value and gateway timers do not add an A2A-specific cap, so a caller can hold the flow for a very long time. Add an explicit maximum or shared clamp before honoring waits above 60 seconds.
