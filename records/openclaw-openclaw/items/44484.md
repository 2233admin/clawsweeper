## Summary

Keep open. Current main has partial effective-tool visibility through `tools.effective`, the Control UI, and sandbox-specific blocked-tool explanations, but it still does not implement strict inheritance or a declared-vs-effective removal trace. Related #59898 is an open, narrower prompt/no-tools fix and its body explicitly does not redesign capability handling.

## Best Possible Solution

Keep this issue open as the umbrella for tool capability determinism. The best path is to extend the tool-policy resolution pipeline to return declared inputs, effective tools, removed tools, and the layer/reason for each removal; expose that through `tools.effective`, `/tools` or status/debug tooling, Control UI, and docs; then add strict inheritance only after maintainers define how unavoidable safety/runtime restrictions should fail closed.
