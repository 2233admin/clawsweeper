## Summary

Keep this issue open. Current main still persists recovery failures through the generic error formatter, and that formatter still returns only `Error.message` for Error instances, so Slack Web API platform errors with `data.needed` and `response_metadata.scopes` remain lossy in delivery queue `lastError` and recovery logs. The open Slack PR is useful related work, but it is extension-log scoped and does not touch the outbound recovery path.

## What This Changes

The issue requests preserving Slack Web API structured error details, especially `missing_scope` needed/granted scope fields, when delivery queue recovery records retry failures.

## Best Possible Solution

Preserve safe structured metadata from Error-like API failures in the shared formatter used by outbound delivery recovery, without importing Slack-specific code into core. Add focused tests for a Slack-shaped `missing_scope` Error and decide whether `missing_scope` should also be classified as a permanent delivery error.

## Reproduction Assessment

Yes. A focused unit test can enqueue a delivery, make recovery's mocked deliver function reject with `Object.assign(new Error("An API error occurred: missing_scope"), { code: "slack_webapi_platform_error", data: { error: "missing_scope", needed: "im:write", response_metadata: { scopes: ["chat:write"], acceptedScopes: ["im:write"] } } })`, and then assert the stored `lastError` includes the structured scope details; current main would only store the message.

## Solution Assessment

Yes, the narrow maintainable direction is a generic safe Error-metadata formatter used by delivery recovery and live queue failure paths. Core should not hard-code Slack or import plugin files, but it can preserve well-known public fields such as `code`, `data.error`, `data.needed`, `data.provided`, `response_metadata.scopes`, `acceptedScopes`, `retryAfter`, and `messages` with existing redaction still applied.
