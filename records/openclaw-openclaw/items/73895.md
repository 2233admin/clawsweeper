## Summary

Keep open. Current main still lacks the requested promotion-grade Gateway v2 metadata envelope: the WS chat schema, agent RPC result path, and OpenAI/OpenResponses compatibility handlers expose routing inputs, raw run results, and basic usage totals, but not the requested config lineage, route/policy metadata, timing breakdown, decision trace, fallback chain, or reasoning-token surface.

## What This Changes

The issue requests an additive Gateway response metadata surface with config lineage, routing and policy fields, timing fields, fallback-chain data, gateway version, decision trace, and token usage including reasoning tokens so a live benchmark smoke can validate Gateway v2 output.

## Best Possible Solution

Design and land an additive typed Gateway v2 metadata envelope that redacts config lineage, exposes bounded route and policy identifiers, includes safe timing and usage breakdowns, and is wired consistently through the WS agent/chat paths and the HTTP compatibility endpoints the benchmark targets, with docs and compatibility tests updated together.

## Reproduction Assessment

Yes. This is a feature gap rather than a runtime crash: static inspection of current main reproduces it because the requested fields are absent from the typed WS schemas, agent result payload, HTTP response builders, and shared usage projection.

## Solution Assessment

Unclear until maintainers approve the metadata contract. The maintainable direction is a typed, redacted, additive protocol envelope with explicit scope/security rules, not a freeform result.meta payload assembled independently per endpoint.
