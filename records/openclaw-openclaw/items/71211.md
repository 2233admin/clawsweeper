## Summary

Keep this open. Current main still builds foreground, background/process, and node-host exec tool results from raw stdout/stderr-derived strings before an exec-owned redaction boundary; downstream UI/event and transcript sanitizers are useful defense in depth but do not make the exec result construction itself safe.

## What This Changes

The issue asks OpenClaw to redact, block, or require explicit reveal for secret-bearing stdout/stderr before exec tool results enter agent context.

## Best Possible Solution

Add a shared exec/tool-result redaction boundary that sanitizes stdout/stderr-derived content and structured details before foreground exec, background process poll/log, node-host exec, and approval follow-up summaries reach agent context, while keeping UI/event/transcript redaction as defense in depth.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints an env-style secret such as `TOKEN=...` flows into foreground, background/process, or node-host tool result content/details before an exec-owned redaction pass.

## Solution Assessment

Yes. Redacting output before tool result construction and covering the shared process/node-host paths is the narrowest maintainable direction; blocking specific secret-file reads could supplement it but would not cover generic secret-bearing stdout/stderr.
