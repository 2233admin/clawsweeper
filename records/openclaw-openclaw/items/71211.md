## Summary

Keep open. Current main still returns stdout/stderr-derived exec output in exec-owned tool result content/details before a dedicated exec-boundary redaction pass; the downstream Control UI/event redaction work is useful but does not close the internal tool-result exposure described here.

## What This Changes

The issue asks OpenClaw to redact or otherwise gate secret-bearing stdout/stderr before exec, process, node-host, and approval follow-up results are delivered back to agents.

## Best Possible Solution

Add a shared exec/tool-payload redaction boundary that applies to stdout/stderr-derived foreground results, runtime outcomes, process poll/log responses, node-host formatted results, and approval follow-up summaries before they enter agent-visible tool results or system events, with regression tests that preserve existing output caps and truncation behavior.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a command that prints an env-style secret flows through `session.aggregated` or `outcome.aggregated` into foreground exec, process poll/log, node-host, and follow-up result text/details before an exec-owned redaction call. I did not run a live secret-printing command because this review is read-only.

## Solution Assessment

Yes. The report's primary direction, redacting stdout/stderr before exec result construction and at shared process/node-host boundaries, is the narrowest maintainable fix. Blocking specific secret-file reads may help as defense in depth, but it would not replace output redaction.
