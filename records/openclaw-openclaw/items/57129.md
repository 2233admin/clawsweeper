## Summary

Current main only partially addresses the requested isolation. Slack app mentions/seeded thread roots can route to per-thread session keys, but ordinary handled top-level Slack channel messages are still explicitly kept on the shared per-channel session, and the docs still describe channel sessions as `agent:<agentId>:slack:channel:<channelId>`. Because this issue asks for fresh isolated runs for new top-level questions broadly, it should stay open for maintainer/product clarification and a focused Slack routing fix.

## Best Possible Solution

Keep this open and decide the intended Slack product behavior explicitly: either make every handled new top-level Slack question route to a fresh session key such as `:thread:<message.ts>` while follow-ups use Slack `thread_ts`, or add a documented config switch that preserves per-channel continuity for always-on rooms. The fix should include routing tests for app mentions, plain handled channel messages, DMs, and Slack thread replies, plus docs that explain the isolation model.
