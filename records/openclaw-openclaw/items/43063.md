## Summary

Keep this PR open. Current main still does not expose a providerUsage or metered OpenRouter billing object from `openclaw models status --json`, but the PR is not merge-ready because it has a TypeScript type error, uses the wrong current-key OpenRouter endpoint, performs authenticated OpenRouter billing calls for non-JSON output, and still reports some failed billing requests as healthy.

## What This Changes

The PR adds an `auth.providerUsage` JSON object to `openclaw models status --json`, adds a core OpenRouter metered billing fetcher with tests, and rewires provider usage fetching for models status output.

## Best Possible Solution

Define a small additive provider-usage JSON contract, extend the provider usage snapshot model deliberately, implement OpenRouter-specific credential resolution and billing endpoint parsing inside `extensions/openrouter` via `resolveUsageAuth`/`fetchUsageSnapshot`, fetch metered billing only for the JSON surface that renders it, report partial/non-2xx failures accurately, and document the fields at `https://docs.openclaw.ai/cli/models`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is `Promise<OpenRouterMeteredUsage | null>`, but this variable stores the awaited snapshot and is later read as an object. Typechecking should fail; use `Awaited<ReturnType<...>>` or the explicit snapshot type.
  - confidence: 0.95
- **[P2] Use OpenRouter's current-key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The key-usage fetch calls `/api/v1/auth/key`, but the current OpenRouter API documents the current key endpoint as `/api/v1/key`. This will turn key usage into non-2xx responses and leave the new JSON payload empty or misleading.
  - confidence: 0.9
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: `openrouterUsage` is only rendered in the JSON payload, but the PR fetches it for every non-plain status invocation. Default rich status and `--check` can therefore perform an unnecessary authenticated OpenRouter billing request and wait on its timeout.
  - confidence: 0.92
- **[P2] Report non-2xx key failures:** `src/infra/provider-usage.openrouter.ts:193-194`
  - body: A non-2xx key response is converted to `null` and is not added to `keyErrors`, so revoked or unauthorized keys can be silently omitted while the provider still reports `status: "ok"`. The JSON health signal should reflect partial or total billing request failures.
  - confidence: 0.9
