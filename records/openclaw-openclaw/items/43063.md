## Summary

Keep this PR open. Current main does not yet expose the requested provider billing shape, but the PR is not merge-ready: the head diff has a TypeScript type error, calls OpenRouter's wrong current-key endpoint, performs OpenRouter billing calls outside the JSON surface that consumes them, and under-reports provider failures. The existing review discussion already raised several of these issues, and the latest head still leaves blocking correctness and billing-contract concerns.

## What This Changes

The PR adds `auth.providerUsage` to `openclaw models status --json`, maps existing windowed provider usage into a new JSON shape, and introduces an OpenRouter metered billing fetcher with focused tests.

## Best Possible Solution

Extend the generic provider-usage contract deliberately, keep OpenRouter-specific credential resolution and HTTP parsing inside the OpenRouter plugin, expose metered snapshots through the existing provider usage hook or a versioned extension of it, fetch billing data only for the JSON surface that emits it, report non-2xx and partial failures accurately, and document the new JSON fields before merge.

## Reproduction Assessment

Yes. The blocking issues reproduce by static inspection of the PR head plus OpenRouter's current API docs: the PR uses a Promise type for an awaited value, calls `/api/v1/auth/key` instead of `/api/v1/key`, and fetches OpenRouter billing in non-JSON status paths that never render it.

## Solution Assessment

No. The requested direction is useful, but this PR is not the best implementation until it follows the provider-owned usage boundary, fixes type safety, uses documented OpenRouter endpoints, gates OpenRouter billing to `--json`, and treats partial billing failures as degraded status.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.94

Full review comments:

- **[P1] Use the awaited OpenRouter usage type:** `src/commands/models/list.status-command.ts:346`
  - body: `ReturnType<typeof loadOpenRouterMeteredUsage>` is the Promise type, but this variable stores the awaited snapshot and is later read as an object. This should fail typechecking; use `Awaited<ReturnType<...>>` or an explicit snapshot type.
  - confidence: 0.96
- **[P2] Gate OpenRouter billing fetches to JSON output:** `src/commands/models/list.status-command.ts:377-380`
  - body: `openrouterUsage` is only emitted in the JSON payload, but this call runs for every non-plain status invocation. Default rich status can now make authenticated OpenRouter billing requests and wait on timeouts even though it never renders the data.
  - confidence: 0.94
- **[P2] Call OpenRouter's current-key endpoint:** `src/infra/provider-usage.openrouter.ts:181`
  - body: The key-usage fetch requests `/api/v1/auth/key`, while OpenRouter documents the current-key endpoint as `/api/v1/key`. Valid keys can therefore produce failed or empty key usage in the new JSON payload.
  - confidence: 0.95
- **[P2] Treat non-2xx key responses as provider errors:** `src/infra/provider-usage.openrouter.ts:193-195`
  - body: A 401, 403, or 500 key response becomes `null`, and the caller only records thrown errors. Invalid or revoked keys can be silently omitted while the provider still reports an apparently healthy status with incomplete data.
  - confidence: 0.91
- **[P2] Mark partial key fetch failures as degraded:** `src/infra/provider-usage.openrouter.ts:272-280`
  - body: When one key throws but another succeeds, `keyErrors` is ignored unless there are zero successful keys. Multi-key setups can therefore return `status: "ok"` while omitting failed profiles from the JSON payload.
  - confidence: 0.89
- **[P2] Surface credits endpoint failures as provider errors:** `src/infra/provider-usage.openrouter.ts:153-155`
  - body: OpenRouter documents `/credits` as requiring a management key, but a non-2xx response is converted to `null` and can still return an `ok` provider when key usage succeeds. That hides real auth or permission failures for the account totals.
  - confidence: 0.87
