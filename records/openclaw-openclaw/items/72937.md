## Summary

Current main already fixes the reported cold `local` memory embedding provider path through the manifest-backed capability provider loader, and that implementation has shipped in `v2026.4.27`. The PR's direct manager-time registration is now obsolete and would be less consistent with the current plugin enablement boundary.

## What This Changes

The PR registers built-in memory embedding providers before `MemoryIndexManager` creates a provider and expands memory host embedding exports.

## Best Possible Solution

Keep the shipped manifest-backed capability provider fallback on main; it resolves cold `local` memory provider lookup while preserving plugin enablement and manifest ownership boundaries.

## Reproduction Assessment

Yes. The report reproduces by running memory status or sync with `agents.defaults.memorySearch.provider = "local"` when the runtime provider registry is cold; current main has a focused cold-registry fallback test and source path for that lookup.

## Solution Assessment

Yes for current main, no for merging this PR. The manifest/capability path is the narrower maintainable solution because it avoids eager global registration and keeps provider availability behind the same plugin enablement rules.
