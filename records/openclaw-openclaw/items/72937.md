## Summary

Keep open. The PR targets a plausible cold memory-provider registry path, but it is not merge-ready because it also rewrites the source memory-host SDK shim into a hand-maintained export list, and overlapping PR #73769 is pursuing a CLI-level fix with targeted regression coverage.

## What This Changes

This PR registers built-in memory embedding providers before `MemoryIndexManager` creates an embedding provider and rewrites `src/memory-host-sdk/engine-embeddings.ts` into explicit core and memory-host exports.

## Best Possible Solution

Adopt one canonical cold-registry fix for local memory embeddings, keep the memory-host SDK bridge package-owned, and add focused regression coverage for `openclaw memory status --deep` or manager initialization with `provider: "local"`.

## Reproduction Assessment

Unclear. The linked issue provides a concrete installed-package reproduction through `openclaw memory status --deep` with `memorySearch.provider = "local"`, but current main also has capability-provider fallback, and this read-only review did not run a live or targeted repro.

## Solution Assessment

No as submitted. The manager-level registration is a plausible narrow fix, but the source memory-host SDK shim rewrite should be removed and maintainers should choose between manager bootstrap, CLI bootstrap, or the existing capability fallback as the intended contract.
