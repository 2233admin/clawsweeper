## Summary

Keep open. Current main still persists Telegram native-command inbound metadata to the slash envelope `SessionKey`, while the same context carries `CommandTargetSessionKey` for the routed target session. The existing regression test also still asserts the old slash-key behavior for a configured ACP topic binding, so this PR remains a focused, relevant fix for the linked #57509 symptom rather than something already implemented or superseded.

## Best Possible Solution

Keep the PR open. The best path is to merge or adapt its narrow Telegram fix so `recordInboundSessionMetaSafe` prefers `CommandTargetSessionKey?.trim() || SessionKey || route.sessionKey`, with regression assertions covering private commands, topic-agent routing, configured ACP topic bindings, and runtime-bound topic sessions. After that, reassess #57509 for any remaining non-native-command restore or model-fallback behavior.
