## Summary

Keep open. Current main still lacks durable ACPX child stderr persistence: the OpenClaw wrapper delegates process lifecycle to the pinned acpx runtime, exposes no stderr log configuration, and has no current tests or docs for persisted stderr artifacts. The overlapping artifact PR remains open and unmerged, and the implementation needs maintainer/upstream review because persisting raw harness stderr is privacy- and retention-sensitive.

## What This Changes

The issue requests on-disk ACPX child-process stderr logging after initialization, with optional log directory and retention controls for post-mortem debugging.

## Best Possible Solution

Add bounded, redacted stderr teeing in the upstream acpx client with typed options for log directory, retention, permissions, and discoverability, then wire those options through the bundled ACPX plugin config/schema/manifest/docs with focused startup and post-initialize stderr tests.

## Risks / Open Questions

- Persisting raw harness stderr can capture credentials, local paths, prompt/tool content, provider diagnostics, or environment details, so default behavior needs redaction, restrictive permissions, bounded retention, and clear operator discoverability.
- The cleanest lifecycle hook appears to live in the upstream openclaw/acpx AcpClient.start() path; an OpenClaw-only wrapper tee could drift from the dependency that owns child-process stdio.
- Open PR #53997 overlaps the stderr artifact shape but is broad, open, and unmerged, so maintainers need to decide whether to review, split, or replace that work before a narrow fix lands.
