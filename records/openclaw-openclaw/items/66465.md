## Summary

Keep open. Current main still does not persist ACPX child-process stderr for post-mortem debugging after initialization; the OpenClaw plugin remains a thin wrapper around the pinned `acpx@0.6.1` runtime, whose client only retains a bounded startup stderr buffer and mirrors stderr when verbose mode is enabled. The overlapping artifact PR is open and unmerged, so it is useful context but not an implemented fix on main.

## What This Changes

Request durable per-subprocess ACPX stderr logs, with possible retention and configurable log directory, so post-init ACP child failures can be debugged after the fact.

## Best Possible Solution

Add bounded, redacted stderr teeing in the upstream `acpx` client with typed options for log directory, retention, permissions, and discoverability, then wire those options through `extensions/acpx` config/schema/manifest/docs with focused tests for startup and post-initialize stderr capture.

## Risks / Open Questions

- Persisting raw ACP child stderr can capture credentials, local paths, prompt/tool content, provider diagnostics, or environment details, so a fix needs redaction, restrictive file permissions, retention, and clear operator discoverability.
- The cleanest implementation appears dependency-owned in `openclaw/acpx` `AcpClient.start()` with typed options; a wrapper-only OpenClaw patch could drift from the runtime that owns child-process lifecycle.
- Open PR #53997 overlaps by adding stderr artifacts, but it is broad, open, and unmerged, so maintainers need to decide whether to review that PR, split it, or implement a narrower upstream acpx option.
