## Summary

Keep open. Current main still has the reported durability gap: the embedded/direct reset path can persist a new session mapping before the successor JSONL exists, and the replay helper can return without creating that target file. The history path then returns an empty message list when no resolved transcript candidate exists, so the existing cleanup/doctor diagnostics do not fully prevent or recover this failure mode.

## What This Changes

The issue requests that context-overflow or compaction reset/rotation never leave `sessions.json` pointing at a nonexistent transcript and preserve or recover the prior valid session history.

## Best Possible Solution

Add a generic reset/rotation invariant for embedded/direct sessions: create or prewarm the successor transcript before making it the active mapping, replay continuity when available, and preserve the prior valid mapping or fail explicitly if the successor cannot be made durable. Keep cleanup/doctor as diagnostics, not the primary recovery mechanism.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with a previous entry whose transcript is missing or has no replayable user/assistant tail, then assert the persisted successor `sessionFile` exists and `readSessionMessages`/`sessions.history` do not silently depend on a nonexistent path.

## Solution Assessment

Yes. The narrowest maintainable fix is at the reset/rotation write boundary, using the existing transcript-header/prewarm patterns, rather than relying on later cleanup of a broken active mapping.
