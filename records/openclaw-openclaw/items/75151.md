## Summary

Keep open. Current main still has a reset path that can persist a new sessionFile before the successor transcript is durable, while the replay helper can return without creating that file and chat.history returns an empty list when no transcript candidate exists. Existing cleanup and doctor diagnostics help find bad rows later, but they do not prevent or recover the broken active mapping.

## What This Changes

The issue reports that context-overflow session reset can point an active session key at a newly generated transcript path that is never created, causing history lookup to appear empty while the prior transcript remains on disk.

## Best Possible Solution

Enforce a generic reset invariant for embedded/direct session resets: create or prewarm the successor JSONL before persisting sessions.json and queued follow-up state, seed continuity when safe, and keep the previous valid mapping or fail explicitly if the successor transcript cannot be made durable.

## Reproduction Assessment

Yes. A focused unit reproduction can call resetReplyRunSession with a previous entry whose sessionFile is missing or has no replayable tail, then assert the persisted successor sessionFile exists and readSessionMessages/chat.history does not silently collapse to empty because the mapped file is missing.

## Solution Assessment

Yes. Fixing transcript durability at the reset write boundary is the narrowest maintainable solution; cleanup and doctor checks should remain diagnostics rather than the primary recovery for a broken active mapping.
