## Summary

Keep open. Current main still has a source-level path where a reset/rotation can persist a new `sessionFile` before the target transcript is durable, while `chat.history` returns an empty list when none of the resolved transcript candidates exists. Existing doctor and cleanup paths detect or prune missing transcript entries, but they do not prevent the bad mapping or recover the prior transcript.

## What This Changes

The issue reports that context-overflow recovery can rotate a Telegram direct session to a new `sessionId` and `sessionFile` that does not exist, orphaning the real previous transcript from active history lookup.

## Best Possible Solution

Add a generic session-reset invariant for embedded/direct reset paths: create or prewarm the successor JSONL transcript before updating `sessions.json` and queued follow-up state, preserve a continuity seed when available, and keep the prior valid mapping or fail explicitly if the successor cannot be made durable. Add regression coverage for missing/empty prior transcripts and `chat.history` after reset.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with a previous entry whose `sessionFile` is missing or has no replayable user/assistant tail, then assert the persisted successor `sessionFile` exists and `readSessionMessages`/`chat.history` does not silently return an empty history because of a missing file.

## Solution Assessment

No current solution is sufficient. The cleanup and doctor paths are useful diagnostics, but the maintainable fix is to enforce transcript durability at the reset/rotation write boundary rather than relying on later cleanup of a broken active mapping.
