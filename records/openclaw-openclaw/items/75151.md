## Summary

Keep open. Current main still allows the auto-reply context-overflow reset path to persist a new session mapping before the successor transcript is guaranteed durable, so the reported missing-transcript mapping remains a valid data-integrity bug rather than a stale or duplicate report.

## What This Changes

The issue requests that session reset or rotation after context overflow never leave `sessions.json` pointing at a nonexistent transcript and preserve or recover usable session history.

## Best Possible Solution

Enforce a session-reset durability invariant in the auto-reply/direct reset path: create or prewarm the successor JSONL before publishing it to `sessions.json`, replay a continuity tail when available, and keep the prior valid mapping or fail explicitly if the successor transcript cannot be made durable.

## Reproduction Assessment

Yes. A focused unit reproduction can call `resetReplyRunSession` with a source entry whose transcript is missing or has no replayable tail, then assert that the persisted successor `sessionFile` exists before `readSessionMessages` or `chat.history` can resolve it.

## Solution Assessment

Yes, the narrowest maintainable fix is at the reset write boundary. Cleanup and doctor checks are useful diagnostics, but they run after the active mapping has already been corrupted and do not preserve continuity.
