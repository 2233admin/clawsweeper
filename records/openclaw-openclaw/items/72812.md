## Summary

Keep open. Current main still has the shared-client teardown path that explains the linked bug, and this open contributor PR appears to address it without a definite correctness or security regression. It should receive normal maintainer validation rather than cleanup closure.

## What This Changes

The PR isolates Codex app-server startup for runs carrying `spawnedBy`, avoids clearing the shared singleton on isolated startup failures, adds abort/cleanup handling for isolated clients, adds regression tests, and includes formatting-only edits in unrelated files.

## Best Possible Solution

Land a narrow Codex app-server lifecycle fix that prevents spawned/helper auth failures from clearing the parent shared client, preserves shared-client behavior for ordinary runs, and keeps deterministic coverage for startup failure, startup timeout, turn-start failure, abort, and normal cleanup paths.

## Reproduction Assessment

Yes. The linked bug gives concrete invalid-token `btw spawn codex 5.5` steps and logs, and current main’s source shows the shared startup client plus unconditional shared-client clearing that explains the reported disconnect.

## Solution Assessment

Yes for the core behavior. Gating isolated startup on `spawnedBy` is the narrow maintainable fix; the safer merge path is to validate the Codex tests and changed gate, then decide whether to trim unrelated formatting churn or add the required changelog entry.
