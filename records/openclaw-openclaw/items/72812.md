## Summary

Keep open for normal maintainer review. Current main still has the shared Codex app-server startup path and unconditional shared-client clearing that match the linked bug, while the PR applies a narrow spawned-run isolation fix with focused regression coverage and no blocking review findings from this read-only pass.

## What This Changes

The PR routes spawned/helper Codex app-server runs to an isolated client when `spawnedBy` is present, avoids clearing the shared singleton for isolated startup failures, closes isolated clients across startup/turn/final cleanup paths, adds regression tests, and includes unrelated formatting-only edits in other areas.

## Best Possible Solution

Land a narrow Codex app-server lifecycle fix that keeps spawned/helper startup failures isolated from the shared singleton, preserves shared-client behavior for ordinary runs, and has deterministic cleanup coverage for startup, timeout, turn-start, abort, and normal completion paths. Maintainers may trim unrelated formatting churn before merge if they want a tighter diff.

## Reproduction Assessment

Yes. The linked bug has clear steps and logs for an invalid bearer-token spawned Codex run, and current main's startup path plus unconditional shared-client clearing gives a high-confidence source-level explanation for the reported disconnect.

## Solution Assessment

Yes for the core behavior. Gating isolated app-server startup on `spawnedBy` is the narrowest maintainable boundary for helper/spawned runs, while keeping normal runs on the existing shared client; the safer review path is to validate the focused tests and changed gate before merge.
