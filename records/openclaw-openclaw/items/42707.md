## Summary

Keep this PR open. Current main still lacks Telegram subagent lifecycle hook registration and still does not hydrate resumed agent runs from persisted session delivery context, so the PR's central behavior is not implemented on main. The patch is stale and needs rebase/cleanup, but it remains a relevant implementation candidate alongside the related open tracker #42986.

## Best Possible Solution

Keep this PR open as a stale but relevant implementation candidate. Rebase it onto current main; add Telegram `registerFull` lifecycle hook wiring like Discord, Matrix, and Feishu; keep Telegram hook code inside the plugin boundary through SDK/local API facades; reuse the existing Telegram thread-binding manager and session-binding adapter; hydrate `resolveAgentRunContext` from the resolved `SessionEntry` on `--session-id` resumes; normalize legacy Telegram target prefixes; and add focused tests for Telegram `sessions_spawn(thread=true, mode="session")`, resume routing hydration, delivery-target resolution, and unbind cleanup. The related open issue #42986 can continue tracking the feature until a corrected implementation lands.

- The submitted patch is not merge-ready as-is because its Telegram production hook crosses the extension boundary with relative imports into core internals; a rebased fix should use `openclaw/plugin-sdk/conversation-runtime` or add a documented SDK seam.

- The PR targets older code shape in several places. Current main already has `defineBundledChannelEntry`, a plugin SDK conversation runtime surface, Telegram thread-binding state sharing, and a global session-binding adapter map, so a rebase should reuse those instead of overwriting them.
