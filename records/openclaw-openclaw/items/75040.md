## Summary

Keep open. Current main still has a credible `thinking` collision path: provider wrappers can write `payload.thinking`, then the generic OpenAI-compatible `extra_body` wrapper runs later, warns, and overwrites the same key. The DeepSeek disabled-thinking failure is already tracked by open related work, but the broader request still needs maintainer scoping because docs currently promise final `extra_body` precedence and the native Ollama `think` path is separate.

## What This Changes

The issue asks OpenClaw to stop duplicate `thinking` request-payload writes and warning spam, while making DeepSeek and Ollama disabled-thinking behavior reliable.

## Best Possible Solution

Define one canonical payload-precedence rule: keep `extra_body` as the explicit final override, suppress or downgrade no-op `thinking` collisions, normalize disabled-thinking aliases before provider wrapping, ensure disabled DeepSeek V4 payloads cannot replay stale `reasoning_content`, and keep native Ollama `think` handling inside the Ollama provider path unless a focused reproduction proves more is needed.

## Reproduction Assessment

Yes for the core collision path: configure an OpenAI-compatible DeepSeek V4 request where the provider wrapper writes `payload.thinking` and `params.extra_body.thinking` is also configured, then inspect the composed payload after post-plugin wrapping. The Ollama stuck-session portion is plausible from the native `think` mapping and thinking-only output handling, but it is not fully reproduced by the source inspection alone.

## Solution Assessment

Unclear. The related PRs address concrete DeepSeek subproblems, but the broader fix needs maintainer scoping so it avoids breaking the documented rule that `extra_body` wins on key collisions.
