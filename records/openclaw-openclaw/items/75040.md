## Summary

Keep open. Current main still has the reported OpenAI-compatible collision path: provider wrappers can write top-level `thinking`, and the later `extra_body` wrapper warns and overwrites that same key. The DeepSeek disabled-thinking subproblems are only partially addressed by related open work, and the Ollama impact follows a separate native `think` path that needs focused scoping rather than issue cleanup.

## What This Changes

The issue requests eliminating duplicate `thinking` request-payload writes and warning spam from `extra_body` collisions while preserving reliable disabled-thinking behavior for DeepSeek and Ollama.

## Best Possible Solution

Define and implement a canonical payload-precedence rule that preserves `extra_body` as the explicit final override, normalizes disabled-thinking aliases before provider wrapping, avoids noisy no-op `thinking` collisions, and keeps native Ollama `think` handling inside the Ollama provider unless a focused reproduction proves a remaining bug there.

## Reproduction Assessment

Yes for the core collision path. Configure an OpenAI-compatible DeepSeek V4 request where the provider wrapper writes `payload.thinking` and `params.extra_body.thinking` is also configured; current main warns and overwrites during post-plugin wrapping.

## Solution Assessment

Unclear. The related DeepSeek PRs address concrete subproblems, but the broader framework behavior needs maintainer scoping so a fix preserves the documented rule that `extra_body` wins on request-key collisions.
