## Summary

Keep open. Current main still has a credible OpenAI-compatible payload collision path: provider wrappers can write `thinking`, then the later `extra_body` wrapper warns and overwrites the same key. The related DeepSeek repair PR is still open and only covers part of the report, while the Ollama impact needs separate scoping because native Ollama uses top-level `think`, not OpenAI-compatible `thinking`.

## What This Changes

The issue requests a framework-level fix so `thinking` is not written twice through provider wrappers and `params.extra_body`, and so disabled thinking works reliably for DeepSeek and Ollama-style local models.

## Best Possible Solution

Keep the issue open and define a canonical payload precedence rule: preserve `extra_body` as the explicit final override, suppress or normalize noisy no-op `thinking` collisions, handle disabled-thinking aliases consistently, ensure disabled DeepSeek payloads cannot replay stale `reasoning_content`, and touch native Ollama only if a focused `think` reproduction proves a separate bug.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a DeepSeek V4 OpenAI-compatible request where the provider wrapper writes `payload.thinking` and the model config also supplies `params.extra_body.thinking`; current main will warn and overwrite in the post-plugin wrapper.

## Solution Assessment

Unclear. The open DeepSeek PR addresses one concrete disabled-thinking failure mode, but the broader framework warning/noise and Ollama claims need maintainer scoping so the fix does not break the documented `extra_body` override contract.
