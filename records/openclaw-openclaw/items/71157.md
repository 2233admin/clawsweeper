## Summary

Keep this issue open. Current main still lacks the requested WhatsApp channel/account expiration config, resolved account state has no effective disappearing-message value, and the active outbound text, media, voice-note companion text, and poll paths do not forward Baileys `ephemeralExpiration`; the two related implementation PRs are closed and unmerged.

## What This Changes

The issue requests WhatsApp channel and per-account configuration that applies a default disappearing-message expiration to outbound OpenClaw replies.

## Best Possible Solution

Implement this in the bundled WhatsApp plugin by adding optional channel/account config and schema/docs metadata, resolving a positive effective expiration with account override semantics, and merging it into Baileys send options on every outbound transport path while preserving existing behavior when unset, zero, or null.

## Reproduction Assessment

Yes. A current-main source check reproduces the absence: the strict config schema lacks the proposed keys, resolved account state has no expiration field, and outbound send code never passes `ephemeralExpiration` to Baileys.

## Solution Assessment

Yes. The requested direction is the narrowest maintainable fix because this is WhatsApp plugin-owned transport metadata, not a generic core channel contract; the pinned Baileys dependency already exposes the needed send option.
