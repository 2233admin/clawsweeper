## Summary

Keep this PR open. Current main has a related Windows output decoder, but the default/non-Windows path still decodes each Buffer with `buffer.toString("utf8")`, so it does not definitely implement the PR’s intended UTF-8 boundary-safe child stdout/stderr decoding. The PR is a focused fix, but it needs to be rebased onto the current decoder so it preserves Windows codepage handling and adds regression coverage for split UTF-8 output.

## Best Possible Solution

Keep the PR open and port the fix onto current main by making the shared child-output decoder boundary-safe for default UTF-8 streams while preserving Windows legacy-codepage fallback. Add focused tests for split UTF-8 chunks on the default path and for stdout/stderr flush behavior, then review or land the rebased PR against the open linked bug.
