## Summary

Keep open. The requested Signal `configPath` feature is valid and upstream `signal-cli` supports a global `--config` option, but this PR is not merge-ready because it wires runtime behavior into obsolete `src/signal/*` files while current main ships the bundled Signal plugin from `extensions/signal/*`.

## What This Changes

This PR adds optional `channels.signal.configPath` schema/type/help/docs/configured-state entries and legacy Signal runtime/tests intended to pass a trimmed `--config` argument when autostarting `signal-cli`.

## Best Possible Solution

Port optional top-level and per-account Signal `configPath` support through the current bundled plugin end to end: strict schema/types, plugin UI hints/help, generated channel metadata, docs, account resolution, monitor autostart propagation, daemon argv construction before `daemon`, blank-value filtering, focused tests, and a user-facing changelog credit for the source contribution.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: `channels.signal.configPath` is absent from strict Signal config validation/types, and the current Signal monitor/daemon cannot pass a config directory to `signal-cli`.

## Solution Assessment

No. The feature direction is sound, but this PR edits obsolete runtime paths; the maintainable solution is a narrow replacement patch against `extensions/signal` plus the current config/docs metadata surfaces.
