## Summary

Keep open. The requested Signal `configPath` support is still missing on current `main`, and the PR is useful source work but not merge-ready because it targets the old `src/signal` runtime while the shipped Signal implementation now lives under `extensions/signal`. The discussion already has maintainer guidance to port the idea, and the author explicitly said they cannot continue the branch.

## What This Changes

The PR adds an optional Signal `configPath` setting to config docs/schema/help/types and passes it as `--config` when the old Signal daemon autostarts.

## Best Possible Solution

Port optional top-level and per-account `configPath` support through the current bundled Signal plugin end to end: strict schema/types, generated channel metadata, UI/help/docs, account resolution, monitor autostart propagation, daemon argv construction before the `daemon` command, blank-value filtering, focused tests, and changelog credit for the source contribution.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: `channels.signal.configPath` is absent from the strict Signal schema/types, and the current Signal monitor/daemon cannot pass any config directory to `signal-cli`.

## Solution Assessment

No. The concept is valid, but this PR implements it in obsolete `src/signal` files; the maintainable fix is a narrow replacement patch against `extensions/signal` and the current generated config surfaces.
