## Summary

Current main still does not support externally managed CDP/noVNC endpoints for sandbox browsers. The sandbox browser path remains Docker-lifecycle based, non-Docker sandbox backends are rejected when browser is enabled, and the config/schema/docs expose only Docker browser settings. The issue is valid and recent, with useful follow-up context around Lightpanda and the broader but closed pluggable-provider PRs, so no conservative close reason applies.

## Best Possible Solution

Keep this issue open. The best implementation is a core sandbox-browser seam: add validated agents.defaults.sandbox.browser external CDP/noVNC endpoint settings or a backend-owned browser endpoint capability, skip Docker lifecycle and Docker registry cleanup only in that explicit mode, preserve current Docker defaults, update config types/zod/generated schema/help/docs, and add regression tests proving a non-Docker sandbox backend can use browser automation without requiring a Docker daemon.

## Risks / Open Questions

- External CDP/noVNC support is security-sensitive: endpoint validation, SSRF/private-network policy, authentication, token redaction, and noVNC exposure need explicit design review.
- Global browser remote-CDP profiles can be mistaken for this feature, but they do not cover sandbox browser lifecycle, noVNC observer URL injection, or non-Docker backend behavior.
- The current plugin sandbox backend API only exposes a browser capability boolean; external plugins cannot supply a browser endpoint/noVNC lifecycle contract without a core seam.
- The broader #41437 approach was closed and #42860 explicitly scoped browser support out, so the remaining work should be a narrow additive design rather than assuming a prior larger PR solved it.
