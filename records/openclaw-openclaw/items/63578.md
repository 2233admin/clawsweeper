## Summary

Keep this PR open. Current main still does not hydrate media/file hints for plugin-handled `send` actions, and the PR is useful because its latest opt-in shape addresses the earlier unconditional-preload concern. It is not ready for maintainer cleanup or merge as-is because it changes a public Plugin SDK type without updating the generated SDK API baseline.

## What This Changes

The PR adds an opt-in `preloadSendMedia` channel action flag, threads it into `hydrateAttachmentParamsForAction` for `send` actions with media/file hints, and adds outbound unit/runner tests for default non-preload and opt-in hydration behavior.

## Best Possible Solution

Land a reviewed opt-in plugin send-media preloading contract, or an equivalent generic seam, with the SDK API baseline/docs updated and focused tests preserving default URL/path forwarding for handlers that already load media inside their own runtime.
