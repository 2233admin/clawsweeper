## Summary

Keep this PR open. Current main still does not hydrate media/file hints for plugin-handled `send` actions, and the PR remains a relevant implementation candidate because its latest diff makes send-media preloading opt-in instead of changing every send path. It is not implemented on main and is not cleanly superseded by the Telegram/buffer-only related work.

## What This Changes

The PR adds an opt-in `preloadSendMedia` channel action flag, threads it into `hydrateAttachmentParamsForAction` for `send` actions with media/file hints, and adds outbound unit/runner tests for default non-preload and opt-in hydration behavior.

## Best Possible Solution

Land a reviewed opt-in plugin send-media preloading contract, or an equivalent generic seam, with focused tests and SDK/plugin documentation while preserving default URL/path forwarding for handlers that already load media inside their own runtime. Keep the related Telegram, Discord, and buffer-only reports coordinated but separate unless one replacement PR intentionally covers the whole cluster.
