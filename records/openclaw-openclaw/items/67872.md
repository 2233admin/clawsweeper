## Summary

Keep this PR open. Current main still drops Slack mention-skipped channel messages before hook dispatch, so the requested behavior is not obsolete, but the submitted patch reads `ingest` from the generic group-policy surface that Slack does not expose as valid public config.

## What This Changes

The PR modifies Slack message preparation to fire an internal `message:received` hook from the no-mention skip path when generic group ingest policy is true, and adds silent-ingest regression tests.

## Best Possible Solution

Add a Slack-owned `ingest` boolean to `channels.slack.channels` with wildcard and channel-specific override semantics, wire it through the Slack type, strict schema, config metadata/docs, and `resolveSlackChannelConfig`, then emit `message:received` from the mention-skip path only when that resolved Slack opt-in is true.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Read ingest from Slack channel config:** `extensions/slack/src/monitor/message-handler/prepare.ts:550-555`
  - body: This calls `resolveChannelGroupPolicy`, which only reads `channels.slack.groups`, but Slack's valid public policy surface is `channels.slack.channels` and the strict Slack schema has no `groups` or channel `ingest` field. As a result, normal Slack configs cannot make `ingestEnabled` true, so the new hook path will not run for the intended users.
  - confidence: 0.93

## Security Review

Status: needs_attention

Summary: The patch intentionally forwards mention-skipped Slack channel content to hooks, but the opt-in is tied to an undocumented invalid config surface.

Concerns:

- **[medium] Use an explicit Slack ingest opt-in:** `extensions/slack/src/monitor/message-handler/prepare.ts:550`
  - body: Mention-skipped channel messages are content that users did not direct at the bot; forwarding them to hooks should be controlled by a documented Slack config field and schema. The current patch gates this on generic `groups.ingest`, which Slack does not expose as a valid public setting.
