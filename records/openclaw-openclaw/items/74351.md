## Summary

Keep open. Current main still blocks IPv6 ULA DNS answers under web_fetch even when the RFC2544 fake-IP opt-in is enabled, and the nearest attempted fix was closed unmerged after an SSRF-widening warning, so this needs maintainer/security design rather than cleanup closure.

## What This Changes

The issue asks for web_fetch's fake-IP proxy SSRF opt-in to cover IPv6 ULA `fc00::/7` answers as well as IPv4 `198.18.0.0/15`, or to expose a separate ULA opt-in.

## Best Possible Solution

Make an explicit maintainer/security decision: either add a separate default-off IPv6 ULA fake-IP opt-in with schema/types/help/docs/cache-key/audit/test coverage, or keep ULA blocked and document that the current RFC2544 option intentionally covers only `198.18.0.0/15`.
