## Summary

Keep this open. Current `main` at `581fbea1d65355917886babffe4161f8b3178a12` still leaves the WebChat and Gateway-backed TUI `chat.send` reply path outside the hook-aware outbound delivery seams for `message_sending`, `message_sent`/internal `message:sent`, and Gateway-injected assistant transcript writes; the broader and narrower related reports remain open, and the earlier lifecycle-hook PR was closed unmerged.

## What This Changes

The issue requests WebChat and `openclaw-tui` outbound assistant replies to fire lifecycle hooks such as `message_sending`, `message_sent`, `before_message_write`, and `agent_end`.

## Best Possible Solution

Land a narrow Gateway chat hook coverage fix before the broader #50126 architecture work: route `chat.send` replies through a hook-aware delivery seam or equivalent local wrapper, emit plugin `message_sent` and internal `message:sent` once for delivered or failed final replies, run `before_message_write` before Gateway-injected assistant transcript appends, and add focused WebChat/Gateway TUI regression tests. Verify `agent_end` in the same tests, but avoid changing it unless the focused test proves a current-main gap.

## Reproduction Assessment

Yes for the title hooks: WebChat and Gateway-backed TUI both go through `chat.send`, and the source trace shows that path bypasses the hook-aware dispatcher and write guard. The `agent_end` claim is less certain because current source shows runner-level `agent_end` wiring, so it needs a focused regression check rather than an assumed fix.

## Solution Assessment

Yes, the issue’s local-fix direction is the best next step: fix the Gateway chat path with existing hook seams and leave the full cross-channel observer architecture to #50126. A safer alternative to a broad rewrite is to add a small shared helper only if needed to preserve exactly-once semantics for final delivery and transcript writes.
