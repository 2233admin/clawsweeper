## Summary

Keep this issue open. Current main still routes QQBot inbound messages through the gateway startup config snapshot, while `bindings` is classified as no-restart config; the linked closing PR is still open and unmerged, so closing this before that PR lands would be premature.

## What This Changes

The issue reports that QQBot peer-specific routing bindings added at runtime are visible in the CLI but are not applied to inbound QQBot messages until the gateway restarts.

## Best Possible Solution

Keep the issue open until PR #73567 is reviewed and either merged or deliberately closed. The desired end state is a narrow QQBot-side fix that resolves the active runtime config for each inbound routing decision, passes that same fresh config through dependent inbound/outbound work, and adds regression coverage for account-binding to peer-binding reroutes without a gateway restart.

## Reproduction Assessment

Yes. The reporter provided concrete steps and logs, and current main's source path still shows QQBot routing from `ctx.cfg` while `bindings` is no-restart config; I did not run a live QQBot account because this review is read-only.

## Solution Assessment

Yes. Fetching the active runtime config at the QQBot gateway boundary is the narrowest maintainable fix; changing the generic route cache or treating all binding edits as restart-required would be broader and would conflict with current reload semantics.
