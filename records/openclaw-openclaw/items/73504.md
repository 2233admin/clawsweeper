## Summary

Keep open. Current main already has the merged #73521 read/search-only Discord execution-mode resolver and tests, so this PR's unconditional gateway resolver is too broad; however, current main still does not forward that resolver through the registered Discord channel wrapper, so the PR contains useful remaining work and should be adapted rather than closed.

## What This Changes

The PR adds a Discord message-action execution-mode resolver that always returns gateway and forwards the Discord action resolver through the registered channel wrapper.

## Best Possible Solution

Update this PR so the adapter keeps the current read/search-only gateway resolver, the registered Discord wrapper forwards that resolver, and focused coverage proves registered `discordPlugin.actions` routes read/search to gateway while leaving other Discord actions local.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: the registered Discord wrapper on current main lacks `resolveExecutionMode`, and the shared runner defaults a missing resolver to local; the PR regression is also source-level because its resolver returns gateway for every action while current tests require non-read/search actions to stay local.

## Solution Assessment

No, not as-is. Forwarding the resolver through the wrapper is the narrow maintainable fix, but replacing the adapter behavior with unconditional gateway execution is broader than the reported read/search hang and conflicts with the current narrowed resolver contract.
