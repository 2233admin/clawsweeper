## Summary

Keep open. The merged #73521 work fixed much of the original Discord read/search and hook report, but current main still does not forward the Discord action resolver through the registered channel wrapper. This PR contains that useful wrapper-forwarding change, but its adapter change is now too broad because it would route all Discord actions through gateway mode instead of preserving the current read/search-only contract.

## What This Changes

The PR adds an unconditional Discord message-action gateway resolver and forwards that resolver through the registered Discord channel wrapper.

## Best Possible Solution

Update this PR or a narrow replacement so the registered Discord wrapper forwards the existing read/search-only resolver, add plugin-level coverage for that wrapper behavior, and leave the #73521 REST timeout and hook fallback intact.

## Reproduction Assessment

Yes. Source-level reproduction is high-confidence: current main’s registered Discord wrapper lacks `resolveExecutionMode`, and the shared runner defaults a missing resolver to local execution. The PR regression is also source-level reproducible because its resolver is unconditional while current tests require non-read/search Discord actions to stay local.

## Solution Assessment

No, not as-is. Forwarding the wrapper resolver is the right narrow fix, but replacing the adapter behavior with unconditional gateway execution is broader than the reported read/search hang and conflicts with the current resolver contract.
