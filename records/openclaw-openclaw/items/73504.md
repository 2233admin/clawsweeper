## Summary

Keep open. The wrapper-forwarding hunk is still useful because current main's registered Discord plugin actions wrapper does not forward `resolveExecutionMode` and the shared runner defaults a missing resolver to local, but the PR's adapter hunk is now too broad because #73521 already added a read/search-only resolver and current tests require other Discord actions to remain local.

## What This Changes

The PR adds a Discord message-action execution resolver and forwards that resolver through the registered Discord channel wrapper so CLI Discord message actions can be routed through gateway execution.

## Best Possible Solution

Update this PR so the production change is the wrapper forwarding in `extensions/discord/src/channel.ts`, preserve #73521's read/search-only adapter resolver, and add registered-plugin coverage proving Discord read/search route through gateway while other actions stay local.

## Reproduction Assessment

Yes. At source level, `discordPlugin` registers the wrapper object from `channel.ts`, that wrapper lacks `resolveExecutionMode`, and the runner defaults missing resolvers to local; the PR regression is also reproducible by comparing its unconditional resolver with tests that require non-read/search actions to stay local.

## Solution Assessment

No, not as-is. Forwarding the resolver through the wrapper is the narrow maintainable fix, but replacing the adapter resolver with unconditional gateway execution widens behavior beyond the reported read/search hang.
