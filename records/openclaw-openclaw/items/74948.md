## Summary

Keep open. Current main still treats a live-PID runtime-deps lock as fresh before legacy age or mtime fallback can run, and the provided discussion links open PR #74950 from the same reporter as the implementation candidate.

## What This Changes

Expire legacy bundled runtime-deps locks with a live or reused PID but no start-time or creation-time evidence once filesystem mtimes show the lock is stale.

## Best Possible Solution

Review and land PR #74950, or make an equivalent narrow predicate change with regression coverage for live-PID legacy locks missing start-time and creation-time evidence while preserving active lock safety for matching start-time owners.

## Reproduction Assessment

Yes. A source-level reproduction is `shouldRemoveRuntimeDepsLock({ pid: 7, lockDirMtimeMs: 1_000, ownerFileMtimeMs: 2_000 }, 1_000_000, () => true, () => null)`, which current main reaches the live-PID `return false` path for.

## Solution Assessment

Yes. The narrowest maintainable fix belongs in `src/plugins/bundled-runtime-deps-lock.ts` with focused tests in `src/plugins/bundled-runtime-deps.test.ts`; a broader locking rewrite is not required for this edge case.
