## Summary

Keep open. Current main still exits the live-PID branch before checking creation-time or filesystem mtime fallback, so the reported legacy Docker lock path remains valid; the linked PR #74950 is still the paired implementation candidate.

## What This Changes

Request to expire stale legacy bundled runtime-deps lock directories when a live reused PID is present but owner start-time and creation-time evidence are absent.

## Best Possible Solution

Land PR #74950 or an equivalent narrow change in `shouldRemoveRuntimeDepsLock()` with regression coverage for live-PID legacy locks missing `starttime` and `createdAtMs`, while preserving active-lock safety when matching start-time evidence exists.

## Reproduction Assessment

Yes. A source-level reproduction on current main is `shouldRemoveRuntimeDepsLock({ pid: 7, lockDirMtimeMs: 1000, ownerFileMtimeMs: 2000 }, 1_000_000, () => true)`, which returns `false` because the live-PID branch exits before the stale mtime fallback.

## Solution Assessment

Yes. The narrow lock predicate fallback plus focused tests is the maintainable fix; the broader runtime-deps repair work in #75183 does not claim this root cause.
