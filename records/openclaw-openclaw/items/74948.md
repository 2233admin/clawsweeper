## Summary

Keep open. Current main still short-circuits any live-PID runtime-deps lock after the optional start-time mismatch check, so a legacy Docker lock with a reused live PID and no start-time evidence never reaches the filesystem-mtime fallback; the same reporter has an open paired fix PR that should be reviewed instead of closing this issue.

## What This Changes

The issue requests expiring legacy bundled runtime-deps locks that lack start-time and creation-time evidence by falling back to lock-directory or owner-file mtimes after the live-PID check.

## Best Possible Solution

Review and land PR #74950, or make the same narrow predicate change with regression coverage for live-PID legacy locks that lack start-time and creation-time evidence while preserving active-lock safety for matching start-time owners.

## Reproduction Assessment

Yes. A source-level reproduction is `shouldRemoveRuntimeDepsLock({ pid: 1, lockDirMtimeMs: 1_000, ownerFileMtimeMs: 2_000 }, 1_000_000, () => true)`, which current main returns `false` because the live-PID branch exits before stale-mtime fallback.

## Solution Assessment

Yes. The narrow maintainable fix belongs in `src/plugins/bundled-runtime-deps-lock.ts` with focused coverage in `src/plugins/bundled-runtime-deps.test.ts`; a broader lock rewrite is not needed for this edge case.
