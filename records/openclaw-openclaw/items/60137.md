## Summary

Keep this PR open. Current main still lacks the two infra behaviors the PR targets: Slack request wrappers with empty/no diagnostic messages are not classified as transient, and null or undefined unhandled-rejection reasons still fall through to the fatal exit path. The patch is narrow and appears functionally sound, but the global null/undefined suppression is a maintainer policy choice rather than something ClawSweeper should auto-close or replace.

## What This Changes

The PR adds Slack request-wrapper fallback detection to `isTransientNetworkError`, treats null or undefined unhandled rejection reasons as non-fatal, and adds focused regression tests for those cases.

## Best Possible Solution

Review and land this PR or an equivalent scoped fix that adds regression coverage for the empty Slack request-wrapper case and the null/undefined Socket Mode rejection case. If maintainers do not want a global null/undefined policy, the safer alternative is a Slack-owned registered handler or normalization path while preserving the reported stability fix.

## Reproduction Assessment

Yes. A focused unit-level reproduction is clear: construct `Object.assign(new Error("A request error occurred: "), { code: "slack_webapi_request_error", original: new Error("") })` or emit an unhandled rejection with `undefined`; current main falls through to false/fatal paths by inspection.

## Solution Assessment

Unclear as a product policy question. The Slack wrapper classifier is a narrow maintainable fix, but globally suppressing null/undefined rejection reasons is broader than Slack and should be explicitly accepted by maintainers or narrowed to the Slack Socket Mode path.
