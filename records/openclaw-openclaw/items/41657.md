## Summary

Keep this issue open. Current main does not fully strip Slack files anymore, but it still only exposes Slack attachments through placeholders plus legacy `MediaPath(s)`/`MediaTypes`; the requested structured agent-visible metadata contract for file id/name/MIME/size is still missing.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata contract to `MsgContext` and the plugin SDK, populate it from Slack `message.files` while preserving current `BodyForAgent` placeholders and `MediaPath(s)` behavior, document the new field, and cover it with Slack plus shared channel contract tests. Review #62792 separately as the narrower Slack media hydration fix.

## Risks / Open Questions

- A Slack-only metadata field would create channel contract drift; the durable path should be a shared additive inbound attachment model across Slack, Discord, Telegram, and other channels.
- Attachment names, MIME types, and sizes are provider/user-controlled metadata. Any implementation should mark them untrusted and avoid exposing Slack private URLs or bearer-token-derived download URLs to agents.
