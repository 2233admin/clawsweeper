## Summary

Current main still does not deliver passive structured Slack attachment metadata to agents. Slack retains file id/name/MIME/size at event type level and can download files into legacy media fields, but the final inbound context is still placeholders plus MediaPath/MediaPaths/MediaTypes, with no shared InboundAttachments-style contract.

## Best Possible Solution

Keep this issue open and implement a backward-compatible shared inbound attachment metadata contract in MsgContext/plugin SDK, then populate it from Slack message.files with provider, provider file id, file name, MIME type, size, downloaded/local-path state, and placeholder text. Preserve existing BodyForAgent placeholders and MediaPath/MediaPaths/MediaTypes behavior, add Slack/shared contract tests, update Slack/plugin docs, and review #62792 separately as narrower media hydration work.

## Clownfish Work Prompt

Implement a backward-compatible shared inbound attachment metadata contract for agents, focused on Slack first.

Observable problem: Slack inbound file events have file id/name/MIME/size, but final agent context only receives placeholders and legacy MediaPath/MediaPaths/MediaTypes. Agents should passively receive structured metadata for Slack attachments without polling Slack.

Expected boundary:
- Add a generic typed metadata field to `MsgContext` / `FinalizedMsgContext`, for example `InboundAttachments?: InboundAttachment[]`, with fields such as `provider`, `kind?`, `fileName?`, `mimeType?`, `sizeBytes?`, `providerFileId?`, `localPath?`, `downloaded`, and `placeholder?`.
- Export the contract through the existing plugin SDK reply/runtime surface as needed; keep it additive and backward compatible.
- Populate the field from Slack `message.files` in the Slack inbound prepare/content path, preserving order and covering both successful downloads and placeholder fallback.
- Preserve existing `BodyForAgent` / `RawBody` placeholders and `MediaPath` / `MediaPaths` / `MediaTypes` behavior.
- Do not expose Slack `url_private`, bearer tokens, or token-authenticated URLs by default.
- Do not change Slack auto-download limits, fetch policy, or unrelated media hydration behavior.
- Update Slack docs and plugin/channel SDK docs if the field is public.
- Add a user-facing changelog entry in the active version with attribution, likely `Thanks @iret77` and `Thanks @neeravmakwana`.

Related refs: this issue, #62792 only as adjacent media hydration work.
