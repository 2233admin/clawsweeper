## Summary

Keep this issue open. Current main no longer fully strips Slack files: Slack models file id/name/MIME/size, downloads accessible files into legacy media fields, emits `[Slack file: ... (fileId: ...)]` placeholders, and documents the `download-file` action. The requested agent-visible structured attachment metadata is still missing from the shared inbound context and Slack finalized context.

## Best Possible Solution

Keep this issue open. Add a backward-compatible shared inbound attachment metadata contract to `MsgContext` and the plugin SDK, populate it from Slack `message.files` while preserving current `BodyForAgent` placeholders and `MediaPath(s)` behavior, document the new field, and add Slack plus shared channel contract tests. Review #62792 separately as the narrower Slack media hydration PR.

## Risks / Open Questions

- A Slack-only metadata field would create channel contract drift; the durable implementation should be additive and shared across channel plugins.
- Attachment names, MIME types, and sizes are provider/user-controlled metadata. Future implementation should treat them as untrusted and avoid exposing Slack private URLs or bearer-token-derived download URLs to agents.
