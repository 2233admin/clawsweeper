## Summary

Current main still partially handles Slack files through downloads, legacy MediaPath/MediaPaths, and text placeholders, but it does not implement the requested passive structured attachment metadata contract with file id, name, MIME type, and size. This remains actionable OpenClaw Slack/plugin-SDK work, not ClawHub cleanup or a duplicate of the narrower open Slack media hydration PR.

## Best Possible Solution

Keep this issue open and implement a backward-compatible shared inbound attachment metadata contract in MsgContext and the plugin SDK, then populate it from Slack message.files while preserving BodyForAgent placeholders and MediaPath/MediaPaths/MediaTypes. Add Slack and shared channel contract tests, document the new field, and review #62792 separately as the narrower Slack media hydration fix.

## Risks / Open Questions

- Closing this now would leave agents dependent on filename/fileId placeholders, legacy media paths, or active download-file calls instead of receiving passive structured file metadata.
- A Slack-only ad hoc field would likely drift from Discord and Telegram attachment behavior; the cleaner path is an additive shared inbound attachment model.
- Attachment names, MIME types, and sizes are provider/user-controlled and should be treated as untrusted metadata; private Slack URLs should not be exposed directly to agents.
