## Summary

Keep this issue open. Current main no longer fully drops Slack files, but it still exposes them to agents through placeholder text plus legacy MediaPath/MediaPaths/MediaTypes. The requested structured inbound file metadata contract for file id, name, MIME type, and size is still missing, and the related open PR #62792 is a narrower Slack media hydration fix rather than a canonical implementation of this metadata request.

## Best Possible Solution

Add a backward-compatible shared inbound attachment metadata contract to MsgContext and the plugin SDK, populate it from Slack message.files while preserving current BodyForAgent placeholders and MediaPath(s)/MediaTypes behavior, document the new field, and cover it with Slack plus shared channel contract tests. Review #62792 separately as the narrower Slack media hydration fix.

## Risks / Open Questions

- A Slack-only field would create channel contract drift; the durable fix should be a shared additive inbound attachment model across channel plugins.
- Attachment names, MIME types, and sizes are provider/user-controlled metadata. Any implementation should mark them untrusted and avoid exposing Slack private URLs or bearer-token-derived download URLs to agents.
