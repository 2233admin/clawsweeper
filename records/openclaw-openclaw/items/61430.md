## Summary

Keep this issue open. Current main still lacks the requested optional exec `description`/`intent` field in the model-facing tool schema, exec runtime params, approval protocol, approval UI, and public docs; the related #61562 was closed as a duplicate of this older tracker, not as a superseding fix.

## What This Changes

Request an optional exec tool description or intent parameter that models can populate and hooks, approval prompts, and audit logs can consume as informational metadata.

## Best Possible Solution

Keep this open and design an additive optional `description` or `intent` field as untrusted display/audit metadata. If maintainers approve the direction, thread it through the exec schema, runtime params, hook-visible params, approval request protocol, approval UI/view model, docs, and focused tests while keeping command-based safety checks authoritative.

## Reproduction Assessment

Not applicable as a feature request. The high-confidence verification path is static inspection of the current exec schema/runtime, approval schema, hook types, UI, and docs, all of which currently lack the requested field.

## Solution Assessment

Unclear until maintainer/security review sets the trust semantics. The narrow additive field is plausible, but the best implementation must explicitly document that intent text is informational and must not replace command-based approval policy.
