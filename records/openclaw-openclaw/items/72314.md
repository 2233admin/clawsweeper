## Summary

Keep this PR open. Current main still requires `MessageSid` for inbound dedupe, so the idless retry problem described by the PR is not implemented on main. The PR remains a focused candidate, but it needs maintainer review/rebase onto the current route-key shape and preservation of provider-id key compatibility.

## Best Possible Solution

Keep this PR open for maintainer review. Best path: rebase it onto current main, implement the idless content/timestamp fallback using the current route-key/JSON key shape, preserve raw provider-id key compatibility, add/keep regression tests for idless retries and non-collapsing same-body different-timestamp messages, then run the targeted inbound-dedupe test and changed gate before merge.
