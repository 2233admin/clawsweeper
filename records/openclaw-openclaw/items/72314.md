## What This Changes

The PR adds a route-scoped SHA-256 content/timestamp fallback dedupe key for idless inbound auto-reply messages, extends inbound dedupe tests, and adds a changelog entry.

## Best Possible Solution

Land the PR after maintainer review; if a channel later needs attachment-aware or retimestamp-tolerant idless dedupe, handle that as a focused follow-up with channel-specific evidence.

## Reproduction Assessment

Yes. On current main, an inbound context with provider/peer data but no MessageSid produces no dedupe key, and dispatch only skips duplicate or inflight claims; the PR's focused tests exercise the fixed idless retry and timestamp-boundary cases.

## Solution Assessment

Yes. The latest patch is the narrowest maintainable solution found here: it changes only inbound dedupe key construction, preserves raw provider-ID suffixes, and requires both text content and Timestamp before creating a fallback key.
