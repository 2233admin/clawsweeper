## Summary

Keep this PR open. Current main still does not restore hook-visible TUI `/new` resets: `/new` only switches to a generated `tui-*` key, while `/reset` is still the path that calls `sessions.reset`. The linked regression issue is still open and this PR uses closing syntax for it, so the best maintainer path is review and refinement of this implementation candidate, not cleanup closure.

## Best Possible Solution

Keep the PR open and review it as the active implementation candidate for the linked regression. The best fix should make TUI `/new` emit `command:new` through the normal reset hook path while preserving per-client `tui-*` isolation, passing the correct old-session context to bundled hooks, avoiding blank session-list clutter, and documenting/validating any additive Gateway protocol field if that field remains part of the solution. After addressing the review comments, run the targeted TUI command-handler and gateway sessions reset tests before merge.
