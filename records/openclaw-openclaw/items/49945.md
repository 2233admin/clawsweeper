## Summary

Keep this PR open. Current main still does not implement the intended TUI `/new` hook-visible reset behavior: `/new` only switches to a unique local `tui-*` session, while `/reset` still uses `sessions.reset`. The PR remains an implementation candidate for open regression #49918, but its protocol/context changes still need maintainer review before merge.

## Best Possible Solution

Keep the PR open and review it as the implementation candidate for #49918. The best fix should make TUI `/new` emit `command:new` through the normal hook-visible reset path while preserving per-client `tui-*` isolation, keep hook context and `event.sessionKey` aligned with the session being ended, avoid blank session-list clutter, and either avoid or properly constrain any Gateway protocol addition. Before merge, clean up changelog attribution and run the targeted TUI command-handler and gateway `sessions.reset` tests.

## Risks / Open Questions

- The PR's additive `sessions.reset.hookSourceKey` field changes the Gateway protocol surface and should be explicitly authorized, documented, or kept internal before merge.
- Review comments identify a remaining attribution risk: hooks may receive `event.sessionKey` for the fresh `tui-*` session while archiving the transcript from the session being left.
- Calling `sessions.reset` on a brand-new `tui-*` key may persist blank session rows before the user sends a message, which can clutter session history.
- The PR changelog entry in the provided diff lacks the required `Thanks @author` attribution.
