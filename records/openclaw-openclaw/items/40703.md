## Summary

Keep this PR open, but do not merge it as-is. Current main still does not have a fail-fast default-model auth guard before channel startup, so the underlying change remains potentially useful; the branch is stale against the current startup split and its guard would reject valid runtime auth modes by using the narrower env/profile helper.

## What This Changes

The PR adds a gateway startup auth guard and focused tests intended to fail before channel startup when the resolved default model provider has no configured auth.

## Best Possible Solution

Revise the guard into the current post-attach sidecar path immediately before channel startup, honor both channel/provider skip flags, use the current runtime auth availability contract with config/workspace/agent context, avoid circular remediation text, and cover env, config, profile, AWS, local, and plugin auth modes in focused tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use runtime auth availability before blocking startup:** `src/gateway/server-startup.ts:47-50`
  - body: The startup gate calls `hasAuthForProvider` with only `provider` and `agentDir`, but that helper only checks env keys and auth-profile presence. Valid current setups using config-backed provider keys, AWS SDK Bedrock auth, local synthetic auth, or plugin synthetic auth can be rejected before channels start.
  - confidence: 0.9

  - body: The error text always suggests `openclaw models set "openai-codex/gpt-5.4"`. When that model is already the failing default without auth, the remediation points users back to the same broken configuration.
  - confidence: 0.82
