## Summary

Keep open. Current main has meaningful untrusted metadata blocks for sender, conversation, reply, thread, location, and recent chat history, but the current inbound message body still flows as ordinary user text after those blocks, and Dreaming can ingest session transcripts into the memory promotion pipeline. The external comment adds useful design constraints around tag-boundary escape/nonce handling and memory provenance, so this is still a real security hardening item rather than implemented cleanup.

## What This Changes

The issue requests structural untrusted-content wrapping and source provenance for inbound channel message bodies before they enter agent context and memory.

## Best Possible Solution

Add a generic core trust-boundary design for channel-originated user bodies: wrap or serialize the live body, reply/thread/history bodies, and media transcripts through a tag-escape-safe or nonce-backed untrusted content format; preserve raw command parsing via separate command fields; and carry channel/source/trust provenance into session transcripts, Dreaming ingestion, promotion records, and retrieval policy with regression tests and docs.

## Reproduction Assessment

Yes. The current gap is reproducible by source trace: a channel turn with `rawBody` becomes `BodyForAgent`, then `get-reply-run` joins that body after the untrusted metadata prefix as plain prompt text, while Dreaming can later ingest session transcripts.

## Solution Assessment

Unclear as proposed. A simple fixed XML tag wrapper is not enough because tag-boundary attacks are plausible; the maintainable solution should use escaped structured data or nonce-backed delimiters and include memory provenance, not only prompt rendering.
