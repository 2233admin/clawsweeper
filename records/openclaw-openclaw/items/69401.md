## Summary

Keep open. Current main has partial hardening, but the requested generic untrusted wrapping for inbound channel message bodies is still not implemented: core prompt assembly joins untrusted metadata blocks with the raw body, several channel plugins still pass raw text as BodyForAgent, and memory/dreaming export strips metadata then indexes plain user text without channel trust provenance.

## Best Possible Solution

Keep this issue open for a maintainer-led security design. The best path is a generic core/channel contract that wraps the current inbound user body with the existing random-id external-content boundary while preserving clean CommandBody/RawBody for command parsing, plus typed channel source/trust metadata that survives into memory/dreaming so retrieval can filter or downgrade channel-originated memories.

## Risks / Open Questions

- Security-sensitive prompt construction: a naive wrapper could duplicate the current body, break command/directive parsing, or destabilize provider transcript and prompt-cache behavior.
- Memory/dreaming provenance is broader than body wrapping; retrieval trust levels and durable attribution need maintainer product/security judgment.
- Docs currently say external-content sanitization applies uniformly to inbound channel content, while current source only applies wrapper mechanics to external tools/metadata and a Discord guild copy, so docs may overstate shipped behavior.
