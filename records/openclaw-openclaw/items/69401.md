## Summary

Keep open. Current main has solid external-content wrapping infrastructure and channel metadata hardening, but the shared inbound reply path still joins the current channel message body into the user prompt as ordinary text. The discussion is also reinforced by the KinThai comment about delimiter breakouts and memory provenance, and there is an open implementation candidate (#64720) that targets the narrow channel-body wrapping gap but has not landed.

## What This Changes

The issue requests structural untrusted-content wrapping and provenance for inbound channel message bodies before they reach agent context and memory.

## Best Possible Solution

Keep this issue open while maintainers review #64720 or a replacement that applies channel-body wrapping at a shared boundary, keeps command/internal-provider behavior intact, and adds regression coverage for major channel paths plus transcript/memory handling. If the body-wrapping PR lands first, preserve the memory provenance and retrieval-filtering portion as #62939 or a narrower follow-up.

## Reproduction Assessment

Yes. A high-confidence reproduction path exists by source inspection: send or prepare any external channel message, observe `BodyForAgent`/`BodyStripped` flow into `baseBodyFinal`, and see that current main appends it without external-content markers.

## Solution Assessment

The requested direction is sound, but current main is not the best end state because it only wraps metadata and some supplemental context. A centralized wrapper with delimiter sanitization, source labels, internal-provider exclusions, tests, and a separate memory-provenance decision is the safer maintainable path.
