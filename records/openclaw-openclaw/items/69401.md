## Summary

Keep #69401 open. Current main has related defenses for untrusted metadata and external tool/hook/file content, but inbound channel message bodies from Slack/Telegram/Discord/WhatsApp still flow as raw user text after metadata prefixes. The commenter’s notes about nonce boundaries, memory persistence, and trust levels are directly relevant to the remaining design work.

## Best Possible Solution

Keep the issue open and implement this as shared core/channel-security work rather than a single plugin change. The likely path is a nonce-safe `wrapUntrustedChannelMessage` or extension of `wrapExternalContent` for channel message bodies, applied at the shared inbound prompt boundary before agent dispatch; set `InputProvenance` with `kind: "external_user"` and `sourceChannel` from channel finalization; carry provenance through session persistence, memory transcript export/retrieval policy, and docs; add regression tests for Slack, Telegram, Discord, WhatsApp, prompt assembly, marker-breakout attempts, and memory ingestion.

## Risks / Open Questions

- Security-sensitive cross-channel change: marker escaping/nonce boundaries, command detection, transcript replay, and memory indexing need focused regression coverage.
- Docs currently state external-content sanitization applies to inbound channel content, but the inspected channel body path does not support that claim yet.
