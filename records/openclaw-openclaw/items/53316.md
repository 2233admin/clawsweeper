## Summary

Keep this PR open. The root cause from the linked iMessage misrouting report is still present on current main: the shared chat-target parser accepts a hex-looking `chat_id:` value as `chatId: 2` because it calls `Number.parseInt` before any digits-only validation. Current iMessage send and allowlist paths still flow through that shared parser, while the PR’s original target file has since become a re-export, so the fix needs to be ported rather than closed.

## Best Possible Solution

Port the PR’s digits-only validation into `src/channels/plugins/chat-target-prefixes.ts` for both `parseChatTargetPrefixesOrThrow` and `parseChatAllowTargetPrefixes`, keep the error message neutral enough for all non-integer values, and add focused regression tests covering iMessage send target rejection plus allowlist non-match. Because BlueBubbles also uses the shared helper, add or review shared-helper coverage so both message channels get the same protection.
