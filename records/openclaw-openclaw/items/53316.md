## Summary

Keep open. The reported routing bug still exists on current main, but this PR no longer patches the live implementation path because iMessage now re-exports and uses the shared channel target parser; the useful fix needs to be ported before merge.

## What This Changes

The PR branch adds decimal-only guards to iMessage `chat_id` parsing and iMessage regression tests for hex `chat_id` rejection plus allowlist false-match prevention.

## Best Possible Solution

Land a narrow shared-parser fix in `src/channels/plugins/chat-target-prefixes.ts` that requires decimal-only `chat_id` values before parsing, throws a neutral invalid-`chat_id` error for send targets, silently rejects invalid allowlist `chat_id` entries, preserves `chat_guid` and `chat_identifier`, and adds shared/iMessage/BlueBubbles regression coverage plus a user-facing changelog credit.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: `Number.parseInt("2ecba0e20f3a4299b0efb228a5990731", 10)` returns `2`, and both live shared parser paths still call `Number.parseInt` without full-string decimal validation.

## Solution Assessment

No. The PR's direction is correct, but the implementation is no longer the best fix because current main routes iMessage and BlueBubbles through `src/channels/plugins/chat-target-prefixes.ts`, not the old iMessage-local helper body.
