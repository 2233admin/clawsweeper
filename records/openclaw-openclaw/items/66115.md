## Summary

Current main still exposes `/v1/models` as an OpenAI-compatible agent-target list, while provider/model discovery remains available only through Gateway RPC `models.list` and Control UI paths. The request is valid but needs maintainer API design because changing `/v1/models` would conflict with the documented agent-first contract.

## Best Possible Solution

Keep this open for maintainer API design. The likely best path is an additive authenticated HTTP discovery endpoint, or explicit extension fields on a compatibility-safe endpoint, backed by existing `models.list` / `loadGatewayModelCatalog()` behavior while preserving `/v1/models` as the documented agent-target list. The design should specify whether the response means configured, allowed, discovered, or live-probed reachable models.

## Risks / Open Questions

- Replacing `/v1/models` outright would break the documented OpenAI-compatible agent-target contract used by clients that select `openclaw/default` or `openclaw/<agentId>`.
- The issue asks for models that are actually reachable, but current catalog paths distinguish configured, allowed, and discovered models; live reachability would need explicit probe, cache, timeout, auth, and error semantics.
- An unauthenticated HTTP discovery endpoint would expose runtime/provider inventory, so the endpoint shape should reuse the existing operator read boundary or an explicitly designed read-scoped contract.
