## Summary

Keep this open. Current main still intentionally exposes OpenClaw agent targets from `/v1/models`, while the provider model catalog is only available through the Gateway `models.list` RPC and UI-backed configured model view; the requested HTTP discovery surface for provider-qualified model IDs remains unresolved.

## What This Changes

The issue requests an HTTP API surface that returns provider-qualified AI model IDs available through the Gateway instead of only OpenClaw agent pseudo-models from `/v1/models`.

## Best Possible Solution

Add an additive provider-model discovery surface, such as a new read-only HTTP endpoint or explicit view parameter backed by the existing `models.list` configured/all catalog, while leaving `/v1/models` agent-first for compatibility and documenting the auth and reachability semantics.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: with OpenAI-compatible HTTP enabled, `GET /v1/models` reaches `handleOpenAiModelsHttpRequest`, which returns only `loadAgentModelIds()`, and the e2e test asserts every id starts with `openclaw`.

## Solution Assessment

Unclear. Replacing `/v1/models` is probably not the best fix because current main intentionally uses that endpoint for agent routing; an additive endpoint or explicit opt-in view is the safer maintainable direction.

## Risks / Open Questions

- The public API shape needs maintainer choice because current docs and history deliberately make `/v1/models` agent-first for OpenAI-compatible routing.
- The requested “actually reachable” semantics are stronger than the existing configured model catalog, which reflects allowlists, provider config, and auth-backed visibility but does not live-probe every model on each list request.
- A provider-catalog HTTP endpoint should keep the same operator-read auth boundary and document whether it exposes configured provider ids, aliases, and auth-visible catalog entries.
