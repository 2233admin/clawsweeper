## Summary

Keep open. Current main still intentionally exposes agent-target IDs from `GET /v1/models`, while the provider-qualified model catalog is available only through the Gateway RPC `models.list`; changing the OpenAI-compatible HTTP surface needs maintainer API/product judgment because chat and embeddings currently treat the OpenAI `model` field as an agent target and use `x-openclaw-model` for provider overrides.

## What This Changes

The issue requests that OpenAI-compatible model discovery expose actual provider-qualified AI model IDs, or add a new endpoint for available backend models, so third-party clients do not need filesystem access to `openclaw.json`.

## Best Possible Solution

Define an explicit HTTP discovery contract for backend provider models, likely by reusing `loadGatewayModelCatalog` and `resolveVisibleModelCatalog` behind an additive endpoint or opt-in view, then align docs and OpenAI-compatible request semantics so clients can safely use the discovered IDs without raw config access.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence reproduction path: enabling the OpenAI-compatible HTTP surface and calling `GET /v1/models` returns only `openclaw` agent-target IDs, and a provider-qualified `model` value on chat completions is rejected unless sent through `x-openclaw-model`.

## Solution Assessment

Unclear. Returning actual model IDs from `/v1/models` alone is not the best fix unless maintainers also want to change the documented agent-first contract; the safer direction is an additive provider-model discovery endpoint or opt-in mode plus coordinated chat/embeddings behavior.
