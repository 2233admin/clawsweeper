## Summary

Keep this issue open. Current main still has no heartbeat-scoped `tools: false`, `disableTools`, or `toolsAllow` config path: heartbeat config is strict, docs list only the existing heartbeat knobs, the runner only passes model/warning/timeout/bootstrap options into `getReplyFromConfig`, and `GetReplyOptions` has no tool-policy field to carry the request down to the embedded runner.

## Best Possible Solution

Keep this issue open and add a heartbeat-scoped tool policy to the existing heartbeat config surface. A conservative path is to support `tools: false` first, or a documented allowlist if maintainers want safe read-only heartbeat tools, then update `HeartbeatSchema`, `AgentDefaultsConfig`, generated config labels/help/docs, `GetReplyOptions`, and `runHeartbeatOnce` so heartbeat runs pass `disableTools` or `toolsAllow` into the embedded runner. Add focused tests proving heartbeat tool definitions are omitted while normal user turns for the same agent keep their configured tools.
