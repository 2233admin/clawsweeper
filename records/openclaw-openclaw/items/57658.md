## Summary

Keep #57658 open. Current main partially solves the practical limited-access symptom with per-device approved scope baselines and scoped token rotation, and tests show reconnects cannot silently widen a read-scoped device back to admin. But the requested custom named roles/role profiles are still missing: `GatewayConfig` has no role-definition field, the CLI only exposes device token rotation with `--role/--scope` and no `openclaw roles create`, and non-operator roles are deliberately restricted to scopes with their own role prefix, so a custom `readonly` or `operatorRW` role cannot map to `operator.read`/`operator.write`.

## Best Possible Solution

Keep the issue open but narrow the implementation direction: build custom role profiles as aliases over the existing operator scope model, with schema/docs for `gateway.roles` or an equivalent access-control section, CLI management such as `openclaw roles create/list/delete`, pairing/approval UI that can assign a named profile, and tests proving named roles resolve to approved scope baselines without allowing scope escalation. The shipped per-device `approvedScopes` and token-rotation behavior should be reused rather than replaced.
