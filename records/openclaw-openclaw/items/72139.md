## Summary

Keep this issue open. Current main still matches the reported Control UI auth-loop path and `chat.history` still performs full synchronous transcript reads before limits are applied. The v2026.4.26 Slack Socket Mode timeout tuning is a partial mitigation, but the follow-up confirmation says it did not resolve the disconnect loop.

## Best Possible Solution

Fix this in core Control UI and Gateway history handling. Make `AUTH_TOKEN_MISMATCH` stop automatic reconnects when no trusted cached-device-token retry can be attempted, and after the single retry is exhausted. Separately, make `chat.history` non-starving by avoiding synchronous full-transcript reads/parses before applying recent-message and byte limits, then add regression coverage for both the auth loop and large-history responsiveness. Treat the Slack Socket Mode timeout tuning as mitigation, not the root fix.
