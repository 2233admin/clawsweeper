## Summary

Keep this issue open. Current main still has a source-backed Control UI `AUTH_TOKEN_MISMATCH` reconnect path when no cached device-token retry is available, and `chat.history` still reads/parses the transcript synchronously before applying the requested message and byte limits; the related open PR only addresses a frontend timeout symptom.

## What This Changes

The issue requests stopping stale Control UI auth reconnect loops and making Gateway `chat.history` avoid event-loop-blocking full transcript work that can starve Slack Socket Mode pongs.

## Best Possible Solution

Fix both root paths in core: make browser and shared Gateway clients treat `AUTH_TOKEN_MISMATCH` as terminal when no trusted cached-device-token retry can be attempted or after that retry is exhausted, and refactor history loading so `chat.history` can return a recent bounded window without synchronous full-transcript reads/parses before limits while preserving branch semantics.

## Reproduction Assessment

Yes. The current Control UI unit test reproduces reconnect after `AUTH_TOKEN_MISMATCH` with no cached retry, and source inspection shows `chat.history` still performs synchronous full-file transcript work before applying limits; only the live Slack timing correlation remains to be rechecked after a patch.

## Solution Assessment

Yes, the issue’s requested direction is the narrow maintainable fix boundary. PR #44897’s frontend timeout and Slack pong-timeout tuning are useful mitigations, but they do not remove the auth retry loop or event-loop-starving history path.
