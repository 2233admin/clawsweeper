## Summary

Keep open. Current main still leaves the reported root paths in place: the browser Control UI can reconnect after `AUTH_TOKEN_MISMATCH` when no cached device-token retry is available, and `chat.history` still performs synchronous full-transcript work before applying the recent-message and byte caps. The related open WebUI timeout PR is a symptom mitigation, not a fix for the auth loop or event-loop starvation path.

## What This Changes

The issue reports that stale Control UI gateway auth retries and blocking `chat.history` work can starve Slack Socket Mode pong handling and cause repeated disconnects.

## Best Possible Solution

Fix both core paths: make `AUTH_TOKEN_MISMATCH` terminal when no trusted cached-device-token retry can be attempted or after the one allowed retry is exhausted, and add a bounded recent-window transcript loading path for `chat.history` so recent-message and byte limits avoid synchronous full-transcript reads before projection.

## Reproduction Assessment

Yes. Current source and tests provide a high-confidence reproduction for the no-cached-token Control UI reconnect path, and source inspection shows `chat.history` still performs synchronous full-file transcript loading before limits; only the live Slack timing needs rechecking after a patch.

## Solution Assessment

Yes. The requested direction is the narrow maintainable fix; PR #44897's frontend timeout and Slack ping/pong tuning are useful mitigations, but they do not remove the auth reconnect loop or the transcript-loading event-loop risk.
