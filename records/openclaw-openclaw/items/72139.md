## Summary

Keep this open. Current main still preserves the Control UI `AUTH_TOKEN_MISMATCH` reconnect path when no cached device-token retry is available, and `chat.history` still synchronously reads/parses the full transcript before applying message and byte limits; the Slack Socket Mode timeout tuning is only a mitigation and the latest discussion confirms it did not restore inbound events.

## What This Changes

The issue requests that stale Control UI auth stop retrying automatically and that `chat.history` avoid event-loop starvation that can trigger Slack Socket Mode disconnects.

## Best Possible Solution

Fix both root paths in core: make browser AUTH_TOKEN_MISMATCH terminal when no trusted cached-device-token retry can be attempted or after the one retry is exhausted, and refactor chat.history to avoid synchronous full-transcript reads/parses before recent-message and byte limits while preserving transcript and branch semantics.

## Reproduction Assessment

Yes for the core code paths: the current Control UI test reproduces reconnect after AUTH_TOKEN_MISMATCH with no cached retry, and static source inspection shows chat.history still performs synchronous full-file transcript work before limits. The live Slack disconnect timing remains to be validated with Socket Mode after a patch.

## Solution Assessment

Yes, the issue’s requested direction is the right fix boundary. Slack Socket Mode timeout tuning and PR #44897’s frontend timeout are useful partial mitigations, but the maintainable solution is to stop the UI auth loop and make Gateway history reads non-starving.
