## Summary

Keep this open. Current main still preserves the Control UI token-mismatch reconnect case when no cached device-token retry is available, and the Gateway still reads/parses the full transcript synchronously before applying chat.history message and byte limits; the related open WebUI PR only addresses a frontend timeout symptom, not these root paths.

## What This Changes

The issue requests that stale Control UI auth stop retrying indefinitely and that chat.history avoid event-loop starvation that can disconnect Slack Socket Mode.

## Best Possible Solution

Fix both root paths in core: make browser AUTH_TOKEN_MISMATCH terminal when no trusted cached-device-token retry can be attempted or after the single retry is exhausted, and refactor chat.history transcript loading so recent-message and byte limits are enforced without synchronous full-transcript work that can starve other WebSocket transports.

## Reproduction Assessment

Yes for the core code paths: current source and tests show the no-retry AUTH_TOKEN_MISMATCH reconnect, and static inspection shows chat.history still performs synchronous full-file transcript work before limiting. The live Slack disconnect timing remains an integration validation step after the repair.

## Solution Assessment

Yes, the issue’s requested direction is the maintainable fix boundary. Slack Socket Mode ping/pong tuning is a useful mitigation, but it does not remove the UI auth loop or the event-loop starvation source.
