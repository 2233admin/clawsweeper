## Summary

Keep this issue open. Current main still has the two root behaviors described in the report: Control UI can continue reconnecting after an `AUTH_TOKEN_MISMATCH` when no cached device-token retry was possible, and `chat.history` still performs synchronous full-transcript reads/parses before applying message and byte limits. The related open PR addresses a frontend loading timeout symptom, not the auth retry loop or history read path that can starve other websocket transports.

## What This Changes

The issue requests that stale Control UI auth stop retrying automatically and that `chat.history` avoid event-loop-starving transcript work so Slack Socket Mode remains connected.

## Best Possible Solution

Fix both root paths in core: make browser `AUTH_TOKEN_MISMATCH` terminal when no trusted cached-device-token retry can be attempted or after the one retry is exhausted, and refactor `chat.history` so recent-message and byte limits are applied without synchronously materializing large full transcripts while preserving branch/compaction semantics.

## Reproduction Assessment

Yes for the core code paths: current tests explicitly preserve reconnect after `AUTH_TOKEN_MISMATCH` when no cached retry exists, and static source inspection shows `chat.history` synchronously full-reads/parses transcripts before limits. The Slack disconnect timing should be live-validated after the code fix.

## Solution Assessment

Yes, the requested direction is the right fix boundary. Slack Socket Mode timeout tuning and PR #44897's frontend timeout are partial mitigations; the maintainable fix is to stop the bad auth loop and make history retrieval non-starving.
