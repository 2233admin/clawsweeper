## Summary

Keep this open. Current main still has both reported root paths: the Control UI can keep reconnecting after an `AUTH_TOKEN_MISMATCH` when no cached device-token retry is available, and `chat.history` still performs synchronous full-transcript work before applying response limits. The related open PR covers a frontend loading-timeout symptom, not the auth-loop and transcript-read causes described here.

## What This Changes

The issue requests stopping stale Control UI auth retry loops and making `chat.history` non-starving so Control UI failures do not degrade Slack Socket Mode connectivity.

## Best Possible Solution

Fix both root paths in core: make browser `AUTH_TOKEN_MISMATCH` terminal when no trusted cached-device-token retry can be attempted or after the single retry is exhausted, and refactor `chat.history`/transcript reading so recent-message and byte limits are applied without synchronously parsing the full transcript on the Gateway event loop. Preserve transcript branch semantics and use Slack Socket Mode only as live validation, not as the sole fix boundary.

## Reproduction Assessment

Yes for the core code paths: the current Control UI test explicitly reproduces the no-device-token `AUTH_TOKEN_MISMATCH` reconnect case, and static source inspection shows `chat.history` still does synchronous full-file transcript work before limits. The live Slack disconnect behavior remains to be verified with Socket Mode after a patch.

## Solution Assessment

Yes, the issue’s requested direction is the right fix boundary. Slack timeout tuning and PR #44897’s UI timeout are useful mitigations, but the maintainable solution is to stop the auth loop and make Gateway history reads non-starving.
