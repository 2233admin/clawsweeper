## Summary

Current main and the latest verified release already carry the Feishu/non-default-agent session fix: Feishu dispatch keeps the resolved route agent through session-store selection, and the reply runner resolves transcript files with agent-aware path options. The issue targets an older reported version and repeats the same root cause described in the linked closed reports.

## What This Changes

The issue requests that Feishu messages routed to non-main agents use that agent's session key, session store, workspace, identity, and transcript directory instead of falling back to `main`.

## Best Possible Solution

Keep the shipped shared session-path implementation and Feishu route-agent dispatch flow. If this still reproduces on `v2026.4.29` or newer, it should come back as a fresh report with current version, bindings output, and route/session log lines.

## Reproduction Assessment

Yes. The old reproduction path is a Feishu account binding to a non-main agent followed by an inbound DM; current source verification shows that route agent and store context now flow through the session-file resolver, so the reported fallback should not reproduce on current main.

## Solution Assessment

Yes. The current solution is the narrow maintainable fix: channel dispatch passes the resolved agent, and the shared session-path helper carries `agentId` plus store-derived session root instead of adding a Feishu-only workaround.
