## Summary

Keep this open. Current main still creates a fresh probe `GatewayClient` for each `probeGateway()` call and has no per-URL `DEVICE_IDENTITY_REQUIRED` short-circuit, so the reporter's corrected RCA remains valid. The exact source-level fix was proposed in #63446, but that PR is closed unmerged; #68280 is adjacent missing-auth fail-fast work and does not currently implement the repeated device-required suppression on main.

## What This Changes

Add source-level throttling for repeated CLI WebSocket probes that receive device-required gateway rejections when the CLI lacks usable device pairing.

## Best Possible Solution

Land a narrow fix at the `probeGateway()` boundary that records recent `DEVICE_IDENTITY_REQUIRED` closes per gateway URL, short-circuits after a small threshold/TTL without constructing another `GatewayClient`, clears on successful connection, and covers the behavior in focused tests. Keep the unauthenticated HTTP health endpoint idea out of this fix unless maintainers separately approve the product/security tradeoff.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to mock `GatewayClient` so repeated `probeGateway()` calls close with code 1008 and reason `device identity required`; current main will construct/start a new mock client each time, and there is no test proving the fourth call short-circuits.

## Solution Assessment

Yes, a #63446-style cache in `src/gateway/probe.ts` is the narrowest maintainable fix because all status/restart callers already cross that boundary. A new unauthenticated `/health` endpoint or only caller-local auth failure handling would be broader and would not directly prove suppression of repeated WebSocket handshake logs.
