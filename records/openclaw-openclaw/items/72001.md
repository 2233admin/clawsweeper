## Summary

Keep open. The PR targets a real current-main Gmail hooks setup bug, but this revision is not ready to merge because it only writes the prefix allowlist: template-rendered Gmail session keys still fail dispatch without `hooks.allowRequestSessionKey=true`, and the unconditional prefix write can make already-valid static-shadow/default-session configurations invalid.

## What This Changes

The PR adds a helper for merging `hooks.allowedSessionKeyPrefixes`, writes `hook:` from `runGmailSetup`, adds Gmail/gateway tests, and appends a changelog fix entry.

## Best Possible Solution

Update the wizard so it writes a complete, internally consistent Gmail hook session policy for the effective mapping it installs: templated Gmail routing should include both `allowRequestSessionKey=true` and a bounded `hook:`/Gmail prefix policy, while user configs that shadow the preset with static mappings or use non-`hook:` default session keys should remain loadable. Add tests for startup validation, actual mapped `/hooks/gmail` dispatch, and the static-shadow/default-session compatibility case.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce the original startup failure: setup writes the Gmail preset, the preset has a templated session key, and `resolveHooksConfig` requires allowed prefixes for effective templated mappings. The PR regression is also reproducible by code inspection against the existing dispatch gate and static-shadow tests; I did not run tests because this review was required to stay read-only.

## Solution Assessment

No. Prefix seeding is necessary for the reported startup validator failure, but it is not sufficient for Gmail delivery and it is too broad when existing mappings make the templated Gmail preset ineffective. A safer fix should update the full session policy conditionally and test both delivery and compatibility paths.
