## Summary

Keep open. Current main confirms the reported Gmail setup bug, but this PR only seeds `hooks.allowedSessionKeyPrefixes` and still leaves templated Gmail deliveries blocked by the dispatch-time `hooks.allowRequestSessionKey` policy; it can also invalidate currently loadable static-shadow configurations with non-`hook:` default session keys.

## What This Changes

The PR adds a Gmail helper for merging `hooks.allowedSessionKeyPrefixes`, writes `hook:` from `runGmailSetup`, adds Gmail and gateway tests, and appends a changelog fix entry.

## Best Possible Solution

Revise the wizard so it writes a complete, internally consistent Gmail hook session policy only when the templated Gmail preset remains effective: include `allowRequestSessionKey=true` with a bounded Gmail hook prefix for preset routing, preserve static shadow/default-session configurations that do not need templated routing, and cover both startup validation and mapped delivery in tests.

## Reproduction Assessment

Yes. The current-main source path is enough to reproduce the original startup failure, and the PR regression is reproducible by following the existing dispatch gate for `mapping-templated` session keys; no mutable or live test run was needed to prove those paths.

## Solution Assessment

No. Prefix seeding is necessary for the startup validator failure, but it is not sufficient for Gmail delivery and is too broad for existing static-shadow configurations; the safer fix is a conditional full session-policy update with compatibility tests.
