## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding even though trusted-proxy is a supported core gateway auth mode, and an open same-author draft PR already targets the exact onboarding gap, so this should wait for maintainer review rather than be closed or auto-repaired.

## What This Changes

The issue requests first-class non-interactive `openclaw onboard` flags for configuring gateway `trusted-proxy` auth and related reverse-proxy settings.

## Best Possible Solution

Review, land, or replace PR #49107 with a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, writes trusted-proxy config through typed flags, preserves runtime fail-closed validation, documents the scripted deployment path, and adds focused tests for valid writes plus missing or unsafe security-critical inputs.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits because only `token` and `password` are accepted.

## Solution Assessment

Yes. The maintainable direction is a core onboarding change because `trusted-proxy` is already a core gateway auth mode and interactive onboarding can configure it; raw config editing or a plugin would bypass the typed onboarding contract.
