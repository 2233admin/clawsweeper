## Summary

Keep open. Current main still has the reported gap: core runtime and interactive gateway setup support `trusted-proxy`, but non-interactive onboarding still types, documents, exposes, and validates only `token` and `password`. The linked same-author draft PR #49107 already targets the exact surface, so this should remain open for maintainer/security review rather than be closed or queued as a separate automated repair.

## What This Changes

Request first-class non-interactive `openclaw onboard` support for configuring core gateway `trusted-proxy` authentication in reverse-proxy deployments.

## Best Possible Solution

Review, land, or replace PR #49107 with a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, writes explicit typed config for `gateway.trustedProxies` and `gateway.auth.trustedProxy.*`, covers Control UI allowed origins for non-loopback deployments, preserves runtime validation, and adds focused docs/tests. Once that path is decided, dedupe #73638 intentionally if it is not the canonical thread.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy`; current main rejects that value in `applyNonInteractiveGatewayConfig` before writing gateway config.

## Solution Assessment

Yes, the requested direction is the right product boundary because `trusted-proxy` is already a core gateway auth mode and the interactive gateway setup can configure it. The safe implementation is first-class typed onboarding flags with validation, not raw post-onboard config editing.
