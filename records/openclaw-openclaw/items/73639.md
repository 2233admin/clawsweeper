## Summary

Keep open. Current main and the latest release still lack the requested non-interactive trusted-proxy onboarding path, while runtime config and interactive onboarding already support trusted-proxy auth. The same-author draft PR #49107 is already the active implementation candidate, so this issue should remain open for maintainer/security review rather than be closed or queued for a separate repair lane.

## What This Changes

The issue requests non-interactive `openclaw onboard` support for configuring gateway `trusted-proxy` authentication behind reverse proxies.

## Best Possible Solution

Keep this issue open until #49107 or a narrow replacement PR lands with typed non-interactive flags, focused tests, docs for scripted deployments, and the existing trusted-proxy fail-closed runtime rules preserved.

## Reproduction Assessment

Yes. A source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits because only `token` and `password` are accepted.

## Solution Assessment

Yes. A core onboarding change is the right direction because trusted-proxy is already a core gateway auth mode and interactive onboarding configures it; raw config editing or a plugin would bypass the typed onboarding contract.
