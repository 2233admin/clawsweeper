## Summary

Current main still has the reported non-interactive onboarding gap: `trusted-proxy` is a supported core gateway auth mode, but `openclaw onboard --non-interactive` only accepts token/password. This should stay open because an open same-author draft PR already targets the exact change and the duplicate-looking same-author issue should be deduped only after maintainers choose the canonical path.

## What This Changes

Add non-interactive `openclaw onboard` support for selecting and safely configuring `trusted-proxy` gateway authentication for scripted reverse-proxy deployments.

## Best Possible Solution

Review, land, or replace PR #49107 with a narrow core onboarding change that accepts `--gateway-auth trusted-proxy`, writes explicit `gateway.trustedProxies` and `gateway.auth.trustedProxy.*` fields through typed flags, preserves existing runtime validation, documents the scripted deployment path, and adds targeted tests for valid config writes plus missing or unsafe security-critical inputs. After that lands or is rejected, intentionally dedupe #73638 if it is not the canonical thread.

## Reproduction Assessment

Yes. The high-confidence source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy ...`; current main exits before writing config because non-interactive gateway auth only accepts `token` or `password`.

## Solution Assessment

Yes. The requested direction belongs in core onboarding rather than a plugin because `trusted-proxy` is already a core gateway auth mode and interactive setup can configure it; the maintainable fix is explicit first-class flags, not raw config editing.
