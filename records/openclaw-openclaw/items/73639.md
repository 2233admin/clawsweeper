## Summary

Keep this issue open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding, while the linked same-author draft PR #49107 is an active implementation candidate for the exact missing surface. Per cleanup policy, the issue should stay open until that PR, or a refreshed equivalent, is reviewed and merged or closed.

## Best Possible Solution

Keep this issue open and review/land or replace PR #49107 with a focused implementation that adds non-interactive onboard flags for `gateway.auth.mode="trusted-proxy"`, `gateway.trustedProxies`, `gateway.auth.trustedProxy.userHeader`, `requiredHeaders`, `allowUsers`, likely `allowLoopback`, and any needed Control UI allowed-origin automation, with targeted tests and docs aligned to the trusted-proxy auth guide.

## Risks / Open Questions

- The implementation touches a security-sensitive gateway auth boundary; it should explicitly clear mutually exclusive token/password fields when switching to trusted-proxy and validate proxy IPs, header names, allowUsers, allowLoopback, and Control UI allowed origins against the current trusted-proxy security model.
- The linked PR is an implementation candidate, not a shipped fix; closing this issue before merge would lose the tracking link for the remaining work.
