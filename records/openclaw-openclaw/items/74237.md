## Summary

Keep open. Current main has partial defenses for blank Feishu messages and generic transcript repair, but it still does not provide the Feishu-scoped stale/corrupt local state repair requested here; the linked fix PR #74397 is open, unmerged, and explicitly targets this issue, so the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

The issue requests Feishu-scoped validation, quarantine, logging, or doctor repair so stale or corrupt Feishu runtime/session state cannot drive sustained gateway CPU usage after startup.

## Best Possible Solution

Review and land #74397, or an equivalent Feishu-owned doctor repair, so `openclaw doctor` can detect and safely archive/rebuild corrupt Feishu local JSON plus Feishu-scoped direct session entries while preserving credentials, non-Feishu sessions, and ACP binding sessions. Keep any shared core changes limited to generic plugin doctor seams if the Feishu plugin cannot express the repair through the existing contract.

## Reproduction Assessment

Unclear for the exact high-CPU loop because the private state files were not provided. The missing repair surface is reproducible from code and can be covered with synthetic corrupt Feishu local JSON and Feishu direct session entries, as the linked PR proposes.

## Solution Assessment

Yes, the linked repair direction is the best fit: the behavior is Feishu-owned state, and the existing channel doctor contract can host a targeted repair without a startup-time core migration. The safer alternative would only be a narrow additive core seam if Feishu cannot enumerate the relevant session stores through current APIs.
