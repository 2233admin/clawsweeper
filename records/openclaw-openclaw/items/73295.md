## Summary

Keep this PR open. Current main still lacks the Discord queued-followup stuck-session circuit breaker, so the change is relevant, but the PR is not merge-ready because the new timer can survive a normal drain and block a later session, and recovery can restart draining before the aborted active run has actually cleared.

## What This Changes

The PR adds a Discord-only stuck-session circuit breaker for queued follow-ups, wires active-run state and abort hooks through the reply runner, exports the new queue helper, and adds focused Vitest coverage.

## Best Possible Solution

Revise this PR in place: keep the feature scoped to Discord queued follow-ups, but tie breaker timer lifetime to queue/session lifetime, replace or clear stale timers on normal drains, and use the existing abort-and-drain or force-clear pattern before scheduling recovery work. Add regression tests for successive sessions on the same queue key and for an abort that does not settle before draining.

## Reproduction Assessment

Yes for the blocking PR findings. The static reproduction path is clear from the PR head: a normal queue drain leaves the new breaker timer map untouched, and the breaker schedules recovery immediately after an abort signal while the same session key can still be registered as active.

## Solution Assessment

No, not as written. The requested circuit breaker is a reasonable narrow direction, but the implementation needs queue-lifetime timer cleanup and drain-after-abort settlement to be the maintainable fix.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Refresh stale breaker timers for new queued work:** `src/auto-reply/reply/queue/circuit-breaker.ts:72-73`
  - body: A breaker timer can remain in `DISCORD_STUCK_SESSION_BREAKER_TIMERS` after a queue drains normally, because the drain cleanup deletes the queue and callback but never clears this new map. A later Discord session on the same `queueKey` returns here and inherits the stale timer instead of arming a fresh protection window, so the new stuck session can be left unprotected. Clear the breaker when the queue drains or replace the existing timer when new queued work arrives.
  - confidence: 0.86
- **[P2] Wait for abort to settle before draining:** `src/auto-reply/reply/queue/circuit-breaker.ts:112`
  - body: The breaker starts the recovery drain immediately after `abortActiveRun` returns true, but that hook maps to an abort signal and the reply-run registry can keep the session key active until cleanup finishes. If the stuck run does not exit promptly, the recovery follow-up can hit `ReplyRunAlreadyActiveError` and leave the queue blocked. Wait for abort settlement or reuse the existing abort-and-drain/force-clear recovery path before scheduling recovery work.
  - confidence: 0.9
