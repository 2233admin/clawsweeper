## Summary

Keep this PR open, but this revision is not merge-ready. Current main still lacks the Discord-specific queued follow-up circuit breaker, and the earlier related PRs are closed unmerged, but the proposed patch starts recovery drain immediately after a boolean abort without waiting for the active reply operation to clear, which can leave a truly stuck session blocking the recovery queue.

## What This Changes

The PR adds a Discord-only follow-up queue circuit breaker under `src/auto-reply/reply` that aborts a non-streaming active run after five minutes, prepends a recovery prompt, wires run-state hooks from `runPreparedReply`, and adds focused Vitest coverage.

## Best Possible Solution

Implement recovery through a path that waits for the active run to settle, force-clears only under an explicit policy, and drains queued follow-ups once the session key is idle. If the behavior remains Discord-specific, the threshold and ownership boundary should be explicit before landing, with focused queue tests plus a safe Discord operational smoke.

## Reproduction Assessment

Yes for the review finding: the static path is high-confidence because current `abortEmbeddedPiRun` only signals cancellation, `createReplyOperation` rejects while the same session key is active, and the PR schedules drain immediately after `abortActiveRun` returns true. A live reproduction would be a non-streaming Discord run that ignores abort while a follow-up is queued.

## Solution Assessment

No, not as written. The desired behavior is reasonable, but the safer implementation should reuse or mirror the existing abort-and-drain/force-clear pattern before scheduling recovery work instead of treating a boolean abort result as session-idle proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Wait for abort to settle before draining:** `src/auto-reply/reply/queue/circuit-breaker.ts:112`
  - body: The breaker starts the follow-up drain as soon as `abortActiveRun` returns true, but the current abort helper only signals cancellation and the reply-run registry keeps the session key active until the operation actually clears. For a truly stuck run that does not exit promptly, the recovery follow-up can hit `ReplyRunAlreadyActiveError` and keep the queue blocked. Wait for the aborted run to drain, or use the existing force-clear recovery path, before scheduling the queued recovery work.
  - confidence: 0.87
