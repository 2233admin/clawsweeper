## Summary

Keep this PR open. Current main has absorbed several Microsoft Teams media fixes, including hosted-content `/$value` fetching, Bot Framework DM media routing, attachment-tag gating, and exact `smba.trafficmanager.net` auth forwarding, but it still does not implement the PR’s channel `/chats/{threadId}/messages` fallback. The PR also contains useful remaining hardening around bounded `/$value` reads, while its tenant-token path and older HTML fallback need maintainer cleanup before merge.

## Best Possible Solution

Keep the PR open for Microsoft Teams maintainer cleanup. Rebase onto current main, preserve the shipped hosted-content, Bot Framework DM, diagnostics, attachment-tag gating, and exact auth-allowlist behavior, then either implement and test the channel `/chats/{threadId}/messages` fallback or explicitly reject it with product/API evidence. Move hosted-content `/$value` reads to `readResponseWithLimit`, and replace the tenant-token overload with a typed provider capability or drop that cross-tenant path from this PR.

## Risks / Open Questions

- Do not merge the PR as-is: the tenant-scoped token path invokes an undeclared overload at an auth-sensitive boundary.
- A direct merge/rebase could regress the current attachment-tag fallback gate back to a broad `text/html` trigger unless handled carefully.
- Closing now would lose the unimplemented channel `/chats/{threadId}/messages` fallback and the no-`Content-Length` bounded-read hardening.
- The `trafficmanager.net` allowlist narrowing may be the right security hardening, but it should be validated against real Bot Framework attachment URLs before shipping.
