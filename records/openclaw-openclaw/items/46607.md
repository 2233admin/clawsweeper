## Summary

Keep open. Current main still lacks the requested headless Google/Gemini audio @file inlining, so the PR remains useful, but it should not merge as-is: the branch is stale against the current provider stream-wrapper architecture, the new prompt-controlled file read is rooted at process cwd instead of the active workspace, and the M4A sniffing path rejects common valid MP4 brands.

## What This Changes

The PR adds Google/Gemini headless payload rewriting that converts local audio @file references in user text parts into Gemini inlineData parts with role, realpath, size, and MIME checks plus regression tests.

## Best Possible Solution

Rework the conversion in the current Google/Gemini provider stream-wrapper path, root all local refs at the active workspaceDir/effectiveWorkspace, keep user-turn-only rewriting, retain realpath/stat/size checks, use robust shared MIME detection for audio including common M4A containers, and land focused regression coverage plus docs/changelog if the behavior ships.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P1] Use the active workspace for audio refs:** `src/agents/pi-embedded-runner/extra-params.ts:251-256`
  - body: resolveGoogleInlineAudioPath bounds refs to process.cwd(). Current embedded runs pass an effectiveWorkspace into applyExtraParamsToAgent and provider hooks, so daemon/headless runs whose cwd differs from the session workspace can read from the wrong root or miss valid refs. Thread the active workspace through this resolver before inlining.
  - confidence: 0.88
- **[P2] Accept common M4A MP4 brands:** `src/agents/pi-embedded-runner/extra-params.ts:242-244`
  - body: The .m4a sniff only accepts a small audio-specific major-brand set. Common M4A files can use ISO MP4 brands such as isom or mp42, so valid audio remains plain text instead of becoming inlineData. Use shared MIME detection or accept compatible M4A brands before rejecting.
  - confidence: 0.8
