## Summary

Keep open. Current main still lacks the requested headless Gemini audio inlining behavior, but this PR is dirty against current main and has merge-blocking correctness/security issues around the file-read boundary, active wrapper placement, and M4A detection.

## What This Changes

The PR rewrites Google/Gemini headless payload text parts so resolvable local audio `@file` references become Gemini `inlineData` parts, with regression tests for inlined, missing, absolute, and oversized refs.

## Best Possible Solution

Implement the rewrite in the active Google provider wrapper or transport seam, thread the session workspace/sandbox media boundary into the resolver, use shared or equivalent MIME detection for supported audio, and cover valid audio, cwd-versus-workspace, sandbox escape, oversize, unresolved refs, and M4A brand cases.

## Reproduction Assessment

Yes. A `google-gemini-cli` user prompt containing a local `@sample.ogg` currently reaches the Google transport as a plain text part on main, while the PR tests demonstrate the intended `inlineData` conversion.

## Solution Assessment

No. The requested behavior is valid, but this PR is not yet the best fix because it uses the process cwd as the file boundary, patches a stale wrapper relative to current main, and duplicates narrow audio sniffing instead of reusing the shared media path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Use the active workspace for audio refs:** `src/agents/pi-embedded-runner/extra-params.ts:251-256`
  - body: `resolveGoogleInlineAudioPath` uses `process.cwd()` as its root, but embedded/headless runs already pass the effective session workspace into `applyExtraParamsToAgent`. When the gateway cwd differs from the workspace or sandbox copy, valid refs are missed or files are read from the wrong boundary; thread the active workspace/sandbox media root into this resolver and realpath-check against it.
  - confidence: 0.88
- **[P2] Move the hook to the active Google wrapper:** `src/agents/pi-embedded-runner/extra-params.ts:368`
  - body: The patch wires the rewrite into a local `createGoogleThinkingPayloadWrapper`, but current main routes Google provider wrapping through `src/plugin-sdk/provider-stream-shared.ts` and `extensions/google/provider-hooks.ts`. Rebase and place the payload hook in the active wrapper or provider transport seam so it actually runs.
  - confidence: 0.77
- **[P2] Accept common M4A MP4 brands:** `src/agents/pi-embedded-runner/extra-params.ts:204`
  - body: The `.m4a` sniff only accepts a small audio-specific major-brand set, so valid M4A files with common MP4 brands such as `isom` or `mp42` remain plain text refs. Reuse shared MIME detection or accept compatible `.m4a` brands before rejecting.
  - confidence: 0.8
