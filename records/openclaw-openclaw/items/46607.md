## Summary

Keep this PR open. Current main still does not implement inline Google/Gemini audio @file reference rewriting for headless payloads, while this PR remains a concrete implementation candidate for the closed #46601 request. The security-sensitive local file read path needs maintainer review before merge, not cleanup closure.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for Google/Gemini headless audio @file inlining. The best path is to adapt it to current main, keep the local-file read path constrained to user turns and workspace-safe real paths, reuse shared MIME/media helpers where possible, preserve the size guard, and land focused regression coverage for successful, missing, absolute/traversing, oversized, and valid .m4a refs.

## Risks / Open Questions

- Security review pass: the PR introduces a prompt-text-to-local-filesystem read path. Before merge, maintainers should verify the workspace/realpath/symlink boundary, user-role filtering, file-size cap, and MIME sniffing against OpenClaw's filesystem trust model.
- A review comment notes the PR's .m4a sniffing may reject valid MP4-family brands such as isom/mp42, so legitimate audio could remain plain text unless the sniffing is broadened or delegated to shared MIME detection.
- The PR may need rebase/adaptation to current provider wrapper structure before landing, because current main routes Google stream wrapping through shared provider-stream helpers and plugin runtime wrapping.
