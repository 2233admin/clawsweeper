## Summary

Keep open. Current main still does not scan Google/Gemini headless payload text for local audio @file references or convert them to Gemini inlineData parts, and this PR remains a concrete implementation candidate for the closed #46601 request.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate for Google/Gemini headless audio @file inlining. The best path is to adapt it to the current Google provider stream-wrapper structure, keep the role/workspace/realpath/size/MIME safeguards, address the .m4a sniffing concern, and land focused regression coverage for the headless payload rewrite.

- Current main moved Google provider wrapping into the plugin/provider-stream structure, so the PR likely needs rebase/adaptation before merge.
