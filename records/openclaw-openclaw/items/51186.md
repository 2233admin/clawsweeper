## Summary

Keep this issue open. Current main has source/test evidence for the hook session rebinding and plugin webhook registry fixes, but the BlueBubbles phone-format routing mismatch remains: inbound DMs still route from the exact sender handle, BlueBubbles handle normalization still does not equate `+1...` and `1...`, and generic binding matching still falls through to wildcard/catch-all on non-exact peer ids.

## Best Possible Solution

Keep this issue open. Implement BlueBubbles-owned phone-handle canonicalization for both initial `resolveAgentRoute` peer matching and configured/runtime conversation binding matching so `+1XXXXXXXXXX` and `1XXXXXXXXXX` select the same peer before wildcard/catch-all routing, with regression coverage for exact binding before wildcard fallback. Preserve the current `/hooks/agent` session rebinding and plugin webhook registry behavior.
