## Summary

Current main still has the reported memory-wiki limitation: compile, local query/get, and status only enumerate direct Markdown children in each top-level wiki page group, so nested pages under sources/ are not compiled or surfaced. This is a valid, narrow plugin bug and a good focused fix-PR candidate.

## Best Possible Solution

Keep this issue open and implement recursive Markdown page discovery inside the memory-wiki plugin. The best fix is a shared deterministic page-file collector used by compile, query/get, and status counting, with focused regression tests for nested sources and generated indexes/digests preserving relative paths. Update user-facing memory-wiki/wiki docs and add a changelog fix entry crediting @arlen8411 if the behavior changes.

## Clownfish Work Prompt

Fix the memory-wiki nested page discovery bug reported in https://github.com/openclaw/openclaw/issues/63208. Current main only scans direct Markdown children under page groups such as sources/, so nested pages like sources/OpenClaw配置文件指南/AGENTS.md are missed by compile, local query/get, and status.

Expected fix boundary: keep the change inside the memory-wiki plugin. Add or extract one shared deterministic Markdown page-file collector for wiki page groups, then use it from compile, query/get, and status counting. Preserve existing flat-page behavior. Traverse only inside the known page-group directories, do not follow symlinks, keep output ordering deterministic, and make an explicit compatibility choice for nested index.md files rather than accidentally indexing generated indexes. Do not add a core migration or bundled-plugin id special case in core.

Likely files: extensions/memory-wiki/src/compile.ts, extensions/memory-wiki/src/query.ts, extensions/memory-wiki/src/status.ts, and a small shared helper if useful; tests in extensions/memory-wiki/src/compile.test.ts, extensions/memory-wiki/src/query.test.ts, and extensions/memory-wiki/src/status.test.ts.
