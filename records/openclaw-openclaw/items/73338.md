## Summary

Keep this PR open. Current main still builds the default TUI Gateway URL from config/env/default port data and the gateway lock payload does not record the resolved runtime port, so the linked active-port bug remains unresolved on main. The PR is a focused implementation candidate, and its latest discussion says it addressed the startup-window review by trusting the lock owner without requiring the listener to be bound yet.

## Best Possible Solution

Keep this PR open for maintainer review. The best product path is to land one canonical fix for #42461 that records the resolved runtime Gateway port in the active lock, exposes a narrow reader that verifies the lock owner, makes default local `openclaw tui` connections use that port before falling back to config/defaults, and keeps regression coverage in `src/infra/gateway-lock.test.ts` plus `src/tui/gateway-chat.test.ts` with user-facing docs updates.
