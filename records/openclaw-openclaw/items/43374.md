## Summary

Keep open. Current main has partial hardening around provider HTTP transport, model fallback classification, Telegram polling watchdogs, and queueing documentation, but it still does not prove the reported four-agent Telegram timeout pattern is fixed. The Telegram runner still derives sink concurrency from the agent max-concurrent setting, the Gateway main lane uses the same configured agent limit, service environments still do not set a libuv threadpool default, and there is no current-main four-agent reproduction or instrumentation evidence tying the simultaneous provider timeouts to a fixed layer.

## What This Changes

Investigate and fix simultaneous LLM API timeouts when multiple Telegram-bound agents run concurrently, including fallback timeout behavior, Telegram polling concurrency, provider transport DNS/Undici behavior, service threadpool defaults, and session/watchdog tuning.

## Best Possible Solution

Keep this as an active reliability investigation until maintainers can reproduce or instrument the four-agent Telegram workload on current main, then land one focused fix in the measured layer: Telegram runner/backpressure, main-lane queueing, provider request transport, session lock timing, service environment defaults, or fallback timeout accounting.
