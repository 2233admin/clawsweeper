## Summary

Keep open. Current main still follows the reported Matrix UIA fallback path, does not parse MAS `org.matrix.cross_signing_reset` challenges or probe `/_matrix/client/v1/auth_metadata`, and still documents the old no-auth/dummy/password sequence. The issue is also paired with open implementation PR #74509, so closing this before that PR lands or is superseded would be premature.

## What This Changes

The issue requests a Matrix plugin fix for MAS-fronted Synapse cross-signing bootstrap so PAT-authenticated bots can use the MSC3967 first-publish path, preserve UIA sessions, and surface MAS reset URLs instead of the misleading password fallback.

## Best Possible Solution

Land or supersede #74509 with a Matrix plugin-owned fix that parses Matrix UIA 401 challenges, threads the UIA session through supported stages, surfaces a typed MAS reset URL for operator approval, keeps MSC3967 first-publish uploads available, and updates focused crypto-bootstrap tests plus Matrix operator docs.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to invoke the current `authUploadDeviceSigningKeys` callback with no configured password while no-auth and dummy UIA requests reject; current tests already assert the same misleading password fallback error, and the issue supplies a concrete MAS 401 response shape.

## Solution Assessment

Yes at the design level, but it is not implemented on main. Handling this in the Matrix plugin's cross-signing auth callback and homeserver-capability probe is the narrowest maintainable path; #74509 is the active candidate to review, land, or supersede.
