## Summary

Keep open. Current main still matches the report: Matrix cross-signing upload UIA only tries no-auth, dummy, and optional password auth, does not inspect advertised UIA stages or preserve the UIA session, and still retries a forced cross-signing reset after an initial bootstrap failure. The same-author implementation PR #74509 is open, so the issue should remain open for maintainer review or supersession rather than being closed independently.

## What This Changes

Request Matrix cross-signing bootstrap support for MAS-fronted Synapse/PAT bot accounts by honoring MSC3967 first-publish behavior, parsing UIA challenges, preserving sessions, and surfacing MAS cross-signing reset URLs instead of a password-only error.

## Best Possible Solution

Land or supersede #74509 with a Matrix plugin-owned fix that parses Matrix UIA 401 challenges, threads the UIA `session`, surfaces a typed MAS reset URL for operator approval, keeps MSC3967 first-publish uploads available, and updates focused crypto-bootstrap tests plus Matrix operator docs.

## Reproduction Assessment

Yes. The current code-level reproduction is high-confidence: make the Matrix SDK auth callback reject no-auth and dummy UIA with no configured password, and current tests assert the same misleading password fallback error; the issue also provides a concrete MAS 401 response shape from a live deployment.

## Solution Assessment

Yes at the design level. Handling this in `extensions/matrix` via the existing matrix-js-sdk `authUploadDeviceSigningKeys` callback is the narrowest maintainable path; the open PR still needs maintainer/security review before it can be treated as the accepted implementation.
