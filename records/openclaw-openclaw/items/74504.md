## Summary

Keep open. Current main still matches the report: Matrix cross-signing UIA only tries no-auth, dummy, and optional password fallback; it does not parse MAS reset stages, UIA params, or UIA session data, and the bootstrap path still forces a reset after initial failure. A same-author implementation PR is open, so the right cleanup is review or supersede that PR rather than closing this issue or starting a separate repair lane.

## What This Changes

Matrix cross-signing bootstrap should handle MAS-fronted Synapse UIA for PAT-authenticated bots by honoring MSC3967 first-publish, preserving UIA session data, and surfacing the MAS cross-signing reset URL instead of suggesting password fallback.

## Best Possible Solution

Land PR #74509 or a reviewed equivalent Matrix plugin fix that parses UIA 401 flows and params, preserves and resends UIA session data, surfaces a typed MAS cross-signing reset URL for operator approval, keeps the MSC3967 no-auth first-publish path available, and updates focused tests plus Matrix operator docs.

## Reproduction Assessment

Yes. The issue provides a concrete MAS/PAT reproduction path, and current-main source/tests reproduce the core behavior: no-auth, dummy, optional password fallback without UIA response parsing or session propagation, followed by a forced reset retry.

## Solution Assessment

Yes, the requested direction is the narrow maintainable path: fix the Matrix plugin's UIA handling around the existing matrix-js-sdk callback and Synapse/MAS contract, without adding a bot-side OAuth bypass or changing core plugin APIs.
