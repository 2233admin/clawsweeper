## Summary

Keep open. Current main still does not add `bailian` or `BAILIAN_API_KEY` to the provider auth surface, and the related target-repo PRs are closed unmerged while the open external fork PR is an implementation candidate rather than shipped behavior.

## What This Changes

The issue requests Bailian provider environment-key support so `BAILIAN_API_KEY` can be resolved for Bailian/Qwen-compatible model configs instead of producing HTTP 401 auth failures.

## Best Possible Solution

If Bailian naming is accepted, add it through the Qwen plugin-owned provider/auth metadata rather than a core-only special case, with regression coverage for auth lookup and provider alias behavior. If the alias is intentionally unsupported, update the Qwen/provider docs to make the supported provider ids, env vars, and SecretRef syntax explicit.

## Reproduction Assessment

Yes. Static source inspection is enough to reproduce the missing lookup: `bailian` has no candidate/evidence/setup mapping on current main, while the reporter also provided live confirmation that adding a Bailian mapping fixed their runtime 401.

## Solution Assessment

Unclear. A one-line env-candidate entry addresses the immediate lookup, but the maintainable fix should decide whether Bailian is an official Qwen compatibility alias and then update the plugin manifest, docs, and tests together.
