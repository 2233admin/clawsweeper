## Summary

Keep open. Current main still lacks a `bailian` provider auth env candidate, `BAILIAN_API_KEY` support, or a `bailian` alias to the bundled Qwen/Alibaba provider path. The report has concrete reproduction details and remains actionable in the OpenClaw repo.

## Best Possible Solution

Keep this issue open. Maintainers should either add a focused compatibility path for `bailian`/`BAILIAN_API_KEY` in the Qwen or Alibaba provider-owned surface with regression coverage for `resolveEnvApiKey("bailian")`, or explicitly document that users must use canonical `qwen`/`alibaba` ids with `QWEN_API_KEY`, `MODELSTUDIO_API_KEY`, or `DASHSCOPE_API_KEY`.

## Evidence

- **Issue and related PR context:** The issue report describes a concrete auth failure with `apiKey: "__env__:BAILIAN_API_KEY"`; the reporter later confirmed that adding a Bailian mapping fixed their local runtime. The provided related target PRs (#42894, #42954, #42978, #46387) are all closed unmerged, so none currently implements this on main.
- **Core auth env candidates omit Bailian:** `CORE_PROVIDER_AUTH_ENV_VAR_CANDIDATES` contains core entries for anthropic, openai, voyage, cerebras, anthropic-openai, and qwen-dashscope, but no `bailian` entry and no `BAILIAN_API_KEY` candidate.
  - file: [src/secrets/provider-env-vars.ts:11](https://github.com/openclaw/openclaw/blob/db40ec404a9178294dba3121758ff59459983a11/src/secrets/provider-env-vars.ts#L11)

A raw `bailian` provider therefore has no current env-key path.
