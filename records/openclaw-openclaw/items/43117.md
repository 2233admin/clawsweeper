## Summary

Keep open. Current main implements the useful subset already noted in the discussion: Gemini Embedding 2 preview, configurable output dimensions, structured Gemini embedding inputs, and image/audio file indexing through the memory file-indexing path. It still does not implement the central conversation-history attachment workflow: session transcript indexing extracts only message text and ignores persisted MediaPath/MediaPaths/MediaType/MediaTypes fields, so prior chat or Telegram attachments are not embedded for memory search. Video and PDF are also still outside the memory multimodal modality set.

## Best Possible Solution

Keep this issue open and narrow implementation to session-memory attachment indexing. Extend the session transcript indexing path to collect eligible persisted media path/type fields, resolve them through existing managed-media and sandbox safety boundaries, and emit structured EmbeddingInput parts for supported attachments while preserving the shipped Gemini Embedding 2 preview, output-dimensionality, and image/audio file-indexing behavior. Add regression coverage for chat or Telegram attachment recall and update docs to distinguish file-path media indexing from conversation-history attachment indexing, with video/PDF support either included deliberately or split into a follow-up.
