## Summary

Keep open. Current main implements the Gemini Embedding 2 subset, configurable output dimensions, and structured image/audio embeddings for file-backed memory paths, but it still does not implement the central requested workflow: embedding prior conversation attachments such as Telegram screenshots/photos from session history. Session transcript indexing flattens user/assistant message text and ignores MediaPath/MediaPaths metadata, so chat attachments are not added to memorySearch unless they also live in an indexed filesystem memory path.

## Best Possible Solution

Keep this issue open and narrow the implementation to session-memory attachment indexing. Extend the memory-core session transcript indexing path to discover eligible MediaPath/MediaPaths and MediaType/MediaTypes metadata, validate and resolve the files through existing managed-media and sandbox boundaries, and emit structured EmbeddingInput chunks for supported attachments while preserving the existing Gemini Embedding 2, outputDimensionality, and file-backed image/audio indexing behavior. Add regression coverage for chat or Telegram attachment recall, and update the memory docs to distinguish file-backed multimodal memory from conversation-history attachment indexing; treat video/PDF as an explicit scoped follow-up unless implemented in the same change.
