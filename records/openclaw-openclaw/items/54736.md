## Summary

Keep open. Current main still has the reported shared silent-token gap: doubled whitespace-separated `NO_REPLY` payloads do not match the core silent predicate, and the normalization path can strip one trailing token while leaving a visible `NO_REPLY`. The Quiubo guard is useful plugin-side mitigation and #69472 was useful prior art, but the related PR is closed unmerged and neither resolves the core behavior on main.

## What This Changes

Suppress token-only replies containing repeated whitespace-separated `NO_REPLY` or equivalent silent-token values so channel plugins relying on core filtering do not receive visible control text.

## Best Possible Solution

Update the shared silent-token predicate or normalization path so payloads made only of repeated whitespace-separated silent tokens are suppressed, while mixed-content replies continue to be delivered without leaked control tokens. Add focused regression coverage for exact matching, repeated tokens, custom tokens, media behavior, and the existing #19537 protections.
