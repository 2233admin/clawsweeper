## Summary

Keep open: current main still has the internal loader/install seams, but the requested operator-facing config field is absent and gateway startup/package-level repair still stages bundled runtime deps without a config opt-out or isolated execution root by default.

## What This Changes

Expose a public `plugins.installBundledRuntimeDeps` opt-out and default bundled plugin runtime dependency staging to isolated execution roots, with docs for the resulting operator matrix.

## Best Possible Solution

Add the config field to the public plugin config contract, thread it into gateway prestage and plugin loading so `false` skips bundled runtime dependency installs, default package-level install execution to an isolated root with atomic replacement, and document the supported operator matrix.

## Reproduction Assessment

Yes for the actionable code gap: source inspection shows the config key is rejected/untagged, the loader has only an internal opt-out, gateway startup does not consult config before prestaging, and package-level repair does not pass an isolated execution root. The production stall and disk-growth reports are credible linked field evidence, but they were not live-reproduced here.

## Solution Assessment

Yes. The narrow maintainable fix is to expose and consistently route the existing internal seams through the public config contract, rather than adding another environment-only workaround; the isolated-default change should be covered with targeted installer tests.
