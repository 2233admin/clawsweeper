## Summary

Keep open. Current `main` still lacks the public config field and default isolated package-level install behavior requested here, while the discussion provides concrete production field data and related runtime-dependency reports. This is actionable as one focused code/docs/test PR rather than cleanup to close.

## What This Changes

Expose an operator config switch to disable bundled runtime-dependency installs, make isolated install execution the default for bundled plugin runtime deps, and document the resulting behavior matrix.

## Best Possible Solution

Add a public boolean config field, thread it through gateway prestage and plugin loading so `false` skips bundled runtime-dependency installation and reports missing deps clearly, make package-level runtime-deps repair use an isolated execution root with atomic replacement by default, and update docs, generated config docs, tests, and changelog for the supported matrix.

## Reproduction Assessment

Yes for the code-level gap: current `main` has a strict public config schema without the requested key, only an internal loader option, gateway prestage that ignores any user opt-out, and package-level repair that defaults execution to the shared install root. The field reports provide credible production symptoms, but this read-only review did not run a live npm staging reproduction.

## Solution Assessment

Yes, with one scope decision. Exposing the existing loader seam and changing the runtime-deps owner path is the narrowest maintainable direction; maintainers should explicitly decide whether bundled channel startup staging is governed by the same `plugins` switch or a separate `channels` switch.
