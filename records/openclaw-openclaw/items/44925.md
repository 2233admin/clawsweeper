## Summary

Keep this open. Current main has useful subagent delivery hardening, but it still has a finite announce retry-limit give-up path, records failed delivery after give-up, treats run timeouts as terminal rather than auto-restarting, and the docs still describe announce as best-effort. Recent discussion adds current 2026.4.26 reproduction and task-ledger evidence, so this is not stale or implemented.

## What This Changes

The issue requests reliable subagent completion handling so completed, timed-out, or artifact-producing `sessions_spawn` work is retried, restarted, verified, or surfaced instead of being silently lost.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is a durable completion and delivery model that separates child completion, parent-session recording, external channel delivery confirmation, retry persistence, timeout policy, deduplication, and artifact/result verification, with regression coverage for fast responders and Telegram-style async delivery.

## Reproduction Assessment

Yes. The April 29 discussion gives a concrete two-agent `sessions_spawn` fast-responder path for the retry-limit warning, and current main still contains the matching finite retry and give-up code path. I did not run live Telegram in this read-only review.

## Solution Assessment

No. Current main is partial hardening, and PR #45200 covers only a last-resort notification for one Pattern 1 symptom; the maintainable fix needs a broader durable completion/delivery policy before a narrow automated repair can safely cover the issue.
