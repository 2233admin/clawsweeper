## Summary

Keep this open. Current main still has the reported finite subagent announce retry-limit give-up path, timeout completion still finalizes as a timeout rather than restart/recovery, docs still describe best-effort delivery with final give-up, and the latest discussion reports the same warning on v2026.4.26 with a concrete fast-responder reproduction. The existing hardening and related open PR cover parts of Pattern 1, but they do not fully address durable delivery, timeout policy, user-visible delivery confirmation, dedupe, or file/result verification.

## What This Changes

The issue requests reliable subagent completion handling: retry or surface failed announces, notify or restart timed-out subagents, and verify file-creation claims before reporting completion.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit completion and delivery state model that separates child completion, parent-session recording, external channel delivery, timeout policy, deduplication, and result verification, with regression coverage for fast responders and Telegram-style async delivery.
