## Summary

Keep open. Current main has additional delivery hardening, but the requested behavior is not fully implemented: announce delivery can still terminate as failed after a finite retry budget, completion-mode dispatch can still return undelivered, timeout handling records a timeout rather than restarting, and recent issue discussion reports the same retry-limit warning with fast responders.

## What This Changes

The issue asks for reliable subagent completion handling: retry or notify on announce failure, restart or notify on timeout, and verify claimed file outputs before announcing completion.

## Best Possible Solution

Keep this as the canonical tracking issue for reliable async subagent completion semantics. The desired end state is an explicit delivery state model that separates child completion, parent-session recording, external channel delivery, retry durability, timeout/restart policy, duplicate prevention, and result/file verification, with regression coverage for fast responders and Telegram-style async delivery.
