## Summary

Keep this PR open. Current main still does not suppress routine WhatsApp gateway banners during prompt assembly, but the PR's submitted disconnect filter is too broad: it would suppress terminal disconnect states that currently share the generic `WhatsApp gateway disconnected (status ...)` queued system event. Security review found no CI, dependency, lockfile, package-script, secret, or artifact-execution changes in the PR context; the blocker is functional correctness.

## Best Possible Solution

Keep the PR open and request a narrower implementation: suppress routine connect banners and only suppress disconnect events proven to be retryable/routine, or change the WhatsApp monitor to enqueue distinct high-signal system events for terminal logged-out/conflict/stopped states before filtering. Add focused tests for both routine flaps and terminal close modes. Treat #72043 as a broader follow-up for configurable operator-facing system-event policy, not as a replacement for this PR.
