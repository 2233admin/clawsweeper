## Summary

Keep open. Current main still has the reported write-before-validation behavior in `plugins enable` and `plugins disable`, and an open linked implementation PR (#73554) uses closing syntax for this issue, so the issue should stay open until that PR is reviewed, landed, or explicitly replaced.

## What This Changes

The issue requests that `plugins enable` and `plugins disable` reject nonexistent plugin ids before writing `plugins.entries` config state.

## Best Possible Solution

Add a pre-write registry lookup for `plugins enable` and `plugins disable`, resolve valid inputs to the canonical plugin id, exit non-zero with a clear `Plugin not found` message on a miss, and add focused policy tests proving no config write or registry refresh occurs for unknown ids.

## Reproduction Assessment

Yes. The source path on current main is enough to reproduce with high confidence: both commands pass the raw requested id into config mutation helpers before any discovered-plugin lookup, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The narrow maintainable solution is to validate the operator-supplied id against the discovered plugin snapshot before mutation while leaving the existing warn-only startup validation for historical stale config entries.
