## Summary

Keep open. Current main still writes plugin config for `plugins enable` and `plugins disable` before validating the requested id against the discovered plugin snapshot, and two open PRs (#73554 and #73623) already reference this issue with closing syntax. The best maintainer path is to review and land one canonical fix rather than close the issue or spawn another implementation lane.

## Best Possible Solution

Review and land one canonical implementation from #73554 or #73623. The fix should resolve the requested id against the discovered plugin snapshot before `enablePluginInConfig` or `setPluginEnabledInConfig`, exit non-zero with a clear `Plugin not found` message on a miss, skip config writes and registry refreshes on failure, and add regression tests for both enable and disable.
