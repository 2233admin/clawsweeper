## Summary

Keep open. Current main still has the reported write-before-validation path for `plugins enable` and `plugins disable`, and the issue is paired with open unmerged PR #73554 using closing syntax, so it should remain open until maintainers land, replace, or close that PR.

## What This Changes

The issue asks for `openclaw plugins enable/disable <id>` to reject nonexistent plugin ids before mutating `plugins.entries` and to exit non-zero with a clear not-found error.

## Best Possible Solution

Add a pre-write lookup in the `plugins enable` and `plugins disable` handlers using the discovered plugin snapshot, resolve accepted aliases to the canonical plugin id, and return exit 1 with `Plugin not found: <id>. Run openclaw plugins list to see installed plugins.` before config writes or registry refresh on a miss. Review and land #73554 if it satisfies that boundary and coverage; otherwise replace it with the same narrow fix.

## Reproduction Assessment

Yes. The issue body gives deterministic CLI steps, and current source directly shows both handlers mutating config before any discovered-plugin lookup; the helper then creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. The narrow maintainable solution is to validate the operator-supplied id against the discovered plugin registry before mutation while leaving existing warn-only stale-entry handling for startup/upgrade resilience.
