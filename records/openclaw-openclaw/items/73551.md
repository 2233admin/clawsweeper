## Summary

Keep open. Current main still mutates plugin policy config before proving the requested plugin id exists, and two open, unmerged PRs (#73554 and #73623) already reference this issue with closing syntax. The right maintainer path is to review and land one canonical fix, then let the merged PR close the issue.

## Best Possible Solution

Review and land one canonical fix from #73554 or #73623. The fix should resolve the requested id against the discovered plugin snapshot before `enablePluginInConfig` or `setPluginEnabledInConfig`, exit non-zero with a clear `Plugin not found` message on misses, skip config writes and registry refreshes on failure, preserve valid id/name/channel-alias behavior, and add focused regression tests for both enable and disable missing-id no-write paths.
