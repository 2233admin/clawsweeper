## Summary

Current main still appears to have the reported bug: `plugins enable` and `plugins disable` mutate plugin config from the raw requested id before checking that the id exists in the discovered plugin snapshot. Two open implementation PRs, #73554 and #73623, reference this issue with closing syntax, so the issue should stay open until one canonical fix is reviewed and merged.

## Best Possible Solution

Keep this issue open and review/land one canonical fix from #73554 or #73623. The fix should resolve the requested id against the discovered plugin snapshot before `enablePluginInConfig` or `setPluginEnabledInConfig`, exit non-zero with a clear `Plugin not found` message on a miss, skip config writes and registry refreshes on failure, and add regression tests for both enable and disable.
