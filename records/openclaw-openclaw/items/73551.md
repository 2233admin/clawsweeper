## Summary

Keep open. Current main still writes `plugins.entries.<id>` from the raw CLI argument before any discovered-plugin lookup, so the stale-entry bug remains; open PR #73554 is the active closing implementation candidate and should be reviewed, landed, or explicitly replaced before closing this issue.

## What This Changes

The issue requests that `plugins enable` and `plugins disable` reject ids that are not discovered plugins before mutating plugin config.

## Best Possible Solution

Add a pre-write registry lookup in the `plugins enable` and `plugins disable` handlers, resolve valid user input to the canonical plugin id, and exit non-zero with a clear `Plugin not found` message before config writes, slot selection, or registry refresh on a miss. Keep the existing warn-only stale-entry validation for resilient startup, and cover both commands with no-write regression tests.

## Reproduction Assessment

Yes. The issue's CLI repro is backed by the current source path: enable/disable pass the raw requested id into config mutation helpers before any discovered-plugin lookup, and the helper creates `plugins.entries.<id>` for arbitrary ids.

## Solution Assessment

Yes. Validating the operator-supplied plugin id against the discovered registry before mutation is the narrow maintainable fix; #73554 is the current implementation candidate to review against alias/canonical-id behavior and no-write coverage.
