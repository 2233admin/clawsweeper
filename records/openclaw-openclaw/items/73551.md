## Summary

Keep open. Current main still mutates plugin config for `plugins enable` and `plugins disable` before proving the requested id exists, and an open linked PR already uses closing syntax for this bug, so maintainers should review, land, or replace that PR before closing the issue.

## What This Changes

The issue asks for `openclaw plugins enable/disable <id>` to reject nonexistent plugin ids with a nonzero exit and without writing stale `plugins.entries` config.

## Best Possible Solution

Add a pre-write lookup in the `plugins enable` and `plugins disable` handlers using the same discovered plugin snapshot used by listing/inspection, resolve valid aliases to the canonical id, and exit 1 with a clear `Plugin not found` message before any config write on a miss.

## Reproduction Assessment

Yes. I did not run the CLI because this review is read-only, but the current source path directly reproduces the reported behavior: both commands pass the operator-supplied id into mutation helpers before any discovered-plugin existence check.

## Solution Assessment

Yes. The narrow maintainable solution is to validate the requested id before mutation and leave existing stale-entry cleanup to `openclaw doctor --fix`; the open linked PR follows that direction and should be reviewed or replaced.
