## Summary

Keep this issue open. Current main has separate image/video generation tools and now accepts broader input modality labels in provider model config, but the reported gap remains: custom `models.providers` model entries still have no output modality/capability field, model selection carries only input metadata, and configured Poe-style models route through the LLM streaming path rather than the image/video generation provider contracts. The stale label does not make this stale enough to close because the report and comments identify a concrete, still-actionable diagnostic/product-boundary issue.

## Best Possible Solution

Keep this open for an explicit product/API decision. The short-term path should make the text-output session boundary clear: document `models.providers` as LLM/chat-output config, add a supported way to declare or reject non-text output models, and emit a clear unsupported media-output diagnostic when such a model is selected. Full Poe image/video support should be implemented through the existing image/video generation provider APIs rather than by sending media-output models through the chat stream adapter.
