## Summary

Keep open. Current main still has the OpenClaw-side contract gap described in the issue: custom `models.providers` entries can describe input modalities but not output modalities, and configured provider models still enter the LLM streaming path rather than the media-generation tool/provider surfaces.

## What This Changes

The issue asks OpenClaw to either support Poe image/video-output models through the proper media-generation runtime or explicitly reject them with clear diagnostics when configured as chat/session models.

## Best Possible Solution

Add an explicit text-output session boundary: keep `models.providers` usable for LLM/chat models, add a supported way to declare or detect non-text output models with a clear unsupported-media diagnostic, and route real Poe image/video generation through `image_generate`/`video_generate` provider surfaces or a Poe plugin implementation.

## Reproduction Assessment

Yes for the OpenClaw-side mismatch: current code accepts configured provider model rows without output metadata, copies them into the model catalog, and resolves them as LLM stream models. The exact live Poe backend response was not rerun here.

## Solution Assessment

No current-main solution exists yet. The narrowest maintainable short-term fix is explicit unsupported media-output handling plus docs; full Poe media generation support should be a separate provider/plugin implementation on the media-generation surfaces.
