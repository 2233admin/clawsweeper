## Summary

Keep open. Current `main` does not implement this feature, the PR has the protected `maintainer` label, and the current PR head still has a concrete P1 security flaw in canonical-path authorization for file-transfer node invokes.

## What This Changes

The PR adds a bundled `file-transfer` plugin with file and directory node-host commands/tools, a plugin node-invoke policy SDK seam, gateway invoke enforcement, media-store subdir hardening, docs/generated baseline updates, dependencies, and tests.

## Best Possible Solution

Keep the plugin-owned policy model, but require every resolved canonical path and dir-fetch entry to be independently authorized by a real allowlist match or a fresh canonical-path approval before bytes are read or written, then rerun the targeted file-transfer, gateway, SDK, type, lint, and changed-surface gates.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Require canonical paths to match file-transfer policy:** `extensions/file-transfer/src/shared/node-invoke-policy.ts:343-344`
  - body: `evaluateFilePolicy()` returns `ok` for `ask: "always"` before checking the allow lists, but this canonical write preflight treats any `ok` result as authorization. With `followSymlinks=true`, an approved requested path can resolve to a target outside `allowWritePaths` and still continue to the mutating invoke. Require canonical rechecks to prove `reason === "matched-allow"`, or prompt/deny the canonical target before continuing, and apply the same rule to dir-fetch entries and final postflight checks.
  - confidence: 0.88

On PR https://github.com/openclaw/openclaw/pull/74134, fix the file-transfer canonical-path authorization bug.
