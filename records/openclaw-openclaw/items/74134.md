## Summary

Keep open. This protected, security-sensitive PR is not implemented on current main, and the latest head still has a blocking canonical-path policy gap: `ask: "always"` can make canonical symlink targets look policy-approved without requiring an allowlist match or a second approval for the resolved path.

## What This Changes

The PR adds an enabled bundled file-transfer plugin, new node-host file/directory commands and agent tools, a plugin node-invoke policy SDK seam, media-store hardening, docs, generated baselines, and tests for binary file operations on paired nodes.

## Best Possible Solution

Keep the PR open, change canonical-path validation so resolved paths require a real `matched-allow` result or a separate approval/deny decision for the canonical path before bytes are returned or written, add regression tests for `ask: "always"` with `followSymlinks=true` across read/write paths, then rerun the targeted file-transfer, gateway policy, SDK export/API, type, and lint gates before maintainer sign-off.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Require canonical file paths to match policy:** `extensions/file-transfer/src/shared/node-invoke-policy.ts:343-344`
  - body: The canonical-path checks treat any `ok` result from `evaluateFilePolicy()` as allowed. With `ask: "always"`, that function returns `ok` before matching `allowReadPaths` or `allowWritePaths`, so a symlinked requested path can be approved while its canonical target outside the allowlist is accepted without a second prompt. Require `reason === "matched-allow"` for canonical paths, or prompt/deny using the canonical path before returning or writing bytes.
  - confidence: 0.86
