## Summary

Keep this PR open. Current main still lacks a same-install running-gateway guard before git-mode update work, so the PR is addressing a real update crash path; however, the proposed patch still has blocking issues around wrapper-backed service commands and gateway restoration after skipped or failed updates.

## What This Changes

The PR adds a git-mode update preflight and tests that detect a running managed gateway for the same install, auto-stop it when restart is enabled, hard-block with `--no-restart`, and allow different installs.

## Best Possible Solution

Revise the PR so git-mode updates detect same-install managed gateways across direct dist entrypoints and wrapper-backed service commands, then either fail closed before mutating the checkout or stop the service only with guaranteed restoration across success, skipped, error, and thrown update paths. Keep `--no-restart` as a clear stop-only block and add focused coverage for wrapper, dirty/skipped, failed-update, and successful restart cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.91

Full review comments:

- **[P1] Handle wrapper-backed service commands:** `src/cli/update-cli/update-command.ts:132`
  - body: `findGatewayServiceEntrypoint` returns the token immediately before `gateway`, but wrapper installs store service args as `[wrapperPath, 'gateway', ...]`. That compares the wrapper path to `dist/index.js`, so documented wrapper-backed services can bypass the guard and still update under a live same-install gateway.
  - confidence: 0.91
- **[P1] Restore the gateway after stopped git updates fail:** `src/cli/update-cli/update-command.ts:1125-1126`
  - body: The PR stops the service before `runGitUpdate`, but the existing skipped/error/throw paths only restart `prePackageServiceStop`, which is never set for git updates. A dirty checkout, failed fetch/rebase, or failed build can therefore leave a previously running managed gateway down even though no update was applied.
  - confidence: 0.9
- **[P2] Remove restart-before-update guidance:** `src/cli/update-cli/update-command.ts:1109-1114`
  - body: In the `--no-restart` block, the message still tells users to stop or restart the gateway before rerunning update. Restarting brings the service back up, so the next update invocation hits the same hard block; the guidance should be stop, update, then restart after the update finishes.
  - confidence: 0.86
