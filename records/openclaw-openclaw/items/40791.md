## Summary

Keep open. Current main still lacks the external rescue watchdog, the overlapping core rescue work is open and unmerged, and this PR branch has concrete correctness and security-hardening blockers that need normal maintainer review rather than conservative cleanup closure.

## What This Changes

The PR adds an opt-in Bash rescue watchdog, a Claude Code incident runner, systemd user timer templates, rescue-watchdog docs, and a few test typing updates.

## Best Possible Solution

Maintainers should choose the canonical recovery architecture first. If this shell watchdog remains desired, keep it scoped to scripts/systemd/docs, fix the state and concurrency bugs, harden the state-file and incident-bundle security model, add focused scripted coverage, and rebase away unrelated test fixes; if #46502 becomes canonical, salvage any useful docs/systemd prior art in a follow-up and supersede this PR after that path lands or is explicitly selected.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Reset suppression state after healthy checks:** `scripts/rescue-watchdog.sh:175-177`
  - body: When the watchdog sees a healthy cycle, it exits before updating `state.env`. A fail -> recover -> fail sequence within `OPENCLAW_RESCUE_COOLDOWN_SEC` can reuse the old fingerprint and skip the new outage, so no incident, runner, or notification fires for the fresh failure.
  - confidence: 0.86
- **[P2] Use collision-proof incident directories:** `scripts/rescue-watchdog.sh:196-198`
  - body: `incident_id` only has second-level granularity and `mkdir -p` reuses an existing directory. Two overlapping timer/manual runs in the same second can write into the same bundle, leaving mixed or partial incident artifacts instead of independent incidents.
  - confidence: 0.88
- **[P2] Persist suppression state before runner execution:** `scripts/rescue-watchdog.sh:250`
  - body: `state.env` is saved only after `RUNNER_CMD` returns. During a long Claude rescue run, another watchdog invocation sees no updated fingerprint and can launch a duplicate runner/notification cycle for the same outage.
  - confidence: 0.89

## Security Review

Status: needs_attention

Summary: The PR introduces an opt-in local command-execution and diagnostic-capture path that needs hardening before it is safe as repo-maintained automation.

Concerns:

- **[medium] Harden sourced watchdog state:** `scripts/rescue-watchdog.sh:51`
  - body: `load_state` sources `$STATE_FILE` from an environment-configurable state directory without regular-file, ownership, mode, or symlink checks. A timer that sources mutable state should fail closed on unsafe state paths because replacing `state.env` turns the next watchdog run into shell execution.
  - confidence: 0.84
- **[medium] Warn about sensitive incident bundles:** `docs/automation/rescue-watchdog.md:38`
  - body: The docs list health snapshots, status output, gateway log tails, git status/diff stats, and runner output in the incident bundle, while the runner adds the incident directory to model context. Operators need explicit guidance on permissions, retention, and possible secret or local diagnostic exposure before enabling this flow.
  - confidence: 0.82
