## Summary

Keep open. The requested TUI reconnect UX is still not implemented on current main, but this PR is not merge-ready because it bypasses the current generic TUI backend contract and weakens macOS SSH host-key verification. Earlier pairing/status review comments were addressed in the branch discussion, but the remaining P1 issues still need a rebase and maintainer review.

## What This Changes

The PR adds TUI gateway-disconnect guidance and Ctrl+R manual reconnect handling, exposes GatewayClient/GatewayChatClient reconnect timer controls, adds macOS remote-gateway probe/auth UI handling, and adjusts diffs cleanup-test/callsite guard coverage.

## Best Possible Solution

Land a rebased and narrowed version that models manual reconnect as an optional TuiBackend capability or gateway-only path, shows disconnect guidance only for reconnectable gateway states, preserves strict macOS SSH host-key checks, and updates TUI docs/tests for any new shortcut.

## Reproduction Assessment

Yes. The backend bug is reproducible by static source inspection of the PR diff against current main: runTui calls client.reconnect() while current TuiBackend and non-gateway implementations do not expose that method. The SSH issue is also directly visible in the PR diff at the added StrictHostKeyChecking=accept-new option.

## Solution Assessment

No. The user-facing reconnect hint is reasonable, but this implementation is not the safest path; reconnect should be routed through a typed optional backend capability or gateway-specific guard, and macOS diagnostics should reuse the shared strict SSH options.

## Review Findings
