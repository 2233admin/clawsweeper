## Summary

Keep open. Current main still has the source-level path described in the report: shared-session heartbeats can select the configured heartbeat model, persist that runtime provider/model into the normal session entry, and the session status card then treats those persisted runtime fields as the selected model when there is no explicit user override.

## What This Changes

The issue requests that heartbeat/internal runs using `agents.defaults.heartbeat.model` stop overwriting or being displayed as the normal user-session runtime model.

## Best Possible Solution

Introduce a run-kind-aware runtime metadata contract that keeps user-turn selected/runtime model fields separate from heartbeat or other internal-run diagnostics, while preserving normal fallback/runtime visibility in status surfaces.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is: configure a shared-session heartbeat with `heartbeat.model` and `isolatedSession` unset, let the heartbeat run, then inspect the session entry or `session_status`; the current code path persists the heartbeat runtime provider/model and status renders it as the primary model.

## Solution Assessment

Yes. The narrowest maintainable fix is a run-kind-aware persistence/display boundary, not a rollback of runtime-model status and not relying on `heartbeat.isolatedSession` as the only mitigation.

## Repair Work Prompt

Fix openclaw/openclaw#74217. Current main lets shared-session heartbeat/internal runs using `agents.defaults.heartbeat.model` persist their runtime provider/model into the normal session entry, and `session_status` then renders that as the selected user-session model when no explicit override exists. Add a run-kind-aware boundary so heartbeat/internal runs do not overwrite `SessionEntry.modelProvider` / `SessionEntry.model` as the user-turn runtime model, or persist their runtime identity separately and label it separately in status. Preserve the merged #55425 behavior for normal user turns: actual runtime/fallback models must still be visible in `session_status`. Do not make `heartbeat.isolatedSession: true` the only fix and do not change unrelated heartbeat delivery semantics. Likely implementation path: thread a run kind or persistence flag from `GetReplyOptions.isHeartbeat` through `runPreparedReply` / `runReplyAgent` into `persistRunSessionUsage` and `persistSessionUsageUpdate`, then skip or separate selected-runtime model persistence for heartbeat/internal runs while keeping usage/context accounting as appropriate. Add regression coverage for a shared-session heartbeat with `heartbeat.model`, no overrides, and `isolatedSession` unset; verify the main session status remains on the configured/default user model while normal user-turn runtime persistence still works. Update heartbeat docs and add a user-facing changelog fix entry with proper contributor attribution.
