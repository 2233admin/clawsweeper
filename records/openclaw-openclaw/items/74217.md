## Summary

Keep this open. Current main still has the core behavior described: shared-session heartbeat runs can select the configured heartbeat model, persist that runtime provider/model onto the same session entry, and `session_status` still treats persisted runtime identity as the selected card model when no explicit model override exists. Recent code now detects and documents “heartbeat model bleed” for context-overflow recovery, but that is a diagnostic/workaround rather than a fix.

## What This Changes

Heartbeat/internal async-completion runs using `agents.defaults.heartbeat.model` should not make a user-facing session status/runtime model appear switched to the heartbeat model.

## Best Possible Solution

Add a run-kind-aware boundary for model persistence and display: heartbeat/internal async-completion runs should either not overwrite the session fields used as selected user-turn model state, or should persist their runtime identity in separate internal-run metadata that status labels separately from the normal selected/configured model.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: run a shared-session heartbeat with `heartbeat.model`, no explicit session override, and empty fallbacks; the heartbeat model is selected, then `modelUsed`/`providerUsed` are persisted to the shared session entry and `session_status` uses those persisted fields as the card model. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

Yes. The narrowest maintainable solution is a run-kind-aware persistence/display split rather than treating every runtime model write as selected user-session model state; that preserves real fallback/runtime diagnostics while stopping heartbeat/internal events from masquerading as a user model switch.

## Repair Work Prompt

Fix the heartbeat/internal runtime model bleed reported in https://github.com/openclaw/openclaw/issues/74217. Current main lets shared-session heartbeat runs select `agents.defaults.heartbeat.model`, then persists that runtime provider/model into `SessionEntry.modelProvider` / `SessionEntry.model`, and `session_status` displays those persisted fields as the selected user-session model when no explicit override exists. Add a run-kind-aware boundary: either prevent heartbeat/internal async-completion runs from overwriting the session fields used for selected user-turn model/status, or persist heartbeat runtime identity separately and label it separately in status. Preserve legitimate fallback/runtime diagnostics, `modelOverrideSource: "auto"`, explicit user overrides, channel model overrides, and normal user-turn runtime usage accounting. Do not solve by disabling heartbeat model selection or by reverting the broader runtime-status behavior from #55425/#65807. Add regression coverage for a shared-session heartbeat with `heartbeat.model`, no explicit override, and empty fallbacks proving status does not imply a user model switch; update heartbeat/status docs and add a user-facing changelog fix entry with proper non-bot attribution.
