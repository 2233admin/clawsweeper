## Summary

Keep open. Current main still has the source-level path described in the report: heartbeat runs can resolve `agents.defaults.heartbeat.model`, then persist that runtime provider/model into the shared user session, and `session_status` uses the persisted runtime identity as the card model when there is no explicit user override. The related status work in #55425 is useful for normal turns, but it also means heartbeat/internal-run metadata needs a separate persistence or display boundary rather than being treated as the selected user-session model.

## What This Changes

The issue asks OpenClaw to prevent heartbeat/internal async-completion model selection from making normal user session status look switched to the heartbeat model.

## Best Possible Solution

Keep #55425's normal-turn runtime model visibility, but add a run-kind-aware runtime metadata boundary so heartbeat, cron, and other internal runs either do not overwrite user-turn session `model`/`modelProvider` or are displayed as separate internal-run diagnostics in status surfaces.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a shared-session heartbeat with `heartbeat.model`, leave `isolatedSession` unset, let the heartbeat run, and then inspect the session entry or `session_status`; current main has an unguarded path from heartbeat model selection to persisted session runtime model fields.

## Solution Assessment

Yes, the narrowest maintainable fix is a run-kind-aware persistence/display boundary. Reverting runtime-model status display would regress #55425, and relying on `heartbeat.isolatedSession` would leave the shared-session bug and misleading status semantics in place.
