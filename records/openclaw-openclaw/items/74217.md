## Summary

Keep this open. Current main still has the reported behavior path: shared-session heartbeat runs can select the configured heartbeat model, persist that runtime provider/model to the same session entry, and have `session_status` display those persisted runtime fields as the selected model when no explicit user override exists. The related status PRs overlap the display surface, but they do not add the run-kind boundary needed for heartbeat/internal runs.

## What This Changes

Heartbeat or internal async-completion runs using `agents.defaults.heartbeat.model` should not make a user-facing session appear switched to the heartbeat runtime model.

## Best Possible Solution

Add a run-kind-aware boundary between selected user-turn model state and internal heartbeat/background runtime identity. The model used by a heartbeat should either not overwrite the session fields used as the selected status-card model, or it should be stored separately and labeled separately in status.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is clear: run a shared-session heartbeat with `heartbeat.model`, no explicit session override, and empty fallbacks; the heartbeat model is selected, persisted to the shared session entry, and then status treats those persisted fields as the card model. I did not run tests because this review is read-only.

## Solution Assessment

Yes. The narrowest maintainable repair is a run-kind-aware persistence/display split rather than treating every runtime model write as selected user-session model state; that preserves normal fallback/runtime diagnostics while stopping heartbeat/internal events from masquerading as a user model switch.

## Repair Work Prompt

Fix the heartbeat/internal runtime model bleed reported in https://github.com/openclaw/openclaw/issues/74217. Current main lets shared-session heartbeat runs select `agents.defaults.heartbeat.model`, then persists that runtime provider/model into `SessionEntry.modelProvider` / `SessionEntry.model`, and `session_status` displays those persisted fields as the selected user-session model when no explicit override exists. Add a run-kind-aware boundary: either prevent heartbeat/internal async-completion runs from overwriting the session fields used for selected user-turn model/status, or persist heartbeat runtime identity separately and label it separately in status. Preserve legitimate fallback/runtime diagnostics, `modelOverrideSource: "auto"`, explicit user overrides, channel model overrides, normal user-turn runtime usage accounting, and the shipped ordinary runtime-status behavior from #55425/#65807. Do not solve this by disabling heartbeat model selection or by reverting runtime model display for normal turns. Add regression coverage for a shared-session heartbeat with `heartbeat.model`, no explicit override, and empty fallbacks proving status does not imply a user model switch. Update heartbeat/status docs if behavior changes and add a user-facing CHANGELOG.md fix entry crediting Thanks @Sitham94; include @Staruy only if the repair directly addresses #74257 too.
