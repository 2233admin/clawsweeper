## Summary

Keep this PR open. Current main still has the Signal daemon shutdown race the PR targets, the linked bug remains open, and the supplied diff is a narrow Signal-owned lifecycle fix with no blocking review findings found in this read-only pass.

## What This Changes

The PR changes Signal daemon shutdown to be asynchronous, wait for process exit, escalate from SIGTERM to SIGKILL after a timeout, await that lifecycle from the Signal monitor, and add focused regression tests.

## Best Possible Solution

Land a Signal-owned lifecycle fix that makes daemon stop idempotent and awaitable, waits for process exit, escalates after a bounded timeout, and keeps regression coverage for both daemon stop semantics and aborted monitor shutdown.

## Reproduction Assessment

Yes. Current main has a high-confidence code-level reproduction path: restart the Signal channel in autostart mode, `stopChannel()` waits for the monitor task, but the monitor currently resolves after a fire-and-forget SIGTERM instead of waiting for the child daemon to exit.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable fix because Signal owns the `signal-cli` daemon lifecycle in autostart mode, and no core/plugin API change is needed for callers to wait on the plugin-owned monitor task.
