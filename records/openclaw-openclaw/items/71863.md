## Summary

Keep this PR open. Current main still has the Signal daemon shutdown race the PR targets: daemon stop is synchronous, monitor cleanup does not await process exit, and gateway channel restart relies on stopChannel waiting for the channel task before starting the channel again. The PR is narrowly scoped to Signal daemon/monitor lifecycle and tests, references the still-open linked bug #22676, and does not introduce unexplained security-sensitive workflow, dependency, lockfile, or package-publishing changes.

## What This Changes

This PR changes the Signal plugin daemon handle to stop asynchronously with SIGTERM/SIGKILL fallback, awaits daemon cleanup from the Signal monitor, and adds focused Signal daemon/monitor lifecycle regression tests.

## Best Possible Solution

Keep this PR open for normal maintainer review, CI, and live Signal restart validation. The best implementation path is the Signal-owned approach proposed here: make daemon stop idempotent and async, send SIGTERM, escalate to SIGKILL after a bounded timeout, await cleanup from monitorSignalProvider, and keep focused regression tests for daemon stop semantics plus aborted monitor shutdown; no core/plugin API change appears necessary.
