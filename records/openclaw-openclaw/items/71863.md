## Summary

Keep this PR open. Current main still has the Signal daemon shutdown race the PR is trying to fix: `SignalDaemonHandle.stop` is synchronous, `spawnSignalDaemon().stop()` only sends SIGTERM, and `monitorSignalProvider` does not await child-process exit before channel restart can start a new daemon. The PR is also paired with the still-open linked bug #22676, so the right next step is maintainer review/CI, not cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review and CI. The best implementation path is the Signal-owned fix proposed here: make daemon `stop()` idempotent and async, send SIGTERM, wait for child exit, escalate to SIGKILL after a bounded timeout, await cleanup from `monitorSignalProvider`, and keep focused regression tests for stop wait/escalation plus aborted monitor shutdown. No core/plugin API change appears necessary for this bug.
