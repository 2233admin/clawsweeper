## What This Changes

The PR makes the Signal daemon stop path await child-process exit, escalates from SIGTERM to SIGKILL after a timeout, awaits Signal monitor cleanup, and adds daemon/monitor regression tests.

## Best Possible Solution

Land a Signal-owned lifecycle fix that makes daemon stop idempotent and awaitable, waits for process exit, escalates after a bounded timeout, and keeps regression coverage for daemon stop semantics and aborted monitor shutdown without adding a core API change.

## Reproduction Assessment

Yes. Current main has a high-confidence code-level reproduction path: restart the Signal channel in autostart mode, `stopChannel()` waits for the monitor task, but the monitor currently resolves after a fire-and-forget SIGTERM instead of waiting for the child daemon to exit.

## Solution Assessment

Yes. The proposed direction is the narrowest maintainable solution because Signal owns the `signal-cli` daemon lifecycle, and the gateway already waits on plugin channel tasks during restart.
