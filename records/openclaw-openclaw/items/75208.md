## Summary

Keep open for maintainer review. Current main still lacks the artifact RPC implementation, so the work is not obsolete, but the branch has a blocking pagination bug and overlaps the maintainer-owned artifact API thread and implementation branch.

## What This Changes

The branch adds Gateway and SDK artifact list/get/download RPCs with cursor pagination and type filtering, generated Swift models, App SDK docs/tests, plus Hyprland screenshot and Windows executable-resolution changes.

## Best Possible Solution

Land the artifact API through the canonical maintainer path after fixing cursor semantics to identify a unique artifact position, adding a multi-artifact-message pagination regression test, and separating unrelated browser or Windows fixes into their own PRs.

## Reproduction Assessment

Yes for the PR regression: create one transcript message with two artifact blocks, call `artifacts.list` with `limit: 1`, then request the next page with the returned cursor. The second artifact is skipped because the cursor only stores `messageSeq`.

## Solution Assessment

No. The overall direction matches the maintainer request, but this patch is not the best final form until pagination uses a stable tiebreaker such as `messageSeq` plus `contentIndex` or artifact id, and the Hyprland docs either match the implemented capture path or move to a separate implementation PR.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve artifacts that share a message sequence:** `src/gateway/server-methods/artifacts.ts:388`
  - body: The cursor only encodes `messageSeq`. If one transcript message contains more artifacts than the page limit, `nextCursor` encodes that message's sequence and the next request filters with `messageSeq > cursorSeq`, so the remaining artifacts from the same message are never returned. Include a tiebreaker such as `contentIndex` or artifact id in the cursor and add a page-split regression test.
  - confidence: 0.95
- **[P3] Align the Hyprland docs with the implemented capture path:** `docs/tools/browser-linux-troubleshooting.md:183-186`
  - body: This section says OpenClaw creates a virtual output and captures it with `grim -o`, but the added implementation only inspects `hyprctl clients -j` and runs `grim -g` against the current window geometry. Either implement the documented virtual-output lifecycle or change the docs to describe the actual fallback path.
  - confidence: 0.88
