## Summary

Keep this PR open. Current main still does not implement the PR’s intended hard approval boundary for destructive or mutating tool calls from external-content hook sessions, and the PR’s own diff remains non-merge-ready because the new guard is not wired into production dispatch and the web_search warning change conflicts with current tests.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up, or supersede it with a production-wired before-tool-call trusted policy. The best implementation should carry external-content provenance into runtime tool context, use or align with isMutatingToolCall, route required approvals through the existing plugin approval path, honor deliberate allowUnsafeExternalContent bypasses, and update tests/docs for any intentional web_search warning behavior change.

## Risks / Open Questions

- Merging this PR as-is would advertise a hard approval boundary while the new guard remains test-only and unused by production tool dispatch.
- The PR’s destructive-tool regex is narrower than current action-aware mutation detection, leaving several existing mutating surfaces outside the proposed gate.
- A raw hook:* session-key check can miss agent-scoped hook sessions unless the production fix carries explicit external-content provenance into the tool-call context.
- Changing web_search to always include the long SECURITY NOTICE contradicts current tests and should be handled as a deliberate behavior/docs update.
