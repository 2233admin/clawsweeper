## Summary

Keep this PR open. Current main still has prompt wrapping and existing generic/plugin approval seams, but it does not implement the PR’s intended hard approval boundary for mutating/destructive tool calls from external hook sessions. The PR also remains non-merge-ready as described in the existing review comments: its new guard is not wired into production dispatch, and its web_search warning behavior conflicts with current tests.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up, but request changes or supersede it with a production-wired policy. The strongest path is a before-tool-call trusted policy that carries external-content provenance into runtime context, reuses `isMutatingToolCall`-style classification, routes required approvals through the existing plugin approval path, honors deliberate `allowUnsafeExternalContent` bypasses, and updates tests/docs for any intentional `web_search` warning behavior change.

## Risks / Open Questions

- Merging this PR as-is would advertise a hard external-content approval boundary while the new guard is not called by production tool dispatch.
- A guard based only on raw `hook:*` session keys can miss agent-scoped hook sessions such as `agent:main:hook:webhook:42`, which current cron normalization produces.
- The PR’s destructive-tool regex is narrower than current action-aware mutation detection, leaving several outbound, control-plane, node, and session mutation paths ungated.
- Changing `web_search` to always include the long security notice contradicts current main’s tests and should be handled as a deliberate behavior/docs change.
