## Summary

Keep this PR open. Current main still does not implement the PR’s intended hard approval boundary for destructive tool calls in external-content hook sessions, and the submitted diff is not merge-ready because its new guard is not wired into production tool dispatch. The security review did not find unrelated dependency, workflow, or package-supply-chain changes in the provided PR file list, but the functional security risk remains: merging it as-is would advertise enforcement that does not run.

## Best Possible Solution

Keep this PR open, but request changes or supersede it with a production-wired core policy. The best implementation is a before-tool-call or trusted-tool-policy gate that carries external-content provenance into tool runtime context, uses `isMutatingToolCall`-style action-aware classification, routes required approvals through the existing plugin approval path, honors deliberate `allowUnsafeExternalContent` bypasses, and updates tests/docs for any intentional `web_search` warning behavior change.

## Risks / Open Questions

- Merging the PR as-is would present a hard external-content approval boundary while the new guard is not invoked in production.
- A guard based only on raw `hook:*` session keys would likely miss normalized agent-scoped hook sessions such as `agent:main:hook:webhook:42`.
- The PR’s destructive-tool regex is narrower than current action-aware mutation detection, leaving several outbound, control-plane, node, and session paths ungated.
- Changing `web_search` to always include the long `SECURITY NOTICE` contradicts current tests and should be treated as an intentional behavior/docs update.
