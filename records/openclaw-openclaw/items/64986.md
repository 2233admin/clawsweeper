## Summary

Keep open. Current main still leaves the reported built-in Test Voice Wake failure source-plausible: the UI tells users to say only a trigger word, while the tester only succeeds when it extracts non-empty command text after the trigger. The overlapping Mac mini report was closed as covered by this issue, so this is now the canonical open tracker rather than a duplicate.

## What This Changes

The issue reports that macOS companion Voice Wake and the built-in Test Voice Wake fail despite microphone and Speech Recognition permissions, enabled Voice Wake settings, configured triggers, and an explicit `en-US` locale.

## Best Possible Solution

Make the built-in Test Voice Wake behavior match its UI contract by accepting trigger-only speech, preferably through a shared helper that mirrors the runtime's trigger-only pause semantics without forwarding to the gateway. Add focused Swift regression coverage for bare and filler-prefixed trigger words, plus a false-positive case for ordinary speech containing the trigger later in the phrase; then use live macOS testing to decide whether any runtime listener bug remains.

## Repair Work Prompt

Fix the built-in macOS Test Voice Wake mismatch tracked by openclaw/openclaw#64986. Current UI in `apps/macos/Sources/OpenClaw/VoiceWakeTestCard.swift` tells users to say only a trigger word, but `VoiceWakeTester` only succeeds when `match.command` is non-empty and its fallback uses `minCommandLength: 1`, so a bare trigger like `openclaw` or filler-prefixed `uh openclaw` can fail. Update the macOS tester path to accept trigger-only speech consistently with the runtime trigger-only pause semantics, preferably via a small shared helper rather than duplicating fragile string logic. Do not make the tester forward to the gateway, do not change Talk Mode/PTT/TTS behavior, and do not broaden gateway/core routing. Add focused Swift regression coverage for trigger-only tester detection and a false-positive case such as ordinary speech ending with a trigger mention. Because this is user-facing, add a single-line active changelog fix entry with `Thanks @zoiks65`. Validate with targeted macOS Swift tests and formatter checks, then run the changed gate via the normal Testbox path if the branch is handed off for merge.
