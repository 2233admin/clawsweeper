## Summary

Keep open. Current main still has only fixed-provider TTS selection plus fallback; there is no `languageRouting` config, `auto: "inbound"` is only an inbound-audio gate, and the batch audio transcription path does not preserve detected-language metadata that TTS routing could consume. The related `before_tts` hook request (#69307) overlaps with per-message voice routing but does not supersede this issue because this request specifically needs a public language-routing contract and STT/reply language source semantics.

## What This Changes

Add backward-compatible language-aware TTS routing so inbound voice replies can choose provider, model, and voice from detected input or reply language while preserving existing fixed TTS behavior as fallback.

## Best Possible Solution

Design a backward-compatible `messages.tts.languageRouting` contract in the TTS config surface, carry optional STT detected-language metadata through media-understanding and inbound context, apply routing before provider synthesis with fixed-provider fallback semantics, and document/test composition with global, per-agent, channel, account, persona, and directive overrides.

## Reproduction Assessment

Not applicable as a feature request. The current-main verification path is source and docs inspection: the config/schema/runtime and STT result contracts do not expose a language route that could be exercised.

## Solution Assessment

Unclear for implementation details. The requested direction is reasonable, but the best maintainable path needs maintainer API design before a fix PR because it touches public config, plugin/provider contracts, and language-detection policy.
