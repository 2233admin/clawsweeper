## Summary

Keep open. Current main still lacks the requested language-aware TTS routing contract: there is no public `messages.tts.languageRouting` config/schema surface, inbound auto-TTS only gates whether audio is synthesized, provider/model/voice selection is still preference/config/persona/directive driven, and batch audio transcription/preflight only preserves transcript text plus model metadata.

## What This Changes

Add optional language-aware routing so inbound auto-TTS can choose provider, model, and voice from detected input or reply language while preserving existing fallback behavior.

## Best Possible Solution

Design a backward-compatible TTS language-routing contract, preserve optional detected-language metadata from audio transcription/preflight, route provider/model/voice before synthesis, and document/test fallback behavior across global, per-agent, channel, account, preference, persona, and directive layers.

## Reproduction Assessment

Yes for the current limitation. Source and docs show that `auto: "inbound"` only gates audio synthesis, while no input/reply language is available to provider routing; no live STT/TTS credentials are needed to verify the missing path.

## Solution Assessment

Unclear for the exact proposed JSON shape. The direction is sound, but the maintainable fix should be a generic TTS/STT config and runtime contract rather than a provider-specific workaround or prompt-only directive convention.
