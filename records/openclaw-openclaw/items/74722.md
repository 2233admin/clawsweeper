## Summary

Keep open. Current main still lacks language-aware TTS routing: the public TTS config/schema has no routing block, inbound auto-TTS is only a modality gate, provider choice is still resolved from explicit overrides, prefs, persona, config, or registry order, and audio transcription outputs do not preserve detected-language metadata. The related discussion and title-search matches are useful context but do not supersede this requested routing feature.

## What This Changes

Request language-aware TTS routing so inbound voice replies can select TTS provider, model, and voice from detected input or reply language while preserving existing fallback behavior.

## Best Possible Solution

Design an additive `messages.tts.languageRouting` contract in the speech-core/public config surface, preserve optional detected-language metadata through STT/media understanding, apply a selected route before provider synthesis, and document/test fallback behavior across global, per-agent, channel, and account TTS overrides.

## Reproduction Assessment

Not applicable as a normal bug reproduction because this is a feature request. Current-main code inspection gives a high-confidence verification path: add the proposed config today and the strict TTS schema/runtime has no field or route logic to consume it.

## Solution Assessment

Unclear until maintainers settle the API shape. The requested direction is plausible, but the maintainable fix should be a reviewed additive config and metadata contract rather than relying on prompt instructions, manual provider switching, or model-emitted TTS directives.
