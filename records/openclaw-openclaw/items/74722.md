## Summary

Keep open. Current main still lacks language-aware TTS routing: the TTS config/schema has no route map, inbound auto-TTS only gates whether synthesis runs, provider/model/voice selection is still driven by prefs/persona/config/directives, and audio transcription has no typed detected-language path into TTS routing. The earlier ClawSweeper comment matches this re-check, and the later CausaMem link is unrelated to the requested TTS behavior.

## What This Changes

Add backward-compatible language-aware TTS routing so inbound voice replies can select a provider, model, and voice from detected input or reply language while preserving existing auto-TTS modality and per-agent override behavior.

## Best Possible Solution

Design a backward-compatible `messages.tts.languageRouting` contract, preserve optional detected-language metadata from audio transcription, choose the route before speech synthesis, and document/test fallback behavior across global, per-agent, channel, account, and plugin TTS surfaces.

## Reproduction Assessment

Yes. The current limitation is reproducible by source and contract inspection: `auto: "inbound"` only gates audio synthesis, and no current TTS/STT path carries a language route input into provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The direction is reasonable, but the best fix needs maintainer API review because it crosses public config, STT/media metadata, plugin runtime contracts, and fallback precedence.
