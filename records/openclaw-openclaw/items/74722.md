## Summary

Keep open. Current main still lacks language-aware TTS routing: the shared TTS config/schema has no `languageRouting` route map, inbound auto-TTS only gates whether synthesis runs, provider/model/voice selection is still driven by prefs/persona/config/directives rather than input or reply language, and audio transcription currently returns transcript text without a typed detected-language path. The earlier ClawSweeper comment aligns with this re-check; the later CausaMem link is unrelated to this TTS routing request.

## What This Changes

Add backward-compatible language-aware TTS routing so inbound voice replies can select a provider, model, and voice from detected input or reply language while preserving existing auto-TTS modality and per-agent override behavior.

## Best Possible Solution

Design a backward-compatible `messages.tts.languageRouting` contract, preserve optional detected-language metadata from audio transcription, choose the route before speech synthesis, and document/test fallback behavior across global, per-agent, channel, account, and plugin TTS surfaces.

## Reproduction Assessment

Yes. The current limitation is reproducible by source and contract inspection: `auto: "inbound"` only gates audio synthesis, and no current TTS/STT path carries a language route input into provider/model/voice selection.

## Solution Assessment

Unclear for the exact proposed JSON shape. The direction is sound, but the best fix needs maintainer API review because it crosses public config, STT/media metadata, plugin runtime contracts, and fallback precedence.
