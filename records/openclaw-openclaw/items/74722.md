## Summary

Keep open. Current `main` still lacks language-aware TTS routing, and the request affects public TTS config plus STT/media metadata contracts, so it needs maintainer API/product review rather than conservative cleanup. The provided related reports are delivery/provider-specific TTS issues, not a canonical tracker for language-based route selection.

## What This Changes

The issue requests language-aware TTS provider, model, and voice routing so auto-TTS replies can synthesize audio with a voice/model matching the inbound or reply language.

## Best Possible Solution

Design a backward-compatible `messages.tts.languageRouting` contract, preserve optional detected-language metadata through audio transcription and inbound context, apply route selection before speech synthesis while respecting existing global/agent/channel/account overrides, and document/test fallback behavior when detection or route matching fails.

## Reproduction Assessment

Not applicable as a feature request. Source inspection gives a high-confidence verification path: `languageRouting` is not in the config/schema, `auto: "inbound"` only gates on `inboundAudio`, and STT preflight does not return detected-language metadata.

## Solution Assessment

Unclear because there is no patch yet. The requested direction is plausible, but the exact config shape and metadata contract should be reviewed by maintainers before implementation because it changes public TTS and media-understanding APIs.
