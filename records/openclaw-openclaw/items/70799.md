## Summary

Keep open for maintainer review. The reported BlueBubbles Private API-off failure is plausible, but this PR is not safe to merge as-is because the added guards preserve the cache-null path named as the production failure and mostly duplicate the existing resolved-account inner guard.

## What This Changes

The PR adds early `getCachedBlueBubblesPrivateApiStatus(...) === false` guards to `markBlueBubblesChatRead` and `sendBlueBubblesTyping` in the BlueBubbles chat helper.

## Best Possible Solution

Resolve the effective BlueBubbles account once, make read receipts and typing indicators best-effort when Private API is disabled or not safely known, and add focused tests for `false`, `null`/fresh-boot, and multi-account/default-account cache behavior while coordinating the overlap with #69932.

## Reproduction Assessment

Yes. The PR gives concrete live steps against a BlueBubbles server with Private API off, and the source-level reproduction is to make `getCachedBlueBubblesPrivateApiStatus` return `null` for mark-read or typing; with this patch's false-only guard, the request still proceeds.

## Solution Assessment

No. The proposed outer guard is not the best fix because it preserves the `null` fallthrough described as the failure mode and duplicates the existing resolved-account inner guard; a safer fix should resolve or refresh account capability before these best-effort Private API calls and cover that path in tests.
