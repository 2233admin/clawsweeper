## Summary

Keep open. Current main still has the Telegram getFile null-return path: media resolution can return null, the single-message handler only sends the user-visible failure warning for thrown errors, and the body builder/test coverage still allows a video placeholder when no media was attached. There are also open implementation PRs (#41048 and #42059), so the issue should stay open until one is merged or closed by maintainers.

## Best Possible Solution

Keep this issue open and resolve it through the focused Telegram plugin fix path, preferably by reviewing and landing #41048 or extracting the equivalent minimal behavior from #42059. The fix should make getFile null/failure distinct from 'no media', send a user-visible warning for failed media resolution, avoid misleading placeholder-only bodies when no attachment exists, preserve caption text, and update the Telegram resolver/body tests.

## Risks / Open Questions

- Telegram media download code has SSRF/proxy/size-limit behavior nearby; a fix should stay inside failure signaling/body construction and avoid broadening download trust policy.
