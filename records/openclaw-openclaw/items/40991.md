## Summary

Current main still has the reported Telegram null-media path: `getFile()` failures can return `null`, the single-message warning only runs for thrown media errors, and the inbound body builder can still emit media placeholder text when no attachment was resolved. The issue is also paired with open implementation PRs, so it should remain open until maintainers land, split, or close that fix path.

## What This Changes

Telegram inbound video/media failures from `getFile()` should produce a clear failed-download state instead of a misleading placeholder-only message with no attachment.

## Best Possible Solution

Keep this issue open and resolve it through the focused Telegram plugin fix path, preferably by reviewing and landing #41048 or extracting the equivalent minimal behavior from #42059. The fix should make getFile null/failure distinct from “no media,” send a user-visible warning for failed media resolution, avoid misleading placeholder-only bodies when no attachment exists, preserve caption text, and update the Telegram resolver/body tests.
