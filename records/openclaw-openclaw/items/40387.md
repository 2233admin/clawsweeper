## Summary

Keep this PR open. Current main still has no EasyRunner guide or navigation entry, and hosting/provider deployment docs are an established OpenClaw docs surface. The PR is documentation-only from a supply-chain standpoint, but the proposed EasyRunner content still needs a rebase plus maintainer docs/security review because it normalizes dangerous Control UI auth guidance and does not align with current gateway bind/origin requirements.

## Best Possible Solution

Keep the PR open and request a rebased docs revision. The best path is to place or link EasyRunner under the current Install > Hosting surface, align the Podman/Caddy flow with current Docker/Podman gateway bind and `allowedOrigins` behavior, remove normal-use `dangerouslyDisableDeviceAuth` guidance, preserve the improved secret-handling and image-build notes, and have maintainers review the EasyRunner-specific operational steps before merge.

## Risks / Open Questions

- Publishing the PR as-is would normalize `gateway.controlUi.dangerouslyDisableDeviceAuth=true` for ordinary reverse-proxy use, contrary to current break-glass security guidance.
- The proposed compose/config flow may fail or require unsafe workarounds because current non-loopback Control UI startup requires explicit allowed origins, and current container docs expect deliberate bind/origin setup.
- The PR has low supply-chain risk because it is docs-only, but it still carries operator-security risk if inaccurate deployment guidance is merged.
