## Summary

Keep open. I swept through current main at db40ec404a9178294dba3121758ff59459983a11 and the issue still matches current behavior: multimodal image/audio chunks can be indexed, but hybrid search still applies the generic weighted BM25+vector merge to every candidate, with vector-only hits carrying `textScore: 0`. There is no modality field in the persisted/search result shape for the merge to use, and current tests only prove literal `image`/`audio` label searches rather than natural-language multimodal recall under default hybrid weights.

## Best Possible Solution

Keep this issue open. The likely fix belongs in `extensions/memory-core` plus `packages/memory-host-sdk`: persist or propagate image/audio modality for multimodal chunks, then make hybrid merging avoid BM25 dampening for those vector candidates while preserving weighted BM25+vector scoring for text chunks. Add focused regression tests for natural-language multimodal recall with default hybrid weights and mixed text/multimodal candidates.

## Risks / Open Questions

- Fixing this by path extension alone could be brittle; the better fix should carry reliable modality metadata from indexing into persisted/search result data or use a tightly bounded classifier.
- Changing hybrid scoring could regress text-on-text ranking unless tests preserve existing BM25+vector behavior for normal Markdown chunks.
- Regression coverage should include mixed Markdown plus image/audio candidates and a natural-language visual/audio query under default 0.7/0.3 weights.
