## Summary

Keep open. Current main still rejects `plugins.entries.exa.config.webSearch.baseUrl` and always sends Exa web-search traffic to the fixed Exa endpoint; this issue is also paired with open, unmerged PR #54939, so the issue should wait for that PR to be reviewed, landed, closed, or replaced.

## What This Changes

Request support for `plugins.entries.exa.config.webSearch.baseUrl` in the bundled Exa web-search plugin, or provide explicit docs/schema guidance when that field is unsupported.

## Best Possible Solution

Land a reviewed Exa-plugin change that adds `webSearch.baseUrl` to manifest/UI metadata, resolves and validates it in the Exa runtime, passes the resolved endpoint through the trusted web-search request path, isolates cache keys by endpoint, and updates Exa/web-search docs and tests. If maintainers choose not to support custom Exa endpoints, document that provider capability difference and improve the validation message instead.
