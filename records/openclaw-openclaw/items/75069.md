## Summary

Keep open. Current main has partial mitigations for the bundled runtime mirror regression, but the remaining reported problem is still real: bundled runtime preparation, mirror/fingerprint work, lock waiting, and package-manager staging still run through synchronous or readiness-critical paths that can delay gateway startup or agent execution. Maintainer discussion explicitly kept the issue open after the partial repairs, and the issue thread includes concrete first-agent-run and acpx-only startup-readiness reproductions.

## What This Changes

The issue asks OpenClaw to stop bundled plugin runtime mirror and runtime-deps preparation from synchronously blocking the gateway main thread during pi-agent invocation and related startup readiness paths.

## Best Possible Solution

Finish this as a staged, profile-backed runtime-loading fix: keep the already-landed hardlink and stale-symlink mitigations, then either defer readiness-critical bundled runtime staging or make the large mirror, fingerprint, lock-wait, and install phases async/cooperative, with bounded event-loop lag proven on a packaged install and the acpx-only startup-readiness repro.

## Reproduction Assessment

Yes. The issue body gives a versioned 2026.4.26 packaged-install first-agent-run lag profile, and NikolaFC added a smaller acpx-only startup-readiness reproduction with cross-version timing; current-main source inspection confirms the remaining synchronous/readiness-critical paths, though post-mitigation timing still needs rerun proof.

## Solution Assessment

Unclear. The suggested async conversion would address event-loop starvation, but the narrowest maintainable fix may be staged readiness deferral/caching plus targeted async conversion of the proven hot phases instead of converting the entire runtime preparation stack in one change.
