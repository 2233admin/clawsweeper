## Summary

Keep open. Current main partially mitigates the shipped 2026.4.26 behavior with a process-level prepared-dist mirror cache and active-provider short-circuiting, but the reported event-loop blocking path is still real enough in source: bundled runtime preparation is still synchronous, the mirror/fingerprint code still uses synchronous filesystem calls, the lock can still block with Atomics.wait, and the hardlink freshness check still uses realpath equality rather than inode identity.

## What This Changes

The issue reports that bundled plugin runtime mirror preparation runs synchronously during pi-agent/tool registry resolution, repeatedly walking bundled dist roots and blocking the Gateway event loop for tens of seconds in 2026.4.26+.

## Best Possible Solution

Keep the issue open and fix the runtime-mirror hot path in staged, profile-backed patches: avoid unnecessary media capability registry loads during tool construction, fix hardlink reuse with inode identity, preserve or strengthen per-process dist-root memoization with freshness invalidation, and move any remaining large mirror/fingerprint work off the synchronous agent invocation path or make it cooperative with event-loop yields.

## Reproduction Assessment

Yes for the released regression: the report includes version, install method, lag output, CPU stack attribution, and a direct bench plan. For current main, reproduction is partial because source inspection confirms remaining synchronous hot-path work, but no live Gateway or bench profile was run in this read-only pass.

## Solution Assessment

Unclear. The suggested full async migration would address event-loop starvation, but it crosses the synchronous plugin-loader contract; a safer first repair is to remove repeated registry/mirror work and fix the hardlink freshness check, then profile before expanding to an async loader/mirror migration.
