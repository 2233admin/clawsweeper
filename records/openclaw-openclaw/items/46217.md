## Summary

Keep open. Current main has zh-CN Control UI support and translated top-level Settings navigation, but the Settings/configuration surface still renders many English literals plus schema-derived English titles and descriptions, so the requested Chinese Settings localization is still unresolved.

## What This Changes

The issue requests Chinese localization for the Control UI Settings section because configuration items still appear in English.

## Best Possible Solution

Keep this open and define a scoped Settings i18n pass: move visible Settings and Quick Settings strings into the English locale source, regenerate non-English bundles through the existing Control UI i18n pipeline, add coverage that prevents new hardcoded Settings copy, and decide how schema title/description metadata should be localized for core and plugin-provided config fields.

## Reproduction Assessment

Yes. Set the Control UI locale to zh-CN and open Settings/Advanced Settings; current main statically renders English literals from the Settings view files and English schema metadata from the generated config schema.

## Solution Assessment

Unclear. The existing Control UI i18n pipeline is the right path for static UI strings, but full configuration-item localization needs a maintainer decision on whether schema/uiHint metadata is translated client-side, served locale-aware, or handled through a staged contract.
