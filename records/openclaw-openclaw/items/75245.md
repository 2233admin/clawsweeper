## Summary

Keep open. Current main still shallow-clones plugin hook context in `api.registerHook()`, so plugin-registered `agent:bootstrap` handlers mutate a copy while the bootstrap path later reads the original event context. The issue describes a real core/plugin SDK contract bug with clear reproduction and no superseding fix.

## What This Changes

The issue requests that mutable `event.context` changes made by plugin-registered hooks, especially `agent:bootstrap` updates to `context.bootstrapFiles`, survive the `registerHook()` wrapper.

## Best Possible Solution

Keep the mutable internal hook contract intact for plugin-registered hooks by exposing or merging back the original context object while temporarily adding and restoring `pluginConfig` per handler, then add a regression test that exercises `api.registerHook('agent:bootstrap')` through bootstrap context resolution.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: load a plugin that registers `api.registerHook('agent:bootstrap')`, assign a sentinel array to `event.context.bootstrapFiles`, and observe that `resolveBootstrapContextForRun()` still consumes the original files because the wrapper mutated only a clone.

## Solution Assessment

Yes. The narrowest maintainable fix is in the plugin registry wrapper so context mutations survive while `pluginConfig` remains per-handler and non-leaking; switching to an immutable return-value hook API would be a larger compatibility change.
