## Summary

Keep open. Current main still shallow-clones plugin hook event context inside `api.registerHook()`, while the agent bootstrap code and docs explicitly rely on `event.context.bootstrapFiles` mutation being visible after hooks run. The report describes a real core/plugin SDK contract bug, not an external plugin-only problem.

## What This Changes

The issue asks OpenClaw to preserve mutable `event.context` changes made by plugin-registered `agent:bootstrap` hooks so adjusted bootstrap files reach prompt/context assembly.

## Best Possible Solution

Keep the mutable internal hook contract intact for plugin-registered hooks by exposing the original context object while temporarily adding and restoring `pluginConfig` per handler, then lock it with a regression test that exercises `api.registerHook('agent:bootstrap')` through bootstrap context resolution.

## Reproduction Assessment

Yes. The high-confidence reproduction path is source-level: a plugin handler registered via `api.registerHook('agent:bootstrap')` mutates a cloned context in `src/plugins/registry.ts`, while `src/agents/bootstrap-hooks.ts` later reads the original event context.

## Solution Assessment

Yes. The narrowest maintainable fix is to adjust the wrapper so context mutations survive while `pluginConfig` remains per-handler and non-leaking; replacing the hook contract with an immutable return-value API would be a larger compatibility change.

## Repair Work Prompt

Fix the mutable hook context regression described in https://github.com/openclaw/openclaw/issues/75245. Current `api.registerHook()` in `src/plugins/registry.ts` shallow-clones `evt.context` to inject `pluginConfig`, which drops `event.context.bootstrapFiles = ...` mutations for `agent:bootstrap`. Preserve the existing mutable internal hook contract while keeping `pluginConfig` per-handler and non-leaking. Prefer temporarily adding `pluginConfig` to the original context, invoking the handler with that same context, and restoring/deleting the previous `pluginConfig` in `finally`; avoid changing hook ordering, global hook registration, or making bootstrap behavior extension-specific. Add regression coverage through the plugin loader/API path: load a test plugin that calls `api.registerHook('agent:bootstrap', ...)`, replaces `bootstrapFiles` with a sentinel file, run bootstrap context resolution, and assert both returned `bootstrapFiles` and `contextFiles` use the sentinel. Keep or extend the existing assertion that `pluginConfig` is visible to the handler but does not permanently leak onto the shared event context. Add a user-facing `CHANGELOG.md` fix entry with valid reporter attribution. Do not introduce a new immutable return-value hook API in this repair.
