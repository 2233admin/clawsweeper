## Summary

Current main still has the reported Zalo proactive-send mismatch. The previous ClawSweeper comment remains accurate: Zalo normalizes optional prefixes but still routes explicit target detection through a numeric-only predicate, so hex-like Bot Platform chat_id strings can fall through to directory lookup and return the reported unknown-target error before the Zalo send helper runs.

## Best Possible Solution

Keep this issue open and fix it in the Zalo plugin: replace the numeric-only `targetResolver.looksLikeId` behavior with a Zalo-specific predicate that accepts real Bot Platform chat_id strings after optional `zalo:`/`zl:` normalization, add focused regression coverage for hex-like chat IDs, and update the Zalo delivery-target docs so they no longer imply numeric-only IDs.
