## Summary

Current main still has the reported Zalo proactive-send mismatch. Zalo strips optional `zalo:`/`zl:` prefixes, but its target resolver still delegates ID detection to the shared numeric-only predicate, so hex-like Zalo Bot Platform `chat_id` values can fall through directory lookup and return `Unknown target ... Hint: <chatId>` before the Zalo send action runs. The existing ClawSweeper comment remains accurate on current main; no related provided item supersedes this Zalo-specific bug.

## Best Possible Solution

Keep this issue open. Update the Zalo plugin's `targetResolver.looksLikeId` to accept real Zalo Bot Platform `chat_id` strings, including hex-like non-numeric IDs after optional `zalo:`/`zl:` prefix normalization. Add a focused regression test through `resolveChannelTarget` or the message action runner using a value like `3becaa50ae12474c1e03`, and update the Zalo CLI/docs target wording to describe Zalo chat IDs without implying numeric-only IDs.
