## Summary

Current main still uses the reported sandbox-browser noVNC stack: Debian installs noVNC, websockify, x11vnc, and Xvfb; the entrypoint starts Xvfb and x11vnc for the noVNC observer path; no TigerVNC/Xtigervnc or other UTF-8 clipboard workaround is present. The latest release tag v2026.4.26 has the same implementation, and the related TigerVNC PR #49618 is closed unmerged, so this remains an actionable sandbox-browser infrastructure bug.

## Best Possible Solution

Keep this issue open. The best fix is a focused sandbox-browser image and entrypoint change that replaces or augments x11vnc with a VNC server that negotiates Extended Clipboard/UTF-8, preserves OpenClaw's noVNC password/token and loopback security properties, bumps SANDBOX_BROWSER_SECURITY_HASH_EPOCH so old containers are recreated, and verifies CJK/Cyrillic clipboard round-trip through noVNC in a Docker smoke path.
