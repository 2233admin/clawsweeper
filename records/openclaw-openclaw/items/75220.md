## Summary

Keep open. Current main fixed the fresh service PATH generator so missing hard-coded fallback directories are no longer emitted, but `gateway install --force` still merges the existing service PATH back into the regenerated environment. That means stale version-manager or package-manager entries can survive a forced reinstall and then be flagged by the same audit path, which matches the reported repair-loop behavior.

## What This Changes

The issue requests that Linux `openclaw gateway install --force` and doctor/status agree on the generated gateway service PATH instead of recreating entries that are later warned as non-minimal.

## Best Possible Solution

Make gateway service repair converge on the same PATH policy that doctor/status audits: preserve safe custom existing PATH entries only when they will not be reported as non-minimal, and add a regression test where a forced reinstall starts from an old unit PATH containing stale version-manager/package-manager entries.

## Reproduction Assessment

Yes. A high-confidence hermetic reproduction is to call `buildGatewayInstallPlan` with an `existingEnvironment.PATH` containing stale `.volta`, `.asdf`, `.nvm/current`, `.fnm/current`, or pnpm entries that are absent from the current minimal set, then pass the resulting environment to `auditGatewayServiceConfig` and observe `gateway-path-nonminimal`.

## Solution Assessment

No, current main is not yet the best solution. The fresh-generation fix is good, but the repair path should also filter or classify preserved existing PATH segments with the same minimal/audit contract instead of reintroducing entries that doctor will warn about.
