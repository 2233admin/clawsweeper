## Summary

Keep open. Current main still has the repair-loop path: forced reinstall and doctor repair pass the existing service environment into install planning, and the PATH merge appends preserved existing PATH entries without applying the same version-manager/package-manager filter that the audit uses.

## What This Changes

The issue reports that Linux `openclaw gateway install --force`/doctor repair can write PATH entries that `openclaw doctor` and gateway status immediately flag as non-minimal.

## Best Possible Solution

Make reinstall/repair PATH planning converge with the audit contract: preserve only existing-service PATH segments that will not be reported as non-minimal, share or mirror the audit predicate intentionally, and add a regression test for an old unit PATH containing stale version-manager/package-manager entries.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildGatewayInstallPlan` with an `existingEnvironment.PATH` containing stale `.volta`, `.asdf`, `.nvm/current`, `.fnm/current`, or pnpm entries that are absent from the generated minimal PATH, then audit the resulting environment and observe `gateway-path-nonminimal`.

## Solution Assessment

No. Current main is only a partial solution because fresh PATH generation omits missing hard-coded fallbacks, while forced reinstall and doctor repair can still preserve the same stale entries from the installed service.
