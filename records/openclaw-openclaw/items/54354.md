## Summary

Keep open. Current main still does not implement `engines.openclaw`-aware npm version selection for plugin install/update, and this PR is an XL public plugin-compatibility and npm-resolution change with concrete blocking findings around tag handling, malformed ranges, and registry metadata parsing.

## What This Changes

The PR adds `engines.openclaw`-aware npm version resolution for plugin install/update, custom semver and registry helpers, an `--ignore-engine` CLI flag, load-time compatibility warnings, and related tests.

## Best Possible Solution

Decide the public plugin compatibility contract first. If `engines.openclaw` is accepted, rework the branch so npm tags and prereleases preserve user intent, registry metadata is parsed defensively, invalid ranges fail closed, non-release host versions avoid noisy warnings, and install/update/load behavior is covered by focused tests; otherwise keep the shipped `openclaw.install.minHostVersion` and ClawHub compatibility model as the supported path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Honor explicit dist-tags during engine resolution:** `src/plugins/install.ts:743-747`
  - body: For specs like `pkg@beta`, `parseRegistryNpmSpec` marks the selector as a tag and `selectorIsPrerelease` stays false. This resolver then scans all package versions, excludes prereleases, and rewrites the request to an exact stable version, so a user asking for the beta channel can silently get a different channel. Resolve the requested tag first or constrain candidates to that tag's version semantics.
  - confidence: 0.9
- **[P1] Reject malformed engine ranges:** `src/infra/semver-range.ts:237-238`
  - body: When a non-empty branch has no tokenized comparators, `allComparators.every(...)` returns true. Unsupported ranges such as `>=2026.3` or `foo` therefore match every host version, bypassing the install/update gate and suppressing load warnings for bad metadata.
  - confidence: 0.92
- **[P2] Validate registry engine values before resolving:** `src/infra/npm-registry-versions.ts:82-87`
  - body: The registry `engines` object is cast to `Record<string, string>` without checking `engines.openclaw`. A package that publishes a non-string value can pass a number into `satisfiesRange`, which calls `range.trim()` and throws during install/update instead of treating the metadata defensively.
  - confidence: 0.88
- **[P2] Preserve prerelease lower bounds for caret and tilde ranges:** `src/infra/semver-range.ts:158-175`
  - body: Caret and tilde parsing replaces the lower-bound prerelease with `null`, so `^2026.3.14-beta.1` becomes `>=2026.3.14` instead of including the beta lower bound. That diverges from npm semver behavior and can reject or select the wrong versions for prerelease engine constraints.
  - confidence: 0.82
- **[P3] Suppress load warnings for unparsable host versions:** `src/plugins/loader.ts:987-995`
  - body: The loader only checks that `VERSION` is truthy before calling `satisfiesRange`. Local or development builds with a non-semver version string will fail every constrained range and warn for all constrained plugins even when compatibility is unknown rather than incompatible.
  - confidence: 0.8

## Security Review

Status: needs_attention

Summary: The patch introduces concrete supply-chain resolution concerns by adding npm registry metadata reads and client-side version selection that can currently select or accept unintended versions.

Concerns:

- **[medium] Explicit npm dist-tags can be overridden:** `src/plugins/install.ts:743`
  - body: The resolver treats tag selectors like `@beta` as a broad package scan and can rewrite them to a stable exact version, which changes the user-requested release channel during plugin installation or update.
  - confidence: 0.9
- **[medium] Malformed compatibility ranges fail open:** `src/infra/semver-range.ts:237`
  - body: Unsupported `engines.openclaw` ranges can evaluate as compatible, bypassing the new install/update gate and hiding load warnings for bad package metadata.
  - confidence: 0.92
- **[low] External registry metadata is not parsed defensively:** `src/infra/npm-registry-versions.ts:82`
  - body: The npm registry `engines.openclaw` value is trusted as a string, so malformed metadata can throw during install/update instead of producing a controlled compatibility result.
  - confidence: 0.88
