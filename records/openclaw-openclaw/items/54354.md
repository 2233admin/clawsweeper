## Summary

Keep this PR open. Current main has overlapping host compatibility enforcement through `openclaw.install.minHostVersion` and ClawHub `pluginApi`/`minGatewayVersion`, but it does not implement this PR's intended npm install/update behavior: selecting the latest compatible npm version from per-version engine metadata. The proposed branch also has unresolved review/security concerns around npm tag semantics, malformed ranges, and untrusted registry metadata, so it needs maintainer review or rework rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best direction is either to rework the feature around the existing `openclaw.install.minHostVersion` / `openclaw.compat` / ClawHub compatibility model, or explicitly decide that npm `engines.openclaw` should become a supported public metadata surface. Any follow-up should preserve npm dist-tag semantics, reject malformed compatibility metadata, guard all registry-derived values, and add focused install/update tests for stable, beta, exact-version, malformed, and non-string metadata cases.
