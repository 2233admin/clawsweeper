## Summary

Keep this PR open. Current main still does not implement the PR’s intended WhatsApp per-agent ack emoji fallback: the WhatsApp ack path reads only `channels.whatsapp.ackReaction.emoji`, so a valid object config without `emoji` resolves to an empty string and suppresses the reaction. The PR discussion is relevant and positive: the contributor addressed earlier review feedback by moving toward a WhatsApp-specific resolver and account-level handling, but the work is still unmerged and needs maintainer review around the SDK export and typed tests.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is to land a WhatsApp-specific ack emoji resolver that preserves no reaction when no WhatsApp ack config exists, skips legacy `messages.ackReaction`, honors account-level precedence, falls back to the responding agent identity only when WhatsApp ack config is intentionally present without an emoji, preserves empty-emoji disable semantics, and covers those cases with typed tests. Maintainers should also decide whether the resolver belongs in the public plugin SDK barrel or stays WhatsApp-local.

## Risks / Open Questions

- The PR proposes exporting resolver helpers through `openclaw/plugin-sdk/channel-feedback`; maintainers should confirm that is the right public SDK surface, or keep the WhatsApp-specific resolver local to the plugin if it is not generic enough.
- The PR’s intended behavior changes partial WhatsApp ack configs without `emoji` from silent to agent-identity ack reactions. That matches the stated goal, but should be accepted deliberately and documented/released as a behavior change.
- The review thread notes `as any` / `typescript/no-explicit-any` suppression in new identity tests; those fixtures should be typed before merge so the object-shaped config contract is actually protected.
