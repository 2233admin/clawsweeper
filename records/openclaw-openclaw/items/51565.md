## Summary

Keep open. Current main still does not implement this PR's intended WhatsApp per-agent ack emoji fallback: WhatsApp ack reactions still read only the configured WhatsApp object emoji, and the generic ack resolver still assumes account/channel ack reactions are strings. The PR remains a focused implementation candidate, but it still needs maintainer review for the public SDK helper/export choice and the outstanding typed-test cleanup noted in review.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is a cleaned implementation that preserves WhatsApp's no-global-ack behavior, adds per-agent identity fallback only when WhatsApp ack config is intentionally present, honors account-level precedence, covers object-without-emoji and empty-emoji edge cases with typed tests, and treats any plugin-SDK export as an intentional public contract.

## Risks / Open Questions

- The current product gap remains for WhatsApp configs that intentionally set `ackReaction` without `emoji`: current main sends no ack reaction instead of using the responding agent identity emoji.
- The proposed `resolveWhatsAppAckEmoji` export would widen the public plugin SDK surface, so maintainers should decide whether that helper belongs in the SDK or should stay WhatsApp-local/internal.
- The PR tests should avoid `as any` / `typescript/no-explicit-any` suppression so the object-shaped config contract remains protected by types.
