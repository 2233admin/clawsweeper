## Summary

Keep this PR open. Current main still does not implement eligibility-based skills snapshot invalidation: the active cache checks only compare watcher/version state and normalized skill filters, while eligibility itself depends on runtime env, binary, remote, and config checks. The PR is not safely closeable as implemented-on-main, and the discussion shows unresolved functional review gaps that should be addressed or superseded by a stronger canonical fix.

## Best Possible Solution

Keep the PR open or supersede it with a canonical fix. The best implementation would centralize a deterministic eligibility/cache signature shared by reply sessions, `openclaw agent`, and isolated cron snapshots; include the actual eligibility inputs used by `shouldIncludeSkill` (`requires.env`, `skills.entries.*.env/apiKey`, relevant `requires.config` paths/defaults, binary availability or a deliberate invalidation strategy for it, remote eligibility, agent skill filters, and snapshot version); persist that metadata on `SessionSkillSnapshot`; and add regression tests for existing sessions where eligibility changes without a `SKILL.md` watcher bump.
