## Summary

Current main still does not implement eligibility-based skillsSnapshot invalidation, and this PR is not yet complete enough to close as obsolete or implemented. The remaining work spans multiple snapshot cache paths and must reflect the same runtime inputs used by skill eligibility checks.

## Best Possible Solution

Keep this PR open for maintainer follow-up or supersede it with a canonical fix. The best implementation is a shared deterministic eligibility/cache signature used by reply sessions, openclaw agent, and isolated cron; it should persist metadata on SessionSkillSnapshot, reflect the same inputs evaluated by shouldIncludeSkill/evaluateRuntimeEligibility, and add regression tests for existing snapshots where env, bin, config, or remote eligibility changes without a SKILL.md watcher bump.

## Risks / Open Questions

- Merging or closing this as handled would leave existing stale-snapshot behavior in current main for eligibility changes that do not bump the SKILL.md watcher or change the skill filter.
- The submitted signature approach, as shown in the provided diff, can miss actual runtime transitions such as binaries installed or removed in an unchanged PATH, env/API-key availability changes, and non-skills requires.config changes.
- The submitted patch only covers reply-session snapshots, while openclaw agent and isolated cron snapshots still reuse cached skills using only missing/version/filter checks.
