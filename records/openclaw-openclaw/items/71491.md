## Summary

Keep open. Current main still lacks a Moonshot/Kimi-specific outgoing payload repair for replayed assistant tool-call messages that are missing top-level `reasoning_content`, and the related fix PR remains open rather than merged. This issue is concrete, current, and distinct from the already-merged native Kimi tool-call ID fix in #70030. https://github.com/openclaw/openclaw/issues/71491

## What This Changes

The issue asks OpenClaw to preserve or safely restore Kimi K2.6 `reasoning_content` on replayed assistant tool-call messages after long-context compaction and cross-model fallback.

## Best Possible Solution

Land a narrow Moonshot-only outgoing payload repair that preserves existing `reasoning_content`, adds a placeholder only when Moonshot thinking is enabled for replayed assistant tool-call messages, and leaves generic OpenAI-compatible replay behavior unchanged.

## Reproduction Assessment

Yes. The issue body and comments give concrete long-session and fallback reproductions, and current source inspection shows the Moonshot wrapper still leaves a thinking-enabled payload’s replayed assistant tool-call messages unchanged when `reasoning_content` is absent.

## Solution Assessment

Yes. A Moonshot stream-wrapper repair is the narrowest maintainable fix because it runs after replay/context assembly, mirrors the existing DeepSeek provider-specific compatibility pattern, and avoids changing generic OpenAI-compatible replay behavior.
