## Summary

Current main still does not support trusted-proxy in non-interactive onboarding. The CLI advertises `--gateway-auth` as token/password, the onboarding option type only allows token/password, and the non-interactive gateway config path rejects any other value. Runtime/config docs and interactive gateway configuration already support `trusted-proxy`, so this is a real core onboarding gap rather than ClawHub/plugin work. The provided related trusted-proxy items cover fallback/internal-caller behavior, not this CLI automation surface.

## Best Possible Solution

Keep this open and implement it in core onboarding: accept `--gateway-auth trusted-proxy`, add explicit non-interactive flags for `gateway.trustedProxies` and `gateway.auth.trustedProxy.*`, reuse the interactive validation model, update CLI docs, and add focused tests for successful config writes plus invalid/missing required fields. The health-check path also needs a deliberate trusted-proxy story, likely requiring `--skip-health` or a proxy-aware probe instead of direct token/password probing.

## Risks / Open Questions

- Closing this would leave scripted reverse-proxy deployments without a supported onboarding path for an auth mode that core already supports at runtime.
- A fix needs careful validation because these flags configure the gateway trust boundary: proxy IPs, user identity header, required headers, allowUsers, and loopback opt-in should be explicit and covered by tests.
