## Summary

Keep open. Current main still rejects trusted-proxy in non-interactive onboarding even though trusted-proxy is a supported gateway runtime mode and the interactive gateway wizard can configure it; the same-author draft PR #49107 remains the active implementation path, while #73639 appears to be a newer duplicate satellite.

## What This Changes

The issue requests non-interactive `openclaw onboard` support for trusted-proxy gateway auth so scripted reverse-proxy deployments can configure the core gateway auth mode without manual config edits.

## Best Possible Solution

Extend core non-interactive onboarding to accept trusted-proxy auth with explicit flags for trusted proxy IPs, identity header, required headers, allowUsers, allowLoopback, and Control UI allowed origins; update focused tests and public docs, and land or replace #49107 with secops review while keeping runtime fail-closed behavior intact.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: pass `gatewayAuth: "trusted-proxy"` into `applyNonInteractiveGatewayConfig`, and current main deterministically exits at the token/password-only guard.

## Solution Assessment

Yes for the requested direction. Trusted-proxy is already a core gateway auth mode and interactive onboarding already supports it, so the narrow maintainable fix belongs in core onboarding rather than a plugin or a manual-config-only doc path.
