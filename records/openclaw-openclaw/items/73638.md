## Summary

Keep open. Current main still rejects `--gateway-auth trusted-proxy` in non-interactive onboarding even though trusted-proxy is a supported core gateway runtime mode and interactive gateway configuration can write it. The same-author draft PR #49107 is the active implementation candidate, while #73639 appears to be the newer duplicate rather than the canonical tracker.

## What This Changes

The issue requests non-interactive `openclaw onboard` support for selecting and configuring `gateway.auth.mode: "trusted-proxy"` in scripted reverse-proxy deployments.

## Best Possible Solution

Implement this in core onboarding by landing or replacing #49107: accept `--gateway-auth trusted-proxy`, add explicit non-interactive flags for trusted proxy IPs, identity header, required headers, allowUsers, allowLoopback, and Control UI allowed origins, clear incompatible token/password state when switching modes, and update focused tests plus public docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: pass `--gateway-auth trusted-proxy` through non-interactive onboarding and `applyNonInteractiveGatewayConfig` deterministically exits because it accepts only `token` or `password`.

## Solution Assessment

Yes for the issue direction. Trusted-proxy auth is already a core gateway mode and interactive onboarding already configures it, so core non-interactive onboarding is the narrow maintainable surface; the exact #49107 implementation still needs maintainer/security review.
