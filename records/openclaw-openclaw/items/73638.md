## Summary

Keep open. Current main still rejects `trusted-proxy` in non-interactive onboarding while the runtime, config schema, docs, and interactive gateway flow already support trusted-proxy auth; the same-author draft PR #49107 remains the active implementation candidate and needs maintainer/security review before this issue can close.

## What This Changes

Request non-interactive `openclaw onboard` support for configuring gateway `trusted-proxy` authentication in scripted reverse-proxy deployments.

## Best Possible Solution

Review, land, or replace #49107 with a narrow current-main onboarding change that accepts `--gateway-auth trusted-proxy`, adds typed flags for trusted-proxy fields, clears incompatible shared-token state, keeps loopback and Control UI origins explicit, updates docs, and adds focused tests for valid and unsafe inputs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `openclaw onboard --non-interactive --gateway-auth trusted-proxy --accept-risk`; current main reaches `applyNonInteractiveGatewayConfig` and exits because only `token` and `password` are accepted.

## Solution Assessment

Yes for the requested direction. Core non-interactive onboarding is the right layer because trusted-proxy is already a core gateway auth mode and interactive onboarding can configure it; raw config editing or a plugin would not address the CLI automation gap safely.
