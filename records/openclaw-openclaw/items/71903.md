## Summary

Keep this PR open. Current main still has the self-PID/no-starttime orphan-lock behavior the PR is trying to change, and active tests still assert that a same-PID legacy lock without `starttime` is reclaimed. The PR is a focused core session-write-lock fix tied to concrete lost-response/session-history reports, but it needs maintainer review and better regression coverage rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to refine the patch, replace the skipped test with explicit regression coverage for not reclaiming a live self-PID/no-starttime lock, preserve tests for dead PID and Linux recycled-PID recovery, and decide whether startup stale-lock cleanup needs a matching guard before landing.

## Risks / Open Questions

- The PR currently skips the legacy same-PID/no-starttime reclamation test instead of replacing it with direct regression coverage for a live self-PID legacy lock plus preserved dead-PID/recycled-PID cleanup behavior.
- Changing no-starttime same-PID reclamation trades off protection against active-parent lock removal against the previously shipped recovery path for legacy stale locks, especially on non-Linux or containerized environments where starttime may be unavailable.
- The PR does not change startup `cleanStaleLockFiles`, which still has separate age-based stale cleanup behavior; maintainers should confirm whether the linked reports require only contended-acquire behavior or also startup cleanup changes.
