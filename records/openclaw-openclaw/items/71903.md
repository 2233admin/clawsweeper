## Summary

Keep this PR open. Current `main` still implements the self-PID/no-starttime reclamation behavior the PR is trying to change, and the active tests still assert that legacy same-PID locks without `starttime` are reclaimed. The PR is a focused core session-lock fix tied to concrete lost-response/session-history reports, but it should be refined rather than closed: it skips the existing behavior test instead of replacing it with direct regression coverage, and the change needs maintainer review because it trades off a previously shipped PID-reuse recovery path.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to refine the patch, replace the skipped test with explicit regression coverage for a live self-PID legacy lock not being reclaimed, preserve stale/recycled/dead-PID cleanup coverage, and then land it if that behavior matches the linked lost-response reports.

## Risks / Open Questions

- The PR currently skips the legacy same-PID/no-starttime reclamation test instead of replacing it with a direct regression test for the live self-PID/no-starttime scenario and separate proof that stale or recycled locks remain reclaimable.
- Because current behavior was shipped to avoid false lock timeouts after PID reuse, maintainers should verify the conservative PID-alive path does not reintroduce stale-lock hangs in container, Windows, or non-Linux legacy-lock cases.
- The proposed implementation should keep comments and behavior aligned around the no-verifiable-starttime boundary; the provided diff treats some valid-starttime cases differently when current starttime cannot be read.
