## Summary

Keep open. Current main still does not implement timestamp-aware WhatsApp batching: the WhatsApp monitor converts and preserves Baileys `messageTimestamp`, but inbound batching is still driven by the shared receive-time debounce timer keyed only by account, conversation, and sender. The config and docs still expose only the existing receive-time `debounceMs` behavior, so delayed same-server-timestamp Baileys deliveries can still split into separate turns when they arrive after the receive-time window.

## Best Possible Solution

Keep this issue open. The best path is a WhatsApp-owned timestamp coalescing implementation, or an additive shared inbound debouncer seam if maintainers want reuse, that groups by account/conversation/sender plus same-or-near WhatsApp `messageTimestamp`, tolerates delayed Baileys delivery beyond the receive-time debounce window, adds fake-timer regression coverage for delayed same-timestamp deliveries, and documents any new config surface.
