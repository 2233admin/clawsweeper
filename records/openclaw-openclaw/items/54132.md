## Summary

Keep open. Current main still lacks the requested Sessions-to-history shortcut, but this PR is not ready to merge because the proposed viewer serves transcript data outside the established authenticated, session-store-aware history path.

## What This Changes

The PR adds a Sessions page History link and subtitle, standalone Control UI `/history` and `/history.js` transcript viewer routes with tests, and a `.vs/` gitignore entry.

## Best Possible Solution

Land a reworked history navigation design that keeps the Sessions shortcut basePath-safe, uses the existing authenticated `/sessions/<key>/history` API or an equally scoped endpoint, respects configured agent/session-store behavior, handles archived and rotating transcript artifacts, and has focused gateway/UI tests for auth, basePath, and transcript-loading boundaries.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.93

Full review comments:

- **[P1] Gate history transcripts behind gateway auth:** `src/gateway/control-ui.ts:551-552`
  - body: `/history.js` is served from the Control UI handler before the existing bootstrap auth and root availability checks, and it serializes local transcript files into JavaScript. Any client that can reach the static Control UI surface can read session history without the `operator.read` gate used by `chat.history`, so transcript access needs to move behind scoped gateway auth or reuse the existing history endpoint.
  - confidence: 0.95
- **[P2] Resolve history files from configured stores:** `src/gateway/control-ui.ts:121-122`
  - body: `loadSessionHistoryFiles` uses `resolveSessionTranscriptsDirForAgent(agentId)`, which ignores `session.store` and the store lookup semantics used by `sessions.list` and `/sessions/<key>/history`. Deployments with a custom session store can show empty or wrong history even though the supported session APIs resolve the correct store from config.
  - confidence: 0.9
- **[P2] Size-check transcripts before reading them:** `src/gateway/control-ui.ts:137-138`
  - body: The byte cap is applied after `fs.readFileSync(filePath)`, so a single oversized transcript is still read into memory synchronously before the cap can exclude it. That defeats the intended 50 MB protection and can block the event loop or spike memory on large logs.
  - confidence: 0.88
- **[P2] Include reset and deleted transcript archives:** `src/gateway/control-ui.ts:129`
  - body: Filtering with `file.endsWith(".jsonl")` drops archive names such as `*.jsonl.reset.<timestamp>` and `*.jsonl.deleted.<timestamp>`, which current session artifact helpers treat as usage-counted transcript history. The new viewer can therefore miss archived logs that users expect this page to show.
  - confidence: 0.86
