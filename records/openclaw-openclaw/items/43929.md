## Summary

Keep this issue open. Current main still exposes `chat.history` as current-segment history only: the protocol has no logical/current scope, the handler reads one active `sessionId`, and reset rollover archives prior transcript files without linking them into a logical history response. Related archive-access work is active, but it has not landed and does not yet supersede this specific WebChat/Gateway request.

## What This Changes

Add an optional logical-history mode to `chat.history` so stable session keys can return current and previous rolled transcript segments with lightweight segment metadata.

## Best Possible Solution

Implement an additive Gateway protocol change that preserves current `chat.history` behavior by default and offers an explicit logical/archive-aware mode returning ordered segment metadata plus bounded display-normalized messages. Coordinate the naming and archive-chain semantics with #45003 and #56131, then update WebChat/Control UI and docs for current, missing, and pruned archive cases.

## Reproduction Assessment

Yes. The source-level path is clear: a stable session key rolls to a new `sessionId`, tests show the old transcript is archived as `.reset.*`, and current `chat.history` reads only the active entry's transcript. I did not run runtime tests because this review was read-only.

## Solution Assessment

Unclear. The requested direction is reasonable, but the exact API should be decided with the open archive-access work; #73883's `includeArchived` is a useful candidate, while this issue's segment metadata and logical-scope semantics still need explicit maintainer review.

- Reset archives can be pruned by `session.maintenance.resetArchiveRetention`, so the product behavior for missing or partial historical segments needs to be explicit.
