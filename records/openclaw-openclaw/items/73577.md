## Summary

Keep this issue open. Current main still restores last-known-good config for invalid reload/startup snapshots without surfacing the original validation path/message in the successful recovery log or recovery notice; the unsupported provider shape explains why the sample can fail, but it does not resolve the operator-visible diagnostic gap.

## Best Possible Solution

Keep this issue open. The best fix is to preserve and surface the original invalid `snapshot.issues` during successful last-known-good recovery in both reload and startup paths, and pass a concise path/message into the recovery log and main-agent notice. Add regression tests that an invalid direct edit recovered from last-known-good still logs the rejected field/schema reason. Separately coordinate with #39766 or a narrower config-schema issue if maintainers want `models.providers.<id>` credential/baseUrl-only entries to validate without `models`.
