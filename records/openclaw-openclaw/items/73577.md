## Summary

Keep open. Current main still restores invalid config from last-known-good with only generic recovery messages, so the operator-visible validation path/message requested here remains missing; the related provider-schema PR is adjacent but does not resolve the diagnostics gap.

The issue asks OpenClaw to surface the validator field/path/rule when an edited config is rejected and restored from last-known-good after a `providers.<id>` change.

## Best Possible Solution

Carry the original invalid `snapshot.issues` through successful last-known-good recovery and surface a concise sanitized path/message in reload logs, startup logs, and the main-agent recovery notice, while preserving the existing recovery behavior and leaving provider schema expansion to its narrower tracker.
