## Summary

Keep open. Current main still restores invalid config from last-known-good with generic recovery messages, while the original validation path/message is only surfaced when recovery does not happen; the related provider-schema PR is adjacent and does not fix the operator-visible diagnostics gap.

## What This Changes

The issue asks OpenClaw to show the specific config validation path and schema reason when an invalid provider config edit is restored from last-known-good instead of only logging generic recovery output.

## Best Possible Solution

Carry the invalid snapshot's formatted validation issues through successful last-known-good recovery and include a concise, redacted path/message summary in reload logs, startup logs, and the main-agent recovery notice, with focused regression tests for both recovery paths. Leave provider schema optionality to #39766 or a separate schema-focused change.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: the current reload and startup recovery tests construct invalid snapshots, then prove successful recovery emits only generic restore output instead of the original validation issue path/message.

## Solution Assessment

Yes. The narrow maintainable solution is to preserve existing last-known-good recovery behavior while propagating the original validation issue summary into operator-visible recovery messages; changing provider schema alone would not solve this diagnostics bug.
