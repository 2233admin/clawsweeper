## Summary

Keep this PR open: the retry-disable direction is plausible, but the current patch drops prepared compaction overrides when it rebuilds the settings manager, and the repeated-thinking hint still is not on the default Qwen openai-completions payload path the PR says it fixes.

## What This Changes

This PR wraps prepared embedded Pi settings in an in-memory retry-disabled snapshot and adds repeated-thinking loop-hint injection inside transport message transformation.

## Best Possible Solution

Keep the PR open and revise it so the retry override is applied without discarding the fully prepared effective settings, then move the repeated-thinking guard to a shared context/completions seam that actually reaches Qwen openai-completions traffic, with focused tests for both paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Preserve prepared compaction overrides:** `src/agents/pi-project-settings.ts:56-59`
  - body: `applyPiCompactionSettingsFromConfig()` applies its prepared reserve/keepRecent changes through `applyOverrides()`, which only updates the manager's effective settings. Rebuilding `flat` from `getGlobalSettings()` and `getProjectSettings()` drops those overrides before wrapping in memory, so this branch can silently undo the compaction preparation that currently happens in this function.
  - confidence: 0.9
- **[P2] Wire the loop hint into completions replay:** `src/agents/transport-message-transform.ts:115`
  - body: The detector is still only called from `transformTransportMessages()`, but default Qwen traffic uses `api: "openai-completions"` and `buildOpenAICompletionsParams()` calls `convertMessages()` directly. Moving the helper before this function's early return does not put the hint on the Qwen path; wire it into the shared context prep or completions builder instead.
  - confidence: 0.91
