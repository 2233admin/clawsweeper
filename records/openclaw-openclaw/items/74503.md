## Summary

Keep this PR open, but do not merge it as-is. The main branch does not already contain the requested retry-disable or repeated-thinking guard, and the PR targets a real open Qwen/openai-completions replay-loop report, but the new detector can keep injecting loop hints from stale history into unrelated later turns.

## What This Changes

The branch disables the embedded Pi SDK retry layer through an in-memory settings snapshot and adds repeated-thinking loop-hint injection to transport message transformation and OpenAI-compatible completions payload building.

## Best Possible Solution

Revise the PR so the retry override remains an in-memory prepared-settings change, and scope the repeated-thinking guard to the active replay suffix or the latest user/tool cycle so old loops cannot affect later unrelated turns. Add focused tests for the prepared settings retry flag, stale-history false positives, and the Qwen/openai-completions payload path.

## Reproduction Assessment

Yes. A focused unit test can build a Context with three historical assistant turns containing identical thinking plus tool calls, append a later normal user prompt, and show that the PR helper still appends the loop hint even though the active turn is unrelated.

## Solution Assessment

No. Disabling the inner SDK retry through an in-memory manager looks like the right direction, but the repeated-thinking detector is too broad; the safer solution is to inspect only the current retry/continuation suffix and prove both responses and completions paths with tests.
