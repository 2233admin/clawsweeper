## Summary

Keep open. Current main still enforces a stored Android gateway TLS pin, only prompts on first-time TLS trust, and has no visible path to accept a rotated reverse-proxy certificate without changing or wiping stored app state. The linked TOFU history makes this a real security-sensitive lifecycle gap rather than something to close as stale or implemented.

## What This Changes

Handle Android node reconnects after a reverse-proxy-terminated TLS certificate rotates without requiring users to clear app data and redo onboarding.

## Best Possible Solution

Add an explicit Android certificate-rotation recovery path that keeps pinning enabled by default, detects a stored-pin mismatch, shows a clear certificate-changed confirmation or pin-reset flow, updates `gateway.tls.<stableId>` only after user approval, and documents/tests the manual/setup-code reverse-proxy path.
