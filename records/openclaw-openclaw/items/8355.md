## Summary

Keep open. Current main still has a batched classic Voice Call STT -> LLM -> TTS path, while realtime voice is a separate bypass mode. The prior streaming TTS PR was closed unmerged, and the requested provider-generic sentence-level LLM-to-TTS-to-audio pipeline remains real work.

## Best Possible Solution

Keep this open as the canonical provider-generic classic Voice Call latency feature. The best implementation is an additive streaming speech/telephony TTS contract plus voice-call wiring that uses embedded-agent partial reply callbacks, sentence splitting, provider streaming audio where available, Twilio/Telnyx playback adapters, and tests for ordering, fallback, abort, and barge-in. Realtime voice should remain documented as a separate full-duplex bypass mode, not treated as the fix for this issue.
