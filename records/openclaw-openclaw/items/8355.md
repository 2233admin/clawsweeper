## Summary

Keep open. Current main still batches the classic Voice Call STT -> LLM -> TTS path: response generation returns one text result, telephony TTS returns one complete Buffer, and Twilio waits for that Buffer before chunking audio. The newer realtime voice mode is documented as a separate full-duplex bypass, and the prior implementation PR #42187 was closed unmerged, so the requested provider-generic sentence-level pipeline remains real work.

## Best Possible Solution

Keep this open as the canonical provider-generic classic Voice Call latency feature. The best implementation is an additive streaming TTS/speech contract plus voice-call wiring that uses embedded-agent partial reply callbacks, a sentence splitter, provider audio chunk streaming where available, and Twilio/Telnyx playback adapters with barge-in, abort, fallback, and ordering tests. Realtime voice should remain documented as a separate bypass mode, not the fix for this request.

## Risks / Open Questions

- A future fix crosses plugin-owned voice-call code and the shared speech provider contract; it should avoid hard-coding one provider if Telnyx/Twilio/provider-generic support is the product goal.
- The current realtime voice path solves low latency by bypassing classic STT -> LLM -> TTS, so it should not be treated as an implementation of this sentence-level TTS pipeline.
