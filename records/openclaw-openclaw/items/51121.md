## Summary

Keep this PR open. Current main still has the un-cleared 500 ms `setTimeout` in `IMessageRpcClient.stop()`, so the intended leak fix is not implemented on main. The PR should not be closed as obsolete, but it likely needs revision because the existing review comment points at a real lifecycle edge: `closed` is created once in the constructor, while `start()` can be called again after `stop()`, so clearing the new timeout can remove the only fallback SIGTERM on a reused client.

## Best Possible Solution

Revise and keep this PR as the implementation candidate: fix `IMessageRpcClient.stop()` so the timeout is cleared when the child closes first, but also make the close waiter per child/start cycle or otherwise preserve the fallback SIGTERM after a restart. Add focused fake-timer tests for child-close-before-timeout, timeout-kills-hung-child, and restart-then-stop behavior, then run the targeted iMessage plugin test lane.
