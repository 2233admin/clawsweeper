## Summary

Keep this PR open. Current main and the latest release still have the un-cleared iMessage `stop()` timeout, so the intended leak fix is not implemented. The PR is a plausible narrow fix candidate, and the author’s scope note plus bot review support that it only targets the iMessage timer path, but the unresolved review concern is real: `IMessageRpcClient.closed` is instance-scoped, so the PR’s `finally { clearTimeout(stopTimer) }` can cancel the fallback `SIGTERM` on a reused client after a prior close has already resolved that promise.

## Best Possible Solution

Revise this PR rather than close it: make the close waiter/reset state per spawned child, then clear the stop timeout in `finally` without losing fallback `SIGTERM` on repeated start/stop cycles. Add focused fake-timer tests for child-close-before-timeout, timeout-kills-hung-child, and restart-then-stop, then run the targeted iMessage plugin test lane before maintainer review/landing.
