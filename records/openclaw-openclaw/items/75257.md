## Summary

Keep this PR open. The linked bug is valid and the patch targets the right voice-call startup path, but the proposed await leaves the pre-start idle timeout active while STT connects, so slow-but-valid transcription handshakes can be closed as a start timeout before the new readiness/failure path finishes.

## What This Changes

The PR changes Twilio voice-call media-stream startup to await realtime STT connect before firing `onConnect`, closes the stream and calls `onDisconnect` on STT connect failure, adds media-stream regression tests, and adds a changelog entry.

## Best Possible Solution

Keep the STT-readiness sequencing change, but clear the accepted Twilio start frame out of pending/pre-start state before awaiting STT readiness, then preserve the intended 1011 close plus `onDisconnect` path for STT connect failures. Add coverage where STT connect takes longer than `preStartTimeoutMs` but less than the provider connect timeout.

## Reproduction Assessment

Yes. The original issue has a high-confidence live reproduction with config and logs; the PR regression can be reproduced with a focused media-stream unit test that sends a valid `start`, makes `connect()` resolve after `preStartTimeoutMs`, and observes an unintended 1008 start-timeout close.

## Solution Assessment

No. The direction is right, but the implementation should first mark a valid `start` as accepted or otherwise cancel the pre-start timer before awaiting STT, so the anti-idle guard does not race the provider readiness wait.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Clear pending start before awaiting STT readiness:** `extensions/voice-call/src/media-stream.ts:317`
  - body: `handleConnection` only clears the pre-start pending timer after `handleStart` returns a session. With this new await, the socket stays pending while STT connects; the default pre-start timeout is 5s, but OpenAI realtime STT allows 10s to connect, so any handshake taking more than 5s is closed as `Start timeout` before this new 1011/onDisconnect path can complete. Clear the pending state once the `start` frame is accepted, or otherwise stop the pre-start timer before awaiting `sttSession.connect()`.
  - confidence: 0.9
