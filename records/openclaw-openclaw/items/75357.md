## Summary

Keep open. Current main still has the usage-aware OpenClaw completions transport, but the no-proxy/no-TLS simple-completion path can remain on the pinned `pi-ai` stream path where generic local/custom `openai-completions` compat may suppress `stream_options.include_usage`; the linked open implementation candidate has not landed on current main.

## What This Changes

The issue requests that simple `openai-completions` sessions request streaming usage from llama.cpp/OpenAI-compatible endpoints and persist non-zero token usage in session JSONL.

## Best Possible Solution

Land the linked implementation candidate or an equivalent narrow target-repo patch that injects or routes `stream_options.include_usage=true` for streaming `openai-completions` simple sessions without requiring fake proxy/TLS overrides, while preserving provider-owned stream wrappers and adding focused regression coverage for usage-only final chunks.

## Reproduction Assessment

Yes. The issue gives concrete llama.cpp curl checks, and the source/dependency path shows a unit-level reproduction: a generic local `openai-completions` simple stream with no proxy/TLS and false streaming-usage compat can avoid both the OpenClaw transport and dependency-level `include_usage`.

## Solution Assessment

Yes. The narrowest maintainable fix is to reuse the existing usage-aware transport behavior or add an equivalent simple-path payload wrapper, rather than repairing session logs after the fact or asking users to configure transport overrides.
