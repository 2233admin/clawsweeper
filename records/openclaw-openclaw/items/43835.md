## Summary

Keep this PR open. The linked workspace-leak bug remains valid on current main, but the proposed patch is not merge-ready because it adds recursive workspace deletion without enough runtime-success, stored-path, and active-reference guards for sandbox state.

## What This Changes

The PR adds recursive cleanup of resolved sandbox workspace directories during sandbox-container pruning, skips shared scope, and checks agent-scope registry entries before deleting.

## Best Possible Solution

Update or replace the branch with backend-aware sandbox-container cleanup that deletes only an owned, recorded, inactive workspace after confirmed runtime removal, skips shared and still-referenced scopes, persists or otherwise verifies the workspace root used by the entry, and adds focused regression coverage for scope-key, active-reference, and removal-failure cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for workspace cleanup:** `src/agents/sandbox/prune.ts:66`
  - body: `entry.sessionKey` is already the normalized registry scope key. In agent scope, passing a stored key such as `agent:ops` back through `resolveSandboxScopeKey()` can resolve to the wrong agent workspace, deleting the wrong directory while leaving the leaked one behind.
  - confidence: 0.91
- **[P1] Confirm runtime removal before deleting the workspace:** `src/agents/sandbox/prune.ts:83`
  - body: The cleanup runs after removal is attempted, but current Docker removal uses an allow-failure path and swallows errors. If the daemon is unavailable or removal fails, this can delete a workspace still mounted by a live runtime.
  - confidence: 0.88
- **[P2] Guard cleanup against every remaining entry for the scope:** `src/agents/sandbox/prune.ts:70-75`
  - body: The remaining-entry check is limited to agent scope and the same registry. Session-scope duplicate entries and browser entries can still reference the same workspace, so pruning one stale entry can delete state still in use.
  - confidence: 0.83
- **[P2] Persist the workspace root before pruning directories:** `src/agents/sandbox/prune.ts:82`
  - body: The branch derives the cleanup path from the current `cfg.workspaceRoot`, but registry entries can outlive config or per-agent root changes. That can leave the original stale directory behind or delete a matching scope directory under the wrong current root.
  - confidence: 0.79

## Security Review

Status: needs_attention

Summary: The diff introduces data-loss-sensitive recursive deletion of sandbox workspace state without sufficient ownership, runtime-success, and active-reference guards.

Concerns:

- **[medium] Recursive deletion can target live or wrong workspace:** `src/agents/sandbox/prune.ts:83`
  - body: `fs.rm(workspaceDir, { recursive: true, force: true })` is security-sensitive in sandbox cleanup because the path is recomputed from current config and the delete is not gated on confirmed runtime removal or every active reference to the stored scope.
  - confidence: 0.88
