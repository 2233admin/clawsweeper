## Summary

Keep this PR open as useful source work, but do not merge the current branch. Current main still has the Docker sandbox workspace leak, while the PR head adds recursive cleanup with wrong-target and premature-deletion risks and is stale relative to the current backend-aware prune path.

## What This Changes

The branch adds non-shared sandbox workspace directory deletion to the sandbox prune helper after expired sandbox containers are removed.

## Best Possible Solution

Land a current-main replacement fix that deletes only Docker-owned non-shared sandbox workspaces after confirmed runtime removal, uses the stored registry scope identity, skips any workspace still referenced by remaining runtime or browser entries, and covers prune plus recreate/config-refresh behavior with focused tests.

## Reproduction Assessment

Yes. Source-level reproduction is high confidence: an expired Docker sandbox registry entry goes through pruneSandboxContainers, which removes the runtime and registry entry on current main but never removes resolveSandboxWorkspaceDir(cfg.workspaceRoot, entry.sessionKey). The PR regressions are also reproducible from source by using a stored agent scope key like agent:ops or a failed docker rm path.

## Solution Assessment

No. Cleaning stale Docker workspaces is the right direction, but this patch is not the best fix because it recomputes stored identity, deletes after an allow-failure removal path, only partially checks remaining users of a scope, and is not aligned with the current backend-manager prune path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Use the stored scope key for workspace cleanup:** `src/agents/sandbox/prune.ts:68`
  - body: Registry entries already store the normalized scope key in entry.sessionKey. Recomputing it in agent scope treats a stored key like agent:ops as malformed and falls back to agent:main, so prune can delete the wrong workspace while leaving the stale one behind.
  - confidence: 0.93
- **[P1] Gate workspace deletion on confirmed runtime removal:** `src/agents/sandbox/prune.ts:83`
  - body: execDocker(..., { allowFailure: true }) returns a nonzero result instead of throwing, but the new fs.rm still runs afterward. If Docker is down or rm -f fails, this can remove the workspace while the container still exists or still has the mount.
  - confidence: 0.9
- **[P2] Check all remaining entries before deleting scoped workspaces:** `src/agents/sandbox/prune.ts:70-76`
  - body: The remaining-entry guard only runs for agent-scope container registry entries. Session scope can also have duplicate entries after container name/prefix changes, and browser registry entries may still be using the same scope workspace, so this can delete a workspace that is still active.
  - confidence: 0.82
- **[P2] Avoid deleting paths from only the current workspace root:** `src/agents/sandbox/prune.ts:82-83`
  - body: The cleanup builds the path from params.cfg.workspaceRoot for every stale entry. If an entry was created under a previous per-agent workspace root, prune can leave the real stale directory behind and delete an unrelated current-root directory with the same scope key.
  - confidence: 0.75
