## Summary

Keep open. Current main covers the runtime compile-cache and public no-respawn tuning pieces, but the central first-class prewarm command/artifact or postinstall priming flow is still missing and needs maintainer product plus supply-chain judgment.

## What This Changes

Request to improve stateless-container cold starts by adding a pre-warmed Node compile-cache flow and a public opt-in to skip the CLI self-respawn path when operators already provide the required Node startup environment.

## Best Possible Solution

Keep the existing NODE_COMPILE_CACHE and OPENCLAW_NO_RESPAWN behavior, then have maintainers choose whether OpenClaw should expose an explicit operator-run warm-cache command with docs and tests. Prefer that explicit command over automatic postinstall priming unless maintainers explicitly accept the install-time execution, cache invalidation, and plugin-import tradeoffs.

## Reproduction Assessment

Yes for the remaining repository gap: current source, docs, tests, and command descriptors show compile-cache/no-respawn support but no first-class warm-cache command or postinstall prewarm flow. The exact reported cold-start timing was not reproduced in this read-only pass.

## Solution Assessment

Unclear. The current env-based pieces are narrow and maintainable, but the remaining prewarm proposal needs product and security posture decisions; an explicit warm-cache command appears safer than automatic postinstall prewarming if maintainers want the feature.
