## Summary

Keep open. Current main covers the public respawn opt-out and operator-managed compile-cache tuning, but it still lacks the requested first-class cache prewarming command, shipped cache artifact, or postinstall cache priming path; those remaining choices need maintainer product and security review.

## What This Changes

The issue requests cold-start improvements by shipping or generating a prewarmed Node compile cache and adding a public opt-in to skip the CLI self-respawn.

## Best Possible Solution

Keep the implemented `OPENCLAW_NO_RESPAWN` and scoped packaged compile-cache behavior, then have a maintainer choose a narrow public shape for build-time prewarming. The likely maintainable path is an explicit `openclaw warm-cache --output` helper with focused CLI tests and Linux/container docs, while treating postinstall priming and shipped cache artifacts as separate release/security decisions.

## Reproduction Assessment

Not applicable as a conventional bug reproduction. The remaining gap is source-inspectable: current main has cache enablement and no-respawn support, but no registered warm-cache command, shipped cache artifact, or postinstall prewarming implementation.

## Solution Assessment

Unclear until maintainers choose the accepted product shape. An explicit opt-in warm-cache command looks narrower and easier to secure than automatic postinstall priming or publishing prewarmed V8 cache artifacts.
