## Summary

Keep open. Current main already has scoped Node compile-cache support and a public no-respawn escape hatch, but the remaining requested upstream surface, a first-class cache prewarm command/artifact or postinstall priming flow, is still absent and needs maintainer product/security judgment.

## What This Changes

The issue requests a first-class way to pre-warm or ship a Node compile cache for cold starts and a public opt-in to bypass CLI self-respawn in controlled container environments.

## Best Possible Solution

Keep the existing env-based compile-cache and no-respawn mechanisms, then decide whether to add an explicit operator-run `openclaw warm-cache --output ...` command with docs and focused tests. Treat automatic postinstall prewarming or broad plugin-import priming as a separate product/security decision rather than the default path.

## Reproduction Assessment

Yes for the remaining repo gap: current main's source, docs, postinstall script, and command descriptors show compile-cache/no-respawn support but no first-class warm-cache command or postinstall priming flow. The exact cold-start speedup remains externally reported and was not reproduced locally.

## Solution Assessment

Unclear. The current env-based no-respawn and scoped compile-cache implementation is maintainable, but the remaining proposal has product and supply-chain choices; an explicit warm-cache command looks safer than automatic postinstall prewarming if maintainers accept the feature.
