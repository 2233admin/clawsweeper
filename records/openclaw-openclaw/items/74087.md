## Summary

Keep this issue open. Current main already has a public no-respawn escape hatch, automatic packaged-launcher Node compile-cache enablement, startup-tuning docs, and stale-cache pruning on postinstall, but it still lacks the central requested pre-warm surface: no `openclaw warm-cache` command, shipped pre-warmed cache artifact, or automatic postinstall cache priming. The remaining shape needs maintainer product/security judgment before implementation.

## Best Possible Solution

Keep the issue open and have maintainers choose a narrow accepted shape. The safest likely path is an explicit, documented `openclaw warm-cache --output <dir>` image-build helper with targeted tests and VPS/container docs; treat automatic postinstall prewarming or shipped prewarmed artifacts as separate product/security decisions. The existing `OPENCLAW_NO_RESPAWN=1` and `NODE_COMPILE_CACHE` guidance can remain the documented public escape hatch unless maintainers want a CLI alias.

## Risks / Open Questions

- Automatic postinstall cache prewarming would expand install-time code execution and should receive explicit maintainer/security review before implementation.
- Pre-warmed Node compile-cache artifacts are sensitive to Node version, architecture, package path, and package update invalidation; stale-cache behavior is already a known adjacent concern from #71348.
- A public respawn bypass can skip bootstrap setup for warning suppression and CA certificate environment if operators set it without providing equivalent environment themselves.
- Best-effort importing plugin entries to prewarm cache may execute top-level plugin code, so the accepted design should avoid surprising side effects or make the risk explicit.
