## Summary

Keep open. The underlying provider-onboarding bug still exists on current main, but this PR is not merge-ready because the LiteLLM part targets deleted legacy command files and the remaining implementation can treat partial token metadata as authoritative.

## What This Changes

The PR adds LiteLLM-compatible `/model/info` probes during Copilot Proxy and LiteLLM onboarding and uses discovered token limits in generated model definitions.

## Best Possible Solution

Refresh this as a provider-owned implementation in `extensions/litellm` and `extensions/copilot-proxy`, using complete `/model/info` token metadata only when it is authoritative and preserving existing configured limits on failure or incomplete responses.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to configure a LiteLLM or Copilot Proxy model whose `/v1/model/info` reports a larger `max_input_tokens`, run onboarding, and inspect that current main still writes static 128000/8192 limits from the provider builders.

## Solution Assessment

No. The proposed direction is correct, but this branch is not the best current fix because part of it targets deleted command onboarding code and the LiteLLM probe can mark fallback values as discovered when metadata is incomplete.
