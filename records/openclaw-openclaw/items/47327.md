## Summary

Keep open. Current main still has no parse-error diagnostics field or CLI rendering for malformed local skills, and this PR remains useful source work, but the proposed patch is incomplete because it only annotates skills that already survived loading and still misses `skills info` and `skills check` output paths.

## What This Changes

The PR adds a `parseError` field to skill/status entries, captures `SKILL.md` read/parse errors while re-reading loaded workspace skills, marks parse-error skills ineligible, and shows parse errors in `openclaw skills list` text/JSON output.

## Best Possible Solution

Preserve read/parse diagnostics at the local skill discovery boundary, carry them through `SkillEntry` and `SkillStatusEntry`, render them consistently in `skills list`, `skills info`, and `skills check` text/JSON output, and cover the behavior with focused loader/status/CLI regression tests while keeping the current tolerant frontmatter parser behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve diagnostics before the loader drops skills:** `src/agents/skills/workspace.ts:513-518`
  - body: The patch records `parseError` only while re-reading skills that were already returned by the loader. On current main, read/parse failures in `loadSingleSkillDirectory` return `null`, so malformed `SKILL.md` files never become status records that list/info/check can render.
  - confidence: 0.88
- **[P2] Handle parse errors in skill info output:** `src/cli/skills-cli.format.ts:189-195`
  - body: `formatSkillInfo` still derives status only from eligible/disabled/allowlist state. A parse-error skill would render as ready or needs setup, and the actual parse error would not appear in the detailed info command.
  - confidence: 0.9
- **[P2] Add a parse-error bucket to skills check:** `src/cli/skills-cli.format.ts:296-318`
  - body: `formatSkillsCheck` still groups every non-eligible, non-disabled, non-blocked skill under missing requirements. Parse-error skills need their own summary and JSON/text section so users do not see an empty or misleading missing-requirements entry.
  - confidence: 0.9
