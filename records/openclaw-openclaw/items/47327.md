## Summary

Keep open. Current main fixes the original colon-in-frontmatter silent-drop case with a tolerant parser, but it does not implement this PR's advertised parse/read-error diagnostic surface; the PR branch is also incomplete because it records diagnostics after failed skills have already been dropped and leaves `skills info` and `skills check` misleading.

## What This Changes

The PR adds `parseError` fields to skill entries and status records, marks parse-error skills ineligible, and renders parse errors in `openclaw skills list` text and JSON output.

## Best Possible Solution

Keep the shipped tolerant frontmatter parser, and only pursue this PR if maintainers still want explicit diagnostics for unrecoverable local skill read/parse failures. If pursued, diagnostics should be captured at discovery time and rendered consistently in `skills list`, `skills info`, and `skills check` without weakening safe file loading.

## Reproduction Assessment

Yes, for the PR defects. Code inspection gives a high-confidence path: make discovery fail before `SkillEntry` creation and no CLI record can show the error; separately inject a parse-error status record and `skills info`/`skills check` still ignore or misbucket it.

## Solution Assessment

No. The submitted approach is too late in the loader pipeline and too narrow in the formatter layer; the safer alternative is discovery-level diagnostics plus complete list/info/check text and JSON coverage while preserving the current tolerant parser behavior.
