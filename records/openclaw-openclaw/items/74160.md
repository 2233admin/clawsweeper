## Summary

Keep this PR open. The reported Anthropic prefill loop is valid and the PR is aimed at the right session-file repair boundary, but the current patch still misses poisoned transcripts where a non-message metadata entry is the file leaf above a trailing errored assistant; the upstream SessionManager still walks that path and replays the assistant message underneath it. The Greptile counter finding has been addressed, but one blocking repair finding remains plus a small docs formatting issue.

## What This Changes

The PR adds a tail-drop pass to `repairSessionFileIfNeeded`, a `droppedTrailingErrorEntries` report/log counter, new session-file repair regressions, and transcript-hygiene documentation for Anthropic trailing assistant prefill loops.

## Best Possible Solution

Repair should preserve non-message metadata rows but reason over the replay-visible tail: scan past replay-transparent metadata when finding the final model-visible message, drop any trailing errored assistant run that would still be replayed as the final assistant turn, keep mid-transcript sentinel rewrites, and add regression coverage for `errored assistant -> model_change` and similar metadata leaves.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is a session JSONL branch shaped as `user -> errored assistant -> model_change` or `thinking_level_change`: the PR stops at the metadata row, while SessionManager still builds replay messages from the path underneath it.

## Solution Assessment

No. The repair location and this-pass counter fix are sound, but stopping at the first non-message tail entry is not the narrowest maintainable fix because some metadata entries are leaf-only state and should be preserved while the repair scans to the last replay-visible message.
