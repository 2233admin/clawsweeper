## Summary

Keep open: this is a fresh current-release Telegram/Gateway reliability report and current `main` does not prove the reported recurring unresponsive-after-error state is fixed. The code has useful partial coverage, including Telegram dispatch fallback replies, runner failure normalization, session reset for specific transcript/session errors, and a long-polling watchdog that restarts Telegram polling, but the reported normal-message failure with recovery only after `openclaw gateway restart` is broader than those implemented paths and lacks logs/root cause evidence for a safe close.

## What This Changes

The issue asks OpenClaw to diagnose and fix an intermittent Telegram gateway failure that returns a generic processing error until manual gateway restart, with automatic recovery and better diagnostic logging.

## Best Possible Solution

Keep the issue open while maintainers collect the failing gateway log window around the April 23 and April 29 occurrences, then either link it to a specific narrow fix such as #74800 or add targeted recovery/diagnostics in the owning layer: shared agent/session recovery for corrupted turns, Telegram dispatch/lane cleanup if a turn leaves state stuck, or Telegram transport recovery if the logs show polling/API liveness loss.
