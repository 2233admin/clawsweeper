## Summary

Keep open: this is a fresh current-release Telegram/Gateway reliability report, and current main does not prove that the reported recurring normal-message failure and post-error unresponsive state is fixed. Main has useful partial recovery for generic agent failures, Telegram dispatch fallback replies, scoped session resets, and polling liveness restarts, but those do not establish recovery from the broader manual-gateway-restart symptom without the failing log window.

## What This Changes

The issue requests root-cause diagnosis, automatic recovery, and better logging for intermittent Telegram Gateway generic processing errors that leave the session unusable until a manual gateway restart.

## Best Possible Solution

Keep this open until maintainers capture the gateway log window and, ideally, a diagnostics/stability export around the next failure; then fix the owning layer narrowly, whether that is agent/session recovery, Telegram dispatch lane cleanup, plugin command handling if logs match #74800, or Telegram transport liveness.

## Reproduction Assessment

No high-confidence way to reproduce the issue yet. The report explicitly says the trigger is unknown/intermittent, and current source inspection only verifies possible fallback and recovery paths, not the persistent wedge after a normal message.

## Solution Assessment

Unclear. A broad automatic gateway restart on every generic processing failure would be a risky product behavior; the safer direction is log-driven targeted recovery plus better diagnostics in the specific layer that leaves the session or transport stuck.
