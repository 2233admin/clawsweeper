## Summary

Keep open: this is a fresh current-release Telegram/Gateway reliability report, and current main still only shows partial recovery coverage. The exact generic error path, Telegram dispatch fallback, polling watchdog, diagnostics recorder, and stuck-session recovery exist, but none proves the reported normal-message wedge that only recovers after `openclaw gateway restart` is fixed.

## What This Changes

The issue asks for root-cause diagnosis, automatic recovery, and better logging for intermittent Telegram gateway failures that leave sessions unresponsive until a manual gateway restart.

## Best Possible Solution

Keep the issue open while maintainers collect the failing Gateway log/stability window, then land a scoped fix in the owning layer: plugin-command result hardening if logs match #74800, Telegram dispatch/lane cleanup if a turn leaves state stuck, agent/session recovery if the embedded run wedges, or Telegram transport recovery if polling/API liveness is the real failure.

## Reproduction Assessment

No: the issue gives platform and occurrence dates, but no reliable trigger, logs, diagnostics bundle, or affected session state. Current main can be inspected for matching fallback and recovery paths, but it cannot reproduce the intermittent wedged state with high confidence.

## Solution Assessment

Unclear: a blanket gateway restart would be a broad recovery policy, and current evidence does not identify the failing layer. The best maintainable solution is to diagnose the root cause first and add targeted recovery or logging where the logs point.
