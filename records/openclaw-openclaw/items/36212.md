## Summary

Keep this issue open. Current main still captures the provider message id on the initial queued FollowupRun, but collect drains, overflow/summary drains, followup runtime setup, and followup delivery do not propagate that id where the agent or Feishu reply delivery can use it. The directly related fix PR in the provided GitHub context is closed and unmerged.

## Best Possible Solution

Implement a narrow core queue/followup continuity fix: normalize queued FollowupRun.messageId values, include trusted per-item message_id metadata in collect prompt blocks, propagate a suitable messageId through summary/single-message drains, pass queued.messageId into runEmbeddedPiAgent as currentMessageId, and feed that currentMessageId into followup delivery reply-threading. Cover collect-mode, summary drain, followup-runner, and followup-delivery behavior with focused regression tests, without Feishu-specific core logic.
