## Summary

Keep this issue open. Current main still captures the original Feishu provider message id and stores it on the initial queued FollowupRun, but collect drains, overflow summary drains, followup runtime, and followup delivery still do not carry that id through as queued-message metadata/currentMessageId. The related fix PR #36235 is closed and unmerged, so there is no merged canonical fix.

## What This Changes

Queued followup messages should preserve the original platform message_id so native reply APIs can target the queued message.

## Best Possible Solution

Implement a narrow core queue/followup continuity fix: normalize queued FollowupRun.messageId values, include trusted per-item message_id metadata in collect prompt blocks, propagate messageId through collect and overflow summary drains, pass queued.messageId into runEmbeddedPiAgent as currentMessageId, and use that currentMessageId in followup delivery reply threading. Keep the fix channel-agnostic and add focused regressions for collect mode, summary drain, followup runner, and followup delivery.
