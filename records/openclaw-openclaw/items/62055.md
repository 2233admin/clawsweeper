## Summary

Keep this issue open. Current main still has no Windows `--stack-size` startup handling, and the CLI respawn planner explicitly returns `null` on `win32`; the unit test suite also asserts that Windows does not respawn. The merged plugin-loading fast-path work in #74173 is relevant to adjacent Windows stack/module-depth problems, but it does not implement the requested CLI startup stack fix for `dashboard` and other heavy admin commands.

## What This Changes

The issue requests a Windows-specific CLI startup fix, likely via `src/entry.respawn.ts`, so heavy CLI/admin commands do not crash from V8 stack exhaustion or heap OOM on native Windows.

## Best Possible Solution

Add a narrow Windows-safe startup path that supplies a larger V8 stack for affected CLI/admin commands when the user has not already set one, while respecting `OPENCLAW_NO_RESPAWN`, help/version/gateway skip policy, and the Windows raw-path safeguards added for onboarding. Back it with unit tests and a native Windows smoke that exercises `dashboard --no-open` and the other reported heavy commands.

## Reproduction Assessment

Yes. The issue has concrete native Windows install/run steps, crash signatures, and a working stack-size workaround from the reporter, plus an independent confirmation on a later release; this review did not live-run Windows but found current main still lacks the requested startup mitigation.

## Solution Assessment

Unclear but actionable. The proposed `--stack-size` respawn direction is plausible, but the current `win32` no-respawn guard was intentional, so the best repair should add a narrow Windows stack-size path rather than blindly removing that guard.
