## Summary

Keep this issue open. Current main still lacks a configurable Telegram TTS delivery mode: the shared TTS config surface has no `delivery`/`separate` option, auto-TTS still returns one reply payload containing both visible text and generated audio, and Telegram still treats normal-length text as a voice/audio caption unless it overflows Telegram's caption limit. The existing ClawSweeper comment kept this open, and the current SHA still supports that conclusion.

## Best Possible Solution

Keep this issue open. Implement a Telegram-owned delivery policy or shared `messages.tts.delivery` contract with `bundled` and `separate` modes, update strict config types/schema/generated docs, and add focused Telegram bot plus direct-send tests. In `separate` mode, Telegram should send visible text as a normal message first, then send the voice note without a caption.
