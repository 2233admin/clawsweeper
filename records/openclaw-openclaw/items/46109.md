## Summary

Keep this issue open. Current main has a substantial SecretRef runtime, CLI, docs, and config-write preflight foundation, but the requested source-aware Control UI/config UX is still not implemented: structured SecretRefs render as read-only generic blank placeholders, SecretInput union analysis still normalizes to the plain string editor path, and the public docs still describe form-mode limitations rather than config/env/unset source badges or resolved status.

## What This Changes

The issue requests first-class `.env` and environment-backed secret configuration UX across CLI and Control UI, including source metadata, env-reference preservation, a secrets/environment helper, migration guidance, and masked-but-visible fields.

## Best Possible Solution

Define the safe SecretRef metadata contract, then add a source-aware SecretInput renderer and supporting metadata flow that shows masked source labels, env variable identity where allowed, resolved/unresolved state, and explicit config/env/unset editing, with docs and tests aligned to the existing secrets CLI and config-write preflight behavior.

## Security Review

Status: needs_attention

Summary: No PR diff was reviewed, but this secret-management UX feature needs focused security review before implementation.

Concerns:

- **[medium] Define safe SecretRef metadata disclosure:**
  - body: Showing env var names, provider aliases, config paths, or resolved/unresolved status can reveal deployment structure even when secret values stay masked. Maintainers should define which metadata is safe before the Control UI exposes it broadly.
  - confidence: 0.86
- **[medium] Avoid resolving or transporting secret values for UI status:**
  - body: The requested resolved/unresolved UX should use bounded metadata or existing preflight/status seams, not raw resolved secret values, so the UI does not broaden secret exposure while improving clarity.
  - confidence: 0.78
