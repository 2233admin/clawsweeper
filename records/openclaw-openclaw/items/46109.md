## Summary

Keep this issue open. Current main has strong SecretRef runtime/CLI support and a Control UI safeguard that prevents structured SecretRefs from being stringified or corrupted, but it still does not provide the requested first-class env-backed secret UX: source badges, env-var identity/resolution state, config/env/unset switching, or a dedicated Secrets/Environment helper in the Control UI.

## Best Possible Solution

Keep this open and design a focused Control UI/CLI follow-up on top of the existing SecretRef contracts: render secret fields with `config` / `env` / `unset` source state, show only approved safe metadata and resolved/unresolved status, add controls for switching between inline config, SecretRef, and unset, and reuse the existing `openclaw secrets` target registry for a dedicated secrets/environment helper.
