## Summary

Keep this issue open. Current main has solid SecretRef runtime, CLI, validation, redaction, and documentation foundations, but the requested first-class Control UI/config UX is still missing: env-backed structured SecretRefs render as blank read-only generic placeholders, SecretInput unions are normalized to the string editor path, and the docs describe that limitation rather than source-aware config/env/unset controls.

## What This Changes

The issue requests first-class `.env` and environment-backed secret UX in the CLI and Control UI, including source badges, masked-but-visible SecretRef state, env reference preservation, a secrets helper, and migration guidance away from plaintext config secrets.

## Best Possible Solution

Define a safe SecretRef metadata contract, then implement a source-aware SecretInput renderer and supporting metadata flow that shows masked source labels, allowed env variable identity when appropriate, resolved/unresolved status, and explicit config/env/unset editing, with Control UI, CLI/docs, and regression tests aligned.

## Reproduction Assessment

Yes for the current UX gap. The focused browser test path constructs an env SecretRef for `channels.discord.token` and asserts the current blank read-only Structured placeholder behavior, which matches the reported ambiguity.

## Solution Assessment

Unclear until maintainers set the security disclosure boundary. The maintainable fix is a bounded metadata contract plus UI/docs/tests, not a cosmetic placeholder change that guesses which secret source details are safe to expose.
