## Summary

Keep open. Current main still validates cron announce channel selection but does not validate explicit primary or failure-destination account IDs against configured channel accounts before persistence, and there is already open PR context for the same bug.

## What This Changes

The issue requests creation-time validation so `openclaw cron create/add --announce --channel <channel> --account <id>` rejects unknown channel account IDs instead of persisting them.

## Best Possible Solution

Land one focused gateway-side validation change that rejects unknown explicit announce account IDs for both `cron.add` and `cron.update`, including failure destinations, with regression tests and a user-facing changelog entry; then close this as implemented.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, and current-main source inspection shows `--account` is forwarded into `delivery.accountId`, shape-checked as a string, and allowed to reach persistence after channel-only validation.

## Solution Assessment

Yes. Fail-fast validation in the gateway cron add/update handlers is the narrowest maintainable fix because it covers CLI and RPC callers before persistence while leaving delivery execution and implicit account resolution unchanged.
