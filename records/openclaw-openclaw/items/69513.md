## Summary

Keep open. Current main still validates cron announce channels but not explicit primary or failure-destination account IDs before persistence, and there is open candidate PR context for the same bug with closing syntax, so this should remain open until maintainers review and land or close that fix path.

## What This Changes

The issue asks OpenClaw to reject or warn when cron announce creation or update receives an explicit delivery accountId that is not configured for the selected channel.

## Best Possible Solution

Land one focused gateway-side validation change that rejects unknown explicit announce account IDs for both cron.add and cron.update, including failure destinations, with regression tests and a user-facing changelog entry; then close this as implemented.

## Reproduction Assessment

Yes. The issue gives concrete CLI steps, and current-main source inspection shows --account is forwarded as delivery.accountId, shape-checked only as a string, and persisted after channel-only validation.

## Solution Assessment

Yes. Gateway-side validation in cron.add and cron.update is the narrowest maintainable fix because it covers CLI and RPC callers before persistence while leaving delivery execution and implicit account resolution unchanged.
