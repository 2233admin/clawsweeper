## Summary

Keep open. Current main has partial protections for direct CLI update teardown and future package tarball import checks, but the live Gateway `update.run` package path can still run the staged global package update and swap from inside the active Gateway process before restart, which matches the reported stale hashed-chunk failure class.

## What This Changes

The issue requests safer package auto-update behavior so stale Rollup chunk references cannot leave Telegram inbound or post-update CLI paths broken until a manual restart.

## Best Possible Solution

Make package updates atomically safe for running OpenClaw processes: gateway-initiated package updates should hand off the swap to an external fresh updater or stop the managed service before mutation, installed-package verification should reject missing relative dist imports before swap, and channel-specific recovery should only be added as a narrow fallback after the generic update guarantee exists.

## Reproduction Assessment

Yes for the failure shape: the issue includes concrete versions, npm-global install layout, Telegram missing-module paths, timestamps, and recovery by restart, and current source still exposes a matching live Gateway package-swap path. The exact historical auto-update race was not replayed because this review is read-only.

## Solution Assessment

No, current main is not yet the best solution. The direct CLI respawn and tarball checks are useful partial fixes, but the maintainable fix is a generic update handoff plus installed dist import-graph verification so stale hashed chunks cannot be introduced or exposed by a live package swap.

## Repair Work Prompt

Fix https://github.com/openclaw/openclaw/issues/75206. The observable bug is that package auto-update can replace the installed OpenClaw dist tree while the live Gateway or old CLI process still has ESM module records pointing at old hashed chunks, causing `ERR_MODULE_NOT_FOUND` for inbound Telegram/runtime imports until restart. Implement the generic safety fix, not a Telegram-only workaround: for gateway `update.run` on package/global installs, hand off package mutation to an external fresh updater or stop the managed service before swapping the package tree; do not mutate the active package root from the live Gateway process and then schedule restart afterward. Extend installed global package verification to reject relative dist imports that point at missing files, reusing or mirroring the existing `scripts/lib/package-dist-imports.mjs` behavior in a TypeScript-safe production path as appropriate. Preserve the existing direct CLI protections: post-core fresh-process resume, Node compile-cache disabling for updated entrypoints, and swallowed best-effort memory-runtime teardown failures. Add focused regression tests for gateway update handoff/mutation blocking, installed package import-graph verification, staged swap failure behavior, and the direct CLI stale-chunk finalizer path. Add a user-facing changelog fix entry crediting `@xonaman` and `@faux123`. Do not broaden this into channel-specific Telegram self-healing unless a generic runtime module-load failure hook already exists.
