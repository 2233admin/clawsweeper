## Summary

Keep open. This is a fresh v2026.4.27 Telegram inbound recurrence after earlier generic stale-chunk update fixes, and current main does not yet prove the requested safety property. Main has useful CLI service guards and release-time package tarball import checks, but install-time verification still does not validate the installed dist import graph, `update.run` can still invoke package update work from the live gateway when restart is available, and there is no source-backed Telegram/module-not-found self-heal or alert path for this failure.

## What This Changes

The issue asks OpenClaw to make package auto-updates atomic and import-safe, and to recover or alert when Telegram inbound hits missing hashed runtime chunks after an update.

## Best Possible Solution

Add a generic update/package safety fix: reject staged packages whose installed dist JavaScript import graph references missing files before swap, and make gateway-initiated package updates hand off to a fresh external updater/restart path instead of mutating the active dist tree in-process. Cover the reported conversation-runtime to inbound.runtime missing-chunk class with focused regression tests, then add narrow channel recovery only if a generic runtime module-load failure hook already exists.

## Reproduction Assessment

Yes for the failure shape: the report gives concrete versions, install method, channel, missing module paths, timestamps, and recovery by restart. The exact old auto-update sequence is not replayable from current main, but the matching update surfaces are clear enough to keep this actionable.

No, current main is not yet the complete fix.
