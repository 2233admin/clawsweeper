## Summary

Keep open. Current main has useful hardening for shell-driven package updates, but the core stale-chunk safety property is still not complete: the Gateway `update.run` path can still call the global package updater from the live Gateway process, and the release validation path explicitly tolerates the same self-swapped `ERR_MODULE_NOT_FOUND` class after a nominally successful package update.

## What This Changes

The issue requests a safe package auto-update boundary so hashed dist chunks cannot be replaced while a live OpenClaw process still references old Telegram inbound/runtime chunks.

## Best Possible Solution

Add a generic package-update safety boundary: package-manager updates initiated from Gateway/auto-update should either delegate the package swap to a fresh external updater before mutating the active dist tree, or fail before mutation with a clear status when a safe handoff cannot be guaranteed. Release/package validation should stop accepting stale self-swapped `ERR_MODULE_NOT_FOUND` as a successful update outcome unless an intentional recovery path also guarantees the old process exits cleanly.

## Reproduction Assessment

Yes. The issue provides versions, install method, missing hashed chunk paths, timestamps, and restart recovery; current main also shows a reproducible source path where `update.run` can replace the global package tree before restart scheduling.

## Solution Assessment

No. Current main’s CLI service guard, managed-service stop, update respawn, and tarball import-graph check are useful partial mitigations, but the narrow maintainable fix is a generic package-update handoff/fail-closed boundary rather than Telegram-specific restart recovery.
