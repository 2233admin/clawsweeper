## Summary

Keep open. The report is fresh, concrete, and not safely closeable as implemented: current main has related post-release gateway lifecycle hardening for stale hashed chunks, but Telegram middleware errors such as the reported `ERR_MODULE_NOT_FOUND` are still only logged, and I did not find a current-main recovery path that restarts the channel or gateway when inbound handler/runtime imports are missing.

## What This Changes

The issue reports that an npm-global auto-update from v2026.4.24 to v2026.4.27 left a running Telegram gateway with stale hashed runtime imports, causing inbound Telegram messages to fail until a manual gateway restart.

## Best Possible Solution

Keep this issue open until maintainers either prove the current gateway lifecycle runtime hardening covers this exact update path and ship it, or add a generic runtime-chunk failure recovery path that validates post-update restart success and escalates missing packaged runtime imports instead of letting inbound channel errors repeat silently.

## Reproduction Assessment

Yes, at the operational level: run a managed npm-global Telegram gateway from v2026.4.24, update to v2026.4.27 without a successful gateway process restart, then send an inbound Telegram message after the package dist files have been replaced. The provided logs and current `bot.catch` behavior make that path credible, though I did not live-run it in this read-only review.

## Solution Assessment

Unclear. The current stable gateway lifecycle runtime boundary is the right generic direction for update-time stale chunk failures, but it is not enough evidence to close because Telegram still only logs handler import failures and there is no shipped release proof for the post-v2026.4.27 hardening.
