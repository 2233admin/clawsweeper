## Summary

Keep open. This is a fresh v2026.4.27 availability report after earlier stale-chunk update fixes, and current main still does not prove the safety property: gateway-triggered updates can still run package update work from the live gateway before restart, installed-package verification checks inventory membership rather than JS import reachability, and Telegram middleware module-load failures are logged without a source-backed restart or alert path.

## What This Changes

Ensure auto-update cannot leave `conversation-runtime` importing missing `inbound.runtime` chunks, and recover or alert when Telegram inbound hits this module-load failure.

## Best Possible Solution

Add a generic update safety guarantee: reject staged or installed OpenClaw packages whose `dist` JavaScript import graph references missing files before activation, and route gateway-triggered package updates through a fresh external updater or clean managed-service handoff so the active dist tree is not mutated in-process. Add Telegram module-load recovery only through a generic runtime failure hook if the update guarantee alone cannot fully protect inbound availability.

## Reproduction Assessment

Yes for the failure class: a focused regression can construct a staged npm package whose `dist` JS imports a missing relative chunk and show that current installed-package verification does not reject it. The exact old auto-update timing is not replayed, but the report and current source make the failure path actionable.

## Solution Assessment

No, current main is not the complete solution. The narrow maintainable repair is generic import-graph verification plus safe gateway update handoff; a Telegram-only restart workaround would be a fallback, not the primary fix.
