## Summary

Keep open. Current main has partial update hardening, but the reported stale hashed-chunk failure is still a real reliability gap: live Gateway/update paths can still mutate the installed package before a guaranteed fresh-process handoff, and current release smoke code explicitly tolerates stale post-swap module imports as a recoverable condition rather than proving they cannot happen.

## What This Changes

The issue asks OpenClaw to prevent npm global auto-update from leaving old Gateway/CLI processes with stale hashed dist chunk references that break Telegram inbound handling until restart.

## Best Possible Solution

Add a generic package-update safety boundary: package-manager updates initiated from a running Gateway or auto-update should hand off the package swap to an external fresh process before mutating the active dist tree, or fail closed when a safe restart cannot be guaranteed. Keep Telegram-specific recovery secondary to that generic update boundary, and preserve release tarball import-graph validation as a separate package-integrity check.

## Reproduction Assessment

Yes. The report gives versions, install method, channel, concrete missing hashed chunk paths, timestamps, and restart recovery, and the follow-up comment gives a second `memory-state` stale-chunk path. The current source review shows plausible paths where package replacement can precede a guaranteed fresh module graph.

## Solution Assessment

No. Current main has partial mitigations, but the best fix is not a Telegram-only restart hook or merely tolerating stale imports in smoke tests; it should remove or fail closed the live-process package-swap path and verify that post-update work runs only from a fresh process/package tree.
