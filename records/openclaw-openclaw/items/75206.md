## Summary

Keep open. Current main has partial package-update hardening, but the Gateway `update.run` path still can invoke the global package update runner from the live Gateway process and only schedules restart after the update returns, so the stale in-memory hashed chunk failure remains plausible for package auto-updates.

## What This Changes

Ensure auto-update/package updates cannot leave the running Gateway with stale hashed ESM chunk imports that break Telegram inbound until restart.

## Best Possible Solution

Add a generic package-update safety boundary for Gateway-initiated updates: package installs must be handed off to a fresh/external updater that can stop or replace the Gateway before mutating the active dist tree, or fail closed with a clear diagnostic when that handoff is unavailable. Keep Telegram-specific recovery secondary to the package update boundary.

## Reproduction Assessment

Yes. The issue and follow-up comment provide concrete versions, install method, missing hashed chunks, timestamps, and restart recovery; current source shows the Gateway RPC path can still reach the package tree swap before the live Gateway process is restarted.

## Solution Assessment

No. Current main has useful CLI and staging mitigations, but they do not apply to the Gateway `update.run` package path; the narrow maintainable fix is an external/fresh-process package-update handoff or a fail-closed Gateway RPC response for package installs.

## Repair Work Prompt

Fix the package-update stale chunk race reported in openclaw/openclaw#75206. Current main still lets Gateway `update.run` call `runGatewayUpdate` in the live Gateway process, which can reach `runGlobalPackageUpdateSteps` and swap the global package tree before the Gateway restarts. Add a generic package-update safety boundary for Gateway-initiated updates: for global/package install surfaces, do not mutate the active package tree from the live Gateway process. Prefer a fresh/external updater handoff that can stop or replace the Gateway before package mutation; if that is not safely available, fail closed with a clear `UpdateRunResult` reason before calling `runGatewayUpdate`. Do not broaden the scope into #64892 raw agent self-update policy, #67702 git-mode-only guarding, or #72813 auto-reply lazy-loader recovery. Add focused tests proving `update.run` does not call the inline package update runner for package/global installs from the Gateway path, and keep existing CLI package update behavior intact. Add a user-facing CHANGELOG.md fix entry with appropriate non-bot attribution.
