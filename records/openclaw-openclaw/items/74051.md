## Summary

Keep open. Current main still has the behavior this PR changes: light dreaming orders by `lastRecalledAt` before `recallCount`, and both rank/apply promotion gates count `dailyCount` through aggregate signal count. The PR is a focused implementation candidate for the linked memory promotion reports, but it changes an existing documented/tested promotion-signal contract and does not cover #71976's separate rehydration-span concern, so it needs maintainer review rather than cleanup closure.

## Best Possible Solution

Review this PR as the active implementation candidate: decide the intended `minRecallCount` semantics for organic recalls, daily signals, and grounded evidence; keep or adjust the focused regression tests; update the Dreaming/memory promotion docs if daily-only signals should no longer satisfy the default gate; and either extend the PR or leave a clear follow-up for #71976's `relocateCandidateRange` span concern before treating the linked issue as fully fixed.
