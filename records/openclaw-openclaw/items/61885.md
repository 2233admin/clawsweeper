## Summary

Keep open. Current main still behaves exactly as reported, but the sessionStorage choice is documented, tested, and tied to prior security hardening, so changing it to localStorage is a maintainer/security product decision rather than a safe cleanup close or automated fix.

## What This Changes

Request to persist the Control UI gateway token in localStorage so it survives closing and reopening the browser.

## Best Possible Solution

Keep the sessionStorage default unless maintainers explicitly choose a durable credential feature. If they do, design it as an opt-in Control UI auth setting with security review, clear docs/tests, and token-rotation behavior rather than silently changing getSessionStorage() to localStorage.

## Reproduction Assessment

Yes. The source and tests give a high-confidence reproduction path: save a Control UI token, observe it is written through sessionStorage and omitted from localStorage, then a new browser session will not reload it.

## Solution Assessment

No. The suggested one-line localStorage switch conflicts with current docs, tests, and prior security hardening; a safer solution would be a maintainer-approved opt-in flow or leaving the current session-scoped behavior unchanged.
