## Summary

Keep this issue open. Current main still lacks an opt-in Slack shared-thread session setting, the Slack routing and thread-context gates still match the reported limitation, and the linked implementation PRs were closed unmerged after explicit trust-boundary concerns rather than replaced by a shipped fix.

## What This Changes

The issue requests an opt-in `session.threadIsolation: false` setting so Slack thread replies can reuse a parent or shared session while preserving Slack thread metadata, per-thread history behavior, and isolated defaults.

## Best Possible Solution

Keep isolated defaults, then design an additive shared-session model with explicit trust boundaries. A future Slack implementation should align inbound and outbound session keys, key thread history by Slack thread identity, track first-seen thread state independently from session existence, preserve `MessageThreadId`, and update config types, schema/help/labels, docs, and focused regression tests.
