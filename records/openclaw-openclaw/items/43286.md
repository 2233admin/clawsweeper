## Summary

Keep open. Current main still routes Slack thread replies to thread-suffixed session keys, the requested shared-thread config surface is absent, and the linked implementation PRs were closed unmerged. The remaining choice is not conservative cleanup because it changes Slack trust boundaries and should be decided with maintainer product/security review.

## What This Changes

Add an opt-in Slack shared-thread session mode so Slack thread replies can reuse the parent or main session while preserving thread targeting and per-thread history context.

## Best Possible Solution

Keep isolated defaults, then design an additive shared-session model with explicit Slack trust boundaries. The implementation should align inbound and outbound session keys, key history by channel/thread identity rather than shared session key, track first-seen thread state independently from session existence, preserve MessageThreadId for reply targeting, document leakage tradeoffs, and add focused routing/config/docs tests.

## Reproduction Assessment

Yes. A static reproduction is high-confidence: current Slack thread replies flow through resolveSlackRoutingContext into resolveThreadSessionKeys without useSuffix:false, and existing tests still assert the :thread:<threadTs> session suffix.

## Solution Assessment

Unclear. The requested threadIsolation:false flag is plausible, but prior discussion and the broader #19929 tracker show this is a trust-boundary design choice; a safer solution may need explicit channel/thread groups or verified single-identity scoping instead of one broad switch.

## Risks / Open Questions

- A broad shared Slack session can expose context from one thread, channel, or user to another if the opt-in boundary is not explicit enough.
- A flat boolean may not be precise enough for mixed workspaces where only selected channels, trusted teams, or single-identity deployments should share context.
- Inbound and outbound key alignment, first-seen thread tracking, and per-thread history keys must be handled together or replies can target correctly while the model receives the wrong context.
