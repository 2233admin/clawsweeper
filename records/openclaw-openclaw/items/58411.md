## Summary

Keep open. Current main still has the slash-command/tool asymmetry: `/acp spawn --bind here` can bind the current Discord conversation in place, but the ACP `sessions_spawn` path only accepts `thread: true` and resolves Discord to child-thread placement. The existing tests explicitly expect a new child thread for the Discord ACP `sessions_spawn` path even when the requester context already has a thread id, so the report is still valid and narrow.

## What This Changes

Add a programmatic `sessions_spawn` option to bind an ACP session to the current Discord conversation or thread in place, matching `/acp spawn --bind here`.

## Best Possible Solution

Add an explicit, additive ACP current-conversation binding mode to `sessions_spawn`, make it mutually exclusive with `thread: true`, reuse the existing session-binding `current` placement semantics, and document the difference between current binds and child-thread binds.

## Reproduction Assessment

Yes. A high-confidence source/test reproduction exists: `src/agents/acp-spawn.test.ts` currently calls `spawnAcpDirect` from a Discord thread context with `thread: true` and asserts child-thread placement and delivery to a new child thread.

## Solution Assessment

Yes, with API naming review. An explicit current-bind option is the narrowest maintainable fix because overloading `thread: true` would blur existing child-thread semantics, while the slash-command path already proves current-placement binding is supported.
