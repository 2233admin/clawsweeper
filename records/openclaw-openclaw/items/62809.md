## Summary

Keep open. Current main still ties OpenAI prompt-cache routing to the per-run model-call session identity, while isolated cron intentionally creates a fresh transcript/session id for every run; that preserves isolation but does not provide the stable cross-run cache-affinity identity requested here.

## What This Changes

The issue requests stable cross-run OpenAI prompt-cache affinity for recurring isolated cron jobs without making those jobs reuse conversation context.

## Best Possible Solution

Add an explicit cron prompt-cache affinity path that derives a stable key from the cron job identity and selected OpenAI-family model, passes it to OpenAI transports separately from `sessionId`/`sessionKey`, and keeps isolated cron transcripts fresh per run.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to run the same isolated cron job twice: current main forces a new cron session id for each isolated run, and OpenAI `prompt_cache_key` is sourced from that per-run session identity rather than a stable job identity.

## Solution Assessment

Yes for the requested direction. The maintainable solution is a narrow cache-affinity seam, not changing isolated cron to reuse persistent sessions or conversation history.
