## Summary

Keep open. Current main still has the exact observe-only active embedded run recovery path described in the report, and QQBot C2C streaming is still wired through awaited `stream_messages` callbacks. No relevant code changed between the reported release and current main, so this is not implemented or superseded.

## What This Changes

Fix QQBot C2C streaming so slow/rate-limited streaming replies do not leave a session permanently frozen when stuck-session recovery sees an active embedded run.

## Best Possible Solution

Add a QQBot-focused streaming recovery path that prevents slow `stream_messages` sends from holding the session lane indefinitely, with bounded finalization/abort behavior and tests for slow stream sends. Keep any core active-run abort policy conservative, explicit, and covered by diagnostics tests before making it default behavior.

## Reproduction Assessment

Yes at the code-path level: current tests and source reproduce the reported `active_embedded_run` observe-only recovery skip when a processing session has an active embedded run. Full live reproduction of QQ's slow `stream_messages` cadence still needs a QQBot environment.

## Solution Assessment

Unclear for a final implementation because the issue suggests both core recovery and QQBot batching changes. The narrowest maintainable direction is to fix QQBot streaming backpressure/finalization first, and only add configurable core active-run abort behavior if the channel-local fix is insufficient.
