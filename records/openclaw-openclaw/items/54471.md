## Summary

Keep open. The PR targets a real ACP parent-streaming gap, but the current branch is not a complete fix: it listens for a `system_event` agent stream while current ACP runtime progress/clarifying messages are surfaced as `status` events that are dropped before `onAgentEvent`, and the added relay path does not update stall/resume progress state.

## What This Changes

The PR adds a `system_event` branch in `startAcpSpawnParentStreamRelay()` so non-empty text can be emitted to the parent session with a fallback ACP context key.

## Best Possible Solution

Update this branch in place so ACP runtime `status` or equivalent progress events are translated into the parent relay contract, forward only non-empty user-safe text, update the same stall/resume state used for assistant output, and add focused regression coverage for `streamTo: "parent"` status/system progress delivery.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an ACP `runTurn` callback receiving `{ type: "status", message: ... }` during a `streamTo: "parent"` spawn: current `agent-command.ts` returns before emitting it, and the parent relay only receives assistant/lifecycle streams on current main.

## Solution Assessment

No. The PR is pointed at the right relay surface, but it is not the narrowest complete fix because it listens for a currently unproduced agent stream instead of wiring the actual ACP runtime status source and counting that output as relay progress.
