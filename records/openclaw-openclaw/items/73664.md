## Summary

Keep this issue open because current main still does not discover llama.cpp runtime context from `/props.n_ctx`, and the GitHub context shows open PR #74057 with closing syntax for the same fix. The issue should stay open until that PR, or an equivalent guarded self-hosted provider change, lands.

## What This Changes

The issue requests that llama.cpp/OpenAI-compatible self-hosted sessions use the actual runtime context window (`n_ctx`) instead of the model training window (`n_ctx_train`) when showing session token percentages.

## Best Possible Solution

Review and land PR #74057, or an equivalent narrow change, so self-hosted OpenAI-compatible discovery can record a positive llama.cpp `/props.n_ctx` runtime cap as `contextTokens` while keeping explicit configured caps authoritative and preserving native context metadata separately.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a llama.cpp server that exposes only `meta.n_ctx_train` from `/v1/models` and `n_ctx` from `/props` will not have its runtime cap discovered on current main because the discovery path only calls `/models`.

## Solution Assessment

Yes. Keeping this issue open while reviewing the linked implementation PR is the best path because the remaining work is narrow, already paired to a closing PR, and belongs in self-hosted provider discovery rather than in the sessions table renderer.
