## Summary

Keep this issue open. Current main has the right abstractions for separating native `contextWindow` from effective runtime `contextTokens`, but it does not yet implement either requested fix for llama.cpp: it does not query llama.cpp `/props` for `n_ctx`, and it does not preserve an existing agent `models.json` context limit as an authoritative runtime cap during provider regeneration. The sessions denominator still comes from persisted or resolved `contextTokens`, so a bad live/discovered value can still make the table misleading.

## Best Possible Solution

Keep the issue open and fix the provider/context-resolution path. The best implementation is to store and prefer an effective runtime `contextTokens` for llama.cpp/OpenAI-compatible local providers: either fetch llama.cpp `/props` and use `n_ctx`, or preserve an explicit configured/runtime cap over live `/v1/models` native metadata. Add focused coverage that a model with `contextWindow`/`n_ctx_train` of 262144 but runtime `contextTokens`/`n_ctx` of 65536 produces `56k/65k`-style session usage rather than `56k/262k`.
