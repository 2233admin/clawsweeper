## Summary

Keep this issue open. Current main still has the generic contextTokens/contextWindow plumbing and manual runtime-cap support, but it does not discover llama.cpp `/props.n_ctx`, and the sessions surfaces still render whatever contextTokens are stored or resolved from config/model metadata.

## Best Possible Solution

Keep this issue open and fix provider/context resolution. For llama.cpp-style OpenAI-compatible providers, add a guarded discovery path that strips `/v1` when needed, queries `/props`, reads positive `n_ctx`, stores/resolves it as `contextTokens`, keeps `contextWindow` as native/training metadata, and preserves explicit configured runtime caps first. Add regression coverage for `/v1/models` returning `meta.n_ctx_train: 262144` while `/props` returns `n_ctx: 65536`, then verify sessions show about `56k/65k` rather than `56k/262k`.
