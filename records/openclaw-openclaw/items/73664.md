## Summary

Keep this issue open. Current main still discovers self-hosted OpenAI-compatible models through `/models` only and does not read llama.cpp `/props.n_ctx`; session display then uses stored/configured/catalog context values. The provided issue context also shows #74057 is an open, unmerged PR with closing syntax for this exact bug, so the right maintainer action is to review that PR rather than close early.

## What This Changes

The issue asks OpenClaw to show the llama.cpp server's actual runtime context window in session token usage instead of the model training context window.

## Best Possible Solution

Review and land #74057 or an equivalent guarded self-hosted provider discovery change that strips a trailing `/v1`, reads a positive llama.cpp `/props` runtime context as `contextTokens`, keeps explicit configured caps authoritative, and adds regression coverage for `/v1/models` reporting a larger training context than the loaded runtime context.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: a llama.cpp-compatible server whose `/v1/models` metadata exposes only the training context will not have its runtime `/props.n_ctx` discovered on current main, and the sessions table then resolves its denominator from stored/configured/catalog context values.

## Solution Assessment

Yes. The best path is to keep this issue open while maintainers review #74057 or the same narrow provider-discovery fix; closing before the linked implementation merges would leave the reported runtime-context mismatch unresolved.
