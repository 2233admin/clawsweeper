## Summary

Keep open. Current main still has no file-scan fallback for `memory_search` when `node:sqlite` is unavailable. The builtin manager opens a SQLite database before search can run; if that fails, the tool returns a disabled/unavailable payload rather than scanning `MEMORY.md` and `memory/*.md`. The related #69199/#59457 work is explicitly scoped to better error messaging, not the fallback requested here.

## Best Possible Solution

Keep #59451 open and implement the fallback in `extensions/memory-core`: when builtin manager creation/search fails specifically because `node:sqlite` is unavailable, scan `MEMORY.md` and `memory/*.md` with the existing memory-file discovery helpers, return ranked keyword snippets, and include explicit response metadata that the results came from degraded file-based search rather than vector/FTS search. Keep #59457/#69199 focused on error-message UX.
