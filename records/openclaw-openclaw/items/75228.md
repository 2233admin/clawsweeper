## Summary

Keep this PR open. The updated branch narrows the earlier security concern by making arbitrary system-trusted stale-pin repair opt-in, but the new opt-in is introduced only as a raw macOS config read; the shared config schema and public config docs still reject unknown `gateway.remote` keys, so the advertised flag is not a supported configuration surface yet.

## What This Changes

The PR adds an opt-in macOS config flag for system-trusted stale gateway TLS pin repair on arbitrary `wss://` hosts, threads that flag through the macOS coordinator, adds Swift coverage, and adds a changelog entry for #74848.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, and if the opt-in policy is accepted, make `gateway.remote.allowSystemTrustedPinRepair` a first-class config key across the shared type, zod schema, labels/help metadata, generated config docs baseline, public docs, and validation tests before merging the macOS behavior change.

## Reproduction Assessment

Yes at code level. A `GatewayTLSValidationFailure(kind: .pinMismatch, storeKey: host:443, systemTrustOk: true)` for a non-loopback, non-`.ts.net` `wss://` URL returns false on current main; the PR’s remaining defect is reproducible by validating config containing `gateway.remote.allowSystemTrustedPinRepair` against the current strict schema.

## Solution Assessment

Not yet. The opt-in approach is safer than silently broadening the default pin repair policy, but this PR is incomplete until the opt-in is supported by the canonical config schema/docs and validation surfaces.
