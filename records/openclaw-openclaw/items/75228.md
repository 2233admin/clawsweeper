## Summary

Keep open. The PR is a useful default-off opt-in direction for the macOS stale TLS pin problem, but the new `gateway.remote.allowSystemTrustedPinRepair` key is not registered in the canonical config contract, so operators enabling it would create schema-invalid OpenClaw config while the macOS app reads the key directly.

## What This Changes

This PR adds a macOS `gateway.remote.allowSystemTrustedPinRepair` opt-in for stale TLS pin auto-repair on arbitrary system-trusted `wss://` hosts, threads it through the coordinator helper, adds Swift tests, and updates the changelog.

## Best Possible Solution

Keep the default loopback/`.ts.net` repair boundary, and if the opt-in policy is accepted, make `gateway.remote.allowSystemTrustedPinRepair` a first-class config key across the shared type, zod schema, labels/help metadata, generated config docs/schema baselines, public docs, and validation tests before merging.

## Reproduction Assessment

Yes. At code level, current main returns false for a non-loopback, non-`.ts.net` `wss://` pin mismatch even with `systemTrustOk: true`; the PR defect is reproducible by inspecting the strict `gateway.remote` schema, which omits the new key that the macOS app reads.

## Solution Assessment

Not yet. The default-off opt-in is safer than silently broadening stale-pin repair, but the PR is incomplete until the new config flag is supported by the canonical config contract and docs surfaces.
