## Summary

Keep open: current main still does not provide the requested server-side WebChat dictation path. WebChat can attach audio files and run realtime Talk, and the media-understanding backend can transcribe audio attachments, but there is no MediaRecorder-based microphone flow or Gateway method that transcribes audio and returns text to the draft without starting an agent run.

## What This Changes

Add a server-side speech-to-text option for WebChat voice input by recording browser audio and sending it to Gateway for configured audio transcription instead of relying on browser SpeechRecognition.

## Best Possible Solution

Implement a focused Control UI/WebChat batch dictation flow: capture microphone audio with getUserMedia plus MediaRecorder, send the completed recording through an authenticated bounded Gateway method, transcribe it through existing tools.media.audio media-understanding plumbing, and append the returned text to the draft while keeping realtime Talk and normal file attachments separate.

## Reproduction Assessment

Not applicable as a bug reproduction; this is a feature request. The high-confidence verification path is current-main inspection: no MediaRecorder path or server-side draft transcription method exists, and the UI test asserts the stale Voice input control is absent.

## Solution Assessment

Yes, the requested direction is the narrow maintainable solution. A dedicated batch dictation method is safer than repurposing chat.send because it can return transcript text without starting an agent run while still reusing existing audio transcription limits and provider/CLI configuration.
