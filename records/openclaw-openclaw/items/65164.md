## Summary

Keep open. Current main has useful partial mitigations for the startup pain described here, including a bounded/non-blocking primary-model prewarm, a startup-prewarm skip env var under the current name, optional model pricing, Telegram Bot API timeout/fallback hardening, Discord REST metadata timeouts, and delayed Gateway WebSocket client startup until the event loop is responsive. Those changes do not prove the broader native Windows gateway-process HTTP failure is fixed: the gateway network bootstrap still only applies env-proxy dispatcher setup, and there is no source, test, or shipped behavior proving Telegram REST, Discord REST, pricing HTTP, and local approval/client handshakes all work together under the reported Windows multi-provider load.

## What This Changes

The issue asks OpenClaw to fix a native Windows gateway-process failure where outbound HTTP/HTTPS calls hang or fail while WebSocket surfaces keep working, and requests a model-prewarm skip workaround.

## Best Possible Solution

Keep the issue open for a focused Windows gateway networking investigation: reproduce on native Windows with Node 22/24 and multiple Discord plus Telegram providers, instrument global undici dispatcher state, active handles, DNS/connect/TLS timing, and channel startup concurrency, then fix the process-wide failure without weakening SSRF/proxy boundaries. The existing startup-prewarm and pricing-disable mitigations should remain as bounded workarounds, with regression coverage proving channel startup is not blocked and Telegram REST, Discord REST, model-pricing/catalog HTTP, and local Gateway client/approval handshakes stay healthy in the same process.
