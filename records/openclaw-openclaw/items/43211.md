## Summary

Current main still does not implement the PR's requested hyphen-stripped allowlist behavior, so the PR is not obsolete. However, the patch should stay open for rework rather than be closed or merged as-is because it broadens a model authorization allowlist across provider IDs that current code treats as distinct unless an explicit alias maps them together.

## Best Possible Solution

Keep this PR open for maintainer or contributor rework. The best path is to provide a concrete reproduction for the runtime path that rewrites a hyphenated provider ID, then either fix that specific normalization path or express the equivalence through explicit provider/plugin alias metadata. Regression coverage should prove known aliases resolve while distinct providers such as `my-provider` and `myprovider` remain separately allowlisted unless explicitly linked.

## Risks / Open Questions

- Merging the PR as written would allow `myprovider/some-model` when only `my-provider/some-model` was configured, even though current main permits those provider IDs to be distinct.
- The PR body asserts that a runtime path strips provider hyphens, but the provided discussion and current main inspection do not identify a concrete reproduction path for that rewrite.
- The proposed tests cover the broadened behavior but not collision cases where hyphenated and non-hyphenated provider IDs are intentionally separate.
- Security review: the PR does not touch supply-chain-sensitive files, but it does touch the model authorization boundary, so the unexplained allowlist expansion is the primary security concern.
