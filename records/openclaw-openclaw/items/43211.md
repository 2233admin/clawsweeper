## Summary

Keep this PR open. Current main still does not implement hyphen-stripped provider allowlist variants, and the PR remains a plausible but security-sensitive fix candidate because unconditional `my-provider/model` -> `myprovider/model` expansion can authorize a distinct provider ID.

## Best Possible Solution

Keep this PR open for maintainer or contributor rework. The best path is to add a concrete reproduction for the runtime path that strips provider hyphens, then either fix that resolver directly or model the equivalence through explicit provider/plugin alias metadata. Regression coverage should prove declared aliases work while distinct providers such as `my-provider` and `myprovider` remain separately allowlisted unless explicitly linked.

## Risks / Open Questions

- Closing this PR would drop a focused fix candidate while current main still has the exact-key allowlist behavior described in the PR.
- Merging the PR as written could broaden the model authorization boundary by treating distinct provider IDs such as `my-provider` and `myprovider` as equivalent without explicit provider/plugin alias metadata.
- The reported runtime path that strips provider hyphens still needs a concrete reproduction or failing test; without that, it is unclear whether the right fix is allowlist aliasing or correcting the upstream resolver that changes the provider ID.
