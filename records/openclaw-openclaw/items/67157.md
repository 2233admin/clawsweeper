## Summary

Keep this PR open for maintainer review. Current main still uses `openclaw.mjs` as the package bin, and the PR is not obsolete, but the proposed bin-target switch conflicts with the release metadata guard and leaves startup, gateway-lock detection, and live-smoke reliability gaps that should block merge until the entrypoint plan is made coherent.

## What This Changes

The PR changes the published npm `openclaw` bin to `scripts/openclaw-runcli-launcher.mjs`, packages that launcher, adds local tarball and live memory-pro host upgrade checks, and adds launcher e2e coverage.

## Best Possible Solution

Revise the PR into a complete package-entrypoint plan: either keep `openclaw.mjs` canonical and add tarball/memory-pro regression coverage around that path, or finish the new bin-target transition by preserving entry bootstrap behavior, updating release guards/tests, aligning gateway argv detection, and hardening the new live/local scripts.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Update the release guard with the bin change:** `package.json:17`
  - body: `package.json` now points `bin.openclaw` at `scripts/openclaw-runcli-launcher.mjs`, but `collectReleasePackageMetadataErrors` still requires `openclaw.mjs`. The branch will fail `pnpm release:openclaw:npm:check` until the release contract and tests are updated, or the bin change is reverted.
  - confidence: 0.96
- **[P1] Preserve the CLI entry bootstrap:** `scripts/openclaw-runcli-launcher.mjs:15-21`
  - body: The new launcher imports `runCli` directly from a generated chunk and exits, bypassing the existing `src/entry.ts` path. That skips startup behavior such as respawn/env normalization, warning filtering, profile/container argv handling, Windows argv normalization, and root fast paths for every npm-installed invocation.
  - confidence: 0.92
- **[P2] Align gateway detection with the new bin target:** `package.json:17`
  - body: Changing the npm bin target can make process command lines expose `scripts/openclaw-runcli-launcher.mjs`, but `isGatewayArgv` only recognizes the current entry paths such as `openclaw.mjs`. Update the gateway argv classifier and tests with the bin transition so a running gateway is not mistaken for a stale process.
  - confidence: 0.86
- **[P2] Run the live smoke against the validated bin:** `scripts/local-upgrade-from-tarball.sh:134`
  - body: The script verifies `$BIN_PATH` resolves to the installed launcher, then runs the regression helper without `OPENCLAW_BIN`. The helper defaults to `openclaw` from `PATH`, so another install can be tested instead of the tarball that was just installed.
  - confidence: 0.9
- **[P2] Await main in the live regression:** `scripts/test-live-memory-pro-cli-host.mjs:271`
  - body: `main()` is async, but the top-level `try/catch` calls it without `await`. Any async failure bypasses the structured JSON error path and surfaces as an unhandled rejection instead, making CI/live diagnostics much worse.
  - confidence: 0.94
- **[P2] Parse stats JSON from stdout first:** `scripts/test-live-memory-pro-cli-host.mjs:84`
  - body: `combined` puts stderr before stdout, and `extractFirstJsonObject` takes the first `{` it sees. A diagnostic object on stderr can be parsed instead of the `memory-pro stats --json` payload, so assemble stdout first or parse stdout directly.
  - confidence: 0.82
- **[P2] Accept mjs run-main chunks:** `scripts/openclaw-runcli-launcher.mjs:9`
  - body: The launcher only searches for `run-main-*.js`, while existing startup code treats built entry extensions as `.js` or `.mjs`. If the build emits an `.mjs` run-main chunk, the installed `openclaw` command will fail with `missing run-main entry`.
  - confidence: 0.78
