## Summary

Keep open: the PR addresses a real config-schema gap, but the patch is incomplete because provider-level native Vertex configs would be accepted by validation and then lose their `google-vertex` API before runtime stream selection.

## What This Changes

The PR adds `google-vertex` to the model API enum/generated config schema and adds tests for native Google Vertex config validation and Google provider normalization.

## Best Possible Solution

Revise the PR so `google-vertex` is accepted consistently by config validation and runtime model materialization, especially provider-level `models.providers.google-vertex.api` inheritance, with a focused inline-provider test proving resolved models keep `api: "google-vertex"`.

## Reproduction Assessment

Yes. A source-backed reproduction path is a config with `models.providers.google-vertex.api: "google-vertex"` and models that omit per-model `api`; after the PR accepts that config, `buildInlineProviderModels` still drops the provider-level API.

## Solution Assessment

No. Adding the enum and regenerated schema is necessary, but the maintainable fix also needs the runtime normalization allowlist and provider-level inheritance coverage so accepted configs reach the native Vertex transport.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve provider-level Vertex APIs:** `src/config/types.models.ts:15`
  - body: Adding `google-vertex` to `MODEL_APIS` makes provider-level `api: "google-vertex"` pass validation, but `normalizeResolvedTransportApi` still filters that value out. When models inherit the API from the provider, `buildInlineProviderModels` returns `api: undefined`, so the Google plugin never selects the native Vertex ADC stream. Add the runtime allowlist entry and cover provider-level inheritance.
