## Summary

Keep this issue open. Current main has adjacent deleted-agent cleanup and runtime guards, but the Control UI Sessions tab still consumes and renders the broad Gateway `sessions.list` result, which intentionally includes disk-discovered agent stores. The browser restore path also still preserves explicit non-default `agent:<id>:*` session selections rather than validating them against configured agents.

## Best Possible Solution

Keep this issue open. Add a Control UI-specific configured-agent boundary: either filter the Sessions tab against `agents.list`, or add an additive `sessions.list` option intended for UI rows while preserving broad Gateway/ACP discovery. Also normalize or reject restored `agent:<id>:*` session keys when `<id>` is no longer configured, falling back to the Gateway session default, with regression tests for both the Sessions tab and browser/URL restore path.

## Risks / Open Questions

- A global change to make Gateway `sessions.list` configured-only would conflict with documented Gateway/ACP disk-only discovery and the #32804 regression test; the safer fix is a Control UI-specific boundary or additive protocol option.
