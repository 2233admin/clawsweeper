## Summary

Keep open. Current main still gates Telegram DM topic renaming on `!entry?.systemSent`, which matches the reporter’s likely regression path when bootstrap/session state exists before the first meaningful user turn; I found no current test or merged follow-up that covers that case.

## What This Changes

The issue requests restoring Telegram DM topic auto-label renames so new DM topics do not keep their provisional fallback title after the first user message.

## Best Possible Solution

Keep the issue open and fix this inside the Telegram plugin by decoupling “needs first auto-label attempt” from `systemSent`, with a focused regression test for a DM-topic session entry that already has bootstrap state before the first user message.

## Reproduction Assessment

Yes. A focused test can seed `loadSessionStore` with `s1: { systemSent: true }`, dispatch a Telegram DM-topic first user message with `RawBody`, and verify current main skips `generateTopicLabel` and `editForumTopic` because the gate is false.

## Solution Assessment

Yes for a narrow plugin-local repair; no core migration or product redesign is needed. The maintainable fix is to track whether the Telegram DM topic label was attempted or completed separately from `systemSent`, then keep the existing config resolution and model-labeling path intact.

Fix OpenClaw issue https://github.com/openclaw/openclaw/issues/62303. Telegram DM topic `autoTopicLabel` currently depends on `!entry?.systemSent`, so newly-created DM topics whose session entry already has bootstrap/system state skip the generated rename and keep the provisional fallback title. Keep the fix inside the Telegram plugin boundary. Do not move this into core, do not change the `autoTopicLabel` default, do not change provider/model selection, and do not add startup/load-time migrations. Add focused tests in `extensions/telegram/src/bot-message-dispatch.test.ts` proving a DM topic with `SessionKey: "s1"`, `RawBody`, and a session store entry like `{ systemSent: true }` still performs exactly one auto-label attempt and calls `bot.api.editForumTopic`; also cover that established/already-labeled sessions do not relabel every turn. Prefer a Telegram-owned durable marker on the session entry, such as under `pluginExtensions`, or an equivalent local state pattern that fits existing session-store conventions.
