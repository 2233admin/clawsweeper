## Summary

Keep this PR open: current main still has the Nextcloud Talk reaction dispatch gap, but the proposed patch is not ready because it ignores shared reaction removal semantics while reporting success for an add.

## What This Changes

This PR adds and registers a Nextcloud Talk message action adapter, adds adapter and reaction sender tests, and adds a changelog entry for the Nextcloud Talk reaction bug.

## Best Possible Solution

Land this PR or an equivalent Nextcloud Talk-owned adapter after it either implements Nextcloud Talk reaction deletion or explicitly rejects unsupported removal requests, keeps focused adapter/sender coverage, and then closes #70110 while resolving the duplicate #72679 path.

## Reproduction Assessment

Yes. On current main, a Nextcloud Talk `react` action selects the channel but `dispatchChannelMessageAction` returns unsupported because the plugin has no `actions.handleAction`; on the PR branch, a direct `react` call with `remove: true` would still call the add sender and return `added`.

## Solution Assessment

No, not as currently proposed. Registering a channel action adapter is the right narrow seam, but the patch should handle or reject `remove: true` before merge because the shared `react` action already exposes removal semantics.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Handle remove reactions before sending:** `extensions/nextcloud-talk/src/message-actions.ts:60-68`
  - body: `remove` is part of the shared `react` action contract, but this handler ignores it and always calls `sendReactionNextcloudTalk`, which POSTs an added reaction and returns `{ added: emoji }`. A call with `emoji: "👍", remove: true` would perform the opposite of the requested action, so implement deletion or reject removal explicitly before sending.
  - confidence: 0.88
