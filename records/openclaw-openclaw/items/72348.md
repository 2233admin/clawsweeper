## Summary

Keep open: current main still advertises Nextcloud Talk reactions without registering a message-action adapter, so the PR targets a real bug. The PR should not merge as-is because its new handler ignores the shared `remove` flag and would add a reaction when the caller requested removal; related PR #72679 also targets #70110, so one canonical fix path should be reconciled.

## What This Changes

The PR adds and registers a Nextcloud Talk message-action adapter, adds adapter and reaction-sender unit coverage, and adds a changelog entry for #70110.

## Best Possible Solution

Land one canonical Nextcloud Talk-owned adapter for #70110 after it preserves the shared reaction contract by handling or rejecting removal requests before POST, keeps the existing SSRF-guarded sender path, and includes focused adapter and sender coverage.

## Reproduction Assessment

Yes. On current main, dispatching a Nextcloud Talk `message` tool `react` action reaches a plugin with no `actions.handleAction`, so dispatch returns unsupported/null despite the reaction capability. On the PR head, calling the new handler with `remove: true` still calls the add-reaction sender and returns `added`.

## Solution Assessment

No, not as currently proposed. Registering a plugin-owned `ChannelMessageActionAdapter` is the right narrow seam, but the handler must not perform an add operation for a removal request; explicit rejection is the conservative fix if DELETE support stays out of scope.
