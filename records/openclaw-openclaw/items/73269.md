## Summary

Keep open. Current main has partial visible-reply controls that reduce this class of narration leaks, but it still does not implement the requested per-agent auto-reply disablement, explicit public/internal audience boundary, or final-only source delivery rule; the related broader boundary report remains open.

## What This Changes

The issue asks OpenClaw to stop automatically delivering assistant narration text to channels when the same assistant turn continues with tool calls, either via a per-agent auto-reply directive or by delivering only final assistant text.

## Best Possible Solution

The durable end state is a shared visible-delivery contract that consistently separates internal assistant text from public source replies across final payloads, block and preview streaming, transcript visible-text extraction, and explicit message-tool sends, with documented per-agent or per-source configuration layered on top.

## Reproduction Assessment

Partly. Current source and unit tests give a high-confidence reproduction of the residual policy gap: direct chats still default to automatic delivery while group/channel turns suppress automatic delivery; the exact text-before-tool-call Feishu sequence still needs a focused synthetic or live fixture.

## Solution Assessment

No. The current messages.visibleReplies mechanism is a useful partial mitigation, but it is not the best complete solution because it lacks per-agent control and an explicit public/internal or final-only assistant text boundary.
