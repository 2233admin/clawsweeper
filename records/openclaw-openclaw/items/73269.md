## Summary

Keep open. Current `main` still provides only a partial mitigation: group/channel turns default to message-tool-only delivery, but direct/source chats still auto-deliver unless globally configured, and there is still no per-agent auto-reply control or public/internal assistant audience boundary. The related #74928 remains open and overlaps the broader contract decision.

## What This Changes

The issue requests suppressing automatic channel delivery for assistant text when tool calls follow, either with per-agent auto-reply control or final-only/public-delivery semantics.

## Best Possible Solution

Define one shared visible-delivery contract that can distinguish public assistant text from internal/runtime text and apply it consistently across preview, block, tool-progress, and final delivery. If maintainers choose #74928 as canonical, preserve this issue’s Feishu/direct-chat reproduction context there before closing this as superseded.

## Reproduction Assessment

Partly. Current source and tests give a high-confidence reproduction of the remaining policy gap: direct chats default to automatic delivery while message-tool-only mode suppresses automatic callbacks and finals; the exact Feishu text-then-tool-call leak still needs a focused fixture or live check.

## Solution Assessment

No. The shipped `messages.visibleReplies` mitigation is useful, but it is not the complete requested solution because it is global/direct opt-in, not per-agent, and it does not add an explicit public/internal or final-only assistant text boundary.
