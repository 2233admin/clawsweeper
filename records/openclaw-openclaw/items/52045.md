## Summary

Keep open. Current main still has a credible browser/gateway state-divergence path after channel setup: `openclaw browser status` goes through `browser.request`, and that handler can return `UNAVAILABLE` when the browser control service gate decides the bundled browser plugin is disabled, even though the gateway process itself is healthy. The related fix PR #52057 is closed unmerged, and the current tests do not cover the Feishu-pairing plus restrictive plugin allowlist scenario described by the reporter.

## What This Changes

The issue reports that `openclaw gateway status` shows a healthy gateway but `openclaw browser --browser-profile openclaw status` reports the gateway closed after Feishu pairing on a fresh Ubuntu install.

## Best Possible Solution

Keep the issue open and land a narrow browser-owned fix that makes the browser control service use the same effective activation semantics documented for root `browser` config, then make `openclaw browser status` report a precise browser-plugin disabled or unauthorized condition instead of implying the gateway itself is down. Add regression coverage for browser status with Feishu enabled and a restrictive plugin allowlist.

## Reproduction Assessment

Yes, for the likely config-level failure path: set up browser config, enable Feishu, and leave `plugins.allow` restrictive without `browser`; current main's browser status path reaches the control-service gate that can return null while the gateway remains healthy. The full live Feishu QR/pairing flow still needs a fresh run to confirm the exact config mutation.

## Solution Assessment

No final patch is present. The closed PR's endpoint-level status check is not sufficient by itself because the observed failure can occur before the status route dispatches, when `startBrowserControlServiceFromConfig` returns null.
