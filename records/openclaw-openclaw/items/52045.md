## Summary

Keep open. Current main still has a real mismatch between plugin activation and the browser control-service gate: setup auto-enable can load the bundled browser plugin under a restrictive plugins.allow, but startBrowserControlServiceFromConfig still checks the raw effective enable state and can return null. That makes openclaw browser status depend on browser.request and report browser control unavailable even while the gateway itself is healthy. The reporter’s Feishu-pairing trigger remains plausible because Feishu setup changes channel config and can interact with restrictive plugin allowlists, while no current regression test covers browser status after channel setup.

## Best Possible Solution

Keep this issue open and land a focused regression fix. The browser control-service gate should use the same resolved/auto-enabled plugin activation semantics that load the browser plugin, so root browser config or browser setup auto-enable keeps browser control available even under restrictive plugins.allow. The CLI/gateway error should distinguish browser control disabled from gateway closed, and tests should cover browser.enabled=true plus plugins.allow omitting browser, plus a channel setup path such as Feishu that leaves browser status working.

## Clownfish Work Prompt

Fix the browser status regression tracked by this issue: after browser setup, pairing/configuring Feishu can leave `openclaw gateway status` healthy while `openclaw browser --browser-profile openclaw status` reports the gateway/browser control as closed or disabled.

Expected boundary:
- Keep the fix in the bundled browser/plugin activation boundary. Do not add core special cases for Feishu beyond a regression test if needed.
- Make `startBrowserControlServiceFromConfig()` / `isDefaultBrowserPluginEnabled()` honor the same setup auto-enable semantics that load the browser plugin: root `browser` config such as `browser.enabled=true` or `browser.profiles.*` should keep the bundled browser control service available even when `plugins.allow` is restrictive and omits `browser`, unless the browser plugin or browser config is explicitly disabled.
- Preserve trust semantics: do not allow arbitrary non-bundled/workspace plugins to bypass `plugins.allow`.
- Improve or preserve precise errors so `openclaw browser status` does not imply the gateway process is down when only browser control is disabled.
- Add regression coverage for `browser.enabled=true` plus restrictive `plugins.allow` omitting `browser`, and for a channel setup-style config path such as Feishu enabling `channels.feishu` while browser status/control service remains available.
- Consider whether docs/changelog need an entry. If adding a user-facing changelog fix, credit the reporter with `Thanks @ylx1314`; do not credit Codex/OpenClaw.
