## Summary

Keep open. The reporter added a concrete reproduction signal: browser status works after setup, then reports the gateway closed after Feishu pairing. Current main still routes `openclaw browser status` through `browser.request`, whose local path can return `UNAVAILABLE` when the browser control service is disabled even though the gateway itself is healthy. The related fix PR #52057 is closed unmerged, and I did not find regression coverage for browser status after Feishu setup/pairing.

## Best Possible Solution

Keep this issue open. Reproduce fresh install plus browser setup plus Feishu pairing on current main, especially with any restrictive `plugins.allow` created or preserved during setup. Then align the browser control-service enablement gate with the manifest/root-config activation behavior, make `openclaw browser status` report the precise disabled/unauthorized condition instead of implying the gateway is closed, and add a regression test for browser status after channel setup.
