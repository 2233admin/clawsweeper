## Summary

Keep open. Current main still preserves only a single caller-supplied active session key during enforce-mode session maintenance; stale pruning, maxEntries capping, load-time maintenance, and disk-budget eviction still do not skip every persisted session with status "running". The related PR #59932 remains the closest candidate, but it is still open and not implemented on main.

## Best Possible Solution

Keep this issue open and use it as the tracking bug until a focused core fix lands. The best fix is a shared protected-session-key source that combines the caller activeSessionKey with all currently active/running sessions, using persisted/in-memory subagent state where needed, then threads that set through stale pruning, maxEntries capping, load-time maintenance, and disk-budget eviction. Review/land a corrected #59932 or replace it with an equivalent patch plus regression tests for multiple running sessions under age, count, and disk-budget pressure.

## Risks / Open Questions

- Concurrent running subagent or cron-created sessions can still be evicted when another session save triggers enforce-mode maintenance and only that caller's activeSessionKey is preserved.
- Oversized-store load/restart paths can still drop running sessions before a later save persists the changed store.
- There is still no targeted regression test proving multiple status:"running" entries survive pruneAfter, maxEntries, and maxDiskBytes enforcement.
