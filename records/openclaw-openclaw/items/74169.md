## Summary

Keep this PR open for maintainer review: the generic timeout fallback change is directionally related, but it does not fix the linked WeCom media-send regression because the message-action path supplies an explicit 10000ms timeout before the fallback in `src/gateway/call.ts` can apply.

## What This Changes

The PR changes `src/gateway/call.ts` so unconfigured gateway calls resolve the shared 15000ms preauth handshake timeout instead of falling back to 10000ms.

## Best Possible Solution

Centralize or align the gateway-executed message-action timeout so slow plugin media sends do not inherit an unrelated 10-second wrapper default, while preserving explicit caller timeouts and adding focused coverage for both the generic gateway-call fallback and the CLI message-action path.

## Reproduction Assessment

Yes. The source-level reproduction path is `messageCommand` -> `runMessageAction` -> gateway-executed plugin action -> `callGatewayLeastPrivilege`, where current main passes `timeoutMs: 10_000`; a fake gateway plugin whose action takes longer than 10 seconds can reproduce the timeout without a live WeCom account.

## Solution Assessment

No. The PR changes only the fallback used when `opts.timeoutMs` is absent, but the linked media-send path reaches the gateway with an explicit 10000ms timeout from `src/infra/outbound/message-action-runner.ts`, so the proposed patch is incomplete for the claimed regression.
