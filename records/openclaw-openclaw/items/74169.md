## Summary

Keep open: the PR is still relevant, but it does not fully fix the linked WeCom media-send regression. It changes the generic gateway-call fallback, while the CLI message-action path still injects and forwards an explicit 10000ms timeout before reaching that fallback.

## What This Changes

The PR changes `src/gateway/call.ts` so calls without an explicit timeout resolve the shared preauth handshake timeout default of 15000ms instead of falling back to 10000ms.

## Best Possible Solution

Align the gateway-executed message-action timeout with the shared preauth timeout default while preserving explicit caller overrides, then add focused regression coverage for both no-config/no-env gateway calls and CLI plugin-dispatched media sends.

## Reproduction Assessment

Yes. The linked report provides a concrete `openclaw message send --channel wecom --media <10MB file>` failure, and current code shows a deterministic static reproduction path where `messageCommand` reaches `callGatewayMessageAction` with an explicit `timeoutMs: 10_000`.

## Solution Assessment

No. This PR changes only the fallback used when no timeout is passed to `callGatewayLeastPrivilege`, but the linked message-action path passes an explicit 10000ms timeout from `src/infra/outbound/message-action-runner.ts`.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P2] Fix the message-action timeout path:** `src/gateway/call.ts:394-396`
  - body: The linked CLI media send goes through `messageCommand` -> `runMessageAction` -> `callGatewayMessageAction`, where the runner still supplies `timeoutMs: 10_000` and forwards it to `callGatewayLeastPrivilege`. Because this change only applies when `opts.timeoutMs` is absent, gateway-executed plugin actions can still fail at 10000ms. Please align the message-action default or stop passing the hard-coded timeout.
