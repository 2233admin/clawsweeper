## Summary

Keep this PR open for maintainer review. Current main still has the generic no-config/no-env gateway wrapper fallback at 10 seconds, so the PR is not implemented on main, but the branch appears mis-scoped for linked #74164 because the CLI/plugin message-action path supplies an explicit 10,000ms timeout before the generic fallback is consulted.

## What This Changes

The PR modifies `src/gateway/call.ts` so `resolveGatewayCallTimeout` always resolves the shared pre-auth handshake timeout default, effectively raising generic no-config gateway-call wrapper timeouts from 10,000ms to 15,000ms.

## Best Possible Solution

Keep this PR open and ask maintainers to choose the intended target. If the goal is generic gateway-call alignment, simplify the ternary and add a regression test proving no env/config now times out at 15 seconds. If the goal is fixing #74164, retarget the timeout boundary in CLI/plugin `message.action` dispatch with focused tests, preserving the plugin boundary and avoiding WeCom-specific core logic.
