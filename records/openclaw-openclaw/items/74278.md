## Summary

Keep this PR open. Current `main` still has the loader fallthrough this PR targets, so the work is not obsolete, but the proposed patch is not merge-ready because it turns whitespace-prefixed, otherwise valid inbound media URIs into `not-found` instead of resolving them, and the linked sandbox image-tool path remains separately covered by open PR #74186.

## What This Changes

The PR adds guarded handling for unresolved `media://` and other scheme-bearing inputs in `loadWebMediaInternal`, plus regression tests for avoiding workspace path joins.

## Best Possible Solution

Normalize and resolve inbound media-store URIs before deciding they are stranded, then keep the defensive no-workspace-join guard for genuinely unresolved scheme-bearing values. The sandbox bridge path should either land through #74186 or be explicitly left as a separate follow-up so #74123 is not closed prematurely.

## Reproduction Assessment

Do we have a high-confidence way to reproduce the issue? Yes. Create an inbound media-store file and call `loadWebMedia("  media://inbound/<id>", { workspaceDir })`; the current source shows the strict precheck misses it, while the proposed test locks in rejection instead of successful hydration.

## Solution Assessment

Is this the best way to solve the issue? No. The patch is useful hardening, but the narrow maintainable fix should let the existing normalized media-reference resolver handle whitespace-prefixed valid `media://` URIs and coordinate separately with the sandbox resolver work in #74186.
