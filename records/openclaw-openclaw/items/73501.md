## Summary

Keep open. The report is fresh, corroborated by multiple users, and current main contains only partial mitigations and diagnostics rather than end-to-end proof that the 4.22/4.23 to 4.26/4.29 reply/reaction latency regression is fixed.

## What This Changes

The issue reports a significant OpenClaw upgrade regression where Feishu replies/reactions and other reply surfaces became much slower after moving from 4.22/4.23 to 4.26 or later with the same configuration.

## Best Possible Solution

Keep this as the broad regression tracker until maintainers measure the same redacted configuration across 4.22 or 4.23, 4.26, v2026.4.29, and current main, then split the dominant bottleneck into focused fixes for Feishu streaming delivery, embedded-run startup, plugin/runtime-deps loading, model/provider resolution, or Active Memory as the measurements dictate.

Yes. The discussion provides a concrete reproduction path: run the same Feishu/Lark or affected channel configuration across the fast and slow versions while recording reaction, first visible reply, final reply, pre-LLM, plugin-load, model-resolution, and embedded-run timings; this read-only review did not execute that live setup.

## Solution Assessment

No. A broad revert or package-size-based fix is not the best path; the maintainable solution is measured, layer-specific repair of the current hot path, using the existing embedded-run stage timing and gateway/plugin diagnostics to identify the bottleneck first.
