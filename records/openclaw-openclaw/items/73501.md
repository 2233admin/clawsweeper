## Summary

Keep this issue open. It is a same-day regression report with several corroborating comments, and current main does not contain a source-backed fix proving that the 2026.4.22-to-2026.4.26 Feishu reaction/reply slowdown is resolved. The current Feishu path still has plausible latency contributors: default sender-name lookup, default typing reaction calls, streaming-card startup/backoff behavior, and serialized inbound handling. Related Feishu/global slowness issues are useful context, but none safely supersedes this version-specific report.

## Best Possible Solution

Keep the issue open and triage it as part of the Feishu/global latency cluster. Ask reporters for a current-release or current-main retest with redacted config, connectionMode/domain, model/provider, gateway diagnostics, and timestamps for inbound receipt, debounce flush, sender lookup, typing reaction create/remove, model first token, streaming-card start/update/close, and final delivery. Have them test toggling channels.feishu.streaming, typingIndicator, and resolveSenderNames separately; then fix the measured owner path with a focused regression test or live proof.
