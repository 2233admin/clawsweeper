## Summary

Keep open. This is a same-day Feishu regression report comparing 2026.4.22 and 2026.4.26, with a second user corroborating broad slowness. Current main has related Feishu startup, timeout, streaming, typing, and sender-lookup controls, but I did not find a current-main fix, reproducible benchmark, or canonical duplicate that proves this specific 2-5s reaction/reply slowdown is resolved.

## Best Possible Solution

Keep this issue open and triage it with the Feishu latency cluster rather than auto-closing it. Ask for a current-release or current-main retest with redacted config, Feishu domain and connectionMode, provider/model choice, gateway logs or diagnostics export, and timings for inbound receipt, debounce flush, sender-name lookup, typing reaction, model first token, streaming card start/update/close, and final delivery. If the measured delay is from default Feishu API calls, adjust defaults or add clearer performance guidance; if it is from plugin/runtime startup or provider latency, fix the owner path with a focused regression test or live proof.
