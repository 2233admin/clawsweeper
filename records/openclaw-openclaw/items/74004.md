## Summary

Keep open. Current main still has only read/adopt behavior for Claude CLI OAuth credentials, while this PR adds the missing reverse write-back path; however, the proposed patch is not ready to land because it writes OAuth secrets into an external Claude CLI store without enough provider/identity guarding and has a concrete token-leak logging issue.

## What This Changes

The PR adds a post-refresh OAuth manager adapter hook and wires it to a new external CLI sync helper that writes refreshed Claude CLI OAuth credentials back through the existing Claude CLI file/Keychain writer.

## Best Possible Solution

If maintainers want bidirectional Claude CLI sync, keep it narrow: write back only verified claude-cli OAuth credentials for the dedicated managed profile, compare identity where available, use redacted error formatting, keep failures best-effort, and add focused tests plus docs/changelog updates. If OpenClaw should preserve the current token-sink/read-managed boundary, close this PR with that product-boundary explanation and keep relying on the #70902-style external read path.

## Reproduction Assessment

Yes. A code-path reproduction is high confidence: on the proposed patch, an expired OAuth credential under the `anthropic:claude-cli` profile id refreshes through oauth-manager and the new hook attempts Claude CLI write-back; a unit test can also show the provider-mismatch case because the helper gates only on profileId.

## Solution Assessment

No, not as currently written. The direction may be viable, but the patch needs provider/identity guards, redacted error logging, and regression coverage before it is the narrowest maintainable way to mutate Claude CLI credentials.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.87

Full review comments:

- **[P1] Gate write-back on the Claude CLI credential:** `src/agents/auth-profiles/external-cli-sync.ts:168-170`
  - body: The new helper writes to the Claude CLI store whenever the profile id is `anthropic:claude-cli`, regardless of `credential.provider` or identity. Profile ids are config/store keys, and the existing read-side path also checks the provider before using an external CLI source; without the same guard here, a mismatched OAuth profile can have its refreshed tokens merged into Claude CLI Keychain/file storage.
  - confidence: 0.88
- **[P2] Redact write-back error messages:** `src/agents/auth-profiles/external-cli-sync.ts:181-186`
  - body: This catch block logs `String(error)` from a credential write-back path. Future writer errors can include serialized token material, while the rest of this area uses `formatErrorMessage()` for best-effort secret redaction; use the same formatter here before merging.
  - confidence: 0.9

## Security Review

Status: needs_attention

Summary: The PR introduces a concrete OAuth secret-handling concern because refreshed credentials can be written to an external Claude CLI store without sufficient provider/identity validation, and one new error path bypasses standard redaction.

Concerns:

- **[medium] External credential write-back lacks provider guard:** `src/agents/auth-profiles/external-cli-sync.ts:168`
  - body: The write-back helper gates on profileId only, so a mismatched OAuth credential stored under the Claude CLI profile id could be merged into the Claude CLI Keychain/file. Require the expected `claude-cli` provider and identity compatibility before calling the writer.
  - confidence: 0.88
- **[low] Write-back error log bypasses redaction:** `src/agents/auth-profiles/external-cli-sync.ts:184`
  - body: Logging `String(error)` in the new catch block can expose OAuth token material if an error embeds request, response, or credential data. Use `formatErrorMessage()` or omit the message for credential-bearing failures.
  - confidence: 0.9
