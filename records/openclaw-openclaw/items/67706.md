## Summary

Keep open. Current main still lacks compaction lifecycle event emission from embedded-runner overflow and timeout recovery, so the PR addresses a real Control UI liveness gap, but the patch is not merge-ready because its completion semantics diverge from the canonical compaction handler and can make failed recovery compactions update memory compaction accounting as if they succeeded.

## What This Changes

The PR imports `emitAgentEvent` into `src/agents/pi-embedded-runner/run.ts` and adds compaction start/end event emission to the embedded runner recovery compaction helpers.

## Best Possible Solution

Land a revised implementation that emits compaction start/end events for overflow and timeout recovery on both the global bus and callback path, matches the canonical `handleCompactionEnd` completion semantics, updates memory accounting to honor `completed`, and adds focused recovery-event regression tests.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Align completed with the compaction result contract:** `src/agents/pi-embedded-runner/run.ts:593`
  - body: The recovery emitter reports `completed: true` whenever `ok && compacted`, but the canonical compaction handler marks completion from actual result presence. Since `CompactResult.result` is optional, `{ ok: true, compacted: true }` would show as completed/retrying even though no compacted result was produced. Please derive `completed` from the presence of `compactResult.result` to match the existing contract.
  - confidence: 0.82
- **[P2] Guard memory accounting against failed end events:** `src/agents/pi-embedded-runner/run.ts:601`
  - body: This path now forwards every recovery `phase: "end"` event to `params.onAgentEvent`, including failures with `completed: false`. The memory-flush runner currently treats any compaction end event as completed and increments persisted compaction count, so a failed timeout/overflow recovery can be recorded as a successful memory compaction. Update that consumer to require `completed === true`, or avoid sending failed end events to that callback.
  - confidence: 0.87
