## Summary

Keep this open. Current main has the generic MCP stdio cleanup from the prior leak work, but the implementation still only closes MCP clients/transports and sends process-tree signals to the wrapper process group; I found no Docker/container ownership tracking or cleanup for containers launched indirectly by npm/npx MCP wrappers during parallel subagent runs.

## What This Changes

The issue requests complete cleanup of Docker containers spawned indirectly by stdio MCP server wrappers during parallel specialist-agent sessions.

## Best Possible Solution

Keep the issue open and design a precise ownership model for MCP-launched containers: either require/record OpenClaw-owned container labels or wrapper metadata at launch, then clean only those containers during session MCP runtime disposal. The existing process-tree kill should remain the generic fallback for normal stdio child processes.

## Reproduction Assessment

Unclear. The report provides concrete steps and observed orphan containers, and source inspection confirms the absence of Docker-aware cleanup, but this review did not live-reproduce the Docker-wrapped UniFi MCP server path.

## Solution Assessment

No, not as the immediate broad sweep suggested in the issue. The maintainable fix should track session-owned containers explicitly and avoid image-name or descendant-process guesses that can delete unrelated Docker workloads.
