## Summary

Keep this open. Current main still cleans up MCP stdio servers by disposing MCP transports and killing the wrapper process group, but I found no Docker/container ownership tracking or removal path for containers indirectly launched by npm/npx MCP wrappers during parallel subagent runs.

## What This Changes

The issue requests complete cleanup of Docker-wrapped stdio MCP servers spawned during parallel specialist-agent sessions so wrapper-launched containers do not remain orphaned after the session ends.

## Best Possible Solution

Keep the issue open and design a precise ownership contract for MCP-launched containers, such as OpenClaw-owned labels or wrapper metadata recorded at launch, then clean only those containers during session MCP runtime disposal while preserving process-tree cleanup for normal stdio servers.

## Reproduction Assessment

Unclear. The issue provides concrete observed Docker output and a plausible parallel-spawn path, and current source inspection confirms no Docker-aware cleanup, but this review did not live-run the UniFi Docker-wrapped MCP server scenario.

## Solution Assessment

No. The broad immediate suggestion to infer child containers from `/proc` or cgroups is risky; the maintainable fix should explicitly track OpenClaw-owned MCP containers and prove cleanup only touches those containers.
