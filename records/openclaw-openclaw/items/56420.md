## Summary

Keep this PR open. Current main still keys Claude permission replies only by the short request id, so a `yes/no <id>` message from another session can satisfy a pending Claude permission. The PR’s central security fix is not implemented on main, and the related open leak issue/PR are orthogonal cleanup around pending-map lifetime rather than a replacement for session binding.

## Best Possible Solution

Review and land this PR or an equivalent focused fix that propagates the originating session key through the Claude permission request schema/server path, stores it with the pending request, rejects replies from other sessions, and adds regression tests for both attacker-session rejection and trusted-session acceptance. Keep the TTL/leak work in #71646/#71648 separate because it does not replace the session-binding security fix.
