## Summary

Keep this PR open. Current main still lacks native outbound WhatsApp mention metadata, and the PR is a real implementation candidate for the open WhatsApp mentions request, but the proposed resolver still has LID precedence/collision edge cases and a default-notification behavior that should get maintainer review before merge.

## What This Changes

This PR adds a WhatsApp outbound mention resolver, wires it into group text/media sends and reply helpers, and adds tests, docs, and a changelog entry for resolving `@+<digits>` and `@<digits>` tokens to Baileys native mention metadata.

## Best Possible Solution

Iterate on this PR rather than close it: keep the scope limited to outbound WhatsApp mention metadata, prefer the correct LID target when Baileys provides both phone and LID identifiers, handle ambiguous digit collisions deliberately, document the default notification behavior, rebase onto current main, and validate the WhatsApp extension tests plus the changed gate.

## Reproduction Assessment

Yes. Current-main source inspection gives a high-confidence reproduction path: a WhatsApp group send containing `@<digits>` reaches Baileys as plain `{ text }` or caption content without `mentions`, while the PR adds focused resolver tests for the intended metadata path. I did not run tests because this review is read-only.

## Solution Assessment

Unclear. The owner module and Baileys `mentions` mechanism are the right narrow direction, but this is not yet the best mergeable form until the LID precedence/collision behavior and default auto-notification policy are resolved or explicitly accepted.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Prefer LID JIDs when a participant has one:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:129-137`
  - body: Baileys participants can expose both a phone JID in `id` and a LID JID in `lid`, but `chooseMentionJid` normalizes `id` first. In that shape, the LID lookup is built with the phone JID, so a `@<lid>` token in a LID-backed group can still send a phone mention that WhatsApp may ignore. Prefer the `lid` field when it is a LID JID, then fall back to `id`/phone metadata.
  - confidence: 0.84
- **[P3] Avoid ambiguous phone and LID digit collisions:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:206-213`
  - body: The resolver always checks `byPhone` before `byLid`, so equal digit strings silently choose the phone participant even when the sender intended the LID participant. Please make collisions explicit, prefer the identifier implied by the matched token, or skip ambiguous matches rather than mentioning the wrong user.
  - confidence: 0.66

## Security Review

Status: needs_attention

Summary: The patch has no supply-chain or secret-handling changes, but it does broaden outbound notification side effects by auto-converting digit tokens into native WhatsApp group mentions.

Concerns:

- **[low] Auto-mentions can notify users from plain generated text:** `extensions/whatsapp/src/inbound/outbound-mentions.ts:206`
  - body: Any matching `@+<digits>` or `@<digits>` token in group text/captions becomes a native WhatsApp mention when participant metadata matches. Because outbound text can be model-generated or relay user-provided content, maintainers should explicitly accept this default or gate it behind a clearer intent/config boundary to avoid unwanted group notifications.
