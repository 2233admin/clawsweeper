## Summary

Keep open. Current main still only skips heartbeats when the main or resolved session command lane is busy; it does not check active cron work, does not define `heartbeat.skipWhenBusy`, and the wake retry path only recognizes `requests-in-flight`. The linked implementation PR #50774 is closed unmerged, and the adjacent open #64963 targets active reply runs rather than cron collisions.

## Best Possible Solution

Keep this issue open and land a focused core fix: always defer heartbeat runs while cron work is active, add the opt-in broader `heartbeat.skipWhenBusy` lane policy if maintainers accept the requested config surface, wire all busy skip reasons through the same wake retry/no-schedule-advance path, and document/test the behavior.

## Clownfish Work Prompt

Implement heartbeat-cron collision avoidance for this issue on current main. Current `runHeartbeatOnce` skips only `CommandLane.Main` and the resolved session lane, and `heartbeat-wake.ts` only retries `requests-in-flight`. Add an always-on heartbeat skip when cron work is active, covering both `CommandLane.Cron` depth for queued/manual runs and cron active-job state for timed/executing runs. Add the requested opt-in `heartbeat.skipWhenBusy` config that also skips when subagent/nested lanes are active; keep it scoped to heartbeat config under `agents.defaults.heartbeat` and `agents.list[].heartbeat`. Return stable skip reasons such as `cron-in-progress` and `lanes-busy`, and update the wake/scheduler paths so these reasons retry after 1s and do not advance the schedule, matching `requests-in-flight`. Add focused tests around cron-active skip, idle no-skip, skipWhenBusy subagent/nested behavior, priority/order, wake retry, and scheduler no-advance behavior. Update config schema/types/help/generated docs as required by repo tooling and public heartbeat/config docs. Include a single-line changelog entry only if repo policy treats the new config/behavior as user-facing; credit `Thanks @scottgl9`.
