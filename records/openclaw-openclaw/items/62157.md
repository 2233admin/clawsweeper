## Summary

Keep this PR open because current main still does not implement the requested text-only post-tool final-payload delivery, but the proposed patch is not merge-ready. The PR correctly identifies the silent-drop class, yet its per-payload `hasSentPayload()` filter can resend already-streamed content when block-stream coalescing merges multiple logical chunks into one outbound payload.

## What This Changes

This PR changes block-streaming final-payload filtering so final replies are filtered with `hasSentPayload()` instead of being dropped wholesale, and updates the related unit test.

## Best Possible Solution

Land a narrow block-streaming fix that preserves genuinely new post-tool final text while still suppressing all content already delivered through exact, logical, coalesced, media, and direct-send paths. The fix should extend the pipeline/build-payload contract rather than relying only on flushed payload content keys, then cover post-tool text, coalesced duplicates, media, errors, abort fallback, and silent-turn behavior with focused tests.

## Reproduction Assessment

Yes. The current-main source/test path reproduces the original missing behavior for text-only finals, and the PR-head code plus the existing review comment give a high-confidence reproduction for the introduced coalescing duplicate: stream `A` and `B` as one coalesced block, then return final payloads `A` and `B`; both can pass the PR filter.

## Solution Assessment

No. The direction is useful, but this patch is not the best fix because it treats `hasSentPayload()` as complete proof for each final payload while the pipeline can coalesce multiple streamed chunks into a different flushed payload. A safer fix should track or derive logical streamed-content coverage before allowing unsent post-tool text through.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep coalesced stream dedupe:** `src/auto-reply/reply/agent-runner-payloads.ts:215-217`
  - body: This filter asks `hasSentPayload()` about each final payload independently, but the PR head only records the content key for the payload actually flushed by the block pipeline. With coalescing enabled, chunks `A` and `B` can be streamed as one `A\n\nB` payload; later final payloads `A` and `B` do not match that sent key and are delivered again at turn end. Please preserve logical/coalesced streamed-content suppression before passing unsent post-tool text through.
  - confidence: 0.88
