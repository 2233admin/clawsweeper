## Summary

Keep this issue open. Current main supports persistent channel/group routing through the generic `channels.modelByChannel` config, but the remaining per-DM part is still not implemented: the reply-time resolver is not given direct peer identifiers, and the resolver returns `null` when both `groupId` and `parentSessionKey` are absent.

## Best Possible Solution

Keep this issue open and finish the existing generic `channels.modelByChannel` implementation for direct conversations. Extend the resolver input and reply call site to include direct peer candidates from the finalized context, persisted session origin, native direct identifiers, and current session key; document supported direct key forms and wildcard behavior; add focused Slack/WhatsApp DM regression coverage while preserving current channel/group precedence.

Current main already supports channel/group entries, but ordinary DMs do not match because `resolveChannelModelOverride` only receives `groupId` and parent session keys.
