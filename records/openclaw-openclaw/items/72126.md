## Summary

Current main still does not implement this PR's intended OpenAI Realtime keychain-ref resolution. The OpenAI realtime voice provider still reads `config.apiKey || process.env.OPENAI_API_KEY` directly for both browser client-secret sessions and server-side bridge creation, so a value like `keychain:openclaw:OPENAI_API_KEY` can still be sent as the literal API key. This PR remains a focused implementation candidate for the linked #72120 repro, with security-sensitive review comments to address before merge.

## Best Possible Solution

Keep this PR open as the active implementation candidate for #72120, but require maintainer review or author iteration before merge. The best fix should resolve `keychain:<service>:<account>` before both OpenAI Realtime browser-session creation and backend bridge creation use the API key, avoid sending unresolved markers in Authorization headers, and add targeted regression tests. Reviewers should also decide whether the fix should use an existing runtime SecretRef seam rather than a provider-local hot-path keychain shell-out, or at minimum avoid memoizing transient failed lookups.
