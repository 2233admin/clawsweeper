## Summary

Keep this PR open for maintainer/author follow-up. Current main still has the reported unresolved Keychain-ref path, and the PR is aimed at the right OpenAI realtime boundary, but the proposed resolver adds an unbounded synchronous Keychain lookup in the gateway request path and the branch is currently dirty against main with a failed parity-gate check.

## What This Changes

The PR adds an OpenAI realtime API-key resolver for `keychain:<service>:<account>` strings, caches resolved values in-process, and routes both browser realtime session creation and backend realtime bridge creation through that resolver.

## Best Possible Solution

Update the PR so Keychain resolution is bounded and covered by focused tests, rebase it onto current main, rerun the OpenAI extension test and changed gate/CI, then merge if the parity-gate result is clean or proven unrelated.

## Reproduction Assessment

Yes. On current main, setting `OPENAI_API_KEY` to `keychain:openclaw:OPENAI_API_KEY` and invoking `createBrowserSession` or `createBridge` reaches the literal env string in the OpenAI bearer/websocket auth path; the linked issue also reports the resulting OpenAI 401 for the browser session path.

## Solution Assessment

No, not as-is. Resolving at the OpenAI realtime API-key boundary is the narrow maintainable direction, but the synchronous Keychain command needs a timeout/fail-closed bound before this is safe for the gateway request path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.78

Full review comments:

- **[P2] Bound the Keychain lookup timeout:** `extensions/openai/realtime-voice-provider.ts:144-150`
  - body: This `execFileSync` runs when a realtime browser session or bridge first needs the API key. With no `timeout`, a locked or prompting Keychain can block the Node event loop indefinitely; existing Keychain readers in this repo bound `security` calls to 5000ms. Add a timeout and continue to fail closed on lookup errors.
