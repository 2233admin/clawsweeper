## Summary

Keep this PR open: it targets a real OpenAI Realtime SecretRef bug that still exists on current main, but the proposed synchronous macOS Keychain lookup is unbounded and the PR is currently dirty with a failed parity-gate check. The branch needs an author or maintainer amendment before it is merge-ready.

## What This Changes

The PR adds an OpenAI Realtime API-key resolver for `keychain:<service>:<account>` strings, caches resolved values in-process, and uses it for both browser realtime session creation and backend realtime bridge creation.

## Best Possible Solution

Amend the PR to bound the Keychain lookup, fail closed on timeout/errors, and cover keychain success, miss, and timeout behavior in focused OpenAI realtime provider tests; then rebase on current main and rerun the OpenAI extension test plus the changed gate/parity check before merging.

## Reproduction Assessment

Yes. Current main has a direct code path from `OPENAI_API_KEY` to the OpenAI Realtime Authorization header and bridge API-key field, and the linked report provides a concrete macOS Keychain setup that produces the observed 401.

## Solution Assessment

No, not as-is. Resolving at the OpenAI realtime API-key boundary is a narrow maintainable direction, but the new synchronous Keychain process call needs a timeout and focused coverage before it is safe for the gateway request path.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.82

Full review comments:

- **[P2] Bound the Keychain lookup timeout:** `extensions/openai/realtime-voice-provider.ts:147-150`
  - body: This `execFileSync` runs the first time a realtime browser session or bridge resolves a `keychain:` API-key ref. Without a timeout, a locked or prompting Keychain can block the Node event loop indefinitely; existing Keychain readers and exec SecretRef resolvers in this repo bound child processes to 5000ms. Add a timeout and keep the lookup fail-closed on errors.
  - confidence: 0.84
