## Summary

Current main still does not implement this PR's intended OpenAI Realtime keychain-ref resolution. The provider still reads `config.apiKey || process.env.OPENAI_API_KEY` directly in both browser client-secret creation and server-side bridge creation, so `keychain:openclaw:OPENAI_API_KEY` can still be sent as a literal API key. The PR remains a focused implementation candidate, but it touches secret resolution and subprocess execution, has unresolved review findings, and needs maintainer/security review rather than automatic closure.

## Best Possible Solution

Keep this PR open as the active implementation candidate for the OpenAI Realtime keychain-ref bug, but require maintainer review or author iteration before merge. The best fix should resolve supported SecretRef/keychain values before both browser client-secret creation and server-side bridge creation use the API key, avoid sending unresolved markers in Authorization headers, avoid permanent caching of transient failures, and add targeted regression tests. Maintainers should decide whether the fix belongs in the existing runtime SecretRef snapshot path or as a carefully bounded OpenAI provider fallback.
