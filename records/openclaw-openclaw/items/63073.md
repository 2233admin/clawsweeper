## Summary

Keep open. Current main still hardcodes the MiniMax usage request to the CN remains endpoint, so the PR is not obsolete, but the proposed patch is not ready because it misses the `minimax-portal` OAuth region path and overlaps with open #65054/#65059.

## What This Changes

This PR adds a MiniMax usage `baseUrl` option, passes a provider/env-derived host from MiniMax provider registration, and updates the usage test to assert the GET request URL and method.

## Best Possible Solution

Land one MiniMax-owned, region-aware usage endpoint resolver that preserves `GET /v1/api/openplatform/coding_plan/remains`, derives global/CN routing from the actual auth/config context including `minimax`, `minimax-portal`, and `MINIMAX_API_HOST`, constrains or explicitly documents custom-origin token routing, updates the MiniMax docs, and adds regression coverage for global/CN API-key and OAuth setups.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Route portal usage through the portal base URL:** `extensions/minimax/provider-registration.ts:260-264`
  - body: `resolveUsageAuth` can return a `minimax-portal` OAuth token, but this branch still reads `models.providers[ctx.provider].baseUrl` while `ctx.provider` is the usage provider `minimax`. CN OAuth onboarding stores the host under `models.providers.minimax-portal.baseUrl`, so those users fall through to the new global default and usage polling can hit the wrong region with the portal token.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff changes the MiniMax usage bearer-token destination based on config/env and needs an explicit trusted-host or token-source-aware routing policy.

Concerns:

- **[medium] Constrain MiniMax usage token routing:** `extensions/minimax/provider-registration.ts:257`
  - body: The PR derives the usage request origin from `models.providers[ctx.provider].baseUrl` or `MINIMAX_API_HOST` after usage auth may have returned a `minimax-portal` OAuth token. That can send the portal usage token to the API-provider host or another configured origin instead of the matching MiniMax region host.
  - confidence: 0.78
