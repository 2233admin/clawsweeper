## Summary

Keep this PR open. Current main has route-scoped messaging dedupe helpers for final/follow-up payload delivery, but the embedded block-reply streaming and message_end paths still suppress by matching text globally. The PR targets a real remaining gap, though the submitted head is not ready as-is because review context flags an unresolved runtime-breaking undefined resolver.

## Best Possible Solution

Keep the PR open and ask for a current-main rebase that reuses the existing shouldSuppressMessagingToolReplies helper in emitBlockChunk and handleMessageEnd, threads the real inbound route/account/thread context into subscribeEmbeddedPiSession, defines/imports the route provider resolver, and aligns message-send target extraction with message-action normalization and Telegram topic handling. Add focused tests for same-route suppression, different-target non-suppression, providerless/implicit current-route sends, synthetic heartbeat/cron/exec providers, and topic/thread parity.
