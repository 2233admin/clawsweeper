## Summary

Keep open. Current main still has the route-scoped duplicate-suppression gap, but this PR head is not mergeable because it calls an undefined resolver and still misses Telegram topic origin context needed for same-route suppression.

## What This Changes

This PR makes embedded messaging-tool duplicate suppression route-aware by adding route context to subscriber paths, widening message-send target extraction, and checking sent targets before suppressing block replies.

## Best Possible Solution

Land a corrected route-scoped dedupe implementation that wires the existing suppression helper into both streamed and message_end block-reply paths, carries provider/target/account/thread context through the embedded subscription boundary, and keeps send-target tracking aligned with outbound delivery normalization.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: current main still suppresses by text only, and the PR head calls an undefined resolver. The Telegram topic duplicate path is traceable through separate `MessageThreadId` origin storage and suppression tests that require thread parity.

## Solution Assessment

No. The PR direction is the right narrow fix, but this branch is not the best mergeable solution until the resolver is defined, origin thread context is propagated, and Telegram thread alias tracking matches runtime behavior.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.96

Full review comments:

- **[P0] Define the routed provider resolver before use:** `src/agents/pi-embedded-runner/run/attempt.ts:383-386`
  - body: `runEmbeddedAttempt` calls `resolveRouteMessageProvider(...)`, but this PR does not declare or import that production symbol. This blocks compilation and would fail embedded runs before tool construction or reply handling.
  - confidence: 0.99
- **[P1] Carry the origin thread into suppression matching:** `src/agents/pi-embedded-runner/run/attempt.ts:1205-1207`
  - body: The subscription receives provider, target, and account context, but not the origin `messageThreadId`/current thread. Telegram stores topic context separately and suppression requires thread parity, so same-topic sends can still emit a duplicate final block reply.
  - confidence: 0.91
- **[P2] Prefer messageThreadId when tracking Telegram sends:** `src/agents/pi-embedded-subscribe.tools.ts:384-389`
  - body: The extractor derives `threadId` from `args.threadId` before `args.messageThreadId`, while Telegram execution resolves `messageThreadId` first. Calls with both aliases can be delivered to one topic and tracked against another.
  - confidence: 0.86
