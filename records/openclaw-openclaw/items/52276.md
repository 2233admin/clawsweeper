## Summary

Keep open. Current main still uses global text-only duplicate suppression in the embedded streamed and message_end block-reply paths, even though a route-aware helper exists. The PR is directionally useful, but its head should not merge because it still has concrete resolver and route-thread defects in the changed code.

## What This Changes

The PR changes embedded messaging-tool duplicate suppression to be route-aware by threading provider, target, account, and channel context through runner/subscriber paths, widening send-target extraction, and adding focused tests.

## Best Possible Solution

Land a corrected route-scoped dedupe implementation that wires the existing suppression helper into both streamed and message_end block-reply paths, derives the real route provider for synthetic automation providers, preserves providerless same-route sends, and carries origin target/account/thread metadata consistently through the embedded subscription boundary.

  - confidence: 0.86
