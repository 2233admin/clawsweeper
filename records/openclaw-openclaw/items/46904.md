## Summary

Keep this PR open. Current main still rejects `message` tool `action: "send"` calls when real poll-creation params are present, while only the narrower zero/default poll-param case is implemented. The PR remains a focused implementation candidate for the message-tool portion of #41199, with maintainer judgment still needed on silent stripping and current-main rebase details.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase or reimplement the sanitizer on current main, keep it symmetric with `hasPollCreationParams`, add regression coverage for camelCase, snake_case, mixed, true/string/non-zero poll params on `action: "send"`, and leave `action: "poll"` validation unchanged. Maintainers should explicitly decide whether silent discard is acceptable or whether low-noise debug observability belongs in the same change.

## Risks / Open Questions

- Silent stripping is a product/debuggability tradeoff: it makes LLM-generated sends succeed, but can hide a caller that intended `action: "poll"`; the existing review discussion already raised observability as the main concern.
- A current-main implementation should strip every field that current `hasPollCreationParams` can detect, including snake_case and dynamic channel-specific `poll*` fields such as `pollDurationSeconds` and `pollPublic`, not only the older shared key list.
- There is overlapping open work around the same message-tool conflict, including #46455 and the canonical issue #41199, so maintainers should pick the best implementation path instead of closing this candidate prematurely.
