## Summary

Keep this PR open. Current main still rejects `message` tool `action: "send"` calls when they contain real poll-creation params; only the narrower zero/default poll-param behavior is implemented. The PR remains a focused implementation candidate for the message-tool portion of #41199, though it needs maintainer review of the silent-stripping tradeoff and likely a current-main rebase.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase or reimplement the sanitizer on current main, make stripping symmetric with `hasPollCreationParams`, add regression coverage for camelCase, snake_case, mixed, true/string/non-zero poll params on `action: "send"`, and leave `action: "poll"` validation unchanged. Maintainers should explicitly choose whether silent discard is acceptable or whether low-noise debug observability belongs in the same change.

## Risks / Open Questions

- Closing this PR now would discard a focused implementation candidate for an unresolved part of #41199 while current main still rejects the reported non-default poll fields.
- The product tradeoff still needs maintainer judgment: silently stripping poll params can make LLM-generated sends succeed, but it may also hide a mistaken `action: "send"` when the caller really intended a poll. The existing review thread already raised observability as the main concern.
- A current-main rebase should preserve symmetry with today's `hasPollCreationParams` behavior, including snake_case and dynamically detected channel-specific `poll*` keys, not only the shared poll keys.
