## Summary

Keep this issue open. Current main still has partial NO_REPLY hardening: suffix stripping exists in normalizeReplyPayload, but gateway live/final chat projection and outbound payload planning still preserve the separated-prefix and terminal-suffix mixed forms that this issue asks to normalize. Related PR #39350 only covers the webchat lead-fragment case for #39335 and does not supersede this broader delivery contract.

## Best Possible Solution

Keep this issue open until maintainers settle and implement one shared silent-token normalization contract across gateway chat projection, outbound payload planning, routeReply/channel delivery, Telegram-class delivery, and prompt/docs guidance. The contract should explicitly cover exact suppression, separated-prefix suppression vs legitimate mentions, glued-prefix cleanup, terminal-suffix stripping with cleaned-content delivery, case handling, media payloads, direct/group rewrite policy, and interaction with related PR #39350.
