## Summary

Keep this issue open. Current main has adjacent NO_REPLY protections, including exact control-token suppression, lead-fragment buffering, auto-reply suffix cleanup, Telegram exact-token normalization, and prompt guidance warning models not to append NO_REPLY. The requested gateway-level behavior is still incomplete: live gateway/TUI projection does not strip standalone terminal suffix tokens, separated malformed prefixes are not discarded by the glued-prefix matcher, and outbound delivery planning still preserves mixed text such as `thanks NO_REPLY`.

## Best Possible Solution

Keep this issue open. Implement a shared silent-token normalization contract for the gateway display/delivery boundary, then wire it into live chat projection, embedded TUI projection, and any outbound delivery path maintainers decide should share the behavior. Add regression tests for exact suppression, separated-prefix discard, glued-prefix stripping, standalone terminal suffix stripping, newline suffix stripping, mixed case, and preservation of legitimate embedded/non-terminal `NO_REPLY` mentions. The Telegram exact-token path should stay covered, but it is only adjacent to this issue rather than a full fix.

## Risks / Open Questions

- A broad suffix rule could remove legitimate explanatory mentions of `NO_REPLY`; the implementation should strip only standalone terminal silent tokens and preserve embedded/non-terminal mentions in tests.
- OpenClaw currently has different contracts for gateway display projection, embedded TUI, auto-reply final normalization, outbound delivery planning, and channel-specific delivery policy. The fix should deliberately choose whether suffix cleanup is display-only, delivery-wide, or both.
