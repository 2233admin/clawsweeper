## Summary

Keep this issue open. Current main at a820a307dfeb950fa633c0ae6fb2386a35a91991 still only has partial NO_REPLY hardening: gateway live/final projection handles exact control text, lead fragments, and glued leading tokens, while separated-prefix and terminal-suffix mixed text still falls through. Outbound payload tests and routeReply tests explicitly preserve the mixed forms this issue asks to normalize, and prompt/docs text still describes exact-token behavior rather than the proposed completion-marker guard.

## Best Possible Solution

Keep this issue open and implement a shared silent-token normalization contract used by gateway live/final chat projection, outbound payload planning, routeReply/channel delivery, and relevant prompt guidance. The contract should define exact suppression, separated-prefix suppression, glued-prefix cleanup, terminal-suffix stripping with cleaned-content delivery, case handling, media payload behavior, direct/group policy and rewrite interaction, and legitimate embedded NO_REPLY mentions, then update focused tests for gateway chat, routeReply, outbound payload planning, Telegram-class delivery, and docs/prompts. If related PR #39350 is still active, fold its gateway lead-fragment work into this broader contract rather than treating it as a complete fix.
