## Summary

Keep this issue open. Current main has adjacent NO_REPLY protections, but the requested gateway-level handling is still incomplete: gateway live chat and embedded TUI projection strip glued leading NO_REPLY and suppress exact control tokens, but they do not strip standalone terminal suffix tokens like `Task complete. NO_REPLY` and do not discard separated malformed prefixes like `NO_REPLY\n\nHello`. The issue discussion also has a fresh 2026.4.x Telegram/Claude Sonnet 4.6 report, so this is not stale cleanup material.

## Best Possible Solution

Keep this issue open. Implement a shared gateway display/delivery normalization helper used by projectLiveAssistantBufferedText and embedded TUI that strips standalone terminal NO_REPLY suffixes, defines and enforces the intended separated-prefix discard behavior, and preserves legitimate embedded/non-terminal NO_REPLY mentions. Add regression tests for exact suppression, separated prefix discard, glued-prefix stripping, terminal suffix stripping, newline suffix stripping, mixed case, and the Telegram/outbound path called out in the latest comment.

- Gateway live chat, embedded TUI, auto-reply routing, outbound channel delivery, Telegram fallback delivery, cron, and subagent announce paths have separate silent-token contracts. The implementation should deliberately choose whether this is display-only normalization or a wider delivery contract.
