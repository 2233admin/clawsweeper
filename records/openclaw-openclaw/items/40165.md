## Summary

Keep this issue open. Current main still does not implement the requested shared robust NO_REPLY handling: gateway live/final chat and embedded TUI only suppress exact control-token text, lead fragments, and glued leading tokens, while outbound delivery planning still has tests that preserve mixed prefix/suffix NO_REPLY text as sendable payloads. The prompt wording now warns not to append NO_REPLY, but the gateway/display/delivery defense-in-depth requested here remains incomplete.

## Best Possible Solution

Keep this open and implement a shared silent-token normalization contract used before gateway display and before delivery-bound channel payloads. It should cover exact suppression, separated-prefix discard or a documented preservation rule, glued-prefix cleanup, standalone terminal suffix stripping with cleaned content delivery, mixed case, media payloads, and legitimate embedded NO_REPLY mentions. Add focused regression tests for gateway live/final chat, embedded TUI, outbound payload planning, routeReply/channel delivery for Telegram/WhatsApp/WebChat-class paths, and keep the prompt/docs wording aligned so NO_REPLY is not treated as a completion marker.

## Risks / Open Questions

- A broad suffix rule could incorrectly remove legitimate user-visible discussion of the literal token; the fix should target standalone terminal sentinel tokens and preserve embedded/non-terminal mentions.
- The normalization paths are split across gateway live/final projection, embedded TUI, routeReply, outbound payload planning, delivery, and channel adapters, so fixing one path can leave leaks elsewhere.
- Direct-chat silent-reply policy and rewrite behavior need an explicit decision so exact NO_REPLY suppression, fallback rewrites, and channel delivery stay consistent.
