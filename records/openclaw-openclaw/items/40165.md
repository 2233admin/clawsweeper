## Summary

Keep open. Current main has adjacent NO_REPLY protections, including exact-token suppression, streamed lead-fragment buffering, glued-prefix stripping, auto-reply suffix cleanup, and Telegram exact-token normalization. The requested shared gateway/outbound behavior is still incomplete: live chat projection can still show terminal suffix tokens like `Task complete. NO_REPLY`, separated malformed prefixes are intentionally not treated as prefixes by the current matcher, and outbound delivery normalization still preserves mixed text containing `NO_REPLY`.

## Best Possible Solution

Implement a shared silent-token normalization contract at the gateway display/delivery boundary, then wire it into live chat projection, embedded TUI/webchat projection, and outbound delivery planning where maintainers want identical behavior. Add regression tests for exact suppression, separated-prefix discard or documented preservation, glued-prefix stripping, standalone terminal suffix stripping, newline suffix stripping, mixed case, and preservation of legitimate embedded/non-terminal `NO_REPLY` mentions. Keep the Telegram exact-token normalization covered as adjacent defense-in-depth, but do not treat it as the full fix for this issue.

## Risks / Open Questions

- A broad suffix rule could remove legitimate explanatory mentions of `NO_REPLY`; the fix should strip only standalone terminal silent tokens and preserve embedded or non-terminal mentions in tests.
- OpenClaw currently has separate contracts for live gateway display, embedded TUI/webchat history, auto-reply final normalization, outbound payload planning, and channel-specific delivery. Maintainers should decide deliberately whether the shared cleanup is display-only, delivery-wide, or both.
