## Risks / Open Questions

- Cron chaining is an execution-orchestration surface: one job can cause another job to run, so maintainers should require tests and review for cycles, disabled or missing targets, one-shot deletion, manual vs scheduled paths, run history, delivery/failure visibility, and bounded behavior under failure.
- The PR branch appears stale against current main's protocol/test layout, including a generated `dist/protocol.schema.json` addition while current main's root `vitest.config.ts` is now a wrapper and generated protocol artifacts have moved/changed shape.
- Current docs steer durable multi-step workflows toward Task Flow/Lobster with cron as the scheduler, so maintainers need a product decision on whether cron-native job chaining should land or be superseded by a Task Flow-oriented follow-up.
