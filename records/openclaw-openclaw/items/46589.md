## Summary

Keep this PR open. Current main covers the cron edit/run-history part of the linked request, but it still lacks the PR's job-to-job pipeline orchestration fields and runtime chaining, and it also lacks the smaller cron UI i18n fixes. This is not implemented on main and needs maintainer review, rebase, or explicit supersession by the Task Flow direction.

## Best Possible Solution

Keep this PR open for maintainer product and code review. The best path is to rebase and narrow it to the still-missing cron chaining and i18n fixes with current protocol/client generation plus focused runtime/UI tests, or explicitly supersede the chaining portion with a Task Flow-oriented follow-up that gives users a supported scheduled pipeline workflow.

## Risks / Open Questions

- The PR branch is stale against current main: `vitest.config.ts` is now a wrapper, and current main has no `dist/protocol.schema.json` while the PR adds one generated artifact.
- Cron chaining is a security/safety-sensitive execution surface because one job can force-run another. Before merge, maintainers should require tests for cycles, disabled or missing targets, one-shot `deleteAfterRun`, manual and scheduled success/failure paths, run history, and failure/delivery visibility.
- Current docs steer durable multi-step workflows toward Task Flow/Lobster with cron as the scheduler, so maintainers should decide whether cron job-to-job chaining is still desired or should be superseded by a Task Flow follow-up.
