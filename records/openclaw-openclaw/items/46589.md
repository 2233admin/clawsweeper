## Summary

Keep open. Current main still does not implement cron success/failure chaining, and the PR head is not ready to merge because direct manual cron runs can complete without dispatching configured chained jobs. The i18n and test-registration fixes look useful, but the feature portion needs maintainer review because it overlaps with the documented Task Flow direction for durable multi-step workflows.

## What This Changes

The PR fixes cron Control UI i18n and test-registration gaps, then adds success/failure cron job chaining fields across cron runtime, gateway schema, generated Swift protocol models, and the Control UI.

## Best Possible Solution

Land or replace only the low-risk Control UI i18n and cron test-registration fixes if maintainers want them independently, then either implement cron-native chaining through one shared post-run dispatch path with protocol/docs/tests for scheduled, queued manual, and direct service runs, or keep durable multi-step orchestration in Task Flow instead of adding a second pipeline primitive to cron.

## Reproduction Assessment

Yes. At PR head, configure a job with onSuccessJobId or onFailureJobId and invoke CronService.run()/ops.run() directly; the inspected source returns after finishPreparedManualRun(), while only enqueueRun() dispatches triggerChainedJob().

## Solution Assessment

No. The i18n fixes are narrow, but the chaining implementation is not the best solution as-is because it splits post-run dispatch across entry points and misses direct manual runs; a centralized post-run chain dispatcher or Task Flow routing is safer.
