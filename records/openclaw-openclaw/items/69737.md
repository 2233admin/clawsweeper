## Summary

Keep open. Current main still has the two reported behaviors: unclassified assistant error prose can be emitted as a user-facing error reply/lifecycle error after redaction, and assistant visible-text extraction still joins multiple matching text blocks. The open related PR #41803 is a good candidate for the raw-error half, but it is still open and does not cover the multi-text delivery policy.

## What This Changes

Request to sanitize Slack/Telegram delivery so raw assistant errorMessage prose is not posted to users and multiple assistant text content items are not concatenated into one duplicate-looking channel post by default.

## Best Possible Solution

Keep this issue open. Review and land #41803 or an equivalent narrow fix for unclassified assistant-error fallback with a regression covering OpenAI 500/help.openai.com/request-ID style prose, then either split the multi-text behavior into its own scoped issue or implement a delivery-boundary selection policy without accidentally changing shared extractor contracts across TUI, gateway display, and chat-history surfaces.
