## Summary

Keep #69737 open. Current main has partial hardening for structured/raw JSON and HTTP-shaped provider errors, but the reported class is not fully implemented: unclassified short `stopReason: error` text can still be surfaced after redaction, and assistant visible-text extraction still joins multiple text content blocks instead of selecting one.

## Best Possible Solution

Keep #69737 open. Best path is to land a minimal delivery/lifecycle fix so `stopReason: "error"` never emits unclassified raw `errorMessage` text to Slack/Telegram users, with a regression test for the plain OpenAI 500/help-link/request-id string. Then split or explicitly scope the multi-text behavior into a narrower owner-reviewed policy change, because current shared extractors intentionally join all matching text blocks and have non-channel callers.
