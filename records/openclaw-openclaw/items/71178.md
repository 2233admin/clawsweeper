## Summary

Keep open. Current main still has the reported `update.run` continuation producer gap: restart continuations are consumed when present, but `update.run` does not accept `continuationMessage`, does not persist `payload.continuation`, and the agent gateway tool does not forward the field. The active path is to review and land PR #74362 or an equivalent fix before closing this issue.

## What This Changes

The issue reports that running `openclaw update` or `update.run` during an in-flight Telegram or Discord agent turn can restart the gateway and silently lose buffered or orphaned assistant replies.

## Best Possible Solution

The desired end state is an additive `update.run` continuation path in main: accept `continuationMessage`, forward it from the agent gateway tool, persist an `agentTurn` restart continuation only for successful update restarts, and keep broader drain/reset recovery as separate focused follow-up work.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, an `update.run` request with a `sessionKey` writes a restart sentinel without `continuation`, and existing tests assert that absence.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge handling are adjacent recovery work, not substitutes for the missing producer field.
