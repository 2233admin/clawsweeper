## Summary

Keep open. Current main still has the `update.run` restart-continuation producer gap: the restart continuation consumer exists, but `update.run` neither accepts `continuationMessage` nor writes `payload.continuation`, and tests still assert the field is absent. Open PR #74362 is the active narrow implementation candidate, so this issue should remain open until that or an equivalent fix lands.

## What This Changes

The issue reports silent Telegram/Discord response loss when `openclaw update` restarts the gateway while an agent turn still owes the user a reply.

## Best Possible Solution

Review and land PR #74362 or an equivalent narrow additive fix that threads `continuationMessage` through the update protocol, restart request parser, agent gateway tool, and successful update restart sentinel. Then split any remaining graceful-drain, post-reset bridge, or coalesced-restart recovery work into focused follow-ups such as #74424.

## Reproduction Assessment

Yes. The source-level reproduction is high confidence: on current main, an `update.run` request with a session key writes a restart sentinel without `continuation`, and existing tests assert that missing field.

## Solution Assessment

Yes for the narrow producer gap. Reusing the existing restart-continuation contract for `update.run` is the smallest maintainable fix; graceful drain and post-reset bridge handling are adjacent recovery work, not substitutes for the missing producer field.
