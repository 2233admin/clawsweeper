## Summary

Keep open. Current main still updates the in-memory cron job before cron-expression validation can fail: disabled jobs skip next-run computation after a schedule edit, so invalid cron expressions can be persisted, and enabling such a job mutates `enabled` before Croner throws. The earlier #74193 fix only maps thrown Croner parse errors to `INVALID_REQUEST`; it does not make `cron.update` atomic or validate disabled schedule edits before mutation.

## What This Changes

The issue asks for invalid `cron edit --cron` expressions to be rejected without mutating disabled cron jobs, and for failed enable attempts not to leave invalid jobs enabled in live state.

## Best Possible Solution

Make cron updates validate and stage the full post-patch job before mutating stored state, then commit the staged job only after schedule computation succeeds. The fix should cover disabled schedule edits and enable transitions, keeping the previous valid schedule/enabled state intact on invalid cron expressions.

## Repair Work Prompt

Fix the cron update atomicity bug reported in openclaw/openclaw#74459. Current behavior: `openclaw cron edit <id> --cron <invalid>` can persist an invalid cron expression when the job is disabled, and `openclaw cron enable <id>` can return `INVALID_REQUEST` while leaving the live job enabled with the invalid schedule. Keep the fix focused in cron service/gateway validation: stage the post-patch job on a clone or otherwise validate the resulting schedule before mutating stored state, compute next-run state on the staged job when schedule/enabled changes require it, and only commit/persist after validation succeeds. Add regression tests proving disabled invalid cron edits reject without changing the previous valid schedule, and failed enable of an invalid stored/job candidate does not leave the live job enabled. Do not rely only on CLI validation, do not broaden generic TypeError/RangeError handling, do not change Croner dependency versions or patches, and do not refactor unrelated cron delivery behavior. Add a single-line `CHANGELOG.md` Fixes entry under Unreleased with `Thanks @bittoby` unless maintainers decide this is internal-only.
