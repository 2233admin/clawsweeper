## Summary

Keep open. Current main still leaves a CLI paired only with `operator.read` unable to reject its own pending repair request because `device.pair.reject` remains pairing-scope gated, gateway authorization runs before the device handler, and the CLI reject path has no local fallback. Current main does use least-privilege scopes for classified CLI methods, which partially reduces future over-broad requests, but it does not provide the requested self-reject, downgrade, or documented bootstrap recovery path.

## What This Changes

The issue requests a supported recovery path for a read-only paired CLI stuck behind an auto-reissued broader pairing repair request that it cannot approve or reject.

## Best Possible Solution

Add or document a bounded recovery flow for same-device local repair deadlocks that lets a limited CLI clear or replace only its own pending upgrade request without silently granting broader operator scopes. The chosen path should update the CLI, gateway/device-pairing auth checks, public devices docs, and regression coverage around read-only paired devices, same-device pending requests, and cross-device denial.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI/device token with only `operator.read`, plus a same-device pending repair request requiring `operator.pairing`; `openclaw devices reject <requestId>` still routes through `device.pair.reject`, which is rejected before the handler and has no CLI fallback.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the best fix depends on maintainer security policy around pairing recovery authority; least-privilege CLI requests alone are not enough because the read-only device still cannot clear or replace the pending repair request.
