## Summary

Keep open. Current main still leaves a read-only paired CLI unable to clear its own pending repair request: `device.pair.reject` is pairing-scope gated before the handler can apply same-device checks, and the CLI reject command has no local loopback fallback. The follow-up discussion also confirms the reporter is blocked in practice and intentionally avoided approving the broader request.

## What This Changes

The issue asks for a supported recovery path or narrower repair-scope flow when a CLI paired with only `operator.read` gets stuck behind an auto-reissued scope-upgrade repair request.

## Best Possible Solution

Add or document a bounded recovery path for same-device local-loopback repair deadlocks without silently granting broader operator scopes. The end state should let a limited CLI clear or replace only its own pending repair request, preserve explicit approval for scope upgrades, and cover the chosen semantics in CLI, gateway pairing, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI/device record with only `operator.read`, followed by a `devices reject` or broader reconnect attempt; current main keeps reject behind `operator.pairing` and has no reject fallback.

## Solution Assessment

No, the current behavior is not the best way to solve the issue because it leaves the recovery path requiring the missing scope. The safest fix direction needs maintainer approval on the authorization model before code changes.
