## Summary

Keep open. Current main still leaves the reported recovery deadlock in place: `device.pair.reject` is pairing-scope gated before the device handler can apply its same-device checks, and the CLI only has local fallback paths for list and approve. The reporter's follow-up confirms this is an active control-plane blocker, while the related closed reports cover approve/list or read-scope behavior rather than reject, downgrade, or narrow repair-scope recovery.

## What This Changes

The issue requests a supported recovery path for a paired CLI limited to `operator.read` when the gateway keeps creating broader pending device repair requests that the CLI cannot approve or reject.

## Best Possible Solution

Define a bounded same-device recovery model for local repair deadlocks, then implement it across the gateway pairing surface, CLI, docs, and regression tests. The end state should let a limited CLI clear or replace only its own pending repair request, preserve explicit approval for scope upgrades, and avoid silently granting broader operator scopes.

## Reproduction Assessment

Yes. A high-confidence source-backed reproduction is a paired CLI/device with only `operator.read`, a pending repair request for broader operator scopes, and `openclaw devices reject <requestId>` failing because `device.pair.reject` requires `operator.pairing` before the handler can apply same-device logic.

## Solution Assessment

Unclear. The report identifies a real bug, but the best fix is a product/security choice between self-reject fallback, narrower repair-scope requests, or a different bootstrap flow; current main does not yet pick or document one.
