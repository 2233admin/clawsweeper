## Summary

Keep open. Current main has a local loopback fallback for `openclaw devices list` and explicit `devices approve`, but `devices reject` still goes through the gateway RPC only, and the docs explicitly describe the fallback as list/approve-only. The broader request for rejecting, downgrading, or requesting narrower repair scopes remains unresolved.

## What This Changes

The issue asks for a supported recovery path when a CLI paired only with `operator.read` receives recurring full-scope operator repair requests that it cannot approve, reject, downgrade, or replace with minimal scopes.

## Best Possible Solution

Add or document a supported recovery that does not require `operator.pairing` from the already scope-limited CLI: most likely a reviewed local-loopback self-reject path for pending repair requests, plus a clear decision on whether CLI repair requests can be narrowed instead of using the full default operator bundle.
