## Summary

Keep open. Current main still leaves the reported read-only CLI recovery deadlock plausible: the CLI requests the broad default operator scope bundle, pairing approve/reject are pairing-scoped gateway methods, and `devices reject` has no local loopback fallback comparable to list/approve.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` when an auto-reissued broader repair pairing request cannot be rejected, approved safely, or narrowed.

## Best Possible Solution

Define an explicit recovery contract for read-only paired CLIs that preserves device identity checks and local-only boundaries, then implement and document that path with focused gateway, infra, CLI, and docs coverage.

## Reproduction Assessment

Yes. A source-level reproduction is high-confidence: a CLI paired with only `operator.read` cannot call `device.pair.reject`, `device.pair.approve`, or other pairing-scoped RPCs, and current CLI fallback only covers list/approve on local loopback, not reject or scope downgrades.

## Solution Assessment

Unclear. Current main is not sufficient, but choosing between self-reject, narrower repair requests, or another bootstrap/recovery flow needs maintainer product and security judgment before a repair PR should be queued.
