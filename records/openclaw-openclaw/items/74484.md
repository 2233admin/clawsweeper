## Summary

Keep open. Current main still leaves a read-only paired CLI unable to reject or replace its own pending scope-upgrade request: the reject RPC is pairing-scope gated, the CLI reject command has no local fallback, and the reporter confirmed this blocks real cron and ACP/sub-agent gateway work. This is not a cleanup-close item because the remaining fix changes gateway pairing recovery authorization semantics.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` when it keeps receiving an over-scoped repair pairing request it cannot approve or reject.

## Best Possible Solution

Add or document a bounded recovery path for same-device local-loopback repair deadlocks without silently granting broader operator scopes. The end state should let the limited CLI clear or replace only its own pending repair request, preserve explicit approval for scope upgrades, and cover the chosen semantics in CLI, gateway pairing, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a paired CLI/device record with only `operator.read`, followed by a reconnect or classified gateway call that requests a broader operator scope; current main creates or refreshes a pending scope-upgrade request while `openclaw devices reject <requestId>` remains blocked behind `operator.pairing`.

## Solution Assessment

Unclear. The issue identifies a real deadlock, but the best fix needs maintainer/security choice between same-device local self-reject, narrower repair-scope requests, or a separate bootstrap/recovery flow.
