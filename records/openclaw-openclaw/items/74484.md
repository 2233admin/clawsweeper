## Summary

Keep open. Current main still supports local loopback fallback for listing and approving device pairing requests, but a read-only paired CLI cannot reject or reshape its own pending scope-upgrade request because `device.pair.reject` remains pairing-scope gated and the CLI reject command has no local fallback. The reporter’s follow-up confirms this is blocking real gateway control-plane work, so this needs maintainer/security semantics rather than cleanup closure.

## What This Changes

The issue requests a supported recovery path for a CLI paired with only `operator.read` when it keeps receiving an over-scoped repair pairing request it cannot approve or reject.

## Best Possible Solution

Add or document a bounded recovery path for a same-device, local-loopback repair deadlock without silently granting broader operator scopes. A good end state would preserve explicit approval for scope upgrades, allow the limited CLI to clear or replace only its own pending repair request, and cover the chosen semantics in CLI, gateway pairing, docs, and regression tests.

## Reproduction Assessment

Yes. A high-confidence reproduction path is a paired CLI/device record with only `operator.read`, followed by a reconnect/request for the current broad CLI operator scope bundle; current main will keep `device.pair.reject` behind `operator.pairing` and the CLI reject path will not fall back locally.

## Solution Assessment

Unclear. The issue correctly identifies the deadlock, but the best fix depends on maintainer policy: same-device local self-reject, narrower repair-scope requests, or a separate bootstrap flow each has different security implications.
