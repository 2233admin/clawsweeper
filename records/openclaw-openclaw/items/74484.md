## Summary

Keep open. Current main still leaves the reported recovery gap: a scope-limited paired CLI can list pending pairing state and approve through the local loopback fallback, but `openclaw devices reject` still goes through the gateway RPC path and therefore requires `operator.pairing`; there is also no documented downgrade or minimal-scope repair request flow for this state.

## What This Changes

The issue asks for a supported way for a scope-limited CLI to reject, downgrade, or narrowly reissue an over-scoped gateway pairing repair request without already having `operator.pairing`.

## Best Possible Solution

Add and document a reviewed recovery path for a locally paired, scope-limited CLI that can resolve its own pending repair request without already holding `operator.pairing`, likely by adding a loopback-only self-reject path and deciding whether repair requests may request narrower operator scopes instead of the full default bundle.
