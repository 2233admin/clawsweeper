## Summary

Keep this issue open. Current main still routes normal replies through the TTS post-processing pipeline, but cron isolated announce delivery sends synthesized/final text payloads directly through outbound delivery after only silent-token normalization. The latest maintainer comment in the issue also explicitly says this TTS-tag case remains open after the related cron media issue was fixed.

## Best Possible Solution

Implement a shared pre-delivery TTS transform for cron isolated `delivery.mode: "announce"` before `deliverOutboundPayloads`, using the same effective config context as normal replies: agent id, channel, account id, final-kind delivery, and session TTS mode where available. Add a focused regression test where a cron isolated announce final reply containing `[[tts]]` or `[[tts:text]]` becomes a synthesized voice/media payload and the visible text no longer includes the raw directive. Keep #33082 treated as related but separate because it covered tool-produced media, not model-emitted TTS tags.
