## Summary

Keep this issue open. Current main still applies `[[tts]]` processing in the normal reply path, but isolated cron `delivery.mode: "announce"` still builds plain text payloads and sends them through outbound delivery after only silent-token normalization. The related cron tool-media fix for #33082 is separate; this model-emitted TTS tag path remains unimplemented.

## Best Possible Solution

Implement a shared pre-delivery TTS transform for isolated cron announce delivery before `deliverOutboundPayloads`, using the same effective context as normal final replies: `cfgWithAgentDefaults`, delivery channel/account, agent id, final kind, and any preserved session `ttsAuto`. Add a cron isolated announce regression test where `[[tts]]` or `[[tts:text]]` output is stripped from visible text and delivered as synthesized voice/media. Keep #33082 treated as related but already separate because it covered tool-produced media, not model-emitted TTS tags.
