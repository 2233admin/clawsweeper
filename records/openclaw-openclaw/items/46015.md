## Summary

Keep open. Current main includes the diagnostics and general per-agent TTS config work, but automatic TTS still treats the selected provider as the primary entry in a fallback chain. A configured ElevenLabs provider can still fall through to Microsoft/Edge after a skip or failure, so the remaining provider-stickiness/fail-closed behavior from the issue discussion is not implemented.

## Best Possible Solution

Keep this issue open as the canonical remaining decision for auto-TTS provider stickiness/fail-closed semantics in Discord. A maintainer should decide whether `messages.tts.provider` should disable fallback for automatic replies, add a separate config flag, or keep fallback expected and update docs/status wording accordingly.
