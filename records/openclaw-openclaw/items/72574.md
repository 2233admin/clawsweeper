## Summary

Keep open. Current main still routes Codex app-server run startup through the shared client path and clears that shared client on startup errors, while an open linked PR explicitly targets this issue with closing syntax; the issue should remain open until that PR or an equivalent fix lands.

## What This Changes

Invalid or expired bearer-token failures in spawned Codex helper runs should fail locally with a 401 without disconnecting the active gateway/session.

## Best Possible Solution

Land PR #72812 or an equivalent narrow fix that gives spawned/helper Codex app-server runs an isolated client, closes only that isolated client on startup/auth failure, preserves shared-client reuse for normal foreground runs, and includes regression coverage for auth failure isolation plus a non-spawned counter-test.

## Reproduction Assessment

Yes. The issue provides concrete command, version, environment, and logs, and current main shows a source-level reproduction path where a spawned Codex startup auth failure can enter shared-client teardown.

## Solution Assessment

Yes. The linked PR’s stated direction is the narrow maintainable fix: isolate helper clients based on real spawn metadata rather than changing global gateway lifecycle behavior.
