## Summary

Keep this open. Current main still has the reported failure boundary: Codex app-server startup uses the shared client factory and clears the shared singleton on startup-path errors, while the linked open PR explicitly closes this issue but has not merged.

## What This Changes

The issue requests that invalid bearer-token failures from `btw spawn`/Codex helper startup fail locally without disconnecting the active OpenClaw gateway/session.

## Best Possible Solution

Review and land PR #72812 or an equivalent narrow fix that isolates spawned/helper Codex app-server startup from the shared client, keeps ordinary runs on the shared client, and preserves explicit cleanup for startup failure, timeout, abort, turn-start failure, and completion paths.

## Reproduction Assessment

Yes. The issue has concrete steps and logs, and current main has a source-level reproduction path where a spawned/helper auth failure can pass through shared app-server startup and clear the shared singleton.

## Solution Assessment

Yes. The linked PR direction is the narrow maintainable solution because it gates isolated startup on real spawned/helper lineage instead of changing the broader shared-client contract for ordinary Codex runs.
