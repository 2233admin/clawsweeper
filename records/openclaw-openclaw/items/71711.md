## Summary

Keep open. Current main still lacks a recurring cron expiration contract: the persisted job type, gateway add/update schemas, CLI add/list surfaces, scheduler result handling, agent cron tool schema, Control UI form, and cron docs only cover one-shot delete-after-run plus recurring rescheduling. The linked #71712 RFC is still open and may affect the public API shape, so this should stay in maintainer review rather than be swept closed.

## What This Changes

Add an optional `activeUntil`/`expiresAt` lifetime and `onExpire` action for recurring `--every` and `--cron` jobs, including scheduler enforcement, CLI/list display, persisted schema, audit logging, and docs.

## Best Possible Solution

Add a backwards-compatible recurring cron lifetime field in the core scheduler contract, validate it for recurring schedules only, enforce it before due execution and startup catch-up, persist the chosen remove/disable outcome, surface it through CLI/RPC/agent tool/UI/docs, and cover both expiration actions with focused cron service and protocol tests.

## Reproduction Assessment

Not applicable as a runtime bug, but there is a high-confidence feature absence check: current main has no accepted cron expiration field or CLI flag, and enabled recurring jobs reschedule instead of expiring, removing, or disabling themselves.

## Solution Assessment

Unclear pending maintainer API choice. The direction belongs in core because persisted cron job lifecycle and gateway scheduling are core-owned, but maintainers should choose the public field names and expiration action semantics before implementation.
