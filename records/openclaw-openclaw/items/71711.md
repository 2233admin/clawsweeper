## Summary

Keep open. Current main still has no recurring cron expiration contract, and the request is concrete core scheduler work rather than a ClawHub/plugin-only idea. Because it changes persisted job shape, Gateway RPC, CLI, agent-tool, UI, docs, default expiration behavior, and overlaps the open agent-facing scheduling RFC, maintainers should settle the public contract before implementation.

## What This Changes

Add an optional per-job `activeUntil`/`expiresAt` lifecycle field for recurring `every` and `cron` schedules so they can stop, remove, or disable themselves after a configured timestamp.

## Best Possible Solution

Define a backward-compatible recurring-job lifecycle contract in the core cron model, validate it only for recurring schedules, enforce expiry before due execution and during maintenance/startup recomputation, persist remove/disable results, emit/audit expiration, and expose identical semantics through Gateway RPC, CLI add/edit/list/show, agent tools, Control UI, and public docs with focused scheduler/protocol/CLI/UI tests.

## Reproduction Assessment

Not applicable as a runtime bug; this is a feature request. The absence is high-confidence because current main has no accepted expiration field or user-facing surface, and recurring jobs keep computing future `nextRunAtMs` values.

## Solution Assessment

Unclear pending maintainer API choice. The direction belongs in core because cron job lifecycle is scheduler-owned persisted state, but maintainers should choose naming, default action, audit semantics, and #71712 alignment before a fix PR starts.
