## Summary

Keep open. Current main still does not implement per-job expiration for recurring cron jobs: the cron schema/types, CLI add/edit surfaces, agent cron tool, Control UI/macOS models, docs, and scheduler lifecycle all still only model one-shot cleanup through deleteAfterRun while recurring every/cron jobs compute a nextRunAtMs. Related #71712 is also still open and depends on recurring expiration semantics, so this needs maintainer API/product design rather than cleanup closure.

## Best Possible Solution

Keep this open and decide the public cron lifecycle contract first: canonical field name, timestamp parsing/timezone behavior, onExpire action/default, audit/task event shape, and interaction with agent-facing scheduling in #71712. Then implement additively across cron types/schema/validators, store normalization, scheduler expiration handling, CLI add/edit/list/show, agent cron tool, Control UI/macOS surfaces, tests, and public docs.

## Risks / Open Questions

- The issue proposes local-time interpretation for timezone-naive activeUntil, while current cron docs say offset-less --at timestamps are UTC unless --tz is supplied; maintainers should settle this before adding a public field.
- activeUntil versus expiresAt and onExpire remove versus disable would become Gateway protocol, CLI, UI, and docs contract, so naming/defaults need product/API review.
- Expiration should be designed with run history, manual cron run behavior, task/audit records, scheduler reloads, and related #71712 agent-created scheduling constraints in mind.
