## Summary

Keep this PR open. Current main still does not implement the shared proxy-fetch circuit breaker proposed here, and the PR discussion identifies real remaining design/security work rather than an obsolete change. The PR is useful, but it needs a rebase and policy-aware revision before it can land.

## Best Possible Solution

Keep this PR open as useful but not mergeable as-is. The best path is a rebased implementation that adds proxy circuit-breaker behavior at the shared proxy helper/plugin-facing surface and active Discord/Telegram paths, preserves managed-proxy fail-closed semantics, allows only one half-open probe per cooldown, fixes direct-error attribution, splits unrelated rate-limit regex changes, and adds focused regression tests for fallback policy, cooldown, single-probe behavior, and recovery.

## Risks / Open Questions

- Security review: the branch's direct fallback after proxy connection errors can bypass the current operator-managed proxy contract, which is used for egress control and SSRF/audit defense.
- Correctness review: the review comments on the PR describe direct-error attribution and half-open concurrency problems that could delay proxy recovery or preserve connection bursts under load.
- Integration risk: the PR diff targets older `src/discord/...` paths, while current main has Discord under `extensions/discord/...`, so it needs a rebase and adaptation to current plugin boundaries.
- Scope risk: the PR also changes unrelated rate-limit matching strings, which should be split out or separately justified from the proxy circuit-breaker work.
