## Summary

Keep open. Current main has not implemented the PR's central proxy circuit-breaker behavior: proxy fetch helpers still dispatch through `ProxyAgent` / `EnvHttpProxyAgent` without circuit state, direct fallback on connect failures, or cooldown tests. The PR is stale against the current plugin layout and is not ready to land as-is because review comments flagged real circuit-state bugs, and the proposed direct fallback has security-policy implications for deployments that require proxy egress.

## Best Possible Solution

Keep the PR open as useful but not mergeable. The best path is a rebased current-main patch that implements a policy-aware proxy circuit breaker in the shared proxy helper and current Discord plugin surfaces, preserves required managed-proxy security semantics, fixes the reviewed half-open/direct-error bugs, and adds focused regression tests for fallback, cooldown, single-probe behavior, and recovery.

## Risks / Open Questions

- The PR's direct fallback on proxy connection errors may be a security regression for deployments that intentionally require all runtime egress through a filtering proxy; the implementation needs an explicit policy split between optional channel/debug proxies and managed security proxy routing.
- The supplied PR diff includes unrelated rate-limit regex changes in agent, cron, and memory retry classification files; those should be separated or justified before review.
- The PR targets older `src/discord/...` paths, while current Discord code lives under `extensions/discord/...`, so it needs a rebase and adaptation to the current plugin boundary.
- The bot review comments describe correctness issues that should be fixed before merge: only record proxy failures when the proxy path was actually used, and gate half-open probes so concurrent callers do not all retry the dead proxy.
