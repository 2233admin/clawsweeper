## Summary

Keep this PR open. Current main still lacks the proposed shared proxy-fetch circuit breaker, and the PR branch is not mergeable as-is because it is stale against current plugin paths and conflicts with the managed proxy fail-closed security contract unless fallback is made policy-aware.

## Best Possible Solution

Keep this PR open as useful but not mergeable as-is. The best path is a rebased, policy-aware implementation that adds proxy connection backoff at the shared proxy helper/plugin-facing surface and current Discord/Telegram call sites, preserves managed-proxy fail-closed semantics, permits only one half-open probe per cooldown, records failures only for requests that actually used the proxy, splits unrelated rate-limit matcher changes, and adds focused tests for fallback policy, cooldown, single-probe behavior, recovery, and managed proxy behavior.

## Risks / Open Questions

- Unconditional direct fallback after proxy connection errors can bypass `proxy.enabled=true` operator egress policy and violate the documented fail-closed security model.
- The PR branch’s gateway catch-block can attribute direct-fetch connection failures to the proxy circuit, extending blackout even when the proxy was not used.
- The half-open implementation described in review comments can allow multiple concurrent probes, recreating the connection burst the circuit breaker is intended to prevent.
- The branch includes unrelated rate-limit matcher changes in agent/cron/memory files, which should be split from the proxy circuit-breaker fix.
- The branch targets stale Discord paths and needs rebase onto the current bundled plugin layout before maintainers can evaluate it against current tests.
