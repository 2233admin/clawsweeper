## Summary

Keep open. Current main still models the macOS app Launch at login setting as live launchd state only: no persisted user intent, no stale/unloaded/repair-needed status, and no Sparkle/post-update reconciliation path. The adjacent merged #40213 KeepAlive fix is shipped, but it does not address the reported update-time persistence/UI drift problem.

## Best Possible Solution

Keep this issue open and fix the macOS app Launch at login contract in the app-owned code path: persist the user's intended launch-at-login preference separately from live launchd state, model disabled versus stale/unloaded/repair-needed states, reconcile or re-bootstrap `ai.openclaw.mac` on app startup and after Sparkle updates, and add focused tests for stale plist, unloaded job, UI reconciliation, and post-update repair behavior.
