## Summary

Keep this PR open for normal maintainer review; current main has not already implemented the PR, and the current branch still has merge-blocking UI issues. The follow-up commit restored the earlier `loadOverview(opts)` and reactive `@state()` regressions, but the new approval log entry is written outside the canonical event buffer and the branch is stale against current main's localized approval UI.

## What This Changes

The PR changes the Control UI approval modal labels, removes a resolved approval from the local queue using a response/fallback id, logs a local approval-resolved event, dispatches a DOM event, and reloads overview data after approval resolution.

## Best Possible Solution

Rebase the PR onto current main, keep the localized approval-label path by updating locale keys instead of hardcoding English strings, and route any new approval-resolution log entry through the same event-buffer flow used for gateway events. If the resolved canonical id is required, update and test the gateway response contract rather than relying on an id field that is not returned today.

## Reproduction Assessment

Yes for the review findings. Source inspection of the PR head shows the new log entry only mutates `eventLog`, while current main overwrites visible logs from `eventLogBuffer`; comparing the PR head with current main also shows the label changes bypass current i18n keys.

## Solution Assessment

No. The approval-flow polish is reasonable, but this branch is not the narrowest maintainable implementation until it preserves the canonical event-log buffer and current localization surface after rebase.
