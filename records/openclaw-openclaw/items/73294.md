## Summary

Keep this PR open. Current main still does not implement the response-watchdog behavior, the two predecessor PRs are closed unmerged, and this PR remains the active narrow implementation candidate; the patch looks directionally sound but needs maintainer review for CI, docs, and notification/noise expectations.

## What This Changes

The PR changes ACP/subagent task creation and the task registry so non-streamed completion-visible runs opt into state-change notifications, newly created running state-change tasks emit an immediate start update, fallback-queued terminal delivery records `session_queued`, and tests cover those cases.

## Best Possible Solution

If maintainers want this watchdog behavior, land an amended version that keeps the new notifications limited to the intended ACP/subagent paths, preserves `session_queued` only when fallback queueing succeeds, updates the background-task docs to match the new defaults, and gets focused task-registry/ACP/subagent proof plus green relevant CI or an explained unrelated-failure record.

## Reproduction Assessment

Yes. A high-confidence source inspection against current main reproduces the missing behavior: ACP/subagent task creation does not pass `state_changes`, `createTaskRecord` does not emit a running update, and the direct-delivery fallback path still records `failed` after successful fallback queueing.

## Solution Assessment

Yes with amendments. The PR uses the existing `notifyPolicy`, requester-origin, idempotency, and fallback-delivery seams rather than adding a Discord-specific path, but it should align docs and clear or explain CI before merge.
