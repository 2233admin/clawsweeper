## Summary

Keep this PR open. Current main still does not implement BlueBubbles tapback/reaction exec approval resolution: BlueBubbles reactions and tapback-shaped messages are forwarded as agent-visible system events, while approval resolution remains available through text `/approve` or other native approval clients. The contribution is still useful, but the current PR design is security-sensitive and needs rework before merge.

## Best Possible Solution

Keep this PR open for rework. The best path is a BlueBubbles-native approval resolver that records the delivered approval prompt message GUID, resolves only reactions against tracked pending approvals, verifies the reactor with the same approver model as `/approve` or native approval clients, uses the narrow approval gateway/runtime SDK surface, handles both `processReaction` webhooks and any legacy tapback-message fallback, and adds focused tests for success, unauthorized senders, stale or untracked GUIDs, spoofed reply text, and gateway failure behavior.

## Risks / Open Questions

- The PR diff parses approval IDs from `replyToBody`, which can be free-form chat text unless the resolver is bound to a bot-sent approval prompt GUID and pending approval state.
- The PR adds raw `callGateway` to a public plugin SDK subpath; current main has narrower approval-specific gateway helpers that better match this use case.
- The PR handles tapback-shaped normal messages, but current BlueBubbles real reaction webhook payloads go through `processReaction`, so installations emitting reaction events would still not resolve approvals.
- Using BlueBubbles `commandAuthorized` alone can diverge from the normal `/approve` authorization path and from native approval approver checks.
