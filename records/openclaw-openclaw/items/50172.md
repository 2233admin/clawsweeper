## Summary

Keep open. Current main does not implement BlueBubbles tapback exec approval resolution, and this PR is not obsolete. It remains a plausible contribution, but the current PR design needs security and functional rework before it should land.

## Best Possible Solution

Keep this PR open for author or maintainer rework. The best path is a BlueBubbles-native approval implementation that records the delivered approval prompt message GUID, resolves tapbacks only against tracked pending approvals, verifies the reactor is authorized for that approval, uses the existing narrow approval runtime/SDK surfaces instead of exporting raw `callGateway`, and covers both real reaction webhooks plus tapback-like message fallback with focused tests.

This means current main has not shipped the PR's intended BlueBubbles reaction approval behavior.

## Risks / Open Questions

- The PR diff parses approval IDs from `replyToBody`, which is untrusted/free-form chat text unless bound to a bot-sent approval message GUID and pending approval state.
- The PR proposes exporting raw `callGateway` from a public plugin SDK subpath, broadening arbitrary gateway RPC access for plugins instead of using a narrow approval runtime/capability.
- The implementation path in the PR touches `processMessage`, but current BlueBubbles reaction webhooks are handled by `processReaction`; real installations that emit reaction events would not resolve approvals through that branch.
- Using the BlueBubbles `commandAuthorized` flag alone can diverge from the normal `/approve` authorization path, which also considers explicit approval authorization and channel-specific approver rules.
