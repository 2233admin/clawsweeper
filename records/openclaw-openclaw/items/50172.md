## Summary

Keep this PR open. Current main still does not implement BlueBubbles tapback/reaction exec approval resolution, and this PR remains an active implementation candidate, but the current PR design has unresolved security and product-contract concerns from maintainer review. The central missing piece is GUID-bound approval resolution for delivered BlueBubbles approval prompts, instead of parsing replied-to chat text or broadening the plugin SDK with raw gateway calls.

## Best Possible Solution

Keep this PR open for rework. The best path is a BlueBubbles approval resolver that records the delivered approval prompt message GUID, resolves only reactions against tracked pending approvals, verifies the reactor with the same approver model used by `/approve` or native approval clients, uses the narrow `resolveApprovalOverGateway` SDK surface, covers both real reaction webhooks and any legacy tapback-message fallback, and adds focused tests for success, unauthorized senders, stale/untracked GUIDs, spoofed reply text, and gateway failure behavior.

## Risks / Open Questions

- Merging the PR as currently described could let a tapback resolve an exec approval based on spoofable or forwarded replied-to text rather than a tracked bot-sent approval prompt GUID.
- The PR’s raw `callGateway` SDK export is broader than the approval-specific `resolveApprovalOverGateway` surface already present on main, increasing plugin SDK privilege surface beyond the feature’s stated need.
- The PR handles tapback-shaped normal messages but, based on current main’s webhook routing and the review comments, still needs explicit coverage for real BlueBubbles `processReaction` webhook events.
- BlueBubbles command authorization alone may not match the native approval approver model used by `/approve`, Discord, Slack, Telegram, and Matrix approval clients.
