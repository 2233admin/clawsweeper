## Summary

Keep open. Current main still full-reindexes changed session transcripts, so the performance problem is not obsolete, but this PR is not merge-ready because it is dirty against main, edits the removed `src/memory` layout, and the proposed incremental path introduces concrete session-memory correctness regressions.

## What This Changes

The PR adds hash-based incremental indexing for session files, chunk offset schema/metadata, and session benchmark/integration tests under the old `src/memory` layout.

## Best Possible Solution

Port the incremental session-indexing idea into the current memory-core plugin and memory-host SDK boundaries, preserve keyword-only session indexing, derive offsets without forking chunking semantics, update `chunks`, vector rows, and `chunks_fts` consistently, and keep regular `MEMORY.md` indexing behavior unchanged.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve FTS-only session indexing:** `src/memory/manager-embedding-ops.ts:828-833`
  - body: The new session path returns as soon as there is no embedding provider. Current `indexFile` still writes FTS-only chunks in that mode, and session sync now calls this method for session files, so users without an embedding provider would stop getting keyword-searchable session transcripts.
  - confidence: 0.91
- **[P2] Refresh FTS rows when metadata moves:** `src/memory/manager-embedding-ops.ts:930-933`
  - body: The hash-match branch updates line and offset metadata only in `chunks`. Keyword search reads line ranges from `chunks_fts`, so reused chunks after truncation or line shifts can still return stale citations unless the matching FTS row is replaced or updated too.
  - confidence: 0.9
- **[P2] Reuse existing chunk sizing semantics:** `src/memory/internal.ts:506`
  - body: The offset chunker counts raw UTF-16 length, while the current chunker uses weighted sizing and surrogate-safe splitting for CJK-heavy text. Switching session sync to this helper can produce different chunks and hashes for Unicode transcripts, defeating reuse and changing chunk boundaries.
  - confidence: 0.82
- **[P3] Correct the overlap offset rewind:** `src/memory/internal.ts:488-489`
  - body: After `flush()` advances by `currentChars - 1`, `carryOverlap()` subtracts an overlap count that still includes the synthetic newline. With overlap enabled, subsequent `startOffset` values can shift one character early, so stored offsets no longer point at the chunk text.
  - confidence: 0.76
