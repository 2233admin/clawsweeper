## Summary

Keep open. Current main still does not strip echoed inbound metadata in the shared outbound delivery normalization path, so this PR is not obsolete or implemented on main. The linked bug report remains open and this PR is the implementation candidate, but the proposed approach still needs maintainer refinement because applying the broad `stripInboundMetadata()` helper to every assistant outbound payload can remove legitimate timestamp-like or quoted metadata content.

## Best Possible Solution

Keep this PR open and review it as the candidate fix for #39847. The best path is to add outbound sanitization before reply directive parsing in the shared outbound payload plan, but use a leading-only or assistant-safe sanitizer, preserve legitimate timestamp-prefixed and quoted metadata text, and add focused regression tests for echoed metadata removal plus non-echo preservation. After that lands, the linked bug can close as implemented.

## Risks / Open Questions

- Applying `stripInboundMetadata()` to every assistant outbound payload can delete intentionally quoted metadata examples or debug output because the helper strips matching metadata blocks throughout the text, not only at an echoed prompt prefix.
- Current main's `stripInboundMetadata()` still strips a leading OpenClaw-format timestamp even when no sentinel is present; the PR's second commit addresses that on the branch, but the mainline helper/tests still show why outbound use needs explicit regression coverage.
