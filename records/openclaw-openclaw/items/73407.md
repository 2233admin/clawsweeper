## Summary

Keep open: current `main` and latest release still carry the WhatsApp plugin -> Baileys -> pinned GPL-3.0 `libsignal-node` dependency path on a published plugin surface, so this is a real compliance/supply-chain concern that needs maintainer/legal direction rather than conservative auto-close.

## What This Changes

The issue reports that the WhatsApp plugin pulls in a GPL-3.0 libsignal dependency, creating a licensing/compliance risk for OpenClaw distribution.

## Best Possible Solution

Keep the issue open until maintainers choose a license posture for the published WhatsApp plugin: replace or isolate the GPL-3.0 libsignal dependency, explicitly approve and document the distribution model, or disable/remove the published WhatsApp plugin path if needed.

## Reproduction Assessment

Yes. This is reproducible by inspecting current `main` package metadata, the lockfile resolution, upstream `libsignal-node` package metadata, and the WhatsApp runtime Baileys socket path; no live WhatsApp session is needed.

## Solution Assessment

Unclear. Keeping the issue open for maintainer/legal direction is the right review outcome, but the actual technical fix depends on the selected policy, so an automated repair should not choose between dependency replacement, isolation, documentation, or removal.
