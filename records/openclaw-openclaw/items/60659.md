## Summary

Keep this issue open. Current main still does not provide a per-agent or per-route-binding session reset policy. The implemented reset surface is global, chat-type, and channel override only; route bindings can only override DM scoping, and the reset resolver is called before any agent/binding-specific reset policy is available.

## Best Possible Solution

Keep this issue open for a generic core config/runtime change. Add a documented reset-policy override at agents.list[] and/or bindings[].session, extend the strict TypeScript and zod schemas plus config docs/help, then resolve the effective reset policy after inbound routing identifies the target agent and binding. Tests should cover precedence against global session.reset, resetByType, and resetByChannel, including the one-shot workflow case described in the issue.

## Risks / Open Questions

- Closing now would drop a still-missing mixed-workflow isolation feature for deployments that need long-lived conversational agents and fresh one-shot processing agents in the same gateway.
- The eventual implementation needs an explicit precedence decision across global session.reset, resetByType, resetByChannel, binding-level policy, and agent-level policy.
- ACP oneshot support could be confused with this request, but it only covers ACP-bound runtime behavior and does not solve generic route-binding or embedded-agent session reset policy.
