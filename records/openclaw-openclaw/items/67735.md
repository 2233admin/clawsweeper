## Summary

Keep this open. Current main still implements WebChat/Control UI as a selected-session surface that defaults to the agent main session, while `identityLinks` canonicalization only runs when routing has a concrete direct peer. The open #61145 PR is a relevant implementation candidate, but it is unmerged and the latest named Telegram-account confirmation adds scope that still needs maintainer product/API review.

## What This Changes

The issue requests WebChat/Control UI support or documentation for using `session.identityLinks` to bind `webchat:default` to the same canonical direct session as linked Telegram DMs instead of always defaulting to `agent:main:main`.

## Best Possible Solution

Review #61145 as the current focused implementation candidate and decide the intended product/API boundary. The desired end state is either an explicit opt-in WebChat/Control UI identity or session mode that resolves through existing `dmScope` and `identityLinks` rules, including the named-account case if intended, or a deliberate docs update that states WebChat remains main-session/operator scoped unless users manually select another session.
