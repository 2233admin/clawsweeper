## Summary

Keep open. Current main still does not implement this PR's intended Telegram waiting reaction or per-message `onQueued` queue callback chain, and the PR remains an open, unmerged implementation candidate with useful review/security follow-up already incorporated on its branch.

## Best Possible Solution

Keep this PR open for maintainer review or an updated branch. The best path is to rebase the callback-based implementation onto current main, add `waiting` to status-reaction types/defaults/config/schema/docs/tests and Telegram variants, extend the command-queue and reply-option types with `onQueued`, thread it through both session and global lanes with a once guard, wire Telegram waiting-to-thinking transitions in the dispatcher, and keep the Telegram account+chat 429 backoff covered by tests.
