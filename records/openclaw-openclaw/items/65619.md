## What This Changes

The issue asks OpenClaw to fix a macOS Gateway regression where explicit `gateway.bind="loopback"` can resolve to `0.0.0.0` after a failed preflight probe and then fail startup with the loopback safety guard.

## Best Possible Solution

Keep explicit `gateway.bind="loopback"` pinned to `127.0.0.1` when the loopback preflight probe fails, leave `auto` and non-loopback fallback policy intact, and add focused regression coverage plus a user-facing changelog credit.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to make the loopback preflight return false, then resolve explicit `gateway.bind="loopback"`; current main returns `0.0.0.0`, and `resolveGatewayRuntimeConfig` throws the reported error.

## Solution Assessment

Yes, the narrow resolver fix is the best maintainable direction. The runtime safety guard should remain fail-closed; the bug is that explicit loopback should not be downgraded to a network bind before reaching that guard.
