## Summary

Keep open. The discussion already corrected the model-pricing theory and localized the original reproducible delay to Gmail watcher startup. Current main still awaits `startGmailWatcherWithLogs()` before channel startup, and normal CLI startup still waits for sidecars by default, so the remaining user-visible startup delay is actionable rather than implemented or obsolete.

## Best Possible Solution

Keep this open and fix the owner path in startup sidecars: start Gmail watcher bootstrap in the background, log that it is continuing asynchronously, preserve existing `OPENCLAW_SKIP_GMAIL_WATCHER` and error handling, and add a regression test that injects a slow `startGmailWatcherWithLogs` and proves channel startup/readiness is not delayed. Separately, triage the latest Bonjour/ACP Windows log if it persists after the sidecar/Gmail fix.
