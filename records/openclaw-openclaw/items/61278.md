## Summary

Keep open. Current main still awaits Gmail watcher startup in the sidecar startup path, and the issue discussion includes a reporter-confirmed bisection showing Gmail watcher startup, not model-pricing, caused the multi-minute delay.

## What This Changes

The issue requests non-blocking Gateway startup when hook or sidecar initialization, especially Gmail watcher startup, performs slow network-dependent work.

## Best Possible Solution

Fix the Gmail-owned startup path narrowly: start Gmail watcher bootstrap in the background from sidecar startup, make that state visible in logs or startup trace, preserve the existing opt-out and error handling, and add a regression test proving slow watcher startup does not delay channel startup or Gateway readiness.

## Reproduction Assessment

Yes. Configure `hooks.enabled=true` with `hooks.gmail.account`, make Tailscale setup or `gog gmail watch start` slow or timing out, and start Gateway through the normal CLI path; current main awaits that watcher before `gateway ready`.

## Solution Assessment

Yes for the localized bottleneck. Backgrounding the Gmail watcher call from `sidecars.gmail-watch` is the narrowest maintainable fix; a broad hook timeout API should wait until multiple hook owners need a shared startup contract.

## Repair Work Prompt

Fix the Gmail watcher startup bottleneck for https://github.com/openclaw/openclaw/issues/61278. Current main awaits `startGmailWatcherWithLogs` inside `sidecars.gmail-watch`, and `startGmailWatcher` can wait on Tailscale setup plus `gog gmail watch start` for up to 120s. Make normal Gateway startup continue without waiting for Gmail watcher bootstrap completion. Keep the change scoped to the Gmail watcher startup path, preserve `OPENCLAW_SKIP_GMAIL_WATCHER`, existing error/warn logging, and reload restart semantics unless explicitly tested. Prefer backgrounding the Gmail watcher call from `startGatewaySidecars` or adding a dedicated lifecycle helper for background startup; emit a clear log/trace line that Gmail watcher bootstrap is running in the background. Add a regression test with a slow or never-resolving `startGmailWatcherWithLogs` mock proving channels/readiness continue without waiting. Do not attempt to fix the separate #74325 `sidecars.channels`/Bonjour stall in this PR. Add a user-facing changelog entry with proper contributor attribution if runtime behavior changes.
