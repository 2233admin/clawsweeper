## Summary

Keep open. The discussion already ruled out model-pricing as the blocking path and localized the original Windows delay to Gmail watcher startup. Current main still awaits Gmail watcher bootstrap before channel startup and default CLI startup still waits for sidecars, so the user-visible startup delay remains actionable.

## Best Possible Solution

Keep this issue open and fix the owner startup path: make Gmail watcher bootstrap non-blocking during Gateway startup, preserve OPENCLAW_SKIP_GMAIL_WATCHER and existing error/skip logging, emit a clear background-start log, and add a regression test proving slow Gmail watcher startup does not delay channel startup/readiness. Treat the newer Bonjour/ACP Windows trace as a possible separate follow-up if it still reproduces after this change.
