## Summary

Keep open. Current main has the browser plugin, CLI, gateway method, and agent guidance for tab/profile control, but the Control UI still has no visible browser panel or login handoff surface; the maintainer discussion also explicitly identifies this as the canonical browser-login handoff work for Google Meet-style flows.

## What This Changes

The issue requests a visible Control UI browser handoff surface for managed browser login workflows, including browser status, active tabs, tab actions, direct access to the managed session, and resume-after-login support.

## Best Possible Solution

Add a first-class Control UI browser surface backed by the existing browser plugin/gateway contracts, with explicit profile/node/tab identity, safe status and tab actions, a manual-login-required state the agent can report, and a resume path that keeps the same managed browser state after the user completes login or permissions.

## Reproduction Assessment

Not applicable as a bug reproduction, but there is a high-confidence verification path for the feature gap: current Control UI navigation, active-tab loading, rendering, and docs contain no browser panel while browser login docs still point users to the agent or CLI.

## Solution Assessment

Yes at the product level: a generic browser handoff panel is the right direction, not a Google Meet-specific workaround. The implementation should keep browser control gated through existing operator/admin scopes and reuse browser.request or a typed additive browser status contract rather than adding hidden plugin-specific UI bypasses.
