## Summary

Keep open. Current main still lacks the central diagnostic-owned recovery path: stuck processing sessions emit `session.stuck` warnings/events, but there is no `stuckSessionAbortMs`, `session.aborted`, run abort, or queue cleanup wired into the diagnostic heartbeat. Related PR #73243 remains open and unmerged for the narrower diagnostics-abort work, while this issue also carries Feishu production logs and broader startup compaction, transcript growth, resource-limit, and channel-liveness concerns that are not all fully resolved on main.

## Best Possible Solution

Keep this issue open and use it as the production outage umbrella until the central recovery path lands. The best maintainer path is to review/land or replace #73243 for diagnostic-owned abort and queue cleanup, keep startup/large-transcript recovery aligned with #64585, #71555, and #60858, then split any remaining Feishu liveness and Linux service memory guardrail work into narrower follow-ups if it is not fixed by the diagnostics change.
