## Summary

Keep open. Current main has useful mitigations for stale lane release, trajectory bounds, Feishu probe timeouts, and Linux memory guidance, but the central outage class is not fully resolved: active embedded runs, active reply work, and active lane tasks still remain warning-only by default, and startup compaction/recovery policy is still split across open follow-ups.

## What This Changes

The issue requests automatic recovery for stuck gateway sessions plus startup compaction, trajectory bounds, memory guidance, Feishu probe isolation, and service PATH repair.

## Best Possible Solution

Keep this issue as the production-outage umbrella until maintainers land or revise the diagnostics abort/reset policy, then split remaining startup compaction, Feishu probe behavior, service memory tuning, and PATH cleanup into narrower owner-scoped follow-ups with clear acceptance criteria.

## Reproduction Assessment

Yes for the central source-level gap: current code and tests show active embedded runs, active reply work, and active lane tasks remain warning-only after the stuck-session threshold. The complete Feishu outage sequence is supported by issue logs but was not live-reproduced in this read-only review.

## Solution Assessment

No, current main is not the complete solution. The safest direction is a configurable diagnostics-owned abort/reset path for confirmed stuck active runs, plus focused follow-ups for startup compaction and the remaining operational mitigations.
