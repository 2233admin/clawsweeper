## Summary

Keep open. Current main has useful mitigations for stale lanes, trajectory sidecar growth, Feishu probe timeouts, service PATH repair, and Linux memory guidance, but it still does not implement the central requested active stuck-session recovery path: diagnostics warn and invoke conservative recovery, while active embedded runs, active reply work, and active lane tasks remain warning-only by default.

## What This Changes

The issue requests automatic recovery for stuck Feishu gateway sessions, startup compaction for oversized existing transcripts, trajectory file bounds, gateway memory-limit guidance, non-blocking Feishu ping behavior, and service PATH repair.

## Best Possible Solution

Keep this as the outage umbrella until maintainers settle and land a configurable, race-safe active stuck-session abort/reset policy, then split the remaining startup compaction, Feishu probe, memory tuning, PATH repair, and transcript/trajectory concerns into focused follow-ups where current main does not already cover them.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests show diagnostics can detect stuck processing sessions, but active embedded runs, active reply work, and active lane tasks are not aborted or reset by the default heartbeat path. The exact Feishu production chain remains log-backed rather than live-reproduced here.

## Solution Assessment

No, current main is not the complete solution because it only releases stale lanes when no active work is present and leaves active stuck work warning-only. The narrow maintainable direction is a diagnostics-owned configurable abort/reset path for confirmed stuck active runs, with startup compaction and channel-specific issues split after that policy is reviewed.
