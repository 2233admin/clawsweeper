## Summary

Keep open. Current main has partial mitigations for stale session lanes, trajectory growth, and Feishu startup probes, but active embedded runs, active reply work, and active lane tasks still remain warning-only by default, and the broad production outage is already tied to open diagnostics abort follow-up work.

## What This Changes

Request auto-recovery for stuck Feishu/gateway sessions plus startup compaction, trajectory limits, memory-limit guidance, Feishu probe isolation, and service PATH repair.

## Best Possible Solution

Keep this as the production outage umbrella until maintainers land or reject a configurable, race-safe stuck-session abort/reset policy. After that, split remaining startup compaction, service memory guidance, Feishu probe, trajectory, and PATH cleanup into narrower trackers if still reproducible.

## Reproduction Assessment

Yes for the central code-path gap: current source and tests show active embedded runs, active reply work, and active lane tasks remain warning-only after the stuck-session threshold. The full Feishu outage sequence has reporter logs but was not live-reproduced here.

## Solution Assessment

No, current main is not the complete solution because it only releases stale lanes with no active work. The likely maintainable direction is the diagnostics-owned abort threshold represented by #73243, plus separate follow-ups for the non-diagnostics sub-requests.
