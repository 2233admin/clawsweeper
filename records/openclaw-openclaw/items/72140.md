## Summary

Keep open. Current main avoids showing a known-stale pre-compaction total by rendering context usage as unknown when `totalTokensFresh` is false, but it still does not satisfy the requested post-compaction display because the shared status formatter ignores the latest compaction checkpoint `tokensAfter`; related PR #72285 remains the right open implementation path to review.

## What This Changes

The issue requests that shared status output report the current post-compaction context size after compaction instead of a stale pre-compaction total.

## Best Possible Solution

Review and land #72285 or an equivalent narrow shared-formatter change that prefers fresh totals, falls back to the latest valid compaction checkpoint `tokensAfter` when totals are stale or missing, and keeps focused regression coverage for the post-compaction status line.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with a stale `totalTokens`, `totalTokensFresh: false`, and a latest compaction checkpoint containing `tokensAfter`; current main ignores the checkpoint and renders unknown context instead of the post-compaction value.

## Solution Assessment

Yes. The narrow maintainable solution is a shared status formatter fallback using the existing freshness marker and checkpoint data, not a channel-specific WhatsApp workaround.
