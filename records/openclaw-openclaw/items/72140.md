## Summary

Keep open. Current main has not implemented the requested post-compaction status display: the shared formatter suppresses stale totals as unknown but still does not use the latest compaction checkpoint `tokensAfter`; open PR #72285 remains the active implementation path.

## What This Changes

The issue asks for `/status` and post-message status cards to show the current post-compaction context size instead of stale pre-compaction totals.

## Best Possible Solution

Keep this open until PR #72285 is reviewed and either landed or replaced. The desired end state is a shared status formatter that prefers fresh totals, falls back to `tokensAfter` from the actual latest valid compaction checkpoint when totals are stale, and has focused regression coverage for stale post-compaction status output.

## Reproduction Assessment

Yes. A source-level reproduction is clear: call `buildStatusMessage()` with stale `totalTokens`, `totalTokensFresh: false`, and a latest compaction checkpoint carrying `tokensAfter`; current main renders unknown context because the formatter never reads that checkpoint.

## Solution Assessment

Yes. The narrow maintainable solution is to fix the display fallback at the shared status boundary and land PR #72285 or an equivalent patch after addressing its checkpoint-selection details; a broader migration is not needed for this issue.
