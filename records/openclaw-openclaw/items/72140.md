## Summary

Keep open. Current main avoids displaying a known-stale pre-compaction total by rendering the context as unknown, but it still does not satisfy the requested post-compaction display because the shared status formatter never falls back to the latest checkpoint `tokensAfter`; the related open PR #72285 is still the right implementation path to review.

## What This Changes

The issue requests that `/status` and shared status cards show the current post-compaction context size instead of a stale pre-compaction total.

## Best Possible Solution

Review and land #72285 or an equivalent narrow change that makes the shared status formatter prefer fresh totals, fall back to the latest valid compaction checkpoint `tokensAfter` only when totals are stale or missing, and keeps focused regression coverage for the post-compaction display.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with stale `totalTokens`, `totalTokensFresh: false`, and a latest compaction checkpoint containing `tokensAfter`; current main renders unknown context because the formatter never reads the checkpoint value.

## Solution Assessment

Yes. The narrow maintainable solution is a shared formatter fallback using the existing freshness marker and checkpoint data, not a channel-specific WhatsApp workaround or a startup migration.
