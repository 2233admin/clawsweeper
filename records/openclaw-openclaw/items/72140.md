## Summary

Keep open. Current main suppresses stale pre-compaction totals, but the shared status formatter still does not fall back to the latest compaction checkpoint `tokensAfter`, so it can render unknown context instead of the requested current post-compaction size; open PR #72285 remains the targeted implementation path.

## What This Changes

The issue requests that shared status output show the current post-compaction context size, with the pre/post compaction delta attached to compaction metadata instead of stale pre-compaction context totals.

## Best Possible Solution

Review and land PR #72285, or an equivalent narrow shared formatter change that prefers fresh totals, falls back to the latest valid checkpoint `tokensAfter` only when totals are stale or missing, and preserves regression coverage for post-compaction status display.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with `totalTokensFresh: false`, stale `totalTokens`, and a latest compaction checkpoint containing `tokensAfter`; current main does not read that checkpoint and therefore cannot render the requested post-compaction count.

## Solution Assessment

Yes. The best fix is a small shared status formatter fallback using the existing freshness marker and checkpoint data, rather than a WhatsApp-specific workaround or a broader session migration.
