## Summary

Keep open. Current main suppresses stale pre-compaction totals by rendering unknown context when `totalTokensFresh` is false, but the shared status formatter still does not fall back to the latest compaction checkpoint `tokensAfter`, so the requested post-compaction context display remains unresolved. The related open PR #72285 is the right implementation path to review rather than closing this early.

## What This Changes

The issue asks for `/status` and shared post-message status cards to show the current post-compaction context size instead of stale pre-compaction totals after manual or automatic compaction.

## Best Possible Solution

Keep this open until #72285 or an equivalent narrow patch lands: update the shared status formatter to prefer fresh totals, fall back to the latest valid compaction checkpoint `tokensAfter` only when totals are stale or missing, keep the compaction delta visible separately, and add regression coverage for stale post-compaction status output.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to call `buildStatusMessage` with a session entry containing stale `totalTokens`, `totalTokensFresh: false`, and a latest compaction checkpoint with `tokensAfter`; current main renders `Context: ?/...` because it never reads that checkpoint.

## Solution Assessment

Yes. The best fix direction is the narrow status-boundary fallback using existing checkpoint data, not a broad storage migration; it should only apply when the cached total is stale or absent so fresh compaction writes remain authoritative.
