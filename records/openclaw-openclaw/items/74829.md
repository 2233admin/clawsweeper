## Summary

Keep this PR open, but the patch is not merge-ready. The underlying current-main regression is real: conformance is still routed through the full/adversarial expected-diagnostics assertion even though the published kitchen-sink package defines conformance as a clean runtime-only personality. However, this branch moves the full runtime config write before plugin install, which writes a plugin-owned channel id before the plugin is installed and can make install-time config validation fail earlier.

## What This Changes

The PR changes the kitchen-sink E2E harness so conformance no longer expects invalid-probe diagnostics, writes the config directory before writing runtime config, moves kitchen-sink runtime configuration before plugin install, and updates the prerelease-plan test guards.

## Best Possible Solution

Keep the conformance assertion on the no-error path, but split the harness setup so only the personality selection is present before install. Write the plugin-owned channel config after install and before enable/list/inspect, or filter install-time diagnostics separately, then add a focused regression guard for that ordering.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction path is the PR branch's `run_success_scenario`: write `channels.kitchen-sink-channel` before `plugins install`, then install reads and validates config before the kitchen-sink package exists, making the channel id unknown.

## Solution Assessment

No. The assertion direction is correct, but moving the whole runtime config before install is not the narrowest maintainable fix; pre-install setup should be limited to the personality setting or diagnostics should be filtered by phase.
