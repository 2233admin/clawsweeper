## Summary

Keep open. Current main still hardcodes the `sessions_send(timeoutSeconds=0)` A2A announce budget to 30 seconds and keeps the initial gateway agent RPC at 10 seconds. The newer configurable `agents.defaults.subagents.announceTimeoutMs` path only covers `subagent_announce` delivery, not the `sessions_send` A2A flow reported here.

## Best Possible Solution

Keep this issue open and fix the `sessions_send` A2A path directly: add an explicit announce timeout surface or config-backed default for `sessions_send`, decide how `timeoutSeconds=0` should bound the background wait/announce flow, remove or make configurable the 10s startup RPC budget if it remains part of the failure mode, and add focused tests covering fire-and-forget A2A replies delayed beyond 30 seconds. The existing subagent announce timeout implementation is useful precedent but is not sufficient by itself.
