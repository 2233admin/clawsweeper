## Summary

Keep this PR open. It addresses a real ACP run cleanup gap on current main, but the current branch still has a blocking fallback timer that can terminate active no-timeout or long-running ACP runs after 24 hours, which conflicts with the documented timeout contract and the gateway timeout resolver.

## What This Changes

The PR adds lifecycle-event cleanup for ACP mode "run" runtimes, passes the initialized runtime close handle into dispatch-failure cleanup, removes ACP run timeout values from backend runtime initialization options, and adds regression tests in the ACP spawn test file.

## Best Possible Solution

Keep the lifecycle end/error cleanup and dispatch-failure cleanup direction, but revise the fallback so it only closes after confirmed terminal task state or after an explicit timeout-derived grace period. The no-timeout path must remain valid, and the broader orphan-process recovery should stay with #68916.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is an ACP mode "run" spawn with runTimeoutSeconds: 0, or another run whose intended lifetime exceeds 24 hours, where no lifecycle terminal event reaches the new listener; the PR head will still close the runtime after the fixed fallback timer.

## Solution Assessment

No. The immediate lifecycle cleanup is the right narrow direction, but the fixed 24-hour fallback is not the best way to solve it because it bypasses existing timeout semantics; a terminal-state or explicit-timeout-based fallback is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Preserve no-timeout ACP runs in fallback cleanup:** `src/agents/acp-spawn.ts:1027-1029`
  - body: The new fallback timer calls cleanup("run-cleanup-timeout") after 24h without checking whether the child run is terminal. runTimeoutSeconds: 0 is documented and resolved as the no-timeout path, so a long-running ACP mode "run" can be closed while still active if the lifecycle event has not arrived. Gate the fallback on terminal task state or derive it from an explicit timeout plus grace period.
