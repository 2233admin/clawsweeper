## Summary

Keep this PR open. The lifecycle-driven ACP run cleanup and dispatch-failure cleanup are useful, but the current patch still has a blocking fallback policy bug: if the lifecycle terminal event is missed, it closes the run runtime after a fixed 24 hours without checking whether the task is actually terminal or whether the run was configured for no timeout or a longer timeout.

## What This Changes

The PR adds lifecycle end/error cleanup for ACP `mode: "run"` runtimes, passes the initialized runtime close handle into dispatch-failure cleanup, removes `runTimeoutSeconds` from ACP runtime initialization options, and adds colocated regression tests.

## Best Possible Solution

Keep the lifecycle end/error cleanup and dispatch-failure cleanup direction, but revise the fallback so it only closes after confirmed terminal task state or after an effective timeout-derived grace period; leave broader parent-reset and orphan-process recovery to #68916.

## Reproduction Assessment

Yes. A focused code-level reproduction is an ACP `mode: "run"` spawn with `runTimeoutSeconds: 0` or a long/default timeout where the lifecycle terminal event is missed; the PR head will close the runtime at the fixed 24-hour fallback instead of respecting the no-timeout or longer-timeout contract.

## Solution Assessment

No. The cleanup direction is narrow and maintainable, but the fixed wall-clock fallback is not the best fix because it bypasses existing timeout semantics; the safer path is task-terminal-state cleanup or fallback timing derived from the explicit/effective run timeout.
