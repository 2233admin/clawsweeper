## Summary

Keep this PR open. The change targets a real ACP run-session cleanup gap that current main still lacks, but the proposed fallback timer can close valid long-running or no-timeout ACP runs before the run is terminal, so the current patch should not merge as-is.

## What This Changes

The PR adds ACP run-mode lifecycle cleanup, passes the initialized runtime close handle into dispatch-failure cleanup, and adds regression tests for those paths.

## Best Possible Solution

Revise the PR so ACP run-mode runtimes close promptly on terminal lifecycle events and dispatch failures, while any fallback cleanup first proves terminal task state or derives its deadline from an explicit run timeout. Keep the broader parent reset and orphan-process recovery work in #68916.

## Reproduction Assessment

Yes. A high-confidence code-level reproduction is to spawn an ACP `mode: "run"` task with `runTimeoutSeconds: 0` or a timeout greater than 24h, miss the lifecycle terminal event, and advance the PR's fallback timer; the new timer closes the runtime without checking whether the run is terminal.

## Solution Assessment

No. Lifecycle-driven cleanup and dispatch-failure cleanup are the right direction, but the fixed 24h fallback is not the narrowest maintainable solution because it conflicts with current timeout semantics; the safer alternative is terminal-state-gated cleanup or a fallback tied to an explicit timeout plus grace.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve no-timeout ACP runs in fallback cleanup:** `src/agents/acp-spawn.ts:1031-1033`
  - body: The fallback timer always calls cleanup after 24h when no terminal lifecycle event is seen. `sessions_spawn` accepts `runTimeoutSeconds: 0` and the agent timeout resolver treats 0 as the no-timeout path, so this can close a legitimate long-running ACP run while it is still active. Gate this fallback on terminal task state or derive it from an explicit timeout plus grace.
