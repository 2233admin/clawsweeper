## Summary

Keep this PR open, but do not merge it as-is. It addresses a real ACP run-mode process leak that current main only handles through later task-maintenance cleanup, but the new fixed 24h fallback can close an active ACP run even when the caller selected the documented no-timeout path.

## What This Changes

The PR adds lifecycle-triggered cleanup for ACP `mode: "run"` sessions, passes the initialized runtime close handle into dispatch-failure cleanup, and adds regression tests in `src/agents/acp-spawn.test.ts`.

## Best Possible Solution

Revise the PR so completed ACP run sessions close promptly on terminal lifecycle events and dispatch failures, while any fallback cleanup first proves the run is terminal or respects an explicit run timeout; keep the broader parent-reset and orphan-process recovery work tracked separately in #68916.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Preserve no-timeout ACP runs in fallback cleanup:** `src/agents/acp-spawn.ts:1031-1033`
  - body: The fallback timer always calls cleanup after 24h when no lifecycle terminal event is seen. `sessions_spawn` documents `runTimeoutSeconds: 0` as the ACP no-timeout path, and the tool normalizes omitted/zero timeouts that way, so a legitimate long-running run can be closed while still active. Gate the fallback on terminal task state or derive it from an explicit timeout instead of using a fixed wall-clock close.
  - confidence: 0.86
