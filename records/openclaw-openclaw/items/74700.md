## Summary

Keep this PR open. It targets a real ACP run-session cleanup gap that current main still does not close immediately, but the proposed fixed 24h fallback can still terminate documented no-timeout ACP runs if the lifecycle terminal event is missed; the second commit does not address that fallback policy.

## What This Changes

The PR adds lifecycle-event cleanup for ACP `mode: "run"` runtimes, passes the initialized runtime close handle into dispatch-failure cleanup, stops forwarding `runTimeoutSeconds` to ACP backend runtime options, and adds regression coverage in `src/agents/acp-spawn.test.ts`.

## Best Possible Solution

Land a revised narrow PR that closes ACP run sessions on terminal lifecycle events and dispatch failures, but replaces the fixed wall-clock fallback with terminal task/session-state proof or an explicit timeout-plus-grace policy; update ACP timeout docs if backend runtime timeout forwarding is intentionally removed, and keep parent-reset and PID-level orphan recovery tracked in #68916.

## Reproduction Assessment

Yes. The code-review reproduction is high confidence: on PR head, a `mode: "run"` ACP spawn with omitted or `0` `runTimeoutSeconds` still schedules unconditional cleanup after 24h if no terminal lifecycle event is observed, despite the documented no-timeout path.

## Solution Assessment

No. The lifecycle and dispatch-failure cleanup direction is the right narrow surface, but the fallback cleanup is not the best fix until it preserves documented no-timeout runs and clarifies the timeout contract.
