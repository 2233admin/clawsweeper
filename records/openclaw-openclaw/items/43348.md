## Summary

Keep this PR open. Current main still does not contain the proposed `acp_send` agent tool, `sessions_cancel` agent tool, or `sessions.cancel` Gateway RPC. Main has adjacent ACP/session controls, including `sessions_send`, `sessions.steer`, `sessions.abort`, `/acp cancel`, task cancellation, and `AcpSessionManager.cancelSession()`, but those do not fully implement this PR's intended agent-facing surfaces. The PR also needs rework before merge because the proposed cancellation and fire-and-forget ACP paths are authorization/delivery sensitive.

## Best Possible Solution

Keep this PR open as a rework candidate. Rebase it onto current main, decide whether a separate `acp_send` tool is still needed beyond the current `sessions_send` behavior, and if agent-facing ACP cancellation is still desired, implement it through the existing session-resolution and ACP-control model with visibility/ownership checks, ACP-only target validation, protocol/tool-catalog/docs updates, and focused regression tests.

## Risks / Open Questions

- The proposed `sessions_cancel` path is authorization-sensitive: an agent-visible tool must reuse session resolution, visibility, ownership, sandbox, and A2A policy checks before cancelling by session key.
- The proposed `acp_send` path must reject non-ACP targets before fire-and-forget dispatch, otherwise output can be accepted into a normal/subagent session with no ACP callback delivery path.
- Adding `sessions.cancel` beside existing `sessions.abort`, `sessions.steer`, `/acp cancel`, task cancellation, and ACP runtime cancellation could fragment cancellation semantics unless protocol docs, scopes, catalog entries, and regression tests are aligned.
