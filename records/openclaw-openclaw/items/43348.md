## Summary

Keep open. Current main still does not implement the PR's two proposed agent tools (`acp_send`, `sessions_cancel`) or the proposed `sessions.cancel` Gateway RPC. Main has partial overlap through `sessions_send`, `sessions.abort`, `/acp cancel`, and ACP task cancellation, but those do not fully cover the PR's intended agent-facing ACP follow-up/cancel surface. The PR remains a plausible rework candidate, with security-sensitive authorization and target-validation issues to fix before merge.

## Best Possible Solution

Keep this PR open as a rework candidate. Rebase onto current main, decide whether a separate ACP-only `acp_send` tool is still needed beyond current `sessions_send` behavior, and if an agent cancellation tool is desired, implement it through the existing `sessions.abort`/ACP cancellation model or a clearly documented new RPC with session resolution, visibility and ownership checks, ACP-only target validation, protocol/docs/tool-catalog updates, and focused regression tests.

## Risks / Open Questions

- The PR's proposed `sessions_cancel` path is authorization-sensitive: without reusing current session resolution, visibility, ownership, and A2A policy checks, a model that learns another ACP session key could cancel a turn outside its intended scope.
- The PR's proposed `acp_send` path resolves visible sessions and dispatches fire-and-forget while relying on ACP callback delivery; it should reject non-ACP targets up front so output is not silently lost for normal/subagent sessions.
- Adding `sessions.cancel` beside shipped `sessions.abort`, `/acp cancel`, task cancellation, and reset/delete ACP cleanup could fragment cancellation semantics unless naming, docs, scopes, and behavior are reconciled.
