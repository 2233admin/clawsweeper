## Summary

Keep open. Current main still lacks the requested vLLM lifecycle and multi-endpoint setup flow, and the PR branch is not merge-ready because the new vLLM handler is not reached by production auth dispatch while unrelated security/build/test-infra scope remains mixed in.

## What This Changes

The PR adds vLLM endpoint management, multi-endpoint model refs, stale default cleanup, docs/tests, and additional A2UI bundling, secrets/zizmor, agents, and memory-test changes.

## Best Possible Solution

Keep the vLLM endpoint-management direction, rebase it onto the current plugin-provider setup, wire the vLLM-specific setup through the production auth path or a generic provider setup seam, keep endpoint/default cleanup in provider-owned code, and split A2UI, zizmor, secrets, and memory-test changes into separately reviewed work.

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P1] Route vLLM auth choices through the new handler:** `src/commands/auth-choice.apply.vllm.ts:114-121`
  - body: `applyAuthChoiceVllm` is added, but production `applyAuthChoice` never imports or invokes it before the generic plugin-provider handler. A normal `vllm` choice still follows the legacy plugin setup, so users skip the new endpoint menu and stale-default cleanup.

  - body: The fallback branch runs `pnpm -s dlx rolldown` and then still requires a persistent `.pnpm` rolldown CLI before running the bundle again. Layouts without `.pnpm` can fail after the temporary bundle succeeds, while pnpm layouts run the bundle twice.
  - confidence: 0.86
