## Summary

Current main still lacks configurable post-failure fallback delivery for hook mappings. Hook config, schema, mapping resolution, hook dispatch, cron delivery types, and the outbound send path all carry one channel/target; existing `fallbackChannel` names are pre-delivery selection helpers, not retry-after-failed-delivery semantics. The related Slack issue is closed and does not supersede this feature request.

## Best Possible Solution

Keep this open for maintainer API design and implementation of a typed hook delivery fallback chain. The likely direction is an ordered fallback target list on hook mappings, carried through schema/types, mapping transforms, hook dispatch, `CronDelivery`, delivery resolution, and final outbound send; docs should explicitly distinguish it from `channel: "last"`, `bestEffort`, and `failureDestination`, and clarify whether WebChat can ever be a fallback destination.

## Risks / Open Questions

- The requested `webchat` fallback example conflicts with current docs that make WebChat internal and not a configurable outbound channel; a maintainer needs to decide whether fallback targets are limited to deliverable channels or whether WebChat gets a new explicit delivery surface.
- Fallback semantics need to define whether each fallback has its own `to`, `accountId`, and `threadId`; a channel-only field may be insufficient when primary and fallback providers use different target identifiers.
- Retrying after an outbound error can duplicate messages if the primary provider accepted a payload but returned or surfaced an ambiguous failure; implementation should distinguish target-resolution failure, send failure, partial payload failure, and best-effort behavior.
- The feature overlaps but is not the same as `bestEffort`, cron `failureDestination`, and tool-context `fallbackChannel`; docs and tests should keep those semantics separate.
