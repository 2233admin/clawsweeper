## Summary

Keep this PR open. The change targets a real long-command blocking problem, but as submitted it creates an unmanaged background-exec path when `process` is denied and leaves the new process-denied behavior contradictory across docs.

## What This Changes

The PR decouples exec auto-backgrounding from `process` tool availability, adjusts hidden-process follow-up text/docs, and adds regression coverage for long exec calls with `process` denied.

## Best Possible Solution

Land a corrected version only if maintainers accept the new process-denied contract: keep the clearer hidden-process messaging, but preserve a finite timeout or reject timeoutless backgrounding whenever `process` is unavailable, then update both exec/background-process docs and add regression coverage for that guard.

## Reproduction Assessment

Yes. The focused path is `createOpenClawCodingTools` with `tools.deny: ["process"]`, then a long `exec` call with a small `yieldMs`; current main proves that path is synchronous, while the PR expects `status: "running"`. The unsafe variant adds `timeout: 0`, which current runtime code and tests prove disables the supervisor timeout.

## Solution Assessment

No, not as submitted. Decoupling backgrounding from process visibility may be acceptable, but the maintainable fix should not let process-denied background sessions opt out of the only remaining lifecycle timeout.
