## Summary

Keep open. Current main still gates exec backgrounding on process-tool availability, so the PR addresses a real behavior gap, but the submitted patch is not merge-ready because it changes process-denied code-execution semantics while leaving follow-up guidance and public docs describing the old contract.

## What This Changes

The PR removes the process-policy-derived exec backgrounding gate, always passes `allowBackground: true`, and adds a regression test for a long exec command when `process` is denied.

## Best Possible Solution

Land a corrected version only if maintainers accept the new process-denied semantics: keep the narrow exec wiring and regression test, pass process-tool availability separately into the running-result text, and update the public exec docs to describe lifecycle timeout or completion-wake behavior when `process` is hidden.

## Reproduction Assessment

Yes. The focused path is `createOpenClawCodingTools` with `tools.deny: ["process"]`, a long exec command, and a small `yieldMs`; current main's source proves the gate disables backgrounding, while the PR adds a test expecting `status: "running"`.

## Solution Assessment

No, not as submitted. Always enabling exec backgrounding is a narrow implementation, but a merge-ready fix must not instruct the agent to use an absent `process` tool and must document the changed process-denied contract.
