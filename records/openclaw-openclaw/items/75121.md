## Summary

Keep open. This is a useful active draft for Telegram forwarded-message context, but the latest head still miscomputes `body_line` for forwarded batch metadata when a forwarded item contains embedded newlines or blank-only text, so the patch is not merge-ready.

## What This Changes

The PR updates Telegram inbound forwarding to keep single-forward bodies clean, add per-item forwarded batch structured context with group visibility filtering, add forwarding prompt regression tests, and includes a small bundled runtime-dependency object construction rewrite.

## Best Possible Solution

Keep the typed structured-context approach, but compute `body_line` from the exact joined `BodyForAgent` output: skip blank-only entries, assign a line only to text that emits prompt body content, and advance by the number of emitted lines for each forwarded item. Add a multiline forwarded-burst regression before landing.

## Reproduction Assessment

Yes. On the latest PR head, a forwarded burst where the first item text is `line one\nline two` and the second item is `next` puts the second item on prompt line 3, but the patch records `body_line: 2` because it increments once per item.

## Solution Assessment

No, not yet. The structured metadata and visibility-gated design is the right narrow direction, but the line-accounting bug must be fixed before this is the best maintainable solution.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Count emitted body lines for forwarded batch metadata:** `extensions/telegram/src/bot-handlers.runtime.ts:317`
  - body: The PR joins forwarded item text with `\n`, but `nextBodyLine++` advances once per item. If one forwarded item contains embedded newlines, the following entry's `body_line` points at the wrong prompt line; blank-only text can also claim a line. Count the actual non-empty emitted body lines before assigning the next `body_line`.
  - confidence: 0.89
