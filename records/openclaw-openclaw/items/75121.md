## Summary

Keep open. The PR targets a reasonable Telegram forwarding-context concern, but the proposed patch is not ready as-is because current main already sends forwarded metadata through structured user-context blocks, and adding the same attribution to `BodyForAgent` duplicates AI-facing prompt context while moving `BodyForAgent` away from the clean-body contract.

## What This Changes

The PR changes Telegram inbound context assembly so direct forwarded messages prefix `BodyForAgent` with a `[Forwarded from ...]` line while leaving `RawBody` and `CommandBody` as the original text, and adds a regression test for that payload field.

## Best Possible Solution

Keep `BodyForAgent` as the clean user body and rely on `ForwardedFrom*` typed fields plus `buildInboundUserContextPrefix(...)` for model-visible forwarding context. If a specific hook or plugin misses forwarded metadata, fix that consumer or hook contract directly and cover the final prompt/hook context rather than concatenating metadata into the body string.

## Reproduction Assessment

Yes. A direct Telegram message with `forward_origin` reproduces the payload distinction on current main: `BodyForAgent` remains the raw body while `ForwardedFrom*` fields are populated; code inspection also shows the normal prompt path already includes those fields as structured forwarded context.

## Solution Assessment

No. The PR's approach is not the best fix because it duplicates metadata in the normal model prompt and conflicts with the documented clean-body plus structured-context migration path; a narrower solution would test or extend the actual consumer that lacks typed forwarded context.
