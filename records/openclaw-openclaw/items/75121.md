## Summary

Keep this draft PR open, but do not merge it as-is. The latest supplied diff still moves Telegram forwarded attribution into the model body even though current main already renders forwarded metadata through typed `ForwardedFrom*` fields and structured prompt context, so normal forwarded turns would be double-attributed.

## What This Changes

The draft branch changes Telegram inbound handling to prefix forwarded-message attribution into `BodyForAgent`, preserve per-message attribution in debounced forward bursts, strip inherited forward metadata from the synthetic burst message, and add forwarded-message regression tests.

## Best Possible Solution

Keep direct-message `BodyForAgent` as clean user text and continue exposing direct forwarded context through typed fields plus `buildInboundUserContextPrefix(...)`. For forwarded debounce bursts, add a structured per-entry context shape or a narrow prompt-context extension that preserves each origin without putting duplicated plaintext metadata into direct forwarded bodies.

## Reproduction Assessment

Yes. The code-path reproduction is a direct Telegram message with `forward_origin`: current main fills `ForwardedFrom*` and structured prompt context, while the PR also inserts the same attribution into `BodyForAgent`, which becomes `BodyStripped`.

## Solution Assessment

No. The proposed string prefix is not the narrowest maintainable fix because it duplicates existing structured metadata and pushes routing/provenance metadata back into the clean model body; typed forwarded context or structured burst context is safer.
