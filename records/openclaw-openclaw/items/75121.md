## Summary

Keep this draft PR open, but it should not merge as-is. Current main already carries Telegram forwarding data as typed `ForwardedFrom*` fields and the normal prompt path renders those fields as structured forwarded context; the branch would also prepend the same attribution into `BodyForAgent`, which then becomes `BodyStripped`, causing duplicate model-visible metadata and weakening the clean-body contract.

## What This Changes

The draft branch prefixes Telegram direct forwarded messages' `BodyForAgent` with a plaintext `[Forwarded from ...]` line while leaving `RawBody` and `CommandBody` clean and adding forwarded-message regression tests.

## Best Possible Solution

Keep `BodyForAgent` as the clean user text and rely on `ForwardedFrom*` typed fields plus `buildInboundUserContextPrefix(...)` for model-visible forwarding context. If a specific hook or plugin consumer lacks forwarded metadata, extend that consumer or hook context with typed forwarded fields and cover the final prompt or hook payload directly.

## Reproduction Assessment

Yes. A direct Telegram message with `forward_origin` has a high-confidence code-path reproduction: current main builds clean `BodyForAgent`, fills `ForwardedFrom*`, and the reply prompt prepends a structured forwarded-context block; the PR would add the same attribution into `BodyForAgent` and therefore into `BodyStripped`.

## Solution Assessment

No. The proposed body-string prefix is not the narrowest maintainable fix because it duplicates existing structured metadata in the normal model prompt and conflicts with the documented migration away from plaintext inbound envelopes.
