## Summary

Keep this draft PR open. The earlier `BodyForAgent` duplication concern has been addressed, but the updated structured burst metadata has a concrete edge-case bug: `body_line` is derived from the original debounced entry index while the actual joined body drops empty text entries, so mixed media-only and text forwarded bursts can point attribution at the wrong prompt line.

## What This Changes

The PR keeps single forwarded Telegram message text clean in `BodyForAgent`, adds per-item `UntrustedStructuredContext` for coalesced forwarded bursts, strips inherited forward metadata from synthetic burst messages, and adds forwarded-message regression tests.

## Best Possible Solution

Keep the typed structured-context approach, but compute any text-line reference from the actual non-empty text entries that survive into the joined `BodyForAgent`, or omit `body_line` for media-only forwarded entries and rely on `item_index`/`message_id` for attribution.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a debounced Telegram forward burst where the first forwarded update has media but no text/caption and the second has text; the joined body has one line, while the second entry's structured context reports `body_line: 2`.

## Solution Assessment

No, not quite yet. The overall direction is the narrow maintainable fix, but the new `body_line` calculation must align with the actual joined prompt body before this is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.83

Full review comments:

- **[P2] Align forwarded batch body_line with joined text:** `extensions/telegram/src/bot-handlers.runtime.ts:233`
  - body: Forwarded burst flushing builds `BodyForAgent` only from non-empty text entries, but this payload uses the original debounced entry index as `body_line`. If a media-only forwarded message arrives before a text forward, the text appears on line 1 while its metadata says line 2, so the model gets misleading attribution. Track the line number only for entries that contribute text, or omit it for media-only entries.
  - confidence: 0.82
