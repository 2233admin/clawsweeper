## Summary

Keep open. The PR addresses a real Telegram forwarded-burst attribution gap, but the current head introduces a blocking regression: the shared synthetic-message helper now strips forward metadata globally, while long text-fragment reassembly still depends on that helper preserving `forward_origin` for `ForwardedFrom*` prompt context.

## What This Changes

The PR updates Telegram forwarded-message handling so single forwards keep typed metadata while forwarded bursts use per-item untrusted structured context, with regression tests for direct forwards, self-forwards, burst coalescing, media-only entries, visibility filtering, and prompt rendering.

## Best Possible Solution

Keep the typed structured-context approach for forwarded bursts, but scope forward-metadata stripping to the multi-message forward-burst synthetic message only. Text-fragment reassembly and other non-batch synthetic messages should preserve Telegram forward fields so existing single-forward prompt metadata remains intact.

## Reproduction Assessment

Yes. Send a forwarded Telegram text message at or above the 4000-character fragment threshold so it enters `flushTextFragments`; in PR head, the helper strips `forward_origin` before `buildTelegramInboundContextPayload` reads it, so `ForwardedFrom*` is not produced. This was verified by source and diff inspection in this read-only review.

## Solution Assessment

No. The overall design is close, but globally stripping forward metadata from the shared helper is broader than the batch fix requires; the safer solution is an explicit strip option used only by forwarded-burst coalescing, plus a regression for long forwarded text fragments.
