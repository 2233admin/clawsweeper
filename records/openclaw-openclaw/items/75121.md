## Summary

Keep open. The latest draft fixes the earlier `body_line` mismatch, but it introduces a security-sensitive context-visibility bypass for Telegram forwarded bursts: per-item forwarded metadata is built in the handler and passed to the agent without the group visibility filter that current main applies to single forwarded/reply metadata.

## What This Changes

The PR changes Telegram inbound handling so forwarded message text stays clean in `BodyForAgent`, debounced forwarded bursts get per-message `UntrustedStructuredContext`, inherited forward metadata is stripped from synthetic burst messages, and forwarded-context regression tests are added.

## Best Possible Solution

Keep the clean `BodyForAgent` plus typed structured-context design, but apply the existing Telegram group `contextVisibility` and sender-allowlist policy to each forwarded batch item before adding it to `UntrustedStructuredContext`; add a regression for allowlist redaction on debounced forwarded bursts before merge.

## Reproduction Assessment

Yes. A high-confidence static reproduction is a Telegram group with `contextVisibility: "allowlist"` and `allowFrom: ["1"]`, where allowed sender `1` forwards a debounced burst whose original forwarded sender is `999`; the PR adds batch structured metadata for `999` even though current single/reply forwarded context is redacted.

## Solution Assessment

No, not yet. The structured metadata direction is narrow and maintainable, but the implementation must reuse the existing forwarded-context visibility decision for batched forwards before it is safe to merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Filter forwarded batch metadata by visibility:** `extensions/telegram/src/bot-handlers.runtime.ts:332`
  - body: The batch path now builds `UntrustedStructuredContext` for every forwarded item and passes it to the agent without the `shouldIncludeGroupSupplementalContext` check used for single forwarded/reply metadata. In a Telegram group configured with `contextVisibility: "allowlist"`, a forward from an unallowed user is currently redacted, but a debounced forwarded burst would still expose that sender's id/username/date in the JSON block. Apply the same forwarded-origin visibility check before adding each batch entry.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff can leak Telegram forwarded-origin metadata across the existing group context-visibility boundary.

Concerns:

- **[medium] Forwarded batch metadata bypasses context visibility:** `extensions/telegram/src/bot-handlers.runtime.ts:332`
  - body: Forwarded batch metadata is created in the handler and copied into `MsgContext` as untrusted structured context, while group `contextVisibility` filtering only runs for the single-message `ForwardedFrom*` and reply-target paths. This can expose blocked forwarded sender IDs/usernames/dates to the agent prompt in allowlist-mode groups.
  - confidence: 0.88
