## Summary

Keep this PR open. Current main still reloads `sessions.list` immediately after `sessions.changed`, so the optimization is not implemented or superseded, but the submitted patch is not merge-ready because it defers the reload without updating the existing synchronous unit test or clearing the pending timer during disconnect cleanup.

## What This Changes

The PR adds Control UI `sessionsRefreshTimer` state and changes `sessions.changed` handling to debounce the full `loadSessions` reconciliation for 5 seconds while keeping the incremental `applySessionsChangedEvent` update immediate.

## Best Possible Solution

Request changes on this PR: keep the immediate incremental session-row update, debounce and coalesce only the full `sessions.list` reconciliation, add fake-timer coverage for delayed and coalesced reloads, initialize timer state in fixtures, and clear any pending timer during Control UI disconnect cleanup.

## Reproduction Assessment

Yes. Current main can be verified by reading `handleGatewayEventUnsafe`: a `sessions.changed` event calls `loadSessions` immediately, and the existing unit test encodes that synchronous behavior; the PR would need fake timers to prove the new delayed/coalesced path.

## Solution Assessment

No, not as submitted. The debounce direction is narrow and maintainable, but the patch needs timer lifecycle cleanup and updated tests before it is the best implementation.
