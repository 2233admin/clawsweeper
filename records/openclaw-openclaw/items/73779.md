## Summary

Keep this PR open. Current main still performs the immediate `sessions.list` reload that the PR is trying to reduce, so the branch is not obsolete; however, the submitted patch needs focused changes before merge because it adds a delayed timer without updating the existing synchronous test, without clearing the timer during Control UI teardown, and without the required user-facing changelog handling.

## What This Changes

The PR adds Control UI `sessionsRefreshTimer` state and debounces the full `loadSessions` reconciliation for `sessions.changed` events while keeping the incremental `applySessionsChangedEvent` update immediate.

## Best Possible Solution

Request changes on this PR: keep the immediate incremental session-list update, debounce only the full `sessions.list` reconciliation, clear any pending reload timer during Control UI disconnect cleanup, cover delayed and coalesced reloads with fake-timer tests, and add the active-version changelog entry if maintainers keep classifying this as a user-visible Control UI performance fix.

## Reproduction Assessment

Yes. Current main can be verified by source inspection: `handleGatewayEventUnsafe` calls `loadSessions` immediately for `sessions.changed`, and the existing unit test encodes that synchronous behavior. The PR regression is likewise reproducible by that test unless timers are advanced.

## Solution Assessment

No, not as submitted. The debounce direction is a narrow maintainable fix for the reported load pattern, but the patch needs timer lifecycle cleanup, updated fake-timer coverage, and changelog handling before it is the best solution.
