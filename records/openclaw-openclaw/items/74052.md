## Summary

Keep this PR open for maintainer review. Current main still has the reported account-resolution ordering bug, and the PR is a focused active fix with no discrete blocking code-review finding found in the diff; public CI still needs follow-up before landing.

## What This Changes

The PR updates outbound message account resolution so explicit `accountId` wins, a calling agent's configured channel/account binding is preferred when `accountId` is omitted, inherited/default account context is last, and regression coverage exercises the combined default-plus-binding case.

## Best Possible Solution

Land this PR or an equivalent narrow change after CI is green or the failing check is clearly unrelated. The desired end state is one shared outbound precedence rule: explicit `accountId`, then the calling agent's channel/account binding, then inherited/default account context.

## Reproduction Assessment

Yes. A focused `runMessageAction` case can set `defaultAccountId`, configure a different binding for `agentId`, omit `params.accountId`, and observe current main choose the default before the binding.

## Solution Assessment

Yes, subject to CI follow-up. The PR changes the shared wrapper/runner precedence instead of adding Slack-specific behavior, and its regression tests cover the missing combined cases without a broader API change.
