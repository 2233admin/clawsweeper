## Summary

Keep this PR open. Current main still creates Feishu synthetic reaction message IDs with a `:reaction:<emoji>:<uuid>` suffix and still passes caller-provided message IDs directly into Feishu SDK API path/data fields. The linked bug #34528 remains open, the prior ClawSweeper review found the same current-main gap, and a maintainer comment says the branch was refreshed as the canonical contributor path with validation run.

## Best Possible Solution

Keep this PR open and review or land the refreshed branch as the canonical fix for #34528. The right implementation direction is a Feishu-plugin-local message ID normalizer applied at every Feishu API boundary, while preserving suffixed synthetic IDs for internal dedupe/session identity, with focused tests for valid suffix stripping and malformed/partial suffix passthrough plus the existing changelog credit.
