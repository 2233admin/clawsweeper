## Summary

Keep open. Current main still lacks the requested `doctor --check-config` surface, and this PR is useful work, but the branch is not merge-ready against current main: it is stale against the registry-backed TTS implementation and still has false-pass cases in config and fallback validation.

## What This Changes

The PR adds an `openclaw doctor --check-config` option, a new deep config validation module for schema/plugin/model/fallback/TTS/channel checks, and formatter-focused tests for the new output.

## Best Possible Solution

Land or replace this with a rebased doctor deep-config mode that uses current registry-backed TTS/model APIs, treats active unresolved env or SecretRef-backed values as failures when they would break an enabled subsystem, validates fallback refs against runtime catalog/provider availability, keeps validation reads side-effect-free, documents the CLI surface, and adds behavior tests for subsystem checks and non-zero exits.

## Reproduction Assessment

Yes. The highest-confidence reproduction is static: applying PR head to current main introduces an import of `TTS_PROVIDERS` from `src/tts/tts.ts`, and current main does not export that symbol. The runtime false-pass cases are also traceable from PR head logic and current config/TTS/model helpers without mutating the checkout.

## Solution Assessment

No. The requested direction is appropriate for the linked config-degradation reports, but this implementation should be updated to current registry APIs and should validate or fail the same runtime conditions it claims to diagnose before merge.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Use the current TTS provider registry:** `src/commands/doctor-check-config.ts:18`
  - body: `TTS_PROVIDERS` is not exported by current `../tts/tts.js`, so this new module will fail typecheck/build after merge. Use registry-backed APIs such as `resolveTtsProviderOrder(configuredProvider, cfg)` or `listSpeechProviders(cfg)` instead of importing the old static provider list.
  - confidence: 0.98
- **[P2] Fail active unresolved env-backed config:** `src/commands/doctor-check-config.ts:72`
  - body: Snapshot warnings are emitted as `warn`, and `hasFailures` only considers `fail`. With an active value like an unset `${OPENAI_API_KEY}`, the preserved placeholder is still a non-empty string, so this command can report success and exit 0 even though the enabled subsystem will run with missing credentials.
  - confidence: 0.9
- **[P2] Validate fallback refs against runtime availability:** `src/commands/doctor-check-config.ts:279-280`
  - body: `resolveModelRefFromString` only proves the fallback string can be parsed. A fallback like `missing-provider/model` can still be counted as validated even though it is not usable at runtime, leaving the requested fallback validation gap uncaught.
  - confidence: 0.92
- **[P2] Reject unparseable qualified primary refs:** `src/commands/doctor-check-config.ts:153-154`
  - body: When `agents.defaults.model` contains a slash but cannot be parsed, such as `anthropic/`, `parseModelRef` returns null and this branch falls through to reporting the resolved fallback default as a healthy primary model. That misses a broken explicitly configured model ref.
  - confidence: 0.84
- **[P2] Keep catalog validation read-only:** `src/commands/doctor-check-config.ts:183`
  - body: Calling `loadModelCatalog({ config: cfg })` uses the default non-read-only path, which can call `ensureOpenClawModelsJson` and create or rewrite `models.json`. A deep config check intended for validation and CI gating should pass `readOnly: true` so it does not mutate user state while checking config.
  - confidence: 0.86
