## Summary

Keep open. Current main still mirrors successful isolated cron announce output into the main-session system-event queue, while the docs describe isolated/background cron work as outside the main conversation; the linked attempted fix PR was closed unmerged, so there is no implemented fix to close against.

## What This Changes

Cron jobs configured with `sessionTarget: "isolated"` should keep each run out of the main conversation instead of accumulating cron output in the main session.

## Best Possible Solution

Implement a narrow cron delivery change that keeps isolated run transcripts isolated for implicit/default announce delivery, while preserving any explicitly configured main-session awareness behavior that maintainers still want. Add regression coverage for the implicit `channel: "last"` path and update changelog/docs if the visible behavior changes.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists on current main: a successful isolated announce delivery reaches `dispatchCronDelivery`, passes `shouldQueueCronAwareness`, and enqueues the delivered text into the resolved main session; current tests also assert that mirroring behavior.

## Solution Assessment

No, current main is not the best solution for this report because it still writes isolated cron output into the main session. The safer fix is to gate awareness on an explicit delivery target or explicit opt-in, matching the direction described in the linked unmerged PR while leaving unrelated cron delivery paths alone.
