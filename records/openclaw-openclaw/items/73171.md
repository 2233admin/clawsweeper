## Summary

Keep this PR open. Current main still does not bound status gateway auth resolution before probing; the newer local status RPC fallback helps a related false-negative path after probe failure, but it does not cover a hung auth lookup before the probe starts. The proposed patch is focused and I found no blocking correctness or supply-chain finding, though it needs normal maintainer review against the newer status fallback code.

## What This Changes

The PR adds a timeout-based empty-auth fallback around status gateway auth resolution and a focused shared status test for a never-settling auth resolver.

## Best Possible Solution

Land a current-main version only if maintainers still want the distinct auth-stall robustness: keep the boundary in the status scan auth-resolution path, add/keep a focused never-settling resolver test, preserve explicit warning behavior, and decide whether rejection should share the same fallback semantics.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is clear from the code: mock resolveGatewayProbeAuthResolution with a never-settling promise and call resolveGatewayProbeSnapshot with a small timeout; current main waits before probe/fallback, while the PR adds coverage for that path.

## Solution Assessment

Yes for the timeout surface, with a rebase caveat. A bounded Promise.race around status auth resolution is the narrowest place to prevent auth lookup from blocking the probe, but it should be reconciled with the newer local RPC fallback and the rejection behavior should be an explicit maintainer choice.
