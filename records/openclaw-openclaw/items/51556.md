## Summary

Keep this PR open for maintainer review. Current main still has the sync stat-error mismatch the PR targets, but the branch is stale against the current shared allowlist helper and the proposed read-file rethrow would leave sync and async read behavior inconsistent.

## What This Changes

The PR changes two catch branches in the old sync pairing allowlist reader in `src/pairing/pairing-store.ts` to rethrow non-ENOENT stat and readFile errors instead of treating the allowlist as missing.

## Best Possible Solution

Refresh the useful stat-error part into `src/pairing/allow-from-store-file.ts`, add focused regression tests for non-ENOENT stat failures, and handle read-file failures consistently across sync and async readers only after deciding the shared fallback policy.

## Reproduction Assessment

Yes. A focused test can stub `fs.statSync` to throw `EACCES` and observe current main returning an empty allowlist while the async stat path rethrows; a second stub where stat succeeds and `readFileSync` throws `EACCES` shows the PR's sync-only read behavior diverging from the async fallback contract.

## Solution Assessment

No. Rethrowing unexpected sync stat errors is the right direction, but this PR also rethrows sync read failures without changing the async reader and targets an old inline helper location; the safer fix is to update the current shared helper and test the stat contract separately from any read-fallback policy change.
