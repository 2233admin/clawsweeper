## Summary

Keep open because current main still lacks the Control UI composer safeguards and regression coverage for numeric-only chat input, and the linked open PR is the correct implementation path before this issue can close.

## What This Changes

The issue reports that Control UI webchat changes pure numeric input such as `123456789` into hyphen-grouped text such as `123-456-789`, breaking numeric IDs, arguments, and URLs.

## Best Possible Solution

Land PR #74369, or an equivalent narrow Control UI change, so the chat composer discourages browser/OS numeric rewriting and tests exact `chat.send` payload preservation without adding server-side deformatting that could corrupt intentionally hyphenated input.

## Reproduction Assessment

Unclear. The issue provides concrete Control UI/Windows steps and a screenshot, and current source shows a plausible path where textarea-rewritten text is sent unchanged, but I did not live-reproduce the Windows browser or IME auto-formatting behavior in this read-only pass.

## Solution Assessment

Yes, the linked direction is the narrowest maintainable fix: add plain-text composer attributes and regression tests at the Control UI boundary. Server-side deformatting would be riskier because hyphenated IDs, phone-like strings, or user prose may be intentional.
