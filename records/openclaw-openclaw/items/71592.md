## Summary

Keep open. Current main still matches the report: shared TUI autocomplete exposes `/status` and `/compact`, but the local TUI command handler has no cases for either command, so the default path sends the raw slash input to the embedded backend and starts an agent run. The related items are adjacent command/autocomplete work, not a canonical fix for this local embedded execution path.

## What This Changes

The issue asks for TUI local embedded `/status` and `/compact` slash commands to be handled or explicitly rejected instead of being submitted to the model as ordinary user text.

## Best Possible Solution

Add a local embedded command boundary for advertised shared commands. `/status` and `/compact` should either execute through a shared command adapter that works without the Gateway or render clear local TUI system responses, and neither command should reach the normal model prompt path in local mode.

## Repair Work Prompt

Fix this issue in OpenClaw. In TUI local embedded mode, advertised known slash commands `/status` and `/compact` currently fall through to `sendMessage(raw)` and start an embedded agent run. Add a narrow local-mode command boundary in `src/tui/tui-command-handlers.ts` and related tests so local `/status` and `/compact` either execute via an appropriate shared/local adapter or render an explicit local TUI system response, but never call `sendChat()` / `agentCommandFromIngress()` as ordinary model text. Preserve gateway-connected behavior for `/status`, `/context`, `/btw`, unknown shared/plugin commands, and existing autocomplete unless a command is explicitly unsupported in local mode. Add focused tests for local `/status` and `/compact`, keep the existing gateway `/status` forwarding test, and add a user-facing changelog Fixes entry with `Thanks @goslingmanagment`. Do not change unrelated command registry, plugin command, or Gateway dispatch behavior.
