Keep this PR open. Current main still clears the in-memory Gateway Token when the Overview WebSocket URL changes, and current browser coverage still asserts that clearing behavior. The PR remains a plausible fix for #41545 with useful regression coverage, but it overlaps with #41546 and #41718 and still needs maintainer judgment on the selected-gateway credential boundary.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best path is to resolve #41545 with one canonical UI change that prevents surprising token loss during ordinary Overview URL edits while preserving the documented selected-gateway credential boundary, then land the chosen implementation with regression coverage and close the other overlapping PRs as superseded.

## Risks / Open Questions

- Preserving a bearer token across arbitrary WebSocket URL edits can send a token selected for one gateway to a different endpoint when the user clicks Connect.
- Because settings are persisted on input, a naive preserve-token fix can write the existing token under new or partial gateway URL sessionStorage keys unless the implementation accounts for that.
- There are multiple open candidate PRs for #41545, so maintainers should pick one canonical implementation and reuse the strongest regression coverage.
