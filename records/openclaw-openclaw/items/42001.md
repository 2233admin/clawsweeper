## Summary

Keep this PR open. Current main still has the behavior the PR targets: editing the Overview WebSocket URL clears the in-memory Gateway Token, and current browser coverage still asserts that clearing behavior. The PR is a focused implementation candidate for the open linked bug, but the exact preservation behavior needs maintainer review because it can carry a selected-gateway credential to a different endpoint.

## Best Possible Solution

Keep the PR open for maintainer review or rework. The best fix should prevent ordinary WebSocket URL edits from unexpectedly erasing a manually entered Gateway Token while preserving the selected-gateway credential boundary, with regression coverage in the Overview/navigation UI tests and explicit handling for when a URL change should require confirmation or a new token.

## Risks / Open Questions

- Merging the exact patch as described could preserve a gateway token across an arbitrary WebSocket endpoint edit and send that credential to the newly entered endpoint on Connect.
- Because settings are saved on input and token storage is keyed by gateway URL, preserving the token while typing may write the existing token under a new or partial gateway URL sessionStorage key.
- There are multiple open implementation candidates for the same linked bug; maintainers should choose one canonical security-aware fix rather than closing this focused PR as cleanup.
