## Summary

Keep this PR open. Current main still clears the in-memory Gateway Token when the Overview WebSocket URL changes, so the PR's intended bug fix has not shipped. The patch is focused, but it changes credential handling and should get maintainer review alongside the linked bug #41545 and competing open PR #41546.

## Best Possible Solution

Keep the PR open for maintainer review or rework. The best fix should prevent ordinary WebSocket URL edits from unexpectedly erasing a manually entered Gateway Token while preserving the selected-gateway credential boundary, with regression coverage in the Overview/navigation UI tests and explicit handling for when a URL change should require confirmation, token re-entry, or a scoped token transfer.

## Risks / Open Questions

- Merging this exact patch could preserve a selected-gateway token across an arbitrary WebSocket endpoint edit, and the Connect path would send that token to the newly entered endpoint.
- Because settings are saved on input and token storage is keyed by gateway URL, preserving the token while typing can write the existing token under a new or partial gateway URL sessionStorage key.
- There are multiple open implementation candidates for #41545, so maintainers should avoid closing one as obsolete until they choose or land a canonical security-aware fix.
