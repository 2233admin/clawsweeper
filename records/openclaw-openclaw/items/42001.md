## Summary

Keep this PR open. Current main still clears the in-memory Gateway Token when the Overview WebSocket URL changes, so the intended bug fix has not shipped. The PR is focused and relevant to the open bug #41545, but it changes credential handling and should get maintainer review alongside the competing open implementation #41546.

## Best Possible Solution

Keep this PR open for maintainer review or rework. The best fix should prevent ordinary WebSocket URL edits from unexpectedly erasing a manually entered Gateway Token while preserving the selected-gateway credential boundary, with regression coverage for Overview/navigation behavior and explicit handling for when a URL change should require confirmation, token re-entry, or a scoped token transfer.

## Risks / Open Questions

- Merging the exact semantics could preserve a selected-gateway token across an arbitrary WebSocket endpoint edit, and the Connect path would send that token to the newly entered endpoint.
- Because settings are saved on input and token session keys are URL-scoped, preserving the token while typing could write the current token under new or partial gateway URL sessionStorage keys.
- There are multiple open implementation candidates for #41545, so maintainers should choose a canonical security-aware fix rather than treating this PR as already implemented.
