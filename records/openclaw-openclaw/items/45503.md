## Summary

Current main has automatic cache-TTL context pruning, live/persisted tool-result size caps, and manual session compaction, but I did not find a manual user- or agent-addressable way to purge a specific tool result or mark a result clear-after-use. The issue should stay open for product/API design rather than stale cleanup.

## Best Possible Solution

Keep this open and make a maintainer product/API decision first. The most conservative direction is a prompt-view-only manual clear/archive mechanism for completed tool results, preserving the raw transcript and provider pairing while allowing a recoverable placeholder in active context; then document it and cover /context, pruning, compaction, and provider replay tests.

## Risks / Open Questions

- A manual purge that rewrites transcript history could break provider replay invariants around assistant tool calls and matching tool results unless it is modeled as a safe prompt-view transformation.
- An agent-accessible clear function or clearAfterUse flag needs recoverability semantics so useful IDs/config values are not lost after the model answers once.
- The user-facing surface is still undecided: slash command, CLI command, context-engine capability, tool-result middleware flag, or a combination.
