## Summary

Keep open. Current main still has automatic cache-TTL pruning, live tool-result truncation, overflow recovery, and whole-session compaction, but no user- or agent-addressable clear/archive/recover surface for a specific completed tool result. The discussion raises recovery and checkpoint semantics, so this needs core product/API design rather than stale cleanup or an autonomous narrow fix.

## What This Changes

Request a manual way to purge, archive, or otherwise remove completed tool results from future context once they are no longer useful.

## Best Possible Solution

Define a core-supported prompt-view clear/archive mechanism for completed tool results that preserves raw transcript history and provider pairing, exposes an intentional user and/or agent surface, supports recovery or durable placeholders, and documents how it interacts with /context reporting, pruning, compaction, context engines, and provider replay.

## Reproduction Assessment

Not applicable as a runtime bug report; this is a feature gap. The gap is reproducible by source inspection on current main: pruning is TTL/policy-driven, /context is inspect-only, /compact is whole-session scoped, and no clear/archive/recover selector exists for individual tool results.

## Solution Assessment

Unclear. The requested command, agent function, and clearAfterUse flag are plausible entry points, but the safer maintainable solution is a core prompt-view archive/clear design that preserves transcript and replay invariants instead of directly deleting history.
