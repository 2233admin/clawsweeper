## Summary

Keep this PR open. Current main still does not implement the requested model-visible hint for skill names called as unknown tools, and the PR is a focused candidate rather than obsolete. It needs rebase and maintainer review because the current pinned pi-agent packages expose `agent.afterToolCall` as a property, not `setAfterToolCall`, and the published agent loop still bypasses `afterToolCall` for immediate missing-tool results.

## Best Possible Solution

Keep this PR open as a useful focused candidate. The best path is to rebase it onto current main, implement the hint at the supported missing-tool result path for the pinned pi-agent packages, avoid private/nonexistent hook APIs, deliberately choose visible-skill matching semantics, keep real tool collisions generic, and add targeted regression tests for the success and non-rewrite cases.

## Risks / Open Questions

- The proposed hook wiring appears incompatible with the currently pinned pi-agent API because it uses `setAfterToolCall` and a private `_afterToolCall` field that were not present in the inspected 0.70.5 package.
- The common missing-tool path may still bypass the proposed rewrite because the inspected 0.70.5 agent loop handles unknown tools as immediate results before `afterToolCall` finalization.
- The PR adds no regression tests for skill match, non-skill unknown tool, tool/skill collisions, `$` or punctuation skill names, or the pinned missing-tool path.
- The unknown-tool parser remains narrow (`[a-z0-9_-]+`) relative to documented/raw skill names and existing command-name sanitization behavior.
