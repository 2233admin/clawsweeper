## Summary

Keep open. Current main still lacks an explicit one-way dispatch/handoff mode for `sessions_send`: the schema has no delivery-intent field, fire-and-forget still schedules the A2A follow-up path, and the A2A helper still performs reply-back turns plus announce delivery. Related open items cover adjacent announce/thread/duplicate-call problems but do not supersede this per-call dispatch semantic.

## Best Possible Solution

Add a backward-compatible explicit delivery intent for `sessions_send`, preferably a conservative `mode: "conversation" | "dispatch"` with current behavior as the default. In dispatch mode, enqueue/persist the message into the target session using the existing inter-session provenance/internal-channel path, return a minimal delivery acknowledgement, and skip `runSessionsSendA2AFlow` entirely so there is no requester-side reply reinjection and no announce unless a separate explicit option is later designed. Update focused tests and the public session tool docs.
