## Summary

Keep open: this is a valid implementation PR for the `/v1/chat/completions` client-tool parity gap, but the proposed stream finalization can still close before pending tool-call metadata is available, and client-tool name conflicts still surface as generic failures.

## What This Changes

The PR adds function `tools`/`tool_choice` handling to `/v1/chat/completions`, forwards resolved client tools into the agent command, maps pending tool calls into Chat Completions non-stream and SSE responses, accepts tool-result messages, and adds gateway tests.

## Best Possible Solution

Revise this branch so Chat Completions keeps the new client-tool protocol support while waiting for the command result before final SSE completion when tool metadata may exist, translating client-tool validation and conflict failures to OpenAI-compatible 400 responses, and preserving the focused gateway regression coverage.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wait for the command result before closing streams:** `src/gateway/openai-http.ts:971-974`
  - body: When `lifecycle:end` arrives before `agentCommandFromIngress(...)` resolves, this timer can close the SSE stream after one second while `resultResolved` is still false. If the command later resolves with `pendingToolCalls`, the code returns because `closed` is already true, so clients never receive `delta.tool_calls` or `finish_reason: "tool_calls"` for the follow-up loop.
  - confidence: 0.91
- **[P2] Return 400 for conflicting client tool names:** `src/gateway/openai-http.ts:839`
  - body: Forwarding `resolvedClientTools` lets duplicate or builtin-conflicting tool names reach the runner, where `createClientToolNameConflictError()` is thrown. This adapter does not catch that error like `/v1/responses` does, so malformed client tool input becomes a generic failure instead of `invalid_request_error`.
  - confidence: 0.78
