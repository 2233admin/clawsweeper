## Summary

Keep this PR open. Current main still lacks the Chat Completions client-tool protocol this branch targets, and the linked #54405 remains a valid open gap, but the proposed stream finalization path can still close before pending tool-call metadata is available. The earlier Greptile error-message polish appears addressed in the second commit, so the remaining blockers are functional review issues rather than cleanup-close evidence.

## What This Changes

The PR adds `/v1/chat/completions` parsing for function `tools` and `tool_choice`, forwards client tools to the agent command, maps pending client-tool calls to non-stream and streaming Chat Completions `tool_calls`, and adds regression coverage in `src/gateway/openai-http.test.ts`.

## Best Possible Solution

Revise this branch so streaming Chat Completions waits for the command result or another proven terminal condition before sending the final SSE chunk, maps client-tool validation/conflict failures to OpenAI-compatible 400 errors, keeps the focused parsing/forwarding/response mapping, and lands with the existing gateway regression tests plus targeted changed-gate proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Wait for the command result before closing streams:** `src/gateway/openai-http.ts:971-974`
  - body: When `lifecycle:end` arrives before `agentCommandFromIngress(...)` resolves, this timer allows `maybeFinalize()` after one second while `resultResolved` is still false. The response can then close with `finish_reason: "stop"`; if the command later resolves with `pendingToolCalls`, the code returns because `closed` is already true, so clients never receive `delta.tool_calls` or the `tool_calls` finish reason needed for the follow-up loop.
  - confidence: 0.91
- **[P2] Return 400 for conflicting client tool names:** `src/gateway/openai-http.ts:839`
  - body: Forwarding `resolvedClientTools` lets duplicate or builtin-conflicting tool names reach the runner, where `createClientToolNameConflictError()` is thrown. Unlike `/v1/responses`, this adapter does not catch that error, so a malformed client tool schema becomes a generic 500/stream error instead of `invalid_request_error`; add the same conflict handling or preflight validation before dispatch.
  - confidence: 0.78
