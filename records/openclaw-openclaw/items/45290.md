## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Avoid stranding deferred silent deliveries:** `src/infra/outbound/deliver.ts:493-501`
  - body: This returns immediately after enqueueing the silent delivery, before any send, ack, fail, or live retry path. Since recovery drains pending snapshots on startup/reconnect rather than continuously watching this queue, normal runtime overlap with a user-visible delivery can leave the follow-up pending until an unrelated later drain.
  - confidence: 0.9
- **[P2] Reject explicit channel failures before fallback:** `src/infra/outbound/message.ts:113-120`
  - body: resolveRequiredChannel now passes the mirrored session fallback even when params.channel is explicit. resolveMessageChannelSelection falls back when the explicit channel is unavailable, so a typo or malformed explicit channel can deliver through the mirrored session channel instead of failing fast.
  - confidence: 0.86
- **[P2] Use the configured session store for Argus checks:** `src/agents/session-write-lock.ts:313-317`
  - body: readArgusSessionContext hard-codes sessions.json beside the session file, but session.store supports configured paths/templates. In those setups Argus metadata is missed, so stale-lock cleanup and the watchdog can still reclaim locks the new protection is meant to preserve.
  - confidence: 0.84

## Security Review

Status: needs_attention

Summary: Supply-chain-sensitive surfaces are unchanged, but the diff introduces a concrete outbound data-routing risk through explicit-channel fallback.

Concerns:

- **[medium] Explicit channel fallback can misroute outbound content:** `src/infra/outbound/message.ts:113`
  - body: The proposed sendMessage fallback applies even when the caller supplied an explicit channel. If that channel is unavailable or malformed, message content can be routed through the mirrored session channel instead of failing, which can expose content on an unintended channel or recipient path.
