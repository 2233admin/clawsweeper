## Summary

Current main still appears to have the reported sandboxed final-reply media gap. When a sandbox root exists, `createReplyMediaPathNormalizer()` sends every non-remote media source through `resolveSandboxedMediaSource()` before trying the agent-scoped outbound media access path, so an absolute host workspace `MEDIA:` path can be rejected as outside the sandbox and dropped. The existing tests explicitly preserve that rejection for sandboxed absolute host-local paths, while only non-sandbox absolute workspace media has positive coverage.

## Best Possible Solution

Keep this issue open and fix the sandboxed normal-reply media path narrowly. When a sandbox root exists, absolute host-local `MEDIA:` paths should be allowed only if they are already managed media or pass the existing agent-scoped outbound media access policy for the active workspace/session, then staged through the existing outbound attachment path. Add positive coverage for a sandboxed Telegram-style final reply using an absolute host workspace media path and negative coverage for unrelated host paths; get secops review for the sandbox/file-read boundary.
