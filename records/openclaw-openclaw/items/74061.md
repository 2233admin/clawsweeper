## Summary

Keep open. Current main still preserves the reported failure mode for sandboxed final replies: once a sandbox root exists, host-absolute local `MEDIA:` paths are resolved through the sandbox mapper first, mapping failures are dropped, and the payload can become the visible `⚠️ Media failed.` fallback before Telegram receives a usable upload path. The astorozhevsky comment is a useful adjacent data point: cron-owned announce delivery appears to have a related `/workspace` to host translation gap, but that scope should be decided deliberately rather than folded in implicitly.

## What This Changes

The issue reports that Telegram final replies using `<final>MEDIA:` with an existing host-local workspace path on Linux are converted to `⚠️ Media failed.` instead of uploading the file.

## Best Possible Solution

Add a narrow, policy-checked bridge for sandboxed final-reply media: after sandbox mapping fails, allow only managed media or host-local files that pass the active agent/session outbound media policy, then stage them through the existing outbound attachment path. Cover the reported Telegram final-reply case, preserve negative tests for unrelated host paths, and explicitly decide whether cron-owned announce delivery should use the same sandbox bridge.

## Reproduction Assessment

Yes. The current source and checked-in tests provide a high-confidence source-level reproduction: with a sandbox root present, an absolute host-local `MEDIA:` path is rejected by sandbox mapping, dropped, and converted to the `Media failed` fallback. I did not run tests because this review was required to keep the checkout read-only.

## Solution Assessment

No. Current main is not the best solution because it keeps a blanket sandbox-first rejection for host-absolute paths that may already be valid under the agent-scoped outbound media policy. The safer solution is a policy-checked fallback, not an unrestricted allowlist.
