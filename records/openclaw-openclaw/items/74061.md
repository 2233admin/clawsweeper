## Summary

Keep open. Current main still preserves the reported sandboxed final-reply failure mode: host-absolute local MEDIA paths are resolved through sandbox mapping first and dropped before Telegram receives a usable upload path; the related cron-owned /workspace case also lacks sandbox bridge wiring.

## What This Changes

The issue asks OpenClaw to deliver local MEDIA paths from Telegram final replies, including sandbox/workspace paths, instead of replacing them with Media failed.

## Best Possible Solution

Add a narrow, policy-checked bridge for sandboxed final-reply media: after sandbox mapping fails, allow only managed media or files that pass the active agent/session outbound media policy, then stage them through the existing outbound attachment path. Cover the reported Telegram-style final reply, keep negative coverage for unrelated host paths, and explicitly decide whether cron-owned announce delivery uses the same bridge.

## Reproduction Assessment

Yes. The current source and checked-in tests provide a high-confidence reproduction path: with sandboxRoot present, an absolute host-local MEDIA path is rejected by sandbox mapping, dropped, and converted to the Media failed fallback.

## Solution Assessment

No. Current main is not the best solution because it keeps a blanket sandbox-first rejection for host-absolute paths that may already be valid under the agent-scoped outbound media policy; the safer fix is a policy-checked fallback rather than unrestricted local path reads.
