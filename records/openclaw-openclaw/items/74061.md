## Summary

Current main still appears to reject the reported sandboxed Telegram final-reply local-media path before the normal agent-scoped outbound media staging path can read it. The issue is new, non-maintainer-authored, protected labels are absent, and adjacent closed reports cover relative paths, non-sandbox workspace media, CLI media roots, or different message-tool payloads rather than this sandbox-plus-absolute-host-path reply path.

## Best Possible Solution

Keep this issue open and fix the sandboxed normal-reply media path narrowly. When a sandbox root exists, absolute host-local `MEDIA:` paths should be allowed only if they are already managed media or pass the existing agent-scoped outbound media access policy for the active workspace/session, then staged through the existing outbound attachment path. Add positive coverage for a sandboxed Telegram-style final reply using an absolute `/home/openclaw/.openclaw/workspace/...` media path and negative coverage for unrelated host paths.

## Clownfish Work Prompt

Fix the Telegram/final-reply local media regression reported in openclaw/openclaw#74061. Current main drops sandboxed absolute host-local `MEDIA:` paths such as `/home/openclaw/.openclaw/workspace/<agent>/src/screenshot.png` before the normal agent-scoped outbound media staging path can handle them, causing `⚠️ Media failed.`. Keep the change narrow in `src/auto-reply/reply/reply-media-paths.ts` and related tests: when `sandboxRoot` exists, allow absolute host-local media only if it is already managed media or passes the existing agent/session/workspace-scoped outbound media access policy, then stage via the existing `resolveOutboundAttachmentFromUrl`/local media persistence path. Do not broadly allow arbitrary host paths, do not weaken sandbox checks for sandbox-relative/container paths, and do not change Telegram transport code unless investigation proves it is needed. Add regression coverage for a sandboxed normal/final reply with an absolute managed workspace path, plus negative coverage that an unrelated host path is still dropped. No changelog entry unless maintainers expect one for this user-facing Telegram media fix.
