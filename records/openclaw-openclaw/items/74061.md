## Summary

Keep open. Current main still preserves the sandboxed final-reply failure path: with a sandbox root present, host-absolute local MEDIA paths are resolved through sandbox mapping first, dropped when outside the sandbox or managed media roots, and converted into the visible media-failure fallback before Telegram receives a sendable attachment. The added cron-owned data point is relevant and appears adjacent rather than a duplicate, because cron direct delivery also lacks sandbox-to-host media translation for /workspace payloads.

## What This Changes

Fix Telegram outbound delivery so final replies with local MEDIA paths generated by agent workflows are staged and uploaded instead of becoming a Media failed fallback.

## Best Possible Solution

Add a narrow, policy-checked bridge for sandboxed final-reply media: allow managed media and files that pass the active agent/session outbound media policy, stage them through the existing outbound attachment path, and keep negative coverage for unrelated host paths. Maintainers should also decide whether cron-owned announce delivery should use the same sandbox-to-host media bridge.

## Reproduction Assessment

Yes. The current source and checked-in tests provide a high-confidence reproduction path: with sandboxRoot present, an absolute host-local MEDIA path outside sandbox/managed roots is rejected, dropped, and converted to the Media failed fallback. I did not run tests because this review is read-only and tests can create artifacts.

## Solution Assessment

No. Current main is not the best solution because it preserves a blanket sandbox-first rejection for host-absolute media that may already be valid under the agent-scoped outbound media policy; the safer fix is a policy-checked staging fallback, not a Telegram-specific upload change or unrestricted allowlist.
