## Summary

Keep open. Current main still drops sandboxed final-reply local media before the agent-scoped outbound media staging path can validate/read it, and the newer cron data point identifies an adjacent sandbox-to-host translation gap rather than a Telegram Bot API upload problem.

## What This Changes

Telegram final replies should deliver local media referenced by `<final>MEDIA:` from an agent workspace path instead of replacing it with `Media failed`.

## Best Possible Solution

Fix the sandboxed normal-reply media path narrowly: after sandbox mapping fails, allow only managed media or paths that pass the existing agent-scoped outbound media policy for the active workspace/session, then stage through the existing outbound attachment path. Add positive coverage for the reported sandboxed Telegram-style final reply and negative coverage for unrelated host paths; separately decide whether cron-owned announce delivery should share the same bridge fix.

## Reproduction Assessment

Yes. The current source and tests provide a high-confidence reproduction path: with `sandboxRoot` present, an absolute host-local `MEDIA:` path is rejected as outside the sandbox root, dropped, and converted to `⚠️ Media failed.`

## Solution Assessment

No. Current main is not the best solution because it keeps a blanket sandbox-first rejection for host-absolute paths that may already be valid under the agent-scoped outbound media policy; the safer fix is a policy-checked fallback, not an unrestricted host-path allowlist.
