## Summary

Keep open. Current main still preserves the reported failure mode for sandboxed final-reply media: host-absolute local paths are resolved as sandbox paths first, rejected when outside the sandbox root, then dropped into the visible `⚠️ Media failed.` fallback. The later cron comment is also useful evidence of the same sandbox-to-host translation gap on an adjacent delivery surface.

## What This Changes

The issue asks OpenClaw to deliver local filesystem media referenced by final `MEDIA:` replies through Telegram instead of replacing them with `Media failed`.

## Best Possible Solution

Narrowly fix sandboxed normal-reply media so host-absolute paths that are valid under the existing agent-scoped outbound media policy can be staged, while unrelated host paths remain blocked. Add positive coverage for a Telegram-style final reply using a host-absolute agent-workspace file and negative coverage for forbidden host paths; decide whether isolated cron direct delivery should share the same bridge or get a separate tracked fix.

## Reproduction Assessment

Yes. Current source plus `reply-media-paths.test.ts` provide a high-confidence reproduction path: with `sandboxRoot` present, an absolute host-local media path is rejected by sandbox resolution, skipped, and converted to `⚠️ Media failed.`

## Solution Assessment

No. The current sandbox-first behavior is not the best solution, and the report's Telegram multipart theory is not where the code points; the safer fix is an upstream policy-checked staging fallback, not a Telegram-specific bypass.
