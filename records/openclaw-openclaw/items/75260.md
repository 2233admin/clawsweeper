## Summary

Close as ClawHub/plugin scope. The PR adds an optional Vesicle iMessage channel as a bundled OpenClaw plugin, while current project direction and plugin docs route optional channel integrations to external plugins published through ClawHub or npm, and the existing SDK already covers the channel, setup, secret, HTTP route, webhook, SSRF, and runtime seams this branch uses. I also found one webhook target ambiguity bug that should be fixed before external publication, but it does not make this a core OpenClaw PR to keep open.

## What This Changes

Adds a bundled `extensions/vesicle` native iMessage channel plugin with Vesicle REST send/probe support, signed webhook ingress, config/secret metadata, docs, generated channel metadata, lockfile entries, and an install smoke script.

## Best Possible Solution

Keep OpenClaw core unchanged for this optional channel and carry Vesicle as an external ClawHub/npm channel plugin using the existing SDK seams. Reopen core work only if the external plugin hits a concrete missing generic plugin API or if maintainers explicitly decide Vesicle should become an official bundled plugin.

## Reproduction Assessment

Not applicable for the requested additive channel feature; the scope decision is based on current main docs/source and the PR diff. For the webhook review finding, a high-confidence repro is two enabled Vesicle accounts on the same webhook path with the same secret, then a signed POST that is accepted for whichever target was registered first.

## Solution Assessment

No. Adding Vesicle as a bundled core plugin is not the narrowest maintainable solution under the current plugin/ClawHub direction; the better path is an external plugin package, with the ambiguous webhook-target handling fixed before publication.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.84

Full review comments:

- **[P2] Reject ambiguous Vesicle webhook targets:** `extensions/vesicle/src/webhook.ts:206-212`
  - body: When two Vesicle accounts share a webhook path and HMAC secret, `targets.find(...)` silently accepts the request for whichever account was registered first. Existing webhook helpers treat this as ambiguous; use the single-target resolver or count matches so inbound messages cannot be routed under the wrong account.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff does not add third-party dependencies or CI supply-chain changes, but its new signed webhook ingress has an account-confusion risk on ambiguous same-path/same-secret matches.

Concerns:

- **[medium] Ambiguous webhook target match:** `extensions/vesicle/src/webhook.ts:206`
  - body: The Vesicle webhook handler authenticates by finding the first target whose HMAC matches. If two accounts reuse the same path and secret, the request is accepted for the first registered account instead of being rejected as ambiguous, which can misroute inbound messages and replies across accounts.
  - confidence: 0.86
