## Summary

Keep open. The PR is a useful focused classifier expansion, but it appears incomplete for the linked bug because the initial live delivery failure path still records every non-abort send error with `failDelivery`, leaving high-confidence permanent failures pending until a later recovery pass retries them. The prior issue discussion also explicitly called out this first-failure gap and raised a maintainer choice between a contained pattern add and a channel-owned classifier seam.

## What This Changes

The PR expands outbound delivery-queue recovery permanent-error regexes for payload-size, auth, and target-not-found failures, adds two recovery regression tests, and adds a changelog entry.

## Best Possible Solution

Land an end-to-end permanent-delivery classification fix that applies to both initial queue cleanup and recovery replay, keeps retry/backoff for transient failures, avoids treating all 400s as permanent, and either uses conservative shared patterns or a channel-owned classifier seam if maintainers choose that boundary.

## Reproduction Assessment

Yes. Current main can be checked by throwing `TelegramError: 400: Bad Request: message is too long` through recovery and by inspecting the live send catch path; recovery lacks the new patterns today, and live send still calls `failDelivery` for non-abort errors.

## Solution Assessment

No. The PR is a good partial recovery-side patch, but it is not the best complete fix for the linked bug until permanent first-send failures are moved to `failed/` immediately and the shared-versus-channel-owned classifier boundary is resolved.
