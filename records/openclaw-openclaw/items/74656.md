## Summary

Keep open. The PR is relevant and fixes the main recovery and first-send gaps, but its best-effort partial-failure logic still leaves a permanent payload in the retry queue whenever another payload failure in the same entry is transient; the branch is also currently dirty against the base.

## What This Changes

The PR expands outbound delivery permanent-error matching, applies permanent classification to live-send cleanup and best-effort partial failures, adds regression tests, and updates the changelog.

## Best Possible Solution

Revise this PR so any permanent best-effort per-payload failure moves the atomic queue entry to failed/, while transient-only failures keep the existing retry path; then rebase and run the targeted outbound tests plus changed-gate validation before landing.

## Reproduction Assessment

Yes. A focused unit test can run bestEffort delivery with two failing payloads, one classified permanent and one transient, and verify the PR currently calls failDelivery instead of moveToFailed.

## Solution Assessment

No. The PR is close, but the best maintainable short-term fix should let permanent partial failures win over transient partial failures because the queue cannot retry only the transient payload while discarding the permanent one.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Move any permanent best-effort failure to failed/:** `src/infra/outbound/deliver.ts:900-903`
  - body: This condition only moves the entry to failed/ when every partial failure is permanent. If one payload fails with `message is too long` and another fails transiently, the permanent payload is still left in `pending/` through `failDelivery`, so recovery can replay an entry that can never fully succeed. Track whether any partial failure is permanent and move the entry to `failed/` when it is.
  - confidence: 0.86
