## Summary

Keep this issue open. Current main still lacks the requested persisted per-job cron hook lifecycle: no `hooks` field in cron job types, Gateway schemas, CLI, or Control UI state, and scheduled/startup/manual cron execution paths still call the core job executor directly. The issue comments add concrete zero-token pre-gate use cases, and open PRs #45129 and #45465 remain implementation candidates that need maintainer and security review rather than cleanup closure.

## Best Possible Solution

Keep this issue open until maintainers either land an additive first-class cron hook lifecycle across persisted cron job types, Gateway schema/RPC, scheduled/startup/manual execution paths, CLI, Control UI, docs, and tests, or explicitly supersede it with a narrower canonical tracker. Treat existing plugin hooks as partial workarounds, not as the requested job-defined cron hook contract.

## Risks / Open Questions

- Shell hooks and webhook hooks would introduce user-configured code execution and outbound network behavior before scheduled model calls, so timeout, logging, secret exposure, sandbox/approval posture, and SSRF-style review are required.
- There are already open implementation PRs #45129 and #45465; starting a separate automated fix lane would likely duplicate or conflict with maintainer review.
- The requested surface spans persisted cron data, Gateway protocol, scheduler run paths, CLI, Control UI, docs, and tests, so this is broader than a narrow cleanup patch.
