## Summary

Keep open. Current main still lacks a persisted cron hook lifecycle, and the discussion plus open implementation candidate show this is active product and security design work rather than conservative cleanup.

## What This Changes

The issue requests pre, post, and error hooks for cron jobs so custom logic can gate, transform, retry, escalate, or notify around scheduled execution.

## Best Possible Solution

Land or explicitly supersede an additive cron hook lifecycle that covers persisted job/config types, Gateway protocol schemas and RPCs, scheduled/startup/manual execution paths, CLI, Control UI, docs, tests, and an operator-controlled security model for user hooks.

## Security Review

Status: needs_attention

Summary: No patch is under review, but the requested shell/webhook cron lifecycle is security-sensitive and needs explicit maintainer design before implementation.

Concerns:

- **[medium] Define user hook execution boundaries:**
  - body: Shell and webhook hooks can execute local commands or outbound network calls before and after scheduled jobs. The design needs clear permission, timeout, logging, failure, and skip semantics so cron hooks do not become an implicit privileged execution path.
