## Summary

Keep open. Current main still lacks the macOS app LaunchAgent ownership guard for doctor service repair, and the linked #67631 report remains open, but this PR should not merge as written because the new label-only ownership check can misclassify a CLI profile named `mac` as OpenClaw.app ownership.

## What This Changes

This PR adds macOS OpenClaw.app LaunchAgent ownership detection to doctor daemon and service-config repair, with docs, changelog, and regression tests so CLI repair skips standalone gateway LaunchAgent mutations when the app owns launchd.

## Best Possible Solution

Revise this PR in place so app ownership detection proves the launchd job is the real OpenClaw.app agent, preferably by checking `ProgramArguments` or another app-specific marker, while preserving the default-profile skip behavior and adding explicit regression coverage for a CLI `OPENCLAW_PROFILE=mac` LaunchAgent.

## Reproduction Assessment

Yes. The original bug has a concrete path in #67631, and the PR regression is reproducible by creating or mocking a CLI LaunchAgent with label `ai.openclaw.mac` under `OPENCLAW_PROFILE=mac`; the new label-only scanner then treats it as OpenClaw.app ownership.

## Solution Assessment

No, not as written. The product direction is narrow and appropriate, but the implementation must distinguish real OpenClaw.app launchd ownership from the supported profile-derived CLI label before merge.
