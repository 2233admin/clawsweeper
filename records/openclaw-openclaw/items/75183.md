## Summary

Keep open. This PR has the protected `maintainer` label and source review found a blocking deferred-startup regression: the PR repins the HTTP route registry after loading startup plugins, but leaves channel registry lookups pinned to the initial pre-runtime registry.

## What This Changes

The PR changes CLI plugin/channel inspection policy, bundled plugin runtime dependency repair and staging, Gateway startup readiness/runtime loading, docs, changelog, and regression tests around bundled runtime deps.

## Best Possible Solution

Update the PR so deferred startup plugin replacement refreshes the channel registry as well as the HTTP route registry, or defer channel pinning until after startup plugins are loaded; add focused regression coverage for channel/outbound lookup after post-bind startup plugin load, then rerun the targeted Gateway/CLI checks and changed gate.

## Reproduction Assessment

Yes. The source-level reproduction is clear: under the PR, Gateway runtime state pins channels before startup plugins load, deferred replacement only repins HTTP routes, and registry-backed outbound adapter lookup reads the pinned channel registry. A focused test can simulate post-bind startup plugin load and assert `loadChannelOutboundAdapter` resolves a loaded startup channel.

## Solution Assessment

No. The PR direction is close, but the current implementation misses the channel registry side of the runtime swap. The safer narrow fix is to repin both channel and HTTP route registries when replacing the attached plugin runtime, or avoid the initial channel pin until runtime plugins are loaded.
