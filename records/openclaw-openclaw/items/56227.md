## Summary

Keep open. Current main has useful restart mitigations, including invalid-config preflight, last-known-good config recovery, stale-listener cleanup, and post-restart health diagnostics, but it does not implement the requested atomic swap. Both the managed-service and SIGUSR1 paths still stop or close the active Gateway before a replacement is proven healthy, so a failed replacement can still leave users without a listener or channel transport.

## Best Possible Solution

Keep this issue open and design a conservative restart handoff. A good path would validate config first, start a replacement on an alternate port/socket or through a supervisor handoff, probe `/readyz`, then switch advertised/connectable endpoints and stop the old Gateway only after the replacement is healthy. If the replacement fails, leave the old Gateway running and deliver a failure notice through the still-active channel. Tests should cover both managed-service restarts and SIGUSR1/config-apply restarts, including replacement startup failure.

## Risks / Open Questions

- A true no-disconnect restart may need a larger design than the current service-manager wrappers support, especially for same-port listeners and channel transports.
- Keeping the old process alive while probing a new one needs careful token, port, discovery, and channel ownership handling to avoid split-brain or duplicate polling.
