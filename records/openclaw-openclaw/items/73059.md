## What This Changes

The issue reports that Windows source installs launched through the global npm junction can skip the Control UI build because `scripts/ui.js` does not call `main()` when the launch path and module path differ.

## Best Possible Solution

Add a small script-owned direct-execution helper that normalizes both the launch path and module path through realpath with a resolved-path fallback, export or isolate it for focused tests, and keep the fix scoped to the UI wrapper plus regression coverage.

## Reproduction Assessment

Yes. The reporter gives a concrete Windows linked-source command, and current main shows the exact guard that would skip `main()` when the entrypoint path and module path differ; a symlinked-entrypoint smoke check reproduced the same silent no-op shape.

## Solution Assessment

Yes. The narrow maintainable fix is to normalize both sides of the direct-run check before calling `main()`, rather than changing gateway auto-build behavior, UI app code, package metadata, or pnpm launcher handling.

## Repair Work Prompt

Fix the Windows linked-source Control UI build wrapper bug reported in https://github.com/openclaw/openclaw/issues/73059. In `scripts/ui.js`, replace the raw `path.resolve(process.argv[1]) === fileURLToPath(import.meta.url)` direct-execution guard with a small helper that compares both paths after realpath normalization, falling back to resolved paths if realpath fails, so junction/symlink launches still call `main()`. Keep the module importable for tests. Add focused coverage in `test/scripts/ui.test.ts` for a junction/symlink-style mismatch between the launch path and module path. Do not change UI app code, package manager resolution, package metadata, or gateway auto-build policy. Related #52291 is adjacent pnpm launcher work and should not be treated as the fix.
