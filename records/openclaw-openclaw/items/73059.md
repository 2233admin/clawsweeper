## Best Possible Solution

Add a small script-owned direct-execution helper that compares the launch path and module path after realpath normalization, keeps the module importable for tests, and adds focused regression coverage for the junction/symlink mismatch without changing UI app code or package metadata.

## Repair Work Prompt

Fix the Windows linked-source Control UI auto-build bug tracked by https://github.com/openclaw/openclaw/issues/73059. Current `scripts/ui.js` decides direct execution by comparing `path.resolve(process.argv[1])` to `fileURLToPath(import.meta.url)`, which can fail when the script is launched through the global npm junction path. Add a small script-owned helper that normalizes both paths through `fs.realpathSync.native` or the closest existing safe realpath helper, falling back to resolved paths if realpath fails, and use it to decide whether to call `main()`. Keep `scripts/ui.js` importable for tests. Add focused regression coverage in `test/scripts/ui.test.ts` that simulates a junction/symlink mismatch between argv1 and the module URL. Treat PR #52291 as adjacent context only; do not change the Windows pnpm quoting behavior unless the direct-execution fix requires a tiny shared helper. Do not modify UI app code, dependency pins, package metadata, or unrelated Control UI asset resolution. Add a single-line changelog Fixes entry if repo policy requires one for this user-visible Windows source-install fix, crediting @LouisGameDev.
