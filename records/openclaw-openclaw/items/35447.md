## Summary

Keep this issue open. Current main at b85edb3 still stores cron job agent/session metadata and stamps it on jobs created through the agent cron tool, but the cron service contract, list options, gateway protocol params, gateway handlers, and tool calls still do not carry or enforce trusted caller context for list/update/remove/run/enqueueRun. The related implementation PRs #36517 and #49174 are closed unmerged; #49175 is a narrower run-history job-name leakage follow-up, and #49183 is a broader policy RFC rather than a replacement.

## Best Possible Solution

Keep this issue open and implement a core cron caller-context contract: thread trusted agentId/sessionKey or an explicit operator/admin override through the agent tool, gateway protocol, gateway handlers, and CronService operations; filter list/listPage; deny update/remove/run/enqueueRun across ownership boundaries; define legacy-job behavior; and add service, gateway protocol/handler, and cron-tool regression tests. Coordinate with #49175 for run-history name leakage and keep #49183 as the broader policy-design discussion.
