## Summary

Keep open. Current main still stores and stamps cron job `agentId`/`sessionKey` metadata on creation, but the cron service contract, list options, gateway protocol, gateway handlers, and cron tool calls still do not carry or enforce trusted caller context for list/update/remove/run/enqueueRun. The related implementation PRs are closed unmerged; #49175 is a narrower run-history name leak and #49183 is a broader policy RFC, not a replacement for this issue.

## What This Changes

Add per-agent/session isolation to cron job visibility and mutations so callers can list, update, remove, and run only jobs owned by their agent/session unless an explicit trusted admin override applies.

## Best Possible Solution

Keep this issue open and implement a core cron caller-context contract: derive trusted `agentId`/`sessionKey` or an explicit operator/admin override from agent/gateway context, thread it through the gateway protocol/handlers and `CronService` operations, filter `list`/`listPage`, deny `update`/`remove`/`run`/`enqueueRun` across ownership boundaries, define legacy-job behavior, and add service, gateway, and cron-tool regression tests. Coordinate with #49175 for run-history job-name leakage and keep #49183 as the broader per-session tool policy discussion.
