## Summary

Keep this PR open. Current main still uses the DST-ambiguous end-of-hour Date setter in both Usage aggregation and selected-hour filtering, and a pure Node check confirms the current loop can fail to advance during the repeated local hour at DST fallback. The PR is not already implemented or superseded by the provided related context; it looks like a focused candidate fix, with one test-cleanup review comment still worth addressing.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate. The best path is a scoped Usage fix that replaces ambiguous local end-of-hour mutation with monotonic next-hour boundary stepping for both aggregation and selected-hour filtering, plus DST fallback/spring-forward regression tests with clean `TZ` restoration.
