## Summary

Keep this PR open. The Discord component-send problem is still real on current main, but the proposed branch needs adaptation to the current split Discord messaging runtime and a required user-facing changelog entry before it can land.

## What This Changes

The PR adds a Discord shared message-tool `components` schema, forwards top-level `params.components` through Discord send/edit actions, and adds focused Discord tests for those paths.

## Best Possible Solution

Land a rebased version that keeps `components` as the documented OpenClaw Discord component-spec object, exposes that schema for shared message sends, forwards top-level send components through `handle-action.ts`, wires component edits into `runtime.messaging.messages.ts` and `runtime.messaging.runtime.ts`, and adds focused tests plus a user-facing changelog entry.

## Reproduction Assessment

Yes. Static current-main inspection gives a high-confidence reproduction path: call the Discord shared `message` tool with `action: "send"` and a top-level `components` object; the schema does not advertise it and the bridge never forwards `params.components`.

## Solution Assessment

No, not as currently proposed. The send direction is correct, but the edit support is implemented against the old monolithic runtime instead of current main's split handlers, and the user-facing fix is missing a changelog entry.
