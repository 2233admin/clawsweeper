## Summary

Keep open: the underlying Discord component send bug is still present on current main, and the PR is useful, but the branch is not merge-ready because its component edit runtime change targets the old monolithic messaging file after main split that logic into dedicated send/message/runtime modules.

## What This Changes

The PR adds a Discord `components` schema contribution to the shared message tool, forwards top-level `components` through Discord send/edit actions, and adds focused tests for those paths.

## Best Possible Solution

Rebase the branch onto current main, keep the schema aligned with the documented OpenClaw Discord component-spec object, forward top-level `components` for send and edit in `handle-action.ts`, and implement component edits in `runtime.messaging.messages.ts` with `editDiscordComponentMessage` exposed through `runtime.messaging.runtime.ts`.

## Reproduction Assessment

Yes. A static current-main reproduction is clear: call the Discord shared `message` tool with `action: "send"` and a top-level `components` object; discovery does not expose that parameter and the local send bridge never forwards it.

## Solution Assessment

No, not as currently proposed. The send/schema direction is the right narrow fix, but the edit support must be moved from the old monolithic runtime hunk into the split current-main messaging runtime files before merge.
