## Summary

Keep this PR open: current main still lacks the requested Discord shared message-tool `components` send path, and the PR is directionally useful, but the latest branch is not correct against current main because its component edit support targets the pre-split Discord messaging runtime instead of the split current-main handlers.

## What This Changes

The PR adds a Discord shared message-tool `components` schema contribution, forwards top-level `components` through Discord send/edit message actions, and adds focused Discord tests for those paths.

## Best Possible Solution

Rebase the PR onto current main and keep the narrow send fix: expose the documented OpenClaw Discord component-spec object on the shared message-tool schema, forward top-level `components` from `send`, and either drop edit support from this PR or wire it through `runtime.messaging.messages.ts` plus `runtime.messaging.runtime.ts` with targeted tests.

Yes. Static current-main inspection gives a high-confidence reproduction path: call the Discord shared `message` tool with `action: "send"` and a top-level `components` object; discovery does not advertise that field and the local bridge never forwards it to the lower sender.

## Solution Assessment

No, not as currently proposed. The send direction is the right narrow fix, but the latest branch also adds edit support in the old monolithic runtime shape, so it must be adapted to the split current-main runtime before merge.
