## Summary

Current main still lacks a TUI plain-text/accessibility mode. The config schema only accepts `ui.seamColor` and `ui.assistant`, TUI/tool render paths still emit emoji and Unicode glyphs unconditionally, and the related implementation PR #9685 is closed unmerged.

## Best Possible Solution

Keep this open and implement it as core TUI/config accessibility work: add a documented `ui.accessibility.plainText` or equivalent config key, validate it, generate config help/labels/schema docs, load it once in `runTui`, thread it through tool display and TUI components, provide ASCII alternatives for emoji/Unicode glyphs, and add focused rendering/config tests. Treat PR #9685 as prior art, not completed work.

## Clownfish Work Prompt

Implement a TUI plain-text accessibility mode for the open issue about disabling emoji and Unicode symbols in the TUI. Use PR #9685 only as prior art; do not copy unrelated changes blindly. Add a documented `ui.accessibility.plainText?: boolean` config option defaulting to false, updating `OpenClawConfig`, zod validation, generated schema help/labels/schema metadata, public config/TUI docs, and changelog with `Thanks @robin24` attribution. Load the setting once in `runTui` and thread it through TUI components without config reads during render. When enabled, make TUI tool headers/summaries and picker/status glyphs use ASCII alternatives: avoid tool emoji prefixes, use `-` instead of box drawing separators, `>` instead of selection arrows, and `...` instead of ellipsis where the TUI itself renders those placeholders. Cover `ChatLog`, `ToolExecutionComponent`, filterable/searchable selectors, settings/theme cursors, and shared tool-display formatting as needed. Keep behavior unchanged when the option is false, avoid changes to messaging-channel emoji/reaction behavior, and keep core generic rather than extension-specific.
