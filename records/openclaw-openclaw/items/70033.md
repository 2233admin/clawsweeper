## Summary

Keep this open. Current main has adjacent malformed streamed tool-call argument repair, but the activation predicate still excludes `bedrock-converse-stream`, while the Bedrock path uses pi-ai's parser behavior that can leave tool-call arguments as `{}` when streamed JSON cannot be parsed. The reported Bedrock/write/exec failure class is therefore not proven fixed on main.

## What This Changes

The issue reports Bedrock/Claude large-content tool calls for write/exec arriving with empty argument objects and causing repeated validation failures.

## Best Possible Solution

Add a Bedrock-aware guard in the provider stream or embedded-runner tool-call repair layer so malformed or unrecoverable empty required-argument tool calls do not execute repeatedly as `{}`. Reuse the existing repair wrapper where it is safe, and surface a clear retryable provider/tool-call error when the original content is not recoverable.

## Reproduction Assessment

Yes for a focused code-level reproduction: simulate a `bedrock-converse-stream` tool call whose streamed input is empty or incomplete for a required-argument tool and verify current main passes `{}` through to validation. The exact live 5-7KB Bedrock threshold remains unproven without AWS credentials and the reported model.

## Solution Assessment

No, current main is not the best solution because its malformed-argument repair is scoped away from Bedrock. The narrow maintainable path is Bedrock-specific stream repair/error handling with regression tests, not changing write/exec schemas or silently chunking all large file content first.
