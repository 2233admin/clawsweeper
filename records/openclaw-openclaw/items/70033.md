## Summary

Keep open. Current main still routes the reported Amazon Bedrock model through `bedrock-converse-stream`, but malformed streamed tool-call argument repair is only enabled for Kimi over `anthropic-messages` and for `openai-completions`; the Bedrock stream path can still finalize missing or unparseable `toolUse.input` as `{}` and then hit write/exec parameter validation. The prior ClawSweeper comment remains accurate against current main, and an unauthenticated GitHub search found no open PR referencing this issue.

## What This Changes

The issue requests that large Amazon Bedrock write/exec tool-call arguments stop degrading into empty `{}` validation failures and instead be repaired or fail closed with a clear bounded retry/error.

## Best Possible Solution

Keep this issue open and implement a focused Bedrock Converse fix in the embedded runner/provider stream path: repair recoverable malformed streamed `toolUse.input`, and fail closed with a clear bounded retry/error when a parameterized tool reaches dispatch with empty or malformed arguments. Add Bedrock fake-stream coverage for large `write` content and `exec` command/heredoc-style arguments while preserving parameterless tools and the existing `openai-completions` repair behavior.
