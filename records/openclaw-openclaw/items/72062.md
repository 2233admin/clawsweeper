## Summary

Keep this issue open. Current main still has per-agent tool-name allow/deny plus positive exec allowlist/script-path matching, but there is no typed, documented, or enforced per-agent negative policy for shell wrapper or script paths, so the reported exec-policy gap remains valid.

## What This Changes

The issue requests a per-agent negative policy for denying shell wrapper or script paths invoked through `tools.exec`.

## Best Possible Solution

Implement a typed and documented per-agent exec path deny policy that reuses the existing shell-wrapper/script candidate resolution, applies before approval prompts and durable approval reuse, makes deny win over allowlist and approvals, covers gateway and node exec, and includes focused regression tests plus public docs.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: configure a per-agent `tools.deny` for an MCP namespace, allow `sh`/`bash` through exec allowlist, and invoke `sh <sensitive-wrapper>`; the tool policy sees only the `exec` tool name while exec approval code evaluates positive allowlist state without a wrapper-path deny.

## Solution Assessment

Unclear. A negative script-path policy addresses the real gap, but the exact API name and nesting should be decided by maintainers because it changes core exec security semantics and likely belongs with the existing `tools.exec` policy fields.
