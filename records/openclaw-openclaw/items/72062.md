## Summary

Keep this issue open. Current main still has per-agent tool-name allow/deny and positive exec allowlist/script-path matching, but no typed, documented, or enforced per-agent negative policy for shell wrapper or script paths, so the reported delegation bypass remains valid.

## What This Changes

The issue requests a per-agent policy for denying specific shell wrapper or script paths invoked through exec, so agents cannot bypass tool-name denies by calling allowed interpreters with sensitive wrapper scripts.

## Best Possible Solution

Add a typed, documented per-agent exec path denial policy that reuses the existing shell-wrapper/script candidate resolution, applies before approval prompts and durable approval reuse, defines deny precedence clearly, and includes regression coverage for gateway and node exec.

## Reproduction Assessment

Yes. The source-level reproduction is high-confidence: configure a per-agent `tools.deny` for an MCP namespace, allow `sh` or `bash` through exec allowlist, then invoke `sh <sensitive-wrapper>`; current tool policy sees only the exec tool name and the exec approval path has no negative script-path rule.

## Solution Assessment

Unclear. A negative wrapper/script path policy is the right class of fix, but maintainers should choose the final API name and nesting, likely close to the existing `tools.exec` policy fields, because it changes exec security semantics.
