## Summary

Keep open. Current main still lacks a typed, documented, and enforced per-agent negative policy for exec wrapper or script paths; the existing tool policy is name-based, while exec script-path resolution is only used for allowlist and approval matching.

## What This Changes

The request adds a per-agent exec policy that denies configured shell or interpreter wrapper script paths so an orchestrator agent cannot invoke sensitive shared scripts directly despite MCP tool-name denies.

## Best Possible Solution

Add a documented per-agent negative exec script-path policy in the core exec security surface, reuse the existing shell-wrapper/script candidate resolution, make deny win before allowlist satisfaction, durable approval reuse, allow-always persistence, and approval prompts, and cover both gateway and node exec with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure per-agent tools.deny for an MCP namespace, allow sh or bash through exec allowlist, then invoke sh <sensitive-wrapper>; current main filters only the exec tool name while exec policy has no script-path deny check.

## Solution Assessment

Unclear. A negative script-path policy is the right class of fix, but the API name, nesting, interpreter coverage, path resolution semantics, and gateway/node precedence should be decided by maintainers because this changes core exec security behavior.
