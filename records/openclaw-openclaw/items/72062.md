## Summary

Keep this issue open. Current main still has per-agent tool-name allow/deny and positive exec allowlist/script-path matching, but no typed, documented, or enforced per-agent negative policy for shell wrapper or script paths, so the reported exec-policy gap remains valid and security-sensitive.

## What This Changes

Request a per-agent exec wrapper/script-path deny policy, such as `agents.list[].tools.deny_wrappers`, that blocks resolved interpreter script operands before exec approval or execution.

## Best Possible Solution

Add a documented, typed per-agent negative exec script-path policy under the exec security surface, reuse the existing shell-wrapper/script candidate resolution, apply deny before allowlist satisfaction, durable approval reuse, allow-always persistence, and approval prompts, and cover both gateway and node exec with focused regression tests.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is available: configure `agents.list[].tools.deny` for an MCP namespace, allow `sh` or `bash` through exec allowlist, then invoke `sh <sensitive-wrapper>`; current tool policy sees only the exec tool name while exec approval code has no negative script-path check.

## Solution Assessment

Unclear. A negative script-path policy is the right class of solution for this gap, but the exact API name, nesting, interpreter coverage, and precedence semantics should be decided by maintainers because this changes core exec security behavior.
