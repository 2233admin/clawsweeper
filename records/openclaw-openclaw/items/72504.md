## Summary

Keep this open. Current main still globally strips the receiving Feishu bot's own mention before agent prompt construction, and open PR #72661 explicitly closes this issue while open PR #72532 is an alternative implementation, so the conservative maintainer outcome is to resolve the PR path before closing the issue.

## What This Changes

Request to fix Feishu mention normalization so multi-bot group messages keep the receiving bot's own mention in agent-facing content instead of causing silent NO_REPLY outcomes.

## Best Possible Solution

Land one focused Feishu plugin fix that strips only a true leading addressing self-mention while preserving inline or repeated self-mentions for multi-bot routing, keeps p2p and group slash-command behavior intact, adds regression coverage, and then closes this issue through the merged PR.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to parse a Feishu group text event whose mentions include Bot A and Bot B while botOpenId is Bot A; current normalizeMentions removes Bot A's placeholder before buildFeishuAgentBody constructs the model-facing prompt.

## Solution Assessment

No for the issue body's broad remove-all-botStripId suggestion, because current code and #35994 history show leading self-mention stripping also protects slash-command dispatch. The best path is the narrower #72661 direction after its slash-boundary and changelog feedback is addressed.
