## Summary

Keep this open. Current main still strips every Feishu mention whose open_id matches the receiving bot before building the agent-facing body, the latest stable release contains the same behavior, and open PR #72661 already references this issue with closing syntax as the narrower implementation candidate.

## What This Changes

The issue asks Feishu group handling to preserve the receiving bot's own <at> mention in multi-bot group messages so each explicitly addressed bot has enough prompt context to respond instead of returning NO_REPLY.

## Best Possible Solution

Review and land #72661 or an equivalent Feishu-owned patch that strips only the leading bot address prefix used for command routing, preserves later or semantic self-mentions in the agent-facing body, adds multi-bot regression coverage, and then closes or supersedes the duplicate open candidate #72532.

## Reproduction Assessment

Yes. A unit-level Feishu text event like `Hey @_bot_a @_bot_b please coordinate`, with both mentions in event.message.mentions and botOpenId set to Bot A, flows through parseFeishuMessageEvent and produces content where Bot A's tag is removed while Bot B's tag remains.

## Solution Assessment

No, the issue's suggested full removal of botStripId is not the best narrow fix because current comments and tests preserve #35994 slash-command behavior. The safer path is the leading-prefix-only strip proposed by #72661, with regression tests for both command routing and multi-bot prompt context.
