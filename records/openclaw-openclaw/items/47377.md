## Summary

Keep open. Current main still stores full cumulative WebChat stream snapshots as committed segments and renders the live stream unchanged; this PR narrows the committed-segment duplication but still lets the live preview repeat already committed text after a tool boundary. The same bug is also covered by overlapping open PRs, so this needs maintainer coordination rather than cleanup closure.

## What This Changes

The PR changes Control UI tool-stream segment commits to store only the newly streamed suffix after the total length of prior committed stream segments.

## Best Possible Solution

Land one focused WebChat streaming fix that treats cumulative assistant snapshots consistently across committed pre-tool segments and the live preview, preferably through a small shared helper or explicit committed-offset state plus regression tests for multi-tool cumulative snapshots.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P2] Deduplicate the live stream preview too:** `ui/src/ui/app-tool-stream.ts:444-451`
  - body: This only slices text when committing `chatStreamSegments`. After a tool starts, the next chat delta still assigns the gateway's full cumulative snapshot to `chatStream`, and `buildChatItems` renders that live stream unchanged after the committed segment. The pre-tool text can still appear again in the live bubble until another tool event or final reload, so the live preview needs the same committed-offset dedupe.
  - confidence: 0.86
