## Summary

Keep open. Current main does not contain the Claude SDK runtime or dispatch surface, so the PR is not obsolete, but the proposed branch still has blocking correctness and security-sensitive parity gaps around SDK settings loading, tool allow/deny mapping, requester role context, and native tool error translation.

## What This Changes

The PR branch adds an opt-in Claude Agent SDK runtime adapter, generated `.claude/agents` files from skills, runtime dispatch plumbing, schema/API/docs updates, and tests, while the current head keeps the legacy embedded runtime as the default.

## Best Possible Solution

Keep the PR open and require a smaller, maintainer-reviewed opt-in runtime implementation that preserves existing OpenClaw tool policy, requester context, project settings, transcript/error semantics, and SDK dependency controls before any later default-flip discussion.

## Reproduction Assessment

Yes. The blocking issues are reproducible by static inspection: compare the PR's `sdk.query({ options })` and native tool bridge with the Claude Agent SDK 0.1.77 `Options` contract plus current OpenClaw tool-policy and embedded-run parameter forwarding.

## Solution Assessment

No. The current branch is not the best merge shape because it introduces the desired SDK seam but leaves policy/settings parity gaps; a safer fix is to keep the opt-in runtime, normalize OpenClaw tool ids to SDK/MCP names, load project settings intentionally, preserve role context, and add focused tests plus live smoke proof.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Normalize SDK deny tool names:** `src/agents/claude-sdk/sdk-tool-policy.ts:56-61`
  - body: OpenClaw configs commonly deny canonical ids such as `exec`, but this helper forwards the raw string into the SDK. The SDK built-in is named `Bash`, so `tools.deny: ["exec"]` still leaves command execution available under the Claude SDK path. Map canonical OpenClaw ids and groups to SDK built-in names before populating `disallowedTools`.
  - confidence: 0.9
- **[P1] Load project settings for SDK runs:** `src/agents/claude-sdk/run.ts:552-563`
  - body: The SDK documents that omitted `settingSources` loads no filesystem settings and must include `project` for `CLAUDE.md`. This options object never sets it, so opt-in Claude SDK runs ignore project `.claude` settings and repo instructions that the runtime may rely on for permissions and behavior. Set the intended sources explicitly and cover it with a test.
  - confidence: 0.88
- **[P2] Map toolsAllow before passing SDK tools:** `src/agents/claude-sdk/run.ts:499-501`
  - body: `toolsAllow` is stored and passed around as OpenClaw tool ids such as `exec` and `read`, but the SDK `tools` option expects SDK built-in names like `Bash`/`Read` or the `claude_code` preset. Forwarding the raw list makes constrained cron/agent turns lose the intended tools instead of matching embedded-runtime allowlist behavior.
  - confidence: 0.87
- **[P2] Forward requester role ids into native tools:** `src/agents/claude-sdk/run.ts:253-256`
  - body: The embedded path forwards `memberRoleIds` into `createOpenClawCodingTools`, where OpenClaw tools receive it as `agentMemberRoleIds`. The SDK path omits that field, so role-scoped group/tool behavior and subagent requester-origin resolution can run without the requester's trusted role context.
  - confidence: 0.84
- **[P2] Mark structured tool failures as MCP errors:** `src/agents/claude-sdk/native-tools-adapter.ts:183-203`
  - body: Many OpenClaw tools report failures via returned `details.status: "failed"` rather than throwing. `agentResultToMcp` currently returns `isError: false` for every non-thrown result, which tells Claude that failed native tool calls succeeded and can make it continue from invalid assumptions.
  - confidence: 0.86
