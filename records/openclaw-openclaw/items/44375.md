## Summary

Keep open. Current main still has the reported default-agent fallback: once `agents.list` is non-empty, unmatched routing chooses `agents.list[].default`, otherwise the first list entry, otherwise `main`. The CLI config set path still applies the requested `agents.list` value without a targeted warning when implicit `main` would be displaced, while docs and tests preserve the first-entry fallback contract.

## Best Possible Solution

Keep this open and implement a conservative UX guard rather than changing routing semantics: when `agents.list` first becomes non-empty without any `default: true` and the prior effective default was implicit `main` or `main` state exists, warn clearly and recommend adding `{ id: "main", default: true }`. Preserve the documented fallback rule unless maintainers choose a breaking config-contract change, and add targeted config/routing tests plus docs/changelog coverage.
