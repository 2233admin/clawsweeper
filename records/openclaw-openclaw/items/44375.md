## Summary

Keep this issue open. Current main still has the reported default-agent fallback: once `agents.list` is non-empty, unmatched routing uses `default: true` if present, otherwise the first list entry, and only falls back to `main` when there are no agent entries. The config set path still applies `agents.list` writes without a targeted implicit-main displacement warning or auto-added `main`, and docs/tests continue to describe and preserve the first-entry fallback.

## Best Possible Solution

Keep this issue open and add a conservative guard in the config mutation and/or doctor surface: when `agents.list` first becomes non-empty without any `default: true` and implicit `main` state likely exists, warn clearly and recommend adding `{ id: "main", default: true }`. Preserve the documented fallback rule unless maintainers choose a breaking config-contract change, and add targeted tests plus routing/config docs updates for the new warning.

## Risks / Open Questions

- Changing the fallback rule outright could break users who intentionally rely on the first `agents.list` entry as the default.
- Auto-inserting `main` during raw config mutation could surprise automation; a warning, dry-run finding, or doctor check is lower-risk.
- Existing orphan/session-key migration coverage helps with some key canonicalization cases, but it does not prevent the silent default-agent routing change reported here.
