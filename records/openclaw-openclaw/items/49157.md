## Summary

Keep open. Current main still has the reported live-PID abandoned session-lock recovery gap: session locks remain age/PID based with a 30-minute stale default, embedded runs can derive very large maxHoldMs budgets from long run timeouts, and the current tests still preserve that oversized hold behavior rather than proving short bounded recovery.

## Best Possible Solution

Keep this issue open as the focused session-lock recovery bug. The best next step is a maintainer-owned design choice for abandoned process-alive locks: either cap embedded-run session-lock maxHoldMs in a way that cannot corrupt active long runs, narrow the lock scope to actual transcript mutation windows, or add a heartbeat/lease signal. Add direct regression coverage for a process that remains alive while its lock-holding operation stops making progress. Coordinate with #13744 for broader configurability and lock-scope work, but do not close this until the live-PID leak path has source and test coverage.
