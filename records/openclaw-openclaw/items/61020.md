## Summary

Keep open. The stale-response replay problem is real on current main because `lastAssistant` is still selected from the full post-run snapshot and is later used for payload fallback, but this PR’s proposed boundary is not safe across compaction paths where the snapshot can be shorter than the pre-prompt history count.

## What This Changes

The PR changes the pi embedded runner to derive `lastAssistant` from `messagesSnapshot.slice(prePromptMessageCount)` and adds a payload helper regression test for idle-timeout behavior with no current assistant.

## Best Possible Solution

Add a compaction-safe production helper for current-turn assistant selection, then use that helper consistently for failover, payload fallback, usage metadata, and hooks without relying on stale array indexes. Cover both zero-token idle timeout and post-compaction current-assistant cases.

## Reproduction Assessment

Yes. A focused code-path reproduction is straightforward: current main can choose a prior assistant when no current assistant exists, while the PR’s proposed slice can lose a current assistant when compaction has shortened or restructured the snapshot before selection.

## Solution Assessment

No. The proposed line fixes the simple idle-timeout case, but the best fix needs a stable turn boundary or message marker that survives compaction rather than reusing `prePromptMessageCount` as an index into a later snapshot.

## Review Findings

- **[P2] Preserve current assistant across compaction:** `src/agents/pi-embedded-runner/run/attempt.ts:1912`
  - body: Slicing `messagesSnapshot` from `prePromptMessageCount` assumes the post-run snapshot still has the same indexes as the pre-prompt history. That drops `lastAssistant` for failover, hook, and payload fallback handling on compaction paths.
