## Summary

Current main still lacks the requested WhatsApp per-group debounce setting. The config type, strict schema, generated metadata, docs, and runtime all continue to expose a single channel/account-level debounce value rather than resolving a debounce window from the inbound group entry.

## What This Changes

Add a per-group `debounceMs` setting for WhatsApp group configuration, with possible later plugin-specific equivalents for Telegram, Discord, and other group-capable channels.

## Best Possible Solution

Implement the WhatsApp plugin path first: accept group-scoped `debounceMs`, resolve exact group before wildcard group before account/channel/global fallback, and make the runtime batching behavior cover the reported media-plus-text sequence without dropping earlier media context. Cross-channel parity should follow as separate plugin-specific work or a consciously designed shared config contract.

## Repair Work Prompt

Implement WhatsApp per-group inbound debounce for the feature request. Add `debounceMs?: number` to WhatsApp group config and the strict Zod schema, update generated channel config metadata/help/docs as required by existing config tooling, and update the WhatsApp inbound runtime so each inbound group message resolves debounce in this order: exact group entry, wildcard group entry, then existing account/channel/global fallback. Use the existing `createInboundDebouncer.resolveDebounceMs` seam rather than adding a separate batching mechanism. Cover the reported image/image/text sequence: media messages in a debounced group must not each trigger separate agent turns, and earlier media context must not be dropped when the batched text message arrives. Keep Telegram/Discord out of scope unless maintainers explicitly request a shared cross-channel contract. Add user-facing docs and a changelog entry with reporter attribution if behavior changes ship.
