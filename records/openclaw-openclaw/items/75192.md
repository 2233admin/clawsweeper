## Summary

Keep open. Current main still has the backend contract needed to create an agent-scoped dashboard session, but the WebChat/Control UI has no create-new-chat flow that lets the user choose an agent; the visible New session button still sends `/new` in the active session. Related #73725 is an open maintainer PR for the generic create-and-switch path, so this should be sequenced with that work rather than closed.

## What This Changes

Add a WebChat/Control UI flow that lets users choose a configured agent when creating a new chat session.

## Best Possible Solution

Land or resolve the generic WebChat `sessions.create` create/switch path first, then add a focused agent picker that reuses `agents.list`, calls `sessions.create` with the chosen `agentId`, switches to the returned session key, refreshes session and identity state, and adds targeted UI/controller tests plus user-facing docs if the visible workflow changes.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction exists: configure multiple agents, open the Control UI chat, and click the plus/New session control; current main sends `/new` in the active session and has no UI path that passes a selected `agentId` to `sessions.create`.

## Solution Assessment

Yes, with sequencing. The narrow maintainable fix is a UI/controller addition over existing `agents.list` and `sessions.create`; no Gateway protocol change appears necessary unless the UI work exposes a concrete missing field.
