## Summary

Keep open. Current main has an agent-aware session dropdown and a backend `sessions.create` RPC that accepts `agentId`, but the visible chat “New session” action still runs `/new` against the current session and there is no Control UI flow to create and switch to a newly selected-agent chat.

## What This Changes

Requested WebChat UI support for creating a new chat session with a selected configured agent.

## Best Possible Solution

Add a focused Control UI new-chat flow that uses the existing `agents.list` data to present configured agents, calls `sessions.create` with the selected `agentId`, switches to the returned session key, refreshes session options/history/avatar, and keeps the existing session dropdown/model/thinking controls intact.

## Reproduction Assessment

Yes. A source-level reproduction is clear: configure multiple agents, open Chat, click the plus/New session control, and current main can only send `/new` in the active session; there is no UI path that passes a selected `agentId` to `sessions.create`.

## Solution Assessment

Yes, the narrowest maintainable solution is a UI-only addition over the existing `agents.list` and `sessions.create` RPCs. A Gateway change is not indicated unless implementation discovers missing metadata or scope handling.
