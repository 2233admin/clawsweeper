## Summary

Keep this issue open. Current main supports Chrome MCP existing-session profiles through a node browser proxy, and it has better diagnostics than older releases, but I did not find source, tests, or history proving that the exact remote node `browser.proxy` consent/attach failure is fixed. The current code still has a node-proxy-specific timeout boundary that differs from the local gateway path described as working in the report.

## Best Possible Solution

Keep the issue open and fix it in the bundled browser plugin's node proxy path. The likely maintainer path is to align `browser.proxy` timeouts with Chrome MCP attach/consent, pass a longer or configurable timeout for existing-session startup and `/tabs`, make abort behavior explicit for Chrome MCP session startup, and add a regression test for `browser.proxy` with `profile=user` where the attach waits for consent or initially closes before approval.
