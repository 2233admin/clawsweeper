## Summary

Keep open. The dependency replacement is still a useful proposal, especially for sharp install pain, but this PR is stale against current main and has merge-blocking behavior regressions: current main moved the sharp-backed implementation into the media-understanding-core plugin, while the PR still rewrites the old core image runtime and leaves remaining sharp consumers unresolved.

## What This Changes

The PR removes sharp install guidance and the root sharp dependency, adds @napi-rs/image with lockfile/native-build entries, and ports image-processing code plus image-related tests from sharp to @napi-rs/image Transformer APIs.

## Best Possible Solution

Rework the migration against current main by replacing or intentionally retaining the plugin-owned media-understanding-core image backend, preserving the public plugin-sdk media contract, keeping backend override compatibility, preserving non-JPEG formats during EXIF normalization, and only removing sharp docs/build approvals after every remaining sharp runtime path is accounted for.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.89

Full review comments:

- **[P2] Preserve source format during EXIF normalization:** `src/media/image-ops.ts:299`
  - body: `normalizeExifOrientation` now returns `transformer.rotate().jpeg(90)` for every non-sips input. The previous sharp path used `rotate().toBuffer()`, which preserved PNG/WebP inputs; this change silently converts them to lossy JPEG and can drop alpha for SDK callers that only requested orientation normalization.
  - confidence: 0.93
- **[P2] Keep the legacy sharp backend override working:** `src/media/image-ops.ts:28`
  - body: Changing the Bun/macOS escape hatch from `OPENCLAW_IMAGE_BACKEND=sharp` to only `napi-rs` makes existing installs with the documented `sharp` override fall back to the sips path after upgrade. Keep `sharp` as a compatibility alias or provide an explicit migration before changing this condition.
  - confidence: 0.87
- **[P2] Keep sharp build approval while sharp remains resolved:** `package.json:964`
  - body: The PR removes `sharp` from `pnpm.onlyBuiltDependencies`, but the PR head lockfile still resolves Baileys with `sharp@0.34.5`. That can leave a remaining sharp install path with its lifecycle scripts unapproved, so either remove the remaining sharp consumer or keep the build approval until the graph is actually sharp-free.
  - confidence: 0.82
