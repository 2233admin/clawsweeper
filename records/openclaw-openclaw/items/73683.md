## Summary

Keep this issue open. Current main still has the reported race shape: WebChat `chat.send` schedules the user transcript write asynchronously and immediately starts the agent dispatch path, while the embedded runner records and submits prompts from the current session history before calling `prompt()`. The existing tests prove the user transcript update is eventually emitted, but not that it completes before the first provider request.

## Best Possible Solution

Fix this in the core WebChat/Gateway path: ensure the accepted `chat.send` user turn is durably appended, or equivalently injected into the embedded runner history, before the first provider request is submitted. Add a targeted regression test that simulates an empty new session with a delayed transcript update and asserts the model submission cannot occur with zero conversation messages. Keep provider-specific fallback handling out of the fix except for coverage using a strict OpenAI-compatible/Z.AI-shaped model case.
