## Summary

Keep this PR open: it addresses a real config-audit credential leak and the centralized snapshot direction is sound, but the proposed redaction remains incomplete for existing supported credential flags outside its hard-coded allowlist.

## What This Changes

The PR centralizes config-audit process snapshotting, redacts argv and execArgv through a new helper, updates write/observe/recovery call sites, adds focused audit tests, and records the fix in the changelog.

## Best Possible Solution

Keep the centralized config-audit process snapshot, but make secret-flag detection cover existing CLI and plugin auth flags before redaction falls back to token-shape patterns, then add tests for omitted flags and the write/observe/recovery audit paths.

## Reproduction Assessment

Yes. Current main has a source-level reproduction through any config write or observe audit with a secret in the first eight argv entries, and the PR finding reproduces by passing an arbitrary value after an omitted flag such as `--custom-api-key` or `--comfy-api-key`.

## Solution Assessment

No, not yet. Centralizing snapshot redaction is the right shape, but the hard-coded allowlist is too narrow for OpenClaw’s existing credential flag surface; a generalized secret-flag classifier or metadata-backed list is safer.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Redact all credential-style CLI flags:** `src/config/io.audit.ts:5-21`
  - body: The helper only masks the next argv element when the previous flag is in `SECRET_FLAG_NAMES`. Existing supported credential flags such as `--custom-api-key`, `--app-token`, `--comfy-api-key`, and `--alibaba-model-studio-api-key` are outside that set; if their value is an arbitrary provider key without a known token prefix, the per-element fallback sees only the standalone value and leaves it in `config-audit.jsonl`. This leaves the reported leak in place for supported credential flags.
  - confidence: 0.9
