## Summary

Keep this PR open. It addresses a real current-main credential-at-rest gap, but the latest head still has a blocking redaction bypass when one bare secret flag is followed by another bare secret flag with an arbitrary value.

## What This Changes

The PR adds config-audit argv/execArgv redaction across write, observe, and recovery audit records, expands regression coverage, and records the security fix in the changelog.

## Best Possible Solution

Keep the centralized audit snapshot redaction, but handle a secret-looking token reached while `redactNext` is set as a new flag so its following value is also redacted; add chained-secret-flag coverage before merging.

## Reproduction Assessment

Yes. Current main is source-reproducible because write, observe, and recovery audit paths persist raw argv; the PR-head blocker is source-reproducible with `['openclaw', '--token', '--api-key', 'plain-provider-key-12345']`.

## Solution Assessment

No. Redacting at the config-audit snapshot boundary is the right maintainable approach, but the current state machine is incomplete because it can swallow a later secret flag and expose that flag's value.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Keep scanning when the next token is another secret flag:** `src/config/io.audit.ts:96-99`
  - body: When `redactNext` is set, this branch masks the current token and continues before checking whether that token is itself a secret flag. With `['openclaw', '--token', '--api-key', 'plain-provider-key-12345']`, `--api-key` is swallowed as the previous value and the following arbitrary key falls through unmasked into the persistent audit record.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff reduces the original audit-log credential exposure but still leaves a concrete argv redaction bypass on the latest PR head.

Concerns:

  - body: The `redactNext` branch masks a later bare secret flag such as `--api-key` as if it were the previous value, then processes the following arbitrary value as standalone text; default fallback patterns do not guarantee that value is redacted before persistence.
