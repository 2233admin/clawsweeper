## Summary

Keep open. The PR uses the right centralization point for config-audit process snapshots, but the latest head still leaves a blocking security gap for supported key-valued credential flags, so it should not merge yet.

## What This Changes

The PR adds a config-audit process snapshot helper, redacts argv and execArgv before persistence, updates write/observe/recovery audit call sites, adds focused tests, and records the security fix in the changelog.

## Best Possible Solution

Keep the centralized config-audit snapshot boundary, but broaden secret-flag detection to cover key-valued credential names, preferably from setup/credential metadata or a shared sensitive-name classifier, and add regression tests for Nostr private keys and Matrix recovery keys across write and observe audit paths.

## Reproduction Assessment

Yes. Current main has a source-level reproduction through any config write or observe audit with a secret in the first eight argv entries; the PR finding reproduces by passing an arbitrary value after `--private-key` or `--recovery-key`, which the latest classifier still treats as non-secret.

## Solution Assessment

No, not yet. Centralizing audit process snapshot redaction is the narrow maintainable shape, but the proposed sensitive-flag classifier is still incomplete for existing OpenClaw credential-bearing CLI surfaces.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Cover key-valued secret flags:** `src/config/io.audit.ts:47-48`
  - body: The classifier still omits supported credential flags such as Nostr `--private-key` and Matrix `--recovery-key`. Those values can be arbitrary `nsec`/hex/recovery strings that do not match the fallback token-shape patterns, so config writes or observe records can still persist plaintext credentials to `config-audit.jsonl`. Cover these key-valued flags, or source sensitivity from credential/setup metadata, and add regression tests for them.
