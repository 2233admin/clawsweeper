## Summary

Keep open. The PR targets a real current-main credential-at-rest leak and uses the right central audit-snapshot boundary, but the latest head can still leave a later secret flag's arbitrary value unredacted when a preceding bare secret flag consumes that flag token as `***`.

## What This Changes

The PR adds config-audit argv/execArgv redaction helpers, routes write/observe/recovery audit records through the shared snapshot helper, adds regression tests, and records the security fix in the changelog.

## Best Possible Solution

Keep the centralized config-audit process snapshot redaction, but adjust the `redactNext` state so a secret-looking `--flag` is not swallowed in a way that exposes its following value. Add a regression test for a missing bare secret flag followed by another secret flag and then run the targeted config audit/recovery tests plus exact-head CI before merge.

## Reproduction Assessment

Yes. Current main is source-reproducible because write, observe, and recovery audit records persist raw first-eight argv arrays; the PR-head blocker is reproducible with the helper input `['openclaw','--token','--api-key','plain-provider-key-12345']`, which leaves the final value plaintext.

## Solution Assessment

No. Redacting at the audit snapshot boundary is the right maintainable direction, but the current fail-closed state machine is not complete because it can consume a later secret flag as a value and fail to redact that flag's value.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.86

Full review comments:

- **[P1] Keep scanning after masked secret-looking flags:** `src/config/io.audit.ts:96-99`
  - body: When `redactNext` is set, this branch masks the next token and immediately `continue`s, even if that token is itself a secret flag. For example, `['openclaw', '--token', '--api-key', 'plain-provider-key-12345']` becomes `['openclaw', '--token', '***', 'plain-provider-key-12345']`, so an arbitrary API key that does not match the fallback token-shape patterns can still land in `config-audit.jsonl`. Please handle secret-looking `--flag` tokens in this state as flags as well, or otherwise mask their following value, and add a regression test for the chained-secret-flag case.
  - confidence: 0.86
