## Summary

Keep this PR open. Current main still does not implement the PR’s intended HTTP-level timeout threading for remote embedding batch upload/create/status/error-read/download calls. The configured batch timeout reaches the provider batch runners, and `fetchWithSsrFGuard` already supports `timeoutMs`, but the active memory-host SDK wrappers and provider batch helpers still cannot accept or forward that value. The PR remains useful context, but it is stale against the current file layout and needs a fresh port rather than cleanup closure.

## Best Possible Solution

Keep this PR open as useful but stale work. The best maintainer path is a fresh port that adds optional `timeoutMs` or deadline-budget support to `packages/memory-host-sdk/src/host/{remote-http,post-json,batch-http,batch-upload}.ts`, threads the remaining budget through `extensions/{openai,voyage,google}/embedding-batch.ts` upload/create/status/error-read/download calls, budgets retries, handles aborts without resubmitting already accepted batches, drops unrelated churn, and adds focused regression tests for SDK helper forwarding and provider batch wait/download paths.

## Risks / Open Questions

- A direct merge is likely stale because the PR edits removed pre-move `src/memory/*` files and includes unrelated formatting/test-support churn outside the memory batch HTTP path.
- A correct port needs one remaining deadline budget across upload, batch create, status polling, terminal error reads, result downloads, and retry attempts; passing the original timeout to each phase can still exceed the configured full-operation timeout.
- Timeout retry behavior must avoid replaying the entire upload/create pipeline after a provider has already accepted a remote batch, or it can duplicate work and leave orphaned provider jobs/files.
- The port must preserve `fetchWithSsrFGuard` SSRF and proxy behavior; swapping to raw fetch or bypassing the shared helper would be a security regression.
