## Summary

Keep this PR open. Current main still does not implement the PR's intended HTTP-level timeout threading for remote embedding batch upload, create, status-poll, error-read, and result-download calls. The branch is stale against the current memory-host SDK plus provider-plugin layout, but the underlying bug remains actionable.

## Best Possible Solution

Keep this PR open as useful but stale work. The best maintainer path is a fresh port that adds optional timeoutMs or shared deadline-budget support to packages/memory-host-sdk/src/host/{remote-http,post-json,batch-http,batch-upload}.ts, threads the remaining budget through extensions/{openai,voyage,google}/embedding-batch.ts upload/create/status/error-read/download calls, avoids resubmitting already accepted provider batches on abort, drops unrelated churn, and adds focused regression tests for SDK forwarding plus provider batch wait/download paths.

## Risks / Open Questions

- Merging this branch as-is is risky because the active implementation moved to packages/memory-host-sdk/src/host and extensions/{openai,voyage,google}/embedding-batch.ts, while the PR diff targets older src/memory paths and includes unrelated formatting/test-support churn.
- A direct timeout retry that reruns the whole batch operation can resubmit provider batches after upload/create already succeeded, causing duplicate remote jobs, cost, or orphaned files.
- The correct fix needs one remaining deadline budget through upload, create, polling, terminal error reads, result downloads, and retry attempts; passing the original timeout to each phase can still exceed the documented full-operation timeout.
