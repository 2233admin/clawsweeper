## Summary

Keep open. Current main does not implement the PR’s intended remote embedding batch HTTP timeout threading. The batch timeout reaches the provider batch runner, but the current memory-host SDK HTTP helpers still do not accept or forward timeoutMs to fetchWithSsrFGuard, and the OpenAI/Voyage/Gemini batch helpers only enforce timeout in polling loops rather than on upload, create, status, error-read, and result-download HTTP calls. The PR also targets old src/memory paths, so the work needs to be ported and re-reviewed rather than closed as shipped.

## Best Possible Solution

Keep the PR open as useful but stale work. The best maintainer path is a fresh port that adds optional timeoutMs to packages/memory-host-sdk/src/host/{remote-http,post-json,batch-http,batch-upload}.ts, threads a single remaining deadline through extensions/{openai,voyage,google}/embedding-batch.ts upload/create/status/error-read/download calls, budgets retry attempts, handles AbortError robustly, and adds focused regression tests in the memory-host SDK plus provider batch wait helpers.
