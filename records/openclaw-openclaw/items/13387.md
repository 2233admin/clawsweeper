## Summary

Current main still does not implement consecutive NO_REPLY/silent-reply pruning in replay context. The embedded runner pipeline sanitizes, validates, heartbeat-filters, history-limits, and repairs tool-use/result pairing, but never collapses consecutive silent assistant turns; the linked implementation PR #13388 is closed and unmerged, so this remains a valid narrow follow-up.

## What This Changes

The issue requests built-in pruning of consecutive assistant NO_REPLY/silent replies from session transcripts so replay context keeps at most one consecutive silent reply.

## Best Possible Solution

Keep this issue open and implement a focused in-memory replay-context prune in the embedded runner history pipeline. Collapse only consecutive assistant silent replies, keep at most one by default, reuse shared silent-token detection, handle legacy string/text-block/JSON-envelope and clean empty-content silent-stop shapes, preserve tool-use/tool-result pairing, update transcript-hygiene docs/changelog, and treat #13388 as prior art rather than merged work.

## Risks / Open Questions

- A fix must preserve isolated intentional silent replies; pruning every NO_REPLY turn would change valid transcript semantics.
- A fix must not remove assistant turns that contain tool_use blocks or otherwise break tool-use/tool-result pairing after history limiting.
- Silent replies appear in multiple shapes: exact string/text block NO_REPLY, JSON action envelopes, and clean empty-content stop turns; matching only one shape would leave the reported replay pollution in place.
- The context-engine assembly runs after the embedded runner pipeline assignment, so the pruned history needs to be the history passed into both normal replay and context-engine prompt assembly.
