## Summary

Current main does not implement the requested `openai-codex` voice transcription path. The existing speech-to-text path is provider/CLI based, with OpenAI STT routed through the normal OpenAI API transcription endpoint, while the Codex media-understanding provider is image-only. Because the request also depends on ChatGPT/Codex quota, billing, and supported-auth semantics, it should stay open for maintainer/product review rather than be swept closed.

## What This Changes

The issue requests inbound voice/audio transcription through the `openai-codex` ChatGPT/Codex-authenticated path, without requiring `OPENAI_API_KEY`, API billing, or normal text-message quota availability.

## Best Possible Solution

Keep the issue open until maintainers decide whether OpenClaw should support Codex-authenticated speech-to-text. If approved, the change should live in the OpenAI plugin/provider-owned media-understanding path, add a real `openai-codex` audio registration and tests, avoid silent fallback to `OPENAI_API_KEY`, and document the exact auth, quota, and billing behavior; if not approved, docs should explicitly call out that current inbound STT uses API-key providers or local CLI transcription.
