## Summary

Keep open. Current main still has a reasoning-enabled GitHub Copilot Opus 4.7 path that emits Anthropic adaptive `output_config.effort` from model ID alone, with no provider/model compatibility rule; current Copilot defaults reduce the common bundled-catalog blast radius by setting Claude reasoning false, but they do not fix the reported payload shape when reasoning is enabled.

## What This Changes

The issue asks OpenClaw to stop sending unsupported Anthropic `output_config.effort` values to `github-copilot/claude-opus-4.7`, where the Copilot proxy rejects non-medium efforts and can put auth profiles into cooldown.

## Best Possible Solution

Add a provider/model-specific adaptive-effort compatibility rule for `github-copilot/claude-opus-4.7`, ideally as a small capability table consulted by the Anthropic request builder or Copilot wrapper, then cover low/minimal/high/xhigh against Copilot while preserving direct Anthropic Opus 4.7 behavior.

## Reproduction Assessment

Yes. The source-level reproduction is a reasoning-enabled `github-copilot/claude-opus-4.7` request with `/think high` or `xhigh`; current main maps that to `output_config.effort` and emits it without a Copilot provider gate, and the issue provides live 400 logs for that payload.

## Solution Assessment

Unclear. A narrow provider gate would likely unblock the reported high/xhigh failure, but the best maintainable fix should reconcile this issue with #70322/#70584 through one provider/model effort capability rule rather than a model-only low/minimal clamp.
