## Summary

Keep open. Current main still arms a claude-cli no-output watchdog, the default resumed CLI watchdog cap is still 180 seconds, and the live/non-live CLI paths still reset liveness from subprocess stdout/stderr rather than a Monitor-aware expected-idle state. The related cap-raising issue and PR remain open and would not by themselves prove the Monitor-specific idle case is fixed.

## Best Possible Solution

Keep this issue open and implement a narrow Monitor-aware expected-idle path for claude-cli. The best fix is to suspend or refresh the no-output watchdog only while a real pending Monitor/OpenClaw MCP long-running tool is active, preserve the watchdog for genuinely stalled silent subprocesses, and add regression coverage for both live claude-stdio and non-live CLI behavior as applicable. Coordinate with #40982/#41022 so a broader cap change does not accidentally close the Monitor-specific bug without coverage.
