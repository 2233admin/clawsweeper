## Summary

Keep open. Current main still has a canvas-only PDF page-image rendering path that returns empty images when the optional canvas import fails, and the open closing PR #75370 is the active implementation candidate. Because the issue is fresh, source-reproducible, and paired with an open PR, it should stay open until that PR or an equivalent fix lands.

## What This Changes

The issue asks OpenClaw to render PDF page images through a Poppler/pdftoppm fallback when the optional @napi-rs/canvas package is unavailable, instead of returning text-only extraction.

## Best Possible Solution

Keep this issue open and review #75370 or an equivalent narrow change in the bundled document-extract plugin. The final implementation should either add a bounded, documented Poppler fallback or surface a clear degradation signal, with focused tests for missing canvas and docs for the optional dependency contract.

## Reproduction Assessment

Yes. A high-confidence static reproduction exists: a low-text PDF reaches the image-rendering branch, loadCanvasModule() throws when @napi-rs/canvas is unavailable, and the extractor returns `{ text, images: [] }`; the agent PDF tool does not pass the warning callback.

## Solution Assessment

Unclear. A pdftoppm fallback is a plausible owner-module fix, but it changes the runtime dependency and external-process security contract, so the best immediate path is maintainer review of the open PR rather than closing the issue early.
