## Summary

Keep open. The issue discussion identifies a real setup/runtime environment mismatch, and current main only partially mitigates it: Gateway RPC config writes now reject unresolved active SecretRefs and daemon install can copy env-backed config refs into the generated service environment, but interactive/provider setup and normal CLI config writes can still persist an env SecretRef based on the shell environment while startup remains fail-fast for enabled model provider API keys.

## What This Changes

The issue reports that model-assisted Moonshot/Kimi setup can persist an env-backed provider API key that works in an interactive shell but is unavailable to the systemd gateway daemon, causing gateway startup failure.

## Best Possible Solution

Add a daemon-aware boundary for post-install provider SecretRef changes: route assistant/TUI/config mutations through daemon-side validation or an OpenClaw-managed daemon-readable secret store, and block or roll back activation when an enabled secondary provider SecretRef cannot resolve in the actual gateway runtime environment.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is to persist models.providers.moonshot.apiKey as an env SecretRef with MOONSHOT_API_KEY absent from the daemon environment; current collection marks that enabled provider key active and startup activation throws the required-secrets failure.

## Solution Assessment

Unclear. Current main's RPC preflight and install-time env propagation are useful partial fixes, but they do not fully cover post-install shell/model-assisted writes or settle whether enabled secondary providers should remain startup-fatal.
