## Summary

Keep this issue open. Current main has added some SecretRef guardrails for gateway RPC config writes, but the reported post-install setup/runtime mismatch is still real: interactive setup can validate against the current shell, direct CLI config writes can persist an env-backed provider key without daemon-environment resolution, and gateway startup still treats unresolved active provider SecretRefs as fatal.

## Best Possible Solution

Keep this issue open for maintainer design on the daemon-aware SecretRef boundary. The likely fix should preflight active provider SecretRefs against the actual gateway service environment before saving or activating post-install config changes, or offer a safe storage path such as the OpenClaw-managed daemon-readable secret environment. If a secondary provider cannot resolve its secret, the product should block or roll back the change rather than leaving the gateway in a startup crash loop.
