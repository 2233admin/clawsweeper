## Summary

Keep open. Current main still does not implement the PR's central `resolveGroupSessionKey()` fix for agent-prefixed Slack thread keys: the parser still checks only the first `From` segment, so `agent:builder:slack:channel:...` is not decoded as Slack. The PR also has a still-relevant functional review concern around unconditionally skipping `agent:<id>:` for main-session keys whose main key matches a channel surface name. Security review did not find supply-chain or code-execution surface changes; the PR is limited to session routing code and tests.

## Best Possible Solution

Keep this PR open as an implementation candidate, but rework the parser to require an explicit group/channel kind after `agent:<id>:<provider>` or reuse the existing `parseRawSessionConversationRef()`/session-conversation parser. Add focused regression coverage for `agent:<id>:slack:channel:<channelId>:thread:<threadTs>` under webchat/provider-fallback contexts and for non-group `agent:<id>:<mainKey>` aliases such as `agent:main:webchat`. Then re-check the restart-sentinel fallback tests so parsed thread targets stay ahead of stale base-session routes while partial account hints are preserved.
