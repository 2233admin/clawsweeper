## Summary

Keep open. Current main still has startup timing traces, generic readiness logging, Gmail watcher lifecycle warnings for some failures, and a plugin-init failure summary, but it does not collect or emit the requested structured Gateway boot outcome summary. The concrete Gmail watcher gap remains because normal startup skips the lifecycle wrapper unless `hooks.gmail.account` is already configured, and the wrapper still suppresses the missing-account reason when called.

## What This Changes

Request to add Gateway startup observability that reports which subsystems and hooks initialized, skipped, or failed, especially Gmail watcher non-start reasons.

## Best Possible Solution

Add one concise, structured, redacted Gateway boot outcome collector after sidecars settle, covering observable startup outcomes for Gmail watcher, internal hooks, channels, plugin services, memory scheduling, recovery sidecars, and gateway_start hooks while preserving current readiness and failure-swallowing behavior.
