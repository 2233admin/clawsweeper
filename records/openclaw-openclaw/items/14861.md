## Summary

Keep open. Current main still has targeted startup logs and startup trace timing, but it does not implement the requested aggregate Gateway boot summary with per-subsystem started/skipped/failed outcomes. The Gmail missing-account path remains silent during normal startup because the gateway only calls the Gmail lifecycle wrapper when an account is configured, and the wrapper still suppresses that reason if called directly.

## Best Possible Solution

Implement a structured Gateway startup outcome collector around `startGatewaySidecars` and post-attach startup. It should record safe started/skipped/failed outcomes for Gmail watcher, internal hooks, channels, plugin services, memory, recovery/restart sidecars, and related startup tasks, then emit one concise normal boot summary after sidecars settle. Add regression coverage for `hooks.enabled=true` with missing `hooks.gmail.account` and for representative aggregate summary output.
