## Summary

Current main still does not implement the requested Gateway boot outcome summary. It has startup timing traces, readiness gating, and targeted sidecar logs, but no structured started/skipped/failed collector, and the Gmail missing-account startup case remains silent during normal Gateway startup.

## What This Changes

Request a Gateway startup boot summary that reports subsystem and hook startup outcomes, especially skipped or failed Gmail watcher initialization reasons.

## Best Possible Solution

Implement a structured, redacted Gateway startup outcome collector around `startGatewaySidecars` and post-attach startup. It should record safe started/skipped/failed outcomes for Gmail watcher, internal hooks, channels, plugin services, memory scheduling, restart/session recovery sidecars, and `gateway_start` hooks, then emit one concise normal boot summary after sidecars settle with regression coverage for `hooks.enabled=true` and missing `hooks.gmail.account`.
