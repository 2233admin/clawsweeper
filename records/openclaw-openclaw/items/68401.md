## Summary

Keep open. Current main has a real fix for the narrower Discord listener-timeout symptom from #73204, but it still does not implement the broader request to let normal same-session channel messages process independently while compaction is running; the core run lane, queued compaction lane, active-run steering, followup queue, and direct compaction lock path still preserve serialization.

## What This Changes

The issue requests that incoming channel messages continue to be accepted or processed with minimal delay while session compaction is running.

## Best Possible Solution

Treat this as a core session-runtime design change: make compaction nonblocking through copy-on-write/background compaction or add a bounded busy-session acknowledgment/offload path, then cover Telegram- and Discord-style inbound messages arriving during compaction without corrupting transcript order, followup delivery, or truncate-after-compaction rotation.
