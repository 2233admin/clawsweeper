## Summary

Keep open. Current main has useful timeout, fallback, cleanup, and stuck-session guardrails, but it still does not prove the reported same-turn near-identical retry churn is fixed: successful timeout compaction can retry the same provider/model, configured fallback paths bypass the newer one-shot idle retry cap, and stuck-session recovery still observes active embedded runs unless active abort is explicitly enabled.

## What This Changes

The issue asks OpenClaw to stop a single assistant turn from repeatedly reissuing near-identical provider requests under the same runId while the session remains stuck in processing.

## Best Possible Solution

Add a focused runtime guard and regression test that tracks same-run timeout retries by candidate, profile, and prompt-size/material-compaction delta, then advances to the next fallback candidate or returns a clear timeout error once the retry budget is exhausted.

## Reproduction Assessment

Yes for the core code path: the existing timeout-compaction harness can simulate repeated timed-out attempts with high prompt usage under one run, and current tests already assert multiple same-run attempts before timeout handling. A live provider/credit reproduction is not necessary to confirm the policy gap.

## Solution Assessment

No, current main is not yet the best solution because it bounds some cleanup and retry paths but does not distinguish productive compaction/profile rotation from unproductive near-identical same-candidate churn. The safer fix is a narrow retry-budget guard in the embedded runner/failover path with regression coverage.
