## Summary

Keep open. Current main has substantial failover and stuck-session guardrails, including a newer one-shot idle-timeout retry cap, but the reported same-turn same-candidate timeout churn is not proven fixed. Current behavior still intentionally permits bounded same provider/model retries through timeout-triggered compaction and auth-profile rotation before model fallback, and I found no regression that asserts near-identical same-candidate requests are suppressed.

## Best Possible Solution

Keep this issue open and decide the intended retry boundary for timeout-driven same-candidate churn. The best implementation path is a focused regression that simulates a single run with configured fallbacks, repeated no-chunk/timeouts, and near-unchanged prompt usage, then ensures OpenClaw either advances after a bounded same-provider/profile attempt budget or surfaces a clear timeout error while preserving useful timeout compaction when it materially reduces context.
