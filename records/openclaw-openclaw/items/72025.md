## Summary

Keep open. Current main still lacks Signal inbound status-reaction lifecycle wiring, and the PR is a focused useful fix, but the latest head still leaves terminal Signal status reactions without the cleanup semantics used by the existing status-reaction channels.

## What This Changes

The PR adds Signal inbound status-reaction wiring in `event-handler.ts` and a new Signal status-reaction test covering direct, group, and no-final-reply dispatch cases.

## Best Possible Solution

Land a narrow Signal plugin fix that reuses the shared status-reaction controller, routes direct and group reactions through the existing Signal helpers, preserves mention gating and no-final restoration, implements terminal clear/restore behavior consistent with existing channels, and updates the changelog plus user-facing config docs.

## Reproduction Assessment

Yes. Current main has a high-confidence static reproduction: a Signal DM or allowed group message with `messages.statusReactions.enabled: true` reaches `dispatchInboundMessage` without status-reaction callbacks, while the Signal send/remove reaction transport already exists.

## Solution Assessment

No, not in the current PR form. The implementation direction is right, but the terminal path should mirror Telegram/Discord/Slack cleanup semantics and the user-facing docs/changelog need to move with the behavior change.
