## Summary

Keep open. Current main still has the reported Feishu topic-session mismatch: tool-only group prompting directs visible output to `message(action=send)`, while the Feishu `send` handler does not carry the topic root or set `replyInThread`; only explicit `thread-reply` gets that transport.

## What This Changes

The issue requests that visible Feishu replies from `group_topic` and `group_topic_sender` sessions post into the original Feishu/Lark topic thread instead of as plain group messages.

## Best Possible Solution

Add a Feishu-owned threading/tool-context or action-resolution path so same-conversation `send` calls from `group_topic` and `group_topic_sender` sessions carry the topic root (`om_*`) into Feishu `reply_in_thread` transport, while ordinary group sends remain top-level.

## Reproduction Assessment

Yes. A high-confidence source reproduction exists: run a Feishu `group_topic` turn with `messages.groupChat.visibleReplies` in message-tool mode, observe the prompt asks for `message(action=send)`, and trace that Feishu `send` does not set `replyInThread` or a topic reply target.

## Solution Assessment

Yes. The best direction is a narrow Feishu/plugin threading fix at the message-tool delivery boundary; changing the generic group prompt for every channel would be broader and would not guarantee transport-level topic delivery.
