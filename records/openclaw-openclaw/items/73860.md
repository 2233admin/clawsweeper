## Summary

Keep open. Current main preserves Feishu `user:ou_...` targets as `open_id`, but the reported Feishu send and `feishu_chat` failure paths still let Lark/Axios HTTP 400 errors surface as generic messages instead of preserving Feishu `code`, `msg`, `log_id`, and troubleshooter details; the remaining live `9499` delivery failure is not yet proven to be target parsing.

## What This Changes

The issue reports that Feishu direct user targets such as `user:ou_...` fail through the shared `message` tool with HTTP 400 while group chat sends work.

## Best Possible Solution

Keep the Feishu direct-message contract for `user:ou_...` targets, add a shared Feishu plugin error normalizer for Lark/Axios API failures, use it from message send and `feishu_chat`, and then use preserved vendor details to decide whether the remaining `9499` delivery failure needs a code fix or Feishu app configuration guidance.

## Reproduction Assessment

Unclear for the live DM delivery failure because this review did not run against a real Feishu tenant. Yes for the confirmed diagnostic bug: mock `client.im.message.create` or `client.contact.user.get` to reject with an Axios-shaped 400 containing `response.data`, then verify current code reports only the generic error.

## Solution Assessment

Yes for the next repair direction: plugin-local error normalization is the narrowest maintainable fix for the confirmed OpenClaw diagnostic gap. It should not change core message routing or claim to solve Feishu `code: 9499` without tenant-backed evidence.
