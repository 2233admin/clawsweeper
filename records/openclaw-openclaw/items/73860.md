## Summary

Keep open. Current main already handles the obvious `user:ou_...` target-prefix path correctly, but the Feishu send and `feishu_chat` paths still allow Lark/Axios HTTP 400 responses to collapse to generic error text instead of preserving Feishu `code`, `msg`, and `log_id`; the live 9499 delivery root cause remains unproven.

## What This Changes

The issue reports HTTP 400 failures when using the shared message tool to send Feishu direct messages to `user:ou_...` targets while group chat sends still work.

## Best Possible Solution

Keep the Feishu direct-message contract for `user:ou_...` targets, add Feishu-plugin error normalization for Lark/Axios API failures in message send and `feishu_chat`, and use the preserved Feishu code/log/troubleshooter details to determine whether the remaining 9499 delivery failure needs code changes or setup guidance.

## Reproduction Assessment

Unclear for the live DM delivery failure because this checkout has no Feishu tenant credentials or OpenClaw Feishu config. Yes for the confirmed diagnostic bug: mock `client.im.message.create` or `client.contact.user.get` to reject with an Axios-shaped 400 containing `response.data`, then observe current code reports only the generic error message.

## Solution Assessment

Yes for the next narrow repair: plugin-local Feishu error normalization is the safest maintainable step, because target parsing already maps `user:ou_...` to `open_id`. A core routing change or claim that 9499 is fixed would be premature without tenant-backed evidence.
