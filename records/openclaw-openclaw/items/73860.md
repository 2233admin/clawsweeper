## Summary

Keep open. Current main does not show the simple Feishu target-prefix bug because `user:ou_...` resolves to `open_id`, but the reported 400 path is still not resolved: the Feishu send and `feishu_chat` member lookup paths can still surface Lark/Axios HTTP errors without the vendor `code`, `msg`, `log_id`, or troubleshooter body that the reporter captured. The live `im.message.create` code 9499 may still be Feishu permission or tenant policy, so this should remain open while a narrow plugin-local diagnostic fix is attempted.

## What This Changes

The issue reports that the Feishu `message` tool returns HTTP 400 when sending `action=send` to `user:ou_...` direct-message targets while group sends work.

## Best Possible Solution

Keep the Feishu direct-message contract for `user:ou_...` targets, add a shared Feishu plugin error normalizer for Lark/Axios API failures, use it from message send and `feishu_chat`, and then use the preserved `code`/`log_id` evidence to decide whether the remaining 9499 delivery failure needs a code fix or Feishu app configuration guidance.

## Reproduction Assessment

Unclear for the live DM delivery failure because this review did not run against a real Feishu tenant. Yes for the confirmed diagnostic bug: mock `client.im.message.create` or `client.contact.user.get` to reject with an Axios-shaped 400 containing `response.data`, then verify current code reports only the generic error.

## Solution Assessment

Yes for the next repair step: plugin-local error normalization is the narrowest maintainable fix for the confirmed OpenClaw diagnostic gap. It should not change core message routing or claim to solve Feishu `code: 9499` until richer live evidence identifies the root cause.
