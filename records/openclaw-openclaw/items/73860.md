## Summary

Keep open. Current main has Feishu outbound support and correctly maps `user:ou_...` targets to `receive_id_type=open_id`, but the reported HTTP 400 path is not proven fixed, and current Feishu send/member lookup error handling can still drop the vendor `code`/`msg`/`log_id` body when the Lark SDK throws an AxiosError.

## What This Changes

The issue requests fixing Feishu direct-message sends through the shared `message` tool when `target=user:ou_<open_id>` returns HTTP 400 while group sends work.

## Best Possible Solution

Keep the current Feishu target normalization, add a shared Feishu API error adapter for thrown SDK/Axios errors, reuse it from message send and `feishu_chat` member lookup paths, and add tests that assert Feishu `code`, `msg`, and `log_id` survive the tool result/log output. If code 9499 remains after that, use the preserved log id to diagnose Feishu permissions or tenant policy with live evidence.

## Reproduction Assessment

Unclear for the exact direct-send code 9499 without a live Feishu app and tenant. There is a high-confidence local reproduction for the diagnostic loss: mock `client.im.message.create` or `contact.user.get` to throw an AxiosError with `response.data`, and current main will surface only the wrapper message.

## Solution Assessment

Yes for a narrow Feishu-plugin repair that preserves vendor error details; no evidence supports closing this as implemented or moving the behavior into core. The actual DM delivery root cause should be investigated only after OpenClaw stops hiding the Feishu response body.
