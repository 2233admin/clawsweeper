## Summary

Keep open. The report is current against v2026.4.26, and current main does not contain a post-release Feishu send-target fix. Current code already normalizes `user:ou_...` to `receive_id_type: "open_id"`, so the remaining 400 needs Feishu API error-body/scopes/tenant investigation rather than closure as already implemented or duplicate.

## Best Possible Solution

Keep open and investigate in the Feishu plugin. Capture the full Feishu/Lark response body for both `im.message.create` and `contact.user.get`, add a regression test for the exact `message` action payload `target=user:ou_...`, then either fix a target/ID-type mismatch, document required Feishu app scopes/tenant constraints, or improve SDK/Axios error unwrapping so users see the actionable vendor error.
