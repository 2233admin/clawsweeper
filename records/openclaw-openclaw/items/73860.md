## Summary

Keep open. Current main already routes `user:ou_...` Feishu targets as `open_id`, so this does not match the older receive_id_type regression; however the Feishu send/chat paths still lose SDK HTTP 400 response bodies, and the reporter supplied useful Feishu `code`/`log_id` evidence that makes a narrow diagnostic repair actionable.

## What This Changes

The issue reports that sending through the shared message tool to Feishu direct-user targets returns HTTP 400 while group chat sends work.

## Best Possible Solution

Keep this issue open and land a Feishu-plugin-local fix that preserves Feishu vendor error bodies for SDK HTTP failures, then use the recovered `code`, `msg`, and `log_id` to decide whether a separate DM-delivery fix or Feishu permissions/docs update is needed.

## Reproduction Assessment

Yes for the OpenClaw diagnostic bug: mock `client.im.message.create` or `client.contact.user.get` rejecting with an Axios-style `{response:{status:400,data:{code,msg,log_id}}}` and current code will surface only the generic message. No high-confidence live reproduction of the Feishu `9499` delivery root cause is available without the reporter's tenant/bot configuration.

## Solution Assessment

Yes, a narrow plugin-local error-body unwrapping fix is the best first step; changing core message-tool routing or guessing a different `receive_id_type` is not supported by current code because `user:ou_...` already resolves to `open_id`.
