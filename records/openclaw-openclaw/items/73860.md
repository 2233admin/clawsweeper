## Summary

Keep open. Current main still has the Feishu user-target send path and the reporter’s `user:ou_...` target maps to `open_id`, but the plugin still drops Feishu/Lark response bodies when the SDK throws on HTTP 400. The discussion includes a native API response with vendor `code: 9499` and a later v2026.4.27 report that Feishu remains broken, so this is still an active Feishu plugin bug/diagnostic gap rather than a cleanup close.

## What This Changes

Fix Feishu direct-message sends through the shared `message` tool for `user:ou_<open_id>` targets that return HTTP 400 while group sends work, and expose useful Feishu API diagnostics.

## Best Possible Solution

Expose Feishu vendor `code`/`msg`/`log_id` from thrown Lark SDK/Axios errors in the Feishu plugin’s message and chat paths first, then use those diagnostics to decide whether a second fix belongs in send payload construction, bot setup/permission docs, or a separate package-runtime issue.

Yes for the reported surface and the current diagnostic gap: the issue gives concrete `message` tool parameters, and current code can be checked by simulating a Lark SDK/Axios error with `response.data`. Live confirmation of vendor code 9499 still needs a Feishu tenant/bot setup.

## Solution Assessment

Unclear for the final DM deliverability root cause, because vendor code 9499 still needs interpretation. Yes for the next narrow fix: preserving Feishu response bodies in plugin-owned error handling is the smallest maintainable step and does not require changing core channel routing.
