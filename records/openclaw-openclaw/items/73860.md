## Summary

Keep open. Current main already routes `user:ou_...` Feishu targets as `open_id` and sends a `receive_id_type`, so this does not look like the older missing-target-type bugs. The reporter's follow-up also shows native Feishu `im.message.create` returning code `9499`, while current OpenClaw still collapses SDK-thrown HTTP 400 responses into `Request failed with status code 400`, losing the vendor `code`, `msg`, and `log_id` needed to diagnose the remaining DM failure.

## What This Changes

The issue requests a fix for Feishu direct-message sends through the shared `message` tool returning HTTP 400 for `user:ou_...` targets while group sends work.

## Best Possible Solution

Keep this issue open and land a narrow Feishu plugin diagnostics fix that preserves Feishu vendor error bodies for message sends and `feishu_chat` failures. If the unwrapped error proves the remaining 9499 is external permissions or tenant policy, follow with targeted setup docs or probe guidance rather than changing the already-supported `user:ou_...` target routing.

## Reproduction Assessment

Yes for the code-level diagnostic bug: an AxiosError-shaped object with `response.data.code`, `msg`, and `error.log_id` currently flows through `formatErrorMessage` as only the generic error message. The live DM delivery root cause remains unclear because the reporter's native Feishu `im.message.create` test also returns code `9499` outside OpenClaw.

## Solution Assessment

Yes for the next fix direction: preserve Feishu API error bodies first, then use those details to diagnose the remaining DM failure. Changing target parsing alone is not the best fix for this issue because current main already maps `user:ou_...` to `open_id`, and #53787 covers the broader undefined `receiveIdType` fallback separately.
