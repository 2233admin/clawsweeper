## Summary

Keep this PR open. Current main still overwrites the existing isolated heartbeat entry when `resolveCronSession({ forceNew: true })` rotates the same `isolatedSessionKey`, and only archives the separate stale-key cleanup path; the PR targets that remaining gap with a narrow archive call and regression test.

## What This Changes

The PR archives the previous isolated heartbeat transcript as `.reset.<timestamp>` when same-key session rotation mints a fresh session ID, and adds a regression test for that path.

## Best Possible Solution

Land this narrow heartbeat-runner fix after normal maintainer review and CI, keeping the archive behavior in the heartbeat/session owner path and relying on the existing session transcript archive helper rather than adding a new storage mechanism.

## Reproduction Assessment

Yes. Seed an isolated heartbeat session entry with an existing `sessionId` and JSONL transcript, run `runHeartbeatOnce` with `isolatedSession: true`, and current main overwrites the entry without archiving that same-key prior transcript; the PR adds this exact regression case.

## Solution Assessment

Yes. Capturing `cronSession.previousSessionId` before the store overwrite and archiving the matching previous entry with `reason: "reset"` is the narrowest maintainable fix because it reuses the existing referenced-session guard and archive helper.
