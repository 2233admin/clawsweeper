## Summary

Keep this open. Current main has mitigated the concrete silent-clobber paths with source-preserving projection and destructive-write rejection, but the root schema still rejects unknown top-level config and bundled plugin CLI command surfaces are still governed by restrictive `plugins.allow`; those remaining points are config-contract and plugin-security policy decisions.

## What This Changes

The issue requests that OpenClaw stop losing unknown or unrelated config data during config round-trips and clarify/fix how restrictive plugin allowlists affect bundled CLI command surfaces.

## Best Possible Solution

Maintainers should explicitly decide the config contract: either keep strict root validation with fail-closed write protections and better diagnostics, or preserve unknown top-level keys as opaque authored data with regression tests and docs. The bundled command allowlist behavior should be handled as a separate plugin-security policy decision.

## Reproduction Assessment

Yes. The historical Windows byte-drop sequence was not rerun in this read-only review, but current source and tests give a high-confidence focused reproduction of the remaining behavior: unknown top-level config is rejected, while the prior projection/drop and destructive-write paths are now preserved or rejected before commit.

## Solution Assessment

Unclear. The suggested root `.passthrough()` change would address unknown-key preservation, but it may not be the best maintainable contract without a typo/stale-key policy; the bundled command exemption is also a security-policy choice rather than a purely mechanical bug fix.
