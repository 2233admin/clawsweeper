## Summary

Keep this issue open. Current main has strong mitigations for the reported silent data-loss path: source/runtime projection preserves source-only keys, destructive size-drop writes are rejected before replacing the config, and plugin installs add the installed plugin to restrictive `plugins.allow`. But the issue’s remaining requested policy changes are not implemented: the root config schema is still strict, validation still rejects unknown top-level fields, and bundled plugin CLI command surfaces still honor `plugins.allow`. Those remaining decisions need maintainer product/security judgment rather than cleanup closure.

## Best Possible Solution

Keep this open for maintainer judgment. The best path is to split the remaining decisions: either keep strict root validation and add regression coverage proving plugin install fails closed without writing when unknown top-level config is present, or deliberately change the root config contract to preserve unknown top-level keys with tests, docs, and doctor/migration policy. Separately decide whether bundled plugin CLI commands should keep honoring `plugins.allow` or receive a narrow OpenClaw-owned exemption.
