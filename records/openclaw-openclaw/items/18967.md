## Summary

Current main still does not implement the requested native parent-scoped send path. Sub-agents can broadly regain `sessions_send` through config, ACP has `streamTo: "parent"`, and announce-chain delivery exists, but there is no `sessions_send_parent` tool or trusted `sessionKey: "parent"` resolver enforcing direct-parent-only, same-tree, no-chaining semantics.

## Best Possible Solution

Keep this issue open for maintainer design. The best direction is a dedicated `sessions_send_parent` tool or a `sessionKey: "parent"` resolver that derives the direct parent from trusted spawn/session metadata, enforces same-tree/direct-parent-only access, rejects sibling/ancestor/cross-agent/missing-parent targets, avoids automatic forwarding beyond the parent, and updates policy, resolver, docs, and regression tests.

## Risks / Open Questions

- A broad `sessions_send` opt-in is not equivalent to a parent-scoped API and can expand session visibility beyond the direct parent if paired with `tools.sessions.visibility="agent"` or `"all"`.
- Any implementation must avoid bypassing OC-07-style session visibility hardening, cross-agent policy, and sandbox clamps.
- ACP parent-owned child sessions already have special A2A loop-avoidance behavior; a native parent-send path should not reintroduce parent/child echo loops or ancestor auto-forwarding.
