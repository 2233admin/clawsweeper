## Summary

Keep open. Current main still performs one SearXNG category request and returns a successful zero-result payload without retrying `general`; the generic web-search fallback only moves to another provider on unavailability, structured missing-key errors, or thrown errors.

## What This Changes

The issue requests retrying a category-specific web search as a general search when the category-specific SearXNG search returns no results.

## Best Possible Solution

Add the retry in the SearXNG plugin client, not the generic provider loop: after an empty categorized response, retry once with `categories=general` when the original effective category set is non-empty and does not already include `general`, while preserving endpoint validation, timeout, count, language, result wrapping, and cache behavior.

## Reproduction Assessment

Yes. A focused source-level reproduction is `runSearxngSearch` with `categories: "weather"` and a mocked SearXNG JSON body of `{"results":[]}`; current code builds only the categorized URL and returns `count: 0`.

## Solution Assessment

Yes, a provider-local fallback is the best direction. Treating zero results as a generic core provider failure would be broader and could incorrectly retry providers or queries where zero results are valid.
