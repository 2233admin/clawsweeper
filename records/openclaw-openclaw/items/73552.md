## Summary

Keep open. Current main has provider fallback for execution failures in auto-detect mode, but it does not retry a category-filtered SearXNG search as a general search when the first response is successful but empty. The requested reliability improvement is still a concrete, narrow gap in the bundled SearXNG web_search path.

## What This Changes

Add a fallback so a category-specific web search that returns no results retries the same query with the general category before returning empty results.

## Best Possible Solution

Implement this as a provider-owned SearXNG fallback first: when a category-filtered SearXNG response is successful but has no usable results, retry the same query once with general search semantics, preserve existing caching and external-content wrapping, and cover it with focused SearXNG client/provider tests. Avoid adding hidden core fallback rules for arbitrary provider result shapes unless multiple providers need a documented SDK contract.

## Reproduction Assessment

Yes. A high-confidence unit reproduction is to mock SearXNG returning {"results": []} for a request with categories such as weather and a non-empty response for the same query with general search; current code makes only the first request and returns count: 0.

## Solution Assessment

Yes, the requested fallback is a reasonable narrow fix when implemented in the SearXNG provider path. A generic core retry based on count/results would be less maintainable because provider schemas and filtered-search semantics are provider-owned.
