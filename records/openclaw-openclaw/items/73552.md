## Summary

Keep open. Current main supports SearXNG categories and cross-provider fallback for execution errors, but the SearXNG provider returns an empty parsed result payload without retrying the same query as general search, so the requested behavior is still missing and actionable.

## What This Changes

The issue requests retrying category-specific web searches with the general category when the category search returns no results.

## Best Possible Solution

Add a provider-owned SearXNG retry path that preserves the requested category search first, then retries the same query/count/language against general search only when a non-general category request returns zero valid results. Keep cross-provider fallback semantics unchanged unless a broader provider contract is explicitly needed.

## Reproduction Assessment

Yes. A focused unit reproduction can mock SearXNG so `categories=weather` returns `{ "results": [] }` while `categories=general` would return rows; current `runSearxngSearch` makes one request and returns `count: 0` without the second request.

## Solution Assessment

Yes, if scoped to the SearXNG plugin. A provider-local fallback is narrower than disabling categories globally or changing `src/web-search/runtime.ts`, because category names and empty-result semantics are provider-specific.

In the bundled SearXNG web_search provider, preserve the requested category search first, but when a request supplies or configures a non-general `categories` value and the parsed SearXNG JSON response has zero valid results, retry the same query/count/language/baseUrl with the general category before returning an empty payload.
