## Summary

Keep open. Current main has SearXNG category support, but it does not retry a successful category search that normalizes to zero results; the generic web-search runtime fallback only switches providers on availability/errors, not on empty category responses. The issue is narrow enough for a focused SearXNG-plugin fix.

## What This Changes

The issue requests retrying category-specific web searches as general web searches when the category-specific search returns no results.

## Best Possible Solution

Add a narrow SearXNG-owned fallback: preserve category-specific routing, but after a successful non-general category request produces zero normalized results, retry once with `categories: "general"`. Keep provider selection, credential handling, SSRF validation, and cross-provider fallback behavior unchanged, and cover the behavior with SearXNG unit tests plus a short docs/changelog update if user-facing behavior changes.

## Reproduction Assessment

Yes. A unit-level reproduction is clear: call `runSearxngSearch` with a non-general `categories` value and a mocked SearXNG JSON response of `{ "results": [] }`; current main returns `count: 0` without issuing a second general-category request.

## Solution Assessment

Yes, if scoped to the SearXNG provider. The proposed category-to-general retry is narrower and more maintainable than disabling categories or changing generic cross-provider fallback semantics, while still preserving the user’s chosen self-hosted provider.
