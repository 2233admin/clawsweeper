## Summary

Keep open. Current main still accepts SearXNG category searches and generic provider fallback exists, but a successful zero-result categorized SearXNG response is returned directly without retrying `general`, so the requested reliability behavior is not implemented or superseded.

## What This Changes

Automatically retry category-specific web search requests with general search when the category-specific search returns no results.

## Best Possible Solution

Add a provider-local SearXNG retry path: when the effective category set is non-empty and does not already include `general`, retry once with `categories=general` after an empty parsed result set, while preserving the existing trusted endpoint validation, timeout, cache, count, language, and response-wrapping behavior.

## Reproduction Assessment

Yes. A high-confidence source-level reproduction is `runSearxngSearch` with `categories: "weather"` and a mocked SearXNG JSON response of `{"results":[]}`; current main builds one categorized URL and returns `count: 0` without a second general-category request.

## Solution Assessment

Yes, if implemented inside the SearXNG plugin. That is narrower than changing core web-search fallback semantics, because a zero-result successful response can be legitimate for other providers and queries.
