## Summary

Keep this issue open. Current main still passes gateway-container-visible sandbox workspace paths directly into managed Docker bind mounts, while the Docker install maps host storage into `/home/node/.openclaw/...` inside the gateway container. The docs now document the DooD host-path parity requirement, but code still does not remap managed sandbox/browser bind sources or enforce parity in Docker setup, and the direct fix PRs #31355 and #31457 are closed unmerged.

## What This Changes

The issue requests a fix for Docker-install/Docker-outside-of-Docker sandbox `workspaceAccess` bind mounts using gateway-internal paths instead of host-resolvable workspace paths.

## Best Possible Solution

Keep this issue open as the canonical Docker-install/Docker-outside-of-Docker sandbox `workspaceAccess` bind-source bug. Maintainers should choose one supported contract: either remap managed sandbox and sandbox-browser bind sources from gateway-container paths to host Docker paths before validation, hashing, and `docker create`, or make the Docker install/setup path enforce host-path parity by default and fail clearly when it is absent. Add regression coverage for `workspaceAccess` `rw`, `ro`, and `none`, plus browser sandbox creation.
