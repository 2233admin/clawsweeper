## Summary

Keep open. Current main still matches the reported regression: the OpenAI plugin creates turn metadata for `openai-codex` ChatGPT routes, the Responses builder includes supplied metadata, and the live send path re-merges that metadata immediately before `client.responses.create(...)`. The closed #74029 attempt is useful context but was builder-only and not merged into current main; the reporter's follow-up correctly points at the turn-state source or final send boundary as the needed fix.

## What This Changes

The issue requests stopping `openai-codex` ChatGPT Responses requests from carrying an unsupported top-level `metadata` field, with regression coverage for the actual SDK send boundary.

## Best Possible Solution

Add a narrow OpenAI/Codex transport fix so ChatGPT-backed `openai-codex-responses` requests never send top-level `metadata`, while preserving `x-openclaw-*` headers and preserving metadata behavior for public OpenAI and Azure Responses. The regression test should capture the final post-merge payload handed to `client.responses.create(...)`, not only the builder output.

## Reproduction Assessment

Yes. The issue has a high-confidence reproduction path from the reporter's live curl/OpenClaw logs, and current main's code path deterministically supplies, builds, and re-merges metadata for `openai-codex` ChatGPT Responses requests when a session id is present.

## Solution Assessment

Yes, the reporter's follow-up direction is the best narrow fix: omit metadata at the OpenAI provider turn-state source for `openai-codex`, or equivalently guard the final metadata merge for `openai-codex-responses`. The earlier builder-only PR is not sufficient because the send path re-adds metadata after `onPayload`.

## Repair Work Prompt

Fix the open issue where ChatGPT-backed `openai-codex-responses` requests carry an unsupported top-level `metadata` field and fail with HTTP 400 `Unsupported parameter: metadata`. Current main emits metadata from `extensions/openai/transport-policy.ts`, includes it in `buildOpenAIResponsesParams`, and re-adds it via `mergeTransportMetadata` immediately before `client.responses.create(...)`. Implement the narrowest fix so the final payload sent for provider `openai-codex` / api `openai-codex-responses` has no top-level `metadata`, while preserving `x-openclaw-session-id`, `x-openclaw-turn-id`, and `x-openclaw-turn-attempt` headers. Preserve metadata for non-Codex `openai-responses` and `azure-openai-responses`. Do not strip `store: false`, `service_tier`, prompt cache, tools, reasoning, or instructions for Codex. Prefer fixing at the OpenAI provider turn-state source (`extensions/openai/transport-policy.ts`) or the final send boundary; avoid a builder-only strip unless the post-merge send path is also protected. Add regression coverage that asserts the final post-merge params passed to `client.responses.create(...)` for an `openai-codex/gpt-5.5` or equivalent `openai-codex-responses` model do not contain `metadata`, and that regular OpenAI/Azure Responses still preserve metadata. Update the changelog with a user-facing fix entry thanking the reporter. Related refs: this issue, closed incomplete PR #74029, related field-compat reports #10140 and #61376.
