## Summary

Current main still matches the reported leak: `cron.runs` with `scope: "all"` builds `jobNameById` from an unfiltered cron job list, and the all-runs reader uses that map for search text and returned `jobName` fields. The exact fix PR and broader isolation PR are both closed unmerged, while the broader per-session cron isolation issue remains open.

## Best Possible Solution

Keep this issue open and fix the cron owner path directly. Thread authenticated caller identity into the cron gateway/service contract, filter the all-scope `jobNameById` source by caller-owned `agentId`/`sessionKey` with an explicit admin override and legacy-job compatibility decision, and add regression coverage proving an all-scope read from one session cannot receive or query another session's job name. Coordinate with #35447, but keep this narrower leak tracked until a fix merges.

- Run logs themselves are still globally readable in all-scope mode; this issue is specifically about job-name enrichment, but the broader run-log access model should be audited with #35447.
