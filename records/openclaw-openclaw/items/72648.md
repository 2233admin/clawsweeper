## Summary

Keep open. Current main does not implement the dmScope-aware exec/cron heartbeat routing requested by #71581, and this PR is still unsafe to merge because the latest head rewrites existing exec-notification and heartbeat-session contracts in ways that can lose delivery context, trust command output as System context, and route heartbeats through raw session keys.

## What This Changes

The PR modifies exec notify-on-exit routing, heartbeat session resolution, and shared DM-policy helpers to normalize system-event routing for single-owner session.dmScope=main configurations.

## Best Possible Solution

Repair this branch narrowly: add dmScope=main single-owner routing after existing canonicalization and ownership checks, and preserve the current exec completion trust, notify delivery context, background-only guard, structured summary, and exec-event heartbeat wake contract.

## Reproduction Assessment

Yes. The regressions are reproducible by static diff against current main and existing tests: the PR removes trusted:false, reads a non-existent deliveryContext field, uses exec-exit/coalesMs, drops canonical heartbeat checks, and changes behavior covered by current exec notify-on-exit tests.

## Solution Assessment

No. This PR is not the best way to solve the issue because it rewrites core session routing and exec notification behavior; the safer path is a small dmScope-aware routing seam that keeps the current delivery, trust, and heartbeat contracts intact.

  - body: The notify-on-exit enqueue no longer passes `trusted: false`. Since `enqueueSystemEvent` defaults missing trust to true, command output can enter the prompt as trusted System context instead of the current untrusted exec-completion event.

  - body: This wake uses `reason: "exec-exit"` and `coalesMs`. The heartbeat classifier only treats `exec-event` as an exec wake, and the option name is `coalesceMs`, so completions lose the intended exec-specific immediate wake behavior.
  - confidence: 0.94
- **[P1] Read notifyDeliveryContext from the session:** `src/agents/bash-tools.exec-runtime.ts:360-361`
  - body: `ProcessSession` stores the preserved origin route in `notifyDeliveryContext`, but this code reads `deliveryContext` through a loose cast. Delayed background completions will lose their pinned channel/topic context and can fall back to mutable session routing.
  - confidence: 0.93
- **[P1] Use dmScope, not session storage scope:** `src/agents/bash-tools.exec-runtime.ts:380-382`
  - body: The normalizer is passed `cfg.session?.scope` instead of `cfg.session?.dmScope`. That mixes storage scope with DM routing scope, so configured `dmScope=main` can be ignored on the path this PR is meant to fix.
  - confidence: 0.91
- **[P1] Preserve canonical heartbeat session checks:** `src/infra/heartbeat-runner.ts:321-327`
  - body: The new resolver returns raw forced/configured keys and falls back to `mainEntry`, removing alias canonicalization and agent-id ownership checks. A malformed, mixed-case, or other-agent key can select or drain the wrong heartbeat session.
  - confidence: 0.94
- **[P2] Keep notify-on-exit background-only:** `src/agents/bash-tools.exec-runtime.ts:325-331`
  - body: The current guard only sends notify-on-exit events for backgrounded sessions; this replacement guard drops that condition. Foreground execs with notify-on-exit enabled can now enqueue system events and wake heartbeats.
  - confidence: 0.89
- **[P2] Keep structured exec completion summaries:** `src/agents/bash-tools.exec-runtime.ts:364-366`
  - body: The new completion text drops the exec id and output tail. Existing behavior and tests rely on those fields so the heartbeat has the command result context it is supposed to process.
  - confidence: 0.9
