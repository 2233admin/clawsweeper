## Summary

Keep open. The linked feature is valid, but the latest PR diff still replaces stable exec notification and heartbeat routing contracts with changes that would trust exec output, lose delivery context, misclassify completion wakes, and bypass canonical session ownership checks.

## What This Changes

The PR adds a shared event-routing key normalizer and rewrites exec notify-on-exit plus heartbeat session resolution to route single-owner DM system events toward an agent main session.

## Best Possible Solution

Keep the PR open and repair it in place with a narrow dmScope=main, single-owner routing seam that runs after existing canonicalization and ownership checks while preserving untrusted exec events, delivery context, structured completion text, and exec-event heartbeat classification.

## Reproduction Assessment

Yes. The regressions reproduce from the fetched PR diff against current main: the changed lines remove trusted:false, use exec-exit, read a non-existent deliveryContext field, and replace canonical heartbeat session selection with raw candidate keys.

## Solution Assessment

No. The current PR is not the best fix because it replaces established session and event contracts; the safer solution is to layer dmScope-aware normalization onto the current canonicalization, trust, delivery, and prompt-classification paths.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.95

Full review comments:

- **[P1] Mark exec completion events untrusted:** `src/agents/bash-tools.exec-runtime.ts:399-403`
  - body: The new notify-on-exit enqueueSystemEvent calls omit `trusted: false`. `enqueueSystemEvent` defaults missing trust to true, so command output and completion text would enter privileged System context instead of the current untrusted exec-completion path.
  - confidence: 0.95
- **[P1] Wake completions with exec-event:** `src/agents/bash-tools.exec-runtime.ts:405-409`
  - body: The branch wakes heartbeat with `reason: "exec-exit"`, but the heartbeat classifier only recognizes `exec-event` as an exec completion. These wakes can miss the exec-event prompt and delivery behavior.
  - confidence: 0.94
- **[P1] Read the stored notify delivery context:** `src/agents/bash-tools.exec-runtime.ts:359-360`
  - body: `ProcessSession` stores the origin route in `notifyDeliveryContext`, but the PR reads `deliveryContext` through a loose cast. Background exec completions would lose the pinned channel/topic and fall back to mutable session routing.
  - confidence: 0.93
- **[P1] Honor dmScope, not session scope:** `src/agents/bash-tools.exec-runtime.ts:391-394`
  - body: The normalizer is passed `cfg.session?.scope` instead of `cfg.session?.dmScope`. A config using `dmScope: "per-channel-peer"` with default storage scope can be treated as `main`, collapsing events that should stay isolated.
  - confidence: 0.9
- **[P1] Preserve canonical heartbeat session checks:** `src/infra/heartbeat-runner.ts:322-331`
  - body: The replacement returns the raw forced/configured `candidateKey` and falls back to `mainEntry`, skipping alias canonicalization and agent-id ownership checks. A mixed-case alias or another agent's key can select or drain the wrong heartbeat session.
  - confidence: 0.94
- **[P2] Keep notify-on-exit background-only:** `src/agents/bash-tools.exec-runtime.ts:325-329`
  - body: `maybeNotifyOnExit` is called on every exec settlement. Dropping the `session.backgrounded` guard means foreground execs with notify-on-exit enabled can enqueue system events and heartbeat wakes.
  - confidence: 0.88
- **[P2] Keep structured exec completion summaries:** `src/agents/bash-tools.exec-runtime.ts:363-365`
  - body: The new text drops the exec id and output tail and uses an `exit code` form that does not match the structured exec completion parser. That can hide the completion payload from exec-event prompts.
  - confidence: 0.9
