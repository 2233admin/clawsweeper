## Summary

Keep open. The linked session-routing problem is real and still tracked by the related open issue, but this PR is not mergeable as written because it regresses established exec-completion trust, delivery, heartbeat classification, and canonical session ownership checks.

## What This Changes

The PR adds a shared event-routing-key normalizer and rewrites exec notify-on-exit plus heartbeat session resolution so single-owner DM system events can route toward an agent main session.

## Best Possible Solution

Repair this PR in place by adding a narrow dmScope=main, single-owner routing seam after existing canonicalization and ownership checks while preserving the current exec completion trust, delivery context, structured summary, background-only guard, and exec-event heartbeat classification.

## Reproduction Assessment

Yes. The regressions are reproducible from the fetched PR diff against current main and the existing tests: the changed lines remove trusted:false, use exec-exit, read a non-existent deliveryContext field, and replace canonical heartbeat session selection with raw candidate keys.

## Solution Assessment

No. The current PR broadly rewrites session resolution and exec notification behavior; the safer fix is to layer dmScope-aware normalization onto the existing canonicalization, trust, delivery, and prompt-classification contracts.

  - body: `ProcessSession` stores the origin route as `notifyDeliveryContext`, but this reads `deliveryContext` through a loose cast. Background exec completions will lose the pinned channel/topic and fall back to mutable session routing.
  - confidence: 0.93

  - body: The normalizer is passed `cfg.session?.scope` instead of `cfg.session?.dmScope`. A config using `dmScope: "main"` with the default storage scope will not trigger the single-owner main-session routing this PR is meant to add.
