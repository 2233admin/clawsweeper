## Summary

Keep open. The linked dmScope heartbeat-routing problem is valid and current main does not implement it, but this PR branch is not safe to merge because it regresses exec completion notification contracts, heartbeat session canonicalization, and security-sensitive session isolation checks.

## What This Changes

The PR adds a shared event-routing key normalizer and rewrites exec system-event notification plus heartbeat session resolution to route internal events toward agent main sessions based on config-derived DM routing state.

## Best Possible Solution

Preserve the existing exec notification and heartbeat session-resolution contracts, then add a narrow dmScope-aware routing step that reuses the current single-owner allowlist precondition after canonicalization and same-agent ownership checks. Cover both exec-event and cron-event wakes without changing unrelated ProcessSession formatting, delivery context, trust flags, or subagent safeguards.

## Reproduction Assessment

Yes. The PR diff statically shows the regressed paths, and current tests already cover notify-on-exit scoping/trust/delivery context plus heartbeat forced-session and subagent fallback behavior; a focused regression should add session.dmScope=main single-owner exec-event and cron-event routing from #71581.

## Solution Assessment

No. The proposed patch is broader than necessary and replaces stable routing/notification behavior; the safer solution is to layer dmScope-aware normalization onto the existing canonicalization, ownership, queue, and exec-event contracts.

  - body: `resolveHeartbeatSession` now returns the raw forced/configured key and falls back to `mainEntry`, dropping alias canonicalization and agent ownership checks. Configured aliases, mixed-case agent keys, or another agent's key can now route or drain the wrong session.

  - body: `maybeNotifyOnExit` is called when every exec process exits, but the new guard no longer checks `session.backgrounded`. Since `tools.exec.notifyOnExit` defaults on, foreground execs with a session key can start queueing completion events and heartbeat wakes.
  - confidence: 0.91
- **[P1] Restore structured untrusted exec completions:** `src/agents/bash-tools.exec-runtime.ts:359-398`
  - body: The new completion text omits the exec id and output tail, so it no longer matches the structured exec completion parser, and the queued event no longer sets `trusted: false`. Background command output can be treated as ordinary trusted heartbeat content instead of an async exec result.
  - confidence: 0.93
- **[P2] Use the exec-event wake reason:** `src/agents/bash-tools.exec-runtime.ts:401-405`
  - body: Notify-on-exit wakes now use `exec-exit`, but heartbeat reason handling only recognizes `exec-event` as the event-driven exec path. These wakes will not build the exec completion prompt or bypass the same gates that current background completion delivery relies on.
  - confidence: 0.9
- **[P2] Read dmScope and agent routing inputs directly:** `src/agents/bash-tools.exec-runtime.ts:486-496`
  - body: The normalizer is fed `session.scope` and defaults missing `agentId` to the main agent, while the linked behavior is controlled by `session.dmScope` and the actual channel/agent allowlist for the event owner. This can collapse isolated DM routing or rewrite another agent's event to the main agent session.
  - confidence: 0.88

## Security Review

Status: needs_attention

Summary: The diff touches session-isolation routing and untrusted exec-output handling, and it weakens existing guardrails, so the security pass is not cleared.

Concerns:

- **[medium] Forced heartbeat keys bypass ownership checks:** `src/infra/heartbeat-runner.ts:271`
  - body: `resolveHeartbeatSession` accepts a raw candidate key and uses `store[candidateKey]` or `mainEntry` after dropping current canonicalization and agent-id verification. Session-key-bearing wake APIs are runtime surfaces, so this weakens isolation between agent sessions.
  - confidence: 0.87
- **[medium] Exec completion output becomes trusted:** `src/agents/bash-tools.exec-runtime.ts:395`
  - body: The new enqueue path omits `trusted: false` for background exec completions. Command output is untrusted input, and current heartbeat logic uses that flag to downgrade owner trust for system-event prompts.
  - confidence: 0.86
- **[medium] Wrong scope field can collapse isolated routing:** `src/agents/bash-tools.exec-runtime.ts:488`
  - body: The new normalizer reads `session.scope` rather than `session.dmScope` and can also resolve missing `agentId` values to the main agent. Explicit non-main DM isolation can be collapsed into a shared main session.
