## Summary

Keep this PR open, but do not merge it as-is. The linked feature request is valid, but the current branch regresses established heartbeat session resolution, exec completion formatting, wake reasons, delivery trust, and still does not cleanly implement the requested dmScope-aware exec/cron routing.

## What This Changes

The PR adds a shared event-routing key normalizer and rewrites background exec notification plus heartbeat session selection to route system events and heartbeat wakes through agent/session ownership keys.

## Best Possible Solution

Keep the existing exec notification and heartbeat session-resolution contracts, then add a narrow dmScope-aware routing seam after canonicalization and agent ownership checks. The fix should use `session.dmScope` plus the real channel/account single-owner allowlist, preserve structured untrusted exec completion events and `exec-event` wakes, and add focused regression coverage for exec-event and cron-event routing under `session.dmScope=main`.

## Reproduction Assessment

Yes. The PR head can be reproduced statically from the diff: changed lines remove the current background guard, canonicalization/ownership checks, structured exec completion text, explicit trust downgrade, and `exec-event` wake reason. Existing tests in `src/agents/bash-tools.test.ts` and `src/infra/heartbeat-runner*.test.ts` cover the contracts that would regress.

## Solution Assessment

No. This is not the best way to solve the linked issue because it replaces stable routing behavior instead of layering dmScope-aware normalization onto the existing canonicalization, ownership, trust, and isolated-session guards.

  - body: The new branch returns the raw `candidateKey` and falls back to `mainEntry`, dropping alias canonicalization and the agent-id ownership check. Forced aliases, mixed-case keys, or another agent's session key can now route or drain the wrong heartbeat session.
  - confidence: 0.94

  - body: `maybeNotifyOnExit` is still called for every process settlement, but the new guard no longer checks `session.backgrounded`. Foreground execs with notify-on-exit enabled can now enqueue system events and heartbeat wakes.
  - confidence: 0.91
- **[P1] Keep structured exec completion events:** `src/agents/bash-tools.exec-runtime.ts:363-365`
  - body: The new completion text is `Exec ... (exit code ...)` or signal-only, with no exec id or output tail. `heartbeat-events-filter` only recognizes the structured `Exec completed (<id>, code ...)` form or `exec finished`, so background completions can miss the exec-event prompt path.
  - confidence: 0.93
- **[P1] Preserve untrusted exec completion prompts:** `src/agents/bash-tools.exec-runtime.ts:371-375`
  - body: These new `enqueueSystemEvent` calls omit `trusted: false`, while current main deliberately downgrades exec completion events. That puts completion text into privileged `System:` context and matches the GHAS code-scanning finding.
  - confidence: 0.9
- **[P2] Use the real dmScope and allowlist inputs:** `src/agents/bash-tools.exec-runtime.ts:391-394`
  - body: The normalizer is fed `sessionAny.dmScope || cfg.session?.scope` plus broad defaults/channel allowlists. `ProcessSession` has no `dmScope`, and the linked behavior is controlled by `session.dmScope` with channel/account allowlists, so the requested routing will not trigger reliably.
  - confidence: 0.88
- **[P2] Guard missing agent ids before main-session rewrites:** `src/agents/bash-tools.exec-runtime.ts:498-501`
  - body: `emitExecSystemEvent` resolves the main session with `opts.agentId || ""`. If the normalizer fires for a single-owner config without an agent id, it can build an empty-agent session key such as `agent::main` and route the event nowhere.
  - confidence: 0.86
