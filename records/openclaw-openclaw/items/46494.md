## Summary

Keep open. Current main has a partial runtime-backed health fix, but the exact health/status mismatch is still actionable: `health` builds Telegram summaries from a runtime projection that can lose `tokenSource`, while `channels.status` uses the richer account snapshot path. There is also an open fork PR with closing syntax for this issue, so the issue should remain open until that work is reviewed/imported or closed.

## Best Possible Solution

Keep this issue open and review/import the open implementation candidate, or land a focused fix that makes `health` use the same safe account snapshot builder as `channels.status` (or otherwise carries safe credential metadata through the runtime projection) and prevents fresh pre-channel-start cache snapshots from hiding newer runtime state. Add regression coverage comparing Telegram `health` and `channels.status` for `running`, `lastStartAt`, `mode`, and `tokenSource`.
