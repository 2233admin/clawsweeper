## Summary

Keep this PR open. Current main still lacks the requested Control UI/Gateway behavior, and the PR is a focused implementation, but it needs revision before merge because the download helper mutates external non-image attachment URLs that may be signed or otherwise query-sensitive.

## What This Changes

The PR normalizes assistant top-level `mediaUrl`/`mediaUrls` into Control UI attachment cards, changes non-image assistant attachments to download links, and adds `Content-Disposition` handling plus Gateway/UI tests for local assistant media downloads.

## Best Possible Solution

Land a narrow version after preserving exact external attachment URLs, keeping `download=1` and `Content-Disposition` scoped to authenticated local/same-origin assistant-media responses, and passing the targeted UI/Gateway tests plus the changed gate.

## Reproduction Assessment

Yes. For the review finding, render an assistant document/audio/video attachment with a signed external URL; current main preserves the exact href, while the PR helper appends `download=1`, which can invalidate signatures or query-sensitive URLs.

## Solution Assessment

No, not as currently written. The overall direction is the right fix boundary, but the safest implementation should avoid mutating external URLs and use server-side attachment disposition only for the authenticated local assistant-media route.
