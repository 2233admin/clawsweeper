# Commit 47b3530

## Summary

Found one regression in Discord REST scheduling: idle cleanup now removes learned shared-bucket mappings immediately, so later bursts can bypass the scheduler’s per-bucket serialization.

## Findings

### Medium: Idle pruning forgets shared Discord buckets and can dispatch over the known remaining quota

- Kind: regression
- File: `extensions/discord/src/internal/rest-scheduler.ts`
- Line: 306
- Evidence: `drainQueues()` now calls `pruneIdleRouteMappings()` whenever a bucket has no active or pending work and is not currently exhausted. That deletes every `routeBuckets` entry for the learned Discord bucket, then `shouldPruneIdleBucket()` deletes the bucket state. The committed test also locks in `activeBuckets === 0` and `routeBucketMappings === 0` immediately after a response with `X-RateLimit-Bucket`.
- Impact: Discord documents `X-RateLimit-Bucket` as the identifier for shared per-route limits and `X-RateLimit-Remaining` as the number of remaining requests that can be made. After this commit, if OpenClaw has already learned that two routes share the same bucket, and the last response says one request remains before reset, going idle drops that state. A later burst to those two routes creates separate route buckets and dispatches both concurrently with `maxConcurrency > 1`, which can exceed the known remaining quota and produce avoidable 429s. Discord also counts non-shared 429s toward the invalid request limit. ([docs.discord.com](https://docs.discord.com/developers/topics/rate-limits))
- Runtime evidence: a focused `tsx` smoke warmed `GET /channels/c1/messages` and `DELETE /channels/c1/messages/m1` with the same `X-RateLimit-Bucket`, `X-RateLimit-Remaining: 1`, and `X-RateLimit-Reset-After: 60`; after warmup, `getSchedulerMetrics()` reported `activeBuckets: 0` and `routeBucketMappings: 0`, and the next two-route burst started both requests immediately.
- Suggested fix: keep learned route-to-bucket mappings and bucket remaining/reset state until reset expiry or a bounded TTL/LRU sweep, rather than pruning immediately when idle. The memory cleanup can still remove stale mappings after their reset window, or after a conservative idle TTL, while preserving shared-bucket serialization across short idle gaps.
