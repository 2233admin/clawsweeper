## Summary

Keep this issue open. Current main only has a narrow mitigation for one static reason code, the broader security-tool category mechanism is not implemented, and an open unmerged PR still references this issue with closing syntax.

## Best Possible Solution

Keep this issue open and have maintainers review PR #1132 and PR #1133, or replace them with a narrower maintained design. The likely path is a security-aware moderation context plus tests proving malicious codes are never suppressed, paired with an admin recovery path for existing moderation holds.

## Risks / Open Questions

- Current main still leaves several security-tool patterns suspicious, including shell execution, file-read plus network send, obfuscated samples, and nonstandard network findings.
- A category-based solution is security-sensitive and should preserve the malicious-code floor while treating false category declarations as evasion evidence.
- The publisher recovery path for users already placed under moderation is not implemented on main.
