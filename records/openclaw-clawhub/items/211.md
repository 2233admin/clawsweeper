## Summary

Keep open. The repository policy for this ClawHub cleanup only allows auto-closing pull requests that are certainly implemented on main, and this item is an issue. Substantively, current main still feeds test/spec/fixture-looking files into both LLM and static moderation paths, and non-forced CLI install/update flows still block or skip suspicious skills. Related PR #273 reduced one false-positive class, but PR #270, which targeted test-file exclusion for this problem, was closed unmerged.

## Best Possible Solution

Keep this issue open as the product/security tracking item. Maintainers should define how scanners treat security-skill tests and fixtures, add shared regression coverage for .test.*, .spec.*, __tests__, fixtures, mocks, and test directories across LLM and static scan paths, preserve scanning for runtime files, and improve manual review/override plus CLI install/update behavior so known false positives do not unnecessarily block security skills.

## Risks / Open Questions

- Closing this would remove the direct tracking item for security-skill false positives while current main still scans test-like source contents and CLI flows still gate suspicious skills.
- A broad ignore for test/spec/fixture paths could become a scanner bypass if runtime code is hidden under those names, so the eventual fix needs explicit semantics and regression coverage.
- The false-positive surface spans LLM evaluation, deterministic static scanning, VirusTotal aggregation, manual review/override behavior, and CLI install/update gating; fixing only one path may leave the user-visible problem unresolved.
