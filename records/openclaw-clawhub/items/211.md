## Summary

Keep open. Repository policy for this ClawHub sweep only permits auto-closing PRs that are certainly implemented on main, and this item is an issue. Substantively, current main still does not exclude test/spec/fixture files from the LLM or static moderation paths, so security-skill test strings can still influence scanner verdicts and non-forced CLI install/update flows still gate suspicious skills.

## Best Possible Solution

Keep this issue open as the product/security tracking item. Define how scanners should treat security-skill tests and fixtures, add regression coverage for .test.*, .spec.*, fixture, and test-directory handling across both LLM and static scan paths, preserve scanning for runtime files, and improve the false-positive review or non-disruptive override path so automated installs and updates are not blocked unnecessarily.

## Risks / Open Questions

- Ignoring all test-looking paths could become a bypass if malicious content is hidden under test/spec/fixture names, so the fix needs explicit scanner semantics and regression coverage.
- The false-positive surface spans LLM evaluation, deterministic static scanning, VirusTotal aggregation, manual overrides, and CLI install/update gating; fixing only one path may leave the user problem unresolved.
- Closing this issue would remove the main tracking item for security-skill false positives while related reports remain open and the targeted test-file exclusion PR was not merged.
