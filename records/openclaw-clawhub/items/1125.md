## Summary

Keep open: current main does not implement install-time OS compatibility checks, so this PR is not obsolete. The proposed patch still has blocking correctness issues around OS alias handling, version-scoped metadata, and backward-compatible response parsing before it is safe to merge.

## What This Changes

The PR adds CLI parsing of skill platform metadata, rejects `clawhub install` when the host OS is not listed in `metadata.os`, and expands `--force` help text to describe bypassing OS mismatch checks.

## Best Possible Solution

Revise the feature so the CLI normalizes host and metadata OS aliases, validates the resolved version's platform metadata, keeps response parsing tolerant of absent metadata where appropriate, and adds focused install tests for compatible, incompatible, alias, `--version`, and `--force` cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Normalize OS aliases before rejecting installs:** `packages/clawhub/src/cli/commands/skills.ts:100`
  - body: The new guard compares `metadata.os` directly to `platform()`, but the documented skill metadata values include `macos` and `windows` while Node reports `darwin` and `win32`. On macOS or Windows this rejects skills that declare the supported documented OS, so normalize both sides before checking membership.
  - confidence: 0.94
- **[P2] Validate OS metadata for the requested version:** `packages/clawhub/src/cli/commands/skills.ts:100`
  - body: The compatibility check uses the skill detail response before `versionFlag` is resolved, and that detail response exposes latest-version metadata. Installing an older version after the latest release changes its OS list can therefore reject a compatible version or allow an incompatible one; check metadata for the resolved version instead.
  - confidence: 0.88
- **[P2] Keep metadata optional in the CLI response schema:** `packages/clawhub/src/schema/schemas.ts:203-206`
  - body: Adding `metadata` as a required property makes installs fail during response parsing for registries or older API responses that omit this newly consumed key, even when no OS restriction exists. The install code already treats `metadata` as optional, so the schema should accept an omitted field as well as null.
  - confidence: 0.78
