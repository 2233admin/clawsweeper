## Summary

Keep this PR open for maintainer review. Current main does not yet implement an OS compatibility guard for `clawhub install`, but the proposed branch is not merge-ready because it compares raw Node platform values to the documented metadata vocabulary and checks latest-version metadata before requested-version resolution.

## What This Changes

The PR adds CLI parsing of skill metadata, rejects `clawhub install` when the host OS is not listed in `metadata.os`, and documents `--force` as an OS-mismatch bypass.

## Best Possible Solution

Revise the OS compatibility feature so the CLI normalizes host and metadata OS aliases, validates the resolved version's metadata, keeps response parsing tolerant of absent metadata where appropriate, and adds focused install tests for compatible, incompatible, alias, `--version`, and `--force` cases.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Normalize OS aliases before rejecting installs:** `packages/clawhub/src/cli/commands/skills.ts:100`
  - body: The new guard compares `metadata.os` directly to `platform()`, but skill metadata is documented as values like `macos` and `windows` while Node reports `darwin` and `win32`. On macOS or Windows this rejects skills that declare the documented supported OS, so normalize both sides before calling `includes()`.
  - confidence: 0.94
- **[P2] Validate OS metadata for the requested version:** `packages/clawhub/src/cli/commands/skills.ts:100`
  - body: The compatibility check runs before `versionFlag` is resolved and uses the detail response's latest-version metadata. Installing an older version after the latest release changes its OS list can therefore reject a compatible version or allow an incompatible one; fetch and validate metadata for the resolved version.
  - confidence: 0.88
- **[P2] Keep metadata optional in the CLI response schema:** `packages/clawhub/src/schema/schemas.ts:203-206`
  - body: Adding `metadata` as a required property makes installs fail during response parsing for registries or older API responses that omit this newly consumed key, even when no OS restriction exists. The install code already treats `metadata` as optional, so the schema should accept omitted/null metadata.
  - confidence: 0.78
