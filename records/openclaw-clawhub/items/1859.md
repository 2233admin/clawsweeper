## Summary

Keep this draft PR open. Current main does not yet contain the full docs/schema/prompt alignment, and the proposed patch has a correctness gap: its static-moderation regression covers top-level `metadata.envVars`, while the added docs tell skill authors to use `metadata.openclaw.envVars`, which the scanner does not currently collect from raw publish metadata.

## What This Changes

This draft PR documents optional skill environment variables via `envVars` entries with `required: false`, adds `envVars` to the CLI schema copy, updates security evaluator wording/output, and adds regression coverage for schema and moderation behavior.

## Best Possible Solution

Land an amended version that consistently supports the documented optional-env shape across the skill-format docs, CLI schema copy, static scanner, security prompt, and tests. In particular, the scanner should treat declarations under raw `metadata.openclaw.envVars` as declared env access, with regression coverage using that documented shape.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P2] Cover the documented nested envVars shape:** `convex/lib/moderationEngine.test.ts:124-125`
  - body: The new regression passes `metadata.envVars` at the top level, but the docs added in this PR tell authors to declare optional variables under `metadata.openclaw.envVars`. The publish path passes raw frontmatter metadata to the static scanner, and `collectDeclaredEnvNames` only reads top-level `envVars`/`requires.env`, so a skill following the new docs can still be flagged for undeclared env access. Please update the scanner and regression to cover `metadata: { openclaw: { envVars: [...] } }`.
  - confidence: 0.91
