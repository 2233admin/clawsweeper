## Summary

Keep open. Current main still renders unmatched local skill content as `UPDATE <registry latest> -> <semver-bumped publish version>`, while `inspect` prints the actual registry latest version. This remains a narrow CLI reporting bug and is not superseded by the related dashboard drift warning.

## What This Changes

The issue asks for `clawhub sync` to report actual registry latest versions accurately instead of presenting semver-bumped publish targets as available registry versions.

## Best Possible Solution

Keep this issue open and land a focused CLI reporting fix. `sync` should remain an upload-local-skills command, but when the resolve API returns `latestVersion` with `match: null`, the report should distinguish the actual registry latest version from the proposed publish version, with tests covering that unmatched-latest case.

## Repair Work Prompt

Fix the `clawhub sync` report for changed local skills whose registry resolve result has `latestVersion` but `match` is null. Preserve `sync` as an upload/publish command and preserve the semver bump calculation used by `resolvePublishMeta`; change the report/selection label so it does not imply the bumped publish target already exists in the registry. A reasonable outcome is to show the registry latest separately from the proposed publish version. Add focused tests for the unmatched-latest case in the sync helper and/or command output. Do not change backend resolve semantics, publish version uniqueness, telemetry payload shape, install/update behavior, or unrelated CLI commands. No changelog entry is expected unless the repository already has a required user-facing changelog path for CLI bug fixes.
