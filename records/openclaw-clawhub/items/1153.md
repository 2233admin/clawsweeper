## Summary

Keep open. This is an open ClawHub issue, and the supplied repository policy only permits auto-close for pull requests that are certainly implemented on current main. Current main still leaves the reporter's owner-facing path as soft delete: CLI/API deletion hides the skill, existing `skillVersions` rows continue to block republishing the same semver, and the destructive hard-delete path is admin/operator controlled production cleanup.

## Best Possible Solution

Keep this issue open for maintainer/operator follow-up. A maintainer should verify production ownership and target records, decide whether blob/file storage needs cleanup, then either run the existing admin hard-delete path for this skill or design an audited owner reset/version-purge workflow if these requests are expected to recur. Triage related open request #1167 at the same time for consistent policy.

## Risks / Open Questions

- The checkout cannot verify whether `JasonStarlight` controls the production owner account or whether `JasonStarlight/phosor-ai` and versions `1.0.0`, `1.0.1`, and `1.0.2` still exist in production.
- Hard delete is destructive production data mutation across versions, fingerprints, embeddings, reports, stats, installs, leaderboard references, aliases, and the skill row; it should be executed or explicitly approved by a maintainer/operator with an audit trail.
- A source search found no `ctx.storage.delete` / `storage.delete(...)` path, so maintainers should decide whether stored file blobs need separate retention or cleanup handling before considering the purge complete.
- Related request #1167 remains open and should be triaged alongside this request for a consistent policy on owner-requested hard deletes and version purges.
