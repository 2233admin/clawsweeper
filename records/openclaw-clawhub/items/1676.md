## Summary

Keep this issue open. Current main has substantial GitHub username auto-sync support, including background sync after sign-in, immutable GitHub numeric-ID lookup, derived-handle updates when the new login is claimable, and personal publisher refresh. But the provided ClawHub cleanup policy only permits automated closure for pull requests that are certainly implemented on main, and this item is an issue. Maintainer judgment is also still needed on whether auto-sync is the desired product answer versus the reporter's requested opt-in/manual rename flow and on the remaining old-handle/profile/transfer details.

## Best Possible Solution

Leave this issue open for maintainer judgment. If production has the current auto-sync behavior and maintainers accept it as the product answer, close manually as implemented or split any remaining work into narrower follow-ups for opt-in rename UX, old-handle reservation/profile redirects, API/docs updates, and transfer-target behavior. If auto-sync is not the intended policy, implement an explicit authenticated rename/accept flow that verifies the GitHub numeric account identity, updates the user and personal publisher handle together, preserves old skill URL behavior, and coordinates with the publisher/org transfer model.
