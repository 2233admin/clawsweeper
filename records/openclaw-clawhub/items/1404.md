## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy only allows auto-closing pull requests that are certainly implemented on current main. The source review also shows the two reported concerns are still actionable: the public skill header still uses compact unlabeled version/download indicators, and search can match slug tokens only after the skill reaches bounded fallback candidates or other index paths, so the reported `yescan` recall problem still needs production/data investigation.

## Best Possible Solution

Keep this issue open for maintainer triage. The practical path is to add explicit labels, tooltips, or accessible text for the skill header version and download/package indicators, then investigate the specific `yescan` recall failure against production data: skill visibility, owner state, moderation flags, search digest row, embedding map, deployment version, and fallback candidate coverage. A focused regression test should cover a short slug-token query like `yescan` matching a multi-token slug like `yescan-scan-universal` once the intended search behavior is confirmed.
