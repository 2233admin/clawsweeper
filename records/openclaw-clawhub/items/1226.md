## Summary

Keep open. The ClawHub cleanup policy does not auto-close issues, and current main still leaves the reported local package publish failure plausible: `package publish .` resolves the source under the resolved CLI workdir, which can be a configured Clawdbot/OpenClaw workspace instead of the caller's plugin repository. The reported repository does contain `openclaw.plugin.json`; after the path bug is fixed, the same package may surface the newer compatibility-metadata validation instead.

## What This Changes

The issue requests that `clawhub package publish . --family code-plugin` correctly detect an existing root `openclaw.plugin.json` when publishing a local code plugin.

## Best Possible Solution

Implement a focused CLI fix so local package publish sources are resolved against the caller's intended package directory when no explicit workdir override is supplied, while preserving explicit workdir/env behavior and GitHub source precedence. Add a regression for the configured-workspace versus plugin-cwd case and keep the newer compatibility validation as the next clear error for packages that still lack required OpenClaw metadata.

Keep the change scoped to package publish/source resolution: preserve explicit `--workdir` and `CLAWHUB_WORKDIR`/`CLAWDHUB_WORKDIR` behavior, preserve GitHub URL and owner/repo shorthand handling, and do not broaden what files are uploaded beyond the selected package folder. Add a regression test that simulates a plugin cwd containing `package.json` and `openclaw.plugin.json` while `opts.workdir` points at a different configured workspace that lacks the manifest; assert the command reads the plugin folder and reaches the current compatibility validation or publish plan rather than `openclaw.plugin.json required`. Coordinate with related #48 if the same cwd/workdir source-resolution helper can cover skill publish too, but do not attempt unrelated package namespace/subdirectory fixes from #1678 or SKILL ignore fixes from #1509 in this PR.
