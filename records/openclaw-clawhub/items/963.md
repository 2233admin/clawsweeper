## Summary

Keep open. Current main still does not implement the central digest-based GitHub backup scan, so this PR is not obsolete. The proposed branch is still not merge-ready because the digest scan trusts owner metadata in a GitHub App-backed write path and drops the existing handle-less owner fallback.

## What This Changes

This PR changes GitHub backup sync to paginate `skillSearchDigest`, lazily fetch version files after a GitHub version mismatch, remove the missing-owner stat, update backup pagination tests, and add retry handling for GitHub backup push conflicts.

## Best Possible Solution

Land a revised digest-based scan that keeps the scan phase lightweight, preserves owner-id fallback for handle-less owners, revalidates current owner eligibility before backup writes, resolves the current owner path for GitHub lookup/write operations, fetches version files only after a mismatch, and retains the push-conflict retry already present on main.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Preserve owner fallback for handle-less users:** `convex/githubBackups.ts:75`
  - body: The digest writer stores `ownerHandle: ''` for visible owners with no handle, while current main falls back to `owner._id`. This guard skips those rows entirely, so public skills from handle-less owners stop being mirrored. Resolve a fallback owner path instead of treating the empty string as ineligible.
  - confidence: 0.94
- **[P1] Revalidate owner state before trusting digest rows:** `convex/githubBackups.ts:72-75`
  - body: The previous scan read the live owner and excluded deleted or deactivated users. This replacement trusts `skillSearchDigest`, but user deletion/deactivation does not directly refresh skill digests by owner user id, so stale rows can still be backed up and then pruned repeatedly. Add live owner validation or guarantee digest refresh for those owner-state changes before enqueueing backups.
  - confidence: 0.88
- **[P2] Resolve the current owner path before backup lookup:** `convex/githubBackups.ts:84`
  - body: Using `digest.ownerHandle` directly can send lookups and backup writes to a stale owner path after a handle change for legacy or owner-user-only skills whose digest was not refreshed. Since prune compares against the live owner path, this can create backup churn. Resolve the current owner handle or fallback before calling GitHub.
  - confidence: 0.8
