## Summary

This issue should stay open. Current main still does not document, normalize, scan, expose via API, or render Claude Code skill lifecycle `hooks`; the only first-class `hooks` surface I found is for plugin/package capability summaries, not skill frontmatter. The request is valid but security-sensitive and needs maintainer product/security policy.

## Best Possible Solution

Keep this issue open for maintainer security/product review. The concrete path is to define an accepted skill hook metadata contract, document it in `docs/skill-format.md`, extend schema/parser/API/UI surfaces to expose hook declarations, add static and LLM scanner coverage for hook commands/events/target files, and decide install/inspect warnings or restrictions for high-privilege hook types before shipping.

## Risks / Open Questions

- Lifecycle hooks can run commands automatically and may block or influence tool use, so accepting and surfacing them needs an explicit security policy for event names, hook types, command targets, and install-time warnings.
- There is a compatibility decision between Claude Code’s top-level `hooks` frontmatter and ClawHub’s current normalized `metadata.openclaw`/Clawdis model.
- Raw frontmatter may be stored on versions, but current public API and UI expose selected parsed metadata; surfacing hooks needs a deliberate schema/API/UI contract rather than a docs-only change.
- Related issue #129 remains open for broader malicious-skill persistence concerns and should be considered context, not a duplicate closure target.
