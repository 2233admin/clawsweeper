## Summary

Keep this issue open. Current main has meaningful ClawHub registry mitigations for malicious skill uploads, including static detection of obfuscated terminal-install payloads, malicious verdict mapping, publish quarantine, owner moderation, and CLI/download blocking. However, this is an issue in openclaw/clawhub, and the repository-specific cleanup policy only permits auto-closing ClawHub pull requests that are certainly implemented on main. The report also includes broader OpenClaw runtime installer sandboxing, script approval, outbound network policy, and keychain-backed credential storage asks that are not proven resolved in this registry repository.

## Best Possible Solution

Keep this issue open for maintainer security triage. Preserve the current ClawHub registry mitigations, then explicitly split or link the remaining work: ClawHub static scanning/abuse-response hardening in this repository, and OpenClaw core installer sandboxing, script approval, outbound network policy, and keychain-backed credential storage in the core repository if still applicable.

## Risks / Open Questions

- Registry static analysis is pattern-based and reduces the reported ClawHub upload path, but it is not a complete replacement for runtime installer sandboxing, script approval, outbound network controls, or secret-storage hardening.
- The ClawHub CLI contains code that reads ~/.openclaw/openclaw.json for workspace/skill-root discovery; this repo does not show a keychain-backed OpenClaw credential-store migration for the issue’s plaintext-secret claim.
- The related closed #123 and current registry mitigations are useful context, but they do not prove that all affected historical content, OpenClaw runtime behavior, or credential handling has been fully remediated.
