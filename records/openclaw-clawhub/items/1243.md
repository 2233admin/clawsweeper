## Summary

Keep this open. The ClawHub cleanup policy does not auto-close issues, and current main still has a credible search-recall gap: search is capped around vector candidates, fallback is skipped once enough primary matches exist, and neither matching path treats skill tags as first-class lexical text. Related #1254 remains the broader open recall issue, while #1326 separately tracks the suspicious-label aspect for this specific skill.

## What This Changes

The issue asks ClawHub search to surface the Amber phone assistant skill for `phone` and `voice` queries based on its title, description, and explicit tags.

## Best Possible Solution

Land one bounded search-recall change, ideally consolidated with #1254, that treats normalized skill tag names and capability tags as lexical candidates while preserving existing moderation filters. Add regression coverage for `phone` and `voice` tag queries, and leave the suspicious-flag review for #1326.

## Repair Work Prompt

Implement a focused ClawHub search-recall fix for this issue and related #1254. Current main stores `tags` and `capabilityTags` in `skillSearchDigest`, but `searchSkills` and `lexicalFallbackSkills` only match display name, slug, and summary, and fallback is skipped once vector-hydrated primary matches fill the limit. Add a bounded or indexed lexical candidate path that treats normalized tag names and capability tags as first-class searchable text for skill search. Prefer using `skillSearchDigest` data; do not scan the full `skills` table or fetch full skill documents for list/search recall. Preserve existing `nonSuspiciousOnly`, `highlightedOnly`, and `capabilityTag` behavior. Do not attempt to clear or reinterpret suspicious flags; #1326 is separate moderation work. Add regression tests showing a skill with `phone` and `voice` tags is returned for those queries even when vector recall/fallback ordering would otherwise miss it. If you add schema/index fields, include the Convex generated updates and a migration/backfill plan or implementation consistent with existing cursor-based patterns. Keep the PR limited to search recall and tests; do not refactor unrelated search UI or moderation code.
