## Summary

Keep open. Current main does not implement `--dry-run` for `clawhub update`: the command registration exposes `--all`, `--version`, and `--force`, while `cmdUpdate` still proceeds to remove/extract files and write metadata/lockfile entries for eligible updates. This is a narrow CLI feature request with clear implementation and validation paths.

## What This Changes

Request adding `--dry-run` to `clawhub update` so users can preview available skill updates without changing installed files or the lockfile.

## Best Possible Solution

Add an update planning path that reuses the existing resolution, moderation, local-change, and semver checks, prints a clear preview of would-update/up-to-date/skipped entries, and returns before any directory, origin metadata, download, extraction, or lockfile write occurs. Document the flag and cover both single-skill and `--all` dry-run behavior in CLI tests.

## Repair Work Prompt

Implement `--dry-run` for `clawhub update`.

Expected behavior:
- Register `--dry-run` on the `update` command and pass it into `cmdUpdate`.
- For `clawhub update <slug> --dry-run` and `clawhub update --all --dry-run`, perform the same read-only resolution/planning checks used by normal update, including latest version lookup, fingerprint resolution, moderation handling, local-change detection, `--version`, and up-to-date checks.
- Print a clear human-readable preview with would-update and already-up-to-date entries; include skipped/blocked/local-change cases where existing behavior would skip.
- In dry-run mode, do not call `rm`, `downloadZip`, `extractZipToDir`, `writeSkillOrigin`, or `writeLockfile`, and do not mutate lockfile state in memory in a way that later writes.
- Preserve existing update behavior when `--dry-run` is not supplied.
- Do not change package publish or sync dry-run behavior.
- Add focused tests in the CLI skills command tests for single-skill and `--all` dry-run, including assertions that no filesystem/download/lockfile writes happen.
- Update `docs/cli.md`; update `docs/quickstart.md` only if the existing update example should mention preview usage.

Related context: package publish and sync already have dry-run patterns, but this request is specifically for installed skill updates.
