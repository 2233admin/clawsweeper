## Summary

Keep open. The ClawHub review policy says issues should remain open, and the current code still has a plausible path matching the report: a publisher can see a plugin in the dashboard, click through to `/plugins/$name`, and the route renders “Plugin not found” whenever the package detail API returns no readable package.

## Best Possible Solution

Keep this issue open. A maintainer should ask for the affected plugin name or URL, then reproduce from the publisher dashboard through `/plugins/$name` and `/api/v1/packages/<name>` while signed in. The likely fix belongs in the plugin detail/dashboard path: ensure owner/publisher authentication survives the detail API request, or show a clear pending/private state instead of a generic not-found page when the viewer owns the package.
