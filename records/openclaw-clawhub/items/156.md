## Summary

Keep this issue open. Current main does record download and star events, but the public counters are still backed by skill document fields that are synced on the 6-hour skill-doc-stat-sync path, so the reported user-visible metric delay remains plausible. The related latency PR was closed unmerged, and the supplied ClawHub policy says issues are not eligible for automatic close in this repository.

## Best Possible Solution

Keep this issue open for maintainer triage. Verify production cron health and event backlog, then either make public counters read a fresher source, adjust the skill document sync schedule with reactive-load safeguards, or document the expected delay and hourly download dedupe behavior if delayed counters are intentional.

## Risks / Open Questions

- Users may still perceive stars and downloads as broken because visible counters can lag behind recorded events until the 6-hour skill document sync runs.
- A successful download can miss its metric if the best-effort recordDownloadInternal call fails or no identity can be derived.
- Repeated downloads by the same API-token user or IP within one hour intentionally do not increment the metric.
