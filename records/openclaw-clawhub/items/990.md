## Summary

Keep open. This is a ClawHub issue, and the repository policy for this cleanup pass says ClawHub issues should remain open even when current main has related implementation. Substantively, main now exposes suspicious/security state through the v1 skill detail, moderation, version, and scan APIs, but the requested static HTML signal for web-fetch consumers is not present in the skill page head metadata, so there is still a concrete maintainer/product decision to make.

## Best Possible Solution

Keep this issue open for maintainer handling. The most concrete path is to add a static head-level signal on skill pages, likely by carrying `moderationInfo` through `fetchSkillPageData`/`buildSkillMeta` and emitting metadata such as `clawhub:suspicious` plus a reason summary when public moderation data exists. Maintainers should also decide whether list/search responses should include a minimal moderation field or whether docs should explicitly direct agents to `/api/v1/skills/{slug}`, `/moderation`, and `/scan` for security checks.
