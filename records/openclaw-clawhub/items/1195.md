## Summary

Keep open. This is a ClawHub issue, and the repository cleanup policy only allows auto-closing PRs that current main certainly implements. Source inspection also confirms current main still contains the reported Convex limit shape: owner-publisher package and skill digest rebuilds can call `.paginate()` repeatedly inside one internal mutation, while the scheduled internal mutations have no continuation-cursor argument.

## What This Changes

Publishing a skill fails because owner-publisher search digest synchronization can run multiple paginated Convex queries within one internal mutation.

## Best Possible Solution

Land one backend fix that processes owner-publisher package and skill digest rebuilds one indexed page per internal mutation invocation, then self-schedules with `continueCursor` until complete, with focused tests for multi-page package and skill paths.

## Repair Work Prompt

Fix the Convex owner-publisher digest sync pagination bug reported by this issue and related reports #1171, #1175, #1176, #1182, #1184, #1197, #1201, and #1205. Current `convex/functions.ts` loops over `.paginate()` inside `syncSkillSearchDigestsForOwnerPublisherId` and `syncPackageSearchDigestsForOwnerPublisherId`; Convex allows only one paginated query per query/mutation invocation. Change the owner-publisher package and skill digest rebuilds to process one indexed page per internal mutation invocation, then self-schedule the next invocation with `continueCursor` until complete. Keep the fix scoped to the digest sync path and its focused tests; do not replace pagination with unbounded `.collect()`, do not add unrelated schema/product changes, and do not broaden the publish flow. Add or update tests in `convex/functions.test.ts` to cover multi-page package and skill paths, cursor propagation, and one `.paginate()` call per invocation. Include a short changelog entry only if this repo's current release-note practice requires one for backend publish fixes.
