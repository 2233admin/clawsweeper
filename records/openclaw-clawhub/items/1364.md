## Summary

Current main still double-serializes the transfer request body: the CLI passes a JSON string into `apiRequest`, and both HTTP transport paths serialize POST bodies again. The server expects an object payload with `toUserHandle`, so the reported failure remains valid and actionable.

## What This Changes

The issue reports that `clawhub transfer request <slug> <handle>` fails because the positional handle is not sent to the API as the `toUserHandle` request field.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from the transfer, rename, and merge command handlers to `apiRequest`, updates command tests to assert object bodies, and preserves the existing API request contract.

## Repair Work Prompt

Fix the CLI request-body serialization bug reported for `clawhub transfer request <slug> <handle>`. Current command handlers pre-serialize request bodies with `JSON.stringify(...)` before passing them to `apiRequest`, while `apiRequest` and the Bun curl fallback are already the JSON serialization boundary. Update `packages/clawhub/src/cli/commands/transfer.ts` so `cmdTransferRequest` passes a plain object body with `toUserHandle` and `message`. Also update `packages/clawhub/src/cli/commands/ownership.ts` so `cmdRenameSkill` and `cmdMergeSkill` pass plain object bodies, since they share the same pattern. Keep the server API contract unchanged and do not make `parseJsonPayload` accept double-encoded strings. Update the matching CLI command tests to assert object bodies are passed to `apiRequest`, and keep HTTP helper tests aligned with `apiRequest` as the only JSON serialization layer. Do not change unrelated command behavior, auth, rate-limit handling, release metadata, or generated Convex files.
