## Summary

Current main still has the reported ClawHub CLI transfer request bug: `cmdTransferRequest` maps the positional handle to `toUserHandle`, but pre-serializes the body before passing it to an HTTP helper that serializes the body again.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from `cmdTransferRequest` to `apiRequest`, updates the transfer test to assert an object payload, and audits/fixes the adjacent ownership rename and merge commands that use the same pre-serialized body pattern. Keep the HTTP helper’s single-serialization contract intact.

## Clownfish Work Prompt

Fix the ClawHub CLI transfer request body serialization bug reported in openclaw/clawhub#1364. In `packages/clawhub/src/cli/commands/transfer.ts`, pass a plain object body to `apiRequest` for `cmdTransferRequest` instead of `JSON.stringify(...)`. Update `packages/clawhub/src/cli/commands/transfer.test.ts` so the request test asserts `body` is an object like `{ toUserHandle: 'alice', message: 'Please take over' }`, not a serialized string. Audit the adjacent ownership commands in `packages/clawhub/src/cli/commands/ownership.ts` and convert the rename/merge POST bodies to plain objects too, with matching test updates in `packages/clawhub/src/cli/commands/ownership.test.ts`. Do not change the `apiRequest` contract in `packages/clawhub/src/http.ts`; its Node and Bun/curl paths should remain responsible for exactly one JSON serialization. Keep the patch focused and avoid unrelated CLI behavior changes. No changelog entry is required unless the repo has an active unreleased CLI changelog convention nearby.
