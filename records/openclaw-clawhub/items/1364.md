## Summary

Current main still pre-serializes the transfer request body before passing it to an HTTP helper that serializes POST bodies itself, so the reported `toUserHandle required` failure remains plausible and actionable. The same pattern is present in adjacent ownership rename/merge commands, and the tests currently assert the incorrect string-body contract.

## Best Possible Solution

Keep this issue open and land a focused CLI bugfix: pass plain object bodies from `cmdTransferRequest`, `cmdRenameSkill`, and `cmdMergeSkill` to `apiRequest`; update the command tests to assert object payloads; keep `packages/clawhub/src/http.ts` as the single JSON serialization boundary; and leave the server API contract unchanged.

## Clownfish Work Prompt

Fix the ClawHub CLI JSON body serialization bug for transfer and adjacent ownership POST commands. In `packages/clawhub/src/cli/commands/transfer.ts`, make `cmdTransferRequest` pass a plain object body like `{ toUserHandle: toHandle, message: options.message }` to `apiRequest` instead of `JSON.stringify(...)`. In `packages/clawhub/src/cli/commands/ownership.ts`, make `cmdRenameSkill` and `cmdMergeSkill` pass `{ newSlug }` and `{ targetSlug }` objects instead of serialized strings. Keep `packages/clawhub/src/http.ts` as the single JSON serialization boundary for both Node fetch and Bun/curl paths, and do not change the public API routes or Convex server contract. No broad refactor or dependency changes. Add a concise changelog/release note only if the repo has an established CLI bugfix note path for this release train.
