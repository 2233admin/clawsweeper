## Summary

This issue should stay open. Current main still pre-serializes the `clawhub transfer request` POST body before passing it to `apiRequest`, and both Node and Bun HTTP paths serialize POST bodies again before sending them. The server still expects an object payload with `toUserHandle`, so the reported `toUserHandle required` failure remains plausible and actionable.

## What This Changes

Fix the CLI transfer request command so the positional handle is sent as the `toUserHandle` API field instead of being lost through double JSON serialization.

## Best Possible Solution

Land a focused CLI fix that passes plain object bodies from `cmdTransferRequest`, `cmdRenameSkill`, and `cmdMergeSkill` to `apiRequest`; update the matching command tests to assert object payloads; keep the HTTP helper as the only JSON serialization boundary and leave the API contract unchanged.

## Repair Work Prompt

Fix the CLI double-JSON serialization bug reported in this issue. In `packages/clawhub/src/cli/commands/transfer.ts`, change `cmdTransferRequest` to pass a plain object body with `toUserHandle` and optional `message` into `apiRequest`, not a pre-serialized string. In `packages/clawhub/src/cli/commands/ownership.ts`, apply the same boundary fix for rename and merge bodies (`newSlug`, `targetSlug`) because they use the same HTTP helper and server-side object contract. Update `packages/clawhub/src/cli/commands/transfer.test.ts` and `packages/clawhub/src/cli/commands/ownership.test.ts` so they assert plain object payloads instead of string containment. Do not change the server API contract, `parseJsonPayload`, rate limiting, auth, registry lookup, or the HTTP helper’s responsibility for JSON serialization. No changelog entry is expected unless this repo has an existing unreleased CLI bugfix notes file that already tracks similar fixes.
