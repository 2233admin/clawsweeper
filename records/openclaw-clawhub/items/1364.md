## Summary

Keep open. This is a ClawHub issue, and the cleanup policy only allows auto-closing ClawHub PRs that are certainly implemented on main. Current main also still appears to contain the reported double-serialization path for `clawhub transfer request`.

## Best Possible Solution

Keep this issue open. The likely fix is to pass a plain object from `cmdTransferRequest` to `apiRequest`, update the transfer command test to assert object payload semantics or final serialized wire payload, and consider auditing nearby CLI POST commands such as rename/merge for the same pattern before cutting a CLI release.

## Evidence

- **Issue discussion identifies concrete root cause:** The reporter’s follow-up comment explains that `transfer request` passes `JSON.stringify(...)` to `apiRequest`, while `apiRequest` serializes `args.body` again, producing a JSON string instead of an object for the server.
- **CLI transfer command still pre-serializes the body:** `cmdTransferRequest` passes `body: JSON.stringify({ toUserHandle: toHandle, message: options.message })` into `apiRequest`.
