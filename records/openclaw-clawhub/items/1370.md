## Summary

Keep this issue open. The repository cleanup policy for openclaw/clawhub says issues should not be auto-closed, and current main still only partially implements owner-facing plugin package lifecycle: package soft-delete storage and a DELETE API exist, but restore/undelete, hide/unhide aliases, CLI commands, dashboard controls, docs, and per-version package de-list workflows are still missing.

## Best Possible Solution

Keep this issue open and finish plugin package lifecycle support end to end: define package-level versus release-level semantics, add package restore/undelete and optional release de-list backend/API paths, expose owner-facing CLI commands and dashboard controls, include a deleted/hidden package restore view, and document the rollback workflow for plugin publishers. Coordinate per-version behavior with open #905 and validate against cleanup cases like #1744.

## Risks / Open Questions

- Owners can withdraw a package through the existing DELETE API, but without owner-facing restore/undelete they may still need maintainer or database intervention afterward.
- Per-version package de-list semantics remain unresolved around latestReleaseId, tags, install resolution, and avoiding accidental re-exposure of a bad release.
- Soft-deleted packages are filtered out of dashboard owner lists, so the existing partial delete path can remove the owner’s visible recovery surface.
