## Summary

Keep this issue open. Current main has partial plugin package lifecycle plumbing, including package and package-release soft-delete fields plus a package DELETE API, but it still lacks the requested owner-facing package hide/delete/undelete workflow across CLI, site UI, restore endpoints, version-level controls, and docs. ClawHub cleanup policy also keeps issues open; only PRs are eligible for implemented-on-main auto-close.

## Best Possible Solution

Keep this issue open and have maintainers define package-level versus release-level lifecycle semantics first. Then implement the chosen owner/admin workflow end to end across Convex package mutations, HTTP API, clawhub CLI commands, owner dashboard or package management UI, tests, and public docs, while coordinating version-specific behavior with #905 and stale plugin cleanup needs from #1744.

## Risks / Open Questions

- A broken plugin package can remain preferred by the ClawHub-first install path until maintainer intervention because owners do not have complete self-service withdrawal and restore controls.
- The existing package DELETE path soft-deletes all active releases, so restore semantics need a clear decision: restore the whole package, restore only previously active releases, or introduce release-level delisting separately.
- A partial fix on only one surface, such as API without CLI/site/docs, would still leave package owners without a reliable recovery workflow.
