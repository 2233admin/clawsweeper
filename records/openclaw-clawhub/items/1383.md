## Summary

Keep open. This is an open ClawHub issue, and the repository cleanup policy does not auto-close issues; current main also still lacks a deterministic CLI/API path for exact-slug or slug-prefix skill enumeration. Existing search has exact/prefix scoring help, but it remains bounded relevance search rather than a complete registry listing mechanism.

## What This Changes

The issue requests registry-style ClawHub CLI discovery for exact skill slugs, slug prefixes, or full hub listings instead of relying only on ranked search results.

## Best Possible Solution

Add a narrow registry discovery surface that preserves ranked search while exposing exact slug lookup and slug-prefix enumeration through indexed, cursor-paginated backend queries, public v1 API support, CLI flags or a hub-list mode with JSON output, and focused docs/tests.

## Repair Work Prompt

Implement deterministic registry discovery for ClawHub skill search/listing without changing existing ranked search semantics. Add exact skill slug lookup and slug-prefix enumeration for public hub skills, using Convex indexed and cursor-paginated access rather than .filter() table scans. A conservative design could add exact/prefix parameters to the v1 skills/search surface or a dedicated list mode, then expose CLI flags such as `clawhub search --exact <slug>`, `clawhub search --prefix <prefix>`, or an equivalent `skills list --source hub`/hub-list mode with JSON support. Preserve existing `search <query...>` output and limits for relevance search. Update CLI/API docs and focused tests. Consider adjacent prefix-recall PR #1256, but do not treat it as sufficient by itself because it targets recall rather than complete enumeration. Do not remove safety filtering, do not add unbounded scans, and do not make unrelated package/release changes.
