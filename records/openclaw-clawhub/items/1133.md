## Summary

Keep open: current main still lacks the requested moderation-hold recovery API, and the PR remains relevant, but the latest patch has unresolved security-sensitive restore semantics and audit accuracy issues that should block merge.

## What This Changes

The PR adds public and internal Convex admin mutations to clear user moderation holds, batch-restore eligible owned skills, add tests, and document the admin recovery CLI path.

## Best Possible Solution

Land a maintainer-reviewed implementation that clears user hold fields, restores only skills genuinely eligible for hold recovery, keeps malicious, manual/quality-hidden, soft-deleted, and unresolved-scan states protected or queued, records cumulative audit data, and documents the admin Convex CLI workflow.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.88

Full review comments:

- **[P1] Treat `not_found` VT results as unresolved:** `convex/skills.ts:4145`
  - body: The restore path only requeues missing, `pending`, or `loading` VT states. Current moderation maps `vtAnalysis.status === "not_found"` to `scanner.vt.pending`, so these rows should remain hidden/queued; as written they become `active/restored.moderation_lift` and drop out of `getPendingScanSkillsInternal`.
  - confidence: 0.92
- **[P1] Preserve pre-hold moderation locks:** `convex/skills.ts:4121`
  - body: `applyUserModerationToOwnedSkillsBatchInternal` overwrites every non-malicious owned skill's reason and hidden timestamp with `user.moderation`, including rows already hidden for `quality.low` or manual review. This predicate then treats those rows as lift candidates and can publish content that was quarantined before the user hold; the hold path needs to preserve or check prior state.
  - confidence: 0.82
- **[P2] Record cumulative restore counts:** `convex/users.ts:813`
  - body: The caller records only the first batch's `restoredCount`; when `scheduledSkills` is true, continuation batches can restore additional skills without updating the `user.moderation.lift` metadata or returned count. Large recoveries will therefore have an incomplete audit trail.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: No supply-chain changes were found, but the moderation restore path can expose unresolved or independently hidden content and needs security review before merge.

Concerns:

- **[high] Unresolved VT results can be published:** `convex/skills.ts:4145`
  - body: The PR treats `not_found` VT status as safe to restore even though current moderation classifies it as pending, allowing a skill to become public and leave the scan queue.
  - confidence: 0.92
- **[high] Independent moderation decisions can be cleared:** `convex/skills.ts:4121`
  - body: Because the existing hold batch overwrites hidden non-malicious skills with `user.moderation`, the new restore path can make previously manual- or quality-hidden skills active.
  - confidence: 0.82
- **[medium] Recovery audit can undercount batch restores:** `convex/users.ts:813`
  - body: Only the first batch count is written to the lift audit log, so large recoveries can omit later restored skills from the security trail.
  - confidence: 0.86
