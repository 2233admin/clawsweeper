## Summary

Keep this issue open. Current main still has the reported anonymous/authenticated split: authenticated downloads use a per-user bucket and skip the IP bucket, while anonymous downloads use a client-IP bucket that falls back to `ip:unknown` if no trusted IP header is available. The false-positive path is plausible and needs production proxy/header and rate-limit log review, not an automatic close.

## What This Changes

Anonymous OpenClaw plugin installs should not hit a false-positive ClawHub download 429 from a shared or unknown IP bucket when the same install succeeds immediately after authentication.

## Best Possible Solution

Keep this open for maintainer and ops follow-up. Inspect production `rate_limit_denied` logs for `/api/v1/packages/*/download` and `/api/v1/download`, especially `kind`, auth state, `ipSource`, `hasClientIp`, and bucket identity. If anonymous traffic is collapsing to `ip:unknown` or a shared proxy IP, fix deployed proxy/header configuration or adjust anonymous download bucketing with an abuse-protection review; track the OpenClaw install-client retry-header display separately if that client still hides `Retry-After` and rate-limit metadata.
