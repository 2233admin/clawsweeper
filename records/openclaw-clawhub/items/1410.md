## Summary

Keep this issue open. Current main still has the reported split: valid Bearer-token requests use a per-user rate-limit bucket, while anonymous downloads use an IP-derived bucket that falls back to `ip:unknown` when no trusted client IP header is available. The reported false-positive path is plausible, but the next decision depends on production proxy/header and rate-limit denial logs plus abuse-protection judgment. ClawHub cleanup policy also says open issues should be reviewed but not auto-closed by this workflow.

## Best Possible Solution

Keep this open for maintainer and ops follow-up. Inspect production `rate_limit_denied` logs for `/api/v1/packages/*/download` and `/api/v1/download`, especially `kind`, `auth`, `ipSource`, `hasClientIp`, and the bucket identity. If anonymous requests are collapsing to `ip:unknown` or a shared proxy IP, fix deployed proxy/header configuration or make an explicit abuse-reviewed change to anonymous download bucketing. Track the OpenClaw install-client retry-header display gap separately if that client still hides `Retry-After` and rate-limit metadata.
