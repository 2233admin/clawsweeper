## Summary

Keep this issue open. Current main still has the reported behavior split: valid Bearer-token requests use a per-user download bucket and bypass the shared IP bucket, while anonymous downloads use an IP bucket that falls back to `ip:unknown` when no trusted client IP header is present. The false-positive path is plausible, but the next useful step needs production proxy/header and rate-limit log review, not an automatic source-only cleanup close.

## Best Possible Solution

Keep this open for maintainer and ops follow-up. Inspect production `rate_limit_denied` logs for `/api/v1/packages/*/download` and `/api/v1/download`, especially `kind`, auth state, `ipSource`, `hasClientIp`, and bucket identity. If anonymous traffic is collapsing to `ip:unknown` or a shared proxy IP, fix the deployed proxy/header configuration or adjust anonymous download bucketing with an abuse-protection review. Track the OpenClaw install-client retry-header display gap separately if that client still hides `Retry-After` and rate-limit metadata.
