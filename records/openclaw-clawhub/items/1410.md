## Summary

Keep this issue open. The repository cleanup policy for ClawHub issues does not permit an automated close here, and current main still has the reported anonymous/authenticated split: authenticated downloads use a per-user bucket, while anonymous downloads use an IP bucket that can fall back to `ip:unknown` when no trusted client IP is available. The remaining decision needs production proxy/header and rate-limit log review, not a source-only cleanup close.

## What This Changes

Anonymous ClawHub plugin installs can return `429 Rate limit exceeded` while the same install succeeds after login, suggesting anonymous download requests may be collapsing into a shared or unknown IP bucket.

## Best Possible Solution

Keep this open until maintainers inspect production rate-limit logs and proxy/header configuration, then either correct real-client-IP attribution or adjust anonymous download bucketing with an abuse-protection review. If the install client still hides retry metadata, track that as a separate client-facing fix.
