## Summary

Keep open. This is an open ClawHub issue, and the repository-specific cleanup policy only permits auto-closing pull requests that are certainly implemented on current main. Source review also does not prove the reported hosted skill record is unstuck: current main has cron polling, queue-health logging, stale fallback, reanalysis, and basic owner-visible pending messaging, but it does not expose the per-item queue/heartbeat/retry status requested in the discussion. Current skill VT logic also still keeps undetected-only engine results pending, while the documented undetected-only clean fallback is package-specific.

## Best Possible Solution

Keep this issue open for maintainer follow-up. The best path is to verify the affected production skill/version record, repair or resync it if still pending, and decide whether to add an owner-visible scan status endpoint/UI with queue position or age, last check/worker heartbeat, retry timing, state transitions, and any maintainer-approved rescan or resync action. If maintainers intentionally want undetected-only VirusTotal engine results to clear skill scans when OpenClaw/LLM/static signals are clean, that policy should be implemented and tested in the skill scan path, not only the package path.
