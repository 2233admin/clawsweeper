## Summary

Keep open. This is a ClawHub issue, and the repository policy says ClawHub issues should remain open unless they are PRs already implemented on main. Substantively, current main still has registry/catalog trust metadata, package scans, trusted-publisher records, and staff audit logs, but no PAP-style principal hierarchy, signed runtime mandates, transport-layer authorization, or inter-agent context-boundary enforcement.

## Best Possible Solution

Keep this issue open for maintainer triage. The best path is to decide whether PAP-style trust belongs in OpenClaw runtime/security architecture, ClawHub package metadata/signing, or both, then define the smallest core authorization and audit boundary before adding any ClawHub registry or plugin integration work.

## Risks / Open Questions

- This is a security-sensitive architecture proposal; closing it without maintainer judgment could skip a real core runtime trust-boundary decision.
- Current ClawHub controls are mostly publish-time, catalog, scan, moderation, and staff-audit controls. They do not solve runtime context bleed between collaborating agent instances.
- The linked OpenClaw runtime issue may be the right canonical implementation home, but the ClawHub registry may still need follow-on metadata, signing, package-policy, or audit integration once maintainers choose the runtime boundary.
