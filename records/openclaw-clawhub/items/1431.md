## Summary

Keep open. This is an open ClawHub issue, and the supplied repository policy says to keep issues open while only auto-closing pull requests that current main definitely implements. Substantively, current main still matches the reported bug: the publish UI lets a user choose an org owner for an existing skill, but `insertVersion` rejects an existing skill whose `ownerPublisherId` differs from the selected publisher as a slug collision. The current transfer path is still user-to-user/personal-publisher based, while open #1635 is the right active implementation thread for org-capable ownership transfers.

## Best Possible Solution

Keep this issue open and route it with #1569 and open PR #1635. The best fix is a maintainer-approved publisher-change or ownership-transfer flow that supports org targets consistently across `skills.ownerPublisherId`, `skillSlugAliases`, publish/edit UI, HTTP API, CLI, docs, tests, audit logging, and any management approval requirements. If ownership moves should not happen through republish, the publish UI should stop offering that path for existing skills and direct users to the transfer flow instead.
