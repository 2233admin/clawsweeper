## Summary

Keep this open. Current main still does not provide re-registration, restore, or GitHub OAuth unbinding after user-initiated account deletion; it intentionally records a permanent deactivation and rejects the next GitHub sign-in. The ClawHub cleanup policy also says issues should stay open for maintainer handling unless they are PRs already implemented on main.

## What This Changes

The issue asks maintainers to unblock GitHub login for a deleted ClawHub account by unbinding the GitHub identity, hard-deleting the old record, or restoring the account.

## Best Possible Solution

Define the intended account lifecycle explicitly: either keep deletion irreversible with a support-only production recovery policy, or add a narrow, audited restore/unbind flow that verifies GitHub identity ownership, preserves ban protections, and covers web and CLI login behavior with tests.

## Risks / Open Questions

- Manual unbinding or hard deletion of auth records is production identity-recovery work and should verify control of the GitHub account before changing account bindings.
- A restore or re-registration path could bypass ban, malware auto-ban, or moderation protections unless it explicitly distinguishes user-initiated deletion from enforcement actions.
- Several related reports remain open, so maintainers may want one canonical product/security decision rather than ad hoc per-user fixes.
