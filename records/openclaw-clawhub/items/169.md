## Summary

Keep open. Repository policy keeps ClawHub issues open for maintainer follow-up, and current main still has a credible clean-install/update fingerprint mismatch: installed/downloaded skill contents can include generated root `_meta.json`, while the resolver compares against published version fingerprints/files that do not include that generated metadata.

## What This Changes

The issue requests that `clawhub update` stop falsely reporting `local changes (no match)` immediately after a clean install or force-update.

## Best Possible Solution

Keep this issue open and land one focused PR that defines one canonical skill-content fingerprint across install, update, sync, downloads, and resolve. The likely fix is to ignore only generated root `_meta.json` in local and downloaded-zip fingerprint calculations, or stop generated metadata from entering installed skill contents, while preserving detection for real user-authored JSON edits. Add regressions for fresh install/force-update followed immediately by update, plus a real local edit still being detected, and coordinate the fix with #178 and #937.

## Repair Work Prompt

Fix the false `clawhub update` local-change warning after a fresh install or force-update. Current main fingerprints installed local text files, but downloaded skill zips can include generated root `_meta.json` while registry resolution compares against published `version.files` fingerprints. Define a single canonical skill-content fingerprint across CLI local hashing, zip hashing, update, sync, and Convex resolve/download behavior.
