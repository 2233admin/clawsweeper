## Summary

Keep this issue open. Current main still exposes package stats but does not record package download or install increments on the package download path, and skill install counts are still driven by authenticated `clawhub sync` telemetry rather than immediate `clawhub install` activity. The related merged work only made package stats visible in API responses; it did not add the missing counter pipeline.

## Best Possible Solution

Keep the issue open. Implement a focused stats pipeline for packages, likely with a package download/install stat event or dedupe path and tests around `/api/v1/packages/{name}/download` plus the OpenClaw plugin install path. Separately, decide whether skill installs should be counted at install/add time or whether the product should keep the current documented `clawhub sync` telemetry model and clarify that behavior in API/CLI docs.

## Risks / Open Questions

- Package authors and users can see package `downloads` and `installs` fields that remain misleading because archive downloads do not touch package counters.
- Adding install-time skill reporting without a product/privacy decision could bypass the documented `clawhub sync` telemetry model and opt-out expectations.
- Package install counting may require coordination with the OpenClaw plugin install flow, since this repository currently exposes package browse/inspect/publish CLI commands but not a package install command.
