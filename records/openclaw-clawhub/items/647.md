## Summary

Keep this open. The repository cleanup policy for ClawHub issues does not allow auto-closing this open issue, and the current main branch still does not prove the reported production GitHub OAuth/session-loss symptom is fixed. Main has frontend callback handling and auth-error UI plumbing, but deleted/deactivated users are still intentionally rejected during OAuth completion, active-user lookup can still return no app user, and several related OAuth-loop/account-deletion reports remain open.

## What This Changes

The issue reports successful GitHub OAuth authorization on clawhub.ai followed by a return to the login state because no active session/user state is established or persisted.

## Best Possible Solution

Keep this open and route it into a canonical OAuth/session-loop investigation. Maintainers should correlate production Convex and frontend logs, verify AUTH_GITHUB_ID/AUTH_GITHUB_SECRET, GitHub callback URLs, CONVEX_SITE_URL, VITE_CONVEX_URL, VITE_CONVEX_SITE_URL, and cookie behavior for clawhub.ai, then reproduce both clean-browser and deleted/deactivated-account cases before deciding on purge, restore, relink, re-registration, or a clearer terminal error/support path.
