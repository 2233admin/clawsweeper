## What This Changes

The PR adds a Convex async dependency-registry scanner for bundled dependency files, schema/cache state, publish scheduling, moderation finding injection, and a small security-scan UI line rendering tweak.

## Best Possible Solution

Land this PR or a successor after regenerating Convex API types, treating unresolved registry lookups as retryable/error unless confirmed misses already make the scan suspicious, making retry state queryable without a fixed in-memory window, and completing maintainer/security review against the dependency-confusion mitigation plan.

## Review Findings

Overall correctness: patch is incorrect

Overall confidence: 0.9

Full review comments:

- **[P1] Regenerate Convex API types for the new module:** `convex/lib/skillPublish.ts:336-338`
  - body: The branch adds `convex/depRegistryScan.ts` and references `internal.depRegistryScan`, but the committed generated API types at the PR head still do not include a `depRegistryScan` module entry. With the generated types committed this way, TypeScript will reject these references during Convex/deploy typecheck. Run Convex codegen and commit the updated `_generated/api.d.ts` entry with the new module.
  - confidence: 0.94
- **[P1] Mark partial registry failures as retryable:** `convex/depRegistryScan.ts:508-518`
  - body: When a registry lookup exhausts retries, this code drops the dependency from `results` and only sets `abortedDueToErrors` after three consecutive failures. If one or two failures are interleaved with successful lookups, the scan can still write `status = "clean"` even though some dependencies were never checked, preventing a later retry. Track unresolved lookups and return `error` unless confirmed missing packages already make the scan suspicious.
  - confidence: 0.91
- **[P2] Index retryable dependency scans:** `convex/depRegistryScan.ts:640-643`
  - body: This retry query only inspects the newest 500 `skillVersions` and filters `depRegistryAnalysis.status` in memory. Older error-state versions outside that window will never be scheduled for retry, so failed scans can remain stuck indefinitely. Store a denormalized status field and query it with an index, or paginate in a way that cannot strand older errors.
  - confidence: 0.86

## Security Review

Status: needs_attention

Summary: The diff is a security-sensitive scanner and currently has concrete false-clean and stale-retry gaps that can leave dependency-confusion exposure unreported.

Concerns:

- **[high] Partial lookup failures can be marked clean:** `convex/depRegistryScan.ts:508`
  - body: Retry-exhausted registry lookups are omitted from the result set unless three failures are consecutive, so a scan can report clean even though some dependencies were never verified against their registries.
  - confidence: 0.91
- **[medium] Error-state rescans can strand older failures:** `convex/depRegistryScan.ts:640`
  - body: The retry action only scans the newest 500 versions and filters a nested status in memory, which can leave older failed dependency-registry scans un-retried indefinitely.
  - confidence: 0.86
