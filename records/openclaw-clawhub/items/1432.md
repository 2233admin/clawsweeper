## Summary

Keep open. The repository policy for this ClawHub sweep does not allow auto-closing open issues, and the substantive bug still appears valid: current main normalizes `metadata.openclaw` for stored/LLM metadata, but static publish moderation still collects declared env vars only from direct `requires`, `env`, `envVars`, and `primaryEnv` fields. The affected live DeepRead skills are still marked suspicious while their hosted SKILL.md files declare `DEEPREAD_API_KEY` in the documented nested `metadata.openclaw` shape.

## What This Changes

Registry and scanner metadata should extract `metadata.openclaw.requires.env`, `primaryEnv`, and homepage from published files so correctly declared DeepRead env requirements do not create false suspicious flags.

## Best Possible Solution

Make static publish and rescan moderation consume the same normalized parsed OpenClaw/Clawdis metadata used by display and LLM context, add a regression test for the inline JSON frontmatter shape from this report, then rescan or manually review the affected DeepRead versions while leaving broader package metadata ingestion to #1053.

## Repair Work Prompt

Update the static scan path so nested `metadata.openclaw`, `metadata.clawdis`, and `metadata.clawdbot` declarations are treated as declared env vars, preferably by reusing or passing the normalized parsed Clawdis/OpenClaw metadata rather than duplicating divergent parsing logic. Add a regression test using the inline JSON frontmatter shape from this report with `DEEPREAD_API_KEY`, plus tests that broad `process.env` access and undeclared env access still produce `suspicious.env_credential_access`. Keep the change scoped to skill publish/static rescan metadata handling; do not implement broad package-level `metadata.json` ingestion from #1053 unless it is necessary for the shared helper and covered separately.
