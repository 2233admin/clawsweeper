## Summary

Keep open. This is a ClawHub issue, so the repository cleanup policy does not allow auto-closing it, and the substantive problem is still only partially addressed: current main parses `metadata.openclaw` for stored/LLM metadata, but static publish moderation still collects declared env vars only from direct fields while the live DeepRead records remain suspicious.

## What This Changes

The issue asks ClawHub to extract `metadata.openclaw.requires.env`, `primaryEnv`, and related runtime metadata from published skill files so legitimate API-key skills are not falsely flagged as suspicious.

## Best Possible Solution

Keep this issue open. The best path is to make static publish moderation consume the same normalized parsed Clawdis/OpenClaw metadata used by display and LLM context, or explicitly collect env declarations from `metadata.openclaw`, `metadata.clawdis`, and `metadata.clawdbot`; add a regression test using the inline JSON frontmatter from this report; then rescan/backfill or manually review the affected published DeepRead versions while leaving broader package-level metadata ingestion to #1053.
