## Summary

Keep open. Current main still uses the apex host as the CLI site/registry and discovery endpoint, while live production now redirects www to the apex host rather than the reported apex-to-www direction; this needs maintainer verification with a fresh production token before changing canonical host behavior. The ClawHub cleanup policy also says issues should stay open, and the related #101 PR was closed unmerged while #99 remains open for adjacent Unauthorized login failures.

## What This Changes

The issue asks to fix CLI token authentication failing when requests to https://clawhub.ai redirect to https://www.clawhub.ai and lose the Authorization header.

## Best Possible Solution

Keep this issue open for maintainer handling. Verify the default CLI login and whoami flow with a fresh production token against the default registry, explicit CLAWHUB_REGISTRY=https://clawhub.ai, and explicit CLAWHUB_REGISTRY=https://www.clawhub.ai. If auth still crosses origins, fix production routing so authenticated API requests stay on one origin, or deliberately change the canonical host across CLI defaults, discovery metadata, docs, tests, workflows, and release notes together. Coordinate with #99, and do not revive #101 without rechecking current production routing first.
