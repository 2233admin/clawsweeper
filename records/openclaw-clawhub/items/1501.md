## Summary

Keep this issue open. Current main preserves the positional recipient handle through the CLI and HTTP transfer request path for user/personal-publisher transfers, but the report has not been reverified on the deployed path, another user reported the same failure, and the likely org/non-user publisher case remains a real open gap tracked by #1675.

## Best Possible Solution

Keep this issue open for maintainer triage. If the recipient is a personal user publisher, ask the reporter to retry the current published CLI/backend and close manually once verified. If the recipient is an org or other non-user publisher, route the product/code work through open #1675 by migrating transfer records and acceptance rules from user ids to publisher target ids with membership checks. If a personal-user transfer still reproduces `toUserHandle required`, add a commander-level regression around `transfer request <slug> <handle> --message ...` and fix the CLI/API request body path.
