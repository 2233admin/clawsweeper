## Summary

Keep open. The repository-specific ClawHub cleanup policy only permits auto-closing PRs that are certainly implemented on main, and this item is an issue. Substantively, current main also does not appear to implement the requested suppression path: the static scanner still flags any code file containing `child_process` plus `exec`/`spawn`, only `suspicious.env_credential_access` is currently externally clearable, and there is no repo symbol or documented field for `openclaw:allow` or `spawns_process`.

## Best Possible Solution

Keep this issue open for maintainer triage. The best path is either to document that intentional process spawns remain review findings with manual moderation as the clearance path, or to implement a narrow scanner allowlist that verifies exact executable/script path, literal args without shell interpretation, constrained environment inheritance, and hook metadata such as `HOOK.md`, with tests in `convex/lib/moderationEngine.test.ts` and docs in `docs/skill-format.md` or `docs/security.md`.

## Risks / Open Questions

- Adding a metadata-based allowlist would be security-sensitive: a weak implementation could let malicious packages self-declare process execution and bypass a critical static finding.
- Leaving the behavior undocumented will continue to create false-positive reports for legitimate gateway or hook skills that spawn narrowly scoped helper processes.
- The issue names a live published skill, so any production moderation clearance is separate from the source change and should remain a maintainer/moderator decision.
