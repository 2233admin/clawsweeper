## Summary

Keep this issue open. Under the supplied openclaw/clawhub policy, issues are not auto-close candidates, and current main still intentionally blocks same-GitHub sign-in after user-initiated deletion rather than implementing the requested re-registration or admin reset path. The newer UI can surface a clearer deleted-account message, but the underlying recovery flow remains unresolved.

## Best Possible Solution

Keep this open for maintainer product and support triage. Decide whether account deletion is permanently irreversible or whether same-GitHub re-registration should be supported. If recovery is intended, add a backend flow that safely unlinks or supersedes the old Convex Auth identity for self-deleted users only, preserves moderation and ownership protections, revokes old tokens, and covers web OAuth plus CLI/API-token flows with tests. If deletion remains irreversible, improve login/support UX and documentation so affected users know whether manual cleanup is possible.

## Risks / Open Questions

- A recovery path must distinguish self-deleted users from banned or moderator-deactivated users so OAuth rebinding cannot bypass moderation.
- A reset/re-registration implementation would need to preserve publisher handles, skill ownership, audit history, token revocation, and CLI/API-token invariants.
- If irreversible deletion remains the intended policy, the hosted login and support flow still needs a durable, actionable outcome for affected users instead of leaving them stuck.
