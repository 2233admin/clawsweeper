## Summary

Keep open. ClawHub cleanup policy does not auto-close issues, and current main still has a credible implementation path for the reported scroll-triggered reorder: the UI replaces the whole search result set after load-more, while backend relevance recall, fallback sizing, and final slicing all depend on the requested limit.

## What This Changes

The issue reports that relevance-sorted `/skills` search results initially omit high-relevance skills near the top and then reorder after the user scrolls enough to trigger additional loading.

## Best Possible Solution

Fix the search pagination/ranking contract so increasing the displayed result count cannot discover or insert higher-ranked items above the already-rendered first page. A durable implementation would either page over a stable ranked result set or decouple candidate recall and lexical fallback from the display limit, with regression coverage comparing the top results for initial and larger requests.

## Repair Work Prompt

The observable bug is that an initial query such as `image` can render one top ordering, then after scroll/load-more the first page changes because the UI replaces all results with a larger `limit` response and backend search recall/scoring depends on that limit. Keep the fix focused on pagination/ranking stability, not a broad rewrite of direct prefix recall tracked by #1254/#1256. Acceptable approaches include serving load-more from a stable ranked result set or making backend candidate expansion and lexical fallback recall independent of display limit so the top 25 for a query are identical whether requesting 25 or 50.
