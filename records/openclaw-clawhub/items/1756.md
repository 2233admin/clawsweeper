## Summary

Keep open. This is an open ClawHub issue, and the repository cleanup policy only allows automated implemented-on-main closes for pull requests. Substantively, current main still has a plausible scroll-triggered reorder path: the skills UI replaces the whole search result array after load-more increases the requested limit, while backend relevance search changes candidate expansion, lexical fallback size, and final slicing from that same limit.

## Best Possible Solution

Keep this issue open and fix the search pagination/ranking contract. The best path is to make relevance search stable across load-more requests, either with a server-side cursor over a fixed ranked result set or by decoupling candidate expansion and fallback recall from the display limit so requesting 50 results cannot discover higher-scoring entries above the already-rendered first 25. Add regression coverage comparing the top results for an initial 25-result request against a larger request for the same query.

## Clownfish Work Prompt

Fix the ClawHub /skills relevance search stability bug where scrolling/load-more can reorder the already-rendered top results for the same query. Current UI requests 25 results first, then increases `searchLimit` to 50 and replaces the whole result array; backend `searchSkills` changes vector candidate expansion, lexical fallback size, and final slicing based on `args.limit`. Make the top N stable across larger limit requests. Preferred boundaries: adjust `convex/search.ts` so candidate recall/scoring for relevance is independent of display limit, or implement an equivalent stable pagination contract; update `src/routes/skills/-useSkillsBrowseModel.ts` only if needed to preserve already-rendered ordering under the new backend contract. Add regression tests proving the first 25 results for a query are identical to the first 25 of a 50-result request when higher-scoring candidates are only discovered by the larger request today.
