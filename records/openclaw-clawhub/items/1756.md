## Summary

Keep open. This is an issue, and the openclaw/clawhub policy for this review run only allows auto-closing pull requests that are certainly implemented on main. Independent of that policy gate, current main still has a plausible version of the reported behavior: search load-more changes the requested limit, replaces the full result array, and the backend relevance search builds candidate/fallback pools from that limit before sorting and slicing.

## Best Possible Solution

Keep this issue open. The best fix is to make relevance search pagination stable, either with server-side cursors over a fixed ranked result set or by decoupling candidate expansion from the displayed limit so increasing the limit cannot discover higher-scoring items above already-rendered results. Add a regression test comparing the top relevance results from the initial 25-result request with the larger load-more request.
