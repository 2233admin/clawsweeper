## What This Changes

The reporter asks for `/skills` relevance search to keep high-relevance results visible on initial render instead of inserting or reordering them only after a scroll/load-more cycle.

## Best Possible Solution

Make `/skills` relevance pagination stable so the first response for a query remains a prefix of later larger or paged responses. A durable implementation should rank over a stable candidate set or page a stable ranked result set, then add regression coverage comparing initial and load-more search requests.

## Repair Work Prompt

Fix `/skills` relevance pagination stability for active search queries. On current main, the UI increases `searchLimit` and replaces `searchResults`, while `convex/search.ts` derives candidate bounds and fallback scan sizes from the requested limit; this can let load-more insert higher-ranked results above the initial page. Implement stable prefix behavior so the first page for a query remains unchanged when loading more, likely by decoupling recall/ranking from display limit or paging over a stable ranked set. Keep ranking semantics as close as possible and avoid broad search product rewrites. Add regression tests that compare a query's initial results with the first entries returned after load-more or a larger limit. Coordinate with adjacent PR #1256 if touching direct lexical recall, but do not fold in unrelated backfill or schema work unless required. If touching skill stat reads, follow `readCanonicalStat` / dual-write migration rules. Include a concise changelog or PR note if user-visible search behavior changes.
