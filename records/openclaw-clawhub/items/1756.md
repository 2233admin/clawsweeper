## Summary

Keep open. Current main still has a plausible path for the reported scroll-triggered search reorder: the UI replaces the whole search result array when load-more increases the requested limit, and the backend relevance search changes its vector candidate pool, exact-match stopping condition, fallback size, and final slice from that same limit. ClawHub cleanup policy also keeps issues open; only PRs can be auto-closed as implemented.

## What This Changes

The issue requests that `/skills` relevance search keep initial top results stable instead of discovering or reordering high-relevance skills only after a scroll-triggered load-more cycle.

## Best Possible Solution

Keep this issue open and fix the search pagination/ranking contract. The most durable path is to make relevance search stable across load-more requests, either by serving a cursor over a fixed ranked result set or by decoupling candidate expansion and lexical fallback recall from the display limit so a 50-result request cannot discover higher-scoring entries above the first 25. Add regression coverage comparing the top results for an initial 25-result request against a larger request for the same query.

## Repair Work Prompt

Fix the `/skills` relevance search pagination/ranking instability reported in openclaw/clawhub#1756. Current behavior: the UI requests 25 results, then scroll load-more requests 50 and replaces the entire result array; backend relevance ranking changes its candidate/fallback pool based on `limit`, so higher-scoring skills can appear above previously rendered results after scrolling. Implement a narrow fix so the top results for a query are stable across load-more requests. Prefer a backend ranking contract fix: either serve a cursor over a fixed ranked result set, or decouple vector candidate expansion and lexical fallback recall from the display limit so requesting 50 cannot discover entries that should have ranked above the first 25. Adjust the client only as needed to fit the backend contract. Add regression coverage that compares the ordered top 25 for the same query when requesting 25 vs. a larger limit, plus any UI coverage needed for load-more behavior. Do not change unrelated ranking weights, release/package metadata, or broad search UX. Add a concise changelog entry only if the repo convention expects user-facing search fixes to be logged.
