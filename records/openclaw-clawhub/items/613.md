## Risks / Open Questions

- Device-flow auth is token issuance infrastructure and needs explicit security review for code entropy, expiry, polling interval/backoff, denial states, code-to-user binding, rate limiting, replay resistance, and auditability.
- The linked implementation PR is open and described as client-side only, so merging a partial CLI flow without server endpoints and a verification page would not solve the reported headless-auth problem.
- The current manual token-paste workaround still requires an already browser-authenticated user to create or obtain the first token, so it does not provide true headless onboarding.
