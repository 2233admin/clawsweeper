## Summary

Keep this issue open. Current main still has the reported failure shape: `clawhub sync` classifies by bare slug, publishes selected skills sequentially, and a publish-time slug conflict from the backend is rethrown without a per-skill recovery path, so one conflicting skill can abort the rest of the sync batch. The ClawHub cleanup policy also says issues stay open unless maintainers handle them explicitly.

## What This Changes

Fix `clawhub sync` so a slug collision owned by another publisher does not prevent the remaining selected skills from syncing.

## Best Possible Solution

Keep the issue open and fix CLI sync narrowly: detect the reported slug-taken publish failure during `cmdSync`, record that skill as skipped or failed with the existing owner URL when present, continue publishing the rest of the selected skills, and print a clear final summary. Avoid changing backend slug uniqueness or introducing broad owner-scoped skill publishing in this fix.
