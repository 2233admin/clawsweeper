## Summary

Keep this issue open. The ClawHub cleanup policy keeps issues open, and current main still does not pass the publishing user's GitHub author metadata into GitHub backup commits; both publish-time and bulk backup paths only carry an owner handle, and the Git commit payload omits author/committer fields.

## Best Possible Solution

Keep this issue open. Implement future-only commit attribution in the GitHub backup path by carrying the publishing user's GitHub identity into backupSkillToGitHub, constructing a privacy-safe Git commit author object, deciding committer behavior explicitly, updating the bulk sync path as well as the publish path, and adding focused tests that assert the GitHub commit payload includes the intended author metadata.

## Risks / Open Questions

- A future implementation should avoid leaking private GitHub email addresses; GitHub's numeric-id noreply format is safer than blindly using stored email.
- Existing bot-authored backup history should remain unchanged unless maintainers explicitly choose a separate history rewrite plan.
- Both publish-time backup and bulk sync paths need the same attribution behavior, or old skills and republished skills will diverge.
