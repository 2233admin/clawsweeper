Keep this issue open. Current main still creates skill backup commits without passing a GitHub user author identity, and ClawHub cleanup policy keeps issues open unless a PR is certainly implemented on main. ## Best Possible Solution

Keep this issue open for maintainer attribution policy, then implement a future-only fix that builds a privacy-safe GitHub author object from existing GitHub identity data, passes it through both publish-time and bulk backup paths, and includes it in both skill and metadata GitHub create-commit payloads. Existing openclaw/skills history should stay untouched unless maintainers explicitly choose a rewrite/backfill.

## Risks / Open Questions

- Do not use private `users.email` values as Git author emails; prefer a GitHub noreply-style address derived from immutable providerAccountId plus current login, or another maintainer-approved privacy-safe source.
- Maintainers need to decide whether the Git author should be the skill owner, the version creator, or the publishing actor, especially for org/shared-owner and bulk-sync cases.
- The GitHub App/bot should probably remain the committer even if the user becomes the author, so write authority stays clear.
- Bulk backup must get any needed identity data without adding a broad unindexed scan or expensive per-item full-document pattern.
