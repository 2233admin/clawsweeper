## Summary

Keep open. The request is still valid: current main creates skill backup commits without passing Git author or committer identity, and ClawHub cleanup policy keeps issues open unless a PR is certainly implemented on main. The remaining blocker is maintainer policy for attribution and privacy-safe email handling, not lack of a code path.

## What This Changes

The issue asks for skill addition/update commits mirrored to the skills repository to use the owner’s GitHub account details as the Git commit author instead of the ClawHub GitHub App/bot identity.

## Best Possible Solution

Define the attribution policy first, then thread a privacy-safe Git author object through publish-time and bulk backup paths into both skill and metadata Git commit payloads, with explicit bot fallback behavior and tests for personal owners, org/publisher ownership, missing identity, and bulk sync.

## Risks / Open Questions

- Maintainers still need to decide whether Git author attribution should represent the publishing actor, a personal owner, an organization/publisher owner, or a bot fallback for bulk and restore-style paths.
- Using stored OAuth email directly as a Git author email could publish private addresses in public Git history; a safe implementation likely needs a maintainer-approved noreply-style source.
- ownerHandle alone is not enough author identity because it can represent a display fallback, a user handle, or organization/publisher ownership rather than a concrete GitHub user account.
