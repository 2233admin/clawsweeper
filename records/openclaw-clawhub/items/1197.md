## Summary

Keep open. This is an open ClawHub issue, and the repository cleanup policy only allows auto-close for PRs that are certainly implemented on current main. The reported Convex failure shape still exists: publisher-triggered owner digest sync now schedules package and skill work separately, but each internal mutation can still loop over `.paginate()` more than once for owners with more than one page of packages or skills.

## Best Possible Solution

Keep this issue open and land one narrow backend fix for the publish-pagination cluster. Convert the package and skill owner-publisher digest sync internal mutations to cursor-based self-scheduling batches that perform at most one `.paginate()` per invocation, preserve indexed `by_owner_publisher` reads, add multi-page regression tests, and then consolidate duplicate reports such as #1195 and #1171 after the fix is merged and deployed.

## Clownfish Work Prompt

Fix the ClawHub publish-pagination bug reported in this issue and related reports (#1171, #1175, #1176, #1182, #1184, #1195, #1205). Current main schedules owner-publisher package and skill digest syncs separately, but `syncPackageSearchDigestsForOwnerPublisherId` and `syncSkillSearchDigestsForOwnerPublisherId` still loop over `.paginate()` inside one internal mutation invocation. Change these owner-publisher digest rebuilds to cursor-based self-scheduling batches that perform at most one `.paginate()` per invocation. Add regression tests showing a multi-page owner-publisher package/skill digest sync schedules continuation instead of issuing a second paginated query in the same invocation.
