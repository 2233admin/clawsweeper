## Summary

Keep open. The report still matches current main: public skill detail responses return forkOf/canonical metadata for referenced skills without applying the normal public visibility checks, and the frontend renders that metadata as “duplicate of @owner/slug”. Repository policy for openclaw/clawhub also says issues should stay open; only PRs that are certainly implemented on main are auto-close candidates.

## Best Possible Solution

Keep this issue open and land a narrow fix in convex/skills.ts: public getBySlug should only include forkOf/canonical metadata when the referenced skill and owner pass the same public visibility rules as normal public skill detail pages. Add regression coverage in convex/skills.public.test.ts for a public skill whose duplicate/canonical target is soft-deleted or otherwise non-public. If maintainers want stored production data cleaned too, clear the affected duplicate relationship through the existing moderator setDuplicate path.
