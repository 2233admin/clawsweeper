## Summary

Keep open. Repository policy keeps ClawHub issues open in this review lane, and current main still only partially addresses the reported artifact false positives: package UI/CLI paths ignore `.git/`, but skill web/API/GitHub publish paths still only drop macOS junk, while `__pycache__/`, `.pyc`, and `.pyo` are not handled by a shared publish-artifact cleanup layer.

## Best Possible Solution

Keep this issue open and land a narrow shared publish-artifact cleanup helper used before storage and before static/LLM moderation across web skill upload, API JSON/multipart skill publish, GitHub import, CLI skill publish, package upload, and package multipart publish where applicable. Cover `.git/`, `__pycache__/`, `.pyc`, and `.pyo`, update tests for each affected path, and document the supported cleanup contract if it becomes user-facing.

## Risks / Open Questions

- Artifact cleanup should happen before persistence and scanner input; suppressing findings after scanning would risk hiding real malicious content instead of removing standard build/VCS artifacts from the published artifact set.
- The ignore helper should be narrowly scoped to well-known generated/VCS artifacts (`.git/`, `__pycache__/`, `.pyc`, `.pyo`) so it does not become a broad scanner bypass.
- Package and skill publish paths use different upload flows today, so a partial fix could leave one path still storing or scanning the reported artifacts.
