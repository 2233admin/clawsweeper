## Summary

Keep open. Current main still contains the reported double-serialization path: the affected CLI commands pass pre-stringified bodies, and the shared HTTP client serializes those bodies again before sending them. The related fix PRs discussed in the issue were closed without merging, and the current tests still assert the buggy string-body shape rather than a plain object.

## Best Possible Solution

Land a scoped CLI fix that passes plain objects to `apiRequest` for transfer request, skill rename, and skill merge; add tests that assert those command bodies are objects, and consider hardening `apiRequest`/curl handling to pass through already-serialized string bodies defensively.
