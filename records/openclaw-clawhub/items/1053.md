## Summary

Keep open. This is an open ClawHub issue, and the repository-specific policy only permits auto-closing ClawHub pull requests that are certainly implemented on current main. Substantively, current main still does not show a complete standalone metadata.json/package.json env-declaration path into normalized registry metadata; related open reports #1432 and #1667 describe the same mismatch pattern.

## Best Possible Solution

Keep this issue open for maintainer triage. Decide whether standalone metadata.json/package.json env declarations are supported skill metadata sources. If yes, parse and merge them during publish into the same normalized parsed.clawdis requires.env/envVars shape used by latestVersionSummary, UI/API install prompts, static scan, and LLM prompt, then add regression tests plus a backfill/rescan path for affected versions. If no, update docs and CLI/server validation so publishers are told to declare env requirements in SKILL.md frontmatter only. Treat uv as already supported unless scanner calibration still warns on it.

## Risks / Open Questions

- Current main has partial frontmatter support, so some newly published skills may no longer hit the exact LLM prompt mismatch when they use metadata.openclaw in SKILL.md; the standalone metadata.json/package.json path still needs an explicit regression test before this can be considered fixed.
- Static scan and LLM scan do not currently appear to share a single normalized metadata source, so fixing only one scanner path could leave user-visible false positives in the other path.
- Related reports #1432 and #1667 remain open and describe the same registry/env mismatch pattern, so maintainers may want to consolidate the fix and rescan/backfill affected skills together.
