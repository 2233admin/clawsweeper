## Summary

Keep open. This is a ClawHub issue, and the repository policy only permits auto-closing ClawHub pull requests that are certainly implemented on current main. The underlying product gap also still exists: current main renders OpenClaw scanner failures as a bare Error, suppresses stored LLM failure summaries in the scan panel, preserves API semantics where an LLM error can make overall security status error, and lacks author-facing docs for what to do next. Related false-positive/rescan issue #669 is still open.

## Best Possible Solution

Keep this issue open for maintainer triage. Add scan-page and public docs guidance explaining what OpenClaw `Error` means, expose a safe scanner failure explanation or stored summary to authors, document whether authors should wait, change the skill, or request a rescan, and define a conservative moderation policy that separates scanner-side LLM failures from actual suspicious evidence. Coordinate with the broader false-positive/rescan workflow in #669.
