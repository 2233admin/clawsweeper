## Summary

Keep open. This is an open ClawHub security-sensitive issue, and the repository policy only permits auto-closing ClawHub PRs that are certainly implemented on current main. The substantive report still reproduces: ClawHub's public skill APIs return 404 for the reported skill, while the public GitHub mirror still serves the skill files, and current main relies on periodic mirror pruning rather than deterministic purge from deletion/moderation paths.

## Best Possible Solution

Keep this issue open for maintainer/security follow-up. The best path is to make malicious or removed skill cleanup deterministic: purge or tombstone openclaw/skills entries from moderation/delete/ban paths, add regression tests around mirror deletion and prune cursor behavior, verify production github-backup-sync/prune health, manually remove known malicious stale entries, and coordinate downstream cache/advisory behavior before closing.

## Risks / Open Questions

- The public GitHub mirror still serves the reported malicious SKILL.md and metadata while ClawHub's public APIs return 404, so downstream installers or marketplaces can keep exposing stale malicious content.
- Mirror cleanup depends on production GitHub App credentials, cron health, cursor progress, batching, and GitHub permissions rather than being deterministic on high-risk delete/moderation actions.
- Deletion, ban, and malware-hiding paths do not appear to synchronously call the existing mirror delete action.
- Existing tests cover backup-page filtering but not prune/delete execution or cron wiring, so regressions in mirror cleanup could pass unnoticed.
- Downstream marketplaces and caches may continue serving stale copies even after the mirror entry is eventually removed.
