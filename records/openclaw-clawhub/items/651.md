## Summary

Keep this issue open. ClawHub review policy keeps issues open, and current main still has a real CLI diagnostics gap: `search` and token login go through proxy/timeout-aware HTTP code, but transport failures can still surface as the raw `fetch failed` message through the shared formatter.

## What This Changes

The issue requests help for `clawhub search "web"` failing with a bare `fetch failed` error, later echoed for token login as well.

## Best Possible Solution

Keep the issue open and land a focused CLI diagnostics improvement in the shared CLI HTTP/error path so DNS, TLS, proxy, timeout, and curl failures produce actionable, redacted messages while preserving existing successful HTTP, status, retry, and rate-limit behavior.

## Repair Work Prompt

Fix the CLI diagnostics gap from openclaw/clawhub#651. Current `clawhub search` and `clawhub login --token` failures can surface as a bare `fetch failed` through `formatError(error)`. Implement a narrow, shared CLI transport-error formatting improvement so DNS, TLS/certificate, proxy, timeout/abort, Undici transport codes, and Bun/curl failures produce actionable, redacted guidance. Keep successful HTTP behavior, HTTP status errors, retry/rate-limit messages, schemas, and request construction unchanged. Do not expose Authorization headers, API tokens, query tokens, proxy credentials, or full sensitive environment values. Add focused tests for the formatter and/or HTTP client paths, including at least one `fetch failed` with a nested cause and one timeout or curl failure case. Add an Unreleased changelog Fixes bullet for the CLI user-facing diagnostics change. Do not broaden this into a network stack rewrite or change install/search/login semantics.
