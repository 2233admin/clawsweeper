## Summary

Keep this issue open. Current main still lacks GitHub Issue Forms, `.github/ISSUE_TEMPLATE/config.yml`, and the requested broad support-intake labels. The related merged rescan-guidance work is useful, but it only handles a narrow labeled rescan path and does not implement structured intake for auth, publish, search, ownership, false-positive appeals, or feature requests. ClawHub policy also keeps issues open unless they are PRs certainly implemented on main.

## Best Possible Solution

Keep this issue open for maintainer/product follow-up. The best path is to approve the support categories, label names, and canonical threads, then add `.github/ISSUE_TEMPLATE` YAML forms plus `config.yml` using form-level static labels where possible. Only add a small `issues: write` triage workflow if GitHub issue-form labels are not enough, and security-review that workflow separately.
