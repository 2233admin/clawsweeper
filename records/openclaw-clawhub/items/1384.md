## Summary

Keep this issue open. Current main still does not contain GitHub issue forms or a broad support-intake labeling setup for the requested ClawHub workflows, and the only related merged automation is the narrower rescan-guidance path that handles explicit rescan/re-evaluation requests after labeling. Repository policy for openclaw/clawhub also says issues are not auto-closed unless they are PRs certainly implemented on main.

## What This Changes

Request structured GitHub issue forms, required support metadata, canonical-thread guidance, and auto-applied triage labels for common ClawHub support categories.

## Best Possible Solution

Keep this issue open for maintainer/product follow-up. The best path is to approve the support categories and label names, add .github/ISSUE_TEMPLATE YAML forms plus config.yml, use form-level static labels where possible, link forms to maintainer-approved canonical threads, and only add a small issues: write triage workflow if GitHub issue-form labels are not enough.
